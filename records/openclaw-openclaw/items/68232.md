## Label Justifications

- `P1`: The issue describes a source-provable Telegram channel workflow break that loses active conversation context for real users during config hot reload.
- `impact:message-loss`: Pending skipped Telegram group messages stored in the in-memory history buffer are discarded when the channel restarts.
- `impact:session-state`: The affected state is Telegram-local conversation context carried between group messages.

## Summary

Keep open: current main still has the Telegram-specific state-loss path. The merged restart-drain work reduces active-run disruption, but a restart-required `channels.telegram.*` reload still stops and starts Telegram with a freshly allocated group-history map.

## What This Changes

The issue reports that Telegram config hot reload restarts the channel and discards in-memory Telegram conversation context.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token/chat/topic, and trim retained entries to the new effective history limit.

## Reproduction Assessment

Yes, by source inspection. A `channels.telegram.*` config change still plans a Telegram channel restart, the reload handler still stops and starts the channel, and the new bot instance still gets an empty `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff during restart-required Telegram reloads; gateway-level restart draining is useful but does not preserve this bot-local map.
