## Summary

Keep open. Current main still restarts the Telegram channel for `channels.telegram.*` hot-reload changes and constructs a fresh Telegram bot with a fresh pending group-history map, while the linked restart-drain work only delays restarts until active work drains.

## What This Changes

The issue asks OpenClaw to avoid losing Telegram in-memory conversation context when config hot reload restarts the Telegram channel.

## Best Possible Solution

Preserve or transfer Telegram pending group history across Telegram restart-required config reloads, scoped by account/token and history key, and trim retained entries to the effective `historyLimit`.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: changing `channels.telegram.historyLimit` matches the Telegram reload prefix, enters `plan.restartChannels`, runs `stopChannel`/`startChannel`, and creates a new bot with an empty `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff for pending group history; the already-merged restart-drain work is useful but does not preserve this Telegram-local context.
