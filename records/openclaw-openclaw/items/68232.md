## Summary

Current main still has the source-level failure path: Telegram config changes are planned as channel restarts, the reload handler stops and starts the channel, and each new Telegram bot gets a fresh pending group-history map. The merged restart-drain work helps active runs drain before restart, but it does not preserve Telegram-local context across the restart.

## What This Changes

The issue asks for Telegram config hot reloads to stop discarding in-memory conversation context, especially pending group history, when `openclaw.json` changes trigger a channel restart.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token and history key and trimmed to the new effective `historyLimit`.

## Reproduction Assessment

Yes. Changing `channels.telegram.historyLimit` follows a clear current-main path from the Telegram reload prefix into `plan.restartChannels`, then `stopChannel`/`startChannel`, then a new Telegram bot with an empty `groupHistories` map; I did not run a live Telegram group in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is Telegram-owned state retention or handoff for pending group history during restart-required reloads; the already-merged restart-drain work is useful but does not preserve this map.
