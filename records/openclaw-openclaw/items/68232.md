## Summary

Keep this open: current main still restarts Telegram for `channels.telegram.*` hot-reload paths and recreates the Telegram bot with a fresh pending group-history map, so the channel-specific state-loss bug remains despite the merged gateway restart-drain work.

## What This Changes

The issue requests that Telegram config hot reloads stop discarding in-memory conversation context such as pending group history when the channel is restarted.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token and chat/topic history key and trimmed to the new effective `historyLimit`.

## Reproduction Assessment

Yes. Changing `channels.telegram.historyLimit` follows a current-main source path from the Telegram reload prefix into `plan.restartChannels`, then `stopChannel`/`startChannel`, then a newly constructed Telegram bot with an empty `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state handoff for pending group history during restart-backed Telegram reloads; the merged gateway restart-drain work is useful but does not preserve this map.
