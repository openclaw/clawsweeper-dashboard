- `impact:message-loss`: The lost `groupHistories` buffer can remove pending Telegram group messages from later prompt context.

## Summary

Current main still has the source-level Telegram state-loss path. Recent restart-drain and Telegram polling-restart fixes reduce adjacent reload damage, but a `channels.telegram.*` reload still restarts the channel and constructs a fresh Telegram bot-local `groupHistories` map.

## What This Changes

The issue asks for Telegram config hot reloads to avoid losing in-memory conversation context when `openclaw.json` changes trigger a channel restart.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped to account, token, chat, and topic, and trim retained entries to the new effective `historyLimit`.

## Reproduction Assessment

Yes, by source inspection. A `channels.telegram.*` config change still plans a Telegram channel restart, the reload handler still stops and starts the channel, and the new Telegram bot still constructs a fresh `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff during restart-required Telegram reloads; gateway-level restart draining and polling restart recovery do not preserve this bot-local context.
