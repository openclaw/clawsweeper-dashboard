## Summary

Keep open. Current main still maps Telegram config changes to a channel restart and recreates Telegram pending group-history state per bot instance, so the restart-drain mitigation does not fix the reported context loss.

## What This Changes

The issue requests that Telegram config hot reloads avoid destroying in-memory Telegram conversation context, especially group pending-history state, when config changes trigger a channel restart.

## Best Possible Solution

Preserve or transfer Telegram pending group history across config-required Telegram restarts, scoped by account/token and history key, and trim retained entries to the effective `historyLimit`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: changing `channels.telegram.historyLimit` matches the Telegram reload prefix, enters `plan.restartChannels`, runs `stopChannel`/`startChannel`, and creates a new bot with an empty `groupHistories` map.

## Solution Assessment

Mostly yes. The narrowest maintainable fix is Telegram-owned state retention or handoff for pending group history; the already-merged restart-drain work is useful but does not preserve this Telegram-local context.
