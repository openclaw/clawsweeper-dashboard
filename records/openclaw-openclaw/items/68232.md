## Summary

Keep open. Current main still turns `channels.telegram.*` edits into a full Telegram channel stop/start, and the pending Telegram group-history buffer is still a fresh in-memory map per bot instance, so the reported state loss remains valid after the restart-drain mitigation in #71465.

## What This Changes

The issue asks for Telegram config hot reload to stop destroying in-memory conversation context when `openclaw.json` changes.

## Best Possible Solution

Preserve or transfer Telegram pending group history across config-required Telegram restarts, scoped by account/token and history key, and trim retained entries to the effective `historyLimit`.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: changing `channels.telegram.historyLimit` matches the Telegram reload prefix, enters `plan.restartChannels`, runs `stopChannel`/`startChannel`, and constructs a new bot with an empty `groupHistories` map.

## Solution Assessment

Unclear. #71465 is a useful drain mitigation, but it does not solve Telegram-local pending history loss; the narrowest maintainable repair is Telegram-owned state retention unless maintainers choose a generic channel reload seam.
