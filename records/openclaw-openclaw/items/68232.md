## Label Justifications

- `P1`: Editing Telegram config can break an active channel workflow by dropping in-memory context during hot reload.
- `impact:message-loss`: Pending group messages stored only in the Telegram group-history map can disappear before they are included in a later mentioned turn.
- `impact:session-state`: The bug is about Telegram bot-local conversation state being discarded when the channel is restarted.

## Summary

Keep open: current main still has the Telegram state-loss path. The merged restart-drain and polling-restart fixes reduce adjacent damage, but a `channels.telegram.*` hot reload still restarts Telegram and the new bot instance starts with a fresh group-history map.

## What This Changes

The issue reports Telegram config hot reloads restarting the channel and dropping in-memory group context such as pending group history.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token/chat/topic and trimmed to the new effective `historyLimit`.

## Reproduction Assessment

Yes, by source inspection. A `channels.telegram.*` change still reaches `plan.restartChannels`, the hot-reload handler still stops and starts Telegram, and `createTelegramBotCore` still creates a fresh `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff during restart-required Telegram reloads; gateway-level restart draining is useful but does not preserve this bot-local map.
