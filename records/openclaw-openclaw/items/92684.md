## What This Changes

The issue asks `openclaw doctor` to stop warning on an empty top-level `groupAllowFrom` when every enabled real account has its own effective non-empty group sender allowlist.

## Best Possible Solution

Make the doctor scan account-aware so fallback parent records do not warn as phantom accounts when all enabled real accounts have effective non-empty group sender allowlists, while preserving true empty-allowlist warnings.

## Reproduction Assessment

Yes from source at medium confidence: current `main` still evaluates the top-level channel record separately before account records, while runtime/account tests show named account group allowlists can be effective for inbound handling. ## Solution Assessment

Yes: a focused doctor-scan fix is the best layer because runtime gating already accepts account-scoped allowlists. Changing channel runtime or adding config would solve the wrong problem.
