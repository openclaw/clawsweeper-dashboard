## Summary

Keep open: the PR is a focused fix for the linked Claude CLI rate-limit issue, but it is not merge-ready because the hard-limit status does not match the official SDK contract and the supplied proof is only test output rather than a real Claude CLI run.

## What This Changes

The branch adds Claude CLI `rate_limit_event` parsing in `extractCliErrorMessage` plus parser and supervisor-capture tests for warning success and hard-limit classification.

## Best Possible Solution

Handle warning statuses as success notices and map the actual hard-limit status to `rate_limit` in the CLI parser/live-session path, backed by a redacted real Claude CLI JSONL fixture and targeted regression tests.

## Reproduction Assessment

Unclear for the original issue: the report includes a detailed live `allowed_warning` stream and current source has a parser gap, but I did not run a Claude Pro Max reproduction. The PR finding is source-reproducible from the branch checking only `denied` while the official SDK contract lists `rejected`.

## Solution Assessment

No as-is. The parser-hook direction is narrow, but the hard-limit literal should match the documented contract or be justified by a redacted raw CLI fixture, and external real behavior proof is still required.
