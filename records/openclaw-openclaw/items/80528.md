## Summary

Keep open: this is a plausible bug-fix PR for the linked Claude CLI cooldown issue, but it should not merge as-is because it lacks real behavior proof and the hard-limit branch keys on `denied` while Claude's Agent SDK docs document the hard status as `rejected`.

## What This Changes

The branch adds explicit Claude CLI `rate_limit_event` handling to the CLI error parser and adds parser/supervisor tests for allowed-warning success and hard-limit classification.

## Best Possible Solution

Handle warning statuses as success notices and map the actual hard-limit status to `rate_limit` in the parser/live-session path, backed by a redacted real Claude CLI JSONL fixture and targeted tests.

## Reproduction Assessment

Unclear for the original issue: the report includes a detailed live `allowed_warning` stream and current source has a parser gap, but I did not run a Claude Pro Max reproduction. The PR review finding is source-backed by the documented rate-limit status contract.

## Solution Assessment

No as-is. The direction is narrow, but the hard-limit status should match documented `rejected` or a redacted real CLI fixture, and external real behavior proof is still required.
