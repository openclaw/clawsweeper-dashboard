## Summary

Keep open. Current main still loads and watches the personal `~/.agents/skills` source unconditionally, while the strict public config schema and docs expose no source-level opt-out. The related #64639 report was already closed as a duplicate of this older canonical issue, so this remains valid narrow core skills/config work rather than cleanup to close.

## What This Changes

Add a configuration option to disable scanning `~/.agents/skills` and possibly other agent skill roots so Claude Code skill directories do not automatically conflict with or bloat OpenClaw skill prompts.

## Best Possible Solution

Add a backwards-compatible `skills.load` source opt-out that is applied before filesystem discovery and watcher registration, preserving today’s default roots and precedence when unset. Update the schema/help/docs/generated config output, add targeted loader and watcher tests for disabled `agents-skills-personal`, and keep native on-demand skill routing as a separate larger design topic.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: a skill under a mocked home `.agents/skills` directory is loaded into the prompt, and the watcher registers the home `.agents/skills` path without a source-specific config guard.

## Solution Assessment

Yes in direction. A source-level opt-out is narrower and more maintainable than requiring many `skills.entries.*.enabled: false` entries, and it avoids conflating this coexistence fix with broader on-demand skill selection work.
