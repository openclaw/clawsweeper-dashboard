## Summary

Current main still provides a forward-or-cancel onboarding flow, and all related implementation PRs are closed unmerged. The feature remains valid but needs maintainer agreement on shared navigation and answer-invalidation semantics before implementation.

## What This Changes

Add backward navigation to the onboarding wizard while preserving prior selections for subsequent forward navigation.

## Best Possible Solution

Define one typed previous-step and history contract with explicit downstream-answer invalidation, then apply it consistently to setup orchestration, prompt adapters, WizardSession, Gateway schemas and handlers, clients, documentation, and regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: onboarding supports forward answer submission and cancellation but has no Back result, history API, or previous-step RPC.

## Solution Assessment

No, adding a local `[b] Back` prompt option alone is not the best solution. A canonical state and history model shared by CLI and Gateway clients is required to prevent inconsistent behavior and stale answers.
