## Summary

Current main still has a forward-or-cancel onboarding flow, and no open or merged related PR now owns the missing Back behavior. The request remains aligned with easier onboarding, but implementation should wait for a maintainer decision on navigation scope and how changing an earlier choice invalidates dependent later answers across CLI, Gateway, and native clients.

## What This Changes

Add backward navigation to the onboarding wizard while preserving previously entered selections.

## Best Possible Solution

Add one typed navigation/history contract to the canonical wizard model, expose it through the Gateway and native-client surfaces, and explicitly invalidate downstream answers whose prerequisites changed while preserving still-valid values.

## Reproduction Assessment

Yes. Current main exposes only forward progression and cancellation in the prompt, session, Gateway, and native-client contracts, so the missing behavior is directly reproducible from source with high confidence.

## Solution Assessment

No, adding `[b] Back` independently to prompt widgets is too narrow. The maintainable solution is a shared state/navigation contract with explicit downstream-answer invalidation across every wizard client.
