## Summary

Keep open: current main still exposes onboarding as a forward/cancel wizard, and the direct Back-navigation implementation was closed unmerged with no shipped replacement.

## What This Changes

The issue requests in-session Back navigation for `openclaw onboard`, preserving earlier answers when users move backward and forward.

## Best Possible Solution

Introduce a shared typed Back/previous contract and setup-flow history model across the wizard prompter, session, Gateway protocol, and clients, rather than a one-off local menu option.

## Reproduction Assessment

Yes, at source level: current `openclaw onboard` uses a sequential flow, and the shared prompt/session/protocol/client surfaces only support next, answer, cancel, and status. I did not run the live wizard because that can write config/state in this read-only review.

## Solution Assessment

Yes directionally, but it needs maintainer product/protocol approval and a cross-surface implementation. The safer solution is a shared typed Back/previous contract rather than adding `[Back]` in only one wizard surface.
