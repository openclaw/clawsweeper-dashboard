## Summary

Keep open: current main still exposes onboarding as a forward/cancel wizard, and the direct Back-navigation PR is closed unmerged with no superseding shipped implementation.

## What This Changes

The issue requests in-session Back navigation for `openclaw onboard`, with prior answers preserved when moving backward and forward.

## Best Possible Solution

Add an additive typed Back/previous contract across the shared wizard prompter, session, Gateway protocol, and clients, with a setup-flow history model that preserves answers while moving backward and forward.

## Reproduction Assessment

Yes, at source level: current `openclaw onboard` uses a sequential flow, and the shared prompt/session/protocol/client surfaces only support next, answer, cancel, and status. I did not run the live wizard because that can write config/state in this read-only review.

## Solution Assessment

Yes directionally, but it needs maintainer product approval and a cross-surface implementation. The safer solution is a shared typed Back/previous contract rather than a one-off `[Back]` option in only part of the wizard.
