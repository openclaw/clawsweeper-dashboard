## Summary

Keep open. The Discord display bug is valid and current main still has the ASCII-only display slug path, but this PR is not merge-safe because it changes Discord guild policy identity from stable guild IDs to mutable slugs and its Unicode fallback does not cover mixed ASCII/non-ASCII channel names promised in the PR body.

## What This Changes

The PR adds a Unicode-preserving fallback for Discord display channel slugs when normalization returns empty, and changes Discord message/component GroupSpace construction to prefer guild slugs over guild IDs.

## Best Possible Solution

Implement a display-only Unicode-preserving Discord channel/thread label path on current main that covers pure and mixed non-ASCII names, while leaving normalizeDiscordSlug, configChannelSlug, allowlist matching, and ID-first GroupSpace policy routing unchanged.

## Reproduction Assessment

Yes. The current-main reproduction is a static path: a channel name such as 실험 normalizes to an empty displayChannelSlug, and message context only emits GroupChannel when that slug is non-empty; the PR defects are also reproducible from the supplied diff and policy lookup code.

## Solution Assessment

No. The display fallback is in the right area, but the proposed patch is not the best fix because it mixes in slug-first GroupSpace behavior and only preserves Unicode when the normalized value is completely empty; the safer fix is a dedicated display-label helper plus focused regression tests.
