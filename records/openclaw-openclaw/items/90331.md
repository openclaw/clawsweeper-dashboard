## Summary

Keep open: source review found a blocking compatibility concern in the new restart preflight. The config self-edit guard coverage is sound, but the PR changes agent-triggered restart from best-effort recovery to fail-closed on local config snapshot failures while sibling restart paths still keep the existing scheduling behavior.

## What This Changes

The PR keeps agent-facing gateway config self-edits inside the narrow allowlist and adds a local config.get preflight that refuses agent-triggered restarts when the config snapshot is invalid or unreadable.

## Best Possible Solution

Either preserve best-effort agent restart as the default or get explicit maintainer acceptance that agent-triggered restart should fail closed on local config snapshot failures while sibling restart paths remain different.

## Reproduction Assessment

Yes by source inspection: current main schedules the gateway restart after the commands.restart gate without config.get, while the PR refuses before scheduling when config.get fails or returns valid:false.

## Solution Assessment

No, not as-is. The protected config-write coverage is appropriate, but the fail-closed restart preflight is a compatibility policy change rather than a clearly best default.
