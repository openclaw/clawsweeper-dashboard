## What This Changes

The branch threads `currentInboundAudio` through active embedded-run queueing, latches accepted steered audio on `ReplyOperation`, and uses it in final inbound-auto TTS decisions.

## Best Possible Solution

Land the operation-local marker approach after maintainers accept the delivery-modality behavior and CI is green, preferring this over persisted session state unless reviewers decide durable metadata is explicitly required.

## Reproduction Assessment

Yes. Source inspection on current main shows a steered audio inbound has `currentInboundAudio`, the active steer queue call does not carry it, and final TTS still uses the original text context's `inboundAudio`; no live WhatsApp timing repro was run here.

## Solution Assessment

Yes. The operation-local marker is the narrowest maintainable fix I found because it avoids persisted session state while latching the audio fact only after Codex accepts active steering; a wrapper-only TTS gate would not fix the missing metadata handoff.
