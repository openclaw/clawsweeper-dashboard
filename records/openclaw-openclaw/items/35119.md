## Summary

Keep open. Current main fixes the original numeric HTTP 500 and JSON Codex `server_error` paths, but the later concrete OpenAI wording beginning `The server had an error processing your request...` is still absent from the classifier and would not create the assistant failover signal needed to reach the configured fallback model.

## What This Changes

The issue asks OpenClaw to treat provider HTTP 500 / Codex server-error responses as failover-eligible so configured model fallbacks are attempted automatically.

## Best Possible Solution

Add a focused regression for an OpenAI Codex assistant error with `stopReason: "error"` and raw text beginning `The server had an error processing your request...`, then add a narrow server-error classifier pattern so the existing assistant failover handoff reaches the configured fallback model.

## Reproduction Assessment

Yes for a deterministic source-level reproduction: the reported raw OpenAI sentence is absent from the server-error patterns, and the assistant path only falls back when the classifier returns a failover signal or the attempt times out. A live upstream 500 replay was not performed.

## Solution Assessment

Yes, a narrow classifier/test update is the best path. The numeric HTTP status and JSON payload fixes should remain as-is; broad fallback-policy changes are not needed for this concrete wording gap.
