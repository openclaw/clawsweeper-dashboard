## Summary

Keep open. Current main still lacks a universal pre-filter raw inbound plugin hook, and the existing WhatsApp `message_received` opt-in remains downstream of access control, route resolution, and group/mention gating. The related WhatsApp-only PR was closed unmerged, so there is no canonical merged implementation to close against.

## What This Changes

Request an observe-only plugin hook that fires for inbound channel messages before allowlist, sender policy, and mention filtering.

## Best Possible Solution

Keep this open for maintainer API design. A maintainable path would add an explicit opt-in, observe-only, privacy-documented SDK/channel contract for pre-filter inbound observation, wire participating channels at the correct point before access and activation filters, and add contract plus channel-ordering tests. If maintainers decide ambient raw observation is out of scope, document that policy and point users to the current downstream `message_received` alternatives.

## Reproduction Assessment

Not applicable as a feature request; there is a high-confidence current-main verification path. Source inspection and targeted searches show no registered raw/pre-filter hook, and WhatsApp still drops or skips messages before downstream `message_received` emission.

## Solution Assessment

Unclear whether the exact proposed `on_raw_inbound` name is the best API. The safer solution is a deliberately named, opt-in, privacy-aware SDK/channel contract rather than a silent global observer or a WhatsApp-only private hook.
