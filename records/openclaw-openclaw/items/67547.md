## Summary

Keep open. Current main has refreshed the tracked hash and committed bundle, but the generator still resolves A2UI dependencies from `ui/node_modules` before root `node_modules` and still lacks the `signal-polyfill` singleton aliases described by the related implementation PR, so the underlying install-layout-sensitive generation path is not fully fixed.

## What This Changes

The issue reports that A2UI generated artifacts can change depending on whether `ui/node_modules` exists during build.

## Best Possible Solution

Fix the generator/resolver so A2UI dependency resolution is byte-stable across install layouts, regenerate the artifacts from that fixed path, and add regression coverage for both `ui/node_modules` and root `node_modules` resolution.

## Reproduction Assessment

Yes. The issue body and Apr 19 follow-up give concrete `pnpm build` dirty-tree reproductions, and current source still contains the layout-sensitive resolver; I avoided running the mutating build step in this read-only review.

## Solution Assessment

No. Current main has refreshed the checked-in hash and bundle, but that is not the narrow durable fix for install-layout-dependent generation; the resolver or singleton alias set still needs to be repaired.
