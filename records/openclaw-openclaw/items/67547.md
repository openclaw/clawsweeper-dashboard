## Summary

Keep open. Current main has removed the tracked A2UI hash/bundle dirt that made the original symptom noisy, but the current Canvas plugin Rolldown config still prefers `ui/node_modules` and still lacks `signal-polyfill` singleton aliases, so the install-layout-dependent bundle output path remains unresolved; https://github.com/openclaw/openclaw/pull/68934 is the related implementation candidate but it targets the old paths.

## What This Changes

The issue asks for A2UI build artifacts to remain reproducible regardless of whether dependencies resolve from root `node_modules` or `ui/node_modules`.

## Best Possible Solution

Apply the singleton/deterministic resolver fix to the current Canvas plugin A2UI Rolldown config, keep generated outputs ignored, and verify the bundle output with and without `ui/node_modules`.

## Reproduction Assessment

Yes. The current source still prefers `ui/node_modules` before root dependencies and lacks `signal-polyfill` aliases, while the dependency contract shows both signal consumers use `signal-polyfill`; I did not command-reproduce because the review must remain read-only.

No. Current main is only a partial solution: the tracked hash/bundle churn is fixed, but the remaining maintainable fix is to update the current Canvas plugin resolver or land a rebased successor to https://github.com/openclaw/openclaw/pull/68934.
