## Summary

Keep open. Current main still has the A2UI generator problem: the committed hash is stale against tracked inputs, the build path still invokes A2UI bundling, and the Rolldown resolver remains install-layout-sensitive without `signal-polyfill` singleton aliases; the related implementation PR is still the right place to review or repair the fix before closing this issue.

## What This Changes

The issue requests byte-stable A2UI bundle and hash generation for the same checkout and lockfile regardless of whether dependencies are present under `ui/node_modules` or root `node_modules`.

## Best Possible Solution

Fix the generator rather than only refreshing checked-in artifacts: canonicalize A2UI dependency resolution across install layouts or add the complete singleton alias set including `signal-polyfill` and `signal-polyfill/`, regenerate the artifacts from current main, and add regression coverage that proves stable generated hash and bundle bytes with and without `ui/node_modules`.

## Reproduction Assessment

Yes. The issue body and Apr 19 comment provide concrete `pnpm build` reproductions, and current main still has a stale tracked hash plus the resolver/alias shape that explains install-layout-dependent output; I avoided the mutating build step to keep the checkout clean.

## Solution Assessment

No, current main is not the best solution yet. The hash-input narrowing is only a partial mitigation; the maintainable fix is a generator-level dependency-resolution or singleton-alias repair with layout-focused regression coverage.
