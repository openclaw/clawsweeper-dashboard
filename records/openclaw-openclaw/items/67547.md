## Summary

Keep open. Current `main` partially mitigates the hash-input side of the report, but the A2UI generator still has an install-layout-sensitive dependency resolver, still lacks a `signal-polyfill` singleton alias, and has no regression coverage proving stable generated bytes across `ui/node_modules` and root `node_modules`; the related #68934 remains open rather than merged evidence.

## What This Changes

The issue requests byte-stable A2UI prebuilt bundle and hash generation for the same checkout and lockfile regardless of whether dependencies resolve from `ui/node_modules` or root `node_modules`.

## Best Possible Solution

Fix the generator rather than only refreshing checked-in artifacts: canonicalize A2UI dependency resolution across install layouts or add the complete singleton alias set including `signal-polyfill` and `signal-polyfill/`, then add regression coverage that proves the generated bundle and hash stay stable with and without `ui/node_modules`. Review and land #68934 or a successor if it covers that full guarantee.

## Reproduction Assessment

Yes. The issue body and Apr 19 comment provide concrete `pnpm build` reproductions, and current `main` still has the resolver/alias shape that explains the install-layout-dependent output; I avoided a live mutating build to keep the checkout clean.

## Solution Assessment

No. Current `main` is only a partial mitigation because the hash inputs are cleaner but the actual bundler resolution can still vary; the narrow maintainable solution is a generator-level dependency-resolution fix plus layout regression coverage.
