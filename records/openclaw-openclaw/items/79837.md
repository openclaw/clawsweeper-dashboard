## Summary

Keep open: current main still has an authorized_user-only Google Vertex ADC runtime path, so service_account JSON credentials remain rejected despite the provider advertising gcloud ADC auth.

## What This Changes

Add Google Vertex support for service_account Application Default Credentials alongside the existing authorized_user ADC flow.

## Best Possible Solution

Land a Google-plugin-owned service_account ADC implementation, likely by fixing or replacing the existing Google Vertex PR, while preserving the authorized_user path and manifest auth-evidence contract.

## Reproduction Assessment

Yes, from source inspection: a service_account ADC file can satisfy the advertised local-file ADC evidence, but the runtime preflight and credential reader require `type === "authorized_user"`. I did not run a live GCP service-account call in this read-only review.

## Solution Assessment

Yes, supporting service_account in the Google plugin's Vertex ADC resolver is the narrow owner-boundary fix. The existing open PR points in that direction, but it needs its review and proof blockers resolved before it can be the final solution.
