## Summary

Keep open: current main still has an `authorized_user`-only Google Vertex ADC runtime path, so `service_account` ADC files remain rejected even though setup can detect the file as gcloud ADC evidence.

## What This Changes

Accept `service_account` ADC JSON keys for the `google-vertex` provider in addition to the existing `authorized_user` ADC flow.

## Best Possible Solution

Add service-account token minting inside the Google plugin's Vertex ADC resolver, preserve the authorized_user path, keep token handling guarded and credential-scoped, and cover both credential types with focused tests plus docs/changelog if behavior changes are user-visible.

## Reproduction Assessment

Yes, from source inspection: a readable `service_account` ADC file can be accepted as gcloud ADC evidence, but the runtime preflight and credential reader require `type === "authorized_user"`. I did not run a live GCP service-account call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Google plugin's Vertex ADC/auth path, using an official Google token-minting path for `service_account` credentials while leaving core auth contracts and existing `authorized_user` behavior intact.
