Close as duplicate/superseded: current main still rejects service_account ADC for Google Vertex, but the same remaining Vertex ADC work is already tracked by https://github.com/openclaw/openclaw/issues/60736 and the open implementation at https://github.com/openclaw/openclaw/pull/60860 already includes service_account handling.

## What This Changes

Request support for service_account ADC JSON credentials in the bundled Google Vertex provider, alongside the existing authorized_user ADC flow.

## Best Possible Solution

Keep the service_account acceptance criterion on the canonical Google Vertex PR or its successor, then close the broader Vertex tracking issue after the merged implementation covers both ADC credential types with tests.

## Reproduction Assessment

Yes. Source inspection shows a service_account ADC file can match the setup evidence path, but the current runtime reader and native transport gate both require type === "authorized_user", so service_account falls through before Vertex bearer auth runs.

## Solution Assessment

Yes for cleanup. Consolidating this into https://github.com/openclaw/openclaw/pull/60860 is narrower for maintainers than tracking a parallel issue, because that open PR already changes the same Google Vertex ADC files and includes service_account coverage.
