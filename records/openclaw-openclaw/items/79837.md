## Summary

Keep open: current main still gates the Google Vertex ADC runtime path on `authorized_user` JSON, so service-account ADC remains unsupported even though the provider setup surface accepts ADC files via `GOOGLE_APPLICATION_CREDENTIALS`. An open Google Vertex PR includes service-account support, but it is unmerged and already has review/proof blockers.

## What This Changes

The issue requests Google Vertex ADC support for `service_account` JSON key files in addition to the existing `authorized_user` refresh-token path.

## Best Possible Solution

Land or supersede a Google-plugin-owned fix that accepts both `authorized_user` and `service_account` ADC, mints bearer tokens through a guarded/injected auth path, declares any direct runtime dependency, and adds focused tests/docs without adding core provider special cases.

## Reproduction Assessment

Yes, from source inspection: with `GOOGLE_APPLICATION_CREDENTIALS` pointing at a `service_account` JSON plus project/location env, setup evidence can mark ADC present, but the Google Vertex runtime gate and credential reader require `type === "authorized_user"`. I did not run a live GCP service-account call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/google/vertex-adc.ts` and adjacent Google plugin tests, preserving the existing `authorized_user` path while adding service-account token minting through the official Google auth flow and provider-local dependency ownership.
