## Summary

Keep open: current main still accepts only `authorized_user` ADC for `google-vertex`, while the plugin manifest and docs expose Vertex as ADC-backed. https://github.com/openclaw/openclaw/issues/71853 fixed the earlier marker/authorized-user path, and https://github.com/openclaw/openclaw/issues/60736 is broader Vertex work, but neither resolves this narrower service-account gap.

## What This Changes

The issue requests that `google-vertex` honor `service_account` ADC JSON files from `GOOGLE_APPLICATION_CREDENTIALS` or the standard gcloud ADC path instead of accepting only `authorized_user` ADC.

## Best Possible Solution

Extend the Google plugin's Vertex ADC resolver to support service-account credentials in the existing ADC path, preserve authorized-user behavior, keep the fix plugin-local, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. From source inspection, a `service_account` ADC file at `GOOGLE_APPLICATION_CREDENTIALS` makes `hasGoogleVertexAuthorizedUserAdcSync()` return false, and the forced marker path later throws because header resolution requires an authorized-user credentials file.

## Solution Assessment

Yes. The narrow maintainable fix is to add a service-account token branch in `extensions/google/vertex-adc.ts`, wire it through the existing Vertex header/stream selection path, and cover both credential types without moving provider-specific auth into core.
