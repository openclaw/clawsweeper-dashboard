## Summary

Keep open: current main still treats Google Vertex ADC as an authorized_user-only runtime path, while the plugin manifest and docs expose google-vertex as gcloud ADC-backed auth evidence. A service_account JSON file is therefore a credible current-main bug, not a stale or implemented report.

## What This Changes

The issue requests service_account Application Default Credentials support for the google-vertex provider in addition to the existing authorized_user ADC path.

## Best Possible Solution

Add provider-local service_account token minting to the Google Vertex ADC resolver, keep authorized_user behavior unchanged, declare any direct runtime dependency, and cover both credential types with focused tests plus docs/changelog updates.

## Reproduction Assessment

Yes, with high-confidence source inspection: a service_account ADC file can satisfy the manifest authEvidence marker, but the runtime stream gate and header resolver both require `type === "authorized_user"`. I did not run a live GCP call in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix belongs in `extensions/google/vertex-adc.ts` and adjacent Google Vertex transport tests, or as a repaired subset of the broader open Vertex PR. Core auth contracts should stay generic.
