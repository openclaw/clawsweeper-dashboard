## Summary

Keep open: current main still has an authorized_user-only Google Vertex ADC runtime path, so service_account JSON credentials remain rejected despite the provider advertising gcloud ADC auth.

## What This Changes

The issue asks the Google Vertex provider to accept service_account ADC JSON credentials in addition to authorized_user ADC for Bearer-token Vertex requests.

## Best Possible Solution

Add service_account token minting inside the google plugin's Vertex ADC resolver, keep authorized_user behavior unchanged, and cover both credential types with focused tests and docs/changelog updates.

Yes, from source inspection: a service_account ADC file can be detected as gcloud ADC evidence, but the runtime preflight and credential reader require `type === "authorized_user"`. I did not run a live GCP service-account call in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix belongs in `extensions/google/vertex-adc.ts` and adjacent Google Vertex tests, using an official Google auth path to mint Bearer tokens for service_account credentials while leaving core auth contracts generic.
