## What This Changes

The issue reports that built-in slash commands can incorrectly consume longer skill command names that share a prefix, such as `/config-check` being handled as `/config`.

## Best Possible Solution

Land one comprehensive boundary-helper fix that covers the shared parser and direct built-in slash handlers, with regression tests for longer command names and valid whitespace, colon, and end-of-command forms.

## Reproduction Assessment

No executed live reproduction was run under the read-only review constraint. Source inspection gives a high-confidence current-main path: `/config-check` satisfies the `/config` prefix parser before the config command gate runs.

## Solution Assessment

Yes, the best fix is a narrow reusable slash-command boundary helper applied to the shared parser and direct built-in handlers. Because two open PRs already claim the issue, the next step is PR review and selection, not a separate repair job.
