## Summary

Keep open: current main still has source-backed prefix matching that can consume `/config-check` as `/config`, and adjacent direct `startsWith` handlers suggest the repair should audit built-in slash boundaries rather than only one parser.

## What This Changes

The issue reports that slash command parsing treats longer slash names with built-in prefixes as the built-in command instead of leaving them available for skill commands.

## Best Possible Solution

Add a reusable slash command boundary helper, apply it to the shared parser and direct built-in handlers such as `/models` and `/allowlist`, and cover `/config-check` skill fallthrough with a focused regression test.

## Reproduction Assessment

No executed high-confidence reproduction was run. Source inspection shows `/config-check` currently satisfies the `/config` prefix parser and that prompt-style skill rewrites still leave the original command body for built-in handlers.

## Solution Assessment

Yes, the best fix is a narrow command-token boundary check reused by built-in slash handlers, with regression coverage for both existing valid forms and longer command names.
