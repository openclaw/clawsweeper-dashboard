## What This Changes

The issue reports built-in slash command parsing consuming longer skill command names that share a prefix with built-in commands such as `/config-check` and `/config`.

## Best Possible Solution

Land one focused PR that adds a reusable slash-command boundary helper, applies it to shared and direct built-in handlers, and covers longer command names with regression tests.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main parses `/config-check` through the `/config` shared parser because it only checks `startsWith` before slicing the command prefix. This review did not run a live channel roundtrip because the checkout was read-only.

## Solution Assessment

Yes, the best fix is a narrow command-token boundary check reused by the shared parser and direct built-in handlers, with focused regression coverage for valid delimiters and longer command names. The existing linked PRs are the right place to choose and land that repair.
