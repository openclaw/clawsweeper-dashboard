## Summary

Keep open: current main still has source-backed prefix matching that can consume longer skill slash commands as built-in commands, and the linked fix PR remains open rather than merged.

## What This Changes

The issue asks for slash command parsing to require a real command-token boundary so longer names like `/config-check` are not captured by `/config`.

## Best Possible Solution

Use one slash-token boundary helper across the shared parser and direct built-in slash handlers, add regressions for valid delimiters and longer command names, then close this issue when the linked PR lands.

## Reproduction Assessment

Yes for source-level reproduction, but no executed live channel run was performed. Current main still accepts `/config-check` as a `/config` prefix match before the config gate can return the reported disabled-command response.

## Solution Assessment

Yes, the maintainable fix is a narrow command-token boundary helper reused by all built-in slash handlers that parse by prefix. The open linked PR is the right direction, but this issue should remain open until that path lands or is narrowed deliberately.
