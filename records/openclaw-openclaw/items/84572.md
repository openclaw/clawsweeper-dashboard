## What This Changes

Require built-in slash command parsers and handlers to match exact command tokens so longer skill commands like `/config-check` are not consumed by `/config`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84584 or an equivalent fix that applies one slash-token boundary helper across the shared parser and direct built-in handlers, then close this issue after merge.

## Reproduction Assessment

Yes for a source-level reproduction path: current main accepts `/config-check` via the `/config` parser because it only checks `startsWith("/config")`, and adjacent direct handlers show the same boundary shape. I did not execute a live channel repro because this review is read-only.

## Solution Assessment

Yes, the best fix is a small shared slash-token boundary helper reused by both the shared parser and direct built-in handlers, with focused regression tests for longer command names and valid whitespace or colon forms.
