## Summary

Keep open for maintainer review. The PR targets the right restart-continuation bug, but the new guard clears the pending session owner before active preparation finishes, leaving an in-flight window where another session can still replace the hooks.

## What This Changes

The PR threads a session key through gateway restart scheduling, adds `emitHooksQueued` and `continuationQueued` reporting, guards coalesced restart hooks by session, and adds targeted restart and gateway-tool tests.

## Best Possible Solution

Keep the scheduler-level guard, but retain the active restart owner through preparation and emission, then add a regression test where a different session coalesces while the first `beforeEmit` promise is still pending.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: schedule session A with a `beforeEmit` promise that stays pending, advance to preparation, then schedule session B and observe that the cleared owner lets B's hooks queue.

## Solution Assessment

No. The session guard is the right direction, but it must preserve the owner for the full preparation/emission cycle instead of clearing it before `beforeEmit` completes.
