## Summary

Keep open: current main and the latest release still expose Feishu `resolveSenderNames` plus contact lookup only, with no native open_id-to-display-name alias map. Related Feishu sender-name PRs overlap attribution behavior, but they do not supersede this requested config contract or settle alias precedence.

## What This Changes

Request a Feishu sender-name alias-map config option that can override or fall back from API-resolved display names for specific user IDs.

## Best Possible Solution

Define a Feishu plugin-owned, account-scoped alias-map config contract, apply it in the sender display-name path with documented precedence, and cover API success, API failure, DM, group, and opt-out behavior in docs and tests.

## Reproduction Assessment

## Solution Assessment

Yes directionally: a Feishu-owned config option is the narrow maintainable boundary, but maintainers should decide alias precedence and coordinate it with adjacent sender-name work before implementation.
