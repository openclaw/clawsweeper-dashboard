## Summary

Keep open: current main still has no Feishu sender alias-map config, and the adjacent Feishu sender-name work does not supersede this custom mapping request.

## What This Changes

Add a Feishu sender-name alias mapping option so operators can map specific sender open_ids to custom display names or fallbacks.

## Best Possible Solution

Add a Feishu plugin-owned alias-map config contract, merge it per account, apply it only to display attribution while preserving ID-based authorization, and document/test DM, group, API-success, API-failure, and opt-out behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug, but yes at source level: the strict Feishu config/docs/runtime path lacks any alias-map key or resolver input, so a configured `senderNameMap` has nowhere to be accepted or applied.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: a Feishu-owned config option is the narrow maintainable boundary, but maintainers should decide alias precedence and sequence it with adjacent sender-name work before implementation.
