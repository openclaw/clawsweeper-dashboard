## Summary

Keep open: current main and the latest release still have no Feishu sender alias-map config or runtime lookup path. This fresh review matches the prior ClawSweeper review, and the adjacent Feishu sender-name PRs do not supersede the custom mapping request.

## What This Changes

The request is to add a Feishu sender-name alias map configuration so specific open_ids can display custom names when API resolution is unavailable or undesired.

## Best Possible Solution

Define a Feishu plugin-owned alias-map config contract, apply it with account-scoped precedence only to display attribution, and document/test API success, API failure, DM, group, and opt-out behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug, but yes at source level: a senderNameMap-style key is absent from the strict config/docs/manifest surfaces, and the resolver has no alias input or lookup branch.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: a Feishu-owned config option is the narrow maintainable boundary, but maintainers should decide alias precedence and coordinate it with adjacent sender-name work before implementation.
