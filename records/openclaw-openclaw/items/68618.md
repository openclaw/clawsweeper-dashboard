Keep open: current main and the latest release still expose only API-backed Feishu sender-name resolution plus an opt-out, with no native alias-map config. The request remains useful, but it needs maintainer product/API judgment because it adds a new Feishu config surface and overlaps open Feishu permission/display-name work.

## What This Changes

Requested native Feishu sender open_id alias mapping so users can customize or fallback display names without editing plugin code.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract: prefer verified low-scope platform data, optionally add a per-account alias map if maintainers want explicit overrides, document precedence, and cover DMs and groups with focused tests.

## Reproduction Assessment

Yes, at source level: current main has a strict Feishu schema with `resolveSenderNames` but no alias-map field, and the sender-name resolver only uses the Feishu contact API/cache path. I did not run a live Feishu tenant because the missing configuration capability is evident from source, docs, tests, and release source.

## Solution Assessment

Unclear: a native map is plausible, but the best permanent behavior needs maintainer choice on whether aliases override API names, only fill failures, or coexist with the broader contact-free Feishu sender-name work. The implementation should stay inside the Feishu plugin rather than core.
