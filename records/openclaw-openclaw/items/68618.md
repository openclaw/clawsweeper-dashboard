## Summary

Keep open: current main and the latest release still have only Feishu API-backed sender-name resolution plus an opt-out, with no native alias-map config; the request is useful but needs maintainer product judgment because it adds a new Feishu config surface and overlaps ongoing permission-reduction work.

## What This Changes

Requests a Feishu channel configuration option that maps sender open_ids to custom display-name aliases for fallback or override behavior.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract: prefer verified low-scope platform data, optionally add a per-account alias map if maintainers want explicit overrides, document precedence, and cover DMs and groups with focused tests.

## Reproduction Assessment

Yes, at source level: current Feishu config is strict and has `resolveSenderNames` but no alias-map field, while the resolver only uses Feishu API lookup plus fallback. I did not run a live Feishu tenant because the missing capability is evident from source and docs.

## Solution Assessment

Unclear: a native map is plausible, but the best permanent fix needs maintainer choice on precedence and overlap with the broader contact-free sender-name work. A narrow implementation should stay inside the Feishu plugin rather than core.
