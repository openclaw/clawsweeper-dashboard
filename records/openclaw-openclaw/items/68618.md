## Summary

Keep open: current main and v2026.6.10 still only support Feishu `resolveSenderNames` as an API-backed boolean lookup, with no native open_id-to-display-name alias map. The request is valid, but it adds Feishu plugin configuration and fallback/display policy surface that overlaps active Feishu permission and display-name work, so it needs maintainer direction rather than cleanup closure.

## What This Changes

Request adding a Feishu sender-name alias map so specific open_id values can display configured names when API-backed sender resolution is unavailable or undesired.

## Best Possible Solution

Choose a maintainer-approved Feishu plugin contract first: decide whether the contact-permission/DM-fallback work should solve the default behavior, then add a documented strict-schema alias map with focused resolver, docs, and tests only if custom overrides remain desired.

## Reproduction Assessment

Yes, at source level: the strict Feishu schema and docs expose only `resolveSenderNames`, and the runtime resolver only uses API-backed sender lookup with open_id fallback. I did not run a live Feishu repro because this read-only review has no channel credentials and the missing config path is clear from source.

## Solution Assessment

Unclear: a native alias map is plausible, but it is not obviously the best fix until maintainers choose how it should interact with the broader Feishu contact-permission and DM display-name fallback work. A safer path is to settle that contract first, then implement one canonical precedence rule inside the Feishu plugin.
