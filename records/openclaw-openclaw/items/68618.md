## Summary

Keep open: current main and the latest release still only support Feishu sender-name API lookup plus an opt-out flag, not a native open_id-to-display-name alias map. The request is valid, but it adds Feishu plugin config and sender-attribution policy surface, and related contact-permission/display-name work is still open rather than merged.

## What This Changes

Add a Feishu channel configuration option mapping sender open_ids to custom display names for fallback or override behavior.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract covering alias precedence, account scoping, contact-free fallback behavior, docs, and regression tests, coordinated with the open contact-permission work.

## Reproduction Assessment

Yes, source-level. Current main has no senderNameMap schema/config path and the sender-name resolver only uses Feishu API results or falls back empty, so the requested alias behavior is absent without needing a live tenant.

## Solution Assessment

Unclear. A native Feishu plugin option may be the right direction, but maintainers should first decide alias precedence, per-account scope, and interaction with contact-free sender lookup work.
