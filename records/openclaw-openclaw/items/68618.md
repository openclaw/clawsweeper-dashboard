## Summary

Keep open: current main still lacks a Feishu sender alias map. The Feishu config schema, docs, and inbound sender resolver only expose `resolveSenderNames` and API-backed lookup, while related PR #73399 is adjacent sender-name work but does not supersede this requested config surface.

## What This Changes

Request a Feishu channel configuration option that maps sender open_ids to custom display aliases for attribution when API name lookup is unavailable or undesired.

## Best Possible Solution

Implement this in the Feishu plugin: add a documented top-level and per-account alias map to the strict config contract, merge it per account, apply aliases only to display attribution, keep authorization ID-based, and cover API-success, API-failure, DM, group, and `resolveSenderNames: false` behavior in focused Feishu tests.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `senderNameMap` is absent from the strict Feishu schemas, local search finds no alias implementation, and the inbound path falls back to the raw open_id when API lookup returns no name.

## Solution Assessment

Yes, a Feishu-owned config option is the narrow maintainable direction. The unresolved part is precedence and sequencing with #73399, not whether the request belongs in core-wide code.
