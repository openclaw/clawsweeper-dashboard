## Summary

Keep open: current main and the latest release still only provide Feishu API-backed sender-name lookup plus the `resolveSenderNames` opt-out, with no native open_id-to-display-name alias map. The request is valid, but it adds Feishu plugin config and sender-attribution policy surface and overlaps open Feishu contact-permission/display-name work, so it needs maintainer direction rather than cleanup closure or an autonomous fix PR.

## What This Changes

Add a Feishu channel configuration option for mapping specific sender open_ids to custom display-name aliases.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract covering alias precedence, per-account scoping, contact-free fallback behavior, docs, and regression tests, coordinated with the open contact-permission work.

## Reproduction Assessment

Yes, source-level. Current main has no Feishu sender alias schema/config path, and the sender-name resolver only uses API results or returns empty on failure.

## Solution Assessment

Unclear. A native Feishu plugin option may be the right shape, but maintainers should first decide alias precedence, per-account scope, privacy posture, and interaction with the contact-free sender lookup work.
