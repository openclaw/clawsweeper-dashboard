## Summary

Keep open: current main and the latest release still support Feishu sender-name API lookup via `resolveSenderNames`, but there is no native open_id-to-display-name alias map. The related Feishu display-name PRs overlap the resolver area, but they change lookup/fallback behavior rather than superseding this requested config contract.

## What This Changes

Requests a Feishu plugin configuration option that maps sender open_ids to custom display names for fallback or override during sender-name resolution.

## Best Possible Solution

Define a Feishu plugin-owned, account-scoped sender alias-map config contract, apply it in the sender display-name path with documented precedence, and cover API success, API failure, DM, group, multi-account, and opt-out behavior in tests and docs.

## Reproduction Assessment

Yes at source level, though this is a feature request rather than broken current behavior: current strict Feishu config omits a `senderNameMap`-style key and the resolver has no alias input or lookup branch.

## Solution Assessment

Yes directionally: Feishu plugin config/runtime/docs are the right boundary, but maintainers should decide alias precedence and coordinate with the active Feishu sender-name PRs before implementation.
