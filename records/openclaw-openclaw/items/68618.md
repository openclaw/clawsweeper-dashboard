## Summary

Keep open: current main and v2026.6.10 still only support the Feishu `resolveSenderNames` opt-out plus API-backed sender-name lookup; there is no native open_id-to-display-name alias map. The request is valid, but it adds Feishu plugin configuration and sender-attribution policy surface that overlaps active Feishu contact-permission/display-name work, so it needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests a native Feishu sender-name alias map so configured open_ids can display custom names or provide fallback names when Feishu API lookup fails.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract covering alias precedence, per-account scoping, contact-free fallback behavior, docs, and regression tests, coordinated with the open contact-permission work.

## Reproduction Assessment

Yes, source-level. Current main has no Feishu sender alias schema/config path, and the sender-name resolver only uses API results or returns empty on failure.

## Solution Assessment

Unclear. A native Feishu plugin option may be the right shape, but maintainers should first decide alias precedence, per-account scope, privacy posture, and interaction with the contact-free sender lookup work.
