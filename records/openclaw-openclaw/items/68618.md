## Summary

Keep open: current main still lacks a Feishu sender alias map. The strict Feishu config, docs, and inbound sender resolver only expose API-backed `resolveSenderNames`, and open PR #73399 is adjacent sender-name work rather than a superseding alias-map implementation.

## What This Changes

Request a Feishu channel `senderNameMap` option that maps Feishu sender IDs to custom display aliases for fallback or override attribution.

## Best Possible Solution

Implement the alias map inside the Feishu plugin config contract, merge it per account, apply it only to display attribution while preserving ID-based authorization, and document/test DM, group, API-success, API-failure, and opt-out behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug, but source inspection is high-confidence: the requested config key is absent from the strict schema/docs, and the inbound sender-name path has no alias input.

## Solution Assessment

Is this the best way to solve the issue? Yes, a Feishu-owned config option is the narrow maintainable direction; maintainers still need to decide alias precedence and coordinate with #73399 before implementation.
