## Summary

Keep open: current main and v2026.6.8 still expose only Feishu API-backed sender-name resolution plus the existing opt-out, with no native sender alias-map config; the request remains valid but needs maintainer product/API judgment because it adds a Feishu config surface and overlaps active display-name permission work.

## What This Changes

Request a Feishu sender open_id-to-display-name alias mapping config for fallback or override of sender names.

## Best Possible Solution

Define a Feishu-plugin-owned sender display-name contract: prefer verified low-scope platform data, optionally add a per-account alias map if maintainers approve explicit overrides, document precedence, and cover DMs and groups with focused tests.

## Reproduction Assessment

Yes. Source inspection shows current Feishu config is strict and only has resolveSenderNames, while the resolver uses Feishu API lookup plus fallback with no native alias-map path.

## Solution Assessment

Unclear. A native alias map is plausible, but the best permanent fix needs maintainer choice on precedence, per-account scope, and overlap with contact-free sender-name work; the implementation should stay inside the Feishu plugin rather than core.
