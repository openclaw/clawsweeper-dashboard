## Summary

Keep open: current main still lacks a Feishu sender alias map. The Feishu plugin only implements API-backed `resolveSenderNames`, the runtime falls back to raw sender IDs when lookup fails, and the adjacent open Feishu display-name PR does not add the requested alias-map configuration surface.

## What This Changes

Request adding a Feishu channel configuration map from sender IDs to display aliases for API fallback or override in DMs and groups.

## Best Possible Solution

Implement this inside the Feishu plugin rather than core: add a documented, per-account-capable alias map to the Feishu config contract, apply it only to display attribution, preserve ID-based authorization behavior, and cover API-success, API-failure, DM, group, and opt-out semantics in focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `senderNameMap` is absent from the strict Feishu runtime schema, and the inbound sender-name path has no alias lookup before falling back to the raw sender ID.

## Solution Assessment

Yes, broadly. A Feishu-owned config option is the narrow maintainable boundary, but maintainers should decide alias precedence and coordinate it with the open Feishu sender-name fallback PR before implementation.
