## Summary

Keep open: current main still trims WebChat message text before optimistic rendering and `chat.send`, and the only directly related fix PRs are closed without merge. The bug is narrow, source-reproducible, and suitable for a focused repair PR.

## What This Changes

The issue reports that WebChat removes leading whitespace from pasted code blocks before sending, breaking ASCII and box-drawing alignment.

## Best Possible Solution

Preserve the original nonblank WebChat message text through host queueing, optimistic display, and `chat.send`, while continuing to use trimmed text only for blank-message guards and local command detection.

## Reproduction Assessment

Yes. Current source shows `handleSendChat()` and `sendChatMessage()` trimming the message before the user-visible optimistic message and `chat.send` payload; a multiline diagram with leading spaces would lose indentation on current main.

## Solution Assessment

Yes. The narrow maintainable fix is the shape shown by https://github.com/openclaw/openclaw/pull/81344: keep exact text for ordinary sends, but keep trim-based blank and slash/stop/reset/BTW command checks.
