## Summary

Keep open: current main still trims WebChat message text before optimistic display and `chat.send`, and the linked fix attempts are closed unmerged rather than shipped.

## What This Changes

The issue reports that WebChat strips leading whitespace from submitted code blocks, breaking ASCII and box-drawing diagram alignment.

## Best Possible Solution

Patch the WebChat send path to keep trimmed text only for guards and command parsing while preserving the original submitted message for optimistic rendering, queued sends, and `chat.send`.

## Reproduction Assessment

Yes from source inspection: current `main` trims the draft in `handleSendChat()` and again in `sendChatMessage()`, which is a high-confidence reproduction path for losing leading spaces. I did not run the browser flow in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow separation between trimmed guard/command text and the original submitted text, with host and controller regression tests. A `trimEnd()`-only host change would be incomplete because the controller currently trims again.
