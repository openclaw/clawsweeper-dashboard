## Summary

Keep open: this PR targets a real session-store lock hot path, but it should not merge until the durability rationale, default-fsync test proof, and required changelog entry are fixed.

## What This Changes

Adds a default-on `durable` option to atomic JSON/text writes, opts the session-store registry write out of fsync, and adds focused `json-files` tests.

## Best Possible Solution

Land a narrow PR that preserves default durable atomic writes, opts only the session registry out, accurately documents the crash-loss tradeoff, strengthens regression tests, and records the user-visible fix in the changelog.

## Reproduction Assessment

Yes, from source inspection: current main awaits temp-file and parent-directory fsync inside the session-store writer path, and the linked reports show that path producing long lock holds under cron load. I did not run a live heavy-cron reproduction in this read-only review.

## Solution Assessment

Yes, with changes: a default-on `durable` option plus a session-registry-only opt-out is the narrowest maintainable code shape. The patch should reword the durability rationale, add direct default-fsync assertions, and include the changelog entry before merge.
