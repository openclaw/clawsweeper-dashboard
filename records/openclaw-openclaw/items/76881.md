## Summary

Keep this PR open. The patch targets a real source-reproducible session-store latency bug, but it still needs changes before merge: the durability rationale overclaims current transcript behavior, the default fsync test does not actually assert sync calls, and the required changelog entry is missing.

## What This Changes

The PR adds a default-on `durable` option to atomic JSON/text writes, opts only the session-store registry write out of fsync, and adds focused `json-files` tests.

## Best Possible Solution

Land a narrow version that preserves default durable atomic writes, opts only the session-store registry out, accurately documents the weaker crash-loss tradeoff, includes a changelog entry, and directly tests both durable modes.

## Reproduction Assessment

Yes. Current main awaits temp-file and parent-directory fsync in the session-store writer path, and the linked field reports show that path causing long lock holds under cron load; I did not run a live heavy-cron reproduction.

## Solution Assessment

Mostly yes. A default-on `durable` option with a session-registry-only opt-out is the narrow maintainable shape, but the patch should fix the durability wording, changelog, and default-mode test proof before merge.
