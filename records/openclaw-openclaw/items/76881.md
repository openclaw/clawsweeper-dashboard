## Summary

Keep open: current main and v2026.5.3 still fsync the temp file and parent directory inside the session-store write path, so the PR remains useful, but the diff overstates transcript durability/reconstruction, lacks the required changelog entry, and has a weak default-durability test.

## What This Changes

Adds an internal `durable` option to atomic JSON/text writes, opts the session-store registry write out of fsync, and adds focused `json-files` tests.

## Best Possible Solution

Land a narrow version that preserves durable atomic writes by default, opts only the session-store registry out, accurately documents the crash-loss tradeoff, directly tests both durability modes, and records the user-facing fix.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.5.3 await temp-file and parent-directory fsync inside the session-store writer path, and linked field reports show long lock holds under cron load. I did not run a live heavy-cron reproduction in this read-only review.

## Solution Assessment

Mostly yes: a default-on `durable` option with a session-registry-only opt-out is the narrowest maintainable fix shape. The patch should reword the recovery rationale, strengthen the default fsync test, and add the required changelog entry before merge.
