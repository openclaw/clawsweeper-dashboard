## Summary

This PR is useful and targets a real session-store latency path, but it should not merge yet because the safety rationale overstates current transcript durability and the change is missing required release-note/test proof.

## What This Changes

Adds a default-on `durable` option to atomic JSON/text writes, opts the session-store registry write out of fsync, and adds focused json-file tests.

## Best Possible Solution

Land a narrow version that keeps durable atomic writes as the default, opts only the session registry out, accurately documents the crash-loss tradeoff, adds the changelog entry, and directly tests both fsync modes.

## Reproduction Assessment

Yes, from source inspection: current main awaits temp-file and parent-directory fsync inside the session-store writer path, and the linked field reports show that path causing long lock holds under cron load. I did not run a live heavy-cron reproduction in this read-only review.

## Solution Assessment

Mostly yes: a default-on `durable` option with a session-registry-only opt-out is the narrowest maintainable shape. The patch still needs corrected durability wording, a changelog entry, and a stronger default-fsync assertion before merge.
