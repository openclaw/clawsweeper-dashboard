## Summary

Keep this PR open: it targets a valid current-main session-store hot path, but the patch still needs changes before merge because it overstates transcript durability, lacks the required changelog entry, and has a default-fsync test that does not actually count fsync calls.

## What This Changes

Adds a default-on `durable` option to atomic JSON/text writes, opts the session-store registry write out of fsync, and adds focused `json-files` tests.

## Best Possible Solution

Land a narrow revision that preserves default durable atomic writes, opts only the session-store registry out, accurately documents the crash-loss tradeoff, includes a changelog entry, and directly tests both durable modes.

## Reproduction Assessment

Yes, from source inspection: current main awaits temp-file and parent-directory fsync inside the session-store writer path, and the linked field reports show that path causing long lock holds under cron load. I did not run a live heavy-cron reproduction in this read-only review.

## Solution Assessment

Mostly yes: a default-on `durable` option with a session-registry-only opt-out is the narrowest code shape. The patch should not claim durable transcript recovery unless that is added, and it needs the required changelog entry before merge.
