## Summary

Keep open and do not merge as-is. The PR is the right narrow memory-core direction, but the new rate-limit classifier still misses code-style `rate_limit` and hyphenated `rate-limit` forms already recognized elsewhere, so those errors can still fall through to generic or missing remediation.

## What This Changes

The PR adds memory-core embedding-error classification and remediation hints to `memory_search` unavailable payloads and `openclaw memory status`, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep this PR as the canonical replacement for #55684, add the missing code-style and hyphenated rate-limit classifications with focused tests in both affected surfaces, then land it after normal CI and maintainer review.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original problem: non-quota provider/auth failures get generic `memory_search` action text and deep status prints only the raw embedding error. The remaining PR defect is reproducible by feeding `openai embeddings failed: rate_limit_exceeded` through the added classifier.

## Solution Assessment

No, not as-is. The approach is narrow and maintainable, but the classifier needs to cover the same rate-limit spellings already recognized by memory embedding policy or share that logic before this fully solves the advertised rate-limit remediation path.
