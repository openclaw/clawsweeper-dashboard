## Summary

Keep open. Current main still uses strict string inequality for both Gateway TLS fingerprint mismatch checks, while this PR narrowly replaces those comparisons with the existing secret-comparison helper. The code change looks functionally correct, but it touches security-sensitive Gateway TLS pinning and needs maintainer review/rebase plus normal landing hygiene before merge.

## What This Changes

The PR imports `safeEqualSecret` into `src/gateway/client.ts` and uses it for the WebSocket `checkServerIdentity` and post-open `validateTlsFingerprint` TLS fingerprint comparisons.

## Best Possible Solution

Land a maintainer-reviewed, current-main version of the Gateway hardening: both normalized TLS fingerprint comparisons should use `safeEqualSecret` or an equivalent constant-time helper, with a changelog entry and focused Gateway/helper validation before merge.

## Reproduction Assessment

Yes. A high-confidence source inspection path reproduces the issue on current main: `src/gateway/client.ts` still contains `fingerprint !== expected` at both TLS fingerprint mismatch checks, and existing Gateway coverage exercises the mismatch path.

## Solution Assessment
