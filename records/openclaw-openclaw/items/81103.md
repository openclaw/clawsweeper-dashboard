## Summary

Keep open: current main still leaves Feishu JSON `file_name` metadata on the inbound save path unnormalized, and the existing test suite explicitly pins that mojibake behavior. The prior merged fix only covered `Content-Disposition`; an open related PR may be the right implementation path, but it is not merged.

## What This Changes

Report that Feishu inbound media filenames from JSON `file_name` are saved as Latin-1 mojibake and should recover CJK names without rewriting valid Latin-1 names.

## Best Possible Solution

Land a focused Feishu inbound filename normalization fix, either by safely extracting the relevant part of https://github.com/openclaw/openclaw/pull/71517 or by applying the existing guarded recovery once before `saveMediaBuffer`, with regression coverage for JSON `file_name` mojibake and valid Latin-1 passthrough.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: current main returns JSON `file_name` values unchanged and the existing Feishu media test asserts the mojibake value for a CJK filename. I did not run a live Feishu receipt in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to normalize the JSON-derived filename at the Feishu inbound boundary before saving, reusing the guarded Latin-1-to-UTF-8 recovery and preserving the false-positive guard. The broader shared decoder PR may also solve it, but it should be landed or split without blocking this concrete bug unnecessarily.
