## Summary

Keep open: current main still lets Feishu JSON-derived `file_name` values bypass the existing CJK mojibake recovery, and the merged Content-Disposition fix intentionally left this separate path unchanged until there was concrete proof.

## What This Changes

Recover CJK mojibake in Feishu inbound JSON `file_name` metadata before saved filenames are written, distinct from the Content-Disposition header path.

## Best Possible Solution

Apply the conservative recovery heuristic at the Feishu inbound JSON/download-metadata boundary, add focused media and bot regression coverage, and leave broader shared filename-decoder migration to https://github.com/openclaw/openclaw/issues/48788 or a reviewed replacement for https://github.com/openclaw/openclaw/pull/71517.

## Reproduction Assessment

Yes. Current source shows JSON `file_name` bypassing recovery and an existing Feishu media test pins the mojibake result; this is a high-confidence source reproduction without running live Feishu.

## Solution Assessment

Yes for the narrow bug: recover only Feishu inbound filenames before storage and keep the Latin-1 false-positive guard covered by tests. A broader shared SDK/export refactor should not be required to fix this report.
