## Summary

Current main still preserves Feishu JSON-derived `file_name` metadata unchanged while the existing mojibake recovery is limited to `Content-Disposition`, so this is a concrete, current Feishu bug rather than a cleanup close.

## What This Changes

The issue requests recovering CJK filenames delivered through Feishu inbound JSON `file_name` metadata, not only filenames decoded from `Content-Disposition` headers.

## Best Possible Solution

Share or rename the conservative filename recovery helper inside the Feishu plugin, apply it before JSON-derived names reach `saveMediaBuffer`, and keep the Latin-1 false-positive guard covered by tests.

## Reproduction Assessment

Yes by source inspection: current main has a direct JSON `file_name` path that bypasses the recovery helper, and the existing Feishu media test pins the mojibake result. No live or test execution was performed in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Feishu-plugin change that reuses the existing conservative CJK recovery heuristic and updates the focused tests, rather than changing core media storage semantics.
