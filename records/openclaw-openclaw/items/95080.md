## Summary

Keep open: the PR targets a real yielded-parent deadlock, but as submitted it broadens subagent terminal task delivery to every blocked successful subagent completion instead of only the failed-delivery give-up recovery path, leaving a concrete duplicate or reordered delivery risk.

## What This Changes

The PR requeues failed subagent delivery as pending when a blocked terminal outcome is recorded and allows blocked subagent terminal task updates to auto-deliver, with focused policy and registry tests.

## Best Possible Solution

Keep the failed-to-blocked requeue, but constrain terminal task auto-delivery to announce give-up recovery and add sibling coverage proving ordinary blocked required completions stay on the existing announce path.

## Reproduction Assessment

Yes. The parent wake-up bug is source-reproducible: current main records failed or blocked subagent delivery but suppresses non-cancelled subagent terminal task updates, so no task fallback wakes the requester; I did not run the live Signal/K8s timing scenario.

## Solution Assessment

No as submitted. The failed-to-blocked requeue is the right repair shape, but the terminal delivery policy must be scoped to that recovery path rather than all blocked subagent successes.
