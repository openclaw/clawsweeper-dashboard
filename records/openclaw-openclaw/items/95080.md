## Summary

Keep open: the bug is valid and the PR targets the right parent wake-up path, but the patch broadens subagent blocked terminal auto-delivery beyond the failed-delivery give-up case and can inject extra parent-visible task events while the normal announce cleanup path still runs.

## What This Changes

The PR requeues failed subagent completion delivery as pending when the run becomes blocked, allows blocked subagent terminal task updates to auto-deliver, and adds focused policy and registry regression tests.

## Best Possible Solution

Keep the failed-to-blocked requeue, but narrow the auto-delivery exception to announce give-up recovery and add sibling coverage proving ordinary progress-only or missing required completions stay on the existing announce path without duplicate task events.

## Reproduction Assessment

Yes. The parent wake-up bug is source-reproducible: current main marks give-up delivery failed/blocked but suppresses non-cancelled subagent terminal task updates, so no task fallback wakes the requester; I did not run the live Signal/K8s timing scenario.

## Solution Assessment

No as submitted. The failed-to-blocked requeue is the right repair shape, but the terminal delivery policy should be scoped to that recovery path instead of all blocked subagent successes.
