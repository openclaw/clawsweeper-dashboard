## Label Justifications

- `P3`: This is a real but non-emergency mention-policy enhancement that needs product/API design before implementation.
- `impact:message-loss`: Implicit reply or bound-session activation can route visible channel replies without an explicit mention, which is a message delivery and routing problem.

## Summary

Keep open. Current main and the latest shipped tag still do not provide a shared strict mention policy; Slack has a scoped `thread.requireExplicitMention` escape hatch, while Discord and other reply-aware channels still treat implicit reply/bound-session facts as activation unless each plugin projects an explicit allowlist.

## What This Changes

Add an opt-in strict mention mode or equivalent config that prevents implicit reply-to-bot or bound-thread participation from satisfying `requireMention` across Slack, Discord, and other reply-aware channels.

## Best Possible Solution

Add a backward-compatible strict implicit-mention policy to the shared activation contract, wire it through affected channel configs and docs, and keep existing implicit behavior as the default unless maintainers approve a migration.

## Reproduction Assessment

Yes, source-level. Current shared gating and channel implementations show implicit mention kinds and bound-thread bypasses still satisfy or suppress `requireMention` without a shared strict mode, although no live transport repro was run.

## Solution Assessment

Unclear until maintainers choose the public API. A strict mode is directionally sound, but it should be additive and coordinated across shared SDK contracts, Slack's existing option, Discord bound-session behavior, and docs.
