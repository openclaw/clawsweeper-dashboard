- `impact:message-loss`: Implicit thread or bound-session activation can cause agents to handle and reply to channel messages that operators expected to require an explicit mention.

## Summary

Keep open. Current main and the latest release still preserve implicit mention activation by default, with only a Slack-scoped `thread.requireExplicitMention` escape hatch and no shared cross-channel strict mention mode; the remaining work is a public config/API decision rather than conservative cleanup.

## What This Changes

The issue requests a strict mention mode that prevents implicit reply-to-bot, thread-participation, or bound-session activation from satisfying `requireMention` across group-capable channels.

## Best Possible Solution

Add a backward-compatible shared strict implicit-mention policy through the SDK/channel activation contract, expose it through the accepted channel config surfaces, keep current implicit behavior as the default unless maintainers choose a migration, and coordinate docs/tests across affected plugins.

## Reproduction Assessment

Yes, source-level. Current shared mention gating treats implicit mention kinds as effective mentions, Slack only exposes a scoped thread option, and Discord can bypass configured mention gating for bound thread sessions; no live transport run was performed.

## Solution Assessment

Unclear until maintainers choose the public API. A strict mode is directionally sound, but the maintainable fix should be additive and coordinated across shared SDK contracts, Slack's existing option, Discord bound-session behavior, and docs.
