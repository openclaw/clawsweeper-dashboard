## Summary

Keep open. Current main has a shared implicit-mention allowlist and a Slack-only `thread.requireExplicitMention` control, but it still lacks the requested general strict mention mode and Discord bound-thread routing still disables `requireMention` before the shared gate can skip unmentioned follow-ups.

## What This Changes

Request an additive strict mention policy for group channel configs so explicit mentions and configured mention patterns pass while implicit reply/thread participation does not.

## Best Possible Solution

Define an additive mention policy on top of `allowedImplicitMentionKinds`, then wire and document it consistently for the affected bundled channel plugins while preserving Slack's existing option as a compatible alias or narrower control.

## Reproduction Assessment

Yes. A source-level reproduction exists for the remaining Discord path: create a bound Discord thread session with guild/channel `requireMention: true`, then send an unmentioned follow-up; current preflight disables `shouldRequireMention` for the bound thread and the test suite asserts that behavior.

## Solution Assessment

Unclear. A strict mode is directionally right, but the best maintainable design should name the public policy once, map it onto the existing implicit-kind allowlist seam, and deliberately handle Slack's already-shipped `thread.requireExplicitMention` option.
