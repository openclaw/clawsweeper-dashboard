- `P3`: This is a real but non-emergency mention-policy enhancement that needs API/product design before implementation.
- `impact:message-loss`: Implicit thread or reply activation can route visible channel replies without an explicit mention, which is a message-delivery and routing problem.

## Summary

Keep open. Current main and the latest shipped tag still lack a shared strict implicit-mention policy; Slack has only a narrower thread-specific workaround, while Discord and other reply-aware channels still need a maintainer-approved config/API decision.

## What This Changes

Request an additive strict mention mode that disables implicit reply-to-bot or thread-participation activation while still allowing explicit bot mentions and configured mention patterns.

## Best Possible Solution

Add a backward-compatible strict implicit-mention policy to the shared activation contract, wire it through affected channel configs and docs, and preserve existing implicit behavior as the default unless maintainers choose a migration.

## Reproduction Assessment

Yes, source-level. Current shared mention-gating, channel docs, and reply-aware channel implementations show implicit reply kinds still satisfy `requireMention` unless a channel-specific allowlist is emptied, and no shared strict config mode exists.

## Solution Assessment

Unclear until maintainers choose the public API. A strict mode is directionally sound, but the maintainable solution should be additive and coordinated across the shared SDK contract, Slack's existing option, Discord thread/session behavior, and docs.
