## Label Justifications

- `P2`: This is a normal-priority media-understanding/provider routing bug with limited but real user impact for custom provider configurations.
- `impact:auth-provider`: The issue concerns model/provider capability metadata and routing decisions for media-understanding image selection.
- `impact:message-loss`: When a text-only model is selected as the image-understanding model, inbound image descriptions are not delivered to the agent context.

## Summary

Keep open. Current main and the latest release still trust user-declared `input: ["image"]` when selecting and registering media-understanding image providers, even when the bundled provider catalog has the same model as text-only; the broader open validation issue does not fully cover this catalog cross-check and diagnostic gap.

## What This Changes

The issue asks OpenClaw to validate or better diagnose media-understanding image-model capability mismatches instead of silently routing images to user-declared text-only models.

## Best Possible Solution

Define a capability-source contract for configured media models, then prevent known text-only catalog rows from being auto-selected for image understanding unless maintainers intentionally add an explicit override path, and preserve the resolved provider/model plus capabilities in failure diagnostics.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 still accept configured `input: ["image"]` as sufficient for image-provider registration/default selection before checking known catalog capabilities; I did not run a live NVIDIA call in this read-only review.

## Solution Assessment

Unclear. The requested guardrail is valid, but maintainers need to choose whether configured capability metadata is authoritative, warning-only, or subordinate to known bundled catalog rows before an automated fix changes routing behavior.
