## Summary

Keep open. Current main still treats a configured model entry with `input` containing `image` as authoritative for media-understanding routing, while the bundled NVIDIA catalog marks the reported Kimi model as text-only; changing that config-versus-catalog authority needs maintainer product judgment rather than cleanup closure.

## What This Changes

Request validation and diagnostics so media-understanding does not silently auto-route images to a user-configured model whose declared image capability conflicts with actual or bundled catalog capabilities.

## Best Possible Solution

Preserve the existing config-authority contract unless maintainers choose otherwise, then add a catalog-aware warning or clearer rejected-image diagnostic with regression coverage for configured image rows that conflict with bundled text-only metadata.

## Reproduction Assessment

Yes for the OpenClaw-side path: current source shows a configured provider model with `input` containing `image` is selected before catalog checks, and the bundled NVIDIA catalog marks the reported Kimi model text-only. I did not live-call NVIDIA, so the provider rejection portion rests on the reporter's log plus the catalog mismatch.

## Solution Assessment

Unclear. A hard catalog refusal is not clearly the best fix because current catalog authority deliberately lets user config override manifests; a maintainer-approved diagnostic or warning path is the safer first design point.
