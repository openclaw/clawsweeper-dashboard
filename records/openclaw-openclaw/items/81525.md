## Summary

Keep open. Current main still trusts user-declared image capability before catalog/default checks and still truncates failed media-understanding diagnostics, so the reported validation and observability gap remains unresolved.

## What This Changes

The issue asks OpenClaw to validate or warn on user-declared image-capable media models that conflict with actual provider/catalog capability, and to make failed image diagnostics actionable.

## Best Possible Solution

Adopt a maintainer-approved catalog-aware warning or validation policy for configured image models, and make failed media-understanding summaries include the attempted provider/model plus the full capability reason.

## Reproduction Assessment

Yes for the OpenClaw-side current-main path: source inspection shows configured image rows are selected before catalog/default checks, and failed summaries truncate the detailed text-only capability error. I did not live-call NVIDIA, so that rejection portion relies on the reporter's log.

## Solution Assessment

Unclear. Hard-refusing catalog-narrower user config may be too aggressive because current behavior intentionally lets configured rows override discovered/catalog rows; warning plus clearer failed-attempt diagnostics is the safer first design point unless maintainers choose strict catalog authority.
