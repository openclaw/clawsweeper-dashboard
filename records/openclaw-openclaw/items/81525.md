## Summary

Keep open. Current main still trusts configured `models.providers.*.models[*].input` as image-capability authority, still lets configured image rows win ahead of manifest defaults, and still collapses failed media-understanding diagnostics before logging them.

## What This Changes

The issue requests validation or clearer diagnostics when media-understanding selects a configured image model whose declared image capability disagrees with provider/catalog reality.

## Best Possible Solution

Adopt a maintainer-approved catalog-aware warning or validation policy for configured image models, and make failed media-understanding summaries include the attempted provider/model plus the full capability reason.

## Reproduction Assessment

Yes for the OpenClaw-side path: source inspection shows configured image rows are trusted before catalog checks, the NVIDIA catalog marks the reported model text-only, and the warning formatter drops the detailed failure reason. I did not perform a live NVIDIA call.

## Solution Assessment

Unclear. A strict refusal of catalog-narrower user config may break intentional custom-provider overrides; a warning policy plus better failed-attempt diagnostics looks like the safer first maintainer decision point.
