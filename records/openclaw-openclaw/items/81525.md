## Summary

Keep open. Current main still trusts user-declared image capability metadata for media-understanding routing before any bundled-catalog cross-check, and the reporter's follow-up identifies a still-present diagnostic truncation path; choosing strict validation versus warning-only behavior needs maintainer product judgment.

## What This Changes

The issue requests safer media-understanding image-model selection and clearer diagnostics when configured model capability metadata conflicts with actual or bundled provider capabilities.

## Best Possible Solution

Add a maintainer-approved guardrail for configured image capability mismatches, and separately preserve attempted provider/model plus full capability error details in failed media-understanding warnings with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw-side path: current source selects configured image-capable rows without a catalog cross-check, and the bundled NVIDIA manifest marks the reported Kimi model text-only. I did not live-call NVIDIA, so the external rejection itself rests on the reporter's log evidence.

## Solution Assessment

Unclear. A hard catalog refusal may not be the best fix because current behavior gives configured catalog rows high authority, so a maintainer-approved warning, skip, probe, or explicit override policy should come before implementation.
