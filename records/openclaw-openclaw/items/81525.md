## Summary

Keep open. Current main still treats configured `input: ["text", "image"]` as authoritative for image routing and still truncates the detailed failed-model diagnostic, while the related closed fixes only addressed lookup/resolution gaps rather than catalog-vs-config validation.

## What This Changes

The issue requests capability validation or clearer diagnostics when media-understanding selects a user-configured image model whose declared `input` conflicts with actual catalog/provider capabilities.

## Best Possible Solution

Add maintainer-approved catalog-aware validation or warnings that preserve intentional overrides, and make failed media-understanding summaries retain attempted provider/model plus resolved input detail.

## Reproduction Assessment

Yes for the OpenClaw-side path: current source selects configured image rows before catalog/default checks, the bundled NVIDIA Kimi row is text-only, and the warning formatter drops the detailed error tail. I did not live-call NVIDIA, so the external provider rejection relies on the reporter's log.

## Solution Assessment

Unclear. Hard-refusing user config that conflicts with bundled catalog metadata may not be the best fix because configured rows currently override discovery/catalog data; a warning plus clearer failed-attempt diagnostics is the safer first design unless maintainers choose strict catalog authority.
