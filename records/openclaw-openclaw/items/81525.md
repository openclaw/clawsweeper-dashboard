## Summary

Keep open. Current main and the latest release still trust user-declared image capability for configured media-understanding models and still truncate failed capability diagnostics, so the reported validation and observability gap remains unresolved.

## What This Changes

The issue asks OpenClaw to validate or clearly diagnose media-understanding image-model selection when user-configured provider metadata declares image support that may conflict with catalog or runtime capability.

## Best Possible Solution

Define a catalog-aware policy for configured image-model capability mismatches, then preserve attempted provider/model and full capability details in failed media-understanding summaries.

## Reproduction Assessment

Yes for the OpenClaw-side path: current source still trusts configured `input: ["image"]` rows for image-model selection while the NVIDIA manifest marks the reported Kimi model text-only. I did not live-call NVIDIA, so the provider rejection portion relies on the reporter's log evidence.

## Solution Assessment

Unclear. A hard catalog refusal may be too aggressive for custom providers; a warning plus clearer failed-attempt diagnostics is the narrower first fix unless maintainers choose strict catalog authority.
