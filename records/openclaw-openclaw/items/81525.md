## Summary

Keep open. Current main still trusts user-declared image capability for media-understanding routing, while the reported NVIDIA catalog entry is text-only and failed summaries still drop useful diagnostic detail; the right fix needs maintainer judgment on config authority versus catalog warnings/probes.

## What This Changes

The issue requests safer media-understanding validation and diagnostics when configured model input metadata says a model supports images but bundled catalog/runtime behavior disagrees.

## Best Possible Solution

Define the config-versus-catalog authority for media input capabilities, then add either a catalog-aware warning/probe or a clearer failed-decision summary with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw-side path: current source selects configured `input: ["image"]` rows before catalog checks, and current formatting can hide the resolved provider/model detail. I did not live-call NVIDIA, so the upstream rejection is supported by the reporter's log and the bundled catalog mismatch.

## Solution Assessment

Unclear. Refusing user overrides against bundled catalog metadata is a product decision; a maintainer-approved warning/probe or narrower failed-summary diagnostic looks safer than changing config authority unilaterally.
