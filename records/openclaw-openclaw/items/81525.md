## Summary

Keep open. Current main still trusts user-configured `models.providers.*.models[].input` for image media-understanding routing before catalog vision checks; deciding whether bundled catalog metadata should warn, constrain, or only diagnose user overrides is a config-contract/product decision, not cleanup closure.

## What This Changes

The issue requests catalog-aware validation, probing, deterministic ordering, or clearer diagnostics when a user-declared image-capable model is text-only in bundled/provider metadata.

## Best Possible Solution

Add a maintainer-approved catalog-aware warning or structured rejection diagnostic that preserves intentional custom-provider overrides unless a documented override policy changes that contract.

## Reproduction Assessment

Yes for the OpenClaw-side routing path: current source selects user-configured image entries before catalog vision checks, and the bundled NVIDIA catalog marks the reported model text-only. The downstream provider rejection rests on the reporter's log because no live NVIDIA call was run.

## Solution Assessment

Unclear. A hard catalog refusal is risky because current behavior gives configured catalog rows authority; a warning or clearer provider-rejection diagnostic is the safer first design point unless maintainers choose a stricter policy.
