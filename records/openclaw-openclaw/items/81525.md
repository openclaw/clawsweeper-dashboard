## Summary

Keep open. Current main still trusts configured `input: ["image"]` model rows ahead of catalog capability checks and still collapses failed media-understanding diagnostics, so the reported validation and observability gap remains unresolved. The open related items are adjacent, but none is a canonical replacement for this exact configured-capability mismatch plus opaque warning path.

## What This Changes

The issue asks OpenClaw to validate or clearly diagnose media-understanding image routing when user-configured model capability metadata disagrees with actual provider/catalog support.

## Best Possible Solution

Adopt a maintainer-approved catalog-aware warning or validation policy for configured image models, and make failed media-understanding summaries include the attempted provider/model plus the full capability failure reason.

## Reproduction Assessment

Yes for the OpenClaw-side path: source inspection shows configured image rows are selected before catalog checks and failed summaries truncate the detailed capability error. I did not live-call NVIDIA, so the external provider rejection portion relies on the reporter's journal logs.

## Solution Assessment

Unclear. A diagnostic-only fix is narrow, but strict catalog rejection versus warning needs maintainer policy because current behavior intentionally gives configured model metadata authority over discovered/catalog metadata.
