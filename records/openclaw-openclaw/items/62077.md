## Summary

Keep open: current main still emits full direct non-webchat Conversation info and Sender metadata for Telegram DMs, no compact/minimal metadata config exists on main, and the active compact-metadata implementation remains an open unmerged PR needing proof and product review.

## What This Changes

The issue asks for a minimal or compact inbound metadata mode for direct chats, especially Telegram DMs, to reduce repeated Conversation/Sender JSON blocks in long-session context.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/72749 or an equivalent backward-compatible opt-in compact direct metadata mode; keep the broader UX discussion aligned with https://github.com/openclaw/openclaw/issues/72704.

## Reproduction Assessment

Yes for the observable prompt-shape complaint: source inspection shows Telegram DMs become direct telegram context, direct non-webchat turns emit full Conversation/Sender blocks, and get-reply-run prepends that prefix. I did not live-measure the hallucination impact.

## Solution Assessment

Unclear as a final product decision. An opt-in compact direct metadata mode is plausible, but the exact config contract and safety-preserving metadata shape need maintainer review through the linked implementation before this issue closes.
