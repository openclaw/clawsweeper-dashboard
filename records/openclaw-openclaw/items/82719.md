## Label Justifications

- `P2`: This is a normal-priority Plugin SDK feature request with clear demand but no current outage or regression.
- `impact:security`: The request changes the trust boundary for non-bundled plugins that could mutate outbound content and model-visible tool results.
- `impact:message-loss`: The requested dispatch hooks include authority to cancel, suppress, or rewrite outbound channel messages.
- `impact:session-state`: Tool-result middleware can change the result the agent sees, affecting same-turn model context and transcript/session state.

## Summary

Keep open. Current main still has no trusted opt-in that lets non-bundled plugins use the requested dispatch and tool-result mutation surfaces, and the remaining work crosses plugin authority, message delivery, and model-visible tool-result trust boundaries.

## What This Changes

The issue requests an explicit trusted-plugin opt-in so user-installed plugins can mutate outbound dispatch and tool-result paths for workflows such as automatic VoiceBox audio attachment.

## Best Possible Solution

Design a narrow per-surface trust policy under the existing plugin hook/config model, then update registry, lazy loading, docs, diagnostics, and contract tests for explicitly opted-in local plugins.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for a source-level feature-gap check: current main blocks non-bundled tool-result middleware and lacks the proposed trusted config path. I did not run the live VoiceBox setup.

## Solution Assessment

Unclear: the capability is plausible, but a broad top-level trusted boolean is probably less maintainable than explicit per-hook or per-surface trust controls with audit/debug behavior.
