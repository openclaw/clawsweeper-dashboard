## Summary

Keep open. Current main still blocks the requested non-bundled tool-result middleware path and has no trusted dispatch opt-in for user plugins; this is a plausible Plugin SDK gap, but it changes plugin authority over outbound delivery and model-visible tool results, so it needs maintainer product and security design instead of cleanup closure.

## What This Changes

The issue requests an explicit trusted-plugin opt-in so user-installed plugins can use dispatch and tool-result mutation hooks currently reserved for bundled plugin paths.

## Best Possible Solution

Design a narrow, explicit per-surface trust policy for local non-bundled code plugins, then update config/schema, registry, lazy middleware loading, docs, and contract tests so only approved dispatch and middleware hooks are elevated.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a Plugin SDK feature request. Source inspection gives a high-confidence feature-gap check: current main still blocks non-bundled tool-result middleware and lacks the proposed trust config surface.

## Solution Assessment

Unclear. The capability is plausible, but the proposed top-level trusted boolean should be reviewed against narrower per-hook or per-surface trust controls before implementation.
