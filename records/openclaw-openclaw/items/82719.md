## Summary

Current main now covers the central request: explicitly enabled installed plugins with manifest-declared contracts can register agent tool-result middleware, and routed ACP/channel replies have a payload-level mutation hook before delivery. The exact `trusted: true` config shape was not added, but the merged manifest-contract and explicit-enablement model is a narrower security boundary for the same use case.

## What This Changes

The issue requests an explicit trust model that lets user-installed plugins opt into dispatch-path and tool-result hooks currently restricted to bundled plugins.

## Best Possible Solution

Keep the manifest-contract plus explicit-enablement trusted-hook model from the merged implementation, and use `reply_payload_sending` for routed ACP/channel payload mutation.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The reported old bundled-only middleware gate is visible in v2026.6.5 source, while current main no longer has that limitation for explicitly enabled installed plugins with matching contracts.

## Solution Assessment

Yes. The current manifest-declared, explicitly enabled trusted-hook path is narrower and more auditable than a broad `trusted: true` config flag, while still covering tool-result mutation and routed outbound payload mutation.
