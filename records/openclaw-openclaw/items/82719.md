## Summary

Keep open. Current main and v2026.5.12 still keep agent tool-result middleware bundled-only, the plugin entry schema has no `trusted` opt-in, and related open middleware PRs do not cover non-bundled dispatch trust; this widens plugin authority over outbound messages and agent-visible tool results, so it needs maintainer product/security design rather than cleanup.

## What This Changes

The issue requests an explicit trust opt-in so user-installed plugins can use dispatch-path hooks and agent tool-result middleware.

## Best Possible Solution

Design an explicit per-surface trust policy under the plugin hook/config model, then update registry, lazy loading, docs, and contract tests before allowing non-bundled plugins to mutate dispatch or tool results.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested non-bundled middleware path remains blocked and no `trusted` config key exists on current main.

## Solution Assessment

Unclear: the capability is plausible, but a broad `trusted: true` boolean is probably too coarse. A per-hook or per-surface trust policy with audit/debug behavior would be safer.
