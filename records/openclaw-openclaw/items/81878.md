## Summary

Keep open: current main still implements the explicit-only native Ollama `num_ctx` behavior that the reporter identifies, and the issue is a real policy/regression conflict rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to preserve or migrate the previous Ollama behavior where configured `contextWindow` could drive native `/api/chat` `options.num_ctx` when `params.num_ctx` is absent.

## Best Possible Solution

Decide the compatibility policy explicitly, then implement a targeted path that preserves intentional user `contextWindow` caps or migrates them to `params.num_ctx` without blindly restoring catalog fallback for every native Ollama model.

## Reproduction Assessment

Yes from source, with high confidence: current main only forwards explicit `params.num_ctx`, and the adjacent tests assert that `contextWindow` and `maxTokens` do not become native Ollama `num_ctx`. I did not run a live Ollama daemon in this read-only review.

## Solution Assessment

Unclear. Reverting `a8c745a6230e1bce61412007c6b21fc48cddb411` wholesale is not the safest fix because it conflicts with the latency fix for https://github.com/openclaw/openclaw/issues/62267; a narrower explicit-config or doctor-migration solution looks more maintainable.
