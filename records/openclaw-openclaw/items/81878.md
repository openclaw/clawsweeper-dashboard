## Summary

Keep open: current main still implements the explicit-only native Ollama `num_ctx` behavior the report identifies, and the remaining decision conflicts with a recently shipped latency fix rather than being stale cleanup.

## What This Changes

The issue asks OpenClaw to preserve configured Ollama `contextWindow` behavior by ensuring native `/api/chat` requests receive the intended `options.num_ctx` or a migration path.

## Best Possible Solution

Decide the compatibility policy explicitly, then implement a targeted path that preserves intentional user context caps or migrates them to `params.num_ctx` without blindly restoring catalog fallback for every native Ollama model.

## Reproduction Assessment

Yes, at source level with high confidence: current main only forwards explicit `params.num_ctx`, and adjacent tests assert that `contextWindow` and `maxTokens` do not become native Ollama `num_ctx`. I did not run a live Ollama gateway request in this read-only review.

## Solution Assessment

Unclear. Reverting `a8c745a6230e1bce61412007c6b21fc48cddb411` wholesale is not obviously the best fix because it conflicts with the latency fix for https://github.com/openclaw/openclaw/issues/62267; a narrower explicit-config or doctor-migration solution looks more maintainable.
