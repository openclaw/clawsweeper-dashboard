## Summary

Keep open: current main still omits native Ollama `options.num_ctx` unless `params.num_ctx` is explicit, the reported regression is source-reproducible, and https://github.com/openclaw/openclaw/pull/81900 is the paired open fix path rather than a reason to close this issue early.

## What This Changes

Ollama native requests should preserve an operator-configured `contextWindow` or `maxTokens` budget after upgrades instead of falling back to Ollama's Modelfile context default when `params.num_ctx` is absent.

## Best Possible Solution

Keep this issue open until maintainers choose and land a narrow compatibility fix, likely either a revised PR that preserves explicit configured budgets or a `doctor --fix` migration to `params.num_ctx` without reintroducing implicit discovered-window forcing.

## Reproduction Assessment

Yes, at source level. Current main's resolver and tests show that `contextWindow` and `maxTokens` are not forwarded as native Ollama `options.num_ctx` when `params.num_ctx` is absent, matching the reported request-body regression.

## Solution Assessment

Unclear. Restoring some compatibility path is warranted, but the exact implementation needs maintainer judgment because a broad fallback may affect discovered/default models while a `doctor --fix` migration or provenance-aware fallback would be narrower.
