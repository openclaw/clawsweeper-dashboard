## Summary

Keep open: current main still implements the explicit-only native Ollama `num_ctx` behavior, the reporter's regression claim is source-reproducible, and an open paired PR now targets this issue directly.

## What This Changes

Restore or migrate native Ollama `/api/chat` context handling so configured `contextWindow` or `maxTokens` no longer falls back to Ollama's smaller default when `params.num_ctx` is omitted.

## Best Possible Solution

Review and land the paired PR or an equivalent narrow fix that preserves explicit `params.num_ctx`, keeps no-budget models under Ollama policy, and covers the compatibility path with tests and docs.

## Reproduction Assessment

Yes. Current main source and tests show that `contextWindow` and `maxTokens` are not forwarded as native Ollama `options.num_ctx`, matching the reporter's described request-body regression; I did not run a live Ollama daemon.

## Solution Assessment

Unclear. Restoring configured-budget fallback is the narrowest code fix for the observed regression, but maintainers should choose whether that is preferred over a `doctor --fix` migration because the current behavior was intentionally shipped for latency and memory concerns.
