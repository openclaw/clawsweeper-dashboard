## Summary

Keep open. Current main still models filesystem policy as a boolean workspace-only switch with single-root read guards, while the related granular policy issue and security hardening PRs remain open; widening direct-session read access needs maintainer security policy before cleanup can close this.

## What This Changes

The issue asks for direct-session `read` to honor the effective filesystem policy instead of staying bound to the workspace root when broader read access is enabled.

## Best Possible Solution

Define one filesystem access contract for read roots, sensitive-path denial, read-vs-write behavior, sandbox mounts, direct-session tool assembly, and per-agent privacy, then route read/write/edit/apply_patch, media loading, node output paths, sandbox bridges, and plugin tool context through that shared policy with regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main still shows boolean-only filesystem policy and single-root guard wiring. The exact macOS direct-session tool-host failure remains unclear until maintainers live-validate that surface with `tools.fs.workspaceOnly=false`.

## Solution Assessment

Unclear as an isolated `read` relaxation. The best direction is the issue's policy-unification goal, but the maintainable fix should add an explicit shared filesystem policy with approved roots and deny rules rather than only bypassing the workspace guard.
