## Summary

Keep open. Current main still has only a boolean filesystem policy, single-root read/tool guards, and no approved-read-roots or sensitive-path contract; the linked security PR #42207 is still open rather than merged, so this needs maintainer policy before implementation.

## What This Changes

The issue asks for the direct-session `read` tool to honor the effective filesystem policy and approved non-workspace roots instead of remaining bound to the workspace root.

## Best Possible Solution

Define one explicit filesystem access contract for `read`, `write`, `edit`, `apply_patch`, prompt media loading, sandbox bridges, and plugin tool context: approved read roots, sensitive-path denial, per-agent privacy semantics, and shared resolver coverage should land before direct-session read is widened.

## Reproduction Assessment

Yes for the source-level mismatch: current main shows boolean-only `tools.fs`, single-root read/guard wiring, and no approved-root schema. The exact macOS direct-session failure should still be live-validated before maintainers widen read access.

## Solution Assessment

No as a standalone `read` relaxation. The maintainable fix is a shared filesystem policy contract with explicit allowed roots and sensitive-path denial, then wiring direct-session read through that contract.
