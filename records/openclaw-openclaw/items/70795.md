## Summary

Keep open. Current main still has a boolean-only filesystem policy and single-root read guards, while the related granular policy and security-hardening work remains open; widening direct-session read access needs an explicit maintainer security contract before cleanup can close this.

## What This Changes

The issue asks for direct-session `read` to honor the effective filesystem policy instead of staying bound to the workspace root when broader read access is enabled.

## Best Possible Solution

Define one filesystem access contract for read roots, sensitive-path denial, read-vs-write behavior, sandbox mounts, direct-session tool assembly, and per-agent privacy, then route all filesystem-facing tools through it with regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main still shows boolean-only filesystem policy and single-root guard wiring. The exact macOS direct-session tool-host failure was not live-reproduced in this read-only review.

## Solution Assessment

Unclear as an isolated `read` relaxation. The maintainable direction is the issue's policy-unification goal, with explicit approved roots and deny rules rather than a local bypass of the workspace guard.
