## Label Justifications

- `P1`: The PR targets a broken scheduled-agent fallback workflow, but its current timeout changes can also regress timed cron run behavior.
- `merge-risk: 🚨 availability`: Dropping the shared TimeoutError abort signal can let fallback work continue after the configured cron watchdog has fired.

- `merge-risk: 🚨 auth-provider`: The PR changes provider/model fallback routing for isolated cron jobs and may contact fallback providers in cases that currently stop at timeout.

## Summary

Keep open, but this PR is not merge-ready: the embedded-result classifier change is useful and already covered by the proof-positive canonical PR, while this PR's unique timeout-abort path can let a fallback continue and perform session or delivery side effects after the scheduler has already marked the cron run timed out.

## What This Changes

The PR wires embedded-result fallback classification into isolated cron agent turns, adds helpers/tests for dropping cron TimeoutError aborts during fallback, and changes finalization to keep visible answers after timeout.

## Best Possible Solution

Keep the classifier repair on the canonical embedded-result fallback path and handle shared-timeout recovery separately with a deadline-preserving design and real cron proof.

## Reproduction Assessment

Yes for the returned-result classifier gap and for the introduced timeout risk at source level. The PR does not include real cron proof for recovering after a wall-clock TimeoutError.

## Solution Assessment

No. Reusing the embedded classifier is the right narrow fix, but dropping the cron wall-clock abort signal is not the best timeout solution because the scheduler already owns that deadline and records the run as timed out.
