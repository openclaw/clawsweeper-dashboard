## Summary

Keep open: the PR is a useful UX direction, but it is not merge-ready because it regresses the promised search bypass, leaves the new hint field out of the protocol schema, and misses the required user-facing changelog entry.

## What This Changes

The PR marks selected config UI fields as advanced, renders top-level advanced fields in a collapsed disclosure, and adds a `docLink` hint rendered as an OpenClaw docs anchor.

## Best Possible Solution

Update this PR so active search always bypasses the advanced disclosure, `docLink` is represented in the protocol/docs contract, focused render/search tests cover the behavior, and the changelog records the UI feature.

## Reproduction Assessment

Yes. A source-level reproduction is enough: a search that matches the top-level section itself, including `tag:advanced` for section nodes, clears `childSearchCriteria`, and the PR then treats search as inactive for the advanced split.

## Solution Assessment

No. The basic/advanced split is a reasonable direction, but the bypass should key off the original active `searchCriteria`, and the new hint field should be added to the protocol schema and public hint surfaces before it is relied on.
