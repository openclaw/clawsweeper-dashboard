## Summary

Keep open. Current main has partial guardrails for task cleanup, load-time session pruning, manual session cleanup, and optional disk budgets, but it still does not resolve the reported default-case memory bloat: under-500 large stores remain live, startup still reads and parses the whole session store, byte-budget cleanup is opt-in, and skill snapshots are still stored inline per session entry.

## What This Changes

The issue asks to bound gateway memory growth by pruning or archiving accumulated session and task records, deduplicating per-session skills snapshots, detecting zombie running tasks, and avoiding eager loading of historical session metadata.

## Best Possible Solution

Keep the existing task-ledger and manual cleanup guardrails, then make an explicit session-storage policy change: add a default live-entry or byte budget that catches large under-500 stores, avoid eager full-store materialization for historical metadata, and replace inline skillsSnapshot copies with a compatible interned or versioned reference model.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline, parses the full sessions.json before cleanup, and would not prune a 471-entry store whose rows are 26 days old under the default 30-day, 500-entry, and 550 high-water behavior. The exact heap-growth number was not reproduced live in this read-only sweep.

## Solution Assessment

Unclear. The current task TTL, lost-task reconciliation, disk-budget option, load-time pruning, and cleanup command are useful partial fixes, but the complete fix needs a maintainer-chosen retention, lazy-loading, and snapshot-deduplication policy rather than a narrow mechanical patch.
