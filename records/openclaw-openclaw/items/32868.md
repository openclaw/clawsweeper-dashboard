## Summary

Keep open. Current main still has the same-channel block-streaming await gap: block replies are queued on the dispatcher, but the callback used by the tool-start flush returns before actual channel delivery settles.

## What This Changes

The issue asks for `text_end` block-streamed same-channel text to be delivered before subsequent tool execution instead of batching at turn end.

## Best Possible Solution

Make accepted same-channel block delivery awaitable in the core reply dispatcher and await it from `dispatch-from-config` before the block-reply flush resolves, while preserving routed-channel behavior, dropped-payload semantics, ACP tracking, and plugin SDK compatibility.

## Reproduction Assessment

Yes. Enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current source shows the flush callback resolves after dispatcher enqueue rather than channel delivery, matching the reported multi-channel batching behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to expose or return an awaitable delivery completion for accepted same-channel block replies and await it at the existing block flush boundary, without changing unrelated disabled-streaming policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/32868 on current main. The observable bug is that with block streaming enabled and `blockStreamingBreak: "text_end"`, same-channel block replies are enqueued on `ReplyDispatcher` but the `onBlockReply` callback resolves before actual channel delivery, so tool execution can continue before the user receives the preceding text. In `src/auto-reply/reply/reply-dispatcher.ts` and `src/auto-reply/reply/reply-dispatcher.types.ts`, make accepted `sendBlockReply` delivery awaitable without breaking existing plugin SDK implementers that return synchronous `true`; preserve `false` for skipped/dropped payloads. In `src/auto-reply/reply/dispatch-from-config.ts`, await accepted same-channel block delivery before resolving the callback used by `blockReplyPipeline.flush()` and `onBlockReplyFlush`; preserve the existing routed-channel `sendPayloadAsync` path, ACP delivered tracking, human-delay ordering, typing/lifecycle behavior, and abort/timeout safety. Use #61974, #63051, and #65761 as references, but avoid their plugin-SDK compatibility and timeout-string pitfalls. Do not broaden the `blockStreamingDefault: "off"` behavior or fold in unrelated #65541 changes. Add or update focused regression tests proving dispatcher delivery is awaitable and dispatch-from-config does not resolve same-channel `onBlockReply` until delivery settles. Add a user-facing changelog entry for the bug fix.
