## Summary

Keep open. Current main and the latest release still have the same source-level gap: the text_end flush is awaited, but the same-channel block reply callback only enqueues dispatcher delivery and returns before the actual channel send drains.

## What This Changes

The issue reports that same-channel block-streamed text is queued until the model turn ends instead of being delivered before the next tool call.

## Best Possible Solution

Implement a narrow awaitable same-channel block-delivery contract so text_end block replies drain through actual dispatcher delivery before tool execution continues, with backward-compatible types, abort/timeout handling, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main awaits the text_end flush boundary, but the same-channel callback enqueues dispatcher delivery and returns before sendChain drains; reporter comments and the duplicate WhatsApp issue confirm the same behavior across real channels.

## Solution Assessment

Yes. The narrow fix is to await actual same-channel dispatcher delivery, either by making sendBlockReply return an awaitable delivery result or by waiting for dispatcher idle at the block-reply flush boundary, rather than adding channel-specific workarounds.
