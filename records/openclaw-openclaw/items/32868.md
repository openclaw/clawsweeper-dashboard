## Summary

Keep open: current main still has the same same-channel block-reply delivery gap, and the latest shipped release also keeps the boolean dispatcher contract. The issue has independent WhatsApp, Telegram, Discord, and WeCom reports plus a community plugin workaround, but no merged core fix.

## What This Changes

The issue asks OpenClaw to deliver `text_end` block-streaming replies to same-channel users before subsequent tool execution begins.

## Best Possible Solution

Land a narrow, abort-aware same-channel delivery barrier so accepted `text_end` block replies drain through the dispatcher before tool execution proceeds.

## Reproduction Assessment

Yes. The source path on current main still flushes the block pipeline before tool execution, but the same-channel callback resolves after queueing dispatcher delivery rather than after channel delivery completes.

## Solution Assessment

Yes. The narrow maintainable fix is to make accepted same-channel block delivery awaitable or await a tightly scoped dispatcher barrier, while avoiding a broader followup lifecycle wait.

Current main awaits routed/cross-channel `sendPayloadAsync`, but same-channel `onBlockReply` only calls `dispatcher.sendBlockReply(normalizedPayload)` while `sendBlockReply` returns boolean after enqueueing async delivery. Update the dispatcher contract or same-channel path so an accepted block reply can be awaited through actual `options.deliver()` completion before tool execution proceeds. Preserve dropped/silent payload semantics, ordering, humanDelay behavior, abort/cancellation behavior, and existing public plugin API expectations; do not implement a broad followup-runner idle wait. Add focused regression coverage for same-channel delivery ordering before tool start, keep cross-channel behavior intact, and add a user-visible changelog fix entry if the repair lane owns changelog updates.
