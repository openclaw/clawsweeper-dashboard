## Summary

Keep open: current main and the latest shipped release still have the same same-channel block-reply delivery gap. The discussion has independent Telegram, Discord, and WeCom confirmations plus a community plugin workaround, but that workaround does not fix the core dispatcher barrier.

## What This Changes

The issue reports that same-channel block-streaming text is queued but not delivered to the chat before the next tool starts when `blockStreamingBreak` is `text_end`.

## Best Possible Solution

Land a narrow, abort-aware same-channel block reply delivery barrier so accepted `text_end` block replies drain through the dispatcher before tool execution proceeds.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main still flushes the block pipeline before tool execution, but the same-channel `onBlockReply` callback resolves after queueing dispatcher delivery rather than after channel delivery completes.

## Solution Assessment

Yes, the best fix direction is a narrow same-channel delivery barrier: return and await a specific accepted block delivery promise or a tightly scoped dispatcher idle wait. The broader https://github.com/openclaw/openclaw/pull/52026 followup-idle approach waits at a different lifecycle boundary and already has a review blocker.
