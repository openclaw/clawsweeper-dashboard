## Summary

Keep open. Current main still matches the reported same-channel bug: cross-channel block replies await delivery, but same-channel block replies only enqueue dispatcher delivery and return before the channel send completes. The issue also has independent Telegram, Discord, WhatsApp, and WeCom confirmations, plus prior unmerged fix attempts.

## What This Changes

The issue requests that `blockStreamingBreak: "text_end"` same-channel block replies be delivered before the next tool executes.

## Best Possible Solution

Land a narrow reply-dispatcher fix that makes accepted same-channel block deliveries awaitable before `onBlockReply` resolves, with compatibility and timeout or abort coverage.

## Reproduction Assessment

Yes, by source inspection: use a same-channel route where `originatingChannel === currentSurface`, enable `blockStreamingBreak: "text_end"`, and emit text before a tool call. The current callback returns after queueing `sendBlockReply`, not after `options.deliver()` completes.

## Solution Assessment

Yes. The narrow maintainable fix is to make same-channel block delivery await actual dispatcher delivery, preferably with a backward-compatible `sendBlockReply` delivery promise or equivalent helper and focused ordering tests.
