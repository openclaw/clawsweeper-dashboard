## Summary

Keep open. Current main still has the same-channel block-streaming await gap: the tool-start path flushes the block pipeline, but the same-channel dispatch callback queues `sendBlockReply` and returns before channel delivery settles; the linked exact-fix PRs are all closed unmerged.

## What This Changes

The issue requests that `blockStreamingBreak: "text_end"` same-channel block replies be delivered to the channel before subsequent tool execution continues.

## Best Possible Solution

Keep this as the canonical open bug and land a focused reply dispatcher/pipeline fix with regression coverage for same-channel `text_end` delivery before tool execution.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current main flushes the block pipeline but the same-channel dispatcher send still resolves before actual channel delivery settles.

## Solution Assessment

Yes. The narrow maintainable solution is to make accepted same-channel block delivery awaitable at the existing flush boundary, while preserving routed-channel behavior, dropped-payload semantics, ACP tracking, human-delay ordering, and plugin SDK compatibility.
