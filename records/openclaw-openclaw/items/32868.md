## Summary

Keep open: current main still queues same-channel block replies through a non-awaitable dispatcher path, so the intended pre-tool `text_end` flush does not prove channel delivery completed. The discussion has independent WhatsApp, Telegram, Discord, and WeCom reports, plus an open but broader followup-idle PR that does not supersede this narrower bug.

## What This Changes

Fix `text_end` block streaming so same-channel block replies are delivered before the next tool executes.

## Best Possible Solution

Land a narrow, abort-aware same-channel delivery barrier so accepted `text_end` block replies drain through the dispatcher before tool execution proceeds.

## Reproduction Assessment

Yes, source-reproducible: current main flushes before tool execution, but the same-channel callback resolves after queueing dispatcher delivery rather than after channel delivery completes. I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make accepted same-channel block delivery awaitable or add a tightly scoped dispatcher barrier, while avoiding the broader followup-idle approach in the open related PR.
