## Summary

Keep open. Current main still queues same-channel block replies through the dispatcher without waiting for channel delivery, so `text_end` flushes can finish before the user-visible block message is actually sent.

## What This Changes

The issue reports that same-channel block-streamed text should reach the chat before later tool execution when `blockStreamingBreak` is `text_end`.

## Best Possible Solution

Land a focused core reply dispatcher fix that makes accepted same-channel block delivery awaitable at the existing flush boundary, with regression coverage and no new delivery policy or config mode.

## Reproduction Assessment

Yes, as a source-level reproduction. Enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current main flushes the block pipeline but the same-channel dispatcher send still resolves before actual channel delivery settles.

## Solution Assessment

Yes. The narrow maintainable fix is to make same-channel block reply delivery awaitable at the existing callback/flush boundary while preserving routed-channel behavior, dropped-payload semantics, dispatcher idle tracking, and plugin SDK compatibility.
