## What This Changes

The issue asks for `text_end` block-streamed replies to be delivered to the same channel before the next tool begins executing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83722, or an equivalent narrow same-channel delivery barrier, then close this issue with the merged commit and validation proof.

## Reproduction Assessment

Yes, source-reproducible: current main still queues same-channel block replies through a boolean dispatcher path while tool start waits only for the block-reply callback chain. Multiple user reports support the same timing failure, though this review did not run a live channel repro.

## Solution Assessment

Yes. The best fix is the existing linked PR or an equivalent narrow dispatcher-drain barrier before tool execution; closing the issue before that PR merges would be premature.
