## Summary

Keep open. Current main still queues same-channel block replies through the dispatcher without awaiting channel delivery, so `text_end` flushes can complete before the user-visible block message is actually sent.

## What This Changes

The issue asks for block-streamed same-channel text replies to be delivered before subsequent tool execution when `blockStreamingBreak` is `text_end`.

## Best Possible Solution

Keep this as the canonical bug and land a focused reply dispatcher/pipeline fix with regression coverage for same-channel `text_end` delivery before tool execution.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls. Current main flushes the block pipeline, but the callback returns after queueing dispatcher delivery rather than after channel delivery settles.

## Solution Assessment

Yes. The narrow maintainable fix is to make accepted same-channel block delivery awaitable at the existing flush boundary while preserving routed-channel behavior and current dispatcher compatibility.
