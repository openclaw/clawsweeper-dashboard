## Summary

Keep open: current main still routes same-channel block replies through a non-awaitable dispatcher call, so the documented `text_end` flush before tool execution does not prove the channel delivery completed. The discussion has independent WhatsApp, Telegram, Discord, and WeCom evidence, and the only open related PR is a broader followup-idle patch with a blocking review rather than a superseding fix.

## What This Changes

The issue requests that same-channel `text_end` block-streamed replies reach channel delivery before subsequent tool execution starts.

## Best Possible Solution

Land a narrow, abort-aware same-channel delivery barrier so accepted `text_end` block replies drain through the dispatcher before tool execution proceeds.

## Reproduction Assessment

Yes, source-reproducible: current main flushes before tool execution, but the same-channel callback resolves after queueing dispatcher delivery rather than after channel delivery completes. Multiple channel reports support the same timing failure, but I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make accepted same-channel block delivery awaitable or add a tightly scoped dispatcher barrier, not a broad followup-idle wait that changes unrelated lifecycle ordering.
