## Summary

Keep open: current main still has the same same-channel block streaming delivery gap. The code flushes block replies before tool start, but the same-channel callback only queues dispatcher delivery and returns before the channel send drains; related comments add Telegram, Discord, WhatsApp, and WeCom evidence, and prior direct fix PRs closed unmerged.

## What This Changes

The issue requests that same-channel `text_end` block-streamed replies be delivered before subsequent tool execution continues.

## Best Possible Solution

Land a narrow, abort-aware delivery barrier for same-channel block replies so `text_end` block flushes drain the relevant dispatcher delivery before tool execution proceeds, without broad followup lifecycle changes.

## Reproduction Assessment

Yes, source-reproducible with high confidence. On current main, the tool-start flush waits for the block pipeline, but the same-channel `onBlockReply` branch returns after `dispatcher.sendBlockReply()` queues async delivery rather than after the channel send completes.

## Solution Assessment

Yes, the best fix direction is the narrow delivery barrier: return/await a specific block delivery promise or use a tightly scoped dispatcher idle wait for the same-channel block path. The broader open followup-idle PR is not the right substitute because it waits at a different lifecycle boundary and has an existing P1 review concern.
