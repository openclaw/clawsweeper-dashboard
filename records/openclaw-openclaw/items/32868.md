## Summary

Keep open. Current main still has the same-channel block-streaming await gap: the dispatcher exposes `sendBlockReply` as boolean-only, schedules actual channel delivery on an internal promise chain, and the same-channel `onBlockReply` path calls it without awaiting delivery, while the cross-channel path awaits `sendPayloadAsync`. The issue has corroborating WhatsApp, Telegram, Discord, and WeCom reports, including a community plugin workaround that parses media from text downstream; related fix PRs for this exact bug are closed unmerged, and open PR #65541 covers disabled block streaming rather than enabled `text_end` same-channel ordering.

## What This Changes

The issue requests that `blockStreamingBreak: "text_end"` same-channel block replies be delivered and awaited before the next tool execution starts instead of batching at turn end.

## Best Possible Solution

Make accepted same-channel block delivery awaitable in the core reply dispatcher and have the `dispatch-from-config` same-channel `onBlockReply` path wait for that accepted delivery before tool-start processing proceeds, while preserving dropped-payload behavior, cross-channel routing, ACP delivered tracking, and plugin SDK contracts.

## Reproduction Assessment

Yes. The issue is reproducible by enabling block streaming with `blockStreamingBreak: "text_end"` on a same-channel chat turn that emits text before sequential tool calls; current source shows the same-channel callback resolves after enqueue rather than after channel delivery, and multiple channel reports confirm the observable batching.

## Solution Assessment

Yes, the narrow fix is to return or otherwise await the dispatcher delivery promise for accepted block replies, then await it in the same-channel block path. Awaiting a broad final `waitForIdle()` may work but is less precise unless scoped carefully around accepted block delivery and dropped-payload semantics.
