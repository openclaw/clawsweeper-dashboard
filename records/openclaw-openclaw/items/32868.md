## Summary

Keep open. Current main still has the same-channel block-streaming await gap: the block pipeline can flush before tool execution, but the same-channel callback only queues dispatcher delivery and returns before the channel send settles.

## What This Changes

The issue asks for `blockStreamingBreak: "text_end"` block replies to reach same-channel users before subsequent tool calls execute instead of batching until the turn completes.

## Best Possible Solution

Make accepted same-channel block delivery awaitable at the existing block-flush boundary while preserving routed-channel behavior, dropped-payload semantics, ACP tracking, and plugin SDK compatibility.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current source shows tool start awaits the pipeline flush, but that flush resolves after dispatcher enqueue rather than channel delivery.

## Solution Assessment

Yes. The narrow maintainable fix is to return or expose an awaitable completion for accepted same-channel `sendBlockReply` delivery and await it from `dispatch-from-config` before the block-reply flush resolves.
