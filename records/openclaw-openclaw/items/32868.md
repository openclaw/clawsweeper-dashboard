## Summary

Keep open. Current main still has the same-channel block-streaming await gap: accepted block replies enqueue dispatcher delivery and return synchronously, while the same-channel path does not await that dispatcher queue before tool execution continues. The closed same-fix PRs were not merged, and the open adjacent PR #65541 covers disabled block streaming rather than this enabled `text_end` same-channel ordering bug.

## What This Changes

Same-channel block streaming with `blockStreamingBreak: "text_end"` should deliver each block reply before the next tool starts instead of batching visible text until the turn ends.

## Best Possible Solution

Make accepted same-channel block delivery awaitable in the reply dispatcher, await that delivery from the same-channel `onBlockReply` path before `text_end` tool-start processing proceeds, and add regression coverage that proves the actual dispatcher `deliver` promise settles before tool execution starts.

## Repair Work Prompt

Fix the same-channel block-streaming delivery ordering bug described in this issue. Current main still has `ReplyDispatcher.sendBlockReply` returning boolean, `createReplyDispatcher` returning `true` immediately after queueing `options.deliver(...)`, and `dispatch-from-config` fire-and-forgetting `dispatcher.sendBlockReply(normalizedPayload)` on the same-channel block path. Implement a narrow core fix that makes accepted same-channel block delivery awaitable and waits for that accepted delivery before `blockStreamingBreak: "text_end"` lets tool execution continue. Preserve dropped-payload `false` behavior, dispatcher ordering, `waitForIdle`, failure counts, `humanDelay`, cross-channel `sendPayloadAsync` behavior, and ACP/direct visible-delivery accounting. Update mocks and types rather than using `any`. Add regression coverage proving the actual dispatcher `deliver` promise settles before tool-start processing continues, not merely that an `onBlockReply` callback promise resolves. Do not change channel-specific plugins or solve adjacent disabled-block-streaming behavior from #65541 in this PR. Add a user-facing changelog fix entry with appropriate reporter credit, because this changes channel delivery behavior.
