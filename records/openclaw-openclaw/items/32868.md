## Summary

Keep open: current main still has the same same-channel block streaming delivery gap, and the canonical duplicate has already been closed back to this issue. The source shows block flushes are awaited before tool-start processing, but the same-channel callback only queues dispatcher delivery and returns before the channel send drains.

## What This Changes

Block streaming in `text_end` mode should deliver same-channel text blocks before the next tool executes.

## Best Possible Solution

Land a narrow dispatcher delivery barrier for same-channel block replies, preserving existing plugin compatibility and abort/timeout behavior while proving `text_end` blocks drain before tool execution continues.

## Reproduction Assessment

Yes. Source inspection on current main shows `handleToolExecutionStart()` awaits block flush plumbing, but the same-channel `onBlockReply` path returns after `dispatcher.sendBlockReply()` queues delivery rather than after the channel send drains.

## Solution Assessment

Yes for the narrow fix direction: make the same-channel block path await the specific dispatcher delivery or a tightly scoped dispatcher idle barrier. The broader open followup-idle PR is not the best shape because it waits in a different lifecycle boundary and has an existing P1 review concern.

Keep the change narrowly scoped to the reply dispatcher/block-reply path: when `blockStreamingBreak: "text_end"` flushes before tool execution, same-channel `onBlockReply` must wait until the queued dispatcher delivery for that block has drained before tool execution continues.
