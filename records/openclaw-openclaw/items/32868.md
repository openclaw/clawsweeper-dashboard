## Summary

Keep open: current main still has the same source-level delivery gap, with same-channel block replies queued through a non-awaitable dispatcher path while the pre-tool flush only waits for the block-reply callback chain. The issue also has independent WhatsApp, Telegram, Discord, and WeCom reports, and the only open related PR is a broader followup-idle patch with a blocking lifecycle review rather than a safe superseding fix.

## What This Changes

The issue reports that same-channel `text_end` block-streaming replies are queued but not delivered to the channel before the next tool executes.

## Best Possible Solution

Land a narrow, abort-aware same-channel delivery barrier so accepted `text_end` block replies drain through dispatcher delivery before tool execution proceeds, while preserving legacy boolean dispatcher compatibility.

## Reproduction Assessment

Yes, source-reproducible: current main flushes before tool execution, but the same-channel callback resolves after queueing dispatcher delivery rather than after the channel send completes. Multiple channel reports support the same timing failure, though this review did not run a live channel repro.

## Solution Assessment

Yes. The best fix is a narrowly scoped same-channel delivery barrier or awaitable accepted block delivery; the broader followup-idle approach in the open related PR is not a safe substitute while it has lifecycle and proof blockers.
