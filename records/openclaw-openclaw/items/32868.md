## Summary

Keep open. Current main still has the same-channel block-streaming await gap: accepted block replies are queued onto the dispatcher send chain but the same-channel callback returns before actual channel delivery settles, while docs and tool-start tests expect `text_end` block output to flush before tool execution proceeds.

## What This Changes

Deliver block-streamed `text_end` replies to the same channel before subsequent tool execution starts.

## Best Possible Solution

Make accepted same-channel block delivery awaitable in the core reply dispatcher and await that delivery from `dispatch-from-config` before tool-start flush processing proceeds, while preserving dropped-payload, routed-channel, ACP tracking, and plugin SDK behavior.

## Reproduction Assessment

Yes. Enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current source shows the callback resolves after enqueue instead of channel delivery, and multiple channel reports confirm the batching symptom.

## Solution Assessment

Yes. The narrow maintainable fix is to expose and await the dispatcher delivery promise for accepted same-channel block replies; a broad `waitForIdle()` can work only if it is carefully scoped around accepted block delivery and dropped-payload semantics.
