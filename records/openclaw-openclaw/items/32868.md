## Summary

Keep open. Current main still has the same-channel block-streaming await gap: the tool-start path flushes the block pipeline, but the same-channel reply callback only queues dispatcher delivery and resolves before the channel send settles.

## What This Changes

The issue asks for same-channel block-streamed text replies to reach the channel before the next tool call executes when `blockStreamingBreak` is `text_end`.

## Best Possible Solution

Make accepted same-channel block delivery awaitable at the existing block-flush boundary while preserving routed-channel behavior, dropped-payload semantics, ACP tracking, and plugin SDK compatibility.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable `blockStreamingDefault: "on"` with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current main flushes the block pipeline but the same-channel dispatcher send still returns before channel delivery settles.

## Solution Assessment

Yes. The narrow maintainable fix is to return or expose an awaitable completion for accepted same-channel `sendBlockReply` delivery, or otherwise await the dispatcher drain for that accepted block, so the existing flush boundary genuinely precedes tool execution.

## Repair Work Prompt

Fix the same-channel block streaming delivery-order bug tracked by this issue. In current main, `handleToolExecutionStart` flushes the block reply pipeline before tool execution, but `dispatch-from-config.ts` calls `dispatcher.sendBlockReply(...)` on the same-channel path without awaiting actual dispatcher delivery; `sendBlockReply` still returns `boolean` while `enqueue` appends `options.deliver(...)` to the internal `sendChain`. Make accepted same-channel block delivery awaitable at the existing `onBlockReply`/pipeline flush boundary, preserving `false`/dropped payload behavior, routed-channel `sendPayloadAsync` behavior, ACP delivered tracking, public plugin SDK compatibility, and dispatcher ordering. Add regression coverage that proves a pending dispatcher delivery promise settles before the next tool execution continues, not merely before `onBlockReply` returns. Do not conflate this with the separate block-streaming-disabled progressive text work in #65541. Add a user-facing `CHANGELOG.md` fix entry if runtime behavior changes.
