## Summary

Keep open. Current main still has the same-channel block-streaming await gap: `text_end` flushes wait for the block pipeline callback, but the same-channel callback queues dispatcher delivery and returns before channel delivery settles.

## What This Changes

The issue reports that same-channel block-streaming replies with `blockStreamingBreak: "text_end"` are not delivered before subsequent tool execution.

## Best Possible Solution

Keep this as the canonical open bug and land a focused reply dispatcher/pipeline fix with regression coverage for same-channel `text_end` delivery before tool execution.

## Reproduction Assessment

Yes, as a source-level reproduction. Enable block streaming with `blockStreamingBreak: "text_end"` on a same-channel turn that emits text before sequential tool calls; current main flushes the block pipeline but the same-channel dispatcher send still resolves before actual channel delivery settles.

## Solution Assessment

Yes. The narrow maintainable fix is to make accepted same-channel block delivery awaitable at the existing flush boundary while preserving routed-channel behavior, dropped-payload semantics, ACP tracking, human-delay ordering, and plugin SDK compatibility.
