## Summary

Keep this PR open: the underlying transcript bug is real and the patch is close, but the current branch is not merge-ready because it still persists synthetic mirrors with a hardcoded model API and lacks after-fix real runtime proof.

## What This Changes

The branch adds an embedded-runner helper plus tests, docs, and changelog text to append delivered `assistantTexts` into the durable session transcript after clean successful attempts.

## Best Possible Solution

Land a rebased patch that keeps the narrow zero-usage transcript mirror, records the actual model API or an explicit mirror discriminator, covers a non-OpenAI provider case, and includes redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main can collect final visible text in `assistantTexts` while the successful attempt path snapshots persisted messages and never appends missing visible assistant text to the session transcript.

## Solution Assessment

No, not yet. The reconciliation boundary is the right narrow fix, but persisting all mirrors as `openai-responses` is not a maintainable durable contract across providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Persist the mirror with the actual model API:** `src/agents/pi-embedded-runner/run/assistant-text-persistence.ts:87`
  - body: The new helper writes every synthetic assistant turn with `api: "openai-responses"` while only provider/model vary. Transport replay treats provider+api+model as identity, so an Anthropic, Ollama, or OpenAI-Codex turn reconciled here is later stored with mismatched API metadata and can take the wrong replay/sanitization path. Thread the actual `params.model.api` or use a deliberate mirror discriminator, and add non-OpenAI coverage.
