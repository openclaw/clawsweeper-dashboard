## Summary

Keep this PR open: it is a relevant implementation for the open WebChat video-upload feature, but the current head still has two blocking correctness issues and only mock/test proof rather than real after-fix WebChat/Gateway behavior proof.

## What This Changes

Adds Control UI/WebChat MP4/M4V/MOV/WebM upload previews, Gateway video validation and native/fallback routing, video input catalog plumbing, Google native video payloads, OpenAI/Anthropic guards, docs, changelog, and tests.

## Best Possible Solution

Carry a single ordered prompt-media sequence through Gateway parsing and dispatch, align the provider-index catalog input contract, then require redacted real WebChat/Gateway proof before maintainer review resumes.

## Reproduction Assessment

Yes for the patch blockers: CI reproduces the provider-index failure, and source inspection shows mixed image/video attachment order is rebuilt incorrectly. Real live provider behavior is not reproduced or proven by the PR evidence.

## Solution Assessment

No, not as submitted. The implementation should keep media ordering stable, settle the catalog input contract consistently, and add real after-fix behavior proof with private details redacted.

  - body: The PR now preserves `audio` and `video` in model catalog normalization, but the provider-index preview test still expects `audio` to be dropped; the latest `checks-node-core-fast-support` run fails on that exact mismatch. A message uploaded as video then image will reach native model delivery as image then video, so carry one ordered media sequence through parsing and dispatch.
  - confidence: 0.9
