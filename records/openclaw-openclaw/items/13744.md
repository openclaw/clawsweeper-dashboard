## Summary

Keep open. Current main has shipped the acquire-timeout and failover-masking parts, but the remaining public stale/max-hold/env policy and the embedded lock lifecycle are still unresolved, with recent operator comments keeping this as the canonical tracker.

## What This Changes

The issue requests configurable session write-lock timeout/stale policy, narrower lock scope, and clearer handling of lock timeouts so they are not reported as model failures.

## Best Possible Solution

Keep this as the canonical session-lock policy tracker, preserve the shipped acquire-timeout/failover fixes, and finish an agreed public stale/max-hold/env policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, at source level. Current main still holds the transcript lock across the embedded prompt lifecycle, and another transcript writer can wait only up to the configured acquire timeout before receiving `SessionWriteLockTimeoutError`.

## Solution Assessment

No. Current main is a useful partial solution, but it only exposes acquire timeout and does not settle the public stale/max-hold/env policy or the broader lock-scope design.
