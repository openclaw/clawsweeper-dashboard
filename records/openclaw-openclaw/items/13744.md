## Summary

Keep open. Current main shipped the acquire-timeout part and failover masking fixes, but stale/max-hold/env policy and the broader lock-lifecycle request are still unresolved; the issue remains the canonical thread with recent operator evidence.

## What This Changes

Request to make session write-lock timeout/stale behavior configurable and reduce the lock lifetime so same-session contention is handled as busy/retry behavior instead of model failover.

## Best Possible Solution

Keep this as the canonical tracker, preserve the shipped acquire-timeout and failover fixes, then land an agreed public lock policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes. A high-confidence source-level path remains: a same-session writer can hold the transcript lock through `activeSession.prompt(...)`, and a later writer waits only until the configured acquire timeout before surfacing a busy-session lock error.

## Solution Assessment

No. Current main is a useful partial solution, but it does not finish the stale/max-hold/env policy or remove the model-run-sized critical section; a maintainable fix needs an explicit transcript-lock lifecycle design.
