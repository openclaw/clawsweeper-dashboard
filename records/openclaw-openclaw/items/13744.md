## Summary

Keep open. Current main has partial mitigation for session-lock contention and failover masking, but it still lacks a public timeout/stale configuration surface and the embedded attempt path still holds the session write lock through model execution before releasing it during teardown.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, narrower lock scope or a per-session writer queue, and error handling that does not report lock contention as model failure.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema/types/help/docs and wire it consistently into embedded attempts plus transcript/session-store mutation paths, while preserving typed lock-timeout failover suppression and staging any deeper lock-scope or single-writer redesign behind focused runner/session tests.

## Reproduction Assessment

Yes. The discussion provides fresh operator reproductions on v2026.4.26, and current-main source inspection shows the embedded path still uses the 10s acquire default and reaches `activeSession.prompt(...)` before releasing the session lock.

## Solution Assessment

Unclear. A config-first timeout/stale knob is a narrow practical mitigation, but the exact public config/env contract and safe lock-lifecycle boundary need maintainer choice; the broader per-session writer queue remains a larger design follow-up.
