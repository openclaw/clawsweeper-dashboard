## Summary

Keep open. Current main partially addresses the report by exposing `session.writeLock.acquireTimeoutMs` and suppressing session-lock timeouts from model failover, but the remaining stale/max-hold/env policy and the broader embedded lock lifecycle are still unresolved.

## What This Changes

Request configurable session write-lock behavior and a narrower lock scope so same-session contention does not surface as model failure or dropped work.

## Best Possible Solution

Keep this as the canonical session-lock policy tracker, preserve the shipped acquire-timeout/failover fixes, and finish an agreed stale/max-hold/env policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes. A high-confidence source-level path remains: one same-session writer can hold the transcript lock through `activeSession.prompt(...)`, while another writer waits only up to the configured acquire timeout before receiving a busy-session lock error.

## Solution Assessment

No. Current main is a useful partial fix, but completing the issue needs an explicit maintainer decision on public lock policy and a safer transcript-lock lifecycle design.
