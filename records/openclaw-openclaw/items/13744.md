## Summary

Keep open. Current main partially satisfies the report by adding `session.writeLock.acquireTimeoutMs` and typed lock-timeout failover handling, but stale/max-hold/env policy and the embedded lock lifecycle remain unresolved.

## What This Changes

Request configurable session write-lock timeout/stale policy, narrower transcript lock scope, and busy-session handling that does not surface as model failure.

## Best Possible Solution

Keep this as the canonical session-lock policy tracker, preserve the shipped acquire-timeout and failover fixes, and finish an agreed stale/max-hold/env policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes. A high-confidence source-level path remains: one same-session writer can hold the transcript lock through `activeSession.prompt(...)`, while another writer waits only up to the configured acquire timeout before receiving `SessionWriteLockTimeoutError`.

## Solution Assessment

No. Current main is a useful partial fix, but resolving the issue well requires explicit policy for public stale/max-hold/env knobs and a safer transcript-lock lifecycle design.
