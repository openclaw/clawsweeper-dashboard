## Summary

Keep open. Current main and v2026.5.2 address the acquire-timeout configuration and failover-masking parts, but the broader stale/max-hold/env policy and embedded lock-scope request are still unresolved.

## What This Changes

Request to make session transcript write-lock timeout/stale policy configurable, narrow embedded lock holding around transcript mutations, and report lock contention separately from model failures.

## Best Possible Solution

Keep the shipped acquire-timeout and failover improvements, then make an explicit maintainer decision on the remaining lock policy and whether to narrow the embedded writer lifecycle in this issue or split it into a canonical follow-up.

Yes. Operator reports show the lock-timeout pressure on released builds, and current-main source inspection shows the embedded path still holds the session lock through activeSession.prompt before end-of-attempt cleanup.

## Solution Assessment

No for closing this item. The current implementation is a useful narrow mitigation, but it does not finish stale/max-hold/env policy or the requested critical-section-only/single-writer lifecycle.
