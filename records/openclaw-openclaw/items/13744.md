## Summary

Keep open. Current main has fixed the biggest config-knob subpart with `session.writeLock.acquireTimeoutMs` and 60s default acquisition waits, and it also suppresses session-lock timeouts from model failover, but this issue remains the canonical tracker for stale/max-hold configuration and the broader embedded-run lock lifecycle/single-writer decision.

## What This Changes

The issue requests configurable session write-lock timeout/stale behavior, a narrower or queued write-lock lifecycle, and clearer non-model error handling for lock contention.

## Best Possible Solution

Keep this as the canonical session write-lock lifecycle tracker, preserve the shipped acquire-timeout config, and have owners decide whether to add stale/max-hold config separately from a deeper single-writer or narrower-lock refactor.

## Reproduction Assessment

Yes at source level for the remaining gap: current main still holds the embedded transcript lock across `activeSession.prompt(...)` and still lacks public stale/max-hold config. I did not live-run a gateway contention scenario against current main after the 60s acquire-timeout mitigation.

## Solution Assessment

No current-main solution is complete. The acquire-timeout and failover fixes are good partials, but the remaining stale/max-hold API and lock lifecycle design should be handled with owner review rather than closed as implemented.
