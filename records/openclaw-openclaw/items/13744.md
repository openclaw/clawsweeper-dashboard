## Summary

Keep open. Current main has shipped an acquire-timeout config and failover-classification mitigation, but it still lacks the requested stale/max-hold/env policy and the embedded lock still spans the model run, with recent maintainer/operator discussion treating this as the canonical remaining item.

## What This Changes

Request to make session write-lock policy configurable, narrow embedded lock lifetime, and surface lock contention separately from provider/model failure.

## Best Possible Solution

Keep this as the canonical tracker, preserve the shipped acquire-timeout and failover fixes, then land an agreed public lock policy plus a designed narrower or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, source-level. Current main still acquires the session lock before the embedded prompt path and releases it in final cleanup, so a long same-session run can keep later writers waiting; I did not run a live concurrent Gateway repro in this read-only review.

## Solution Assessment

No for closing this issue. Current main is a useful partial solution, but it does not complete stale/max-hold/env configurability or the requested narrower lock lifecycle.
