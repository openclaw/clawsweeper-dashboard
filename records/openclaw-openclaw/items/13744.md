## Summary

Keep open. Current main has partial mitigations for acquire-timeout configuration and failover classification, but the remaining stale/max-hold/env policy and the requested narrower or single-writer transcript lifecycle are still unresolved and treated as the canonical thread in recent discussion.

## What This Changes

Request to expose session write-lock timeout/stale policy through config or env, narrow the embedded lock hold window, and report lock contention as a busy/retryable condition instead of model failover.

## Best Possible Solution

Keep this as the canonical tracker, preserve the shipped acquire-timeout and failover fixes, then land an agreed public lock policy plus a designed narrower or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, source-level. Current main acquires the session lock before the embedded prompt path and releases it in final cleanup, so a long same-session run can keep later writers waiting; I did not run a live concurrent Gateway repro in this read-only review.

## Solution Assessment

No for closing this issue. Current main is a useful partial solution, but the best fix still needs explicit maintainer decisions for stale/max-hold/env policy and the lock lifecycle boundary.
