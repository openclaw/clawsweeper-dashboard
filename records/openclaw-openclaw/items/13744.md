## Summary

Keep open. Current main covers the acquire-timeout configuration and lock-timeout failover masking, but stale/max-hold/env policy and the embedded transcript lock lifecycle remain unresolved while recent discussion still reports operator-visible contention.

## What This Changes

Make session transcript write-lock timeout/stale policy configurable and narrow the embedded run lock lifecycle so lock contention does not surface as model failure.

## Best Possible Solution

Keep this as the canonical session-lock policy tracker, preserve the shipped acquire-timeout and failover fixes, and finish an agreed stale/max-hold/env policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, at source level. One same-session writer can hold `sessionLock` through `activeSession.prompt(...)` and cleanup, while another writer waits only up to the configured acquire timeout before receiving `SessionWriteLockTimeoutError`.

## Solution Assessment

No. Current main is a useful partial solution, but fully solving the report needs explicit public policy for stale/max-hold/env knobs and a safer transcript-lock lifecycle design.
