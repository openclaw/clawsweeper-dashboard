## Summary

Keep open. Current main has implemented the acquire-timeout configuration and the failover-masking fix, but the issue's remaining stale/max-hold/env policy and broader embedded-run lock lifecycle are still unresolved, with recent discussion treating this as the canonical tracker.

## What This Changes

The issue requests configurable session write-lock timeout/stale policy, narrower lock scope, and clearer lock-timeout handling so concurrent session writes do not surface as model failover.

## Best Possible Solution

Keep this as the canonical session-lock policy tracker, preserve the shipped acquire-timeout/failover fixes, and finish an agreed public stale/max-hold/env policy plus a designed narrower critical section or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, at source level. Current main still holds the session write lock through `activeSession.prompt(...)`, so another writer can wait only up to `session.writeLock.acquireTimeoutMs` before receiving `SessionWriteLockTimeoutError`.

## Solution Assessment

No. Current main is a useful partial solution, but the remaining public stale/max-hold/env policy and the larger lock-scope design still need a maintainer decision.
