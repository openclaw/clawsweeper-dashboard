## Summary

Keep open. Current main and the 2026.5.2 changelog show the acquire-timeout part is implemented, but the requested stale/max-hold/env policy and the broad embedded lock lifetime remain unresolved.

## What This Changes

The issue requests configurable session write-lock timeout and stale behavior, narrower lock lifetime around session mutation, and lock-timeout handling that does not look like model/provider failure.

## Best Possible Solution

Keep the shipped acquire-timeout and failover fixes, then decide whether to split the remaining stale/max-hold/env knobs from a deeper single-writer or narrower-lock lifecycle change.

## Reproduction Assessment

Yes. Operator comments show released-build lock contention, and current-main source inspection shows the remaining request: only acquireTimeoutMs is public while the embedded attempt still releases the session lock at end-of-attempt cleanup after prompt execution.

## Solution Assessment

No for closing this item. The current implementation is a useful partial mitigation, but it does not finish the requested stale/max-hold/env configurability or the narrower lock lifecycle.
