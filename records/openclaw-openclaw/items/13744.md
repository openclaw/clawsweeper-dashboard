## Summary

Keep open. Current main has shipped the acquire-timeout config and failover-classification pieces, but the requested stale/max-hold/env policy and a truly narrower or queued transcript lifecycle remain unresolved and active discussion treats this as the canonical tracker.

## What This Changes

Request to expose session write-lock timeout/stale policy and reduce lock scope so same-session concurrent traffic does not fail as model failover.

## Best Possible Solution

Keep this as the canonical tracker, preserve the shipped acquire-timeout and failover fixes, then land an agreed public lock policy plus a designed narrower or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, source-level. Current main still holds the session lock through activeSession.prompt and releases it during late attempt cleanup, so a long same-session run can block later writers until the configured acquire timeout.

## Solution Assessment

No for closing this issue. The current implementation is a useful partial mitigation, but it does not complete stale/max-hold/env configurability or the requested narrower lock lifecycle.
