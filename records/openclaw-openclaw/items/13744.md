## Summary

Keep open. Current main has partial mitigations for session-lock contention and failover masking, but the requested public lock timeout/stale configuration and truly narrower embedded lock lifecycle are still missing, and the discussion has fresh operator evidence from v2026.4.26 and May 2, 2026.

## What This Changes

Request configurable session write-lock timeout/stale/max-hold behavior, narrower lock scope around transcript mutations, and clearer handling of lock timeouts instead of model failover errors.

## Best Possible Solution

Add a supported session write-lock configuration contract, wire it through embedded runs and session-store/transcript mutation paths, keep typed lock-timeout failover suppression, and stage any deeper single-writer or per-file lock redesign with focused runner/session tests.

## Reproduction Assessment

Yes. The discussion provides fresh v2026.4.26 and May 2 operator evidence, and source inspection shows the embedded path still uses the 10s acquire default while the lock is held through prompt execution.

## Solution Assessment

Unclear. A config-first fix looks like a narrow operational unblock, but maintainers still need to choose the public config/env contract and decide how far to narrow the embedded lock lifecycle now versus a later single-writer refactor.
