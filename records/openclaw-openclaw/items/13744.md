## Summary

Keep open. Current main has useful partial fixes for acquire-timeout configuration and failover classification, but the remaining public stale/max-hold/env policy and the requested narrower or single-writer transcript lifecycle are still unresolved; recent discussion treats this as the canonical tracker.

## What This Changes

The issue requests configurable session write-lock timeout/stale behavior, narrower lock scope around session writes, and clearer handling of lock timeouts so they are not reported as model failures.

## Best Possible Solution

Keep this as the canonical tracker, preserve the shipped acquire-timeout and failover fixes, then land an agreed public lock policy plus a designed narrower or single-writer transcript lifecycle.

## Reproduction Assessment

Yes, source-level. Current main acquires the session write lock before embedded session prep, calls `activeSession.prompt(...)` while still inside that locked lifecycle, and releases during final cleanup, so a long same-session run can keep later writers waiting until `session.writeLock.acquireTimeoutMs`.

## Solution Assessment

No for closing this issue. Current main is a useful partial solution, but the best fix still needs explicit maintainer decisions for stale/max-hold/env policy and the embedded lock lifecycle boundary.
