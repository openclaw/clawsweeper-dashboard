## Summary

Keep open. Current main has a strong partial mitigation for the hardcoded 10s acquire timeout and failover masking, but the requested narrower lock lifecycle and stale/max-hold/env policy are not fully implemented.

## What This Changes

Request to expose session write-lock timing policy, narrow the embedded-run lock lifecycle, and handle lock contention separately from model/provider failures.

## Best Possible Solution

Keep the current acquire-timeout and failover improvements, then either finish a documented lock policy plus narrower writer lifecycle here or split the deeper lock-scope work into a dedicated canonical follow-up.

## Reproduction Assessment

Yes. Operator reports reproduce the lock-timeout pressure on released builds, and source inspection of current main shows the embedded path can still hold the session lock through `activeSession.prompt(...)`.

## Solution Assessment

Unclear. The current acquire-timeout config is the right narrow mitigation, but it does not fully solve stale/max-hold policy or the broader lock-scope/queue behavior requested here.
