## Summary

Current main and release v2026.6.5 still synthesize `<media:audio>` for blank Signal audio sends, so the user-visible request remains unresolved. The proposed global config is not automatically the best fix because iMessage now avoids visible text through an attachment-specific transport while retaining an internal echo marker; maintainers should choose whether Signal needs the same channel-owned treatment or whether a genuine cross-channel preference warrants a new config surface.

Likely related people: `steipete` (high confidence) for the messages-config and plugin runtime boundary, `omarshahine` (high confidence) for the recent iMessage media-only transport design, and `Hua688` (medium confidence) for recent Signal send-path work.

## What This Changes

Requests an opt-in setting to suppress visible media placeholder text for media-only outbound messages while preserving current fallback behavior by default.

## Best Possible Solution

First make Signal media-only delivery use a transport-valid non-visible body or attachment-only operation while keeping placeholder data internal for echo/dedupe; add a documented default-preserving global setting only if maintainers confirm users need policy control across multiple channels after that audit.

## Reproduction Assessment

Yes at source level: a blank Signal media send with an `audio/*` attachment deterministically enters `extensions/signal/src/send.ts` and becomes `<media:audio>` before RPC delivery. A real Signal client run is still needed to prove the current API-mode display and any replacement body.

## Solution Assessment

No, the proposed global config is not yet proven to be the best solution. The narrower maintainable direction is to adopt iMessage's channel-owned separation of visible transport text and internal echo state where Signal's dependency contract permits, then decide whether any cross-channel preference remains.
