## Summary

Keep open: current main still has no `messages` suppression config and Signal still synthesizes `<media:${kind}>` for blank media-only sends. A recent merged iMessage fix covers part of the visible-placeholder problem, but it does not satisfy the requested cross-channel opt-in config surface.

## What This Changes

Request adds an opt-in `messages` config option to suppress synthetic `<media:audio>` / `<media:...>` fallback text for media-only messaging sends.

## Best Possible Solution

Approve a default-off `messages` media-placeholder policy, keep iMessage's attachment-only behavior, wire Signal and any remaining outbound placeholder paths through the config, update schema/docs/generated config artifacts, and add focused regressions.

## Reproduction Assessment

Yes, for the current behavior by source inspection. Blank Signal media sends with recognized audio/media MIME types still synthesize `<media:${kind}>`, and the strict `messages` schema has no accepted suppression key.

## Solution Assessment

Yes, directionally. A default-off `messages` option is the narrow maintainable approach, but the final option shape and Signal transport policy should be approved before implementation.
