## Summary

Keep open: current main still lacks a `messages` suppression option and Signal still synthesizes visible media placeholder text for blank media-only sends. The iMessage-specific fix has shipped, but that only covers one channel and does not satisfy this issue's cross-channel, opt-in config request.

## What This Changes

The issue requests a default-off messages config option to suppress visible media placeholder text such as `<media:audio>` for media-only channel sends.

## Best Possible Solution

Approve a default-off `messages` media-placeholder policy, keep the current iMessage attachment-only visible behavior, wire Signal and any remaining outbound placeholder paths through the chosen config, and update schema, docs, metadata, and focused tests together.

## Reproduction Assessment

Yes, by source inspection. Blank Signal media sends with recognized audio/media MIME types still synthesize `<media:${kind}>`, and the strict `messages` schema has no accepted suppression key.

## Solution Assessment

Yes, directionally. A default-off `messages` option is the narrow maintainable path, but the exact option shape and Signal transport fallback policy need maintainer approval before implementation.
