## Summary

Keep open: current main still lacks any `messages` media-placeholder suppression option, and Signal still synthesizes `<media:audio>` for blank audio media sends. The merged iMessage-only fix partially reduces the visible placeholder problem, but it does not resolve the canonical cross-channel config/product request.

## What This Changes

The issue requests a default-off configuration option to suppress visible `<media:audio>` placeholder text for media-only messaging sends.

## Best Possible Solution

Approve and document a default-off media-placeholder policy under `messages`, keep the current iMessage no-visible-text behavior, and wire Signal plus any remaining outbound placeholder paths through schema, help/docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection. Captionless Signal media replies pass empty text plus media into `sendMessageSignal`, which synthesizes `<media:${kind}>`; audio MIME types therefore become `<media:audio>`.

## Solution Assessment

Yes, directionally. A default-off `messages` option is a maintainable shape, but the exact option contract and Signal fallback behavior need maintainer approval.
