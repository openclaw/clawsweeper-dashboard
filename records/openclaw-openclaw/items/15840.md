## Summary

Keep open. Current main and v2026.6.6 still synthesize `<media:audio>` for blank Signal audio/media sends, no `messages` suppression config exists, and the related implementation pull requests are closed unmerged; iMessage now solves the visible sibling symptom through channel-owned attachment delivery, so the remaining decision is whether Signal needs the same treatment or a global config option.

## What This Changes

The issue requests a default-preserving config option to suppress visible media placeholder text such as `<media:audio>` on media-only messaging sends.

## Best Possible Solution

Resolve Signal media-only delivery with a non-visible, transport-valid channel behavior if possible, and add a documented default-off global setting only if maintainers confirm cross-channel policy control is still needed.

## Reproduction Assessment

Yes, at source level: a blank Signal media send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. A live Signal client run is still needed before choosing the replacement transport behavior.

## Solution Assessment

No, the requested global config option is not yet proven to be the best solution. iMessage now avoids visible placeholder text through a channel-owned attachment path, so Signal should be audited against its transport contract before adding a cross-channel preference.
