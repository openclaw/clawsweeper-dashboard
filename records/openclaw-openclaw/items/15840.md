## Summary

Keep open: current main still lacks a `messages` media-placeholder suppression option, and the remaining Signal outbound path still synthesizes visible `<media:audio>` text for captionless audio/media sends. Recent iMessage work partially fixed visible outbound text there, but it does not resolve the canonical cross-channel config/product request.

## What This Changes

The issue requests a default-off `messages` config option to suppress synthesized `<media:audio>` / `<media:...>` fallback text for media-only voice or audio sends.

## Best Possible Solution

Approve and document a default-off media-placeholder policy under `messages`, keep existing fallback behavior by default, wire Signal and any remaining outbound placeholder paths through schema/help/docs, and preserve iMessage’s internal echo behavior with focused tests.

## Reproduction Assessment

Yes, by source inspection. Captionless Signal media replies pass empty text plus `mediaUrl` into `sendMessageSignal`, which maps `audio/*` to `audio` and synthesizes `<media:audio>` before dispatch.

## Solution Assessment

Yes, directionally. A default-off `messages` option is a maintainable shape, but the exact option contract and Signal fallback behavior need maintainer approval before this should be automated.
