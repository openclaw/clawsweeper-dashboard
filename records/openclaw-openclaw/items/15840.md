## Summary

Keep open. Current main and the latest release still synthesize visible Signal `<media:audio>` text for blank audio/media sends, and the requested `messages` suppression config does not exist; the merged iMessage sibling fix narrows the remaining decision to Signal transport behavior versus an approved global config surface.

## What This Changes

Request a default-off `messages` configuration option to suppress automatic media placeholder text such as `<media:audio>` for media-only voice/audio sends.

## Best Possible Solution

Resolve Signal media-only delivery with a transport-valid non-visible body or attachment-only send if possible; add a documented default-off global `messages` setting only if maintainers confirm cross-channel policy control is needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. I did not run a live Signal client, so transport-compatible replacement behavior still needs proof.

## Solution Assessment

No, the requested global config option is not yet proven to be the best solution. iMessage now avoids visible placeholder text through channel-owned attachment delivery, so Signal should be audited against its transport contract before adding a cross-channel preference.
