## Label Justifications

- `P3`: This is a shipped but bounded presentation issue affecting media-only Signal messages without data-loss, security, or availability impact.
- `impact:other`: Signal recipients can see OpenClaw's internal media marker as an unwanted caption on attachment-only messages.

## Summary

Current main and release `v2026.6.5` still expose `<media:audio>` on blank Signal attachment sends, while the iMessage case was fixed separately; this remains a valid but narrowed Signal behavior decision, not cleanup-ready.

## What This Changes

Requests an opt-in setting to suppress visible media placeholder text on media-only channel sends.

## Best Possible Solution

Keep media markers internal and implement a Signal-plugin-owned attachment-only or non-visible body strategy proven against both native signal-cli and the supported container backend; add a global config only if additional current channels demonstrate the same policy need.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.5` deterministically convert blank Signal audio sends into `<media:audio>` before RPC delivery. This read-only review did not perform a fresh live Signal send.

## Solution Assessment

No, the proposed global config is not currently the best-supported fix; the merged iMessage correction and upstream native Signal contract favor a narrower Signal-owned solution, subject to container-backend proof.
