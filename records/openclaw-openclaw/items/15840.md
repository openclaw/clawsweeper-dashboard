## Summary

Keep open: current main and v2026.5.12 still lack the requested `messages` config option, and Signal still synthesizes `<media:audio>` / `<media:${kind}>` as the outbound body for blank media-only sends. The merged iMessage-only fix removes visible iMessage placeholder text, but it does not resolve the canonical cross-channel config/product request.

## What This Changes

The issue requests a default-off configuration option to suppress generated media placeholder text such as `<media:audio>` for media-only channel sends.

## Best Possible Solution

Approve and document a default-off media-placeholder policy under `messages`, keep iMessage's current empty-visible-text behavior, and wire Signal plus any remaining outbound placeholder paths through config schema, help/labels, docs, and focused regression tests.

## Reproduction Assessment

Yes, by source inspection. Captionless Signal media replies pass empty text plus media into `sendMessageSignal`, which synthesizes `<media:${kind}>`; audio MIME types therefore become `<media:audio>`.

## Solution Assessment

Yes, directionally. A default-off `messages` option is a maintainable shape, but the final option name, granularity, and Signal fallback behavior need maintainer approval.
