## Summary

Keep open: current main still lacks a `messages` media-placeholder suppression option, and Signal still synthesizes visible `<media:audio>` text for captionless audio attachments. The iMessage visible-send case has since been fixed, but the canonical cross-channel config request is not implemented and the related implementation PRs are closed unmerged.

## What This Changes

The issue requests a default-off `messages` config option to suppress visible media placeholder text such as `<media:audio>` for audio-only messaging sends.

## Best Possible Solution

Approve and document a default-off `messages` media-placeholder policy, preserve current fallback behavior by default, wire Signal through schema/help/docs/tests, and keep iMessage’s visible-delivery fix intact.

## Reproduction Assessment

Yes, by source inspection. Captionless Signal media replies pass empty text plus `mediaUrl` into `sendMessageSignal`, audio MIME maps to `audio`, and the send path synthesizes `<media:audio>` before dispatch.

## Solution Assessment

Yes, directionally. A default-off `messages` option is maintainable, but maintainers need to choose the exact config contract and Signal hidden-body/fallback semantics before implementation.
