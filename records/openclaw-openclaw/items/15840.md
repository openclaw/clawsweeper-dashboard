## Summary

Keep this open. Current main and the latest release still have the Signal and iMessage media-placeholder behavior, the requested messages config key is absent, and the directly linked implementation PRs are closed unmerged.

## What This Changes

The issue requests an opt-in messages configuration setting to suppress visible media fallback text such as `<media:audio>` for media-only channel sends.

## Best Possible Solution

Keep this open as the canonical maintainer product/config request and decide a default-off messages option that preserves accessibility fallback behavior while suppressing visible channel placeholders where safe.

## Reproduction Assessment

Yes, at source level: current Signal and iMessage code still constructs media placeholder text or echo text for blank media-only sends, and audio MIME classifies to `audio`. No live transport reproduction was run.

## Solution Assessment

Unclear as an implementation choice: an opt-in config is plausible and default-preserving, but maintainers need to approve the config shape and channel-specific fallback behavior before a fix PR.
