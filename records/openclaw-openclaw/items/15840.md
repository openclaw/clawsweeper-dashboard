## Summary

Keep open. Current main still synthesizes visible media placeholders for blank media-only Signal and iMessage sends, the shared audio MIME classifier still maps audio attachments to the `audio` placeholder kind, and the strict `messages` config surface still has no suppression option; the related implementation PRs are closed unmerged.

## What This Changes

The issue requests a default-off `messages` config option to suppress visible media fallback text such as `<media:audio>` for media-only messaging sends.

## Best Possible Solution

Implement a documented, default-off `messages` setting that preserves existing placeholders by default while letting Signal and iMessage media-only outbound paths suppress visible placeholder text when enabled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send a media-only Signal or iMessage reply with an empty caption and an `audio/*` or other known media MIME type, and current main constructs a visible `<media:...>` body before delivery.

## Solution Assessment

Yes. A default-off config option wired through the shared messages config and the channel-owned send paths is the narrowest maintainable solution; the Signal transport detail should be handled without changing the default accessibility/fallback behavior.
