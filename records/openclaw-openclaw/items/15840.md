## Summary

Keep open. Current main and v2026.6.10 still synthesize the Signal media-only fallback body from the attachment MIME kind, so audio attachments still become visible `<media:audio>` text, and the requested `messages` suppression config is not present.

## What This Changes

The issue requests a default-off `messages` config option to suppress synthetic media placeholder text for audio or other media-only channel sends.

## Best Possible Solution

Resolve the remaining Signal media-only audio visibility with a transport-valid hidden-body or attachment-only path, and add a documented default-off global `messages` setting only if maintainers confirm cross-channel policy control is needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. I did not run a live Signal client, so the transport-compatible replacement still needs proof.

## Solution Assessment

Unclear. A global default-off config option may be valid, but the iMessage sibling fix shows channel-owned attachment delivery can solve part of the family, so Signal should be audited against its transport contract before adding cross-channel policy surface.
