## Summary

Keep open. Current main still synthesizes visible media placeholder text for blank media-only Signal and iMessage sends, audio MIME types still map to the `audio` placeholder kind, and the strict `messages` config surface still has no suppression option. The related implementation PRs are closed unmerged, so the requested behavior remains unimplemented.

## What This Changes

Add a default-off `messages` configuration option to suppress visible `<media:audio>` and other media placeholder text for media-only outbound messages while preserving current fallback behavior by default.

## Best Possible Solution

Keep this as the canonical request and implement a documented, default-off `messages` option that is accepted by the strict config schema, represented in generated config metadata/help/docs, and honored by Signal and iMessage media-only outbound paths without changing default accessibility/fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send a media-only Signal or iMessage reply with an empty caption and an `audio/*` or image MIME type, and current main constructs `<media:audio>` or `<media:image>` before delivery.

## Solution Assessment

Yes. A default-off config option wired through the shared messages config and the channel-owned send paths is the narrowest maintainable solution; the Signal implementation detail should account for any transport requirement for non-empty attachment text.
