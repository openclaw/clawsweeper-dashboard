## Summary

Keep open. Current main and the latest release still synthesize visible Signal media placeholder text for blank audio/media sends, and the requested `messages` suppression config is still absent; the merged iMessage fix narrows but does not close the remaining Signal/config product decision.

## What This Changes

The issue requests a default-off configuration option to suppress automatic `<media:audio>` or other media placeholder text for media-only channel sends.

## Best Possible Solution

Resolve the remaining Signal media-only audio visibility with a transport-valid hidden-body or attachment-only path, and add a documented default-off global `messages` setting only if maintainers confirm cross-channel policy control is needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. I did not run a live Signal client, so the transport-compatible replacement still needs proof.

## Solution Assessment

No, the requested global config option is not yet proven to be the best solution. iMessage now avoids visible placeholder text through channel-owned attachment delivery, so Signal should be audited against its transport contract before adding a cross-channel preference.
