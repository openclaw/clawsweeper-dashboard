## Summary

Keep open: current main still has no `messages` media-placeholder suppression option, and the Signal media-only send path still synthesizes visible `<media:audio>` text from captionless audio attachments. iMessage has a later partial fix for visible outbound placeholder text, but that does not resolve the canonical cross-channel config/product request.

## What This Changes

The issue requests a default-off configuration option to suppress automatic `<media:audio>` or related media placeholder text for media-only messaging sends.

## Best Possible Solution

Approve and document a default-off `messages` media-placeholder policy, preserve current fallback behavior by default, wire Signal and any remaining outbound placeholder paths through schema/help/docs, and keep iMessage’s internal echo placeholder separate from visible delivery.

## Reproduction Assessment

Yes, by source inspection. Captionless Signal media replies pass empty text plus `mediaUrl` into `sendMessageSignal`, audio MIME maps to `audio`, and the send path synthesizes `<media:audio>` before dispatch.

## Solution Assessment

Yes, directionally. A default-off `messages` option is a maintainable direction, but the exact config contract and Signal fallback semantics should be approved before implementation.
