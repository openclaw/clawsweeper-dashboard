## Summary

Keep open. Current main and v2026.6.10 still synthesize visible Signal media-placeholder text for blank audio/media sends, while the requested strict `messages` config option is still absent and needs a maintainer product/config decision.

## What This Changes

Request a default-off `messages` option to suppress synthetic `<media:audio>` / `<media:...>` text for media-only voice or audio sends while preserving fallback behavior by default.

## Best Possible Solution

Resolve the remaining Signal media-only audio visibility with a transport-valid hidden-body or attachment-only path, and add a documented default-off `messages` setting only if maintainers confirm cross-channel policy control is needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment maps to `audio` and then `sendMessageSignal` sets the RPC body to `<media:audio>`. I did not run a live Signal client, so the transport-compatible replacement still needs proof.

## Solution Assessment

Unclear. The requested global config option is plausible, but iMessage now solved its sibling issue channel-locally, so maintainers should first decide whether the remaining fix is Signal-specific delivery behavior or a cross-channel messages policy.
