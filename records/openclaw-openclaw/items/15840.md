## Summary

Keep open. Current main and the latest release still synthesize visible Signal media-placeholder text for blank audio/media sends, while the requested suppression setting remains absent from the strict messages config surface; iMessage is now only a partial sibling fix, not a full implementation of this request.

## What This Changes

The issue requests a default-off messages configuration option to suppress visible media placeholder text such as `<media:audio>` for media-only outbound messages.

## Best Possible Solution

Resolve the remaining Signal media-only audio visibility with a transport-valid hidden-body or attachment-only path, and add a documented default-off global `messages` setting only if maintainers confirm cross-channel policy control is needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. I did not run a live Signal client in this read-only cleanup pass, so the transport-compatible replacement still needs proof.

## Solution Assessment

Unclear. The requested global config option may be useful, but iMessage now solved the sibling problem with channel-owned attachment delivery, so Signal transport behavior should be audited before adding a cross-channel preference.
