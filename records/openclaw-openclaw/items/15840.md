## Summary

Keep open. Current main and the latest release still synthesize `<media:audio>` for blank Signal audio/media sends and have no `messages` suppression config; the merged iMessage work only removed the sibling visible iMessage placeholder, so a maintainer still needs to choose the remaining Signal/global policy.

## What This Changes

Request a default-off `messages` config option or equivalent behavior to suppress visible media placeholder text for audio/voice media-only sends.

## Best Possible Solution

Resolve Signal media-only delivery with a non-visible, transport-valid channel behavior if possible; add a documented default-off global setting only if maintainers confirm cross-channel policy control is still needed.

## Reproduction Assessment

Yes, at source level: a blank Signal send with an `audio/*` attachment deterministically becomes `<media:audio>` before RPC delivery. I did not run a live Signal client, so transport-compatible replacement behavior still needs proof.

## Solution Assessment

No, the requested global config option is not yet proven to be the best solution. iMessage now avoids visible placeholder text through channel-owned attachment delivery, so Signal should be audited against its transport contract before adding a cross-channel preference.
