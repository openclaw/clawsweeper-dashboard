## Summary

Keep open: the report is concrete and current source still has spawn paths that can throw `EBADF` without a safe retry, especially service-managed child-adapter calls with no fallback and bundled MCP stdio startup. Current exec-host non-PTY calls already use `pipe-closed`, so the repair should target the remaining shared spawn surfaces rather than assume every exec path still uses inherited stdin.

## What This Changes

The issue reports gateway `spawn EBADF` failures under very high file descriptor counts and requests safer stdin/fallback handling for child process spawns plus investigation of the underlying fd leak.

## Best Possible Solution

Centralize safe spawn behavior for service-managed gateways: use pipe/closed stdin where no interactive input is needed, retry `EBADF` with a non-detached pipe fallback, and cover MCP stdio startup plus supervisor child spawns without adding config.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. The issue includes concrete logs and current source inspection shows retry gaps that are source-reproducible, but the high-fd launchd setup was not recreated.

## Solution Assessment

Unclear as written: changing only `createChildAdapter` would miss the direct MCP stdio spawn path the issue also names. The narrow maintainable fix is to apply the safe `EBADF` fallback pattern consistently across the supervisor child adapter and MCP stdio startup, with regression tests.
