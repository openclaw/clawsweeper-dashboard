## Summary

Keep open: current main has mitigated some related FD-pressure paths, and host exec calls now request piped stdin explicitly, but the generic child adapter still defaults no-input spawns to inherited stdin and its EBADF retry only disables detach. The linked fixing PR is closed unmerged, so the requested spawn hardening is still not implemented on main.

## What This Changes

The issue asks to harden gateway child-process spawning under high file-descriptor pressure by avoiding inherited stdin in service mode and adding a pipe-stdin EBADF fallback.

## Best Possible Solution

Add a narrow child-adapter hardening patch with regression tests for non-TTY/service defaults and an EBADF retry that switches to fully piped stdio without changing interactive stdin behavior.

## Reproduction Assessment

No high-confidence live reproduction was performed in this read-only review. The issue body provides concrete macOS launchd logs, and current source still shows the inherited-stdin/no-stdio-changing-fallback path that matches the proposed failure mode.

## Solution Assessment

Mostly yes: the requested child-adapter hardening is the narrowest maintainable fix, as long as it preserves explicit interactive stdin behavior and adds focused EBADF fallback coverage.
