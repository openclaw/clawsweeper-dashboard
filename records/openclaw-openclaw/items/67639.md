## What This Changes

This PR threads status scan `sourceConfig` into the memory status snapshot/runtime bootstrap path, adds focused status and memory-runtime tests, and includes unrelated provider/channel type cleanup.

## Best Possible Solution

Land a narrow rebased fix that carries `overview.sourceConfig` as internal `activationSourceConfig` through the status memory probe into runtime bootstrap while preserving current main's status fast paths, scoped memory-slot loading, and plugin hook contract.

Yes. A high-confidence source-level reproduction is to use a status overview whose resolved `cfg` differs from `sourceConfig.plugins.allow` and trace `executeStatusScanFromOverview` through `resolveStatusMemoryStatusSnapshot`; current main never passes the source config into that chain.

## Solution Assessment

No, not as this PR head stands. The activation-source plumbing is the right direction, but the branch must preserve `onlyPluginIds`, the `"cli"` memory purpose, and current status/read-only channel fast paths before it is the best fix.
