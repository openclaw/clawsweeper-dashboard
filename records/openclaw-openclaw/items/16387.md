## Summary

Keep open: current main still treats configured provider and request headers as literal strings, while the open related PRs address narrower typed affinity, schema, or static per-scope header paths rather than the requested generic session/agent/run header interpolation.

## What This Changes

The issue requests dynamic interpolation for custom LLM provider headers so runtime values such as session, agent, or run IDs can be used in provider requests.

## Best Possible Solution

Keep this as the canonical generic session-aware provider-header request until maintainers choose a narrow documented placeholder resolver or explicitly decide that typed provider-specific affinity contracts are the durable path.

## Reproduction Assessment

Yes. Source inspection on current main shows provider and request header values are sanitized and merged as literal strings with no session, agent, or run placeholder resolver.

## Solution Assessment

Unclear. Generic templating covers the Fireworks and Langfuse/LiteLLM use cases, but the open typed-affinity PRs may be safer for provider-specific cache routing, so maintainers should choose the durable contract before implementation.
