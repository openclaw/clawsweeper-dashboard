## Summary

Keep this issue open because current main still treats configured provider headers as literal strings, and the viable open Fireworks PR covers only session-affinity headers through the existing typed compatibility seam. The additional LiteLLM/Langfuse request for arbitrary session-, agent-, or run-aware headers remains unresolved and needs an explicit maintainer product/API decision rather than being silently folded into the Fireworks fix.

## What This Changes

Add runtime-derived session, agent, and run values to custom LLM provider request headers, primarily for cache affinity and tracing.

## Best Possible Solution

Land or reject the typed Fireworks affinity repair independently, then define a narrowly scoped runtime request-header extension—preferably typed placeholders or a provider/plugin hook with explicit precedence and data-exposure semantics—only if maintainers want to support the broader tracing use case.

## Reproduction Assessment

Yes for the reported limitation: current main sanitizes and merges configured header strings without expanding session, agent, or run values. The claimed cache-hit and tracing improvements are external provider outcomes rather than a failing current-main invariant.

## Solution Assessment

No, unrestricted header string templating is not yet established as the best solution. Fireworks fits the existing typed affinity contract, while the broader Langfuse case should use a deliberately scoped runtime hook or placeholder contract with explicit security and precedence rules.
