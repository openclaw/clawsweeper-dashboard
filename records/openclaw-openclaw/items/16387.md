## Summary

Keep open. Current main still forwards configured provider headers and request headers as literal strings, the Fireworks plugin does not add session-affinity headers, and the linked dynamic-header implementation PRs remain closed unmerged while no open replacement PR is tracking the fix.

## What This Changes

The issue requests runtime interpolation for configured custom provider headers so placeholders like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be sent per LLM request.

## Best Possible Solution

Add a narrow documented dispatch-time resolver for configured provider headers and request.headers that supports only explicit session, agent, and run identity placeholders while preserving existing static behavior.

## Reproduction Assessment

Yes. A current-main config such as `models.providers.fireworks.headers["x-session-affinity"] = "{{session.key}}"` would flow through the sanitizer, merge, and OpenAI-family transport header path as the literal string because those paths have no session/run template context.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured custom provider headers dynamic.
