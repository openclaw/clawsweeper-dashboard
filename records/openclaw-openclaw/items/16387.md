## Summary

Keep open: current main and the latest release still pass configured provider headers through as static strings, and the related dynamic-header implementation PRs are closed unmerged. The current provider turn-state hook can add provider-owned headers, but it does not resolve custom-provider config placeholders for Fireworks, LiteLLM, or similar proxy use cases.

## What This Changes

The issue requests dynamic placeholder interpolation in custom LLM provider headers, such as resolving session, agent, and run values into `models.providers.*.headers` or `request.headers`.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for provider headers covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving existing static, SecretRef, auth, retry, attribution, and provider-plugin behavior.

## Reproduction Assessment

Yes for source-level behavior: a configured header value such as `{{session.key}}` is copied and merged as a literal string today. I did not run a live provider request in this read-only review, and this remains a feature request rather than broken documented behavior.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only plugin hook can cover one vendor but not arbitrary custom provider config or the LiteLLM/Langfuse-style placeholders.
