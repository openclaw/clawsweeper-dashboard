## Summary

Keep open: current main and the latest release still pass configured provider headers through as literal strings, while the related implementation PRs are closed unmerged. Existing provider-plugin turn hooks can cover vendor-owned dynamic headers, but they do not provide the requested custom-provider config placeholders.

## What This Changes

The issue requests dynamic interpolation for custom provider HTTP headers so runtime values like session, agent, or run identifiers can be sent to providers such as Fireworks.ai and LiteLLM.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for models.providers.*.headers and models.providers.*.request.headers covering {{session.key}}, {{agent.id}}, and {{run.id}} while preserving static header, SecretRef, auth, attribution, retry, and provider-plugin behavior.

## Reproduction Assessment

Yes for source-level behavior: a configured header value such as {{session.key}} is copied and merged as a literal string today. I did not run a live provider request in this read-only review, and this remains a feature request rather than a broken documented behavior.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only plugin hook can cover one vendor but not arbitrary custom provider config or the LiteLLM/Langfuse-style agent/session placeholders.
