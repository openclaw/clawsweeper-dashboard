## Summary

Keep open: current main still normalizes configured provider headers as literal strings, has no dispatch-time placeholder context for `{{session.key}}` or `{{run.id}}`, and the exact dynamic-header PR attempts at https://github.com/openclaw/openclaw/pull/16448 and https://github.com/openclaw/openclaw/pull/18213 remain closed unmerged. The adjacent open work at https://github.com/openclaw/openclaw/pull/79990 adds static per-agent/job header scopes, not runtime interpolation.

## What This Changes

Support runtime template interpolation for custom LLM provider headers so session, agent, and run identifiers can drive provider cache or trace routing.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers`, covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving existing static header behavior and security rules.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms current main would pass placeholders such as `{{session.key}}` through as literal header text.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
