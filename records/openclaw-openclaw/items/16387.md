## Summary

Keep open: current main still treats configured provider/request headers as literal config values, while native session-affinity support and the adjacent Fireworks PR only cover known provider behavior rather than arbitrary dynamic header templating for custom providers.

## What This Changes

Request dynamic interpolation in custom LLM provider header config so headers can use runtime session, agent, or run values for Fireworks prompt cache affinity and LiteLLM/Langfuse-style tracing.

## Best Possible Solution

Add a narrow allowlisted dispatch-time resolver for provider and provider-request headers, preserving static headers, SecretRef resolution, auth injection, provider hooks, caller precedence, docs, and provider-native affinity paths.

## Reproduction Assessment

Yes, source-level: current header resolution and merge paths have no runtime placeholder context, so configured placeholders are forwarded as literal header values.

## Solution Assessment

Yes: a narrow OpenClaw-level resolver is the best remaining shape because OpenClaw owns the session, agent, and run identity needed for custom-provider headers; Fireworks-only affinity support is useful but insufficient.
