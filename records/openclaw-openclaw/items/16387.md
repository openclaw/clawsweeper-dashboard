## Summary

Keep open: current main still treats configured provider headers as literal strings, does not give the provider-header merge path session/run template context, and the Fireworks plugin only handles Kimi thinking payloads, not affinity headers. The exact implementation attempts are closed unmerged, while adjacent open PRs cover static scoped headers or tracing rather than this allowlisted runtime templating request.

## What This Changes

Request dynamic placeholder interpolation in custom LLM provider header config so runtime session, agent, and run values can be sent to providers such as Fireworks.ai and LiteLLM.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for models.providers.*.headers and models.providers.*.request.headers, covering {{session.key}}, {{agent.id}}, and {{run.id}} while preserving existing static behavior and security rules.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives a high-confidence verification path: placeholders such as {{session.key}} would currently flow through as literal header text.

## Solution Assessment

Yes. An OpenClaw-level allowlisted resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
