## Summary

Keep open: current main and v2026.6.8 still treat configured provider/request headers as static strings, and newer prompt-cache affinity or request-header work only overlaps with this session/agent/run templating request.

## What This Changes

Request dynamic interpolation for model-provider header config so values such as session, agent, or run identity can be sent per LLM request.

## Best Possible Solution

Keep this as the canonical dynamic provider-header templating item until maintainers choose a narrow documented contract that preserves static headers, SecretRef resolution, auth injection, provider attribution, and existing prompt-cache behavior.

## Reproduction Assessment

Yes for the reported limitation: current main sanitizes and merges configured header strings without expanding session, agent, or run values. I did not run tests in this read-only review, but the missing resolver is clear from source and docs.

## Solution Assessment

Unclear that generic string templating is the best final API. A safer maintainer path is to choose an explicit provider-header context contract, possibly using typed affinity or request-context fields where they cover the use case.
