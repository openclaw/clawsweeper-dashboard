## Summary

Keep this issue open: current main and the latest release still treat configured provider headers as literal strings, and the open related PRs solve narrower provider-specific affinity or schema gaps rather than the requested generic session/agent/run header interpolation.

## What This Changes

Request dynamic interpolation for custom model-provider headers, such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`, so provider requests can use per-session routing and tracing metadata.

## Best Possible Solution

Keep this as the canonical generic session-aware provider-header request until maintainers choose a narrow documented placeholder resolver or decide that typed provider-specific affinity APIs are the durable contract.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.10` shows configured provider and request headers are sanitized and merged as literal strings with no session, agent, or run placeholder resolver.

## Solution Assessment

Unclear. Generic templating covers the Fireworks and Langfuse/LiteLLM use cases, but the open typed affinity PRs may be safer for provider-specific cache routing, so maintainers should choose the durable contract before implementation.
