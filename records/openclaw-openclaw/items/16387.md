## Summary

Keep open: current main still treats configured provider headers as literal strings, so the requested session/agent/run-aware header templating is not implemented. The newer provider-owned transport identity hook helps native provider plugins, but it does not resolve user-configured custom-provider header values; the earlier implementation PRs are closed unmerged and no open generic fix PR owns this exact work.

## What This Changes

Allow custom LLM provider header config to interpolate session, agent, and run placeholders so headers like Fireworks `x-session-affinity` can vary per session.

## Best Possible Solution

Add a documented, allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers` covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving current static-header behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that a configured value like `{{session.key}}` would be forwarded literally today.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only plugin wrapper would not cover custom provider config or the LiteLLM/Langfuse use case, while arbitrary template evaluation would be too broad.
