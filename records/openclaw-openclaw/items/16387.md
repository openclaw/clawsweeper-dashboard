## Summary

Keep open: current main still has no generic dynamic provider-header templating, and the only open adjacent work appears to cover Fireworks session-affinity headers through a compat flag rather than the broader `{{session.key}}` / `{{agent.id}}` / `{{run.id}}` header contract requested here.

## What This Changes

Request dynamic placeholders in custom LLM provider headers so session, agent, or run identity can drive provider-side cache routing and tracing.

## Best Possible Solution

Review and either land or revise the Fireworks affinity PR, then keep this issue narrowed to a generic allowlisted provider-header resolver only if arbitrary `{{session.key}}`, `{{agent.id}}`, or `{{run.id}}` headers remain a desired contract.

## Reproduction Assessment

Yes. Source inspection shows configured provider headers are sanitized and merged as literal strings, and the current request wrapper does not carry a generic session/run placeholder context into provider headers.

## Solution Assessment

Unclear as a final product path. A narrow allowlisted resolver is the best fit for arbitrary dynamic headers, but open PR https://github.com/openclaw/openclaw/pull/90033 may be the better narrower fix for the Fireworks-specific affinity case.
