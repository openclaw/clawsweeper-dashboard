## Summary

Keep open: current main still treats configured provider headers as literal strings, the Fireworks plugin does not add session-affinity headers, the exact dynamic-templating implementation attempts are closed unmerged, and the adjacent open work at https://github.com/openclaw/openclaw/pull/79990 adds static per-agent/job header scopes rather than runtime placeholder interpolation.

## What This Changes

The issue requests runtime interpolation for configured custom-provider headers so placeholders such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be sent to LLM providers.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers`, covering only explicit runtime placeholders while preserving existing static behavior and security rules.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms the requested config would currently pass placeholders such as `{{session.key}}` through as literal header text.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
