## Summary

Keep this open: current main and v2026.6.6 still treat configured provider headers as static literal strings, and the newer provider-header issues do not cover per-session placeholder interpolation for normal model turns.

## What This Changes

The issue requests runtime interpolation for custom provider headers so session, agent, or run identifiers can be sent to providers such as Fireworks.ai and LiteLLM.

## Best Possible Solution

Keep this as the canonical dynamic session-aware provider-header item until maintainers choose a narrow, documented header-context contract that preserves static headers, SecretRef resolution, auth injection, provider attribution, and existing prompt-cache behavior.

## Reproduction Assessment

Yes for the reported limitation: current main sanitizes and merges configured header strings without expanding session, agent, or run values. I did not run tests because this was a read-only review, but the missing resolver is clear from source and docs.

## Solution Assessment

Unclear that generic string templating is the best final shape. A safer maintainer path is to choose an explicit provider-header context contract, possibly using typed affinity/run-context fields where they cover the use case, before queueing a fix.
