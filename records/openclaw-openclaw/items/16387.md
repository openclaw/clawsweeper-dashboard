## Summary

Keep open: current main still treats configured provider headers as literal strings, the Fireworks plugin does not add session-affinity headers, the exact implementation attempts were closed unmerged, and the adjacent open work at https://github.com/openclaw/openclaw/pull/79990 adds static per-agent/job header scopes rather than runtime template interpolation.

## What This Changes

Request runtime placeholders in configured LLM provider headers so values such as `x-session-affinity` can use session, agent, or run identifiers.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers`, covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving existing static behavior and security rules.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms the requested config would currently pass placeholders such as `{{session.key}}` through as literal header text.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
