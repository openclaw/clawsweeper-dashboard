## Summary

Keep open: current main still forwards configured provider headers as literal strings, so the requested session/agent/run-aware header templating is not implemented. The provider-owned turn-state hook helps native provider plugins, but it does not provide a documented resolver for user-configured custom-provider headers; the two related implementation PRs are closed unmerged and no open PR currently owns the generic fix.

## What This Changes

The issue requests runtime interpolation for custom provider headers so values can use session, agent, or run identifiers for provider cache affinity and tracing.

## Best Possible Solution

Add a documented, allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers` covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`, while preserving current static-header behavior and existing auth, SecretRef, retry, attribution, and provider-plugin semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that a configured value like `{{session.key}}` would be forwarded literally today.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only plugin wrapper would not cover custom provider config or the LiteLLM/Langfuse use case, while arbitrary template evaluation would be too broad.
