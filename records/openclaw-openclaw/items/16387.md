## Summary

Keep open. Current main still treats configured provider headers and request headers as literal strings, while the dynamic-header PRs linked from the discussion are closed unmerged and the merged static-header propagation work does not cover runtime placeholders.

## What This Changes

Request dynamic interpolation in custom LLM provider header config for placeholders like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` so providers can receive per-session or per-run routing headers.

## Best Possible Solution

Add a narrow documented dispatch-time resolver for configured provider headers and request headers that supports explicit session, agent, and run identity placeholders while preserving existing static header behavior.

## Reproduction Assessment

Yes for the missing behavior. Source inspection shows `{{session.key}}` would be sanitized, merged, and sent as a literal configured header because the current header paths have no resolver or session/run context.

## Solution Assessment

Yes. A generic OpenClaw resolver is the narrowest maintainable direction because OpenClaw owns the requested identity values and the feature needs to work for arbitrary custom providers, not only Fireworks.
