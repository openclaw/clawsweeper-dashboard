## Summary

Keep open: current main still normalizes and merges configured provider headers as literal strings, while provider-owned turn-state hooks and prompt_cache_key support do not provide the requested generic config placeholder contract. The related implementation PRs are closed unmerged, and the provider/config gap remains narrow and source-confirmed.

## What This Changes

Request support for allowlisted runtime placeholders in configured provider headers so custom LLM providers can receive session-, agent-, or run-aware headers.

## Best Possible Solution

Add a documented, allowlisted dispatch-time resolver for models.providers.*.headers and models.providers.*.request.headers covering {{session.key}}, {{agent.id}}, and {{run.id}}, while preserving static headers, SecretRef handling, auth injection, retries, attribution, and provider-plugin semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that a configured value like {{session.key}} would be forwarded literally today.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only wrapper would not cover custom provider config or the LiteLLM/Langfuse use case, while arbitrary template evaluation would be too broad.
