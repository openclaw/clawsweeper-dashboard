## Summary

Keep this issue open: current main still treats configured provider headers as literal strings, and the open related PRs cover narrower static/per-agent or typed Fireworks affinity paths rather than the requested generic `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` interpolation.

## What This Changes

The issue requests runtime interpolation for custom provider header values so provider requests can carry session-aware headers for Fireworks caching and custom-provider tracing.

## Best Possible Solution

Keep this as the canonical generic session-aware provider-header request until maintainers choose a durable contract: a narrow documented placeholder resolver, a typed-header strategy that also covers custom-provider tracing, or an explicit rejection of generic templating.

## Reproduction Assessment

Yes. Source inspection on current main shows configured provider and request headers are sanitized and merged as literal strings, with no session, agent, or run placeholder resolver.

## Solution Assessment

Unclear. Generic templating is the broadest fit for Fireworks plus Langfuse/LiteLLM, but the open typed affinity PR may be safer for the Fireworks-specific path, so maintainers should choose the permanent provider-header contract first.
