## Summary

Current main still does not implement the central request: configured provider headers are sanitized and merged as literal strings, while the open related PRs cover typed session-affinity/schema companions rather than arbitrary `{{session.key}}`, `{{agent.id}}`, or `{{run.id}}` header interpolation.

## What This Changes

Request to support dynamic runtime placeholders in custom model-provider headers so provider requests can include session-, agent-, or run-specific values for cache affinity and tracing.

## Best Possible Solution

Keep this as the canonical generic session-aware provider-header request until maintainers choose either a narrow documented placeholder resolver or a typed-header strategy that also covers custom-provider tracing needs.

## Reproduction Assessment

Yes for the reported limitation: source inspection shows configured provider and request headers are sanitized and merged as literal strings with no session, agent, or run placeholder resolver. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear. Generic header templating is plausible and covers the Langfuse/LiteLLM use case, but the open typed affinity PRs may be the safer Fireworks-specific path, so maintainers should choose the durable contract before implementation.
