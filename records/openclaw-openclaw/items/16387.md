- `impact:auth-provider`: The issue concerns model-provider request headers, provider cache routing, tracing metadata, and credential-adjacent request configuration.

## Summary

Keep open. Current main and v2026.6.6 still treat configured provider headers as static strings, while the open Fireworks affinity PRs cover a narrower `compat.sendSessionAffinityHeaders` path rather than the broader `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` templating request.

## What This Changes

The issue requests dynamic interpolation in custom model-provider headers so runtime session, agent, or run values can be sent to providers such as Fireworks.ai or LiteLLM.

## Best Possible Solution

Define the maintainer-approved request-header contract first: either land the narrower Fireworks compat path and keep this issue for explicit placeholders, or add a documented resolver/plugin seam that supports only approved runtime values while preserving static headers, SecretRef handling, auth, retries, and header precedence.

## Reproduction Assessment

Yes, source-level. Current main copies configured provider and request header strings literally through sanitization and merge paths, and the docs still describe those headers as static rather than templated.

## Solution Assessment

Unclear. A narrow explicit resolver is plausible, but maintainers should first decide whether the open compat-based Fireworks PRs satisfy the urgent cache-affinity case or whether a broader placeholder/plugin API is the right permanent contract.
