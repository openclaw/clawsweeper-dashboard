## Summary

Current main still supports only literal custom-provider headers; session and run identity never reach the request-header resolution path. The exact implementation attempts are closed unmerged, while the issue discussion has already narrowed this to a maintainable runtime interpolation feature.

## What This Changes

Add allowlisted runtime placeholders to custom provider headers so requests can use stable session-, agent-, or run-scoped values.

## Best Possible Solution

Resolve a closed set of placeholders such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` once at canonical model-request dispatch, preserving literal headers, SecretRef handling, auth and attribution precedence, retries, plugin wrappers, and compaction behavior.

## Reproduction Assessment

Yes at source level: current main copies configured header strings literally, and the active request wrapper does not receive sessionKey or runId. This read-only review did not perform a live Fireworks cache-hit comparison.

## Solution Assessment

Yes, with a narrower contract than general templating: an allowlisted resolver at the canonical request boundary is the best fit because OpenClaw owns these runtime identities and must apply them consistently across normal and compaction calls.
