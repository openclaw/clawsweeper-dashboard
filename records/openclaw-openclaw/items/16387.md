## Summary

Keep open: current main still treats configured provider headers as static strings and has no dispatch-time context for `{{session.key}}` or `{{run.id}}`; prior exact PRs are closed unmerged, and the adjacent open static-header PR does not cover runtime interpolation.

## What This Changes

Request dynamic interpolation of runtime session, agent, and run values in custom LLM provider headers for Fireworks affinity and similar tracing use cases.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers` covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms current main would pass placeholders such as `{{session.key}}` through as literal header text.

## Solution Assessment

Yes. An OpenClaw-level allowlisted resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
