- `P2`: The issue has real provider cost, cache-affinity, and observability impact, but it is a bounded provider-header feature rather than an outage or data-loss incident.
- `impact:auth-provider`: The requested behavior changes model-provider request headers, cache-affinity routing, and auth-adjacent provider metadata.

## Summary

Keep open: current main and v2026.6.10 still treat configured provider/request headers as literal strings, and the open related PRs cover narrower typed affinity, schema, or static header surfaces rather than generic session/agent/run header interpolation.

## What This Changes

Request dynamic interpolation in custom LLM provider headers so per-request session, agent, or run identifiers can be sent to provider/proxy endpoints.

## Best Possible Solution

Keep #16387 as the canonical generic provider-header templating request until maintainers choose a narrow documented placeholder resolver or explicitly prefer typed provider-specific affinity/run-context contracts.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.10 shows configured provider and request header values are sanitized and merged as literal strings with no session, agent, or run placeholder resolver.

## Solution Assessment

Unclear. Generic templating covers the Fireworks and Langfuse/LiteLLM use cases, but maintainers still need to choose whether the durable contract should be generic header templates, typed affinity flags, or a narrower run-context header map.
