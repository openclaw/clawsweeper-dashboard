## Summary

Keep open: current main still treats configured provider headers as literal strings, Fireworks does not add session-affinity headers itself, and the prior dynamic-header PRs are closed unmerged with no generic replacement tracking the requested behavior.

## What This Changes

Request dynamic placeholder interpolation for configured LLM provider headers so values like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be sent per request.

## Best Possible Solution

Add a narrow documented dispatch-time resolver for configured provider headers and `request.headers` with an explicit, small placeholder allowlist while preserving existing static-header behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a config such as `models.providers.fireworks.headers["x-session-affinity"] = "{{session.key}}"` would pass through sanitizer, merge, and OpenAI transport paths as a literal string because those paths have no session/run template context.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured custom provider headers dynamic.
