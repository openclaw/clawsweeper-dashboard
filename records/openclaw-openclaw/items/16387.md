## Summary

Keep open: current main and v2026.5.20 still treat configured provider headers as literal strings, and the only open related PR is provider-specific ZAI stickiness rather than the generic `{{session.key}}`/`{{agent.id}}`/`{{run.id}}` header-template contract requested here.

## What This Changes

Requested dynamic interpolation for custom provider header config so session-, agent-, and run-specific values can be sent to upstream model providers.

## Best Possible Solution

Add a documented, allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers` covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`, while preserving static headers, SecretRef resolution, auth injection, retries, attribution, and provider-plugin semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that a configured value like `{{session.key}}` would be forwarded literally today.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; provider-specific wrappers such as the open ZAI PR do not cover custom Fireworks/LiteLLM config, while arbitrary template evaluation would be too broad.
