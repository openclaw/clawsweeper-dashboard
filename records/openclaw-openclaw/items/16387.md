## Summary

Keep open: current main and the latest release still treat configured provider headers as literal strings, and the open companion PRs cover typed Fireworks affinity/schema acceptance rather than the generic dynamic header-template contract requested here.

## What This Changes

Request dynamic interpolation of provider header config values such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` for custom LLM provider requests.

## Best Possible Solution

Keep this as the canonical session-aware provider-header item until maintainers choose a durable contract, then either land the typed affinity/schema companions for the Fireworks path or implement a narrow documented placeholder resolver that preserves static headers, auth, SecretRef, attribution, and caller precedence.

## Reproduction Assessment

Yes for the reported limitation: source inspection shows configured provider headers are sanitized and merged as literal strings with no session, agent, or run placeholder resolver. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear. Generic header templating is plausible for the Langfuse/LiteLLM use case, but the typed affinity-header companion may be the safer Fireworks path; maintainers should choose the durable config contract before new implementation work proceeds.
