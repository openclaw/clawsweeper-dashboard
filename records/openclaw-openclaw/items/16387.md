## Summary

Keep open: current main and v2026.6.5 still treat configured provider headers as literal static strings, while the closed implementation attempts were not merged and no canonical open replacement owns the remaining dynamic header placeholder work.

## What This Changes

Request dynamic interpolation for custom model-provider headers so values like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be sent as per-session provider request headers.

## Best Possible Solution

Add a documented allowlisted dispatch-time resolver for provider headers covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving existing static, SecretRef, auth, retry, attribution, and provider-plugin behavior.

## Reproduction Assessment

Yes, at source level: a configured header value such as `{{session.key}}` is copied, sanitized, and merged as a literal string today. I did not run a live Fireworks or LiteLLM request in this read-only review.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; Fireworks-only plugin logic or prompt-cache-key support covers adjacent cases but not arbitrary custom-provider header placeholders.
