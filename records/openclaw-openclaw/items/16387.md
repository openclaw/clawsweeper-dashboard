## Summary

Keep open: current main still forwards configured provider headers as static strings, so the requested session/run-aware interpolation is not implemented. Static header propagation and prompt_cache_key support are adjacent improvements, but they do not solve dynamic `x-session-affinity` or LiteLLM/Langfuse header values for custom provider config.

## What This Changes

The issue requests dynamic placeholder interpolation in custom LLM provider headers, such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`.

## Best Possible Solution

Add a documented, allowlisted dispatch-time resolver for `models.providers.*.headers` and `models.providers.*.request.headers` covering `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` while preserving current static-header behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that a value like `{{session.key}}` would be forwarded literally today.

## Solution Assessment

Yes. A narrow allowlisted OpenClaw-level resolver is the maintainable path; a Fireworks-only plugin wrapper would not cover custom provider config or the LiteLLM/Langfuse use case, while arbitrary template evaluation would be too broad.
