## Summary

Keep this issue open: current main and the latest release still treat configured provider headers as literal static strings, while the open companion PRs cover narrower typed Fireworks affinity/schema paths rather than the requested generic `{{session.key}}`, `{{agent.id}}`, or `{{run.id}}` header interpolation.

## What This Changes

The issue requests dynamic runtime interpolation for configured custom-provider headers so session, agent, or run identifiers can be used in provider requests.

## Best Possible Solution

Keep this as the canonical generic session-aware provider-header request until maintainers choose either a narrow documented placeholder resolver or a typed-header strategy that also covers custom-provider tracing needs.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows configured provider and request headers are sanitized and merged as literal strings with no session, agent, or run placeholder resolver.

## Solution Assessment

Unclear. Generic header templating is plausible and covers the Langfuse/LiteLLM use case, but the open typed affinity PRs may be the safer Fireworks-specific path, so maintainers should choose the durable contract before implementation.
