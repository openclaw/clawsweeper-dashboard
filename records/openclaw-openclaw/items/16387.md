## Summary

Keep open: current main still treats configured provider headers and request headers as literal strings, Fireworks does not add session-affinity headers itself, and no open linked fix PR covers session/run header templating.

## What This Changes

Request dynamic interpolation in custom LLM provider headers so values like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can drive Fireworks or LiteLLM cache and tracing affinity.

## Best Possible Solution

Add a narrow documented dispatch-time resolver for configured provider headers and `request.headers` with an explicit placeholder allowlist while preserving existing static-header behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection still confirms the requested config, such as `x-session-affinity: {{session.key}}`, would currently pass through as a literal string.

## Solution Assessment

Yes. An OpenClaw-level resolver is the narrow maintainable direction because OpenClaw owns the requested session, agent, and run identity values, while provider plugins can add provider-owned headers but cannot make arbitrary user-configured headers dynamic.
