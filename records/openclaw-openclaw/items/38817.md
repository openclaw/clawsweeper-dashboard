## Summary

Keep open. Current main still relies on model `compat.unsupportedToolSchemaKeywords` to strip rejected JSON Schema keywords, and the Volcengine Kimi/Ark catalog rows still do not declare that compat metadata; the direct fix PRs are closed unmerged and the open Kimi compaction PR tracks a different failure mode.

## What This Changes

The issue requests that Volcengine-hosted Kimi/Ark models stop receiving unsupported tool-schema keywords such as `minLength`, `maxLength`, and `minItems`.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine Kimi/Ark catalog rows, with focused regression coverage, while keeping core normalization generic.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does show current main will leave `minLength`/`maxLength` in built-in tool schemas for the affected Volcengine rows because those rows lack the compat metadata that triggers stripping.

## Solution Assessment

Yes, the best narrow fix is provider metadata rather than provider-name branching in core. The current generic compat seam already strips declared unsupported keywords, so Volcengine should feed that seam from its manifest catalog.
