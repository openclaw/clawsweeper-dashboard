## Summary

Keep open. Current main still relies on resolved model compat metadata to strip unsupported tool-schema keywords, but the Volcengine Ark/Kimi catalog rows do not declare `compat.unsupportedToolSchemaKeywords`, so the reported `minLength` rejection can still reach the provider; the two direct fix PRs are closed unmerged and the open Kimi compaction PR tracks a different failure mode.

## What This Changes

The issue reports Volcengine-hosted Kimi models rejecting tool schemas that contain JSON Schema validation keywords such as `minLength`, causing tool calls to fail.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine Ark/Kimi catalog rows, include BytePlus only if confirmed, keep core normalization generic, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to resolve a current Volcengine Ark/Kimi model row, normalize a tool schema containing `minLength`, and observe that current main leaves the keyword because the resolved model has no `compat.unsupportedToolSchemaKeywords`.

## Solution Assessment

No for the literal provider/model substring detection proposed in the issue. The narrower maintainable solution is provider manifest compat metadata feeding the existing generic normalization seam, with no core Volcengine/Kimi branching.
