## Summary

Keep open. Current main still has the generic `compat.unsupportedToolSchemaKeywords` path, but the affected Volcengine Ark/Kimi catalog rows do not declare that metadata, so `minLength` is still not stripped for those models; the two direct fix PRs are closed unmerged and the remaining open Kimi PR tracks a different schema-size/malformed-tool-call problem.

## What This Changes

Volcengine/Kimi tool schemas should strip unsupported JSON Schema validation keywords such as `minLength` before sending tool parameters to affected models.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine Ark/Kimi catalog rows, with focused tests proving the metadata reaches tool normalization; include BytePlus rows only if their backend behavior is confirmed, and keep core normalization generic rather than adding provider/model substring checks.

Yes. A high-confidence code-path reproduction is to resolve a current Volcengine Ark/Kimi model row, normalize a tool schema containing `minLength`, and observe that current main leaves the keyword because the row has no compat metadata; the live provider rejection itself was reported but not rerun here.

## Solution Assessment

Yes. The narrow maintainable solution is to feed the existing generic compat seam from provider catalog metadata, rather than adding Volcengine/Kimi substring detection to core tool normalization.
