## Summary

Keep open: current main still strips unsupported tool-schema keywords only from model compat metadata, and the Volcengine/Ark/Kimi catalog rows still do not declare that metadata. The direct fix PRs #38883 and #46887 are closed unmerged, while #44888 tracks a separate Kimi schema-compaction problem.

## What This Changes

Volcengine-hosted Kimi models should strip unsupported JSON Schema validation keywords from tool parameter schemas so built-in tool calls do not fail provider validation.

## Best Possible Solution

Add provider-owned compat.unsupportedToolSchemaKeywords metadata to the affected Volcengine/Ark/Kimi catalog rows, include BytePlus only if the same backend behavior is confirmed, and keep core normalization driven by the existing generic compat seam.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear current-main path: Volcengine/Kimi models without compat metadata keep minLength in tool schemas, matching the reporter's provider validation error; no live API call was run.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is provider catalog metadata feeding the existing compat seam, not a new provider-name branch in core normalization.
