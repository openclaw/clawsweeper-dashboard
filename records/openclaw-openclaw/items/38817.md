## Summary

Keep open. Current main still has the generic compat-driven schema stripping path, but the Volcengine Ark/Kimi catalog rows do not declare `compat.unsupportedToolSchemaKeywords`, so the reported `minLength` rejection can still reach the provider.

## What This Changes

Volcengine-hosted Kimi models should strip unsupported tool-schema validation keywords such as `minLength`, `maxLength`, and `minItems` before sending tool definitions.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine Ark/Kimi catalog rows, and include BytePlus only if the same backend limitation is confirmed, while keeping core normalization generic.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to resolve a current Volcengine Ark/Kimi model row, normalize a tool schema containing `minLength`, and observe that current main leaves the keyword because the row has no compat metadata.

## Solution Assessment

No for the literal core substring detection proposed in the issue. The narrower maintainable fix is manifest compat metadata feeding the existing generic normalization seam.
