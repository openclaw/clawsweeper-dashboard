## Summary

Keep open. Current main still has the generic compat-based schema cleanup, but the affected Volcengine/Ark/Kimi catalog rows do not declare the unsupported schema keywords, the two direct fix PRs are closed unmerged, and the open Kimi compaction PR tracks a different failure mode.

## What This Changes

The issue asks OpenClaw to stop sending unsupported JSON Schema keywords such as `minLength` to Volcengine-hosted Kimi models during tool calls.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine/Ark/Kimi catalog rows, include BytePlus only if it shares the backend behavior, and keep core schema normalization generic.

## Reproduction Assessment

Yes for a source-level path. Current main passes model compat into tool normalization, Volcengine/Kimi rows do not provide unsupported-keyword compat, and existing built-in tool schemas still contain `minLength`/`maxLength` fields that would survive for those models.

## Solution Assessment

No for the issue body's proposed provider-string fallback in core. The narrower maintainable fix is to populate provider-owned catalog compat metadata so the existing generic normalization path handles the affected models.
