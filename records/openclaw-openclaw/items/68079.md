## Summary

Keep open: the provider-local optimization is useful and has sufficient live proof, but the current patch would regress current ZAI thinking behavior, accepts header-invalid session IDs, can duplicate the session header under different casing, and adds a stale Pi runtime package.

## What This Changes

The PR adds a ZAI provider stream wrapper, tests, docs, changelog, and package/lockfile updates to inject `X-Session-Id` from the agent `sessionId` for prompt-cache stickiness.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that composes with current `wrapZaiStreamFn`, validates and canonicalizes `X-Session-Id`, uses the current `@earendil-works/*` Pi runtime packages, and keeps focused tests/docs aligned.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the branch would replace current ZAI thinking-wrapper behavior, and a Node `Headers` probe reproduces invalid header rejection plus duplicate-cased header normalization. The cache improvement itself is supported by the contributor's live benchmark and inspected chart.

## Solution Assessment

No as currently implemented. Provider-local header injection is the right boundary, but the patch needs to preserve the current ZAI stream behavior, sanitize and replace the header safely, and use the current Pi runtime namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Compose with the existing ZAI stream wrapper:** `extensions/zai/index.ts:279`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which sets `tool_stream` and patches disabled or preserved thinking payloads. Replacing it with this new wrapper drops those covered thinking behaviors after merge.
  - confidence: 0.93
- **[P2] Reject header-invalid session IDs:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts non-empty strings containing CR, LF, or NUL. Fetch `Headers` rejects those values, so a malformed `sessionId` would make otherwise valid ZAI requests fail before they are sent.
  - confidence: 0.9
- **[P2] Replace session headers case-insensitively:** `extensions/zai/stream.ts:39-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization comma-joins the values, which can send the wrong sticky-routing key.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: Current main uses `@earendil-works/pi-ai@0.74.0`, but this PR adds `@mariozechner/pi-ai@0.69.0` for ZAI. That pulls a second stale provider runtime into the plugin instead of using the current namespace.
  - confidence: 0.9
