## Summary

Keep open: the provider-local direction is useful and the PR includes real workload proof, but the current branch still has correctness and security blockers around session-id plumbing, ZAI wrapper composition, header normalization, raw session-id exposure, and stale Pi dependency namespace.

## What This Changes

The PR adds a ZAI provider stream wrapper, tests, docs/changelog text, package metadata, and lockfile changes to inject `X-Session-Id` from `sessionId` for sticky prompt-cache routing.

## Best Possible Solution

Land a repaired provider-local ZAI wrapper that preserves current thinking/tool-stream behavior, receives a stable opaque affinity key through embedded provider-stream options, normalizes session headers safely, and uses the current Pi package namespace.

## Reproduction Assessment

Yes for the PR review blockers: current-main source shows provider-owned streams do not pass `sessionId`, current ZAI thinking behavior would be dropped by the wrapper replacement, and a Node `Headers` probe reproduces invalid-value rejection and duplicate-cased header merging. The cache-rate improvement itself was not independently rerun.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve current ZAI stream behavior, pass an opaque valid affinity key through the run path, normalize existing session headers, and use current runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass the run session id into provider streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but current embedded provider-owned streams are wrapped without copying the run session id into stream options. Normal ZAI calls can therefore skip the new header while the tests pass by invoking the wrapper with a synthetic `sessionId`.
  - confidence: 0.88
- **[P2] Compose with the current ZAI wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both tool-stream defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with only `wrapZaiProviderStream` drops those covered behaviors, so ZAI thinking settings can regress after rebase.
  - confidence: 0.94
- **[P2] Hash the affinity value before sending it:** `extensions/zai/stream.ts:42`
  - body: The new header forwards the raw OpenClaw session identifier to ZAI, but the provider only needs a stable routing key. Derive a stable opaque provider-local value so user or client-chosen session labels are not exposed to the external provider.
  - confidence: 0.8
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including values with header-invalid control bytes. Once copied into `X-Session-Id`, those values can be rejected during Fetch header normalization before a valid ZAI request is sent.
  - confidence: 0.9
- **[P2] Replace existing x-session-id case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main provider code uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.93
