## Summary

Keep open: the feature is useful and has live proof, but the current branch is conflicting with main and still has concrete correctness and security blockers around stream composition, session-id plumbing, header safety, and the Pi package namespace.

## What This Changes

The PR adds ZAI provider `X-Session-Id` sticky-routing header injection from a session identifier, with tests, docs, changelog, package metadata, and lockfile updates.

## Best Possible Solution

Land a repaired provider-local ZAI affinity implementation that preserves current thinking/tool-stream behavior, receives a stable safe session affinity key on normal ZAI calls, normalizes headers case-insensitively, and stays on the current Pi package namespace.

## Reproduction Assessment

Yes for the PR review blockers: current-main source shows provider-owned streams are wrapped without `sessionId`, current ZAI has thinking patches the PR must preserve, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate header merging. The live cache-rate improvement itself was not independently rerun.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to compose with current ZAI stream behavior, pass a safe affinity key through real provider-owned calls, normalize existing headers, and use current runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass session ids into provider-owned streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but current embedded provider-owned streams are wrapped without copying the run session id into stream options. Normal ZAI calls can therefore skip the new header while the tests pass by invoking the wrapper with a synthetic `sessionId`.
  - confidence: 0.88
- **[P2] Compose with the current ZAI wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both tool-stream defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with only `wrapZaiProviderStream` drops those covered behaviors after rebase, so ZAI thinking settings can regress.
  - confidence: 0.94
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR/LF and NUL bytes. Once copied into `X-Session-Id`, Fetch/Undici header normalization can throw before a valid ZAI request is sent.
  - confidence: 0.9
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main provider code uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.93
