## Summary

Keep this PR open, but it is not mergeable yet: the proposed ZAI sticky-routing wrapper misses current session-id plumbing, replaces current ZAI thinking behavior, accepts unsafe header values, exposes the raw session id, and adds the stale Pi package namespace.

## What This Changes

The PR adds a ZAI provider stream wrapper, tests, docs/changelog text, and package metadata to inject `X-Session-Id` from `options.sessionId` for prompt-cache sticky routing.

## Best Possible Solution

Land a rebased provider-local ZAI wrapper that preserves current thinking/tool-stream behavior, receives a stable opaque affinity key, normalizes headers safely, uses the current Pi package namespace, and keeps focused tests/docs aligned.

## Reproduction Assessment

Yes for the PR review blockers: source inspection shows the missing provider-stream `sessionId` plumbing and dropped ZAI thinking wrapper, and a Node `Headers` probe reproduces invalid header value rejection plus duplicate header merging. The cache-rate improvement itself was not independently rerun.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to compose with current ZAI stream behavior, pass an opaque valid affinity key, normalize existing headers, and use current runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass the run session id into provider streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` only reads `options.sessionId`, but the current provider-owned stream path does not pass the run session id into `wrapEmbeddedAgentStreamFn`. Normal ZAI calls can therefore skip the new header while tests pass by calling the wrapper with a synthetic `sessionId`.
  - confidence: 0.88
- **[P2] Compose with the current ZAI wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both tool-stream defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with only `wrapZaiProviderStream` drops those covered behaviors, so ZAI thinking settings can regress after rebase.
  - confidence: 0.94
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR, LF, and NUL bytes. Once copied into `X-Session-Id`, Fetch/Undici header normalization can throw before a valid ZAI request is sent.
  - confidence: 0.9
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Hash the affinity value before sending it:** `extensions/zai/stream.ts:42`
  - body: The provider only needs a stable routing key, but the patch forwards the raw OpenClaw session identifier to ZAI. Derive a stable opaque provider-local value so user or client-chosen session labels are not exposed externally.
  - confidence: 0.8
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main provider code uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.93
