## Summary

Keep open: the ZAI sticky-routing feature remains useful, but the PR is not safe to merge against current main because it drops existing ZAI thinking behavior, may not receive the run session id on normal provider-owned calls, accepts unsafe header values, duplicates case-variant headers, and adds a stale Pi runtime dependency.

## What This Changes

Adds a ZAI provider stream wrapper, tests, docs/changelog, package metadata, and lockfile changes to inject `X-Session-Id` from stream options for sticky prompt-cache routing.

## Best Possible Solution

Land a repaired provider-local ZAI affinity implementation that preserves existing ZAI thinking/tool-stream behavior, propagates a safe stable affinity key, normalizes headers, and uses the current Pi package namespace.

## Reproduction Assessment

Yes for the PR blockers: current-main source shows the missing provider-stream `sessionId` path and existing ZAI thinking wrapper, and a local Node header probe reproduces invalid-value rejection plus duplicate header merging. The live ZAI cache-rate improvement itself was not independently rerun.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but this patch needs to compose with current main, use a propagated safe affinity key, sanitize and replace headers case-insensitively, and avoid the stale package namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass the run session id into provider streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but current provider-owned stream resolution wraps `params.providerStreamFn` without passing the run `sessionId`. Normal ZAI calls can therefore miss the new header even though the unit tests pass with synthetic options.
  - confidence: 0.9
- **[P2] Preserve the current ZAI thinking wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both `tool_stream` defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with only `wrapZaiProviderStream` drops those covered behaviors after rebase.
  - confidence: 0.94
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string up to 256 chars, including CR/LF and NUL bytes. Once copied into `X-Session-Id`, Fetch/Undici header normalization can throw before a valid ZAI request is sent.
  - confidence: 0.91
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: Current main provider code uses `@earendil-works/pi-ai@0.74.0`, but this PR imports and adds `@mariozechner/pi-ai@0.69.0`. That pulls a second stale provider runtime into the ZAI plugin without a compatibility reason.
  - confidence: 0.93
