## Summary

Keep this PR open: the ZAI sticky-routing idea is useful and the provided benchmark proof is meaningful, but the current diff still has blocking correctness, privacy, and dependency issues before it can merge.

## What This Changes

The PR adds ZAI provider docs, tests, changelog, package changes, and a stream wrapper that injects `X-Session-Id` from `options.sessionId` for prompt-cache sticky routing.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that preserves current thinking/tool-stream behavior, sends an opaque valid stable affinity key, replaces conflicting headers canonically, uses the current Pi runtime namespace, and proves the normal ZAI run path receives the session key.

## Reproduction Assessment

Yes for the review blockers: current-main source and tests show the ZAI stream-wrapper contract, the PR diff replaces it, and a Node header probe reproduces invalid-value rejection plus duplicate-cased header merging. The cache improvement itself is supported by the contributor's live benchmark chart rather than a local live ZAI run.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve the current ZAI wrapper, use an opaque and header-valid key, replace existing session headers case-insensitively, use current dependencies, and prove the session id reaches normal ZAI calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Plumb the session id into the ZAI wrapper:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` only reads `options.sessionId`, but the current embedded provider stream path needs to pass the run session id into the stream options before this wrapper runs. Otherwise normal ZAI calls can skip the new header while the unit tests still pass because they call the wrapper directly with a synthetic `sessionId`.
  - confidence: 0.78
- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:282`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which applies tool-stream defaults and patches disabled or preserved thinking payloads. Replacing that registration with this new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.93
- **[P2] Send an opaque affinity key instead of the raw session id:** `extensions/zai/stream.ts:42`
  - body: The new header forwards `options.sessionId` verbatim to ZAI. OpenClaw session ids can encode channel, group, topic, or user routing metadata, but the provider only needs a stable affinity key, so this should derive a stable opaque value instead of exposing the raw session id.
  - confidence: 0.86
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR, LF, and NUL. Once copied into `X-Session-Id`, those values can throw during Fetch header normalization before a valid ZAI request is sent.
  - confidence: 0.9
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new ZAI files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.9
