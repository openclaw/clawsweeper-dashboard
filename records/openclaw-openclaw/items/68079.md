## Summary

Keep open: the provider-local sticky-routing idea is useful and the proof is credible, but the current branch still has localized correctness and security blockers against current main.

## What This Changes

The PR adds ZAI docs, tests, package changes, and a stream wrapper intended to inject an `X-Session-Id` header for prompt-cache sticky routing.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that preserves current thinking/tool-stream behavior, passes an opaque valid affinity key through the normal run path, and keeps dependencies on the current Pi namespace.

## Reproduction Assessment

Yes for the review blockers: current-main source shows provider-owned stream wrappers do not receive `sessionId`, current ZAI tests cover thinking payload behavior that the PR would drop, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate-cased header merging. The cache improvement itself is supported by the PR's live benchmark screenshot rather than a local live ZAI run.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve the current ZAI wrapper, use a header-valid opaque key, replace existing session headers canonically, and use the current Pi runtime namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the run session id to provider streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` only reads `options.sessionId`, but current embedded provider-owned streams are wrapped without adding the run `sessionId` to options. Normal ZAI calls can therefore miss the new header while these tests still pass because they call the wrapper directly with a synthetic `sessionId`.
  - confidence: 0.83
- **[P2] Compose with the current ZAI wrapper:** `extensions/zai/index.ts:282`
  - body: Current main routes ZAI through a wrapper that applies tool-stream defaults and patches disabled or preserved thinking payloads. Replacing the registration with this new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.94
- **[P2] Hash the affinity value before sending it:** `extensions/zai/stream.ts:42`
  - body: The new header forwards the raw OpenClaw session identifier to ZAI, but the provider only needs a stable routing key. Derive a stable opaque provider-local value instead of exposing user or client-chosen session labels to the external provider.
  - confidence: 0.78
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR, LF, and NUL. Once copied into `X-Session-Id`, those values are rejected during Fetch header normalization before a valid ZAI request can be sent.
  - confidence: 0.91
- **[P2] Replace existing x-session-id case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.92
