## Summary

Keep open: the ZAI session-affinity feature is plausible and the PR has real behavior proof, but the current branch is not merge-ready because it drops current ZAI thinking payload behavior and has focused header/dependency defects.

## What This Changes

The PR adds a ZAI provider stream wrapper, tests, docs, changelog, package, and lockfile changes to inject an `X-Session-Id` header from `sessionId` for prompt-cache stickiness.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that keeps existing thinking/tool-stream behavior while adding safe `X-Session-Id` injection and aligned tests/docs.

## Reproduction Assessment

Yes for the review blockers: current-main tests define the ZAI thinking payload behavior, and source inspection of the PR diff shows the new wrapper would bypass that patch; a Node Request/Headers probe confirms the invalid and duplicate header cases.

## Solution Assessment

No as implemented. Provider-local header injection is the right boundary, but it should compose with current `wrapZaiStreamFn`, validate/canonicalize the header safely, and use the current Pi runtime namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:279`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which applies tool_stream defaults and patches disabled or preserved-thinking payloads. Replacing that registration with this new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.94
- **[P2] Reject HTTP-invalid session IDs before injecting:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including NUL bytes. Once copied into `X-Session-Id`, those values throw during Fetch/Request header normalization before a valid ZAI request can be sent.
  - confidence: 0.86
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:39-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new ZAI files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.9
