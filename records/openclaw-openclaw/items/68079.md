## Summary

Keep open: the feature is useful and has live benchmark evidence, but the current PR head still has merge-blocking correctness and security issues against current main, including missing session-id plumbing, dropped ZAI thinking behavior, raw/invalid header handling, duplicate header normalization, and a stale Pi package namespace.

## What This Changes

The PR adds ZAI provider stream tests, docs, changelog, package changes, and a stream wrapper to inject `X-Session-Id` for prompt-cache sticky routing.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that preserves current thinking/tool-stream behavior, injects a valid opaque affinity key through the normal run path, and stays on the current Pi package namespace.

## Reproduction Assessment

Yes for the review blockers: current-main source shows provider-owned streams do not receive `sessionId`, existing ZAI tests cover thinking behavior the PR drops, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate-cased header merging. The cache improvement itself is supported by the PR's inspected live benchmark screenshot, not by a local live ZAI run.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve the current ZAI wrapper, pass a valid opaque affinity key through the run path, normalize existing session headers, and use the current Pi runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the run session id to provider streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but embedded provider-owned streams are wrapped without copying the run `sessionId` into options. Normal ZAI calls can therefore skip the new header while these tests still pass because they invoke the wrapper with a synthetic `sessionId`.
  - confidence: 0.83
- **[P2] Compose with the current ZAI wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both tool-stream defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing the registration with this new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.94
- **[P2] Hash the affinity value before sending it:** `extensions/zai/stream.ts:42`
  - body: The new header forwards the raw OpenClaw session identifier to ZAI, but the provider only needs a stable routing key. Derive a stable opaque provider-local value so user or client-chosen session labels are not exposed to the external provider.
  - confidence: 0.78
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR, LF, and NUL. Once copied into `X-Session-Id`, those values are rejected during Fetch header normalization before a valid ZAI request can be sent.
  - confidence: 0.91
- **[P2] Replace existing x-session-id case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.92
