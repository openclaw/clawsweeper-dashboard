## Summary

Keep this PR open: the ZAI sticky-routing feature is provider-scoped and backed by useful live evidence, but the current branch still has concrete merge blockers in stream composition, header handling, and dependency alignment.

## What This Changes

The PR adds a ZAI stream wrapper that injects `X-Session-Id` from `options.sessionId`, updates ZAI provider registration, adds stream-wrapper tests, declares a Pi runtime dependency, and updates docs/changelog text.

## Best Possible Solution

Keep the feature ZAI-owned, but repair the branch so session-header injection composes with current ZAI thinking/tool-stream handling, validates and replaces the header safely, and uses the current Pi package namespace/version.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR bypasses current ZAI thinking wrapping, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate-casing combination. The feature itself is supported by the PR's live benchmark rather than a failing current-main bug reproduction.

## Solution Assessment

No. A provider-local wrapper is the right boundary, but this branch is not mergeable until it preserves the existing ZAI wrapper, sanitizes and canonicalizes the header, and aligns with current Pi runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:279`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which preserves tool-stream defaults and patches disabled/preserved thinking payloads. Replacing it with this new wrapper drops that existing behavior after rebase, so ZAI thinking settings can regress.
  - confidence: 0.9
- **[P2] Reject header-invalid session IDs:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts non-empty strings containing CR, LF, NUL, or other invalid header bytes. Once those values are copied into `X-Session-Id`, Fetch/Undici header normalization can throw and fail an otherwise valid ZAI request; omit invalid values instead.
  - confidence: 0.91
- **[P2] Replace session headers case-insensitively:** `extensions/zai/stream.ts:39-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Case-insensitive header normalization can combine them into `old, new`, breaking the sticky-routing key; remove any existing session header before setting the canonical one.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace and version:** `extensions/zai/package.json:8`
  - body: The PR adds `@mariozechner/pi-ai@0.69.0` and imports old Pi package names while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a stated compatibility reason.
  - confidence: 0.9
