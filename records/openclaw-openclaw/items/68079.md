Full review comments:

- **[P1] Use an opaque affinity key for the provider header:** `extensions/zai/stream.ts:34-42`
  - body: This copies `options.sessionId` directly into a third-party ZAI request header. Current main has `promptCacheKey` as a distinct provider-affinity value, and recent cache-affinity fixes rely on that key staying stable while run/session ids rotate, so the PR needs an approved opaque key or explicit maintainer privacy policy before merge.
  - confidence: 0.88
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading `options.headers` before adding `X-Session-Id` leaves an existing `x-session-id` key in place. Fetch/Undici normalizes those as the same header and combines values, so ZAI can receive `old, new` instead of one sticky-routing key.
  - confidence: 0.93
- **[P2] Reject header-invalid session ids before injection:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR/LF and other invalid header bytes. Once copied into `X-Session-Id`, Node `Headers` throws before the request is sent, turning a caller-provided session id into a ZAI transport failure.
  - confidence: 0.89
- **[P3] Drop the release-owned changelog edit:** `CHANGELOG.md:21`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns that file. Keep the release-note context in the PR body or squash message and remove this direct changelog change.
  - confidence: 0.95
