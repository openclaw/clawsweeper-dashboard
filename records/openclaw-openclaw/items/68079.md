## Summary

Keep open: the PR targets a useful ZAI prompt-cache stickiness improvement and includes live chart proof, but the current branch still drops current ZAI thinking behavior, can miss the normal run session id path, accepts invalid or duplicate headers, and adds a stale Pi runtime dependency.

## What This Changes

Adds a ZAI provider stream wrapper, tests, docs/changelog, package metadata, and lockfile changes to inject `X-Session-Id` from stream options for prompt-cache sticky routing.

## Best Possible Solution

Land a rebased provider-local ZAI affinity implementation that composes with current `wrapZaiStreamFn`, receives a safe stable affinity key before header injection, normalizes existing session headers, and uses the current `@earendil-works` Pi packages.

## Reproduction Assessment

Yes for the PR review blockers: source inspection shows the wrapper-order/sessionId and current ZAI thinking behavior, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate header merging. I did not independently rerun the live ZAI cache-rate improvement.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve current ZAI stream behavior, use a propagated and safe affinity key, normalize headers, and avoid the stale package namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass the run session id into the affinity wrapper:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads only `options.sessionId`, but this provider wrapper is applied outside the stream layer that currently adds the run session id. Normal agent calls can therefore skip the new header even though the unit tests pass with synthetic options.
  - confidence: 0.9
- **[P2] Compose with the current ZAI thinking wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies `tool_stream` defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with only `wrapZaiProviderStream` drops those covered behaviors after rebase.
  - confidence: 0.94
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: The validator accepts any non-empty trimmed string up to 256 chars, including CR, LF, and NUL bytes. Once copied into `X-Session-Id`, Fetch/Undici header normalization throws and the ZAI request never starts.
  - confidence: 0.92
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before setting `X-Session-Id` leaves a prior `x-session-id` in place. Header normalization combines the values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: Current main provider code uses `@earendil-works/pi-ai` 0.75.1, but this PR imports and declares `@mariozechner/pi-ai` 0.69.0. That pulls a second stale provider runtime into the ZAI plugin without a compatibility reason.
  - confidence: 0.93
