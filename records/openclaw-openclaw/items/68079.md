## Summary

Keep this PR open: the ZAI sticky-routing feature is useful and has inspected live benchmark proof, but the current head still has concrete correctness and security blockers and current main has not implemented the change.

## What This Changes

The PR adds a ZAI provider stream wrapper that injects an `X-Session-Id` header for prompt-cache sticky routing, with tests, docs, changelog, package, and lockfile updates.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that preserves current thinking/tool-stream behavior, injects a valid opaque affinity key through the normal run path, and uses the current Pi package namespace.

## Reproduction Assessment

Yes for the review blockers: current-main source shows the wrapper ordering/session-id gap and tested ZAI thinking behavior, and the Node `Headers` probe reproduces invalid-value rejection plus duplicate-cased header merging. The cache improvement itself is supported by the inspected PR screenshot, not by a local live ZAI run.

## Solution Assessment

No as currently implemented. Provider-local sticky routing is the right boundary, but the patch needs to preserve the current ZAI wrapper, pass or derive the affinity key in the correct layer, normalize headers, and avoid stale runtime dependencies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the run session id before the ZAI wrapper reads it:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but current run setup applies provider wrappers outside the inner wrapper that can add the run `sessionId`. Normal ZAI calls can therefore skip the new header while the tests pass only because they call the wrapper with a synthetic `sessionId`.
  - confidence: 0.84
- **[P2] Preserve the current ZAI thinking wrapper:** `extensions/zai/index.ts:327`
  - body: Current main's ZAI wrapper applies both tool-stream defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing registration with the new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.94
- **[P2] Hash the affinity value before sending it:** `extensions/zai/stream.ts:42`
  - body: The new header forwards the raw OpenClaw session identifier to ZAI, but the provider only needs a stable routing key. Derive a stable opaque provider-local value so user or client-chosen session labels are not exposed to the external provider.
  - confidence: 0.78
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts any non-empty trimmed string, including CR, LF, and NUL. Once copied into `X-Session-Id`, those values are rejected during Fetch header normalization before a valid ZAI request can be sent.
  - confidence: 0.91
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization combines the two values, such as `old, new`, which can break the sticky-routing key.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.92
