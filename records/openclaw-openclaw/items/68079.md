## Summary

Keep open: the provider-local idea is useful and the live proof is adequate, but the current patch still has localized correctness and security blockers around session plumbing, current ZAI thinking behavior, header normalization, and the Pi runtime namespace.

## What This Changes

The PR adds a ZAI stream wrapper, tests, package dependency, docs, and changelog entry to inject `X-Session-Id` from `sessionId` for Z.AI prompt-cache sticky routing.

## Best Possible Solution

Land a repaired provider-local ZAI implementation that preserves current thinking/tool-stream behavior, sends a validated opaque affinity key through normal embedded runs, normalizes existing headers, and uses the current Pi package namespace.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the current stream plumbing and ZAI wrapper behavior the PR must preserve, and a Node `Headers` probe reproduces invalid-value rejection and duplicate-cased header merging. I did not rerun a live ZAI workload; the cache improvement is supported by the inspected PR screenshot.

## Solution Assessment

No, not as currently implemented. Provider-local sticky routing is the right boundary, but this patch needs to compose with current ZAI behavior, receive the run session id on real paths, hash or otherwise opaque the affinity value, normalize headers, and use current runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the run session id into wrapped streams:** `extensions/zai/stream.ts:34`
  - body: `createZaiSessionIdHeaderWrapper` reads `options.sessionId`, but the normal embedded stream resolution path does not pass the run `sessionId` into provider/default boundary-aware streams before this wrapper is applied. Real ZAI calls can therefore skip the new header while the unit tests pass because they invoke the wrapper with a synthetic `sessionId`.
  - confidence: 0.84
- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:282`
  - body: Current main's ZAI wrapper applies both `tool_stream` defaults and thinking payload patches for disabled or preserved-thinking modes. Replacing the registration with this new wrapper drops those covered behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.94
- **[P2] Send an opaque affinity value:** `extensions/zai/stream.ts:42`
  - body: The new header forwards the raw OpenClaw session identifier to ZAI, but the provider only needs a stable routing key. Derive a stable opaque provider-local value so user or client-chosen session labels are not exposed to the external provider.
  - confidence: 0.78
- **[P2] Reject header-invalid session ids:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts non-empty strings that can contain CR, LF, or NUL. Once copied into `X-Session-Id`, those values can fail Fetch header normalization before a valid ZAI request is sent.
  - confidence: 0.91
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading existing headers before adding `X-Session-Id` leaves a prior `x-session-id` key in place. Header normalization treats those as the same header and combines them, which can produce an invalid or non-sticky affinity key such as `old, new`.
  - confidence: 0.91
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new runtime file imports old `@mariozechner/*` packages and adds `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.92
