## Summary

Keep open: the ZAI sticky-routing feature is useful and has live benchmark proof, but this branch is not mergeable as-is because it bypasses current ZAI thinking payload handling, accepts header-invalid session IDs, can duplicate differently cased session headers, and adds an older renamed Pi runtime dependency.

## What This Changes

The PR adds a ZAI provider stream wrapper that injects `X-Session-Id` from `options.sessionId`, plus ZAI stream tests, a plugin dependency, and docs/changelog updates.

## Best Possible Solution

Land a repaired ZAI-local wrapper that injects the sticky-routing header while preserving current ZAI thinking/tool-stream behavior, canonicalizing header handling, and using the current Pi package namespace.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR bypasses current ZAI wrapper behavior, and a Node `Headers` probe reproduces invalid-value rejection plus duplicate-casing header combination. The feature itself is a provider-performance enhancement supported by the PR's live benchmark rather than a current-main bug reproduction.

## Solution Assessment

No as currently implemented. A provider-local wrapper is the right boundary, but this branch needs to compose with current ZAI thinking handling, sanitize and replace the session header safely, and align with current Pi runtime packages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:279`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which preserves tool-stream defaults and patches disabled or preserved thinking payloads. Replacing that registration with this new wrapper drops those current behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.92
- **[P2] Reject header-invalid session IDs:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts non-empty strings containing CR, LF, NUL, or other header-invalid bytes. Once copied into `X-Session-Id`, Fetch/Undici header normalization can throw and fail an otherwise valid ZAI request; omit invalid values instead.
  - confidence: 0.92
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:39-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Case-insensitive header normalization can combine them into `old, new`, breaking the sticky-routing key; remove any existing session header before setting the canonical one.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new ZAI files import old `@mariozechner/*` packages and add `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.9
