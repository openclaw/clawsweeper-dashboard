## Summary

Keep open: the requested ZAI prompt-cache optimization is plausible and has useful live proof, but the current branch would regress existing ZAI thinking payload handling, accepts header-invalid session IDs, can emit duplicate session headers, and adds a stale renamed Pi runtime dependency.

## What This Changes

The branch adds a ZAI provider stream wrapper that injects `X-Session-Id` from `sessionId`, plus tests, package metadata, docs, changelog, and lockfile updates.

## Best Possible Solution

Land a repaired provider-local wrapper that preserves existing ZAI thinking and tool-stream behavior, safely canonicalizes `X-Session-Id`, uses current Pi packages, and keeps focused docs/tests aligned.

## Reproduction Assessment

Yes for the review blockers: current-main source shows the ZAI thinking wrapper that the branch bypasses, and a Node header probe reproduces invalid control-byte rejection plus duplicate header merging. The requested optimization itself is supported by the contributor's live benchmark rather than a current-main bug reproduction.

## Solution Assessment

No as currently implemented. A provider-local session header wrapper is the right boundary, but it needs to compose with current `wrapZaiStreamFn`, sanitize and replace the header safely, and align with current Pi package names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Compose with the current ZAI stream wrapper:** `extensions/zai/index.ts:279`
  - body: Current main routes ZAI through `wrapZaiStreamFn`, which preserves tool-stream defaults and patches disabled or preserved thinking payloads. Replacing that registration with the new wrapper drops those behaviors, so ZAI thinking settings can regress after merge.
  - confidence: 0.92
- **[P2] Reject header-invalid session IDs:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts non-empty strings containing CR, LF, NUL, or other control bytes. Once copied into `X-Session-Id`, Node/Fetch header validation can throw before the request is sent, so invalid header values should be omitted.
  - confidence: 0.92
- **[P2] Canonicalize existing X-Session-Id headers:** `extensions/zai/stream.ts:39-42`
  - body: Spreading existing headers and then adding `X-Session-Id` leaves a prior `x-session-id` key in place. Case-insensitive normalization combines those values, which can send `old, new` instead of the intended sticky-routing key.
  - confidence: 0.9
- **[P2] Use the current Pi package namespace:** `extensions/zai/package.json:8`
  - body: The new ZAI code imports old `@mariozechner/*` packages and adds `@mariozechner/pi-ai@0.69.0`, while current main uses `@earendil-works/*@0.74.0`. That installs a second stale provider runtime and can diverge transport behavior without a compatibility reason.
  - confidence: 0.9
