## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Forward sessionId before relying on the wrapper:** `extensions/zai/index.ts:154`
  - body: This wrapper only injects the header when `options.sessionId` is present, but current main does not pass `sessionId` into the normal provider-owned, boundary-aware, or prompt-cache fallback stream branches. As a result the ZAI agent path this PR is meant to optimize will usually call the wrapper without the value it needs; either include the propagation fix or depend on the generic affinity PR that supplies it.
  - confidence: 0.9
- **[P1] Use an approved opaque affinity key or document raw session IDs:** `extensions/zai/stream.ts:42`
  - body: The new header sends OpenClaw's raw `sessionId` to ZAI. Session ids can be user-supplied or encode task/workspace context, while the existing stream contract already has a separate `promptCacheKey` for provider affinity, so maintainers need an explicit raw-vs-opaque decision before this becomes default provider-visible metadata.
  - confidence: 0.86
- **[P2] Replace existing session headers case-insensitively:** `extensions/zai/stream.ts:40-42`
  - body: Spreading caller headers and then adding `X-Session-Id` leaves an existing `x-session-id` key in place. Fetch header normalization treats those names as the same header and can send a comma-joined value such as `old, new`, which undermines sticky routing or can be rejected by the provider.
  - confidence: 0.88
- **[P2] Reject Fetch-invalid session ids before injection:** `extensions/zai/stream.ts:21`
  - body: `isInjectableSessionId` accepts strings containing CR, LF, or NUL, but creating a request with those header values throws before the provider call is sent. The wrapper should reject invalid HTTP header bytes instead of turning a bad session id into a ZAI transport failure.
  - confidence: 0.85
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:19`
  - body: `CHANGELOG.md` is release-owned in this repository; normal feature PRs should keep release-note context in the PR body or final squash message instead of editing the changelog directly.
  - confidence: 0.9
