## Summary

Keep open. Current main still grants `internalDiagnostics` only to bundled diagnostics exporter services, while official install metadata and docs still expose npm/ClawHub install paths for `diagnostics-prometheus`; the reported npm/global path can therefore load without ever subscribing to diagnostics events.

## What This Changes

The issue asks OpenClaw to stop allowing a nonfunctional npm-installed `diagnostics-prometheus` path that loads but lacks the `internalDiagnostics` capability and returns empty metrics.

## Best Possible Solution

Pick one supported contract and align all surfaces: ship the exporter through a trusted bundled/ClawHub path that receives the capability, or reject/document npm installs as unsupported until an explicit trust mechanism exists.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: npm-installed plugins are discovered as `global`, the service context grants `internalDiagnostics` only to `bundled`, and the Prometheus service returns early when the capability is missing. I did not run the live gateway install because this review had to remain read-only.

## Solution Assessment

No single automatic fix is clearly best yet. The report’s alternatives touch trusted capability policy, packaging/release mechanics, install rejection, and docs, so maintainers should choose the supported trust/install contract before implementation.
