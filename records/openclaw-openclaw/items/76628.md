## Summary

Keep open. Current main still grants `internalDiagnostics` only to bundled diagnostics exporter services while official docs and catalog metadata advertise external install paths; a global/npm install can load and bind the endpoint without ever receiving events.

## What This Changes

The issue requests that the Prometheus diagnostics plugin either work on supported install paths or fail/document unsupported paths instead of loading with empty metrics and error logs.

## Best Possible Solution

Align one supported contract across docs, install metadata, and runtime: either make the exporter available through a trusted path that receives `internalDiagnostics`, or reject/document npm/global installs as unsupported until that trust path exists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: installed plugins are discovered as `global`, the capability gate accepts only `bundled`, and the Prometheus service returns before subscribing when the capability is absent.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on a maintainer trust/distribution decision, because packaging the exporter, rejecting unsupported installs, documenting limitations, or adding a trust mechanism have different security and product tradeoffs.
