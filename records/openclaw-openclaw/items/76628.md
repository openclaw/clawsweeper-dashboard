## Summary

Keep open. Current main still grants `internalDiagnostics` only to bundled diagnostics exporter services, while the official Prometheus plugin metadata and docs still expose external install routes that can load as `global`; that path still reaches the service without the capability and cannot emit metrics.

## What This Changes

Make the official `diagnostics-prometheus` install path either receive the trusted diagnostics capability or fail/document unsupported external installs so the Prometheus endpoint does not return an empty body.

## Best Possible Solution

Choose one supported contract and align code, install metadata, and docs: ship the exporter through a trusted path that receives `internalDiagnostics`, or reject/document external npm installs until explicit official-source trust exists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: installed/global plugins are discovered as `global`, the service context grants `internalDiagnostics` only to bundled diagnostics exporters, and the Prometheus service returns before subscribing when the capability is missing; I did not run the gateway because this review is read-only.

## Solution Assessment

Unclear. No single automatic fix is clearly best because the repair crosses privileged capability policy, packaging/ClawHub release behavior, install rejection, and docs; maintainers should pick the supported trust/install contract first.
