## Summary

Keep open. Current main and the latest tagged release still join every eligible assistant text block before user-visible delivery, and the discussion confirms this remains a standalone delivery-policy bug rather than an implemented or superseded request.

## What This Changes

The issue requests a delivery-layer policy for assistant messages with multiple eligible text content items so user-facing channels send one selected block instead of a concatenation.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, preserve shared transcript extraction as concatenation by default, and cover multi-text assistant messages plus unchanged transcript/TUI/history behavior in tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a synthetic assistant message with two eligible `text` blocks flows through the newline join in `extractAssistantVisibleText` and then through `handleMessageEnd` delivery. I did not need to execute tests to establish that path.

## Solution Assessment

Unclear. A delivery-scoped selector appears to be the narrowest maintainable direction, but the exact pick-one rule is still a maintainer policy choice before a repair should encode it.
