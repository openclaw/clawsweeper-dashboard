## Summary

Keep open: current main and v2026.6.10 still join multiple eligible assistant text blocks before final delivery, while the related commentary/final-answer leak was fixed separately and does not cover this same-phase pick-one policy.

## What This Changes

The issue requests a maintainer-approved policy for user-facing delivery when one assistant message contains multiple eligible text blocks.

## Best Possible Solution

Add a maintainer-approved delivery-boundary selector for multiple eligible final text items while preserving shared transcript/history concatenation and adding regression coverage for both surfaces.

## Reproduction Assessment

Yes, at source level: a message with multiple matching assistant text blocks is still extracted by joining all matches, and final delivery consumes that joined text. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the selector policy. A delivery-only selector is the narrower maintainable direction; changing the shared extractor default would broaden the blast radius across transcript and display surfaces.
