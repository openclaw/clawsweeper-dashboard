## Summary

Keep open. Current main and the latest tagged release still join every eligible assistant text block before delivery, and the discussion confirms this is a standalone delivery-policy bug rather than a duplicate or shipped fix.

## What This Changes

Requests a delivery-layer policy that selects one assistant text block for channel-visible replies instead of concatenating multiple text content items.

## Best Possible Solution

Add a delivery-scoped selector before channel emission that chooses a deterministic public reply text block while leaving shared transcript extraction concatenating by default, with regression coverage for both delivery and non-delivery consumers.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks flows through extractAssistantVisibleText into handleMessageEnd and produces newline-joined text for delivery; no private Slack transcript is needed.

## Solution Assessment

Unclear. Delivery-scoped selection is the narrowest maintainable direction, but the exact selector rule still needs maintainer choice before encoding channel-visible behavior.
