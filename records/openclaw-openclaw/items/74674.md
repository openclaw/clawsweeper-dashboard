## Summary

Keep open. Current main still routes channel-visible assistant text through helpers that join every eligible text block, while transcript-oriented surfaces still rely on joined output, so this remains a valid delivery-policy bug rather than cleanup that can close.

## What This Changes

Request a delivery-layer policy that selects one user-visible assistant text block when a turn contains multiple eligible text content items, instead of concatenating them into a single channel reply.

## Best Possible Solution

Add an explicit delivery-boundary text selector for embedded channel delivery, preserve shared transcript extraction as concatenation by default, and cover same-phase multi-text assistant messages in delivery tests plus unchanged transcript/TUI/history behavior.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show the delivered text is joined; no private Slack transcript is required.

## Solution Assessment

Unclear. A delivery-only selector is the narrow maintainable direction, but the exact pick-one rule still needs owner choice before an automated repair should encode policy.
