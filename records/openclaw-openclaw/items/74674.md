## Summary

Keep open. Current main still routes user-visible final delivery through helpers that join all eligible assistant text blocks, and the latest release tag still has the same delivery path. The discussion confirms this remains a standalone delivery-policy bug, not a resolved duplicate of the related raw-error report.

## What This Changes

The issue requests a delivery-layer policy so Slack/Telegram final replies choose one assistant text block instead of concatenating multiple text content items.

## Best Possible Solution

Add a delivery-scoped selector before channel emission while preserving concatenation for transcript, TUI, gateway dump, and chat-history surfaces, then cover both paths with focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an assistant message with two eligible `text` blocks flows through the newline join in `extractAssistantVisibleText` and then through `handleMessageEnd` delivery.

## Solution Assessment

Unclear. A delivery-scoped selector appears to be the narrowest maintainable fix, but the specific pick-one policy should be chosen by maintainers before a repair encodes it.
