## Summary

Keep open. Current main still joins every eligible assistant text block before channel-visible delivery, the latest tagged release has the same path, and the discussion confirms this is a standalone delivery-policy bug rather than cleanup that can close.

## What This Changes

The issue asks for a delivery-layer pick-one policy so multi-text assistant messages do not post concatenated near-duplicate text to user-facing channels.

## Best Possible Solution

Add an explicit delivery-scoped selector before joined text reaches channel adapters, preserve shared transcript extraction as concatenation by default, and cover multi-text assistant messages in delivery tests plus unchanged transcript/TUI/history behavior.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show the delivered text is joined; no private Slack transcript is required.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrow maintainable direction, but the exact pick-one rule still needs owner choice before an automated repair should encode policy.
