## Summary

Keep open. Current main still joins all eligible assistant text blocks in the shared and embedded visible-text extractors, and the embedded message-end delivery path consumes that joined string; transcript-facing surfaces still have explicit joined-output expectations, so this remains a real scoped delivery-policy bug rather than cleanup that can close.

## What This Changes

The issue requests a deterministic user-facing delivery policy that selects one visible assistant text item instead of concatenating multiple same-phase text content blocks.

## Best Possible Solution

Add an explicit delivery-boundary selector for user-visible assistant text while preserving shared transcript extraction as concatenation by default, then cover same-phase multi-text assistant messages in embedded delivery tests and keep TUI/gateway/history behavior unchanged.

## Reproduction Assessment

Yes. A synthetic assistant message containing two eligible `text` blocks is enough to trace current main from `extractAssistantVisibleText` through `handleMessageEnd`, where the joined string is used for delivery.

## Solution Assessment

Unclear until maintainers choose the exact selector rule. The narrow maintainable direction is a delivery/content-classification selector, not changing the shared extractor default used by transcript-oriented surfaces.
