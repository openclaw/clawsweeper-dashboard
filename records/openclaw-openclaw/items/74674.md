## Summary

Keep open. Current main and the latest release still join every eligible assistant text block before user-visible delivery, and the discussion confirms this remains standalone delivery-policy work rather than an implemented or superseded item.

## What This Changes

Request a delivery-boundary policy for assistant messages with multiple text content items so channel-visible replies pick one block instead of concatenating near-duplicate variants.

## Best Possible Solution

Add an explicit delivery-scoped selector upstream of channel emission, keep shared transcript extraction as concatenation by default, and cover multi-text delivery plus unchanged TUI/gateway/history behavior in tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a synthetic assistant message with two eligible text blocks is newline-joined by `extractAssistantVisibleText` and then delivered by `handleMessageEnd`.

## Solution Assessment

Unclear. A delivery-scoped selector upstream of the join appears to be the narrowest maintainable direction, but the exact pick-one rule is still a maintainer policy choice.
