## Summary

Keep open. Current main and the latest release still concatenate every eligible assistant text block before final channel delivery, and the discussion confirms this is a standalone delivery-policy bug rather than an implemented or superseded request.

## What This Changes

Request a delivery-scoped policy for assistant messages with multiple text content blocks so user-facing channels send one selected reply instead of concatenating variants.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, keep shared transcript extraction as concatenation by default, and cover multi-text assistant messages plus unchanged TUI/gateway/history behavior in focused tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` blocks is enough to trace current main through `extractAssistantVisibleText` into `handleMessageEnd` and show the final delivery text is joined.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrow maintainable direction, but the exact pick-one rule still needs maintainer choice before a repair should encode policy.
