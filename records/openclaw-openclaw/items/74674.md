## Summary

Keep open. Current main still filters eligible assistant text blocks by phase and joins them before embedded message-end delivery, while related discussion scopes this separately from #69737 and current non-delivery tests still expect joined transcript-style output.

## What This Changes

The issue requests a delivery-boundary policy that selects one user-visible assistant text item from multiple eligible assistant text blocks without changing transcript-oriented joining.

## Best Possible Solution

Add a delivery-boundary text selection helper for embedded channel replies, keep shared transcript extraction concatenating by default, and cover same-phase multi-text assistant messages in delivery tests plus unchanged transcript/TUI/history tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main from extractAssistantVisibleText through handleMessageEnd and show that the delivered visible text is the joined string; no private Slack transcript is required.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrowest maintainable direction appears to be a delivery-only selector upstream of the join, not changing the shared extractor default.
