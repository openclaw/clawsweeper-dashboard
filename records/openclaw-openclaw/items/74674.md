## Summary

Keep open. Current main still concatenates all eligible assistant text blocks before embedded message delivery, while transcript-oriented callers still have tests or call sites that rely on joined text, so this remains a valid delivery-policy bug rather than cleanup that can close.

## What This Changes

Requests a deterministic pick-one policy for user-visible delivery when one assistant message contains multiple eligible text content items, without changing transcript/history-oriented concatenation behavior.

## Best Possible Solution

Add an explicit delivery-boundary text selection helper for embedded channel delivery, leave shared transcript extraction as concatenation by default, and cover same-phase multi-text assistant messages in delivery tests plus unchanged transcript/TUI/history tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text content blocks is enough to trace current main from extractAssistantVisibleText through handleMessageEnd and show that the delivered text is the joined string; no private Slack transcript is required.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrow maintainable direction is delivery-only selection upstream of the join, not a global shared-extractor default change.
