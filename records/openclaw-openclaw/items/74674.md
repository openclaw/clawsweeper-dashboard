## Summary

Keep open. Current main and the latest release still concatenate every eligible assistant text block before user-visible delivery, while the related discussion scopes this as a standalone delivery policy bug rather than a duplicate or implemented request.

## What This Changes

Request a delivery-layer policy so assistant messages with multiple eligible text blocks emit one user-facing reply instead of concatenating variants, while preserving joined transcript/history surfaces.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, keep shared transcript extraction as concatenation by default, and cover multi-text assistant messages plus unchanged transcript/TUI/history behavior in tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` content blocks can be traced through `extractAssistantVisibleText` into `handleMessageEnd`, where the delivered visible text is the newline-joined result.

## Solution Assessment

Unclear. The issue correctly scopes the fix to delivery/public visible selection rather than changing shared defaults, but the exact pick-one rule still needs maintainer choice before implementation.
