## Summary

Keep open: current main and the latest release still join multiple eligible assistant text blocks before final user-facing delivery, and the parent issue closure explicitly left this narrower multi-text policy item open for maintainer choice.

## What This Changes

The issue requests a delivery-layer policy for assistant messages with multiple text content items so user-facing channels do not post concatenated near-duplicate text.

## Best Possible Solution

Add a maintainer-approved delivery-boundary selector for multiple eligible final text items, likely preserving shared transcript/history concatenation and covering delivery plus non-delivery regression tests.

## Reproduction Assessment

Yes, at source level: an assistant message with multiple matching text blocks is still extracted by joining all matching blocks, and final delivery consumes that joined visible text. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the policy, but changing only the delivery boundary is the narrowest maintainable direction. A shared extractor default change would be too broad because tests and callers preserve concatenation for transcript/display/history surfaces.
