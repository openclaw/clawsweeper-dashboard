## Summary

Keep open. Current main and the latest release still route channel-visible assistant delivery through helpers that join all eligible text blocks, and the related issue discussion confirms this is a standalone delivery-policy bug rather than a duplicate of the raw-error or audience-boundary reports.

## What This Changes

The issue requests a delivery-layer policy that selects one assistant text content item instead of posting a newline-joined concatenation when a single assistant message contains multiple visible text blocks.

## Best Possible Solution

Add a delivery-scoped selector before final channel emission, preserve concatenation for transcript/debug/TUI/history surfaces, and cover both delivery and non-delivery behavior with focused regression tests.

## Reproduction Assessment

Yes, source-level. An assistant message with two eligible `text` blocks flows through the newline join in `extractAssistantVisibleText` and then into `handleMessageEnd` delivery.

## Solution Assessment

Unclear. A delivery-scoped selector appears to be the narrowest maintainable fix, but maintainers should choose the public pick-one rule before a patch encodes it.
