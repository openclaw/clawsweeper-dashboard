## Summary

Keep open. Current main and the latest release still join all eligible assistant text blocks before channel delivery, and the issue discussion confirms this is a standalone delivery-policy bug rather than a duplicate of the raw-error or broader audience-boundary reports.

## What This Changes

The issue requests a delivery-layer pick-one policy for assistant messages that contain multiple eligible text content items.

## Best Possible Solution

Add a delivery-scoped selector before channel emission, preserve concatenation for transcript/debug/TUI/history surfaces, and cover both boundaries with focused regressions.

## Reproduction Assessment

Yes, source-level. An assistant message with two eligible text blocks flows through newline joining in extractAssistantVisibleText and then into handleMessageEnd delivery.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrowest maintainable direction, but maintainers should choose the public pick-one rule before a patch encodes it.
