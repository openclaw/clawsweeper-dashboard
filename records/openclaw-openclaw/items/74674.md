## Summary

Keep open. Current main still joins all eligible assistant text blocks before the embedded delivery path, and existing transcript/display surfaces still encode joined-text behavior, so the requested delivery-only selection policy is not implemented or superseded.

## What This Changes

The issue asks OpenClaw to stop concatenating multiple assistant text content items at channel delivery and instead apply an explicit pick-one policy while preserving transcript-style joining elsewhere.

## Best Possible Solution

Add an explicit delivery-boundary selector after maintainers choose the policy, preserve shared transcript extraction as concatenation by default, and cover same-phase multi-text assistant messages in embedded delivery tests while keeping transcript/TUI/history behavior unchanged.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` blocks is enough to trace current main from `extractAssistantVisibleText` into `handleMessageEnd`, where the joined string is used for delivery; no private Slack data is required.

## Solution Assessment

Unclear until the selector rule is chosen. The narrow maintainable direction is a delivery-only selector, not changing the shared extractor default that other surfaces already rely on.
