## Summary

Keep open. Current main still joins all eligible assistant text blocks in the embedded and shared visible-text extractors, and the message-end delivery path consumes that joined string; the follow-up discussion confirms this remains a valid standalone delivery-policy bug rather than stale cleanup.

## What This Changes

Define a delivery-layer policy so embedded channel replies select one assistant text item instead of posting multiple joined text blocks, while preserving concatenation for transcript-oriented surfaces.

## Best Possible Solution

Add an explicit delivery-boundary text selector before the newline join, preserve shared transcript/history extraction as concatenation by default, and cover same-phase multi-text assistant messages in delivery tests plus unchanged transcript/TUI/history behavior.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show the delivered text is joined; no private Slack transcript is required.

## Solution Assessment

Unclear. A delivery-only selector upstream of the join appears to be the narrow maintainable direction, but the exact selector policy still needs maintainer choice before an automated repair encodes it.
