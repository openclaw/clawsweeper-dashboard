## Summary

Keep open. Current main still feeds newline-joined assistant text blocks into channel-visible delivery, while transcript-style surfaces retain concatenation as expected behavior, so this remains a valid delivery-policy bug.

## What This Changes

Request a delivery-layer policy so channel-visible replies select one assistant text item instead of concatenating multiple eligible assistant text blocks.

## Best Possible Solution

Add an explicit delivery-boundary selector/classifier that emits one public assistant text block, while preserving concatenation for transcript, TUI, gateway, and history surfaces and covering both paths with focused tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show the delivered text is newline-joined before channel emission.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrow maintainable direction, but the exact pick-one rule should be chosen by an owner before automation encodes policy.
