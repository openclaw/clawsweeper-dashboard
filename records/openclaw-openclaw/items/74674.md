- `impact:message-loss`: The issue concerns duplicated or incorrectly combined channel-visible messages when one assistant turn contains multiple visible text blocks.

## Summary

Keep open: current main and the latest release still concatenate eligible assistant text blocks before channel delivery, and the related raw-error issue was closed with this follow-up explicitly left open. The remaining work is a bounded delivery-policy decision, not stale cleanup.

## What This Changes

The issue requests a delivery-boundary policy so assistant messages with multiple visible text blocks do not get concatenated into one user-facing channel reply.

## Best Possible Solution

Add a delivery-only visible-text selector with focused regression coverage so channel replies emit one chosen final text block while transcript/history-style extractors keep concatenation unless maintainers explicitly choose a shared selector API.

## Reproduction Assessment

Yes, at source level. A message with two eligible text blocks is still joined by the embedded/shared extractors and then used by the message-end delivery path; I did not run a live Slack reproduction.

## Solution Assessment

Unclear until maintainers choose the policy. The narrowest maintainable direction appears to be delivery-only selection with regression tests that prove transcript/history callers keep their existing concatenation behavior.
