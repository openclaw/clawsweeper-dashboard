## Summary

Keep open. Current main still concatenates same-phase assistant text blocks before the embedded delivery boundary, and the surrounding tests confirm that transcript/display surfaces intentionally preserve joined multi-block text, so this remains a valid delivery-policy bug rather than cleanup that can be closed.

## What This Changes

Request a delivery-layer policy that selects one assistant text content item for user-facing channel posts instead of concatenating multiple same-phase text items.

## Best Possible Solution

Add an explicit delivery-boundary text selector before the join, preserving shared transcript extraction as concatenation by default, then cover same-phase multi-text assistant messages in embedded delivery tests while keeping TUI, gateway, and chat-history behavior unchanged.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` blocks is enough to trace the current source path through `extractAssistantVisibleText` and `handleMessageEnd` to a joined delivery string; no private Slack transcript is needed.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrow maintainable direction appears to be delivery-scoped selection, preferably at content-type classification before the join, rather than changing the shared extractor default.
