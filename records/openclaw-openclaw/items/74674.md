## Summary

Keep open: current main and v2026.6.10 still concatenate all eligible assistant text blocks before final channel delivery, and the parent issue was closed while explicitly leaving this policy follow-up open.

## What This Changes

Request to define a delivery-layer policy that selects one eligible assistant text content item instead of posting the concatenation when multiple text blocks are present.

## Best Possible Solution

Adopt a maintainer-approved delivery-boundary pick-one rule for multiple eligible final text items while preserving shared transcript, history, gateway, and TUI concatenation behavior.

## Reproduction Assessment

Yes, at source level: an assistant message with multiple matching text blocks is still extracted by joining all matching blocks, and final delivery consumes that joined visible text. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the policy. A delivery-boundary selector is the narrowest maintainable direction, while changing the shared extractor default would be too broad for current non-delivery callers and tests.
