## Summary

Keep open: current main and v2026.6.10 still join all eligible assistant text blocks before final user-facing delivery, and the remaining work is the maintainer policy choice for the delivery boundary rather than a stale or already-fixed report.

## What This Changes

The issue asks for a delivery-boundary policy that chooses one visible assistant text item instead of concatenating multiple text content items into a single channel reply.

## Best Possible Solution

Add a maintainer-approved delivery-only selector for multiple eligible final text items while preserving shared concatenation semantics for transcript and history surfaces, with focused regression coverage for both paths.

## Reproduction Assessment

Yes, at source level: an assistant message with multiple matching text blocks is still extracted by joining all matching blocks, and final delivery consumes that joined visible text. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the policy; a delivery-only selector is the narrowest maintainable direction. A shared extractor default change would be too broad because current callers and tests preserve concatenation outside delivery.
