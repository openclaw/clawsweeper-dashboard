## Summary

Keep open: current main and the latest release still join multiple eligible assistant text blocks before user-facing delivery, and the related discussion intentionally left this narrower multi-text policy issue open for maintainer choice.

## What This Changes

The issue requests a delivery-boundary policy so channel replies choose one assistant text item instead of posting concatenated multi-text variants.

## Best Possible Solution

Implement a narrow delivery-boundary selector, preserve existing concat semantics for transcript/debug surfaces unless maintainers explicitly change them, and add regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection on current main shows a message with multiple eligible assistant text blocks is still joined by the embedded/shared extractors and then used by the message-end delivery path; I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the policy. The narrowest maintainable direction appears to be delivery-only selection with tests proving transcript/history callers keep their existing concatenation behavior.
