## Summary

Keep open. Current main and the latest release still concatenate every eligible assistant text block before delivery, and the open audience-boundary thread is broader rather than a clear superseding fix for this narrower delivery-policy bug.

## What This Changes

Issue requests a delivery-boundary policy that selects one user-visible assistant text item instead of posting newline-concatenated multiple text content items.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, keep shared transcript extraction as concatenation by default, and cover duplicate multi-text delivery plus unchanged TUI/history/session behavior in tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` content blocks is enough to trace current main through `extractAssistantVisibleText` into `handleMessageEnd` and show the delivered visible text is joined.

## Solution Assessment

Unclear. A delivery-only override is the narrowest maintainable direction, but the exact pick-one rule is a user-visible policy choice that should be selected before an automated repair encodes it.
