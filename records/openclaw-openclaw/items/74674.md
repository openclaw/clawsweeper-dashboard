## Summary

Keep open. Current main and the latest tagged release still join every eligible assistant text block before user-visible delivery, and the related discussion confirms this remains a standalone delivery-policy bug rather than an implemented or superseded request.

## What This Changes

The issue asks for a delivery-layer policy that selects one assistant text item instead of concatenating multiple text content blocks in user-visible channel replies.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, preserve shared transcript extraction as concatenation by default, and cover multi-text assistant messages plus unchanged transcript/TUI/history behavior in tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` content blocks is enough to trace current main through `extractAssistantVisibleText` into `handleMessageEnd` and show the delivered visible text is joined.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrow maintainable direction, but the exact pick-one rule still needs maintainer choice before a repair should encode policy.
