## Summary

Keep open: current main and latest release still join multiple eligible assistant text blocks before visible delivery, and the related raw-error issue was closed with this multi-text policy follow-up explicitly left open. The remaining work is real but needs a maintainer decision on delivery selection semantics and sibling-surface preservation.

## What This Changes

The issue asks OpenClaw to stop concatenating multiple assistant text content items at the user-facing delivery boundary and choose an explicit pick-one policy while preserving transcript-like callers.

## Best Possible Solution

Adopt a maintainer-approved delivery-boundary selector, likely last non-empty text unless maintainers choose otherwise, while leaving shared transcript/display extractors concatenating by default and adding focused regression coverage for both delivery and non-delivery callers.

## Reproduction Assessment

Yes, source-level. On current main, a single assistant message with multiple eligible text blocks still flows through extractAssistantVisibleText, joins all matching parts, and handleMessageEnd uses that joined value for visible delivery.

## Solution Assessment

No merged solution is present yet. The best fix is not a broad shared-default change; it should be a narrow delivery-boundary selection policy or explicit selector that preserves transcript-style callers.
