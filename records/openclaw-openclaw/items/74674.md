## Summary

Keep open. Current main still joins all eligible assistant text blocks before the embedded delivery path emits the final user-facing reply, while transcript/display surfaces still have tests or call sites that depend on joined text behavior. The related reports in the provided context do not supersede this narrower selector-policy bug; martingarramon's follow-up confirms it is a standalone delivery-side/content-classification target.

## What This Changes

The issue requests a deterministic delivery-boundary policy so assistant messages with multiple visible text content items are not concatenated into one user-facing channel reply.

## Best Possible Solution

Add an explicit delivery-boundary selector upstream of the join, preserving shared transcript extraction as concatenation by default. Cover same-phase multi-text assistant messages in embedded delivery tests and keep TUI, gateway preview, and chat-history behavior unchanged unless a maintainer explicitly chooses a broader contract change.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible same-phase `text` blocks is enough to trace the current source path from extraction through `handleMessageEnd` delivery; no private Slack transcript is needed.

## Solution Assessment

Unclear until maintainers choose the selector policy. The narrow maintainable direction appears to be a delivery-only/content-classification selector rather than changing the shared extractor default.
