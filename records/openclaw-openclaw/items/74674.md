## Summary

Keep open. Current main and v2026.5.2 still join every eligible assistant text block before message-end delivery, while non-delivery surfaces still rely on concatenated text, so the issue remains valid and needs a maintainer selector-policy decision.

## What This Changes

The issue requests a delivery-layer policy that emits one assistant text block instead of concatenating multiple eligible text content items into the user-facing channel reply.

## Best Possible Solution

Add an explicit delivery-scoped selector before channel emission, preserve shared transcript/TUI/history concatenation by default, and cover both multi-text delivery and unchanged non-delivery behavior in regression tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` blocks follows the source path from newline join in `extractAssistantVisibleText` into `handleMessageEnd` final delivery; I did not need live Slack access to establish that path.

## Solution Assessment

Unclear. A delivery-scoped selector appears to be the narrowest maintainable fix, but the exact pick-one rule is still a maintainer policy choice.
