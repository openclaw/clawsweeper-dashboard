## Summary

Keep open. Current main still joins all eligible assistant text blocks before embedded channel delivery, while existing transcript/TUI-oriented paths still expect joined text; the remaining work is a real delivery-policy bug, not cleanup that can close.

## What This Changes

The issue requests a delivery-layer policy that avoids concatenating multiple assistant text content items into one user-facing channel message while preserving transcript-oriented concatenation elsewhere.

## Best Possible Solution

Add an explicit delivery-boundary selector for embedded channel replies, keep shared transcript extraction concatenating by default, and cover both same-phase multi-text delivery and unchanged transcript/TUI/history behavior with focused regression tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text content blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show that the delivery text is the joined string; no private Slack transcript is required.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrow maintainable direction is a delivery-only selection helper upstream of the join, not a global shared-extractor behavior change.
