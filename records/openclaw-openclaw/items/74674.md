- `impact:message-loss`: The issue concerns duplicated or incorrectly combined channel-visible messages when one assistant turn contains multiple visible text blocks.

## Summary

Keep open: current main and the latest release still join multiple eligible assistant text blocks before channel delivery, and the related raw-error issue was closed with this follow-up intentionally left open. The remaining work is a bounded delivery-policy decision, not stale cleanup or a duplicate of the open provider snapshot-collapse PR.

## What This Changes

The issue requests a delivery-layer pick-one policy for assistant messages with multiple eligible text content items while preserving transcript-style concatenation elsewhere.

## Best Possible Solution

Add a delivery-boundary visible-text selector with focused regression coverage so channel replies emit one chosen final text block while transcript, TUI, gateway, and chat-history extractors keep concatenation unless maintainers intentionally change that contract.

## Reproduction Assessment

Yes, at source level. A message with multiple eligible text blocks is still joined by the embedded/shared extractors and then used by the message-end delivery path; I did not run a live Slack reproduction.

## Solution Assessment

Unclear until maintainers choose the policy. The narrowest maintainable direction appears to be a delivery-only selector plus regressions proving non-delivery transcript-style callers keep their existing concatenation behavior.
