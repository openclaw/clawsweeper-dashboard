## Summary

Keep open. Current main still concatenates every eligible assistant text block before channel-visible delivery, while transcript-style callers still rely on joined output, so this remains a valid delivery-policy bug rather than conservative cleanup.

## What This Changes

The issue requests a delivery-layer policy for assistant messages with multiple visible text blocks so channel output does not concatenate near-duplicate public replies.

## Best Possible Solution

Add an explicit delivery-scoped public reply selector before channel adapters, keep shared transcript extraction concatenating by default, and cover multi-text delivery plus unchanged transcript/TUI/history behavior.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible text blocks is enough to trace current main through extractAssistantVisibleText into handleMessageEnd and show the delivered text is joined; no private Slack transcript is required.

## Solution Assessment

Unclear. A delivery-scoped selector is the narrow maintainable direction, but the exact pick-one rule still needs owner choice before automation should encode policy.
