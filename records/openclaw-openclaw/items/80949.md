## Summary

Keep open. The branch targets a real shared delivery diagnostic gap, but the proposed warning is emitted as an internal-only trusted `log.record`, so it is not operator-visible through the normal logger or public diagnostic stream; maintainers should also reconcile it with the overlapping open diagnostic PR.

## What This Changes

The PR adds an aggregate warning path and reply-dispatch tests for substantive final replies suppressed by `message_tool_only` delivery.

## Best Possible Solution

Land one canonical privacy-preserving diagnostic that is visible to operators by default while preserving the no-auto-forward contract and never logging suppressed final text.

## Reproduction Assessment

Yes, at source level. Current main resolves group/channel turns to `message_tool_only`, sets delivery suppression, and skips ordinary final payloads; linked Telegram and Slack reports provide live examples of the same successful-but-invisible turn class.

## Solution Assessment

No. The shared dispatch boundary and aggregate metadata are the right direction, but emitting a trusted internal `log.record` does not create the operator-facing warning the PR is intended to add.

## Review Findings
