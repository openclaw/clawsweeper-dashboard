## What This Changes

The issue reports that the macOS VoiceWakeOverlay can pinwheel because an attributed-text SwiftUI change path synchronously recalculates layout and mutates observed overflow state.

## Best Possible Solution

Make overlay height measurement pure or safely coalesced, update overflow state outside the synchronous SwiftUI view-update path with a no-op guard, and keep the WorkActivityStore menubar watchdog work separate.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/43480. Current main has VoiceWakeOverlayView synchronously calling updateWindowFrame from onChange(of: controller.model.attributed), and measuredHeight writes controller.model.isOverflowing during the resulting layout calculation. Keep the fix scoped to the VoiceWakeOverlay controller/view path: make height measurement pure or safely coalesce overflow-state writes outside the SwiftUI view-update cycle with a no-op guard, and defer/coalesce frame updates so transcript changes cannot re-enter multiple times per frame. Add focused regression coverage where feasible around overflow/frame updates not synchronously mutating observed state from the view update path. Do not change WorkActivityStore or the unrelated menubar watchdog PR. Include a changelog entry if the fix is user-visible, and capture macOS manual proof before landing: repeatedly trigger Voice Wake or push-to-talk overlay and confirm no pinwheel plus no SwiftUI 'Modifying state during view update' or 'onChange ... multiple times per frame' log spam.
