## What This Changes

The issue reports a macOS VoiceWake overlay SwiftUI render loop that can pinwheel the app when attributed text changes trigger synchronous window resizing.

## Best Possible Solution

Land a focused macOS overlay fix that keeps layout measurement free of unnecessary observable side effects, defers or coalesces resize work outside SwiftUI's active update transaction, and proves the running overlay no longer warns or pinwheels.

## Reproduction Assessment

Yes at the source level: current main still has the reported `onChange` to resize to measurement to observable-state-write chain. I did not reproduce the intermittent pinwheel in a signed macOS app during this read-only review.

## Solution Assessment

Yes in direction: separating observable mutation from synchronous layout measurement is the narrow maintainable fix. The final repair should also coalesce or guard deferred resize work and include real overlay proof.

## Repair Work Prompt

Fix the macOS VoiceWake overlay SwiftUI render loop described in https://github.com/openclaw/openclaw/issues/43480. Keep the change narrowly scoped to the VoiceWake overlay window/view path: break the synchronous cycle where `VoiceWakeOverlayView` observes `model.attributed`, calls `updateWindowFrame(animate:)`, and `measuredHeight()` writes `model.isOverflowing` during the same view update. Prefer deferring the resize and avoiding redundant overflow writes, or an equivalent owner-local fix that preserves current overlay sizing behavior. Add focused regression coverage in the macOS tests for unchanged overflow measurements and resize scheduling behavior where feasible. Do not broaden voice session behavior, gateway behavior, app lifecycle, config, or unrelated overlays. Do not edit CHANGELOG.md.
