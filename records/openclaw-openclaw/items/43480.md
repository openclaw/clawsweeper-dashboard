## Summary

Keep open: current main still contains the reported VoiceWakeOverlay update cycle, and the linked menubar-freeze PR is a different WorkActivityStore issue rather than a fix for this overlay path.

## What This Changes

The issue reports a SwiftUI render loop in the macOS VoiceWakeOverlay where synchronous layout measurement updates observable overflow state and can freeze the app.

## Best Possible Solution

Make the overlay height calculation pure or safely coalesced, update overflow state outside the synchronous SwiftUI view-update path with a no-op guard, and add focused regression coverage plus macOS manual proof for repeated voice transcript updates.

## Reproduction Assessment

Yes, at source level. Current main still has `onChange(of: model.attributed)` calling `updateWindowFrame`, which reaches `measuredHeight()` and synchronously writes observable `model.isOverflowing`; live macOS proof was not established in this Linux review environment.

## Solution Assessment

Yes, a narrow overlay fix is the right direction. The best patch should break this synchronous state/update cycle in `VoiceWakeOverlayView` and `VoiceWakeOverlayController+Window` rather than rely on the separate WorkActivityStore menubar watchdog PR.

## Repair Work Prompt

Fix the macOS VoiceWakeOverlay SwiftUI render loop reported in https://github.com/openclaw/openclaw/issues/43480. On current main, `VoiceWakeOverlayView` calls `controller.updateWindowFrame(animate: true)` from `onChange(of: controller.model.attributed)`, and `VoiceWakeOverlayController+Window.measuredHeight()` synchronously mutates observable `model.isOverflowing`; break that synchronous update cycle without changing the overlay feature surface. Keep the fix scoped to the macOS voice wake overlay, preserve overflow behavior, avoid broad UI refactors and avoid changing WorkActivityStore/menubar stale-state policy. Add focused regression coverage or a narrow test seam where feasible, and add a user-facing changelog entry because this is a macOS app freeze fix. Validate with the narrow macOS Swift overlay tests plus a real macOS manual check that triggers the voice wake overlay repeatedly and shows no SwiftUI modifying-state/onChange multiple-times-per-frame warnings.
