## What This Changes

The issue reports a macOS VoiceWakeOverlay SwiftUI render loop where attributed-text changes trigger window measurement that synchronously mutates observed overflow state, causing intermittent pinwheel hangs.

## Best Possible Solution

Make overlay height measurement side-effect-free or safely coalesced, update overflow state outside SwiftUI's synchronous render/update cycle with equality guards, and verify the real macOS overlay no longer pinwheels or emits update-loop warnings.

## Reproduction Assessment

Yes at source level: current main still routes attributed changes into updateWindowFrame, which calls measuredHeight and synchronously writes observed isOverflowing state. I did not run a live macOS pinwheel reproduction in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow VoiceWakeOverlay controller/view change that removes synchronous observed-state mutation from the render/update path. The related WorkActivityStore watchdog PR addresses a separate menu-bar display-state problem.
