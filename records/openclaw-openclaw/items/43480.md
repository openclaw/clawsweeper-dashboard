## Repair Work Prompt

Fix the macOS VoiceWakeOverlay SwiftUI render-loop freeze reported in https://github.com/openclaw/openclaw/issues/43480. Keep the repair in the overlay owner surface: `VoiceWakeOverlayView.swift`, `VoiceWakeOverlayController+Window.swift`, and focused tests under `apps/macos/Tests/OpenClawIPCTests`. Break the synchronous loop where `onChange(of: controller.model.attributed)` calls `updateWindowFrame`, `targetFrame()` calls `measuredHeight()`, and `measuredHeight()` mutates `model.isOverflowing`. Prefer making height measurement side-effect free, or moving the overflow state update into a guarded/coalesced MainActor update that only runs after the SwiftUI view update cycle and only when the value changes. Also defer/coalesce attributed-text frame changes if needed. Do not treat #60990 as the fix; it is menubar activity-state work. Add focused regression coverage for the overflow/update coalescing behavior, update `CHANGELOG.md` with a single-line user-facing fix entry crediting `@gdiab` if appropriate, and avoid changing unrelated voice wake, gateway, provider, or menubar behavior.
