Keep open: current main and the latest shipped tag still contain the reported VoiceWakeOverlay synchronous update cycle, and the related menubar-freeze PR is a separate WorkActivityStore change.

## What This Changes

The issue reports a macOS VoiceWakeOverlay SwiftUI render loop caused by synchronous window-frame updates and observable overflow-state mutation during overlay rendering.

## Best Possible Solution

Make overlay height measurement pure or safely coalesced, update overflow state outside the synchronous SwiftUI view-update path with a no-op guard, and add focused regression coverage plus macOS manual proof.

## Reproduction Assessment

Yes, at source level. Current main still has `onChange(of: model.attributed)` calling `updateWindowFrame`, which reaches `measuredHeight()` and synchronously writes observable `model.isOverflowing`; live macOS proof was not established in this review.

## Solution Assessment

Yes, a narrow overlay fix is the right direction. The separate menubar WorkActivityStore PR does not touch this overlay path, so it should not be treated as the solution for this issue.
