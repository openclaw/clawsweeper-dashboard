## Summary

Keep open: current main still has a source-reproducible path to the reported typing-with-no-visible-reply symptom, but the Feishu-specific expectation conflicts with the documented generic group visible-reply policy, so this needs maintainer policy/docs review rather than cleanup closure.

## What This Changes

The issue asks for Feishu group @mentions to send the generated agent response back to the group instead of ending with typing and `queuedFinal=false, replies=0`.

## Best Possible Solution

Maintainers should decide whether Feishu group @mentions should override the generic room visible-reply default, then either document the required `messages.groupChat.visibleReplies: "automatic"` setting on the Feishu page or add a targeted Feishu behavior change with regression coverage.

No, not for the reporter's exact live setup. Source and tests show a default group/channel path that can produce typing plus `queuedFinal=false` with no final reply, but the issue lacks effective visible-reply and tool-policy config for a high-confidence live Feishu reproduction.

## Solution Assessment
