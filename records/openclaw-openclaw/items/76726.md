## Summary

Keep open: current `main` still has a source-reproducible path to the reported Feishu symptom, but the behavior is tied to the generic group visible-reply policy and needs maintainer policy/docs judgment rather than cleanup closure.

## What This Changes

Feishu group @mentions are reported to start typing and run the agent but produce no visible group reply, with `queuedFinal=false` and `replies=0`.

## Best Possible Solution

Maintainers should choose the intended Feishu group policy, then either add a targeted automatic visible-delivery path with regression coverage or update Feishu troubleshooting to direct users to `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

No, not for the reporter's exact setup. Source inspection shows the default-config path that produces typing plus `queuedFinal=false`/`replies=0`, but the issue lacks effective visible-reply/tool-policy config and I did not run a live Feishu group test.

## Solution Assessment

Unclear. Forcing Feishu group @mentions to auto-post is not obviously the narrowest fix because it conflicts with the documented generic group visible-reply default; the maintainable path depends on a maintainer policy choice.
