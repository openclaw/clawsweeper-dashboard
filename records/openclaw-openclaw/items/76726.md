## Summary

Keep open: current main still has a source-backed path where a Feishu group @mention is processed, typing remains visible, and normal final delivery is suppressed under the generic group visible-reply default. The remaining question is whether Feishu @mentions should follow that generic policy or get Feishu-specific automatic visible replies.

## What This Changes

Request Feishu group @mentions that produce an agent response to deliver that response back to the group instead of ending with `replies=0`.

## Best Possible Solution

Maintainers should decide whether Feishu group @mentions should override the generic group visible-reply default; then either add a targeted Feishu automatic-reply path with regression coverage or update Feishu troubleshooting docs to point users at `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

No, not at high confidence for the reporter's exact setup. Source inspection gives a clear default-config path to typing plus `queuedFinal=false, replies=0`, but the issue lacks effective visible-reply and tool-policy config for a live Feishu reproduction.

## Solution Assessment

Unclear. Forcing automatic Feishu group replies would override the current generic group/channel policy, so the maintainable path needs a maintainer policy decision before a code or docs change.
