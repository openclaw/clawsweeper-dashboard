## Summary

Keep open: the bug is real on current main, but this external PR still has blocking proof and implementation gaps around shared media URI validation and production `workspaceDir` coverage.

## What This Changes

The PR adds image-tool handling, tests, and a changelog entry for loading bare and `MEDIA:`-prefixed `media://inbound/<id>` refs from inbound media storage.

## Best Possible Solution

Land a rebased or replacement patch that preserves or resolves `media://` managed-media refs before workspace or sandbox path normalization, uses the shared media-reference/web-media contract, and adds `workspaceDir` regression coverage.

## Reproduction Assessment

Yes, at source level. Current main passes `workspaceDir` into `createImageTool`, classifies `media://` as supported, and can still resolve it into a workspace `media:/inbound/...` path before `loadWebMedia` can hydrate managed media.

## Solution Assessment

No. The PR direction is useful, but the current branch should be rebased or replaced to use the shared inbound media resolver and to cover the `workspaceDir` agent-session path instead of adding a separate custom parser.
