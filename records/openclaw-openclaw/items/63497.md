## Summary

Keep open. The bug target is valid on current main, but this PR should not merge as-is because it adds a custom `media://inbound` parser at a shared media access boundary, lacks after-fix real behavior proof, and overlaps with other open media URI fixes that need sequencing.

## What This Changes

This PR adds image-tool handling, tests, and a changelog entry for bare and `MEDIA:`-prefixed `media://inbound/<id>` image references.

## Best Possible Solution

Land a rebased or replacement fix that preserves managed `media://` refs through the shared media-reference/loadWebMedia contract before workspace or sandbox path handling, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main passes `workspaceDir` into `createImageTool`, and the image tool can resolve `media://inbound/<id>` under that workspace before `loadWebMedia` sees the managed URI; a direct path check produces the reported `media:/inbound` shape.

## Solution Assessment

No. The PR targets the right failure, but the maintainable fix should reuse the shared inbound media resolver/loader contract instead of adding a custom regex/direct `resolveMediaBufferPath` parser on the access boundary.
