## Summary

This PR should stay open because it addresses a real image-tool/media URI bug, but it is not merge-ready: current `main` still has the image tool passing `media://inbound/...` into `loadWebMedia` with `workspaceDir`, the PR adds a separate inbound URI parser instead of using the shared media-reference resolver, and there is still no after-fix real behavior proof for the Telegram/control-UI path that motivated the change.

## What This Changes

The PR changes `src/agents/tools/image-tool.ts` to accept bare and `MEDIA:`-prefixed `media://inbound/<id>` image references, adds image-tool regression tests, and adds a changelog fix entry.

## Best Possible Solution

Rebase or replace this PR with a fix that routes `media://inbound` normalization through the shared media-reference/media-store resolver, keeps the image-tool tests, and includes redacted real behavior proof for the image-tool or Telegram/control-UI flow.

## Reproduction Assessment

Yes at source level: current `main` can pass `media://inbound/<id>` through `createImageTool` into `loadWebMedia` with `workspaceDir`, and the related reports show the resulting malformed workspace path. I did not run a live Telegram/ZAI reproduction in this read-only review.

## Solution Assessment

No, not as currently patched: the bug should be fixed at or through the shared media-reference/media-store resolver instead of adding a second inbound URI parser in the image tool. The tests are useful and should be kept around the final boundary.
