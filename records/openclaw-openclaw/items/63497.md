## Summary

Keep open. Current main still has a source-level `media://inbound` image-tool failure path when `workspaceDir` is present, and this PR targets a real bug, but the proposed branch is not merge-ready because it adds a custom inbound-media URI parser instead of using the shared media-reference contract and still lacks after-fix real behavior proof.

## What This Changes

The PR adds image-tool handling, tests, and a changelog entry for bare and `MEDIA:`-prefixed `media://inbound/<id>` references.

## Best Possible Solution

Land a rebased or consolidated fix that preserves managed `media://inbound` references through the shared media-reference/loadWebMedia contract before workspace or sandbox path handling, with focused regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create the normal image tool with `workspaceDir` and pass `media://inbound/<id>`; current main can route that value into `resolve(options.workspaceDir, normalizedRef)` before managed-media hydration runs.

## Solution Assessment

No. The PR targets the right failure, but the maintainable fix should reuse `normalizeMediaReferenceSource`/`resolveInboundMediaReference`/`loadWebMedia` semantics instead of adding a custom regex and direct `resolveMediaBufferPath` parser in the image tool.
