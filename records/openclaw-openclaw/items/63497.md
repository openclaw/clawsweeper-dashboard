Keep open. The PR targets a real image-tool bug, but the current diff still adds a custom `media://inbound` parser instead of using the shared media-reference resolver, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds image-tool handling, tests, and a changelog entry for bare and `MEDIA:`-prefixed `media://inbound/<id>` image references.

## Best Possible Solution

Land a rebased or consolidated fix that preserves managed `media://` refs through the shared media-reference/loadWebMedia contract before workspace or sandbox path handling, with focused regression coverage and real behavior proof.

Yes. A high-confidence source-level path is to create the normal image tool with `workspaceDir` and pass `media://inbound/<id>`; current main can turn it into a workspace `media:/inbound` path before managed-media hydration runs.

## Solution Assessment

No. The PR targets the right failure, but the maintainable fix should reuse the shared inbound media-reference/loadWebMedia contract instead of adding a custom regex and direct `resolveMediaBufferPath` parser on the access boundary.
