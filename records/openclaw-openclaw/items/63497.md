## Summary

Keep open. The underlying image-tool bug is still source-reproducible on current main, but this PR should not merge as-is because it lacks real behavior proof, adds a custom media URI parser that diverges from the shared resolver, and misses the normal `workspaceDir` path; related open work at https://github.com/openclaw/openclaw/pull/74186 and https://github.com/openclaw/openclaw/pull/74278 should be sequenced deliberately.

## What This Changes

The PR adds image-tool handling, tests, and a changelog entry for resolving bare and `MEDIA:`-prefixed `media://inbound/<id>` image references.

## Best Possible Solution

Land a rebased or replacement patch that preserves `media://` managed-media refs before workspace or sandbox path normalization, uses the shared media-reference/web-media contract, and adds `workspaceDir` regression coverage.

## Reproduction Assessment

Yes, at source level. Current main passes `workspaceDir` into `createImageTool`, classifies `media://` as supported, and then can resolve it as workspace-relative because the image-tool join does not exclude `refInfo.isMediaStoreUrl`.

## Solution Assessment

No. The PR direction is useful, but the current branch should be rebased or replaced to use the shared inbound media resolver and to cover the `workspaceDir` agent-session path instead of adding a separate custom parser.
