## Summary

Keep open. The PR targets a real image-tool/media-store bug, and current main still has the production `workspaceDir` path that can corrupt `media://inbound/...` into a workspace-relative `media:/inbound/...` path. The branch is useful, but it predates the current shared media-reference helpers and should be updated or replaced rather than closed as implemented.

## What This Changes

This PR teaches the image tool to resolve bare and `MEDIA:`-prefixed `media://inbound/<id>` references from the inbound media store, with image-tool tests and a changelog fix entry.

## Best Possible Solution

Update `src/agents/tools/image-tool.ts` so media-store refs are preserved or resolved before workspace-relative resolution, preferably by using `resolveMediaReferenceLocalPath` or by excluding `refInfo.isMediaStoreUrl` from the `workspaceDir` branch. Add regression coverage through `createImageTool` or `createOpenClawTools` with `workspaceDir` and `fsPolicy: { workspaceOnly: true }` for both bare and `MEDIA:`-prefixed inbound refs, while leaving the sandbox-specific bridge fix coordinated with #74186.

## Reproduction Assessment

Yes. A focused reproduction can create `OPENCLAW_STATE_DIR/media/inbound/<id>`, construct the image tool with a `workspaceDir`, and call it with `media://inbound/<id>`; current main’s path logic deterministically rewrites the URI to a malformed workspace path before media-store resolution.

## Solution Assessment

No. The PR direction is correct, but on current main the maintainable fix should use the shared media-reference contract and cover the production `workspaceDir` path instead of adding a bespoke inbound URI branch and tests that omit the failing composition.
