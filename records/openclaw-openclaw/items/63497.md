## Summary

Keep open. The underlying bug is still valid on current main and this PR targets the right surface, but the branch is currently conflicting, lacks required after-fix real behavior proof, and adds a second media URI parser instead of reusing the shared inbound-media resolver contract.

## What This Changes

The PR adds image-tool handling, tests, and a changelog entry for bare and `MEDIA:`-prefixed `media://inbound/<id>` image references.

## Best Possible Solution

Land a rebased or consolidated fix that preserves managed `media://inbound` references until the shared media-reference/loadWebMedia boundary, covers the normal `workspaceDir` path, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Create the normal image tool with `workspaceDir` and pass `media://inbound/<id>`; current main can route it through `resolve(options.workspaceDir, normalizedRef)`, and a direct Node path check produces the malformed `media:/inbound` path shape.

## Solution Assessment

No. The PR targets the correct bug, but the maintainable fix should reuse the shared media-reference/loadWebMedia semantics instead of adding a custom regex and direct media-store ID parser in the image tool.
