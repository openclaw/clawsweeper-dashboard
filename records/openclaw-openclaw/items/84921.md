## Summary

Keep this open: current main still sends Codex app-server compaction to the native thread path with a finite waiter, while `/compact` and `/status` can surface different context budgets; the exact native timeout needs live Telegram/Codex proof before a safe fix is designed.

## What This Changes

The issue reports long Telegram direct Codex app-server sessions timing out during compaction after `/status` shows a larger context window than `/compact`, leaving the session unrecoverable without `/new`.

## Best Possible Solution

Align Codex runtime status, manual `/compact`, auto-compaction thresholds, and timeout recovery around one trustworthy native-budget model, then surface a specific Telegram recovery message when native compaction cannot complete.

## Reproduction Assessment

No, not as a high-confidence live repro. The source path and sanitized latest-release evidence explain how the failure can occur, but this review did not run a long Telegram direct Codex app-server session to force the native compaction timeout.

## Solution Assessment

Unclear. A hard-coded 200k cap would be brittle; the safer direction is to make Codex native budget selection, early compaction thresholds, and timeout recovery explicit, tested, and visible to the channel user.
