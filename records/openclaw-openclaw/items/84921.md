## Summary

Keep this open: the report describes a current Codex app-server long-session failure that is not implemented away on main, and the source still routes non-context-engine Codex `/compact` through native app-server compaction with a finite waiter and no recovery fallback when that native operation times out.

The issue asks OpenClaw to avoid unrecoverable Codex app-server Telegram sessions when `/status` and `/compact` disagree on the effective context budget and native compaction times out.

## Best Possible Solution

Align Codex runtime status, manual compaction, auto-compaction thresholds, and recovery around one trustworthy native-budget model, then surface a specific Telegram recovery message when native compaction cannot complete.

## Reproduction Assessment

## Solution Assessment

Unclear. A hard-coded 200k cap would be too brittle; the safer direction is to make Codex native budget selection and timeout recovery explicit, tested, and visible to the channel user.
