## Summary

Keep this open: current main and v2026.5.22 include a retry mitigation for native Codex compaction timeouts, but the report's larger problem remains unresolved because native compaction can still fail after the bounded wait, OpenClaw still ignores local compaction model/provider overrides for this path, and related follow-up work is still open.

## What This Changes

The issue reports long Telegram direct sessions on the Codex app-server becoming unrecoverable when `/compact` uses a narrower native budget than the user-facing status window and native compaction times out.

## Best Possible Solution

Define a maintainer-approved Codex compaction policy that aligns status, manual `/compact`, automatic thresholds, and Telegram recovery messaging, then land the focused native-budget/timeout fix alongside the related https://github.com/openclaw/openclaw/issues/86019 and https://github.com/openclaw/openclaw/pull/85958 decisions.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction against current main. The source path and reporter evidence support the failure shape, and tests cover simulated native timeout retry, but the full long Telegram/Codex session timeout was not reproduced here.

## Solution Assessment

Unclear. The shipped retry is only a partial mitigation; the maintainable fix needs an explicit maintainer decision on native Codex budget ownership, force-local/context-engine escape behavior, and the user-facing recovery path.
