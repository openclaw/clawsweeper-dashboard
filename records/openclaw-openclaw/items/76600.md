## Summary

Keep open: the reporter's follow-up supersedes the original raw-string prompt theory, but current main still has a credible Claude ACP failure path when `sessions_spawn` includes `runTimeoutSeconds`. OpenClaw persists that timeout as ACP runtime options, replays it through `session/set_config_option`, and the pinned Claude ACP adapter does not offer a `timeout` config option.

## What This Changes

Make ACP Claude `sessions_spawn` runs with timeouts avoid unsupported timeout config replay and expose the underlying adapter error details.

## Best Possible Solution

Keep the OpenClaw/Gateway turn deadline for `runTimeoutSeconds`, skip or gate unsupported Claude ACP timeout config replay, and add regression coverage plus clearer nested ACP error details.

## Reproduction Assessment

Yes, by source inspection plus the reporter's trace: call `sessions_spawn` with `runtime: "acp"`, `agentId: "claude"`, and a positive `runTimeoutSeconds`; current main stores that as `timeoutSeconds` and replays `timeout` through `session/set_config_option` before the prompt. I did not run a live Claude ACP session in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is in ACP runtime option replay for the acpx/Claude boundary, not in prompt conversion. The published acpx client already normalizes text prompts before JSON-RPC, while Claude ACP rejects the unsupported `timeout` config option.
