## Summary

Close as superseded: the observable Claude ACP `sessions_spawn` failure is already tracked by open PR #75306, and current source/dependency inspection points to the `runTimeoutSeconds` -> `timeout` config-control path rather than OpenClaw sending raw string prompts.

## What This Changes

The issue asks OpenClaw to make Claude ACP `sessions_spawn` runs succeed on 2026.5.2 by sending schema-compatible prompt payloads and exposing the underlying ACP error details.

## Best Possible Solution

Review and land #75306 or an equivalent acpx-runtime fix, preserving the manager-level timeout watchdog while avoiding unsupported Claude ACP session config replay.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the observed run: `sessions_spawn` with `runtime: "acp"` and `runTimeoutSeconds` stores a timeout runtime option, current main forwards it as `session/set_config_option(timeout)`, and Claude ACP 0.31.x rejects unknown config ids. I did not run a live Claude session in this read-only review.

## Solution Assessment

Yes: the best current path is the narrower fix already tracked by #75306, because the published `acpx@0.6.1` client wraps text-only prompts into content-block arrays before `session/prompt`. Prompt normalization should only change if a real OpenClaw payload trace still proves raw strings after the timeout replay is removed.
