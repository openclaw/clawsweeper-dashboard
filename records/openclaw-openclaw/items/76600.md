## Summary

Keep open: the latest trace corrected the suspected raw-string prompt cause and current main still has an actionable ACP Claude timeout-config failure path. The code maps `sessions_spawn` timeouts into a generic `timeout` ACP config option, the acpx wrapper forwards that option for non-Codex agents, and `claude-agent-acp@0.31.4` only exposes mode/model/effort config options.

## What This Changes

The issue asks OpenClaw to fix ACP Claude `sessions_spawn` failures when timeout overrides are replayed as unsupported runtime config and to surface the underlying adapter error.

## Best Possible Solution

Keep the issue open and fix the ACP spawn/control path so `runTimeoutSeconds` remains an OpenClaw turn timeout while unsupported backend config keys are not replayed to Claude ACP, with clearer nested ACP error details.

## Reproduction Assessment

Yes, from source inspection plus the reporter's wrapper trace: pass `runTimeoutSeconds` or `timeoutSeconds` to `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` and current main will persist `timeoutSeconds`, replay `session/set_config_option` as `timeout`, and hit Claude ACP's unsupported config option path before `session/prompt`. I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

No, the original string-prompt normalization is not the best fix after the reporter's trace. The narrow maintainable fix is to gate or avoid timeout config replay for backends that do not advertise/support it while retaining OpenClaw's own turn timeout and improving nested ACP error surfacing.
