## Summary

Keep open: the reported live failure is detailed, but the string-prompt root-cause hypothesis does not match current main plus the pinned `acpx@0.6.1` runtime source. OpenClaw still delegates turns to acpx, and acpx converts string prompts to text content-block arrays before `session/prompt`, so maintainers need a current live ACP trace before changing code.

## What This Changes

The issue asks OpenClaw to normalize text-only Claude ACP prompts to content-block arrays and expose underlying invalid-params details instead of only `ACP_TURN_FAILED`.

## Best Possible Solution

Capture the actual outbound `session/prompt` payload from current main or a rebuilt release image, then add regression coverage at the acpx runtime boundary and patch OpenClaw/acpx only if a string still reaches the adapter.

## Reproduction Assessment

No. The reporter's direct wrapper string probe reproduces the adapter validation error, but current main plus the pinned acpx runtime source shows OpenClaw's delegated path converts string prompts to content-block arrays before `connection.prompt`; I did not establish a live current-main failure.

## Solution Assessment

Unclear. A core-side normalization patch is not the best fix unless a current wire trace proves the string reaches the adapter; if it does, the narrowest maintainable fix likely belongs at the acpx prompt-construction/client boundary with regression coverage.
