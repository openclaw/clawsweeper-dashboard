## Summary

Keep open: the reporter’s follow-up trace supersedes the original prompt-shape theory and points to a real current-main ACP timeout-control replay failure. Current source still stores `runTimeoutSeconds` as a runtime timeout option, replays it as `session/set_config_option` key `timeout`, and the pinned Claude ACP adapter rejects unknown config options before the prompt runs.

## What This Changes

The issue asks OpenClaw to stop failing Claude ACP `sessions_spawn` runs when timeout fields are passed and to preserve the underlying ACP adapter error details.

## Best Possible Solution

Decouple OpenClaw’s turn timeout from backend config replay: keep using `runTimeoutSeconds` for Gateway/control-plane timeout enforcement, send ACP `timeout` config only when the adapter explicitly supports it, and propagate nested ACP error details when a control fails.

## Reproduction Assessment

Yes, by calling `sessions_spawn` with `runtime: "acp"`, `agentId: "claude"`, and `runTimeoutSeconds` or legacy `timeoutSeconds` against the pinned acpx 0.6.1 plus Claude ACP 0.31.4 path. I did not run the live harness here, but current source and the reporter’s wrapper trace line up on the same failing `session/set_config_option` request.

## Solution Assessment

Yes: fixing runtime-control replay is the narrow maintainable path, not wrapping the prompt string, because acpx already normalizes text prompts and the traced failure happens before `session/prompt`. The repair should also make the nested ACP `data.details` visible enough for future adapter incompatibilities.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76600. The current issue title’s prompt-shape theory was corrected by the reporter: acpx 0.6.1 normalizes text prompts, but `sessions_spawn({ runtime: "acp", agentId: "claude", runTimeoutSeconds/timeoutSeconds })` stores the timeout in ACP runtime options, `buildRuntimeConfigOptionPairs` replays it as `session/set_config_option` key `timeout`, and `@agentclientprotocol/claude-agent-acp@0.31.4` rejects that key with `Unknown config option: timeout` before the prompt runs. Make `runTimeoutSeconds` continue to enforce OpenClaw/Gateway/control-plane turn timeouts, but do not send ACP `timeout` config to adapters unless the runtime has explicitly advertised/supports that config key. Preserve adapters that do support timeout if there is a reliable advertised-key path; do not add a new user config knob and do not change dependency pins. Improve the error path so nested ACP details from failed session controls are surfaced instead of only `Internal error` where feasible. Add focused regression coverage for Claude/non-Codex ACP timeout replay and keep the existing Codex timeout suppression behavior. Add a user-facing `CHANGELOG.md` fix entry.
