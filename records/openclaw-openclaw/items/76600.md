## Summary

Keep open: the reporter’s correction supersedes the original raw-string prompt theory, but current main still has an actionable ACP timeout replay bug on the documented Claude ACP spawn path.

## What This Changes

The issue asks OpenClaw to make `sessions_spawn(runtime="acp", agentId="claude")` work when a run timeout is supplied and to expose the underlying ACP adapter error.

## Best Possible Solution

Keep the local run-timeout watchdog semantics, stop replaying unsupported timeout config to Claude ACP unless the adapter explicitly advertises it, and preserve the nested ACP error details in the reported failure.

## Reproduction Assessment

Yes, source-reproducible with high confidence: supply `runTimeoutSeconds` to a Claude ACP `sessions_spawn` run on current pins and the code maps it to `session/set_config_option` key `timeout`, which the published Claude adapter rejects. I did not execute a live run, but the reporter’s JSON-RPC trace matches the current source path.

## Solution Assessment

Yes, the corrected fix direction is the narrowest maintainable path: avoid treating OpenClaw’s local timeout as a generic adapter config unless support is advertised. Changing prompt string conversion would patch the wrong layer because `acpx@0.6.1` already wraps string prompts before `session/prompt`.

## Repair Work Prompt

Fix this issue by preventing `sessions_spawn(runtime="acp", agentId="claude")` with `runTimeoutSeconds`/`timeoutSeconds` from replaying OpenClaw’s local timeout as unsupported ACP `session/set_config_option` key `timeout` unless the backend/adapter explicitly advertises that config key. Preserve `timeoutSeconds` as the OpenClaw turn watchdog, add regression coverage for Claude/non-Codex ACP timeout handling, and improve the surfaced error so nested ACP details such as `Unknown config option: timeout` are not collapsed to a generic internal error. Do not change prompt content-block normalization unless a failing test proves that separate bug still exists. Add a `CHANGELOG.md` fix entry because this is user-facing ACP runtime behavior.
