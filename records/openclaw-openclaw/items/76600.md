## Summary

Keep open: the reporter’s correction is credible and current `main` still has a source path that replays `runTimeoutSeconds` as ACP `timeout` config for Claude, while the pinned Claude adapter rejects unknown config options.

## What This Changes

This issue asks OpenClaw to stop failing Claude ACP `sessions_spawn` runs when `runTimeoutSeconds`/`timeoutSeconds` is supplied and to expose the nested ACP adapter error more clearly.

## Best Possible Solution

Keep the issue open and patch ACP runtime timeout handling so OpenClaw uses `runTimeoutSeconds` as its local turn watchdog without replaying unsupported `timeout` config to Claude, while preserving explicit backend config controls and surfacing ACP `data.details` in failures.

## Reproduction Assessment

Yes, with high confidence from source and the reporter's live trace: `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run", runTimeoutSeconds: 45 })` flows into `session/set_config_option timeout`, and the pinned Claude adapter rejects unknown config ids. I did not rerun the live path in this read-only checkout.

## Solution Assessment

Yes, the corrected direction is the narrow maintainable fix: stop treating OpenClaw's run timeout as a generic adapter config option unless the backend actually supports it, and improve error wrapping to include nested ACP details. The original string-prompt normalization theory is not the best fix for this issue.

Fix issue https://github.com/openclaw/openclaw/issues/76600. Current `sessions_spawn({ runtime: "acp", agentId: "claude", runTimeoutSeconds: N })` persists `timeoutSeconds`, `buildRuntimeConfigOptionPairs` turns that into ACP config key `timeout`, and `acpx@0.6.1` forwards it to `claude-agent-acp@0.31.4`, which rejects unknown config options. Keep `runTimeoutSeconds` as the OpenClaw/Gateway turn watchdog and background-run timeout, but do not replay it as generic `session/set_config_option` for Claude/non-advertising backends.
