## Summary

Keep open: the corrected issue trace points to a real current-main ACP timeout-control bug, not the original raw-string prompt theory. Current code still maps `runTimeoutSeconds` into a `timeout` ACP config option and forwards it for non-Codex acpx agents such as Claude.

## What This Changes

Prevent Claude ACP `sessions_spawn` runs with a timeout from failing on unsupported `session/set_config_option` replay, and surface the nested ACP error details.

## Best Possible Solution

Keep the Gateway run timeout, but only send ACP `timeout` through `session/set_config_option` when the backend or agent is known to support it, and retain useful adapter error details when config replay fails.

## Reproduction Assessment

Yes, source-reproducible with high confidence: a Claude ACP spawn with `runTimeoutSeconds` stores `timeoutSeconds`, converts it to `timeout`, and current acpx code forwards it for non-Codex agents. The reporter's live trace confirms `claude-agent-acp@0.31.4` rejects that replay before the prompt turn.

## Solution Assessment

Yes for the corrected diagnosis: the narrow fix is in ACP runtime option application or acpx capability filtering, not raw prompt conversion. The repair should preserve run timeout enforcement outside the unsupported adapter config option.

## Repair Work Prompt

Fix openclaw/openclaw#76600. The corrected reproduction shows `sessions_spawn({ runtime: "acp", agentId: "claude", runTimeoutSeconds/timeoutSeconds: ... })` fails before `session/prompt` because OpenClaw/acpx replays `session/set_config_option` with key/configId `timeout`, and `claude-agent-acp@0.31.4` rejects it as `Unknown config option: timeout`. Do not implement raw prompt wrapping for this issue.
