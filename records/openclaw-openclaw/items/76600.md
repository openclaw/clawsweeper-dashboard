## Summary

Keep open: the issue discussion corrected the root cause to ACP timeout config replay, and current main still maps `runTimeoutSeconds` to a `timeout` runtime config option that the acpx wrapper forwards for Claude/non-Codex ACP agents.

## What This Changes

The issue asks OpenClaw to make Claude ACP `sessions_spawn` with `runTimeoutSeconds` avoid unsupported `timeout` config replay and surface nested ACP adapter errors.

## Best Possible Solution

Keep the Gateway-side turn timeout, but stop replaying `timeout` to Claude ACP unless the adapter advertises support, and preserve the nested adapter error details when runtime config replay fails.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction. Current main maps `runTimeoutSeconds` to `timeoutSeconds`, replays it as ACP config key `timeout`, and forwards that key for Claude/non-Codex ACP agents; the reporter's wrapper trace shows Claude ACP rejects that exact request.

## Solution Assessment

Yes for the corrected direction, not for the original raw-string prompt suspicion. The narrow maintainable fix is at the runtime-control/acpx boundary: keep OpenClaw's outer timeout while only sending backend config keys that the adapter supports or that the owner wrapper knows are safe.
