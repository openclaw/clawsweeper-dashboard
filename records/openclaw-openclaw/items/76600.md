## Summary

Keep open: the reporter's corrected trace and current source point to a real ACP timeout config replay bug, not the original raw-string prompt suspicion. Current main still maps `runTimeoutSeconds` to an ACP `timeout` config option that Claude ACP 0.31.4 does not advertise or accept.

## What This Changes

Request that ACP `sessions_spawn` Claude runs with a timeout succeed by avoiding unsupported `timeout` config replay and surfacing nested ACP adapter errors.

## Best Possible Solution

Keep `runTimeoutSeconds` as an OpenClaw-enforced turn timeout unless the active ACP adapter explicitly supports a timeout config key, with regression coverage for Claude/acpx timeout spawns and clearer nested ACP error details.

## Reproduction Assessment

Yes, with high confidence from source plus the reporter's live trace, though not from an in-checkout live run. `sessions_spawn` stores the timeout, OpenClaw converts it to ACP config key `timeout`, acpx forwards it, and Claude ACP 0.31.4 rejects that key before `session/prompt`.

## Solution Assessment

Yes: the narrow fix is timeout-control replay, not text prompt wrapping. acpx already normalizes string prompts before `session/prompt`, so OpenClaw should avoid sending unsupported `timeout` config options while still enforcing the local turn timeout.

## Repair Work Prompt

Fix the ACP timeout replay bug tracked by https://github.com/openclaw/openclaw/issues/76600. Current root cause: `sessions_spawn(runtime="acp", agentId="claude", runTimeoutSeconds/timeoutSeconds=...)` stores `runtimeOptions.timeoutSeconds`; `buildRuntimeConfigOptionPairs` maps that to ACP config key `timeout`; `applyManagerRuntimeControls` sends `session/set_config_option`; `acpx@0.6.1` forwards it; `claude-agent-acp@0.31.4` rejects `timeout` before `session/prompt`. Do not implement the original prompt-array wrapping as the primary fix, because acpx normalizes string prompts before `session/prompt`. Keep `runTimeoutSeconds` as the local OpenClaw/Gateway turn timeout, and avoid sending `timeout` as an ACP runtime config option unless the active adapter explicitly supports it; preserve explicit unsupported-control behavior for truly unsupported user-set config keys where appropriate. Add regression coverage showing a Claude/acpx-style backend with no `timeout` config support still reaches the prompt/turn path when `runTimeoutSeconds` is set, and that nested ACP adapter details are surfaced when control replay does fail. Update `docs/tools/acp-agents.md` and `CHANGELOG.md` if the user-facing timeout semantics or diagnostics text changes. Do not change dependency pins or vendor package code unless a maintainer has provided an acpx release or explicit direction.
