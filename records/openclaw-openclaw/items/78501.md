## Summary

Keep this PR open: current main still has the reported heavy-check lock gaps, but the proposed branch is not mergeable as-is because it lacks real behavior proof and adds a packaged preinstall failure mode that can break consumer installs.

## What This Changes

The PR expands local heavy-check locking for type-aware oxlint, sharded oxlint, and scoped tooling tests, adds an install-pressure preinstall guard, and adjusts related script, cron, and Codex tests.

## Best Possible Solution

Keep the narrow lock-gap fixes, remove or strictly source-checkout-gate the fatal preinstall guard, then rebase and merge only after redacted after-fix proof is supplied.

## Reproduction Assessment

Yes. Source inspection on current main shows the type-aware explicit oxlint, sharded oxlint, and scoped tooling-test paths can bypass the shared lock; I did not run the Raspberry Pi RSS scenario or execute the PR branch.

## Solution Assessment

No, not as currently proposed. The lock serialization changes are narrow, but the fatal preinstall pressure guard should be removed or gated on an actual OpenClaw source checkout before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid treating every Git worktree as a source checkout:** `scripts/preinstall-package-manager-warning.mjs:124`
  - body: This preinstall guard uses `git rev-parse --is-inside-work-tree` from the lifecycle cwd to decide whether the install is a source checkout. During a packaged install under `node_modules/openclaw` inside a consumer project that is itself a Git repo, that still returns true, so a low-memory or high-load host can get exit code 1 from OpenClaw's preinstall and fail the install. Gate this on an actual OpenClaw source checkout marker, or keep packaged preinstall warnings non-fatal.
  - confidence: 0.88
