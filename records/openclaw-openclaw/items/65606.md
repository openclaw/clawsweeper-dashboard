## Summary

Keep open. The QQBot 4009 bug is real on current main, but this PR is not mergeable as the fix because it patches the removed legacy gateway entry point while the active QQBot gateway now runs through the bridge and engine gateway files.

## What This Changes

The branch adds a QQBot close-code session-recovery helper and test, then rewires the legacy `extensions/qqbot/src/gateway.ts` close handler so 4009 preserves session state while 4006/4007 clear it.

## Best Possible Solution

Port the intended 4009 policy into the active engine gateway: `SESSION_TIMEOUT` should refresh as needed but preserve `sessionId` and `lastSeq`, while 4006/4007 keep clearing session state. Add focused regression coverage on `ReconnectState.handleClose`, update the stale `SESSION_TIMEOUT` comment, and add the required changelog entry for the user-facing fix.

Yes. A deterministic source-level reproduction exists on current main: `ReconnectState.handleClose(GatewayCloseCode.SESSION_TIMEOUT, false)` returns `clearSession: true`, and `GatewayConnection.handleClose` then deletes the state required for the next HELLO resume path.

## Solution Assessment

No. The intended behavior is the right direction, but this PR applies it to a removed legacy file; the narrow maintainable fix belongs in `extensions/qqbot/src/engine/gateway/reconnect.ts` with active-path tests.
