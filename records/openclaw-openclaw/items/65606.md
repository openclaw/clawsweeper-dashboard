## Summary

Keep open. The reported QQBot 4009 behavior is still present on current main and the latest release, but this PR changes the removed legacy gateway file instead of the active QQBot engine gateway path, so the intent is useful but the submitted patch is not merge-ready.

## What This Changes

The PR adds a QQBot close-code recovery helper and regression test, then wires the legacy `extensions/qqbot/src/gateway.ts` close handler so 4009 preserves session state while 4006/4007 clear it.

## Best Possible Solution

Port the intended policy into `ReconnectState.handleClose` in `extensions/qqbot/src/engine/gateway/reconnect.ts`: 4006/4007 should continue clearing session state and re-identifying, while 4009 should refresh as needed but preserve `sessionId` and `lastSeq` so the existing HELLO resume path runs. Add focused active-path regression coverage and update the `SESSION_TIMEOUT` comment instead of reviving the removed legacy gateway file.

Yes. A deterministic source-level reproduction exists on current main: `ReconnectState.handleClose(GatewayCloseCode.SESSION_TIMEOUT, false)` returns `clearSession: true`, and `GatewayConnection.handleClose` then deletes the state required for HELLO resume.

## Solution Assessment

No. The intended behavior is correct, but this PR is not the best implementation because it targets `extensions/qqbot/src/gateway.ts`, which current main no longer uses; the narrow maintainable fix belongs in the active engine gateway files.
