## Summary

Keep open. The underlying QQBot 4009 bug is still present on current main, and the PR is useful source work but not mergeable as-is because it patches a legacy gateway file that current main no longer contains or starts.

## What This Changes

The PR adds a QQBot close-code recovery helper/test and rewires the legacy `extensions/qqbot/src/gateway.ts` close handler so 4009 preserves session state while 4006/4007 still clear it.

## Best Possible Solution

Port the intended 4009 policy into the active QQBot engine gateway: `SESSION_TIMEOUT` should keep `sessionId` and `lastSeq` so the next HELLO can RESUME, while 4006/4007 continue clearing session state, with active-path regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A deterministic source-level reproduction exists: `ReconnectState.handleClose(GatewayCloseCode.SESSION_TIMEOUT, false)` returns `clearSession: true` on current main, and `GatewayConnection.handleClose` then deletes the state required for the next HELLO resume path.

## Solution Assessment

No. The intended behavior is the right fix direction, but this PR applies it to a stale legacy file; the maintainable solution is a port to `extensions/qqbot/src/engine/gateway/reconnect.ts` with tests on the active path.
