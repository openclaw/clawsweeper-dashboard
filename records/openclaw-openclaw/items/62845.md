## Summary

Keep open: the PR still has merge blockers in the proposed patch, and the external contributor proof gate is missing. The shipped gateway path already covers the original WebChat leak, but this branch also changes client-side behavior and `/activate` parsing without updating the matching reload and command-registry paths.

## What This Changes

The PR changes WebChat silent-reply stream filtering, adds controller tests, adds a `/activate` activation-command alias, and adds changelog entries.

## Best Possible Solution

Either narrow this PR to a complete client-side WebChat defense that shares the reload/helper contract, or close it in favor of the shipped gateway suppression and finish `/activate` through the dedicated activation PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for both blockers: lowercase silent finals diverge between the PR controller and the reload helper, and `/activate` remains absent from registry-backed command detection used by WhatsApp group gating.

## Solution Assessment

No. The browser-side suppression direction is reasonable, but the current patch is not the narrow complete fix because it leaves related decision points out of sync and mixes in an incomplete activation alias change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register `/activate` in command detection:** `src/auto-reply/group-activation.ts:31`
  - body: Changing `parseActivationCommand()` alone leaves `/activate mention` outside the registry-backed control-command detector. WhatsApp group gating uses `hasControlCommand(commandBody, params.cfg)` for the owner no-mention bypass, so this alias can still be skipped before the activation handler runs unless the registry and coverage are updated too.
  - confidence: 0.91
- **[P2] Align lowercase silent finals with history reload:** `ui/src/ui/controllers/chat.ts:21`
  - body: The controller now drops lowercase `no_reply`, but `ui/src/ui/chat-event-reload.ts` still treats that same final payload as renderable. When a deferred `session.message` reload is waiting, `app-gateway` can clear it without replaying history, so update the reload helper and tests to use the same case-insensitive exact-token contract.
  - confidence: 0.86
- **[P2] Avoid dropping legitimate single-letter fallback replies:** `ui/src/ui/controllers/chat.ts:78`
  - body: `isSilentReplyTerminalStream()` treats any terminal fallback text equal to `N` as silent, even when the final/abort event has no silent `NO_REPLY` payload proving it was a sentinel fragment. That can make a real one-character assistant reply disappear after streaming, so scope the `N` suppression to confirmed silent finals or remove it.
  - confidence: 0.78
