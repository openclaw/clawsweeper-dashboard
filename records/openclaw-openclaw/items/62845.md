## Summary

Keep open. Current main still lacks the client-side lowercase and stream-prefix suppression this PR targets, but the latest PR head still leaves a single-letter `N` stream snapshot reachable through terminal fallback and is missing the required user-facing changelog entry.

## What This Changes

The PR updates Control UI chat silent-token filtering for case-insensitive `NO_REPLY` and uppercase prefix fragments, adds regression tests, and adds `/activate` as an alias for `/activation` in auto-reply command parsing.

## Best Possible Solution

Land a focused browser-side silent-token fix that aligns exact-token matching with the shared server contract, suppresses terminal prefix leftovers including the first `N` snapshot only in the safe terminal fallback context, keeps legitimate natural-language `No...` messages visible, and records the user-facing behavior in the changelog. The `/activate` alias can land with the same PR only if maintainers accept the extra scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the current code and PR diff: a stream sequence `N` then `NO`/`NO_` followed by a silent final leaves `state.chatStream` as `N`, and the PR's terminal fallback still treats that as renderable.

## Solution Assessment

No. The PR is close, but the current helper/fallback combination does not fully solve the stream-fragment case described in the PR body; the safer solution is to handle the single-letter terminal leftover without broadening live delta suppression of legitimate one-letter replies.
