## Summary

Keep open. Current main still has exact, case-sensitive client-side NO_REPLY filtering in the Control UI chat stream and tests that intentionally keep lowercase `no_reply`, so this PR is not obsolete. The proposed branch is directionally useful, but its prefix helper still misses the single-character `N` stream snapshot called out in the PR body, which can still be persisted through the terminal fallback path.

## What This Changes

The PR adds client-side case-insensitive NO_REPLY filtering and uppercase prefix-fragment suppression in the Control UI chat stream, adds regression tests, and also adds `/activate` as an alias for `/activation`.

## Best Possible Solution

Land a focused client-side silent-token fix that keeps the browser stream/fallback behavior aligned with the server contract, including regression coverage for lowercase exact finals and terminal prefix-fragment leftovers. The `/activate` alias can land separately or stay only if maintainers accept the extra scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: on current main, a final assistant payload containing `no_reply` is treated as visible by the existing test, and terminal fallback only rejects exact `NO_REPLY`, so uppercase prefix fragments can become assistant messages.

## Solution Assessment

No. The PR is close, but it does not fully solve the stream-fragment case it describes because a first `N` delta remains outside the prefix helper and can still be persisted when the final payload is silent or missing; dropping that terminal leftover would make the fix complete.
