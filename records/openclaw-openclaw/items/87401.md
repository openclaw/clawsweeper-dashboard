## Label Justifications

- `P2`: A normal Telegram photo workflow can deliver media that the agent cannot inspect, but the blast radius is limited to Telegram media context.
- `impact:message-loss`: A delivered Telegram photo can become unavailable to the agent's image-inspection path even though the media was downloaded.

## Summary

Keep open: current main and the latest release still expose Telegram photo media as `telegram:file/...` in model-facing context, which the image tool rejects, and the prior attempted fix was closed unmerged after a visibility-boundary finding.

## What This Changes

The issue asks Telegram media context to resolve `telegram:file/<file_id>` photo references to an image-tool-readable local or `media://inbound/*` reference before the agent sees them.

## Best Possible Solution

Fix this inside the Telegram plugin by projecting only visibility-allowed, hydrated local paths or canonical `media://inbound/*` refs into model/tool-facing context while keeping Telegram file IDs as private metadata.

## Reproduction Assessment

Yes at source level: current main writes `telegram:file/...` into Telegram cached media, serializes it into prompt context, and the image tool rejects unsupported schemes. I did not run a live Telegram photo in this read-only review.

## Solution Assessment

Yes, the best fix belongs at the Telegram media/context handoff, not in the core image tool. The repair should reuse the existing inbound media contract and apply the same visibility policy used for reply context.
