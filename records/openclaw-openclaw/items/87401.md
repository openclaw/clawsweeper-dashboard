## Summary

Keep open: current main and v2026.6.6 still expose Telegram cached photo media as `telegram:file/...` in model-facing prompt context, while the image tool rejects that scheme. The attempted linked fix is closed unmerged, and the related sticker issue is adjacent rather than a complete replacement for this photo report.

## What This Changes

The issue reports that Telegram photo references reach the image tool as unsupported `telegram:file/<file_id>` values instead of an image-tool-readable local or media reference.

## Best Possible Solution

Fix the Telegram plugin media-context handoff so hydrated photo media reaches model-facing context as a local path or `media://inbound/*` reference, while keeping Telegram file IDs as plugin-private metadata.

## Reproduction Assessment

Yes at source level: current main persists `telegram:file/...`, exposes it as prompt `media_ref`, and rejects that scheme in the image tool. I did not run a live Telegram photo flow in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is at the Telegram media/context handoff, reusing existing local or `media://inbound/*` support. Teaching the core image tool to resolve Telegram file IDs would cross the plugin boundary and duplicate channel-specific ownership.
