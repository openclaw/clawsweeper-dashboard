## Label Justifications

- `impact:message-loss`: A delivered Telegram photo can become unavailable to the agent's image-inspection path even though the media was downloaded.

## Summary

Keep open: current main and v2026.6.6 still expose Telegram photo reply context as `telegram:file/...`, which the image tool rejects, and the related fix PR was closed unmerged after a security-boundary review finding.

## What This Changes

The issue reports that Telegram photos can reach the image tool as unresolved `telegram:file/<file_id>` references instead of an image-tool-readable local or `media://inbound/*` path.

## Best Possible Solution

Keep the fix in the Telegram plugin: project only visible, hydrated media paths or `media://inbound/*` refs into model/tool-facing context while retaining Telegram file IDs as private metadata.

## Reproduction Assessment

Yes at source level: current main still serializes cached Telegram media as `telegram:file/...`, and the image tool rejects that unsupported scheme. I did not run a live Telegram photo through the transport in this read-only review.

## Solution Assessment

Yes, the right direction is to fix the Telegram media/context handoff rather than adding Telegram-specific URI support to the core image tool. The repair should also incorporate the visibility-gating lesson from the closed attempted PR.
