## Label Justifications

- `P2`: A Telegram image-reference bug breaks photo inspection in a normal channel workflow with limited blast radius.
- `impact:message-loss`: A delivered Telegram photo attachment becomes unusable by the agent's image-inspection path even though the media was downloaded.

## Summary

Keep open: current main and the latest release still expose Telegram photo refs as `telegram:file/<file_id>` in conversation context, while the image tool only accepts file paths, file/data/http URLs, or canonical `media://inbound/*` refs.

## What This Changes

The issue reports that Telegram photo references reach the image tool as unresolved `telegram:file/<file_id>` URIs instead of usable local or `media://` paths.

## Best Possible Solution

Normalize Telegram media context to an image-tool-readable local or `media://inbound/*` reference before it reaches the agent, while keeping Telegram file IDs as plugin-internal metadata if still needed.

## Reproduction Assessment

Yes from source inspection, but not from a live Telegram run: current main still emits `telegram:file/<file_id>` into Telegram conversation context and the image tool rejects unsupported schemes. A focused regression can simulate a Telegram photo/reply chain and assert the model-facing context contains `mediaPath` or `media://inbound/*` instead.

## Solution Assessment

Yes: the narrow maintainable fix is at the Telegram context/media handoff, reusing the existing canonical `media://inbound/*` and local media path support. Teaching the core image tool to resolve Telegram file IDs would cross the plugin boundary and duplicate channel-specific ownership.
