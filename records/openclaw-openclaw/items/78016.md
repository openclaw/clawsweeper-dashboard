## Summary

Keep open: current Matrix inbound media handling saves audio attachments but lacks the channel-owned voice/audio preflight path used by other chat plugins, so the reported parity gap is plausible and narrow enough to fix in the Matrix plugin.

## What This Changes

Request Matrix inbound voice messages to be transcribed and understood like voice inputs on Discord, Telegram, WhatsApp, and Feishu.

## Best Possible Solution

Add Matrix-owned inbound audio/voice preflight using the existing `tools.media.audio` helper, feed the transcript into mention detection and final inbound context, and document the Matrix behavior without moving owner-specific policy into core.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection gives a clear current-main path where Matrix downloads media only after mention gating and never calls the shared audio preflight helper; a live Element voice-message run was not performed.

## Solution Assessment

Is this the best way to solve the issue? Yes: a Matrix plugin-local preflight wired to the existing media-runtime STT contract is narrower and more maintainable than changing core media understanding or adding a new config option.
