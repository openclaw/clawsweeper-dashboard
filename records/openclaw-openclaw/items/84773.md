## Summary

Keep open: the compact result idea is reasonable, but this implementation drops the internal `sourceReply` contract that current Codex/WebChat delivery depends on, so it would regress visible current-run replies. It also only provides mock/unit proof for a UI-visible WebChat behavior.

## What This Changes

The PR changes internal WebChat `message_tool_only` sends to return a compact message tool result while adding a regression assertion for the internal-source path.

## Best Possible Solution

Keep the compact user-visible tool content, but preserve a runtime-only source-reply handoff for Codex/WebChat projection and prove the full WebChat/TUI current-run path after the patch.

## Reproduction Assessment

Yes at source level: current `main` extracts `details.sourceReply` from the message tool result, while this PR’s compact details omit that field. I did not run a live WebChat/TUI session in this read-only review.

## Solution Assessment

No: compacting the visible tool result is a good direction, but this patch is not the best implementation because it removes the runtime payload contract. The safer fix is to keep display content compact while preserving source-reply telemetry for the WebChat/TUI projection path.
