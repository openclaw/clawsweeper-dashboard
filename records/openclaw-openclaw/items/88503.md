## Summary

Keep this PR open for normal review: the patch is focused and I did not find a line-level correctness bug, but the supplied live proof only shows a normal WebChat send and still does not exercise the same-Gateway reconnect replay race that this change is meant to protect.

## What This Changes

The PR changes Control UI Webchat queue sends so same-scope reconnects can retry in-flight sends, stale send completions do not remove replayed queue items, gateway-scope changes do not replay old sends, and in-flight queue entries cannot be removed.

## Best Possible Solution

Land after redacted real Gateway plus browser proof demonstrates a same-scope reconnect during an in-flight Webchat send replays once without duplicate output or a stuck queued message.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main stops the old Gateway client with a non-retryable 'gateway client stopped' Error while Webchat only requeues retryable or gateway-closed-style send failures. I did not run a live browser race because this review is read-only.

## Solution Assessment

Mostly yes: the PR targets the right UI/Gateway boundary and adds focused regression coverage for stale completions and same-scope retryability. The remaining gap is proof, not an identified code defect: the real run should exercise the reconnect/replay behavior instead of only a normal send.
