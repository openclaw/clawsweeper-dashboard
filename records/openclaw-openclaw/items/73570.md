## Summary

Keep this PR open, but do not merge this revision. Current main still lacks the requested internal pre-send hook bridge, while the PR patch introduces blocking behavior bugs around cancelled sends, rewritten content, and the unresolved `message_sending` versus `message:sending` event-key contract.

## What This Changes

The PR adds a fire-and-forget internal `message:sending` hook emission in `src/infra/outbound/deliver.ts` after the plugin `message_sending` hook returns.

## Best Possible Solution

Amend the PR into a documented, tested internal pre-send hook surface: emit only for non-cancelled sends, pass the effective post-rewrite content, settle the event-key compatibility contract, and add focused regression coverage plus a changelog entry before landing the linked fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main runs the plugin `message_sending` hook but does not trigger an internal pre-send event, and directory-hook metadata is registered by literal event key.

## Solution Assessment

No. Bridging the pre-send path is the right direction, but this patch is not the best solution yet because it notifies before cancellation, sends stale pre-rewrite content, and does not settle whether existing `message_sending` hook registrations should be supported or migrated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Do not notify internal hooks for cancelled sends:** `src/infra/outbound/deliver.ts:1032`
  - body: The new `fireAndForgetHook` block runs before the `hookResult.cancelled` guard. If a `message_sending` plugin returns `{ cancel: true }`, this emits an internal pre-send notification for a message that will not be delivered, so the guard needs to run before the internal hook or the condition needs to include `!hookResult.cancelled`.
  - confidence: 0.94
- **[P2] Pass the rewritten hook content to internal subscribers:** `src/infra/outbound/deliver.ts:1041`
  - body: This builds the internal event from the original `payloadSummary`, but `applyMessageSendingHook` returns rewritten text in `hookResult.payloadSummary` when a plugin modifies outbound content. Internal subscribers would see stale pre-rewrite content; use `hookResult.payloadSummary.hookContent ?? hookResult.payloadSummary.text`.
  - confidence: 0.95
- **[P2] Make the bridged event match user hook registrations:** `src/infra/outbound/deliver.ts:1035-1036`
  - body: The linked report describes standalone hooks registered for `message_sending`, but `createInternalHookEvent("message", "sending", ...)` dispatches only `message` and `message:sending` because internal hook metadata is registered verbatim. Either add an explicit compatibility path for `message_sending` or update the contract, docs, and tests so users register the supported key.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `src/infra/outbound/deliver.ts:1032`
  - body: This is a user-facing hook fix, and the branch only changes delivery code. Repo policy requires user-facing fixes to add a single-line `CHANGELOG.md` entry in the active Unreleased Fixes section with contributor credit.
  - confidence: 0.9
