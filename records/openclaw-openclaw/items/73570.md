## Summary

Keep open. The PR targets a plausible outbound hook gap, but the latest diff still emits the new pre-send internal hook for messages that will not be delivered, sends stale pre-rewrite content, and does not satisfy the linked `message_sending` directory-hook registration key.

## What This Changes

The PR adds a fire-and-forget internal `message`/`sending` hook dispatch during outbound delivery and updates outbound delivery tests to expect the extra pre-send notification.

## Best Possible Solution

Define and document the internal pre-send hook contract, then emit it only for payloads that will actually be delivered, using effective post-plugin content and compatibility for the accepted registration key.

## Reproduction Assessment

Yes, at source level. Current main has no internal pre-send bridge, and the latest PR diff still places the new dispatch before cancellation/deliverability checks while using the original payload summary and a key that does not match the linked `message_sending` registration.

## Solution Assessment

No. The branch is in the right delivery area, but it is not the best mergeable fix until the event key and observe-only semantics are decided and the implementation uses the effective deliverable payload.
