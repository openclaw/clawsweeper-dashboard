## Summary

Keep open. Current main still lacks the requested pre-send internal hook bridge, but this PR is not mergeable yet because it would miss the reported standalone hook key and notify internal hooks with stale or canceled-send data.

## What This Changes

The PR adds a fire-and-forget internal `message:sending` trigger in outbound delivery after plugin `message_sending` processing.

## Best Possible Solution

Define a documented and tested internal pre-send hook surface: emit only for non-canceled sends, pass the effective post-rewrite content, decide whether `message_sending` is a compatibility alias or unsupported, and add focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. The linked issue gives concrete standalone-hook steps, and source inspection confirms current main only invokes the plugin `message_sending` path while internal dispatch reaches `message` and `message:<action>` registrations.

## Solution Assessment

No. The direction is plausible, but this patch is not the best solution yet because it preserves the event-key mismatch and introduces canceled-send and stale-content regressions.

Full review comments:

- **[P2] Do not notify internal hooks for canceled sends:** `src/infra/outbound/deliver.ts:1032`
  - body: The new `fireAndForgetHook` block runs before the `hookResult.cancelled` guard. If a `message_sending` plugin cancels delivery, this still emits a pre-send internal event for a message that will not be sent, so the guard needs to run before the internal hook or be included in the condition.
  - confidence: 0.94
- **[P2] Pass rewritten hook content to internal subscribers:** `src/infra/outbound/deliver.ts:1041`
  - body: This builds the internal event from the original `payloadSummary`, but `applyMessageSendingHook` returns rewritten text in `hookResult.payloadSummary` when a plugin modifies outbound content. Internal subscribers would see stale pre-rewrite content; use the payload summary from `hookResult`.
  - confidence: 0.95

  - body: The linked report registers standalone hooks for `message_sending`, but `createInternalHookEvent("message", "sending", ...)` dispatches only `message` and `message:sending` because internal hook metadata is registered verbatim. Either add a compatibility path for `message_sending` or update the contract, docs, and tests so users register the supported key.
  - confidence: 0.88
