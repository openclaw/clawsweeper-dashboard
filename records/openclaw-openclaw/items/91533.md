## Summary

Keep open: the patch is the right general direction, but it is not merge-ready because a legacy global avatar can still reappear after a scoped clear, and the PR lacks real two-agent after-fix proof.

## What This Changes

The PR changes Control UI assistant avatar persistence from one browser-local key to per-agent localStorage keys with a legacy global-key fallback.

## Best Possible Solution

Land a narrow Control UI storage fix that scopes assistant avatar overrides per agent while treating the legacy global key as a one-time migration or suppressible fallback, with focused regression tests and real two-agent proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main reads and writes all assistant avatar overrides through the single `openclaw.control.assistant.v1` key, so changing one agent's override affects the next agent that loads the same key. I did not run a live UI repro because this review is read-only.

## Solution Assessment

No, not yet. Per-agent keys are the right layer, but this implementation must fix legacy fallback/clear semantics and show real two-agent behavior before it is the best fix.
