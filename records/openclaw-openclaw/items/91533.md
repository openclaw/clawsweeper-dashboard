## Summary

Keep this PR open, but it is not merge-ready: the patch moves the main avatar read/write path toward per-agent storage, yet legacy fallback clearing and bootstrap/reload behavior can still reapply the wrong avatar, and the supplied proof is only a code snippet rather than a real two-agent UI check.

## What This Changes

The PR scopes Control UI assistant-avatar localStorage reads and writes with `openclaw.control.assistant.v1:<agentId>` keys and adds a legacy global-key fallback.

## Best Possible Solution

Use one canonical per-agent assistant-avatar storage path across render, identity refresh, clear, and bootstrap, with an explicit legacy migration or tombstone strategy plus focused tests and real UI proof.

## Reproduction Assessment

Yes at source level: current main and v2026.6.1 use one global assistant-avatar localStorage key, and the linked issue gives direct switch-agent steps. I did not run a live browser reproduction because this was a read-only review.

## Solution Assessment

No. Scoping the key is the right direction, but the current patch is incomplete because legacy fallback clearing and the bootstrap override caller can still leak the wrong avatar across agents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Make scoped clears suppress the legacy fallback:** `ui/src/ui/storage.ts:386`
  - body: When a legacy global avatar exists, `loadLocalAssistantIdentity(agentId)` falls back to that value whenever the scoped key is absent. But clearing an agent now removes only the scoped key, so the next identity reload falls back to the old global avatar and the user cannot actually clear the displayed override for that agent. Preserve a per-agent empty marker, migrate/drop the legacy value intentionally, or otherwise make clear semantics match the fallback behavior.
  - confidence: 0.9
- **[P2] Scope the bootstrap override caller too:** `ui/src/ui/storage.ts:358`
  - body: The PR scopes render and `agent.identity.get`, but bootstrap still calls `loadLocalAssistantIdentity()` with no agent id through `applyLocalAssistantAvatarOverride`. On reload for a non-default active session, that sibling path can still apply the global browser-local avatar before the gateway identity refresh. Thread the active agent id into bootstrap and cover reload behavior.
  - confidence: 0.87
