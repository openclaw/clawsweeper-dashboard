## Summary

Keep open: the scoped-avatar direction matches the linked bug, but this patch currently has a type error and leaves the bootstrap local-avatar path unscoped, so it is not merge-ready.

## What This Changes

The PR scopes Control UI assistant-avatar localStorage reads and writes to `openclaw.control.assistant.v1:<agentId>` with a legacy global-key fallback.

## Best Possible Solution

Finish one canonical scoped assistant-avatar storage path across render, identity loading, saving, and bootstrap, with tests for per-agent values plus legacy global fallback.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.1` use one global assistant-avatar localStorage key, and the linked issue gives direct switch-agent steps. I did not run a live browser reproduction because this review was read-only.

## Solution Assessment

No: scoping the storage key is the right direction, but this PR is incomplete because it breaks the override state type and misses the bootstrap override caller. The maintainable fix is to thread the active normalized agent id through every local assistant-avatar read/write path and test the upgrade fallback.
