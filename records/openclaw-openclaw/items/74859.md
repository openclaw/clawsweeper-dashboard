## Summary

Keep open. Current main still has the pre-aborted signal race in the embedded runner, and open PR #74979 already uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that embedded runs avoid starting `activeSession.prompt()` when the run abort signal is already aborted, preventing a floating Agent loop after interrupt/timeout cleanup.

## Best Possible Solution

Review and land PR #74979, or an equivalent narrow fix that checks abort state before every embedded prompt start and adds an escaped-agent cleanup backstop; treat a configurable per-run LLM-call cap as separate hardening.

## Reproduction Assessment

Yes. The deterministic source path is clear on current main: a pre-aborted `params.abortSignal` aborts `runAbortController`, then the runner still reaches `abortable(activeSession.prompt(...))`, where the prompt promise is created before `abortable()` rejects.

## Solution Assessment

Yes. The narrow maintainable fix is a pre-prompt abort guard plus a cleanup backstop for escaped Agent state; the proposed max-call cap is useful defense-in-depth but not required to fix this bug.
