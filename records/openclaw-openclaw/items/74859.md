## Summary

Keep open. Current `main` still has the source-level pre-aborted prompt-start race, and the supplied GitHub context shows open PR #74979 is already paired with this issue via closing syntax, so the issue should stay open until that PR or an equivalent replacement merges.

## What This Changes

Ensure aborted embedded Pi runs, especially pre-aborted signals from interrupt mode, cannot start or leave an unowned Agent prompt loop that continues making LLM calls.

## Best Possible Solution

Review and land #74979, or an equivalent narrow replacement, with a guard before every embedded Pi prompt-start boundary, cleanup for escaped active runs and queued messages, and focused regression coverage for pre-aborted and normal prompt paths. Treat the configurable max-LLM-call cap as separate hardening unless maintainers want it in the same change.

## Reproduction Assessment

Yes. The issue includes a deterministic pre-aborted `AbortSignal` test shape, and current `main` still shows `activeSession.prompt(...)` starting before `abortableWithSignal(...)` can reject a pre-aborted signal.

## Solution Assessment

Yes for the narrow repair direction. A pre-prompt abort guard plus escaped-agent cleanup backstop is the smallest maintainable fix; the proposed per-run LLM cap is useful defense-in-depth but needs separate product/config review.
