## Summary

Keep open. Current main still creates `activeSession.prompt(...)` before `abortable()` can observe a pre-aborted run signal, and open PR #74979 is explicitly paired with this bug using closing syntax, so the issue should remain open until that PR is revised, merged, or replaced.

## What This Changes

The issue reports that a pre-aborted run signal can be wrapped around an already-started Pi `activeSession.prompt()` promise, leaving an unowned Agent loop that continues making LLM calls after attempt cleanup.

## Best Possible Solution

Revise and land #74979, or an equivalent narrow replacement, so the embedded runner checks the run abort state before every Pi prompt-start boundary, cancels any escaped active run and clears queues during abort cleanup, and adds regression coverage for pre-aborted, mid-abort, and normal prompt paths. Keep any configurable per-run LLM hard cap as separate defense-in-depth unless maintainers intentionally broaden this fix.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current-main source plus the pinned Pi 0.70.6 contract still show the same call-order hazard; this read-only review did not execute the test.

## Solution Assessment

Yes, the narrow fix direction is the best maintainable path: guard before starting `activeSession.prompt()` and add cleanup backstops for escaped Pi Agent work. The proposed max-LLM-call cap is useful hardening but should not be required to fix this specific prompt-start race.
