## Label Justifications

- `P1`: The issue describes an urgent broken agent workflow where interrupt mode can leave unbounded background LLM calls after the user-facing run exits.
- `impact:session-state`: The attempt lifecycle can finish while a separate Agent run remains active, creating run/session state drift.
- `impact:crash-loop`: The zombie Agent loop can keep runtime work alive indefinitely and degrade process availability until restart.

## Summary

Keep open: current main and latest release still evaluate `activeSession.prompt()` before `abortable()` can reject an already-aborted run signal, and the linked fix PR at https://github.com/openclaw/openclaw/pull/74979 is open but not merged.

## What This Changes

The issue requests fixing pre-aborted abort handling so `abortable(activeSession.prompt())` cannot start a detached Agent loop after an interrupted run exits.

## Best Possible Solution

Land the linked guard PR or an equivalent current-main patch that checks the run abort signal before `activeSession.prompt()` and proves the pre-aborted path with production-path regression coverage.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.6` still evaluate `activeSession.prompt()` before `abortable()` sees an already-aborted signal; I did not execute the supplied Vitest in this read-only review.

## Solution Assessment

Yes for the core fix direction: a pre-prompt abort guard at `promptActiveSession` is the narrowest maintainable fix for the evaluation-order bug. The proposed configurable LLM-call hard cap is a separate product/config decision, not required to fix the central bug.
