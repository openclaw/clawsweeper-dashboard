## Label Justifications

- `P1`: The issue describes an urgent broken agent workflow where interrupt mode can leave unbounded background LLM calls after the user-facing run exits.
- `impact:crash-loop`: The reported zombie Agent loop can keep runtime work alive indefinitely and degrade process availability until restart.
- `impact:session-state`: The attempt lifecycle can finish while a separate Agent run remains active, creating run/session state drift.

## Summary

Keep open: current main and the latest release still call `activeSession.prompt()` before the pre-aborted run signal is checked by `abortable()`, so the reported zombie-loop surface remains present. An open linked PR already targets this issue and should be reviewed or landed before the issue closes.

## What This Changes

Fix the embedded runner so a pre-aborted run signal cannot start a floating Agent prompt loop after the attempt exits.

## Best Possible Solution

Review and land the linked guard PR, or an equivalent current-main patch that checks the run abort signal before calling `activeSession.prompt()` and covers the pre-aborted path.

## Reproduction Assessment

Yes, source-level: current main evaluates `activeSession.prompt()` before `abortable()` can reject a pre-aborted signal, and the Agent loop/dependency path supports the reported floating-run behavior. I did not execute the supplied Vitest in this read-only review.

## Solution Assessment

Yes, the best immediate fix is a pre-prompt abort guard at `promptActiveSession` plus focused regression coverage. A configurable per-run LLM-call hard cap would be a separate product/config decision, not required for the central bug fix.
