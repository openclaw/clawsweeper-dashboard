## Label Justifications

- `P1`: The issue describes a broken interrupt-mode agent workflow that can continue making unbounded background LLM calls after the visible run exits.
- `impact:session-state`: The attempt lifecycle can finish while a separate Agent run remains active, creating run/session state drift.
- `impact:crash-loop`: The zombie Agent loop can keep runtime work alive indefinitely and degrade process availability until restart.

## Summary

Keep open: current main and the latest release still construct `activeSession.prompt(...)` before `abortable()` can reject an already-aborted run signal, and the linked open fix PR at https://github.com/openclaw/openclaw/pull/74979 is still the active implementation path.

## What This Changes

Prevent pre-aborted embedded agent runs from starting a floating `activeSession.prompt()` loop after interrupt or timeout aborts.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74979, or an equivalent current-main patch that checks the run abort signal before `activeSession.prompt()` and proves the production path with focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main calls `activeSession.prompt(...)` before `abortable()` can reject a pre-aborted signal, and the Agent runtime then owns a separate abort controller. I did not execute the supplied Vitest because this was a read-only cleanup review.

## Solution Assessment

Yes, the pre-prompt abort guard at `promptActiveSession` is the narrow owner-boundary fix for the central evaluation-order bug. The proposed configurable LLM-call cap would be a separate product safety net, not required to fix this issue.
