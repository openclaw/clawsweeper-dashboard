The PR exports `hasTelegramErrorCode`, uses it in Telegram sendChatAction 401 classification, and adds regression tests for structured 429/5xx messages containing `401` plus a true structured 401.

## Best Possible Solution

Land one focused handler fix that uses structured Telegram HTTP codes, short-circuits known transient network errors before any text fallback, and includes accepted real/runtime proof plus regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main evaluates broad `is401Error` text matching before the transient branch, so a transient error message containing `401` can still trigger the suspension path; I did not run tests because this review is read-only.

## Solution Assessment

No. Structured `error_code` handling is the right layer for Telegram HTTP errors, but the PR must also route known recoverable network errors before the bare text fallback or remove that numeric fallback.
