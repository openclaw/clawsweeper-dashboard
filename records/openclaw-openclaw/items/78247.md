## Summary

Keep open. This is a useful skill enhancement with sufficient terminal proof, but the PR head still has a JSON correctness bug for the current CodexBar total-only per-model token shape.

## What This Changes

The PR adds per-model token totals to `skills/model-usage` `--mode all` text/JSON output and updates the skill docs and tests.

## Best Possible Solution

Land the skill enhancement after tracking per-token-field availability and adding a `totalTokens`-only regression test, while leaving cost normalization to the canonical open issue.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a skill feature PR. The review finding is source-reproducible from the PR head and CodexBar contract: total-only `modelBreakdowns[]` rows produce `tokensAvailable: true` with split fields serialized as zero.

## Solution Assessment

No, not yet. Parsing token data at the `modelBreakdowns[]` boundary is the right direction, but JSON output must distinguish unavailable split fields from real zero-token splits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unknown per-model token splits:** `skills/model-usage/scripts/model_usage.py:320-323`
  - body: Current CodexBar per-model breakdowns can expose `totalTokens` without input/output/cache split fields. This code gates every split field only on `tokens_available`, so a total-only row serializes missing split fields as `0` rather than `null`; consumers cannot distinguish unknown splits from real zero usage. Track split-field availability separately and emit `null` when a split was not present.
