## Summary

Keep open. The PR is a useful skill enhancement that is not on current main, but it should fix one JSON-shape bug before merge: current CodexBar per-model breakdowns expose `totalTokens` without per-model input/output/cache splits, and the patch would serialize those unknown split fields as `0`. The PR body includes sufficient terminal proof for the changed script behavior, and the security pass found no concrete concern.

## What This Changes

The PR updates the `model-usage` skill script, docs, and tests to include per-model token totals in `--mode all` text and JSON output when `modelBreakdowns[]` exposes token fields.

## Best Possible Solution

Land a corrected skill enhancement that reports per-model total tokens when present, preserves unknown split fields as unknown in JSON, and leaves cost normalization to https://github.com/openclaw/openclaw/issues/37878.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a skill feature PR. Current main source confirms the missing token summaries, and the PR body provides a terminal smoke run for the new behavior.

## Solution Assessment

No, not yet. Parsing token data at the `modelBreakdowns[]` boundary is the right direction, but the JSON output should not turn unavailable per-model split fields into zeroes for current CodexBar total-only rows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Preserve unknown per-model token splits:** `skills/model-usage/scripts/model_usage.py:320-323`
  - body: Current CodexBar per-model breakdowns expose `totalTokens` but not per-model input/output/cache splits. With this patch, those missing fields stay at the dataclass default `0` and `build_json_all` serializes them whenever `tokensAvailable` is true, so real `--mode all --format json` output would claim zero input/output/cache tokens instead of unknown. Track per-field availability and emit `null` for split fields that were not present.
