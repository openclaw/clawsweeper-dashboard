## Summary

Keep open: this is a focused fix for the right MCP-native OAuth gateway path and the PR body includes real live-output proof, but the patch leaves pre-existing token stores without `expires_at` waiting 50 minutes before the first proactive refresh, so already-expired tokens can keep failing after upgrade.

## What This Changes

The PR adds a lifecycle-owned proactive OAuth refresh timer for MCP-native `auth: "oauth"` MCP session transports, persists `expires_at`, and carries the OAuth provider/server URL through transport resolution so sessions can stop the timer on dispose.

## Best Possible Solution

Refresh unknown-expiry MCP OAuth stores promptly, add a focused regression for a stored token with `refresh_token` and no `expires_at`, then land this proactive-refresh fix.

## Reproduction Assessment

Yes for the review finding. Current main stores MCP OAuth tokens without `expires_at`, the PR schedules missing-expiry tokens with a 50-minute fallback, and the linked report shows expired stored tokens are the failing gateway state; I did not run a live one-hour expiry test.

## Solution Assessment

No, not yet. Proactive refresh is a reasonable owner-boundary fix, but unknown-expiry upgraded stores need a prompt first refresh so the patch actually heals existing affected gateways.
