## Summary

This should stay open and needs one focused fix before merge: current main has the reported Telegram media-alias gap and the PR mostly uses the right media path with live proof, but the new multi-media action path would hand the last send result to delivery pinning even though OpenClaw's delivery contract pins the first delivered message.

## What This Changes

The PR normalizes Telegram `sendMessage` media aliases, top-level paths, `mediaUrls`, and structured attachments, then sends them sequentially through the existing Telegram media helper with regression tests.

## Best Possible Solution

Keep the alias and media fan-out direction, but track the first successful Telegram media send id for delivery pinning and add a multi-attachment `delivery.pin` regression before landing.

## Reproduction Assessment

Yes by source inspection: current main exposes `path`/`filePath` and structured attachment media aliases, while Telegram action runtime only forwards `mediaUrl`/`media`. The PR finding is also source-reproducible because the helper returns the last media result while the delivery contract says to pin the first delivered message.

## Solution Assessment

No as written. The media normalization and existing Telegram media helper are the right direction, but the PR should preserve the first successful send id for delivery pinning before merge.
