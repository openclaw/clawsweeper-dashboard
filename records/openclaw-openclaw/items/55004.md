## Summary

Keep open: current main still lacks the local dashboard reachability gate, but this PR head is stale/conflicting and would regress token redaction and TLS-aware dashboard links; it also still lacks after-fix real behavior proof.

## What This Changes

The PR adds a local dashboard `/healthz` preflight before clipboard/browser auto-open, daemon/config remediation output, a daemon status field, and dashboard command tests.

## Best Possible Solution

Rebase or replace the preflight on current main so local auto-open side effects are gated while preserving token redaction, SecretRef behavior, TLS-aware links, `--no-open`, custom binds, and daemon status diagnostics.

## Reproduction Assessment

Yes, by source inspection and the linked fresh-install logs: on current main, `openclaw dashboard` reaches clipboard/browser side effects without a local reachability gate. I did not run a live repro in this read-only review.

## Solution Assessment

No. The preflight direction is appropriate, but this branch is not the best implementation until it is rebased or reworked without regressing token redaction, TLS links, and current dashboard fallback contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep bearer tokens out of dashboard logs:** `src/commands/dashboard.ts:149`
  - body: This logs `dashboardUrl`, which includes `#token=...` when token auto-auth is enabled. Current main logs only the clean base URL and has regression coverage for that, so this would re-expose gateway bearer tokens in runtime logs.
  - confidence: 0.92
- **[P1] Stop passing tokens into SSH hints:** `src/commands/dashboard.ts:187-190`
  - body: When browser open fails, this passes the bearer token into `formatControlUiSshHint`, and the returned hint is logged. Current main deliberately calls that helper without a token, so this would leak the token on the SSH fallback path.
  - confidence: 0.9
- **[P2] Preserve TLS-aware dashboard links:** `src/commands/dashboard.ts:136-141`
  - body: This call omits `tlsEnabled`, so `gateway.tls.enabled` would still produce plaintext dashboard links. Current main passes TLS state and the dashboard docs require `https`/`wss` in that mode.
  - confidence: 0.88

## Security Review
