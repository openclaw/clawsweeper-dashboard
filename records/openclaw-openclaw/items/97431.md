## Summary

Keep open: the PR targets the right Feishu invalid-token bug, but the retry can clear the streaming-card token cache while retrying SDK client requests that use the Lark SDK token cache, and the retry loop can lose Feishu diagnostics when token-invalid appears on the final retry slot. It also needs redacted real Feishu runtime proof before merge.

## What This Changes

Adds Feishu token-invalid code detection, an optional `requestFeishuApi` invalidation hook, streaming-card cache invalidation wiring, and unit tests.

## Best Possible Solution

Move invalid-token recovery to the Feishu client/token owner boundary so SDK-backed sends, media, and streaming-card paths invalidate the token they actually use, retry once, preserve diagnostics, and include focused tests plus redacted live Feishu proof.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection and the linked issue show current main treats 99991663/99991664 as non-retryable on the Feishu send helper path.

## Solution Assessment

No. Token-invalid classification is the right direction, but the retry must invalidate the SDK token/cache used by the captured client request and preserve the original Feishu error on retry exhaustion.
