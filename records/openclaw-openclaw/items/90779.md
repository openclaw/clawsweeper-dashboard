## Summary

Keep open: the PR targets a real current-main source path, but it still lacks after-fix proof from a patched real plugin-channel login and the automatic gateway restart is an availability-sensitive lifecycle choice.

## What This Changes

This PR adds a channel-login fallback that calls `gateway.restart.request` when post-login `channels.start` returns unloaded-plugin invalid/unknown-channel errors, with focused CLI tests.

## Best Possible Solution

Land this only if maintainers want a post-login safe-restart mitigation in addition to the startup-planner work, after redacted real plugin-channel proof shows login, restart request, post-restart running status, and ideally inbound delivery.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The source path is reproducible: current main calls `channels.start` after login, and the gateway rejects a channel absent from the active registry with the reported invalid/unknown-channel errors.

## Solution Assessment

Acceptable mitigation, not yet proven best. Restart is an existing recovery seam for process-stable plugin metadata, but the broader externally-installed plugin startup failures may need the startup-planner fix rather than relying on login to restart the gateway.
