## Summary

Keep open: this is a focused candidate fix for a real post-login channel reconciliation bug, but it is not merge-ready because the restart detector can match the wrong error class, the supplied proof does not show a patched external-plugin login recovering after restart, and maintainers still need to accept login-triggered safe restart as the intended recovery path.

## What This Changes

The PR adds a CLI fallback that requests `gateway.restart.request` when post-login `channels.start` fails with selected unloaded-channel-plugin error text, plus regression tests for that fallback.

## Best Possible Solution

Land one canonical lifecycle fix after the detector is narrowed to structured registry-miss request errors, maintainers accept safe restart as the intended recovery, and redacted external-plugin proof shows login, restart, and recovery.

## Reproduction Assessment

Yes at source level: current main still calls `channels.start` after local login, and gateway normalization can reject an unloaded external channel from the active registry. I did not run a live Weixin or third-party QR-login setup in this read-only review.

## Solution Assessment

No, not as written. The restart fallback is a plausible mitigation, but it should use the gateway request-error contract and needs real external-plugin proof plus maintainer agreement that restart is the canonical recovery.
