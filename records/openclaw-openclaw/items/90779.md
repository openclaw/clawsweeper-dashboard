## Summary

## What This Changes

The PR adds a CLI fallback that requests `gateway.restart.request` when post-login `channels.start` fails because the channel plugin is not loaded in the running gateway.

## Best Possible Solution

Land this only after maintainers confirm safe gateway restart is the intended recovery for unloaded channel plugins and the PR body includes redacted real plugin-channel proof showing login, restart request, post-restart running status, and preferably inbound delivery.

## Reproduction Assessment

Yes for source-level reproduction, but I did not establish a live failing run. Current main calls `channels.start` after local channel login, and the gateway rejects a channel absent from the running plugin registry as `unknown channel`, matching the linked report’s logs.

## Solution Assessment

Acceptable mitigation, not fully proven best. Restart is an existing seam for plugin registry changes, but the automatic restart fallback needs maintainer acceptance and real after-fix proof before it is the best merge path.
