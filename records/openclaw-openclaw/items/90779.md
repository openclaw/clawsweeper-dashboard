## Summary

Keep open: current main and the latest release still have the post-login `channels.start`-only reconcile path, and this PR is a plausible mitigation, but the supplied proof does not show the patched real plugin-channel login triggering restart and reaching a running state. The automatic safe-restart behavior also needs maintainer acceptance because it can affect active gateway work.

## What This Changes

The PR adds a CLI fallback that requests `gateway.restart.request` when post-login `channels.start` fails for an unloaded channel plugin, plus focused mocked tests.

## Best Possible Solution

Land this only after maintainers accept safe gateway restart as the intended recovery for unloaded channel plugins and the PR body includes redacted real plugin-channel proof showing login, restart request, post-restart running status, and preferably inbound delivery.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The source path is reproducible: current main calls `channels.start` after login, and the gateway rejects a channel absent from the active registry with the reported invalid/unknown channel errors.

## Solution Assessment

Acceptable mitigation, not yet proven best. Restart is an existing seam for plugin registry changes, but automatic restart from login needs maintainer acceptance and real after-fix proof; an explicit plugin reload path would be the cleaner direction if maintainers reject the restart tradeoff.
