## Summary

Keep open. The shared-validator fix is the right boundary and the supplied proof covers the normal unreadable-schema case, but the new prefix-based rethrow can let plugin-controlled schema-read errors bypass terminal-text sanitization.

## What This Changes

The PR wraps plugin JSON-schema validation internals so unreadable schema objects report sanitized invalid-schema errors and adds cached and uncached regression coverage.

## Best Possible Solution

Keep the shared validator approach, but brand or re-sanitize internal invalid-schema errors and add a regression for prefixed unreadable-schema trap messages before merge.

## Reproduction Assessment

Yes, from source inspection: a schema Proxy whose trap throws an Error beginning with `invalid schema:` reaches the new `runSchemaOperation` prefix branch and is rethrown before sanitization. I did not execute the repro because this review was read-only.

## Solution Assessment

No as submitted. The shared validator is the best fix location, but the implementation should not trust an arbitrary error-message prefix to identify already-sanitized internal errors.
