## Summary

Keep open: the shared validator is the right place for this fix, but the patch still has a concrete sanitizer bypass because an unreadable schema can throw an `invalid schema:`-prefixed error and be rethrown without `sanitizeTerminalText`. The PR is also draft and labeled `maintainer`, so it needs explicit maintainer readiness after the code repair.

## What This Changes

The PR wraps shared plugin JSON Schema validation operations so unreadable schema objects raise invalid-schema errors, with cached and uncached regression coverage.

## Best Possible Solution

Keep the shared-validator mitigation, but make internal invalid-schema detection unspoofable and prove prefixed, newline, and ANSI schema-trap errors stay sanitized before merge.

## Reproduction Assessment

Yes. Source inspection shows current main can leak raw unreadable-schema traversal errors, and the PR branch source shows a reproducible sanitizer bypass when the thrown schema error message starts with `invalid schema:`.

## Solution Assessment

No. The shared validator is the right layer, but the current implementation should use an internal sentinel or private error class instead of trusting a public message prefix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not trust the invalid-schema message prefix:** `src/plugins/schema-validator.ts:66-67`
  - body: Here any caught `Error` whose message starts with `invalid schema:` is treated as an internal sanitized error and rethrown unchanged. An unreadable schema Proxy can throw `new Error("invalid schema:\n...")` from `ownKeys`, which is the same path this PR is hardening, and that bypasses `sanitizeTerminalText`. Use an internal sentinel/error class or always rebuild external errors through `invalidSchemaError`, then add a regression for newline/ANSI escaping.
  - confidence: 0.9
