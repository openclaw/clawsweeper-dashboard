## Summary

Keep open. This is a useful additive media-understanding seam with supplied real-provider proof, but the PR currently makes the advertised OpenAI diarized path fail-prone because the runner still supplies OpenClaw's default audio prompt while OpenAI's diarize model does not support prompts.

## What This Changes

The PR adds typed audio transcript segments, forwards diarized OpenAI-compatible audio options from providerOptions, exports the segment type through the Plugin SDK, refreshes the SDK baseline, and adds focused media-understanding tests.

## Best Possible Solution

Make diarized audio requests prompt-safe, keep the additive segment contract narrow, then land after maintainer approval of the SDK/providerOptions semantics and docs expectation.

## Reproduction Assessment

Yes for the PR defect by source inspection: the runner resolves a default audio prompt and passes it alongside the new diarized responseFormat. The feature request itself is not a current-main bug; current main has no segments path to reproduce.

## Solution Assessment

No, not yet. The patch shape is close, but diarized OpenAI-compatible requests should suppress or reject prompts before merge so the advertised OpenAI diarize path is compatible with the upstream contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Suppress prompts for diarized audio requests:** `src/media-understanding/runner.entries.ts:683-687`
