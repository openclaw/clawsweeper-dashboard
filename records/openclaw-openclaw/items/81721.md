## Summary

Keep open. The PR adds a useful additive diarized transcript seam with sufficient real provider proof, but it still sends the runner's default audio prompt on the configured diarized OpenAI path, which conflicts with the OpenAI transcription contract and can break official diarized requests.

## What This Changes

The PR forwards audio `response_format` and `chunking_strategy`, preserves typed transcript segments, re-exports the segment type from the Plugin SDK, refreshes the generated SDK baseline hash, and adds focused media-understanding tests.

## Best Possible Solution

Land the additive segment contract after guarding or rejecting the unsupported OpenAI diarized prompt combination and confirming maintainer approval for the public SDK/output addition.

## Reproduction Assessment

Yes for the review finding by source inspection: PR head can set `responseFormat` to `diarized_json` while the runner still supplies the default audio prompt, and OpenAI documents that prompts are unsupported for `gpt-4o-transcribe-diarize`. I did not run a live official OpenAI request in this read-only review.

## Solution Assessment

No, not yet. Reusing existing `providerOptions` and preserving `segments` is a good shape, but the implementation needs to suppress or reject the unsupported prompt/model/response-format combination before the official OpenAI path is safe.
