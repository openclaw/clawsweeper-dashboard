## Summary

This active external feature PR should stay open, but the current patch still has a blocking compatibility bug for the official OpenAI diarized transcription path: the new diarized request fields can be enabled through media provider options while the runner still sends the default audio prompt, which OpenAI documents as unsupported for `gpt-4o-transcribe-diarize`.

## What This Changes

The PR adds typed diarized audio transcript segments, forwards `response_format` and `chunking_strategy` through the OpenAI-compatible audio helper, re-exports the segment type from the Plugin SDK surface, refreshes the generated SDK baseline hash, and adds focused tests.

## Best Possible Solution

Land the additive diarized transcript seam after adding a narrow guard for the official diarized OpenAI prompt incompatibility and after maintainer approval of the new SDK/output contract.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR head can set `responseFormat` to `diarized_json` while the runner still supplies the default audio prompt, and OpenAI documents prompts as unsupported for `gpt-4o-transcribe-diarize`. I did not run a live OpenAI request in this read-only review.

## Solution Assessment

No, not yet; reusing existing `providerOptions` and preserving `segments` is a good shape, but the implementation needs to suppress or reject the unsupported prompt/model/response-format combination before it is safe for the official OpenAI path.
