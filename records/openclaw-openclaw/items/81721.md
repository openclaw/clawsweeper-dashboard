## Summary

Keep open: this fresh feature PR is not implemented on current main, and the diff has a blocking OpenAI diarization compatibility bug plus missing real behavior proof.

## What This Changes

The PR adds typed diarized transcript segments to media-understanding audio outputs, maps providerOptions response_format/chunking_strategy into audio requests, appends those multipart fields in the OpenAI-compatible helper, and adds mocked tests.

## Best Possible Solution

Land the additive media-understanding contract only after the request builder omits unsupported prompt fields for diarized OpenAI requests and the contributor adds redacted real provider proof that segments flow through.

## Reproduction Assessment

Yes for the review finding: source inspection shows OpenClaw always supplies the default audio prompt, and the PR's helper still includes prompt when sending diarized_json. The upstream OpenAI contract says gpt-4o-transcribe-diarize does not support prompt.

## Solution Assessment

No; the general contract shape is reasonable, but the current request shape will break the PR's official OpenAI diarized path. The safer fix is to gate or suppress prompt by model/response-format support and add a regression test for gpt-4o-transcribe-diarize.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Omit prompts for diarized OpenAI transcriptions:** `src/media-understanding/openai-compatible-audio.ts:105-106`
  - body: When a user configures the PR's own OpenAI diarized path with `gpt-4o-transcribe-diarize`, `runProviderEntry` still passes the default audio prompt (`Transcribe the audio.`), and this helper includes it in the multipart body alongside `response_format=diarized_json`. OpenAI documents that `prompt` is not supported for that model, so the official provider rejects the request before any `segments[]` can be returned. Suppress the default prompt for diarized OpenAI requests, or only send a prompt when the selected transcription model supports it. ([platform.openai.com](https://platform.openai.com/docs/api-reference/audio/createTranscription.class?utm_source=openai))
  - confidence: 0.9
