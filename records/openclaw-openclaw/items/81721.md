## Summary

Keep open: the PR is useful and active, but current head still leaves the OpenAI Codex audio path outside the diarized prompt guard, so a configured Codex diarized transcription can fail before sending the provider request.

## What This Changes

The PR adds diarized JSON audio segment support, forwards OpenAI-compatible audio response_format/chunking_strategy options, preserves segments in media-understanding results, exports the segment type through the Plugin SDK, and adds focused tests.

## Best Possible Solution

Keep the provider-option and segment-preservation seam, but make the prompt guard cover every OpenAI-backed diarized audio provider or centralize that decision in the OpenAI-compatible helper with regression coverage.

## Reproduction Assessment

Yes for the review finding: source inspection shows provider openai-codex with model gpt-4o-transcribe-diarize and response_format=diarized_json keeps the default audio prompt, then reaches the helper path that rejects prompts. I did not execute tests in this read-only sweep.

## Solution Assessment

No as-is: the main seam is maintainable, but the prompt guard is too provider-specific for an OpenAI-backed provider already registered in core. The safer fix is to cover openai-codex or centralize implicit-prompt handling in the OpenAI-compatible helper.
