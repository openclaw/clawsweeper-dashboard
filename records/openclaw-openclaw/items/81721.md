## Summary

Keep this PR open, but it still needs a narrow correctness fix before merge: the diarized prompt guard only handles `openai`, while `openai-codex` routes through the same OpenAI-compatible audio helper and can still fail before fetch. The supplied real provider output is sufficient and CI is green, but the remaining blocker is source-proven.

## What This Changes

This PR adds typed diarized transcript segments, forwards OpenAI-compatible audio `response_format` and `chunking_strategy` provider options, preserves returned `segments[]`, exports the segment type through the Plugin SDK, and adds focused media-understanding tests.

## Best Possible Solution

Keep the segment-preservation and provider-option seam, but make the diarized prompt decision cover every OpenAI-compatible diarized audio path and add an `openai-codex` regression before merge.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows `openai-codex` remains outside the runner’s default-prompt omission but still routes to the helper that rejects prompts for the diarized model. I did not run tests because this was a read-only sweep.

## Solution Assessment

No as-is: the main seam is maintainable, but the prompt guard is still too provider-specific for an OpenAI-compatible provider already registered in core. The safer fix is to cover `openai-codex` or centralize implicit-prompt handling in the helper/runner contract with regression coverage.
