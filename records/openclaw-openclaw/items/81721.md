## Summary

Keep open: this is a useful additive media-understanding feature with sufficient real-provider proof, but the current PR head has a blocking correctness issue for non-`openai` OpenAI-compatible diarized transcription providers and still needs maintainer approval for the public SDK/output contract.

## What This Changes

The branch forwards diarized audio request fields, preserves normalized transcript `segments[]` through media-understanding result types, exports the segment type through the Plugin SDK, refreshes the generated SDK baseline hash, and adds focused tests.

## Best Possible Solution

Fix the non-`openai` OpenAI-compatible diarized prompt path, add focused coverage for that provider shape, then land the additive segment contract after maintainer approval and green CI.

## Reproduction Assessment

Yes for the review finding by source inspection: configure a non-`openai` OpenAI-compatible audio provider with `response_format: diarized_json`, and the PR head runner leaves the default audio prompt in place because of the provider-id guard. As a feature PR, there is no original bug reproduction to run.

## Solution Assessment

No as-is: reusing `providerOptions` and carrying normalized segments is a good narrow direction, but the prompt handling is tied too tightly to provider id `openai`. The safer fix is to base unsupported-prompt handling on diarized response/model semantics and cover a non-`openai` compatible provider in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle diarized prompts for compatible providers:** `src/media-understanding/runner.entries.ts:453-459`
  - body: `resolveAudioPromptForProvider` returns before checking `responseFormat` for any provider id other than `openai`. A configured OpenAI-compatible provider such as `openai-codex` or the OCI-style provider used in the proof can set `response_format: "diarized_json"`, but the runner will still pass the default `"Transcribe the audio."` prompt, causing exact OpenAI diarized models to be rejected by the helper or prefixed compatible models to receive an unsupported prompt. Move this handling off the hard-coded provider id and add a non-`openai` provider test.
  - confidence: 0.87
