## Summary

Keep open: the linked issue is valid and this PR moves in the right gateway-delegation direction, but the current diff introduces blocking CLI regressions by bypassing existing message-action normalization for presentation-only and local-media sends, and it lacks the required changelog entry.

## What This Changes

The PR forwards Telegram message-action execution mode through the bundled wrapper, changes message gateway calls to use the normal gateway helper, adds a Telegram CLI send fast path through `message.action`, and adds focused regression tests.

## Best Possible Solution

Keep the wrapper `resolveExecutionMode` fix and the gateway-helper cleanup, but change the CLI Telegram path so it either reuses the existing message-action normalization pipeline or faithfully preserves its contract: presentation-only sends, parsed `presentation`/`delivery`, explicit local-media root expansion, dry-run behavior, and existing gateway scoping. Add the missing Unreleased/Fixes changelog entry and focused regressions before review against #75477.

## Reproduction Assessment

Yes. The linked issue gives a concrete live path, and current `main` explains it: Telegram declares gateway execution but the bundled wrapper does not forward that mode while the CLI loads channel plugins before dispatch. The PR regressions are also reproducible by comparing the fast-path diff with the existing docs, runner parsing, and media-root normalization paths.

## Solution Assessment

No, not as-is. Gateway delegation is the right fix direction, but this fast path is too narrow; the safer solution is to preserve the existing normalization/media-access behavior or limit the fast path to cases it can forward without changing the documented send contract.
