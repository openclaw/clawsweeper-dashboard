## Summary

Keep this PR open, but it should not merge as-is. Current main still has the Telegram gateway-dispatch gap the PR is trying to fix, and the wrapper execution-mode part is useful; however, the added CLI fast path regresses documented presentation-only sends, bypasses local-media normalization, and misses the required user-facing changelog entry.

## What This Changes

The PR forwards Telegram message-action gateway execution mode through the bundled plugin wrapper, changes message-action gateway calls to use the normal CLI gateway helper, adds a Telegram-specific CLI send fast path, and adds focused regression tests.

## Best Possible Solution

Merge a repaired version that keeps the Telegram wrapper `resolveExecutionMode` fix and CLI gateway-helper cleanup, but either routes only simple text Telegram sends through the fast path or makes the fast path preserve the normal runner's JSON parsing, presentation-only support, delivery hints, and local-media access behavior. Add focused regression tests for those contracts and a single-line Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes. The linked issue gives a concrete `openclaw message send --channel telegram --account default --target ... --message ... --json` path, and current-main source shows the CLI still loads configured channel plugins while the Telegram wrapper drops the gateway execution-mode declaration.

## Solution Assessment

No, not as-is. Delegating Telegram message sends to the live gateway is the right direction, but the PR's broad CLI fast path bypasses existing normalization and media-access contracts that current message sends rely on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve presentation-only Telegram sends:** `src/cli/program/message/helpers.ts:76-77`
  - body: The fast path catches every non-dry-run Telegram send with a target, but then rejects calls unless `--message` or `--media` is present and forwards `presentation`/`delivery` as raw strings. The documented CLI contract allows `--presentation` alone, and the normal runner parses these JSON options before plugin dispatch, so direct `message.action` dispatch will fail presentation-only/button sends or ignore parsed delivery hints.
  - confidence: 0.88
- **[P2] Keep local media sends on the normalized path:** `src/cli/program/message/helpers.ts:88-90`
  - body: This fast path also intercepts `openclaw message send --channel telegram --media /path/...` and forwards only `mediaUrl` to the gateway. The gateway `message.action` handler dispatches directly to Telegram without `mediaLocalRoots`, while the existing runner expands local media parent roots before dispatch, so explicit files outside default OpenClaw roots can regress to local media access failures.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/cli/program/message/helpers.ts:158-160`
  - body: This is a user-facing CLI/Telegram bug fix, but the PR does not update `CHANGELOG.md`. Repo policy requires user-visible fixes to be listed under Unreleased/Fixes, so release notes would miss the Telegram CLI hang fix.
  - confidence: 0.97
