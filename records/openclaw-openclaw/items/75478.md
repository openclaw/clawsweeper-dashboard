## Summary

Keep open: the PR targets a real Telegram CLI gateway-dispatch bug, but the proposed fast path regresses existing presentation-only and local-media send contracts and is missing the required changelog entry.

## What This Changes

The PR forwards Telegram action execution mode through the bundled wrapper, switches message-action gateway calls to `callGateway`, adds a Telegram CLI gateway fast path, and adds focused tests.

## Best Possible Solution

Keep the wrapper execution-mode and gateway-helper direction, but make the Telegram CLI gateway path preserve the existing send normalization contract for presentation, delivery, and local media before adding changelog coverage.

## Reproduction Assessment

Yes. The linked issue gives a concrete `openclaw message send --channel telegram ... --json` path, and current-main source shows the CLI still loads configured channel plugins before dispatch while the Telegram wrapper drops the gateway execution-mode declaration.

## Solution Assessment

No, not as proposed. Delegating Telegram sends to the live gateway is the right direction, but the fast path needs to either reuse the normal normalization pipeline or stay limited to simple text sends so it does not break richer CLI sends.
