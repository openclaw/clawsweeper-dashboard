## Summary

Keep open: the linked Telegram CLI hang is valid on current main, and this PR is pointed at the right gateway-owned execution path, but the patch is not merge-ready because the new CLI fast path bypasses existing message-action normalization and media access behavior.

## What This Changes

The PR forwards Telegram message-action execution mode through the bundled plugin wrapper, switches message-action gateway calls to `callGateway`, adds a Telegram CLI `send` fast path that calls `message.action`, and adds focused tests.

## Best Possible Solution

Keep the wrapper execution-mode fix and CLI gateway-scope cleanup, but make the Telegram CLI route preserve the existing message-action normalization contract or restrict the fast path to cases it can faithfully forward. The final patch should cover presentation-only sends, delivery JSON, explicit local media, and release notes before merge.

## Reproduction Assessment

Yes. The linked issue gives a concrete live path, and current-main source explains it: Telegram declares gateway execution while the bundled wrapper does not forward that mode and the CLI loads the channel registry before dispatch.

## Solution Assessment

No, not as-is. Gateway delegation is the right direction, but the proposed fast path is narrower than the existing CLI contract and bypasses parsing/media access that the normal runner provides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve presentation-only Telegram sends:** `src/cli/program/message/helpers.ts:75-76`
  - body: The fast path handles every non-dry-run Telegram send with a target, but it rejects the request unless `--message` or `--media` is present and forwards `presentation`/`delivery` as raw strings. The documented CLI contract allows `--presentation` alone, and the normal runner parses these JSON options before plugin dispatch; direct `message.action` dispatch will fail or lose delivery semantics on this path.
  - confidence: 0.88
- **[P2] Keep local media sends on the normalized path:** `src/cli/program/message/helpers.ts:87-88`
  - body: This fast path also catches `openclaw message send --channel telegram --media /path/...` and forwards only `mediaUrl` to the gateway. The gateway `message.action` handler dispatches params directly, while Telegram local-file access depends on dispatch options such as `mediaLocalRoots`, so explicit local files outside default roots can regress to local-media access failures.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/cli/program/message/helpers.ts:158`
  - body: This is a user-facing CLI/Telegram bug fix, but the PR does not update `CHANGELOG.md`. Repo policy requires user-visible fixes to be listed under `Unreleased` / `Fixes`, so the release notes would miss the CLI hang fix.
  - confidence: 0.97
