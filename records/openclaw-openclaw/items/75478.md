## Summary

Keep open: the PR addresses a real Telegram CLI gateway-dispatch bug, but the new fast path bypasses existing message-action normalization for presentation/delivery payloads and local-media access handling, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR forwards Telegram message-action execution mode through the bundled plugin wrapper, switches message gateway calls to the normal gateway helper, adds a Telegram CLI send gateway fast path, and adds focused regression tests.

## Best Possible Solution

Keep the wrapper `resolveExecutionMode` fix and the gateway-helper cleanup, but adjust the CLI fast path so it preserves the existing message-action normalization and local-media access contract, or limit the fast path to cases it can faithfully forward. Add focused regressions for presentation-only and local-media Telegram sends plus an `Unreleased` changelog fix entry, then review it against #75477.

## Reproduction Assessment

Yes. The linked issue gives a concrete live path, and current main source explains it: Telegram's implementation declares gateway execution, but the bundled wrapper does not forward that mode while the CLI loads the channel registry before dispatch.

## Solution Assessment

No, not as-is. Delegating Telegram sends to the live gateway is the right direction, but the new CLI fast path is too narrow because it bypasses presentation/delivery parsing and local-media root expansion that the normal message-action runner already provides.

Full review comments:

- **[P2] Preserve presentation-only Telegram sends:** `src/cli/program/message/helpers.ts:75-76`
  - body: The fast path now handles every non-dry-run Telegram send with a target, but it rejects sends unless `--message` or `--media` is present and forwards `presentation`/`delivery` as raw strings. The documented CLI contract allows `--presentation` alone, and the normal runner parses these JSON options before plugin dispatch; `message.action` dispatches directly to Telegram, so presentation-only/button sends will fail or lose parsed delivery options on this path.
  - confidence: 0.88
- **[P2] Keep local media sends on the normalized path:** `src/cli/program/message/helpers.ts:87-88`
  - body: This fast path also catches `openclaw message send --channel telegram --media /path/...`. It forwards only `mediaUrl` to the gateway, but the gateway `message.action` handler dispatches params directly and Telegram only receives `mediaLocalRoots` from dispatch options. That drops the parent-root expansion current CLI sends use for explicit local media paths, so local files outside default OpenClaw roots can regress to `LocalMediaAccessError`.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/cli/program/message/helpers.ts:158`
  - body: This is a user-facing CLI/Telegram bug fix, but the PR does not update `CHANGELOG.md`. Repo policy requires user-visible fixes to be listed under `Unreleased` / `Fixes`, so the release notes would miss the CLI hang fix.
  - confidence: 0.97
