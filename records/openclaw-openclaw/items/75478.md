## Summary

Keep open. The linked Telegram CLI hang is real, and part of this PR is aimed at the right execution-mode boundary, but the proposed CLI fast path bypasses existing send normalization for documented presentation-only/delivery inputs and explicit local-media access, and it is missing the required user-facing changelog entry.

## What This Changes

The branch forwards Telegram message-action execution mode through the bundled plugin wrapper, changes message-action gateway calls to the general gateway helper, adds a Telegram CLI gateway fast path for sends, and adds focused regression tests.

## Best Possible Solution

Keep the wrapper execution-mode fix and the gateway-helper cleanup, but repair the Telegram CLI path so it avoids local Telegram runtime loading while preserving the same normalization, JSON parsing, delivery options, and local-media access contract as the normal message-action runner. Add focused regressions for presentation-only and local-media Telegram sends plus an Unreleased/Fixes changelog entry, then review it against #75477.

## Reproduction Assessment

Yes. The linked issue gives a concrete live path, and current main source explains it: Telegram declares gateway execution, but the bundled wrapper does not forward that mode while the CLI loads configured channel plugins before dispatch.

## Solution Assessment

No, not as-is. Delegating Telegram sends to the live gateway is the right direction, but the new CLI fast path is narrower than the existing send contract; the safer fix is to reuse the runner's normalization/media-access path or faithfully mirror it before direct gateway dispatch.
