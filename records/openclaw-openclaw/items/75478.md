## Summary

Keep open: this PR targets a real Telegram CLI hang, but the current diff would regress richer Telegram sends by bypassing existing message normalization before gateway dispatch.

## What This Changes

The PR forwards Telegram action execution mode through the bundled wrapper, switches message gateway calls to `callGateway`, and adds a Telegram CLI send fast path with focused tests.

## Best Possible Solution

Keep the wrapper execution-mode and CLI gateway direction, but preserve the existing send normalization contract for presentation, delivery, and local media before adding release-note coverage.

## Reproduction Assessment

Yes. The linked #75477 path is concrete, and current main still loads configured channel plugins before dispatch while the Telegram wrapper does not expose gateway execution mode.

## Solution Assessment

No, not as proposed. Delegating Telegram sends to the live gateway is the right direction, but the fast path must share or faithfully reproduce the runner's normalization instead of bypassing it.

  - body: The fast path catches every non-dry-run Telegram send with a target, then rejects calls unless `--message` or `--media` is present.
