## Summary

Keep open: the updated proof is now useful, but the patch still changes shared Unix cleanup semantics without carrying the actual detached/process-group fact through existing detached cleanup callers. Several current callers spawn detached children and still call `killProcessTree` or `signalProcessTree` without `detached: true`, so this can leave descendants running when the leader PID has exited or the `ps` probe cannot see it.

## What This Changes

The PR adds a Unix `ps -p <pid> -o pgid=` group-leader check before negative-PID group signaling in `killProcessTree` and `signalProcessTree`, plus tests for non-leader fallback and explicit `detached: true` override.

## Best Possible Solution

Before merge, carry the actual detached/process-group fact into every relevant cleanup path or provide an equivalent helper contract that preserves detached descendant cleanup after leader exit, with focused tests and terminal process-group proof.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` still default omitted-detached Unix cleanup to group kill, and the PR-introduced remaining risk is visible in detached callers that still omit `detached: true`. I did not rerun the live macOS launchd gateway scenario in this read-only review.

## Solution Assessment

No as submitted. The group-leader check is a plausible safety gate for attached non-leader children, but the best fix must also preserve detached process-tree cleanup for existing callers that know they spawned detached children.
