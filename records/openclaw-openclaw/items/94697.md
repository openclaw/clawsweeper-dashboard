## Summary

Keep this PR open, but do not merge it as submitted: it addresses a real gateway SIGTERM bug, yet the new `ps -p <pid>` gate can suppress valid detached process-group cleanup after the original group leader exits, which can leave descendants running.

## What This Changes

The PR adds a Unix process-group-leader check before `killProcessTree` or `signalProcessTree` uses negative-PID group signaling, and adds kill-tree tests for non-leader fallback.

## Best Possible Solution

Keep the safety goal, but adjust the patch to carry actual detached/process-group state or otherwise preserve `-pid` cleanup for verified detached groups, with regression coverage and real process proof for both attached and detached cases.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 still default omitted-detached Unix cleanup to group-kill, matching the linked gateway SIGTERM report. I did not live-reproduce the macOS launchd gateway restart, but I did prove the PR-introduced detached-group edge with a contained local process smoke.

## Solution Assessment

No as submitted. The helper-side group-leader check is a plausible part of the fix, but the best solution must preserve valid detached group cleanup after the leader exits or when callers assert detachment.
