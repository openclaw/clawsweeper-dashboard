## Summary

Keep open: current `main` does not prove the requested force-index behavior is fixed. The linked watcher issue and open fallback PR cover stale indexes from missed watcher events, but this issue reports a distinct manual `openclaw memory index --force` path that can return success while the sessions backlog remains dirty.

## What This Changes

The issue reports that `openclaw memory index --force` exits successfully before processing the full memory-search sessions backlog or logging why it stopped.

## Best Possible Solution

Add a verified force-index completion contract: reproduce the large sessions backlog or a provider-throttled equivalent, make `--force` either drain the enabled sources or fail/warn with remaining backlog details, and keep the watcher fallback work separate.

## Reproduction Assessment

No high-confidence current-main reproduction was run. The report has concrete steps, and source inspection shows the CLI can print success after sync returns without verifying remaining dirty/backlog state.

## Solution Assessment

Unclear. Making `--force` fail loudly or continue until clean is likely the right product behavior, but maintainers need current-main/provider-backed proof to choose between retry/backoff, completion verification, and better status/error surfacing.
