## Summary

Keep open. Current main still wires ACP `streamTo="parent"` through the parent relay, but that relay only surfaces assistant text and lifecycle status while dropping structured tool, command output, patch, approval, item, plan, and thinking streams. The prior ClawSweeper review and related closed channel-visibility issue do not supersede this narrower ACP parent-relay request; the remaining blocker is maintainer policy for verbosity and redaction.

## What This Changes

Request ACP `streamTo="parent"` to expose child tool calls, tool results, and thinking/progress details to the parent session instead of only assistant text and lifecycle updates.

## Best Possible Solution

Add a narrow ACP parent-relay policy that emits bounded, redacted summaries for structured progress streams while preserving existing assistant/lifecycle summaries, gated by the maintainer-chosen verbose/default behavior and covered by focused relay tests plus ACP docs updates.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to start `startAcpSpawnParentStreamRelay()`, emit a matching child-run `tool` or `thinking` event, and observe that the callback returns at the non-lifecycle guard without enqueueing a parent system event.

## Solution Assessment

Unclear until maintainers choose the visibility policy. The implementation likely belongs in `src/agents/acp-spawn-parent-stream.ts`, but raw full tool/result/thinking relay would be less maintainable than redacted summaries with explicit verbosity gating.
