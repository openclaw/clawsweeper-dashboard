## Summary

Keep open. Current main still wires ACP `streamTo="parent"` through the parent relay, but the relay only surfaces assistant text and lifecycle status; structured tool, command output, patch, approval, item, plan, and thinking streams are still dropped before they can enqueue parent-session system events. The related closed channel-visibility request is broader and ClawHub-oriented, so it does not supersede this ACP runtime relay gap.

## What This Changes

The issue asks for ACP child runs using `streamTo="parent"` to expose tool calls, tool results, and thinking progress to the parent session instead of only assistant text and lifecycle events.

## Best Possible Solution

Add a narrow ACP parent-relay policy that preserves existing assistant and lifecycle summaries while emitting bounded, redacted summaries for structured progress streams, gated by the maintainer-chosen verbose/default behavior, with focused relay tests and ACP docs updates.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to start `startAcpSpawnParentStreamRelay()`, emit a matching child-run `tool`, `command_output`, or `thinking` agent event, and observe that the callback returns at the non-lifecycle guard without enqueueing a parent system event.

## Solution Assessment

Unclear until maintainers choose the visibility and redaction policy. The implementation likely belongs in `src/agents/acp-spawn-parent-stream.ts`, but raw full tool/result/thinking relay would be weaker than redacted summaries with explicit verbosity gating.
