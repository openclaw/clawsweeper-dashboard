## Summary

Keep open: the PR addresses a real workspace-only filesystem boundary gap, but the current implementation is too broad because it disables the shared Codex native surface instead of only constraining native filesystem access, and it still lacks real behavior proof.

## What This Changes

The PR disables the Codex app-server native tool surface when effective OpenClaw `tools.fs.workspaceOnly` is true and adds tests for agent, default-agent, runtime-session, global, opt-out, and malformed-config policy resolution.

## Best Possible Solution

Enforce workspace-only at the Codex filesystem/native filesystem-execution boundary while preserving unrelated native surfaces, unless maintainers explicitly approve a broad fail-closed upgrade behavior with matching docs and proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current OpenClaw fs policy is enforced in OpenClaw-owned tools, while upstream Codex app-server raw filesystem methods are separate from that wrapper. The PR body does not include a live after-fix run, so runtime proof remains missing.

## Solution Assessment

No. The reported gap is real, but disabling the shared native surface is not the narrowest maintainable fix because it changes unrelated Codex capabilities; a filesystem-boundary guard or maintainer-approved compatibility break is needed.
