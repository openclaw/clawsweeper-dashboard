## Summary

Keep open: the PR targets a real workspace-only filesystem gap, but current head still applies that filesystem policy by disabling the whole Codex native surface and still lacks external real behavior proof, so it is not merge-ready.

## Best Possible Solution

## Reproduction Assessment

Yes, source-level: current main enables the broad Codex native surface when sandbox and allowlist gates pass, while OpenClaw-owned file tools separately enforce `tools.fs.workspaceOnly`. I did not establish a live after-fix Codex app-server run for this PR.

## Solution Assessment

No: the PR addresses a real security gap, but using the broad native-surface switch is not the narrowest maintainable fix for a filesystem-only policy. A narrower filesystem/native-execution boundary would preserve unrelated Codex native capabilities.
