## Summary

Keep open. The linked docs problem is valid, but this PR is not mergeable as-is because the docs wording over-narrows `--load networkidle` support to managed profiles and the branch includes an unrelated OpenRouter runtime change that can break `openrouter/auto`.

## What This Changes

This PR updates three browser docs pages to qualify `wait --load networkidle` support and changes OpenAI transport request construction to strip `openrouter/` from OpenRouter model IDs.

## Best Possible Solution

Keep this as a docs-only fix that scopes the unsupported case to existing-session/Chrome MCP profiles, and move any OpenRouter model-id behavior change to a separate tested PR with real proof.

## Reproduction Assessment

Yes. The docs contradiction is reproducible by reading current main, and source/tests establish that `networkidle` is rejected only on existing-session while non-existing-session paths go through Playwright.

## Solution Assessment

No. A docs-only clarification is the right direction, but this implementation is too narrow for remote CDP and bundles an unrelated provider runtime change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Describe the unsupported case as existing-session only:** `docs/tools/browser-control.md:289`
  - body: This qualifier says `--load networkidle` requires a managed `openclaw` profile, but remote CDP profiles are `openclaw` driver profiles too and the non-existing-session route forwards waits to Playwright. Please phrase the limitation as existing-session/Chrome MCP only so Browserless or other remote-CDP users are not told the supported path is unavailable.

  - confidence: 0.93
