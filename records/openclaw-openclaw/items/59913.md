## Summary

Keep open. Current main still contains both reported paths: OpenAI-compatible image parts are forwarded as `existingImages` and can return before the prompt image sanitizer when the prompt has no image references, while lifecycle `error` events in the streaming handler still request finalization before visible error content is guaranteed. The issue is also paired with open same-author PR #62070 using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that OpenAI-compatible `/v1/chat/completions` image parts be sanitized/resized before provider handoff and that streaming provider errors be surfaced before the terminal SSE marker.

## Best Possible Solution

Review and land PR #62070 or an equivalent narrow fix that sanitizes `existingImages` on the no-ref prompt path and writes one visible streaming error chunk before final SSE finalization. Keep the separate iOS/client-side image resizing work tracked by #68524/#73710 separate from this server-side gateway bug.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: send an OpenAI-compatible `image_url` part with ordinary text and no prompt file/media refs, and the current runner returns `existingImages` before sanitization; a lifecycle `error` event before the async catch writes content can finalize the stream first.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: sanitize gateway-provided images in the shared prompt image loader and handle lifecycle streaming errors before finalization. The fix is not present on current main, and PR #62070 is the current implementation candidate.
