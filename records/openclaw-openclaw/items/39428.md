## Summary

Keep open: current main and v2026.6.10 still route named Discord accounts through explicit binding tiers and then the default agent, while the same-name account fallback remains a security-sensitive product decision with a conflicting open fork PR candidate.

## What This Changes

The issue requests Discord inbound routing to map a named account to an agent with the same ID when no explicit binding exists.

## Best Possible Solution

Maintainers should decide whether explicit bindings remain the contract or land a guarded Discord-only fallback with an agent-list guard, distinct diagnostics, docs, regression tests, and redacted real Discord proof.

## Reproduction Assessment

Yes, at source level. Discord passes accountId into resolveAgentRoute, and the resolver falls through to the configured default agent after explicit binding tiers when no binding matches.

## Solution Assessment

Unclear. A guarded Discord-only accountId-to-agent fallback is plausible, but current docs support explicit bindings and repository policy treats fallback behavior as a product/security decision.
