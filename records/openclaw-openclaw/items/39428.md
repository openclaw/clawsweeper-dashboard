## Summary

Keep open: current main still falls through to the default route after explicit binding tiers, and https://github.com/openclaw/openclaw/pull/39485 directly targets this issue but remains open and unmerged. The remaining question is a security-sensitive Discord routing contract, not cleanup-safe closure.

## What This Changes

Requests Discord routing to implicitly map an inbound Discord account ID to a same-named configured agent when no explicit binding matches, instead of falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or an equivalent maintainer-approved Discord routing change with a configured-agent guard, distinct diagnostics, docs/tests, and redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: a Discord route with accountId "agentA", an agents.list entry for "agentA", and no binding still falls through to the default route on current main. I did not run live Discord.

## Solution Assessment

Unclear. A same-name fallback is plausible if maintainers accept the convention, but current docs specify explicit bindings and the accepted implementation should guard against phantom agents, use clear diagnostics, and include real Discord proof.
