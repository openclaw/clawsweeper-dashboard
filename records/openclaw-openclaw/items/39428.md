## Label Justifications

- `P1`: Discord multi-account traffic can reach the wrong active agent and permission profile when operators omit explicit bindings.
- `impact:session-state`: Messages and status attach to the default agent session rather than the intended account-specific session.
- `impact:security`: A restricted Discord account can fall through to a more privileged default agent and tool policy.
- `impact:auth-provider`: The inbound account identity resolves to an unintended agent and model selection.

## Summary

Current main still performs the reported default-agent fallback, and no merged or viable canonical replacement implements same-name routing. The request remains open, but because current documentation makes explicit bindings the routing contract, automatic name matching is a compatibility-sensitive product change rather than a strict bug fix.

## What This Changes

Add Discord routing that selects a configured same-named agent when its account ID matches and no explicit binding applies, while preserving default-account behavior.

## Best Possible Solution

Define a single Discord-owned routing policy shared by all Discord inbound surfaces that verifies an actual configured agent, preserves explicit-binding precedence, excludes the reserved default account, records a distinct match reason, and documents the compatibility change.

## Reproduction Assessment

Yes, the reported fallback is source-reproducible: without a matching binding, Discord passes its account ID into a resolver whose unmatched path selects the default agent. This read-only review did not run a live Discord environment.

## Solution Assessment

No, not as previously proposed: same-name inference is plausible, but the earlier shared-resolver patch could route to phantom agents and did not settle compatibility or complete Discord-surface ownership.
