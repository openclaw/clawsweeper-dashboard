## Summary

Keep open: current main still falls through to the default agent after explicit Discord/account binding tiers, and https://github.com/openclaw/openclaw/pull/39485 directly targets this issue with closing syntax but remains open. Maintainers still need to decide whether implicit same-name Discord account routing should become supported policy.

## What This Changes

Request Discord inbound routing to map an unmatched accountId to a same-named agent before falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or an equivalent maintainer-approved Discord routing change with explicit policy, guards, diagnostics, docs, tests, and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: a Discord route with accountId "agentA", an agents.list entry for "agentA", and no bindings still reaches the default fallback on current main. I did not run live Discord, and the unresolved part is whether same-name fallback is the intended contract.

## Solution Assessment

Unclear. The linked resolver fallback is plausible if maintainers accept the convention, but the maintainable version should require a configured agent, use distinct diagnostics, update docs, and include real Discord proof.
