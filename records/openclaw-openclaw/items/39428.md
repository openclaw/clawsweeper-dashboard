## Summary

Keep open: current main and v2026.5.12 still route unmatched Discord accounts through explicit binding tiers and then the default agent, while https://github.com/openclaw/openclaw/pull/39485 references this issue with closing syntax but has not merged. The requested implicit same-name routing is security-sensitive and still needs maintainer product/security review.

## What This Changes

The issue requests Discord account IDs that match agent IDs to route implicitly to that same-named agent instead of falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement only after it requires an existing configured agent, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the shared resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. A same-name Discord fallback is plausible, but it changes routing/security semantics and should land only with maintainer approval, a configured-agent guard, distinct diagnostics, regression coverage, docs, and real Discord proof.
