## Summary

Keep open: current main still routes unmatched Discord accounts through explicit binding tiers and then the default agent, and the open linked implementation PR is not merged. The requested same-name fallback is security-sensitive and changes the documented explicit-binding contract, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

Route Discord inbound messages to a same-named configured agent when the Discord account ID matches the agent ID instead of falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement design that maps only configured agents, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. A same-name Discord fallback is plausible, but it changes routing/security semantics and should land only with maintainer approval, a configured-agent guard, distinct diagnostics, regression coverage, docs, and real Discord proof.
