## Summary

Keep open: current main and v2026.5.12 still fall through to the default agent when no explicit Discord account binding matches, and an open implementation PR with closing syntax already owns the immediate fix path.

## What This Changes

The issue requests implicit Discord routing from a channel account ID to a same-named agent when no explicit binding exists.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or an equivalent maintainer-approved routing change with an existing-agent guard, distinct diagnostics, docs/tests, and redacted Discord proof.

## Reproduction Assessment

Yes, at source level: Discord forwards accountId into resolveAgentRoute, and when no binding tier matches, current main returns resolveDefaultAgentId with matchedBy default. I did not run live Discord.

## Solution Assessment

Unclear. A same-name fallback is plausible, but it changes a permission-sensitive routing convention and should land only with maintainer approval, an existing-agent guard, distinct match reporting, regression coverage, docs, and live Discord proof.
