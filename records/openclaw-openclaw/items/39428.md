## Summary

Keep open: current main still routes Discord account traffic only through explicit binding tiers before default fallback, and open PR #39485 directly targets this issue with closing syntax but has not merged.

## What This Changes

The issue requests Discord inbound routing to select a same-named agent for a Discord accountId when no explicit binding exists, instead of falling back to the default agent.

## Best Possible Solution

Review and land PR #39485 or an equivalent maintainer-approved fix that adds a narrow Discord-only fallback after explicit binding tiers, skips accountId "default", requires an existing matching agent, and includes regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call resolveAgentRoute or resolveDiscordConversationRoute with channel "discord", accountId "agentA", agents.list containing "agentA", and no bindings; current main reaches the default fallback instead of selecting agentA.

## Solution Assessment

Yes, if maintainers accept the implicit-routing policy. The narrowest maintainable direction is Discord-only same-name fallback after explicit bindings and before default fallback, with a guard for accountId "default" and regression tests.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because default fallback can route a named Discord account intended for a restricted agent into the default agent.

Concerns:

- **[medium] Default fallback can bypass per-agent restrictions:** `src/routing/resolve-route.ts:818`
  - body: When no explicit binding matches, current main falls through to the configured default agent. In the reported multi-account Discord setup, that can send messages from an account meant for a restricted same-named agent to the broader default agent unless operators add explicit bindings.
  - confidence: 0.84
