## Summary

Keep open: current main still routes Discord accountId values only through explicit binding tiers before falling back to the default agent, and open PR #39485 directly targets this issue with closing syntax but has not merged.

## What This Changes

The issue requests Discord inbound routing to implicitly choose an agent whose ID matches the Discord accountId when no explicit binding matches, instead of falling back to the default agent.

## Best Possible Solution

Review and land PR #39485 or an equivalent maintainer-approved fix that adds a narrow Discord-only same-name fallback after explicit binding tiers and before default fallback, with regression coverage and a default-account guard.

## Reproduction Assessment

Yes. A source-level reproduction is clear: resolve a Discord route with accountId "agentA", agents.list containing "agentA", and no bindings; current main reaches the default fallback instead of selecting the same-named agent.

## Solution Assessment

Unclear until maintainers accept the routing policy change. If accepted, the narrowest maintainable fix is the linked Discord-only fallback after explicit binding tiers, with tests for positive matching and the "default" account guard.

## Security Review

Status: needs_attention

Summary: The report is security-sensitive because default fallback can route a Discord account intended for a restricted same-named agent into the broader default agent.

Concerns:

- **[medium] Default fallback can cross agent policy boundaries:** `src/routing/resolve-route.ts:818`
  - body: When no explicit binding matches, current main falls through to the configured default agent. In a multi-account Discord setup, an account intended for a restricted same-named agent must be explicitly bound or inbound messages can run under the default agent's sandbox, tools, and auth context.
