## Summary

Keep open: current main and v2026.5.12 still route unmatched Discord accounts through the documented explicit binding tiers and then to the default agent, while the linked closing PR remains open and needs review before this issue can be resolved.

## What This Changes

Requests Discord inbound routing to map a Discord account ID to a same-named agent when no explicit binding matches, instead of falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement only after the implicit fallback contract is approved, guarded to configured agents, documented, tested, and proven with redacted Discord runtime evidence.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no explicit binding the resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. Same-name Discord fallback may be useful, but it changes routing and security semantics and should land only after maintainer approval with configured-agent guarding, docs/tests, and real Discord proof.
