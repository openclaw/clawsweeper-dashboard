## Label Justifications

- `P1`: Wrong-agent Discord routing can cross session, model, workspace, tool, and permission boundaries in real multi-account setups.
- `impact:security`: The reported fallback can send traffic intended for a restricted Discord account into a broader default agent.
- `impact:session-state`: Messages can attach to the default agent session instead of the intended same-named agent session.
- `impact:auth-provider`: The route can select the wrong agent and model context for a Discord account.

## Summary

Keep open: current main and v2026.5.22 still route unmatched Discord accounts through explicit binding tiers and then the default agent. The requested same-name account fallback remains a permission-sensitive routing contract decision; the target-repo PR that tried it was closed unmerged, while only a fork PR remains open.

## What This Changes

The issue asks for Discord inbound messages from an account whose name matches an agent ID to route to that matching agent instead of falling back to the default agent.

## Best Possible Solution

Choose and document one Discord account routing contract, guard it against non-existent or runtime-ineligible agents, add focused resolver and Discord route tests, and collect redacted real Discord multi-account proof before landing.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and current main falls through to the default agent after explicit binding tiers. I did not run live Discord proof in this read-only review.

## Solution Assessment

Unclear. The requested same-name fallback would solve the reported setup, but it changes the documented routing contract; maintainers should choose between guarded implicit fallback, explicit account bindings, or another account-level contract.
