## Summary

Keep open: current main and the latest release still route unmatched Discord accounts through the explicit binding tiers and then the default agent, while the linked implementation PR remains open and conflicting. Because the requested same-name fallback affects permission-sensitive agent routing and changes the documented explicit-binding contract, this needs maintainer review rather than cleanup closure.

## What This Changes

The issue requests implicit Discord routing from a Discord account id to a same-named agent when no explicit binding is configured.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement design that only maps configured agents, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. A same-name Discord fallback is plausible, but it changes the documented explicit-binding contract and should land only with maintainer approval, a configured-agent guard, distinct diagnostics, regression coverage, docs, and real Discord proof.
