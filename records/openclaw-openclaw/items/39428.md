## Summary

Keep open: current main and v2026.5.12 still route unmatched Discord accounts through explicit binding tiers and then the default agent, while an open PR references this issue with closing syntax but has not landed. The requested implicit same-name routing is security-sensitive and not yet part of the documented routing contract.

## What This Changes

Requests Discord routing to infer an agent from a matching non-default account name when no explicit binding matches.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement only after it requires an existing configured agent, reports implicit matches distinctly, updates docs/tests, and includes redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. A same-name Discord fallback is plausible, but it changes routing/security semantics and should land only with maintainer approval, a configured-agent guard, distinct diagnostics, regression coverage, docs, and real Discord proof.
