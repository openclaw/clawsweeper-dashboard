## Summary

Keep open: current main still routes Discord accountId through the generic binding resolver and falls back to the configured default agent when no explicit binding matches. Open PR #39485 directly targets this issue with closing syntax, but it is still open and unmerged, so the issue should remain open until that PR or an equivalent guarded fix lands.

## What This Changes

The issue requests Discord inbound routing to implicitly match a Discord account name to an agent ID before falling back to the default agent.

## Best Possible Solution

Review and land PR #39485 or an equivalent maintainer-approved fix that adds a narrow Discord-only fallback after explicit binding tiers and before default-agent fallback, skips accountId "default", requires an existing matching agent, and includes regression coverage for positive matching and default-account behavior.

## Reproduction Assessment

Yes. The issue has concrete multi-account Discord steps, and the same behavior can be reproduced at the resolver level by calling resolveAgentRoute or resolveDiscordConversationRoute with channel "discord", accountId "agentA", matching agents.list, and no bindings; current main falls through to matchedBy "default".

## Solution Assessment

Yes, the narrow maintainable direction is a Discord-only implicit fallback after explicit bindings and before default fallback. The open PR is a plausible implementation candidate, though maintainers may want an unambiguous match reason such as implicit.account rather than reusing binding.account.
