## Summary

Keep open: current main still falls through to the default agent when no explicit Discord account binding matches, and open PR #39485 directly targets this issue with closing syntax but is unmerged.

## What This Changes

Request that Discord inbound routing map a non-default Discord accountId to an existing same-named agent after explicit bindings fail instead of falling back to the default agent.

## Best Possible Solution

Review and land PR #39485 or an equivalent maintainer-approved fix that adds a narrow Discord-only fallback after explicit binding tiers, skips accountId "default", requires an existing matching agent, and includes regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call resolveAgentRoute or resolveDiscordConversationRoute with channel "discord", accountId "agentA", agents.list containing "agentA", and no bindings; current main reaches the default fallback instead of selecting agentA.

## Solution Assessment

Yes, if maintainers accept the policy change. The narrowest maintainable solution is a Discord-only implicit fallback after explicit binding tiers and before default fallback, with tests for positive matching and the default-account guard.

## Security Review

Status: needs_attention

Summary: The report is security-sensitive because default fallback can route a Discord account intended for a restricted agent into the broader default agent.

Concerns:

- **[medium] Default fallback can bypass per-agent restrictions:** `src/routing/resolve-route.ts:818`
  - body: When no explicit binding matches, current main falls through to the configured default agent. In the reported multi-account Discord setup, that can send messages from an account meant for a restricted same-named agent to the default agent unless operators add explicit bindings.
  - confidence: 0.84
