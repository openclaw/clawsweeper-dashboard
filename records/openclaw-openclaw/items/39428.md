## Label Justifications

- `P1`: Discord wrong-agent routing can cross session, workspace, model, tool, and permission boundaries in real multi-agent setups.
- `impact:security`: Routing a restricted Discord account into the default agent can bypass intended per-agent tool, permission, or workspace restrictions.
- `impact:session-state`: Messages can attach to the default agent session instead of the intended same-named agent session.
- `impact:auth-provider`: The reported route selects the wrong agent/model context for a Discord account.

## Summary

Keep open: current main and v2026.5.20 still route unmatched Discord accounts through explicit binding tiers and then the default agent, while the open same-name routing PR remains unmerged and product/security-sensitive.

## What This Changes

The issue requests Discord inbound messages from an account whose name matches an agent ID to route to that agent instead of the default agent when no explicit binding exists.

## Best Possible Solution

Approve one Discord routing policy that either keeps explicit bindings as required or adds a configured-agent-only same-name fallback with distinct diagnostics, docs, regression tests, and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes the account ID into resolveAgentRoute, and current main falls through to the default agent when no explicit binding matches. I did not run live Discord proof in this read-only review.

## Solution Assessment

Unclear. Same-name fallback may be useful, but current docs/tests define explicit bindings followed by default routing, so the permanent fix needs maintainer approval plus configured-agent guards, diagnostics, docs, tests, and real Discord proof.
