## Summary

Keep open: current main and the latest release still route unmatched named Discord accounts through the default-agent fallback, and the open linked fork PR remains a candidate but is conflicting and still needs maintainer product/security review.

## What This Changes

The issue requests Discord inbound routing to infer the agent from a same-named Discord account when no explicit binding matches.

## Best Possible Solution

Maintainers should decide whether explicit bindings remain the supported contract or land a guarded Discord-only fallback with an agent-list guard, distinct diagnostics, docs, regression tests, and redacted real Discord proof.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and the resolver falls through to the configured default agent after explicit binding tiers. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. A guarded Discord-only accountId-to-agent fallback is plausible, but current docs support explicit bindings and AGENTS.md treats fallback behavior as a product/security decision.
