## Summary

Keep open: current main and the latest release still fall through to the default agent when no explicit Discord account binding matches, and https://github.com/openclaw/openclaw/pull/39485 is an open implementation PR with closing syntax for this issue. Because this affects permission-sensitive agent routing, it is not safe maintainer cleanup to close.

## What This Changes

Requests implicit Discord account-to-agent routing when a Discord account id matches an agent id and no explicit binding is configured.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or an equivalent maintainer-approved routing change with an existing-agent guard, distinct diagnostics, docs/tests, and redacted Discord proof; keep explicit bindings as the supported workaround until then.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the resolver reaches the default-agent return path. I did not run live Discord.

## Solution Assessment

Unclear. A same-name fallback is plausible, but it changes the documented explicit-binding contract and should only land with maintainer approval, an existing-agent guard, distinct match reporting, regression coverage, docs, and real Discord proof.
