## Label Justifications

- `P1`: Wrong-agent Discord routing can cross session, model, workspace, tool, and permission boundaries in real multi-account setups.
- `impact:security`: The reported fallback can send traffic intended for a restricted Discord account into a broader default agent.
- `impact:session-state`: Messages can attach to the default agent session instead of the intended same-named agent session.
- `impact:auth-provider`: The route can select the wrong agent and model context for a Discord account.

## Summary

Keep open: current main and v2026.5.22 still use explicit routing bindings and then default-agent fallback for unmatched Discord accounts, while the target-repo implementation PRs for same-name or account-level Discord routing are closed unmerged. The remaining work is a permission-sensitive routing contract decision rather than conservative cleanup.

## What This Changes

Request Discord inbound messages from an account to route to a same-named agent before falling back to the default agent when no explicit binding exists.

## Best Possible Solution

Choose one maintainer-approved Discord account routing contract, document it, guard it to configured runtime-eligible accounts and agents, add focused resolver and Discord route tests, and require redacted real Discord proof before landing.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and current main falls through to the default agent after explicit binding tiers. I did not run live Discord proof in this read-only review.

## Solution Assessment

Unclear. The requested same-name fallback would solve the reported setup, but it changes the documented routing contract; a safer final path may be account-level agent binding or a guarded implicit fallback after product and security review.
