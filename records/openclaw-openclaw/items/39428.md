## Summary

Keep open: current main and v2026.5.12 still use explicit binding tiers followed by the default-agent fallback, and the open closing PR still needs product/security review before this issue can be resolved.

## What This Changes

The issue requests Discord inbound routing to choose a same-named configured agent when the Discord account id matches the agent id instead of falling back to the default agent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/39485 or a replacement only after the implicit fallback contract is approved, guarded to existing configured agents, documented, tested, and proven with redacted Discord runtime evidence.

## Reproduction Assessment

Yes, at source level: Discord passes accountId into resolveAgentRoute, and with no matching binding the resolver reaches the default-agent return path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. Same-name Discord fallback may be useful, but it changes routing and security semantics and should land only with configured-agent guarding, distinct diagnostics, docs/tests, real Discord proof, and maintainer approval.
