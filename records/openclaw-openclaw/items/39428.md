## Summary

Keep open: current main and the latest release still route unmatched Discord account traffic through explicit bindings and then the default agent. The open linked fork PR remains the implementation candidate, but the implicit accountId-to-agent fallback still needs maintainer product and security review.

## What This Changes

The issue asks Discord inbound messages from an account named like an agent to route to that matching agent instead of falling back to the default agent.

## Best Possible Solution

Review, refresh, and either land or reject the linked Discord-only routing fallback with explicit security/product rationale; otherwise document that explicit account bindings are required.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord passes the named accountId into resolveAgentRoute, and current main returns the configured default agent when no explicit binding tier matches; I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear. The linked PR is a plausible narrow fix, but the best solution depends on whether maintainers want implicit account-name matching as supported routing behavior or want explicit bindings to remain the only contract.
