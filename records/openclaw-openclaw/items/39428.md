## Summary

Keep open: current main and v2026.6.6 still route unmatched Discord account traffic through explicit binding tiers and then the default agent. The open fork PR remains a candidate, but it is conflicting and the implicit accountId-to-agent fallback still needs maintainer product and security approval.

## What This Changes

The issue requests Discord inbound routing to select an agent whose id matches the Discord accountId when no explicit binding exists.

## Best Possible Solution

Review, refresh, and either land or reject a narrow Discord accountId-to-agent fallback with explicit security/product rationale; otherwise document that explicit bindings are required.

## Reproduction Assessment

Yes. Source inspection shows Discord passes the named accountId to resolveAgentRoute, and current main falls through to the default agent when no explicit binding tier matches; I did not run a live Discord setup in this read-only review.

## Solution Assessment

Unclear. A Discord-only accountId-to-agent fallback is a plausible narrow repair, but the best solution depends on whether maintainers want implicit name matching or explicit bindings as the only supported contract.
