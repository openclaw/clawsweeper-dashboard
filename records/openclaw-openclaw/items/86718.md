## Summary

Keep open: current main and the latest release still use an in-process JSON usage-cost cache and queued refresh path, while the issue discussion contains current-release OOM/outage evidence for the same usage/cost failure class.

## What This Changes

The issue requests that heavy `sessions.usage` and `usage.cost` processing stop starving the Gateway event loop and taking HTTP/WebSocket/channel paths offline under memory pressure.

## Best Possible Solution

Move or bound usage-cost cache storage and refresh work outside the Gateway hot path, then prove large-corpus `usage.cost`, `sessions.usage`, HTTP, health, and WebSocket responsiveness in a live setup.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.10` still full-parse the generated cache and run queued usage refresh in-process. I did not run a fresh live large-cache replay in this read-only review, but the issue comments provide current-release live OOM and outage evidence.

## Solution Assessment

Unclear as a final implementation choice: offloading or bounding the work is the right class of fix, but a TTL tweak alone would be too narrow. The best solution should settle the storage/isolation design while preserving the existing usage API contract.
