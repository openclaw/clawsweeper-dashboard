## Summary

Keep open: the report describes a fresh, high-impact Gateway availability failure that is only partially mitigated by current usage-cache performance work, and current main does not prove HTTP/WS health stays responsive during heavy sessions usage/cost refresh under memory pressure.

## What This Changes

The issue asks OpenClaw to prevent sessions usage/cost work from starving the Gateway event loop and taking HTTP, WebSocket, status, and channel workflows offline under memory pressure.

## Best Possible Solution

Keep this as a core Gateway stability bug until usage/cost refresh is bounded, cancellable, degraded, or isolated enough that health, HTTP, and WebSocket handshakes remain responsive under the reported session and memory load.

No high-confidence current-main reproduction was established. The report has detailed live evidence and the current usage/cost code path is source-traceable, but there is no deterministic load fixture proving the outage on current main after the latest perf commits.
