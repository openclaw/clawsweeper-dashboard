## Summary

Keep open: current main still keeps the usage/cost cache as large JSON in the Gateway process, schedules queued refreshes in-process, and the latest 2026.6.8 comment adds current-release hard-OOM evidence for the same usage/cost cluster.

## What This Changes

The issue reports Gateway HTTP/WS and channel outages when `sessions.usage` / `usage.cost` work runs under large session-cache and memory pressure.

## Best Possible Solution

Keep this as the canonical usage/cost Gateway starvation tracker until maintainers land and prove off-main-process or bounded incremental aggregation with degraded fast paths that keep HTTP, WebSocket, status, and channels responsive.

## Reproduction Assessment

Yes for the issue class, but not as a local current-main replay: the 2026.6.8 comment provides current-release OOM evidence with a 216 MB cache, and current main still parses, sorts, and refreshes that usage/cost cache in the Gateway process.

## Solution Assessment

No; the current background request-path mitigation is only partial. The maintainable fix likely needs a maintainer-owned choice between worker/process isolation, SQLite or incremental cache storage, bounded chunking, and degraded priority fast paths.
