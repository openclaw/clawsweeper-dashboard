## Label Justifications

- `P1`: The issue describes an urgent Gateway availability failure affecting dashboard HTTP/WebSocket, status probes, and live channels.
- `impact:message-loss`: The reported Gateway stalls produce WebSocket 1006 closes and Telegram/WhatsApp timeout or reconnect behavior.
- `impact:crash-loop`: Related current-release evidence shows hard V8 OOM aborts and restart loops from oversized usage-cost cache parsing.

## Summary

Keep open: current main still stores usage/cost data in a whole-file JSON cache, parses and stringifies it in the Gateway process, and runs queued refresh work in-process; the current-release OOM and stall reports therefore remain actionable rather than implemented.

## What This Changes

The issue asks OpenClaw to keep Gateway HTTP, WebSocket, status, and channel paths responsive when sessions usage/cost processing runs under large session corpora and memory pressure.

## Best Possible Solution

Keep this as the canonical usage/cost Gateway availability tracker until maintainers land and prove a bounded storage/refresh design that keeps health, HTTP, WebSocket, status, and channel paths responsive under large usage-cost datasets.

## Reproduction Assessment

Yes for the issue class, from source and live report evidence: current main still parses, summarizes, sorts, and refreshes large usage-cost cache data in the Gateway process. I did not run a local large-profile replay, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No; the current background refresh mode is only a partial mitigation for the older synchronous request-path rebuild. The remaining best fix needs a maintainer-approved storage or workload-isolation design plus representative live Gateway proof.
