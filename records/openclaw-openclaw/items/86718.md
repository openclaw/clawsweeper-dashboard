## Summary

Keep open: current main still has the usage/cost cache and refresh path in the Gateway process, and the newer current-release OOM comments show the same central failure class has not been retired by a merged fix.

## What This Changes

This issue reports Gateway HTTP/WebSocket unavailability, channel degradation, and hard-OOM variants caused by usage/cost cache work under memory pressure.

## Best Possible Solution

Move usage-cost state and refresh work out of the Gateway hot path with a bounded canonical cache design, then prove large-corpus `usage.cost`, `sessions.usage`, health, HTTP, and WebSocket responsiveness in a live setup.

## Reproduction Assessment

Yes at source level, but not from a fresh local live run: current source still full-parses `.usage-cost-cache.json` and runs queued usage-cache refresh in-process. Recent v2026.6.8 comments add live OOM and outage evidence for the same usage/cost cluster.

## Solution Assessment

Unclear as a final implementation choice: another request-path background flag or TTL tweak is too narrow. The best fix should address cache storage bounds and refresh isolation while preserving the usage API contract.
