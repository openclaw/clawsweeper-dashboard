## Summary

Keep open. The current-main merge shape is now focused on the ClawHub verify-output files, but the trusted URL helper still accepts branch-like commit values such as `main`, so the new `openclaw.verifiedSourceUrl` can be advertised as commit-pinned when it is not; the contributor proof is also mock-only for the external PR gate.

## What This Changes

The branch adds `openclaw.verifiedSourceUrl` to `openclaw skills verify` JSON from server-resolved ClawHub provenance, reuses that helper for install persistence, updates focused tests/docs, and includes generated Codex prompt snapshot refreshes that are already present on current main.

## Best Possible Solution

Promote `verifiedSourceUrl` only when ClawHub provenance carries a full resolved commit SHA, add negative coverage for branch/tag-like commit values, and refresh the PR with real CLI proof or explicit maintainer acceptance of the no-live-sample limitation.

## Reproduction Assessment

Not applicable as a bug reproduction: this PR adds an additive CLI JSON output field. The behavior is source-reviewable and mock-tested, but the positive path has not been exercised against a live ClawHub provenance response.

## Solution Assessment

No. Rebuilding the URL from server-resolved `repo` + `commit` + `path` is the right boundary, but the implementation must validate that `commit` is actually a resolved commit SHA before advertising the result as commit-pinned.
