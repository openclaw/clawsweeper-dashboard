## Summary

Keep open: current main documents and hardens the local ACP/acpx Discord binding path, but the requested Raspberry Pi Gateway plus remote Mac harness topology still lacks a node/gateway selector and needs maintainer product and security direction.

## What This Changes

Requests official support or documentation for Discord ACP sessions where a Raspberry Pi Gateway triggers Claude/Codex harness execution on a remote Mac node.

## Best Possible Solution

Decide the supported topology explicitly: either document ACP/acpx harnesses as Gateway-host-only, or design a typed remote ACP backend/node selector with security model, Discord binding docs, and regression coverage.

## Reproduction Assessment

Yes for the remaining limitation: source inspection shows ACP binding/runtime inputs accept only harness/backend/mode/cwd-style local options, and docs require the harness command on the Gateway host. I did not live-run the full Raspberry Pi, Discord, and Mac topology.

## Solution Assessment

No single implementation is clearly best yet. The maintainable path needs a maintainer choice between a docs-only Gateway-host stance and a new remote ACP or node-routing contract.
