## Summary

Keep open. Current main clarifies that ACP/acpx harnesses run on the Gateway host and mitigates one stale-binding error path, but it still has no first-class way to place Claude/Codex ACP harness execution on a separate Mac node from a Discord workspace flow.

## What This Changes

The issue asks for official support or clear guidance for Discord ACP sessions where the Gateway runs on one host and the external coding harness runs on a remote Mac node.

## Best Possible Solution

Keep this as a maintainer product/design issue whose end state is either explicit Gateway-host-only documentation for Discord ACP, or a first-class remote ACP harness placement design with typed node selection, credential handling, approvals, audit logging, docs, and regression coverage.

## Reproduction Assessment

Unclear for the full live failure because the Pi/Mac/Discord topology was not run end to end. The remaining product gap is source-reproducible: current docs, strict schemas, spawn params, runtime inputs, and manager handoff expose no node or host selector for ACP harness placement.

## Solution Assessment

Unclear. A docs-only clarification is the narrowest safe improvement for current behavior, while actual remote Mac ACP harness execution needs product and security design before code changes.
