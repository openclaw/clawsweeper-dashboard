## Summary

Keep open: current main clarifies and hardens the local ACP/acpx path, but the requested remote Mac harness topology still needs a maintainer product decision and likely overlaps adjacent cross-gateway work rather than being fully implemented.

## What This Changes

Request support or clearer official guidance for Discord-triggered ACP harness sessions where the Gateway runs on one host and Claude/Codex runs on a remote Mac node.

## Best Possible Solution

Decide the supported topology explicitly: either document ACP/acpx harnesses as Gateway-host-only and route remote compute through cross-gateway sessions, or design a remote ACP backend/node selector with Discord binding docs.

## Reproduction Assessment

Yes for the remaining limitation: current source shows ACP/acpx has no remote node selector and docs require the harness command on the Gateway host. I did not live-run the full Pi, Discord, and Mac topology.

## Solution Assessment

No, current main is not the full solution for remote Mac harness execution. The narrow maintainable path needs maintainer direction between a docs-only host-local stance and extending ACP/runtime or cross-gateway routing.
