## Summary

Keep open. Current main still treats ACP harness execution as Gateway-host-local and has no ACP node/host selector, while the requested Pi Gateway plus remote Mac harness topology would require a product/API decision rather than conservative cleanup.

## What This Changes

Issue asks OpenClaw to clarify or support an ACP/Discord topology where the Gateway runs on a Raspberry Pi while Codex or Claude harnesses execute on a remote Mac node.

## Best Possible Solution

Maintainers should decide the support boundary: either design a node-aware ACP runtime binding/config surface, or explicitly document that ACP harnesses run on the Gateway host and point remote Mac workflows to supported alternatives.

## Reproduction Assessment

No high-confidence end-to-end live reproduction is available. Source inspection does show the non-ACP session error path and the absence of any ACP node/host selector, but the reported `acpx exited with code 1` turn failure was not reproduced live.

## Solution Assessment

No. Current main partially clarifies binding behavior and stale ACP metadata handling, but it does not implement the requested remote harness topology; the narrow maintainable path needs a product decision before code work.
