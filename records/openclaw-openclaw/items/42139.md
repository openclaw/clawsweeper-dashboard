## Summary

Keep open. Current main documents ACP/acpx as a Gateway-host external harness path and has clearer Discord ACP binding behavior, but it still does not implement first-class placement of Claude/Codex ACP harness execution on a separate Mac node. That remaining capability crosses product and security boundaries, so this is not conservative cleanup.

## What This Changes

The issue asks whether Discord ACP spawn can run Claude/Codex on a separate Mac node while the Gateway runs elsewhere, and requests official configuration guidance or documentation/feature tracking if not.

## Best Possible Solution

Keep this as a maintainer product/design issue. The end state should either explicitly document the current Gateway-host-only ACP/acpx model in the Discord ACP path, or add a first-class remote ACP harness placement design with typed config, node selection, credential handling, approvals, audit logging, Discord binding semantics, schema docs, and regression coverage.

## Reproduction Assessment

Unclear for the full live failure because the Raspberry Pi, Mac mini, Discord, and acpx topology was not run end to end. There is a high-confidence static reproduction of the remaining product gap: current docs, strict schemas, spawn params, runtime inputs, and manager handoff expose no node or host selector for ACP harness placement.

## Solution Assessment

Unclear. A docs-only clarification is the narrowest safe improvement for current behavior, while actual remote Mac ACP harness execution needs product and security design before code changes.
