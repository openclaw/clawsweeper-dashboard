## Summary

Keep open: current main clarifies and hardens the local ACP/acpx Discord binding path, but the requested Raspberry Pi Gateway plus remote Mac harness topology still lacks a node/gateway selector and needs maintainer product direction.

## What This Changes

The issue asks whether Discord `/acp spawn` can use a Raspberry Pi Gateway while running Claude/Codex ACP harnesses on a remote Mac mini, and requests either official configuration docs or feature tracking.

## Best Possible Solution

Decide the supported topology explicitly: either document ACP/acpx harnesses as Gateway-host-only and point remote compute to nodes/cross-gateway flows, or design a remote ACP backend/node selector with Discord binding docs.

## Reproduction Assessment

Yes for the remaining limitation: current source and docs show ACP/acpx has no remote node selector and expects the harness command on the Gateway host. I did not live-run the full Pi, Discord, and Mac topology.

## Solution Assessment

No, current main is not the full solution for remote Mac harness execution. The maintainable path needs a maintainer choice between a docs-only Gateway-host stance and a new remote ACP or cross-gateway routing contract.
