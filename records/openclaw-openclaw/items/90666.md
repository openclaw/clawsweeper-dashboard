Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep timeout cleanup armed after cancel:** `src/cron/active-jobs.ts:136`
  - body: `cancelCronJobRun` aborts the same controller that the timeout watchdog observes. In `executeJobCoreWithTimeout`, the watchdog callback returns immediately once that signal is already aborted, and it is the only path that resolves `timeoutPromise` and calls `cleanupTimedOutCronAgentRun`. If the underlying cron runner stalls or ignores the operator abort, cancelling before the configured timeout now prevents the existing timeout cleanup from firing, leaving the cron execution stuck instead of cleaning it up. Please add a separate cancellation/cleanup race or otherwise keep timeout cleanup active after operator abort, with a regression that cancels a timeout-enabled run whose runner does not resolve on abort.
