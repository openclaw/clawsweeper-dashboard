## Summary

Keep open: the PR fixes the central isolated cron cancellation path, but it is not merge-ready because the same active-handle path also makes main-session cron task rows look cancellable even though that work is already enqueued into a session run the cron frame does not own.

## What This Changes

The PR adds live abort handles for active cron runs, routes cron task cancellation through Gateway first, and updates task registry, command, and cron timer tests for cron cancellation.

## Best Possible Solution

Land one cron cancellation path that only marks task rows cancelled when the live process owns the underlying run, while keeping main-session cron rows explicitly non-cancellable unless their downstream session work is also cancelled and proven.

## Reproduction Assessment

Yes for source-level review: current main returns the unsupported-runtime path for cron task rows, and the PR-introduced main-session edge is traceable from task row creation through executeMainSessionCronJob. I did not run tests because this is a read-only review.

## Solution Assessment

No: the isolated cron cancellation approach is plausible, but registering all cron core executions is not the best fix because main-session cron work has a different owner boundary. The safer repair is to limit cancellation to live isolated cron task runs or explicitly implement and prove downstream session cancellation.
