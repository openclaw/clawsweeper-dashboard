## Summary

Keep open: current main still routes gateway exec through child-process supervision and has no merged recovery for the reported low-threshold macOS `spawn EBADF` mode. The related open high-FD reports and https://github.com/openclaw/openclaw/pull/77751 are useful context, but they target service/high-file-descriptor behavior rather than this report's claimed 5-10 spawn, non-FD-exhaustion failure.

## What This Changes

The issue reports macOS gateway exec failures with `spawn EBADF` after a small number of repeated spawn calls, suspected to involve libuv/kqueue child-process monitoring state.

## Best Possible Solution

Prove the failure on macOS with repeated gateway exec calls, then either add a narrow supervisor/spawn recovery with regression coverage or document a runtime support boundary if the failure is purely upstream.

## Reproduction Assessment

No high-confidence live reproduction was established. The report has concrete macOS steps and current source still exercises repeated gateway child spawns, but the specific 5-10-call EBADF threshold was not reproduced on current main.

## Solution Assessment

Unclear until the macOS failure is reproduced. Periodic gateway restarts should remain an operational workaround; the maintainable fix is a narrow spawn/supervisor mitigation only if current-main proof shows OpenClaw can recover cleanly.
