## Summary

Keep open: current main still routes local gateway exec through child process spawns, and the existing EBADF handling only retries through supplied fallback options rather than addressing the reported macOS repeated-spawn failure. The open related fix at https://github.com/openclaw/openclaw/pull/77751 targets the high-file-descriptor service case in https://github.com/openclaw/openclaw/issues/77750, not this report's claimed low-threshold, no-FD-exhaustion failure mode.

## What This Changes

The issue reports that repeated macOS gateway `exec` calls start failing with `spawn EBADF` after roughly 5-10 successful spawns and recover only after a Gateway restart.

## Best Possible Solution

Prove the failure on macOS with a repeated gateway exec loop, then add the narrowest supervisor/spawn recovery or document a runtime support boundary if the issue is purely upstream.

## Reproduction Assessment

No, not at high confidence: the report has concrete steps and current source still exercises repeated child spawns, but no live macOS/Node 25 loop was established in this review. The next proof should run 20+ simple `exec host=gateway` calls against current main on macOS with sandbox off.

## Solution Assessment

Unclear until the macOS failure is reproduced. A narrow supervisor/spawn recovery is likely the maintainable direction, while periodic Gateway restarts should remain an operational workaround rather than product behavior.
