## Summary

Keep open. Current main already carries cron job identifiers and running markers through the model, Gateway protocol, UI types, runtime, tests, and CLI list formatting, but the Control UI job cards and agent cron panel still do not display each job ID and still derive visible status from lastStatus instead of state.runningAtMs.

## What This Changes

Request to make each scheduled cron job show its stable identifier and whether it is currently running.

## Best Possible Solution

Render each cron job's stable ID and an active running status in the main Control UI cron list and the agent-specific cron panel, using existing state.runningAtMs data and adding focused English i18n plus UI tests for running, idle, disabled, and identifier display.

## Reproduction Assessment

Yes for the visible gap. A static render path using a CronJob with id and state.runningAtMs would still show status from state.lastStatus and never render job.id in the Control UI card or agent cron panel.

## Solution Assessment

Yes, a focused UI presentation fix is the best path. The backend, protocol, runtime, and CLI already expose the required data, so a new core cron API or schema change would be unnecessary unless implementation discovers a missing typed UI adapter.
