## Summary

Keep open. Current main still carries cron job IDs and running-state data through the contract, runtime, tests, CLI, and docs, but the Control UI cron list and agent cron panel do not visibly render the stable job ID or treat `state.runningAtMs` as the active status.

## What This Changes

The issue asks for each timed cron task to show its identifier and whether it is currently running.

## Best Possible Solution

Render each cron job's stable ID and active running status in the main Control UI cron list and the agent-specific cron panel using existing `CronJob.id` and `state.runningAtMs` data.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `CronJob.id` and `state.runningAtMs` reach the UI renderers, while the relevant render and formatter functions omit them from visible output.

## Solution Assessment

Yes. A focused Control UI presentation change is the narrowest maintainable solution because the backend contract, runtime state, service tests, CLI formatter, and docs already support the requested fields.
