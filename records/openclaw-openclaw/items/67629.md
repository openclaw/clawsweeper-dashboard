## Summary

Keep open. Current main already carries cron job IDs and `state.runningAtMs` through the model, protocol, runtime, tests, and CLI, but the Control UI cron list and agent cron panel still do not visibly render the stable job ID or active running state.

## What This Changes

Request that each timed cron task visibly show whether it is running and expose a stable identifier.

## Best Possible Solution

Render each cron job's stable ID and active running status in the main Control UI cron list and the agent-specific cron panel using existing `state.runningAtMs` data, with English i18n, focused UI tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows a `CronJob` with `id` and `state.runningAtMs` would flow into Control UI renderers that display `lastStatus`/next/last metadata and never render the stable job ID or running marker.

## Solution Assessment

Yes. A focused Control UI presentation change is the narrowest maintainable path because backend, protocol, runtime, and CLI support already exist.
