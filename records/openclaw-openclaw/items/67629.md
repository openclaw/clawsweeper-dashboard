## Summary

Keep open. Current main and the latest release still expose cron job IDs and running markers through the backend, protocol/UI types, docs, and CLI, but the Control UI cron list and agent cron panel do not visibly render the stable job ID or a running indicator.

## What This Changes

The issue asks for each timed cron task to visibly show whether it is running and to include a stable identifier.

## Best Possible Solution

Add a narrow Control UI display update that shows job.id and a running indicator derived from state.runningAtMs, or currentStatus if the related read-model PR lands first, in both the cron list and agent cron panel.

## Reproduction Assessment

Yes, source-level. Render a CronJob with id and state.runningAtMs: current main carries both fields to UI types, but the current UI renderers display name/schedule/last status/next/last data instead of the stable ID or running state.

## Solution Assessment

Yes. A narrow Control UI presentation change is the best fix because the core model, gateway schema, and CLI already expose the requested facts; backend protocol changes should be avoided unless the related read-model PR lands first.
