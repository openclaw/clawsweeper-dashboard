## Summary

Keep open. Current main and the latest release still carry cron job IDs and running-state data through the typed cron contract, Gateway projection, and CLI, but the Control UI cron list and agent cron panel do not visibly render stable IDs or derive active status from `state.runningAtMs`.

## What This Changes

The issue asks for each timed cron task to show whether it is running and to expose a stable identifier.

## Best Possible Solution

Render `job.id` and a running status derived from `state.runningAtMs` in the Control UI cron list and agent cron panel, with focused UI tests and no backend or CLI contract changes.

Yes, as source-reproducible current-main behavior. The typed cron data path exposes `CronJob.id` and `state.runningAtMs`, while the relevant UI renderers omit the stable ID and use `lastStatus` instead of the active running marker.

## Solution Assessment

Yes. A focused Control UI presentation change is the narrowest maintainable solution because the backend, Gateway projection, and CLI behavior already expose the needed data.
