## Summary

Keep open. Current `main` already carries cron job IDs and `state.runningAtMs` through the model, Gateway protocol, runtime, tests, and CLI list formatting, but the documented Control UI cron list and agent cron panel still do not visibly render the stable job ID or active running state.

## What This Changes

The issue requests visible running state and identifiers for each timed cron task.

## Best Possible Solution

Render each cron job's stable ID and active running status in the main Control UI cron list and the agent-specific cron panel using existing `state.runningAtMs` data, with focused English i18n and UI tests.

## Reproduction Assessment

Yes. A static render path with a `CronJob` containing `id` and `state.runningAtMs` still flows through UI code that displays `lastStatus`/next/last metadata and never renders the job ID in either Control UI cron surface.

## Solution Assessment

Yes. A focused Control UI presentation fix is the narrowest maintainable solution because backend, protocol, runtime, and CLI support already exist.
