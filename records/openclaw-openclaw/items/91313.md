## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Omit null clears when replacing payload kinds:** `packages/gateway-protocol/src/schema/cron.ts:256-257`
  - body: Adding `null` to the agentTurn patch schema also accepts updates like `{ payload: { kind: "agentTurn", message: "...", model: null } }` when the existing job is not an agentTurn. That path goes through `buildPayloadFromPatch`, which copies `patch.model` and `patch.fallbacks` into the new persisted payload, so the job can store `null` instead of omitting the optional fields. Please either reject null clears on kind replacement or make the replacement builder drop them, with service/Gateway coverage.
  - confidence: 0.93
- **[P2] Align CronPayloadPatch with the new null-clear contract:** `src/cron/service/jobs.ts:926-931`
  - body: These new branches rely on `patch.model` and `patch.fallbacks` being `null`, but `CronAgentTurnPayloadPatch` still derives those fields from `model?: string` and `fallbacks?: string[]`; only `toolsAllow` is nullable. That leaves typed cron callers and the service contract out of sync with the Gateway schema, and may fail the TypeScript build around the null comparisons. Add explicit nullable model/fallbacks patch fields in `src/cron/types.ts`.
  - confidence: 0.88
