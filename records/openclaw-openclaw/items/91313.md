## Summary

Keep open: the PR fixes much of the cron null-clear path, but it still advertises a model-only null clear through the tool schema without making the tool infer an `agentTurn` payload for that shape, and the supplied proof is unit-test output rather than a real cron update.

## What This Changes

The PR changes cron update payload schemas, normalization, service merge logic, and tests so `payload.model: null` and `payload.fallbacks: null` can clear stored overrides.

## Best Possible Solution

Align the tool schema, canonicalization, protocol validation, normalization, and service merge so nullable cron payload clears work for the advertised model-facing shape, then require redacted real Gateway or CLI proof of a stored override being cleared.

## Reproduction Assessment

Yes from source: current main drops `payload.model: null` and `payload.fallbacks: null` during normalization and then preserves existing stored overrides during merge. The remaining PR bug is also source-reproducible because the tool schema accepts model-only null clears while the unchanged canonicalizer does not infer `agentTurn` for `model === null`.

## Solution Assessment

No, not yet: the implementation is in the right ownership areas, but it leaves the advertised model-only null-clear shape incomplete and lacks real behavior proof. The best fix is to add model-null kind inference plus focused tool/Gateway coverage before merge.
