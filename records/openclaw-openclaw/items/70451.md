## Summary

Keep open. Current main still routes hooks CLI report generation through full plugin diagnostics and runtime module loading; the bootstrap policy avoids preloading plugins for the hooks command, but the command body still enters the runtime-loading path that matches the reported Windows timeout class.

## What This Changes

The issue reports that Windows users on OpenClaw 2026.4.21 and Node 24 see `openclaw hooks enable/list/status` hang for about 60 seconds before SIGKILL, and asks for hook registration alternatives or a fix ETA.

## Best Possible Solution

Make `openclaw hooks list/info/check/enable/disable` build its report from workspace hooks plus manifest or metadata snapshot data for plugin-managed hooks. The command should preserve plugin-managed hook display and toggle restrictions while avoiding full plugin runtime imports on the CLI control-plane path.

## Reproduction Assessment

Unclear for the live SIGKILL because this review did not run Windows/Node 24. The source-level path is reproducible: the affected hooks commands call `buildHooksReport`, which calls full plugin diagnostics and reaches `loadOpenClawPlugins` with module loading enabled.

## Solution Assessment

Yes, the narrow maintainable solution is to move hooks CLI reporting to snapshot or manifest-backed hook metadata. A broad Windows plugin-loader speedup alone is not the best terminal fix because hook listing and toggling should not depend on importing plugin runtimes.
