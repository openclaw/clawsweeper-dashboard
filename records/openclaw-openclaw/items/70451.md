## Summary

Keep open. Current main and the latest release still route hooks list/info/check/enable/disable through plugin diagnostics and runtime plugin registration, and the active repair candidate has not merged or fully proved the lingering-handle path.

## What This Changes

The issue reports Windows `openclaw hooks enable/list` timing out because hooks CLI reporting can enter plugin diagnostic/runtime loading.

## Best Possible Solution

Land or replace the narrow hooks one-shot exit fix, then use Windows or temporary hanging-plugin proof to decide whether hooks reporting also needs a metadata/snapshot-only plugin hook path.

## Reproduction Assessment

Yes by source inspection: current main hooks commands load plugin diagnostics, plugin diagnostics can execute plugin `register()` code, and successful hooks actions do not force process exit. I did not run a live Windows/Node 24 reproduction.

## Solution Assessment

Unclear until the active PR has stronger proof: the one-shot exit approach is narrow for lingering handles, but a metadata-only hooks report would be safer if the timeout happens before output.
