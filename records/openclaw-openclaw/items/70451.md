## Summary

Keep open. Current main still routes the hooks CLI report through full plugin diagnostics, so the Windows timeout remains a plausible source-backed bug rather than an implemented or duplicate cleanup candidate.

## What This Changes

The issue reports Windows/Node 24 timeouts and SIGKILLs when running `openclaw hooks enable/list/status`, with questions about managing hook registration while the plugin registry hang is addressed.

## Best Possible Solution

Move `openclaw hooks list/info/check/enable/disable` to a metadata-first hook discovery path that avoids full plugin runtime imports while still showing plugin-managed hook entries and preserving toggle restrictions.

## Reproduction Assessment

Unclear for the live SIGKILL because no Windows/Node 24 run was performed. The source-level reproduction path is clear: the affected hooks commands all reach `buildHooksReport`, which still invokes full plugin diagnostics with module loading enabled.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the hooks CLI on manifest/config/workspace hook metadata instead of broad plugin runtime diagnostics; a general Windows plugin-loader speedup would not remove the unnecessary CLI control-plane dependency.

## Repair Work Prompt

Fix the Windows hooks CLI timeout tracked by https://github.com/openclaw/openclaw/issues/70451. Current `src/cli/hooks-cli.ts` builds hook reports by calling `buildPluginDiagnosticsReport`, which loads plugin runtime modules. Change `openclaw hooks list/info/check/enable/disable` so hook discovery and toggle validation use workspace/managed/bundled/plugin hook metadata or a bounded metadata snapshot helper, not full plugin diagnostics/runtime imports. Preserve plugin-managed hook display (`plugin:<id>`) and keep plugin-managed hooks non-toggleable through the hooks CLI. Do not broaden command bootstrap plugin loading, do not add core plugin IDs, and do not change dependency patches or installer behavior. Add focused tests that would fail if the hooks CLI report path calls diagnostics/runtime module loading for normal list/check/enable/disable paths. Add a user-facing CHANGELOG entry crediting the reporter if the fix changes CLI behavior.
