## Summary

Keep open. Current main and the latest release still route hooks commands through plugin diagnostics that can execute plugin `register()` code, and successful hooks actions still return without a forced one-shot exit; the linked repair PR is open and conflicting, so this is not implemented yet.

## What This Changes

The issue reports Windows `openclaw hooks` commands timing out or being SIGKILLed while CLI plugin/bootstrap code runs.

## Best Possible Solution

Land or replace the narrow hooks one-shot exit fix, then use Windows or temporary hanging-plugin proof to decide whether hooks reporting also needs a metadata-only plugin-hook path.

## Reproduction Assessment

Partly. Source inspection gives a credible lingering-handle reproduction path: hooks commands load plugin diagnostics, diagnostics can execute plugin register code, and the success path does not force exit; I did not run the exact Windows/Node 24 SIGKILL scenario.

## Solution Assessment

Unclear as a final fix. The linked one-shot exit approach is the narrowest fix for post-output lingering handles, but a metadata-only hooks reporting path may be safer if current Windows hangs happen before output.
