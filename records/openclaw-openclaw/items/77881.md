## Summary

Keep open: current main still returns an empty phase signal store for non-ENOENT read failures, and the next phase-signal write can replace the existing history with only the current cycle's data.

## What This Changes

The issue asks memory-core dreaming to recover only missing or corrupt phase-signal files while propagating other I/O read failures so accumulated phase-signal history is not overwritten.

## Best Possible Solution

Rethrow non-ENOENT, non-SyntaxError phase-signal read errors so the existing dreaming workspace failure path logs the failure and skips the destructive write; cover it with a focused regression test.

## Reproduction Assessment

Yes, from source inspection: make `fs.readFile` for `memory/.dreams/phase-signals.json` throw `EACCES` while valid recall and phase-signal data exist, then call `recordDreamingPhaseSignals`. Current main will convert that read failure into an empty phase-signal store and write it back; I did not execute this because the sweep is read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is to keep first-run `ENOENT` and corrupt-JSON recovery but rethrow other I/O errors. That uses the existing workspace-level dreaming catch instead of adding a new config option or policy path.
