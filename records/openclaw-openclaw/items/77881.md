## Summary

The issue is valid on current main: the phase-signal reader still converts every non-ENOENT read failure into an empty store, and the update path can then persist that empty-derived store over accumulated dreaming signals.

## What This Changes

The issue asks memory-core to preserve accumulated dreaming phase-signal history by not treating non-ENOENT `phase-signals.json` read failures as an empty store.

## Best Possible Solution

Have memory-core treat operational phase-signal store read errors as fatal for the current update/rank cycle while keeping first-run missing-file and corrupt-JSON recovery behavior.

## Reproduction Assessment

Yes, from source inspection: induce or mock EACCES on `memory/.dreams/phase-signals.json`, call `recordDreamingPhaseSignals`, and the current code will read an empty fallback and write it back. I did not execute that filesystem-error scenario in this read-only review.

## Solution Assessment

Yes; the narrow maintainable fix is to rethrow non-ENOENT, non-`SyntaxError` read errors from `readPhaseSignalStore` so the existing dreaming workspace catch path logs the failure and avoids writing a replacement store.
