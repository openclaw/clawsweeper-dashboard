## Summary

Keep open. Current main still catches every `readPhaseSignalStore` failure as an empty phase-signal store, the latest shipped release still has the same behavior, and multiple open fix PRs already reference this issue with closing syntax, so the issue should stay open until one fix lands.

## What This Changes

The issue requests that memory-core preserve accumulated dreaming phase-signal data by rethrowing operational read errors instead of treating them as an empty store.

## Best Possible Solution

Land one focused memory-core fix that preserves ENOENT and corrupt-JSON recovery while rethrowing operational phase-signal read errors, with regression coverage, then resolve the duplicate open PRs.

## Reproduction Assessment

Yes, from source inspection: make `memory/.dreams/phase-signals.json` fail with a non-ENOENT, non-SyntaxError read error during `recordDreamingPhaseSignals`, and current main will continue from an empty fallback and write the derived store back. I did not run that filesystem-error scenario because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to rethrow non-recoverable phase-signal store read errors so the existing dreaming workspace catch path records the failure and avoids replacing the stored signals.
