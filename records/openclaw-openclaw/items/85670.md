## Summary

Keep open: the PR still contains a blocking ordering regression because it runs legacy OAuth sidecar migration and stale OAuth shadow cleanup concurrently even though the latter intentionally depends on the former’s persisted auth-store rewrite. It also still lacks after-fix real behavior proof for the claimed doctor timing improvement.

## What This Changes

The PR changes `runDoctorRepairSequence` so plugin dependency cleanup, legacy OAuth sidecar repair, and stale OAuth shadow cleanup run together in one `Promise.all`.

## Best Possible Solution

Keep dependent OAuth repairs serial, parallelize only repair steps that do not read or write the same auth profile stores, add focused sequencing coverage, and require redacted patched-run doctor timing proof before merge.

## Reproduction Assessment

Yes for the PR-introduced defect from source inspection: stale-shadow cleanup skips legacy sidecar refs, and sidecar migration removes those refs. I did not live-reproduce the original VPS doctor timing regression.

## Solution Assessment

No; this is not the best merge shape because it parallelizes auth repairs that share persisted auth profile state. The safer fix is to keep the auth dependency ordered and parallelize only independent cleanup work.
