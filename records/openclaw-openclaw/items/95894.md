## Label Justifications

- `P2`: The issue describes a real plugin startup crash path with limited blast radius to Express 4.x plugins installed through managed npm roots.
- `impact:crash-loop`: Affected plugins can fail their HTTP route startup path when path-to-regexp 8.x is forced into an Express 4 dependency graph.
- `maturity:stable`: The affected CLI plugin install-source feature belongs to the M4 Stable CLI surface, even though the broader plugin architecture surface is still M3 Beta.

## Summary

Keep open: current main still copies the host workspace override set, including path-to-regexp 8.4.0, into managed npm plugin roots, while the reported plugin path depends on Express 4.x and Express 4.21.2 declares path-to-regexp 0.1.12. The merged managed-peer-pin fix for a related EOVERRIDE issue does not remove or scope the path-to-regexp override for third-party Express 4 plugins.

## What This Changes

The issue asks OpenClaw to stop managed plugin installs from forcing Express 4.x plugins onto an incompatible path-to-regexp 8.x dependency graph.

## Best Possible Solution

Define a maintainer-approved managed-npm-root override policy that preserves plugin-owned Express 4 dependency graphs without globally downgrading OpenClaw's Express 5/router path, with focused install/update tests and dependency-resolution docs.

## Reproduction Assessment

Yes. Source inspection shows managed npm installs still inherit the host path-to-regexp 8.4.0 override, while Express 4.21.2 declares path-to-regexp 0.1.12; I did not run a live plugin install in this read-only cleanup review.

## Solution Assessment

No for the issue's blanket downgrade option: OpenClaw's current Express 5/router stack depends on path-to-regexp 8.x. The best fix is a scoped managed-root policy or maintainer-approved validator path that preserves plugin dependency ownership without breaking core.
