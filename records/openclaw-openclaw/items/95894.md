## Label Justifications

- `P2`: The issue reports a real managed npm plugin startup crash path with limited blast radius to Express 4.x plugins installed through OpenClaw-managed roots.
- `impact:crash-loop`: Affected Express 4.x plugins can fail their HTTP route startup path when path-to-regexp 8.x is forced into their dependency graph.

## Summary

Keep open: current main and v2026.6.10 still propagate host workspace overrides, including path-to-regexp 8.4.0, into managed npm plugin roots. Express 4.x requires path-to-regexp 0.1.x while OpenClaw's current Express 5/router stack needs 8.x, so the remaining work is a maintainer policy decision for scoped managed-root override behavior or validation, not a safe cleanup close.

## What This Changes

The issue asks OpenClaw to stop managed plugin installs from forcing host path-to-regexp 8.x into Express 4.x plugin dependency graphs, either through central dependency policy or install-time validation.

## Best Possible Solution

Define a narrow managed npm-root dependency policy that preserves plugin-owned dependency graphs without globally downgrading OpenClaw's Express 5 path, and add docs plus regression coverage for install/update behavior.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current installs copy host workspace overrides into managed npm roots, the host override pins path-to-regexp 8.4.0, and Express 4.21.2 metadata requires path-to-regexp 0.1.12. I did not run a live plugin install in this read-only review.

## Solution Assessment

No, the issue's preferred blanket downgrade is not the best fix because OpenClaw's Express 5/router stack depends on path-to-regexp 8.x. The safer solution is a maintainer-approved scoped override policy or validator that preserves plugin dependency ownership without breaking core.
