## Label Justifications

- `P2`: This is a normal-priority plugin-install/runtime compatibility bug with a clear affected dependency path but limited blast radius to Express 4-based npm plugins.
- `impact:crash-loop`: The report is about Express 4 plugin HTTP routes throwing during startup/load so the plugin server cannot serve its routes.

## Summary

Keep open: current main still has a managed npm plugin-install path that reuses host pnpm overrides, including `path-to-regexp: 8.4.0`, while Express 4.x packages require the 0.1.x line and the reported example plugin depends on Express 4 without its own override. The prior Microsoft Teams path-to-regexp fixes are useful adjacent context, but they fixed a bundled Teams/Express 5 route-pattern problem rather than this external plugin dependency-resolution problem.

## What This Changes

The issue requests a core plugin-install fix so Express 4.x-based npm plugins do not have to carry their own `path-to-regexp` override to avoid route startup crashes.

## Best Possible Solution

Define a scoped managed-plugin dependency policy that preserves plugin-owned Express 4 dependency resolution while keeping OpenClaw security overrides explicit and tested, with a regression fixture for an Express 4 plugin package.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live-run no: current install code applies host overrides to managed npm plugin roots, and npm metadata shows the reported plugin depends on Express 4 while Express 4 requires `path-to-regexp` 0.1.x.

## Solution Assessment

Is this the best way to solve the issue? Unclear: a blanket root override to `path-to-regexp` 0.1.x would conflict with OpenClaw's Express 5 dependency, so maintainers should choose a scoped managed-plugin override policy or validator with regression coverage.
