## Label Justifications

- `P2`: The issue reports a real plugin startup/runtime crash path with limited blast radius to Express 4.x plugins installed through managed npm roots.
- `impact:crash-loop`: Affected plugins can fail their HTTP route startup path when path-to-regexp 8.x is forced into an Express 4 dependency graph.

## Summary

Keep open: current main still applies the host path-to-regexp 8.4.0 override into managed npm plugin roots, and dependency metadata confirms that breaks Express 4.x plugin graphs while a blanket downgrade would conflict with OpenClaw's Express 5/router 2.x stack.

## What This Changes

The issue requests a central OpenClaw packaging or plugin-validation fix for Express 4.x plugins resolving incompatible path-to-regexp versions during plugin installs.

## Best Possible Solution

Define a narrow managed-npm-root dependency policy that preserves plugin-owned Express 4 dependency graphs without globally downgrading OpenClaw's Express 5/router path, with docs and regression coverage for install/update behavior.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current managed npm installs inherit the host path-to-regexp 8.4.0 override, while Express 4.x package metadata requires path-to-regexp 0.1.x. I did not run a live plugin install because this cleanup review is read-only.

## Solution Assessment

No, the issue's preferred blanket downgrade is not the best fix because OpenClaw's current Express 5/router stack depends on path-to-regexp 8.x. The safer direction is a maintainer-approved scoped managed-root policy or validator that preserves plugin dependency ownership without breaking core.
