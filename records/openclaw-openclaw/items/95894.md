## Label Justifications

- `P2`: The issue reports a real managed-plugin startup crash path with limited blast radius to Express 4.x plugins installed through OpenClaw-managed npm roots.
- `impact:crash-loop`: Affected plugins can fail route initialization or startup when `path-to-regexp` 8.x is forced into an Express 4.x dependency graph.

## Summary

Keep open: current main and the latest release still export the host `path-to-regexp` 8.4.0 override into managed npm plugin roots, while Express 4.x plugin graphs require `path-to-regexp` 0.1.x. The merged stale-pin fix only addressed npm `EOVERRIDE`, not this transitive override conflict.

## What This Changes

The issue requests a core managed-plugin install fix so Express 4.x plugins are not forced onto OpenClaw's `path-to-regexp` 8.x override and do not crash at route startup.

## Best Possible Solution

Define a maintainer-approved managed npm override policy that prevents host-only pins from forcing incompatible transitive versions into third-party plugin graphs, with focused install/update tests and docs for the supported plugin dependency boundary.

## Reproduction Assessment

Yes. Source inspection shows managed npm plugin installs inherit OpenClaw's `path-to-regexp` 8.4.0 override, while Express 4.x registry metadata requires `path-to-regexp` 0.1.12; I did not run a live install because this review is read-only.

## Solution Assessment

No, the issue's preferred blanket downgrade is not the best fix because OpenClaw's current Express 5/router dependency path requires `path-to-regexp` 8.x. A scoped managed-root override policy or targeted validator is the maintainable direction.
