## Label Justifications

- `P2`: The issue describes a normal-priority plugin install/runtime bug with limited blast radius to Express 4.x plugins.
- `impact:crash-loop`: The reported failure is a runtime crash when Express 4.x plugin route code loads with an incompatible `path-to-regexp` version.

## Summary

This issue is still necessary: current main and the latest release both propagate OpenClaw workspace overrides into managed plugin npm roots, and that includes a `path-to-regexp` 8.x pin that conflicts with Express 4.x plugin dependency contracts.

## What This Changes

The issue asks OpenClaw to prevent Express 4.x-based plugin installs from resolving an incompatible `path-to-regexp` 8.x version through core-managed package override behavior.

## Best Possible Solution

Adopt a scoped managed-npm override policy that keeps OpenClaw-owned pins where needed but does not force `path-to-regexp` 8.x onto Express 4 plugin subtrees, with regression coverage for an Express 4 plugin install.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: managed plugin installs read and apply OpenClaw workspace overrides, and the current override set includes `path-to-regexp: 8.4.0` while Express 4 declares `path-to-regexp: 0.1.12`. A live packaged install/startup repro was not run in this read-only sweep.

## Solution Assessment

No, the proposed global override downgrade is not the best fix because Express 5/router legitimately uses `path-to-regexp` 8.x. The better direction is a scoped managed-plugin override policy or dependency planner that preserves plugin package contracts without losing OpenClaw-owned security pins where they are valid.
