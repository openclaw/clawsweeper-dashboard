## Label Justifications

- `P2`: The report describes a normal-priority plugin startup crash with limited blast radius to Express 4-based plugins installed through managed npm roots.
- `impact:crash-loop`: The issue is about plugin HTTP routes failing at runtime with a route parser crash when the plugin loads Express 4 against an incompatible path-to-regexp.

## Summary

Keep open: current main and the latest release still propagate OpenClaw workspace overrides into managed npm plugin projects, and the reported Express 4 plugin example depends on Express 4 without its own override, so the central compatibility problem remains actionable.

## What This Changes

The issue asks OpenClaw core to prevent Express 4.x-based plugins from resolving an incompatible path-to-regexp version or to warn plugin authors before runtime crashes.

## Best Possible Solution

Keep the issue open for a managed npm plugin install policy fix that scopes or filters host overrides so OpenClaw security pins do not force incompatible versions into plugin-owned dependency graphs.

## Reproduction Assessment

Yes at source level: current main injects OpenClaw workspace overrides into managed npm plugin projects, the reported package depends on Express 4 without its own override, and Express 4 expects path-to-regexp 0.1.x while the host override pins 8.4.0. I did not run the full external plugin startup path.

## Solution Assessment

No for the preferred root override as stated: a single global path-to-regexp override cannot satisfy both Express 4 and the Express 5/router dependency shape. The better fix is to change managed plugin install override propagation or validation with explicit maintainer approval.
