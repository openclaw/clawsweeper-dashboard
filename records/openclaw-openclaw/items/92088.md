## Label Justifications

- `P1`: A transient update-command failure can strand the primary gateway offline and break all channel workflows until manual intervention.
- `impact:crash-loop`: The issue concerns process-level availability because the managed gateway remains inactive after the handoff fails.
- `impact:message-loss`: While the gateway is stopped, channel messages cannot receive agent responses.

## Summary

Current main still permits a transient managed-update failure to become a sustained outage: after the gateway exits, every update-command failure path records an error but performs no service recovery. This is a distinct, actionable reliability bug rather than work already covered by the earlier handoff fixes.

## What This Changes

Ensure failed managed update handoffs restore gateway availability and finalize restart-sentinel state with the actual update and recovery outcome.

## Best Possible Solution

Add one failure-only recovery routine to the detached handoff owner that restarts the same managed gateway through its canonical supervisor contract, records the recovery outcome, and leaves successful restart ownership unchanged.

## Reproduction Assessment

Yes at source level. Current main waits for the gateway process to exit and then has explicit spawn, exit-code, signal, and helper failure paths that return without invoking any managed-service restart.

## Solution Assessment

Yes, with a narrow refinement: recovery belongs at the existing detached handoff boundary and should use established supervisor identity rather than adding configuration, reporter-specific shell workarounds, or a second successful restart path.
