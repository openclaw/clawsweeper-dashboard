## Label Justifications

- `P1`: A transient managed-update failure can leave the primary gateway unavailable until an operator manually restarts it.
- `impact:crash-loop`: The issue concerns process-level availability because the managed gateway remains stopped after the failed handoff.
- `impact:message-loss`: A stopped gateway suppresses channel responses for the duration of the outage.

## Summary

The issue remains necessary: current main records detached update failures without restoring the managed gateway, while the specifically linked fix PR is open, cleanly mergeable, and backed by real systemd before/after proof. Keep the issue open until that PR merges or maintainers choose a replacement.

## What This Changes

Make managed update handoffs restore the gateway service when the update command fails after stopping the service.

## Best Possible Solution

Land a detached-helper recovery path that uses the canonical service identity and an idempotent supervisor start operation on every update-command failure, with focused systemd, launchd, and Windows wiring tests and no recovery action after successful updates.

## Reproduction Assessment

Yes from source inspection and linked live evidence. Current main has detached-helper spawn and nonzero-exit paths with no service restoration, and the linked PR shows an Ubuntu systemd reproduction with an inactive service and unavailable gateway before the patch.

## Solution Assessment

Yes, provisionally. The detached helper is the narrowest viable recovery owner because it survives the update process that may stop the gateway; the linked approach reuses canonical service identities and idempotent platform start commands rather than adding a competing update path.
