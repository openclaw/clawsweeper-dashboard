## Label Justifications

- `P1`: A transient failure in the standard managed update path can leave the primary gateway offline until an operator manually recovers it.
- `impact:crash-loop`: The issue concerns process-level gateway availability and failed recovery after a managed lifecycle operation.
- `impact:message-loss`: The stopped gateway prevents channel responses and processing until service recovery.

## Summary

Current main still has the reported availability gap: the detached updater converts command failures into terminal sentinel errors but performs no compensating gateway restart. The stale pending-sentinel concern is already substantially handled, while the post-stop service recovery remains necessary.

## What This Changes

The issue requests best-effort restoration of a managed gateway when the detached update command fails after stopping it, plus terminal cleanup of stale update state.

## Best Possible Solution

Give the detached handoff an explicit, race-safe failure cleanup that makes one best-effort restart of the exact prior systemd or launchd service after post-stop failure, while retaining the existing terminal sentinel ownership checks.

## Reproduction Assessment

Yes from source and production evidence, though this read-only review did not execute a live systemd reproduction. The supplied logs show stop followed by exit 127, and every matching current-main failure branch exits without a gateway recovery action.

## Solution Assessment

Yes in direction, but a broad shell trap is not ideal. The narrowest maintainable fix is explicit supervisor-aware cleanup that preserves sentinel ownership and cannot double-restart after a successful update.
