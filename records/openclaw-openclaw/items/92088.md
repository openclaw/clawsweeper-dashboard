## Summary

Current main still has the reported failure class: the managed handoff records update-command failure but does not restore the stopped gateway. The issue is recent, unprotected, has no linked fix PR, and is a clear bounded core reliability repair.

## What This Changes

Ensure a failed managed update handoff restarts the previously stopped gateway and does not leave stale restart state.

## Best Possible Solution

Make the detached managed handoff own best-effort service restoration after every post-parent-exit update failure, using the canonical supervisor/service identity contract and preserving sentinel ownership safeguards.

## Reproduction Assessment

Yes, at high-confidence source level: after the parent gateway exits, current main handles update-command startup and exit failures without any visible service restoration, matching the supplied systemd logs. This read-only review did not execute a destructive managed-service update scenario.

## Solution Assessment

Yes, with one refinement: recovery belongs in the canonical detached handoff lifecycle and should reuse supervisor/service identity rather than hard-code a caller-side fallback. Focused tests should cover spawn failure, nonzero exit, successful recovery attempt, and sentinel ownership.

In the canonical managed-service update handoff, ensure that once the gateway parent has exited, every update-command failure path—spawn error, nonzero exit, signal exit, or helper exception—makes a best-effort attempt to restore the previously managed gateway service. Reuse existing supervisor/service identity contracts and keep behavior portable; do not add a config option, runtime fallback stack, or Linux-only hard-coded policy when an existing daemon/service abstraction can own it. Preserve the current sentinel ownership rules so unrelated or newer restart work is never overwritten, and keep the absolute `systemd-run` resolution. Add focused regression tests for failed command startup and failed command exit, including recovery invocation and sentinel failure state. Do not edit `CHANGELOG.md`; include user-visible release-note context in the PR body or commit message.
