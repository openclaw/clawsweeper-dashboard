## Summary

Keep open: the PR is a focused, still-useful fix for a subagent completion delivery edge case that current main does not already solve. I found no blocking code-review findings, but the merge still carries maintainer-visible message-delivery and session-state risk because it makes the visible message-tool failure terminal after requester wake is credited.

## What This Changes

The PR separates subagent requester-wake success from required visible message-tool delivery status and persists failed visible-delivery accounting for subagent completions.

## Best Possible Solution

Land this only if maintainers accept the terminal no-retry semantics for this failure mode; otherwise require a retry-preserving design before merge.

## Reproduction Assessment

Yes, at source level: current main returns an undelivered direct result for required message-tool subagent completion with no message-tool evidence, while the PR changes that path to requester wake delivered plus visible failure metadata. I did not run a live transport reproduction in this read-only review.

## Solution Assessment

Mostly yes: splitting requester wake from visible delivery failure is a narrow maintainable shape and the tests cover the new accounting. The unresolved question is whether terminal no-retry semantics are the desired product behavior for required visible delivery failures.
