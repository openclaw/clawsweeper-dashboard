## Summary

Keep open for maintainer review: the patch looks coherent and I found no blocking code defect, but it intentionally makes required visible-delivery failure terminal after requester wake, which is a message-delivery and session-state decision.

## What This Changes

The PR splits subagent requester-wake success from required visible message-tool delivery failure, persists visible failure metadata, and marks linked detached task delivery as failed.

## Best Possible Solution

Land this only if maintainers accept terminal visible-failure semantics; otherwise require a retry-preserving design before merge.

## Solution Assessment

Mostly yes: splitting requester wake from visible delivery failure is a narrow maintainable shape, and the new tests cover the accounting path. The unresolved question is whether terminal no-retry semantics are the desired behavior for required visible delivery failures.
