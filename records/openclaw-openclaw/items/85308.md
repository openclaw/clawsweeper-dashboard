## Summary

Keep open: the bug shape is valid, but this patch is not merge-ready because it turns a required visible message-tool failure into `delivered: true`, which downstream cleanup treats as a delivered completion. The PR also lacks after-fix real behavior proof from a real setup.

## What This Changes

The branch adds requester-wake and visible-delivery status fields to subagent announcement/run records, changes message-tool failure paths to return `delivered: true` with `visibleDeliveryStatus: failed`, and adds focused regression tests.

## Best Possible Solution

Keep requester wake bookkeeping separate from visible delivery success, preserving the wake timestamp/status while leaving required visible delivery failed or retryable until message-tool delivery is proven or explicitly exhausted.

## Reproduction Assessment

Yes from source inspection: current main returns a failed delivery result for required message-tool output that is absent, and the PR changes that same path to `delivered: true`. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No. Preserving requester wake is the right direction, but using `delivered: true` for a failed visible delivery is not the narrowest maintainable fix; the lifecycle should consume a separate requester-wake signal without marking visible delivery and task notification as delivered.
