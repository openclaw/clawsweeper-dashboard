## Summary

Keep open: the proof gap is now mostly addressed, but the patch still makes a required visible message-tool failure look delivered in the linked task state, which can hide a lost channel reply while clearing retry state.

## What This Changes

The PR splits subagent requester wake success from visible message-tool delivery failure metadata and clears pending final delivery when the requester handoff runs.

## Best Possible Solution

Keep requester wake delivery separate, but make the required visible-delivery failure terminal state visible through failed task/status semantics or another explicit maintainer-approved failure status before clearing retries.

## Reproduction Assessment

Yes, at source level: current main returns `delivered: false` for required message-tool no-evidence paths and lifecycle cleanup keeps those completions in retry or failed delivery handling. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. Splitting requester wake from visible delivery status is the right shape, but the patch should not surface a required visible delivery failure as linked task delivery success without an explicit maintainer decision.
