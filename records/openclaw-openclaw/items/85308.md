## Summary

Keep open: the lifecycle split is useful, but this PR is not merge-ready because the new failed visible-delivery state is written with an announced timestamp while the existing SQLite hydration path reclassifies any announced timestamp as delivered after restart.

## What This Changes

The PR separates requester wake accounting from required visible message-tool delivery failure for subagent completions and records the visible failure in lifecycle/task delivery state.

## Best Possible Solution

Preserve requester-wake and visible-delivery as separate states, make the failed visible-delivery state round-trip through SQLite, and then have maintainers explicitly accept or reject the terminal no-retry semantics.

## Reproduction Assessment

Yes. Source inspection shows the PR writes a failed delivery with `announcedAt`, while current SQLite hydration treats `completion_announced_at` as delivered; I did not run tests because this was a read-only review.

## Solution Assessment

No. The lifecycle split is plausible, but the current patch is not the best mergeable solution until failed visible-delivery state survives the SQLite restart path and current-head behavior proof is refreshed.
