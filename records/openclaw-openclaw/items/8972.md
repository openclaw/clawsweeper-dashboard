## Summary

Keep open. Current main still resolves Discord guild channel entries as exact id/name/slug keys, parent fallbacks, or the literal `*`, so glob support remains a new access-control feature rather than an obsolete cleanup item.

## What This Changes

Add glob or wildcard pattern entries to Discord guild channel allowlists so dynamically named channels can be accepted without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a documented Discord-owned pattern syntax and precedence model, preserve exact id/name/slug and parent/literal `*` behavior, update schema/types/docs/diagnostics, and add focused allowlist regression coverage.

## Reproduction Assessment

No: this is not a bug reproduction because the issue requests a new feature. Source inspection gives high confidence that `ws-*` is currently treated as a literal key and will not match `ws-task-1` unless an exact, parent, or literal `*` entry also matches.

## Solution Assessment

Unclear until maintainers approve the public glob syntax and access-control precedence. A narrow Discord allowlist implementation is plausible, while category inheritance should remain tied to https://github.com/openclaw/openclaw/issues/60582 unless the designs are intentionally combined.
