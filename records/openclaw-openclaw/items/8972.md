## Label Justifications

- `P2`: This is a normal-priority Discord workflow enhancement with clear user value but no current core outage or regression.
- `impact:security`: Pattern matching would change which Discord guild channels are authorized to trigger agent processing.
- `impact:message-loss`: Messages from newly created prefix-related channels are currently suppressed until each channel is explicitly configured.

## Summary

Current main and the latest release still support only exact/normalized Discord channel keys, thread-parent fallback, and the literal `*` fallback, so prefix-scoped dynamic channels remain unsupported; no merged or viable open PR supersedes this request, but the authorization grammar and precedence need maintainer product/security judgment.

## What This Changes

Add pattern matching for Discord guild channel allowlist entries so dynamically created channels with matching names can be accepted without per-channel config edits and gateway restarts.

## Best Possible Solution

Keep the shipped exact and literal-star behavior, and add a Discord-owned, documented pattern form only after maintainers approve deterministic precedence and audit/status semantics, with focused tests covering intended matches, overlaps, threads, disabled entries, and denied near-matches.

## Reproduction Assessment

Yes, at source level: configure a non-empty guild channel map containing `ws-*` but no literal `*`, then send in `ws-task-1`; current code treats `ws-*` as an ordinary key, finds no supported match, and denies the channel. No live Discord run was performed during this read-only review.

## Solution Assessment

Unclear pending maintainer design approval. Implementing matching in the Discord-owned allowlist path is narrower than changing all shared channels, but the syntax, precedence, and security diagnostics must be chosen explicitly before it is the best fix.
