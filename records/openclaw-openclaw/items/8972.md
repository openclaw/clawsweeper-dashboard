## Label Justifications

- `P2`: This is a normal-priority Discord allowlist feature with clear user impact but a bounded channel/config surface.
- `impact:security`: The requested behavior changes which Discord channels an agent is allowed to listen to, so it affects an access-control boundary.
- `impact:message-loss`: Without pattern support, messages in newly created matching Discord channels are suppressed until operators patch config and restart the gateway.

## Summary

Keep this issue open: current main and the latest release still treat Discord guild channel entries as exact id/name/slug keys, parent fallbacks, or the literal `*` fallback, not glob patterns, and no merged PR or canonical issue now owns the glob request.

## What This Changes

The issue requests glob or wildcard matching in Discord guild channel allowlists so newly created channels like `ws-*` can be allowed without config edits and gateway restarts.

## Best Possible Solution

Keep this open and design a Discord-owned pattern channel match contract that preserves exact/parent precedence and literal `*` fallback, then update schema/types/docs/status/audit behavior and regression tests; keep category inheritance in the separate open category issue unless maintainers intentionally combine the designs.

## Reproduction Assessment

Yes, from source: configure a Discord guild `channels` entry such as `ws-*`, then send from `ws-task-1`; current main checks exact id/name/slug keys and literal `*`, so the pattern key does not match and the configured channel gate resolves to denied.

## Solution Assessment

Unclear for implementation shape: glob support is plausible, but the best fix needs a maintainer-approved access-control contract for syntax, precedence, escaping, and interaction with category-level and literal-wildcard behavior.
