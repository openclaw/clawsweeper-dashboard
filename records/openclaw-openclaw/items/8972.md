## Label Justifications

- `P2`: This is a normal-priority Discord channel configuration improvement with real workflow impact but no evidence of an urgent regression.
- `impact:security`: The request changes which Discord guild channels can trigger an agent through allowlist matching.
- `impact:message-loss`: Current exact channel allowlists can silently drop intended messages from dynamically created Discord channels until config is edited and the gateway restarts.

## Summary

Keep open: current main and the latest release still treat Discord guild channel entries as exact channel id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback. Pattern matching remains a security-sensitive Discord config feature that needs maintainer design before implementation.

## What This Changes

The issue requests glob or wildcard pattern entries in Discord guild channel allowlists so dynamically created channels like `ws-task-1` can be admitted without per-channel config edits and gateway restarts.

## Best Possible Solution

Implement a documented Discord-owned pattern channel entry only after maintainer approval, preserving exact and parent precedence, keeping literal `*` as the broad fallback, defining pattern ordering, and updating types, schema, docs, audit/status output, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `guilds.<id>.channels` with `ws-*`, send from a channel whose slug is `ws-task-1`, and observe the current resolver only checks exact keys plus literal `*`, then returns `{ allowed: false }` on no match.

## Solution Assessment

Unclear until maintainers choose the security semantics. A narrow Discord-owned matcher extension is plausible, but the best fix must define precedence and keep category inheritance and broader access-list redesigns separate unless maintainers intentionally combine them.
