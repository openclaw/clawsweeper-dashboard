## Label Justifications

- `P2`: This is a bounded Discord enhancement with real workflow impact and a workaround, but it is not an urgent outage or regression.
- `impact:security`: Discord channel pattern matching would decide which guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created channels are currently denied until each channel has an exact configured entry or the broad `*` fallback is used.

## Summary

Keep open: current main still treats Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback, so `ws-*` remains an ordinary unmatched key and the requested pattern allowlist behavior is still missing in current main and the latest release.

## What This Changes

Request support for glob or wildcard Discord guild channel allowlist entries so dynamically named channels can be authorized without editing config for each new channel.

## Best Possible Solution

Approve one documented Discord-owned channel pattern contract, then update allow-list matching, exported types/schema, docs, audit/status behavior, and focused regression tests together.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. A narrow Discord-owned pattern contract is the right implementation boundary if that direction is approved.
