## Label Justifications

- `impact:security`: Channel matching determines which Discord messages cross the configured authorization boundary and reach the agent.
- `impact:message-loss`: Under the current allowlist contract, messages from dynamically created channels without an exact or literal-star match are suppressed.

## Summary

Keep this issue open: current main still interprets Discord channel-map keys as exact id/name/slug entries, thread-parent entries, or the literal `*` fallback, so `ws-*` does not match `ws-task-1`. The request remains distinct from the open category-inheritance proposal, while changing this authorization boundary requires an explicit maintainer decision on syntax and precedence.

## What This Changes

Add pattern-based Discord guild channel allowlist entries so dynamically created matching channels work without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned, documented pattern form with deterministic precedence after exact channel and thread-parent matches but before literal `*`; constrain which channel attributes patterns can match, preserve deny-by-default behavior, expose match provenance to status/audit paths, and add focused authorization tests plus Discord documentation.

## Reproduction Assessment

Yes at source level: configure a non-empty Discord guild channel map containing `ws-*` but no literal `*`, then send in `ws-task-1`; current main treats `ws-*` as an ordinary record key and resolves the new channel as denied. No live Discord run was needed to establish the matching result from the closed source path.

## Solution Assessment

Unclear until maintainers choose the public syntax, matched channel attributes, and overlap precedence. A narrow Discord-owned pattern contract is preferable to changing the documented channel map into additive overrides or silently combining it with category inheritance.
