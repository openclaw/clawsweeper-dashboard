## Summary

Keep open: current main and v2026.6.8 still treat Discord guild channel allowlist keys as exact channel keys, one parent fallback, or the literal `*`, so a key like `ws-*` is not evaluated as a glob and the requested access-control feature remains missing.

## What This Changes

Request support for glob-style Discord guild channel allowlist entries so dynamically created channels like `ws-task-1` can be admitted without per-channel config edits and restarts.

## Best Possible Solution

Approve and document one Discord-owned selective pattern contract, then update the matcher, Discord allowlist path, types/schema, docs, status/audit behavior, and focused regression tests together.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence; if that direction is approved, the narrow implementation boundary is the Discord-owned allowlist path plus aligned schema/docs/tests.
