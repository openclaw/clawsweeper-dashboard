## Label Justifications

- `P2`: This is a bounded Discord enhancement with real workflow impact and a restart/config-edit workaround, but it is not an urgent outage.
- `impact:security`: Glob channel keys would decide which Discord guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created Discord channels are currently denied unless each channel is listed exactly or a broad `*` fallback is used.

## Summary

Keep open: current main and v2026.6.8 still only support exact Discord channel keys, parent fallback, normalized names, and the literal `*` fallback, so the requested `ws-*` style glob allowlist remains missing and needs maintainer-approved authorization semantics.

## What This Changes

Request to let Discord guild channel allowlist entries match dynamic channel names with glob or wildcard patterns, with category-based inheritance noted as an alternative.

## Best Possible Solution

Approve one documented Discord-owned pattern contract, then update the allow-list resolver, exported types, Zod schema, docs, audit/status behavior, and regression tests together while keeping category and hot-reload work separate.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. If approved, the narrowest maintainable solution is a Discord-owned pattern contract rather than bundling category inheritance, additive overrides, or hot-reload into the same change.
