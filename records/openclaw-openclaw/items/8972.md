- `impact:message-loss`: Under the current non-empty channel allowlist, messages from dynamically created unmatched channels are denied and not processed.

## Summary

Current main and v2026.6.5 still lack partial-wildcard channel matching, and live search found no separate canonical issue or open PR for that exact capability. The issue should remain open, but implementation needs maintainer product/security judgment because name patterns expand an authorization boundary and must define precedence against exact channel, thread-parent, category, and literal `*` behavior.

## What This Changes

Add glob-style Discord guild channel allowlist entries so dynamically created matching channels are accepted without config edits and gateway restarts.

## Best Possible Solution

Keep this as the canonical pattern-matching request and define a documented Discord-owned pattern form with exact/normalized exact and thread-parent precedence, deterministic pattern ordering, literal `*` as the final fallback, matched-key audit metadata, schema/help/docs updates, and regression coverage; keep category inheritance and broader user/role access policy in their linked issues unless maintainers intentionally unify the designs.

## Reproduction Assessment

Yes, at source level with high confidence: configure a guild channel key such as `ws-*` and send from `ws-task-1`; current main treats the configured key literally, finds no exact/parent/`*` match, and resolves the channel as denied. This was established by source and release inspection rather than a live Discord run.

## Solution Assessment

Unclear. Pattern matching is a reasonable narrow direction, but the proposed bare glob-key syntax is not yet the best approved solution until maintainers decide the authorization grammar, match ordering, normalization, audit behavior, and relationship to category-based configuration.

## Risks / Open Questions

- A broad or ambiguous pattern can authorize newly created channels that were never explicitly reviewed, so escaping, normalization, case behavior, and denial precedence must be specified.
- Multiple matching patterns need deterministic ordering; exact id/name/slug and thread-parent matches should not be silently displaced.
- Status, audit, and diagnostics must preserve the matched pattern key so operators can understand why a message was accepted or denied.
- The separate category proposal at https://github.com/openclaw/openclaw/issues/60582 and broader access-policy proposal at https://github.com/openclaw/openclaw/issues/69748 could create overlapping precedence rules if designed independently.
