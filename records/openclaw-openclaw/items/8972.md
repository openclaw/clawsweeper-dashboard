## Summary

Keep open. Current main still handles Discord guild channel entries as exact id/name/slug keys, parent fallbacks, or the literal `*`; `ws-*` remains an ordinary unmatched record key, and the related category/access-list items do not supersede this narrower pattern request.

## What This Changes

Request support for glob/wildcard Discord guild channel allowlist entries so dynamically created matching channels work without config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned types, schema, docs/status/audit behavior, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A guild channel config containing `ws-*` and an incoming channel slug `ws-task-1` will miss because the matcher uses exact lookups plus parent and literal `*` fallback, and unmatched configured channels deny.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord allow-list implementation is plausible, but it should follow product/security review rather than automatic cleanup.
