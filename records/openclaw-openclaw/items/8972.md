## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface but no urgent regression or outage.
- `impact:security`: The request changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until config is patched or a broader wildcard is used.

## Summary

Keep open. Current main and the latest release still treat Discord guild channel entries as exact id/name/slug matches, one thread-parent fallback, or the literal `*`; glob keys such as `ws-*` remain a new Discord access-control feature, not an implemented behavior.

## What This Changes

Request support for wildcard or glob-style Discord guild channel allowlist entries so dynamically created channels can be accepted without adding each channel key to config.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, preserve existing exact and literal `*` behavior, update types/schema/docs/status/audit output, and add focused allowlist/preflight regression coverage while coordinating category and access-list work with the related issues.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary record key, while runtime lookup only checks exact id/slug/name candidates, thread-parent fallback, and the literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the maintainable solution needs explicit access-control precedence and coordination with category/access-list design work.
