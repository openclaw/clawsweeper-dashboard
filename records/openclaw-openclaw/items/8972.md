## Label Justifications

- `impact:security`: The requested feature changes which Discord guild channels are authorized to trigger OpenClaw.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until configuration is patched and the gateway is restarted.

## Summary

Keep open. Current main and the latest shipped tag still implement exact Discord channel key matching plus thread-parent and literal `*` fallback only; adding glob syntax would widen a Discord channel authorization boundary and needs maintainer product/security review.

## What This Changes

The issue requests glob/wildcard support in Discord guild channel allowlists so dynamically created channels like `ws-task-1` can match a configured pattern such as `ws-*` without config edits and gateway restarts.

## Best Possible Solution

Keep this as an open Discord plugin design task: define a documented pattern syntax and precedence model, preserve exact and literal `*` behavior, update types/schema/docs/status/audit output, and add focused allowlist/preflight regression tests.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary `channels` record key, while runtime lookup only checks exact id/slug/name candidates, thread-parent fallback, and literal `*`, so `ws-task-1` remains denied unless `*` or an exact entry is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the safest implementation needs explicit access-control precedence and coordination with the category and access-list design issues.
