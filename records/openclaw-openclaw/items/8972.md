## Label Justifications

- `P2`: This is a meaningful but bounded Discord configuration improvement that needs design review rather than emergency handling.
- `impact:security`: The requested matcher controls which Discord channels may deliver messages into the agent workflow.
- `impact:message-loss`: Dynamically created channels that are intended to match a naming convention are currently denied and their messages are not processed.

## Summary

Current main still treats Discord guild channel keys as exact ID/name/slug entries, parent fallbacks, or the literal `*` fallback, so the requested pattern capability remains missing. This should stay open for maintainer product and security judgment because glob precedence changes an access-control boundary, while the category alternative remains separately tracked.

## What This Changes

Add glob or wildcard matching for Discord guild channel allowlist entries so dynamically created matching channels do not require config edits and restarts.

## Best Possible Solution

Define a documented Discord-owned pattern form with deterministic exact, normalized, parent, pattern, and literal-`*` precedence; expose the selected match in status/security-audit output and add focused access-control tests, while leaving category inheritance to its canonical issue.

## Reproduction Assessment

Yes, at source level with high confidence: configure `ws-*` under a non-empty Discord guild `channels` block and a channel named `ws-task-1` has no exact candidate match, so only a separately configured literal `*` could allow it. This is a clear source reproduction, not an executed live Discord run.

## Solution Assessment

No, the proposal is not yet specific enough to be the best implementation: a generic shared glob branch risks sibling channels, and the category alternative is separate work. The safer design is a Discord-owned pattern contract with explicit ordering, audit visibility, docs, and regression coverage.
