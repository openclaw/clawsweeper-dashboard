## Summary

Keep this issue open. Current main still treats Discord guild channel allowlist entries as exact channel id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback; a key like `ws-*` remains a normal record key with no glob evaluation, and the public schema/docs do not define a pattern or category contract.

## What This Changes

The issue requests glob or wildcard pattern support in Discord guild channel allowlists so dynamic channels matching names such as `ws-*` are accepted without per-channel config updates.

## Best Possible Solution

Implement a documented Discord-owned channel pattern form with explicit precedence, likely by extending the shared matcher in an opt-in way. Keep exact id/name/slug and thread-parent behavior stable, preserve literal `*` as the broad fallback, update exported types/Zod schema/docs/status or audit surfaces, and add focused tests for dynamically created matching channels.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a Discord guild with `channels: { "ws-*": { enabled: true } }` and evaluate a channel named `ws-task-1`; current code only compares exact id/name/slug candidates and the literal `*`, then returns `{ allowed: false }` on a miss.

## Solution Assessment

Unclear. A glob key such as `ws-*` is plausible UX, but the best implementation needs maintainer-defined access-control precedence; category inheritance should remain under #60582 and additive override semantics under #73387 unless maintainers decide to merge those designs.
