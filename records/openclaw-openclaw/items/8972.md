## Summary

Keep open: current main still treats Discord guild channel allowlist entries as exact channel id/name/slug keys, thread-parent fallbacks, or the literal `*`; glob keys such as `ws-*` are not implemented and need maintainer-defined authorization semantics.

## What This Changes

Add glob/wildcard matching for Discord guild channel allowlist entries so dynamically created channels like `ws-task-1` can be accepted by a `ws-*` config entry without per-channel config edits and gateway restarts.

## Best Possible Solution

Implement a documented Discord-owned pattern matching contract with explicit precedence, schema/types/docs/status/audit coverage, and regression tests, while keeping category inheritance under #60582 unless maintainers deliberately merge the designs.

## Reproduction Assessment

Yes for the current missing capability by source inspection: configure a guild channel entry such as `"ws-*": { "enabled": true }` and a Discord channel named `ws-task-1`; current main only checks exact id/name/slug keys, parent keys, and the literal `*`, so the pattern falls through to `{ allowed: false }`. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Unclear. The proposed glob UX is plausible, but the maintainable fix needs owner-defined authorization precedence and should stay separate from category inheritance unless maintainers choose to combine those designs.
