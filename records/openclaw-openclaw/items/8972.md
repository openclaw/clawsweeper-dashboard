## Label Justifications

- `P2`: This is a bounded Discord configuration improvement with meaningful workflow impact but no current outage or regression.
- `impact:security`: Discord channel matching is an ingress allowlist boundary that determines which channel messages may reach the agent.
- `impact:message-loss`: Messages from dynamically created matching channels are suppressed until each channel is explicitly configured or all channels are broadly admitted with `*`.

## Summary

Current main and the latest release still support only exact channel id/name/slug keys, thread-parent fallback, and the literal `*` fallback; selective patterns such as `ws-*` remain unsupported, and no canonical issue or viable PR supersedes that request.

## What This Changes

Add Discord guild channel allowlist glob matching so dynamically created channels can be accepted without per-channel config edits and gateway restarts.

## Best Possible Solution

Add a documented Discord-owned channel-pattern form with exact and parent matches taking precedence, deterministic ordering among patterns, literal `*` last, and audit-visible match metadata; keep category inheritance in https://github.com/openclaw/openclaw/issues/60582.

## Reproduction Assessment

Yes at source level: `ws-*` is stored as an ordinary record key, Discord supplies only exact channel candidates, and only literal `*` receives fallback semantics. No runtime test was executed because this was a strictly read-only review.

## Solution Assessment

The prefix-pattern direction is reasonable, but the proposed category alternative should remain separate and maintainers must approve the syntax, ordering, and access-control precedence before implementation.
