## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface but no urgent regression or outage.
- `impact:security`: The requested behavior changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until config is patched, a broader fallback is used, and the gateway is restarted.

## Summary

Keep open. Current main and v2026.5.22 still treat Discord guild channel entries as exact id/name/slug keys, one parent fallback, or the literal `*`; glob keys such as `ws-*` remain a new Discord access-control feature that needs maintainer product and security direction.

## What This Changes

Add glob or wildcard matching for Discord guild channel allowlist entries so dynamically created matching channels can be accepted without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a documented Discord-owned pattern syntax and precedence model, preserve exact id/name/slug, disabled-entry, thread-parent, and literal `*` behavior, then update schema, docs, status/audit output, and focused allowlist/preflight regression coverage.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary record key while runtime lookup only checks exact id/slug/name candidates, one parent fallback, and literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the maintainable solution needs explicit access-control precedence and coordination with category/access-list design work.
