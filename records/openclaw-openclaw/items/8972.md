## Summary

Keep this issue open: current main and the latest release still lack pattern evaluation, the related category and broader access-list issues do not replace this narrower request, and no viable implementation PR was found. The remaining decision is the security-sensitive public syntax and precedence for pattern matches.

## What This Changes

Add glob or wildcard matching to Discord guild channel allowlists so dynamically created channels can be authorized without per-channel configuration changes.

## Best Possible Solution

Define a first-class Discord-owned pattern form with deterministic precedence, then align exported types and schema, audit/status diagnostics, docs, and focused authorization tests; keep category inheritance in https://github.com/openclaw/openclaw/issues/60582 and the broader access-list redesign in https://github.com/openclaw/openclaw/issues/69748.

## Reproduction Assessment

Yes, at source level. Configure a non-empty guild channel map containing only `ws-*` and resolve `ws-task-1`; current main checks exact ID/slug/name, optional thread parent, and literal `*`, then returns `allowed: false`.

## Solution Assessment

Yes in direction, but not as silent glob interpretation of every existing record key. The narrowest maintainable solution is an explicit typed pattern contract with documented precedence and security-focused regression coverage.
