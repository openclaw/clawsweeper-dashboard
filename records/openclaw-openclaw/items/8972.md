## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with clear impact but no current outage or urgent regression.
- `impact:security`: The requested wildcard behavior changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Current exact-only channel allowlists can cause intended dynamically created Discord channels to be ignored until config is patched or a broader literal wildcard is used.

## Summary

Keep open: current main and v2026.6.5 still treat Discord guild channel entries as exact id/name/slug keys, parent fallbacks, or the literal `*`, so `ws-*` remains an unimplemented Discord access-control feature that needs maintainer design.

## What This Changes

Add pattern-based Discord guild channel allowlist keys such as `ws-*` so dynamically created matching channels can be accepted without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned glob syntax and precedence model, preserve exact and literal `*` behavior, update types/schema/docs/audit output, and add focused allowlist/preflight regression coverage while keeping category inheritance and broader access-list redesign on their own tracks.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary record key, while runtime lookup only checks exact id/slug/name candidates, parent fallback, and literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support fits the dynamic-channel workflow, but the maintainable fix needs explicit access-control precedence and coordination with category/access-list design work.
