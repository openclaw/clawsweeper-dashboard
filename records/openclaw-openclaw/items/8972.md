## Label Justifications

- `P2`: This is a normal-priority Discord channel configuration improvement with real workflow impact but no evidence of an urgent regression.
- `impact:security`: The request changes which Discord guild channels can trigger an agent through allowlist matching.
- `impact:message-loss`: Current exact channel allowlists can silently drop intended messages from dynamically created Discord channels until config is edited and the gateway restarts.

## Summary

Keep open: current main and the latest release still treat Discord guild channel entries as exact channel id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback; glob matching remains missing and needs maintainer-approved security semantics.

## What This Changes

Add glob or wildcard pattern matching for Discord guild channel allowlist entries so dynamically named channels can be allowed without per-channel config edits and gateway restarts.

## Best Possible Solution

Approve a documented Discord-owned pattern channel entry only after defining precedence, escaping, audit/status output, docs, schema/types, and regression tests, while keeping category inheritance as separate work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `guilds.<id>.channels` with `ws-*`, send from a channel whose slug is `ws-task-1`, and observe that current resolution checks exact keys plus literal `*` before returning `{ allowed: false }` on no match.

## Solution Assessment

Unclear until maintainers choose the security semantics. A narrow Discord-owned matcher extension is plausible, but the best fix must define precedence and keep category inheritance and broader access-list redesigns separate unless maintainers intentionally combine them.
