- `impact:security`: Channel-pattern matching determines which Discord messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created channels are currently suppressed until each channel receives an exact configuration entry.

## Summary

Current main and v2026.6.6 still interpret Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the broad literal `*`; selective patterns such as `ws-*` remain unsupported, and the separate category and access-policy issues do not supersede this narrower request.

## What This Changes

Add selective wildcard matching to Discord guild channel allowlists so dynamically created matching channels work without per-channel configuration updates.

## Best Possible Solution

Approve and document one Discord-owned selective pattern syntax with exact and thread-parent matches taking precedence, deterministic ordering among multiple patterns, literal `*` remaining the broad fallback, and aligned config types, validation, audit/status output, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, normalized, parent, or literal `*` match and therefore resolves denied. No live Discord run was needed to establish the missing matcher branch.

## Solution Assessment

Yes, a narrow selective pattern contract is the best fit for the stated dynamic-channel workflow, but maintainers must choose its permanent syntax and security precedence first. Category inheritance and the broader user/role access-policy redesign should remain separate work unless maintainers deliberately unify those designs.
