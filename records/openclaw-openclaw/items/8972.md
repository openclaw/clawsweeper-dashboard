## Label Justifications

- `P2`: This is a normal-priority Discord configuration enhancement with bounded scope but meaningful access-control impact.
- `impact:security`: Selective channel pattern matching changes which Discord channels are authorized to send messages into the agent workflow.
- `impact:message-loss`: Messages from dynamically created channels are currently suppressed until each channel is configured or the broad literal wildcard is enabled.

## Summary

Current main still treats Discord guild channel keys as exact id/name/slug entries, thread-parent fallbacks, or the literal `*` fallback, so selective keys such as `ws-*` remain unsupported. The request is distinct from the open category and access-list designs, but it needs an explicit maintainer product/security decision because matching precedence changes the channel authorization boundary.

## What This Changes

Add selective glob or wildcard matching to Discord guild channel allowlists so dynamically created matching channels work without config edits and gateway restarts.

## Best Possible Solution

Add an explicit Discord-owned channel-pattern contract with deterministic precedence after exact and thread-parent matches but before literal `*`, while aligning types, schema/help, diagnostics, docs, and regression coverage and leaving category inheritance and sender-role policy to their separate canonical issues.

## Reproduction Assessment

Yes. On current main, configure a non-empty Discord guild channel map containing `ws-*` and send from `ws-task-1`; source shows no exact, parent, or literal `*` match, so the channel remains denied.

## Solution Assessment

Yes in direction, but not as implicit globbing of every existing key. The narrowest maintainable solution is an explicit Discord-owned pattern form with defined precedence, compatibility behavior, diagnostics, and security-focused tests.
