## Summary

Keep open. Current main still treats Discord guild channel entries as exact id/name/slug matches, thread-parent matches, or the literal `*` fallback; glob keys like `ws-*` are not implemented, and the remaining change needs an access-control/product decision rather than cleanup closure.

## What This Changes

Add glob-style Discord guild channel allow-list entries so dynamically created matching channels can be allowed without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with schema/docs/audit updates and focused regression coverage for dynamic matching.

## Reproduction Assessment

Yes for the current limitation by source inspection: a key such as `ws-*` is stored as a normal channel entry and never compared as a glob against `ws-task-1`. I did not run a live Discord gateway repro because the matcher, docs, and tests already establish the current behavior.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow implementation in the Discord allow-list matcher is plausible, but it should be gated by product/security review rather than queued as an automatic fix.
