## Summary

Keep open: current main still resolves Discord guild channel allowlists through exact id/name/slug, thread-parent, and literal `*` matches only, so a key like `ws-*` is not evaluated as a glob. Related category, additive-override, sender-access, and hot-reload items overlap the workflow but do not implement this pattern matching contract.

## What This Changes

Request support for Discord guild channel allowlist patterns, such as `ws-*`, so dynamically created matching channels can be admitted without per-channel config edits and restarts.

## Best Possible Solution

Approve and document one Discord-owned pattern contract, then update the resolver/preflight path, exported types, Zod schema, docs/status/audit behavior, and focused tests while keeping category inheritance and hot-reload separate.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, normalized, parent, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. If approved, the narrowest maintainable path is a Discord-owned pattern tier using the current config shape and existing glob tooling, not bundling category inheritance, hot reload, or sender access-list redesign into the same change.
