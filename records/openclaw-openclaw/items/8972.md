## Label Justifications

- `P2`: This is a bounded Discord enhancement with real workflow impact and a restart/config-edit workaround, but it is not an urgent outage.
- `impact:security`: Discord channel pattern matching would decide which guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created Discord channels are currently denied unless each channel is listed exactly or a broad `*` fallback is used.

## Summary

Keep open: current main and v2026.6.8 still treat Discord guild channel entries as exact id/name/slug keys, parent fallbacks, or the literal `*` fallback, so a key such as `ws-*` is not evaluated as a glob and the requested pattern allowlist behavior remains missing.

## What This Changes

Add Discord guild channel allowlist support for glob or wildcard keys so dynamically created matching channels can be accepted without per-channel config edits.

## Best Possible Solution

Approve one documented Discord-owned pattern contract, then update the allow-list resolver, exported types, Zod schema, docs, audit/status behavior, and focused regression tests together while keeping category and hot-reload work separate.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. A narrow Discord-owned pattern contract is the right implementation boundary if that direction is approved.
