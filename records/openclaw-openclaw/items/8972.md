## Summary

Keep open. Current main still treats Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback, so a key like `ws-*` is not evaluated as a glob and this remains access-control-sensitive feature work requiring maintainer product/security review.

## What This Changes

The issue requests glob or wildcard Discord guild channel allowlist entries so dynamically created channels can be admitted without per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned types, Zod schema, docs/status/audit behavior, and focused regression tests while keeping category inheritance and broader access-list redesigns separate unless maintainers intentionally combine them.

## Reproduction Assessment

Yes, at source level. A guild channel config containing `ws-*` and an incoming channel slug `ws-task-1` misses because the matcher uses exact lookups plus parent and literal `*` fallback, and unmatched configured channels deny.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord-owned allow-list implementation is plausible, but it should follow product/security review rather than automatic cleanup.
