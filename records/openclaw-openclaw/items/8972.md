## Summary

Keep open. Current main and the latest release still treat Discord guild channel entries as exact id/name/slug candidates, thread-parent fallbacks, or the literal `*` fallback; a key like `ws-*` is not evaluated as a glob, and the remaining work is an access-control-sensitive feature decision rather than a cleanup close.

## What This Changes

The issue requests glob or wildcard matching for Discord guild channel allowlist entries so dynamically created matching channels do not require config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned channel pattern contract with explicit precedence and security semantics, then update the matcher, Discord config types/schema/docs/status or audit surfaces, and focused regression tests while keeping category inheritance and access-list redesign scoped to their canonical discussions unless maintainers intentionally combine them.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that a `ws-*` entry is treated as a literal channel key and will not match `ws-task-1` on current main or v2026.5.12.

## Solution Assessment

Unclear until maintainers choose the public syntax and access-control precedence. A narrow Discord-owned matcher extension is plausible, but it should not be automated before product and security review of pattern ordering and channel access widening.
