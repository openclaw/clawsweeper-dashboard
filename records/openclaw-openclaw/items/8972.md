## Label Justifications

- `P2`: This is a bounded Discord channel enhancement with real workflow impact and a restart/config-edit workaround, but it is not an urgent outage.
- `impact:security`: Glob channel keys would decide which Discord guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created Discord channels are currently denied unless each channel is listed exactly or a broad `*` fallback is used.

## Summary

Keep open: current `main` and the latest `v2026.6.10` release still treat Discord guild channel entries as exact id/name/slug keys, one thread-parent fallback, or the literal `*` fallback, so `ws-*` style pattern allowlists remain missing and need maintainer-approved authorization semantics.

## What This Changes

Request support for glob or wildcard Discord guild channel allowlist entries such as `ws-*` so dynamically created matching channels can be admitted without per-channel config edits.

## Best Possible Solution

Keep this as the canonical Discord glob/pattern allowlist tracker and approve one documented Discord-owned pattern contract before updating resolver behavior, exported types, Zod schema, docs, audit/status output, and focused tests.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. If approved, the narrowest maintainable solution is a Discord-owned pattern contract rather than bundling category inheritance, additive overrides, or hot reload into the same change.
