## Summary

Keep open. Current main and the latest release still treat Discord guild channel entries as exact id/name/slug matches, thread-parent fallbacks, or the literal `*` entry; glob keys like `ws-*` are not implemented and need maintainer product/security review because this widens Discord channel access.

## What This Changes

The issue requests glob or wildcard-style Discord channel allowlist entries so dynamically created channels matching names like `ws-*` do not need per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned exported types, strict schema, docs, audit/status behavior, and focused regression coverage.

## Reproduction Assessment

Yes for the current limitation by source inspection: a key such as `ws-*` is stored as an ordinary channel entry and is never compared as a glob against `ws-task-1`. I did not run a live Discord gateway repro because the matcher, docs, and tests establish the current behavior.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord allow-list implementation is plausible, but it should follow product/security review rather than an automatic cleanup fix.
