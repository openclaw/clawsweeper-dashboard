## Summary

Keep open. Current main and the latest release still only support exact Discord channel id/name/slug matches, thread-parent fallback, and the literal `*` entry; glob keys like `ws-*` remain unimplemented and need an access-control/product decision before a fix.

## What This Changes

Request support for glob or wildcard Discord guild channel allowlist entries so dynamic channels like `ws-task-1` can match `ws-*` without per-channel config edits and restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned exported types, Zod schema, docs, audit/status behavior, and focused regression coverage.

## Reproduction Assessment

Yes for the current limitation by source inspection: a key such as `ws-*` is stored as an ordinary channel entry and is never compared as a glob against `ws-task-1`. I did not run a live Discord gateway repro because the matcher, docs, and tests establish the current behavior.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord allow-list implementation is plausible, but it should follow product/security review rather than an automatic cleanup fix.
