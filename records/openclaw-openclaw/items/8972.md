## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface but no urgent production regression.
- `impact:security`: The requested wildcard matching changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until config is patched and the gateway is restarted.

## Summary

Keep open. Current main and v2026.5.19 still only match exact Discord channel id/name/slug, one parent fallback, or the literal `*`; a key such as `ws-*` is a plain record key today, and adding it is an access-control config/product decision.

## What This Changes

The issue requests glob or wildcard Discord guild channel allowlist keys so dynamically created channels such as `ws-task-*` can be allowed without config edits and gateway restarts.

## Best Possible Solution

Define a documented Discord pattern allowlist syntax and precedence model, preserve exact matches and literal `*`, update types/schema/docs/status/audit output, and add regression tests while coordinating category support through https://github.com/openclaw/openclaw/issues/60582 and access-list redesign through https://github.com/openclaw/openclaw/issues/69748.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary `channels` record key, while runtime lookup only checks exact id/slug/name candidates, one parent fallback, and the literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the reported dynamic-channel workflow, but the safest implementation needs explicit access-control precedence and coordination with the separate category and access-list redesign issues.
