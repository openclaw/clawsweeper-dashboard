## Label Justifications

- `P2`: This is a normal-priority Discord channel configuration improvement with real workflow impact but no evidence of an urgent regression.
- `impact:security`: The request changes which Discord guild channels can trigger an agent through allowlist matching.
- `impact:message-loss`: Current exact allowlists can silently drop intended messages from dynamically created Discord channels until config is edited and the gateway reloads or restarts.

## Summary

Keep open: current main and v2026.6.5 still resolve Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*`; `ws-*` is still not evaluated as a glob and the security semantics need maintainer approval.

## What This Changes

The issue requests glob or wildcard pattern entries for Discord guild channel allowlists so dynamically created matching channels can be accepted without per-channel config edits.

## Best Possible Solution

Approve and implement a documented Discord-owned pattern channel-entry contract with explicit match precedence, escaping, schema/types, docs, audit/status output, and regression coverage, while keeping category inheritance and broader access-list redesigns separate unless maintainers intentionally merge those scopes.

## Reproduction Assessment

Yes, by source inspection. Configure `guilds.<id>.channels` with `ws-*`, send from a channel whose slug is `ws-task-1`, and current resolution checks exact keys plus literal `*` before returning `{ allowed: false }` on no match.

## Solution Assessment

Unclear until maintainers choose the security semantics. A narrow Discord-owned matcher extension is plausible, but the best fix must define precedence and keep category inheritance and broader access-list redesigns separate unless maintainers intentionally combine them.
