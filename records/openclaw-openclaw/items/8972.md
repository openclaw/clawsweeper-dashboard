## Summary

Current main still lacks pattern evaluation, and no related item safely supersedes this distinct request. The feature remains useful but needs an explicit maintainer decision because it expands Discord authorization semantics and overlaps adjacent category and access-list designs.

## What This Changes

Add pattern-based Discord guild channel allowlist entries so dynamically created matching channels work without per-channel configuration changes.

## Best Possible Solution

Define a Discord-owned selector contract with explicit syntax and deterministic precedence—exact ID, name, slug, and thread-parent matches first, approved pattern matches next, and literal `*` last—then align config types and schema, public docs, status and security-audit reporting, and focused allow/deny regression tests.

## Reproduction Assessment

Yes, at source level with high confidence: current main compares configured channel keys by exact or normalized equality and only recognizes the literal `*` fallback, so `ws-*` cannot match `ws-task-1`. No live Discord run was performed because the missing evaluation branch is conclusive.

## Solution Assessment

Unclear. A Discord-owned pattern selector is a plausible bounded solution, but maintainers must first choose its syntax, precedence, name-versus-ID security posture, and relationship to category inheritance and the broader access-list proposal.

- Pattern matching expands a Discord authorization boundary; unclear precedence could admit unintended channels, while incorrect normalization could continue suppressing intended messages.

- Channel-name patterns are mutable and less stable than Discord IDs, so the security model must explicitly state whether matching names or slugs is acceptable and how collisions or renames affect access.
