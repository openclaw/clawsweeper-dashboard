## Label Justifications

- `P2`: This is a bounded, useful Discord configuration enhancement with real workflow impact but no urgent regression or outage.
- `impact:security`: Discord guild channel matching is an inbound authorization boundary, and pattern semantics determine which channels may trigger the agent.
- `impact:message-loss`: Messages from dynamically created channels are currently suppressed until an exact or literal-wildcard channel entry authorizes them.

## Summary

Current main still treats Discord guild channel entries as exact ID/name/slug keys, thread-parent fallbacks, or the literal `*`; the requested pattern matching remains a distinct, useful feature, but it needs maintainer product and security decisions because matching precedence changes an inbound authorization boundary.

## What This Changes

Add glob-style matching to Discord guild channel allowlist keys so dynamically created channels can be authorized without individual config edits and gateway restarts.

## Best Possible Solution

Define an explicit Discord-owned channel-pattern contract, retain exact and thread-parent matches above patterns and literal `*`, specify deterministic ordering for overlapping patterns, expose the selected key in diagnostics, and add focused authorization, schema, and documentation coverage.

## Reproduction Assessment

Yes, from current source with high confidence: configure a non-empty Discord guild `channels` map containing `ws-*` but no literal `*`, then send in `ws-task-1`; the key is never evaluated as a pattern and the channel resolves to denied.

## Solution Assessment

Unclear pending maintainer design approval: glob matching directly addresses dynamic channel names, but the best maintainable implementation depends on choosing public syntax, overlap ordering, and whether pattern logic belongs in Discord or a deliberately expanded shared contract.
