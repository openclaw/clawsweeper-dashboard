## Label Justifications

- `P2`: This is a bounded Discord enhancement with real workflow impact, but it is not an urgent outage or regression.
- `impact:security`: Discord channel pattern matching would decide which guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created channels are currently denied until each channel has an exact configured entry or the broad `*` fallback is used.

## Summary

Keep open: current main and v2026.6.6 still only match Discord guild channel keys by exact id/name/slug, thread parent, or literal `*`; selective patterns such as `ws-*` remain a product/security decision, and related category, additive-override, and hot-reload issues do not replace this narrower request.

## What This Changes

Request support for selective Discord guild channel allowlist patterns such as `ws-*` so dynamic channels can be admitted without editing config for each channel.

## Best Possible Solution

Approve and document one Discord-owned selective pattern contract with exact and thread-parent matches first, deterministic ordering among patterns, literal `*` remaining the broad fallback, and aligned types, schema, docs, audit/status output, and regression coverage.

## Reproduction Assessment

Yes, source-level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. No live Discord run was needed to establish the missing matcher branch.

## Solution Assessment

Yes, a narrow selective-pattern contract is the best fit for the stated dynamic-channel workflow, but maintainers must choose the permanent syntax and security precedence first. Category inheritance, additive overrides, and hot reload should remain separate unless maintainers deliberately unify those designs.
