## Summary

Keep open: current main and v2026.6.10 still lack Discord channel glob/pattern allowlist matching; related category, additive-override, sender-access, and hot-reload issues overlap the workflow but do not implement the requested `ws-*` matching contract.

## What This Changes

The issue requests glob or wildcard channel allowlist entries for Discord guild channels so dynamically named channels can be authorized without per-channel config edits and restarts.

## Best Possible Solution

Approve one documented Discord-owned pattern contract, then update the resolver, exported types, Zod schema, docs, audit/status behavior, and focused preflight/matcher tests while keeping category inheritance and hot-reload separate.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, parent, normalized, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. If approved, the narrowest maintainable path is a Discord-owned pattern contract rather than bundling category inheritance, additive overrides, sender access-list redesign, or hot-reload into the same change.
