## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface and no urgent production regression.
- `impact:security`: The request changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until configuration is patched and the gateway is restarted.

## Summary

Keep open. Current main and the latest shipped release still treat Discord guild channel entries as exact id/name/slug keys, one-level parent fallbacks, or the literal `*`; adding `ws-*` would be a new Discord access-control config feature, not an obsolete cleanup item.

## What This Changes

The issue requests glob or wildcard support for Discord guild channel allowlist entries so dynamically created matching channels can be accepted without config edits and gateway restarts.

## Best Possible Solution

Define a documented Discord-owned pattern syntax and precedence model, preserve exact and literal `*` behavior, update types/schema/docs/status/audit output, and add focused allowlist/preflight regression tests while coordinating category and access-list design with the related open issues.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary `channels` record key, while runtime lookup only checks exact id/slug/name candidates, thread-parent fallback, and the literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the safest implementation needs explicit access-control precedence and coordination with category/access-list design work.
