## Label Justifications

- `P2`: This is a bounded Discord enhancement with real workflow impact and a workaround, but it is not an urgent outage or regression.
- `impact:security`: Discord channel pattern matching would decide which guild messages cross the configured channel authorization boundary.
- `impact:message-loss`: Messages from intended dynamically created Discord channels remain denied until each channel has an exact configured entry or the broad literal `*` fallback is used.

## Summary

Keep open: current main and v2026.6.8 still treat Discord guild channel entries as exact id/name/slug matches, parent fallback, or the literal `*`, so a key like `ws-*` remains an unmatched channel entry and the requested pattern allowlist behavior is still missing.

## What This Changes

The issue requests glob or wildcard-style Discord guild channel allowlist entries so dynamically created matching channels can be accepted without per-channel config edits and restart churn.

## Best Possible Solution

Approve one documented Discord-owned channel pattern contract, then update allow-list matching, exported types/Zod schema, docs, audit/status behavior, and focused regression tests together while keeping category inheritance and hot-reload work separate unless maintainers intentionally combine them.

## Reproduction Assessment

Yes, at source level: with a nonempty Discord guild `channels` map containing `ws-*`, a channel named `ws-task-1` has no exact, normalized, parent, or literal `*` match and resolves denied. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the permanent pattern syntax and authorization precedence. A narrow Discord-owned pattern contract is the maintainable boundary if that direction is approved.
