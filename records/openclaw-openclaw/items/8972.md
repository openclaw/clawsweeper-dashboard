## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface but no urgent production regression.
- `impact:security`: The request changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until configuration is patched and the gateway is restarted.

## Summary

Keep open. Current main and v2026.5.20 still treat Discord guild channel keys as exact id/name/slug, thread-parent, or literal `*` matches; supporting `ws-*` would add public access-control semantics rather than clean up an obsolete report.

## What This Changes

Request adding glob/wildcard Discord guild channel allowlist entries so dynamic channels like `ws-task-1` can be allowed without config edits and gateway restarts.

## Best Possible Solution

Define a documented Discord-owned pattern syntax and precedence model, preserve exact and literal `*` behavior, update types/schema/docs/status/audit output, and add focused regression tests while coordinating category and access-list design with the related open issues.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary `channels` record key, while runtime lookup only checks exact id/slug/name candidates, thread-parent fallback, and the literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the safest implementation needs explicit access-control precedence and coordination with category/access-list design work.
