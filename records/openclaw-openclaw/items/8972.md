## Label Justifications

- `P2`: This is a normal-priority Discord access-control enhancement with a clear affected surface and no urgent production regression.
- `impact:security`: The request changes which Discord guild channels are authorized to trigger OpenClaw from configured allowlist entries.
- `impact:message-loss`: Exact-only allowlists can cause intended dynamically created Discord channels to be ignored until config is patched and the gateway restarts.

## Summary

Keep open. Current main and latest release still treat Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*`; glob keys like `ws-*` remain a new Discord access-control feature that needs maintainer product/security review.

## What This Changes

Request support for Discord guild channel allowlist glob or wildcard patterns so dynamically created matching channels can be authorized without per-channel config edits and restarts.

## Best Possible Solution

Define a documented Discord-owned pattern syntax and precedence model, preserve existing exact and literal `*` behavior, update types/schema/docs/status/audit output, and add focused allowlist/preflight regression coverage while keeping category and access-list work coordinated with the related issues.

## Reproduction Assessment

Yes, at source level. A guild channel key like `ws-*` is stored as an ordinary record key, while runtime lookup only checks exact id/slug/name candidates, thread-parent fallback, and the literal `*`, so `ws-task-1` remains denied unless a broader fallback is configured.

## Solution Assessment

Unclear until maintainers choose the public config contract. Glob support is coherent for the dynamic-channel workflow, but the maintainable solution needs explicit access-control precedence and coordination with category/access-list design work.
