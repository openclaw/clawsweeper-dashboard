## Label Justifications

- `P2`: This is a valid Discord access-control improvement with limited blast radius, but it needs maintainer design review before implementation.
- `impact:security`: Glob entries could authorize many Discord guild channels that exact allowlists currently deny by omission.
- `impact:message-loss`: Today a dynamically created channel like `ws-task-1` is silently dropped when only `ws-*` is configured.

## Summary

Keep open. Current main still treats Discord guild channel keys as exact id/name/slug matches, thread-parent fallbacks, or the literal `*` fallback, so `ws-*` is not evaluated as a glob and this remains security-sensitive feature work.

## What This Changes

Add glob or wildcard pattern support to Discord guild channel allowlists so dynamically created matching channels do not require per-channel config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned types, Zod schema, docs/status/audit behavior, and focused regression tests.

## Reproduction Assessment

Yes, at source level. A guild channel config containing `ws-*` and an incoming channel slug `ws-task-1` misses because the matcher uses exact lookups plus parent and literal `*` fallback, then unmatched configured channels deny.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord-owned allow-list implementation is plausible, but it should follow product/security review rather than automatic cleanup.
