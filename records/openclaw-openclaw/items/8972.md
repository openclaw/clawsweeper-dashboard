## Summary

Keep open. Current main still treats Discord guild channel entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback; glob/category semantics are not defined in code, schema, or docs, so this is a real feature request that needs maintainer design rather than cleanup closure.

## What This Changes

Request glob or wildcard Discord guild channel allowlist keys so newly created matching channels can be accepted without config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned, documented pattern syntax and precedence model, update the config contract/docs/status diagnostics, and add regression coverage proving matching dynamic channels are allowed without per-channel config edits; keep category inheritance scoped to #60582 unless maintainers merge the designs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main would treat `ws-*` as a literal key and deny `ws-task-1` unless an exact, parent, or literal `*` entry matches.

## Solution Assessment

Unclear on the exact syntax until maintainers decide the access-control precedence. A narrow Discord allow-list implementation is plausible, but it should be designed with schema/docs/tests rather than closed as already implemented or moved out of the repo.
