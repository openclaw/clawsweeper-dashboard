## Summary

Keep open. Current main and the latest release still treat Discord guild channel entries as exact id/name/slug matches, parent fallbacks, or the literal `*` entry; glob keys such as `ws-*` are not implemented, and the related category/additive-access issues do not supersede this narrower request.

## What This Changes

Add Discord guild channel allowlist pattern/glob support so dynamically created channels matching entries such as `ws-*` can be accepted without config edits and gateway restarts.

## Best Possible Solution

Define a Discord-owned pattern syntax and precedence model, then implement it in the existing allow-list path with aligned exported types, strict schema, docs/status/audit behavior, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A guild channel config containing a key like `ws-*` and an incoming channel slug `ws-task-1` will miss because the matcher uses exact lookups plus parent and literal `*` fallback; current tests also show unmatched configured channels deny.

## Solution Assessment

Unclear until maintainers choose the public pattern syntax and access-control precedence. A narrow Discord allow-list implementation is plausible, but it should follow product/security review rather than automatic cleanup.
