## Summary

Keep open. The PR has useful live proof for direct inline-eval materialization, but the gateway-approved stored-plan path still loses the approved venv interpreter path and crosses an exec approval boundary.

## What This Changes

The PR materializes exact direct Python and Node inline eval node-host commands into deterministic private scripts before `system.run` approval and adds approval/dispatch coverage.

## Best Possible Solution

Keep the materialization approach, but carry enough inline-eval plan context through stored `systemRunPlan` dispatch so direct node calls and gateway-approved replays preserve the approved interpreter path and mutable script binding.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: build a materialized venv Python plan, then invoke `system.run` with `command: prepared.plan.argv`, `rawCommand: prepared.plan.commandText`, and `systemRunPlan: prepared.plan` to mirror gateway replay. I did not run a local test because this review is read-only.

## Solution Assessment

No, not yet. The materialization design is reasonable, but the current dispatch condition and test cover the direct inline command path instead of the gateway-forwarded stored-plan path that users actually hit after approval.
