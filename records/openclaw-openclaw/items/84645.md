## Summary

Keep open: this is focused and well-proven, but it intentionally changes a node-host exec approval security boundary and still needs maintainer acceptance plus docs alignment before merge.

## What This Changes

The PR materializes exact direct Python/Node inline eval commands into private deterministic scripts during node-host `system.run.prepare` and reuses the approved rewritten argv during `system.run` dispatch.

## Best Possible Solution

Land only after the exec-approval docs describe supported materialized forms, unsupported fail-closed forms, and generated-script artifact behavior, with explicit maintainer acceptance of the changed approval boundary.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.10 shows direct interpreter/runtime commands without one stable file operand still fail approval planning with `SYSTEM_RUN_DENIED`, while the PR body supplies live gateway output for the new materialized Python path and a fail-closed Node-sensitive case.

## Solution Assessment

Mostly, but not merge-ready. Materializing the snippet to a private file and binding that file fits the existing mutable-file approval contract, but the docs and maintainer policy acceptance need to catch up before this is the best shipped solution.
