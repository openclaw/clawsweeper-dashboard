## Summary

Keep this PR open: current main and the latest release still deny the direct inline-eval approval path, so the work is not obsolete, but the branch is not merge-ready because it changes a security-sensitive approval boundary and leaves operator docs stale.

## What This Changes

The PR materializes exact direct Python and Node inline eval commands into deterministic private script files before node-host approval planning, binds the generated script through mutableFileOperand, and reuses the approved rewritten argv during dispatch.

## Best Possible Solution

Land this only after the exec-approval docs describe the supported materialized forms, unsupported fail-closed forms, and generated-script artifact behavior, with explicit maintainer acceptance of the changed approval boundary.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.9 shows direct interpreter/runtime commands without one stable file operand still fail approval planning with SYSTEM_RUN_DENIED, while the PR body supplies live gateway output for the new materialized Python path and a fail-closed Node-sensitive case.

## Solution Assessment

Mostly, but not merge-ready. Materializing the snippet to a private file and binding that file fits the existing mutable-file approval contract, but the docs and maintainer policy acceptance need to catch up before this is the best shipped solution.
