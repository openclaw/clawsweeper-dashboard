## Summary

Keep this PR open: current main still denies the direct inline-eval approval path, while the PR contains a coherent implementation, focused tests, and sufficient live gateway proof. The remaining concern is maintainer acceptance of a security-boundary change that writes generated scripts under the state directory before approved execution.

## What This Changes

The PR materializes exact direct Python and Node inline eval argv into private generated scripts for node-host `system.run` approval, reuses the rewritten plan during dispatch, and adds plan/dispatch tests.

## Best Possible Solution

Land a maintainer-reviewed version that keeps ambiguous inline eval fail-closed, binds generated scripts with `mutableFileOperand`, and has an accepted lifecycle policy for state-dir materialized scripts.

## Reproduction Assessment

I did not run local tests because this review is read-only, but the PR body includes live gateway before/after-style output for the changed behavior.

## Solution Assessment

Yes, conditionally: materializing exact direct forms before approval binding is a maintainable way to avoid allowing raw inline eval. The maintainer decision is whether the generated state-dir script lifecycle is the right security boundary.
