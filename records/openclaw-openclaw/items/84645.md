## Summary

## What This Changes

The PR rewrites exact direct Python and Node inline eval commands into deterministic private script files during node-host approval planning and reuses the materialized plan during approved execution.

## Best Possible Solution

Keep the materialization design, but carry the original inline-eval identity through stored-plan replay and cover strictInlineEval allow-always non-persistence for both direct and gateway-forwarded executions.

## Reproduction Assessment

Yes, from source inspection. Gateway replay forwards `systemRunPlan.argv` and `systemRunPlan.commandText`, which misses the PR's `approvalPlan.commandPreview === command.commandText` detection and leaves `materializedInlineEvalHit` null.

## Solution Assessment

No, not yet. The materialization approach is maintainable, but the current replay detection must preserve the original inline-eval identity before it is safe to merge.
