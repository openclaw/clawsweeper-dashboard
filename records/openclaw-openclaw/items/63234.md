## Summary

Keep open. Current main still has generic plugin-loader and tool-catalog paths that can validate a required plugin schema against missing `entry.config` as `{}` and emit the reported `[plugins] <id> invalid config` log shape; the recent cold-registry fix covers path-based tool loading, but not a required-config preservation regression for the valid-config case described here.

## What This Changes

The issue asks OpenClaw to stop repeated invalid-config startup logs for a valid required plugin config during gateway startup and tool registration.

## Best Possible Solution

Land a generic loader/tool-catalog regression fix that preserves `plugins.entries.<id>.config` for required native tool plugins across startup, standalone cold loads, `tools.catalog`, and post-ready tool resolution without adding UnraidClaw-specific core logic.

## Reproduction Assessment

Yes for a source-level generic path: a native tool plugin fixture with required `serverUrl` and `apiKey` config can exercise startup, cold registry load, `tools.catalog`, and `resolvePluginTools` and assert valid config is never observed as `{}`. The exact UnraidClaw environment is not directly reproducible from this checkout because that plugin is absent.

## Solution Assessment

Yes, the best direction is a narrow generic loader/catalog fix with regression coverage. That keeps ownership at the plugin runtime boundary and avoids downstream plugin-specific behavior in core.
