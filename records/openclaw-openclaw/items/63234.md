## Summary

Keep open. Current main and the latest release still have loader paths that validate missing plugin config as `{}` and emit the reported `invalid config` log shape, while the related fix PR remains open and has not proven the broader startup/catalog path for this canonical issue.

## What This Changes

The issue asks OpenClaw to stop repeated false `invalid config` startup logs for a working plugin whose config is present under `plugins.entries.<id>.config`.

## Best Possible Solution

Land or adapt a generic plugin-loader/config propagation fix that preserves valid wrapped plugin config across startup, snapshot, CLI metadata, tool catalog, and activation paths, with strict activation validation retained and regression coverage for false log spam.

## Reproduction Assessment

Yes for the generic source path: a strict-schema plugin whose `entry.config` is omitted in a non-activating loader or CLI metadata path is validated as `{}` and can emit the reported required-field errors. I did not live-reproduce the exact UnraidClaw Docker setup because that plugin is not in the checkout.

## Solution Assessment

Yes: the maintainable fix belongs in the generic loader/config propagation path, not in an UnraidClaw-specific workaround. The open related PR is a plausible starting point, but this issue should remain open until that fix or an equivalent one lands with startup/catalog coverage.
