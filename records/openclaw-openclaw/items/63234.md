## Summary

Keep open. Current main still has the generic plugin-loader paths that can validate an omitted plugin config as `{}` and emit the same repeated `invalid config` messages, and the related open PR is not merged or explicitly closing this canonical issue yet.

## What This Changes

The issue asks OpenClaw to stop repeated invalid-config logs for a valid UnraidClaw `plugins.entries.unraidclaw.config` during gateway startup.

## Best Possible Solution

Review, land, or adapt the open loader fix so valid `plugins.entries.<id>.config` is preserved across startup, metadata, catalog, and tool-discovery paths, with regression coverage that prevents valid config from being revalidated as `{}`.

## Reproduction Assessment

Yes for the generic source path: current main validates undefined plugin config as `{}` and has multiple invalid-config log sites reachable from non-activating loader/catalog phases. No live UnraidClaw repro was run because that plugin is not present in the checkout.

## Solution Assessment

Yes, the maintainable fix is a generic loader/tool-discovery validation guard or config propagation repair, not an UnraidClaw-specific special case. The open related PR is a plausible starting point but should be checked against this issue's startup/catalog behavior before closing.
