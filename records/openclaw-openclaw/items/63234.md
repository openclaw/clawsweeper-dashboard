## Summary

Keep open. Current main and the latest release still have the generic plugin-loader path that validates omitted plugin config as `{}` and emits the reported `invalid config` message shape, while the related fix PR remains open and still needs maintainer review before this issue can be considered resolved.

## What This Changes

The issue asks for valid `plugins.entries.<id>.config` values to stop producing repeated false `invalid config` startup/preflight log lines for working plugins.

## Best Possible Solution

Review, land, or replace the generic loader/config propagation fix so non-activating startup, snapshot, CLI metadata, and catalog paths do not warn on omitted config while real activation still validates strict plugin schemas.

## Reproduction Assessment

Yes for the generic source path: a strict-schema plugin whose `entry.config` is omitted in a non-activating loader path is validated as `{}` and can emit missing-required-field errors. The exact UnraidClaw Docker setup was not live-reproduced because that plugin is not in current main.

## Solution Assessment

Yes: the best fix belongs in the generic plugin loader/config propagation boundary, not in an UnraidClaw-specific workaround. The open related PR is a plausible path, but it still needs maintainer review for startup/catalog coverage and validation-boundary safety.
