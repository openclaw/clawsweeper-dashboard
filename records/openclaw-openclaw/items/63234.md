## Summary

Keep open. Current main still has generic plugin loader paths that validate `entry?.config`, coerce a missing value to `{}`, and emit the reported `[plugins] <id> invalid config` shape for required schemas. The adjacent shipped work covers plugin-local invalid-config degradation, but I did not find coverage proving a valid required plugin config is preserved across startup, registry reuse, `tools.catalog`, and later tool resolution; the exact UnraidClaw plugin is also not present in this checkout.

## What This Changes

The issue requests eliminating repeated UnraidClaw plugin invalid-config startup logs when valid `plugins.entries.<id>.config` eventually registers successfully.

## Best Possible Solution

Add a narrow generic regression fix in the plugin loader/tool-catalog path so resolved `plugins.entries.<id>.config` for required native tool plugins remains available across gateway startup, active registry reuse, `tools.catalog`, and post-ready tool resolution, without hard-coding an UnraidClaw id or changing plugin author contracts unnecessarily.

## Reproduction Assessment

Yes for a high-confidence generic reproduction path: create a fixture native tool plugin with required `serverUrl` and `apiKey`, load it with valid `plugins.entries.<id>.config`, then exercise gateway startup, registry reuse, `tools.catalog`, and post-ready tool resolution while asserting no `{}` validation or repeated invalid-config logs. The exact UnraidClaw reproduction is unclear because that plugin is not present in current main.

## Solution Assessment

Yes, a generic loader/catalog regression fix is the best direction because the observable failure is config propagation across core plugin phases, not Unraid-specific behavior. A hard-coded UnraidClaw workaround would violate the plugin boundary and leave other required-config plugins exposed.
