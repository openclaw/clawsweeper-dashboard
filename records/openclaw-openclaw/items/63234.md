## Summary

Keep open. Current main and v2026.5.18 still have the generic loader paths that can validate omitted plugin config as `{}` and emit the reported `invalid config` log shape, while the related fix PR remains open and still needs maintainer review for the validation boundary.

## What This Changes

The issue requests that valid plugin config under `plugins.entries.<id>.config` stop producing repeated missing-required-field `invalid config` startup logs for plugins that ultimately load successfully.

## Best Possible Solution

Review, land, or replace the generic loader/config propagation fix so valid wrapped plugin config is preserved through startup, snapshot/metadata, catalog, and activation paths while strict activation validation remains intact.

## Reproduction Assessment

Yes for the generic source path: a strict-schema plugin whose `entry.config` is omitted in a non-activating loader or CLI metadata path is validated as `{}` and can emit the reported required-field errors. I did not live-reproduce the exact UnraidClaw Docker setup because that plugin is not in the checkout.

## Solution Assessment

Yes: the maintainable fix belongs in the generic plugin loader/config propagation path, not in an UnraidClaw-specific workaround. The open related PR is a plausible starting point, but this issue should remain open until that fix or an equivalent one lands with startup/catalog coverage.
