## Summary

Keep open. Current main and the latest release still contain the generic loader paths that can validate an omitted plugin config as `{}` and emit the reported `invalid config` log shape; the closely related fix PR remains open and unmerged, so this issue is still the canonical tracker for the valid-config log-spam problem.

## What This Changes

The issue reports repeated false `invalid config` logs for a working UnraidClaw plugin whose valid values are under `plugins.entries.unraidclaw.config`.

## Best Possible Solution

Land or adapt a generic plugin-loader/config propagation fix that preserves valid `plugins.entries.<id>.config` across startup, snapshot, CLI metadata, tool catalog, and activation paths while keeping real activation validation strict.

## Reproduction Assessment

Yes for the generic source path: a strict-schema plugin whose `entry.config` is omitted in a non-activating loader or CLI metadata path is still validated as `{}` and can emit the reported required-field errors. I did not live-reproduce the exact UnraidClaw Docker setup because that plugin is not in this checkout.

## Solution Assessment

Yes: the maintainable fix belongs in the generic loader/config propagation path, not in an UnraidClaw-specific workaround. The open related PR is a plausible starting point, but this issue should remain open until that fix or an equivalent one lands with startup/catalog coverage.
