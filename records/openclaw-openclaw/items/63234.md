## Summary

Keep open. Current main still has the generic loader paths that turn missing plugin config into the reported missing-required-field `invalid config` log shape, while UnraidClaw itself is absent from the tree and there is no regression proving valid required plugin config survives startup, registry reuse, and `tools.catalog` phases.

## What This Changes

The issue asks OpenClaw to stop repeated startup `invalid config` logs for a valid UnraidClaw plugin config containing `serverUrl`, `apiKey`, and `tlsSkipVerify`.

## Best Possible Solution

Add a generic regression fix that preserves resolved `plugins.entries.<id>.config` for required native tool plugins across startup, registry reuse, `tools.catalog`, and post-ready tool resolution, with no UnraidClaw-specific core branch.

## Reproduction Assessment

Yes for a high-confidence generic reproduction path: use a fixture native tool plugin with required `serverUrl` and `apiKey`, load it with valid `plugins.entries.<id>.config`, then exercise startup/registry reuse/`tools.catalog` and assert no `{}` validation or repeated invalid-config logs. The exact UnraidClaw reproduction is not available because that plugin is absent from current main.

## Solution Assessment

Yes, a generic loader/catalog regression fix is the best direction. It matches the plugin boundary, avoids hard-coding UnraidClaw in core, and can be validated with focused synthetic coverage.
