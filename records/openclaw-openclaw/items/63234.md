## Summary

Keep open. Current main and the latest release still have the generic plugin-loader path that validates omitted plugin config as `{}` and emits the reported `invalid config` message shape; https://github.com/openclaw/openclaw/pull/70811 is the likely repair path, but it remains open.

## What This Changes

The issue asks OpenClaw to stop valid UnraidClaw plugin config from producing repeated startup `invalid config` logs when early plugin-load phases do not see resolved config.

## Best Possible Solution

Land or replace the generic plugin-loader fix so non-activating loads defer validation only when config is not materialized, while real activation and explicit config still validate strict schemas.

## Reproduction Assessment

Yes for the generic source path: a strict-schema plugin whose `entry.config` is omitted in a non-activating loader path is validated as `{}` and can emit missing-required-field errors. The exact UnraidClaw Docker setup was not live-reproduced because that plugin is not in current main.

## Solution Assessment

Yes: the best fix belongs in the generic plugin loader/config propagation boundary, not in an UnraidClaw-specific workaround. The open related PR is a plausible implementation path but still needs maintainer review before this issue can close.
