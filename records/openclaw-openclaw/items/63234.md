## Summary

Keep open. Current main still has generic plugin-loader paths that validate `entry?.config`, turn a missing value into `{}`, and emit `[plugins] <id> invalid config` from repeated runtime/metadata loads; the exact UnraidClaw plugin is not in this checkout and there is still no regression proving valid required plugin config survives startup/catalog reuse. A referenced fix attempt exists but is divergent from current main, so it does not close this issue.

## What This Changes

The issue asks OpenClaw to stop UnraidClaw from repeatedly logging missing required config during startup when `plugins.entries.unraidclaw.config` already contains the required values.

## Best Possible Solution

Land a narrow generic regression fix that preserves resolved `plugins.entries.<id>.config` for required native tool plugins across startup, active registry reuse, `tools.catalog`, and post-ready tool resolution, with coverage for both valid config preservation and log spam prevention.

## Reproduction Assessment

No, not for the exact UnraidClaw environment because that plugin is absent from current main. We do have a clear source-reproducible synthetic path: create a native tool plugin with required `serverUrl` and `apiKey`, load it with valid `plugins.entries.<id>.config`, then exercise startup/registry reuse/`tools.catalog` and assert no `{}` validation or repeated invalid-config logs.

## Solution Assessment

Yes, a narrow generic loader/catalog fix is the best direction. It matches the plugin boundary and avoids hard-coding UnraidClaw; the prior dedupe-only commit is not on current main and should not replace proof that valid config survives the implicated phases.
