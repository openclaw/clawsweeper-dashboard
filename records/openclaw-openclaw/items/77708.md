## Summary

Keep open: the PR now includes credible live Voice Call proof, but source review still finds a route-effective config mismatch and a core/plugin boundary blocker before merge.

## What This Changes

The branch adds opt-in Voice Call realtime agent-context and consult-policy config, provider instruction injection, Google realtime bridge stabilization, capability-provider loading changes, docs, changelog, and tests.

## Best Possible Solution

Build realtime instructions from the per-call effective Voice Call config, and express realtime provider activation through a generic manifest/plugin-owned contract before merging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a per-number route with a different `agentId`, enable `realtime.agentContext`, and start an inbound realtime call; provider instructions are built from the global config while consults use the route-effective config.

## Solution Assessment

No. The direction is useful, but the current implementation should derive instructions after route resolution and keep realtime capability activation generic instead of hard-coding bundled plugin config in core.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Build realtime context from the effective call route:** `extensions/voice-call/src/runtime.ts:322-328`
  - body: This builds provider instructions once from the top-level Voice Call config during runtime startup. Per-number routing stores a matched route on the call record, and realtime consults already resolve that route-effective config, so a routed line with its own `agentId` can inject the global persona into the realtime provider while consults use the routed agent. Build instructions after route resolution or pass a per-call instruction resolver into `RealtimeCallHandler`.
  - confidence: 0.88
- **[P2] Keep realtime capability loading generic:** `src/plugins/capability-provider-runtime.ts:347-354`
  - body: This makes core inspect `plugins.entries.voice-call` and `plugins.entries.google-meet` to decide which realtime providers to load. That only covers bundled plugin configs and leaves external plugins with the same capability outside this activation model when another provider is already active. Move this through manifest metadata or a generic plugin-owned requested-capability contract.
  - confidence: 0.8
