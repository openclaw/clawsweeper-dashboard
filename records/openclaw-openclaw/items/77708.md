## Summary

Keep open: the PR still has a route-aware correctness bug in the new realtime instruction construction, and the contributor has not provided after-fix real voice-call proof. The protected `maintainer` label also means this needs explicit maintainer handling before merge.

## What This Changes

The PR adds opt-in Voice Call realtime agent-context and consult-policy config/docs/tests, updates plugin metadata and changelog baselines, and also changes Google realtime reconnect/audio behavior plus capability-provider loading.

## Best Possible Solution

Make realtime instruction construction route-aware after the call record is registered, keep the docs/config metadata aligned, and require a real redacted voice-call log or terminal proof before merge.

## Reproduction Assessment

Yes, from source inspection: configure `numbers` with a route-specific `agentId`, enable `realtime.agentContext`, and start an inbound realtime call to that route. The PR builds provider instructions from the global config while the existing consult path uses the route-effective config.

## Solution Assessment

No. The narrower maintainable fix is to derive realtime instructions after route resolution, or pass a route-aware instruction resolver into `RealtimeCallHandler`, then cover the mismatch with a regression test.
