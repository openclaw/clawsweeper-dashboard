## Summary

Keep open: this maintainer-labeled feature PR still has P2 correctness issues around route-specific realtime instructions and forced consult-policy behavior. The earlier generic realtime provider-loading concern appears addressed by the updated direct capability-provider resolution.

## What This Changes

The PR adds opt-in Voice Call realtime agent-context and consult-policy config, Google Meet/Voice Call realtime routing and audio stabilization, generic realtime provider loading, docs, changelog, and tests.

## Best Possible Solution

Derive realtime provider instructions from the per-call effective Voice Call config, then align `consultPolicy` implementation and docs so light turns are not forced through the full agent unless that is an explicit product choice.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a per-number route with a different `agentId`, enable `realtime.agentContext`, and start a routed realtime inbound call; provider instructions are built from the global config while consults resolve the route-effective config.

## Solution Assessment

No. The feature direction is useful, but the current patch should build instructions after route resolution and either keep consult policy as model guidance or explicitly implement a product-approved forced-consult mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Build realtime context from the effective call route:** `extensions/voice-call/src/runtime.ts:322-327`
  - body: This builds provider instructions once from the top-level Voice Call config during runtime startup. Per-number routing can override `agentId`, and realtime consults already resolve that route-effective config, so a routed line can inject the global persona into the realtime provider while consults use the routed agent. Build instructions after route resolution or pass a per-call instruction resolver into `RealtimeCallHandler`.
  - confidence: 0.9
- **[P2] Do not force consults for every final transcript:** `extensions/voice-call/src/webhook/realtime-handler.ts:569-580`
  - body: The docs and prompt say `consultPolicy` only guides the realtime model and that `always` consults before substantive answers, but this schedules a forced full-agent consult for every non-empty final user transcript when the provider does not call the tool. A simple greeting can trigger a second forced consult response; gate this fallback to substantive/tool-needed turns or make the forced behavior an explicit documented mode.
  - confidence: 0.82
