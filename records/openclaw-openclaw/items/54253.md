## Summary

Keep open. Current main still routes OpenRouter/OpenAI-compatible model turns through the shared guarded fetch path and Undici-backed runtime fetch; I did not find a current-main RISC-V64 fallback, compatibility diagnostic, or live proof that the original transport failure is fixed. The service-proxy workaround in the comments is useful, but it is a separate symptom surface from the RISC-V64 reports where direct HTTP worked.

## What This Changes

The issue reports that OpenClaw chat turns fail with `LLM request failed: network connection error` on Linux RISC-V64 after npm install, OpenRouter onboarding, and TUI use, while the same API key works on x86 systems.

## Best Possible Solution

Keep this as the canonical RISC-V64 provider transport tracker until a maintainer or reporter verifies current main on Linux RISC-V64. If Undici/fetch remains the failure boundary while Node `https` succeeds, add either a narrowly guarded provider transport fallback or a clear compatibility diagnostic in the shared provider request path, while leaving service proxy cleanup as a separate environment issue.

## Reproduction Assessment

Unclear. The issue and related #58150 provide concrete RISC-V64 setup details plus a useful direct-HTTP contrast, but this review had no RISC-V64 host to verify current-main OpenRouter/OpenAI-compatible provider calls or separate the Undici transport failure from proxy-service configuration issues.

## Solution Assessment

No. Closing or changing shared provider networking based only on the proxy workaround would be premature; the maintainable path is to first confirm the RISC-V64 transport boundary on current main, then make the smallest provider-transport fix or diagnostic that preserves SSRF guard and proxy behavior.
