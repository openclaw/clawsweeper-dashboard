## Summary

Keep this open: the report matches a real current cron/embedded-runner timeout path, and current docs promise cron-owned timeout handling rather than a raw `CommandLaneTaskTimeoutError` from the `cron-nested` lane.

## What This Changes

Fix isolated cron agent-turn runs that stall until the `cron-nested` command lane watchdog fires, so they either complete or fail through documented cron timeout handling.

## Best Possible Solution

Cron should own this timeout outcome: preserve the `cron-nested` lane release behavior, but convert or avoid the raw lane watchdog failure so run history, task status, and failure delivery use the documented cron timeout path.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The source path is clear: a 300000 ms resolved embedded agent timeout becomes a 330000 ms lane watchdog, and the command queue emits the exact `CommandLaneTaskTimeoutError` seen in the logs.

## Solution Assessment

No, the current behavior is not the best way to solve the issue. The lane watchdog is useful to release stuck queue slots, but cron should surface its own documented timeout/failure result rather than leaking an internal lane error to the whole run.
