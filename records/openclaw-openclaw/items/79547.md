## Summary

Keep this PR open: it targets a valid memory-core bug and includes useful real behavior proof, but the patch adds a text-only cross-message skip that can drop a legitimate assistant reply after a user-typed marker prompt.

## What This Changes

This PR filters runtime-only event marker turns and their immediate assistant cron-delivery replies from memory-host session exports, extends memory-core/session-reader coverage, and adds a changelog entry for the related bug report.

## Best Possible Solution

Keep the fix in memory-host session-corpus ingestion, but make runtime-only turn identification non-spoofable and cover both generated-runtime and user-typed-marker cases.

## Reproduction Assessment

Yes, at source level: current main emits runtime-only turns with the marker prompt and buildSessionEntry has no corresponding runtime-event filter before session-corpus export. I did not run a local failing repro because this review is read-only.

## Solution Assessment

No. Filtering the generated turn is the right boundary, but this implementation identifies the pair solely by user-visible text; the safer fix is to carry or consume runtime-only provenance or another non-user-spoofable transcript marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84
