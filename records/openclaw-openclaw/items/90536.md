## Summary

Keep open: current main and v2026.6.9 still request only the base OpenAI ChatGPT OAuth scopes, while upstream Codex and the reporter's live smoke point to connector scopes as the accepted model-access path.

## What This Changes

The issue asks OpenClaw to update OpenAI ChatGPT OAuth scopes so `openai/gpt-5.5` works through ChatGPT/Codex OAuth instead of failing with missing scope and falling back.

## Best Possible Solution

Live-validate the connector scope set, update the OpenAI plugin authorize and preflight scope behavior, and add focused tests plus an after-login `openai/gpt-5.5` smoke proving no fallback.

## Reproduction Assessment

Yes from source and reporter proof: current main builds the OAuth authorize URL without the connector scopes while the reporter showed those scopes restore `openai/gpt-5.5`. I did not run a live OAuth login in this read-only review.

## Solution Assessment

No for the literal `model.request` request. The best fix is a plugin-owned connector-scope update matching upstream Codex after maintainer-approved live validation.
