## Summary

Keep open: current main and v2026.6.8 still build the OpenAI browser OAuth flow with only identity/offline scopes, while the reporter's follow-up and upstream Codex source point to connector scopes as the likely model-access consent path. The exact OAuth scope change should be maintainer-approved and live-tested because prior history shows a literal model.request addition can break the browser login itself.

## What This Changes

The issue asks OpenClaw to request the OpenAI ChatGPT OAuth scopes required for openai/gpt-5.5 model access and avoid falling through after missing-scope 401s.

## Best Possible Solution

Maintainers should live-validate the approved OpenAI ChatGPT OAuth consent scope, update the OpenAI plugin's browser authorize and preflight scope behavior if confirmed, and add focused scope plus missing-scope user-visible fallback coverage.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection confirms current main and v2026.6.8 still omit the connector scopes that the reporter validated locally and upstream Codex currently requests.

## Solution Assessment

Unclear as an implementation approval question: adding model.request literally is not the best fix because prior evidence shows that can break OAuth authorization. The safer path is maintainer-approved connector-scope validation, then a small OpenAI plugin change with tests.
