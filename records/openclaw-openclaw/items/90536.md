## Summary

Keep open: current main and v2026.6.8 still build the OpenAI ChatGPT OAuth authorize flow without the connector scopes that upstream Codex currently uses and that the reporter validated locally. No open or merged PR found in the related search owns this remaining fix, and the literal `model.request` wording is likely not the safe repair because the reporter and prior merged scope work show that scope can break browser OAuth authorization.

## What This Changes

The issue requests fixing OpenAI ChatGPT OAuth consent scopes so GPT-5.5 model calls do not fail with missing-scope errors and fall back to another configured model.

## Best Possible Solution

Maintainers should live-validate the accepted OpenAI ChatGPT OAuth connector-scope set, update the OpenAI plugin's authorize and preflight scope behavior, and add focused tests plus user-visible fallback/auth-failure coverage.

## Reproduction Assessment

Yes from source for the reported scope mismatch: current main constructs the OAuth authorize URL from a scope constant that lacks the connector scopes used by upstream Codex and validated by the reporter. No live OAuth run was performed in this read-only review, so the exact live failure was not reproduced here.

## Solution Assessment

No for the literal `model.request` proposal; the better fix is a plugin-owned connector-scope update after maintainer-approved live validation. The OpenAI plugin OAuth flow is the right owner boundary, but the final API scope contract is product/API-sensitive.
