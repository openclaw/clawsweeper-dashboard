## Summary

Keep open: current main and the latest release still generate the OpenAI ChatGPT OAuth browser flow with only identity/offline scopes, while the reporter's follow-up and upstream Codex source point to connector scopes as the accepted model-access consent path. The older merged scope PR is relevant context because it shows `model.request` was rejected as a forced browser scope, but it does not solve this current OpenAI provider auth gap.

## What This Changes

The issue requests fixing OpenAI ChatGPT OAuth scopes so `openai/gpt-5.5` can run through the selected OAuth profile instead of hitting missing-scope 401s and falling back silently.

## Best Possible Solution

Maintainers should live-validate the current OpenAI ChatGPT OAuth consent contract, update the OpenAI plugin's browser scope to the approved connector-scope shape if confirmed, and add focused tests for authorize URL generation plus missing-scope user-visible failure handling.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this review. Source inspection confirms current main and v2026.6.6 omit the connector scopes that the reporter validated and upstream Codex now requests, so the issue is source-reproducible with medium confidence.

## Solution Assessment

Unclear as an implementation approval question. Updating the scope string is a small code change, but the exact OAuth consent surface should be maintainer-approved and live-tested because prior history shows `model.request` alone can break browser authorization.
