## Summary

Keep open for maintainer judgment. Current main matches documented default `steer` behavior that can batch queued Feishu messages into the active run, but the report asks whether that default is wrong for direct messages rather than pointing to a narrow source-level bug.

## What This Changes

The issue asks for rapid Feishu messages queued behind a busy session to receive independent replies instead of being folded into the active run's reply context.

## Best Possible Solution

Maintainers should decide whether Feishu/direct-message bursts should keep documented `steer` behavior with guidance to use `/queue followup` or global `messages.queue.mode`, or intentionally change the default/per-channel policy with matching schema, docs, and regression coverage.

## Reproduction Assessment

Yes for the documented merge path: current source defaults busy sessions to `steer`, sends queued text into the active run, and docs say `steer` drains all pending messages together. I did not establish a live Feishu-only reproduction of a separate transcript bleed bug.

## Solution Assessment

Unclear. A code change is not obviously the best fix because current docs define the observed batching as expected `steer` behavior and `followup` already provides later independent turns; changing defaults needs maintainer product judgment.
