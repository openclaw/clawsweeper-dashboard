## Label Justifications

- `P2`: The issue blocks real Feishu interactive-card workflows but is scoped to one channel plugin rather than core runtime availability.
- `impact:message-loss`: Interactive card text, button labels, selections, and callback payloads can fail to reach the agent as usable inbound message content.
- `impact:data-loss`: The requested form and selector values are user-provided data that current callback parsing can discard before agent dispatch.

## Summary

Keep open: current main and the latest release still do not provide a complete Feishu interactive-card parsing path for direct inbound card messages or card-action sibling fields, and several overlapping Feishu PRs/issues need a canonical maintainer decision instead of another cleanup close.

## What This Changes

Add Feishu interactive-card parsing so inbound cards, readback/quoted cards, buttons, dropdowns, and form values reach the agent as usable content.

## Best Possible Solution

Choose one canonical Feishu-plugin implementation that handles direct inbound cards, readback/quoted and merge-forward card variants, and card action/form callbacks through a documented agent-visible payload contract with focused tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A direct inbound Feishu `message_type: "interactive"` reaches `parseMessageContent` without an interactive branch, and card-action callbacks drop sibling fields before fallback dispatch.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but maintainers should consolidate the overlapping parser/readback/action PRs and decide the callback payload contract before merge.
