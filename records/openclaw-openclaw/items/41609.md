## Summary

Keep open: current main and v2026.5.22 still leave direct Feishu inbound interactive-card parsing incomplete, and the card-action path still has unresolved form/select callback payload work. Several open related PRs cover pieces of the fix, but none has merged as the canonical solution.

## What This Changes

Request parsing and display of Feishu interactive-card message content, including card text, button labels, dropdown selections, and form values.

## Best Possible Solution

Choose one canonical Feishu plugin implementation that handles direct inbound cards, readback/quoted and merge-forward card variants, and card action/form callbacks through a documented agent-visible payload contract with focused tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A direct Feishu `message_type: "interactive"` reaches `parseMessageContent()` without an interactive branch, and card-action callbacks normalize only `action.value` plus `tag`.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but maintainers should consolidate the overlapping parser/readback/action PRs and decide the callback payload contract before merge.
