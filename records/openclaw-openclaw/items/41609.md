## Summary

Keep open. Current main and the latest release still route direct inbound Feishu interactive-card events through a parser with no interactive-card branch; merged work has improved fetched/read-message card parsing, but it does not solve the central inbound event path or the callback-field scope.

## What This Changes

The issue requests complete parsing of inbound Feishu Interactive Card message content, including card text, button labels, selected options, and form values.

## Best Possible Solution

Keep this as the Feishu plugin tracker, share or extract the existing interactive-card text parser for direct inbound events, and split callback option/form passthrough into a focused maintainer-reviewed follow-up unless maintainers approve one combined scope.

## Reproduction Assessment

Yes, at source level: an inbound Feishu event with `message_type: "interactive"` reaches `parseFeishuMessageEvent`, which calls `parseMessageContent`, and that parser has no interactive-card branch. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as one broad fix. Reusing the existing card text extractor for direct inbound events looks like the narrowest parser repair, while option/form callback payload passthrough needs a maintainer-approved encoding/API choice.
