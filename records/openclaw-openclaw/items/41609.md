## Summary

Keep open. Current main still lacks live inbound `interactive`/`interactive_card` handling in the Feishu receive parser, while existing card parsing only covers fetched/thread-message paths and related parser/action PRs are still open.

## What This Changes

The issue requests parsing Feishu interactive-card message content, including card text, buttons, dropdown selections, and form values, instead of exposing only an unsupported/type placeholder.

## Best Possible Solution

Consolidate the related Feishu parser/action PRs into one plugin-owned change that shares card content extraction across live receive, fetched/quoted, merged/thread paths, and handles callback option/form fields in the existing card-action path.

## Reproduction Assessment

Yes, at source level. Passing an `im.message.receive_v1` event with `message_type: "interactive"` or `"interactive_card"` through `parseFeishuMessageEvent()` still reaches `parseMessageContent()`, which has no card branch and returns the raw payload.

## Solution Assessment

No, current main is not the best solution because it leaves live inbound parsing split from the fetched-message card parser. The narrow maintainable fix is to share the parser and add direct inbound plus card-action callback coverage.
