## Summary

Keep open. Current main partially parses fetched or quoted Feishu interactive cards, but this issue’s central inbound message-event and card-action content request is still not solved; the merged repair at https://github.com/openclaw/openclaw/pull/72397 explicitly left the broader inbound work open, and the overlapping open PRs are not safe canonical replacements yet.

## What This Changes

Request Feishu inbound interactive-card support so agents can read card text, buttons, options, and form data instead of placeholders or raw type markers.

## Best Possible Solution

Keep this issue as the canonical tracker, then land or replace a narrow Feishu plugin fix that parses inbound cards and fetches documented card content where needed while splitting card action callbacks into a separately reviewed scope if necessary.

## Reproduction Assessment

Yes, source-reproducible. Current main routes inbound Feishu events through `parseMessageContent()` with no `interactive` case, and merge-forward submessages also default interactive content to `[interactive]`; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Unclear as a single patch. The Feishu plugin is the right boundary, but the best solution needs maintainer scoping across inbound card parsing, documented card-content fetch behavior, and separate card action callback data before choosing among overlapping PRs.
