## Summary

Keep open. Current main still routes inbound Feishu `message_type: "interactive"` events through a parser with no interactive-card branch, while the merged related work only partially covers fetched/read-message card parsing and separate callback-field PRs remain unresolved.

## What This Changes

The issue requests Feishu Interactive Card message and callback content parsing so card text, buttons, options, and form values reach the agent instead of placeholders or raw type data.

## Best Possible Solution

Keep this as the canonical Feishu plugin tracker, share the existing interactive-card text parser with the inbound event path, and split callback option/form passthrough into a focused maintainer-reviewed PR unless maintainers approve a combined scope.

## Reproduction Assessment

Yes at source level: an inbound Feishu event with `message_type: "interactive"` reaches `parseMessageContent`, which has no interactive-card branch. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as one broad change. The Feishu plugin is the right boundary, but maintainers should decide whether inbound card text parsing and callback option/form payload passthrough are one repair or separate focused fixes.
