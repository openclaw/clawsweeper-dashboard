## Summary

Keep open. Current main still only parses Feishu `interactive` content on fetched/read-message paths, while the inbound message-event path has no `interactive` branch and the related open PR is an implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to parse Feishu Interactive Card message-event content, including card text and interactive fields, instead of exposing only an unsupported placeholder or type marker.

## Best Possible Solution

Land or replace a narrow Feishu-plugin fix that wires inbound `message_type: "interactive"` through the existing card parser, while keeping card-action form/select field passthrough tracked in the narrower callback issues unless maintainers explicitly combine the scope.

## Reproduction Assessment

Yes from source inspection: current main routes inbound Feishu message events through `parseMessageContent()` with no `interactive` branch, so card events do not reach the existing card parser. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as one broad fix: Feishu is the right plugin boundary, but inbound card text parsing, optional raw-card fetching, merge-forward cards, and card-action form/select payload passthrough should be scoped deliberately before selecting the canonical implementation.
