## Summary

Keep open. Current main and the latest release still have no inbound Feishu `interactive` branch in the message-event parser; the merged related work at https://github.com/openclaw/openclaw/pull/72397 only covers fetched/read-message card parsing, and the callback-field subset is still tracked by open narrower work.

## What This Changes

The issue requests complete parsing of inbound Feishu interactive card content, including card text, button labels and values, dropdown options, and form input values.

## Best Possible Solution

Keep this as the broad Feishu plugin tracker, share or align the fetched-message card parser with inbound event parsing, and let the open callback-field item own form/input/option passthrough unless maintainers choose a combined design.

## Reproduction Assessment

Yes, at source level: an inbound Feishu event with `message_type: "interactive"` reaches `parseMessageContent`, which has no interactive-card branch. Live Feishu tenant proof is still missing for the exact service payload and callback-field variants.

## Solution Assessment

No, closing now would not be the best solution because current main only partially solves fetched/read-message card parsing. The best path is a Feishu-plugin repair that reuses the existing parser for inbound events and a separate or explicitly combined maintainer-approved callback-field shape.
