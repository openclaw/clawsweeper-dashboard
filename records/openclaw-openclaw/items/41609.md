## Summary

Keep open. Current main and the latest release only parse Feishu interactive cards on fetched/quoted/thread-history helper paths; inbound message events still use a parser with no `interactive` branch, and the open related PRs are not viable canonical fixes yet.

## What This Changes

Request Feishu/Lark inbound interactive-card messages to expose readable card content, including text and interactive fields, instead of placeholders or raw JSON.

## Best Possible Solution

Keep this as the canonical tracker, then land or replace a narrow Feishu plugin fix for inbound interactive-card text parsing with real Feishu proof while splitting card action callbacks if they need a separate event/API decision.

## Reproduction Assessment

Yes, at source level. An inbound Feishu event with `message_type: "interactive"` reaches `parseFeishuMessageEvent()` and then `parseMessageContent()`, which has no interactive branch; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Unclear as one patch. The Feishu plugin is the right boundary, but maintainers should scope simple inbound text parsing separately from raw-card context propagation and card-action callback handling.
