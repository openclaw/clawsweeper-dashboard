## Summary

Keep open. Current main still does not fully handle Feishu interactive-card content across direct inbound events, read/message readback, merge-forward sub-messages, and card action payload fields; related open PRs cover overlapping slices and need maintainer consolidation rather than another close sweep.

## What This Changes

Requested Feishu interactive-card receive/read handling that surfaces card text, button data, dropdown options, and form values instead of placeholders or raw type data.

## Best Possible Solution

Consolidate the Feishu plugin work into one bounded implementation that handles direct inbound events, message reads, thread and merge_forward reads, and card action payload fields with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main routes direct inbound interactive cards and merge-forward interactive sub-messages through parsers without interactive branches, while readback and action callback paths only cover part of the requested card payload.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but maintainers should consolidate the overlapping parser, readback, merge_forward, and card-action PRs around a verified API request shape and stable agent-facing text/raw payload semantics.
