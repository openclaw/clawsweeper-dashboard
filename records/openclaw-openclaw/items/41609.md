## Label Justifications

- `P2`: The issue blocks real Feishu interactive-card workflows but is scoped to one channel plugin rather than core runtime availability.
- `impact:data-loss`: Button, dropdown, and form values are user-provided data that can be stripped before agent dispatch.
- `impact:message-loss`: Interactive card text and callback content can fail to reach the agent as usable inbound message content.

## Summary

Keep open: current main and the latest shipped release still lack a complete Feishu interactive-card path for direct inbound card messages and card-action sibling fields, while overlapping open PRs need one canonical maintainer decision.

## What This Changes

The issue requests Feishu inbound interactive cards and card callbacks to expose complete card text, button, dropdown, and form values to the agent instead of only a placeholder or type marker.

## Best Possible Solution

Choose one canonical Feishu-plugin implementation that handles direct inbound card messages, readback and merge-forward card variants, and card action/form callbacks through a documented agent-visible payload contract with focused tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A direct inbound Feishu message with message_type interactive reaches parseMessageContent without an interactive branch, and card-action callback parsing narrows action to value/tag before fallback dispatch.

## Solution Assessment

Unclear as a final implementation because main has partial readback parsing but not the complete inbound and callback contract. The best fix should stay in the Feishu plugin and consolidate the overlapping open parser/action PRs instead of landing another partial parser.
