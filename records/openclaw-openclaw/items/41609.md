## Summary

Keep open. Current main still leaves live Feishu interactive-card message events on the generic inbound parser, which has no `interactive` or `interactive_card` branch; the shipped parser work only covers fetched/quoted message content and related open PRs still need maintainer consolidation.

## What This Changes

The issue requests Feishu inbound interactive-card parsing so message events expose readable card text, button labels/values, dropdown selections, and form inputs instead of placeholders or raw type data.

## Best Possible Solution

Keep this as the canonical Feishu inbound interactive-card parsing tracker and consolidate the related PRs into one Feishu-plugin fix that shares the card parser across live receive, fetched/quoted, and merged-message paths, supports both `interactive` and `interactive_card`, and coordinates action selection/form fields with #42908.

## Reproduction Assessment

Yes, at source level. Passing an `im.message.receive_v1` event with `message_type: "interactive"` or `"interactive_card"` through `parseFeishuMessageEvent()` still reaches `parseMessageContent()`, which lacks a card branch and therefore cannot extract readable card content.

## Solution Assessment

No, current main is not the best solution because it leaves live inbound parsing split from the existing fetched-message card parser. The narrow maintainable fix is extension-owned parser sharing plus direct inbound regression tests and callback-field coverage where Feishu supplies action data.
