## Summary

Keep open. Current main has partial Feishu interactive-card parsing for fetched messages, but the live inbound event path still does not parse `interactive` card content, and the related implementation work in https://github.com/openclaw/openclaw/pull/56795 plus narrower parser follow-ups remains unmerged.

## What This Changes

Request Feishu inbound interactive-card messages to expose readable card content instead of only a placeholder, type, or raw card payload.

## Best Possible Solution

Land one consolidated Feishu plugin fix that shares interactive-card parsing between inbound events and fetched-message reads, adds focused inbound-event and parser tests, and folds in the narrow card-format cases from the related open PRs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an inbound Feishu event with `message_type: "interactive"` is routed through `parseFeishuMessageEvent` into `parseMessageContent`, which has no interactive-card branch. I did not run a live Feishu tenant callback in this read-only review.

## Solution Assessment

Yes, keeping this open is the right outcome; closing would be premature because current main and the latest release still miss the inbound event parsing path. The narrow maintainable direction is to review and consolidate the open Feishu PRs rather than starting a duplicate repair branch.
