## Summary

Keep open: current main and latest release still do not parse direct inbound Feishu interactive-card messages, and the remaining action/form/readback work overlaps active PRs that need maintainer consolidation rather than a fresh automated fix.

## What This Changes

The issue asks OpenClaw to parse Feishu interactive-card content, including card text, button/dropdown values, and form input values, instead of surfacing only a placeholder or raw type.

## Best Possible Solution

Consolidate one Feishu-plugin implementation that shares bounded card parsing across direct inbound and readback paths, then handle action/form callback payloads through an explicit maintainer-approved payload contract.

## Reproduction Assessment

Yes, at source level. A direct inbound Feishu event with message_type "interactive" reaches parseMessageContent without an interactive branch, while callback fields such as option/options/form_value are still not represented in the current handler contract.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but the best path is maintainer consolidation of the overlapping parser/readback PRs plus a clear decision on card action/form callback payload scope.
