## Label Justifications

- `P2`: The issue affects Feishu interactive-card content reaching agents, but the blast radius is limited to one bundled channel plugin.
- `impact:message-loss`: Feishu card text and interaction fields can be hidden or dropped before the agent can read the user's message or card action.

## Summary

Keep open. Current main and the latest release still lack inbound Feishu `interactive` message-event parsing, while the merged/readback fixes only cover fetched messages and the callback-field work remains a separate open scope.

## What This Changes

The issue requests Feishu interactive-card message and callback content parsing so agents can see card text, button values, selector choices, and form input data.

## Best Possible Solution

Keep this as the broad Feishu plugin tracker, reuse or extract the existing interactive-card parser for inbound message events, and resolve callback field passthrough through the narrower open callback work or an explicit combined maintainer-approved shape.

## Reproduction Assessment

Yes, at source level: inbound Feishu events with `message_type: "interactive"` go through `parseFeishuMessageEvent` into `parseMessageContent`, which has no interactive-card branch. Callback form/select fields are also source-visible as dropped because the callback normalizer returns only `action.value` and `tag`.

## Solution Assessment

No, closing this issue is not the best solution because current main only fixes fetched/read-message card content. The maintainable fix should stay in the Feishu plugin, share the existing parser with inbound events, and make a deliberate callback payload-shape decision.
