## Summary

Keep open: current main still leaves direct inbound Feishu `interactive` messages on the unhandled parser path, and callback form/selection fields still need a maintainer-approved payload contract before more automated fixes are useful.

## What This Changes

The issue asks Feishu interactive card message events and callbacks to expose complete card text, controls, and form/selection data instead of placeholders or raw/partial values.

## Best Possible Solution

Consolidate one Feishu-plugin implementation that handles direct inbound, readback/quoted, merge_forward, and card action/form callbacks through a clear payload contract with focused tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A direct inbound Feishu event with `message_type: "interactive"` reaches `parseMessageContent` without an interactive branch, and callback fields such as option/options/form_value are still outside the current handler contract.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but the maintainable path is to consolidate the overlapping parser/readback PRs and make an explicit callback payload contract decision.
