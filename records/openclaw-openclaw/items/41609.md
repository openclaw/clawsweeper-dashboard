## Summary

Current main and v2026.6.5 still send live inbound Feishu interactive-card events through a generic parser with no interactive-card branch, while the existing card extractor is limited to fetched/read paths. The issue remains necessary and is paired with the open implementation at https://github.com/openclaw/openclaw/pull/56795; maintainers should resolve that PR’s payload-fetch and data-boundary decisions and require real-tenant proof before landing.

## What This Changes

Parse live inbound Feishu interactive card messages into readable card text and interaction content instead of passing raw JSON or placeholder content to the agent.

## Best Possible Solution

Keep one Feishu-owned card extractor shared by live inbound, quoted, read, and forwarded paths; use event content when complete, fetch richer content only when required, preserve a readable fallback on fetch failure, and prove both paths in a real Feishu tenant before landing.

## Reproduction Assessment

Yes, at source level: a live `interactive` event reaches `parseMessageContent`, which has no interactive-card branch, while the fetched-message path already contains the expected extractor. A real Feishu tenant run remains necessary to verify the exact callback and follow-up API payload shapes.

## Solution Assessment

Unclear. Reusing the existing Feishu extractor is the correct owner-boundary direction, but the extra message fetch and raw-payload propagation in the open PR require maintainer approval and live proof before that implementation can be considered the best fix.
