## Summary

Keep open. Current `origin/main` still lacks inbound Feishu `interactive`/`interactive_card` message-event parsing and still narrows card-action callbacks to `action.value`/`tag`; related merged work only covers fetched/read card content, while callback field passthrough remains a narrower open scope.

## What This Changes

The issue requests complete Feishu interactive-card parsing for inbound card messages and card callbacks, including card text, buttons, dropdown selections, and form input values.

## Best Possible Solution

Keep this as the broad Feishu plugin tracker, reuse or extract the existing interactive-card parser for inbound message events, and resolve callback field passthrough through the narrower linked callback work or an explicit combined maintainer-approved shape.

## Reproduction Assessment

Yes, source-level: inbound Feishu events call `parseMessageContent(event.message.content, event.message.message_type)`, and that parser has no interactive-card branch; callback payload parsing also narrows actions to `value` and `tag`. I did not run a live Feishu tenant event in this read-only review.

## Solution Assessment

No, closing is not the best solution because current main only covers fetched/read-message card parsing and still drops or bypasses the inbound/callback paths. The maintainable fix should stay inside the Feishu plugin and make a deliberate split-or-combine decision for inbound parsing versus card-action passthrough.
