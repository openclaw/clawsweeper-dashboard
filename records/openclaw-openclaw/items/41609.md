## Summary

Keep open. Current main still has incomplete Feishu interactive-card handling: direct inbound events and merge-forward sub-messages are not parsed through an interactive-card branch, readback fetches do not request richer card payloads, and the remaining fixes are split across active Feishu parser/readback/action items that need consolidation.

## What This Changes

The issue asks Feishu inbound, readback, and card-action paths to expose complete interactive-card content instead of placeholders, raw JSON, or missing button/form values.

## Best Possible Solution

Consolidate the Feishu plugin work into one bounded implementation that handles direct inbound events, message reads, thread/merge-forward reads, and card action payloads with focused regression tests and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main routes direct inbound interactive cards and merge-forward interactive sub-messages through parsers without interactive branches, and readback only has partial flat-card extraction without requesting richer card payloads.

## Solution Assessment

No final solution is on main yet. The Feishu plugin is the right boundary, but maintainers should consolidate the overlapping parser/readback/action PRs around a verified API request shape, bounded parsing, and stable RawBody/user-text semantics.
