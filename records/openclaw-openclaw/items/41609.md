## Summary

Keep open. Current main still routes live Feishu `interactive` / `interactive_card` receive events through a parser with no card branch, while the merged quoted-card parser work in https://github.com/openclaw/openclaw/pull/72397 explicitly left the broader inbound-card work to https://github.com/openclaw/openclaw/pull/56795 and this issue.

## What This Changes

Request Feishu interactive card message and callback parsing so inbound cards expose card text, button labels/values, dropdown options, and form inputs to agents.

## Best Possible Solution

Land one Feishu-plugin-owned parser/action change that shares card extraction across live receive, fetched, thread, and merge paths and models card callback option/form data against Feishu's documented payloads.

## Reproduction Assessment

Yes, at source level. An `im.message.receive_v1` payload with `message_type: "interactive"` or `"interactive_card"` flows through `parseFeishuMessageEvent()` to `parseMessageContent()`, which still has no card branch and returns the raw payload.

## Solution Assessment

No, current main is not the best solution because live receive parsing, fetched/thread parsing, merge-forward formatting, and card-action callback fields remain split. The safer fix is a consolidated Feishu-plugin parser/action path with focused tests rather than another partial branch.
