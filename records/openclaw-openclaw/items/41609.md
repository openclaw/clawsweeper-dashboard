## Summary

Keep open. Current main still leaves the broader Feishu interactive-card request unresolved: direct inbound card messages lack a dedicated parser path, fetched/read parsing is only partial, and card-action callbacks still ignore selection/form fields unless they are inside `action.value`.

## What This Changes

The issue requests complete Feishu interactive-card parsing for inbound messages and card callbacks, including card text, button values, dropdown options, and form inputs.

## Best Possible Solution

Land one Feishu-plugin-owned parser/action path, or an explicitly coordinated set of PRs, that shares card extraction across live receive, fetched/read, thread/quote paths and preserves `option`, `options`, and `form_value` callback payloads with focused regression tests.

## Reproduction Assessment

Yes, at source level. An `im.message.receive_v1` payload with `message_type: "interactive"` flows through `parseFeishuMessageEvent()` into `parseMessageContent()`, which has no card branch, and card-action selection/form fields are not represented in the current fallback path.

## Solution Assessment

No, current main is not the best solution. The maintainable fix is a consolidated Feishu-plugin parser/action change that reuses a shared extractor, uses the correct Feishu card payload contract, and preserves existing message context compatibility.
