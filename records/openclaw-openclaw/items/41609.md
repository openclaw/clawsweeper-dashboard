## Summary

Keep open. Current main and the latest release still do not parse Feishu inbound `message_type: "interactive"` events through the card parser; only fetched/read-message interactive cards are partially covered, and the related PRs are still open rather than merged fixes.

## What This Changes

The issue requests Feishu inbound Interactive Card message parsing so card text, controls, options, and form-like content are visible to agents instead of placeholders.

## Best Possible Solution

Keep this issue open for a maintainer-chosen Feishu plugin fix that parses inbound `message_type: "interactive"` events, while using the separate card-action issues for form/select callback passthrough unless maintainers explicitly merge those scopes.

## Reproduction Assessment

Yes from source inspection for the inbound path: current main and the latest release route Feishu message events through `parseMessageContent`, and that function has no `interactive` case. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as one broad fix: the Feishu plugin is the correct boundary, but maintainers should decide whether inbound card text parsing, optional raw-card fetching, and callback form/action payload passthrough are one change or separate focused fixes.
