## Summary

Keep open. Current main still has the Feishu interactive-card receive gap: live `im.message.receive_v1` events go through `parseMessageContent()`, which has no `interactive` or `interactive_card` branch, while the existing card parser is confined to fetched/read message parsing in `send.ts`. Related open PRs appear to be partial implementation candidates, not a reason to close this issue before merge.

## What This Changes

The issue requests that the Feishu plugin parse inbound interactive-card message content and card-action selections/forms instead of exposing only a placeholder, type marker, or raw payload.

## Best Possible Solution

Keep this as the canonical Feishu direct inbound interactive-card parsing issue. A maintainable fix should keep the behavior extension-owned, share or move the existing card parser into the live inbound path, support both `interactive` and `interactive_card` in `parseMessageContent()` and merged/forwarded submessage formatting, preserve or fetch raw card payload data when Feishu supplies only a placeholder, coordinate the card-action selection/form fields with #42908, add direct inbound regression tests, and update Feishu docs if setup requirements change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass an `im.message.receive_v1`-style event with `message_type: "interactive"` or `"interactive_card"` through `parseFeishuMessageEvent()`; current main calls `parseMessageContent()`, which has no card branch and therefore does not extract readable card content.

## Solution Assessment

No. Current main is not the best solution because it leaves fetched-message card parsing and live inbound parsing split; the safer fix is a narrow Feishu-plugin change that reuses one parser across receive, fetched/read, and merged-message paths with focused tests.
