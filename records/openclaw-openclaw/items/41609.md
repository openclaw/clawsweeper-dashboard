## Summary

Keep open: current main and the latest release still lack a complete Feishu interactive-card path for direct inbound card messages and card-action form payloads, and the related open PRs need a canonical maintainer decision rather than another cleanup close.

## What This Changes

The issue requests Feishu inbound interactive-card parsing so agents can see card text, button labels, dropdown selections, and form values instead of placeholders or incomplete callback payloads.

## Best Possible Solution

Choose one Feishu plugin implementation that handles direct inbound cards, readback/quoted and merge-forward card variants, and card-action form callbacks through a documented agent-visible payload contract with focused tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. A direct inbound Feishu `message_type: "interactive"` reaches `parseMessageContent` without an interactive branch, and card-action callbacks drop sibling fields before fallback dispatch.

## Solution Assessment

Unclear until maintainers choose the payload contract. The Feishu plugin is the right boundary, but the maintainable solution should consolidate direct inbound parsing, readback parsing, and callback payload forwarding rather than landing another partial parser.
