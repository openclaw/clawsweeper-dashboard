## Summary

Keep open. Current main still leaves Feishu live inbound interactive-card events on an unhandled parser path, and the remaining work is split across active Feishu PRs rather than landed on main.

## What This Changes

Parse Feishu Interactive Card message and callback content so agents can see card text, button labels, selected options, and form values.

## Best Possible Solution

Land one Feishu-plugin-owned parser/action path, or an explicitly coordinated set of PRs, that shares card extraction across live receive, fetched/thread/merge paths and preserves option/options/form_value callback payload fields with focused tests.

## Reproduction Assessment

Yes, at source level. An im.message.receive_v1 payload with message_type interactive or interactive_card flows through parseFeishuMessageEvent() into parseMessageContent(), which has no card branch and returns raw content for unhandled types.

## Solution Assessment

No, current main is not the best solution. The maintainable fix is a consolidated Feishu-plugin parser/action change that uses Feishu's documented card payload contracts and keeps RawBody/context compatibility intact.
