## Summary

Keep open. Current main and v2026.6.10 still leave direct inbound Feishu `interactive` message events without an interactive-card parser, and card-action callbacks still normalize the action payload down to `value`/`tag`; related open work covers narrower merge-forward and callback subsets rather than replacing this broad tracker.

## What This Changes

The issue requests Feishu interactive card message parsing so agents can see card text, button labels, option selections, and form values instead of placeholder or type-only content.

## Best Possible Solution

Keep this as the broad Feishu plugin tracker, wire the existing interactive-card text extractor into direct inbound message events, and resolve raw callback-field passthrough through the narrower linked issue or an explicit combined maintainer-approved shape.

## Reproduction Assessment

Yes, source-level: direct inbound Feishu events call `parseMessageContent`, which has no interactive-card branch, and card-action parsing preserves only `action.value`/`tag`. I did not run a live Feishu tenant event in this read-only review.

## Solution Assessment

No, closing is not the best solution because current main and the latest release still miss the direct inbound and callback-field paths. The maintainable fix should stay inside the Feishu plugin and deliberately split or combine this broad tracker with the narrower callback-field issue.
