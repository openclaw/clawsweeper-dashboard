## Summary

Keep open: current main and v2026.5.3 still apply a POSIX group/other mode-bit check to the native hook relay bridge directory, which matches the reported Windows failure path. The discussion also correctly flags that a simple Windows skip would remove validation around a security-sensitive relay directory.

## What This Changes

The issue asks OpenClaw to use Windows-appropriate validation for the Codex native hook relay bridge directory instead of POSIX mode-bit checks that can false-fail on Windows.

## Best Possible Solution

Keep the Unix owner/private-mode checks, add an explicit Windows ACL validation path for the bridge directory, and fail closed or repair with a documented ACL reset when validation is unavailable.

## Reproduction Assessment

Yes for source reproduction: current main reaches the exact throwing branch when `chmodSync(0o700)` leaves group/other mode bits set, and Node documents that Windows does not implement owner/group/other permission distinctions. I did not run a live Windows harness.

## Solution Assessment

No implemented solution exists yet. The reporter's `process.platform !== "win32"` guard is a useful diagnostic workaround, but the durable fix should preserve the bridge directory's safety guarantees with Windows ACL validation rather than disabling the check.
