## Summary

Keep open: current main still applies POSIX group/other mode-bit validation to the native hook relay bridge directory, and the reported Windows failure path remains source-reproducible. The fix should be platform-specific because the guard protects relay bridge token files.

## What This Changes

Make native hook relay bridge directory permission validation on Windows use Windows-appropriate ACL/ownership checks instead of POSIX group/other mode bits that block the Codex harness.

## Best Possible Solution

Add an explicit Windows branch using the existing ACL inspection/reset approach or equivalent trusted-principal validation for the bridge directory, while preserving POSIX checks elsewhere and adding focused Windows-aware regression coverage.

## Reproduction Assessment

Yes for source reproduction: current main calls the bridge directory setup during Codex relay registration, and that setup throws the reported error if Windows leaves synthesized group/other mode bits after `chmodSync(0o700)`. I did not run a live Windows harness in this read-only pass.

## Solution Assessment

No implemented solution exists yet. The reporter's platform skip is directionally useful for diagnosis, but the maintainable fix is a Windows-specific ACL validation path rather than dropping the security check.
