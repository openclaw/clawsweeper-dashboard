## Summary

Keep open: current main still applies POSIX group/other mode-bit validation to the native hook relay bridge directory, and the Windows discussion points to a real platform-specific validation gap rather than a stale report.

## What This Changes

The issue asks OpenClaw to use Windows-appropriate validation for the native hook relay bridge directory so Codex harness startup is not blocked by POSIX mode-bit false positives.

## Best Possible Solution

Keep the Unix owner and private-mode checks intact, add a Windows ACL/ownership validation branch for bridge directory and registry files, and cover it with focused relay tests plus Windows harness proof.

## Reproduction Assessment

Yes for source reproduction: current main still reaches the exact throw when `lstat`/`chmodSync` leaves group/other bits set. I did not run a live Windows Codex harness, so the reproduction is source-backed rather than live-reproduced.

## Solution Assessment

No implemented solution exists yet. The reporter's win32 guard is directionally useful for proving the failure, but the maintainable fix should preserve relay-token safety with a real Windows ACL/ownership check.
