## Summary

Keep open: current `main` and `v2026.5.3` still apply POSIX group/other mode-bit validation to the native hook relay bridge directory, matching the reported Windows Codex harness failure path.

## What This Changes

The issue asks for Windows-appropriate native hook relay bridge directory validation instead of a POSIX mode-bit check that can false-positive on Windows.

## Best Possible Solution

Keep the Unix owner and private-mode checks intact, add a Windows-specific validation path that does not rely on POSIX group/other bits, and cover it with focused relay tests plus Windows startup validation.

## Reproduction Assessment

Yes for source reproduction: the exact reported error is reachable on current `main` when `chmodSync(0o700)` leaves group/other mode bits set on Windows. I did not run a live Windows harness in this read-only pass.

## Solution Assessment

No implemented solution exists yet. The reporter's `process.platform !== "win32"` guard is plausible, but the maintainable fix should make the platform boundary explicit and preserve the relay bridge safety guarantees.
