## Summary

Keep open: current main still applies POSIX group/other mode-bit validation to the native hook relay bridge directory, and the discussion identifies a real Windows-specific permission-model mismatch that still needs a secure platform branch.

## What This Changes

Fix the Windows native hook relay bridge permission check so Codex harness startup is not blocked by POSIX mode-bit validation on a Windows bridge directory.

## Best Possible Solution

Keep the Unix private-mode and ownership checks intact, add a Windows-specific ACL/ownership validation path for the bridge directory, and cover both branches with focused relay tests plus Windows startup validation.

## Reproduction Assessment

Yes for source reproduction: current main still reaches the reported error when chmod does not clear group/other mode bits, and Node documents that Windows does not implement owner/group/other permission distinctions. I did not run a live Windows harness.

## Solution Assessment

No implemented solution exists yet. The reporter's win32 guard explains the failure mode, but the maintainable fix should use an explicit Windows ACL branch rather than dropping permission validation.
