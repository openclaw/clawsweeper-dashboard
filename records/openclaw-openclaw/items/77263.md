## Summary

Current `main` still contains the Windows-sensitive POSIX mode-bit validation that matches the reported Codex harness startup failure, and the supported Codex native hook relay path reaches that check. Keep this open because the fix needs a Windows-specific safety policy for a native relay directory.

## What This Changes

The issue asks OpenClaw to stop rejecting Windows native hook relay bridge directories solely because POSIX group/other mode bits remain set after `chmod(0o700)`.

## Best Possible Solution

Keep the Unix owner/private-mode checks, add an explicit Windows path that validates the bridge directory with Windows-appropriate ACL or ownership semantics, and cover both branches with focused relay tests plus Windows startup validation.

## Reproduction Assessment

Yes for source reproduction: current `main` reaches the exact reported throw when `chmodSync(0o700)` leaves group/other mode bits set, and Node documents that Windows does not implement owner/group/other permission distinctions. I did not run a live Windows gateway harness in this read-only review.

## Solution Assessment

No implemented solution exists yet. The reporter’s platform guard is directionally plausible, but the best fix should make the Windows security boundary explicit rather than skipping the relay-directory safety check entirely.
