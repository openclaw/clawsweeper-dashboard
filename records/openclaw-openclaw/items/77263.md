## Summary

Keep open: current main and the latest release still apply POSIX group/other mode-bit validation to the native hook relay bridge directory, which matches the Windows failure path and still needs a Windows-safe permission policy.

## What This Changes

The issue requests a Windows-appropriate native hook relay bridge directory permission check instead of POSIX group/other mode-bit validation in the Codex harness.

## Best Possible Solution

Keep the Unix owner/private-mode checks, add a Windows-specific ACL or ownership validation path for the bridge directory and registry files, and cover it with focused relay tests plus a Windows Codex harness smoke.

## Reproduction Assessment

Yes for source reproduction: the current-main Codex registration path reaches the exact throw when Windows-style synthesized mode bits remain nonzero after `chmodSync(0o700)`. I did not run a live Windows gateway/Codex harness in this read-only pass.

## Solution Assessment

No implemented fix exists yet. The reporter's skip is a useful workaround, but the durable repository fix should preserve the relay bridge security boundary with Windows-specific ACL validation rather than dropping the check entirely.
