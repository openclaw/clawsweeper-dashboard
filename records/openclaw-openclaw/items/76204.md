## Summary

Current main already covers the reported empty-prompt path: runtime-only events with an empty transcript prompt are submitted with a non-empty marker prompt and hidden runtime context, ordinary blank visible prompts are skipped before provider submission, and heartbeat/wake runs use a non-empty transcript marker.

## What This Changes

Validate or neutralize empty gateway user prompts so malformed wake/system events do not reach model providers or poison auth-profile cooldowns.

## Best Possible Solution

Keep the current marker-prompt/runtime-context split and blank-prompt preflight guard; reopen only if a new report shows a distinct provider submission path still accepts empty content.

## Reproduction Assessment

Yes. The source-level reproduction is `effectivePrompt` containing a runtime event with `transcriptPrompt: ""`; current tests assert that this now becomes a non-empty marker prompt, and a separate test verifies blank visible prompts are skipped before provider submission.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix because it preserves runtime event context for the model without persisting or forwarding an empty user message, while the generic skip guard covers non-runtime blank prompts.
