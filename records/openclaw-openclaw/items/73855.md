## Summary

Keep open. The hosted-installer drift guard is useful and the earlier PR-trigger/concurrency defects were fixed, but the latest workflow still runs the live CDN comparison directly on installer-source pushes before the existing website sync workflow can publish the same source.

## What This Changes

The PR adds a GitHub Actions workflow that compares `scripts/install.sh` with `https://openclaw.ai/install.sh` on workflow-file PRs, installer-source pushes, daily schedule, and manual dispatch.

## Best Possible Solution

Keep the drift guard, but run the live hosted comparison only after `Website Installer Sync` has completed successfully, or limit live drift detection to schedule/manual runs while workflow syntax is validated separately.

## Reproduction Assessment

Yes. The PR's own GitHub Actions run fetched the hosted installer and failed on a real SHA mismatch, and source inspection shows the direct `push` trigger can run before `website-installer-sync.yml` publishes the same source.

## Solution Assessment

No. A live CDN check is the right direction, but attaching it directly to installer-source pushes is not the narrowest maintainable trigger because the publisher workflow completes later.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Run drift checks after installer sync completes:** `.github/workflows/check-hosted-installer-sync.yml:4-8`
  - body: This push trigger checks the live hosted installer on the same `main` push that changes `scripts/install.sh`, but the hosted file is only copied to openclaw.ai later by `Website Installer Sync` after its verifier jobs pass. Normal installer changes can therefore leave an expected red run until a later schedule; trigger this after the sync workflow succeeds, keep it scheduled/manual, or add a deployment-aware wait.
