## Summary

Keep open. Current main does not include an equivalent hosted-installer drift workflow, and the linked installer bug remains open, but the current PR head still has the same two merge-blocking workflow bugs called out earlier: pull requests that change `scripts/install.sh` compare unpublished PR bytes against the deployed hosted installer, and scheduled runs share the same cancellation group as main pushes.

## What This Changes

The PR adds `.github/workflows/check-hosted-installer-sync.yml`, a GitHub Actions workflow that fetches `https://openclaw.ai/install.sh`, compares its SHA-256 digest with `scripts/install.sh`, and runs on push, pull request, daily schedule, and manual dispatch.

## Best Possible Solution

Keep the checksum guard, but make the live source-vs-host comparison a post-merge, scheduled, or manual signal instead of a pull-request check for installer source changes. The workflow should also isolate scheduled concurrency from push concurrency, then be validated with workflow sanity before it is used to close the linked installer issue.

## Reproduction Assessment

Yes. The current PR head still includes `scripts/install.sh` in the `pull_request` path filter, the compare step exits on any source-hosted hash mismatch, the live source and hosted hashes differ, and the PR head sync check is failing.

## Solution Assessment

No. The idea is useful, but this implementation is not the best mergeable form because it compares in-flight PR bytes against already-deployed CDN bytes and lets unrelated main pushes cancel scheduled drift alerts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit live checksum checks on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but the PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from `pull_request` validation, or skip the live checksum comparison for pull request events.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22`
  - body: `schedule` and `push` runs on `main` both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group or otherwise avoid cross-event cancellation.
  - confidence: 0.91
