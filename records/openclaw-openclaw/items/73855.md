## Summary

Keep open. The PR head still contains the same two workflow blockers after the author’s re-review comment: pull requests that edit the installer are still compared against the already-deployed hosted installer, and scheduled runs still share the same cancellation group as pushes to `main`.

## What This Changes

The PR adds a GitHub Actions workflow that compares `scripts/install.sh` with the hosted `https://openclaw.ai/install.sh` on push, pull request, schedule, and manual dispatch.

## Best Possible Solution

Keep the checksum guard, but make live source-vs-host comparison a post-merge, scheduled, or manual signal, and isolate scheduled concurrency from push cancellation.

## Reproduction Assessment

Yes. The PR head still includes the problematic `pull_request` path and concurrency group, the source and hosted hashes differ, and the PR head’s hosted-installer sync check is failing.

## Solution Assessment

No. The direction is useful, but this implementation is not the narrowest mergeable form because it checks undeployed PR bytes against deployed CDN bytes and can cancel the scheduled drift alert from a main push.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit live-host comparisons on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but the PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job exits on any hash mismatch, future installer-source PRs will fail before deployment can make the hosted copy match. Remove this path from `pull_request` validation, or skip the live checksum comparison for pull request events.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group or otherwise prevent cross-event cancellation.
  - confidence: 0.91
