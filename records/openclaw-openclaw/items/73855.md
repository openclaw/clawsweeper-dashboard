## Summary

Keep open. The hosted-installer drift guard is not obsolete and current main does not have an equivalent byte-for-byte sync workflow, but this PR head still contains the two previously reported workflow blockers despite the follow-up comment saying they were fixed, and its own new workflow is failing on the checksum step.

## What This Changes

The PR adds `.github/workflows/check-hosted-installer-sync.yml`, a GitHub Actions workflow that checks out the repo, downloads `https://openclaw.ai/install.sh`, and compares its SHA-256 digest with `scripts/install.sh` on selected push, pull request, schedule, and manual runs.

## Best Possible Solution

Keep the checksum guard idea, but make the live source-vs-host comparison run only in contexts where deployed bytes are expected to match, and keep scheduled drift checks in a distinct concurrency group from push runs. The existing install smoke workflow can remain focused on executable installer behavior while this new guard covers byte drift.

## Reproduction Assessment

Yes for the introduced CI failure mode: the PR head still includes `scripts/install.sh` under `pull_request.paths`, the current source and hosted installer hashes differ, and the PR's `Check Hosted Installer Sync` run failed in the `Compare checksums` step.

## Solution Assessment

No. A read-only checksum guard is a reasonable way to detect hosted installer drift, but this implementation is not the best mergeable form because it compares in-flight PR source against deployed bytes and lets scheduled checks share cancellation with push runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Limit live-host comparisons on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but a pull-request checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from `pull_request` validation, or skip the live checksum comparison for `pull_request` events.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation groups:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both use the default branch ref, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group or otherwise keep scheduled runs from being cancelled by push runs.
  - confidence: 0.91
