## Summary

Keep open. The hosted-installer drift guard is a useful direction, but the current PR head still contains the two previously reported workflow bugs and lacks after-fix real behavior proof; current main also now has `website-installer-sync.yml`, so this needs to be rebased into that flow rather than comparing undeployed refs to the live CDN.

## What This Changes

Adds a GitHub Actions workflow that downloads `https://openclaw.ai/install.sh` and compares its SHA256 with `scripts/install.sh` on push, pull request, schedule, and manual dispatch.

## Best Possible Solution

Keep a hosted-installer drift guard, but run live CDN comparisons only on deployed states, schedule/manual checks, or after the existing website sync/deploy path has completed.

## Reproduction Assessment

Yes. Source inspection of the current PR head shows the PR-trigger and concurrency bugs directly, and current source-vs-hosted hashes differ, so the failure mode is high-confidence without mutating the checkout.

## Solution Assessment

No. The direction is useful, but this implementation still compares undeployed PR bytes to deployed CDN bytes and lets schedule/push runs cancel each other; integrating with `website-installer-sync.yml` or limiting live checks to deployed states is narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Limit live-host comparisons to deployed installer states:** `.github/workflows/check-hosted-installer-sync.yml:7-11`
  - body: This workflow still runs the live CDN checksum when `scripts/install.sh` changes on a pull request, so the PR checkout is compared with the already-deployed `openclaw.ai` installer before any deployment can update it. Any legitimate installer-source PR will fail this guard even when correct; keep PR validation to the workflow file/syntax, or skip the live checksum outside deployed/scheduled/manual states.
  - confidence: 0.98
- **[P2] Separate scheduled concurrency from push runs:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both resolve `github.ref` to `refs/heads/main`, so this group allows a main push to cancel the daily drift check. Include `github.event_name` or otherwise isolate scheduled checks so the daily signal is not suppressed.
  - confidence: 0.93
