## Summary

Keep open. The hosted-installer drift guard is useful and is not already on current main, but the latest PR head still has two merge-blocking workflow bugs: it live-compares undeployed PR installer bytes against the deployed CDN copy, and scheduled checks can be cancelled by main-push runs.

## What This Changes

The PR adds `.github/workflows/check-hosted-installer-sync.yml`, a scheduled/push/PR/manual GitHub Actions workflow that fetches `https://openclaw.ai/install.sh` and fails when its SHA256 differs from `scripts/install.sh`.

## Best Possible Solution

Keep the checksum guard, but make live hosted-vs-source comparison a post-merge, scheduled, or manual signal, and split scheduled concurrency from push concurrency.

## Reproduction Assessment

Yes. The PR head file still contains the problematic trigger and concurrency group, the live source/hosted hashes differ, and the PR-head hosted-installer sync workflow is currently failing.

## Solution Assessment

No. The direction is sound, but this implementation is not the narrowest mergeable form because it checks PR bytes against deployed CDN bytes and allows cross-event cancellation of the scheduled drift check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit live-host comparisons on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but the checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job exits on any hash mismatch, future installer-source PRs can fail before deployment can make the hosted copy match. Remove this path from `pull_request` validation, or skip the live checksum comparison for pull request events.
  - confidence: 0.98
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the group or otherwise keep scheduled runs in a separate concurrency lane.
  - confidence: 0.92
