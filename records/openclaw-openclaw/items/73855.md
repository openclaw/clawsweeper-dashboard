## Summary

Keep open. The drift guard is useful, but the PR still runs the live CDN checksum directly on main pushes that change installer source, before the existing website sync workflow can publish the same source, so future installer merges can leave main red.

## What This Changes

The PR adds a GitHub Actions workflow that downloads `openclaw.ai/install.sh` and fails when its SHA256 differs from `scripts/install.sh` on schedule, manual dispatch, main pushes, and workflow-file PRs.

## Best Possible Solution

Keep the drift guard, but run the live CDN comparison only after `website-installer-sync` succeeds for a trusted main push, or limit live checks to schedule/manual while PRs only validate workflow syntax.

## Reproduction Assessment

Yes. Source inspection shows the PR's push trigger can fetch the live CDN before `website-installer-sync` finishes publishing, and the PR check log confirms the workflow fails on live checksum mismatch.

## Solution Assessment

No. The direction is useful, but running the live comparison directly on installer-source pushes is not the narrowest maintainable solution; a trusted post-sync check or schedule/manual guard avoids red main runs.
