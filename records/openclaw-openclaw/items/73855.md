## Summary

Keep open: the drift guard is useful, but the PR still races the existing Website Installer Sync workflow on installer-source pushes, so it needs a narrow trigger-ordering repair before merge.

## What This Changes

Adds a GitHub Actions workflow that fetches `https://openclaw.ai/install.sh` and compares its SHA256 with `scripts/install.sh` on schedule, manual dispatch, main pushes, and workflow-file PRs.

## Best Possible Solution

Run the live CDN comparison after Website Installer Sync succeeds for a trusted main push, or keep live checks to schedule/manual dispatch while PRs only validate workflow syntax.

## Reproduction Assessment

Yes. Source inspection shows the new push trigger can run the live comparison on the same main push that starts Website Installer Sync, whose publish job runs only after verifier jobs; the latest PR check also shows the live comparison failing on a hosted/source hash mismatch.

## Solution Assessment

No. The drift guard is useful, but running it directly on installer-source pushes is not the narrowest maintainable solution because it can mark main red before the hosted installer has been published.
