## Label Justifications

- `P1`: The issue describes a user-visible gateway reliability regression where Active Memory can block normal replies and make channel workflows unusable.
- `impact:crash-loop`: Reporter logs and comments include gateway hangs, health timeouts, high CPU, event-loop stalls, and restart/boot overload symptoms.

## Summary

Keep open. Current main and the latest release still have a source-reproducible Active Memory reply-path risk: the prompt-build hook awaits recall, recall still runs an embedded agent without a dedicated lane, and unset embedded lanes resolve to the serial main lane; QMD startup has been partially mitigated but does not close the broader reliability/defaults tracker.

## What This Changes

The issue asks OpenClaw to make Active Memory recall and QMD startup safer so memory work cannot block or overload interactive gateway replies.

## Best Possible Solution

Keep this as the canonical reliability tracker while maintainers land or replace the direct lane/deadline fixes, then split remaining Active Memory default-mode, default-model, QMD startup, and status/doctor policy into focused follow-ups.

## Reproduction Assessment

Yes, at source level: before_prompt_build awaits Active Memory recall, recall still calls runEmbeddedPiAgent without a lane, omitted lanes resolve to main, and command lanes default to one active task. I did not run a live multi-agent gateway canary in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The narrow lane/deadline repairs should land or be replaced first, while nonblocking mode, default recall model, QMD startup policy, and status/doctor warnings need focused maintainer-reviewed follow-ups.
