## Label Justifications

- `P1`: The issue reports source-backed gateway availability and reply-latency failures affecting real interactive users when Active Memory is enabled.
- `impact:crash-loop`: The reported event-loop stalls, high CPU, gateway health timeouts, and channel reconnect flaps are process-level availability failures.

## Summary

Keep open. Current main and v2026.6.10 include useful mitigations, but the central reliability problem remains source-reproducible: Active Memory still awaits a blocking embedded recall inside `before_prompt_build`, still inherits the current session model when no memory model is configured, and still uses one process-wide `active-memory` command lane by default.

## What This Changes

The issue requests safer Active Memory and QMD startup behavior so memory recall and boot-time QMD work do not block replies or overload multi-agent gateways.

## Best Possible Solution

Make the reply-critical Active Memory path explicitly safe by default, likely through maintainer-approved nonblocking or single-shot semantics, per-agent/session recall isolation, cheap-model guidance or defaults, and QMD startup controls that stay lazy unless operators opt in.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: enable Active Memory for an eligible interactive session with a slow recall path, and `before_prompt_build` awaits an embedded recall on the single `active-memory` lane before the reply can proceed. I did not run a live multi-agent gateway in this read-only review.

## Solution Assessment

No, current main is a set of useful mitigations rather than the best final solution. The maintainable end state needs a product decision on blocking versus nonblocking recall, lane isolation, recall model/defaults, and QMD startup policy.
