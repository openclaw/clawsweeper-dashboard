## Summary

Keep open. Current main has partial mitigations for QMD startup laziness, Active Memory timeout racing, and timeout circuit breaking, but the central reply-path issue remains: eligible Active Memory turns still await a blocking embedded recall run, still inherit the active/session model by default, and still have a broad default hook budget. PR #73667 is open, draft, unmerged, and explicitly paired with this issue, so this should remain open until that PR or a replacement lands.

## What This Changes

Request safer Active Memory and QMD defaults so blocking per-message Active Memory recall and multi-agent QMD startup work do not delay or overload gateway replies.

## Best Possible Solution

Keep this as the canonical tracker until #73667 or a replacement lands a maintainable default profile: Active Memory should fail open with a short strict reply-path budget or become opportunistic by default, inherited-model behavior should be made safe or clearly diagnosed, recall concurrency should be bounded, and QMD startup should stay lazy unless an explicit serialized or jittered refresh policy is configured.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by enabling Active Memory for an eligible interactive session with `config.model` unset: the `before_prompt_build` hook awaits embedded recall using the current/session model and the default 45s hook budget. The latest comment also reports the same core slowdown on v2026.4.29 with one agent and no QMD.

## Solution Assessment

No. Current main is only a partial solution because it bounds hung recall and makes QMD startup lazy by default, but it does not make Active Memory nonblocking or short-budgeted by default and still inherits slow active models. The narrowest maintainable path is to review, land, or supersede #73667 with the remaining semantics covered.
