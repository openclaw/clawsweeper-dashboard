## Label Justifications

- `P1`: Multiple real-user reports show Active Memory and QMD work making gateway replies slow or unreliable enough to affect normal channel workflows.
- `impact:crash-loop`: Reporter logs and comments describe gateway health timeouts, event-loop stalls, high CPU, startup/boot overload, and channel liveness failures.

## Summary

Keep open. Current main and v2026.6.8 include useful mitigations, but the central reliability problem remains source-reproducible: Active Memory still gates prompt construction on embedded recall, can inherit the active chat model, and uses a single process-wide recall lane while the remaining safe-default, config, model-routing, QMD startup, and diagnostic choices need maintainer direction.

## What This Changes

The issue asks OpenClaw to make Active Memory recall and QMD startup safer for multi-agent gateways so blocking embedded recall and boot-time memory work do not overload reply paths.

## Best Possible Solution

Keep this as the canonical reliability tracker until maintainers choose and land the safe-default policy for blocking versus opportunistic recall, recall model selection, per-agent/session recall isolation, QMD startup work, and operator diagnostics.

## Reproduction Assessment

Yes, at source level: current main still awaits Active Memory recall in before_prompt_build, recall can inherit the current chat model, and the dedicated recall lane remains a single serial process-wide lane by default. I did not run a live multi-agent gateway canary.

## Solution Assessment

No. Current main is only a partial mitigation: it fixes the main-lane self-deadlock and improves QMD startup defaults, but the remaining safe-default, config/API, model-routing, and concurrency decisions still need maintainer direction.
