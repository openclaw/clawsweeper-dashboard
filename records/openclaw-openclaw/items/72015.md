## Label Justifications

- `P1`: The issue reports real user-facing agent/channel reply stalls and gateway overload in normal interactive use.
- `impact:crash-loop`: Reporter logs and comments describe high CPU, liveness warnings, gateway health timeouts, and restart/boot overload symptoms.
- `impact:message-loss`: The affected before_prompt_build path can block or suppress normal replies across chat channels when Active Memory recall times out.

## Summary

Keep open. Current main has useful QMD startup and Active Memory timeout mitigations, but the reply-path risk remains source-reproducible and this issue is already paired with open fix PRs rather than ready for cleanup close.

## What This Changes

The issue asks OpenClaw to make Active Memory and QMD startup safer by bounding or decoupling blocking recall, avoiding heavy default recall models, adding concurrency/startup controls, and surfacing operational cost.

## Best Possible Solution

Keep this as the canonical reliability tracker until the open lane/latency fixes land or are replaced, then split remaining nonblocking, default-model, QMD policy, and status/doctor work into focused follow-ups.

## Reproduction Assessment

Yes, at source level: current main still awaits Active Memory recall from before_prompt_build, that embedded recall omits a lane, omitted lanes resolve to main, and command lanes are serial by default. I did not run a live gateway canary.

## Solution Assessment

No single current fix is enough yet. A dedicated Active Memory recall lane is the narrow first repair, while nonblocking mode, default recall model, QMD startup policy, and status/doctor warnings need maintainer product decisions or narrower follow-ups.
