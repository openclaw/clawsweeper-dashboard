## Summary

Keep open. Current main has partial mitigations for Active Memory timeout handling and QMD startup breadth, but eligible replies still await Active Memory recall in `before_prompt_build`, the recall still runs an embedded agent, and the default model path still inherits the current/session model when no plugin model is configured; open draft PR #73667 remains the direct closing-PR path for this issue.

## What This Changes

The issue asks to make Active Memory and QMD startup safer by avoiding blocking per-message embedded recall, inherited slow model defaults, weak cancellation, concurrency stacking, and QMD boot bursts.

## Best Possible Solution

Review, repair, land, or replace #73667 so Active Memory fails open for normal replies by default, avoids inherited slow primary models unless explicitly chosen, bounds repeated recall work, and keeps QMD startup lazy or explicitly serialized.

## Reproduction Assessment

Yes. Enable Active Memory for an eligible direct session with `config.model` unset: current main still awaits `maybeResolveActiveRecall` in `before_prompt_build` and tests assert use of the current session model; the issue discussion adds multiple macOS/Linux logs showing the same slowdown pattern.

## Solution Assessment

No. Current main is a partial mitigation because it improves timeout/setup-grace and QMD startup behavior but preserves blocking recall and session-model inheritance; the narrower maintainable path is to finish or replace the linked reliability PR before closing this issue.
