## Summary

Keep open. Current main has useful mitigations for hook deadlines, QMD startup laziness, and the already-merged dedicated recall lane, but the central reliability problem is not fully solved: Active Memory is still a blocking pre-reply embedded-agent pass, can still inherit the active chat model, and remaining lane/default/mode choices are split across open follow-up work.

## What This Changes

The issue requests safer Active Memory and QMD reliability defaults so memory recall does not block replies or overload multi-agent gateways.

## Best Possible Solution

Keep this umbrella open until maintainers land or replace the focused Active Memory latency work, then split any remaining model-default, nonblocking/single-shot, QMD startup, lane-concurrency, and diagnostics policy into narrower follow-ups.

## Reproduction Assessment

Yes, for source-level reproduction: current main still awaits Active Memory recall in before_prompt_build, can inherit the active chat model, and routes all recall through a process-wide active-memory lane. I did not run a live current-main multi-agent gateway canary.

## Solution Assessment

No, current main is only a partial solution. The best path needs maintainer-reviewed choices for nonblocking or single-shot recall, model defaults, QMD startup behavior, lane concurrency, and operator diagnostics rather than another isolated narrow repair.
