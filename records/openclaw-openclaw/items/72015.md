## Summary

Keep open. Current main has partial reliability mitigations for QMD startup and Active Memory timeout racing, but the central reported behavior remains: eligible reply-path hooks still await an embedded recall agent, default to the current/session model when no plugin model is set, and keep a 45s default hook envelope. The provided GitHub context also shows open draft PR #73667 uses closing syntax for this issue, so the issue should remain the canonical tracker until that PR or a replacement lands.

## What This Changes

The issue requests safer Active Memory and QMD defaults so memory recall does not block normal replies or create startup bursts on multi-agent gateways.

## Best Possible Solution

Active Memory should become fail-open for normal replies by default: use a small strict pre-reply budget or opportunistic injection, avoid inheriting slow primary models unless explicitly chosen, bound recall concurrency and cleanup, and keep QMD startup lazy unless operators deliberately opt into serialized or jittered refresh work.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: enable Active Memory for an eligible direct session with config.model unset, then the current before_prompt_build hook awaits maybeResolveActiveRecall, resolves the current/session model, and runs an embedded agent before the main reply can proceed.

## Solution Assessment

No. Current main is only a partial solution because timeout racing and QMD lazy startup do not remove the blocking recall default or inherited-model behavior; reviewing, repairing, landing, or superseding PR #73667 is the narrower maintainable path.
