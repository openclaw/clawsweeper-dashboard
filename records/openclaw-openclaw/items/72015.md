## Label Justifications

- `P1`: The issue has multiple real-user reports of Active Memory and QMD work making gateway replies slow or unreliable enough to affect normal channel workflows.
- `impact:crash-loop`: Reporter logs and comments describe gateway health timeouts, event-loop stalls, high CPU, restart/boot overload, and channel liveness failures.

## Summary

Keep open. Current main and the latest release include useful mitigations, but they do not solve the central reliability problem: Active Memory still gates prompt construction on embedded recall, can inherit the active chat model, and runs recall through one process-wide serial lane while the remaining default/config decisions need maintainer judgment.

## What This Changes

The issue requests safer Active Memory recall and QMD startup behavior so memory work cannot block replies or overload multi-agent gateways.

## Best Possible Solution

Keep this as the canonical reliability tracker until maintainers choose and land the safe-default policy for blocking versus opportunistic recall, recall model selection, recall lane/concurrency isolation, QMD startup work, and operator diagnostics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main still awaits Active Memory recall in `before_prompt_build`, recall can inherit the current chat model, and the dedicated recall lane remains a single process-wide serial lane; I did not run a live multi-agent gateway canary.

## Solution Assessment

No. Current main is only a partial mitigation: it fixes the main-lane self-deadlock and improves QMD startup defaults, but the remaining safe-default and concurrency decisions still need maintainer direction.
