## Summary

Keep open. Current main has partial mitigations for hard timeouts and QMD startup breadth, but Active Memory still awaits embedded recall on the prompt-build path, still inherits the current/session model when no plugin model is configured, and an open draft PR still uses closing syntax for this same reliability issue.

## What This Changes

The issue requests reliability changes so Active Memory recall does not block normal replies and QMD boot initialization does not overload multi-agent gateways.

## Best Possible Solution

Review, repair, land, or replace the open Active Memory reliability PRs, then close this issue only after the merged fix covers the reply-critical recall path and the desired QMD/default policy is settled.

## Reproduction Assessment

Yes, at source level. Enabling Active Memory for an eligible interactive session still makes before_prompt_build await maybeResolveActiveRecall, and with config.model unset the recall model resolution still falls through to the current run model before the agent default and fallback.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable path is to land or replace the linked fixes for the Active Memory critical path, while leaving broader nonblocking/default-model/QMD policy choices to maintainer review.
