## Label Justifications

- `P1`: The issue describes a user-visible gateway reliability regression where Active Memory can block normal replies and make channel workflows unusable.
- `impact:crash-loop`: Reporter logs and comments include gateway hangs, health timeouts, high CPU, event-loop stalls, and restart/boot overload symptoms.

## Summary

Keep open. Current main has useful Active Memory timeout and QMD startup mitigations, but the reply-critical path is still source-reproducible: Active Memory recall is awaited from `before_prompt_build`, can inherit the chat model, and still relies on the embedded-runner lane default while direct fixing work remains open.

## What This Changes

The issue asks OpenClaw to make Active Memory recall and QMD startup fail open and bounded so they do not block replies or overload gateways.

## Best Possible Solution

Keep this as the canonical reliability tracker until the direct lane/deadline fixes land or are replaced, then split remaining Active Memory default-mode, default-model, QMD startup, and status/doctor policy into focused follow-ups.

## Reproduction Assessment

Yes, at source level: `before_prompt_build` awaits Active Memory recall, recall still calls `runEmbeddedPiAgent` without a lane, omitted lanes resolve to `main`, and command lanes default to one active task. I did not run a live gateway canary.

## Solution Assessment

No, current main is only a partial solution. The narrow lane/deadline repairs should land or be replaced first, while nonblocking mode, default recall model, QMD startup policy, and status/doctor warnings need focused maintainer-reviewed follow-ups.
