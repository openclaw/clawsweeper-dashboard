## Summary

Keep open. The report is still a valid reliability bug on current main: Active Memory remains on the reply-critical `before_prompt_build` path and still runs an embedded recall subagent, while an open draft PR with closing syntax and a narrower open lane-deadlock report are already tracking implementation follow-up.

## What This Changes

The issue requests safer Active Memory reply-time behavior and QMD startup throttling so multi-agent gateways do not block user replies or overload memory backends.

## Best Possible Solution

Land or replace the open Active Memory/QMD reliability PR, and coordinate with the narrower lane-deadlock report so reply-critical memory recall is bounded, fail-open, and isolated from the main user-turn lane.

## Reproduction Assessment

Yes, at source level. Enable Active Memory for an eligible direct session with `config.model` unset or slow, and current main still awaits an embedded recall run from the `before_prompt_build` hook before building the user reply.

## Solution Assessment

No final fix is present yet. Current main includes useful timeout and QMD-startup mitigations, but the best path is to finish the existing reliability PR or an equivalent maintainer-approved change that removes or isolates the reply-critical blocking behavior.
