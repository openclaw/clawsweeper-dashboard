## Label Justifications

- `P1`: The issue describes a user-visible gateway reliability regression where Active Memory can delay or block normal replies and make channel/gateway workflows unusable.
- `impact:crash-loop`: Reporter logs and comments include gateway health timeouts, event-loop stalls, high CPU, restart/boot overload, and channel liveness failures.

## Summary

Keep open. Current main has useful mitigations for Active Memory timeouts and QMD startup, but the central reliability concern remains: Active Memory is still a blocking `before_prompt_build` recall path, its dedicated recall lane is still one global serial lane, and the remaining safe-default choices require maintainer product/API judgment.

## What This Changes

The issue asks OpenClaw to make Active Memory and QMD memory startup safe for multi-agent gateways by bounding blocking recall, avoiding heavy default recall models, and preventing startup bursts.

## Best Possible Solution

Keep this as the canonical reliability tracker until maintainers choose and land the blocking-recall, recall-lane/concurrency, default-model, and QMD startup policy changes, then split any residual Active Memory quality issues into narrower follow-ups.

## Reproduction Assessment

Yes, at source level: current main still awaits Active Memory recall from `before_prompt_build`, recall can inherit the chat model, and the dedicated recall lane remains a single global serial lane. I did not run a live multi-agent gateway reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The best fix needs maintainer-chosen safe defaults around blocking vs nonblocking recall, lane/concurrency isolation, recall model selection, and QMD startup policy rather than closing after the narrower lane fix.
