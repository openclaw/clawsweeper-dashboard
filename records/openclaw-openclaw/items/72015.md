## Label Justifications

- `P1`: The issue reports source-backed gateway availability and reply-latency failures affecting real interactive users when Active Memory is enabled.
- `impact:crash-loop`: The reported high CPU, event-loop stalls, health timeouts, and channel reconnect flaps are process-level availability failures.

## Summary

Keep open. Current main and v2026.6.10 include useful mitigations, especially the dedicated recall lane and safer QMD startup defaults, but the central reliability problem remains source-reproducible: Active Memory still awaits embedded recall during prompt construction, can inherit the active conversation model, and uses one process-wide `active-memory` lane whose default concurrency is still one.

## What This Changes

The issue requests safer Active Memory and QMD startup behavior so blocking memory recall and boot work do not stall replies or overload multi-agent gateways.

## Best Possible Solution

Make Active Memory reply-critical behavior safe by default through maintainer-approved nonblocking or single-shot semantics, per-agent/session isolation, explicit fast recall-model policy, and lazy or bounded QMD startup controls.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: enable Active Memory for an eligible interactive session with slow or queued recall, and prompt construction awaits the embedded recall path on the single `active-memory` lane before the reply can proceed.

## Solution Assessment

No. Current main has useful mitigations, but the best fix still needs maintainer direction on blocking versus nonblocking recall, lane isolation, recall model defaults, and QMD startup policy.
