## Summary

Keep this open. Current main contains useful partial mitigations for QMD startup laziness, hard timeout racing, and Active Memory timeout backoff, but the central reliability request remains unresolved and the paired draft implementation PR is still open and unmerged.

## What This Changes

The issue asks for safer Active Memory and QMD startup defaults for multi-agent gateways, including fail-open or nonblocking recall, stricter latency budgets, safer model selection, recall concurrency limits, QMD boot control, and better operator diagnostics.

## Best Possible Solution

Keep the issue open until #73667 or a replacement lands a coherent safe default profile: Active Memory should fail open within a small user-visible budget, avoid surprising slow-model inheritance or warn clearly, bound stacked recall work, keep QMD startup lazy or explicitly concurrency-controlled for multi-agent gateways, and update docs/status/doctor guidance with the shipped semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by enabling Active Memory for an eligible interactive session without config.model: before_prompt_build awaits maybeResolveActiveRecall and getModelRef inherits the active/session model; for QMD, opted-in startup refresh still walks configured agents and syncs those selected by default or explicit memory-search settings.

## Solution Assessment

No. Current main is a partial mitigation rather than the best final solution: it adds lazy QMD defaults and Active Memory timeout/circuit-breaker behavior, but does not provide the requested nonblocking or deadline mode, safer default model selection, recall concurrency controls, or full operational diagnostics.
