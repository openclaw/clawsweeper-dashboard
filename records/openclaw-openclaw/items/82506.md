## Summary

Keep open: the PR’s core direction is useful, but the current head still regresses iOS no-route fallback and includes unrelated cron/pairing CI-stabilization churn.

## What This Changes

The PR changes gateway approval delivery to report attempted versus delivered state, makes configured exec/plugin approval forwarder failures visible, adds regression coverage, and includes unrelated cron/pairing edits.

## Best Possible Solution

Keep the richer delivery result contract, fix each backend to distinguish no eligible route from attempted failed delivery, revert unrelated cron/pairing churn, and avoid closing the linked issue until follow-up dispatch failures are covered or separately tracked.

## Reproduction Assessment

Yes, at source level: current iOS delivery returns `false` for no eligible/visible target and the PR maps that `false` to `attempted: true` before `approval-shared` consults turn-source fallback. I did not run a live iOS or external-channel reproduction in this read-only review.

## Solution Assessment

No: surfacing explicit approval delivery failures is the right direction, but this implementation needs richer iOS delivery state or narrower attempted accounting, and the unrelated cron changes should be removed from this gateway fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve iOS no-route fallback:** `src/gateway/server-methods/exec-approval.ts:398`
  - body: Mapping every iOS `handleRequested` result through `{ attempted: true }` treats the existing `false` no-route cases as explicit delivery attempts. Because `approval-shared` now skips `hasApprovalTurnSourceRoute` once `attempted` is true, a request that could fall back to the originating chat route can now expire as `no-approval-route`; return richer state from iOS delivery or only set `attempted` after a real push attempt fails.
  - confidence: 0.91
- **[P2] Keep persisted cron payload validation strict:** `src/cron/persisted-shape.ts:55-61`
  - body: This branch removes the trim check from persisted cron payload validation, so whitespace-only `systemEvent.text` and `agentTurn.message` values are no longer rejected at load. That is an unrelated production behavior change for a gateway approval PR and can let malformed persisted jobs enter the scheduler path; revert it or handle it in a targeted cron fix with coverage.
  - confidence: 0.82
