## Summary

Keep open: the failure-surfacing direction is useful and the PR proof is sufficient, but the latest head still has two blocking correctness issues in iOS route accounting and unrelated cron validation.

## What This Changes

The PR changes gateway exec/plugin approval delivery to report attempted versus delivered state, makes all-target approval forwarder failures visible, adds regression coverage, adjusts cron store validation tests, and moves two plugin tests onto `openclaw/plugin-sdk/test-env` helpers.

## Best Possible Solution

Keep the richer delivery result contract, make each backend distinguish no eligible route from attempted failed delivery, remove unrelated cron validation churn, and avoid closing the linked issue until follow-up dispatch failures are fixed or separately tracked.

## Reproduction Assessment

Yes, source-reproducible: current iOS delivery returns `false` for no eligible targets, while the PR maps that to `attempted: true` and `approval-shared` then skips turn-source fallback. I did not run a live iOS or external-channel reproduction in this read-only review.

## Solution Assessment

No: surfacing explicit approval delivery failures is the right direction, but this implementation needs richer iOS delivery state or narrower attempted accounting, plus removal of the unrelated cron validation change.

  - confidence: 0.84
