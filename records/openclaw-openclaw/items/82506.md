## Summary

The PR is useful but not ready: it fixes explicit approval delivery failure handling while introducing a regression in iOS no-route fallback accounting.

## What This Changes

The PR changes gateway exec/plugin approval delivery to report `{ delivered, attempted }`, makes configured approval forwarding fail visibly when all targets fail, and adds regression tests for explicit dispatch failure.

## Best Possible Solution

Keep the richer delivery result contract, but make each delivery backend distinguish no eligible route from an attempted failed delivery before suppressing fallback routing.

## Reproduction Assessment

Yes, at source level: current iOS push delivery returns `false` for no eligible route, and the PR maps that result to `attempted: true`, preventing the existing turn-source fallback check. I did not run a live iOS or external-channel reproduction in this read-only review.

## Solution Assessment

No: surfacing explicit approval delivery failures is the right direction, but this implementation needs richer iOS delivery state or narrower attempted accounting so no-route cases still fall back correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve iOS no-route fallback:** `src/gateway/server-methods/exec-approval.ts:401`
  - body: Mapping every iOS `false` result to `attempted: true` also covers normal no-route cases where `handleRequested` found no visible or eligible iOS target. Because `approval-shared` now skips `hasApprovalTurnSourceRoute` whenever `attempted` is true, an approval with a valid turn-source chat route but no iOS push route can resolve as `no-approval-route` instead of falling back to that route. Please return richer state from the iOS backend or only set `attempted` after an actual push attempt fails.
