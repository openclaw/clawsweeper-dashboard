## Summary

Keep open: the PR is a plausible partial fix for configured approval-target delivery failures, but it introduces a blocking iOS fallback regression and does not fully cover the linked follow-up delivery failure class.

## What This Changes

The PR makes gateway exec/plugin approval delivery return `{ delivered, attempted }`, makes the forwarder throw when all configured targets fail, and adds regression tests for failed explicit approval dispatch.

## Best Possible Solution

Keep the richer delivery result contract, but make each delivery backend distinguish no eligible route from attempted failed delivery and either address the follow-up failure class here or avoid closing the linked issue as fully fixed.

## Reproduction Assessment

Yes, at source level: current main returns `false` from iOS delivery for no visible/eligible targets, and the PR maps that false value to `attempted: true`, which suppresses the existing turn-source fallback path. I did not run a live external-service reproduction.

## Solution Assessment

No: the attempted-delivery contract is the right direction, but this implementation conflates no-route and failed-attempt results for iOS push delivery. A safer fix would return richer delivery state from iOS push and preserve fallback when no push was actually attempted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve iOS no-route fallback:** `src/gateway/server-methods/exec-approval.ts:401`
  - body: Mapping every iOS `false` result to `attempted: true` also covers the normal cases where `handleRequested` found no visible/eligible iOS target. Because `approval-shared` now skips `hasApprovalTurnSourceRoute` when `attempted` is true, an approval with a valid turn-source chat route but no iOS target will expire as `no-approval-route` instead of falling back to that route. Please only set `attempted` when an actual push attempt failed, or have the iOS delivery backend return richer state.
