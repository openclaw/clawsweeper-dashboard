## Summary

Keep open: the PR is aimed at a real approval-delivery bug and the supplied terminal proof is convincing, but the latest head still regresses iOS no-route fallback and carries an unrelated cron validation relaxation.

## What This Changes

The branch changes exec/plugin approval delivery to track attempted versus delivered dispatch, surface configured target failures, and add gateway/forwarder regression coverage while also changing cron and pairing test/validation files.

## Best Possible Solution

Keep the richer delivery result contract, make each backend distinguish no eligible route from attempted failed delivery, remove unrelated cron/pairing churn, and leave the linked issue open until all dispatch/follow-up failure paths are handled or split.

## Reproduction Assessment

Yes, at source level: current main logs failed configured approval dispatch without surfacing it, and the PR body includes terminal proof against the real forwarder module. The PR regression is also source-provable because iOS false delivery is mapped to attempted=true before turn-source fallback is considered.

## Solution Assessment

No: surfacing explicit approval delivery failures is the right direction, but this implementation needs richer iOS delivery state or narrower attempted accounting and should drop unrelated cron validation changes.
