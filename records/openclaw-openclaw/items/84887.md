## Summary

Keep open: current `main` and the latest shipped tag still build runtime LLM allowlist refs with raw provider/model interpolation, so an OpenRouter-shaped model id can surface as `openrouter/openrouter/...`. The reporter also narrowed the provider-precedence root cause to the Lossless issue, leaving this OpenClaw item as a focused host diagnostics bug.

## What This Changes

Fix runtime LLM allowlist normalization and diagnostics so provider-qualified model ids are not reported with a duplicated provider prefix.

## Best Possible Solution

Add a shared or local canonical model-ref formatter for runtime policy checks, use it in `runtime.llm` diagnostics, and cover the OpenRouter provider-qualified case with a regression test.

## Reproduction Assessment

Yes for a source-level reproduction: a provider `openrouter` selection with model id `openrouter/gpt-5.4-mini` flows into string interpolation that can produce `openrouter/openrouter/gpt-5.4-mini`. I did not execute a live gateway run in this read-only review.

## Solution Assessment

Yes, a narrow normalization/formatting fix in the runtime LLM policy path is the best fit. The Lossless provider-precedence problem is separately tracked upstream and should not be solved in OpenClaw core here.

## Repair Work Prompt

Fix the runtime LLM allowlist diagnostic/model-key normalization bug reported in https://github.com/openclaw/openclaw/issues/84887. In `src/plugins/runtime/runtime-llm.runtime.ts`, stop formatting normalized provider/model refs with raw `${provider}/${model}` when the model id may already carry the provider prefix; use `modelKey` from `src/agents/model-ref-shared.ts` or an equivalent shared helper that avoids `openrouter/openrouter/...` while preserving valid multi-segment OpenRouter model ids such as `openrouter/anthropic/claude-sonnet-4-6`. Add focused regression coverage in `src/plugins/runtime/runtime-llm.runtime.test.ts` for a plugin/runtime LLM model override where selection is provider `openrouter`, model id `openrouter/gpt-5.4-mini`, and `allowedModels` includes `openrouter/gpt-5.4-mini`; assert the allowlist check and denial diagnostic use the canonical single-prefixed ref. Check the adjacent plugin subagent fallback path in `src/gateway/server-plugins.ts` for the same helper pattern, but do not broaden behavior beyond model-ref formatting. Do not change Lossless provider precedence behavior; that is tracked upstream in https://github.com/Martian-Engineering/lossless-claw/issues/738. Add a changelog fix entry if maintainer policy for user-visible diagnostics applies.
