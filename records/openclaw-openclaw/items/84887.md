## Summary

Keep open: current main still formats runtime LLM allowlist and diagnostic refs with raw provider/model interpolation, so a provider-qualified OpenRouter model id can surface as `openrouter/openrouter/...`. The reporter narrowed the upstream Lossless provider-precedence issue separately, leaving this as a focused OpenClaw diagnostics bug.

## What This Changes

Fix runtime LLM allowlist normalization and diagnostics so provider-qualified model ids are not reported with a duplicated provider prefix.

## Best Possible Solution

Use the existing `modelKey` behavior or a small shared formatter for runtime LLM policy refs, add the OpenRouter provider-qualified regression, and leave Lossless provider precedence to its upstream issue.

## Reproduction Assessment

Yes for a source-level reproduction: a provider `openrouter` selection with model id `openrouter/gpt-5.4-mini` reaches interpolation that can produce `openrouter/openrouter/gpt-5.4-mini`. I did not execute a live gateway run in this read-only review.

## Solution Assessment

Yes, a narrow normalization/formatting fix in the runtime LLM policy path is the maintainable solution. The Lossless provider-precedence problem is separately owned upstream and should not be solved in OpenClaw core here.
