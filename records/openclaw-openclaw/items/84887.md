## Summary

Keep open: current main still formats runtime LLM allowlist checks with raw provider/model interpolation, so a selection like provider `openrouter` plus model id `openrouter/gpt-5.4-mini` can become `openrouter/openrouter/gpt-5.4-mini`. The reporter narrowed the Lossless provider-precedence problem to the upstream Lossless issue, leaving this as a focused OpenClaw diagnostics bug.

## What This Changes

Normalize runtime LLM allowlist checks and diagnostics so provider-qualified model ids are not reported with a duplicated provider prefix.

## Best Possible Solution

Use one canonical provider/model formatter for runtime policy comparisons and diagnostics, preferably reusing the existing `modelKey` behavior, and add a regression for provider `openrouter` with model id `openrouter/gpt-5.4-mini`.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: a resolved selection with provider `openrouter` and model id `openrouter/gpt-5.4-mini` flows through `${provider}/${model}` formatting and can produce `openrouter/openrouter/gpt-5.4-mini`. I did not execute a live gateway run in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow normalization/formatting repair in the runtime LLM policy path, not a broad Lossless provider-precedence change. The upstream Lossless precedence issue should remain separate.

## Repair Work Prompt

Fix OpenClaw issue https://github.com/openclaw/openclaw/issues/84887. In `runtime.llm.complete`, normalize provider/model refs for allowlist comparisons and error diagnostics without duplicating a provider prefix when `normalizeModelRef(provider, modelId)` returns a model id already prefixed by the same provider. Prefer reusing the existing `modelKey` behavior from the model-ref helpers rather than adding an ad hoc formatter. Add a regression in `src/plugins/runtime/runtime-llm.runtime.test.ts` where provider selection resolves to provider `openrouter`, model id `openrouter/gpt-5.4-mini`, and `plugins.entries.<id>.llm.allowedModels` contains `openrouter/gpt-5.4-mini`; the allowlist check should not produce or require `openrouter/openrouter/gpt-5.4-mini`. Consider whether the same formatter should also be used by the adjacent subagent override policy in `src/gateway/server-plugins.ts`, but do not change provider precedence or Lossless-specific `LCM_SUMMARY_MODEL` behavior here; that is tracked upstream in https://github.com/Martian-Engineering/lossless-claw/issues/738 and separately in https://github.com/openclaw/openclaw/issues/83465. Add a changelog entry only if maintainers consider this user-visible diagnostics behavior release-note-worthy.
