## Summary

Keep open: current main and the latest release still build the runtime LLM allowlist and denial refs with raw provider/model interpolation, and multiple open PRs already reference this issue with closing syntax, so it should close only after one fix lands.

## What This Changes

The issue asks OpenClaw to normalize runtime LLM allowlist diagnostics so provider-qualified model IDs do not appear as double-prefixed refs.

## Best Possible Solution

Land one focused fix that uses the existing canonical modelKey behavior for runtime LLM allowlist checks and diagnostics, with a regression for the OpenRouter provider-qualified shape and an explicit decision on the adjacent gateway fallback path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: provider `openrouter` plus model ID `openrouter/gpt-5.4-mini` flows through normalizeModelRef and then raw interpolation, producing `openrouter/openrouter/gpt-5.4-mini`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is narrow: reuse or mirror modelKey after normalizeModelRef in the runtime LLM policy path and add a regression test. The Lossless provider-precedence behavior is separately tracked upstream and should not be solved in OpenClaw core here.
