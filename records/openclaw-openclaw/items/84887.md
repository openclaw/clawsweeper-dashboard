## Summary

Keep open: current main and the latest release still format runtime LLM policy refs with raw provider/model interpolation, so the reported double-prefix diagnostic remains source-reproducible. Three open closing PRs already target the same bug, so this issue should stay open until one canonical fix lands: https://github.com/openclaw/openclaw/pull/84934, https://github.com/openclaw/openclaw/pull/84940, or https://github.com/openclaw/openclaw/pull/84946.

## What This Changes

The issue requests consistent runtime LLM allowlist normalization so provider-qualified model ids do not appear as duplicated provider refs in policy checks or diagnostics.

## Best Possible Solution

Land one narrow modelKey-based policy normalization fix with regression coverage for provider-qualified OpenRouter-style refs, keeping the Lossless provider-precedence issue separate.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: provider openrouter plus modelId openrouter/gpt-5.4-mini flows through normalizeModelRef and then raw interpolation into openrouter/openrouter/gpt-5.4-mini. I did not run a live gateway/plugin reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing modelKey-style canonical formatter in runtime LLM policy checks and diagnostics, with focused regression coverage. The broader Lossless provider-precedence problem remains external to OpenClaw core for this item.
