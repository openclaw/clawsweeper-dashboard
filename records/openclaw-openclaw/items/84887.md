- `P2`: The issue is a valid runtime diagnostics and allowlist-matching bug with a narrow provider/model policy surface.
- `impact:auth-provider`: The affected behavior is provider/model override policy and diagnostics for plugin runtime LLM calls.

## Summary

Keep open: current main still formats runtime LLM allowlist refs with raw provider/model interpolation, so the reported double-prefix diagnostic remains source-reproducible. The issue already has open closing PRs, so it should stay open until one of those fixes lands.

## What This Changes

Fix runtime LLM allowlist normalization so provider-qualified model ids are not reported or checked with a duplicated provider prefix.

## Best Possible Solution

Land one focused PR that uses the canonical `modelKey` behavior for runtime LLM allowlist checks and diagnostics, with a regression for provider-qualified OpenRouter refs and an explicit scope decision for the gateway fallback policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `provider=openrouter` plus `modelId=openrouter/gpt-5.4-mini` flows through raw `${provider}/${model}` formatting and can become `openrouter/openrouter/gpt-5.4-mini`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or mirror `modelKey` for the runtime policy key/error path and cover the OpenRouter provider-qualified case with a regression test; the Lossless provider-precedence issue remains external to OpenClaw.
