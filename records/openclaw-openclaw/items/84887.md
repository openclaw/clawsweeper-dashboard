- `impact:auth-provider`: The issue concerns provider/model override policy and model-choice diagnostics for plugin runtime LLM calls.

## Summary

Keep open: current `main` and the latest release still build the runtime LLM allowlist and diagnostic refs with raw provider/model interpolation, and two open PRs are linked as fixes but neither has merged. The report is narrowly scoped to OpenClaw host-side diagnostics; the Lossless provider-precedence side is already tracked upstream at https://github.com/Martian-Engineering/lossless-claw/issues/738.

## What This Changes

The issue requests canonical provider/model normalization for runtime LLM allowlist checks and diagnostics so provider-qualified model IDs do not render or match as double-prefixed refs.

## Best Possible Solution

Land or supersede with a focused patch that uses `modelKey()` for both allowlist entries and resolved model refs in the runtime policy paths, with regression coverage for provider-qualified OpenRouter model IDs, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: current `main` normalizes `provider=openrouter` plus `modelId=openrouter/gpt-5.4-mini` and then interpolates provider/model again for the policy key and diagnostic. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow canonical-key change already proposed by the linked focused PR, plus regression tests for the runtime LLM and related gateway policy diagnostics. The upstream Lossless provider-precedence bug should remain outside this OpenClaw fix.
