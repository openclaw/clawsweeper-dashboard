## Summary

Keep open: current main and the v2026.5.19 release still build runtime LLM allowlist diagnostics by interpolating provider/model after normalization, so an OpenRouter-shaped model id can be reported as `openrouter/openrouter/...`. The discussion correctly narrows the OpenClaw work to host-side diagnostics while leaving the Lossless provider-precedence incident upstream.

## What This Changes

Normalize runtime LLM allowlist checks and diagnostics so provider-qualified model IDs are not rendered with a duplicated provider prefix.

## Best Possible Solution

Add a shared runtime model-ref formatting/normalization helper for allowlist comparison and errors, then cover `provider="openrouter"`, `modelId="openrouter/gpt-5.4-mini"`, and `allowedModels=["openrouter/gpt-5.4-mini"]` with a focused regression test.

## Reproduction Assessment

Yes from source, though not executed live in this read-only pass. The current path can resolve provider `openrouter` with model id `openrouter/gpt-5.4-mini` and then formats it as `openrouter/openrouter/gpt-5.4-mini` before the allowlist check.

## Solution Assessment

Yes. The narrow maintainable fix is host-side canonical formatting for runtime LLM allowlist comparison and diagnostics, while the Lossless provider-precedence issue should remain in the external Lossless tracker.
