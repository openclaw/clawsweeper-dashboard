## Summary

Keep open: current `main` and the latest release still build runtime LLM allowlist refs with raw provider/model interpolation, so the reported `openrouter/openrouter/...` diagnostic remains source-reproducible. The issue is already linked to active closing PRs, so it should stay open until one focused fix lands.

## What This Changes

The issue requests that runtime LLM allowlist checks and diagnostics avoid doubled provider prefixes when a resolved model id is already provider-qualified.

## Best Possible Solution

Consolidate around one focused fix that uses the existing canonical model-ref formatter in runtime LLM policy checks and diagnostics, with regression coverage for provider-qualified OpenRouter-style refs.

## Reproduction Assessment

Yes. Source inspection on current `main` shows provider `openrouter` plus model id `openrouter/gpt-5.4-mini` can be formatted as `openrouter/openrouter/gpt-5.4-mini` before the allowlist check; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable solution is to reuse the existing `modelKey()` formatter in the runtime policy path and add a regression test; the Lossless provider-precedence bug remains correctly tracked upstream.
