## What This Changes

The issue asks OpenClaw's read tool to access macOS paths whose directories or filenames mix CJK and ASCII characters instead of failing with ENOENT after spaces appear at character-boundaries.

## Best Possible Solution

Run a live macOS/oMLX/Qwen mixed-path read-tool check with raw tool-call arguments captured; fix OpenClaw if it mutates exact arguments, otherwise keep de-spacing as an opt-in plugin/hook policy or a maintainer-approved guarded fallback.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection shows a path that reaches the resolver with inserted ordinary spaces will miss deterministically, but it does not prove where the mutation originates on current macOS/oMLX.

## Solution Assessment

Unclear. There is no proposed patch, and the safest fix depends on proving whether the mutation happens in OpenClaw argument handling, a provider adapter, or the local model/backend; a blanket de-spacing heuristic is not clearly safe as a core default.
