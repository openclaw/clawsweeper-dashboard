## Summary

Keep open. Current main and v2026.5.12 still forward read-tool path arguments as supplied, and the adjacent malformed-tool-call repair only handles invalid JSON, so a valid tool call whose CJK+ASCII path was already space-mutated would still reach filesystem access unchanged; live macOS/oMLX proof is still missing.

## What This Changes

The issue requests that the read tool preserve and resolve macOS paths whose directory or filename mixes CJK and ASCII characters instead of failing with ENOENT after added spaces.

## Best Possible Solution

First run a live macOS/oMLX/Qwen mixed-path read-tool check, then either fix exact argument preservation with focused regression coverage or add guarded diagnostics without fuzzy filesystem guessing.

## Reproduction Assessment

No. The issue body has concrete user steps and source inspection shows current main would fail if it receives the reported space-mutated argument, but I did not establish a live macOS/oMLX reproduction of the mutation on current main.

## Solution Assessment

Unclear. There is no proposed patch, and the safest fix boundary depends on whether the mutation occurs in the local backend, provider/tool-call transport, or OpenClaw argument handling; a broad path-guessing fallback is not the best first fix.
