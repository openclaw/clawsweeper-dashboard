## Summary

Keep open. Current main and v2026.6.8 still do not prove a fix for the reported CJK+ASCII path argument mutation; the read path code preserves the supplied string, so the missing evidence is a live macOS/oMLX trace showing where the argument is being changed before the read tool receives it.

## What This Changes

The issue reports ENOENT failures when the read tool is asked to read macOS paths whose directory or filename mixes CJK and ASCII characters.

## Best Possible Solution

Capture raw read-tool arguments in a live macOS/oMLX repro; if OpenClaw mutates them, fix the argument-preservation layer, and if the model emits already-spaced paths, keep core unchanged or approve an opt-in plugin/resolver policy with tests.

## Reproduction Assessment

No, not with high confidence. The reporter provided plausible macOS/oMLX steps and logs, but current source only proves that a path already mutated with inserted spaces will fail; it does not prove where the mutation happens on current main.

## Solution Assessment

Unclear. A narrow argument-preservation fix would be best if a live trace proves OpenClaw changes the path, while a global CJK/ASCII de-spacing heuristic is a product decision and should not be automated as a default core fix.
