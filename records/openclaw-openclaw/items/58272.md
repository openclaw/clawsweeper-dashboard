## Summary

Keep open. Current main still appears to pass read-tool path arguments through without a CJK/ASCII boundary repair, and there is no merged duplicate or shipped fix; the missing piece is live macOS/oMLX proof of where the path string is mutated.

## What This Changes

The issue asks OpenClaw's read tool to preserve mixed CJK+ASCII directory and filename paths on macOS instead of failing with ENOENT after spaces are inserted.

## Best Possible Solution

Add focused regression coverage for mixed CJK+ASCII read-tool arguments and the local-provider tool-call boundary, then fix argument preservation or emit precise diagnostics without broad filesystem guessing.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows that if current main receives the reported space-mutated path as valid JSON, it forwards that value to the read tool and would hit ENOENT for the real unspaced filename.

## Solution Assessment

Unclear. There is no proposed patch, and the safest fix boundary appears to be provider/tool-call argument handling or diagnostics rather than a read-tool fallback that guesses at filenames.
