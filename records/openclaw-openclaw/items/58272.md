## Summary

Keep open. Current main still appears to pass read-tool paths exactly as supplied, and the adjacent tool-call repairs do not correct a valid JSON argument whose CJK+ASCII path text has already been space-mutated.

## What This Changes

The issue reports macOS read-tool ENOENT failures when directory names or filenames mix CJK and ASCII characters because the path appears to gain inserted spaces before filesystem access.

## Best Possible Solution

Add focused regression coverage for mixed CJK+ASCII read-tool arguments and the local-provider tool-call boundary, then fix argument preservation or emit precise diagnostics without broad filesystem guessing.

## Reproduction Assessment

No high-confidence live reproduction is available yet. Source inspection shows that if current main receives the reported space-mutated path as valid JSON, it forwards that path unchanged to the read tool, but the macOS/oMLX mutation still needs live proof.

## Solution Assessment

Unclear. There is no proposed patch, and the safest fix boundary depends on whether the mutation occurs in provider/tool-call argument generation, OpenClaw stream handling, or local backend behavior.
