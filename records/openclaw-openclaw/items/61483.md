Full review comments:

- **[P1] Require policy before defaulting Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises it fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config or a reviewed destination policy. Keep it out of defaults, or add and document the reviewed gateway policy before default-allowing this egress path.
  - confidence: 0.9
- **[P2] Refresh command declarations when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is already connected can leave `http.request` undeclared until reconnect or reapproval, making the setting appear broken on the normal off-to-on path.
  - confidence: 0.86
- **[P2] Expose HTTP response truncation to callers:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:155-160`
  - body: `readResponseBody` stops at the 5 MB cap but returns the partial body without any `truncated` signal. Callers can receive invalid partial JSON or HTML with no way to distinguish it from a complete response, so the result should expose truncation explicitly or fail with a clear error.
  - confidence: 0.78
