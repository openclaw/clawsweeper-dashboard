## Summary

Keep open: current main still lacks this feature, and the PR remains meaningful, but it should not merge until maintainers choose the Android HTTP egress policy and fix the connect-time command refresh path.

## What This Changes

The PR adds an Android `http.request` node command with a native HTTP handler, Settings toggle, gateway allowlist entry, protocol constants, and Android/gateway tests.

## Best Possible Solution

Land a narrower implementation after the Android HTTP egress policy is chosen, then enforce that policy in the gateway/handler and refresh or reset the active node command snapshot when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source default-allows `http.request`, validates only URL scheme, and only persists the toggle while current main checks active session declared commands. I did not run a live Android E2E path in this read-only review.

## Solution Assessment

No. The Android handler is a plausible owner, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary until maintainers choose the egress and destination policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit policy before defaulting Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises it fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config or a reviewed destination policy. Keep it out of defaults, or add and document the reviewed gateway policy before default-allowing this egress path.
  - confidence: 0.9
- **[P2] Refresh the node session when HTTP Access turns on:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is already connected can leave `http.request` undeclared until reconnect or re-approval, making the toggle appear broken on the normal off-to-on path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces a new Android HTTP proxy/egress path without a reviewed gateway default or destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`

  - confidence: 0.9

  - confidence: 0.86
