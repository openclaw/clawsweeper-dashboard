## Summary

Keep open: this is a plausible Android/gateway feature PR, but current main does not already implement it and the PR still needs maintainer security/product policy for Android HTTP egress plus a command-surface refresh fix before merge.

## What This Changes

The PR adds an Android `http.request` node command with an Android HTTP handler, Settings toggle, gateway Android allowlist entry, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose and encode or document the Android HTTP egress policy, keep the command explicitly gated or destination-limited, and refresh or reconnect the active node session when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and a preference-only toggle setter while current main forwards only commands declared by the active session.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for this new egress feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require policy before default-allowing HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android defaults lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config. Keep this behind `gateway.nodes.allowCommands`, or add and document a reviewed destination policy before allowing it by default.
  - confidence: 0.88
- **[P2] Refresh commands when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway authorizes invokes against the command list captured by the active node session. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, so the new setting can look broken on the normal off-to-on path.
  - confidence: 0.9
