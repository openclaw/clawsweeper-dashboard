## Summary

This should stay open because current `main` does not implement the Android HTTP command, and the PR still introduces a new cross-network Android HTTP egress path that needs explicit security/product policy review before merge.

## What This Changes

The PR adds an Android `http.request` node invoke command with a handler, Android settings/runtime gating, gateway Android allowlisting, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose the Android HTTP egress policy, keep raw `http.request` explicitly opt-in or destination-limited, and make the Android setting update the gateway-visible command set predictably.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows Android default allowlisting, scheme-only URL validation, and a preference-only toggle setter. Current main shows `node.invoke` authorizes against the active session's declared command list before forwarding.

## Solution Assessment

No; Android is a plausible runtime owner, but default gateway allowlisting plus arbitrary HTTP(S) destinations is not the narrowest maintainable policy for a new cross-network egress feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require policy before default-allowing HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config. Keep this behind explicit `gateway.nodes.allowCommands`, or add and document a reviewed destination policy before allowing it by default.
  - confidence: 0.88
- **[P2] Refresh commands when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway authorizes invokes against the command list captured by the active node session. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, so the new setting can look broken on the normal off-to-on path.
  - confidence: 0.9

## Security Review
