Full review comments:

- **[P1] Require policy before defaulting Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` makes this phone-network HTTP proxy available by default whenever the Android app advertises it, while the handler accepts arbitrary HTTP(S) URLs. Keep this command explicit via `gateway.nodes.allowCommands`, or add the reviewed destination/operator policy before default-allowing it.
  - confidence: 0.9
- **[P1] Constrain HTTP targets before shipping the proxy:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: `validateUrl` only checks the scheme, so a gateway invoke can make the phone fetch loopback, private, link-local, metadata, or other local-network endpoints reachable from the device. If this feature is intended for public web fetches or sensitive handoff only, enforce that destination policy here before opening the connection.
  - confidence: 0.86
- **[P2] Refresh command declarations when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands at connect time and the gateway checks the active session's command list. Enabling HTTP Access after the node is already connected can leave `http.request` undeclared until reconnect or reapproval, making the normal off-to-on path appear broken.
  - confidence: 0.86
- **[P2] Expose body truncation in the HTTP result:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:155-160`
  - body: `readResponseBody` stops at the 5 MB cap but returns the partial body without any `truncated` signal. Callers can receive invalid partial JSON or HTML with no way to distinguish it from a complete response, so the result should expose truncation explicitly or fail with a clear error.
  - confidence: 0.78
