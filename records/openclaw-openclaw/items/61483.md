## Summary

Keep open for maintainer review: the feature is real and not on current main, but the PR still adds a security-sensitive Android HTTP egress path without a settled gateway/default destination policy and has a connected-session toggle bug.

## What This Changes

The PR adds an Android `http.request` node command, Android HTTP handler and Settings toggle, a gateway Android allowlist entry, and Android/gateway tests.

## Best Possible Solution

Land a narrower implementation only after maintainers choose the Android HTTP egress policy, keep the command explicitly gated or destination-limited as needed, and refresh or reconnect the active node session when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and a preference-only toggle setter while current main forwards only commands declared by the active session. I did not run Android locally in this read-only review.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations and no active-session refresh is not the narrowest maintainable boundary for this feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without an explicit operator or destination policy. Keep it out of defaults, or add and document a reviewed policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh the node session after enabling HTTP access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in `buildNodeConnectOptions()` at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, making the normal off-to-on path appear broken.
  - confidence: 0.9

## Security Review
