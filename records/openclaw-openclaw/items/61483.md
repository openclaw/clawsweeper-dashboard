## Summary

Keep this PR open, but it is not merge-ready: it is the canonical implementation path for the Android HTTP-node feature, yet the current diff adds a new cross-network egress path as an Android default without a reviewed destination policy and leaves the enabled switch unavailable to already-connected sessions until reconnect.

## What This Changes

The PR adds an Android `http.request` node command, handler, capability advertising, Settings toggle, gateway allowlisting, and Android/gateway tests.

## Best Possible Solution

Land only after maintainers choose the Android HTTP egress policy, make `http.request` explicit or destination-policy-backed, refresh or intentionally reconnect the node session when access changes, and document the shipped behavior.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows default Android allowlisting, scheme-only URL validation, and no active-session refresh when HTTP Access changes. Current gateway source shows the command list is captured from `connect.commands` at registration time.

## Solution Assessment

No. Android may be an appropriate owner for this capability, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require egress policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `http.request` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises it fetch arbitrary HTTP(S) URLs from the phone network without explicit operator allowlisting or destination policy. Keep this command explicitly allowlisted, or add a reviewed destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only updates prefs, but the gateway checks the command list captured by the active node session before forwarding invokes. Turning the switch on after connecting leaves `http.request` undeclared until reconnect, so the visible enable path can appear broken.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a general Android HTTP egress path without a reviewed default/destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once the Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config. That creates a new cross-network egress path from the phone network.
  - confidence: 0.88
- **[medium] Handler accepts any HTTP(S) destination:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: The handler filters only by URL scheme, leaving loopback, private, link-local, metadata, and local-network targets reachable from the Android device unless maintainers deliberately choose and document that policy.
