## Summary

Keep open: current main does not implement `http.request`, and this PR is the canonical implementation path, but the exact head still needs maintainer security/product judgment for Android HTTP egress policy plus a session-refresh fix before merge.

## What This Changes

The PR adds an Android `http.request` node command with an HTTP handler, Settings toggle, capability advertising, gateway allowlisting, and Android/gateway tests.

## Best Possible Solution

Land only after maintainers choose and document the Android HTTP egress policy, make forwarding explicit or destination-policy-backed, refresh or reconnect when access changes, and rerun Android/gateway validation.

## Reproduction Assessment

Yes for the review blockers: exact-head PR source shows default Android allowlisting, scheme-only URL validation, and no active-session refresh. Current and PR-head gateway source shows the connected `nodeSession.commands` snapshot is checked before forwarding invokes.

## Solution Assessment

No. Android may be the right owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require egress policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without explicit operator allowlisting or destination policy. Keep this behind `gateway.nodes.allowCommands`, or add a reviewed destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway checks the command list captured by the active node session before forwarding invokes. Turning the switch on after connecting leaves `http.request` undeclared until reconnect, so the visible enable path can appear broken.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a general Android HTTP egress path without a reviewed default or destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once the Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config. That creates a new cross-network egress path from the phone network.
  - confidence: 0.88
