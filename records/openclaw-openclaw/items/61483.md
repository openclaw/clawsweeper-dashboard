Keep this PR open as the canonical implementation candidate; the linked feature issue was closed in favor of this PR and current main still has no Android `http.request` command. ## What This Changes

The branch adds an Android `http.request` node capability, handler, Settings toggle, gateway Android allowlist entry, command constant, and Android/gateway tests.

## Best Possible Solution

Land a narrower implementation after maintainers choose the Android HTTP egress policy, enforce that policy in both gateway and handler code, refresh or reapprove command snapshots when the toggle changes, expose truncation explicitly, and document the operator/security model.

## Solution Assessment

No. The Android handler is a plausible owner for the capability, but default gateway allowlisting plus unconstrained destinations is not the narrowest maintainable solution until maintainers choose the egress policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require policy before defaulting Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises it fetch HTTP(S) destinations from the phone network without explicit operator config or a reviewed destination policy. Keep it out of defaults, or add and document the reviewed gateway policy before default-allowing this egress path.
  - confidence: 0.9
- **[P1] Constrain HTTP destinations before proxying requests:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: `validateUrl` only checks for `http` or `https`, so this command can reach loopback, private, link-local, metadata, or other local-network targets visible to the phone. Add destination filtering or an explicit maintainer-approved policy before exposing this as a gateway-invoked command.
  - confidence: 0.88
- **[P2] Refresh commands after toggling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is already connected can leave `http.request` undeclared until reconnect or reapproval, making the setting appear broken on the normal off-to-on path.
  - confidence: 0.86
- **[P2] Expose when HTTP responses are truncated:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:155-160`
  - body: `readResponseBody` stops at the 5 MB cap but returns the partial body without any `truncated` signal. Callers can receive invalid partial JSON or HTML with no way to distinguish it from a complete response, so the result should expose truncation explicitly or fail with a clear error.
  - confidence: 0.78
