## Summary

Keep this PR open, but it should not merge as-is: the Android HTTP proxy capability is real and has proof, but the current diff creates a new cross-network egress path without a settled destination policy and leaves the connected-session enable path stale until reconnect.

## What This Changes

This PR adds an Android `http.request` node invoke command with handler, settings/prefs, registry/dispatcher wiring, gateway allowlist support, and Android/gateway tests.

## Best Possible Solution

Land only after maintainers choose the egress policy, make `http.request` explicit or destination-policy-backed, refresh or reconnect the node session when access changes, and document the shipped behavior.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows default Android allowlisting, scheme-only URL validation, and no active-session refresh when HTTP Access changes; current gateway code shows why the connected-session path fails until reconnect.

## Solution Assessment

No. Android is a plausible owner for this capability, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable security boundary for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require egress policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `http.request` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises it fetch arbitrary HTTP(S) URLs from the phone network without explicit operator allowlisting or destination policy. Keep this command dangerous/explicitly allowlisted, or add a reviewed destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only updates prefs, but the gateway checks the command list captured by the active node session before forwarding invokes. Turning the switch on after connecting leaves `http.request` undeclared until reconnect, so the visible enable path appears broken.
  - confidence: 0.9
