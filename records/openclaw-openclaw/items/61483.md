## Summary

Keep open. Current main does not implement the Android `http.request` node command, and this PR still needs security/product judgment plus a small behavior fix before it is mergeable, even though the linked demo is useful evidence for the intended workflow.

## What This Changes

The PR adds an Android `http.request` node command, Android HTTP handler, Settings toggle, gateway command allowlist entry, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose the Android HTTP egress policy, with explicit gateway/operator gating or destination limits as needed and a session refresh or reconnect flow when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only destination validation, and a preference-only toggle setter, while current main forwards only commands declared by the active node session. I did not run Android locally in this read-only review.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations and no active-session refresh is not the narrowest maintainable boundary for this feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any approved Android node that advertises it fetch arbitrary HTTP(S) destinations through the phone network without an explicit operator or destination policy. Keep it out of defaults, or add and document a reviewed policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access is enabled:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in its connect options and the gateway checks the active session's declared commands. Enabling HTTP Access while already connected can leave `http.request` undeclared until reconnect or reapproval, making the new switch appear broken on the normal off-to-on path.
  - confidence: 0.9

## Security Review
