## Summary

Keep open: current main still lacks this feature, and the PR needs maintainer product/security policy before merge because it default-allows a new Android HTTP egress path and accepts any HTTP(S) destination from the phone network.

## What This Changes

The PR adds an Android `http.request` node invoke command with protocol constants, an Android HTTP handler, Settings toggle, gateway Android allowlist entry, and Android/gateway tests.

## Best Possible Solution

Choose the Android HTTP egress and destination policy first, then land a revised implementation that applies that policy in the gateway and makes the Android toggle update the active session or clearly require reconnect/re-approval.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and a preference-only toggle while current main checks active session declared commands. I did not run a live Android E2E path in this read-only review.

## Solution Assessment

No. The Android runtime handler may be the right owner, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary until maintainers choose the security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any paired Android node that advertises it fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config or a reviewed destination policy. Keep it out of defaults, or add and document the reviewed gateway policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh session commands after enabling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in `buildNodeConnectOptions()` at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect or re-approval, making the new toggle appear broken on the normal off-to-on path.
  - confidence: 0.86

## Security Review

Status: needs_attention
