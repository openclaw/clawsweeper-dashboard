## Summary

Keep this PR open: current main does not implement the Android `http.request` node command, and the PR is still the canonical implementation path, but it needs security/product policy decisions and a session-refresh fix before merge.

## What This Changes

The PR adds an Android `http.request` node command with a handler, capability advertising, Settings opt-in, gateway allowlisting, and Android/gateway tests.

## Best Possible Solution

Land only after maintainers choose the Android HTTP egress policy, make `http.request` explicit or destination-policy-backed, refresh or intentionally reconnect the node session when access changes, and document the shipped behavior.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows default Android allowlisting, scheme-only URL validation, and no active-session refresh when HTTP Access changes. Current gateway source shows `connect.commands` is captured into the node session and checked before forwarding invokes.

## Solution Assessment

No. Android may be an appropriate owner for this capability, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require egress policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` allows any paired Android node that advertises it to fetch arbitrary HTTP(S) destinations from the phone network without explicit operator allowlisting or destination policy. Keep this command behind `gateway.nodes.allowCommands`, or add a reviewed destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway checks the command list captured by the active node session before forwarding invokes. Turning the switch on after connecting leaves `http.request` undeclared until reconnect, so the visible enable path can appear broken.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a general Android HTTP egress path without a reviewed default or destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`

  - confidence: 0.88
