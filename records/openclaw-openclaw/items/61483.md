## Summary

Keep this PR open: current main does not implement the feature, and the branch contains useful Android work and linked real-behavior proof, but the proposed default Android HTTP egress path still needs maintainer security/product policy and an active-session toggle fix before merge.

## What This Changes

The PR adds an Android `http.request` node invoke command, Android HTTP execution and Settings toggle wiring, gateway allowlist support, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose the Android HTTP egress and destination policy, then refresh or reconnect the active node session when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows Android default allowlisting, scheme-only URL validation, and a preference-only toggle setter while current main checks active session declared commands. I did not run a live Android E2E path in this read-only review.

## Solution Assessment

No. The Android handler may be the right runtime owner, but default gateway allowlisting plus unconstrained HTTP(S) destinations and no active-session command refresh is not the narrowest maintainable boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any paired Android node that advertises it fetch arbitrary HTTP(S) destinations from the phone network without a reviewed operator or destination policy. Keep it out of defaults, or add and document the reviewed policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh session commands after toggling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in `buildNodeConnectOptions()` at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, making the new toggle appear broken on the normal off-to-on path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a new Android HTTP egress/proxy path without a reviewed gateway default or destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once an Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config or reviewed destination policy.
  - confidence: 0.88
- **[medium] Handler accepts any HTTP(S) destination:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`

  - confidence: 0.84
