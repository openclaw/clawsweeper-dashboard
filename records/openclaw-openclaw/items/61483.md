## Summary

Keep open: the feature is useful and has linked proof, but PR-head source still default-allows a new Android HTTP egress path without a reviewed gateway or destination policy and does not refresh active session commands after the HTTP Access toggle changes.

## What This Changes

The branch adds an Android `http.request` node invoke command, Android UI/preferences gating, gateway allowlist support, and Android/gateway tests.

## Best Possible Solution

Land a narrower version after maintainers choose the Android HTTP egress policy, then refresh active command declarations when HTTP Access changes and align tests/docs with that boundary.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the Android default allowlist, scheme-only URL validation, and preference-only toggle setter while current main forwards only commands declared by the active session. I did not run a real Android device in this read-only review.

## Solution Assessment

No. Android is a plausible runtime owner for the handler, but default gateway allowlisting plus unrestricted HTTP(S) destinations is not the narrowest maintainable boundary for this feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets a paired Android node that advertises it fetch arbitrary HTTP(S) URLs from the phone network without explicit gateway/operator config or destination policy. Keep it out of defaults, require existing `gateway.nodes.allowCommands`, or add and document the reviewed policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh commands after toggling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in `buildNodeConnectOptions()` at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, making the new toggle appear broken on the normal off-to-on path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention because the patch creates a phone-network HTTP proxy path without an explicit gateway or destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once an Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config.
  - confidence: 0.88
- **[medium] Handler accepts any HTTP(S) destination:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: The handler filters only by URL scheme, leaving loopback, private, link-local, metadata, and local-network targets reachable from the Android device unless maintainers deliberately choose and document that policy.
  - confidence: 0.84
