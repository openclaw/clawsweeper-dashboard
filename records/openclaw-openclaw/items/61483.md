## Summary

This PR should stay open: it adds a plausible Android/gateway capability, but the branch still needs maintainer security policy direction before merge and has a concrete session-refresh bug in the normal enable path.

## What This Changes

The PR branch adds an Android `http.request` node command, Android prefs/UI gating, handler and dispatcher wiring, gateway allowlist support, and Android/gateway tests.

## Best Possible Solution

Land an Android-owned implementation only after maintainers choose the egress policy, likely by making the command explicitly allowlisted or policy-backed with destination rules, then refresh the connected node session and document the command before shipping.

## Reproduction Assessment

Yes for the review blockers. The PR head source shows default Android allowlisting and scheme-only URL validation, and it shows the toggle setter only updates prefs while advertised commands are built at connect time.

## Solution Assessment

No. Android is a plausible owner for the capability, but default gateway allowlisting plus scheme-only URL validation is not the narrowest maintainable solution for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `http.request` to Android platform defaults lets any paired Android node that advertises it become an arbitrary HTTP(S) egress proxy from the phone network. Keep it out of defaults, require explicit allowlisting, or add reviewed destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway checks the command list declared by the active node session. Turning the default-off switch on while already connected leaves `http.request` unavailable until a reconnect, so the new toggle appears broken on the normal enable path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a general-purpose Android HTTP egress path without a reviewed gateway allowlist and destination policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once the Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config. That creates a new cross-network egress path from the phone's LAN and should be treated as security-sensitive before merge.
  - confidence: 0.88
- **[medium] Handler accepts any http or https destination:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: The handler filters only by URL scheme, leaving loopback, private, link-local, metadata, and local-network targets in scope from the Android device. That bypasses the gateway's existing proxy/SSRF egress assumptions unless maintainers deliberately choose that policy.
  - confidence: 0.84
