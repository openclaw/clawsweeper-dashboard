## Summary

This PR should stay open: it adds a plausible Android node HTTP proxy capability, but the current diff still needs maintainer security/product judgment before OpenClaw defaults to forwarding arbitrary HTTP(S) egress through a paired phone.

## What This Changes

The branch adds an Android `http.request` node command, an opt-in Settings toggle, Android handler/registry/dispatcher wiring, gateway Android allowlisting, and Android/gateway tests.

## Best Possible Solution

Land a narrower version after maintainers choose the Android HTTP egress policy, document or encode the destination/allowlist rules, and ensure the Settings toggle updates the gateway-visible command set or clearly reconnects.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and no active-session refresh. Current main shows `node.invoke` is gated by the active session's command list before forwarding.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for a new cross-network egress feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require egress policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without explicit operator allowlisting or a reviewed destination policy. Keep this behind explicit `gateway.nodes.allowCommands`, or add and document the intended destination policy before merge.
  - confidence: 0.88
- **[P2] Refresh the node session when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway checks the command list captured by the active node session before forwarding invokes. Turning the switch on after connecting leaves `http.request` undeclared until reconnect, so the visible enable path can appear broken.
  - confidence: 0.9

## Security Review

Status: needs_attention
