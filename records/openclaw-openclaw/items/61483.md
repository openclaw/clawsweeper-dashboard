## Summary

Keep open: current main does not implement the Android `http.request` node command and the related feature issue was closed in favor of this PR, but the PR still needs maintainer security/product judgment before it can merge because it default-allows a new Android HTTP egress path and has a source-proven connected-session toggle gap.

## What This Changes

The PR adds an Android `http.request` node invoke command with Android protocol/registry/dispatcher/UI/preferences support, a gateway allowlist entry, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose the Android HTTP egress policy, with explicit gateway/operator gating or destination limits as needed and an active-session command refresh when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and a preference-only toggle setter while current main forwards only commands declared by the active session. I did not run Android locally in this read-only review.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations and no active-session refresh is not the narrowest maintainable boundary for this feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android platform defaults lets any paired Android node that advertises it fetch arbitrary HTTP(S) destinations from the phone network without an explicit gateway/operator or destination policy. Keep it out of defaults, or add and document the reviewed policy before default-allowing this egress path.
  - confidence: 0.88
- **[P2] Refresh session commands after toggling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the node declares commands in `buildNodeConnectOptions()` at connect time and the gateway checks the active session's declared commands. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, making the new toggle appear broken on the normal off-to-on path.
  - confidence: 0.9
