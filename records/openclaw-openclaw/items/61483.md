## Summary

Keep open. Current main does not implement `http.request`, and the PR is not merge-ready because it adds a general Android HTTP egress path without an accepted gateway/node security policy, plus a connected-session refresh gap and missing docs/changelog.

## What This Changes

The PR adds an Android `http.request` node invoke command, Android HTTP Access prefs/UI, gateway Android allowlist support, protocol constants, handlers, and Android/gateway tests.

## Best Possible Solution

Accept the Android-owned implementation only after maintainers choose the egress policy, likely by making `http.request` explicitly opt-in or policy-backed with destination rules, then refresh the connected node session and document the command before shipping.

## Reproduction Assessment

Yes. The PR head source shows `http.request` is default-allowlisted for Android and accepts any `http`/`https` destination, while the gateway invokes only commands declared in the current node session; enabling HTTP Access only writes prefs and does not refresh that session.

## Solution Assessment

No. Android is a plausible owner for this capability, but default gateway allowlisting plus scheme-only URL validation is not the narrowest maintainable solution for a new cross-network egress path.

  - body: Adding `http.request` to Android platform defaults means any paired Android node that advertises it can become an arbitrary HTTP(S) egress proxy without explicit gateway config or reviewed destination policy. Turning the default-off switch on while already connected leaves `http.request` undeclared until a reconnect, so the new toggle appears broken on the normal off-to-on path.
  - confidence: 0.9

  - body: This adds a public Android setting, node command, and JSON request/response contract, but the branch does not update the changelog or node/Android docs. Operators need the opt-in behavior, parameters, limits, and security expectations before a user-facing feature ships.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces an opt-in but general-purpose Android HTTP proxy, and the current branch default-allows it without a reviewed egress/SSRF policy.

Concerns:

  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once the Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config. That creates a new cross-network egress path from the phone's LAN and should be treated as security-sensitive before merge.
