## Summary

Keep open. Current main does not implement the Android `http.request` node command, and the PR still needs maintainer security/product policy before merge because it adds a new Android HTTP egress path with default gateway allowlisting and no destination policy.

## What This Changes

The PR adds an Android `http.request` node command, Android HTTP handler, Settings toggle, gateway allowlist entry, and Android/gateway tests.

## Best Possible Solution

Land a narrower version only after maintainers choose the Android HTTP egress and destination policy, then keep command advertisement and active-session refresh behavior consistent with existing node command gates.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows Android default allowlisting, scheme-only destination validation, and a preference-only toggle setter while current main checks active session declared commands. I did not run live Android E2E proof in this read-only review.

## Solution Assessment

No. The Android handler may be the right runtime owner, but default gateway allowlisting plus unconstrained HTTP(S) destinations and no active-session command refresh is not the narrowest maintainable boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate Android HTTP egress behind explicit policy:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android `PLATFORM_DEFAULTS` makes every paired Android node that advertises the new command available as an arbitrary HTTP(S) egress path. Keep this out of defaults, or add the reviewed gateway/operator and destination policy before default-allowing it.
  - confidence: 0.88
- **[P2] Refresh commands after enabling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only saves the preference, but commands are declared in connect options and the gateway checks the active session's `commands`. Turning the switch on while already connected leaves `http.request` undeclared until reconnect, so the new setting appears broken on the normal off-to-on path.
  - confidence: 0.9
