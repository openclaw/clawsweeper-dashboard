## Summary

Keep this PR open. It is a concrete Android HTTP-node feature with recorded real-behavior proof, but it still needs maintainer security/product judgment before merge because it creates a new phone-network HTTP egress path and default-allows that command without a reviewed destination policy.

## What This Changes

The PR adds an Android `http.request` node command, Android settings/runtime gating, gateway Android allowlisting, and Android/gateway tests.

## Best Possible Solution

Land a narrower version after maintainers choose and encode or document the Android HTTP egress policy, keep the command explicitly gated or destination-limited, and refresh the active node session when HTTP Access changes.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows default Android allowlisting, scheme-only URL validation, and a preference-only toggle setter. Current main shows `node.invoke` checks the active session's declared command list before forwarding.

## Solution Assessment

No. Android may be the right runtime owner for the handler, but default gateway allowlisting plus unconstrained HTTP(S) destinations is not the narrowest maintainable boundary for a new cross-network egress feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require policy before default-allowing HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `NODE_HTTP_REQUEST_COMMAND` to Android defaults lets any paired Android node that advertises the command fetch arbitrary HTTP(S) destinations from the phone network without explicit operator config. Keep this behind `gateway.nodes.allowCommands`, or add and document a reviewed destination policy before allowing it by default.
  - confidence: 0.88
- **[P2] Refresh commands when HTTP Access changes:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only writes the preference, but the gateway authorizes invokes against the command list captured by the active node session. Enabling HTTP Access after the node is connected can leave `http.request` undeclared until reconnect, so the new setting can look broken on the normal off-to-on path.
  - confidence: 0.9
