## Summary

Keep open. Current main still lacks `http.request`, and this PR is the active implementation candidate for the use case also described in #65440, but it should not merge until maintainers decide the Android HTTP egress policy and the branch addresses the remaining toggle and documentation gaps.

## What This Changes

The PR adds Android `http.request` node command support, an HTTP Access setting, gateway allowlist wiring, protocol constants, handlers, and Android/gateway tests.

## Best Possible Solution

Accept an Android-owned implementation only after `http.request` is dangerous/explicitly opt-in or policy-backed with destination controls, refreshes advertised commands on toggle changes, and ships docs plus changelog coverage.

## Reproduction Assessment

Yes. Inspecting PR head is enough: Android platform defaults include `http.request`, `HttpHandler` accepts any `http`/`https` URL, and `setHttpAccessEnabled` only writes prefs while gateway dispatch checks `nodeSession.commands` from the active session.

## Solution Assessment

No. Android is a plausible owner for this capability, but default gateway allowlisting plus scheme-only destination validation is not the narrowest maintainable security boundary for a new cross-network egress path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require policy before default-allowing HTTP proxying:** `src/gateway/node-command-policy.ts:103`
  - body: Adding `http.request` to Android platform defaults makes any paired Android node that advertises it a generic HTTP(S) egress proxy without explicit gateway opt-in or destination policy. This can reach the phone's LAN, private, loopback, or metadata addresses, so keep it dangerous/explicitly allowlisted or add a reviewed destination policy before forwarding.
  - confidence: 0.89
- **[P2] Refresh the node session after toggling HTTP Access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:801-803`
  - body: `setHttpAccessEnabled` only updates preferences, but the gateway authorizes against the command list declared by the active node session. Turning the default-off switch on while already connected leaves `http.request` undeclared until reconnect, so the normal toggle path appears broken.
  - confidence: 0.9
- **[P3] Document and changelog the new command:** `src/infra/node-commands.ts:8`
  - body: This adds a public Android setting plus a new `node.invoke` command and JSON contract, but the branch changes no docs or `CHANGELOG.md`. Operators need the opt-in behavior, parameters, limits, and security expectations before this user-facing feature ships.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces an opt-in but general-purpose Android HTTP proxy, and the current branch default-allows it without a reviewed egress/SSRF policy.

Concerns:

- **[medium] Default allowlists arbitrary Android HTTP egress:** `src/gateway/node-command-policy.ts:103`
  - body: `NODE_HTTP_REQUEST_COMMAND` is added to Android platform defaults, so once the Android app advertises it the gateway can forward arbitrary URL fetches without explicit operator config. That creates a new cross-network egress path from the phone's LAN and should be treated as security-sensitive before merge.
  - confidence: 0.89
- **[medium] Handler accepts any http or https destination:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:49`
  - body: The handler only filters by URL scheme, so loopback, private, link-local, metadata, and local-network destinations from the Android device remain in scope. That bypasses the gateway process proxy/SSRF egress controls and needs a deliberate policy decision.
  - confidence: 0.86
