## Summary

Keep open. Current main still has the reported split: status reads persisted session-store rows, while agent-side session tools apply the default session visibility policy and can hide an existing same-agent WhatsApp direct session that was not spawned by the requester.

## What This Changes

The issue asks for WhatsApp direct sessions that are visible in status and usable by cron delivery to also be discoverable and targetable through agent session tools for on-demand automation.

## Best Possible Solution

Keep this open until maintainers choose the intended visibility policy. If on-demand automation should target existing same-agent WhatsApp DMs, implement a generic deliverable-session discovery or target-resolution path that remains visibility-aware and covers WhatsApp direct sessions; otherwise document the required tools.sessions.visibility setting for operators who want that behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create a persisted same-agent WhatsApp direct session, leave tools.sessions.visibility unset so it defaults to tree, then compare status output with sessions_list or sessions_send from a different same-agent requester session that did not spawn that WhatsApp session.

## Solution Assessment

Unclear as a product decision. Making WhatsApp DMs always visible is not obviously the best fix because it bypasses an intentional privacy boundary; a generic visibility-aware deliverable-session mechanism or explicit config/docs guidance is safer.
