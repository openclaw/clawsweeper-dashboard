## Summary

Keep open. Current main still has the reported split: status and cron-capable session rows can show/use a WhatsApp direct session, while agent session tools keep the default `tree` visibility boundary and can hide a same-agent direct session that was not current or spawned by the requester.

## What This Changes

The issue requests that WhatsApp direct sessions visible in status and usable by cron delivery also be discoverable and targetable through agent session tools.

## Best Possible Solution

Define one generic policy for deliverable-session discovery: either explicitly surface same-agent channel sessions through a visibility-aware path, or document `tools.sessions.visibility=agent/all` as the supported opt-in for this workflow.

## Reproduction Assessment

Yes, by source inspection: with `tools.sessions.visibility` unset, default `tree` visibility lets status show a stored same-agent WhatsApp row while `sessions_list` and `sessions_send` can filter or forbid it if it is not current or requester-spawned. I did not run a live macOS WhatsApp Desktop reproduction in this Linux review.

## Solution Assessment

Unclear. A WhatsApp-only live-discovery bypass would sidestep the generic session privacy model; a generic deliverable-session policy or explicit configuration guidance is the safer maintainable direction.
