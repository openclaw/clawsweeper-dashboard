## Summary

Keep open. Current main still has a source-proven split where status and cron-visible stored WhatsApp session rows can exist while agent session tools filter or forbid the same target under the default session visibility policy; resolving that is a generic session visibility/source-of-truth decision, not conservative cleanup.

## What This Changes

The issue requests consistent WhatsApp direct session discovery and targeting when status and cron delivery show an active WhatsApp session.

## Best Possible Solution

Define one generic deliverable-session discovery contract: either make the supported opt-in visibility mode explicit for this workflow, or add a tested source-of-truth policy that aligns status, cron delivery, sessions_list, and sessions_send without a WhatsApp-only bypass.

## Reproduction Assessment

Yes, source-level. With default tree visibility, sessions_list and sessions_send can omit or forbid a stored same-agent WhatsApp direct session that is not current or requester-owned, while status still lists stored rows; I did not run a live macOS WhatsApp Desktop repro.

## Solution Assessment

Unclear. A WhatsApp-only live-discovery bypass would sidestep the generic session privacy model; maintainers should decide whether deliverable-session discovery should widen generically or whether the documented visibility opt-in is the supported workflow.
