## Summary

Keep open. Current main still has the reported split: status and cron-capable stored delivery context can see/use a WhatsApp direct session, while agent session tools apply the default session visibility boundary and can hide same-agent sessions that were not spawned by the requester.

## What This Changes

The issue asks for WhatsApp direct sessions that are visible in status and usable by cron delivery to also be consistently discoverable and targetable by agent session tools.

## Best Possible Solution

Keep the generic session privacy model intact while maintainers choose either a visibility-aware deliverable-session discovery path or explicit guidance that operators must opt into tools.sessions.visibility=agent/all for same-agent WhatsApp DM targeting.

## Reproduction Assessment

Yes. Persist a same-agent WhatsApp direct session that was not spawned by the requester, leave tools.sessions.visibility unset so it defaults to tree, then compare openclaw status --json with sessions_list or sessions_send from another same-agent requester session.

## Solution Assessment

Unclear. A WhatsApp-only live discovery bypass would address one surface but risks sidestepping generic session visibility; a policy-aware generic path or clear configuration guidance is safer.
