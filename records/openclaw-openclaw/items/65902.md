## Summary

Keep open. Current main still has the reported split between status/cron-visible session rows and agent session tools, and resolving it requires a session visibility/source-of-truth decision rather than conservative cleanup.

## What This Changes

The issue asks for WhatsApp direct sessions that work for status and cron delivery to be exposed consistently to agent-side session discovery and send tools.

## Best Possible Solution

Define a generic deliverable-session discovery policy, or document the existing tools.sessions.visibility=agent/all opt-in as the supported workflow instead of adding a WhatsApp-only bypass.

## Reproduction Assessment

Yes, by source inspection: with default tree visibility, sessions_list and sessions_send can omit or forbid a stored same-agent WhatsApp direct session that is not current or requester-owned, while status still lists store rows. I did not perform a live macOS WhatsApp Desktop reproduction.

## Solution Assessment

Unclear. A WhatsApp-only live-discovery bypass would sidestep the generic session privacy model; maintainers should decide whether to widen deliverable-session discovery generically or make the current visibility opt-in explicit.
