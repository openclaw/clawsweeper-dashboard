## Summary

Keep open. Current main still has a source-backed split: status and cron/direct-delivery paths can record and surface a WhatsApp direct session, while agent-facing session tools still apply the default tree-scoped visibility guard and can omit or forbid that same target; the follow-up discussion correctly frames this as a generic session source-of-truth and visibility-policy problem rather than a WhatsApp transport-only failure.

## What This Changes

Investigate inconsistent WhatsApp direct-session discovery where status and cron delivery can see a session but agent-side session tools may not expose or target it.

## Best Possible Solution

Define one generic deliverable-session discovery contract that aligns status, cron delivery, sessions_list, and sessions_send while preserving explicit session visibility boundaries.

## Reproduction Assessment

Yes, source-level. On current main, a stored same-agent WhatsApp direct session that is not the requester/current tree can be listed by status and used by cron-direct delivery while sessions_list skips it and sessions_send forbids it under default tree visibility.

## Solution Assessment

Unclear. A WhatsApp-only live AX discovery bypass would sidestep the generic session privacy model; maintainers should decide whether discovery widens generically or whether explicit tools.sessions.visibility opt-in is the supported workflow.
