## Summary

Keep open. Current main still has the reported split: status and cron-capable stored delivery context can expose/use a WhatsApp direct session, while agent session tools apply the default tree visibility boundary and can hide a same-agent direct session that was not spawned by the requester.

## What This Changes

The issue asks for WhatsApp direct sessions that are visible to status and cron delivery to be consistently discoverable and targetable from agent-side session tools.

## Best Possible Solution

Keep the generic session privacy model intact while maintainers choose whether to add a visibility-aware deliverable-session discovery path or document that operators must opt into tools.sessions.visibility=agent/all for same-agent WhatsApp DM targeting.

## Reproduction Assessment

Yes. Persist a same-agent WhatsApp direct session that was not spawned by the requester, leave tools.sessions.visibility unset so it defaults to tree, then compare openclaw status --json with sessions_list or sessions_send from another same-agent requester session.

## Solution Assessment

Unclear. A WhatsApp-only discovery bypass is not the safest maintainable fix because it would sidestep generic session visibility; a generic policy-aware path or explicit configuration guidance is safer.
