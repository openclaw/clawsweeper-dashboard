## Summary

Keep open. Current main still has the reported split: CLI status and cron delivery can see or use a persisted WhatsApp direct session, while agent-side session tools still apply the default `tools.sessions.visibility=tree` boundary and can hide same-agent sessions that were not spawned by the requester.

## What This Changes

Report asks for WhatsApp direct sessions that appear in status and cron delivery to be consistently discoverable and targetable by agent session tools.

## Best Possible Solution

Keep this open until maintainers choose the intended visibility policy, then either add a generic visibility-aware deliverable-session discovery/targeting path or document the required `tools.sessions.visibility` setting for operators who want same-agent WhatsApp DM targeting.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to persist a same-agent WhatsApp direct session that was not spawned by the requester, leave `tools.sessions.visibility` unset so it defaults to `tree`, then compare `openclaw status --json` with `sessions_list` or `sessions_send` from a different same-agent requester session.

## Solution Assessment

Unclear as a product decision. Making WhatsApp DMs unconditionally visible is not the safest fix; a generic visibility-aware deliverable-session path or explicit config/docs guidance would preserve the existing privacy model better.
