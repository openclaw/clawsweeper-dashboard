## Summary

Keep open. Current main still has a source-backed split where stored WhatsApp session rows can appear in status and cron-delivery paths while agent session tools hide or forbid the same target under the default tree-scoped visibility policy; the later issue comment also frames this as a structural source-of-truth problem, not a WhatsApp transport-only failure.

## What This Changes

The issue asks for WhatsApp direct sessions that appear in status and work for cron delivery to also be consistently discoverable and targetable by agent session tools.

## Best Possible Solution

Define one generic deliverable-session discovery contract that aligns status, cron delivery, sessions_list, and sessions_send while preserving explicit session visibility boundaries.

## Reproduction Assessment

Yes, source-level. On current main, a stored same-agent WhatsApp direct session that is not the requester/current tree can be listed by status while sessions_list skips it and sessions_send forbids it under default tree visibility; no live macOS WhatsApp Desktop repro was run.

## Solution Assessment

Unclear. A WhatsApp-only live-discovery bypass would sidestep the generic session privacy model; maintainers should decide whether discovery should widen generically or whether the documented visibility opt-in is the supported workflow.
