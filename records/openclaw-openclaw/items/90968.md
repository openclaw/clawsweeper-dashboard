## Label Justifications

- `P1`: This PR targets a broken managed ACP/OpenCode first-turn workflow and currently introduces a first-turn runtime-control regression.
- `merge-risk: 🚨 compatibility`: Existing startup runtime options such as timeoutSeconds can stop taking effect during upgrade if the full option signature is treated as already applied.
- `merge-risk: 🚨 security-boundary`: A startup permissionProfile could be skipped before the first turn because the patch can mark it applied without sending the approval_policy config control.

## Summary

Keep open: the underlying ACP/OpenCode startup-control bug is valid, but this patch is not safe because it marks every runtime control as applied even though startup only applies model/thinking/cwd; it also lacks real behavior proof.

## What This Changes

This PR sets the cached ACP runtime handle's applied control signature during session initialization and adds a regression test so startup model/thinking options are not resent as config controls on the first turn.

## Best Possible Solution

Track startup-applied controls separately from post-start controls so model/thinking are not resent to OpenCode, while timeout, permissions, mode, and backend extras still apply before the first turn; then add redacted real OpenCode/ACP proof.

## Reproduction Assessment

Yes, from source inspection: `sessions_spawn` can pass startup `timeoutSeconds`, `ensureSession` cannot receive it, and the new cache signature would make the first `runTurn` skip applying it as a config control.

## Solution Assessment

No. The fix is aimed at the right ACP startup bug, but the all-options signature is too broad; the maintainable fix needs to distinguish controls actually applied during startup from controls still requiring post-start application.
