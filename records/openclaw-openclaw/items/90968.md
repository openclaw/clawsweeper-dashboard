## Summary

Keep open. The replay-suppression direction is useful, but the latest head still is not merge-ready because it blanket-strips Claude ACP model/thinking controls in core despite the ACPX adapter already preserving valid Claude model ids, leaves cached/update replay paths ambiguous, and has no latest-head real behavior proof.

## What This Changes

The PR changes ACP startup/resume/control-cache handling so startup model/thinking/cwd are not replayed as config controls, and adds Claude-specific stripping of model/thinking for managed ACP spawn/init/resume paths with regression tests.

## Best Possible Solution

Keep the startup replay suppression, but make one canonical Claude ACP runtime-option policy that preserves valid Claude model ids, rejects or strips unsupported non-Claude model/thinking controls consistently, and proves startup, resume, cached replay, and explicit update behavior on the latest head.

## Reproduction Assessment

Yes from source, though I did not run a live ACP session in this read-only review. Current main can replay model/thinking as ACP config controls, and related Claude ACP evidence shows non-Claude model/thinking controls are rejected by that runtime.

## Solution Assessment

No. The applied-options delta cache is the right direction for replay suppression, but blanket core stripping is too broad and the Claude policy should preserve valid Claude ids while covering every runtime-option entry point consistently.
