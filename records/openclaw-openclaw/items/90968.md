- `merge-risk: 🚨 security-boundary`: The same baseline reset can prevent approval_policy from being applied because model/thinking are sent first and may fail before permission controls run.

## Summary

Keep open: the revised baseline fixes the initial first-turn startup model/thinking resend path, and the supplied live OpenCode proof is useful, but the patch still drops the applied baseline after runtime-option updates and can reopen the same failure before permission/timeout/mode controls are applied.

## What This Changes

The PR replaces ACP runtime-control cache signatures with an applied-runtime-options baseline so startup-applied model/thinking/cwd can be skipped while post-start controls still reconcile before turns.

## Best Possible Solution

Preserve the matching startup-applied baseline for unchanged model/thinking/cwd when runtime options are persisted, and reapply only changed or not-yet-applied post-start controls with regression coverage.

## Reproduction Assessment

Yes from source inspection: initialize/resume seeds model/thinking, runtime-option persistence clears `appliedRuntimeOptions`, and the next turn builds the full config-option list with model/thinking before permission/timeout. I did not run a live repro in this read-only review.

## Solution Assessment

No. The applied-baseline approach is the right layer, but clearing that baseline wholesale after runtime-option persistence is not the best fix because it reopens the unsupported model/thinking resend path.
