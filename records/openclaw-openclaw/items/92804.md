## Summary

Keep open: the PR is a focused ACP `final_only` bug-fix attempt, but it defers only final text while still force-flushing tool/status deliveries at intermediate `done` events, so the turn-scoped message-delivery boundary remains incomplete.

## What This Changes

The PR changes the ACP reply projector to preserve accumulated `final_only` text across intermediate `done` resets and adds projector regression tests for multi-invocation text accumulation.

## Best Possible Solution

Land one canonical ACP projector fix that treats intermediate `final_only` `done` as an invocation boundary, preserves accumulated text/tool/status/limit/separator state until dispatch turn completion, and includes real ACP gateway or transport proof.

## Reproduction Assessment

Yes at source level: current main flushes and resets the ACP projector on every `done` or `error`, while dispatch also performs a later turn-level flush. I did not establish a live WeChat reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible text-preservation patch, but the best fix should preserve the whole `final_only` turn-scoped buffer, including tool/status deliveries, until the existing dispatch turn-completion flush.
