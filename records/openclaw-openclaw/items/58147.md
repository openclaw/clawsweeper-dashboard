## Summary

Keep open. Current main still has no path for `agents.defaults.timeFormat` to affect Control UI/WebChat timestamp rendering: the gateway defaults payload omits it, the UI types cannot carry it, and the browser formatter uses locale defaults directly. The prior ClawSweeper comment remains directionally correct on the newer main SHA.

## What This Changes

The issue requests that Control UI/WebChat session duration and timestamp display honor `agents.defaults.timeFormat` instead of relying on locale-default 12h/24h formatting.

## Best Possible Solution

Expose a browser-safe `timeFormat` value through `sessions.list` defaults or an equivalent Control UI payload, then apply explicit 12h/24h formatting in the relevant timestamp renderers while preserving locale behavior for `auto`; document the UI scope if maintainers decide the setting should apply beyond prompt text.

## Reproduction Assessment

Yes for a source-level reproduction: configure `agents.defaults.timeFormat: "24"` and inspect the current `sessions.list` defaults plus `formatChatTimestampForDisplay`; there is no way for that preference to reach or alter the formatter. I did not launch the UI in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to thread the existing config value through the gateway/UI contract and update browser-safe formatting plus focused tests. If maintainers want `timeFormat` to remain prompt-only, the safer alternative is to document that boundary and track a separate UI display preference.
