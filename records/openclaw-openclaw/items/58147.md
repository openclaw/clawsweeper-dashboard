## Summary

Keep open: current main still defines `agents.defaults.timeFormat`, but the sessions/defaults gateway contract and Control UI/WebChat timestamp renderers still have no path for that value, so the preference mismatch remains source-reproducible.

## What This Changes

The issue asks for Control UI/WebChat session duration and timestamp display to honor the existing `agents.defaults.timeFormat` setting.

## Best Possible Solution

Thread the existing `timeFormat` value through the sessions/defaults UI contract and apply explicit `auto`/`12`/`24` clock formatting in the affected Control UI/WebChat timestamp renderers, while keeping stopwatch-style durations separate unless product docs define them.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure `agents.defaults.timeFormat` to `"24"`, then inspect `sessions.list` defaults and the UI timestamp renderers; the value cannot currently reach or alter browser timestamp formatting.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing config value through the gateway/UI contract and preserve locale-default behavior for `auto`; adding a separate UI preference would be broader than this report.
