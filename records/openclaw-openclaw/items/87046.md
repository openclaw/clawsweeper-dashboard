## Label Justifications

- `P1`: The PR targets a reported broken Discord/cron gateway workflow, but the proposed fix can change runtime plugin disablement behavior.
- `merge-risk: 🚨 compatibility`: Merging this can change empty-plugin-scope reload semantics by keeping previously registered plugin harnesses alive instead of clearing them.
- `merge-risk: 🚨 security-boundary`: A disabled or inactive plugin's harness could remain callable after the active registry is empty, weakening plugin disablement as an operator boundary.

## Summary

Keep open. The reported gateway/runtime failure is plausible, but this patch restores every prior agent harness into an empty plugin scope, which can leave plugin-owned harnesses callable after their plugin is no longer active; the claimed `claude-cli` path also looks like CLI-backend metadata, not an embedded AgentHarness registry problem.

## What This Changes

The PR snapshots and restores the global agent harness registry in the plugin loader and gateway empty-plugin-scope branches, and adds a loader regression test expecting a previously registered harness to survive.

## Best Possible Solution

Fix empty-scope reloads so required provider or CLI-runtime metadata survives without leaving plugin-owned agent harnesses callable when their plugin is inactive, then cover both the real CLI/backend failure and plugin-disablement behavior.

## Reproduction Assessment

Partly: source inspection shows the registered-harness wipe on current main, and the proposed test would cover that path. The exact `claude-cli` production failure is not reproduced with high confidence because current docs and source classify `claude-cli` as a CLI backend alias rather than an AgentHarness.

## Solution Assessment

No. Restoring every previous harness is broader than the ownership boundary, and the test proves `codex`-style harness preservation rather than the reported `claude-cli` runtime path; a narrower metadata or owner-aware fix is safer.
