## Summary

Keep open: current main still has the empty-plugin-scope harness wipe this PR targets, but the branch should not merge as-is because it restores every previously registered harness into an intentionally empty active plugin registry.

## What This Changes

The branch snapshots and restores registered agent harnesses in the plugin loader and gateway empty-scope reload paths, and adds a loader regression test.

## Best Possible Solution

Rebase onto current main and fix the empty-scope lifecycle so required always-on harness availability is preserved without keeping disabled or unscoped plugin-owned harnesses callable, with loader and gateway tests covering both cases.

## Reproduction Assessment

Source-reproducible: current main has both empty-scope branches clearing the global harness registry without restoration, and the PR defect follows from restoring all prior harnesses into an empty active registry. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Snapshot/restore is a plausible mitigation, but restoring every prior harness is not the best fix because it breaks the active plugin ownership boundary; the safer path is a narrower lifecycle rule with explicit ownership tests.
