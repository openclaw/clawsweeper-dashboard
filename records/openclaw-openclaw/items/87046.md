## Summary

Keep open: the reported empty-scope harness wipe is still real on current main and the latest release, but this branch is not merge-ready because the restore-all approach can keep plugin-owned harnesses callable after the active registry has been set to empty, and the proof does not show a patched gateway run.

## What This Changes

The branch snapshots and restores registered agent harnesses in the plugin loader and gateway empty-plugin-scope paths and adds a loader regression test.

## Best Possible Solution

Preserve required always-on provider harness availability without reviving disabled or unscoped plugin-owned harnesses, with loader and gateway tests plus redacted patched-gateway proof.

## Reproduction Assessment

Yes at source level. Current main has both empty-scope branches clearing the process-global harness registry without restoration, and the proposed test path reproduces the registry wipe, but I did not run a live gateway timing repro in this read-only review.

## Solution Assessment

No. Snapshot-and-restore is a plausible mitigation, but restoring every prior harness is too broad because it crosses the active plugin ownership boundary; the safer fix is ownership-aware preservation with explicit tests.
