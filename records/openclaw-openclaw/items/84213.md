## Label Justifications

- `P2`: This is a user-facing native Windows CLI regression for shell completion setup, but it does not make the core agent runtime unusable.
- `impact:crash-loop`: The reported command hangs indefinitely and must be killed, which is a process-level availability failure for that CLI workflow.

## Summary

Keep open. Current main still routes the reported direct completion commands through full command/subcommand/plugin registration before generating output or writing caches, and the open related PRs cover adjacent doctor/update timeout paths rather than proving the native Windows direct command hang is fixed.

## What This Changes

The issue asks OpenClaw to stop hanging on native Windows when generating cached shell completions for zsh, PowerShell, and write-state modes.

## Best Possible Solution

Make direct completion generation bounded and Windows-safe while preserving explicit plugin-owned completions, then prove packaged native Windows zsh, PowerShell, and `--write-state` runs exit promptly and create the expected cache files.

## Reproduction Assessment

No high-confidence live reproduction yet. The issue gives concrete native Windows steps and current source confirms all reported commands share the eager completion-registration path, but this review did not run native Windows.

## Solution Assessment

Yes for keeping this as a narrow CLI completion fix, but not by globally disabling explicit plugin completion registration. The repair should bound or harden the direct completion path while preserving the plugin-completion contract.
