## What This Changes

The PR adds Amazon Bedrock discovery `modelContextWindowOverrides` for regex-based per-model context-window overrides, with tests, manifest schema/UI hints, docs, config typing, generated SDK baseline hash updates, and an unrelated current-main QQBot/tsdown cleanup diff.

## Best Possible Solution

Land one canonical Bedrock context-window fix. If this regex override approach is selected, make overrides use the same normalized candidate IDs as known lookup, keep built-in precedence deterministic, update the docs with an unshadowed example, add the required changelog entry, and rebase away unrelated current-main noise before merge.

## Reproduction Assessment

Yes. Current main can be inspected statically: unknown Bedrock discovery models still fall back to `defaultContextWindow`, and a focused PR test can reproduce the override bug by passing a prefixed profile/model ID that reaches `resolveKnownContextWindow` without a base-model ARN while using a base-ID regex such as `^example\.`.

## Solution Assessment

No, not as written. The configurable override design is plausible, but the implementation does not apply regexes to the resolver's normalized candidate IDs, and the docs/changelog are not merge-ready; #73382 remains a narrower alternative for the Claude-specific bug.
