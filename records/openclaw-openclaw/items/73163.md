## Summary

Keep open. The PR addresses a valid Control UI operator UX gap, but the latest head still derives the warning from raw startup config instead of the resolved gateway runtime state, so CLI/runtime overrides can make the warning fire or stay quiet for the wrong server behavior.

## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI device-identity failures, focused suppression tests, and a changelog entry.

## Best Possible Solution

Thread the resolved Control UI enabled state and resolved auth mode into the startup logging path, or pass a small precomputed warning context, so the warning reflects the server that actually bound while leaving the existing WebSocket/device-identity enforcement unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: start with config that looks warning-worthy, then pass runtime overrides such as `controlUiEnabled: false` or `auth.mode: "trusted-proxy"`; the PR helper still reads `cfgAtStart` and can warn anyway. The inverse case, config-disabled Control UI with a runtime enable override, can also stay quiet incorrectly.

## Solution Assessment

No. Startup logging is the right surface, but this implementation is not the narrowest correct fix until it uses resolved runtime state rather than re-inferring actual gateway behavior from raw config.

## Review Findings

Overall correctness: patch is incorrect

Thread the resolved state into this logger and cover those override cases.
