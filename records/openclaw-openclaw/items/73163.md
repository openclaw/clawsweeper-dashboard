## Summary

Keep this PR open, but the latest head still needs a correction before merge: the new warning is derived from raw startup config, while the gateway already resolves runtime overrides before post-attach startup logging.

## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI device-identity failures, focused startup-log tests, and a changelog entry.

## Best Possible Solution

Keep the startup warning, but thread the resolved Control UI enabled state and resolved auth mode, or a small precomputed warning context, from server startup through the post-attach logger. Add regression coverage for runtime override cases without changing WebSocket/device-identity enforcement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: config that looks warning-worthy plus runtime overrides such as controlUiEnabled: false or auth.mode: "trusted-proxy" can still warn, while config-disabled UI plus a runtime enable override can stay quiet incorrectly.

## Solution Assessment

No. Startup logging is the right surface, but this implementation is not the narrowest correct fix until it uses the same resolved runtime state as the server that actually bound.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use resolved runtime state for the warning:** `src/gateway/server-startup-log.ts:66-72`
  - body: `logGatewayStartup` derives Control UI enabled state and trusted-proxy suppression from raw `cfg`, but startup resolves runtime overrides before this log runs. With overrides like `controlUiEnabled: false` or `auth.mode: "trusted-proxy"`, the warning can fire for a Control UI/auth mode that did not actually start, and the inverse runtime-enable case can miss the warning. Thread resolved state into this logger and cover those cases.
  - confidence: 0.9
