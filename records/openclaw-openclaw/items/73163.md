## Summary

Keep open. This PR targets a real gap tracked by #71669, but the current head still derives the new warning from startup config instead of the resolved runtime state, so CLI/runtime overrides can produce a warning for a Control UI/auth mode that did not actually start.

## What This Changes

The branch adds a gateway startup warning for non-loopback plain-HTTP Control UI secure-context/device-identity rejection, focused suppression tests, and a changelog entry.

## Best Possible Solution

Make the startup warning reflect the gateway that actually started: use resolved Control UI enabled state, resolved auth mode, TLS state, and listen hosts, while preserving the existing WebSocket/device-identity enforcement and keeping the changelog entry aligned with the final behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: start from config that would warn, then apply runtime overrides such as `controlUiEnabled: false` or `auth.mode: "trusted-proxy"`; the PR helper still reads `cfgAtStart` and can emit the warning anyway.

## Solution Assessment

Mostly yes. Startup logging is the narrowest maintainable place for this operator guidance, but this implementation should use resolved runtime state before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use resolved runtime state for the warning:** `src/gateway/server-startup-log.ts:66-72`
  - body: `logGatewayStartup` derives Control UI enabled and auth mode from `cfg`, but startup resolves CLI/runtime overrides before this log runs and then passes only `cfgAtStart`. With config that looks warning-worthy but start options such as `controlUiEnabled: false` or `auth.mode: "trusted-proxy"`, the warning can fire for a Control UI/auth mode that did not actually start; thread resolved values into the logger and cover those quiet cases.
