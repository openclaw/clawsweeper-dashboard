## Summary

Keep this draft PR open, but the patch is not merge-ready: it adds a sticky persisted preference marker that can make a later temporary `/usage` choice survive rollover, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds explicit `--persist` handling for reply/session preference commands so selected usage, model, and thinking settings can be marked for persistence across session rollovers.

## Best Possible Solution

If maintainers want this flag, keep the explicit `--persist` model but make persistence follow the latest user intent: add the marker only for `--persist`, remove it for ordinary updates, and prove the transition across rollover.

## Reproduction Assessment

Yes from source inspection: after `/usage full --persist`, a later `/usage tokens` without `--persist` updates `responseUsage` but leaves `persistentPreferenceFields` containing `responseUsage`, so the later supposedly session-scoped value can still roll over.

## Solution Assessment

No; the marker approach is plausible, but the best version must clear old markers when the user makes a non-persistent update and should include maintainer-approved unset semantics plus real behavior proof.
