## Summary

Keep open. Current main still ignores fetched Feishu mention metadata in the quoted/thread message parser, and an open implementation PR already references this issue with closing syntax, so the issue should wait for that PR or an equivalent focused fix to land.

## What This Changes

Resolve Feishu quoted/replied message `@_user_N` placeholders by applying fetched mention metadata, matching the live inbound message path.

## Best Possible Solution

Review and narrow PR #48976, or land an equivalent Feishu-plugin-only patch that models fetched `mentions`, adapts them into the existing normalization helper, applies it after fetched content parsing for quoted/root/thread messages, and adds focused `send.test.ts` regression coverage.

## Reproduction Assessment

Yes. A mocked `getMessageFeishu()` or `listFeishuThreadMessages()` response with `body.content` containing `@_user_1` plus fetched mention metadata would still return the raw placeholder because the current parser ignores `mentions`.

## Solution Assessment

Yes. The narrow maintainable fix is inside the Feishu plugin fetched-message parser, reusing the existing mention normalization path with a small adapter and regression tests rather than changing core behavior.
