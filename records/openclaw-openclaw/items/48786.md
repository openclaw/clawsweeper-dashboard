## Summary

Keep open. Current main still parses Feishu API-fetched quoted/root/thread message content without modeling or normalizing fetched-message mention metadata, while the live inbound path already normalizes Feishu mention placeholders before agent context. The same-repo fixing PR was closed unmerged as an unmergeable broad branch, and a still-open fork PR references this issue but is the same conflicting 18-file branch, so this issue should remain the canonical tracker until a narrow Feishu-only fix lands or the linked fork PR is closed/superseded.

## What This Changes

The issue requests resolving Feishu quoted/replied-message mention placeholders like `@_user_N` into readable mention markup when quoted content is fetched for agent context.

## Best Possible Solution

Land a narrow Feishu-plugin-only fix that models fetched-message mention metadata, adapts it to the established mention-normalization contract, applies it for quoted/root/thread fetched content, and adds focused `send.test.ts` regression coverage while leaving unrelated authorization, session, dependency, and changelog surfaces untouched.

## Reproduction Assessment

Yes, at source level: current main fetches quoted/root/thread Feishu messages through `getMessageFeishu()` and `listFeishuThreadMessages()`, then returns parsed body text without reading or normalizing fetched-message mentions. I did not run a live Feishu tenant repro in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow Feishu plugin change at the fetched-message parser boundary, reusing or adapting the existing mention-normalization contract. The broad linked branch is not the best landing path because it mixes the useful Feishu fix with unrelated changes and is conflicting.
