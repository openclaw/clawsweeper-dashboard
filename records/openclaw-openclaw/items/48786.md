## Label Justifications

- `P1`: This is a user-facing Feishu channel workflow bug where quoted/thread context can be delivered to the agent with unresolved mention placeholders.
- `impact:session-state`: The affected surface is agent-visible quoted and thread context, where mention state drifts from the display-resolved Feishu message metadata.

## Summary

Keep open. Current main still parses fetched Feishu quoted/thread message content without reading fetched-message mentions, while live inbound Feishu messages already normalize mention placeholders; the prior upstream PR was closed unmerged and the remaining open fork PR is not a landed fix.

## What This Changes

Feishu quoted/replied message context should resolve fetched-message mention placeholders instead of exposing raw `@_user_N` tokens to the agent.

## Best Possible Solution

Land a narrow Feishu-plugin-only fix that adapts fetched-message `mentions` to the existing normalization helper for quoted, root, and thread-history content, with focused regression tests and real Feishu proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: a fetched Feishu message with `body.content` containing `@_user_1` and SDK `mentions` metadata will currently return the parsed text unchanged because `parseFeishuMessageItem()` never reads `item.mentions`. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, normalizing fetched Feishu message mentions inside the Feishu plugin parser is the narrow owner-boundary fix. The safe implementation should adapt the SDK `{ key, id, id_type, name }` mention shape instead of blindly casting it to the live event shape.
