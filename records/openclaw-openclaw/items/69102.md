## Summary

Keep open: current main and v2026.6.10 still lack direct Google/Gemini Flex service-tier request plumbing. The OpenRouter workaround in the discussion is useful, and https://github.com/openclaw/openclaw/issues/90243 overlaps on tier selection, but neither replaces direct Gemini `service_tier` support.

## What This Changes

Request support for configuring Gemini Flex service tier by sending Google `service_tier`/`serviceTier` on direct Gemini model requests.

## Best Possible Solution

Have maintainers approve a Google-provider-owned service-tier params contract, then wire direct Gemini `serviceTier`/`service_tier` into the Google request payload with focused tests and docs while coordinating with the Vertex mapping work.

## Reproduction Assessment

Yes. Source inspection shows configured Google params currently have no path into a Google `serviceTier` or `service_tier` request field, matching the reported missing capability without needing a live paid Gemini Flex call.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrowest maintainable direction appears to be a Google-provider-owned per-model params contract, not query-string model refs or a cron-only control by default.
