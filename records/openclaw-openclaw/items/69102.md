## Summary

Keep open: current main still lacks native direct Google/Gemini Flex service-tier request plumbing. The OpenRouter workaround in the discussion is useful for users who can route through OpenRouter, and https://github.com/openclaw/openclaw/issues/90243 overlaps on Google tier selection, but neither replaces direct Google `service_tier` support.

## What This Changes

Request native Gemini Flex service-tier support so Google/Gemini model requests can specify the Flex tier from OpenClaw configuration or request params.

## Best Possible Solution

Have maintainers approve a Google-provider-owned params contract, then wire direct Gemini `serviceTier`/`service_tier` into the Google request payload with focused tests and docs while coordinating the separate Vertex physical-model mapping work.

## Reproduction Assessment

Yes, source-level. Current main has no direct Google `serviceTier` or `service_tier` path in the Google transport request types, payload builder, or extra-param normalization.

## Solution Assessment

Unclear until maintainers choose the public API shape. A Google-provider-owned per-model/request params contract looks maintainable; query-string model refs or the OpenRouter workaround are not the best native direct-Google fix.
