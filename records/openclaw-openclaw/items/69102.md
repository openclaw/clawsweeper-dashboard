## Summary

Keep open: current main and the latest release still lack native direct Google/Gemini Flex service-tier request support. The OpenRouter workaround in the discussion is useful, and the adjacent Vertex issue overlaps tier usage, but neither satisfies this direct Google provider request.

## What This Changes

Request support for configuring Gemini Flex by forwarding `service_tier`/`serviceTier: "flex"` on direct Google Gemini model requests, including non-time-sensitive cron or heartbeat workloads.

## Best Possible Solution

Have maintainers approve a Google-plugin-owned params contract, then wire Gemini Flex into the direct Gemini request payload with focused tests and docs while coordinating only the shared design points with the Vertex physical-model mapping work.

## Reproduction Assessment

Yes, source-reproducible: the path from configured model params into the direct Google transport has no service-tier canonicalization, option field, or payload emission. I did not run a live paid Gemini Flex call in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrowest maintainable direction appears to be Google-provider-owned per-model params support, not query-string model refs or cron-only service-tier controls by default.
