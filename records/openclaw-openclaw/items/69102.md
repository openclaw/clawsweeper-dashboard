## Summary

Keep open. Current main still has no Google/Gemini Flex service-tier config or payload emission path; the duplicate was closed back to this canonical issue, and the adjacent google-genai PR remains open with different scope and review blockers.

## What This Changes

Request support for configuring Google Gemini Flex by forwarding `service_tier`/`serviceTier: "flex"` on Gemini requests.

## Best Possible Solution

Add Google-plugin-owned `serviceTier`/`service_tier` normalization and forwarding into direct Gemini payloads, document Flex timeout expectations, and cover the params path with focused tests before considering broader query-string or cron-specific APIs.

## Reproduction Assessment

Yes for source-level proof: current main has no Google params field, normalization path, or payload emission path for `serviceTier`/`service_tier`. I did not run a live Gemini call because this read-only review should not use external credentials.

## Solution Assessment

Mostly yes: provider-owned params support is the narrowest maintainable direction for the core request. Query-string model refs and cron/heartbeat-specific controls should stay separate unless maintainers explicitly choose those public APIs.
