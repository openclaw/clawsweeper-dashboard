## Summary

Keep open: current main and the latest release still lack native Google/Gemini Flex service-tier request support; the OpenRouter workaround in the comments is useful but does not satisfy the direct Google provider contract.

## What This Changes

The issue requests a Google/Gemini configuration path that sends `service_tier: flex` on Gemini API requests.

## Best Possible Solution

Have maintainers approve a Google-plugin-owned per-model params contract, then wire Gemini Flex into the direct Google generateContent payload with focused tests and docs while leaving Vertex physical-model mapping to its separate issue.

## Reproduction Assessment

Yes, source-reproducible: Google documents Flex as a request option, and current main has no Google transport option, extra-param handling, docs, or payload emission for Gemini Flex service tier. I did not run a live paid Gemini Flex call in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrowest maintainable direction appears to be Google-provider-owned per-model params support, not query-string model refs or cron-only service-tier controls by default.
