## Summary

Keep open. Current main still lacks Google/Gemini Flex service-tier plumbing, the duplicate report is already closed back to this canonical issue, and no live implementation PR was found.

## What This Changes

The issue requests Google Gemini Flex support by allowing Gemini requests to send `service_tier: "flex"`.

## Best Possible Solution

Add Google-plugin-owned `serviceTier`/`service_tier` normalization and forwarding into the direct Gemini request payload, then document Flex timeout behavior and cover the params path with focused tests.

## Reproduction Assessment

Yes for source-level reproduction: current main has no Google params field or payload emission path for `serviceTier`/`service_tier`, so a configured Flex value cannot reach Gemini. I did not run a live Gemini request because this read-only review should not use external credentials.

## Solution Assessment

Mostly yes: provider-owned params support is the narrowest maintainable direction for the core request. Query-string model refs and cron/heartbeat-specific controls should stay separate unless maintainers explicitly choose those APIs.
