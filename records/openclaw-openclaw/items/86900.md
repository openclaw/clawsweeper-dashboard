## What This Changes

The PR changes staged compaction to return typed fallback results, stop split summarization after two consecutive generic fallbacks, and updates compaction regression tests for the new result shape.

## Best Possible Solution

Keep the typed breaker in the built-in LLM compaction path only if maintainers explicitly accept the fallback summary shape produced when the breaker opens.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main's staged compaction loop calls summarizeWithFallback for each split, and summarizeChunks wraps each generateSummary call in retryAsync with three attempts. The PR's focused tests model a persistent non-terminal summarizer failure and show the call-count behavior after the fix.

## Solution Assessment

Yes, this is an acceptable narrow mitigation because it uses typed fallback state instead of string-prefix control flow and keeps the breaker inside the built-in staged split loop. The remaining question is product/session-state acceptance of the breaker-open fallback text, not a concrete code defect.
