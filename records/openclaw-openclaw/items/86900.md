## Summary

Keep open: current main and v2026.6.1 still run every staged split and final merge attempt, while this PR adds the missing typed fallback signal and breaker. I did not find a concrete line-level defect, but the two-fallback threshold is a session-state policy choice that needs maintainer acceptance before merge.

## What This Changes

The branch changes built-in compaction so `summarizeWithFallback` returns typed fallback metadata, `summarizeInStages` stops after two generic fallback split results, and focused tests cover breaker behavior.

## Best Possible Solution

Land a maintainer-approved breaker policy that caps repeated summarizer calls, preserves successful partial summaries, and makes the outage fallback behavior explicit without adding new config.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.1 call `summarizeWithFallback` for every staged split and then the final merge, while `retryAsync` allows three attempts per failed chunk. I did not run a failing live provider scenario locally in this read-only review.

## Solution Assessment

Yes, if maintainers accept the outage policy: typed fallback metadata is a clean control signal and avoids string-prefix inference or new config. The remaining question is whether the hard-coded threshold and returned fallback shape are the desired session-state tradeoff.
