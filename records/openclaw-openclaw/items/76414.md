## Summary

Keep open: current main still rewrites `HOME` for local Codex app-server shells without ensuring the OpenClaw CLI resolves the gateway's active config, so config/message commands can load defaults and Signal can fall back to port `8080`. The source path and reporter logs make this a narrow repair candidate.

## What This Changes

The issue reports that OpenClaw CLI commands run inside an agent sandbox can read sandbox-local/default config instead of the host config, causing Signal sends to use the default port rather than the configured daemon port.

## Best Possible Solution

Propagate the resolved OpenClaw config/state identity into local Codex app-server child environments before `HOME` is rewritten, while preserving explicit user overrides and the isolated Codex home.

## Reproduction Assessment

Yes. A local Codex app-server stdio launch with no explicit `OPENCLAW_CONFIG_PATH` rewrites `HOME` to `agents/<id>/agent/codex-home/home`; current config resolution then derives a missing sandbox config and `loadConfig()` returns defaults, which sends Signal toward `8080`.

## Solution Assessment

Yes. Setting or inheriting the active `OPENCLAW_CONFIG_PATH` at the Codex app-server environment bridge is narrower than changing global config fallback behavior, changing Signal defaults, or relying on symlinks in the isolated home.
