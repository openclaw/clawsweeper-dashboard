## Summary

Keep open. Current main documents the OpenAI/Codex provider/runtime split and exposes runtime metadata on some session/agent surfaces, but `models.list` and the WebUI picker still do not carry effective runtime/auth-route metadata needed to distinguish the reported GPT routes.

## What This Changes

The issue asks for model picker and configure surfaces to distinguish GPT provider/auth route and agent runtime instead of collapsing distinct routes into the same visible model label.

## Best Possible Solution

Add an additive model-choice metadata path for effective provider, auth route/profile family, and agent runtime, then render it in WebUI and relevant picker surfaces without changing canonical provider/model refs or existing aliases.

## Reproduction Assessment

Yes for the central UI/API limitation: source inspection shows the model picker payload and label logic lack runtime/auth-route metadata. The reporter's exact beta-install latency and auth-path outcome was not live-reproduced.

## Solution Assessment

Unclear as an implementation path: the requested direction is valid, but the best fix should be an additive gateway/catalog metadata design rather than manual alias generation alone.
