## Summary

Keep open: current main documents and implements provider/model runtime policy, but the `models.list` catalog contract and Control UI picker label path still expose only model/provider/alias-style data, so the picker cannot show effective GPT/Codex auth route or agent runtime without a product/protocol decision.

## What This Changes

Requests that WebUI/channel model pickers show the effective provider/auth route and agent runtime for GPT/Codex choices instead of collapsing distinct routes into the same visible `gpt-5.5` label.

## Best Possible Solution

Add an additive model-choice metadata contract for effective provider, auth route/profile family, and agent runtime, then update WebUI and relevant picker surfaces to display it while preserving canonical provider/model refs and existing aliases.

## Reproduction Assessment

Yes for the central UI/API limitation: source inspection shows the model picker payload and label logic lack runtime/auth-route metadata. I did not live-reproduce the reporter's exact latency/auth-path outcome from the beta install.

## Solution Assessment

Unclear as a final implementation until maintainers choose the additive metadata shape. The best direction is protocol/catalog metadata plus picker rendering, not manual alias generation alone.
