## Summary

Keep open: the Telegram approval-reaction feature is useful and has live Telegram proof, but the branch currently accepts an undocumented allow-always reaction path because it registers the full shared approval-reaction decision set while the PR docs promise only thumbs-up and thumbs-down reactions.

## What This Changes

The PR adds Telegram approval-reaction handling, persistence-backed reaction target binding, focused Telegram tests, and Telegram docs for resolving pending approvals from message reactions.

## Best Possible Solution

Constrain Telegram reaction targets to `allow-once` and `deny`, add a regression that infinity does not resolve `allow-always`, and keep `/approve <id> allow-always` as the manual Telegram path unless maintainers explicitly choose to document and prove the broader shortcut.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR passes all pending approval actions into the Telegram reaction target, while the shared SDK resolves infinity to `allow-always` when that decision is present. I did not run tests because this was a read-only review.

## Solution Assessment

No as currently written: reusing the shared reaction bindings wholesale is too broad for Telegram's documented thumbs-up/thumbs-down behavior. The narrow maintainable fix is to filter Telegram's registered reaction decisions or explicitly document and prove the broader infinity shortcut.
