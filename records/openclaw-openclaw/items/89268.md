## Best Possible Solution

Keep the reaction-target binding approach, but rethrow non-expired resolver failures with focused retry coverage and add redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, from source inspection: the PR's `message_reaction` approval branch logs and swallows non-expired resolver errors, while the Telegram middleware only keeps an update retryable when the handler throws. I did not run the PR tests in this read-only review.

## Solution Assessment

No: the target-store binding is a reasonable owner-boundary fit, but the handler needs the same retry semantics as approval callbacks and live Telegram proof before this is the best merge shape.
