## Summary

Keep open: the feature is useful and not already on current main, but the patch is not merge-ready because bound approval reactions are resolved only after generic Telegram reaction notification/access gates, and the external PR has no live Telegram proof.

## What This Changes

The PR adds Telegram approval-reaction target storage, binds pending native approval prompts to Telegram messages, intercepts `message_reaction` updates to resolve approvals, and adds focused tests.

## Best Possible Solution

Keep the Telegram-owned target store and shared SDK reaction mapping, but move bound approval handling ahead of generic reaction notification/access gates and prove the fixed path in a live Telegram approval flow before merge.

## Reproduction Assessment

Yes, source-reproducible for the PR defect: the existing handler returns for `reactionNotifications: "off"` and generic authorization failures before the PR-added approval lookup. A focused test should cover a bound approval reaction under those settings.

## Solution Assessment

No: the owner boundary is plausible, but resolving bound approvals after generic reaction gates is too narrow. The maintainable fix is approval-first handling like Matrix/iMessage, followed by the existing generic reaction path for non-approval reactions.
