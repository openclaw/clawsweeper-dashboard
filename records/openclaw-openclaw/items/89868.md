## Summary

This should stay open: current main and the latest release still have a source-proven abort propagation gap where delegated context-engine compaction can keep running after the outer overflow-recovery wait aborts, matching the reported stop-button context loss.

## What This Changes

The issue reports that pressing stop during auto-compaction aborts the caller wait but still lets the in-flight delegated compaction persist a compacted transcript.

## Best Possible Solution

Forward the context-engine compact abortSignal through delegateCompactionToRuntime into compactEmbeddedAgentSessionDirect, add regression coverage for the delegation hop, and track the contextWindow model-limit concern separately if needed.

## Reproduction Assessment

Yes, source-reproducible: a delegated context-engine compaction receives an outer abortSignal but delegateCompactionToRuntime does not pass it into compactEmbeddedAgentSessionDirect. I did not live-reproduce the long-session UI stop sequence.

## Solution Assessment

Yes: the narrow maintainable fix is at the delegation boundary, not a rollback layer after compaction has already persisted. The model contextWindow concern should be split from this stop-abort repair.
