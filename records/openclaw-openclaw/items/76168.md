## Summary

Keep open. Current main still lets `openclaw message send` load the configured channel plugin registry before the message command's normal error-to-`exit(1)` wrapper, and Telegram's loaded surface can still reach `send.js`/`grammy`, so the plugin-load failure path is not proven fixed by the newer gateway-execution work.

## What This Changes

The issue asks for `openclaw message send` to avoid spinning after Telegram plugin load failure by either fixing bundled runtime dependency loading before import or failing fast with a clean nonzero exit.

## Best Possible Solution

Keep the issue open until message CLI plugin-registry failures are converted into a bounded, observable nonzero exit, while coordinating any Telegram gateway fast path with #75477/#75478.

## Reproduction Assessment

No. The production report is detailed and current-main source shows the plugin-load exception path before the clean `exit(1)` wrapper, but I did not establish a deterministic local Telegram/runtime-deps CPU-spin reproduction.

## Solution Assessment

Yes for the narrow first repair: route `ensurePluginRegistryLoaded` failures through the same failure/finalization path as message delivery failures. Gateway delegation is a useful complementary fix, but it does not replace a generic clean-exit guard for plugin-load exceptions.
