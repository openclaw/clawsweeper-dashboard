## Summary

Keep open. Current main still loads the configured channel plugin registry before the message command's normal error/exit wrapper, so a `PluginLoadFailureError` can bypass the tested `exit(1)` path; the related gateway-delegation and timeout reports remain open and do not prove this failure mode is fixed.

## What This Changes

The issue asks for `openclaw message send` to either load bundled channel runtime deps correctly or fail fast instead of leaving a spinning `openclaw-message` child after plugin load failure.

## Best Possible Solution

Keep this open until message CLI plugin-registry failures become a bounded, observable nonzero exit, while coordinating any Telegram gateway fast path with #75477/#75478 and leaving timeout/SIGTERM policy to #75895.

## Reproduction Assessment

No: the report has detailed production logs and current-main source shows a plugin-load exception path before the clean exit wrapper, but I did not establish a deterministic Telegram/runtime-deps reproduction on current main.

## Solution Assessment

Yes for the narrow repair: route `ensurePluginRegistryLoaded` failures through the same nonzero exit/finalization path as message delivery failures, and avoid broader gateway-delegation or timeout policy changes unless coordinated with the existing related items.
