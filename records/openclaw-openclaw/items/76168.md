## Summary

Keep open. The production report has concrete logs and current main still strict-loads the configured channel plugin registry before the message command's clean exit wrapper, so a `PluginLoadFailureError` can escape the tested `exit(1)` path; related #75478 and #75895 remain open and do not prove this failure mode is fixed.

## What This Changes

The issue asks for `openclaw message send` to either load bundled channel runtime deps correctly or fail fast instead of leaving an `openclaw-message` child spinning after plugin load failure.

## Best Possible Solution

Keep this open until message CLI plugin-registry failures become a bounded nonzero exit with regression coverage, and avoid local Telegram runtime loading for gateway-owned sends if the delegation path lands.

## Reproduction Assessment

No: there is detailed production evidence and a clear current-main source path where plugin-load exceptions occur before the clean exit wrapper, but I did not establish a deterministic local Telegram/runtime-deps reproduction for the CPU spin.

## Solution Assessment

Yes for the narrow first repair: wrap or move `ensurePluginRegistryLoaded` so plugin-load failures use the same nonzero exit/finalization path as message delivery failures, while coordinating broader Telegram gateway delegation with #75478.

## Repair Work Prompt

Fix the `openclaw message send` plugin-load failure path from https://github.com/openclaw/openclaw/issues/76168. Current main calls `ensurePluginRegistryLoaded` in `src/cli/program/message/helpers.ts` before `runCommandWithRuntime`, so `PluginLoadFailureError` can bypass the helper's clean `exit(1)` path. Make plugin-registry load failures log a clear error, run any intended bounded cleanup that is safe for this path, and exit nonzero without leaving an `openclaw-message` process alive. Add focused regression coverage in `src/cli/program/message/helpers.test.ts` for `ensurePluginRegistryLoaded` throwing. Coordinate with https://github.com/openclaw/openclaw/pull/75478 for any Telegram gateway-delegation changes; do not duplicate or hard-code a broader channel-specific fast path unless that PR has landed or the maintainer asks. Add a user-facing changelog fix entry. Do not change bundled runtime dependency repair policy beyond what is needed for the fail-fast CLI behavior.
