## Summary

Keep open. Current main still lets message CLI plugin-registry load failures escape before the message command's normal error/exit wrapper, so the generic fail-fast part of the report is not fixed even though current package dependency layout may reduce the original `grammy` missing-module trigger.

## What This Changes

Report requests that `openclaw message send` load bundled channel dependencies correctly or fail fast instead of leaving an `openclaw-message` child spinning after a Telegram plugin load failure.

## Best Possible Solution

Keep the message CLI's plugin-registry load failures inside a bounded nonzero command exit path with regression coverage, while leaving Telegram gateway delegation and timeout policy to their narrower related items.

## Reproduction Assessment

No, not at high confidence: I did not reproduce a current-main `openclaw-message` 100% CPU spin. Source inspection does show that strict plugin-load exceptions can still occur before the helper's clean error/exit wrapper.

## Solution Assessment

Yes for the narrow repair: wrapping or moving `ensurePluginRegistryLoaded` into the same bounded error path as `messageCommand` is the smallest maintainable fix. A Telegram-specific gateway fast path can remain separate because it does not cover all plugin-load failures.
