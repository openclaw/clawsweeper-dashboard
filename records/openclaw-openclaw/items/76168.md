## Summary

Keep open. Current main still loads configured channel plugins before the message helper's normal error/exit wrapper, while runtime plugin loading remains strict, so a plugin load failure can bypass the tested nonzero exit path; the related gateway-delegation PR is still open overlap, not proof this failure mode is fixed.

## What This Changes

The issue asks for `openclaw message send` to avoid leaving an `openclaw-message` child spinning after a bundled plugin/module load failure, either by loading runtime deps correctly or by failing fast and exiting cleanly.

## Best Possible Solution

Make message CLI plugin-registry failures a bounded, observable nonzero exit through the same finalization path as delivery failures, while coordinating any broader Telegram gateway fast path with #75478.

## Reproduction Assessment

No. The production report is detailed and current-main source shows the registry-load exception path before clean exit handling, but I did not establish a deterministic Telegram/runtime-deps repro for the high-CPU spin.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76168. In current main, `src/cli/program/message/helpers.ts` calls `ensurePluginRegistryLoaded(...)` before `runCommandWithRuntime`, so `PluginLoadFailureError` from configured-channel loading can bypass the message action's `failed`/`exit(1)` path and may leave message CLI resources alive. Add a focused repair that converts plugin-registry load failures into the same bounded nonzero-exit/finalization path as message delivery failures, with regression coverage where mocked `ensurePluginRegistryLoaded` throws. Preserve successful sends, delivery failures, read stop-hook skipping, and bounded `gateway_stop` behavior. Do not implement a broad Telegram gateway-delegation rewrite unless strictly required; coordinate with #75478/#75477. Add a user-facing `CHANGELOG.md` fix entry unless maintainers classify the change as internal-only.
