## Summary

Keep open. Current main still has the reported vulnerable shape: `message send` loads channel plugins before dispatch, plugin load failures are configured to throw, and there is no message-CLI timeout or explicit exit boundary around that pre-dispatch load; related open items overlap but have not landed a fix.

## What This Changes

The issue requests that `openclaw message send` avoid leaving an `openclaw-message` child spinning after bundled channel plugin load failures, either by fixing runtime-dependency loading before imports or by failing fast and exiting cleanly.

## Best Possible Solution

Add a message-CLI-owned fail-fast and timeout/abort boundary that starts before plugin registry loading, and separately preserve Telegram gateway delegation without bypassing existing send normalization contracts.

## Reproduction Assessment

Yes for the source-level failure path: mock or force `ensurePluginRegistryLoaded` to throw or never settle before `messageCommand`, and current main has no local catch, timeout, or signal abort around that step. The exact high-CPU production spin still needs a process-level regression once a fix is implemented.

## Solution Assessment

Yes, the maintainable direction is narrow CLI lifecycle handling plus the existing gateway-dispatch contract; fixing this in generic startup or by adding Telegram-specific core policy would be broader than necessary.
