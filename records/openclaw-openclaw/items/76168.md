## Summary

Keep open. Current main still loads the configured channel plugin registry before the message command's error/exit wrapper, while runtime loading throws on plugin load errors; the related open Telegram delegation and timeout items do not prove this plugin-load-failure path is fixed.

## What This Changes

Ensure `openclaw message send` either has usable bundled runtime dependency paths before channel imports or exits cleanly after plugin load failure instead of leaving a spinning `openclaw-message` child.

## Best Possible Solution

Keep this open until message CLI plugin-registry failures become a bounded nonzero exit with regression coverage, while letting the Telegram gateway-delegation work remain a separate coordinated fix.

## Reproduction Assessment

No: source inspection shows registry-load exceptions still occur before the helper's clean exit wrapper, but I did not establish a deterministic current-main Telegram/runtime-deps run that produces the 100% CPU spin.

## Solution Assessment

Yes for the narrow repair: wrap or move `ensurePluginRegistryLoaded` so plugin-load failures use the same bounded nonzero exit path as delivery failures, while coordinating the broader Telegram gateway-delegation path with #75478.
