## Summary

Keep open. Current main still loads the message command's plugin registry before the helper's error-to-exit wrapper, while strict runtime loading can throw `PluginLoadFailureError`; the related Telegram delegation and timeout reports remain open and do not prove this failure path is fixed.

## What This Changes

The issue asks for `openclaw message send` to fail fast or load bundled runtime dependencies correctly when a channel plugin load fails, especially for Telegram.

## Best Possible Solution

Bound message CLI plugin-registry failures inside the command's nonzero exit/finalization path, then let the separate Telegram gateway-delegation work avoid unnecessary local Telegram runtime loading.

## Reproduction Assessment

No. The production report has concrete logs and current main shows a clear source path where plugin-load exceptions escape the tested `exit(1)` wrapper, but I did not establish a deterministic local Telegram/runtime-deps CPU-spin reproduction.

## Solution Assessment

Yes for the narrow first repair. The maintainable fix is to wrap or move `ensurePluginRegistryLoaded` so plugin-load failures use the same bounded nonzero exit path as delivery failures, while coordinating broader Telegram delegation with #75478.
