## Summary

Keep open. Current main still loads the message CLI plugin registry before the command's normal error/exit wrapper, and strict plugin loading can still throw `PluginLoadFailureError` before `openclaw message send` reaches the tested `exit(1)` path.

## What This Changes

The issue asks for `openclaw message send` to avoid stuck high-CPU `openclaw-message` children after Telegram/plugin load failures by fixing runtime-dep loading order or exiting cleanly.

## Best Possible Solution

Convert message CLI plugin-registry failures into a bounded observable nonzero exit, then address runtime-dependency ordering only if a focused repro proves it remains necessary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the report has detailed production logs but no deterministic local repro. Source review does show a high-confidence structural path where strict plugin-load failures bypass the clean message CLI exit handling.

## Solution Assessment

Is this the best way to solve the issue? Yes for a narrow first fix: wrap or move `ensurePluginRegistryLoaded` so `PluginLoadFailureError` goes through the same error/finalization path as delivery failures, without adding Telegram-specific core policy or broad runtime-deps changes without repro proof.
