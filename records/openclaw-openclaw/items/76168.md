## Summary

Keep open. Current main still loads the scoped channel plugin registry before the message command's normal runtime error wrapper, and strict plugin loading can still throw `PluginLoadFailureError`; related #75478 and #75895 are open and do not yet implement the generic clean-exit behavior for this failure path.

## What This Changes

The issue requests that `openclaw message send` avoid spinning `openclaw-message` child processes after bundled channel plugin load failures by either fixing runtime dependency loading or failing fast with a nonzero exit.

## Best Possible Solution

Keep this open until message CLI registry-load failures are converted into a bounded, observable nonzero exit, with runtime-dependency ordering changes deferred until a focused repro proves that separate bug.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the report includes detailed production logs and current-main source review shows a high-confidence structural failure path, but the full high-CPU spin has not been reproduced deterministically.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow first fix: route `ensurePluginRegistryLoaded` failures through the same message CLI error/finalization path as delivery failures, then handle bundled runtime-dependency ordering only if a repro isolates it.
