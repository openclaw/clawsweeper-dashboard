## Summary

Keep open. The report is not implemented on current main: the message action still performs strict plugin loading before its normal failure/exit wrapper, so a plugin-load exception can bypass the command's tested `exit(1)` path.

## What This Changes

The issue asks for `openclaw message send` to use bundled runtime dependency paths correctly and to exit nonzero instead of leaving an `openclaw-message` child spinning after plugin load failures.

## Best Possible Solution

Keep this open until message CLI plugin-load failures are converted into a bounded, observable nonzero exit, with the runtime-dependency ordering fixed only after an isolated repro or test fixture proves it.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the report includes detailed production logs, but no deterministic local repro; current-main code review does give a high-confidence structural path where plugin-load failures bypass the message action's clean exit handling.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow first fix: route `ensurePluginRegistryLoaded` failures through the same nonzero exit/finalization path as message delivery failures, then separately address the runtime-dependency ordering if it can be reproduced.
