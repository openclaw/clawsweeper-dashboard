## What This Changes

The PR derives readable plain text from Telegram HTML when an HTML parse-mode send is rejected without caller-supplied `plainText`, preserving anchor URLs as `label (url)` and adding a regression test.

## Best Possible Solution

Land the narrow Telegram fallback hardening after the normal required checks, keeping explicit caller `plainText` authoritative and deriving readable fallback text only for HTML-mode sends without one.

## Reproduction Assessment

Yes. Current main source shows an HTML-mode send can enter the HTML parse-error retry path with fallback text derived from rendered HTML when no caller `plainText` is supplied; I did not rerun a live before-case in this read-only review.

## Solution Assessment

Yes. The PR addresses the implicated sender-level fallback without reopening the lazy sender/core cron path already fixed by https://github.com/openclaw/openclaw/pull/81758, and it preserves caller-provided `plainText` as the stronger contract.
