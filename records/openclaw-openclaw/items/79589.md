## Summary

Keep open. Current main still exposes a FIFO-only command queue with no priority option, and two open PRs already use closing syntax for this issue, so the issue should stay paired with that review path until one implementation lands or is rejected.

## What This Changes

Request priority levels for command queue entries so latency-sensitive work can overtake background work within the same lane while preserving same-priority FIFO ordering and preventing starvation.

## Best Possible Solution

Review and land one canonical priority-scheduling implementation with queue API, foreground/background caller mapping, docs, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes. Current main has no priority field on `CommandQueueEnqueueOptions` and drains lanes with `state.queue.shift()`, so a later latency-sensitive entry cannot overtake earlier background work in the same lane.

## Solution Assessment

Unclear until the linked PR review is resolved. A priority field with in-place selection and starvation handling is a plausible direction, but maintainers should choose one implementation path and document the caller priority contract.
