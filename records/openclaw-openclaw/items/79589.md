## Summary

Keep open: current main and v2026.6.8 have static foreground/normal/background queue priority, but the requested starvation guard is still missing from the shipped/current queue path and open closing PRs already own the remaining scheduler-policy work.

## What This Changes

The issue requests command-queue priority levels plus a starvation guard so urgent work can run ahead of background work in the same lane while preserving FIFO among equal-priority entries.

## Best Possible Solution

Review and land one maintainer-approved command-queue aging policy with regression coverage for static priority, equal-priority FIFO, and low-priority aging; keep this issue open until that PR merges or a canonical replacement is chosen.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: hold a lane busy, enqueue background work, keep adding foreground work, and current static priority insertion plus `shift()` provides no age-based promotion.

## Solution Assessment

Unclear. The command queue is the right owner boundary, but maintainers still need to decide the aged-background-versus-fresh-foreground policy before either linked PR is the best fix.
