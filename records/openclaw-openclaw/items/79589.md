## Summary

Keep this open: current main still has FIFO-only command lane draining with no enqueue priority option, and two open PRs explicitly close this issue, so the issue should remain paired with that review path until one implementation lands or is closed.

## What This Changes

The issue requests priority-aware command queue scheduling within a lane, preserving FIFO order among equal priorities and preventing starvation of lower-priority work.

## Best Possible Solution

Land or explicitly close the linked priority-scheduling PRs so OpenClaw ends up with one documented command queue priority contract and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows no `priority` enqueue option and a FIFO `state.queue.shift()` drain path, with existing tests asserting FIFO order.

## Solution Assessment

Unclear until the linked PRs are reviewed. A priority field plus starvation guard is a plausible direction, but the canonical implementation and caller wiring need maintainer acceptance before this issue is resolved.
