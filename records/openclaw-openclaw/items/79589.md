## Summary

Current main still has FIFO-only command lane draining and no enqueue priority option, while two open PRs already use closing syntax for this issue. Keep this open until maintainers choose, land, or close the linked implementation path.

## What This Changes

Add priority levels to the command queue so latency-sensitive work can jump ahead of background work within a lane while preserving FIFO ties and starvation protection.

## Best Possible Solution

Review and land one canonical priority-queue implementation, preferably with caller wiring, docs, regression coverage, and real behavior proof, then close this issue from the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the enqueue options lack priority and `drainLane` dequeues with `state.queue.shift()`, so the missing priority capability is directly observable.

## Solution Assessment

Unclear until maintainer review. A priority field with in-place selection and starvation protection is a plausible narrow design, but the core scheduling semantics and overlapping PRs need one accepted implementation path.
