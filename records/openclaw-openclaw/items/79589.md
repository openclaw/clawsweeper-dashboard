## Summary

Current main still has FIFO-only command lane draining with no priority field, and this issue is already paired with two open closing PRs, so it should remain open until maintainers choose and land or close the implementation path.

## What This Changes

The issue requests priority-aware command queue scheduling so urgent lane work can bypass lower-priority queued work while preserving same-priority FIFO order and preventing starvation.

## Best Possible Solution

Review and land one canonical priority-queue implementation with normalized priority values, starvation coverage, foreground/background caller wiring where needed, and updated docs/tests; close this issue only after the chosen PR merges.

## Reproduction Assessment

Yes for the current limitation: source inspection shows no `priority` enqueue option, FIFO `state.queue.shift()` lane draining, and docs that describe FIFO queue behavior. I did not run a live latency repro because the missing priority capability is directly visible in current main and the latest release.

## Solution Assessment

Unclear until maintainers choose between the open implementation PRs. A priority field plus in-place priority scan and starvation guard is a plausible narrow design, but the accepted fix should reconcile https://github.com/openclaw/openclaw/pull/75299 with the broader caller-wiring work in https://github.com/openclaw/openclaw/pull/80199.
