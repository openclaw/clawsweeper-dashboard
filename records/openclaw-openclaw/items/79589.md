## Summary

Current main still exposes only FIFO command-lane draining and no enqueue priority option, and this issue has open closing PRs, so it should stay open until the linked implementation path is merged, revised, or closed.

## What This Changes

Add priority levels to the command queue so urgent work can run before lower-priority entries in the same lane while preserving FIFO order within a priority and preventing starvation.

## Best Possible Solution

Review the active priority-queue PRs, pick one canonical implementation, and land a documented queue priority contract with focused regression coverage before closing this issue.

## Reproduction Assessment

Yes. Source inspection on current main shows no `priority` field in the enqueue options and FIFO draining via `state.queue.shift()`, with docs and tests also reflecting FIFO behavior.

## Solution Assessment

Unclear until the linked PRs finish review. A priority field plus starvation guard is a plausible narrow design, but the exact queue API and foreground/background mapping need maintainer approval because they change core scheduling semantics.
