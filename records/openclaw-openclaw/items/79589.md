## Summary

Current main still has FIFO-only lane draining and no enqueue priority contract, so the issue remains valid; it should stay open because two open PRs already use closing syntax and own the implementation path.

## What This Changes

The issue requests priority-aware command queue scheduling so higher-priority entries can run before lower-priority queued work within the same lane while preserving equal-priority FIFO order and preventing starvation.

## Best Possible Solution

Review and land or supersede the active priority-queue implementation, preferably one that includes the queue API contract, caller priority wiring, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the enqueue options lack priority and `drainLane()` selects work with `state.queue.shift()`, which directly demonstrates the FIFO-only behavior.

## Solution Assessment

Unclear until maintainer review chooses between the linked implementations. The priority field plus in-place selection and starvation guard is a plausible direction, but the canonical scheduling contract should be settled in the open PR review.
