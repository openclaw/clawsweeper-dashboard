## Summary

Current main still drains each command lane FIFO and exposes no priority option, and the issue is already paired with an open closing PR, so it should stay open until that PR is reviewed, merged, or closed.

## What This Changes

Request priority-aware command queue scheduling within a lane, preserving equal-priority FIFO ordering and preventing starvation.

## Best Possible Solution

Review and either land or revise the linked priority-scheduling PR so the command queue has one documented priority contract and regression coverage in `src/process`.

## Reproduction Assessment

Yes. Source inspection on current main shows the enqueue options lack priority and the drain loop uses `state.queue.shift()`, so the reported FIFO-only behavior is directly reproducible from the implementation.

## Solution Assessment

Unclear until the linked PR is reviewed. A priority field plus in-place selection and starvation coverage is a plausible narrow design, but the core scheduling semantics should be accepted by maintainers before this issue is closed.
