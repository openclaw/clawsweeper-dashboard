## Summary

Keep open: current main still exposes only FIFO command-lane scheduling with no priority option, and two open PRs already use closing syntax for this issue, so the issue should remain paired with that review path.

## What This Changes

Request adds priority levels to the command queue so urgent work can jump lower-priority queued work within the same lane while preserving equal-priority FIFO ordering and starvation protection.

## Best Possible Solution

Choose one canonical priority-queue implementation, document the scheduling contract, land focused regression and real behavior proof, then close the issue from the merged PR.

## Reproduction Assessment

Yes. Source inspection on current main shows the enqueue options lack priority and the drain loop uses FIFO `state.queue.shift()`, so the missing capability is directly verifiable from the implementation.

## Solution Assessment

Unclear until maintainer review settles the canonical PR. Priority-aware in-lane selection with FIFO ties and starvation protection is a plausible design, but the exact API and caller priority policy are product/runtime decisions.
