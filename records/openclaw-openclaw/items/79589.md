## Summary

Current main still has FIFO-only command queue draining and no enqueue priority option, while two open PRs already use closing syntax for this issue. This should stay open until the active priority-queue PR path is merged, revised, or closed.

## What This Changes

The issue requests priority support for command-queue entries so latency-sensitive work can jump ahead of lower-priority background work within the same lane without losing FIFO ordering among equal priorities.

## Best Possible Solution

Use the active PR review path to land one documented priority contract with regression coverage, then close this issue after the selected PR merges.

## Reproduction Assessment

Yes. Source inspection on current main shows there is no priority field and the drain loop uses FIFO `shift()`, so the reported missing capability is directly reproducible from the implementation.

## Solution Assessment

Unclear until the linked PR review completes. A priority field with in-place selection and starvation protection is a plausible narrow direction, but the scheduling contract should be finalized by maintainers before closing this issue.
