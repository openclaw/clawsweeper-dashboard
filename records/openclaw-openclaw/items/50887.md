## Label Justifications

- `impact:message-loss`: The reported stream assembly path can replace delivered Chinese characters with U+FFFD, corrupting message content.
- `impact:session-state`: The corrupted response text can flow into captured output or transcripts and become session context drift.

## Summary

Keep open: current main and the latest release still carry a boundary-unsafe default UTF-8 stream decode path for child output, and the prior target-repo fix PRs were closed unmerged rather than shipped.

## What This Changes

The issue requests fixing Chinese UTF-8 corruption in streamed webchat/group-chat agent responses when multi-byte characters are split across output chunks.

## Best Possible Solution

Land a narrow target-repo fix in the shared output decoder so the default UTF-8 path is stateful across chunks, preserve Windows legacy-codepage fallback behavior, and add split-CJK regression coverage for decoder and child-adapter output.

## Reproduction Assessment

Yes for a source-level reproduction: split a Chinese UTF-8 codepoint across two Buffer chunks on the current default child-output decoder path and per-chunk decoding emits replacement characters. No live webchat/cursor2api reproduction was run.

## Solution Assessment

Yes on direction: the durable fix belongs in createWindowsOutputDecoder's shared default UTF-8 path, not only in a child-local wrapper, because current stdout and stderr handling already centralizes decoding there.
