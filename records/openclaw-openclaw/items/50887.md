## Summary

Keep open: current main and the latest release still have a byte-boundary-unsafe default UTF-8 decoder path relevant to streamed child/process output, and https://github.com/openclaw/openclaw/pull/51303 is still open with closing syntax for this issue. The issue should be resolved by landing, revising, or superseding that linked PR rather than being closed early.

## What This Changes

The issue asks OpenClaw to stop corrupting Chinese characters during streamed WebChat group responses by fixing UTF-8 chunk-boundary assembly.

## Best Possible Solution

Resolve through https://github.com/openclaw/openclaw/pull/51303 or a narrow replacement that makes createWindowsOutputDecoder boundary-safe on the default UTF-8 path, preserves the Windows legacy-codepage fallback, adds split-CJK regression coverage, and verifies a real child-output or WebChat/cursor2api stream.

## Reproduction Assessment

Yes for the core split-byte failure: current source has a per-chunk Buffer.toString("utf8") default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

Yes on direction, but not yet as a closed issue outcome: stateful UTF-8 decoding is the right primitive, and the maintainable fix belongs in createWindowsOutputDecoder while preserving the Windows fallback and adding regression coverage.
