## Label Justifications

- `P2`: This is a real user-visible streaming text corruption bug with a localized decoder fix path, not a crash, security bypass, or total channel outage.
- `impact:message-loss`: The reported symptom corrupts delivered chat message text by replacing individual Chinese characters with U+FFFD.
- `impact:session-state`: Corrupted assistant output can be persisted into the session transcript and reused as conversation context.

## Summary

Keep open: current main and the latest release still have a byte-boundary-unsafe default UTF-8 decoder path, and an open linked PR already targets this issue with closing syntax, so the issue should be resolved by landing or superseding that PR rather than closing early.

## What This Changes

The issue reports Chinese text corruption in streamed WebChat/group-chat replies where split UTF-8 byte sequences are decoded into U+FFFD replacement characters.

## Best Possible Solution

Resolve through https://github.com/openclaw/openclaw/pull/51303 or a narrow replacement that makes createWindowsOutputDecoder boundary-safe on the default UTF-8 path, preserves the Windows legacy-codepage fallback, adds split-CJK coverage, and verifies a real child-output or WebChat/cursor2api stream.

## Reproduction Assessment

Yes for the core split-byte failure: current source has a per-chunk Buffer.toString("utf8") default decoder path, and a focused Node probe reproduces U+FFFD when Chinese UTF-8 bytes are split across chunks. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

Yes on direction, but not yet through a closed issue outcome: stateful UTF-8 decoding is the right primitive, and the maintainable fix belongs in createWindowsOutputDecoder while preserving the Windows fallback and adding regression coverage.
