## What This Changes

The issue reports streamed Chinese WebChat/group-chat responses randomly replacing individual characters with U+FFFD, likely from UTF-8 chunk-boundary decoding.

## Best Possible Solution

Land a narrow replacement in core that makes createWindowsOutputDecoder boundary-safe on the default UTF-8 path, preserves the Windows legacy-codepage fallback, adds split-CJK regression coverage, and confirms whether the reported WebChat route also needs JSONL/socket handling.

Yes for the source-level failure: current main still decodes default UTF-8 chunks with Buffer.toString("utf8"), and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api setup was not live-run in this read-only review.

## Solution Assessment

Yes on direction: stateful UTF-8 decoding belongs in the shared createWindowsOutputDecoder boundary while preserving Windows legacy-codepage behavior. The closed child-local PR is no longer the best patch shape for current main.
