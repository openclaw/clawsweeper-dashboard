## Summary

Keep open. Current main and the latest release still contain a chunk-boundary-unsafe default UTF-8 decode path relevant to streamed child output, and open PR #51303 already targets this issue with closing syntax, so early cleanup would be premature.

## What This Changes

Fix WebChat group-chat streaming corruption where Chinese/CJK characters can appear as U+FFFD when UTF-8 bytes are split across chunks.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes the shared default UTF-8 stream decoder boundary-safe on non-Windows, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for the decoder and child-adapter wiring, and verifies the WebChat/cursor2api path plus any JSONL socket or truncation surfaces involved.

## Reproduction Assessment

Partially yes: the byte-boundary corruption class is reproducible with a read-only Node split-CJK probe and matches the current default decoder code. The full cursor2api/WebChat group-chat scenario was not live-reproduced here.

## Solution Assessment

No, closing this now is not the best way to solve it. The maintainable path is to review, land, revise, or supersede PR #51303 with focused regression coverage rather than closing the issue or queueing a duplicate repair lane.
