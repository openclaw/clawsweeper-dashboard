## Summary

Keep open: current main still has a boundary-unsafe default UTF-8 decode path for streamed child output on non-Windows platforms, and open PR #51303 already targets this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to preserve Chinese UTF-8 characters during WebChat group-chat streaming instead of replacing random characters with U+FFFD.

## Best Possible Solution

Land PR #51303 or a narrow superseding fix that makes default UTF-8 stream decoding boundary-safe, preserves Windows legacy-codepage fallback behavior, and adds split-CJK regression coverage for the decoder and child adapter route.

## Reproduction Assessment

Source-level yes: splitting Chinese UTF-8 bytes and decoding each chunk independently reproduces U+FFFD, and current main still does that in the default decoder branch. The full cursor2api/WebChat setup was not live-run.

## Solution Assessment

Yes in direction: the narrow maintainable fix is at the shared stream-decoder/child-output boundary, with regression tests. Because child.ts already delegates to createWindowsOutputDecoder on current main, the safest current fix point is the default decoder branch or an equivalent narrowly scoped replacement.
