## Summary

Keep open: current main and the latest release still have a boundary-unsafe default UTF-8 decode path for streamed child output, and open PR #51303 already targets this issue with closing syntax.

## What This Changes

The issue reports random U+FFFD replacement characters in longer Chinese WebChat group-chat responses, likely from UTF-8 multi-byte stream chunks decoded independently.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes default UTF-8 stream decoding boundary-safe, preserves Windows legacy-codepage fallback behavior, and adds split-CJK regression coverage for both the decoder and child adapter wiring.

## Reproduction Assessment

Yes for the byte-boundary corruption class: a read-only split-CJK Node probe produced U+FFFD with Buffer.toString, and current main uses that same default decoder path for child output chunks. The full cursor2api/WebChat group-chat scenario was not live-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is a boundary-safe shared decoder change, using StringDecoder or streaming TextDecoder semantics while preserving Windows legacy-codepage fallback. Closing this issue before PR #51303 merges or is superseded would be premature.
