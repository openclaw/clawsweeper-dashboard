## Summary

Keep open: current main and v2026.6.6 still have boundary-unsafe per-chunk UTF-8 decoding on relevant stream surfaces, and an open closing PR exists but is cross-repo and conflicting rather than merged.

## What This Changes

The issue reports Chinese characters being replaced with U+FFFD during streamed webchat/group-chat responses, likely from UTF-8 byte sequences split across stream chunks.

## Best Possible Solution

Resolve this through maintainer review of the open closing PR or a narrow target-repo replacement that makes the shared default UTF-8 decoder stateful across chunks, preserves Windows legacy-codepage fallback, and adds split-CJK coverage for the decoder plus the relevant child/socket delivery path.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: split a Chinese UTF-8 codepoint across Buffer chunks on the current default decoder path and per-chunk `buffer.toString("utf8")` can emit U+FFFD. I did not run the full cursor2api/WebChat group-chat scenario.

## Solution Assessment

Yes on direction, but not by closing this issue now: the maintainable fix belongs in the shared default UTF-8 decoder, with proof for the child and socket paths. The open closing PR or a narrow replacement should land before the issue closes.
