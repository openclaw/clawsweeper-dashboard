## Summary

Keep open: current main and v2026.6.10 still contain a default UTF-8 streaming decode path that can emit U+FFFD when CJK bytes split across chunks, and no same-repository merged PR owns the fix.

## What This Changes

Fix streaming assembly so Chinese/CJK characters are preserved instead of being replaced with U+FFFD in WebChat/group-chat responses.

## Best Possible Solution

Land or sponsor a narrow shared streaming-decoder fix for default UTF-8 output paths, with split-CJK coverage and route proof for the reported WebChat/provider flow.

## Reproduction Assessment

Yes at source level: current main still has a per-chunk UTF-8 decode branch and a focused split-byte probe shows the U+FFFD failure mode. The full WebChat/cursor2api-go environment was not live-reproduced here.

## Solution Assessment

Yes on direction, but not through the currently linked child-local fork as-is. The best fix is the shared default decoder path plus focused WebChat/provider route proof.
