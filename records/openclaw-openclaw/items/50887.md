## Summary

Keep open: current main still has a boundary-unsafe UTF-8 decode branch that can produce U+FFFD for split CJK bytes, and the issue is already paired with open closing pull request https://github.com/openclaw/openclaw/pull/51303.

## What This Changes

The issue asks OpenClaw to stop corrupting Chinese characters with U+FFFD during streamed WebChat group responses.

## Best Possible Solution

Make createWindowsOutputDecoder boundary-safe for the default UTF-8 path, preserve the Windows legacy-codepage fallback, add default-decoder and child-adapter coverage, and verify the reported WebChat route before closing.

## Reproduction Assessment

Yes, at source level. A split multibyte UTF-8 CJK sequence reproduces U+FFFD with per-chunk Buffer.toString("utf8"), and current main still uses that pattern in the default shared decoder path.

## Solution Assessment

No, closing this issue now is not the best path. The narrow maintainable solution is to resolve, revise, land, or supersede https://github.com/openclaw/openclaw/pull/51303 with a fix in the shared decoder rather than opening a duplicate repair lane from the issue.
