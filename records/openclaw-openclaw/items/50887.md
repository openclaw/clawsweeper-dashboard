## Summary

Keep open: current main and the latest release still contain byte-boundary-unsafe UTF-8 decoding on the default output-decoder path, and an open linked PR already targets this issue with closing syntax. The issue should be resolved by revising, landing, or superseding that PR rather than closing early or queueing a duplicate fix.

## What This Changes

The issue reports Chinese streaming text corruption where multi-byte UTF-8 characters are replaced with U+FFFD during WebChat/group-chat delivery.

## Best Possible Solution

Resolve through https://github.com/openclaw/openclaw/pull/51303 or a narrow replacement that makes the default UTF-8 output decoder boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for the default decoder and child adapter wiring, and verifies the reported WebChat/cursor2api path or an equivalent real stream.

## Reproduction Assessment

Yes for the core split-byte failure: current source has a per-chunk Buffer.toString("utf8") default decoder path, and a focused Node probe reproduces U+FFFD when Chinese UTF-8 bytes are split across chunks. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

Yes on direction, but not yet on the open PR shape: stateful UTF-8 decoding is the right primitive, but the maintainable fix belongs in createWindowsOutputDecoder while preserving the Windows fallback and adding regression coverage.
