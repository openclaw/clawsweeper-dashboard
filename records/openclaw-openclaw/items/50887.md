## Summary

Keep open: current `main` and the latest release still have a chunk-boundary-unsafe default UTF-8 decode path for child/process output, and the only open linked fix is a conflicting fork PR rather than a landed target-repo fix.

## What This Changes

The issue reports Chinese text corruption in webchat group responses, likely from UTF-8 multibyte sequences being decoded across streaming chunk boundaries.

## Best Possible Solution

Land or sponsor a narrow shared-decoder fix that makes the default UTF-8 stream path stateful, preserves the Windows legacy-codepage fallback, and adds split-CJK regression coverage for the decoder and child-output wiring.

## Reproduction Assessment

Yes at source level: the current default decoder path still decodes each non-Windows output chunk independently with `buffer.toString("utf8")`, and a focused byte-split probe reproduces U+FFFD output. The full WebChat/cursor2api setup was not live-run in this read-only review.

## Solution Assessment

Yes on the fix direction, but not through issue closure yet. The maintainable fix point is the shared output decoder default path plus focused regression tests, while the open fork PR is conflicting and child-local.
