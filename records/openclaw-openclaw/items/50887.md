## Summary

Keep open. Current main still contains a boundary-unsafe default UTF-8 decode path for streamed child output, the latest release tag contains the same fallback, and open PR #51303 already targets this issue with closing syntax, so this should be resolved through that PR or a narrow replacement rather than closed now.

## What This Changes

The issue reports random U+FFFD replacement characters in longer Chinese WebChat group responses, likely from UTF-8 byte sequences split across streaming boundaries.

## Best Possible Solution

Review, land, revise, or supersede PR #51303 with a narrow shared decoder fix that makes default UTF-8 child-output decoding boundary-safe, preserves the Windows legacy-codepage fallback, and adds focused split-CJK regression coverage for the decoder and child adapter route.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially yes: the byte-boundary corruption class reproduces with a read-only Node split-CJK probe and matches the current non-Windows child-output decoder path, but the full cursor2api/WebChat group-chat route was not live-run here.

## Solution Assessment

Is this the best way to solve the issue? Yes, the maintainable fix belongs in the shared decoder/child adapter path rather than in WebChat rendering; early closure is not appropriate while PR #51303 is still open and unmerged.
