## Summary

Keep open: current main and the latest release still decode the default child-output UTF-8 stream per Buffer chunk, while the same-root upstream PRs are closed unmerged and the linked fork PR is open but conflicting.

## What This Changes

The issue reports random U+FFFD replacement characters in longer Chinese WebChat group-chat responses, likely from UTF-8 stream chunks split across multibyte character boundaries.

## Best Possible Solution

Land or sponsor a narrow shared-decoder fix that makes the default UTF-8 stream path stateful, preserves Windows legacy-codepage fallback, and adds split-CJK coverage for the decoder and child-output wiring.

## Reproduction Assessment

Yes at source level: current main still decodes the default child-output stream path per Buffer chunk, and a focused split-byte probe shows the U+FFFD failure mode. The full WebChat/cursor2api setup was not live-reproduced here.

## Solution Assessment

Yes on direction, but not through the current linked fork PR as-is. The best fix point is the shared output decoder default path plus focused route coverage, not another child-local patch that bypasses the Windows compatibility wrapper.
