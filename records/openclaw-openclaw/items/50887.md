## Summary

Keep open. Current `main` still has a chunk-boundary-unsafe default UTF-8 decode path feeding streamed child output, and open PR #51303 already targets this issue with closing syntax, so closing or queueing a duplicate repair lane would be premature.

## What This Changes

Fix streamed WebChat/group-chat output so Chinese/CJK UTF-8 characters are not randomly replaced with U+FFFD replacement characters.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes the shared default UTF-8 output decoder boundary-safe on non-Windows, preserves the Windows legacy-codepage fallback, adds split-CJK regression coverage for the decoder and child adapter wiring, and verifies any JSONL socket or WebChat/provider route needed for the reported setup.

## Reproduction Assessment

Partially yes: the byte-boundary corruption class is reproducible with a read-only Node split-CJK probe and matches current source. The full cursor2api/WebChat group-chat scenario was not live-reproduced here.

## Solution Assessment

No, closing this issue now is not the best way to solve it. The maintainable path is to review, land, revise, or supersede PR #51303 with focused regression coverage rather than closing the issue or creating a duplicate repair lane.
