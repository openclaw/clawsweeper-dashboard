## Summary

Keep open: current main still has a boundary-unsafe UTF-8 decode path relevant to streamed child/process output, and the issue is paired with an open unmerged PR that uses closing syntax for this issue. The shipped v2026.5.12 tag still contains the same default decoder fallback, so this is not implemented yet.

## What This Changes

The issue requests fixing streamed Chinese/CJK output corruption where split UTF-8 chunks can become U+FFFD replacement characters in WebChat/group-chat responses.

## Best Possible Solution

Revise or replace the open fix PR so the default branch of createWindowsOutputDecoder is UTF-8 boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK regression coverage for the shared decoder and child adapter wiring, and verifies the reported route or an equivalent real child-output stream.

## Reproduction Assessment

Yes for the core byte-boundary failure: current main has a per-chunk Buffer.toString("utf8") path and a focused Node probe reproduces U+FFFD corruption for split Chinese bytes. No for the full WebChat/cursor2api setup, which was not live-run in this read-only review.

## Solution Assessment

Not yet: the right primitive is stateful UTF-8 decoding, but the maintainable fix should live in the shared decoder rather than only in stale child-adapter handlers. The linked open PR should be reviewed, revised, landed, or superseded before this issue closes.
