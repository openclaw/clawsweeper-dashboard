## Label Justifications

- `P2`: This is a real user-visible streaming text corruption bug with a localized decoder fix path, not a crash, security bypass, or total channel outage.
- `impact:message-loss`: The reported symptom corrupts delivered chat message text by replacing individual Chinese characters with U+FFFD.
- `impact:session-state`: Corrupted assistant output can be persisted into the session transcript and reused as conversation context.

## Summary

Keep open: current main and the latest release still have a default per-chunk UTF-8 decode path that can emit U+FFFD, and the issue is paired with an open PR that uses closing syntax, so it should be resolved by landing, revising, or superseding that PR.

## What This Changes

The issue reports streamed Chinese output in WebChat group chat being corrupted when split UTF-8 characters are decoded as U+FFFD replacement characters.

## Best Possible Solution

Move stateful UTF-8 buffering into createWindowsOutputDecoder, preserve the Windows legacy-codepage fallback, add split-CJK coverage for the default decoder and child adapter wiring, and resolve the linked open PR through review, revision, landing, or supersession.

## Reproduction Assessment

Yes for the core split-byte failure: current main has a per-chunk Buffer.toString("utf8") default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. No full WebChat/cursor2api live reproduction was run in this read-only review.

## Solution Assessment

The right primitive is stateful UTF-8 decoding, but the maintainable fix belongs in the shared decoder rather than bypassing it in the child adapter. The linked open PR should be revised, landed, or superseded before this issue closes.
