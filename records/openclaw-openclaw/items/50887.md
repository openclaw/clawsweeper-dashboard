## Summary

Keep open: current main and the latest release still contain a boundary-unsafe default UTF-8 decode path for streamed child output, and open PR #51303 already targets this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to stop replacing streamed Chinese characters with U+FFFD in WebChat/group-chat responses when UTF-8 characters are split across chunks.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes default UTF-8 stream decoding boundary-safe, preserves Windows legacy-codepage fallback behavior, and adds split-CJK regression coverage for the decoder and child adapter wiring.

## Reproduction Assessment

Partially yes: the byte-boundary corruption class is reproducible with a read-only Node split-CJK probe and matches the current default decoder code. The full cursor2api/WebChat group-chat setup was not live-reproduced here.

## Solution Assessment

Yes: the maintainable fix is a narrow boundary-safe decoder change at the shared child-output seam, with tests that preserve the Windows legacy-codepage behavior. Closing this issue before PR #51303 lands or is superseded would be premature.
