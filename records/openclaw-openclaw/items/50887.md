## Summary

Keep open: current main and v2026.5.20 still have a byte-boundary-unsafe default UTF-8 path in the shared child-output decoder, and https://github.com/openclaw/openclaw/pull/51303 is an open closing PR for this issue, so the issue should be resolved by landing, revising, or superseding that PR rather than closed early.

## What This Changes

The issue requests a fix for Chinese characters being corrupted to U+FFFD during streamed WebChat/group-chat responses.

## Best Possible Solution

Resolve through https://github.com/openclaw/openclaw/pull/51303 or a narrow replacement that makes createWindowsOutputDecoder boundary-safe on the default UTF-8 path, preserves Windows legacy-codepage fallback, adds split-CJK regression coverage, and verifies a real child-output or WebChat/cursor2api stream.

## Reproduction Assessment

Yes for the core split-byte failure: current source has a per-chunk Buffer.toString("utf8") default decoder path, and a focused Node probe reproduces U+FFFD for split Chinese bytes. The full WebChat/cursor2api route was not live-run in this read-only review.

## Solution Assessment

Yes on direction, but not as a closed issue outcome yet: stateful UTF-8 decoding is the right primitive, and the maintainable fix belongs in createWindowsOutputDecoder while preserving the Windows fallback and adding regression coverage.
