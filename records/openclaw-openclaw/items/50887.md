## Summary

Keep open: current main and v2026.6.8 still have a boundary-unsafe default UTF-8 stream decoder for non-Windows child output, and the remaining fix is not landed; the open fork PR that references this issue is conflicting and should be resolved by maintainer review rather than treated as implementation proof.

## What This Changes

The issue reports UTF-8 streaming corruption where Chinese response characters in WebChat group chat are replaced by U+FFFD.

## Best Possible Solution

Land or sponsor a narrow upstream fix that makes the shared default UTF-8 output decoder stateful, preserves the Windows legacy-codepage fallback, adds split-CJK regression coverage for the default path and child wiring, and then resolves the conflicting fork PR.

## Reproduction Assessment

Yes at source level: current main still decodes the default child-output stream path per Buffer chunk, and a focused Node probe reproduces U+FFFD for split Chinese UTF-8 bytes. The full WebChat/cursor2api setup was not live-reproduced.

## Solution Assessment

Yes on direction, but not through the current fork PR as-is. The best fix point is the shared decoder default path plus focused route coverage, not another child-local patch that bypasses the Windows compatibility wrapper.
