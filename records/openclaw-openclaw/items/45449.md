## What This Changes

The PR adds Discord `ackReactionTiming` config, docs, and tests, and changes the Discord message handler so ack reactions can either fire on receipt or be deferred until `onAgentRunStart`.

## Best Possible Solution

Port the behavior to the current `extensions/discord` plugin implementation with Discord-scoped channel/account `ackReactionTiming`, keep `received` as the default, start the deferred lifecycle only from `onAgentRunStart`, preserve ack scope gating and cleanup semantics, and update config metadata, docs, tests, and changelog together.

Yes. Source inspection on current main is enough: the active Discord handler queues the ack reaction before dispatch, the helper immediately calls `setQueued()`, and no config/docs surface exposes `ackReactionTiming`.

## Solution Assessment
