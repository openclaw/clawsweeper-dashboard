## Summary

Keep this PR open. Current main still lacks Discord ack reaction timing and still starts the Discord ack/status reaction lifecycle at message receipt; the PR remains a useful implementation candidate, but it targets the old `src/discord` layout and needs to be ported to the active Discord plugin surface before merge.

## What This Changes

The PR adds Discord `ackReactionTiming` config/docs/tests intended to preserve receipt-level ack reactions by default while optionally deferring the initial ack until `onAgentRunStart`.

## Best Possible Solution

Port the feature to `extensions/discord` with a Discord-scoped `channels.discord.ackReactionTiming` and account override, keep the default as `received`, preserve the lifecycle guards from the later PR commit, update config metadata/docs, and add regression coverage for default immediate ack, run-start deferral, scope gating, and no pre-run status reaction.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the gap: the active Discord handler calls the queue helper before dispatch, and neither config schemas nor docs expose an `ackReactionTiming` option.

## Solution Assessment

No, not as the PR currently stands. The proposed behavior is a good narrow fix, but the implementation is on the old `src/discord` path; the maintainable solution is to port it to the active `extensions/discord` plugin surface.
