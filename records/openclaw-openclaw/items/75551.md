## Summary

Keep this issue open. Current main still has a credible Discord-owned gap: the managed proxy contract covers WebSocket egress, but the Discord gateway WebSocket path does not explicitly preserve the original Discord gateway hostname for proxied WSS TLS validation, and existing Discord proxy tests only cover the channel-specific proxy agent path.

## What This Changes

The issue asks OpenClaw to make Discord gateway WebSocket connections work with `proxy.enabled`/`proxy.proxyUrl` by validating TLS against `gateway.discord.gg` rather than the local forward proxy host, or to provide an equivalent per-channel proxy bypass.

## Best Possible Solution

Keep the issue open and fix the Discord-owned gateway transport so managed and channel-specific proxied WSS connections preserve the original Discord gateway hostname for TLS validation, backed by a focused regression test and any necessary docs clarification for Discord proxy behavior.

## Reproduction Assessment

Yes. The issue gives a concrete managed proxy config, Discord account setup, and the exact TLS hostname mismatch; current-main source inspection shows the Discord WSS path lacks explicit hostname preservation and has no managed-proxy regression coverage.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable direction is likely a Discord transport fix plus regression tests, but a maintainer should approve the exact TLS/proxy approach before implementation.
