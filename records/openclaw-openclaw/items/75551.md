## Summary

Keep this issue open. Current main still has a credible gap between the managed proxy contract and the Discord gateway WebSocket implementation: the public proxy docs promise WebSocket egress through `proxy.enabled`, while the Discord gateway path only wires an explicit Discord/debug proxy agent and does not pin the WSS TLS `servername` to the original Discord gateway host.

## What This Changes

The issue asks OpenClaw to make Discord gateway WebSocket-over-HTTP-proxy TLS validation use the original target host, such as `gateway.discord.gg`, when `proxy.enabled` routes traffic through an operator-managed proxy.

## Best Possible Solution

Keep the issue open and fix the Discord-owned gateway transport so WSS connections preserve the original gateway hostname for TLS validation under managed and channel-specific HTTP proxy paths, with a regression test covering `proxy.enabled`/`OPENCLAW_PROXY_URL` and `gateway.discord.gg` servername behavior.

## Reproduction Assessment

Yes. The issue gives concrete config, proxy URL, channel setup, and the exact TLS hostname error; current source inspection shows there is no Discord regression test or explicit WSS `servername` preservation for the global proxy path.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable fix appears to be in `extensions/discord/src/monitor/gateway-plugin.ts`, likely by deriving the WSS URL hostname and passing it as `servername` while adding a focused Discord gateway proxy regression test.
