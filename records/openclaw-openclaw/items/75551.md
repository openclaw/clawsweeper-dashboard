## Summary

Keep this open. The report is current, detailed, and source-backed: current main has Discord-specific proxy handling only for `channels.discord.proxy`, while the new managed `proxy.enabled` path routes WebSocket traffic through process-level proxy hooks and there is no Discord gateway fix equivalent to the Telegram transport change cited in the report.

## What This Changes

The issue requests fixing Discord gateway WebSocket-over-managed-proxy TLS hostname validation so `proxy.enabled` can be used without Discord reconnect loops.

## Best Possible Solution

Add a Discord-owned managed-proxy gateway transport path that preserves end-to-end TLS validation against `gateway.discord.gg` when `proxy.enabled` is active, with a focused regression test using a local CONNECT proxy and docs/changelog updates if the public proxy contract changes.

## Reproduction Assessment

Yes. The issue provides exact config, environment, and primary TLS error, and current source shows a plausible gap: Discord explicit proxy support is keyed to `channels.discord.proxy` while the managed proxy uses process-level hooks. A targeted automated repro should simulate a local CONNECT proxy and assert the TLS servername/hostname remains the Discord gateway host.

## Solution Assessment

Unclear until maintainers choose the safest proxy boundary. The reporter’s primary direction, Discord-owned explicit WebSocket proxy handling that preserves target TLS identity, looks narrower than adding a broad bypass knob; a global per-channel bypass should be secondary unless owner review finds explicit proxying unsafe.
