## Summary

This issue is a duplicate of #74809, which remains open and already tracks the same `proxy.enabled` plus local HTTP proxy failure, the same Discord TLS hostname error, and the same managed-proxy/global-agent root cause. The detailed 2026.4.29 logs and Telegram parity note are useful context, but they do not indicate a separate root cause from the canonical top-level network proxy bug.

## What This Changes

Fix Discord gateway WebSocket proxy handling so TLS certificate validation uses the original Discord gateway hostname when `proxy.enabled` routes traffic through an HTTP forward proxy.

## Best Possible Solution

Keep #74809 as the canonical open tracker and fold this issue's 2026.4.29 logs, Discord bot impact, and Telegram parity context into that single managed-proxy repair. The eventual fix should preserve proxy coverage and target-host TLS validation rather than creating a Discord-only bypass or disabling certificate checks.
