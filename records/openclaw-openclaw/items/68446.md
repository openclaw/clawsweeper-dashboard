## Summary

Keep open: this PR targets a real WhatsApp authorization boundary, but the branch is now conflicting/stale against current main, still leaves a command-authorization fallback path in its own head, and changes a fallback contract that current main documents and tests preserve.

## What This Changes

This PR changes WhatsApp inbound policy so group allowlists use only explicit `groupAllowFrom`, disables one effective-list fallback call, and adds regression tests for group allowlist behavior.

## Best Possible Solution

Decide and document the WhatsApp group sender fallback contract, then rebase onto the current shared ingress path and align inbound policy, command authorization, docs, and regression tests together.

## Reproduction Assessment

Yes, source-level. The PR head omits `groupAllowFromFallbackToAllowFrom: false` from command authorization while the helper falls back unless that flag is disabled; current main docs and tests also prove the fallback-contract conflict without running a live WhatsApp session.

## Solution Assessment

No. The branch disables fallback in one resolver but not command authorization, and it does not reconcile the documented/tested fallback behavior now present on current main.
