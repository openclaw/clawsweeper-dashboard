## Summary

Keep open: this PR targets a real WhatsApp authorization problem, but the branch is not mergeable as-is because the PR head leaves a command-authorization fallback path, conflicts with current main’s documented/tested fallback contract, and lacks after-fix real WhatsApp proof.

## What This Changes

This PR changes WhatsApp inbound policy so group sender allowlists use only explicit `groupAllowFrom` and adds regression tests for DM `allowFrom` not opening group allowlist handling.

## Best Possible Solution

Decide the WhatsApp group sender fallback contract, then rebase onto the current stable ingress path and align inbound policy, command authorization, docs, tests, and runtime proof together.

## Reproduction Assessment

Yes, source-level. The PR head omits `groupAllowFromFallbackToAllowFrom: false` from command authorization while the shared helper falls back unless that flag is disabled; current main docs and tests also show the contract conflict.

## Solution Assessment

No. The branch disables fallback in one resolver on stale code, leaves command authorization incomplete on the PR head, and does not reconcile the current documented/tested WhatsApp fallback behavior.
