## Summary

Keep this PR open: it targets a real WhatsApp group-authorization regression, but the submitted head is still incomplete for command authorization and lacks external real behavior proof before merge.

## What This Changes

The PR changes WhatsApp inbound policy so DM `allowFrom` is not resolved as `groupAllowFrom`, and adds inbound-policy regression tests for group allowlist behavior.

## Best Possible Solution

Rebase onto current main, keep WhatsApp group access driven by explicit `groupAllowFrom` or group entries rather than DM `allowFrom`, cover both auto-reply and command authorization, and align the WhatsApp docs with that contract.

## Reproduction Assessment

Yes. Source inspection of PR head shows the group command path still passes DM `allowFrom` into the shared command gate without disabling fallback, and the shared helper falls back from empty `groupAllowFrom` to `allowFrom` unless explicitly told not to.

## Solution Assessment

No. The conversation-policy change is in the right direction, but the submitted branch is incomplete and stale; the safer fix is a rebased patch that carries the strict group boundary through command authorization too.
