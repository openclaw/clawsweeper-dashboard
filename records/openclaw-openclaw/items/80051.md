## Summary

Keep open: the PR targets a real ACP timeout bug, but the current patch still forwards all closed-gate events through `onEvent`, which is user-visible in ACP dispatch, and the external proof is mock-only.

## What This Changes

The PR changes `consumeAcpTurnStream` to call `onEvent` and throw captured errors after `eventGate` closes, adds unit tests, and updates a UI i18n raw-copy baseline entry.

## Best Possible Solution

Preserve late error and terminal diagnostics after timeout through a filtered or non-visible path, with helper plus dispatch/projector coverage proving visible output stays suppressed.

## Reproduction Assessment

Yes. Source inspection shows current main closes `eventGate` in the timeout path while `consumeAcpTurnStream` keeps draining, and any later event hits the closed-gate `continue` before terminal tracking or error conversion; I did not run a live ACP timeout in this read-only review.

## Solution Assessment

No. Capturing late errors is the right direction, but forwarding every closed-gate event through `onEvent` is not the narrowest maintainable fix because ACP dispatch routes that callback to visible reply projection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter late events before invoking onEvent:** `src/acp/control-plane/manager.turn-stream.ts:32`
  - body: After timeout closes the gate, this calls `onEvent` for every ACP event before checking the type. ACP dispatch passes `onEvent` to the reply projector, which renders `text_delta`, `status`, and `tool_call`, so a late stream can emit visible output after the turn has already timed out; filter to late error/terminal diagnostics or use a non-visible path.
