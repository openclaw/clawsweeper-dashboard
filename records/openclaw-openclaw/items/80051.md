## Summary

Keep open: the bug is source-reproducible, but this PR still forwards all closed-gate events into the user-visible ACP reply path and lacks required real behavior proof from a real ACP setup.

## What This Changes

The PR updates ACP turn-stream consumption to forward events after the timeout gate closes, capture late error events, throw saved stream errors regardless of gate state, and add helper unit tests.

## Best Possible Solution

Update the ACP timeout contract so late errors and terminal signals remain observable through a non-visible or filtered path, keep late visible output suppressed, cover the helper plus dispatch/projector boundary, and add redacted real ACP proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: close `eventGate.open` during an ACP timeout, then have `runtime.runTurn()` yield an `error` or output event; current main skips it before delivery or error surfacing, and I did not execute tests because this review is read-only.

## Solution Assessment

No. Always throwing captured stream errors is aligned with the bug, but forwarding every late event through `onEvent` is not the narrowest maintainable fix because ACP dispatch wires `onEvent` to visible reply projection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not forward late output through onEvent:** `src/acp/control-plane/manager.turn-stream.ts:32`
  - body: After the timeout gate closes, `manager.core` has already rejected the turn and `dispatch-acp` passes `onEvent` into the visible reply projector. Calling `onEvent` for every closed-gate event here lets late `text_delta`, `status`, or `tool_call` events render after the timeout path; filter the late branch to errors/terminal diagnostics or another non-visible channel.
  - confidence: 0.9
