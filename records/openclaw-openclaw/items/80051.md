## Summary

Keep open: the underlying ACP timeout bug is source-reproducible, but this PR still forwards every closed-gate ACP event through the visible `onEvent` path and only provides mocked/unit proof, so it should not merge as-is.

## What This Changes

The PR changes `consumeAcpTurnStream` to call `onEvent` and collect `error` events after the timeout gate closes, and adds focused helper unit tests.

## Best Possible Solution

Define and implement a post-timeout stream contract that preserves late error and terminal diagnostics without sending visible reply output after the timeout, with helper and dispatch/projector coverage.

## Reproduction Assessment

Yes. Source inspection shows current main closes `eventGate` in the timeout path while `consumeAcpTurnStream` keeps draining, and any event yielded afterward hits the closed-gate `continue` before delivery or error conversion; I did not run a live ACP timeout in this read-only review.

## Solution Assessment

No. Capturing late errors is the right direction, but forwarding every closed-gate event through `onEvent` is not the narrowest maintainable fix because ACP dispatch uses that path for visible projector output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter late events before invoking onEvent:** `src/acp/control-plane/manager.turn-stream.ts:32`
  - body: After timeout closes the gate, this calls `onEvent` for every ACP event before checking the type. ACP dispatch passes `onEvent` to the reply projector, which renders `text_delta`, `status`, and `tool_call`, so a late stream can emit visible output after the turn has already timed out; filter to late error/terminal diagnostics or use a non-visible path.
