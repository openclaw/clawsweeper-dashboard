## Summary

Keep open: the PR targets a real current-main ACP timeout bug, but the patch is not merge-ready because it forwards all late events, including visible output, through `onEvent` after timeout and still lacks real behavior proof from an ACP setup.

## What This Changes

The PR changes ACP turn-stream handling so late post-timeout events still reach the non-output event observer and late errors are thrown, with new unit coverage.

## Best Possible Solution

Revise the PR so late timeout handling observes errors and terminal signals without forwarding late visible output to reply projectors, add boundary coverage for that behavior, and include redacted real ACP proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: close the ACP turn gate during timeout, then let the runtime iterator yield an `error` or event; current main skips it before delivery or error surfacing. I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR fixes late error visibility in the helper, but forwarding every late event through `onEvent` is not the narrowest maintainable solution because downstream ACP dispatch treats `onEvent` as visible reply delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not forward late output through onEvent:** `src/acp/control-plane/manager.turn-stream.ts:32`
  - body: After a timeout, `manager.core` has already rejected the turn and `dispatch-acp` passes `onEvent` into the reply projector. Forwarding every closed-gate event here means late `text_delta` or `tool_call` events can still be rendered to the user after the timeout/error path, which breaks the timeout boundary the PR says it preserves. Filter the late path to error/terminal events or a non-visible diagnostic channel instead.
  - confidence: 0.9
