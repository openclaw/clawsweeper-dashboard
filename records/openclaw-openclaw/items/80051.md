## Summary

Keep open: the underlying ACP timeout bug is source-reproducible, but this PR still forwards every late runtime event into `onEvent`, which is wired to visible ACP reply projection, and the external PR still lacks real after-fix ACP proof.

## What This Changes

The PR changes ACP turn-stream consumption to call `onEvent` after timeout gate closure, capture late error events, throw saved stream errors regardless of gate state, and add helper unit tests.

## Best Possible Solution

Define the ACP post-timeout stream contract, surface late errors or terminal diagnostics through a filtered non-visible path, keep late visible output suppressed, and cover both the helper and dispatch/projector boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: close `eventGate.open` during an ACP timeout, then have `runtime.runTurn()` yield an `error` or output-like event; current main skips it before delivery or error surfacing.

## Solution Assessment

No. Throwing captured stream errors is aligned with the bug, but forwarding every late event through `onEvent` is not the narrowest maintainable fix because ACP dispatch wires `onEvent` to visible reply projection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Filter late events before calling onEvent:** `src/acp/control-plane/manager.turn-stream.ts:32`
  - body: After the timeout gate closes, this branch calls `onEvent` before checking the event type. In ACP dispatch, `onEvent` is the reply projector, which renders text, status, and tool-call events, so late output can appear after the turn already timed out; filter this path to late error/terminal diagnostics or another non-visible channel.
