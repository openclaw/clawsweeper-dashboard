## Summary

Keep open: the PR addresses a real source-level cache-retention gap, but it currently widens the resolver past the documented non-Anthropic Bedrock guard and still lacks real backend payload proof.

## What This Changes

This PR passes explicit cacheRetention through the embedded-runner resolver, emits prompt_cache_retention="24h" for long OpenAI-compatible completions payloads, and adds resolver/transport tests plus changelog text.

## Best Possible Solution

Keep the transport-side long-retention support, but narrow the resolver so OpenAI-compatible completions passthrough does not bypass the existing Bedrock exclusion, then require a redacted real backend trace before merge.

## Reproduction Assessment

Yes at source level: current main returns undefined before reading explicit cacheRetention for non-Anthropic, non-Google openai-completions providers. I did not establish a live Docker/oMLX reproduction in this read-only review.

## Solution Assessment

No, not as written. The transport-side 24h mapping is plausible for the intended bug, but the resolver widening must preserve the established non-Anthropic Bedrock exclusion or use a narrower compatibility signal.
