## Summary

Keep open: this PR targets a real OpenAI-compatible completions cache-retention regression, but the current resolver change still bypasses a documented and tested non-Anthropic Bedrock exclusion, and the supplied proof is a resolver demo rather than a real backend payload trace.

## What This Changes

This PR passes explicit `cacheRetention` through the embedded-runner resolver for OpenAI-compatible completions providers, adds `prompt_cache_retention="24h"` to long-retention completions payloads, and adds tests plus changelog text.

## Best Possible Solution

Keep the transport-side long-retention payload support, but narrow the resolver so OpenAI-compatible completions passthrough does not override the existing non-Anthropic Bedrock exclusion, then require a redacted real backend trace before merge.

## Reproduction Assessment

Source-level yes: current main returns `undefined` before reading explicit `cacheRetention` for non-Anthropic, non-Google providers, so the embedded wrapper drops the value for OpenAI-compatible completions. I did not establish a live Docker/oMLX reproduction in this read-only review.

## Solution Assessment

No, not as written. The transport-side `24h` mapping is consistent with OpenAI's prompt-cache-retention contract, but the resolver widening must preserve the established non-Anthropic Bedrock exclusion or use a narrower compatibility signal. ([platform.openai.com](https://platform.openai.com/docs/guides/prompt-caching?utm_source=openai))
