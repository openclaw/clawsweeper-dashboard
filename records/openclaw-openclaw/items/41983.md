## Summary

Keep open: the PR targets a real contention problem between concurrent gateway sessions, and current main already mitigates the stale-bucket memory concern with pruning and a hard cap, but the proposed key change weakens the existing actor-scoped write throttle by giving every reconnect a fresh control-plane budget.

## What This Changes

The PR changes gateway control-plane write rate-limit keys to include `connId` for all clients and adds a test showing same-device/IP sessions get separate write budgets.

## Best Possible Solution

Preserve an actor-scoped write budget unless maintainers explicitly choose a layered design that adds per-connection fairness without allowing reconnects to bypass the device/IP throttle.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: under the PR key, two requests with the same device/IP but different `connId` values resolve to different buckets, and a reconnect can repeat that indefinitely within the same 60-second actor window.

## Solution Assessment

No. The PR solves same-device/IP session contention by replacing the actor key with the connection key; a safer solution would keep the device/IP cap and add any per-connection fairness as an additional layer after maintainer approval.
