## Summary

Keep open. Current main still uses aggregate `deviceId+clientIp` control-plane write budgets except for the no-identity fallback, while this PR intentionally changes a security-sensitive gateway throttling boundary to per-connection budgets and still needs maintainer policy acceptance plus docs/changelog/test cleanup before merge.

## What This Changes

The PR changes gateway control-plane write rate-limit keys to include `connId` whenever present and adds a regression test showing a second concurrent session gets its own `config.patch` budget.

## Best Possible Solution

Decide the gateway security policy first. If maintainers want independent budgets per active connection, update this PR or a replacement to keep the current pruning/cap safeguards, document the new scope, add the required changelog entry, rename the misleading test, and rerun the focused gateway checks plus the changed gate.

## Reproduction Assessment

Yes: source and test inspection provide a high-confidence reproduction path for the behavioral change. Current main expects a known-IP client with `connId` to share `unknown-device|10.0.0.10`, while the PR changes that key to include `|conn=conn-fallback`; tests were not run because this was a read-only review.

## Solution Assessment

Unclear: appending `connId` is the narrow implementation if per-session budgets are desired, but it is not clearly the best policy because it weakens aggregate write throttling. The safer path is explicit maintainer acceptance before aligning docs, changelog, and tests.
