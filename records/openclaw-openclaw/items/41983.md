## Summary

Keep this PR open. Current main still enforces the documented aggregate control-plane write budget, so this remains an unmerged gateway policy/security change rather than cleanup-close material.

## What This Changes

The PR changes control-plane rate-limit keys to include `connId` whenever present and adds a concurrent-session budget isolation regression test.

## Best Possible Solution

Keep the PR open for gateway policy review. If maintainers want per-connection budgets, rebase it onto current main, preserve pruning/cap safeguards, update docs and changelog, and document the security impact; otherwise close it manually with the aggregate-throttling rationale.

## Reproduction Assessment

Yes: source and test inspection provide a high-confidence reproduction path for the behavioral difference, because current main expects `connId` to be ignored when `clientIp` is present. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear: appending `connId` is a small maintainable implementation only if maintainers accept independent write budgets per concurrent session. The safer path is an explicit gateway policy/security decision before merging or replacing this PR.
