## Summary

Keep open for maintainer review, but this patch is not mergeable as-is: current main and the public Gateway docs define the control-plane write limit per device/IP actor, while the PR makes that privileged write budget renewable per connection and has no real behavior proof.

## What This Changes

The PR changes gateway control-plane write rate-limit keys to include `connId` whenever present and adds a test expecting a second concurrent session to receive a separate write budget.

## Best Possible Solution

Keep the actor-scoped `deviceId+clientIp` throttle as the baseline, and add any per-connection fairness as a documented additional layer only after maintainer security/product approval.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: after exhausting a write budget under one `connId`, changing `connId` creates a distinct key and a fresh bucket. No current-main bug is reproduced because current main intentionally keeps the device/IP key when identity is present.

## Solution Assessment

No. Replacing the actor key with a connection key is not the narrowest maintainable solution; preserve the existing actor-level guard and layer connection fairness separately only if maintainers choose that policy.
