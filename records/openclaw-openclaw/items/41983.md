## Summary

This PR should stay open for maintainer judgment rather than cleanup close: current main and the public docs keep control-plane write limits actor-scoped by `deviceId+clientIp`, while the branch changes that policy to grant fresh privileged write budgets per connection and still has no real behavior proof.

## What This Changes

The PR changes gateway control-plane write throttling to include `connId` in the rate-limit key and adds coverage for separate write budgets across concurrent sessions.

## Best Possible Solution

Keep `deviceId+clientIp` as the baseline privileged-write throttle; add any per-connection fairness only as a separate documented layer after explicit product/security approval and real gateway proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: changing `connId` changes the rate-limit key and gives a fresh bucket while the actor identity is unchanged. No current-main bug was reproduced because current main intentionally keeps actor-scoped keys when identity is present.

## Solution Assessment

No. Replacing the actor key with a connection key is not the narrowest maintainable solution; preserve the existing actor-level guard and layer connection fairness separately only if maintainers choose that policy.
