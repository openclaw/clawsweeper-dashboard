## Summary

Keep open. Current main does not implement this PR's intended per-connection isolation, and the submitted branch should not merge as-is because it changes the documented rate-limit boundary and predates the main-branch bucket cleanup/cap that prevents unbounded state growth.

## What This Changes

The PR changes the gateway control-plane write rate-limit key to append `connId` whenever available and adds a test that concurrent sessions do not share the write budget.

## Best Possible Solution

Decide the intended control-plane rate-limit actor explicitly; if per-connection isolation is accepted, land it on top of main's pruning/cap with docs, changelog, and tests for both isolation and cleanup.

## Reproduction Assessment

Yes, at source level. The PR key appends `connId` for each connection while the submitted branch has no close-hook or periodic prune for stale connection buckets; current main and docs show the existing per-device/IP behavior.

## Solution Assessment

No. The branch changes a documented rate-limit boundary without docs or changelog and needs to preserve main's bucket cleanup before this can be considered a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound buckets before keying by connection:** `src/gateway/control-plane-rate-limit.ts:27-31`
  - body: This changes every connected client from one device/IP bucket into a unique `connId` bucket, but the PR head still stores buckets in a module-level map with no eviction on close or periodic pruning. Reconnect churn leaves old connection buckets unreachable, so gateway memory can grow without bound.
  - confidence: 0.9
- **[P3] Align docs with the new rate-limit actor:** `src/gateway/control-plane-rate-limit.ts:31`
  - body: The gateway docs currently promise control-plane writes are limited per `deviceId+clientIp`, but this line makes the effective limit per connection whenever `connId` exists. Operators will see a different throttle boundary than the documented one.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff changes a control-plane rate-limit boundary and introduces concrete resource-exhaustion concerns on the submitted branch.

Concerns:

- **[medium] Unbounded per-connection rate-limit buckets:** `src/gateway/control-plane-rate-limit.ts:27`
  - body: Keying by `connId` on the PR head creates a fresh bucket for each reconnect, but stale buckets are not removed from the map, so repeated reconnects can grow gateway memory indefinitely.
  - confidence: 0.9
- **[low] Rate-limit budget can be multiplied by connections:** `src/gateway/control-plane-rate-limit.ts:31`
  - body: Moving the default actor from device/IP to connection lets the same authorized device gain additional `config.apply`, `config.patch`, and `update.run` attempts by opening more concurrent sessions, which needs an explicit maintainer policy decision.
  - confidence: 0.78
