## What This Changes

The PR changes gateway control-plane write rate-limit keys to include `connId` whenever present and adds a test for separate budgets across concurrent gateway sessions.

## Best Possible Solution

Preserve an aggregate device/IP or server-derived abuse boundary while adding any desired connection isolation deliberately, with updated docs, focused tests, and redacted real gateway proof before merge.

## Reproduction Assessment

Yes for the behavior change: source inspection shows current main shares the control-plane write bucket by `deviceId+clientIp`, while the PR would split that bucket by `connId`. It is not a proven bug under current docs because the docs describe the shared bucket as the existing contract.

## Solution Assessment

No, not as-is. The narrow implementation demonstrates the intended isolation, but the safer solution needs an explicit maintainer boundary decision, aggregate abuse protection or updated policy, passing typecheck, and real behavior proof.
