## Summary

Keep open: the PR changes a documented gateway security boundary from an actor-scoped `deviceId+clientIp` write budget to a per-connection budget, which lets the same actor renew the privileged write throttle by reconnecting. Current main still preserves `connId` only as a last-resort fallback when both device and IP are unknown, and the prior stale-bucket memory concern is already handled on main.

## What This Changes

The PR changes gateway control-plane write rate-limit keys to include `connId` whenever present and adds a test showing separate write budgets for concurrent sessions.

## Best Possible Solution

Keep the documented actor-scoped write budget, and add per-connection fairness only as a second layer if maintainers explicitly want that gateway policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: under the PR key, repeated `config.patch` calls from the same device/IP can get a fresh budget after changing `connId` by reconnecting.

## Solution Assessment

No. Replacing the actor key with a connection key is not the narrowest maintainable solution; preserving the `deviceId+clientIp` guard and optionally adding a second per-connection limiter would be safer.
