## Summary

Keep open: the PR targets a source-visible same-device/IP contention case, but the diff changes a sensitive control-plane write limiter from actor-scoped to connection-scoped without an aggregate cap or explicit maintainer policy. Current main has already addressed the Greptile memory-growth concern with pruning and a hard bucket cap, so the remaining blocker is rate-limit semantics plus small PR hygiene.

## What This Changes

The PR changes the gateway control-plane write rate-limit key to include `connId` for connected clients and adds a test showing separate `config.patch` budgets for two concurrent sessions.

## Best Possible Solution

If maintainers want connection isolation, layer it under an explicit aggregate device/IP guard or document the weaker per-connection policy, then update the test name and changelog.

## Reproduction Assessment

Yes, by source inspection. Current main returns the same rate-limit key for clients that share device/IP even when their `connId`s differ, so two concurrent sessions share the three-write-per-minute bucket; no tests were run because this was a read-only sweep.

## Solution Assessment

No. The diff is the smallest code change for per-connection isolation, but it removes the established actor-level budget for sensitive writes; the safer maintainable path is an explicit layered limit or maintainer-approved policy change.
