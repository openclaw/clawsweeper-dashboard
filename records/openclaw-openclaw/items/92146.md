## Summary

Keep open: this is a focused fix for a current P1 isolated-cron media/yield regression, and current main does not contain the yield-aware wait gate. The patch looks reasonable, but the external PR proof is still source-test/terminal proof rather than a full isolated cron callback-continuation run.

## What This Changes

The PR adds a yield-aware completion-required media wait helper, wires it into embedded attempt finalization, and adds regression coverage for preserving non-yield cron media waits while skipping the post-yield wait.

## Best Possible Solution

Land a yield-aware wait gate only with focused proof that yielded isolated cron media runs resume through the completion event while ordinary non-yield cron media runs still wait synchronously.

## Reproduction Assessment

Yes at source level: current main aborts the run controller with reason `sessions_yield` and then can enter the completion-required media waiter with that already-aborted signal. I did not execute a full live cron/provider reproduction in this read-only review.

## Solution Assessment

Likely yes as a narrow owner-boundary fix: the wait decision is exactly where the aborted signal is reused, and the patch preserves normal non-yield waits. The remaining gap is proof that the completion callback drives the next isolated cron step end to end.
