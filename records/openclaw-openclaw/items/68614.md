## Summary

Keep open. The underlying Google Chat delivery gap still appears valid on current main, but this PR is not landable as-is because it is dirty against the current split delivery/turn-kernel code and leaves a new timeout armed when dispatch exits without a delivery callback.

## What This Changes

The PR changes Google Chat reply handling to start a 25-second fallback timer after creating a typing placeholder, post a fallback message on timeout, delete the placeholder, and send the eventual agent reply as a new message.

## Best Possible Solution

Port the useful timeout behavior to current main's split Google Chat code: preserve the existing typingIndicator default and reaction fallback, start a bounded timer only after a typing message is created, pass an expired-deferred flag into delivery, reuse the extracted send/update fallback logic, and always clear the timer when the turn dispatch exits without delivery.

## Reproduction Assessment

Yes. A focused unit test can fake timers, create a message-mode typing placeholder, make dispatch return without invoking `deliver`, advance 25 seconds, and observe the PR's spurious fallback; current main also has no timer path for long-running delivery before the first reply callback.

## Solution Assessment

No. The PR is directionally useful, but it targets the old monolithic monitor shape and clears the timer only from `onDeliver`; a safer fix belongs in `extensions/googlechat/src/monitor.ts` plus `extensions/googlechat/src/monitor-reply-delivery.ts` with `try/finally` cleanup around dispatch completion.

## Review Findings

Overall correctness: patch is incorrect

In those cases the 25-second callback still posts the fallback and deletes the typing message after the turn has already ended, so wrap the dispatch await in `try/finally` and clear the timer there.
