## Summary

Keep this issue open. Current main and the latest tagged release still implement a phase-only assistant visible-text contract; public/internal audience metadata is neither parsed nor enforced at the shared extractor, OpenResponses schema, UI projection, or channel delivery gate.

## What This Changes

The issue requests an explicit public/internal assistant audience contract at shared visible-text and delivery boundaries so internal assistant text is not selected for user-visible delivery.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and a deliberate legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` like `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current final-answer path because `audience` is ignored by the parser and extractor.

## Solution Assessment

Yes. The report's shared-contract direction is the narrow maintainable fix shape; a channel-only cleanup would not consistently protect shared visible extraction, replay, OpenResponses, history/UI projection, and message-end delivery.
