## Summary

Keep this open. Current main and the latest v2026.4.29 release still expose a phase-only assistant visibility contract: audience metadata is not parsed, represented in OpenResponses, enforced by the shared visible extractor, or checked at channel delivery. The related issues cover adjacent phase leaks and delivery selection problems, but this issue's public/internal audience boundary remains distinct and unresolved.

## What This Changes

The issue requests an explicit public/internal assistant audience contract for visible text extraction and channel delivery, including fail-closed handling for invalid audiences and suppression of exact silent sentinels.

## Best Possible Solution

Define a versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider stream parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` containing `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current final-answer visible path because the parser and extractor ignore `audience`.

## Solution Assessment

Yes. The shared-contract direction is the maintainable fix shape; solving this only in one channel delivery path would not protect shared extraction, replay, OpenResponses, history/UI projection, and message-end delivery consistently.
