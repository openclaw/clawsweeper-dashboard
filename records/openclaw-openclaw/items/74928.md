## Summary

Keep open. Current main and the latest shipped v2026.4.29 code still expose a phase-only assistant text contract at the shared parser, visible extractor, OpenResponses schema, UI projection, and channel delivery boundary; the reporter's package-level audience smoke remains consistent with current source.

## What This Changes

Request an explicit assistant audience contract so public/internal assistant text is encoded, parsed, propagated, and enforced before user-visible delivery.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider stream parsing, replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed, exact silent-token suppression, and an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature: {"v":1,"id":"x","phase":"final_answer","audience":"internal"}` follows the current parser/extractor path because `audience` is ignored and `final_answer` text is selected as visible.

## Solution Assessment

Yes, the shared contract direction in the report is the maintainable fix shape. A channel-only sanitizer would not consistently protect replay, OpenResponses, history/UI projection, and shared visible-text extraction.
