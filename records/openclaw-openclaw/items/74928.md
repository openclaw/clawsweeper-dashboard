## Summary

Keep this open. Current main still implements assistant visibility with phase-only metadata, so the reported public/internal audience boundary is still missing and the related issues only cover adjacent leakage or selection symptoms.

## What This Changes

The issue requests a shared assistant audience contract that marks assistant text as public or internal and enforces that boundary at visible extraction and channel delivery.

## Best Possible Solution

Define a versioned `public | internal` assistant audience contract in shared message content, preserve it through provider parsing/replay/history/UI projection, and enforce public-only selection at delivery with an explicit legacy phase fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence synthetic reproduction: a signed assistant text block with `phase: "final_answer"` and `audience: "internal"` is parsed as final-answer text because the current parser discards audience metadata.

## Solution Assessment

Yes for the requested direction. A shared audience contract is the narrow maintainable fix; patching only one channel send path would not protect the shared extractor, replay, history, and UI surfaces.
