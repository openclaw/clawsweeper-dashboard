## Summary

Keep this open. Current main still has a phase-only assistant visibility contract, so `audience: "internal"` on a synthetic `final_answer` text signature is ignored by shared extraction and downstream delivery paths.

## What This Changes

The issue requests a versioned public/internal assistant audience contract across signature parsing, provider replay, visible-text extraction, OpenResponses/history projection, and channel delivery.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract in shared message content, propagate it through provider parsing/replay and OpenResponses/history/UI projection, then enforce public-only visible selection at delivery with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` is parsed as final-answer only, so `extractAssistantVisibleText` returns the text on current main.

## Solution Assessment

Yes. The report's shared-contract direction is the maintainable fix; solving only one channel delivery path would leave the shared extractor, provider replay, OpenResponses, and UI/history surfaces without the boundary.
