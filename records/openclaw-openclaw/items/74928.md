## Summary

Keep this open. Current main still has a phase-only assistant visibility contract: `audience` is not parsed, encoded, schema-valid, or checked before shared visible extraction and channel delivery.

## What This Changes

Add an explicit public/internal assistant audience boundary for visible assistant text and delivery selection, while preserving legacy phase-only compatibility where appropriate.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract in shared message content, propagate it through provider parsing/replay and OpenResponses/history/UI projection, then enforce public-only selection at visible delivery boundaries with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current final-answer path because the parser drops `audience`.

## Solution Assessment

Yes for the requested direction. A shared audience contract is the narrow maintainable solution; channel-local filtering would not protect shared extractors, replay, history, or UI surfaces consistently.
