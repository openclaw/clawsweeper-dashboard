## Summary

Keep this open. Current main and the latest release still implement a phase-only assistant visibility contract; public/internal audience metadata is not parsed, propagated, or enforced at the shared extractor, OpenResponses schema, UI projection, or channel delivery boundary.

## What This Changes

The issue requests an explicit `public`/`internal` assistant audience contract for visible-text extraction and delivery, including fail-closed handling for invalid audiences and suppression of exact silent sentinels.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` like `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` still follows the current final-answer visible path because `audience` is ignored by the parser and extractor.

## Solution Assessment

Yes. The report's shared-contract direction is the maintainable fix shape; solving this only in one channel delivery path would not protect shared visible extraction, replay, OpenResponses, history/UI projection, and message-end delivery consistently.
