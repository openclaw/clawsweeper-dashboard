## Summary

Keep this issue open. Current main still implements a phase-only assistant visible-text boundary: `public`/`internal` audience metadata is not parsed, exposed, or enforced in the shared extractor, OpenResponses schema, Control UI projection, or embedded channel delivery path, and the latest tagged release shows the same contract.

## What This Changes

The issue requests an explicit public/internal assistant audience contract for visible delivery so internal assistant text and exact silent sentinels cannot be selected as user-visible text.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and a deliberate legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` containing `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current final-answer visible path because the parser and extractor ignore `audience`.

## Solution Assessment

Yes. The report’s shared-contract direction is the narrow maintainable fix shape; a channel-only sanitizer would leave replay, OpenResponses, history/UI projection, and shared extraction with inconsistent semantics.
