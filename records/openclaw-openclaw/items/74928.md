## Summary

Keep open. The reporter's sanitized package evidence and the prior ClawSweeper comment still match current main: assistant text metadata is phase-only, shared visible extraction ignores any public/internal audience field, and delivery suppression only gates commentary-phase output. This remains a security-sensitive cross-surface contract decision rather than a narrow cleanup close.

## What This Changes

Request an explicit public/internal assistant audience contract for assistant text signatures, visible-text extraction, replay/history, OpenResponses, UI, and channel delivery, with fail-closed handling for invalid audience values and legacy phase fallback where needed.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate it through provider stream parsing, transcript/history replay, OpenResponses, Control UI, and channel delivery, and enforce public-only visible selection with invalid or unknown audience values failing closed while preserving deliberate legacy phase-only compatibility.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` JSON containing `v: 1`, an id, `phase: "final_answer"`, and `audience: "internal"` follows the current source path because audience is ignored and final-answer text is selected as visible.

## Solution Assessment

Yes, the requested shared contract direction is the maintainable solution. A delivery-only sanitizer would leave replay, history, OpenResponses, and UI extraction with inconsistent semantics.
