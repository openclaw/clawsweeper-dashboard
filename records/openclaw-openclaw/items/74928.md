## Summary

Keep this open. Current main still has a phase-only assistant visibility contract, so a signed `final_answer` block with `audience: "internal"` is still treated as visible by the shared extractor and delivery code.

## What This Changes

Add an explicit public/internal assistant audience boundary for visible-text extraction, provider replay, transcript projection, and channel delivery.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract in shared message content, propagate it through provider parsing/replay and OpenResponses/history/UI projection, then enforce public-only visible selection at delivery with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current `final_answer` path because the parser drops `audience`, so source inspection gives a high-confidence reproduction path.

## Solution Assessment

Yes for the direction. A shared contract is the narrow maintainable solution; patching only one channel send path would not protect the shared extractor, replay, history, or UI surfaces.
