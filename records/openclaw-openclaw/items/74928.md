## Summary

Keep this open. Current main still implements assistant visibility as a phase-only contract; `audience: "public" | "internal"` is not parsed, propagated, exposed in OpenResponses, or enforced at the shared visible-text and channel delivery boundary.

## What This Changes

The issue requests an explicit public/internal assistant audience contract for visible assistant text selection and delivery.

## Best Possible Solution

Define one versioned assistant audience contract in shared message content, propagate it through provider parsing/replay, OpenResponses, history/UI projection, and channel delivery, then enforce public-only visible selection with a narrow legacy fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` containing `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current final-answer visible path because the parser and extractor ignore `audience`.

## Solution Assessment

Yes. The report's shared-contract direction is the maintainable fix shape; solving only the embedded channel delivery path would not protect the shared extractor, replay, OpenResponses, or UI/history surfaces.
