## Summary

Keep this open. Current main still has a phase-only assistant visibility contract: `audience` is neither parsed nor enforced at the shared visible-text extractor, and downstream embedded delivery suppresses commentary only before using that extractor.

## What This Changes

The issue requests an explicit public/internal assistant audience contract so internal assistant text is never selected for user-visible channel delivery.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract in shared message content and OpenResponses schema, propagate it through provider parsing/replay and UI/history projection, then enforce public-only visible selection at delivery with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` is parsed as final-answer only, so `extractAssistantVisibleText` still returns the text on current main.

## Solution Assessment

Yes. The report's shared-contract direction is the maintainable fix; solving only one channel scrubber would leave shared extraction, provider replay, OpenResponses, and UI/history surfaces without the audience boundary.
