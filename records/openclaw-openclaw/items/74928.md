## Summary

Keep this open. Current main still implements assistant visibility as a phase-only contract: signatures, OpenResponses schemas, provider replay, shared extraction, UI extraction, and embedded delivery do not parse or enforce a `public`/`internal` assistant audience boundary, and the related open issues do not supersede this specific contract request.

## What This Changes

The issue requests a shared public/internal assistant audience contract for visible assistant-text extraction and channel delivery.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract, propagate it through signatures/OpenResponses/replay/history/UI, enforce public-only selection at visible delivery boundaries, and keep legacy phase-only fallback narrowly scoped for old transcripts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` has its audience ignored and remains eligible for final-answer extraction.

## Solution Assessment

Yes for the direction, but it needs a maintainer-owned shared contract rather than per-channel filtering. The narrow maintainable solution is a versioned audience field enforced at shared visible extraction and delivery boundaries.
