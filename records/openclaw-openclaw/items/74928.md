## Summary

Keep this open. Current main and the latest release still expose a phase-only assistant visibility contract; public/internal audience metadata is not parsed, propagated, or enforced at the shared visible extraction, OpenResponses, UI projection, or channel delivery boundary.

## What This Changes

The issue requests an explicit public/internal assistant audience contract for assistant text signatures and visible delivery selection, with internal text failing closed and exact silent sentinels suppressed.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and an explicit legacy phase-only compatibility path.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` such as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` still follows the current final-answer visible path because `audience` is ignored by the parser and extractor.

## Solution Assessment

Yes, the report's shared-contract direction is the right shape. A one-channel sanitizer would not protect shared visible extraction, replay, OpenResponses, history/UI projection, and message-end delivery consistently.
