## Summary

Keep this open. Current main and the latest shipped v2026.5.2 tag are still phase-only at the shared signature, visible-extraction, OpenResponses, provider transport, and embedded delivery surfaces, so the requested public/internal assistant audience boundary remains unresolved.

## What This Changes

Request an explicit public/internal assistant audience contract through assistant text signatures, provider/replay schemas, shared visible-text helpers, and channel delivery selection.

## Best Possible Solution

Define one versioned public/internal assistant audience contract in shared message content and gateway/provider schemas, propagate it through replay/history/UI surfaces, and enforce public-only selection at visible delivery boundaries with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a text block signed with phase final_answer and audience internal has its audience ignored by parseAssistantTextSignature and remains eligible for extractAssistantVisibleText.

## Solution Assessment

Yes for the requested direction, pending maintainer-owned semantics. A shared versioned audience gate is the maintainable boundary; per-channel filtering would leave provider, replay, history, UI, and shared extraction paths inconsistent.
