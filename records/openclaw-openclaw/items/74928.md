## Summary

Keep open. Current main still has a phase-only assistant text contract and no public/internal assistant audience gate at shared visible-text extraction, OpenResponses schema, Control UI extraction, or message-end delivery; the reporter's sanitized package-artifact evidence remains consistent with the current source.

## What This Changes

Add an explicit public/internal assistant audience contract for assistant text signatures and enforce it at visible extraction and delivery boundaries while preserving legacy phase fallback where needed.

## Best Possible Solution

Define a versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider stream parsing, replay, transcript/history projection, OpenResponses, UI extraction, and channel delivery, then enforce public-only visible selection with explicit invalid-audience fail-closed behavior and a narrow legacy phase-only compatibility path.

## Reproduction Assessment

Yes. A synthetic assistant message with a text block signed as `{ v: 1, id: "x", phase: "final_answer", audience: "internal" }` follows the current parser/extractor path because `audience` is ignored and `final_answer` text is selected as visible.

## Solution Assessment

Yes. The requested shared contract is the maintainable direction; a delivery-only scrubber would leave replay, history projection, OpenResponses, and Control UI with inconsistent semantics.
