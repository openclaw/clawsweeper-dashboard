## Summary

Keep open. Current main and the latest shipped v2026.4.29 code still have a phase-only assistant text contract at the shared visible-text, OpenResponses, UI extraction, and channel message-end delivery boundaries; the reporter's sanitized package-artifact evidence remains consistent with current source.

## What This Changes

The issue requests an explicit public/internal assistant audience contract for visible assistant delivery, with invalid audiences failing closed and exact silent sentinels suppressed at visible extraction/delivery boundaries.

## Best Possible Solution

Define one versioned assistant audience contract in the shared message-content layer, propagate `public` and `internal` through provider parsing/replay, OpenResponses, transcript/history projection, UI extraction, and channel delivery, then enforce public-only visible selection with invalid audiences failing closed and a deliberate legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant message with a text block signature like `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` follows the current parser/extractor path because `audience` is ignored and `final_answer` text is selected as visible.

## Solution Assessment

Yes, the report's shared contract direction is the maintainable fix shape. A delivery-only sanitizer would not consistently protect replay, OpenResponses, history/UI projection, and shared visible-text extraction.
