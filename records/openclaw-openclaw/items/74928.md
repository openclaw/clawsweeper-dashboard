## Summary

Keep this open. Current main still implements a phase-only visibility contract; `audience` is neither parsed nor enforced before shared visible-text extraction, embedded channel delivery, OpenResponses replay/schema, or UI projection.

## What This Changes

The issue requests an explicit `public | internal` assistant audience contract for assistant text signatures and user-visible delivery selection.

## Best Possible Solution

Define one versioned assistant audience contract in shared message content, propagate it through provider stream parsing, replay, history, gateway schema, and UI projection, then enforce public-only selection at delivery boundaries with a narrow legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` JSON containing `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` still follows the final-answer path because the current parser drops `audience` and extraction checks only phase.

## Solution Assessment

Yes, the requested shared audience contract is the right direction. A channel-local filter would leave shared extraction, replay, history, and UI projection with inconsistent internal/public semantics.
