## Summary

Keep open. Current main has fixed or documented adjacent parts of this DeepSeek/model-switching path, including DeepSeek onboarding/catalog docs, models.json merge behavior, default-model allowlist upserts, and unmanaged gateway restart handling. The core UX report is still not fully resolved because the agent-facing gateway config tool still blocks models.providers writes, there is no first-class models add provider command, configured model views intentionally preserve agents.defaults.models allowlist semantics, and existing sessions still need an explicit session model switch or reset rather than silently adopting a new default.

## What This Changes

The issue asks for a smoother provider/model switch flow when moving from qwen3.6-plus to deepseek-v4-pro, covering provider config mutation, global versus agent model config sync, allowlist diagnostics, gateway restart, and active-session model adoption.

## Best Possible Solution

Keep the already-landed DeepSeek/default-model/restart improvements, then split the remaining UX into maintainer-owned changes: a safe custom-provider setup flow, status or doctor diagnostics for provider/models.json/allowlist mismatches, and a clear explicit current-session switch or reset path instead of silent active-session remapping.

## Reproduction Assessment

Unclear. The issue provides concrete models, version, OS, setup, and symptoms, and current source confirms several remaining static behaviors; a high-confidence end-to-end reproduction still needs the exact patch payload and the reporter's global plus agent model config state.

## Solution Assessment

Unclear. The requested automatic all-provider allowlist expansion and active-session remapping may conflict with intentional allowlists and session stability; a narrower CLI/status/doctor design with explicit session switching looks safer.
