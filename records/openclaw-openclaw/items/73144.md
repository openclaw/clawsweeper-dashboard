## Summary

Keep open. Current main has improved adjacent pieces of the DeepSeek/model-switching path, but it still does not resolve the full UX bundle: the agent-facing config patch guard still blocks provider config writes, there is no generic provider-add CLI, configured model views still preserve allowlist semantics, and active-session changes still need explicit session handling rather than silent remapping on reload.

## What This Changes

The issue requests a smoother model-switching/setup flow for moving from `bailian/qwen3.6-plus` to `deepseek/deepseek-v4-pro`, covering provider config writes, global/agent model config consistency, allowlist warnings, gateway restart behavior, and active-session model refresh.

## Best Possible Solution

Keep the already-landed DeepSeek/default-model/restart improvements, then split the remaining UX into maintainer-approved pieces: a safe custom-provider setup path, doctor/status diagnostics for provider/models.json/allowlist mismatches, and an explicit current-session switch or reset flow instead of silent active-session remapping.

## Reproduction Assessment

Unclear. The report gives enough environment and model detail for a likely reproduction, and current-main static inspection confirms several remaining behaviors, but high-confidence end-to-end reproduction still needs the exact patch payload plus the reporter's global and agent model config state.

## Solution Assessment

Unclear. The requested automatic provider allowlist expansion and active-session remapping may not be the safest design; a narrower CLI/status/doctor workflow with explicit session switching or reset would better preserve existing trust and session semantics.
