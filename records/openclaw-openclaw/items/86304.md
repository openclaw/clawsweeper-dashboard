## Summary

Keep open: current main still trusts the Dashboard key's stored sessionId for list rows and chat.history, and I did not find an implemented repair that reconstructs a wrong canonical Dashboard mapping from orphaned transcript or trajectory evidence.

## What This Changes

The issue reports an upgrade regression where Dashboard history entries load the wrong conversation because sessions.json maps a Dashboard key to the wrong sessionId while the original transcript/trajectory remains on disk.

## Best Possible Solution

Keep this tracked until maintainers choose a diagnostic or doctor repair that validates corrupted Dashboard entries against store/transcript/trajectory evidence, with redacted fixtures and regression coverage.

## Reproduction Assessment

No high-confidence end-to-end upgrade reproduction is available without a redacted fixture. Source inspection does show that if sessions.json maps a Dashboard key to the wrong sessionId, current main will list and load history through that wrong mapping.

## Solution Assessment

Unclear. Repairing the stored mapping may be the right outcome, but the durable rule for choosing the correct orphan transcript or trajectory file needs maintainer review before changing user history.
