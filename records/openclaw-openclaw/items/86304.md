## Summary

Keep open: current main still uses the stored Dashboard entry sessionId for list rows, previews, and chat.history, and the current migration only canonicalizes session keys rather than repairing an already-canonical Dashboard key that points at the wrong transcript.

## What This Changes

The issue reports that after upgrade, a Dashboard history key can map to the wrong sessionId in sessions.json while the original trajectory file remains on disk.

## Best Possible Solution

Keep this issue open until maintainers choose a doctor or diagnostic repair that can validate corrupted Dashboard entries against store, transcript, and trajectory evidence without adding a steady-state runtime fallback.

## Reproduction Assessment

Yes at source level: if sessions.json maps a Dashboard key to the wrong sessionId, current main will list and load history through that wrong mapping. No high-confidence live upgrade reproduction is available without a redacted sessions.json plus transcript or trajectory fixture.

## Solution Assessment

Unclear: repairing the stored mapping is likely the right outcome, but the durable rule for selecting the correct orphaned transcript or trajectory file needs maintainer review before changing user history.
