## Summary

Keep open: current main and the latest release still trust the stored Dashboard session row, so a wrong `sessions.json` mapping can still list and load the wrong preserved history. The fix needs maintainer review because automatic relinking could attach a Dashboard key to the wrong transcript or trajectory file.

## What This Changes

The issue reports a post-upgrade Dashboard history mapping drift where a Dashboard key points at the wrong `sessionId` even though the expected trajectory file remains on disk.

## Best Possible Solution

Add a maintainer-approved diagnostic or doctor repair that validates corrupted Dashboard mappings against store metadata, transcript headers, trajectory evidence, and redacted fixtures, and refuses ambiguous relinks.

## Reproduction Assessment

No high-confidence end-to-end upgrade reproduction is available without a redacted fixture. Source inspection does show that if `sessions.json` maps a Dashboard key to the wrong `sessionId`, current main lists and loads history through that wrong mapping.

## Solution Assessment

Unclear. Repairing the stored mapping is likely needed, but the durable rule for choosing the correct orphan transcript or trajectory file requires maintainer review before changing user history.
