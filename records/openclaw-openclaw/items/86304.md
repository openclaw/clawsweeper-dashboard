## Summary

Keep open: current main and the latest shipped release still drive Dashboard list/history from the stored SessionEntry.sessionId, so a corrupted Dashboard key mapping can still load the wrong preserved history. The repair rule needs maintainer judgment because choosing a replacement orphan transcript or trajectory file can mis-associate user history.

## What This Changes

The issue reports an upgrade regression where a Dashboard session key in sessions.json points to the wrong sessionId, causing preserved historical conversation data to load through the wrong mapping.

## Best Possible Solution

Add a maintainer-approved diagnostic or doctor repair that validates corrupted Dashboard mappings against store, transcript header, and trajectory evidence with redacted fixtures, and refuses ambiguous relinks.

## Reproduction Assessment

No high-confidence end-to-end upgrade reproduction is available without a redacted fixture. Source inspection does show that if sessions.json maps a Dashboard key to the wrong sessionId, current main lists and loads history through that wrong mapping.

## Solution Assessment

Unclear. Repairing the stored mapping may be right, but the durable rule for choosing the correct orphan transcript or trajectory file needs maintainer review before changing user history.
