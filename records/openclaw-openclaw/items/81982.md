## Summary

Current main and the latest release already preserve configured model fallback chains for override/current-fallback model selections, which matches the reported `next=none` failure on older 2026.5.7 logs.

## What This Changes

The issue reports that a deferred config restart can leave an embedded run with an empty model fallback chain, so a 429 surfaces instead of trying configured fallback models.

## Best Possible Solution

Keep the shipped fallback-chain handling in v2026.5.12 and current main; any remaining report should be a fresh issue against v2026.5.12 or newer with new `model-fallback/decision` logs.

## Reproduction Assessment

No: I did not reproduce the interruption on current main. The 2026.5.7 logs line up with the older override-model fallback-chain bug, and current source plus regression tests show the chain is preserved now.

## Solution Assessment

Yes: the current fix is the narrow maintainable path because it preserves the configured candidate list in the model fallback resolver instead of adding restart-specific special cases.
