## Summary

Keep open. The PR targets a real, source-reproducible gateway auth problem, but the branch is stale against current main's separate scope-clearing helper, lacks after-fix real gateway proof, and changes a WebSocket authorization boundary that needs maintainer/security approval.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so successful token/password operator auth can return `allow`, and adds one token-path regression assertion.

## Best Possible Solution

Define one explicit WebSocket device-less token/password scope policy, then land a current-main implementation that aligns connect policy, scope clearing, auth tests, protocol/security docs, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level: current main tests connect a valid shared-token/password operator without device identity and assert that requested scopes are cleared, making operator.write-scoped RPCs fail. I did not run a live remote WebSocket gateway in this read-only review.

## Solution Assessment

No. The PR touches the right policy surface, but a rebased fix must also align current main's scope-clearing helper, compatibility tests, security/protocol docs, and real gateway proof before this auth-boundary change is mergeable.
