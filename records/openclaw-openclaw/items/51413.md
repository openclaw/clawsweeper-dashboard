## Summary

Keep open. The PR targets a source-reproducible gateway auth problem, but the current diff is incomplete against current main, lacks contributor real-behavior proof, and changes a security-sensitive WebSocket operator authorization boundary that needs maintainer/secops approval.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so successful token/password operator auth can return `allow`, and adds a token-path policy test.

## Best Possible Solution

Approve one WebSocket token/password device-less scope policy, then land a rebased implementation that aligns the allow decision, scope-clearing helper, integration tests, docs, and real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid shared-token operator without device identity, clears its scopes, and checked-in tests expect a scoped RPC to fail with `missing scope`; I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The PR's direction may be valid after policy approval, but as written it changes the allow decision without aligning current main's scope-clearing helper, handler behavior, docs, or real behavior proof.

## Review Findings

Overall correctness: patch is incorrect
