## Summary

Keep open. The PR targets a real linked gateway auth regression, but the proposed allow-rule change is not sufficient against current main because a separate missing-device scope-clearing helper still strips token/password scopes, and the requested behavior changes a WebSocket authorization boundary that needs maintainer security/product review.

## What This Changes

The PR threads auth method into the WebSocket missing-device policy, allows successful token/password operator auth to skip device identity, and adds a token-path regression test.

## Best Possible Solution

Land one coherent maintainer-approved gateway auth policy that updates both the missing-device decision and the scope-clearing path, with token and password regression coverage proving the approved write-RPC behavior; otherwise document that remote backend clients must use device identity, device tokens, trusted-proxy, or another supported operator surface.

## Reproduction Assessment

Yes. A high-confidence code/test reproduction exists: current main allows valid device-less shared-token/password operator connects but clears requested scopes, and `chat.send` requires `operator.write`; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR's direction may be viable only after maintainer authorization-boundary approval, but the current patch does not update the scope-clearing helper on main and therefore would not prove the reported `chat.send` behavior.

## Review Findings
