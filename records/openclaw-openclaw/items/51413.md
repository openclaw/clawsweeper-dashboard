## Summary

Keep open: current `main` does not implement the requested non-local token/password scope retention, and this PR is now incomplete against the current gateway auth flow because scope clearing is handled by a separate helper that still strips those scopes. The remaining question is an authorization-boundary decision, especially with #51396 and #57889 still open.

## What This Changes

The PR threads auth method into missing-device evaluation so successful token/password operator auth can skip device identity, and adds a token-path connect-policy regression test.

## Best Possible Solution

Land one maintainer-approved gateway auth policy that either preserves scopes for approved remote token/password backend clients across both the missing-device decision and scope-clearing paths with token/password regression coverage and docs, or keeps the current device-identity/trusted-proxy requirements documented.

## Reproduction Assessment

Yes. Source and compatibility-test inspection give a high-confidence path: a device-less shared-token/password operator connect can succeed, then a write-scoped RPC fails after requested scopes are cleared; no live test was run because this review is read-only.

## Solution Assessment

No. The PR's direction may be viable only after maintainer authorization approval, but the current patch does not update the scope-clearing helper on `main` and conflicts with current tests/docs that limit preservation to local backend handoff.
