## Summary

Keep open. The PR targets a real source-reproducible gateway scope-clearing symptom, but it is not mergeable as-is because current main still clears token/password scopes through a newer helper path, the change crosses a remote WebSocket authorization boundary, and contributor real behavior proof is missing.

## What This Changes

The branch passes authMethod into WebSocket missing-device policy, allows successful token/password operator auth to skip device identity, and adds a token-path policy test.

## Best Possible Solution

Define and approve one WebSocket token/password device-less scope policy, then land a rebased implementation that updates the allow decision, scope-clearing helper, integration tests, and protocol/security docs together.

## Reproduction Assessment

Yes. Source inspection and checked-in tests show current main accepts a valid device-less shared-token operator connection, clears scopes, and expects a scoped RPC to fail with `missing scope`; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The PR's direction may be valid after policy approval, but as written it changes the allow decision without aligning current main's scope-clearing helper, handler behavior, docs, or real behavior proof.
