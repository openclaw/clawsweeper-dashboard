## Summary

Keep this PR open, but the current patch is not merge-ready: current main still clears device-less token/password operator scopes through the separate scope-clearing helper, and the requested remote scope preservation conflicts with current docs/tests that reserve preservation for local backend handoff paths.

## What This Changes

The PR threads `authMethod` into gateway missing-device evaluation so successful token/password operator auth can allow device-less non-local clients, and adds a token-path unit regression test.

## Best Possible Solution

Adopt one secops-approved gateway auth policy that either preserves scopes for an explicitly trusted remote backend path across both missing-device and scope-clearing logic with docs/tests, or keeps the current device-identity requirement documented and closes the linked reports accordingly.

## Reproduction Assessment

Yes, from source and compatibility-test inspection: a device-less shared-token/password operator can connect but has requested scopes cleared, causing write/admin RPCs to fail with missing scope. No live test was run because this review is read-only.

## Solution Assessment

No. The proposed direction may be valid only after maintainer approval, but the current patch changes the allow decision without updating the current scope-clearing policy and conflicts with docs/tests that limit device-less scope preservation to local backend handoff.
