## Summary

Keep open. The PR targets a real source-reproducible gateway scope symptom, but the branch is stale against current main, incomplete for the current scope-clearing path, and changes a security-sensitive remote device-less operator auth boundary without real behavior proof or maintainer policy approval.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so successful token/password operator auth can allow device-less non-local clients, plus a token-path regression test.

## Best Possible Solution

Settle one WebSocket device-less token/password scope policy, then land a current-main implementation that aligns connect policy, scope clearing, regression tests, protocol docs, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level: current tests expect valid shared-token/password operator connects with `device: null` to have scopes cleared and a scoped RPC to fail. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The direction may be acceptable only after policy approval, but this patch changes the allow decision without matching current main's scope-clearing helper, docs/tests, or real behavior proof.
