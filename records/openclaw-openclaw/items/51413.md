## Summary

Keep open. Current main still does not implement the intended non-local, device-less token/password WebSocket scope retention, and this PR is now incomplete against main because it only changes the missing-device allow decision while the newer scope-clearing helper still strips token/password scopes. The linked bug and a similar open implementation candidate remain unresolved, and the requested behavior changes an authorization boundary.

## What This Changes

The PR threads `authMethod` into the WebSocket missing-device policy, treats successful token/password auth as sufficient for an operator allow decision, and adds a token-path regression test.

## Best Possible Solution

Keep this PR open for maintainer review with #51396 and #57889. The desired end state is either a documented, tested WebSocket policy that updates both the missing-device decision and scope-clearing path for the approved remote shared-secret backend case, or an explicit decision that remote device-less WebSocket operators must use device identity, device tokens, trusted-proxy, or the HTTP trusted-operator surface instead.

## Reproduction Assessment

Yes. Current main has a high-confidence code/test reproduction path: valid device-less shared-token/password operator connects are allowed but their scopes are cleared, and `chat.send` remains an `operator.write` method. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR's direction may be viable only after maintainer authorization-boundary review, but the current patch is not sufficient on main because the separate scope-clearing helper still strips token/password scopes. A safer fix would centralize the approved preservation rule and cover token plus password with a write-RPC regression.
