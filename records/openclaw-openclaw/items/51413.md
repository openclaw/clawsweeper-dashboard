## Summary

Keep open. The PR targets a real gateway auth cluster, but it is stale/incomplete against current main and changes a security-sensitive remote device-less operator scope policy that needs maintainer/secops review plus real behavior proof before merge.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy, treats successful token/password operator auth as sufficient to allow device-less operators, and adds a token-path unit test.

## Best Possible Solution

Define one explicit WebSocket auth policy, then align the missing-device decision, scope-clearing helper, tests, and protocol/security docs around either an approved remote token/password exception or a documented device-identity requirement.

## Reproduction Assessment

Yes, from source and checked-in tests. Current main accepts a valid device-less shared-token operator connection, clears its scopes, and expects a scope-gated RPC to fail with `missing scope`; I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The PR's direction may be valid after policy approval, but as written it changes the allow decision without updating current main's scope-clearing policy and conflicts with docs/tests that limit device-less preservation to direct-local backend handoff.
