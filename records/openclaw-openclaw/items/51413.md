## Summary

Keep open: this PR is not ready to merge against current main because it changes the missing-device allow decision for remote token/password operator auth without updating the current scope-clearing policy, and the requested trust-boundary change still needs maintainer/security approval plus real behavior proof.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so successful token/password operator auth can allow device-less non-local clients, and adds a token-path unit regression test.

## Best Possible Solution

Adopt one secops-approved WebSocket auth policy that either preserves remote shared-token/password operator scopes across both missing-device and scope-clearing logic with docs and end-to-end coverage, or documents that remote clients must use device identity.

## Reproduction Assessment

Yes, from source and checked-in tests: current main's valid shared-token/password device-less operator path clears scopes and expects a scope-gated RPC to fail with `missing scope`. I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The PR's direction may be valid after maintainer approval, but against current main it changes the allow decision without updating the scope-clearing policy and conflicts with docs/tests that limit device-less preservation to direct-local backend handoff.
