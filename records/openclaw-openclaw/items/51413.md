## Summary

Keep open. The PR targets a real gateway auth-scope problem, but on current main it is incomplete because the separate scope-clearing helper still strips token/password scopes, and the remote device-less operator trust boundary needs maintainer/security approval plus real behavior proof.

## What This Changes

The PR threads `authMethod` into gateway WebSocket missing-device policy so successful token/password operator auth can return `allow`, and adds a token-path regression assertion.

## Best Possible Solution

Define and approve one WebSocket token/password device-less operator scope policy, then land a current-main fix that aligns connect policy, scope clearing, auth baselines, protocol docs, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid shared-token/password operator without device identity, clears requested scopes, and checked-in tests expect the follow-up scoped RPC to fail with `missing scope`.

## Solution Assessment

No. The PR touches the right decision point, but current main has a separate scope-clearing helper and baselines that must also change, and the remote device-less operator trust boundary needs explicit maintainer/security approval.
