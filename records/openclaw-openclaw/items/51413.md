## Summary

Keep open. The PR is still an incomplete fix against current main and it changes a security-sensitive remote device-less operator scope boundary that current tests and docs deliberately keep narrow.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so token/password-authenticated non-local operator clients can be allowed without device identity, and adds a token-path unit regression test.

## Best Possible Solution

Adopt one reviewed gateway auth contract for headless shared-auth clients, likely configured-scope or device-token-backed, then align source, regression tests, and protocol/security docs; otherwise keep documenting that remote clients need device identity.

## Reproduction Assessment

Yes. Current source and checked-in auth tests show a valid shared-token/password operator without device identity connects with cleared scopes and then fails scoped RPCs with `missing scope`; I did not run a live remote gateway because this review is read-only.

## Solution Assessment

No. The requested direction may be valid only after maintainer/security approval, and the patch is incomplete against current main because it changes the allow decision without reconciling the separate scope-clearing helper and documented trust boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align scope clearing with the new allow path:** `src/gateway/server/ws-connection/connect-policy.ts:117-124`
  - body: On current main, allowing token/password auth in `evaluateMissingDeviceIdentity` is not enough: missing-device sessions still call `shouldClearUnboundScopesForMissingDeviceIdentity`, which returns true for token/password auth and clears scopes before the session proceeds. A remote token/password operator would still lose `operator.admin`/`operator.write`, so the approved policy has to update the clearing helper/handler and add end-to-end coverage.
  - confidence: 0.9
