## Summary

Close this PR as superseded: the underlying gateway token-scope problem is now tracked by the canonical `gateway.auth.tokenScopes` design/implementation path, while this branch is conflicting, changes a security boundary by trusting self-declared scopes, and misses the current-main scope-clearing helper.

## What This Changes

The PR threads `authMethod` into WebSocket missing-device policy so successful token/password operator auth can return `allow`, and adds a token-path regression assertion.

## Best Possible Solution

Continue the maintainer-approved `gateway.auth.tokenScopes` path with bounded privileged-scope rules, aligned docs/tests, and real gateway proof instead of trusting client-declared device-less scopes by default.

## Reproduction Assessment

Yes, at source level. Current main accepts device-less shared-token/password operator connections, clears their requested scopes, and checked-in tests expect scoped RPCs to fail with `missing scope`.

## Solution Assessment

No. This PR touches one decision point but not current-main scope clearing, and the safer maintainable direction is the bounded configured token-scope contract in the canonical maintainer PR.

Full review comments:

- **[P2] Align scope clearing with the allow rule:** `src/gateway/server/ws-connection/connect-policy.ts:117-124`
