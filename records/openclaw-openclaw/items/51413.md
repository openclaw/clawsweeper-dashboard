## Summary

Keep open, but this patch should not merge as-is. It relaxes the current gateway auth boundary by letting device-less shared-token/password WebSocket operators keep self-declared scopes, while current main, docs, and tests intentionally clear those scopes except for narrow approved paths; the PR also lacks after-fix real behavior proof.

## What This Changes

The branch threads `authMethod` into missing-device policy, treats successful token/password operator auth as an allow decision, and adds a focused connect-policy unit test.

## Best Possible Solution

Define an explicit headless WebSocket client contract, then land a current-main implementation that uses paired device identity/device tokens or a reviewed configured-scope path with aligned tests and protocol docs.

## Reproduction Assessment

Yes. Current source and checked-in tests show a high-confidence path: connect as an operator with a valid shared token/password and no device identity, receive `hello-ok` with cleared scopes, then observe a scoped RPC fail with `missing scope`; I did not run live tests because this review is read-only.

## Solution Assessment

No. Preserving all self-declared token/password scopes is not the narrowest maintainable fix because current main treats those scopes as unbound without device identity; a safer direction is an explicit maintainer-approved scope grant or device-token contract.
