## Summary

This PR should stay open: the latest head fixes the option-key poisoning direction, but cache hits can still return JSON before the underlying fs-safe root, hardlink, and symlink validation contracts run.

## What This Changes

The branch adds an opt-in process-scoped stat-validated cache for sync JSON reads, routes selected helpers through it, and keeps the legacy `loadJsonFile` path on the raw reader.

## Best Possible Solution

Keep the performance win only after cache hits preserve the full fs-safe contract, or move the optimization to a caller-owned discovery/snapshot cache where the boundary has already been validated.

## Reproduction Assessment

Yes for the PR-introduced defects by source inspection. A focused test can cache a root read, then repeat through an escaping root-relative path or after adding a hardlink; a separate regular-read test can replace the cached path with a symlink to the same file.

## Solution Assessment

No. The current cache shape is useful for the measured startup cost, but the best fix must either rerun or fully encode the raw fs-safe validation contract before serving hits.

  - body: `readRootJsonObjectSync` excludes `rootDir` and `relativePath` from the cache key and serves a cached value by resolved absolute path.   - body: For root reads with `rejectHardlinks: true`, a file can be cached while it has one link and then gain another hardlink without changing `mtimeMs`, `size`, or `ino`. The next strict read returns the cached success at this line, while the raw fs-safe reader would reject `nlink > 1`.
  - confidence: 0.9
- **[P1] Revalidate regular-file symlink state before hits:** `src/infra/json-files.ts:49-50`

  - confidence: 0.88

## Risks / Open Questions

- A cached root-read success can be reused for a different `rootDir`/`relativePath` pair that resolves to the same absolute path before fs-safe checks whether that request is inside the requested root.
- A strict root read cached before a hardlink is added can keep returning success because the cache hit does not compare `nlink` or rerun `rejectHardlinks` validation.
- A regular cached read can survive replacement of the path with a symlink to the same file because the cache uses `fs.statSync`, which follows symlinks, while the raw reader rejects final-path symlinks.
