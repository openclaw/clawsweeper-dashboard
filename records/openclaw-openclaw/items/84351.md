## Summary

Keep open: the PR targets a real startup hot path, but the current cache can bypass fs-safe root-read safety options and the maintainer-labeled PR should stay under maintainer review.

## What This Changes

The PR wraps sync JSON readers in `src/infra/json-files.ts` with a process-local stat-validated cache and routes two infra import sites through it.

## Best Possible Solution

Keep the performance work only if cached reads preserve the full upstream fs-safe contract; prefer domain-scoped manifest/snapshot caching, or key/bypass root-boundary reads with focused security regression coverage.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows the cached root-read result is returned before fs-safe receives the later call's `rejectHardlinks`, `maxBytes`, or `rootRealPath`. I did not run a live repro because this review is read-only.

## Solution Assessment

No. The narrow maintainable path is to preserve fs-safe root-read semantics by bypassing this cache for root-boundary reads or keying it on all safety-relevant options with regression tests.

## Review Findings

Full review comments:

- **[P1] Key root-read cache by safety options:** `src/infra/json-files.ts:82-83`
  - body: `readRootJsonObjectSync` is cached only by resolved path, so a cached `ok` result is returned before the upstream reader can apply a later call's `rootRealPath`, `rejectHardlinks`, or `maxBytes`.   - confidence: 0.92
