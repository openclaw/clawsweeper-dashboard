## Summary

82dc858e892f7af7e80a5ea543ca355eea1b08e7 addresses the safety-options part of the earlier review, but it does not clear the PR: the same path-only cache still crosses `tryReadJsonSync` and `readRootJsonObjectSync`, so a root-bounded read can return a plain cached parse without running the fs-safe root reader.

## What This Changes

The PR adds an opt-in, stat-validated process cache for sync JSON reads, routes one package-update root read through it, and keeps legacy `loadJsonFile` on the raw reader.

## Best Possible Solution

Keep the performance win only if cache identity is partitioned by reader contract and root-boundary semantics, or limit this PR to plain `tryReadJsonSync` caching while root-bounded reads stay on the raw fs-safe path.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: a mixed plain/root read of the same resolved path hits the shared map before the raw fs-safe reader runs. I did not run tests because this pass is read-only.

## Solution Assessment

No. The latest commit fixes the strict-options bypass, but the best fix still needs separate cache identity by API contract or no root-reader caching at this layer.
