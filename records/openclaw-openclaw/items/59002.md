## Summary

Keep open. Current `main` still has the same credible Gmail hook setup/session-policy regression: the setup path installs the templated Gmail preset without the complete session policy that startup and runtime dispatch require, and the linked repair remains open and partial.

## What This Changes

The issue reports that Gmail inbound hook sessions and downstream Signal notifications stopped after upgrading from OpenClaw 2026.3.28 to 2026.3.31.

## Best Possible Solution

Revise the Gmail setup path to emit a complete, intentional session policy for the effective mapping it installs, then repair or replace https://github.com/openclaw/openclaw/pull/72001 with startup and runtime `/hooks/gmail` dispatch coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the wizard installs the templated Gmail preset, current gateway startup requires prefixes, and mapped delivery still rejects templated keys unless `hooks.allowRequestSessionKey` is enabled. I did not run live Gmail/Tailscale/Signal proof in this read-only pass.

## Solution Assessment

No landed solution exists. The current open repair at https://github.com/openclaw/openclaw/pull/72001 appears necessary but incomplete because it seeds prefixes without settling or enabling the runtime templated-session policy.
