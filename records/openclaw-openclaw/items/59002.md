## Summary

Keep open. Current main and the latest release still have a credible Gmail hook session-policy mismatch: the Gmail setup path installs the templated Gmail preset, while startup/runtime policy requires explicit bounded request-session-key settings for that preset to create per-message sessions and dispatch notifications.

## What This Changes

The issue reports that, after updating to 2026.3.31, inbound Gmail no longer creates `hook:gmail:<id>` sessions or triggers downstream Signal notifications.

## Best Possible Solution

Revise or replace the open Gmail wizard repair so the emitted config matches the effective mapping: either bounded per-message Gmail sessions with runtime coverage, or a static Gmail mapping when request-derived session keys stay disabled.

## Reproduction Assessment

Yes, at source level: current main installs the templated Gmail preset without the complete session policy, and mapped dispatch rejects template-derived keys unless `hooks.allowRequestSessionKey` is enabled. I did not run live Gmail Pub/Sub or Signal proof in this read-only pass.

## Solution Assessment

No landed solution exists. The open repair at https://github.com/openclaw/openclaw/pull/72001 is relevant but incomplete because prefixes alone satisfy only startup validation, not the runtime templated-session gate or the static-vs-per-message policy choice.
