## Summary

Keep open. Current main still has credible Gmail hook regressions in the setup and routing path: the generated Gmail config enables the templated Gmail preset without writing the session-key policy that current hook routing requires, and the Tailscale setup helper still appends the push token as a query parameter. The related PR #60574 is useful evidence, but it was closed unmerged and only covers the Tailscale-token portion.

## What This Changes

The issue reports that Gmail Pub/Sub hook events stopped creating `hook:gmail:<id>` sessions and stopped delivering configured Signal notifications after updating from OpenClaw 2026.3.28 to 2026.3.31.

## Best Possible Solution

Keep the issue open and fix the Gmail integration end to end: generated Gmail setup should either write the safe policy needed for the templated `hook:gmail:<message-id>` preset or install a static safe mapping, the Tailscale push endpoint should not carry the push token in the query string, and focused tests should cover setup output, mapped Gmail dispatch, and the token-free endpoint shape.

## Reproduction Assessment

Yes. There is a high-confidence code-level reproduction path: current setup writes the Gmail preset without the session-key policy current hook routing requires, and existing gateway tests show such templated mappings are rejected before dispatch; live Gmail/Tailscale/Signal was not exercised in this read-only pass.

## Solution Assessment

No current solution is landed. The related PR #60574 addresses the Tailscale query-token part but was closed unmerged and does not cover the setup-policy mismatch, so the best repair should handle both paths together.
