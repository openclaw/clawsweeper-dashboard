## Summary

Keep open. Current main and the latest v2026.5.22 release still show the Gmail setup/runtime policy mismatch: setup enables the templated Gmail preset, while startup and mapped dispatch require an explicit bounded request-session-key policy before per-message Gmail sessions can work.

## What This Changes

The issue reports that after updating to 2026.3.31, Gmail watcher delivery remains healthy but Gmail hook sessions and downstream Signal notifications stop appearing.

## Best Possible Solution

Make Gmail setup emit one coherent end-to-end session policy: either bounded per-message Gmail sessions with allowRequestSessionKey, constrained prefixes, dispatch coverage, and live proof, or a static Gmail mapping/default-session path that keeps request-derived session keys disabled.

## Reproduction Assessment

Yes, at source level: current main installs the templated Gmail preset without the complete session policy, and mapped dispatch rejects template-derived keys unless hooks.allowRequestSessionKey is enabled. I did not run live Gmail Pub/Sub or Signal proof in this read-only pass.

## Solution Assessment

No landed solution exists. The related open repair at https://github.com/openclaw/openclaw/pull/72001 is relevant, but adding prefixes alone addresses startup validation without settling the runtime templated-session gate or static-vs-per-message policy choice.
