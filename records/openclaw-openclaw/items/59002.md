## Summary

Keep open. Current main and the latest release still have a credible Gmail hook session-policy mismatch: the Gmail setup path enables the templated Gmail preset, while startup and runtime require explicit bounded request-session-key policy before per-message Gmail sessions can dispatch.

## What This Changes

The issue reports that Gmail inbound hook sessions and downstream Signal notifications stopped after upgrading from OpenClaw 2026.3.28 to 2026.3.31.

## Best Possible Solution

Revise Gmail setup so the emitted config matches the effective Gmail mapping policy end to end: bounded per-message sessions with `allowRequestSessionKey` and prefixes, or a static Gmail mapping while request-derived session keys stay disabled.

## Reproduction Assessment

Yes, at source level: current main installs the templated Gmail preset without the complete session policy, and mapped dispatch rejects template-derived keys unless `hooks.allowRequestSessionKey` is enabled. I did not run live Gmail Pub/Sub or Signal proof in this read-only pass.

## Solution Assessment

No landed solution exists. The open repair at https://github.com/openclaw/openclaw/pull/72001 is relevant but incomplete because prefixes alone satisfy only startup validation, not the runtime templated-session gate or the static-vs-per-message policy choice.
