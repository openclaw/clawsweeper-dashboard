## Summary

This still looks valid on current main: Gemini CLI OAuth token exchange and project discovery still use the strict guarded-fetch default, while the fetch-guard contract says strict mode ignores ordinary proxy env vars unless the managed proxy lifecycle is active. An open fix PR already targets the same bug, so this issue should stay open until that work lands or is replaced.

## What This Changes

Request that google-gemini-cli OAuth complete behind a macOS HTTP(S) proxy by honoring proxy routing during token exchange and follow-up Google project discovery calls.

## Best Possible Solution

Land a narrow Google plugin fix that uses the existing trusted env-proxy guarded-fetch helper for the known Google OAuth/project endpoints, with regression coverage for env proxy routing and no change to global strict SSRF defaults.

## Reproduction Assessment

Yes, by source inspection: run Gemini CLI OAuth with `HTTPS_PROXY`/`HTTP_PROXY` set on a proxy-only network and current `fetchWithTimeout` will still call the strict guarded-fetch default for token exchange and project discovery. I did not perform the live macOS OAuth flow, so this is `source_reproducible` rather than an executed reproduction.

## Solution Assessment

Yes, the narrow maintainable direction is to route only these trusted first-party Google OAuth/project requests through trusted env-proxy mode when env proxy applies. Changing the global fetch-guard default would be broader and riskier.
