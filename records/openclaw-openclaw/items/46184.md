## Summary

Keep open: current main still routes google-gemini-cli OAuth token exchange and project discovery through the strict SSRF guarded-fetch path, which the repo tests document as ignoring environment proxy variables. Open PR #47518 directly targets this failure, so the issue should wait for that PR or a narrow replacement rather than be closed.

## What This Changes

The issue requests google-gemini-cli OAuth to complete behind a macOS HTTP(S) proxy instead of failing after the browser callback during token exchange.

## Best Possible Solution

Land a provider-owned fix that opts only the known Google OAuth/project-discovery endpoints into trusted env-proxy routing with regression coverage, while preserving strict SSRF defaults elsewhere.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: run `openclaw models auth login --provider google-gemini-cli --set-default` with direct Google egress blocked and `HTTP_PROXY`/`HTTPS_PROXY` set; current main reaches `fetchWithSsrFGuard` in strict mode, which repo tests say ignores env proxy settings.

## Solution Assessment

Yes for the env-proxy failure. The narrow maintainable solution is a google-gemini-cli-auth-local change to use trusted env-proxy mode for the known first-party Google OAuth endpoints, not a global relaxation of guarded-fetch strict defaults.
