## Summary

Keep open. Current main still routes Gemini CLI OAuth token exchange and project-discovery requests through the default strict guarded-fetch path, which does not honor ordinary env proxy settings unless trusted proxy mode or the managed OpenClaw proxy lifecycle is active. The linked #47518 PR is an open implementation candidate for this exact plugin-owned behavior, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that Gemini CLI OAuth token exchange and follow-up Google requests work behind a macOS HTTP(S) proxy instead of failing with `TypeError: fetch failed`.

## Best Possible Solution

Review and land #47518 or a narrower replacement that keeps the change inside the Google plugin OAuth helper: opt the known first-party Google OAuth/project endpoints into trusted env-proxy routing only when the target should use env proxy, preserve strict DNS-pinned defaults elsewhere, cover `HTTP_PROXY`/`HTTPS_PROXY` and `NO_PROXY`, and document whether macOS system proxy without inherited env is supported.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: with a proxy-required network and no managed OpenClaw proxy lifecycle, `openclaw models auth login --provider google-gemini-cli --set-default` reaches token exchange through `extensions/google/oauth.http.ts`, while the guard tests show strict mode ignores env proxy by default.

## Solution Assessment

Unclear as a complete solution. Enabling trusted env-proxy mode for the Google OAuth/project helper is the likely narrow fix for env-proxy setups, but maintainers still need to review #47518's SSRF boundary and decide whether macOS system proxy without env vars is in scope.
