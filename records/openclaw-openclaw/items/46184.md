Keep open: current main still routes Gemini CLI OAuth token, userinfo, and project-discovery calls through the default strict guarded-fetch path, so the reported proxy failure remains source-reproducible. ## What This Changes

The issue reports that google-gemini-cli OAuth token exchange fails behind a macOS HTTP(S) proxy because the guarded fetch path does not route trusted Google OAuth calls through the proxy environment.

## Best Possible Solution

Land a narrow current-main fix in the Google plugin OAuth HTTP helper that uses target-aware trusted env-proxy mode for trusted Google OAuth calls, with focused regression coverage and no unrelated Discord, LINE, Nostr, or plugin-SDK churn.

## Reproduction Assessment

Yes, by source inspection plus the reporter's manual patch result. Current main's Gemini OAuth helper does not opt into trusted env-proxy mode, while the guarded-fetch implementation only uses EnvHttpProxyAgent for explicit trusted env-proxy calls.

## Solution Assessment

No for the existing open PR as-is. The right fix direction is narrow: update only the Google OAuth fetch helper to use target-aware trusted env proxy handling and add focused tests, rather than merging the broad stale branch.

## Repair Work Prompt

Keep the patch narrow: update the Google plugin OAuth HTTP helper so trusted Google OAuth/token/userinfo/project-discovery calls use `mode: "trusted_env_proxy"` only when `shouldUseEnvHttpProxyForUrl(url)` says the target should use an HTTP(S) env proxy; do not use a broad `hasProxyEnvConfigured` gate, do not add a config option, and do not touch Discord, LINE, Nostr, plugin-SDK, or unrelated dispatcher behavior. Add focused tests that prove the helper passes trusted env-proxy mode when HTTPS_PROXY/HTTP_PROXY applies and does not do so for NO_PROXY or non-matching env cases.
