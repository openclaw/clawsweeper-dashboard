## Summary

Keep open: current main and the latest shipped release still leave the Gemini CLI OAuth HTTP helper on the default strict guarded-fetch path, so the proxied macOS OAuth failure remains source-reproducible. The broad related fix PR was closed without merge, and no linked merged PR implements the narrow trusted-env-proxy fix.

## What This Changes

The issue asks the `google-gemini-cli` OAuth token and identity requests to work behind a macOS HTTP(S) proxy instead of failing with `TypeError: fetch failed`.

## Best Possible Solution

Land a narrow bundled Google plugin fix that opts the trusted first-party Google OAuth endpoints into target-aware env-proxy routing, with regression coverage and redacted real proxy proof when feasible.

## Reproduction Assessment

Yes. Source inspection shows current main still sends Gemini CLI OAuth token, userinfo, and project-discovery requests through the strict guarded-fetch default, and the reporter confirmed a local trusted-env-proxy patch resolves the proxied failure.

## Solution Assessment

Yes for a narrow provider-local helper change; no for reviving the broad closed PR at https://github.com/openclaw/openclaw/pull/47518. The maintainable path is to update the Google OAuth HTTP helper and focused tests while preserving the existing guarded-fetch proxy contract.
