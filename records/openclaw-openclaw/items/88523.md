## Label Justifications

- `P1`: The PR addresses a broken subagent completion handoff path where parent agents can miss child results under concurrent completion backpressure.
- `merge-risk: 🚨 message-delivery`: Merging changes when subagent completion messages give up versus keep retrying, which can affect whether parent agents receive completion results.

## Summary

Keep open for PR repair and maintainer review: current main still has the retry-cap path that matches the reported dropped completion handoffs, and the PR is targeted, but the new persisted pending marker can stay stale after a later announce attempt fails before producing a delivery result.

## What This Changes

The PR adds an optional `lastHandoffPending` delivery flag and uses it in subagent cleanup/resume retry guards so busy-parent completion handoffs can retry past the 3-attempt cap until the hard expiry, with regression coverage.

## Best Possible Solution

Clear the pending-handoff marker for each new attempt or no-result failure so only the current `completion_handoff_pending` result gets the cap exemption, then keep the hard-expiry bound and merge after maintainer review.

## Reproduction Assessment

Yes, source inspection and the reporter log give a high-confidence path: current main returns `completion_handoff_pending` for busy parent completion handoffs while both cleanup and resume guards still apply the retry cap. I did not run a live multi-subagent gateway reproduction.

## Solution Assessment

No, not quite: the overall approach is narrow, but the marker must be cleared for no-result/thrown attempts so stale pending state cannot exempt genuine failures. After that repair, the persisted-marker approach is cleaner than raising retry constants globally.
