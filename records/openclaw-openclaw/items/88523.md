## Summary

Keep this PR open, but do not merge it yet: the proposed helper change is in the right area, but the resumed-run preflight still gives up as soon as `attemptCount >= MAX_ANNOUNCE_RETRY_COUNT`, so the reported concurrent completion drop can still occur after the scheduled retry.

## What This Changes

The PR exempts transient `completion_handoff_pending` completion announces from the retry-count cap in the cleanup decision helper, threads that reason through lifecycle cleanup, and adds focused decision-helper tests.

## Best Possible Solution

Make retry-cap handling flow-aware at both the cleanup decision helper and the `resumeSubagentRun` preflight, while preserving the cap for non-completion and non-pending failures plus the 30-minute hard expiry.

## Reproduction Assessment

Yes from source inspection, though I did not run a live multi-agent repro. A `completion_handoff_pending` announce can exceed the cap, the PR helper returns `retry`, and the resumed entrypoint still gives up before another announce attempt.

## Solution Assessment

No. The helper change is directionally right, but the best fix must also update the resumed retry guard and cover that path with a regression test.
