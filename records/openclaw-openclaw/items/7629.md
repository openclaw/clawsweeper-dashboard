## Summary

Keep open: current main and v2026.6.8 still leave `memory-lancedb` on vector-only retrieval with no FTS/BM25 index, no hybrid config, and no query-text path into LanceDB search. The earlier implementation PR is closed unmerged and too broad to be the canonical fix path, so this issue remains the focused tracker.

## What This Changes

Add BM25 plus vector hybrid retrieval to the `memory-lancedb` plugin so exact keyword matches participate in memory recall.

## Best Possible Solution

Keep this as a focused `memory-lancedb` parity enhancement owned by the plugin: add an approved hybrid config/default, maintain a LanceDB FTS index on `text`, pass query text through recall/search paths, implement verified JS hybrid or local fusion behavior, preserve vector-only fallback, and document the final contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `memory-lancedb`, store text where an exact token should matter, and recall/search still sends only an embedding vector to LanceDB `vectorSearch` with no FTS/BM25 path.

## Solution Assessment

Unclear as proposed. The owner boundary is right, but the best implementation needs maintainer-approved defaults and a verified LanceDB JS hybrid or local-fusion path rather than copying the Python-style issue sketch directly.
