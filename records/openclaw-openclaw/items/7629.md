## Summary

Keep open. Current main and the latest release still leave `memory-lancedb` on vector-only retrieval, while the default memory engine already has BM25 + vector hybrid search; the earlier matching PR was closed unmerged and is not a viable canonical fix.

## What This Changes

Add hybrid BM25 + vector retrieval to `memory-lancedb`, including FTS indexing, configurable weighting, query-text propagation, and fallback behavior for exact-keyword recall parity.

## Best Possible Solution

Keep this as a focused `memory-lancedb` parity enhancement owned inside the plugin: add maintainer-approved hybrid config/defaults, create or maintain an FTS index on `text`, thread query text through recall/forget/CLI/duplicate paths, preserve vector fallback, and cover exact-keyword recall with targeted tests.

## Reproduction Assessment

Yes, as source proof rather than a bug repro: current main and `v2026.6.6` show `memory-lancedb` embedding queries and calling `vectorSearch` only, with no FTS index or hybrid config. I did not run a live LanceDB setup because the missing path is clear in the implementation and tests.

## Solution Assessment

Yes on direction, not on the old branch: the maintainable fix belongs in `extensions/memory-lancedb` rather than core memory, but the closed broad PR is not the best landing vehicle. The best implementation should be a new focused plugin change with dependency-verified LanceDB JS API use and upgrade-safe fallback behavior.
