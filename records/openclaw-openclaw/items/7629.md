## Summary

Keep open. Current main still leaves the bundled memory-lancedb plugin on vector-only retrieval: it opens or creates the LanceDB table without an FTS index, exposes no hybrid/BM25 config, passes only embedding vectors into recall/search paths, and tests still assert vectorSearch. The linked implementation PR was closed unmerged, while the default builtin memory engine’s hybrid BM25 + vector support is a separate code path and does not give LanceDB parity.

## What This Changes

Add BM25/full-text retrieval to the bundled memory-lancedb plugin and combine it with vector similarity for recall/search parity.

## Best Possible Solution

Keep this as a focused bundled memory-lancedb parity enhancement: add a typed hybrid config, maintain an FTS index on the text column, propagate original query text alongside embeddings, merge BM25 and vector candidates with a tested fallback when LanceDB native hybrid is unavailable, preserve vector-only behavior when FTS setup fails or hybrid is disabled, and document the new knobs.

## Reproduction Assessment

Not applicable as a pure feature request, but the current-main gap is high-confidence: source and tests show vector-only LanceDB recall/search with no FTS index or hybrid config. A future regression test can reproduce the user problem by storing an exact symbol/env-var memory and asserting a lexical query can rank it through the LanceDB plugin.

## Solution Assessment

Yes, adding this inside extensions/memory-lancedb is the right boundary because the default memory engine already has its own hybrid path and this request is specifically LanceDB parity. The exact implementation should not assume Python-style native hybrid APIs; it should verify the pinned JS dependency and use separate FTS/vector queries plus local fusion if that is the maintainable path.
