## Summary

Current main still lacks the requested capability, the pinned LanceDB TypeScript client supports a focused plugin-local implementation, and no merged or viable open PR owns the work. The issue should remain open, but default enablement, ranking semantics, score thresholds, fallback behavior, and whether another backend-specific config surface fits the memory convergence direction require maintainer product judgment first.

## What This Changes

Add full-text indexing and hybrid BM25-plus-vector retrieval to the bundled memory-lancedb plugin, with configuration for enablement and ranking behavior.

## Best Possible Solution

Keep the implementation entirely in `extensions/memory-lancedb`, thread original query text through every retrieval caller, create and maintain an FTS index, use a supported LanceDB TypeScript hybrid/reranker path, define one stable score contract for recall and duplicate checks, preserve an explicit vector-only mode, and document and test fresh-table, existing-table, and FTS-failure behavior.

## Reproduction Assessment

Yes for the stated limitation: current-main source shows no FTS index and every database search receives only a vector, so exact-keyword BM25 retrieval cannot occur. This is a reproducible feature gap, not a broken existing contract.

## Solution Assessment

No, not exactly as proposed: the plugin boundary and hybrid goal are sound, but the implementation must follow the pinned TypeScript API, and maintainers must choose default enablement, reranking, threshold semantics, index-failure behavior, and whether weights should be public configuration.

## Risks / Open Questions

- Hybrid ranking can silently change recall order, minimum-score meaning, duplicate detection, forget candidates, and which memories are injected into prompts for existing users.
- A vectorWeight/textWeight surface could imply linear score fusion even though the supported dependency path naturally provides RRF; maintainers need to choose one stable contract rather than expose misleading knobs.
- Failing open to vector-only search may hide broken FTS setup indefinitely, while failing closed would break existing recall; index lifecycle and operator-visible fallback policy need an explicit decision.

## Close Comment

Thanks for the idea. I checked the current extension path, and this is a better fit for [ClawHub.com](https://clawhub.ai/) than OpenClaw core.

The requested hybrid retrieval is useful, but it no longer needs to be added to OpenClaw’s bundled plugin: the existing plugin API and LanceDB JavaScript contract are sufficient, and the published ClawHub `memory-lancedb-pro` plugin already provides vector-plus-BM25 retrieval. Keeping a second bundled implementation would add config, ranking, migration, and fallback policy to core for capability users can install externally.

So I’m closing this as a scope-fit item for the plugin/community path. Please upload or publish it through [ClawHub.com](https://clawhub.ai/) so it can live as an installable community skill instead of a bundled OpenClaw core change.
