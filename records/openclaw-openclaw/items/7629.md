## Summary

Keep open. Current main still leaves the bundled `memory-lancedb` plugin on vector-only retrieval, with no FTS/BM25 index setup, no hybrid config surface, and no query text passed into LanceDB search paths; the related implementation PR was closed unmerged.

## What This Changes

Add BM25/full-text retrieval to the bundled `memory-lancedb` plugin and combine it with vector similarity for LanceDB memory recall/search parity.

## Best Possible Solution

Keep this as a focused bundled `memory-lancedb` parity enhancement: add typed hybrid config, maintain an FTS index on `text`, propagate original query text alongside embeddings, preserve vector-only fallback, and document/test the new behavior.

## Reproduction Assessment

Not applicable as a pure feature request, but the current-main gap is high-confidence from source inspection: LanceDB recall/search is vector-only and has no FTS index or hybrid config. A future regression test can store an exact symbol/env-var memory and assert a lexical query ranks it through the LanceDB plugin.

## Solution Assessment

Yes. The right boundary is `extensions/memory-lancedb`, because builtin `memory_search` already has its own hybrid path and this request is specifically LanceDB parity; the implementation should verify the JS LanceDB API and use a tested local fusion fallback if native hybrid is not suitable.
