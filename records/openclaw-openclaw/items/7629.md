## Summary

Keep open: current main and the latest release still provide vector-only retrieval in the bundled LanceDB backend. An external enhanced plugin exists, but it does not settle whether the supported bundled backend should gain hybrid parity; that scope and upgrade behavior need maintainer judgment.

## What This Changes

Add BM25 plus vector hybrid retrieval to the bundled memory-lancedb plugin.

## Best Possible Solution

Decide whether the bundled LanceDB backend remains a parity target; if accepted, implement hybrid retrieval entirely within `extensions/memory-lancedb` using verified JavaScript APIs, explicit existing-table FTS index lifecycle, stable score semantics, vector-only fallback, and real local LanceDB proof.

## Reproduction Assessment

Yes at the source level: current main passes only embeddings into `MemoryDB.search`, creates no FTS index, and tests assert `vectorSearch`, so exact-keyword retrieval cannot participate. This demonstrates a feature gap, not a failure of an existing documented LanceDB contract.

## Solution Assessment

Unclear. Plugin-owned FTS and hybrid fusion are technically plausible, but the proposed config shape, default-on behavior, score mapping, fallback policy, and bundled-product scope require maintainer decisions before implementation.
