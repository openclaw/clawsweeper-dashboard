## Summary

Keep open: current main and v2026.6.5 still leave the official memory-lancedb plugin on vector-only retrieval, while the matching implementation PR was closed unmerged and no viable superseding core/plugin work owns the parity request. The related memory-lancedb-pro discussion shows a community/ClawHub path for a fuller retrieval pipeline, but it does not implement this requested capability in the official plugin.

## What This Changes

The issue requests hybrid BM25 plus vector retrieval for the memory-lancedb plugin so exact keyword matches can participate in long-term memory recall.

## Best Possible Solution

Keep this open as a maintainer-reviewed memory-lancedb parity enhancement: implement plugin-owned hybrid retrieval with explicit config, FTS index maintenance on text, query-text propagation, JavaScript API proof, fallback behavior, docs, and focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug repro, but source inspection gives high confidence the capability is absent: current memory-lancedb creates no FTS index, exposes no hybrid config, and passes only vectors into db.search.

## Solution Assessment

Is this the best way to solve the issue? The direction is plausible, but the best implementation needs maintainer choices for official-plugin scope, defaults, ranking weights, reranker policy, and fallback semantics; the clean boundary is memory-lancedb, not core runtime or another broad mixed-surface PR.
