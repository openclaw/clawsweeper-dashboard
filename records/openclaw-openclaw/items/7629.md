## Summary

Current main still provides vector-only LanceDB retrieval, and no merged or viable open PR owns the missing hybrid behavior. The feature is technically feasible inside the plugin, but its default, score semantics, threshold compatibility, and failure policy need maintainer product judgment.

## What This Changes

Add BM25 plus vector hybrid retrieval to the memory-lancedb plugin, including FTS indexing, query-text propagation, and configuration.

## Best Possible Solution

Keep the implementation plugin-local and define one maintainer-approved contract for FTS index lifecycle, query fusion, normalized scores, defaults, thresholds, and failure handling, then align every search caller, tests, manifest, and public documentation.

## Reproduction Assessment

Yes, at source level: current main has no FTS index, lexical query path, hybrid configuration, or query-text-aware database search. This demonstrates the missing capability, though it is a feature gap rather than a broken established contract.

## Solution Assessment

No, not exactly as proposed: plugin-local hybrid search is the right ownership boundary, but the pinned TypeScript client requires explicit FTS and reranker composition, and maintainers must first define compatible scoring, defaults, and failure behavior.

- Enabling hybrid retrieval by default would change existing users' ranking behavior, while silent vector fallback after an FTS failure could conceal a degraded configuration.
- Adding another backend-specific configuration surface may conflict with the repository's stated direction to converge on one recommended memory path.
