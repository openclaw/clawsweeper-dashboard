## Summary

Keep open: current main and the latest release still leave `memory-lancedb` on vector-only LanceDB retrieval, while the direct implementation attempt at https://github.com/openclaw/openclaw/pull/7636 is closed unmerged, conflicting, and too broad to be the canonical fix path.

## What This Changes

The issue requests hybrid BM25 plus vector search support for the official `memory-lancedb` plugin.

## Best Possible Solution

Keep this as a focused `memory-lancedb` parity enhancement: add typed plugin config and docs, create or maintain a LanceDB FTS index on `text`, pass query text through recall/forget/CLI/hooks, use the verified LanceDB JS hybrid/rerank path or a tested local fusion fallback, and preserve vector-only behavior when hybrid is disabled or unavailable.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `memory-lancedb`, store a memory whose exact keyword is not favored by embeddings, and current recall still passes only vectors into LanceDB `vectorSearch` with no FTS/BM25 path. I did not run a live LanceDB scenario because this review is read-only.

## Solution Assessment

Unclear until maintainer policy is chosen. `memory-lancedb` is the right owner boundary, but the best implementation needs an approved default, fallback, and config shape instead of blindly enabling new hybrid behavior for existing installs.
