## Summary

Keep open: this focused draft feature PR appears to cover the linked cron alert timestamp request and now has both terminal and Telegram-visible proof, but it remains a draft and overlaps another open implementation at https://github.com/openclaw/openclaw/pull/77811.

## What This Changes

The PR adds a cron alert event-time formatter, threads eventTimeMs through repeated and gateway failure-alert paths, adds event-time text to auto-disable notifications, and updates focused cron/gateway tests.

## Best Possible Solution

Land one canonical implementation that formats event time at the cron alert boundary using existing run metadata and schedule timezone, without adding new cron config or changing routing semantics.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection gives a high-confidence current-main check: repeated, first-failure, and auto-disable alert text currently omit event-time text while run/event metadata already carries timestamps.

## Solution Assessment

Yes, this formatter-plus-plumbing approach is the narrow maintainable direction for the linked request because it covers repeated alerts, first-failure destination/fallback messages, webhook payloads, and auto-disable text. The remaining decision is maintainer reconciliation with the sibling implementation and draft status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the cron scheduler timezone fallback:** `src/cron/alert-time.ts:6`
  - body: For cron jobs created without `--tz`, the scheduler and docs use the gateway host local timezone, but this formatter labels those event times as UTC. On non-UTC hosts, a default cron job that runs at local 09:00 would alert as a different UTC wall-clock time, which reintroduces the misleading timestamp problem this PR is trying to fix. Resolve missing cron `tz` the same way `schedule.ts` does, and reserve UTC for non-cron or invalid-timezone fallbacks.
  - confidence: 0.87
