## Summary

Keep open: the PR has the right overall shape and convincing real behavior proof, but it still has a blocking timezone bug. Cron schedules without `--tz` are scheduled in the gateway host timezone on current main, while the new formatter labels those alert event times as UTC.

## What This Changes

The branch adds a shared cron alert event-time formatter, threads event timestamps through cron failure alert delivery and webhook paths, and updates focused cron/gateway tests.

## Best Possible Solution

Land one cron timestamp implementation after the formatter shares the existing schedule timezone semantics, keeping alert routing and config schema unchanged.

## Reproduction Assessment

Yes at source level: current main alert builders omit event-time text while cron finished events already carry run timing. The PR regression is also source-reproducible because its new formatter returns UTC for cron schedules without `tz`, while current scheduler/docs use the gateway host timezone.

## Solution Assessment

No, not as written. The formatter-plus-plumbing approach is maintainable, but missing cron `tz` must resolve through the same host-local fallback as the scheduler before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the cron scheduler timezone fallback:** `src/cron/alert-time.ts:7`
  - body: For cron jobs created without `--tz`, the scheduler and docs use the gateway host local timezone, but this formatter labels those event times as UTC. On non-UTC hosts, a default cron job that runs at local 09:00 would alert as a different UTC wall-clock time, which reintroduces the misleading timestamp problem this PR is trying to fix. Resolve missing cron `tz` the same way `schedule.ts` does, and reserve UTC for non-cron or invalid-timezone fallbacks.
  - confidence: 0.87
