## Summary

The PR is useful and has real Telegram proof, but it still formats cron jobs without an explicit `tz` as UTC while current cron scheduling and docs use the gateway host timezone for that case. That would make the new alert timestamp wrong on non-UTC hosts, so the patch should not merge as-is.

## What This Changes

This PR adds a cron alert timestamp formatter, threads event times through repeated failure, failure-destination, webhook, and schedule-error notifications, and updates focused cron/gateway tests.

## Best Possible Solution

Land the formatter and plumbing after it shares the cron scheduler's existing timezone fallback for cron schedules without `tz`, while keeping UTC for non-cron schedules or invalid timezone recovery.

## Reproduction Assessment

Yes for source-level behavior: current main omits event-time text from cron failure alerts while finished cron events already carry run timing, and the PR's remaining timezone mismatch is visible from the formatter versus scheduler fallback contract. I did not run a live outage/reconnect reproduction in this read-only review.

## Solution Assessment

No, not as written. The formatter-plus-plumbing approach is the right narrow shape, but the missing-`tz` fallback must match `resolveCronTimezone()` before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the cron scheduler timezone fallback:** `src/cron/alert-time.ts:6`
  - body: For cron jobs created without `--tz`, the scheduler and docs use the gateway host local timezone, but this formatter labels those event times as UTC. On non-UTC hosts, a default cron job that ran at local 09:00 would alert with a different UTC wall-clock time, reintroducing the misleading timestamp problem this PR is trying to fix. Resolve missing cron `tz` the same way `schedule.ts` does, and reserve UTC for non-cron or invalid-timezone fallback cases.
