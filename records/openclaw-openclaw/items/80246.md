## Summary

Keep open: this is a focused implementation for the linked cron alert timestamp feature and it has real behavior proof, but the current diff has a blocking timezone correctness issue for cron jobs created without an explicit `--tz`.

## What This Changes

The PR adds a cron alert timestamp formatter, threads event times through repeated failure alerts, failure-destination messages, and webhook payloads, and updates focused cron/gateway tests.

## Best Possible Solution

Land one narrow implementation that formats cron alert event time with the existing schedule timezone semantics, keeps routing/schema unchanged, and uses UTC only for non-cron or invalid-timezone fallback cases.

## Reproduction Assessment

Yes for the source-level behavior: current main alert builders omit event-time text while cron finished events already carry `runAtMs`. The blocking timezone bug is also source-reproducible because scheduler fallback uses host local timezone and the new formatter uses UTC.

## Solution Assessment

No, not as written. The formatter-plus-plumbing approach is the right shape, but it must share cron's existing no-`tz` timezone fallback before the PR is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the cron scheduler timezone fallback:** `src/cron/alert-time.ts:6`
  - body: For cron jobs created without `--tz`, the scheduler and docs use the gateway host local timezone, but this formatter labels those event times as UTC. On non-UTC hosts, a default cron job that runs at local 09:00 would alert as a different UTC wall-clock time, which reintroduces the misleading timestamp problem this PR is trying to fix. Resolve missing cron `tz` the same way `schedule.ts` does, and reserve UTC for non-cron or invalid-timezone fallbacks.
  - confidence: 0.87
