## Summary

Keep open: the PR addresses a real cron-alert usability gap and has supplied proof, but the current patch still has a blocking timezone mismatch for cron schedules without `tz`. Current cron scheduling and docs use the gateway host timezone in that case, while the new formatter labels those alert times as UTC.

## What This Changes

This PR adds a cron alert event-time formatter, threads `eventTimeMs` through cron failure-alert delivery and webhook payloads, and updates focused cron/gateway tests.

## Best Possible Solution

Land a narrow version that formats alert event time with the existing cron schedule timezone semantics, falls back to UTC only for non-cron or invalid-timezone cases, and keeps routing/schema unchanged.

## Reproduction Assessment

Yes for the source-level behavior: current main alert builders omit event-time text while cron finished events already carry `runAtMs`. The PR defect is also source-reproducible because the new formatter uses UTC where the scheduler/docs use the host timezone for cron schedules without `tz`.

## Solution Assessment

No, not as written. The formatter-plus-plumbing approach is the right shape, but it needs to share the scheduler's existing timezone fallback before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the cron scheduler timezone fallback:** `src/cron/alert-time.ts:6`

On non-UTC hosts, a default cron job would alert with a different wall-clock time than the schedule that actually fired, so resolve missing cron `tz` the same way `schedule.ts` does and reserve UTC for non-cron or invalid-timezone fallback cases.
