## Summary

Keep this PR open. Current main still rejects empty and whitespace-only `write.content`, while the proposed one-flag validator change matches the existing `allowEmpty` mechanism and is covered by focused tests.

## What This Changes

The PR allows empty or whitespace-only content for the `write` tool by setting `allowEmpty: true` on its required `content` parameter group and adding validator regression tests.

## Best Possible Solution

Land this narrow validation change with its focused tests, keeping path, missing-content, and non-string-content rejection strict.

## Reproduction Assessment

Yes. On current main, `REQUIRED_PARAM_GROUPS.write.content` lacks `allowEmpty` and `assertRequiredParams` trims before accepting a string, so `content: ""` or whitespace-only content follows the reported missing-content error path; the linked issue also includes live retry logs.

## Solution Assessment

Yes. Adding `allowEmpty: true` only to the write `content` group is the narrowest maintainable fix because the shared validator already preserves missing-key and non-string rejection, and the pinned upstream write tool accepts string content without a non-empty constraint.
