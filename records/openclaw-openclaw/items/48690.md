## Summary

Close: current main and release v2026.5.7 already contain a safer caller-owned lane task timeout that covers this PR's central bug, and the branch's hard default timeout would now regress long-running caller budgets.

## What This Changes

The PR adds a command-lane task timeout error, a default 15-minute Promise.race timeout in pump(), a taskTimeoutMs enqueue option, and queue tests for timeout/unblock/reset behavior.

## Best Possible Solution

Keep the shipped caller-owned queue timeout and use a narrow follow-up from the canonical stuck-lane issue if any remaining runtime path still needs a timeout budget.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a never-settling enqueueCommandInLane task occupying activeTaskIds and preventing queued work until a timeout or reset releases the lane; current main now has regression coverage for that path.

## Solution Assessment

No. This PR is no longer the best way to solve it because it applies a global 15-minute fallback, while the shipped mainline implementation keeps taskTimeoutMs caller-owned and propagates embedded-run budgets explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve caller timeout budgets:** `src/process/command-queue.ts:142`
  - body: Using `entry.taskTimeoutMs ?? DEFAULT_TASK_TIMEOUT_MS` makes every enqueue without an explicit override fail after 15 minutes, regardless of the caller's existing runtime policy. Agent turns and cron work can legitimately use longer or disabled budgets, so this can report false `CommandLaneTaskTimeoutError`s and let later lane work overlap while the original task is still unwinding; the mainline fix keeps lane timeouts caller-owned instead.
  - confidence: 0.86
