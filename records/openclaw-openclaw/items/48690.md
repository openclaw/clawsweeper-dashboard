## Summary

Current main already implements and ships the command-lane task-timeout recovery, and it does so with caller-owned timeouts rather than this branch's hard global fallback. The branch should close because merging it now would be redundant and less safe than the shipped main implementation.

## What This Changes

The branch adds a command-lane task timeout wrapper, a timeout error type, per-enqueue `taskTimeoutMs`, timeout diagnostics, and regression tests for stuck lane recovery.

## Best Possible Solution

Keep the shipped caller-owned timeout design on main and close this stale branch; any remaining production tuning should happen as narrow follow-up issues against the current implementation.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a one-concurrency lane with a never-settling first task and queued work behind it; current main has a regression test for that exact stuck-lane recovery path.

## Solution Assessment

No for this branch as-is. The current main implementation is the narrower maintainable solution because it keeps timeouts caller-owned and avoids imposing a global 15-minute budget on long-running or disabled-timeout work.
