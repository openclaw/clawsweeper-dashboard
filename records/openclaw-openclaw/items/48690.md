## Summary

Current main and v2026.5.7 already ship the lane-timeout fix via caller-owned `taskTimeoutMs` handling and embedded-run timeout wiring. The remaining branch behavior is an older hard default for every queue entry, which is superseded by the shipped implementation and would be riskier for longer or no-timeout runs.

## What This Changes

This PR adds `CommandLaneTaskTimeoutError`, a default lane task timeout around queued work, `taskTimeoutMs` enqueue plumbing, timeout diagnostics, and command queue timeout tests.

## Best Possible Solution

Keep the shipped caller-owned timeout implementation and close this stale source branch instead of merging its queue-wide fallback.

## Reproduction Assessment

No, not as a current-main failure. The original source-level stuck-lane path is covered by the shipped timeout implementation and a regression test for a never-settling queued task releasing the lane.

## Solution Assessment

No, this PR is no longer the best way to solve the issue. Current main already uses the narrower caller-owned timeout budget, while this branch would reintroduce a hard default that can override longer or disabled caller timeouts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve caller timeout budgets:** `src/process/command-queue.ts:142`
  - body: The fallback `entry.taskTimeoutMs ?? DEFAULT_TASK_TIMEOUT_MS` gives every enqueue without an explicit option a hard 15-minute cutoff. Callers with longer or disabled runtime budgets can then be rejected while their underlying task keeps running, allowing later lane work to overlap with still-mutating work; keep timeout ownership at the caller as current main does.
  - confidence: 0.9
