## What This Changes

The PR adds a command-lane task timeout API and tests so a never-settling active task releases its lane and queued work can continue.

## Best Possible Solution

Keep the current shipped implementation and close this obsolete PR; any remaining raw-queue timeout policy should be tracked as a narrower follow-up, not by landing this branch.

## Reproduction Assessment

I did not run tests because this was a read-only review, but the focused regression test is present in source.

## Solution Assessment

No for the PR branch as-is: its global default queue timeout can override longer or disabled caller runtime budgets. Yes for current main: it keeps the queue timeout caller-owned and threads embedded-run and cron timeout policies through their owning surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve caller runtime budgets before timing out lanes:** `src/process/command-queue.ts:142`
  - body: The fallback `entry.taskTimeoutMs ?? DEFAULT_TASK_TIMEOUT_MS` gives every queue task a hard 15-minute timeout when callers do not opt in. Existing agent and cron callers have longer or disabled timeout policies, so this can reject legitimate work while the underlying task keeps running and allow later lane work to overlap it; thread the caller-owned timeout into the queue instead of applying a global default.
  - confidence: 0.89
