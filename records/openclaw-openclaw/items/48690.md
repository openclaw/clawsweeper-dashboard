## Summary

Current main already implements the useful lane-timeout behavior with a safer caller-owned timeout model, regression coverage, and release provenance; this PR branch is superseded and its blanket default timeout would be risky to merge.

## What This Changes

The PR adds a command-lane task timeout, `CommandLaneTaskTimeoutError`, a `taskTimeoutMs` enqueue option, timeout diagnostics, and queue timeout tests.

## Best Possible Solution

Keep the shipped caller-owned timeout implementation on main and close this superseded branch instead of merging its blanket timeout policy.

## Reproduction Assessment

Yes for the original bug by source inspection: a never-settling queued task on a maxConcurrent=1 lane prevents `completeTask()` and `pump()` from running. Current main has a focused regression test proving a timed task releases the lane and drains queued work.

## Solution Assessment

No for this PR branch; its default 15-minute timeout overrides longer caller budgets. Current main's caller-owned queue timeout plus embedded-run and cron-specific timeout wiring is the narrower maintainable solution.
