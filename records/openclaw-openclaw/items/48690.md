## Summary

Current main implements the PR's core liveness goal with a safer caller-owned timeout design: the command queue can release timed-out tasks, embedded agent runs thread their own runtime budget into the queue, and cron has its own timeout/cleanup wrapper. The PR branch is now obsolete and still carries the earlier hard default-timeout concern, so it should close rather than merge.

## What This Changes

This PR adds a command-lane task timeout, a `CommandLaneTaskTimeoutError`, a `taskTimeoutMs` enqueue option, timeout diagnostics, and command-queue tests for releasing stuck lane work.

## Best Possible Solution

Keep the current-main caller-owned timeout design and close this obsolete branch. The linked liveness report can be closed by a separate issue review once maintainers are comfortable that current main covers the reported production surfaces.

## Reproduction Assessment

Yes. The stuck-lane failure mode is reproduced by a never-settling `enqueueCommandInLane` task with queued work behind it, and current main has a focused regression test proving the timed-out lane drains the queued task.

## Solution Assessment

Yes for current main, not for this branch. The maintainable solution is to keep queue timeout support opt-in and thread surface-specific budgets from embedded agent and cron code; the PR's queue-wide default is less safe because it can override long or disabled caller policies.
