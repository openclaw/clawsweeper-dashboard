## What This Changes

Adds a task-level timeout option to command lane queue execution so a hung lane task can be rejected and queued work can continue.

## Best Possible Solution

## Reproduction Assessment

Yes for the underlying old failure mode as a source-level reproduction: a never-settling task in a maxConcurrent=1 lane blocks queued work until a task timeout releases it. I did not run tests in this read-only review, and current main already has regression coverage for the timeout path.

## Solution Assessment

No for merging this branch as-is. Current main is the better solution because queue timeouts are caller-owned, tied to embedded run budgets, and progress-aware instead of applying this PR's broad default timeout to every queued task.
