## Summary

Current main and release v2026.5.3 already implement the lane-timeout recovery with a safer caller-owned timeout model, so this stale PR should close rather than merge over the shipped behavior.

## What This Changes

This PR adds a per-task timeout option, timeout error, diagnostics, and command-queue tests so hung lane tasks release queued work.

## Best Possible Solution

Keep the shipped caller-owned timeout implementation and close this stale source PR as superseded by current main.

## Reproduction Assessment

No current-main failure reproduction remains. Source inspection shows finite taskTimeoutMs paths now release stuck command-lane tasks, and regression coverage verifies queued work drains after a timeout.

## Solution Assessment

No, this PR is no longer the best fix. Current main uses caller-owned lane timeouts derived from the actual agent run budget, which is safer than the PR's blanket default timeout for every queue entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve caller timeout budgets before timing lane tasks:** `src/process/command-queue.ts:142`
  - body: Using `entry.taskTimeoutMs ?? DEFAULT_TASK_TIMEOUT_MS` gives every enqueue without an explicit override a hard 15-minute cap. Callers can legitimately run longer or disable timeouts, so this can reject the lane promise while the original work keeps mutating state and later work starts behind it; the timeout should be owned by the caller's runtime budget instead of a queue-wide default.
  - confidence: 0.91
