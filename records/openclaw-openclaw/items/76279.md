## Summary

Keep open: the reported gap is real on current main and the PR is pointed at the right runtime hook, but the current patch is not merge-ready because it skips valid engines without `afterTurn` and its new fixtures fail the current assistant-message type contract.

## What This Changes

The PR changes CLI agent compaction so completed CLI turns are fed into the context engine and adds focused tests for that lifecycle.

## Best Possible Solution

Mirror the harness and embedded context-engine lifecycle in the CLI path: ingest the new CLI turn after transcript persistence via `afterTurn` when available, otherwise `ingestBatch` or per-message `ingest`, while preserving current initialization and compaction behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows CLI transcripts are persisted and then passed to a lifecycle that never feeds the completed turn into `afterTurn`, `ingestBatch`, or `ingest`.

## Solution Assessment

No, not as written. Calling the context engine from the CLI path is the right direction, but the patch must preserve engines without `afterTurn`, fix the invalid assistant-message fixtures, and rebase over current initialization/import changes.
