## Summary

Keep open: current main still has the reported CLI context-engine ingestion gap, and this PR is pointed at the right runtime path, but the head is not mergeable as-is because it skips valid engines without `afterTurn` and its new `SessionManager.appendMessage()` fixtures fail test type checking.

## What This Changes

The PR adds post-turn context-engine ingestion to the CLI compaction lifecycle and expands `cli-compaction` tests for normal, no-budget, throwing, and missing-`afterTurn` cases.

## Best Possible Solution

Mirror the harness and embedded lifecycle for CLI turns after transcript persistence: call `afterTurn` when available, otherwise feed only the new messages through `ingestBatch` or per-message `ingest`, while keeping ingestion independent from compaction budget.

## Reproduction Assessment

Yes. Source inspection on current main shows CLI turns are persisted and then routed through `runCliTurnCompactionLifecycle`, which never feeds the completed turn into `afterTurn`, `ingestBatch`, or `ingest`.

## Solution Assessment

No, not as written. Calling the context engine from the CLI path is the right direction, but the patch should preserve valid engines without `afterTurn`, keep current main initialization during rebase, and use valid assistant-message fixtures.
