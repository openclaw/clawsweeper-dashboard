## Summary

Keep this PR open: the reported CLI context-engine ingestion gap is still present on current main and the PR is aimed at the right code path, but the branch still has blocking correctness issues before merge.

## What This Changes

The PR changes CLI agent compaction to feed the just-completed CLI turn into the context engine and adds focused tests for the new ingestion lifecycle.

## Best Possible Solution

Mirror the harness and embedded lifecycle for CLI turns after transcript persistence: call `afterTurn` when available, otherwise feed new messages through `ingestBatch` or per-message `ingest`, while keeping no-budget ingestion separate from compaction.

## Reproduction Assessment

Yes. Source inspection on current main shows CLI transcripts are persisted and then passed to a lifecycle that never feeds the completed turn into `afterTurn`, `ingestBatch`, or `ingest`.

## Solution Assessment

No, not as written. Calling the context engine from the CLI path is the right direction, but the patch should preserve valid engines without `afterTurn`, fix the type-invalid fixtures, and rebase over the current initialization/import changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

  - body: `afterTurn` is optional while `ingest` is required by the context-engine contract. This branch silently skips valid engines that only implement `ingestBatch` or `ingest`, so CLI turns can still remain invisible to those plugins; mirror the embedded/harness fallback path instead.
  - confidence: 0.92

  - body: The new `SessionManager.appendMessage()` fixtures append assistant objects that fail the current message type contract because assistant messages require api/provider/model/usage/stopReason fields. Build the fixtures with a valid assistant-message helper before merge.
  - confidence: 0.9
