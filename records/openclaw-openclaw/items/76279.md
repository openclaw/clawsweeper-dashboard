## Summary

Keep open: the PR addresses a real CLI context-engine gap, but the patch is not merge-ready because it only handles engines with `afterTurn` and omits the required ingest fallback for valid context-engine plugins.

## What This Changes

The PR adds CLI post-turn context-engine `afterTurn` ingestion before the existing CLI compaction logic and adds unit coverage for normal, no-budget, throwing, and no-`afterTurn` cases.

## Best Possible Solution

Update the CLI lifecycle to mirror the embedded context-engine post-turn contract: initialize/resolve the active engine safely, call `afterTurn` when available, otherwise call `ingestBatch` or per-message `ingest` for the just-appended turn, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A focused unit reproduction can use a valid `ContextEngine` with required `ingest()` but no `afterTurn`; the proposed no-`afterTurn` path currently asserts no ingestion, which demonstrates the remaining contract gap without needing a live gateway.

## Solution Assessment

No. Calling `afterTurn` for Lossless-Claw is the right direction, but the maintainable fix should preserve the documented fallback path for all valid context engines and include the required release note.
