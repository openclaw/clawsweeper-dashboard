## Summary

Keep open. Current main partially mitigates the original release-delete race through fs-safe's snapshot-safe release path, but OpenClaw still handles `file_lock_stale` by rechecking and path-removing a same-PID/no-starttime session lock, so the reported race remains source-reproducible and is not superseded by an open canonical fix.

## What This Changes

The issue reports a session write-lock race where async release cleanup can remove a newly acquired lock for the same session.

## Best Possible Solution

Make the session-level stale cleanup snapshot-safe or add a release-in-progress guard around same-session reacquire, then cover unawaited release plus same-session reacquire with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: an unawaited release removes the in-memory held entry, a same-session reacquire sees a same-PID/no-starttime lock as reclaimable after fs-safe fails closed, and the session-level recovery path removes the lock path before retrying.

## Solution Assessment

Yes. The narrow maintainable fix is in the session write-lock primitive or its stale cleanup helper, not in timeout policy, public config, or broader lock-scope redesign.
