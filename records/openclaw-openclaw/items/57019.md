## Summary

Keep open. Current main still has the source-backed stale-recovery path that can remove a lock sidecar by path after stale classification, and https://github.com/openclaw/openclaw/pull/83265 is the open closing PR for this exact issue, so this should stay open until that PR is landed or declined.

## What This Changes

The issue asks OpenClaw to prevent session write-lock stale cleanup from deleting a newly changed or freshly acquired lock file.

## Best Possible Solution

Review and land or decline https://github.com/openclaw/openclaw/pull/83265; if it lands, close this as implemented by snapshot-safe fs-safe stale recovery.

## Reproduction Assessment

Yes, at source level. Current main has SESSION_LOCKS.acquire -> file_lock_stale -> same-PID/no-starttime stale classification -> path-based fs.rm -> retry; I did not run a synthetic race harness.

## Solution Assessment

Yes. The open PR's direction is the narrow owner-boundary fix: keep OpenClaw's stale-owner classification, but let fs-safe perform remove-if-unchanged deletion from the observed lock snapshot.
