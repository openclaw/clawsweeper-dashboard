## Summary

Keep open. Current main removed the original local HELD_LOCKS release implementation, but the session wrapper still has a source-backed stale-recovery path that can classify same-PID/no-starttime locks as reclaimable and then delete the lock by path before retrying.

## What This Changes

The issue reports that session write-lock cleanup can remove a freshly acquired same-session lock through stale or release recovery ordering.

## Best Possible Solution

Make session-level stale cleanup snapshot-safe, preferably by using fs-safe's remove-if-unchanged recovery hook or an equivalent observed-snapshot guard, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level. The path is SESSION_LOCKS.acquire -> file_lock_stale -> same-PID/no-starttime stale classification -> path-based fs.rm -> retry; I did not run a synthetic race harness in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the session write-lock stale-removal path or its fs-safe integration, not in timeout policy, config, or a broader lifecycle redesign.
