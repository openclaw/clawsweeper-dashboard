## Summary

Keep open. Current main still deletes the in-memory held-lock entry before async close/unlink cleanup completes, and the same-PID/no-starttime orphan path can still reclaim and remove the lock file during that gap; the reporter's follow-up lowers practical severity but does not harden the primitive.

## What This Changes

The issue asks OpenClaw to harden session write-lock release so async cleanup cannot delete a newly acquired lock file for the same session.

## Best Possible Solution

Harden the lock primitive by keeping the held entry visible until close/unlink cleanup finishes or adding an explicit releasing guard, then cover unawaited release plus same-session reacquire with a focused regression test.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call release() without awaiting it, reacquire the same session while the old lock file still exists, and use a same-PID payload without starttime so orphan-self detection observes no HELD_LOCKS entry.

## Solution Assessment

Yes. A narrow fix in src/agents/session-write-lock.ts is the best path; it should preserve dead-PID and verified recycled-PID cleanup while preventing release-in-progress cleanup from deleting a fresh lock.

## Repair Work Prompt

Fix the session write-lock release race tracked by https://github.com/openclaw/openclaw/issues/57019. Current main deletes the HELD_LOCKS entry before async close/unlink cleanup completes, so a same-process reacquire can treat the old no-starttime lock file as orphan-self, remove/recreate it, and then have the old cleanup delete the new lock file. Keep the lock visible until cleanup is complete, or add an explicit releasing guard that shouldTreatAsOrphanSelfLock respects. Add a focused regression test in src/agents/session-write-lock.test.ts that starts release() without awaiting it, controls cleanup timing or simulates a same-PID/no-starttime payload, reacquires the same session, and proves old cleanup cannot remove the newly held lock. Keep scope to src/agents/session-write-lock.ts, src/agents/session-write-lock.test.ts, and a single-line CHANGELOG.md Fixes entry for the session reliability fix. Do not implement sessionWriteLock config, broad lock-scope changes, provider failover changes, or transcript-write redesigns. Before editing, inspect overlapping PR #71903 and preserve intended dead-PID/verified recycled-PID recovery semantics.
