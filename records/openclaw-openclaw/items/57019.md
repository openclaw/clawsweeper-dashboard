## Summary

Keep open. Current main has the fs-safe snapshot-safe release mitigation, but the OpenClaw session wrapper still handles `file_lock_stale` by re-reading and path-removing a same-PID/no-starttime lock before retrying, so a same-session concurrent acquire can still remove a fresh lock through the higher-level recovery path.

## What This Changes

The issue reports a session write-lock race where async lock release or stale recovery can delete a newly acquired lock file for the same session.

## Best Possible Solution

Make the session-level stale cleanup snapshot-safe, or add a release-in-progress guard around same-session reacquire, and cover unawaited release plus concurrent same-session reacquire with focused regression tests.

## Reproduction Assessment

Yes, at source level. The current path is SESSION_LOCKS.acquire -> file_lock_stale -> same-PID/no-starttime stale classification -> path-based fs.rm -> retry, which is enough to reproduce the unsafe interleaving without relying on a new product behavior.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the session write-lock primitive or stale-removal helper; changing timeout policy, adding config, or redesigning the whole lock lifecycle would be broader than this bug requires.

## Repair Work Prompt

Fix the session write-lock stale recovery race reported at https://github.com/openclaw/openclaw/issues/57019. Current main uses @openclaw/fs-safe for snapshot-safe release, but src/agents/session-write-lock.ts still handles file_lock_stale by re-inspecting and path-removing lockPath in removeReportedStaleLockIfStillStale. Make that recovery deletion snapshot-safe or add an in-process release/reclaim guard so a same-session concurrent acquire cannot delete a lock file that changed after inspection. Preserve existing stale/dead-PID/recycled-PID/non-OpenClaw-owner/orphan-self recovery semantics, avoid adding public config or changing lock timeout policy, and do not change package dependencies unless a maintainer explicitly approves. Add focused regression coverage in src/agents/session-write-lock.test.ts for unawaited release plus concurrent same-session reacquire/no-starttime behavior.
