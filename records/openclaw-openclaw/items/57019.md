## Summary

Keep open. Current main and the latest release still contain the wrapper-level stale-lock retry path that re-reads a reported stale lock and removes it by path, and an open closing PR already owns the focused fix.

## What This Changes

The issue reports a session write-lock race where stale cleanup can remove a newer lock file and allow concurrent writes to the same session transcript.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83265, or an equivalent fs-safe snapshot-guarded stale recovery, then close this issue as implemented after merge and release proof.

## Reproduction Assessment

Yes, source-level. I did not run a live race in this read-only pass, but current main still has the `file_lock_stale` catch, stale re-read, path unlink, and retry path, with production comments reinforcing the failure mode.

## Solution Assessment

Yes. Moving stale sidecar deletion to fs-safe's existing `remove-if-unchanged` recovery is the narrow owner-boundary fix because fs-safe owns the observed lock snapshot needed to decide whether unlinking is still safe.
