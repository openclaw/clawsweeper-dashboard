## Summary

Keep open. Current main has the fs-safe snapshot-safe release mitigation, but the OpenClaw session wrapper still catches `file_lock_stale`, re-reads the lock path, and removes it by path before retrying, so the remaining race is still source-reproducible in the session-level recovery path.

## What This Changes

The issue reports a session write-lock race where asynchronous release or stale recovery can delete a newly acquired same-session lock file.

## Best Possible Solution

Make the session-level stale cleanup snapshot-safe, preferably by using fs-safe’s approved `remove-if-unchanged` path or an equivalent observed-snapshot guard, and cover same-session stale retry with focused regression tests.

## Reproduction Assessment

Yes, at source level. The current path is `SESSION_LOCKS.acquire` -> `file_lock_stale` -> same-PID/no-starttime stale classification -> path-based `fs.rm` -> retry; I did not run a live race harness in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the session write-lock stale-removal path or its fs-safe integration, not in timeout policy, config, or a broader lock lifecycle redesign.
