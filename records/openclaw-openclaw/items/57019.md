## Summary

Keep open. Current main still deletes the in-memory held-lock entry before async close/unlink cleanup completes, and the same-PID/no-starttime contended acquire path can still force-reclaim that lock file during the gap.

## What This Changes

The issue asks to prevent `releaseHeldLock` from deleting a new same-session lock file after an async release/reacquire race.

## Best Possible Solution

Harden the session write-lock primitive so a lock remains visible, or explicitly marked releasing, until close/unlink cleanup has finished, with focused regression coverage for release-in-progress reacquisition.

## Reproduction Assessment

Yes, at source level. Call `release()` without awaiting it, reacquire the same session while the old lock file still exists, and use a same-PID payload without starttime so orphan-self detection sees no `HELD_LOCKS` entry.

## Solution Assessment

Yes. A narrow fix in `src/agents/session-write-lock.ts` is the best path; broader lock timeout, queueing, or scope changes belong to separate issues such as #13744.

## Repair Work Prompt

Fix openclaw/openclaw#57019. Current main's `releaseHeldLock` deletes `HELD_LOCKS` before async `handle.close()`/`fs.rm()` cleanup, allowing an unawaited release plus immediate reacquire of the same session with a same-PID/no-starttime payload to reclaim the old lock and then have the old release remove the new lock. Update `src/agents/session-write-lock.ts` to keep release-in-progress state visible until cleanup completes, preferably by moving `HELD_LOCKS.delete` until after close/unlink or by adding an explicit releasing guard. Preserve reentrant counts, forced test cleanup, watchdog shutdown, dead-PID/recycled-PID cleanup, `cleanStaleLockFiles` behavior, and existing lock timeout defaults. Add a focused regression in `src/agents/session-write-lock.test.ts` that controls release cleanup timing and proves a fresh reacquired lock file is not deleted by the prior release. Add a single-line `CHANGELOG.md` Fixes entry for the user-facing transcript corruption risk. Check related #75822 and #71903, but do not revert their self-orphan behavior or add new config knobs.
