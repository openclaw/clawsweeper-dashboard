## Summary

Keep open. Current main still removes the in-memory held-lock entry before async close/unlink cleanup completes, and contended acquisition can still reclaim same-PID lock files without starttime during that window; the reporter’s follow-up lowers practical exposure but does not remove the primitive-level race.

## What This Changes

The issue requests hardening the session write-lock release path so async cleanup cannot delete a newly acquired lock file for the same session.

## Best Possible Solution

Harden the lock primitive by keeping the held entry visible until close/unlink cleanup finishes or by adding an explicit releasing guard, with focused coverage for unawaited release plus same-session reacquire.

## Reproduction Assessment

Yes, at source level. Call release() without awaiting it, reacquire the same session while the old lock file still exists, and use a same-PID payload without starttime so orphan-self detection observes no HELD_LOCKS entry.

## Solution Assessment

Yes. A narrow change in src/agents/session-write-lock.ts is the maintainable fix; broader queueing/config changes belong to separate lock-contention requests, not this race.

## Repair Work Prompt

Fix the session write-lock release race tracked by https://github.com/openclaw/openclaw/issues/57019. In src/agents/session-write-lock.ts, prevent release-in-progress cleanup from deleting a newly acquired lock for the same normalized session. Prefer either keeping the HELD_LOCKS entry visible until handle.close() and fs.rm() complete, or adding an explicit releasing guard that shouldTreatAsOrphanSelfLock respects. Preserve dead-PID, recycled-PID, payload-less/malformed stale-lock cleanup, and the recent untracked self-owned lock reclaim behavior from PR #75822. Add a focused regression test in src/agents/session-write-lock.test.ts for calling release() without awaiting it, then immediately contending/reacquiring the same session with a no-starttime same-PID payload so the old cleanup cannot remove the fresh lock. Add a single-line CHANGELOG.md fix entry if this is user-facing. Do not add new config options, change lock timeout policy, or broaden session queue behavior.
