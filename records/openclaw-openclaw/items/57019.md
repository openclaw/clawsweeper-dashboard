## Summary

Keep open. The original release-ordering concern is now partly mitigated by fs-safe snapshot cleanup, but current main and the latest release still have an unsafe session stale-recovery path that revalidates a stale lock and then deletes the lock file by path; the open closing pull request is the focused fix path.

## What This Changes

The issue requests a fix for session write-lock stale/release races so cleanup cannot remove a changed or newly-acquired `.jsonl.lock` and allow concurrent transcript writes.

## Best Possible Solution

Keep stale-owner classification in `src/agents/session-write-lock.ts`, but delegate stale sidecar removal to fs-safe's unchanged-snapshot contract and close this after the linked fix lands and is verified.

## Reproduction Assessment

Yes, at source level. Current main still has the `file_lock_stale` catch -> stale re-read -> path-based `fs.rm` -> retry flow, and issue comments add production reports on `v2026.5.18` across Linux/WSL2 and macOS.

## Solution Assessment

Yes. The linked pull request is the narrowest maintainable fix because it preserves OpenClaw's stale-owner policy while moving deletion to fs-safe's unchanged-snapshot recovery instead of duplicating path-only cleanup in the wrapper.
