## Label Justifications

- `P1`: The report had multiple production confirmations of session-lock failures affecting cron and isolated-session workflows.
- `impact:data-loss`: The reported race could remove mutual-exclusion protection and allow concurrent writes to session JSONL transcripts.
- `impact:session-state`: The affected surface is the session write-lock mechanism that protects transcript and agent session consistency.

## Summary

Close as implemented. Current main and stable v2026.5.22 now route session stale-lock removal through fs-safe's snapshot-guarded `remove-if-unchanged` contract instead of re-reading a reported stale lock and deleting the path directly; the still-open linked fix PR is superseded by the shipped implementation.

## What This Changes

The issue asks to prevent session stale-lock release or recovery from deleting a newly acquired `.jsonl.lock` file.

## Best Possible Solution

Keep the shipped fs-safe snapshot-guarded recovery in core and close the superseded open fix PR separately if maintainers agree it no longer carries unique work.

## Reproduction Assessment

Yes for the affected releases: v2026.5.20 source still had the `file_lock_stale` re-read plus path-only `fs.rm()` path, and production comments cover v2026.5.18 and v2026.5.19. Current main no longer has that path.

## Solution Assessment

Yes. Delegating stale sidecar deletion to fs-safe's `remove-if-unchanged` path is the narrow owner-boundary fix because the lock manager owns the observed sidecar snapshot.
