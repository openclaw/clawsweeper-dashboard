## Summary

Keep open. Current main moved session locks onto the fs-safe manager, and that manager now protects its own release cleanup from deleting a changed lock file, but OpenClaw still path-removes a reported stale same-PID/no-starttime session lock after the manager fails closed, leaving the release-in-progress/fresh-acquire race source-reproducible.

## What This Changes

The issue asks to prevent session write-lock release cleanup from deleting a newly acquired lock during same-session reacquisition.

## Best Possible Solution

Add a narrow release-in-progress guard or equivalent unchanged-lock protection around the session-level stale cleanup, then cover unawaited release plus same-session reacquire with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence interleaving: call release() without awaiting it, reacquire the same session while the old no-starttime same-PID lock still exists, let fs-safe fail closed, then have removeReportedStaleLockIfStillStale path-remove after the old release or another actor has made room for a fresh lock.

## Solution Assessment

Yes. The narrow maintainable fix is still in the session write-lock primitive: keep release-in-progress locks out of orphan-self reclamation or make the session-level stale removal snapshot-safe, without changing lock timeout policy or adding configuration.
