## Summary

Keep open. Current main still removes the in-process session lock entry before async close/unlink cleanup finishes, and the same-PID/no-starttime orphan path can still turn that gap into lock-file removal. The reporter's follow-up and representative callers lower practical severity, but the lock primitive itself is still not hardened.

## What This Changes

The issue requests hardening session write-lock release so async cleanup cannot delete a lock file that was reacquired during release-in-progress.

## Best Possible Solution

Keep the issue open and harden the lock primitive itself: either keep the held entry visible until close/unlink cleanup completes, or add an explicit releasing guard that same-PID orphan detection respects. Cover it with a focused regression test for unawaited release plus same-session reacquire under a no-starttime payload.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: call release() without awaiting it, reacquire the same session while the old lock file still exists, and use a same-PID payload without starttime so shouldTreatAsOrphanSelfLock observes no HELD_LOCKS entry.

## Solution Assessment

Yes, a narrow session-write-lock fix is the best path. Moving HELD_LOCKS.delete until after cleanup, or adding a releasing guard, addresses the primitive directly without broad lock redesign or the larger timeout/config work tracked elsewhere.

## Repair Work Prompt

Fix the session write-lock release race tracked by https://github.com/openclaw/openclaw/issues/57019. Current main deletes the HELD_LOCKS entry before async close/unlink cleanup completes, so a same-process reacquire can treat the old no-starttime lock file as orphan-self and remove/recreate it before the old cleanup deletes the new file. Keep the lock visible until cleanup is complete, or add an explicit releasing guard that shouldTreatAsOrphanSelfLock respects. Add a focused regression test in src/agents/session-write-lock.test.ts that starts release() without awaiting it, simulates a same-PID payload without starttime or otherwise controls cleanup timing, reacquires the same session, and proves the old cleanup cannot remove the newly held lock. Keep the change scoped to src/agents/session-write-lock.ts and its colocated test; do not implement sessionWriteLock config, broad lock-scope changes, provider failover changes, or transcript-write guard redesigns. Before editing, inspect overlapping PRs #71903 and #44510 for conflicts, but do not fold their broader behavior into this fix. No changelog entry is needed unless maintainers request one for user-visible lock behavior.
