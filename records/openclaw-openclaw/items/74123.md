## Summary

Keep open. Current main still has the media URI corruption paths, and two open implementation PRs already reference this issue with closing syntax, so this should remain open until maintainers land or consolidate that work.

## What This Changes

The issue asks OpenClaw to resolve `media://inbound/...` image references to managed inbound media instead of treating the URI as a workspace-relative `media:/...` path.

## Best Possible Solution

Land or replace the linked implementation work with one focused fix that preserves inbound media ID validation and sandbox path boundaries, prevents scheme-bearing values from being joined under `workspaceDir`, handles sandbox bridge ordering before direct path resolution, and adds regression coverage for the image tool, sandbox resolver, and shared media loader.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `media://inbound/example.png` reaches path resolution paths that treat it as a relative string, and Node resolves that shape to `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes. Keeping the issue open while #74186 and #74278 are reviewed or consolidated is the narrowest maintainable path; a separate repair job would duplicate already-open implementation work.
