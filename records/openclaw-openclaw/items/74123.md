## Summary

Keep open. Current main still contains the `media://` URI corruption route in sandbox media resolution and a related shared-loader workspace-join route; because open implementation PRs #74186 and #74278 reference this issue with closing syntax, the issue should remain open until one consolidated fix lands or those PRs are resolved.

## What This Changes

The issue asks OpenClaw to resolve `media://inbound/<image-file>` references to managed inbound media instead of corrupting them into workspace-relative `media:/inbound/...` paths.

## Best Possible Solution

Land or replace the existing implementation PRs with one focused fix that handles canonical `media://inbound/<id>` before sandbox bridge path resolution, prevents any scheme-bearing value from being joined under `workspaceDir`, preserves inbound media ID validation and sandbox path boundaries, and adds regression coverage for both surfaces.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: sandboxed image input reaches `resolveSandboxedBridgeMediaPath`, direct bridge resolution handles `media://inbound/example.png` as a relative path, and Node path resolution produces a cwd-relative path containing `media:/inbound/example.png`.

## Solution Assessment

Yes. The best current path is to keep this issue open while maintainers review or consolidate #74186 and #74278; starting a separate repair lane would duplicate already-open implementation work.
