## Summary

Keep open. Current main still has the reported `media://` to `media:/` workspace-path corruption path in both image-tool and sandbox media resolution, and two open implementation PRs already reference this issue with closing syntax, so this should stay open until maintainers land or consolidate the fix.

## What This Changes

The issue asks for inbound `media://inbound/...` image references to resolve to OpenClaw-managed media files instead of malformed workspace paths like `media:/inbound/...`.

## Best Possible Solution

Land or replace the linked implementation work with one focused fix that preserves inbound media ID validation, prevents scheme-bearing values from being joined under `workspaceDir`, handles sandbox bridge ordering before direct path resolution, and adds regression tests for image-tool, sandbox resolver, and shared media-loader paths.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `media://inbound/example.png` reaches path resolution branches that treat it as a relative string, and Node resolves that shape to `/workspace/media:/inbound/example.png`, matching the reported error.

## Solution Assessment

Yes. Keeping this issue open while maintainers review or consolidate #74186 and #74278 is the narrowest maintainable path; a separate repair job would duplicate already-open closing PRs.
