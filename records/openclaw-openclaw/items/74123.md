## Summary

Keep open. Current main still has the `media://` path-corruption path in the sandbox resolver and shared media loader, and open implementation PRs already reference this issue with closing syntax, so the issue should stay open until one fix lands or those PRs are resolved.

## What This Changes

The issue requests that `media://inbound/<image>` image references resolve to OpenClaw-managed inbound media instead of being joined under the workspace as a malformed `media:/inbound` path.

## Best Possible Solution

Land or replace the existing implementation PRs with one focused fix that handles canonical `media://inbound/<id>` before sandbox bridge path resolution, prevents scheme-bearing values from being joined under `workspaceDir`, and adds regression coverage for both the sandbox resolver and stranded shared-loader path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed image reference reaches `resolveSandboxedBridgeMediaPath`, direct bridge resolution treats `media://inbound/example.png` as relative input, and Node resolves it to `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes. Keeping this issue open while maintainers review or consolidate #74186 and #74278 is the best current path; a new separate repair job would duplicate already-open implementation work.
