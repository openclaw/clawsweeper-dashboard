## Summary

Keep open. Current main still has the reported `media://` path-corruption path in the sandbox resolver and shared media loader, and two open implementation PRs already reference this issue with closing syntax, so the issue should remain open until one consolidated fix lands.

## What This Changes

The issue asks for `media://inbound/<image>` image references to resolve to OpenClaw-managed media files instead of being joined under the workspace as a malformed `media:/...` path.

## Best Possible Solution

Land or replace the open implementation PRs with one focused fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, prevents scheme-bearing values from being joined under `workspaceDir`, and adds regression coverage in `sandbox-media-paths.test.ts` and `web-media.test.ts`.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed image reference reaches `resolveSandboxedBridgeMediaPath`, direct bridge resolution treats `media://inbound/example.png` as relative input, and Node resolves it to `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes. Reviewing or consolidating #74186 and #74278 is the best current path because they target the two concrete failure seams without requiring a broader media API redesign.
