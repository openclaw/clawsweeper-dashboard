## Summary

Keep open. Current main still contains the resolver ordering that can turn a canonical `media://inbound/...` image reference into a workspace-relative `media:/inbound/...` path, and open implementation PRs #74186 and #74278 already reference this issue with closing syntax, so the issue should remain open until one fix lands or those PRs are resolved.

## What This Changes

The issue requests that `media://inbound/...` image references resolve to OpenClaw-managed inbound media instead of being treated as malformed workspace-relative paths.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge or workspace-relative path resolution, preserves inbound media validation and sandbox path boundaries, and adds focused regression coverage for sandboxed and workspace-backed image-tool inputs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a sandboxed or workspace-backed image-tool call with `media://inbound/example.png`; current source routes the string through path resolution, and Node resolves it as `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes. The narrow maintainable solution is to handle scheme-bearing managed media before bridge/workspace path resolution, with media-loader defense in depth; keeping the issue open while #74186 and #74278 are reviewed is the right issue workflow.
