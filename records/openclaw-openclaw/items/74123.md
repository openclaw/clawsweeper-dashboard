## Summary

Keep open. Current main still contains the reported `media://` URI corruption paths, and open implementation PRs #74186 and #74278 already reference this issue with closing syntax, so the issue should stay open until maintainers land or consolidate the fix.

## What This Changes

Treat `media://inbound/<image>` references as OpenClaw-managed inbound media so image understanding uses the stored file instead of resolving the URI as a workspace-relative path.

## Best Possible Solution

Land or consolidate a focused fix that treats `media://` as a media-store URI before workspace or sandbox path resolution, preserves inbound media validation and local-root boundaries, and adds regression coverage for the image tool, `loadWebMedia`, and sandbox resolver paths.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main can feed `media://inbound/example.png` into `path.resolve(workspaceDir, ...)`, and Node produces the malformed `/workspace/media:/inbound/example.png` path reported by the user.

## Solution Assessment

Yes for triage. Keeping this issue open while maintainers review or consolidate #74186 and #74278 is the narrowest maintainable path because those open PRs already target the affected resolver surfaces.
