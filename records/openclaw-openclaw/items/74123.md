## Summary

Keep open. Current main still has the reported media URI corruption path: the image tool can workspace-resolve `media://inbound/...` as a relative path, the sandbox bridge resolves raw `media://` input before the inbound fallback can run, and linked open PRs #74186 and #74278 already reference this issue with closing syntax.

## What This Changes

The issue asks for inbound `media://inbound/...` image references to resolve to OpenClaw-managed media files instead of malformed workspace paths such as `media:/inbound/...`.

## Best Possible Solution

Land or consolidate the linked implementation work into one focused fix that treats `media://` as a media-store URI before workspace or sandbox path resolution, preserves inbound media ID validation, and adds regression coverage for image-tool, sandbox resolver, and shared `loadWebMedia` paths.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current path resolution can apply workspace or sandbox `path.resolve` to `media://inbound/example.png`, and Node resolves that shape to a `media:/inbound/...` workspace child path before media loading.

## Solution Assessment

Yes. Keeping this issue open while maintainers review, land, or consolidate #74186 and #74278 is the narrowest maintainable path; a separate ClawSweeper repair job would duplicate already-open closing PRs.
