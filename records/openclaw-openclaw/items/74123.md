## Summary

Keep open. Current `main` still contains the reported URI corruption path, and two open implementation PRs already reference this issue with closing syntax, so maintainers should review or consolidate that work instead of closing this issue now.

## What This Changes

The issue asks OpenClaw to resolve `media://inbound/<image>` references as managed media files instead of workspace-relative paths like `media:/inbound/<image>`.

## Best Possible Solution

Land or consolidate the linked implementation work into one focused fix that treats `media://` as a media-store URI before image-tool workspace resolution and sandbox bridge resolution, preserves inbound media ID validation and sandbox boundaries, and adds regression coverage at the image tool, sandbox resolver, and shared `loadWebMedia` seams.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current `main` can feed `media://inbound/example.png` into workspace or sandbox `path.resolve`, and Node turns that into `/workspace/media:/inbound/example.png` before media loading.

## Solution Assessment

Yes. Keeping this issue open while maintainers review or consolidate #74186 and #74278 is the narrowest maintainable path because both PRs already target the same user-visible failure with closing syntax.
