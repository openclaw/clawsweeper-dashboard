## Summary

This PR should stay open: it addresses the reported sidecar status gap, but the new OAuth eligibility short-circuit can mark valid Codex CLI bootstrap-backed profiles as missing before the existing bootstrap resolver runs.

## What This Changes

The PR preserves legacy Codex OAuth `oauthRef` metadata, reports unresolved sidecar OAuth profiles as missing with `unresolved_ref`, forwards that reason through status payloads, and refreshes gateway auth status after doctor auth-profile repairs.

## Best Possible Solution

Revise auth health so legacy `oauthRef` profiles fail closed only after existing external CLI/bootstrap resolution has had a chance to supply usable credentials, then keep the gateway refresh piece and split any broader linked-issue follow-ups if needed.

## Reproduction Assessment

Yes by source inspection and existing coverage: current main documents tokenless Codex profiles using Codex CLI OAuth, while the PR returns missing before that resolver runs. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

No, not as written: the sidecar classification is the right direction, but it needs to preserve the existing external CLI bootstrap contract before declaring OAuth credentials missing.
