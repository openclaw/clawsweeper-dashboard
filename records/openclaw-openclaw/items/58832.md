## Summary

Keep open. Current main still routes top-level Slack DMs with `replyToMode="all"` through the message timestamp as a routed thread id, and the test suite still asserts a `:thread:` suffix for that case. The report remains a narrow, actionable Slack plugin bug rather than a stale, duplicate, or already-implemented cleanup.

## What This Changes

Requests Slack top-level direct messages with `dmScope: "per-channel-peer"` and reply threading enabled to keep the base direct-message session key instead of appending `:thread:<message ts>`.

## Best Possible Solution

Fix Slack plugin routing so top-level direct messages never use their own timestamp as an inbound session-key thread id, while actual Slack thread replies still use the parent `thread_ts` and all existing reply-delivery metadata remains available where needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the top-level DM test with `replyToMode="all"` expects a `:thread:500.000` session suffix, and the routing code deterministically derives that suffix from the message timestamp.

## Solution Assessment

Yes. The narrow maintainable solution is in Slack routing and Slack tests, not core session-key construction: suppress the direct-message `autoThreadId` for top-level DMs while preserving real thread replies and reply metadata.
