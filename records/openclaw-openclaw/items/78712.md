## Summary

Keep open. Current main and shipped v2026.5.7 still restore full default operator scopes for shared-secret callers on trusted-operator plugin routes, current tests assert admin-capable behavior, and an open linked fix PR should be reviewed before this issue closes.

## What This Changes

The issue asks OpenClaw to stop trusted-operator plugin HTTP routes from granting shared-secret callers implicit admin-capable runtime scopes.

## Best Possible Solution

Review and land a narrow gateway fix, likely through https://github.com/openclaw/openclaw/pull/78759 or a replacement, that removes implicit admin scope for shared-secret plugin routes and aligns tests/docs around the trusted-operator contract.

## Reproduction Assessment

Yes. Current-main source inspection shows token/password auth sets `trustDeclaredOperatorScopes: false`, and the trusted-operator resolver maps that to `CLI_DEFAULT_OPERATOR_SCOPES` containing `operator.admin`; current tests assert the same path.

## Solution Assessment

Unclear. The report points at the right code path, but the safest fix needs maintainer/security judgment because current code has an explicit trusted-operator opt-in while plugin-route docs describe conservative shared-secret scopes.
