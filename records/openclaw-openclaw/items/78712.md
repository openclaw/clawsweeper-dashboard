## Summary

Keep open: current main and the latest release still grant default operator scopes, including admin, to shared-secret callers on `trusted-operator` plugin HTTP routes, while docs and newer admin-route surfaces show this is a security/product contract that needs explicit maintainer decision rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to stop shared-secret callers from receiving implicit admin-capable runtime scopes on `trusted-operator` plugin HTTP routes.

## Best Possible Solution

Maintainers should settle the `trusted-operator` plugin-route contract first, then either document the shipped full-operator behavior clearly or land a security-reviewed compatibility-aware change with tests, docs, and real Gateway proof.

## Reproduction Assessment

Yes, from source inspection: token/password auth sets `trustDeclaredOperatorScopes: false`, and `trusted-operator` plugin route scope resolution converts that path into default scopes containing `operator.admin`. I did not run a live Gateway repro because this is a read-only review.

## Solution Assessment

Unclear. The narrow code change in the closed PRs is plausible, but the best fix depends on a maintainer/security decision because current shipped code, tests, and admin HTTP RPC docs treat some shared-secret plugin HTTP routes as full trusted-operator surfaces.
