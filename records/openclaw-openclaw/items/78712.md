## Summary

Keep open. Current main still gives shared-secret callers full default operator scopes on `trusted-operator` plugin routes, while plugin-route docs describe shared-secret runtime scopes as write-only; the issue also has an open fixing PR with closing syntax, so closing before merge would be premature.

## What This Changes

The issue requests preventing shared-secret bearer callers on `trusted-operator` plugin HTTP routes from receiving implicit admin runtime scopes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78759 or an equivalent corrected fix that aligns the resolver, dispatch tests, docs, and changelog around an explicit non-implicit-admin contract for shared-secret plugin routes.

## Reproduction Assessment

Yes. Current-main source inspection shows the shared-secret trusted-operator path returns the default scope set, and current tests assert `operator.admin` is present; I did not run tests because this was a read-only review.

## Solution Assessment

Yes on direction, unclear on the exact mergeable patch. Narrowing shared-secret trusted-operator scopes matches the documented plugin-route contract, but the final fix must update dispatch-level expectations and get maintainer security review before merge.
