## Summary

Keep open. Current main still restores full default operator scopes for shared-secret callers on `trusted-operator` plugin routes, and current tests assert admin-capable behavior; this needs security/maintainer handling rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop `trusted-operator` plugin HTTP routes from granting default admin-capable runtime scopes to shared-secret bearer callers with narrow or absent scope headers.

## Best Possible Solution

Align the `trusted-operator` route contract, docs, and tests so shared-secret bearer callers cannot receive implicit admin runtime scopes without a clear audited route-level policy or identity-bearing admin scope.

## Reproduction Assessment

Yes. Current-main source inspection and existing tests show a shared-secret bearer request to a `trusted-operator` plugin route receives default scopes containing `operator.admin`; I did not run a live gateway because this was a read-only review.

## Solution Assessment

Unclear. The report points at the right code path, but the safest fix direction needs maintainer/secops judgment because current tests and bundled routes intentionally use the `trusted-operator` escape hatch.
