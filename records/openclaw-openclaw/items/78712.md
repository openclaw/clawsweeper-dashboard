## Summary

Keep open: current main source and tests support the reported shared-secret-to-admin runtime scope path, while plugin-route docs and the earlier scope-hardening changelog describe a write-only fallback. Because the `trusted-operator` surface is intentional and used by bundled routes, this needs maintainer/security policy judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent gateway-auth plugin HTTP routes using `gatewayRuntimeScopeSurface: "trusted-operator"` from giving shared-secret bearer callers implicit `operator.admin` runtime scopes.

## Best Possible Solution

Decide the trusted-operator plugin-route contract explicitly: either narrow shared-secret plugin routes back to write/default-deny with regression tests and a changelog entry, or document the opt-in admin semantics and add clearer auditing guidance for route authors.

## Reproduction Assessment

Yes. Source inspection at current main shows token/password auth sets declared scopes untrusted and the trusted-operator resolver maps that case to the full default operator scope set; existing unit and integration tests encode the same path, though I did not run them because this review is read-only.

## Solution Assessment

Unclear. A narrow code fix is feasible, but the current behavior is an explicit, tested `trusted-operator` route mode used by bundled plugins, so the best solution depends on maintainer security policy for that surface.
