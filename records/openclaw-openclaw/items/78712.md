## Summary

Keep open. Current main and v2026.5.7 still contain the shared-secret trusted-operator scope-widening path, and an open PR already closes this issue, so this should remain open until that PR is reviewed, proven, and either landed or rejected.

## What This Changes

The issue reports that shared-secret bearer callers on plugin HTTP routes using `gatewayRuntimeScopeSurface: "trusted-operator"` receive default operator scopes, including `operator.admin`, instead of a conservative route runtime scope.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/78759 after aligning resolver behavior, route-level tests, docs or release notes, and proof so shared-secret trusted-operator plugin routes cannot receive implicit admin scopes.

## Reproduction Assessment

Yes. Source inspection on current main shows token/password auth sets `trustDeclaredOperatorScopes` false and trusted-operator route resolution returns `CLI_DEFAULT_OPERATOR_SCOPES` containing `operator.admin`; existing tests assert the same path.

## Solution Assessment

Unclear for the final patch. The linked PR’s central resolver direction is plausible, but it still needs security review, route-level test alignment, and contributor real behavior proof before this issue is solved.
