## Summary

Keep open: current main and the latest release still contain the reported trusted-operator shared-secret admin-scope behavior, and the related fix PRs closed unmerged. This needs maintainer security/product contract review because current tests, docs, and bundled trusted-operator routes disagree on whether shared-secret plugin routes should be write-only or full operator surfaces.

## What This Changes

The issue reports that shared-secret callers on trusted-operator plugin HTTP routes receive admin-capable runtime scopes instead of remaining least-privilege.

## Best Possible Solution

Maintainers should settle the trusted-operator shared-secret contract, then align source, tests, and docs in one change: either explicitly document these routes as full operator surfaces or make plugin trusted-operator admin behavior require an identity-bearing/admin-scoped path with upgrade guidance.

## Reproduction Assessment

Yes. Source inspection on current main shows shared-secret auth sets `trustDeclaredOperatorScopes: false`, and trusted-operator plugin route scope resolution then returns the default operator scope set including `operator.admin`; current tests assert the same path.

## Solution Assessment

No single mechanical fix is clearly best yet. Removing the fallback is plausible, but the safer maintainable solution needs maintainer security/product confirmation because shipped routes and docs encode competing expectations.
