## Summary

This PR is useful and has real terminal proof, but the current head is still not mergeable because the guard only blocks remote node-host configs when a saved node token exists and it duplicates narrower loopback logic than the existing gateway helper.

## What This Changes

Adds a `models auth login` preflight for remote node hosts, one regression test, and a changelog entry.

## Best Possible Solution

Land or replace this with a focused guard that blocks all non-loopback node-host auth-profile writes using shared loopback semantics, with regression coverage for tokenless remote configs and local loopback forms.

## Reproduction Assessment

Yes. Source inspection on current main shows `modelsAuthLoginCommand` persists profiles to the local agent store and does not inspect node-host config, while `runNodeHost` can save a remote gateway target; I did not run a live OAuth flow.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow fix, but the current PR should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the remote-node guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from runtime config or environment. With the `!nodeHostConfig?.token` return here, a tokenless remote node-host config can still run provider auth and write local profiles, leaving the reported credential-consumption path reachable.
  - confidence: 0.92
- **[P2] Use the shared loopback host check:** `src/commands/models/auth.ts:708-712`
  - body: This local helper only accepts empty, `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already handles the repo's local gateway forms such as `localhost.`, 127/8, bracketed IPv6, and IPv4-mapped loopback. Reuse the shared helper so valid local node-host configs are not blocked.
  - confidence: 0.86
