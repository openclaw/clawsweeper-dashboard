## Summary

Keep this PR open for changes. The direction and terminal proof are useful, but the current guard still lets tokenless remote node-host configs proceed into local auth-profile writes, and it duplicates narrower loopback parsing than the shared gateway helper.

## What This Changes

This PR adds a pre-auth remote node-host guard and regression test for `openclaw models auth login`, updates the changelog, and removes an unused daemon inspect helper.

## Best Possible Solution

Merge a repaired pre-auth guard that blocks all non-loopback node-host auth-login local writes before provider auth, treats missing gateway host as local, reuses shared loopback detection, and keeps focused tokenless and loopback regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a `node.json` with a non-loopback `gateway.host` and no `token`, then `modelsAuthLoginCommand` returns past the new guard and current auth login persists provider profiles through the local `agentDir`.

## Solution Assessment

No. Blocking local writes is a reasonable short-term direction, but the current PR ties remote-node detection to token presence and duplicates loopback parsing instead of using the existing gateway contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block tokenless remote node hosts:** `src/commands/models/auth.ts:728`
  - body: The guard returns whenever `nodeHostConfig.token` is absent, but node-host config can legitimately contain a non-loopback `gateway` without a saved token; `runNodeHost` persists the gateway before resolving credentials. That leaves the reported local auth-profile write path reachable for tokenless remote node hosts, so decide remote-ness from the non-loopback gateway config instead and cover that case.
  - confidence: 0.93
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:719-723`
  - body: This helper only allows `localhost`, `127.0.0.1`, and `::1`, while the gateway helper also accepts forms such as `localhost.`, bracketed IPv6, mapped IPv6, and the broader 127/8 loopback range. A local node-host setup using those supported forms would be blocked after upgrade, so reuse the shared gateway loopback helper and handle a missing host as local.
  - confidence: 0.88
