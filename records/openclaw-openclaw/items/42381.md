## Summary

Keep open: this PR addresses a real credential-write bug, but the current head still leaves tokenless remote node-host configs unblocked and duplicates narrower loopback logic than the shared gateway helper.

## What This Changes

The PR adds an early `models auth login` guard for node-host configs, regression coverage, and a changelog entry for blocking remote node credential writes.

## Best Possible Solution

Land a focused guard that blocks non-loopback `nodeHostConfig.gateway.host` regardless of optional token, reuses the shared gateway loopback classifier, and covers tokenless remote plus local loopback allow cases.

## Reproduction Assessment

Yes. Source inspection shows current main writes provider auth profiles to the local `agentDir`, while node-host config can point at a remote gateway and resolve credentials from env/config without `node.json.token`.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow direction, but this PR should key off the gateway host without requiring `token` and should reuse the shared loopback classifier.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P1] Block tokenless remote-node auth writes:** `src/commands/models/auth.ts:620`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the gateway target before resolving credentials from config/env. Returning when the token is absent lets tokenless remote node hosts still perform the local auth write this guard is meant to block.
  - confidence: 0.93
- **[P2] Reuse the shared loopback classifier:** `src/commands/models/auth.ts:611-615`
  - body: The new helper only accepts `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already handles other local forms such as `localhost.`, 127/8, bracketed IPv6, and IPv4-mapped loopback. Valid local node-host configs could be rejected.
  - confidence: 0.88
