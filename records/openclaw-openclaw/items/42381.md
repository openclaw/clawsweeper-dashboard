## Summary

Keep open: this is the right focused fix path for the linked P1 auth bug and the PR includes real terminal proof, but the current head still bypasses tokenless remote node-host configs and duplicates narrower loopback logic, so it needs repair before merge.

## What This Changes

The PR adds a pre-auth guard to `models auth login` for remote node-host configurations, focused regression coverage, and a changelog entry.

## Best Possible Solution

Block all non-loopback node-host auth-login writes before provider auth, regardless of saved token, using the shared gateway loopback semantics with focused tokenless and loopback regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main resolves and persists provider auth through the local `agentDir` with no node-host guard, and the PR body supplies terminal proof for the intended tokenful remote-node guard path.

## Solution Assessment

No. Blocking before provider auth is the narrow maintainable direction, but this diff should remove the token precondition, reuse the shared loopback helper, and add tokenless-remote plus valid-loopback coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token gate from the remote-node guard:** `src/commands/models/auth.ts:727`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the gateway target before resolving token/password from config or environment. Returning when the token is missing leaves tokenless remote node hosts able to enter provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared gateway loopback helper:** `src/commands/models/auth.ts:718-723`
  - body: This helper only treats empty host, `localhost`, `127.0.0.1`, and `::1` as local. The shared gateway helper already covers localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so this PR can falsely block valid local node-host configs.
  - confidence: 0.86
