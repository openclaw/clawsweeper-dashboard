## Summary

This PR should stay open because it targets a real P1 auth bug, but the current patch still leaves a credential-write bypass for tokenless remote node-host configs and duplicates narrower loopback logic that can block valid local hosts.

## What This Changes

The PR adds a pre-auth guard, regression test, and changelog entry for blocking `openclaw models auth login` on remote node-host configurations.

## Best Possible Solution

Merge a repaired version that blocks every non-loopback node-host auth-login write before provider auth while preserving loopback behavior through the shared gateway helper.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main resolves a local `agentDir` and persists provider auth results there while node-host config can point at a remote gateway. The PR body also supplies after-patch terminal proof for the intended early guard path.

## Solution Assessment

No. Blocking before provider auth is the right narrow direction, but the patch should remove the optional-token precondition, reuse the shared loopback helper, and cover tokenless remote plus loopback cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block tokenless remote node hosts:** `src/commands/models/auth.ts:728`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving token/password from config or environment. Returning when the token is missing leaves tokenless remote node hosts able to enter provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:719-723`
  - body: This helper only treats empty host, `localhost`, `127.0.0.1`, and `::1` as local. The existing gateway helper already handles localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so this PR can reject valid local node-host configs.
  - confidence: 0.86
