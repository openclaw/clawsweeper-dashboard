## Best Possible Solution

Refresh the branch and repair the guard to block any node-host config with a non-loopback gateway before provider auth, using the shared gateway loopback helper with tokenless and accepted-loopback tests.

## Reproduction Assessment

Yes. Source inspection shows current main writes provider auth profiles through the local `agentDir`, and the PR body includes terminal proof for the token-bearing non-loopback node-host guard path.

## Solution Assessment

No. The direction is right, but the implementation should block based on the non-loopback gateway config rather than `nodeHostConfig.token` and should reuse `src/gateway/net.ts` loopback semantics.

A local gateway using one of those forms would be treated as remote and blocked, so import the shared helper and add allow-case coverage.
