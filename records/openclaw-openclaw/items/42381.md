## Summary

Keep open: this is a real auth-provider bug and the PR is close to the right direction, but the current branch still leaves a wrong-host credential-write path reachable for tokenless remote node-host configs and can fail closed for loopback host forms accepted by the gateway.

## What This Changes

This PR adds a `models auth login` guard for remote node-host credential writes, a regression test and changelog entry, and removes an unused daemon inspect helper.

## Best Possible Solution

Repair the guard to block any non-loopback node-host config before provider auth, reuse the shared gateway loopback helper, cover tokenless remote and loopback allow cases, then land the fail-closed behavior with the existing changelog note.

## Reproduction Assessment

Yes, source-level. Current main persists provider auth through the local `agentDir`, and PR head still returns early for tokenless non-loopback node-host configs before reaching the new guard failure.

## Solution Assessment

No. The fail-closed guard is the right direction, but the implementation should key off non-loopback node-host gateway state rather than token presence and should reuse the shared loopback contract.
