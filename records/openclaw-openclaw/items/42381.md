## Summary

Keep open: the PR targets a real auth-write bug and now includes usable terminal proof, but the current head still leaves tokenless remote node hosts able to write local auth profiles and duplicates narrower loopback semantics than the shared gateway helper.

## What This Changes

The PR adds a pre-login remote node-host guard to `models auth login`, a regression test, and a changelog entry.

## Best Possible Solution

Repair this PR by blocking non-loopback node-host auth-login writes regardless of saved token presence, reusing the shared loopback helper, and adding focused coverage for tokenless remote configs and accepted loopback forms.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main resolves a local `agentDir` and writes provider profiles there while node-host config can point at a remote gateway; I did not run a live OAuth flow.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow fix, but this implementation should not depend on `nodeHostConfig.token` and should reuse the existing gateway loopback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token gate from the remote-node guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from config or environment. With this check, a tokenless remote node host still reaches provider auth and can write profiles into the local agent store, leaving the reported credential-consumption path reachable.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host check:** `src/commands/models/auth.ts:710-712`
  - body: This helper only allows an empty host, `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already accepts the loopback forms OpenClaw supports, including 127/8, bracketed IPv6, and IPv4-mapped loopback. Valid local node-host configs such as `[::1]` or `127.0.0.2` would be blocked by the PR-local helper.
  - confidence: 0.86
