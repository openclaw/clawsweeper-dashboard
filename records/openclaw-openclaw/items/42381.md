## Summary

Keep this PR open, but do not land it as-is. Current main still has the reported local auth-write behavior, the linked bug remains open, and the PR is a focused fix attempt; however the proposed guard is ineffective for normal current node-host configs and the branch is currently dirty/unmergeable.

## What This Changes

The PR adds a node-host config check before `modelsAuthLoginCommand` and a regression test so remote node hosts with non-loopback gateways reject provider auth login before credentials are persisted locally.

## Best Possible Solution

Land a rebased fix that detects node-host remote gateway state from current node-host config, blocks the unsafe local credential write before provider auth consumes refresh material, uses the shared loopback classifier, and covers the normal node.json shape plus loopback allow paths in `src/commands/models/auth.test.ts`.

## Reproduction Assessment

Yes. The linked bug provides a concrete remote node-host setup and current main source shows the login path persists auth results to the resolved local `agentDir`; the PR-specific failure is reproducible by static inspection because `runNodeHost` saves gateway info without `token` while the new guard returns early when `token` is absent.

## Solution Assessment

No. Blocking remote node-host auth writes is the right direction, but this PR's token gate misses the current node-host config written by `runNodeHost`, and its loopback helper is narrower than the shared gateway helper already used elsewhere.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not gate the guard on nodeConfig.token:** `src/commands/models/auth.ts:544-545`
  - body: `runNodeHost` currently saves node id/display name/gateway info to node.json but never writes `token`, so a normal remote node host will load a config without `token` and this early return lets the OAuth flow run and write credentials locally. Detect the node-host remote gateway from the current config shape instead of requiring `token`.
  - confidence: 0.93
- **[P2] Use the shared loopback host classifier:** `src/commands/models/auth.ts:537-539`
  - body: This local helper only allows `localhost`, `127.0.0.1`, and `::1`, while OpenClaw's shared `isLoopbackHost` also handles valid loopback forms such as `127.0.0.2`, `[::1]`, and IPv4-mapped loopback. With a local node host configured through one of those forms, the PR would incorrectly block auth as remote.
  - confidence: 0.88
