## Summary

Keep open: this is a focused auth bug fix with sufficient terminal proof and green checks, but the current PR head still leaves tokenless remote node-host configs able to write local profiles and duplicates narrower loopback semantics than the shared gateway helper.

## What This Changes

The PR adds a `models auth login` guard for remote node hosts, regression coverage, and a changelog entry.

## Best Possible Solution

Update this PR or a replacement to block all non-loopback node-host auth-login writes using the shared gateway loopback helper, with focused tests for tokenless remote configs and local loopback variants.

## Reproduction Assessment

Yes. Source inspection shows current main resolves a local `agentDir` and persists provider auth profiles there while node-host config can point at a remote gateway; the PR body also supplies terminal proof of the proposed guard path.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow fix, but this implementation should remove the `token` precondition and reuse the existing shared loopback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token gate from the remote-node guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from runtime config or environment. Returning when the token is missing leaves tokenless remote node hosts able to enter provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:708-712`
  - body: This local helper only treats empty host, `localhost`, `127.0.0.1`, and `::1` as local. `src/gateway/net.ts` already covers localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so this PR can reject valid local node-host configs.
  - confidence: 0.86
