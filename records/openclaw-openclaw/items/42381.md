## Summary

Keep open: current main still lacks the remote-node auth guard, and the PR is a focused active fix path, but the current diff still leaves tokenless remote node-host configs able to write local auth credentials.

## What This Changes

The PR adds a pre-auth remote-node guard and regression test for `openclaw models auth login`, adds a changelog entry, and removes an unused daemon inspect helper.

## Best Possible Solution

Merge a repaired version that blocks non-loopback node-host auth-login writes before provider auth starts, treats absent gateway host as local, and uses the repo's shared loopback semantics with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main persists `models auth login` results through the local `agentDir` while `node.json` can point at a non-loopback gateway, and the PR's tokenless bypass is visible in the guard condition.

## Solution Assessment

No. A pre-auth guard is the right narrow direction, but this implementation should not key off the optional node token and should reuse shared loopback semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block tokenless remote node hosts:** `src/commands/models/auth.ts:728`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` stores the remote gateway target before resolving credentials from config or environment. With this return, a saved remote gateway config without a stored token still proceeds into provider auth and local profile writes.
  - confidence: 0.92
- **[P2] Use shared loopback semantics:** `src/commands/models/auth.ts:719-723`
  - body: This helper only accepts a small literal subset of loopback hosts. The existing gateway helper covers localhost dot variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so local node-host configs using those forms would be blocked incorrectly.
  - confidence: 0.86
