## Summary

Keep open: the bug is real and current main still lacks a remote node-host auth-login guard, but this PR still has blocking issues in the guard conditions and loopback handling.

## What This Changes

The PR adds a `models auth login` remote node-host guard, regression coverage, and a changelog entry.

## Best Possible Solution

Block all non-loopback node-host auth-login writes before provider auth using shared `src/gateway/net.ts` loopback semantics, with focused tokenless-remote and loopback tests.

## Reproduction Assessment

Yes. Source inspection shows `models auth login` resolves and writes through a local `agentDir` while node-host config can point at a remote gateway, and the PR body supplies terminal proof of the proposed early guard path.

## Solution Assessment

No. Blocking before provider auth is the right narrow direction, but this diff should remove the optional-token precondition and reuse the shared loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token gate from the remote-node guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the gateway target before resolving credentials from config or environment. Returning when the token is missing leaves tokenless remote node hosts able to enter provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:708-712`
  - body: This helper only treats empty host, `localhost`, `127.0.0.1`, and `::1` as local. The shared gateway helper already covers localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so this PR can reject valid local node-host configs.
  - confidence: 0.86
