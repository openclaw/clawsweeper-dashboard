## Summary

Keep open: the PR targets a real auth-write bug and now includes contributor terminal proof, but the live head still leaves tokenless remote node-host configs able to write local provider auth profiles and uses narrower loopback logic than the shared gateway helper.

## What This Changes

The PR adds a `models auth login` remote node-host guard, regression coverage, and a changelog entry.

## Best Possible Solution

Land or replace this PR with a focused guard that blocks all non-loopback node-host auth-login writes using shared loopback semantics, leaving credential sync for a separate explicit API design.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes `models auth login` provider profiles to the local agent store while saved node-host config can point at a remote gateway.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow fix, but this implementation should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from runtime config or environment. Returning when the saved token is absent leaves tokenless remote node hosts able to run provider auth and write local profiles, so the reported credential-consumption path remains reachable.
  - confidence: 0.92
- **[P2] Use the shared loopback host check:** `src/commands/models/auth.ts:708-712`
  - body: This helper only accepts empty host, `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already accepts local gateway forms such as localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback. Reusing the shared helper avoids blocking valid local node-host configs.
  - confidence: 0.86
