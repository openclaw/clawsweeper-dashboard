## Summary

Keep open: the linked auth bug is still present on current main, and this PR is the active fix path, but the current patch still leaves a tokenless remote node-host write path open, uses narrower loopback semantics than the shared gateway helper, and lacks required external real behavior proof.

## What This Changes

This PR adds a `models auth login` preflight guard for remote node-host configs, regression coverage, a changelog entry, and a real-behavior-proof workflow/script addition that is already present on current main.

## Best Possible Solution

Land a rebased branch or replacement that blocks any non-loopback node-host `models auth login` write using the shared loopback helper, covers tokenless and loopback cases, and includes redacted real remote node-host/gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main resolves a local agent directory and persists provider auth profiles there while node-host config can point at a remote gateway; I did not run a live remote OAuth flow.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow fix direction, but this implementation should drop the token precondition, reuse shared loopback semantics, and add the missing coverage/proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:620`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote `gateway` target before credentials are resolved from runtime config or environment. With this return path, a remote node host without a persisted token can still run provider auth and write local profiles, leaving the reported refresh-token consumption path open.
  - confidence: 0.92
- **[P2] Use the shared loopback host check:** `src/commands/models/auth.ts:611-615`
  - body: This local helper only allows empty, `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already treats localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback as local. Reusing that helper avoids blocking valid local node-host configs.
  - confidence: 0.87
