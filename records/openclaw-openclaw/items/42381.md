## Label Justifications

- `P1`: The PR addresses a real remote gateway auth workflow where provider credentials can be consumed or stored on the wrong host and break model calls.
- `merge-risk: 🚨 compatibility`: The patch intentionally changes remote node-host auth login to fail closed and may also reject some loopback host forms if the helper is not repaired.
- `merge-risk: 🚨 auth-provider`: The changed path controls provider OAuth/token profile writes and can leave credentials on the wrong side of the node/gateway boundary if incomplete.

## Summary

Keep open: the reported auth-boundary problem is valid on current main, but the latest PR head still misses real remote node-host configs because the guard depends on an optional `node.json` token and copies narrower loopback semantics than the gateway uses.

## What This Changes

The PR adds a pre-auth `models auth login` guard for remote node-host configs, regression coverage, a changelog entry, and an unrelated removal of an unused daemon inspect helper.

## Best Possible Solution

Merge a repaired pre-auth guard that blocks non-loopback node-host auth login before provider auth, reuses shared gateway loopback semantics, covers tokenless remote and loopback allow cases, and leaves gateway-side credential forwarding for a separate design if desired.

## Reproduction Assessment

Yes. Source inspection shows current main writes provider auth profiles through the local `agentDir`, and the PR body proves the token-bearing remote-node guard path while the PR diff shows a tokenless bypass remains.

## Solution Assessment

No. Blocking remote node-host auth login is the right narrow direction, but this patch should not depend on `nodeHostConfig.token` and should reuse the existing gateway loopback helper before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block tokenless remote node hosts:** `src/commands/models/auth.ts:715`
  - body: `node.json` only stores the gateway target; gateway auth can come from config or env, and the `token` field is optional and not written by the current node-host runner. With this early return, a non-loopback node host without `nodeHostConfig.token` still continues into provider auth and local `auth-profiles.json` writes, leaving the reported credential-boundary bug reachable.
  - confidence: 0.94
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:705-710`
  - body: This local helper is narrower than `src/gateway/net.ts:isLoopbackHost`, which already accepts forms like `localhost.`, bracketed `::1`, and IPv4-mapped loopback. A copied subset can incorrectly fail closed for local node-host auth login after upgrade, so the guard should reuse the shared gateway semantics and cover those allow cases.
  - confidence: 0.86
