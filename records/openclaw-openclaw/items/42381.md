## Summary

Keep open: this PR targets a real auth bug and has usable proof, but the current head still misses tokenless remote node-host configs and reimplements narrower loopback semantics than core already provides.

## What This Changes

The PR adds a `models auth login` guard, regression coverage, and a changelog entry to block local credential writes on remote node hosts.

## Best Possible Solution

Land or replace this PR with a focused guard that blocks all non-loopback node-host auth-login writes using shared loopback semantics, while leaving credential forwarding/sync for a separate explicit design.

## Reproduction Assessment

Yes. Source inspection shows current main resolves a local `agentDir` and persists provider auth profiles there while node-host config can point at a remote gateway; the PR body also shows terminal proof of the intended guard path.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow solution, but this implementation should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from runtime config or environment. Returning when the saved token is absent leaves tokenless remote node hosts able to run provider auth and write local profiles, so the reported credential-consumption path remains reachable.
  - confidence: 0.92
- **[P2] Use the shared loopback host check:** `src/commands/models/auth.ts:708-712`
  - body: This helper only accepts empty host, `localhost`, `127.0.0.1`, and `::1`, while `src/gateway/net.ts` already handles localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback. Reusing the shared helper avoids blocking valid local node-host configs.
  - confidence: 0.86
