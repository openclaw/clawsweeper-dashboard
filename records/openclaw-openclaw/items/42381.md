## Summary

Keep open: the PR targets a real auth-provider bug, but the current head still leaves one remote-node credential-write path reachable and uses a narrower loopback check than the shared gateway helper.

## What This Changes

The PR adds a pre-auth guard and regression test for `openclaw models auth login` on remote node hosts, plus a changelog entry and a small daemon inspect cleanup.

## Best Possible Solution

Merge a repaired pre-auth guard that blocks non-loopback node-host auth-login writes before provider auth starts, treats absent gateway host as local, and reuses the shared gateway loopback semantics with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes provider auth profiles through the local `agentDir`, and the PR head still permits a tokenless `node.json` with a non-loopback gateway.

## Solution Assessment

No. The pre-auth guard is the right narrow direction, but this implementation should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block tokenless remote node hosts:** `src/commands/models/auth.ts:728`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` persists the remote gateway before credentials are resolved from config or env. Returning here when the token is absent still lets a node with a remote gateway proceed into local auth-profile writes, leaving the reported credential mismatch reachable.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host check:** `src/commands/models/auth.ts:719-723`
  - body: This helper only accepts a small literal subset of loopback hosts. The existing gateway helper covers localhost dot variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so local node-host configs using those forms would be blocked incorrectly.
  - confidence: 0.86
