## Summary

Keep this PR open: it targets a valid auth bug, but the current branch still misses tokenless remote node-host configurations and lacks required real behavior proof for the external auth-flow change.

## What This Changes

The PR adds a remote node-host guard to `models auth login`, regression coverage, a changelog entry, and real-behavior-proof workflow/script files.

## Best Possible Solution

Use a focused rebased auth fix that blocks all non-loopback remote node-host auth-login writes, reuses shared loopback semantics, covers tokenless and loopback cases, and includes redacted real remote node-host proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes provider auth profiles to the local resolved `agentDir` and does not check node-host remote gateway config before provider auth can run.

## Solution Assessment

No. Blocking remote node-host auth-login writes is the right narrow direction, but this branch should drop the token precondition, use the shared loopback helper, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:620`
  - body: `NodeHostConfig.token` is optional, and node-host gateway auth can come from config or `OPENCLAW_GATEWAY_*` after the remote gateway target is saved. This early return leaves tokenless remote node hosts able to run provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Use the shared loopback host check:** `src/commands/models/auth.ts:611-615`
  - body: This helper only accepts empty, `localhost`, `127.0.0.1`, and `::1`; `src/gateway/net.ts` already handles 127/8, bracketed IPv6, IPv4-mapped loopback, and localhost variants. Reusing that helper avoids blocking valid local node-host configs.
  - confidence: 0.87
