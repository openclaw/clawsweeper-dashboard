## Summary

Keep open: this PR targets a real P1 auth bug, but the current head still leaves tokenless remote node-host configs able to write local provider credentials and duplicates narrower loopback semantics than the shared gateway helper.

## What This Changes

The PR adds a guard, regression test, and changelog entry to block `openclaw models auth login` on remote node-host configurations before credentials are written locally.

## Best Possible Solution

Block all non-loopback node-host auth-login writes regardless of stored token, reuse `src/gateway/net.ts` loopback semantics, and add focused tokenless-remote plus loopback coverage.

## Reproduction Assessment

Yes. Source inspection shows current main writes provider auth results through the local `agentDir` while node-host config can point at a remote gateway; the PR body also supplies terminal proof of the intended early guard path.

## Solution Assessment

No. Blocking before provider auth is the right narrow direction, but this diff should remove the optional-token precondition and reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token gate from the remote-node guard:** `src/commands/models/auth.ts:727`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the gateway target before resolving token/password from config or environment. Returning when the token is missing leaves tokenless remote node hosts able to enter provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host helper:** `src/commands/models/auth.ts:718-723`
  - body: This helper only treats empty host, `localhost`, `127.0.0.1`, and `::1` as local. The shared gateway helper already covers localhost variants, 127/8, bracketed IPv6, and IPv4-mapped loopback, so this PR can reject valid local node-host configs.
  - confidence: 0.86
