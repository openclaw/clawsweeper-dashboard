## Summary

Keep this PR open: the underlying current-main bug is still present, but the provided head patch still leaves tokenless remote node-host configs unblocked and duplicates narrower loopback logic than the existing gateway helper.

## What This Changes

The PR adds a `models auth login` guard, regression coverage, and a changelog entry to block local provider-profile writes on remote node hosts.

## Best Possible Solution

Land a focused version that blocks non-loopback `nodeHostConfig.gateway.host` regardless of `token`, reuses `src/gateway/net.ts` loopback classification, and covers tokenless remote plus loopback allow cases.

## Reproduction Assessment

Yes. Source inspection on current main shows `modelsAuthLoginCommand` writes profiles through the local `agentDir`, while node-host gateway config can point at a remote host and credentials can come from env/config without `node.json.token`.

## Solution Assessment

No. Blocking remote node-host auth-login writes is the right narrow fix, but this PR should key off the non-loopback gateway host without requiring `token` and should reuse the shared loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the remote-node guard:** `src/commands/models/auth.ts:619`
  - body: `NodeHostConfig.token` is optional, and node-host auth can come from env/config/password paths while `node.json.gateway.host` still points at a remote gateway. Returning when the token is absent leaves the local auth-write path reachable for those remote node hosts.
  - confidence: 0.93
- **[P2] Reuse the shared loopback classifier:** `src/commands/models/auth.ts:610-615`
  - body: The new helper only accepts `localhost`, `127.0.0.1`, and `::1`, while gateway code already treats 127/8, bracketed IPv6, IPv4-mapped loopback, and localhost absolute-form names as local. Valid local node-host configs could be blocked as remote.
  - confidence: 0.88
