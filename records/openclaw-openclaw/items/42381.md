## Summary

Keep open: the PR targets a real credential-write bug and now includes usable real-behavior proof, but the current head still leaves a tokenless remote node-host path unguarded and duplicates narrower loopback semantics than core already uses.

## What This Changes

The branch adds a node-host config guard to `modelsAuthLoginCommand`, a regression test, and a changelog entry for blocking remote node-host auth-login writes.

## Best Possible Solution

Land a focused guard that blocks any saved non-loopback node-host gateway before local auth/profile writes, reuses the shared gateway loopback helper, and keeps regression coverage plus the changelog entry.

## Reproduction Assessment

Yes. Current main writes provider auth profiles via the local `agentDir` and never checks saved node-host gateway config, while node-host config can point at a remote gateway with credentials supplied from env/config.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow direction, but this implementation should remove the token precondition and use the shared loopback helper before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:717`
  - body: `NodeHostConfig.token` is optional, and `runNodeHost` saves the remote gateway target before resolving credentials from env/runtime config. Returning when the saved token is absent leaves remote node hosts authenticated by env/config able to run provider auth and write local profiles.
  - confidence: 0.92
- **[P2] Reuse the shared loopback host check:** `src/commands/models/auth.ts:708-713`
  - body: The new helper only accepts empty, `localhost`, `127.0.0.1`, and `::1`, while the existing gateway helper accepts established local forms such as `localhost.`, 127/8, bracketed IPv6, and IPv4-mapped loopback. This can block valid local node-host configs as if they were remote.
  - confidence: 0.86

## Security Review

  - body: Because the guard returns when `nodeHostConfig.token` is absent, a node host with a saved remote gateway and credentials supplied through env/config can still consume and store provider credentials on the node instead of the gateway.
