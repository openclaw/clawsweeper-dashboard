## Summary

Keep open: this PR targets a valid auth bug linked to an open closing issue, but the current head still token-gates the remote-node guard, duplicates the gateway loopback classifier, and lacks real remote-node behavior proof.

## What This Changes

The PR adds a `models auth login` preflight guard based on node-host config, a regression test, and a changelog entry for remote node-host credential writes.

## Best Possible Solution

Update this PR or replace it with a focused guard that blocks non-loopback node-host auth-login writes regardless of token, reuses the shared gateway loopback classifier, covers tokenless and loopback cases, and includes redacted real remote-node proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main resolves a local agent directory and persists provider auth profiles there, while node-host config can point at a remote gateway where model calls happen; I did not run a live remote gateway OAuth flow.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow direction, but this implementation should key off the non-loopback gateway host without requiring `token` and should reuse the existing gateway loopback classifier.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:620`
  - body: `NodeHostConfig.token` is optional, and node-host gateway auth can come from environment or runtime config while `nodeHostConfig.gateway.host` still points at a remote gateway. Returning when `token` is absent leaves the local auth-write path reachable for those remote node hosts.
  - confidence: 0.93
- **[P2] Reuse the shared loopback classifier:** `src/commands/models/auth.ts:611-615`
  - body: This helper only treats empty, `localhost`, `127.0.0.1`, and `::1` as local, while gateway code already accepts broader loopback forms like `127/8`, `[::1]`, IPv4-mapped loopback, and `localhost.`. Valid local node-host configs could be blocked as remote.
  - confidence: 0.88
