## Summary

Keep open: this is a focused auth-provider bug fix with useful proof, but the current PR head still misses a real remote-node config shape and can reject local loopback gateway forms that the shared gateway code accepts.

## What This Changes

The PR adds a `models auth login` preflight that blocks node hosts pointed at a non-loopback gateway before writing provider auth profiles locally, adds a regression test and changelog entry, and also removes an unrelated daemon inspect helper.

## Best Possible Solution

Repair the guard so it uses the same gateway loopback semantics as runtime code, blocks remote node-host configs regardless of stored token presence, preserves local loopback auth login, and lands as a focused auth fix with tests.

## Reproduction Assessment

Yes. Source inspection shows current main writes provider auth profiles through the local `agentDir`, while node-host config can persist a non-loopback gateway without a stored token; the PR body supplies terminal proof for the token-bearing guarded path.

## Solution Assessment

No as currently written. Blocking remote-node auth writes is the right narrow direction, but this implementation should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback contract before merge.
