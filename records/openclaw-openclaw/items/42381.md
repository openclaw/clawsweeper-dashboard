## Label Justifications

- `P1`: The PR fixes a credential-placement bug that can break real model auth flows and consume OAuth refresh tokens on the wrong host.
- `merge-risk: 🚨 compatibility`: The intended fix changes an existing remote node-host command from permissive local writes to a fail-closed error, and the current loopback helper can block supported local forms.
- `merge-risk: 🚨 auth-provider`: The patch touches provider auth login and still leaves a tokenless path that can write credentials to the wrong auth-profile store.

## Summary

This PR is a useful, active fix path with sufficient terminal proof for the covered scenario, but the current head is not merge-ready because the guard still depends on a saved node token and uses narrower loopback detection than the gateway code.

## What This Changes

This PR adds a pre-auth remote node-host guard to `models auth login`, a regression test, a changelog note, and an unrelated unused inspect-helper removal.

## Best Possible Solution

Merge a repaired pre-auth guard that blocks all non-loopback node-host auth-login local writes, treats missing gateway host as local, reuses shared loopback detection, and keeps focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a tokenless `node.json` with a non-loopback gateway host bypasses the PR guard and then the auth code can persist provider credentials through the local `agentDir`; I did not run the flow in this read-only review.

## Solution Assessment

No. The fail-closed pre-auth guard is the right narrow shape, but as written it should key on non-loopback gateway config rather than saved-token presence and should reuse the shared gateway loopback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block tokenless remote node configs:** `src/commands/models/auth.ts:728`
  - body: The guard returns whenever `nodeHostConfig.token` is absent, but node-host config persists `gateway.host` separately and the token is optional. Password/env-token or not-yet-paired remote node configs can still proceed into local provider auth writes, so this should key on a non-loopback gateway host instead of saved token presence.
  - confidence: 0.91
- **[P2] Reuse the gateway loopback helper:** `src/commands/models/auth.ts:719-723`
  - body: This helper only accepts empty, `localhost`, `127.0.0.1`, and `::1`, while the gateway helper already handles supported loopback forms such as `localhost.`, 127/8, bracketed IPv6, and mapped IPv4 loopback. Reusing the shared helper avoids fail-closing valid local node-host setups after upgrade.
  - confidence: 0.86
