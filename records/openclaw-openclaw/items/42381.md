## Summary

This PR addresses a real P1 auth-provider/security bug and has useful proof, but it is not merge-ready because the current branch still gates the remote-node block on a node token and copies narrower loopback semantics than the shared gateway code.

## What This Changes

The branch adds a `models auth login` preflight for node-host config, blocks non-loopback remote node hosts before local auth-profile writes, adds regression coverage, updates the changelog, and removes an unused daemon inspect helper.

## Best Possible Solution

Repair the guard so it uses the same gateway loopback semantics as the runtime, blocks remote node-host configs regardless of token presence, keeps local loopback auth login working, and lands with focused auth tests only.

## Reproduction Assessment

Yes for source reproduction: current main writes provider auth results through the local `agentDir`, and node-host config can persist a non-loopback gateway without a stored token. I did not run the command because this review is read-only, but the PR body supplies terminal proof for the token-bearing guarded path.

## Solution Assessment

No as currently written: blocking remote node-host auth writes is the right direction, but the implementation should not depend on `nodeHostConfig.token` and should reuse the shared gateway loopback semantics before merge.
