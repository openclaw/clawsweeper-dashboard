## Summary

Keep open: this is a useful auth bug fix path, but the current PR is not merge-ready because it still has an auth-sensitive guard gap, duplicates narrower loopback logic than the gateway uses, lacks after-fix real remote node-host proof, and is currently conflicting.

## What This Changes

The PR adds a pre-login guard, regression test, and changelog entry to stop `openclaw models auth login` from writing local provider credentials on remote node hosts.

## Best Possible Solution

Update or replace this PR with a focused guard based on non-loopback `nodeHostConfig.gateway.host` regardless of token, reuse the shared gateway loopback classifier, add tokenless and loopback regression coverage, then attach redacted real remote-node proof before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main resolves a local agent store and persists provider auth profiles there while node-host config can point model execution at a remote gateway; I did not run a live OAuth flow.

## Solution Assessment

No. Blocking remote node-host auth writes is the right narrow direction, but this implementation should not depend on `nodeHostConfig.token` and should reuse the existing gateway loopback classifier.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove the token precondition from the guard:** `src/commands/models/auth.ts:620`
  - body: `NodeHostConfig.token` is optional, and node-host gateway auth can come from runtime config or environment while `nodeHostConfig.gateway.host` still points at a remote gateway. Returning when `token` is absent leaves the local auth-write path reachable for those remote node hosts.
  - confidence: 0.93
- **[P2] Reuse the shared loopback classifier:** `src/commands/models/auth.ts:611-615`
  - body: This helper only treats empty, `localhost`, `127.0.0.1`, and `::1` as local, while gateway code already accepts broader loopback forms like `127/8`, `[::1]`, IPv4-mapped loopback, and `localhost.`. Valid local node-host configs could be blocked as remote.
  - confidence: 0.88
