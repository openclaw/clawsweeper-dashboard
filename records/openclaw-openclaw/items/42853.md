## Summary

Keep open. Current main still lacks a durable gateway heap configuration path and the linked Linux/macOS OOM reports make the capability useful, but this branch is not mergeable as-is because it weakens service environment hardening, targets stale setup paths, loses current node-service PATH handling, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `gateway.nodeMaxOldSpaceMb` config/schema/help surface, injects a resolved V8 heap flag into gateway and node service `NODE_OPTIONS`, threads install helpers, adds an onboarding prompt, and expands service environment tests.

## Best Possible Solution

Ship a refreshed, reviewed implementation that derives a heap-only service startup value from typed config or a maintainer-approved default, ports the prompt to `src/wizard/setup.gateway-config.ts`, preserves current PATH handling, and validates regenerated launchd/systemd/Windows/node-host service metadata.

## Reproduction Assessment

Yes, at source level: current main has no `gateway.nodeMaxOldSpaceMb` schema/type field and generated gateway/node service environments do not set a managed heap flag. The full OOM workloads from https://github.com/openclaw/openclaw/issues/52092 and https://github.com/openclaw/openclaw/issues/76658 were not live-reproduced here.

## Solution Assessment

No. A typed durable heap config is a plausible direction, but this PR should not persist ambient `NODE_OPTIONS`, should handle both Node heap flag spellings if it preserves existing heap options, and must be rebased onto the active setup and node-service install paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Generate a heap-only NODE_OPTIONS value:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends the caller's existing `NODE_OPTIONS`, and this line writes that value into generated service metadata. That bypasses current install hardening: running install with `NODE_OPTIONS="--require /tmp/evil.js"` can persist an arbitrary startup hook. Derive the service value only from the approved heap config/default, or use an equivalent non-ambient startup mechanism.
  - confidence: 0.92
- **[P2] Detect underscore heap flags before appending:** `src/daemon/service-env.ts:269`
  - body: Node accepts `--max_old_space_size` as a `NODE_OPTIONS` heap flag too, but the helper only checks for the hyphenated spelling. A user with `NODE_OPTIONS=--max_old_space_size=2048` would get a second heap flag appended, so the generated value can unexpectedly override their existing limit.
  - confidence: 0.86
- **[P2] Port the heap prompt to the active setup wizard:** `src/wizard/onboarding.gateway-config.ts:310-318`
  - body: Current main uses `src/wizard/setup.gateway-config.ts`, and `src/wizard/onboarding.gateway-config.ts` is no longer tracked. Adding the heap prompt here means users will not see or clear the new setting through the current setup flow after rebase.
  - confidence: 0.9
- **[P2] Preserve node service PATH additions:** `src/commands/node-daemon-install-helpers.ts:60-63`
  - body: Current main passes `extraPathDirs` from the selected `nodePath` into `buildNodeServiceEnvironment`. This replacement call adds `nodeMaxOldSpaceMb` but omits that argument, so supervised node services can lose the selected Node toolchain on PATH.
  - confidence: 0.82
