## Summary

Keep open for maintainer follow-up, but do not merge this branch as-is. The heap-limit configuration addresses real Gateway OOM reports, yet the PR currently persists ambient `NODE_OPTIONS` into generated service metadata, misses a Node-supported heap flag spelling, does not actually pass loaded config into the node-service install caller, is stale against current service-environment plumbing, and lacks real regenerated-service proof.

## What This Changes

Adds a `gateway.nodeMaxOldSpaceMb` config/schema/onboarding field and injects a computed Node heap flag into generated gateway/node service `NODE_OPTIONS` with unit tests.

## Best Possible Solution

Land or replace this with a heap-only managed config/default path that preserves the `NODE_OPTIONS` blocklist, current service PATH/env planning, node-host config threading, and real install/update proof across service managers.

## Reproduction Assessment

No high-confidence live OOM reproduction was run in this read-only pass. The PR defects are source-reproducible from the PR head and current main: ambient `NODE_OPTIONS` is copied into service metadata, underscore heap flags are missed, and the node install caller does not pass loaded config.

## Solution Assessment

No. A typed heap setting is the right direction, but this implementation should be reworked to generate heap-only startup options from config/default, preserve current service-env APIs, pass config through node installs, handle Node flag aliases, and prove real daemon regeneration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid copying ambient NODE_OPTIONS into managed services:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends to `env.NODE_OPTIONS`, and this assignment writes the result into generated service metadata. That bypasses the current host-env policy that blocks `NODE_OPTIONS`, so an install-time value like `--require /tmp/evil.js` or `--import ...` can become durable daemon startup control; generate the heap flag from the typed config/default only, without carrying arbitrary ambient flags.
  - confidence: 0.94
- **[P2] Detect underscore heap flags before appending:** `src/daemon/service-env.ts:269-271`
  - body: Node accepts `--max_old_space_size` as the same heap limit flag, but this check only looks for `--max-old-space-size`. A user with `NODE_OPTIONS=--max_old_space_size=2048` would get a second hyphenated flag appended, and the later flag wins, silently replacing their existing heap choice.
  - confidence: 0.9
- **[P2] Pass the loaded config into node service installs:** `src/commands/node-daemon-install-helpers.ts:60-63`
  - body: This helper reads `params.config?.gateway?.nodeMaxOldSpaceMb`, but the node install CLI call still does not pass its loaded `config` into `buildNodeInstallPlan`. As a result, independently installed node services keep auto-detecting and ignore the user's configured heap cap on the actual install path.
  - confidence: 0.87
