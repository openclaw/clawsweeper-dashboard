## Summary

Keep open: current main still lacks the requested gateway heap-limit config and service heap injection, and the linked OOM report remains an active user problem. The PR is useful source work, but it is not mergeable as-is because it would persist ambient NODE_OPTIONS into durable service startup and it is stale against the current setup and node-service install paths.

## What This Changes

This PR adds a gateway.nodeMaxOldSpaceMb config/schema/help field, injects NODE_OPTIONS heap limits into gateway and node service environments, threads the config through install helpers, adds a heap prompt to the old onboarding flow, and adds service-env tests.

## Best Possible Solution

Ship a refreshed implementation tied to #52092 that adds the documented config contract, generates a deterministic heap-only service startup value without preserving ambient NODE_OPTIONS, recognizes both accepted Node heap flag spellings where relevant, ports the prompt to the current setup flow with blank-to-auto behavior, preserves current node-service PATH handling, adds the required changelog entry, and covers gateway/node install behavior with focused tests.

## Reproduction Assessment

Yes. #52092 gives concrete cron catch-up steps and a working manual --max-old-space-size workaround for the user-visible OOM, and the PR regressions are statically reproducible from the patch plus local Node probes for NODE_OPTIONS startup preloads and both heap flag spellings.

## Solution Assessment

No. Raising the supervised Node heap is a plausible narrow direction, but this PR should not preserve ambient NODE_OPTIONS in durable service metadata and must be rebased onto the current setup and node-service install paths before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Generate a heap-only NODE_OPTIONS value:** `src/daemon/service-env.ts:340`
  - body: resolveNodeOptions preserves or appends the caller's existing NODE_OPTIONS, and buildCommonServiceEnvironment writes that into the durable service environment. That bypasses current install hardening: an install run with NODE_OPTIONS="--require /tmp/evil.js" would persist that startup hook. Generate only the heap flag from config/default, or use an equivalent non-ambient startup mechanism.
  - confidence: 0.92
- **[P2] Respect underscore heap flags before appending:** `src/daemon/service-env.ts:265`
  - body: Node accepts --max_old_space_size as well as --max-old-space-size, but the helper only checks for the hyphen spelling. A user who already pinned heap with the underscore form would get a second heap flag appended, and the generated value can override their limit.
  - confidence: 0.86
- **[P2] Port the prompt to the current setup wizard:** `src/wizard/onboarding.gateway-config.ts:307`
  - body: The heap prompt is added to src/wizard/onboarding.gateway-config.ts, but current main uses src/wizard/setup.gateway-config.ts and the onboarding file is gone. Without moving this into the active setup flow, users will not see or clear the new config through setup.
  - confidence: 0.88
- **[P2] Preserve node service PATH additions:** `src/commands/node-daemon-install-helpers.ts:57-60`
  - body: The PR's node install call adds nodeMaxOldSpaceMb but is based on the old buildNodeServiceEnvironment({ env }) call. Current main also passes extraPathDirs from the selected nodePath; the rebase needs to keep that argument or supervised node services can lose the selected node toolchain on PATH.
  - confidence: 0.8
