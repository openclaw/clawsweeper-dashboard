## Summary

Keep open: current main still lacks durable gateway heap configuration and the linked Linux/macOS OOM reports show real user impact, but this PR is not merge-ready because it persists ambient `NODE_OPTIONS`, targets stale setup code, and needs rebase work for current node-service install behavior.

## What This Changes

Adds `gateway.nodeMaxOldSpaceMb` config/schema/help, injects a resolved Node heap flag into gateway/node service environments, threads daemon install helpers, adds a wizard prompt, and extends service-env tests.

## Best Possible Solution

Ship a refreshed implementation for #52092 and #76658 that adds the config contract, generates a deterministic heap-only startup value, ports setup UI to the active wizard, preserves node-service PATH handling, and updates tests, generated config docs, and changelog.

## Reproduction Assessment

Yes. The linked Linux and macOS reports provide concrete OOM paths and manual heap workarounds, and current-main source inspection confirms there is no durable gateway heap config or service `NODE_OPTIONS` emission today.

## Solution Assessment

No. The durable config direction is plausible, but this patch should not persist ambient `NODE_OPTIONS` and must be rebased onto the active setup and node-service install paths before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Generate a heap-only NODE_OPTIONS value:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends the caller's existing `NODE_OPTIONS`, and `buildCommonServiceEnvironment` writes that into durable service metadata. An install run with `NODE_OPTIONS="--require /tmp/evil.js"` would persist that startup hook despite the current blocked-env contract. Generate only the heap flag from config/default, or use an equivalent non-ambient startup mechanism.
  - confidence: 0.92
- **[P2] Detect underscore heap flags before appending:** `src/daemon/service-env.ts:265`
  - body: Node accepts `--max_old_space_size` as well as `--max-old-space-size`, but the helper only checks the hyphen spelling. A user who already pinned heap with the underscore form would get a second heap flag appended and may have their intended limit overridden.
  - confidence: 0.88
- **[P2] Port the prompt to the active setup wizard:** `src/wizard/onboarding.gateway-config.ts:307`
  - body: The heap prompt is added to `src/wizard/onboarding.gateway-config.ts`, but current main uses `src/wizard/setup.gateway-config.ts` and the onboarding file is gone. Without moving this into the active setup flow, users will not see or clear the new config through setup.
  - confidence: 0.9
- **[P2] Preserve node service PATH additions:** `src/commands/node-daemon-install-helpers.ts:57-60`
  - body: The node install call adds `nodeMaxOldSpaceMb` based on the old `buildNodeServiceEnvironment({ env })` call. Current main also passes `extraPathDirs` from the selected `nodePath`; the rebase needs to keep that argument or supervised node services can lose the selected node toolchain on `PATH`.
  - confidence: 0.82
- **[P3] Update generated config docs and changelog:** `src/config/types.gateway.ts:413`
  - body: This introduces a public gateway config field and changes installed service behavior, but the PR does not update the changelog, generated config schema/reference output, or gateway configuration docs. The new key would be missing from the published config surface and likely trip generated-drift checks.
  - confidence: 0.78
