## Summary

Keep open: the user problem is real and current main still lacks a durable gateway heap setting, but this PR is stale against current main and introduces a security-sensitive startup-environment regression by preserving ambient NODE_OPTIONS in generated services.

## What This Changes

This PR adds a `gateway.nodeMaxOldSpaceMb` config/schema/help field, injects a V8 heap limit into gateway and node service environments, threads the config through install helpers, adds an onboarding prompt, and extends service-env tests.

## Best Possible Solution

Ship a refreshed implementation tied to #52092 and #76658 that adds the config contract, generates a deterministic heap-only service startup value, ports the prompt to the current setup flow, preserves node-service PATH handling, documents the behavior, adds a changelog entry, and covers gateway/node install regeneration with focused tests.

## Reproduction Assessment

Yes. The OOM need has concrete related reproductions in #52092 and #76658, and the PR regressions are source-reproducible from the head diff plus local Node probes for NODE_OPTIONS startup behavior and accepted heap flag spellings.

## Solution Assessment

No. Raising supervised Node heap is a plausible direction, but this patch should not persist ambient NODE_OPTIONS and must be rebased onto the current setup and node-service install paths before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Generate a heap-only NODE_OPTIONS value:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends the caller's existing `NODE_OPTIONS`, and this line writes that into durable service metadata. That bypasses the current install hardening that strips `NODE_OPTIONS`: an install run with a startup hook such as `--require` would persist it for the supervised gateway/node service. Generate only the heap flag from config/default, or use an equivalent non-ambient startup mechanism.
  - confidence: 0.92
- **[P2] Respect underscore heap flags before appending:** `src/daemon/service-env.ts:269`
  - body: Node accepts `--max_old_space_size` as well as `--max-old-space-size`, but the helper only checks the hyphen spelling. A user who already pinned heap with the underscore form would get a second heap flag appended, and the later value can override their intended cap.
  - confidence: 0.86
- **[P2] Port the prompt to the current setup wizard:** `src/wizard/onboarding.gateway-config.ts:310`
  - body: The heap prompt is added to `src/wizard/onboarding.gateway-config.ts`, but current main uses `src/wizard/setup.gateway-config.ts` for gateway setup. Without moving this into the active setup flow, users will not see or clear the new config through setup.
  - confidence: 0.9
- **[P2] Preserve node service PATH additions:** `src/commands/node-daemon-install-helpers.ts:60-63`
  - body: Current main passes `extraPathDirs: resolveDaemonNodeBinDir(nodePath)` so supervised node services keep the selected node toolchain on PATH. This PR's node install call omits that argument, so a rebase that keeps this shape can regress sibling tools like npm or pnpm for node services.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.gateway.ts:413`
  - body: This is a user-facing feature that adds a gateway config field and changes installed service behavior, but the PR head does not update `CHANGELOG.md`. Add a single-line entry under the active version without bot attribution.
  - confidence: 0.78
