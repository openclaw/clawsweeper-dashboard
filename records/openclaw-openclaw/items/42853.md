## Summary

Keep open. The linked Linux/macOS reports make the heap-limit work useful and current main plus the latest release still lack a supported Gateway heap config, but this PR needs code changes, maintainer policy review, and contributor real behavior proof before it can merge.

## What This Changes

Adds `gateway.nodeMaxOldSpaceMb` config/schema/help, generated `NODE_OPTIONS` heap flags for gateway and node service environments, install-plan threading, a wizard prompt, and service-env tests.

## Best Possible Solution

Ship a refreshed or replacement PR that adds a typed heap-only Gateway/node service config, generates only an approved `--max-old-space-size` value, updates the active setup/config surface, preserves PATH/TLS hardening, and proves regenerated service metadata across supported daemon targets.

## Reproduction Assessment

Yes at source level: current main and the latest release have no `gateway.nodeMaxOldSpaceMb`, schema/help metadata for it, or generated service heap `NODE_OPTIONS`. I did not live-reproduce the long-running Linux/macOS OOM workloads in this read-only pass.

## Solution Assessment

No. A typed heap-limit config is the right direction, but this branch should generate only an approved heap flag, detect both Node flag spellings, move the prompt to the active setup wizard, preserve current node-service PATH behavior, and include real install/update proof before merge.

- **[P1] Generate only the managed heap flag:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends the caller's ambient `NODE_OPTIONS`, and `buildCommonServiceEnvironment` writes that value into generated service metadata.   - confidence: 0.9

  - body: Current main uses `src/wizard/setup.gateway-config.ts` from the setup flow, and `src/wizard/onboarding.gateway-config.ts` is no longer present. After rebase, adding the prompt only to the old onboarding path would not expose or clear the new setting through the active setup flow.
  - confidence: 0.9
