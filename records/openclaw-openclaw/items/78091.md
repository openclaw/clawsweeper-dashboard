## Label Justifications

- `P2`: The issue affects Open-WebUI session continuity and session-store growth for a bounded Gateway HTTP integration path, without evidence of a release-blocking outage.
- `impact:session-state`: One logical chat can split into many independent backend sessions, losing natural transcript continuity and bloating session state.
- `maturity:stable`: The affected Gateway runtime HTTP APIs are part of `gateway-runtime`, which is scored M4 Stable with Stable HTTP APIs in `qa/maturity-scores.yaml`.

## Summary

Keep open: current main and v2026.6.10 still fall back to a random OpenAI-compatible session when no explicit session key or OpenAI `user` is present, and the direct closing PR is open, conflicting, and not yet proof-ready.

## What This Changes

Reuse a stable OpenClaw session for one Open-WebUI chat instead of creating a new random OpenAI-compatible session for each message.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/78124, or replace it with an equivalent narrow Gateway fix that supports Open-WebUI's current chat-id signal, preserves documented precedence, updates docs/tests, and includes real multi-turn proof.

## Reproduction Assessment

Yes, at source level: `/v1/chat/completions` reaches `resolveSessionKey`, which uses `randomUUID()` when neither `x-openclaw-session-key` nor OpenAI `user` is present. I did not run a live Open-WebUI setup in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the contract. Header-based correlation is plausible, but the best fix must support Open-WebUI's actual forwarded chat-id header, preserve existing precedence unless intentionally changed, and avoid a shared default-session fallback.
