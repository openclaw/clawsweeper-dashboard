## Summary

Keep open: current main and the latest shipped release still use a random OpenAI-compatible session when neither `x-openclaw-session-key` nor OpenAI `user` is supplied, upstream Open-WebUI has a stable forwarded chat-id header, and the linked implementation PR is still open rather than merged.

## What This Changes

The issue requests stable Open-WebUI chat-to-session correlation for the OpenAI-compatible `/v1/chat/completions` endpoint instead of per-request random sessions.

## Best Possible Solution

Define and document a stable-session contract for OpenAI-compatible clients, add regression coverage for Open-WebUI's forwarded chat id, and require real multi-turn proof that one Open-WebUI chat reuses one OpenClaw session.

## Reproduction Assessment

Yes at source level: current main falls back to `randomUUID()` for Chat Completions requests without `x-openclaw-session-key` or OpenAI `user`. I did not run a live Open-WebUI container in this read-only review.

## Solution Assessment

Yes in direction: explicit client-supplied chat/session ids are the maintainable fix, while a global default session or IP/user-agent hash would risk merging unrelated conversations. The current implementation path is the open linked PR or a replacement that handles Open-WebUI's actual header and documents precedence.
