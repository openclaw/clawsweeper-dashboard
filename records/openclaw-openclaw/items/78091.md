## Summary

Keep open: current main and the latest stable release still create a random OpenAI-compatible session when no explicit OpenClaw session key or OpenAI `user` is supplied, and the direct closing implementation PR is still open rather than merged.

## What This Changes

The issue asks OpenClaw’s OpenAI-compatible gateway to reuse a stable session for one Open-WebUI chat instead of creating a new random session per message.

## Best Possible Solution

Land or replace the linked PR with a documented stable-session contract for OpenAI-compatible clients, including Open-WebUI’s actual forwarded chat-id header, precedence tests, and redacted multi-turn proof.

## Reproduction Assessment

Yes, at source level: current main falls back to `randomUUID()` for Chat Completions requests without `x-openclaw-session-key` or OpenAI `user`. I did not run a live Open-WebUI container in this read-only review.

## Solution Assessment

Yes in direction: an explicit client-supplied chat/session id is the maintainable fix, while a global default session or IP/user-agent hash would risk merging unrelated conversations. The open implementation path should handle Open-WebUI’s actual header and document precedence.
