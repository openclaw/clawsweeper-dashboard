## Summary

Keep open: current main and the latest release still create a random OpenAI-compatible session when neither `x-openclaw-session-key` nor OpenAI `user` is supplied, and the linked fixing PR is open, conflicting, and still needs repair/proof.

## What This Changes

The issue requests stable `/v1/chat/completions` session reuse for Open-WebUI chats instead of creating a new random OpenClaw session per message.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/78124, or replace it with an equivalent narrow Gateway fix that supports Open-WebUI's real chat-id header, preserves existing session precedence, updates docs/tests, and includes real multi-turn proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `/v1/chat/completions` reaches `resolveSessionKey`, which uses `randomUUID()` when neither an explicit OpenClaw session key nor OpenAI `user` is present. I did not run a live Open-WebUI repro in this read-only review.

## Solution Assessment

Yes, the right end state is a narrow Gateway session-correlation fix, but the open PR is not best as-is because it misses Open-WebUI's default header and still needs precedence/docs/proof cleanup.
