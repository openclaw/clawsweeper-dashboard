## Label Justifications

- `P2`: The bug affects session continuity and session-store growth for Open-WebUI-style OpenAI-compatible clients, with a bounded Gateway HTTP blast radius.
- `impact:session-state`: The issue creates many independent backend sessions for one logical chat and loses natural conversation continuity.

## Summary

Keep open: current main and v2026.6.8 still fall back to a random OpenAI-compatible session when neither `x-openclaw-session-key` nor OpenAI `user` is present, and the direct fixing PR remains open, conflicting, and blocked on proof/contract repair.

## What This Changes

The issue asks the OpenAI-compatible chat endpoint to derive a stable Open-WebUI conversation session instead of creating a new random session per message.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/78124, or replace it with an equivalent narrow Gateway fix that supports Open-WebUI's real chat-id header, preserves documented session precedence, updates tests/docs, and includes real multi-turn proof.

## Reproduction Assessment

Yes, at source level: `/v1/chat/completions` reaches `resolveSessionKey`, which calls `randomUUID()` when neither `x-openclaw-session-key` nor OpenAI `user` is present. I did not run a live Open-WebUI repro in this read-only review.

## Solution Assessment

Yes for the narrow direction, but not via the current PR as-is. The maintainable fix is a small Gateway session-correlation contract that reads Open-WebUI's real forwarded chat id, preserves existing precedence, and proves the real multi-turn flow.
