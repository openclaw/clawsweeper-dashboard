## Label Justifications

- `P2`: The issue affects session continuity and session-store growth for Open-WebUI-style OpenAI-compatible clients, with a bounded Gateway HTTP blast radius.
- `impact:session-state`: One logical chat can split into many independent backend sessions, losing natural transcript continuity and bloating session state.

## Summary

Keep open: current main and the latest release still use a random OpenAI-compatible session when no explicit OpenClaw session key or OpenAI `user` is present, and the direct closing PR is still open and unmerged.

## What This Changes

The issue asks OpenClaw to reuse a stable OpenAI-compatible session for Open-WebUI chat turns instead of creating a new random backend session per message.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/78124, or replace it with an equivalent narrow Gateway fix that supports Open-WebUI's actual chat-id header, preserves documented session precedence, updates tests/docs, and includes real multi-turn proof.

## Reproduction Assessment

Yes, at source level: `/v1/chat/completions` reaches `resolveSessionKey`, which uses `randomUUID()` when neither `x-openclaw-session-key` nor OpenAI `user` is present. I did not run a live Open-WebUI setup in this read-only review.

## Solution Assessment

Unclear for the exact requested shape: header-based correlation is plausible, but the best fix should support Open-WebUI's actual forwarded chat-id header, preserve existing precedence unless maintainers approve a break, and avoid a shared default session fallback.
