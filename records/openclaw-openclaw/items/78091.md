## Summary

Keep open: current main still generates a random OpenAI-compatible session when the request lacks an explicit OpenClaw session key or OpenAI `user`, and the linked closing PR is still open rather than merged.

## What This Changes

Request that OpenAI-compatible Chat Completions reuse a stable OpenClaw session for each Open-WebUI chat instead of creating a random session per request.

## Best Possible Solution

Review and land the linked PR or a replacement that derives the session from Open-WebUI's stable forwarded chat id, documents the contract, and proves multi-turn reuse before closing this issue.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection shows a Chat Completions request without `x-openclaw-session-key` or `user` uses `randomUUID()`, and current docs describe stateless per-request behavior. I did not run a live Open-WebUI container, but upstream source shows the stable chat id is forwarded as `X-OpenWebUI-Chat-Id` when enabled.

## Solution Assessment

The maintainable direction is an explicit client-supplied chat/session id, not a global default session or IP/user-agent hash that could mix unrelated chats. The open linked PR is the right review path, but it still needs to honor Open-WebUI's actual header contract and provide real behavior proof.
