## Summary

Keep open: current main/source and docs confirm the Chat Completions endpoint is intentionally stateless unless the caller supplies `user` or `x-openclaw-session-key`, so the Open-WebUI persistence request needs a deliberate compatibility design rather than cleanup closure.

## What This Changes

Request persistent session reuse for Open-WebUI chats using OpenClaw's OpenAI-compatible Chat Completions endpoint instead of creating per-request `agent:<agentId>:openai:<uuid>` sessions.

## Best Possible Solution

Define an explicit Chat Completions session-correlation contract for third-party clients, preferably using a stable per-chat request field or header when present and documenting safe behavior when no per-chat identity exists.

## Reproduction Assessment

Yes for the observed session proliferation path: current source deterministically creates a random `openai:<uuid>` session whenever a Chat Completions request omits both `x-openclaw-session-key` and `user`. I did not run a live Open-WebUI container, so the upstream request shape still needs validation.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. Reusing a stable chat id would be maintainable if Open-WebUI sends one, but falling back to a global/default session or IP/user-agent hash is a product and safety decision because it can mix unrelated conversations.
