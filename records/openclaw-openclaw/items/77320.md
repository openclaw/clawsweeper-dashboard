## Summary

Keep open: current main still documents and tests message-tool-only group/channel suppression as intentional, and the remaining ask is a product/observability decision rather than a fixed Slack transport bug.

## What This Changes

The issue requests Slack group/channel diagnostics or fallback behavior when a `message_tool` visible-reply turn ends with final text but no `message(action=send)` tool call.

## Best Possible Solution

Decide the policy first; the narrowest maintainable end state is a scoped turn-end diagnostic for suppressed non-empty final text with no successful message-tool delivery, while preserving tool-only privacy unless maintainers explicitly opt into fallback delivery.

## Reproduction Assessment

Yes, source-reproducible: current dispatch code and tests show group/channel `message_tool_only` mode processes the turn while suppressing block/final Slack-channel delivery. I did not run a live Slack/claude-cli reproduction for the nondeterministic model turn shape.

## Solution Assessment

Unclear: the hard diagnostic is likely the safest first step, but auto-promote or a fallback mode would change documented private-by-default room behavior. Maintainers should choose the log level, trigger conditions, and whether fallback delivery is allowed before implementation.
