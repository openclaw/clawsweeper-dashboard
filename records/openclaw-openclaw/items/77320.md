## Summary

Keep open: current main intentionally suppresses automatic Slack group/channel final replies in message-tool mode, but this remains a legitimate product/observability request because only verbose pre-run suppression logging exists and there is no terminal diagnostic or fallback for a completed non-empty private final answer.

## What This Changes

Request a Slack/group-chat diagnostic or fallback when a `message_tool`-only turn finishes with non-empty final text but no `message(action=send)` delivery.

## Best Possible Solution

Preserve the tool-only default; if maintainers want better observability, add a scoped terminal diagnostic for suppressed non-empty final text before considering any opt-in fallback delivery mode.

## Reproduction Assessment

Yes, source-reproducible: current main resolves Slack group/channel turns to `message_tool_only` and suppresses automatic final/block delivery. I did not establish the nondeterministic live Claude `[thinking,text]` turn in this read-only review.

## Solution Assessment

Unclear: a terminal diagnostic is likely the narrowest maintainable improvement, while auto-promotion or a new fallback mode changes the documented privacy semantics and needs maintainer product judgment.
