## Summary

Keep open: current main still has the shared `message_tool_only` suppression path, and the remaining request is unresolved observability or fallback semantics rather than an already-fixed Slack transport bug. The issue is also the canonical thread used to close a duplicate observability report, while the related diagnostic PR at https://github.com/openclaw/openclaw/pull/81018 is still open with review blockers.

## What This Changes

The issue asks OpenClaw to make group/channel `message_tool` reply suppression diagnosable or safer when an agent finishes with text but no message-tool send.

## Best Possible Solution

Preserve the tool-only privacy contract, but land or replace the open diagnostic PR with a default-visible, deduped warning or diagnostic event for suppressed non-empty final text with no committed message-tool delivery.

## Reproduction Assessment

Yes, at source level: current main resolves group/channel turns to `message_tool_only`, suppresses ordinary final delivery, and has tests confirming no visible final is queued. The comments add fresh Slack, Telegram, and Discord live examples, but I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a default-visible diagnostic is the narrowest maintainable improvement, while reaction signals, auto-promotion, or a fallback delivery mode need an explicit privacy/product decision.
