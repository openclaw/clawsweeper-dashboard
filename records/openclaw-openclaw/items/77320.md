## Summary

Keep open: current main still intentionally keeps group/channel final text private in `message_tool` visible-reply mode, and the remaining request is for new observability or fallback semantics rather than a fixed Slack transport bug.

## What This Changes

The issue requests diagnostics or fallback delivery when Slack group turns in `visibleReplies="message_tool"` finish with final text but no message-tool send.

## Best Possible Solution

Decide the product semantics first; if observability is desired, add a scoped terminal diagnostic for suppressed non-empty final text with no successful message-tool delivery while preserving the private-by-default room contract.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show group/channel `message_tool_only` mode suppresses automatic final/block delivery while still running the agent. I did not establish a deterministic live Slack/claude-cli reproduction in this read-only review.

## Solution Assessment

Unclear: the hard diagnostic is the narrowest likely improvement, but auto-promotion or a new fallback mode would change documented privacy and visible-reply semantics. Maintainers should choose the log level and fallback policy before implementation.
