## Summary

Keep open: current main still intentionally suppresses automatic Slack group/channel replies in `message_tool` mode and has no terminal diagnostic or fallback for a final-text turn with no message-tool send. The remaining ask is an observability or fallback policy change, not behavior already implemented on main.

## What This Changes

Request a Slack/group-chat diagnostic or fallback when a `message_tool`-only turn ends with final text but no message-tool send.

## Best Possible Solution

Decide the product semantics first; if observability is desired, add a scoped terminal diagnostic for suppressed non-empty final text with no message-tool delivery while preserving the default private-by-design behavior.

## Reproduction Assessment

Yes, source-reproducible: current dispatch code and tests show group/channel `message_tool_only` mode suppresses final/block replies while still running the agent. I did not establish a deterministic live Slack/claude-cli `[thinking,text]` reproduction in this read-only review.

## Solution Assessment

Unclear: the hard diagnostic is the narrowest likely improvement, but auto-promote or fallback delivery would change documented private-by-default group/channel semantics. Maintainers should choose the warning level and fallback policy before implementation.
