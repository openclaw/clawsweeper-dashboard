## Summary

Keep open: current main still intentionally suppresses automatic group/channel final delivery under message-tool-only mode, and the remaining gap is the requested operator-visible diagnostic rather than an already-fixed Slack transport bug.

## What This Changes

The issue requests Slack group-thread observability or fallback delivery when `visibleReplies="message_tool"` suppresses a completed assistant text turn with no `mcp__openclaw__message` call.

## Best Possible Solution

Add a default warn-level diagnostic for completed message-tool-only turns that have visible assistant text but no committed message-tool delivery; keep auto-promotion or a new fallback mode as separate product work.

## Reproduction Assessment

No; this review did not live-reproduce the Slack/Claude turn shape. Source and tests do show current main suppresses group/channel final delivery under message-tool-only mode, and the issue supplies production logs for the Slack symptom.

## Solution Assessment

Yes, for a diagnostic-only change: a warn-level no-message-tool/no-delivery log is narrow and preserves the documented privacy default. Auto-promotion or a new `visibleReplies` fallback mode would need separate product judgment.
