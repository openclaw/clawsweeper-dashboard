## Summary

Keep open: this is still the canonical cross-channel `visibleReplies="message_tool"` observability/fallback issue. Current main preserves the private-by-default behavior and still lacks the requested default-visible diagnostic or fallback, while the related open fix PR at https://github.com/openclaw/openclaw/pull/81018 is not merged and has unresolved review/proof concerns.

## What This Changes

Add a cross-channel diagnostic or fallback when a group/channel tool-only visible-reply turn ends with non-empty assistant text and no successful message-tool send.

## Best Possible Solution

Keep this as the canonical issue and land a shared dispatch or runner-level default-visible diagnostic first; treat auto-promotion or a new fallback mode as an explicit product/privacy decision.

## Reproduction Assessment

Yes, at source level: current main resolves group/channel turns to `message_tool_only`, suppresses ordinary final delivery, and tests assert no final reply is queued. The issue comments add live Slack, Telegram, and Discord examples, but I did not run a live channel repro here.

## Solution Assessment

Unclear: a non-verbose structured diagnostic is the narrowest maintainable first step, while auto-promotion or a new fallback mode needs maintainer approval because it can reveal private finals in rooms.
