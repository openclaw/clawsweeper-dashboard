## Summary

Keep open: current main still suppresses ordinary group/channel final replies in `message_tool_only` mode and does not implement the requested terminal diagnostic or fallback behavior; recent Slack and Telegram reports show the same source-policy path, but changing warning/fallback semantics needs maintainer product judgment.

## What This Changes

Request a diagnostic or fallback when group/channel `message_tool` visible-reply mode suppresses a text-only final assistant turn with no message-tool send.

## Best Possible Solution

Add a scoped structured diagnostic for non-empty final text with no committed message-tool delivery in `message_tool_only` mode, then decide separately whether any opt-in fallback or auto-promotion is acceptable.

## Reproduction Assessment

Yes, source-reproducible: current main resolves group/channel turns to `message_tool_only`, skips ordinary final delivery under `suppressDelivery`, and has tests asserting no final reply is queued. I did not run a live Slack or Telegram reproduction in this read-only review.

## Solution Assessment

Unclear: a warning-only diagnostic is the narrowest likely improvement, but auto-promote or a new fallback mode would change documented private-by-default room behavior. Maintainers should choose the policy before implementation.

- A warn/error for every private final answer could create noise in intentionally quiet group/channel rooms; maintainers should choose log level and dedupe/rate-limit behavior.

- The exact `[thinking,text]` live turn shape is model-dependent; source tests can prove the policy path, but live Slack/Telegram proof would still be useful for any behavior-changing fix.
