## Summary

Keep open: current main still treats non-tool final text in group/channel `message_tool` mode as intentionally private output, and the requested terminal diagnostic or fallback semantics are not implemented yet.

## What This Changes

The issue requests Slack group-thread observability or fallback delivery when a turn produces final text without any `mcp__openclaw__message` tool call under `messages.groupChat.visibleReplies="message_tool"`.

## Best Possible Solution

Decide the product semantics first; the narrowest maintainable change is a scoped terminal diagnostic for suppressed non-empty final text with no message-tool send while preserving private-by-default group behavior.

## Reproduction Assessment

Yes, source-reproducible: current resolver, Slack dispatch integration, suppression code, and tests show group/channel `message_tool_only` mode suppresses automatic final/block replies while still running the agent. I did not run a live Slack/claude-cli reproduction in this read-only review.

## Solution Assessment

Unclear: the hard diagnostic is likely the smallest useful improvement, but auto-promote delivery or a new fallback mode changes documented room privacy semantics. Maintainers should choose the diagnostic level and fallback policy before implementation.
