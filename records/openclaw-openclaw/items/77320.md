## Summary

Keep open: current main still suppresses ordinary group/channel final replies under `message_tool_only` and only emits a verbose-only suppression line, while recent Slack, Telegram, and Discord comments show the operator-visible diagnostic gap remains. The active diagnostic PR at https://github.com/openclaw/openclaw/pull/81018 is not merged and still needs revision/review.

## What This Changes

Request a cross-channel diagnostic or fallback when a group/channel turn in `messages.groupChat.visibleReplies="message_tool"` produces final text but no successful message-tool send.

## Best Possible Solution

Add a shared, default-visible diagnostic for suppressed non-empty finals under `message_tool_only` with no successful message-tool delivery, then separately decide whether auto-promotion or a new fallback mode should exist.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main resolves group/channel turns to `message_tool_only` and skips ordinary final payload delivery, and the issue has live Slack, Telegram, and Discord examples. I did not run a live Slack/Telegram/Discord repro in this read-only review.

## Solution Assessment

Unclear for the full request: a diagnostic-first fix is the narrowest maintainable path, but auto-promotion or fallback modes need an explicit privacy/product decision. The current main behavior is documented but lacks the requested normal operator-visible signal.
