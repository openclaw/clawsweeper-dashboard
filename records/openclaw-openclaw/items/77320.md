## Summary

Keep open: current main intentionally suppresses automatic Slack group/channel final replies in message-tool-only mode, and the requested diagnostic or fallback semantics need maintainer UX/privacy judgment rather than conservative cleanup closure.

## What This Changes

Requests Slack group-channel observability or fallback handling when `messages.groupChat.visibleReplies="message_tool"` suppresses a non-empty final assistant text turn with no message-tool call.

## Best Possible Solution

Decide the product semantics first; if observability is desired, add a narrowly scoped terminal diagnostic for suppressed non-empty final text with no successful message-tool delivery while preserving private-by-default room behavior.

## Reproduction Assessment

Yes, source-reproducible: current Slack dispatch passes `message_tool_only` into the shared reply path, and current tests assert group/channel final replies are not automatically sent in that mode. I did not run a live Slack/claude-cli reproduction.

## Solution Assessment

Unclear: the hard diagnostic is the narrowest likely improvement, but auto-promote or a fallback mode would change documented privacy semantics. Maintainers should choose warning level, scope, and fallback policy before implementation.
