## Summary

Keep open: current main still intentionally suppresses normal final replies in group/channel `message_tool_only` mode, and the remaining request is a cross-channel observability or fallback policy decision that is not merged yet. Recent comments add Telegram and Discord live reproductions, and the related diagnostic PR is still open with a review blocker.

## What This Changes

Request a gateway-level diagnostic or fallback for group/channel text-only final replies suppressed under `messages.groupChat.visibleReplies="message_tool"`.

## Best Possible Solution

Preserve the tool-only privacy contract, but add a default-visible structured diagnostic for suppressed substantive final text with no successful message-tool delivery; treat auto-promotion or reaction signals as separate product decisions.

## Reproduction Assessment

Yes at source level: configure a group/channel turn with `message_tool_only`, return a non-reasoning text final without `deliverDespiteSourceReplySuppression`, and current dispatch skips final delivery. I did not run a live Slack/Telegram/Discord reproduction.

## Solution Assessment

Unclear for the full request: the shared dispatch boundary is the right place for diagnostics, but fallback delivery, reactions, and log severity need maintainer policy. The open diagnostic PR is close in scope but verbose-only logging does not meet the hard diagnostic request.
