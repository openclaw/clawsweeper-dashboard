## Summary

Keep open: current main still matches the reported behavior, and the remaining decision is whether to add diagnostics or fallback semantics for a documented message-tool-only mode. #77260 is related but tracks command/system replies, not Slack agent-turn final text.

## What This Changes

The issue requests diagnostics or fallback delivery when a Slack group-chat agent turn ends with non-empty final text but no `mcp__openclaw__message` call under `messages.groupChat.visibleReplies="message_tool"`.

## Best Possible Solution

Decide the product semantics first; the narrowest likely end state is a scoped terminal diagnostic for suppressed non-empty final text while preserving private-by-default group behavior.

## Reproduction Assessment

Yes, source-reproducible: current source and tests show Slack group/channel `message_tool` mode suppresses automatic final/block replies while still running the agent. I did not run a live Slack/claude-cli reproduction, and the exact model turn shape is nondeterministic.

## Solution Assessment

Unclear: the hard diagnostic is probably the safest first step, but fallback delivery or a new `message_tool_or_last_text` mode would be a product/privacy policy change rather than a narrow transport bug fix.
