- `P3`: This is an optional new local voice-provider capability rather than a broken documented workflow.
- `impact:auth-provider`: The request is about provider routing and avoiding cloud-provider credential requirements in Talk Mode.

## Summary

Close as ClawHub/provider-plugin work: the local/self-hosted STT use case is valid, but current main already has realtime transcription and realtime voice plugin seams, and Talk consumes registered provider configs without a missing core API identified.

## What This Changes

The issue requests local or self-hosted STT endpoint support for realtime Talk Mode voice paths so users can avoid cloud API keys.

## Best Possible Solution

Publish a ClawHub realtime voice or realtime transcription provider plugin for the local ASR engine, and open a narrower core issue only if that plugin exposes a concrete missing SDK hook.

## Reproduction Assessment

Not applicable; this is a feature request for a new local/self-hosted provider path, not a report of broken current behavior. Source and docs confirm the existing route is plugin-provider discovery rather than a built-in local STT server.

## Solution Assessment

No for the proposed core/baseUrl direction; adding local ASR endpoint policy to the bundled OpenAI provider would make optional provider behavior part of core. The cleaner solution is a ClawHub provider plugin using the existing realtime transcription or realtime voice SDK seams.
