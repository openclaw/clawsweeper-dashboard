## Summary

Keep open: the PR targets a real replay-sanitizer bug, but the current predicate is too broad and would also catch Anthropic-compatible providers that current main intentionally keeps out of signature-preserving replay. The PR also has focused Vitest evidence, not real after-fix replay proof from a running setup.

## What This Changes

The PR exports the Anthropic API helper, changes replay-history sanitization to always strip invalid thinking signatures for Anthropic/Bedrock model APIs, and adds three sanitizer regression tests.

## Best Possible Solution

Land a replay-side sanitizer only after the signed-thinking predicate is scoped to providers or policies that actually require replay signatures, with regressions for Kimi/GitHub Copilot compatibility and separate tracking for write-side prevention if it remains out of scope.

## Reproduction Assessment

Yes from source for both the reported sanitizer branch and the PR regression: current main skips `stripInvalidThinkingSignatures` when `policy.preserveSignatures` is false, while the PR’s new `isAnthropicApi` gate would also match non-preserving `anthropic-messages` providers such as Kimi. I did not run a live interrupted Anthropic continuation in this read-only review.

## Solution Assessment

No. Replay-side sanitization is the right narrow layer for the defensive fix, but the current model-API predicate is too broad; the signed-thinking requirement should come from provider identity or an explicit replay-policy capability, with non-signature providers still gated by `policy.preserveSignatures`.
