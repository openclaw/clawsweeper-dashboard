## Summary

Keep open for maintainer review: the patch is a narrow, plausible fix for the malformed context-engine assemble result crash, but it intentionally changes an external plugin failure from downstream crash to warn-and-fallback behavior, so compatibility semantics and validation scope should be accepted before merge.

## What This Changes

The PR adds runtime validation for `assembleHarnessContextEngine()` results and tests malformed context-engine assemble payloads that omit or misuse the required `messages` array.

## Best Possible Solution

Land the harness-level `messages` array guard if maintainers accept messages-only validation and warn-and-fallback recovery; otherwise extend or relocate validation at the context-engine boundary before merge.

## Reproduction Assessment

Yes. Current main returns assemble results unchecked and callers consume `assembled.messages`, while the PR body includes before/after terminal output using a malformed lossless-claw-shaped result.

## Solution Assessment

Mostly yes. Validating the required `messages` array at the shared harness boundary is narrow and covers embedded and Codex callers, but maintainers should decide whether registry quarantine or `estimatedTokens` validation belongs in the same change.
