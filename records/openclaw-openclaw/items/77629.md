## Summary

Keep open: the docs-only PR is useful, but the new hook guidance overstates `activation.onChannels` as a Gateway startup hook-loading trigger for chat-path hooks, which current source does not support.

## What This Changes

The PR adds plugin docs explaining chat-path hook runtime activation and expands the manifest `onCapabilities` table entry for hook-only plugins.

## Best Possible Solution

Land the docs clarification after rewording the hook-loading paragraph to name the actual Gateway startup triggers and, ideally, include a small `activation.onCapabilities: ["hook"]` manifest example.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the docs defect: the PR text names `onChannels`, while the current startup planner only treats hook capability metadata or explicit hook policy as hook startup intent.

## Solution Assessment

No; docs-only is a reasonable way to address the linked report, but this patch should not present `activation.onChannels` as sufficient for hook-only chat-path startup unless it adds a precise caveat about configured-channel ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Remove onChannels from hook startup guidance:** `docs/plugins/hooks.md:134`
  - body: The new hook docs tell hook-only/non-capability plugins they can load for chat-path hooks by matching `activation.onChannels`, but Gateway startup hook loading only checks `activation.onCapabilities: ["hook"]` or explicit hook policy before admitting hook plugins.
