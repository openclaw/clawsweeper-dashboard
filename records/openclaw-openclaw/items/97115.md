## Label Justifications

- `impact:message-loss`: The cron path can accept an empty or incomplete embedded result and finish without the useful answer the fallback chain should have produced.

## Summary

Keep open: current main still lets the isolated cron fallback path accept returned empty or incomplete embedded results as successful attempts because that path omits the existing embedded-result classifier, while sibling agent paths pass it.

## What This Changes

The issue reports that isolated cron model fallback accepts zero-token or aborted model results as success instead of trying configured fallback models.

## Best Possible Solution

Reuse the existing embedded result fallback classifier and exhausted-result merge helper in isolated cron, with focused regression coverage; leave broader timeout-budget semantics for separate proof.

## Reproduction Assessment

Yes for the empty-result fallback gap: current source shows isolated cron omits result classification while runWithModelFallback only retries returned result-level failures when a classifier is supplied; no live Gemini cron run was executed.

## Solution Assessment

Yes for the narrow classifier gap: matching the normal agent command path is the best bounded fix, while changing shared cron timeout signal behavior should wait for separate proof.
