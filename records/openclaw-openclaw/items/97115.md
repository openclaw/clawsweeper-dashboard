## Summary

Keep open: current main and latest release still leave the isolated cron fallback path without result-level classification, and current main still forwards the cron abort signal through the fallback chain; open closing PRs now own the repair path.

## What This Changes

The issue reports isolated cron `agentTurn` fallback chains failing when cron timeouts share an aborted signal or empty/zero-token embedded results are accepted as successful model output.

## Best Possible Solution

Land one focused fix that covers both result-level embedded-result classification and cron timeout abort inheritance, with regression coverage for empty/incomplete results, timeout recovery, and operator-cancel/restart behavior; then close this issue and reconcile the partial related reports.

## Reproduction Assessment

Yes. Source inspection on current main shows the cron path omits the classifier hooks that `runWithModelFallback` requires for returned-result failover, and the issue provides a focused failing test shape for that path; the timeout-abort half is also source-plausible from the shared abort signal.

## Solution Assessment

Yes for the desired fix shape: reuse the existing embedded result classifier and adjust cron timeout-abort handling rather than adding new config or provider-specific policy. The exact PR to land still needs maintainer review because https://github.com/openclaw/openclaw/pull/97255 covers both dimensions while https://github.com/openclaw/openclaw/pull/96529 and https://github.com/openclaw/openclaw/pull/97132 cover only the classifier gap.
