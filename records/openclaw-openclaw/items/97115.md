## Summary

Keep open. Current main still has a cron-specific fallback gap: isolated cron uses `runWithModelFallback` without the embedded-result classifier that normal agent turns use, so an empty or zero-output successful-looking run can stop the fallback chain before the next candidate is tried.

## What This Changes

The issue asks for isolated cron agentTurn fallback to continue past primary timeouts and zero-token empty responses instead of treating the bad fallback as success or persisting it in session state.

## Best Possible Solution

Keep this open and first align isolated cron with the existing embedded-result fallback classifier, then separately settle any cron timeout-budget semantics with focused regression coverage.

## Reproduction Assessment

Yes at source level, not live: normal agent turns classify empty embedded results before accepting a fallback candidate, while the isolated cron executor omits that classifier. The Gemini-specific zero-token payload still needs a real cron proof run after a proposed fix.

## Solution Assessment

The narrow best first fix is to reuse the existing embedded-result fallback-classification contract in isolated cron. Changing shared AbortController wall-clock timeout semantics should be a separate maintainer-reviewed timeout policy decision.
