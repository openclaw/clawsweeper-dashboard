## Summary

Keep open: current main and v2026.6.10 still have the returned-result cron fallback gap, and the open fix PR has not merged; the broader timeout/cache subclaims also need follow-up rather than cleanup closure.

## What This Changes

The issue reports that isolated cron `agentTurn` fallback chains can stop on empty/zero-token embedded results, abort later candidates with a shared timeout signal, and persist a bad fallback model in cron session state.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/96529 for the returned-result fallback gap, then re-test the timeout and session-state subclaims on current main before closing or splitting them.

## Reproduction Assessment

Yes for the returned-result fallback gap at source level: current main omits cron result classification while the fallback helper only retries ok returned results when a classifier is supplied. The shared timeout and cache-poisoning parts are plausible but not fully reproduced in this read-only pass.

## Solution Assessment

Yes for the returned-result slice: reusing the existing embedded result classifier and exhaustion merge helper in the cron embedded path is the narrowest maintainable fix. The timeout/cache portions need separate proof after the linked PR lands rather than being folded into the same fix blindly.
