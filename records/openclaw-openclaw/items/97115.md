## Label Justifications

- `P1`: The issue affects scheduled agent work with configured fallback chains and can make cron jobs silently fail to produce or deliver expected output.
- `impact:message-loss`: The reported cron path can finish without useful assistant output or delivery when a fallback returns an empty completed result.
- `impact:auth-provider`: The bug is in provider/model fallback classification and routing across configured fallback candidates.
- `impact:session-state`: The report includes poisoned cron session model state after a bad fallback candidate is accepted.

## Summary

Keep open: current main still lets isolated cron accept an embedded empty/zero-token result as a successful fallback attempt because the cron path does not pass the existing embedded-result fallback classifier, while the normal agent path does.

## What This Changes

The issue asks OpenClaw to make isolated cron agentTurn fallback chains retry after empty/zero-token provider results and avoid aborting later fallback attempts with a shared cron timeout signal.

## Best Possible Solution

Reuse the existing embedded-agent fallback classifier in isolated cron and add focused cron regression coverage; leave broader cron timeout budget semantics to a separate maintainer-reviewed change if still reproducible.

## Reproduction Assessment

Yes for the empty-result fallback gap: current source shows isolated cron omits the classifier that existing tests prove is required to retry empty embedded results; no live Gemini cron run was executed.

## Solution Assessment

Yes for the narrow classifier gap: matching the normal agent command path is the best bounded fix, while changing the shared cron timeout signal should wait for separate proof.
