## Summary

Keep open. The PR targets a real missing per-agent foreground lane capability, but the current head still has a security-sensitive ingress lane override and two resolver/defaults defects that should block merge.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane`, wires gateway and auto-reply agent runs through resolved command lanes, applies configured lane concurrency, updates schema/docs/changelog, and adds tests.

## Best Possible Solution

Keep the per-agent/default command-lane direction, but make network-facing lane selection server-controlled or allowlisted, keep default concurrency attached only to the default lane, align resolved config inheritance, rebase, and rerun the changed gate.

## Reproduction Assessment

Yes. #43235 and #42686 provide the shared-`main` lane bottleneck scenario, and current main plus the PR head make the PR defects source-reproducible by static inspection of lane precedence, queue lane creation, and config fallback behavior.

## Solution Assessment

No, not as proposed. The new config surface is a reasonable narrow direction, but the ingress override policy and resolver/defaults behavior need to be corrected before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The configured per-agent/default lane is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` still accepts arbitrary strings and the command queue creates unknown lanes on demand, so callers can choose unique lanes instead of the configured caps; ignore request lanes here or validate them against a server-owned allowlist.
  - confidence: 0.88
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: When defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit it when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the same defaults fallback pattern as nearby inheritable fields.
  - confidence: 0.8

## Security Review

Status: needs_attention
