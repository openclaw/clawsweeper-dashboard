## Summary

Keep open. Current main still lacks per-agent/default command-lane configuration, and this PR targets the right user problem, but the proposed patch is not safe to merge because it preserves arbitrary network-facing lane overrides and has two concrete resolver/config-default defects already reflected in review discussion.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config/schema/docs, routes gateway and auto-reply agent runs through the configured lane, registers configured lane concurrency, adds tests, updates generated config baseline hashes, and adds a changelog entry.

## Best Possible Solution

Keep the feature direction, but make gateway ingress lane selection server-controlled or restricted to a server-owned allowlist, fix command-lane concurrency inheritance so default caps stay with the default lane unless an agent explicitly sets its own cap, align the resolved agent config with defaults inheritance, then rerun the focused config/gateway/auto-reply tests and docs baseline checks before landing.

## Reproduction Assessment

Yes. The user problem is reproducible from #43235's Feishu multi-agent `lane=main` wait logs plus current main's shared global lane path; the PR defects are statically reproducible from the new resolver and ingress code without needing a full runtime run.

## Solution Assessment

No, not as proposed. The PR is aimed at the right core boundary for routed foreground agent runs, but the safest maintainable solution should not let network callers bypass configured lanes with arbitrary names and should not carry a default lane cap onto a different per-agent lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so authenticated clients can send unique lane names and run outside configured per-agent/default caps. Ignore ingress lane overrides or validate them against a server-owned allowlist.
  - confidence: 0.88
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so inherit `defaults.maxConcurrent` only when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to consumers of the resolved config; use the same defaults fallback pattern as nearby inheritable fields.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff leaves a network-facing lane override path that can bypass configured concurrency caps by creating arbitrary queue lanes.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: Gateway `AgentParamsSchema` accepts `lane` as an arbitrary string, and the command queue creates state for new lane names. Because the PR resolves `request.lane` before the configured per-agent/default lane, an authenticated client can choose unique lanes to avoid configured concurrency caps and increase concurrent work.
  - confidence: 0.88
