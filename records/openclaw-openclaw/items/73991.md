## Summary

Keep open. Current main still lacks the per-agent command-lane config surface, and this PR targets a real #43235 bottleneck, but the proposed patch is not safe to merge because gateway clients can still override configured lanes and one resolver fallback misapplies default lane concurrency to unrelated per-agent lanes.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` schema/docs/config handling, applies configured lane concurrency, and routes gateway plus auto-reply agent runs onto configured command lanes.

## Best Possible Solution

Keep the opt-in command-lane direction, but make gateway ingress lane selection server-controlled or allowlisted, keep default lane concurrency attached only to the default lane unless explicitly overridden, align `ResolvedAgentConfig` with the defaults fallback contract, and then revalidate schema/docs plus targeted gateway and auto-reply tests before merge.

## Reproduction Assessment

Yes. #43235 provides a concrete Feishu/main-lane wait scenario, and the PR defects are reproducible by static inspection of the diff together with current `AgentParamsSchema` and `command-queue` behavior.

## Solution Assessment

No. The PR uses reasonable config and queue surfaces, but the request-lane precedence and resolver fallback behavior make this implementation unsafe and incorrect as proposed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so configured per-agent/default caps are not authoritative for network-facing agent calls. Ignore request lanes at this boundary or validate them against a server-owned allowlist.
  - confidence: 0.86
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54-58`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff lets authenticated gateway callers choose arbitrary command lanes ahead of server-configured lane policy.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: The gateway agent RPC accepts arbitrary `lane` strings, and the command queue creates new lane states for unknown names. Because the PR resolves `request.lane` before the configured per-agent/default lane, authenticated callers can choose unique lanes instead of the intended server-owned lane policy.
  - confidence: 0.86
