## Summary

Keep open. Current main still lacks the requested per-agent/default command lane path, so the PR is not obsolete, but the head under review should not merge as-is because the new gateway path leaves configured lane isolation bypassable by arbitrary client-provided lane names and has concrete resolver/defaults defects.

## What This Changes

The PR adds opt-in `agents.defaults.commandLane` and `agents.list[].commandLane` config with schema/help/docs/changelog updates, gateway and auto-reply lane wiring, configured lane concurrency registration, and tests.

## Best Possible Solution

Keep the opt-in core queue/config direction, but make gateway ingress lane selection server-controlled or explicitly allowlisted, keep default concurrency attached only to the default lane unless an agent inherits that lane id, align `ResolvedAgentConfig` with the defaults contract, and then refresh docs/schema/tests before landing.

## Reproduction Assessment

Yes. The underlying issue has a high-confidence source reproduction path through current main: gateway agent runs pass no configured lane, `resolveGlobalLane()` falls back to `main`, and #43235 supplies concrete Feishu wait evidence. The PR defects are reproducible by static inspection of the new resolver and ingress precedence, with straightforward targeted unit coverage.

## Solution Assessment

No, not as proposed. The selected config surface is the narrow maintainable place to solve the issue, but the implementation should not allow arbitrary network-facing lane overrides to bypass server policy, and the resolver must not mix default-lane concurrency into unrelated custom lanes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so an authorized client can choose unique lanes and run outside configured per-agent/default caps. For network ingress, make lane selection server-controlled or validate it against a bounded allowlist.
  - confidence: 0.86
- **[P2] Keep default concurrency on the default lane:** `src/config/agent-command-lanes.ts:54-58`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so inherit `defaults.maxConcurrent` only when the agent also inherits the default lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config, so use the same defaults fallback pattern as the other inheritable fields.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR leaves configured command-lane isolation bypassable by arbitrary network-facing lane values.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: Gateway `AgentParamsSchema` accepts `lane` as any string, and the command queue creates state for new lane names. Because the PR resolves `request.lane` before the configured per-agent/default lane, an authorized caller can choose unique lanes to avoid configured concurrency caps and increase concurrent work.
  - confidence: 0.86

## Risks / Open Questions

- The PR lets authorized gateway callers choose arbitrary lane names, creating unbounded lane states and bypassing the configured per-agent/default concurrency caps.
- The new resolver can apply a default lane's `maxConcurrent` to an unrelated per-agent lane id when the agent omits its own cap.
- Changing `request.lane` precedence may affect existing CLI/manual lane users, so the security fix needs a maintainer decision on server-controlled lanes versus a strict allowlist/admin-only override.
