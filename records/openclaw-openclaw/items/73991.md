## Summary

Keep open. Current main still lacks per-agent foreground command lanes, so this PR is not obsolete, but the current PR head is not mergeable because caller-supplied gateway lanes can bypass the new server config and the resolver applies default lane concurrency to unrelated per-agent lanes.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config, schema/docs/generated baselines, gateway and auto-reply lane routing, lane concurrency wiring, tests, and a changelog entry.

## Best Possible Solution

Keep the per-agent command-lane direction, but make gateway lane selection server-controlled or explicitly allowlisted, keep default concurrency attached to the default lane only, align resolved agent config with defaults, and revalidate the focused gateway/config/docs tests.

## Reproduction Assessment

Yes for source behavior and PR defects: current main has only the shared foreground lane unless a caller supplies `lane`, and the PR head source shows request-lane precedence plus resolver fallbacks. The exact Feishu deployment latency was not live-reproduced.

## Solution Assessment

No, not as proposed. The config direction is reasonable, but the PR should not let caller-supplied lane names override server-owned policy and should not inherit default concurrency across unrelated lane ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1084`
  - body: The new configured lane is bypassed whenever a gateway client supplies `request.lane`. Because the protocol accepts any string and the queue creates lanes on demand, authenticated callers can choose unique lanes instead of the server-configured per-agent/default caps; ignore request lanes here or validate them against a server-owned allowlist.
  - confidence: 0.9
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.8
