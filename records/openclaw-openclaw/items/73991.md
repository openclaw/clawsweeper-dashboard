## Summary

Keep open. The PR targets a valid gap reported in #43235 and #42686, and the Feishu reproduction context is useful, but the current patch still has a security-sensitive gateway ingress blocker plus two resolver correctness issues before it should merge.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config/schema/docs, wires configured lanes into gateway and auto-reply agent execution, applies configured lane concurrency, and updates tests, generated config baselines, and the changelog.

## Best Possible Solution

Keep the opt-in core command-lane direction, but make gateway ingress lane selection server-controlled or allowlisted, keep default lane concurrency attached only to the default lane unless explicitly overridden, align `ResolvedAgentConfig` with the defaults fallback contract, and then revalidate schema/docs plus targeted gateway and auto-reply tests before merge.

## Reproduction Assessment

Yes. The #43235 Feishu report gives a concrete main-lane wait scenario, and the PR defects are reproducible by static inspection: a request with a unique `lane` bypasses configured lanes, and a defaults lane with `maxConcurrent` bleeds onto a per-agent lane id that omits its own cap.

## Solution Assessment

No, not as proposed. The implementation chooses the right core config and queue surfaces, but the gateway ingress precedence and resolver fallback behavior make the patch unsafe to merge without rework.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so configured per-agent/default caps are not authoritative for network-facing agent calls. Ignore request lanes at this boundary or validate them against a server-owned allowlist.
  - confidence: 0.86
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54-58`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff gives authenticated gateway request lanes precedence over server-configured command lanes, leaving the new configured isolation bypassable.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: The gateway agent RPC accepts arbitrary `lane` strings, and the command queue creates new lane states for unknown names. Because the PR resolves `request.lane` before the configured per-agent/default lane, authenticated callers can choose unique lanes instead of the intended server-owned lane policy.
  - confidence: 0.86
