## Summary

Keep open. Current main still lacks per-agent/default command lane configuration, and this PR is aimed at the right core boundary, but the patch is not mergeable as proposed because it preserves untrusted gateway lane overrides and has concrete resolver/defaulting bugs.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config/schema/docs, wires resolved lanes into gateway and auto-reply agent execution, registers configured lane concurrency, adds tests, updates generated config baseline hashes, and adds a changelog entry.

## Best Possible Solution

Keep the opt-in core queue/config direction, but make gateway ingress lane selection server-controlled or allowlisted, fix command-lane concurrency inheritance so default caps stay on the default lane unless explicitly overridden, align `ResolvedAgentConfig` with the defaults fallback contract, and revalidate docs/schema plus targeted gateway and agent tests before landing.

## Reproduction Assessment

Yes. The linked Feishu multi-agent report provides concrete `lane=main` wait evidence, current main source confirms default inbound runs share the global `main` lane after per-session serialization, and the PR defects are reproducible by static inspection of the new resolver and ingress code.

## Solution Assessment

No, not as proposed. The implementation targets the right maintainable surface, but configured lane isolation should not be bypassable by arbitrary client-provided lane strings, and default lane concurrency should not bleed onto unrelated per-agent lane ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so authenticated clients can choose unique lanes and run outside configured per-agent/default caps. For network ingress, ignore `request.lane` or validate it against a server-owned allowlist.
  - confidence: 0.86
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54-58`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id; otherwise leave the custom lane at its explicit cap or the queue default.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any consumer of the resolved config; use the defaults fallback or do not expose this as a resolved field.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR leaves configured command-lane isolation bypassable by arbitrary network-facing `request.lane` values.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: Gateway `AgentParamsSchema` accepts `lane` as an arbitrary string, and the command queue creates state for new lane names. Because the PR resolves `request.lane` before the configured per-agent/default lane, an authenticated client can choose unique lanes to avoid configured concurrency caps.
  - confidence: 0.86
