## Summary

Keep open. Current main still lacks a per-agent command-lane config path, and this PR remains the active implementation candidate, but the proposed patch has blocking gateway resource-control and resolver-default defects that should be addressed before merge.

## What This Changes

This PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config with schema/docs updates, gateway and auto-reply lane routing, lane concurrency application, and tests.

## Best Possible Solution

Keep the command-lane config direction, but make gateway lane choice server-controlled or allowlisted, keep default `maxConcurrent` attached only to the default lane, and align resolved agent config with defaults before revalidating.

## Reproduction Assessment

Yes. The linked report gives a concrete shared-`main` bottleneck scenario, and the PR defects are source-reproducible from request-lane precedence, queue lane creation, and resolver fallback behavior at PR head.

## Solution Assessment

No, not as proposed. The public config direction is reasonable, but the current diff lets caller-supplied lanes override server-configured policy and misapplies default concurrency to unrelated agent lanes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new configured lane is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the queue creates lanes on demand, so authenticated callers can choose unique lanes instead of the server-configured per-agent/default caps; ignore request lanes here or validate them against a server-owned allowlist.
  - confidence: 0.9
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.8
