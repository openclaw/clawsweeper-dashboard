## Summary

Keep open. Current main still lacks a per-agent command-lane config surface, and this PR targets the linked Feishu/main-lane bottleneck, but the patch should not merge as-is because gateway lane selection remains client-controllable and two defaulting paths misrepresent configured lane policy.

## What This Changes

The PR adds `agents.defaults.commandLane` and per-agent `commandLane` config, schema/docs/baselines/changelog updates, gateway and auto-reply lane wiring, lane concurrency application, and tests.

## Best Possible Solution

Keep the command-lane config direction, but make network-facing lane choice server-controlled or allowlisted, keep default `maxConcurrent` attached only to the default lane, and align resolved agent config with defaults before revalidating.

## Reproduction Assessment

Yes. #43235/#42686 provide a concrete shared-`main` lane bottleneck scenario, and the PR defects are reproducible by static inspection of the request-lane precedence, queue lane creation, and resolver fallback code.

## Solution Assessment

No, not as proposed. The config and queue surfaces are reasonable, but the gateway ingress precedence and resolver/defaults behavior need rework before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new configured lane is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so an authenticated caller can choose unique lanes instead of the server-configured per-agent/default caps. Ignore request lanes at this boundary or validate them against a server-owned allowlist.
  - confidence: 0.88
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete gateway resource-control concern because arbitrary request lanes override server-configured command lanes.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`

Because the PR resolves `request.lane` before the configured per-agent/default lane, authenticated callers can choose unique lanes instead of the intended server-owned lane policy.
