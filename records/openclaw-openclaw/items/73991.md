## Summary

Keep open. Current main still lacks the requested per-agent command-lane configuration, so this PR targets a real reported bottleneck, but the proposed patch is not safe to merge because network-facing callers can still override the configured lane and the resolver applies default-lane concurrency to unrelated per-agent lanes.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config/schema/docs, applies configured lane concurrency, and routes gateway plus auto-reply agent runs through the resolved command lane.

## Best Possible Solution

Keep the opt-in core command-lane direction, but make network ingress lane selection server-controlled or allowlisted, keep default lane concurrency attached only to the default lane unless explicitly overridden, align the resolved agent config with defaults inheritance, and then re-run the targeted gateway/config/auto-reply tests plus config docs generation before merge.

## Reproduction Assessment

Yes. The original linked report gives a concrete main-lane wait scenario, and the PR blockers are reproducible by static inspection of the diff against current main: arbitrary request lanes override configured lanes, and default `maxConcurrent` bleeds into a different per-agent lane id.

## Solution Assessment

No. The PR uses the right broad surfaces, but the current implementation leaves configured isolation bypassable and misapplies defaults; the safer fix is to make lane policy authoritative on the server side and tighten the resolver semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and the command queue creates lane state on demand, so configured per-agent/default caps are not authoritative for network-facing agent calls. Ignore request lanes here or validate them against a server-owned allowlist.
  - confidence: 0.86
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any future consumer of the resolved config; use the defaults fallback or avoid exposing it as resolved.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete gateway ingress DoS risk by letting authenticated callers choose arbitrary command-lane names ahead of server-configured lane policy.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: The gateway agent RPC accepts arbitrary `lane` strings, and the command queue creates lane states for unknown names. Because this PR resolves `request.lane` before the configured per-agent/default lane, callers can choose unique lanes instead of the intended server-owned lane policy.
  - confidence: 0.86
