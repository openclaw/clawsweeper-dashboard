Keep open. Current main deliberately exposes `sessions_send` in the QA coding profile while `message` remains messaging/channel-specific or explicitly forced, and the exact failing runtime-tool fixture is part of the still-open https://github.com/openclaw/openclaw/pull/80323. ## What This Changes

The issue asks maintainers to decide whether the QA-lab runtime message fixture should target the default `sessions_send` tool or remain a channel-profile `message(action="send")` scenario.

## Best Possible Solution

Resolve this in the Phase 2 harness by treating `sessions_send` as the default runtime send coverage and leaving `message(action="send")` to channel-visible scenarios, unless maintainers explicitly decide to broaden coding-profile tools.

## Solution Assessment

Unclear until maintainers choose the QA contract. The narrow maintainable direction appears to be a fixture/report wording update in https://github.com/openclaw/openclaw/pull/80323, not widening the coding-profile tool surface by default.
