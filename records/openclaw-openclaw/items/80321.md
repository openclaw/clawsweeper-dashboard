## Summary

Keep open. Current main still encodes two different intended surfaces: QA-lab parity scenarios run with the coding profile where `sessions_send` is included and `message` is excluded unless explicitly forced, while channel-visible scenarios exercise `message(action="send")`; the linked https://github.com/openclaw/openclaw/pull/80323 is still open, so this remains an active QA-contract decision rather than cleanup-close material.

## What This Changes

The issue asks maintainers to decide whether QA-lab runtime message coverage should target `sessions_send` or channel-profile `message(action="send")`.

## Best Possible Solution

Resolve the Phase 2 QA-lab contract by mapping default runtime message-send coverage to `sessions_send` and keeping `message(action="send")` in channel-visible scenarios unless maintainers intentionally broaden coding-profile tools.

## Reproduction Assessment

No high-confidence current-main runtime repro exists because the exact failing runtime-tool fixture is not checked in. Source inspection does confirm the ambiguity: coding-profile QA exposes `sessions_send` and excludes `message`, while checked-in channel coverage uses `message(action="send")`.

## Solution Assessment

Unclear. Retargeting or renaming the fixture inside https://github.com/openclaw/openclaw/pull/80323 appears narrower than widening the coding profile, but maintainers should make that QA-harness boundary decision.
