## Summary

Keep open. Current main still feeds Dreaming through the shared usage-counted session listing and only classifies cron runs from the session key/path set or narrow archive-content hints; it does not follow stored subagent lineage such as `spawnedBy`, so the reported parentage gap remains distinct from https://github.com/openclaw/openclaw/issues/72611.

## What This Changes

The issue reports Dreaming session-corpus pollution from reset/deleted transcript archives and cron-spawned subagent transcripts that are not classified as cron-derived.

## Best Possible Solution

Add a Dreaming-owned ingestion boundary that excludes archive duplicates from the session corpus while preserving memory_search archive indexing, and extend cron classification to follow stored subagent lineage such as `spawnedBy` or `parentSessionKey`.

## Reproduction Assessment

Yes for source reproduction: current main lists usage-counted archives for the Dreaming sweep and does not classify cron-descended subagents from stored lineage. I did not run a live workspace reproduction.

## Solution Assessment

No, the issue's literal one-line shared-helper swap is probably not the best fix because tests document reset/deleted archive indexing for memory_search. The maintainable fix is a Dreaming-specific corpus eligibility boundary plus lineage-aware cron classification.
