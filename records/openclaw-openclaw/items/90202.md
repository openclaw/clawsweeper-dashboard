- **[P1] Persist local-import rows in the real source-sync store:** `extensions/memory-wiki/src/source-sync-state.ts:5`
  - body: The PR widens `MemoryWikiImportedSourceGroup`, but current main's plugin-state-backed source-sync store still normalizes entries by accepting only `bridge` and `unsafe-local`. In a registered plugin runtime, `local-import` entries would be filtered out on write/read, so skip/prune behavior from the new tests will not hold for the real store. Please update the normalizer and add plugin-state coverage for `local-import` rows.
  - confidence: 0.9
- **[P2] Check the source-sync cap before writing local pages:** `extensions/memory-wiki/src/local-import.ts:210-213`
  - body: Bridge and unsafe-local imports assert the projected source-sync capacity before page writes because the store caps entries at 20,000. This path starts writing every local page first and only fails at the final state write, so an oversized configured directory can leave unmanaged `sources/local-import-*` pages behind. Reuse the existing capacity assertion for `group: "local-import"` before the writes and cover the oversized case.
  - confidence: 0.9
- **[P2] Document the new local-import config and command:** `extensions/memory-wiki/src/cli.ts:1120-1125`
  - body: This adds both `plugins.entries.memory-wiki.config.localImports` and `openclaw wiki local-import import`, but the public Memory Wiki and wiki CLI docs still only describe the existing import commands and config keys. Because this is a new local-file import API, the docs need to explain configuration, how it differs from one-shot ingest and unsafe-local, and what local path/privacy boundary operators are accepting.
