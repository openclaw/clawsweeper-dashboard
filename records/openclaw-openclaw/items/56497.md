## Summary

This PR should stay open but is not ready to merge: current main does not implement Volcengine Responses session caching, and the branch still has source-level blockers around package imports, cache invalidation, retry output isolation, and documented defaults. The happy-path screenshot proof is sufficient for the real-behavior gate, but the opt-in vendor retention semantics still need provider-owner review.

## What This Changes

The PR adds a Volcengine provider `wrapStreamFn` session-cache wrapper, cache helper tests, and docs for configuring `params.cache` with the `openai-responses` API.

## Best Possible Solution

Land a provider-local Volcengine Responses cache only after it uses current pi package contracts, invalidates on full Responses input changes, retries through an isolated stream, aligns docs with runtime defaults, and clearly documents opt-in retention behavior.

## Reproduction Assessment

Not applicable as a current-main bug because current main has no Volcengine session-cache implementation. The PR blockers are source-reproducible from the branch diff, current package declarations, and the pi v0.74.0 Responses transport contract.

## Solution Assessment

No, not as written. The provider-local boundary is appropriate, but the implementation needs the dependency, invalidation, retry, docs/default, and retention issues resolved before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main declares `@earendil-works/pi-*`, but this new module imports `@mariozechner/pi-*`. Those packages are not declared in this workspace, so the provider will fail to resolve the cache module after merge.
  - confidence: 0.97
- **[P1] Invalidate cached response IDs when input changes:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash covers tools and instructions but not the full Responses `input`, where this transport can carry system/developer prompt content. A prompt or profile change in the same session can reuse `previous_response_id` and trim away the updated prompt before sending.
  - confidence: 0.91
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls back into `pipeStream` with the same output stream. If the failed attempt emitted `start` or partial content before the error event, the retry output is concatenated with events from the failed attempt.
  - confidence: 0.88
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans. Users following the docs do not get the documented default behavior.
  - confidence: 0.84
