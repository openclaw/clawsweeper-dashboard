## What This Changes

The PR adds an opt-in Volcengine Ark Responses session-cache wrapper, cache tests, provider wiring, and English provider/model docs for `params.cache`.

## Best Possible Solution

Keep the feature provider-local and opt-in, rebase it onto the current Volcengine plugin surface, and land only after cache invalidation, retry isolation, docs/code defaults, and focused tests are fixed.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: current main lacks the cache wrapper, the PR imports undeclared packages, and focused tests can exercise `volcengine-plan` hook routing, system-prompt invalidation, and stale-response retry output reuse.

## Solution Assessment

No. A provider-local opt-in wrapper is the right boundary, but this patch needs to be rebased to the current dependency namespace and Volcengine hook surface, with stricter invalidation, isolated retries, and docs/code default alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current Pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-agent-core` and `@earendil-works/pi-ai`, but this new file imports the old `@mariozechner/*` packages. Those packages are not declared in the workspace, so the PR will fail to build/typecheck on current main until the imports are updated.
  - confidence: 0.95
- **[P1] Preserve Volcengine plan hook coverage:** `extensions/volcengine/index.ts:81`
  - body: The PR head adds `wrapStreamFn` on an older provider entry that lacks current main's `hookAliases: ["volcengine-plan"]`. Hook lookup resolves by provider id, aliases, and hookAliases, so `volcengine-plan/*` cache settings will miss the wrapper unless the PR is rebased while preserving hook coverage for both provider ids.
  - confidence: 0.9
- **[P1] Invalidate cached sessions when system input changes:** `extensions/volcengine/cache.ts:277-278`
  - body: The invalidation hash only covers `target.tools` and `target.instructions`, but the Responses path carries the system/developer prompt as an `input` item. A prompt or profile change in the same session can therefore reuse the old response id, trim the payload to continuation input, and run under stale instructions.
  - confidence: 0.9
- **[P2] Retry stale response ids through isolated output:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response branch retries by calling `pipeStream(output, false)` with the same output stream. If the failed attempt emitted a start or partial event before the retry decision, consumers can see failed-attempt events followed by retry events; buffer until retry is ruled out or retry through a fresh output path.
  - confidence: 0.86
- **[P2] Align the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` when it is explicitly configured. Either make the code default match the docs or document the real request shape.
  - confidence: 0.84
