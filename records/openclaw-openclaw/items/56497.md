## Summary

This PR still contains unique Volcengine session-cache feature work, but it is not ready to merge against current main because the branch is stale/conflicting and has source-reproducible correctness blockers in the new cache wrapper.

## What This Changes

The PR adds an opt-in Volcengine Responses session-cache wrapper, wires it through `wrapStreamFn`, adds cache tests, and documents `params.cache` configuration.

## Best Possible Solution

Land a provider-local Volcengine Responses cache wrapper after provider-owner approval with current pi imports/deps, full input-based invalidation, isolated stale-id retry, accurate defaults/docs, and focused tests.

## Reproduction Assessment

Not applicable for the requested feature: this adds new opt-in Volcengine provider behavior rather than fixing a current-main bug. The review findings are source-reproducible from the PR head, current main package contracts, and pi-ai `v0.74.0` Responses source.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but it must be rebased, use current package contracts, invalidate against full Responses input, isolate retry output, align documented defaults, and get retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package imports:** `extensions/volcengine/cache.ts:1-3`
  - body: This new module imports `@mariozechner/pi-*`, but current main declares and uses `@earendil-works/pi-*`; after rebase these imports will not resolve, and the Volcengine package manifest still lacks the runtime pi deps used by provider stream wrappers.
  - confidence: 0.97
- **[P1] Hash the Responses input before continuing:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash only covers `target.tools` and `target.instructions`, while the Responses system/developer prompt is carried in `input`. A prompt/profile change can reuse the old `previous_response_id`, then continuation mode trims away the updated prompt and runs under stale instructions.
  - confidence: 0.92
- **[P2] Retry stale response IDs with a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the first attempt has already pushed `start` or partial events before the error, consumers can see failed-attempt events concatenated with the retry result.
  - confidence: 0.88
- **[P2] Align the documented thinking default:** `extensions/volcengine/cache.ts:111`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans, so users following the documented default will not get the documented payload behavior.
  - confidence: 0.86
