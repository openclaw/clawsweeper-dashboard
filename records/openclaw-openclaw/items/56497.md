- **[P1] Rebase the Pi imports onto the current package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*` 0.75.4 and has no `@mariozechner/pi-*` packages, so this new file will fail module resolution until the branch is rebased and the imports/types are updated.

  - body: The Responses transport puts the system/developer prompt in `input`, but this hash only covers `tools` and `instructions`. With a stable session id, prompt or profile changes can reuse `previous_response_id` and then trim the updated prompt out of the continuation request.
  - confidence: 0.93
- **[P1] Gate vendor response storage behind explicit approval:** `extensions/volcengine/cache.ts:320`
  - body: The upstream Responses payload starts with `store: false`, but this wrapper forces `store = true` whenever cache is enabled. That changes vendor-side retention of prompt/session content without an explicit documented opt-in or provider-owner approval.
  - confidence: 0.88
- **[P2] Retry stale-response failures with a fresh output stream:** `extensions/volcengine/cache.ts:354`
  - body: The retry path recurses into `pipeStream(output, false)` using the same output stream. If the failed attempt emitted any start or delta events before the stale-response error, those events remain queued and will be concatenated with the retry response.
  - confidence: 0.82
- **[P2] Align the documented thinking default with the implementation:** `docs/providers/volcengine.md:73`
  - body: The docs say omitted `cache.thinking` falls back to `true`, but `resolveCacheConfig` leaves it undefined unless the user explicitly sets a boolean, so the documented default does not match the payload behavior.
  - confidence: 0.86
