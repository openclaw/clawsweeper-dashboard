## Best Possible Solution

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but this implementation needs dependency-contract fixes, complete Responses input invalidation, isolated retry behavior, docs/default alignment, and retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This new module imports `@mariozechner/pi-*`, but current main depends on `@earendil-works/pi-*` 0.74.0 and the Volcengine package does not declare the old namespace. As merged, the module will fail to resolve or violate the plugin runtime dependency contract.
  - confidence: 0.97
- **[P1] Invalidate cache when Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash only covers `target.tools` and `target.instructions`, but pi's Responses transport carries the system/developer prompt in `target.input`. With a stable session ID, prompt or profile changes can reuse the old `previous_response_id`, trim the request, and continue under stale instructions.
  - confidence: 0.94
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events, the retry appends a second attempt to the same stream, so callers can see concatenated events.
  - confidence: 0.89
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends a `thinking` body for explicit booleans. Users following the docs will not get the documented default behavior.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow, dependency-install, or command-execution supply-chain issue was found, but the cache feature changes vendor-side response retention semantics.

Concerns:

- **[low] Approve vendor response retention semantics:** `extensions/volcengine/cache.ts:320`
