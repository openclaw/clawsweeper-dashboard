## Summary

Keep this PR open: the Volcengine session-cache feature is useful and has inspected real-behavior screenshots, but the current branch is not merge-ready because it still has current-main compatibility, session-state, and security-boundary blockers.

## What This Changes

The PR adds an opt-in Volcengine Responses session cache wrapper, focused unit tests, and docs for `params.cache`.

## Best Possible Solution

Keep the feature inside the Volcengine plugin, but merge only after rebasing to current dependency names, scoping cache state to the full runtime boundary, invalidating on prompt/tool/input-prefix changes, and documenting the provider-side storage TTL.

## Reproduction Assessment

Yes for the blocking PR issues by source inspection: current main uses `@earendil-works/*`, carries non-Codex Responses system prompts in `input`, and exposes `authProfileId` on stream options while the PR omits those facts from cache safety. I did not run a live Volcengine repro in this read-only review.

## Solution Assessment

No. A Volcengine-local wrapper is a plausible home for the feature, but this implementation needs current-main dependency repair and stronger session/auth/prompt boundaries before it is the maintainable solution.

## Risks / Open Questions

- Merging the branch as-is can break current-main builds because the new file imports the old `@mariozechner/*` package names instead of the current `@earendil-works/*` packages.
- The module-level cache can reuse a `previous_response_id` across the same model/session when auth profile or ownership context changes, which risks stale or cross-boundary session continuation.
