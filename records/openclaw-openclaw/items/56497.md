## Summary

Current main still lacks Volcengine session caching, and the PR is useful provider-local work, but the latest diff is not ready to merge because it misses the coding provider hook path, can reuse stale system/developer instructions, reuses emitted stream output during retry, and lacks the required changelog entry.

## What This Changes

The PR adds a Volcengine stream wrapper, unit tests, and docs for opt-in Ark Responses session caching using `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the implementation provider-local, cover both `volcengine` and `volcengine-plan`, include system/developer prompt state in cache invalidation, retry stale response ids without exposing failed-attempt events, document the opt-in provider-side store/cache behavior clearly, and add the required changelog entry before running the focused plugin tests and changed gate.

Yes. A focused unit test can resolve/wrap `volcengine-plan` and observe that the wrapper is absent; another can reuse a session id while changing `context.systemPrompt`; a fake stream can emit a partial event before a stale-response error to show mixed retry output.

## Solution Assessment

No, not as currently written. The provider-local wrapper is the right boundary, but the patch needs hook coverage, stricter invalidation, retry isolation, and changelog coverage before it is the maintainable fix.

  - body: `wrapStreamFn` is added only to the `volcengine` plugin entry, while hook lookup for runtime providers matches `provider.id`, `aliases`, or `hookAliases`.   - confidence: 0.86
- **[P1] Invalidate cache when system input changes:** `extensions/volcengine/cache.ts:244-245`
  - body: The cache hash only includes `target.tools` and `target.instructions`, but pi-ai Responses places `context.systemPrompt` into `target.input` as a leading system/developer item. A same-session prompt/profile change can reuse `state.responseId`, after which continuation mode trims input and drops the updated instruction.
  - confidence: 0.9
- **[P2] Retry without reusing emitted stream output:** `extensions/volcengine/cache.ts:313`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted a start or partial event before the error, consumers receive failed-attempt events followed by retry events; buffer until retry is decided or retry through a fresh output path.
  - confidence: 0.84
