## Summary

Keep open: current main does not have Volcengine Responses session caching, but this PR still has blocking compile/runtime issues and a provider-retention product review point before it can merge.

## What This Changes

The branch adds an opt-in Volcengine `params.cache` Responses session-cache wrapper, wires it through `wrapStreamFn`, adds unit tests, and documents the new config.

## Best Possible Solution

Land a provider-local Volcengine cache wrapper only after it compiles on the current pi namespace, invalidates against the full Responses input, isolates retry output, aligns documented defaults with code, and has owner approval for retention semantics.

## Reproduction Assessment

Not applicable. This is a new opt-in provider feature PR rather than a current-main bug report; the review is based on the PR diff, current main, upstream pi-ai 0.74.0 source, prior review comments, and screenshots.

## Solution Assessment

No, not as written. The provider-local wrapper is a plausible direction, but the current patch has compile/runtime blockers and retention semantics that need provider-owner product review before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Import the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This file imports `@mariozechner/pi-*`, but current main declares `@earendil-works/pi-*` packages. The new cache module will fail to resolve until these imports are rebased to the current package names.
  - confidence: 0.97
- **[P1] Invalidate cache against the full Responses input:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash only covers `tools` and `instructions`, while current pi-ai puts the system/developer prompt in `input`. A prompt/profile change can reuse the old `previous_response_id`, then continuation mode trims away the updated prompt and runs under stale instructions.
  - confidence: 0.91
- **[P2] Retry stale response IDs with a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the first attempt has already pushed `start` or partial events before the error, consumers can see failed-attempt events concatenated with the retry result.
  - confidence: 0.87
- **[P2] Align the thinking default with the docs:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans. Users following the documented default will not get the documented payload behavior.
  - confidence: 0.86
