## Summary

Keep open. Current main still does not implement Volcengine Responses session caching, and the PR is useful provider-local work, but the proposed patch is not merge-ready because it misses the coding provider hook path and has cache invalidation and retry-stream correctness bugs.

## What This Changes

The PR adds an opt-in Volcengine stream wrapper, unit tests, and provider docs for Ark Responses session caching with `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the implementation provider-local, cover both `volcengine` and `volcengine-plan`, include system/developer input in session invalidation, isolate stale-id retries so failed-attempt events cannot leak, document the opt-in provider-side store/cache behavior clearly, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. A focused unit test can resolve `volcengine-plan` and observe no wrapper, another can change `context.systemPrompt` within the same session and see the old `responseId` reused, and a fake stream can emit a partial event before a stale-id error to show mixed retry output.

## Solution Assessment

No, not as currently written. The provider-local wrapper is the right boundary, but the patch needs hook coverage, stricter invalidation, retry isolation, and changelog coverage before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Register the cache wrapper for volcengine-plan:** `extensions/volcengine/index.ts:81`
  - body: `wrapStreamFn` is only registered on the `volcengine` plugin entry, while runtime hook lookup matches the requested provider against the plugin id, aliases, or hook aliases. Calls using `volcengine-plan/*` therefore never receive this wrapper even when cache params are enabled; add hook coverage for the coding provider and test both ids.
  - confidence: 0.88
- **[P1] Invalidate cache when system input changes:** `extensions/volcengine/cache.ts:277-278`
  - body: The cache hash only includes `target.tools` and `target.instructions`, but the Responses transport serializes `context.systemPrompt` into `target.input` as a leading system/developer item. A same-session prompt/profile change can reuse `state.responseId`, after which continuation mode trims the input and drops the updated instruction.
  - confidence: 0.9
- **[P2] Retry without reusing emitted stream output:** `extensions/volcengine/cache.ts:354-355`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted a start or partial event before the error, consumers receive failed-attempt events followed by retry events; buffer until retry is decided or retry through a fresh output path.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `docs/providers/volcengine.md:36`
  - body: This is a user-facing feature PR and the diff does not update `CHANGELOG.md`. Add a single-line entry in the active Changes section with contributor credit for `@stayrascal` before merge.
  - confidence: 0.92
