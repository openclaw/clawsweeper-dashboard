## Summary

Keep open; current main does not implement Volcengine session caching, and this PR remains useful, but source review found provider-local blockers that should be fixed before merge.

## What This Changes

The PR adds an opt-in Volcengine Ark Responses session-cache wrapper, provider wiring, unit tests, and documentation for `params.cache`.

## Best Possible Solution

Keep the feature provider-local and opt-in, cover both Volcengine provider ids, invalidate continuations on all stable request inputs, isolate stale-response retries, align docs with behavior, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: request `volcengine-plan/*` with cache params, change system/developer input within the same session, or use a fake stream that emits a start event before a stale previous-response error.

## Solution Assessment

No. The provider-local wrapper is the right boundary, but this patch needs hook coverage, stricter invalidation, retry isolation, and user-facing release/docs consistency before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Register the wrapper for volcengine-plan:** `extensions/volcengine/index.ts:81`
  - body: `wrapStreamFn` is added only to the provider entry whose id is `volcengine`. Runtime hook lookup matches the requested provider id against `id`, `aliases`, and `hookAliases`, while the manifest exposes `volcengine-plan` as a separate provider id, so cache params on coding models will not enter this wrapper. Add hook coverage and tests for both Volcengine provider ids.
  - confidence: 0.89
- **[P1] Invalidate cached continuations when system input changes:** `extensions/volcengine/cache.ts:277-278`
  - body: The continuation hash only includes `target.tools` and `target.instructions`, but this Responses path carries the effective system/developer prompt in `input`. A prompt or profile change in the same session can reuse the old `previous_response_id`, trim the request to trailing input, and drop the new instruction. Include the stable system/developer input in invalidation before entering continuation mode.
  - confidence: 0.91
- **[P2] Retry stale response ids without reusing emitted output:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream used by the failed attempt. If the first attempt already emitted a start or partial event, the consumer sees failed-attempt events followed by retry events. Buffer until retry is ruled out, or retry through a fresh output path.
  - confidence: 0.86
- **[P3] Align the documented thinking default with the wrapper:** `docs/providers/volcengine.md:72-73`
  - body: The docs say omitted `thinking` falls back to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only writes `target.thinking` when the setting is explicitly `true` or `false`. Either make the implementation default to enabled thinking or document that omission leaves provider/default behavior untouched.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `docs/providers/volcengine.md:39`
  - body: This is a user-facing provider feature and the PR does not update `CHANGELOG.md`. Add one single-line entry under the active Changes section, crediting `@stayrascal`.
  - confidence: 0.94
