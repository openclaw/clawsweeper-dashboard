## Summary

Keep open: current main does not implement the feature, and the branch has meaningful unique work, but the patch is not mergeable because several source-reproducible correctness blockers and a retention-semantics approval issue remain.

## What This Changes

The PR adds an opt-in Volcengine Responses session-cache wrapper, provider wiring, unit tests, and docs for `params.cache`.

## Best Possible Solution

Land a provider-local Volcengine Responses session-cache feature after rebase, correct pi imports, input-based invalidation, isolated retry streams, docs/default alignment, and approved retention semantics.

## Reproduction Assessment

Yes for the PR blockers: the namespace mismatch, input-hash gap, retry stream reuse, and docs/default mismatch are source-reproducible from the PR head and upstream pi 0.74.0 source. No current-main bug is reproduced because main does not implement this opt-in feature.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but the implementation needs dependency-contract fixes, full Responses input invalidation, fresh retry streams, docs/default alignment, and retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This module imports `@mariozechner/pi-*`, but current main depends on `@earendil-works/pi-*` 0.74.0 and the Volcengine package does not declare the old namespace. As merged, the new file will fail module resolution or violate the plugin runtime dependency contract.
  - confidence: 0.97
- **[P1] Hash the Responses input before reusing cache:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash only covers `tools` and `instructions`, while pi Responses carries the system/developer prompt in `input`. A prompt or profile change in the same session can reuse the old `previous_response_id`, trim the request, and continue under stale instructions.
  - confidence: 0.93
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The retry path calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events before the stale-response error, those events are concatenated with the retry output.
  - confidence: 0.88
- **[P2] Align the thinking default with the docs:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to true, but `resolveCacheConfig` leaves it undefined and the wrapper only sends `thinking` for explicit booleans. Users following the docs will not get the documented default behavior.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow, dependency-install, or command-execution supply-chain issue was found, but the cache feature changes vendor-side response retention semantics.

Concerns:

- **[low] Approve vendor response retention semantics:** `extensions/volcengine/cache.ts:320`
