The PR adds opt-in Volcengine Ark Responses session caching with provider wiring, cache tests, and docs for `params.cache`.

## Best Possible Solution

Keep the feature provider-local and opt-in, rebase it onto the current Volcengine plugin surface, and land only after cache invalidation, retry isolation, docs/code defaults, focused tests, and live proof are fixed.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: current main lacks the cache wrapper, the PR imports undeclared package names, and the PR head can be checked against current provider hook and upstream Responses payload contracts.

## Solution Assessment

No. A provider-local opt-in wrapper is the right boundary, but this patch needs to be rebased to the current dependency namespace and Volcengine hook surface, with stricter invalidation, isolated retries, and docs/code default alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the current Pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main declares the Pi packages as `@earendil-works/*`, but this new file imports `@mariozechner/pi-agent-core` and `@mariozechner/pi-ai`. Those runtime imports are not declared in the workspace, so the PR will fail to build or load against current main until the namespace is updated.
  - confidence: 0.97
- **[P1] Preserve Volcengine plan hook coverage:** `extensions/volcengine/index.ts:81`
  - body: The PR head adds `wrapStreamFn` on an older provider entry that omits current main's `hookAliases: ["volcengine-plan"]`. Provider hook lookup uses that alias, so `volcengine-plan/*` cache settings will miss this wrapper unless the PR is rebased while preserving the current provider fields.
  - confidence: 0.92
- **[P1] Invalidate cached sessions when system input changes:** `extensions/volcengine/cache.ts:276-278`
  - body: The invalidation hash only covers `target.tools` and `target.instructions`, but the Responses path puts the system/developer prompt in `target.input`. A prompt/profile change can therefore reuse the old `previous_response_id`, trim the payload to continuation input, and run under stale instructions.
  - confidence: 0.91
- **[P2] Retry stale response IDs through isolated output:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same event stream. If the failed attempt emitted a start or partial event before the retry decision, consumers can receive failed-attempt events followed by retry events in one response.
  - confidence: 0.86
- **[P2] Align the documented thinking default:** `extensions/volcengine/cache.ts:110-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` when configured. Either make the code default match the docs or document the actual request shape.
  - confidence: 0.84
