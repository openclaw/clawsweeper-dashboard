This PR should stay open because current main does not implement Volcengine Responses session caching and the branch still contains unique feature work, but the current patch is not mergeable as written due to concrete dependency, cache-invalidation, retry-stream, and docs/default mismatches.

## What This Changes

The PR adds opt-in Volcengine Responses session caching, provider wiring, cache tests, and docs for `params.cache`.

## Best Possible Solution

Land a provider-local Volcengine Responses cache feature after it rebases cleanly, uses current pi imports, invalidates on full Responses input changes, isolates retry output, aligns docs with runtime defaults, and documents approved retention semantics.

## Reproduction Assessment

Not applicable as a current-main bug because main does not implement this opt-in feature. The merge blockers are source-reproducible from the PR head, current package declarations, and pi v0.74.0 Responses source.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but the implementation needs dependency-contract fixes, full input-based invalidation, isolated retry behavior, and explicit retention-semantics review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the declared pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This new module imports `@mariozechner/pi-*`, but current main declares `@earendil-works/pi-*` 0.74.0 and the Volcengine package does not add the old namespace. As merged, the cache wrapper will fail to resolve these imports.
  - confidence: 0.97
- **[P1] Invalidate the cache when Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash only covers `tools` and `instructions`, but this Responses path carries the system/developer prompt in `input`. With a stable session, a prompt change can reuse the old `previous_response_id`, trim the request, and run under stale instructions.
  - confidence: 0.92
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events before the stale-response error, those events remain and get concatenated with retry output.
  - confidence: 0.88
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans. Users following the docs do not get the documented default.
  - confidence: 0.86
