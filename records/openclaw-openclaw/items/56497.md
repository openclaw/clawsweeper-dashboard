## Summary

Keep open. Current main does not implement Volcengine session caching, and this PR is a plausible provider-local feature, but the head is not mergeable as-is because it still uses old pi imports, misses full input-based cache invalidation, and needs explicit review of vendor retention semantics.

## What This Changes

The PR adds an opt-in Volcengine Responses session-cache wrapper, unit tests, and provider docs for `params.cache`.

## Best Possible Solution

Land a provider-local Volcengine Responses session-cache feature only after it aligns with current SDK/dependency contracts, treats vendor storage semantics as explicit opt-in behavior, and covers invalidation/retry edge cases with tests.

## Reproduction Assessment

Not applicable for the feature request as a current-main bug; current main lacks the optional cache wrapper. The blocking review findings are source-reproducible from the PR head plus the upstream pi v0.74.0 Responses contract.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but it needs current imports, full input-based cache invalidation, isolated retry output, docs/default alignment, and retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*` 0.74.0, but this new module imports `@mariozechner/pi-*`. Rebasing or merging the branch as-is leaves the cache wrapper with unresolved imports.
  - confidence: 0.97
- **[P1] Invalidate cache entries when Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash only covers `tools` and `instructions`, but pi builds the system/developer prompt into `input`. A prompt/profile change can therefore reuse an old `previous_response_id` and trim away the updated prompt.
  - confidence: 0.92
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events, consumers can see those events concatenated with the retry result.
  - confidence: 0.88
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans.
  - confidence: 0.86
