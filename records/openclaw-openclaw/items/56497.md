## Summary

Keep open: current main does not implement Volcengine session caching, but this PR is not mergeable as-is because it imports the old pi package namespace, can reuse stale cache state after prompt changes, reuses retry output streams, and needs explicit review of vendor-side retention semantics.

## What This Changes

The branch adds an opt-in Volcengine `params.cache` Responses session-cache wrapper, tests, and docs for `previous_response_id`, `store`, `expire_at`, and cache token reuse.

## Best Possible Solution

Land a provider-local Volcengine session-cache wrapper only after rebase, current pi imports, input-aware invalidation, fresh retry streams, accurate docs/tests, and explicit opt-in retention approval.

## Reproduction Assessment

Not applicable for the requested feature itself because this adds opt-in Volcengine behavior rather than fixing a current-main bug. The blocking review findings are source-reproducible from the PR head, current main package declarations, and pi 0.74.0 Responses source.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but the patch must be rebased, use current package contracts, invalidate against full Responses input, isolate retry output, align documented defaults, and get retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*` 0.74.0 and no longer declares `@mariozechner/pi-*`. As written, this new module leaves the cache wrapper with unresolved imports after rebase/build.
  - confidence: 0.97
- **[P1] Invalidate cache entries when Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash only covers `tools` and `instructions`, but pi builds the system/developer prompt into `input`. A prompt/profile change can therefore reuse an old `previous_response_id` and then trim away the updated prompt on continuation turns.
  - confidence: 0.92
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events, consumers can see those events concatenated with the retry result.
  - confidence: 0.88
- **[P2] Align the thinking default with the docs:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans. Users following the example may not get the documented thinking setting.
  - confidence: 0.86
