The PR adds an opt-in Volcengine Responses session-cache wrapper, wires it into the provider, adds cache tests, and documents `params.cache` usage.

Land a provider-local Volcengine Responses cache feature only after it aligns with current pi imports, invalidates on full Responses input changes, isolates retry output, documents retention semantics, and has provider-owner approval.

Not applicable as a current-main bug because this is an opt-in provider feature. The merge blockers are source-reproducible from the PR head, current package declarations, and the pi v0.74.0 Responses source.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but the patch needs current package contracts, full input-based invalidation, isolated retry behavior, and explicit retention-semantics review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*` 0.74.0, but this new module imports `@mariozechner/pi-*`, so merging or rebasing as-is leaves unresolved imports for the cache wrapper.
  - confidence: 0.97
- **[P1] Invalidate cache entries when Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash covers only `tools` and `instructions`, but pi builds the system/developer prompt into `input`; a prompt/profile change can therefore reuse a stale `previous_response_id` and trim away the updated prompt.
  - confidence: 0.92
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream, so any `start` or partial events already emitted by the failed attempt can be concatenated with retry output.
  - confidence: 0.88
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` returns `thinking: undefined` and the wrapper only sends `thinking` for explicit booleans.
  - confidence: 0.86
