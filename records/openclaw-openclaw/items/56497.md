## Summary

Keep open: current main still lacks the Volcengine session-cache feature, but this PR is not merge-ready because the branch has source-reproducible correctness blockers and changes provider-side response retention behavior that needs owner approval.

## What This Changes

The PR adds an opt-in Volcengine Responses session-cache wrapper, wires it into the provider, adds cache tests, and documents `params.cache`.

## Best Possible Solution

Land a provider-local Volcengine Responses session-cache feature after rebase, current pi imports, input-based invalidation, isolated retry behavior, docs/default alignment, and approved retention semantics.

## Reproduction Assessment

No for a current-main bug: current main does not implement this opt-in feature. Yes for PR blockers: the namespace mismatch, input-hash gap, retry stream reuse, and docs/default mismatch are source-reproducible from the PR head and pi 0.74.0 source.

## Solution Assessment

No, not as written. The provider-local wrapper is the right boundary, but it needs dependency-contract fixes, full Responses input invalidation, retry isolation, docs/default alignment, and retention-semantics approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This module imports `@mariozechner/pi-*`, but current main declares `@earendil-works/pi-*` 0.74.0 and the Volcengine plugin package does not declare the old namespace. As merged, the new module will fail to resolve or violate the plugin runtime dependency contract.
  - confidence: 0.97
- **[P1] Hash the Responses input before reusing cache:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash only covers `tools` and `instructions`, but pi's Responses transport puts the system/developer prompt in `input`. With a stable session ID, a prompt/profile change can reuse the old `previous_response_id`, trim the request, and continue under stale instructions.
  - confidence: 0.93
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The retry path calls `pipeStream(output, false)` with the same output stream. If the failed attempt already emitted `start` or partial events before the stale-response error, those events remain and get concatenated with retry output.
  - confidence: 0.88
- **[P2] Align the thinking default with the docs:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `thinking` for explicit booleans. Users following the docs will not get the documented default behavior.
  - confidence: 0.86
