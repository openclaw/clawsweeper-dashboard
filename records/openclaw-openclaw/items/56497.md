Full review comments:

- **[P1] Rebase the Pi imports onto the current namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*`, while this new file imports `@mariozechner/pi-*`. That stale namespace will fail against the current dependency contract, so the branch needs to be rebased and updated before merge.
  - confidence: 0.96
- **[P1] Hash the full Responses input before continuing cached sessions:** `extensions/volcengine/cache.ts:276-278`
  - body: The invalidation hash only includes `target.tools` and `target.instructions`, but this Responses path can carry system/developer prompt changes inside `input`. A prompt/profile update can therefore reuse the old `previous_response_id`, then drop the new instructions on the continuation request.
  - confidence: 0.91
- **[P1] Require explicit approval before forcing vendor storage:** `extensions/volcengine/cache.ts:319-320`
  - body: This wrapper sets `store = true` for every cache-enabled Volcengine Responses request, while current main only forces provider-managed storage for known native OpenAI/Azure routes. That changes conversation retention semantics and needs explicit provider policy/docs approval before landing.
  - confidence: 0.9
- **[P1] Return a fresh stream for stale response retries:** `extensions/volcengine/cache.ts:354`
  - body: The retry path calls `pipeStream(output, false)` with the same output stream. If the failed attempt emitted any events before the stale-response error, the consumer will see those partial events followed by the retry's events.
  - confidence: 0.84
- **[P2] Align the documented thinking default with runtime behavior:** `docs/providers/volcengine.md:72-73`
  - body: The docs say omitted `cache.thinking` defaults to true, but the implementation and tests leave it `undefined` unless configured. Users following the docs will get different request payloads than described.
  - confidence: 0.88
