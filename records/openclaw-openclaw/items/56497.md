## Summary

Keep open: current main still lacks Volcengine session caching, and this PR is useful but not merge-ready because the branch has blocking compile/runtime/docs issues plus a provider-retention product review point.

## What This Changes

Adds an opt-in Volcengine `openai-responses` session-cache wrapper configured by `params.cache`, with `previous_response_id`, `caching`, `store`, and `expire_at` handling plus provider docs and cache tests.

## Best Possible Solution

Land a provider-local Volcengine cache wrapper only after rebasing onto current provider surfaces, fixing cache invalidation/retry/default semantics, and documenting the opt-in vendor retention behavior clearly.

## Reproduction Assessment

Not applicable. This is a new provider feature PR rather than a current-main bug report, so review is based on the PR diff, current-main Volcengine behavior, dependency source, and the supplied screenshots.

## Solution Assessment

No, not as currently written. A provider-local wrapper is plausible, but the current diff needs namespace, prompt-invalidation, retry-isolation, default/docs, and retention-policy fixes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This new file imports `@mariozechner/pi-agent-core` and `@mariozechner/pi-ai`, but current main only declares and uses the `@earendil-works/*` pi packages. As written, the added module will not resolve in this workspace after merge.
  - confidence: 0.97
- **[P1] Include input prompt changes in the cache hash:** `extensions/volcengine/cache.ts:277`
  - body: Responses carries the system/developer prompt in `input`, but this hash only covers `tools` and `instructions`. After a prompt or profile change, the wrapper can reuse the old `previous_response_id` and trim the updated prompt from the continuation request, leaving the next turn under stale instructions.
  - confidence: 0.92
- **[P2] Retry stale response IDs with a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry calls `pipeStream(output, false)` with the same output stream. If the failed attempt emitted start, text, or tool events before the invalid-response error, those partial events are delivered before the retry output.
  - confidence: 0.86
- **[P2] Align the thinking default with the docs:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `target.thinking` when the value is explicit. Users following the documented default will not get the documented payload behavior.
  - confidence: 0.85
