## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Port the wrapper to SDK-owned stream imports:** `extensions/volcengine/cache.ts:1-3`
  - body: This new runtime file imports stream and LLM contracts from `@mariozechner/*`, but current main exposes those contracts through `openclaw/plugin-sdk/agent-core` and `openclaw/plugin-sdk/llm`, and the Volcengine package only depends on `@openclaw/plugin-sdk`. As submitted, the branch needs a current-main port before it can compile and validate normally.
  - confidence: 0.94
- **[P1] Invalidate cache when Responses input changes:** `extensions/volcengine/cache.ts:276-278`
  - body: The cache hash only covers `target.tools` and `target.instructions`, while current Responses payloads carry the system/developer prompt and replayed message history in `target.input`. A prompt/profile/history change can therefore reuse an old `previous_response_id`, then continuation mode trims the request to trailing items so Volcengine never sees the updated input state.
  - confidence: 0.91
- **[P2] Disclose retained Volcengine response state:** `extensions/volcengine/cache.ts:319-330`
  - body: Enabling cache unconditionally sends `store=true` and `expire_at`, retaining provider-side response state for the configured TTL. The PR security section says data access scope is unchanged, and the docs do not disclose this retention behavior, so maintainers and users cannot make an informed opt-in decision.
  - confidence: 0.88
- **[P3] Align the documented thinking default:** `docs/providers/volcengine.md:72-73`
  - body: The docs say `thinking` falls back to `true`, but `resolveCacheConfig` leaves it `undefined` unless the user explicitly sets a boolean and the wrapper only writes `target.thinking` when that value exists. Either set the code default or stop documenting `thinking: true` as the default.
  - confidence: 0.82
