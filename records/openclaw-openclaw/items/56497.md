## Summary

Keep open. Current main still lacks Volcengine session-cache support, and the PR is useful provider-local work, but the current patch is not merge-ready because the hook misses the `volcengine-plan` provider path, cache invalidation can miss system/developer prompt changes, stale-response retries can reuse already-emitted stream output, and the user-facing feature needs a changelog entry.

## What This Changes

The PR adds a Volcengine stream wrapper, tests, and docs for opt-in Ark Responses session caching via `params.cache`, managing `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the implementation provider-local, cover both `volcengine` and `volcengine-plan`, include system/developer prompt state in cache invalidation before trimming continuation input, isolate stale-response retries from already-emitted output, document the opt-in provider-side cache/store behavior clearly, and add the required changelog entry before validation.

## Reproduction Assessment

Yes. A focused unit test can resolve/wrap `volcengine-plan` and observe the wrapper is absent; another can reuse one session id while changing `context.systemPrompt`; a fake stream can emit a start/partial event before a stale-response error to show mixed retry output.

## Solution Assessment

No, not as currently written. The provider-local wrapper is the right boundary, but the patch needs hook coverage, stricter invalidation, retry isolation, and changelog coverage before it is the maintainable fix.
