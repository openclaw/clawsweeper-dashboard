## Summary

This PR should stay open: current main does not implement Volcengine Responses session caching, and the branch still has concrete correctness and provider-retention issues that need repair or maintainer/provider review before merge.

## What This Changes

The PR adds an opt-in Volcengine `params.cache` wrapper for the Responses API, wires it into the Volcengine provider, adds cache unit tests, and documents the configuration.

## Best Possible Solution

Keep the feature provider-local, but land it only after the cache wrapper matches current pi package contracts, invalidates on full Responses input changes, isolates stale-response retries, documents opt-in retention behavior, and receives provider-owner review.

## Reproduction Assessment

Not applicable as a current-main bug; this is an opt-in provider feature. The merge blockers are source-reproducible from the PR diff and current package/provider contracts.

## Solution Assessment

No, not as written. A provider-local wrapper is the right boundary, but the current patch needs correctness fixes and explicit retention-semantics review before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: Current main depends on `@earendil-works/pi-*`, but this new module imports `@mariozechner/pi-*`. Rebasing or merging as-is leaves the cache wrapper on the wrong package namespace for the current tree.
  - confidence: 0.97
- **[P1] Invalidate cache entries when the Responses input changes:** `extensions/volcengine/cache.ts:277`
  - body: The invalidation hash covers tools and `instructions`, but the Responses path carries system/developer prompt content in `input`. A prompt/profile change can reuse the old `previous_response_id`, trim the new prompt away, and continue under stale instructions.
  - confidence: 0.91
- **[P2] Retry stale response IDs through a fresh stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry pipes the second attempt into the same output stream, so any start or partial events emitted before the failed attempt can be concatenated with retry output. The retry should create an isolated stream for the second attempt.
  - confidence: 0.88
- **[P2] Match the documented thinking default:** `extensions/volcengine/cache.ts:111-113`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` returns `thinking: undefined` and the wrapper only sends the field for explicit booleans. Users following the docs will not get the documented default behavior.
  - confidence: 0.84
