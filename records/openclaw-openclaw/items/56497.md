## Summary

Keep this PR open: the feature is plausible and the screenshots show a cache-read happy path, but current main does not already implement it and the branch still has blocking build/correctness issues plus a product/privacy decision around vendor response retention.

Adds an opt-in Volcengine Responses session-cache wrapper configured through `params.cache`, wires it through `wrapStreamFn`, and adds provider docs plus unit tests.

## Best Possible Solution

Land a provider-local cache wrapper only after owner acceptance of retention semantics, a rebase to the current pi namespace and provider hooks, and coverage for prompt changes, retries, and documented defaults.

## Solution Assessment

No, not as written. A provider-local wrapper is a plausible direction, but the branch needs namespace, prompt-invalidation, retry-isolation, documented-default, and retention-policy fixes before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current pi package namespace:** `extensions/volcengine/cache.ts:1-3`
  - body: This new module imports `@mariozechner/pi-*`, but current main declares and uses `@earendil-works/pi-*` packages. After rebasing, the added module will not resolve until these imports are updated.
  - confidence: 0.97
- **[P1] Invalidate cache when input prompt changes:** `extensions/volcengine/cache.ts:277`
  - body: The cache hash ignores the system/developer prompt because pi-ai puts it in `input`, not `instructions`. With a stable session id, prompt or profile changes can reuse the old `previous_response_id` and then trim the updated prompt from the continuation request.
  - confidence: 0.91
- **[P2] Retry stale response IDs with an isolated stream:** `extensions/volcengine/cache.ts:354`
  - body: The stale-response retry recurses with the same output stream. If the failed attempt emitted start, text, or tool events before the error, those events are delivered before the retry output.
  - confidence: 0.84
- **[P2] Make the thinking default match the docs:** `extensions/volcengine/cache.ts:111`
  - body: The docs say omitted `cache.thinking` defaults to `true`, but `resolveCacheConfig` leaves it `undefined`, so the wrapper sends no `thinking` field unless users set it explicitly.
  - confidence: 0.86
