## Summary

This PR should stay open: it adds a useful Volcengine provider feature, but the branch still has merge-blocking session-state and provider-side context-retention risk. The main correctness problem is that cache continuation can reuse a prior provider `responseId` after OpenClaw has shortened the current input, so Volcengine may continue from context that the current request no longer sends.

## What This Changes

The PR adds an opt-in Volcengine Responses API session-cache wrapper, cache helper tests, provider wiring, and docs for `params.cache`.

## Best Possible Solution

Keep the feature in the Volcengine plugin, but invalidate the provider-side cache whenever the current OpenClaw request no longer proves the same prompt, tools, and retained input context before using `previous_response_id`.

## Reproduction Assessment

Yes for the review finding: source inspection shows a shorter `input` keeps the prior cache state and can still reuse `previous_response_id`. I did not run a live Volcengine repro in this read-only review.

## Solution Assessment

No as currently proposed: the plugin boundary is right, but reusing provider-side response ids after input shortening is not the safest cache contract. The better path is to reset or re-key the cache unless the retained prompt, tools, and conversation input are proven unchanged.

- **[P1] Reset cache when the request input shrinks:** `extensions/volcengine/cache.ts:349-352`
  - body: This branch keeps the previous Responses cache when `input.length < state.lastInputLength`, treating it as compaction.
