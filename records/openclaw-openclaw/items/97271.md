## Summary

Keep open for maintainer review: the patch is a useful, mostly correct hardening of current main's unbounded Telegram Bot API reads, with no blocking code finding found. The main maintainer decision is whether to accept the new 4 MiB fail-closed cap as-is and whether to cover the remaining lower-risk getChat reader in the same sweep.

## What This Changes

The PR replaces Telegram getMe, getWebhookInfo, getChatMember, and getUpdates JSON body reads with capped `readResponseWithLimit` reads, adds a probe-test mock adjustment, and adds a local proof script.

## Best Possible Solution

Land the bounded-read mitigation after maintainers accept the 4 MiB fail-closed cap, with getChat either included now or explicitly left as a follow-up for complete Telegram Bot API coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection on current main clearly shows unbounded `.json()` reads in the targeted Telegram paths, and the PR body provides a negative-control stream proof for unbounded `.json()` behavior, but I did not run a current-main OOM reproduction end to end.

## Solution Assessment

Is this the best way to solve the issue? Yes for the targeted high-risk paths: reusing the existing plugin SDK response-limit helper is the narrow owner-boundary fix, though a complete sweep would also bound the lower-risk getChat lookup path.
