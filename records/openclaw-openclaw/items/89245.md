## Summary

Keep open. Current main and the latest release still have reset/new and runner-driven session rotation paths that can persist a new session id, sessionFile, and usage-family entry before a transcript artifact is guaranteed, so the reported Feishu direct missing-transcript state drift is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent Feishu direct `/reset` or `/new` recovery from advancing session-family metadata to missing transcript IDs and generic failure loops.

## Best Possible Solution

Add a generic core session lifecycle invariant so reset/new only records a new family member after a minimal transcript is durable, or keeps/remaps to the last valid transcript with explicit repair guidance.

## Reproduction Assessment

Yes at source level, but not via a live current-main Feishu replay: current reset/new and runner-driven reset paths can persist new session metadata before transcript durability is guaranteed.

## Solution Assessment

Unclear as a ready repair plan. The best direction is generic core session lifecycle hardening with Feishu/Codex proof, not a Feishu-only patch.
