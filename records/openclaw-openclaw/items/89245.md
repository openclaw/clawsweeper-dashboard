## Summary

Keep open. Current main and v2026.6.9 still have direct `/new` and `/reset` session paths that can persist new lineage/session-file metadata before a durable transcript artifact is guaranteed; adjacent Gateway reset hardening does not cover the reported Feishu direct path.

## What This Changes

Ensure Feishu direct `/reset` and `/new` cannot leave session-family IDs or session-file references pointing to missing transcripts or fall into a generic failure loop.

## Best Possible Solution

Make direct and runner reset/new flows commit session lineage only after a minimal durable transcript exists, or retain/remap the last valid transcript with explicit repair behavior, then prove Feishu/Codex recovery end to end.

Yes at source level, but not as a fresh live replay. Current main still has direct `/new` and `/reset` paths that can persist new lineage and session-file metadata without a durable transcript guarantee.

Unclear. The source points to the invariant that must hold, but maintainers still need to choose the exact durability and repair semantics before an automated fix is safe.
