## Summary

Keep open. Current `main` still has direct auto-reply `/new` and `/reset` paths that can publish a new session id, `sessionFile`, and usage-family lineage before any durable transcript artifact is guaranteed; the newer lifecycle transcript-header guard appears limited to the Gateway reset service path and does not prove the reported Feishu direct broken-lineage case is fixed.

## What This Changes

The issue reports Feishu direct becoming unusable after repeated `/reset` and `/new` actions when session lineage advances to transcript ids that have no durable session artifacts, causing generic external-run failure replies.

## Best Possible Solution

Make all direct and runner reset/new paths publish session lineage only after a minimal durable transcript exists, or retain/remap the last valid transcript with explicit session-repair behavior, then prove the Feishu/Codex recovery path end to end.

## Reproduction Assessment

Yes for source-level reproduction of the unsafe invariant: current main still has reset/new paths that can persist new lineage and session-file metadata without a durable transcript guarantee, though no fresh live Feishu replay was run.

## Solution Assessment

Unclear as a product policy choice; the source points to the needed invariant, but maintainers need to choose the exact recovery semantics and user-facing repair behavior before an automated fix should proceed.
