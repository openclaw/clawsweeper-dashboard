## Label Justifications

- `P2`: The issue describes a bounded but user-visible session recovery failure in a direct channel after reset/new actions.
- `impact:session-state`: The central problem is session-family metadata pointing at session ids whose transcript artifacts do not exist.
- `impact:message-loss`: Follow-up Feishu messages can fail into platform fallback copy instead of producing a normal assistant reply.

## Summary

Keep open. Current main still lets reset/new session metadata and usage-family IDs advance before a durable transcript artifact is guaranteed; adjacent compaction and Feishu doctor work helps, but does not prevent the reported reset/new missing-transcript state.

## What This Changes

The issue asks for transactional Feishu direct `/reset` and `/new` session recovery so session metadata cannot point at missing transcript artifacts or trap users in generic failure replies.

## Best Possible Solution

Add a generic core session lifecycle invariant so reset/new either creates a durable minimal transcript before persisting the new family member or keeps/remaps to the last valid transcript with explicit repair guidance.

## Reproduction Assessment

No high-confidence live current-main replay was run. Source inspection supports the reported shape: reset/new persists new session metadata before transcript creation is guaranteed, and the reporter supplied concrete Feishu filesystem evidence for missing artifacts.

## Solution Assessment

Unclear until maintainers choose the invariant boundary. The best direction is core session lifecycle hardening plus focused Feishu/Codex proof, not closing under the existing doctor cleanup path.
