## Summary

Keep open: the Telegram mirror gap is real by source inspection, but the proposed preview-finalized mirror can double-write over-limit streamed final text because follow-up chunks already go through the newly mirrored send path.

## What This Changes

The PR adds Telegram transcript mirroring for direct outbound replies and preview-finalized replies, exports the transcript helper through the plugin SDK runtime barrel, refreshes the SDK API baseline, adds tests, and updates the changelog.

## Best Possible Solution

Keep the Telegram-owned mirror approach, but make transcript mirroring single-source for each delivered final answer; after that, maintainers can accept the proof override and land it to close https://github.com/openclaw/openclaw/issues/75991.

## Reproduction Assessment

Yes by source inspection and supplied before-fix logs: current main carries Telegram session context into dispatch, but direct `deliverReplies` finishes without appending an assistant delivery mirror. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No, not as submitted: the optional callback is the right owner-boundary shape, but the preview-finalized branch mirrors the whole final while long-final follow-up chunks are already mirrored through `sendPayload`. The safer fix is to mirror either the full final once or each delivered chunk once, not both.
