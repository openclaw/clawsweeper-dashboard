## Summary

Keep open: the doctor false positive is a valid bug, but this PR still masks invalid alternate-memory-plugin configs and the supplied real-behavior proof remains placeholder-heavy.

## What This Changes

The PR suppresses the doctor missing-memory-plugin note for non-default memory slot owners, adds two doctor-memory-search unit tests, and adds a changelog entry.

## Best Possible Solution

Have `memory-lancedb` expose the generic memory capability runtime, or require explicit active plugin ownership before doctor suppresses the missing-runtime note.

## Reproduction Assessment

Yes, at source level. Current main warns when backend config resolution returns null, and `memory-lancedb` is documented as a slot-owned active memory plugin while current main does not register a memory runtime for it.

## Solution Assessment

No. The suppression target is valid, but slot text alone is not the narrowest maintainable proof of active ownership; runtime capability registration or explicit configured/registry-confirmed ownership is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Require active ownership before suppressing:** `src/commands/doctor-memory-search.ts:323`
  - body: This treats a slot-only alternate memory selection as healthy because `plugins.entries[memorySlot]?.enabled` is `undefined` when the entry is missing, so `undefined !== false` returns true. With `plugins.slots.memory = "memory-lancedb"` but no valid configured entry, LanceDB registers only its warning service and no memory tools/hooks, and doctor would now hide the missing-runtime note for an unavailable memory plugin. Require an explicit configured entry or registry/capability-confirmed active owner before returning here.
  - confidence: 0.87
