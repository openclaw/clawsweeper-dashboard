## Summary

Keep open. Current `main` still preserves only one detected client tool call through the embedded runner metadata and both `/v1/responses` emitters still select a single pending call; the issue is now paired with open closing PR #75580, so it should remain open until that PR lands or is closed.

## What This Changes

The issue requests that `/v1/responses` return every client function tool call when an agent invokes multiple client tools in one turn, preserving order in non-streaming and streaming responses.

## Best Possible Solution

Land a reviewed fix equivalent to #75580: collect client tool calls in order at the runner boundary, carry them as ordered `pendingToolCalls`, emit every call from both OpenResponses modes, and keep focused regression coverage for JSON and SSE output order.

## Reproduction Assessment

Yes. The issue has a high-confidence source reproduction on current `main`: repeated client tool callbacks overwrite one variable, and the gateway later emits only `pendingToolCalls[0]`. A live curl is not required to verify the lossy path.

## Solution Assessment

Yes. The array-through-the-pipeline approach is the narrow maintainable solution because it fixes the first lossy layer and the two HTTP emitters without changing the public plugin or client tool schema.
