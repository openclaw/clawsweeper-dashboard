## Summary

Keep open. Current main still preserves only one detected client tool call through the embedded runner metadata, and both non-streaming and streaming `/v1/responses` handlers select a single pending call, so the linked closed PR has not been replaced by an implemented fix.

## What This Changes

The issue asks OpenClaw to return every client-side `function_call` output item, in order, when an agent calls multiple client tools during one `/v1/responses` turn.

## Best Possible Solution

Implement ordered client-tool-call arrays end to end: collect all detected calls in the embedded attempt result, carry them through run metadata as `pendingToolCalls`, emit every pending call in both `/v1/responses` response modes, and add order-preserving regression tests.

## Reproduction Assessment

Yes. The issue's `/v1/responses` scenario maps directly to current source: each client tool callback overwrites one variable, and the gateway later emits only `pendingToolCalls[0]`; a live curl was not needed for high-confidence source reproduction.

## Solution Assessment

Yes. The narrow array-based repair described by the issue and closed PR #52287 is the maintainable fix because it addresses the first lossy layer and both response emitters without changing the public plugin API.
