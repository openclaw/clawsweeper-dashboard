## Label Justifications

- `P1`: A PVC-backed pod can remain Ready while normal runtime writes fail, so managed deployments may continue routing user traffic to a pod that cannot reliably process work.
- `impact:message-loss`: The report names message handling and heartbeat/cron write paths as examples that can fail while the pod remains in service endpoints.
- `impact:session-state`: The failure mode affects transcript, media, and session persistence under the workspace/state paths.
- `impact:other`: The core maintainer-visible impact is an incorrect Kubernetes readiness signal for storage-unwritable gateway pods.

## Summary

Keep open: current main and the latest release still lack a workspace/storage writability signal in gateway readiness, and several open PRs are candidate fixes that have not merged yet.

## What This Changes

The issue requests that `/readyz` return not-ready when PVC-backed workspace or state storage can no longer accept critical writes, while keeping `/healthz` shallow.

## Best Possible Solution

Land a single gateway-owned storage writability readiness probe that makes `/readyz` return 503 while critical state/workspace paths are unwritable, keeps `/healthz` live, caches the probe, and covers the chosen storage roots with focused tests plus real gateway proof.

## Reproduction Assessment

Yes. I did not fill a PVC live in this review, but source inspection gives a high-confidence reproduction path: make runtime workspace writes fail with ENOSPC and current `main` still computes `/readyz` from startup, draining, channel, and event-loop signals only.

## Solution Assessment

Yes. The right endpoint is readiness, not liveness: `/readyz` should fail when critical storage cannot be written, while `/healthz` stays shallow so a full PVC does not cause restart churn.
