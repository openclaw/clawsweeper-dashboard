## Summary

Keep open: current main and the latest release still do not include a workspace or storage writability signal in gateway readiness, and an open clean PR already targets this issue with closing syntax.

## What This Changes

The issue asks for `/readyz` to return NotReady when critical PVC-backed workspace or state storage cannot be written, while keeping `/healthz` shallow.

## Best Possible Solution

Land one canonical gateway-owned storage writability readiness probe, keep `/healthz` shallow, then close this issue and redirect the closed sibling PRs to the landed fix.

## Reproduction Assessment

Yes. Source inspection shows `/readyz` is derived from startup, draining, event-loop, and channel health only, so storage writes can fail with ENOSPC without making readiness fail; I did not fill a live PVC in this read-only review.

## Solution Assessment

Yes. Readiness is the right endpoint for removing an unwritable pod from service endpoints, while `/healthz` should stay shallow because restarting will not free a full PVC.
