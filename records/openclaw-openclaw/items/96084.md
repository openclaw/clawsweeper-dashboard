- `P1`: A false-ready pod can keep receiving managed-deployment traffic while message/session/media persistence writes fail.
- `impact:message-loss`: The report describes user-visible message and media/session write paths failing while traffic continues to route to the pod.

- `impact:other`: The central impact is an incorrect Kubernetes readiness signal for a storage-unhealthy gateway, which is operationally significant but not covered by a more specific owned label.

## Summary

Keep open: current main and the latest release still lack a workspace/state writability signal in gateway readiness, and the open linked PR is the active candidate fix rather than merged implementation.

## What This Changes

The issue requests that `/readyz` return NotReady when critical OpenClaw workspace/state writes fail with disk-full or unwritable-storage errors while `/healthz` remains shallow liveness.

## Best Possible Solution

Land a focused gateway readiness fix that probes critical workspace/state writability with a cached write/fsync/unlink check, reports a clear failing reason on `/readyz`, and leaves `/healthz` as shallow liveness.

## Reproduction Assessment

Yes from source inspection, though I did not run a live PVC-fill scenario. Current `/readyz` delegates to readiness state that has no workspace write probe, so an ENOSPC write failure outside the channel-health path will not make readiness fail.

## Solution Assessment

Yes. A cached critical workspace/state writability signal in gateway readiness is the narrow maintainable fix, with the open linked PR as the current implementation candidate that still needs review and proof.
