## Summary

Keep open: current main and v2026.6.9 still compute gateway readiness from startup, draining, channel, and event-loop state without any critical workspace/storage writability signal, while several linked PRs are open candidate fixes rather than merged implementations.

## What This Changes

The issue requests that gateway `/readyz` return NotReady when critical PVC-backed workspace or state storage is unwritable or returns ENOSPC, while `/healthz` remains live.

## Best Possible Solution

Land one gateway-owned cached writability probe over the critical state/workspace roots, make `/readyz` return 503 with a clear reason while storage is unwritable, keep `/healthz` shallow, and validate fresh plus recovered storage behavior.

## Reproduction Assessment

Yes. I did not fill a PVC live, but source inspection gives a high-confidence path: make critical workspace writes fail with ENOSPC and current main still computes `/readyz` only from startup, draining, channel, and event-loop signals.

## Solution Assessment

Yes. Readiness is the right layer because Kubernetes should stop routing to a pod whose critical storage is unwritable, while liveness should stay shallow because restarting does not free a full PVC.
