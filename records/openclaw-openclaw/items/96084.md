## Summary

Keep open: current main and v2026.6.10 still compute `/readyz` without any workspace or storage writability signal, and open candidate PRs are implementation paths rather than reasons to close the issue before merge.

## What This Changes

The issue requests that gateway `/readyz` return NotReady when PVC-backed workspace or state storage is full or unwritable while `/healthz` remains live.

## Best Possible Solution

Land or reject one canonical gateway-owned storage-readiness implementation that probes critical writable roots, keeps `/healthz` shallow, returns `/readyz` 503 on unwritable storage, and proves recovery after storage becomes writable again.

## Reproduction Assessment

Yes, source-level. I did not fill a PVC in this review, but current main's `/readyz` path has no storage writability input, so an ENOSPC workspace write failure would not change readiness when the existing startup, draining, channel, and event-loop signals pass.

## Solution Assessment

Yes, the requested endpoint boundary is the right fix shape: readiness should fail for critical storage writability failures while liveness stays shallow. The best path is to review and land one canonical open PR rather than queue another parallel fix.
