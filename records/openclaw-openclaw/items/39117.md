## Summary

Keep open, but the PR is not mergeable as-is: current main has already moved part of the recovery/typing surface to safer generic paths, while this branch still introduces an active-run race and caller-spoofable reset audit attribution.

## What This Changes

The PR branch adds embedded-run watchdog lane recovery, aborted-run payload suppression, queued busy receipts, sessions.reset audit metadata/protocol fields, and focused tests.

## Best Possible Solution

Split or replace the branch with narrower changes: keep recovery aligned with the current diagnostic abort/drain path, handle busy feedback through the canonical queue UX issue, and add reset audit metadata only if attribution is server-derived and documented.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the watchdog clears active state while the old attempt can still be streaming, and sessions.reset audit attribution is accepted from RPC params. I did not run runtime tests in this read-only review.

## Solution Assessment

No. This PR is not the best current solution because main already has a safer generic recovery path, while the branch bundles runtime recovery, queue UX, payload behavior, and public audit protocol changes with correctness and security issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep active run marked until it stops:** `src/agents/pi-embedded-runner/run/attempt.ts:1258-1262`
  - body: The watchdog callback force-clears the session lane and ACTIVE_EMBEDDED_RUNS entry while the timed-out run is still known to be streaming. A later same-session run can then observe the session as idle before the old attempt has stopped emitting or writing transcript state, so keep the active marker until abort/drain confirms termination or gate force-clear on a confirmed stop.
  - confidence: 0.88
- **[P2] Derive reset audit actor server-side:** `src/gateway/server-methods/sessions.ts:307-312`
  - body: sessions.reset accepts triggeredBy and triggerSource from RPC params and persists them as audit attribution. Any caller allowed to reset a session can spoof who or what caused the reset, so authoritative actor/source fields should come from authenticated request context or be stored separately as caller-supplied hints.
  - confidence: 0.78
