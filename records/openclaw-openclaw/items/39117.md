## Summary

Keep this PR open for maintainer handling: it still contains potentially useful busy-session and queued-feedback work, but the branch is not safe to merge as-is because it mixes several behavior surfaces and introduces a force-clear race plus spoofable reset-audit attribution.

## What This Changes

The PR branch adds embedded-run watchdog lane recovery, aborted-run assistant payload suppression, queued busy receipts, sessions.reset audit metadata/protocol fields, and focused tests.

## Best Possible Solution

Split or replace the branch with narrower changes: keep recovery aligned with current abort-and-drain diagnostics, add aborted-payload suppression if still needed, route busy feedback through the canonical busy-feedback issues, and add reset audit metadata only with server-derived attribution and documented protocol semantics.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the watchdog release path clears active state while the attempt may still be streaming, and the audit fields are accepted from caller params. No runtime tests were run because this was a read-only review.

## Solution Assessment

No. The PR is not the best current solution because main already has a safer generic recovery path, while this branch bundles runtime recovery, queue UX, payload behavior, and public audit protocol changes with concrete correctness and security issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep active run marked until it stops:** `src/agents/pi-embedded-runner/run/attempt.ts:1254-1262`
  - body: The watchdog callback force-clears the session lane and ACTIVE_EMBEDDED_RUNS entry while the timed-out run is still known to be streaming. A later same-session run can then observe the session as idle before the old attempt has stopped emitting or writing transcript state, so keep the active marker until abort/drain confirms termination or gate force-clear on a confirmed stop.
  - confidence: 0.88
- **[P2] Derive reset audit actor server-side:** `src/gateway/server-methods/sessions.ts:307-312`
  - body: sessions.reset accepts triggeredBy and triggerSource from RPC params and persists them as audit attribution. Any caller allowed to reset a session can spoof who or what caused the reset, so authoritative actor/source fields should come from authenticated request context or be stored separately as caller-supplied hints.
  - confidence: 0.78

## Security Review
