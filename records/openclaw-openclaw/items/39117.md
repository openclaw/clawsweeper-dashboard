## What This Changes

The PR branch adds embedded-run watchdog lane recovery, aborted-run assistant payload suppression, queued busy receipts, sessions.reset audit metadata/protocol fields, and focused tests for those paths.

## Best Possible Solution

Split or replace the PR with smaller changes: keep recovery aligned with the current diagnostic abort-and-drain/force-clear model, add aborted-run payload suppression against the current payload media/directive logic if still desired, decide queued feedback through the canonical busy-feedback issues, and only add reset audit metadata through the session-reset service with documented protocol semantics and trustworthy attribution.

## Reproduction Assessment

Yes for the blocking PR regression by code inspection: the PR watchdog clears the active run registry while the same attempt is still known to be streaming, which allows later same-session work to observe the session as idle before the old attempt has stopped. For the remaining requested features, current-main source checks reproduce absence rather than runtime behavior; no tests were run because this review is read-only.

## Solution Assessment

No. The PR is not the best current solution because main now has a safer generic recovery path, while this branch mixes recovery, chat UX, payload behavior, and protocol audit changes and introduces a force-clear race plus spoofable audit metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep the run active until it actually stops:** `src/agents/pi-embedded-runner/run/attempt.ts:1254-1262`
  - body: The watchdog release timer fires after the abort is still streaming, then resets the session lane and clears ACTIVE_EMBEDDED_RUNS. That lets the next same-session run start while the old attempt can still emit replies or write transcript state, especially when session concurrency is above one. Reset queue bookkeeping only after abort/drain confirms the attempt stopped, or keep the active-run marker until final cleanup.
  - confidence: 0.88
- **[P2] Derive reset audit actor server-side:** `src/gateway/server-methods/sessions.ts:307-312`
  - body: sessions.reset accepts triggeredBy/triggerSource from RPC params and persists them as lastResetBy/lastResetSource. Any client allowed to call sessions.reset can therefore spoof the audit attribution. If these are audit fields, derive the actor/source from the authenticated request or store caller-supplied hints separately from authoritative metadata.
  - confidence: 0.78
