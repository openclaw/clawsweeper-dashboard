## Summary

Keep this open: current main and v2026.6.10 still derive the macOS LaunchAgent plist path from `HOME`, and the linked closing PR is still open and unmerged rather than shipped behavior.

## What This Changes

The issue reports that `openclaw gateway install` fails with launchctl error 5 when the macOS home directory is on an external APFS volume.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/90923 with one canonical boot-volume-aware LaunchAgent path resolver used by install, repair, restart, uninstall, status/audit, doctor detection, and update-helper paths.

## Reproduction Assessment

Yes, at source level: current main still writes and bootstraps the LaunchAgent plist under `HOME`, which is the external APFS path the report says launchd rejects. A full live repro needs macOS with an external APFS home and was not rerun here.

## Solution Assessment

Yes, the maintainable fix is a single boot-volume-aware LaunchAgent path resolver reused by every launchd lifecycle path. The active linked PR should remain the implementation path unless maintainers replace it with an equivalent canonical fix.
