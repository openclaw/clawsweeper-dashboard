## Summary

Keep this issue open: current main and the latest release still resolve the macOS LaunchAgent plist under HOME, and the linked open PR is a fix candidate rather than merged behavior.

## What This Changes

The issue asks OpenClaw to install and bootstrap the macOS gateway LaunchAgent from the boot-volume LaunchAgents directory when the user's home directory is on an external APFS volume.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90923 or an equivalent canonical boot-volume-aware LaunchAgent path resolver that keeps install, restart, uninstall, repair, status/audit, doctor detection, and app-adjacent lookup surfaces aligned.

## Reproduction Assessment

Yes, at source level: current main writes and bootstraps the LaunchAgent plist under HOME, which is the path the report says launchd rejects on an external APFS volume. I did not perform a fresh live macOS external-volume run in this checkout.

## Solution Assessment

Yes, the best fix shape is a single boot-volume-aware LaunchAgent path resolver reused by every launchd lifecycle path. The active linked PR should remain the implementation path until it lands or is replaced by an equivalent fix.
