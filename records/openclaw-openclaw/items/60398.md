## Summary

Keep open. Current main still resolves the macOS Gateway LaunchAgent plist under the resolved HOME directory and bootstraps that same path with launchctl, so the external-APFS-home failure remains actionable and is not implemented or superseded on main.

## What This Changes

Handle macOS Gateway installation when the user's home directory lives on an external APFS volume by using a launchd-bootstrap-safe per-user LaunchAgents plist location.

## Best Possible Solution

Add a shared macOS LaunchAgent plist-location resolver that detects a non-bootstrappable external HOME LaunchAgents directory and uses the boot-volume per-user LaunchAgents path consistently across install, stage, bootstrap repair, uninstall cleanup, restart handoff, update restart helper, status/doctor inspection, service audit, and the macOS app snapshot. Preserve existing labels, plist permissions, environment-wrapper behavior, log paths, and Linux/Windows service behavior.

## Reproduction Assessment

Yes. The report gives concrete macOS external-APFS HOME steps, and current-main code inspection reproduces the path decision by setting HOME to `/Volumes/.../Users/name`: the resolver still returns `/Volumes/.../Library/LaunchAgents/ai.openclaw.gateway.plist` and install bootstraps that path.

## Solution Assessment

Yes. A narrow shared resolver plus lifecycle wiring is the best maintainable direction; a docs-only workaround would leave install, restart, doctor/status, and app inspection inconsistent. The implementation should preflight the fallback path before force-unloading a working service.
