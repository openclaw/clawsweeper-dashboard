## Summary

Keep open. Current main and the latest shipped release still resolve the macOS Gateway LaunchAgent plist under the effective HOME directory and pass that same path to launchctl bootstrap, so an external-APFS home can still hit the reported launchd error path; I found no merged or open PR that supersedes this issue.

## What This Changes

Request that macOS gateway install use a launchd-safe boot-volume LaunchAgents path when HOME is on an external APFS volume.

## Best Possible Solution

Add one shared macOS LaunchAgent plist-location resolver that chooses a launchd-safe boot-volume path when HOME is external, then use it consistently across install, repair, restart, status/audit, update restart, and the macOS app snapshot.

## Reproduction Assessment

Yes, source-backed rather than live: the issue gives concrete external-APFS HOME steps, and current main still resolves and bootstraps the plist from HOME/Library/LaunchAgents. I did not run a live macOS external-APFS smoke in this Linux review.

## Solution Assessment

Yes. A shared resolver is the narrow maintainable fix; an install-only workaround would leave restart, repair, audit, update restart, and macOS app status paths inconsistent.
