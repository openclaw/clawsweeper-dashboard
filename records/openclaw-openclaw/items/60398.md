## Label Justifications

- `P1`: A supported macOS gateway install can fail completely for users whose HOME is on an external APFS volume, and --force can unload a working service before failing to re-bootstrap it.
- `impact:crash-loop`: The issue is about LaunchAgent bootstrap failure and gateway process availability on macOS.

## Summary

Keep open. Current main and v2026.5.18 still derive the macOS LaunchAgent plist path from HOME and pass that same path to launchctl bootstrap, so an external-APFS home can still hit the reported error 5 path; the merged launchd hardening PR fixed adjacent stdio/log failures but not the external-home plist location.

## What This Changes

The issue asks for macOS gateway install to place the LaunchAgent plist on a launchd-safe boot-volume path when the user home directory is on an external APFS volume.

## Best Possible Solution

Add one shared macOS LaunchAgent plist-location resolver that chooses a launchd-safe boot-volume path when HOME is external, then use it consistently across install, restart, audit/status, update restart, and macOS app snapshot paths.

## Reproduction Assessment

Yes, source-backed rather than live-reproduced here. The issue gives concrete external-APFS HOME steps, and current main still resolves and bootstraps the plist from HOME/Library/LaunchAgents.

## Solution Assessment

Yes. A shared LaunchAgent plist resolver is the narrow maintainable fix; an install-only special case would leave restart, update, audit/status, and macOS app surfaces inconsistent.
