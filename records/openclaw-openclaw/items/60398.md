## Summary

This issue should stay open: current main and the latest release still derive macOS LaunchAgent plist paths from `$HOME`, and the live closing path is https://github.com/openclaw/openclaw/pull/90923. The report is actionable and paired with an open implementation candidate, so cleanup should wait for that PR to land or be replaced.

## What This Changes

The issue reports that `openclaw gateway install` fails with launchd bootstrap error 5 when the user's home directory is on an external APFS volume and asks OpenClaw to write/bootstrap the LaunchAgent plist from the boot-volume user directory instead.

## Best Possible Solution

Land the open boot-volume-aware LaunchAgent resolver PR or an equivalent successor that makes the path canonical across daemon lifecycle and relevant app readers, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` still derive the LaunchAgent plist from `$HOME`, matching the reporter's external APFS home reproduction and launchd error 5 failure mode; I did not run a live macOS external-volume repro in this read-only review.

## Solution Assessment

Mostly yes. The open PR's single canonical resolver is the right fix shape, but the final solution should satisfy the sibling macOS app path-alignment concern already raised on that PR before this issue closes.
