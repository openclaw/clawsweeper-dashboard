## Summary

Keep this open: current main and v2026.6.10 still derive macOS LaunchAgent plist paths from `HOME`, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks macOS `gateway install` to place and bootstrap the LaunchAgent plist from the boot-volume user directory when the user's home is on an external APFS volume.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/90923 or an equivalent canonical boot-volume-aware LaunchAgent path resolver across install, repair, restart, uninstall, inspection, doctor, and update recovery paths.

## Reproduction Assessment

Yes, at source level: current main still writes and bootstraps the LaunchAgent plist under `HOME`, which is the external APFS path the report says launchd rejects. A full live repro needs macOS with an external APFS home and was not rerun here.

## Solution Assessment

Yes, the maintainable solution is a single boot-volume-aware LaunchAgent path resolver reused by every macOS launchd lifecycle path. The active linked PR should remain the implementation path unless maintainers replace it with an equivalent canonical fix.
