## Summary

Keep open. Current `main` still writes and bootstraps the macOS Gateway LaunchAgent plist under the resolved HOME directory, so the external-APFS HOME failure remains source-backed and actionable.

## What This Changes

Report that `openclaw gateway install` fails when macOS HOME is on an external APFS volume because launchd refuses to bootstrap the generated LaunchAgent plist from that volume.

## Best Possible Solution

Add one shared macOS LaunchAgent plist-location resolver that uses a launchd-safe boot-volume plist path when HOME is external, then wire it through install, stage, repair, uninstall, restart, status/audit, update restart, and the macOS app view.

## Reproduction Assessment

Yes, source-backed. The issue gives concrete external-APFS HOME steps, and current source still resolves and bootstraps the LaunchAgent plist under HOME, which would be `/Volumes/.../Library/LaunchAgents` for the reported setup.

## Solution Assessment

Yes. A shared LaunchAgent plist resolver is the narrow maintainable fix; changing only install would leave repair, restart, update restart, status/audit, and the macOS app inconsistent.
