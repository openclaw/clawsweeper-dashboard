## Summary

Keep open. Current main still resolves the macOS Gateway LaunchAgent plist under the resolved HOME directory and bootstraps that same path, so the external-APFS-home install failure remains source-backed and actionable.

## What This Changes

The issue requests that macOS gateway install avoid bootstrapping the LaunchAgent plist from an external APFS home directory.

## Best Possible Solution

Add one shared macOS LaunchAgent plist-location resolver that uses a launchd-safe boot-volume plist path when HOME is external, then wire it through install, stage, repair, uninstall, restart, status/audit, update restart, and the macOS app view.

## Reproduction Assessment

Yes, source-backed. The report gives concrete external-APFS HOME steps, and current-main code inspection shows that an external HOME still yields a /Volumes/.../Library/LaunchAgents plist that install passes to launchctl bootstrap.

## Solution Assessment

Yes. A shared LaunchAgent plist resolver is the narrow maintainable fix; a docs-only workaround or install-only special case would leave restart, repair, update restart, status/audit, and the macOS app inconsistent.
