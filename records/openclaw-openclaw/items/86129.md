## Summary

Keep open: current main still separates structured Ollama `thinking`/`reasoning` fields but treats inline `message.content` from native Ollama as visible text, so the reported Kimi Cloud reasoning leak remains plausible and actionable rather than implemented or superseded.

## What This Changes

The issue reports that Ollama-hosted Kimi models can emit inline reasoning into user-visible chat even when thinking is disabled, and asks for response-side stripping or model metadata to prevent that leak.

## Best Possible Solution

Add a narrow Ollama-provider Kimi Cloud response sanitizer that only strips the confirmed inline reasoning prefix/delimiter before visible text is emitted, with regression tests and live proof against the affected model when feasible.

## Reproduction Assessment

No live repro was run because it needs an Ollama Cloud Kimi setup, but there is a medium-confidence source reproduction path: current main treats inline `message.content` from native Ollama as visible text while only separating structured `thinking`/`reasoning` fields.

## Solution Assessment

Yes, the best fix appears to be provider-owned and narrow. A gateway-wide delimiter stripper or broad text heuristic should wait for stronger proof because it could corrupt unrelated model output.
