## Summary

Keep open: current main and v2026.5.22 still treat Ollama `message.content` as visible text, while existing reasoning fixes cover native fields, tagged thinking, Slack delivery, and OpenCode Go replay payloads rather than this Ollama Kimi inline-content leak.

## What This Changes

The issue requests an Ollama Kimi response sanitizer so inline model reasoning is not rendered as chat text when thinking is disabled.

## Best Possible Solution

Add a provider-owned Ollama Kimi cloud response sanitizer that buffers or reclassifies pre-delimiter inline reasoning before any text deltas reach channel delivery, with focused tests for streaming and final-message paths.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main sends the disable knobs but still emits all Ollama `message.content` chunks as visible text. I did not run a live Ollama Cloud Kimi request, so this is not marked live reproduced.

## Solution Assessment

Yes, a provider-owned Ollama Kimi sanitizer is the narrowest maintainable direction. A generic gateway splitter would be riskier because the delimiter and failure mode are model/provider-specific and the leak occurs during streaming before final output cleanup.
