## Summary

Keep open: current main and v2026.5.22 still only handle Ollama Kimi thinking as request/structured-response fields; inline `message.content` is accumulated as visible text with no Kimi Cloud response stripper, and the reporter added screenshot evidence after the prior review.

## What This Changes

The issue requests an Ollama-provider sanitizer so Kimi Cloud inline reasoning is not rendered as normal chat output when thinking is off.

## Best Possible Solution

Add a narrow Ollama-provider Kimi Cloud response sanitizer for the confirmed inline reasoning prefix, with focused stream tests and redacted live proof; keep broader user-configurable filtering in the separate product discussion.

## Reproduction Assessment

No high-confidence live reproduction was run because it needs an Ollama Cloud Kimi setup. Source inspection plus the reporter's screenshot make the path source-reproducible: inline `message.content` is delivered as visible text unchanged.

## Solution Assessment

Yes, a provider-owned sanitizer is the narrowest maintainable direction for this issue. A gateway-wide output filter is broader and should stay in the separate product discussion unless maintainers choose that policy explicitly.
