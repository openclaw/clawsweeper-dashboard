## Summary

Keep open: current main and the latest release still forward Ollama Kimi Cloud request-side thinking controls but treat inline `message.content` as visible text, so the reported reasoning leak remains plausible, security-sensitive, and not superseded by the older closed Kimi/opencode-go reports.

## What This Changes

The issue requests an Ollama-provider response sanitizer so Kimi Cloud inline reasoning emitted despite `Think: off` is not rendered in chat.

## Best Possible Solution

Add a narrow Ollama-provider Kimi Cloud response sanitizer that strips only confirmed inline reasoning before visible chat emission, with focused unit coverage and live redacted proof against an affected model.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review because it requires an Ollama Cloud Kimi setup. The source path is clear enough to reproduce with mocked chunks: structured thinking is separated, while inline `message.content` is emitted unchanged as visible text.

## Solution Assessment

Yes, a provider-owned sanitizer is the narrowest maintainable solution if live proof confirms the delimiter and variants. A gateway-wide delimiter stripper should be avoided unless maintainers intentionally choose a broader cross-provider policy.
