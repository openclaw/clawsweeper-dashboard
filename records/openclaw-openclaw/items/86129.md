## Summary

Keep open: current main and the latest release still only handle Ollama Kimi thinking on the request side; inline `message.content` from Ollama is accumulated as visible text with no Kimi-specific response sanitizer, so the reported reasoning leak remains plausible and actionable.

## What This Changes

The issue requests an Ollama-provider fix so `ollama/kimi-k2.6:cloud` and related Kimi Cloud models do not render inline reasoning text when thinking is off.

## Best Possible Solution

Add a narrow Ollama-provider Kimi Cloud response sanitizer that strips only the confirmed inline reasoning prefix before emitting visible text, with regression coverage and live proof when available.

## Reproduction Assessment

No high-confidence live reproduction was run in this review because it needs an Ollama Cloud Kimi setup, but the source path is clear: structured thinking is separated while inline `message.content` is emitted as visible text unchanged.

## Solution Assessment

Yes, a provider-owned sanitizer is the narrowest maintainable path; a gateway-wide delimiter stripper would need stronger proof to avoid corrupting unrelated model output.
