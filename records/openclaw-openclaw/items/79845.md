## Summary

Keep open for contributor and maintainer follow-up. The feature is not implemented on current main and could be useful, but this PR is not mergeable as-is: it corrupts the gateway update handler, adds a UI listener for an event the broadcaster would reject, misses reactive UI state, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a proposed `update.progress` event type, Control UI progress state, update-banner rendering, and CSS for live update progress in the Control UI.

## Best Possible Solution

Implement an additive, admin-scoped `update.progress` contract around the existing `runGatewayUpdate` progress callbacks, preserve `update.run` and `update.status`, declare reactive Control UI state, cover gateway/UI tests, and include redacted live gateway/browser proof while coordinating broader UX with https://github.com/openclaw/openclaw/issues/65011.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR diff shows `src/gateway/server-methods/update.ts` reduced to invalid non-TypeScript bytes, and current broadcaster code shows `update.progress` would be dropped as an unknown event. The requested feature itself still lacks live gateway/browser proof.

## Solution Assessment

No: the current patch is not the best way to solve the issue because it removes the existing update handler, does not register a scoped broadcast event, and does not make the new UI state reactive. The safer path is a small additive backend event plus focused UI state/rendering and tests.
