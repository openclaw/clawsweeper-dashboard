## Summary

Keep open: the feature direction is useful, but the current patch would not deliver progress frames to the Control UI because the new event is not registered with the Gateway broadcaster; the provided proof also does not show the real update-progress path running.

## What This Changes

The PR adds gateway `update.progress` broadcasts during `update.run` and Control UI state/rendering for an inline update progress panel.

## Best Possible Solution

Implement a typed, admin-scoped update progress event with broadcaster coverage, UI tests for event handling/rendering, corrected progress math, and redacted real behavior proof from an actual update run.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the added `update.progress` broadcasts use an event name current main denies by default, so the new UI listener will not receive frames. The requested feature itself still needs real gateway/browser proof after repair.

## Solution Assessment

No: the proposed direction is reasonable, but this patch is not the best implementation until the event is explicitly scoped, covered by tests, and shown in a real Control UI update flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Register the update progress broadcast event:** `src/gateway/server-methods/update.ts:75`
  - body: The new `context.broadcast("update.progress", ...)` calls use an event name that the Gateway broadcaster currently drops because it is not in `EVENT_SCOPE_GUARDS` and unknown non-plugin events return false. As written, the Control UI listener will never receive these progress frames; add the event with the appropriate admin scope and coverage.
  - confidence: 0.94
- **[P3] Advance the completed-step progress width:** `ui/src/ui/app-render.ts:1586-1589`
  - body: `index` is documented as 0-based, but the progress bar width uses `index / total`. On `step-complete`, step 1 of 10 still renders 0% and every completed step is one behind; derive completed count as `index + 1` for completed steps.
  - confidence: 0.86
