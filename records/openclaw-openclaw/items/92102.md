## What This Changes

The issue reports that plain `openclaw memory status` shows `Vector store: unknown` even when indexed vector data exists, while `--deep` reports ready.

## Best Possible Solution

Land one canonical memory-core fix, preferably using a vector-specific persisted readiness signal with real `openclaw memory status` proof, then close this issue through the merged PR.

## Reproduction Assessment

Yes, source-reproducible: plain `memory status` can use a status-only manager, skip vector probing, return undefined `storeAvailable`, and render `Vector store: unknown`. I did not run the CLI because this review is read-only and local CLI/tests can write state or caches.

## Solution Assessment

Partly. The requested bug fix is needed, but a broad chunk-count fallback is less precise than a vector-specific persisted signal such as `vectorDims`, because non-vector or degraded chunk rows can exist.
