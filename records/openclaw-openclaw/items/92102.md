## Summary

Current main and the latest release still preserve the reported fast-path behavior: plain `memory status` skips vector probing, `MemoryManager.status()` returns an undefined builtin vector-store state, and the CLI renders that as `unknown`. This issue should stay open because https://github.com/openclaw/openclaw/pull/92114 is an open linked fix candidate and https://github.com/openclaw/openclaw/pull/93268 is a competing open implementation path that maintainers still need to choose between.

## What This Changes

`openclaw memory status` should avoid showing `Vector store: unknown` for an already indexed builtin vector database when the fast path skips sqlite-vec probing.

## Best Possible Solution

Land one narrow memory-core fix that infers persisted vector-store readiness on the plain status path without probing embeddings or sqlite-vec, then close this issue through the merged PR.

## Reproduction Assessment

Yes from source inspection: a status-purpose builtin memory manager skips vector probing, returns undefined `storeAvailable`, and the CLI renders that as `Vector store: unknown`. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

Yes on direction, but not yet on the exact PR to land. A vector-specific persisted signal such as `vectorDims` looks safer than a generic chunk-existence fallback, and maintainers should consolidate the open PRs before merge.
