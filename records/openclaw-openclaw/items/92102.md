## Summary

Current main still has the reported fast-path mismatch: a fresh status-purpose memory manager leaves vector availability unprobed, so plain status can render `unknown` even though deep status later proves sqlite-vec ready. The issue is distinct from the related provider/index-identity and native-extension-loading reports and remains a narrow, actionable diagnostic bug.

## What This Changes

Make plain `openclaw memory status` report existing semantic vector-index readiness accurately without requiring the slow deep probe.

## Best Possible Solution

Expose a cheap persisted semantic-vector-index state separately from current-process sqlite-vec probe availability, then render plain status from that explicit contract while retaining deep status as the active runtime check.

## Reproduction Assessment

Yes at source level. Current main creates a fresh status manager with nullable vector availability and exports it without the deep vector probe, directly matching the supplied plain-versus-deep output.

## Solution Assessment

No for the exact suggested `chunks` query. The issue warrants a narrow fix, but generic chunks prove neither semantic-vector persistence nor successful sqlite-vec loading; those states should be represented explicitly.
