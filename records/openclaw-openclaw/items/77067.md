## Summary

Keep this PR open: it targets a real QQBot reliability gap, but the patch still relies on socket inactivity rather than validating HEARTBEAT_ACK receipt, so it can miss the silent-drop case it is meant to fix.

## What This Changes

The PR adds QQBot WebSocket compression, TCP keepalive/socket timeout handling, heartbeat interval logging, and a new keepalive-focused test file.

## Best Possible Solution

Add an explicit QQBot heartbeat ACK watchdog tied to the server-provided heartbeat interval, keep TCP keepalive as a transport assist if desired, and cover the real GatewayConnection lifecycle with regression tests plus a changelog entry.

## Reproduction Assessment

Yes, at source level: current main sends HEARTBEAT on the server interval and ignores HEARTBEAT_ACK, so a connection that remains open while inbound frames stop has no ACK-based reconnect path. I did not establish a live QQ network failure path during this read-only review.

## Solution Assessment

No: TCP keepalive and logging are useful, but the proposed fixed socket timeout is not the narrowest reliable fix for missing heartbeat ACKs. The safer solution is a protocol-level ACK deadline that closes or terminates the WebSocket when the expected ACK does not arrive.
