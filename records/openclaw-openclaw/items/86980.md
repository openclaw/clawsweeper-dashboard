## Summary

Keep open: current main still has a plausible CLI/control-plane boundary for this report, and the related plugin LLM auth work is adjacent rather than a closing fix. Root help can enter plugin CLI descriptor loading when plugin config affects help, CLI metadata registration passes an empty runtime object, and runtime inspection still module-loads the target plugin while docs promise scriptable JSON output.

## What This Changes

The issue asks for CLI help and plugin inspection to avoid misleading lossless-claw runtime LLM warnings/startup logs and keep `--json` output clean when plugin metadata is loaded.

## Best Possible Solution

Keep this open and make CLI help/inspection distinguish metadata-only registration from gateway/runtime execution while ensuring `--json` keeps plugin diagnostics off stdout.

## Reproduction Assessment

No high-confidence live reproduction was run against current main because it requires the external lossless-claw setup. Source inspection does show the reported boundary: root help can load plugin CLI metadata with an empty runtime object, and runtime inspect module-loads the target plugin.

## Solution Assessment

Unclear until maintainers choose the permanent CLI metadata/runtime boundary. The safest direction is to keep metadata/help on manifest or explicit metadata-only surfaces and reserve `runtime.llm.complete` diagnostics for real runtime contexts.
