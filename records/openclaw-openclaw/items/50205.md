## Summary

Keep open. Current main has a bundled Google provider and a `google-vertex` surface, but it still has no accepted `requestLabels` config and the Google/Vertex request body builder never emits `labels`. The discussion’s Vertex-only correction is important and matches Google’s Vertex AI docs, which document custom labels for `generateContent` and `streamGenerateContent` cost attribution. ([cloud.google.com](https://cloud.google.com/vertex-ai/generative-ai/docs/multimodal/add-labels-to-api-calls?utm_source=openai))

## What This Changes

The issue requests configurable static labels on outbound Gemini or Vertex generateContent requests so GCP billing can attribute costs by deployment, environment, or workload.

## Best Possible Solution

Add a provider-owned Google Vertex request-label path: validate and accept static labels in config, propagate them through model/provider resolution, inject them as `labels` only for `google-vertex` generate/streamGenerateContent requests, and document the Vertex-only billing attribution behavior.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: adding `models.providers.google-vertex.requestLabels` hits the strict provider schema without that key, and even a bypassed config would not reach the payload because the Google transport request type and builder have no labels field.

## Solution Assessment

The requested direction is correct for Vertex AI, but not for direct Gemini API calls. The narrow maintainable solution is Vertex-only label validation and injection in the Google provider path, rather than a generic arbitrary body patch across all providers.
