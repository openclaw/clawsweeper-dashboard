## Summary

Keep open: this is a real Gmail watcher lifecycle bug and the PR is focused, but the proposed re-entry cleanup can still hang watcher restart/config reload because it waits only for the child `exit` event after SIGTERM/SIGKILL. The supplied runtime proof covers respawn and shutdown cleanup, but not the repeated-start/config-reload path central to the fix.

## What This Changes

The PR tracks Gmail watcher respawn timeouts, clears old renewal/respawn timers, kills the previous `gog serve` process on `startGmailWatcher` re-entry, and adds mocked lifecycle tests.

## Best Possible Solution

Land a bounded lifecycle cleanup that stores and clears respawn timers, settles old-child shutdown on `exit`, `close`, `error`, or a final timeout, and proves repeated start or config reload leaves one active watcher.

## Reproduction Assessment

Yes, source-reproducible: current main creates an untracked respawn timeout and overwrites watcher process/renewal state on repeated start. I did not execute the repro because this review must keep the checkout read-only.

## Solution Assessment

No, not yet: storing and clearing respawn state is the right direction, but exit-only old-child cleanup is not robust enough for a restart path. Settle cleanup on more lifecycle events or a final timeout and add proof for the repeated-start path.
