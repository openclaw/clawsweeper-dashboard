## Summary

Keep open: this PR targets a real Gmail watcher lifecycle leak and moves cleanup before async setup, but the hard-shutdown timeout still resolves before the old child exits, leaving a stale exit handler able to clobber watcher state or schedule another respawn. The supplied terminal proof is useful, but it does not exercise the repeated-start path in a real runtime setup.

## What This Changes

The PR adds a Gmail watcher re-entry guard that clears the existing renewal interval, terminates an existing `gog serve` child before async setup, and adds focused Vitest coverage.

## Best Possible Solution

Make re-entry a single owner transition: clear the timer, terminate the old process, keep the old exit path retired until the child has actually exited after escalation, then start the replacement with regression coverage and redacted real repeated-start proof.

## Reproduction Assessment

Yes, source-level: current main overwrites the Gmail watcher child and renewal interval on repeated successful starts without clearing the old handles. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the guard belongs in the right place, but it must not release shutdown ownership until the old child has actually exited after timeout escalation. The PR also needs real repeated-start proof rather than only build, grep, startup, and mocked-test output.
