## Summary

Keep open: the PR targets a real Gmail watcher interval leak on current main, but the patch still makes direct re-entry a partial lifecycle transition and the supplied proof does not exercise a real restart or re-entry path.

## What This Changes

This PR adds a `clearInterval(renewInterval)` guard in `startGmailWatcher` plus fake-timer regression tests for repeated Gmail watcher starts.

## Best Possible Solution

Make Gmail watcher re-entry an atomic lifecycle transition: stop or reuse the existing serve before replacing process and timer ownership, then prove one live serve and one renewal timer remain.

## Reproduction Assessment

Yes, at source level: repeated successful `startGmailWatcher` calls on current main overwrite `renewInterval`, and `stopGmailWatcher` can only clear the latest handle. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Clearing the old interval after spawning a replacement process does not make re-entry safe when an existing `gog serve` still owns the port; the fix should handle process and timer ownership together.
