## Summary

Keep open: the PR targets a real Gmail watcher lifecycle leak on current main, but it is not merge-ready because the new re-entry cleanup can wait forever on an old child that never emits `exit`, and the supplied runtime proof does not exercise the central repeated-start/config-reload path.

## What This Changes

The branch tracks the Gmail watcher respawn timeout, clears renewal and respawn timers on re-entry/stop, terminates an existing `gog serve` child before starting a replacement, and adds watcher lifecycle tests.

## Best Possible Solution

Bound old-process cleanup on `exit`, `close`, `error`, or a final timeout after escalation, add focused regression coverage for a child that does not emit `exit`, and provide redacted live proof that repeated start leaves one active `gog serve` process.

## Reproduction Assessment

Yes, at source level: current main overwrites the stored Gmail watcher process and renewal interval on repeated successful starts, and the crash respawn timeout is fire-and-forget. I did not run the repo tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: tracking the respawn timeout and cleaning prior state before async setup is the right direction, but exit-only cleanup settlement is not safe enough for restart/config-reload availability. The safer fix is to settle on `exit`, `close`, `error`, or a final timeout and prove the live repeated-start path.
