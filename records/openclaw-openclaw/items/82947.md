## Summary

Keep open: this targets a valid current-main Gmail watcher leak, but the latest patch still has re-entry ownership races and the supplied terminal proof does not exercise a real watcher re-entry or restart path.

## What This Changes

The PR adds a `startGmailWatcher` re-entry guard and tests intended to stop an existing Gmail watcher process and renewal timer before starting a replacement.

## Best Possible Solution

Make watcher re-entry a single owner transition: stop or cancel existing process, interval, and pending respawn ownership before async replacement work, wait or escalate until the old serve is gone, then prove a real re-entry leaves one serve and one renewal timer.

## Reproduction Assessment

Yes, source-level: current main overwrites `watcherProcess` and `renewInterval` on repeated successful starts without clearing the old handles. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the direction is right, but the guard needs to own the old watcher before async watch setup and must not spawn a replacement until the old process is confirmed stopped or force-killed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move re-entry shutdown before async watch setup:** `src/hooks/gmail-watcher.ts:284-289`
  - body: This guard runs only after `startGmailWatch` has awaited, while `currentConfig` already points at the new config. If the old child exits during that await, its exit handler sees `shuttingDown === false` and queues the 5s respawn; this PR then spawns another serve at line 311, so the queued respawn can still create a duplicate process. Stop or mark the old watcher as shutting down before the async replacement work, or track and cancel the restart timeout.
  - confidence: 0.88
- **[P2] Do not spawn while the old process may still be alive:** `src/hooks/gmail-watcher.ts:300-302`
  - body: On the 1s timeout this resolves without SIGKILL or an exit confirmation, after `oldProcess` has been detached from `watcherProcess`. A slow or ignored SIGTERM can leave the old `gog serve` alive and untracked while the replacement starts on the same bind/port. Reuse the stop path's escalation behavior or fail the restart until the old process exits.
  - confidence: 0.82
