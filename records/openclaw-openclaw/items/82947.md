## Summary

Keep open: this PR addresses a real Gmail watcher lifecycle leak, but the new re-entry cleanup can hang while waiting only for the old child's `exit` event, and the supplied runtime proof still does not exercise live repeated start/config reload.

## What This Changes

The PR stores and clears Gmail watcher respawn timers, clears renewal intervals on re-entry/stop, kills the old `gog serve` process before spawning a replacement, and adds mocked lifecycle tests.

## Best Possible Solution

Keep the respawn/interval cleanup direction, but make old-process cleanup settle on `exit`, `close`, `error`, or a final timeout after escalation, add focused regression coverage for that path, and include redacted repeated-start runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main overwrites the stored watcher process and renewal interval on repeated successful starts, and the crash respawn timeout is not stored. I did not execute the repro because this review must keep the checkout read-only.

## Solution Assessment

No, not yet: tracking the respawn timeout and clearing prior state before async setup is the right direction, but exit-only cleanup settlement is not safe enough for a startup/reload lifecycle path.
