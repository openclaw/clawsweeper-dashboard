## What This Changes

The PR updates `src/hooks/gmail-watcher.ts` to clear any existing `renewInterval` before scheduling a new Gmail watcher renewal timer and adds a changelog entry.

## Best Possible Solution

Land the focused timer cleanup once after-fix behavior proof shows a restarted watcher no longer leaves the old renewal timer active, with a focused fake-timer regression test if maintainers want a source-level guardrail.

## Reproduction Assessment

Yes, source-level: current main assigns a new `renewInterval` in `startGmailWatcher` without clearing the old handle, while `stopGmailWatcher` already has the expected cleanup pattern. A mocked two-start/fake-timer check would reproduce duplicate renewal callbacks; I did not run tests because this review is read-only.

## Solution Assessment

Yes for the stated timer leak: clearing the old handle before replacing it is the narrow maintainable cleanup and matches the existing stop path. It should not be treated as a full idempotent-start solution because child process cleanup remains owned by `stopGmailWatcher` and the reload lifecycle.
