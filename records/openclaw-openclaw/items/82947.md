## Summary

Keep open: current main has a source-level Gmail watcher process/timer leak and this PR addresses the right lifecycle path, but the patch still has an unbounded exit-only cleanup wait and the supplied real proof does not exercise repeated start/config reload.

## What This Changes

This PR tracks the Gmail watcher respawn timeout, clears old watcher timers on re-entry and stop, terminates the old `gog serve` process before a replacement start, and adds mocked lifecycle tests.

## Best Possible Solution

Bound old-process cleanup on `exit`, `close`, `error`, or a final timeout after escalation, keep the respawn/interval cleanup, add focused regression coverage for the non-`exit` child path, and require redacted real repeated-start proof.

## Reproduction Assessment

Yes, source-reproducible: current main overwrites the stored watcher process and renewal interval on repeated successful starts, and the crash respawn timeout is not stored. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: tracking the respawn timeout and cleaning prior state before async setup is the right direction, but exit-only cleanup settlement is not safe enough for a startup/reload lifecycle path.
