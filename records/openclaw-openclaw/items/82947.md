## Summary

Keep open: the source-level interval leak is real and the patch is focused, but the current diff is not merge-ready because it can clear the renewal timer for a still-running first `gog serve` when a duplicate start spawns a second process that then fails to bind. The supplied proof also shows build output, compiled code, gateway startup, and tests, but not the real re-entry path this PR changes.

## What This Changes

The PR adds a `clearInterval(renewInterval)` guard in `startGmailWatcher` and fake-timer regression tests for repeated starts creating only one renewal callback.

## Best Possible Solution

Make `startGmailWatcher` re-entry a single owner transition: stop or reuse the existing watcher before replacing the process and timer, then prove a real restart or re-entry leaves one live serve process and one renewal timer.

## Reproduction Assessment

Yes, source-level: current main overwrites `renewInterval` without clearing the previous handle, so repeated successful starts can leave an unreachable timer. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: clearing the old interval after spawning a replacement process does not make re-entry safe when the old `gog serve` is still active. The maintainable fix should handle process and timer ownership together.
