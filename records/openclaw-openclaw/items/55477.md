## Summary

Keep open, but not merge-ready. The attribution problem is valid, but this patch mutates an internal stream option that the current LiteLLM/OpenAI-compatible request path does not serialize, and it would export stable session/channel identifiers without an approved opt-in privacy boundary.

## What This Changes

The branch adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` before outbound model calls.

## Best Possible Solution

Define one opt-in, provider-owned request metadata contract for supported LiteLLM/OpenAI-compatible payloads, coordinate it with https://github.com/openclaw/openclaw/issues/63901, and cover payload shape, privacy boundaries, tests, docs, and redacted LiteLLM proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the patch only enriches `options.metadata`, while the current LiteLLM/OpenAI-compatible path builds a separate request payload and only supports payload mutation through `onPayload` or existing extra-body seams. The original downstream LiteLLM callback claim was not reproduced live because the PR provides no inspectable real-environment artifact.

## Solution Assessment

No. The narrow maintainable solution is an opt-in provider request metadata contract that patches the supported wire payload, not a broad attempt-runner wrapper that may be invisible to the transport and can leak session routing identifiers by default.

- **[P2] Propagate metadata through the request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2453-2461`
  - body: This wrapper only adds fields to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds the outbound request body separately before calling `client.chat.completions.create`. Since the completions payload builder never reads `options.metadata`, LiteLLM callbacks still will not receive the advertised `session_key`, `message_channel`, or `context_limit` fields.
  - confidence: 0.93

  - body: The wrapper forwards stable session keys and channel identifiers whenever those params exist, with no LiteLLM-specific opt-in or documented privacy contract. Because session keys can encode platform, channel, topic, or direct-message routing data, this should be decided in the provider metadata API rather than enabled globally in the attempt runner.
  - confidence: 0.88
