## Summary

Keep open, but do not merge as-is: current source shows the new runner metadata does not reach the LiteLLM request payload, and the export policy is not approved.

## What This Changes

The PR adds a runner-level stream wrapper that tries to merge `session_key`, `message_channel`, and `context_limit` into model call metadata.

## Best Possible Solution

Implement a provider-owned LiteLLM/OpenAI-compatible metadata path that serializes an approved opt-in field set into the chat-completions payload with focused tests and redacted LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection shows LiteLLM uses `openai-completions`, while the completions transport builds the request payload without reading `options.metadata`, so the PR's fields do not reach LiteLLM callbacks.

## Solution Assessment

No. A runner-level wrapper is too broad and misses the LiteLLM wire payload; the maintainable fix belongs in the provider/transport layer with explicit metadata export policy.
