## Summary

Keep open. The observability goal is useful, but the PR is not merge-ready because it writes an unconsumed stream option instead of the LiteLLM request payload, and it exports session identifiers without an approved provider/privacy boundary.

## What This Changes

The PR adds a runner-level `streamFn` wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Define one opt-in, provider-owned request metadata contract for supported LiteLLM/OpenAI-compatible payloads, settle it with https://github.com/openclaw/openclaw/issues/63901, and cover payload shape, privacy boundaries, docs, and redacted real LiteLLM proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main routes LiteLLM through `openai-completions`, and that transport builds and sends params without consuming arbitrary `options.metadata`. The downstream LiteLLM callback claim still lacks inspectable live proof.

## Solution Assessment

No. A runner-level wrapper is too broad, misses the actual LiteLLM wire payload, and should be replaced by a provider-owned opt-in serialization path with an approved field boundary.
