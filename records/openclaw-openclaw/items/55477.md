## Summary

Keep open: the patch does not currently make LiteLLM receive the metadata, and it adds a broad outbound identifier path before the privacy boundary is decided.

## What This Changes

This PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into outbound model-call `options.metadata`.

## Best Possible Solution

A narrow LiteLLM-owned path should serialize approved metadata into the OpenAI-compatible request payload, preserve existing provider metadata, and leave other providers untouched.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: configure LiteLLM, follow the `openai-completions` stream path, and observe that the current payload builder never reads the PR's `options.metadata`.

## Solution Assessment

No. A runner-level wrapper is both too broad for identifier privacy and insufficient for LiteLLM because the relevant transport drops the metadata before sending the request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata through the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds `chat.completions.create` params without reading or merging `options.metadata`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata to LiteLLM or explicit opt-in:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for every provider whenever session fields are present. Since `session_key` can encode channel, peer, group, topic, thread, or direct-message identifiers, this needs a LiteLLM-only or explicit opt-in boundary before providers or wrappers can receive it.
  - confidence: 0.84
