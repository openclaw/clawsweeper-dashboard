## Summary

Keep this PR open, but it is not mergeable as proposed: the runner wrapper adds metadata to `options`, while the LiteLLM route currently sends OpenAI-compatible chat-completions params that never read or serialize that field, and exporting stable session identifiers also needs an explicit provider/privacy boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that stamps `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Implement a LiteLLM/provider-owned opt-in metadata path that serializes approved fields into the OpenAI-compatible chat-completions payload, with focused transport/plugin tests and redacted live callback proof.

## Reproduction Assessment

Yes. Source inspection shows the PR writes `options.metadata`, while the active LiteLLM `openai-completions` path builds and sends chat-completions params without reading or merging that field.

## Solution Assessment

No. The runner-level wrapper is too broad, misses the LiteLLM wire payload, and should be replaced by provider-owned transport metadata with an explicit export policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata into the LiteLLM payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes `options.metadata`, but LiteLLM uses the OpenAI-compatible completions transport and `buildOpenAICompletionsParams` never reads that option before `client.chat.completions.create`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.92

  - confidence: 0.86
