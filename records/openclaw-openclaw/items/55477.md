## Summary

Keep this PR open. The goal is useful, but the patch should not merge as written because the LiteLLM route uses OpenAI-compatible chat completions, and current main does not copy `options.metadata` into that request payload. It also sends stable session/channel identifiers from a runner-level wrapper without a LiteLLM-only or explicit opt-in boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` `streamFn` wrapper that merges `session_key`, `message_channel`, and `context_limit` into outbound model-call `options.metadata` when those run parameters are present.

## Best Possible Solution

Add a narrow, tested LiteLLM/OpenAI-completions metadata path that serializes only approved string fields into the actual request payload, preserves existing caller/provider metadata, gates the behavior by LiteLLM or explicit config, and documents the privacy semantics for session identifiers.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: LiteLLM resolves to `openai-completions`, and the current completions transport builds the chat-completions payload without reading or merging `options.metadata`.

## Solution Assessment

No. A runner-level wrapper is both too broad for privacy and insufficient for LiteLLM because the OpenAI-completions transport drops the metadata before the request payload is sent; a provider-owned or explicitly gated transport metadata seam is safer.
