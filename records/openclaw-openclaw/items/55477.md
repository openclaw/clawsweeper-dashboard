## Summary

Keep open for maintainer review: the patch is not safe to merge because it writes metadata only to runner options that the LiteLLM/OpenAI-compatible payload builder does not serialize, and it exports session identifiers without a provider/privacy opt-in. The contributor's claimed LiteLLM verification is also not attached as inspectable real behavior proof.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into model-call `options.metadata`.

## Best Possible Solution

Implement this as a provider-owned LiteLLM/OpenAI-compatible metadata path that serializes an approved, opt-in field set into the chat-completions payload, with focused tests and redacted LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection against the PR diff and current main shows LiteLLM uses `api: "openai-completions"` while `buildOpenAICompletionsParams` never reads `options.metadata`, so the added fields do not reach the request payload.

## Solution Assessment

No. The runner-level wrapper is too broad and does not reach LiteLLM; the maintainable fix belongs in the LiteLLM/provider transport layer with explicit opt-in policy.

## Review Findings
