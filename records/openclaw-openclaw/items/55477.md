## Summary

Keep open: this PR targets a useful LiteLLM attribution feature, but the current diff is not mergeable because it writes metadata into a runner-level options object that the LiteLLM OpenAI-compatible transport never serializes, and it exports stable session/channel identifiers without a provider opt-in privacy boundary.

## What This Changes

The branch adds a runEmbeddedAttempt stream wrapper that merges session_key, message_channel, and context_limit into model-call options.metadata.

## Best Possible Solution

Implement this in the LiteLLM/provider transport path with an explicit opt-in metadata contract, focused transport tests, and redacted LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection of the PR diff plus current main shows LiteLLM uses api "openai-completions", while buildOpenAICompletionsParams sends a request payload that never reads options.metadata.

## Solution Assessment

No. The runner-level wrapper is too broad and does not reach the LiteLLM request payload; the maintainable fix belongs in provider-owned transport metadata with explicit opt-in policy.
