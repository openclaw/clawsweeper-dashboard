## Summary

Keep open. The PR targets a useful provider attribution gap, but the branch still writes untyped stream options instead of the LiteLLM/OpenAI-compatible request payload, lacks an opt-in privacy boundary, and overlaps the canonical metadata design in https://github.com/openclaw/openclaw/issues/63901 plus the related open implementation PR at https://github.com/openclaw/openclaw/pull/54725.

## What This Changes

The PR adds a `runEmbeddedAttempt` `streamFn` wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Define one opt-in, provider-owned request metadata contract for supported LiteLLM/OpenAI-compatible payloads, then cover payload shape, privacy boundaries, docs, tests, and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main routes LiteLLM through `openai-completions`, and that transport builds and sends request params without consuming arbitrary `options.metadata`. The original downstream LiteLLM callback claim has not been live-reproduced with inspectable proof.

## Solution Assessment

No. A runner-level wrapper is too broad and misses the actual LiteLLM wire payload; the maintainable path is a provider-owned opt-in serialization contract with an approved field boundary.
