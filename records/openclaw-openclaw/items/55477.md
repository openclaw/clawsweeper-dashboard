## Summary

Keep this PR open, but do not merge the current patch: it writes metadata onto runner options, while the LiteLLM route uses the OpenAI-compatible completions transport that does not read or serialize that options metadata into the request body, and the session identifier export needs an explicit provider/privacy boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into outbound model-call `options.metadata` when those run parameters are present.

## Best Possible Solution

Add a narrow, tested LiteLLM-owned metadata path that serializes approved string fields into OpenAI-compatible request metadata, preserves existing provider metadata, gates the behavior by LiteLLM or explicit opt-in, and documents the privacy semantics.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: configure a LiteLLM model, follow the `openai-completions` transport, and observe that the current completions payload builder never reads the PR's `options.metadata`.

## Solution Assessment

No. A runner-level wrapper is too broad for session identifier privacy and insufficient for LiteLLM because the OpenAI-compatible completions transport drops that metadata before sending the request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata through the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds `chat.completions.create` params without reading or merging `options.metadata`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata to LiteLLM or explicit opt-in:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for every provider whenever session fields are present. Since `session_key` can include platform routing identifiers such as channel, group, topic, thread, or DM IDs, this needs a LiteLLM-only or explicit opt-in boundary before providers or wrappers can receive it.
  - confidence: 0.84
