## Summary

Keep open: the PR addresses a plausible LiteLLM attribution feature, but the patch does not actually serialize the new metadata into the LiteLLM chat-completions request and it would export stable session/channel identifiers to every provider without an explicit privacy gate.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into model-call `options.metadata`.

## Best Possible Solution

Add a provider-owned LiteLLM metadata path that serializes an approved field set into the OpenAI-compatible chat-completions payload behind an explicit privacy gate, with focused tests and redacted real LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the current patch failure: LiteLLM uses `openai-completions`, and current `buildOpenAICompletionsParams` never reads or serializes `options.metadata` into the request payload.

## Solution Assessment

No. The runner-level wrapper is too broad for session identifier privacy and insufficient for LiteLLM delivery; the maintainable path is provider/transport-owned serialization with explicit opt-in policy.
