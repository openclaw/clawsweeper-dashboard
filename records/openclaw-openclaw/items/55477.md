## Summary

Keep this PR open, but it is not merge-ready: current main still shows that LiteLLM uses the OpenAI-compatible completions transport, and the proposed runner wrapper only writes `options.metadata` without adding those fields to the actual request payload. The prior ClawSweeper review blockers still apply: real LiteLLM proof is missing and the session-identifier export needs maintainer API/privacy review.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Define one provider-owned, privacy-conscious LiteLLM/OpenAI-compatible request metadata contract that serializes approved fields into the actual payload, covers every advertised hook path, and includes focused transport tests plus redacted live LiteLLM callback proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main routes LiteLLM through `openai-completions`, and that transport builds and sends params without consuming arbitrary `options.metadata`. The downstream callback scenario itself still lacks inspectable live proof.

## Solution Assessment

No. A runner-level wrapper is too broad, misses the LiteLLM wire payload, and should be replaced by a provider-owned opt-in serialization path with an approved field boundary.
