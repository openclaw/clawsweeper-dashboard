## Summary

Keep open: the requested LiteLLM metadata forwarding is not on current main, and this PR is not mergeable as written because it writes to `options.metadata` without serializing that metadata into the LiteLLM OpenAI-compatible request payload and without a privacy boundary for session identifiers.

## What This Changes

Adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into outbound model-call `options.metadata`.

## Best Possible Solution

Add a LiteLLM/provider-owned metadata path that serializes an approved field set into OpenAI-compatible payloads behind an explicit privacy gate, with focused tests and redacted real LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection is enough: LiteLLM uses `openai-completions`, and current `buildOpenAICompletionsParams` never reads `options.metadata`, so this wrapper cannot deliver the requested metadata to LiteLLM callbacks.

## Solution Assessment

No. The runner-level wrapper is too broad for session identifier privacy and insufficient for LiteLLM delivery; the maintainable path is provider/transport-owned serialization with explicit opt-in policy.

## Review Findings
