## Summary

Keep open: the LiteLLM attribution goal is valid, but this PR is not mergeable because it adds metadata at a runner-wide layer that the LiteLLM OpenAI-compatible transport does not serialize, and it sends stable session/channel identifiers without a provider-specific or explicit opt-in boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into model-call `options.metadata` when session fields are present.

## Best Possible Solution

Add a narrow, tested LiteLLM or OpenAI-completions metadata path that serializes approved string fields only for LiteLLM or explicit opt-in, preserves existing provider metadata, and documents the privacy semantics.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: configure LiteLLM, follow `extensions/litellm/provider-catalog.ts` to the `openai-completions` transport, and observe that `buildOpenAICompletionsParams` never reads the PR's `options.metadata`.

## Solution Assessment

No. A runner-level wrapper is too broad for stable session identifiers and insufficient for LiteLLM unless the OpenAI-compatible request payload actually serializes the metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata through the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds `chat.completions.create` params without reading `options.metadata`. LiteLLM callbacks therefore will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata to LiteLLM or explicit opt-in:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for every provider when session fields exist, and `session_key` can include channel, thread, group, or DM identifiers. Keep this behind a LiteLLM/provider-owned or explicit opt-in boundary before sending it outside OpenClaw.
  - confidence: 0.84
