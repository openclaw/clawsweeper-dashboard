## Summary

Keep open for maintainer review: the feature is useful, but the proposed runner-level wrapper does not deliver metadata through LiteLLM's current OpenAI-compatible payload path and would export stable session/channel identifiers for every provider without an explicit privacy boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` `streamFn` wrapper that merges `session_key`, `message_channel`, and `context_limit` into model-call `options.metadata` when session fields are present.

## Best Possible Solution

Add a LiteLLM-owned, tested metadata path that serializes an approved field set into the OpenAI-compatible request payload behind a provider-specific or explicit privacy gate.

## Reproduction Assessment

Yes. Static source inspection is high-confidence: LiteLLM is registered as `openai-completions`, and that transport never reads the PR's `options.metadata` before constructing the outbound chat-completions payload.

## Solution Assessment

No. A runner-level wrapper is too broad for session identifier privacy and insufficient for LiteLLM delivery; the maintainable fix belongs in the LiteLLM/provider transport metadata path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata through the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM uses the `openai-completions` transport and that path builds the chat-completions params without reading `options.metadata`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata to LiteLLM or explicit opt-in:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for every provider whenever session fields are present. Because `session_key` can embed platform routing identifiers such as direct-message, guild, group, topic, thread, or channel IDs, this needs a LiteLLM-specific or explicit opt-in boundary before merge.
  - confidence: 0.84
