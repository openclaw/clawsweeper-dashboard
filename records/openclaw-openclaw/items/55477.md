## Summary

Keep open: the patch is still not mergeable because it adds a runner-level `options.metadata` wrapper, but current `main` routes LiteLLM through the OpenAI-completions transport that does not serialize that metadata, and exporting stable session/channel identifiers still needs an explicit provider/privacy decision plus real behavior proof.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Add a LiteLLM/provider-owned, tested transport metadata path that serializes an approved field set into the OpenAI-compatible request payload behind an explicit privacy gate while preserving existing provider metadata behavior.

## Reproduction Assessment

Yes. Static source inspection is high confidence: LiteLLM uses `openai-completions`, and current `buildOpenAICompletionsParams` never reads or serializes `options.metadata` into the outbound chat-completions payload.

## Solution Assessment

No. The proposed runner-level wrapper is both too broad for session identifier privacy and insufficient for LiteLLM delivery; the maintainable fix belongs in a provider/transport metadata path with explicit opt-in policy.
