## Summary

Keep open for maintainer review. The requested provider attribution remains useful and is still tracked by the open canonical metadata issue, but this PR does not serialize the new fields into the LiteLLM/OpenAI-compatible request payload and it introduces an unresolved privacy boundary for session identifiers.

## What This Changes

The branch adds a wrapper in `runEmbeddedAttempt` that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for each model call.

## Best Possible Solution

Define one opt-in, provider-owned request metadata contract for supported LiteLLM/OpenAI-compatible payloads, coordinate it with https://github.com/openclaw/openclaw/issues/63901, and cover payload shape, privacy boundaries, docs, and redacted real LiteLLM proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main routes LiteLLM through `openai-completions`, and that transport builds and sends request params without consuming arbitrary `options.metadata`. The downstream LiteLLM callback claim still lacks inspectable live proof.

## Solution Assessment

No. A runner-level wrapper is too broad, misses the actual LiteLLM wire payload, and should be replaced by a provider-owned opt-in serialization path with an approved field boundary.
