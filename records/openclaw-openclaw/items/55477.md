## Summary

Keep open, but do not merge as-is: current main still shows the LiteLLM path never reads `options.metadata`, and the PR would export stable session routing identifiers without a provider-specific opt-in boundary or inspectable real behavior proof.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Implement a LiteLLM/provider-owned opt-in metadata export that serializes approved fields into the OpenAI-compatible chat-completions payload, with focused transport/plugin tests and redacted live LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection is sufficient: LiteLLM is registered as `openai-completions`, the PR only writes `options.metadata`, and the completions builder/sender never reads that field before `chat.completions.create`.

## Solution Assessment

No. The runner-level wrapper is too broad, misses the LiteLLM wire payload, and should be replaced by provider-owned opt-in metadata serialization.
