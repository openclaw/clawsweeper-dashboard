## Summary

Keep this PR open, but do not merge it as-is: the runner-level wrapper writes to `options.metadata`, while the LiteLLM route uses the OpenAI-completions transport that does not serialize that field into the request payload, and the proposed export of stable session identifiers needs an explicit privacy boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into model-call `options.metadata`.

## Best Possible Solution

Add a LiteLLM-owned, tested transport metadata path that serializes an approved field set into the OpenAI-compatible request payload behind an explicit privacy gate while preserving existing provider metadata.

## Reproduction Assessment

Yes. Static source inspection is high confidence: LiteLLM uses `openai-completions`, and the OpenAI-completions transport on current main never reads `options.metadata` when constructing the outbound chat-completions payload.

## Solution Assessment

No. A global runner wrapper is both too broad for session identifier privacy and insufficient for LiteLLM delivery; the maintainable fix belongs in the LiteLLM/provider transport metadata path with an approved opt-in boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata through the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds the chat-completions params without reading `options.metadata`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata to LiteLLM or explicit opt-in:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for every provider whenever session fields are present. Because `session_key` can embed platform routing identifiers such as direct-message, guild, group, topic, thread, or channel IDs, this needs a LiteLLM-specific or explicit opt-in boundary before merge.
  - confidence: 0.84
