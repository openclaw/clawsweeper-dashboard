## Summary

Keep open: the attribution goal is useful and overlaps an open canonical provider-metadata issue, but this patch is not merge-ready because it writes uncontracted stream options that the LiteLLM/OpenAI-compatible payload path does not serialize and it exports session/channel identifiers without an opt-in privacy boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Define one opt-in, provider-owned request metadata contract for supported LiteLLM/OpenAI-compatible payloads, coordinate it with the canonical metadata issue, and cover payload shape, privacy boundaries, focused tests, docs, and redacted LiteLLM proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: LiteLLM is routed through `openai-completions`, and that transport builds a request payload without reading `options.metadata`. The claimed local LiteLLM callback behavior was not live-reproduced because the PR has no inspectable artifact.

## Solution Assessment

No. The maintainable fix is an opt-in provider request metadata contract that patches supported wire payloads, not a broad attempt-runner wrapper that may be invisible to the transport and can leak session routing identifiers by default.

- **[P2] Propagate metadata through the request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2453-2461`
  - body: This wrapper only adds fields to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds the outbound request body separately before calling `client.chat.completions.create`. Since the completions payload builder never reads `options.metadata`, LiteLLM callbacks still will not receive the advertised `session_key`, `message_channel`, or `context_limit` fields.
  - confidence: 0.93

  - body: The wrapper forwards stable session keys and channel identifiers whenever those params exist, with no LiteLLM-specific opt-in or documented privacy contract. Because session keys can encode platform, channel, topic, or direct-message routing data, this should be decided in the provider metadata API rather than enabled globally in the attempt runner.
  - confidence: 0.88
