## Summary

Keep open. The attribution goal is useful, but the patch writes to a stream option that the OpenAI-compatible/LiteLLM request path does not serialize, and exporting raw session/channel identifiers still needs maintainer privacy/API judgment plus inspectable real LiteLLM proof.

## What This Changes

The branch wraps `runEmbeddedAttempt`'s `streamFn` to merge `session_key`, `message_channel`, and `context_limit` into `options.metadata` before model calls.

## Best Possible Solution

Choose one opt-in provider attribution contract, serialize only approved metadata through the supported request-payload/provider seam, and land it with privacy docs, focused payload tests, and redacted LiteLLM callback proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the OpenAI-compatible transport builds a request payload and invokes `options.onPayload`, but never reads `options.metadata`, so the proposed fields would not reach LiteLLM through that path. The claimed local LiteLLM verification is not inspectable from the PR.

## Solution Assessment

No. The goal is valid, but stamping runner-level `options.metadata` is not the maintainable fix; the safer path is a provider payload or SDK seam with an explicit opt-in privacy contract.
