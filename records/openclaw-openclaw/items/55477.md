## Summary

Keep open: the requested LiteLLM attribution is useful, but this PR writes metadata only to runner-level `options.metadata`, which the current LiteLLM/OpenAI-compatible completions path does not serialize into the outbound request. The patch also exports stable session/channel identifiers without an approved provider-specific privacy gate and lacks inspectable real behavior proof.

## What This Changes

The PR adds a runner-level `streamFn` wrapper that injects `session_key`, `message_channel`, and `context_limit` into model call options for LiteLLM attribution.

## Best Possible Solution

Implement a provider-owned LiteLLM/OpenAI-compatible metadata path that serializes an approved opt-in field set into the chat-completions payload, with focused transport/plugin tests and redacted LiteLLM callback proof.

## Reproduction Assessment

Yes. Source inspection reproduces the blocker: LiteLLM uses `openai-completions`, and the current completions transport ignores `options.metadata` before sending the chat-completions payload.

## Solution Assessment

No. The runner-level wrapper is too broad, misses the LiteLLM wire payload, and should be replaced by a provider/transport-layer implementation with explicit metadata export policy.
