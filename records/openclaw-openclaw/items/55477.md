## Summary

Keep open: the attribution goal is useful, but this patch is not mergeable because it writes session fields to an unconsumed `options.metadata` bag, exports stable session identifiers without an approved provider opt-in/privacy boundary, and still lacks inspectable real LiteLLM proof.

## What This Changes

The branch adds a `runEmbeddedAttempt` stream wrapper that merges `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Define a provider-owned, privacy-conscious LiteLLM metadata export path that serializes only approved fields into the OpenAI-compatible request payload, with focused transport/plugin tests and redacted live LiteLLM callback proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main registers LiteLLM as `openai-completions`, and the completions options, builder, and send path do not consume arbitrary `options.metadata`. The downstream LiteLLM callback scenario itself still lacks inspectable live proof.

## Solution Assessment

No. A runner-level wrapper is too broad, misses the LiteLLM wire payload, and should be replaced by provider-owned opt-in metadata serialization with an approved field boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Propagate metadata into the LiteLLM payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes `options.metadata`, but LiteLLM uses the OpenAI-compatible completions transport and `buildOpenAICompletionsParams` never reads that option before `client.chat.completions.create`. As written, LiteLLM callbacks still will not receive these fields.

  - confidence: 0.86
