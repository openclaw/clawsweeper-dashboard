## Summary

Keep open: current main still has a source-backed Telegram auto-TTS path that can delete the active preview and resend the final media-bearing reply after newer user messages. The related open TTS delivery-mode issues are useful context, but they are broader product/API requests and do not supersede this recent regression report.

## What This Changes

The issue reports a Telegram auto-TTS regression where a late generated voice attachment recreates or reorders an already-visible text reply after the user has responded.

## Best Possible Solution

Preserve/finalize the visible Telegram text in place and deliver late final-mode TTS audio without deleting or recreating that text; keep broader configurable split/suppress delivery work in the related feature threads.

## Reproduction Assessment

Yes at source level, with reporter steps and screenshot support: preview text can be visible first, then final auto-TTS adds media, bypasses preview finalization, clears the preview, and sends a new media message. I did not run a live current-main Telegram session.

## Solution Assessment

Yes: a narrow Telegram final TTS delivery fix is the best next step. Adding a new TTS delivery-mode config would be broader than this regression and is already tracked by related feature issues.
