## Summary

Keep open: current main still has a source-backed Telegram auto-TTS ordering path where late synthesized audio turns the final text reply into a media-bearing payload, causing Telegram to clear the active preview and send a new durable message instead of preserving the already visible text in place.

## What This Changes

Keep Telegram text replies in place and ordered when late TTS audio is added after the assistant text is already visible.

## Best Possible Solution

Fix Telegram final TTS handling so the already visible streamed text finalizes or remains in place while late audio is delivered without changing the text message order or timestamp; leave the broader text-then-audio product redesign to https://github.com/openclaw/openclaw/issues/80630.

## Reproduction Assessment

Yes, source-backed: enable Telegram preview streaming with auto-TTS, let the assistant text become visible, then let final TTS add media after the user replies. Current main can still route that media-bearing final through normal send instead of preview finalization; I did not run a live Telegram session.

## Solution Assessment

Yes: the best fix is a narrow Telegram final media/TTS finalization change that preserves the existing text preview and sends late audio without re-sending the text. A repo-wide TTS follow-up redesign is broader and already tracked separately.
