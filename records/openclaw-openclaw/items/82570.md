## Summary

Keep open: current main still has a source-backed delayed auto-TTS ordering path, and the issue is already paired with an open implementation PR that uses closing syntax but has not merged.

## What This Changes

The issue requests that delayed TTS audio be added to an existing assistant reply without removing, reordering, or retimestamping the already visible text message.

## Best Possible Solution

Land the linked PR or an equivalent narrow fix that associates delayed TTS media with the original assistant reply in display history, then separately decide any broader text-then-audio delivery redesign in https://github.com/openclaw/openclaw/issues/80630.

## Reproduction Assessment

Yes, source-backed: enable auto-TTS, let the assistant text appear, reply before synthesis finishes, then let the delayed TTS media append after the later user message. I did not run a live repro in this read-only review.

## Solution Assessment

Yes: associating the delayed TTS media with the original assistant message is the narrow maintainable fix for the reported ordering and timestamp regression. A global TTS follow-up-message redesign is broader and belongs in the separate feature discussion.
