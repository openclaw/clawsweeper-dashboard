## Summary

Keep open: this is a recent Telegram auto-TTS/preview-finalization regression candidate with concrete source evidence. Current main still has a path where a final media-bearing TTS payload bypasses streamed text finalization and is sent through normal media delivery, which can move the voice/text bubble after later user replies.

## What This Changes

The issue asks OpenClaw to preserve an already-visible Telegram assistant text reply's order and timestamp when auto-TTS audio is added later.

## Best Possible Solution

Fix Telegram final TTS delivery so streamed/finalized text remains in place and late audio delivery cannot delete or recreate it after newer user messages; keep https://github.com/openclaw/openclaw/issues/80630 as the broader product request for universal follow-up audio behavior.

## Reproduction Assessment

We have a source-backed reproduction path plus reporter steps and screenshot, but no live current-main Telegram run in this review. The key path is final auto-TTS turning the reply into a media payload after Telegram preview streaming has already shown text.

## Solution Assessment

A narrow Telegram finalization/delivery fix is the best next step. A repo-wide separate-follow-up audio feature would be broader than this regression and is already represented by the related feature request.
