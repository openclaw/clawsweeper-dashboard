## Summary

Keep open: current main still has a source-backed Telegram auto-TTS ordering regression path where late synthesized audio turns the final reply into a media-bearing payload, bypassing preview finalization and using normal send delivery after the text preview was already visible.

## What This Changes

The issue reports a Telegram auto-TTS regression where late-arriving voice output can cause an already displayed text reply to be removed/reinserted after a newer user reply.

## Best Possible Solution

Fix Telegram final TTS delivery so already visible streamed text remains in place and late audio is delivered without changing the text message order or timestamp, while leaving the broader follow-up-audio feature request to https://github.com/openclaw/openclaw/issues/80630.

## Reproduction Assessment

Yes, source-backed: enable Telegram preview streaming with auto-TTS, let text become visible, then let final TTS add media to the final payload; current main still routes that media final through normal send instead of preview finalization. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes: a narrow Telegram final media/TTS finalization fix is the best path. A repo-wide text-then-audio delivery redesign is broader and already tracked separately.
