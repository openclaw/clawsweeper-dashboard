## Summary

Keep open: current main still has a source-backed auto-TTS ordering path where late audio turns an already visible final reply into a media-bearing payload and Telegram sends it normally after clearing the preview. The linked WebChat PR is open and unmerged, and it does not cover the Telegram-visible path described by the report.

## What This Changes

Reports that with auto-TTS enabled, a late voice/audio attachment can move an already visible assistant text reply after a user's intervening reply and change its timestamp/order.

## Best Possible Solution

Fix Telegram final TTS delivery so already visible preview text stays in place and late audio is delivered without reordering or timestamp changes; keep broader follow-up audio UX in https://github.com/openclaw/openclaw/issues/80630.

## Reproduction Assessment

Yes, source-backed: enable Telegram preview streaming with auto-TTS, let final text become visible, then let final TTS add media to the payload. Current main still routes that media-bearing final through normal send instead of preview finalization; I did not run a live Telegram repro.

## Solution Assessment

Yes: the narrow maintainable fix is in Telegram final media/TTS finalization, not a repo-wide text-then-audio redesign. The linked WebChat PR should either be retargeted as WebChat-only or expanded before it closes this issue.
