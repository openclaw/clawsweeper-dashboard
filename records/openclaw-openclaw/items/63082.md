## Summary

Keep open: the underlying shell-profile bug is valid, but this PR still does not wire the new resolver into the runtime code that actually checks and writes completion profile files, so the central behavior remains unfixed and doctor can report a path different from the one modified.

## What This Changes

The PR adds an environment-aware shell profile resolver to `src/cli/completion-cli.ts`, adds resolver unit tests in `src/cli/completion-cli.test.ts`, and changes doctor completion messages to print the resolver result.

## Best Possible Solution

Move the profile-path resolver into `src/cli/completion-runtime.ts`, make install/check/slow-pattern detection/doctor/setup/update share that one runtime path decision, preserve existing completion script/cache behavior, and keep broader PowerShell onboarding polish scoped to #44296 unless the shared resolver naturally covers it.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: with `ZDOTDIR` or `XDG_CONFIG_HOME` set, current main and this PR's unchanged `installCompletion` still choose `$HOME`-based zsh/fish profile paths, while the PR tests only the new helper in `completion-cli.ts`.

## Solution Assessment

No. The requested direction is right, but the PR implements the resolver in the wrong module and does not change the runtime installer/checker surfaces that determine actual behavior; the narrower maintainable fix is a shared runtime resolver used everywhere.
