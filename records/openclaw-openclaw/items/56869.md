## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clarify CIDR NO_PROXY support in the Compose sample:** `docker-compose.yml:32`
  - body: The `NO_PROXY` sample uses CIDR ranges as a general Docker gateway example, but process-level EnvHttpProxyAgent paths still use Undici's matcher, which does not understand CIDR. OpenClaw's CIDR extension only gates trusted-env proxy selection in guarded provider helpers, so users can still see private-IP traffic proxied. Use exact host/domain entries here or document the limitation explicitly.
  - confidence: 0.86
- **[P2] Clarify CIDR NO_PROXY support in the docs sample:** `docs/install/docker.md:195`
  - body: The docs repeat the same CIDR `NO_PROXY` value without explaining that it is only honored by OpenClaw's target-aware proxy decision, not by every EnvHttpProxyAgent-backed gateway path. This can make users expect private-network bypasses that some outbound paths will not provide.
  - confidence: 0.84
- **[P3] Remove duplicate Linux extra_hosts guidance:** `docs/install/docker.md:201-205`
  - body: Current `docker-compose.yml` already contains the Linux `host.docker.internal:host-gateway` mapping. Telling users to add the same block again, and adding another commented block near `environment`, creates stale guidance and can lead to duplicate YAML keys if copied literally.
  - confidence: 0.87
- **[P2] Scope the all-outbound proxy claim:** `docs/install/docker.md:207`
  - body: The final sentence says this routes all outbound gateway-container requests through the local proxy, but `NO_PROXY` intentionally bypasses some targets and not every gateway dependency has the same EnvHttpProxyAgent behavior. Narrow this to HTTP(S) egress that honors `HTTP_PROXY`/`HTTPS_PROXY`.
  - confidence: 0.78
