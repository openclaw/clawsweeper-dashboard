Full review comments:

- **[P2] Use NO_PROXY entries the env proxy actually honors:** `docker-compose.yml:32`
  - body: The new Compose sample includes CIDR ranges, but the gateway env-proxy path installs Undici's `EnvHttpProxyAgent`, whose `NO_PROXY` parser only handles exact host, suffix, and optional-port entries. Users copying this will expect private IP ranges to bypass the proxy, but those entries are treated as literal hostnames.
  - confidence: 0.9
- **[P2] Use supported NO_PROXY entries in the docs sample:** `docs/install/docker.md:195`
  - body: This docs sample repeats the same CIDR `NO_PROXY` values. Since Undici `EnvHttpProxyAgent` does not honor CIDR ranges, the published Docker guide would mislead users running mixed public and private-network traffic through the gateway container.
  - confidence: 0.9
- **[P2] Remove the duplicate extra_hosts block:** `docker-compose.yml:33-34`
  - body: Current main already includes an active `extra_hosts` mapping for `host.docker.internal` in the bundled Compose service. Adding another commented service-level block invites users to uncomment a duplicate key instead of relying on the existing mapping.
  - confidence: 0.86
- **[P2] Scope the all-outbound proxy claim:** `docs/install/docker.md:207`
  - body: `HTTP_PROXY` and `HTTPS_PROXY` cover env-aware HTTP clients, not every possible gateway egress path. Current network-proxy docs are explicit about managed proxy coverage and limits, so this sentence should not promise that all outbound requests route through these env vars.
  - confidence: 0.84
