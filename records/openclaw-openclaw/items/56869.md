## Best Possible Solution

Align the Docker guidance with current managed-proxy docs or narrow the env-var examples to the runtime paths they actually cover, remove duplicate host mapping guidance, and add redacted Docker proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use NO_PROXY values that match the runtime:** `docker-compose.yml:32`
  - body: The new `NO_PROXY` sample uses CIDR ranges, but the gateway also installs Undici's global `EnvHttpProxyAgent` for proxy env vars, and upstream Undici treats `NO_PROXY` entries as literal host/suffix/port values rather than CIDR ranges. Users copying this can expect private IP ranges to bypass the proxy when some gateway env-proxy traffic will not.
  - confidence: 0.88
- **[P2] Use supported NO_PROXY entries in the docs sample:** `docs/install/docker.md:195`
  - body: The Docker guide repeats the CIDR `NO_PROXY` values. Because the docs present this as gateway-container proxy setup, the example should not publish private-range bypass syntax that is only valid for selected OpenClaw guarded helper paths and not for the global Undici env-proxy dispatcher.
  - confidence: 0.88
- **[P2] Remove the duplicate extra_hosts block:** `docker-compose.yml:33-34`
  - body: Current `main` already includes an active `extra_hosts` mapping for `host.docker.internal` in this service. Adding another commented service-level block next to the env example invites users to uncomment a duplicate key instead of using the existing mapping.
  - confidence: 0.9
- **[P2] Scope the all-outbound proxy claim:** `docs/install/docker.md:207`
  - body: `HTTP_PROXY` and `HTTPS_PROXY` env vars do not guarantee every gateway egress path is routed through the local proxy; current network-proxy docs describe the broader managed proxy path and its limits. This sentence should be narrowed so users do not treat the env sample as full egress coverage.
  - confidence: 0.84
