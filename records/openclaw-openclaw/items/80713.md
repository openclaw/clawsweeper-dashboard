## Summary

Keep open: this is a useful iMessage tapback fix, but the latest diff still has a P2 own-mode ID-shape bug that can drop reactions on regular bot sends. The PR is also draft and carries the protected `maintainer` label, so cleanup closure is not appropriate.

## What This Changes

The PR routes inbound iMessage tapbacks as untrusted reaction system events, adds `reactionNotifications` config/docs/generated metadata, records tool-sent bridge results in the reply cache, and adds regression tests.

## Best Possible Solution

Keep the reaction-system-event design, but make prefixed and bare iMessage message IDs match consistently across tapback targets, echo cache, and reply-cache ownership checks before merge.

## Reproduction Assessment

Yes. Current main source-reproducibly turns `is_reaction` iMessage payloads into normal inbound text, and the PR finding is source-reproducible with a cached `p:0/GUID` bot send plus a normalized bare `GUID` tapback target.

## Solution Assessment

No, not as currently written. The reaction-event direction is maintainable, but default `own` mode needs prefixed/bare ID matching before it reliably handles bot-authored iMessage targets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match prefixed tapback target IDs in own mode:** `extensions/imessage/src/monitor/inbound-processing.ts:83`
  - body: `normalizeReactionValue` strips the `p:0/` prefix from `associated_message_guid`, but `sendMessageIMessage` and the echo/reply caches preserve native IDs such as `p:0/imsg-1`. In default `own` mode, a tapback on a regular bot send then looks up `imsg-1` while the cache is keyed by `p:0/imsg-1`, so the reaction is dropped instead of queued. Please normalize both cache writes and lookup keys, or try both forms during ownership checks.
  - confidence: 0.87
