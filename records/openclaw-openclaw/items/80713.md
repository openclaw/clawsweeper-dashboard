## Summary

Keep open: this is an active iMessage bug-fix PR with protected `maintainer` handling, and the latest head fixes the earlier prefixed-ID concern but still has a blocking reply-cache lookup bug in default `own` mode.

## What This Changes

The branch routes inbound iMessage tapbacks to untrusted reaction system events, adds `reactionNotifications` config/docs/generated metadata, records bridge-sent replies/effects/attachments in the reply cache, and adds regression coverage.

## Best Possible Solution

Keep the reaction-system-event design, but fix default `own` mode to call the reply-cache ownership lookup with the message ID plus chat/account context before merge.

## Reproduction Assessment

Yes. Current main source-reproducibly formats `is_reaction` payloads as normal message text, and the PR defect is source-reproducible by following the default reply-cache lookup after an echo-cache miss.

## Solution Assessment

No, not as currently written. The overall approach is maintainable, but the production reply-cache fallback must be wired to the existing two-argument API before this can reliably handle tool-sent bridge messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the reply-cache lookup its message ID:** `extensions/imessage/src/monitor/inbound-processing.ts:604-610`
  - body: `isKnownFromMeMessageId` is an object-shaped test hook, but the production fallback `isKnownFromMeIMessageMessageId` takes `(messageId, ctx)`. Please wrap the default call or align the signatures, and cover the non-injected path.
