## Summary

Keep open: this is an active draft PR with protected maintainer handling and a valid iMessage tapback bug fix direction, but the latest diff still has a P2 message-id normalization bug that can drop default own-mode reactions for cached `p:0/...` bot message ids.

## What This Changes

The PR routes iMessage inbound tapbacks into reaction system events, records tool-sent iMessage outputs as bot-authored targets, and adds reaction notification config, docs, generated metadata, changelog, and tests.

## Best Possible Solution

Keep the reaction-system-event direction, but make iMessage id matching consistent across tapback targets, echo cache, and reply cache before merge.

## Reproduction Assessment

Yes. The reported bug is source-reproducible on current main because `is_reaction` payloads are converted to normal inbound text, and the PR finding is source-reproducible with a cached `p:0/...` bot message id plus a tapback target normalized to the bare id.

## Solution Assessment

No, not as currently written. The reaction-event approach is maintainable, but the id normalization needs to check or store both prefixed and bare iMessage ids before the default own-mode path is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Match both prefixed and bare tapback target ids:** `extensions/imessage/src/monitor/inbound-processing.ts:83`
  - body: `normalizeReactionValue` removes the `p:0/` part from `associated_message_guid`, but current send and reply-cache code preserve native ids such as `p:0/imsg-1` as cache keys. In default `own` mode, a tapback on that cached bot message looks up `imsg-1` while the cache key is `p:0/imsg-1`, so the reaction is dropped instead of queued. Please normalize cache writes too or try both id forms during lookup.
  - confidence: 0.86
