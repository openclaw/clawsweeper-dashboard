## Summary

Keep open: current main still lacks the requested OpenClaw Jiti cache-dir override, and the PR has a blocking compatibility regression because its fallback forces `fsCache: true` when no OpenClaw override is set, overriding Jiti's own cache env defaults.

## What This Changes

The PR adds an `OPENCLAW_JITI_CACHE_DIR` Jiti `fsCache` override to shared plugin loader options, the plugin SDK root alias, and the Matrix wrapper, plus disables `fsCache` in one git-path regression fixture.

## Best Possible Solution

Land a rebased implementation that only sets `fsCache` to a trimmed non-empty `OPENCLAW_JITI_CACHE_DIR`, otherwise leaves it undefined, scopes loader caching by the effective option when needed, and documents the operator env surface with targeted regression coverage.

## Reproduction Assessment

Yes. The PR defect is source-reproducible: applying the diff with `OPENCLAW_JITI_CACHE_DIR` unset and `JITI_FS_CACHE=false` would still pass explicit `fsCache: true`, as shown by the PR diff and Jiti v2.7.0 option resolution.

## Solution Assessment

No. The requested override is reasonable, but this patch is not the best implementation because preserving compatibility requires omitting `fsCache` unless the OpenClaw-specific override is explicitly set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve Jiti fsCache defaults when no override is set:** `src/plugins/sdk-alias.ts:455-458`
  - body: `resolveJitiFsCache()` returns `true` when `OPENCLAW_JITI_CACHE_DIR` is absent or blank, so callers pass an explicit `fsCache: true` to `createJiti()`. Jiti resolves `JITI_FS_CACHE`/`JITI_CACHE` in its defaults and then applies user options afterward, so deployments that already disable filesystem cache writes with `JITI_FS_CACHE=false` would start writing cache files again.
  - confidence: 0.94
- **[P2] Add fsCache to loader cache isolation:** `src/plugins/sdk-alias.ts:466`
  - body: The loader options become dependent on an environment-controlled cache directory, but the loader cache key is still based on `tryNative` and aliases. Tests or long-lived processes that change the effective cache setting can reuse a Jiti loader created with a different `fsCache`; include the effective value or otherwise scope cache reuse.
  - confidence: 0.84
