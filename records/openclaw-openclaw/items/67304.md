## Summary

Keep this PR open, but request changes before merge. Current main still lacks the requested `OPENCLAW_JITI_CACHE_DIR` support and the PR targets real plugin-loader paths, but the proposed fallback forces `fsCache: true` when the OpenClaw override is unset, which changes existing Jiti env behavior and should block landing.

## What This Changes

The PR adds `OPENCLAW_JITI_CACHE_DIR` handling to the shared plugin loader, SDK root alias, and Matrix runtime wrapper, and disables Jiti filesystem caching in one git-path regression test fixture.

## Best Possible Solution

Land a narrow implementation that passes a trimmed `OPENCLAW_JITI_CACHE_DIR` only when it is non-empty, otherwise leaves `fsCache` unset so Jiti keeps its own defaults and env controls. Include cache-key isolation if the effective option can vary, add focused tests for the helper/direct loaders, and record the user-facing env var in the changelog and any appropriate operator docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to apply the PR, leave `OPENCLAW_JITI_CACHE_DIR` unset, set `JITI_FS_CACHE=false`, and observe that OpenClaw passes explicit `fsCache: true`, which overrides Jiti's env-derived default.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best implementation because it changes existing Jiti env semantics; the safer fix is to omit `fsCache` unless the OpenClaw-specific override is explicitly set.
