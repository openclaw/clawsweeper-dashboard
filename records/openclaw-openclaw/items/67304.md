## Summary

Keep open: current main still lacks the requested `OPENCLAW_JITI_CACHE_DIR` support, but this patch is not merge-ready because it forces Jiti filesystem caching on when the OpenClaw override is unset, omits cache-key isolation for the new effective option, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a new `OPENCLAW_JITI_CACHE_DIR` environment override to Jiti plugin loader paths and disables Jiti filesystem caching in one regression fixture.

## Best Possible Solution

Land a rebased implementation that only supplies a trimmed non-empty OpenClaw cache directory override, otherwise leaves `fsCache` undefined so Jiti honors its own defaults/env, scopes loader caching by the effective option where needed, and adds focused tests plus operator-facing docs/changelog coverage.

## Reproduction Assessment

Yes. The PR defect is source-reproducible from the diff plus Jiti v2.7.0 option resolution: with `OPENCLAW_JITI_CACHE_DIR` unset and `JITI_FS_CACHE=false`, the patch still passes explicit `fsCache: true`.

## Solution Assessment

No. The requested override is reasonable, but the patch should preserve Jiti defaults unless the OpenClaw-specific env var is non-empty and should isolate cached loaders by the effective cache option.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

- **[P1] Preserve Jiti fsCache defaults when no override is set:** `src/plugins/sdk-alias.ts:455-458`

Jiti resolves `JITI_FS_CACHE`/`JITI_CACHE` in its defaults and then applies user options, so this would regress existing deployments that disable filesystem cache writes through Jiti env vars.
  - confidence: 0.94

Tests or long-lived embedded processes that change the effective setting can reuse a Jiti loader created with a different cache directory or cache-disabled state.
  - confidence: 0.84
