## Summary

Keep open: current main still does not implement `OPENCLAW_JITI_CACHE_DIR`, but this PR is not merge-ready because it overrides Jiti's existing cache-disable env behavior when the OpenClaw variable is unset, does not scope cached loaders by the effective cache option, and lacks required after-fix real behavior proof.

## What This Changes

The branch adds `OPENCLAW_JITI_CACHE_DIR` handling to shared plugin Jiti options, SDK root-alias loading, the Matrix runtime wrapper, and disables `fsCache` in one git-path regression test fixture.

## Best Possible Solution

Land a narrow env override only after maintainers accept the surface: preserve Jiti's native env/default behavior when unset, isolate loader reuse by the effective cache option, rebase the Matrix wrapper portion onto current main, and add focused tests plus operator-facing documentation.

## Reproduction Assessment

Yes for the PR defects: source inspection plus the Jiti v2.7.0 contract shows that the added fallback forces `fsCache: true` when the OpenClaw override is absent, and current loader-cache code keys only on aliases and `tryNative`. This is a feature PR rather than a broken-current-main bug report.

## Solution Assessment

No. The requested override is plausible, but the current patch should omit `fsCache` unless the OpenClaw override is non-empty, include the effective cache option in loader reuse if it can vary, and rebase away stale Matrix wrapper assumptions.

A test or embedded long-lived process that changes the effective cache setting can reuse a Jiti loader created with a different cache path or cache-disabled state; include the effective `fsCache` value or otherwise prevent cross-setting reuse.
  - confidence: 0.84
