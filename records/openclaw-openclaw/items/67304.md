## Summary

Keep open: current main still lacks `OPENCLAW_JITI_CACHE_DIR`, and the feature idea is plausible, but this PR is not mergeable as-is because it changes Jiti default/env behavior, does not isolate cached loaders by the effective cache option, is stale against the current Matrix runtime wrapper, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds `OPENCLAW_JITI_CACHE_DIR` to Jiti loader options in the shared plugin loader, SDK root alias, and Matrix runtime wrapper, plus disables `fsCache` in one git-path regression test fixture.

## Best Possible Solution

Land a narrow cache-dir override only after maintainers accept the env surface: pass a trimmed path only when non-empty, otherwise leave `fsCache` undefined so Jiti keeps its own env/default behavior; isolate loader reuse by the effective cache option where it can vary; drop stale Matrix wrapper churn; and add focused tests plus operator-facing docs or changelog handling.

## Reproduction Assessment

Yes for the PR defects: source inspection plus the Jiti v2.7.0 source contract show that the patch forces `fsCache: true` when the OpenClaw override is absent and makes shared loader options depend on a setting that is not in the loader cache key. This is a feature PR, not a broken-current-main bug report.

## Solution Assessment

No. The requested override is plausible, but the current patch should preserve Jiti defaults when unset, include cache-option isolation if needed, rebase away the stale Matrix wrapper hunk, and include proof/docs before merge.

  - confidence: 0.84
