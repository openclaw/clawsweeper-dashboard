## Summary

Keep open: current main still lacks the requested cache-directory override, but this PR should not merge as written because it overrides Jiti's existing fsCache env controls when the new OpenClaw env var is unset and lacks real after-fix behavior proof.

## What This Changes

The PR adds `OPENCLAW_JITI_CACHE_DIR` to shared plugin loader, SDK root alias, and old Matrix runtime Jiti calls, plus disables `fsCache` in a git-path regression fixture.

## Best Possible Solution

Land a rebased implementation that only supplies a trimmed non-empty OpenClaw cache directory, otherwise leaves `fsCache` undefined so Jiti keeps honoring its own defaults, with cache-key isolation, focused tests, and operator docs/changelog coverage.

## Reproduction Assessment

Yes. The PR defect is source-reproducible: with `OPENCLAW_JITI_CACHE_DIR` unset and `JITI_FS_CACHE=false`, the diff still passes explicit `fsCache: true`, and Jiti v2.7.0 applies user options after env-derived defaults.

## Solution Assessment

No. The requested override is reasonable, but the current patch should omit `fsCache` unless the OpenClaw-specific override is non-empty and should isolate cache reuse by the effective setting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve Jiti fsCache defaults when unset:** `src/plugins/sdk-alias.ts:455-458`
  - body: `resolveJitiFsCache()` falls back to `true`, so OpenClaw now passes an explicit `fsCache` option even when the new env var is absent or blank. Jiti resolves `JITI_FS_CACHE`/`JITI_CACHE` first and then applies user options, so this regresses deployments that already disable filesystem cache writes; omit `fsCache` unless the OpenClaw override is non-empty and apply the same rule to the duplicated loader call sites.
  - confidence: 0.94
- **[P2] Include fsCache in loader cache keys:** `src/plugins/sdk-alias.ts:466`
  - body: This line makes shared loader options depend on an env-controlled cache directory, but the loader cache key still only includes `tryNative` and aliases. Tests or long-lived processes that change the effective cache setting can reuse a Jiti loader created with a different `fsCache`; include the effective value or otherwise scope cache reuse.
  - confidence: 0.84
