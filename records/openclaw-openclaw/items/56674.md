## Summary

This PR should stay open: it targets a real OpenResponses reasoning gap on current main, but the branch still has a security-sensitive design concern around publishing raw reasoning on the shared agent-event bus and lacks real after-fix behavior proof.

## What This Changes

The PR adds raw reasoning capture from embedded-agent thinking events and returns it from `/v1/responses` as non-stream reasoning output items plus streaming `response.reasoning.*` SSE events, with focused gateway and agent tests.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit reasoning-visibility opt-in that preserves channel/plugin gates, then document the HTTP contract and cover streaming, non-streaming, and plugin-isolation behavior.

## Reproduction Assessment

Yes, at source level. Current main has no Responses reasoning output/events, and the PR diff shows where raw thinking would be captured and emitted for the HTTP surface.

## Solution Assessment

No. Returning reasoning on `/v1/responses` is a reasonable direction, but publishing hidden raw thinking onto the shared agent-event bus before an explicit visibility boundary is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This emits `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. Plugin agent-event subscriptions without a `streams` filter receive all shared agent events, so hidden raw reasoning can become visible to plugins even though server-chat suppresses websocket/node broadcasts. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
- **[P2] Document the new Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the OpenResponses docs still say `reasoning` is accepted and ignored. Merging this would leave the documented HTTP API contract wrong for clients trying to use or avoid the new reasoning surface.
  - confidence: 0.82
