## Summary

Keep open: current main still does not implement Responses reasoning output, but this PR still has a blocking raw-reasoning exposure concern and lacks contributor real behavior proof before merge.

## What This Changes

The PR adds `/v1/responses` reasoning output items and `response.reasoning.*` SSE events by emitting embedded-agent `thinking-raw` events and collecting them in the gateway, with agent and gateway regression tests.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit visibility opt-in that preserves plugin/channel reasoning gates, then update the Responses docs and focused gateway/agent/plugin tests.

## Reproduction Assessment

Yes, from source inspection. Current main documents and implements `/v1/responses` as assistant/tool-output-only for reasoning, while the PR diff clearly changes the path by publishing `thinking-raw` events for HTTP capture.

## Solution Assessment

No. Returning reasoning on the HTTP API is a useful direction, but routing hidden raw thinking through the shared agent-event/plugin subscription bus before an explicit visibility gate is not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This emits `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. `emitAgentEvent` feeds the shared listener set, and plugin agent-event subscriptions without a stream filter receive all streams, so hidden raw reasoning can become visible to plugins even though server-chat suppresses websocket/node broadcasts. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
