## Summary

Keep open. The feature is useful, but this PR still needs maintainer API/security review because it publishes hidden raw reasoning on the shared agent-event bus, and it lacks required after-fix real behavior proof.

## What This Changes

The PR adds `/v1/responses` reasoning output items and `response.reasoning.*` SSE events by publishing embedded-agent raw thinking events, collecting them in the gateway, and adding focused agent/gateway tests.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit visibility opt-in that preserves channel/plugin reasoning gates, then update the Responses docs and focused gateway/agent/plugin tests.

## Reproduction Assessment

Yes, at source level. Current main lacks `/v1/responses` reasoning output/events, and the PR source clearly shows the new raw-reasoning publication path before the existing visibility gate.

## Solution Assessment

No. Returning reasoning on the HTTP API is a valid direction, but publishing hidden raw thinking through the shared agent-event/plugin bus is not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This emits `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. Plugin agent-event subscriptions without a `streams` filter receive all shared agent events, so hidden raw reasoning can become visible to plugins even though server-chat suppresses websocket/node broadcasts. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
- **[P2] Document the new Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the OpenResponses docs still say `reasoning` is accepted and ignored. Merging this would leave the documented HTTP API contract wrong for clients trying to use or avoid the new reasoning surface.
  - confidence: 0.82
