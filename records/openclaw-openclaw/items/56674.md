## Summary

Keep this PR open for maintainer review. The feature gap is real on current main, and the PR fixes several earlier review-thread issues, but the current approach still publishes raw reasoning on the global agent-event bus before the existing reasoning visibility gate, which leaves a concrete privacy/API contract blocker before merge.

## What This Changes

The PR adds raw thinking capture from embedded agent runs and returns reasoning output items plus reasoning SSE events from `/v1/responses`, with gateway and agent tests and server-chat suppression for `thinking-raw` broadcasts.

## Best Possible Solution

Ship OpenResponses reasoning support only after the capture path is explicitly gated by documented reasoning visibility or an authorized per-request HTTP opt-in, keeps raw thinking out of broad plugin/session event streams by default, updates docs and changelog, and covers reasoning-off, authorized opt-in, streaming, tool-call ordering, and plugin-subscription exposure cases.

## Reproduction Assessment

Yes. Source inspection is enough: current main gates reasoning events behind `state.streamReasoning`, while the PR emits `thinking-raw` before that gate and the plugin host bridge can dispatch unfiltered agent-event subscriptions.

## Solution Assessment

No. Returning reasoning from `/v1/responses` is the right direction, but publishing raw thinking through the global agent-event bus before the visibility decision is not the safest implementation; an HTTP-scoped capture path or explicit gate should come first.
