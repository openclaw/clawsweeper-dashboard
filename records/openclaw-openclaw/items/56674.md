## Summary

Keep open: the feature is useful, but the patch is not merge-ready because it publishes raw reasoning on the shared agent-event path before an explicit plugin/visibility boundary, leaves the public OpenResponses docs stale, and lacks after-fix real behavior proof.

## What This Changes

The PR adds raw embedded-agent reasoning capture and exposes it as OpenResponses reasoning output items plus streaming `response.reasoning.*` events from `/v1/responses`.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit reasoning-visibility opt-in that does not publish hidden raw thinking to general plugin/agent subscribers, then document and validate non-streaming, streaming, and plugin-isolation behavior.

## Reproduction Assessment

Yes, at source level for the missing API surface. Current main documents `/v1/responses` reasoning as ignored and lacks reasoning stream event types, while the PR diff adds the capture and emission path.

## Solution Assessment

No, not yet. Returning reasoning on `/v1/responses` is a reasonable direction, but the current shared-event implementation bypasses plugin visibility controls and the public API docs remain wrong.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This publishes `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. `server-chat` suppresses websocket and node-session sends, but plugin host-hook subscriptions without a `streams` filter still receive shared agent events, so hidden raw reasoning can become visible to plugins. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
- **[P2] Document the new Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the OpenResponses docs still say `reasoning` is accepted and ignored. Merging this would leave the documented HTTP API contract wrong for clients trying to use or avoid the new reasoning surface.
  - confidence: 0.82
