## Summary

Keep open. Current main still lacks the requested OpenResponses reasoning output, but this PR is not mergeable as-is because it publishes hidden raw reasoning through the shared agent-event bus and still lacks after-fix real behavior proof.

## What This Changes

The PR adds `/v1/responses` reasoning output items and `response.reasoning.*` SSE events by emitting embedded-agent raw thinking events and collecting them in the gateway HTTP handler.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit visibility opt-in that preserves plugin/channel reasoning gates, then update the Responses docs and focused gateway/agent/plugin tests.

## Reproduction Assessment

Yes, from source inspection. Current main is assistant/tool-output-only for `/v1/responses`, while the PR head emits raw thinking before the reasoning visibility gate and the plugin event bridge forwards unfiltered shared events to plugin subscriptions.

## Solution Assessment

No. Returning reasoning on the HTTP API is a useful direction, but routing hidden raw thinking through the shared agent-event/plugin subscription bus before an explicit visibility gate is not the safest implementation.
