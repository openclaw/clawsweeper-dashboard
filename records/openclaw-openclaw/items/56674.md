## Summary

Keep open: the feature is still useful and current main still lacks HTTP reasoning output, but this PR is not merge-ready because raw reasoning exposure is not gated, the public API contract is undocumented, real behavior proof is missing, and the branch is currently conflicting.

## What This Changes

The PR captures embedded-agent thinking and returns it from `/v1/responses` as reasoning output items and `response.reasoning.*` SSE events.

## Best Possible Solution

Ship a refreshed, documented HTTP reasoning contract that preserves current default silence, scopes raw capture away from broad plugin-visible events, and exposes reasoning only through an explicit policy such as existing `reasoningDefault` or a per-request opt-in.

## Reproduction Assessment

Yes, at source level. Current main documents Responses `reasoning` as ignored, serializes assistant text without collecting reasoning, and streams only assistant/lifecycle events; I did not run a live reasoning-capable model in this read-only review.

## Solution Assessment

No, not yet. Returning structured HTTP reasoning is plausible, but this implementation needs an explicit visibility/security gate, a plugin-safe capture path, docs, rebase, and real behavior proof before it is the maintainable solution.
