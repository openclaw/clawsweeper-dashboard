## Summary

This PR should stay open as a substantive Gateway feature proposal, but it is not merge-ready: the current branch still needs real behavior proof and maintainer judgment on the raw reasoning visibility boundary before exposing reasoning through `/v1/responses`.

## What This Changes

The PR adds raw embedded-agent reasoning capture and returns it as `/v1/responses` reasoning output items plus streaming `response.reasoning.*` SSE events.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit reasoning-visibility opt-in that preserves channel/plugin visibility gates, then document and validate non-streaming, streaming, and plugin-isolation behavior.

## Reproduction Assessment

Yes, at source level. Current main documents `/v1/responses` reasoning as ignored and has no reasoning output/event schema, while the PR diff shows the new capture and emission path.

## Solution Assessment

No. Returning reasoning through `/v1/responses` is a reasonable feature direction, but publishing hidden raw thinking through the shared agent-event bus before an explicit visibility boundary is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This emits `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. Plugin host-hook subscriptions without a `streams` filter receive all shared agent events, so hidden raw reasoning can become visible to plugins even though `server-chat` suppresses websocket and node broadcasts. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
- **[P2] Document the new Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the OpenResponses docs still say `reasoning` is accepted and ignored. Merging this would leave the documented HTTP API contract wrong for clients trying to use or avoid the new reasoning surface.
  - confidence: 0.82
