## Summary

Keep this PR open. The feature is useful and current main still lacks OpenResponses reasoning output, but the proposed patch publishes raw thinking onto the global agent-event path before the documented reasoning visibility gate, which leaves a privacy/API-contract blocker before merge.

## What This Changes

The PR adds raw thinking capture from embedded agent runs, returns reasoning items and reasoning SSE events from `/v1/responses`, suppresses the raw stream from server-chat broadcasts, and adds gateway/agent tests.

## Best Possible Solution

Ship a revised OpenResponses reasoning implementation that is explicitly gated by documented reasoning visibility or an authorized HTTP opt-in, keeps raw thinking out of broad plugin/session event streams by default, documents the API shape, adds the required changelog entry, and covers reasoning-off, authorized opt-in, streaming, tool-call ordering, and plugin-subscription exposure cases.

## Reproduction Assessment

Yes. Source inspection is enough: current main only emits formatted `thinking` when `state.streamReasoning` is enabled, while the PR diff emits `thinking-raw` before that visibility check and the HTTP handler/plugin bridge can observe that raw stream.

## Solution Assessment

No. Exposing reasoning on `/v1/responses` is the right product direction, but publishing raw thinking on the global agent-event bus before visibility authorization is not the safest implementation; a scoped internal capture path or explicit gate should come first.
