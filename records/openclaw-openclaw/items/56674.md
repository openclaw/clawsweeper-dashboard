## Summary

Keep open: this is a plausible feature PR, but the current head still exposes raw reasoning on the shared agent-event bus before an explicit visibility boundary and lacks after-fix real behavior proof; the public OpenResponses docs would also be stale if merged as-is.

## What This Changes

The PR adds OpenResponses reasoning output items and streaming reasoning events by emitting raw embedded-agent thinking events and collecting them in the gateway HTTP handler.

## Best Possible Solution

Expose Responses reasoning through a scoped HTTP capture path or explicit reasoning-visibility opt-in that preserves channel/plugin gates, then document and validate the non-stream, streaming, and plugin-isolation behavior.

## Reproduction Assessment

Yes, at source level. Current main has no Responses reasoning output/events, and the PR diff shows the new raw-thinking event capture and emission path that would change the HTTP surface.

## Solution Assessment

No. Returning reasoning on /v1/responses is a reasonable feature direction, but publishing hidden raw thinking onto the shared agent-event bus before an explicit visibility boundary is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate raw reasoning before shared event publishing:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: This emits `thinking-raw` through `emitAgentEvent` before the reasoning visibility gate. Plugin host-hook subscriptions without a `streams` filter receive shared agent events, so hidden raw reasoning can become visible to plugins even though server-chat suppresses websocket and node broadcasts. Keep HTTP capture scoped internally or require an explicit reasoning visibility opt-in before publishing raw thinking on the shared bus.
  - confidence: 0.9
- **[P2] Document the new Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the OpenResponses docs still say `reasoning` is accepted and ignored. Merging this would leave the documented HTTP API contract wrong for clients trying to use or avoid the new reasoning surface.
  - confidence: 0.82
