## Summary

Keep open: current main still lacks Responses reasoning output, but this PR is not ready to merge because the patch exposes reasoning without an explicit visibility gate, publishes raw thinking through a shared event surface reachable by plugin host hooks, leaves public Responses docs contradictory, and has no real behavior proof.

## What This Changes

The PR adds raw embedded-agent reasoning capture and serializes it as `/v1/responses` reasoning output items plus streaming `response.reasoning.*` events, with regression tests around raw dedupe and server-chat suppression.

## Best Possible Solution

Expose reasoning through an explicitly gated HTTP contract, keep raw capture scoped away from shared plugin events, update Responses docs/tests, and coordinate `/v1/chat/completions` on the canonical issue.

## Reproduction Assessment

Yes, at source level. Current main documents Responses `reasoning` as ignored, lacks reasoning stream event types, and serializes assistant text only; I did not run a live reasoning model.

## Solution Assessment

No, not yet. Returning HTTP reasoning is a plausible direction, but the patch needs an explicit visibility gate, a scoped internal capture path, and public docs before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate HTTP reasoning behind an explicit opt-in:** `src/gateway/openresponses-http.ts:815-820`
  - body: This returns collected `thinking-raw` as a reasoning item whenever the model emits it, without checking the request, session `/reasoning`, or agent `reasoningDefault` visibility. A reasoning-capable model can therefore expose hidden thinking through `/v1/responses` by default; gate collection/serialization on an explicit HTTP reasoning policy.
  - confidence: 0.88
- **[P1] Keep raw thinking off shared plugin event delivery:** `src/agents/pi-embedded-subscribe.ts:592-596`
  - body: `thinking-raw` is still published with `emitAgentEvent`. `server-chat` suppresses WebSocket/node forwarding, but plugin host-hook subscriptions without a `streams` filter receive all agent events, so hidden model reasoning can reach plugins; use a scoped HTTP collector or make this stream explicitly opt-in before the plugin bridge.
  - confidence: 0.9
- **[P2] Document the Responses reasoning contract:** `src/gateway/open-responses.schema.ts:351-360`
  - body: This adds public `response.reasoning.*` events and reasoning output items, but the Responses docs still say `reasoning` is ignored and the SSE event list omits the new events. Clients cannot rely on the new output shape, gating, or ordering until the public contract is updated.
  - confidence: 0.84
