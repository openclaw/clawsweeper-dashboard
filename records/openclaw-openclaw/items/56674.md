## Summary

Keep open. Current main still lacks OpenResponses reasoning output, but this PR should not merge as-is because it routes hidden raw thinking through the shared agent-event/plugin subscription surface and is missing required changelog coverage.

## What This Changes

The PR adds raw embedded-agent thinking capture and maps it to OpenResponses reasoning output items and `response.reasoning.*` SSE events, with gateway, server-chat, and embedded-agent tests.

## Best Possible Solution

Expose OpenResponses reasoning through a scoped HTTP/internal capture path or a documented explicit visibility opt-in, then add changelog/docs and tests for reasoning-off, streaming, tool-call ordering, and plugin subscription non-exposure.

## Reproduction Assessment

Yes. Source inspection reproduces the feature gap on current main: `/v1/responses` ignores reasoning fields, has no reasoning SSE schemas, and only serializes assistant/tool output; the PR-introduced privacy issue is also source-reproducible from the raw event publication plus plugin subscription dispatch path.

## Solution Assessment

No. Returning reasoning on the OpenResponses API is a reasonable direction, but publishing hidden raw thinking on the shared agent-event bus is not the narrowest maintainable solution; a scoped capture channel or explicit authorization gate should come first.
