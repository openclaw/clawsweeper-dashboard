## Summary

Keep open, but do not merge as-is. The feature is still a plausible core change, yet this PR currently has blocking gateway reload, strict schema, public type, SDK compatibility, and security-sensitive hot-reload regressions, plus no real behavior proof.

## What This Changes

The PR adds `agents.defaults.sessionLaneMaxConcurrent` and wires it into session lane creation/startup/reload while changing agent-default config types, gateway reload handlers, and the plugin SDK config runtime barrel.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves current reload, secrets/shared-auth, plugin reload, SDK, cron-nested, and timeout behavior, with explicit per-session ordering semantics and real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection for the current product gap and PR-head regressions. Current main still creates session lanes with `maxConcurrent: 1`, while the PR-head blockers are visible in changed source; no live runtime reproduction was run in this read-only pass.

## Solution Assessment

No. Configurable session-lane concurrency is a plausible direction, but this branch is not the best implementation because it mixes the feature with unrelated reload, schema, SDK, cron-lane, and security-sensitive regressions.
