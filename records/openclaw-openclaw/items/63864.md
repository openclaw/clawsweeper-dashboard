## Summary

Keep open. The feature is not implemented on current main, and this PR is not mergeable as-is because it still removes required gateway reload/security contracts, misses the strict config schema, changes public SDK/type surfaces unrelated to the feature, lacks real behavior proof, and needs maintainer judgment on same-session parallelism semantics.

## What This Changes

The PR adds an `agents.defaults.sessionLaneMaxConcurrent` option, wires it into session lane creation/startup/reload, and rewrites gateway reload handlers plus agent-default and SDK config-runtime exports.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves managed reload, plugin reload, SDK exports, cron-nested behavior, and explicitly defines same-session ordering semantics.

## Reproduction Assessment

Yes, by source inspection. Current main hardcodes new lanes to `maxConcurrent: 1`, while the PR's blockers are visible by comparing the diff to current gateway reload, schema, SDK, and queue contracts.

## Solution Assessment

No. Configurable session-lane concurrency may be useful, but this branch is not the best implementation because it removes reload/security contracts, misses schema/docs/tests, and does not establish the supported same-session ordering model.
