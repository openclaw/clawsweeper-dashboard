## Summary

Keep open. The requested configurable session-lane concurrency is not implemented on current main, but this PR still has blocking correctness and security-sensitive reload regressions plus missing real behavior proof before it can merge.

## What This Changes

The PR adds an `agents.defaults.sessionLaneMaxConcurrent` option and wires it through session lane creation, gateway startup/reload handling, config types, and the plugin SDK runtime barrel.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves managed reload, plugin reload, SDK compatibility, cron-nested behavior, and explicitly defines same-session ordering semantics with real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection for the current limitation and the PR blockers. Current main still creates session lanes with `maxConcurrent: 1` and documents one active run per session, while the PR-head missing export/schema/reload issues are visible in source.

## Solution Assessment

No. Configurable session-lane concurrency is a plausible direction, but this branch is not the best implementation because it mixes the feature with stale reload, config type, SDK, and cron-lane regressions and lacks real behavior proof.
