## Summary

Keep open: the feature request is plausible, but the current PR head still has blocking gateway reload, config schema, SDK contract, and cron-lane regressions, plus no real behavior proof for the advertised same-session concurrency behavior.

## What This Changes

The PR adds an `agents.defaults.sessionLaneMaxConcurrent` resolver and wires it into session lane creation/startup/reload while also rewriting gateway reload handling, agent-default types, and the SDK config-runtime barrel.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves managed reload, plugin reload, SDK exports, cron-nested behavior, and explicitly defines same-session ordering semantics.

## Reproduction Assessment

Yes, by source inspection. Current main hardcodes new lanes to `maxConcurrent: 1`, while the PR's merge blockers are visible in the diff and adjacent current-main callers/tests.

## Solution Assessment

No. Configurable session-lane concurrency may be useful, but this branch is not the best implementation because it removes reload/security contracts, misses schema/docs/tests, and does not establish the supported same-session ordering model.
