## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks configurable per-session lane concurrency, so the feature direction remains relevant, but this branch introduces blocking gateway startup/reload, config-schema, SDK export, and public type-surface regressions, plus a security-sensitive reload/auth/secrets regression.

## What This Changes

The PR adds `agents.defaults.sessionLaneMaxConcurrent`, applies it to session lanes at startup/reload, stores the session-lane default in shared queue state, and changes config type, gateway reload, and plugin SDK runtime exports.

## Best Possible Solution

Keep the PR open for revision or replacement. The maintainable path is a narrow implementation that settles the public key name, updates the config type, strict zod schema, generated config metadata/help, docs, and focused queue/reload tests, wires startup and hot reload through shared queue state, and preserves existing gateway reload/auth/secrets, SDK export, and cron-nested lane behavior. Leave the broader auto-yield and lane warning work tracked by #53023 unless implemented separately.

## Reproduction Assessment

Yes. Current main can be inspected directly: session lanes are created with `maxConcurrent: 1`, and embedded agent runs enqueue through `session:<key>` lanes. The PR defects are also statically reproducible from the PR head: a missing ESM export, a strict schema/key mismatch, and a removed SDK export still imported by WhatsApp runtime.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best way to solve it because it mixes the intended queue change with unrelated gateway reload, secrets/auth, SDK, and public type-surface regressions. A narrower patch that preserves existing contracts and adds schema/docs/tests is safer.
