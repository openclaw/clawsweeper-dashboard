## Summary

Keep open: current main still lacks configurable per-session lane concurrency, but this PR remains unsafe to merge because it lacks real behavior proof and introduces blocking gateway reload, config, SDK, and cron-lane regressions.

## What This Changes

The PR adds an `agents.defaults.sessionLaneMaxConcurrent` option and wires it through command queue state, gateway lane setup/reload handling, agent-default types, and the plugin SDK config runtime barrel.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves managed reload, plugin reload, SDK compatibility, cron-nested behavior, and explicitly defines same-session ordering semantics with real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection. Current main still creates session lanes with `maxConcurrent: 1` and documents one active run per session, while the PR-head blockers are visible in the changed source; I did not run a live gateway or Telegram reproduction.

## Solution Assessment

No. Configurable session-lane concurrency is a plausible direction, but this branch is not the best implementation because it mixes the feature with unrelated reload, config type, SDK, cron-lane, and security-sensitive regressions.
