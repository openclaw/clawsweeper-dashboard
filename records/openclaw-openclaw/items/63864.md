## Summary

Keep this PR open, but it is not mergeable as-is: the feature is still potentially useful for related session-starvation reports, while the current branch regresses gateway reload, config schema/type, SDK, and cron-lane behavior and still lacks required real behavior proof.

## What This Changes

The PR adds an `agents.defaults.sessionLaneMaxConcurrent` setting for session lanes and rewrites related queue, gateway reload, agent-default config type, and SDK export surfaces.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves managed/plugin reload, SDK compatibility, cron-nested behavior, and explicitly defines same-session ordering semantics with real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main hardcodes new session lanes to `maxConcurrent: 1` and the docs guarantee one active run per session, while the PR-head blockers are visible in source. I did not run live gateway or Telegram proof in this read-only review.

## Solution Assessment

No. Configurable session-lane concurrency is a plausible direction, but this branch is not the best implementation because it mixes the option with reload, schema, SDK, config type, and cron-lane regressions; a narrow additive change is safer.
