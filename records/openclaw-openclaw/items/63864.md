## Summary

Keep open. Current main still lacks configurable session-lane concurrency, but this PR remains unsafe to merge because it deletes gateway reload handling, omits the strict config schema, breaks public type/SDK surfaces, drops cron-nested concurrency, and still lacks real behavior proof.

## What This Changes

The PR adds `agents.defaults.sessionLaneMaxConcurrent`, wires it into session lane creation and gateway lane setup/reload, and modifies config type and plugin SDK exports.

## Best Possible Solution

Rework this as a narrow additive queue/config change that preserves current reload, shared-auth, SDK, cron-nested, type, schema, docs, and test contracts, then add real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection. Current main still creates session lanes with `maxConcurrent: 1` and has no session-lane resolver in gateway lane setup; the PR-head blockers are visible statically in the diff.

## Solution Assessment

No. Configurable session-lane concurrency is a reasonable direction, but this branch is not the best implementation because it mixes the feature with unrelated gateway reload, SDK, cron, and public config regressions.
