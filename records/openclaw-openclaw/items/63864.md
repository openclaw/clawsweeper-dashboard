## Summary

This PR should stay open for maintainer review: current main still lacks configurable per-session lane concurrency, but the branch still has blocking gateway reload, config schema, SDK, and cron-lane regressions and lacks real behavior proof.

## What This Changes

The branch adds `agents.defaults.sessionLaneMaxConcurrent`, wires it into session lane creation and gateway lane application/reload, and changes config and plugin SDK exports.

## Best Possible Solution

Rework this as a narrow additive config/schema/docs/test change that preserves existing reload, secrets/shared-auth, plugin reload, SDK, and cron-nested behavior, then verify it in a real Telegram or gateway setup.

## Reproduction Assessment

Yes, by source inspection. Current main still hardcodes new lane concurrency to 1 and the PR-head blockers are visible in the diff; no live Telegram or gateway proof was supplied.

## Solution Assessment

No. Configurable session-lane concurrency is plausible, but this branch is not the best implementation because it mixes the feature with unrelated reload, schema, SDK, and cron regressions.
