## Summary

Keep open, but this PR should not merge as-is. Current main still lacks configurable per-session lane concurrency, yet this branch has blocking gateway reload, strict-schema, public config type, SDK export, and cron-lane regressions, plus no after-fix real behavior proof.

## What This Changes

The branch adds `agents.defaults.sessionLaneMaxConcurrent`, wires it into session lane creation/startup/reload, and changes agent-default config types, gateway reload handlers, and the plugin SDK config runtime barrel.

## Best Possible Solution

Rework this as a narrow additive config, schema, docs, and test change that preserves current reload, secrets/shared-auth, plugin reload, SDK, cron-nested, and timeout behavior, with explicit same-session ordering semantics and real Telegram or gateway proof.

## Reproduction Assessment

Yes, by source inspection. Current main still creates session lanes with `maxConcurrent: 1`, and the PR-head blockers are visible in the changed source; no live gateway or Telegram reproduction was run in this read-only pass.

## Solution Assessment

No. Configurable session-lane concurrency is a plausible direction, but this branch is not the best implementation because it mixes the feature with unrelated reload, config type, SDK, cron-lane, and security-sensitive regressions.
