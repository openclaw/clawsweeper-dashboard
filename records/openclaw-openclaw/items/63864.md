## Summary

Keep open. Current main still lacks configurable session-lane concurrency, but this PR head is not mergeable as-is because it breaks gateway reload startup, removes security-sensitive reload handling, leaves the new config key out of the strict schema, and regresses existing public type/SDK surfaces.

## What This Changes

The PR adds `agents.defaults.sessionLaneMaxConcurrent` and wires it into session lane creation, gateway lane startup/reload, config types, and the plugin SDK config-runtime barrel.

## Best Possible Solution

Keep the feature direction but replace or heavily revise the branch as a narrow queue/config change: settle the public key name, update types, strict schema, generated config metadata/help, docs, changelog, and focused queue/reload tests, and preserve current gateway reload/auth/secrets, SDK, and cron-nested lane behavior.

## Reproduction Assessment

Yes. Current main can be statically reproduced by inspecting `getLaneState()` and `runEmbeddedPiAgent()`; the PR defects are also statically reproducible from the PR head through the missing export, missing strict schema key, removed SDK export, and deleted reload security wrapper.

## Solution Assessment

No. Configurable session-lane concurrency is plausible, but this branch is not the best implementation because it mixes the queue feature with unrelated gateway reload/security, SDK, and public type-surface regressions; a smaller patch preserving current contracts is safer.
