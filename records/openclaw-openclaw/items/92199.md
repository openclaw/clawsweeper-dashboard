## Label Justifications

- `P2`: This is a repeatable, bounded channel-output regression affecting long-running task feedback, with a working explicit-message-tool workaround and no runtime crash.
- `impact:message-loss`: Assistant text generated during the ACP turn is silently omitted from the user-visible WeChat result.

## Summary

Current main and the latest release still use invocation-level `done` handling that can emit multiple `final` deliveries within one ACP turn, matching the reported silent loss when the WeChat consumer retains only the last delivery. The issue remains necessary, but implementation should stay consolidated in the open shared-projector fix rather than revive the two closed duplicate PRs; that canonical PR still has a known pending tool/status delivery blocker and lacks live transport proof.

## What This Changes

Preserve all direct-reply text generated between ACP tool calls so WeChat receives the complete turn output instead of only the last segment.

## Best Possible Solution

Keep `final_only` state turn-scoped in the shared ACP projector, preserving text, limits, separators, pending tool summaries, and status output across invocation boundaries, then emit the complete ordered payload exactly once at terminal turn flush and prove it through a real redacted ACP transport run.

## Reproduction Assessment

Yes at source level: current main can produce multiple `final` deliveries inside one ACP turn because invocation `done` handling flushes before the later outer turn flush, and the related runtime scripts demonstrate that shape. A high-confidence live WeChat reproduction is not yet available.

## Solution Assessment

No, the closed issue-specific PR should not be revived, and the open canonical PR is not yet complete because it can still flush pending tool/status output at intermediate boundaries. The narrowest maintainable fix is one shared projector invariant that defers the entire final-only payload until the true turn boundary.
