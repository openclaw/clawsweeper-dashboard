## Summary

Keep open. Current main still has the reported retention/lifecycle ambiguity: `.deleted.*` is the retained transcript suffix, `pruneAfter` still controls both visible-entry pruning and `.deleted.*` hard deletion, and the related open items cover adjacent pieces without superseding this narrower terminology/config split request.

## What This Changes

The issue asks OpenClaw to make session-retention terminology and configuration clearer by separating retained transcript archiving from physical deletion semantics.

## Best Possible Solution

Define an additive, compatibility-preserving session archive lifecycle contract: keep reading legacy suffixes, choose approved non-destructive naming for new retained transcripts, split visible-entry and retained-transcript retention where appropriate, and update schema, docs, cleanup, indexing, and tests together.

## Reproduction Assessment

Yes, for the described current behavior by source inspection. Current main uses `pruneAfterMs` for both stale session entry pruning and `.deleted.*` archive cleanup, and the cron reaper uses one `sessionRetention` value for the same two-stage lifecycle.

## Solution Assessment

Unclear as a direct implementation. The problem is real, but the safest solution is a maintainer-approved lifecycle/config contract rather than only renaming `.deleted.*` or adding one more local retention key.
