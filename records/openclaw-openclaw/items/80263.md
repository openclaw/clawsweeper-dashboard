## Summary

Keep open. Current main and the latest release still use `.deleted.*` archives and reuse `pruneAfter` or `cron.sessionRetention` for both store-entry pruning and retained-transcript cleanup; some docs now clarify subagent and cron behavior, but the requested terminology and config split remain unresolved.

## What This Changes

The issue requests clearer session lifecycle terminology/docs and separate retention knobs for visible session entries versus retained transcript archives.

## Best Possible Solution

Design a compatibility-preserving lifecycle contract: keep reading legacy `.deleted.*`, choose the approved non-destructive archive term or location, add an explicit deleted-archive retention or detach/delete split, and update schema, help, docs, and tests across session maintenance, cron, and subagents.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the reported current semantics: `pruneAfter` detaches stale visible entries and also cleans old `.deleted.*` archives, while cron uses `sessionRetention` for both steps.

## Solution Assessment

Unclear pending maintainer design. The requested direction is reasonable, but the narrowest maintainable fix needs an explicit compatibility, naming, config-default, and destructive-retention policy before implementation.
