## Summary

Keep open. Current main still uses `.deleted` archives and still reuses `session.maintenance.pruneAfter` for both visible-entry pruning and retained `.deleted` archive cleanup; related archive-lifecycle issues overlap, but they do not fully cover this issue’s requested config/API split and terminology decision.

## What This Changes

The issue requests clearer session-retention terminology, a non-destructive archive suffix, separate visible-entry and retained-transcript retention knobs, and lifecycle docs for session, cron, and subagent cleanup.

## Best Possible Solution

Define a compatibility-preserving session archive lifecycle: keep reading legacy suffixes, choose explicit retained-transcript retention semantics for `.deleted` archives, then update config schema/help, docs, cleanup code, and tests together.

## Reproduction Assessment

Yes, for the current behavior. Source and focused tests show stale entries are pruned with `pruneAfterMs`, transcripts are renamed to `.deleted.*`, and old `.deleted.*` archives are later removed using the same retention window.

## Solution Assessment

Unclear as a direct implementation. The requested direction is reasonable, but the durable fix needs a maintainer-approved compatibility and retention-policy decision rather than a narrow automated bug patch.
