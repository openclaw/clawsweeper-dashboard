## Summary

Keep open. Current main still has the reported retention ambiguity: `.deleted.*` is the retained archive suffix, `session.maintenance.pruneAfter` both removes stale visible store entries and later hard-deletes deleted archives, and the requested split requires a retention/config product decision rather than a narrow bug fix.

## What This Changes

Requests clearer session archive lifecycle terminology, a split between visible-entry and retained-transcript retention settings, and docs for subagent and cron retention behavior.

## Best Possible Solution

Define a compatibility-preserving session archive lifecycle contract with explicit names and separate visible-entry versus retained-transcript retention, then align config schema, help text, docs, cleanup behavior, and tests around that contract.

## Reproduction Assessment

Yes, at source level. Current main shows `session.maintenance.pruneAfter` pruning visible store entries and later removing `.deleted.*` archives, with cron `sessionRetention` following the same one-window pattern.

## Solution Assessment

Unclear as a direct implementation. A compatibility-preserving lifecycle/config design is safer than a rename-only or one-off docs patch, and maintainers need to choose the retention semantics first.
