## Summary

Keep open: current main still uses the described `.deleted.*` archive naming and shared retention windows, and changing suffixes or splitting retention knobs is a user-facing config and compatibility decision rather than an auto-closeable bug fix.

## What This Changes

Clarify session retention terminology and configuration so retained transcript archives are not described as deleted and visible-entry retention is not conflated with physical removal retention.

## Best Possible Solution

Design an additive retention lifecycle contract that preserves recognition of existing `.deleted.*` archives, introduces clearer retained-transcript terminology if maintainers want it, and updates config schema, docs, CLI behavior, and tests together.

## Reproduction Assessment

Not applicable as a bug reproduction because the report explicitly describes intentional but confusing semantics. Source and tests do confirm the current behavior: visible entries are pruned, transcripts are renamed to `.deleted.*`, and later cleanup reuses the same retention window.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The requested direction is plausible, but the safest implementation would be additive terminology and new knobs while continuing to recognize existing `.deleted.*` archives.
