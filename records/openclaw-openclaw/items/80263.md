## Summary

Keep open: current main still has the reported retention terminology and double-duty retention behavior, and the reporter's follow-up highlights a distinct public config/API semantics problem that is not fully owned by the adjacent archive-renaming or retention-cleanup issues. This needs maintainer product direction rather than cleanup closure.

## What This Changes

Request clearer session-retention lifecycle semantics by renaming `.deleted.*`, splitting visible-entry and retained-transcript retention knobs, and documenting cron/subagent archive behavior.

## Best Possible Solution

Define an additive retention lifecycle contract that preserves legacy archive compatibility, separates visible-session and retained-transcript retention if maintainers approve, and updates schema, help, docs, cleanup behavior, and regression tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config semantics request. Source, docs, tests, and the shipped release confirm the described current behavior exists.

## Solution Assessment

Unclear until maintainers choose the public compatibility policy. The requested direction is plausible, but the best implementation should be additive and continue recognizing existing `.deleted.*` archives.
