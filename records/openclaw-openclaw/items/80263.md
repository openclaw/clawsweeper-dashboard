## Summary

Keep open: current main and v2026.6.10 still use `.deleted.*` for retained transcripts and still use `pruneAfter` both to detach visible session entries and to physically remove deleted archives. The remaining work is a real core session-lifecycle/API decision, not stale cleanup.

## What This Changes

Clarify core session-retention lifecycle semantics by renaming retained transcript archive states and splitting or documenting retention controls for visible entries, retained archives, cron sessions, and subagent cleanup.

## Best Possible Solution

Define one maintainer-approved session archive lifecycle contract, then update schema, help text, docs, tests, and migration/compatibility behavior so visible-entry retention and retained-transcript deletion are explicit.

## Reproduction Assessment

Yes, at source level: current main still creates `.jsonl.deleted.*` archives and still wires deleted archive cleanup to the same `pruneAfter` window used for visible session entry pruning. I did not run a live Gateway flow because the code and tests already prove the current semantics.

## Solution Assessment

Unclear as a concrete implementation: the direction is valid, but exact suffix names, retention-key names, and migration/compatibility behavior need maintainer design because they touch shipped archive filenames and public config semantics.
