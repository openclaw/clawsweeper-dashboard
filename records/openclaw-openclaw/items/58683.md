## Summary

Keep open: this is still the canonical implementation path for heartbeat daypart scheduling, but the PR is not merge-ready because it lacks contributor real-behavior proof and still has source-level scheduler defects against the current heartbeat model.

## What This Changes

The PR adds optional `heartbeat.schedule[]` time-of-day interval blocks across heartbeat config schema/types/help/docs, runner scheduling logic, and focused tests.

## Best Possible Solution

Land a rebased core heartbeat scheduling implementation that composes with the phase scheduler, preserves fallback and active-hours semantics, and covers fallback gaps plus DST boundaries in tests and docs.

## Reproduction Assessment

Yes for the PR-introduced scheduler defects by source inspection: uncovered schedule gaps return `null`, and boundary delays use fixed minute-of-day arithmetic. The requested feature itself is not reproducible on current main because current main does not expose `heartbeat.schedule`.

## Solution Assessment

No. The requested capability belongs in core heartbeat scheduling, but this patch needs to be rebased onto the current phase scheduler and corrected for fallback gaps and DST boundary wakes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recompute due time when schedule falls back to every:** `src/infra/heartbeat-runner.ts:1152`
  - body: When a boundary enters an uncovered schedule gap, `resolveScheduleIntervalMs(...)` returns `null`, so this guard leaves `nextDueMs` based on the previous window. A `2h` overnight window ending at `08:00` with fallback `every: "30m"` can wait until the stale `2h` due time instead of honoring the fallback; recompute with `scheduleMs ?? agent.intervalMs` before the due check.
  - confidence: 0.92
- **[P2] Use timezone-aware instants for boundary wakes:** `src/infra/heartbeat-active-hours.ts:190`
  - body: `resolveNextWindowBoundaryMs` derives delays from fixed `24 * 60` minute arithmetic, which is wrong on DST transition days. For example, in `America/New_York`, a spring-forward boundary from `01:30` to `03:00` is 30 real minutes but this computes 90, so interval transitions can wake about an hour late; compute the next local boundary as a real instant or narrow the documented guarantee and tests.
  - confidence: 0.78
