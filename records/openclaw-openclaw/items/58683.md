## Summary

This PR should stay open because current main still lacks daypart heartbeat scheduling, but the proposed patch is not ready to merge due to missing real behavior proof and scheduler correctness issues in the PR head.

## What This Changes

The PR adds optional `heartbeat.schedule[]` time-of-day interval blocks across heartbeat docs, config schema/types/help metadata, runner scheduling logic, and tests.

## Best Possible Solution

Land a rebased core heartbeat scheduler extension that preserves the current phase-aligned runner contract, handles schedule gaps and timezone boundaries deliberately, documents the new config, and includes redacted real gateway proof.

## Solution Assessment

No. Daypart heartbeat intervals plausibly belong in core, but this patch needs real behavior proof plus fallback-gap and timezone-boundary fixes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor fallback intervals after schedule gaps:** `src/infra/heartbeat-runner.ts:1152-1155`
  - body: When a boundary moves from a scheduled window into an uncovered gap, `resolveScheduleIntervalMs(...)` returns `null`, so this block leaves `nextDueMs` from the prior window. Recalculate against `scheduleMs ?? agent.intervalMs` so the documented fallback cadence takes effect.
  - confidence: 0.92
- **[P2] Compute boundary delays from timezone instants:** `src/infra/heartbeat-active-hours.ts:190`
  - body: `resolveNextWindowBoundaryMs` uses fixed `24 * 60` minute arithmetic, so DST transition days can schedule a local boundary roughly an hour early or late. Compute the next local boundary as an epoch instant or narrow the documented guarantee and tests.
  - confidence: 0.8
