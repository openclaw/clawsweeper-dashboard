## Summary

Keep open. Current `main` still lacks the requested `heartbeat.schedule[]` config surface, and this PR is the canonical implementation thread for the closed daypart heartbeat issues, but the branch is conflicting with the newer phase-based scheduler and still has source-reproducible scheduler defects before merge.

## What This Changes

The PR adds optional `heartbeat.schedule[]` time-of-day heartbeat intervals with schema/help/docs updates, scheduler integration, and focused tests.

## Best Possible Solution

Land a rebased core heartbeat implementation that integrates daypart intervals with the current phase scheduler, preserves active-hours gating and fallback semantics, and includes schema/docs/generated-config coverage plus scheduler tests for fallback gaps and DST boundaries.

## Reproduction Assessment

Yes for the PR-introduced bug by source inspection: configure a scheduled window that ends into an uncovered period with fallback `every`, and the tick-time branch skips due-time recalculation when the schedule resolver returns `null`. Reproducing the feature on current `main` is not applicable because current `main` does not expose the config field.

## Solution Assessment

No. The feature belongs in core heartbeat scheduling, but this patch needs a rebase onto the phase scheduler and a corrected effective-interval calculation before it is the narrow maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recompute due time when schedule falls back to `every`:** `src/infra/heartbeat-runner.ts:1153`
  - body: When a boundary enters an uncovered period, `resolveScheduleIntervalMs(...)` returns `null`, so this branch keeps the previous window's `nextDueMs`. That can delay fallback `every` heartbeats; recompute with the effective interval (`scheduleMs ?? agent.intervalMs`) before deciding whether the agent is due.
  - confidence: 0.92
- **[P2] Compute boundary delays from timezone-aware instants:** `src/infra/heartbeat-active-hours.ts:191`
  - body: `resolveNextWindowBoundaryMs` uses fixed 24-hour minute-of-day arithmetic, so DST transition days can schedule a boundary wake about an hour early or late while the docs promise minute-level transitions. Compute the next local boundary as a real instant in the configured timezone, or narrow the documented behavior and tests accordingly.
  - confidence: 0.78
