## Summary

Keep open: current main still lacks daypart heartbeat scheduling, and this PR remains the canonical implementation, but the branch is not mergeable as-is because its runner integration misses fallback gaps, its boundary delay is not timezone-instant-aware, and external real behavior proof is still missing.

## What This Changes

The branch adds an optional `heartbeat.schedule[]` config with schema/help/docs updates, scheduler integration, and focused tests for time-of-day heartbeat intervals.

## Best Possible Solution

Land a rebased core heartbeat implementation that extends the current phase scheduler with daypart intervals, preserves active-hours gating and fallback semantics, and adds schema/docs/generated-config coverage plus scheduler tests for fallback gaps and DST boundaries.

## Reproduction Assessment

Yes for the PR-introduced fallback-gap bug by source inspection: configure a scheduled window that ends into an uncovered period with fallback `every`, and the tick-time branch skips due-time recalculation when the schedule resolver returns `null`. Reproducing the feature on current main is not applicable because current main does not expose the config field.

## Solution Assessment

No. The feature direction belongs in core heartbeat scheduling, but this patch needs to be rebased onto the current phase scheduler and corrected for fallback-gap and timezone-boundary behavior before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recompute due time when schedule falls back to `every`:** `src/infra/heartbeat-runner.ts:1152-1155`
  - body: When a boundary moves from a scheduled window into an uncovered period, `resolveScheduleIntervalMs(...)` returns `null`, so this guard does not recalculate `nextDueMs` from the fallback `every`. An overnight `2h` window ending at `08:00` with `every: "30m"` can keep waiting until the old 2h due time instead of switching to the fallback cadence.
  - confidence: 0.92
- **[P2] Compute boundary delay with timezone-aware instants:** `src/infra/heartbeat-active-hours.ts:189-201`
  - body: This computes the next boundary delay with minute-of-day arithmetic and a fixed 24h day, so DST transition days can wake about an hour early or late in timezones like `America/New_York`. Build the boundary as a real instant in the configured timezone, or document and test weaker semantics.
  - confidence: 0.78
