## Summary

Keep open: current main still lacks the requested heartbeat daypart scheduling surface, and the related feature issues were closed in favor of this PR, but the branch is not merge-ready because it is conflicting, lacks contributor real behavior proof, and still has source-level scheduler defects.

## What This Changes

The PR adds optional `heartbeat.schedule[]` time-of-day interval blocks across heartbeat config schema/types/help/docs, runner scheduling logic, and focused tests.

## Best Possible Solution

Implement daypart heartbeat intervals as a first-class core scheduler extension that composes with phase alignment and active-hours gating, with schema/docs/tests plus real gateway proof.

## Reproduction Assessment

Yes for the PR-introduced scheduler defects by source inspection: the runner skips recalculation when the schedule resolver returns `null`, and boundary wake delays use fixed minute-of-day arithmetic. The requested feature itself is not reproducible on current main because current main has no `heartbeat.schedule` schema or type surface.

## Solution Assessment

No. The requested capability likely belongs in core heartbeat scheduling, but this patch needs a rebase onto the current phase scheduler plus fixes for fallback gaps and DST boundary wakes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recompute due time when schedule falls back to every:** `src/infra/heartbeat-runner.ts:1152-1156`
  - body: When a boundary moves into a gap, `resolveScheduleIntervalMs(...)` returns `null`, so this block does not recompute from the fallback base interval. A `2h` overnight window ending at `08:00` with fallback `every: "30m"` can keep the stale due time instead of running around the fallback cadence; recompute from `scheduleMs ?? agent.intervalMs` before the due check.
  - confidence: 0.92
- **[P2] Compute boundary delay with timezone-aware instants:** `src/infra/heartbeat-active-hours.ts:190`
  - body: `resolveNextWindowBoundaryMs` derives delays from fixed `24 * 60` minute-of-day arithmetic. On DST transition days, a local boundary can be about an hour closer or farther in real time, so interval transitions can wake late or early; compute the next local boundary as a real instant or narrow the documented guarantee and tests.
  - confidence: 0.78
