## Summary

Keep open: current main still lacks daypart heartbeat scheduling, but this external feature PR is not merge-ready because its runner integration is stale against the phase scheduler, has source-level transition bugs, and lacks real behavior proof.

## What This Changes

The PR adds optional `heartbeat.schedule[]` daypart interval config across schema/types/help/docs, heartbeat runner scheduling logic, and focused tests.

## Best Possible Solution

Land a rebased core heartbeat scheduling implementation that composes with the phase scheduler, preserves fallback and active-hours semantics, and covers fallback gaps plus DST boundaries in tests/docs.

## Reproduction Assessment

Yes for the PR-introduced fallback-gap bug by source inspection: a scheduled window ending into an uncovered period makes `resolveScheduleIntervalMs` return `null`, so the tick-time guard skips fallback `every` recomputation. The requested feature itself is not reproducible on current main because current main does not expose `heartbeat.schedule`.

## Solution Assessment

No. The requested capability belongs in core heartbeat scheduling, but this patch needs to be rebased onto the current phase scheduler and corrected for fallback gaps and DST boundary wakeups before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recompute due time when schedule falls back to every:** `src/infra/heartbeat-runner.ts:1152`
  - body: When a boundary enters an uncovered period, `resolveScheduleIntervalMs(...)` returns `null`, so this guard skips recalculating `nextDueMs` from the fallback base interval. Recompute with the effective interval (`scheduleMs ?? agent.intervalMs`) before deciding whether the agent is due.
  - confidence: 0.92
- **[P2] Use timezone-aware instants for boundary wakes:** `src/infra/heartbeat-active-hours.ts:190`
  - body: `resolveNextWindowBoundaryMs` derives delays from fixed 24-hour minute-of-day arithmetic, which is wrong on DST transition days. Compute the next local boundary as a real instant in the configured timezone, or narrow the documented transition guarantee and tests accordingly.
  - confidence: 0.78
