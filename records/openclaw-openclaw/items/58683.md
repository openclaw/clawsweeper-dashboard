## Summary

Keep open: this is a useful core heartbeat feature, but the PR is not merge-ready because current main still lacks `heartbeat.schedule[]`, the branch has schedule fallback and DST boundary issues, and external real behavior proof is missing.

## What This Changes

Adds optional `heartbeat.schedule[]` daypart interval config with schema/help/docs updates, runner boundary scheduling logic, and focused tests.

## Best Possible Solution

Land a rebased core heartbeat scheduling implementation that preserves phase scheduling, handles fallback gaps and DST boundaries, and documents the final `heartbeat.schedule[]` semantics with focused tests.

## Reproduction Assessment

Yes for the PR-introduced fallback-gap bug by source inspection: an uncovered schedule period returns `null`, so the tick-time guard skips fallback `every` recomputation. The feature itself is not reproducible on current main because current main does not expose `heartbeat.schedule[]`.

## Solution Assessment

No. Core heartbeat scheduling is the right place, but this patch needs fallback recomputation, timezone-aware boundary handling or narrowed docs, and a rebase onto current phase scheduling before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recompute due time when schedule falls back to every:** `src/infra/heartbeat-runner.ts:1152`
  - body: When a schedule window ends into an uncovered period, `resolveScheduleIntervalMs(...)` returns `null`, so this guard leaves `nextDueMs` on the old window interval instead of recalculating from fallback `every`. That can delay a fallback heartbeat by the previous window's longer cadence.
  - confidence: 0.92
- **[P2] Use timezone-aware instants for boundary wakes:** `src/infra/heartbeat-active-hours.ts:190`
  - body: This fixed `24 * 60` minute arithmetic is wrong on DST transition days, so a boundary in a configured timezone can wake about an hour early or late while the docs promise minute-level transitions. Compute the next local boundary as a real instant, or narrow the documented guarantee and tests.
  - confidence: 0.78
