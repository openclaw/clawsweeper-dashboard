## Best Possible Solution

## Reproduction Assessment

Yes for the blocking PR bug: configure an overnight schedule with `every: "2h"` ending into an uncovered period with fallback `every: "30m"`; at the boundary, `resolveScheduleIntervalMs(...)` returns `null`, so the PR's tick-time branch keeps the prior long `nextDueMs`. Current-main reproduction of the feature itself is not applicable because the config field is absent.

## Solution Assessment

No: the feature direction belongs in core heartbeat scheduling, but this patch is not yet the best implementation because fallback-gap transitions are wrong and the branch is stale against the current phase scheduler. The safer repair is to compute an effective interval as `resolveScheduleIntervalMs(...) ?? agent.intervalMs` anywhere due time is recalculated, then rebase and validate.
