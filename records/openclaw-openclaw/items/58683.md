## Summary

Keep this PR open. Current main still does not implement time-of-day heartbeat schedules, and the PR is directionally useful, but the current head is dirty against main and still has blocking scheduler behavior around fallback gaps plus required release-note/documentation cleanup before merge.

## What This Changes

The PR adds `heartbeat.schedule[]` config, schema/help/docs, schedule resolution helpers, runner integration, and tests for time-of-day heartbeat intervals.

## Best Possible Solution

Land a rebased core heartbeat implementation that keeps the current phase-based scheduler, adds `heartbeat.schedule[]` as a validated config/docs surface, computes due times from the effective interval including fallback gaps, handles or accurately documents DST boundary behavior, and ships focused scheduler/schema/docs coverage plus a changelog entry.

## Reproduction Assessment

Yes for the blocking PR bugs: configure a long scheduled window that ends into an uncovered gap with a shorter fallback `every`, then the PR's tick-time recalculation leaves `nextDueMs` based on the old window because `resolveScheduleIntervalMs(...)` returns `null`. Current-main reproduction of the feature itself is not applicable because `heartbeat.schedule` is absent.

## Solution Assessment

No: the feature belongs in core heartbeat behavior, but this patch is not yet the best implementation because it is stale against current scheduler code and mishandles fallback-gap and DST boundary transitions. The safer path is to rebase onto the phase scheduler and compute an `effectiveInterval = scheduleMs ?? agent.intervalMs` everywhere due time is recalculated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recompute due time when falling back to every:** `src/infra/heartbeat-runner.ts:1151-1153`
  - body: When a schedule boundary moves into an uncovered gap, `resolveScheduleIntervalMs(...)` returns `null`, so this guard skips recalculating `nextDueMs` from the fallback `every` interval. A previous long window can delay fallback heartbeats far past the documented cadence; recompute with `scheduleMs ?? agent.intervalMs` instead.
  - confidence: 0.93
- **[P2] Add the required changelog entry:** `docs/gateway/heartbeat.md:185`
  - body: This adds a user-facing heartbeat config feature and docs, but the PR does not update `CHANGELOG.md`. Repo policy requires a single-line entry under the active version with attribution for user-facing `feat` changes.
  - confidence: 0.88
- **[P2] Compute DST-aware boundary delays:** `src/infra/heartbeat-active-hours.ts:189-191`
  - body: `resolveNextWindowBoundaryMs` derives the next boundary from minute-of-day arithmetic with a fixed 24-hour day. On DST transition days this can wake about an hour early or late, so daypart interval changes miss the intended local boundary; compute the next local boundary as a real timezone-aware instant or explicitly narrow the documented guarantee.
  - confidence: 0.82
