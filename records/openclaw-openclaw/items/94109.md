## Summary

Keep open, but this PR is not merge-ready: the performance goal is useful, but the patch skips the cleanup that currently protects CI/package builds from stale outputs before tsdown runs with `--no-clean`.

## What This Changes

The PR wraps the tsdown wrapper’s four pre-build cleanup calls in a `process.env.CI !== "true"` guard so CI runs skip them.

## Best Possible Solution

Keep CI stale-output cleanup before `--no-clean`, then optimize only a measured safe cleanup sub-step or runner path with focused tests and Windows/Testbox proof.

## Reproduction Assessment

Yes. Source inspection of PR head shows `CI=true` bypasses all four cleanup calls before the wrapper invokes tsdown with `--no-clean`; the pasted Windows snippets support the timing claim but not artifact safety.

## Solution Assessment

No. This is a plausible performance shortcut, but the best fix keeps the stale-output cleanup contract and targets only a proven redundant slow path.
