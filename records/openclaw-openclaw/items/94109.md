## Summary

Keep open: the PR is focused, but it is not safe to merge as written because it skips the wrapper cleanup in every `CI=true` build while the wrapper still runs tsdown with `--no-clean`, and the submitted proof does not demonstrate Windows CI timing or stale-output safety.

## What This Changes

The PR wraps four pre-tsdown cleanup calls in `scripts/tsdown-build.mjs` with a `process.env.CI !== "true"` guard so CI runs skip them before invoking tsdown.

## Best Possible Solution

Keep the stale-output cleanup invariant in CI and optimize the expensive Windows path narrowly, such as timing or splitting cleanup steps and skipping only a proven redundant step with CI proof.

## Reproduction Assessment

Yes from source inspection: with the patch, `CI=true` bypasses all four cleanup calls, while the wrapper still constructs tsdown args with `--no-clean`. I did not run the mutating build command because this review is read-only.

## Solution Assessment

No. This is a plausible performance shortcut, but the safer fix is to preserve the stale-output cleanup contract and target only the expensive redundant cleanup path with Windows CI proof.
