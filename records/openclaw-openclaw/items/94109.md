## Label Justifications

- `P2`: This is a normal-priority infrastructure performance PR with limited direct user impact but a concrete CI artifact-safety risk.
- `merge-risk: 🚨 automation`: The diff can make CI/Testbox artifact builds run over restored stale outputs because wrapper cleanup is skipped while tsdown still uses `--no-clean`.

## Summary

Keep open: the performance idea is understandable, but the current patch skips the whole CI cleanup boundary while tsdown still runs with `--no-clean`, so restored build artifacts can survive into CI/Testbox outputs.

## What This Changes

The branch wraps the four pre-tsdown cleanup calls in `scripts/tsdown-build.mjs` so they are skipped when `process.env.CI === "true"`.

## Best Possible Solution

Keep stale-output cleanup in CI, then optimize only a measured slow sub-step with focused tests and Windows/Testbox proof for restored cache paths.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the merge risk: with `CI=true`, the new guard bypasses all four cleanup calls, while CI/Testbox workflows can restore `dist/` and `dist-runtime/` before `pnpm build:ci-artifacts`.

## Solution Assessment

No. This is a plausible performance shortcut, but it is too broad because it removes the stale-output safety boundary; the safer fix keeps cleanup and targets only the proven redundant or slow step.
