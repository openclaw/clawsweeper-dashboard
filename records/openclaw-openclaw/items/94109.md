## Summary

Keep this PR open, but not merge-ready: the Windows timing improvement is plausible, yet the patch skips every wrapper cleanup step while the wrapper still invokes tsdown with `--no-clean`, leaving stale restored build-output risk unproven.

## What This Changes

The PR guards four pre-build cleanup calls in `scripts/tsdown-build.mjs` with `process.env.CI !== "true"` so CI runs skip them before invoking tsdown.

## Best Possible Solution

Keep CI stale-output cleanup before `--no-clean`, then optimize only a measured safe slow sub-step with focused tests and Windows/Testbox proof.

## Reproduction Assessment

Yes. Source inspection of PR head shows `CI=true` bypasses all four cleanup calls before the wrapper invokes tsdown with `--no-clean`; the pasted Windows snippets support the timing claim but not stale-output safety.

## Solution Assessment

No. This is a plausible performance shortcut, but the best fix keeps the stale-output cleanup contract and targets only a proven redundant slow path.
