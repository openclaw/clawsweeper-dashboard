## Summary

Keep open: this PR is not obsolete, but it is not merge-ready because the CI guard bypasses the current cleanup-before-`--no-clean` safety boundary while CI artifact jobs can restore stale output directories.

## What This Changes

The branch wraps the four tsdown pre-build cleanup calls so they run only when `process.env.CI !== "true"`.

## Best Possible Solution

Keep stale-output cleanup in CI, then optimize only a measured slow sub-step with focused tests and Windows/Testbox proof.

## Reproduction Assessment

Yes. Source inspection shows the proposed `CI=true` branch bypasses all four cleanup calls before a `--no-clean` tsdown invocation; current CI workflows can restore output directories before running that build path.

## Solution Assessment

No. This is a plausible performance shortcut, but it is too broad because it removes the stale-output safety boundary; the safer fix keeps cleanup and targets only the proven redundant or slow step.
