## Summary

Keep this PR open but blocked: the Windows CI speedup is plausible, but the patch skips every tsdown wrapper cleanup step before a `--no-clean` build while CI artifact paths can restore `dist/` and `dist-runtime/`, so stale-output safety is not established.

## What This Changes

The PR wraps the four pre-tsdown cleanup helpers in `scripts/tsdown-build.mjs` with a `process.env.CI !== "true"` guard so CI builds skip them.

## Best Possible Solution

Keep stale-output cleanup before CI `--no-clean` tsdown builds, then optimize only a measured cleanup sub-step or runner path with focused tests and Windows/Testbox proof.

## Reproduction Assessment

Yes for the proposed behavior: source inspection of the PR head shows `CI=true` bypasses all four cleanup calls before the wrapper invokes tsdown with `--no-clean`. The pasted Windows snippets support the timing claim, but not artifact safety.

## Solution Assessment

No. This is a plausible performance shortcut, but the best fix keeps the stale-output cleanup contract and targets only a proven redundant slow path.
