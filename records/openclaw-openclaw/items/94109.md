## Summary

Keep this PR open, but it is not merge-ready: the diff skips every tsdown wrapper cleanup step in CI while the wrapper still runs tsdown with `--no-clean`, and current CI artifact paths can restore old build outputs before `pnpm build:ci-artifacts`. The pasted Windows logs are useful timing evidence, but they do not prove stale-output safety.

## What This Changes

The PR guards the four pre-build cleanup calls in `scripts/tsdown-build.mjs` so they run only when `process.env.CI !== "true"`.

## Best Possible Solution

Keep CI stale-output cleanup before `--no-clean`, then optimize only a measured safe cleanup sub-step with focused tests and Windows/Testbox proof.

## Reproduction Assessment

Yes. Source inspection shows `CI=true` bypasses all four cleanup calls before the wrapper invokes tsdown with `--no-clean`; the missing proof is artifact safety after restored stale outputs, not whether the guard takes effect.

## Solution Assessment

No. This is a plausible performance shortcut, but the safer fix keeps the stale-output cleanup contract and targets only a proven redundant slow path.
