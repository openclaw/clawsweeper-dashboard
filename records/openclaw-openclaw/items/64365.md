## Summary

Keep this PR open: current main still lacks outbound Slack `reply_broadcast` support, and the PR is a useful implementation candidate, but the current patch is not merge-ready because it documents unsupported message-tool parameters, wires discovery into an older Slack schema surface, and accepts `replyBroadcast` for upload/media paths that do not honor it.

## What This Changes

The PR adds a Slack `replyBroadcast` parameter across docs, action runtime, action adapter, message dispatch, tool schema, send payload construction, and tests for broadcasted Slack thread replies.

## Best Possible Solution

Implement this as Slack-owned outbound support: expose a discoverable `replyBroadcast` field through the current `describeSlackMessageTool` schema contribution API, map it from message-tool `send` params through Slack dispatch/action/send layers, emit `reply_broadcast: true` only for valid threaded `chat.postMessage` sends and only once for chunked output, document current message-tool parameters, and add focused Slack tests plus a changelog entry while leaving broader mirroring policy to #63904.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current main has no outbound `replyBroadcast`/`reply_broadcast` path, while the PR diff shows the docs/schema/upload defects by following `message` tool dispatch into Slack send and upload branches.

## Solution Assessment

No. The requested direction is valid, but this patch is not the best mergeable form until it moves schema wiring to `message-tool-api.ts`, avoids silently accepting unsupported upload broadcasts, fixes docs to use current message-tool fields, and adds the required changelog coverage.
