## Summary

Keep open: current main has fallback transition state, lifecycle events, WebUI status, and verbose-only chat notices, but it still lacks the requested default-on bidirectional per-session/per-topic notification policy and disable/config surface.

## What This Changes

The issue requests built-in user-visible notifications when model fallback activates and when the primary model recovers, delivered to the affected session/topic by default with an opt-out config.

## Best Possible Solution

Define a product-approved fallback notification contract that reuses the existing session fallback state, emits activation and recovery notices to the affected conversation target, and documents the default/config behavior alongside the recovery design.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main only provides verbose-gated chat notices plus lifecycle/status observability, not default-on fallback/recovery notifications.

## Solution Assessment

Unclear until maintainer product design is done. The existing fallback transition state is a good implementation base, but the requested default chat behavior and config contract need an explicit policy decision before a narrow fix is safe.
