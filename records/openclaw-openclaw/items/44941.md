## Summary

Keep open. Current main has fallback transition tracking, lifecycle events, verbose-mode reply notices, /status visibility, and Web UI indicators, but it still does not provide default-on activation and recovery notifications in the affected chat/session/topic when verbose mode is off.

## What This Changes

Request default-on bidirectional model fallback notifications for fallback activation and primary recovery, scoped to the affected session/topic, with a disable config.

## Best Possible Solution

Define a product-owned fallback notification policy that sends one-time activation and recovery notices to the affected conversation by default, redacts sensitive details, documents throttling/recovery semantics, and exposes stable disable/tuning config.

## Reproduction Assessment

Yes. Source and existing e2e coverage provide a high-confidence path: simulate fallback and recovery with resolvedVerboseLevel set to off, then observe lifecycle events while reply payloads omit both activation and cleared notice text.

## Solution Assessment

No. The existing verbose-only notices and lifecycle events are useful groundwork, but they do not solve the requested default-on affected-conversation behavior; maintainer-owned policy should come before a code change.
