## Summary

Keep open. Current main still has fallback transition state, lifecycle events, verbose-mode reply notices, /status visibility, and Web UI indicators, but it does not provide default-on fallback activation and recovery notifications in the affected chat/session/topic when verbose mode is off.

## What This Changes

Request default-on per-session/per-topic notifications when model fallback activates and when the primary model recovers, with a disable config.

## Best Possible Solution

Define a product-owned fallback notification policy that sends one-time activation and recovery notices to the affected conversation by default, redacts sensitive failure reasons, documents throttling/recovery semantics, and exposes a stable config switch to disable or tune the behavior.

## Reproduction Assessment

Yes. The current e2e fixture simulates fallback and recovery with `resolvedVerboseLevel` set to `off` and asserts lifecycle events are emitted while activation and cleared notice text is absent from reply payloads; I inspected this path read-only rather than executing tests.

## Solution Assessment

No. The existing verbose-only notices are useful groundwork, but they do not solve the requested default-on affected-conversation behavior; the safer next step is maintainer-owned policy before implementation.
