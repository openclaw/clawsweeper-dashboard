## Summary

Keep open. Current main has fallback transition state, lifecycle events, verbose-mode reply notices, /status visibility, and Web UI indicators, but it still does not provide default-on activation and recovery notifications in the affected chat/session/topic when verbose mode is off, and there is no dedicated disable/tuning config for that behavior.

## What This Changes

The issue requests built-in, default-on per-session/per-topic notifications when model fallback activates and when the primary model recovers, plus a config switch to disable it.

## Best Possible Solution

Define a product-owned fallback notification policy that sends one-time activation and recovery notices to the affected conversation by default, redacts or normalizes sensitive failure reasons, documents throttling and recovery semantics, and exposes a stable config switch to disable or tune the behavior.

## Reproduction Assessment

Yes. The current e2e fixture simulates fallback and recovery with resolvedVerboseLevel set to off and asserts lifecycle events are emitted while activation and cleared notice text is absent from reply payloads; I inspected this path read-only rather than executing tests.

## Solution Assessment

Unclear. The requested default-on direction may be right for production reliability, but the exact implementation needs maintainer decisions on wording, redaction, throttling, recovery truth, and config shape before an automated narrow patch is safe.
