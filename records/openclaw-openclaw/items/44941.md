## Summary

Keep open: current main has fallback state tracking, lifecycle events, `/status` visibility, and verbose-mode inline switch/recovery notices, but it still suppresses user-visible fallback and fallback-cleared messages outside verbose mode and has no default notification config. The remaining work overlaps active product/API decisions in https://github.com/openclaw/openclaw/issues/44935, https://github.com/openclaw/openclaw/issues/79163, and the open implementation PRs.

## What This Changes

The issue requests default per-session/per-topic notifications when model fallback activates and when the primary model recovers.

## Best Possible Solution

Define one maintained fallback-transition notification contract covering default/opt-in policy, per-session delivery, dedupe, recovery coupling, and hook/API needs, then land one coherent implementation instead of parallel partial PRs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/policy request. Source and existing tests do give high-confidence proof of the current gap: verbose-off replies suppress both fallback and fallback-cleared user-visible text.

## Solution Assessment

No, current main is only a partial solution because the visible notice path is tied to verbose mode and no default notification policy/config exists. The narrowest maintainable path is a maintainer-approved notification contract, likely coordinated with the existing open PRs and the recovery design issue.
