## Summary

Keep open. The PR is still useful source work because current main still defaults heartbeat runs to the shared session, but the branch is stale and would regress current main's base-session event consumption when omitted `isolatedSession` starts using the `:heartbeat` session by default. The initial Greptile docs/test concerns were addressed in the PR head, but current main has since added heartbeat event-routing safeguards and `config-agents` docs that the branch needs to preserve/update before merge.

## What This Changes

The PR changes heartbeat runtime behavior, config docs/type comments, and regression tests so omitted `isolatedSession` uses a `:heartbeat` session by default while `isolatedSession: false` keeps the shared transcript path.

## Best Possible Solution

Rebase the PR onto current main and apply the default flip only on top of the current heartbeat runner, preserving `consumeInspectedSystemEvents`, isolated heartbeat key canonicalization, stale isolated-key cleanup, untrusted-event classification, and isolated delivery-route suppression; then update the active config-agent docs and add focused coverage for cron/wake event consumption under the new default.

## Reproduction Assessment

Yes for the review findings. PR head `753dfe5` flips omitted `isolatedSession` to the isolated session, while current main shows cron main jobs enqueue events on the base key and the PR head lacks the current success-path event drain that prevents replay.

## Solution Assessment

No as currently staged. The one-line `!== false` condition is the narrow shape for the requested default change, but the branch must be rebased onto current main and preserve newer event-consumption/routing behavior before it is a safe fix.
