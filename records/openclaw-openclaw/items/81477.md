## Summary

Keep this PR open for now: it targets a plausible SecretRef channel-action problem, but the PR lacks real behavior proof and introduces a hot outbound-path clone of the full secrets runtime snapshot. Current main already supplies the normal gateway request context from the resolved runtime config snapshot after secrets activation.

## What This Changes

The branch imports secrets runtime into gateway send handling, prefers the active resolved secrets config in `resolveRequestedChannel`, adds a mocked `message.action` regression test, and adds a changelog note.

## Best Possible Solution

Keep resolved configuration flowing through `GatewayRequestContext`; if a real late-binding gap exists, add a narrow config-only runtime accessor or fix the context supplier instead of cloning full secrets snapshots inside outbound request handling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The PR only adds a mocked unit scenario where the handler context returns raw SecretRefs, while current main's normal gateway startup activates the resolved secrets snapshot and routes `getRuntimeConfig()` through that snapshot.

## Solution Assessment

Is this the best way to solve the issue? No. Pulling the global active secrets snapshot from `send.ts` bypasses the existing request-context config seam and deep-clones the full snapshot on every outbound request; a context-level resolved config or config-only accessor would be narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid cloning the full secrets snapshot on sends:** `src/gateway/server-methods/send.ts:134-136`
  - body: `resolveRequestedChannel` runs for `message.action`, `send`, and `poll`. After gateway startup an active secrets snapshot is normally present, and `getActiveSecretsRuntimeSnapshot()` deep-clones the config, source config, auth stores, and web-tools state before returning, so this adds O(snapshot) work to every outbound request. Keep using the resolved request-context config or add a config-only accessor instead.
