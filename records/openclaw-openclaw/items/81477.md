## Summary

Keep open: the PR has a concrete P2 regression in a shared gateway hot path and still lacks after-fix real behavior proof; current main normally routes resolved secrets through the request context already.

## What This Changes

The PR changes gateway channel resolution to prefer `getActiveSecretsRuntimeSnapshot()?.config`, adds a mocked `message.action` regression test, and adds a changelog entry for SecretRef-backed Discord/Telegram actions.

## Best Possible Solution

Keep resolved configuration flowing through `GatewayRequestContext`; if a real late-binding gap exists, fix the context supplier or add a narrow config-only runtime accessor with real Discord/Telegram proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. Source inspection shows normal gateway startup activates resolved secrets into `getRuntimeConfig()` and `message.action` forwards that config, while the PR only proves a mocked raw-context scenario.

## Solution Assessment

Is this the best way to solve the issue? No. Pulling the global active secrets snapshot in `send.ts` bypasses the request-context seam and deep-clones full runtime state on outbound requests; a context-level fix or config-only accessor is narrower.
