## Summary

Keep open: this recent external PR still has a blocking hot-path design issue and no real behavior proof, while current main already appears to route normal gateway runtime config through the resolved secrets snapshot.

## What This Changes

The PR imports the secrets runtime into gateway send handling, prefers the active secrets snapshot config in channel resolution, adds a mocked message.action regression test, and adds a changelog note.

## Best Possible Solution

Keep resolved configuration flowing through GatewayRequestContext; if a real late-binding gap exists, add a narrow config-only runtime accessor or fix the context supplier with live gateway proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The only failing path shown is a mocked handler context returning raw SecretRefs, while source inspection shows normal gateway startup activates a resolved secrets snapshot that getRuntimeConfig already reads.

## Solution Assessment

Is this the best way to solve the issue? No. The branch bypasses the request-context config seam and deep-clones the full secrets snapshot on the shared send/poll/message.action path; a context-level resolved config or config-only accessor would be narrower.
