## Summary

Keep open: the PR still has a blocking hot-path issue and lacks contributor-supplied real behavior proof, while current main already routes normal gateway message actions through the resolved request-context runtime config.

## What This Changes

The PR changes gateway channel resolution to prefer the active secrets runtime snapshot config, adds a mocked `message.action` regression test, and adds an Unreleased changelog note.

## Best Possible Solution

Keep resolved configuration flowing through `GatewayRequestContext`; if a real late-binding gap exists, add a narrow config-only runtime accessor or fix the context supplier with live Discord/Telegram proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The PR only demonstrates a mocked context returning raw SecretRefs, while current main's gateway startup activates the resolved secrets snapshot and `send.ts` reads `context.getRuntimeConfig()`.

## Solution Assessment

Is this the best way to solve the issue? No. Pulling the global active secrets snapshot from `send.ts` bypasses the request-context config seam and deep-clones full snapshot state for `message.action`, `send`, and `poll`; a context-level fix or config-only accessor would be narrower.
