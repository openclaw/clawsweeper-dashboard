## Summary

Keep open. Current main still matches the reported bug: direct subagent announce delivery calls the Gateway `agent` method with session/channel context but no provider/model override, and the shared ingress model-selection path still selects agent default, stored session override, then explicit run override without consulting `channels.modelByChannel`. The prior ClawSweeper comment reached the same conclusion, and the relevant code has not been fixed on current main.

## What This Changes

Subagent announce delivery should use the parent/requester session channel's `channels.modelByChannel` model when no explicit per-run or stored session model override exists.

## Best Possible Solution

Add channel-model resolution to the shared Gateway/agent ingress model-selection path, using requester session/channel context when there is no explicit per-run override and no stored session model/provider override. Preserve `explicit > stored session > channel config > agent default`, reuse existing model alias and allowlist handling, and add focused regression tests for direct subagent announce plus explicit/stored precedence.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: direct subagent announce enters `agentCommandFromIngress` without provider/model, and that path never calls the channel override resolver before running the model fallback chain.

## Solution Assessment

Yes. Fixing shared ingress model selection is the narrowest maintainable solution because it covers direct announce and other Gateway agent ingress calls consistently without adding announce-specific model injection that could drift from reply/status behavior.
