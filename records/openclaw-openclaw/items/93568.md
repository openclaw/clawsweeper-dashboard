## Summary

Current main and v2026.6.9 still restrict explicit plugin allowlist auto-append to ClickClack, and live GitHub shows an open closing PR for the generic setup fix. This issue is still necessary and should stay open until that PR lands or maintainers choose a narrower prompt/trust design.

## What This Changes

The issue asks channel setup to complete plugin trust/bootstrap for Weixin by adding or prompting for `openclaw-weixin` in `plugins.allow` after explicit selection instead of failing after install.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/93603 so explicit channel/plugin setup handles restrictive `plugins.allow` atomically while preserving denylist and global-disable semantics, then close this issue after merge.

## Reproduction Assessment

Yes from source inspection: current main only auto-allowlists `clickclack` after an allowlist block, while channel setup and onboarding installation route selected plugins through that helper. I did not run a live interactive Weixin flow in this read-only review.

## Solution Assessment

Unclear as a product policy question: the linked PR is the clean central helper fix if maintainers accept explicit selection as the allowlist trust gesture across callers. If that is too broad, the safer solution is to scope auto-allowlisting to channel setup or add an explicit prompt before mutating `plugins.allow`.
