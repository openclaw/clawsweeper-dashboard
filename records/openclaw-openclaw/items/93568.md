## Summary

Current main and the latest release still keep explicit plugin allowlist auto-append limited to ClickClack, while an open linked PR is the active candidate fix for the Weixin setup failure. This issue should stay open until that PR lands, is revised, or maintainers choose a narrower trust/prompt design.

## What This Changes

Issue asks for Weixin channel setup to auto-allowlist an explicitly selected plugin when `plugins.allow` is restrictive, instead of installing it and then failing on plugin activation.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/93603 so explicit channel/plugin setup handles restrictive `plugins.allow` atomically while preserving denylist and global-disable semantics, then close this issue after merge.

## Reproduction Assessment

Yes from source inspection: current main only auto-allowlists `clickclack` after an allowlist block, while channel setup and onboarding plugin installation route selected plugins through that helper. I did not run a live interactive Weixin flow in this read-only review.

## Solution Assessment

Unclear as a product policy question: the linked PR is the clean central helper fix if maintainers accept explicit selection as the allowlist trust gesture across callers. If that is too broad, the safer solution is to scope auto-allowlisting to channel setup or add an explicit prompt before mutating `plugins.allow`.
