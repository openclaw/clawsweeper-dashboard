## Summary

Keep open: current main and v2026.6.9 still keep explicit plugin allowlist auto-append limited to ClickClack, while the open linked PR is the active candidate fix for the Weixin setup failure. The issue should stay open until that PR lands, is revised, or maintainers choose a narrower prompt/trust design.

## What This Changes

The issue asks guided channel setup to complete plugin allowlist bootstrapping for explicitly selected channel plugins such as Weixin instead of installing the plugin and then failing on a restrictive plugins.allow list.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/93603 so explicit channel/plugin setup handles restrictive plugins.allow atomically while preserving denylist and global-disable semantics, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a restrictive plugins.allow list, selecting a non-ClickClack plugin reaches enableExplicitlySelectedPluginInConfig and returns blocked by allowlist instead of appending the selected plugin id.

## Solution Assessment

Unclear as a product policy question. The linked PR is the clean central helper fix if maintainers accept explicit selection as the allowlist trust gesture across callers; if that is too broad, the safer solution is to scope auto-allowlisting to channel setup or add an explicit prompt before mutating plugins.allow.
