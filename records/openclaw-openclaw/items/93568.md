## Summary

Current main and v2026.6.9 still keep the explicit allowlist retry limited to ClickClack, so the Weixin setup failure remains. This issue should stay open because https://github.com/openclaw/openclaw/pull/93603 is still the active linked fix candidate and the remaining decision is the plugin allowlist trust scope.

## What This Changes

The issue reports that guided Weixin channel setup installs the plugin but then fails because a restrictive plugins.allow list is not updated for the explicitly selected plugin.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/93603 so explicit channel/plugin setup handles restrictive plugins.allow while preserving global-disable and denylist semantics, then close this issue after merge.

## Reproduction Assessment

Yes from source inspection: current main only auto-allowlists clickclack after an allowlist block, while channel setup and onboarding plugin installation route selected plugins through that helper. I did not run a live interactive Weixin flow in this read-only review.

## Solution Assessment

Unclear as a product-policy boundary: the linked PR is the clean central helper fix if maintainers accept explicit selection as the plugin allowlist trust gesture. If that is too broad, the safer solution is to scope auto-allowlisting to guided channel setup or add an explicit prompt before mutating plugins.allow.
