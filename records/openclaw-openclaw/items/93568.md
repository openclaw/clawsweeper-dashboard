## Summary

Current main and v2026.6.9 still keep explicit plugin allowlist auto-append limited to ClickClack, while https://github.com/openclaw/openclaw/pull/93603 is the open closing fix candidate for the Weixin setup failure. This issue should stay open until that PR lands, is revised, or maintainers choose a narrower prompt/trust design.

## What This Changes

The issue requests that explicitly selecting Weixin during channel setup also complete the plugin allowlist and enablement bootstrap when `plugins.allow` is restrictive.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/93603 so explicit channel/plugin setup handles restrictive `plugins.allow` atomically while preserving denylist and global-disable semantics, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 only auto-allowlist `clickclack` after an allowlist block, while the Weixin setup/install path uses the same helper for a different plugin id; I did not run a live interactive Weixin setup in this read-only review.

## Solution Assessment

Unclear as a product policy question. The linked PR is the clean central helper fix if maintainers accept explicit selection as the allowlist trust gesture across callers; if that is too broad, the safer solution is to scope auto-allowlisting to channel setup or prompt before mutating `plugins.allow`.
