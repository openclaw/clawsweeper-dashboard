## Summary

Keep open: current main still has a source-level path where cron auto-TTS for QQBot can produce a generated local audio payload, QQBot can return an error result with no delivery identity, and shared media delivery still counts that result as delivered.

## What This Changes

The issue reports that QQBot cron delivery drops messages when `messages.tts.auto` is `always`, while the cron run still shows success/delivered.

## Best Possible Solution

Keep cron auto-TTS working for QQBot, but either stage generated audio through a QQBot-allowed media path or make no-identity QQBot media results fail delivery accounting instead of reporting success.

## Reproduction Assessment

Yes, from source inspection: current main can route cron auto-TTS into QQBot media delivery, accept a QQBot no-identity error result, and still classify the media payload as sent. I did not run a live QQBot cron reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve the existing cron/TTS feature while correcting media staging and delivery accounting. Disabling TTS for cron or adding a new config option would avoid the symptom without fixing the broken success signal.
