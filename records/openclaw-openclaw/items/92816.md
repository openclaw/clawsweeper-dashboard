## What This Changes

The issue reports QQBot cron deliveries losing generated TTS voice messages when `messages.tts.auto` is `always`, while the cron run is shown as delivered or successful.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/92947 or an equivalent scoped QQBot temp-root media trust fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source-reproducible: cron applies auto-TTS, speech-core writes the voice file under OpenClaw's temp root, QQBot still rejects that local root, and shared media delivery can count a no-identity media result as sent. I did not run a live current-main QQBot cron reproduction.

## Solution Assessment

Yes: the best fix is a plugin-local QQBot temp-root trust path that preserves realpath containment and makes the voice actually arrive. An accounting-only fix would make the failure honest but would not satisfy the reporter's expected delivery behavior.
