## Label Justifications

- `P1`: The issue affects real async/subagent workflows by surfacing completion without a usable result to users in UI and Telegram contexts.
- `impact:message-loss`: The reported and source-backed failure mode is a completed task whose usable result payload is missing from the requester handoff.
- `impact:session-state`: The bug involves parent/child session completion state being promoted before the child result is durably available.

## Summary

Keep open. Current main and v2026.5.18 still have a source-backed path where completion-mode subagent announce can send `(no output)` as `task_completion.result` when no child output is available, while the relevant hardening PRs remain open.

## What This Changes

The issue asks OpenClaw to defer async/subagent completion announcements until a usable user-facing result, structured delivery payload, or sanitized fallback payload exists.

## Best Possible Solution

Consolidate and land the subagent completion hardening so completion handoffs persist, wait for, or synthesize a real user-safe payload before announcement, with regression coverage rejecting placeholder-only results.

## Reproduction Assessment

Yes, at source level. Current main can read no child output, continue through retry exhaustion, and still deliver `(no output)` as the completion result; I did not run a live Telegram/OpenRouter repro.

## Solution Assessment

Yes directionally. The requested fail-closed/defer behavior matches the docs contract; the maintainable path is to consolidate the existing completion-delivery hardening rather than add another display-layer filter.
