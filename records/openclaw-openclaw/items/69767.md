## Summary

Keep open. Current main and the latest release still have a source-backed path where completion-mode subagent announce can send `(no output)` as `task_completion.result`, while related hardening PRs remain open and unmerged.

## What This Changes

The issue asks OpenClaw to defer async/subagent completion announcements until a usable result payload, structured payload, or sanitized fallback exists.

## Best Possible Solution

Land or replace the queue-first/result-payload hardening so completion handoffs defer until a real user-safe payload or structured fallback exists, with regression coverage rejecting placeholder-only results.

## Reproduction Assessment

Yes, at source level. Current main has a focused path where child history contains only user text, no latest assistant reply exists, and completion-mode announce still delivers `(no output)` in the requester prompt; I did not run live Telegram/OpenRouter.

## Solution Assessment

No. Current main is not the best solution because it promotes a placeholder as completion data; the maintainable fix is to persist, wait for, or synthesize a real user-safe payload before announcing, likely by consolidating the open work in https://github.com/openclaw/openclaw/pull/68464 and https://github.com/openclaw/openclaw/pull/76927.
