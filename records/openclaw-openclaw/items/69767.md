## Summary

Keep open. Current main still has a code- and test-backed path that reads no child output, substitutes `(no output)`, and delivers it as `task_completion.result`; the broader hardening work at https://github.com/openclaw/openclaw/pull/68464 is still open rather than merged.

## What This Changes

Fix async/subagent completion delivery so requester-facing completion is not announced until a usable result payload or safe fallback is available.

## Best Possible Solution

Land or replace the hardening path that persists a user-safe completion payload and makes no-payload completion fail closed, with regression coverage replacing the current `(no output)` expectation.

## Reproduction Assessment

Yes. Current main has a focused test path where child history contains only user text, delivery still succeeds, and the requester prompt contains `(no output)`; I did not run a live Telegram/OpenRouter reproduction in this read-only review.

## Solution Assessment

No. Current main is not the best solution because it promotes a placeholder as a completion result; the maintainable fix is to defer/retry or persist a real user-safe payload before announcing completion.
