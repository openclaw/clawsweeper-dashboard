## Summary

Keep open. Current main and the latest release still have a source-backed path where subagent completion can proceed with `(no output)` as the `task_completion.result`, while the central hardening work remains unmerged in https://github.com/openclaw/openclaw/pull/68464.

## What This Changes

The issue asks OpenClaw to defer async/subagent completion announcements until a usable result payload or safe fallback is available.

## Best Possible Solution

Land or replace the completion-delivery hardening so the requester receives a real user-safe payload, and update regression coverage to reject placeholder-only completion handoffs.

## Reproduction Assessment

Yes. Current main has a focused source/test path where child history contains only user text, no latest assistant reply exists, and completion-mode announce still delivers `(no output)` in the requester prompt; I did not run live Telegram/OpenRouter.

## Solution Assessment

No. Current main is not the best solution because it promotes a placeholder as completion data; the maintainable fix is to persist, wait for, or synthesize a real user-safe payload before announcing.
