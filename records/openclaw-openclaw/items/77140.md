## Summary

Keep open: the PR targets a real exec-resource safety gap, but the proposed guard has blocking classifier gaps and should not merge as-is.

## What This Changes

The PR adds a Gateway exec preflight that classifies heavy commands, refuses unisolated always-heavy validation runs, allows explicit isolation wrappers, and adds analyzer/blocking tests.

## Best Possible Solution

Keep the exec-entry guard direction, but normalize package-manager argv, parse systemd-run properties only before the child-command separator, add regression coverage, and document the user-facing behavior before merge.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows packageManagerHeavyReason reads args[0], so pnpm --dir ui test is not classified as heavy, and package.json already contains that command shape.

## Solution Assessment

No. An exec-entry guard is a plausible narrow solution, but this implementation needs option parsing, stricter systemd-run property parsing, changelog/docs follow-through, and maintainer approval of the policy defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse package-manager options before classifying:** `src/agents/bash-tools.exec.ts:1207`
  - body: The guard decides the package-manager command from args[0], so any global option before the script or subcommand is treated as non-heavy. The repo already uses pnpm --dir ui test; with this code first is --dir and the function returns null, allowing an unisolated heavy test run. Please skip recognized package-manager options and their values before checking install/lint/build/test script names, and cover --dir, -C, and workspace/filter forms.
  - confidence: 0.9
- **[P2] Stop accepting child argv as systemd properties:** `src/agents/bash-tools.exec.ts:1198-1202`
  - body: isExplicitSystemdMemoryScope scans the full argv for MemoryMax=, including arguments after the systemd-run -- command separator. That means systemd-run --user --scope -- corepack pnpm lint:core MemoryMax=6G is treated as isolated even though no MemoryMax property is applied to the scope. Split systemd-run options from the child command and only accept -p/--property MemoryMax before --.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/agents/bash-tools.exec.ts:1634-1636`
  - body: This changes user-facing exec behavior by refusing commands and introducing OPENCLAW_EXEC_HEAVY_ENFORCEMENT, but the PR does not update CHANGELOG.md. Repo policy requires user-facing fix/feat/perf changes to add an active-version changelog entry before merge.
