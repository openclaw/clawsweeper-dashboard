## Summary

This PR should stay open: it targets a real Gateway exec resource-safety gap, but the current draft still has a systemd-run bypass and lacks contributor-supplied real Gateway behavior proof.

## What This Changes

The PR adds a Gateway exec preflight analyzer that blocks known-heavy validation commands unless an approved isolation wrapper or emergency bypass is present, with tests and a changelog entry.

## Best Possible Solution

Tighten systemd-run parsing so only MemoryMax before the payload separator counts as isolation, classify unapproved systemd-run payloads normally, add regression coverage, then require real Gateway exec proof before merge.

## Reproduction Assessment

Yes at source level: current main has no heavy-command preflight on the documented unsandboxed Gateway exec path, and the linked issue describes concrete production incidents. I did not establish a live OOM or Gateway restart reproduction during this read-only review.

## Solution Assessment

No as currently proposed: an exec-entry guard is a plausible narrow fix, but the systemd-run wrapper parsing needs tightening and the contributor still needs to provide real Gateway behavior proof.
