## Summary

Keep open: the PR targets a real Gateway exec safety gap that current main still lacks, but the current diff still has correctness/security blockers around trusted isolation detection and lacks real Gateway exec proof.

## What This Changes

The PR adds a Gateway exec heavy-command analyzer/refusal, regression tests for known-heavy validation commands and wrappers, and a changelog entry.

## Best Possible Solution

Land an exec-entry guard only after approved isolation wrappers are verified, unapproved systemd-run payloads are classified normally, docs describe the new behavior, and real Gateway exec proof demonstrates both refusal and approved allow paths.

## Reproduction Assessment

Yes at source level: current main has no heavy-command preflight before local Gateway exec proceeds toward host execution. No live reproduction of the full OOM/restart or host-hang incident was established in this review.

## Solution Assessment

No as currently proposed: an exec-entry guard is a plausible direction, but this implementation trusts unverified wrappers and under-parses systemd-run payloads. The safer path is to verify approved isolation before bypassing classification, classify unapproved wrapper payloads normally, and document/prove the actual Gateway behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify the heavy-run wrapper before trusting it:** `src/agents/bash-tools.exec.ts:1187-1191`
  - body: The analyzer treats any executable whose basename is `openclaw-heavy-run` as isolated, including a workspace-local script with that name. That lets an unbounded wrapper run `corepack pnpm lint:core` while bypassing the new guard; require a configured/approved wrapper path or keep classifying the payload unless the wrapper is verified.
  - confidence: 0.86
- **[P2] Classify systemd-run payloads unless MemoryMax is real:** `src/agents/bash-tools.exec.ts:1203-1211`
  - body: `isExplicitSystemdMemoryScope` scans tokens after `--` and `analyzeHeavyExecCommand` does not unwrap unapproved `systemd-run` payloads. As a result, `systemd-run --user --scope -- corepack pnpm lint:core` is treated as light, while a payload arg like `MemoryMax=6G` can make the wrapper appear approved; stop option parsing at `--` and classify the payload unless a pre-payload memory property is present.
  - confidence: 0.9
- **[P3] Document the exec heavy-command guard:** `CHANGELOG.md:155`
  - body: The changelog announces a new refusal/bypass behavior, but `docs/tools/exec.md` still says full-trust host exec has no extra heuristic prefilter. Update the exec docs with known-heavy refusals, approved isolation wrappers, maybe-heavy diagnostics, and the emergency bypass before shipping this behavior.
  - confidence: 0.84
