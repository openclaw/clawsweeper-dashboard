## Summary

Keep open: this is a plausible exec-entry resource guard, and the latest force-push fixed the earlier basename-only wrapper trust issue, but the current patch still has a concrete systemd-run parsing bypass plus missing real Gateway proof and docs/policy follow-through.

## What This Changes

The PR adds a Gateway exec heavy-command analyzer/refusal, regression coverage in the exec preflight tests, and a changelog entry for unisolated validation commands.

## Best Possible Solution

Land an exec resource-policy guard only after systemd-run payload parsing is fail-closed, the behavior is documented, and real Gateway exec proof shows both unisolated refusal and approved isolated execution.

## Reproduction Assessment

Yes at source level for the review finding: `systemd-run` accepts `COMMAND [ARGUMENTS...]` without `--`, while the PR parser treats no-separator invocations as having no payload to reclassify. No live reproduction of the original OOM/hard-hang incident was established in this read-only review.

## Solution Assessment

No as currently proposed: an exec-entry guard is a reasonable direction, but this patch still leaves a systemd-run bypass and lacks real Gateway proof and docs for the new policy. A safer version should parse or reject no-separator systemd-run payloads, document the wrapper/bypass contract, and prove the runtime path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse systemd-run commands without --:** `src/agents/bash-tools.exec.ts:1285-1287`
  - body: `systemd-run` accepts `COMMAND [ARGUMENTS...]` without a `--` separator, but this parser sets `payload` to `[]` whenever `--` is absent. That lets an unapproved wrapper such as `systemd-run --user --scope -p TimeoutStopSec=60 corepack pnpm lint:core` avoid both approval as an isolated MemoryMax scope and reclassification as a heavy payload, so the new guard can still be bypassed.
  - confidence: 0.89
- **[P3] Document the heavy-command guard:** `CHANGELOG.md:430`
  - body: The changelog announces a new exec refusal, wrapper allowlist, and emergency bypass, but the exec docs still describe full/off host exec as having no extra heuristic prefilter. Please add the user-facing docs before shipping the behavior so operators know which commands are refused, how approved wrappers are configured, and when the bypass is acceptable.
  - confidence: 0.84
