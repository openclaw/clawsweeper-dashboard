## Summary

Keep open: the PR targets a real source-reproducible Gateway exec resource-safety gap, but the current draft still has a concrete systemd-run bypass, stale exec docs for the new refusal path, and no real Gateway behavior proof.

## What This Changes

The PR adds a Gateway exec preflight classifier/refusal for known-heavy validation commands, tests for classifier and execution-path behavior, and a changelog entry.

## Best Possible Solution

Tighten systemd-run parsing, update the exec docs for the new guard and bypass semantics, and require after-fix Gateway exec proof before this PR leaves draft.

## Reproduction Assessment

No for the full systemd OOM incident: I did not live-reproduce a Gateway restart or host hang. Yes at source level for the unsafe acceptance path: current main has no heavy-command preflight before gateway exec proceeds toward launching the requested command, and the linked issue provides concrete production incidents.

## Solution Assessment

No as currently proposed: an exec-entry guard is a plausible narrow fix, but the systemd-run wrapper parsing, docs, and real Gateway proof need completion. The safer path is to classify unapproved systemd-run payloads normally and only treat MemoryMax before the payload separator as approved isolation.

## Review Findings

- **[P2] Classify unapproved systemd-run payloads:** `src/agents/bash-tools.exec.ts:1297-1301`

If the guard lands, the docs need to describe known-heavy refusal, approved isolation wrappers, maybe-heavy diagnostics, and the emergency bypass.
