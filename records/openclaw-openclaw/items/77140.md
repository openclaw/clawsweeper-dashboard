## Summary

Keep open: this draft external PR targets a real exec resource-safety gap, but the patch is not merge-ready because the new guard still has concrete bypasses, docs remain inconsistent with the new refusal behavior, and the supplied proof stops at the analyzer rather than a real Gateway exec path.

## What This Changes

The PR adds a Gateway exec preflight analyzer/refusal for known-heavy validation commands, targeted classifier/execution tests, and a changelog entry.

## Best Possible Solution

Land an exec-entry guard only after it verifies approved isolation wrappers, classifies unapproved systemd-run payloads normally, documents the new refusal and bypass semantics, and includes real Gateway exec proof.

## Reproduction Assessment

Yes at source level for the unsafe acceptance path: current main has no heavy-command preflight before local gateway exec proceeds toward host execution. No for the full OOM/restart incident, because this review did not live-reproduce Gateway memory pressure or a host hang.

## Solution Assessment

No as currently proposed: an exec-entry guard is a plausible direction, but this implementation trusts unverified wrappers and under-parses systemd-run payloads. The safer fix is to verify approved isolation before bypassing classification, classify unapproved wrapper payloads normally, and align docs and proof with the actual Gateway behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify openclaw-heavy-run before trusting it:** `src/agents/bash-tools.exec.ts:1187-1191`
  - body: The analyzer treats any executable whose basename is `openclaw-heavy-run` as isolated, including `./openclaw-heavy-run` from the workspace. A command can therefore route `corepack pnpm lint:core` through a local script with that name and bypass the guard without any resource limit. Require a configured/approved wrapper path or keep classifying the payload unless the wrapper is verified.
  - confidence: 0.86
- **[P2] Classify unapproved systemd-run payloads:** `src/agents/bash-tools.exec.ts:1297-1301`
  - body: `systemd-run --user --scope -- corepack pnpm lint:core` is not approved isolation, but the analyzer does not unwrap the payload for normal classification, so it is treated as light. The MemoryMax scan also looks after the payload separator, letting a payload argument such as `MemoryMax=6G` make the wrapper appear approved. Stop parsing wrapper properties at `--` and classify the payload unless a real pre-payload MemoryMax policy is present.
  - confidence: 0.9
- **[P3] Document the heavy-command refusal path:** `src/agents/bash-tools.exec.ts:1674-1675`
  - body: This call adds a new heuristic refusal path for gateway exec, but the exec docs still say full-trust host exec has no extra heuristic prefilter. If this guard lands, the docs need to describe known-heavy refusals, approved isolation wrappers, maybe-heavy diagnostics, and the emergency bypass.
  - confidence: 0.84
