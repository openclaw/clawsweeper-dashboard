## Summary

Keep open: the safety direction is plausible and the changelog gap is fixed, but this draft PR still lacks real Gateway behavior proof and has a systemd-run wrapper gap that can bypass the intended MemoryMax requirement.

## What This Changes

## Best Possible Solution

Land a maintainer-approved exec guard that blocks known-heavy gateway exec unless sandboxed or verifiably resource-limited, with regression coverage, changelog/docs alignment, and real behavior proof; keep https://github.com/openclaw/openclaw/issues/77121 open until an implementation merges.

## Reproduction Assessment

## Solution Assessment

No as currently proposed: the exec-entry guard is a plausible narrow mitigation, but the systemd-run payload handling needs tightening and the contributor still needs real Gateway behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Classify non-limited systemd-run payloads:** `src/agents/bash-tools.exec.ts:1299-1303`
  - body: The guard only treats systemd-run as approved when isExplicitSystemdMemoryScope returns true, but otherwise it falls through without inspecting the command after `--`. Because the shared payload extractor does not unwrap systemd-run, `systemd-run --user --scope -- corepack pnpm lint:core` is reported as non-heavy even though it lacks the MemoryMax limit this PR requires for the allowed systemd path. Please unwrap and classify the payload unless the systemd-run invocation has a real MemoryMax property before the command separator.
  - confidence: 0.86
