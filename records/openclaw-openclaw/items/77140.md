## Summary

Keep open: this PR targets a real unsafe exec path for https://github.com/openclaw/openclaw/issues/77121, but the current patch still has a blocking systemd-run bypass, insufficient real Gateway exec proof, and missing docs for the new refusal behavior.

## What This Changes

The PR adds a Gateway exec preflight classifier/refusal for known-heavy validation commands, targeted tests, and a changelog entry.

## Best Possible Solution

Keep the exec-entry guard direction, but tighten systemd-run payload parsing, document the guard and bypass semantics, and require real Gateway exec proof before merge.

## Reproduction Assessment

No for the full systemd OOM incident because I did not live-reproduce a Gateway restart or host hang. Yes at source level for the unsafe acceptance path: current main has no heavy-command preflight before local gateway exec proceeds toward process launch, and the linked issue provides concrete production incidents.

## Solution Assessment

No as currently proposed: an exec-entry guard is a plausible narrow solution, but the systemd-run parser, docs, and real Gateway proof are incomplete. The safer path is to classify unapproved systemd-run payloads normally and only treat MemoryMax before the payload separator as approved isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Classify unapproved systemd-run payloads:** `src/agents/bash-tools.exec.ts:1297-1301`
  - body: The analyzer treats approved `systemd-run` scopes as safe, but unapproved wrappers are never unwrapped for normal payload classification. Because the helper also scans `MemoryMax=` across the full argv instead of stopping before `--`, `systemd-run --user --scope -- corepack pnpm lint:core` can pass as light and a payload `MemoryMax=` argument can make an unsafe wrapper look approved.
  - confidence: 0.9
- **[P3] Document the new exec refusal path:** `src/agents/bash-tools.exec.ts:1674-1675`
  - body: This call adds a new heuristic refusal path for gateway exec, but `docs/tools/exec.md` still says `security=full` plus `ask=off` has no extra heuristic prefilter. If the guard lands, the docs need to describe known-heavy refusal, approved isolation wrappers, maybe-heavy diagnostics, and the emergency bypass.
  - confidence: 0.84
