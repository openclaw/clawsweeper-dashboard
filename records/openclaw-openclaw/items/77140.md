## Summary

Keep open: the PR targets a real Gateway exec resource-policy gap, but the current diff misses package-manager commands that put global options before the script/subcommand, so existing forms like `pnpm --dir ui test` can bypass the stated heavy-command block.

## What This Changes

The PR adds a Gateway exec preflight that classifies heavy validation commands, refuses unisolated always-heavy local runs, permits configured isolation wrappers, and adds analyzer/blocking tests.

## Best Possible Solution

Normalize package-manager argv by skipping recognized global options and option values before classifying install/lint/build/typecheck/test scripts, add regression tests for `pnpm --dir ui test` and similar workspace/filter forms, then get maintainer approval on policy defaults.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the PR bug: `packageManagerHeavyReason` reads `args[0]`, so `pnpm --dir ui test` starts with `--dir` and is not classified as heavy. The repository already contains that exact option-before-test shape in `package.json`.

## Solution Assessment

No, not yet. An exec-entry guard is a plausible narrow direction, but this implementation needs package-manager option parsing and maintainer approval for the command policy, wrapper, and bypass semantics before merge.

## Review Findings
