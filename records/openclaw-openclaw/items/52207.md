## Summary

Keep open: current main already ships the SearXNG provider, but generic Tavily `web_search` freshness still is not implemented on main, and this live PR remains a mixed, conflicting branch with blocking correctness/security issues and missing real behavior proof.

## What This Changes

The branch adds a bundled SearXNG web-search plugin, generic Tavily `web_search` freshness mapping, docs/config/onboarding/generated metadata, and unrelated Pi embedded runner provider-normalization changes.

## Best Possible Solution

Keep the shipped SearXNG implementation from https://github.com/openclaw/openclaw/pull/57317 and land Tavily freshness as a focused Tavily-plugin change with request/cache coverage, docs, and no Pi runner or superseded SearXNG churn.

## Reproduction Assessment

Yes. Source inspection is enough for this PR review: current main's Tavily generic provider exposes only `query` and `count`, while the PR branch adds freshness handling and also shows the blocking unrelated changes.

## Solution Assessment

No. The mixed branch is not the best way to solve the remaining issue; a narrow Tavily freshness follow-up is more maintainable than merging superseded SearXNG code plus Pi runner changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:31-37`
  - body: The unconditional `isOpenRouter` branch rewrites every OpenRouter `openai-completions` model to `openai-responses`, including custom/proxied `baseUrl` deployments that may only implement Chat Completions. This unrelated transport change can break configurations that explicitly selected the existing API surface.
  - confidence: 0.86
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:322-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config path and still hit missing-key errors at runtime.
  - confidence: 0.82
- **[P2] Prompt SearXNG users for the base URL:** `extensions/searxng/src/searxng-search-provider.ts:55-69`
  - body: This provider advertises and stores an optional SearXNG API key as the onboarding credential, but the search client actually needs `baseUrl` to reach a self-hosted instance. Selecting SearXNG can therefore write the user's setup input to the wrong field and leave the provider pointed at the default localhost URL.
  - confidence: 0.78
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:330-339`
  - body: SearXNG can run with a configured base URL and no API key, but this heuristic only counts SearXNG when an API key or `SEARXNG_API_KEY` exists. A keyless SearXNG configuration can run web search while the audit reports it disabled.
  - confidence: 0.78
