## Summary

Keep open. Current main already shipped the SearXNG provider, but it still does not implement the remaining generic Tavily `web_search` freshness support, and the PR branch is dirty plus mixes that useful Tavily feature with superseded SearXNG code and unrelated provider-runtime changes.

## What This Changes

The PR branch adds a bundled SearXNG web-search plugin, adds Tavily generic `web_search` freshness-to-days support, updates docs/config/onboarding/generated metadata, and changes Pi embedded model-provider normalization code.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317. If Tavily freshness is still desired, land a narrow Tavily-plugin change against current main that accepts OpenClaw's documented freshness values, maps them deliberately to Tavily request/cache behavior, updates focused Tavily/web-search tests and docs, and avoids legacy config paths or provider-normalization changes.

## Reproduction Assessment

Yes. Static inspection against current main and the PR head is enough: current main lacks generic Tavily freshness, while the PR head shows the freshness mapper, unread Tavily config path, unrelated OpenRouter normalization, and SearXNG audit/default issues directly in changed files.

## Solution Assessment

No. This PR is not the best solution because it combines a superseded SearXNG provider, one still-useful Tavily feature, stale config surfaces, and unrelated model transport changes; the safer path is a Tavily-only patch plus focused docs/tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-37`
  - body: The new `isOpenRouter` branch rewrites every OpenRouter model using `openai-completions` to `openai-responses`, even when `baseUrl` points at a custom or proxied endpoint. Those deployments may only implement Chat Completions, so this unrelated change can break configurations that explicitly chose that API surface.
  - confidence: 0.86
- **[P2] Map the documented freshness values:** `extensions/tavily/src/tavily-search-provider.ts:36-42`
  - body: The generic `web_search` docs and existing normalization use `day`, `week`, `month`, and `year`, but this mapper only handles `pd`, `pw`, `pm`, and `py`. A documented call such as `freshness: "week"` would validate at the tool surface but send no Tavily recency filter.
  - confidence: 0.83
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:322-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is designed to work with a base URL and no API key, but this audit heuristic only counts SearXNG when an API key or `SEARXNG_API_KEY` exists. A keyless SearXNG configuration can run web search while the audit reports it disabled.
  - confidence: 0.78
- **[P2] Avoid defaulting SearXNG to blocked localhost:** `extensions/searxng/src/config.ts:5`
  - body: The provider defaults to `http://localhost:8080`, while private-network access defaults to false and is enforced by the network guard. A default SearXNG selection therefore points at localhost and rejects its own request until users discover and set a separate flag.
  - confidence: 0.74
