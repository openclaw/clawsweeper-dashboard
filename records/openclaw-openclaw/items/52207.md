## Summary

Keep open: current main already ships the SearXNG portion, but the Tavily freshness gap still exists and this PR mixes that useful change with stale SearXNG/config wiring, an audit concern, and unrelated provider-normalization changes.

## What This Changes

The PR adds a bundled SearXNG web-search plugin, Tavily `web_search` freshness-to-days support, related config/docs/onboarding/generated metadata, and Pi embedded provider-normalization changes.

## Best Possible Solution

Keep the shipped SearXNG provider from #57317, and land Tavily freshness as a narrow Tavily-plugin change with focused request/cache mapping, docs, and tests.

## Reproduction Assessment

Yes. Static inspection is enough: current main lacks generic Tavily freshness, while the PR diff shows the freshness mapper, unread config path, security-audit heuristic, and unrelated provider normalization directly in changed files.

## Solution Assessment

No. This PR is not the best path because it combines a superseded provider implementation with one still-useful Tavily feature and unrelated runtime behavior; a Tavily-only patch is safer and easier to review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Restrict OpenRouter response normalization to supported hosts:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-38`
  - body: The new `isOpenRouter` branch rewrites every OpenRouter `openai-completions` model to `openai-responses`, even when `baseUrl` points at a custom or proxied endpoint. Those deployments may only implement Chat Completions, so this unrelated change can break configurations that explicitly chose that API surface.
  - confidence: 0.86
- **[P2] Map the documented freshness values:** `extensions/tavily/src/tavily-search-provider.ts:36-42`
  - body: The shared `web_search` surface documents `day`, `week`, `month`, and `year`, but this mapper only handles `pd`, `pw`, `pm`, and `py`. A documented call such as `freshness: "week"` would validate at the tool surface but send no Tavily recency filter.
  - confidence: 0.83
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is designed to work with a base URL and no API key, but this audit heuristic only counts SearXNG when an API key or `SEARXNG_API_KEY` exists. A keyless SearXNG configuration can run web search while the audit reports it disabled.
  - confidence: 0.78
