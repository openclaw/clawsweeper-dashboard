Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:31-37`
  - body: The new `isOpenRouter` condition rewrites every OpenRouter model using `openai-completions` to responses transport, even when the user configured a custom/proxy `baseUrl`. Existing proxies may only implement Chat Completions, so this unrelated change can break working OpenRouter deployments.
  - confidence: 0.87
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.82
- **[P2] Use the generic freshness values Tavily callers already send:** `extensions/tavily/src/tavily-search-provider.ts:76-78`
  - body: The shared `web_search` schema describes freshness as `day/week/month/year`, but this mapper only accepts `pd/pw/pm/py`. Typical generic calls with `freshness: "week"` will silently omit Tavily's `days` filter, so the new feature will not work through the documented tool contract.
  - confidence: 0.8
- **[P2] Count keyless SearXNG selection in web-search audits:** `src/security/audit-extra.sync.ts:333-337`
  - body: SearXNG is intended to work without an API key, but this audit heuristic still reports web_search as enabled only when a key or env var is present. A configured keyless SearXNG base URL can therefore be active while the small-model exposure audit underreports the tool.
  - confidence: 0.78
