## Summary

Keep open: current main already shipped the SearXNG provider, but it still does not implement Tavily freshness for the generic `web_search` path, and this PR remains mixed-scope with blocking review findings.

## What This Changes

The PR adds a bundled SearXNG web-search provider, Tavily generic `web_search` freshness support, onboarding/config/docs/generated metadata, and Pi embedded provider-normalization changes.

## Best Possible Solution

Keep the shipped SearXNG provider from #57317, and land Tavily freshness as a narrow Tavily-plugin change with provider-compatible freshness normalization, request/cache coverage, docs, tests, and no unrelated Pi runner changes.

## Reproduction Assessment

Yes. Static inspection is enough: current main lacks Tavily generic freshness, while the PR diff shows the proposed mapper, unread config path, audit heuristic, and unrelated provider normalization directly in changed files.

## Solution Assessment

No. This PR is not the best path because it mixes a superseded SearXNG implementation, one still-useful Tavily feature, security-sensitive audit behavior, and unrelated runtime changes; a Tavily-only follow-up is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-38`
  - body: The new OpenRouter branch rewrites `openai-completions` models to `openai-responses` even when `baseUrl` points at a custom or proxied endpoint. Those deployments may only implement Chat Completions, so this unrelated change can break configurations that explicitly chose that API surface.
  - confidence: 0.86
- **[P2] Map the documented freshness values:** `extensions/tavily/src/tavily-search-provider.ts:36-42`
  - body: The shared `web_search` surface documents `day`, `week`, `month`, and `year`, but this mapper only handles `pd`, `pw`, `pm`, and `py`. A documented call such as `freshness: "week"` would validate at the tool surface but send no Tavily recency filter.
  - confidence: 0.83
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is designed to work with a base URL and no API key, but this audit heuristic only counts SearXNG when an API key or `SEARXNG_API_KEY` exists. A keyless SearXNG configuration can run web search while the audit reports it disabled.
  - confidence: 0.78
