## Summary

Keep open: the SearXNG half is already shipped on main, but the useful Tavily freshness support is still missing and the current branch is mixed, conflicting, and has correctness/security blockers plus no real behavior proof.

## What This Changes

The PR adds a SearXNG web-search plugin, Tavily generic `web_search` freshness handling, docs/config/onboarding metadata, and unrelated Pi embedded runner provider-normalization changes.

## Best Possible Solution

Keep the shipped SearXNG implementation from main and land Tavily freshness as a focused Tavily-plugin change that maps current `day`/`week`/`month`/`year` freshness values to Tavily `time_range`, includes cache/request coverage, and avoids the unrelated Pi runner/config churn.

## Reproduction Assessment

Yes. Source inspection shows current main lacks generic Tavily freshness support, while the PR diff shows the intended Tavily path and the blocking mixed-scope changes.

## Solution Assessment

No. Merging this mixed branch is not the best solution; the maintainable path is a narrow Tavily freshness follow-up on current main while keeping the already-shipped SearXNG provider.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Map current freshness values to Tavily time_range:** `extensions/tavily/src/tavily-search-provider.ts:65-76`
  - body: The shared `web_search` tool now exposes `freshness` as `day`, `week`, `month`, or `year`, and Tavily's Search API supports those values through `time_range`; this branch only accepts `pd`/`pw`/`pm`/`py` and sends `days`, so current valid calls like `{ freshness: "week" }` are silently ignored for Tavily. ([docs.tavily.com](https://docs.tavily.com/documentation/api-reference/endpoint/search?utm_source=openai))
  - confidence: 0.86
- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:31-37`
  - body: The new `isOpenRouter` branch rewrites every OpenRouter `openai-completions` model to `openai-responses`, even when the user configured a custom or proxied `baseUrl`. That can break deployments that intentionally use a Chat Completions-only endpoint and is unrelated to the web-search feature.
  - confidence: 0.86
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: The strict runtime schema accepts `tools.web.search.tavily.apiKey`, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the newly accepted config shape and still get missing-key errors at runtime.
  - confidence: 0.82
- **[P2] Prompt SearXNG users for the base URL:** `extensions/searxng/src/searxng-search-provider.ts:55-69`
  - body: The provider advertises and stores an optional SearXNG API key as the onboarding credential, but SearXNG search needs the self-hosted `baseUrl`. Selecting SearXNG can write the user's setup input to the wrong field and leave the provider pointed at the fallback URL.
  - confidence: 0.78
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG can run with a configured base URL and no API key, but the new exposure-audit heuristic counts only key-based SearXNG setup. A keyless SearXNG configuration can therefore run web search while the audit treats web search as disabled.
  - confidence: 0.78
