## Summary

Keep open: current main and v2026.6.9 still leave `secrets audit` scoped to config/auth/model/`.env` stores and do not scan or explicitly exclude generated `service-env/*.env` files; the discussion also narrowed the desired fix to audit truthfulness rather than automatic cleanup.

## What This Changes

The issue asks for `secrets audit` to scan OpenClaw-managed generated service-env files or clearly report that those files are outside the audit scope.

## Best Possible Solution

Keep this open and first make `secrets audit` truthful about generated service-env coverage without printing secret values; handle service-env repair and OAuth residue cleanup as separate maintainer-approved follow-ups.

## Reproduction Assessment

Yes, at source level: current `runSecretsAudit` never enumerates generated service-env files, while LaunchAgent install writes and status reads those files as managed service environment. I did not replay the macOS LaunchAgent migration live in this read-only review.

## Solution Assessment

No complete solution exists on current main. The best first fix is narrow audit truthfulness, while deletion/regeneration and OAuth residue cleanup need separate security and owner decisions.
