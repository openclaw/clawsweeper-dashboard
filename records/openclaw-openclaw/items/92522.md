## Summary

Keep open: current main can report a clean secrets audit without scanning an OpenClaw-managed LaunchAgent environment file that may retain plaintext credentials. The service-env audit and repair gap is concrete, but the additional OAuth-profile deletion request needs separate credential-lifecycle judgment.

## What This Changes

Audit and repair OpenClaw-managed LaunchAgent service environment files after secrets migrate to SecretRefs, with a safer path for stale OAuth residue cleanup.

## Best Possible Solution

Make the secrets audit inspect canonical OpenClaw-managed service-env files and make migration or doctor safely remove migrated managed keys, while tracking OAuth-profile retirement separately unless maintainers define an explicit provider-selection and credential-removal contract.

## Reproduction Assessment

Yes, at source level. Current main scans the primary `.env` but not the generated LaunchAgent service-env file, while daemon tests confirm that file can contain provider keys and is reported as a file-backed environment source.

## Solution Assessment

No, not as one bundled cleanup rule. Service-env audit and migrated-key repair are the narrow security fix; automatic OAuth-profile deletion requires a separate decision about active provider credentials and fallback behavior.
