## Label Justifications

- `impact:security`: An OpenClaw-managed file may retain plaintext credentials while the security audit incorrectly reports no plaintext findings.
- `impact:auth-provider`: The retained values and additional residue concern provider API keys, SecretRefs, and authentication profiles.

## Summary

Keep open: current main and v2026.6.6 can report a clean secrets audit without inspecting an OpenClaw-managed LaunchAgent service-env file that may still contain plaintext provider credentials; the OAuth-residue cleanup request should be separated or explicitly scoped before implementation.

## What This Changes

Extend secrets auditing and repair so managed macOS service environment files cannot retain unreported plaintext credentials after SecretRef migration, with a safe path for stale authentication residue.

## Best Possible Solution

Treat generated service-env files as canonical secrets-audit inputs and add a doctor or forced-service-regeneration repair that removes obsolete managed credential keys without deleting operator-owned values; handle OAuth profile retirement as a separately scoped cleanup contract if necessary.

## Reproduction Assessment

Yes at source level: current main scans the state `.env` but not the managed LaunchAgent service-env file, while the daemon code can write and preserve credential-bearing values there. A live macOS reproduction would strengthen the exact repair-path proof but is not needed to establish the audit omission.

## Solution Assessment

No as one combined implementation: adding managed service-env inventory and repair is the narrow maintainable fix, while direct removal of stale OAuth database rows needs a separately defined safety and ownership contract unless existing doctor APIs already cover it.
