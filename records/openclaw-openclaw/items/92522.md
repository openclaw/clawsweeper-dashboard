## Summary

Keep open: current main and the latest release still let `openclaw secrets audit` report only the canonical config/auth/model/`.env` audit scope while LaunchAgent-managed service env files can separately retain plaintext provider credentials.

## What This Changes

The issue asks OpenClaw secrets auditing and SecretRef migration cleanup to account for managed macOS LaunchAgent service environment files that can retain plaintext provider keys.

## Best Possible Solution

Add generated service-env files to the secrets-audit inventory, or emit an explicit out-of-scope warning, and scope any destructive OAuth-profile cleanup as a separate doctor/auth-state contract.

## Reproduction Assessment

Yes at source level: current audit assembly omits generated LaunchAgent service-env files while LaunchAgent code persists service environment values there. A live macOS run would strengthen repair proof, but it is not needed to see the inventory gap.

## Solution Assessment

No as one combined implementation: service-env audit coverage has a clear boundary, while OAuth profile deletion needs a separate safety decision before any cleanup command rewrites auth SQLite rows.
