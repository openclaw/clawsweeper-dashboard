## Summary

Keep open: current main and v2026.6.9 still leave `secrets audit` blind to generated LaunchAgent service-env files, while the related open PR handles a different `.env` scrub over-deletion path.

## What This Changes

The issue asks `openclaw secrets audit` to report or explicitly scope out OpenClaw-managed LaunchAgent service-env plaintext after SecretRef migration.

## Best Possible Solution

Keep this open and make `secrets audit` truthful first: safely scan generated service-env files or emit an explicit out-of-scope warning, then handle service-env repair and OAuth residue cleanup as separate maintainer-approved doctor/daemon flows.

## Reproduction Assessment

Yes, at source level: current `runSecretsAudit` does not enumerate generated LaunchAgent service-env files, while daemon code writes and reads those files as file-sourced service environment. I did not replay a live macOS SecretRef migration in this read-only review.

## Solution Assessment

Yes for the audit-truthfulness direction, but no for bundling deletion or repair into an unsupervised fix. The maintainable path is a narrow redacted audit-scope change first, with service-env regeneration and OAuth cleanup handled separately after maintainer/security direction.
