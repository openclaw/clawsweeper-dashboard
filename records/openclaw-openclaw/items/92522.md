## Summary

Keep open: current main and latest release still do not include generated LaunchAgent service-env files in `secrets audit`, and service audit explicitly treats file-only managed env values as non-embedded. The older plist-inline SecretRef issues were closed by the service-env wrapper work, but this report is a distinct post-migration audit/repair gap.

## What This Changes

The issue requests that secrets audit or migration account for plaintext secrets left in OpenClaw-managed macOS LaunchAgent service environment files after SecretRef migration.

## Best Possible Solution

Have security owners decide whether generated service-env files become first-class secrets audit inputs, doctor repair inputs, or both, then implement that path with focused regression coverage and split OAuth residue cleanup if its deletion rules differ.

## Reproduction Assessment

Yes at source level: `runSecretsAudit` does not enumerate generated service-env files, and the service audit test proves file-only managed env values are not flagged. I did not replay the full macOS LaunchAgent migration live in this read-only pass.

## Solution Assessment

No complete current solution is visible. Inline service metadata repair does not cover generated service-env file residue, and OAuth residue cleanup needs explicit safety rules before automation deletes auth state.
