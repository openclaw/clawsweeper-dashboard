## Summary

Keep open: current main still leaves the reported audit truthfulness gap. `secrets audit` scans config/auth/model stores and the normal `.env`, while LaunchAgent can materialize managed values in `service-env` and current service-audit behavior deliberately does not flag file-only managed env values; the maintainer comment’s split between audit truthfulness and later repair/cleanup is the right boundary.

## What This Changes

The issue asks for `openclaw secrets audit` to account for plaintext secrets left in managed LaunchAgent service env files after SecretRef migration, with separate safer cleanup for stale OAuth/profile residue.

## Best Possible Solution

Add a narrow audit truthfulness change that detects or explicitly scopes generated service-env files without exposing secret values, then handle service-env repair/regeneration and OAuth residue cleanup as separate maintainer-approved follow-ups.

## Reproduction Assessment

Yes, at source level. Current main’s audit path does not enumerate generated `service-env` files, while LaunchAgent status can read those files and marks their values as file-sourced.

## Solution Assessment

Partly. Audit truthfulness is the narrow first fix, but bundling automatic service-env deletion and OAuth residue cleanup into the same change is not the best path without maintainer/security ownership decisions.
