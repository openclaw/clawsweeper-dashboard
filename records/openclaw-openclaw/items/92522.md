## Label Justifications

- `impact:security`: A clean secrets audit can miss plaintext credentials left in an OpenClaw-managed service environment file.
- `impact:auth-provider`: The affected values are provider API keys, SecretRefs, and stale auth profile residue.

## Summary

Keep open: current main and the latest release still leave managed service-env files outside `openclaw secrets audit`, while the existing doctor/service audit path only reports inline embedded managed env values, not file-only LaunchAgent env plaintext. The OAuth residue cleanup request should be scoped separately before implementation.

## What This Changes

The issue reports that `openclaw secrets audit --json` can report clean after SecretRef migration while an OpenClaw-managed macOS LaunchAgent service env file still contains plaintext copied secrets.

## Best Possible Solution

Add managed service-env inventory to the secrets/security migration story or make the audit output explicitly point users to the doctor service-env repair path, while keeping OAuth profile cleanup as a separate safety-scoped decision.

## Reproduction Assessment

Yes at source level: current `runSecretsAudit` never scans the managed service-env path, while LaunchAgent parsing and doctor service audit handle service environment provenance elsewhere. I did not rerun the full macOS LaunchAgent migration live in this read-only review.

## Solution Assessment

Unclear as one combined change: service-env audit or a clear audit warning plus doctor repair is the right narrow direction, but stale OAuth profile deletion needs a separate owner/safety contract.
