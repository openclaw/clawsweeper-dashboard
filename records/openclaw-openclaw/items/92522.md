## Summary

Keep open: current main still lets `openclaw secrets audit --json` omit the managed LaunchAgent `service-env` file, while LaunchAgent code can store credential-bearing environment values there. Current doctor/service logic adds partial inline-service repair, but it explicitly treats EnvironmentFile-backed values as acceptable and no duplicate or merged fix now owns the remaining audit/cleanup gap.

## What This Changes

The issue asks OpenClaw to make SecretRef migration and secrets audit account for plaintext credentials left in managed macOS LaunchAgent service-env files, with a related request for safer OAuth residue cleanup.

## Best Possible Solution

Extend secrets audit/apply/doctor to inventory and safely repair OpenClaw-owned service-env credentials while splitting OAuth profile residue into an explicit cleanup contract if needed.

## Reproduction Assessment

Yes at source level: current main scans .env/auth/models/config storage but not the generated LaunchAgent service-env file, while LaunchAgent code can write and read credential values from that file. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

No, not as one combined change: service-env audit and managed-value repair is the narrow source fix, while OAuth residue deletion needs a separately defined safety contract.
