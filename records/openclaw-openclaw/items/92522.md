## Summary

Keep open: current main and v2026.6.6 still do not cover the reported generated LaunchAgent service-env file residue. The related inline-service repair path is useful, but `secrets audit` only scans the normal config/auth/model stores and the service audit explicitly skips file-only EnvironmentFile values, which matches the reported `service-env` file surface.

## What This Changes

The issue asks OpenClaw to detect or safely repair plaintext provider credentials left in managed LaunchAgent service-env files after migrating those credentials to SecretRefs.

## Best Possible Solution

Keep this open for a security-owner decision on whether generated service-env files become first-class `secrets audit` inputs, doctor repair inputs, or both, and split OAuth profile residue cleanup if it needs a separate safety contract.

## Reproduction Assessment

Yes at source level: `runSecretsAudit` does not enumerate generated service-env files, LaunchAgent status marks generated env-file values as `file`, and service audit has a test proving file-only managed env values are not flagged. I did not run a live macOS LaunchAgent migration in this read-only pass.

## Solution Assessment

No complete current solution is visible. Inline supervisor metadata repair is not enough for the generated service-env file case, and the OAuth residue cleanup should be explicitly scoped before implementation.
