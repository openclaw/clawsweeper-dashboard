## Summary

Keep this PR open, but do not merge it as-is: the proposed wiring appears to run recovery after the existing observe step records the same suspicious signature that the recovery helper uses as a dedupe skip, so the central config-restoration behavior likely still does not fire.

## What This Changes

The PR wires suspicious config recovery into three config read entry points so a detected truncated `openclaw.json` can be restored from backup and re-read before becoming the active snapshot.

## Best Possible Solution

Move the recovery attempt before suspicious-read observation, or change the observe/recovery dedupe contract so first-read recovery still fires, then add a regression test through the public config read path plus redacted after-fix runtime proof.

## Reproduction Assessment

Yes, from source inspection: the proposed path calls recovery after `readConfigFileSnapshotInternal()` has already observed the suspicious snapshot, and the recovery helper skips the same signature. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No; the PR touches the right area, but the call appears too late in the control flow. The maintainable fix is to recover before observation or split the observer dedupe from recovery, with a public-read-path regression test.
