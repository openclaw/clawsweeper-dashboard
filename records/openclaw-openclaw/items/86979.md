## Summary

Keep this PR open: the config-truncation bug is real and the pre-observe hook is the right area, but this patch is not merge-ready because it treats a partial recovered raw/parsed pair as a full snapshot and still lacks after-fix real behavior proof.

## What This Changes

The branch imports `maybeRecoverSuspiciousConfigRead` into `src/config/io.ts` and invokes it from `finalizeReadConfigSnapshotInternalResult` before `observeConfigSnapshot`.

## Best Possible Solution

Keep the single pre-observe recovery point, but make it rebuild or re-read a full `ConfigFileSnapshot` only after a confirmed restore, with focused success/failure tests and redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source-reproducible rather than locally executed: current main has the recovery helper but the read finalizer only observes snapshots, and the linked reports/PR body include real truncation logs. I did not run a live induced truncation path in this read-only review.

## Solution Assessment

No. Calling recovery before observation is the right direction, but this implementation should rebuild or re-read the full snapshot after confirmed restore and preserve the clobbered result on copy failure.
