## What This Changes

The PR imports `maybeRecoverSuspiciousConfigRead` and calls a new helper from the normal, plugin-metadata, and write-prep config snapshot read wrappers in `src/config/io.ts`.

## Best Possible Solution

Move read recovery into the internal read flow before `observeConfigSnapshot` records or dedupes suspicious signatures, preserve fresh plugin metadata and write options after restore, add public-read-path tests, then provide redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source-level: PR head calls recovery after `readConfigFileSnapshotInternal()`, while current main observes the suspicious snapshot during internal finalization and the recovery helper skips an already-observed signature. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No; the PR touches the right surface, but the recovery call is too late in the control flow. The maintainable fix is to recover before observation or split the observe/recovery dedupe contract, with focused public-read-path coverage.
