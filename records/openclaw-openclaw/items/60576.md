## Summary

Keep open. Current main still exits on nullish unhandled-rejection reasons, and upstream Slack socket-mode can reject the start promise with an undefined value when a close event emits a disconnected state without an error. The PR is a useful, narrow implementation candidate, but it needs the required changelog entry and maintainer sign-off on whether this should be a process-wide policy or Slack-scoped handler.

## What This Changes

The PR adds an early warning-and-return path for null or undefined unhandled-rejection reasons in the global handler and adds regression tests for both values.

## Best Possible Solution

Land a policy-approved version of the fix with focused regression tests and a changelog entry: either keep the core nullish-reason guard if maintainers accept the global policy, or move the suppression into a Slack-owned unhandled-rejection handler if the desired boundary is owner-specific.

## Reproduction Assessment

Yes. On current main, emitting process unhandledRejection with undefined or null reaches exitWithTerminalRestore("unhandled rejection"), and Slack socket-mode upstream can reject undefined when its Disconnected event carries no error argument.

## Solution Assessment

Unclear. The proposed code is narrow and technically consistent with the observed Slack dependency path, but the product boundary is broader than the diff because it suppresses every empty process-wide rejection reason; a Slack-scoped handler is the safer alternative if maintainers want owner-specific behavior.
