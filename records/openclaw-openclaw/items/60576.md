## Summary

Keep open. Current main still has no null/undefined guard in the global unhandled-rejection handler, so those rejection reasons fall through the existing fatal fallback. The merged browser-specific #73983 work is related but does not implement this PR’s generic infra behavior, and its context explicitly kept this PR separate.

## What This Changes

The PR adds a warning-and-return path for null or undefined unhandled-rejection reasons and regression tests proving those reasons do not trigger process exit.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and land either the same core null/undefined guard with regression tests after maintainer sign-off on process-wide policy, or an extension-owned Slack handler if maintainers want core to keep treating empty unknown rejections as fatal.

## Reproduction Assessment

Yes. A process unhandledRejection event with undefined or null currently bypasses the registered-handler, abort, fatal/config, and transient checks, then reaches exitWithTerminalRestore("unhandled rejection"). The existing fatal-detection test helper is the right focused reproduction path once edits are allowed.

## Solution Assessment

Unclear. The patch is narrow and has sensible tests, but the policy choice is broader than the code size because it suppresses all empty process-wide rejection reasons; a Slack-scoped handler via registerUnhandledRejectionHandler would be safer if maintainers want owner-specific behavior.
