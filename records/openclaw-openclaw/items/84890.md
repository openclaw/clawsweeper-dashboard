## Summary

This is a plausible focused gateway availability fix, but it should stay open because the PR still lacks after-fix real behavior proof for the runtime upgrade/chunk-rotation path and because the eager load intentionally makes lifecycle-runtime import failures affect gateway startup earlier.

## What This Changes

The PR eager-loads `src/cli/gateway-cli/lifecycle.runtime.ts` at gateway loop startup and adds SIGUSR1 handler rejection handling that marks restart tokens consumed on failure.

## Best Possible Solution

Land the narrow lifecycle-runtime priming fix only after redacted after-fix runtime proof shows SIGUSR1 restart completes across package-tree rotation or the equivalent launchd update flow.

## Reproduction Assessment

Yes, at source level: current main lazily imports the lifecycle runtime inside the SIGUSR1 handler and restart scheduling coalesces while the emitted token is unconsumed. I did not run a real package-swap or chunk-rotation reproduction, and current docs/source now route supervised package-manager `update.run` through a managed handoff.

## Solution Assessment

Mostly yes: eager-loading the existing lifecycle hub is a small maintainable fix for the lazy import failure mode, and the catch unblocks the restart token defensively. The merge should wait for after-fix runtime proof because the change intentionally moves lifecycle-runtime import failures into gateway startup.
