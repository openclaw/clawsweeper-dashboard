## Label Justifications

- `P1`: The PR targets a broken gateway update/restart workflow that can leave users running an old process against swapped package files.
- `merge-risk: 🚨 availability`: The diff intentionally broadens gateway startup dependencies by loading the lifecycle runtime before lock acquisition, which can turn a transitive lifecycle import failure into startup failure.

## Summary

Keep open for maintainer review and possible landing: the patch is focused, the source path matches the reported restart deadlock, and the updated PR body now supplies live after-fix terminal proof. I found no blocking correctness finding; the remaining concern is the deliberate availability tradeoff from eager-loading the lifecycle runtime before gateway lock acquisition.

## What This Changes

The PR eager-loads the gateway lifecycle runtime at gateway-loop startup and catches SIGUSR1 handler rejections so restart tokens are marked handled on failure.

## Best Possible Solution

Land the lifecycle-runtime priming fix after maintainer acceptance of the startup fail-fast behavior, and keep the distinct restartIntent/stale-token fix tracked through https://github.com/openclaw/openclaw/pull/84334.

## Reproduction Assessment

Yes, source-reproducible: current main lazily imports `lifecycle.runtime.ts` inside the SIGUSR1 handler, and the restart scheduler coalesces while an emitted token remains unconsumed. I did not run a local package-swap reproduction in this read-only review.

## Solution Assessment

Yes, mostly: eager-loading the existing lifecycle runtime loader is a narrow fix that covers the affected lifecycle callsites without changing restart-token semantics. A focused regression test would improve durability, but I did not find a better small implementation path.
