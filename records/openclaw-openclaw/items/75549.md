## Summary

Keep this PR open. The append-mode direction is useful and current main still has the overwrite-only gap, but the current head still lets an append mutate after the call is canceled while it is waiting behind the same-file mutation queue.

## What This Changes

The PR adds an optional `append` parameter to the agent `write` tool, routes host/workspace/sandbox append writes, adds focused append tests, and records the change in `CHANGELOG.md`.

## Best Possible Solution

Land the append flag after it matches the existing write tool safety envelope: unchanged overwrite default, flat provider-safe schema, workspace/sandbox boundaries preserved, same-file serialization, abort checks inside the queued mutation path, and regression coverage for host, workspace-only, sandbox, concurrent, and canceled append calls.

## Reproduction Assessment

Yes. Main is source-reproducible as overwrite-only outside memory flush, and the PR bug is source-reproducible by queuing an append behind an existing same-path mutation, aborting before the queued callback runs, and observing that `ops.appendFile` still executes.

## Solution Assessment

No. The optional `append` flag is the narrow maintainable solution, but this implementation needs queued-abort handling before it is consistent with upstream write semantics.
