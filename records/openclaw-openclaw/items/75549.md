## Summary

Keep open. The linked overwrite-only write-tool problem is real on current main, but this PR is not mergeable as written because the append path bypasses existing schema, path-resolution, validation, and sandboxed-write contracts.

## What This Changes

The PR adds an optional `append` parameter to the model-facing `write` tool, adds host append operations and workspaceOnly tests, and adds a changelog entry for cron and monitor workflows.

## Best Possible Solution

Land a corrected append implementation that preserves the existing write contract: a flat provider-safe schema, shared path resolution, unchanged required-parameter validation, host and sandbox write support, workspace boundary behavior unchanged, and focused regressions for tilde paths, `@` paths, malformed append calls, and sandboxed append.

## Reproduction Assessment

Yes. Current-main source inspection shows ordinary write remains overwrite-only outside memory-flush runs, and the PR defects are reproducible from the diff plus upstream TypeBox and pi-coding-agent path-resolution contracts.

## Solution Assessment

No. An optional append flag is a reasonable narrow direction, but this implementation is not the best fix until it reuses the existing write semantics and covers sandboxed writes rather than introducing a parallel append path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep append schemas top-level object:** `src/agents/pi-tools.read.ts:814-824`
  - body: `Type.Intersect` emits a top-level `allOf`, and OpenClaw intentionally preserves top-level `allOf` during tool-schema normalization. That breaks the existing provider-facing contract that model tools have a top-level object schema, so add `append` by extending the existing object schema instead.
  - confidence: 0.92
- **[P2] Reuse write path resolution for append:** `src/agents/pi-tools.read.ts:837-838`
  - body: The append branch resolves paths with `path.resolve(ops.root, filePath)` instead of the upstream write tool's `resolveToCwd` handling. Paths normal write supports, including `~/...` and `@/...`, are treated as workspace-relative and append to the wrong file.
  - confidence: 0.9
- **[P2] Validate append calls before writing:** `src/agents/pi-tools.read.ts:827-835`
  - body: For `append: true`, this bypasses `wrapToolParamValidation` and turns missing or non-string `content` into an empty string, returning success after mutating the file. Append mode should enforce the same required `path` and `content` validation as overwrite mode.
  - confidence: 0.86
- **[P2] Cover sandboxed write tools too:** `src/agents/pi-tools.read.ts:660`
  - body: This wraps only the host write factory, but sandboxed writable runs skip the host write tool and add `createSandboxedWriteTool` separately. Those cron or agent workflows would still expose overwrite-only `write`, so the advertised general append mode is incomplete.
  - confidence: 0.82
