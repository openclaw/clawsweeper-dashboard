## Summary

Keep open. Current main still has the general `write` tool as overwrite-only outside the memory-flush wrapper, so the PR addresses a real gap, but the proposed patch has blocking correctness issues: it changes the write schema to a top-level `allOf`, bypasses existing path normalization and argument validation on append calls, and does not cover sandboxed write tools.

## What This Changes

The PR adds an optional `append` parameter to the host write tool path, routes append calls through new append operations, adds append behavior tests, and adds a changelog entry for cron and monitor workflows.

## Best Possible Solution

Land a corrected append implementation that preserves existing write semantics: a flat provider-safe schema, existing path normalization and required-parameter validation, host and sandbox write support, workspace boundary behavior unchanged, and focused regression tests for tilde paths, schema shape, malformed append calls, and sandboxed append.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the original overwrite-only behavior outside memory flush, and the PR defects are reproducible from the diff plus dependency contracts: TypeBox `Intersect` produces top-level `allOf`, and upstream write path normalization is bypassed by the new append branch.

## Solution Assessment

No. Adding an optional append flag is a reasonable direction, but this implementation is not the narrowest maintainable fix until it reuses the existing write contract for path resolution and validation, keeps schemas provider-safe, and covers sandboxed writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a flat object schema for append:** `src/agents/pi-tools.read.ts:814-824`
  - body: `Type.Intersect` emits a top-level `allOf`, and OpenClaw's schema normalizer intentionally preserves top-level `allOf`. That breaks the existing contract that every model-facing tool has a top-level `type: "object"` schema, which some providers require and `createOpenClawCodingTools` already tests.
  - confidence: 0.92
- **[P2] Preserve write path normalization for append:** `src/agents/pi-tools.read.ts:837-838`
  - body: The append branch bypasses `createWriteTool`'s `resolveToCwd` path handling and calls `path.resolve(root, filePath)` directly. Paths that normal write currently supports, such as `~/.openclaw/...` from the linked cron report or `@/abs/path`, are treated as workspace-relative paths, so append writes the wrong file.
  - confidence: 0.9
- **[P2] Validate append calls before writing:** `src/agents/pi-tools.read.ts:827-835`
  - body: For `append: true`, the wrapper skips the already-validated tool and only checks `path`; a missing or non-string `content` becomes an empty string and returns success. Append mode should preserve the same required-parameter validation as overwrite mode before mutating the file.
  - confidence: 0.86
- **[P2] Add append support to sandboxed writes:** `src/agents/pi-tools.read.ts:661`
  - body: This wraps only `createHostWorkspaceWriteTool`, but sandboxed runs skip that host write tool and add `createSandboxedWriteTool` later. Sandboxed cron or agent workflows with writable workspaces would still expose overwrite-only `write`, so the advertised general append mode remains unavailable there.
  - confidence: 0.82
