## Summary

Keep open. Current main and the latest release still read native ClawHub skill archives with `response.arrayBuffer()` after the ClawHub request timeout has been cleared, so a headers-then-stalled body can still block skill installation and prevent channel final/error delivery; the linked fix PR was closed unmerged.

## What This Changes

The issue asks for chat-triggered skill installation to acknowledge/progress and finish with completion or failure instead of leaving the agent silent with queued follow-ups.

## Best Possible Solution

Add a shared timeout or signal-aware body-consumption path for ClawHub archive downloads, cover stalled-mid-stream skill archive reads with regression tests, and verify chat channels still deliver progress plus a final success or error.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a fetch implementation can return 200 headers with a body whose `arrayBuffer()` never resolves, and current `downloadClawHubSkillArchive` awaits it after the request timer is cleared. I did not run the live Telegram/Feishu E2E path in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to keep an OpenClaw-owned abort timeout alive through archive body consumption, preferably via a shared ClawHub download helper, rather than treating this as a Telegram-only transport problem.
