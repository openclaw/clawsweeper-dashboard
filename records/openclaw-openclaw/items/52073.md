## Summary

Keep open. Current main still leaves native ClawHub archive body consumption outside the active request timeout, so a server that returns headers and then stalls can keep skill installation pending and prevent visible completion or error delivery. The prior linked fix attempt was closed unmerged, and the newer Feishu report suggests maintainers should also verify the shared long-running channel liveness path after the ClawHub timeout is fixed.

## What This Changes

The issue requests that skill installation triggered from a chat channel acknowledge promptly, show progress, and proactively deliver completion or failure instead of making the agent appear silent until another message arrives.

## Best Possible Solution

Keep the issue open and land a focused reliability fix that keeps an abort or idle timeout active through ClawHub archive body consumption, adds a stalled-body regression test, and verifies chat-channel progress plus final/error delivery for the skill-install path. If the Feishu long-task report still reproduces after that, split it into a shared channel liveness follow-up rather than hiding it under the ClawHub download fix.

## Reproduction Assessment

Yes, for the original skill-install root cause there is a high-confidence code-level reproduction: a ClawHub endpoint that returns headers and then stalls during the body read will leave `response.arrayBuffer()` pending after the request timeout is cleared. The newer Feishu report is corroborating user evidence, but it lacks logs and may be a separate long-task path.

## Solution Assessment

Yes for the immediate skill-install failure: the narrow maintainable fix is to bound or stream the ClawHub archive body read, not to remove normal per-chat serialization. It is unclear whether that fully solves the newer Feishu case, so the repair should include channel delivery verification and leave room for a separate shared-liveness follow-up.
