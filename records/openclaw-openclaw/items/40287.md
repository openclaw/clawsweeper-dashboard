## Summary

Keep open. The Slack `file_shared` gap still appears valid on current main, but this PR cannot merge as-is because it targets the legacy `src/slack` path, omits setup/docs subscription updates, uses a weak history lookup, and lacks real Slack proof.

## What This Changes

The PR adds a legacy Slack `file_shared` event fallback under `src/slack`, dedupes file messages, surfaces attachment notes, and extends Slack attachment types.

## Best Possible Solution

Implement the fallback in `extensions/slack`, add Slack manifest/docs/test coverage for `file_shared`, target history retrieval around the event timestamp, preserve media safeguards and dedupe behavior, and require redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main registers only `message` and `app_mention` for inbound Slack message handling and setup/docs omit `file_shared`; a live repro would use Slack Socket Mode with a large or multi-file upload that emits `file_shared` without a normal message event.

## Solution Assessment

No. The PR identifies the right gap, but the patch is not the best implementation because it targets the removed legacy Slack path, omits setup/docs subscription changes, and relies on a sliding recent-history window instead of an event-timestamp-bounded lookup.

  - confidence: 0.91

  - body: After a delay, the fallback fetches only a recent message window using `limit`, so busy channels can still push the upload out of the result set. Use the file event timestamp with `oldest`/`latest` and `inclusive`, or an equivalent targeted lookup.
  - confidence: 0.88
