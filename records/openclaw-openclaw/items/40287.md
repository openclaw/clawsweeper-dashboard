## Summary

Keep this PR open. Current main still lacks a Slack `file_shared` handler and setup/docs subscription entry, so the user problem is not implemented, but the PR is not mergeable as-is because it targets the removed `src/slack` layout and still needs setup plus lookup hardening before it can reliably fix large Slack uploads.

## What This Changes

The PR adds a Slack `file_shared` fallback handler under `src/slack`, marks normal file messages for deduplication, surfaces Slack attachment notes, and extends Slack attachment typing.

## Best Possible Solution

Port the useful idea into the active `extensions/slack` plugin: add and register a `file_shared` handler, update the generated Slack setup manifest and public Slack docs, hydrate the matching message with a timestamp-anchored bounded history lookup, dedupe Slack retries plus normal `message`/`file_share` delivery, handle multi-file uploads, preserve the current Slack media security/download limits, and cover it with focused Slack plugin tests.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: current main registers only `message` and `app_mention` Slack inbound events and the setup manifest omits `file_shared`; a live reproduction would share a large or multi-file upload in Slack Socket Mode and observe no inbound dispatch when Slack sends only `file_shared`.

## Solution Assessment

No. The PR addresses the right class of Slack fallback, and its latest commit fixed some earlier review comments, but the current patch is not the best solution because it edits the removed `src/slack` tree and omits required setup/docs plus timestamp-anchored history lookup.
