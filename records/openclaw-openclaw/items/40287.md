## Summary

Keep open: the Slack `file_shared` gap still exists on current main, but this external PR is not mergeable as-is because it targets the removed `src/slack` legacy path, misses setup/docs event subscriptions, and lacks real behavior proof.

## What This Changes

The branch adds a Slack `file_shared` fallback handler and attachment-note handling in the legacy `src/slack` runtime path.

## Best Possible Solution

Implement the fallback in `extensions/slack`: subscribe setup/docs to `file_shared`, register an active-plugin handler, hydrate the matching Slack message with a timestamp-bounded lookup, dedupe against normal message delivery, preserve existing file/media handling, and add focused Slack tests.

## Reproduction Assessment

Yes from source inspection, not a live Slack run. Current main registers only `message`/`app_mention` for inbound Slack messages and setup/docs omit `file_shared`; a live repro would share a large or multi-file upload in Slack Socket Mode and verify no dispatch occurs when Slack emits only `file_shared`.

## Solution Assessment

No. The PR points at a real gap, but the current patch is not the best fix because it edits the removed `src/slack` runtime, misses setup/docs subscriptions, and uses a sliding recent-history lookup instead of anchoring around the file event.
