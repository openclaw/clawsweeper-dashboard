## Summary

Keep open: the PR targets a real Slack large/multi-file fallback gap, but the patch is stale against the active Slack plugin layout and should not merge as-is.

## What This Changes

The branch adds a legacy Slack `file_shared` fallback handler under `src/slack`, marks normal file messages for deduplication, appends attachment notes, and extends Slack attachment typing.

## Best Possible Solution

Port the useful fallback into `extensions/slack`: register and test `file_shared`, update generated setup/docs manifests, hydrate the matching message with a timestamp-bounded lookup, preserve existing media safety limits, and add a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main registers only `message` and `app_mention` as inbound Slack message events, and the setup/docs manifests omit `file_shared`; a live probe would share a large or multi-file upload and check that no OpenClaw dispatch occurs when Slack emits only `file_shared`.

## Solution Assessment

No. The PR targets the right Slack fallback, but it is not the best current fix because it edits removed legacy files, does not update setup/docs event subscriptions, and leaves the fallback history lookup vulnerable to a sliding recent-history window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Port the handler into the active Slack plugin:** `src/slack/monitor/events.ts:3`
  - body: Current main no longer has the `src/slack` runtime this patch edits; the active Slack channel lives under `extensions/slack/src`. As written, the new handler is not wired into the runtime that ships on main and needs to be ported instead of recreating legacy Slack files.
  - confidence: 0.97
- **[P2] Subscribe Slack apps to file_shared events:** `src/slack/monitor/events.ts:21-24`
  - body: Registering a handler is not enough for installed Slack apps to receive these events. The generated Slack manifest and public setup docs still omit `file_shared`, so users following setup will not get the fallback event this handler depends on.
  - confidence: 0.9
- **[P2] Anchor the history lookup to the file event timestamp:** `src/slack/monitor/events/file-shared.ts:84-87`
  - body: The fallback still fetches recent channel history without `oldest` or `latest` tied to the `file_shared` event. Busy channels can push the upload outside that window, making the large-file fallback silently drop the message it is meant to recover.
  - confidence: 0.85
