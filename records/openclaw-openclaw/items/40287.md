## Summary

Keep open: the Slack `file_shared` gap is still valid on current main, but this branch is stale and incorrect because it edits the removed legacy Slack path and does not update the active plugin setup surface.

## What This Changes

The PR adds a Slack `file_shared` fallback handler, file/message dedupe, forwarded attachment notes, and Slack attachment typing under the legacy `src/slack` path.

## Best Possible Solution

Implement the fallback in `extensions/slack`, add `file_shared` to generated setup/docs, hydrate with a timestamp-bounded lookup, preserve existing media limits and message dedupe, and prove it with a real Slack run.

## Reproduction Assessment

Yes, by source inspection: current main only registers `message` and `app_mention` for inbound Slack messages and its setup manifest omits `file_shared`. A live repro would share a large or multi-file upload where Slack emits only `file_shared` and confirm no inbound dispatch occurs.

## Solution Assessment

No. The requested direction is useful, but this patch is not the best solution because it edits the removed legacy Slack runtime, leaves setup/docs unsubscribed to `file_shared`, and uses a sliding recent-history lookup instead of anchoring around the file event.
