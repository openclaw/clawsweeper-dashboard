## Summary

Keep open: the Slack file_shared fallback remains a valid missing capability, but this PR is not mergeable as-is because it targets the legacy Slack path, misses setup/docs subscription work, still has a brittle history lookup, and lacks real Slack proof.

## What This Changes

The PR adds Slack file_shared fallback handling, dedupe state, forwarded-attachment notes, and Slack attachment typing under src/slack.

## Best Possible Solution

Implement the fallback in extensions/slack, add Slack manifest/docs/test coverage for file_shared, use an event-timestamp-bounded history lookup with dedupe, preserve existing forwarded-attachment media handling, and require redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection shows current main and the latest release subscribe to message/app_mention but not file_shared, and the setup manifest/docs do not request file_shared; a live repro would use Slack Socket Mode with a large or multi-file upload that emits file_shared without a normal message event.

## Solution Assessment

No. The PR identifies the right gap, but the patch is not the best implementation because it targets the legacy src/slack path, omits setup/docs subscription changes, and uses a sliding recent-history window instead of a timestamp-bounded lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Move the fallback into the active Slack plugin:** `src/slack/monitor/events.ts:5`
  - body: Current main runs Slack from extensions/slack/src, not the legacy src/slack path this PR modifies. Registering the fallback here will not wire it into the runtime users run.
  - confidence: 0.97
- **[P2] Subscribe file_shared in setup and docs:** `src/slack/monitor/events/file-shared.ts:50`
  - body: The new listener depends on Slack delivering file_shared, but the branch does not update the generated Slack manifest, setup tests, or user setup docs to request that bot event. Newly configured apps still will not send this fallback event.
  - confidence: 0.91
- **[P2] Anchor the history lookup to the file event:** `src/slack/monitor/events/file-shared.ts:82-84`
  - body: After a delay, the fallback fetches only a recent message window using limit. In busy channels the upload can still fall outside that window, so the only fallback path silently drops the event; use the file event timestamp with oldest/latest/inclusive or equivalent targeted lookup.
  - confidence: 0.88
