## Summary

Keep open: the PR targets a real remaining Slack `file_shared` gap, but the current patch is not mergeable because it edits the removed legacy `src/slack` path, misses setup/docs subscription changes, uses a fragile history lookup, and still lacks real Slack proof.

## What This Changes

The PR adds a Slack `file_shared` fallback, dedupe, attachment note handling, and Slack attachment types under the legacy `src/slack` path.

## Best Possible Solution

Implement the fallback in `extensions/slack`, update Slack manifest/docs/test coverage for `file_shared`, use a timestamp-bounded or paginated message lookup, preserve existing media and dedupe safeguards, and require redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection shows current main has no active `file_shared` registration and the setup surfaces do not subscribe it; a live repro would use Slack Socket Mode with a large or multi-file upload that emits `file_shared` without a normal message event.

## Solution Assessment

No. The requested fallback is useful, but this patch is not the best implementation because it targets the removed legacy Slack path and leaves setup/docs plus the history lookup incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Move the fallback into the active Slack plugin:** `src/slack/monitor/events.ts:5`
  - body: Current main no longer has a runnable `src/slack` implementation; Slack runtime code lives under `extensions/slack/src`. Registering the fallback here leaves the feature unwired on current main and keeps the branch conflicting against the plugin migration.
  - confidence: 0.97
- **[P2] Subscribe file_shared in setup and docs:** `src/slack/monitor/events/file-shared.ts:50`

  - confidence: 0.91
- **[P2] Anchor the history lookup to the file event:** `src/slack/monitor/events/file-shared.ts:82-85`
  - body: After a delay, the fallback fetches only a recent channel window with `limit`, so busy channels can still push the upload out of the result set. Use the file event timestamp with a bounded lookup or paginate around the event instead of relying on the latest messages.
  - confidence: 0.88
