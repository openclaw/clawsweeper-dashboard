## Summary

Keep open: current main still lacks a Slack `file_shared` event path, and this is the canonical open tracker, but the PR branch is not merge-ready because it edits the removed legacy `src/slack` runtime, misses event-subscription setup/docs, uses an unanchored history lookup, and lacks real behavior proof.

## What This Changes

The PR branch adds a legacy `src/slack` `file_shared` fallback handler, dedupe plumbing, an attachment-note helper, and Slack attachment typings.

## Best Possible Solution

Implement the fallback in `extensions/slack`, update Slack manifest/docs to subscribe to `file_shared`, hydrate the matching message with a timestamp-bounded lookup and dedupe, preserve current attachment/media handling, and require redacted real Slack proof before merge.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main registers only `message` and `app_mention` for inbound Slack messages and omits `file_shared` from setup/docs. A live repro would share a large or multi-file upload in Socket Mode and verify no inbound dispatch occurs when Slack emits only `file_shared`.

## Solution Assessment

No. The requested direction is valid, but this branch is not the best implementation because it edits removed legacy files, misses the Slack app subscription surface, and uses a sliding recent-history lookup instead of anchoring the lookup to the file event timestamp.
