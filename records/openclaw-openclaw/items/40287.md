## Summary

Keep open: the Slack `file_shared` gap still exists on current main and this PR remains the canonical open tracker, but the branch is not mergeable as-is because it targets the removed legacy Slack path, omits setup/docs subscription changes, uses a sliding history lookup, and lacks real behavior proof.

## What This Changes

The PR adds a legacy `src/slack` `file_shared` event handler, dedupe sets, attachment-note rendering, and Slack attachment typing for large-file fallback handling.

## Best Possible Solution

Implement the fallback in `extensions/slack`, update Slack setup/docs/tests for `file_shared`, use a timestamp-bounded lookup or equivalent targeted retrieval, preserve media safeguards and dedupe behavior, and require redacted real Slack proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main registers only `message` and `app_mention` for inbound Slack messages and setup/docs omit `file_shared`. A live repro would use Slack Socket Mode with a large or multi-file upload that emits `file_shared` without a normal `message` event; I did not run that live Slack workspace flow here.

## Solution Assessment

No. The PR identifies the right gap, but the current patch is not the best solution because it targets the removed legacy path, omits the required setup/docs subscription change, and uses a recent-history window instead of a targeted event-timestamp lookup.

  - confidence: 0.88
