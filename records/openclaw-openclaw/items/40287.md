## Summary

Keep open: the Slack fallback gap still exists on current main, but this PR is not mergeable as-is because it edits removed legacy `src/slack` paths and omits required active-plugin setup/docs and a timestamp-anchored history lookup.

## What This Changes

The PR adds a legacy Slack `file_shared` fallback handler under `src/slack`, marks normal file messages for deduplication, appends forwarded-attachment notes, and extends Slack attachment typing.

## Best Possible Solution

Port the useful fallback into `extensions/slack`: register and test a `file_shared` handler, add `file_shared` to generated setup/docs manifests, hydrate the matching Slack message with a bounded event-timestamp lookup, and preserve existing Slack media safety limits.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main registers only `message` and `app_mention` as inbound Slack message events and the generated Slack manifest omits `file_shared`; a live probe would share a large or multi-file upload in Slack Socket Mode and observe no OpenClaw dispatch when Slack sends only `file_shared`.

## Solution Assessment

No. The PR targets the right Slack fallback problem, but it is not the best fix because it edits removed legacy files and does not update the active Slack plugin setup/docs or anchor the history search to the file event timestamp.

  - body: Current main no longer has the `src/slack` runtime this patch edits; the active Slack channel is under `extensions/slack/src`. As written, the fallback handler is not wired into the runtime that ships on main and needs to be ported instead of recreating legacy Slack files.

  - body: After the fixed delay this still fetches only recent channel history without `oldest` or `latest` tied to the `file_shared` event. Busy channels can push the upload outside that window, making the fallback silently drop the large-file message it is meant to recover.
  - confidence: 0.85
