## Summary

This PR should stay open as useful source work, but it is not mergeable: the active Slack implementation on current main is under `extensions/slack`, while the PR edits the removed legacy `src/slack` path, and current setup/docs still do not subscribe Slack apps to `file_shared`.

## What This Changes

The PR adds a legacy Slack `file_shared` fallback handler, marks normal file messages for deduplication, adds forwarded-attachment notes, and extends Slack attachment typing.

## Best Possible Solution

Port the useful fallback into `extensions/slack`: register and test a `file_shared` handler, add `file_shared` to generated setup/docs manifests, hydrate the matching Slack message with a timestamp-bounded lookup, preserve existing media safety limits, and add a changelog entry.

## Reproduction Assessment

Yes, at source level. Current main has no active `file_shared` handler or manifest subscription, so a Slack app following OpenClaw setup cannot deliver that fallback event into the Slack message pipeline.

## Solution Assessment

No. The PR targets the right Slack fallback gap, but the implementation edits the legacy `src/slack` runtime, misses setup/docs subscription, and keeps a sliding history lookup that can miss the uploaded file in busy channels.

  - body: Current main no longer ships the `src/slack` runtime this patch edits; the active Slack channel lives under `extensions/slack/src`. As written, this fallback is not wired into the implementation users run and needs to be ported instead of adding legacy files.

  - confidence: 0.9

  - body: After the fixed delay this still fetches the most recent channel messages without an `oldest` or `latest` bound tied to the `file_shared` event. Busy channels can push the upload outside that window, making the fallback silently drop the file message it is meant to recover.
  - confidence: 0.86
