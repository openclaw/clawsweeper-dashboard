## Summary

Keep open. The idea is useful, but the branch is not merge-ready: it exposes session identity on a public provider hook without cache-safe plumbing, runtime-plan coverage, a settled privacy contract, or real downstream attribution proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` and threads them through one embedded-runner extra-params call path.

## Best Possible Solution

Define one privacy-scoped provider request metadata contract, make it cache-safe and available on every advertised hook path, then cover it with focused tests, docs, and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main has a session-agnostic prepared-param cache and runtime-plan resolver, and the patch does not cover every context it would advertise. The original LiteLLM spend-log behavior was not reproduced live.

## Solution Assessment

No. Exposing raw session fields on one prepared-param path is not the best final shape; the safer solution is a privacy-scoped provider metadata contract with cache/runtime-plan coverage and proof.

  - confidence: 0.84
