## Summary

Keep open. The requested provider attribution seam is still relevant, but this patch is not mergeable because it makes prepared extra params session-dependent without cache isolation, misses the runtime-plan preparation path, exposes session fields inconsistently through inherited hook contexts, and lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` to provider extra-param preparation types and threads them through `resolvePreparedExtraParams`, `applyExtraParamsToAgent`, and one embedded-runner call site.

## Best Possible Solution

Define a privacy-conscious provider request/session metadata contract, thread it through each advertised hook path consistently, isolate or avoid caches for session-dependent values, and cover it with focused tests and docs while coordinating with https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defects: source inspection shows a session-agnostic prepared-param cache, a runtime-plan preparation path that would not receive the new session fields, and inherited provider hook contexts that would be advertised without consistent population. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this narrow thread-through is not the best mergeable solution because cache isolation, runtime-plan preparation, public hook context consistency, tests/docs, and privacy/API scope need to be solved together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include session identity in prepared-param cache keys:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can return values derived from `ctx.sessionKey` or `ctx.sessionId`, the existing prepared-param cache can reuse one session's `user` value for another session with the same provider/model/config. Add session identity to the cache key, disable caching for session-dependent preparation, or move dynamic request identity out of cached extra params.
  - confidence: 0.9
- **[P2] Prepare runtime extra params with session context:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main computes `preparedRuntimeExtraParams` before `applyExtraParamsToAgent` and passes it via `options.preparedExtraParams`, so adding session arguments only to the apply call does not make `prepareExtraParams` see them on that path. Thread the fields through the runtime-plan `resolveExtraParams` path or prepare after session context is available.
  - confidence: 0.88
- **[P2] Keep inherited provider hook contexts populated:** `src/plugins/types.ts:465-476`
  - body: Adding fields to `ProviderPrepareExtraParamsContext` also advertises them on transport and stream-wrapper contexts derived from that type, but the patch only supplies them to the prepare hook. Either split the base context or populate the fields everywhere the public types expose them.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch can misattribute or leak session identity through cached provider extra params.

Concerns:
