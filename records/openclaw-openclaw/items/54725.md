## Summary

Keep open: current main still does not expose session identity on the provider prepare hook, and the proposed patch has merge-blocking API and cache issues that need maintainer direction before repair.

## What This Changes

The PR branch adds optional `sessionKey`/`sessionId` fields to `ProviderPrepareExtraParamsContext` and passes them from embedded attempts into provider extra-param preparation.

## Best Possible Solution

Define one provider session/request metadata contract, thread it through every advertised hook path that exposes it, isolate session-dependent prepared-param caching, and add SDK docs, regression tests, and a changelog entry.

## Reproduction Assessment

Yes for the PR defects: source inspection of current main shows cached prepared extra params without session identity and a runtime-plan preparation path that bypasses the PR's new apply arguments. The original downstream spend-log use case was not live-tested.

## Solution Assessment

No. The direction is useful, but this patch is too narrow for a public provider SDK contract; the safer solution is a reviewed metadata/session design with cache isolation and coverage for every exposed hook path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include session identity in the extra-param cache key:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can depend on `ctx.sessionKey` or `ctx.sessionId`, the existing prepared-param cache can reuse one session's returned `user` value for a later session with the same provider/model/config. Include the session fields in the key or avoid caching session-dependent prepared params.
  - confidence: 0.9
- **[P2] Thread session fields through runtime-plan preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main resolves `preparedRuntimeExtraParams` through the runtime plan before `applyExtraParamsToAgent`, and the apply path skips preparation when that value is supplied. Passing session fields only to `applyExtraParamsToAgent` leaves the runtime-plan path without the new context.
  - confidence: 0.86
- **[P2] Keep advertised hook contexts in sync:** `src/plugins/types.ts:465-476`
  - body: Adding these fields to `ProviderPrepareExtraParamsContext` also advertises them on derived transport and stream-wrapper contexts, but the patch only populates the prepare hook. Either split the types or pass the fields everywhere the public types expose them.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/plugins/types.ts:465`
  - body: This is a public provider SDK feature, but the PR only changes source files. Add a `CHANGELOG.md` entry for the new provider hook context fields.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff can misattribute session identity through cached prepared extra params.

Concerns:

- **[medium] Session identity can leak through cached extra params:** `src/agents/pi-embedded-runner/extra-params.ts:159`
  - body: A provider that returns `user: ctx.sessionKey` from `prepareExtraParams` can have that prepared result cached and reused for another session because the cache key omits the new fields, sending the wrong session identity to downstream provider logs.
  - confidence: 0.84
