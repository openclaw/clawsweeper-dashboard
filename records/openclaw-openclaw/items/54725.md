## Summary

Keep open. The provider observability idea is useful, but this PR is not merge-ready because it exposes session-derived identity through a session-agnostic prepared-param cache, misses the runtime-plan preparation path, broadens inherited public hook contexts without consistently populating them, and still lacks real downstream attribution proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` and threads them through one embedded-runner extra-params call path.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, make session-dependent prepared params cache-safe and runtime-plan aware, populate every public hook that advertises the fields, then cover it with focused tests, docs, and redacted downstream proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main has a session-agnostic prepared-param cache, a runtime-plan prepare path without session context, and inherited hook contexts that would be advertised without consistent population. The original downstream spend-log behavior was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch only threads two raw identifiers through one path and misses cache isolation, runtime-plan preparation, inherited public hook contexts, privacy review, and real provider proof.
