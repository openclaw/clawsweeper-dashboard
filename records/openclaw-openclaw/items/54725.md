## Summary

Keep open. The requested provider attribution seam is useful, but the PR is not mergeable as written: it makes provider extra-param preparation session-dependent without cache isolation, misses the runtime-plan preparation path, advertises fields on inherited hook contexts that it does not populate, and lacks contributor-supplied real LiteLLM/OpenRouter proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and threads them through embedded-runner extra-param preparation.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, thread it through every advertised hook path consistently, make session-dependent values cache-safe, and cover the chosen behavior with focused payload and hook-context tests plus docs.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main shows a session-agnostic prepared-param cache, a runtime-plan preparation path that would not receive the new fields, and inherited provider hook contexts that the PR would advertise without consistent population. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable shape until cache isolation, runtime-plan preparation, inherited hook consistency, privacy/API scope, and real provider proof are resolved together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include session identity in prepared-param cache keys:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Adding `sessionKey` and `sessionId` to `prepareExtraParams` makes the hook output potentially session-dependent, but the prepared-param cache key still ignores those values. A provider returning `user: ctx.sessionKey` can have one session's identity reused for another session with the same provider/model/config.
  - confidence: 0.9
- **[P2] Prepare runtime extra params with session context:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main prepares runtime extra params through the runtime plan and then passes them as `options.preparedExtraParams`, so adding session arguments only to this later apply call does not make `prepareExtraParams` see them on that path. Thread the fields through the runtime-plan resolver or prepare after session context is available.
  - confidence: 0.88
- **[P2] Keep inherited provider hook contexts populated:** `src/plugins/types.ts:465-476`
  - body: Adding fields to `ProviderPrepareExtraParamsContext` also advertises them on transport and stream-wrapper contexts derived from that type, but the patch only supplies them to the prepare hook. Either split the base context or populate the fields everywhere the public types expose them.
  - confidence: 0.84
