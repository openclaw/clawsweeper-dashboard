## Summary

Keep open, but not merge-ready. The hook direction is useful and not implemented on current main, but this branch is merge-conflicting and needs a cache-safe, runtime-plan-aware, privacy-scoped metadata contract plus real downstream proof before maintainers should consider it.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and forwards them through the embedded runner extra-param path.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, thread it through every advertised hook path consistently, make session-dependent prepared params cache-safe, and cover it with focused tests plus redacted downstream proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main shows a session-agnostic prepared-param cache, a runtime-plan path that prepares without session context, and inherited hook contexts that would be advertised without consistent population. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch should not merge until the metadata contract, cache isolation, runtime-plan threading, inherited hook consistency, tests, and real provider proof are handled together.
