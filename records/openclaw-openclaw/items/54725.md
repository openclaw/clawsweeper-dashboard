## Summary

Keep open. The provider attribution seam is still useful and not implemented on current main, but this patch is not mergeable as written because it makes provider extra params session-dependent without cache isolation, misses the runtime-plan preparation path, and lacks real behavior proof from a downstream provider setup.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to provider extra-param preparation and threads them through one embedded-runner call site.

## Best Possible Solution

Define a privacy-conscious provider request/session metadata contract, thread it through each advertised hook path consistently, isolate or avoid caches for session-dependent values, and cover the behavior with focused tests and docs.

## Reproduction Assessment

Yes for the PR defects: source inspection shows session-agnostic prepared-param caching and a runtime-plan preparation path that would not receive the new session fields. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this narrow thread-through is not the best mergeable solution because cache isolation, runtime-plan preparation, public hook context consistency, and privacy/API scope need to be solved together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include session identity in prepared-param cache keys:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can return values derived from `ctx.sessionKey` or `ctx.sessionId`, the existing prepared-param cache can reuse one session's `user` value for later sessions with the same provider/model/config. Add session identity to the cache key, disable caching for session-dependent preparation, or move dynamic request identity out of cached extra params.
  - confidence: 0.9
- **[P2] Prepare runtime extra params with session context:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main computes `preparedRuntimeExtraParams` before `applyExtraParamsToAgent` and then passes it via `options.preparedExtraParams`, so adding session arguments only to the apply call does not make `prepareExtraParams` see them on that path. Thread the fields through the runtime-plan `resolveExtraParams` path or prepare after session context is available.
  - confidence: 0.88
- **[P2] Keep inherited provider hook contexts populated:** `src/plugins/types.ts:465-476`
  - body: Adding fields to `ProviderPrepareExtraParamsContext` also advertises them on transport and stream-wrapper contexts derived from that type, but the patch only supplies them to the prepare hook. Either split the base context or populate the fields everywhere the public types expose them.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch can misattribute or leak session identity through cached provider extra params.

Concerns:

- **[medium] Session identity can leak through cached extra params:** `src/agents/pi-embedded-runner/extra-params.ts:159`
  - body: A provider returning `user: ctx.sessionKey` from `prepareExtraParams` can have that prepared result cached and reused for another session because the cache key omits the new session fields, sending the wrong session identity to downstream provider logs.
