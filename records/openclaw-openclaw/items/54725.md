## Summary

Keep open: the provider-attribution idea is useful, but this branch is not merge-ready because session-dependent prepared params would be cache-unsafe, runtime-plan preparation still misses session context, inherited hook contexts would over-advertise the fields, raw session ids need privacy review, and no real downstream proof is attached.

## What This Changes

The branch adds optional `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` and threads them through one embedded-runner extra-params path.

## Best Possible Solution

Define one privacy-scoped provider request metadata contract, make session-dependent prepared params cache-safe and runtime-plan aware, populate every advertised hook path consistently, then cover it with focused tests, docs, and redacted downstream proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main has a session-agnostic prepared-param cache, a runtime-plan prepare path without session context, and inherited hook contexts that would be advertised without consistent population. The original downstream spend-log behavior was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch only threads two raw identifiers through one path and misses cache isolation, runtime-plan preparation, inherited public hook contexts, privacy review, and real provider proof.

## Security Review

Status: needs_attention

Summary: The patch introduces a privacy-sensitive session-identity path that can be misattributed through caching or forwarded to downstream provider logs without a settled contract.
