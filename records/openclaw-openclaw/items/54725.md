## Summary

Keep open. The PR targets a useful provider attribution gap, but the branch is conflicting and not merge-ready because the added session-aware hook output can be cached across sessions, the runtime-plan path still prepares without session context, inherited public hook contexts would not be populated consistently, and the downstream behavior lacks inspectable real proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and threads them through one embedded-runner extra-params path.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, thread it through every public hook path that advertises it, make session-dependent prepared params cache-safe, and cover it with focused tests plus redacted downstream proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main has a session-agnostic prepared-param cache, a runtime-plan prepare path without session context, and inherited hook contexts that would be advertised without consistent population. The original downstream spend-log behavior was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch should not merge until the metadata contract, cache isolation, runtime-plan threading, inherited hook consistency, tests, and real provider proof are handled together.

## Security Review

Status: needs_attention

Summary: The patch introduces a privacy-sensitive session-identity path that can be misattributed through caching or forwarded to downstream provider logs without a settled contract.

Concerns:

- **[medium] Session identity can leak through cached extra params:** `src/agents/pi-embedded-runner/extra-params.ts:159`
  - body: A provider returning `user: ctx.sessionKey` from `prepareExtraParams` can have that prepared result cached and reused for another session because the cache key omits the new session fields.
  - confidence: 0.84

  - body: The added JSDoc encourages forwarding session keys to LiteLLM/downstream proxies, but session keys can encode channel, account, peer, tenant, or task identifiers and should be explicit, sanitized, or opt-in before becoming a public provider API expectation.
  - confidence: 0.82
