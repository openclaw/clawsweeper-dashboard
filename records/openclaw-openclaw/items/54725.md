## Summary

Keep open. The PR targets a real provider attribution gap, but the patch is not merge-ready because it would advertise session fields through inherited provider hook contexts without consistently populating them, and current main now resolves prepared extra params through the runtime-plan path before `applyExtraParamsToAgent`.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` and threads them through embedded extra-param preparation.

## Best Possible Solution

Add a reviewed provider metadata/session identity contract that is documented, populated everywhere it is advertised, and covered across embedded and runtime-plan paths with focused tests.

## Reproduction Assessment

Yes for the patch defect: static inspection of current main shows the derived hook contexts and runtime-plan preparation path that the PR does not handle. The original LiteLLM spend-log attribution use case is plausible from the PR body but was not live-tested in this read-only review.

## Solution Assessment

No, not as proposed. Passing session fields only through the original prepare path is too narrow; the safer fix is to either scope the type to that hook only or consistently thread the fields through every public context that exposes them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Populate session fields on every exposed hook context:** `src/plugins/types.ts:476-481`
  - body: Adding `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` also exposes them on transport and stream-wrapper contexts, but the patch only populates the prepare hook. Plugins reading the newly typed fields from those other hooks would get `undefined`, so either thread the fields through each context builder or split the type so only the intended hook advertises them.
  - confidence: 0.86
- **[P2] Thread session fields through runtime-plan preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main resolves `preparedRuntimeExtraParams` through `AgentRuntimePlan.transport.resolveExtraParams` before calling `applyExtraParamsToAgent`, and `applyExtraParamsToAgent` skips preparation when that value is provided. Passing session fields only into `applyExtraParamsToAgent` will not reach `prepareExtraParams` after rebase, so the runtime-plan resolver and call site need the same context.
  - confidence: 0.83
