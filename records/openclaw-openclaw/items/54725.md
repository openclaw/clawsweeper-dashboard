## Summary

Keep open. The provider-metadata direction is useful, but the PR is not merge-ready because the session-aware hook output would still be cached without session identity, current main has a runtime-plan preparation path that would miss the new fields, and the PR has no contributor-provided real LiteLLM/OpenRouter proof.

## What This Changes

The PR adds optional sessionKey/sessionId fields to ProviderPrepareExtraParamsContext and forwards them through embedded-runner extra-param preparation at one attempt call site.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, thread it through every advertised hook path consistently, make session-dependent prepared params cache-safe, and cover it with focused hook-context/payload tests plus redacted downstream proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main shows a session-agnostic prepared-param cache, a runtime-plan path that prepares without session context, and inherited hook contexts that would be advertised without consistent population. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch should not merge until the metadata contract, cache isolation, runtime-plan threading, inherited hook consistency, tests, and real provider proof are handled together.
