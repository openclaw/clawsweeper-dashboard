## Summary

Keep open: current main still lacks this provider session-identity hook, but the PR is not merge-ready because it exposes session-dependent data on a cached hook, misses the runtime-plan preparation path, and needs a coherent public hook contract with the related metadata work.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to provider extra-param preparation and attempts to pass them from embedded runs into provider hooks.

## Best Possible Solution

Define a coherent provider session/request metadata contract, thread it through every advertised runtime path that exposes it, isolate cached results by session when needed, and cover it with tests, docs, and changelog.

## Reproduction Assessment

Yes for the patch defects: source inspection of current main shows cached prepared extra params without session identity and a runtime-plan preparation path that bypasses the PR’s new arguments. The original LiteLLM spend-log use case was not live-tested.

## Solution Assessment

No. The direction is useful, but the diff is too narrow for the public API it changes; the safer solution is a reviewed provider metadata/session contract with cache isolation and runtime-plan coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Include session identity in the extra-param cache key:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can depend on `ctx.sessionKey` or `ctx.sessionId`, the existing prepared-param cache can reuse the first session's returned `user` value for later sessions with the same provider/model/config. Add the session fields to the cache key or avoid caching session-dependent prepared params.
  - confidence: 0.9
- **[P2] Thread session fields through runtime-plan preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main resolves `preparedRuntimeExtraParams` through the runtime plan before calling `applyExtraParamsToAgent`, and the apply path skips preparation when that value is supplied. Passing session fields only to `applyExtraParamsToAgent` means `prepareExtraParams` still will not see them on that path.
  - confidence: 0.86
- **[P2] Keep advertised hook contexts in sync:** `src/plugins/types.ts:465-476`
  - body: Adding the fields to `ProviderPrepareExtraParamsContext` also advertises them on transport and stream-wrapper contexts that derive from that type, but the patch only populates the prepare hook. Either split the types or pass the fields everywhere the public types expose them.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/plugins/types.ts:465`
  - body: This is a public provider SDK feature, but the PR only changes source files. Add a `CHANGELOG.md` entry for the new provider hook context fields.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch can misattribute session identity through cached provider extra params.

Concerns:

- **[medium] Session identity can leak through cached extra params:** `src/agents/pi-embedded-runner/extra-params.ts:159`
  - body: A provider that returns `user: ctx.sessionKey` from `prepareExtraParams` can have that prepared result cached and reused for another session because the cache key omits the new fields, sending the wrong session identity to downstream provider logs.
  - confidence: 0.84
