## Summary

Keep open. Current main still lacks this provider extra-param session metadata seam, but this PR is not mergeable as-is because it leaves session-derived prepared params cache-unsafe, misses the runtime-plan preparation path, exposes inconsistent hook context fields, and still lacks contributor-supplied real provider proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to provider extra-param preparation types and tries to thread those values through embedded-runner extra-param preparation.

## Best Possible Solution

Define a privacy-conscious provider request metadata contract, thread it through every advertised hook path consistently, make session-dependent values cache-safe, and cover the chosen behavior with focused payload and hook-context tests plus docs.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main shows a session-agnostic prepared-param cache, a runtime-plan preparation path that would not receive the new fields, and derived provider contexts that the PR would advertise without consistent population. The original downstream spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable shape because cache isolation, runtime-plan preparation, hook-context consistency, real provider proof, and privacy/API scope need to be resolved together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include session identity in prepared-param cache keys:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can return values derived from `ctx.sessionKey` or `ctx.sessionId`, the existing prepared-param cache can reuse one session's `user` value for another session with the same provider/model/config. Add session identity to the cache key, disable caching for session-dependent preparation, or move dynamic request identity out of cached extra params.
  - confidence: 0.9
- **[P2] Prepare runtime extra params with session context:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main computes prepared runtime extra params through the runtime plan and passes them as `options.preparedExtraParams`, so adding session arguments only to the later apply call does not make `prepareExtraParams` see them on that path. Thread the fields through the runtime-plan resolver or prepare after session context is available.
  - confidence: 0.88
- **[P2] Keep inherited provider hook contexts populated:** `src/plugins/types.ts:465-476`
  - body: Adding fields to `ProviderPrepareExtraParamsContext` also advertises them on transport and stream-wrapper contexts derived from that type, but the patch only supplies them to the prepare hook. Either split the base context or populate the fields everywhere the public types expose them.
  - confidence: 0.84
