## Summary

Keep open. The PR targets a real provider API gap, but it is not merge-ready against current main because the new typed session fields would be visible on derived provider hook contexts without being populated there, and current main now resolves prepared extra params through the runtime-plan path before `applyExtraParamsToAgent`.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and threads them from embedded attempts into provider extra-param preparation.

## Best Possible Solution

Expose session identity through one coherent provider hook contract, with fields populated everywhere the public types advertise them, focused contract tests for the runtime-plan/embedded paths, and provider SDK docs/changelog updates if the public API expands.

## Reproduction Assessment

Yes for the PR defect: static inspection of current main shows the added base type would expose fields on transport and wrap hooks while the current context builders do not populate them. The original LiteLLM spend-log attribution use case is plausible from the PR body but was not live-tested in this read-only review.

## Solution Assessment

No, not as proposed. Passing session fields to only `prepareExtraParams` is too narrow for the inherited public hook types and current runtime-plan flow; the safer fix is either to scope the type narrowly to the prepare hook or thread the fields consistently through all advertised hook contexts with tests.
