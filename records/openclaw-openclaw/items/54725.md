## Summary

Keep open. The provider-session attribution direction is useful, but this PR is not merge-ready: it makes prepared extra params session-dependent without cache isolation, misses the runtime-plan preparation path, advertises fields on derived hook contexts it does not populate, and lacks after-fix real behavior proof.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to provider extra-param preparation and passes them from the embedded run path so provider plugins can tag downstream requests by session.

## Best Possible Solution

Define a privacy-safe provider request metadata contract, thread session identity through every hook that advertises it, avoid cross-session prepared-param caching, and cover LiteLLM/OpenRouter attribution with focused tests and docs.

## Reproduction Assessment

Yes for the patch defects: source inspection shows cached prepared extra params without session identity and a runtime-plan preparation path that would not receive the new fields. The original LiteLLM spend-log behavior was not live-reproduced.

## Solution Assessment

No. The direction is reasonable, but the diff is too narrow for the public provider API it changes; the safer solution is a reviewed request/session metadata contract with cache isolation and runtime-plan coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add session identity to the prepared-param cache key:** `src/agents/pi-embedded-runner/extra-params.ts:159-160`
  - body: Once `prepareExtraParams` can read `ctx.sessionKey` or `ctx.sessionId`, the existing cache can reuse one session's returned `user` value for another session with the same provider/model/config. Include the identity in the cache key or avoid caching session-dependent prepared params.
  - confidence: 0.9
- **[P2] Thread session fields through runtime-plan preparation:** `src/agents/pi-embedded-runner/run/attempt.ts:2299-2300`
  - body: Current main prepares `preparedRuntimeExtraParams` through the runtime plan before `applyExtraParamsToAgent`, and the apply path skips preparation when that value is supplied. Passing fields only to the apply call leaves this path without session context.
  - confidence: 0.86
- **[P2] Keep exposed provider hook contexts consistent:** `src/plugins/types.ts:465-476`
  - body: `ProviderExtraParamsForTransportContext` and `ProviderWrapStreamFnContext` derive from `ProviderPrepareExtraParamsContext`, so adding fields there advertises them on hooks this patch does not populate. Split the types or pass the values to every exposed context.
  - confidence: 0.84
