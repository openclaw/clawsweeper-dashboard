## Summary

Keep open. The requested provider-session metadata remains unsolved on current main, but this PR is merge-conflicting, lacks contributor real-behavior proof, and would introduce cache and public-hook consistency problems if merged as-is.

## What This Changes

This branch adds optional sessionKey/sessionId to ProviderPrepareExtraParamsContext and threads them through embedded-runner extra-param preparation at one attempt call site.

## Best Possible Solution

Define one privacy-conscious provider request metadata contract, thread it through every advertised hook path consistently, make session-dependent prepared params cache-safe, and cover it with focused hook-context and payload tests plus docs.

## Reproduction Assessment

Yes for the PR defects by source inspection: current main shows a session-agnostic prepared-param cache, a runtime-plan preparation path that would not receive the new fields, and inherited provider hook contexts that the PR would advertise without consistent population. The original downstream LiteLLM spend-log problem was not reproduced live.

## Solution Assessment

No. The direction is useful, but threading raw session fields only through this prepare hook is not the narrowest mergeable solution until cache isolation, runtime-plan preparation, inherited hook consistency, privacy/API scope, and real provider proof are resolved together.
