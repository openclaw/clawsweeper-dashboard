## Summary

Keep open: the provider hook seam is still missing on current main, but this PR is not ready against the current runner because session-dependent prepared params would either miss the main precompute path or be cached without session identity.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and threads them from the embedded runner toward provider extra-param hooks.

## Best Possible Solution

Land a rebased additive provider-hook API that exposes session identity consistently at the first prepared-param resolution point, prevents cross-session cache reuse, adds regression coverage and a changelog entry, and coordinates with #63901/#73492 if maintainers want one broader metadata seam.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main resolves prepared extra params before the PR's updated apply call and caches prepared results without session identity. No runtime test was run because this was a read-only review.

## Solution Assessment

No. The proposed seam is plausible, but the current patch is not the best mergeable form until it is rebased to current main, threads the fields into the first `resolvePreparedExtraParams` call, fixes cache scoping, and adds coverage/release notes.
