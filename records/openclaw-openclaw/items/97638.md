## Label Justifications

- `P3`: This is an opt-in operational-cost feature request for custom sandbox deployments, not a current regression, crash, data-loss, or security bypass report.
- `impact:other`: The reported impact is redundant disk usage and session startup overhead from sandbox skill copies, which is maintainer-visible but outside the more specific impact label taxonomy.

## Summary

Keep open for maintainer product review: current main still always requests sandbox skill materialization and has no skipSkillsSync config, but the requested opt-out changes config surface and sandbox skill-read invariants, so it is not a safe automatic repair candidate.

## What This Changes

Request to add an opt-in config flag that skips copying eligible skills into sandbox workspaces for custom sandbox backends that preinstall skills.

## Best Possible Solution

Have sandbox/agents owners decide whether this belongs as an explicit sandbox config field or a backend capability, then require docs, schema, and regression coverage that preserves readable skill paths for default sandboxes.

## Reproduction Assessment

Yes, at source level for the current behavior: current main calls sandbox skill sync for both non-rw and rw sandbox workspace layouts, and tests assert those calls. The exact scale impact was not independently measured in this review.

## Solution Assessment

Unclear: a global agents.defaults.skipSkillsSync flag is plausible, but a backend-scoped capability or sandbox-owned config may be the safer owner boundary because current sync protects readable sandbox skill paths.
