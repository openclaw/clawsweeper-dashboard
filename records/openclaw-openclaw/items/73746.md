Full review comments:

- **[P1] Reconcile new statuses with persisted parsing:** `src/tasks/task-registry.types.ts:7-8`
  - body: Current main restores `task_runs.status` through the strict `parseTaskStatus` set in `task-registry.types.ts`. This PR introduces persisted `awaiting_approval` and `waiting_external` states, so a rebased version must update that parser and restore tests; otherwise blocked tasks written by the new runtime can fail to load after restart with `Invalid persisted task status`.
  - confidence: 0.86
- **[P2] Preserve blocked terminal outcomes in summaries:** `src/tasks/task-status.ts:222-228`
  - body: Existing tasks use `status: "succeeded"` plus `terminalOutcome: "blocked"` to mean the task needs follow-up, and current terminal/TaskFlow code treats that separately from completion. The new operational summary handles every `succeeded` task as `finished · completed` with `next: no action`, so CLI and auto-reply summaries can tell users no action is needed for a blocked task.
  - confidence: 0.88
- **[P2] Avoid announcing resumed tasks as started:** `src/tasks/task-status.ts:341-342`
  - body: When approval resolves, `task-registry.ts` moves the task from `awaiting_approval` back to `running`, but `buildTaskLifecycleEventFromRecord` calls `buildTaskLifecycleEvent(task)` without the previous status. That path treats any running task with no previous status as `task.started`, so a resumed approval wait can emit a second "Background task started" message instead of a resumed/progress update.
  - confidence: 0.78
