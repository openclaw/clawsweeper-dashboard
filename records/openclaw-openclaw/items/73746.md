## Summary

Keep open. Current main does not implement the new intermediate task states, and the PR is still useful, but the current patch is not merge-ready because PR head still drops the new statuses during task creation and lacks the required docs/changelog follow-through; the earlier Greptile transition concerns appear addressed in the latest head.

## What This Changes

The PR adds `awaiting_approval` and `waiting_external` task states, treats them as active across task registry/flow/audit/maintenance/CLI/auto-reply rendering, and adds shared operational/lifecycle formatting plus focused task tests.

## Best Possible Solution

Make the two intermediate statuses first-class wherever task status values are parsed, normalized, displayed, audited, documented, and announced, while preserving the TaskFlow compatibility projection to `running`; then land this PR or an equivalent replacement after focused task tests, docs checks, changelog audit, and the changed gate pass.

## Reproduction Assessment

Yes. On PR head, call `createTaskRecord` with `status: "awaiting_approval"` or `status: "waiting_external"`; the call reaches `normalizeTaskStatus`, which still omits both values and returns `queued`.

## Solution Assessment

No, not as currently proposed. The product direction is maintainable and the transition fixes appear to have landed, but the implementation must update the canonical normalizer and add regression coverage plus user-facing docs and changelog before merge.
