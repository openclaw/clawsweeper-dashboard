## Summary

Keep open. Current main still has the old task-status model, and this PR is the active implementation path, but PR head still loses the new statuses when they enter through the normalized create/update paths. The author did address the Greptile transition-review items in 7f755fe, so the right outcome is a focused repair rather than closing or superseding the PR.

## What This Changes

The branch adds `awaiting_approval` and `waiting_external` task states plus lifecycle/operational formatting across task registry, Task Flow, CLI/auto-reply rendering, audit/maintenance, and focused task tests.

## Best Possible Solution

Make the intermediate task states first-class everywhere task statuses are parsed, normalized, stored, displayed, audited, documented, and announced, while preserving the Task Flow compatibility projection to `running`. Then land this PR or an equivalent replacement with focused task regression coverage and the required docs/changelog alignment.

## Reproduction Assessment

Yes. On PR head, call `createTaskRecord` or the run-scoped update path with `status: "awaiting_approval"` or `status: "waiting_external"`; `normalizeTaskStatus` still falls back to `queued`, which is visible from the source path without needing a broad test run.

## Solution Assessment

No, not in its current form. The direction is maintainable, but the safer implementation is to update or centralize status normalization, add create/update-path regression coverage, and align the public docs and changelog before merge.
