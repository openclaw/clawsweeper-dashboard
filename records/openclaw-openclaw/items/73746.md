## Summary

Keep open. Current main still has the old coarse task states, so the PR is not obsolete, but the PR head remains merge-blocked because the newly added statuses are not preserved by task status normalization and the user-facing docs/changelog updates are still missing.

## What This Changes

The PR adds `awaiting_approval` and `waiting_external` task statuses, treats them as active across task registry/flow/audit/maintenance/CLI/auto-reply rendering, and adds lifecycle/operational formatting plus focused task tests.

## Best Possible Solution

Make the two intermediate statuses first-class anywhere task status values are parsed, normalized, displayed, audited, documented, and announced, while preserving the TaskFlow compatibility projection to `running`; then land this PR or an equivalent replacement after targeted task tests, docs checks, changelog audit, and the changed gate pass.

## Reproduction Assessment

Yes. On PR head, call `createTaskRecord` with `status: "awaiting_approval"` or `status: "waiting_external"`; the call reaches `normalizeTaskStatus`, which still omits both values and returns `queued`.

## Solution Assessment

No, not as currently proposed. The product direction is maintainable, and the Greptile transition concerns are addressed, but the implementation should update the canonical normalizer and regression coverage plus user-facing docs and changelog before merge.
