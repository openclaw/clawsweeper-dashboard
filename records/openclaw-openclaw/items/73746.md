- **[P2] Preserve the new statuses in normalization:** `src/tasks/task-registry.types.ts:7-8`
  - body: Adding `awaiting_approval` and `waiting_external` to `TaskStatus` does not make them persistable while `normalizeTaskStatus` still allow-lists only the old values. The create and run-scoped update paths normalize before storing, so explicit uses of the new states fall back to `queued`.
  - confidence: 0.97

  - body: When a progress summary sanitizes to empty, this lifecycle path still builds a `task.progressed` event with stage-only detail and emits a background-task update. That regresses the existing internal-context redaction behavior, which returns `null` for sanitized-empty progress text.
  - confidence: 0.93
- **[P2] Pass the flow revision when cancelling:** `src/tasks/task-registry.test.ts:2293-2297`
  - body: The added managed-flow test calls `requestFlowCancel` without the required `expectedRevision`. Current main requires that revision, so the test type check cannot pass until it supplies the created flow's revision.
  - confidence: 0.9
- **[P3] Document the new task statuses:** `src/commands/tasks.ts:97-98`
  - body: The PR exposes `awaiting_approval` and `waiting_external` in CLI and auto-reply output, but the background task docs still describe only `queued`, `running`, and terminal states. Users need the new public status values documented with the lifecycle change.
  - confidence: 0.84
