## Summary

Keep open: the reported stable Weixin failure is credible, and current main still has the exact `message.action` unsupported-action guard plus an isolated-cron announce path that can expose the message tool; the direct fallback uses durable outbound delivery, so the remaining work is to isolate which current path still reaches action dispatch before changing core or plugin contracts.

## What This Changes

The issue reports isolated cron `delivery.mode=announce` to `openclaw-weixin` failing because delivery reaches `message.action=send` instead of the channel plugin's outbound text send path.

## Best Possible Solution

Make isolated cron announce delivery consistently reach durable outbound text delivery for action-less external channels, and add focused regression proof that a failed message-tool action cannot suppress the final announce fallback.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully for current main: the stable-release repro is concrete and the exact error source is present, but I did not run a live Weixin cron job and the current-main path that still reaches action dispatch needs a trace.

## Solution Assessment

Is this the best way to solve the issue? Unclear: using durable outbound text delivery is the right final-delivery direction, but current main already does that for direct fallback, so the best fix must first identify whether message-tool action routing or plugin action support is the real remaining boundary.
