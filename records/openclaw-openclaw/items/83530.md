- `P1`: The issue describes active user-facing Telegram message loss in the latest release line, where completed assistant output is partially delivered.
- `impact:message-loss`: The reported failure drops the suffix of a completed assistant reply before it reaches Telegram.

## Summary

Current main already fixes the reported prefix-only final payload path in the embedded runner, and the existing regression test covers the exact one-streamed-prefix versus complete-final-answer shape from the report.

## What This Changes

The issue reports that Telegram final delivery can send a strict prefix of the final assistant text even though the complete final answer exists in OpenClaw run metadata.

## Best Possible Solution

Keep the runner-level canonical final-answer selection on main and ship it; reopen or file a new report only if prefix-only truncation recurs on a build containing b278098a7c.

## Reproduction Assessment

Yes, at source level. v2026.5.12 only preferred the canonical final answer when more than one streamed assistant text existed, while the report's one-prefix-payload shape would keep the stale prefix; current main adds a focused regression test for that case.

## Solution Assessment

Yes. Fixing this in runner payload construction is narrower and more maintainable than a Telegram-only delivery patch because it selects the canonical final assistant text before any channel transport receives the payload.
