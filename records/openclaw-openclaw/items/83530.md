## Summary

Keep open. The report is a concrete Telegram message-loss bug: current main has related canonical-final and ellipsis-truncation safeguards, but I did not find a guard that repairs the reported clean strict-prefix final payload when `runResult.meta.finalAssistantVisibleText` already has the complete answer.

## What This Changes

The issue reports Telegram final delivery sometimes sending a strict prefix of the complete assistant response instead of the canonical final assistant text.

## Best Possible Solution

Add a narrow final-delivery guard that promotes the canonical final assistant text only for one safe text-only final payload that is a strict prefix of the canonical text, with regression coverage and Telegram proof for the streaming final path.

## Reproduction Assessment

No live Telegram reproduction was run. Source inspection gives a medium-confidence reproduction boundary: current main only repairs ellipsis-style truncation and does not compare a clean strict-prefix payload against canonical final assistant metadata before Telegram delivery.

## Solution Assessment

Yes, the reporter's mitigation shape is likely the narrowest maintainable fix if implemented at the final payload handoff and guarded to text-only, non-error, non-reasoning, non-media, single-payload deliveries.
