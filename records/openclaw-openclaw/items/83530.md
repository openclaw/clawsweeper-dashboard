## Label Justifications

- `P1`: The report describes a real Telegram channel workflow delivering only a prefix of an otherwise complete assistant response in the latest release.
- `impact:message-loss`: The affected behavior silently drops the tail of a user-visible Telegram reply while the complete response exists internally.

## Summary

Close as implemented on current main. The latest release source still had the single-prefix gap, but current main now prefers the structured current-turn final assistant answer over streamed preview fragments and has a regression test for the exact incomplete-preview shape.

## What This Changes

The issue reports Telegram final replies sometimes using an intermediate streaming prefix even though run metadata contains the complete assistant response.

## Best Possible Solution

Keep the current-main payload assembly fix and let the next release carry it; the channel-specific prefix replacement workaround should not be needed once users are on a build containing 7d1317634e5b814d859636338829e2699136449d.

## Reproduction Assessment

Yes, source-level against the reported release: `v2026.5.12` only preferred the canonical final answer when more than one streamed text existed, so a single streamed prefix could remain the final payload. Current main adds a regression test for that exact incomplete-preview case and passes the longer final answer through payload assembly.

## Solution Assessment

Yes. Fixing the shared embedded-run payload assembly is better than a Telegram-only delivery patch because all final delivery paths receive the canonical current-turn assistant answer before channel formatting or streaming finalization.
