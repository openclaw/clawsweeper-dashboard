## Summary

Keep open: this PR addresses a real manual cron active-job gap on current main, has sufficient live terminal proof, and I found no blocking correctness or security issues.

## What This Changes

The PR marks prepared manual cron runs active until `finishPreparedManualRun` completes and adds regression tests for successful and throwing manual runs.

## Best Possible Solution

Land this PR, or an equivalent patch, so manual cron runs use the same active-job ownership contract as scheduled runs; keep the linked issue open until merge.

## Reproduction Assessment

Yes. Source inspection shows the current manual-run producer path creates a cron task without marking the job active, and the PR body supplies live before/after sqlite and gateway-log proof; I did not run local tests because this was a read-only review.

## Solution Assessment

Yes. Reusing the existing `markCronJobActive`/`clearCronJobActive` contract on the missing manual-run path is the narrowest maintainable fix and avoids changing durable recovery or public API behavior.
