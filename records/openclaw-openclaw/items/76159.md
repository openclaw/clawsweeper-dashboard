## Summary

Keep open: current main still has only an internal embedded-runner silent-empty opt-in; cron job payloads/schema/docs do not expose the requested per-job policy, and the fresh v2026.5.12 production comment confirms the false-positive still matters.

## What This Changes

The issue requests an opt-in cron job flag that treats clean empty assistant stops as successful intentional silent runs.

## Best Possible Solution

Make an explicit maintainer decision on the cron success contract, then add a documented payload/schema/executor policy that forwards to the existing runner seam if intentional empty stops are supported.

## Reproduction Assessment

Yes, source-level. Current main can be traced from cron payload/schema through the isolated executor to `runEmbeddedPiAgent`, and the requested silent-empty policy is not exposed or forwarded.

## Solution Assessment

Unclear until maintainer signoff. Reusing the existing runner gate is the narrow implementation seam, but adding a public cron flag changes automation success semantics and should be a product/API decision.
