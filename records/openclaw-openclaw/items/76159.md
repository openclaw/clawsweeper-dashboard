## Summary

Keep open. Current main has the embedded-runner mechanism to accept clean empty assistant stops, but cron still has no public per-job field and does not forward any job-derived silent-empty policy into that runner option.

## What This Changes

The issue requests an opt-in per-cron setting that treats clean zero-payload assistant stops as successful silent runs instead of cron errors.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through normalization, gateway protocol schema, CLI/tool creation and edit surfaces, and the isolated executor into the existing runner gate while preserving default failure behavior.

## Reproduction Assessment

Yes by source inspection: current main can accept clean empty assistant replies only when the caller passes `allowEmptyAssistantReplyAsSilent`, and isolated cron currently has no payload field or forwarding path for that option. I did not run a live cron job because this review was required to stay read-only.

## Solution Assessment

Yes directionally: reusing the existing runner gate behind an opt-in cron policy is the narrow maintainable direction. The exact public field placement and naming still need maintainer signoff before implementation.
