## Summary

Keep open: current main and the latest release still lack the requested per-job silent-stop cron contract, and the open related PR only covers the narrower `delivery.mode="none"` handoff while remaining conflicting.

## What This Changes

The issue requests an opt-in cron job flag that treats intentional empty assistant stops as successful silent runs instead of cron errors with backoff.

## Best Possible Solution

Define one documented cron silent-success contract, reuse the existing runner primitive where safe, and add schema, type, docs, and regression coverage for both accepted silence and default incomplete-turn protection.

## Reproduction Assessment

Yes. Source inspection shows current main has no accepted `acceptSilentStop` field and does not pass the existing silent-empty option from cron into the embedded runner; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers should first decide whether the supported contract should be a new flag, `delivery.mode="none"`, command cron jobs for deterministic scripts, or a narrower combination.
