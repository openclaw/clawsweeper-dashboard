## Summary

Keep open. Current main has an internal runner gate for clean empty assistant stops, but isolated cron jobs still do not expose or forward a per-job silent-empty policy; current docs and related #68452 only cover exact `NO_REPLY` silent-token paths.

## What This Changes

The issue requests an opt-in per-cron setting that treats clean `stopReason=stop` runs with zero assistant payloads as successful silent jobs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through payload types, normalization/update surfaces, gateway protocol schema, executor forwarding, and regression coverage while preserving the current default failure behavior.

## Reproduction Assessment

Yes by source inspection. The runner can accept clean empty stops only when `allowEmptyAssistantReplyAsSilent` is set, current tests prove the no-flag path still errors, and isolated cron never passes a job-derived equivalent.

## Solution Assessment

Yes directionally, but not ready for automatic implementation. Reusing the existing runner gate behind an opt-in cron policy is the narrow maintainable path, while the field name and public protocol semantics need maintainer approval.
