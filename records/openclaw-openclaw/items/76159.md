## Summary

Keep open. Current main has the embedded-runner switch needed to accept clean empty assistant stops, but isolated cron jobs still do not expose or forward a per-job silent-empty policy; the shipped #68452 work covers exact `NO_REPLY` tool-result paths, not zero-payload assistant stops without that sentinel.

## What This Changes

Request an opt-in cron job flag that treats `stopReason=stop` with zero payloads as a successful silent cron run.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through normalization, gateway protocol schema, CLI/tool creation surfaces, and the isolated executor into the existing runner gate while preserving the default failure behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: the runner can accept clean empty stops only with `allowEmptyAssistantReplyAsSilent`, isolated cron never passes that option, and cron converts fatal embedded error payloads into job errors and counter increments.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: reusing the existing runner gate behind an opt-in cron payload policy is the narrow maintainable path, but the public API shape needs maintainer signoff before implementation.
