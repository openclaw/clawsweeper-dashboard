## Summary

Keep open. Current main has a runner-level opt-in for clean empty assistant stops, but cron jobs still lack a public per-job field and do not forward that opt-in into isolated agent execution.

## What This Changes

Request an opt-in cron job flag so clean `stopReason=stop` turns with zero payloads count as successful silent runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through cron normalization, Gateway protocol schema, CLI/tool surfaces, and the isolated executor into the existing runner gate while preserving the current default failure behavior.

## Reproduction Assessment

Yes. Source inspection shows clean empty assistant stops are accepted only when `allowEmptyAssistantReplyAsSilent` is true, while isolated cron neither exposes nor forwards that flag and cron converts the resulting error payload into job error/backoff state.

## Solution Assessment

Yes, directionally. Reusing the existing runner gate behind an opt-in cron payload field is the narrow maintainable path, but the field name and public contract need maintainer approval.
