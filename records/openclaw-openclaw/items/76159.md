## Summary

Keep open. Current main still has only an internal embedded-runner switch for clean empty assistant stops; isolated cron jobs do not expose or forward a per-job silent-empty contract, so this remains a product/config feature request.

## What This Changes

Request adding an opt-in per-cron job flag so clean stopReason=stop, zero-payload agent turns count as successful silent runs instead of errors/backoff.

## Best Possible Solution

Add a documented opt-in cron agentTurn silent-empty policy that flows through types, normalization, protocol schema, CLI/tool surfaces, docs, and executor forwarding into the existing runner gate while keeping the default failure behavior unchanged.

## Reproduction Assessment

Yes. Source inspection shows clean empty assistant stops are accepted only when allowEmptyAssistantReplyAsSilent is true, while isolated cron neither exposes nor forwards that flag and cron still turns fatal empty/error payloads into job errors/backoff.

## Solution Assessment

Yes, directionally, with maintainer signoff. Reusing the existing runner gate behind an opt-in cron field is the narrow maintainable path, but the public contract shape needs approval first.
