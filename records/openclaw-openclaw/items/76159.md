## Summary

Keep open. Current main still only accepts clean empty assistant stops when a caller passes `allowEmptyAssistantReplyAsSilent`; cron `agentTurn` jobs neither expose the requested `acceptSilentStop` policy nor forward an equivalent flag, and the narrower `delivery.mode: "none"` subcase is already tracked separately at https://github.com/openclaw/openclaw/issues/79069.

## What This Changes

Request adds an opt-in cron job policy that treats clean zero-payload `stop` turns as successful silent runs instead of errors that trigger cron backoff.

## Best Possible Solution

Define a documented cron `agentTurn` silent-empty policy, or explicitly choose existing delivery-mode semantics instead, then thread that approved policy through normalization, schema/CLI surfaces, and the isolated executor into the runner flag while preserving default failure behavior.

## Reproduction Assessment

Yes, by source inspection: the embedded runner can accept clean empty stops only when `allowEmptyAssistantReplyAsSilent` is passed, and the cron path does not expose or forward that policy. I did not run a live cron/model reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer signoff: reusing the existing runner gate is the narrow implementation path, but adding `acceptSilentStop` as public cron configuration is a product/API decision. The narrower `delivery.mode: "none"` propagation path is already tracked at https://github.com/openclaw/openclaw/issues/79069.
