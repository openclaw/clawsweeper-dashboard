## Summary

Keep open: current main and v2026.6.8 still lack the requested per-job `acceptSilentStop` cron contract, and cron still does not pass the existing silent-empty runner option. Related issues and PRs cover narrower or competing fixes, so the broader product/config decision remains open.

## What This Changes

The issue requests a per-job cron option that treats intentionally empty `stopReason=stop` agent turns as successful silent runs instead of incomplete/error runs.

## Best Possible Solution

Define one documented cron silent-success contract, thread it through the existing runner primitive for both CLI and embedded cron where safe, and add schema/types/docs plus regression coverage that preserves real empty-response failure protection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has no `acceptSilentStop` schema/type field, cron does not pass `allowEmptyAssistantReplyAsSilent`, and empty cron output still reaches error-classification paths; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers should first choose whether the supported contract is an explicit flag, `delivery.mode: "none"`, a cron default, or command cron for deterministic script workloads.
