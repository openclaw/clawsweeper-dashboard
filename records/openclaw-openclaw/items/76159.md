## Summary

Keep open: current main still lacks the requested per-job `acceptSilentStop` cron configuration, and the existing runner opt-in is not exposed through cron payload/types/schema. The open related work only covers the narrower `delivery.mode: "none"` handoff, while the latest production comment confirms the broader silent-cron false-positive still matters on v2026.5.12.

## What This Changes

The issue requests an opt-in cron job flag that treats clean `stopReason="stop"` runs with zero payloads as successful intentional silence.

## Best Possible Solution

Land the narrow no-delivery handoff if desired, then make an explicit maintainer decision on a documented `acceptSilentStop` or equivalent cron policy for jobs that may intentionally emit nothing.

## Reproduction Assessment

Yes, source-level. Current main has the embedded-runner opt-in but the cron payload/schema/executor do not expose or forward the requested silent-stop policy, and the issue includes a concrete production report on v2026.5.12.

## Solution Assessment

Unclear until maintainer signoff. Reusing the existing runner gate is the right implementation seam, but adding a public `acceptSilentStop` cron flag changes automation success semantics and should not be treated as a mechanical bug fix.
