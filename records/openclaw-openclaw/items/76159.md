## Summary

Keep open. Current main has a runner-level opt-in for clean empty assistant stops, but cron job payloads, protocol schema, normalization, and the isolated executor still do not expose or forward a per-job silent-empty policy; the closed #68452 path only covers exact `NO_REPLY` tool-result quiet successes.

## What This Changes

Request an opt-in cron job setting that treats `stopReason=stop` with zero payloads as a successful silent run instead of an incomplete-turn error.

## Best Possible Solution

Add a documented, default-off cron `agentTurn` silent-empty policy that normalizes through create/edit/API/tool surfaces and forwards to `allowEmptyAssistantReplyAsSilent` for isolated execution without changing the default failure path.

## Reproduction Assessment

Yes, by source inspection. Clean empty assistant stops are accepted only when `allowEmptyAssistantReplyAsSilent` is true, while isolated cron neither stores nor forwards a per-job equivalent and cron converts resulting error payloads into error/backoff state.

## Solution Assessment

Yes, directionally. Reusing the existing runner gate behind an opt-in cron field is the narrow maintainable path, but the field name and API/docs shape need maintainer approval first.
