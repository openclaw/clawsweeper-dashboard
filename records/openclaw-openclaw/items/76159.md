## Summary

Keep open: current main and v2026.6.10 still lack the requested per-job silent-stop cron contract, and the cron executor still does not pass the existing silent-empty runner opt-in. Related open PRs cover overlapping all-cron or delivery-mode fixes, but they do not settle this broader public config/product decision.

## What This Changes

The issue asks for an opt-in per-cron-job `acceptSilentStop` flag so empty assistant replies can be treated as successful silent cron runs.

## Best Possible Solution

Define one documented cron silent-success policy, reuse the existing runner opt-in at the cron executor boundary where appropriate, and update schema/types/CLI/docs/tests only for the chosen public contract.

## Reproduction Assessment

Yes, at source level. Current main has no accepted per-job silent-stop field, cron does not pass `allowEmptyAssistantReplyAsSilent`, and empty cron output still reaches the existing error-classification paths.

## Solution Assessment

Unclear. A per-job flag is plausible, but current code already offers command payloads for deterministic scripts and open related PRs propose executor-level silent-empty policy, so maintainers need to choose the stable product contract first.
