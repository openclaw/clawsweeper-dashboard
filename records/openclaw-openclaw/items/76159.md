## Label Justifications

- `P2`: The issue describes concrete false error/backoff behavior in cron automation, but the fix is a bounded product/config decision rather than an emergency runtime outage.
- `impact:other`: The affected surface is cron run status, backoff, and failure-alert noise for intentionally silent jobs, which is maintainer-visible but outside the specific owned impact labels.

## Summary

Keep open: current main still lacks the requested per-job `acceptSilentStop` cron contract, the latest release lacks it too, and the related open PR only covers the narrower `delivery.mode="none"` path for another issue.

## What This Changes

The issue requests a per-job cron opt-in that treats intentional empty assistant stops as successful runs without incrementing error backoff or sending failure alerts.

## Best Possible Solution

Define one documented opt-in cron silent-success contract, reuse the existing runner primitive where it is safe, and add schema/type/docs/tests for both accepted silence and default incomplete-turn protection.

## Reproduction Assessment

Yes. Source inspection shows current main has no `acceptSilentStop` field, does not pass the existing silent-empty runner option from cron, and still increments cron error counters when the run is classified as an error; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers should choose between a broad `acceptSilentStop` contract, making `delivery.mode="none"` the supported opt-in, or routing deterministic script cases to command cron jobs.
