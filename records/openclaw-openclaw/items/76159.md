## Summary

Keep open: current main has the lower-level silent-empty runner behavior, but cron jobs cannot opt into it today, so the reported cron backoff/error path remains a real missing feature.

## What This Changes

The issue requests a per-cron-job opt-in flag that treats intentional `stopReason=stop` runs with no reply payloads as successful silent runs.

## Best Possible Solution

Add a backward-compatible boolean on cron `agentTurn` payloads, expose it through RPC/CLI/tool/docs, and wire it to the existing empty-as-silent runner path without changing the default incomplete-turn behavior.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is available: an isolated cron agent turn that returns an empty clean assistant stop reaches cron without `allowEmptyAssistantReplyAsSilent`, so the existing incomplete-turn path can produce an error payload that cron records as an execution error.

## Solution Assessment

Yes, the requested opt-in is the narrow maintainable direction because OpenClaw already has a guarded lower-level empty-as-silent seam; the repair should reuse that seam rather than weakening global empty-response detection.
