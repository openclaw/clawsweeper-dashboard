## Summary

Keep open. Current main has a runner-level gate for clean empty assistant stops, but isolated cron jobs do not expose or forward any per-job silent-empty policy; the related #68452 work only shipped exact `NO_REPLY` tool-result handling, so this requested zero-payload cron behavior remains a product/API decision.

## What This Changes

Add an opt-in per-job cron flag that treats clean `stopReason=stop` zero-payload isolated agent turns as successful silent runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that normalizes, validates, persists, forwards to the existing runner gate, and preserves default failure behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: clean empty assistant stops are only accepted when `allowEmptyAssistantReplyAsSilent` is passed, and isolated cron never passes a job-derived equivalent.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: reusing the existing runner gate behind an opt-in cron policy is the narrow maintainable path, but the public API shape needs maintainer approval.
