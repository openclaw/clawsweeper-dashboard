## Summary

Keep open. Current main has an internal embedded-runner option for clean empty assistant stops, but isolated cron jobs still have no per-job payload/protocol field and do not forward that option; the closed #68452 path only covers exact `NO_REPLY` tool-result silence, not a zero-payload assistant stop.

## What This Changes

Request an opt-in per-job cron flag such as `acceptSilentStop` so intentional `stopReason="stop"` runs with no payloads are recorded as successful instead of incomplete-turn errors with backoff.

## Best Possible Solution

Add a documented opt-in `agentTurn` silent-empty policy that flows through cron payload types, gateway schema, creation/edit surfaces, docs, and the isolated executor into the existing runner gate while preserving the current default error behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by source inspection: clean empty assistant stops are accepted only behind `allowEmptyAssistantReplyAsSilent`, and isolated cron never exposes or forwards that option, so the existing incomplete-turn path still applies.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an exact API, but directionally yes: reusing the existing runner gate behind a per-job cron policy is the narrow technical path, while the public flag name and scope need maintainer signoff.
