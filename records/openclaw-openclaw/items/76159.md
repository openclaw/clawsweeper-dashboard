## Summary

Keep open: current main still has no per-job `acceptSilentStop` contract and does not thread the existing silent-empty runner option through isolated cron agent turns, while the related open PR is narrower, conflicting, and unmerged.

## What This Changes

Request an opt-in cron-job contract that treats intentional no-output `stopReason="stop"` agent runs as successful instead of cron errors.

## Best Possible Solution

Define one public cron silent-success contract, then align schema, runner classification, docs, and focused regression tests around it without weakening normal incomplete-turn detection.

## Reproduction Assessment

Yes at source level: current cron execution can call the embedded runner without the silent-empty option, and error outcomes increment `consecutiveErrors`. I did not run a live cron/provider reproduction in this read-only review.

## Solution Assessment

Unclear. The requested flag is plausible, but maintainers should choose the public contract because existing docs already define `delivery.mode="none"`, exact `NO_REPLY` silence, and command cron jobs, while the runner deliberately guards ambiguous post-tool empty stops.
