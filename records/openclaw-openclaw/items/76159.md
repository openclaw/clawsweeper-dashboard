## Summary

Keep open. Current main has a reusable embedded-runner gate for clean empty assistant stops, but isolated cron jobs still cannot configure or forward that policy, and the shipped related fix only covers exact `NO_REPLY` tool-result paths.

## What This Changes

Request an opt-in per-job cron flag that treats clean `stopReason=stop` zero-payload `agentTurn` runs as successful intentional silence.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through cron types, protocol validation, creation/editing surfaces, and the isolated executor into the existing runner gate while preserving the default error behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: the runner accepts clean empty stops only with `allowEmptyAssistantReplyAsSilent`, cron cannot set or forward that option, and tests preserve the default error path.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: reusing the existing runner gate behind an opt-in cron policy is the narrow implementation path, but the public field shape needs maintainer approval.
