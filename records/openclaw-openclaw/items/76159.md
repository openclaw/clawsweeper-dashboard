## Summary

Keep open: current main and the latest release still lack the requested per-job `acceptSilentStop` contract, and the remaining decision is broader than the open `delivery.mode="none"` repair path.

## What This Changes

The issue requests an opt-in cron job flag that treats intentional empty assistant stops as successful silent cron runs instead of incomplete-turn errors.

## Best Possible Solution

Define a documented opt-in cron silent-success contract that threads the existing runner option only for approved job modes while preserving default incomplete-turn protection.

## Reproduction Assessment

Yes. Source inspection shows current main has no `acceptSilentStop` field, the API schema rejects unknown cron fields, and the cron executor does not pass the existing silent-empty runner option; I did not execute a live cron in this read-only review.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers need to choose between this broader contract, the narrower `delivery.mode="none"` fix path, or directing deterministic no-output work to command cron payloads.
