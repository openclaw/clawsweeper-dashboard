## Summary

Keep open: current main still lacks a public cron-job flag or equivalent policy that forwards intentional empty assistant stops into the embedded runner’s silent-empty path, and the latest issue comment reports the same false-positive error behavior on v2026.5.12.

## What This Changes

Requests an opt-in cron job flag that treats an empty assistant stop with no payloads and no tool errors as a successful silent run.

## Best Possible Solution

Define and document a maintainer-approved cron silent-empty policy, then wire it through cron types, protocol schema, executor parameters, runner tests, cron tests, and docs.

## Reproduction Assessment

Yes, source-level for the current limitation: cron payload/types/schema cannot carry the requested flag, and the executor does not forward the existing runner opt-in. The production comments provide a concrete real-world trigger, but I did not run a live cron/model reproduction.

## Solution Assessment

Unclear until maintainer signoff. Reusing `allowEmptyAssistantReplyAsSilent` is the narrow implementation seam, but exposing a public cron flag changes success semantics and should be decided intentionally.
