## Summary

Keep open. Current main has the runner-level silent-empty safety gate, but cron jobs still do not expose or forward a per-job `acceptSilentStop`-style contract, and the shipped `NO_REPLY` fix only covers explicit silent-token outcomes rather than clean zero-payload assistant stops.

## What This Changes

The issue requests an opt-in per-cron-job flag that treats clean `stopReason="stop"` runs with zero payloads as successful silent cron runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` contract that forwards to the existing runner silent-empty gate, records the run as ok/no-delivery, and leaves default empty-response, timeout, provider-error, tool-error, and side-effect behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an isolated cron `agentTurn` ending in a clean empty assistant stop reaches `runEmbeddedPiAgent` without the silent-empty flag, so the runner can still produce the incomplete-turn error that cron then counts for backoff.

## Solution Assessment

Yes. Reusing `allowEmptyAssistantReplyAsSilent` behind an opt-in cron payload field is the narrowest maintainable direction because the runner helper already carries the safety gates for aborts, timeouts, tool errors, committed messaging delivery, and side effects.

For isolated embedded cron runs only, forward the flag to `runEmbeddedPiAgent` as `allowEmptyAssistantReplyAsSilent` so a clean empty assistant stop becomes the existing silent `NO_REPLY` outcome, records the cron run as `ok`, resets `consecutiveErrors`, suppresses user delivery, and still writes run history. Do not change default empty-response behavior, direct/user chat behavior, timeout/abort/provider-error/tool-error handling, committed message-tool delivery handling, or side-effect replay guards.
