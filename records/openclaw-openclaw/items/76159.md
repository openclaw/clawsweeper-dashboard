## Summary

Keep open. Current main has the embedded-runner switch needed to accept clean empty assistant stops, but isolated cron jobs still do not expose or forward a per-job silent-empty policy, so the requested public cron behavior remains unimplemented.

## What This Changes

Request an opt-in cron `acceptSilentStop` job flag that treats clean zero-payload assistant stops as successful silent runs without error backoff.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through types, normalization, protocol schema, CLI/tool surfaces, and the isolated executor into the existing runner gate while keeping the default failure behavior unchanged.

## Reproduction Assessment

Yes. Source inspection plus existing runner tests show clean empty assistant stops are accepted only when `allowEmptyAssistantReplyAsSilent` is true, while isolated cron neither exposes nor forwards that flag.

## Solution Assessment

Yes, directionally. Reusing the existing runner gate behind an opt-in cron field is the narrow maintainable path, but the public field name and policy shape need maintainer signoff.
