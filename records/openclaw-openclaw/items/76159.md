## Summary

Keep open. Current main has the runner-level silent-empty gate, but cron jobs still do not expose or forward a per-job silent-stop contract; the closed #68452 work only covers explicit `NO_REPLY` tool-result paths, not a clean zero-payload assistant stop.

## What This Changes

Request an opt-in per-cron job flag that treats clean zero-payload assistant stops as successful silent cron runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that forwards to the existing runner gate, records clean empty stops as ok/no-delivery, and leaves current defaults unchanged.

## Reproduction Assessment

Yes. Source inspection shows a clean empty assistant stop is treated as silent only when `allowEmptyAssistantReplyAsSilent` is true, while isolated cron does not pass that flag and cron converts resulting error payloads into job errors/backoff.

## Solution Assessment

Yes, with maintainer signoff on the public contract. Reusing the existing runner gate behind an opt-in cron field is the narrowest maintainable direction because the runner already owns the safety checks for clean silent completion.
