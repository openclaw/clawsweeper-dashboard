## Summary

Keep open. Current main has the embedded-runner machinery to accept clean empty assistant stops only when a caller opts in, but isolated cron jobs still do not expose or forward a per-job silent-empty policy; the shipped adjacent fix only covers exact NO_REPLY tool-result paths.

## What This Changes

Add an opt-in per-cron-job flag so intentional stopReason=stop runs with zero payloads complete successfully instead of counting as errors and triggering cron backoff.

## Best Possible Solution

Add a documented opt-in cron agentTurn silent-empty policy that flows through cron types, normalization, protocol, CLI/tool surfaces, and the isolated executor into allowEmptyAssistantReplyAsSilent while preserving the default failure behavior.

## Reproduction Assessment

Yes. Source inspection shows clean empty assistant stops are accepted only behind allowEmptyAssistantReplyAsSilent, while isolated cron neither exposes nor forwards that flag and cron converts resulting error payloads into job errors and backoff.

## Solution Assessment

Yes directionally, but it needs maintainer signoff. Reusing the existing runner gate behind an opt-in cron payload field is the narrow maintainable path, but the public API shape is a product/protocol decision.
