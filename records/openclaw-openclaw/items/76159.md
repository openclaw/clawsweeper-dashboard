## Summary

Keep open: current main and v2026.6.8 still lack the requested per-job `acceptSilentStop` cron contract, and the existing silent-empty runner option is not threaded through the cron executor. The related open issues and PRs cover narrower `delivery.mode="none"` or current CLI empty-response regressions, so they do not settle the broader public cron config decision.

## What This Changes

Request an opt-in per-job cron flag that treats an intentionally empty assistant stop as a successful cron run without incrementing errors, backoff, or failure notifications.

## Best Possible Solution

Define one documented cron silent-success contract, thread the existing runner primitive through CLI and embedded cron where safe, and add schema/types/docs plus regression coverage that preserves real empty-response failure protection.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `acceptSilentStop` field, cron does not pass `allowEmptyAssistantReplyAsSilent`, and empty cron output still reaches the existing error-classification paths; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers should first choose whether the stable contract is an explicit flag, `delivery.mode: "none"`, a cron default, or command cron for deterministic script workloads.
