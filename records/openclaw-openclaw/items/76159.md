## Summary

Keep open. Current main still does not implement the requested per-job silent-stop policy: the embedded runner has an explicit empty-assistant opt-in, but cron payload/types/schema and the cron executor do not expose or forward `acceptSilentStop`; the active related PR only covers the narrower `delivery.mode: "none"` case.

## What This Changes

Request to add a per-job `acceptSilentStop` cron flag so intentional empty assistant stops can be treated as successful silent runs.

## Best Possible Solution

Have maintainers choose the public cron policy: land the narrower `delivery.mode: "none"` handoff if desired, and separately add a documented per-job silent-empty flag only if jobs that may also emit output should be able to opt into silent success.

## Reproduction Assessment

Yes, source-level. Current main can accept clean empty assistant stops only when the embedded runner receives `allowEmptyAssistantReplyAsSilent`, and the cron path neither exposes the requested flag nor forwards an equivalent policy.

## Solution Assessment

Unclear until maintainer signoff. Reusing the existing runner gate is the narrow implementation seam, but adding `acceptSilentStop` as public cron configuration changes automation success semantics and should not be treated as a mechanical bug fix.
