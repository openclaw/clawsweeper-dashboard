## Summary

Keep open. Current main still lacks the requested per-job cron `acceptSilentStop` policy; it only has a lower-level embedded-runner opt-in, and the open related PR covers the narrower `delivery.mode: "none"` case rather than the broader public cron flag requested here.

## What This Changes

Add an opt-in cron job `acceptSilentStop` policy so clean `stopReason=stop` runs with zero visible payloads can count as success for intentionally silent jobs.

## Best Possible Solution

Decide the public cron policy explicitly: land the narrow `delivery.mode: "none"` handoff if approved, then add a documented `agentTurn` silent-empty flag only if maintainers want silent success for jobs that may otherwise use normal delivery.

## Reproduction Assessment

Yes, source-level. Current main exposes the runner-only silent-empty gate but cron payload/schema/executor paths neither accept nor forward the requested per-job policy, so an empty assistant stop still follows the incomplete-turn error path unless a caller sets the internal flag.

## Solution Assessment

Unclear until maintainer signoff. Reusing `allowEmptyAssistantReplyAsSilent` is the narrow implementation seam, but exposing `acceptSilentStop` as public cron configuration changes automation success semantics; https://github.com/openclaw/openclaw/pull/81374 is only the narrower no-delivery fix.
