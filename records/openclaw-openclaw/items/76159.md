## Summary

Keep open. Current main has a runner-level silent-empty seam and shipped support for explicit `NO_REPLY`, but cron still has no per-job `acceptSilentStop` contract and the isolated cron executor does not pass the runner flag, so clean zero-payload cron stops can still be classified as errors and enter backoff.

## What This Changes

Add an opt-in per-cron `acceptSilentStop` flag that treats clean zero-payload assistant stops as successful silent cron runs without error notifications or backoff.

## Best Possible Solution

Add an additive `agentTurn` payload flag, wire it only for isolated embedded cron turns to the existing runner silent-empty gate, document it, and leave default empty-response behavior unchanged.

## Reproduction Assessment

Yes. A focused cron test can mock an isolated embedded agent turn ending with `stopReason: "stop"`, zero payloads, no tool errors, and an opt-in job flag; current main has no cron flag or runner forwarding, so the same shape still follows the incomplete-turn error path.

## Solution Assessment

Yes. Reusing `allowEmptyAssistantReplyAsSilent` behind an explicit cron opt-in is the narrowest maintainable fix because the runner already preserves the timeout, abort, tool-error, side-effect, and `stopReason=error` safety gates.
