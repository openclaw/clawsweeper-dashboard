## Label Justifications

- `P2`: The report is a valid bounded automation reliability problem, but it is not a security emergency, data loss, crash loop, or unusable core runtime.
- `impact:other`: The affected behavior is false cron failure/backoff and alert noise for intentionally silent automation, which is maintainer-visible but outside the specific owned impact categories.

## Summary

Keep open: current main still has the runner-level silent-empty primitive, but cron jobs do not expose or thread a per-job opt-in, so intentional silent cron turns can still flow into the error/backoff path.

## What This Changes

Request a per-cron-job opt-in that treats intentional empty terminal assistant stops as successful silent cron executions.

## Best Possible Solution

Define the cron contract, then add a narrow opt-in through cron schema/tool/CLI surfaces, isolated-run execution, status/history handling, and regression tests while preserving current default behavior.

## Reproduction Assessment

Yes. Source inspection shows an isolated cron run can still receive an empty-terminal error payload from the embedded runner, then cron records an error, increments `consecutiveErrors`, and applies alert/backoff behavior; I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. The requested flag is a plausible small product surface, but the best solution needs maintainer agreement on whether this belongs as a per-job cron option, a prompt-policy-derived runner option, or a different cron payload mode.
