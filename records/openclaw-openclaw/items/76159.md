## Summary

Keep open. Current main has a runner-level silent-empty opt-in, but cron job definitions, protocol schema, CLI/tool surfaces, and docs still do not expose a per-job way to opt clean zero-payload assistant stops into that path.

## What This Changes

Request a per-job cron flag that treats clean `stopReason=stop` runs with zero payloads as successful silent runs instead of cron errors/backoff.

## Best Possible Solution

Add a documented opt-in agentTurn cron payload policy that forwards to the existing runner gate, records clean empty stops as ok/no-delivery, and preserves the current default error behavior.

## Reproduction Assessment

Yes. Source and regression tests show clean zero-payload assistant stops error without `allowEmptyAssistantReplyAsSilent`, and isolated cron does not currently pass any equivalent flag into the runner.

## Solution Assessment

Yes as a direction, pending maintainer approval. Reusing the existing runner gate behind an opt-in cron field is the narrowest implementation path, but the public contract needs product/API signoff.
