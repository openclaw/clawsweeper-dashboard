## Summary

Keep open. Current main has a runner-level gate for clean empty assistant stops, but isolated cron jobs do not expose or forward any per-job silent-empty policy, and the documented cron path still requires an explicit `NO_REPLY` sentinel for intentional silence.

## What This Changes

Add an opt-in per-cron setting that treats clean zero-payload assistant `stop` runs as successful silent runs without error counters or backoff.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through validation and the isolated executor into the existing runner gate while preserving the default failure behavior.

## Reproduction Assessment

Yes by source inspection: clean empty assistant stops are only silent when `allowEmptyAssistantReplyAsSilent` is true, isolated cron never passes that option, and tests assert the no-flag path returns the incomplete-turn error. I did not run a live cron job because the source and regression coverage already show the current-main path.

## Solution Assessment

Unclear until maintainers choose the public API shape. Directionally, reusing the existing runner gate behind an opt-in cron policy is the narrow maintainable solution, but the exact field placement and documented semantics need product review.
