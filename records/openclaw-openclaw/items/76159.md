## Summary

Keep open. Current main has a runner-level silent-empty gate, but cron jobs still have no public per-job field and do not forward that gate, so the requested cron behavior remains unimplemented.

## What This Changes

Request adds a per-cron `acceptSilentStop` opt-in that treats clean zero-payload `stop` turns as successful silent runs.

## Best Possible Solution

Add an additive cron `agentTurn` opt-in field that forwards to `allowEmptyAssistantReplyAsSilent`, records the run as ok/no-delivery for clean empty stops, and leaves default empty-response, timeout, provider-error, tool-error, and side-effect behavior fail-closed.

## Reproduction Assessment

Yes. Source inspection and existing runner tests give a high-confidence path: without a cron field forwarding `allowEmptyAssistantReplyAsSilent`, a clean empty assistant stop can still produce the incomplete-turn error that cron counts for backoff.

## Solution Assessment

Yes, pending maintainer approval. Reusing the existing runner silent-empty gate behind an additive per-job cron option is the narrowest maintainable direction because the runner already owns the safety checks.
