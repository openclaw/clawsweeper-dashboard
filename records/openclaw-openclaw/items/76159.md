## Summary

Keep open. Current main has a safety-scoped embedded-runner switch for clean empty assistant stops, but isolated cron jobs still do not expose or forward a per-job silent-empty contract; the shipped #68452 work only covers exact `NO_REPLY` tool-result paths, not a clean zero-payload assistant stop.

## What This Changes

Request an opt-in per-cron-job flag that lets intentionally silent zero-payload `agentTurn` runs complete as ok instead of erroring and entering backoff.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through cron normalization, protocol schema, CLI/tool surfaces, and the isolated executor into the existing runner gate while keeping the default failure behavior unchanged.

## Reproduction Assessment

Yes. Source inspection shows clean empty assistant stops are accepted only when `allowEmptyAssistantReplyAsSilent` is true, while isolated cron neither exposes nor forwards that flag and cron converts resulting error payloads into job errors/backoff.

## Solution Assessment

Yes, directionally, with maintainer signoff. Reusing the existing runner gate behind an opt-in cron field is the narrow maintainable path, but the public field name and docs/protocol shape need product approval.
