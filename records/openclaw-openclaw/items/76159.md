## Summary

Keep open. Current main still has the embedded-runner gate needed to treat clean empty assistant stops as silent, but isolated cron jobs do not expose or forward a per-job policy, and the existing shipped fix covers exact `NO_REPLY` sentinel paths rather than zero-payload assistant stops.

## What This Changes

The issue requests an opt-in cron-job setting that treats clean `stopReason=stop` zero-payload assistant turns as successful silent cron runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through cron types, protocol schema, cron tool/docs, normalization/persistence, and the isolated executor to the existing runner gate while preserving the default failure behavior.

## Reproduction Assessment

Yes, by source inspection: a cron `agentTurn` with `stopReason=stop` and no payloads still reaches the embedded-runner empty-response error path because cron never passes `allowEmptyAssistantReplyAsSilent`. The runner tests also assert the no-flag default surfaces `Agent couldn't generate a response`.

## Solution Assessment

Yes directionally: reusing the existing guarded runner gate behind an opt-in cron policy is the narrow maintainable path. The exact public API shape, including whether it should be named `acceptSilentStop`, needs maintainer signoff first.
