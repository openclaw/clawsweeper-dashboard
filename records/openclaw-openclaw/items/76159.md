## Summary

Keep open. Current main has adjacent silent-token and runner-level support, but it does not expose a per-job `acceptSilentStop` cron contract or pass the existing silent-empty runner gate from cron, so the reported clean empty-stop cron case remains actionable.

## What This Changes

The issue requests an opt-in cron-job flag that treats intentionally silent `stopReason=stop` runs with zero payloads as successful.

## Best Possible Solution

Add a documented opt-in cron job contract that forwards to the existing silent-empty runner gate for isolated agent-turn jobs, records an ok/no-delivery run, and leaves default empty-response, timeout, provider-error, and tool-error behavior unchanged.

## Reproduction Assessment

Yes. A focused test can mock an isolated cron agent turn ending with a clean empty assistant stop and zero payloads; current cron does not pass the silent-empty runner flag, so the runner/finalizer path still produces an error outcome that increments cron backoff.

## Solution Assessment

Yes. Reusing `allowEmptyAssistantReplyAsSilent` behind an opt-in cron field is the narrowest maintainable fix because the hard safety gates for tool errors, side effects, timeouts, and `stopReason=error` already exist in the runner helper.

Fix https://github.com/openclaw/openclaw/issues/76159. Add an opt-in cron job field, using the requested `acceptSilentStop?: boolean` name unless maintainers choose a different public name, so isolated agent-turn cron jobs can treat a clean empty assistant stop with zero payloads as intentional silence. Keep the default false. Wire the field through stored cron job types, normalization/create/update paths, gateway protocol schemas, cron CLI add/edit if applicable, and the agent `cron` tool schema/recovery path. In the isolated cron executor, pass the existing `allowEmptyAssistantReplyAsSilent` runner option only when the job is an isolated/current/session agent-turn job with `acceptSilentStop === true`; do not broaden this to main system-event jobs. Preserve the runner safety gates: timeouts, aborts, `stopReason="error"`, run-level provider errors, tool errors, and potential side effects must still fail. Ensure the resulting silent-empty run records as `status: "ok"`, clears `consecutiveErrors`, does not send the incomplete-turn notification, and still appears in cron run history. Update docs for cron jobs/CLI and add a user-facing changelog entry crediting @RonnyMarkus if policy allows. Do not change the existing explicit `NO_REPLY` semantics except where tests need to prove both paths remain quiet successes.
