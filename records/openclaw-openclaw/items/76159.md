- `impact:other`: The issue affects cron job execution status, failure alerts, and scheduling backoff rather than a more specific ClawSweeper impact class.

## Summary

Keep open: current main still has the silent-empty runner primitive, but cron jobs do not expose a per-job opt-in and the cron path still treats the resulting empty-terminal error payload as a failed run with error backoff.

## What This Changes

The issue requests a per-job `acceptSilentStop` cron option that treats an empty `stop` assistant turn as a successful silent cron run.

## Best Possible Solution

Expose a narrow maintainer-approved silent-empty cron semantic through cron payload/protocol/CLI/tool/docs and route it to the existing runner silent-empty support while keeping default error behavior unchanged.

## Reproduction Assessment

Yes, from source inspection: current cron schemas lack the requested flag, the cron executor does not pass the existing silent-empty runner option, and the empty-terminal error path flows into cron error/backoff state. I did not run a live cron reproduction in this read-only review.

## Solution Assessment

Unclear if the exact `acceptSilentStop` name is the best API, but the maintainable shape is a narrow per-job opt-in that reuses the existing runner silent-empty primitive. Maintainers should decide the public config semantics before implementation.
