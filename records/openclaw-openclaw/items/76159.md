## Summary

Keep open. Current main has the embedded-runner gate that can turn clean empty assistant stops into silent `NO_REPLY`, and #68452 covered exact `NO_REPLY` tool-result paths, but isolated cron jobs still do not expose or forward a per-job policy for zero-payload assistant stops.

## What This Changes

Request a per-job cron `acceptSilentStop` opt-in so clean `stopReason=stop` runs with zero payloads are treated as successful silent runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through normalization, gateway protocol schema, CLI/tool creation surfaces, and the isolated executor into the existing runner gate while preserving the default failure behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: the runner only accepts clean empty stops with `allowEmptyAssistantReplyAsSilent`, isolated cron never passes that option, and cron converts fatal embedded error payloads into job errors and counter increments.

## Solution Assessment

Is this the best way to solve the issue? Directionally yes, reusing the existing runner gate behind an opt-in cron policy is the narrow maintainable path; the exact public field name and semantics still need maintainer approval.
