## Summary

Keep open: current main and v2026.6.10 still lack the requested per-job cron silent-stop contract, and the existing runner opt-in is not threaded through the cron executor. Related open issues and PRs overlap the same empty-cron symptom, but they do not settle the broader public config/product decision requested here.

## What This Changes

The issue requests an opt-in cron-job setting that treats intentionally empty assistant stops as successful silent cron runs.

## Best Possible Solution

Define one documented cron silent-success contract, thread the existing runner primitive through both CLI and embedded cron where appropriate, and add schema/types/docs plus regression coverage that preserves real empty-response failure protection.

## Reproduction Assessment

Yes, source-reproducible: current main has no accepted `acceptSilentStop` field, the cron executor does not pass `allowEmptyAssistantReplyAsSilent`, and empty CLI output still fails when the flag is absent. I did not run a live cron because this review is read-only.

## Solution Assessment

Unclear. A per-job flag is plausible, but maintainers should first decide whether the permanent contract is explicit per-job config, `delivery.mode: "none"`, an all-cron default, or command cron for deterministic script cases.
