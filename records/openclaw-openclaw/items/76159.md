## Summary

Keep open. Current main has the internal runner gate for clean empty assistant stops, but isolated cron jobs do not expose or forward a per-job silent-empty policy; the shipped #68452 work covers exact `NO_REPLY` tool-result paths, not zero-payload assistant stops without that sentinel.

## What This Changes

Request adds an opt-in per-cron-job flag that treats clean zero-payload `stop` turns as successful silent cron runs instead of incomplete-turn errors.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through normalization, gateway schema, CLI/hook/edit surfaces, and the isolated executor into the existing runner gate, with regression coverage for ok status, counter reset, delivery suppression, and run history.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: the runner can accept clean empty stops only with `allowEmptyAssistantReplyAsSilent`, isolated cron never passes that option, and cron converts fatal embedded error payloads into job errors and counter increments.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: reusing the existing runner gate behind an opt-in cron payload policy is the narrow maintainable path, but the public API shape needs maintainer signoff before implementation.
