## Summary

Keep open. Current main still has the embedded-runner gate for clean empty assistant stops, but isolated cron jobs do not expose or forward any per-job policy equivalent to the requested flag, so this remains a valid feature/config request rather than implemented cleanup.

## What This Changes

Request an opt-in per-cron-job flag that treats clean zero-payload `stopReason="stop"` agent turns as successful silent runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through normalization, gateway protocol schema, CLI/tool creation surfaces, and the isolated executor into the existing runner gate while preserving current failure defaults.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: clean empty assistant stops only become silent when `allowEmptyAssistantReplyAsSilent` is true, and isolated cron never passes that option.

## Solution Assessment

Is this the best way to solve the issue? Directionally yes, reusing the existing runner gate behind an opt-in cron policy is the narrow path, but maintainers should decide the public flag name and contract before implementation.
