## Summary

Keep open. Current main has a guarded runner path that can turn clean empty assistant stops into silent `NO_REPLY`, and the adjacent #68452 fix is already shipped for exact `NO_REPLY` tool results, but isolated cron jobs still have no public per-job field and do not forward the runner flag.

## What This Changes

Request an opt-in cron job setting that treats intentional zero-payload assistant stops as successful silent runs.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` silent-empty policy that flows through types, normalization, protocol schema, CLI/tool surfaces, docs, and the isolated executor into the existing runner gate while preserving the default failure behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows clean empty assistant stops are only accepted when `allowEmptyAssistantReplyAsSilent` is supplied, while isolated cron neither exposes nor forwards that flag.

## Solution Assessment

Yes, directionally, with maintainer signoff. Reusing the existing runner gate behind an opt-in cron field is the narrow maintainable path, but changing the public cron contract needs product/API approval.
