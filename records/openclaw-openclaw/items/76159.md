## Summary

Keep open. Current main has the runner-level silent-empty opt-in, but cron job payload/schema/CLI/tool surfaces still do not expose a per-job flag and the cron executor does not forward the runner flag, so the requested behavior remains missing.

## What This Changes

Request a per-job cron `agentTurn` flag that treats a clean zero-payload assistant stop as a successful silent run.

## Best Possible Solution

Add a documented, additive cron `agentTurn` boolean that forwards to `allowEmptyAssistantReplyAsSilent`, records the run as ok/no-delivery, and leaves default empty-response behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a cron `agentTurn` can reach `runEmbeddedPiAgent` without `allowEmptyAssistantReplyAsSilent`, and existing runner tests show that the same clean empty assistant stop becomes an incomplete-turn error without the flag.

## Solution Assessment

Yes, with maintainer approval for the public contract. Reusing the existing runner silent-empty gate behind an opt-in cron payload field is the narrowest maintainable implementation path.
