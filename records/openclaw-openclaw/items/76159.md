## Summary

This should stay open: current `main` has the runner mechanics for treating clean empty assistant turns as silent, but cron job definitions do not expose an `acceptSilentStop`-style flag and cron does not pass the runner flag, so the requested per-job behavior is not implemented.

## What This Changes

Add a per-job opt-in cron flag so intentional zero-payload `stopReason=stop` agent-turn runs are recorded as successful silent cron runs instead of errors.

## Best Possible Solution

Expose a public cron agent-turn opt-in, wire it to `allowEmptyAssistantReplyAsSilent`, preserve the default error behavior, and document that the flag is for jobs whose prompt policy intentionally allows no visible output.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a cron agent-turn that produces zero payloads cannot currently set or pass the runner’s silent-empty flag, while the runner tests show empty turns are silent only when that flag is true.

## Solution Assessment

Yes, the requested direction is a narrow fit if implemented through the existing runner seam. The safest shape is to add the public cron schema/normalization field and pass it only for agent-turn cron jobs, rather than weakening global incomplete-turn classification.
