## Summary

Keep open. Current main has a runner-level silent-empty gate, but isolated cron jobs still have no public per-job flag and do not forward that gate, so the requested cron behavior remains unimplemented.

## What This Changes

Add a per-job cron option that treats clean zero-payload assistant stops as intentional quiet success for jobs that are allowed to produce no output.

## Best Possible Solution

Add a documented opt-in cron `agentTurn` contract that normalizes through CLI/API schemas and forwards to the existing runner safety gate while leaving default empty-response behavior unchanged.

## Reproduction Assessment

Yes. Source inspection and runner regression tests show clean empty assistant stops only become silent success when `allowEmptyAssistantReplyAsSilent` is true, and isolated cron never passes that flag.

## Solution Assessment

Unclear until maintainers approve the public cron option name, but reusing the existing runner silent-empty gate behind an opt-in job field is the narrowest maintainable direction.
