## Summary

Keep open. Current main has reduced onboarding friction with QuickStart, a post-setup TUI/Web hatch, and a chat-first Crestodian preview, but it still does not implement the requested value-first default flow before setup/customization; the remaining work needs maintainer UX and security scoping.

## What This Changes

Request a first-run onboarding flow that shows a useful starter task before optional personalization such as agent name, reply style, symbols, or persona preferences.

## Best Possible Solution

Define a maintainer-approved first-run contract that keeps mandatory security/auth decisions explicit while moving optional personalization behind a starter task or template-based value moment.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source and docs inspection show the current first-run ordering, but there is no failing current-main bug to reproduce.

## Solution Assessment

Unclear: the value-first direction is plausible, but it is not yet specified enough to be the best implementation path. Maintainers need to decide which prompts are non-essential and which security/auth decisions must remain before any starter task runs.
