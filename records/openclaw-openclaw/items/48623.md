## Summary

Keep open. Current main still persists `lastGood` but does not use it to choose the next auth profile, while the original shared-timeout symptom is partly mitigated and the slug-generator timeout, but not model, override has been addressed.

## What This Changes

Request failover-aware auth-profile ordering that prefers the last successful profile, per-profile retry timeout isolation, and a lightweight model override for slug generation.

## Best Possible Solution

Add failover-aware auth-profile selection that can prefer the most recently successful usable profile after rate-limit/auth/billing failures while preserving normal round-robin scheduling; treat the slug-generator model override as a separate config decision if maintainers still want it.

## Reproduction Assessment

Yes for the main remaining ordering concern: source and tests show `lastGood` is recorded but ignored by auth-profile ordering, session override selection, and embedded-run profile rotation. The original shared-timeout cascade is not reproduced from current source because each embedded attempt now schedules its own timeout.

## Solution Assessment

No for a global lastGood sort: that would overwrite the current round-robin behavior rather than adding failure-recovery semantics. The safer direction is explicit failover-aware ordering plus separate maintainer scope for slug model configuration and the overlapping #55897/#50217 work.
