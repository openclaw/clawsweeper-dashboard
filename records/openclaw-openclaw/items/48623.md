## Summary

Keep open. Current main still intentionally uses auth-profile round-robin ordering from `usageStats.lastUsed` instead of `lastGood`, while the shared per-attempt timeout symptom is partly mitigated and the slug-generator model override remains unresolved.

## What This Changes

The issue asks for auth-profile failover to prefer the last successful profile, avoid timeout-budget cascades across rotations, and allow lightweight slug-generation model selection.

## Best Possible Solution

Add an explicit failover-aware auth-profile selection path that can prefer the most recently successful usable profile after rate-limit/auth/billing failures while preserving normal round-robin semantics, and split slug-generator model selection into a separate config/product decision if maintainers want it.

## Reproduction Assessment

Yes for the remaining ordering behavior: current source and tests show `lastGood` is persisted but intentionally not used for selection, and session/embedded failover consumes that resolved order. The original shared timeout cascade is not reproduced from current source because attempts now get fresh attempt timers.

## Solution Assessment

No, replacing round-robin globally is not the best fix because current docs and tests intentionally preserve it. The safer solution is a maintainer-approved failover-aware mode or policy option, plus separate handling for slug-generator model selection.
