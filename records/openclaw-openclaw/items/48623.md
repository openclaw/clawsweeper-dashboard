## Summary

Keep open. Current main still records `lastGood` but does not use it to order automatic auth-profile selection, and both session override selection and embedded failover consume that precomputed round-robin/LRU order. The embedded retry timeout complaint is partly mitigated by fresh per-attempt abort timers, and slug generation now honors the general agent timeout, but the central failover-aware ordering request and slug-specific model override remain unresolved.

## What This Changes

This issue asks OpenClaw to prefer the last successful auth profile during failover recovery, avoid cascading embedded-run timeouts across profile rotations, and allow slug generation to use a lightweight task-specific model.

## Best Possible Solution

Add an explicit failover-aware auth-profile selection mode that prefers the most recently successful usable profile after rate-limit/auth/billing failures while preserving normal round-robin behavior for routine scheduling. Cover `resolveAuthProfileOrder`, session override selection, and embedded-run profile rotation with focused tests, then split or separately track slug-generator model override and cooldown-isolation work if maintainers want that surface handled independently.

## Reproduction Assessment

Yes for the main remaining ordering bug: configure multiple profiles, record `lastGood` for the backup, leave the default profile older in `usageStats.lastUsed`, and `resolveAuthProfileOrder` will still return LRU round-robin order instead of preferring `lastGood`. The shared-timeout portion is only partially reproducible from current source because each embedded attempt now creates its own abort timer.

## Solution Assessment

No; current main is only a partial mitigation. The maintainable fix is not to replace round-robin globally, but to introduce failover-aware lastGood/MRU ordering for recovery paths and keep ordinary scheduling semantics intact.
