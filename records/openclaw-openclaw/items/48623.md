## Summary

Keep open. Current main partially mitigates the timeout side and the slug-generator timeout, but it still intentionally preserves round-robin auth-profile ordering over `lastGood`, and the remaining model/ordering requests require maintainer product/config policy rather than conservative cleanup closure.

## What This Changes

The issue requests auth-profile failover to prefer the last successful profile, avoid shared timeout behavior across failover attempts, and allow lightweight per-task model choices for slug generation.

## Best Possible Solution

Split the remaining work: keep the current per-attempt timeout behavior, use https://github.com/openclaw/openclaw/issues/55897 and https://github.com/openclaw/openclaw/pull/50217 for auth-profile ordering/persistence decisions, and open a separate product/config item if embedded task model overrides should be supported.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence reproduction for the `lastGood` ordering part because current code and tests still prefer oldest-first round-robin, but the full live Anthropic 429-to-backup cascade was not replayed against current main.

## Solution Assessment

No as a single automatic fix. The timeout portion is partly mitigated, while `lastGood` priority and per-task model selection require maintainers to choose an ordering/config contract before implementation.
