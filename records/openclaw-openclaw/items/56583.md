## Summary

Keep open for maintainer handling. Current main has already moved overload pacing to a different shipped contract, and this PR is stale against that surface while also mixing in live-test HOME/profile changes with concrete correctness and security regressions.

## What This Changes

The PR raises embedded Pi overload failover backoff to a configurable 30s ceiling via `agents.defaults.embeddedPi.overloadBackoffMaxMs`, updates config schema/help/tests, and adds live-test HOME/profile staging changes.

## Best Possible Solution

If overload pacing still needs product work, implement it against the shipped `auth.cooldowns` model with focused schema/docs/tests. Split any live-test HOME/profile staging changes into a separate PR and fix the profile-loading and config-staging issues there.

## Reproduction Assessment

Yes for the PR findings: the provided diff and review comments show ordinary `installTestEnv()` can source the profile before live-only setup, and the staging path copies only the active config file. The current-main behavior was verified by source/docs inspection rather than running tests, per the read-only review constraint.

## Solution Assessment

No. The PR is no longer the narrowest maintainable fix because it targets an obsolete overload config path and bundles unrelated live-test harness changes; the safer path is a rebased, auth-cooldowns-only overload change plus a separate live-test fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate profile sourcing to live setup:** `test/test-env.ts:338`
  - body: The patch calls `loadProfileEnv(realHome)` for every `installTestEnv()` invocation. A developer profile that exports live flags or provider credentials can turn ordinary non-live test workers into live/network runs before isolated live setup has run.
  - confidence: 0.9
- **[P2] Refresh the live env snapshot after loading the profile:** `test/test-env.ts:336`
  - body: `liveEnvSnapshot` is captured before profile loading, but `stageLiveTestState()` uses that snapshot to choose `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. Live runs relying on `~/.profile` for those paths will stage the wrong config or state.
  - confidence: 0.86
- **[P2] Preserve config include files when staging live HOME:** `test/test-env.ts:315-316`
  - body: The staging code copies only the active config file into the temp home. Config include files resolved relative to the original config root can disappear, so valid live configs with sibling includes fail unless the user opts into the real HOME.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR can source live credentials and live-test flags into ordinary non-live test processes.

Concerns:

- **[medium] Profile sourcing can leak live credentials into unit tests:** `test/test-env.ts:338`
  - body: Because profile env loading runs outside the live-only path in the PR, non-live test setup can import provider keys, GitHub tokens, and live flags from a developer profile before isolation. That broadens secret exposure and can trigger expensive or flaky live network suites unexpectedly.
  - confidence: 0.88
