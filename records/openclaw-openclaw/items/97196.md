## Summary

Keep open: current main still has the source-level path that can put OpenRouter nested `reasoning.effort` and DeepSeek-native `reasoning_effort` on the same request, and the linked fix PRs are still open rather than merged.

## What This Changes

OpenRouter-routed DeepSeek V4 requests can fail with HTTP 400 because OpenClaw can send both nested `reasoning.effort` and top-level `reasoning_effort`.

## Best Possible Solution

Land one provider-owned OpenRouter DeepSeek V4 fix that sends exactly one reasoning control field, preserves replay/backfill behavior, and then close this issue after the chosen PR merges.

## Reproduction Assessment

Yes at source level: current main can build nested OpenRouter `reasoning.effort` and then run the DeepSeek V4 wrapper that adds top-level `reasoning_effort` on the same payload. I did not run a live OpenRouter request in this read-only review.

## Solution Assessment

Unclear for the issue body's exact suggested fix, because a gate-only change can miss the OpenRouter provider wrapper composition. The best fix should cover the OpenRouter-owned wrapper path and make the final payload choose exactly one effort representation.
