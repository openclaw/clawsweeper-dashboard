## Summary

Keep open: current main still has a source-reproducible path where OpenRouter DeepSeek V4 models can receive both nested `reasoning.effort` and top-level `reasoning_effort`, and the newer live OpenRouter catalog reports these models support reasoning so the builder-side gate can be active.

## What This Changes

The issue reports that OpenRouter-routed DeepSeek V4 requests fail with HTTP 400 because OpenClaw sends both OpenRouter and native DeepSeek reasoning-effort fields.

## Best Possible Solution

Keep this open and land a narrow OpenRouter DeepSeek V4 payload fix that preserves replay backfill while ensuring the final request sends exactly one effort field, with regression coverage for the composed wrapper order.

## Reproduction Assessment

Yes at source level: current main can mark the OpenRouter DeepSeek V4 models as reasoning-capable, build nested OpenRouter reasoning, and then run the DeepSeek V4 wrapper that adds top-level `reasoning_effort`. I did not execute a live OpenRouter call in this read-only pass.

## Solution Assessment

The issue body's merged-compat gate idea is not the best full fix if it drops OpenRouter DeepSeek V4 replay backfill. The better fix is to keep the OpenRouter-owned DeepSeek V4 wrapper but make the final payload choose exactly one effort representation and test that composed path.
