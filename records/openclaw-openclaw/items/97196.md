## Summary

Keep open: current main still has a source-reproducible path where OpenRouter DeepSeek V4 models can receive both nested `reasoning.effort` and top-level `reasoning_effort`, and prior related fixes are adjacent rather than resolving this exact composed-wrapper interaction.

## What This Changes

The issue reports DeepSeek V4 models routed through OpenRouter failing because both `reasoning_effort` and nested `reasoning.effort` can be sent in one request.

## Best Possible Solution

Land a narrow OpenRouter DeepSeek V4 payload fix that preserves replay/backfill behavior while ensuring the final request sends exactly one reasoning-effort representation, with a composed-wrapper regression test.

## Reproduction Assessment

Yes at source level: current main can detect OpenRouter compat, build nested `reasoning.effort` for reasoning-capable DeepSeek V4 routes, and then run the OpenRouter DeepSeek V4 wrapper that adds top-level `reasoning_effort`. I did not execute a live OpenRouter call in this read-only pass.

## Solution Assessment

No, the issue body's merged-compat gate may be too broad if it drops OpenRouter DeepSeek V4 replay/backfill behavior. The better fix is to keep the OpenRouter-owned DeepSeek V4 wrapper path but make the final payload choose exactly one effort representation and test that composed order.
