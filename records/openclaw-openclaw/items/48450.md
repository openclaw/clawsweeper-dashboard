## Summary

Keep open. Current main still creates the Discord thread first, then awaits a separate initial-message post for non-forum threads; if that second call rejects, the runtime never returns the created thread id and the generic tool adapter reports an error. The related #18117 change explains why this path exists, but it did not add partial-success handling, and search did not find an open paired fix PR for this issue.

## What This Changes

The issue asks Discord `thread-create` to avoid returning a generic error without the created thread id after Discord already created the thread but a later initial-message step failed.

## Best Possible Solution

Fix the Discord plugin `thread-create` path so the created thread id is preserved and surfaced when the non-forum follow-up initial-message post fails. Keep forum/media behavior unchanged, make the result shape additive, and add regression coverage for first REST POST succeeds plus second REST POST rejects.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock the Discord REST client so the first thread-create POST resolves `{ id: "t1" }` and the second `Routes.channelMessages("t1")` POST rejects; current main will throw before returning `t1`.

## Solution Assessment

Yes, the best fix is a narrow Discord plugin change around the two-step thread-create helper/runtime result. Broad retry deduplication or same-name active-thread reuse would be less precise because duplicate thread names can be intentional and the real ambiguity is the missing partial-success return.
