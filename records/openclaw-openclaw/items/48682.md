## Summary

Keep open: the underlying `auth.order` bug still exists on current main, but this PR's current approach is stale and introduces concrete credential-selection regressions around disabled discovery and explicit profile exclusions.

## What This Changes

The PR adds a core implicit GitHub Copilot resolver and expands models-config tests so implicit Copilot discovery follows `auth.order` and fallback cases.

## Best Possible Solution

Move the fix into the GitHub Copilot plugin auth/catalog path using the existing order-aware provider-auth seam, while preserving env-token precedence, explicit provider config, discovery disablement, and `auth.order` exclusions.

## Reproduction Assessment

Yes. Source inspection of current main shows `resolveFirstGithubToken` reads the first stored Copilot profile after env-token precedence, while the credential semantics docs make `auth.order` the profile-selection contract.

## Solution Assessment

No. The requested behavior is valid, but the PR implements it in stale core provider orchestration and violates plugin disablement plus explicit auth-order semantics; the safer fix belongs in `extensions/github-copilot`.
