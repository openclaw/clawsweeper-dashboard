## Summary

Keep open: the linked bug is still source-reproducible on current main and the latest release, but this PR should not merge as-is because it adds a core Copilot fallback that can bypass disabled discovery and explicit auth-order exclusions. It also lacks inspectable real behavior proof and is currently conflicting against main after the provider-discovery file split.

## What This Changes

This branch adds a core models-config fallback for implicit GitHub Copilot provider discovery that selects Copilot auth profiles using `auth.order` and expands models-config regression tests for ordered and unresolved profile fallback paths.

## Best Possible Solution

Fix the shared Copilot plugin auth helper, or a narrow plugin-facing SDK seam it uses, so catalog and embeddings honor ordered eligible profiles while preserving env-token precedence, discovery disablement, and explicit auth-order exclusions.

## Reproduction Assessment

Yes. By source inspection on current main, configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, and the catalog path reaches `resolveFirstGithubToken`, which selects `profileIds[0]`.

## Solution Assessment

No. The bug is real, but this PR solves it with a core fallback after plugin discovery; the narrower maintainable fix belongs in the Copilot plugin helper or SDK auth seam and must preserve disablement plus auth-order exclusions.
