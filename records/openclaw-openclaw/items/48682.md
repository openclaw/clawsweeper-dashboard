## Summary

Keep this PR open, but do not merge it as-is. The underlying Copilot auth.order bug still exists on current main, while the proposed core fallback is stale against the current provider split and introduces a credential-use regression by bypassing plugin-owned discovery controls.

## What This Changes

The PR adds a core implicit GitHub Copilot provider resolver and expands models-config tests for auth.order-driven Copilot profile selection and fallback behavior.

## Best Possible Solution

Move the fix into the GitHub Copilot plugin auth/catalog path, using the existing order-aware provider-auth SDK seam or a plugin-local helper that delegates to it. Keep env-token precedence, explicit models.providers config, plugin discovery disablement, and documented auth.order exclusion semantics intact, with focused coverage for ordered profiles, unresolved ordered tokenRefs, stale stored order, and disabled discovery.

## Reproduction Assessment

Yes. The linked bug and PR body give a concrete multiple-profile auth.order scenario, and current main source plus the existing models-config test still show first stored profile selection instead of order-aware selection.

## Solution Assessment

No. The requested behavior is valid, but this PR solves it in stale core provider code and bypasses plugin-owned discovery controls; the narrower maintainable solution is in extensions/github-copilot auth/catalog code using the provider-auth order seam.

## Review Findings
