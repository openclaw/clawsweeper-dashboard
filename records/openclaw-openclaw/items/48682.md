## Summary

Keep open. The underlying bug is still present on current main, but this PR is not merge-ready because its core fallback can bypass plugin disablement, conflicts with documented auth-order exclusion semantics, and lacks inspectable real behavior proof.

## What This Changes

The PR adds core implicit GitHub Copilot provider discovery code and regression tests so generated models config follows `auth.order` when selecting stored Copilot profiles.

## Best Possible Solution

Implement order-aware token selection inside `extensions/github-copilot/auth.ts` or a provider-auth SDK helper it uses, preserving env-token precedence, discovery disablement, explicit `auth.order` exclusions, and coverage for catalog discovery plus embeddings.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, then run implicit Copilot discovery or Copilot embeddings; current main reaches `profileIds[0]`.

## Solution Assessment

No. The bug is real, but the PR adds a core fallback after plugin discovery; the narrower maintainable fix is in the GitHub Copilot plugin-owned helper or SDK auth seam, while preserving disablement and order exclusions.
