## Summary

Keep open: the linked bug is still valid on current main, but this PR is stale and routes Copilot credential selection through a core fallback that can bypass plugin discovery controls and documented auth-order exclusions.

## What This Changes

The branch adds a core fallback resolver for implicit `github-copilot` discovery and expands models-config tests for `auth.order` profile selection and fallback cases.

## Best Possible Solution

Rework the fix inside the GitHub Copilot plugin auth/catalog path using the order-aware provider-auth seam, with regression coverage for disabled discovery and excluded profiles.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveFirstGithubToken` selects `profileIds[0]` from `listProfilesForProvider`, while the credential semantics docs make `auth.order` the selection/filtering contract.

## Solution Assessment

No. The requested behavior is right, but this PR solves it by adding a core fallback after plugin discovery; the maintainable fix belongs in the plugin-owned Copilot auth/catalog path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Respect disabled Copilot discovery:** `src/agents/models-config.providers.ts:806-811`
  - body: The new fallback injects `github-copilot` whenever plugin discovery did not populate it, but intentional disablement also produces that absence. With Copilot discovery disabled, this can re-add the provider and trigger token exchange despite the user's opt-out.
  - confidence: 0.86
- **[P2] Preserve explicit auth.order exclusions:** `src/agents/models-config.providers.ts:930-931`
  - body: After ordered candidates are exhausted, this fallback tries raw stored profiles that may have been omitted from explicit `auth.order`. The credential semantics doc says omitted profiles are excluded, so this can select a credential the operator intentionally left out.
  - confidence: 0.82
