## Summary

Keep this PR open, but do not merge it as-is. Current main still has the reported Copilot profile-order bug, while the PR patch is stale against the current plugin-owned discovery path and introduces a blocking regression by re-adding Copilot outside plugin discovery controls.

## What This Changes

The PR adds a core implicit GitHub Copilot provider resolver and expands models-config regression coverage for auth.order and fallback profile selection.

## Best Possible Solution

Rework the fix inside the GitHub Copilot plugin auth/catalog path, using the existing order-aware provider-auth SDK seam or an equivalent plugin-local helper. Preserve env-token precedence, explicit provider config behavior, plugin discovery disablement, and documented `auth.order` exclusion semantics, with focused plugin and models-config coverage for ordered profiles, unresolved ordered tokenRefs, stale stored order, and discovery disablement.

## Reproduction Assessment

Yes. The PR body and linked bug give a concrete multiple-profile `auth.order` scenario, and current main source plus regression tests still show first-profile selection rather than order-aware selection.

## Solution Assessment

No. The intended product fix is valid, but this PR solves it in stale core code and bypasses plugin-owned discovery controls; the narrower maintainable path is to rework `extensions/github-copilot/auth.ts` and the plugin catalog path.
