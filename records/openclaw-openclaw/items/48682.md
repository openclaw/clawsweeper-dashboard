## Summary

Keep open. The linked bug is valid, but this branch is conflicting, lacks inspectable real behavior proof, and introduces auth-boundary regressions by adding a core Copilot fallback instead of fixing the shared Copilot auth resolver.

## What This Changes

The PR changes implicit GitHub Copilot model-provider discovery to choose stored profiles according to `auth.order` and adds models-config regression tests for ordered and unresolved profile fallback cases.

## Best Possible Solution

Rebase and implement ordered Copilot profile selection inside `extensions/github-copilot/auth.ts` or a narrow SDK auth seam shared by catalog and embeddings, preserving env precedence, discovery disablement, and explicit `auth.order` exclusions.

## Reproduction Assessment

Yes. By source inspection on current main, multiple stored `github-copilot` profiles with `auth.order.github-copilot` preferring a non-first profile still reach `resolveFirstGithubToken`, which selects `profileIds[0]` when env tokens are absent.

## Solution Assessment

No. The bug is real, but this PR solves it with a conflicting core post-discovery fallback; the maintainable fix belongs in the shared Copilot auth helper or SDK auth seam and must preserve discovery disablement plus auth-order exclusions.
