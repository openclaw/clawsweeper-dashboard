## Summary

Keep open: the intended regression fix is valid, but the PR’s new catalog-to-npm path drops existing catalog safety metadata before invoking the npm installer.

## What This Changes

The PR resolves bare official external plugin ids through the official catalog before raw npm fallback, with CLI tests and a changelog entry for the Brave install regression.

## Best Possible Solution

Keep the catalog-before-npm behavior, but thread catalog `pluginId` and `expectedIntegrity` through the new plan and installer call before merge.

## Reproduction Assessment

Yes. Source inspection of current main plus the linked issue shows bare `brave` falls through to raw npm when no bundled Brave source is present, while the official catalog already maps `brave` to `@openclaw/brave-plugin`.

## Solution Assessment

No, not yet. Resolving official ids before npm is the narrow maintainable fix, but this PR should preserve the catalog safety metadata already used by other install paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass catalog safety metadata into npm installs:** `src/cli/plugins-install-command.ts:769`
  - body: The new official-id branch resolves through the combined official catalog, then calls the npm installer with only the resolved spec. Catalog entries can carry `expectedIntegrity`, and existing catalog repair installs also pass `expectedPluginId`, so this path would bypass those checks for bare-id installs. Thread the catalog plugin id and expected integrity through the plan and into `tryInstallPluginOrHookPackFromNpmSpec` before merging.
  - confidence: 0.87

## Security Review
