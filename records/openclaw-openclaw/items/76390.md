## Summary

Keep open: the PR targets a real Brave install regression and the resolver direction is sound, but the new catalog-to-npm path drops catalog identity/integrity metadata that other catalog install flows preserve.

## What This Changes

The PR maps bare official catalog plugin ids such as `brave` to their catalog npm package before raw npm fallback, with CLI tests and a changelog entry.

## Best Possible Solution

Update the PR so catalog-resolved npm installs carry the expected plugin id and expected integrity through to `installPluginFromNpmSpec`, then keep the existing Brave, explicit npm, and bundled precedence coverage.

## Reproduction Assessment

Yes, from source inspection and the linked issue context: in packaged-style installs without a bundled Brave source, current main falls through from bare `brave` to raw npm while the catalog already maps `brave` to `@openclaw/brave-plugin`. I did not execute the install path in this read-only review.

## Solution Assessment

No, not yet: resolving official ids before npm is the right narrow fix, but the implementation should preserve catalog safety metadata instead of calling the npm installer with only the resolved spec.

## Review Findings

Overall correctness: patch is incorrect
