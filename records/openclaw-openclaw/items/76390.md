## Summary

Keep open: the PR targets a real packaged-install regression, but the current patch adds a catalog-to-npm route without preserving the catalog's expected plugin id and integrity metadata, so it should not merge as-is.

## What This Changes

The PR resolves bare official external plugin ids such as `brave` through the official external catalog before raw npm fallback in `openclaw plugins install`.

## Best Possible Solution

Keep the catalog-first install behavior, but thread catalog `pluginId` and `expectedIntegrity` through the plan/helper into `installPluginFromNpmSpec` and add regression coverage for an integrity-pinned catalog entry.

## Reproduction Assessment

Yes, from source inspection: current main falls through from bare `brave` to raw npm when no bundled Brave source is available, while the official catalog already maps `brave` to `@openclaw/brave-plugin`. I did not execute the CLI because this review is read-only and `pnpm` is unavailable in the environment.

## Solution Assessment

No, not yet: resolving official ids before npm is the right narrow direction, but the patch should preserve the catalog trust contract by carrying expected plugin id and integrity into the npm installer.
