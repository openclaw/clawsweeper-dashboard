## Label Justifications

- `P2`: The issue is a concrete platform improvement for transcript recovery and plugin/audit tooling, but it is not an urgent runtime outage.
- `impact:session-state`: The affected state is the durable association between a logical session key and its transcript file across resets and rotations.
- `impact:data-loss`: The report describes historical transcript identity becoming unrecoverable after `sessions.json` no longer maps the old session id.

## Summary

Keep open: current main and latest release still write session transcript headers without `sessionKey` or `agentId`, and the earlier implementation attempt at https://github.com/openclaw/openclaw/pull/56082 was closed unmerged. The request remains valid, but persisting logical session keys can expose channel/user/topic identifiers in portable transcript files, so it needs maintainer approval for the metadata/privacy contract before a fix PR.

## What This Changes

The issue requests persisting `sessionKey` and related identity metadata in session transcript JSONL headers so orphaned transcripts remain self-identifying.

## Best Possible Solution

Define a small backward-compatible transcript-header contract for normalized `sessionKey` and derived `agentId`, then update every header writer plus docs and focused transcript/reset/compaction tests together.

## Reproduction Assessment

Yes at source level: current main's shared header helper and all inspected writer paths omit `sessionKey` and `agentId`. A live repro would be opening any newly created session JSONL header, but this read-only review did not create files.

## Solution Assessment

Unclear until maintainers approve the persisted metadata/privacy contract. If approved, the maintainable fix is a shared header contract used by all writers, not plugin-local mapping files or a one-path patch.
