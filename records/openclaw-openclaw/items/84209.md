## Label Justifications

- `P2`: This is a concrete platform improvement with clear affected files and recovery/audit value, but it is not an urgent runtime outage.
- `impact:data-loss`: The reported impact is loss of the durable mapping between transcript files and their logical session identity after reset or store mutation.
- `impact:session-state`: The affected behavior is session transcript identity and state reconstruction across session id rotations and compaction.

## Summary

Keep open: current `main` and the latest release still write JSONL session headers without logical session identity, while the remaining choice is a new persisted metadata contract rather than a narrow bug fix.

## What This Changes

Persist `sessionKey`, and likely derived `agentId`, in session transcript JSONL headers so transcript files can identify their logical conversation without `sessions.json`.

## Best Possible Solution

Define a small transcript-header metadata contract, write normalized `sessionKey` and derived `agentId` on every new header path, document the privacy implications, and use the open restore PR only if it satisfies that contract or can be split cleanly.

## Reproduction Assessment

Yes, from source inspection: the current header writers construct JSONL session headers without `sessionKey` or `agentId`, and the latest release has the same shape. I did not run a live gateway because this review is read-only and must not create session files.

## Solution Assessment

Unclear until maintainers settle the persisted metadata contract. Adding `sessionKey` and derived `agentId` across all header creation paths looks like the right direction, but privacy, versioning, and coverage of reset/compaction paths should be decided before landing.
