## Summary

Keep open: current main and the latest shipped tag still write session transcript headers without `sessionKey` or `agentId`, and the earlier implementation PR was closed unmerged. The request is source-reproducible, but the persisted metadata/privacy contract needs maintainer approval before code should move.

## What This Changes

The issue requests adding persistent `sessionKey` and possibly `agentId` metadata to session transcript JSONL headers for orphan recovery, plugin identity, audit, and portability.

## Best Possible Solution

Define a small backward-compatible transcript-header metadata contract, write normalized `sessionKey` and derived `agentId` consistently on every new transcript header path, and document the privacy/portability tradeoff with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the production transcript header writers still emit JSONL session headers without `sessionKey` or `agentId`; I did not run a live gateway because this read-only review should not create artifacts in the checkout.

## Solution Assessment

Unclear until maintainers settle the public contract. Adding `sessionKey` and derived `agentId` across all header creation paths looks maintainable, but the exact fields, versioning, and metadata exposure need approval first.

## Risks / Open Questions

- Persisting `sessionKey` can expose channel, user, topic, or job identifiers wherever transcript files or support bundles are shared, so maintainers should approve the privacy contract before implementation.
- A partial implementation would leave some transcript creation paths self-identifying and others anonymous unless normal append, direct append, embedded/bootstrap, reset, fork, and compaction successor paths are covered together.
- Existing docs and tooling currently treat `sessions.json` as the durable `sessionKey -> sessionId` authority, so the new header contract needs docs and regression coverage to avoid competing interpretations.
