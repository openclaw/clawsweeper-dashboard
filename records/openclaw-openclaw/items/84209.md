- `impact:data-loss`: The issue is about losing the durable mapping between transcript files and their logical session identity after reset or store mutation.

## Summary

Keep open: current main and the latest release still write JSONL session headers without session identity, and the earlier duplicate cleanup accidentally left the remaining feature without a canonical tracker. The request is well supported by source inspection, but it needs a maintainer decision on the persisted metadata contract and privacy/versioning before implementation.

## What This Changes

The issue requests persisting `sessionKey`, and optionally `agentId`, in JSONL session transcript headers so orphaned transcripts can self-identify after reset or rotation without `sessions.json`.

## Best Possible Solution

Define a small transcript-header metadata contract, write normalized `sessionKey` and derived `agentId` on every new header path, preserve old-reader compatibility, and document the metadata/privacy implications with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: the current header writers construct JSONL session headers without `sessionKey` or `agentId`, and the latest release has the same shape. I did not run a live gateway because this read-only review must not create session files.

## Solution Assessment

Unclear until maintainers settle the public contract. Adding `sessionKey` and derived `agentId` across all header creation paths looks like the right direction, but field set, versioning, and metadata exposure should be decided before a PR lands.
