## Summary

Keep open: current main and the latest shipped release still write transcript JSONL headers without `sessionKey` or `agentId`, the prior duplicate pair closed without leaving a canonical open item, and the earlier implementation PR was closed unmerged. The request is source-confirmed, but persisting session identity needs maintainer approval for the header metadata and privacy contract before implementation.

## What This Changes

The issue requests persisting `sessionKey` and optionally `agentId` in JSONL transcript session headers so transcript files remain self-identifying after store rotation or reset.

## Best Possible Solution

Define a small backward-compatible transcript-header metadata contract, write normalized `sessionKey` and derived `agentId` on every new header path, and document the privacy/portability tradeoff with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows each production header writer still emits a JSONL session header without `sessionKey` or `agentId`; I did not create live session files because this review is read-only.

## Solution Assessment

Unclear until maintainers settle the public contract. Adding `sessionKey` and derived `agentId` across all header creation paths looks maintainable, but the exact fields, versioning, and metadata exposure need approval first.

## Risks / Open Questions

- Persisting `sessionKey` can expose channel, user, topic, job, or plugin identifiers anywhere transcript files or support bundles are shared, so the field set needs an explicit privacy decision.
- A partial implementation would leave some transcript files self-identifying and others anonymous unless normal append, direct append, embedded/bootstrap, reset, and compaction successor paths are covered together.
- The public transcript-header contract and docs do not currently promise these fields, so readers and support tooling need a deliberate compatibility story.
