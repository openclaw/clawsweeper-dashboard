## Summary

Keep open: the PR is the right narrow fix for the Control UI `/new` hook regression, but the changelog entry still points at superseded unmerged PR #77004 instead of this corrected PR.

## What This Changes

The PR adds an explicit `emitCommandHooks` opt-in to `sessions.create`, passes it from the Control UI `/new` session-creation path, emits the command/reset/session hook chain for that opted-in path, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land the explicit `emitCommandHooks` gate with the added regression coverage after correcting the release-note PR reference or removing the superseded PR number.

## Reproduction Assessment

Yes, source-level. Current main routes typed Control UI `/new` into `sessions.create`, and that handler creates a child session without the documented `command:new` hook path used by `session-memory`.

## Solution Assessment

Yes, the explicit opt-in parameter is the narrow maintainable fix because only the Control UI command path should restore webchat hook delivery; ordinary SDK parent-session creates should stay hook-free unless explicitly opted in.
