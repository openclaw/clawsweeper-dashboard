The PR adds an explicit `emitCommandHooks` opt-in to `sessions.create`, passes it from the Control UI `/new` path, emits command/reset/session hooks for that path, and adds gateway/UI coverage plus a changelog entry.

## Best Possible Solution

## Reproduction Assessment

Yes, source-level. Current main routes Control UI typed `/new` into `sessions.create`, and that handler creates the child session without the documented `command:new` hook path used by `session-memory`.

## Solution Assessment

Yes, the explicit opt-in parameter is the narrow maintainable fix because only the Control UI command path should restore webchat hook delivery; ordinary SDK parent-session creates should stay hook-free unless explicitly opted in.
