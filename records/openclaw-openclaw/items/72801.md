## Summary

Keep open. Current main still generates an empty isolated Codex ACP config, so the user problem is real, but this draft PR is dirty against main and its parser only handles one explicit table shape while missing supported Codex project-trust TOML forms and the required changelog entry.

## What This Changes

The PR changes the ACPX Codex auth bridge to copy trusted Codex project declarations and the ACPX workspace cwd into the isolated Codex home while adding auth-bridge tests that auth and notify config remain excluded.

## Best Possible Solution

Keep the fix owner-local in `extensions/acpx`: inherit only Codex trusted project declarations plus the active workspace cwd into the isolated Codex home, handle Codex-supported TOML project shapes, keep auth/notify/profile config excluded, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is to put a trusted project in `CODEX_HOME/config.toml` and call `prepareAcpxCodexAuthConfig`; current main writes only a generated header to the isolated config, so the trusted project cannot propagate.

## Solution Assessment

No, not as written. The direction is maintainable, but the line-based parser is too narrow for Codex-supported TOML shapes; the safer repair is a rebased ACPX-local extraction helper with coverage for explicit tables, inline `[projects]` entries, and top-level inline `projects` maps.
