## Summary

Keep open: current main still drops Codex trusted-project config when ACPX creates its isolated CODEX_HOME, and this PR is the right owner-local direction but remains incomplete because it parses only one TOML shape and lacks the required changelog entry.

## What This Changes

The PR updates the ACPX Codex auth bridge and tests to copy trusted Codex project declarations plus the workspace cwd into the isolated Codex home while continuing to exclude auth, notify hooks, and other source Codex config.

## Best Possible Solution

Keep the fix in extensions/acpx with TOML-aware extraction/rendering of only trusted project declarations plus the active cwd, preserve auth/notify exclusion, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: put a trusted project entry in CODEX_HOME/config.toml and call prepareAcpxCodexAuthConfig; current main writes an isolated config containing only the generated header, so the trust entry is lost.

## Solution Assessment

No, not as written. The direction is maintainable and owner-local, but the implementation should parse Codex TOML semantically across supported project-map shapes and include the required changelog coverage.
