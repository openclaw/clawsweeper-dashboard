## Summary

Keep open: this PR addresses a real ACPX Codex trust inheritance gap on current main, but the draft patch is incomplete because its extractor only handles one TOML shape and it lacks the required user-facing changelog entry.

## What This Changes

The PR changes the ACPX Codex auth bridge to inherit trusted Codex project declarations plus the active workspace cwd into the isolated CODEX_HOME config and adds bridge tests for that behavior.

## Best Possible Solution

Keep the fix owner-local in extensions/acpx, parse Codex config with TOML semantics, copy only trusted project declarations plus the active cwd into the isolated CODEX_HOME, and add focused tests plus a single-line changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is to create CODEX_HOME/config.toml with a trusted project entry, call prepareAcpxCodexAuthConfig, and inspect the generated stateDir/acpx/codex-home/config.toml; current main writes only the generated header.

## Solution Assessment

No, not as written. The direction is right, but the implementation should handle Codex-supported TOML shapes and be rebased onto current main so it does not drift from the current ACPX wrapper behavior.
