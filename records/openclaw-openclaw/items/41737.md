## Best Possible Solution

Rework the endpoint lifecycle as a vLLM plugin/provider setup improvement on current main, split unrelated infrastructure/security changes, and require redacted real vLLM setup proof before merge.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows normal vLLM auth choices are consumed by the plugin-provider handler before the new vLLM handler can run, and the A2UI dlx fallback can fail after a successful temporary bundle run. The requested feature itself is not a bug reproduction; current main simply lacks that lifecycle flow.

## Solution Assessment

No. The requested capability is reasonable, but this branch is not the best merge path until it is rebased onto the current plugin/provider setup contract, the unrelated security/build/test-infra changes are split out, and real behavior proof is added.
