The PR adds a vLLM endpoint management flow with multi-endpoint/model selection and stale-default cleanup, plus related docs/tests and unrelated A2UI, memory-test, secret-baseline, and zizmor changes.

## Best Possible Solution

Rework this as a focused vLLM plugin/provider setup change on current main, split unrelated infra/security/test churn, wire it through the current auth dispatcher, and add redacted real endpoint proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows normal vLLM choices are consumed by the plugin-provider handler before the new lifecycle handler can run, and the A2UI fallback can exit after invoking pnpm dlx. The requested vLLM lifecycle flow itself is feature work, not a bug reproduction.

## Solution Assessment

No. The capability is reasonable, but this branch is not the best merge path until it is rebased onto the current plugin/provider setup contract, unrelated security/build/test changes are split out, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route vLLM choices through the lifecycle handler:** `src/commands/auth-choice.apply.vllm.ts:114-125`
  - body: `applyAuthChoiceVllm` is added but never registered in `applyAuthChoice`'s handler list. Because the plugin-provider handler runs first and resolves `vllm`, users still hit the existing self-hosted plugin setup and bypass the new endpoint-management and stale-cleanup logic.
  - confidence: 0.93
- **[P2] Stop after the pnpm dlx fallback succeeds:** `scripts/bundle-a2ui.sh:95-103`
  - body: This fallback already runs `pnpm -s dlx rolldown`, then requires a persistent `.pnpm` CLI and invokes bundling again. If that local CLI is absent, the script can fail after the fallback command has already done the bundle work, or run the bundle twice when it is found.
  - confidence: 0.86
