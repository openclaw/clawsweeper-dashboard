## Summary

Keep this PR open because the vLLM endpoint-management feature is not already on current main, but the branch is not mergeable as-is: the new vLLM setup handler is not reached by normal auth dispatch and the branch mixes unrelated build/security/test-infra changes into the provider feature.

## What This Changes

The PR adds a core vLLM endpoint-management setup flow with multi-endpoint provider keys, model scanning/selection, stale default cleanup, docs/tests, and unrelated A2UI, memory-test, secret-baseline, and zizmor changes.

## Best Possible Solution

Rework the endpoint lifecycle as a vLLM plugin/provider setup flow, split unrelated A2UI/zizmor/memory-test changes, and require redacted real setup proof before merge.

## Reproduction Assessment

Yes for the blocking PR defects. Source inspection at PR head shows the new vLLM handler is present but not registered in auth dispatch, and the A2UI fallback logic can fail after a successful dlx run.

## Solution Assessment

No. The desired vLLM feature is reasonable, but this PR is not the best merge path until it is rebased onto the plugin/provider setup contract and unrelated build/security/test-infra changes are split out.
