## Summary

Keep open: current main still uses plugin-owned single-endpoint vLLM setup, and this PR is not merge-ready because the new vLLM management handler is unreachable while unrelated build/security/test-infra changes are mixed into the branch.

## What This Changes

The PR adds interactive vLLM endpoint management, multi-endpoint provider keys, stale default cleanup, docs, tests, and unrelated A2UI, SQLite-test, secret-baseline, and zizmor changes.

## Best Possible Solution

Move the useful vLLM endpoint-management work into the plugin/provider setup surface, add only generic core seams that third-party plugins can use, keep focused tests/docs, and split unrelated CI/build/test-infra changes into separate PRs.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the new vLLM handler is present but not registered, while the plugin-provider handler resolves the vLLM choice first.

## Solution Assessment

No. The requested vLLM capability is reasonable, but this PR needs a rebase onto current plugin-owned setup surfaces and should split unrelated build/security/test-infra changes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route vLLM choices through the setup handler:** `src/commands/auth-choice.apply.vllm.ts:114-125`
  - body: `applyAuthChoiceVllm` is added here, but `applyAuthChoice` never registers it and still runs the plugin-provider handler first. Provider choices such as `vllm` are consumed before this code can run, so the new endpoint-management and stale-cleanup flow is skipped in normal setup.
  - confidence: 0.93
- **[P2] Stop after the pnpm dlx Rolldown fallback:** `scripts/bundle-a2ui.sh:95-103`
  - body: This fallback already runs `pnpm -s dlx rolldown`, then looks for a persistent `.pnpm` CLI and runs bundling again. Environments without that pnpm layout can fail after a successful fallback, while pnpm installs can bundle twice.
  - confidence: 0.86
