## Summary

Keep open: current main still lacks the requested multi-endpoint vLLM lifecycle flow, but this PR is not safe to merge because the new vLLM handler is unreachable and the branch mixes unrelated build/security policy changes into the feature.

## What This Changes

The PR adds vLLM endpoint lifecycle management, multi-endpoint provider keys, model selection, stale default cleanup, docs/tests, and also changes A2UI bundling, SQLite-gated memory tests, secret baseline data, and zizmor configuration.

## Best Possible Solution

Rebase the useful vLLM endpoint-management work onto current main's plugin-owned provider setup architecture, wire it through the production provider auth/model-picker path, preserve SecretRef/auth-profile semantics, cover stale default and fallback cleanup with focused tests, update public vLLM docs, and split build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. The blocking PR behavior is source-reproducible: at PR head the new vLLM setup handler is never registered, while the plugin-provider handler still consumes the vLLM choice first.

## Solution Assessment

No. The requested vLLM capability is reasonable, but this PR is not the best merge shape until it is rebased onto current plugin/provider setup surfaces and unrelated infrastructure changes are separated.
