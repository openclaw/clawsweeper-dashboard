## Summary

Keep open. Current main still has only the single-provider vLLM setup path, so the feature is not implemented on main; however, the PR is not merge-ready because the new vLLM setup handler is unreachable in the PR head and the branch mixes unrelated build/scanner/test-policy changes into provider work.

## What This Changes

The PR adds vLLM endpoint lifecycle management, multi-endpoint provider keys, model discovery/selection, stale vLLM default cleanup, docs, and tests, while also changing A2UI bundling, secret-scan baseline data, SQLite-gated memory tests, and zizmor configuration.

## Best Possible Solution

Rebase the useful vLLM work onto current main's plugin/provider setup architecture, implement endpoint lifecycle management through the production provider auth/model-picker path, preserve SecretRef/auth-profile semantics, keep stale default and fallback cleanup covered by focused tests, update the public vLLM docs, and split build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. The blocking behavior is reproducible by source inspection: at PR head, selecting vLLM is handled by the plugin-provider auth path before the new applyAuthChoiceVllm helper can run, so the endpoint management flow and stale cleanup are skipped.

## Solution Assessment

No. The requested vLLM behavior is reasonable, but this PR is not the best merge shape until it is rebased onto current plugin-owned provider setup surfaces and unrelated infrastructure changes are separated.
