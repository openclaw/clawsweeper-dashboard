## Summary

Keep open: current main does not implement multi-endpoint vLLM management, and this PR is still not merge-ready because normal vLLM choices route through the plugin-provider setup path before the new handler and the branch mixes unrelated build, scanner, and test-infra changes.

## What This Changes

The PR adds a vLLM endpoint-management setup flow with multi-endpoint keys, model scanning and selection, stale default cleanup, docs, tests, and extra build/CI/test-infra changes.

## Best Possible Solution

Rework the vLLM endpoint-management flow through the plugin/provider setup contract, keep only generic core seams, and split unrelated script, scanner, and test-infra changes.

## Reproduction Assessment

Yes. Source inspection of the PR-head diff and current-main dispatcher shows the new vLLM handler is present in the PR but is not reached by normal vLLM auth choices because plugin-provider auth dispatch runs first.

## Solution Assessment

No. The desired vLLM capability is reasonable, but this PR is not the best merge path until it is rebased onto the current plugin-owned setup surface and unrelated build/security/test-infra edits are split out.
