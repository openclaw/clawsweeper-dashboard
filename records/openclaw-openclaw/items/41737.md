## Summary

Keep open. The vLLM management feature is still not present on current main, but this PR is not mergeable as-is because the new vLLM setup handler is not on the production auth path and the branch mixes unrelated build/security/test-infra changes into a stale provider-architecture patch.

## What This Changes

The PR branch rewrites vLLM interactive setup for endpoint lifecycle management and multi-endpoint model selection, adds stale-default cleanup/tests/docs, and also touches A2UI bundling, memory SQLite test gating, secret-scan baseline data, and zizmor configuration.

## Best Possible Solution

Rebase the useful vLLM endpoint-management work onto current main's plugin-owned provider setup surface, keep SecretRef/auth-profile semantics intact, cover stale default and fallback pruning with focused tests, update public docs and changelog, and split build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. Source inspection against PR head shows applyAuthChoice only dispatches through loaded plugin providers and the added applyAuthChoiceVllm helper is not registered, so selecting vLLM does not enter the new endpoint lifecycle path.

## Solution Assessment

No. The requested vLLM behavior is reasonable, but this PR should be rebased and narrowed onto the current plugin/provider setup architecture instead of merging an XL branch with unrelated build and security-policy edits.
