The PR adds a vLLM setup lifecycle flow with saved endpoints, multi-model discovery/selection, stale default cleanup, docs/tests, and also modifies A2UI bundling, memory tests, the secret baseline, and zizmor config.

## Best Possible Solution

Rework this as a focused vLLM plugin/provider setup change on current main, split unrelated infra/security/test churn, wire the flow through the current auth dispatcher, and add redacted real endpoint proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows normal vLLM choices are consumed by the plugin-provider handler before the new lifecycle handler can run, and the A2UI dlx fallback can fail after a successful temporary run. The requested vLLM lifecycle flow itself is a feature request, not a bug reproduction.

## Solution Assessment

No. The requested capability is reasonable, but this branch is not the best merge path until it is rebased onto the current plugin/provider setup contract, unrelated security/build/test changes are split out, and real behavior proof is added.
