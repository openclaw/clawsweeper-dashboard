## Summary

Keep open. Current main still lacks the requested vLLM endpoint lifecycle flow, but this PR is not correct or mergeable as-is because the new vLLM handler is unreachable, the branch is dirty against current main, and it mixes unrelated build/security-policy changes into the provider feature.

## What This Changes

The PR adds a managed vLLM setup flow with endpoint CRUD, multi-endpoint keys, model discovery/selection, stale default cleanup, docs, tests, and incidental A2UI, memory-test, secret-scan, and zizmor changes.

## Best Possible Solution

Rebase the useful vLLM work onto current main's plugin/provider setup architecture, route endpoint management through the production provider setup path, preserve SecretRef/auth-profile semantics, cover stale default and fallback cleanup with focused tests, update public docs and changelog, and split unrelated infra/security changes.

## Reproduction Assessment

Yes. The main blocker is reproducible by source inspection: at PR head, selecting vLLM is handled by the existing plugin-provider dispatch before the new applyAuthChoiceVllm helper can run.

## Solution Assessment

No. The requested vLLM lifecycle behavior is reasonable, but this PR is not the best merge shape until it is rebased onto current provider-plugin surfaces and unrelated build/security/test-infra changes are separated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route vLLM choices through the setup handler:** `src/commands/auth-choice.apply.vllm.ts:121-125`
  - body: `applyAuthChoiceVllm` is added, but `applyAuthChoice` never imports or registers it; the first handler still consumes plugin choices like `vllm`. Normal vLLM setup therefore skips the endpoint-management menu and stale-model cleanup entirely.
  - confidence: 0.93
- **[P2] Stop after the pnpm dlx fallback:** `scripts/bundle-a2ui.sh:95-103`
  - body: The fallback branch already runs `pnpm dlx rolldown`, then requires a project-local `.pnpm` Rolldown CLI and runs the bundle again. Layouts without `.pnpm` can fail after the temporary bundle succeeds, while pnpm layouts run it twice.
  - confidence: 0.86
