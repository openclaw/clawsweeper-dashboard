## Summary

Keep open for maintainer review. Current main does not implement the requested multi-endpoint vLLM lifecycle, but this PR is not merge-ready because the new vLLM handler is not wired into auth dispatch and the branch mixes unrelated build, scanner, and test-infra changes.

## What This Changes

The branch adds vLLM endpoint management and stale default cleanup with docs/tests, while also changing A2UI bundling, SQLite-gated memory tests, secret-scan baseline data, and zizmor configuration.

## Best Possible Solution

Rebase and split the vLLM endpoint lifecycle work onto the current plugin-owned provider setup path, preserve SecretRef/auth-profile semantics, cover stale defaults and fallbacks with focused tests, update public vLLM docs, and move A2UI/zizmor/test-infra changes to separate reviews.

## Reproduction Assessment

Yes. Source inspection at PR head shows the new vLLM setup handler is not imported or registered by auth-choice dispatch, while selecting vLLM is handled by the plugin-provider auth path before the new lifecycle code can run.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best merge shape until it is rebased onto current main's plugin/provider setup architecture and unrelated build/security/test-infra changes are separated.

  - body: `applyAuthChoiceVllm` contains the new endpoint-management flow, but `applyAuthChoice` never imports or registers it, and the plugin-provider handler still consumes `vllm` choices first. Normal vLLM setup therefore skips the endpoint menu and stale-model cleanup entirely.
