## Summary

Keep open. The vLLM endpoint-management goal is still not implemented on current main, and the PR contains useful work plus many addressed review follow-ups, but the current PR head is not merge-ready: the new vLLM handler is not registered in the production auth-choice dispatch path, the A2UI fallback can fail after a successful dlx bundle, and the branch mixes unrelated build/security/test-policy edits into the provider change.

## What This Changes

The PR adds a vLLM endpoint management setup flow with multi-endpoint keys, model discovery/selection, stale default cleanup, docs, and tests, while also changing A2UI bundling, secret-scan baseline entries, SQLite memory tests, and zizmor configuration.

## Best Possible Solution

Rebase the useful vLLM provider work onto current main's bundled vLLM plugin/provider setup surface, wire endpoint management through the production auth and model-picker paths, keep stale vLLM default and fallback cleanup covered by focused tests, update the public vLLM docs, and split build/security/test-infra edits into separate reviews.

## Reproduction Assessment

Yes. The blocking behavior is reproducible by source inspection: at PR head, selecting vLLM goes through applyAuthChoiceLoadedPluginProvider in the registered handler list, while the new applyAuthChoiceVllm function is never imported or registered.

## Solution Assessment

No. The requested vLLM behavior is reasonable, but this PR is not the best merge shape until it is rebased onto the current plugin-owned provider setup architecture and unrelated build/security/test-policy edits are separated.
