## Summary

Keep open for maintainer review: source and dependency inspection support this as a focused Matrix bug fix with sufficient live proof and no line-level blocking findings, but it changes long-running Matrix retry/session state and overlaps with another open candidate PR.

## What This Changes

The PR updates the Matrix decrypt bridge to retain exhausted failed-decryption retry state, requeue it on Matrix crypto key signals via `MatrixEvent.attemptDecryption`, clear retained state on stop, and add focused regression tests.

## Best Possible Solution

Land this focused Matrix decrypt-bridge retry fix after owner review accepts the retry-state behavior, then close the linked bug and supersede the broader overlapping candidate if maintainers choose this path.

## Reproduction Assessment

Yes, at source level: current main exhausts failed decryptions into a non-retryable set while later crypto key signals only process the pending retry map. I did not run a live current-main Matrix reproduction in this read-only review, but the PR body supplies live after-fix Synapse output for the recovery path.

## Solution Assessment

Yes, this appears to be the best current fix shape: it stays in the Matrix decrypt retry owner, uses the matrix-js-sdk event retry contract that bypasses the `decryptEventIfNeeded` `clearEvent` guard, and avoids broad sync error string classification. A formal API seam remains a maintainer ownership choice rather than a concrete patch defect.
