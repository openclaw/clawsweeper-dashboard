## Summary

Keep this PR open, but it is not merge-ready: the chosen retry shape is dependency-incompatible because it requeues the same MatrixEvent after matrix-js-sdk has marked the failed decrypt with clearEvent, so a later `decryptEventIfNeeded()` call can resolve without attempting decryption.

## What This Changes

The PR changes the Matrix decrypt bridge to retain exhausted decrypt retry state, requeue it on Matrix crypto key signals, clear retained exhausted state on stop, and add Matrix bridge/lifecycle regression tests.

## Best Possible Solution

Revise the Matrix decrypt-bridge fix so late key availability uses a matrix-js-sdk-supported retry path, prove it with a regression that exercises the real SDK event contract, and add redacted live encrypted-room recovery proof before merge.

## Reproduction Assessment

No for the original live Docker/Matrix outage: this review did not reproduce it against a real homeserver. Yes for the PR defect: source inspection of matrix-js-sdk 41.6.0 shows a failed decrypt sets `clearEvent`, which makes `shouldAttemptDecryption()` false for the same event.

## Solution Assessment

No. Keeping retry ownership in the decrypt bridge is the right layer, but requeueing the same failed MatrixEvent is not the best fix unless the code first uses a dependency-supported way to make that event decryptable again.
