## Summary

Keep open. Current main still exposes `memorySearch.fallback` as a provider string and reuses the resolved primary `remote`/provider options in both startup and runtime fallback paths, so the Gemini-primary plus OpenAI-compatible fallback endpoint case is still not configurable independently; #63990 is broader multi-index work, not a replacement for this narrow config gap.

## What This Changes

Request support for separate memorySearch fallback provider, model, and remote endpoint settings so primary and fallback embedding providers can use independent configuration.

## Best Possible Solution

Add a backward-compatible fallback object form that preserves string and `"none"` configs while resolving fallback provider, model, remote, local, input-type, and secret metadata independently across schema, runtime, docs, generated config metadata, and focused tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the strict schema/type contract, and the existing string fallback paths reuse the primary resolved `remote` settings.

## Solution Assessment

Yes for this narrow issue. An additive fallback object is the smallest maintainable fix; the multi-index/model-aware failover design should remain in #63990 rather than being mixed into this repair.

## Repair Work Prompt

Fix the memorySearch fallback remote config gap tracked by https://github.com/openclaw/openclaw/issues/43760. Add a backward-compatible fallback object form, for example `memorySearch.fallback: { provider, model?, remote?, local?, inputType?, queryInputType?, documentInputType?, outputDimensionality? }`, while preserving existing string fallback ids and `"none"`. Resolve primary and fallback provider settings independently so a Gemini primary can use env/default Gemini auth while an OpenAI-compatible fallback can use its own `remote.baseUrl` and `remote.apiKey`. Update public config types, zod schema, generated schema/help/labels/sensitive metadata, runtime startup fallback, runtime provider-state fallback, docs, and focused tests. Keep this single-index behavior narrow; do not implement the multi-index/model-aware failover design from #63990. Add a user-facing changelog entry unless the final diff is test-only.
