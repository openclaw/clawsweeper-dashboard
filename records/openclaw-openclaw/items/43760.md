## Summary

Keep open. Current main still has the reported memorySearch limitation: fallback is a provider string in the public config contract, and both startup and runtime fallback paths reuse the same resolved remote/options block. The related implementation PR is closed unmerged, and the broader related issue is also closed, so this narrower fallback-provider/model/remote gap remains active.

## What This Changes

Add a way for memorySearch fallback embeddings to use independent provider, model, remote endpoint, headers, input, dimensionality, and local settings while preserving the existing string fallback shorthand.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that keeps fallback: "provider" and "none" working, but lets object fallback settings resolve independently through schema, public types, runtime provider creation, generated config metadata, docs, and regression tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the schema/type contract, and the existing fallback paths reuse the single resolved remote/settings block.

## Solution Assessment

Yes. A fallback object is the narrowest maintainable solution because it preserves existing configs and avoids a growing set of separate top-level keys such as fallbackRemote and fallbackModel.
