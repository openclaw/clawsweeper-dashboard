## Summary

Keep open. Current main still has the exact narrow config gap: `memorySearch.fallback` is a string, fallback activation reuses the shared `memorySearch.remote`, and the closed implementation PR was explicitly superseded in favor of this issue rather than merged.

## What This Changes

Request support for an independent `memorySearch` fallback provider/model/remote configuration so a Gemini primary does not inherit an OpenAI-compatible fallback endpoint.

## Best Possible Solution

Add a backward-compatible fallback object shape for provider/model/remote overrides on current main, while leaving multi-index failover semantics to #63990.

## Reproduction Assessment

Yes, for the reported feature gap: source inspection shows one shared `memorySearch.remote` object is passed to both the primary and fallback provider paths, and Gemini consumes that base URL. I did not run a live SiliconFlow/Gemini request in this read-only review.

## Solution Assessment

Yes, for this narrow issue: a backward-compatible `fallback: { provider, model?, remote? }` shape is the maintainable fix already identified by the superseded PR review. The broader model-aware multi-index design should stay separate in #63990.
