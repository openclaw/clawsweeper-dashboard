## Summary

Keep open. Current main still exposes memorySearch fallback as a string provider id and reuses one resolved remote/options block for primary and fallback embedding providers; the open multi-index issue is broader and does not supersede this narrower config gap.

## What This Changes

Request separate provider, model, and remote settings for memorySearch fallback embeddings so a Gemini primary can fall back to an OpenAI-compatible endpoint such as SiliconFlow.

## Best Possible Solution

Add a backward-compatible fallback object form that resolves fallback provider, model, remote, local, and secret metadata independently while preserving existing string and "none" configs.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: the requested object form is outside the schema/type/docs contract, and existing startup/runtime fallback paths reuse the primary resolved settings.

## Solution Assessment

Yes. For this narrow issue, an additive fallback object is the most maintainable fix; the broader multi-index/model-space failover design should remain in #63990.
