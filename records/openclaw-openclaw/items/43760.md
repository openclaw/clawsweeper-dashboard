## Summary

Keep open. Current main still has a string-only memorySearch fallback and one shared resolved remote/options block that is passed to both primary and fallback embedding providers, so the independent fallback endpoint requested here remains unresolved; the open multi-index issue is related but broader.

## What This Changes

Support separate memorySearch fallback provider/model/remote configuration so a primary Gemini embedding provider can fall back to an OpenAI-compatible endpoint without sharing the primary remote block.

## Best Possible Solution

Add a backward-compatible fallback object form that independently resolves fallback provider, model, remote, local, input-type, and secret metadata while preserving existing string and "none" configs.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: the requested object form is outside the schema/type/docs contract, and both startup and runtime fallback paths reuse the primary resolved settings.

## Solution Assessment

Yes. An additive fallback object is the narrowest maintainable direction for this request; it should preserve existing string configs and leave the broader multi-index design to #63990.
