## Summary

Keep open. Current main still has the reported memorySearch limitation: fallback is a provider string, and both startup and runtime fallback paths reuse the same resolved remote/options block; the prior implementation PR is closed unmerged, while #63990 tracks a broader multi-index design rather than this narrow config gap.

## What This Changes

Request independent memorySearch fallback provider/model/remote configuration so a Gemini primary can use its own credentials while an OpenAI-compatible fallback uses a separate endpoint such as SiliconFlow.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that preserves string and "none" configs while resolving fallback provider, model, remote, local, and secret metadata independently across schema, public types, runtime creation, docs, and focused regression tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the schema/type contract, and the existing string fallback paths reuse the single resolved remote/options block.

## Solution Assessment

Yes. For this narrower issue, a fallback object is the most maintainable additive fix; the broader multi-index/model-space failover design should remain in #63990 rather than being mixed into this repair.
