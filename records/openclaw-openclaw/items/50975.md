## Summary

Close as superseded: the prefixless Anthropic token premise matches a closed report that the reporter resolved as a wrong-flow paste, while standard Anthropic API-key auth is already handled through the separate current-main API-key route. The branch also weakens setup-token validation and has no real behavior proof.

## What This Changes

The branch lowers Anthropic setup-token minimum length in `src/commands/auth-token.ts` and accepts non-`sk-ant-oat01-` values of at least 40 characters.

## Best Possible Solution

Keep setup-token validation scoped to supported setup-token material, use the existing Anthropic API-key auth route for standard API keys, and open a fresh auth-contract issue only if there is real proof of a supported prefixless setup-token format.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows short legacy-prefixed setup tokens skip the proposed length guard. No for the claimed new prefixless setup-token format; the related report was resolved as a wrong-flow paste and this PR provides no real Anthropic auth proof.

## Solution Assessment

No. A validator-only patch would weaken guardrails and blur setup-token versus API-key credential semantics; the maintainable path is to preserve the split unless a proven new Anthropic token contract requires a coordinated code, docs, and test change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject too-short prefixed setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so a truncated legacy setup token such as `sk-ant-oat01-x` would be accepted and stored instead of failing during paste/setup. Keep any prefix relaxation separate from an unconditional minimum-length guard.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch weakens an auth-token validation guard by allowing malformed legacy-prefixed Anthropic setup tokens through.

Concerns:

- **[low] Malformed legacy tokens bypass validation:** `src/commands/auth-token.ts:33`
  - body: Because the proposed length check only applies to prefixless values, too-short `sk-ant-oat01-` setup tokens would pass upfront validation and be stored, weakening the existing credential guardrail and making later auth failures harder to diagnose.
  - confidence: 0.9
