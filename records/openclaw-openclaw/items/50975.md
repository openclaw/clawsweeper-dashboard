## Summary

Keep this PR open, but it is not mergeable as-is: setup-token validation now lives in the shared provider helper, and this branch would weaken the legacy setup-token guard while leaving Anthropic API-key versus setup-token runtime semantics unresolved.

## What This Changes

The PR relaxes Anthropic setup-token validation in the command auth-token helper by lowering the minimum length and allowing values without the legacy `sk-ant-oat01-` prefix.

## Best Possible Solution

Confirm the supported Anthropic credential formats, keep API-key and setup-token modes distinct, then update the shared helper, Anthropic runtime classification, docs, and focused tests together.

## Reproduction Assessment

Yes for the PR regression by source inspection: `sk-ant-oat01-x` would skip the branch length check because it has the legacy prefix. Current main still rejects that value through the shared validator.

## Solution Assessment

No. The patch changes a stale implementation target, weakens validation for malformed legacy tokens, and does not update the active shared helper, runtime credential classification, or focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject too-short prefixed setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values such as `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep prefix relaxation separate from an unconditional minimum-length check.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff weakens an auth-token validation guard by allowing malformed legacy-prefixed Anthropic setup tokens through.

Concerns:

- **[low] Malformed legacy tokens bypass validation:** `src/commands/auth-token.ts:33`
  - body: Because length validation only applies to prefixless tokens, a too-short value with the legacy Anthropic prefix would pass upfront validation and be stored as a credential, making later auth failures harder to diagnose and weakening the existing guardrail.
  - confidence: 0.9
