## Summary

Keep open, but do not merge this branch as-is: current main still has the underlying Anthropic auth friction, while this PR is stale against the shared validator and weakens the existing setup-token guard.

## What This Changes

The PR lowers Anthropic setup-token validation from 80 to 40 characters and accepts tokens that do not start with `sk-ant-oat01-`.

## Best Possible Solution

Confirm the credential formats, keep API-key and setup-token modes distinct, and update the shared validator/runtime/tests only for the credential path that is actually supported.

## Reproduction Assessment

Yes. The PR regression is reproducible by inspection with `sk-ant-oat01-x`: the proposed guard skips the length check because the prefix matches, while current main rejects that value as too short.

## Solution Assessment

No. The patch changes a stale implementation location, weakens validation for malformed legacy tokens, and does not update the downstream Anthropic runtime behavior or focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject too-short prefixed setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values such as `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep prefix relaxation separate from an unconditional minimum-length check.
  - confidence: 0.94

## Security Review

Status: needs_attention
