## Summary

Keep open. Current main still has the reported split: `models status` can report local OAuth profile health from expiry metadata while the Codex app-server path forces a real OAuth refresh on the first agent turn, so the reporter’s 401 failure is still plausible and not fully covered by the older closed OAuth-sync issues.

## What This Changes

Fresh OpenAI Codex OAuth or device-code login should make the first real `openclaw agent` request succeed, or status should avoid reporting the profile as healthy when the runtime refresh path cannot use it.

## Best Possible Solution

Fix the Codex app-server OAuth refresh/login-token contract so fresh browser and device-code credentials survive the first real request, and make status/probe output distinguish local profile metadata from app-server viability.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The reporter provides concrete current-version steps, and source inspection shows the local-status path plus forced app-server refresh path that can produce the mismatch.

## Solution Assessment

Unclear because there is no patch yet and the failing 401 needs live/provider-contract investigation. The maintainable direction is to fix the Codex runtime refresh path first, then adjust status/probe reporting if the runtime cannot validate tokens cheaply.
