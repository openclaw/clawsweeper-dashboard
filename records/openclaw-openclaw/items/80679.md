## Summary

Keep open. Current main and the latest release still validate the full gateway attachment base64 payload with a regex in the reported `chat.send` path, and the paired open validator PR has not merged or supplied inspectable real WebChat proof.

## What This Changes

The issue reports mobile WebChat image attachments failing `chat.send` with a `RangeError` in gateway base64 validation, plus a secondary `sharp` optional-dependency error in image processing.

## Best Possible Solution

Land a non-decoding linear gateway base64 validator with focused regression coverage and redacted real WebChat/gateway proof, while tracking the `sharp` packaging behavior separately.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The reporter's logs map to the current `chat.send` parser path, but a direct Node regex probe with large strings did not throw in this checkout.

## Solution Assessment

Yes, the narrow maintainable direction is replacing whole-payload regex validation with a non-decoding linear validator that preserves the existing size guards. A Buffer round trip or a standalone UI cap would not be the safest complete fix for the reported gateway path.
