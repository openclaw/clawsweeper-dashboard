## Summary

Keep open. Current main and the latest release still run the full attachment base64 payload through the regex in `isValidBase64`, and the same reporter has an open repair PR that is not ready to land yet.

## What This Changes

Fix a gateway `chat.send` crash where large mobile WebChat image attachments can hit `RangeError: Maximum call stack size exceeded` during base64 validation.

## Best Possible Solution

Review and land the paired stack-safe validator PR after its lint and real-behavior-proof blockers are resolved, then close this issue from that merged fix.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The reported production stack names `isValidBase64`, and current source still has the same whole-payload regex in the `chat.send` attachment path, but a focused Node probe did not trigger `RangeError`.

## Solution Assessment

Yes, the best narrow fix is to replace the regex with a non-decoding linear base64 validator while preserving the existing pre-decode size guards and invalid-base64 behavior. The open paired PR is on that path, but it still needs lint cleanup and real WebChat/gateway proof before merge.
