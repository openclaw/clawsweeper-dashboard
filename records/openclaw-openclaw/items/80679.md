## Summary

Keep open. The production stack from `v2026.5.7` maps to a current-main `chat.send` path that still validates the whole attachment payload with one regex before size checks and offload, so this is a narrow gateway attachment bug rather than cleanup-close material.

## What This Changes

The issue requests that large mobile webchat image attachments stop crashing `chat.send` in base64 validation and stage/deliver normally.

## Best Possible Solution

Replace the regex-based base64 validation in `src/gateway/chat-attachments.ts` with a stack-safe linear validator or canonicalization path, add multi-megabyte attachment regression coverage, and leave `sharp` packaging to the canonical tracker.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The issue has an exact `v2026.5.7` stack naming `isValidBase64`, current source still has the same whole-payload regex path, and a direct Node regex probe did not reproduce the RangeError locally.

## Solution Assessment

Yes for the primary bug: the narrow maintainable fix is to remove unbounded regex validation from the gateway attachment parser without changing the attachment protocol. Reusing or adapting the existing linear `canonicalizeBase64` helper is safer than trusting large payloads by prefix alone, and the `sharp` symptom should remain separate.
