## Summary

Keep open. Current main still validates the full gateway attachment base64 payload with a regex before size checks, and the open paired validator PR has not merged; the later WebChat optimistic-state fix did not remove this gateway parser path.

## What This Changes

The issue reports mobile WebChat image sends failing with a `RangeError` in gateway base64 attachment validation and also notes a related `sharp` optional-dependency error.

## Best Possible Solution

Land or replace the paired gateway validator fix with a non-decoding linear validator plus focused regression coverage, then close this issue and related duplicates; track the `sharp` packaging behavior separately if needed.

## Reproduction Assessment

No, not as a live current-main browser reproduction. The reporter's stack names `RegExp.test` inside `isValidBase64`, and current main still calls that whole-payload regex before size checks in `chat.send` attachment parsing.

## Solution Assessment

Yes, the narrow maintainable fix is to replace the gateway attachment regex with non-decoding linear validation while preserving the existing pre-decode size guard. A Buffer round-trip or client-only mitigation would be weaker for this code path.
