## Summary

Keep open. Current main still runs a whole-payload base64 regex in the gateway attachment parser, the reported WebChat path still sends base64 attachment content into `chat.send`, and a paired open PR already targets this validator path but has not merged.

## What This Changes

The issue reports mobile WebChat image sends failing with a stack overflow in gateway base64 attachment validation.

## Best Possible Solution

Land or replace the paired gateway validator PR with a non-decoding linear validator, focused regression coverage, and redacted real WebChat proof; handle the `sharp` packaging symptom as a separate follow-up.

## Reproduction Assessment

No. I did not establish a high-confidence live current-main reproduction, but current source still matches the reported `chat.send` attachment parser path and whole-payload `RegExp.test` stack frame.

## Solution Assessment

Yes, with one correction: the best fix is a non-decoding linear validator that preserves the existing pre-decode size guard, not a Buffer round-trip that decodes before the size check.
