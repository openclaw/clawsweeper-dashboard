## Summary

Keep open. Current `main` and the latest tagged release still validate the full attachment base64 payload with the same regex before offload, and the active paired PR at https://github.com/openclaw/openclaw/pull/81911 has not landed yet.

## What This Changes

The issue requests fixing a mobile webchat image-send crash where large base64 attachment validation can throw `RangeError: Maximum call stack size exceeded` in the gateway parser.

## Best Possible Solution

Replace the gateway regex validator with a non-decoding linear base64 validator, add focused multi-megabyte attachment regression coverage, and land the active PR only after lint and real WebChat/gateway proof are addressed.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. The reporter’s gateway logs name `isValidBase64` and current source still matches that path, but a focused current-main Node regex probe up to 10 MiB did not throw.

## Solution Assessment

Yes, the narrow maintainable direction is to remove unbounded regex validation while preserving the existing pre-decode size guard. The open paired PR is the right review path once it fixes lint and supplies inspectable real behavior proof.
