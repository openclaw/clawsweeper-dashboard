## Summary

Keep open. Current main still validates the entire attachment base64 payload with one regex before image offload, and the reported shipped `v2026.5.7` stack points at that same parser path; the separate missing-`sharp` symptom is already tracked elsewhere.

## What This Changes

The issue requests a fix for mobile WebChat image sends that fail with `RangeError: Maximum call stack size exceeded` during gateway base64 validation.

## Best Possible Solution

Replace the regex-based validator with stack-safe base64 validation in the gateway attachment parser, add a multi-megabyte attachment regression test, and leave the `sharp` dependency fix to its existing issue and PR.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The report includes an exact `v2026.5.7` stack and current source still has the same whole-payload regex path, but a direct Node probe of large strings did not reproduce the RangeError locally.

## Solution Assessment

Yes for the primary failure: the narrow maintainable fix is to remove unbounded regex validation from `src/gateway/chat-attachments.ts` while preserving invalid-base64 and size-limit checks. A linear character/padding validator plus the existing decoded-size verification is safer than trusting the data URL prefix or doing a broad protocol change.

## Repair Work Prompt

Fix the primary WebChat/mobile image attachment stack overflow reported in https://github.com/openclaw/openclaw/issues/80679. Current `src/gateway/chat-attachments.ts` validates the entire base64 payload with `/^[A-Za-z0-9+/]+={0,2}$/` before image offload; replace that with a stack-safe linear validation path that preserves length, padding, invalid-character, decoded-size, and existing max-size behavior. Add focused regression coverage for a multi-megabyte base64 image attachment through `parseMessageWithAttachments`, including that malformed characters/padding are still rejected and oversized images still fail before decode/offload. Do not change the attachment protocol, do not trust the data URL prefix alone, and do not address the missing-`sharp` package dependency here; that is tracked by https://github.com/openclaw/openclaw/issues/77760 and https://github.com/openclaw/openclaw/pull/77791. Add a changelog fix entry if the repair is user-visible.
