## Summary

Keep open. Current main and the latest release still validate the full attachment base64 payload with one regex before size/offload handling, and the open validator PR is unmerged and still missing inspectable real behavior proof.

## What This Changes

The issue reports `chat.send` failing with `RangeError: Maximum call stack size exceeded` when large mobile WebChat image attachments reach `isValidBase64`.

## Best Possible Solution

Land a non-decoding, stack-safe base64 validator with focused multi-megabyte attachment coverage and redacted real WebChat proof, while tracking the `sharp` packaging behavior separately.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The report includes exact gateway logs and current source still routes large WebChat base64 through the same regex, but a direct large-string Node probe did not throw.

## Solution Assessment

Yes for the primary failure: replacing the regex with a non-decoding linear validator is the narrow maintainable fix because it preserves pre-decode size checks. The open PR is in the right area but still needs inspectable real behavior proof and ideally focused regression coverage.
