## Summary

This PR is a focused and useful fix for the shared `/models` picker and fallback allowlist path, but the latest diff still leaves the new alias-first fallback branch outside the existing manifest/plugin model-id normalization contract. That can create raw or wrongly attributed allowed keys for plugin-normalized aliases, so it should stay open until that narrow blocker is repaired.

## What This Changes

The branch updates shared model fallback allowlist construction to resolve bare fallback aliases through the configured alias index, adds model-selection and `/models` regression coverage, and records a changelog entry.

## Best Possible Solution

Keep the alias-index approach, but make the new fallback-only alias branch use the same normalization context as the existing default/exact ref parsing path and cover that with a manifest-normalized alias regression test.

## Reproduction Assessment

Yes from source inspection and the PR's copied live output: current main can synthesize `openai-codex/opus-cli` for a bare fallback alias because `buildAllowedModelSetWithFallbacks` resolves fallbacks without consulting the alias index. I did not execute the repro locally in this read-only review.

## Solution Assessment

Not yet. Resolving fallback aliases through the alias index is the right narrow seam, but the implementation needs to forward the existing manifest/plugin normalization context before it is the safest fix.
