## Summary

Keep open: this PR addresses a real ClawDock bug still present on current main and the latest release, but the current diff has a merge-blocking token argv exposure in the helper.

## What This Changes

The PR updates the ClawDock shell helper and changelog so `clawdock-dashboard` rewrites loopback dashboard URLs to the Docker-published gateway port, appends browser auth tokens, and redacts tokenized terminal output.

## Best Possible Solution

Land a helper-scoped remap that resolves the published Compose port, preserves scheme/path and token redaction, encodes tokens without argv exposure, and coordinates with the separate no-deps dashboard lookup fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a Docker-published gateway port different from `18789`, run `clawdock-dashboard`, and current main opens the CLI-container loopback URL unchanged.

## Solution Assessment

Partly. Fixing `scripts/clawdock/clawdock-helpers.sh` is the narrowest maintainable direction, but this PR should encode the token without raw child-process argv exposure and preserve the related https://github.com/openclaw/openclaw/pull/80494 `--no-deps` helper fix before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid passing dashboard tokens through argv:** `scripts/clawdock/clawdock-helpers.sh:74-78`
  - body: `_clawdock_urlencode` is called with the gateway token and then starts `python3` or `node` with that token as a positional argument. Local process listings can expose child-process argv values while the helper runs, so feed the token through stdin or encode it without a subprocess argument before appending it to the browser URL.
  - confidence: 0.88
