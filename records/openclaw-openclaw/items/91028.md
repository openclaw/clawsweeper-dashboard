Full review comments:

- **[P1] Mask credentials through Lobster's final env merge:** `extensions/lobster/src/lobster-runner.ts:313-314`
  - body: This deletes the gateway credential keys from the copied env, but `@clawdbot/lobster` later builds the actual runtime env as `{ ...process.env, ...ctx.env }`. If the gateway process has `OPENCLAW_URL`, `OPENCLAW_TOKEN`, `CLAWD_URL`, or `CLAWD_TOKEN`, those keys are restored before workflow shell steps run; pass explicit `undefined` denylist values or otherwise override the dependency merge and test through the real `createToolContext` path.
  - confidence: 0.91
- **[P2] Preserve llm-task auth and model policy:** `extensions/lobster/src/lobster-tool.ts:277-288`
  - body: The new adapter calls `runEmbeddedAgent` directly with only provider/model/timeout, so embedded `llm.invoke --provider openclaw` bypasses the existing `llm-task` handling for plugin defaults, `allowedModels`, `authProfileId`, thinking policy, temperature, and max token limits. The bridge should reuse or share that policy path unless maintainers intentionally define a different contract.
  - confidence: 0.86
- **[P1] Avoid shadowing Lobster's HTTP openclaw provider:** `extensions/lobster/src/lobster-tool.ts:324`
  - body: Registering the direct adapter under `openclaw` changes `llm.invoke --provider openclaw` in embedded runs because Lobster resolves `ctx.llmAdapters[provider]` before its built-in HTTP `openclaw` adapter. Use a distinct provider name or get maintainer approval and docs for the compatibility change.
  - confidence: 0.84
