Full review comments:

- **[P1] Keep stripped credentials out of Lobster's final env:** `extensions/lobster/src/lobster-runner.ts:293-294`
  - body: This deletes the gateway keys from the ctx object passed to Lobster, but `@clawdbot/lobster` builds its runtime env as `{ ...process.env, ...ctx.env }`. Because deleted keys are absent from `ctx.env`, any `OPENCLAW_URL`/token values in the gateway process env are added back before `exec` and `openclaw.invoke` run, so workflow shell steps can still read them. Override the keys through the final dependency path and add a test that exercises the real embedded runtime merge.
  - confidence: 0.95
- **[P1] Preserve llm-task policy before injecting the adapter:** `extensions/lobster/src/lobster-tool.ts:332-333`
  - body: The default Lobster registry already exposes `llm.invoke`, and this PR always injects an `openclaw` direct adapter whenever the Lobster tool is available. That lets Lobster workflows invoke OpenClaw LLM runs without the existing llm-task optional-tool enablement, `allowedModels`, `defaultAuthProfileId`, model-alias resolution, or documented tuning path. Reuse/extract the llm-task policy or add an explicit maintainer-approved Lobster LLM policy before exposing this bridge.
  - confidence: 0.9
- **[P2] Forward llm.invoke generation options:** `extensions/lobster/src/lobster-tool.ts:277-288`
  - body: Lobster's `llm.invoke` payload includes `temperature` and `maxOutputTokens`, but this `runEmbeddedAgent` call only forwards timeout, provider, and model. Embedded users passing `--temperature` or `--max-output-tokens` will silently get defaults, unlike the existing llm-task path which forwards `streamParams`. Map those payload fields or reject them explicitly.
  - confidence: 0.87
