## Summary

Keep this PR open, but it is not merge-ready: the branch addresses a real embedded Lobster gap, yet the current patch still misses the real dependency env merge and creates a parallel LLM policy path.

## What This Changes

The PR changes the Lobster plugin to pass host-provided embedded runtime bridges, strip gateway env credentials, and map Lobster `llm.invoke --provider openclaw` to `api.runtime.agent.runEmbeddedAgent`.

## Best Possible Solution

Keep the bridge idea, but land it only after final-env credential masking, `llm-task` policy parity or shared reuse, explicit provider-name approval, and live embedded invocation proof.

## Reproduction Assessment

Yes by source for the review blockers: `@clawdbot/lobster` re-merges `process.env` into the final runtime env, and the new adapter directly calls `runEmbeddedAgent` without the existing `llm-task` policy fields. I did not run a live model workflow in this read-only review.

## Solution Assessment

No; the current branch is a plausible bridge shape, but not the best fix until it masks credentials after the real dependency merge and reuses or exactly preserves the established `llm-task` policy path.

- **[P1] Mask credentials through the final Lobster env merge:** `extensions/lobster/src/lobster-runner.ts:293-294`

  - confidence: 0.94

Embedded workflows could therefore run with different model/auth policy than the documented `llm-task` path this feature is replacing.
