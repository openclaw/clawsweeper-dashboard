## Summary

Keep open. Current `main` already has provider-hook caching and active-registry reuse, but the minimal-profile path still constructs the OpenClaw tool set and performs media/image capability snapshot work before final profile filtering, so the reported pre-stream overhead is not fully addressed.

## What This Changes

The issue asks OpenClaw to avoid multi-second pre-stream delays from eager media/web capability discovery and repeated provider-hook resolution under `tools.profile: "minimal"`.

## Best Possible Solution

Land a narrow performance fix that makes core tool construction profile-aware before media/image/PDF availability checks and preserves the existing provider-hook and registry caches.

## Reproduction Assessment

Yes for a source-level reproduction path: run a minimal-profile Gateway or embedded turn with `recordToolPrepStage` timing and compare `core-plugin-tools` and `stream-setup` before and after. Exact reported seconds still need macOS LaunchAgent profiling.

## Solution Assessment

Unclear as a complete fix direction because current `main` already has provider-hook caching and active-registry reuse. The safer approach is targeted lazy/profile-aware media construction plus memoization only where current profiling still shows uncached lookups.

## Repair Work Prompt

Fix the remaining minimal-profile pre-stream performance regression from https://github.com/openclaw/openclaw/issues/76203. Current `main` already has provider-hook caching and active-registry reuse, so do not replace those systems. Make `createOpenClawTools`/`createOpenClawCodingTools` avoid media/image/PDF capability metadata work when the resolved tool allowlist cannot expose those tools, especially under `tools.profile: "minimal"`. Ensure image-tool availability also receives or respects the effective allowlist before calling `loadCapabilityMetadataSnapshot`. Add focused regression tests proving a minimal allowlist does not load capability metadata for image/media/PDF factories while explicit allowlists and explicit model configs still work. Keep the fix in the agent tool-construction/profile boundary unless profiling proves another current cache miss. Add a user-facing changelog entry if the repo policy treats this perf fix as user-facing.
