## Summary

Keep open. Current main has already added provider-hook caching and active-registry reuse for part of the reported stream-setup cost, but minimal-profile runs still enter media/image capability availability work before final profile filtering, so the central performance request is not fully implemented.

## What This Changes

The issue requests lazy or cached media/web provider discovery and provider-hook resolution so minimal-profile turns do not spend several seconds preparing unused tools before streaming.

## Best Possible Solution

Land a focused performance fix that makes core tool construction profile-aware before media/image capability checks, preserves explicit opt-ins, and adds regression coverage proving minimal-profile tool prep does not resolve unused media capability providers.

## Reproduction Assessment

Yes for a source-level reproduction path: run a minimal-profile Gateway or embedded turn with the existing `recordToolPrepStage` timing callback and compare `core-plugin-tools` and `stream-setup` before and after. The exact reported seconds still need macOS LaunchAgent profiling to confirm magnitude.

## Solution Assessment

Unclear as a complete fix direction because main already has provider-hook caching and active-registry reuse. The safer path is a targeted lazy/profile-aware construction change plus measurement of any remaining uncached provider-hook calls.
