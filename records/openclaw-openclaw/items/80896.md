## Summary

Keep open: the bootstrap key-proof hardening is useful, but the latest head still changes setup-code bootstrap from the current silent handoff into a manual approval flow, and the PR has a protected label plus no after-fix real behavior proof.

## What This Changes

The PR adds bootstrap public-key proof validation and forwards the verified handshake payload, while modifying gateway bootstrap pairing tests and approval handling.

## Best Possible Solution

Require current-handshake device-key proof at the bootstrap verifier and gateway seam while preserving setup-code auto-approval, bounded node/operator handoff, token revocation, and failure recovery.

Yes. Source inspection is high-confidence: current main expects setup-code bootstrap to return `hello-ok` with no pending request, while PR head forces bootstrap-token pairings to `silent: false` and expects `PAIRING_REQUIRED`.

## Solution Assessment

No. Passing public-key proof into the verifier is the right boundary, but replacing the established setup-code handoff with manual approval is not the narrowest maintainable fix.
