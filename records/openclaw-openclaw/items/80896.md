## Summary

Keep open: the security hardening target is plausible, but the patch is not mergeable because it removes the documented and tested QR/setup-code bootstrap handoff. The external PR also lacks after-fix real behavior proof, and the protected `maintainer` label means this needs explicit maintainer handling.

## What This Changes

The PR adds public-key proof to bootstrap token verification and threads the verified device signature payload through gateway connect auth, while changing QR/setup-code bootstrap pairing behavior and tests.

## Best Possible Solution

Add the public-key proof check at the bootstrap verifier and gateway auth seam while preserving the existing setup-code bootstrap profile handoff, fresh-node silent approval, bounded node/operator token issuance, and bootstrap-token revocation behavior.

## Reproduction Assessment

Yes. Source inspection shows current main has a documented/tested setup-code bootstrap handoff path, while PR head forces bootstrap-token pairings to require manual approval; the existing gateway suite provides a focused reproduction path.

## Solution Assessment

No. Requiring proof of the submitted device key is the right boundary, but deleting `allowSilentBootstrapPairing`, `approveBootstrapDevicePairing`, and the `hello-ok` token handoff is not the narrowest maintainable fix.
