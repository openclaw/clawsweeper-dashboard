## Summary

Keep open: this protected gateway/auth PR still lacks external real behavior proof and latest head changes setup-code bootstrap onboarding from the current auto-approved `hello-ok` handoff into an interactive approval flow.

## What This Changes

The branch adds public-key proof validation to device bootstrap-token verification, forwards the verified handshake payload into that verifier, and rewrites gateway/device-pairing tests around bootstrap approval and handoff.

## Best Possible Solution

Require current-handshake device-key proof at the bootstrap verifier and gateway seam while preserving current setup-code auto-approval, bounded node/operator handoff, token revocation, and failure recovery.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main expects setup-code bootstrap to auto-approve and return `hello-ok`, while PR head forces the same bootstrap-token path through `PAIRING_REQUIRED`.

## Solution Assessment

No. Passing public-key proof into the verifier is the right boundary, but replacing the established setup-code handoff with interactive approval is not the narrowest maintainable fix.
