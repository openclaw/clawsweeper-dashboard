## Summary

Keep open. The security hardening is useful, but the latest head still regresses setup-code bootstrap by forcing bootstrap-token pairing through manual approval, and the external PR has no real behavior proof.

## What This Changes

The branch adds public-key proof forwarding and validation for bootstrap token binding, then rewrites gateway/device bootstrap pairing tests and pending approval handling around that path.

## Best Possible Solution

Require current-handshake public-key proof before binding a bootstrap token while preserving setup-code auto-approval, bounded node/operator token handoff, token revocation, and failure recovery.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main expects setup-code bootstrap to return `hello-ok` with no pending request, while PR head sets bootstrap-token pairings to non-silent and expects `PAIRING_REQUIRED`.

## Solution Assessment

No. Requiring proof at the verifier and gateway auth seam is the right boundary, but replacing the established setup-code handoff with manual approval is not the narrowest maintainable fix.
