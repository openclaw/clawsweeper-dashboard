## Summary

Keep open: the verifier hardening is useful, but the latest head still changes setup-code bootstrap from silent handoff to interactive approval and lacks external real behavior proof on a protected gateway/auth PR.

## What This Changes

The PR requires bootstrap-token verification to receive device public-key proof, forwards the verified handshake proof through gateway auth, and rewrites bootstrap pairing tests around approval and token binding.

## Best Possible Solution

Require current-handshake device-key proof at the bootstrap verifier and gateway seam while preserving setup-code auto-approval, bounded node/operator handoff, token revocation, and failure recovery.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main expects fresh setup-code bootstrap to auto-approve and return hello-ok, while PR head forces bootstrap-token pairing through PAIRING_REQUIRED before manual approval.

## Solution Assessment

No. Passing public-key proof into the verifier is the right boundary, but replacing the established setup-code handoff with interactive approval is not the narrowest maintainable fix.
