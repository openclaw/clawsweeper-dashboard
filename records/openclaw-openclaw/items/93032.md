## Label Justifications

- `P1`: The report describes all cron jobs silently failing to fire after a real upgrade, with the gateway otherwise appearing healthy.
- `impact:message-loss`: Scheduled outbound work, transcript loggers, and heartbeats can be suppressed when the scheduler loads zero jobs.
- `impact:session-state`: The in-memory scheduler state can diverge from persisted SQLite cron rows until a second hard restart.

## Summary

Keep open. Current main still has a plausible stale in-memory cron store path after an empty first load, and the post-release SQLite WAL maintenance changes do not by themselves reload cron state or prove the single-restart upgrade path is fixed.

## What This Changes

After upgrading to v2026.6.6, the reporter asks OpenClaw to ensure a single gateway restart loads migrated SQLite cron jobs into the scheduler instead of requiring a second hard restart.

## Best Possible Solution

Make cron startup/import ordering deterministic so the first live scheduler load sees migrated SQLite rows, and add regression coverage for the v2026.5.28-to-v2026.6.6 LaunchAgent upgrade path.

## Reproduction Assessment

No live macOS LaunchAgent upgrade repro was run. Source inspection shows a plausible stale-cache path: `ensureLoaded` can keep an empty in-memory store while later `cron list/status/start` reads do not force a reload after SQLite rows become visible.

## Solution Assessment

Unclear until the startup ordering is proven. The safest direction is to complete legacy cron import before cron can load, or explicitly force-refresh the cron service after migration commits, instead of treating WAL checkpointing alone as the fix.
