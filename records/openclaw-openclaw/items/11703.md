## Summary

Keep open. Current main still uses receive-time debounce for WhatsApp inbound batching; `messageTimestamp` is preserved on the inbound message but is not part of the debounce key, timer, flush policy, or config surface.

## What This Changes

The issue requests coalescing WhatsApp inbound messages that share the same Baileys server `messageTimestamp` into one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only messages from the same account, conversation, and sender, with conservative tolerance/defaults and docs for any new config surface.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after the configured receive-time `debounceMs` will enter separate receive-time buffers because the current key omits the timestamp and the timer starts at enqueue time.

## Solution Assessment

Yes, directionally, but the exact policy needs maintainer judgment. The maintainable path is WhatsApp plugin-level timestamp coalescing, not changing shared inbound debounce semantics for every channel.
