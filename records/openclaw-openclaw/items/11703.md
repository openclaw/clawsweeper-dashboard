## Summary

Keep open. Current main still uses receive-time inbound debouncing for WhatsApp; Baileys `messageTimestamp` is preserved on the inbound message but is not used as a coalescing key, timer, or documented config mode.

## What This Changes

Request batching simultaneous WhatsApp text messages by Baileys `messageTimestamp` instead of only by receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when server timestamps match or fall within a small documented tolerance, while preserving existing immediate-flush behavior for media, location, reply-context, self-message, access-control, and control-command paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account, chat, and sender with the same `messageTimestamp`, where the second arrives after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path, with config/docs only as needed, rather than changing shared inbound debounce semantics for every channel.
