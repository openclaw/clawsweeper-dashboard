## Summary

Keep open. Current main and v2026.6.10 still batch WhatsApp inbound messages with the receive-time debouncer; `messageTimestamp` is preserved as event metadata but is not used to coalesce late Baileys deliveries, and the remaining default/config semantics need maintainer direction.

## What This Changes

The issue requests batching WhatsApp messages by WhatsApp server `messageTimestamp` instead of only by receive-time debounce.

## Best Possible Solution

Approve a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting any config/default contract maintainers choose.

## Reproduction Assessment

Yes, at source level. A same-key WhatsApp message that arrives after the receive-time debounce timer has flushed cannot join the earlier batch because `messageTimestamp` is stored on the event but not used to reopen, bucket, or coalesce flushed buffers.

## Solution Assessment

Unclear until maintainers choose the timestamp/default/config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
