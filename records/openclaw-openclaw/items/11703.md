## Summary

Keep open. Current `main` still uses receive-time inbound debounce for WhatsApp; `messageTimestamp` is preserved on the message but is not used as a coalescing key, flush policy, or config surface, and the existing maintainer/ClawSweeper comments remain accurate.

## What This Changes

Request to batch related WhatsApp inbound messages by WhatsApp `messageTimestamp` instead of only by receive-time debounce.

## Best Possible Solution

Add a WhatsApp plugin-level timestamp coalescing policy for text-only same-account/conversation/sender messages, with conservative defaults/config and docs, while preserving existing immediate-flush cases.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after the configured receive-time `debounceMs` will enter separate receive-time buffers because the current key omits the timestamp and the timer starts at enqueue time.

## Solution Assessment

Yes as a direction, but only with a narrow WhatsApp-owned policy. Changing the shared inbound debouncer for every channel would be broader than needed; the unresolved maintainer choice is the exact default, tolerance, and config shape.
