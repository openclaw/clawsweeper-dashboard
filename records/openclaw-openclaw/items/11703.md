## Summary

Keep open. Current main and the latest shipped tag still batch WhatsApp inbound messages with a receive-time debounce keyed by account, conversation, and sender; Baileys `messageTimestamp` is preserved as metadata but not used for coalescing, and the open composing-aware debounce PR is adjacent rather than a superseding timestamp fix.

## What This Changes

Request timestamp-based WhatsApp inbound coalescing so same-time forwarded/comment or rapid-fire messages arrive as one agent turn instead of separate turns.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only same account, conversation, and sender messages with conservative tolerance, schema/types/docs, and focused fake-timer coverage while preserving media, command, access-control, reply-context, and self-message behavior.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing rather than a broad shared debouncer behavior change, coordinated with the open composing-aware debounce PR.
