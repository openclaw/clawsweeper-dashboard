## Summary

Keep open. Current main still stores WhatsApp `messageTimestamp` as message metadata while batching only through the receive-time debounce timer, so the requested timestamp-based coalescing is not implemented and needs maintainer decisions before a repair lane should change behavior.

## What This Changes

Request timestamp-based WhatsApp batching so same or near-same WhatsApp server timestamps can be delivered as one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account/conversation/sender messages, with conservative tolerance/defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp messages with the same `messageTimestamp` but delivered after `debounceMs` has elapsed will flush separately because the current buffer is receive-timer based and deleted on flush.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing, not a broad shared debouncer semantic change unless maintainers intentionally want cross-channel behavior.
