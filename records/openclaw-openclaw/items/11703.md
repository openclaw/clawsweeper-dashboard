## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata only; inbound batching remains keyed by account/conversation/sender and flushed by receive-time debounce, so the requested timestamp coalescing behavior is still missing.

## What This Changes

The issue requests batching WhatsApp inbound messages by WhatsApp `messageTimestamp` rather than receive time so forwarded or simultaneous messages can arrive as one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same account/conversation/sender messages, with conservative scope and documented config/default behavior.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered more than `debounceMs` apart will flush separately because current buffering is keyed without the timestamp and the timer starts on enqueue.

## Solution Assessment

Yes, with maintainer policy first. The maintainable direction is a WhatsApp plugin-level timestamp coalescing policy rather than changing shared inbound debounce semantics for every channel.
