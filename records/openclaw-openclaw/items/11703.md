## Summary

Keep open. Current main and v2026.6.10 still use account/conversation/sender receive-time debouncing for WhatsApp; Baileys `messageTimestamp` is preserved as event metadata but is not used as a coalescing key or late-arrival grouping rule, and the remaining default/config semantics need maintainer direction.

## What This Changes

The issue requests WhatsApp inbound batching based on Baileys `messageTimestamp` so simultaneous forward-and-comment messages can be delivered to the agent as one turn instead of separate receive-time debounced turns.

## Best Possible Solution

Approve a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting any config/default contract maintainers choose.

## Reproduction Assessment

Yes, at source level. A same-key WhatsApp message that arrives after the receive-time debounce timer has flushed necessarily starts a new batch because `messageTimestamp` is stored on the event but not used to reopen, bucket, or coalesce flushed buffers.

## Solution Assessment

Unclear until maintainers choose the timestamp/default/config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
