## Label Justifications

- `impact:session-state`: A forwarded item and its accompanying comment can reach separate agent turns, fragmenting the conversational context used for the response.

## Summary

Current main still uses a receive-time quiet window, so delayed WhatsApp parts with the same server timestamp can become separate agent turns. The request remains useful and is not owned by the per-group debounce issue, but timestamp collision, maximum-wait, and compatibility semantics require maintainer product judgment before implementation.

## What This Changes

Add WhatsApp-specific batching that uses platform message timestamps to keep delayed parts of one send action in a single agent turn.

## Best Possible Solution

Implement a bounded, backward-compatible WhatsApp-owned coalescing policy that treats server timestamp as one grouping signal, preserves the current default, and explicitly defines sender/conversation boundaries, ordering, maximum wait, timestamp collisions, commands, media, location, and reply-context behavior.

## Reproduction Assessment

Yes at source level: if Baileys delivers the second message after the configured receive-time quiet window, current main has already flushed the first buffer and creates a separate turn. This review did not perform a live WhatsApp account reproduction of the reported decryption delay.

## Solution Assessment

No, not exactly as proposed. Server timestamps are relevant evidence, but equal-timestamp bucketing with a 50–100 ms receive window neither spans the reported delay nor safely handles one-second timestamp collisions; maintainers should approve a bounded channel-owned contract first.

## Risks / Open Questions

- WhatsApp server timestamps have one-second precision, so grouping solely by timestamp equality can combine independent rapid messages unless account, conversation, sender, message type, and a bounded collision policy are enforced.
- The proposed 50–100 ms flush window does not address the reported 5–10 second Baileys delivery gap; a longer maximum wait changes latency and needs an explicit compatibility decision.
- Adding generic `messages.inbound` options would expand a global config surface for channel-specific behavior; repository policy favors keeping this policy inside the WhatsApp plugin unless other channels demonstrate the same contract.
