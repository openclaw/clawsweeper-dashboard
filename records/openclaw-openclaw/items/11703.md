## Summary

Keep open. Current main and the latest release still use receive-time WhatsApp inbound debouncing; `messageTimestamp` is preserved as metadata but is not part of the debounce key, timer source, config schema, or documented batching behavior.

## What This Changes

The issue requests batching WhatsApp inbound messages by Baileys `messageTimestamp` so simultaneous forward-plus-comment messages arrive in one agent turn.

## Best Possible Solution

Choose a maintainer-approved WhatsApp-owned compatibility contract, then add conservative timestamp coalescing for text-only same-account, same-conversation, same-sender messages with schema/types/docs and focused fake-timer coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the second message is enqueued.

## Solution Assessment

Unclear for the exact proposed config shape. A WhatsApp-owned timestamp coalescing path with conservative defaults is more maintainable than broadly changing the shared debouncer for every channel.
