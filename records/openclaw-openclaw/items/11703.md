## Summary

Keep open. Current main and v2026.5.19 still batch WhatsApp inbound messages through receive-time debounce only; `messageTimestamp` is preserved as metadata but is not part of the debounce key, timer, config schema, or docs, and the adjacent composing-aware debounce PR does not implement timestamp coalescing.

## What This Changes

Request batching WhatsApp messages that share or nearly share Baileys `messageTimestamp` rather than only receive-time `debounceMs`, so forward-plus-comment or rapid sends land in one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same-account, same-conversation, same-sender messages with conservative tolerance, schema/types/docs, fake-timer coverage, and coordination with adjacent WhatsApp debounce work.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path with conservative defaults/tolerance, not a broad shared debouncer semantic change.
