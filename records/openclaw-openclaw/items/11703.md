## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata but batches inbound WhatsApp messages only through receive-time debounce, so the reported source-level gap remains.

## What This Changes

Request timestamp-aware WhatsApp inbound batching so same-server-timestamp messages can arrive as one agent turn instead of separate receive-time debounced turns.

## Best Possible Solution

Add a WhatsApp-owned timestamp coalescing policy for text-only same-account, same-conversation, same-sender messages, with explicit safety rules and documentation for any config/default behavior.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after the receive-time `debounceMs` window will flush separately because the key omits the timestamp and the timer is enqueue-time based.

## Solution Assessment

Yes with maintainer policy input. The narrow maintainable solution is WhatsApp plugin-level timestamp coalescing, not changing shared inbound debounce semantics for every channel.
