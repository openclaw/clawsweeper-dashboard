## Label Justifications

- `P2`: This is a valid but bounded WhatsApp batching/context-quality improvement with a latency-heavy workaround, not an outage.
- `impact:session-state`: Messages intended as one user action can be split into separate agent turns, fragmenting the context presented to the agent.

## Summary

Keep open. Current main and the latest release still batch WhatsApp inbound messages by local enqueue/debounce timing, while WhatsApp server timestamps are only preserved and used to order messages that already reached the same flush.

## What This Changes

The issue requests WhatsApp inbound batching based on WhatsApp server message timestamps rather than local receive/debounce time.

## Best Possible Solution

Implement a bounded WhatsApp-owned text coalescer keyed by account, conversation, sender, and server-time bucket, with maximum-wait and immediate-flush rules; add config only after maintainers choose the contract.

## Reproduction Assessment

Yes at source level: after the receive-time debounce flushes, a later Baileys message necessarily enters a new buffer and agent turn even when its WhatsApp timestamp matches. I did not run a live WhatsApp reproduction.

## Solution Assessment

No, the exact timestamp-equality/config proposal is not clearly the best final shape because second-resolution timestamps need tighter boundaries, maximum-wait semantics, and preserved immediate-flush behavior. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer.
