## Summary

Keep open. Current main and the latest release still use receive-time debounce for WhatsApp inbound batching; WhatsApp server timestamps are preserved and later used for ordering, but they do not keep late-arriving same-timestamp messages in one agent turn.

## What This Changes

The issue requests WhatsApp inbound batching by Baileys `messageTimestamp` so simultaneous forward/comment or rapid messages arrive as one agent turn instead of separate turns.

## Best Possible Solution

Approve and implement a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting the chosen config/default contract.

## Reproduction Assessment

Yes at source level. A late Baileys message with the same WhatsApp timestamp necessarily starts a new receive-time debounce buffer after the previous timer has flushed; I did not run a live WhatsApp reproduction.

## Solution Assessment

Unclear until maintainers decide the timestamp and config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
