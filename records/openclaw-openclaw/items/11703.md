## Summary

Keep open. Current main and v2026.5.20 still batch WhatsApp inbound messages through the shared receive-time debounce path; `messageTimestamp` is preserved as metadata, but it is not used as a grouping key, timer source, schema field, or documented behavior.

## What This Changes

The issue requests batching WhatsApp messages by Baileys `messageTimestamp` so simultaneously sent forward/comment or rapid messages arrive in one agent turn instead of separate receive-time debounce turns.

## Best Possible Solution

Implement a WhatsApp-owned, backwards-compatible timestamp coalescing mode for same-account, same-conversation, same-sender text messages, with conservative tolerance, schema/types/docs, and focused fake-timer coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path with conservative defaults and tolerance, not a broad shared debouncer semantic change.
