## Summary

Keep open. Current main still implements WhatsApp inbound batching as a receive-time debounce and preserves `messageTimestamp` only as message metadata, while the related composing-aware PR is adjacent and does not supersede timestamp-based coalescing.

## What This Changes

The issue requests WhatsApp inbound batching by WhatsApp server `messageTimestamp` so simultaneous forward-plus-comment messages become one agent turn.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account, conversation, and sender messages with conservative defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush as separate turns because the buffer is deleted by the receive-time timer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing, not a broad shared debouncer semantic change, with maintainer choice on tolerance and defaults.
