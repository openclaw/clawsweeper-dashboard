## Summary

Keep open. Current main still batches WhatsApp inbound messages by receive-time debounce only; `messageTimestamp` is preserved as metadata/context but is not used as a coalescing key, timer, or config-controlled grouping rule.

## What This Changes

The issue requests batching WhatsApp messages by Baileys/WhatsApp `messageTimestamp` so delayed same-action deliveries can arrive as one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps match or fall within a very small documented tolerance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account, chat, and sender with the same `messageTimestamp`, where the second arrives after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path rather than changing shared inbound debounce semantics for every channel.
