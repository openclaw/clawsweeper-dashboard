## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata, but inbound batching is keyed by account, conversation, and sender and is flushed by receive-time timers; the related composing-aware debounce PR is adjacent but does not implement timestamp-based coalescing.

## What This Changes

Requested behavior is to batch WhatsApp inbound messages by WhatsApp server `messageTimestamp` instead of only receive-time debounce.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account, conversation, and sender messages with conservative defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` has elapsed will flush separately because the first receive-time buffer is deleted before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing, not a broad shared debouncer semantic change unless maintainers intentionally want this across channels.
