## Summary

Keep open. Current main still batches WhatsApp inbound text through the shared receive-time debounce path; WhatsApp server timestamps are preserved on the inbound message but are not used as a coalescing key, timer, or config-controlled grouping rule.

## What This Changes

Request to batch WhatsApp forward/comment or rapid-fire message bursts by WhatsApp `messageTimestamp` rather than only by gateway receive time.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender, while preserving current media, location, reply-context, access-control, self-message, and control-command flushing behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account, chat, and sender with the same `messageTimestamp`, where the second arrives after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path and use a documented timestamp-coalescing rule, rather than changing shared inbound debounce semantics for every channel.
