## Summary

Keep open. Current main still uses WhatsApp `messageTimestamp` only as inbound metadata; batching is still keyed by account, conversation, and sender and flushed by receive-time debounce, so the requested timestamp coalescing behavior remains unimplemented.

## What This Changes

Request to batch WhatsApp messages with the same or very close server `messageTimestamp` instead of relying only on receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same account/conversation/sender messages, with conservative documented defaults and regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after the receive-time debounce window will flush separately because the active buffer has already been released and the key does not include a timestamp bucket.

## Solution Assessment

Yes on direction, but maintainer policy is needed first. The maintainable approach is a WhatsApp plugin-level timestamp coalescing policy rather than changing shared inbound debounce semantics for every channel.
