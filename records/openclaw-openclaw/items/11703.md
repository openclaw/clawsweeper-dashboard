## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as inbound metadata, but batching remains keyed to account, conversation, sender, and receive-time debounce windows; there is no timestamp coalescing behavior or config/docs surface yet.

## What This Changes

The issue requests WhatsApp inbound batching based on WhatsApp server `messageTimestamp` so same-send forward/comment bursts arrive as one agent turn.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only same account/conversation/sender messages, keep media/control/reply flushing behavior intact, and align config/docs if a new knob is introduced.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` text messages with the same `messageTimestamp`, delivered to OpenClaw more than `debounceMs` apart, will pass through separate receive-time debounce windows and flush as separate turns.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path and avoid changing shared debounce semantics for every channel unless an additive hook is needed.
