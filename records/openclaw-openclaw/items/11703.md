## Summary

Keep open. Current main and v2026.6.8 still use a receive-time WhatsApp debounce window; `messageTimestamp` is preserved as event metadata/order input, but it is not a coalescing key or late-arrival grouping rule.

## What This Changes

The issue requests WhatsApp inbound batching by Baileys `messageTimestamp` so messages sent together but delivered seconds apart reach the agent as one turn.

## Best Possible Solution

Approve a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting any chosen config/default contract.

## Reproduction Assessment

Yes, at source level. A same-key WhatsApp message that arrives after the receive-time debounce timer has flushed necessarily starts a new batch, because `messageTimestamp` is stored on the event but not used to reopen or coalesce flushed buffers.

## Solution Assessment

Unclear until maintainers choose the timestamp/default/config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
