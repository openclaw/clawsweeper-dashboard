## Label Justifications

- `P2`: The issue causes fragmented WhatsApp context and wasted turns but is channel-specific, has a higher-debounce workaround, and does not make the core runtime unusable.
- `impact:session-state`: Messages intended as one input can be split across agent turns, changing the conversational context available to each run.

## Summary

Current main still batches WhatsApp messages with a receive-time timer; the server timestamp only orders messages that already entered the same buffer, so messages delivered after the timer expires remain separate agent turns. The request is valid and aligned with improving major channels, but safe coalescing semantics need a maintainer product decision before implementation.

## What This Changes

Add WhatsApp inbound batching that can associate messages by their WhatsApp server timestamp rather than relying only on arrival-time debounce.

## Best Possible Solution

Define a WhatsApp-owned, backward-compatible coalescing policy for text-only messages keyed by account, conversation, sender, and a narrowly bounded server-timestamp group, with explicit command/media flush rules and focused fake-timer plus live WhatsApp proof.

## Reproduction Assessment

Yes at source level: when a second WhatsApp message arrives after the configured receive-time timer has flushed, current main necessarily starts another turn even when the server timestamps match. The issue includes observed delivery gaps, although this read-only review did not run a live WhatsApp reproduction.

## Solution Assessment

Unclear. Server-timestamp-aware grouping addresses the reported cause, but the proposed timestamp-only key and 50–100 ms flush window do not cover the reported 5–10 second delivery gap safely without a maintainer-approved grouping and latency policy.

## Risks / Open Questions

- WhatsApp timestamps have second-level precision, so timestamp-only grouping could merge unrelated messages sent during the same second unless sender, conversation, arrival bounds, and flush boundaries are defined carefully.
- A long coalescing grace period could add latency to normal messages; a short period may not cover the reported multi-second Baileys decryption delay.
- Media, commands, reply context, access-control results, connection shutdown, and durable-delivery accounting must preserve their current immediate or ordered behavior.
