## Summary

Keep open. Current main still uses receive-time debouncing for WhatsApp inbound messages; `messageTimestamp` is preserved as message metadata/context but is not used to coalesce late Baileys deliveries into one agent turn. The existing maintainer and bot comments remain accurate against the current checkout.

## What This Changes

Request WhatsApp inbound batching to group messages by WhatsApp server `messageTimestamp` rather than only by local receive-time debounce windows.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps match or fall within a very small documented tolerance. Preserve existing media, location, reply-context, access-control, self-message, dedupe, control-command, and shared debouncer behavior for other channels.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account/chat/sender with the same `messageTimestamp`, where the second is delivered after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes, with one boundary condition: the maintainable fix should be WhatsApp-owned timestamp coalescing rather than changing shared inbound debounce semantics for every channel. The exact config/default should be conservative and documented because WhatsApp timestamps have second-level precision.
