## Summary

Keep open. Current main still has no WhatsApp timestamp coalescing path: `messageTimestamp` is normalized into inbound message metadata, while batching remains driven by the shared receive-time debounce timer keyed only by account, conversation, and sender.

## What This Changes

Request timestamp-aware batching for WhatsApp inbound messages so same-chat messages with identical or very close WhatsApp server `messageTimestamp` values can be delivered as one agent turn instead of separate receive-time debounce turns.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender, preserving existing media, access-control, self-message, reply-context, and control-command flushing behavior and documenting any config surface.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` text messages with the same `messageTimestamp`, delivered to OpenClaw more than `debounceMs` apart, still pass through separate receive-time debounce windows and flush as separate turns.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path and avoid changing shared debounce semantics for every channel unless an additive helper hook is needed.
