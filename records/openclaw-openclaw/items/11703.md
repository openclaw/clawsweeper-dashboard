## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata while batching through receive-time debounce, so timestamp-based coalescing remains unresolved and needs maintainer decisions around semantics and config.

## What This Changes

Request timestamp-based batching for WhatsApp inbound messages so same or nearby `messageTimestamp` messages can arrive as one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account/conversation/sender messages, with conservative documented tolerance/defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after `debounceMs` has elapsed will flush separately because the current buffer is receive-timer based and is deleted on flush.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing policy, not a broad shared debouncer semantic change unless maintainers intentionally want cross-channel behavior.
