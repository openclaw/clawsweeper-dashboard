## Summary

Keep open. Current main and v2026.6.10 still use receive-time WhatsApp inbound debouncing keyed by account, conversation, and sender; `messageTimestamp` is preserved as event metadata but is not used to coalesce late Baileys deliveries, and the remaining timestamp/default/config semantics need maintainer direction.

## What This Changes

Request WhatsApp to group delayed text messages by Baileys server `messageTimestamp` rather than only the receive-time debounce window.

## Best Possible Solution

Approve and implement a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving current immediate-flush exclusions and documenting the chosen config/default contract.

## Reproduction Assessment

Yes, at source level. A same-key WhatsApp text message that arrives after the receive-time debounce timer has flushed necessarily starts a new batch because `messageTimestamp` is stored on the event but not used to reopen, bucket, or coalesce flushed buffers.

## Solution Assessment

Unclear until maintainers choose the timestamp/default/config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
