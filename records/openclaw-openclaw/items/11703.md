## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata only while inbound batching is keyed and flushed by receive-time debounce, so the requested timestamp coalescing behavior remains unimplemented.

## What This Changes

Request timestamp-based WhatsApp inbound coalescing so messages with the same or close Baileys `messageTimestamp` batch together instead of relying only on receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same account/conversation/sender messages, with conservative documented defaults and regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after the receive-time debounce window will flush separately because the first buffer is deleted by the timer and the key has no timestamp bucket.

## Solution Assessment

Unclear for the exact proposed shared config shape. The maintainable direction is WhatsApp plugin-level timestamp coalescing, not a broad change to shared inbound debounce semantics unless maintainers intentionally want cross-channel behavior.
