## Label Justifications

- `P2`: This is a valid but bounded WhatsApp batching/context-quality improvement with a latency-heavy workaround, not an outage.
- `impact:session-state`: Messages intended as one user action can be split into separate agent turns, fragmenting the context presented to the agent.

## Summary

Keep open. Current main and the latest release still batch WhatsApp inbound messages by local enqueue/debounce timing; WhatsApp server timestamps are preserved and used for ordering only after messages have already reached the same flush.

## What This Changes

The issue requests WhatsApp inbound batching by WhatsApp server message timestamps instead of only local receive/debounce timing.

## Best Possible Solution

Approve and implement a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate flush behavior and adding focused docs/tests for the chosen config contract.

## Reproduction Assessment

Yes at source level: after the receive-time debounce flushes, a later Baileys message necessarily enters a new buffer and agent turn even when its WhatsApp timestamp matches. I did not run a live WhatsApp reproduction.

## Solution Assessment

Unclear until maintainers decide the timestamp and config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
