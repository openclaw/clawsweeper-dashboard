## Summary

Keep open: current main still preserves WhatsApp `messageTimestamp` as message metadata but batches inbound WhatsApp text through the normal receive-time debounce path. The related composing-aware debounce PR is adjacent waiting UX, not a timestamp-based coalescing implementation, so this still needs a maintainer decision on tolerance, defaults, and config shape.

## What This Changes

Request timestamp-aware WhatsApp inbound batching so same-server-timestamp messages arrive in one agent turn instead of depending only on receive-time debounce.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account/conversation/sender messages, with conservative tolerance/defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` elapses will flush separately because current main deletes the receive-time buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing, not a broad shared debouncer semantic change unless maintainers intentionally want this behavior across channels.
