## Summary

Keep open. Current main still batches WhatsApp inbound messages by receive-time debounce keyed on account, conversation, and sender, while `messageTimestamp` is preserved as metadata and not used as a coalescing key or timer; the open composing-aware debounce PR is adjacent but does not supersede timestamp-based batching.

## What This Changes

Request batching WhatsApp messages by WhatsApp server `messageTimestamp` so forwarded/comment or simultaneous messages can reach the agent as one turn even when Baileys delivers them after the receive-time debounce window.

## Best Possible Solution

Add WhatsApp plugin-level timestamp coalescing for text-only same account, conversation, and sender messages with conservative tolerance, schema/types/docs, and focused fake-timer regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing rather than changing the shared debouncer contract broadly, with maintainer choice on tolerance and defaults.
