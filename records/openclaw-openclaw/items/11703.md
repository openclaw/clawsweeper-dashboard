## Summary

Keep open. Current main still lacks timestamp-based WhatsApp coalescing: `messageTimestamp` is preserved as inbound metadata, but batching still uses the shared receive-time debounce key/timer and the exposed config/docs surface only covers `debounceMs`.

## What This Changes

The issue requests batching WhatsApp inbound messages by WhatsApp server `messageTimestamp` rather than only by receive-time debounce so forwarded/comment pairs remain in one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only messages from the same account, conversation, and sender, with config/docs aligned if a new knob is introduced.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` text messages with the same `messageTimestamp`, where the second reaches OpenClaw after `debounceMs`, will use separate receive-time debounce windows and flush as separate turns.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path and preserve shared inbound debounce semantics for other channels.

Implement this issue as a narrow WhatsApp plugin change. Add timestamp-based coalescing for text-only inbound WhatsApp messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance, even if Baileys delivers them after the normal receive-time `debounceMs` window. Keep core debounce semantics generic; do not add WhatsApp-specific ids or policy to shared core. Preserve media/attachment immediate flushing, reply context, access control, self-message handling, control-command behavior unless explicitly and narrowly covered, read receipts, append-upsert handling, and inbound dedupe.
