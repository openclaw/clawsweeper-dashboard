## Label Justifications

- `P2`: This is a concrete WhatsApp batching improvement with limited blast radius, but it needs behavior choices rather than urgent runtime handling.
- `impact:session-state`: Splitting same-action WhatsApp messages into separate agent turns can lose conversational context and alter the session transcript.

## Summary

Keep open. Current main and the latest release still use receive-time debounce for WhatsApp inbound batching; `messageTimestamp` is preserved as metadata but not used in the debounce key, timer, schema, or docs, and the adjacent composing-aware debounce PR does not implement timestamp coalescing.

## What This Changes

Request to batch same-action WhatsApp inbound messages using Baileys `messageTimestamp` instead of only receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same-account, same-conversation, same-sender messages with conservative tolerance, schema/types/docs, fake-timer coverage, and coordination with adjacent debounce work.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path with conservative defaults/tolerance, not a broad shared debouncer semantic change.
