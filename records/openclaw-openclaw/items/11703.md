## Summary

Keep open. Current main and the latest shipped tag still batch WhatsApp inbound messages with a receive-time debounce keyed by account, conversation, and sender; Baileys `messageTimestamp` is preserved as metadata but is not used for timestamp coalescing, and the related composing-aware debounce PR is adjacent rather than a superseding fix.

## What This Changes

Request timestamp-based WhatsApp inbound batching so delayed Baileys messages with identical or nearby server timestamps arrive as one agent turn.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only same account, conversation, and sender messages with conservative tolerance, schema/types/docs, and focused fake-timer coverage, while keeping shared debouncer semantics unchanged unless a small additive hook is needed.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is WhatsApp-owned timestamp coalescing rather than a broad shared debouncer behavior change, with maintainer choice on tolerance and defaults.
