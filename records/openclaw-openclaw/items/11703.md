- `impact:session-state`: Splitting same-action WhatsApp messages into separate agent turns can lose conversational context and alter the session transcript.

## Summary

Keep open. Current main and v2026.5.19 still preserve WhatsApp `messageTimestamp` only as metadata while inbound batching remains keyed and timed by receive-time debounce; no canonical PR currently implements timestamp coalescing.

## What This Changes

The issue asks for WhatsApp messages with identical or near-identical server `messageTimestamp` values to be coalesced into one agent turn despite delayed Baileys delivery.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only same-account, same-conversation, same-sender messages with conservative tolerance, schema/types/docs, fake-timer coverage, and coordination with adjacent WhatsApp batching work.

## Reproduction Assessment

Yes, source-level. Current main builds the WhatsApp debounce key without `messageTimestamp` and uses a receive-time `setTimeout`, so two same-key text messages with the same server timestamp delivered after `debounceMs` expires will flush separately.

## Solution Assessment

Unclear for the exact proposed `messages.inbound.coalesceByTimestamp` config shape because timestamp tolerance and defaults are product decisions. The maintainable direction is a WhatsApp-owned timestamp coalescing path, not a broad shared debouncer semantic change for every channel.
