## Summary

Keep open. Current main and the latest shipped tag still use receive-time WhatsApp inbound debouncing; `messageTimestamp` is preserved on the message but is not used as a coalescing key, timer input, or config surface, and the open composing-aware debounce PR is adjacent rather than a timestamp-coalescing fix.

## What This Changes

Request to batch WhatsApp inbound messages by WhatsApp server `messageTimestamp` so delayed Baileys deliveries from the same send action reach the agent as one turn.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only same-account, same-conversation, same-sender messages with conservative tolerance, schema/types/docs, and fake-timer coverage while preserving media, command, access-control, reply-context, and self-message behavior.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the buffer before the later message arrives.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path, coordinated with the open composing-aware debounce PR, rather than a broad shared debouncer semantic change.
