## Summary

Keep open. Current main and v2026.5.22 still use receive-time WhatsApp inbound debouncing; `messageTimestamp` is preserved as metadata but not used as a grouping key, timer source, schema option, or documented batching mode. The open composing-aware debounce PR is adjacent, but it does not own this timestamp-coalescing request.

## What This Changes

Request WhatsApp inbound batching for same or near-same Baileys `messageTimestamp` messages instead of relying only on receive-time debounce windows.

## Best Possible Solution

Choose the WhatsApp-owned product contract first, then add backward-compatible timestamp coalescing for same-account, same-conversation, same-sender text messages with schema/docs/types and focused fake-timer coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the second message is enqueued.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path with conservative defaults and documented tolerance, not a broad semantic change to the shared debouncer.
