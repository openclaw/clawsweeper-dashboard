## Summary

Keep open. Current main and v2026.5.20 still batch WhatsApp inbound messages through receive-time debounce; `messageTimestamp` is preserved as message metadata but is not used as a grouping key, timer source, schema option, or documented batching behavior. The related open composing-aware debounce PR is adjacent but does not solve same-timestamp coalescing.

## What This Changes

The issue requests batching same-chat WhatsApp inbound messages by Baileys `messageTimestamp` so delayed simultaneous messages arrive in one agent turn.

## Best Possible Solution

Implement a WhatsApp-owned, backward-compatible timestamp coalescing mode for same-account, same-conversation, same-sender text messages, with conservative tolerance, docs/schema/types, and focused fake-timer coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer has already removed the first buffer.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp-owned timestamp coalescing path with conservative defaults and documented tolerance, not a broad semantic change to the shared debouncer.
