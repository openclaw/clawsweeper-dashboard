## Summary

Current main still cannot coalesce a WhatsApp message that arrives after the receive-time debounce buffer has flushed, even when it shares the earlier message’s WhatsApp timestamp. The report remains useful and has no viable replacement PR, but the safe retention/tolerance policy needs maintainer judgment because timestamp equality alone can falsely merge distinct messages.

## What This Changes

Batch delayed WhatsApp messages from the same user action using WhatsApp message timestamps rather than only local receive-time debounce.

## Best Possible Solution

Define a WhatsApp-owned bounded late-arrival policy keyed by account, conversation, sender, and timestamp bucket, with explicit expiry and exclusions for media, locations, control commands, reply-context boundaries, and already-started agent turns; reuse the existing debounce path where possible and add configuration only if no safe default policy exists.

## Reproduction Assessment

Yes, at source level: once the receive-time timer flushes a sender’s buffer, current main has no mechanism to attach a later same-timestamp WhatsApp delivery to that completed batch. The reported 5–10 second Baileys delay was not reproduced live in this read-only review.

## Solution Assessment

No, not as proposed: a 50–100 ms flush after the last same-timestamp arrival cannot bridge the reported 5–10 second delivery gap, while timestamp equality without a bounded lifecycle risks false merges. A WhatsApp-owned late-arrival policy is the narrower maintainable direction.
