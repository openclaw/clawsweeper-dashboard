## Summary

Keep open. Current main and v2026.6.8 still batch WhatsApp inbound messages through a receive-time debounce window; `messageTimestamp` is preserved and used for ordering/event metadata, but it is not a coalescing key or late-arrival grouping rule.

## What This Changes

Request to batch WhatsApp inbound messages by WhatsApp server `messageTimestamp` so delayed same-action messages arrive in one agent turn.

## Best Possible Solution

Approve a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting the chosen config/default contract.

## Reproduction Assessment

Yes, at source level. A same-key WhatsApp message that arrives after the receive-time debounce timer has flushed necessarily starts a new batch, because `messageTimestamp` is stored on the event but not used to reopen or coalesce flushed buffers.

## Solution Assessment

Unclear until maintainers choose the timestamp/default/config contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by timestamp equality.
