## Summary

Keep open. Current main still batches WhatsApp inbound messages through receive-time debounce only; `messageTimestamp` is preserved as metadata but is not part of the debounce key, flush policy, or config surface. The request is valid source-level feature work, but the exact timestamp coalescing policy and config/default behavior need maintainer judgment before an automated fix lane.

## What This Changes

Request to batch delayed WhatsApp messages that share the same WhatsApp server timestamp into one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only messages from the same account, conversation, and sender, preserving existing media, location, reply-context, access-control, self-message, and control-command bypass behavior and documenting any new config surface.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered more than `debounceMs` apart will flush as separate turns because the current key omits the timestamp and the timer is based on receive/enqueue time.

## Solution Assessment

Yes, with a narrow boundary. The maintainable direction is a WhatsApp plugin-level timestamp coalescing policy, not a broad change to shared inbound debounce semantics for every channel.
