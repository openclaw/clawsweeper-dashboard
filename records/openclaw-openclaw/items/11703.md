## Summary

Keep open. Current main still batches WhatsApp inbound messages through the shared receive-time debounce path; `messageTimestamp` is preserved as metadata but is not used as a coalescing key, timer, or config-controlled grouping rule.

## What This Changes

Request timestamp-aware WhatsApp batching so simultaneous forward-plus-comment or rapid-fire messages with the same or nearby WhatsApp server timestamp arrive as one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only messages from the same account, conversation, and sender, preserving existing media, reply-context, access-control, self-message, and control-command behavior.

## Reproduction Assessment

Yes, source-level. Two WhatsApp `notify` text messages from the same account, chat, and sender with the same `messageTimestamp`, where the second arrives after `debounceMs`, will flush as separate turns because the timer is receive-time based.

## Solution Assessment

Yes, with a narrow boundary. The maintainable fix should live in the WhatsApp plugin path and use a documented timestamp-coalescing rule rather than changing shared inbound debounce semantics for every channel.
