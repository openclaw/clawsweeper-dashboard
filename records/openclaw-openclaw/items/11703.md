## Summary

Keep open. Current main and the latest release still preserve WhatsApp `messageTimestamp` but batch inbound WhatsApp text through a receive-time debounce key, so late-arriving same-server-timestamp messages can still split into separate agent turns; the remaining decision is the WhatsApp-specific timestamp/default/config contract.

## What This Changes

Request to batch late-arriving simultaneous WhatsApp messages using Baileys `messageTimestamp` instead of only receive-time debounce.

## Best Possible Solution

Implement a WhatsApp-owned bounded server-timestamp coalescer for text-only same-account, same-conversation, same-sender messages, preserving immediate-flush exclusions and documenting the chosen config/default contract.

## Reproduction Assessment

Yes, at source level. A late Baileys message with the same WhatsApp timestamp starts a new receive-time debounce buffer after the previous timer has flushed because the debounce key excludes the timestamp and the timer is based on arrival time.

## Solution Assessment

Unclear until maintainers choose the timestamp and config/default contract. A WhatsApp-owned bounded coalescer is safer than changing the generic shared debouncer or grouping solely by second-resolution timestamp equality.
