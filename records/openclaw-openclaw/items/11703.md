## Summary

Keep open. Current main still uses WhatsApp account/conversation/sender receive-time debounce while preserving `messageTimestamp` only as metadata, and the adjacent composing debounce PR does not implement timestamp-based coalescing.

## What This Changes

Request timestamp-based batching for WhatsApp messages that share the same or very close Baileys `messageTimestamp`, especially forward-plus-comment bursts delivered several seconds apart.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only messages from the same account, conversation, and sender, with conservative documented tolerance/defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after `debounceMs` has elapsed will flush separately because the first receive-time buffer is deleted and the key has no timestamp bucket.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp plugin-level timestamp coalescing policy, not a broad shared debouncer semantic change unless maintainers explicitly choose cross-channel behavior.
