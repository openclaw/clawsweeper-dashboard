## Summary

Keep open. Current main still uses receive-time WhatsApp inbound debouncing; `messageTimestamp` is preserved for metadata/access checks but is not used as a coalescing key, timer source, config option, or documented batching mode. The related composing-aware and per-group debounce items are adjacent, not a replacement for this timestamp-coalescing request.

## What This Changes

Request WhatsApp inbound batching by WhatsApp server `messageTimestamp` so simultaneously sent forward/comment or rapid-fire messages stay in one agent turn despite delayed Baileys delivery.

## Best Possible Solution

Choose a WhatsApp-owned compatibility contract, then add conservative timestamp coalescing for same-account, same-conversation, same-sender text messages with schema/types/docs and focused fake-timer coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp text messages with the same `messageTimestamp` but delivered after `debounceMs` expires will flush separately because the receive-time timer deletes the first buffer before the second message is enqueued.

## Solution Assessment

Unclear for the exact proposed config shape. A WhatsApp-owned timestamp coalescing path with conservative defaults is more maintainable than broadly changing the shared debouncer for every channel.
