## Summary

Keep open. Current main still batches WhatsApp inbound text through account/conversation/sender receive-time debounce while preserving `messageTimestamp` only as message metadata; the related composing debounce PR is adjacent but does not implement timestamp coalescing.

## What This Changes

Request timestamp-based coalescing for WhatsApp inbound messages so same or near server-timestamp messages from one chat and sender batch into one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Add WhatsApp-owned timestamp coalescing for text-only same account/conversation/sender messages, with conservative documented tolerance/defaults, schema/types/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Two same-sender WhatsApp `notify` messages with the same `messageTimestamp` but delivered after `debounceMs` has elapsed will flush separately because the first buffer is deleted by the receive-time timer and the key has no timestamp bucket.

## Solution Assessment

Unclear for the exact proposed config shape. The maintainable direction is a WhatsApp plugin-level timestamp coalescing policy, not a broad shared debouncer semantic change unless maintainers intentionally want cross-channel behavior.
