## Summary

Keep open: current main still leaves the relevant Gateway connect auth fields unbounded, and this PR is the active fix path with sufficient terminal proof. The patch direction looks correct, but it still needs the public protocol limits documented and Gateway/security approval for the cap values before merge.

## What This Changes

The PR adds bounded connect-auth schema primitives, applies 256-character caps to bootstrap/device tokens and 4096-character caps to shared token/password fields, adds validator coverage, and records the change in the changelog.

## Best Possible Solution

Document the per-field wire limits, approve the cap values for the Gateway protocol, and merge this PR so the linked issue closes through the normal merge path.

## Reproduction Assessment

Yes. Source inspection on current main shows sub-64 KiB auth strings pass ConnectParamsSchema and can reach shared-secret or bootstrap/device-token auth paths before rejection; I did not run a live WebSocket upgrade in this read-only review.

## Solution Assessment

Mostly yes. Schema-level maxLength validation is the narrowest boundary before handshake auth work; the safer finish is to document the new wire limits and get owner/security approval for the cap values.
