## Summary

Keep open: current main and the latest release still lack the requested schema-boundary auth limits, and this PR remains the active candidate fix; the remaining blocker is compatibility acceptance for the new per-field caps.

## What This Changes

The PR adds TypeBox maxLength primitives for gateway connect-frame auth strings, applies them to `ConnectParamsSchema.auth`, and adds validator coverage for oversized values.

## Best Possible Solution

Merge this schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and make the protocol contract explicit; otherwise narrow the cap to generated bootstrap/device token fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded auth strings at `ConnectParamsSchema`, then the first-frame handler validates those params before entering auth paths that can do length-scaling comparisons; I did not run a live WebSocket repro in this read-only review.

## Solution Assessment

Yes as a mitigation, with a maintainer decision still open. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether these exact caps are the desired compatibility line.
