## Summary

Keep open: current main and the latest release still have the unbounded connect-frame auth schema, and the schema-bound hardening is useful. The PR should not merge as-is until maintainers decide the shared-secret wire contract, because the new 4096-character cap can reject existing config-backed gateway token/password values that are currently accepted.

## What This Changes

This PR adds TypeBox max-length primitives for gateway connect-frame auth fields, applies them in the protocol schema, adds validator tests, and edits the release changelog.

## Best Possible Solution

Keep protocol-schema caps for machine-issued bootstrap/device credentials, and either preserve the existing shared-secret contract or land a maintainer-approved shared-secret maximum with matching config, docs, and upgrade coverage.

## Reproduction Assessment

Yes, by source inspection: current main validates unbounded connect-frame auth strings before routing them into gateway auth, and safeEqualSecret work scales with provided byte length. I did not execute a runtime repro because this review was read-only.

## Solution Assessment

No: schema-layer bounds are the right place for protocol-issued bootstrap/device credentials, but capping config-backed shared secrets at 4096 without a matching public/config contract is not yet the safest complete fix.
