Keep open: source review supports the security hardening direction and the contributor supplied sufficient after-fix proof, but this is still an unmerged Gateway wire-contract change that needs Gateway/security owner signoff and a small protocol-docs follow-up before merge.

## What This Changes

The branch caps Gateway connect-frame `auth.bootstrapToken`/`auth.deviceToken` at 256 chars and `auth.token`/`auth.password` at 4096 chars, adds validator tests, and adds a changelog entry.

## Best Possible Solution

Land the schema caps with the protocol docs updated and Gateway/security owner approval for the cap values, then let the linked issue close through the merge.

## Reproduction Assessment

Yes, source-level: current main accepts sub-64 KiB oversized connect auth strings at `ConnectParamsSchema`, then can route bootstrap/device candidates into token comparisons under the device-bootstrap scan path. I did not run a live WebSocket upgrade in this read-only review.

## Solution Assessment

Mostly yes: schema-level `maxLength` rejection is the narrowest boundary before handshake auth work. The safer finish is to document the new wire limits and get owner/security approval for the exact caps.
