## Summary

Keep this PR open: current main still accepts unbounded connect-frame auth strings, and the PR addresses the right pre-auth protocol boundary with focused tests and real validator proof. The remaining blocker is maintainer acceptance of the new auth length caps as a compatibility tradeoff, not a discrete patch defect.

## What This Changes

The PR adds bounded TypeBox string primitives for connect-frame auth fields, applies them in the gateway protocol schema, and adds protocol validator tests for at-cap and oversized auth values.

## Best Possible Solution

Merge this focused schema-boundary mitigation after gateway/security maintainers explicitly accept the 256/4096 auth cap compatibility line, then let the linked tracking issue close through the merge.

## Reproduction Assessment

Yes from source inspection, though I did not run a live server in this read-only review. Current main accepts unbounded connect auth strings at schema validation, then the pre-auth path can pass those strings into length-scaling secret comparison and serialized bootstrap/device-token verification.

## Solution Assessment

Yes, bounding at ConnectParamsSchema is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison work. The only unresolved choice is whether the proposed 256/4096 limits are the compatibility line maintainers want to support.
