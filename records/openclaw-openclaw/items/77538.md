## Summary

Keep open: current main still has the unbounded Gateway connect-auth schema gap, and this PR is an active, focused hardening path with real behavior proof and green CI. The remaining work is Gateway/security maintainer signoff on the public cap values plus a small protocol-docs update.

## What This Changes

The PR adds bounded Gateway connect-frame auth schema primitives, applies 256-character bootstrap/device-token and 4096-character shared-secret limits, adds validator regression tests, and updates the changelog.

## Best Possible Solution

Land the schema-level hardening after documenting the auth limits in the Gateway protocol docs and getting Gateway/security owner approval on the cap values.

## Reproduction Assessment

Yes, source-level: current main validates connect params before auth work while all four `auth.*` fields are unbounded, and bootstrap/device candidates reach stored-token verification through `safeEqualSecret`. I did not run a full WebSocket upgrade in this read-only review.

## Solution Assessment

Mostly yes: rejecting oversized values in the protocol schema is the narrow boundary fix before handshake auth and constant-time comparison work. The safer finish is docs plus maintainer/security approval for the exact public limits.
