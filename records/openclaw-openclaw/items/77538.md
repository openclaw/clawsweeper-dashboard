## Summary

Keep open: current main and the latest release still accept unbounded connect-frame auth strings, and this PR remains the direct candidate fix, but the new per-field limits are a compatibility-sensitive protocol contract that needs gateway/security acceptance before merge.

## What This Changes

The PR adds maxLength-backed gateway protocol primitives for connect-frame auth fields and validator tests for at-cap and oversized auth payloads.

## Best Possible Solution

Merge this schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and the protocol contract is documented; otherwise narrow the cap to generated bootstrap/device token fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded auth strings at `ConnectParamsSchema`, then the first-frame handler validates those params before entering auth paths that can do length-scaling comparisons; I did not run a live WebSocket repro in this read-only review.

## Solution Assessment

Yes as a mitigation, with a maintainer decision still open. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether these exact caps are the desired compatibility line.
