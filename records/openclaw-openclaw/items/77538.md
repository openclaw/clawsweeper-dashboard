## Summary

Keep open: this is useful gateway pre-auth hardening, but it changes the connect-frame protocol acceptance contract and still needs gateway/security compatibility acceptance before merge.

## What This Changes

The PR adds maxLength-bound TypeBox primitives for connect-frame auth fields, applies them to `auth.token`, `auth.bootstrapToken`, `auth.deviceToken`, `auth.password`, and `auth.approvalRuntimeToken`, and adds validator regression tests for oversized values.

## Best Possible Solution

Merge this schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and the protocol contract is documented; otherwise narrow the cap to generated bootstrap/device token fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded connect auth strings at schema validation, then passes accepted shared-secret/bootstrap/device token values into pre-auth auth handling and length-scaling comparison paths.

## Solution Assessment

Yes as a mitigation, with a maintainer cap decision still open. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether these exact caps are the desired compatibility line.
