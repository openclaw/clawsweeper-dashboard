## Summary

Keep open: the security mitigation is plausible and well-proven, but it changes the public connect-auth acceptance contract, and current docs plus the latest release only establish the broader pre-auth frame limit, so gateway/security owner acceptance is still needed before merge.

## What This Changes

The PR adds TypeBox maxLength primitives for connect-frame auth strings, applies them to all five `ConnectParamsSchema.auth` fields, and adds validator tests for at-cap and oversized values.

## Best Possible Solution

Merge this schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and make the protocol contract explicit; otherwise narrow the cap to generated bootstrap/device token fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded connect auth strings at schema validation, then the pre-auth path can pass those strings into length-scaling secret comparison and serialized bootstrap-token verification.

## Solution Assessment

Yes, with a maintainer decision still open. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether these exact caps are the desired compatibility line.
