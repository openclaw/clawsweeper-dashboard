## Summary

Keep this PR open: the security hardening is aimed at a real pre-auth amplification path, but the proposed schema caps are not yet aligned with the existing shared gateway credential contract, so the patch can reject previously accepted token/password configurations before auth runs.

## What This Changes

The PR adds maxLength-backed primitives for connect-frame `auth.*` fields, applies them to `ConnectParamsSchema`, adds focused validator tests, and updates the changelog.

## Best Possible Solution

Land the pre-auth schema hardening after the credential-length contract is consistent across the wire schema, config/startup validation, protocol docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main accepts a sub-64 KiB connect frame with oversized `auth.*` strings, then later compares those strings through `safeEqualSecret` on auth/bootstrap paths. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: schema-level rejection is the right narrow layer for attacker-controlled connect payloads, but this patch also changes the accepted shared credential contract without matching config validation or docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Align auth caps with configured credential contracts:** `src/gateway/protocol/schema/frames.ts:64-67`
  - body: The new 256-character schema cap also applies to `auth.token`, but `gateway.auth.token` and remote token inputs are still accepted as unconstrained `SecretInput` values and the docs only tell operators to use a long random token. An existing gateway configured with a 257+ character shared token would pass config/startup resolution, then every WS connect would be rejected as invalid params before auth comparison. Please either document and validate the same cap at the credential/config boundary, or split the wire caps so shared tokens/passwords preserve the existing credential contract while bootstrap/device tokens stay tightly bounded.
  - confidence: 0.83
