## What This Changes

The PR adds bounded Gateway connect auth string primitives, applies them to `ConnectParamsSchema`, adds validator regression tests, and adds changelog entries.

## Best Possible Solution

Land the schema-level caps with one accurate changelog entry, protocol docs for the new auth field limits, and maintainer/security approval of the chosen bounds.

## Reproduction Assessment

Yes, source-level. Current main accepts sub-64 KiB oversized `auth.*` strings at the connect schema, and those values can reach shared-secret or bootstrap/device-token secret comparison paths before rejection.

## Solution Assessment

Mostly yes: rejecting oversized auth values in the protocol schema is the narrow boundary fix before handshake secret comparison. The safer finish is to document the new public limits and collapse the stale changelog duplicate before maintainer/security review.
