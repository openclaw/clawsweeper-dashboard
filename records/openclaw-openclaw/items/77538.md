## Summary

Keep open: this is a focused gateway security hardening PR with a source-reproducible pre-auth cost problem, but it changes the connect protocol compatibility line and still needs gateway/security owner acceptance or a narrower cap before merge.

## What This Changes

The PR adds TypeBox maxLength primitives for connect-frame auth strings, applies them to the `ConnectParamsSchema.auth` fields, and adds validator coverage for at-cap and oversized values.

## Best Possible Solution

Merge this schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and make the protocol contract explicit; otherwise narrow the caps to generated bootstrap/device token fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded connect auth strings at schema validation, then the pre-auth path can pass those strings into length-scaling secret comparison and serialized bootstrap/device-token verification.

## Solution Assessment

Yes, with a maintainer decision still open. Bounding `ConnectParamsSchema` is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison; the unresolved part is whether these exact caps are the desired compatibility line.
