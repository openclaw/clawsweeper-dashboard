## Summary

Keep open: current main still has unbounded Gateway connect auth strings, and this PR applies a narrow schema-level bound with regression coverage and real validator proof. The remaining blocker is maintainer/security review of the public protocol caps plus a small protocol docs update.

## What This Changes

The branch adds Gateway connect-auth max-length schema primitives, applies them to `ConnectParamsSchema.auth`, adds validator regression tests, and adds a changelog entry.

## Best Possible Solution

Land the schema-level caps after documenting the public auth limits and getting Gateway/security maintainer signoff on the chosen values.

## Reproduction Assessment

Yes, source-level: current main validates connect params before auth work while the four `auth.*` fields are unbounded, and oversized bootstrap/device tokens reach `safeEqualSecret` through the bootstrap/device-token verification paths. I did not live-drive a full WebSocket upgrade in this read-only review.

## Solution Assessment

Mostly yes: rejecting oversized values in the protocol schema is the narrow boundary fix before handshake secret comparison. The safer finish is to document the changed wire contract and get maintainer/security approval for the exact caps.
