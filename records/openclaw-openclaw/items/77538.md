## Summary

Keep open: this PR fixes a real Gateway pre-auth schema-boundary issue, and the latest branch proof shows the intended validator behavior. Two narrow cleanup items remain before merge: remove the stale duplicate changelog entry and document the new public connect auth field limits.

## What This Changes

The PR adds bounded Gateway connect auth string schema primitives, applies 256/4096 character caps to connect auth fields, adds validator regression tests, and adds changelog entries.

## Best Possible Solution

Land the schema-level caps with one accurate changelog entry, Gateway protocol documentation for the per-field limits, and maintainer/security approval of the chosen bounds.

## Reproduction Assessment

Yes, source-level: current main validates connect params before auth work while the `auth.*` fields are unbounded, and oversized bootstrap/device tokens can reach the `safeEqualSecret` comparison path under the bootstrap-state lock. I did not live-drive a full WebSocket upgrade in this read-only review.

## Solution Assessment

Mostly yes: rejecting oversized auth values in the connect schema is the narrow boundary fix before deeper auth comparison work. The safer finish is to document the new protocol limits, collapse the stale duplicate changelog entry, and get maintainer/security signoff on the cap values.
