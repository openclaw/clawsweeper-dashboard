## Best Possible Solution

Merge this focused schema-boundary mitigation after gateway/security owners accept or adjust the 256/4096 caps and update the protocol contract if needed; let the linked issue close on merge.

## Reproduction Assessment

Yes. Source inspection shows current main accepts unbounded connect auth strings at schema validation, then the pre-auth path can pass those strings into length-scaling secret comparison and serialized bootstrap/device-token verification.

## Solution Assessment

Yes, with one maintainer decision still open. Bounding at ConnectParamsSchema is the narrowest effective layer because it rejects oversized values before auth resolution and token comparison work; the unresolved part is whether these exact caps are the desired compatibility line.
