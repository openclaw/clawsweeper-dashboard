## Summary

Keep open: the source-level fix direction is sound for the unbounded pre-auth auth fields, but this external security PR still lacks real behavior proof and has minor docs/changelog cleanup before merge.

## What This Changes

The PR adds max-length primitives for Gateway connect auth fields, applies them to ConnectParamsSchema, adds protocol validation tests, and adds changelog entries.

## Best Possible Solution

Land a schema-level cap after the contributor supplies redacted real gateway/WebSocket proof, the duplicate changelog is collapsed, protocol docs mention the field limits, and maintainer/security review agrees on the caps.

## Reproduction Assessment

Yes, source-level: current main's ConnectParamsSchema uses unbounded Type.String auth fields, validateConnectParams runs before auth work, and the bootstrap/device-token paths eventually call safeEqualSecret on attacker-provided strings. I did not run a live WebSocket reproduction in this read-only review.

## Solution Assessment

Mostly yes: rejecting oversized auth values at the protocol schema is the narrow boundary fix and avoids reaching secret comparison work. The PR still needs proof plus documentation and changelog cleanup before it is merge-ready.
