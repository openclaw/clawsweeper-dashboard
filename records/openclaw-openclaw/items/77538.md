## Summary

Keep this PR open for a small follow-up. The schema-level hardening is the right layer and the follow-up commit fixes the earlier 256-character shared-secret mismatch, but the new wire credential caps still need to be made visible at the protocol/config boundary and the changelog entry needs the contributor attribution required by repo policy.

## What This Changes

The PR adds maxLength-backed connect auth primitives, applies them to `ConnectParamsSchema`, adds focused validator coverage, and records the Gateway security fix in the changelog.

## Best Possible Solution

Land the pre-auth schema hardening after the 4096-character shared-secret cap is also surfaced through startup/config diagnostics and public protocol docs, with the missing changelog credit added.

## Reproduction Assessment

Yes. Source inspection shows current main accepts sub-64 KiB oversized `auth.*` strings at connect validation, and bootstrap/device paths can then compare those values through length-scaling `safeEqualSecret` calls under the bootstrap-state lock.

## Solution Assessment

No, not yet. Schema-level rejection is the narrow maintainable fix for attacker-controlled connect payloads, but the patch introduces shared-secret wire caps without matching public docs or startup/config diagnostics for operators.
