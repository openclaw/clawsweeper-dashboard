## Summary

Keep open: current main still accepts unbounded connect-frame auth strings, and this PR is a valid security-hardening direction, but the latest diff still needs the shared-secret cap contract resolved and external real behavior proof before merge.

## What This Changes

The PR adds bounded TypeBox primitives for connect-frame auth fields, applies them to `ConnectParamsSchema`, adds validator coverage, and records the fix in the changelog.

## Best Possible Solution

Land schema-level bounds for bootstrap/device tokens after the shared-secret wire cap is either made an explicit documented startup-validated credential contract or removed from shared-secret fields.

## Reproduction Assessment

Yes. Source inspection shows current main accepts sub-64 KiB oversized `auth.*` strings at connect validation, and the bootstrap/device path can compare attacker-provided token length through `safeEqualSecret` while iterating stored state under a lock.

## Solution Assessment

No, not yet. Schema-level rejection is the narrow right shape for protocol-issued bootstrap/device tokens, but the shared-secret cap needs a published startup/config contract or should not apply to operator-configured shared secrets.
