## Summary

Keep open: current main still lacks the requested connect-frame auth bounds and this PR is the active candidate, but the patch should not merge as-is because `Type.String({ maxLength })` is a grapheme-count limit and does not byte-bound the pre-auth secret-comparison work this hardening is meant to cap.

## What This Changes

The branch adds bounded TypeBox primitives for five connect-frame `auth.*` fields and a new gateway-protocol test for at-cap and over-cap auth values.

## Best Possible Solution

Keep the schema-boundary fix, but make the bound byte-accurate or ASCII-shaped for protocol-issued tokens, add non-ASCII oversized regressions, and merge only after maintainers accept the stricter auth-field caps.

## Reproduction Assessment

Yes. Source inspection shows the PR relies on TypeBox `maxLength`, and TypeBox 1.1.39 enforces that as grapheme count while the vulnerable comparison path allocates and scans by UTF-8 byte length.

## Solution Assessment

No. The schema boundary is the right layer, but the current primitive is not the best fix for byte/allocation hardening; use a byte-length guard or an ASCII token shape for protocol-issued credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use a byte or ASCII bound for auth tokens:** `packages/gateway-protocol/src/schema/primitives.ts:32-34`
  - body: `Type.String({ maxLength })` is not a byte limit in TypeBox 1.1.39; it checks grapheme count. A payload made from a small number of graphemes with many combining marks can still fit under this schema cap while carrying tens of KB of UTF-8 data into `safeEqualSecret`, which pads and scans by byte length. Since this PR is meant to bound pre-auth allocation work, add a byte-length guard or restrict protocol-issued tokens to an ASCII token shape, and cover the non-ASCII oversized case in tests.
  - confidence: 0.92
