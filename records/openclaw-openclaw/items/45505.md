## Summary

Keep open: current main and v2026.6.6 still have the narrow counter-only late-success reconciliation, while the requested first-class compaction-completion boundary across session state remains a product/architecture decision.

## What This Changes

The issue requests a generalized reconciliation model for post-timeout compaction completion so session counters, token freshness, status/UI state, and related consumers converge after late success.

## Best Possible Solution

Define a narrow core compaction-completion reconciliation boundary for monotonic compaction-derived session state, then migrate only clearly related counters and token freshness there while leaving unrelated delivery and timeout behavior out of scope.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main shows the requested convergence model is still fragmented, though a live timeout-late-success run was not executed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the ownership boundary; the issue direction is plausible, but the best solution should be a narrow compaction-derived-state reconciler rather than a broad timeout framework.
