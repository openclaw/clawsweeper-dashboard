## Summary

Keep open. Current main still has the narrow late-success counter/timestamp repair, while the requested authoritative reconciliation boundary for token freshness, session rotation, status-visible state, memory/auth consumers, checkpoints, and lifecycle semantics is not implemented.

## What This Changes

Requests a generalized reconciliation boundary for compaction completions that become known after run finalization or timeout handling has moved on.

## Best Possible Solution

Introduce one idempotent compaction-success reconciliation boundary that accepts authoritative completion facts and converges only monotonic, clearly compaction-derived session state.

## Reproduction Assessment

Yes, at source level. Current main shows the late-success hook still reaches a count-only helper, while richer token, rotation, status, memory, auth, and checkpoint state are updated through separate paths.

## Solution Assessment

Yes, the issue's direction is the maintainable shape if scoped carefully. The safer version is a single idempotent reconciliation boundary, not more scattered finalizer writes or a broad timeout framework.
