## Summary

Current main fixes the original stale compaction counter, but successful-compaction event handling and run-finalization accounting remain parallel state writers. The broader ownership and convergence question is therefore still necessary and needs maintainer design judgment.

## What This Changes

Generalize successful late-compaction reconciliation into one authoritative, monotonic path for durable session state.

## Best Possible Solution

Define a lifecycle-owned reconciliation result containing only durable facts from successful compaction, then make session accounting apply it idempotently while keeping presentation and delivery state separate.

## Reproduction Assessment

Yes at source level: current main visibly separates successful-compaction count reconciliation from finalization-time token, usage, and session accounting, although no live timeout scenario was run during this read-only review.

## Solution Assessment

Yes directionally: one narrow durable reconciliation boundary is preferable to adding further parallel repairs, but maintainers must first decide its exact data contract and lifecycle owner.
