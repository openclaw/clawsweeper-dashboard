## Summary

Keep open: the narrow late-success compaction counter repair exists, but current main and the latest release still split compaction-derived session state across the event handler, run finalization, token/status readers, memory flush gating, auth-profile rotation, and cron/session update paths.

## What This Changes

The issue requests a generalized authoritative reconciliation boundary for compaction completion that can converge compaction-derived session state after timeout-late success.

## Best Possible Solution

Define a narrow core compaction-completion reconciliation boundary for monotonic compaction-derived session state, then migrate only clearly related counters/token freshness there while leaving unrelated delivery and timeout behavior out of scope.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main shows the requested convergence model is still fragmented, though a live timeout-late-success run was not executed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the ownership boundary; the issue direction is plausible, but the best solution should be a narrow compaction-derived-state reconciler rather than a broad timeout framework.
