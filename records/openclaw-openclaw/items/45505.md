## Summary

Keep open. Current main still has the narrow late-success `compactionCount` reconciliation, but token/status freshness, memory flush markers, auth-profile rotation, and checkpoint metadata remain separate paths, so the broader state-convergence design follow-up is not implemented or safely superseded.

## What This Changes

Request to generalize timeout-late compaction completion reconciliation so compaction-derived session state converges after success is learned late.

## Best Possible Solution

Introduce one idempotent compaction-success reconciliation boundary that accepts authoritative completion facts and converges only monotonic compaction-derived session state, with explicit exclusions for unrelated timeout and UI delivery policy.

## Reproduction Assessment

Yes, at source level. Current main routes timeout-late compaction success through a counter-only helper while token freshness, status rendering, memory/auth consumers, and checkpoint state remain on separate update paths.

## Solution Assessment

Yes, the requested direction is maintainable if scoped carefully. The best path is a single idempotent reconciliation boundary for compaction-derived facts, not scattered finalizer writes or a broad timeout framework.
