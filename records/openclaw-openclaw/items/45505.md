Keep open. Current main still has a narrow late-success compaction-count repair, while token freshness, status rendering, memory/auth consumers, lifecycle behavior, and checkpoint state remain on separate reconciliation paths that need maintainer scoping.

## What This Changes

The issue requests a generalized authoritative post-compaction completion reconciliation model across session counters, token snapshots, status/UI state, lifecycle events, and timeout-late-success recovery.

## Best Possible Solution

Introduce one idempotent compaction-success reconciliation boundary that accepts authoritative completion facts and converges only monotonic, clearly compaction-derived session state.

## Reproduction Assessment

Yes, at source level. Current main still routes timeout-late compaction success through a count-only helper while token freshness, status rendering, memory/auth consumers, and checkpoint state remain on separate update paths.

## Solution Assessment

Yes, the requested direction is the maintainable shape if scoped carefully. The safer path is a single idempotent reconciliation boundary, not scattered finalizer writes or a broad timeout framework.
