## Summary

Keep open. Current main still implements the predecessor count-only late compaction reconciliation, but the broader authoritative post-timeout compaction completion model requested here is not implemented across token freshness, rotated session identity, status-visible context state, memory-flush gating, and auth-profile rotation.

## What This Changes

Introduce a generalized, monotonic post-timeout compaction completion reconciliation model for compaction-derived session state after late success.

## Best Possible Solution

Define one idempotent compaction-success reconciliation boundary that owns durable compaction-derived updates and deliberately routes count, token freshness, rotated session identity, and dependent status/memory/auth state through it.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: simulate a successful compaction_end after run finalization has moved on and current main’s late path can only repair compactionCount/updatedAt. The richer state remains tied to finalizer-time metadata paths.

## Solution Assessment

No. The current count-only repair was the right narrow predecessor, but the requested full solution should be a single idempotent reconciliation helper or event rather than additional scattered finalizer writes.
