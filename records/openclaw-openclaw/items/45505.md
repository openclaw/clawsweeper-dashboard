## Summary

Keep open: current main has the narrow late-success `compactionCount` reconciliation from the related merged PR, but the broader request is still real because compaction-derived session state remains split across completion handling, run finalization, memory flush gating, auth-profile rotation, cron usage persistence, and status readers.

## What This Changes

The issue asks OpenClaw to generalize late compaction-completion reconciliation so compaction-derived session state converges after timeout-late success.

## Best Possible Solution

Define a narrow core compaction-completion reconciliation boundary for monotonic compaction-derived session state, then migrate only clearly related counters and token freshness there while leaving delivery/UI timing and unrelated timeout behavior out of scope.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main shows the requested convergence model is still fragmented, though I did not run a live timeout-late-success repro in this read-only sweep.

## Solution Assessment

Is this the best way to solve the issue? The issue direction is reasonable, but the exact implementation needs maintainer product/architecture judgment so it does not become a broad timeout framework or over-reconcile unrelated UI/delivery state.
