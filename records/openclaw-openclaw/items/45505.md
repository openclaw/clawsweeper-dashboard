## Summary

Keep open. Current main still has the narrow post-success `compactionCount` reconciliation from the earlier fix, but the broader session-state convergence problem remains split across token/status, memory flush, auth-profile, and checkpoint paths; the open adjacent PRs cover pieces of that surface but do not supersede this design follow-up.

## What This Changes

The issue asks for a generalized post-timeout compaction completion reconciliation model across compaction-derived session state.

## Best Possible Solution

Introduce one idempotent compaction-success reconciliation boundary that accepts authoritative completion facts and converges only monotonic, clearly compaction-derived session state while leaving unrelated timeout/UI policy out of scope.

## Reproduction Assessment

Yes, at source level. Current main routes timeout-late compaction success through a counter-only helper while token freshness, status rendering, memory/auth consumers, and checkpoint state remain on separate update paths.

## Solution Assessment

Yes, the requested direction is maintainable if scoped carefully. The best path is a single idempotent reconciliation boundary for compaction-derived facts, not scattered finalizer writes or a broad timeout framework.
