## Summary

Keep open. Current main covers runtime/scoped Node compile-cache support and a public no-respawn env knob, but the requested first-class prewarmed/shipped cache workflow is still absent and needs maintainer product and supply-chain direction.

## What This Changes

Request a cold-start optimization that prewarms or ships Node compile-cache data and exposes a public opt-in to avoid CLI self-respawn in controlled container environments.

## Best Possible Solution

If accepted, prefer an explicit operator-run warm-cache command with cache scoping, invalidation tests, and container/server docs; defer postinstall or shipped-cache behavior until supply-chain policy is settled.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence proof of the remaining gap: runtime compile-cache and OPENCLAW_NO_RESPAWN exist, but no first-class prewarm command, shipped cache artifact, or postinstall warm path exists on current main.

## Solution Assessment

Unclear. The existing OPENCLAW_NO_RESPAWN path is a narrow maintainable solution for respawn overhead, but maintainers still need to choose the prewarming surface and supply-chain posture for compile-cache artifacts.
