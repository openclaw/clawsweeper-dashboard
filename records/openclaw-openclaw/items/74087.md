## Summary

Keep open. Current main already covers runtime compile-cache enablement and a public no-respawn opt-in, but the requested first-class cache prewarming surface, shipped cache artifact, or automatic postinstall warm path is still absent and needs maintainer direction.

## What This Changes

Request to reduce stateless container cold starts by adding a prewarmed Node compile-cache surface and a public opt-in to skip CLI self-respawn.

## Best Possible Solution

Keep the issue open until maintainers choose the public prewarming surface. The safest likely end state is an explicit operator-run `openclaw warm-cache --output ...` command with container/VPS docs and focused tests, while leaving automatic postinstall priming or shipped cache artifacts for explicit supply-chain approval.

## Reproduction Assessment

Yes for the remaining repository gap: current source, docs, tests, and command catalogs show compile-cache and no-respawn support but no first-class warm-cache or postinstall prewarm path. The exact performance numbers still need a cold-start benchmark on a current packaged build.

## Solution Assessment

Unclear for the remaining work. The existing env knobs are narrow and maintainable, but the prewarm mechanism is a product and supply-chain choice; an explicit command appears safer than automatic postinstall or shipped artifacts if maintainers accept the feature.
