## Summary

Keep open. Current main implements the reusable Node compile-cache path and the public `OPENCLAW_NO_RESPAWN` opt-in, but it still does not provide a first-class prewarm command, shipped cache artifact, or automatic postinstall warm path requested for stateless container cold starts.

## What This Changes

Request to add a first-class Node compile-cache prewarming path and a public no-respawn opt-in for faster container cold starts.

## Best Possible Solution

If maintainers accept the feature, prefer an explicit operator-run `openclaw warm-cache` command with cache scoping, invalidation tests, and container/server docs; defer postinstall warming or shipped artifacts until supply-chain policy is settled.

## Reproduction Assessment

Yes for the remaining repository gap: current source, docs, tests, and command catalogs show runtime compile-cache and no-respawn support but no first-class cache prewarming command, postinstall warm path, or shipped cache artifact. The exact performance gain still needs a cold-start benchmark on a current packaged build.

## Solution Assessment

Unclear. The existing env knobs are narrow and maintainable, but choosing between an explicit prewarm command, postinstall warming, or shipped cache artifacts is a product and supply-chain decision rather than a single obvious repair.
