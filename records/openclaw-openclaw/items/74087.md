## Summary

Keep open. Current main already has packaged-install Node compile-cache enablement, docs for `NODE_COMPILE_CACHE`, and the public `OPENCLAW_NO_RESPAWN` escape hatch, but it still lacks the requested first-class cache prewarming command, shipped cache artifact, or automatic postinstall prewarm path. The remaining decision has product and supply-chain implications, so it should stay with maintainers.

## What This Changes

The issue requests cold-start improvements by adding a prewarmed Node compile-cache workflow or artifact and a public opt-in to bypass CLI self-respawn.

## Best Possible Solution

Keep the issue open until maintainers choose the public prewarming surface. If accepted, the safest likely end state is an explicit operator-run `openclaw warm-cache` command with focused tests and VPS/container docs; automatic postinstall priming or shipped cache artifacts should wait for explicit approval of install-time execution and cache invalidation tradeoffs.

## Reproduction Assessment

Yes for the remaining repository gap: current source, docs, command catalogs, and postinstall script inspection show compile-cache/no-respawn support but no first-class warm-cache command or postinstall prewarm path. The performance speedup claim itself still needs a cold-start benchmark on current packaged builds to quantify.

## Solution Assessment

Unclear. The already-implemented env knobs are narrow and maintainable, but the remaining prewarm path is a product and supply-chain decision; an explicit command appears safer than automatic postinstall prewarming if maintainers want the feature.
