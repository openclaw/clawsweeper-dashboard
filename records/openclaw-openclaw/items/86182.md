## Summary

Keep open: current main and v2026.6.6 include the related alpha-bucket picker work, but the remaining issue is the still-real Discord picker capacity/design constraint rather than a completed implementation.

## What This Changes

The issue asks maintainers to track Discord picker capacity limits around 25 select options, 5 action rows, and 100-character custom IDs for very large wildcard provider/model configs.

## Best Possible Solution

Keep this as the Discord picker design tracker and choose a durable search or state-token architecture only if the bucketed picker still causes real user pain.

## Reproduction Assessment

Yes, at source level: current main still exposes the 25-option, 5-row, and 100-character Discord constraints in the picker code, with tests documenting the bucket workaround. I did not run a live Discord repro.

## Solution Assessment

Unclear: the merged bucket implementation is a good partial mitigation, but the remaining structural concern needs a maintainer choice between autocomplete/search, modal flow, or short server-side state tokens.
