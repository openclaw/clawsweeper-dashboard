## Summary

Keep open: current main and v2026.6.8 include the alpha-bucket mitigation, but the central structural Discord picker limits remain and the next step is a maintainer UX/state-model decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to track structural Discord model-picker limits from 25 select options, 5 action rows, and 100-character custom IDs for large wildcard model/provider catalogs.

## Best Possible Solution

Keep this as the Discord picker design tracker, then implement one maintainer-approved search or short-token state architecture in the Discord plugin if real large-catalog usage warrants it.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still expose the 25-option, 5-row, and 100-character Discord picker constraints in code and tests. I did not run a live Discord repro with a 1000+ model catalog.

## Solution Assessment

Unclear: the shipped bucket implementation is a reasonable partial mitigation, but the remaining issue needs a maintainer choice between autocomplete/search, modal flow, short-lived server-side state, or a narrower no-change decision.
