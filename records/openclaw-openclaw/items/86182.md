## Summary

Keep open: current main and v2026.6.9 still contain the bucketed Discord picker workaround, but they do not implement a larger search/modal/session-state redesign for very large wildcard catalogs or future row/custom_id pressure.

## What This Changes

Track a Discord model picker redesign for very large provider/model catalogs that exceed select-menu, action-row, and custom_id capacities.

## Best Possible Solution

Keep this as the canonical Discord picker capacity tracker until maintainers choose a Discord-local search, modal, or short-lived state-token design and validate it against large wildcard catalogs.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 still encode the 25-option, 5-row, and 100-character constraints and rely on bucket/customId workarounds. I did not run a live Discord repro with an extreme wildcard catalog.

## Solution Assessment

Unclear: the issue is the right tracker, but implementation should wait for maintainers to choose the durable Discord UX and state model. Autocomplete, modal search, and server-side state each solve different parts of the browse and custom_id pressure.
