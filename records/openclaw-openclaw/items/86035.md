## Label Justifications

- `P1`: The issue reports a current Control UI/WebChat input regression that makes normal CJK chat entry unusable for affected users.

## Summary

The report remains necessary: current main and the latest release still update the controlled Control UI textarea draft on every input event, while the prior related work only guarded Return handling for IME composition.

## What This Changes

The issue requests fixing the Control UI/WebChat composer so CJK IME composition is not interrupted before committed text reaches the textarea draft.

## Best Possible Solution

Add composition-state handling in the Control UI composer so resize and slash-menu behavior can still observe textarea contents while draft commits are deferred until `compositionend`.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: the textarea is controlled, Enter composition is guarded, but input events still commit draft state immediately. I did not run the full macOS Chrome IME workflow, so this is source-reproducible rather than directly reproduced here.

## Solution Assessment

Yes, the likely solution is narrow: keep the existing composer, but track composition state and commit the final textarea value on `compositionend` instead of changing gateway, CLI, TUI, or native macOS text-view behavior.
