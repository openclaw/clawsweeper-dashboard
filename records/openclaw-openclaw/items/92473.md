## Label Justifications

- `P2`: The issue reports a bounded but real disclosure of internal routing details without outage, data loss, credential exposure, or an unusable workflow.
- `impact:auth-provider`: The recipient-facing notice reveals the selected and active provider/model routing state and fallback reason.

## Summary

Current main and the latest release intentionally send fallback transitions through the normal reply surface, including when verbose mode is off, so this is a valid privacy/presentation concern but not a regression against the established contract. Keep it open for a maintainer decision on whether fallback transparency should be operator-only, opt-in, or recipient-visible by default.

## What This Changes

Change model fallback transition notices so internal provider and model-routing details are not exposed to external message recipients by default.

## Best Possible Solution

Define one core reply-presentation policy that distinguishes operator-facing status from recipient-facing content, retains fallback details in structured logs and lifecycle events, and applies the chosen upgrade-safe default consistently before transport dispatch.

## Reproduction Assessment

Yes, at source level. Shipped and current code deterministically creates fallback status payloads, bypasses source suppression for them, and tests delivery with verbose mode off; the reporter also supplied a post-restart workaround smoke, though this review did not run a live external channel.

## Solution Assessment

No, not yet. Channel-specific gating or a new configuration key would be premature; the best fix is a single shared operator-versus-recipient visibility policy, reconciled with https://github.com/openclaw/openclaw/pull/69399 before implementation.
