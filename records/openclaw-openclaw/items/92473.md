## Summary

Current main intentionally sends model-fallback transitions as operational status messages on the normal reply surface, including when source replies are otherwise suppressed, and the shipped docs describe that behavior. The report identifies a valid privacy and presentation concern, but changing it would reverse an established product contract and conflicts with the still-open opt-in proposal at https://github.com/openclaw/openclaw/pull/69399, so maintainers need to choose one shared visibility policy before implementation.

## What This Changes

Prevent raw model-fallback routing notices from reaching external user-facing messaging recipients while retaining internal telemetry and lifecycle events.

## Best Possible Solution

Choose a single product-wide visibility rule for fallback status messages, implement it at the shared reply/delivery boundary, and update the existing transition tests and model-failover docs; avoid per-channel exceptions and add a config knob only if maintainers explicitly want operator choice.

## Reproduction Assessment

Yes at the source-contract level: current main deliberately places fallback notices in normal reply delivery, and focused E2E coverage requires them across verbose levels. The review did not run a live external transport, but the shared delivery path and the reporter's v2026.6.5 smoke make the behavior high-confidence.

## Solution Assessment

No, the suggested channel gate or new config flag is not yet the best fix because both would choose product policy implicitly. The maintainable path is a shared visibility decision first, followed by one central delivery rule and aligned tests/docs.
