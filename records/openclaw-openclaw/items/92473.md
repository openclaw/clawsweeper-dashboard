## Summary

Keep open. Current main and v2026.6.9 still intentionally deliver model-fallback and fallback-cleared notices on the reply surface, while closed opt-out requests already point back to this issue as the canonical visibility-policy decision.

## What This Changes

The issue asks OpenClaw to keep model fallback notices internal-only or gate them so external user-facing messaging channels do not receive raw model-routing banners.

## Best Possible Solution

Settle one maintainer-approved fallback-notice audience policy in the shared reply runner, keep logs and lifecycle events intact, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.9 still prepend fallback/fallback-cleared notices to reply payloads, mark them for source-suppression-bypass delivery, and test visible notices while verbose is off. I did not force a live provider fallback in this read-only review.

Unclear as a direct suppression patch. The local workaround is plausible, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy, operator transparency, and async-notice requests do not diverge.
