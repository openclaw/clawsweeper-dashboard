## Summary

Keep open. Current main and v2026.6.8 still intentionally deliver model fallback and fallback-cleared notices on the same user-facing reply surface, and related suppression/opt-out reports were closed in favor of this issue as the canonical visibility-policy decision.

## What This Changes

The issue asks OpenClaw to keep model fallback notices internal-only, or gate them by surface/channel or configuration, so raw provider/model-routing banners do not reach external message recipients.

## Best Possible Solution

Define one maintainer-approved fallback-notice visibility policy in the shared reply runner, keep lifecycle/log telemetry intact, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes. Current code still prepends fallback notices to normal reply payloads, and current tests/docs assert same-surface delivery including verbose-off runs; I did not force a live provider-fallback scenario in this read-only review.

## Solution Assessment

Unclear as a direct suppression patch. The local workaround is plausible, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy, operator transparency, and async-notice requests do not diverge.
