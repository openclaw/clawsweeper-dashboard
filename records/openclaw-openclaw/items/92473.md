## Summary

Keep open: current main and v2026.6.8 still intentionally deliver model fallback and fallback-cleared notices on the user-facing reply surface, so suppressing or gating them for external channels remains an unresolved product/config policy decision.

## What This Changes

Suppress or gate model fallback and fallback-cleared notices from external user-facing channels while preserving internal logs and lifecycle telemetry.

## Best Possible Solution

Define one maintainer-approved fallback-notice visibility policy in the shared reply runner, keep lifecycle/log telemetry intact, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, at source level: current code, docs, and tests still deliver fallback notices on the user-facing reply surface, including verbose-off runs. I did not force a live provider-fallback scenario in this read-only review.

## Solution Assessment

Unclear as a direct suppression patch. The local workaround is plausible, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy, operator transparency, and async-notice requests do not diverge.
