## Summary

Keep open. Current main and the latest release still intentionally send model fallback and fallback-cleared notices through the normal reply surface, so the reported external-channel exposure remains an unresolved fallback-notice visibility policy question rather than an already-fixed issue.

## What This Changes

The issue asks OpenClaw to keep model fallback notices internal-only or gate their delivery to external user-facing channels instead of sending raw fallback banners to recipients.

## Best Possible Solution

Define one shared fallback-notice visibility policy in the reply runner, keep lifecycle events/logs available, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds and prepends fallback notices to user-visible reply payloads, and E2E tests assert visible fallback and fallback-cleared text. I did not force a live provider outage in this read-only review.

## Solution Assessment

No, not as a mechanical suppression patch. The local workaround proves one possible behavior, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy and operator transparency do not diverge by transport.
