## Summary

Keep open: current main and the latest release still intentionally send model fallback and fallback-cleared notices on the normal reply surface, including through source-suppression-bypass delivery, so the reported external-recipient exposure remains a live product-policy question rather than implemented cleanup.

## What This Changes

The issue asks OpenClaw to stop raw model fallback notices from reaching external user-facing channels while retaining internal telemetry and lifecycle visibility.

## Best Possible Solution

Define one shared fallback-notice visibility policy, likely surface-aware or opt-in for external recipient banners, while preserving lifecycle events, logs, docs, and focused regression coverage for the chosen default.

## Reproduction Assessment

Yes, at source level. Current main deterministically builds and delivers fallback notice payloads through the shared reply path, and tests assert visible Model Fallback output even with verbose mode off.

## Solution Assessment

No, not as a channel-specific suppression patch. The maintainable fix is a shared reply visibility policy or config shape because current source, tests, and docs intentionally make these notices user-visible.
