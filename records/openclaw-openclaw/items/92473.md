## Summary

Keep open. Current main and the latest release still intentionally deliver model fallback and fallback-cleared notices on the normal reply surface, so this remains a maintainer product decision about fallback-notice audience policy rather than an already-fixed bug.

## What This Changes

Request to keep model fallback and fallback-cleared notices out of external user-facing channels, or make their delivery surface/config controlled.

## Best Possible Solution

Define one shared fallback-notice audience policy in the reply runner, preserve lifecycle events/logs for observability, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes. Current main builds and prepends fallback notices to user-visible reply payloads, latest release docs describe the same behavior, and E2E coverage asserts the notices appear on normal fallback transitions.

## Solution Assessment

No, not as a mechanical suppression patch. The local workaround proves one possible behavior, but the best solution needs a maintainer-chosen shared visibility policy so privacy, operator transparency, and channel behavior stay consistent.
