## Summary

Keep open. Current main and v2026.6.8 still intentionally deliver model fallback and fallback-cleared notices on the normal reply surface, including source-suppression bypass behavior, so this remains a maintainer visibility-policy decision rather than an already-fixed issue.

## What This Changes

Request to make model fallback notices internal-only or surface/channel/config-gated so external user-facing channels do not receive raw model-routing banners.

## Best Possible Solution

Define one shared fallback-notice visibility policy in the shared reply runner and dispatch path, keep lifecycle/log telemetry available, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds and prepends fallback notices to reply payloads, and E2E coverage asserts they appear with verbose mode off. I did not force a live provider outage in this read-only review.

## Solution Assessment

Unclear as a mechanical suppression patch. The local workaround proves one possible behavior, but the best fix needs a maintainer-chosen visibility policy so external-channel privacy and operator transparency do not diverge by transport.
