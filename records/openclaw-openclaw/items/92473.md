## Summary

Current `main` and release `v2026.6.6` intentionally send model fallback transitions on the normal reply surface, including external messaging paths. The report therefore identifies a credible privacy and presentation concern, but changing this established behavior requires a shared product-policy decision rather than a channel-specific suppression patch.

## What This Changes

Prevent raw model fallback and recovery status banners from reaching unintended external messaging recipients while retaining internal logs and lifecycle telemetry.

## Best Possible Solution

Define one typed, shared visibility policy for operational fallback messages at the reply-delivery boundary, using a privacy-safe external-recipient default while preserving lifecycle and log telemetry plus any explicitly approved operator-facing visibility mode.

## Reproduction Assessment

Yes, from source with high confidence: current `main` deliberately prepends fallback notices to normal reply payloads, and focused E2E coverage verifies this with verbose mode disabled. The reporter's exact deployment was not recreated, but the implicated external delivery behavior follows directly from the shared reply path.

## Solution Assessment

No, neither a channel-specific gate nor a new config flag is yet the best established solution. The narrow maintainable direction is a single shared delivery-policy decision reconciled with https://github.com/openclaw/openclaw/pull/69399 and the shipped behavior from https://github.com/openclaw/openclaw/pull/82676.
