## Summary

Keep open: current main and the latest release still intentionally send model fallback and fallback-cleared notices on the normal reply surface, including outside verbose mode, so the reported external-channel disclosure concern remains a live product-policy question rather than cleanup that can close.

## What This Changes

The issue asks OpenClaw to stop sending raw model fallback/fallback-cleared notices to external user-facing channels unless that visibility is explicitly allowed.

## Best Possible Solution

Define one shared core fallback-notice visibility policy: lifecycle events and logs should keep the transition data, while external user-facing replies should omit it by default or show it only through an explicit opt-in, verbose, or operator-owned surface, with docs and regression tests updated together.

## Reproduction Assessment

Yes, at source level. Current main constructs fallback notice payloads, marks them deliverable despite source-reply suppression, and has a verbose-off E2E expectation for the visible banner; I did not run tests because this review is read-only.

## Solution Assessment

No final implementation path is best yet. The narrow fix should be shared in core reply presentation policy after maintainers choose whether the default is opt-in, verbose-only, operator-only, or surface-aware.
