## Summary

Keep open. Current main and v2026.6.8 still intentionally send model fallback and fallback-cleared notices on the normal reply surface, including with verbose mode off, so this remains the canonical visibility-policy decision rather than an already-fixed issue.

## What This Changes

Request to stop raw model fallback notices from reaching external user-facing channel replies, or gate that delivery by shared surface/channel policy or config.

## Best Possible Solution

Maintainers should define one shared fallback-notice visibility policy in the reply runner, keep lifecycle/log telemetry available, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-level: current main builds fallback notices into reply payloads, and E2E coverage asserts the notices are visible even with verbose mode off. I did not force a live provider outage in this read-only review.

## Solution Assessment

Unclear as a mechanical patch. The reporter's workaround proves suppression is possible, but the best fix needs a maintainer-chosen shared visibility policy rather than a one-off external-channel drop.
