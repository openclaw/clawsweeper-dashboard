## Summary

Current main intentionally sends model fallback and recovery notices on the normal reply surface, bypassing source-reply suppression, and the public docs describe that behavior. The report therefore identifies a valid recipient-privacy and presentation concern, but resolving it requires a maintainer decision about the shared visibility contract and reconciliation with the opposite opt-in request in https://github.com/openclaw/openclaw/issues/67197 and https://github.com/openclaw/openclaw/pull/69399.

## What This Changes

Stop model fallback and recovery banners from reaching external messaging recipients while retaining internal logs and lifecycle events.

## Best Possible Solution

Choose one shared audience policy for operational notices—ideally distinguishing operator-owned surfaces from external recipients—then enforce it centrally with explicit default, documentation, and upgrade semantics.

## Reproduction Assessment

Yes at source level. Current main deterministically creates fallback and recovery payloads, marks them to bypass source suppression, and sends them through the normal reply surface, matching the reported external delivery.

## Solution Assessment

Not yet. Central shared policy is safer than channel-specific gating, but maintainers must first decide whether external notices should be disabled, opt-in, verbose-only, or audience-aware.
