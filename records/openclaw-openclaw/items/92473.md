## Summary

Current main and release `v2026.6.6` intentionally deliver model-fallback transitions on the normal user reply surface, including with verbose mode off. The report raises a valid privacy and presentation concern, but changing this behavior requires a shared product-policy decision rather than a channel-specific suppression patch.

## What This Changes

Keep automatic model-fallback and recovery notices out of external user-facing channel replies while retaining internal logs and lifecycle telemetry.

## Best Possible Solution

Adopt one shared visibility policy for typed fallback status notices, retaining lifecycle/log telemetry and existing transcript/TTS exclusions, then align the runner, focused delivery tests, and failover documentation with that decision.

## Reproduction Assessment

Yes at source level: current main and release `v2026.6.6` intentionally place fallback transitions on the normal user reply surface, and focused E2E tests assert that behavior. A live external-channel run was not performed during this read-only review.

## Solution Assessment

No, neither per-channel filtering nor a new configuration option is established as the best solution. The narrow maintainable boundary is the shared typed reply-status policy, and maintainers must first choose the intended default.
