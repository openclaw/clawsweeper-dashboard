## Summary

Keep open. Current main and the latest release still intentionally send model fallback and fallback-cleared notices on the normal reply surface, including with verbose mode off, so the reported external-channel leak remains a real visibility-policy question rather than an already-fixed bug.

## What This Changes

The issue asks OpenClaw to keep model fallback notices internal or otherwise prevent raw fallback banners from reaching external user-facing channels.

## Best Possible Solution

Define one shared fallback-notice visibility policy in the shared reply runner, keep lifecycle events/logs available, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds and prepends fallback notices to user-visible reply payloads, and E2E tests assert that they appear with verbose mode off. I did not force a live provider outage in this read-only review.

## Solution Assessment

Not as a mechanical suppression patch. The local workaround proves a possible behavior, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy and operator transparency do not diverge by transport.
