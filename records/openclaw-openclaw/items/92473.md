- `P2`: The issue is a confirmed, limited-blast-radius external delivery bug, but it does not make the core runtime unusable.
- `impact:message-loss`: An internal status payload is misrouted to an external message recipient.

## Summary

Current main and the latest release deliberately deliver model-fallback status payloads through normal external reply routing, so the reported leak remains present; however, related work proposes opt-in user-facing notices, making the permanent surface policy a maintainer decision rather than a safe automatic removal.

## What This Changes

Prevent internal model-fallback notices from reaching external message recipients while retaining fallback telemetry for operators.

## Best Possible Solution

Make external fallback banners disabled by default, preserve lifecycle events and operator diagnostics, and centralize any intentional user-facing opt-in instead of using source-suppression bypass metadata.

## Reproduction Assessment

Yes at source level: current main marks fallback notices to bypass source suppression, prepends them to reply payloads, and includes a Telegram-context test expecting final delivery. No live transport reproduction was run during this read-only review.

## Solution Assessment

Unclear as a final product policy: suppressing external delivery by default is the narrowest safe correction, but maintainers should reconcile it with the related opt-in user-notification proposal before implementation.
