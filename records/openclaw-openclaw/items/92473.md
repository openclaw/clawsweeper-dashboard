## Summary

Keep open. Current main still intentionally sends model-fallback and fallback-cleared notices through normal user-facing reply delivery, while the remaining request is a real visibility/privacy policy decision about whether external recipients should see provider/model routing details.

## What This Changes

The issue asks OpenClaw to keep model fallback notices internal-only, or otherwise gate them by surface/channel, so external user-facing messaging recipients do not receive raw fallback banners.

## Best Possible Solution

Define one shared fallback-notice audience policy for reply delivery, then update the central runner, docs, and E2E coverage so external-recipient and operator/internal surfaces behave consistently.

## Reproduction Assessment

Yes at source level: current main builds fallback notice payloads, marks them as source-suppression bypass notices, and tests assert the notice appears even with verbose mode off. I did not run a live provider-failure path in this read-only review.

## Solution Assessment

Unclear as an implementation request: suppressing external delivery may be the right privacy default, but the durable fix needs a shared maintainer policy rather than one channel-specific filter.
