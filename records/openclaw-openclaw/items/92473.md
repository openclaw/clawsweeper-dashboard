## Summary

Keep open. Current main and v2026.6.6 intentionally deliver model fallback and fallback-cleared notices on the normal reply surface, including outside verbose mode, so the reported behavior is source-reproducible and remains the canonical visibility-policy question rather than something already implemented or superseded.

## What This Changes

The issue requests a shared policy so raw model fallback notices do not reach external user-facing channel recipients while fallback telemetry remains available internally.

## Best Possible Solution

Define one shared fallback-notice visibility policy at the reply-presentation layer, then update the runner, docs, and regression tests so external-recipient replies follow that policy while lifecycle events and logs keep the routing telemetry.

## Reproduction Assessment

Yes, at source level. Current main deterministically builds fallback and fallback-cleared reply payloads, marks them for source-suppression bypass, and tests verbose-off delivery; I did not force a live external-channel fallback.

## Solution Assessment

No automatic fix yet. The narrow maintainable path is a shared visibility policy in the reply layer, not per-channel suppression, and maintainers need to decide whether the default becomes surface-aware, opt-in, or remains user-visible.
