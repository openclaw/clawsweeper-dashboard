- `P2`: The shipped behavior can disclose internal routing details during fallback transitions, but the blast radius is bounded and resolution requires a normal-priority product decision.
- `impact:auth-provider`: The notice exposes selected and active provider/model routing information.

## Summary

Current main intentionally sends fallback transitions on the normal reply surface, including direct messaging channels, so this is not obsolete or already fixed. The report raises a valid privacy and audience-boundary concern, but resolving it requires maintainers to decide whether notices are operator-only, recipient-visible, or governed by a shared recipient-aware policy; the open opt-in proposal does not safely supersede this issue because it predates the shipped default and is currently conflicted.

## What This Changes

Change model fallback notice visibility so raw provider/model routing details do not reach unintended external message recipients.

## Best Possible Solution

Adopt one central recipient-aware status-delivery contract: retain fallback details in verbose logs and lifecycle events, show user-facing notices only to audiences maintainers explicitly classify as appropriate, and cover direct, routed, and source-suppressed delivery paths with upgrade-aware tests and docs.

## Reproduction Assessment

Yes at the source level: current main explicitly marks fallback notices to bypass source suppression, and a focused test proves delivery on a direct Telegram reply path. This read-only review did not force a live provider outage, so the evidence is source and test based rather than a fresh live reproduction.

## Solution Assessment

No, neither a channel-by-channel gate nor a new config flag is yet the best solution. A shared recipient-aware delivery policy is narrower and more maintainable, but maintainers must first choose the intended default and upgrade behavior.
