## Summary

Keep open. Current main and v2026.6.8 still intentionally send same-surface model fallback notices, narrower opt-out requests were closed in favor of this issue, and adjacent open work asks for more async visibility, so the remaining decision is a maintainer-owned fallback-notice visibility policy.

## What This Changes

Request to make model fallback and fallback-cleared notices internal-only or surface/channel-gated so external recipients do not see raw provider/model fallback banners.

## Best Possible Solution

Define one maintainer-approved fallback-notice visibility policy in the shared reply runner, keep lifecycle/log telemetry intact, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes. Current source, docs, release content, and E2E tests show fallback notices are intentionally built as same-surface reply payloads and can bypass normal source-reply suppression; I did not force a live provider fallback.

## Solution Assessment

Unclear as a direct suppression patch. The local workaround is plausible, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy, operator transparency, and async-notice requests do not diverge.
