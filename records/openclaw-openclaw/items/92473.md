## Summary

Keep open. Current main and the latest release still intentionally deliver model-fallback and fallback-cleared notices on the normal reply surface, including a suppression-bypass path, so the issue remains necessary as the canonical maintainer decision point for fallback-notice visibility in external channels.

## What This Changes

The issue asks OpenClaw to stop exposing raw model fallback notices to external user-facing message recipients, or to add a fallback-notice visibility policy/configuration.

## Best Possible Solution

Define one shared fallback-notice visibility policy, likely making external-channel fallback banners opt-in or surface-aware while preserving lifecycle events and verbose/operator telemetry.

## Reproduction Assessment

Yes, at source level: current main constructs fallback and fallback-cleared notice payloads, marks them to bypass source-reply suppression, and dispatches them on ordinary reply paths. I did not run a live provider timeout scenario in this read-only review.

## Solution Assessment

Unclear as an implementation until maintainers choose the policy. A shared reply-presentation policy is the best shape; suppressing notices inside individual channel adapters would duplicate core fallback behavior and leave sibling transports inconsistent.
