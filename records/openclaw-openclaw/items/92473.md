## Summary

Keep open. Current main only narrows the leak for ambient `room_event` delivery; it still intentionally builds visible fallback and fallback-cleared reply payloads, and the latest release still documents and ships same-surface fallback notices, so the external-channel visibility policy remains unresolved.

## What This Changes

The issue asks OpenClaw to make model fallback notices internal-only or surface-gated so external recipients do not see raw provider/model fallback banners.

## Best Possible Solution

Maintainers should define one shared fallback-notice visibility policy at the reply-delivery boundary, keep lifecycle/log telemetry independent, and update docs plus regression coverage for direct, group/channel room-event, and internal surfaces.

## Reproduction Assessment

Yes. `origin/main` source and tests still generate visible fallback/fallback-cleared reply payloads outside verbose mode, and current main only adds a room-event suppression backstop; no live external channel run was executed in this read-only review.

## Solution Assessment

No, this should not close as fixed and should not be handled as a mechanical suppression patch. The best solution needs maintainer confirmation of the intended direct, group/channel, and internal surface policy because current docs and shipped behavior intentionally make fallback transitions visible.
