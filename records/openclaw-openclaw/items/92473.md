## Summary

Keep open. Current main and the latest release still intentionally deliver model fallback and fallback-cleared notices on the user-facing reply surface, while closed related opt-out requests already point back to this issue as the canonical visibility-policy decision.

## What This Changes

Request to make model fallback and fallback-cleared notices internal-only or surface-gated so external channels do not receive raw model-routing banners.

## Best Possible Solution

Settle one maintainer-approved fallback-notice audience policy in the shared reply runner, keep logs and lifecycle events intact, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-level: current main still builds fallback/fallback-cleared notice payloads, marks them for source-suppression-bypass delivery, and tests verbose-off same-surface delivery. I did not force a live provider fallback in this read-only review.

## Solution Assessment

Unclear as a direct suppression patch. The local workaround is plausible, but the best fix needs a maintainer-chosen shared visibility policy so external-channel privacy, operator transparency, and async-notice requests do not diverge.
