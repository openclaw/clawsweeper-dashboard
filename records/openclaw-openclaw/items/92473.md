## Summary

Keep open. Current main and the latest release still intentionally deliver model fallback and fallback-cleared notices on the reply surface, so the reported external-channel visibility problem remains a maintainer product-policy decision rather than an already-fixed issue.

## What This Changes

The issue asks OpenClaw to stop delivering model fallback banners to external user-facing channels, or make fallback-notice visibility surface-aware or configurable.

## Best Possible Solution

Define one shared fallback-notice visibility policy in the reply runner, keep lifecycle/log telemetry available, and update docs plus E2E coverage for the chosen external-channel behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds and prepends fallback notices to reply payloads, and E2E coverage asserts they appear with verbose mode off. I did not force a live provider outage in this read-only review.

## Solution Assessment

No, not as a mechanical suppression or one-off config patch. The best fix needs a maintainer-chosen shared visibility policy so privacy expectations and operator transparency do not diverge by transport.
