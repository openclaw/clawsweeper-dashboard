## Summary

Keep open. Current main still lacks a durable accepted-inbound store/replay path for Slack events interrupted by SIGTERM, and the relevant implementation candidate remains open at https://github.com/openclaw/openclaw/pull/49431 rather than merged.

## What This Changes

The issue requests wiring a durable SQLite message store into the active Slack gateway path to replay in-flight messages after SIGTERM or restart.

## Best Possible Solution

Land, split, or supersede the pending-inbound work with a durable channel/gateway ingress handoff, startup replay, shutdown handling, idempotent delivery semantics, and regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Reproduction Assessment

Yes, source-reproducible. The Slack handlers accept events into in-memory dedupe/debounce and live dispatch, while current main has no durable accepted-inbound/message-store symbols to replay the event after SIGTERM.

## Solution Assessment

Unclear for the exact SQLite API shape, but the durable pending-inbound handoff is the right direction. The maintainable path is to review, land, split, or replace https://github.com/openclaw/openclaw/pull/49431 instead of creating a duplicate repair lane.
