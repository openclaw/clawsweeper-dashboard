## Summary

Keep open. Current main still has in-process Slack ingress, memory-only accepted-message dedupe/debounce, restart/shutdown drain mitigations, and durable outbound/restart-continuation pieces, but no durable accepted-inbound Slack/gateway store and replay path for a Socket Mode event already acknowledged before SIGTERM; the relevant implementation candidate remains open and unmerged.

## What This Changes

Request to persist accepted inbound Slack/gateway messages and replay or requeue them across SIGTERM/restart so in-flight acknowledged events are not silently lost.

## Best Possible Solution

Land, split, or replace the pending-inbound recovery work with a durable accepted-inbound handoff, startup replay, shutdown handling, idempotent delivery semantics, and regression plus real restart proof for an accepted Slack turn interrupted before reply delivery.

## Reproduction Assessment

Yes, at source level: Slack Bolt ACKs Events API requests before listener dispatch, while current Slack ingress only marks the event in memory and dispatches live work. A SIGTERM after ACK and before completion still has no durable accepted-inbound record to replay.

## Solution Assessment

Unclear for the exact SQLite API shape, but durable accepted-inbound handoff is the right direction. The maintainable path is to review, land, split, or replace https://github.com/openclaw/openclaw/pull/49431 rather than start a duplicate repair lane.
