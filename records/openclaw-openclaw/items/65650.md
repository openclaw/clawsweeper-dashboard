## Summary

Keep open. Current main has restart deferral and shutdown drain work, but it still lacks a durable accepted-inbound Slack/gateway store and replay path for messages already acknowledged before SIGTERM; the related pending-inbound implementation remains open rather than merged.

## What This Changes

The issue requests wiring a durable SQLite-style message store into the active gateway path so accepted Slack messages can be replayed after SIGTERM or restart.

## Best Possible Solution

Land, split, or supersede the pending-inbound recovery work with a durable ingress handoff, startup replay, shutdown handling, idempotent delivery semantics, and regression coverage for an accepted Slack turn interrupted before reply delivery.

## Reproduction Assessment

Yes, source-reproducible. Current Slack ingress accepts events into in-memory dedupe/debounce and live dispatch without a durable accepted-inbound store, so a SIGTERM after Socket Mode ACK can still lose the event.

## Solution Assessment

Unclear for the exact SQLite API shape, but a durable pending-inbound handoff is the right direction. The maintainable path is to review, land, split, or replace https://github.com/openclaw/openclaw/pull/49431 instead of creating a duplicate repair lane.
