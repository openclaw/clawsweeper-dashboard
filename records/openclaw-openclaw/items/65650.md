## Summary

Keep open. Current main still lacks a durable accepted-inbound store/replay path for Slack events interrupted by SIGTERM, and the closest implementation candidate remains open at https://github.com/openclaw/openclaw/pull/49431 with review blockers rather than merged.

## What This Changes

The issue requests durable message-store wiring in the active gateway/Slack path so accepted in-flight messages can survive SIGTERM and replay after restart.

## Best Possible Solution

Land, split, or supersede the pending-inbound work with a durable channel/gateway ingress handoff, startup replay, shutdown handling, idempotent delivery semantics, and regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Reproduction Assessment

Yes, at source level: Slack accepts events into in-memory dedupe/debounce and live dispatch, while current main has no durable accepted-inbound store symbols to replay after SIGTERM. I did not run a live Slack restart reproduction in this read-only review.

## Solution Assessment

Unclear for the exact SQLite API shape, but durable accepted-inbound handoff is the right direction. The maintainable path is to review, land, split, or replace https://github.com/openclaw/openclaw/pull/49431 rather than opening a duplicate repair lane.
