## Summary

Keep open. Current main still lacks a durable accepted-inbound store/replay path for Slack Socket Mode messages interrupted by SIGTERM; the active path uses in-process dedupe/debounce and direct dispatch, while open PR #49431 remains the relevant implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Wire a durable SQLite-backed inbound message store into active gateway/Slack dispatch so accepted messages survive SIGTERM or restart and replay on startup.

## Best Possible Solution

Keep this issue open until OpenClaw has a durable channel/gateway ingress handoff with startup replay, shutdown handling, idempotent delivery semantics, and regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Reproduction Assessment

Yes. The report gives a concrete Slack Socket Mode plus SIGTERM scenario, and current-main source inspection confirms accepted Slack events can proceed through memory-only dedupe/debounce into live dispatch without a durable replay record.

## Solution Assessment

Unclear for the exact SQLite file/API shape, but the durable pending-inbound handoff is the right direction. The maintainable path is to review, land, split, or supersede #49431 rather than treating outbound delivery queues or restart-continuation recovery as sufficient.
