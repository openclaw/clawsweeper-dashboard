## Summary

Keep open. Current main still does not have a durable accepted-inbound message store before active Slack/gateway dispatch; it has in-memory debounce, session metadata recording, restart deferral, and outbound/session delivery recovery, but those do not replay a Slack Socket Mode event that was already accepted and then killed mid-turn. Open PR #49431 remains the relevant implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks to persist accepted inbound Slack/gateway messages before active dispatch and replay or requeue them across SIGTERM/restart so acknowledged events are not lost.

## Best Possible Solution

Keep this issue open until OpenClaw has a durable pending-inbound handoff at the channel/gateway ingress boundary, with startup replay, shutdown/restart handling, idempotent delivery semantics, and regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Reproduction Assessment

Yes. The report gives a concrete Slack Socket Mode/SIGTERM scenario, and current-main code inspection confirms accepted Slack events can move from Bolt handlers through process-local debounce into active dispatch without a durable replay record.

## Solution Assessment

No, current main is not yet the solved state. The maintainable direction is a generic pending-inbound/replay seam, likely by reviewing, landing, splitting, or superseding #49431, rather than treating outbound delivery queues or restart-sentinel continuation as sufficient.
