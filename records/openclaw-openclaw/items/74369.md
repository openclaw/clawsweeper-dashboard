## Summary

Keep open for maintainer review: this is a focused candidate for the linked numeric Control UI bug, current main and the latest release still lack the proposed composer safeguards, and the remaining question is the intentional composer-wide typing-assistance tradeoff.

## What This Changes

The PR adds plain-text-related textarea attributes to the Control UI chat composer and a rendered-attribute regression assertion.

## Best Possible Solution

Land the composer-boundary mitigation if maintainers accept the global typing-assistance tradeoff; otherwise choose a narrower input policy before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection shows the composer forwards textarea.value into the send path, so browser/OS rewriting at the textarea boundary would reach chat.send.

## Solution Assessment

Yes, the composer textarea is the right boundary to prevent value rewriting before draft capture. Downstream deformatting would be broader and could corrupt intentionally formatted text, but maintainers still need to accept the typing-assistance tradeoff.
