## Summary

Keep open. Current main and the latest release still render the Control UI chat composer without the proposed plain-text safeguards; the patch is focused and clean, but merge should stay with maintainers because it intentionally changes browser typing assistance for all composer input.

## What This Changes

The PR adds five plain-text/autocorrection-related attributes to the Control UI chat composer textarea and a regression test asserting those attributes.

## Best Possible Solution

Keep the fix at the composer textarea boundary so numeric text is preserved before draft capture, rather than adding downstream deformatting that could corrupt intentional separators.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection and the linked report show that current main captures textarea.value and forwards it through chat.send, but the browser or OS numeric auto-formatting trigger was not reproduced here.

## Solution Assessment

Yes, this is the best fix shape for the reported numeric corruption. The composer textarea is the narrow boundary where browser/OS rewriting can enter the draft; downstream message normalization would be broader and could corrupt intentionally formatted text.
