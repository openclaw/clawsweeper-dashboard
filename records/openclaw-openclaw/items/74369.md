## Summary

Keep open. Current main and the latest release still lack the composer attributes that this PR adds, and the patch is a focused candidate fix, but merging it should remain a maintainer decision because it changes browser typing-assistance defaults for every Control UI composer draft.

## What This Changes

The PR adds autocomplete, autocorrect, autocapitalize, spellcheck, and inputmode attributes to the Control UI chat textarea, plus a regression test asserting those attributes.

## Best Possible Solution

Keep the fix at the composer textarea boundary and land it only after maintainers accept the global typing-assistance tradeoff, rather than adding downstream numeric deformatting that could corrupt intentional separators.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection and the linked report support the path because current main captures textarea.value and sends that draft without numeric repair, but the browser/OS formatter trigger itself was not reproduced here.

## Solution Assessment

Yes, the composer textarea is the best fix boundary for the reported numeric corruption because it prevents browser/OS rewriting before draft capture. Downstream deformatting would be broader and could corrupt intentionally formatted text, but maintainers still need to accept the typing-assistance tradeoff.
