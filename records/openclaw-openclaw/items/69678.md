## Summary

Keep open. Current main still hard-caps isolated cron run-log summaries at 2000 UTF-16 code units before the Gateway appends `CronRunLogEntry.summary`, while the existing `cron.runLog` config only controls log-file pruning by bytes and retained line count. The issue is recent, concrete, source-reproducible, not superseded by the closed related PR, and narrow enough for a focused repair.

## What This Changes

Request to make cron run-log summary truncation configurable, or raise the hard-coded 2000-character cap, so long agent outputs remain auditable in `openclaw cron runs`.

## Best Possible Solution

Add a bounded `cron.runLog.maxSummaryChars` option under the existing run-log config, apply it anywhere cron derives persisted summary text, keep delivery payloads and `outputText` unchanged, and update schema help/docs/tests so operators understand total disk usage remains bounded by `maxBytes` and `keepLines`.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: any isolated cron final assistant/output text longer than 2000 UTF-16 code units flows through `pickSummaryFromOutput` and is reduced to 2000 characters plus an ellipsis before persistence.

## Solution Assessment

Yes. The suggested config option in `cron.runLog` is the narrowest maintainable fix because it changes only the run-log readability cap and reuses the existing retention boundary instead of altering delivery, session storage, or cron execution semantics.
