## Summary

Keep open: current main and v2026.5.12 still hard-code the cron run-log summary cap at 2000 characters, and no matching implementation PR or canonical issue now owns the requested configurable or larger summary cap. The remaining decision is user-facing policy for the cap/default/config contract, so this should not be swept closed or auto-fixed without maintainer direction.

## What This Changes

The issue requests raising or configuring the cron run-log summary truncation limit so long agent outputs remain readable in run history.

## Best Possible Solution

Define a bounded cron run-log summary cap contract, likely under `cron.runLog`, then align config schema/help, docs, helper plumbing, persistence behavior, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Any cleaned cron output longer than 2000 UTF-16 code units passes through `pickSummaryFromOutput`, receives a 2000-character prefix plus ellipsis, and is then persisted as the run-log `summary`.

## Solution Assessment

Unclear until maintainers choose policy. Adding `cron.runLog.maxSummaryChars` is a narrow implementation path, but the default value and whether `summary` should serve as full audit text are user-facing product decisions.
