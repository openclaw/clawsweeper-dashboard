## Label Justifications

- `P2`: This is a normal-priority cron auditability improvement with a clear code path and limited blast radius, but it can mislead users reviewing past deliveries.
- `impact:data-loss`: The run-log entry silently persists only the first 2000 characters plus an ellipsis for long assistant output, losing audit content from stored history.

## Summary

Keep open: current main and v2026.5.18 still hard-code the cron run-log summary cap at 2000 UTF-16 code units, with no duplicate or closing PR owning the configurable or larger-cap behavior. The code path is clear, but the remaining fix needs a maintainer choice about whether summaries are a short preview, a configurable audit field, or a separate full-output audit surface.

## What This Changes

Request that cron run-log summaries stop hard-truncating long assistant output at 2000 characters, preferably via a new cron.runLog summary cap or a larger default.

## Best Possible Solution

Define a bounded cron run-log summary contract, likely under cron.runLog or a dedicated audit-output path, then align types, schema, help text, docs, persistence plumbing, and regression coverage.

## Reproduction Assessment

Yes, at source level. Any cleaned cron output longer than 2000 UTF-16 code units passes through pickSummaryFromOutput, receives a 2000-character prefix plus ellipsis, and is then persisted as the run-log summary.

## Solution Assessment

Unclear until maintainers choose policy. A cron.runLog.maxSummaryChars setting is a plausible narrow path, but the default value and whether summaries should serve as full audit text are user-facing decisions.
