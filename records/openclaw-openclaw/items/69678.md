## Label Justifications

- `P2`: Cron run-log truncation affects auditability and operator diagnosis for long outputs, but the affected surface is limited to retained cron history summaries.
- `impact:data-loss`: The issue is about silently dropping the tail of long delivered agent text from the persisted run-log summary field.

## Summary

Keep open: current main and the latest release still hard-code the 2000-character cron summary cap, and `cron.runLog` has no supported `maxSummaryChars` setting. The source-level behavior is clear, but choosing a larger fixed cap versus a public config option is a maintainer product/config decision.

## What This Changes

The issue asks OpenClaw to stop hard-truncating cron run-log summaries at 2000 UTF-16 characters, either by adding a `cron.runLog.maxSummaryChars` setting or by raising the hard cap.

## Best Possible Solution

Keep this in core cron history and resolve it with an approved summary-retention policy, updating the helper, schema/help/docs, and focused cron tests if a configurable cap is chosen.

## Reproduction Assessment

Yes, at source level. Passing text longer than 2000 characters through the current summary picker yields a 2000-character prefix plus an ellipsis, and finished cron events store that summary directly in run history.

## Solution Assessment

Unclear until maintainers choose the product shape. A larger fixed cap is the narrowest runtime change, while `cron.runLog.maxSummaryChars` gives operators control but adds a public config contract.
