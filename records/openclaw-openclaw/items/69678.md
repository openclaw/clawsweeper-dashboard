## Label Justifications

- `impact:data-loss`: The persisted cron run-log summary silently drops the tail of long assistant output, leaving incomplete audit data for past runs.

## Summary

Keep open: current main and v2026.6.10 still hard-code cron run-log summaries at 2000 characters, and no canonical merged or open PR found in this pass changes the run-log summary cap. The remaining blocker is a maintainer product decision: raise the fixed cap or add a deliberate public config key.

## What This Changes

The issue requests making cron run-log summaries retain more than the current hard-coded 2000-character cap, possibly via a new `cron.runLog.maxSummaryChars` setting.

## Best Possible Solution

Have a maintainer choose the summary-retention policy, then update the cron helper plus config schema/docs/tests only if a new public setting is approved.

## Reproduction Assessment

Yes. Source inspection shows any cleaned text longer than 2000 characters passed to `pickSummaryFromOutput` becomes a 2000-character prefix plus an ellipsis, and that selected value is what the cron run-log writer persists.

## Solution Assessment

Unclear until maintainers choose the product shape. A larger fixed cap is the narrowest runtime change, while a `cron.runLog.maxSummaryChars` key gives operator control but adds public config/schema/docs surface.
