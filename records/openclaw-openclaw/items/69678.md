- `impact:data-loss`: Cron run history silently drops the tail of long delivered agent text from the retained summary field.

## Summary

Keep open: current main and the latest release still hard-code the 2000-character cron summary cap, and there is no supported `cron.runLog.maxSummaryChars` config. The source-level issue is clear, but the fix needs a maintainer decision because adding a new config key is a public config-surface change.

## What This Changes

The issue requests a longer or configurable cron run-log summary limit so long agent outputs remain auditable in `openclaw cron runs`.

## Best Possible Solution

Keep this in core cron history and resolve it with an approved summary-retention policy, updating the helper, schema/help/docs, and focused cron tests if a configurable cap is chosen.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` both cap `pickSummaryFromOutput` at 2000 characters, and the adjacent test suite currently expects long summaries to truncate; I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product shape. Raising the built-in cap avoids a new config surface, while `cron.runLog.maxSummaryChars` matches the reporter's proposal but requires schema/help/docs and compatibility review.
