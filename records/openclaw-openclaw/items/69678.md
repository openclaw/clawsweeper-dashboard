## Label Justifications

- `P2`: The issue affects cron run-history auditability for long agent outputs, but the blast radius is limited to retained cron summaries rather than live delivery.
- `impact:data-loss`: The persisted run-log summary silently drops the tail of long delivered assistant text, making past run audit data incomplete.

## Summary

Keep open: current main and v2026.6.10 still hard-code the 2000-character cron summary cap, and `cron.runLog` still exposes only row/legacy byte retention settings. The remaining choice is a maintainer product/config decision: raise the built-in cap or add a new public config key.

## What This Changes

The issue asks to avoid the hard-coded 2000-character cron run-log summary cap, either by adding `cron.runLog.maxSummaryChars` or raising the default.

## Best Possible Solution

Have a maintainer choose the summary-retention policy, then update the cron helper, config schema/help/docs if needed, and focused cron/config tests in one small core PR.

## Reproduction Assessment

Yes, source-reproducible: passing text longer than 2000 characters through `pickSummaryFromOutput` returns a 2000-character prefix plus an ellipsis, and finished cron events store that selected summary directly in run history.

## Solution Assessment

Unclear until maintainers choose the product shape. A larger fixed cap is the narrowest runtime change, while `cron.runLog.maxSummaryChars` gives operators control but adds schema, help, docs, and upgrade-review surface.
