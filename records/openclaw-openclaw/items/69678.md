## Summary

Keep open. Current main and the latest release still hard-cap persisted cron run-log summaries at 2000 UTF-16 code units, `cron.runLog` still only configures file pruning, and no duplicate or fixing PR surfaced in the related search.

## What This Changes

The issue asks for cron run-log summaries to avoid the hard-coded 2000-character cap, preferably by adding a `cron.runLog.maxSummaryChars` config option or raising the default.

## Best Possible Solution

Add a bounded `cron.runLog.maxSummaryChars` option with schema/help/docs/tests and feed it through summary selection before JSONL persistence, while leaving delivery payloads and full `outputText` unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any isolated cron final output longer than 2000 UTF-16 code units flows through `pickSummaryFromOutput` and is persisted as a 2000-character prefix plus ellipsis.

## Solution Assessment

Yes, if maintainers accept a configurable cap. Adding the option under the existing `cron.runLog` group is narrower than changing delivery semantics or storing duplicate full transcripts in the run log.
