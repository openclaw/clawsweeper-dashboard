## Summary

Keep open: current main and the latest release still hard-code a 2000-character cron run-log summary, and the requested fix needs a user-facing cap/default policy decision before implementation.

## What This Changes

Request to make cron run-log summary truncation configurable or much larger so long delivered agent outputs remain readable in cron runs history.

## Best Possible Solution

Define a bounded cron run-log summary cap contract, likely under `cron.runLog`, and wire it through config schema, docs/help, summary selection, persistence tests, and run-log pruning behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows any cleaned cron output longer than 2000 UTF-16 code units returns `truncateUtf16Safe(..., 2000)` plus ellipsis through `pickSummaryFromOutput` before the summary is persisted.

## Solution Assessment

Unclear until maintainers choose policy. Adding `cron.runLog.maxSummaryChars` is a narrow implementation path, but the default value and whether `summary` should serve as audit text are user-facing config decisions.
