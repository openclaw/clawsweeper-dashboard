## Summary

Keep open: current main and v2026.5.7 still persist only a 2000-code-unit cron summary, and fixing it requires a user-facing run-log cap/default decision rather than an already-implemented cleanup.

## What This Changes

Request to make cron run-log summary retention longer or configurable so long agent outputs remain readable in `openclaw cron runs`.

## Best Possible Solution

Add a bounded `cron.runLog.maxSummaryChars` option with schema/help/docs/tests and feed it into summary selection before JSONL persistence, while preserving delivery payloads and full `outputText`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any isolated cron final output longer than 2000 UTF-16 code units flows through `pickSummaryFromOutput` and is persisted as a 2000-character prefix plus ellipsis.

## Solution Assessment

Yes, if maintainers accept the configurable cap direction. Adding the option under the existing `cron.runLog` group is narrower than changing delivery semantics or storing duplicate full transcripts in the run log.
