## Summary

This PR should stay open but not merge as-is: it targets a valid abort/stale-output bug, yet the patch is stale against current runner APIs and only cleans after aborted output can already be delivered.

## What This Changes

The PR adds an embedded-runner abort-path helper that strips trailing aborted assistant messages from live session state and SessionManager persistence after prompt aborts.

## Best Possible Solution

Suppress aborted assistant output in the subscriber before stream or block delivery, then strip trailing aborted assistant entries from live state and persistence using `agent.state.messages`, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. #48241 gives rapid Telegram steps and timestamped logs, and current source shows an aborted assistant `message_end` can still pass through delivery before post-catch cleanup.

## Solution Assessment

No. The PR is aimed at the right bug, but the current patch is not the best fix because it uses a stale dependency API and cleans too late in the event flow.
