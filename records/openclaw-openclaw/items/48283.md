## Summary

Keep this PR open, but it needs changes before merge. The reported bug is valid and the PR is a useful source candidate, but the branch is stale against current main and the proposed cleanup happens after aborted assistant output can already be persisted or delivered.

## What This Changes

The PR adds an abort-path helper in the embedded runner to strip trailing assistant messages with `stopReason: "aborted"` from in-memory session state and SessionManager file entries.

## Best Possible Solution

Land a current-main fix that suppresses or strips aborted assistant output before subscription consumers, SessionManager persistence, and channel block replies can observe it, while reusing the existing tail-cleanup shape for session state and adding regression coverage for ordinary aborts.

## Reproduction Assessment

Yes. The linked issue gives concrete Telegram rapid-message steps and logs, and the source-level path can be reproduced by injecting an assistant `message_end` with `stopReason: "aborted"` and visible text into the embedded subscription/session path.

## Solution Assessment

No. The PR points in the right direction, but post-catch cleanup is too late for `message_end` delivery/persistence and the helper uses an Agent API that current main does not expose.
