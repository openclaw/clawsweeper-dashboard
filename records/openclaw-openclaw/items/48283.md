## Summary

Keep open. The linked bug is plausible and worth fixing, but this PR needs revision: it strips aborted assistant messages immediately on the abort catch path, while the runner can still have the original prompt/session abort settling, so the stale assistant entry can be appended after the strip has already run. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds abort-path cleanup that removes trailing assistant messages with `stopReason: "aborted"` from the in-memory session and session file before persistence.

## Best Possible Solution

Rebase onto current main, reuse or generalize the current sessions-yield stripping helper, wait for tracked prompt/session abort settlement before stripping, then add focused regression coverage and real runtime proof.

## Reproduction Assessment

No, not as a live current-main reproduction from this review. The linked issue has strong operational evidence and current source shows a plausible abort-settlement race, but I did not run a real Telegram/provider reproduction in the read-only review.

## Solution Assessment

No. The cleanup direction is right, but this branch strips too early in the abort path; the maintainable fix should wait for prompt/session abort settlement and fit the current split helper structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Wait for abort settlement before stripping:** `src/agents/pi-embedded-runner/run/attempt.ts:2616`
  - body: `promptActiveSession()` wraps `activeSession.prompt()` in `abortable(...)`, so this catch path can run as soon as the runner abort signal fires while the original prompt/session abort is still settling. Stripping the current tail here can miss the `stopReason: "aborted"` assistant that Pi appends during that later unwind, leaving the stale response in the transcript. Please wait for the tracked prompt/session abort settlement before stripping, as the sessions-yield path does, and cover that ordering with a regression test.
  - confidence: 0.84
