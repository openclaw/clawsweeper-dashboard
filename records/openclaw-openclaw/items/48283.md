## Summary

Keep open. The underlying abort-garbage bug remains plausibly valid, but this PR is not mergeable as-is because it strips aborted assistant messages before prompt/session abort settlement and still lacks after-fix real behavior proof.

## What This Changes

This PR adds a non-yield abort cleanup helper in the embedded Pi runner to remove trailing assistant messages with `stopReason: "aborted"` from memory and session file state.

## Best Possible Solution

Rebase and rework the PR to wait for tracked abort settlement before removing untrusted aborted assistant output, add focused regression coverage, and provide redacted real runtime proof.

## Reproduction Assessment

No, not as a high-confidence live current-main Telegram/provider reproduction. The linked report has strong operational evidence, and source inspection shows abort buffers plus abort-settlement timing paths that can preserve or broadcast nonblank aborted assistant text.

## Solution Assessment

No. The cleanup direction is plausible, but this patch strips too early in the abort path and lacks runtime proof; the maintainable fix should wait for abort settlement and avoid regressing legitimate abort partial retention.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Wait for abort settlement before stripping:** `src/agents/pi-embedded-runner/run/attempt.ts:2616`
  - body: `promptActiveSession()` wraps `activeSession.prompt()` in `abortable(...)`, so this catch path can run as soon as the runner abort signal fires while the original prompt/session abort is still settling. Stripping the current tail here can miss the aborted assistant message appended during that later unwind, leaving the stale response in the transcript; wait for the tracked abort settlement before stripping and cover that ordering with a regression test.
