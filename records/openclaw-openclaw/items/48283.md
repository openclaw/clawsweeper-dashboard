## Summary

Keep open: the linked bug is plausible, but this PR does not fix the user-visible gateway buffer path and lacks after-fix real behavior proof. Current docs and tests intentionally retain aborted partials, so the final behavior needs maintainer review rather than merging this narrow runner cleanup as-is.

## What This Changes

The PR adds embedded-runner abort cleanup that strips trailing assistant messages with `stopReason: "aborted"` from live session state and session-manager file entries.

## Best Possible Solution

Replace or revise this PR with a gateway-level abort buffer gate that prevents stale or invalid aborted content from being broadcast or persisted, while preserving or explicitly changing the documented partial-retention contract.

## Reproduction Assessment

Yes for source-level reproduction: current main tests seed `chatRunBuffers` with nonblank text and assert that abort broadcasts or persists assistant content. I did not live-reproduce the exact Telegram/sub2api runtime race in this read-only review.

## Solution Assessment

No. Cleaning the embedded runner session tail after `activeSession.prompt()` aborts is too late and too narrow for the gateway path that broadcasts and persists buffered abort text.
