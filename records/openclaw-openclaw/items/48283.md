## Summary

Keep open: the PR targets a real, source-reproducible abort bug, but this branch is not mergeable as-is because it uses a stale Pi Agent transcript API, cleans up after user-visible delivery can already happen, and lacks required real Telegram behavior proof.

## What This Changes

Adds a post-prompt abort cleanup helper in the embedded runner that removes trailing `stopReason: "aborted"` assistant messages from in-memory and SessionManager transcript state.

## Best Possible Solution

Gate `stopReason: "aborted"` assistant output before subscriber stream or block delivery, then strip trailing aborted transcript entries through `agent.state.messages` and SessionManager persistence with focused regression coverage and Telegram proof.

## Reproduction Assessment

Yes, at source level. The linked report provides rapid Telegram steps and timestamped logs, and current main shows aborted assistant `message_end` content can still pass through subscriber delivery before this PR's post-catch cleanup would run.

## Solution Assessment

No. The PR points at the right bug, but the implementation uses a stale dependency API and runs after visible delivery can already happen; the narrow maintainable fix should suppress aborted output in the subscriber path and use current transcript replacement APIs.
