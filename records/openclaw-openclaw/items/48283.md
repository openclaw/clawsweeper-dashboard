## Summary

Keep open. The linked bug still appears valid on current main, but this PR should not merge as-is because it uses a stale Agent API shape and only cleans up after the prompt abort path, while current message_end handling can still deliver aborted assistant text to subscribers and channel replies first.

## What This Changes

The PR adds an abort-path helper in the embedded runner to strip trailing assistant messages with `stopReason: "aborted"` from in-memory session state and SessionManager file entries.

## Best Possible Solution

Land a current-main fix that suppresses aborted assistant output before `message_end` subscribers or block replies can observe it, then strips any aborted assistant tail from live Agent state and SessionManager persistence using the current `state.messages` shape, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. The linked issue gives concrete Telegram rapid-message steps and logs, and a focused source-level reproduction can inject an assistant `message_end` with `stopReason: "aborted"` and visible text into the embedded subscription path.

## Solution Assessment

No. The PR is pointed at the right bug, but the current diff is not the best fix because it uses a stale `replaceMessages` API and runs after visible `message_end` handling can already emit or persist the bad assistant content.
