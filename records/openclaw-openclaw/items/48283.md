## Summary

Keep open: the PR targets a real abort/stale-content bug, but the patch only trims the embedded runner session tail after prompt abort while current gateway code can still broadcast and persist abort buffer text, and the external PR still lacks required real behavior proof.

## What This Changes

The PR adds `stripAbortedAssistantMessages()` to `src/agents/pi-embedded-runner/run/attempt.ts` and calls it when `activeSession.prompt()` throws a runner abort.

## Best Possible Solution

Gate or validate abort buffers before user-visible delivery and transcript persistence, while preserving documented legitimate partials or updating the abort partial-retention contract with tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: current main can be driven with a nonblank `chatRunBuffers` entry and an abort to broadcast and persist that text. I did not live-reproduce the exact Telegram/sub2api race from the report.

## Solution Assessment

No. The proposed cleanup runs after embedded prompt abort and does not gate the gateway buffer path that delivers and persists abort text, so it is too narrow for the reported user-visible stale content.
