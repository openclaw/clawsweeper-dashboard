## Summary

Keep open. The PR targets a real abort-output bug, but the current patch strips aborted assistant messages before prompt/session abort settlement can finish and it still lacks required real behavior proof; current main only has adjacent abort-settle cleanup, not this non-yield stale-output gate.

## What This Changes

The PR adds a non-yield abort cleanup helper in the embedded Pi runner to remove trailing assistant messages with `stopReason: "aborted"` from memory and session file state on abort.

## Best Possible Solution

Rework the patch to wait for tracked abort settlement before removing untrusted aborted assistant output, add focused regression coverage, and include redacted real runtime proof.

## Reproduction Assessment

No, not as a high-confidence live current-main Telegram/provider reproduction. Source inspection shows abort buffers plus abort-settlement timing paths that can preserve or broadcast nonblank aborted assistant text, matching the reported failure class.

## Solution Assessment

No. The cleanup direction is plausible, but this patch strips too early in the abort path and needs to preserve or explicitly revise the existing legitimate abort-partial contract.

## Review Findings
