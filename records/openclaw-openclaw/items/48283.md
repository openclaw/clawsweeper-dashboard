## Summary

Keep open: the PR targets a real abort-output bug, but the branch is not merge-ready because it uses a stale transcript replacement API, cleans up after aborted output can already be delivered, and has no after-fix real behavior proof.

## What This Changes

The PR adds an embedded-runner abort-path helper that strips trailing aborted assistant messages from in-memory and SessionManager transcript state after `activeSession.prompt` aborts.

## Best Possible Solution

Gate `stopReason: "aborted"` assistant output before subscriber stream or block delivery, then strip trailing aborted transcript entries through current `agent.state.messages` and SessionManager persistence with focused regression coverage and Telegram proof.

## Reproduction Assessment

Yes at source level. The linked report gives rapid Telegram steps and timestamped logs, and current source shows `handleMessageEnd` can emit aborted assistant content before this PR's post-catch cleanup would run.

## Solution Assessment

No. The PR is aimed at the right bug, but the patch uses a stale dependency API and cleans too late; the safer repair suppresses aborted output in the subscriber path and uses `agent.state.messages` for transcript cleanup.
