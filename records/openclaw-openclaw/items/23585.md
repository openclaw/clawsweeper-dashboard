## Summary

Keep open. Current main and v2026.6.10 still reject trusted-proxy requests when the configured identity header is absent; related merged work covers local-direct password and loopback proxy policy, not a headerless default identity, and accepting one needs maintainer/security approval.

## What This Changes

The issue requests a trusted-proxy default or fallback user for trusted proxy requests whose configured identity header is absent on Cloudflare WebSocket upgrades.

## Best Possible Solution

Keep trusted-proxy identity validation fail-closed unless maintainers approve a narrow single-principal fallback mode with explicit config, source/loopback/header rules, allowlist semantics, audit/docs updates, and regression coverage.

## Reproduction Assessment

Yes for the OpenClaw source path: current main deterministically rejects a trusted-proxy request without the configured identity header, and a focused test asserts trusted_proxy_user_missing. I did not live-reproduce Cloudflare's WebSocket header stripping.

## Solution Assessment

Unclear pending security policy. The requested defaultUser/fallbackUser is mechanically plausible, but it changes the trusted-proxy trust model; safer defaults are fail-closed identity headers, a documented alternate auth path, or a maintainer-approved single-principal mode.
