## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, and the requested fallback/default user would add a new security-sensitive gateway auth mode rather than fix a broken existing contract.

## What This Changes

Add a trusted-proxy fallback/default user so requests from a trusted proxy can authenticate when the configured identity header is missing.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, add an explicit fallback identity contract across config types, schemas, docs, audit guidance, and gateway/control-ui tests, and if rejected document Cloudflare no-identity WebSocket setups as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show a trusted-proxy request from a trusted source without the configured user header returns trusted_proxy_user_missing. The Cloudflare-specific WebSocket header-loss claim was not live-verified in this review.

## Solution Assessment

Unclear. A defaultUser/fallbackUser option could solve the reported deployment, but it is not automatically the safest fix because it would authenticate requests with no identity header at the gateway boundary.
