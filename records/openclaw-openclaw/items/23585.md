## Summary

Current main still fails closed when trusted-proxy identity is absent, and the config/docs/tests do not expose a fallback/default identity contract. This should stay open because the requested behavior is a new security-sensitive auth path and config option that needs maintainer/security policy, not conservative cleanup.

## What This Changes

The issue requests a trusted-proxy fallback/default user when the configured identity header is absent on WebSocket requests behind Cloudflare Zero Trust.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, define the fallback identity contract across auth logic, exported config types, strict schema, docs, audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified in this read-only review.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth path.
