## Summary

Keep open. Current main still fails closed when a trusted-proxy request lacks the configured identity header, and neither the config contract nor docs expose a default or fallback identity option. The report remains distinct from the closed local-direct password fallback work, and the requested behavior changes a security-sensitive gateway auth boundary that needs maintainer/security review.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or `fallbackUser` option so requests from a configured trusted proxy can authenticate when the identity header is absent.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity contract across exported config types, runtime/generated schemas, docs, security audit guidance, and gateway/control-ui tests; if rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point operators to an auth or proxy path that reliably forwards identity headers.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: a trusted-proxy request from a trusted source with no configured user header reaches `trusted_proxy_user_missing`, and the current unit test asserts that rejection. The Cloudflare-specific WebSocket header-loss behavior was not live-reproduced in this read-only review.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could solve the reported deployment, but it is not automatically the safest fix because it would authorize no-identity requests at the gateway boundary; the maintainable path needs explicit security semantics or a documented unsupported configuration.
