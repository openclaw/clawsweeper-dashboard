## Summary

Keep open. Current main and the latest release still fail closed when the trusted-proxy identity header is absent, and the requested fallback user would add a new no-identity auth success path that needs gateway/security policy approval.

## What This Changes

Requests an explicit trusted-proxy default or fallback user config so trusted proxy requests can authenticate when the configured identity header is missing.

## Best Possible Solution

Keep this open for gateway/security policy review; if accepted, add an explicit fallback identity contract across auth logic, config types/schema, docs, audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket proxy paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes, for the OpenClaw-side behavior at source level: current main returns `trusted_proxy_user_missing` when a trusted-proxy request lacks the configured user header. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not yet clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
