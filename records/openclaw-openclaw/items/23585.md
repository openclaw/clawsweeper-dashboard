## Summary

Keep open: current main and the latest release still reject trusted-proxy requests when the configured identity header is absent, and the merged local-direct password fallback does not solve this Cloudflare/WebSocket proxy-identity case. Adding a fallback/default identity would be a new security-sensitive auth/config behavior that needs gateway/security maintainer approval.

## What This Changes

The issue requests a trusted-proxy fallback/default user for requests from a trusted proxy when the configured identity header is absent, especially Cloudflare WebSocket upgrade paths.

## Best Possible Solution

Keep this open for gateway/security owner review; either add an explicit, audited trusted-proxy fallback identity option across config, schema, docs, security audit, and auth tests, or document Cloudflare/IP-whitelist/no-identity WebSocket setups as unsupported and point users to a proxy path that reliably forwards identity headers or a different auth mode.

## Reproduction Assessment

Yes, source-level: current main still returns trusted_proxy_user_missing when the trusted-proxy identity header is absent, and current config/schema/docs expose no fallback identity option. I did not run a live Cloudflare Tunnel because the OpenClaw-side behavior is directly visible in source and tests.

## Solution Assessment

Unclear. A default/fallback user may be the right product answer for this deployment shape, but it is not a narrow bug fix; it adds new auth semantics and needs explicit gateway/security maintainer approval before implementation.
