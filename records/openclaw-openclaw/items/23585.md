## Summary

Keep open: current main and v2026.6.5 still fail closed when the trusted-proxy identity header is absent, and adding a fallback user would be a new security-sensitive auth/config contract that needs maintainer policy approval.

## What This Changes

The issue requests an explicit trusted-proxy fallback/default user when the configured identity header is absent on WebSocket requests from a trusted proxy.

## Best Possible Solution

Keep strict per-request identity as the default; if maintainers accept this deployment mode, add an explicitly high-risk fallback option with mandatory allowlisting, clear session/audit semantics, security-audit warnings, docs, and focused auth/control-ui tests.

## Reproduction Assessment

Yes from source: a trusted-proxy request without the configured identity header deterministically returns `trusted_proxy_user_missing`, matching the reported gateway failure. The Cloudflare-specific WebSocket header omission was not independently live-reproduced.

## Solution Assessment

Unclear pending maintainer/security policy: the proposed option directly addresses the deployment, but reliable WebSocket identity forwarding or another auth mode preserves stronger per-request attribution and remains safer by default.
