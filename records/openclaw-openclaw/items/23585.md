## Summary

Keep open. Current main still fails closed when a trusted-proxy request lacks the configured identity header, and neither the config contract nor the docs expose a default or fallback user option. The discussion already separates this Cloudflare/WebSocket missing-header case from the closed local-direct fallback work, and the requested behavior changes a security-sensitive gateway auth boundary that needs maintainer/security review.

## What This Changes

The issue requests a `defaultUser` or `fallbackUser` trusted-proxy auth option so requests from a trusted proxy can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, add an explicit fallback identity contract across config types, schemas, docs, audit guidance, and gateway/control-ui tests, and if rejected document Cloudflare no-identity WebSocket setups as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: a trusted-proxy request from a trusted source with no configured user header returns `trusted_proxy_user_missing`, and `src/gateway/auth.test.ts:720` asserts that rejection. The Cloudflare-specific WebSocket header-loss behavior was not live-reproduced here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could solve the reported deployment, but it is not automatically the safest fix because it would authorize no-identity requests at the gateway boundary.
