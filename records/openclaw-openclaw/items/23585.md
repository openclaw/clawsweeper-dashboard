## Summary

Keep open. Current main still fails closed when a trusted-proxy request lacks the configured identity header, and accepting a fallback identity would expand a security-sensitive gateway auth boundary rather than clean up an already-fixed bug.

## What This Changes

Request support for a `defaultUser`/`fallbackUser` trusted-proxy setting that authenticates trusted-proxy requests when the configured identity header is absent.

## Best Possible Solution

Decide the auth policy explicitly; either add a documented fallback identity contract across config, schemas, docs, audit guidance, and tests, or document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer auth alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare-specific WebSocket header-loss claim was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser`/`fallbackUser` option could solve the reported deployment, but it is not yet clearly the safest maintainable fix because it would authenticate no-identity requests at the gateway boundary.
