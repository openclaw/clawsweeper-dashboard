## Summary

Current main and v2026.6.6 still deliberately reject trusted-proxy requests without the configured identity header. The requested fallback remains a distinct, unresolved security and product-policy choice rather than an already-fixed bug or duplicate.

## What This Changes

Add an optional trusted-proxy fallback identity for requests from a trusted proxy when the configured user header is absent.

## Best Possible Solution

Keep fail-closed identity validation unless maintainers approve an explicit single-principal proxy mode with narrow source requirements, allowlist semantics, security-audit warnings, documentation, and fresh-install and upgrade coverage.

## Reproduction Assessment

Yes for the OpenClaw path: current main deterministically rejects a trusted-proxy request without the configured identity header. The reported Cloudflare WebSocket header omission remains externally supplied evidence rather than an independently reproduced environment.

## Solution Assessment

Unclear pending security policy: the proposed fallback is mechanically direct, but it changes the trust model; preserving fail-closed identity headers or using another supported authentication path remains safer by default.
