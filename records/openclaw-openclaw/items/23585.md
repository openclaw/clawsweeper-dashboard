## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is missing, and accepting a fallback/default identity would add a new security-sensitive auth path rather than fix broken documented behavior.

## What This Changes

The issue requests a `defaultUser` or `fallbackUser` option for trusted-proxy auth when the configured identity header is absent on WebSocket requests.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, define an explicit fallback identity contract across auth logic, exported config types, strict schema, docs, security audit guidance, and gateway auth tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth path.
