## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is absent, and adding a fallback/default user would change a security-sensitive gateway auth boundary rather than fix broken documented behavior; the related local/internal fallback work at https://github.com/openclaw/openclaw/issues/17761 and https://github.com/openclaw/openclaw/issues/78684 does not implement this request, and https://github.com/openclaw/openclaw/issues/69066 tracks a different service-identity design.

## What This Changes

The issue requests a `defaultUser` or `fallbackUser` trusted-proxy config option so trusted proxy requests can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep this open for maintainer/secops policy review; if accepted, add an explicit fallback identity contract across config types, strict schema, docs, security audit guidance, and trusted-proxy auth tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified in this review.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could address the reported deployment, but it is not clearly the safest maintainable solution because it would authenticate requests that carry no user identity header at the gateway boundary.
