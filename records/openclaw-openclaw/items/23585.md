## Summary

Current main still has no trusted-proxy default/fallback identity; source, strict config schema, docs, and tests all preserve the identity-header-required behavior. This should stay open because the requested fallback adds a new security-sensitive auth path and config contract, not a narrow documented bug fix.

## What This Changes

Add a trusted-proxy `defaultUser` or `fallbackUser` option so requests from trusted proxies can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, define the fallback identity contract across auth logic, exported config types, strict schema, docs, audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth path.
