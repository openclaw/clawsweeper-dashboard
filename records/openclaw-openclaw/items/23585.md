## Summary

Keep open. Current main and the latest release still reject trusted-proxy requests when the configured identity header is absent, and adding a fixed fallback identity remains a security-sensitive auth/config feature that needs maintainer approval.

## What This Changes

The issue requests a trusted-proxy fallback/default identity for requests from a trusted proxy when the configured identity header is absent, especially Cloudflare Tunnel WebSocket upgrades.

## Best Possible Solution

Keep fail-closed identity validation unless maintainers approve an explicit single-principal trusted-proxy mode with narrow source, loopback, allowlist, audit, docs, and regression-test coverage; if rejected, document Cloudflare no-identity WebSocket setups as unsupported.

## Reproduction Assessment

Yes for the OpenClaw path: current main deterministically rejects a trusted-proxy request without the configured identity header. The Cloudflare WebSocket header loss remains reporter-supplied evidence rather than a live reproduction from this review.

## Solution Assessment

Unclear pending security policy. The requested fallback is mechanically direct, but it changes the trusted-proxy trust model; fail-closed identity headers or a documented alternate auth path remain safer defaults unless maintainers explicitly accept a single-principal mode.
