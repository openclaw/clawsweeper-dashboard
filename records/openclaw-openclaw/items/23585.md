## Summary

Keep open: current main and the latest release still fail closed when the trusted-proxy identity header is absent, while the requested fallback identity would add a new security-sensitive config and auth success path that needs maintainer/security policy first.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or `fallbackUser` config that authenticates trusted-proxy requests when the configured identity header is absent.

## Best Possible Solution

Keep this open for gateway/security policy review; if accepted, define an explicit fallback identity contract across auth logic, config types, strict schema, docs, audit warnings, and regression tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket proxy paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source, tests, docs, and `v2026.5.12` show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not yet clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
