## Summary

Keep open: current main and the latest release still fail closed with `trusted_proxy_user_missing` when the configured trusted-proxy identity header is absent, and adding a fallback/default user would create a new security-sensitive auth success path.

## What This Changes

The issue requests an explicit trusted-proxy fallback/default identity for requests from configured trusted proxies when the identity header is absent.

## Best Possible Solution

Keep this open for gateway/security policy review; if accepted, define an explicit fallback identity contract across auth logic, config types, strict schema, docs, audit warnings, and regression tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket proxy paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes, for the OpenClaw-side behavior at source level: current main returns `trusted_proxy_user_missing` when a trusted-proxy request lacks the configured user header. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not yet clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
