## Summary

Keep open. Current main and the latest shipped release still fail closed when the trusted-proxy identity header is absent, and adding a fallback/default identity would create a new security-sensitive auth path that needs maintainer/security policy rather than conservative cleanup.

## What This Changes

Add a trusted-proxy default or fallback identity to use when the configured user identity header is absent.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, define the fallback identity contract across auth logic, config types, strict schema, docs, audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth path.
