## Summary

Keep open. Current main still fails closed when trusted-proxy auth lacks the configured identity header, and accepting a fallback/default identity would change the gateway auth trust boundary enough to require maintainer/security approval first.

## What This Changes

Request to add a trusted-proxy fallback/default user so trusted proxy requests can authenticate when the identity header is absent.

## Best Possible Solution

Keep this open for gateway/security policy review; if accepted, define an explicit fallback identity contract across auth logic, config types/schema, docs, audit warnings, and regression tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket proxy paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes, for the OpenClaw-side behavior at source level: current main returns trusted_proxy_user_missing when a trusted-proxy request lacks the configured user header. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A defaultUser or fallbackUser option could unblock the reported deployment, but it is not yet clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
