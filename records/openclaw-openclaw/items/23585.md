## Summary

Keep open. Current main still fail-closes trusted-proxy auth when the configured identity header is absent, while the requested fallback user would add a new security-sensitive auth/config behavior rather than fix a broken documented contract.

## What This Changes

Request to add a trusted-proxy default or fallback user config so trusted-proxy requests without the identity header can authenticate as a configured user.

## Best Possible Solution

Keep this open for maintainer/secops policy review; if accepted, add an explicit fallback identity contract across config types, schema, docs, audit guidance, and auth tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return trusted_proxy_user_missing. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A defaultUser or fallbackUser option could unblock the reported deployment, but it is not clearly the safest maintainable fix because it would create a no-identity trusted-proxy auth path.
