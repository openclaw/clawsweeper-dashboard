## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is absent, and the requested fallback identity is a security-sensitive new auth success path that needs maintainer/security policy approval rather than cleanup closure.

## What This Changes

The issue asks for a trusted-proxy `defaultUser` or `fallbackUser` config option so requests from trusted proxies can authenticate when the configured identity header is missing.

## Best Possible Solution

Keep this issue open for gateway/security policy review; if accepted, add an explicit fallback identity contract across auth logic, config types/schema, generated schema/docs, audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket proxy paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes, for the OpenClaw-side behavior at source level: current main returns `trusted_proxy_user_missing` when a trusted-proxy request lacks the configured user header. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not yet clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
