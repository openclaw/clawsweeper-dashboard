## Summary

Current main and the latest release still fail closed when a trusted-proxy request lacks the configured identity header, and no config/schema/docs contract exposes a fallback user. This should stay open because accepting no-identity trusted-proxy requests would be a new security-sensitive auth path that needs maintainer/security policy before implementation.

## What This Changes

The issue requests a trusted-proxy fallback/default user when the configured identity header is absent, mainly for Cloudflare Zero Trust WebSocket paths.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, define the fallback identity contract across auth logic, exported config types, strict schema, generated docs/schema, security audit guidance, and tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source, tests, docs, and v2026.5.12 release inspection show trusted-proxy requests missing the configured user header return trusted_proxy_user_missing. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A defaultUser or fallbackUser option could unblock the reported deployment, but it is not clearly the safest maintainable solution because it creates a no-identity trusted-proxy auth success path.
