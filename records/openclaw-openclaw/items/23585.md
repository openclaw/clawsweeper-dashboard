## Summary

Keep open. Current main and the latest release still fail closed when a trusted-proxy request lacks the configured identity header, and adding a default identity would change a security-sensitive Gateway auth boundary that needs maintainer/security approval.

## What This Changes

The issue requests a trusted-proxy default or fallback user so headerless requests from trusted proxies can authenticate as that configured user.

## Best Possible Solution

Keep fail-closed identity validation unless maintainers approve a narrow single-principal trusted-proxy mode with explicit source, loopback, allowlist, audit, docs, and regression-test coverage.

## Reproduction Assessment

Yes for the OpenClaw path: current main deterministically rejects a trusted-proxy request without the configured identity header. The Cloudflare WebSocket header-loss claim remains reporter-supplied rather than live-reproduced in this read-only review.

## Solution Assessment

Unclear pending security policy. The requested fallback is mechanically direct, but it changes the trusted-proxy trust model; fail-closed identity headers or a documented alternate auth path remain safer defaults unless maintainers approve a single-principal mode.
