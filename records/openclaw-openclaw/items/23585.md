## Summary

Keep open. Current main and the latest release still reject trusted-proxy requests when the configured identity header is absent, and accepting a fallback identity would add a security-sensitive Gateway auth/config surface that needs maintainer, product, and security approval.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or fallback identity option for trusted proxy requests, especially Cloudflare WebSocket upgrade paths, when the identity header is not forwarded.

## Best Possible Solution

Keep fail-closed identity validation unless maintainers approve a narrow explicit fallback identity mode with source, loopback, required-header, allowlist, audit, docs, and regression-test coverage; if rejected, document Cloudflare IP-whitelist/no-identity WebSocket setups as unsupported and point users to a proxy path that forwards identity headers or to shared-secret/Tailscale alternatives.

## Reproduction Assessment

Yes for the OpenClaw source path: current main deterministically rejects a trusted-proxy request without the configured identity header as `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss claim remains reporter-supplied rather than live-reproduced in this read-only review.

## Solution Assessment

Unclear pending security policy. The requested fallback is mechanically direct, but it changes the trusted-proxy trust model; fail-closed identity headers or a documented alternate auth path remain safer defaults unless maintainers approve a single-principal mode.
