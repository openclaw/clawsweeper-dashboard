## Summary

Current main and v2026.6.5 still reject trusted-proxy requests when the configured identity header is missing; the requested fallback remains a security-sensitive new authentication path requiring explicit maintainer policy approval.

## What This Changes

Add an optional configured identity used when a request comes from a trusted proxy but lacks the trusted-proxy user header.

## Best Possible Solution

Keep strict per-request identity as the default; if maintainers accept this deployment mode, add an explicitly high-risk fallback option with mandatory allowlisting, precise session/audit identity semantics, security-audit warnings, and focused fresh-install and upgrade documentation.

## Reproduction Assessment

Yes from source: a trusted-proxy request without the configured identity header deterministically returns `trusted_proxy_user_missing`, matching the reported gateway failure; the Cloudflare-specific header omission was not independently reproduced.

## Solution Assessment

Unclear pending maintainer policy: the proposed option directly addresses the deployment, but reliable WebSocket identity forwarding or another authentication mode preserves stronger per-request attribution and remains safer by default.
