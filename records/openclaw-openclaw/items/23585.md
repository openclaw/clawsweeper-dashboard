## Summary

Keep open. Current main still requires a trusted-proxy identity header and fails closed when it is absent; adding a default/fallback user would be a new security-sensitive auth path that needs maintainer/secops policy approval.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or `fallbackUser` config option for requests from trusted proxies when the configured identity header is missing.

## Best Possible Solution

Keep this open for maintainer/secops policy review; if accepted, add an explicit fallback identity contract across config types, schema, docs, audit guidance, and auth tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified in this read-only review.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable fix because it would create a no-identity trusted-proxy auth path.
