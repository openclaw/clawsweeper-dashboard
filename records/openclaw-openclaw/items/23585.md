## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is absent, and the config, schema, docs, and tests do not expose a default or fallback user option; the requested change would alter a security-sensitive gateway auth boundary.

## What This Changes

Add an explicit trusted-proxy default or fallback user for trusted proxy requests that arrive without the configured identity header, especially Cloudflare WebSocket upgrade paths.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, add an explicit fallback identity contract across config types, zod/generated schemas, docs, audit guidance, and gateway/control-ui tests, and if rejected document Cloudflare no-identity WebSocket setups as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: a trusted-proxy request from a trusted source without the configured user header returns trusted_proxy_user_missing, and the current auth test asserts that rejection. The Cloudflare edge-to-origin WebSocket header loss was not live-verified here.

## Solution Assessment

Unclear. A defaultUser or fallbackUser option could solve the reported deployment, but it is not automatically the safest fix because it would authenticate no-identity requests at the gateway boundary.
