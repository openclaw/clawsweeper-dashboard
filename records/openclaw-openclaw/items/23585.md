## Summary

Keep open. Current main still fail-closes trusted-proxy auth when the configured identity header is absent, and adding a fallback/default identity would be a new security-sensitive auth path rather than a fix to the documented contract.

## What This Changes

Request support for a trusted-proxy `defaultUser` or `fallbackUser` config value that authenticates requests from trusted proxies when the configured identity header is missing.

## Best Possible Solution

Keep this open for maintainer/security review; if accepted, add an explicit fallback identity contract across config types, strict schema, docs, audit guidance, and gateway auth tests, and if rejected, document no-identity Cloudflare/IP-whitelist WebSocket paths as unsupported with safer alternatives.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests show trusted-proxy requests missing the configured user header return `trusted_proxy_user_missing`. The Cloudflare WebSocket header-loss behavior itself was not live-verified here.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` option could unblock the reported deployment, but it is not clearly the safest maintainable fix because it would create a no-identity trusted-proxy auth path.
