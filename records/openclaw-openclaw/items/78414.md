## Summary

Keep open: the PR targets a real Gateway reload bug and includes convincing logs, but the current patch changes documented `hot` reload behavior by queueing an automatic restart before the existing hot-mode warning path.

## What This Changes

The PR adds Gateway config-reload detection and tests for enabling an unloaded channel plugin, plus a changelog entry.

## Best Possible Solution

Revise the PR to make unloaded-channel activation observable and recoverable through the existing reload-mode policy, or explicitly update docs and tests if maintainers choose a hot-mode exception.

## Reproduction Assessment

Yes for the review finding: PR head queues a restart before the hot-mode warning branch, while current main and docs define hot mode as warning-only with operator-managed restart. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair for the linked wedge, but it needs either to route through the existing hot-mode contract or receive an explicit maintainer decision to change that contract.
