## Summary

Keep open: this targets a real Gateway/WhatsApp activation wedge and the PR body now supplies after-fix log proof, but the patch auto-restarts even when `gateway.reload.mode` is `"hot"`, bypassing the documented hot-mode contract that restart-required changes warn and leave restart to the operator.

## What This Changes

The PR adds a Gateway config reload guard, regression tests, and a changelog entry to restart when a live config edit enables an unloaded channel plugin.

## Best Possible Solution

Land a version that makes unloaded channel activation observable and recoverable while preserving the documented hot/manual behavior unless maintainers explicitly update the reload-mode contract.

## Reproduction Assessment

Yes, at source level: current main has a clear reload path where restart-required config changes in hot mode warn and return without rebuilding or restarting the plugin registry, and the linked production report supplies WhatsApp activation wedge evidence. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair for hybrid/restart behavior, but calling `queueRestart` before the hot-mode guard changes the documented hot-mode semantics without an explicit contract update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

  - body: The new unloaded-channel guard calls `queueRestart(...)` even when `gateway.reload.mode` is `"hot"`.
