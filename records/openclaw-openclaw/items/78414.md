## Summary

Keep open: the PR targets a real Gateway/WhatsApp activation wedge and includes convincing after-fix logs, but the current diff auto-restarts even when `gateway.reload.mode` is `hot`, which conflicts with the documented manual-restart contract for hot mode.

## What This Changes

The PR adds Gateway config-reload detection for unloaded channel-plugin enable transitions, queues a restart, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Land a revision that makes unloaded channel activation observable and recoverable while preserving the hot-mode contract unless maintainers explicitly update that contract and its docs/tests.

## Reproduction Assessment

Yes for the review finding: PR head queues a restart before the hot-mode guard, while current main and docs define hot mode as warning-only for restart-required changes. I did not run a live current-main Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair, but this implementation needs either to route through the existing hot-mode warning path or receive an explicit maintainer decision to change the reload-mode contract.
