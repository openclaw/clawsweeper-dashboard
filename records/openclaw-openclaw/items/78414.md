## Summary

Keep open: the PR targets a real Gateway/WhatsApp reload wedge and includes convincing after-fix logs, but the diff still changes `gateway.reload.mode="hot"` from warning-only/operator-managed restart to automatic restart for this activation case. That semantic change needs maintainer agreement before merge.

## What This Changes

This PR adds Gateway config-reload logic and tests to restart when a live config edit enables an unloaded channel plugin, plus a changelog entry.

## Best Possible Solution

Make unloaded-channel activation observable and recoverable through the existing reload-mode policy, or explicitly update the docs and tests if maintainers choose a hot-mode exception.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows `queueRestart(...)` runs before the `settings.mode === "hot"` warning path, while current main and docs define hot mode as operator-managed for restart-required changes. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair for the linked wedge, but the implementation should either preserve the reload-mode contract or come with an explicit maintainer-approved contract change.
