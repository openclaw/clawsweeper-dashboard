## Summary

Keep open: this PR is a plausible fix with sufficient real behavior proof, but the current diff still changes the documented `gateway.reload.mode: "hot"` contract by auto-restarting before the existing hot-mode warning branch can apply.

## What This Changes

Adds Gateway config-reload detection for false/missing-to-true channel enable transitions when the channel plugin is not loaded, queues a restart, adds focused tests, and updates the changelog.

## Best Possible Solution

Revise the PR to preserve the existing reload-mode contract, or make an explicit maintainer-approved hot-mode exception with matching docs and tests before landing it with the linked issue.

## Reproduction Assessment

Yes for the review finding: PR head queues a restart before the hot-mode warning branch, while current main and docs define hot mode as warning-only with operator-managed restart. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair, but this implementation needs either to route through the existing hot-mode policy or receive an explicit maintainer decision to change that contract.
