## Summary

Keep open: the PR addresses a real Gateway/WhatsApp activation wedge, but as written it changes `gateway.reload.mode="hot"` from warning/manual restart to automatic restart for this case, and current main does not yet contain the guard.

## What This Changes

The branch adds an unloaded-channel activation detector in Gateway config reload, focused regression tests, and a changelog entry.

## Best Possible Solution

Land a revision that makes unloaded channel activation observable and recoverable while preserving the hot-mode contract unless maintainers explicitly update that contract and its docs/tests.

## Reproduction Assessment

Yes for the review finding: source inspection shows PR head queues a restart in `hot` mode before the warning-only branch, while current docs and current main define hot mode as operator-managed for restart-needed changes. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair, but this implementation needs either to preserve the existing hot-mode warning path or receive an explicit maintainer decision to change that reload-mode contract.
