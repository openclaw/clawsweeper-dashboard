## Summary

Keep open: this PR targets a source-backed gateway reload bug and includes focused tests/changelog, but it still lacks after-fix real behavior proof and the diff appears to bypass the documented `gateway.reload.mode="hot"` no-auto-restart contract.

## What This Changes

The PR adds a config-reload guard, regression tests, and a changelog entry to restart the Gateway when `channels.<id>.enabled` flips to true for an unloaded channel plugin.

## Best Possible Solution

Land a version that preserves the intended reload-mode contract, keeps the regression coverage and changelog, and includes real post-fix Gateway proof showing the restart and subsequent channel plugin load.

## Reproduction Assessment

Yes, at source level: current main has a clear reload path where restart-required changes in `gateway.reload.mode="hot"` warn and return, while the related report supplies production evidence for the WhatsApp activation wedge. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow fix, but this implementation queues a restart before preserving hot-mode manual-restart behavior; the safer path is to honor that contract or get an explicit maintainer product decision plus live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve hot-mode no-auto-restart behavior:** `src/gateway/config-reload.ts:329`
  - body: The new unloaded-channel guard calls `queueRestart(...)` even when `gateway.reload.mode` is `"hot"`. That mode is documented and currently implemented as “warn when restart is needed; the operator handles it,” so deployments that selected hot mode to avoid automatic restarts will now restart on a `channels.<id>.enabled` flip. Please either keep the existing hot-mode warning/manual path for this guard or update the product contract deliberately.
  - confidence: 0.82
