## Summary

Keep open: the branch still makes `gateway.reload.mode: "hot"` auto-restart for unloaded-channel activation, which conflicts with the current warning/manual-restart contract unless maintainers explicitly choose this exception.

## What This Changes

The PR adds Gateway config-reload detection for unloaded channel enable transitions, focused regression tests, and a changelog entry.

## Best Possible Solution

Revise the PR to make unloaded-channel activation observable and recoverable through the existing reload-mode policy, or explicitly update docs/tests if maintainers choose a hot-mode exception.

## Reproduction Assessment

Yes for the review finding: PR head queues a restart before the hot-mode warning branch, while current main and docs define hot mode as warning-only with operator-managed restart. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No as written. Restart-on-activation is a plausible narrow repair, but this implementation needs either to route through the existing hot-mode warning path or receive an explicit maintainer decision to change that contract.
