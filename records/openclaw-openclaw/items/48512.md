## Summary

This should stay open because current main does not yet have the pending-tool-call idle-flush guard, but this PR is not mergeable as-is: the branch mixes the intended runner fix with a very large unrelated generated tree and other behavior changes, and the core flush loop still has correctness issues.

## What This Changes

The PR attempts to add a `hasPendingToolCalls` hint to pi-embedded idle cleanup while also carrying broad unrelated generated artifacts, hooks, UI, session, gateway, and delivery changes.

## Best Possible Solution

Replace this branch with a narrow runner-only fix that bridges `Agent.state.pendingToolCalls` into every relevant idle-flush cleanup path, preserves one total timeout budget, treats the probe as best-effort, and adds focused regression tests without unrelated artifacts or feature changes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a credible current-main path: cleanup waits for idle once, then flushes pending tool results, while pi-agent-core tracks in-flight tool calls separately in `state.pendingToolCalls`; I did not run a deterministic live overloaded-error repro.

## Solution Assessment

Is this the best way to solve the issue? No. The adapter direction is plausible, but the branch is too broad and the proposed loop still mishandles timeout accounting and best-effort cleanup safety; a narrow replacement is the maintainable path.

- **[P1] Remove the generated dist.bak payload:** `dist.bak/account-lookup-BCxEEOS8.js:1`
  - body: The PR adds a large generated `dist.bak` tree and then ignores that directory in `.gitignore`. That mixes unreviewed build output into an otherwise narrow runner fix and masks future changes to those files, so this branch should drop the generated payload before merge.
  - confidence: 0.98
- **[P2] Use the remaining timeout for each idle re-wait:** `src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:56`
  - body: The loop starts a total cleanup budget, but every retry still calls `waitForAgentIdleBestEffort` with the original full timeout. If the first idle wait resolves near the limit and pending tools remain, cleanup can block for another full timeout and hold runner resources longer than callers requested.
  - confidence: 0.92
- **[P2] Keep the pending-tool probe non-throwing:** `src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:72`
  - body: `hasPendingToolCalls.call(opts.agent)` runs in cleanup paths but is not wrapped. If the adapter or a future agent implementation throws during shutdown, this helper can reject before the error-path cleanup disposes the session, so the probe should be best-effort like `waitForIdle`.
  - confidence: 0.84
- **[P2] Use execFile for structured extraction:** `src/hooks/bundled/session-memory/handler.ts:184`
  - body: This adds a shell command string that interpolates `sessionId` and a workspace path. Even though normal session IDs are expected to be tame, this is an unnecessary command-injection surface in an unrelated hook change; use `execFile`/`spawn` with an argument array if this behavior is kept.
  - confidence: 0.78
