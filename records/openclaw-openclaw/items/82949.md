## Summary

Keep open. The PR targets a valid source-reproducible `sessions_send` identity bug and uses the right code path, but it currently resolves names through a helper that falls back to `main` for non-agent session keys, and the supplied proof is source-probe only rather than a real OpenClaw run.

## What This Changes

This PR threads OpenClaw config into `sessions_send` A2A context builders so prompts can include configured requester and target identity names, with focused helper and announce tests.

## Best Possible Solution

Emit identity-name lines only when the session key parses to a configured agent id, cover parsed and non-parsed keys in regression tests, then verify an actual A2A `sessions_send` run.

## Reproduction Assessment

Yes. Source inspection shows current main only emits session/channel lines, and the linked issue can be reproduced by building A2A context with configured `agents.list[].identity.name` and an `agent:habit:...` requester key.

## Solution Assessment

No, not quite. Config-threading is the right narrow shape, but the implementation should parse agent session keys explicitly and skip name emission when parsing fails instead of using a helper that defaults to `main`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip identity names when the session key is not parsed:** `src/agents/tools/sessions-send-helpers.ts:57-58`
  - body: `resolveAgentIdFromSessionKey()` falls back to `main` when a session key is legacy, malformed, or otherwise not `agent:<id>:...`, so this can add `Agent ... name: <main identity>` for a sender/target that was not actually resolved to `main`. That preserves the exact kind of sender misattribution this fix is meant to avoid; parse the session key and omit the name when no agent id is present.
  - confidence: 0.88
