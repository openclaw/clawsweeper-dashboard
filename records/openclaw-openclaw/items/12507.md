## Summary

Keep open: current main has ClawHub trust-envelope and Skill Card surfaces from https://github.com/openclaw/openclaw/pull/86699, but it still does not persist trusted skill digests/signatures or enforce verification before SKILL.md content enters runtime prompts.

## What This Changes

The issue requests signed integrity verification, trusted publisher provenance, and unsigned-skill policy for SKILL.md distribution and runtime loading.

## Best Possible Solution

Adopt a maintainer-approved skill provenance contract: ClawHub publishes trusted digest/signature and publisher identity, install/update persists that state, runtime loading re-verifies canonical skill contents, and unsigned or invalid skills follow an explicit compatibility and capability policy.

## Reproduction Assessment

Yes, at the source level: current main loads SKILL.md and builds skill snapshots without trusted signature, expected digest, or verification state. This is a missing security capability rather than a regression in an existing documented contract.

## Solution Assessment

Unclear as an implementation plan: end-to-end provenance and signing is the right direction, but Ed25519 signatures, degraded modes, trust roots, revocation, migration, and capability enforcement need maintainer security/product decisions before code should land.
