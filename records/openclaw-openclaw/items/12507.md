## Summary

Keep open. Current main and the latest release still lack native trusted SKILL.md signature verification; the related SIE issue is closed as a duplicate of this issue, and the open SHA-256 PR is partial local tamper detection rather than the requested publisher/signature trust model.

## What This Changes

Request cryptographic integrity and signature verification for SKILL.md/native skills, with ClawHub signing metadata and restricted behavior for unsigned skills.

## Best Possible Solution

Keep this as the canonical native skill-signature tracker and define one end-to-end trust contract across ClawHub metadata, install/update persistence, loader-time verification, status surfaces, docs, revocation, and unsigned-skill policy.

## Solution Assessment

Unclear until maintainers choose the trust model. Cryptographic skill verification is a valid direction, but a maintainable solution needs trusted publisher/signature metadata, loader-time verification, status reporting, revocation handling, and unsigned capability policy rather than a local hash-only check.
