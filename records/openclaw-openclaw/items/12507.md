## Summary

Keep open. Current main and the latest release still do not provide trusted SKILL.md signature verification; related SIE work was closed as a duplicate of this tracker and the open SHA-256 PR is only partial tamper-detection, not publisher/signature provenance.

## What This Changes

The issue requests cryptographic integrity/signature verification for SKILL.md skills, with unsigned skills restricted or degraded by policy.

## Best Possible Solution

Keep this as the canonical native skill-signature tracker and define one end-to-end trust contract across ClawHub metadata, archive verification, persisted origin/lock state, loader-time verification, status surfaces, docs, revocation, and unsigned-skill policy.

## Reproduction Assessment

Not applicable. This is a feature/security-design request rather than broken existing behavior, and current-main plus latest-release source inspection confirms the requested native signature and loader verification path is absent.

## Solution Assessment

Unclear until maintainers choose the trust model. Cryptographic skill verification is a valid direction, but the maintainable solution needs trusted publisher/signature metadata, loader-time verification, status reporting, revocation handling, and unsigned capability policy rather than a local hash-only check.
