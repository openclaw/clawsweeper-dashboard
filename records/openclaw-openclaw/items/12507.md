## Label Justifications

- `P2`: This is meaningful security hardening for skill distribution and runtime trust, but it needs product/security design and is not an active reproduced emergency.
- `impact:security`: The issue concerns supply-chain integrity, publisher authenticity, local tamper detection, and capability boundaries for untrusted skill instructions.

## Summary

Keep open: current main has partial ClawHub verification, Skill Card, and install provenance metadata, but runtime skill loading still consumes local SKILL.md bytes without trusted digest/signature revalidation or unsigned/invalid-skill capability policy.

## What This Changes

The issue requests cryptographic integrity/signature verification for SKILL.md skills and policy restrictions for unsigned or tampered skills.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, bind artifact and SKILL.md identity into install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes, source-reproducible: current main reads SKILL.md into Skill records and snapshots without comparing a trusted signature, persisted digest, verification envelope, or unsigned capability tier. v2026.6.8 has the same loader/type gap.

## Solution Assessment

No, not as a one-shot Ed25519/degraded-mode patch without design. The maintainable path is to extend the existing ClawHub verification and install metadata into runtime enforcement after maintainers settle trust roots, revocation, compatibility, and operator override behavior.
