## Summary

Keep open. Current main still lacks native signed or trusted-integrity skill loading, and the issue discussion plus the closed SIE duplicate treat this as the canonical skill trust-boundary tracker rather than stale cleanup.

## What This Changes

Request content hash/signature verification for SKILL.md skills, including ClawHub signing metadata, loader checks, and restricted behavior for unsigned skills.

## Best Possible Solution

Keep this as the canonical skill-signature trust-model tracker and design one end-to-end contract across ClawHub metadata, install/update persistence, loader-time verification, status surfaces, docs, and unsigned-skill policy.

## Reproduction Assessment

Not applicable. This is a feature/security-design request rather than a broken existing behavior report, and source inspection confirms the requested signing and loader verification path is absent on current main.

## Solution Assessment

Unclear as an implementation path until maintainers choose the trust model. A hash-only audit check is useful but incomplete; the maintainable solution should define signed metadata, loader policy, and user-visible verification state together.
