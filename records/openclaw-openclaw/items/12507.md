## Summary

Current main adds ClawHub trust-envelope and security-verdict reporting, but still does not authenticate installed skill bytes or carry verification state into runtime loading; this remains the canonical security and product-design issue.

## What This Changes

Add trusted cryptographic integrity and publisher-signature verification for distributed skills, with restricted capabilities for unsigned skills.

## Best Possible Solution

Have ClawHub issue a versioned, cryptographically authenticated envelope bound to an immutable skill artifact and publisher identity; verify it before extraction, persist the trusted binding, revalidate installed contents before trusted use, expose status diagnostically, and adopt an explicit migration and capability policy for unsigned or modified skills.

## Reproduction Assessment

Yes, at source level. Current main records registry and version metadata, fetches registry verdicts, and reads local SKILL.md content without comparing the loaded bytes to a persisted trusted digest or signature.

## Solution Assessment

No, not fully as proposed. Ed25519 detached signatures are plausible, but a loader-only check is insufficient without registry trust roots, exact artifact binding, install persistence, revocation handling, local-modification semantics, and an upgrade-safe enforcement policy.
