## Summary

Keep open. Current main still lacks native signed/integrity-verified skill loading: ClawHub skill metadata and local lock/origin records do not carry trusted digest/signature state, the installer does not compare the downloaded archive to trusted metadata, and the runtime loader reads SKILL.md into skill snapshots without verification or unsigned-skill policy.

## What This Changes

The issue requests cryptographic integrity and signature verification for skills, including signed SKILL.md handling, trusted metadata, and policy restrictions for unsigned skills.

## Best Possible Solution

Keep this as the canonical skill-signature trust-model tracker and design one end-to-end contract spanning ClawHub metadata, install/update persistence, loader-time verification, status surfaces, docs, and unsigned-skill policy.

## Reproduction Assessment

Not applicable. This is a feature/security-design request rather than a broken existing behavior report, and source inspection confirms the requested signing and loader verification path is absent on current main.

## Solution Assessment

Unclear as an implementation path until maintainers choose the trust model. A hash-only audit check would be useful but incomplete; the maintainable solution should define signed metadata, loader policy, and user-visible verification state together.
