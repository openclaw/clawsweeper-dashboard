## Summary

Keep open: current main has ClawHub verification, provenance, and status surfaces, but still does not bind runtime `SKILL.md` loading to a trusted signature/digest or define an unsigned-skill capability policy.

## What This Changes

The issue requests signed or otherwise trusted integrity verification for skills before they are loaded into agent context.

## Best Possible Solution

Keep this issue as the canonical native skill integrity/signature work and extend ClawHub trust metadata into load-time enforcement after maintainers settle trust roots, revocation, unsigned compatibility, and operator override behavior.

## Reproduction Assessment

Yes. Source inspection shows current main reads local `SKILL.md` into Skill records and runtime snapshots without comparing a trusted signature, persisted digest, revocation state, or unsigned capability tier.

## Solution Assessment

No, not as a one-shot Ed25519/degraded-mode patch without design. The maintainable path is to extend current ClawHub verification and provenance into runtime enforcement after maintainers choose trust roots, revocation, compatibility, and operator override behavior.
