## Summary

Current main has partial ClawHub verification, trust display, and local install provenance, but it still does not enforce trusted SKILL.md signatures or define unsigned-skill capability policy at runtime. This remains valid canonical security/product work, not conservative cleanup-closeable.

## What This Changes

The issue requests cryptographic signature or integrity verification for SKILL.md skills, with unsigned-skill restrictions tied into capability and security-profile policy.

## Best Possible Solution

Keep this issue open as the canonical native skill integrity/signature tracker and extend ClawHub trust metadata into runtime enforcement after maintainers settle trust roots, revocation, unsigned compatibility, and operator override behavior.

## Reproduction Assessment

Yes, as a source-level feature gap. Current main reads SKILL.md into Skill records and runtime snapshots without comparing a trusted signature, persisted digest, revocation state, or unsigned capability tier.

## Solution Assessment

No, not as a one-shot Ed25519/degraded-mode patch without design. The maintainable path is to extend current ClawHub verification and provenance into runtime enforcement after maintainers choose trust roots, revocation, compatibility, and operator override behavior.

## Risks / Open Questions

- A local hash-only implementation could create false assurance unless hashes are bound to trusted ClawHub publisher/signature or digest metadata and rechecked at runtime load.
