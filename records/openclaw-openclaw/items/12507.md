## Summary

Keep open: current main has useful ClawHub verification, install metadata, and UI/CLI trust surfaces, but it still does not bind runtime SKILL.md loading to trusted signatures or persisted digest state, and unsigned/invalid skill policy remains a security/product decision.

## What This Changes

The issue requests cryptographic integrity and signature verification for SKILL.md/native skill contents, including trust metadata, runtime verification, and unsigned-skill capability policy.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, persist authenticated artifact and content identity into install metadata, revalidate canonical skill contents before runtime use, and define a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes, at source level. Current main still reads local SKILL.md into runtime skill records without comparing trusted ClawHub signature, verification, archive digest, or skill-file digest state.

## Solution Assessment

No, the exact phased Ed25519/degraded-mode proposal is not yet a settled best fix. The maintainable direction is to extend the existing ClawHub verification, install metadata, and runtime loader path rather than add a parallel unauthenticated hash-only system.
