## Summary

Current main still lacks trusted publisher signatures, persisted expected integrity, and load-time verification or capability policy for skills. The narrower SIE proposal was closed in favor of this canonical issue, while the hash-only implementation PR closed unmerged because it neither established trusted provenance nor delivered reliable verification.

## What This Changes

Add trusted integrity and signature verification for distributed skills, with policy-based restrictions for unsigned or invalid skills.

## Best Possible Solution

Adopt a threat-modeled provenance contract where ClawHub supplies trusted version digests or signatures and publisher identity, installs persist that state, runtime loading re-verifies canonical skill contents, and unsigned or invalid skills follow an explicitly approved compatibility policy.

## Reproduction Assessment

Yes at the source level: current main clearly loads skill content without trusted signature or provenance state, although this is a missing capability rather than a regression in an existing contract.

## Solution Assessment

The end-to-end provenance direction is appropriate, but Ed25519 signatures and degraded capability modes are not implementation-ready until maintainers choose the signing authority, trust and revocation model, compatibility behavior, and enforcement boundary.

## Risks / Open Questions

- A hash-only design can create false assurance because a digest computed from received bytes does not independently authenticate the publisher or registry intent.
- Fail-closed or capability-restricted loading could break existing local, bundled, managed, GitHub-sourced, and legacy ClawHub skills without an explicit migration and operator policy.
- The design must settle key rotation, revocation, canonicalization, local-edit semantics, offline behavior, and verification timing before enforcement is safe.
