## Summary

Keep this issue open: current main and v2026.6.5 expose a ClawHub trust-envelope lookup, but neither the installed artifact nor locally loaded `SKILL.md` is cryptographically bound to that envelope, and no runtime policy restricts unsigned, invalid, or modified skills.

## What This Changes

Add trusted cryptographic integrity and publisher-signature verification for distributed skills, with policy-based restrictions for unsigned skills.

## Best Possible Solution

Use ClawHub as the provenance and publisher-signature authority, bind its authenticated artifact identity to persisted install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, and locally modified skills across every supported skill root.

## Reproduction Assessment

Yes at the source level: current main’s ClawHub verification path can be traced to registry metadata, while the local loader demonstrably reads `SKILL.md` without using that result. This is a missing security capability rather than a regression in an existing signature contract.

## Solution Assessment

No, the issue’s exact phased proposal is not yet a settled best fix because Ed25519 trust roots, revocation, all-root coverage, unsigned behavior, and capability restrictions require product and security decisions. The narrowest maintainable direction is to extend the existing ClawHub envelope and lifecycle metadata rather than add a parallel verification system.
