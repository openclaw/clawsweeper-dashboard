## Summary

Keep open. Current main and the latest release still lack native trusted SKILL.md signature verification, while the closed SIE issue points back here as the canonical tracker and the open SHA-256 PR is only partial local tamper detection.

## What This Changes

The issue requests cryptographic signing and verification for SKILL.md skills, including ClawHub metadata, install persistence, loader enforcement, and unsigned-skill policy.

## Best Possible Solution

Keep this as the canonical native skill-signature tracker and define one end-to-end trust contract across ClawHub metadata, archive verification, persisted origin/lock state, loader-time verification, status surfaces, docs, revocation, and unsigned-skill policy.

## Reproduction Assessment

Not applicable. This is a feature/security-design request rather than broken existing behavior, and current-main source inspection confirms the requested native signature and loader verification path is absent.

## Solution Assessment

Unclear until maintainers choose the trust model. Cryptographic skill verification is a valid direction, but a maintainable solution needs trusted publisher/signature metadata, loader-time verification, status reporting, revocation handling, and unsigned capability policy rather than a local hash-only check.
