## Summary

Keep open: current main and the latest release have ClawHub verification, install provenance, and status surfaces, but runtime SKILL.md loading still does not require a trusted signature/digest or define unsigned-skill capability policy.

## What This Changes

The issue requests cryptographic integrity/signature verification for SKILL.md skills, with unsigned skills restricted or degraded before runtime use.

## Best Possible Solution

Keep this issue as the canonical native skill integrity/signature tracker and extend ClawHub trust metadata into runtime enforcement after maintainers settle trust roots, revocation, unsigned compatibility, and operator override behavior.

## Reproduction Assessment

Yes, as a source-level feature gap. Current main reads local SKILL.md into Skill records and runtime snapshots without comparing a trusted signature, persisted digest, revocation state, or unsigned capability tier.

## Solution Assessment

No, not as a one-shot Ed25519/degraded-mode patch without design. The maintainable path is to extend current ClawHub verification and provenance into runtime enforcement after maintainers choose trust roots, revocation, compatibility, and operator override behavior.
