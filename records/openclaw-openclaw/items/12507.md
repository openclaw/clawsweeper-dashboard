## Summary

Keep open. Current main still lacks native trusted SKILL.md signature verification, and the related open SHA-256 PR is partial local tamper detection rather than the requested publisher/signature trust model.

## What This Changes

The issue requests cryptographic integrity/signature verification for SKILL.md-based skills, including trusted publisher metadata, loader enforcement, and restricted behavior for unsigned skills.

## Best Possible Solution

Keep this as the canonical native skill-signature tracker and define one end-to-end trust contract across ClawHub metadata, install/update persistence, loader-time verification, status surfaces, docs, revocation, and unsigned-skill policy.

## Reproduction Assessment

Not applicable. This is a feature/security-design request rather than broken existing behavior, and current-main source inspection confirms the requested native signature and loader verification path is absent.

## Solution Assessment

Unclear until maintainers choose the trust model. Cryptographic skill verification is a valid direction, but a maintainable solution needs trusted publisher/signature metadata, loader-time verification, status reporting, revocation handling, and unsigned capability policy rather than a local hash-only check.
