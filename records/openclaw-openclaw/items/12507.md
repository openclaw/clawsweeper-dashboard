## Summary

Keep open. Current main still has native ClawHub skill install/update and local SKILL.md loading without trusted digest/signature metadata, persisted verification state, or an unsigned-skill policy, and the existing maintainer/Codex discussion already treats this as the canonical skill integrity work.

## What This Changes

Add cryptographic integrity and signature verification for SKILL.md-based skills, including policy for unsigned or unverified skills.

## Best Possible Solution

Define a native skill trust contract: ClawHub should expose trusted digest/signature/publisher metadata for skill versions, OpenClaw should verify archives before extraction, persist expected integrity/signature state in origin/lock records, recheck SKILL.md or full skill contents during snapshot/load, surface verification status in CLI/UI/status, and document/enforce the unsigned-skill policy.

## Reproduction Assessment

Yes. The gap is reproducible by tracing current main: `openclaw skills install` downloads/extracts a ClawHub archive without comparing trusted integrity metadata, then `loadSkillsFromDirSafe` later reads SKILL.md into the prompt path without verification state.

## Solution Assessment

Unclear. The requested signing direction is appropriate, but the best maintainable fix needs a small security design first so registry metadata, runtime load policy, persisted lock/origin fields, and capability restrictions stay consistent.
