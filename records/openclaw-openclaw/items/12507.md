## Summary

Keep open: current main and v2026.6.8 include useful ClawHub trust-envelope, security-verdict, Skill Card, and install metadata surfaces, but runtime skill loading still consumes local SKILL.md bytes without revalidating a trusted signature/digest or applying an unsigned/invalid-skill capability policy.

## What This Changes

The issue requests cryptographic integrity and signature verification for SKILL.md skills, including trusted metadata, runtime verification, and restricted behavior for unsigned skills.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, bind authenticated artifact and content identity into install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes, source-reproducible: current main reads SKILL.md into Skill records and snapshots without comparing a trusted signature, persisted digest, verification envelope, or unsigned capability tier. v2026.6.8 has the same loader/type gap.

## Solution Assessment

No, not exactly as a one-shot Ed25519/degraded-mode patch without design. The maintainable path is to extend the existing ClawHub verification and install metadata into runtime enforcement after maintainers settle trust roots, revocation, compatibility, and operator override behavior.
