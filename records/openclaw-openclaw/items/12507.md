## Summary

The central request remains unresolved: current main offers ClawHub verification and security-verdict queries, but signatures are advisory, installed identity is not bound to persisted trusted integrity metadata, and the loader still accepts local `SKILL.md` content without signature-based policy.

## What This Changes

Add trusted integrity and cryptographic signature verification for skills, with restricted behavior for unsigned or unverified content.

## Best Possible Solution

Define an end-to-end, upgrade-safe trust contract in ClawHub and OpenClaw: bind signed publisher provenance to exact artifacts, verify before installation, persist canonical verified identity, recheck installed content at audit/load boundaries, expose status clearly, and enforce unsigned policy only through an explicit rollout with safe defaults and revocation handling.

## Reproduction Assessment

Yes at source level: current main can query advisory verification data, while the install metadata and local loader paths show no persisted signature binding or unsigned enforcement. This is a missing security capability rather than a failing pre-existing contract.

## Solution Assessment

No, loader-only detached-signature verification is not by itself the best solution. The maintainable solution must first define trusted registry and publisher semantics, exact artifact binding, persistence, revocation, upgrade behavior, and where capability restrictions are enforced.
