## Summary

Keep open: current main and the latest release expose ClawHub verification/status surfaces, but runtime skill loading still reads local `SKILL.md` bytes without enforcing a trusted signature, comparing persisted digest state, or applying an unsigned-skill capability policy.

## What This Changes

The issue requests cryptographic integrity and signature verification for `SKILL.md` skill contents, including trusted publisher metadata and policy differences for unsigned skills.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, bind authenticated artifact and content identity into install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes at the source level: current main still reads local `SKILL.md` into skill records without comparing stored ClawHub verification, archive digest, or skill-file digest state. I did not run a live tamper scenario because this review is read-only.

## Solution Assessment

No, the exact phased Ed25519/degraded-mode proposal is not yet a settled best fix. The maintainable direction is to extend the existing ClawHub verification, install metadata, and runtime loader path rather than add a parallel unauthenticated hash-only system.
