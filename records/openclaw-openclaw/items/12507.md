## Summary

Keep open: current main added ClawHub trust-envelope/status surfaces, but the runtime still loads local `SKILL.md` bytes without enforcing a trusted signature, comparing the persisted digest, or applying an unsigned-skill capability policy.

## What This Changes

Request native integrity/signature verification for skills so modified or unsigned `SKILL.md` content is detected before runtime use.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, bind authenticated artifact and content identity into install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes at the source level: current main records some ClawHub trust metadata, but `loadSkillsFromDirSafe` still reads the local `SKILL.md` into the skill snapshot without comparing trusted signature/digest state. I did not run a live tamper scenario because this review is read-only.

## Solution Assessment

No, the issue’s exact phased Ed25519/degraded-mode proposal is not yet a settled best fix. The maintainable direction is to extend the existing ClawHub verification, install metadata, and runtime loader path rather than add a parallel unauthenticated hash-only system.
