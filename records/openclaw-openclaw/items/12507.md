## Summary

Keep open: current main and v2026.6.6 have useful ClawHub verification, status, Skill Card, and install metadata surfaces, but runtime skill loading still consumes local SKILL.md bytes without enforcing a trusted signature, comparing persisted digest state, or applying an unsigned-skill capability policy.

## What This Changes

The issue requests cryptographic integrity and signature verification for SKILL.md-based skills, including restricted behavior for unsigned or untrusted skills.

## Best Possible Solution

Use ClawHub as the provenance/signature authority, bind authenticated artifact and content identity into install metadata, revalidate canonical skill contents before runtime use, and apply a maintainer-approved policy for unsigned, invalid, revoked, or locally modified skills.

## Reproduction Assessment

Yes. At the source level, current main still reads local SKILL.md into runtime skill records without comparing trusted ClawHub signature, verification, archive digest, or skill-file digest state.

## Solution Assessment

No, the exact phased Ed25519/degraded-mode proposal is not yet a settled best fix. The maintainable direction is to extend the existing ClawHub verification, install metadata, and runtime loader path rather than add a parallel unauthenticated hash-only system.
