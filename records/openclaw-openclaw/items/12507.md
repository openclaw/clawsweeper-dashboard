## Summary

Keep this issue open as the canonical remaining native-skill integrity work. Current main now exposes a ClawHub trust-envelope verification command, but it does not bind that result to installed bytes, persist trusted integrity/signature state, recheck local skill content before runtime use, or enforce a defined unsigned-skill capability policy.

## What This Changes

Add cryptographic integrity and publisher-signature verification for skills, with explicit capability restrictions for unsigned or unverified content.

## Best Possible Solution

Extend the existing `clawhub.skill.verify.v1` contract into one end-to-end provenance path: authenticate expected artifact/content metadata, compare it before extraction, persist the verified identity and digest in canonical install state, reverify content before runtime use, expose status through existing CLI/UI surfaces, and adopt an explicit migration-compatible policy for unsigned or unverifiable skills.

## Reproduction Assessment

Yes at source level. Current main directly shows that native installation records omit trusted integrity/signature data and runtime skill snapshots carry no verification state, although this is a missing security capability rather than a conventional failing-runtime bug.

## Solution Assessment

The end-to-end goal is appropriate, but the issue’s exact Ed25519 and degraded-mode design is not yet proven to be the best complete policy. The narrowest maintainable direction is to extend the existing ClawHub trust envelope through install persistence and runtime verification, with maintainers deciding authority, migration, and enforcement defaults.
