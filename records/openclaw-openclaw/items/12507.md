## Summary

Keep open. Current `main` still lacks trusted native skill signing or integrity verification at ClawHub metadata, install persistence, and runtime `SKILL.md` loading boundaries, and the existing maintainer discussion already treats this as the canonical skill integrity work rather than cleanup to close.

## What This Changes

Request native content integrity and signature verification for `SKILL.md` skills, including trusted metadata, runtime verification, and policy for unsigned skills.

## Best Possible Solution

Define a native skill trust model that adds trusted digest/signature/publisher metadata to ClawHub, verifies archives before extraction, persists expected integrity in origin/lock state, revalidates skill contents during snapshot/load, and exposes a documented unsigned-skill policy across CLI/UI/status surfaces.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the native ClawHub skill path and runtime loader have no trusted signature or persisted integrity state, while only the separate plugin package path verifies a trusted archive hash.

## Solution Assessment

Unclear. The requested direction is valid security work, but the best fix needs maintainer decisions on publisher trust, ClawHub API metadata, migration, runtime enforcement, and unsigned capability policy before a narrow patch is safe.
