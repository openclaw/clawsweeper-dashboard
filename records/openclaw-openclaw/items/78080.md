## Summary

Keep open: current main still returns `null` for ordinary non-DM Discord text that matches `hasControlCommand()` before the later Discord routing, authorization, mention, or agent enqueue paths can handle it.

## What This Changes

The issue asks for Discord guild text slash/control commands such as `/steer ...` to reach the normal text-command handling path or produce a visible rejection instead of being silently consumed.

## Best Possible Solution

Keep the native Discord interaction echo drop, but let ordinary guild text control-command messages proceed into the existing Discord text-command authorization and mention-bypass path with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `preflightDiscordMessage` for a normal guild message with content like `/steer test` and a non-native-command message type; current main returns `null` at the early `hasControlCommand()` guard before agent enqueue.

## Solution Assessment

Yes. The narrowest maintainable fix is to remove or relocate the early silent return so the existing command gate handles authorized and unauthorized text commands, while preserving the later native `ChatInputCommand`/`ContextMenuCommand` echo drop.
