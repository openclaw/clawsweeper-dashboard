## Summary

Keep open: current main still lets a non-empty heartbeat reply payload made of the internal stream-error sentinel pass through heartbeat normalization and into durable channel delivery, so the reported Feishu-visible placeholder blast is still a plausible current bug.

## What This Changes

The issue asks OpenClaw to suppress repeated internal stream-error placeholder text from user-visible heartbeat fallback replies.

## Best Possible Solution

Add a narrow heartbeat-runner guard that treats placeholder-only stream-error sentinel output as internal empty/failed heartbeat output before durable channel delivery, while preserving the persisted sentinel contract for replay safety.

## Reproduction Assessment

Yes at source level. A mocked heartbeat reply payload containing only repeated `STREAM_ERROR_FALLBACK_TEXT` would pass current normalization and reach `sendDurableMessageBatch`; I did not run a live Feishu/provider fallback reproduction.

## Solution Assessment

Yes. The narrowest maintainable fix is to classify placeholder-only heartbeat output before delivery, not to remove the sentinel or add channel/provider-specific behavior.
