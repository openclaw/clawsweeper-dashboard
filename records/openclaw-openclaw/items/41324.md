## Summary

Keep this PR open: it addresses a real Feishu Bitable permission gap that is not implemented on current main, but the current patch still has correctness/security review blockers and lacks required real Feishu behavior proof.

## What This Changes

This PR adds requester auto-grant support to `feishu_bitable_create_app`, a `grant_to_requester` tool parameter, requester permission response fields, and direct tool registration to read Feishu requester context.

## Best Possible Solution

Land a corrected Feishu Bitable requester grant that validates SDK response codes, preserves requester ID type, updates security-audit coverage and regression tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings from source inspection: a nonzero `permissionMember.create` response or a Feishu event with only `sender_id.user_id` would exercise the incorrect PR paths. The intended live Feishu Bitable behavior still lacks after-fix proof.

## Solution Assessment

No. Mirroring `feishu_doc` is the right direction, but this patch needs response validation, ID-type-safe requester handling, security-audit/test updates, and real behavior proof before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate permission responses before reporting success:** `extensions/feishu/src/bitable.ts:393-402`
  - body: `permissionMember.create()` resolves to a Feishu response with `code` and `msg`, and existing permission code treats nonzero `code` as failure. This branch sets `requester_permission_added` after any resolved response, so missing scopes or invalid members can be reported as a successful grant.
  - confidence: 0.91
- **[P2] Match requester member_type to the ID format:** `extensions/feishu/src/bitable.ts:397-398`
  - body: The trusted requester string can come from a `sender_id.user_id` fallback when `open_id` is unavailable, but this always sends `member_type: "openid"`. Those Feishu deliveries would fail the auto-grant for legitimate requesters.
  - confidence: 0.86
- **[P3] Include Bitable grants in the Feishu security audit:** `extensions/feishu/src/bitable.ts:541-544`
  - body: Adding `grant_to_requester` creates a new automatic Bitable sharing path, but the Feishu security audit still warns only for doc creation grants. Operators relying on that audit would miss the new auto-share surface.
  - confidence: 0.78
