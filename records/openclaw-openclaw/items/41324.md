## Summary

Keep open: current main still lacks Bitable requester grants and this PR is a relevant Feishu feature, but the branch should not merge as-is because the new permission path can report failed grants as success, can use the wrong Feishu member ID type, lacks security-audit coverage, and has no real Feishu behavior proof.

## What This Changes

The PR changes `extensions/feishu/src/bitable.ts` to add `grant_to_requester`, read Feishu requester context for `feishu_bitable_create_app`, call `drive.permissionMember.create` for new Bitables, and return requester permission status fields.

## Best Possible Solution

Land a corrected Bitable requester grant that validates Feishu response codes, handles `openid` versus `userid`, updates security audit and regression coverage, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: have `permissionMember.create` resolve with a nonzero `code`, or use a Feishu requester ID that came from the `user_id` fallback, and the PR path misreports or misroutes the grant.

## Solution Assessment

No. Mirroring `feishu_doc` is a reasonable direction, but this diff needs response validation, ID-type-safe requester handling, audit/test updates, and real behavior proof before it is mergeable.

## Review Findings

Full review comments:

- **[P1] Validate permission responses before reporting success:** `extensions/feishu/src/bitable.ts:393-402`
  - body: `permissionMember.create()` resolves to a Feishu response with `code` and `msg`, and existing permission code treats nonzero `code` as failure. This branch sets `requester_permission_added` after any resolved response, so missing scopes or invalid members can be reported as a successful grant.
  - confidence: 0.91
- **[P2] Match requester member_type to the ID format:** `extensions/feishu/src/bitable.ts:397-398`
  - body: The trusted requester string can come from a `sender_id.user_id` fallback when `open_id` is unavailable, but this always sends `member_type: "openid"`. Those Feishu deliveries would fail the auto-grant for legitimate requesters.
  - confidence: 0.86
- **[P3] Include Bitable grants in the Feishu security audit:** `extensions/feishu/src/bitable.ts:541-544`
  - body: Adding `grant_to_requester` creates a new automatic Bitable sharing path, but the Feishu security audit still warns only for doc creation grants. Operators relying on that audit would miss the new auto-share surface.
  - confidence: 0.78
