## Summary

Keep open; current main still lacks Bitable requester auto-granting, but this PR is not merge-ready because the new grant path can misreport failed Feishu permission calls, assumes every requester id is an open_id, and misses the Feishu audit/changelog surface for a user-facing access-control feature.

## What This Changes

The PR adds requester-context permission granting to `feishu_bitable_create_app`, a `grant_to_requester` tool parameter, and requester permission status fields in `extensions/feishu/src/bitable.ts`.

## Best Possible Solution

Revise the Bitable grant implementation to validate Feishu `code`/`msg` responses, preserve or detect the requester id type, update Feishu audit/tests/changelog, and then send the feature through normal maintainer review.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: mock `drive.permissionMember.create` to resolve `{ code: 999, msg: "missing scope" }` and the patch still reports `requester_permission_added: true`; pass a Feishu requester derived from `sender_id.user_id` and the patch submits it as `member_type: "openid"`.

## Solution Assessment

No. Mirroring `feishu_doc` is a reasonable direction, but the implementation should validate permission responses, handle requester id type correctly, and cover the audit/test/changelog surfaces before merge.
