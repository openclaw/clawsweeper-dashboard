## Summary

Keep open: current main still lacks Bitable requester grants, and this PR addresses a real Feishu parity gap, but the patch has merge-blocking correctness issues around permission response handling and requester ID type handling.

## What This Changes

The PR adds `grant_to_requester` support to `feishu_bitable_create_app`, grants the trusted Feishu requester edit access after Bitable creation, and returns requester permission status metadata.

## Best Possible Solution

Land a corrected Bitable requester-grant implementation that validates SDK responses, uses an ID-type-safe permission member type, extends audit and regression coverage, and records the user-facing change in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: mock `permissionMember.create` to resolve a nonzero `code` and the patch still reports success; route a Feishu sender that only has `user_id` and the patch submits it as `member_type: "openid"`.

## Solution Assessment

No. Mirroring `feishu_doc` is the right direction, but this implementation should validate permission responses, preserve or infer requester ID type, and update audit/test/changelog coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate permission responses before reporting success:** `extensions/feishu/src/bitable.ts:390-399`
  - body: `client.drive.permissionMember.create()` resolves with `code`/`msg`, and existing Feishu permission code treats nonzero `code` as failure. This path sets `requester_permission_added` after any resolved response, so missing scopes or invalid members can be reported as a successful grant.
  - confidence: 0.91
- **[P2] Match member_type to the requester ID format:** `extensions/feishu/src/bitable.ts:394-395`
  - body: The trusted requester string can fall back to `sender_id.user_id` when `open_id` is absent, but this always sends `member_type: "openid"`. User-id-only Feishu deliveries will fail the new auto-grant path for legitimate requesters.
  - confidence: 0.86
- **[P3] Include Bitable grants in the Feishu security audit:** `extensions/feishu/src/bitable.ts:538-544`
  - body: This adds a new automatic Bitable sharing path through `grant_to_requester`, but the Feishu security audit currently warns only for doc creation grants. Operators relying on the audit would miss the new Bitable auto-share surface.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `extensions/feishu/src/bitable.ts:538-544`
  - body: This is a user-facing Feishu feature, but the patch only changes `extensions/feishu/src/bitable.ts`. Repo policy requires a `CHANGELOG.md` entry for user-facing feature changes.
  - confidence: 0.84
