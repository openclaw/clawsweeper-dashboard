## Summary

Keep open. This is a maintainer-owned, protected-label PR, and the proposed scanner has a concrete false-negative bug: any file-level `afterEach`, `afterAll`, or `finally` suppresses all `mkdtemp` findings in that file even when the cleanup is unrelated.

## What This Changes

This PR adds a standalone `lint:tmp:test-temp-cleanup` script that scans tracked test files for `mkdtemp` usage without obvious scoped cleanup, plus Vitest coverage for the scanner and JSON output.

## Best Possible Solution

Keep the guard standalone, but tighten it to validate cleanup scope per `mkdtemp` call and add a regression fixture where an unrelated hook or `finally` does not suppress a leak.

## Reproduction Assessment

Yes. From source inspection, a test file with an unrelated `afterEach`, `afterAll`, or `finally` plus an uncleaned `mkdtemp` reaches line 47 and is suppressed, so the false negative is source-reproducible.

## Solution Assessment

No. The whole-file cleanup signal is too coarse for the stated guard; the maintainable fix is to classify cleanup around each `mkdtemp` call or test block instead of suppressing the entire file.
