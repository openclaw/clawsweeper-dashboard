## Summary

Keep open. Current main still leaves the reported Telegram delivery reliability cluster unresolved across all three central parts: live sends still get a fresh queue entry per enqueue, recovery still does not classify the reported Telegram terminal errors as permanent, and direct Telegram action sends still bypass the chunked HTML-mode text path.

## What This Changes

The issue requests idempotent Telegram delivery queue retry semantics, permanent classification for Telegram terminal errors, and pre-splitting long Telegram messages before Telegram's 4096-character limit.

## Best Possible Solution

Land one focused reliability fix that gives each logical outbound send stable queue update semantics, classifies unrecoverable Telegram delivery errors before live or recovery retry, and routes every Telegram text-send surface through a chunk-aware <=4000/4096-character path.

## Reproduction Assessment

Yes. The issue includes concrete production queue/log evidence, and current-main source inspection gives a high-confidence focused reproduction path: send or replay a Telegram payload over 4096 characters that fails with Telegram 400 and observe retryable queue state plus direct action non-chunking.

## Solution Assessment

Yes. The requested direction is the narrow maintainable solution, with shared queue retry semantics in outbound core and Telegram-specific terminal-error and chunking behavior kept at the Telegram plugin seam.
