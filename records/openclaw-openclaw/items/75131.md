## Summary

Keep open. Current main has Telegram long-message splitting, but the delivery queue still creates a fresh UUID per enqueue and recovery still misses the reported terminal Telegram errors; the linked implementation candidate should be reviewed before this issue closes.

## What This Changes

The issue requests idempotent outbound delivery queue retries, permanent-fail handling for terminal Telegram 400/403 errors, and Telegram text splitting before the 4096-character Bot API limit.

## Best Possible Solution

Land or replace the linked fix with a focused outbound reliability change that passes a stable logical delivery key into queue storage, reuses one queue entry on retry, classifies the reported Telegram terminal errors as permanent, preserves current Telegram chunking, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: direct sends call enqueueDelivery without a queue identity, enqueueDelivery generates a fresh UUID, and the permanent-error classifier lacks the reported Telegram terminal messages.

## Solution Assessment

Yes. The narrow maintainable fix belongs in outbound queue identity and recovery error classification while preserving the existing Telegram chunking path; no config or product decision is needed.
