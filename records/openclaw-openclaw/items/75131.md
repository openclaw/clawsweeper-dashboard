## Summary

Keep open. Current main still leaves the reported Telegram delivery reliability cluster unresolved: permanent Telegram size/auth errors are not classified, live retry attempts still get fresh queue entries, and the Telegram action path can still bypass the chunk-aware send mode.

## What This Changes

The issue requests idempotent Telegram delivery retries, permanent-failure handling for unrecoverable Telegram 400/403 errors, and pre-splitting long Telegram sends before they hit Telegram's message limit.

## Best Possible Solution

Land one focused reliability fix through the existing outbound queue and Telegram send seams: keep a stable retry identity for the same logical send, classify unrecoverable Telegram provider failures before retrying, and route every Telegram text send surface through the existing chunk-aware path.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the reported Telegram permanent errors miss isPermanentDeliveryError, repeated live attempts enqueue new IDs, and Telegram action sends can reach the non-chunked markdown/default branch.

## Solution Assessment

Yes. The requested direction is the narrow maintainable solution if implemented through the existing delivery-queue storage/recovery and Telegram outbound/action send seams, with explicit tests to avoid deduping legitimate duplicate sends.
