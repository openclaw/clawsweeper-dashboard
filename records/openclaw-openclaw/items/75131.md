## Summary

Keep open. Current main still matches the reported reliability cluster: live delivery creates a new queue entry for each send attempt, the recovery permanent-error matcher still omits the reported Telegram terminal errors, and the direct Telegram action send path can still reach the non-chunked markdown send mode.

## What This Changes

Make Telegram delivery retries idempotent, permanently fail unrecoverable Telegram 400/403 delivery errors, and pre-split outbound Telegram text above Telegram's message length limit.

## Best Possible Solution

Land one focused reliability fix that gives a logical send a stable queued identity, classifies unrecoverable Telegram delivery failures before recovery retry, and ensures every Telegram text-send surface uses a chunk-aware path at or below the documented 4000/4096 character limit.

## Reproduction Assessment

Yes. The issue includes concrete production queue/log evidence, and current-main source provides a high-confidence source-level reproduction path with mocked Telegram long-message failures and repeated live send attempts.

## Solution Assessment

Yes. The requested direction is the narrow maintainable repair: keep Telegram-specific chunking in the Telegram plugin/send surface and either extend the generic permanent-error classifier carefully or add a provider-aware classifier seam without moving bundled plugin policy into unrelated core paths.
