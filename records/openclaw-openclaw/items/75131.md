## Summary

Keep open: current main only partially addresses the report. Telegram outbound now has a rich-message chunking path, but the shared delivery recovery classifier still misses the reported terminal Telegram errors, and the queue layer still has no stable logical enqueue key; the linked fork PR is open but unmerged and explicitly leaves Telegram chunking unchanged.

## What This Changes

The issue requests idempotent delivery queue retries, permanent classification for terminal Telegram delivery errors, and pre-splitting or otherwise avoiding too-long Telegram outbound messages.

## Best Possible Solution

Keep this as the canonical tracking issue until a merged OpenClaw PR covers logical queue ids, terminal Telegram delivery classification, and the current Telegram long-text behavior with focused regression tests.

## Solution Assessment
