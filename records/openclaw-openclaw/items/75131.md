## Summary

Keep open. Current main has addressed the long Telegram text splitting path, but the queue still creates a fresh delivery UUID per send attempt and the recovery permanent-error classifier still misses several reported terminal Telegram 400/403 cases.

## What This Changes

The issue asks for idempotent delivery-queue retry identity, permanent Telegram 400/403 recovery classification, and pre-splitting Telegram sends at the platform limit.

## Best Possible Solution

Land a focused reliability fix that preserves current Telegram chunking while adding stable delivery-queue identity for logical direct sends and explicit permanent classification for the reported Telegram terminal errors.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: repeated direct sends still reach enqueueDelivery, which generates a fresh UUID, and the recovery classifier still does not match the reported terminal Telegram errors.

## Solution Assessment

Yes, with scope trimmed to the remaining gaps. The maintainable fix belongs at the outbound delivery queue and Telegram error-boundary layer rather than in the agent/session layer, while keeping existing long-message chunking intact.
