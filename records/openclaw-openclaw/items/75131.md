## Summary

Keep open. Current main has addressed the Telegram long-message chunking part, but the outbound queue still creates a fresh persisted UUID for each enqueue and the recovery permanent-error classifier still misses several reported terminal Telegram errors; the supplied open fork PR with closing syntax should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for Telegram outbound retries to reuse one logical delivery queue entry, classify terminal Telegram 400/403 errors as permanent, and pre-split long sends under Telegram limits.

## Best Possible Solution

Review, land, or replace the linked implementation candidate with a focused outbound reliability fix that carries stable logical delivery identity into the queue, adds exact Telegram terminal-error classification, preserves current Telegram chunking, and includes regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: direct sends with an idempotency key still enqueue without a queue identity and enqueueDelivery still mints a fresh UUID, while the reported Telegram terminal errors are absent from PERMANENT_ERROR_PATTERNS.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse a stable logical queue entry for retried logical sends and expand the permanent-error classifier with the exact Telegram terminal strings while keeping the already-present Telegram chunking path.
