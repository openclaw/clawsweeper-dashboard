## Summary

Keep open. Current main still leaves the reported Telegram delivery reliability cluster unresolved: permanent Telegram size/auth errors are not fully classified, live sends still allocate a new queue entry per send attempt, and one Telegram action path can still bypass the chunk-aware outbound adapter.

## What This Changes

The issue requests idempotent Telegram delivery retries, permanent-failure handling for unrecoverable Telegram 400/403 errors, and pre-splitting outbound Telegram text at the platform limit.

## Best Possible Solution

Land one focused reliability change through the existing outbound queue and Telegram send seams: preserve one queue entry per logical send, classify unrecoverable Telegram delivery failures before retrying, and route every Telegram text-send surface through the existing chunk-aware path.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: the reported Telegram permanent errors miss the classifier, live retries allocate new queue UUIDs, and the Telegram action send path can call the non-chunked default branch for long text.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented through existing queue storage/recovery and Telegram outbound/action send boundaries, with tests preventing broad dedupe or classifier regressions.
