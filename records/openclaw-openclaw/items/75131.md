## Summary

Keep open. Current main still leaves the reported bug cluster unresolved: Telegram permanent size/auth errors are not classified as permanent, each live delivery enqueue still gets a fresh queue UUID, and the Telegram plugin action path can still bypass the chunk-aware outbound adapter for long text.

## What This Changes

Fix Telegram delivery retries to be idempotent at the queue layer, classify unrecoverable Telegram 400/403 send failures as permanent, and split long Telegram sends before Telegram's message-size limit fails them.

## Best Possible Solution

Land one focused reliability fix through the existing outbound queue and Telegram send seams: add a stable retry identity for the same logical send, classify unrecoverable Telegram 400/403 failures before retrying, and make every Telegram text-send surface use the chunk-aware ≤4000/4096-character path.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: the reported Telegram permanent errors miss the classifier, repeated non-recovery sends enqueue new UUIDs, and the Telegram action path reaches the non-chunked default send branch for long markdown-mode text.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented through existing queue storage/recovery and Telegram outbound/action send seams, with care to avoid deduping intentional repeated messages.
