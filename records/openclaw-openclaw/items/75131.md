## Summary

Keep open. Current main has Telegram text chunking and recovery no longer re-enqueues during replay, but it still creates a fresh delivery-queue UUID for each live enqueue and the permanent-error classifier still misses several reported Telegram terminal errors.

## What This Changes

The issue requests idempotent delivery retries, permanent failure classification for Telegram terminal errors, and Telegram message pre-splitting around Telegram's 4096-character limit.

## Best Possible Solution

Review and land or replace the active fixes so current main gains stable logical queue IDs for live retry paths and precise permanent-error classification, while keeping the existing Telegram chunking behavior.

## Reproduction Assessment

Yes, by source inspection. Current main still generates a fresh queue UUID per enqueue and leaves the reported Telegram terminal errors out of the permanent classifier, so those remaining failures have a clear source-level reproduction path.

## Solution Assessment

Mostly yes. The requested direction is appropriate, but the maintainable path is to preserve the existing Telegram chunking implementation and focus the remaining fix on stable queue idempotency plus narrowly tested permanent-error patterns.
