## Summary

Keep open. Current main still leaves the reported Telegram delivery reliability cluster unresolved: the permanent-error matcher omits the reported Telegram size/bot/deactivated failures, live outbound sends still create a new queue entry for each non-recovery attempt, and Telegram message actions can still call the non-chunked default send path. The related #74321 was closed as a narrower duplicate of this broader issue, not as a shipped fix.

## What This Changes

The issue requests idempotent Telegram delivery retries, permanent-failure handling for unrecoverable Telegram 400/403 responses, and pre-splitting send-message text to fit Telegram message limits.

## Best Possible Solution

Land one focused reliability fix that preserves a stable retry identity for the same logical delivery attempt, classifies unrecoverable Telegram delivery errors before retry/recovery, and routes every Telegram text-send surface through the existing chunk-aware <=4000/4096-character path.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: the reported permanent Telegram errors miss isPermanentDeliveryError, repeated live send attempts call enqueueDelivery with fresh UUIDs, and Telegram action sends can reach the default non-chunked text branch for long text.

## Solution Assessment

Yes, with implementation care. The requested direction is the narrow maintainable fix, but the PR should avoid deduping legitimate duplicate sends and should keep Telegram-specific behavior at the Telegram/plugin seam or behind a generic delivery-error classifier contract.
