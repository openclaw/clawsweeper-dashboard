## Summary

Keep open. Current main still lacks stable logical delivery queue identity and still misses the reported terminal Telegram error strings in the permanent-error classifier; Telegram long-message chunking is present, but the remaining work is only partially covered by open implementation candidates.

## What This Changes

The issue requests idempotent durable queue retries, permanent classification for deterministic Telegram send failures, and pre-splitting long Telegram messages before delivery.

## Best Possible Solution

Review and land or replace the open implementation path with one focused outbound reliability change that reuses a stable logical delivery queue entry, classifies the reported deterministic Telegram failures as permanent, preserves current Telegram chunking, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: direct durable sends still enqueue through a fresh UUID path, and the reported terminal Telegram errors still miss the recovery classifier and remain retryable.

## Solution Assessment

Yes. The narrow maintainable solution is to carry a stable logical send key into durable queue storage and expand the permanent-error classifier while keeping Telegram-layer chunking in place; the linked open PRs should be reviewed or consolidated before closing this issue.
