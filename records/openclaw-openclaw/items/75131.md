Keep open. Current main still mints fresh delivery-queue UUIDs and still omits several reported terminal Telegram errors from permanent-error classification; the Telegram chunking part is already covered, and the open implementation candidate at https://github.com/haoruilee/openclaw/pull/6 references this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests idempotent outbound delivery queue retries, Telegram permanent-error classification, and 4096-character pre-splitting for send-message delivery.

## Best Possible Solution

A merged fix should route an explicit logical send key into the queue, reuse/update the same entry on retry, classify the reported terminal Telegram errors as permanent, and keep the existing Telegram chunking path covered by regression tests.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: direct sends still enqueue through a fresh UUID path, and the permanent-error classifier still omits the reported Telegram terminal strings.

## Solution Assessment

Yes. The narrow maintainable solution is to fix queue identity in outbound delivery storage, add exact terminal provider-error classification in recovery, and preserve the existing Telegram chunking behavior.
