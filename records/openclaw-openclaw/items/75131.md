## Summary

Keep open. Current main still has the queue idempotency and permanent-error classifier gaps, while Telegram long-message chunking is now present and an open implementation candidate references this issue with closing syntax.

## What This Changes

The issue asks for Telegram outbound delivery retries to reuse a logical queue entry, terminal Telegram errors to fail permanently, and long Telegram messages to be split before sending.

## Best Possible Solution

Land or replace the linked implementation with a focused outbound reliability fix that reuses a stable logical delivery queue entry, classifies the reported terminal Telegram errors as permanent, preserves current Telegram chunking, and adds regression coverage plus the maintainer-owned changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: direct durable sends still enqueue through a fresh UUID path, and reported terminal Telegram errors still miss the recovery classifier and fall through to failDelivery retry state.

## Solution Assessment

Yes. The narrow maintainable fix is to carry a stable logical send key into delivery queue storage and expand permanent-error classification while preserving Telegram-layer chunking; the linked open implementation should be reviewed or replaced before closing the issue.
