## Summary

Keep open. Current main still leaves the reported reliability cluster unresolved: Telegram permanent size/auth errors miss the permanent-failure classifier, live enqueue still creates a new queue UUID for every send attempt, and Telegram action sends can still bypass the chunk-aware outbound adapter. The related #74321 was closed as a narrower duplicate of this broader canonical issue, not as a shipped fix.

## What This Changes

Fix Telegram delivery retries to be idempotent at the queue layer, classify unrecoverable Telegram 400/403 failures as permanent, and pre-split long Telegram sends before Telegram rejects them.

## Best Possible Solution

Land one focused reliability fix through existing outbound queue and Telegram send seams: persist a stable retry identity for the same logical send, classify unrecoverable Telegram delivery failures before retrying, and make every Telegram text-send surface use the existing chunk-aware <=4000/4096-character path.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the reported Telegram permanent errors miss isPermanentDeliveryError, repeated non-recovery sends enqueue new UUIDs, and Telegram action sends reach the non-chunked markdown-mode branch for long text.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented through the existing queue storage/recovery and Telegram outbound/action send seams, with explicit care to avoid deduping legitimate duplicate sends.
