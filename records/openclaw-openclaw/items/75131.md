Fix the remaining Telegram delivery reliability gaps from https://github.com/openclaw/openclaw/issues/75131. Current main already chunks long Telegram text in extensions/telegram/src/send.ts, so preserve that behavior and do not rewrite the chunker unless a failing test proves a bypass. Implement stable queue identity/idempotency for direct logical sends so a retry/replay of the same send updates the existing delivery-queue entry instead of creating another UUID; carry an explicit idempotency key or equivalent stable queue key through sendMessage/message-action direct delivery into enqueueDelivery, keeping recovery skipQueue semantics intact. Extend delivery permanent-error classification to cover the reported Telegram terminal failures, including '400 message is too long', '400 chat not found', '400 bot was blocked by the user', '403 Forbidden: bot can't send messages to bots', and '403 user is deactivated', while leaving transient 429/5xx/network errors retryable. Add focused regression tests for queue idempotency/update behavior, recovery moving these Telegram errors to failed/, and direct Telegram action/core send long-text chunk coverage if not already covered. Add a user-facing CHANGELOG.md fix entry. Do not move Telegram-specific policy into unrelated core defaults beyond the existing generic classifier seam.
