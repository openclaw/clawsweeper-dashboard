## Summary

Keep open: current main still lacks a stable logical queue key for retries and still misses the reported deterministic Telegram permanent-error strings; Telegram long-text handling is partially addressed through the rich-message chunker but does not close the whole delivery-queue report.

## What This Changes

The issue requests idempotent delivery retries, permanent failure classification for deterministic Telegram 400/403 errors, and pre-splitting long Telegram sends before platform rejection.

## Best Possible Solution

Review and land, adopt, or replace the open logical-queue-id PR, fold in the deterministic-error classifier coverage, and decide whether the current rich-message chunking fully supersedes the 4,096-character split request.

## Reproduction Assessment

Yes for the current-main source path: queue entries still get fresh UUIDs and the permanent classifier still misses several reported Telegram terminal errors. I did not run a live Telegram repro, and the chunking subclaim is only partially source-proven because current rich-message sends chunk at 32,768 characters.

## Solution Assessment

Keeping this open is the best current outcome because an open closing PR owns the idempotency repair, while current main still leaves meaningful classifier and split-limit questions. A duplicate ClawSweeper fix PR would risk racing the linked implementation path.
