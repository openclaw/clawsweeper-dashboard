## Summary

Keep open. Current main still lacks the remaining queue idempotency and Telegram permanent-error classification fixes, while Telegram long-message chunking is already present and an open linked implementation candidate should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for Telegram outbound delivery to avoid duplicate queue entries on retry, classify terminal Telegram failures as permanent, and avoid oversized Telegram sends.

## Best Possible Solution

Review and land or replace haoruilee/openclaw#6 with a focused outbound reliability fix that adds stable queue identity, exact Telegram permanent-error classification, regression tests, and a changelog entry while preserving existing Telegram chunking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: direct outbound sends still enqueue through a fresh UUID path, and recovery still treats several reported Telegram terminal errors as transient because the classifier omits those strings.

## Solution Assessment

Yes. The narrow maintainable fix is to carry a stable logical delivery key into the queue layer, update the existing queue entry on failed attempts, and add exact terminal Telegram patterns while leaving the already-present Telegram chunking path intact.
