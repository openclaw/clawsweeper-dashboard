## Label Justifications

- `P1`: The report describes a source-reproducible Telegram delivery workflow that can jam retries and saturate the gateway event loop for real users.
- `impact:message-loss`: The affected outbound delivery path can leave Telegram messages failed, replayed, duplicated, or stuck in retry recovery.

## Summary

Keep open: current main and v2026.6.8 still create fresh outbound queue IDs and still miss several reported Telegram permanent-error strings. Open linked PRs cover active repair paths, while the 4096-character split part is partly covered by current Telegram chunking and needs reconciliation rather than a duplicate splitter.

## What This Changes

The issue asks OpenClaw to make outbound delivery retries idempotent, permanently fail deterministic Telegram delivery errors, and ensure Telegram text is split below Telegram's 4096-character limit.

## Best Possible Solution

Review and land, adopt, or replace the linked idempotency and classifier PRs, while treating Telegram long-text splitting as already mostly covered unless a fresh 4096-specific failure reproduces.

## Reproduction Assessment

Yes, by source inspection: current main still creates fresh queue IDs on enqueue and the classifier lacks several reported permanent Telegram error strings. I did not run a live Telegram account reproduction in this read-only review.

## Solution Assessment

No final fix is on current main yet. The best path is maintainer review of the linked open PRs rather than closing this issue or spawning a competing repair; the Telegram split portion should be revalidated against current chunking before more implementation work.
