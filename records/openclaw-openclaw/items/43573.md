## Summary

Keep open: current main still lacks immediate ACP spawn task seeding and this PR addresses a valid bug, but the branch is stale/unmergeable, carries unrelated CI workflow changes, and has a correctness issue where `taskDelivered` is true even when seed persistence failed.

## What This Changes

The PR seeds ACP `sessions_spawn` transcripts with the initial task, forwards input provenance through dispatch, dedupes the first replayed prompt, adjusts prompt-only session-manager normalization, adds tests/changelog, and changes several CI workflows to reconstruct PR merge results.

## Best Possible Solution

Land a narrowed ACP transcript seed implementation on current main: persist the task before dispatch with provenance, dedupe only the first seeded replay, report delivery status truthfully, and split unrelated CI workflow work.

## Reproduction Assessment

Yes, source-reproducible. Current main dispatches `params.task` to ACP without seeding the transcript before spawn returns, while the later ACP transcript write happens only after the turn completes; the discussion also includes immediate-history sampling that hit empty histories.

## Solution Assessment

No. The ACP seeding/dedupe direction is right, but this PR should be rebased or replaced, narrowed to the ACP surface, and fixed so delivery confirmation is only true after the seed write succeeds.
