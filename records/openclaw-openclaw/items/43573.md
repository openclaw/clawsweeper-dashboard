## Summary

Keep open. The PR targets a real ACP `sessions_spawn` history gap that is still visible on current main, but the branch should not merge as-is because it can report `taskDelivered: true` after best-effort transcript seeding fails, its changelog entry misses contributor credit, and it mixes the ACP fix with unrelated workflow/token changes that need separate maintainer review.

## What This Changes

The PR seeds ACP `sessions_spawn` transcripts with the initial task, forwards input provenance into the first ACP dispatch, dedupes the later replayed prompt, adjusts prompt-only SessionManager handling, adds regression tests and a changelog entry, and changes several CI/labeler workflow files.

## Best Possible Solution

Land a narrow ACP-only fix on current main that writes the initial spawn task to the child transcript before returning delivery confirmation, passes provenance through the matching dispatch, dedupes only the first seeded replay by marker/provenance, preserves prompt-only transcript normalization, adds focused regression coverage, and moves any CI/token workflow changes to a separate maintainer-owned review.

## Reproduction Assessment

Yes. Spawn an ACP session with `sessions_spawn(runtime: "acp", task: ...)`, inspect `sessions_history` or `chat.history` before the first assistant reply, and current main shows that immediate history reads the transcript while the spawn path only dispatches the task and persists transcript text later.

## Solution Assessment

No as-is. The seed-and-dedupe direction is the right product fix, but the PR should be rebased onto current main, narrowed to ACP transcript behavior, report delivery only after the seed is actually persisted, and fix the changelog credit before merge.
