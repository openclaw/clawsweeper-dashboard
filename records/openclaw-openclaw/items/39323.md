## Summary

Keep open: current main still lacks proactive pairing-request owner notifications, and this PR is the canonical remaining implementation path, but the branch is stale against current hook, gateway startup, and outbound-delivery seams and still lacks real behavior proof.

## What This Changes

The PR adds a root `pairing.notify` config block, emits a `pairing:request` internal hook for new pending pairing requests, registers owner notification dispatch at gateway startup, and adds notification tests.

## Best Possible Solution

Land a rebased, documented implementation that emits the current hook event, registers in the active post-attach gateway startup path, routes through the channel-keyed outbound adapter/deps path with target normalization, and includes redacted real notification proof.

## Reproduction Assessment

## Solution Assessment

No, not as-is. The direction is plausible, but the implementation needs a current-main rebase, active hook/startup/outbound seams, docs, target normalization, and real behavior proof before it is the maintainable solution.
