## Summary

Keep open: current main still lacks proactive owner notification for pairing requests, but this PR is stale against the current hook/startup split and still has outbound routing, Discord targeting, and docs blockers before it can merge.

## What This Changes

The PR adds a `pairing.notify` config surface, a `pairing:request` internal hook, owner notification dispatch, gateway registration, and tests.

## Best Possible Solution

Land a rebased, documented, maintainer-reviewed implementation that uses the current hook type module, registers in the active post-attach sidecar path, routes notification delivery through a generic outbound channel seam, normalizes channel-specific targets where needed, and keeps notification text sanitized and outside agent context.

Not applicable as a bug reproduction; this is a feature PR. Source inspection confirms current main does not implement owner notifications, and the Discord blocker is source-backed by the current bare-numeric recipient rejection path.

## Solution Assessment

No, not as-is. The feature direction is reasonable, but the branch needs a current-main rebase plus hook/startup/outbound/docs fixes before it is the narrowest maintainable solution.
