## Summary

## What This Changes

The PR adds a `pairing.notify` config block, a `pairing:request` internal hook fired from channel pairing creation, owner-notification dispatch, gateway startup registration, and pairing notification tests.

## Best Possible Solution

Land a rebased, documented, maintainer-reviewed implementation that emits a current hook event, registers in the active gateway sidecar path, uses the channel-id keyed outbound seam, normalizes channel-specific targets, and keeps requester text sanitized outside agent context.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source inspection on current main gives a high-confidence behavior check: the requested `pairing.notify`/`pairing:request` owner notification surface is absent and pairing request creation only persists pending requests.

## Solution Assessment

No, not as-is. The direction is plausible, but the branch must be rebased to the current hook, gateway startup, and outbound seams, add docs and Discord target handling, and include real behavior proof before it is the narrowest maintainable solution.
