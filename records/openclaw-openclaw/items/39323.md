## Summary

Keep open: current main still lacks proactive owner notifications for new pairing requests, but this PR is not merge-ready because it needs maintainer design approval, a rebase onto the current hook/startup/outbound seams, and after-fix real behavior proof.

The PR adds a `pairing.notify` config block, a `pairing:request` hook, owner-notification sending logic, gateway registration, and tests for new pairing request notifications.

## Best Possible Solution

Land a maintainer-approved design that emits a typed pairing-request event from the current pairing-store path, registers notification handling in post-attach startup, routes through the generic outbound adapter path, documents the config, and includes redacted real-channel proof.

## Solution Assessment

No, not as currently implemented. The direction is plausible, but the branch targets stale hook/startup seams and bypasses the current outbound adapter contract, so the safer path is rebase plus maintainer design approval and live proof.
