## Summary

Keep open: current main still lacks proactive owner notification for pairing requests, and the related issue was closed in favor of this PR rather than because the behavior shipped. The PR remains useful as the canonical feature discussion, but the current branch is stale/conflicting, misses real behavior proof, and targets outdated hook/startup/outbound seams.

## What This Changes

The PR adds owner notifications for new pairing requests through a new `pairing.notify` config block, a `pairing:request` internal hook, gateway registration, channel sending logic, and notifier tests.

## Best Possible Solution

Land a maintainer-approved design that emits a typed pairing-request event from the current pairing-store path, registers notification handling in post-attach gateway startup, routes through generic outbound adapters, documents the config, and includes redacted real-channel proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Current source and docs show manual pending-request listing/approval but no proactive owner notification on request creation.

## Solution Assessment

No, not as currently implemented. The direction is plausible, but the branch targets stale hook/startup seams and bypasses the current outbound adapter contract, so the safer path is a rebase onto current contracts plus maintainer design approval and real-channel proof.
