## Summary

Keep this PR open: current main still lacks proactive owner notifications for DM pairing requests, and the closed companion issue points here as the canonical work, but the branch needs real behavior proof plus rebase/design fixes before merge.

## What This Changes

The PR adds a `pairing.notify` config block, emits a `pairing:request` hook when new pairing requests are stored, registers an owner notifier at gateway startup, sends notifications through configured channels, and adds tests.

## Best Possible Solution

Land a maintainer-approved design that emits a typed pairing-request event from the current pairing-store path, registers notification handling in post-attach startup, routes through channel-ID outbound adapters, documents `pairing.notify`, and includes redacted real-channel proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source and docs give a high-confidence current-behavior check: current main records pending pairing requests and documents manual approval, but does not proactively notify the owner when requests are created.

## Solution Assessment

No, not as currently implemented. The direction is useful, but the branch needs to rebase onto the current hook/startup/outbound contracts and get maintainer approval for the new config and notification security boundary.
