## Summary

Keep this PR open. Current main still lacks proactive DM-pairing owner notification, but the branch is not merge-ready because it is stale against the current gateway startup layout and still has a concrete Discord target-routing bug.

## What This Changes

The PR adds a `pairing.notify` config/schema surface, a `pairing:request` internal hook emitted from new DM pairing requests, owner-notification dispatch code and tests, and gateway startup registration.

## Best Possible Solution

Land a rebased, documented, maintainer-reviewed implementation that adds a versioned internal or plugin pairing-request hook emitted only for fresh DM pairing requests, then sends owner notifications with fixed-format sanitized content, normalized per-channel targets, PII-safe logs, schema/docs alignment, and focused tests around the shared pairing path.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction; this is a feature PR. Current-main absence is high-confidence by source inspection, and the Discord target defect is reproducible from the current `sendMessageDiscord` recipient parsing path and existing bare-numeric rejection test.

## Solution Assessment

No, not as-is. The feature direction is appropriate, but the current branch should be rebased onto `server-startup-post-attach.ts`, normalize Discord notification targets, and then rerun focused pairing/outbound validation before it is the best maintainable fix.

After rebasing, place this registration in the active `startGatewaySidecars` path or the hook will not be registered at gateway startup.
  - confidence: 0.78
