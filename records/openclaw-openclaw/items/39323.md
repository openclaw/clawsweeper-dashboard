## Summary

Keep open: current main still lacks proactive owner notifications for new DM pairing requests, and the linked feature issue was already closed as superseded by this PR. The branch is still not mergeable as-is because it targets stale hook/startup seams, bypasses the current outbound adapter contract, and lacks contributor real behavior proof.

## What This Changes

The PR adds a `pairing.notify` config block, a `pairing:request` hook, owner-notification sending logic, gateway registration, and tests for new pairing request notifications.

## Best Possible Solution

Land a maintainer-approved design that emits a typed pairing-request event from the current pairing-store path, registers notification handling in the active post-attach startup seam, routes delivery through the generic outbound adapter path, documents the config, and includes redacted real-channel proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source and docs inspection on current main show manual pending-request listing/approval but no proactive owner notification on request creation.

## Solution Assessment

No, not as currently implemented. The direction is plausible, but the branch targets stale hook/startup seams and duplicates outbound delivery instead of using the current channel adapter contract.
