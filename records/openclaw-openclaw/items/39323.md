## Summary

Keep open: current main still lacks proactive owner notification for new pairing requests, but this PR is not merge-ready because it targets stale hook/startup/outbound seams and still lacks required real behavior proof.

## What This Changes

The PR adds `pairing.notify`, a `pairing:request` internal hook from new pairing requests, owner notification dispatch through configured messaging channels, and tests.

## Best Possible Solution

Land a rebased, documented implementation that emits the active hook event from the current pairing-store path, registers in post-attach startup, routes through the channel-keyed outbound adapter with target normalization, and includes redacted live notification proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Source inspection on current main shows pending requests are stored and listed manually, with no proactive owner-notification config or hook.

## Solution Assessment

No, not as currently implemented. The direction is plausible, but the branch needs rebasing onto current hooks/startup/outbound seams, documentation and target normalization, plus real runtime proof before it is the maintainable solution.
