## Summary

Keep open: current main still does not implement proactive owner notifications for new DM pairing requests, but this PR should not merge as-is because it is stale against the current gateway startup split and still forwards Discord notification targets in a form the current Discord sender rejects.

## What This Changes

The PR adds a `pairing.notify` config/schema surface, a `pairing:request` internal hook emitted from new pairing requests, owner-notification dispatch code and tests, and gateway startup registration.

## Best Possible Solution

Land a rebased, documented, maintainer-reviewed implementation that emits a fresh-request-only pairing event from the shared DM pairing path, registers notification delivery in the active gateway sidecar startup path, normalizes per-channel notification targets, keeps notification text fixed-format and sanitized, masks owner PII in logs, and updates docs/changelog with focused pairing and outbound-channel tests.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction; this is a feature PR. Current-main absence is source-backed, and the Discord defect is reproducible from the current `sendMessageDiscord` recipient parsing path plus the existing bare-numeric rejection test.

## Solution Assessment

No, not as-is. The feature direction is reasonable, but the current branch needs a rebase onto the post-attach startup path, Discord target normalization, and docs/changelog follow-through before it is the narrowest maintainable solution.
