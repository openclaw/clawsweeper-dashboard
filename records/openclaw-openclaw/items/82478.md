## Summary

Keep open: the patch has blocking correctness issues, does not actually avoid `reload()` on cache hits, has broken TTL test code, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an in-memory cache for embedded Pi `DefaultResourceLoader` instances keyed by workspace and agent directory, with TTL/invalidation helpers, tests, and reload markers in the attempt and compaction paths.

## Best Possible Solution

Rework the fix so stable discovery results can be reused without reusing per-run `settingsManager` or inline extension state, and prove repeated embedded runs with redacted after-fix timing logs.

## Reproduction Assessment

Unclear for live reproduction: the PR body and related issue include credible timing logs, and source inspection confirms current main awaits upstream synchronous reload. This review did not reproduce the 5-9s stall in a real embedded run.

## Solution Assessment

No: this PR caches a mutable loader across per-run settings while still calling `reload()` on cache hits. A safer solution should cache only stable discovery work or return explicit reload state while rebuilding run-specific settings and extension factories.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Make the TTL test callback async:** `src/agents/pi-embedded-runner/resource-loader.test.ts:227-240`
  - body: This new test uses `await` inside a non-async `it` callback, so the file will not parse and the branch cannot rely on this coverage. Mark the callback async or switch to fake timers before merge.
  - confidence: 0.99
- **[P2] Align the TTL test with the minimum clamp:** `src/agents/pi-embedded-runner/resource-loader.test.ts:229-242`
  - body: `OPENCLAW_RESOURCE_LOADER_CACHE_TTL_MS` is set to `100`, but `resolveCacheTtlMs()` clamps positive values to at least 10,000 ms. Waiting 150 ms will not expire the entry, so this expectation is wrong even after the callback is made async.
  - confidence: 0.98
- **[P2] Skip reloads on cache hits:** `src/agents/pi-embedded-runner/resource-loader.ts:132-134`
  - body: Returning a cached loader still sends both callers through `await resourceLoader.reload()`, and upstream `reload()` immediately calls `packageManager.resolve()`. Cache hits therefore keep paying the synchronous discovery cost this PR is meant to remove.
  - confidence: 0.97
- **[P2] Keep per-run loader inputs fresh:** `src/agents/pi-embedded-runner/resource-loader.ts:132-134`
  - body: The cached loader was constructed with the first run's `settingsManager` and `extensionFactories`, but later calls with the same workspace and agent dir can have different compaction/context-pruning config. Reusing that loader can omit newly required inline extensions or reload resources from stale settings.
  - confidence: 0.93
