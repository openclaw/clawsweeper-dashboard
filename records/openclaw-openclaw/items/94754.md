## Summary

Keep open: the PR is a useful partial Telegram fix, but it still misses marker-derived silent delivery across media fan-out and native Telegram replies, and the supplied proof is mocked rather than live Telegram proof.

## What This Changes

The PR parses a trailing `notify=false` marker in Telegram text sends, strips it from delivered text, maps it to `disable_notification`, and adds unit coverage for basic text sends.

## Best Possible Solution

Normalize standalone trailing `notify=false` before Telegram payload fan-out or carry the derived silent flag through every affected Telegram send path, then prove marker removal and silent delivery with redacted live Telegram evidence.

## Reproduction Assessment

Yes, at source level: current main can carry fallback text ending in `notify=false`, while Telegram delivery only maps structured `silent` metadata to `disable_notification`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Parsing only inside `sendMessageTelegram` is too low in the stack for batch-level silent intent and does not cover native replies; the safer fix normalizes once before Telegram fan-out or shares the same normalizer across both reply implementations.
