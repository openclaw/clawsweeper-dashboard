## Summary

Keep open. Current main still has the live-PID orphan-lock failure mode for transcript locks: a fresh lock owned by a still-running process is not stale until age exceeds the 30-minute default, and embedded runs can still request a very large watchdog hold budget.

## What This Changes

The issue asks OpenClaw to recover session write locks leaked by a still-running Gateway process without leaving a session blocked for the stale-lock window.

## Best Possible Solution

Keep this as the focused live-PID session-lock recovery bug and implement an explicit progress/lease or bounded recovery path that cannot unlock genuinely active transcript writes; use #13744 for broader configurability and lock-scope follow-up.

## Reproduction Assessment

Yes. A source-level reproduction is to create a fresh .jsonl.lock payload with a live Gateway/current PID while no matching in-process HELD_LOCKS entry exists, then contend for the same session file; current inspect/reclaim logic treats it as non-stale until staleMs expires and the caller times out.

## Solution Assessment

No. Current main contains useful mitigations, but it still relies on age or a potentially huge watchdog hold budget rather than a bounded live-PID lease/progress signal for abandoned lock contexts.
