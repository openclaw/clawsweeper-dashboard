## Summary

Keep open. Current main still allows a fresh lock file owned by a live PID to block contenders until timeout, and embedded transcript locks can still receive a very large watchdog budget derived from run timeout, so the live-Gateway abandoned-lock failure class is not fully fixed.

## What This Changes

The issue requests robust recovery for leaked session write locks when a Gateway process stays alive but the lock-holding async context stops making progress.

## Best Possible Solution

Add a bounded live-PID progress signal or hard recovery budget for session transcript locks, with regression coverage for a lock file owned by a still-running process but absent from the in-process held-lock map. Keep #13744 for broader configurability and lock-scope work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a recent `.jsonl.lock` payload with a live PID and valid start time while no matching `HELD_LOCKS` entry exists, then call `acquireSessionWriteLock`; current inspection logic treats it as non-stale and waits until the acquisition timeout.

## Solution Assessment

Unclear. A heartbeat lease is a robust direction, but the narrowest maintainable first fix may be a bounded max-hold/progress policy plus explicit orphan recovery; maintainers should choose the safety bound before a code PR changes reclaim semantics.
