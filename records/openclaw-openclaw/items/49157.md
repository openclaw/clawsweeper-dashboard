## Summary

Keep open. Current main has useful partial mitigations, but the live-PID same-process lock-leak class is still not fully bounded because active held locks can keep `HELD_LOCKS` populated and avoid orphan-self reclaim until the watchdog budget or 30-minute stale age is reached; #13744 is related broader configurability/scope work, not a terminal duplicate.

## What This Changes

The issue asks OpenClaw to recover session transcript write locks leaked by a still-running Gateway process before they can deadlock session operations for tens of minutes.

## Best Possible Solution

Keep this as the focused live-PID session-lock recovery bug: add a bounded progress/lease or equivalent abandoned-context recovery mechanism for transcript locks, while leaving broader lock configurability and scope work to #13744.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to acquire a session lock with a live PID and large `maxHoldMs`, abandon the release path while keeping the process alive, then attempt a second acquisition and observe that the existing live lock is not reclaimed before acquire timeout.

## Solution Assessment

No. Current main is a partial mitigation, not the best complete solution, because recovery still depends on 30-minute stale age or a potentially huge watchdog budget rather than a bounded live-PID progress signal.
