## Summary

Keep open. Current main still has the reported live-PID orphan-lock failure mode: a fresh lock file owned by the still-running Gateway PID is treated as valid until the 30-minute stale age or an oversized in-process watchdog budget expires, and the recent cleanup work does not add a bounded progress/lease signal for an abandoned lock context.

## What This Changes

The issue asks OpenClaw to recover leaked session write locks when a long-running Gateway process remains alive but the lock-holding operation has stopped releasing the `.jsonl.lock` file.

## Best Possible Solution

Keep this issue as the focused live-PID session-lock recovery bug. The maintainable end state is a bounded progress signal or tighter critical-section design that reclaims abandoned locks without unlocking active transcript writes, with #13744 carrying broader configurability and lock-scope follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a current-format `.jsonl.lock` payload with the current/Gateway PID, matching process start time, and a recent `createdAt`, while no matching in-process `HELD_LOCKS` entry exists; current main treats that lock as non-stale and a contending acquire waits until its timeout.

## Solution Assessment

No. Current main has useful partial mitigations for exit cleanup, PID recycling, old-format self locks, and failover classification, but it still lacks the bounded live-PID lease/progress recovery needed for the reported abandoned-lock class.
