## Summary

Keep open. Current main has provider usage snapshots, provider-owned usage hooks, a gateway usage RPC, and Overview/status usage hints, but it still lacks the requested manual Control UI fuel gauge at the run/model decision point and the requested source, confidence, stale, and unavailable semantics.

## What This Changes

Request a manual Control UI provider-capacity gauge that refreshes provider-reported remaining limits and reset windows before starting expensive runs.

## Best Possible Solution

Define a normalized provider-capacity metadata contract, keep provider-specific fetching in provider-owned usage hooks, and render a manual refresh affordance near the selected model/send controls once maintainers choose the UI surface.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence absence check: usage data exists in gateway/status and Overview surfaces, but no manual decision-point capacity gauge exists in the chat/run toolbar.

## Solution Assessment

Unclear. The existing provider usage hooks and gateway methods are the right foundation, but the metadata contract, reliability semantics, and Control UI placement need maintainer direction before a narrow implementation is clearly the best fix.
