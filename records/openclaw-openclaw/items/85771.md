## Summary

Keep open: current main still appends a renderable final assistant payload to the WebChat message state without checking whether the same assistant turn is already the visible history tail, and the only matching fix PR is closed unmerged.

## What This Changes

The issue reports an intermittent WebChat frontend duplicate assistant bubble after a final `chat.status` event while the backend transcript contains only one assistant message.

## Best Possible Solution

Add a narrow tail-signature de-duplication guard for final assistant payloads in `handleChatEvent`, with regression tests for history-before-final and legitimate repeated assistant text separated by a user turn.

## Reproduction Assessment

Yes at source level, though not live in a browser: if `chatMessages` already ends with the persisted assistant reply and `handleChatEvent` receives a matching active-run `final` payload, current main appends the same assistant message again.

## Solution Assessment

Yes, the best fix is a controller-level duplicate-tail guard around final assistant payload appends, not backend transcript changes, because the report and docs distinguish live delivery projection from the durable JSONL transcript.
