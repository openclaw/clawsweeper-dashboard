## Summary

Keep open: current main still has a WhatsApp directory gap. The WhatsApp group directory is config-backed today, while live participating-group metadata is only hydrated inside the inbound runtime path and is not exposed through a live directory adapter.

## What This Changes

The issue asks for `openclaw directory groups list --channel whatsapp` to discover groups from the linked WhatsApp account instead of returning only configured group allowlist entries.

## Best Possible Solution

Add a WhatsApp-owned live `listGroupsLive` adapter backed by account-scoped runtime metadata or a clearly defined Baileys query path, while keeping the config-backed group list as the offline fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI chooses `listGroupsLive ?? listGroups`; WhatsApp only defines `listGroups`, and that lister only returns configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but the maintainable fix should use the existing live-directory contract and a maintainer-chosen WhatsApp runtime/cache boundary instead of adding Baileys calls to the config helper.
