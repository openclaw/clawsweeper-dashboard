## Summary

Keep open: current main still has the reported WhatsApp directory gap, but solving it adds live WhatsApp group discovery rather than fixing a behavior already promised by the current config-backed directory contract.

## What This Changes

The issue requests that `openclaw directory groups list --channel whatsapp` discover WhatsApp groups the linked account belongs to before those group JIDs are configured.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter backed by a clear account-scoped live query or runtime metadata cache, while keeping the config-backed lister as the offline fallback and documenting the behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI selects `listGroupsLive ?? listGroups`; WhatsApp only defines `listGroups`, and that implementation reads configured `channels.whatsapp.groups` keys, so empty or omitted config yields no discoverable groups.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the maintainable fix should use the existing live-directory contract and a maintainer-chosen WhatsApp runtime/cache boundary instead of adding provider calls to the config helper.
