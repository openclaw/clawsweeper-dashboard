## Summary

Keep open: current main still has the WhatsApp directory gap. The CLI can prefer a live group reader, but WhatsApp registers only the config-backed reader; live participating-group metadata is hydrated in the inbound runtime and is not exposed as directory output.

## What This Changes

Request live WhatsApp group discovery from `openclaw directory groups list --channel whatsapp` instead of returning only configured `channels.whatsapp.groups` entries.

## Best Possible Solution

Add a WhatsApp-owned live group directory source backed by active listener metadata or a documented Baileys query path, with config-backed listing retained as the offline fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI falls back to WhatsApp's config-backed `listGroups`, which reads `channels.whatsapp.groups` keys and returns no entries when the map is empty.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the maintainable path should use a WhatsApp-owned live directory/runtime boundary rather than changing the generic config helper.
