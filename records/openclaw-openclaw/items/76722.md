## Summary

Keep open: current main still has a WhatsApp directory gap. WhatsApp group listing is config-backed, while live participating-group metadata is only hydrated inside the inbound runtime path and is not exposed through a live directory adapter.

## What This Changes

The issue asks for `openclaw directory groups list --channel whatsapp` to discover the linked WhatsApp account's joined groups instead of returning only configured group allowlist keys.

## Best Possible Solution

Add a WhatsApp-owned live group directory adapter backed by an explicit runtime/cache boundary, with the current config-backed list retained as the offline fallback.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the CLI would use `listGroupsLive`, but WhatsApp only provides config-backed `listGroups`, which enumerates `channels.whatsapp.groups` keys. I did not run a live WhatsApp account reproduction in this read-only pass.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but maintainers should choose the live-directory/runtime-cache/privacy boundary before implementation rather than adding an ad hoc Baileys call in the config helper.
