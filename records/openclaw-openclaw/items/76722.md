## Summary

Keep open: current main still has the WhatsApp directory gap. The command falls back to the config-backed group allowlist, while live WhatsApp group membership is only hydrated inside runtime metadata and is not exposed through the directory adapter.

## What This Changes

The issue asks for `openclaw directory groups list --channel whatsapp` to discover WhatsApp groups the linked account belongs to instead of returning only configured group JIDs.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` implementation backed by the active runtime or gateway-owned metadata cache, keep config-backed entries as the offline fallback, and update WhatsApp directory tests and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw directory groups list --channel whatsapp` selects the WhatsApp `listGroups` fallback, which only returns configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but the maintainable fix should use the existing live-directory contract and WhatsApp runtime/cache ownership rather than calling Baileys from the config helper.
