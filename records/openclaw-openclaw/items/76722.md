## Summary

Keep open: current main still has a WhatsApp directory gap. The observed empty result is explained by source and docs: WhatsApp group listing is config-backed today, while live Baileys group membership is hydrated only inside runtime metadata and is not exposed through the directory adapter.

## What This Changes

Requests that WhatsApp group directory listing discover groups from the linked account instead of only returning `channels.whatsapp.groups` config keys.

## Best Possible Solution

Add a WhatsApp-owned live group directory adapter that reads participating groups from the active runtime or gateway-owned cache, includes group names when available, and keeps config-backed group entries as the offline fallback.

## Reproduction Assessment

Yes. Source inspection reproduces the reported behavior: `openclaw directory groups list --channel whatsapp` falls back to the config-backed `listGroups` path, and that path can only return configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but the narrow maintainable solution should use the existing `listGroupsLive` directory contract and WhatsApp runtime/cache ownership instead of calling Baileys from the config directory helper.
