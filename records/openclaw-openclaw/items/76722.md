## Summary

Keep open: current main still exposes only config-backed WhatsApp group directory entries, while the CLI already has a live-directory seam that WhatsApp does not implement. Docs and tests confirm this is current behavior, so this should not be closed as cleanup.

## What This Changes

Request live WhatsApp group discovery for `openclaw directory groups list --channel whatsapp` so a linked account can find group JIDs before configuring allowlists.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter that returns participating group JIDs and names from the active runtime or gateway-owned cache, while preserving config-backed groups as the offline fallback.

## Reproduction Assessment

Yes. Source inspection shows `openclaw directory groups list --channel whatsapp` falls back to WhatsApp `listGroups`, and that implementation can only return configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the suggested fix should probably use the existing `listGroupsLive` directory contract and WhatsApp runtime/cache ownership rather than putting Baileys calls in the config helper.
