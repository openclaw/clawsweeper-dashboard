## Summary

Keep open: current main still has a WhatsApp group directory gap. The command path can use live directory adapters, but WhatsApp only registers a config-backed group lister, so an unconfigured account can still return no groups even when the linked WhatsApp account belongs to groups.

## What This Changes

Add a way for `openclaw directory groups list --channel whatsapp` to discover WhatsApp groups the linked account belongs to instead of returning only configured allowlist keys.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` implementation backed by account-scoped active socket state or a documented group metadata cache, while keeping the existing config-backed lister as the offline fallback.

## Reproduction Assessment

Yes, source-reproducible. With empty `channels.whatsapp.groups`, the helper reads no map keys and the CLI falls back to WhatsApp's config-backed `listGroups` because no `listGroupsLive` is registered.

## Solution Assessment

Unclear. Live discovery is likely the right user outcome, but adding Baileys calls directly to the config helper would cross ownership boundaries; the narrower maintainable path is a WhatsApp-owned live directory adapter plus config fallback.
