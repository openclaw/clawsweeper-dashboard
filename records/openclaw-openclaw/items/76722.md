## Summary

Keep open: current main still only lists configured WhatsApp group allowlist keys, so the requested live group membership directory remains missing. This is a feature/UX gap rather than a strict regression because current docs describe WhatsApp group IDs as log/config-backed, not dynamically discoverable.

## What This Changes

The issue requests that `openclaw directory groups list --channel whatsapp` discover groups the linked WhatsApp account belongs to instead of returning only preconfigured group IDs.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter backed by account-scoped runtime metadata or a clearly defined Baileys query path, keep the config-backed lister as the offline fallback, and update tests/docs for empty-config discovery behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI would use `listGroupsLive` if WhatsApp had one, but WhatsApp only registers the config-backed `listGroups`, which returns configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but the maintainable fix should use a WhatsApp-owned live directory/runtime seam rather than adding provider calls inside the config helper.
