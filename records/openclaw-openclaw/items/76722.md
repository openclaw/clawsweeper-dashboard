## Summary

Keep open: current main still lacks a WhatsApp live group directory, so the command can only list configured group allowlist keys while participating WhatsApp groups remain available only inside runtime metadata hydration.

## What This Changes

Expose WhatsApp group membership through `openclaw directory groups list --channel whatsapp` instead of returning only configured `channels.whatsapp.groups` keys.

## Best Possible Solution

Add a WhatsApp-owned live group directory adapter that reads participating groups from active runtime or gateway-owned cached metadata, includes group names when available, and keeps config-backed entries as the offline fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with no configured WhatsApp group keys, the plugin exposes only config-backed `listGroups`, so the CLI receives an empty list even though runtime may have hydrated participating groups.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the maintainable fix should use the existing `listGroupsLive` directory contract and WhatsApp runtime/cache ownership rather than calling Baileys from the config helper.
