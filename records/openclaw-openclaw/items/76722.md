## Summary

Keep open: current main and the latest release still expose WhatsApp group directory results from configured group allowlist keys only, while the existing Baileys participating-group fetch is internal runtime metadata and not wired to `openclaw directory groups list`.

## What This Changes

The issue requests dynamic WhatsApp group directory discovery so users can list groups the linked account belongs to before configuring group access.

## Best Possible Solution

Add a WhatsApp-owned live directory path that lists participating groups from the active WhatsApp runtime or gateway-owned cache, includes group names when available, and preserves config-backed allowlist behavior as the fallback.

## Reproduction Assessment

Yes, source-level: current main routes WhatsApp group directory listing to `channels.whatsapp.groups` map keys only, so account membership cannot affect the result. I did not perform a live WhatsApp account test in this read-only review.

## Solution Assessment

Unclear: dynamic discovery is likely the right user outcome, but the maintainable implementation should use the existing live-directory seam and gateway-owned WhatsApp runtime/cache rather than adding provider calls to the config helper without owner review.
