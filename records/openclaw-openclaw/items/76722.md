## Summary

Keep open: current main still cannot discover WhatsApp group memberships; the command falls back to a config-backed list and no canonical duplicate or closing PR exists. This needs owner/product judgment because exposing live account group membership crosses runtime state and privacy boundaries.

## What This Changes

Expose WhatsApp group membership through `openclaw directory groups list --channel whatsapp` instead of only returning configured group allowlist keys.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter backed by account-scoped Baileys group metadata or a clearly owned runtime cache, while keeping the config-backed group list as the offline fallback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with no configured `channels.whatsapp.groups` keys, the WhatsApp directory adapter can only return an empty config-backed list. I did not run a live WhatsApp account, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. Dynamic discovery likely matches the desired user outcome, but the maintainable solution should use the existing live-directory contract with a maintainer-chosen WhatsApp runtime/cache and privacy boundary instead of adding ad hoc Baileys calls to the config helper.
