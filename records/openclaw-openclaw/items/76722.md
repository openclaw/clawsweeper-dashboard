## Summary

Keep open: current main and the latest release still expose WhatsApp groups through the config-backed allowlist path only, while live participating-group metadata is hydrated inside the inbound runtime and not surfaced through a directory adapter.

## What This Changes

The issue asks for WhatsApp group directory discovery to return groups the linked account belongs to instead of only configured `channels.whatsapp.groups` allowlist entries.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter backed by an account-scoped runtime metadata cache or an explicit Baileys query path, keep the config-backed lister as the offline fallback, and document the discovery behavior.

## Reproduction Assessment

Yes, by source inspection. On current main, `openclaw directory groups list --channel whatsapp` falls back to WhatsApp's config-backed `listGroups`, which can only return keys already present in `channels.whatsapp.groups`.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the best implementation should use the existing live-directory contract and a maintainer-chosen WhatsApp runtime/cache boundary rather than adding live provider calls to the generic config helper.
