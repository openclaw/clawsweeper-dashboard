## Summary

Keep open: current main still has the WhatsApp group directory gap. Source, docs, and tests show the command is config-backed today, while live Baileys group membership is hydrated only inside the WhatsApp runtime path and is not exposed through the directory adapter.

## What This Changes

The issue asks for WhatsApp group directory listing to discover groups the linked account belongs to instead of only returning configured group JID keys.

## Best Possible Solution

Add a WhatsApp-owned live group directory adapter that reads participating groups from the active runtime or a gateway-owned cache, includes group names when available, and preserves config-backed entries as the offline fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `openclaw directory groups list --channel whatsapp` falls back to `listGroups`, and the WhatsApp `listGroups` implementation can only return configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the maintainable implementation should use the existing `listGroupsLive` directory contract and WhatsApp runtime/cache ownership instead of calling Baileys from the config helper.
