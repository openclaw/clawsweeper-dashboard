## Summary

Keep open: current main still has no live WhatsApp group-membership directory; it only lists configured WhatsApp group IDs, while the docs describe the current config-backed behavior and log-based JID workaround. This is better classified as a WhatsApp plugin feature/product decision than a broken existing CLI contract.

## What This Changes

Request live WhatsApp group discovery for `openclaw directory groups list --channel whatsapp` when the linked account belongs to groups but `channels.whatsapp.groups` is empty.

## Best Possible Solution

Implement a WhatsApp-owned `listGroupsLive` backed by account-scoped Baileys group metadata or an explicit runtime cache, while keeping config-backed `listGroups` as the offline fallback and aligning CLI/docs guidance.

## Reproduction Assessment

Yes for the observed empty-list path: source inspection shows that with no configured `account.groups` keys, WhatsApp directory group listing returns an empty config-backed list. I did not run a live WhatsApp account, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. Dynamic discovery likely matches the user outcome, but the maintainable solution should use the existing `listGroupsLive` contract with a WhatsApp-owned runtime/cache and privacy boundary instead of adding ad hoc live calls to the config helper.
