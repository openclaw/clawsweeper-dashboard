## Summary

Keep open: current main still exposes WhatsApp group listing as a config-backed directory only. The CLI can prefer a live group adapter when a channel provides one, but WhatsApp currently registers only the config lister, so groups that exist only in the linked account membership remain undiscoverable through this command.

## What This Changes

Request that WhatsApp group directory listing return groups the linked account belongs to, or provide a better empty-state path when no group IDs are configured.

## Best Possible Solution

Add a WhatsApp-owned `listGroupsLive` adapter backed by account-scoped runtime metadata or an approved Baileys query path, while keeping the config-backed `listGroups` as the offline fallback and documenting the empty state.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI selects `listGroupsLive ?? listGroups`; WhatsApp lacks `listGroupsLive`, and its `listGroups` path returns only configured group-map keys.

## Solution Assessment

Unclear. Dynamic discovery is the right user outcome, but the maintainable implementation should use the existing live-directory contract and a WhatsApp-owned runtime boundary rather than adding provider calls inside the shared config helper.
