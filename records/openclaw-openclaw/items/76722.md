## Summary

Keep open: current main still only exposes a config-backed WhatsApp group directory, while live participating-group metadata is hydrated in the inbound runtime and not surfaced through the directory CLI.

## What This Changes

Requests WhatsApp group discovery so `openclaw directory groups list --channel whatsapp` returns groups the linked account belongs to even when `channels.whatsapp.groups` is empty.

## Best Possible Solution

Add a WhatsApp-owned live `listGroupsLive` path backed by an active linked account or deliberately scoped cached metadata, while keeping the config-backed lister as the offline fallback and documenting the no-live-session behavior.

## Reproduction Assessment

Yes, by source inspection. `openclaw directory groups list --channel whatsapp` selects `listGroupsLive ?? listGroups`, WhatsApp only provides `listGroups`, and that lister returns configured `channels.whatsapp.groups` keys.

## Solution Assessment

Unclear. Dynamic discovery is likely the right user outcome, but the maintainable fix should be a WhatsApp-owned live directory adapter or cache boundary, not a change to the generic config helper.
