## Summary

Keep open: current main still returns only WhatsApp group IDs already configured, while the core directory command already has a live-reader seam that WhatsApp has not implemented.

## What This Changes

Add WhatsApp group discovery so `openclaw directory groups list --channel whatsapp` can show groups from the linked account without pre-seeding group JIDs.

## Best Possible Solution

Add a WhatsApp-owned live group directory path through a session-safe Baileys metadata seam, while keeping configured `groups` as the allowlist/fallback output.

## Reproduction Assessment

Yes, from source inspection: the WhatsApp adapter only exposes config-backed `listGroups`, and the helper reads `account.groups` map keys. With no configured group keys, the command path returns an empty list even if the linked account belongs to groups.

## Solution Assessment

Unclear until maintainers choose the WhatsApp session access model. A `listGroupsLive` implementation is the right shape, but it must avoid disrupting the gateway-owned Baileys session.
