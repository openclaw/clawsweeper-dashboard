## Summary

Keep open: current main still lists WhatsApp groups from configured allowlist keys only, and the WhatsApp plugin does not implement the existing live group-directory hook. Because current docs/tests describe the config-backed behavior, this is best classified as a live-discovery feature/product gap rather than an already-fixed strict bug.

## What This Changes

The issue asks for `openclaw directory groups list --channel whatsapp` to discover WhatsApp groups the linked account has joined, instead of only returning `channels.whatsapp.groups` entries.

## Best Possible Solution

Add a WhatsApp-owned live group directory path that uses the active WhatsApp runtime or a gateway-owned metadata cache for joined groups, while preserving config-backed allowlist listing as the fallback.

## Reproduction Assessment

Yes, source-level: with empty or omitted `channels.whatsapp.groups`, current main resolves only the configured group map and returns no group entries regardless of WhatsApp account membership. I did not perform a live WhatsApp account test in this read-only review.

## Solution Assessment

Unclear: live discovery is likely the right user outcome, but the maintainable implementation should be a WhatsApp-owned `listGroupsLive` path using the active runtime/cache rather than adding provider calls to the config helper.
