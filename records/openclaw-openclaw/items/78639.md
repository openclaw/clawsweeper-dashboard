## Summary

Keep open: this PR targets a real current-main sync-FS hot path and includes credible live runtime proof, but the current head is dirty against main and the async auto-enable resolver bypasses main's metadata-snapshot reuse on the same hot path.

## What This Changes

This PR adds async plugin registry, metadata, and auto-enable paths for gateway channel status/start/send, updates WhatsApp Web close classification and reconnect logging, and adds changelog/test coverage.

## Best Possible Solution

Rebase and update the PR so the async resolver preserves current main's compatible metadata-snapshot reuse before filesystem discovery, then keep the async registry work and WhatsApp transport recovery covered by focused tests and live proof.

## Reproduction Assessment

Yes. Current main source still routes channels.status, channels.start, and messages.send through synchronous plugin auto-enable discovery, and the PR body provides redacted live before/after logs for the slow-filesystem WhatsApp burst path.

## Solution Assessment

Not yet. Asyncizing the registry/read path is the right direction, but the async auto-enable resolver should mirror current main's current-metadata-snapshot reuse before falling back to filesystem discovery.
