## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reuse current metadata snapshots in the async auto-enable path:** `src/config/plugin-auto-enable.shared.ts:998-1002`
  - body: `resolvePluginAutoEnableManifestRegistryAsync` always calls `loadPluginMetadataSnapshotAsync` when a registry is needed. The sync resolver first reuses `getCurrentPluginMetadataSnapshot`, and current gateway status/start/send paths rely on that prepared snapshot to avoid rediscovery; port that reuse into the async path before switching the hot paths over.
  - confidence: 0.86
- **[P2] Include behavior-affecting inputs in the async dedupe key:** `src/plugins/plugin-registry-snapshot.ts:449-464`
  - body: The async in-flight key hashes only install record ids and a small env subset, but the registry read can depend on install record values and env-derived default state/npm roots. Concurrent callers with the same plugin ids but different paths/specs or env roots can await the first caller's snapshot.
  - confidence: 0.84
- **[P2] Mirror current registry freshness checks in the async path:** `src/plugins/plugin-registry-snapshot.ts:490-514`
  - body: The async validation chain must preserve current main's persisted-registry checks for stale diagnostics, missing config-path activation metadata, recovered install record path existence, and allowed bundled source overlays. Otherwise a rebased async path can accept stale registries or reject valid current-main roots differently than the sync path.
  - confidence: 0.78
