## Summary

Keep open: this PR targets a real gateway/WhatsApp performance problem and includes useful live A/B proof, but it is not merge-ready because the current head has a failing gateway-methods check and the new async registry inflight cache can reuse snapshots across distinct request inputs.

## What This Changes

The branch adds async plugin registry/metadata/auto-enable paths, switches gateway channel status/start/send to them, changes WhatsApp reconnect classification/logging, and adds changelog entries.

## Best Possible Solution

Land a corrected version that keeps plugin metadata fresh, removes sync filesystem work from gateway hot paths, isolates async snapshots by all behavior-affecting inputs, and covers gateway/registry/WhatsApp behavior with focused tests.

## Reproduction Assessment

Yes. The linked issue gives concrete WSL2/WhatsApp self-DM steps and current main still follows the sync registry validation path; I did not live-run the slow-filesystem setup here.

## Solution Assessment

No, not yet. The overall async direction is maintainable, but this patch needs the failing gateway mocks fixed and the async registry cache keyed by all behavior-affecting inputs before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update the gateway mocks for async auto-enable:** `src/gateway/server-methods/channels.ts:245`
  - body: The PR switches channels.status/channels.start and send to `applyPluginAutoEnableAsync`, but the existing gateway tests still mock only `applyPluginAutoEnable`. The current head is failing gateway-methods with "No applyPluginAutoEnableAsync export is defined" until those mocks and expectations are updated.
  - confidence: 0.98
- **[P2] Key inflight registry snapshots by all inputs:** `src/plugins/plugin-registry-snapshot.ts:449-455`
  - body: This inflight key omits behavior-affecting inputs such as `workspaceDir` and env-derived state/bundled plugin paths. Two concurrent async calls with the same policy hash but different workspace or env can await the first promise and receive the wrong plugin registry snapshot.
  - confidence: 0.9
- **[P2] Remove sync recovery reads from the async path:** `src/plugins/plugin-registry-snapshot.ts:503-506`
  - body: `loadSnapshotInstallRecordsAsync` still reaches the managed-npm recovery scan, which uses sync JSON reads and `fs.statSync`. On installs with managed npm plugins, the new channels.status async path can still block the event loop during persisted registry validation.
  - confidence: 0.82
- **[P3] Preserve the active metadata scan phase:** `src/plugins/plugin-metadata-snapshot.ts:206`
  - body: The sync metadata scan preserves the active diagnostics phase, but the async variant hard-codes `phase: "startup"`. Gateway diagnostics for channel/status scans will lose the actual phase labels that are needed to correlate event-loop stalls with channel operations.
  - confidence: 0.78
