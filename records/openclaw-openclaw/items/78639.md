## Summary

Keep open: the reported slow-filesystem gateway/WhatsApp behavior is real and the async direction is useful, but this PR still has correctness issues in the new async resolver and inflight dedupe path that should be fixed before merge.

## What This Changes

The PR adds async plugin registry, metadata, and auto-enable paths for gateway status/start/send calls and changes WhatsApp Web transport close handling so retryable closes stay inside the monitor.

## Best Possible Solution

Keep the async read-path and WhatsApp transport recovery direction, but preserve current snapshot/memo reuse, make async dedupe safe for distinct install record values, reconcile current-main registry safeguards during rebase, and add focused regression coverage.

## Reproduction Assessment

Yes, source-backed. Current main still routes channels.status, channels.start, and messages.send through synchronous plugin auto-enable discovery, and the PR body supplies live slow-filesystem WhatsApp A/B logs; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. Asyncizing the persisted-read path is the right maintainable direction, but the patch needs to mirror current metadata snapshot reuse and make async inflight dedupe safe for different install record contents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reuse the current metadata snapshot in the async resolver:** `src/config/plugin-auto-enable.shared.ts:998-1001`
  - body: The sync resolver first checks `getCurrentPluginMetadataSnapshot` and only falls back to disk-backed metadata loading when no compatible snapshot exists. The new async resolver skips that fast path and always calls `loadPluginMetadataSnapshotAsync`, so gateway status/start/send can still rescan plugin metadata after startup already handed the gateway a compatible snapshot.
  - confidence: 0.89
- **[P3] Key async dedupe by full install record inputs:** `src/plugins/plugin-registry-snapshot.ts:449-450`
  - body: `buildAsyncRegistrySnapshotKey` only includes the install record ids. Because `loadInstalledPluginIndex` consumes and hashes the record values, two concurrent callers with the same ids but different install paths or specs can share the first caller's snapshot; include a stable digest of the records or skip dedupe when custom records are supplied.
  - confidence: 0.82
