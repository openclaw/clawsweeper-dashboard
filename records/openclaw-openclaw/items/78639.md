- **[P2] Preserve current metadata reuse in the async hot path:** `src/gateway/server-methods/channels.ts:245-249`
  - body: Current main routes gateway status/start/send through `resolveGatewayPluginConfig`, which reuses the prepared current plugin metadata snapshot and caches the auto-enabled config. This PR switches those hot paths to `applyPluginAutoEnableAsync` directly, and the async resolver always calls `loadPluginMetadataSnapshotAsync`; port the async path through the current snapshot/cache lifecycle instead of reintroducing per-call metadata resolution.
  - confidence: 0.87
- **[P2] Include all behavior-affecting inputs in async dedupe:** `src/plugins/plugin-registry-snapshot.ts:449-464`
  - body: The async in-flight key only hashes install record ids plus a small env subset. Registry reads can depend on install record values and env-derived default state/npm roots, so concurrent callers with the same plugin ids but different paths/specs or state roots can await the wrong snapshot; include those inputs or narrow dedupe to truly identical params.
  - confidence: 0.84
- **[P2] Mirror current registry freshness checks in async validation:** `src/plugins/plugin-registry-snapshot.ts:490-515`
  - body: Current main rejects persisted registries with stale diagnostics, missing config-path activation metadata, mismatched source overlays, and env-aware recovered install records. The async validation chain only checks a subset, so it can accept stale persisted metadata or derive under different conditions than the sync path after rebase.
  - confidence: 0.8
