## Summary

Keep this PR open: it targets a real, source-reproducible gateway/WhatsApp performance bug and includes sufficient live A/B log proof, but the current patch still has merge-blocking correctness issues in the gateway test mocks, async registry dedupe key, and remaining sync managed-npm recovery reads.

## What This Changes

This PR adds async plugin registry, metadata, and auto-enable paths for gateway channel status/start/send, changes WhatsApp Web reconnect close classification/logging, and updates the changelog.

## Best Possible Solution

Land a corrected version that makes the gateway hot-path registry validation truly async without persistent metadata caching, preserves per-call snapshot isolation and diagnostics attribution, and keeps WhatsApp transport recovery covered by focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows channels.status/start/send still route through synchronous plugin registry validation, and the linked issue plus PR body give a concrete slow-filesystem WhatsApp burst reproduction path; I did not live-run the PVC or WSL setup in this read-only review.

## Solution Assessment

No, not yet. The async direction is the right maintainable fix, but this patch needs input-safe inflight dedupe, removal of the remaining sync recovery scan, updated gateway mocks/tests, and preserved diagnostics phase before merge.

Full review comments:

- **[P2] Update the gateway mocks for async auto-enable:** `src/gateway/server-methods/channels.ts:245-248`
  - body: The PR switches channels.status, channels.start, and send to applyPluginAutoEnableAsync, but the existing gateway tests still mock only applyPluginAutoEnable. Those mocked modules will not provide the newly imported export, so the affected test files fail before exercising the behavior.
  - confidence: 0.98
- **[P2] Key async snapshot dedupe by every behavior input:** `src/plugins/plugin-registry-snapshot.ts:444-455`
  - body: The inflight key omits behavior-affecting inputs such as workspaceDir, env-derived flags/bundled paths, and installRecords. Concurrent async calls with the same state/file/policy hash but different registry context can receive the first caller's snapshot.
  - confidence: 0.9
- **[P2] Remove sync recovery reads from the async path:** `src/plugins/plugin-registry-snapshot.ts:503-506`
  - body: loadSnapshotInstallRecordsAsync still reaches the managed-npm recovery scan, which uses sync JSON reads and fs.statSync. Managed npm plugin installs can therefore still block the event loop during the new async persisted-registry validation path.
  - confidence: 0.84
- **[P3] Preserve the active metadata scan phase:** `src/plugins/plugin-metadata-snapshot.ts:202-214`
  - body: The sync metadata scan preserves the active diagnostics phase, but the async variant hard-codes phase: "startup". Gateway channel/status scans will lose the phase labels needed to correlate event-loop stalls with channel operations.
  - confidence: 0.78
