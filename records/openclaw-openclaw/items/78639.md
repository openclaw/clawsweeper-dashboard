## Summary

Keep open: this PR targets a real current-main gateway/WhatsApp performance bug and includes useful live A/B proof, but the current head has confirmed gateway-methods failures plus async registry correctness blockers that should be fixed before merge.

## What This Changes

The branch adds async plugin registry, metadata, and auto-enable paths for gateway channel status/start/send, changes WhatsApp reconnect close classification/logging, and updates the changelog.

## Best Possible Solution

Land a corrected version that removes sync filesystem work from gateway hot paths while preserving fresh plugin metadata, isolates async snapshot dedupe by all behavior-affecting inputs, and keeps WhatsApp lifecycle versus transport close handling covered by focused tests.

## Reproduction Assessment

Yes, source inspection plus the linked issue gives a high-confidence reproduction path: slow-filesystem gateway use with WhatsApp self-DM traffic still reaches synchronous plugin registry validation on current main. I did not live-run the slow-filesystem setup in this review.

## Solution Assessment

No, not yet. The async direction is the right maintainable fix, but this patch needs the failing gateway mocks/tests fixed, the async registry key hardened, and the remaining sync recovery scan removed or isolated before merge.

Full review comments:

- **[P2] Update the gateway mocks for async auto-enable:** `src/gateway/server-methods/channels.ts:245`
  - body: The PR switches channels.status, channels.start, and send to applyPluginAutoEnableAsync, but the existing gateway tests still mock only applyPluginAutoEnable.   - body: This inflight key omits behavior-affecting inputs such as workspaceDir and env-derived state or bundled plugin paths. Concurrent async calls with the same policy hash but different workspace or env can share the first promise and receive the wrong registry snapshot.
  - confidence: 0.9

  - body: loadSnapshotInstallRecordsAsync still reaches the managed-npm recovery scan, which uses sync JSON reads and fs.statSync. Installs with managed npm plugins can still block the event loop during the new async persisted-registry validation path.
  - confidence: 0.82
- **[P3] Preserve the active metadata scan phase:** `src/plugins/plugin-metadata-snapshot.ts:206`
  - body: The sync metadata scan preserves the active diagnostics phase, but the async variant hard-codes phase: "startup". Gateway channel/status scans will lose the phase labels needed to correlate event-loop stalls with channel operations.
  - confidence: 0.78
