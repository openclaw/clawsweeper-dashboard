## Summary

Keep this PR open: the direction and supplied live proof are useful, but the latest head still leaves synchronous plugin metadata discovery on the new async gateway path and has an unsafe async inflight dedupe key for custom install records.

## What This Changes

The PR adds async plugin registry/metadata auto-enable paths for gateway channel status/start/send calls and changes WhatsApp Web close handling so transport-class closes retry inside the monitor.

## Best Possible Solution

Keep the async read-path and WhatsApp retry direction, but make candidate detection fully async, preserve current metadata snapshot reuse, key async dedupe by full install-record inputs, and add focused regression coverage before merge.

## Reproduction Assessment

Yes, source-backed. Current main still sends channels.status and messages.send through synchronous plugin auto-enable discovery, and the PR body plus linked issue include live slow-filesystem WhatsApp evidence; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. Asyncizing the persisted-read path is the right maintainable direction, but this patch still invokes synchronous discovery before the async path and needs safer dedupe semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Make candidate detection async before using the async resolver:** `src/config/plugin-auto-enable.apply.ts:84`
  - body: The new async entry point still calls `detectPluginAutoEnableCandidates`, which calls the synchronous manifest registry resolver whenever configured channels/providers may need auto-enable. For WhatsApp configs this can still enter the sync metadata/registry loader before the async resolver runs, so the gateway hot path can continue blocking on the same FS checks this PR is meant to remove.
  - confidence: 0.9
- **[P2] Reuse the current metadata snapshot before async discovery:** `src/config/plugin-auto-enable.shared.ts:998-1001`
  - body: The synchronous resolver first checks `getCurrentPluginMetadataSnapshot` and only falls back to disk-backed metadata loading when no compatible snapshot exists. The async resolver skips that fast path and always calls `loadPluginMetadataSnapshotAsync`, so gateway status/start/send can still rescan metadata after startup already produced a compatible snapshot.
  - confidence: 0.88
- **[P3] Key async dedupe by full install record inputs:** `src/plugins/plugin-registry-snapshot.ts:449-450`
  - body: `buildAsyncRegistrySnapshotKey` only includes the install record ids, but index construction consumes the full record contents and stores a hash of each record. Two concurrent callers with the same plugin ids but different specs or install paths can share the first caller's snapshot; include a stable digest of the records or skip dedupe when custom records are supplied.
  - confidence: 0.8
