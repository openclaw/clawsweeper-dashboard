## Summary

Keep open: this is a plausible active bug-fix PR with sufficient live behavior proof, but the current patch still misses the existing metadata-snapshot fast path and has an unsafe async dedupe key for custom install records.

## What This Changes

The PR adds async plugin registry, metadata, and auto-enable paths for gateway status/start/send calls and changes WhatsApp Web connect-close handling/logging so transport-class closes retry inside the monitor.

## Best Possible Solution

Keep the async read-path and WhatsApp transport recovery direction, but preserve current snapshot/memo reuse, isolate async dedupe by full install-record inputs, reconcile current-main metadata memoization, and add focused regression coverage before merge.

## Reproduction Assessment

Yes, source-backed. Current main still routes channels.status, channels.start, and messages.send through synchronous plugin auto-enable discovery, and the linked issue plus PR body include live slow-filesystem WhatsApp logs; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. Asyncizing the persisted-read path is the right maintainable direction, but the patch needs to mirror current metadata snapshot reuse and make async inflight dedupe safe for different install record contents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reuse the current metadata snapshot before async discovery:** `src/config/plugin-auto-enable.shared.ts:998-1001`
  - body: The synchronous resolver first asks `getCurrentPluginMetadataSnapshot` and only falls back to disk-backed metadata loading when no compatible snapshot exists. The new async resolver skips that fast path and always calls `loadPluginMetadataSnapshotAsync`, so gateway status/start/send can still rescan plugin metadata after startup already handed the gateway a compatible snapshot.
  - confidence: 0.89
- **[P3] Key async dedupe by full install record inputs:** `src/plugins/plugin-registry-snapshot.ts:449-450`
  - body: `buildAsyncRegistrySnapshotKey` only includes the install record ids. Two concurrent callers with the same plugin ids but different specs or install paths will share the first snapshot even though `loadInstalledPluginIndex` consumes the full record contents; include a stable digest of the records or skip dedupe when custom records are supplied.
  - confidence: 0.8
