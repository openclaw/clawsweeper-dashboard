## Summary

Keep open: the PR targets a real gateway event-loop and WhatsApp reconnect problem with useful live proof, but the latest head still misses current main's metadata-snapshot reuse in the new async resolver and has one edge-case async dedupe correctness issue.

## What This Changes

The PR adds asynchronous plugin registry, metadata, and auto-enable paths for gateway channel status/start/send calls and changes WhatsApp Web reconnect handling/logging for transport-class close codes.

## Best Possible Solution

Keep the async registry/read-path direction and WhatsApp transport recovery, but update the async resolver to preserve current main's compatible metadata-snapshot reuse and tighten the async dedupe key before merge.

## Reproduction Assessment

Yes. Current main source still routes channels.status, channels.start, and messages.send through synchronous plugin auto-enable discovery, and the PR body provides redacted live before/after logs for the slow-filesystem WhatsApp burst path.

## Solution Assessment

No, not yet. Asyncizing the persisted-read path is the right maintainable direction, but the proposed async auto-enable resolver should mirror current main's metadata-snapshot reuse before falling back to filesystem discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reuse the current metadata snapshot before async discovery:** `src/config/plugin-auto-enable.shared.ts:998-1001`
  - body: The sync resolver first asks `getCurrentPluginMetadataSnapshot` and only loads metadata from disk if no compatible snapshot exists. The new async resolver goes straight to `loadPluginMetadataSnapshotAsync`, so gateway status/start/send calls will rescan plugin metadata even after startup handed a compatible snapshot to the gateway, reintroducing discovery work on the hot path this PR is trying to remove.
  - confidence: 0.88
- **[P3] Key async dedupe by full install record inputs:** `src/plugins/plugin-registry-snapshot.ts:449-450`
  - body: The async inflight key only includes `Object.keys(params.installRecords)`. Two concurrent callers with the same plugin ids but different specs or install paths will share the first snapshot, even though `loadInstalledPluginIndex` consumes the record contents to build the registry; include a stable digest of the records or skip dedupe when custom records are supplied.
  - confidence: 0.77
