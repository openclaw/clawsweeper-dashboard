## Summary

Keep open. Current main and the latest release still reject configured custom basenames and external absolute bootstrap paths, and the related open issues/PRs have not settled the product/security contract for explicit bootstrap file trust boundaries.

## What This Changes

The issue requests that explicitly configured `bootstrap-extra-files.paths` load arbitrary basenames and trusted absolute files instead of being filtered by the recognized-bootstrap basename and workspace-root guards.

## Best Possible Solution

Define the bootstrap-extra-files trust contract first, then implement one documented path with focused tests: workspace-contained arbitrary basenames if approved, external files only behind an explicit reviewed opt-in, and visible diagnostics for skipped configured files.

## Reproduction Assessment

Yes, from source inspection: configured paths enter `loadExtraBootstrapFilesWithDiagnostics`, non-whitelisted basenames are rejected before the read path, and accepted names still go through a workspace-root guarded open. I did not run a live bootstrap because this review is read-only.

## Solution Assessment

Unclear. Bypassing the whitelist and workspace boundary would address the deployment described in the issue, but the maintainable solution needs an explicit trust contract, docs, diagnostics, and regression tests before changing prompt file-read authority.
