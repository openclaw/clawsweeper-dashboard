## Summary

Keep open. Current main and the latest release still enforce the documented basename allowlist and workspace-root guard for configured bootstrap extras, while the reporter’s requested absolute-path behavior changes a prompt-context file-read boundary that needs maintainer product/security review.

## What This Changes

Request to let explicit `bootstrap-extra-files.paths` load custom basenames and absolute paths outside the workspace, and to surface skipped-path diagnostics.

## Best Possible Solution

Define the bootstrap-extra-files trust contract first, then either add visible diagnostics under the current restrictions or deliberately support explicit arbitrary/external files with opt-in policy, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows configured paths flow into `loadExtraBootstrapFilesWithDiagnostics`, where non-whitelisted basenames are rejected and reads remain rooted at the workspace; I did not run a live bootstrap because this review is read-only.

## Solution Assessment

Unclear. Bypassing the whitelist and workspace boundary would address the reporter’s deployment, but it changes the documented security contract; a maintainer-approved contract with diagnostics, docs, and tests is the safer path.
