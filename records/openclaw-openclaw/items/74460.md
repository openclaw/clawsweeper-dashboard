## Summary

Keep open. Current main has already absorbed the Windows SystemRoot/WINDIR/ProgramFiles blocklist work from related #74456, but it still does not block COMSPEC from workspace dotenv files while the Windows .cmd wrapper path reads process.env.ComSpec. The PR remains a narrow, relevant hardening patch with no blocking review findings, but it should be reconciled with the still-open related Windows ACL hardening in #74458 before merge.

## What This Changes

The PR adds COMSPEC, PROGRAMFILES, PROGRAMW6432, SYSTEMROOT, and WINDIR to the workspace dotenv blocklist and adds unit coverage for mixed-case and uppercase Windows shell trust-root keys.

## Best Possible Solution

Land one canonical Windows dotenv hardening that preserves the already-merged SystemRoot/WINDIR/ProgramFiles coverage, adds COMSPEC with focused dotenv coverage, and reconciles any remaining ACL command-path validation from #74458.

Yes. A source-level reproduction exists: a workspace .env key named ComSpec is currently accepted by loadWorkspaceDotEnvFile because COMSPEC is absent from the blocklist, and Windows batch-command execution reads process.env.ComSpec in src/process/exec.ts.

Yes for the narrow COMSPEC workspace dotenv gap. The maintainable final path is to rebase or fold this small blocklist/test addition into the canonical Windows hardening work so duplicate blocklist entries and overlapping tests do not diverge.
