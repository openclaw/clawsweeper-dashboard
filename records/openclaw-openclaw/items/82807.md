## Summary

Keep open: the report matches a plausible Discord progress-preview regression, not a generic Discord send-splitting gap. Current Discord direct sends chunk over-limit text, but the progress-preview finalization path can accept a short ellipsis-truncated final snapshot and edit the draft without the transcript-backed recovery Telegram already has.

## What This Changes

Fix Discord delivery of roughly 3000-character assistant replies that are truncated instead of delivered complete or chunked in openclaw 2026.5.16-beta.3.

## Best Possible Solution

Add Discord progress-preview parity with Telegram: detect ellipsis-truncated final snapshots, recover a current-turn full transcript final when it extends the snapshot, and deliver that text through normal Discord chunking.

## Reproduction Assessment

No high-confidence live current-main repro was run. ## Solution Assessment

Yes, the narrow maintainable fix is to reuse or port the existing Telegram truncated-final recovery pattern for Discord progress previews. Changing generic Discord chunking first would be less targeted because the normal send path already chunks long text.
