## Summary

Keep open: current main still lacks per-agent session maintenance, and this PR is the canonical tracker, but the branch is not mergeable as-is because it leaves normal runtime writes on global maintenance, can apply agent policies to shared or arbitrary stores, is dirty against current main, and lacks real behavior proof.

## What This Changes

Adds `agents.list[].maintenance` config support, shared maintenance schema extraction, per-agent/global merge logic, cleanup and store plumbing, generated schema updates, and resolver tests.

## Best Possible Solution

Rebase the feature onto the cleanup service, resolve per-agent policy from a normalized single config snapshot, apply agent policies only to owned per-agent runtime writes/stores, keep shared or arbitrary store cleanup on a safe global policy unless entries are partitioned, preserve `pruneDays` compatibility, and land docs plus focused tests with real-run proof.

## Reproduction Assessment

Not applicable; this is a new public config feature rather than a current-main bug. Source inspection shows current main only exposes global `session.maintenance` and no per-agent maintenance field or resolver path.

## Solution Assessment

No. The direction is plausible, but the current patch is not the best mergeable solution because normal write paths remain global and cleanup can apply agent-specific retention to stores that are shared or not owned by that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Thread per-agent maintenance through runtime writes:** `src/config/sessions/store.ts:395-400`
  - body: `resolveMaintenanceConfig(opts?.agentId)` only works when callers pass an agent id, but normal session writes still pass a pre-resolved global `maintenanceConfig` or no agent id. As a result, per-agent pruning, caps, and disk budgets are not enforced during ordinary traffic.
  - confidence: 0.88
- **[P1] Keep shared-store cleanup on the global policy:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` can dedupe multiple agents down to one shared store path, so resolving cleanup mode from the surviving `target.agentId` can prune entries for another agent in that same JSON store. Shared stores need global policy or entry partitioning before destructive cleanup uses agent overrides.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to explicit stores:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id only as a placeholder for an arbitrary path. Passing that synthetic id into maintenance selection can apply the default agent's retention policy to a store it does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: The lookup compares configured and requested ids verbatim, while existing agent resolution normalizes ids before matching. Mixed-case or human-readable ids can resolve elsewhere but silently miss their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` together. Since resolution prefers `pruneAfter`, a supported per-agent `pruneDays` override would not replace the global age window.
  - confidence: 0.84
