## Summary

Keep open. Current main still exposes session maintenance only as global `session.maintenance`, so the PR is not obsolete or implemented, but the latest proposed patch has blocking functional issues around schema compatibility, normal runtime enforcement, and cleanup target safety.

## What This Changes

The PR adds an `agents.list[].maintenance` config surface, extracts a shared session maintenance Zod schema, merges per-agent overrides over global session maintenance, threads an agent id into session cleanup/save options, regenerates base config schema, and adds resolver tests.

## Best Possible Solution

Keep this PR open for author revision or maintainer consolidation. A mergeable implementation should preserve optional config compatibility, resolve effective per-agent maintenance from one normalized config snapshot, apply it consistently to runtime reads/writes and manual cleanup, keep shared and explicit-store cleanup on a safe policy, and update generated schema, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature PR, but the blocking findings have high-confidence source-level reproduction paths. Current main and the supplied PR diff show where automatic writes still use global maintenance and where cleanup target semantics can apply an agent policy to the wrong store.

## Solution Assessment

No. The direction is useful, but this patch is not yet the best maintainable solution because it exposes a new data-retention surface before schema compatibility, runtime enforcement paths, and shared-store cleanup semantics are safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep maintenance optional in config schemas:** `src/config/zod-schema.agent-runtime.ts:798`
  - body: `SessionMaintenanceSchema` is a required object schema, and adding it directly to `AgentEntrySchema` makes every agent entry require `maintenance`; the same extracted schema also replaces the previously optional `session.maintenance`. Existing configs that rely on defaults will fail validation before runtime can use those defaults, so the call sites need optional variants.
  - confidence: 0.9
- **[P1] Thread per-agent policy into normal writes:** `src/config/sessions/store.ts:292`
  - body: Adding `agentId` to save options does not make automatic maintenance honor agent overrides unless normal callers pass the effective policy. Reply/session writes already pre-resolve global `session.maintenance`, and boot/heartbeat writes still have no agent-aware policy, so regular traffic will ignore `agents.list[].maintenance`.
  - confidence: 0.88
- **[P1] Fall back to global policy for shared stores:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` dedupes identical store paths, but resolving maintenance from the surviving `target.agentId` applies one agent's retention limits to the whole shared JSON store. In shared-store setups, a stricter override can prune another agent's sessions; shared targets need a global or entry-aware policy.
  - confidence: 0.9
- **[P2] Detach explicit store cleanup from agent overrides:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, the target resolver uses the default agent id only as a placeholder for an arbitrary path. Feeding that synthetic id into maintenance resolution makes explicit file cleanup inherit the default agent's override, which can delete entries from a store the default agent does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:139`
  - body: The new override lookup compares `a.id === agentId` verbatim, while normal agent routing normalizes ids. Config entries such as `Ops` or `My Agent` can resolve elsewhere but miss their maintenance override here, silently falling back to the global policy.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging:** `src/config/sessions/store-maintenance.ts:157`
  - body: A spread merge can leave global `pruneAfter` and per-agent legacy `pruneDays` on the same object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will not override the global age window.
  - confidence: 0.84
