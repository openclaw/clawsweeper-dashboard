## Summary

Keep this PR open, but it is not merge-ready. Current main still documents and enforces session maintenance as a global policy, and this PR's per-agent config direction is useful but has blocking correctness issues around runtime writes, shared stores, explicit store cleanup, and legacy override semantics.

## What This Changes

The PR adds per-agent `agents.list[].maintenance` config, shared maintenance schema reuse, per-agent merge resolution, cleanup/save-path agent-id threading, generated schema updates, and resolver tests.

## Best Possible Solution

Land a rebased implementation that resolves effective maintenance from one normalized config snapshot, wires owned per-agent runtime writes consistently, keeps shared or arbitrary store cleanup on a safe global policy unless entries are partitioned by owner, preserves legacy `pruneDays` precedence, and updates docs, changelog, generated schema, and focused regression coverage together.

## Reproduction Assessment

Not applicable as a feature PR. The blocking findings have source-level paths through the reply write path, cleanup target resolution, shared-store dedupe, explicit `--store` handling, and current agent-id normalization contracts.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best mergeable form because it advertises per-agent retention while leaving key enforcement paths global and applying agent-specific policies to store scopes that may not belong to that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread per-agent maintenance into runtime writes:** `src/config/sessions/store.ts:398-400`
  - body: Adding `agentId` to save options does not make normal traffic use per-agent retention because the main reply path already passes a pre-resolved global `maintenanceConfig`, and other runtime write sites still do not provide an agent id. Configured per-agent `pruneAfter`, `maxEntries`, and disk budgets would not be enforced during the writes that normally run maintenance.
  - confidence: 0.88
- **[P1] Use global cleanup policy for shared stores:** `src/commands/sessions-cleanup.ts:314-315`
  - body: `--all-agents` can dedupe several agents down to one shared store path, but this change resolves cleanup policy from the surviving target agent. A stricter override on that one agent can prune entries in the shared JSON file that belong to another agent, so shared targets need global policy or entry-owner partitioning first.
  - confidence: 0.9
- **[P2] Keep explicit store cleanup off agent overrides:** `src/commands/sessions-cleanup.ts:373-377`
  - body: For `--store`, target resolution uses the default agent id only as a placeholder for an arbitrary file path. Passing that synthetic id into maintenance resolution makes manual cleanup inherit the default agent's override and can delete entries from a store that the default agent does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This exact `a.id === agentId` lookup diverges from the rest of agent resolution, which normalizes configured and requested ids before matching. A config entry such as `Ops` or `My Agent` can resolve elsewhere but miss its maintenance override here and silently fall back to global retention.
  - confidence: 0.86
- **[P2] Resolve pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` on the same object. Because the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override would not override the global age window.
  - confidence: 0.84
