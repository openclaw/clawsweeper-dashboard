## Summary

Keep open. Current main still has only global session maintenance, and this PR has blocking source-review issues: normal runtime writes remain on global policy, cleanup can apply agent policies to shared or arbitrary stores, and the resolver misses existing agent-id and legacy `pruneDays` semantics; the external PR also lacks real behavior proof.

## What This Changes

Adds `agents.list[].maintenance` config support, shared maintenance schema extraction, per-agent/global merge logic, cleanup and store plumbing, generated schema changes, and resolver tests.

## Best Possible Solution

Land a rebased implementation only after it resolves per-agent maintenance from a normalized config snapshot, applies agent policies to owned per-agent stores and runtime writes, keeps shared or arbitrary store cleanup on a safe global policy unless entries are partitioned, preserves legacy `pruneDays`, and updates docs, schema, and focused regression coverage together.

## Reproduction Assessment

Not applicable; this is a new public config feature rather than a current-main bug report. Source inspection confirms current main only exposes global `session.maintenance` and no per-agent maintenance field.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable form because key runtime write paths remain global and cleanup can apply an agent-specific policy to stores that may not belong to that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Thread per-agent maintenance through runtime writes:** `src/config/sessions/store.ts:398-400`
  - body: The new resolver only uses the per-agent override when callers pass an agent id, but normal reply/gateway/heartbeat writes still omit it or pass a pre-resolved global maintenance config. That means pruning, capping, and disk-budget enforcement still use the global policy during ordinary traffic, so the feature only works in limited cleanup paths.
  - confidence: 0.88
- **[P1] Keep shared-store cleanup on the global policy:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` can dedupe multiple agents down to one store path, but this change resolves maintenance from whichever target agent survives. A stricter override on that agent can prune entries belonging to another agent in the same JSON store.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to explicit stores:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary path. Passing that synthetic id into maintenance selection can apply the default agent's retention policy to a store it does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching maintenance overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: The lookup compares configured and requested ids verbatim, while existing agent resolution normalizes ids before matching. Mixed-case or human-readable ids can resolve elsewhere but silently miss their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` together. Since resolution prefers `pruneAfter`, a supported per-agent `pruneDays` override would not replace the global age window.
  - confidence: 0.84
