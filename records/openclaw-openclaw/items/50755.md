## Summary

Keep open: current main still lacks per-agent session maintenance, but this PR is not merge-ready because it is conflicting with current main, lacks real behavior proof, and still has destructive retention risks around normal writes, shared stores, and manual store cleanup.

## What This Changes

The PR adds `agents.list[].maintenance` overrides, extracts a shared maintenance schema, adds resolver tests, and threads agent context through session cleanup and store-save paths.

## Best Possible Solution

Rebase onto current main and implement per-agent maintenance through normalized, owner-aware `maintenanceConfig` resolution, keeping shared/manual stores on global or entry-aware policy with docs, focused tests, and redacted real CLI/runtime proof.

## Reproduction Assessment

## Solution Assessment

No. The feature direction is plausible, but this diff needs a current-main rebase and owner-aware retention semantics before it is the maintainable way to add per-agent session maintenance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate per-agent maintenance into resolved write configs:** `src/config/sessions/store.ts:395-399`
  - body: This new `agentId` option does not integrate with current main's `maintenanceConfig` contract. Normal runtime writes already pass a pre-resolved global config, so after rebase those callers can bypass the per-agent resolver and keep enforcing global retention during ordinary traffic.
  - confidence: 0.9
- **[P1] Keep shared cleanup stores on a safe policy:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` target resolution dedupes identical store paths, so `target.agentId` can be only one representative for a shared `sessions.json`. Applying that representative agent's override can prune or cap sessions belonging to another agent in the same file.
  - confidence: 0.9
- **[P2] Detach manual store cleanup from default-agent overrides:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary operator-selected path. Passing that id into the save path makes manual cleanup inherit the default agent's override instead of the global policy for the selected store.
  - confidence: 0.86
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This exact `a.id === agentId` lookup does not match the rest of agent/session routing, which normalizes ids. A configured id like `Ops` or `My Agent` can resolve elsewhere but miss its maintenance override here and silently fall back to the global policy.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging:** `src/config/sessions/store-maintenance.ts:162`
  - body: The raw spread can keep both a global `pruneAfter` and an agent `pruneDays` on the merged object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will not win over a global `pruneAfter`.
  - confidence: 0.83
