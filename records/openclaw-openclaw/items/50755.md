## Summary

Keep open. The feature is not implemented on current main, the exact same-feature issue is already closed in favor of this PR, and the PR still has blocking correctness issues plus missing contributor real behavior proof.

## What This Changes

The PR adds `agents.list[].maintenance` session retention overrides, extracts a shared maintenance schema, updates generated config schema, threads `agentId` into cleanup/store paths, and adds resolver tests.

## Best Possible Solution

Rebase onto the cleanup service, define the per-agent retention contract in docs/help, resolve policy from one normalized config snapshot, apply agent policy only to agent-owned runtime writes/stores, keep shared or arbitrary stores on a safe global policy unless entries are partitioned, preserve `pruneDays` compatibility, and add real-run proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new public config feature. The review findings are source-reproducible from the PR diff plus current-main target resolution and writer semantics.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable solution because runtime writes remain effectively global, cleanup can apply one agent's policy to shared or explicit stores, and the branch is behind the current cleanup-service architecture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Thread per-agent maintenance through runtime writes:** `src/config/sessions/store.ts:399-400`
  - body: `resolveMaintenanceConfig(opts?.agentId)` only changes behavior when callers pass an agent id, but ordinary session-store writes still pass a fully resolved global maintenance config or no agent id. Per-agent pruning, caps, and disk budgets therefore will not run during normal traffic, only in the paths that explicitly adopt this new option.
  - confidence: 0.88
- **[P1] Keep shared-store cleanup on the global policy:** `src/commands/sessions-cleanup.ts:165`
  - body: `previewStoreCleanup()` now resolves maintenance from `target.agentId` for the entire JSON store. Because `--all-agents` dedupes multiple agents that share a store path, the surviving agent's stricter policy can prune entries belonging to another agent in that same store.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to explicit stores:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary path. Passing that synthetic id into `updateSessionStore()` can make an operator-selected store inherit the default agent's per-agent retention policy, even when that store belongs to something else.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This lookup compares the configured id and requested id verbatim, while current agent resolution normalizes both sides before matching. Config entries such as `"Ops"` or `"My Agent"` can resolve elsewhere but silently miss their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` on the same object. Since resolution prefers `pruneAfter`, a supported per-agent `pruneDays` override will not replace the global age window.
  - confidence: 0.84
