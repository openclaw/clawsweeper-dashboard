## Summary

Keep open. Current main still has only global session maintenance, and the PR is a plausible feature direction, but the patch is not mergeable because normal runtime writes remain global, destructive cleanup can apply an agent policy to shared or arbitrary stores, the branch is dirty against current main, and contributor real behavior proof is missing.

## What This Changes

Adds `agents.list[].maintenance` config, shared maintenance schema extraction, per-agent/global merge logic, cleanup/store plumbing, generated schema updates, and resolver tests.

## Best Possible Solution

Rebase onto the cleanup service, define per-agent retention as a documented public config contract, resolve policy from one normalized config snapshot, apply agent policies only to agent-owned runtime writes/stores, keep shared or arbitrary stores on a safe global policy unless entries are partitioned, preserve `pruneDays` compatibility, and add docs plus focused tests with real-run proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new public config feature. Source inspection of current main shows only global `session.maintenance` and no accepted per-agent `maintenance` field or agent-aware resolver path.

## Solution Assessment

No. The direction is plausible, but this patch is not the best mergeable solution because normal runtime writes still use global maintenance and cleanup can apply agent-specific retention to shared or arbitrary stores.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Thread per-agent maintenance through runtime writes:** `src/config/sessions/store.ts:395-400`
  - body: `resolveMaintenanceConfig(opts?.agentId)` only takes effect when callers pass an agent id, but normal session writes on current main still pass a global `maintenanceConfig` or no agent id. Per-agent pruning, caps, and disk budgets would therefore not be enforced during ordinary traffic.
  - confidence: 0.88
- **[P1] Keep shared-store cleanup on the global policy:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` can dedupe multiple agents down to one shared store path. Resolving cleanup mode from the surviving `target.agentId` can prune entries for another agent in that same JSON store, so shared stores need global policy or entry partitioning before destructive cleanup uses agent overrides.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to explicit stores:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id only as a placeholder for an arbitrary path. Passing that synthetic id into maintenance selection can apply the default agent's retention policy to a store it does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This lookup compares configured and requested ids verbatim, while existing agent resolution normalizes ids before matching. Mixed-case or human-readable ids can resolve elsewhere but silently miss their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` together. Since resolution prefers `pruneAfter`, a supported per-agent `pruneDays` override would not replace the global age window.
  - confidence: 0.84
