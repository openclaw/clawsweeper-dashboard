## Summary

Keep this PR open for maintainer handling: current main has not implemented per-agent session maintenance, and the proposed patch still has merge conflicts plus concrete retention/data-loss correctness issues before it can merge.

## What This Changes

The PR adds an `agents.list[].maintenance` config surface, shared maintenance schema extraction, per-agent/global merge logic, cleanup/store plumbing, generated schema updates, and resolver tests.

## Best Possible Solution

Land a rebased implementation that resolves per-agent maintenance from one normalized config snapshot, applies it only to owned per-agent stores/runtime writes, keeps shared or arbitrary store cleanup on a safe global policy unless entries are partitioned, preserves legacy `pruneDays` behavior, and updates docs, changelog, schema, and regression coverage together.

## Reproduction Assessment

Not applicable; this is a feature PR rather than a reported current-main bug. The remaining blockers are high-confidence source-review paths through runtime writes, cleanup target resolution, shared-store dedupe, explicit `--store`, and agent-id normalization.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best mergeable form because it leaves key enforcement paths global and can apply agent-specific cleanup policies to stores that may not belong to that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread per-agent maintenance into runtime writes:** `src/config/sessions/store.ts:398-400`
  - body: Adding an `agentId` save option is not enough while normal traffic still passes pre-resolved global maintenance or omits an agent id. Per-agent `pruneAfter`, `maxEntries`, and disk budgets would not be enforced on the write paths that normally run maintenance.
  - confidence: 0.88
- **[P1] Use global cleanup policy for shared stores:** `src/commands/sessions-cleanup.ts:314-315`
  - body: `--all-agents` can dedupe multiple agents to one shared store path, but the patch resolves cleanup from the surviving target agent. A stricter override on that agent can prune entries belonging to another agent in the same JSON file.
  - confidence: 0.9
- **[P2] Keep explicit store cleanup off agent overrides:** `src/commands/sessions-cleanup.ts:373-377`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary path. Passing that synthetic id into maintenance resolution can apply the default agent's override to a store it does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: The new lookup compares configured and requested ids verbatim, but existing agent resolution normalizes ids before matching. Mixed-case or human-readable ids can resolve elsewhere while silently missing their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` together. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override would not replace the global age window.
  - confidence: 0.84
