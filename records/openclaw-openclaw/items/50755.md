Full review comments:

- **[P1] Propagate per-agent maintenance into runtime writes:** `src/config/sessions/store.ts:395-399`
  - body: This path only resolves per-agent maintenance when callers pass `opts.agentId`, but current main's normal reply/session writes precompute and pass a global `maintenanceConfig`. Without updating that caller contract, normal traffic will keep enforcing global retention instead of the new per-agent policy.
  - confidence: 0.88
- **[P1] Keep shared cleanup stores on a safe policy:** `src/commands/sessions-cleanup.ts:311-314`
  - body: `--all-agents` target resolution can dedupe multiple agents to the same store path, but this line chooses one representative agent's policy for the whole file. A stricter override on one agent can prune or cap sessions belonging to another agent sharing that `sessions.json`.
  - confidence: 0.9
- **[P2] Detach manual store cleanup from default-agent overrides:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary operator-supplied path. Passing that id into the save path makes manual cleanup inherit the default agent's override instead of the global policy for the selected store.
  - confidence: 0.86
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This exact id comparison does not match the rest of agent/session routing, which normalizes requested and configured ids. A configured id such as `Ops` can resolve elsewhere but miss its maintenance override here and silently fall back to global policy.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging:** `src/config/sessions/store-maintenance.ts:162`
  - body: A raw spread can keep both global `pruneAfter` and agent `pruneDays` on the merged object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will not win over a global `pruneAfter`.
  - confidence: 0.83
