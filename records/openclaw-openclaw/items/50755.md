## Summary

Keep open. Current main still has only global session maintenance, so the PR is not obsolete, but the proposed branch exposes per-agent retention settings before the normal runtime write paths and manual cleanup target semantics are safe.

## What This Changes

The PR adds `agents.list[].maintenance`, extracts a shared maintenance schema, merges per-agent overrides into maintenance resolution, threads agent ids into cleanup/save paths, regenerates base config schema, and adds resolver tests.

## Best Possible Solution

Finish this PR or replace it with a version that resolves effective maintenance from one normalized config snapshot, applies it consistently to runtime writes and manual cleanup, keeps shared and arbitrary store cleanup on a safe policy, preserves legacy `pruneDays` precedence, and updates generated schema, docs, changelog, and focused tests together.

## Reproduction Assessment

Not applicable as a feature PR. The blocking review findings have high-confidence source-level reproduction paths through the main reply write path, cleanup target dedupe, explicit `--store` target handling, and existing agent-id normalization code.

## Solution Assessment

No. The direction is useful, but the current diff is not the best mergeable form because it advertises per-agent retention while leaving important enforcement paths global and introducing unsafe cleanup behavior for shared or arbitrary stores.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread per-agent maintenance into runtime writes:** `src/config/sessions/store.ts:292`
  - body: Adding `agentId` to save options is not enough while the main reply/session path pre-resolves `maintenanceConfig` from `cfg.session?.maintenance` and passes that global policy into `updateSessionStore`. Normal traffic would keep pruning/capping with the global policy even when an agent override is configured.
  - confidence: 0.88
- **[P1] Use global cleanup policy for shared stores:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` dedupes identical store paths, but this change applies the surviving target agent's override to the entire JSON file. In a shared-store setup, one agent's stricter override can prune another agent's sessions; shared targets should use a global policy or partition by entry ownership first.
  - confidence: 0.9
- **[P2] Keep explicit store cleanup off agent overrides:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, the resolver uses the default agent id as a placeholder for an arbitrary file path. Feeding that synthetic id into maintenance resolution makes explicit cleanup inherit the default agent's override and can delete entries from a store the default agent does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:139`
  - body: This lookup compares the configured id to the requested id verbatim, while existing routing normalizes ids before matching. A config entry such as `Ops` or `My Agent` can resolve elsewhere but miss its maintenance override here, silently falling back to global retention.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:157`
  - body: A spread merge can leave global `pruneAfter` and per-agent legacy `pruneDays` on the same object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override would not override the global age window.
  - confidence: 0.84
