## Summary

## What This Changes

The PR adds `agents.list[].maintenance` overrides, extracts a shared maintenance schema, adds per-agent resolution tests, and threads an agent id into session cleanup/save paths.

## Best Possible Solution

Rebase onto current main, design per-agent maintenance around normalized real agent ownership, keep shared/manual stores on global or entry-aware policy, and prove the behavior with focused tests plus a real CLI/config run.

## Reproduction Assessment

Yes for the PR review findings by source inspection, though no live reproduction was run. The PR's `opts.agentId` save path conflicts with current main's pre-resolved maintenanceConfig writes and target resolution that can be shared or synthetic.

## Solution Assessment

No. The schema extraction is a reasonable building block, but the implementation needs a current-main rebase and an owner-aware policy model before it is the maintainable way to add per-agent session retention.

Full review comments:

- **[P1] Propagate per-agent maintenance into normal writes:** `src/config/sessions/store.ts:399`
  - body: This only resolves per-agent maintenance when callers pass `opts.agentId`, but current runtime write paths precompute and pass a global `maintenanceConfig` from `sessionCfg?.maintenance`. Normal traffic would continue enforcing global retention while the new behavior mostly works from manual cleanup, so the advertised per-agent override is incomplete.
  - confidence: 0.88
- **[P1] Keep shared cleanup stores on a safe policy:** `src/commands/sessions-cleanup.ts:314`
  - body: `--all-agents` target resolution can dedupe multiple agents to the same store path, but this line chooses a single representative agent's mode/policy for the whole file. A stricter override on one agent can prune or cap sessions that belong to another agent sharing that `sessions.json`.
  - confidence: 0.9
- **[P2] Detach manual store cleanup from default-agent overrides:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary operator-supplied path. Passing that id into the save path makes manual cleanup inherit the default agent's retention override instead of the global maintenance policy for the selected store.
  - confidence: 0.86
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This exact id comparison does not match the rest of agent/session routing, which normalizes requested and configured ids. A configured id such as `Ops` can resolve elsewhere but miss its maintenance override here and silently fall back to global policy.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging:** `src/config/sessions/store-maintenance.ts:162`
  - body: A raw spread can keep both global `pruneAfter` and agent `pruneDays` on the merged object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will not win over a global `pruneAfter`.
  - confidence: 0.83
