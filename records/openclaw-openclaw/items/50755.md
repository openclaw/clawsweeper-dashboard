## Summary

Keep open: the feature is plausible, but this branch is incomplete and unsafe for merge because normal session writes still do not reliably use per-agent maintenance and cleanup can apply one agent's policy to shared or manual stores.

## What This Changes

Adds an `agents.list[].maintenance` config surface, shared maintenance schema, override merge logic, cleanup/save-path threading, generated schema updates, and per-agent resolution tests.

## Best Possible Solution

Rebase onto current main, resolve per-agent maintenance from normalized agent config for real agent-owned targets, keep shared/manual stores on global policy, and prove the behavior with focused tests plus a real CLI/config run.

## Reproduction Assessment

No live reproduction was run in this read-only review, but source inspection gives a high-confidence path for the review findings. The PR's target.agentId-based cleanup and save-option approach conflicts with current target resolution, normal write paths, and maintenance field precedence.

## Solution Assessment

No. The schema extraction is reasonable, but the current implementation is not the narrowest maintainable solution because it leaves normal writes global and applies per-agent policy to shared/manual stores.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Propagate per-agent maintenance into normal writes:** `src/config/sessions/store.ts:400`
  - body: This only changes save-time resolution when callers pass `opts.agentId`, but the normal runtime save paths still do not reliably pass an agent-specific policy. In current main, the reply path precomputes and passes a global `maintenanceConfig`, so regular traffic would continue enforcing global `session.maintenance` and the advertised per-agent prune/maxEntries/disk-budget behavior would only work from manual cleanup.
  - confidence: 0.88
- **[P1] Keep shared stores on the global policy:** `src/commands/sessions-cleanup.ts:315`
  - body: `--all-agents` target resolution dedupes identical store paths and keeps one representative agent. Resolving maintenance from that representative means a stricter override can prune or cap the entire shared JSON file, including sessions owned by other agents; shared stores need global policy or ownership-aware per-entry cleanup.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to --store:** `src/commands/sessions-cleanup.ts:377`
  - body: For `--store`, the resolver fills `target.agentId` with the default agent only as a placeholder for an arbitrary operator-supplied path. Passing that id into the save path makes manual cleanup inherit the default agent's override, so a standalone store can be pruned with the wrong policy.
  - confidence: 0.86
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: This exact `a.id === agentId` match disagrees with the rest of agent resolution, which normalizes configured and requested ids. A config entry like `"Ops"` or `"My Agent"` can resolve elsewhere but miss its maintenance override here and silently fall back to the global policy.
  - confidence: 0.86
- **[P2] Resolve deprecated pruneDays before merging:** `src/config/sessions/store-maintenance.ts:162`
  - body: A raw spread can keep both global `pruneAfter` and agent `pruneDays` on the merged object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will never win over a global `pruneAfter`, silently ignoring the agent-specific retention window.
  - confidence: 0.83
