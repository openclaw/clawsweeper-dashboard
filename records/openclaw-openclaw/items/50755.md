## Summary

Keep open: current main still has only global session maintenance, but this PR is not mergeable as-is because the proposed per-agent policy is only partially wired and can apply destructive cleanup settings to shared or arbitrary stores.

## What This Changes

The PR adds an `agents.list[].maintenance` config surface, shared maintenance schema extraction, per-agent/global merge logic, cleanup/store plumbing, generated schema updates, and resolver tests.

## Best Possible Solution

Land a rebased implementation that resolves per-agent maintenance from one normalized config snapshot, applies it only to owned per-agent stores/runtime writes, keeps shared or arbitrary store cleanup on a safe global policy unless entries are partitioned, preserves legacy `pruneDays` behavior, and updates docs, schema, generated config, and regression coverage together.

## Reproduction Assessment

Not applicable; this is a feature/config PR rather than a current-main bug report. The blocking issues are source-review paths through runtime writes, cleanup target resolution, shared-store dedupe, explicit `--store`, and agent-id normalization.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best mergeable form because it leaves key enforcement paths global and can apply agent-specific cleanup policies to stores that may not belong to that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread per-agent maintenance through runtime writes:** `src/config/sessions/store.ts:398-400`
  - body: The new resolver only uses per-agent config when callers pass an agent id, but normal reply/gateway/heartbeat writes still omit it or pass pre-resolved global maintenance. That leaves prune, cap, and disk-budget enforcement on the global policy during normal traffic.
  - confidence: 0.88
- **[P1] Keep shared-store cleanup on the global policy:** `src/commands/sessions-cleanup.ts:314`
  - body: `--all-agents` can collapse multiple agents to one store path, but this change resolves maintenance from whichever target agent survives. A stricter override on that agent can prune entries belonging to another agent in the same JSON file.
  - confidence: 0.9
- **[P2] Do not apply default-agent overrides to explicit stores:** `src/commands/sessions-cleanup.ts:374-377`
  - body: For `--store`, target resolution uses the default agent id as a placeholder for an arbitrary path. Passing that synthetic id into maintenance selection can apply the default agent's retention policy to a store it does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching maintenance overrides:** `src/config/sessions/store-maintenance.ts:141`
  - body: The lookup compares configured and requested ids verbatim, while existing agent resolution normalizes ids before matching. Mixed-case or human-readable ids can resolve elsewhere but silently miss their maintenance override here.
  - confidence: 0.86
- **[P2] Resolve legacy pruneDays before merging overrides:** `src/config/sessions/store-maintenance.ts:162`
  - body: A spread merge can keep global `pruneAfter` and per-agent legacy `pruneDays` together. Since resolution prefers `pruneAfter`, a supported per-agent `pruneDays` override would not replace the global age window.
  - confidence: 0.84
