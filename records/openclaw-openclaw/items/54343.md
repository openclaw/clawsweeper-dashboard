## Summary

Keep open for maintainer review: the goal may be valid, but the current branch is not safe to merge because it drops the existing sender authorization gate and still needs a maintainer policy decision on whether chat-command `/stop` should be stricter than direct abort RPCs.

## What This Changes

The PR changes the auto-reply `/stop` handler to require `operator.admin` for internal gateway clients by replacing the existing sender gate with `requireGatewayClientScopeForInternalChannel`.

## Best Possible Solution

If maintainers approve this hardening, keep `rejectUnauthorizedCommand(params, "/stop")`, import the internal scope helper, add an internal-channel `operator.admin` check immediately after the sender gate, and add focused scope tests plus a changelog entry.

## Reproduction Assessment

Yes for the PR defects: static inspection of the PR patch and current helper contract reproduces the missing import and the dropped external sender gate. The underlying authorization request is only partly reproducible as a product bug because direct abort RPCs still intentionally use `operator.write` for owned aborts.

## Solution Assessment

No. Replacing `rejectUnauthorizedCommand` is not the best solution; the maintainable path is an additive internal scope check after the existing sender authorization gate, pending maintainer approval of the stricter `/stop` policy.
