## Summary

Keep open: this is a plausible authorization-hardening proposal, but the current branch is incorrect because it calls a helper that is not imported/exported on current main and removes the existing sender authorization gate. The stricter `/stop` admin policy also needs maintainer confirmation and contributor real behavior proof before merge.

## What This Changes

The PR replaces the auto-reply `/stop` sender authorization check in `src/auto-reply/reply/commands-session-abort.ts` with an `operator.admin` gateway scope check.

## Best Possible Solution

If maintainers want this hardening, keep `rejectUnauthorizedCommand(params, "/stop")`, add the exported `requireGatewayClientScope` check immediately after it, and add focused `/stop` tests for internal scopes and external unauthorized senders.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR diff and current main shows the unimported/non-exported helper call and the dropped sender gate. No high-confidence current-main product bug is established because direct abort RPCs remain `operator.write` scoped.

## Solution Assessment

No. Replacing the sender gate is not the best fix; the safer shape is an additive gateway scope check after existing sender authorization, pending maintainer approval of the stricter policy.
