## Summary

Keep this PR open for maintainer review, but do not merge it as-is. The intended hardening may be useful, but the latest patch still calls an unimported helper and replaces the existing global sender authorization gate with an internal-channel-only scope helper, which creates a concrete external-channel authorization regression.

## What This Changes

The PR changes the auto-reply `/stop` handler in `src/auto-reply/reply/commands-session-abort.ts` to require internal gateway clients to have `operator.admin`.

## Best Possible Solution

If maintainers want this hardening, keep the existing sender authorization gate, import the internal scope helper, add the `operator.admin` check immediately after the existing gate for internal channels only, and add focused tests for internal `operator.write` blocked, internal `operator.admin` allowed, and external unauthorized senders still blocked.

## Reproduction Assessment

Yes for the PR defects: static inspection of the latest patch and current helper contract shows the missing import and the external sender-gate removal. The underlying request is only partly reproducible as a policy bug because current direct abort RPCs still intentionally accept `operator.write`.

## Solution Assessment

No. Replacing `rejectUnauthorizedCommand` is not the best way to solve the issue; the maintainable path is an additive internal scope gate after the existing sender gate, contingent on maintainer approval of the stricter `/stop` policy.
