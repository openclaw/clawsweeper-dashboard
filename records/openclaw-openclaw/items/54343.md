## Summary

Keep open: the authorization hardening goal is plausible, but this patch is not mergeable because it calls a helper that current main does not export and removes the existing sender authorization gate; the stricter /stop scope policy also needs maintainer confirmation.

## What This Changes

The PR replaces the /stop handler's sender authorization gate with an operator.admin gateway-client scope check in src/auto-reply/reply/commands-session-abort.ts.

## Best Possible Solution

If maintainers approve the stricter /stop policy, keep rejectUnauthorizedCommand(params, "/stop"), add the exported requireGatewayClientScope check immediately after it, and cover internal operator.write rejected, internal operator.admin allowed, and external unauthorized sender blocked.

## Reproduction Assessment

Yes for the PR defects: the live diff and current source show the unexported helper call and the removed sender gate. No high-confidence current-main product bug is established for the proposed operator.admin /stop policy because direct abort RPCs remain operator.write scoped.

## Solution Assessment

No. Replacing the sender gate is not the narrow maintainable fix; any scope hardening should be additive after existing sender authorization and should wait for maintainer confirmation of the /stop policy.
