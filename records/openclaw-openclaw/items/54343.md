## Summary

This PR should stay open, but it is not mergeable as written: it calls a gateway-scope helper that current main does not export or import, and it replaces the existing sender authorization gate instead of adding scope hardening after it. The requested `/stop` policy also needs maintainer confirmation because direct abort RPCs are still scoped as `operator.write`.

## What This Changes

The PR changes the auto-reply `/stop` command handler to require `operator.admin` gateway-client scope instead of using the existing sender authorization gate.

## Best Possible Solution

If maintainers approve stricter `/stop` hardening, preserve the existing sender gate, add an internal gateway `operator.admin` gate after it, and cover internal write/admin plus external unauthorized sender cases.

## Reproduction Assessment

Yes for the PR defects: the live diff and current source show the missing exported/imported helper and the removed sender gate. The underlying product policy bug is not fully established because current main still scopes direct abort RPCs as `operator.write`.

## Solution Assessment

No. Replacing the sender gate is not the narrow maintainable fix; any approved scope hardening should be additive after existing sender authorization and should include focused regression coverage.
