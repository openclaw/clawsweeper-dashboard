## Summary

Keep open: the PR targets a real config CLI bug, but the current patch globally disables numeric parent array detection, so documented bracket-index paths can materialize objects instead of arrays. The PR also needs stronger after-fix CLI proof before merge.

## What This Changes

The PR changes config set/patch path mutation calls to always preserve numeric-looking path segments as object keys and adds a Discord guild snowflake set regression test.

## Best Possible Solution

Carry bracket/index intent or schema-aware container metadata through config set so numeric Record keys stay objects while real array paths still create arrays, then prove the actual CLI flow and add regression coverage for both cases.

## Reproduction Assessment

Yes, source-reproducible: current `config set` can create an array for a missing numeric-looking Discord guild key, while the PR regression is also source-obvious because `parsePath` erases bracket-origin information before `setAtPath`. I did not execute the CLI in this read-only review.

## Solution Assessment

No. Preserving numeric Record keys is the right direction, but forcing `numericObjectKeys: true` globally is not the best fix because it breaks documented bracket-index array creation; the fix needs to preserve that distinction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve bracket-index array creation:** `src/cli/config-cli.ts:1751`
  - body: Setting `numericObjectKeys` to `true` for `setAtPath` makes the path writer ignore numeric next segments when choosing whether to auto-create `[]`. Because `parsePath` returns the same string for `0` from `.0` and `[0]`, a documented path like `openclaw config set agents.list[0].id main` on a config without `agents.list` would now build an object at `agents.list` instead of an array and fail schema validation. Keep bracket-index array creation working while preserving numeric Record keys.
  - confidence: 0.91
