## Summary

Keep open; the proposed fix targets a real config CLI bug, but the current patch would break bracket-index array creation because the path parser drops bracket provenance and `setAtPath` relies on numeric detection to create arrays.

## What This Changes

The PR changes config set/patch mutations to always preserve numeric-looking path segments as object keys and adds a Discord guild snowflake regression test.

## Best Possible Solution

Preserve numeric Record keys without losing bracket-index array creation, ideally by carrying bracket/index intent through path parsing or by using schema-aware container creation, then prove the actual CLI flow.

## Reproduction Assessment

Yes from source inspection: current `config set` passes `numericObjectKeys: false`, so a numeric snowflake segment can auto-create an array where Discord expects a record. I did not run the CLI because this review is constrained to read-only checkout inspection.

## Solution Assessment

No. The fix direction is right for Discord snowflake keys, but globally forcing object-key semantics removes the existing array auto-creation behavior for documented bracket-index paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve bracket-index array creation:** `src/cli/config-cli.ts:1751`
  - body: Setting this option to `true` globally makes `setAtPath` ignore every numeric next segment when choosing whether to auto-create `[]`. `parsePath` returns the same string for `0` whether it came from `.0` or `[0]`, so a documented command like `openclaw config set agents.list[0].id main` on a config without `agents.list` would now build an object at `agents.list` instead of an array and fail schema validation. Keep bracket-index array creation working while preserving numeric Record keys.
  - confidence: 0.91
