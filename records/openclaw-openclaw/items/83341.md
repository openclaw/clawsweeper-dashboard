## Summary

Keep open: the PR targets a real config-set bug, but forcing numeric object semantics globally drops bracket-index intent and can break documented array writes; the supplied proof is a standalone probe rather than the patched CLI.

## What This Changes

The PR changes `runConfigOperations` to pass `numericObjectKeys: true` for both set and patch mutations and adds a config-set regression test for Discord snowflake keys.

## Best Possible Solution

Preserve bracket/index intent or use schema-aware container selection so numeric Record keys remain object keys while real array paths still create arrays, with focused tests and redacted CLI proof.

## Reproduction Assessment

Yes, at source level: current `setAtPath` creates array parents for all-digit next segments in set mode, while `channels.discord.guilds` is a schema-backed object record. I did not execute the CLI because this review is read-only.

## Solution Assessment

No. The fix direction is right for snowflake keys, but the patch should preserve bracket-index intent or choose containers from schema instead of forcing every numeric-looking segment to be an object key.
