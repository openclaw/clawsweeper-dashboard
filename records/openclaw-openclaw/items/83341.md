## Summary

Keep open: the PR targets a real config set bug, but the current patch can regress documented bracket-index array paths because bracket notation is flattened before `setAtPath`, and the supplied proof does not exercise the patched CLI against an on-disk config.

## What This Changes

The PR changes the config CLI path writer to always treat numeric-looking path segments as object keys and adds a config set regression test for a Discord guild snowflake key.

## Best Possible Solution

Carry bracket/index intent through parsed path segments, or use schema-aware missing-container creation, so numeric Record keys become objects while documented bracket array paths remain arrays.

## Reproduction Assessment

Yes, at source level: current main passes `numericObjectKeys` only for patch mode, and `setAtPath` creates arrays for numeric-looking next segments while `channels.discord.guilds` is a string-keyed record. I did not execute the CLI because this review is read-only.

## Solution Assessment

No. The fix direction is right for Discord snowflake keys, but forcing all writes into object-key semantics is not the narrowest maintainable solution because bracket array intent is already flattened before `setAtPath`.
