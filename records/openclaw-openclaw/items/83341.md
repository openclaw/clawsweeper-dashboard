## Summary

Keep open: the PR targets a real source-reproducible config CLI bug, but the proposed global object-key setting would break documented bracket-index array paths and the supplied proof does not show the patched CLI writing a real config file.

## What This Changes

The PR changes config set/patch path mutation to always pass `numericObjectKeys: true` and adds a regression test for setting a Discord guild snowflake key.

## Best Possible Solution

Make `config set` distinguish numeric Record keys from bracket array indices, or use schema-aware container creation, while preserving array paths and adding focused regression coverage plus real CLI proof.

## Reproduction Assessment

Yes, at source level: current main creates array parents for all-digit next segments in `config set`, while `channels.discord.guilds` is a record. The PR-introduced regression is also source-clear because `parsePath` erases whether `0` came from `[0]` or `.0`.

## Solution Assessment

No. Preserving numeric Record keys is the right direction, but forcing object-key semantics globally breaks documented array path creation; the safer fix must preserve bracket/index intent or consult schema shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve bracket-index array creation:** `src/cli/config-cli.ts:1751`
  - body: Forcing `numericObjectKeys` to `true` here makes missing parents become objects even when the user used bracket syntax. Since `parsePath` stores both `.0` and `[0]` as the same string, a documented command like `openclaw config set agents.list[0].id main` on a config without `agents.list` would create an object where `AgentsSchema` requires an array. Preserve bracket-index intent while fixing numeric Record keys.
  - confidence: 0.91
