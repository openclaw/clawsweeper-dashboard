## Summary

Keep open: current main and the latest shipped tag still use a single Codex app-server shared-client slot that is cleared when the compatibility key changes, and `agentDir` is part of that key. This issue is already paired with the open fixing PR at https://github.com/openclaw/openclaw/pull/79513, so it should stay open until that PR or a replacement lands.

## What This Changes

Replace the Codex harness single global shared app-server client with key-aware reuse so agents with distinct `agentDir` values do not evict each other.

## Best Possible Solution

Land a keyed shared-client pool through the linked PR or a replacement, keeping `agentDir` in the key while closing stale secret/start generations; handle dynamic-tool fingerprint compaction separately if still needed.

## Reproduction Assessment

Yes. By source inspection, current main deterministically creates distinct keys for distinct `agentDir` values and then clears the only global shared client on key mismatch; the issue comments and duplicate report add live multi-agent symptoms.

## Solution Assessment

Yes, with one constraint. A keyed pool is the narrow maintainable fix because it preserves per-agent Codex home/auth isolation, but it must also evict stale secret-bearing or start-option generations rather than keeping every old key alive.
