## Label Justifications

- `P1`: The original bug affected real multi-agent Codex harness turns and could abort or wedge active user workflows.
- `impact:session-state`: The issue describes active runs and app-server bindings becoming inconsistent across agent directories.
- `impact:message-loss`: Related reproductions show active turns going silent or failing when another agent starts a Codex app-server path.
- `impact:crash-loop`: The issue centers on app-server client closes, process churn, and restart/retry failures under multi-agent load.

## Summary

Close: current main and the latest stable release use a keyed Codex app-server shared-client registry, so different agent dirs no longer evict each other. The old follow-up PR is closed unmerged, and the remaining startup retry/readiness failure is already tracked separately.

## What This Changes

The issue requests replacing the Codex app-server single shared-client slot with per-compatible-key reuse so agents with different `agentDir` values do not evict each other's app-server clients.

## Best Possible Solution

Close this issue as implemented and keep the narrower startup retry/readiness work in https://github.com/openclaw/openclaw/issues/83959.

## Reproduction Assessment

Yes. The historical source had one shared `client`/`promise`/`key` slot while `agentDir` was part of the key, and the current regression test captures the formerly failing two-agent-dir path.

## Solution Assessment

Yes. The keyed registry is the narrow maintainable fix for cross-agent eviction because it keeps `agentDir` isolation while preventing unrelated keys from closing each other's active clients; startup retry warm-up remains a separate follow-up.
