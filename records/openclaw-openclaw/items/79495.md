## Summary

Current main still has the reported cross-agent eviction path: the Codex app-server shared client is a single global entry, while the compatibility key includes `agentDir`, so alternating compatible agents can close each other's warm client.

## What This Changes

This issue asks for the Codex app-server shared client to be pooled by compatible start-options key instead of using one singleton that is cleared when agents have different `agentDir` values.

## Best Possible Solution

Keep `agentDir` in the compatibility key, replace the singleton with a keyed shared-client pool, add alternating-agent regression coverage, and leave dynamic-tool fingerprint compaction as a separate follow-up unless it stays very small.

## Reproduction Assessment

Yes by source inspection: current main deterministically creates distinct keys for distinct `agentDir` values, then clears the only global shared client when that key changes. I did not run a live Raspberry Pi or multi-agent gateway scenario.

## Solution Assessment

Yes: a keyed pool is the narrow maintainable fix because it preserves per-agent `HOME`/`CODEX_HOME`/auth isolation while avoiding cross-agent eviction. Removing `agentDir` from the key would be riskier because current docs and auth bridging depend on per-agent isolation.
