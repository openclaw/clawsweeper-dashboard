## Summary

Keep open: current main still has the reported Codex shared-client singleton eviction behavior, and https://github.com/openclaw/openclaw/pull/79513 is an open closing PR for this issue rather than a merged fix. The latest discussion also adds production validation on `openclaw@2026.5.12-beta.8`, so this remains an active narrow Codex plugin bug.

## What This Changes

Request to replace the Codex harness shared app-server singleton with keyed per-start/auth/agent client pooling so different `agentDir` values do not evict each other's warm clients.

## Best Possible Solution

Land a narrowed Codex plugin fix that pools warm shared clients by compatible key for distinct `agentDir` values while actively closing stale auth/env/start generations and preserving per-agent `CODEX_HOME` isolation; treat dynamic-tool fingerprint compaction as a separate follow-up unless explicitly scoped.

## Reproduction Assessment

Yes by source inspection: current main deterministically creates distinct keys for distinct `agentDir` values and then clears the only global shared client when the key changes. The discussion adds real production reports, but I did not run a live multi-agent gateway reproduction in this review.

## Solution Assessment

Yes, the keyed-pool direction is the narrow maintainable fix because it preserves per-agent auth/session isolation while avoiding cross-agent eviction. The implementation should also evict stale secret/env/start generations so token rotation and cleanup behavior remain intact.
