## Summary

Current main still has the reported singleton eviction path, and an open linked fix PR already targets it, so this should remain open until https://github.com/openclaw/openclaw/pull/79513 lands or is replaced. The related active-turn report at https://github.com/openclaw/openclaw/issues/80618 adds useful reproduction evidence but was closed into this canonical track.

## What This Changes

The issue requests replacing the Codex app-server shared-client singleton with per-key pooling so agents with distinct agentDir values do not evict each other's warm clients.

## Best Possible Solution

Land the keyed shared-client pool from the linked PR, preserving agentDir/auth/start-option isolation, and leave compact dynamic-tool fingerprints as a separate follow-up unless it stays clearly scoped.

## Reproduction Assessment

Yes by source inspection: current main computes distinct keys for distinct agentDir values and clears the only global shared client when that key changes. I did not rerun a live Raspberry Pi or multi-agent gateway scenario during this read-only review.

## Solution Assessment

Yes: pooling by compatibility key is the narrow maintainable fix because it keeps per-agent HOME/CODEX_HOME/auth isolation while preventing unrelated agent dirs from closing each other's clients. Removing agentDir from the key would be riskier.
