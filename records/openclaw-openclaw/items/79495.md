## What This Changes

The issue asks for Codex app-server shared clients to be pooled per compatible start-options key so different `agentDir` values do not evict each other.

## Best Possible Solution

Land a repaired keyed shared-client pool that preserves per-agent isolation and stale auth/env/start-option cleanup, then close this canonical issue through the merged PR.

## Reproduction Assessment

Yes, by source inspection. Current main deterministically derives different keys for different `agentDir` values and clears the single global shared client when the key changes; I did not run the live multi-agent Codex harness workload.

## Solution Assessment

Yes, the keyed-pool direction is the narrow maintainable fix, as long as it keeps `agentDir` in the compatibility key and preserves cleanup for rotated auth/env/start options. The best path is to review and land the linked PR after that blocker is addressed.
