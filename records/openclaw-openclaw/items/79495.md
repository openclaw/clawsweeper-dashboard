## Summary

Keep open. Current main still has the reported single-slot Codex app-server shared-client eviction path, and an open linked fix PR already targets the same bug, so this issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue reports that the Codex harness app-server shared client is a single process-global slot, so changing start-option keys across agents can evict another agent's active client.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79513 or an equivalent keyed shared-client pool that preserves per-agent isolation, adds alternating-agent regression coverage, and leaves compact dynamic-tool fingerprints to a separate scoped follow-up.

## Reproduction Assessment

Yes. By source inspection on current main, distinct agentDir values produce distinct compatibility keys and getSharedCodexAppServerClient clears the only global client when the key changes; I did not run the live multi-agent gateway scenario.

## Solution Assessment

Yes. A keyed pool is the narrow maintainable fix because it keeps per-agent HOME/CODEX_HOME/auth isolation while preventing unrelated active clients from being closed on key mismatch.
