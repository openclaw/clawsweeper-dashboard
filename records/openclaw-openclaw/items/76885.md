## Summary

Keep this PR open: the direct `stageSystemdService` preservation cases are improved, but the doctor/install repair path can still render file-owned provider secrets as inline `Environment=` values after preserving them in `gateway.systemd.env`.

## What This Changes

The PR merges existing `gateway.systemd.env` entries with state-dir `.env` entries, clears tracked OpenClaw-managed inline keys, adds systemd regression tests, and records a changelog fix for #76860.

## Best Possible Solution

Preserve operator-owned env-file keys across re-stage while ensuring those same keys are excluded from inline systemd `Environment=` output, with regression coverage through the doctor/install planning path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `readSystemdServiceExecStart` merges EnvironmentFile keys into the command env, `buildGatewayInstallPlan` preserves safe existing keys, and the PR then keeps non-dotenv keys in the inline environment when rendering the unit.

## Solution Assessment

No. The direction is right, but the implementation needs to carry file-owned key knowledge into inline filtering or otherwise exclude merged env-file keys from `Environment=` before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep file-only secrets out of Environment=:** `src/daemon/systemd.ts:596`
  - body: When doctor rebuilds the service plan from an existing unit, provider keys loaded from `gateway.systemd.env` are preserved into `environment`. This merge writes those keys back to the env file, but `environmentSansDotEnvEntries` only removes state-dir `.env` keys, so the same provider secret can also be emitted as an inline `Environment=` value in the unit. Please filter inline env against the keys retained in the env file, or carry file-only source metadata through staging, so env-file secrets stay file-owned.
  - confidence: 0.9
