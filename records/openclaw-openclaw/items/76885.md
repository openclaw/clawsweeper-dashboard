## Summary

Keep this PR open: it addresses a real destructive systemd env-file wipe, but the proposed merge can preserve stale managed keys that systemd will prefer over fresh inline service values; it also lacks the required user-facing changelog entry.

## What This Changes

The PR updates systemd staging to merge existing `gateway.systemd.env` values with state-dir `.env` values and adds a regression test for preserving operator-added provider keys.

## Best Possible Solution

Preserve unmanaged operator-added env-file keys while dropping any preserved key that staging will emit inline, then add focused regression coverage and an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes, at source level. Current main writes `gateway.systemd.env` from only `dotenvVars`, so a pre-existing operator key absent from the state-dir `.env` is discarded on re-stage; I did not execute a live systemd run because this review was read-only.

## Solution Assessment

No. The PR uses the right preservation direction, but preserving every existing key lets stale env-file values override fresh inline values that current staging intentionally emits.

## Review Findings
