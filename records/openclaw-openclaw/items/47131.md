## Summary

Keep open: current main still supports outbound reply tags and reply-threading policy, but there is no inbound parser that strips trailing `reply in thread`/`rit` text or carries a per-turn force-thread flag; the linked implementation PR is closed and unmerged.

## What This Changes

Add an inbound directive that strips trailing `reply in thread`/`rit`/`thread`/`in thread` from user messages and forces the assistant reply to target the inbound message thread.

## Best Possible Solution

Add a shared auto-reply inbound directive parser that strips only validated trailing triggers, carries a typed per-turn force-thread intent, feeds the existing reply payload threading path, and covers supported channel behavior with focused tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence even though this is a feature request rather than a bug. On current main, trailing `rit`/`reply in thread` text remains in inbound bodies, no context carrier exists, and reply payload construction has no inbound-force parameter.

## Solution Assessment

Yes. A narrow shared inbound directive that translates into the existing reply-to-current payload path is the maintainable approach; channel-specific parsing or config-first work would fragment the behavior.
