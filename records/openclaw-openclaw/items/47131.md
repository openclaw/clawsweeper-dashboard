## Summary

Keep this issue open. Current main still supports outbound assistant reply tags and reply-threading policy, but it has no shared inbound parser that strips trailing `reply in thread` / `rit` / `thread` / `in thread` text or carries that user intent into reply payload construction; the related implementation PR is closed and unmerged.

## What This Changes

Add a built-in inbound directive that strips trailing reply-in-thread trigger phrases and forces the outbound response to reply to the current inbound message.

## Best Possible Solution

Add suffix-only inbound directive extraction in the shared auto-reply context path, carry a typed per-turn flag, and map it onto the existing current-message reply-threading path without channel-specific parsing.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new user-facing directive request. Source inspection gives high confidence that current main does not implement the requested inbound preprocessing path.

## Solution Assessment

Yes. A shared inbound directive feeding the existing reply payload threading path is the narrow maintainable approach; LLM prompt instructions or channel-specific parsing would be less reliable and less consistent.
