## Summary

Keep this issue open. Current main still only supports outbound assistant reply tags, implicit reply-threading policy, and channel-specific threading behavior; it does not implement a shared inbound trailing directive that strips `reply in thread` or `rit` before the model sees it and forces reply-to-current. The related implementation PR is closed and unmerged, so it remains useful source context rather than a reason to close.

## What This Changes

The issue requests a built-in inbound directive that strips trailing reply-in-thread trigger phrases from user messages and forces the outgoing response to reply to the current inbound message.

## Best Possible Solution

Implement this in the shared auto-reply inbound pipeline: strip deliberate trailing trigger text from every model/command body field, including pre-populated `BodyForAgent`; carry a typed per-turn flag on `MsgContext`; and translate that flag through the existing reply payload threading path as an explicit reply-to-current request. Keep channel-specific thread settings separate, use a conservative trigger set, and update docs/changelog when the behavior ships.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: pass inbound text such as `hello rit` through `finalizeInboundContext`; current main preserves the suffix in prompt/command fields and has no context flag or payload-builder parameter that can force reply-to-current from that inbound text.

## Solution Assessment

Yes, with constraints. A narrow shared inbound directive that feeds the existing reply payload threading path is the maintainable direction; channel-specific parsing or merging the old PR as-is would be weaker because the closed PR had concrete review concerns to address.
