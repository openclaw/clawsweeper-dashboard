## Summary

Keep open. Current main still supports assistant-output reply tags and per-channel reply policies, but not a shared inbound suffix directive; the global trigger vocabulary needs maintainer product approval before implementation.

## What This Changes

The issue requests a built-in inbound suffix directive that strips phrases like `reply in thread` or `rit` from user input and forces the next response into the originating thread.

## Best Possible Solution

If maintainers accept the feature, add a typed suffix-only directive in shared inbound finalization, carry it into the existing reply-threading payload path, document the approved trigger policy, and cover stripping plus reply targeting with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the shared inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and whether this belongs in core or behind a stronger plugin/API hook. The maintainable implementation should reuse shared inbound finalization and the existing reply-threading payload path, not channel-specific prompt instructions.
