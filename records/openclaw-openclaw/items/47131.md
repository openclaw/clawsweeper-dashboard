## Summary

Keep open. Current main still has outbound reply tags and reply-threading policy, but it does not implement the requested shared inbound suffix directive; because this would add magic user-facing trigger words, maintainers should decide the vocabulary and config policy before implementation.

## What This Changes

Add a built-in inbound directive that strips trailing phrases like `reply in thread` or `rit` from user messages and forces the response to reply to the current message.

## Best Possible Solution

If accepted, add shared suffix-only extraction in the inbound auto-reply path, carry typed per-turn intent, map it onto the existing current-message reply-threading machinery, and cover it with focused tests and docs after maintainers choose the trigger policy.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the inbound preprocessing path is not implemented.

## Solution Assessment

Unclear until maintainers confirm the magic suffix vocabulary and whether trigger configuration belongs in core config. If accepted, the narrow maintainable approach is a shared inbound directive feeding the existing reply-threading path rather than channel-specific prompt instructions.
