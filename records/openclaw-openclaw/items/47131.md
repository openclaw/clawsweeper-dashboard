## Summary

Keep this issue open. Current main still has outbound reply tags and reply-threading policy, but no shared inbound parser that strips trailing `reply in thread` / `rit` / `thread` / `in thread` text and carries that user intent into reply payload construction; the related implementation PR is closed and unmerged.

## What This Changes

Request a built-in inbound suffix directive that strips phrases such as `reply in thread` or `rit` before the agent run and forces the response to thread to the current inbound message.

## Best Possible Solution

If accepted, add suffix-only extraction in the shared inbound auto-reply path, carry a typed per-turn flag, and map it onto the existing current-message reply-threading machinery with focused tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new user-facing directive request. Source inspection gives high confidence that current main does not implement the requested inbound preprocessing path.

## Solution Assessment

Unclear until a maintainer confirms the default trigger vocabulary and whether it should be configurable. Architecturally, a shared inbound directive feeding the existing reply-threading path is the narrow maintainable direction if the feature is accepted.
