## Summary

Keep open. Current main and v2026.5.22 still support reply threading through channel `replyToMode` and assistant-output `[[reply_to_*]]` tags, but the requested shared inbound suffix directive remains missing; the blocker is maintainer product/API judgment around global magic phrases such as bare `thread`.

## What This Changes

Request a built-in inbound suffix directive that strips phrases such as `reply in thread` or `rit` from user text and forces the assistant reply to target the inbound message thread.

## Best Possible Solution

If maintainers accept the feature, define a narrow suffix-only inbound directive contract, carry it as typed intent from shared inbound finalization into existing reply payload resolution, document the approved trigger policy, and test stripping plus reply targeting.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive, not broken existing behavior. Source inspection on current main gives high confidence that the shared inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and core/plugin boundary. The maintainable path should reuse shared inbound finalization and the existing reply-threading payload path rather than channel-specific prompt instructions.
