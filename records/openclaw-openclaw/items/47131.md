## Summary

Keep this issue open. Current `main` still has outbound assistant reply tags and reply-threading policy, but no inbound parser that strips trailing `reply in thread`/`rit`/`thread`/`in thread` text or carries a per-turn force-thread flag from user input; the linked implementation PR is closed and unmerged.

## What This Changes

Request an inbound user directive that strips trailing thread-trigger phrases from incoming messages and forces the assistant response to reply to the current message thread.

## Best Possible Solution

Add the directive in the shared auto-reply inbound pipeline and translate it through the existing reply payload threading path.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: pass inbound text such as `hello rit` through `finalizeInboundContext` on current `main`; the suffix remains in the prompt/command bodies and no per-turn flag exists to force reply-to-current.

## Solution Assessment

Yes. A narrow shared inbound directive feeding the existing reply payload path is the maintainable direction; channel-specific parsing or config-first work would be a less consistent fix.
