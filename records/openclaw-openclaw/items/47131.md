## Summary

Keep this issue open. Current `main` still supports outbound assistant reply tags and existing reply-threading policy, but it does not strip inbound trailing `reply in thread`/`rit` text or carry a per-turn flag that forces reply-to-current; the linked implementation PR is closed and unmerged source context.

## What This Changes

The issue requests a built-in inbound directive that strips trailing reply-in-thread trigger phrases from user messages and forces the outgoing response to reply to the current inbound message.

## Best Possible Solution

Implement the directive in the shared auto-reply inbound pipeline, carry a typed per-turn flag through `MsgContext`, and translate it through the existing reply payload threading path with docs, changelog, and focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: pass inbound text such as `hello rit` through `finalizeInboundContext`; current `main` preserves the suffix in prompt/command fields and has no flag or payload-builder input to force reply-to-current from that inbound text.

## Solution Assessment

Yes, with constraints. A narrow shared inbound directive feeding the existing reply payload path is the maintainable direction; channel-specific parsing or merging the old PR as-is would be weaker because the trigger and reply-policy semantics still need tightening.
