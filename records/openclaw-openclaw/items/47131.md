## Summary

Keep open: current main and v2026.6.10 still lack the inbound natural-language reply-in-thread suffix directive. The only direct implementation PR is closed unmerged, and the remaining blocker is product/API shape rather than conservative cleanup.

## What This Changes

The issue requests an inbound user directive that strips suffixes like `reply in thread` or `rit` and forces the outbound response to reply to the current message thread.

## Best Possible Solution

Define a maintainer-approved per-message reply override contract first, then implement a narrow shared inbound directive or plugin/API seam with regression coverage for prompt stripping and supported threaded channels.
