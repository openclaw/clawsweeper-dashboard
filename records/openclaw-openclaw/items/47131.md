## Summary

Keep open. Current main still has assistant-output reply tags and channel reply policies, but not a shared inbound suffix directive; accepting global phrases like bare `thread` needs maintainer product approval.

## What This Changes

The issue requests a built-in inbound message suffix directive such as `reply in thread` or `rit` that strips the suffix before the model sees it and forces the response to reply to the inbound message.

## Best Possible Solution

If maintainers accept the feature, add a typed suffix-only inbound directive in shared inbound finalization, carry it into the existing reply-threading payload path, document the approved trigger policy, and test stripping plus reply targeting.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the shared inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and whether this belongs in core or behind a stronger plugin/API hook. The maintainable path should reuse shared inbound finalization and existing reply-threading payloads, not channel-specific prompt instructions.
