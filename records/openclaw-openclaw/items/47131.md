## Summary

Keep open. Current main and the latest release still have assistant-output reply tags and channel `replyToMode` policy, but not the requested shared inbound suffix directive; the remaining blocker is maintainer product/API judgment around global magic phrases like bare `thread`.

## What This Changes

Add a built-in inbound user directive so suffixes like `reply in thread` or `rit` are stripped before the agent sees them and force the response to thread to the inbound message.

## Best Possible Solution

If maintainers accept the feature, add a typed suffix-only inbound directive in shared inbound finalization, carry it into existing reply-threading payload resolution, document the approved trigger policy, and test stripping plus reply targeting.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the shared inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and whether this belongs in core or behind a stronger plugin/API hook. The maintainable implementation should reuse shared inbound finalization and the existing reply-threading payload path, not channel-specific prompt instructions.
