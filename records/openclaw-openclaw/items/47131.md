## Summary

Keep open. Current main still supports outbound reply tags and implicit reply-threading policy, but it does not implement a shared inbound suffix parser that strips `reply in thread` / `rit` / `thread` / `in thread` and carries that intent into reply payload construction; the linked implementation PR is closed and unmerged.

## What This Changes

The issue requests a built-in inbound suffix directive that strips phrases like `reply in thread` and forces replies into the current message thread.

## Best Possible Solution

If accepted, add suffix-only extraction in the shared inbound auto-reply path, carry a typed per-turn flag, and map it onto the existing current-message reply-threading machinery with focused tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence the inbound preprocessing path is not implemented.

## Solution Assessment

Unclear until maintainers confirm the magic trigger vocabulary and whether it should be configurable. Architecturally, a shared inbound directive feeding the existing reply-threading path is the narrow maintainable direction if the feature is accepted.
