## Summary

Keep open. Current main still has outbound reply tags, channel reply policies, and some channel-specific thread config, but it does not implement the requested shared inbound suffix directive; global trigger words such as `thread` need a maintainer product decision before code work is safe.

## What This Changes

Add a built-in inbound suffix directive that strips phrases like `reply in thread` or `rit` from user input and forces the response to reply to the triggering message.

## Best Possible Solution

If maintainers accept the feature, add a typed suffix-only inbound directive in shared inbound finalization, route it into the existing reply-threading payload path, document the approved trigger policy, and cover `BodyForAgent` plus command-body stripping behavior with focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and whether this should be built into core or exposed through a stronger plugin/API hook. The maintainable implementation would reuse shared inbound finalization and existing reply-threading payload logic, not channel-specific prompt instructions.
