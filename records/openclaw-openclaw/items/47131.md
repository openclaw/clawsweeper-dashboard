## Summary

Keep open. Current main still has outbound reply tags and reply-threading policy, but no shared inbound suffix directive; accepting global magic trigger words such as `thread` needs maintainer product judgment before implementation.

## What This Changes

The issue requests a built-in inbound suffix directive that strips phrases like `reply in thread` or `rit` from user messages and forces the outbound response into the current thread.

## Best Possible Solution

If accepted, add a typed suffix-only inbound directive in shared inbound finalization, feed existing reply-threading payload logic, document the chosen trigger policy, and cover agent/command stripping tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new user-facing directive. Source inspection on current main gives high confidence that the inbound suffix parser and carrier field do not exist.

## Solution Assessment

Unclear until maintainers approve the trigger vocabulary and config policy. The maintainable path would be a shared inbound directive feeding the existing reply-threading path, not channel-specific prompt instructions.
