## Summary

Keep open: current main still rejects object-form SecretRefs on core MCP `env` and `headers` values, and PR #69417 already tracks the exact implementation path rather than needing duplicate work.

## What This Changes

Request SecretRef object support for core `mcp.servers.*.env.*` and `mcp.servers.*.headers.*` values so MCP credentials can be kept out of plaintext config.

## Best Possible Solution

Finish and land PR #69417 or an equivalent narrow replacement covering schema/types, generated schema, runtime resolution, target registry, docs, and regression tests for core MCP env/header SecretRefs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a structured SecretRef under `mcp.servers.demo.env.API_KEY` or `mcp.servers.demo.headers.Authorization`; current Zod, generated schema, and exported types only accept primitive values there.

## Solution Assessment

Yes as a direction. The maintainable fix is to reuse the existing SecretInput schema, resolver, and target-registry model for core MCP values; PR #69417 already follows that path and should be reviewed rather than duplicating the work.
