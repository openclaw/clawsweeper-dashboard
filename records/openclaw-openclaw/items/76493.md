## Summary

Keep open: current main still rejects object-form SecretRefs on top-level MCP env/header values, and the active implementation path is https://github.com/openclaw/openclaw/pull/69417 rather than a reason to close or queue duplicate work.

## What This Changes

Allow structured SecretRef values in top-level `mcp.servers.*.env.*` and `mcp.servers.*.headers.*` config values instead of only scalar literals.

## Best Possible Solution

Land a refreshed version of https://github.com/openclaw/openclaw/pull/69417 that reuses the existing SecretInput registry/runtime model, resolves only active MCP env/header surfaces, updates audit/configure/docs/tests, and receives security-owner review.

## Reproduction Assessment

Yes, at source level. A structured SecretRef under `mcp.servers.demo.env.API_KEY` or `mcp.servers.demo.headers.Authorization` is rejected by the current schema/types, and object values would be dropped by MCP launch coercion if they bypassed validation.

## Solution Assessment

Yes as a direction, but it is not on main yet. The narrow maintainable solution is to extend the existing SecretInput schema, target registry, runtime resolution, audit/configure, and docs model rather than adding an MCP-only secret path.
