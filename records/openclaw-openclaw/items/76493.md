## Summary

Keep this open: current main still rejects object-form SecretRefs on core `mcp.servers` env/header values, and the already-open PR https://github.com/openclaw/openclaw/pull/69417 is the right implementation path to finish rather than closing this issue or queuing duplicate work.

## What This Changes

Allow SecretRef objects for core `mcp.servers.*.env.*` and `mcp.servers.*.headers.*` values so MCP credentials can live outside plaintext `openclaw.json`.

## Best Possible Solution

Finish and land https://github.com/openclaw/openclaw/pull/69417 after its changelog blocker and SecretRef/security-owner review are handled, covering schema, runtime resolution, target registry, docs, and regression tests for env/header values.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a structured SecretRef under `mcp.servers.demo.env.API_KEY` or `mcp.servers.demo.headers.Authorization`; current Zod, generated schema, and exported types all accept only primitive values there, though I did not run a live config command because this review is read-only.

## Solution Assessment

Yes as a direction. The narrow maintainable solution is to reuse the existing `SecretInput` schema/resolver/target-registry model for core MCP env/header values, not add ad hoc MCP-only secret resolution; the open PR already follows that shape but still needs review cleanup.
