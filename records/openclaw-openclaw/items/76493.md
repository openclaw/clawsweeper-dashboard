## Summary

Keep open: current main still rejects structured SecretRef objects in top-level MCP env/header config, and the active implementation path is an open, conflicting PR that still needs maintainer/security compatibility review before this issue can close.

## What This Changes

Request SecretRef object support in top-level `mcp.servers.*.env.*` and `mcp.servers.*.headers.*` so MCP credentials do not have to stay plaintext in `openclaw.json`.

## Best Possible Solution

Land a maintainer/security-reviewed MCP SecretRef implementation that widens the schema, resolves only active env/header refs through the runtime secret snapshot, preserves literal compatibility, respects stdio env filtering, and updates docs plus focused tests.

## Reproduction Assessment

Yes, at source level. Current main’s MCP schema and exported type contract accept only scalar env/header values, and the launch coercion path drops object values such as structured SecretRefs unless they are resolved earlier.

## Solution Assessment

Unclear as an automated fix path. The direction belongs in core because this is a missing credential surface, but the safest implementation needs maintainer/security decisions on literal-string compatibility and MCP env filtering before merge.
