## Summary

Keep open: current main still rejects object-form SecretRefs on core MCP env/header values, and https://github.com/openclaw/openclaw/pull/69417 is the active implementation path rather than a reason to close or queue duplicate work.

## What This Changes

The issue asks core `mcp.servers.*.env.*` and `mcp.servers.*.headers.*` values to accept SecretRef objects instead of only primitive literals.

## Best Possible Solution

Land a refreshed version of https://github.com/openclaw/openclaw/pull/69417 that reuses the existing SecretInput registry/runtime model, resolves only active MCP env/header surfaces, updates audit/configure/docs/tests, and receives security-owner review.

## Reproduction Assessment

Yes, at source level. A structured SecretRef under `mcp.servers.demo.env.API_KEY` or `mcp.servers.demo.headers.Authorization` is rejected by the current schema/types, and object values would be dropped by the MCP launch coercion if they bypassed validation.

## Solution Assessment

Yes as a direction, but not yet on main. The narrow maintainable solution is to reuse the existing SecretInput schema, target registry, runtime resolution, audit/configure, and docs model rather than adding an MCP-only secret path.
