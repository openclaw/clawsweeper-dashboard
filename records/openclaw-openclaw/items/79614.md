## Summary

Keep open. Current main now has a foreground reply freshness fence for one stale-final delivery class, but it still lacks the requested host-owned finalize-time latest-user snapshot, complaint flag, re-anchor prompt, and config contract.

## What This Changes

Request a host-level stale-reply re-anchoring contract that rechecks the newest user message at finalize time, detects stale-reply complaints, and can force one bounded re-draft.

## Best Possible Solution

Define an explicit core freshness contract around finalization and delivery that composes with the existing foreground fence, documents config and observability, and adds focused Telegram/direct-chat, complaint re-anchoring, and overlap tests.

## Reproduction Assessment

No high-confidence live current-main stale visible reply reproduction was established. Source inspection does show the requested finalize-time latest-user signal and complaint flag are absent, while current tests cover only foreground delivery suppression.

## Solution Assessment

Unclear. A bounded finalize-time re-draft is plausible, but maintainers should decide whether the right durable fix is finalize re-anchoring, an extension of the current delivery fence, queue policy changes, or a plugin API contract with explicit config defaults.
