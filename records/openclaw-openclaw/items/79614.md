## Summary

Keep open. Current `main` has default active-run steering and a bounded `before_agent_finalize` revision hook, but it still does not provide the requested host-owned latest-user snapshot, complaint signal, or built-in stale-reply re-anchor pass; the related open reports and hook PR are adjacent, not a complete superseding implementation.

## What This Changes

Requests a host-owned stale-reply recheck that snapshots the newest user message at finalization, detects stale-reply complaints, and can re-anchor or redraft once via config.

## Best Possible Solution

Define a host-owned finalization freshness contract that exposes latest inbound state and a conservative complaint signal to the harness, with bounded redraft behavior, observability, config/schema docs, and focused Pi/Codex regression tests.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main lacks the requested finalize-time latest-user snapshot and complaint flag, while the report supplies plausible external operator evidence.

## Solution Assessment

Unclear until maintainer product review. A finalize-time freshness signal fits the existing `before_agent_finalize` revision gate, but default complaint detection and redraft semantics should be designed as a runtime/config contract rather than copied directly from a local regex plugin.
