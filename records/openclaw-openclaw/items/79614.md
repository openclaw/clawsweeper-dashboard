## Summary

Keep open. Current main has active-run steering and a foreground delivery fence, but it still does not provide the requested host-owned latest-user snapshot, complaint flag, or built-in bounded re-anchor contract at finalization time.

## What This Changes

Request a host-owned stale-reply recheck that snapshots the newest user message before finalization, detects stale-reply complaints, and triggers one bounded re-anchor pass without relying on a local plugin.

## Best Possible Solution

Define an explicit host-owned freshness contract at the appropriate finalization or pre-delivery boundary, then add config/schema/docs, observability, and focused Pi/Codex regression coverage while coordinating with the related stale-intent and stale-delivery trackers.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection does confirm that the requested finalize-time latest-user snapshot, complaint flag, and built-in re-anchor contract are still absent.

## Solution Assessment

Unclear until maintainer design review. A bounded finalize-time recheck is plausible because `before_agent_finalize` can request one extra pass, but the default policy and config semantics are broader than a narrow bug fix.
