## Summary

Keep open. Current main still has the same source-level gap: cron can preserve a mixed-case Matrix target only when `currentDeliveryContext.to` reaches `createCronTool`, but the Claude CLI MCP loopback path still scopes tools with session key, channel, account, and owner context only. That leaves implicit cron delivery able to fall back to `parseAgentSessionKey`, whose documented contract lowercases returned routing values.

## What This Changes

The issue asks OpenClaw to preserve canonical mixed-case Matrix room IDs when `mcp__openclaw__cron action=add` uses `sessionTarget: "current"` through the Claude CLI MCP loopback path without explicit delivery.

## Best Possible Solution

Carry canonical current delivery target and thread context through the Claude CLI MCP loopback path into `createOpenClawTools`/`createCronTool`, or resolve the canonical target from persisted session metadata before cron falls back to session-key parsing. Keep the seam generic for case-sensitive channel targets and add regression coverage for mixed-case Matrix room IDs through loopback cron creation.

## Reproduction Assessment

Yes. The issue provides concrete v2026.4.27 reproduction data, stored job JSON, and Matrix errors, and current main still shows the missing loopback target propagation that allows the lowercased session-key fallback.

## Solution Assessment

Yes. Extending the existing delivery-context seam into loopback tool resolution is the narrowest maintainable direction; a Matrix-specific cron special case would leave the generic case-sensitive target problem in place.

## Repair Work Prompt

Fix the current-session cron Matrix room-id casing regression reported in this issue. When Claude CLI bundle-MCP/loopback invokes `mcp__openclaw__cron action=add` with `sessionTarget: "current"` and no explicit `delivery`, the cron tool must receive the canonical current delivery target instead of falling back to the lowercased session key. Keep the fix generic for case-sensitive channel targets, not Matrix-specific. Likely approach: thread a case-preserving current channel target and thread id from the CLI runner into the loopback MCP server config/env, parse those headers without lowercasing in `resolveMcpRequestContext`, include them in the loopback tool cache key, forward them through `resolveGatewayScopedTools`, and pass them into `createOpenClawTools`/`createCronTool` via the existing `currentDeliveryContext` seam. Use closed PR #75303 as source context, but verify against current main before reusing anything. Preserve owner-only loopback bearer behavior, do not broaden plugin/tool policy scope, do not add Matrix-specific cron hacks, and do not touch dependencies. Add focused regression tests for mixed-case Matrix target propagation through loopback tool resolution plus the existing cron context behavior, and add a user-facing `CHANGELOG.md` fix entry with appropriate reporter/source credit.
