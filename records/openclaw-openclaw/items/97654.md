## Summary

Keep open: current main still lacks a cron/run-start diagnostic for an explicit `web_search` allowlist when no search provider is registered; existing diagnostics only catch a tool-call error after the model actually calls the tool.

## What This Changes

The issue requests a clear warning or failure when a cron `toolsAllow` list includes `web_search` but all web-search provider plugins are disabled.

## Best Possible Solution

Add a narrow cron/agent preflight diagnostic warning for runtime `toolsAllow` entries whose provider-backed tool family cannot be used, starting with `web_search`, and cover it with isolated-cron regression tests.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced: current main shows provider absence is only detected inside `runWebSearch` execution, while cron diagnostics are derived from agent result payloads after the run. A live cron job with all providers disabled would still be useful proof for the exact delivered-apology behavior.

## Solution Assessment

Yes for a warning-style repair: the best narrow fix is to surface missing provider-backed tool availability in cron/run diagnostics before dispatch or at run start, not to add a new config option or rewrite provider selection.
