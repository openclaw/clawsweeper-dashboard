## Summary

This looks like a real, current bug-class report rather than cleanup: current main still exposes managed `web_search` with a documented `query` argument but does not mark it required in the generic schema sent to models, while provider execution fails when `query` is absent. The reported Ollama/gemma4 behavior should stay open for a narrow schema/validation hardening pass, with live Ollama proof if feasible.

## What This Changes

The issue asks OpenClaw to make `web_search` reliably use the documented `query` argument instead of repeated invalid `queries` tool calls.

## Best Possible Solution

Keep the issue open and make the managed `web_search` argument contract match provider execution, with focused regression coverage for the generic schema and Ollama payload path.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does reproduce the failure path: current main rejects tool args without a string `query`, while the serialized `web_search` schema does not mark `query` as required.

Yes, the narrow maintainable direction is to harden the managed `web_search` schema/validation in the core tool path rather than changing Ollama provider HTTP behavior. If the local model still ignores the corrected schema, the documented Ollama tool-schema compatibility setting remains the fallback.
