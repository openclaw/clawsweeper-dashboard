## Summary

Keep open: current main and the shipped v2026.5.12 release still expose managed `web_search` with a documented required `query` argument, but the generic schema sent to models does not mark `query` as required while execution rejects calls that omit it. The issue has no linked merged fix or matching duplicate, so this is a narrow bug repair candidate.

## What This Changes

The issue reports that OpenClaw/Ollama web search prompts gemma4 to call `web_search` with `queries` instead of the expected `query` string.

## Best Possible Solution

Make the managed `web_search` schema match the documented and runtime-required `query` contract, then prove the Ollama/gemma4 path with focused regression tests and live validation when feasible.

## Reproduction Assessment

No high-confidence live end-to-end reproduction was established. Source inspection provides a focused current-main reproduction path: pass `{ queries: ["..."] }` through managed `web_search` and the Ollama provider reaches the required `query` validation failure while the model-facing schema does not mark `query` required.

## Solution Assessment

Yes, the narrow maintainable fix is to harden the managed `web_search` schema/validation contract first, not to change Ollama HTTP endpoint behavior. If live Ollama proof shows the model still ignores the corrected schema, add the smallest compatible argument repair with tests.
