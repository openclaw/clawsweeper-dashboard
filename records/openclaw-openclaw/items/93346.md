- `impact:session-state`: The reported behavior is drift between cached UI selection, persisted session fields, and the effective runtime model state.

## Summary

Keep open: current main and the latest release still prefer the local chat model override cache in the picker, while Gateway session rows still publish one selected-or-runtime model field rather than a distinct effective runtime model. Two linked UI-only PR attempts were closed unmerged, and maintainer discussion points to an explicit selected-versus-effective Gateway/UI contract before implementation.

## What This Changes

The issue requests that the web chat model dropdown reflect the session's effective runtime model, or clearly show selected and effective models when fallback/default drift occurs.

## Best Possible Solution

Expose selected/preferred and effective/runtime model identity separately in Gateway rows, then update the dropdown/status UI and regression tests around that contract.

## Reproduction Assessment

Yes, source-reproducible: current UI resolves the picker from chatModelOverrides before sessions.list, and Gateway rows still expose selected-or-runtime model identity as one field. I did not run a live fallback scenario in this read-only review.

## Solution Assessment

No, a cache-only UI patch is not the best fix. The maintainable solution needs a selected/effective model contract that preserves pending user selections while exposing runtime fallback state.
