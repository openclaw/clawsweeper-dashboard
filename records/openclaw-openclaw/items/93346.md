## Summary

Keep open: current main and the latest release still have the cache-first chat model selector and selected-over-runtime Gateway row projection, so the misleading dropdown state is not safely resolved.

## What This Changes

The issue requests that the web chat model dropdown reflect the effective runtime model when fallback/default drift makes it differ from the selected model.

## Best Possible Solution

Define a Gateway session metadata contract that exposes selected/preferred and effective/runtime model identity separately, then update the chat picker/status UI and regression tests around that contract.

## Reproduction Assessment

Yes, source-reproducible: the current selector returns the local cached model before it reads the active server row, and Gateway row fields still choose selected model identity over runtime identity. I did not run a live Gateway/provider fallback repro in this read-only review.

## Solution Assessment

Unclear: the requested direction is right, but the maintainable fix likely needs a selected-versus-effective Gateway contract rather than only changing UI cache priority.
