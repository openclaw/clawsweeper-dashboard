## Summary

Keep open: current main and the latest release still prefer the cached chat model override in the UI and still collapse selected/runtime model identity in the Gateway session row. Newer strict fallback rules reduce one silent-fallback path, but they do not resolve the stale dropdown contract reported here.

## What This Changes

The issue asks for the chat model dropdown to reflect or clearly distinguish the selected/preferred model from the effective runtime model after fallback/default drift.

## Best Possible Solution

Define selected/preferred and effective/runtime model identity separately in the Gateway/UI contract, then update the dropdown/status display and focused regression coverage around that contract.

## Reproduction Assessment

Yes. Source inspection shows a cached chatModelOverrides value wins before the active server row is read, and Gateway still projects selected model identity before runtime identity.

## Solution Assessment

Unclear until maintainer direction. The right fix is not the previously attempted UI-only cache priority change; it needs a selected-versus-effective Gateway/UI contract that preserves pending user selections.
