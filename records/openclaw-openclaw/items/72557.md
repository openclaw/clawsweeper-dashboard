## Summary

Keep this PR open. The latest head fixes the earlier duplicate-id, normalization, noop, docs, and changelog feedback, but it still introduces a registry rollback leak and a public Plugin SDK baseline drift before it is merge-ready; it also still claims the linked footer feature is resolved even though the TUI consumer is deliberately deferred.

## What This Changes

The PR adds a public `PluginStatusProvider` type and `registerStatusProvider` plugin API, stores normalized status providers in the plugin registry, exports the type through the plugin SDK, and updates docs, tests, fixtures, and changelog text.

## Best Possible Solution

Land this as a coherent API-only precursor only after the registry rollback path and generated Plugin SDK baseline are updated, and keep the visible footer work accurately tracked until a TUI consumer defines ordering, hiding, theming, and truncation behavior.

## Reproduction Assessment

Not applicable as a feature PR, but the review findings have high-confidence static reproduction paths: register a status provider and throw during plugin registration to exercise loader rollback, and compare the public SDK export/API change against the generated baseline hash contract.

## Solution Assessment

Unclear as currently patched. A narrow `registerStatusProvider` seam is plausible, but the implementation needs registry snapshot coverage and SDK baseline regeneration, and the PR should not claim the visible footer feature is resolved while the render path is deferred.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot status providers during plugin rollback:** `src/plugins/registry.ts:457-460`
  - body: `registerStatusProvider` appends to a new registry array, but `snapshotPluginRegistry` / `restorePluginRegistry` do not snapshot that array. If a plugin registers a status provider and then throws during `register`, the loader restores every existing registry array except `statusProviders`, leaving a provider from a plugin that failed to load. Add `statusProviders` to the loader snapshot and restore paths, with rollback coverage.
  - confidence: 0.91
- **[P2] Commit the Plugin SDK API baseline hash:** `src/plugin-sdk/plugin-entry.ts:121`
  - body: This PR changes the public Plugin SDK surface by adding `registerStatusProvider` to `OpenClawPluginApi` and exporting `PluginStatusProvider`, but it does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. `pnpm plugin-sdk:api:check` compares that generated hash, so the SDK API gate should report baseline drift until the hash is regenerated and committed.
  - confidence: 0.94
- **[P2] Do not mark the footer feature resolved yet:** `CHANGELOG.md:9`
  - body: The changelog says this API-only seam resolves #64086, but the PR body explicitly defers the TUI render path. Since plugins still cannot surface anything in the footer after this merges, use a non-closing reference or include the consumer so the remaining visible behavior stays tracked.
  - confidence: 0.87
