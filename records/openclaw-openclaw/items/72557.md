## Summary

Keep this PR open. The proposed status-provider API is a plausible narrow precursor for ContextClaw and the broader footer work, but the latest head still has merge-blocking mechanical issues: the new mutable registry array is not part of loader rollback, the public Plugin SDK baseline hash is not updated, and the PR still uses closing language for #64086 even though it deliberately defers the TUI render path.

## What This Changes

The PR adds a `PluginStatusProvider` type and `registerStatusProvider` plugin API, stores status-provider registrations in the plugin registry, exports the type through the Plugin SDK, updates docs/changelog text, and adds registry tests plus fixture updates.

## Best Possible Solution

Land this as an API-only precursor after the registry rollback, Plugin SDK baseline, and public wording are consistent; keep #64086 and #53774 open until a follow-up wires provider output into the TUI footer or broader footer-template behavior.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction, but the review findings have high-confidence static reproduction paths: register a status provider and throw during plugin load to exercise rollback, inspect the unchanged SDK baseline hash against the new export, and compare #64086's requested TUI polling with the API-only diff.

## Solution Assessment

No, not as currently patched. The narrow API seam is a maintainable direction, but the patch should first include rollback coverage, regenerated SDK API baseline proof, and non-closing references unless it also implements the visible footer consumer.
