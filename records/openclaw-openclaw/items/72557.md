## Summary

Keep open. This is useful source work, but the PR still introduces blocking issues around plugin-load rollback, the public Plugin SDK baseline, and closing the visible footer-status issue before a TUI consumer exists; it also lacks after-fix real behavior proof from a real setup.

## What This Changes

The PR adds a PluginStatusProvider/registerStatusProvider plugin API with registry storage, SDK exports, docs, changelog text, and registry tests, without wiring a TUI render-path consumer.

## Best Possible Solution

Land the API precursor only after rollback handling, SDK baseline regeneration, non-closing footer issue wording, and real plugin-registration proof are in place, while keeping the visible footer/template work tracked separately.

## Reproduction Assessment

Yes for the review findings. Source and PR-head inspection show the new statusProviders array is not part of loader rollback, the public SDK export lacks the tracked baseline hash update, and the changed wording would close the visible footer-status issue before rendering exists.

## Solution Assessment

No, not as currently patched. A narrow registerStatusProvider seam is a plausible maintainable direction, but this PR still needs lifecycle rollback coverage, API baseline updates, accurate public wording, and real behavior proof before merge.
