## Summary

Keep this PR open. Current main has not implemented the requested plugin footer-status API, so the PR is not obsolete, but the current patch still needs maintainer API/render-contract review and has concrete merge blockers around advertised TUI behavior, id normalization, and the required changelog entry.

## What This Changes

The PR adds a public `PluginStatusProvider` type and `registerStatusProvider` plugin API, stores registrations in the plugin registry, exports the type through the plugin SDK, and updates docs, fixtures, and registry tests.

## Best Possible Solution

Land a coherent footer-status contract: either include the TUI consumer that reads registered providers with defined ordering, hiding, theming, and truncation rules, or keep this as an explicitly API-only precursor and leave the visible footer feature tracked by the linked issue. In either path, normalize stored ids and add the active changelog entry before merge.

## Reproduction Assessment

Not applicable as a feature PR, but the review findings have high-confidence static checks: compare the PR docs/API additions to `src/tui/tui.ts:updateFooter`, and inspect `registerStatusProvider` with a padded first id followed by the canonical id.

## Solution Assessment

Unclear as currently patched. A narrow plugin API seam is plausible, but advertising it as a TUI footer contribution and resolving the footer feature before any render consumer exists is not the safest maintainable endpoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire status providers before documenting footer output:** `docs/plugins/sdk-overview.md:87`
  - body: The docs now tell plugin authors that `api.registerStatusProvider(...)` registers a TUI footer status contribution, but the PR never wires `registry.statusProviders` into the TUI footer. `updateFooter` still renders only the fixed built-in fields, so a plugin can call this API and see no visible footer output. Either add the consumer here or make the docs/scope clearly API-only and leave the visible footer feature open.
  - confidence: 0.9
- **[P2] Store normalized status provider ids:** `src/plugins/registry.ts:460-462`
  - body: This trims `provider.id` for validation and duplicate lookup but stores the original provider object. If the first registration uses `" contextclaw "`, the stored entry keeps the spaces, a later `"contextclaw"` registration is not detected as duplicate, and downstream consumers receive a noncanonical id. Store a provider copy with the trimmed id or compare against trimmed stored ids.
  - confidence: 0.91
- **[P2] Add the public plugin API changelog entry:** `src/plugins/types.ts:2370`
  - body: This adds a user-facing plugin SDK method and documentation, but the PR does not include `CHANGELOG.md`. Repo policy requires `feat`/public SDK changes to add an active changelog entry with appropriate external attribution before merge.
  - confidence: 0.86
