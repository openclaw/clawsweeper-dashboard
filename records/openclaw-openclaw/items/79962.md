## Summary

Keep open: the PR is a plausible additive Gateway feature, but the current branch misses rotated-family predecessor IDs, lacks visible after-fix real behavior proof, and overlaps active companion-session API design work.

## What This Changes

Adds `includeLineage` to `sessions.resolve` across Gateway RPC, the embedded gateway stub, Swift protocol models, docs, tests, and changelog so clients can request session-family metadata.

## Best Possible Solution

Revise the Gateway contract so current and archived family session IDs resolve with explicit visibility and ambiguity behavior, coordinate it with the active durable-lineage seam, and require real Gateway output proof before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: a row with current `sessionId` `sess-current` and `usageFamilySessionIds` containing `sess-old` will not be selected by `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. I did not run tests because this was a read-only review.

## Solution Assessment

No. Adding opt-in lineage to `sessions.resolve` is a reasonable narrow direction, but this implementation should resolve archived family IDs and settle the public API shape with the related durable-lineage work before landing.
