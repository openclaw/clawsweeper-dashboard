## Summary

Keep open: the feature is useful, but the PR currently misses the rotated-family lookup path and lacks visible after-fix proof; related companion-lineage API work is still active.

## What This Changes

The branch adds `includeLineage` to `sessions.resolve`, serializes optional lineage through Gateway RPC and the embedded stub, and updates protocol schema, Swift params, docs, tests, and changelog.

## Best Possible Solution

Revise the Gateway contract so `sessions.resolve` can resolve both current and archived family session IDs while preserving visibility and ambiguity rules, then add real Gateway output proof before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row with current `sessionId: "sess-current"` and `usageFamilySessionIds: ["sess-old", "sess-current"]`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The selector only considers current `entry.sessionId` or the store key, so it reports no session.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a plausible narrow surface, but this implementation should match archived family IDs and coordinate with the open companion-lineage contract before the public API lands.
