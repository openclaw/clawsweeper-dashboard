## Summary

Keep open: this is a plausible Gateway feature PR, but the patch still misses archived family session-id lookup and the external real behavior proof is not actually attached.

## What This Changes

Adds an optional `includeLineage` parameter to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded stub, and updates docs, Swift protocol models, tests, and changelog.

## Best Possible Solution

Keep the PR open for a revised narrow Gateway contract that resolves both current and archived family session ids while preserving visibility and ambiguity rules, then require real Gateway output proof before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row whose current `sessionId` is `sess-current` and whose `usageFamilySessionIds` includes `sess-old`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The selector never includes that row, so the proposed feature would report `No session found`.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow surface, but the implementation should match archived family ids and coordinate with https://github.com/openclaw/openclaw/issues/79903 and https://github.com/openclaw/openclaw/pull/79970 before this public contract lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve archived lineage ids before returning lineage:** `src/gateway/sessions-resolve.ts:274`
  - body: `sessions.resolve({ sessionId, includeLineage: true })` only reaches this return after the selector has matched the current `entry.sessionId` or store key. After reset or rotation, predecessor ids live only in `usageFamilySessionIds`, so resolving one of those ids returns `No session found` instead of the canonical key and lineage. Include family ids in the session-id match path and add a predecessor-id regression.
