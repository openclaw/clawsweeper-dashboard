## What This Changes

The PR adds optional `includeLineage` support to `sessions.resolve`, returns lineage metadata through Gateway RPC and the embedded gateway stub, and updates Swift protocol models, docs, tests, and `CHANGELOG.md`.

## Best Possible Solution

Keep the PR open, rebase it cleanly, remove the release-owned changelog edit, and revise the lineage resolver so archived family session ids follow the same canonical visibility and ambiguity policy as current `sessionId` resolution.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row whose current `sessionId` is `sess-current` and whose `usageFamilySessionIds` includes `sess-old`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The selector never includes that row, so the proposed feature reports no session instead of canonical lineage.

## Solution Assessment

Unclear as a final product shape. Adding `includeLineage` to `sessions.resolve` is a plausible narrow seam, but the implementation should first support archived family ids or explicitly narrow the contract, and maintainers need to reconcile it with the open companion-session lineage issue.
