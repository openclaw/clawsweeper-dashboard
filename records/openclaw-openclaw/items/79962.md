## Summary

Keep open: this is a plausible Gateway feature, but the PR still misses predecessor session IDs stored in usage lineage and the external-contributor proof is only test-command claims, not real Gateway or client output.

## What This Changes

The PR adds an optional `includeLineage` flag to `sessions.resolve`, returns lineage metadata through the Gateway RPC and embedded stub, and updates protocol docs, Swift models, changelog, and focused tests.

## Best Possible Solution

Extend the sessionId match path to include `usageFamilySessionIds` under the existing visibility and ambiguity rules, add predecessor-ID regression coverage, and settle the additive Gateway/SDK lineage shape against the open companion-seam design before merge.

## Reproduction Assessment

The feature request itself is not a bug, but the blocking PR finding is source-reproducible: a stored entry with `usageFamilySessionIds: ["sess-old", "sess-current"]` and current `sessionId: "sess-current"` will not match a `sessions.resolve` request for `sessionId: "sess-old"`. I did not run tests because this read-only review cannot execute artifact-producing checks.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable direction, but the implementation should resolve archived family IDs as well as the current ID and needs maintainer review for the public protocol shape.
