## Summary

Keep open: the feature is not implemented on current main, and this PR is not ready to merge because the proposed resolver still misses archived family session ids and the external proof is test-only.

## What This Changes

The PR adds an optional `includeLineage` flag to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded gateway stub, updates protocol docs/Swift models, and adds focused tests.

## Best Possible Solution

Land this surface only after `sessionId` lookup includes known family ids while preserving visibility and ambiguity rules, with regression coverage and real Gateway or companion proof.

## Reproduction Assessment

Not applicable for the feature request itself. For the blocking finding, source inspection gives a high-confidence path: after reset, the prior id lives in `usageFamilySessionIds`, while this PR’s selection path still only matches current `entry.sessionId` or the store key.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a plausible narrow surface, but the implementation should resolve archived family ids before it satisfies the rotated-lineage contract.
