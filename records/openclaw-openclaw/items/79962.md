## What This Changes

The PR adds an opt-in `includeLineage` parameter to Gateway `sessions.resolve`, serializes lineage through the Gateway and embedded stub, updates Swift/docs/tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Settle the public lineage contract with maintainers, then rebase onto current main, resolve predecessor family ids or narrow the documented contract, remove the release-owned changelog edit, and add redacted real Gateway runtime proof.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection: a row with current `sessionId` `sess-current` and `usageFamilySessionIds: ["sess-old", "sess-current"]` will not be selected by `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The PR feature itself is new API work rather than a broken existing behavior report.

## Solution Assessment

Unclear as the final product shape; `includeLineage` on `sessions.resolve` is plausible, but it must either support archived family ids or explicitly document a narrower current-id-only contract. Maintainers should choose that protocol boundary before merge.
