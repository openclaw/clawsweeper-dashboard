## Summary

Keep open: the PR adds a plausible Gateway lineage surface, but the proposed session-id lookup still cannot resolve archived family ids stored only in lineage metadata, and the external after-fix proof is not visible in the PR body or comments.

## What This Changes

Adds `includeLineage` to `sessions.resolve`, serializes optional lineage through Gateway RPC and the embedded stub, and updates docs, Swift protocol models, tests, and changelog.

## Best Possible Solution

Revise the Gateway resolver contract so current and archived family ids resolve through the same visibility and ambiguity policy, then land it with redacted real Gateway output proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row whose current `sessionId` is `sess-current` and whose `usageFamilySessionIds` contains `sess-old`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The PR selector never includes that row, so the proposed feature returns `No session found`.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow direction, but the implementation should match archived family ids and coordinate with https://github.com/openclaw/openclaw/issues/79903 and https://github.com/openclaw/openclaw/pull/79970 before this public contract lands.

## Review Findings
