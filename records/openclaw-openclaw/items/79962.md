## Summary

Keep open: the PR adds a useful Gateway lineage feature, but the proposed lookup only reaches lineage serialization after matching the current session id or store key, so archived family ids still return not found. External after-fix proof is also still placeholder-only, and the public contract overlaps active companion lineage design work.

## What This Changes

Adds an optional `includeLineage` parameter to `sessions.resolve`, returns lineage through Gateway and embedded-stub responses, and updates protocol schema, Swift models, docs, and tests.

## Best Possible Solution

Revise the Gateway contract to resolve both current and archived family session ids under existing visibility and ambiguity rules, coordinate the API shape with the active lineage seam, then attach redacted real Gateway output proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row with current `sessionId` `sess-current` and `usageFamilySessionIds` containing `sess-old`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The selector omits that row, so the proposed feature returns `No session found`.

## Solution Assessment

No. Adding opt-in lineage to `sessions.resolve` is a plausible narrow surface, but the implementation must match archived family ids and coordinate with https://github.com/openclaw/openclaw/issues/79903 and https://github.com/openclaw/openclaw/pull/79970 before this public contract lands.
