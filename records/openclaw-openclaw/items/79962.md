## Summary

Keep open: this is a new Gateway protocol feature with a plausible surface, but the patch still misses rotated-family lookup for archived `usageFamilySessionIds` and has only test-command proof, so it needs contributor proof and maintainer API review before merge.

## What This Changes

The PR adds an optional `includeLineage` flag to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded stub, updates protocol docs/Swift models, and adds focused resolver tests.

## Best Possible Solution

Land this surface only after the sessionId lookup path includes known family ids while preserving visibility and ambiguity rules, with regression coverage and real Gateway/companion proof.

## Reproduction Assessment

Not applicable: this is a feature PR, not a broken existing contract. Source inspection verifies current main lacks `includeLineage`, and the proposed behavior can be reviewed against existing rotation lineage fields.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow surface, but the implementation should resolve archived family session ids as well as the current id before it satisfies the rotated-lineage contract.
