## Summary

Keep open: the PR targets a real public Gateway seam that current main still lacks, but the current patch does not resolve archived lineage session IDs and still needs contributor-visible runtime proof plus maintainer API review before it can merge.

## What This Changes

Adds optional `includeLineage` support to Gateway `sessions.resolve`, serializes the richer result through RPC and the embedded stub, and updates Swift models, docs, changelog, and tests.

## Best Possible Solution

Revise the Gateway contract so `sessions.resolve` can resolve both current and archived family session IDs while preserving visibility and ambiguity rules, then review it with the broader durable-lineage design.

## Reproduction Assessment

Yes for the PR defect by source inspection: a row with current `sessionId: "sess-current"` and `usageFamilySessionIds: ["sess-old", "sess-current"]` will not be selected when resolving `sessionId: "sess-old"`, because the selector only checks current `entry.sessionId` and key equality.

## Solution Assessment

No. Adding opt-in lineage to `sessions.resolve` is a plausible narrow surface, but this implementation should also match archived family IDs and be coordinated with https://github.com/openclaw/openclaw/issues/79903 before the public contract lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve archived lineage ids before returning lineage:** `src/gateway/sessions-resolve.ts:274`
  - body: `sessions.resolve({ sessionId, includeLineage: true })` only reaches this return after selection has matched the current `entry.sessionId` or store key. After reset or rotation, predecessor IDs live only in `usageFamilySessionIds`, so resolving an archived family ID returns `No session found` instead of the canonical key and lineage. Include family IDs in the match path and add a predecessor-ID regression test.
  - confidence: 0.9
