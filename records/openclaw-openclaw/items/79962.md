## Summary

Keep open: this is a recent external feature PR for a new Gateway protocol surface, and the branch still has a concrete correctness blocker plus mock-only real behavior proof. Current `main` does not already implement the requested `includeLineage` contract, so none of the close reasons apply.

## What This Changes

The branch adds an optional `includeLineage` flag to Gateway `sessions.resolve`, threads lineage through RPC and the embedded gateway stub, updates protocol docs/Swift models, and adds focused resolver tests.

## Best Possible Solution

Land a reviewed `sessions.resolve` lineage surface only after sessionId lookup includes known family ids while preserving visibility and ambiguity rules, with focused regression coverage and real Gateway or companion proof.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. For the PR finding, source inspection is enough: the patch returns lineage only after current sessionId/key selection, while reset stores predecessor ids in `usageFamilySessionIds`.

## Solution Assessment

No. Adding `includeLineage` to the existing resolver is a plausible narrow API, but the implementation should resolve archived family session ids and provide real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve archived family session IDs:** `src/gateway/sessions-resolve.ts:271`
  - body: `sessions.resolve({ sessionId, includeLineage: true })` only reaches this return after the match path finds the current `entry.sessionId` or store key. After reset or compaction, predecessor ids live in `entry.usageFamilySessionIds`, so resolving an archived id returns `No session found` instead of the canonical key and lineage. Include family ids in the sessionId match path and add a predecessor-id regression.
