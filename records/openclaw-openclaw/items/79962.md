## Summary

Keep open: the PR is a plausible additive Gateway feature, but the patch still misses archived session-family ids, has only test-based proof, and needs maintainer API review before merge.

## What This Changes

The PR adds `includeLineage` to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded gateway stub, and updates protocol docs, Swift models, changelog, and focused tests.

## Best Possible Solution

Keep the additive Gateway surface only after `sessionId` matching includes `usageFamilySessionIds`, preserves visibility and ambiguity rules, and maintainers approve the public API shape.

## Reproduction Assessment

Not applicable as a user bug because this is a feature PR. For the patch finding, source inspection is high-confidence: predecessor ids are stored in `usageFamilySessionIds`, but the PR selects rows using only current `sessionId` or store key before serializing lineage.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow surface, but the implementation must resolve archived family ids as well as the current id before it satisfies the rotated-lineage contract.
