## Summary

Keep open: this feature is not on current main, and the PR still misses archived family-session lookup while the external proof is only formatter/Vitest evidence.

## What This Changes

The PR adds an optional `includeLineage` flag to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded stub, updates protocol docs/Swift models, and adds focused resolver tests.

## Best Possible Solution

Land the surface only after `sessionId` lookup includes known family ids while preserving visibility and ambiguity rules, with regression coverage and real Gateway or companion proof.

## Reproduction Assessment

Not applicable for the feature itself; this is a new public Gateway contract, not a broken existing behavior. For the patch finding, source inspection gives a high-confidence path: predecessor ids are stored in `usageFamilySessionIds`, but the PR still selects only by current `entry.sessionId` or key before serializing lineage.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a plausible narrow surface, but the implementation should broaden selection to known family ids and cover predecessor-id resolution before it satisfies the rotated-lineage contract.
