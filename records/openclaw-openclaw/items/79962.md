## Summary

Keep open: this is a plausible Gateway feature, but the PR still misses archived family-session id resolution and lacks visible after-fix real behavior proof; the public API shape also overlaps active companion-seam work.

## What This Changes

Adds optional `includeLineage` support to Gateway `sessions.resolve`, threads the payload through RPC and the embedded stub, and updates protocol schema, Swift models, docs, tests, and changelog.

## Best Possible Solution

Revise the Gateway contract so session resolution can match both current and archived family session ids with explicit ambiguity handling, then add predecessor-id regression coverage and real Gateway proof before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row with current `sessionId: "sess-current"` and `usageFamilySessionIds: ["sess-old", "sess-current"]`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The selector never includes that row, so the PR would return `No session found` instead of the canonical key and lineage.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow surface, but the implementation must match archived family ids and coordinate with the active durable lineage API work before becoming a public contract.
