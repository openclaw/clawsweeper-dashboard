## Summary

Keep open: current main does not expose `includeLineage`, and the proposed Gateway contract still misses archived family session IDs that the linked lineage feature needs. The PR also remains blocked on contributor real behavior proof and maintainer API review.

## What This Changes

The branch adds `includeLineage` to Gateway `sessions.resolve`, serializes optional lineage metadata through Gateway RPC and the embedded stub, and updates protocol docs, Swift model generation, tests, and changelog.

## Best Possible Solution

Land this surface only after the sessionId lookup path includes known family IDs while preserving visibility and ambiguity rules, with regression coverage and real Gateway or companion-client proof.

## Reproduction Assessment

Not applicable: this is a new Gateway feature, not a broken existing contract. Source inspection verifies current main lacks `includeLineage`, and the patch can be reviewed against existing rotation-lineage fields.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a reasonable narrow surface, but the implementation should resolve archived family session IDs as well as the current ID before it satisfies the rotated-lineage contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Match archived family session ids:** `src/gateway/sessions-resolve.ts:271`
  - body: `sessions.resolve({ sessionId, includeLineage: true })` only reaches this return after the match path finds the current `entry.sessionId` or store key. After reset or rotation, predecessor IDs live in `entry.usageFamilySessionIds`, so resolving an archived ID still returns `No session found` instead of the canonical key and lineage. Include family IDs in the sessionId match path and add a predecessor-ID regression.
