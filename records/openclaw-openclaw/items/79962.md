## Summary

Keep open: current main does not expose `includeLineage`, and the PR still has a correctness gap for archived family session ids plus missing after-fix real behavior proof.

## What This Changes

The PR adds an optional `includeLineage` parameter to `sessions.resolve` and threads lineage serialization through Gateway RPC, the embedded gateway stub, Swift protocol models, docs, tests, and changelog.

## Best Possible Solution

Revise the Gateway contract so `sessions.resolve` resolves both current and archived family session ids while preserving visibility and ambiguity rules, then require real Gateway output proof before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row whose current `sessionId` is `sess-current` and whose `usageFamilySessionIds` includes `sess-old`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The PR selector excludes that row, so the request returns `No session found` before lineage can be serialized.

## Solution Assessment

No. Adding `includeLineage` to `sessions.resolve` is a plausible narrow surface, but this implementation should match archived family ids and be coordinated with the active durable lineage seam before becoming a public protocol contract.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Resolve archived lineage ids before returning lineage:** `src/gateway/sessions-resolve.ts:274`

Include family ids in the session-id match path and add a predecessor-id regression.
