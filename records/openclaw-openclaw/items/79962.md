## Summary

Keep open: the feature direction is useful, but the current patch misses archived family session IDs before lineage is returned, and the external PR still lacks visible after-fix runtime proof. The public contract also overlaps the active lineage API work at https://github.com/openclaw/openclaw/issues/79903 and https://github.com/openclaw/openclaw/pull/79970.

## What This Changes

Adds an optional `includeLineage` flag to `sessions.resolve`, serializes lineage through Gateway RPC and the embedded stub, and updates protocol docs, Swift models, tests, and changelog.

## Best Possible Solution

Revise the Gateway lineage seam so it resolves current and archived family session IDs with explicit ambiguity/visibility behavior, then land it with real Gateway output proof or defer to the canonical lineage-selection stack.

## Reproduction Assessment

Yes for the PR defect by source inspection: create a row with current `sessionId: "sess-current"` and `usageFamilySessionIds: ["sess-old", "sess-current"]`, then call `sessions.resolve({ sessionId: "sess-old", includeLineage: true })`. The PR selector never includes that row, so it returns `No session found` before lineage can be serialized.

## Solution Assessment

No. Adding opt-in lineage to `sessions.resolve` is plausible, but this implementation must match archived family IDs and be coordinated with https://github.com/openclaw/openclaw/issues/79903 and https://github.com/openclaw/openclaw/pull/79970 before becoming a public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve archived lineage ids before returning lineage:** `src/gateway/sessions-resolve.ts:274`

After reset or rotation, predecessor IDs live only in `usageFamilySessionIds`, so resolving one of those IDs returns `No session found` instead of the canonical key and lineage; include family IDs in the match path and add a predecessor-ID regression.
