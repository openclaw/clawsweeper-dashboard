## Summary

Keep open: the PR is a useful lineage slice, but source inspection shows predecessor session IDs stored only in `usageFamilySessionIds` still will not resolve, and the external-PR proof is test-only.

## What This Changes

The PR adds `includeLineage` to `sessions.resolve`, serializes optional lineage metadata through Gateway and embedded gateway paths, and updates docs, changelog, tests, and Swift protocol params.

## Best Possible Solution

Keep the lineage work focused on the narrow public resolver seam, make predecessor/family IDs discoverable through the same canonical selection policy, and avoid closing the broader companion-seams umbrella until the remaining slices land or are explicitly superseded.

## Reproduction Assessment

Yes, from source inspection. After reset, the previous ID is stored in `usageFamilySessionIds`, but the resolver still only searches `entry.sessionId` and key before the new lineage serialization runs.

## Solution Assessment

No, not as currently proposed. Adding `includeLineage` to `sessions.resolve` is a plausible small seam, but it must also resolve known family IDs and should not close the broader companion-seams umbrella.
