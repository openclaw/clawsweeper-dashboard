## Summary

Keep open: current main and v2026.5.7 still generate a canonical macOS LaunchAgent PATH without `/opt/homebrew/bin` or `/opt/homebrew/sbin`, while Homebrew Node is a documented/recommended macOS install path and the existing runtime-dir prepend only partially covers fresh installs.

## What This Changes

The issue asks macOS Gateway LaunchAgent service PATH generation to include Homebrew directories so `openclaw gateway restart` does not leave the service unable to find Homebrew-installed runtime/tools.

## Best Possible Solution

Treat `/opt/homebrew/bin` and `/opt/homebrew/sbin` as stable macOS service PATH entries while preserving the existing filters against user/workspace/version-manager PATH injection, and add focused regression coverage for install/restart PATH rendering.

## Reproduction Assessment

Yes for the source-level bug: `buildServiceEnvironment` on Darwin currently produces a PATH without `/opt/homebrew/bin` or `/opt/homebrew/sbin`, and the test suite asserts that value. I did not run a real macOS LaunchAgent restart in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in macOS service PATH generation and its tests, not in caller-specific restart logic. The implementation should preserve the current minimal-path/security intent and avoid reintroducing user version-manager directories.
