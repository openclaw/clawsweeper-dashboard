## Label Justifications

- `P2`: The report describes a limited provider-fallback delay that affects rate-limited OpenRouter chains but does not make the core runtime unusable.
- `impact:auth-provider`: The affected surface is provider routing and fallback after a rate-limited model provider response.

## Summary

Current main and the shipped v2026.5.12 release already prevent the reported long SDK Retry-After stall by converting over-cap retryable provider responses into immediate SDK errors, allowing OpenClaw model failover to continue.

## What This Changes

The issue asks OpenClaw to avoid stalling a provider fallback chain when OpenRouter returns a long Retry-After value that exceeds the configured retry wait cap.

## Best Possible Solution

Keep the shipped SDK retry cap behavior and ask affected users on 2026.4.21 to upgrade to v2026.5.12 or newer, with OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS available for a stricter local cap.

## Reproduction Assessment

No current-main failure was reproduced because the code is already fixed. The old failure mode is source-reproducible from the SDK retry contract, and current tests cover the over-cap Retry-After path that used to stall failover.

## Solution Assessment

Yes. The current fix is the narrow maintainable path: it preserves normal short SDK retries, injects the SDK-supported no-retry header only for over-cap retryable responses, and lets the existing model fallback chain handle the surfaced error.
