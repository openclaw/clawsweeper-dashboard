## Summary

Keep open: current main and the latest release still use a strict retry cap comparison, so `Retry-After: 60` remains SDK-retryable at the default 60-second cap. The related fix PR targeted the right narrow path but was closed unmerged after the proof gate, so the issue still needs a small code/test/docs fix with real fallback proof if feasible.

## What This Changes

The issue reports that OpenRouter 429 responses with `Retry-After: 60` stall model fallback instead of surfacing immediately when the retry wait reaches the configured SDK cap.

## Best Possible Solution

Make the SDK retry cap explicitly inclusive at the boundary, add a `Retry-After: 60` default-cap regression test, update retry/failover docs, and capture redacted real or equivalent fallback proof when practical.

## Reproduction Assessment

Yes, at source level: current main resolves the default retry cap to 60 seconds and uses a strict `>` comparison, so `Retry-After: 60` does not receive `x-should-retry: false`. I did not run a live OpenRouter 429 timing repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to align the cap predicate, regression coverage, and docs around the intended at-cap behavior. The closed PR had the likely code shape, but it still needed docs alignment and real behavior proof before merge.
