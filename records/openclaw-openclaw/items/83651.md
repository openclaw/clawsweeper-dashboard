## Label Justifications

- `P2`: This was a real provider fallback delay, but current evidence shows it is already fixed and limited to retry/failover latency rather than data loss or outage.
- `impact:auth-provider`: The issue concerns provider routing and model fallback behavior for OpenRouter rate limits.

## Summary

Current main and the v2026.5.12 release already implement the requested SDK Retry-After cap for OpenAI-compatible providers such as OpenRouter: a configured cap below the provider Retry-After value makes the SDK surface the 429 so OpenClaw can continue model failover.

## What This Changes

The issue requests that OpenRouter fallback chains skip long SDK Retry-After sleeps once the delay exceeds the configured max wait cap.

## Best Possible Solution

Keep the shipped SDK retry cap; if OpenRouter still stalls on v2026.5.12 or newer with the env cap set below Retry-After, collect fresh redacted provider response logs in a new issue.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. Source and existing tests show current main should inject `x-should-retry: false` for a 60 second Retry-After when `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS=10`, so the reported 2026.4.21 stall is obsolete unless fresh current-release logs show otherwise.

## Solution Assessment

Yes. The shipped fix is the narrow transport-layer solution: it uses the OpenAI SDK's `x-should-retry` contract to preserve short SDK retries while surfacing long waits to OpenClaw's existing rate-limit model-failover path.
