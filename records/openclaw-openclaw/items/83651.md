## Summary

Keep open. Current main still uses a strict retry cap comparison, so `Retry-After: 60` at the default 60-second cap remains SDK-retryable; the linked open fix PR at https://github.com/openclaw/openclaw/pull/83659 references this issue with closing syntax, so the issue should stay open until that PR or a successor lands.

## What This Changes

The issue asks OpenClaw to avoid SDK-internal sleeps when an OpenRouter 429 `Retry-After` reaches the configured retry cap so fallback providers are tried promptly.

## Best Possible Solution

Align the retry-cap code, tests, and docs around an explicit inclusive or exclusive boundary, then close this issue when the linked fix PR or a replacement merges with proof that the OpenRouter fallback stall is gone.

## Reproduction Assessment

Yes. Source inspection shows current main leaves an exact `Retry-After: 60` retryable at the default 60-second cap, and the OpenAI/Anthropic SDKs sleep retry-after values unless `x-should-retry: false` is injected.

## Solution Assessment

Unclear as currently proposed. The predicate change in the linked PR is the narrow likely fix, but maintainers still need the boundary documented and real fallback-chain proof before the issue should close.
