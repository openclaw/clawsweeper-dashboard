## Summary

Keep this open: current `main` and the latest release still use a strict greater-than retry cap, so `Retry-After: 60` at the default 60-second cap remains SDK-retryable. The directly linked repair PR targeted the right boundary but was closed unmerged, and a later user report still describes delayed OpenRouter fallback on 2026.5.18.

## What This Changes

The issue asks OpenClaw to surface long OpenRouter `Retry-After` rate-limit responses promptly so the provider fallback chain can advance instead of waiting inside the SDK.

## Best Possible Solution

Align code, tests, and docs around an explicit inclusive retry cap so equal-to-cap `Retry-After` responses surface immediately while below-cap retry sleeps remain SDK-managed.

## Reproduction Assessment

Yes from source inspection: current `main` leaves a retryable 429 with `Retry-After: 60` untouched at the default 60-second cap, and the OpenAI/Anthropic SDKs then sleep the parsed retry-after duration unless `x-should-retry: false` is injected. I did not run a live OpenRouter rate-limit scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the cap boundary inclusive in `src/agents/provider-transport-fetch.ts`, add the exact 60-second regression test, and update the retry/failover docs to match the new contract.
