## Summary

Keep open: current main and the latest release still use a strict retry cap comparison, so `Retry-After: 60` at the default 60-second cap remains SDK-retryable before OpenClaw can advance the fallback chain; the only matching fix PR closed unmerged after the proof gate.

## What This Changes

The issue asks OpenClaw to stop SDK-internal retry sleeps at the configured Retry-After cap so OpenRouter fallback chains can advance without a 60-second stall.

## Best Possible Solution

Treat retry-after values at or above the configured SDK retry cap as too long, align the retry/failover docs, add a default-cap equality regression test, and verify the fallback path no longer waits 60 seconds.

## Reproduction Assessment

Yes from source inspection, but not from a live OpenRouter run: current main parses `Retry-After: 60`, compares it with the default cap using `>`, and therefore does not set `x-should-retry: false`. The checked upstream SDK contracts then sleep Retry-After values unless that header disables retry.

## Solution Assessment

Yes, with docs and proof: an at-or-above-cap predicate plus an equality regression test is the narrow code fix. The safer complete path also updates the public retry/failover wording and proves the behavior with a real or synthetic SDK fallback scenario.
