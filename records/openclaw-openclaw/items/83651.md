## Summary

Keep open: current main still treats `Retry-After: 60` as SDK-retryable at the default 60-second cap, so the reported provider fallback stall can still happen. A focused PR addressed this boundary but was closed unmerged for missing real behavior proof.

## What This Changes

Treat OpenRouter/OpenAI-compatible `Retry-After` values at the SDK retry cap as too long so model fallback can proceed without waiting inside the provider SDK.

## Best Possible Solution

Make the SDK retry cap explicitly inclusive, update retry/failover docs and boundary tests, then close after proof shows `Retry-After: 60` no longer stalls fallback routing.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current main leaves `Retry-After: 60` retryable at the default cap, and the OpenAI/Anthropic SDKs then honor `retry-after` unless `x-should-retry: false` is injected.

## Solution Assessment

Yes: the narrow maintainable fix is to make at-cap retry-after responses bypass SDK retries and align docs/tests with that inclusive cap behavior.
