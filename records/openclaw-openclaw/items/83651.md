## Summary

Keep open: current `main` still uses a strict retry-after cap boundary, so `Retry-After: 60` at the default 60-second cap is not surfaced immediately for fallback handling; the linked fix PR was closed unmerged.

## What This Changes

The issue reports that OpenRouter `Retry-After: 60` responses can stall model fallback before later providers are tried.

## Best Possible Solution

Make the SDK retry cap inclusive for retry-after values at the configured threshold, add equality regression coverage, and update retry/failover docs; capture redacted real fallback-chain proof if feasible.

## Reproduction Assessment

Yes from source inspection, but not from a live OpenRouter run: current main parses `Retry-After: 60` as equal to the default cap and leaves it retryable because the bypass comparison is strict `>`.

## Solution Assessment

Yes: the narrow maintainable fix is to align the retry cap boundary, regression tests, and docs around at-or-above-cap behavior rather than adding a new retry configuration path.
