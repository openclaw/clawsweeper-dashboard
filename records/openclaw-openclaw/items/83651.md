## Summary

This should stay open: current main and the latest release still use a strict greater-than retry cap, so `Retry-After: 60` at the default 60-second cap remains SDK-retryable instead of surfacing immediately for fallback. The linked fix PR was closed unmerged after the proof/docs gate, and a second reporter confirmed similar fallback delay on 2026.5.18.

## What This Changes

The issue requests that OpenClaw bypass SDK-internal retry sleeps when an OpenRouter `Retry-After` value reaches the configured retry wait cap so later fallback providers can run promptly.

## Best Possible Solution

Make the SDK retry wait cap inclusive at the equality boundary, add a default `Retry-After: 60` regression test, and update retry/failover docs so fallback behavior and operator-facing cap semantics match.

## Reproduction Assessment

Yes, source inspection gives a high-confidence focused repro path: a retryable response with `Retry-After: 60` at the default cap is left retryable because current main uses `>` rather than an inclusive comparison. I did not run a live OpenRouter 429 scenario.

## Solution Assessment

Yes, the narrow maintainable fix is in `src/agents/provider-transport-fetch.ts`: make the cap boundary inclusive, cover the default 60-second case, and align the docs. The previously linked PR was directionally right but incomplete because it lacked proof and docs alignment.
