## Summary

Keep open: current `main` and the shipped `v2026.5.18` tag still use a strict `Retry-After > maxWaitSeconds` comparison, so the reported `Retry-After: 60` boundary is not fixed. The only exact companion fix at https://github.com/openclaw/openclaw/pull/83659 was closed unmerged for missing real behavior proof, and the later comment reports similar behavior on the latest stable release.

## What This Changes

The issue requests that provider responses with `Retry-After` at the configured SDK retry wait cap bypass SDK-internal retry sleeps so model fallback can proceed promptly.

## Best Possible Solution

Make the SDK retry cap explicitly inclusive for `Retry-After >= OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS`, add the equality regression test, and update retry/failover docs while preserving the disable and short-wait behavior.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.5.18` keep `Retry-After: 60` retryable because the default cap is 60 and the predicate is strict `>`. I did not live-trigger OpenRouter's rate limit in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make the existing cap boundary inclusive, add a regression test for the 60-second case, and align the docs. A broader fallback rewrite is not needed for the reported boundary.
