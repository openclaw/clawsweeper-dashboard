## Summary

Keep open for maintainer review: current main still lacks memory-lancedb isolation, but this draft PR is not mergeable because it targets an obsolete recall hook path, has scope-key and migration observability defects, and lacks real two-agent LanceDB proof.

## What This Changes

The PR adds an opt-in `agentScoping` setting to memory-lancedb, an `agentId` LanceDB column/migration, and scoped memory tool/hook reads and writes.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that derives one trusted scope or partition key, applies it consistently to tools, before_prompt_build auto-recall, auto-capture, duplicate checks, migrations, CLI/docs, and tests, and aligns with the broader partition work.

## Reproduction Assessment

Yes, source-level. Configure two agents or users against the same memory-lancedb dbPath, store a memory from one, then recall from the other; current main has no row field or query filter that can prevent cross-scope results.

## Solution Assessment

No. The requested direction is valid, but this branch is not the best mergeable solution until it is rebased onto current main, scopes the active hook path, uses one trusted validated key, reports migration failures, and includes real two-agent proof.
