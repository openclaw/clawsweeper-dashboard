## Summary

Keep this PR open: the memory-lancedb isolation problem is still real on current main, but this draft branch is stale against the active prompt-build hook, has blocking correctness/security concerns, and lacks contributor real-behavior proof.

## What This Changes

The PR adds an optional `agentScoping` config and manifest field, stores an `agentId` column in LanceDB rows, and converts memory tools/hooks to filter store/recall/forget operations by the current agent.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes auto-recall, auto-capture, duplicate checks, tools, and delete paths through one validated agent scope key, with logged migration diagnostics, focused regression coverage, docs for the config/migration behavior, and redacted real-environment proof.

## Reproduction Assessment

Yes, source-level. Current main stores and recalls from one shared LanceDB table with no `agentId` column or `agentScoping` config; I did not run a live two-agent LanceDB environment.

## Solution Assessment

No. The direction is valid, but this branch is not the best mergeable form until it is rebased to `before_prompt_build`, uses one validated scope key, reports migration failures, and includes real behavior proof.
