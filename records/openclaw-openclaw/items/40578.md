## Summary

Keep open: current main still has an unscoped memory-lancedb store, so the feature is not implemented, but this draft PR is not mergeable as-is because it targets stale hook/tool paths and has scoped-access correctness issues.

## What This Changes

The PR adds an optional `agentScoping` setting, an `agentId` LanceDB column/migration, scoped memory tool and hook calls, manifest config metadata, and a small unit-test update.

## Best Possible Solution

Land a rebased, maintainer-reviewed memory-lancedb isolation design that scopes tools, duplicate checks, auto-recall, and auto-capture on current hook paths; uses one explicit scope-key contract; logs migration failures; and includes docs, changelog, and two-agent regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a single shared `MemoryDB` and unscoped `db.search`/`db.store`/`db.delete` calls; the PR-specific regressions are visible directly in the patch.

## Solution Assessment

No. The PR is directionally useful, but the mergeable solution should be rebased onto current main and settle one stable agent-scope contract before storing or filtering memory rows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent scope key:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When scoping is enabled, this patch stores raw `ctx.agentId` values but filters reads and deletes with a stripped value. IDs containing characters outside the filter allowlist can become unrecoverable or collide, so validate/reject or encode once and use that same key everywhere.
  - confidence: 0.91
- **[P2] Scope the current prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and tests assert `before_agent_start` is not registered. This patch scopes only the legacy hook path, so auto-recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Report agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic.
  - confidence: 0.84
- **[P3] Instantiate tool factories in the live smoke path:** `extensions/memory-lancedb/memory-lancedb.live.test.ts:75-80`
  - body: This patch converts memory tools to factories, but the live memory-lancedb smoke still retrieves registered tools and calls `.execute` on the raw value. Once factory registration lands, that live path must invoke the factory with a context first.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and has scoped-access bugs that can hide, leak, or misroute memories if merged as-is.

Concerns:

- **[high] Agent ID normalization can break isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter on a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee.
  - confidence: 0.9
- **[high] Automatic recall can remain unscoped:** `extensions/memory-lancedb/index.ts:588`
  - body: The PR scopes the legacy auto-recall hook while current main uses `before_prompt_build`, so a stale rebase could still inject memories across agents on the active prompt path.
  - confidence: 0.88
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows failures without logging, leaving operators with no warning that scoped filtering may not be active on existing tables.
  - confidence: 0.82
