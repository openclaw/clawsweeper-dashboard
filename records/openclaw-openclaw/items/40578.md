## Summary

Keep open. The PR targets a real memory isolation gap, but current main still has unscoped LanceDB memory paths and this draft branch has unresolved correctness, security-boundary, and real-behavior-proof blockers rather than being obsolete or safely closeable.

## What This Changes

The branch adds optional `agentScoping` to `memory-lancedb`, adds an `agentId` LanceDB column/migration, and routes memory store/search/delete through per-agent tool and hook context.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes auto-recall, auto-capture, duplicate checks, tools, and deletes through one validated scope identity, with migration diagnostics, config/docs alignment, regression coverage, and redacted real two-agent proof.

## Reproduction Assessment

Yes, source-level. Configure two agents against the same memory-lancedb `dbPath`, store from one scope, then recall from the other; current main has no row field, scoped DB selection, or search filter to prevent cross-agent results.

## Solution Assessment

No. The direction is plausible, but this branch is not the best mergeable form until it is rebased to current main, scopes `before_prompt_build`, uses one validated scope key, reports migration failures, and includes real behavior proof.
