## Summary

Keep this draft PR open: current main still has the memory-lancedb cross-agent sharing problem, but the branch is not mergeable as-is because it scopes a stale hook path, lacks real multi-agent behavior proof, and has scoped-access correctness/security issues.

## What This Changes

The PR adds an opt-in `agentScoping` setting, an `agentId` LanceDB field/migration, scoped memory tool/hook reads and writes, and a small tool-factory test adjustment for `memory-lancedb`.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes tools, duplicate checks, auto-recall, and auto-capture on current hook paths with one validated scope-key contract, migration diagnostics, docs/changelog, and two-agent regression plus live behavior proof.

## Reproduction Assessment

Yes, source-level: current main constructs one shared `MemoryDB` and the tool/hook paths search/store/delete without an agent scope. I did not run a live multi-agent LanceDB reproduction in this read-only review.

## Solution Assessment

No. The requested direction is valid, but this branch is not the best mergeable solution until it is rebased onto current main and uses one stable, validated scope-key contract across writes, reads, deletes, duplicate checks, and active hooks.

  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build` and tests assert the legacy hook is not registered. This branch scopes only `before_agent_start`, so after rebase automatic recall would still search the shared DB unless the current hook gets the same agent scope.
  - confidence: 0.9

  - body: The branch writes `ctx.agentId` into stored rows but builds read/delete filters from a separately stripped value.   - confidence: 0.82

  - body: `migrateAgentIdColumn` swallows schema or `addColumns` failures without logging. Operators can enable scoping while an existing table remains effectively unscoped, with no diagnostic explaining why isolation is not active.
  - confidence: 0.86
- **[P3] Update the live smoke for tool factories:** `extensions/memory-lancedb/index.ts:351-362`
  - body: This patch converts the memory tools to factories, but the existing live smoke retrieves the registered tool and calls `.execute` directly. Once this change lands, the live test must instantiate the factory with a context before calling `execute`.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes memory data isolation semantics and has scoped-access defects that could hide, leak, or misroute memories if merged as-is.

Concerns:

- **[high] Automatic recall can remain unscoped:** `extensions/memory-lancedb/index.ts:588`
  - body: The branch scopes the stale `before_agent_start` hook while current main uses `before_prompt_build`; after rebase, the active automatic recall path can still inject memories from the shared table.
  - confidence: 0.88
- **[medium] Agent scope-key mismatch can break isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Writes use the raw context value while reads/deletes filter on a separately stripped value, so an unexpected context value can make scoped memories unreachable or collide with another normalized key.
  - confidence: 0.82
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The `agentId` column migration catches all errors without warning, leaving operators without a signal that scoped filtering may not be available for existing tables.
  - confidence: 0.84
