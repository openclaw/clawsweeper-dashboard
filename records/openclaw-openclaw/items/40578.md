## Summary

Keep open. This is a plausible memory-lancedb isolation feature, but the draft PR targets an obsolete auto-recall hook on current main, has scope-key and migration defects, lacks real behavior proof, and overlaps a still-open partition-design issue.

## What This Changes

Adds optional `agentScoping` config to `memory-lancedb`, an `agentId` LanceDB column/migration, scoped memory tool factories and hook changes, plus config, manifest, and test updates.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that uses one trusted partition key consistently across tools, auto-recall, auto-capture, duplicate checks, forget/delete, CLI/operator paths, migrations, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. With two agents or user partitions sharing the same `memory-lancedb` `dbPath`, current main stores rows without a scope key and memory recall searches the shared table, so cross-scope recall is structurally possible; I did not run live LanceDB proof in this read-only review.

## Solution Assessment

No. The direction is valid, but this branch should not merge until it rebases onto `before_prompt_build`, derives one validated scope key, reports migration failures, covers the live factory path, and includes real two-agent proof.
