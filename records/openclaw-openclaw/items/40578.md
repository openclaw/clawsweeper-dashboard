## Summary

Keep open: current main still uses one shared memory-lancedb store, so the PR targets a real gap, but the proposed patch is not mergeable as-is because it scopes a legacy hook path, handles agent IDs inconsistently, and leaves migration failures hard to diagnose.

## What This Changes

The PR adds an optional `agentScoping` setting to memory-lancedb, an `agentId` LanceDB column/migration, scoped memory tool and hook calls, manifest config metadata, and a small test update.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes `memory_recall`, `memory_store`, `memory_forget`, duplicate checks, auto-recall, and auto-capture on the current hook paths; uses one explicit scope-key contract; preserves legacy shared rows intentionally; logs migration failures; and adds two-agent isolation coverage plus config/docs updates.

## Reproduction Assessment

Yes. Current main can be verified statically: one shared `MemoryDB` is created and all memory operations omit agent identity. The PR-specific failures are visible in the patch by comparing raw `ctx.agentId` writes with normalized read/delete filters and the legacy hook being scoped instead of current `before_prompt_build`.

## Solution Assessment

No. The PR is directionally useful, but it is not the best mergeable solution because it is based on a legacy hook path and does not define one stable agent-scope key. A safer fix should rebase onto current main and settle the isolation contract before storing or filtering data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent scope key:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When scoping is enabled, this patch stores the raw `ctx.agentId` but reads and deletes with a stripped value. IDs containing characters such as `/` or spaces can become unrecoverable or collide with another scope, so validate/reject or encode once and use that same key for writes, duplicate checks, reads, and deletes.
  - confidence: 0.93
- **[P2] Scope the current prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and tests assert `before_agent_start` is not registered. This patch only scopes the legacy hook path, so automatic recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Report agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic, so surface a warning through the plugin logger.
  - confidence: 0.84
- **[P3] Instantiate tool factories in the live smoke path:** `extensions/memory-lancedb/memory-lancedb.live.test.ts:75-80`
  - body: This patch converts memory tools to factories, but the live memory-lancedb smoke test still retrieves registered tools and calls `.execute(...)` on the raw value. Once this changed registration lands, that live path will throw unless it invokes the factory with a context first.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and has concrete scoped-access correctness issues that can leak, hide, or misroute memories.

Concerns:

- **[high] Agent ID normalization can break memory isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter on a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
- **[high] Automatic recall can remain unscoped after rebase:** `extensions/memory-lancedb/index.ts:588`
  - body: The PR scopes the legacy auto-recall hook, while current main uses `before_prompt_build`. If rebased without moving the scope check, automatic prompt memory injection can still leak memories across agents.
  - confidence: 0.88
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows errors without logging, so a failed schema update can leave existing tables unscoped while operators receive no warning.
  - confidence: 0.82
