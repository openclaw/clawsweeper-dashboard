## Summary

Keep open: current main still has unscoped memory-lancedb storage and recall, but this draft PR is not safe to merge because it targets a stale hook path, uses inconsistent agent scope handling, hides migration failures, and only provides unit-test output rather than real multi-agent proof.

## What This Changes

The branch adds an optional `agentScoping` config/manifest field and changes memory-lancedb storage, search/delete, hooks, and tools to tag and filter memories by `ctx.agentId`.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes auto-recall, auto-capture, duplicate checks, tools, delete paths, and docs through one validated agent scope key while preserving legacy shared memories intentionally.

## Reproduction Assessment

Yes, source-level. Current main stores, searches, and auto-recalls memory-lancedb rows from one shared table with no agent scope; I did not run a live two-agent LanceDB setup.

## Solution Assessment

No. The direction is valid and uses an existing plugin tool-context seam, but this draft needs a rebase to the active hook path, one validated scope key, logged migration failures, focused docs/tests, and real behavior proof.

  - body: This branch adds scoped recall under `before_agent_start`, but current main injects memory through `before_prompt_build`. After rebase, automatic recall can still query the shared table and inject another agent's memories unless the active prompt-build path is scoped too.

  - confidence: 0.88

  - confidence: 0.86

  - body: The diff changes memory tools to factories so they can receive `ctx.agentId`, but the live smoke still reads registered tools and calls `.execute` directly. A live run will throw before exercising store, recall, or forget.
  - confidence: 0.82
