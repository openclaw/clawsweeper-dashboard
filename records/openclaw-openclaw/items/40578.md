## Summary

## What This Changes

The PR adds an opt-in `agentScoping` config option to `memory-lancedb`, stores an `agentId` column, and attempts to scope memory recall/store/forget plus lifecycle hooks by the current agent.

## Best Possible Solution

Land a fresh or rebased memory-lancedb isolation design that scopes tools plus current before_prompt_build/agent_end hooks by one validated runtime identity, preserves default shared behavior intentionally, surfaces migration failures, and includes real two-agent plus upgrade proof.

## Reproduction Assessment

Yes at source level: current main constructs one shared MemoryDB and calls search/store/delete without agent identity, so the reported shared-memory behavior is clear from source. I did not run a live two-agent LanceDB setup, so real runtime reproduction is still missing.

## Solution Assessment

No: the direction is useful, but this branch is stale and still has scope-key, migration, and proof defects. The maintainable path is a rebased implementation on current hooks with one validated identity, explicit migration behavior, and real two-agent proof.

- **[P1] Scope the active prompt-build hook:** `extensions/memory-lancedb/index.ts:591`
  - body: Current main moved memory auto-recall to `before_prompt_build`, and the docs mark `before_agent_start` as compatibility-only.   - confidence: 0.9
- **[P1] Validate one scope key before storing:** `extensions/memory-lancedb/index.ts:428`
  - body: This writes the raw `ctx.agentId`, while `search()` and `delete()` strip characters before filtering. Agent IDs like `agent/123` are stored under one value and later queried as `agent123`, so scoped memories can disappear or collide with another agent's scope.
  - confidence: 0.93
- **[P2] Report failed agentId migrations:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows every error, so an operator enabling scoped mode on a table where `addColumns` or `schema` fails gets no warning that the isolation column was not added. Surface this through plugin logging or an explicit diagnostic state.
  - confidence: 0.86
- **[P2] Update live tool callers for factories:** `extensions/memory-lancedb/index.ts:351`
  - body: Switching memory tools to factories means the registered live-test tool value is now a function, but `memory-lancedb.live.test.ts` still calls `.execute` directly. That live verification will throw before exercising the changed behavior unless the factory is unwrapped there too.
  - confidence: 0.88
