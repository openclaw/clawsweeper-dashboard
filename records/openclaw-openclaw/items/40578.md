## What This Changes

The PR adds an opt-in `agentScoping` config option to `memory-lancedb`, adds an `agentId` column/migration, and threads agent scope through memory tools and lifecycle hooks.

## Best Possible Solution

Land a fresh or rebased memory-lancedb isolation design that scopes tools plus current `before_prompt_build`/`agent_end` hooks by one validated runtime identity, preserves default shared behavior intentionally, surfaces migration failures, and includes real two-agent and upgrade proof.

## Reproduction Assessment

Yes at source level: current main constructs one shared `MemoryDB` and calls search/store/delete without agent identity, so the reported shared-memory behavior is clear from source. I did not run a live two-agent LanceDB setup, so real runtime reproduction is still not established here.

## Solution Assessment

No: the direction is useful, but this branch is stale and still has scope-key and migration defects. The maintainable path is a rebased implementation on current hooks with one validated identity, explicit migration behavior, and real two-agent proof.
