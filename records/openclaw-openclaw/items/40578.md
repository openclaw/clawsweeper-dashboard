## Summary

Keep open: current main still lacks memory-lancedb agent scoping, and this draft PR remains the canonical implementation track, but the branch is conflicting, misses the active prompt-build recall path, and lacks required real two-agent LanceDB proof.

## What This Changes

The PR adds an opt-in `agentScoping` config, an `agentId` LanceDB column/migration, scoped search/delete behavior, and context-aware memory tool factories for the memory-lancedb plugin.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes auto-recall, auto-capture, duplicate checks, tools, delete paths, and operator surfaces through one validated agent scope key, preserves legacy shared memories intentionally, logs migration failures, updates docs/tests, and includes live two-agent proof.

## Reproduction Assessment

Yes, source-level. Current main stores, searches, and auto-recalls memory-lancedb rows from one shared table without an agent scope; I did not establish a live two-agent LanceDB run.

## Solution Assessment

No. The direction is valid, but this PR scopes the legacy hook instead of the active prompt-build hook, uses inconsistent scope-key handling, hides migration failures, and lacks real behavior proof.
