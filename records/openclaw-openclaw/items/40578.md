## Summary

Keep open. Current main still lacks memory-lancedb isolation, and this draft PR is a plausible implementation track, but it is not merge-ready because the branch is stale against the active prompt-build hook, lacks real two-agent proof, and has blocking scope-key and migration defects.

## What This Changes

The branch adds an opt-in `agentScoping` config to memory-lancedb, adds an `agentId` LanceDB column/migration, filters memory search/delete paths by agent context, and updates config metadata plus a unit test mock.

## Best Possible Solution

Land a rebased isolation design that uses one trusted scope identity across tools, auto-recall, auto-capture, duplicate checks, delete/forget, CLI/operator paths, migration diagnostics, docs, regression tests, and redacted real two-agent proof.

## Reproduction Assessment

Yes, source-level. Current main uses one memory-lancedb `MemoryDB`, one `memories` table, and unscoped search/store/delete calls from tools plus `before_prompt_build` and `agent_end`, so two agents sharing one `dbPath` can recall shared memories.

## Solution Assessment

No. Agent scoping is a plausible direction, but this branch is not the best mergeable form until it is rebased to current main, scopes the active prompt-build hook, uses one validated scope key, handles migration failure visibly, and includes real behavior proof.
