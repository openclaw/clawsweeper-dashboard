## Summary

Keep open. Current main still uses one shared LanceDB memory store, and this PR is the canonical implementation track, but the branch is draft/conflicting with blocking privacy-boundary findings and missing real two-agent proof.

## What This Changes

The branch adds an optional memory-lancedb `agentScoping` config, an `agentId` LanceDB column/migration, and scoped memory tool/hook behavior with updated plugin metadata and tests.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes auto-recall, auto-capture, duplicate checks, tools, and deletes through one validated scope identity, with migration diagnostics, docs, regression coverage, and redacted real two-agent proof.

## Reproduction Assessment

Yes, source-level. Configure two agents against the same memory-lancedb `dbPath`, store from one scope, then recall from the other; current main has no row field, scoped DB selection, or search filter to prevent cross-agent results.

## Solution Assessment

No. The direction is plausible, but this branch is not the best mergeable form until it is rebased to the current prompt-build hook path, uses one validated scope key, reports migration failures, and includes real behavior proof.

Full review comments:

- **[P2] Scope the active prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-601`
  - body: This branch scopes auto-recall on the legacy `before_agent_start` hook, but current main injects LanceDB memories from `before_prompt_build`. After rebase, the active prompt-build path can still call the shared `db.search()` and inject another agent's memory unless that path is scoped too.
  - confidence: 0.91

  - body: The patch stores raw `ctx.agentId` values but searches and deletes after stripping disallowed characters. IDs such as `agent/one` can be stored under one key and read/delete under another, making scoped memories disappear or collide after normalization.
  - confidence: 0.88
- **[P3] Report failed agentId migrations:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn()` catches schema or `addColumns` failures without warning. If an existing table cannot add `agentId`, operators can enable scoping while the table remains unscoped with no diagnostic explaining why isolation is not working.
  - confidence: 0.86
- **[P3] Unwrap tool factories in the live smoke:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch converts registered memory tools to factories, but the live memory smoke still reads `registeredTools[i].tool` and calls `.execute` directly. A live run will throw before exercising store, recall, or forget unless the test invokes the factory with a tool context.
  - confidence: 0.82
