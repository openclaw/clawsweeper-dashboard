## Summary

Keep this PR open, but do not merge it as-is. Current main still has the unscoped memory-lancedb behavior the PR is trying to address, while the proposed patch is stale against current main and introduces blocking correctness issues around agent identity, active auto-recall hooks, and migration observability.

## What This Changes

The PR adds an optional `agentScoping` setting to memory-lancedb, adds an `agentId` LanceDB column and best-effort migration, filters memory reads/deletes by agent, and refactors memory tools into context-aware factories.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes `memory_recall`, `memory_store`, `memory_forget`, `before_prompt_build`, and `agent_end`; uses one validated or encoded scope key consistently; preserves or migrates legacy shared rows deliberately; logs migration failures; updates docs/config metadata; and adds two-agent isolation regression coverage.

## Reproduction Assessment

Yes. Current main can be checked statically: memory-lancedb has one shared `MemoryDB`, no `agentId` schema/config, and unscoped `db.search`/`db.store` calls. The PR-specific key mismatch is reproducible from the patch with an agent ID containing a slash or space.

## Solution Assessment

No. The PR is directionally useful, but it is not the best mergeable solution because it scopes the legacy auto-recall hook path and does not define a consistent agent-ID storage/filter contract for the memory isolation boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one stable agent scope key:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When scoping is enabled, this patch stores the raw `ctx.agentId` but reads and deletes with a stripped value. IDs containing characters such as `/` or spaces can become unrecoverable or collide with another scope, so validate/reject or encode once and use that same key for writes, duplicate checks, reads, and deletes.
  - confidence: 0.93
- **[P2] Scope the active prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and tests assert `before_agent_start` is no longer registered. This patch scopes the legacy hook path, so automatic recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Report agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic, so surface a warning through the plugin logger.
  - confidence: 0.84
- **[P3] Instantiate tool factories in the live smoke path:** `extensions/memory-lancedb/index.ts:351`
  - body: This patch converts memory tools to factories, but the live memory-lancedb smoke test still retrieves registered tools and calls `.execute(...)` on the raw value. Once this changed registration lands, that live path will throw unless it invokes the factory with a context first.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, workflows, or secret handling, but it changes a memory isolation boundary and has concrete scoped-access correctness concerns.

Concerns:

- **[high] Agent ID normalization can break memory isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter on a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows errors without logging, so a failed schema update can leave existing tables unscoped while operators receive no warning.
  - confidence: 0.82
