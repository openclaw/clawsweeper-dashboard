## Summary

Keep open. The PR addresses a real, still-unimplemented memory-lancedb isolation problem, but it is a draft/conflicting external branch with missing real behavior proof and merge-blocking defects around the active recall hook, scope-key consistency, and migration diagnostics.

## What This Changes

The PR adds an opt-in `agentScoping` setting for memory-lancedb, an `agentId` LanceDB column/migration, and scoped memory tool/hook reads and writes.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that derives one trusted scope key, applies it consistently to tools, `before_prompt_build` auto-recall, auto-capture, duplicate checks, migrations, CLI/docs, and tests, and aligns the product boundary with the broader partition work.

## Reproduction Assessment

Yes, source-level. Configure two agents or users against the same memory-lancedb `dbPath`, store a memory from one, then recall/search from the other; current main has one table and no stored or queried scope key to prevent cross-scope results.

## Solution Assessment

No. The requested direction is valid, but this branch is not the best mergeable solution until it is rebased onto current main, scopes the active hook path, uses one validated scope-key contract, surfaces migration failures, and includes real two-agent proof.
