## Summary

Keep this draft PR open: the per-agent memory isolation request is still not implemented on current main, but the proposed patch is stale against the current hook path and has blocking scoped-access bugs.

## What This Changes

The PR adds an opt-in `agentScoping` setting to `memory-lancedb`, an `agentId` LanceDB column/migration, scoped search/delete/store behavior, config metadata, and a small tool-factory test update.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that scopes tools, duplicate checks, auto-recall, and auto-capture on current hook paths with one validated agent scope key, migration diagnostics, docs/changelog, and two-agent regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows one shared `MemoryDB` and unscoped memory operations, and the PR-specific regressions are visible directly in the proposed diff.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best mergeable solution until it is rebased onto current main and uses one stable, validated scope-key contract across writes, reads, deletes, and hooks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent scope key:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When scoping is enabled, this patch writes raw `ctx.agentId` values but reads/deletes with a stripped value. Agent IDs containing disallowed characters can become unrecoverable or collide, so validate/reject or encode once and use that same key everywhere.
  - confidence: 0.91
- **[P2] Scope the current prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build` and tests assert `before_agent_start` is not registered. This patch scopes only the legacy hook path, so auto-recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Report agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` swallows schema or `addColumns` failures without warning. Operators can enable scoping while existing tables stay effectively unscoped with no diagnostic, so thread a logger or return a warning signal.
  - confidence: 0.84
- **[P3] Update live smoke for tool factories:** `extensions/memory-lancedb/index.ts:351-362`
  - body: This patch converts memory tools to factories, but the live smoke path still calls `.execute` on the raw registered tool. Once this lands, the live test must instantiate the factory with a context before calling `execute`.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes memory data isolation semantics and has scoped-access bugs that can hide, leak, or misroute memories if merged as-is.

Concerns:

- **[high] Agent ID normalization can break isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes use raw `ctx.agentId` while scoped reads/deletes filter on a stripped version; unexpected IDs can become unrecoverable or collide with another normalized ID.
  - confidence: 0.9
- **[high] Automatic recall can remain unscoped:** `extensions/memory-lancedb/index.ts:588`
  - body: The PR scopes the stale `before_agent_start` hook while current main uses `before_prompt_build`, so a stale rebase can still inject memories across agents on the active prompt path.
  - confidence: 0.88
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The `agentId` column migration catches failures without logging, leaving operators without a warning that scoped filtering may not be active on existing tables.
  - confidence: 0.82
