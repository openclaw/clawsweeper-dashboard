## Summary

Keep open: the requested memory isolation problem is real on current main, but this draft branch is stale against the active hook path, has scoped-access defects, and lacks required real multi-agent behavior proof.

## What This Changes

The PR adds an opt-in `agentScoping` setting, an `agentId` LanceDB field/migration, and scoped memory tool/hook reads and writes for the memory-lancedb plugin.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that derives one trusted scope key, applies it consistently to tools, duplicate checks, auto-recall, auto-capture, migrations, CLI semantics, docs, and tests, while coordinating the product boundary with https://github.com/openclaw/openclaw/issues/63550.

## Reproduction Assessment

Yes, source-level: current main stores and searches through one shared LanceDB table with no agent, user, tenant, namespace, or partition key in the schema or query paths. I did not run a live two-agent LanceDB setup in this read-only review.

## Solution Assessment

No. The requested direction is valid, but this branch is not the best mergeable solution until it is rebased onto current main, uses one validated scope-key contract, scopes the active hook path, surfaces migration failures, and includes real two-agent proof.

  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and tests assert the legacy hook is not registered. This branch scopes `before_agent_start`, so after rebase automatic recall would still read the shared DB unless the current hook receives the same scope.

  - confidence: 0.86

  - body: `migrateAgentIdColumn` catches schema or `addColumns` failures without logging. Operators can enable scoping while an existing table remains effectively unscoped, with no diagnostic explaining why isolation is not active.
  - confidence: 0.86

  - body: This patch converts memory tools to factories, but the live smoke still retrieves the registered tool and calls `.execute` directly. Once this change lands, the live test must instantiate the factory with a context before calling `execute`.
  - confidence: 0.82
