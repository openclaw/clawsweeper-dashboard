## Summary

Keep open: current main still lacks memory-lancedb agent scoping, and related reports were closed to consolidate on this PR, but the branch is draft/conflicting and has blocking correctness/security gaps plus mock-only proof.

## What This Changes

Adds optional `agentScoping` config to the memory-lancedb plugin, an `agentId` LanceDB column/migration, and scoped memory tool/hook behavior based on the calling agent context.

## Best Possible Solution

Land a rebased memory-lancedb isolation design that uses one trusted partition key consistently across tools, active hooks, duplicate checks, forget/delete, CLI/operator surfaces, migrations, docs, and regression plus live two-agent proof.

## Reproduction Assessment

Yes, source-level. Current main stores MemoryEntry rows with no agent/user scope and auto-recall calls `db.search` from `before_prompt_build` without a scope filter; no live two-agent LanceDB run was provided.

## Solution Assessment

No. The direction is valid, but this branch should not merge until it rebases onto the active hook/config surface, uses one validated scope key, reports migration failures, updates the live factory path, and includes real two-agent proof.
