## Summary

Keep open: current main still keeps sender names/ids out of trusted inbound metadata, while the remaining fix requires a maintainer security/product decision and reconciliation of overlapping open implementation PRs.

## What This Changes

Request adding a trusted, platform-resolved sender display/principal identity to inbound metadata so agents do not infer names from untrusted conversational context.

## Best Possible Solution

Choose one canonical trusted requester identity contract, preferably stable-id/identityLinks-backed, reconcile the open Discord-specific and generic PRs, and land it with tests, docs, privacy notes, and real setup proof while keeping mutable display names out of trusted metadata.

## Reproduction Assessment

Yes for the source-level implementation gap: current main has no trusted sender display/principal field in inbound metadata and renders sender identity only in untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: adding raw sender_display_name to trusted metadata is weaker than a canonical, stable-id-backed principal. The identityLinks/requester_identity direction looks safer, but maintainers need to choose the public contract before merge.
