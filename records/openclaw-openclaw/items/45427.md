## Summary

Keep open: current main and the latest release still omit a trusted sender display/principal field from inbound metadata. The linked Discord implementation is the direct path, but it is still open and proof-gated, and the trusted-identity semantics need maintainer product/security review.

## What This Changes

The issue requests a trusted sender display/principal field in inbound metadata so agents do not infer human identity from untrusted conversation text.

## Best Possible Solution

Define a canonical trusted sender principal, preferably operator-controlled through identityLinks or a documented channel-directory contract, then land the Discord/core plumbing with tests, docs, privacy alignment, and real setup proof.

## Reproduction Assessment

Yes for the implementation gap: current main and v2026.5.7 source show no trusted sender display-name/principal field in inbound metadata. I did not reproduce the reported production DM misroute in a live fleet.

## Solution Assessment

Unclear as originally proposed: promoting a raw sender_display_name into trusted metadata may conflict with the current prompt-trust and privacy boundary. The identityLinks-based sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it still needs maintainer review and proof.
