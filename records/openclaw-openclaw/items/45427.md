## Summary

Keep open: current main and the latest release still lack a trusted sender identity field in the inbound metadata prompt, and the remaining fix is already being explored by open related PRs with competing schemas that need maintainer security/product judgment.

## What This Changes

The issue requests trusted sender identity metadata for inbound messages so agents do not resolve recipients from untrusted conversation context.

## Best Possible Solution

Choose one stable provider-id-backed identity contract, likely based on `session.identityLinks`, keep mutable display names in untrusted context, and reconcile https://github.com/openclaw/openclaw/pull/70944 with https://github.com/openclaw/openclaw/pull/82353 before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.22` have no trusted sender-name, `sender_principal`, or `requester_identity` field, and sender facts remain in untrusted metadata blocks. I did not live-reproduce the reported March 13, 2026 production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `session.identityLinks`-backed contract such as `requester_identity` or `sender_principal`.
