## Summary

Keep open: current main still omits a trusted sender display/principal field from inbound metadata, and the open Discord implementation remains proof-gated. The request is valid but needs maintainer product/security judgment before adding human identity data to trusted prompt metadata.

## What This Changes

Request is to add a trusted sender identity/name field to inbound metadata so agents do not infer people from untrusted conversational context.

## Best Possible Solution

Define a canonical trusted sender principal, preferably operator-controlled through identityLinks or another documented channel-directory contract, then land the Discord/core plumbing with tests, docs, and real setup proof.

## Reproduction Assessment

Yes for the implementation gap: current main and v2026.5.7 source show no trusted sender display-name/principal field in inbound metadata. I did not reproduce the reported production DM misroute in a live fleet.

## Solution Assessment

Unclear as originally proposed: promoting raw sender_display_name into trusted metadata may conflict with current prompt-trust and privacy boundaries. The identityLinks-based sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it still needs maintainer review and proof.
