## Summary

Keep open: current main and latest release still keep sender identity in untrusted prompt context and do not provide a trusted requester identity field. The issue remains valid, but the right fix needs maintainer product/security judgment because raw trusted display names are weaker than a stable `session.identityLinks`-backed principal, and open related PRs propose overlapping schemas.

## What This Changes

The issue asks OpenClaw to add a trusted sender name or equivalent identity field to inbound metadata so agents do not infer recipients from untrusted conversational context.

## Best Possible Solution

Choose one stable provider-id-backed identity contract, likely through `session.identityLinks`, keep mutable display names in untrusted context, and reconcile the generic and Discord-specific PRs before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.22` have no trusted sender-name or canonical requester identity field, and sender facts remain in untrusted metadata blocks. I did not live-reproduce the reported production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed field such as `requester_identity` or `sender_principal`.
