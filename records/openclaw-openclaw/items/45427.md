- `impact:message-loss`: The failure mode is messages being routed to the wrong person after identity was inferred from untrusted context.

## Summary

Keep open: current main and the latest release still omit a trusted sender principal/name from the inbound metadata envelope, and the open follow-up work has not yet resolved the product/security contract for canonical identity.

## What This Changes

Request to add a trusted sender identity field to inbound metadata so agents do not infer human recipients from untrusted conversational context.

## Best Possible Solution

Choose one stable provider-id-backed `session.identityLinks` identity contract, reconcile the generic and Discord-specific PRs, and keep mutable display names in untrusted context.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.19` have no trusted canonical sender identity field and keep sender identifiers/names in untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `sender_principal` or `requester_identity`.
