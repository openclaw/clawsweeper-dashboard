## Summary

Keep open: current main still keeps human sender names out of trusted inbound metadata and only surfaces sender labels in untrusted user context, while the related implementation PR remains open for proof and maintainer acceptance.

## What This Changes

Requests adding a trusted sender display or canonical name to inbound metadata so agents do not infer identity from untrusted conversation context.

## Best Possible Solution

Settle and land a trusted canonical identity field through the inbound metadata contract, preferably derived from existing `identityLinks` or a channel-owned trusted lookup, with display names kept untrusted unless their source and semantics are explicit.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature/contract request. Source inspection on current main gives a high-confidence verification path: the trusted inbound payload excludes sender identity while sender labels are rendered as untrusted user context.

## Solution Assessment

Unclear for raw `sender_display_name` as requested. The related `sender_principal` approach via existing `session.identityLinks` looks safer than trusting mutable display names, but it still needs real behavior proof and maintainer product/security acceptance.
