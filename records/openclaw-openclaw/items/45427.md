## Summary

Keep this open: current main and the latest release still do not expose a trusted sender identity in the inbound metadata envelope, and the current implementation deliberately keeps sender names in untrusted context. The remaining work is a maintainer security/product decision between the open identity-contract paths, not a cleanup close.

## What This Changes

The issue requests a trusted sender display-name or canonical identity field in inbound metadata so agents do not infer recipients from untrusted conversational context.

## Best Possible Solution

Settle one trusted identity contract backed by stable channel IDs and session.identityLinks, keep human display names untrusted, then land a refreshed canonical PR or close the competing branch.

## Reproduction Assessment

No for the full wrong-recipient DM incident: that depends on fleet prompts, session state, and delivery history that were not replayed. Source inspection does confirm the current-main gap: trusted metadata has no canonical sender identity while SenderName/SenderId remain in untrusted user-context blocks.

## Solution Assessment

No, not as the raw trusted sender-name proposal. The safer solution is a canonical identity derived from stable sender IDs and operator-configured identityLinks, with maintainers choosing the field name and resolver boundary before merge.
