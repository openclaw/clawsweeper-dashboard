- `P1`: The issue describes production DM misrouting and session contamination from an identity trust gap affecting real agent/channel workflows.
- `impact:security`: The requested change changes which human identity facts are treated as trusted prompt metadata.
- `impact:message-loss`: The reported failure mode is messages being sent to the wrong person because the agent inferred identity from untrusted context.

## Summary

Keep open: current main and v2026.5.19 still keep sender identity out of trusted inbound metadata, and the remaining decision is the trusted canonical identity contract rather than a raw display-name field.

## What This Changes

Add a trusted sender display/name identity to inbound metadata so agents do not infer a recipient from untrusted conversation context.

## Best Possible Solution

Choose one stable provider-id-backed `session.identityLinks` identity contract, reconcile the generic and Discord-specific PRs, and avoid promoting mutable display names as trusted metadata.

## Reproduction Assessment

Yes for the source-level gap: current main and v2026.5.19 have no trusted sender display/principal field and keep sender identity in untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: a raw trusted display name is weaker than a canonical stable-id-backed principal. The best path is a maintainer-approved identityLinks-backed contract, not multiple independent prompt field names.
