## Summary

Keep open: current main and the latest release still lack a trusted sender display/principal field in the inbound metadata envelope, and the remaining work is a security/product contract choice with overlapping open implementation PRs rather than a safe standalone cleanup close.

## What This Changes

Add a trusted, OpenClaw-resolved sender display/principal field to inbound metadata so agents do not infer requester identity from untrusted conversation text.

## Best Possible Solution

Choose one stable-id-backed requester identity contract, reconcile the generic and Discord-specific PRs, and land it with focused tests, docs/privacy notes, and redacted real setup proof while avoiding mutable display names as trusted metadata.

## Reproduction Assessment

Yes for the source-level implementation gap: current main has no trusted sender display/principal field in inbound metadata and renders sender identity only as untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: adding a raw trusted display name is weaker than a canonical stable-id-backed principal. The `session.identityLinks` based `requester_identity` or `sender_principal` direction looks safer, but maintainers need to choose the public contract before merge.
