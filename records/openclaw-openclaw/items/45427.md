## Summary

Keep open: current main still does not expose a trusted sender display-name/principal in inbound metadata, and the related #70944 implementation is open and needs rework. This changes prompt trust and privacy posture, so it needs maintainer direction before any cleanup close.

## What This Changes

Request adding an OpenClaw-resolved trusted sender display name or principal to inbound metadata so agents do not infer identity from conversational context.

## Best Possible Solution

Define a canonical trusted sender identity contract, likely a sender_principal derived from operator-controlled identityLinks or a clearly documented channel-directory source, then wire it through inbound metadata with focused Discord/core tests and docs.

## Reproduction Assessment

Yes for the implementation gap: current main source and tests show trusted inbound metadata omits sender identity while SenderName is only rendered in untrusted user-context blocks. No live reproduction was attempted for the reported production DM misroute itself.

## Solution Assessment

Unclear: adding a raw trusted sender_display_name may not be the safest contract because names are mutable and currently treated as untrusted. The sender_principal direction in #70944 looks safer, but it needs maintainer review and completion before this issue can close.
