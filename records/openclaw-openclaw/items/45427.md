## Summary

Keep open. Current main and v2026.6.6 still keep sender identity out of the trusted inbound metadata envelope; the open Discord sender-principal PR is useful implementation evidence, but it is unmerged and the trusted identity schema remains a maintainer/security decision.

## What This Changes

Request to add a trusted, platform-resolved sender identity/name to inbound metadata so agents do not infer recipients from untrusted conversational context.

## Best Possible Solution

Approve one canonical trusted sender identity contract, likely a stable principal derived from configured identityLinks rather than a raw display name, then rebase and land https://github.com/openclaw/openclaw/pull/70944 or an equivalent current-main implementation with focused tests and real behavior proof.

## Reproduction Assessment

No for the full production misroute: the fleet prompts, delivery history, and session state were not replayed. Source inspection does confirm the current-main capability gap: trusted metadata lacks a canonical sender identity while sender display data remains in untrusted context blocks.

## Solution Assessment

Unclear: adding a raw trusted sender_display_name field is not obviously the best fix because it would make human-identifying data authoritative in the system prompt. The safer direction appears to be a canonical principal derived from stable channel IDs and identityLinks, with maintainers choosing the final schema.
