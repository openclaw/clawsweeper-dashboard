## Label Justifications

- `P1`: The issue reports production DM misrouting and session contamination from an identity trust gap in agent/channel workflows.
- `impact:security`: The requested change affects which human identity facts are safe to promote into trusted model metadata.
- `impact:message-loss`: The reported failure mode is messages being routed to the wrong recipient after identity was inferred from untrusted context.
- `impact:session-state`: The discussion describes polluted agent context after the wrong identity entered the session.

## Summary

Keep open: current main and v2026.5.20 still do not provide a trusted sender name or canonical requester identity in the inbound trusted metadata. The source-level gap is real, but the safe fix needs maintainer product/security judgment because raw display names are mutable and the two open implementation PRs propose different trusted identity shapes.

## What This Changes

Request adding a trusted sender identity/name field to the inbound metadata envelope so agents do not derive routing identity from untrusted conversation context.

## Best Possible Solution

Choose one stable provider-id-backed trusted identity contract, likely based on `session.identityLinks`, keep mutable display names in untrusted context, and reconcile https://github.com/openclaw/openclaw/pull/82353 with https://github.com/openclaw/openclaw/pull/70944 before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and v2026.5.20 have no trusted sender-name or canonical requester identity field, and sender facts remain in untrusted metadata blocks. I did not live-reproduce the reported March 13 production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `requester_identity` or `sender_principal`.
