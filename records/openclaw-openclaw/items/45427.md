## Summary

Keep open: current main and the latest release still do not provide a trusted sender-name or canonical requester identity in inbound prompt metadata, and the remaining work is security-sensitive because raw display names are weaker than a stable provider-id-backed identity contract.

## What This Changes

Request to add a platform-resolved trusted sender identity/name to inbound metadata so agents do not infer identity from untrusted conversation context.

## Best Possible Solution

Adopt one stable provider-id-backed identity contract, likely based on `session.identityLinks`, keep mutable display names in untrusted context, and reconcile the open generic and Discord-specific PRs before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.20` have no trusted sender-name or canonical requester identity field, and sender facts remain in untrusted metadata blocks. I did not live-reproduce the reported March 13 production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `requester_identity` or `sender_principal`.
