## Summary

Keep open: current main and the latest release still omit a trusted sender display/principal field from inbound metadata, and the linked Discord implementation remains open. The request is valid but security-sensitive because current prompt hardening deliberately keeps human names out of trusted metadata.

## What This Changes

The issue requests a trusted platform-resolved sender display/principal name in inbound metadata so agents do not infer identities from untrusted conversation context.

## Best Possible Solution

Define a canonical trusted sender principal, preferably from operator-controlled identityLinks or a documented channel-directory source, then thread that field through inbound metadata with Discord/core tests, docs, and real setup proof before closing the issue.

## Reproduction Assessment

Yes for the implementation gap: current main source, tests, and release contents show no trusted sender display-name/principal field. I did not reproduce the reported production DM misroute in a live fleet.

## Solution Assessment

Unclear as proposed: promoting a raw sender_display_name into trusted metadata may conflict with current prompt-trust and privacy boundaries. The identityLinks-based sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it still needs maintainer review and proof.
