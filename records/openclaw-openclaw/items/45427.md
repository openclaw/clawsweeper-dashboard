## Summary

Keep open: current main still lacks a trusted sender display-name/principal in the inbound metadata envelope, and the related implementation is still open while privacy work raises unresolved product/security questions around raw human names.

## What This Changes

Add a trusted platform-resolved sender display name or principal to inbound prompt metadata so agents do not infer identity from untrusted conversation context.

## Best Possible Solution

Define a canonical trusted sender identity contract, likely sender_principal from operator-controlled identityLinks or a documented channel-directory source, then wire it through inbound metadata with focused Discord/core tests and docs.

## Reproduction Assessment

Yes for the implementation gap: current main source and tests show trusted inbound metadata omits sender identity while SenderName is rendered only in untrusted user-context blocks. I did not reproduce the reported production DM misroute itself.

## Solution Assessment

Unclear: adding a raw trusted sender_display_name is not clearly the safest fix because names are mutable and current code explicitly keeps them untrusted. The sender_principal direction in #70944 looks closer, but it needs maintainer review and completion before this can close.
