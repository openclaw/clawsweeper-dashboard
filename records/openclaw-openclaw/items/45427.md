## Summary

Keep open. Current main and v2026.6.8 still omit trusted sender identity from the authoritative inbound metadata payload; the related open Discord `sender_principal` PR is useful but unmerged/conflicting and the trusted identity schema remains a maintainer/security decision.

## What This Changes

Request to add a trusted sender name/display or canonical identity to inbound metadata so agents do not infer recipients from untrusted conversation context.

## Best Possible Solution

Approve one canonical trusted sender identity contract, likely a stable principal derived from configured `session.identityLinks` rather than a raw display name, then land https://github.com/openclaw/openclaw/pull/70944 or an equivalent current-main implementation with focused tests and proof.

## Reproduction Assessment

No for the full production misroute; the fleet prompts, delivery history, and session state were not replayed. Source inspection does confirm the current-main capability gap: trusted metadata lacks canonical sender identity while sender display data remains in untrusted blocks.

## Solution Assessment

Unclear: adding raw `sender_display_name` is not obviously best because it would make human-identifying data authoritative in the system prompt. A canonical principal derived from stable channel IDs and `session.identityLinks` appears safer, but maintainers need to choose the final schema.
