## Summary

Keep open: current main and v2026.5.12 still omit a trusted sender display/principal field from inbound prompt metadata. The linked Discord and gateway provenance PRs are useful implementation paths, but they remain open and the identity trust semantics need maintainer product/security review.

## What This Changes

Request adding a trusted resolved sender display/principal field to inbound metadata so agents do not infer names from untrusted conversation context.

## Best Possible Solution

Define one canonical trusted sender principal contract, preferably operator-controlled through identityLinks or a documented channel-directory seam, then land aligned Discord/core plumbing with tests, docs, privacy notes, and real setup proof.

## Reproduction Assessment

Yes for the implementation gap: current main and v2026.5.12 source show no trusted sender display-name/principal field in inbound metadata. I did not reproduce the reported production DM misroute in a live fleet.

## Solution Assessment

Unclear as originally proposed: promoting a raw sender_display_name into trusted metadata may conflict with the current prompt-trust and privacy boundary. The identityLinks-based sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it still needs maintainer review and proof.
