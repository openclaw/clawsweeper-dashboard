## Label Justifications

- `impact:security`: The issue is about preventing untrusted conversational names from being treated as trusted identity for routing private messages.

## Summary

Keep open: current main and the latest release still do not expose a trusted sender name or principal in the inbound metadata envelope, and the related Discord implementation at https://github.com/openclaw/openclaw/pull/70944 is still open rather than merged. The issue is real, but the literal trusted display-name shape needs maintainer product/security review because current code deliberately keeps human names in untrusted prompt context.

## What This Changes

The issue requests a trusted sender display/principal identity in the inbound metadata envelope so agents do not infer names from untrusted conversation context.

## Best Possible Solution

Adopt a maintainer-approved trusted principal design, likely identityLinks-backed and channel-resolved, while keeping raw human display names in untrusted context unless a stronger trust contract is defined.

## Reproduction Assessment

No high-confidence live reproduction was established here, but the source path is clear: current main has no trusted sender name/principal field and keeps sender labels in untrusted metadata. The production misroute itself would need a real channel/session replay to reproduce end to end.

## Solution Assessment

No, the literal trusted sender_display_name proposal is not clearly the best shape. A stable identityLinks-backed sender_principal is safer because display names can be spoofable and privacy-sensitive.
