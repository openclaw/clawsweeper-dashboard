## Summary

Keep this open: current main and v2026.6.6 still do not expose a trusted sender identity field, and sender display data remains explicitly untrusted. The remaining work is a maintainer security/product decision between the open ID-backed identity-contract PRs, not a cleanup close.

## What This Changes

The issue requests a trusted sender display identity in the inbound metadata envelope so agents do not infer names from untrusted conversational context.

## Best Possible Solution

Settle one stable ID-backed trusted identity contract, likely based on `session.identityLinks`, then land or close the competing open PRs under that chosen schema.

## Reproduction Assessment

No for the full wrong-recipient incident: that depends on fleet prompts, session state, and delivery history that were not replayed. Source inspection does confirm the current-main gap: trusted metadata lacks a canonical sender identity while sender display data remains in untrusted context blocks.

## Solution Assessment

No as a raw trusted sender-name field. The safer solution is a canonical identity derived from stable channel IDs and operator-configured identityLinks, with maintainers choosing the field name and resolver boundary before merge.
