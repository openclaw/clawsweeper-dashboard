## Summary

Keep open. Current main carries `SenderName` through the Discord and agent-runner paths, but the prompt metadata contract still does not expose an explicit `sender_name` or `sender_display_name` field; it only folds the value into a generic `sender` field and a separate sender block labeled as untrusted metadata. Because the request is about trusted identity metadata and misrouting prevention, closing as implemented would be too strong.

## What This Changes

The issue requests that OpenClaw add a trusted, platform-resolved sender display name to inbound message metadata so agents do not infer names from conversation context.

## Best Possible Solution

Define an explicit inbound identity contract for display names. A conservative implementation would add an optional `sender_display_name` or `sender_name` field derived from `SenderName` to the per-turn user-context metadata and tests, while preserving the existing rule that arbitrary human strings stay out of the stable trusted system prompt unless maintainers intentionally change that trust model.

## Reproduction Assessment

Yes for the reported metadata gap: current main can be statically checked in `buildInboundUserContextPrefix`, where `sender_id` exists but no explicit `sender_name` or `sender_display_name` key is emitted. The production misrouting incident itself is not reproducible from the repository alone.

## Solution Assessment

Unclear. Adding a display-name field is narrow, but the requested placement in trusted metadata needs maintainer judgment because current code deliberately separates human names from the trusted system block.

## Risks / Open Questions

- Changing this exactly as requested could conflict with the current prompt-design rule that keeps human names out of the trusted system metadata block for prompt-cache and injection safety.
- Not every channel has the same directory-backed display-name semantics as Discord, so the contract should define whether this is optional and whether it means username, display name, nickname, or server directory name.
- The production Mendel misroute was not live-reproduced here; the high-confidence check is limited to the missing explicit metadata field on current main.
