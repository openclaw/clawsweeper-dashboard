## Summary

Keep open: current main still has no trusted sender name/principal in the inbound metadata envelope, and the remaining work is a security/product contract decision with overlapping open implementation PRs.

## What This Changes

Add a trusted platform-resolved sender display or principal field to inbound metadata so agents do not infer sender identity from untrusted conversation context.

## Best Possible Solution

Choose one stable-id-backed trusted requester identity contract, reconcile the generic and Discord-specific PRs, and land it with focused tests, docs/privacy notes, and redacted real setup proof while keeping mutable display names out of trusted metadata.

## Reproduction Assessment

Yes for the source-level implementation gap: current main has no trusted sender display/principal field in inbound metadata and renders sender identity only in untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: adding a raw trusted display name is weaker than a canonical stable-id-backed principal. The `identityLinks`/`requester_identity` direction looks safer, but maintainers need to choose the public contract before merge.
