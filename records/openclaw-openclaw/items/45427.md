## Summary

Keep open. Current main still keeps human sender names out of the trusted inbound metadata payload and only renders sender labels in untrusted user-context blocks, while the related Discord PR is open and still needs maintainer/proof follow-up.

## What This Changes

Add a trusted resolved sender display name or principal to inbound metadata so agents do not infer identity from untrusted conversation context.

## Best Possible Solution

Choose the trusted identity contract, then expose it through the trusted inbound metadata path with Discord coverage and docs, preferably reusing `session.identityLinks` when operator-controlled principals are the intended source of truth.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence that current main lacks a trusted sender name/principal field, but the production misrouting incident itself is not replayable from the issue data.

## Solution Assessment

Unclear. The related identityLinks-based `sender_principal` approach looks safer than trusting mutable display names, but maintainers need to decide whether it satisfies the original server-directory `sender_display_name` request.
