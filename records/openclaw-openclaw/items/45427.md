## Summary

Keep this open: current main and v2026.6.6 still do not expose a trusted sender identity field, and current code deliberately keeps sender names in untrusted context. The remaining work is a maintainer security/product decision between the open identity-contract paths, not a cleanup close.

## What This Changes

Request to add trusted OpenClaw-resolved sender identity metadata to inbound prompts so agents do not infer recipient identity from untrusted conversational context.

## Best Possible Solution

Settle and document one stable ID-backed trusted identity contract, likely using session.identityLinks, then land or close the competing open PRs under that chosen schema.

## Reproduction Assessment

No for the full wrong-recipient incident: that depends on fleet prompts, session state, and delivery history that were not replayed. Source inspection does confirm the current-main gap: trusted metadata lacks a canonical sender identity while sender display data remains in untrusted context blocks.

## Solution Assessment

No as a raw trusted sender-name field. The safer solution is a canonical identity derived from stable channel IDs and operator-configured identityLinks, with maintainers choosing the field name and resolver boundary before merge.
