## Summary

Keep open. Current main and the latest release still do not provide a trusted sender identity in the inbound system metadata, and the open Discord implementation is only a candidate because it is unmerged, dirty against base, and still needs maintainer/security approval for the model-visible identity contract.

## What This Changes

Request to add a trusted sender identity field to inbound metadata so agents do not infer recipients from untrusted conversational names.

## Best Possible Solution

Approve one canonical trusted sender identity contract, preferably a stable principal derived from channel IDs and session.identityLinks, then land a rebased implementation with focused prompt, routing, and channel tests plus real behavior proof.

## Reproduction Assessment

No for the full production misroute: the fleet prompts, delivery history, and session state were not replayed. Source inspection does confirm the current-main gap: trusted metadata lacks a stable sender identity while sender names and IDs remain in untrusted context blocks.

## Solution Assessment

No for the literal raw sender_display_name proposal because making mutable human names trusted model metadata has privacy and spoofing tradeoffs. The safer direction is a maintainer-approved canonical principal derived from stable channel IDs and identityLinks.
