## Summary

Keep open: current main still lacks a trusted sender display/principal field in inbound prompt metadata, and the only related implementation remains open, dirty, proof-gated, and tied to unresolved trusted-identity semantics.

## What This Changes

Add a trusted sender display/principal field to inbound metadata so agents do not resolve sender names from untrusted conversational context.

## Best Possible Solution

Define a canonical trusted sender principal contract, preferably operator-controlled through identityLinks or a documented channel-directory seam, then land aligned Discord/core plumbing with tests, docs, privacy notes, and real setup proof.

## Reproduction Assessment

Yes for the implementation gap: current source has no trusted sender display/principal field in the inbound metadata payload and keeps sender identity in untrusted context. I did not reproduce the reported production misrouting incident live.

## Solution Assessment

Unclear as originally proposed: promoting raw sender_display_name into trusted metadata is weaker than an operator-controlled canonical principal. The sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it still needs refresh, proof, and maintainer security/product acceptance.
