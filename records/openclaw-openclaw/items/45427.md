## Label Justifications

- `P1`: The issue reports production DM misrouting and session contamination from an identity trust gap in agent/channel workflows.
- `impact:security`: The requested change decides which human identity facts are safe to promote into trusted model metadata.
- `impact:message-loss`: The reported failure mode is messages being routed to the wrong recipient after identity was inferred from untrusted context.
- `impact:session-state`: The discussion describes polluted agent context after the wrong identity entered the session.

## Summary

Keep open: current main and v2026.5.20 still omit a trusted sender name or canonical requester identity from the inbound metadata envelope, and the remaining fix needs maintainer schema and security-boundary approval before the related open PRs can settle it.

## What This Changes

Add a trusted platform-resolved sender name or canonical sender identity to inbound metadata so agents do not infer recipients from untrusted conversation context.

## Best Possible Solution

Choose one stable provider-id-backed trusted identity contract, likely based on `session.identityLinks`, keep mutable display names in untrusted context, and reconcile the generic and Discord-specific PRs before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.20` have no trusted canonical sender identity field and keep sender id/name in untrusted context. I did not live-reproduce the March 13 production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `requester_identity` or `sender_principal`.
