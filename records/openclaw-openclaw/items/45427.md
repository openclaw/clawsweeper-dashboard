## Summary

Keep open: current main and the latest release still lack a trusted sender identity field, and the remaining fix needs a maintainer product/security choice rather than a raw trusted display-name addition.

## What This Changes

Add trusted inbound sender identity metadata so agents do not infer recipient identity from untrusted conversational context.

## Best Possible Solution

Choose one stable provider-ID-backed identity contract, likely using `session.identityLinks`, keep mutable display names in untrusted context, and reconcile the generic and Discord-specific PRs before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.20` have no trusted sender-name or canonical requester identity field, and sender facts remain in untrusted metadata blocks. I did not live-reproduce the reported production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name is weaker than a stable-ID-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `requester_identity` or `sender_principal`.
