## Summary

Keep open. Current main still exposes platform-resolved `SenderName`/`SenderId` in `MsgContext`, but the AI-facing trusted inbound metadata intentionally omits sender identity and renders sender name/id in user-role untrusted metadata; changing that intersects identity trust and prompt-cache policy.

## What This Changes

The issue requests a trusted platform-resolved sender display name in inbound message metadata alongside the trusted sender id.

## Best Possible Solution

Define a documented per-turn sender identity contract that exposes stable `sender_id` plus a display-only platform-resolved label without making mutable names an authorization identity, while preserving prompt-cache boundaries.

## Reproduction Assessment

Yes. A focused source check reproduces the gap: build the trusted inbound metadata with `SenderName`/`SenderId` and the trusted JSON omits both, while the user-context prefix renders them under untrusted metadata labels.

## Solution Assessment

Unclear. The requested field is directionally valid, but the exact “trusted sender display name” fix needs maintainer/security judgment because current code deliberately separates stable trusted metadata from mutable human labels.
