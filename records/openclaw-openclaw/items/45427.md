## Label Justifications

- `P1`: The issue describes production DM misrouting and session contamination from an identity trust gap affecting real agent/channel workflows.
- `impact:security`: The requested change changes which human identity facts are treated as trusted prompt metadata.
- `impact:message-loss`: The reported failure mode is messages being delivered to the wrong person because the agent inferred identity from untrusted context.
- `impact:session-state`: The discussion describes cascading session-state pollution after the wrong identity entered the agent context.

## Summary

Keep open: current main and v2026.5.12 still do not expose a trusted sender display/principal field in the inbound metadata envelope, while related open PRs propose overlapping safer identity-link-backed contracts that need maintainer product and security review before this issue can close.

## What This Changes

Request a trusted inbound sender identity field so agents do not infer recipient/person identity from untrusted conversational context.

## Best Possible Solution

Choose one stable-provider-id-backed requester identity contract, likely based on `session.identityLinks`, then reconcile the generic and Discord-specific PRs with tests, docs/privacy notes, and redacted real setup proof while avoiding mutable display names as trusted metadata.

## Reproduction Assessment

Yes for the source-level implementation gap: current main has no trusted sender display/principal field and keeps sender identity in untrusted context. I did not reproduce the reported production DM misrouting incident live.

## Solution Assessment

Unclear as originally proposed: a raw trusted display name is weaker than a canonical stable-id-backed principal. The best path is a maintainer-approved identityLinks-backed contract, not multiple independent prompt field names.
