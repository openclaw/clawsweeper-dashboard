## Label Justifications

- `P1`: The issue reports real production DM misrouting and session contamination from an identity trust gap in agent/channel workflows.
- `impact:security`: The requested change decides which human identity facts are safe to promote into trusted model metadata.
- `impact:message-loss`: The failure mode is messages being routed to the wrong recipient after identity was inferred from untrusted context.
- `impact:session-state`: The discussion describes polluted agent context after the wrong identity entered the session.

## Summary

Keep open: current main and the latest release still omit a trusted canonical sender identity from inbound metadata, while the related open PRs need maintainer schema and security-boundary alignment before this can close.

## What This Changes

Request to expose trusted sender identity/name metadata at inbound message ingestion so agents do not infer recipients from untrusted conversational context.

## Best Possible Solution

Choose one stable provider-id-backed `session.identityLinks` identity contract, keep mutable display names in untrusted context, and reconcile the open generic and Discord-specific PRs before landing.

## Reproduction Assessment

Yes for the source-level gap: current main and `v2026.5.20` have no trusted canonical sender identity field and keep sender identifiers/names in untrusted context. I did not live-reproduce the reported production DM misrouting incident.

## Solution Assessment

Unclear as originally proposed: a trusted raw display name would be weaker than a stable-id-backed canonical principal. The safer path is a maintainer-approved `identityLinks`-backed contract such as `sender_principal` or `requester_identity`.
