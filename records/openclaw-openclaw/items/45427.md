## Summary

Keep open: current main and v2026.5.7 still omit a trusted sender display-name/principal from inbound metadata, while the related sender_principal implementation remains open and unmerged. This is a prompt-trust and privacy contract, so it should not be closed as cleanup or sent to a standalone repair lane yet.

## What This Changes

Request to add a trusted platform-resolved sender display name or principal to the inbound metadata envelope so agents do not infer identity from untrusted conversation context.

## Best Possible Solution

Define a canonical trusted sender principal contract, preferably derived from operator-controlled identityLinks or a clearly documented channel-directory source, then wire it through inbound metadata with focused Discord/core tests, docs, and real setup proof.

## Reproduction Assessment

Yes for the implementation gap: current main source, tests, and release contents show no trusted sender display-name/principal field. I did not reproduce the reported production DM misroute itself in a live fleet.

## Solution Assessment

Unclear as proposed: adding a raw trusted sender_display_name may not be the safest contract because names are mutable and currently untrusted. The sender_principal direction in https://github.com/openclaw/openclaw/pull/70944 looks narrower, but it needs maintainer review and proof before this issue can close.
