## Summary

This PR still targets a real Telegram UX gap on current main, but it should not merge as-is. The callback-based approach is sound and the PR head appears to address earlier review comments around session-lane queuing, emoji collision, reaction 429 backoff, and async typing composition; the remaining blockers are the missing public config-contract follow-through for the new `waiting` key and the required changelog entry.

## What This Changes

The PR adds a per-message `onQueued` callback through the command queue, embedded runner, auto-reply, and Telegram dispatch so queued Telegram messages can show a `waiting` status reaction, with controller tests and Telegram reaction 429 backoff.

## Best Possible Solution

Land the per-message callback design after adding `waiting` to the public status-reaction config contract, generated schema/help/docs, API checks if needed, focused tests, and the changelog, while preserving the session/global once-guard and per-account+chat Telegram 429 backoff already present in the PR head.

## Reproduction Assessment

Yes. Send a second Telegram message in the same session while the first run occupies the session lane; current main has no `onQueued` callback or `waiting` controller state, so the message can show thinking while still waiting. The config finding is reproducible by trying to configure `messages.statusReactions.emojis.waiting`, which is absent from the strict schema.

## Solution Assessment

No, not yet. The per-message callback is the right maintainable direction because it avoids sampling global lane state, but the PR needs config/schema/generated-doc and changelog follow-through before it is complete.
