## Summary

Keep open. Current main still lacks the waiting status reaction and per-message queued callback, and the PR’s callback design is the right direction, but the branch is not merge-ready because it still misses current public type/config/generated/changelog follow-through.

## What This Changes

The PR adds a per-message queued callback through the command queue, embedded runner, auto-reply, and Telegram dispatch so queued Telegram messages can show a waiting status reaction, with controller tests and Telegram reaction 429 backoff.

## Best Possible Solution

Land the per-message callback design after rebasing it onto current main, adding the current shared type definitions, adding waiting to the public status-reaction config/schema/generated/help/docs surfaces, preserving the session/global once-guard and per-account+chat 429 backoff, and adding the required changelog entry.

## Reproduction Assessment

Yes. Send a second Telegram message in the same session while the first run occupies the session lane; current main has no onQueued callback or waiting controller state, so the queued message can show thinking while it is still waiting. The review findings also reproduce by static inspection: waiting is absent from the strict config schema/types/generated help, the current shared option types do not expose onQueued, and the PR diff lacks a changelog entry.

## Solution Assessment

No, not yet. The per-message callback is the narrow maintainable design because it avoids sampling global lane state or leaking other users’ activity, but this PR needs current-main type/config/generated-doc/changelog follow-through before merge.
