## Summary

Keep open: current main still lacks the queued-message waiting reaction path, and this PR remains a plausible focused Telegram UX feature, but it is not merge-ready without conflict resolution and real Telegram proof.

## What This Changes

The branch adds a Telegram waiting status reaction by threading per-message `onQueued` callbacks through the command queue, agent runner, auto-reply options, and Telegram status reaction controller, plus focused tests and 429 backoff.

## Best Possible Solution

Rebase onto current main, preserve the current lane-timeout and Telegram status-reaction plumbing, then land only after a live Telegram run shows queued messages moving from acknowledged to waiting to thinking.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main sets Telegram thinking eagerly and has no per-message queued callback, which matches the reported UX gap, but no real Telegram run was inspected.

## Solution Assessment

Yes, the per-message `onQueued` callback is the right shape because it avoids sampling global queue state and preserves user isolation. The branch still needs a rebase and live Telegram proof before it is a mergeable solution.
