## Summary

Keep open. Current main still lacks Telegram queued-waiting status reactions and the PR contains useful targeted work, but it is not merge-ready because the new runtime `waiting` emoji key is missing from the public config/schema/help/generated surfaces and the branch needs a current-main rebase.

## What This Changes

The PR threads a per-message `onQueued` callback through the command queue, embedded runner, auto-reply, and Telegram dispatch to show a waiting status reaction, adds waiting status-reaction controller and Telegram emoji variants/tests, and adds Telegram reaction 429 backoff.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires queued feedback from both session and global lanes with a once guard, preserves the account+chat 429 backoff and current embedded lane timeout behavior, and updates the config type, strict Zod schema, generated schema/help, docs, and focused tests for the new `waiting` status key.

## Reproduction Assessment

Yes. Send a second Telegram message in the same session while a first run occupies the session lane; current main sets `thinking` before execution and has no `onQueued`/`waiting` path. For the review finding, a `messages.statusReactions.emojis.waiting` override is not accepted by the current strict config schema because that key is absent.

## Solution Assessment

No, not yet. The callback-based approach is the narrow maintainable direction and the PR head addressed the earlier session-lane and 429-backoff comments, but the public config contract/generation gap and stale branch state need to be fixed before merge.
