## Summary

Keep open: the PR targets the right QQ Bot media bug and adds focused test coverage, but the diff also moves persisted QQ Bot data for custom-home users without migration/fallback, does not honor `OPENCLAW_STATE_DIR`, and lacks real behavior proof.

## What This Changes

The PR adds a QQ Bot helper that resolves data/media roots from `OPENCLAW_HOME` and a regression test for accepting media under `$OPENCLAW_HOME/.openclaw/media/qqbot`.

## Best Possible Solution

Keep the media-root fix, but derive QQ Bot roots from the full state-dir contract and either migrate/fallback existing HOME-based QQ Bot data for custom-home upgrades or leave data paths compatible while moving only media roots.

## Reproduction Assessment

Yes, from source: current main validates QQ Bot media under `HOME/.openclaw/media` while core state/media helpers use the configured OpenClaw home. I did not run a live QQ Bot send in this read-only review.

## Solution Assessment

No. The media fix belongs in the QQ Bot path helper, but the proposed resolver should match full state-dir precedence and preserve upgraded data before merge.

Include `OPENCLAW_STATE_DIR` handling in the local resolver or inject the resolved state dir through the plugin boundary.
