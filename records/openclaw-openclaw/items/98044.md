## Summary

Keep open: current main and v2026.6.10 still route Android setup QR scanning through a generic scanner failure while camera permission is declared, tracked, and reported elsewhere, so the reported denied-permission setup confusion remains unresolved.

## What This Changes

The issue requests Android QR setup to request camera permission when needed, explain denied camera permission, and offer an app-settings action when Android will not show the runtime prompt.

## Best Possible Solution

Add narrow Android onboarding handling that distinguishes scanner startup failure from OpenClaw camera-permission denial, uses the existing permission/settings affordances when camera is genuinely required, and preserves the Google Code Scanner path.

## Reproduction Assessment

Yes at source level: current main still starts the Android QR scanner without a permission-specific branch and only shows a generic scanner error, while camera permission is declared and modeled elsewhere. No live Android device reproduction was run in this read-only review.

## Solution Assessment

Yes with a dependency caveat: the narrow fix belongs at Android onboarding's scan/error UI boundary, but it should not blindly require app CAMERA for Google Code Scanner because that API is documented as permissionless.
