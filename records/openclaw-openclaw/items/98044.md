## Summary

Keep open: current main and v2026.6.10 still start Android QR scanning without first checking or requesting camera permission, and the only scan failure shown is a generic scanner error even though the app already declares and tracks camera permission elsewhere.

## What This Changes

Request Android onboarding QR setup to request camera permission, explain denial, and offer an app-settings action when Android will not prompt again.

## Best Possible Solution

Handle camera permission at the Android QR scan entry point, show a specific inline denied/permanently-denied state with an app-settings action, and leave broader gateway/auth recovery messaging to the separate Android setup issue.

## Reproduction Assessment

No live repro was run in this read-only review, but the source path is high-confidence: tapping scan calls startScan directly and current release/main only render a generic scanner failure while camera permission is declared and modeled elsewhere.

## Solution Assessment

Yes, the request is the right narrow fix: the scan entry point should own the camera permission prompt and denied-state copy instead of sending users into gateway/auth recovery.
