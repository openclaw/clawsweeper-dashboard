## Summary

Keep open: current main and the latest release still start Android onboarding QR scanning without first checking/requesting camera permission, while existing Android permission infrastructure already supports camera prompts and app-settings fallback elsewhere.

## What This Changes

Request that Android QR setup prompt for camera permission, explain denied/permanently denied camera state, and offer Android app settings when needed.

## Best Possible Solution

Handle camera permission at the Android QR scan entry point, reuse the existing permission/settings fallback seam, and keep gateway/auth recovery messaging separate from local permission failures.

## Reproduction Assessment

No live-device reproduction was run, but source inspection provides a high-confidence reproduction path: tapping Scan setup code invokes `qrScanner.startScan()` without checking `Manifest.permission.CAMERA`, and failures show only generic scanner copy.

## Solution Assessment

Yes. The narrow fix is to put the permission prompt, denied copy, and settings action at the QR scan entry point while reusing the existing Android permission requester/settings patterns.
