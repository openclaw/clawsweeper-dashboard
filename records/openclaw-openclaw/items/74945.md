## Summary

Keep this PR open, but it still needs changes before merge: current main does not implement the linked model-ref canonicalization bug, and this PR fixes the right surface, but the latest diff adds a full default catalog load to normal model-run dispatch.

## What This Changes

The PR canonicalizes CLI `infer model run --model` refs against the model catalog for local and gateway dispatch, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep catalog-backed case repair, preserve mixed-case canonical IDs and auth-profile suffixes, and land one canonical #73715 fix that does not add broad side-effectful catalog discovery to ordinary model probes.

## Reproduction Assessment

Yes. The linked issue gives concrete CLI steps, and current-main source shows the raw `--model` value still reaches both local preparation and gateway override parsing.

## Solution Assessment

No, not as-is. Catalog-aware canonicalization is a reasonable fix direction, but the patch should use a gated, read-only, or narrower lookup so canonical explicit model runs keep the shipped lean dispatch path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Avoid loading the full catalog for every model run:** `src/cli/capability-cli.ts:575`
  - body: `canonicalizeCliModelRef` calls default `loadModelCatalog()` for every non-empty `<provider/model>` ref before it knows whether case repair is needed. Gate catalog lookup to refs needing case repair, or use a read-only/narrow lookup so canonical explicit runs stay lean.
