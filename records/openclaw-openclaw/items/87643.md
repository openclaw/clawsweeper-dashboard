## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass session auth profiles to title generation:** `src/auto-reply/reply/session-title-generator.ts:119-128`
  - body: The title generator forwards only provider/model into `generateConversationLabel`, but current session model changes can persist `authProfileOverride` alongside those fields. A session pinned to a specific profile can therefore summarize with default credentials or fail even though the main reply used the selected profile; carry the session auth profile through this path or skip generation when one is active.
  - confidence: 0.9
- **[P1] Do not enable background summaries by default:** `src/auto-reply/reply/session-title-generator.ts:31-32`
  - body: Because this returns only when `enabled === false`, every existing config with no `sessionTitle` key starts making extra background model calls and writing `autoTitle` after the threshold. Make generation explicit opt-in, or get maintainer approval plus upgrade/privacy docs and tests for the default-on behavior.
  - confidence: 0.88
- **[P2] Keep context titles ahead of generated titles:** `src/gateway/session-utils.ts:228-230`
  - body: Current main prefers `subject` before falling back to transcript-derived titles, but this insertion returns `autoTitle` first. For group/channel sessions that already have a subject, generating a title can hide the established context label; keep `subject` ahead of generated metadata and test that precedence.
  - confidence: 0.84
- **[P2] Preserve display names in the UI helper:** `ui/src/ui/session-display.ts:100-104`
  - body: The helper now returns `derivedTitle` before `displayName`, so any row carrying both a distinct derived title and a manual display name will hide the user rename in the Control UI. Keep `displayName` ahead of `derivedTitle` here as well as in the server-derived title helper.
  - confidence: 0.79
