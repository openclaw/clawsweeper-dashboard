## Summary

Keep open: this is a fresh, specific Discord message-loss report against the current durable-send path, and current main/release source still lacks terminal proof that every planned text chunk was actually delivered before reporting success. The linked regression-test PR is open and test-only, so it is useful context but not an implemented fix yet.

## What This Changes

The issue asks OpenClaw to prevent Discord durable multi-chunk replies from silently dropping later chunks after transient delivery errors.

## Best Possible Solution

Land or extend focused durable Discord regression coverage, then make durable delivery reconcile intended text chunks against delivered message identities and surface partial_failed when they diverge, with live Discord validation if feasible.

## Reproduction Assessment

No high-confidence current-main reproduction is available from this review. The report gives concrete v2026.5.12 live steps and current main still matches the implicated path, but I did not run a live Discord scenario or prove the exact drop point from source alone.

## Solution Assessment

Unclear. Per-chunk retry and the open test PR cover one adjacent path, but the maintainable fix should prove intended chunk count versus acknowledged delivery and report partial_failed for any missing chunk.
