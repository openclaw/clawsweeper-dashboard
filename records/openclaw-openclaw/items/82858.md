Keep open: the report is fresh, specific, and describes a plausible Discord message-loss regression; current source shows intended durable chunk fan-out and retry behavior, but there is not enough live or regression-test proof to close it as fixed or unreproducible.

## What This Changes

The issue reports that Discord long replies split into multiple chunks deliver only the first chunk after the v2026.5.12 durable-send migration.

## Best Possible Solution

Add a focused regression that drives a durable Discord final reply through two planned text chunks, injects a transient failure on a later chunk, and proves every chunk is either delivered after retry or reported as partial_failed with visible diagnostics; then validate with a real Discord scenario if possible.

## Reproduction Assessment

No. The report gives concrete v2026.5.12 live steps, but I did not reproduce against current main or a live Discord account, and current source inspection shows the intended fan-out path should send each chunk.

## Solution Assessment

Unclear. The reporter’s per-chunk retry workaround may reduce the symptom, but the maintainable fix should first identify whether the drop is in durable fan-out, Discord adapter retry/result accounting, preview finalization fallback, or delivery queue handling.
