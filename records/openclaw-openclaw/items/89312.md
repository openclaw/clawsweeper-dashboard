## Summary

Keep open: current main and the latest release still have the direct client-tool descriptor reads, but this protected draft is not the best landing shape as submitted because it rereads descriptors across reservation, conflict checks, Tool Search planning, and registration while the newer overlapping PR uses a stronger snapshot-once boundary.

## What This Changes

The PR adds guarded client-tool descriptor reads and regressions so malformed client `function`, name, description, parameters, or tag accessors are skipped or degraded while healthy client tools remain usable.

## Best Possible Solution

Land one reconciled descriptor-snapshot fix that snapshots client tool descriptors once per attempt and reuses those snapshots for reserved-name checks, Tool Search planning, conflict checks, and registration.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.6 shows direct `tool.function` and `tool.function.name` reads in conflict checks, runner reservation, allowlist planning, and registration, and the PR body records focused red/green tests for throwing getters.

## Solution Assessment

No. The guard is useful, but the best fix is the snapshot-once shape from the related draft or an equivalent refactor, because this branch still lets stateful descriptors disagree between admission and registration.
