## Summary

Keep open: this maintainer-labeled draft should not be cleanup-closed, and the branch is not merge-ready because it guards descriptor reads but still rereads the same client descriptors at later admission and registration points. The related open draft at https://github.com/openclaw/openclaw/pull/89699 is relevant because it threads a single descriptor snapshot through those consumers.

## What This Changes

The branch adds guarded client tool descriptor reads and applies collected client tool names across conflict checks, Tool Search planning, bundle MCP/LSP reservation, allowlist collection, conversion, and regressions.

## Best Possible Solution

Use one immutable client-tool descriptor snapshot per attempt and pass it through reservation, Tool Search allowlists, conflict checks, and registration, then choose or rebase against the related draft at https://github.com/openclaw/openclaw/pull/89699.

## Reproduction Assessment

Yes. Source inspection of current main shows direct `tool.function` and `tool.function.name` reads in conflict checks, bundle reservation, allowlist planning, and conversion, so throwing getters can abort those paths without a new product mode.

## Solution Assessment

No. Guarded descriptor reads belong in this owner boundary, but this branch snapshots only names and then rereads the descriptors later; the safer solution is one descriptor snapshot reused across all attempt consumers.
