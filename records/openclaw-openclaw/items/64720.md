## Summary

Keep open. The security goal is valid and current main still lacks generic external-content wrapping for channel body/thread text, but this branch is not merge-ready because it derives trust from route/provider strings that misclassify heartbeat and cron prompts, is currently conflicting with main, and has only unit/CI proof.

## What This Changes

The PR adds `wrapExternalContent`-based wrapping for channel message bodies and thread context, a `channel` external-content source label, and inbound text unit coverage.

## Best Possible Solution

Define an owner-approved ingress trust boundary, wrap only external channel body/thread text with the existing nonce-backed helper, preserve internal scheduler/system-event providers, and prove both live channel/prompt assembly behavior and scheduler upgrade behavior.

## Reproduction Assessment

Yes, by source inspection. Current main still sends channel body/thread text into prompt assembly without `wrapExternalContent`, and the PR shows concrete heartbeat and cron-event misclassification paths through `OriginatingChannel` and the provider allowlist.

## Solution Assessment

No as written. Wrapping external channel content is the right direction, but the fix should use explicit ingress trust rather than delivery-route metadata and must preserve internal scheduler/system-event prompts.
