## Summary

Keep open. The PR targets a real untrusted channel-body hardening gap, but the current diff is not mergeable because it classifies trust from routing metadata, misses existing scheduler providers, and lacks required real behavior proof.

## What This Changes

The PR adds `wrapExternalContent` wrapping for channel message bodies and thread context in auto-reply prompt assembly, adds inbound-text unit tests, and adds a `channel` external-content source label.

## Best Possible Solution

Classify wrapping from explicit ingress/source trust, keep nonce-backed wrapping for real external channel body and context text, preserve raw summaries and command parsing, and cover scheduler plus channel cases in tests while leaving memory provenance to the canonical issue.

## Reproduction Assessment

Yes. Static source inspection establishes both the current-main hardening gap and the PR regression: current main does not wrap live channel body/thread text, while the PR chooses `OriginatingChannel` before `Provider` even though scheduler contexts use `Provider` for the internal source class.

## Solution Assessment

No. The wrapper direction is right, but route metadata plus an incomplete trusted-provider allowlist is not the safest trust boundary; the fix should use explicit ingress/source classification and cover heartbeat, cron-event, exec-event, and real channel cases.
