## Summary

Keep open. The security hardening direction is valid, but this PR still misclassifies internal heartbeat/cron prompts because it derives wrapping from reply-routing metadata and an incomplete trusted-provider allowlist; it also lacks real after-fix behavior proof.

## What This Changes

The PR wraps channel message bodies and thread context with external-content boundaries, adds a channel source label, and adds inbound text unit tests.

## Best Possible Solution

Revise this PR to derive wrapping from explicit ingress/source trust metadata, keep nonce-backed wrapping for real external channel body and thread text, preserve raw summaries, and leave memory provenance to the linked broader issue.

## Reproduction Assessment

Yes. Static inspection is enough: current main sends channel bodies as plain prompt text, and the PR head wraps any OriginatingChannel or unknown provider, including heartbeat/cron route contexts.

## Solution Assessment

No. The wrapper direction is right, but route metadata plus a hard-coded negative allowlist is not the safest trust boundary; wrapping should be driven by explicit ingress/source classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-271`
  - body: `OriginatingChannel` is reply-routing metadata, not ingress trust metadata. Heartbeat and cron turns can carry an external delivery channel while the prompt source remains internal, so this wraps trusted scheduler prompts as untrusted channel text and can degrade scheduled actions.
  - confidence: 0.9
- **[P1] Trust heartbeat and cron-event providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` treats every provider except `node`, `cli`, and `exec-event` as external, but current main defines `heartbeat` and `cron-event` as system-event providers. Internal scheduler turns with those providers now get fenced even when there is no inbound channel message.
  - confidence: 0.88
