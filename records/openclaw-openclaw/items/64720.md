## Summary

Keep open. Current main still has the channel body/thread-text protection gap, but this PR is not safe to merge as written because it derives trust from route metadata and omits existing internal scheduler providers from the trusted set.

## What This Changes

The PR adds external-content wrapping for channel message bodies and thread context text in auto-reply prompt assembly, adds a channel source label, and adds inbound text unit tests.

## Best Possible Solution

Land a revised security-boundary implementation that wraps genuinely external channel body and thread text using explicit ingress/source trust metadata, preserves trusted scheduler/system-event prompts, and leaves durable memory provenance to the canonical design issue.

## Reproduction Assessment

Yes. Source inspection on current main shows channel body and thread context are sanitized but not structurally wrapped, and source inspection of the PR diff shows trusted scheduler/provider regressions.

## Solution Assessment

No. Nonce-backed external-content wrapping is the right direction, but this patch should derive wrapping from explicit ingress/source trust metadata instead of `OriginatingChannel` and a local negative allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:267-268`
  - body: `OriginatingChannel` is reply-route metadata, not proof that the prompt text came from an external user. Heartbeat and cron runs can set `Provider` to `heartbeat` or `cron-event` while carrying an external `OriginatingChannel` for delivery, so this branch wraps trusted scheduler prompts as attacker-supplied channel content and can change scheduled automation behavior.
  - confidence: 0.91
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` marks every provider except `node`, `cli`, and `exec-event` as external. Current scheduler paths use `heartbeat` and `cron-event` as internal system-event providers, so internal automation prompts without an originating channel would now be fenced as untrusted external content.
  - confidence: 0.88
