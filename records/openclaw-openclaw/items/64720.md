## Summary

Keep open. The PR is a useful security-hardening direction, but the latest head still has blocking trust-classification regressions and the contributor has only unit/CI proof, not after-fix real behavior proof.

## What This Changes

The PR wraps channel message bodies and thread context with `wrapExternalContent`, adds a `channel` external-content source label, and adds inbound-text unit tests.

## Best Possible Solution

Revise the PR to derive wrapping from explicit ingress/source trust metadata, keep nonce-backed wrapping for real external channel body and thread text, and leave durable memory provenance to the linked broader issue.

## Reproduction Assessment

Yes. Source inspection on current main shows channel body and thread context enter prompt assembly without `wrapExternalContent`; the PR's remaining regression is also source-reproducible from the provider/route fields.

## Solution Assessment

No. Nonce-backed wrapping is the right direction, but this patch derives trust from `OriginatingChannel` and an incomplete trusted-provider set; explicit ingress/source trust metadata is the safer boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-271`
  - body: At PR head, `OriginatingChannel` is used before `Provider` to decide whether text is external. That field is delivery-route metadata, so heartbeat or cron turns routed to an external channel can have trusted scheduler prompts fenced as attacker-supplied content.
  - confidence: 0.9
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: The new allowlist omits `heartbeat` and `cron-event`, even though current main treats both as system-event providers. Internal scheduler turns without an originating channel would now be wrapped as untrusted external content.
  - confidence: 0.88
