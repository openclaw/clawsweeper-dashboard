## Summary

Keep open. Current main still has the channel-body/thread-context trust-boundary gap, so the PR is addressing real security work, but the current branch still has trust-classification blockers and lacks contributor real behavior proof.

## What This Changes

The PR adds external-content wrapping for channel message bodies and thread context in auto-reply prompt assembly, adds a `channel` external-content source label, and adds inbound-text unit tests.

## Best Possible Solution

Land a revised security-boundary implementation that wraps genuinely external channel body and thread text using explicit ingress/source trust metadata, preserves trusted scheduler/system-event prompts, and leaves broader memory provenance to https://github.com/openclaw/openclaw/issues/69401.

## Reproduction Assessment

Yes. Source inspection on current main shows channel body and thread context are not structurally wrapped, and source inspection of the PR head shows the remaining regressions from route-first provider selection and the incomplete trusted-provider allowlist.

## Solution Assessment

No. Nonce-backed external-content wrapping is the right direction, but this patch should derive wrapping from explicit ingress/source trust metadata instead of `OriginatingChannel` and a local negative allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-271`
  - body: `OriginatingChannel` is reply-route metadata, not proof that the prompt text came from an external user. Heartbeat or cron runs routed to Slack/Telegram can have `Provider` set to `heartbeat`/`cron-event` while `OriginatingChannel` names the delivery channel, so this wraps trusted scheduler prompts as untrusted external content and can change scheduled automation behavior.
  - confidence: 0.91
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: The trusted-provider set omits `heartbeat` and `cron-event`, which current main treats as internal system-event providers. Internal scheduler runs without an originating channel would now be fenced as untrusted external content, changing trusted automation prompts even though no channel user supplied that body.
  - confidence: 0.88
