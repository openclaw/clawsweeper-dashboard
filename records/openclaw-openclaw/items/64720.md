## Summary

Keep this PR open. Current main still leaves channel message bodies and thread context without external-content wrapping, so the security hardening is addressing a real gap, but the current branch is not safe to merge because it derives trust from reply-routing metadata and an incomplete trusted-provider allowlist, which can misclassify heartbeat and cron scheduler prompts as untrusted channel content.

## What This Changes

The PR adds channel external-content wrapping helpers, applies them to auto-reply base and thread-context bodies, adds inbound-text unit tests, and adds a channel source label to external-content metadata.

## Best Possible Solution

Add an explicit ingress trust/source classification for prompt wrapping, then wrap only true external channel body and thread-context text while preserving trusted heartbeat, cron-event, exec-event, node, and CLI turns. Keep the channel source label and raw queue summaries, and add regression coverage for both external channel input and internal scheduled turns.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a heartbeat or cron-event turn can carry OriginatingChannel for delivery while Provider remains heartbeat or cron-event, and the PR currently prefers OriginatingChannel for wrapping; internal-only heartbeat and cron-event turns are also treated as external because those providers are absent from the trusted set.

## Solution Assessment

No. The direction is right, but the current implementation is not the best fix because it treats routing metadata as ingress trust metadata; the safer solution is an explicit trust/source seam or provider-based classification that does not reclassify scheduler prompts.
