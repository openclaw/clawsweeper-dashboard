## Summary

Keep open, but do not merge as-is. The hardening direction is useful and matches the remaining security problem tracked in https://github.com/openclaw/openclaw/issues/69401, but the diff still misclassifies trusted scheduler prompts as external channel text and lacks after-fix real behavior proof.

## What This Changes

The PR adds `wrapExternalContent`-based wrapping for channel message bodies and thread context in auto-reply prompt assembly, plus inbound-text unit coverage and a `channel` external-content source label.

## Best Possible Solution

Revise this PR to classify wrapping from explicit ingress/source trust, keep nonce-backed wrapping for real external channel body and context text, preserve raw summaries and command parsing, and cover heartbeat/cron/exec plus channel cases in tests while leaving broader memory provenance to the linked canonical issue if needed.

## Reproduction Assessment

Yes. Static source inspection is enough: current heartbeat/cron/exec contexts can carry an external `OriginatingChannel` for delivery while `Provider` remains the trusted ingress source, and the PR chooses `OriginatingChannel` first.

## Solution Assessment

No. The wrapping direction is right, but route metadata plus an incomplete trusted-provider allowlist is not the safest trust boundary; use an explicit ingress/source classification that reuses current system-event provider knowledge.
