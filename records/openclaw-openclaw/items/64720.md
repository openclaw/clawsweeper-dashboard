## Summary

Keep open. Current main still lacks the requested structural wrapping for channel-originated body/thread text, but the PR head is not mergeable as-is because it derives trust from reply-route metadata, omits trusted scheduler providers, and lacks real behavior proof for this security boundary.

## What This Changes

The PR wraps channel message body and thread context text with `wrapExternalContent`, adds a `channel` external-content source label, and adds inbound-text unit tests.

## Best Possible Solution

Land a revised security-boundary implementation that wraps genuinely external channel body and thread text using explicit ingress/source trust metadata, preserves scheduler/system-event prompts, and coordinates durable memory provenance through the related open issue.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main passes channel body/thread context into prompt assembly without structural external-content wrapping, and the PR head shows trust-classification regressions from route-first provider selection and an incomplete trusted-provider allowlist.

## Solution Assessment

No. Nonce-backed external-content wrapping is the right primitive, but this patch should derive wrapping from explicit ingress/source trust metadata or the existing system-event provider contract instead of `OriginatingChannel` plus a local negative allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-274`
  - body: `OriginatingChannel` is reply-route metadata, not proof that the prompt text came from an external user. Heartbeat and cron runs can carry a channel route while `Provider` is `heartbeat` or `cron-event`, so this wraps trusted scheduler prompts as attacker content and can distort scheduled automation behavior.
  - confidence: 0.91
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` marks every provider except `node`, `cli`, and `exec-event` as external. Current internal scheduler paths use `heartbeat` and `cron-event`, so those trusted automation prompts can now be fenced as untrusted channel content.
  - confidence: 0.88
