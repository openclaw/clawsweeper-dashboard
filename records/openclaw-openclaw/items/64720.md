## Summary

Keep open. The security goal is valid, but the PR currently infers external-channel trust from routing/provider strings, which can reclassify trusted heartbeat and cron-event prompts as attacker-supplied content; it also lacks required real behavior proof and is currently conflicting with main.

## What This Changes

The PR wraps channel message bodies and thread context with `wrapExternalContent`, adds a `channel` external-content source label, and adds inbound text unit tests.

## Best Possible Solution

Define an owner-approved ingress trust boundary, wrap only external channel body/thread text with the existing nonce-backed helper, preserve internal scheduler/system-event providers, and prove both live channel/prompt assembly behavior and scheduler upgrade behavior.

## Reproduction Assessment

Yes, by source inspection. Current main lacks structural external-content wrapping for channel body/thread text, and the PR diff shows concrete heartbeat and cron-event misclassification paths through `OriginatingChannel` and the provider allowlist.

## Solution Assessment

No as written. Wrapping external channel content is the right direction, but the fix should use explicit ingress trust rather than delivery-route metadata and must preserve internal scheduler/system-event prompts.
