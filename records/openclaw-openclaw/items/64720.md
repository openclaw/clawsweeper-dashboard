## Summary

Keep open. The PR addresses a real prompt-boundary gap, but the current head still has blocking trust-classification regressions for scheduler/system-event turns and only provides unit-test proof, not after-fix real behavior proof.

## What This Changes

The PR wraps channel message bodies and thread context with `wrapExternalContent`, adds a `channel` external-content source label, and adds inbound text unit tests.

## Best Possible Solution

Revise this PR to derive wrapping from explicit ingress/source trust metadata, keep nonce-backed wrapping for real external channel body and thread text, preserve raw queue summaries, and leave durable memory provenance to the linked broader issue.

## Reproduction Assessment

Yes. Source inspection is enough: current main sends channel body and thread context as plain prompt text, while the PR head wraps based on route/provider strings that also occur on internal heartbeat and cron-event paths.

## Solution Assessment

No. The wrapper direction is right, but using `OriginatingChannel` and a negative provider allowlist is not the safest trust boundary; the code should use explicit ingress/source classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-271`
  - body: `OriginatingChannel` is reply-routing metadata, not ingress trust metadata. Heartbeat and cron turns can carry an external delivery route while the prompt source remains internal, so this wraps trusted scheduler prompts as untrusted channel text and can degrade scheduled actions.
  - confidence: 0.9
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` treats every provider except `node`, `cli`, and `exec-event` as external, but current main defines `heartbeat` and `cron-event` as system-event providers. Internal scheduler turns without an originating channel now get fenced even when there is no inbound channel message.
  - confidence: 0.88
