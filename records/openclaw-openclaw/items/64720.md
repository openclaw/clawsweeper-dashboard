## Summary

Keep open: the hardening goal is valid on current main, but the patch has source-classification blockers and lacks required real behavior proof, so it is not merge-ready.

## What This Changes

The PR adds external-content boundary wrapping for channel message bodies and thread context in auto-reply prompt assembly, plus inbound-text unit coverage and a `channel` external-content source label.

## Best Possible Solution

Use explicit ingress/source trust classification, wrap only true external channel body and thread text with existing nonce-backed boundaries, preserve raw summaries and command parsing, and leave memory provenance to https://github.com/openclaw/openclaw/issues/69401.

## Reproduction Assessment

Yes. Static source inspection is enough: current heartbeat, cron, and exec contexts can carry an external `OriginatingChannel` for delivery while `Provider` remains the internal source, and the PR chooses `OriginatingChannel` first.

## Solution Assessment

No. The wrapping direction is sound, but the trust boundary should be derived from ingress/source provider rather than route metadata and should reuse current system-event provider classification.
