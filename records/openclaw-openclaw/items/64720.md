## Summary

Keep open. Current main still leaves channel body and thread context text structurally unframed, but this PR would classify trusted heartbeat/cron prompts as external channel content and still lacks after-fix real behavior proof.

## What This Changes

The PR adds external-content wrapping helpers/tests, wraps channel body and thread context text during auto-reply prompt assembly, and adds a channel source label to external-content metadata.

## Best Possible Solution

Land a revised security-boundary implementation that wraps genuinely external channel body/thread text using explicit ingress trust metadata, preserves heartbeat/cron/system-event prompts, and coordinates memory provenance through https://github.com/openclaw/openclaw/issues/69401.

## Reproduction Assessment

Yes. Source inspection on current main shows channel body and thread context are sanitized but not structurally wrapped, and the PR diff clearly shows the scheduler/provider misclassification path.

## Solution Assessment

No. Nonce-backed external-content wrapping is the right direction, but this patch should derive trust from explicit ingress/source metadata instead of OriginatingChannel and a local negative allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:267-268`
  - body: OriginatingChannel is reply-route metadata, not proof that the prompt text came from an external user. Heartbeat and cron runs can set Provider to heartbeat or cron-event while carrying an external OriginatingChannel for delivery, so this branch wraps trusted scheduler prompts as attacker content and can change scheduled automation behavior.
  - confidence: 0.91
- **[P1] Trust scheduler event providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: isExternalChannelProvider marks every provider except node, cli, and exec-event as external. Current scheduler paths use heartbeat and cron-event as internal system-event providers, so internal automation prompts without an originating channel would now be fenced as untrusted external content.
  - confidence: 0.88
