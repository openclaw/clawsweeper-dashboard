## Summary

Keep open: current main still has a real channel-body prompt-injection hardening gap, but this PR is not ready to merge because the latest head derives trust from routing metadata, misses internal scheduler providers, is dirty against the base branch, and lacks required real behavior proof.

## What This Changes

The PR adds channel message body and thread-context wrapping with `wrapExternalContent`, a `channel` external-content source label, and unit tests for inbound text helpers.

## Best Possible Solution

Use an explicit ingress/source trust classification, wrap real external channel body and context text with the nonce-backed external-content wrapper, preserve raw summaries and command parsing, and leave memory provenance to the canonical follow-up unless this PR is expanded deliberately.

## Reproduction Assessment

Yes. Static source inspection is enough: scheduler contexts can carry `OriginatingChannel` for delivery while `Provider` remains `heartbeat`, `cron-event`, or `exec-event`, and the PR chooses `OriginatingChannel` first.

## Solution Assessment

No. The hardening direction is right, but the trust boundary should come from explicit ingress/source classification rather than route metadata, with tests covering channel and scheduler cases.
