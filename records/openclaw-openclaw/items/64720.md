## Summary

Keep open: current main still leaves channel body and thread-context prompt text unwrapped, but this PR uses routing/provider strings as the trust boundary and lacks real after-fix channel proof, so it needs revision before merge.

## What This Changes

The PR wraps channel message bodies and thread context with `wrapExternalContent`, adds a `channel` external-content source label, and adds inbound text unit tests.

## Best Possible Solution

Revise the PR to derive untrusted wrapping from explicit ingress/source trust metadata, keep nonce-backed wrapping for real external channel body and thread text, and leave durable memory provenance to the linked broader issue.

## Reproduction Assessment

Yes. Source inspection on current main shows channel `BodyForAgent` is only normalized/sanitized, `runPreparedReply` passes body and thread context into prompt assembly without external-content wrapping, and current tests show system-event prompts can carry route metadata.

## Solution Assessment

No. Nonce-backed wrapping is the right direction, but this PR derives trust from `OriginatingChannel` and an incomplete trusted-provider set; explicit ingress trust metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from ingress trust:** `src/auto-reply/reply/get-reply-run.ts:270-271`
  - body: `OriginatingChannel` is reply-routing metadata, not ingress trust metadata. Heartbeat and cron turns can carry an external delivery route while the prompt source remains internal, so this wraps trusted scheduler prompts as untrusted channel text and can degrade scheduled actions.
  - confidence: 0.9
- **[P1] Trust scheduler providers:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` treats every provider except `node`, `cli`, and `exec-event` as external, but current main defines `heartbeat` and `cron-event` as system-event providers. Internal scheduler turns without an originating channel now get fenced even when there is no inbound channel message.
  - confidence: 0.88
