## Summary

Keep open: current main still has inconsistent stale `plugin.approval.waitDecision` handling across the reported approval surfaces, while the existing exec-approval path already has the fail-closed/null-decision pattern this issue is asking to extend.

## What This Changes

Classify stale or missing `plugin.approval.waitDecision` ids as a typed unavailable/declined outcome across Codex app-server, before_tool_call, and native hook relay approval waits.

## Best Possible Solution

Reuse or extend the shared approval-not-found classifier, map known stale plugin wait errors to the existing fail-closed unavailable/declined outcomes in the three reported callers, and preserve throwing for unrelated waitDecision failures.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: mock `plugin.approval.waitDecision` to reject with `approval expired or not found` or `unknown or expired approval id` in the three caller paths and current main will not consistently produce the typed unavailable/declined outcome. I did not run a live turn in this read-only review.

## Solution Assessment

Yes, the suggested direction is the narrowest maintainable fix: centralize stale approval-id classification and return existing fail-closed approval outcomes while leaving unrelated errors exceptional.
