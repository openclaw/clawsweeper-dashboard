## Summary

Keep open: the PR is aimed at a real, source-reproducible Gateway reset bug, but the patch misses the already-stale upgrade state reported by #77770 and the new live proof actually shows the stale filename still active after reset.

## What This Changes

The branch extracts generated session-file rewrite logic into shared session helpers, applies it during Gateway session reset for direct store transcript paths, and adds reset/helper regression tests.

## Best Possible Solution

Fix reset path selection for direct-store OpenClaw-generated filenames even when the filename's embedded id differs from the current stored sessionId, keep explicit custom paths stable, add the stale-entry regression test, attach after-fix live proof, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main is source-reproducible because Gateway reset passes the old stored sessionFile into a resolver that prefers it, and the linked issue plus PR proof show the stale filename/header-id mismatch.

## Solution Assessment

No. The direction is narrow, but the implementation only handles generated filenames that match the current stored sessionId; it should also detect already-stale generated direct-store filenames and rotate those to the new session id.
