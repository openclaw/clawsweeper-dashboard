## Summary

Keep open: this is an active, focused contributor PR with sufficient live proof for the reported MiMo v2.5-pro failure, but the proposed model matchers miss a documented affected model, `mimo-v2-flash`, so the patch should not merge as-is.

## What This Changes

Adds Xiaomi MiMo DeepSeek-style `reasoning_content` replay handling through endpoint attribution, OpenAI-completions compat defaults, Xiaomi plugin thinking/replay hooks, a custom-provider fallback wrapper, and regression tests.

## Best Possible Solution

Land the Xiaomi compatibility fix after covering all documented MiMo tool-call thinking models, including `mimo-v2-flash`, while preserving Flash's upstream default-disabled thinking behavior.

## Reproduction Assessment

Yes. The linked issue and PR body provide concrete pre-fix 400 logs for MiMo v2.5-pro multi-turn tool calls, and source inspection shows current main lacks the Xiaomi wrapper/fallback that would add `reasoning_content`.

## Solution Assessment

No, not yet. The fix direction matches the existing DeepSeek-style architecture, but the current matcher should cover the documented `mimo-v2-flash` thinking-mode case before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cover MiMo V2 Flash in the replay matcher:** `extensions/xiaomi/thinking.ts:3-8`
  - body: Xiaomi's official docs and the related live-tested PR https://github.com/openclaw/openclaw/pull/81573 both list `mimo-v2-flash` as affected when thinking is enabled, but this new set only matches Pro/Omni/v2.5 IDs. Because the proxy fallback mirrors the same omission, `mimo-v2-flash` still will not get the DeepSeek-style `thinking` payload or assistant `reasoning_content`, so multi-turn tool-call requests can still hit the documented 400. Add Flash to the Xiaomi/proxy matchers and cover it in tests while preserving its default-disabled thinking policy.
  - confidence: 0.87
