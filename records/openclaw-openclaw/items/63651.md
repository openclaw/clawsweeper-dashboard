## Summary

Keep this PR open, but do not merge it as-is. The Windows duplicate restart output path is real on current main, but the patch removes the only human success line for completed schtasks/login-item restarts and bundles unrelated browser CDP SSRF policy work into a gateway output fix.

## What This Changes

The PR removes Windows schtasks/login-item restart stdout writes, changes scheduled-task helper logic and lowercase normalization, and adds browser CDP profile SSRF validation with trusted-mode DNS-skip behavior.

## Best Possible Solution

Land a focused gateway-only fix that emits exactly one human success message after the final completed Windows restart flow, including stale-PID retry, and keep the browser CDP SSRF policy work in a separate security-reviewed PR.

## Reproduction Assessment

Yes. Current main has deterministic source evidence: schtasks/login-item restart paths write to stdout, and the stale-PID post-restart path can call the same restart implementation again. A Windows live run would confirm the exact console output, but the duplicate path is clear from code.

## Solution Assessment

No. Removing the service-level writes without adding a central completed-restart success message would de-duplicate retries by making successful completed restarts silent. The safer solution is to centralize one final success line in the lifecycle flow and split unrelated browser changes.
