## Summary

Keep open: the duplicate Windows restart output is source-reproducible, but this PR is not merge-ready because it removes both service-layer success messages without adding a single central non-JSON success line, lacks real Windows output proof, is dirty, and bundles unrelated browser CDP SSRF changes.

## What This Changes

The branch removes Windows schtasks/startup-fallback restart stdout success writes and also changes browser CDP profile SSRF validation/tests plus schtasks helper code.

## Best Possible Solution

Land a narrow gateway-only fix that emits exactly one completed Windows restart success line after the final retry/health path, and split or drop the browser CDP policy work from this PR.

## Reproduction Assessment

Yes, by source inspection: current main writes Windows restart success inside restartStartupEntry/restartScheduledTask, and stale-PID recovery can call service.restart a second time. I did not run a live Windows schtasks command in this read-only review.

## Solution Assessment

No. Deleting both service-layer writes relies on a completed-restart emit path that does not print non-JSON success output, so the safer fix is a single central success message after the final retry/check path and a separate review for browser CDP SSRF changes.
