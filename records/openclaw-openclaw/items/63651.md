## Summary

Keep open. Current main still has the Windows duplicate-output path, but this PR is not mergeable as-is because it can silence normal completed restarts, bundles unrelated security-sensitive browser CDP changes, and lacks after-fix real behavior proof.

## What This Changes

The PR removes Windows schtasks/startup-fallback restart stdout success writes, refactors scheduled-task run helpers, normalizes several string comparisons, and adds browser CDP profile SSRF validation changes and tests.

## Best Possible Solution

Implement a narrow gateway-only fix that centralizes exactly one completed restart success line after the final retry/health path, and split any browser CDP SSRF policy work into a separate security-reviewed PR.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main writes restart success inside the Windows service implementation, and stale-PID recovery can call that implementation again. I did not run a live Windows schtasks reproduction in this read-only review.

## Solution Assessment

No. Removing both service-layer writes without adding one central completed-restart message makes ordinary successful Windows restarts silent, and the browser CDP SSRF work should be split from the gateway output fix.
