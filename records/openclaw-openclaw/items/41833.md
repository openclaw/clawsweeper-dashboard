## Summary

Keep this open. Current main now has opt-in inferred commitments, but that implementation is a scoped heartbeat check-in memory, not the requested deferred-work primitive with explicit add/done operations, cron/task completion reconciliation, overdue escalation, and startup/session surfacing for unresolved promised work.

## What This Changes

Requested behavior: add a native commitment/deferred-task primitive with durable completion tracking, overdue escalation, status visibility, and CLI/API operations above the existing cron/task automation.

## Best Possible Solution

Design a first-class commitment layer that reuses cron for timing and tasks/Task Flow for execution records: explicit `add/list/done/cancel/status` operations, durable links to scheduled runs or background tasks, completion reconciliation, overdue or failed-run escalation through configured channels, and session/startup surfacing for unresolved commitments. Keep inferred heartbeat check-ins as a separate or lower-level input path rather than treating `sent` as completed work.

## Reproduction Assessment

Yes. Current-main source and docs provide a high-confidence gap check: `openclaw commitments` can inspect/dismiss inferred follow-ups, while the requested `commit add/done` scheduling and completion-verification path is absent; the issue discussion also gives a concrete live failure pattern.

## Solution Assessment

Unclear. A native layer above cron/tasks appears directionally appropriate for durable promises, but the exact capture, escalation, and completion semantics need maintainer product design before this is safe as a narrow implementation PR.
