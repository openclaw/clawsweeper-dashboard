## Summary

Keep open: the exact empty-array liveness theory is contradicted by source, because successful empty `getUpdates` responses publish connected status, but the restart-loop symptom remains source-actionable through the isolated worker timeout path when `timeoutSeconds` exceeds the 45s `getUpdates` guard.

## What This Changes

The issue reports Telegram idle polling accounts being marked disconnected and restarted in a loop after upgrading to OpenClaw 2026.5.12.

## Best Possible Solution

Keep the issue open and fix isolated Telegram polling so its long-poll timeout cannot outlive the hard `getUpdates` request guard, while preserving empty successful polls as liveness.

## Reproduction Assessment

Yes, at source level for configs where Telegram isolated polling receives `timeoutSeconds` above the 45s `getUpdates` guard: an idle long poll can be aborted before `poll-success`, leaving the account disconnected until health-monitor restart. The exact reporter config is missing, so this is not a live reproduced current-main case.

## Solution Assessment

Yes: the narrow maintainable fix is to align the isolated worker long-poll timeout with the existing 45s request-timeout contract. Adding a new `healthMonitor.requireUpdatesForLiveness` option would not address the timeout mismatch and would add an unnecessary second policy surface.

## Repair Work Prompt

Fix the Telegram idle isolated-polling restart loop reported at https://github.com/openclaw/openclaw/issues/83264. Current main passes `channels.telegram.timeoutSeconds` / account timeout into `createTelegramIngressWorker`, and `extensions/telegram/src/telegram-ingress-worker.runtime.ts` uses that value as the Bot API `getUpdates` long-poll `timeout` while `fetchJson` aborts at `TELEGRAM_GET_UPDATES_REQUEST_TIMEOUT_MS` (45s). Bound or derive the isolated worker Bot API long-poll timeout so it cannot outlive the hard request guard, preserving the existing behavior where a successful empty `getUpdates` array posts `poll-success` and calls `notePollSuccess`. Add focused regression coverage for isolated ingress with `timeoutSeconds > 45` showing idle polling does not remain disconnected or hit the hard abort before liveness is published. Do not add a new healthMonitor config option, weaken gateway health policy, or conflate this with Telegram 409 conflict handling. Include a user-facing changelog fix entry if preparing the PR for landing.
