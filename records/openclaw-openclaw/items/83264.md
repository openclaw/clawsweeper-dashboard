## Summary

Keep open: the exact claim that empty completed `getUpdates` responses are ignored does not match current or v2026.5.12 source, but the same restart-loop symptom is source-plausible when isolated polling receives a configured Telegram `timeoutSeconds` above the 45s `getUpdates` request guard.

## What This Changes

The issue reports that idle Telegram polling accounts on OpenClaw 2026.5.12 restart repeatedly because the health monitor does not observe liveness while accounts receive no incoming updates.

## Best Possible Solution

Keep the issue open and make isolated polling use a bounded long-poll timeout that cannot outlive the `getUpdates` request guard, while preserving empty successful polls as liveness.

## Reproduction Assessment

Yes at source level for configs with Telegram `timeoutSeconds` greater than the 45s `getUpdates` guard: idle polls can abort before `poll-success`, leaving the account disconnected until the health monitor restarts it. The exact reporter config is missing, so this is not a live reproduced current-main case.

## Solution Assessment

Yes: the narrow maintainable fix is to align the isolated worker long-poll timeout with the existing 45s request-timeout contract, not to add a new liveness config flag.

## Repair Work Prompt

Fix the Telegram isolated polling idle health-monitor false positive from https://github.com/openclaw/openclaw/issues/83264. Current source passes `channels.telegram.timeoutSeconds` into the isolated ingress worker as the raw Bot API `getUpdates.timeout`, while OpenClaw's request-timeout contract aborts `getUpdates` at 45s and ignores higher configured timeouts; idle accounts with a higher timeout can abort before `poll-success`, leaving `connected=false` and causing health-monitor `disconnected` restarts. Keep core health policy semantics, do not add a new config option, and preserve the existing behavior that a completed empty `getUpdates` result marks polling live. Prefer a bounded worker poll timeout below or aligned with the request guard, with focused regression coverage for high `timeoutSeconds` plus no updates. Add a changelog entry because this is a user-visible Telegram reliability fix.
