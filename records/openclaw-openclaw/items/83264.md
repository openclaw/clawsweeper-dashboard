## Summary

Keep open: current main still has a source-plausible Telegram restart-loop path for idle accounts when isolated polling receives a configured `timeoutSeconds` above the 45s `getUpdates` request guard. The report's exact empty-array theory is not what the code does, because empty successful polls are treated as liveness, but the same symptom remains credible through the isolated worker timeout mismatch.

## What This Changes

Investigate a reported OpenClaw 2026.5.12 Telegram idle-account health-monitor false positive that restarts polling workers in a loop.

## Best Possible Solution

Bound the isolated worker's Bot API long-poll timeout so it cannot outlive the existing 45s `getUpdates` request guard, and keep successful empty polls as liveness without adding a new health-monitor config flag.

## Reproduction Assessment

Yes at source level: with Telegram `timeoutSeconds` greater than 45, an idle isolated `getUpdates` call can be aborted before Telegram returns an empty array, leaving the account disconnected until the health monitor restarts it. The reporter's exact config was not provided, so this is not a live reproduced current-main case.

## Solution Assessment

Yes: the narrow maintainable fix is to align isolated long-poll timeout behavior with the existing Telegram request-timeout contract, not to add a new liveness policy flag or change gateway health semantics.

## Repair Work Prompt

Fix the Telegram idle-account restart loop tracked by https://github.com/openclaw/openclaw/issues/83264. Current main passes `channels.telegram.timeoutSeconds` into the isolated ingress worker, while `fetchJson` aborts `getUpdates` after `TELEGRAM_GET_UPDATES_REQUEST_TIMEOUT_MS` (45s); when configured `timeoutSeconds` is higher, idle long polls can abort before returning `[]`, so no `poll-success` marks liveness and the gateway health monitor can restart the account as disconnected. Keep the fix inside the Telegram plugin owner boundary. Make isolated polling use a bounded Bot API long-poll timeout that cannot outlive the existing request guard, preserve the current behavior that successful empty `getUpdates` responses count as liveness, and do not add a new healthMonitor config flag or change generic gateway health policy. Add focused regression coverage for a high configured timeout in isolated polling and keep existing timeout helper tests green. Add a user-facing changelog entry if the repair changes shipped behavior.
