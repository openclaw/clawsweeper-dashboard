## Summary

Keep open: current main still forces Codex app-server turns in an active OpenClaw sandbox from `danger-full-access` to `workspace-write`, and the resulting turn policy hardcodes `networkAccess: false`, which matches the reported loss of outbound access. The exact beta-only timeline is not proven because the latest release tag also contains this policy path, but there is no current-main fix to close against.

## What This Changes

The issue reports that sandboxed Codex app-server agents cannot use outbound network access after OpenClaw clamps native Codex sandboxing to `workspace-write`.

## Best Possible Solution

Keep the OpenClaw sandbox boundary, but make Codex native app-server turns honor the existing sandbox egress contract, likely through an explicit externally managed sandbox policy, with regression coverage for network-enabled sandbox agents.

## Reproduction Assessment

Yes, source-reproducible: with OpenClaw sandboxing enabled, current main clamps Codex app-server `danger-full-access` to `workspace-write`, and turn policy then sends `networkAccess: false`. I did not run the reporter's live DNS or Google API flow.

## Solution Assessment

Unclear until owner review: disabling the sandbox is not the right fix, and a bare new per-agent switch could widen egress too much. The maintainable path should derive Codex network policy from the existing sandbox egress configuration without weakening isolation.
