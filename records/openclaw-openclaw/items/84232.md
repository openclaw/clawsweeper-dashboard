## Summary

Keep this PR open: it is a plausible direction for implicit source replies, but it intentionally leaves explicit current-channel targets on required durable delivery, while the linked Slack repro logs show the failing calls include `target` plus `bestEffort:false`. It also lacks after-fix real Slack proof.

## What This Changes

The PR changes `runMessageAction` to force `bestEffort:true` for `message_tool_only` sends without an explicit target and adds Slack-style regression tests for implicit versus explicit target sends.

## Best Possible Solution

Classify message-tool sends to the active source conversation as source replies even when the model supplies the current target, keep required durability for true out-of-band targets, and prove the Slack group/channel flow in a real setup.

## Reproduction Assessment

Yes at source level, but not with live Slack proof. The linked logs show a Slack `message_tool_only` payload with `target` and `bestEffort:false`, and current code still maps that to required durable delivery that requires `reconcileUnknownSend`.

## Solution Assessment

No. The patch helps omitted-target calls, but the reported payload includes the current Slack target, so the safer solution is to distinguish current-source targets from true out-of-band targets and then prove the real Slack flow.
