## Summary

Keep open: the PR addresses a real acpx config gap, but it should not merge as-is because structured args are concatenated without quoting before acpx's command-line parser, so whitespace or quoted args break. The user-facing feature also needs the required changelog entry.

## What This Changes

The PR adds optional `agents.<id>.args: string[]` support to the acpx plugin config, manifest schema, resolver, and config tests.

## Best Possible Solution

Keep the new config surface only if structured args are encoded losslessly at the acpx string boundary, with regression tests and the required changelog/docs follow-through.

## Reproduction Assessment

Yes, from source inspection: configure `command: "node"` with `args: ["/tmp/my adapter.mjs", "--verbose"]`; this PR builds `node /tmp/my adapter.mjs --verbose`, and acpx 0.6.1 then splits it on whitespace before spawn. I did not run the test suite because the review is read-only and `pnpm` is unavailable.

## Solution Assessment

No. The direction is reasonable, but the implementation should quote or otherwise preserve every structured arg before handing a command string to acpx, and it needs the required changelog entry.
