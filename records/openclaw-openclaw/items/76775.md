## Summary

Keep open: the feature is useful, but the current patch is not merge-ready because it flattens structured argv back into an ambiguous command string and omits the required user-facing changelog entry.

## What This Changes

The PR adds optional `agents.<id>.args: string[]` to ACPX plugin agent overrides and appends those args to the configured agent command before handing it to the ACPX registry.

## Best Possible Solution

Keep the ACPX `args` config feature, but serialize args losslessly under the current acpx string contract or move to a structured acpx registry API if that becomes available, with regression coverage for whitespace and quotes plus changelog/docs alignment.

## Reproduction Assessment

Yes for the review finding: `args: ["/tmp/My Adapter.mjs", "--name=foo bar"]` would be converted to a single string that acpx@0.6.1 later splits into extra argv tokens. I verified this from the PR diff and the pinned acpx source rather than by running tests.

## Solution Assessment

No. Joining with spaces reintroduces the quoting problem that structured `args` is meant to avoid; the narrower maintainable fix is to quote each appended arg correctly or preserve structure until spawn.
