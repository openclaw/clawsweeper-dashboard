## Summary

Keep open: the feature is useful, but the patch loses argv boundaries by joining `args` with spaces before acpx re-parses the command string, and it lacks the required changelog coverage.

## What This Changes

The PR adds optional `agents.<id>.args` to the ACPX plugin config schema and manifest, then appends those args to agent command overrides with unit coverage.

## Best Possible Solution

Keep the config feature, but serialize `args` losslessly or add a structured acpx registry API, cover whitespace/quote cases, and update docs/changelog for the new ACPX plugin config.

## Reproduction Assessment

Yes for the review finding: with `args: ["/tmp/My Adapter.mjs", "--name=foo bar"]`, the patch serializes a single string that acpx splits into extra argv tokens. I verified the path from the PR diff and acpx@0.6.1 source rather than by running tests.

## Solution Assessment

No. `args.join(" ")` is not the best implementation because it reintroduces the quoting problem; the safer fix is to quote or otherwise preserve argv boundaries and add regression coverage for whitespace and quote characters.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve argv boundaries when serializing args:** `extensions/acpx/src/config.ts:244`
  - body: This joins `args` with spaces before handing the result to acpx, but acpx later calls `splitCommandLine()` and then `spawn(command, args)`. An entry like `"/tmp/My Adapter.mjs"` or `"--name=foo bar"` is split into multiple argv tokens, which is the quoting problem this structured field is meant to avoid. Please shell-quote each arg or keep command/args structured until acpx can spawn them directly.
