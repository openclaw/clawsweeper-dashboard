## Summary

Keep open: the ACPX args feature is useful, but the current patch loses argv boundaries by joining structured args with spaces before acpx parses and spawns the command, and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds optional `plugins.entries.acpx.config.agents.<id>.args` schema and manifest support, then flattens those args onto ACPX agent command override strings with config tests.

## Best Possible Solution

Keep the config feature, but serialize `args` losslessly using acpx-compatible quoting or a structured upstream API, add whitespace/quote regression coverage, and update changelog/docs for the new ACPX plugin config.

## Reproduction Assessment

Yes for the review finding: with `args: ["/tmp/My Adapter.mjs", "--name=foo bar"]`, the PR serializes a single command string that acpx later splits into extra argv tokens. This was verified from the PR diff and acpx@0.6.1 source rather than a runtime test.

## Solution Assessment

No. Adding `args` is a reasonable feature direction, but `args.join(" ")` reintroduces the quoting problem; the narrower maintainable fix is acpx-compatible quoting or preserving structured argv until spawn.
