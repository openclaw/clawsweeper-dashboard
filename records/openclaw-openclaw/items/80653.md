## Summary

Keep open: current main still restores same-bot Telegram offset state by bot id only, then drops lower update IDs as already accepted. The linked fix PR at https://github.com/openclaw/openclaw/pull/80671 uses closing syntax and is still open, so this issue should stay open until that PR or a replacement lands.

## What This Changes

Telegram long-polling should recover after BotFather token rotation instead of reusing a stale persisted getUpdates offset and dropping new messages.

## Best Possible Solution

Land a reviewed Telegram plugin fix that binds persisted offsets to a non-secret token epoch or equivalent startup repair, preserves safe upgrade behavior, and adds same-bot token-rotation regression coverage.

## Reproduction Assessment

Yes. A current-main source repro is clear: persist a version 2 high `lastUpdateId` for bot id `111111`, restart with token `111111:rotated-secret`, then lower incoming update IDs are skipped as `accepted-watermark`.

## Solution Assessment

Yes. The narrow maintainable direction is in the Telegram offset-store/startup path using a non-secret token fingerprint or equivalent epoch; the linked PR is the right review lane, with any PR-specific defects handled there before merge.
