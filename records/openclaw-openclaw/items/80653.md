## Summary

Keep open: current main still has a plausible stale-offset drop path for same-bot Telegram token rotation, because the persisted offset is validated by bot id rather than the actual token epoch and lower update IDs are skipped as already accepted.

## What This Changes

The issue asks Telegram long-polling to reset stale persisted update offsets after BotFather token rotation so new lower update IDs are received.

## Best Possible Solution

Bind Telegram offset state to a non-secret token epoch/fingerprint or equivalent startup repair, treat legacy bot-id-only offsets carefully, and add a regression for same-bot token rotation with lower update IDs.

## Reproduction Assessment

Yes, at source level: a persisted high `lastUpdateId` for the same bot id is restored, and an incoming lower `update_id` is rejected by `createTelegramUpdateTracker` as `accepted-watermark`. I did not run a live Telegram/BotFather rotation in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Telegram plugin offset-store/startup path. Avoid reintroducing a pre-poll `getUpdates` confirmation call because current tests and docs intentionally avoid that conflict-prone path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80653 in the Telegram plugin. Current main keys persisted update offsets to account + bot id, so a BotFather token rotation for the same bot can keep the old `lastUpdateId` and cause `createTelegramUpdateTracker` to drop lower `update_id` values as `accepted-watermark`. Implement a non-secret token epoch/fingerprint in the offset state, or an equivalent startup repair, so same-bot token changes reset or ignore stale offsets without storing the raw token. Treat legacy bot-id-only records carefully, add regression tests for same bot id/different token with lower `update_id`, and do not reintroduce a pre-poll `getUpdates` confirmation call because Telegram polling tests intentionally avoid that 409-prone path.
