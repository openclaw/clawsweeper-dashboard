## Summary

Keep open: current main still restores same-bot Telegram offsets by bot id only, and an open closing PR specifically targets this issue but is not merged yet.

## What This Changes

The issue reports that rotating a Telegram bot token can leave OpenClaw using a stale persisted long-poll update offset, silently dropping new messages.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/80671 with a Telegram plugin-local token-epoch/fingerprint repair that handles released v2 offset files and adds regression coverage.

## Reproduction Assessment

Yes, at source level: a same-bot persisted high lastUpdateId is accepted by readTelegramUpdateOffset and then lower update_id values are rejected by the tracker as accepted-watermark. I did not run a live Telegram/BotFather rotation in this read-only review.

## Solution Assessment

Yes in direction: the narrow fix belongs in the Telegram offset-store and monitor startup path using a non-secret token epoch or equivalent. Because an open closing PR already targets this, maintainer review should happen there before this issue closes.
