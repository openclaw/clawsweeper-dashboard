## Summary

Keep open: current main invalidates Telegram polling offsets when the numeric bot id changes, but BotFather token regeneration for the same bot can keep that id stable while changing the secret, so the stale offset can still be reused and lower update ids are rejected before dispatch.

## What This Changes

The issue reports that rotating a Telegram bot token can leave OpenClaw using a stale long-polling update offset, causing new messages to be ignored until state is cleared.

## Best Possible Solution

Make Telegram offset persistence credential-rotation aware without storing raw tokens, and preserve the existing no-startup-getUpdates behavior that avoids 409 conflict loops.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a clear reproduction model: a same-bot token change can keep the stored botId match, reuse the old lastUpdateId, and then reject lower update_id values as accepted-watermark.

## Solution Assessment

Yes, the best fix is narrow and belongs in the Telegram plugin: make the offset record token-rotation aware with a secret-safe fingerprint and focused regression coverage, without adding a config knob or a broad core migration.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80653 in the Telegram plugin. Current main stores only lastUpdateId plus numeric botId in extensions/telegram/src/update-offset-store.ts, so BotFather token regeneration for the same bot can reuse the old offset; createTelegramUpdateTracker then drops lower update_id values as accepted-watermark. Implement a secret-safe token-rotation-aware offset contract, for example by storing a one-way fingerprint of the full token or equivalent non-raw token identity and treating missing/mismatched fingerprints as stale. Do not persist raw Telegram bot tokens. Preserve the existing behavior that monitor startup does not issue a separate getUpdates confirmation, because that was intentionally avoided after 409 conflict loops. Add focused regression tests for same-bot-token-secret rotation and legacy/no-fingerprint records, plus any tracker/monitor coverage needed to show lower fresh update ids are accepted after reset. Changelog is expected if the fix is user-visible.
