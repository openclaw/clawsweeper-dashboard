## Summary

Keep this PR open. Current main has partial legacy filtering for heartbeat/envelope leaks, but it does not implement the PR's source-tagged `internal_system` path; the PR still targets a real user-visible leak class, while the current diff should not merge until the provenance marker is restricted to trusted internal ingress.

## What This Changes

The branch tags heartbeat/cron/exec-event turns with `InputProvenance.kind = "internal_system"`, hides matching user messages in gateway and Control UI history, adds focused regressions, and includes incidental formatting/import churn in adjacent tests/plugins.

## Best Possible Solution

Land the source-tagged design only after tightening the trust boundary: set `internal_system` at trusted internal heartbeat/cron/exec origins, reject or strip untrusted `internal_system` provenance at gateway ingress such as `agent.run`, keep gateway projection as the shared suppression point with Control UI defense-in-depth, and preserve legacy text/envelope cleanup for old transcripts.

## Reproduction Assessment

Yes. The leak class has concrete Control UI/webchat reports in the PR discussion and related #72918/#75232, and current-main source inspection shows exec/cron/heartbeat turns are not source-tagged. The PR-specific regression is also source-reproducible: a caller can submit `agent.run` with `inputProvenance.kind === "internal_system"`, which the branch then treats as authority to hide the turn from history.

## Solution Assessment

No, not as-is. Source-tagging internal turns at origin is the maintainable direction, but the current patch must first make `internal_system` a trusted-origin-only marker before gateway/UI history filters use it to hide user-role messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Gate hidden provenance on trusted ingress:** `src/gateway/chat-display-projection.ts:396`
  - body: The new history filter drops any user message with `provenance.kind === "internal_system"`, but `agent.run` accepts request `inputProvenance` from the public schema and normalizes it without the admin-scope gate that `chat.send` uses for system provenance. A gateway caller can persist an arbitrary user turn with that kind and this branch will hide it from chat/session history, so reject or strip `internal_system` at untrusted ingress before filtering on it.
  - confidence: 0.84
