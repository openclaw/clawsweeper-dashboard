## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the repo TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, while this repo declares and consistently imports `typebox`; loading this file can fail before `sessions_start` is usable.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses any agent-scoped key, this bootstraps only the agent's main session and continues there. A missing key like `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning not found.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping replacements:** `src/gateway/sessions-bootstrap.ts:117-122`
  - body: `applySessionsPatchToStore()` preserves an existing entry. If a dead main entry exists without `sessionId`, bootstrap can still lack a new id; if `force` is used on an active entry, it can keep the old transcript instead of creating a fresh session.
  - confidence: 0.88
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. Restricted setups could start or probe sessions the model should not access.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal write scopes will hit the default admin requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter it out before execution.
  - confidence: 0.86
- **[P2] Preserve storage errors from bootstrap failures:** `src/gateway/server-methods/sessions.ts:416-424`
  - body: The auto-start path rewrites every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`, so transcript or filesystem failures are reported as a bad key instead of an infrastructure error callers can retry or diagnose.
  - confidence: 0.84
- **[P2] Restore the prior session entry on bootstrap failure:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, rollback unconditionally deletes the store entry. In replacement or force flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
- **[P2] Emit session change events after sessions.start creates:** `src/gateway/server-methods/sessions.ts:863-895`
  - body: `sessions.start` can create a new session but returns without emitting the same session-change notification used by other create paths, so subscribed clients can keep stale session lists until a later mutation.
  - confidence: 0.8
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48-hour jump failure mode, so the fallback should stay anchored near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path records schedule errors instead of aborting, and force runs should preserve that behavior.
  - confidence: 0.84
- **[P2] Send read receipts before waiting for terminal status:** `src/agents/tools/sessions-send-tool.a2a.ts:54-58`
  - body: The receipt path runs only after `agent.wait` returns `ok`, which is a terminal snapshot. Long-running fire-and-forget sends can time out even though processing started, so no receipt is emitted for the delivery-tracking behavior this change introduces.
  - confidence: 0.86
