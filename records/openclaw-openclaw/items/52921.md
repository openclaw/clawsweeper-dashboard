## Summary

Keep open: current main still has the missing-session failure, but this PR is not ready because it mixes the session recovery work with unrelated cron changes and leaves blocking lifecycle, policy, catalog, and scheduler defects.

## What This Changes

The PR adds structured session-miss errors, `sessions_send` auto-bootstrap/message tracking/read-receipt behavior, a `sessions.start` gateway method and `sessions_start` tool, generated Apple/tool-display updates, and cron scheduler fallback/force-run changes.

## Best Possible Solution

Land a focused fix for #52875 that defines the missing-session recovery contract, preserves session visibility/A2A/sandbox rules, updates protocol schemas/generated clients/tool catalogs/docs/tests together, and moves cron scheduler changes into a separate review path.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_send` reaches `handleSessionSend` and returns `INVALID_REQUEST`/`session not found` when the configured target has no active `sessionId`, matching the linked regression report.

## Solution Assessment

No. The PR targets a real failure mode, but the current implementation is not the narrowest maintainable solution because it broadens gateway/tool behavior without preserving existing policy and includes unrelated cron changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the repo's TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but the repo depends on `typebox` and the PR does not add `@sinclair/typebox` to package metadata. Loading this tool can fail module resolution before `sessions_start` is usable.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses an agent-scoped key, this path bootstraps the agent's main session and continues with that canonical key. A missing key like `agent:ops:incident-123` would be delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves existing entries. If the dead main entry exists without `sessionId`, or `force` is meant to replace a stuck session, this bootstrap path does not allocate a fresh id before transcript creation and can fail or reuse the stuck transcript.
  - confidence: 0.88
- **[P2] Preserve bootstrap infrastructure errors:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: The auto-start path rewrites every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`. Transcript or storage failures should remain infrastructure errors so callers can retry or diagnose them correctly.
  - confidence: 0.84
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:38-45`
  - body: `sessions_start` forwards arbitrary `agentId` values to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by existing session tools. Restricted agents can start or probe sessions outside their allowed surface.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal write scopes will hit the default admin requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes `sessions_start`, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter the recovery tool out before execution.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the degraded fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48-hour jump failure mode, so the fallback should stay near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches those failures and records schedule errors, so force-run preflight should use the same recoverable behavior.
  - confidence: 0.84
- **[P2] Restore prior session entries on bootstrap rollback:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
