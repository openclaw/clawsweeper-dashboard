## What This Changes

## Best Possible Solution

Keep the narrower current-main auto-create path for configured agent main-session sends; revisit `sessions.start`, read receipts, or structured error-code expansion only in a focused policy-reviewed PR.

## Reproduction Assessment

Yes for the original source-level failure: the PR base returned a missing-session error before any bootstrap path. Current main no longer reproduces the central configured-agent main-session case because it creates the session before sending.

## Solution Assessment

No, this PR is not the best way to solve it. Current main fixes the central recovery path more narrowly, while this branch adds broader protocol/tool/read-receipt and cron changes with blocking defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-428`
  - body: When `sessions.send` misses any agent-scoped key, this bootstraps the agent main session and continues there, so a missing named session can be silently delivered into `agent:<id>:main` instead of returning not found.
  - confidence: 0.91
- **[P1] Allocate a fresh session id before bootstrapping:** `src/gateway/sessions-bootstrap.ts:117-122`
  - body: `applySessionsPatchToStore()` preserves an existing entry, so dead or forced main-session recovery can keep a missing or stale `sessionId` and fail or reuse the old transcript instead of creating a fresh session.
  - confidence: 0.88
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`.
  - confidence: 0.92
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence when this recovery path is used, reintroducing the 48-hour jump failure mode covered by existing cron regression tests.
  - confidence: 0.87
