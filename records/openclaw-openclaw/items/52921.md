## Summary

Keep open: this is the active implementation candidate for a real missing-session recovery problem, but the current branch still has blocking correctness issues, a session visibility/security concern, and no after-fix real behavior proof.

## What This Changes

The branch adds structured session-miss errors, `sessions.send` auto-bootstrap, a `sessions.start` gateway method and `sessions_start` agent tool, delivery metadata/read receipts, generated Apple/tool-display updates, and cron fallback/force-run changes.

## Best Possible Solution

Land a focused session recovery change that preserves existing visibility/A2A/sandbox policy, updates gateway protocol schemas/generated clients/tool catalogs/docs/tests together, and moves cron scheduler fixes to a separate review path.

## Reproduction Assessment

Yes for the current source-level limitation: current `main` returns `session not found` before any bootstrap path and has no `sessions.start`/`sessions_start` surface. I did not run the original Ubuntu/MiniMax upgrade scenario live.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best shape because it broadens gateway/tool lifecycle behavior without preserving existing access policy and bundles unrelated cron scheduler changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the repo TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but package metadata only declares `typebox`. Loading this file can fail module resolution before `sessions_start` is usable.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses any agent-scoped key, this bootstraps only the agent's main session and then continues with that key. A missing key like `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning not-found.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:117-122`
  - body: `applySessionsPatchToStore()` preserves an existing store entry. If the dead main entry exists without `sessionId`, this patch does not allocate a new id before transcript creation, so bootstrap can fail or keep the caller attached to the stuck session.
  - confidence: 0.89
- **[P2] Preserve bootstrap infrastructure errors:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: This maps every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`. Transcript or storage failures should remain infrastructure errors such as `UNAVAILABLE` so callers can retry or diagnose them correctly.
  - confidence: 0.84
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:32-47`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. In restricted setups, a model could start or probe sessions it should not access.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal `operator.write` scopes will hit the default admin requirement when invoking the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter the recovery tool out before execution.
  - confidence: 0.86
- **[P2] Send read receipts before terminal completion:** `src/agents/tools/sessions-send-tool.a2a.ts:54`
  - body: The receipt path only runs after `agent.wait` returns `ok`, which means a terminal snapshot. For fire-and-forget sends where the target starts but runs longer than the wait window, no receipt is emitted even though the agent did begin processing.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Seeding the degraded fallback from `now + 48h` can skip the next daily occurrence whenever this path is used. Existing regression coverage documents this two-day jump failure mode, so the fallback should stay anchored near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-544`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path records schedule errors instead of aborting the operator action.
  - confidence: 0.84
- **[P2] Restore the prior session entry on rollback:** `src/gateway/sessions-bootstrap.ts:139-140`
  - body: When transcript creation fails, rollback unconditionally deletes the store entry. In `force` or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
