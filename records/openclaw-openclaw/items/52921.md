## Summary

Keep this PR open because it targets a real regression that is still present on current main, but the proposed branch is not mergeable as-is: the session recovery API bypasses existing visibility policy, the gateway/protocol/tool catalog changes are incomplete, and the branch mixes unrelated cron scheduler changes that introduce regressions already covered by existing tests.

## What This Changes

The branch adds structured session-miss error codes, a `sessions.start` gateway method and `sessions_start` agent tool, `sessions_send` auto-bootstrap/message tracking/read-receipt behavior, generated Apple/tool-display updates, and cron scheduler fallback/force-run changes.

## Best Possible Solution

Land a focused fix for the linked missing-session regression that defines the structured missing-session recovery contract, preserves current session visibility/A2A/sandbox rules, updates gateway protocol schemas/generated clients/tool catalogs/docs/tests together, and moves the cron scheduler changes into a separate review path.

## Reproduction Assessment

Yes. The linked bug has a concrete path on current main: `sessions_send` to a configured agent with no active persisted `sessionId` reaches `handleSessionSend` and returns `INVALID_REQUEST`/`session not found` instead of bootstrapping or returning a structured session-miss code.

## Solution Assessment

No. The PR addresses the right user problem, but this branch is not the best solution because it broadens protocol and tool behavior while leaving correctness and security-policy gaps; the safer path is a narrower session-only recovery change that reuses the existing visibility and method-scope contracts.

- **[P1] Use the repo's TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`

  - body: When `sessions.send` misses an agent-scoped key, this branch bootstraps the agent's main session and continues with that canonical key. A missing key like `agent:ops:incident-123` would be delivered into `agent:ops:main` instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves an existing store entry. If a dead main entry exists without `sessionId`, this path does not allocate a new id before transcript creation, so bootstrap can fail and roll back instead of recovering the session.
  - confidence: 0.88
- **[P2] Preserve unavailable bootstrap failures:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: The auto-start path rewrites every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`. Transcript or storage failures should remain infrastructure errors so callers can retry or diagnose them correctly.
  - confidence: 0.84
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards arbitrary `agentId` values to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. Restricted agents can start or probe sessions outside their allowed surface.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal write scopes will hit the default admin requirement instead of being able to use the new write API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter out the recovery tool before execution.
  - confidence: 0.86
- **[P1] Keep cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48-hour jump failure mode.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path records those schedule errors instead of aborting the run.
  - confidence: 0.84

  - body: When transcript creation fails, rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
