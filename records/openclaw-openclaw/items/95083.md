## Summary

The central fix is useful, but the current patch is too broad: adding `stdio` to the unconditional MCP transport union lets `config validate` pass URL-only `transport: "stdio"` configs that the runtime still skips. The PR should stay open for a narrow validation repair and a conflict-free refreshed head.

## What This Changes

The PR widens MCP server config validation and exported config types to accept `transport: "stdio"` and adds schema regression coverage for command-bearing servers.

## Best Possible Solution

Accept explicit stdio only for the command-as-stdio contract, keep URL-only remote MCP validation aligned with runtime, add regression coverage for both shapes, and land one conflict-free candidate for the canonical issue.

## Reproduction Assessment

Yes. Source inspection shows `openclaw config validate` reaches `OpenClawSchema.safeParse`, whose current MCP schema rejects explicit stdio while the runtime accepts command-bearing servers as stdio; the new over-broad edge is also source-reproducible from the runtime unsupported-transport branch.

## Solution Assessment

No. The intended schema/type alignment is the right layer, but the unconditional union is too broad; the best fix is conditional validation that accepts `stdio` only for command-bearing servers and keeps URL-only remote configs aligned with runtime.
