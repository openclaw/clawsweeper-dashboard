## Summary

Keep open: the latest branch fixed the earlier unconditional-wake compatibility problem by adding an explicit `notificationWake.reactions` policy, but the opt-in wake path is still incomplete because heartbeat preflight does not classify `notifications-event` wake requests as wake payloads. It also still lacks live Telegram reaction proof for the changed transport behavior.

## What This Changes

The PR adds Telegram reaction keys plus config/schema/docs/tests for `notificationWake.reactions`, so accepted reactions queue by default and can opt into routed heartbeat wakes.

## Best Possible Solution

Keep queued reaction events as the default, land one shared notification-wake implementation with heartbeat-runner support, and require redacted live Telegram reaction proof before merge.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reactions and returns without a wake request, while the PR adds an opt-in wake path. I did not run a live Telegram reaction update in this read-only review.

## Solution Assessment

No, not as-is: the opt-in policy shape is better than the old unconditional wake, but it must also teach heartbeat preflight to treat notification wakes as wake payloads. The broader shared policy PR already shows that missing runtime piece.
