## Summary

Keep this PR open: the code path looks like a plausible narrow Telegram fix, but it is still an active external PR with an open linked issue, a failing real-behavior-proof check, and maintainer/product review still needed for changing accepted reaction notifications from passive context into wake-triggering input.

## What This Changes

The PR makes accepted Telegram reaction events include stable standard/custom reaction keys and request an immediate heartbeat wake after a system event is successfully enqueued.

## Best Possible Solution

Land a Telegram-owned implementation that wakes only after accepted, deduplicated reaction events, keeps reaction text untrusted, exposes stable standard/custom reaction keys, and includes redacted live proof of a real reaction-only wake.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reaction events and does not request a heartbeat wake. I did not run a live Telegram reaction in this read-only review.

## Solution Assessment

Mostly yes for the narrowed code path: using the existing heartbeat wake after a successful system-event enqueue is cleaner than synthesizing a Telegram message turn, and the PR preserves existing access gates. The remaining blockers are live reaction proof and maintainer approval for the default behavior change.
