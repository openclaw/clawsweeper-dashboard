## Summary

Keep open. Current main still lacks Telegram reaction semantics, custom emoji reaction handling, stable reaction keys, and reaction-triggered heartbeat wake; this PR remains the active implementation candidate for the linked open feature/bug work, but the patch is not merge-ready because it conflicts with current SDK guardrails and generated-surface expectations.

## What This Changes

The PR adds Telegram `reactionSemantics` config, reaction-key normalization for standard and custom emoji, semantic reaction event text and wake behavior, docs, tests, and supporting plugin SDK exports.

## Best Possible Solution

Keep the feature in the Telegram plugin boundary, update the branch to current main, use focused runtime SDK subpaths or a narrow approved wake seam, regenerate the config/API baselines, and land with targeted coverage for emoji, custom emoji, `wake|queue|ignore`, auth filtering, duplicate suppression, and queue-only generic reactions.

## Reproduction Assessment

Yes. On current main, a `message_reaction` update with a standard emoji only queues generic text through `enqueueSystemEvent`, and a `custom_emoji` reaction is filtered out because only `ReactionTypeEmoji` entries are collected.

## Solution Assessment

No, not as-is. The product direction is reasonable, and prior review feedback on collisions, invalid keys, and duplicate wake suppression appears addressed in the PR discussion, but the patch should be rebased onto current main's focused SDK surfaces and regenerated baselines before it is the maintainable fix.
