## Summary

Keep open: current main still lacks the proposed reaction semantics and mapped wake contract, and this PR remains plausible but is blocked by mock-only proof plus a maintainer choice between this broad API and the narrower wake fix.

## What This Changes

The PR adds Telegram `reactionSemantics` config, reaction-key normalization for emoji/custom emoji, mapped queue/ignore/wake handling via heartbeat, docs/generated metadata, SDK config exports, and focused tests.

## Best Possible Solution

Land or supersede this with a Telegram-owned reaction contract that normalizes standard/custom emoji, preserves authorization and routing limits, wakes only configured meaningful reactions, and includes schema, docs, tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset by source inspection: current main receives authorized Telegram `message_reaction` updates, enqueues system events, and returns without a heartbeat or message-processing wake. The broader `reactionSemantics` mapping is new public feature behavior rather than a pre-existing failing contract.

## Solution Assessment

Unclear pending maintainer scope choice. The implementation direction is coherent, but maintainers need to choose this broad semantics API versus the narrower wake-only implementation in https://github.com/openclaw/openclaw/pull/75899 and require live Telegram proof before merge.
