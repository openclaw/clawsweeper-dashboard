## Summary

Keep open: the code shape looks sound and I found no blocking correctness defect, but this external Telegram transport PR still lacks proof of an actual reaction-only wake and changes accepted reaction notifications from passive context into turn-triggering input, so it needs maintainer/product review before merge.

## What This Changes

The branch updates Telegram reaction handling to append stable standard/custom reaction keys, downgrade reaction events, request a heartbeat after successful enqueue, and update tests, docs, and changelog.

## Best Possible Solution

Land a Telegram-owned patch that wakes only after accepted and deduplicated reaction events, keeps reaction text untrusted, records stable reaction keys, and includes redacted live proof of a real reaction-only wake.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reaction events and returns without a heartbeat wake. I did not run a live Telegram reaction in this read-only review.

## Solution Assessment

Mostly yes: using the existing heartbeat wake path after successful system-event enqueue is narrower than synthesizing a Telegram message turn, and the PR preserves existing reaction/access gates. The remaining questions are product approval for waking by default and live Telegram proof.
