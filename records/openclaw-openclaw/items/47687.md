## Summary

Keep open. The code change is narrow and appears directionally correct, but this external Telegram behavior PR still lacks proof that a real reaction-only update wakes the routed session; the posted getMe/canary evidence verifies Telegram access and source shape, not the changed user path.

## What This Changes

The PR updates Telegram reaction handling to emit stable standard/custom reaction keys, downgrade reaction system events, request a heartbeat after successful enqueue, and document/test that behavior.

## Best Possible Solution

Land a Telegram-owned patch that wakes only after accepted and deduplicated reaction events, keeps reaction text untrusted, exposes stable reaction keys, and is backed by redacted real Telegram reaction proof.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram emoji reactions and returns without calling `requestHeartbeat`. I did not establish a live Telegram reaction repro in this read-only review.

## Solution Assessment

Mostly yes: using the existing heartbeat wake path after successful system-event enqueue is narrower than synthesizing a Telegram message turn and matches existing interaction wake patterns. The remaining issue is proof and maintainer approval for the default behavior change.
