## Summary

Keep open: current main still has the passive Telegram reaction path and this PR is a plausible narrow fix, but the external PR merge gate is not met because proof is mock-only and the default wake behavior still needs maintainer approval.

## What This Changes

The PR updates Telegram reaction handling to emit stable standard/custom reaction keys, enqueue reaction system events as untrusted, request an immediate heartbeat wake after successful enqueue, and update docs/changelog/tests.

## Best Possible Solution

Land a Telegram-owned wake implementation only after maintainer approval of the wake semantics, preserving existing access gates and untrusted event handling, with focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main's message_reaction handler enqueues accepted standard emoji system events and returns without calling requestHeartbeat or processMessage. I did not run live Telegram proof in this read-only review.

## Solution Assessment

Mostly yes: using the existing heartbeat wake path after successful enqueue is narrower than synthesizing a Telegram message turn, and the PR preserves access gates and untrusted event handling. The remaining question is whether maintainers accept immediate wake as the default behavior for accepted reaction notifications.
