## Summary

Keep open. Current main still lacks Telegram reaction semantics and mapped wake behavior, so the contribution remains useful for the linked Telegram reaction requests, but the current PR head is not merge-ready because it uses deprecated broad SDK subpaths and leaves generated config/SDK artifacts plus changelog coverage incomplete.

## What This Changes

The PR adds Telegram `reactionSemantics` config, standard/custom emoji reaction-key normalization, semantic reaction system events, mapped heartbeat wake behavior, docs, tests, and supporting config/SDK exports.

## Best Possible Solution

Land a rebased Telegram-owned implementation that keeps generic reactions queue-only, maps configured standard/custom emoji reactions to semantic system events, wakes only authorized mapped `wake` reactions after a successful enqueue, and keeps schema, docs, generated metadata, SDK baselines, and changelog aligned with current main's focused plugin SDK boundaries.

## Reproduction Assessment

Yes. On current main, a Telegram `message_reaction` update with a standard emoji follows the handler path that queues generic text only; a `custom_emoji` reaction is filtered out because only `ReactionTypeEmoji` entries are collected, and there is no config key that can request a wake.

## Solution Assessment

No, not as-is. The product direction is narrow and maintainable, and earlier review comments show several semantic edge cases were addressed, but this branch must be rebased onto current main's focused SDK subpaths and regenerated public artifacts before it is the best fix.
