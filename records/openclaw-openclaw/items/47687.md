## Summary

Keep open: the code path looks focused and I found no blocking code findings, but the updated proof still does not show a real Telegram reaction-only input waking the routed session. This also needs maintainer/product approval because it changes accepted Telegram reactions from passive context into wake-triggering input under existing reaction notification settings.

## What This Changes

The PR updates Telegram reaction handling to emit stable standard/custom reaction keys, downgrade reaction system events, request a heartbeat after successful enqueue, and document/test that behavior.

## Best Possible Solution

Land a Telegram-owned patch that wakes only after accepted and deduplicated reaction events, keeps reaction text untrusted, records stable reaction keys, and includes redacted live proof of a real reaction-only wake.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reaction events and returns without a heartbeat wake. I did not run a live Telegram reaction in this read-only review.

## Solution Assessment

Mostly yes: using the existing heartbeat wake path after successful system-event enqueue is narrower than synthesizing a Telegram message turn, and the PR preserves existing reaction/access gates. The remaining question is maintainer approval for making existing accepted reaction notifications wake by default.
