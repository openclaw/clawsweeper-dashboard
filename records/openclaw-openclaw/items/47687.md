## Summary

Keep open: this PR is useful but not merge-ready because the opt-in notification wake request is not classified by the heartbeat runner, and live Telegram reaction proof is still absent. ## What This Changes

The branch adds `notificationWake.reactions` config/schema support, a shared notification system-event helper, Telegram standard/custom reaction keys, docs/generated config updates, and focused tests.

## Best Possible Solution

Use one canonical shared notification wake implementation, preserve queue-by-default compatibility, classify `notification-wake:*` as a heartbeat wake payload, update the Plugin SDK API baseline, and require redacted live Telegram reaction proof before merge.

Yes at source level: on this PR head, a Telegram reaction with `notificationWake.reactions: "wake"` calls `requestHeartbeat` using `source: "notifications-event"`, while heartbeat preflight does not treat that as a wake payload. I did not run the PR branch because this review is read-only.

## Solution Assessment

No: the queue-default plus opt-in policy is a reasonable direction, but this branch is incomplete without heartbeat-runner classification, Plugin SDK API baseline updates, and live Telegram proof. The overlapping shared-policy branch already demonstrates the missing runner fix.
