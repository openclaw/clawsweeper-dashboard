## Summary

Keep this PR open. Current main still lacks Telegram reaction semantics and reaction-triggered wake behavior, so the contribution is not obsolete, but the branch is not merge-ready against current main because it reintroduces forbidden broad SDK imports, omits generated config/API artifacts, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds Telegram `reactionSemantics` config, standard/custom emoji reaction-key normalization, semantic reaction system events, mapped wake behavior, docs, tests, and supporting SDK/config exports.

## Best Possible Solution

Land a rebased Telegram-owned implementation that keeps generic reactions queue-only, maps configured standard/custom emoji reactions to semantic system events, wakes only authorized mapped `wake` reactions after a successful enqueue, and keeps schema, docs, generated metadata, SDK baselines, and changelog aligned.

## Reproduction Assessment

Yes. A current-main `message_reaction` payload with a standard emoji follows the handler path that queues generic text only; a `custom_emoji` payload is filtered out because only `ReactionTypeEmoji` entries are collected, and no current config key can request a wake.

## Solution Assessment

No, not as-is. The product direction is reasonable, and prior comments show several correctness issues were addressed, but the branch needs current-main SDK import cleanup, generated baseline updates, and changelog coverage before it is the narrow maintainable fix.
