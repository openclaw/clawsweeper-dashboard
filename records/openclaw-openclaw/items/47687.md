## Summary

Keep open: the PR addresses a real Telegram reaction workflow gap, but the patch makes accepted reactions wake sessions unconditionally instead of preserving queued-only behavior by default, and the supplied proof still does not show a live Telegram reaction update waking the routed session.

## What This Changes

The branch updates Telegram reaction handling, docs, tests, and changelog so accepted standard/custom emoji reactions include stable reaction keys and request an immediate heartbeat after enqueue.

## Best Possible Solution

Use a shared notificationWake.reactions policy that preserves queued-only behavior by default and only wakes Telegram reactions when the operator explicitly opts in, with live Telegram reaction proof before merge.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reaction system events and returns without requesting a heartbeat. I did not run a live Telegram reaction update in this read-only review.

## Solution Assessment

No: the direct unconditional wake is not the safest product shape because it changes existing reactionNotifications behavior for upgrades. The safer path is a shared opt-in wake policy that keeps queue as the default.
