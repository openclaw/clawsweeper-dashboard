- `P2`: This is a normal-priority Telegram workflow/config change with real but limited routing and notification impact.

## Summary

Keep open: the implementation direction is useful, but this PR is not merge-ready because the new policy precedence conflicts with the documented per-agent override behavior and the PR still lacks after-fix live Telegram reaction proof.

## What This Changes

The PR adds `notificationWake.reactions` config and SDK plumbing, routes Telegram reaction system events through a shared wake helper with stable standard/custom emoji keys, updates heartbeat gating, docs, generated metadata, and tests.

## Best Possible Solution

Align the policy precedence with the documented per-agent override contract or explicitly document/test channel-owned precedence, choose the canonical shared policy branch, and require redacted live Telegram proof before merging.

## Reproduction Assessment

Yes at source level: current main enqueues accepted Telegram reaction events and returns without a heartbeat wake, and the PR precedence issue is visible from the resolver order versus added config help text. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is: the queue-by-default wake policy is a maintainable direction, but precedence, live transport proof, and overlap with the shared-policy branch need to be settled before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Let routed agent overrides win:** `src/infra/notification-system-events.ts:158-162`
  - body: The new help text presents `agents.list[].notificationWake` as a per-agent override for routed notification events, but this resolver checks `channelConfig?.notificationWake` before `resolveAgentNotificationWake`. With `channels.telegram.notificationWake.reactions = "wake"` and a routed agent set to `"queue"` or `"off"`, the channel value wins, so that agent cannot keep noisy reactions passive or suppressed as documented. Please either move routed-agent policy ahead of channel-level policy, or explicitly document and test that channel policy is intended to dominate agent policy.
  - confidence: 0.82
