## Summary

Keep open: current main still treats accepted Telegram reactions as passive system events, and the PR’s source-level fix looks coherent, but merge should wait for maintainer approval of the default wake semantics and proof that a real Telegram reaction update wakes the routed session.

## What This Changes

The PR changes Telegram reaction handling to emit stable standard/custom reaction keys, mark reaction events untrusted, and request an immediate heartbeat after a successful accepted reaction enqueue.

## Best Possible Solution

Land a Telegram-owned patch that wakes only after accepted and deduplicated reaction events, keeps reaction content untrusted, and explicitly resolves default-versus-opt-in wake semantics with live reaction proof.

## Reproduction Assessment

Yes at source level: current main’s Telegram `message_reaction` handler enqueues accepted emoji reaction events and does not request a heartbeat wake. I did not run a live Telegram reaction in this read-only review.

## Solution Assessment

Mostly yes at code level: using the existing heartbeat wake path after successful `enqueueSystemEvent` is narrower than synthesizing a Telegram message turn. The remaining decision is whether existing `reactionNotifications` should become wake-triggering by default or whether wake needs an explicit opt-in mode.
