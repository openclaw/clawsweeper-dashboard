## Label Justifications

- `P2`: This is a normal-priority Telegram workflow improvement with limited channel scope but real upgrade and proof questions.
- `merge-risk: 🚨 session-state`: The PR can start new agent turns from reaction-only events for existing sessions that previously only queued passive context.
- `merge-risk: 🚨 message-delivery`: Changing accepted reactions into immediate wakes can cause new outbound replies or agent processing from a user action that previously produced no turn.
- `merge-risk: 🚨 security-boundary`: Reaction-only events from allowed Telegram senders become agent-turn triggers without an explicit wake policy, which broadens the activation boundary even though the event is marked untrusted.

## Summary

Keep open: current main still leaves Telegram reaction events as queued context only, but this PR changes existing reaction-notification setups into immediate agent-turn triggers without an opt-in wake policy. A maintainer-owned shared policy PR now appears to be the safer canonical direction, and this branch still needs real reaction-only wake proof before merge: https://github.com/openclaw/openclaw/pull/83611

## What This Changes

The PR changes Telegram reaction handling to emit stable standard/custom reaction keys, request a heartbeat after successful reaction event enqueue, update Telegram tests, and document the new behavior.

## Best Possible Solution

Route reaction system events through the shared notification wake policy, or an equivalent Telegram-owned opt-in, so queued remains the upgrade-safe default while stable reaction keys, untrusted metadata, and wake behavior are tested and documented.

## Reproduction Assessment

Yes at source level: current main registers `message_reaction`, passes existing gates, enqueues a system event, and returns without `requestHeartbeat`. I did not run a live Telegram reaction in this read-only review.

## Solution Assessment

No as written: stable reaction keys and untrusted event metadata are useful, but unconditional wake is not the safest upgrade shape. The better path is an explicit notification wake policy with queued default and wake opt-in coverage.
