## Summary

Keep open: current main still preserves structured Gateway `errorKind` values only up to the ACP translator, where every non-refusal error is returned to ACP clients as `end_turn`; fixing this safely needs an ACP-visible contract decision rather than a narrow translator-only patch.

## What This Changes

Preserve structured ACP-visible backend error classifications instead of collapsing non-refusal ACP error events to normal turn completion.

## Best Possible Solution

Define a documented additive ACP error-classification surface that preserves Gateway `errorKind` or equivalent failed-turn codes without inventing unsupported `StopReason` values, then update translator/runtime docs and focused tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with `errorKind: "timeout"` reaches `src/acp/translator.ts`, and the focused test currently asserts the ACP prompt resolves with `stopReason: "end_turn"`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but adding unsupported `StopReason` literals would not be the maintainable fix.
