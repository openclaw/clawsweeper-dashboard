## Summary

Current main partially preserves gateway chat error classification, but the ACP translator still collapses every non-refusal error to `end_turn`; this remains real work and is not a duplicate of the narrower Anthropic rate-limit copy issue.

## What This Changes

Request to preserve structured ACP backend error classes through the ACP translator instead of returning `end_turn` for all non-refusal errors.

## Best Possible Solution

Define a documented ACP bridge error-classification contract, carry the selected kind through runtime/chat/translator boundaries, and add focused tests for refusal plus transient failure classes without regressing existing ACP stop-reason compatibility.

## Reproduction Assessment

Yes. Source inspection and existing targeted tests show `errorKind: "timeout"` currently resolves to `end_turn`, while only `refusal` is preserved at the ACP translator boundary.

## Solution Assessment

Unclear until maintainers choose the ACP-facing contract. A narrow code change is possible after that decision, but mapping non-refusal errors into ad hoc stopReason strings would be less maintainable than a documented protocol/runtime extension.
