## Summary

Keep open: current main and the latest release still collapse non-refusal ACP error kinds to `end_turn`, and https://github.com/openclaw/openclaw/pull/91603 is an open linked implementation candidate that should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error classes instead of collapsing all ACP error states into normal end-of-turn completion.

## Best Possible Solution

Land a refreshed version of https://github.com/openclaw/openclaw/pull/91603, or replace it with a narrower ACP translator fix that preserves structured error information without conflicting client-visible error semantics.

## Reproduction Assessment

Yes, by source inspection: current main still maps non-refusal ACP `state: "error"` events to `{ stopReason: "end_turn" }`, and focused tests assert that for `timeout` and unknown kinds. I did not run a live ACP client scenario in this read-only review.

## Solution Assessment

No, closing this issue now is not the best path because the linked implementation PR is still open and unresolved. The maintainable solution is to settle the ACP prompt error contract and then land or replace that candidate fix.
