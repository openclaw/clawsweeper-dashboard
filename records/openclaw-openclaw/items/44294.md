## Label Justifications

- `impact:session-state`: ACP prompt responses currently represent some failed backend turns as normal end-of-turn state.
- `impact:auth-provider`: The requested classification includes provider/backend failures such as rate limits, refusals, and server/runtime errors.

## Summary

Keep this open: current main and the latest release only preserve the ACP refusal case, while timeout/rate-limit-style backend failures still collapse to normal `end_turn`; the remaining work needs an ACP-facing error contract decision rather than cleanup closure.

## What This Changes

The issue requests preserving structured ACP backend error classes so clients can distinguish failures like refusals, timeouts, rate limits, and server errors from normal end-of-turn completion.

## Best Possible Solution

Define the supported ACP-facing failure contract, then map gateway/backend error classes through that contract without misreporting operational failures as normal completion.

## Reproduction Assessment

Yes, from source inspection: an ACP chat error with `errorKind: "timeout"` still resolves as `{ stopReason: "end_turn" }`, and the colocated test asserts that behavior. I did not run tests because this review is read-only and the source/test path is sufficient to prove the current behavior.

## Solution Assessment

No, the current refusal-only mapping is not the full requested solution. The best path is contract-first: decide how ACP clients should receive non-refusal backend failures, then add focused translator tests for that representation.
