## Summary

Keep this open: current main and the latest release still collapse non-refusal ACP backend errors to normal `end_turn`, and the linked open PR is an implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Preserve structured ACP backend error classifications so clients can distinguish operational failures and refusals from normal end-of-turn behavior.

## Best Possible Solution

Define the ACP-facing failure contract, then land or replace the linked PR with focused translator coverage that preserves backend error classes without misreporting operational failures as normal completion.

## Reproduction Assessment

Yes, source-level: an ACP chat error with `errorKind: "timeout"` still resolves as `{ stopReason: "end_turn" }`, and the focused test asserts that behavior. I did not run tests because this review is read-only and the source/test path is sufficient to prove current behavior.

## Solution Assessment

No, current main is not the full solution. The best path is contract-first: decide how ACP clients should receive non-refusal backend failures, then land a focused translator change and tests.
