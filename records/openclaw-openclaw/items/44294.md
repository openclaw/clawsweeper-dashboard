## Summary

Keep open. Current main still only preserves the ACP-compatible `refusal` distinction; `timeout`, `rate_limit`, `context_length`, and `unknown` gateway `errorKind` values still resolve as plain `end_turn` with no `PromptResponse._meta` carry-through, matching the prior ClawSweeper review rather than resolving it.

## What This Changes

Preserve structured gateway backend `errorKind` values in ACP prompt responses instead of exposing non-refusal backend failures only as normal `end_turn` completions.

## Best Possible Solution

Keep Gateway `errorKind` as-is, preserve ACP enum compatibility, and expose the backend error class through documented `PromptResponse._meta` with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an ACP chat event with `state: "error"` and `errorKind: "timeout"`; the current harness expects `{ stopReason: "end_turn" }`, and `finishPrompt` has no `_meta` path.

## Solution Assessment

Yes. The narrow maintainable solution is to keep `refusal` mapped to ACP's standard refusal stop reason, keep other backend failures on standard `end_turn`, and carry the structured class in ACP `_meta` because the pinned SDK does not allow custom stop reasons.
