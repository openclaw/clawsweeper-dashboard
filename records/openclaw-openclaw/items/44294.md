## Summary

Keep open. Current main still reads gateway `errorKind` on ACP chat error events but only preserves the `refusal` distinction as an ACP stop reason; non-refusal classes still resolve as plain `end_turn` with no response metadata, so the reported client-visible gap remains valid and narrow.

## What This Changes

The issue asks the ACP translator to preserve structured backend error classes so clients can distinguish refusals, timeouts, rate limits, and server/context errors instead of treating every error like normal end-of-turn completion.

## Best Possible Solution

Add an ACP-compatible projection for gateway error classification: keep stop reasons within the SDK union, preserve `refusal` as `stopReason: "refusal"`, carry non-refusal backend classes in a documented `PromptResponse._meta` field, and update focused ACP tests plus user-facing docs/changelog.
