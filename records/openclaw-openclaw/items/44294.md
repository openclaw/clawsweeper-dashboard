## Label Justifications

- `P2`: This is a normal-priority ACP error-signaling gap with clear source proof but limited blast radius and an API/product decision pending.
- `impact:session-state`: The affected value is the ACP prompt terminal stop reason, which clients use to interpret turn/session completion state.
- `impact:auth-provider`: The lost classifications include provider/backend failures such as rate limits and timeouts that influence retry and provider-handling behavior.

## Summary

Keep open: current main still maps non-refusal Gateway ACP errorKind values such as timeout and rate_limit to ACP end_turn, and the remaining fix needs a maintainer decision on the ACP-visible error contract rather than a narrow code-only cleanup.

## What This Changes

The issue asks ACP to expose structured backend error classes such as refusal, timeout, rate_limit, and server_error instead of reporting non-refusal errors as end_turn.

## Best Possible Solution

Define a documented additive ACP error-classification surface that carries Gateway errorKind or equivalent error codes alongside compatible StopReason values, then update translator/runtime schema docs and focused tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with errorKind: "timeout" reaches the ACP translator and the current regression test asserts the prompt resolves with stopReason: "end_turn".

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported StopReason values is not the narrow maintainable fix.
