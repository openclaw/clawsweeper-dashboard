## Summary

Keep open: current main and the latest v2026.5.22 release still preserve only `refusal` and collapse other structured Gateway error kinds to ACP `stopReason: "end_turn"`; the durable fix needs a maintainer ACP contract decision rather than a translator-only literal change.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error classifications so ACP clients can distinguish refusals, timeouts, rate limits, and server failures from normal end-of-turn completion.

## Best Possible Solution

Define a documented additive ACP error-classification or failed-turn surface that carries Gateway `errorKind` without inventing unsupported `StopReason` values, then update translator/runtime docs and targeted tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with `errorKind: "timeout"` reaches `src/acp/translator.ts`, and the focused test currently asserts the ACP prompt resolves with `stopReason: "end_turn"`; tests were not run because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but emitting new `StopReason` literals such as `timeout` or `rate_limit` would not be compatible with the pinned ACP SDK contract.
