## Summary

Keep open: current main and latest release still collapse non-refusal ACP backend error kinds into normal `end_turn`, and the linked open implementation PR uses closing syntax for this issue. The remaining work is maintainer review of the client-visible ACP error contract, not conservative cleanup closure.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error classifications so clients can distinguish refusals, timeouts, rate limits, context length, and server errors instead of receiving normal `end_turn` for every non-refusal error.

## Best Possible Solution

Land a refreshed ACP fix through the linked PR or a narrow replacement after maintainers choose a stable structured error contract, with focused tests and real ACP client proof.

## Reproduction Assessment

Yes. From source, ACP `state: "error"` currently resolves non-refusal `errorKind` values as `end_turn`, and adjacent tests assert that behavior for `timeout` and unknown kinds.

## Solution Assessment

Unclear. Preserving richer error kinds is directionally useful, but the exact ACP client contract is compatibility-sensitive and the open implementation PR needs maintainer review before changing rejection/stop behavior.
