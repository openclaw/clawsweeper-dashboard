## Summary

Keep open: current main and the latest stable release still map non-refusal ACP backend errors to normal `end_turn` completion, and the open linked PR is the active implementation candidate that uses closing syntax for this issue. The remaining work is maintainer review of the client-visible ACP error contract, not cleanup closure.

## What This Changes

The issue requests preserving structured ACP backend error classes so clients can distinguish refusals, timeouts, rate limits, server errors, and similar failures from normal end-of-turn completion.

## Best Possible Solution

Use a documented ACP-compatible channel for backend error classes, keep the existing refusal and token-limit semantics intentional, and land the linked PR only after focused tests and real ACP client proof support the chosen contract.

## Reproduction Assessment

Yes, at source level: current main maps non-refusal ACP backend errors to end_turn, and the focused tests assert that timeout and unknown error kinds resolve that way. I did not run a live ACP client, but the translator path and tests are enough to prove the current mapping.

## Solution Assessment

Unclear: preserving structured error data is the right direction, but maintainers still need to choose the stable ACP/JSON-RPC error contract before the linked PR is the best fix. The narrowest path is to finish review on the existing linked PR rather than queue a second implementation.
