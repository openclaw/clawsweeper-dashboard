## Label Justifications

- `P2`: This is a normal-priority ACP error-signaling gap with clear source proof but limited blast radius and a protocol/product decision pending.
- `impact:session-state`: The affected value is the ACP prompt terminal result, which clients use to interpret turn and session completion state.
- `impact:auth-provider`: The lost classifications include provider/backend failure categories such as rate limits and timeouts that influence retry and provider handling.

## Summary

Keep open: current main still only maps ACP `errorKind: "refusal"` to `stopReason: "refusal"`; timeout, rate_limit, context_length, and unknown Gateway error kinds still collapse to `end_turn`, and fixing that safely needs a maintainer decision on the ACP-visible error contract.

## What This Changes

Preserve structured ACP backend error kinds so clients can distinguish backend failures from normal ACP end-of-turn completion.

## Best Possible Solution

Define a documented additive ACP error-classification surface that carries Gateway `errorKind` or equivalent error codes alongside compatible ACP stop reasons or failed-turn data, then update translator/runtime schema docs and focused tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with `errorKind: "timeout"` reaches the ACP translator and the current focused test asserts the prompt resolves with `stopReason: "end_turn"`; I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported `StopReason` values would not be the narrow maintainable fix.
