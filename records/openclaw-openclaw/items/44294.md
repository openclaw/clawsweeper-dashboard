## Summary

This should stay open: current main still preserves Gateway `errorKind` only up to the ACP translator, where non-`refusal` backend errors resolve as normal `end_turn`. The request is valid, but the remaining work needs an ACP-visible contract decision because the pinned ACP SDK `StopReason` union does not include timeout, rate-limit, context-length, or server-error stop reasons.

## What This Changes

Request to preserve structured ACP backend error classifications so clients can distinguish refusals, timeouts, rate limits, and server failures from normal end-of-turn completion.

## Best Possible Solution

Define a documented additive ACP error-classification surface that carries Gateway `errorKind` without breaking existing `StopReason` compatibility, then update translator/runtime schema docs and focused tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Gateway chat `state: "error"` event with `errorKind: "timeout"` reaches the ACP translator, and the current targeted test asserts the prompt resolves as `end_turn`.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported `StopReason` values is not the narrow maintainable fix.
