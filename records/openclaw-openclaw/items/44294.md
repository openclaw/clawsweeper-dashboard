## Summary

Keep open: current main and the latest release still collapse non-refusal ACP backend error states into normal `end_turn` completion, and the linked open implementation at https://github.com/openclaw/openclaw/pull/91603 uses closing syntax for this issue. The remaining work is maintainer review of the client-visible ACP error contract, not cleanup closure.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error classes so ACP clients can distinguish refusals, timeouts, rate limits, context length, and server failures from normal end-of-turn completion.

## Best Possible Solution

Land a refreshed ACP fix through the linked PR or a narrow replacement after maintainers choose a non-conflicting structured error contract, then close this issue with source, tests, release, and real ACP client proof.

## Reproduction Assessment

Yes, at source level: current main maps non-refusal ACP backend error events to `end_turn`, and the focused tests assert that behavior for `timeout` and unknown `errorKind`. I did not run a live ACP client, but the translator path and tests are enough to reproduce the current mapping from source.

## Solution Assessment

Unclear: preserving structured error data is the right direction, but maintainers still need to choose the stable ACP/JSON-RPC error contract before the linked PR is the best fix. The narrowest path is to finish review on the existing linked PR rather than queue a second implementation.
