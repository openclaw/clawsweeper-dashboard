## Label Justifications

- `P2`: This is a bounded ACP correctness problem that harms retry and UX decisions but does not make the core runtime unusable.
- `impact:session-state`: A failed ACP turn is currently reported as normal terminal completion, obscuring the actual run outcome.
- `impact:auth-provider`: Provider and backend failure categories such as refusal and rate limiting are flattened at the ACP translation boundary.

## Summary

Current main and v2026.6.5 still collapse timeout, rate-limit, and unknown ACP backend failures into normal `end_turn` completion. The linked open PR addresses the real gap, but its compatibility-sensitive JSON-RPC error semantics need maintainer agreement and real-client proof before merge.

## What This Changes

Preserve structured ACP backend failure kinds so clients can distinguish operational errors from normal turn completion.

## Best Possible Solution

Use the SDK-supported request-error data channel with a documented OpenClaw error-kind shape, retain explicit refusal and context-length terminal mappings, and add focused client compatibility proof before landing the linked PR.

## Reproduction Assessment

Yes at source level: current main and its tests directly show timeout and unknown backend errors resolving as `end_turn`. A live ACP client run was not performed, but it is not needed to establish the present translator mapping.

## Solution Assessment

Unclear: the linked PR changes the correct owner boundary and the SDK supports structured error data, but maintainers still need to choose stable error-code semantics and verify existing client behavior before calling it the best fix.
