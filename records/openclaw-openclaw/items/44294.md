## Label Justifications

- `P2`: This is a normal-priority ACP error-signaling gap with clear source proof, limited blast radius, and a pending protocol/product decision.
- `impact:session-state`: The affected value is the ACP prompt terminal result that clients use to interpret turn and session completion state.
- `impact:auth-provider`: The lost classifications include provider/backend failure categories such as rate limits and timeouts that influence retry and provider handling.

## Summary

Keep open: Gateway-side `errorKind` classification is present, but ACP clients still receive `end_turn` for non-refusal backend errors, and the pinned ACP SDK stop-reason contract has no timeout/rate-limit/server-error values to map to safely.

## What This Changes

Request ACP to preserve backend error classifications so clients can distinguish refusals, timeouts, rate limits, and server failures instead of receiving normal `end_turn` for most error states.

## Best Possible Solution

Define a documented additive ACP error-classification surface that preserves Gateway `errorKind` or equivalent failed-turn codes without inventing unsupported `StopReason` literals, then update translator/runtime docs and focused tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with `errorKind: "timeout"` reaches `src/acp/translator.ts`, and the focused test currently asserts the ACP prompt resolves with `stopReason: "end_turn"`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but adding unsupported `StopReason` literals would not be the maintainable fix.
