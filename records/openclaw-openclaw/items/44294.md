## Summary

This should stay open: current main and the latest release still map only ACP/Gateway `errorKind: "refusal"` to ACP `refusal`, while timeout, rate-limit, context-length, and unknown errors still resolve as normal `end_turn`. The remaining work is valid, but it needs an ACP-facing contract decision before implementation.

## What This Changes

Preserve backend error classifications such as timeout and rate-limit across the ACP translator instead of resolving all non-refusal error events as `end_turn`.

## Best Possible Solution

Define an additive, documented ACP bridge error-classification surface that carries Gateway `errorKind` without breaking existing ACP `stopReason` compatibility, then cover refusal and transient failure classes with focused tests.

## Reproduction Assessment

Yes. From source inspection, a Gateway chat `state: "error"` event with `errorKind: "timeout"` flows through the ACP translator and the current test asserts it resolves to `end_turn`; I did not execute tests in this read-only review.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported `stopReason` literals would be less maintainable than a documented metadata or runtime-contract extension.
