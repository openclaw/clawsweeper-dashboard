## Summary

Keep open. Current main still preserves only the ACP-compatible `refusal` distinction, while `timeout`, `rate_limit`, `context_length`, and `unknown` gateway `errorKind` values resolve as plain `end_turn` with no `PromptResponse._meta` carry-through.

## What This Changes

Request that ACP backend error classifications be preserved for clients instead of collapsing all non-refusal ACP error events into `end_turn`.

## Best Possible Solution

Keep the Gateway schema as-is, map `refusal` to ACP `stopReason: "refusal"`, keep other backend failures on standard `end_turn`, and expose the structured backend class through documented `PromptResponse._meta` with focused regression coverage.

## Reproduction Assessment

Yes. The current test harness can reproduce the behavior by sending an ACP chat event with `state: "error"` and `errorKind: "timeout"`; it resolves as `{ stopReason: "end_turn" }` and `finishPrompt` has no `_meta` path.

## Solution Assessment

Yes. A narrow ACP bridge change is the best fix because the pinned ACP SDK does not allow custom stop reasons, while its `PromptResponse._meta` field is the compatible extension point.
