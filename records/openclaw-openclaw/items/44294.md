## Summary

Keep open. Current main still preserves only the ACP-compatible `refusal` distinction; `timeout`, `rate_limit`, `context_length`, and `unknown` Gateway error kinds still resolve as plain `end_turn` with no `PromptResponse._meta` carry-through.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error kinds instead of flattening all non-refusal errors to `end_turn`.

## Best Possible Solution

Keep standard ACP stop reasons, keep `refusal` mapped to ACP `refusal`, and expose the Gateway backend error class through a documented `PromptResponse._meta` field with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an ACP chat event with `state: "error"` and `errorKind: "timeout"`; current main resolves `{ stopReason: "end_turn" }` and has no `_meta` path in `finishPrompt`.

## Solution Assessment

Yes. The narrow maintainable solution is to keep ACP enum-compatible stop reasons and carry the structured backend class in `_meta`, because the pinned ACP SDK does not allow `timeout` or `rate_limit` as stop reasons.

## Repair Work Prompt

Fix the ACP bridge gap for https://github.com/openclaw/openclaw/issues/44294. Preserve standard ACP `StopReason` values: `errorKind: "refusal"` should still produce `stopReason: "refusal"`, and other Gateway error kinds should keep ACP-compatible `stopReason: "end_turn"`. Add a documented `PromptResponse._meta` carry-through for Gateway `errorKind` values (`timeout`, `rate_limit`, `context_length`, `unknown`, and likely `refusal` too if useful) so ACP clients can distinguish backend failures from normal completion. Update focused tests in `src/acp/translator.error-kind.test.ts` and any affected stop-reason coverage; do not add custom ACP stop reasons or broaden Gateway protocol enums unless a current enum is missing. Update `docs/cli/acp.md` and `CHANGELOG.md` because this is user/client-visible bridge behavior.
