- `impact:session-state`: The issue is about ACP prompt terminal state being reported as normal completion when backend failure state should remain distinguishable.

## Summary

Keep open: current main and the latest release still collapse non-refusal ACP gateway error kinds such as `timeout` and `unknown` to `end_turn`, and the open linked implementation PR is not merged yet. The remaining work is real, but it is compatibility-sensitive because it changes the ACP `session/prompt` result contract.

## What This Changes

Request structured ACP backend error classification so ACP clients can distinguish refusals, context-length errors, timeouts, rate limits, and server failures instead of treating all non-refusal errors as `end_turn`.

## Best Possible Solution

Land a refreshed ACP contract fix through https://github.com/openclaw/openclaw/pull/91603 or a narrow replacement, preserving gateway `errorKind` for ACP clients with non-conflicting codes or data fields, focused tests, and real ACP client proof.

## Reproduction Assessment

Yes, by source inspection: current main and `v2026.6.6` still map non-refusal ACP gateway error events to `end_turn`, and the adjacent test asserts that for `timeout` and unknown kinds. I did not run a live ACP client scenario in this read-only review.

## Solution Assessment

Not yet. The direction is valid, but the best solution is a maintainer-approved ACP error contract, then landing the linked PR or a replacement after compatibility, error-code, and real-behavior proof concerns are resolved.
