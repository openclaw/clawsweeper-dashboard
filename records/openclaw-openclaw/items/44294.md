## Summary

This should stay open: current main still preserves only `refusal` at the ACP translator boundary, while timeout, rate-limit, context-length, and unknown backend errors still resolve to normal `end_turn` semantics. The remaining work is real, but it needs a maintainer decision on the ACP-facing error contract before automation changes protocol-visible behavior.

## What This Changes

The issue asks OpenClaw to preserve structured ACP backend error kinds instead of mapping non-refusal backend failures to normal end-of-turn behavior.

## Best Possible Solution

Define an additive, documented ACP bridge error-classification surface, carry Gateway `errorKind` through that surface, and add focused translator/runtime tests for refusal plus transient failure classes without breaking existing stopReason compatibility.

## Reproduction Assessment

Yes, from source inspection: a Gateway chat `state: "error"` event with `errorKind: "timeout"` flows through the ACP translator and the current test asserts it resolves to `end_turn`. I did not execute tests in this read-only review.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving non-refusal backend classes is the right direction, but inventing unsupported stopReason literals would be less maintainable than a documented protocol/runtime extension or metadata field.
