## Summary

This should stay open: current main and the latest v2026.5.12 release still preserve Gateway error classes only up to the ACP translator, where non-refusal error kinds resolve as normal `end_turn`. The report is valid, but the remaining change needs an ACP-facing contract decision before implementation.

## What This Changes

Preserve structured Gateway/ACP backend error classes in the ACP bridge instead of treating non-refusal error events as normal `end_turn` completions.

## Best Possible Solution

Define a documented, additive ACP error-classification surface that carries Gateway `errorKind` without breaking existing `stopReason` compatibility, then cover refusal and transient failure classes with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Gateway chat `state: "error"` event with `errorKind: "timeout"` reaches the ACP translator, and the current test asserts it resolves to `end_turn`; I did not execute tests in this read-only review.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported `StopReason` literals would be less maintainable than an additive metadata or runtime-contract extension.
