- `impact:auth-provider`: The lost classifications include provider/backend failure categories such as rate limits and timeouts that influence retry and provider handling.

## Summary

Keep open: current main and v2026.5.20 still collapse non-refusal Gateway `errorKind` values to ACP `stopReason: "end_turn"`, and the safe fix needs an ACP-visible contract decision rather than a translator-only literal change.

## What This Changes

Request to add or consume ACP-visible structured backend error classification so timeout, rate-limit, server, and refusal errors are not all exposed as normal end turns.

## Best Possible Solution

Define a documented additive ACP error-classification or failed-turn surface that carries Gateway `errorKind` without inventing unsupported `StopReason` values, then update translator/runtime docs and focused tests together.

## Reproduction Assessment

Yes, at source level. A Gateway chat error with `errorKind: "timeout"` reaches `src/acp/translator.ts`, and the focused test currently asserts the ACP prompt resolves with `stopReason: "end_turn"`; tests were not run because the review is read-only.

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but adding unsupported `StopReason` literals would not be the maintainable fix.
