## Summary

Keep open: current main and the latest release still collapse non-refusal Gateway errorKind values to ACP end_turn. The gap is source-reproducible, but the fix needs a maintainer decision on an ACP-visible error contract because the pinned ACP SDK StopReason union has no timeout, rate-limit, context-length, or server-error values.

## What This Changes

The issue asks ACP to preserve structured Gateway backend error classes instead of translating non-refusal errors to stopReason end_turn.

## Best Possible Solution

Define a documented additive ACP error-classification surface that carries Gateway errorKind alongside existing StopReason compatibility, then update translator/runtime schema docs and focused tests together.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.12 shows a Gateway chat error with errorKind: "timeout" reaches the ACP translator and resolves as stopReason: "end_turn".

## Solution Assessment

Unclear until maintainers choose the ACP-visible contract. Preserving backend error class information is the right direction, but inventing unsupported StopReason values is not the narrow maintainable fix.
