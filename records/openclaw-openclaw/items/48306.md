## Summary

The issue should stay open because current `main` still depends on a partly manual macOS release/appcast handoff and the narrower asset-verification fix is only present in an open, unmerged PR, while the split-version visibility and graceful failure parts remain unresolved.

## What This Changes

The issue requests a reliable macOS app update path where Sparkle appcast entries only advertise reachable app artifacts, recovery releases include usable macOS assets, and app/gateway version skew is visible or handled safely.

## Best Possible Solution

Land or supersede the release-asset guardrail work, then add any needed app/gateway version-skew visibility in the macOS app or gateway status path so a missing app update cannot become a silent operator trap.

## Reproduction Assessment

Yes for the release-pipeline gap at source level: current `main` still has a validation-only public macOS workflow and no merged post-publish asset check. No for the full operator split-brain loop, which would require a real macOS app/gateway setup and a bad release state to replay.
