## Summary

Keep open. The current appcast and latest stable release are healthy for v2026.4.29, so this is not an active latest-release outage, but current main still does not enforce the release/appcast asset invariant that this issue reports. The public macOS release workflow remains validation-only, appcast generation can still publish a derived GitHub release URL without reachability proof, `release-check.ts` validates Sparkle version shape rather than enclosure asset existence, and the directly related verifier PR #66213 is still open.

## What This Changes

The issue asks for the macOS updater/release path to stop advertising missing app assets and to make app/gateway version drift safer or more visible.

## Best Possible Solution

Keep this as the canonical macOS updater/release consistency issue until release-owner work lands and ships: verify `.zip`, `.dmg`, and `.dSYM.zip` assets before or immediately after stable publish, verify the appcast enclosure URL before `appcast.xml` advances, protect appcast ownership, and consider client-side messaging or fallback behavior for missing artifacts.

## Reproduction Assessment

Yes, for the reported release-state class: the historical v2026.3.13 and v2026.4.12 reports plus current source show the appcast path can be advanced from derived URLs without asset reachability proof. The active 404 does not reproduce on v2026.4.29 because the current release assets and appcast enclosure are reachable.

## Solution Assessment

No, current main is not yet the best end state: a healthy latest release and checklist docs reduce the immediate outage but do not make the release/appcast handoff atomic. The maintainable fix is an automated release guard plus appcast ownership, with updater UX improvements as a follow-up if needed.
