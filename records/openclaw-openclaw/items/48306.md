## Summary

Keep open: current main no longer advertises a dead latest macOS app asset, but the public repo still lacks the post-publish asset/appcast guard that would prevent this release failure class from recurring, and the focused repair PR is still open with review blockers.

## What This Changes

The issue asks OpenClaw to prevent macOS Sparkle appcasts/releases from advertising missing macOS assets and to reduce app/gateway version-skew fallout.

## Best Possible Solution

Land or replace the focused post-publish guard in https://github.com/openclaw/openclaw/pull/66213 so stable macOS releases verify assets and appcast state before clients see them; add client-side version-drift UX only if current builds still show noisy recovery behavior after the release guard is fixed.

## Reproduction Assessment

Yes for the release artifact failure class, though not from a live current-app run. The old affected releases still have zero assets and 404 zip URLs, while current source shows the public workflow delegates asset upload/appcast publication and the public checks do not verify enclosure reachability.

## Solution Assessment

No close yet. The narrow maintainable solution is the related release asset/appcast verifier or an equivalent release-owner-approved guard; a broader Sparkle rewrite should remain separate unless the guard does not stop recurrence.
