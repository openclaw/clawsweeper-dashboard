## Summary

Keep open. The PR targets a valid Feishu audio-duration bug, but the current head is conflicting, lacks sufficient real behavior proof, and still introduces blocking regressions around local SDK imports, download metadata, MIME-aware routing, incomplete duration coverage, and synchronous probing.

## What This Changes

The PR changes `extensions/feishu/src/media.ts` to probe local audio/video duration with `ffprobe` and pass that value into Feishu file uploads.

## Best Possible Solution

Replace this branch with a narrow Feishu media patch on current main that preserves existing routing and metadata helpers, computes a finite millisecond duration from the actual uploaded native or transcoded audio/video bytes, avoids blocking the gateway event loop, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection follows message-tool `media` / `asVoice` through Feishu action handling to `sendMediaFeishu`, where Opus/Ogg audio reaches `uploadFileFeishu` without `duration`; the Feishu upload docs say omitted duration prevents concrete duration display.

## Solution Assessment

No. This PR is not the best fix because it refactors established routing and metadata code, adds synchronous process execution in the send path, and only fills duration for local-path sends instead of the actual uploaded bytes.

  - confidence: 0.87

  - body: Calling `spawnSync("ffprobe", ...)` inside `sendMediaFeishu` blocks the Node event loop until `ffprobe` exits or times out. A slow or malformed local file can stall unrelated gateway message handling, so duration probing should use an async helper or an existing bounded media runtime path.
