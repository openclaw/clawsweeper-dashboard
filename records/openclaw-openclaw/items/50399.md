## Summary

Keep open: current main still has the Feishu duration gap, but this PR is not landable as-is because it regresses Feishu media metadata/routing, probes only a subset of supported send paths, and lacks after-fix OpenClaw behavior proof.

## What This Changes

The PR changes Feishu media sending to probe local audio/video duration with ffprobe and pass that value into Feishu file uploads while refactoring download metadata and media routing in `extensions/feishu/src/media.ts`.

## Best Possible Solution

Land a narrow Feishu media fix, likely as a refreshed or replacement PR, that computes finite duration from the uploaded native or transcoded audio/video bytes while preserving current metadata extraction and MIME-aware routing.

## Reproduction Assessment

Yes. Source inspection shows current main can forward `duration` in `uploadFileFeishu`, but `sendMediaFeishu` never supplies it for routed audio/video uploads; the Feishu upload contract explains why that produces missing duration display.

## Solution Assessment

No. The PR targets the right Feishu bug, but the patch is broader than needed, regresses established metadata/routing behavior, misses remote and buffer send paths, and blocks the event loop with synchronous ffprobe.

Full review comments:

- **[P2] Restore the Feishu-local runtime import:** `extensions/feishu/src/media.ts:5`
  - body: `openclaw/plugin-sdk/feishu` is not the Feishu plugin's current local runtime barrel and is not exposed by current package exports, so this branch will fail to resolve after rebase. Keep these types/helpers behind `../runtime-api.js` as main does.
  - confidence: 0.9
- **[P2] Preserve Feishu download metadata extraction:** `extensions/feishu/src/media.ts:148`
  - body: `downloadImageFeishu` now returns only `{ buffer }`, and the resource path only reads lowercase `response.headers`. That drops current `contentType`, `filename*`, SDK field, and `data` fallbacks used by inbound media saving/routing, so Feishu downloads can lose MIME and filenames silently.
  - confidence: 0.9
- **[P2] Keep routing on the MIME-aware helper:** `extensions/feishu/src/media.ts:483-487`
  - body: The inline routing treats every `video/*` response as Feishu `mp4` and treats MIME-identified Opus/Ogg audio with a generic filename as `stream`. That regresses current main's centralized MIME/extension contract and can send playable media as files or unsupported videos as media messages.
  - confidence: 0.88
- **[P2] Compute duration for the bytes being uploaded:** `extensions/feishu/src/media.ts:494`
  - body: The new duration code only runs for non-HTTP `mediaUrl`, but `sendMediaFeishu` also supports remote URLs and `mediaBuffer`. Those audio/video paths still upload without duration, so the reported missing-duration behavior remains for supported sends.
  - confidence: 0.87
- **[P2] Avoid synchronous ffprobe in the send path:** `extensions/feishu/src/media.ts:501-509`
  - body: Calling `spawnSync("ffprobe", ...)` inside `sendMediaFeishu` blocks the Node event loop until ffprobe exits or times out. A slow or malformed media file can stall unrelated gateway message handling, so duration probing should be async and bounded.
  - confidence: 0.84
