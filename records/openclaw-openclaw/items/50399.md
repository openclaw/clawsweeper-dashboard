## Summary

Keep this PR open, but it is not landable as-is: it targets a valid Feishu duration bug, yet the branch regresses Feishu metadata/routing behavior, lacks convincing OpenClaw real-behavior proof, and adds blocking `ffprobe` execution on the outbound send path.

## What This Changes

The PR changes `extensions/feishu/src/media.ts` to probe local audio/video duration with `ffprobe`, pass `duration` into Feishu file uploads, and refactor download metadata extraction plus outbound media routing.

## Best Possible Solution

Land a narrow Feishu plugin fix or replacement PR that computes a finite millisecond duration from the exact uploaded native or transcoded audio/video bytes and passes it through `uploadFileFeishu` while preserving existing download metadata and MIME-aware routing.

## Reproduction Assessment

Yes. Current main routes Feishu Ogg/Opus media to `uploadFileFeishu` without `duration`, and Feishu's upload contract says omitted duration prevents concrete duration display; I did not run a live Feishu UI send.

## Solution Assessment

No. The PR targets the right bug, but it is broader than necessary, regresses current metadata/routing behavior, misses supported remote/buffer send paths, and adds synchronous probing to the send path.

  - body: The replacement parser only reads lowercase `response.headers` and a plain `filename=` value. The current helper also handles header aliases, SDK/data fallbacks, `filename*`, and UTF-8 recovery, so inbound Feishu resources can lose MIME types and non-ASCII filenames.
  - confidence: 0.9
- **[P2] Keep routing on the MIME-aware helper:** `extensions/feishu/src/media.ts:483-487`
  - body: The inline routing maps every `video/*` response to Feishu `mp4` and maps MIME-identified Ogg/Opus audio with a generic filename to `stream`. That regresses current main's centralized MIME/extension contract and can send supported media with the wrong Feishu message type.
  - confidence: 0.88
- **[P2] Compute duration from the uploaded media bytes:** `extensions/feishu/src/media.ts:494`
  - body: The new duration code only runs for non-HTTP `mediaUrl`, but `sendMediaFeishu` also supports remote URLs, buffers, and now voice-intent transcoded buffers. Those supported audio/video paths would still upload without `duration`, leaving the reported bug unfixed there.
  - confidence: 0.86

  - body: `spawnSync("ffprobe", ...)` blocks the Node event loop until the process exits or times out. A slow or malformed caller-selected file can stall unrelated gateway message handling, so duration probing should be async and bounded.
