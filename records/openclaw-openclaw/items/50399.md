## Summary

Keep open. Current main still lacks automatic Feishu duration propagation on the normal media send paths, and this PR is not landable because it regresses Feishu routing/download metadata and adds direct synchronous ffprobe execution in the outbound path. The remaining user-facing bug is also tracked by #53798.

## What This Changes

The PR changes Feishu media upload handling to derive audio/video duration with ffprobe, pass it to file upload, and rewrites parts of Feishu download metadata and outbound media routing in `extensions/feishu/src/media.ts`.

## Best Possible Solution

Replace or substantially rewrite the PR as a small Feishu-only fix on current main: keep `extractFeishuDownloadMetadata`, `resolveFeishuOutboundMediaKind`, and `audioAsVoice` behavior; derive a millisecond duration after media normalization using the shared async ffprobe helper; pass it only for proven Feishu Opus/MP4 uploads; and add focused regression tests for duration propagation plus existing metadata/routing behavior.

## Reproduction Assessment

Yes. The current-main code path is reproducible by inspection and with a focused unit test: `sendMediaFeishu` uploads Opus/MP4 through `uploadFileFeishu` without `duration`, and both message-tool media and reply media route through that function.

## Solution Assessment

No. The PR targets the right API field, but the implementation is not the best fix because it rewrites stable routing/metadata code, probes only raw non-HTTP local paths, and uses synchronous direct process execution instead of the existing shared ffprobe helper.

Full review comments:

- **[P2] Route Feishu helper imports through the local barrel:** `extensions/feishu/src/media.ts:5`
  - body: This production file imports `openclaw/plugin-sdk/feishu` from inside the Feishu plugin. Current guardrails include Feishu in the local-barrel checks and reject this pattern, so the channel import guardrail test will fail; keep the type/runtime imports behind `../runtime-api.js` or another local barrel.
  - confidence: 0.9
- **[P2] Restore Feishu download metadata extraction:** `extensions/feishu/src/media.ts:148`
  - body: Returning only `{ buffer }` for image downloads and replacing the shared metadata helper with direct `response.headers[...]` parsing loses content type and filename metadata from alternate SDK shapes and RFC 5987 `filename*` headers. Inbound media saving relies on this metadata before falling back to MIME sniffing, so existing downloads can get wrong MIME types or filenames.
  - confidence: 0.9
- **[P2] Keep MIME-specific Feishu media routing:** `extensions/feishu/src/media.ts:483-487`
  - body: The new routing maps every `video/*` payload to Feishu `file_type: "mp4"`, while `audio/*` only becomes voice when the filename extension is Opus/Ogg. Current main only treats known Feishu-compatible video MIME types as media and accepts `audio/ogg`/`audio/opus` by MIME, so generic CDN URLs and unsupported videos will be sent with the wrong file/message type.
  - confidence: 0.88
- **[P2] Probe duration after media normalization:** `extensions/feishu/src/media.ts:494`
  - body: Duration probing only runs for non-HTTP `mediaUrl` strings. Remote URLs, buffer-backed sends, home-relative/file-url forms that normalize through `loadWebMedia`, and converted voice media can still upload Opus/MP4 without duration, leaving the reported Feishu duration display bug in place on supported send paths.
  - confidence: 0.86
- **[P2] Use the shared async ffprobe helper:** `extensions/feishu/src/media.ts:501`
  - body: Calling `spawnSync` inside `sendMediaFeishu` blocks the gateway event loop for up to the timeout on every probed attachment and bypasses the repo's trusted `runFfprobe`/system-binary resolution path. Use the existing async helper with argument arrays and normalized buffer/temp-file input instead.
  - confidence: 0.84
