## Summary

Keep open. The Feishu duration bug is valid, but this branch is not landable against current main because it imports a missing SDK subpath, regresses established Feishu media metadata/routing behavior, only probes duration for one supported input path, and still lacks convincing after-fix OpenClaw/Feishu proof.

## What This Changes

The PR changes Feishu media sending to probe local audio/video duration with ffprobe and pass it to file uploads, while also replacing current media-routing and download-metadata logic in `extensions/feishu/src/media.ts`.

## Best Possible Solution

Replace this with a narrow Feishu media patch that keeps the current routing and metadata helpers, computes a finite millisecond duration for native/transcoded audio and supported video uploads, adds focused regression tests, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes, source inspection plus the open related report gives a high-confidence path: Feishu `.opus`/`.ogg` routes to an audio upload while current `sendMediaFeishu` does not pass `duration`. I did not run a live Feishu UI send in this read-only review.

## Solution Assessment

No. The maintainable fix should preserve current Feishu routing/metadata contracts and add a duration helper plus tests; this PR rewrites those contracts and leaves supported media paths without duration.

  - confidence: 0.93

Current main routes those MIME types to Feishu `audio`, so this branch can still send playable voice media as a file and skip the new duration path.
  - confidence: 0.91
- **[P2] Compute duration for all supported media inputs:** `extensions/feishu/src/media.ts:494-495`
  - body: The duration probe only runs for non-HTTP `mediaUrl` values. `sendMediaFeishu` also supports `mediaBuffer` and remote URLs that have already been loaded into `buffer`; those OPUS/MP4 paths still upload without `duration`, so the reported Feishu duration symptom remains for supported callers.
