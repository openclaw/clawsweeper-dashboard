## Summary

Keep open. The Feishu duration bug is real and still tracked by the open canonical issue, but this PR is not landable against current main because it only partially supplies duration and regresses newer Feishu media routing/download handling.

## What This Changes

The PR modifies `extensions/feishu/src/media.ts` to probe local audio/video duration with `ffprobe` and pass that duration into Feishu file uploads.

## Best Possible Solution

Revise or replace this PR with a current-main Feishu-only patch that preserves centralized routing and metadata extraction, computes finite millisecond duration for supported voice/video uploads, and adds focused media tests.

## Reproduction Assessment

Yes, from source inspection. Current main routes Opus/Ogg media to Feishu audio upload but does not pass `duration`, and https://github.com/openclaw/openclaw/issues/53798 provides the matching message-tool reproduction path.

## Solution Assessment

No. The PR’s intent is right, but the current branch is not the best fix because it rewrites routing/metadata code, only handles raw local paths, and bypasses the shared ffprobe helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Populate duration for every supported upload source:** `extensions/feishu/src/media.ts:494`
  - body: The new duration logic only runs for a raw non-HTTP `mediaUrl`. `sendMediaFeishu` also accepts `mediaBuffer` and remote URLs loaded into buffers, so supported Opus/MP4 sends through those paths still upload without `duration` and keep the same Feishu UI duration problem.
  - confidence: 0.88
- **[P2] Keep MIME routing centralized:** `extensions/feishu/src/media.ts:483-487`
  - body: This branch replaces `resolveFeishuOutboundMediaKind` with inline checks that map any `video/*` to `mp4` and only extension-named audio to `opus`. That diverges from current main’s exact MIME routing and risks regressing generic URL audio/video handling; preserve the helper and extend it instead.
  - confidence: 0.82
- **[P2] Restore full download metadata extraction:** `extensions/feishu/src/media.ts:180-185`
  - body: The new parser only reads lowercase `response.headers` and simple `filename=...`, and image downloads still return only `{ buffer }`. Current main preserves content type, header aliases/data fields, and CJK/Latin-1 filename recovery, so this would regress inbound MIME and filename handling.
  - confidence: 0.86
- **[P2] Use the shared ffprobe runner:** `extensions/feishu/src/media.ts:501-509`
  - body: Calling `spawnSync("ffprobe")` directly runs through the process PATH and blocks the gateway event loop during outbound sends. Current media runtime already has trusted `execFile`-based `runFfprobe`; use that helper and feed the prepared media safely.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The patch introduces a new direct external-process execution path for `ffprobe` instead of the trusted shared media runtime helper.

Concerns:

- **[medium] Direct ffprobe PATH execution:** `extensions/feishu/src/media.ts:501`
  - body: `spawnSync("ffprobe", ...)` resolves the executable from the inherited PATH and bypasses OpenClaw’s trusted system-binary resolver used by `runFfprobe`, creating an avoidable local execution/supply-chain risk in the send path.
  - confidence: 0.74
