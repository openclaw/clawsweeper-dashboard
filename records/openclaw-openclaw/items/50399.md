## Summary

Keep open. The Feishu duration bug is valid and still not fully fixed on current main, but this PR is not merge-ready because it is stale against the current Feishu media path, weakens existing metadata/routing behavior, and introduces direct synchronous ffprobe execution instead of using the shared media helper.

## What This Changes

The PR modifies `extensions/feishu/src/media.ts` to probe local audio/video duration with ffprobe, pass `duration` to Feishu file uploads, and rewrite parts of Feishu download metadata extraction and outbound media routing.

## Best Possible Solution

Replace or substantially rewrite the PR as a small Feishu-only fix on current main: keep the existing metadata extraction, `resolveFeishuOutboundMediaKind`, `audioAsVoice`, and local runtime barrels; derive duration from the normalized buffer with the shared async ffprobe helper; pass it only for proven Feishu Opus/MP4 uploads; and add focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Current main is reproducible by source inspection and a focused unit test: send an Opus or MP4 through `sendMediaFeishu` and assert the mocked `im.file.create` payload lacks `duration`, while Feishu's own docs say omitting duration prevents duration display.

## Solution Assessment

No. The PR targets the right Feishu API field, but it is not the best solution because it rewrites stable routing/metadata code, misses buffer/remote/converted media paths, and runs ffprobe synchronously through PATH instead of the repository's trusted async helper.
