## Summary

Keep open. The PR targets a real Feishu audio-duration bug, but the current branch is not landable because it regresses existing Feishu media routing and download metadata, violates the current extension import guardrail, and only computes duration for a subset of supported sends.

## What This Changes

The PR changes `extensions/feishu/src/media.ts` to probe local audio/video duration with `ffprobe` and pass that value into Feishu file uploads.

## Best Possible Solution

Replace this branch with a narrow Feishu media patch on current main that keeps the existing routing and metadata helpers, computes a finite millisecond duration from the actual uploaded audio/video bytes, avoids blocking the gateway event loop, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call the current main Feishu `sendMediaFeishu` audio route and inspect that `client.im.file.create` receives no `duration`, while Feishu's upload contract says omitted duration prevents concrete duration display.

## Solution Assessment

No. This PR is not the best fix because it refactors established routing/metadata code, adds a synchronous process call in the send path, and only fills duration for local-path sends; a smaller current-main patch should preserve existing helpers and add duration coverage.

  - confidence: 0.9

  - body: Replacing `resolveFeishuOutboundMediaKind` with inline checks changes the MIME contract: generic `audio/opus` URLs without an `.opus`/`.ogg` name fall back to `stream`, while every `video/*` becomes Feishu `mp4`. That regresses the existing exact MIME/extension routing and can send playable voice notes as files or unsupported videos as MP4 media.
  - confidence: 0.88
- **[P2] Compute duration for the uploaded media bytes:** `extensions/feishu/src/media.ts:494`
  - body: The new duration code only runs for a non-HTTP `mediaUrl`, but `sendMediaFeishu` also supports `mediaBuffer` and remote URLs after `loadWebMedia`. Those supported audio/video paths still upload without duration, so the reported missing-duration behavior remains for non-local sends and any buffer-backed media.
  - confidence: 0.87
- **[P2] Move ffprobe off the synchronous send path:** `extensions/feishu/src/media.ts:501-510`
  - body: Calling `spawnSync("ffprobe", ...)` inside `sendMediaFeishu` blocks the Node event loop until `ffprobe` exits or hits the 10s timeout. A slow or malformed local file can stall unrelated gateway message handling, so duration probing should use an async helper or an existing media-runtime path with bounded behavior.
  - confidence: 0.84
