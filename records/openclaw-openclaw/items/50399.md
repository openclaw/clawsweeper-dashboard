## Summary

Keep open. The Feishu duration bug is real and still source-reproducible on current main, but this PR is not landable because it weakens current metadata/routing behavior, only covers local `mediaUrl` duration probing, blocks the event loop with synchronous `ffprobe`, and still lacks real OpenClaw behavior proof.

## What This Changes

The PR changes Feishu media sending to probe local audio/video duration with `ffprobe` and pass it into file uploads while also refactoring download metadata extraction and outbound media routing in `extensions/feishu/src/media.ts`.

## Best Possible Solution

Land a narrow current-main Feishu media patch, or a replacement PR, that computes a finite millisecond duration from the actual uploaded native or transcoded audio/video bytes and passes it through `uploadFileFeishu` while preserving existing download metadata and MIME-aware routing.

## Reproduction Assessment

Yes. Source inspection shows `sendMediaFeishu` routes native Opus/Ogg audio to `uploadFileFeishu` without `duration`, and Feishu's upload contract explains why that omits concrete duration display; I did not perform a live Feishu UI send.

## Solution Assessment

No. The PR targets the right bug, but the patch is broader than necessary, regresses established metadata/routing behavior, misses remote and buffer-backed sends, and uses synchronous probing in the outbound send path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restore the Feishu-local runtime import:** `extensions/feishu/src/media.ts:5`
  - body: `openclaw/plugin-sdk/feishu` is not exported by the current package and bypasses the Feishu plugin's local runtime barrel. Keep the type/helper import on the local Feishu runtime surface used by main so this resolves after rebase.
  - confidence: 0.9
- **[P2] Preserve Feishu download metadata extraction:** `extensions/feishu/src/media.ts:148`
  - body: `downloadImageFeishu` now returns only `{ buffer }`, and the resource path only reads lowercase `response.headers`. That drops current `contentType`, `filename*`, SDK field, and `data` fallbacks used by inbound media saving/routing, so downloads can lose MIME and filenames silently.
  - confidence: 0.9
- **[P2] Keep routing on the MIME-aware helper:** `extensions/feishu/src/media.ts:483-487`
  - body: The inline routing maps every `video/*` response to Feishu `mp4` and maps MIME-identified Opus/Ogg audio with a generic filename to `stream`. That regresses current main's centralized MIME/extension contract and can send playable media as files or unsupported videos as media messages.
  - confidence: 0.88
- **[P2] Compute duration for the uploaded bytes:** `extensions/feishu/src/media.ts:494`
  - body: The new duration code only runs for non-HTTP `mediaUrl`, but `sendMediaFeishu` also supports remote URLs and `mediaBuffer`. Those supported audio/video paths still upload without `duration`, so the reported missing-duration behavior remains there.
  - confidence: 0.87
- **[P2] Avoid synchronous ffprobe in the send path:** `extensions/feishu/src/media.ts:501-509`
  - body: Calling `spawnSync("ffprobe", ...)` inside `sendMediaFeishu` blocks the Node event loop until ffprobe exits or times out. A slow or malformed file can stall unrelated gateway message handling, so duration probing should be async and bounded.
  - confidence: 0.84
