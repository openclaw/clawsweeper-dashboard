## Summary

Keep open: this PR targets a real Feishu duration bug that current main still does not fully solve, but the branch is not landable as-is because it conflicts with newer Feishu media code, weakens metadata/routing behavior, adds synchronous ffprobe execution, and lacks branch-specific real behavior proof.

## What This Changes

The PR modifies `extensions/feishu/src/media.ts` to probe local audio/video duration with `ffprobe`, pass that duration to Feishu file uploads, and rewrite parts of Feishu media download metadata and outbound routing.

## Best Possible Solution

Land a narrow Feishu replacement patch that preserves current metadata and MIME-aware routing, computes a finite millisecond duration from the actual uploaded native or transcoded audio/video bytes, and covers the duration field with focused Feishu tests.

## Reproduction Assessment

Yes, from source and the Feishu API contract: current main sends Feishu audio/video uploads without `duration`, while the Feishu upload API says omitted duration prevents concrete duration display. I did not run a live Feishu UI send in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but it is broader than necessary, regresses current metadata/routing behavior, misses URL and buffer-backed audio/video duration paths, and adds synchronous probing to the send path.

Full review comments:

- **[P2] Restore the local Feishu runtime import:** `extensions/feishu/src/media.ts:5`
  - body: `openclaw/plugin-sdk/feishu` is not a current package export, and bundled extension source is guarded against importing its own plugin facade. Keep the current local/runtime SDK imports so the branch can pass contracts after rebase.
  - confidence: 0.9

  - body: `downloadImageFeishu` now returns only `{ buffer }`, dropping the current `contentType` extracted from Feishu response metadata. Callers that save or route downloaded images can silently lose the MIME hint and fall back to weaker detection.
  - confidence: 0.88
- **[P2] Restore the full Feishu resource metadata parser:** `extensions/feishu/src/media.ts:181-191`
  - body: The replacement parser only reads lowercase `response.headers` and plain `filename=`. Current main also handles header aliases, SDK/data fallbacks, `filename*`, and UTF-8 recovery, so inbound resources can lose MIME types and non-ASCII filenames.
  - confidence: 0.9
- **[P2] Keep outbound routing on the MIME-aware helper:** `extensions/feishu/src/media.ts:483-487`
  - body: The inline routing maps every `video/*` response to Feishu `mp4` and maps MIME-identified Ogg/Opus audio with a generic filename to `stream`. That regresses current main's centralized MIME/extension contract and can send supported media with the wrong Feishu message type.
  - confidence: 0.88
- **[P2] Compute duration for every uploaded media byte source:** `extensions/feishu/src/media.ts:494`
  - body: Duration probing only runs for a non-HTTP `mediaUrl`, but `sendMediaFeishu` also supports remote URLs, buffers, and voice-intent transcoded buffers. Those supported audio/video paths would still upload without `duration`, leaving the reported symptom unfixed there.
  - confidence: 0.86
- **[P2] Avoid blocking ffprobe on the gateway send path:** `extensions/feishu/src/media.ts:501-509`
  - body: `spawnSync("ffprobe", ...)` blocks the Node event loop until the process exits or times out. A slow or malformed caller-selected file can stall unrelated gateway message handling during outbound Feishu sends.
  - confidence: 0.84
