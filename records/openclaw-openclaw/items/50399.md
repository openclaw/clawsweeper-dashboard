## Summary

Keep open: the Feishu duration bug remains valid on main, but this PR is not landable because it regresses existing media metadata/routing paths and introduces a synchronous PATH-resolved ffprobe call.

## What This Changes

This PR changes Feishu media sending to probe local audio/video duration with ffprobe, pass `duration` to uploads, and rewrites download metadata extraction and outbound media routing in `extensions/feishu/src/media.ts`.

## Best Possible Solution

Apply a small Feishu-only patch on current main that preserves existing metadata extraction, MIME-aware routing, and `audioAsVoice`, then computes duration through the shared async ffprobe helper after media preparation for native Opus/MP4 uploads.

## Reproduction Assessment

Yes by source inspection: current main accepts and forwards `duration` in `uploadFileFeishu`, but `sendMediaFeishu` never supplies it for Opus/MP4 uploads. The Feishu API docs state omitted audio/video duration prevents displaying the specific file duration.

## Solution Assessment

No. The PR targets the right Feishu API field, but it is not the best implementation because it rewrites stable routing/metadata code, misses supported buffer/remote/transcoded media paths, and runs ffprobe synchronously through PATH instead of the existing trusted async helper.

  - body: Mapping any `video/*` payload to `mp4` and requiring audio extensions for Opus drops current MIME-aware routing. CDN-style `audio/ogg` without a useful filename can become a generic file, while non-MP4 videos can be sent down Feishu's MP4 media path.
  - confidence: 0.89

  - body: The duration probe only runs for raw local `mediaUrl` strings, so `mediaBuffer`, remote downloads, and `audioAsVoice` transcode output can still upload Opus/MP4 without `duration`. That leaves supported send paths with the reported missing-duration behavior.
  - confidence: 0.88
- **[P2] Use the shared async ffprobe helper:** `extensions/feishu/src/media.ts:501-509`
  - body: Calling `spawnSync("ffprobe", ...)` in outbound delivery blocks the gateway event loop and resolves through PATH. The repo already exposes `runFfprobe`, which uses trusted system-binary resolution and async `execFile`.
  - confidence: 0.85
