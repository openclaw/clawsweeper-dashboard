## Summary

Keep open. Current main still has no `feature-lab` CLI surface, and the proposed implementation is a useful small feature PR but still has source-backed blockers around service resolution, test isolation, and required public-surface docs/changelog.

## What This Changes

This PR adds a top-level `openclaw feature-lab` diagnostic command, lazy CLI registration, status implementation, and unit coverage for deployed SHA/service wiring.

## Best Possible Solution

Land a revised command that reuses the gateway service resolution/parsing contract, keeps tests isolated from real home service files, and adds the required CLI docs and changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: the PR reads only the default systemd unit path while current main supports `OPENCLAW_SYSTEMD_UNIT` and profile-derived service names, and the added test writes into the real home service directory.

## Solution Assessment

No. The feature direction is narrow, but the implementation should use `resolveSystemdUserUnitPath` or `readSystemdServiceExecStart`, inject temp paths or HOME for tests, and add docs/changelog before merge.
