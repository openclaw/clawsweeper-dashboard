## Summary

Keep open. Current main does not implement a feature-lab CLI command, and the PR is a real implementation attempt, but the proposed patch still has merge-blocking issues around systemd service resolution and test isolation plus required docs/changelog coverage.

## What This Changes

The PR adds an `openclaw feature-lab` CLI command, a status reader for local feature-lab deployment metadata, command registration/descriptors, and unit coverage for deployed SHA and service wiring.

## Best Possible Solution

Keep the diagnostic only if it is a documented core CLI surface that uses the existing gateway service-resolution contract and isolated tests; otherwise move the feature-lab status check into the feature-lab deployment tooling rather than adding a public command.

## Reproduction Assessment

Yes. With `OPENCLAW_PROFILE=work` or `OPENCLAW_SYSTEMD_UNIT=custom-unit`, current main expects a different user systemd unit path, while the PR still reads only the default `openclaw-gateway.service`; the test side effect is also directly visible from its `os.homedir()` writes.

## Solution Assessment

No. The narrowest maintainable fix is to reuse `resolveSystemdUserUnitPath` or the existing daemon service reader, inject filesystem/home/env paths for tests, and add the required docs and changelog entry.
