## Summary

Keep open. The PR is not obsolete on current main, but the proposed command is not merge-ready because it bypasses existing service-unit resolution, its test mutates the real user systemd directory, and the new user-facing CLI surface is missing release/docs follow-through.

## What This Changes

The PR adds a core `openclaw feature-lab` CLI command with descriptor/registration entries, a status reader for feature-lab root/install/systemd metadata, and unit coverage for deployed SHA and service wiring.

## Best Possible Solution

Keep the command only if it uses the existing gateway service resolution contract, keeps tests isolated from real user service files, and lands with the required changelog/docs updates; otherwise move the diagnostic into the feature-lab deployment tooling instead of the core CLI.

## Reproduction Assessment

Yes. With `OPENCLAW_PROFILE=work` or `OPENCLAW_SYSTEMD_UNIT=custom-unit`, current main expects a profile/custom systemd unit path, while the PR code still reads only the default `openclaw-gateway.service`; the test side effect is also directly visible from the added `os.homedir()` writes.

## Solution Assessment

No. The direction is narrow, but the implementation should reuse `resolveSystemdUserUnitPath(process.env)` or the same daemon service reader and inject filesystem/home paths for tests instead of duplicating service discovery and writing real home state.
