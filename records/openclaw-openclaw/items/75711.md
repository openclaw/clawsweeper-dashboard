## What This Changes

The PR adds a top-level `openclaw feature-lab` CLI command that reports feature-lab deployment paths, deployed SHA, systemd service command wiring, and JSON/text status output with unit coverage.

## Best Possible Solution

Land a revised command only after it reuses the existing daemon service resolution/reading contract, keeps filesystem and home paths injectable for tests, and adds the CLI reference page, docs index/nav entry, and active changelog line; otherwise keep this diagnostic in feature-lab deployment tooling.

## Reproduction Assessment

Yes. A source-level reproduction is enough: with `OPENCLAW_PROFILE=work` or `OPENCLAW_SYSTEMD_UNIT=custom-unit`, current main resolves a non-default systemd unit path while the PR reads only the default unit file; the test side effect is visible from its `os.homedir()` service-file write.

## Solution Assessment

No. The feature direction is narrow, but the current implementation bypasses the maintained service resolver and lacks test isolation plus required public-command docs/changelog; the safer path is to reuse `resolveSystemdUserUnitPath` or the daemon service reader and inject paths for tests.
