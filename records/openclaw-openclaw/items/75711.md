## Summary

Keep open: this is a plausible new CLI diagnostic, but the branch still misreads custom/profiled systemd units, writes tests into the real user service directory, lacks docs for a public command, and does not include inspectable after-fix output proof.

## What This Changes

The PR adds a top-level `openclaw feature-lab` command, lazy CLI registration, status implementation, and unit coverage for deployed SHA/service wiring.

## Best Possible Solution

Land only after the command reuses the existing gateway systemd resolver/reader, tests isolate all service files under temp state, public CLI docs are added, and real status output proof is attached.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the two blocking defects: the PR hardcodes the default systemd unit path while current main supports custom/profiled unit names, and its unit test writes into the real home service directory.

## Solution Assessment

No. The direction may be useful, but the implementation should reuse `resolveSystemdUserUnitPath` or `readSystemdServiceExecStart`, inject temp paths or HOME for tests, add CLI docs, and include real output proof before merge.
