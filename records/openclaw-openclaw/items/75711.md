## Summary

Keep open. Current main does not include a `feature-lab` command, and the latest provided PR diff still has blocking correctness and test-isolation issues: it bypasses the existing profile/custom systemd unit resolver and its unit test writes under the runner's real user service directory. The new public CLI surface also needs docs and changelog coverage before merge.

## What This Changes

The PR adds a core `openclaw feature-lab` CLI command that reports local feature-lab root/install metadata, deployed SHA, systemd service file/command, and whether the service points at the feature install, with unit coverage for SHA and service wiring.

## Best Possible Solution

Keep the command only if it follows the existing daemon service resolution contract, keeps tests fully isolated through injected home/service paths or temp HOME, and lands with a CLI docs page/index entry plus the required changelog line; otherwise keep the diagnostic inside feature-lab deployment tooling until the core CLI surface is explicitly accepted.

## Reproduction Assessment

Yes. A source-level reproduction is enough: with OPENCLAW_PROFILE=work or OPENCLAW_SYSTEMD_UNIT=custom-unit, current main resolves a non-default unit path, while the PR reads only openclaw-gateway.service; the real-home test side effect is directly visible from os.homedir() writes in the added test.

## Solution Assessment

No. The command direction is narrow, but the implementation should reuse resolveSystemdUserUnitPath or the daemon service reader and inject filesystem/home paths for tests before this is the maintainable solution.
