## Summary

Keep open: current main has useful mitigations, but normal session cleanup still does not reconcile unindexed transcript and trajectory files by default, so the reported disk bloat remains actionable.

## What This Changes

The issue requests automatic cleanup or reconciliation of session transcript and trajectory files left on disk but missing from sessions.json after gateway restarts.

## Best Possible Solution

Extend session maintenance so `openclaw sessions cleanup` can report and prune unreferenced session artifacts using existing retention rules, while preserving QMD and referenced-checkpoint safeguards.

## Reproduction Assessment

Yes from source inspection: with the default maintenance config, unindexed `.jsonl` and trajectory files are only detected by doctor or removed when a configured disk budget is exceeded. I did not run a live filesystem repro because this review is read-only.

## Solution Assessment

No, the current mitigations are not the best complete solution: doctor is interactive/manual and disk-budget cleanup is opt-in via configuration. The narrow maintainable fix is to make normal session cleanup reconcile unreferenced physical artifacts with retention-safe dry-run and enforce behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77608. Current main only removes unreferenced physical session artifacts when `session.maintenance.maxDiskBytes` is configured and exceeded, while doctor only interactively archives primary orphan transcripts. Extend normal session cleanup/maintenance so dry-run reports and enforce mode prunes unreferenced session artifacts under the agent sessions directory: primary `.jsonl` transcripts missing from `sessions.json`, orphan `.trajectory.jsonl` / `.trajectory-path.json` sidecars, and unreferenced compaction checkpoint transcripts. Reuse existing `session.maintenance` retention semantics rather than adding a new config option unless unavoidable. Preserve active/referenced session artifacts, compaction checkpoints referenced from store entries, and existing QMD-session safeguards. Keep the fix in the generic session maintenance layer, not a gateway-specific startup migration. Update docs and add a user-facing changelog entry. Do not change unrelated session routing, provider, or plugin behavior.
