## Summary

Keep open: current main has partial mitigations, but default session cleanup still does not reconcile unindexed transcript and trajectory artifacts, so the reported disk-bloat path remains actionable.

## What This Changes

The issue requests cleanup of session `.jsonl` and `.trajectory.jsonl` files that exist on disk but are no longer referenced by `sessions.json` after gateway restart or crash cycles.

## Best Possible Solution

Extend the existing sessions cleanup maintenance path to detect, dry-run report, and safely prune unreferenced session artifacts using current retention and reference-protection rules.

## Reproduction Assessment

Yes from source inspection: with default maintenance settings, create an unreferenced primary transcript or trajectory sidecar next to `sessions.json`; cleanup will not report it because the normal path iterates only indexed entries and disk-budget scanning is disabled without `maxDiskBytes`. I did not run a live filesystem repro because this review is read-only and `pnpm` is unavailable here.

## Solution Assessment

Yes: keeping this open for a focused repair is the narrow maintainable path. The best fix is to reuse the existing artifact reference scanner in normal session cleanup rather than adding startup migrations or a new configuration knob.
