## Label Justifications

## Summary

Keep this open because current and shipped behavior intentionally treats schema and invalid-request failures as terminal, while the report raises a meaningful but unresolved request for candidate-specific recovery. The exact rejected payload is needed to distinguish an Anthropic-only incompatibility from malformed state that every fallback would reject.

## What This Changes

Allow configured fallback models to recover an active run when Anthropic rejects a tool or message payload with `invalid_request_error`.

## Best Possible Solution

Keep globally malformed requests terminal, but add a provider-scoped fallback or deterministic payload-repair path when a redacted captured failure proves the request is Anthropic-specific and another candidate can validly continue the turn.

## Reproduction Assessment

Yes for the current terminal behavior: current source, regression tests, and the v2026.6.5 documentation show schema and invalid-request failures normally do not trigger fallback. There is not yet a high-confidence reproduction proving that the same rejected turn succeeds on the configured fallback because the complete payload is absent.

## Solution Assessment

No, treating every Anthropic `invalid_request_error` like a transient 429 or 503 is too broad. The narrow maintainable path is a provider/model-specific incompatibility classification or deterministic request repair backed by a captured failing payload and cross-provider validation.
