## Summary

Keep open: current main still has a source-reproducible fallback gap for Anthropic schema-rejection payloads. The model fallback loop can continue after thrown provider failures, but embedded non-business error payloads are not currently promoted into fallback classifications for Anthropic-style models.

## What This Changes

The issue asks OpenClaw to treat Anthropic `invalid_request_error` tool-payload/schema rejection as eligible for the configured fallback model instead of ending the session.

## Best Possible Solution

Define a narrow policy for pre-output Anthropic schema/tool-payload rejection: either repair/validate the bad payload before submit, or promote only safe `invalid_request_error` shapes into fallback with regression and live Anthropic proof.

## Reproduction Assessment

No live repro was run, but the source path is clear: current main's embedded result classifier only converts auth/auth-permanent/billing error payloads into fallback attempts, while the reported Anthropic schema rejection is a non-business provider error payload on a non-GPT5 model.

## Solution Assessment

Unclear as a final policy: treating every `invalid_request_error` as fallback-eligible may hide deterministic payload bugs, so the best fix needs maintainer-approved criteria plus tests and a live Anthropic proof path.
