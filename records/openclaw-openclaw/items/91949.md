## Summary

Keep this open: current main still treats Anthropic-style request-schema/tool-payload rejections as terminal `format` failures, and the documented failover policy says invalid-request errors usually do not rotate or fall back. The remaining work is real, but it needs a maintainer decision on whether this Anthropic shape is replay-safe fallback, pre-submit validation, or a narrower repair path.

## What This Changes

The issue asks OpenClaw to keep sessions recoverable when Anthropic returns `invalid_request_error` for a tool payload/schema rejection, instead of terminating before trying configured fallback models.

## Best Possible Solution

Add a maintainer-approved narrow recovery path: either validate/repair the Anthropic tool payload before submit, or classify only proven replay-safe Anthropic schema rejection shapes as fallback-eligible with regression coverage and live/redacted proof.

## Reproduction Assessment

Yes from source for the current policy behavior, but not as a live Anthropic repro. Current main maps unclassified HTTP 400/422 provider rejections to `format`, and the failover policy surfaces deterministic format failures unless a narrow retry flag opts in.

## Solution Assessment

Unclear. The requested broad fallback behavior is not the current documented policy; the best fix needs maintainer approval for a narrow Anthropic/schema-validation recovery path rather than a blanket `format` fallback change.
