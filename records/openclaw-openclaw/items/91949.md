## Summary

Keep this open: current main and v2026.6.6 still treat format/invalid-request provider rejections as terminal by default, while the report raises a plausible Anthropic-specific recovery gap that needs redacted payload proof and a maintainer policy call.

## What This Changes

The issue asks for Anthropic `invalid_request_error` schema/tool-payload rejections to trigger the configured model fallback instead of ending the session with a terminal provider-schema error.

## Best Possible Solution

Keep globally malformed requests terminal by default, then add a provider-scoped payload repair or fallback exception only if redacted captured payload evidence proves the failure is Anthropic-specific and safe to continue on another candidate.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and v2026.6.6 show format/invalid-request failures normally do not trigger fallback. No for the full Anthropic-to-OpenRouter recovery claim because the complete rejected payload and live acceptance by the fallback candidate are missing.

## Solution Assessment

No: the requested blanket treatment of Anthropic `invalid_request_error` like 429/503 is too broad against the current terminal format-failure policy. A captured-payload-backed provider-specific repair or fallback exception is the safer direction.
