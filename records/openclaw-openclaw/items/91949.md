## Summary

Keep this open: current main and v2026.6.6 still intentionally surface body-bearing format/invalid-request failures instead of falling back, while the requested Anthropic-specific recovery needs redacted payload proof and a maintainer policy decision.

## What This Changes

The issue requests treating Anthropic `invalid_request_error` tool-payload rejections as model-fallback eligible instead of terminating the active agent session.

## Best Possible Solution

Keep malformed request payloads terminal by default, then add a narrow Anthropic-specific repair or fallback exception only after redacted payload proof shows the fallback candidate can validly continue the turn.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and v2026.6.6 show format/invalid-request failures normally do not trigger fallback. No for the full live recovery claim, because the redacted payload does not prove the fallback model would accept the same turn.

## Solution Assessment

No for the requested blanket fallback treatment; it is broader than the current failover contract. The safer path is a captured-payload-backed provider-specific repair or fallback exception that preserves terminal handling for globally malformed requests.
