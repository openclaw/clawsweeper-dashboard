## Summary

Current main intentionally treats body-bearing HTTP 400/422 request-schema rejections as `format` failures and surfaces deterministic format failures instead of advancing the model chain. The report is still necessary because provider-specific transcript incompatibilities can be recoverable on another configured model, but the proposed rule to fallback on every Anthropic `invalid_request_error` is too broad without the complete rejected payload and a typed safety boundary.

## What This Changes

Allow configured model fallback to recover an agent turn when Anthropic rejects a request or tool payload with `invalid_request_error`.

## Best Possible Solution

Keep shared request/schema corruption terminal, but define a typed candidate-specific incompatibility signal for proven provider-only transcript or tool-schema rejection, preserve the original diagnostic, and cover both safe cross-provider recovery and non-retryable poisoned-state cases.

## Reproduction Assessment

Yes for the current-main terminal path: a body-bearing Anthropic HTTP 400 is classified as `format`, and deterministic format failures surface without model fallback. No for the reporter's claimed provider-specific recoverability until the complete redacted rejected request and a real fallback comparison are available.

## Solution Assessment

No, not as proposed: making every Anthropic `invalid_request_error` equivalent to 429/503 would retry deterministic malformed state. The best direction is a narrow typed classification for provider-specific incompatibility, with terminal behavior retained for shared transcript and tool-payload corruption.
