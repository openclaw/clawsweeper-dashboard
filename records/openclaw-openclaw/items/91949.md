## Summary

Keep open. Current main and the latest release still keep Anthropic schema/tool-payload `invalid_request_error` in the terminal format-failure path unless a narrow retry-repair path opts in, and the related open PR explicitly excludes this colon-suffixed schema-rejection variant.

## What This Changes

The issue requests treating Anthropic `invalid_request_error` schema/tool-payload failures as fallback-eligible instead of terminating the active session.

## Best Possible Solution

Add a maintainer-approved narrow recovery policy: either repair/preflight the safe Anthropic request-shape failures before retry, or allow fallback only for proven replay-safe subclasses while leaving deterministic malformed payloads terminal.

## Reproduction Assessment

Yes from source, but not from a live Anthropic run. Current code, tests, and docs show the reported schema/tool-payload error text stays in the terminal format-failure path unless a narrow repair path opts in.

## Solution Assessment

Unclear. Broad fallback for all `invalid_request_error` responses is not the best fix; the safer direction is a narrow repair/preflight or allowlisted fallback policy for replay-safe Anthropic payload failures.
