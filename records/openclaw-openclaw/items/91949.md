## Summary

Keep this open: current main and v2026.6.6 still intentionally treat generic provider schema/tool-payload rejections as terminal format failures, while the report may describe either a distinct Anthropic tool-payload bug or the signed-thinking replay class already tracked elsewhere; the redacted payload is the blocker for a safe close or fix boundary.

## What This Changes

The issue requests that Anthropic `invalid_request_error` responses for tool-call payload/schema problems trigger the configured fallback model instead of terminating the session.

## Best Possible Solution

Keep format failures terminal by default, then add a narrow payload-backed repair or fallback exception only for a proven Anthropic-specific recoverable class; route signed-thinking evidence to https://github.com/openclaw/openclaw/issues/85781 if that is the real payload.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and v2026.6.6 show `format`/invalid-request failures normally surface instead of falling back. No for the full cross-provider recovery claim because the complete rejected Anthropic payload is still redacted.

## Solution Assessment

No for the blanket requested fix: treating all Anthropic `invalid_request_error` responses like 429/503 is broader than the documented failover policy. The safer solution is a captured-payload-backed retry repair or fallback exception that does not undo terminal handling for deterministic malformed requests.
