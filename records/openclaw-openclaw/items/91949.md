## Summary

Keep this open: current main and the latest release still intentionally surface format/invalid-request failures instead of falling back, while the report raises a plausible provider-specific recovery gap that needs payload evidence and a maintainer policy decision.

## What This Changes

The issue asks OpenClaw to treat Anthropic invalid_request_error responses for tool-call payload rejection as fallback-eligible instead of terminating the active session.

## Best Possible Solution

Keep globally malformed requests terminal, then add a provider-scoped fallback or deterministic payload-repair path only if a redacted captured failure proves the request is Anthropic-specific and another candidate can validly continue the turn.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and the v2026.6.6 tag show format/invalid-request failures normally do not trigger fallback. No for the full cross-provider recovery claim, because the complete rejected payload is missing.

## Solution Assessment

No: the requested blanket treatment of Anthropic invalid_request_error like 429/503 is too broad. The safer solution is a captured-payload-backed provider-specific repair or fallback exception that does not undo the terminal format-failure policy.
