## Summary

Keep this open: current main and the latest release still intentionally treat unclassified 400/422 request-shape rejections as `format` failures that surface instead of falling back, while this report raises a plausible Anthropic-specific recovery gap that needs a redacted payload and maintainer policy decision.

## What This Changes

The issue requests that Anthropic `invalid_request_error` tool-payload/schema failures use the configured fallback model instead of terminating the current session.

## Best Possible Solution

Keep globally malformed requests terminal by default, then add a provider-scoped repair or fallback exception only if a full redacted failure fixture proves the rejected Anthropic payload is safe to repair or replay on a fallback candidate.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and the latest release show `format`/invalid-request failures normally do not trigger fallback. No for the full live recovery claim because the complete rejected Anthropic payload is missing.

## Solution Assessment

No for the requested blanket treatment of Anthropic `invalid_request_error` like 429/503. The maintainable path is a payload-backed provider-specific repair or fallback exception that does not undo the terminal format-failure policy for deterministic malformed requests.
