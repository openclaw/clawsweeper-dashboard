## Summary

Keep this open: current main and the latest release still treat provider format/invalid-request failures as terminal by default, while the report raises a plausible Anthropic-specific recovery gap that needs the rejected payload and a maintainer policy call before changing fallback behavior.

## What This Changes

The issue requests making Anthropic invalid_request_error/tool-payload rejections fallback-eligible instead of terminating the agent session.

## Best Possible Solution

Keep malformed requests terminal by default, then add a provider-scoped repair or fallback exception only if a redacted captured failure proves the request is Anthropic-specific and replay-safe.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and the v2026.6.6 tag show format/invalid-request failures normally do not trigger fallback. No for the full cross-provider recovery claim, because the complete rejected payload is missing.

## Solution Assessment

No: the requested blanket treatment of Anthropic invalid_request_error like 429/503 is too broad. The safer solution is a captured-payload-backed provider-specific repair or fallback exception that does not undo the terminal format-failure policy.
