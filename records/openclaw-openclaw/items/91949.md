## Summary

Current `main` still treats request-shape `format` failures as terminal on the prompt/request path, while docs say model fallback advances on auth/rate-limit/timeout classes and usually surfaces invalid-request errors. The report is recent, concrete, and not superseded, but changing that policy for Anthropic tool-payload rejections needs maintainer/product judgment and live repro before a fix PR.

## What This Changes

Request to make Anthropic `invalid_request_error` tool-call payload rejections fall back to configured backup models instead of ending the session.

## Best Possible Solution

Maintain the current terminal default for generic format errors unless maintainers approve a narrow Anthropic/tool-payload retry-repair path, ideally backed by a live repro and regression tests for both the recovered and still-terminal cases.

## Reproduction Assessment

Yes from source for the policy path: current `main` classifies request-shape failures as `format` and the prompt-stage decision surfaces terminal format errors instead of entering model fallback. No live Anthropic reproduction was run, so the exact provider payload recovery remains unproven.

## Solution Assessment

Unclear. Treating every Anthropic `invalid_request_error` as fallback-eligible is not obviously the best fix because docs and code intentionally keep generic format errors terminal; the safer direction is a narrow retry-repair policy or pre-submit payload validation after maintainers decide the contract.
