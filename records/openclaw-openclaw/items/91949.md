## Summary

Keep open: current main and v2026.6.8 still route Anthropic schema/tool-payload `invalid_request_error` responses through the terminal format-failure path, and the related open PR only covers the bare `LLM request failed.` wrapper while explicitly leaving this variant out.

## What This Changes

The issue requests fallback or recovery when Anthropic returns `invalid_request_error` for schema or tool-payload failures instead of terminating the active session.

## Best Possible Solution

Add a maintainer-approved narrow recovery policy: preflight or repair proven-safe Anthropic request-shape failures, or allow fallback only for replay-safe subclasses while keeping deterministic malformed payloads terminal.

## Reproduction Assessment

Yes from source, but not from a live Anthropic run. Current code, tests, and docs show the reported schema/tool-payload error text stays in the terminal format-failure path unless a narrow repair path opts in.

## Solution Assessment

Unclear. Broad fallback for all `invalid_request_error` responses is not the best fix; the safer direction is a narrow repair/preflight or allowlisted fallback policy for replay-safe Anthropic payload failures.
