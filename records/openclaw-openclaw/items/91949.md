## Summary

Keep open. Current main still needs maintainer direction for whether Anthropic request-shape `invalid_request_error` should trigger a narrow repair/fallback path, because the documented default keeps format/invalid-request failures terminal unless a known retry-repair path opts in.

## What This Changes

The issue requests fallback or recovery when Anthropic rejects a tool-call payload with `invalid_request_error` instead of terminating the agent session.

## Best Possible Solution

Keep this open for a maintainer-approved recovery policy: either add a narrow repair/fallback path for safe Anthropic request-shape failures, or keep them terminal and add preflight/session repair that prevents poisoned payloads from being sent.

## Reproduction Assessment

Yes from source, but not from a live Anthropic run. Current code and docs show format/invalid-request failures are terminal unless a known repair path opts into retry; the exact redacted provider payload still needs live proof.

## Solution Assessment

Unclear. Broadly treating every `invalid_request_error` as fallback-eligible is not the best fix; the safer direction is a narrow allowlisted repair/fallback path or preflight validator for replay-safe Anthropic payload failures.
