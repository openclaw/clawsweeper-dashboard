## Summary

Keep this open: current main and v2026.6.6 intentionally surface format/invalid-request failures instead of falling back, but the report describes a plausible Anthropic-specific recovery gap that needs the full redacted rejected payload and a maintainer policy decision.

## What This Changes

The issue requests treating Anthropic tool-payload invalid_request_error responses as fallback-eligible instead of terminating the active session.

## Best Possible Solution

Keep malformed requests terminal by default, and add a narrow provider-owned classifier, repair, or fallback exception only if captured payload proof shows a recoverable Anthropic-specific shape.

## Reproduction Assessment

Partly: yes for the current terminal behavior, because source, tests, docs, and v2026.6.6 show format/invalid-request failures normally do not trigger fallback. No for the full Anthropic-to-fallback recovery claim, because the complete rejected payload is redacted and no live provider repro was run.

## Solution Assessment

No for the requested blanket fallback treatment. A payload-backed provider-specific repair or fallback exception is safer than treating all Anthropic invalid_request_error responses like 429 or 503 failures.
