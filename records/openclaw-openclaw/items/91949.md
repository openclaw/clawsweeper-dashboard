## Summary

Keep this open: current main and v2026.6.5 still treat invalid-request/format rejections as terminal except explicit repair paths, while the report raises a meaningful provider-specific recovery request that needs a complete redacted payload and maintainer policy decision.

## What This Changes

The issue asks OpenClaw to treat Anthropic `invalid_request_error` tool-payload or schema rejections as fallback-eligible instead of terminating the active session.

## Best Possible Solution

Keep format errors terminal by default, then add a targeted Anthropic-specific payload repair or fallback eligibility path only after a captured redacted payload proves the rejection is candidate-specific.

## Reproduction Assessment

Yes for the current terminal handling: source, tests, and v2026.6.5 docs show invalid-request/format failures do not normally trigger fallback. No for the stronger claim that the same rejected turn succeeds on the configured fallback, because the complete payload is absent.

## Solution Assessment

No for the requested blanket fallback rule. The safer direction is targeted provider/model-specific repair or fallback eligibility backed by a captured payload and cross-provider validation.
