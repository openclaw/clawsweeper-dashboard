## Summary

Keep this open: current main and v2026.6.6 still classify body-bearing 400/422 provider schema rejections as `format` and intentionally surface non-retryable format failures instead of using the configured model fallback, while the report still lacks the redacted rejected payload needed to prove a fallback candidate can safely continue the same turn.

## What This Changes

The issue requests treating Anthropic `invalid_request_error` schema/tool-payload failures as model-fallback-eligible instead of terminating the active session.

## Best Possible Solution

Keep globally malformed format requests terminal, then add a provider-scoped payload repair or fallback exception only after a redacted captured Anthropic failure proves the request is provider-specific and replay-safe for the next candidate.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and v2026.6.6 show format/invalid-request failures normally do not trigger fallback. No for the full cross-provider recovery claim, because the complete rejected payload is missing.

## Solution Assessment

No: the requested blanket treatment of Anthropic `invalid_request_error` like 429/503 is too broad. The safer fix is a captured-payload-backed provider-specific repair or fallback exception that does not undo the terminal format-failure policy.
