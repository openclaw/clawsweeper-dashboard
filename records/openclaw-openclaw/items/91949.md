## Summary

Keep open: current main repaired a related signed-thinking replay class, but generic Anthropic 400/422 invalid-request or format failures still surface as terminal unless a narrow repair path opts in. The reported payload is redacted too far to prove this is already fixed or safely covered by the signed-thinking tracker.

## What This Changes

The issue asks OpenClaw to treat Anthropic `invalid_request_error` tool-payload/schema rejections as fallback-eligible instead of terminating the active session.

## Best Possible Solution

Keep generic malformed request payloads terminal, but add a provider-scoped repair or fallback path only when a complete redacted payload and live proof show the rejection is Anthropic-specific and safe to continue on another candidate.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, docs, and PR history show format/invalid-request failures do not generally trigger fallback. No for the specific reported payload succeeding on fallback, because the rejected content is redacted.

## Solution Assessment

No, a blanket `invalid_request_error` fallback is too broad. The maintainable fix is a targeted provider/model repair or fallback rule backed by the exact rejected payload and proof that another candidate can validly continue.
