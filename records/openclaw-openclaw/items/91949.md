## Summary

Keep open: current main still treats broad Anthropic invalid_request_error/schema-payload failures as terminal format errors unless a specific repair path opts in, so this needs a maintainer product decision and live proof before changing fallback behavior.

## What This Changes

Treat Anthropic invalid_request_error from invalid tool/schema payloads as fallback-eligible instead of ending the session.

## Best Possible Solution

Define a narrow recovery/fallback policy for replay-safe invalid_request cases, add targeted validation or repair before retry, and keep deterministic schema errors terminal by default unless the retry path is proven safe.

## Reproduction Assessment

Source-reproducible: current main classifies unclassified 400/422 invalid requests as format failures and surfaces terminal format failures instead of broad fallback. No live Anthropic session was run for this review.

## Solution Assessment

Unclear: broad fallback for all invalid_request_error responses is not proven to be the safest fix. A narrow replay-safe repair path with provider-specific proof is more maintainable than retrying deterministic schema errors by default.
