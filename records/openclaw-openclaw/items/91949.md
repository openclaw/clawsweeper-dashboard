## What This Changes

The issue requests that Anthropic `invalid_request_error` tool-payload/schema rejections either be repaired before submission or trigger the configured fallback model instead of ending the active session.

## Best Possible Solution

Keep generic malformed request payloads terminal, but add a provider/model-specific recovery or fallback classifier only for captured invalid-request subtypes proven safe to replay or repair.

## Reproduction Assessment

Yes for the current terminal behavior: source and release docs show generic invalid-request format failures do not advance fallback. The exact Anthropic payload is still needed to reproduce the claimed safe fallback success.

## Solution Assessment

No for broad fallback on every `invalid_request_error`; that would be too coarse for deterministic malformed payloads. A narrow captured-payload repair or provider-specific fallback classifier is the maintainable path.
