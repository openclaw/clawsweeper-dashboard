## Summary

Keep open: current main and v2026.6.6 still treat generic 400/422 `invalid_request_error` payload/schema failures as terminal format failures unless a known repair path opts in. The related thinking-signature recovery fix in https://github.com/openclaw/openclaw/pull/92916 does not settle this broader Anthropic tool-payload fallback request.

## What This Changes

Request to make Anthropic `invalid_request_error` tool-payload rejections trigger the configured fallback model instead of ending the agent session.

## Best Possible Solution

Add a narrow repair or retry-once path for the specific Anthropic tool/transcript invalid-request pattern, while keeping generic malformed payloads terminal unless maintainers explicitly change the fallback contract.

## Reproduction Assessment

No for the exact Anthropic session: the issue includes logs but no deterministic prompt/session artifact. Source inspection does show current main still classifies generic 400/422 invalid-request rejections as terminal `format` failures instead of fallback-eligible ones.

## Solution Assessment

Unclear: globally treating `invalid_request_error` as fallback-eligible conflicts with current docs and tests. A safer permanent fix is a scoped validator/repair/retry path for the concrete Anthropic tool or transcript error.
