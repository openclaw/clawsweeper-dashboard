## Summary

Keep open: current main and the latest release still treat generic Anthropic format/invalid_request failures as terminal unless a narrow retry-repair path opts in, and no canonical related item safely owns this broader fallback-policy change.

## What This Changes

The issue asks OpenClaw to treat Anthropic invalid_request_error tool/schema payload rejections as fallback-eligible instead of terminating the session.

## Best Possible Solution

Maintainers should choose a narrow policy: either keep generic invalid_request_error terminal and fix the specific stale tool/replay payload source, or explicitly classify only proven recoverable Anthropic invalid_request_error shapes for fallback with regression tests and live proof.

## Reproduction Assessment

Yes for the source-level policy path, but not for a live Anthropic run. Current code surfaces format-classified invalid_request failures unless allowFormatRetry is set, and embedded error payloads only promote auth/billing cases to fallback.

## Solution Assessment

Unclear. The requested fallback behavior is plausible, but current docs intentionally keep most invalid-request errors terminal, so the best fix needs a maintainer-approved recoverable-error boundary rather than a blanket classifier change.
