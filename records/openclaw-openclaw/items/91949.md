## What This Changes

The issue requests that Anthropic `invalid_request_error` tool-payload failures advance to configured model fallbacks instead of terminating the active session.

## Best Possible Solution

Add a targeted recovery path only after a redacted full payload proves an Anthropic-specific request incompatibility that another configured candidate can validly handle.

## Reproduction Assessment

Yes for the current terminal behavior: source, tests, and v2026.6.5 docs show format/invalid-request failures do not normally advance fallback. No for the requested cross-provider recovery, because the complete rejected payload is missing.

## Solution Assessment

No, treating every Anthropic `invalid_request_error` like a transient 429 or 503 is too broad. The safer path is a provider/model-specific incompatibility classifier or deterministic request repair backed by a captured failing payload.
