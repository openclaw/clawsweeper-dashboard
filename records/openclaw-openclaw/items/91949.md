## Summary

Keep open. Current main and the latest release still treat provider `format`/invalid-request failures as terminal by default, and the exact Anthropic raw error is elided enough that this should not be closed as implemented or as a duplicate of the narrower thinking-signature cluster.

## What This Changes

The issue asks OpenClaw to recover from Anthropic `invalid_request_error` tool/request payload failures by using configured fallback models instead of terminating the active session.

## Best Possible Solution

Keep this open until maintainers decide the narrow recovery policy: either add a specific retry-repair/fallback path for safe Anthropic request-shape failures, or keep them terminal and add preflight/session repair that prevents poisoned payloads from being sent.

## Reproduction Assessment

Yes from source, not from a live Anthropic run: current failover policy treats `format` failures as terminal unless `allowFormatRetry` is explicitly set, and tests assert assistant format failures surface instead of falling back. The exact Anthropic payload from the reporter was not live-reproduced.

## Solution Assessment

Unclear. Making every `invalid_request_error` fallback-eligible would conflict with current docs/tests, so the best fix likely needs a narrow allowlisted repair/fallback path or preflight validation for the specific safe Anthropic payload failures.
