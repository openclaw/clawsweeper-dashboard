## Summary

Keep this open: current main and the latest release still treat most provider format/invalid-request failures as terminal, while this report describes a real session-ending Anthropic failure that needs a maintainer-approved recovery or fallback policy.

## What This Changes

The issue asks OpenClaw to recover from Anthropic `invalid_request_error` tool/payload rejections instead of terminating the active session without trying the configured fallback model.

## Best Possible Solution

Define a narrow recovery path for the specific Anthropic tool/transcript payload failure: validate or repair the bad payload before submission, or mark only proven repairable format failures as retry/fallback eligible, with tests and docs updated together.

## Reproduction Assessment

Yes at source level, with medium confidence: current policy surfaces `format` failures instead of fallback, and current error helpers map schema-style 400/422/`invalid_request` failures into that area. No live Anthropic run reproduced the exact reporter payload in this review.

## Solution Assessment

Unclear as requested: broad fallback for all `invalid_request_error` is not obviously the best fix because current docs intentionally treat most format errors as terminal. A safer solution is a maintainer-approved, provider-specific repair or retry path for the exact recoverable tool/transcript class.
