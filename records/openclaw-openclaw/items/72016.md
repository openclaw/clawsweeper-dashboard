## Summary

Keep open. Current main still lacks the requested public, upgrade-safe doctor-check extension API: the health contribution model and runner are core-owned/static, the plugin API has no doctor-check registrar, and the existing doctor plugin contract only covers migration/compatibility helpers. The issue is actionable, but it needs maintainer design/security decisions before implementation.

## What This Changes

The issue requests a plugin-style extension point for custom `openclaw doctor` checks and optional fixes that survive npm upgrades.

## Best Possible Solution

Add a narrow, additive public doctor contribution contract with plugin attribution, deterministic ordering, duplicate handling, focused contract tests, and docs; start with read-only diagnostics or tightly-gated repair support, then consider the multi-gateway overlap check as a separate built-in or example once the trust policy is clear.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is an API/feature request. The high-confidence verification path is source inspection: current main has no public plugin-SDK doctor-check registration surface for the main doctor health flow.

## Solution Assessment

Unclear until maintainers decide the public API and repair trust policy. A plugin-SDK extension is the right direction for upgrade-safe custom checks, but plugin-owned `fix` support should not land without explicit consent and unattended-fix rules.
