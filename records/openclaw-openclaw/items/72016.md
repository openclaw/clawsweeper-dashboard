## Summary

Keep open. Current main still lacks a public doctor-check registration surface: the health contribution model is core-only, the doctor health run loop is a static in-file list, and the plugin SDK/runtime-doctor surfaces do not expose a generic check/fix API. The request is valid, but the repair path creates a public SDK seam plus plugin-owned repair authority, so it needs maintainer API and trust-policy direction before an automated fix PR.

## What This Changes

Requests a plugin-extensible doctor API so operators can add custom environment checks and persistent fix flows that survive package upgrades.

## Best Possible Solution

Keep this as a plugin-SDK design item. Add an additive doctor-check contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, list/filter support, tests, and docs; start with read-only diagnostics, then add repair support only after the interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. The verification path is high-confidence source inspection: current main has no public doctor-check registration API or equivalent plugin contribution registry for doctor health checks.

## Solution Assessment

Unclear until maintainers choose the API and trust model. A narrow additive SDK seam is the maintainable direction, while local source patches or ad hoc plugin CLI commands would not satisfy the integrated doctor workflow requested here.
