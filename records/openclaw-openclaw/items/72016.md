## Summary

Keep open. Current `main` still lacks a public plugin-SDK registration surface for custom `openclaw doctor` checks; the existing doctor flow is core-owned/static, and adding plugin-owned fixes needs maintainer API and trust-policy direction rather than cleanup closure.

## What This Changes

Expose an upgrade-safe extension point for `openclaw doctor` checks and optional fixes so operators can add environment-specific diagnostics without patching source after each npm upgrade.

## Best Possible Solution

Treat this as a plugin-SDK design item: add a narrow additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only diagnostics and add repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path: current `main` has no public doctor-check registration API or equivalent plugin contribution registry for the main doctor health flow.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam appears maintainable, while local source patches or standalone plugin CLI commands would not provide the integrated, upgrade-safe doctor workflow being requested.
