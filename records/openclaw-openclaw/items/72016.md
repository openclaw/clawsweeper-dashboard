## Summary

Keep open. Current main and v2026.6.10 have structured doctor health checks, a public health helper subpath, plugin doctor-contract migrations, and a narrow bundled Policy bridge, but they still do not provide an upgrade-safe public path for arbitrary non-bundled plugins to contribute doctor checks and repair-capable fixes; the remaining work needs maintainer product and security direction.

## What This Changes

The issue requests a public, upgrade-safe doctor extensibility API so plugins or local checks can add diagnostics and optional fixes without editing OpenClaw source.

## Best Possible Solution

Define an additive, opt-in doctor health extension contract using the existing health registry as the base, with source attribution, bounded plugin loading, explicit repair consent/unattended policy, public docs, and focused SDK/CLI tests.

## Reproduction Assessment

Yes for the source-level feature gap, not for a runtime bug. Current main and v2026.6.10 show doctor loading disabled for normal plugins, a Policy-only bundled health bridge, and no arbitrary installed-plugin doctor extension loading contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The existing structured health registry is the right foundation, but the public loading boundary, SDK surface, check selection, and repair consent policy need an explicit decision before implementation.
