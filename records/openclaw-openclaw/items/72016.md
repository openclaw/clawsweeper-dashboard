## Summary

Keep open. Current main and v2026.6.10 have the structured doctor health-check contract and `plugin-sdk/health`, but the active automatic bridge still only loads the bundled Policy plugin, so the requested upgrade-safe external plugin doctor extension path remains a product and security design item.

## What This Changes

The issue requests an upgrade-safe plugin/extensibility API for custom `openclaw doctor` checks and optional fixes so local environment-specific checks survive npm upgrades.

## Best Possible Solution

Define a narrow, security-reviewed external plugin doctor registration contract, likely manifest or public-surface based, and keep custom checks outside core while preserving explicit consent for any repair path.

## Reproduction Assessment

Yes for the source-level feature gap: current main and v2026.6.10 expose structured doctor health checks, but the inspected automatic bridge registers only bundled Policy checks, not arbitrary installed plugin checks.

## Solution Assessment

Unclear as proposed because the broad API shape needs maintainer selection; the maintainable path is a small explicit registration/loading contract plus security rules, not ad hoc doctor subcommand or runtime loading expansion.

## Risks / Open Questions

- A general plugin doctor hook could load arbitrary plugin runtime during doctor, so maintainers need a trust, timeout, redaction, and owner-policy model before expanding beyond bundled Policy checks.

- The current partial SDK surface may make plugin authors think registration is automatic when external installed plugins still need a durable active-command-path registration mechanism.
