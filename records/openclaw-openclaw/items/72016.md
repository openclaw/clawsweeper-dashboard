## Summary

Keep open. Current main still has a static core-owned doctor health flow plus limited plugin/channel compatibility adapters, but no public upgrade-safe SDK registrar for arbitrary doctor checks or plugin-owned repair hooks; the follow-up comments accurately identify the remaining API and trust-policy gap.

## What This Changes

Request an upgrade-safe plugin/custom extension point for `openclaw doctor` checks and optional environment-specific repair actions.

## Best Possible Solution

Design a narrow additive doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, list/filter UX, focused tests, docs, and separately gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. Source inspection gives a high-confidence verification path: current main has no public doctor-check registrar and the doctor health flow only runs its static core list.

## Solution Assessment

Unclear until maintainers settle the public API and repair trust policy. A small additive SDK seam appears maintainable, while local source edits or standalone plugin CLIs would not provide the integrated upgrade-safe doctor workflow requested.
