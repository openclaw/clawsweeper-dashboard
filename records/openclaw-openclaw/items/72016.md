## Summary

Keep open. Current main has the structured doctor health-check contract and a public `plugin-sdk/health` subpath, but the automatic doctor command path still only bridges the bundled Policy plugin, so upgrade-safe external plugin doctor checks and repair authority remain a product/security decision.

## What This Changes

Requested extensibility for `openclaw doctor` so custom environment checks and possible fixes can survive npm upgrades outside local core source edits.

## Best Possible Solution

Define a narrow, security-reviewed external plugin doctor registration contract, likely manifest or public-surface based, and keep custom checks outside core while requiring explicit consent for any repair path.

## Reproduction Assessment

Yes for the source-level feature gap: current main and v2026.6.10 expose structured doctor health checks, but the inspected automatic bridge registers only bundled Policy checks rather than arbitrary installed plugin checks.

## Solution Assessment

Unclear as proposed because maintainers still need to choose the public API, loading boundary, and repair-consent model; a small explicit registration contract is safer than broad doctor runtime loading.
