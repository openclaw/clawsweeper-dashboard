## Summary

Keep open. Current main still has a static core doctor health contribution list plus narrow plugin compatibility hooks, but no public plugin-SDK registrar for arbitrary custom doctor checks or plugin-owned repair flow.

## What This Changes

Request an upgrade-safe plugin/SDK extension point for custom `openclaw doctor` checks and possible `--fix` repairs.

## Best Possible Solution

Design a narrow additive doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, docs, and separately gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. Source inspection gives a high-confidence verification path: current main has no public doctor-check registrar and the doctor health flow only runs its static core list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam appears maintainable, while local source edits or ad hoc plugin CLI commands would not provide the integrated upgrade-safe doctor workflow requested.
