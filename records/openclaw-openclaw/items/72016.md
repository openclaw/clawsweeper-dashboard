## Summary

Keep open. Current main still lacks a public doctor-check registration surface: doctor health contributions are core-only, the run loop is a static in-file list, and the plugin SDK/runtime-doctor surfaces do not expose a registerDoctorCheck-style API. The request is valid, but the repair half is a public SDK and trust-policy design item, so it should not be auto-closed or sent directly to a repair lane.

## What This Changes

Expose a plugin/SDK extension point for `openclaw doctor` checks and optional fixes so environment-specific diagnostics can survive npm upgrades.

## Best Possible Solution

Keep this as a plugin-SDK design item. Add a narrow additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only diagnostics, then add repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path: current main has no public doctor-check registration API or equivalent plugin contribution registry for doctor health checks.

## Solution Assessment

Unclear until maintainer API and trust-policy direction lands. A narrow additive SDK seam appears maintainable, but using ad hoc plugin CLI commands or local source patches would not solve the requested integrated doctor workflow.
