## Summary

Keep open. Current main still has a core-owned static doctor health flow and a migration-focused plugin doctor contract, but no public plugin-SDK surface for custom doctor checks or plugin-owned repair policy; the remaining work needs maintainer API and trust-policy direction.

## What This Changes

Request to make `openclaw doctor` extensible so persistent plugins or local packages can add custom setup checks and optional repair playbooks across upgrades.

## Best Possible Solution

Treat this as a public plugin-SDK design item: add a narrow additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, and docs; start with read-only diagnostics or clearly gated repair support rather than local source patching.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API/feature request. The verification path is high-confidence source inspection: current main has no public doctor-check registration API or equivalent plugin contribution registry for the main doctor health flow.

## Solution Assessment

Unclear until maintainers settle the API and trust policy. A narrow additive SDK seam appears maintainable; standalone local source edits or plugin CLI commands would not provide the integrated, upgrade-safe doctor workflow being requested.
