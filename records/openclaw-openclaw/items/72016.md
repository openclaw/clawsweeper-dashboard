## Summary

Keep open. Current main still has a static core doctor health flow without a public plugin registrar for custom doctor checks, and the discussion has unresolved SDK shape plus repair-trust decisions that need maintainer direction.

## What This Changes

Expose an upgrade-safe plugin/SDK extension point for custom `openclaw doctor` checks and optional fixes.

## Best Possible Solution

Design and ship a narrow additive public doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, docs, and separately gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. The feature gap is high-confidence by source inspection on current main: there is no public doctor-check registrar and the health flow runs a static core contribution list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam appears maintainable, while local source edits or plugin-owned side commands would not provide the integrated upgrade-safe doctor workflow requested.

## Security Review

Status: needs_attention

Summary: The requested future API is security-sensitive because plugin-owned doctor fixes would run in local repair paths with config and state access.

Concerns:

- **[medium] Define trust and consent for plugin doctor fixes:** `docs/plugins/architecture.md:444`
  - body: A future SDK would let plugin-owned code run during doctor and, if fix-enabled, mutate config or host state. Native plugins run in-process and are not sandboxed, so the contract needs attribution, prompts or unattended-mode policy, logs, and docs before unattended repairs are allowed.
  - confidence: 0.9
