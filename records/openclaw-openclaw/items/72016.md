## Summary

Keep open. Current main still lacks a public plugin SDK/API for registering custom doctor checks, and the requested repair path crosses trust-policy decisions that maintainers should settle before implementation.

## What This Changes

Request an upgrade-safe doctor extensibility API so plugins or local extensions can add environment-specific health checks and optional fixes.

## Best Possible Solution

Design an additive public doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, docs, and separately gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. The feature gap is high-confidence by source inspection: current main has no public doctor-check registrar and runs a static core contribution list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam is the maintainable direction; local source edits or plugin-owned side commands would not provide the integrated upgrade-safe doctor workflow requested.
