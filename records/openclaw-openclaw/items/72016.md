## Summary

Keep open. The issue discussion correctly identifies a real current-main gap: doctor health checks are still a static core-only contribution list, and there is no public plugin SDK registrar for custom doctor checks or plugin-owned fixes.

## What This Changes

Request an upgrade-safe plugin/API surface for custom `openclaw doctor` checks and optional repair actions.

## Best Possible Solution

Design an additive public doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, docs, and separately gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. The feature gap is high-confidence from source inspection: current main has no public doctor-check registrar and runs a static core contribution list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam is the maintainable direction; local source edits or plugin-owned side commands would not provide the integrated upgrade-safe doctor workflow requested.
