## Summary

Keep open. Current main partially addresses the foundation with a structured health-check registry and bundled policy checks, but the requested upgrade-safe custom doctor extension path is still unresolved for non-bundled plugins because `openclaw doctor` does not load general plugin runtime and the remaining repair/trust policy needs maintainer direction.

## What This Changes

Request an upgrade-safe plugin or extension API for custom `openclaw doctor` checks and optional `--fix` repairs that survive npm upgrades.

## Best Possible Solution

Define an additive doctor health extension contract that loads opted-in plugin checks with clear attribution in lint and repair modes, and gate plugin-owned repairs behind explicit consent, policy controls, docs, and focused contract tests.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence that current main does not support the requested end-to-end custom doctor extension path: `doctor` has `loadPlugins: "never"`, and only core plus bundled policy checks are registered automatically.

## Solution Assessment

Unclear until maintainers choose the plugin loading and repair trust policy. The current structured health registry is a good base, but the maintainable solution should make plugin attribution, consent, and non-bundled loading semantics explicit instead of relying on ad hoc runtime imports.
