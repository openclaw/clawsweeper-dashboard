## Summary

Keep open. Current main and the latest release have the structured health-check registry and `plugin-sdk/health`, but the regular plugin API still lacks a first-class external doctor-check registration lifecycle and the automatic bridge is limited to bundled Policy, so the requested upgrade-safe custom doctor extension surface remains a product and security design item.

## What This Changes

Request an upgrade-safe plugin/API extension surface for custom `openclaw doctor` checks and optional repair actions.

## Best Possible Solution

Keep this issue open as the canonical design item for external doctor extensions, evolving the existing `plugin-sdk/health` contract into an explicit plugin API with safe default linting, scoped loading, and opt-in repair.

## Reproduction Assessment

Yes, source inspection gives a high-confidence gap: current main has a structured health registry and a Policy-only bridge, but `OpenClawPluginApi` has no first-class doctor or health-check registration method for external plugins.

## Solution Assessment

No current implementation fully solves the issue. The best path is likely to reuse `plugin-sdk/health`, but maintainers still need to decide plugin loading, trust boundaries, and repair consent semantics before implementation.
