## Summary

Keep open. Current main still has only core-owned doctor health contributions plus channel/provider-specific doctor helper seams; it does not expose the requested public plugin-SDK contract for arbitrary persistent doctor checks or plugin-owned fixes. The issue discussion already narrowed the remaining work to API location, contribution attribution, CLI plumbing, and repair trust policy, so this is a real design item rather than stale cleanup.

## What This Changes

The issue requests a supported doctor extension API so operators can add custom environment checks and optional fixes that survive npm upgrades.

## Best Possible Solution

Keep this as a plugin-SDK design item. Add an additive, read-only doctor contribution contract first with plugin attribution, deterministic ordering, duplicate-id handling, docs, and focused contract tests; add plugin-owned repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path: current main has no public registerDoctorCheck-style API or equivalent generic plugin doctor contribution registry.

## Solution Assessment

Unclear until maintainer API and trust-policy direction lands. A narrow additive plugin-SDK doctor contribution seam looks maintainable, but plugin-owned repair authority and unattended fix behavior need explicit design before implementation.

## Risks / Open Questions

- The repair half of the requested API is security-sensitive: plugin-owned fixes could mutate config, state, or host setup under doctor repair modes without a settled consent and trust policy.
- A diagnosis-only doctor contribution API, repair/fix API, CLI filtering/listing behavior, and the bundled multi-gateway overlap check may need to be split so the public SDK seam stays reviewable.
- Doctor currently tries to keep non-interactive runs from eagerly loading plugin runtime; a new contribution registry needs an explicit loading model so health checks do not regress startup or headless doctor performance.
