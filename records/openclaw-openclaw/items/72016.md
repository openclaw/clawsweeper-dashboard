## Summary

Keep open. Current main still has a static core-owned doctor health flow and helper-only `runtime-doctor` SDK exports; exposing custom doctor checks and fixes is a real public SDK plus repair-trust design gap, not a cleanup-close candidate.

## What This Changes

Request an upgrade-safe plugin/SDK extension point for adding custom `openclaw doctor` checks and optional fixes, motivated by multi-gateway configuration drift.

## Best Possible Solution

Design and ship a narrow additive doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate handling, focused tests, docs, and separately consent-gated repair support.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. The feature gap is high-confidence by source inspection: current main has no public doctor-check registrar and runs a static core contribution list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam appears maintainable; local source edits or ad hoc plugin CLI commands would not provide the integrated upgrade-safe doctor workflow requested.

## Security Review

Status: needs_attention

Summary: The requested future API is security-sensitive because plugin-owned doctor fixes would run in-process and may mutate local config or host state.
