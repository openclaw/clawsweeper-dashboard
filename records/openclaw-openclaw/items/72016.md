## Summary

Keep open. Current `main` still has only a core-owned, static doctor health contribution list plus a migration-focused plugin doctor contract; it does not expose a public plugin-SDK registration surface for custom `openclaw doctor` checks or plugin-owned repair policy.

## What This Changes

The issue requests an upgrade-safe way for plugins or local extensions to add custom `openclaw doctor` checks and optional fixes.

## Best Possible Solution

Treat this as a public plugin-SDK design item: add a narrow additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, focused tests, and docs; start with read-only diagnostics and add repair support only after interactive and unattended-fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API/feature request. The verification path is high-confidence source inspection: current `main` has no public doctor-check registration API or equivalent plugin contribution registry for the main doctor health flow.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam appears maintainable, while local source patches or standalone plugin CLI commands would not provide the integrated, upgrade-safe doctor workflow being requested.
