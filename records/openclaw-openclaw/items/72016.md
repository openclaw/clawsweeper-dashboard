## Summary

Keep open. Current `main` still has the API gap described in the discussion: doctor health checks are assembled as core-only contributions, and plugins do not have a supported registrar for upgrade-safe custom checks or plugin-owned fixes.

## What This Changes

The issue requests an upgrade-safe extension point for custom `openclaw doctor` checks and optional fixes.

## Best Possible Solution

Design an additive public doctor-check SDK contract with plugin attribution, deterministic ordering, duplicate-id handling, docs/tests, and separately gated repair support for plugin-owned fixes.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an API feature request. The gap is high-confidence from source inspection: current `main` has no public doctor-check registrar and runs a static core contribution list.

## Solution Assessment

Unclear until maintainers settle the public API and trust policy. A narrow additive SDK seam is the maintainable direction; local source edits or plugin-owned side commands would not provide the integrated upgrade-safe doctor workflow requested.
