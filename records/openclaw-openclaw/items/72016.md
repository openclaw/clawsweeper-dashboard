## Summary

Keep open. Current main has a structured doctor health-check registry and a public `plugin-sdk/health` subpath, but the automatic doctor bridge is still hard-wired to the bundled Policy plugin and there is no first-class external plugin doctor registration API, so the requested upgrade-safe custom doctor extension surface remains a product and security design item.

## What This Changes

The issue requests a public, upgrade-safe doctor extension API so custom checks and optional fixes can survive OpenClaw upgrades.

## Best Possible Solution

Keep this issue open as the canonical design item for external doctor extensions, evolving the existing `HealthCheck` and `doctor --lint` contracts into an explicit plugin API with safe default linting, scoped loading, and opt-in repair.

## Reproduction Assessment

Yes, source inspection gives a high-confidence gap: current main has internal and bundled health-check plumbing, but no first-class external plugin doctor registration path in `OpenClawPluginApi`. This is not a runtime bug reproduction; it is a source-reproducible missing capability.

## Solution Assessment

No current implementation fully solves the issue. The best path is likely to reuse the existing health-check contracts, but maintainers still need to decide plugin loading, trust boundaries, and repair consent semantics before implementation.
