## Summary

Keep open. Current `main` still has a core-owned static doctor health flow and no public plugin-SDK registration surface for custom doctor checks; this is not a ClawHub-close case because the requested plugin cannot be built cleanly without a new API seam and repair trust policy.

## What This Changes

The issue requests an upgrade-safe extension point for custom `openclaw doctor` checks and optional fixes that can survive npm upgrades.

## Best Possible Solution

Add a narrow additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only diagnostics or explicitly gated repairs, then document interactive and unattended fix behavior before exposing plugin-owned mutation.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. Source inspection provides a high-confidence verification path: current `main` has no public doctor-check registration API and the main doctor health loop is still a fixed core list.

## Solution Assessment

Unclear until maintainers settle the public API and repair trust model. A narrow SDK seam is the maintainable direction, while local source patches or separate plugin CLI commands would not provide the integrated, upgrade-safe doctor workflow requested.
