## Summary

Keep open. Current main still only strips historical OpenAI-compatible reasoning for Gemma 4, while Qwen/oMLX-style `reasoning_content` remains replayable in same-model history; the related attempted repairs are closed unmerged.

## What This Changes

Fix a regression where Qwen/oMLX-style OpenAI-compatible `reasoning_content` from a prior assistant turn is replayed into later conversation history and breaks follow-up requests.

## Best Possible Solution

Add targeted replay-history hygiene for affected Qwen/oMLX-style OpenAI-compatible reasoning fields so historical provider reasoning blocks are removed or downgraded before follow-up requests, while preserving explicit provider contracts that require reasoning replay and same-turn tool-call continuations.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve/sanitize `openai-completions` history for a Qwen/oMLX model with a prior assistant thinking block signed `reasoning_content`; current policy does not enable `dropReasoningFromHistory`, matching the reporter’s two-turn failure flow.

## Solution Assessment

Yes, the best repair direction is a replay-policy fix rather than a generic JSON escaping patch. The existing Gemma 4 path provides a narrow pattern, but the scope needs to cover the affected Qwen/oMLX-style OpenAI-compatible replay path without breaking providers that intentionally preserve reasoning.
