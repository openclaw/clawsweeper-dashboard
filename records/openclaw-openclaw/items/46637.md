## Summary

Keep open. Current main still captures OpenAI-compatible `reasoning_content` as replayable thinking and only enables history stripping for Gemma 4, so the Qwen/oMLX-style same-model replay path described here remains valid. The related attempted repair PRs are closed unmerged, and the issue is narrow enough for a focused fix PR.

## What This Changes

The issue asks OpenClaw to prevent Qwen/oMLX `reasoning_content` from being replayed in conversation history in a way that breaks follow-up OpenAI-compatible requests.

## Best Possible Solution

Add targeted replay-history hygiene for affected Qwen/oMLX-style OpenAI-compatible reasoning fields so prior-turn provider reasoning blocks are removed or downgraded before follow-up requests, while preserving same-turn tool-call continuations and explicit provider contracts that require reasoning replay.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a prior assistant thinking block with `thinkingSignature: "reasoning_content"` into same-model `openai-completions` history for a non-Gemma Qwen/oMLX model and observe that current policy does not enable `dropReasoningFromHistory`; the issue also provides a concrete two-turn Telegram/oMLX flow.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is replay-policy hygiene for affected Qwen-style OpenAI-compatible history, not a generic JSON escaping patch or unconditional stripping across all providers.
