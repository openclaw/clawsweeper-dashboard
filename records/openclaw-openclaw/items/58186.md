## Summary

Keep open. Current main still lacks hook mapping `systemPrompt` support, and this PR remains useful source work, but the latest branch is stale against the moved hook request handler and current cron executor, so it would not reliably deliver Gmail system instructions after rebase.

## What This Changes

The PR adds optional literal `hooks.mappings[].systemPrompt` support, a default Gmail preset system prompt, hook-to-cron payload plumbing, config/docs metadata, tests, and a changelog entry.

## Best Possible Solution

Land a rebased repair that adds literal-only hook mapping `systemPrompt` end to end through the current config schema, Gmail preset, mapped hook handler, cron job payload, isolated cron executor, current public docs page, changelog credit, and focused tests while keeping webhook-derived email content in the user-message/external-content path.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: a mapped Gmail hook expecting a literal `systemPrompt` in the cron payload would fail on current main because the config, mapping action, request handler, cron payload, and executor all omit the field.

## Solution Assessment

No, not as currently rebased. The literal-only `systemPrompt` design is the right security boundary, but the patch must be ported from stale `server-http.ts` and `run.ts` call sites to `src/gateway/server/hooks-request-handler.ts` and `src/cron/isolated-agent/run-executor.ts`.
