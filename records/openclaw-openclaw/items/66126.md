## Summary

Keep open. Current main has partial mitigations for persisted-message continuation and post-compaction repeated-tool aborts, but isolated cron still enters the embedded runner with raw prompt text and no turn/message identity, so ordinary overflow compaction can still retry `params.prompt` instead of resuming a stable cron/system turn.

## What This Changes

The issue asks OpenClaw to stop treating post-compaction cron/system retry prompts as fresh user input that can repeat deterministic tool failures.

## Best Possible Solution

Define one shared compaction-resume contract for cron, system-event, and user turns that preserves turn identity, inbound metadata, and in-flight tool/result state across compaction, with isolated-cron overflow regression coverage.

## Reproduction Assessment

Yes, source-reproducible. The current cron path has no `currentMessageId`, while the runner still falls back to `params.prompt` unless a narrow continuation condition fires.

## Solution Assessment

Unclear. The requested retry-of-turn or reply-obligation-ledger direction matches the missing state, but a cron-only marker patch would be too narrow without a shared compaction-resume contract for related overflow cases.
