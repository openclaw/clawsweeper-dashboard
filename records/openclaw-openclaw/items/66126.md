## Summary

Keep open. Current main has useful mitigations for persisted user-message replay and repeated identical post-compaction tool loops, but isolated cron embedded runs still lack durable current-turn identity and the overflow retry path can still fall back to the original prompt text.

## What This Changes

The issue reports that auto-compaction retry can re-inject cron or system prompts as fresh user input and loop tool calls after context overflow.

## Best Possible Solution

Define a shared compaction-resume contract that preserves durable turn identity, in-flight state, and reply obligations across cron, system-event, and user turns, then cover isolated cron overflow with regression tests.

## Reproduction Assessment

Yes at source level: isolated cron still enters `runEmbeddedPiAgent` without `currentMessageId`, while the runner falls back to `params.prompt` unless mid-turn recovery or a persisted-message guard fires. I did not run a live current-main cron/model loop in this read-only pass.

## Solution Assessment

Unclear. The requested retry-id/message-ledger direction matches the missing state, but the maintainable fix should be a shared compaction-resume contract rather than a cron-only marker patch.
