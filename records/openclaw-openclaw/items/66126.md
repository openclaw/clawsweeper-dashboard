## Summary

Keep open. Current main still rebuilds the embedded-run prompt from `params.prompt` on each retry and continues after successful overflow compaction with `retrying prompt`; the compaction subscriber only resets attempt-local state, not durable turn/message identity. Existing tests still encode repeated post-compaction attempts, and related compaction-resume reports #59618, #63216, and #74257 are still open rather than merged/shipped replacements.

## What This Changes

Prevent compaction retries for cron/system-triggered embedded runs from treating the original prompt as fresh input and re-running deterministic tool-call failures.

## Best Possible Solution

Define a stable embedded-run compaction-resume contract: preserve current turn identity and original inbound message metadata across compaction, mark resumed attempts distinctly from fresh cron/system/user input, avoid replaying deterministic failed tool cascades as new work, and cover cron/system-event overflow cases with regression tests shared with the related compaction-resume reports.

## Reproduction Assessment

Yes. The issue body and follow-up comment provide live log reproduction, and current tests/source provide a focused reproduction path: trigger context overflow during an embedded run, allow compaction to succeed, and observe the same prompt retried through the outer loop.

## Solution Assessment

Unclear. The proposed retry-of-turn/message-ledger direction is likely the right shape, but maintainers should choose the cross-surface resume contract before implementing a cron-only or prompt-marker patch.
