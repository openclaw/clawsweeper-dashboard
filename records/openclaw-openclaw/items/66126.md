## Summary

Keep open. Current main has added useful mitigations for persisted user-turn replay and identical post-compaction tool loops, but the cron embedded-run path still lacks a durable current-turn identity and the overflow retry loop can still fall back to the original prompt text.

## What This Changes

The issue asks OpenClaw to stop treating post-compaction retries for cron/system prompts as fresh user input that can repeat deterministic tool failures indefinitely.

## Best Possible Solution

Define a shared compaction-resume contract that preserves durable turn identity, in-flight state, and reply obligations across cron, system-event, and user turns, then cover isolated cron overflow with focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main still lets cron enter the embedded runner without currentMessageId, while the runner falls back to params.prompt after compaction unless mid-turn recovery or the persisted-message guard fires.

## Solution Assessment

Unclear. The requested retry-id/message-ledger direction matches the missing state, but the maintainable fix should be a shared compaction-resume contract rather than a cron-only marker patch.
