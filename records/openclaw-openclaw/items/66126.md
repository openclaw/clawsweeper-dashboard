## Summary

Keep open. Current main still rebuilds and retries the original prompt for ordinary overflow compaction; only the narrower mid-turn precheck path now switches to a continuation prompt, so the reported cron/system reinjection bug remains valid.

## What This Changes

Stop auto-compaction retry from treating an isolated cron/system prompt as a fresh user turn after overflow recovery.

## Best Possible Solution

Define one shared compaction-resume contract that preserves current turn identity, inbound message metadata, and in-flight tool/result state across cron, system-event, and user overflow recovery, then cover it with focused embedded-runner and isolated-cron regression tests.

## Reproduction Assessment

Yes, source-reproducible. The issue logs match current log strings, and current main still has a focused path where overflow compaction succeeds, the loop continues, and the next attempt defaults back to `params.prompt` unless it is the narrow mid-turn precheck case.

## Solution Assessment

Unclear. The requested retry-of-turn or reply-obligation-ledger direction matches the missing state, but a cron-only marker patch would be too narrow; maintainers should settle the shared compaction-resume semantics first.
