## Summary

Keep open. Current main has a partial persisted-message continuation fix and a post-compaction loop guard, but isolated cron still invokes the embedded runner with raw prompt text and no current message id, so ordinary overflow compaction can still fall back to retrying `params.prompt` rather than resuming a stable cron/system turn.

## What This Changes

The issue asks OpenClaw to stop treating post-compaction cron/system retries as fresh prompts so failed tool-call turns do not re-enter an overflow loop.

## Best Possible Solution

Define one shared compaction-resume contract for cron, system-event, and user turns that preserves turn identity, inbound metadata, and in-flight tool/result state across compaction, then cover isolated-cron overflow with regression tests.

## Reproduction Assessment

Yes, source-reproducible. The issue logs match current retry log strings, and current main still has a cron path where no `currentMessageId` is available, so overflow compaction falls back to `params.prompt` unless a narrower continuation condition fires.

## Solution Assessment

Unclear. The requested retry-of-turn or reply-obligation-ledger direction matches the missing state, but a cron-only marker patch would be too narrow without a shared compaction-resume contract for the related overflow cases.
