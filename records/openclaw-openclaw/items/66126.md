## Summary

Keep open. Current main still retries ordinary overflow compaction by rebuilding the attempt from `params.prompt`; only the newer mid-turn precheck path gets a continuation prompt. The cron executor still passes fixed cron prompt text into the embedded runner, the compaction subscriber only resets attempt-local state on `willRetry`, and related compaction-resume reports remain open rather than merged or shipped replacements.

## What This Changes

The issue asks OpenClaw to make post-compaction retry/resume turn-stable so cron/system prompts are not replayed as fresh user input after overflow compaction.

## Best Possible Solution

Define a shared embedded-run compaction-resume contract: preserve current turn identity and original inbound message metadata across compaction, resume from existing transcript/tool-result state when work already started, distinguish resumed attempts from fresh cron/system/user input, and add regression coverage for cron/system-event overflow alongside #59618, #63216, and #74257.

## Reproduction Assessment

Yes. The issue body provides live logs, and current source/tests provide a focused reproduction path: trigger embedded-run context overflow, allow compaction to succeed, and observe the same prompt retried through the outer loop for up to three compaction attempts.

## Solution Assessment

Unclear. The proposed retry-of-turn/message-ledger direction matches the missing state, but maintainers should settle the cross-surface resume semantics before taking a cron-only marker patch.
