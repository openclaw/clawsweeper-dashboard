## Summary

Keep open. Current main has useful mitigations for persisted user-message replay and repeated post-compaction tool loops, but the cron/system path still lacks durable turn identity and can still fall back to retrying the original prompt text after overflow compaction.

## What This Changes

The issue reports that overflow compaction retries in cron/system sessions can re-inject the original prompt as fresh input, causing repeated tool calls and lane-blocking loops.

## Best Possible Solution

Define a shared compaction-resume contract for cron, system-event, and user turns that preserves turn identity and in-flight state across compaction, then cover isolated cron overflow with regression tests.

## Reproduction Assessment

Yes, source-reproducible. The current cron path enters the embedded runner without currentMessageId, while the runner still falls back to params.prompt unless mid-turn recovery or a persisted-message guard fires.

## Solution Assessment

Unclear. The requested retry-of-turn or reply-obligation-ledger direction matches the missing state, but the maintainable fix should be a shared compaction-resume contract rather than a cron-only marker patch.
