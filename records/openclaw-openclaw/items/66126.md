## Label Justifications

- `P1`: The report has credible source-level support for repeated compaction retry loops that can stall real multi-channel agent workflows for minutes.
- `impact:session-state`: The failure is about compaction, retry, transcript continuation, and current-turn identity becoming detached from the original cron/system turn.
- `impact:message-loss`: The retry loop can replay or misclassify inbound work and prevent normal channel replies from being delivered while the lane is blocked.
- `impact:crash-loop`: The reported behavior is an availability loop with repeated compaction/retry cycles and long lane waits that make channels appear hung.

## Summary

Keep open. Current main has partial replay prevention for persisted user turns, but isolated cron runs still enter the embedded runner without a durable current-turn identity and the overflow retry path can still reuse the original prompt; related open compaction issues and the open suppress-hook PR do not supersede this cron/system turn-state report.

## What This Changes

The issue reports that overflow auto-compaction retries cron/system turns as fresh prompts, replaying failed tool cascades and blocking channels.

## Best Possible Solution

Define a shared compaction-resume contract that preserves durable turn identity, in-flight state, and reply obligations across cron, system-event, and user turns, then cover isolated cron overflow with focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main still lets isolated cron enter the embedded runner without currentMessageId, while the runner falls back to params.prompt after compaction unless a mid-turn or persisted-message guard fires; I did not run a live cron loop.

## Solution Assessment

Unclear. The requested retry-id/message-ledger direction matches the missing state, but the maintainable fix should be a shared compaction-resume contract rather than a cron-only marker patch.
