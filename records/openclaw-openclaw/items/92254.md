## Label Justifications

- `P1`: The PR targets a real auth/session regression in channel model pickers, but the current patch still leaves one main Discord path broken and can mis-preserve auth routing.
- `merge-risk: 🚨 auth-provider`: The diff changes when `authProfileOverride` is retained, which can affect provider credential selection after a model picker switch.
- `merge-risk: 🚨 session-state`: The patch writes persisted session model/auth override state and can leave the session with either deleted or incompatible auth profile state depending on the picker path.

## Summary

Keep this PR open: the Telegram direct-persist change is useful, but the Discord normal hidden-command path still reaches the destructive helper without preserve semantics, and the new unconditional preserve flag can retain an incompatible auth profile after a cross-provider model switch.

## What This Changes

The PR adds `preserveAuthProfileOverride: true` to Telegram and Discord model picker direct-persist paths and adds a Discord regression test for preserving an auth profile override during picker selection.

## Best Possible Solution

Route Telegram direct persist, Discord direct persist, and Discord hidden `/model` persistence through one provider-aware auth-profile preservation policy, then add regression coverage for both direct and successful hidden-command paths plus live Telegram callback proof.

## Reproduction Assessment

Yes, by source inspection: current main deletes auth profile fields when the helper is called without `profileOverride` or `preserveAuthProfileOverride`, and the picker/directive paths show those call shapes. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is a plausible partial fix, but the best fix must cover Discord's successful hidden-command path and should use provider-aware preservation rather than unconditional `true`.
