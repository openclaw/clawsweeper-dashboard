## Summary

Keep open. Current main still has a source-reproducible lossy-success path: the full child result is captured and placed in the completion handoff, but requester-side direct delivery treats any visible requester payload as delivered, so a short prefix can suppress fallback without an integrity signal.

## What This Changes

Ensure sub-agent completion announces preserve or signal the full child result instead of silently delivering only a short prefix to the requester session.

## Best Possible Solution

Keep the typed completion handoff, add deterministic integrity detection for prefix-only or materially incomplete requester output, and fall back to the already-captured child result with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Telegram reproduction was established, but source inspection shows a current-main path where a short visible requester payload is accepted as successful without comparing it to the captured child result.

## Solution Assessment

Is this the best way to solve the issue? No; the current implementation is not sufficient because it treats any visible requester-agent output as complete delivery. A narrow integrity check plus fallback to the frozen/internal child result is the safer maintainable fix.

Current main captures the full child result, but src/agents/subagent-announce-delivery.ts treats any visible requester-agent output as successful and only falls back to the child result when there is no visible output.
