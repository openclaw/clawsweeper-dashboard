## Summary

Keep open. Current main still has a source-reproducible lossy-success path: the full child result is captured and placed in the internal completion event, but requester-side direct delivery treats any visible requester payload as delivered, so a prefix-only announce can suppress fallback without an integrity signal.

## What This Changes

The issue requests lossless or detectable sub-agent completion announce delivery when the requester session receives only a short prefix of the child result body.

## Best Possible Solution

Add deterministic completion-delivery integrity detection for prefix-only or materially incomplete requester output, then fall back to the already-captured child result with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Telegram reproduction was established, but source inspection shows a current-main path where a short visible requester payload is accepted as successful without comparing it to the captured child result.

## Solution Assessment

Is this the best way to solve the issue? No; the current implementation is not sufficient because it treats any visible requester-agent output as complete delivery. A narrow integrity check plus fallback to the frozen/internal child result is the safer maintainable fix.
