## Summary

Keep open. Current main still has the reported lossy-success path, and the issue is already paired with open draft PR #76528 rather than something that should be closed before merge.

## What This Changes

The issue reports that sub-agent completion announcements can arrive as only a short prefix of the final result while the full child transcript remains recoverable through session history.

## Best Possible Solution

Land or replace #76528 with a focused integrity check that detects prefix-only or materially incomplete requester output and falls back to the captured child result with regression coverage and docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Telegram reproduction was established, but source inspection gives a high-confidence path: any non-empty requester payload suppresses fallback without checking whether it is only a prefix of the captured child result.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable path is to preserve the requester-agent primary delivery while adding completion-body integrity detection and falling back to the frozen child result; #76528 appears to target that boundary but is still open.
