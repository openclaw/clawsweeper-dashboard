## Summary

Keep open. Current main still has a source-reproducible lossy-success path: the full child result is captured and carried in the completion event, but requester-side direct delivery treats any visible requester payload as delivered without checking whether it is only a prefix or materially incomplete.

## What This Changes

The issue reports that completed sub-agent announce events sometimes deliver only a short prefix of the final result to the requester session even though the full sub-agent transcript remains recoverable via sessions_history.

## Best Possible Solution

Add deterministic completion-delivery integrity detection for prefix-only or materially incomplete requester output, then fall back to the already-captured child result with focused regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established, but source inspection gives a clear current-main path: any non-empty requester payload marks completion delivery successful even when the captured child result is longer and available for fallback.

## Solution Assessment

No; the current implementation is not the best solution because it only detects missing output, not incomplete output. The narrower maintainable fix is an integrity check against the internal completion result plus fallback to that result when delivery is materially incomplete.
