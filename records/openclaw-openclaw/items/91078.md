## Summary

## What This Changes

The branch adds a Codex app-server startup helper and regression tests so OpenClaw sandbox exec-server runs start/resume Codex threads with `danger-full-access` while preserving the existing sandbox policy for non-exec-server runs.

## Best Possible Solution

Keep the override tied to the existing sandbox exec-server predicate, land only after CI and sandbox-boundary review, and prefer one live sandboxed Codex app-server smoke before merge if maintainers want end-to-end proof.

## Reproduction Assessment

Yes, from source plus the PR's incident logs: current main can start/resume a Codex thread with `workspace-write` while the turn uses OpenClaw `externalSandbox`, matching the nested-sandbox failure mode. I did not run a live deployed gateway turn myself.

## Solution Assessment

Yes, this looks like the best narrow fix: it reuses the existing `shouldRequireCodexSandboxExecServerEnvironment` guard instead of changing global Codex defaults or adding another config path. The safer alternative would be only to add an end-to-end sandbox smoke before landing, not to move the fix elsewhere.
