## Summary

## What This Changes

The PR adds a Codex app-server startup resolver that switches the app-server thread sandbox to `danger-full-access` only when OpenClaw sandbox exec-server owns native tools, with focused unit coverage.

## Best Possible Solution

Keep the guarded startup override if maintainers accept the Codex external-sandbox contract, update the Codex harness reference for this exception, and preferably add one deployed sandboxed Codex native `apply_patch` smoke before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can still select a Codex `workspace-write` thread sandbox for active OpenClaw sandbox runs, while the PR proof shows the real-config resolver flips only the sandbox exec-server path. I did not run a full current-main live Gateway reproduction.

## Solution Assessment

Mostly yes: the startup resolver is the right owner boundary because thread start/resume params own the Codex thread sandbox and the existing exec-server path already owns `externalSandbox` turn policy. The remaining issue is docs and maintainer acceptance of the security-boundary tradeoff, not a different code layer.
