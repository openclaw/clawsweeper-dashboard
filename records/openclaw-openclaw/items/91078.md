## Summary

Keep open: the code change is a plausible fix for the nested Codex filesystem sandbox path, but it still needs the Codex sandbox docs updated and explicit maintainer acceptance of the security-boundary change before merge.

## What This Changes

The PR adds a guarded Codex app-server startup resolver that switches the thread sandbox to `danger-full-access` only when OpenClaw sandbox exec-server owns native tool execution, with focused unit coverage.

## Best Possible Solution

Keep the guarded resolver if maintainers accept the upstream Codex external-sandbox contract, update the Codex harness docs for this exception, and use a live sandboxed Codex smoke if maintainers want stronger merge proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the current-main source path can still select `workspace-write` for sandboxed Codex exec-server startup while the PR proof shows the resolver flipping that path, but I did not establish a full live current-main `apply_patch` failure in this checkout.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: the startup resolver is the right owner boundary for the Codex thread sandbox, while the existing exec-server path already owns environment selection and `externalSandbox` turn policy; the missing piece is docs and maintainer signoff.
