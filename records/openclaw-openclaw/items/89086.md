## What This Changes

The PR prefixes HTTP status codes onto node-host browser proxy errors so downstream stale-target classification can see messages like `404: tab not found`.

## Best Possible Solution

Land the narrow producer-side formatter after redacted patched gateway plus remote browser-node proof shows the status-preserved 404 crosses `node.invoke` and triggers the existing stale-target retry, or after maintainers explicitly accept the split proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current `runBrowserProxyCommand` strips the 404 status while `isChromeStaleTargetError` requires `404:` plus `tab not found`; I did not run a live remote browser node in this read-only review.

## Solution Assessment

Yes for the code path. Preserving the status at the node proxy producer is the narrow maintainable fix; broadening the consumer classifier would weaken the existing HTTP-status guard and risk matching non-HTTP tab-not-found strings.
