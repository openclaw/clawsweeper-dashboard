## Summary

Keep open: current main and the latest release still contain the MiniMax M3 `medium` to `adaptive` fallback path, and the only linked fix PR was closed unmerged as unsafe because `off` may disable MiniMax M3 output.

## What This Changes

The issue asks OpenClaw to stop unsupported fixed-budget thinking levels such as `medium` from silently resolving to provider-managed `adaptive` on adaptive-only profiles like MiniMax M3.

## Best Possible Solution

Land a maintainer-approved fallback policy that prevents silent adaptive cost escalation while proving the final MiniMax M3 payload or live response does not become disabled or empty.

## Reproduction Assessment

Yes for the source path: current main still lets MiniMax M3 expose `off|adaptive`, ranks `adaptive` equal to `medium`, and applies the shared fallback in cron/runtime paths. A live MiniMax token-cost measurement was not run here.

## Solution Assessment

No approved fix shape exists yet. The shared resolver is the right choke point, but the closed candidate showed that resolving MiniMax M3 `medium` to `off` needs provider payload or live proof so it does not become disabled/empty output.
