## Summary

Keep open: current main and v2026.6.6 still have the reported MiniMax M3 fallback path, and the only linked fix PR was closed unmerged after maintainer review because the proposed `medium` to `off` fallback could make M3 unusable.

## What This Changes

Prevent stored or default fixed-budget thinking levels such as `medium` from silently resolving to provider-managed `adaptive` on MiniMax M3 and other adaptive-only thinking profiles.

## Best Possible Solution

Adopt a maintainer-approved, cost-safe adaptive-only fallback policy that never silently escalates fixed-budget requests to `adaptive`, while proving MiniMax M3 still returns visible output for supported default and explicit thinking paths.

## Reproduction Assessment

Yes for the source path: current main still maps MiniMax M3 `medium` fallback to `adaptive` through the shared resolver. No live MiniMax run was performed to verify the exact token-cost multiplier.

## Solution Assessment

Unclear: the shared resolver/profile boundary is the right area, but the closed linked PR showed that resolving fixed budgets to `off` is not proven safe for MiniMax M3. The best fix needs maintainer fallback policy plus live or payload-level MiniMax proof.
