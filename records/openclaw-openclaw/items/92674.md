## Summary

Keep open: current main and v2026.6.8 still contain the reported adaptive-only fallback behavior, and the linked attempted fix at https://github.com/openclaw/openclaw/pull/92803 was closed unmerged because resolving MiniMax-M3 fixed-budget requests to `off` can produce disabled or empty-output behavior.

## What This Changes

The issue asks OpenClaw to stop silently remapping unsupported fixed-budget thinking requests such as `medium` to provider-managed `adaptive` on adaptive-only profiles like MiniMax-M3.

## Best Possible Solution

Choose a provider-safe fallback policy that prevents silent fixed-budget-to-adaptive cost escalation without sending disabled MiniMax-M3 requests, then back it with resolver, payload-path, docs, and live or redacted MiniMax proof.

## Reproduction Assessment

Yes for the source path: current main shows MiniMax-M3 exposes `off|adaptive`, `adaptive` ranks equal to `medium`, and cron applies the shared fallback. I did not live-run MiniMax to verify the reporter's exact token-cost multiplier.

## Solution Assessment

No, the closed linked PR's `medium -> off` fallback alone is not the best solution because it can route MiniMax-M3 into disabled-thinking empty output. The right solution needs a maintainer-approved provider-safe fallback and final payload or live proof.
