- `impact:message-loss`: The issue is about duplicated stale Slack media being delivered as if it belonged to each text-only reply.

## Summary

Keep open. Current main and v2026.5.19 still have the source-level Slack media replay path, and the open closing PR remains the paired fix path but is conflicting and needs a maintainer decision on the documented root-media inheritance contract.

## What This Changes

The issue reports Slack text-only thread replies being sent to the model with the parent message's media reattached as fresh inbound media.

## Best Possible Solution

Land a rebased Slack fix or replacement that stops repeated parent-media rehydration, preserves reply-owned media and intentional root-media context unless maintainers explicitly change the contract, and updates tests/docs for the final behavior.

## Reproduction Assessment

Yes, source-level: current main hydrates `starter.files` when a Slack thread reply has no direct media and then forwards `threadStarterMedia` as effective inbound media. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Unclear. The issue is real, but the linked never-hydrate fix is broader than the documented Slack root-media contract; a narrower rebased fix or an explicit maintainer-approved contract change is safer.
