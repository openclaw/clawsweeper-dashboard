## Label Justifications

- `P2`: This is a real Slack media/context bug with limited channel-specific blast radius and an existing linked fix path.
- `impact:message-loss`: The issue is about duplicated stale Slack media being delivered as if it belonged to each text-only reply.
- `impact:session-state`: The model receives stale parent media as current turn context, causing thread context drift across replies.

## Summary

Keep open. Current main and the latest tagged release still hydrate Slack thread-starter files for text-only thread replies and forward them as inbound media, while the linked closing PR remains open and conflicting.

## What This Changes

The issue asks Slack thread replies to include only media attached to the current reply instead of re-attaching parent-message media on every text-only reply.

## Best Possible Solution

Land a rebased Slack fix that stops repeated starter-media rehydration after the initial thread context is recorded, preserves reply-owned media and intentional first-turn root-media context, and updates the Slack docs to match the final contract.

## Reproduction Assessment

Yes. Source inspection on current main shows a text-only Slack thread reply with starter files and no direct media hydrates `starter.files` into `threadStarterMedia`, then forwards that fallback into inbound media.

## Solution Assessment

Unclear as requested. The broad never-hydrate direction is not the safest default because current docs support first-turn root-media inheritance; the maintainable fix should stop repeated hydration or explicitly update the contract.
