## Summary

Keep open. Current main still hydrates Slack thread-starter files for text-only thread replies and forwards them as inbound media, while the focused closing PR remains open and unmerged.

## What This Changes

Slack thread replies with no media should not re-download or attach media from the parent thread-starter message.

## Best Possible Solution

Land #60353 or a narrow equivalent in the Slack plugin so parent media remains available only as thread history/context while direct reply media is still delivered normally.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a text-only Slack thread reply yields null direct media, hydrates `starter.files`, and forwards that fallback as the inbound media payload.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the Slack plugin's inbound thread assembly: stop hydrating thread-starter media for replies while preserving starter text, thread history, and reply-owned media.
