## Summary

Keep open; this PR should not merge as-is because it turns the auto-generated provider hostname allowlist into a private-network SSRF bypass, carries unrelated active-memory and doctor changes, and has only unit-test proof.

## What This Changes

The PR branch updates SSRF private-network skip logic to treat `hostnameAllowlist` as a bypass, adds SSRF tests, and also carries unrelated active-memory and doctor migration changes.

## Best Possible Solution

Keep `hostnameAllowlist` as host scoping, preserve explicit `request.allowPrivateNetwork` as the private/internal network opt-in, and land a focused SSRF regression only if a live Docker/local-provider repro still fails with that opt-in.

## Reproduction Assessment

Yes for the patch risk: source inspection shows provider `baseUrl` creates `hostnameAllowlist`, and this PR makes that list skip reserved/private checks. No high-confidence after-fix real Docker/provider reproduction is included in the PR.

## Solution Assessment

No. The narrower maintainable solution is to keep explicit `allowPrivateNetwork` as the trust boundary and add regression coverage for `host.docker.internal` that also preserves metadata/internal-host protections.
