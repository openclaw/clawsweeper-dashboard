## Summary

The linked spawn failure is source-supported, but this PR does not preserve the caller boundary described in its body: interactive and programmatic callers currently share the same `thinking` field, so the patch would also silently downgrade unsupported user-entered values. Focused runtime proof is also still missing.

## What This Changes

The PR changes unsupported thinking-level handling in `agentCommand` so only `thinkingOnce` throws and `thinking` overrides use the existing supported-level fallback.

## Best Possible Solution

Add a typed internal source or fallback policy that only programmatic spawn/session callers can select, retain strict validation for user-entered `thinking`, and cover both paths with focused tests plus a redacted runtime spawn transcript.

## Reproduction Assessment

Yes, at source level: `sessions_spawn` sends its override as gateway `thinking`, which becomes `thinkOverride` and currently enters the unsupported explicit-level throw. Neither the failing nor fixed child run was executed in the supplied evidence.

## Solution Assessment

No. Guarding on `thinkingOnce` does not distinguish interactive and programmatic callers because production callers currently use `thinking`; the maintainable fix needs an explicit caller policy or owner-boundary normalization.
