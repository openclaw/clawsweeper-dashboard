## Summary

Keep this PR open. Current main still lacks the requested Mantle `discovery.enabled` opt-out, and the PR is a focused implementation candidate for the linked bug, but the patch should be updated before merge because it gates on a registration-time config snapshot rather than the runtime config pattern already used by the sibling Bedrock provider.

## What This Changes

This PR adds an `amazon-bedrock-mantle` manifest config/UI hint for `discovery.enabled` and returns `null` from the Mantle provider catalog handler when the captured plugin config disables discovery.

## Best Possible Solution

Land an updated Mantle-owned fix that adds the manifest config schema/UI hint, resolves current plugin config from `ctx.config` with the same fallback pattern as `amazon-bedrock`, passes that config into `resolveImplicitMantleProvider`, exits before IAM token generation when `discovery.enabled === false`, and documents/tests the disabled, enabled, and unset behaviors.

## Reproduction Assessment

Yes. The underlying bug is reproducible from source inspection and the linked report: on current main the catalog path always calls the resolver, and the resolver attempts IAM bearer-token generation when no explicit Mantle token is present, producing the reported discovery log when generation is unavailable.

## Solution Assessment

No, not as-is. The requested direction is right, but the patch should use the runtime plugin config pattern and put the gate at the resolver boundary so the opt-out consistently prevents IAM token generation across catalog and direct resolver callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Read Mantle discovery config from runtime config:** `extensions/amazon-bedrock-mantle/register.sync.runtime.ts:24-26`
  - body: The gate checks the `api.pluginConfig` value captured when the plugin registered, but provider catalog execution already receives the current config in `ctx.config`. After config reloads or included plugin config changes, setting `plugins.entries.amazon-bedrock-mantle.config.discovery.enabled: false` can leave this captured object unset and the handler will still call `resolveImplicitMantleProvider`, so IAM token generation and the reported log spam continue. Mirror the sibling Bedrock provider's runtime config lookup and pass the config into the resolver before token generation.
  - confidence: 0.87
