## Summary

Keep open. The reported Mantle discovery log spam is valid and current main still lacks the opt-out, but this PR is not ready because it reads the startup `api.pluginConfig` snapshot and gates only the catalog handler instead of using live runtime config and stopping at the resolver boundary.

## What This Changes

This PR adds an `amazon-bedrock-mantle` manifest config/UI hint for `discovery.enabled` and skips catalog discovery when the captured plugin config disables it.

## Best Possible Solution

Land a Mantle-owned fix that adds the manifest config/UI hint, reads live plugin config from `ctx.config` with the startup fallback pattern, passes it into `resolveImplicitMantleProvider`, returns before IAM token generation when disabled, and covers docs, changelog, and regression tests.

## Reproduction Assessment

Yes. Current main source inspection and the linked report show the catalog path always calls the resolver, and the resolver attempts IAM token generation when no explicit Mantle token exists; the PR-specific bug reproduces when runtime config changes after registration or the resolver is called outside the catalog hook.

## Solution Assessment

No. The requested opt-out is the right direction, but this patch should use the same live runtime-config pattern as `amazon-bedrock` and put the opt-out at the resolver boundary so every discovery path avoids IAM token generation.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Move the Mantle discovery gate into the resolver:** `extensions/amazon-bedrock-mantle/register.sync.runtime.ts:24-26`
  - body: The new guard checks the `api.pluginConfig` snapshot captured at plugin registration, but provider catalog execution receives current config in `ctx.config` and `resolveImplicitMantleProvider` is still callable without any opt-out. After a config reload, or from any direct resolver caller, setting `discovery.enabled: false` can still reach IAM token generation and produce the log spam this PR is meant to suppress. Mirror the sibling Bedrock runtime-config lookup and return from the resolver before token generation.
