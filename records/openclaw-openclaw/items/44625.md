## Summary

Close as implemented on main. Current Slack capability diagnostics read granted scopes from `auth.test` response metadata before the legacy Slack scope APIs, and the reported CLI path now goes through that helper; this is present on current `main` but not in the latest local release tag `v2026.4.29`.

## What This Changes

The issue asks for `openclaw channels capabilities` to report Slack scopes normally for modern `xoxb-*` bot tokens instead of returning `auth.scopes` and `apps.permissions.info` `unknown_method` errors.

## Best Possible Solution

Keep the Slack plugin-local `auth.test` scope discovery on main and let the next OpenClaw release ship it; no core channel API change is needed.

## Reproduction Assessment

Yes. The issue has concrete CLI steps and a follow-up reproduction against the old helper path, while current main has a focused regression test for the modern bot-token `auth.test` response metadata path.

## Solution Assessment

Yes. A Slack plugin-local scope discovery change is the narrowest maintainable fix because the existing capabilities path already delegates to the plugin helper and the Slack SDK exposes granted scopes in response metadata.
