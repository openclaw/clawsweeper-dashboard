## Summary

Keep open: the interrupt handling direction is valid and the supplied terminal proof is useful, but the latest patch now drops the existing dashboard launch after successful git installs because that path bypasses the final doctor block where the flag is set.

## What This Changes

The PR adds installer INT/TERM abort traps, handles final doctor exit code 130 as an interruption, and moves dashboard-opening intent into successful doctor handling.

## Best Possible Solution

Land a status-aware installer change that aborts cleanly on INT/TERM, treats doctor exit 130 as interruption, and opens the dashboard only after successful paths that currently open it, including successful git installs.

## Reproduction Assessment

Yes. Source inspection shows current main lacks INT/TERM traps and uses doctor command control flow that can continue after interruption; the PR body also supplies before/after terminal proof for SIGINT during the npm install path, though I did not run the live interactive doctor flow.

## Solution Assessment

No. The trap and exit-code direction is appropriate, but the latest dashboard gating is too broad because it removes the successful git-install dashboard path; the safer fix is status-aware gating rather than deleting the earlier flag unconditionally.
