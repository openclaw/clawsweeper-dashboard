## Summary

Keep open: the patch improves installer interrupts when the child process actually dies from SIGINT, but source inspection shows the interactive doctor cancellation path can still exit 0 and be treated as success, leaving the reported upgrade-doctor/dashboard behavior unresolved.

## What This Changes

The PR adds INT/TERM trapping and doctor exit-code handling in `scripts/install.sh`, gates dashboard launch on doctor success, and adds a changelog fix entry.

## Best Possible Solution

Keep the installer-owned fix, but make doctor cancellation surface as a non-success installer-visible state before setting `doctor_ok` or opening the dashboard, then rebase onto current `main`.

## Reproduction Assessment

Yes for source-level reproduction: current `main` treats successful doctor exit as success and doctor prompt cancellation can exit 0. I did not run a live `curl | bash` upgrade-doctor session in this read-only review.

## Solution Assessment

No. The shell trap and dashboard gating are the right area, but the PR still accepts doctor exit 0 as success even though prompt cancellation currently exits 0.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not treat cancelled doctor runs as successful:** `scripts/install.sh:2997-2998`
  - body: The interactive doctor path can still return 0 when a Clack prompt is cancelled because `guardCancel` calls `runtime.exit(0)`. In that case the parent shell trap is not enough, this branch sets `doctor_ok=1`, and the installer can proceed to plugin updates/dashboard open after the user pressed Ctrl+C. Make doctor cancellation surface as 130 or another installer-detectable cancelled state before marking the doctor run successful.
  - confidence: 0.88
