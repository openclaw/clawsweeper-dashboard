## Summary

Keep this PR open, but do not merge it as-is: it addresses a real installer bug, yet the proposed doctor handling can still treat the reported interactive Clack cancellation path as a successful doctor run and proceed to dashboard/plugin work.

## What This Changes

The PR adds INT/TERM handling to `scripts/install.sh`, preserves the doctor subprocess exit status before plugin updates/dashboard opening, and adds a changelog fix entry.

## Best Possible Solution

Make doctor cancellation observable to the installer even when Clack returns a zero-exit cancel path, then prove an interactive upgrade doctor Ctrl+C exits cleanly without plugin updates or dashboard opening.

## Reproduction Assessment

Yes by source inspection: current main has only an EXIT cleanup trap, doctor prompt cancellation routes through `guardCancel` with exit 0, and the installer treats a zero-exit doctor as successful. I did not run a live TTY upgrade in this read-only review.

## Solution Assessment

No. The patch catches INT/TERM and doctor exit 130, but the reported interactive doctor cancellation can still return 0, so the installer needs a cancellation signal/exit contract that cannot be confused with doctor success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Don't treat cancelled doctor prompts as success:** `scripts/install.sh:2997-2998`
  - body: The failing path is Ctrl+C inside interactive `openclaw doctor`: current doctor cancellation goes through Clack and `guardCancel`, which exits 0. With this branch, `doctor_exit == 0` still sets `doctor_ok=1`, so a cancelled prompt can proceed to plugin updates and dashboard opening; the parent shell trap is not a reliable fallback when the foreground child handles SIGINT and exits 0. Make doctor cancellation observable or non-zero before marking doctor success.
