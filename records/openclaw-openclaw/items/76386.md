## Summary

Keep open: the PR is a focused installer bug fix with useful proof, but the latest proposed merge still has blocking control-flow issues. The upgrade path can still carry a dashboard-open flag from the earlier migration doctor into a later failed final doctor, and the new nonzero `run_doctor` return can abort a fresh install with existing config at an unguarded call site.

## What This Changes

The branch adds installer INT/TERM abort handlers, changes `run_doctor` to return status, and moves some dashboard-open flags behind doctor success checks.

## Best Possible Solution

Keep the signal trap, but make dashboard launch depend on the specific doctor/onboarding branch that actually completed successfully and update every `run_doctor` call site for the new status-return contract.

## Reproduction Assessment

Yes, at source level. Current main lacks an INT/TERM trap and has doctor/dashboard control flow that can continue after failed doctor handling; the PR body also supplies terminal before/after proof for SIGINT during install, though I did not run the live interactive doctor flow.

## Solution Assessment

No. The trap is the right direction, but the proposed merge is not yet the narrow safe fix because it leaves dashboard intent stale across doctor phases and changes `run_doctor` failure behavior without updating all call sites.
