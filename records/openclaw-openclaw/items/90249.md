## Summary

Keep open: the core Google shared converter fix is plausible, but review found a supported bundled Google transport path that still reads the same tool schema property directly and would retain the unreadable-schema crash behavior.

## What This Changes

The PR changes Google/Gemini tool conversion to skip unreadable tool schemas, preserve valid sibling declarations, and omit toolConfig when no declarations remain.

## Best Possible Solution

Use one shared Google tool-declaration conversion behavior, or equivalent tests in both supported Google text paths, so unreadable schemas are skipped consistently and all-skipped requests omit both tools and toolConfig.

## Reproduction Assessment

Yes from source: current main directly reads tool.parameters while building Google function declarations, so a throwing getter can abort conversion before healthy sibling tools are sent. I did not run tests because this was a read-only review.

## Solution Assessment

No: the proposed fix improves the core shared provider path, but the registered bundled Google transport plugin keeps the same direct schema read and original tool-count toolConfig gating.
