## Best Possible Solution

Route the skills CLI through configured remote Gateway resolution, return the complete `SkillStatusReport`, preserve local/offline fallback, and add focused tests plus redacted real Gateway proof.

## Reproduction Assessment

## Solution Assessment

No. The proposed direction is right, but this implementation passes a configured remote URL as a CLI override, so configured credentials are blocked and the central fix usually falls back to the old local status path.

- **[P1] Let configured remote auth flow through:** `src/cli/skills-cli.ts:114`
  - body: Passing `gatewayUrl` as `opts.url` turns the configured remote URL into a CLI URL override. The Gateway client deliberately refuses URL overrides without explicit credentials and credential resolution returns `{}` for non-env overrides, so this throws before `skills.status` runs and the catch falls back to the old local-only status path. Let the Gateway call resolve the configured remote target/credentials itself instead.
  - confidence: 0.93

  - body: This branch constructs `{ skills, workspaceDir }`, but `SkillStatusReport` requires `managedSkillsDir` and the JSON formatters read that field. Returning a partial report breaks the typed contract and drops report metadata; return the Gateway report after validating the expected shape, or include the required fields.
  - confidence: 0.88
