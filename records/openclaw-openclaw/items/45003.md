## Label Justifications

- `P2`: The issue affects session-history recovery and continuity for reset or upgraded users, but the core runtime still operates and the data is usually still on disk until retention pruning.
- `impact:session-state`: Archived transcript fragments can become unreachable through normal history tools, breaking session and context continuity after resets.
- `impact:data-loss`: Users experience vanished history, and retention pruning can later turn undiscovered reset archive files into actual transcript loss.

## Summary

Keep open. Current main and latest release still lack a settled archive/family history contract for normal `chat.history` and `sessions_history`, while overlapping open PRs cover competing missing-active, include-archived, and family-history directions that need maintainer API/product selection.

## What This Changes

Request a recovery utility or built-in path to restore or re-index `.jsonl.reset.*` session transcript archives so reset history remains accessible after daily/manual resets.

## Best Possible Solution

Define and land one canonical recovery contract: preferably a read-only Gateway/tool history path with ordered segment/source metadata, plus a mutating doctor/restore path only if maintainers explicitly approve that workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_history` and `chat.history` still have no archive/family selector and read the currently mapped active transcript; I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No. The proposed mutating script is a useful emergency workaround, but the best durable solution is a maintainer-approved read-only archive/family history contract with stable metadata, with restore/re-indexing handled only through an explicit doctor-style workflow if desired.
