## Label Justifications

- `P2`: The issue affects session-history recovery and continuity for upgraded or reset users, but the core runtime still operates and the data is usually still on disk until retention pruning.
- `impact:session-state`: Archived transcript fragments can become unreachable through normal history tools, breaking session and context continuity after resets.
- `impact:data-loss`: Users experience vanished history, and retention pruning can later turn undiscovered reset archive files into real transcript loss.

## Summary

Keep open. Current main and latest release still expose normal `chat.history` and `sessions_history` as active-transcript-only paths, while the remaining archive/family-history fixes are open PRs that need maintainer API and product selection.

## What This Changes

Request a supported way to restore or expose session history archived as `.jsonl.reset.<timestamp>` after manual or daily session reset behavior.

## Best Possible Solution

Define and land one canonical, additive history recovery contract that exposes active plus archived or family transcript segments with stable source metadata; add a mutating restore or doctor path only if maintainers deliberately approve that workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows no archive/family selector in `ChatHistoryParamsSchema` or `SessionsHistoryToolSchema`, and `chat.history` reads only the currently mapped transcript; I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No. The proposed mutating restore script is a useful stopgap, but the safer durable fix is a read-only Gateway/tool history contract with segment/source metadata, or an explicit doctor/restore path only after maintainer product approval.
