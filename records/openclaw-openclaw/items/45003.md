## Summary

Keep open: current main and the latest release still lack a supported restore/re-index command or archive-history selector, and the thread is paired with open contributor PRs that need maintainer product/API review before the issue can close.

## What This Changes

The issue requests a utility or supported workflow to restore and re-index `.jsonl.reset.<timestamp>` session transcript archives after daily/manual reset archiving hides prior sessions from normal history surfaces.

## Best Possible Solution

Choose and land a backward-compatible archive recovery contract, preferably read-only by default, with bounded Gateway reads, segment metadata, docs, tests, and a mutating restore path only if maintainers approve re-index semantics.

## Reproduction Assessment

Yes, at source level. Current main creates `.jsonl.reset.<timestamp>` archives during reset, but `chat.history` and `sessions_history` still expose only active transcript reads and there is no restore/import command in the inspected CLI/script surfaces.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The maintainable path should be a reviewed archive-read or restore contract coordinated with the open related PRs and the logical-history issue.
