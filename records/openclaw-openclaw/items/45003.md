- `impact:data-loss`: Archived transcripts disappear from supported surfaces and may later be pruned, creating perceived or eventual loss of user history.

## Summary

Keep open: current main and v2026.5.20 still lack a restore/re-index command or approved archived-history selector, and the related implementation PRs remain open with product, proof, or safety questions.

## What This Changes

Request a recovery utility or supported archive-history surface for session transcripts renamed to `.jsonl.reset.<timestamp>` by reset rotation.

## Best Possible Solution

Choose one backward-compatible archive recovery contract: keep current active-history behavior as the default, add explicit bounded archive/logical-history access with segment metadata, and add mutating restore/re-index only after identity, ownership, and retention semantics are approved.

## Reproduction Assessment

Yes, at source level. Current main still creates `.jsonl.reset.<timestamp>` archives and can prune them, while `chat.history`, `sessions_history`, and slash commands provide no restore/re-index command or approved archive-history selector.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The safer path is an approved read-only archive/logical-history contract first, with mutating restore behavior added only after session identity, ownership, visibility, and retention semantics are settled.
