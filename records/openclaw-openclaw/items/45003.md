## Summary

Keep open. Current main and v2026.6.10 still lack a restore/re-index utility or an explicit archived/family history selector; the implemented fallback only reads the newest reset archive when the active transcript is missing, and the remaining direction is already tied to open archive-history PRs that need maintainer product/security review.

## What This Changes

Request to restore and re-index `.jsonl.reset.*` session archives, later expanded into a broader archived-session recovery and discoverability path after daily resets.

## Best Possible Solution

Land one maintainer-approved read-only archive/family history API, then decide separately whether a guarded doctor/import restore path is needed for users who want to mutate archived files back into active session state.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 expose no archive/family selector and only fall back to the newest reset archive when the active transcript is missing. I did not run a live multi-day reset scenario in this read-only review.

## Solution Assessment

No, the original rename/re-index script is not clearly the best permanent fix because it mutates archived session state and can blur identity metadata. A read-only, bounded archive/family retrieval API plus an audited doctor/import path if needed is safer.

## Risks / Open Questions

- A restore script that renames archives back into live `.jsonl` files can mutate session identity and indexing state, so it needs a maintainer-approved migration/doctor shape rather than an ad hoc script.

- Open PRs already cover overlapping retrieval paths, so queueing another autonomous fix risks competing API surfaces (`includeArchived` versus `includeFamily`).
