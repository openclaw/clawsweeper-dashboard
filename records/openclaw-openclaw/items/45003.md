## Summary

Keep open: current main and the latest release still lack a supported restore/re-index command or approved archived-history selector, while related implementation PRs remain open and competing design choices need maintainer review.

## What This Changes

The issue requests a utility or supported recovery path to restore `.jsonl.reset.<timestamp>` session archives and rebuild session history visibility after reset rotation.

## Best Possible Solution

Choose one backward-compatible archive recovery contract: keep active-history behavior as the default, add explicit bounded archive/logical-history access with segment metadata, and add mutating restore only after identity, authorization, and retention semantics are approved.

## Reproduction Assessment

Yes, at source level. Current main still creates `.reset.<timestamp>` archives and can prune them, while `chat.history`, `sessions_history`, and slash commands provide no supported restore/re-index command or approved archive-history selector.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The safer path is an approved read-only archive/logical-history contract first, with mutating restore behavior added only after session identity, ownership, visibility, and retention semantics are settled.
