## Summary

Keep open. Current main and the latest release still do not provide a restore/re-index utility or an explicit archive/family history selector for `sessions_history` and `chat.history`; the useful remaining work is covered by open implementation/product-direction threads rather than already implemented code.

## What This Changes

The issue requests a core recovery path for `.jsonl.reset.*` session archives, originally as a restore/re-index script and later as read-only access to archived/logical session history.

## Best Possible Solution

Land or narrow a maintainer-approved archived/logical-history recovery contract with bounded read-only access, source metadata, retention behavior, tests, and docs; keep mutation-style restore as a deliberate doctor/recovery decision rather than an ad hoc script.

## Reproduction Assessment

Yes. At source level, current main has reset archive fallback only when the active transcript is missing, and neither `sessions_history` nor `chat.history` exposes a selector for archived/family history; I did not run a live reset scenario in this read-only review.

## Solution Assessment

Unclear as proposed. A restore/re-index script could help already affected users, but the more maintainable solution is an explicit read-only archive/family history API or maintainer-approved doctor recovery path with clear ownership and retention semantics.
