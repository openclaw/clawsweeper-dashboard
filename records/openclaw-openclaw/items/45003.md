## Summary

Keep open. Current main and v2026.6.8 still provide only a same-session reset-archive fallback when the active transcript is missing; they do not provide the requested restore/re-index workflow or a public archive/family selector, and open implementation PRs still need maintainer API/security direction.

## What This Changes

Requests a restore and re-index utility for session transcript files archived as `.jsonl.reset.<timestamp>` by reset maintenance.

## Best Possible Solution

Adopt one maintainer-approved, bounded, header-checked read-only archive/family history API with source metadata, then add a doctor-style mutating restore only if maintainers still want emergency recovery.

## Reproduction Assessment

Yes, at source level: current main exposes no `sessions_history` archive/family selector and only falls back to a same-session reset archive when the active transcript is missing. I did not run a live WebChat reset because this review is read-only.

## Solution Assessment

No, the standalone mutating restore script is a useful stopgap but not the best permanent core shape by itself. A read-only archive/family API should be settled first, with restore/re-index as explicit doctor-style recovery if maintainers want it.
