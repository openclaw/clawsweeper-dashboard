## Summary

Keep open: current main still lacks a supported restore/list/read path for reset archives in `sessions_history`, `chat.history`, or the Control UI session list, and several open PRs now cover competing slices of the recovery design that need maintainer direction.

## What This Changes

The issue requests a utility or built-in recovery path to restore and re-index `.reset.*` archived session transcripts after daily or manual session resets.

## Best Possible Solution

Choose a maintainer-approved archive recovery contract that can list and read reset archives without unsafe mutation by default, then land one coherent slice or split the restore, API, and UI pieces intentionally.

## Reproduction Assessment

Yes, at source level. Create or observe a reset archive, then query current-main `sessions_history` or `chat.history`: the schemas expose no archive/logical-history parameter and the reader path resolves only active transcript candidates.

## Solution Assessment

No for the original standalone rename/re-index script as the durable fix. It may help affected users, but the maintainable product direction needs maintainer choice between read-only archive discovery, explicit restore, and logical-history semantics before landing one of the open PR slices.
