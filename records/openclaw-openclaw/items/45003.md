## Summary

Keep open. Current main and v2026.6.10 still lack the requested restore/re-index utility and also lack an explicit `chat.history` / `sessions_history` archive-family selector; the closest remaining implementation path is the open, maintainer-review PR at https://github.com/openclaw/openclaw/pull/90239 plus the canonical live report at https://github.com/openclaw/openclaw/issues/92062.

## What This Changes

The issue asks for a recovery utility or supported recovery path for session transcripts archived as `.jsonl.reset.<timestamp>` after reset rotation.

## Best Possible Solution

Land or narrow one maintainer-approved, default-off, bounded, header-checked family-history recovery contract, then decide whether a separate doctor-style restore/re-index command is still needed for already affected users.

## Reproduction Assessment

Yes, at source level. Current main has only active-missing reset archive fallback, while `chat.history` and `sessions_history` expose no archive/family selector or restore/re-index command; I did not run a live multi-day reset scenario in this read-only review.

## Solution Assessment

Unclear as originally proposed. A restore/re-index script could help already affected users, but the safer maintainable path is a bounded read-only family-history API or a deliberate doctor recovery command with source metadata and retention semantics.
