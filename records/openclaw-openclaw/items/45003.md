## Summary

Keep open: current main and the latest release still lack a supported reset-archive restore command or `chat.history`/`sessions_history` archive selector, while overlapping open PRs cover competing fallback, explicit archive-read, and mutating restore designs that need maintainer direction.

## What This Changes

The issue requests a supported way to restore or recover `.jsonl.reset.<timestamp>` session archives and rebuild session indexing after reset archiving hid prior session history.

## Best Possible Solution

Choose one maintainer-approved archive recovery contract that is read-only by default, preserves reset semantics and retention behavior, and adds a mutating restore path only if indexing, ownership filtering, and legacy archive handling are explicitly designed.

## Reproduction Assessment

Yes, at source level. Current main creates `.jsonl.reset.<timestamp>` archives during reset, but `chat.history` and `sessions_history` still resolve only the active transcript path and expose no archive selector or restore command.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. It can be useful for already affected users, but the maintainable path should be a reviewed archive-read or restore contract coordinated with the open related PRs.
