## Label Justifications

- `P2`: The issue affects session-history recovery and continuity after resets, but the runtime still operates and the data is usually on disk until retention pruning.
- `impact:session-state`: Archived transcript fragments can become unreachable through normal history tools, breaking session and context continuity after resets.
- `impact:data-loss`: Users experience vanished history, and retention pruning can later delete undiscovered reset archives.

## Summary

Keep open. Current main and v2026.6.6 still expose only active-session history, while open related PRs cover overlapping archive, missing-active, and family-history designs that need a maintainer API/product decision before this cleanup can close.

## What This Changes

Request a supported utility or recovery path for `.jsonl.reset.*` session archives so history archived by daily or manual resets can become visible again.

## Best Possible Solution

Define one canonical read-only archive/family history contract with ordered segment/source metadata, then add a doctor-style mutating restore only if maintainers explicitly approve that workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions_history` and `chat.history` still have no archive/family selector and read the currently mapped active transcript; I did not run a live reset reproduction.

## Solution Assessment

No. The proposed mutating restore script is a useful emergency workaround, but the durable fix should be a maintainer-approved read-only archive/family history contract, with restore/re-indexing only as an explicit doctor-style workflow if desired.
