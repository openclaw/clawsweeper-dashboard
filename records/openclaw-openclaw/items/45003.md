## Summary

Keep open: current main still lacks a supported restore command or read/list path for `.jsonl.reset.<timestamp>` archives through `chat.history`, `sessions_history`, or the session UI, and active related PRs are still unmerged. The remaining work needs maintainer direction on read-only archive visibility versus mutating restore semantics.

## What This Changes

Request a built-in utility or supported recovery path for session transcripts archived as `.reset.<timestamp>` after manual or daily session resets.

## Best Possible Solution

Choose one maintainer-approved archive recovery contract that is read-only by default, preserves reset semantics and bounded history behavior, and adds a mutating restore path only if ownership filtering, retention, and legacy archive handling are explicitly designed.

## Reproduction Assessment

Yes, at source level. Current main creates `.jsonl.reset.<timestamp>` archives during reset, but `chat.history` and `sessions_history` only resolve the active transcript path and expose no archive selector or restore command.

## Solution Assessment

No for the standalone rename/re-index script as the durable repo-level solution. It may help already affected users, but the maintainable fix should be a reviewed archive-read or restore contract coordinated with the open PRs.
