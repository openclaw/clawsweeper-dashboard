## Summary

Keep open: current main still lacks a restore/re-index command or an approved archive-history contract for reset transcripts. The source gap is still present, while related open PRs cover overlapping partial fixes and need maintainer product/API review before this issue can close.

## What This Changes

Request a utility or supported recovery path to restore `.jsonl.reset.<timestamp>` session archives and rebuild session history after daily reset archival.

## Best Possible Solution

Choose one backward-compatible archive recovery contract, preferably read-only by default with bounded Gateway reads, segment metadata, docs, tests, and a mutating restore/re-index path only if maintainers approve it.

## Reproduction Assessment

Yes, at source level. Current main creates `.jsonl.reset.<timestamp>` archives and prunes them by retention policy, but `chat.history` and `sessions_history` still expose only the current active transcript and there is no restore/re-index command.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The safer path is an approved archive-read contract first, with mutating restore behavior added only after session identity and retention semantics are settled.
