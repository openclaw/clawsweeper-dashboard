## Summary

Keep open: current main still creates and prunes reset archives, but it does not provide the requested restore/re-index utility or first-class archived-history access through `chat.history` / `sessions_history`, and same-author PRs are already active implementation candidates for part of the gap.

## What This Changes

The issue requests a utility to restore `.jsonl.reset.<timestamp>` session transcript archives back into active session files and rebuild the session index after daily reset archiving made history disappear from normal session surfaces.

## Best Possible Solution

Keep this open until maintainers choose the recovery path: land or revise the archive-aware history PRs for first-class read-only access, and add a conservative CLI/doctor restore only if a mutating recovery workflow is still desired for already affected users.

## Reproduction Assessment

Yes, at source level: once a transcript is renamed to `.jsonl.reset.<timestamp>`, `sessions_history` calls `chat.history` with only `sessionKey` and `limit`, and `chat.history` reads only the current session entry's transcript. I did not run a live reset because this review was read-only.

## Solution Assessment

Unclear. The proposed restore script is a useful stopgap for already affected users, but the safer long-term design is read-only archive access through `chat.history` / `sessions_history`, with an explicit doctor or CLI repair only for approved mutating recovery cases.
