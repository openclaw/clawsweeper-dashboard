## Summary

Keep open. Current main and latest release still expose active-session-only history through normal `chat.history` and `sessions_history`, while open related PRs are still competing implementation candidates rather than merged fixes.

## What This Changes

Request to restore or re-expose reset-archived session transcripts so upgraded users can recover sidebar/tool access to prior session history.

## Best Possible Solution

Define one canonical, additive, read-only Gateway and tool history contract that exposes active plus archived or family transcript segments with stable source metadata, then close this issue after that path merges or maintainers explicitly reject it.

## Reproduction Assessment

Yes, at source level: current main has no archive/family selector in `ChatHistoryParamsSchema` or `SessionsHistoryToolSchema`, and `chat.history` reads only the current mapped transcript. I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No, the mutating restore script is a useful stopgap but not the best durable fix. A read-only Gateway/tool history contract with segment identity is safer for current users, remote clients, and downstream local-history tools.
