## Summary

Keep open: current main fixes part of the daily-reset freshness concern, but it still lacks a supported restore command or read/list path for reset archives through `chat.history`, `sessions_history`, or the session UI, and several active PRs cover overlapping recovery designs that need maintainer direction.

## What This Changes

The issue asks for a supported utility or recovery path to restore or re-index `.jsonl.reset.<timestamp>` session archives so reset history becomes accessible again.

## Best Possible Solution

Choose one maintainer-approved archive recovery contract that is read-only by default, preserves reset semantics and bounded history behavior, and only adds a mutating restore path if ownership filtering and legacy archive handling are explicitly designed.

## Reproduction Assessment

Yes, at source level. Current main has no archive selector or restore command, and the active `chat.history`/`sessions_history` path resolves only the current transcript candidates, so a `.jsonl.reset.<timestamp>` archive is not recoverable through those supported surfaces.

## Solution Assessment

No for the original standalone rename/re-index script as the durable fix. It may help already affected users, but the maintainable direction needs a reviewed archive-read or restore contract with bounded reads, ownership filtering, docs, and coordinated PR scope.
