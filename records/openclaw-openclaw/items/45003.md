## Summary

Keep open. Current main and the latest release only have a narrow same-session reset-archive fallback when the active transcript is missing; they still do not provide the requested restore/re-index workflow or a public `sessions_history` archive/family selector, and open related PRs still need maintainer API direction.

## What This Changes

The issue requests a recovery script or equivalent built-in path for restoring and re-indexing session transcripts archived as `.reset.<timestamp>` files after reset behavior made history disappear from normal session views.

## Best Possible Solution

Adopt one maintainer-approved read-only archive/family history API that preserves the existing missing-active fallback and returns ordered source metadata; only add a doctor-style restore/re-index command if maintainers decide a mutating emergency recovery path is still needed.

## Reproduction Assessment

Yes at source level. Current main exposes no `sessions_history` archive/family selector, and a newer live report documents the same user-visible gap against v2026.6.5; I did not run a live WebChat reset because this review is read-only.

## Solution Assessment

No, the standalone mutating restore script is a useful emergency workaround but not the best permanent core shape by itself. The maintainable solution should settle a read-only archive/family history API first, with mutating restore handled as explicit doctor-style recovery if desired.
