## Summary

Keep open: current main still creates `.jsonl.reset.<timestamp>` archives but `chat.history` and `sessions_history` expose only the current active transcript path, and the open related PRs need maintainer product/API review rather than a competing cleanup close.

## What This Changes

Add a supported utility or recovery path to restore and re-index session transcript archives created by reset rotation.

## Best Possible Solution

Choose and land a backward-compatible archive recovery contract, preferably read-only by default, with bounded Gateway reads, segment metadata, docs, tests, and a mutating restore/re-index path only if maintainers approve that behavior.

## Reproduction Assessment

Yes, at source level. Current main archives reset transcripts and prunes them by retention policy, but the inspected Gateway protocol, `chat.history`, and `sessions_history` paths do not expose an archive selector or restore/re-index command.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The maintainable path should be a reviewed archive-read or restore contract coordinated with the open related PRs and the logical-history issue.
