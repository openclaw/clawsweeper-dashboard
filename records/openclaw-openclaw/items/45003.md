## Summary

Keep open: current main still creates reset archives and has some archive indexing improvements, but it does not provide the requested restore/re-index utility or first-class archived-history access through `chat.history` / `sessions_history`. The same-author follow-up PRs and broader logical-history issues make this a maintainer direction call rather than a safe cleanup close.

## What This Changes

Request a utility that renames `.jsonl.reset.<timestamp>` session transcript archives back to active `.jsonl` files and rebuilds `sessions.json` after daily reset rollover.

## Best Possible Solution

Keep this open until maintainers choose the recovery path: land or revise archive-aware history access, then add a conservative doctor/CLI restore only if mutating recovery is still desired for already affected users.

## Reproduction Assessment

Yes, at source level: reset rollover still renames transcripts to `.jsonl.reset.<timestamp>`, and current `chat.history` / `sessions_history` read only the active entry without an archive selector. I did not run a live reset because this review was read-only.

## Solution Assessment

Unclear. The proposed restore script is a useful stopgap, but the safer maintainable direction is read-only archive access through Gateway/tool surfaces, with any mutating restore limited to an explicit doctor or CLI repair after maintainer approval.
