## Label Justifications

- `P2`: The issue affects history/context recovery for upgraded users, but the core runtime still runs and the data is usually still on disk.
- `impact:session-state`: The request concerns transcript and session continuity becoming unreachable after reset-created session rotations.
- `impact:data-loss`: Users can perceive lost history and archived transcripts may later be pruned without a supported recovery path.

## Summary

Keep open. Current main still lacks a supported restore/import or archive-family history path for normal `chat.history` / `sessions_history` use, and the same-author PR plus the newer canonical bug report remain open, so this still needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue requests a utility or equivalent supported product path to restore or access session transcripts archived as `.reset.<timestamp>` files after daily or manual resets.

## Best Possible Solution

Define one additive, bounded, read-only Gateway/session-history contract that exposes active plus archived or family transcript segments with stable source metadata, then have agent tools and UI consume that without renaming archives into active sessions.

## Reproduction Assessment

Yes, at source level: current main has no archive/family selector in `ChatHistoryParamsSchema` or `SessionsHistoryToolSchema`, and the Gateway reads only the current mapped transcript. I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No, the mutating restore script is a useful stopgap but not the best durable fix. A read-only Gateway/tool contract with segment identity is safer and aligns better with local and remote clients.
