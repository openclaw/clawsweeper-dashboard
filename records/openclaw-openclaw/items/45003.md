## Label Justifications

- `P2`: The issue affects session-history recovery for upgraded/reset users, but the core runtime still operates and the data is usually still on disk until retention pruning.
- `impact:session-state`: Archived transcript fragments can become unreachable through normal history tools, breaking session and context continuity after resets.
- `impact:data-loss`: Users experience vanished history, and retention pruning can later turn undiscovered archive files into real transcript loss.

## Summary

Keep open. Current main and the latest release still expose normal `chat.history` and `sessions_history` as active-transcript-only paths, while the related archive/family-history PRs are still open implementation candidates rather than merged fixes.

## What This Changes

The issue requests a supported way to restore or access session transcript history archived as `.jsonl.reset.<timestamp>` after reset-driven session rollover.

## Best Possible Solution

Define and land one canonical, additive history recovery contract that exposes active plus archived or family transcript segments with stable source metadata; use a restore script only if maintainers deliberately want a mutating doctor-style recovery path.

## Reproduction Assessment

Yes. Source inspection shows current main has no archive/family selector in `ChatHistoryParamsSchema` or `SessionsHistoryToolSchema`, and `chat.history` reads only the currently mapped transcript; I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No. The proposed mutating restore script is a useful stopgap, but the safer durable fix is a read-only Gateway/tool history contract with segment/source metadata, or an explicit doctor/restore path only after maintainer product approval.
