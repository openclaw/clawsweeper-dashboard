## Summary

Keep open. Current main still produces and prunes reset archives, but it does not provide the requested bulk restore/re-index utility and the supported `chat.history` / `sessions_history` surfaces still read the active transcript only. The provided discussion also shows same-author open PRs #60409 and #73883 as active implementation candidates, so this issue should stay open until those are reviewed, landed, or explicitly superseded by a maintainer recovery design.

## What This Changes

Request a first-party utility to restore `.jsonl.reset.*` session archive files back into accessible session history and rebuild the session index, with related follow-up around first-class archived-history access.

## Best Possible Solution

Keep the issue open until maintainers choose the recovery path: land or revise the active archived-history PRs for first-class read-only access, and add a conservative CLI/doctor repair only if maintainers still want a mutating restore/re-index workflow for already affected users. The end state should preserve stable session identity, original/restored archive paths, reset timestamps, and safe retention behavior for downstream history viewers.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: after a reset produces a `.jsonl.reset.<timestamp>` archive, `chat.history` and `sessions_history` on current main resolve only the active `sessions.json` entry and expose no archive selector; a live repro was not run to avoid writing files in this read-only review.

## Solution Assessment

Unclear. The proposed restore script is a useful stopgap for already affected users, but the more maintainable solution is first-class read-only archive access through `chat.history` / `sessions_history`, with a doctor or CLI repair only for explicit recovery cases that must mutate files.
