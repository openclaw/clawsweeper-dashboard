## Label Justifications

- `P2`: The issue affects history and context recovery for upgraded users, but the core runtime still operates and the data is usually still on disk until retention pruning.
- `impact:session-state`: The request concerns transcript and session continuity becoming unreachable after reset-created session rotations.
- `impact:data-loss`: Users can perceive lost history, and reset archives may later be pruned without a supported recovery path.

## Summary

Keep open. Current main and v2026.6.6 still expose active-session-only history for normal `chat.history` and `sessions_history`, and this issue is paired with an open same-author implementation PR plus newer canonical family-history work, so cleanup closure would be premature.

## What This Changes

Request a supported way to restore or access session transcript history archived as `.jsonl.reset.<timestamp>` after daily or manual session resets.

## Best Possible Solution

Define one additive, bounded, read-only Gateway and tool history contract that exposes active plus archived or family transcript segments with stable source metadata, then close this issue after the paired or canonical implementation merges or maintainers explicitly reject the direction.

## Reproduction Assessment

Yes, at source level: current main has no archive/family selector in `ChatHistoryParamsSchema` or `SessionsHistoryToolSchema`, and `chat.history` reads only the current mapped transcript. I did not run a live upgrade/reset reproduction in this read-only review.

## Solution Assessment

No, the mutating restore script is a useful stopgap but not the best durable fix. A read-only Gateway/tool history contract with segment identity is safer for current users, remote clients, and downstream local-history tools.
