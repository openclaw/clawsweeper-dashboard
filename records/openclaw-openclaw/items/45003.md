## Label Justifications

- `P2`: This is a normal-priority session recovery improvement with real user impact but limited blast radius and no emergency runtime outage.
- `impact:session-state`: The issue is about session transcript continuity and agent/context recovery after reset rotation.
- `impact:data-loss`: Archived transcripts can disappear from supported surfaces and may later be pruned, creating perceived or eventual loss of user history.

## Summary

Keep open: current main still has no restore/re-index command and no approved archive-history contract for reset transcripts. The directly related recovery PRs remain open, so the remaining work still needs maintainer product/API review rather than cleanup closure.

## What This Changes

Request a utility or supported recovery path to restore and re-index session transcripts archived as `.jsonl.reset.<timestamp>` after reset rotation.

## Best Possible Solution

Choose one backward-compatible archive recovery contract, preferably read-only by default with bounded Gateway reads, segment metadata, docs, tests, and a mutating restore/re-index path only if maintainers approve it.

## Reproduction Assessment

Yes, at source level. Current main creates `.jsonl.reset.<timestamp>` archives and can prune them by retention policy, while `chat.history` and `sessions_history` still expose only the current active transcript and there is no restore/re-index command.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The safer path is an approved archive-read contract first, with mutating restore behavior added only after session identity and retention semantics are settled.
