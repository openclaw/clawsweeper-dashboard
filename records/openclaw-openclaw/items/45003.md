## Summary

Keep open: current main still does not provide a supported restore, archive-listing, or archive-chain read contract for reset transcripts through the normal session history surfaces, and several open PRs cover overlapping slices that need maintainer direction.

## What This Changes

The issue requests a utility or supported recovery path to restore and re-index session transcripts archived as `.reset.*` files after reset behavior made them disappear from normal history surfaces.

## Best Possible Solution

Choose a maintainer-approved archive recovery contract that favors safe read-only discovery by default, then land coordinated PRs for the narrow fallback, explicit archive reads or metadata, and any intentional restore command/script.

## Reproduction Assessment

Yes, at source level. Create or observe a `.jsonl.reset.<timestamp>` archive and query current-main `chat.history` or `sessions_history`: the protocol/tool schemas expose no archive option, and the reader path resolves only active transcript candidates.

## Solution Assessment

No for the standalone rename/re-index script as the durable product fix. It may help affected users, but the maintainable solution needs a maintainer-approved recovery/read contract and coordination with the open archive-history PRs.
