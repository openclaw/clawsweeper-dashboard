## Label Justifications

- `P2`: The gap repeatedly hides prior conversation history but does not make the core runtime unusable.
- `impact:data-loss`: Archived transcripts appear lost to users and may become permanently unavailable when retention pruning runs.
- `impact:session-state`: The issue concerns continuity and recoverability across reset-generated session segments.

## Summary

Current main still exposes only the active session transcript through normal history paths, and the reporter’s two implementation PRs remain open. A newer June 2026 report confirms the broader continuity gap, so this issue still needs one canonical maintainer product/API decision rather than closure or another parallel fix.

## What This Changes

Add a supported path to recover or read reset-archived session transcripts so prior conversation history remains accessible after rollover.

## Best Possible Solution

Define one additive, bounded Gateway contract for read-only access to current and archived logical-session segments, preserve stable segment identity and retention status, and have agent tools and UI consume it without renaming transcripts or rebuilding the active index.

## Reproduction Assessment

Yes at source and report level: current history requests follow the active session without an archive selector, and https://github.com/openclaw/openclaw/issues/43929 documents a concrete June 1, 2026 rollover where retained prior context was absent from normal visible history. No new live rollover was executed during this read-only review.

## Solution Assessment

No, the proposed destructive restore script is not the best permanent solution; a bounded read-only Gateway archive/history contract preserves session identity and supports local and remote clients.
