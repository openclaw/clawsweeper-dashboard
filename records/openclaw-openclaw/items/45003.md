## Summary

Keep open: current main still lacks a supported restore/import or read-only archive-history contract for `.jsonl.reset.<timestamp>` transcripts, and the remaining direction needs maintainer product/privacy review with active related PRs already in flight.

## What This Changes

Request a recovery path for session transcripts archived as `.jsonl.reset.<timestamp>`, initially as a restore/reindex script and later as a read-only archive access roadmap.

## Best Possible Solution

Review the active archive-history PRs, choose a bounded Gateway/tool visibility contract for reset archives, document it, and only add a restore/reindex script if maintainers still want a mutable recovery path after read-only access is settled.

## Reproduction Assessment

Yes, at source level. Current main still creates `.jsonl.reset.<timestamp>` archives but `chat.history` and `sessions_history` only expose the active transcript path, so the reported access gap is clear without a live Gateway run.

## Solution Assessment

Unclear until maintainers choose the product contract. A read-only bounded Gateway/tool path looks safer than renaming archived files back into active sessions, but the active related PRs should be reviewed or revised first.
