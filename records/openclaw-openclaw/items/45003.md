## Summary

Keep open: current main and v2026.5.20 still have no restore/re-index command or approved archive-history selector, and the related PRs remain open with product, proof, and safety questions. The issue is valid, but it should stay as a maintainer/product tracker rather than be closed as implemented or queued as a single automated fix.

## What This Changes

The issue requests a supported restore/re-index utility or recovery path for session transcripts archived as `.reset.<timestamp>` after reset rotation.

## Best Possible Solution

Choose one backward-compatible archive recovery contract: keep current history as the default, add explicit bounded archive/logical-history access with segment metadata, and add mutating restore/re-index only after maintainers approve identity, ownership, and retention semantics.

## Reproduction Assessment

Yes, at source level. Current main still creates `.jsonl.reset.<timestamp>` archives and can prune them, while `chat.history`, `sessions_history`, sessions-history HTTP, and slash commands provide no restore/re-index command or approved archive-history selector.

## Solution Assessment

No for a standalone rename/re-index script as the durable repo-level solution. The safer path is an approved read-only archive/logical-history contract first, with mutating restore behavior added only after session identity, ownership, visibility, and retention semantics are settled.
