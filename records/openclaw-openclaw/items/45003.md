## Label Justifications

- `P2`: The issue affects recoverability of archived session history after resets, but the core runtime still operates and related fixes need normal maintainer review.
- `impact:session-state`: Archived transcript fragments can become unreachable through normal history tools, breaking session and context continuity after resets.
- `impact:data-loss`: Users experience vanished history, and retention pruning can later delete undiscovered reset archives.

## Summary

Keep open. Current main and the latest release still lack a supported archive/family-history or restore contract for `.reset.*` transcripts, and the open related PRs cover overlapping fallback, explicit archive, and family-history designs that need maintainer product/API direction before this can close.

## What This Changes

The issue requests a utility or built-in recovery path to restore or otherwise access session transcripts archived as `.jsonl.reset.<timestamp>` after reset rotation.

## Best Possible Solution

Define one canonical read-only archive/family-history contract with ordered segment/source metadata first, then add a mutating doctor-style restore only if maintainers explicitly approve that workflow.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` expose only active-session `chat.history`/`sessions_history` parameters, and the gateway reads the current mapped transcript rather than archived or family segments.

## Solution Assessment

No. The proposed restore script is a useful emergency workaround, but the maintainable core direction should be an approved read-only archive/family history contract, with mutating restore/reindexing only as a separate explicit workflow if desired.
