## Summary

Keep open: the PR targets a real native chat exec approval loop and has strong Telegram proof, but it adds a hard-coded native approval channel registry instead of using the existing plugin approval-capability contract, which is a merge-blocking compatibility defect.

## What This Changes

The PR extends gateway-host exec approval handling so native chat approval clients wait inline and return approved or denied command results to the active agent turn, with regression tests for native and non-native channels.

## Best Possible Solution

Keep the native chat inline-resume behavior, but drive native-client detection from the existing approval-capability/native-runtime contract instead of a bundled channel ID list.

## Reproduction Assessment

Yes from source inspection: current main only waits inline when `turnSourceChannel` normalizes to webchat, so native chat channels fall through to the async approval-pending followup path even though the docs promise approved runs feed lifecycle results back to the agent.

## Solution Assessment

No. The behavioral direction is right, but the best fix should reuse the existing approval-capability/native-runtime contract instead of adding a static core list of native approval channel IDs.
