## Summary

Keep open. Current main does not yet implement explicit session-label precedence or the dedicated `deriveSessionTitle` module/tests, so this PR is still useful, but the latest patch still introduces a concrete regression by replacing defensive `normalizeOptionalString` handling with direct `.trim()` calls on shallowly loaded session-store fields.

## What This Changes

The PR extracts gateway session title derivation into `src/gateway/session-title.ts`, re-exports it from `session-utils`, moves title tests into a dedicated test file, and changes explicit session labels to outrank transcript-derived fallback titles.

## Best Possible Solution

Merge a corrected version that preserves the extraction, compatibility re-export, explicit `entry.label` precedence, and `origin.label` ordering behind transcript-derived titles, while retaining defensive normalization for persisted title candidates and adding release-note coverage; leave the broader #39722 behavior to its canonical fix path.

## Reproduction Assessment

Yes. Current main can be checked by calling `deriveSessionTitle({ sessionId, updatedAt, label: "My App" }, "heartbeat")`, which follows the existing first-user-message fallback instead of the label; the PR regression is reproducible with malformed store data such as `displayName: 42` because the new helper calls `.trim()` directly.

## Solution Assessment

No as submitted. The module extraction and explicit-label precedence are narrow and maintainable, but the helper should keep `normalizeOptionalString` for persisted optional fields before this can merge safely.
