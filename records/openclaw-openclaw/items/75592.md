## Summary

Keep open. The PR adds useful plugin-owned voice-call functionality that current main does not already have, but it is not merge-ready because it injects private caller context from low-assurance caller ID, defaults transcript logs outside the configured store path, and lacks required user-facing docs/changelog coverage.

## What This Changes

The PR adds realtime voice caller-context config keyed by caller phone number, injects caller-specific greetings/profile/voice-card instructions into the realtime handler, and adds optional JSONL transcript-fragment logging with tests.

## Best Possible Solution

Revise this PR in the voice-call plugin so caller-specific context is either gated by a stronger verified identity signal or documented as an explicit low-assurance opt-in with private-context safeguards, transcript logs honor configured storage or require an explicit path, and docs plus changelog describe setup, privacy, and retention behavior.

## Reproduction Assessment

Yes. A static reproduction is clear: configure realtime.callerContext for a matching Twilio From value and the PR builds provider instructions plus greeting/profile context from that value; configure config.store and omit realtime.transcriptLog.path and the PR helper still resolves the log under the default home directory.

## Solution Assessment

No. The plugin-owned placement is right, but this is not the best safe solution until the caller-ID trust model, transcript storage default, and user-facing privacy documentation are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid treating caller ID as verified identity:** `extensions/voice-call/src/webhook/realtime-handler.ts:116-124`
  - body: The new prompt says caller identity comes from a verified caller ID and injects profile/voice-card context based only on callerMeta.from. Current realtime inbound acceptance is normalized Twilio From matching allowFrom, and the docs explicitly say that is low-assurance caller-ID filtering, so spoofed caller ID could trigger personalized greetings or private context injection. Please require stronger verification or make this an explicit low-assurance opt-in that does not inject private context before confirmation.
  - confidence: 0.88
- **[P2] Honor the configured voice-call store for transcript logs:** `extensions/voice-call/src/webhook/realtime-handler.ts:154-155`
  - body: When transcript logging is enabled without an explicit realtime.transcriptLog.path, resolveTranscriptLogPath always writes under ~/.openclaw/voice-calls. Installations that set the plugin store path will get realtime aftercare logs somewhere else, contrary to the new schema comment and the existing CallManager store behavior. Thread the configured store path into the realtime handler or require an explicit path.
  - confidence: 0.82
- **[P3] Document the new realtime caller-context config:** `extensions/voice-call/src/config.ts:324-327`
  - body: This adds user-facing realtime config keys and reads local profile/voice-card files, but the PR does not update the Voice Call docs or CHANGELOG.md. Users need setup syntax plus privacy/retention warnings, and repo policy requires a changelog entry for this user-facing feature.
  - confidence: 0.9
