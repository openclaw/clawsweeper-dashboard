## Summary

Keep open. The PR is not obsolete on current main and it implements useful voice-call functionality in the plugin-owned area, but the proposed patch is not merge-ready because it injects private caller context from low-assurance caller ID, defaults transcript logs away from the configured voice-call store, and adds user-facing config without docs/changelog coverage.

## What This Changes

The PR adds realtime voice-call caller-context configuration keyed by caller ID, exact known-caller greetings with bounded profile/voice-card prompt injection, and optional JSONL transcript fragment logging.

## Best Possible Solution

Revise the plugin implementation so caller-specific context is explicitly low-assurance or gated by stronger identity verification, transcript logging follows the configured voice-call store or requires an explicit path, and the user-facing config is documented with privacy/retention guidance plus a changelog entry.

## Reproduction Assessment

Yes. Configure realtime.callerContext for a matching From number and simulate a Twilio realtime webhook; the PR injects configured profile/voice-card instructions and greeting from that caller ID. Configure config.store with transcriptLog enabled and no transcriptLog.path; the PR still resolves logs under the default home voice-calls path.

## Solution Assessment

No. The plugin-owned location is right, but the proposed design is not the safest maintainable solution until the caller identity model, transcript storage default, and documentation/changelog gaps are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid treating caller ID as verified identity:** `extensions/voice-call/src/webhook/realtime-handler.ts:116`
  - body: The new caller-context prompt says the caller identity comes from a verified caller ID and injects profile/voice-card context based only on callerMeta.from. Current realtime inbound acceptance is allowlist matching against the provider-supplied From value, and the docs warn this is low-assurance caller-ID filtering, so spoofed caller ID can trigger personalized greetings or private context injection. Please require stronger verification or make this a clearly low-assurance opt-in that does not imply verified identity for private context.
  - confidence: 0.88
- **[P2] Honor the configured voice-call store for transcript logs:** `extensions/voice-call/src/webhook/realtime-handler.ts:154-155`
  - body: When transcript logging is enabled without an explicit realtime.transcriptLog.path, resolveTranscriptLogPath always writes under ~/.openclaw/voice-calls. Installations that set the plugin store path will get aftercare logs in a different location than the rest of the voice-call state, contrary to the new schema comment. Thread the configured store path into the realtime handler or require an explicit path.
  - confidence: 0.82
- **[P3] Document the new realtime caller-context config:** `extensions/voice-call/src/config.ts:324-327`
  - body: This adds user-facing realtime callerContext and transcriptLog keys and reads local profile/voice-card files, but the PR does not update the Voice Call docs or CHANGELOG.md. Users need setup syntax plus privacy/retention warnings, and repo policy requires a changelog entry for user-facing feature changes.
  - confidence: 0.9
