## Summary

Keep open. The PR is useful and in the right voice-call plugin boundary, but it is not merge-ready: it injects private caller context based on low-assurance caller ID, routes default transcript logs outside the configured voice-call store, writes transcript fragments synchronously in the live realtime path, and lacks the user-facing docs/changelog/config metadata needed for the new privacy-sensitive config.

## What This Changes

The PR adds voice-call realtime caller-context config keyed by caller phone number, injects optional local profile/voice-card context and caller-specific greetings into realtime sessions, and adds optional JSONL transcript-fragment logging with tests.

## Best Possible Solution

Revise the PR inside the voice-call plugin so caller-specific context is either gated by a stronger verified identity signal or explicitly documented as a low-assurance opt-in with private-context limits, transcript logging honors configured storage and uses non-blocking writes, and docs, changelog, and plugin config metadata describe setup, privacy, and retention behavior.

## Reproduction Assessment

Yes. A static reproduction is clear: configure realtime.callerContext for a Twilio From value and the PR injects local profile/voice-card instructions with “verified caller ID” wording; configure config.store and enable transcriptLog without a path and the PR still resolves the log under the default home directory.

## Solution Assessment

No. The plugin-owned placement is right, but this is not the best maintainable solution until the caller-ID trust model, transcript storage/write path, and user-facing privacy/config documentation are fixed.
