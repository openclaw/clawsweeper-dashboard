## Summary

Keep this PR open, but it should not merge as-is: the proposed caller-ID-keyed private context injection conflicts with the documented low-assurance caller-ID model, and transcript logging needs storage/privacy follow-through.

## What This Changes

The PR adds Voice Call realtime caller context keyed by caller ID, caller-specific opening greetings/profile and voice-card injection, and optional realtime transcript JSONL logging.

## Best Possible Solution

Keep the feature in the Voice Call plugin, but gate private caller context behind an explicit low-assurance or verified-identity design, make transcript logging honor configured storage, and document setup, privacy, storage, and retention.

## Reproduction Assessment

Yes. A source-level reproduction is clear from the PR: configure realtime.callerContext for a matching From number and the provider instructions receive caller-specific private context; enable transcriptLog without a path and the helper resolves a hardcoded home-store path.

## Solution Assessment

No. Plugin-owned placement is right, but the proposed design is not ready until identity assurance, transcript storage, and user-facing privacy documentation are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid treating caller ID as verified identity:** `extensions/voice-call/src/webhook/realtime-handler.ts:116-124`
  - body: The new prompt and profile/voice-card injection are selected from the provider-supplied caller number, while current docs explicitly describe that number as low-assurance caller-ID filtering. A spoofed caller ID could unlock personalized greetings or private context before stronger verification, so gate private context behind verified identity or an explicit low-assurance confirmation flow.
  - confidence: 0.88
- **[P2] Honor the configured voice-call store for transcript logs:** `extensions/voice-call/src/webhook/realtime-handler.ts:154-155`
  - body: When transcript logging is enabled without an explicit path, this helper hardcodes ~/.openclaw/voice-calls/realtime-transcripts.jsonl even though the config comment promises the voice-call store directory and the manager already honors config.store. Custom-store installations would split realtime aftercare logs into a different location.
  - confidence: 0.82
- **[P3] Document realtime caller-context and transcript logging:** `extensions/voice-call/src/config.ts:324-327`
  - body: This adds user-facing config keys that read local files and persist transcript fragments, but the Voice Call docs do not explain the syntax, caller-ID assurance limits, storage path, interim transcript handling, retention, or privacy expectations.
  - confidence: 0.86
