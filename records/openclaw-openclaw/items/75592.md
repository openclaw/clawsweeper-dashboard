## Summary

Keep open. The PR is useful voice-call-owned work, but it is not merge-ready because it introduces a security-sensitive caller-ID trust assumption for private realtime context, routes default transcript logs outside configured voice-call storage, and lacks required docs/changelog coverage for new user-facing config.

## What This Changes

The PR adds voice-call realtime caller-context and transcript-log configuration, injects configured caller profile/voice-card instructions and greetings into `RealtimeCallHandler`, and adds tests/fixtures for inbound caller context plus JSONL transcript logging.

## Best Possible Solution

Keep the PR open and revise it so per-caller context does not imply strong identity from caller ID, private profile/voice-card injection has a clear security model, transcript logs honor the configured voice-call store or require an explicit path, and docs/changelog explain setup, privacy, and retention behavior.

## Reproduction Assessment

Yes. Configure `realtime.callerContext` for a phone number and simulate a Twilio realtime webhook with matching `From`; the PR builds provider instructions and greetings from that caller ID before any stronger identity check. Configure a custom voice-call `store` and omit `realtime.transcriptLog.path`; the PR helper still resolves logs under the default home path.

## Solution Assessment

No. The plugin-owned location is right, but this is not the best safe solution until the caller identity assumption and transcript storage default are fixed; a safer direction is explicit low-assurance caller-ID wording or a stronger verification gate before injecting private context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid treating caller ID as verified identity:** `extensions/voice-call/src/webhook/realtime-handler.ts:116`
  - body: The new caller-context prompt says the caller identity comes from a verified caller ID and injects profile/voice-card context based only on `callerMeta.from`. Current realtime inbound acceptance is normalized Twilio `From` matching `allowFrom`, and the docs explicitly warn that caller-ID filtering is low assurance, so a spoofed caller ID could trigger personalized greetings or private context injection into the realtime provider. Please require a stronger verified identity signal or make this an explicit low-assurance opt-in that does not inject private context before caller confirmation.
  - confidence: 0.88
- **[P2] Honor the configured voice-call store for transcript logs:** `extensions/voice-call/src/webhook/realtime-handler.ts:154-155`
  - body: When transcript logging is enabled without an explicit `realtime.transcriptLog.path`, `resolveTranscriptLogPath` always writes under `~/.openclaw/voice-calls`. Installations that set the plugin `store` path will get aftercare logs in a different location than the rest of the voice-call state, contrary to the new schema comment and operational expectations. Thread the configured voice-call store path into the realtime handler or require an explicit path.
  - confidence: 0.82
- **[P3] Document the new realtime caller-context config:** `extensions/voice-call/src/config.ts:324-327`
  - body: This adds user-facing realtime config keys (`callerContext`, `transcriptLog`) and reads local profile/voice-card files, but the PR does not update `docs/plugins/voice-call.md` or `CHANGELOG.md`. Users need setup syntax plus privacy/retention warnings, and OpenClaw policy requires a changelog entry for user-facing feature changes.
  - confidence: 0.9
