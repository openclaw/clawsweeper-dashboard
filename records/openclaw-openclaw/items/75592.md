## Summary

Keep open. Current main does not implement realtime caller-context or transcript-log support, and this PR is not merge-ready because it injects private caller context from low-assurance caller ID and adds transcript persistence without the right storage/docs guardrails. The existing ClawSweeper comment already flagged these points, and the PR head still matches that reviewed diff.

## What This Changes

The PR adds voice-call realtime caller-context config keyed by caller phone number, injects caller-specific greeting/profile/voice-card instructions into the realtime handler, and adds optional JSONL transcript-fragment logging with tests.

## Best Possible Solution

Revise the voice-call plugin implementation so caller-specific context is gated by a stronger verified identity signal or an explicit low-assurance opt-in with confirmation/private-context safeguards, transcript logging honors the configured store or requires an explicit path, and docs plus changelog cover setup, privacy, and retention.

## Reproduction Assessment

Yes. A static reproduction is clear: configure realtime.callerContext for a matching Twilio From value and the PR injects the configured name/profile/voice-card into provider instructions; configure config.store and omit realtime.transcriptLog.path and the PR resolves logs under the default home path instead of the configured store.

## Solution Assessment

No. The plugin-owned placement is appropriate, but the current PR is not the best safe fix until the caller-ID trust model, transcript storage default, and user-facing privacy documentation are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid treating caller ID as verified identity:** `extensions/voice-call/src/webhook/realtime-handler.ts:116-124`
  - body: The new prompt and private profile/voice-card injection are selected only from callerMeta.from, while current inbound acceptance is provider-supplied From matched against allowFrom and documented as low-assurance caller-ID filtering. A spoofed caller ID could trigger personalized greetings or private context before stronger verification, so gate this behind stronger identity proof or make it a low-assurance opt-in that confirms identity before injecting private files.
  - confidence: 0.88
- **[P2] Honor the configured voice-call store for transcript logs:** `extensions/voice-call/src/webhook/realtime-handler.ts:154-155`
  - body: When transcript logging is enabled without realtime.transcriptLog.path, this helper hardcodes ~/.openclaw/voice-calls/realtime-transcripts.jsonl even though the schema says the default is the voice-call store directory and CallManager already honors config.store. Installations with a custom store will silently split realtime aftercare logs into a different location.
  - confidence: 0.82
- **[P3] Document the realtime caller-context config:** `extensions/voice-call/src/config.ts:324-327`
  - body: This adds user-facing realtime.callerContext and transcriptLog keys that read local files and persist transcript fragments, but the PR does not update Voice Call docs or CHANGELOG.md. Users need setup syntax plus privacy, storage, and retention guidance, and repo policy expects a changelog entry for this feature.
  - confidence: 0.9
