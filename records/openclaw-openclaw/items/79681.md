## Summary

Keep open. The report now has an independent v2026.5.7 repro plus a source-level explanation, and current main still contradicts the documented `thinking` typing contract by ignoring reasoning deltas until renderable text has already appeared; current main only appears to cover the tool-start part of the suggested fix.

## What This Changes

The issue asks OpenClaw to keep Telegram typing keepalives visible during long active agent work, especially before the first assistant output and during tool-heavy runs.

## Best Possible Solution

Restore the documented typing lifecycle so reasoning deltas in `thinking` mode start and refresh keepalives before first text, preserve the current tool-start behavior, add focused regression coverage, and live-verify Telegram DM behavior when feasible.

## Reproduction Assessment

Yes at source level: current main’s `createTypingSignaler` ignores the first `signalReasoningDelta()` until `hasRenderableText` is true, while docs say `thinking` mode starts on the first reasoning delta. I did not run a live Telegram DM repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the shared typing signaler: let reasoning deltas in `thinking` mode start or refresh typing before first text while keeping heartbeat/disabled/silent-message guards intact. Telegram-specific changes should wait unless live logs show the channel API path failing separately.
