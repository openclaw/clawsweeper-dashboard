## Summary

Keep open: current main provides generic channel status and logout plus a WhatsApp-specific QR login UI, but it still lacks an account-addressable plugin pairing contract and a safe browser-driven plugin install flow. The request fits OpenClaw’s setup and web-ergonomics direction, while the permanent UI, protocol, plugin ownership, provenance, and consent model require maintainer product design rather than a narrow autonomous fix.

## What This Changes

Add a Control UI “Linked Apps” hub for discovering, installing, pairing, and managing channel plugins with QR-based login, initially targeting WeChat.

## Best Possible Solution

Extend the existing Channels surface with an additive, typed, account-addressable pairing capability declared by channel plugins, and design plugin installation as a separate provenance-aware consent flow that keeps channel-specific login mechanics inside each plugin.

## Reproduction Assessment

Not applicable: this issue requests a new generic Control UI capability, and current source inspection confirms the capability is absent rather than showing a failure of an existing contract.

## Solution Assessment

No, not exactly as proposed: a parallel hub with one-click installation would duplicate channel management and blur security boundaries; extending the existing Channels UI through typed plugin capabilities is the narrower maintainable direction.

## Risks / Open Questions

- A browser-triggered plugin install can execute third-party package code and must preserve source provenance, install policy, permissions, restart behavior, and explicit operator consent.
- A generic QR-login API would become a compatibility-sensitive gateway/plugin contract across multiple channels and accounts; a WeChat-specific core shortcut would create ownership and maintenance drift.
- A separate Linked Apps hub could duplicate the existing Channels settings surface and create two competing sources of channel status and connection management.
