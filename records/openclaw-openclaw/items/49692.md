## Summary

Keep open. Current main has friendlier restart copy and safe-restart preflight, but it still rejects new command enqueues during gateway drain before they can enter the preserved queue, so the requested automatic retry/replay behavior remains unresolved; https://github.com/openclaw/openclaw/pull/46303, https://github.com/openclaw/openclaw/issues/55412, and https://github.com/openclaw/openclaw/issues/57425 are related but not a merged replacement for this issue.

## What This Changes

The issue requests friendlier gateway drain messaging and automatic retry or replay for inbound messages rejected during gateway restart drain.

## Best Possible Solution

Keep this open until OpenClaw either lands a native bounded FIFO drain-window retry/replay path or explicitly scopes the remaining work into the narrower retry and graceful-restart tracks while preserving the friendlier fallback copy.

## Reproduction Assessment

Yes. The high-confidence source path is gateway restart calling markGatewayDraining, then enqueueCommandInLane rejecting GatewayDrainingError before the message can be queued; the current command-queue test asserts that behavior.

## Solution Assessment

Unclear as a final product path. A bounded runner retry like https://github.com/openclaw/openclaw/issues/55412 may handle the narrow visible error, while durable queue/replay and restart recovery need maintainer scoping with https://github.com/openclaw/openclaw/pull/46303 and https://github.com/openclaw/openclaw/issues/57425.
