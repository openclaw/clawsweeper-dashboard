## Summary

Keep open: current main has friendlier restart copy, safe-restart preflight, and restart-sentinel continuation paths, but new command enqueues during gateway drain still reject before they can enter the preserved queue, so automatic retry/replay remains unresolved.

## What This Changes

Requests friendlier drain messaging and automatic FIFO retry/replay of inbound messages rejected while the gateway is draining for restart.

## Best Possible Solution

Keep this issue as the canonical tracker and land a bounded drain-window retry/replay design, likely with an internal-only enqueue bypass and per-channel FIFO ordering, while leaving broader durable restart recovery to the graceful-restart track.

## Reproduction Assessment

Yes. The high-confidence source path is restart calling markGatewayDraining, enqueueCommandInLane rejecting GatewayDrainingError before queueing, and the current test suite asserting that rejection path.

## Solution Assessment

Unclear as a final product path. The local dist-patch direction shows a narrow in-process possibility, but current code intentionally rejects drain-window enqueues and the open PR history shows maintainers still need to scope safe replay semantics.
