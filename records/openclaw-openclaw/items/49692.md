## Label Justifications

- `P1`: The issue describes real channel messages being lost or requiring manual resend during gateway restarts, which is a broken agent/channel workflow affecting users.
- `impact:message-loss`: The core report is that inbound messages during the gateway drain window are rejected or lost instead of being replayed after restart.

## Summary

Keep open: current main and the latest release have friendlier restart copy, but new command enqueues still reject during gateway drain before a queue entry is created, so automatic retry/replay of the original inbound message remains unresolved. The duplicate retry issue was already closed back to this tracker, while the broader restart-recovery umbrella remains separate at https://github.com/openclaw/openclaw/issues/57425.

## What This Changes

Make gateway restart drain replies friendlier and queue/replay unprocessed inbound messages after restart instead of surfacing GatewayDrainingError to channel users.

## Best Possible Solution

Keep this as the narrow drain-window tracker: implement a bounded FIFO queue/replay path for inbound work that arrives during SIGUSR1 drain, integrate it with restart lifecycle copy, and leave the broader manifest/session-recovery program to https://github.com/openclaw/openclaw/issues/57425.

## Reproduction Assessment

Yes, source-level with high confidence. Restart marks gatewayDraining, enqueueCommandInLane rejects before creating a QueueEntry, resetAllLanes only drains already queued work, and the current test suite asserts the rejection path.

## Solution Assessment

Unclear as a final implementation. The local workaround shows a narrow in-process path is plausible, but the maintainable fix needs maintainer decisions on FIFO replay, deduplication, idempotency, and how it composes with safe restart and broader recovery work.
