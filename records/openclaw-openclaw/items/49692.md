## Summary

Keep open. Current main has friendlier restart copy, but inbound command work is still rejected during the gateway drain window before it can enter the preserved queue, so the lost-message/manual-resend behavior remains. The related Kafka plugin PR was closed unmerged, while #46303 and #57425 are still open and need maintainer selection rather than another automatic close.

## What This Changes

The issue requests friendlier gateway-drain user messaging and automatic retry/replay of inbound messages rejected while the gateway restarts.

## Best Possible Solution

Native in-process restart handling should accept and preserve inbound command entries during the drain window, keep FIFO ordering, avoid starting new work while draining, and let `resetAllLanes()` replay preserved entries after restart. Keep the friendly fallback for non-replayable drain failures, and handle durable crash/full-stop recovery and broader session recovery through #57425 or a reviewed follow-up.

## Reproduction Assessment

Yes. The high-confidence unit path is `markGatewayDraining()` followed by `enqueueCommand()`, which current main still expects to reject with `GatewayDrainingError` before any queue entry is preserved.

## Solution Assessment

Yes for the narrow remaining bug: queueing new command entries while skipping active drain during SIGUSR1 restart is the smallest maintainable in-process fix. Broader persistent replay should stay under explicit maintainer review because it changes durability and replay boundaries.
