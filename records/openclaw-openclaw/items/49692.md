## Summary

Keep open. Current main has friendlier restart copy, but the core lost-message behavior remains: new command work is still rejected during the gateway drain window before it can enter the queue that `resetAllLanes()` preserves for post-restart replay. The related Kafka plugin PR was closed unmerged, the broader restart-recovery umbrella remains open, and an overlapping open PR needs maintainer selection rather than another automatic close.

## What This Changes

The issue asks for friendlier user-facing gateway restart-drain copy and automatic replay of inbound messages that arrive while the gateway is draining for restart.

## Best Possible Solution

Native in-process restart handling should accept and preserve inbound command entries during the SIGUSR1 drain window, keep per-lane FIFO ordering, avoid starting new work while draining, and let `resetAllLanes()` replay preserved entries after restart. Keep the friendlier fallback message for truly non-replayable restart failures, while leaving durable crash/full-stop recovery and broader session manifests to the restart-recovery umbrella.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `markGatewayDraining()` followed by `enqueueCommand()`: current main rejects with `GatewayDrainingError`, and the existing command queue test asserts that behavior.

## Solution Assessment

Yes for the narrow remaining bug: queueing but not draining new command entries during an in-process restart is the smallest maintainable fix because the queue reset path already preserves and pumps queued work. Broader durable replay should be handled separately because it changes persistence and duplicate-delivery semantics.
