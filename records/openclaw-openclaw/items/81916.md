## Summary

Keep open: the latest head fixes stale output usage, but it still adds all bytes after the stale usage line, including pre-compaction tool/output bytes before the later compaction marker, so the reported compaction loop can remain. The real behavior proof also needs a refresh for the current head.

## What This Changes

The branch teaches preflight compaction to detect post-usage compaction markers, cap stale transcript usage/output, and adds regressions for stale usage and fresh tail pressure.

## Best Possible Solution

Land an estimator that splits or ignores stale pre-compaction tail bytes once a post-usage compaction marker proves the usage snapshot is stale, while preserving fresh latest-usage and real post-usage-tail pressure.

## Reproduction Assessment

Yes for a source-level path: a transcript with a stale usage line, a large pre-compaction tool/output line, then a compaction marker will still add that pre-marker tail at the PR head. I did not run the focused test in this read-only pass.

## Solution Assessment

No. The marker-gated prompt/output cap is the right direction, but the fix should also stop pre-compaction tail bytes from being counted as active post-compaction pressure and refresh proof for the latest head.
