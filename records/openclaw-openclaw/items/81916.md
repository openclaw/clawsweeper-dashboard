## Summary

Keep open: the latest branch is a plausible narrow fix, but it still lets marker-proven stale output and pre-marker tail pressure survive when the stale prompt snapshot is not far enough above the active replay estimate, and the real-behavior proof names an older head SHA.

## What This Changes

The branch adds post-compaction marker detection to the preflight transcript estimator and tests stale usage/tail handling plus no-marker fresh tail pressure.

## Best Possible Solution

Land a marker-aware estimator fix that drops stale output and pre-marker tail whenever a post-usage compaction marker proves the usage snapshot is stale, while preserving no-marker fresh usage plus tail pressure.

## Reproduction Assessment

Yes, source-level reproduction is clear: use a transcript with a post-usage compaction marker, moderate stale prompt usage, and large stale output or pre-marker tail. The latest PR head keeps that stale pressure because hasStaleUsageSnapshot remains false unless the prompt-disparity heuristic also trips.

## Solution Assessment

No. The marker-gated direction is right, but once a post-usage marker proves the usage snapshot predates compaction, stale output and pre-marker tail removal should not also depend on the prompt-disparity heuristic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drop marker-proven stale output and tail pressure:** `src/auto-reply/reply/agent-runner-memory.ts:610-613`
  - body: `hasStaleUsageSnapshot` still requires the stale prompt input to exceed `estimatedMessageTokens * 2 + 10000`. When a compaction marker follows the last usage record but the stale input is only moderately above the active replay, the code keeps stale `outputTokens` and full `trailingBytes`, so the next preflight can still compact immediately after compaction. Treat the post-usage marker as stale evidence for output/tail pressure and add that regression.
