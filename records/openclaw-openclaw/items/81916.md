## Summary

Keep open: the PR targets a real stale-usage compaction bug and includes usable proof, but the current diff still caps legitimate latest transcript usage whenever recent replay is much smaller, which can skip needed preflight compaction near the guard.

## What This Changes

The PR bounds raw transcript prompt-usage snapshots against recent active replay estimates in preflight compaction and adds a focused stale-usage regression test.

## Best Possible Solution

Keep the stale-usage bound, but make it apply only when usage is known stale or post-compaction, while preserving the existing raw latest usage plus trailing-byte floor and adding a threshold regression.

## Reproduction Assessment

Yes for the source-level PR regression: current main has a combined-tail test with 86k latest usage plus a 36 KB tool tail that expects compaction, and this clamp can reduce that floor below the threshold. The original production Discord/Pi loop was not live-rerun.

## Solution Assessment

No. Bounding stale transcript usage is the right direction, but this implementation over-bounds legitimate latest usage; the safer fix should preserve the current tail-pressure floor while capping only stale snapshots.
