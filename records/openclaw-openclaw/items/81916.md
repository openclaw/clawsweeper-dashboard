## Summary

Keep open: the stale-usage cap is useful, but this patch can undercount a valid latest usage record plus post-usage tool tail, so it needs a narrow estimator/test repair before merge.

## What This Changes

The PR clamps large transcript usage snapshots against a recent message replay estimate in preflight compaction and adds a stale-usage regression test.

## Best Possible Solution

Keep the stale-usage bound, but apply it only to truly stale pre-compaction snapshots while preserving latest usage plus tail pressure with regression coverage.

## Reproduction Assessment

Yes for the source path: the PR proof covers stale usage after compaction, and current main has a regression requiring 86k latest usage plus a 36 KB tail to compact at >=96k. I did not run a live Discord/Pi loop.

## Solution Assessment

No. Bounding stale transcript usage is the right direction, but this implementation does not distinguish stale pre-compaction records from valid latest usage plus post-usage tail pressure.
