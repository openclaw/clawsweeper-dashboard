## Summary

Keep open: this is the canonical active PR for the linked stale preflight compaction bug, but the current head still adds stale output tokens from the same marked-stale usage snapshot, so a high pre-compaction completion can keep forcing unnecessary preflight compaction.

## What This Changes

The PR adds compaction-marker detection, bounds marked-stale transcript prompt usage against recent replay estimates, and adds focused preflight compaction regressions.

## Best Possible Solution

Land a corrected estimator that treats prompt and output from a marked-stale usage snapshot consistently while preserving fresh latest-usage and post-usage-tail protections, with targeted regressions.

## Reproduction Assessment

Yes for a source-level PR regression: a transcript with a high-output usage record before a compaction marker and a small active replay follows the PR's stale-marker branch, but `resolveEffectivePromptTokens` still adds the stale output. I did not run the live Discord/Pi loop.

## Solution Assessment

No. The marker-gated prompt cap is the right direction, but the fix should also stop stale output from contributing to projected active context while preserving fresh output-token pressure.
