## Summary

Keep this PR open: the second commit fixes the earlier tail-pressure blocker, but the stale-usage path still carries `outputTokens` from the same pre-compaction usage snapshot, so a large stale completion can still force another early preflight compaction.

## What This Changes

The PR adds compaction-marker-aware bounding for stale transcript usage estimates and regression coverage for stale and fresh preflight compaction cases.

## Best Possible Solution

Land a corrected estimator that treats prompt and output from a marked stale usage snapshot consistently while retaining fresh latest-usage and post-usage-tail protections, with targeted regressions.

## Reproduction Assessment

Yes for a source-level path: a transcript with a high-output usage record before a compaction marker and a small active replay follows the PR's stale-marker branch, but `resolveEffectivePromptTokens` still adds the stale output. I did not run the live Discord/Pi loop.

## Solution Assessment

No. The marker-gated prompt cap is the right direction, but the fix should also stop stale output from contributing to the projected active context while preserving fresh output-token pressure.
