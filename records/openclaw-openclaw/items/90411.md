## Summary

Keep open: this maintainer/draft PR is protected from cleanup, and the patch usefully hardens the global command registry but misses the non-activating gateway registry path that still stores the original mutable command object.

## What This Changes

The PR adds guarded snapshot/copy logic around plugin command definitions and tests unreadable/native metadata plus post-registration mutation behavior in the global command registry.

## Best Possible Solution

Land a complete snapshot contract that stores immutable guarded command snapshots in both global `pluginCommands` and non-activating `registry.commands`, with focused gateway/registry regression coverage.

## Reproduction Assessment

Yes by source inspection. Current main reads and stores plugin-owned command metadata directly, and the PR still leaves the non-activating `registry.commands` path storing the original object consumed by gateway `commands.list`.

## Solution Assessment

No. The global-registry snapshot layer is the right direction, but the best fix also needs to store the validated snapshot in non-activating registry command registrations.
