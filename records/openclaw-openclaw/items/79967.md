## Summary

Keep open: the PR targets a real cron model-rejection observability gap, but the patch currently says the run is falling back even though the code returns an error before any fallback run. The submitted proof is also mock-only, so this needs contributor follow-up before merge.

## What This Changes

The PR adds a `model-fallback/decision` subsystem warning when `resolveCronModelSelection` rejects an isolated cron `payload.model`.

## Best Possible Solution

Keep the fail-closed behavior and add an accurate `model-fallback/decision` warning that describes the rejected cron model as an error, with focused regression coverage or live cron proof showing the real log output.

## Reproduction Assessment

Yes, by source inspection: a rejected cron `payload.model` reaches `resolveCronModelSelection`, returns `ok: false`, and `run.ts` records a `cron-preflight` error without any `model-fallback/decision` log call on current main. I did not run a live cron job in this read-only review.

## Solution Assessment

No, not as written. The narrow fix direction is good, but the warning must not claim `falling back` or expose a `fallbackModel` when the branch returns an error before fallback execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Avoid saying rejected cron models fall back:** `src/cron/isolated-agent/model-selection.ts:149-151`
  - body: This branch returns `ok: false`, and the caller records a `cron-preflight` error before any model run happens. Emitting `fallbackModel` plus `falling back to ...` will tell operators that the rejected cron is continuing on the default model, which is the confusion this fix is meant to remove; log the rejection/error outcome instead.
