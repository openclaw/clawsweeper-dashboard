## Summary

Keep this PR open, but do not merge it as-is: the intended WARN is useful, yet the new log says the run is falling back even though current cron code immediately returns an error, and the supplied proof is only a mock simulation.

## What This Changes

The PR adds a `model-fallback/decision` subsystem WARN in `resolveCronModelSelection` when a cron `payload.model` override is rejected before returning `ok: false`.

## Best Possible Solution

Keep the PR focused on an accurate rejection/error warning for cron model selection, with real behavior proof and targeted regression coverage for the emitted log fields.

## Reproduction Assessment

Yes, from source inspection: current `main` rejects a disallowed cron `payload.model` without emitting through `model-fallback/decision`. A live cron run would strengthen this, but the missing log branch is clear in the current code.

## Solution Assessment

No. Adding the WARN is a plausible fix direction, but this implementation should not describe a fallback because the runner returns an error before any fallback attempt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not log a fallback that never happens:** `src/cron/isolated-agent/model-selection.ts:149-151`
  - body: This warning records `fallbackModel` and says the cron is `falling back`, but `resolveCronModelSelection` immediately returns `ok: false` and `runCronIsolatedAgentTurn` turns that into an error before any model call. That would tell operators the default/agent model was used when the run actually failed, recreating the routing confusion this log is meant to remove. Please log this as a rejected/failed cron selection, or omit fallback fields unless a real fallback is attempted.
  - confidence: 0.91
