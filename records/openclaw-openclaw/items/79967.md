## Summary

Keep open. The direction is useful, but the patch is not mergeable because it logs a rejected cron model as a fallback even though the run exits with a `cron-preflight` error, and the supplied proof is mock-only.

## What This Changes

The PR imports a model-fallback subsystem logger and emits a warning when `resolveCronModelSelection` rejects an isolated cron `payload.model`.

## Best Possible Solution

Keep the fail-closed behavior and add an accurate `model-fallback/decision` warning that records the rejection/error outcome without implying fallback, backed by real cron/log proof or focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: on current main, a rejected cron `payload.model` returns `ok: false` in `resolveCronModelSelection`, and `run.ts` records a `cron-preflight` error before any provider/model run. I did not execute a live cron job in this read-only review.

## Solution Assessment

No, not as written. Adding the warning is the right narrow direction, but it must describe the rejection/error outcome rather than logging `fallbackModel` and `falling back to ...` on a branch that does not fall back.

## Review Findings

Overall correctness: patch is incorrect
