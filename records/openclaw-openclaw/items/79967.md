## Summary

Keep open: the PR targets a real cron observability gap, but the current diff would log that cron is falling back to another model even though the code returns an error before any fallback attempt.

## What This Changes

The PR adds a `model-fallback/decision` WARN in cron model selection when an explicit cron `payload.model` is rejected.

## Best Possible Solution

Keep the change focused on an accurate rejected-model warning for the fail-closed cron path, with targeted regression coverage and redacted real runtime proof.

## Reproduction Assessment

Yes, from source inspection: current main rejects a disallowed cron `payload.model` without emitting through `model-fallback/decision`, and the caller returns a cron-preflight error before any model run. I did not run a live cron scenario in this read-only review.

## Solution Assessment

No. Emitting a WARN is the right direction, but this patch should log an accurate rejected or failed selection without fallback fields or `falling back` wording because no fallback happens on this path.
