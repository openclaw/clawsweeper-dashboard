## Summary

Keep this PR open: the requested warning is a useful narrow fix for https://github.com/openclaw/openclaw/issues/79955, but the current patch logs that cron is falling back even though the current code fails closed before running the job, and the supplied proof is only a mock simulation.

## What This Changes

The PR imports the subsystem logger and emits a `model-fallback/decision` warning when `resolveCronModelSelection` rejects an explicit cron `payload.model`.

## Best Possible Solution

Emit a single `model-fallback/decision` warning that accurately says the cron `payload.model` was rejected and the run failed before execution, then cover that log path with focused resolver or cron-run coverage while preserving the existing `ok: false` behavior.

## Reproduction Assessment

Yes from source inspection: current main rejects the disallowed cron `payload.model` in `resolveCronModelSelection` without emitting to `model-fallback/decision`, and `run.ts` converts that result into a preflight error before model fallback can run.

## Solution Assessment

No as written: logging the rejection is the right narrow direction, but this implementation describes a fallback that the current control flow does not perform; the safer fix is an accurate rejection/fail-closed warning plus targeted coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make the warning match the fail-closed path:** `src/cron/isolated-agent/model-selection.ts:149-151`
  - body: At this point `resolveCronModelSelection` immediately returns `ok: false`, and the caller turns that into a cron preflight error before `runWithModelFallback` can run. Logging `fallbackModel` and saying `falling back to ...` will mislead operators into thinking the rejected cron used the default model, so the warning should say the run failed/rejected before execution or otherwise remove the fallback wording.
