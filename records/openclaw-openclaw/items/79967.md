## Summary

Keep open: the requested cron rejection warning is plausible, but the PR still reports a fallback model and says it is falling back even though the current code returns an error before any model call; the supplied proof is only an inline mock, not a real OpenClaw cron/module run.

## What This Changes

The PR adds a `model-fallback/decision` WARN in cron model selection when a cron `payload.model` override is rejected.

## Best Possible Solution

Emit an accurate rejection warning on the fail-closed cron path, omit fallback wording or fields unless a fallback is actually attempted, and add focused regression coverage for the log fields.

## Reproduction Assessment

Yes, source-reproducible: current main rejects a disallowed cron `payload.model` without emitting through `model-fallback/decision`, and the caller returns an error before any model call. I did not establish a live cron reproduction in this read-only review.

## Solution Assessment

No. Adding a WARN is a reasonable direction, but this patch should describe a rejected/failed cron selection instead of claiming a fallback that the runtime does not perform.

## Review Findings
