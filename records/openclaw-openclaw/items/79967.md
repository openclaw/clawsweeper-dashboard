## Summary

Keep open: the linked observability gap is plausible, but this PR is not mergeable as written because the new warning says the cron job falls back even though the current code returns an error, and the supplied proof only mocks the logger.

## What This Changes

The branch imports a model-fallback subsystem logger and emits a warning when cron `payload.model` validation returns an error.

## Best Possible Solution

Keep cron fail-closed and add an accurate `model-fallback/decision` warning for rejected `payload.model` outcomes, backed by real cron log proof or focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main returns `ok: false` for a rejected cron `payload.model`, and `run.ts` converts that into a `cron-preflight` error before provider execution. I did not execute a live cron job in this read-only review.

## Solution Assessment

No, not as written. Adding a warning is a narrow maintainable direction, but the log must describe the rejection/error outcome rather than a fallback that does not happen.
