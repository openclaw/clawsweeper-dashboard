## Summary

Keep open: the PR targets a real observability gap, but the current diff misdescribes the fail-closed cron path as a fallback and the supplied proof is only a mock logger simulation rather than real OpenClaw cron output.

## What This Changes

The PR imports the model-fallback subsystem logger and emits a WARN before returning `ok: false` for rejected cron `payload.model` values.

## Best Possible Solution

Add an accurate `model-fallback/decision` warning that names the rejected payload model and error outcome without implying execution fallback, then back it with real cron log proof or focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main returns `ok: false` for a rejected cron `payload.model`, and `run.ts` turns that into a `cron-preflight` error before provider execution. I did not run a live cron job in this read-only review.

## Solution Assessment

No, not as written. Adding a warning is a narrow maintainable direction, but the log must describe the rejection/error outcome rather than a fallback that does not happen.
