This is a fresh Windows browser-control startup report against v2026.5.12 with concrete reporter evidence, and current main still has the named caller passing `loaderFilename: modulePath`; however the shared loader cache also normalizes that value, so the exact unnormalized runtime boundary needs a Windows packaged repro before cleanup or closure.

The issue reports a Windows browser-control startup crash when a raw drive-letter import path reaches the ESM loader.

## Best Possible Solution

Reproduce against the packaged Windows browser-control startup path, then normalize the remaining raw drive-letter boundary once at the shared loader/caller seam and add a focused regression covering the generated package path.

## Solution Assessment
