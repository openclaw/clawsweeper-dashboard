## Summary

The latest head fixes the previous classified `server_error` gap, but the PR still needs changes: normalization now always synthesizes clean `currentAttemptReplayMetadata` when the field is absent, so the intended legacy fallback to cumulative `replayMetadata` is bypassed before the retry gate runs.

## What This Changes

The PR adds optional current-attempt replay metadata, derives it during embedded-attempt normalization, uses it for silent-error retry eligibility, routes classified `server_error` into that retry gate, and adds focused tests.

## Best Possible Solution

Land one canonical fix that has the attempt producer emit explicit current-attempt replay metadata, preserves conservative fallback when that field is absent, keeps classified `server_error` retry coverage, and blocks same-attempt side effects.

## Reproduction Assessment

Yes, at source level: current main blocks retry on cumulative replay metadata, and PR head shows the new fallback can be bypassed because normalization always creates `currentAttemptReplayMetadata`. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted: separating current-attempt metadata is the right direction, but computing it unconditionally in normalization is not the safest contract boundary. The better fix is to emit explicit per-attempt metadata from the attempt producer and leave absent metadata conservative.
