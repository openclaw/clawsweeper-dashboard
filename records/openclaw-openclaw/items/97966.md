## Summary

Keep open: the linked bug is real and source-reproducible on current main, but PR head still has a blocking retry-safety defect where prior dirty replay state can hide same-attempt unsafe tool side effects.

## What This Changes

The PR adds per-attempt replay metadata for empty provider-error retry gating and routes classified `server_error` through the retry gate.

## Best Possible Solution

Keep the per-attempt metadata direction, but source it from an attempt-local tracker that starts clean for each attempt and records same-attempt tools, messaging deliveries, cron adds, spawns, async work, and missing metadata independently of prior session state.

## Reproduction Assessment

Yes. Current main source shows later attempts receive accumulated replay state and the silent-error retry helper rejects attempts whose resolved replay metadata has `hadPotentialSideEffects`, matching the linked issue's prior-tool plus transient provider-error path.

## Solution Assessment

No as submitted. The PR chooses the right boundary by separating current-attempt metadata from cumulative metadata, but the boolean delta source is not safe when the initial replay state is already dirty.
