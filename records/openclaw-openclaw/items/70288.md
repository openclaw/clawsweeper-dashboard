## Summary

Keep open. The reported bug is valid, but the current PR is still incomplete: it changes the direct dispatch return value while the production buffered wrapper still normalizes `queuedFinal` back to false when no final reply was enqueued in the current cycle. The branch also lacks the required changelog entry for a user-facing fix.

## What This Changes

The branch changes the plugin-bound handled branch in `dispatchReplyFromConfig` to return `queuedFinal: true` and updates three direct dispatch tests to expect that result.

## Best Possible Solution

Carry an explicit plugin-claimed or response-committed signal through the production dispatch result without treating missing, no-handler, declined, error, or cancelled final delivery as a completed visible response. Cover the direct dispatch, wrapper, and Telegram fallback paths, and add the active-version changelog entry for the user-visible fix.

## Reproduction Assessment

Yes. The source path is enough to reproduce at code level: a plugin-bound `handled` claim can return raw `queuedFinal: true` with zero final counts, then `dispatchInboundMessage` converts it to false and Telegram’s fallback condition still matches when no delivery occurred.

## Solution Assessment

No. Returning true in `dispatchReplyFromConfig` is necessary but not sufficient because the production wrapper conflates a committed async response with an in-cycle final enqueue count; the fix needs to preserve that distinction through `dispatchInboundMessage`.
