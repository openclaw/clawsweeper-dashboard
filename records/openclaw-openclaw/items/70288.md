## Summary

Keep open. The PR targets a real bug and the direct-dispatch change is useful, but it is incomplete because the production Telegram path consumes the finalized buffered-dispatch result, where an async plugin claim with zero final replies is still normalized back to `queuedFinal: false`.

## What This Changes

The PR changes the plugin-bound handled branch to report `queuedFinal: true` and updates three direct `dispatchReplyFromConfig` expectations.

## Best Possible Solution

Carry an explicit plugin-claimed or response-committed signal through `DispatchFromConfigResult` finalization so handled async claims suppress channel fallback, while missing, no-handler, declined, error, and cancelled-final paths keep their current non-final behavior. Add coverage at the wrapper level and a Telegram regression test for a handled async claim with no immediate visible reply.

## Reproduction Assessment

Yes. The high-confidence reproduction path is code-level: a plugin-bound `inbound_claim` handled result can commit to an async response with `counts.final === 0`, the wrapper finalizer returns `queuedFinal: false`, and Telegram then enters its no-delivery fallback branch. The PR body's `openclaw-codex-app-server` verification is useful supporting live-context evidence.

## Solution Assessment

No. Returning `queuedFinal: true` from the direct `dispatchReplyFromConfig` branch is necessary but not sufficient, because production callers read the finalized buffered-dispatch result. The narrower maintainable fix is to preserve the committed-response distinction through finalization instead of relying on final reply counts alone.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve claimed responses through the wrapper:** `src/auto-reply/reply/dispatch-from-config.ts:658`
  - body: The new `queuedFinal: true` return is still normalized by `dispatchInboundMessage` to `result.queuedFinal && counts.final > 0`. A handled async plugin claim normally queues no final reply in this dispatch cycle, so Telegram still receives `queuedFinal: false` and can send the silent-reply fallback. Please carry the plugin-claimed response signal through the wrapper and cover that path.
