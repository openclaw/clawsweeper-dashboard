## Summary

This issue should stay open: current `main` still keeps status context lookups read-only/non-refreshing and gateway early startup does not warm the context-window cache, while https://github.com/openclaw/openclaw/pull/92968 is an open closing PR for the same bug.

## What This Changes

The issue requests warming the context-window cache at gateway startup or status resolution so the first cold-start `/status` reports model-specific context limits instead of the 200k fallback.

## Best Possible Solution

Keep this issue open until the linked PR or a coordinated replacement lands a deterministic first-cold-status fix that preserves the intended status/gateway startup safety boundaries.

## Reproduction Assessment

Yes at source level, but not from a live cold-start run: current `main` still has status paths that pass `allowAsyncLoad: false` and no gateway startup context-window warmup. The exact visible failure should be proven by the linked PR before merge.

## Solution Assessment

Yes, this is an appropriate core bug to track, but the final implementation should be the reviewed linked PR or a coordinated replacement. The fix should make the first returned status deterministic without accidentally turning status rendering into unbounded provider/catalog startup IO.
