## Summary

Keep open: current main and the latest release still keep status/session context-window lookup on read-only no-async paths without Gateway startup warming, so the cold first-status 200K fallback remains source-reproducible. The active fix path appears to be https://github.com/openclaw/openclaw/pull/92930, so this issue should wait for that PR to land, be replaced, or be explicitly rejected.

## What This Changes

The issue requests warming model context-window metadata before the first cold Gateway status response so status displays model-specific limits instead of the 200K fallback.

## Best Possible Solution

Keep this issue open, review https://github.com/openclaw/openclaw/pull/92930 as the active fix candidate, and have the eventual merged fix explicitly close or reference this report.

## Reproduction Assessment

Yes, as a source-level reproduction rather than a live run. Current main still routes status/session context lookup through cfg-backed, allowAsyncLoad:false paths while Gateway startup does not warm the context-window cache.

## Solution Assessment

Unclear whether the exact proposed shape is the best final fix until the active PR is reviewed. The maintainable end state should make the first status result deterministic without reintroducing unbounded provider/catalog work into Gateway hot paths.
