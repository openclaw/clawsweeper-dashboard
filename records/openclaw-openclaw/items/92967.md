## Summary

Keep open: current main and the latest release still leave cold status/session context-window reads on synchronous no-discovery paths, while https://github.com/openclaw/openclaw/pull/92930 is the active candidate fix for the same 200K first-status fallback.

## What This Changes

The issue asks OpenClaw to warm or otherwise populate the context-window cache before a cold gateway status read so the first `/status` does not display the 200K fallback for every model.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92930 or a narrow successor that fixes first-status context windows without making status rendering block on unsafe provider/catalog I/O.

## Reproduction Assessment

Yes, from source inspection: current main still has status/session context-window paths that pass `allowAsyncLoad: false`, and gateway startup still does not call `ensureContextWindowCacheLoaded`. I did not run a live cold-start gateway in this read-only review.

## Solution Assessment

Keeping this issue open is the best triage outcome because current main is not fixed and an open PR already owns the likely repair. The exact implementation should be reviewed on that PR because the status/startup cache-warming tradeoff is compatibility-sensitive.
