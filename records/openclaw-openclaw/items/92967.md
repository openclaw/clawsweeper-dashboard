## Summary

Keep this open. Current main partially covers CLI status through read-only static catalog metadata, but the cold Gateway/chat status path still does not warm context-window cache at startup and `buildStatusText` still performs non-refreshing context lookups that can fall back to 200k.

## What This Changes

The issue asks for cold-start Gateway/status context-window metadata to be warmed or otherwise prepared so the first `/status` call reports model-specific limits instead of the 200k default.

## Best Possible Solution

Preserve first-status correctness through a bounded read-only/static or background-warmed context path, without blocking Gateway readiness or broadening provider/catalog imports on hot paths.

## Reproduction Assessment

Yes, at source level: current main still has non-refreshing chat status lookups and no Gateway startup context-window warmup. I did not run a live cold Gateway `/status` scenario in this read-only cleanup review.

## Solution Assessment

Unclear for the exact proposed startup-warm approach. The best fix should keep the read-only/status startup contract while making first status output use static or already-warmed model-specific context metadata.
