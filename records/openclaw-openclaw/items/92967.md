## Summary

This issue should stay open: current main still keeps the relevant status context-window lookups read-only/non-refreshing and Gateway startup does not warm the context-window cache, while https://github.com/openclaw/openclaw/pull/92968 is an open closing PR for the same report. ## What This Changes

Request to make cold-start Gateway `/status` show model-specific context windows on the first call instead of falling back to the 200K default.

## Best Possible Solution

Review and land or replace one canonical PR that makes the first cold status response deterministic while preserving status read-only and startup lazy-loading boundaries.

## Reproduction Assessment

Yes, from source inspection: current main still has status paths passing `allowAsyncLoad: false` and no Gateway startup context-window warmup. I did not run a live cold-start Gateway scenario in this read-only review.

## Solution Assessment

Unclear whether the proposed startup warmup is the best final fix because current status paths intentionally avoid provider/catalog IO while rendering. The best fix should make first status output deterministic without broadening hot startup/lazy-loading behavior more than necessary.
