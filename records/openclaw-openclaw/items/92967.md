This issue should stay open: current main and the latest release still keep the relevant status lookups non-refreshing, and the issue is paired with an open closing implementation PR that has not landed.

## What This Changes

The issue asks OpenClaw to warm or prepare model context-window metadata so the first cold-start `/status` reports the model-specific context window instead of the 200K fallback.

## Best Possible Solution

Keep this issue open until one canonical linked PR makes the first cold `/status` result deterministic while preserving bounded status and Gateway startup behavior.

## Reproduction Assessment

Yes, source-reproducible: current main still has `/status` resolution paths that pass `allowAsyncLoad: false`, and Gateway startup does not warm `ensureContextWindowCacheLoaded`. I did not run a live cold-start gateway scenario in this read-only review.

## Solution Assessment

Unclear whether the proposed startup/background warming is the best final fix. The maintainable solution should make the first formatted status response use bounded model-specific metadata without unbounded provider/catalog work in hot paths.
