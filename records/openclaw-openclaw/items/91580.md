## Summary

Keep open: this targets a real dense-stream overhead bug and includes useful Testbox proof, but the current patch can emit stale assistant-message updates when a provider omits `partial` without mutating the prior snapshot.

## What This Changes

The branch makes plain `text_delta` events lightweight for OpenAI-compatible, OpenAI Responses, and Ollama streams, updates agent-loop handling, and adds regression tests.

## Best Possible Solution

Keep the lightweight-delta direction, but make partial-less text deltas reconstruct or mutate the current assistant text consistently across providers and add Ollama-backed coverage before merge.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from the linked issue and PR proof: a dense fake OpenAI-compatible backend emitting 30,000 one-character chunks exposes repeated full-partial work. I did not rerun it in this read-only review.

## Solution Assessment

No, not as currently patched. Lightweight text deltas are the right direction, but the implementation needs provider-independent current-message reconstruction and explicit stream-contract compatibility handling before merge.

- **[P2] Rebuild the current message when partial is omitted:** `packages/agent-core/src/agent-loop.ts:405`
  - body: This fallback only works for streams that mutate the previous partial object. Ollama builds fresh snapshot messages from `accumulatedVisibleContent`, and after this PR its `text_delta` events no longer carry a new snapshot, so `message_update.message` stays at the `text_start` snapshot until `text_end`; extension consumers that read the message field get stale assistant content during streaming. Update the agent loop to apply `event.delta` to the current text block, or keep a lightweight mutable current message for Ollama, and cover that path.
