## Summary

Keep open: the linked SSH tunnel bug is real, but this PR wires the port probe to the remote SSH hostname instead of the local loopback bind address, which would break normal remote SSH targets before `ssh` starts. It also needs real after-fix behavior proof before merge.

## What This Changes

The PR changes `startSshPortForward` to pass the parsed SSH target host into `ensurePortAvailable` before starting the local SSH port forward.

## Best Possible Solution

Probe `127.0.0.1` at the SSH tunnel caller, remove the issue-specific implementation comment, and add focused coverage plus real terminal/log proof for the preferred-port fallback path.

## Reproduction Assessment

Yes for the review finding: the source path passes `parsed.host` to a local `net.listen` bind, and a Node runtime probe shows a non-local hostname fails with `EADDRNOTAVAIL`. The original issue remains source-reproducible on current main because the SSH preflight is still hostless while the tunnel path uses `127.0.0.1`.

## Solution Assessment

No. Passing `parsed.host` is the wrong layer because that value is the remote SSH target; the narrow maintainable fix is to pass the local loopback address that the SSH tunnel owns, `127.0.0.1`, with focused regression proof.
