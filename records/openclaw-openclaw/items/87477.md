## Summary

Keep open for maintainer merge review: current main and v2026.5.27 still lack the package install/load gate, and this PR is a viable focused fix with no blocking code findings found. The remaining concern is an intentional compatibility policy change: too-new external package plugins will now fail install or skip load until the operator upgrades OpenClaw or pins a compatible plugin.

## What This Changes

The branch adds an `openclaw.compat.pluginApi` package metadata contract, npm/package install preflight checks, discovery and manifest-registry load skips, docs, and focused regression coverage.

## Best Possible Solution

Land this fail-closed package API gate with the documented compatibility contract, then track compatibility-aware package selection as a separate follow-up.

## Reproduction Assessment

Yes, source inspection shows current main only enforces the ClawHub plugin API gate and npm install metadata lacks the package `openclaw.compat.pluginApi` check before managed-root mutation. The linked issue and PR proof give a concrete WhatsApp/package-version scenario.

## Solution Assessment

Yes, if maintainers accept the fail-closed policy: checking package metadata before mutation and again during discovery/registry load is the narrowest boundary for preventing later runtime API crashes. A compatible-version resolver is useful follow-up work, not required for this first guard.
