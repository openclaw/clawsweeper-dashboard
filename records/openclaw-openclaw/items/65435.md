## Summary

Keep open. Current main still lacks the chunk-table fallback for no-provider, FTS-unavailable memory search, so this PR addresses a real gap, but the PR head is not mergeable as-is: it is dirty against current main and introduces fallback behavior that can override vector-only semantics, bypass merged hybrid ranking, and miss current source-filter contracts.

## What This Changes

The PR adds a memory-core chunk-scan text fallback when FTS5 is unavailable, re-enables memory search tests, skips trigram helper tests without FTS5, and carries stale fixture adjustments in plugin registry, doctor migration, and gateway close tests.

## Best Possible Solution

Rebase the branch onto current main and land a scoped memory-core fallback that scans `chunks` only for the lexical path that lacks FTS, preserves vector-only and merged hybrid ranking semantics, threads current source filters through the fallback, and drops stale unrelated fixture/doctor changes unless they are still required after rebase.

## Reproduction Assessment

Yes. With indexed `chunks`, no embedding provider, and unavailable or disabled FTS, current main returns `[]` before querying `chunks`; schema inspection shows `chunks` can exist even when the FTS virtual table cannot be created.

## Solution Assessment

No. The PR direction is useful, but the current patch is not the best fix because it widens lexical fallback into vector-only/provider paths, bypasses the merged hybrid ranking pipeline, and needs to be reconciled with current source-filter behavior.

Full review comments:

- **[P2] Preserve vector-only mode when fallback is lexical:** `extensions/memory-core/src/memory/manager.ts:400-401`
  - body: This branch also runs when `hybrid.enabled` is false. If the embedding is a zero vector or vector search fails, vector-only configurations now scan chunks lexically and can return substring hits where the previous vector-only path returned no results; gate the chunk fallback away from `!hybrid.enabled`.
  - confidence: 0.9
- **[P2] Keep relaxed keyword fallback on merged ranking:** `extensions/memory-core/src/memory/manager.ts:420`
  - body: Returning raw `keywordResults` here skips `mergeHybridResults`, including temporal decay and MMR ordering. The relaxed fallback should preserve the merged keyword-backed entries so stale or duplicate chunks do not jump ahead of the configured ranking policy.
  - confidence: 0.88
- **[P2] Thread source filters into fallback scans:** `extensions/memory-core/src/memory/manager.ts:526-535`
  - body: The fallback helper builds a filter from all enabled sources. Current main passes requested source filters into FTS and vector search before ranking and limiting, so after rebase this fallback can return hits from the wrong corpus unless it accepts and uses the same source filter list.
  - confidence: 0.84
