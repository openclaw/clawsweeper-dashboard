## Summary

Keep open: current main still has the no-FTS memory recall gap, but this PR is not merge-ready because it lacks affected-runtime proof and still changes vector-only/provider search behavior, ranking/source-filter handling, and scan bounds.

## What This Changes

The PR adds a chunks-table lexical fallback for memory search when FTS5 is unavailable, re-enables memory-core tests, and includes adjacent doctor/plugin-registry/gateway fixture fixes.

## Best Possible Solution

Land a rebased, bounded fallback scoped to the missing-FTS keyword gap that preserves provider/vector-only semantics, source filters, temporal/MMR ranking, and includes focused tests plus redacted affected-runtime proof.

## Reproduction Assessment

Yes, at source level: current main records FTS unavailable when FTS5 table creation fails, and the no-provider path then returns an empty result set instead of searching indexed chunks. I did not run a live affected Node runtime in this read-only review.

## Solution Assessment

No as written. A fallback is a reasonable direction, but this branch needs a rebase plus vector-only, ranking, source-filter, and bounded-scan fixes before it is the narrow maintainable solution.

  - body: This branch also runs when `query.hybrid.enabled` is false or FTS is unavailable for a provider-backed search. If vector search returns no rows, the PR scans chunks lexically, so vector-only configurations can return substring hits instead of honoring the existing vector-only contract.

  - body: Current main passes requested source filters into vector and FTS searches before ranking and limiting. The new fallback builds the default source filter only, so corpus-scoped memory searches can rank and limit against the wrong source after rebase.
  - confidence: 0.84
- **[P2] Bound the chunks-table fallback scan:** `extensions/memory-core/src/memory/manager.ts:531-537`
  - body: The fallback selects every chunk row and scans text in JavaScript before slicing to `limit`. On affected no-FTS installs with large memory stores, each search can load and process the whole chunks table on the gateway path; use bounded SQL terms or batched rowid scanning with yields.
  - confidence: 0.82
