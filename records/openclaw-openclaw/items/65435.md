## Summary

Keep open. Current main still has the underlying unavailable-FTS memory search gap, but this PR's current head introduces provider/vector-only regressions, bypasses the existing merged hybrid ranking fallback, and does not preserve the newer source-filter contract that current main relies on.

## What This Changes

The PR adds a memory-core chunks-table keyword fallback when FTS5 is unavailable, re-enables/skips related memory tests, and adjusts stale doctor, plugin-registry, and gateway test fixtures.

## Best Possible Solution

Land a narrow memory-core fix on current main that scans `chunks` only for the unavailable-FTS lexical path, threads caller source filters through that scan, preserves provider-backed vector-only semantics, and keeps relaxed hybrid fallback ordered through the existing merged ranking pipeline.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: with indexed `chunks`, no provider, and unavailable or disabled FTS, `search()` returns `[]`; the PR regressions are also visible directly in the provider fallback and hybrid ranking branches at the PR head.

## Solution Assessment

No. The implementation direction is useful, but this PR head is not the best fix until it avoids lexical fallback in provider/vector-only paths, keeps merged ranking for relaxed hybrid results, and carries current source filters into the chunks scan.
