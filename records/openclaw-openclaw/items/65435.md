## Summary

Keep open; the fallback is a plausible direction for the open memory-search bug, but the current branch lacks real affected-runtime proof, is conflicting against current main, and still introduces search-contract and runtime-safety regressions that should block merge.

## What This Changes

The branch adds a memory-core chunks-table keyword fallback when FTS5 is unavailable, re-enables or skips related memory tests, and includes unrelated fixture/config test adjustments for plugin registry, doctor migration, and gateway close tests.

## Best Possible Solution

Land a rebased memory-core fallback limited to the missing-FTS lexical gap, preserving vector-only/provider semantics, source filters, merged ranking, and bounded scans with focused tests plus redacted affected-runtime proof.

## Reproduction Assessment

Yes, at source level: if `CREATE VIRTUAL TABLE ... USING fts5` fails, current main records FTS unavailable, and no-provider search returns empty results while provider-backed search skips keyword recall. I did not run a live Node build without FTS5 in this read-only review.

## Solution Assessment

No, not as written. A chunk-table fallback is a reasonable repair direction, but this PR needs a rebase plus vector-only, ranking, source-filter, bounded-scan, and affected-runtime proof before it is the narrow maintainable solution.
