## Summary

Keep open: current main still has a source-reproducible no-provider/no-FTS memory search gap, but this PR head is conflicting and still changes vector-only, hybrid ranking, and source-filter behavior, with no real FTS-unavailable runtime proof after the fix.

## What This Changes

The PR adds a chunks-table lexical fallback for memory search when FTS5 is unavailable, re-enables memory-core tests, skips FTS-dependent helper tests on unsupported SQLite builds, and adjusts unrelated CLI/gateway/doctor fixtures.

## Best Possible Solution

Land a rebased memory-core fallback that scans chunks only for the no-provider/no-FTS lexical path while preserving vector-only semantics, hybrid ranking, source filters, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows that with indexed chunks, no embedding provider, and disabled or unavailable FTS, current main returns [] even though chunks can exist without chunks_fts.

## Solution Assessment

No. The fallback direction is appropriate, but this PR head is not the best mergeable fix because it reaches vector-only/provider paths, bypasses merged ranking, omits current source filters, and lacks real runtime proof.

  - body: This branch also runs when `query.hybrid.enabled` is false or FTS is unavailable for a provider-backed search.   - body: Current main passes requested source filters into vector and FTS searches before ranking and limiting. This fallback builds the default source filter only, so after rebase it can rank and limit against the wrong corpus and suppress requested hits.
  - confidence: 0.84
