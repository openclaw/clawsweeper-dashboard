## Summary

Keep open: current main and the latest release still use a hostless `isPortBusy` probe on the `gateway --force` path, and the related open PR explicitly leaves this lower-harm sibling surface for a separate follow-up.

## What This Changes

The issue requests fixing `gateway --force` port-busy detection so an IPv4-only listener is not missed by the `isPortBusy` preflight.

## Best Possible Solution

Land a narrow follow-up that makes the gateway force busy check use the shared multi-address port probe and adds focused regression coverage for an IPv4-only listener.

## Reproduction Assessment

Yes for the affected dual-stack/macOS path: bind `127.0.0.1:<port>` and exercise the current `isPortBusy` hostless probe. This review verified the source path and sibling proof, but did not mark it locally reproduced because the Linux runner returned `EADDRINUSE` for the hostless probe.

## Solution Assessment

Yes. Reusing the existing multi-address port-inspection logic is the narrowest maintainable fix because it keeps port probing policy in one helper instead of adding another gateway-specific address-family rule.
