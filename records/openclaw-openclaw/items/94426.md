## What This Changes

The issue reports that the `gateway --force` port-busy preflight misses IPv4-only listeners because `isPortBusy` uses a hostless listen probe.

## Best Possible Solution

Review and land one focused linked PR that reuses the existing multi-endpoint port probe for `isPortBusy`, then close this issue after the merge.

## Reproduction Assessment

Yes, from source inspection: `isPortBusy` still performs a hostless bind and `forceFreePortAndWait` exits early when that probe says the port is free. I did not run a live socket repro in this read-only sweep, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to route `isPortBusy` through the existing multi-family `checkPortInUse` helper and add focused regression coverage. The practical next step is to review one linked PR rather than open a new repair lane.
