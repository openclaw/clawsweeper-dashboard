## What This Changes

The issue reports that `isPortBusy` in the `gateway --force` port-reclaim flow can miss IPv4-only listeners because it uses a hostless listen probe.

## Best Possible Solution

Land one focused PR that routes `isPortBusy` through the existing multi-family port check, preserves inconclusive-probe handling, and adds focused `gateway --force` regression coverage.

## Reproduction Assessment

No high-confidence live repro was established in this read-only review. The source-level path is clear: current `main` still uses a hostless `isPortBusy` probe and the force flow exits when that helper reports false.

## Solution Assessment

Yes, the requested direction is the best fix shape: reuse the existing multi-family port probe instead of adding another one-off host probe. The landing PR should handle `unknown` distinctly so an inconclusive bind probe does not look like a free port.
