Report that `gateway --force` can miss an IPv4-only listener because `isPortBusy` probes a port without specifying a host.

## Best Possible Solution

Review and land one focused linked PR that reuses or cleanly exposes the multi-family port probe for `isPortBusy`, with a targeted regression test for the force-port path.

## Reproduction Assessment

Yes for source-level reproduction: current main still uses the hostless `isPortBusy` probe, and `forceFreePortAndWait` returns early when that helper reports free. I did not run a live IPv4-only listener repro in this read-only review.

## Solution Assessment

Yes, the best fix remains to route `isPortBusy` through the existing multi-family port probe rather than adding another one-off host probe, then cover the force path with focused regression coverage.
