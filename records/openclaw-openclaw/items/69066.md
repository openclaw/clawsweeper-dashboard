## Summary

Keep open. Current main and the latest shipped release still do not provide a distinct gateway-owned local service principal/service-token path, and the active discussion includes concrete trusted-proxy user impact plus design constraints from the trusted-proxy feature history.

## What This Changes

Request to split gateway-owned internal service authentication from user-facing gateway auth modes with a phased local service principal/token design.

## Best Possible Solution

Keep the RFC open until maintainers approve the local-service-principal contract, then implement a small additive SecretRef-backed loopback-only service-token path with focused auth, client, config, docs, and regression tests before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source inspection on current main shows trusted-proxy mode rejects loopback requests with trusted_proxy_loopback_source before any service-token path exists, and tests assert local-direct trusted-proxy credentials remain rejected.

## Solution Assessment

Unclear until maintainer auth/security review. The RFC direction is plausible, but the maintainable first step should follow the loopback source plus SecretRef-backed local_service path described in the maintainer comment rather than adding more trusted-proxy fallbacks.
