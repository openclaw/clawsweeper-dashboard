## Summary

Keep open. Current main and v2026.5.18 still expose trusted-proxy, token, password, Tailscale, device, and bootstrap auth paths, but not the requested independent service-principal axis; the issue also has recent user deployment evidence and a maintainer-authored refinement that makes this a security/product contract decision rather than stale cleanup.

## What This Changes

The issue requests a phased gateway auth design that separates internal service identity from user-facing auth modes and local operator auth.

## Best Possible Solution

Approve and implement a narrow loopback-only SecretRef-backed local_service principal first, keep trusted-proxy user semantics unchanged, and track named external service principals separately.

## Reproduction Assessment

Yes, source-level. Current main has no service-principal auth result or gateway.service config, and the auth code plus tests still show internal callers relying on trusted-proxy/password fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the security contract. The strongest path in the discussion is a loopback-only SecretRef-backed local_service principal first, rather than adding more trusted-proxy fallback branches or broad external principals immediately.
