## Summary

Keep open. Current main still lacks a separate gateway service/local_service auth path, loopback trusted-proxy/local-direct callers still hit the trusted-proxy rejection path, and the active maintainer discussion narrows this into a security-sensitive credential design that needs approval before implementation.

## What This Changes

Request an RFC/design path to separate gateway internal service identity from user auth for trusted-proxy deployments and loopback internal callers.

## Best Possible Solution

Keep the RFC open until maintainers approve the narrowed SecretRef-backed, loopback-only local_service contract, then implement it additively with schema, docs, tests, and a migration plan before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence current-main reproduction path: loopback trusted-proxy/local-direct token or password credentials still return trusted_proxy_loopback_source, and there is no service credential path to try first.

## Solution Assessment

Unclear until maintainer auth/security approval. The maintainable direction is the narrowed loopback plus SecretRef-backed local_service principal described in the maintainer comment, not widening trusted-proxy headers or reviving broad local password fallbacks.
