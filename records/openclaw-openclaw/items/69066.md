## Summary

Keep open. The RFC remains active gateway auth/security design: current main has tactical trusted-proxy relief paths, but no first-class loopback-only service identity or `local_service` principal, and a trusted-proxy author explicitly narrowed the acceptable Phase 1 direction.

## What This Changes

The issue proposes separating gateway-local service identity from user-facing gateway auth modes so internal callers stop depending on trusted-proxy, password, token, or device-pairing fallbacks.

## Best Possible Solution

Keep the RFC open until maintainers settle the auth/security shape, then implement a small additive loopback-only SecretRef-backed `local_service` principal with focused gateway auth/client/config/docs tests before retiring legacy fallbacks.

## Reproduction Assessment

Yes, for the current coupling. Source, docs, and tests show internal/local-direct callers in trusted-proxy mode still depend on password fallback or reject with trusted-proxy reasons rather than a separate service identity path.

## Solution Assessment

Unclear until maintainer security review. The requested direction is plausible and supported in discussion, but the best implementation should follow the narrowed loopback plus SecretRef-backed local service principal design rather than broadening trusted-proxy fallbacks.
