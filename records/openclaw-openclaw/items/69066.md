## Summary

Keep open. Current main still lacks an independent local service identity path: gateway auth results, config, client connect payloads, docs, and tests all keep internal callers on the existing shared-secret, device, or trusted-proxy branches. The issue also has fresh deployment confirmations and maintainer design feedback, so this is active gateway auth/security design work rather than cleanup closure.

## What This Changes

The issue requests a first-class local/internal gateway service identity path separate from user-facing gateway auth modes such as trusted-proxy, token, and device pairing.

## Best Possible Solution

Keep the RFC open and implement the approved local service identity as small additive PRs: SecretRef-backed gateway-local token config/storage, GatewayClient injection, auth result/principal handling, loopback peer-address validation, docs and doctor guidance, and regression tests across none/token/trusted-proxy modes before retiring legacy fallbacks.

## Reproduction Assessment

Yes for the current coupled behavior. Current tests reproduce valid local-direct trusted-proxy token attempts rejecting with trusted_proxy_loopback_source while password fallback succeeds, and the issue comments add real nginx/oauth2-proxy/Keycloak and Traefik/authentik deployment confirmations.

## Solution Assessment

Unclear until maintainer auth/security review completes. The likely best direction is the narrower shape from nickytonline's comment: actual loopback peer address plus SecretRef-backed local service token, timing-safe comparison, invalid-token fail-closed behavior, and a single coarse local_service principal rather than treating localhost as trusted-proxy traffic.
