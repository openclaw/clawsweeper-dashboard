## Summary

Keep open. Current main still has no gateway-local service identity path: internal clients authenticate through existing token, password, bootstrap, or device credentials, while trusted-proxy local-direct handling remains coupled to loopback rejection and password fallback. The recent issue comments add real deployment confirmations and maintainer design constraints, so this is active auth/security design work rather than cleanup closure.

## What This Changes

Request to add a first-class gateway-local service identity/authentication path separate from user and trusted-proxy auth for internal OpenClaw callers.

## Best Possible Solution

Keep the RFC open for maintainer auth/security review, settle the Phase 1 local service-auth model, then implement small additive PRs covering SecretRef-backed token generation/preservation/rotation, loopback-only validation using the actual socket peer address, GatewayClient injection, an explicit local_service auth result/principal, docs, doctor/migration behavior, and targeted regression tests.

## Reproduction Assessment

Yes. Current tests reproduce the coupled behavior by rejecting local-direct trusted-proxy token attempts with trusted_proxy_loopback_source while accepting only password fallback, and the issue comments add nginx/oauth2-proxy plus Traefik/authentik deployment confirmations.

## Solution Assessment

Unclear until maintainer security review. The best direction appears to be nickytonline's narrower local SecretRef-backed service token for direct loopback calls, not treating localhost as trusted-proxy traffic or adding more fallback branches to the existing user-auth decision tree.
