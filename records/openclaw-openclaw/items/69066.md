## Summary

Keep open. Current main and the latest shipped release still use tactical trusted-proxy/password fallback paths rather than the first-class service-principal auth contract requested here, and the discussion now has maintainer guidance plus additional deployment evidence that needs gateway/auth owner review.

## What This Changes

Requests a separate gateway service-identity auth axis for internal callers, distinct from user auth modes and local operator auth.

## Best Possible Solution

Approve and implement a narrow loopback-only SecretRef-backed `local_service` principal evaluated before user auth, then track scoped external service principals as a separate follow-up design.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.5.18` expose only the existing gateway auth modes and tests/docs still rely on trusted-proxy/password fallback behavior, so the requested service-principal path is absent.

## Solution Assessment

Unclear until gateway/auth owners approve the contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed `local_service` principal first, with external scoped principals handled separately.
