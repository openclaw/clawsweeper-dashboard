## Summary

Keep open. Current main still lacks a separate gateway-owned service identity path, and the issue remains an active auth/security RFC rather than stale cleanup or an implemented request.

## What This Changes

The issue requests a first-class internal service identity path for gateway-owned local callers, separate from user-facing gateway auth modes.

## Best Possible Solution

Keep the RFC open until maintainers settle the local-service-principal contract, then implement a small additive loopback-only SecretRef-backed service token path with focused auth/client/config/docs tests before retiring tactical fallbacks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a trusted-proxy config with a loopback request and no proxy identity headers rejects as `trusted_proxy_loopback_source`, and there is no service-token credential path to try first. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainer security review. The RFC direction is plausible, but the best solution should follow the narrowed loopback plus SecretRef-backed `local_service` design rather than broadening trusted-proxy fallbacks.
