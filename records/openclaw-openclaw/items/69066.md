## Summary

Keep open. Current main has Phase 0 trusted-proxy relief through explicit loopback/password paths, but it still lacks the first-class `local_service` or named service-principal auth contract requested here, and the active discussion has narrowed this into a security-sensitive gateway credential design that needs owner approval.

## What This Changes

The issue requests a first-class gateway service identity so internal and third-party service callers do not depend on user auth modes.

## Best Possible Solution

Approve a narrow loopback-only SecretRef-backed `local_service` principal first, keep user trusted-proxy semantics separate, then track named external service principals with scoped allowlists as a follow-up design if maintainers accept that direction.

## Reproduction Assessment

Yes, source-level. Current auth code, config schema, docs, and tests show there is no service-principal path and that internal callers still rely on trusted-proxy/password fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed `local_service` principal first, not more trusted-proxy fallback branches or a broad external-principal system in the first step.
