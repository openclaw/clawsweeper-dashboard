## Summary

Keep open. Current main still has no separate gateway service/local_service auth result or config surface, and trusted-proxy mode still returns before any local token/password credential can authorize an internal caller; the active maintainer discussion narrows this into a security-sensitive credential design that needs approval before implementation.

## What This Changes

The issue requests a first-class gateway local service identity that is separate from user-facing gateway auth modes for internal callers such as subagents, CLI probes, cron, browser tooling, and native approvals.

## Best Possible Solution

Keep the RFC open until maintainers approve the narrowed SecretRef-backed, loopback-only local_service contract, then implement it additively with schema, docs, tests, and a migration plan before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence current-main reproduction path: loopback trusted-proxy/local-direct token or password credentials still return trusted_proxy_loopback_source, and there is no service credential path to try first.

## Solution Assessment

Unclear until maintainer auth/security approval. The maintainable direction appears to be the narrowed loopback plus SecretRef-backed local_service principal described in the maintainer comment, not widening trusted-proxy headers or relying on more fallback branches.
