- `impact:auth-provider`: The affected surface is gateway auth mode routing, trusted-proxy behavior, token/password fallback, and future service principal resolution.

## Summary

Keep open. Current main and the latest release still use tactical trusted-proxy/local password fallback behavior rather than a first-class service-principal contract, and the discussion has narrowed this into a gateway/auth security design that needs maintainer approval.

## What This Changes

The issue requests separating gateway internal service identity from user auth so internal callers, local operators, and trusted-proxy users no longer share one gateway auth decision tree.

## Best Possible Solution

Approve a narrow loopback-only SecretRef-backed local_service principal first, keep trusted-proxy user semantics separate, and split cross-host named service principals into a later scoped design.

## Reproduction Assessment

Yes, source-level. Current auth code, config schema, docs, and v2026.5.18 all show no service-principal path, and internal callers still depend on shared token/password/trusted-proxy fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed local_service principal first, with external named principals split into a later design.
