## Summary

Keep open. Current main and v2026.5.18 still lack a separate gateway service or `local_service` auth principal; the remaining work is an active, security-sensitive gateway credential design rather than stale cleanup.

## What This Changes

Request to split gateway internal service/local operator auth from user auth by adding a first-class service identity path instead of extending `gateway.auth.mode` trusted-proxy fallbacks.

## Best Possible Solution

Keep the RFC open until gateway/security owners approve a narrow additive `local_service` contract, then implement it across auth, credentials, config/schema, internal caller injection, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. Current auth types, config types, docs, and tests show no service/local_service auth path, and trusted-proxy internal callers still rely on password fallback or fail with trusted-proxy reasons.

## Solution Assessment

Unclear until gateway/security owners approve the contract. The strongest current direction is a loopback-only SecretRef-backed `local_service` principal first, with external scoped principals handled as a later design rather than folded into Phase 1.
