## Summary

Keep open. Current main and the latest release still only expose the tactical trusted-proxy/password/local-direct auth paths, while the requested first-class local service identity remains a security-sensitive gateway/auth design decision with active maintainer refinement and user deployment evidence.

## What This Changes

The issue requests an RFC-level split between user authentication, internal service identity, and local operator authentication in the OpenClaw gateway.

## Best Possible Solution

Approve and implement a narrow loopback-only SecretRef-backed local_service principal first, keep trusted-proxy user semantics unchanged, and track named external service principals as a separate follow-up RFC.

## Reproduction Assessment

Yes, source-level. Current main has no service-principal auth result or gateway.service config, and the auth code plus tests still show internal callers relying on trusted-proxy/password fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the security contract. The best-supported path is the maintainer-refined loopback-only SecretRef-backed local_service principal first, rather than adding more trusted-proxy fallback branches or broad external principals in the first step.
