## Summary

Keep open. Current main and the latest release still implement only the trusted-proxy/password fallback paths, while the requested first-class service-principal auth contract remains a security-sensitive design decision with active maintainer discussion.

## What This Changes

Request to separate gateway internal service identity from user auth with an orthogonal service-principal credential path for internal callers and later external principals.

## Best Possible Solution

Approve a narrow loopback-only SecretRef-backed `local_service` principal first, keep user trusted-proxy semantics separate, and track named external service principals as a follow-up design.

## Reproduction Assessment

Yes, source-level. Current auth code, schema, tests, release tag, and docs show no service-principal path and show internal callers still rely on trusted-proxy/password fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the security contract. The narrowest maintainable direction appears to be loopback-only SecretRef-backed `local_service` first, not additional trusted-proxy fallback branches or a broad external-principal system in the first step.
