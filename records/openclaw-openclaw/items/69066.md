## Summary

Keep open. Current main still lacks a separate gateway-owned local service principal, and the live discussion contains concrete trusted-proxy/internal-caller impact plus maintainer design constraints that need auth/security approval before implementation.

## What This Changes

The issue requests an RFC/design for gateway-owned internal service identity separated from user auth, trusted-proxy auth, and local operator auth.

## Best Possible Solution

Keep the RFC open until maintainers settle the local_service contract, then implement an additive loopback-only SecretRef-backed service token path with config/schema/docs/tests before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source and tests show trusted-proxy mode rejects loopback/local-direct requests with trusted_proxy_loopback_source before any service credential path, and current main has no service credential to try first.

## Solution Assessment

Unclear until maintainer auth/security approval. The maintainable direction is the narrowed SecretRef-backed, loopback-only local_service principal described in the maintainer comment, not widening trusted-proxy or local password fallback behavior.
