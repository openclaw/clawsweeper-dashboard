## Summary

Keep open. Current main still has no independent gateway-local service identity, and the active discussion has narrowed this into auth/security design work rather than cleanup closure.

## What This Changes

The issue requests separating gateway internal service identity from user-facing gateway auth modes so local/internal clients no longer depend on trusted-proxy or local-direct fallback branches.

## Best Possible Solution

Keep the RFC open, settle the local service-auth trust model with maintainers, then implement it as small additive PRs that do not expand trusted-proxy fallback semantics.

## Reproduction Assessment

Yes. Current source and tests reproduce the coupled behavior: local-direct trusted-proxy token attempts fail with trusted_proxy_loopback_source, while only the configured password fallback is accepted.

## Solution Assessment

Unclear pending maintainer security review. The best direction appears to be the narrower SecretRef-backed local service token described by nickytonline, not another fallback branch inside the user-auth trusted-proxy path.
