## Summary

Keep open. Current main still has only the existing user/auth-mode branches and same-host trusted-proxy fallback knobs; it does not have a separate gateway-owned local service principal, and the live discussion includes concrete user impact plus maintainer design constraints.

## What This Changes

Request to add an orthogonal gateway-owned local service identity/service-token path for internal gateway callers, separate from user auth and trusted-proxy modes.

## Best Possible Solution

Keep the RFC open until maintainers settle the local-service-principal contract, then implement a small additive loopback-only SecretRef-backed service token path with focused auth, client, config, docs, and regression tests before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: trusted-proxy mode with a loopback request and no proxy identity headers returns trusted_proxy_loopback_source, and there is no service-token credential path to try first.

## Solution Assessment

Unclear until maintainer security review. The RFC direction is plausible, but the narrow maintainable solution should follow the loopback source plus SecretRef-backed local_service design described in the maintainer comment rather than widening trusted-proxy fallback behavior.
