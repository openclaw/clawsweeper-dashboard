## Summary

Keep open. Current main and the latest release still have tactical trusted-proxy/local-direct fallback behavior, but no first-class service-principal or local-service authentication contract, and the discussion has narrowed the next step into a gateway/auth security design that needs maintainer approval.

## What This Changes

The issue requests an RFC-level split between user auth, local service identity, and local operator auth in the gateway.

## Best Possible Solution

Approve a narrow loopback-only SecretRef-backed local service principal first, keep trusted-proxy user semantics separate, and track named external service principals with scoped allowlists as a follow-up design if maintainers accept that direction.

## Reproduction Assessment

Yes, source-level. Current auth code, config schema, docs, and tests show no service-principal path exists and internal callers still depend on trusted-proxy/password fallback behavior rather than a separate service identity axis.

## Solution Assessment

Unclear until gateway/auth owners approve the contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed local service principal first, with broader external named principals split into a later design.
