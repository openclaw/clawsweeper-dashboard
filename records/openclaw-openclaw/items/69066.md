## Summary

Keep open. Current main has tactical trusted-proxy password and allowLoopback relief, but it still has no first-class gateway-local service identity path; active deployment confirmations plus maintainer feedback make this auth/security design work rather than cleanup closure.

## What This Changes

This RFC requests a separate gateway-local service identity for internal callers, independent of user-facing gateway auth modes and trusted-proxy handling.

## Best Possible Solution

Keep the RFC open for maintainer auth/security review, then implement a small additive loopback-only SecretRef-backed local service principal with docs, doctor/install behavior, and focused regression tests before retiring legacy fallbacks.

## Reproduction Assessment

Yes, for the coupled current behavior. Source and tests show local-direct trusted-proxy token attempts still reject with trusted_proxy_loopback_source while internal callers can only use the password fallback path.

## Solution Assessment

Unclear until maintainer security review. The current fallback paths are tactical; the safer direction is the maintainer-suggested loopback plus SecretRef-backed local service token, not trusting localhost as proxy identity or adding more user-auth fallback branches.
