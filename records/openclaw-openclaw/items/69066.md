## Summary

Keep open. Current main and the latest stable release still lack a first-class service-principal auth path, and the active discussion has narrowed this into a security-sensitive gateway credential contract that needs owner review before implementation.

## What This Changes

Request to separate gateway internal service identity from user authentication and local operator auth.

## Best Possible Solution

Approve a narrow loopback-only SecretRef-backed local_service principal first, keep user trusted-proxy semantics separate, and track external named service principals as a separate follow-up if maintainers accept that direction.

## Reproduction Assessment

Yes, source-level. Current auth code, config schema, docs, and tests show there is no service-principal path and that internal callers still rely on trusted-proxy/password fallback behavior.

## Solution Assessment

Unclear until gateway/auth owners approve the contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed local_service principal first, not more trusted-proxy fallback branches or a broad external-principal system in the first step.
