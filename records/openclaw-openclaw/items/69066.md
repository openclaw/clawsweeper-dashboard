## Summary

Keep open. Current main still has no separate gateway service/local_service auth path, source and tests still keep loopback trusted-proxy/local-direct callers on the trusted-proxy rejection path, and the maintainer discussion narrows this into a security-sensitive local service credential design that needs approval before implementation.

## What This Changes

The RFC requests a new gateway-local service identity/auth axis for internal callers, separate from trusted-proxy and other user auth modes.

## Best Possible Solution

Keep the RFC open until maintainers approve the narrowed local_service contract, then implement an additive SecretRef-backed loopback-only service token path with schema, docs, tests, and a migration plan before retiring tactical trusted-proxy fallbacks.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence current-main reproduction path: loopback trusted-proxy/local-direct token or password credentials still return trusted_proxy_loopback_source, and no service credential path exists to try first.

## Solution Assessment

Unclear until maintainer auth/security approval. The maintainable direction is the narrowed loopback plus SecretRef-backed local_service principal described in the maintainer comment, not widening trusted-proxy headers or reviving broad local password fallbacks.
