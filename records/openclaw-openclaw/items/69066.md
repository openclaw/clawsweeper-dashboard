## Summary

Keep open. Current main still has no separate gateway service/local_service auth result or config surface, trusted-proxy auth still rejects loopback before token/password branches can authorize internal callers, and the recent maintainer discussion narrows this into a security-sensitive credential design that needs approval before implementation.

## What This Changes

Requests a new gateway-local service identity so internal gateway callers authenticate independently from user-facing auth modes such as trusted-proxy.

## Best Possible Solution

Keep the RFC open until maintainers approve a loopback-only SecretRef-backed local_service contract, then implement it additively with config/schema, token storage and rotation, internal caller injection, docs, and regression tests before retiring tactical fallback paths.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence current-main reproduction path: trusted-proxy mode rejects loopback local-direct token/password credentials with trusted_proxy_loopback_source, and there is no service credential path to try first.

## Solution Assessment

Unclear until maintainer auth/security approval. The narrowest maintainable direction appears to be the proposed loopback plus SecretRef-backed local_service principal, not adding more trusted-proxy fallback branches.
