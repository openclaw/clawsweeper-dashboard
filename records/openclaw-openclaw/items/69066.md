## Summary

Keep open. Current main and the latest release still have no separate gateway service or local_service auth principal, while the issue discussion has narrowed this into a security-sensitive gateway credential design that needs explicit owner approval.

## What This Changes

The issue requests an RFC-backed split between user auth, local operator auth, and internal gateway service identity for direct gateway calls.

## Best Possible Solution

Keep the RFC open until gateway/security owners approve a loopback-only SecretRef-backed local_service contract, then implement it additively across auth, credentials, config/schema, internal caller injection, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. Current source and tests show trusted-proxy mode rejects loopback local-direct token/password credentials with trusted_proxy_loopback_source, and there is no service credential path to try first.

## Solution Assessment

Unclear until gateway/security owners approve the auth contract. The narrowest maintainable direction appears to be a loopback-only SecretRef-backed local_service principal, not more trusted-proxy fallback branches.
