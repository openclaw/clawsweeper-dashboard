## Summary

Keep open: current main and v2026.6.6 still route Feishu text edits through the card-update PATCH endpoint, which matches the reported card-JSON parse failure when the target message is an interactive card. The empty presentation payload is already normalized away, so the fix should correct Feishu edit routing rather than close this as implemented.

## What This Changes

The issue asks OpenClaw to stop Feishu `message.edit` from failing with `parse card json err` when a plain-text edit follows an interactive-card send with an empty presentation payload.

## Best Possible Solution

Update the Feishu plugin edit path to ignore empty presentations, use Feishu's text/post edit API for text edits, and use valid interactive-card JSON for card edits, with regression tests covering text-only edits and card-origin text replacements.

## Reproduction Assessment

Yes at source level: editing a Feishu interactive-card message with plain text currently sends post content through `client.im.message.patch`, while Feishu's docs reserve PATCH for card updates. I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

No, dropping empty `presentation` alone is not the best fix because the text edit path still uses the card-only endpoint. The maintainable fix is to route Feishu edits by message/requested content type and keep card updates as valid card JSON.
