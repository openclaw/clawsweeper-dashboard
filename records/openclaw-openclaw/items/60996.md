## Summary

Keep open. Current main still does not provide all-platform subcommand-level exec allowlisting, but this PR is not merge-ready because its convenience subcommand expansion cannot match real commands and the new `ask="auto"` path can silently allow commands that should prompt or remain unknown.

## What This Changes

The PR enables all-platform `argPattern` checks, adds subcommand-pattern parsing and docs, introduces static/LLM command safety classification, and wires a new `ask="auto"` exec approval mode into gateway exec.

## Best Possible Solution

Ship the subcommand allowlist change as a narrow exec-approval enhancement with explicit argv semantics, policy normalization, docs, and focused tests; keep any automatic safety classifier fail-closed and behind separate security/product review.

## Reproduction Assessment

Yes. The broken `git add:*` path is reproducible by source inspection of the PR expansion versus `matchArgPattern`'s argv-slice matching, and the `ask="auto"` prompt bypass follows directly from the PR's `on-miss` mapping plus current `requiresExecApproval` semantics.

## Solution Assessment

No. The current PR direction is too broad and unsafe as written; the maintainable path is to fix the all-platform argPattern behavior first and treat `ask="auto"` as a separate fail-closed approval-policy design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expand subcommand entries into executable and argv patterns:** `src/infra/exec-allowlist-pattern.ts:126-131`
  - body: `git add:*` expands to `pattern: "git add"` and `argPattern: "^git add\\s+.*$"`, but allowlist matching checks the executable separately and tests `argPattern` against args after argv[0]. The documented convenience syntax will silently fail to match real commands.
  - confidence: 0.93
- **[P1] Route auto prompt decisions to a real approval:** `src/agents/bash-tools.exec-host-gateway.ts:124-125`
  - body: Mapping classifier `prompt` decisions to `on-miss` does not force an approval in `security="full"`; `requiresExecApproval` only prompts for `on-miss` on allowlist misses. Elevated commands classified as prompt-worthy can therefore execute without confirmation.
  - confidence: 0.91
- **[P1] Fail closed for unknown auto-mode commands:** `src/infra/exec-safety-analysis.ts:222-228`
  - body: Commands that miss the hard-coded regexes fall through to `level = "safe"`, and `ask="auto"` then auto-allows them. Unknown commands such as uncommon admin tools or scripts should prompt instead of being trusted by default.
  - confidence: 0.9
- **[P2] Preserve auto in every exec ask validator:** `src/infra/exec-approvals.ts:48-49`
  - body: This accepts `auto` in `normalizeExecAsk`, but the same file's persisted policy sanitization and `isExecAsk` helper still only accept `off|on-miss|always`. Loaded approval policies can strip or ignore the new mode.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR changes host exec approval behavior and introduces concrete fail-open paths in `ask="auto"`; no new third-party dependency source was found.

Concerns:

- **[high] Unknown commands auto-allow in auto mode:** `src/infra/exec-safety-analysis.ts:228`
  - body: The static analyzer defaults unrecognized commands to `safe`, and the classifier converts `safe` into `allow`, which can bypass approval for commands the heuristic does not understand.
  - confidence: 0.9
- **[high] Prompt decisions can execute under full security:** `src/agents/bash-tools.exec-host-gateway.ts:125`
  - body: Classifier `prompt` decisions are mapped to `on-miss`, but `on-miss` only prompts in allowlist mode. In `security="full"`, elevated commands can run without the intended confirmation.
  - confidence: 0.91
