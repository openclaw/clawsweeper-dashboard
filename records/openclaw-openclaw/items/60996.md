## Summary

Keep open. Current main still has the underlying all-platform subcommand allowlist gap, but this PR is not mergeable as written because the proposed subcommand expansion does not match existing argv semantics and the new auto classifier introduces fail-open exec approval paths.

## What This Changes

The PR removes the Windows-only `argPattern` guard, adds subcommand pattern expansion and docs, adds semantic/LLM command classification, and introduces an `ask="auto"` exec approval mode.

## Best Possible Solution

Ship the all-platform `argPattern` work as a narrow allowlist enhancement with explicit argv semantics, focused tests, docs, and changelog; treat `ask="auto"` and command classification as a separate fail-closed security design.

## Reproduction Assessment

Yes. The `git add:*` mismatch is reproducible by comparing the PR expansion to current `matchArgPattern`'s `argv.slice(1)` contract, and the auto prompt bypass follows from current `requiresExecApproval` semantics.

## Solution Assessment

No. This PR is too broad and unsafe as written; the safer path is to split out the subcommand allowlist change and require any classifier-based auto mode to fail closed and route prompt decisions to a real approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expand subcommand entries into executable and args:** `src/infra/exec-allowlist-pattern.ts:126-131`
  - body: `git add:*` expands into a pattern containing `git add` and an `argPattern` that also expects `git add`, but allowlist matching checks the executable separately and tests the regex against args after argv[0]. The documented syntax will silently fail to match real commands.
  - confidence: 0.93
- **[P1] Route prompt decisions to an actual approval:** `src/agents/bash-tools.exec-host-gateway.ts:124-125`
  - body: Mapping classifier `prompt` results to `on-miss` does not force an approval in `security="full"`; current approval semantics only prompt for `on-miss` in allowlist mode. Elevated commands can therefore run without the intended confirmation.
  - confidence: 0.91
- **[P1] Fail closed for unrecognized auto commands:** `src/infra/exec-safety-analysis.ts:222-228`
  - body: Commands that miss the hard-coded safety patterns fall through as `safe`, and `ask="auto"` converts `safe` into allow. Unknown scripts or admin tools should prompt instead of being trusted by default.
  - confidence: 0.9
- **[P2] Preserve auto through policy normalization:** `src/infra/exec-approvals.ts:48-49`
  - body: This accepts `auto` in `normalizeExecAsk`, but the same module's persisted policy sanitization and `isExecAsk` helper still only accept `off|on-miss|always`. Loaded approval policies can strip or ignore the new mode.
  - confidence: 0.88
- **[P2] Keep no-argv allowlist paths from regressing:** `src/infra/exec-command-resolution.ts:346`
  - body: Enabling `argPattern` globally means matcher call paths that intentionally pass no argv, such as shell positional-carrier matching, can no longer match entries with an `argPattern` on POSIX. That changes previously path-matched allowlist behavior into unexpected denials.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR changes host exec approval behavior and introduces fail-open paths in the proposed `ask="auto"` mode.

Concerns:

- **[high] Unknown commands auto-allow in auto mode:** `src/infra/exec-safety-analysis.ts:228`
  - body: The static analyzer defaults unrecognized commands to `safe`, and the classifier converts `safe` into allow, bypassing approval for commands the heuristic does not understand.
  - confidence: 0.9
- **[high] Prompt decisions can execute under full security:** `src/agents/bash-tools.exec-host-gateway.ts:125`
  - body: Classifier `prompt` decisions are mapped to `on-miss`, but `on-miss` only prompts in allowlist mode; under `security="full"`, elevated commands can run without the intended confirmation.
