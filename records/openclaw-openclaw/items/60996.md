## Summary

Current main already implements the central all-platform exec `argPattern` behavior with narrower matcher, docs, and test coverage, while the PR's extra `ask="auto"` classifier remains unsafe and should not be merged from this branch.

## What This Changes

The branch removes the platform guard for exec allowlist `argPattern`, adds subcommand-pattern helpers/docs, and introduces static/LLM command classification plus an `ask="auto"` approval mode.

## Best Possible Solution

Keep the current main implementation for argv-level allowlist matching and route any future classifier or `ask="auto"` work through a separate fail-closed security design.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR helper expansion conflicts with the current `argv.slice(1)` argPattern contract, and the proposed auto-mode prompt path conflicts with `requiresExecApproval` semantics.

## Solution Assessment

No. The current main implementation is the better narrow solution for all-platform `argPattern`; the classifier and `ask="auto"` behavior should be split into a separate product/security review if revived.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expand subcommand entries into executable and args:** `src/infra/exec-allowlist-pattern.ts:126-131`
  - body: `git add:*` expands into a pattern containing `git add` and an argPattern that also expects `git add`, but allowlist matching checks the executable separately and tests the regex against args after argv[0]. The documented syntax will silently fail to match real commands.
  - confidence: 0.93
- **[P1] Route prompt decisions to an actual approval:** `src/agents/bash-tools.exec-host-gateway.ts:124-125`
  - body: Mapping classifier `prompt` results to `on-miss` does not force an approval in `security="full"`; current approval semantics only prompt for `on-miss` in allowlist mode, so elevated commands can run without the intended confirmation.
  - confidence: 0.91
- **[P1] Fail closed for unrecognized auto commands:** `src/infra/exec-safety-analysis.ts:222-228`
  - body: Commands that miss the hard-coded safety patterns fall through as `safe`, and `ask="auto"` converts `safe` into allow. Unknown scripts or admin tools should prompt instead of being trusted by default.
  - confidence: 0.9
- **[P2] Preserve auto through policy normalization:** `src/infra/exec-approvals.ts:48-49`
  - body: The patch accepts `auto` in `normalizeExecAsk`, but persisted policy sanitization and the `isExecAsk` helper still only accept `off|on-miss|always`, so loaded approval policies can strip or ignore the new mode.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR changes host exec approval behavior and introduces fail-open paths in the proposed `ask="auto"` mode.

Concerns:

- **[high] Unknown commands auto-allow in auto mode:** `src/infra/exec-safety-analysis.ts:228`
  - body: The static analyzer defaults unrecognized commands to `safe`, and the classifier converts `safe` into allow, bypassing approval for commands the heuristic does not understand.
  - confidence: 0.9
- **[high] Prompt decisions can execute under full security:** `src/agents/bash-tools.exec-host-gateway.ts:125`
  - body: Classifier `prompt` decisions are mapped to `on-miss`, but `on-miss` only prompts in allowlist mode; under `security="full"`, elevated commands can run without the intended confirmation.
  - confidence: 0.91
