## Summary

Keep open: current main does not have ACP backend fallbacks, and the PR is a useful feature path, but the latest head still misses the registry-level `ACP_BACKEND_UNAVAILABLE` case for a configured unhealthy primary backend.

## What This Changes

The PR adds `acp.fallbacks` config and updates ACP `runTurn` to retry pre-output backend availability failures on configured fallback runtime backends with handle cleanup, tests, and a changelog entry.

## Best Possible Solution

Land only after the failover gate covers the intended pre-output unavailable-backend paths and maintainers accept the new ACP config contract.

## Reproduction Assessment

Yes for the review finding by source inspection: a named unhealthy runtime backend throws `ACP_BACKEND_UNAVAILABLE`, while the PR classifier excludes that code. Not applicable as a current-main bug because this is a new feature PR.

## Solution Assessment

No as-is; the narrow fix is to include pre-output `ACP_BACKEND_UNAVAILABLE` in backend failover and add a focused regression, while preserving the existing no-failover-after-output guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include unhealthy-backend errors in failover:** `src/acp/control-plane/manager.core.ts:772-773`
  - body: The failover gate only accepts `ACP_TURN_FAILED`/`ACP_SESSION_INIT_FAILED`, but `requireAcpRuntimeBackend` throws `ACP_BACKEND_UNAVAILABLE` when the configured backend is registered but unhealthy, before any output can occur. With `acp.backend` set to that primary and a healthy `acp.fallbacks` entry, `runTurn` records terminal failure instead of trying the fallback, so the advertised unavailable-backend failover misses this real registry path.
