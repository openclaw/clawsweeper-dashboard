## Summary

Keep open: current `main` does not already provide ACP backend fallback, and the latest PR head still has a source-reproducible cached fallback handle cleanup defect plus missing real behavior proof.

## What This Changes

The PR adds an `acp.fallbacks` config option and changes ACP `runTurn` to retry pre-output unavailable, quota, or rate-limit failures on configured fallback backends.

## Best Possible Solution

Keep the config-backed ACP failover design only after backend replacement closes stale cached handles on every mismatch path and a redacted live terminal/log/recording demonstrates real pre-output backend failover.

## Reproduction Assessment

Yes, for the PR-introduced defect by source inspection: a prior successful fallback can leave a fallback runtime cached, then the next turn forces primary metadata and clears that cached handle without `runtime.close`. I did not run tests because this review is read-only.

## Solution Assessment

No. ACP backend failover may be a reasonable feature, but this branch should close cached fallback handles on the return-to-primary path, cover that cache state in tests, and include real failover proof before merge.
