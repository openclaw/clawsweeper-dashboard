## Summary

Keep open. Current main does not implement ACP backend failover or an `acp.fallbacks` config key, and this PR is the active implementation attempt, but the proposed branch is not merge-ready because it regresses ACP turn completion semantics, derives failover candidates before resolved session metadata, can leak persistent handles while switching backends, and mixes unrelated runtime-dependency/package-resolution changes into the ACP feature.

## What This Changes

The PR adds an `acp.fallbacks` config option, changes ACP `runTurn` to try configured backend fallbacks for unavailable/rate/quota/limit failures, and also modifies install-smoke parsing plus bundled plugin runtime-dependency staging.

## Best Possible Solution

Keep this PR open but request changes. The ACP failover implementation should be rebased onto current main, narrowed to ACP config/control-plane/docs/tests, preserve existing terminal-done validation, derive backend candidates from resolved session metadata plus registry behavior, close or retire old handles before switching, avoid retries after output/tool effects, use precise retry classification, add focused manager tests and docs/changelog, split unrelated runtime-deps/install-smoke work, and correct the closing reference.

## Reproduction Assessment

Yes. The current-main absence of `acp.fallbacks` is directly reproducible with source searches, and the PR regressions can be exercised with focused ACP manager mocks: a stream that ends without `done`, a binding/agent metadata backend with no global `acp.backend`, and a persistent handle failover after partial output.

## Solution Assessment

No. The proposed branch is directionally relevant, but it is not the best way to solve the issue as-is because it bypasses existing stream-consumption guarantees, ignores current backend precedence, and mixes unrelated package-resolution changes; a narrower ACP-only patch on current main is the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore terminal done-event validation:** `src/acp/control-plane/manager.core.ts:868-888`
  - body: Current main rejects ACP streams that end without a terminal `done` event. This patch inlines stream consumption and then treats any stream with no `streamError` as successful, so a backend that silently exits without `done` will mark the turn and background task succeeded instead of failed.
  - confidence: 0.94
- **[P1] Derive backend candidates after session resolution:** `src/acp/control-plane/manager.core.ts:724-735`
  - body: The new candidate list and missing-backend guard run before `resolveSession()`, so sessions whose backend comes from binding or agent metadata, or from registry auto-selection when global `acp.backend` is unset, now throw `ACP_BACKEND_MISSING` before the valid backend can be used.
  - confidence: 0.91
- **[P1] Close the old handle before switching backends:** `src/acp/control-plane/manager.core.ts:749-750`
  - body: On fallback attempts this clears the runtime cache without closing the previous persistent handle. Once the cache entry is discarded, idle eviction and local session accounting can no longer reclaim that old backend session, so repeated failovers can leak backend handles.
  - confidence: 0.87
- **[P1] Block failover after output or tool calls:** `src/acp/control-plane/manager.core.ts:1022-1028`
  - body: The failover gate ignores whether the first backend already emitted text or tool calls. If an unavailable-looking error arrives after partial output or side effects, the caller can receive output from backend A followed by a second full attempt on backend B.
  - confidence: 0.86
- **[P2] Tighten the failover error classifier:** `src/acp/control-plane/manager.core.ts:1022-1027`
  - body: Matching `msg.includes("rate")` and `msg.includes("limit")` is too broad for deciding backend failover. Unrelated `ACP_TURN_FAILED` messages containing words such as `generated` can be misclassified and retried on another backend instead of surfacing the original error.
  - confidence: 0.82
- **[P2] Invalidate runtime-deps stamps for the new payload:** `scripts/stage-bundled-plugin-runtime-deps.mjs:921-940`
  - body: The runtime-deps change stages new `node_modules/openclaw` contents, but the branch keeps the same staging fingerprint inputs/version. Existing stamped caches can therefore be considered fresh and skip restaging, leaving upgraded installs without the newly materialized SDK payload.
  - confidence: 0.84
