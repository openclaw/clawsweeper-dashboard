## Summary

Keep open: the branch now addresses the earlier oneshot and error-chain review items, but current PR head still has a source-proven cached fallback handle leak when a later turn returns to the configured primary backend, and the thread still only shows CI/test validation rather than real ACP backend failover proof.

## What This Changes

The PR adds `acp.fallbacks` config/schema/help/changelog plumbing and rewrites ACP `runTurn` to retry unavailable, quota, or rate-limit ACP backend failures across configured fallback runtime backends.

## Best Possible Solution

Close cached fallback handles before forcing a persisted fallback session back to the configured primary backend, add a regression test that primes the cached fallback runtime, and provide redacted live ACP failover proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced defect: after a successful fallback leaves a cached fallback runtime, a later turn forces `metaWithBackend` back to the primary and `ensureRuntimeHandle` clears the mismatched cache without closing it. I did not run tests because this review is read-only.

## Solution Assessment

No. ACP backend failover may be a reasonable feature, but this branch should preserve cached runtime cleanup on the primary-return path and include real failover proof before it is the best merge path.
