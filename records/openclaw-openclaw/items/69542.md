## Summary

Keep open: current main does not implement ACP backend fallbacks, and this PR still has blocking ACP correctness issues, an unrelated plugin runtime-dependency staging change, and no contributor-provided real behavior proof.

## What This Changes

The PR adds an `acp.fallbacks` config field, wraps ACP turn execution in a backend failover loop, updates config metadata, and changes install-smoke JSON parsing plus bundled plugin runtime-dependency staging.

## Best Possible Solution

Request changes or replace this with an ACP-only failover design that preserves current stream guarantees, resolves backend candidates after session metadata precedence, closes or retires old handles safely, avoids retrying after emitted output or side effects, uses precise retry classification, and splits runtime-deps/install-smoke work into separately reviewed changes.

## Reproduction Assessment

Yes for the PR defects: source comparison shows current main rejects no-`done` streams and honors resolved ACP metadata, while the PR head bypasses those paths. I did not run tests because this review was required to remain read-only.

## Solution Assessment

No. Backend failover may be a useful direction, but this implementation is not the narrow maintainable fix because it weakens stream completion handling, computes candidates before existing precedence resolution, risks handle leaks and duplicate side effects, and mixes in unrelated runtime-deps work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore terminal done-event validation:** `src/acp/control-plane/manager.core.ts:868-888`
  - body: Current main rejects ACP streams that end without a terminal `done` event. This patch inlines stream consumption and records success whenever `streamError` is empty, so a backend that silently exits after output is marked succeeded instead of failed.
  - confidence: 0.94
- **[P1] Derive backend candidates after session resolution:** `src/acp/control-plane/manager.core.ts:724-736`
  - body: The new candidate list and missing-backend guard run before `resolveSession()`. Bound sessions whose backend comes from binding metadata, agent runtime defaults, or registry selection now throw `ACP_BACKEND_MISSING` before the valid backend can be used.
  - confidence: 0.91
- **[P1] Close the old handle before switching backends:** `src/acp/control-plane/manager.core.ts:749-750`
  - body: On fallback attempts this clears the runtime cache without closing the previous persistent handle. Once the cache entry is discarded, idle eviction and local session accounting can no longer reclaim that old backend session.
  - confidence: 0.87
- **[P1] Block failover after output or tool calls:** `src/acp/control-plane/manager.core.ts:1022-1028`
  - body: The failover gate ignores whether the first backend already emitted text or tool calls. If an unavailable-looking error arrives after partial output or side effects, callers can receive output from backend A followed by a second full attempt on backend B.
  - confidence: 0.86
- **[P2] Tighten the failover error classifier:** `src/acp/control-plane/manager.core.ts:1022-1027`
  - body: Matching `msg.includes("rate")` and `msg.includes("limit")` is too broad for backend failover decisions. Unrelated `ACP_TURN_FAILED` messages containing words such as `generated` can be retried on another backend instead of surfacing the original error.
  - confidence: 0.82
- **[P2] Split runtime-deps staging from ACP failover:** `scripts/stage-bundled-plugin-runtime-deps.mjs:921-940`
  - body: This PR materializes a host `openclaw` package under staged plugin runtime deps, changing package resolution on a plugin execution path. That may be valid separately, but it is unrelated to ACP failover and lacks cache/fingerprint proof in this branch.
  - confidence: 0.84
