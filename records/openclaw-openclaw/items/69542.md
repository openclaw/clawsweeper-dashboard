## Summary

Keep open: the feature is not on current main, and the latest PR head still has a P2 cached-runtime cleanup bug on the post-failover return-to-primary path. The submitted proof is a build/check/test run, not real ACP backend failover behavior.

## What This Changes

This PR adds `acp.fallbacks` config/schema/help/changelog support and changes ACP `runTurn` to retry configured fallback runtime backends for pre-output unavailable, quota, or rate-limit failures.

## Best Possible Solution

Keep the ACP failover design only after the primary-return cleanup path closes stale cached fallback handles and real proof shows pre-output backend failover across configured ACP backends.

## Reproduction Assessment

Yes, for the PR-introduced defect by source inspection: after a successful fallback leaves a fallback runtime cached, a later turn overrides metadata back to the configured primary and clears the cached fallback without closing it. I did not run tests because this review is read-only.

## Solution Assessment

No. ACP backend failover may be a reasonable feature, but this branch should close stale cached fallback handles on the return-to-primary path, add regression coverage for that cache state, and include real failover proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
