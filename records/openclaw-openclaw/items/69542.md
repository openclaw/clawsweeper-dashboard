## Summary

Keep open: current main does not implement ACP backend fallback, but this PR is not ready to merge because the branch regresses ACP stream completion checks, backend resolution, persistent-handle cleanup, and mixes in an unrelated runtime-deps staging change. The claimed closing issue is also unrelated to ACP failover.

## What This Changes

The PR adds an `acp.fallbacks` config field, wraps ACP `runTurn` in a backend failover loop, updates config schema/help/labels, and also changes install-smoke JSON parsing plus bundled plugin runtime-dependency staging.

## Best Possible Solution

Keep the PR open but request changes or a narrowed replacement. The maintainable path is an ACP-only failover design on current main that preserves `consumeAcpTurnStream`, derives backend candidates after session resolution, avoids retries after emitted output or tool calls, closes or safely retires old persistent handles before switching, uses precise retry classification, adds focused ACP manager tests/docs/changelog, and splits runtime-deps/install-smoke changes into separate reviewed work.

## Reproduction Assessment

Yes. A source-level reproduction is clear: current main has a test for streams ending without `done`, while the PR branch would mark the same no-error/no-done stream successful; similar focused manager mocks can cover metadata-derived backend selection and partial-output failover.

## Solution Assessment

No. The direction is relevant, but this implementation is not the best fix because it bypasses current stream guarantees, computes backend candidates before existing precedence resolution, leaks or loses old persistent handles on backend switches, and mixes unrelated package-resolution changes into the ACP feature.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore terminal done-event validation:** `src/acp/control-plane/manager.core.ts:868-888`
  - body: Current main rejects ACP streams that end without a terminal `done` event. This patch inlines stream consumption and treats any stream with no `streamError` as successful, so a backend that silently exits without `done` will mark the turn and background task succeeded instead of failed.
  - confidence: 0.94
- **[P1] Derive backend candidates after session resolution:** `src/acp/control-plane/manager.core.ts:724-736`
  - body: The new candidate list and missing-backend guard run before `resolveSession()`, so sessions whose backend comes from binding or agent metadata, or from registry auto-selection when global `acp.backend` is unset, now throw `ACP_BACKEND_MISSING` before the valid backend can be used.
  - confidence: 0.91
- **[P1] Close the old handle before switching backends:** `src/acp/control-plane/manager.core.ts:746-750`
  - body: On fallback attempts this clears the runtime cache without closing the previous persistent handle. Once the cache entry is discarded, idle eviction and local session accounting can no longer reclaim that old backend session, so repeated failovers can leak backend handles.
  - confidence: 0.87
- **[P1] Block failover after output or tool calls:** `src/acp/control-plane/manager.core.ts:1022-1028`
  - body: The failover gate ignores whether the first backend already emitted text or tool calls. If an unavailable-looking error arrives after partial output or side effects, the caller can receive output from backend A followed by a second full attempt on backend B.
  - confidence: 0.86
- **[P2] Tighten the failover error classifier:** `src/acp/control-plane/manager.core.ts:1022-1027`
  - body: Matching `msg.includes("rate")` and `msg.includes("limit")` is too broad for deciding backend failover. Unrelated `ACP_TURN_FAILED` messages containing words such as `generated` can be misclassified and retried on another backend instead of surfacing the original error.
  - confidence: 0.82
- **[P2] Invalidate runtime-deps stamps for the new payload:** `scripts/stage-bundled-plugin-runtime-deps.mjs:921-940`
  - body: The runtime-deps change stages new `node_modules/openclaw` contents, but the branch keeps the same staging fingerprint version and does not fingerprint the copied package/dist payload. Existing stamped caches can therefore be considered fresh and skip restaging, leaving upgraded installs without the newly materialized SDK payload.
  - confidence: 0.84
