## Summary

Keep open: the PR adds a plausible ACP failover feature with supplied terminal proof, but the latest diff still misses the registry-level unhealthy-backend path, so a configured primary backend that throws `ACP_BACKEND_UNAVAILABLE` will not try `acp.fallbacks`.

## What This Changes

The PR adds `acp.fallbacks` config and updates ACP turn dispatch to retry configured backup runtime backends on pre-output availability failures, with config metadata, tests, and a changelog entry.

## Best Possible Solution

Update the failover classifier and focused tests so configured fallbacks cover all pre-output backend availability failures, including `ACP_BACKEND_UNAVAILABLE`, then land only after maintainer approval for the new config behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is to register an unhealthy configured primary backend and a healthy fallback, set `acp.backend` plus `acp.fallbacks`, and call `runTurn`; the registry throws `ACP_BACKEND_UNAVAILABLE`, which the PR classifier excludes before it can try the fallback.

## Solution Assessment

No, not yet. The overall design is close, but the maintainable fix is to treat registry-level `ACP_BACKEND_UNAVAILABLE` as failover-worthy before output and add a regression test for that path.
