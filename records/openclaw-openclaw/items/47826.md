## Summary

Keep open: current main and v2026.5.20 still lack the requested lightweight GET REST endpoints for session, agent, and status queries, while the adjacent admin HTTP RPC plugin is POST-only, default-off, and not an equivalent read-only REST surface.

## What This Changes

The issue requests authenticated Gateway HTTP REST endpoints for common read-only session, agent, and status queries so web integrations can avoid slow CLI status calls or WebSocket RPC setup.

## Best Possible Solution

Choose one canonical authenticated Gateway HTTP query surface, then implement it over existing Gateway RPC/session semantics with route/auth/schema/docs/tests and a deliberate data-exposure boundary.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or combined status queries, and unmatched /api paths are not handled by the Control UI fallback.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a single authenticated HTTP query contract over existing Gateway methods rather than a second overlapping control surface.

## Risks / Open Questions

- A new read-only HTTP query surface can expose session keys, agent metadata, status details, and possibly transcript-adjacent data, so auth, scopes, origin/CORS behavior, and response-field minimization need explicit review.
- OpenClaw now also has a default-off admin HTTP RPC plugin; adding separate REST endpoints without a product decision could create overlapping Gateway query/control surfaces.
- The closed unmerged REST PR for this issue showed concrete implementation pitfalls around query validation and CORS policy that a future design should address.
