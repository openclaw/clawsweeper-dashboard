## Summary

Keep open. Current main and the latest release still lack the requested read-only session/agent REST query endpoints; the directly related implementation PR is closed unmerged, and the older open session-management issue is adjacent but explicitly does not replace this read-only query tracker.

## What This Changes

The issue requests lightweight Gateway HTTP REST endpoints for read-only session and agent queries so integrations can avoid slow `openclaw status --json` cold starts and WebSocket client setup.

## Best Possible Solution

Define a reviewed read-only HTTP query surface that dispatches to canonical `sessions.list`, `sessions.get`, and `agents.list` handlers with existing schemas, auth/scopes, tests, and public docs, possibly by extending the opt-in admin HTTP RPC surface instead of duplicating business logic.

## Reproduction Assessment

Yes, at source level. Current main has canonical Gateway RPC methods for the data but no dedicated read-only REST handlers, and the opt-in admin HTTP RPC allowlist exposes `agents.list` while omitting `sessions.list` and `sessions.get`.

## Solution Assessment

No, not as a direct set of bespoke handlers yet. The maintainable path is to reuse existing Gateway method schemas and handlers, with maintainers deciding whether the HTTP owner is admin HTTP RPC or a narrower read-only REST contract.
