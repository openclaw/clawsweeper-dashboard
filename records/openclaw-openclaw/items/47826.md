## Summary

Keep open. Current main still lacks the requested read-only Gateway REST endpoints for sessions, agents, session detail, or combined status; the only linked implementation PR is closed unmerged, and the related session-management REST tracker is adjacent rather than a replacement.

## What This Changes

Request to add authenticated Gateway REST GET endpoints for common read queries so integrations can avoid slow CLI cold-starts and WebSocket RPC setup.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over existing `agents.list`, `sessions.list`, and `sessions.get` semantics, with strict route/auth/schema tests and public docs coordinated with the session-management REST tracker.

## Reproduction Assessment

Yes. Source inspection on current main shows no route helper or request stage for the requested sessions, agents, or status REST endpoints, and unmatched paths fall through to the 404 tail.

## Solution Assessment

Unclear until maintainers approve the public API and auth shape. The maintainable direction is an authenticated REST wrapper over existing read RPC methods, not the closed PR as-is because it already raised parsing and CORS concerns.
