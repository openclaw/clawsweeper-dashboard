## Summary

Keep open. Current main still lacks the requested read-only Gateway REST query endpoints: the HTTP router registers probes, OpenAI-compatible endpoints, tools invoke, session history/kill, plugin routes, media, and Control UI, then returns 404 for unhandled paths. The related implementation PR is closed unmerged, and #20934 overlaps on session REST design but is broader and does not fully cover the requested agents/status query surface.

## What This Changes

The issue requests lightweight authenticated Gateway HTTP REST endpoints for read-only session and agent queries so integrations can avoid CLI cold-start and WebSocket client overhead.

## Best Possible Solution

Define a small authenticated read-only Gateway REST contract over the existing `agents.list`, `sessions.list`, and `sessions.get` behavior, with an optional combined status endpoint if maintainers want it. The implementation should enforce the existing operator read boundary, add route/auth/schema tests, document the public contract on the Gateway docs, and coordinate endpoint naming with #20934 before accepting code.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: current main has no registered handler for the requested `/api` read endpoints, Control UI excludes `/api` paths, and unhandled HTTP requests fall through to 404.

## Solution Assessment

Unclear until maintainers choose the public API shape. The narrow maintainable direction appears to be an authenticated read-only REST layer over existing RPC helpers and scopes, not reviving the closed PR as-is or routing command text through the OpenAI-compatible chat path.
