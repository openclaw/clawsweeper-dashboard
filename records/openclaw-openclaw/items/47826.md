## Summary

Keep open. Current main still lacks the requested lightweight read-only REST endpoints for gateway session and agent queries, while the adjacent admin HTTP RPC plugin is an opt-in POST RPC bridge and does not provide the proposed GET query surface.

## What This Changes

The issue requests authenticated Gateway HTTP REST endpoints for common read-only session and agent queries so integrations do not need the slow CLI or WebSocket JSON-RPC client.

## Best Possible Solution

Adopt one canonical authenticated HTTP query surface, either by documenting and expanding admin HTTP RPC for trusted automation or by adding a narrow read-only REST API over existing Gateway RPC semantics with route, auth, schema, and docs coverage.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or status queries, and unmatched paths fall through to 404 while Control UI intentionally avoids /api fallback handling.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable path is to wrap existing Gateway RPC semantics behind one authenticated HTTP query surface without duplicating the admin HTTP RPC plugin's role.
