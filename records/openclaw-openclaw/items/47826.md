## Summary

Keep open. Current main and the latest release still lack the requested lightweight GET REST endpoints for session and agent queries; the adjacent admin HTTP RPC plugin is POST-only, default-off, and not a direct read-only REST query surface.

## What This Changes

Request to add authenticated Gateway HTTP REST endpoints for common read-only session, agent, and status queries so integrations can avoid CLI cold-start and WebSocket setup costs.

## Best Possible Solution

Choose one canonical authenticated HTTP query surface: either expand and document admin HTTP RPC for trusted automation, or add a narrow read-only REST API over existing Gateway RPC semantics with route, auth, schema, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or status queries, and unmatched paths fall through to 404 while /api paths are excluded from Control UI fallback.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is a single authenticated HTTP query contract over existing Gateway RPC/session data rather than a second overlapping control surface.
