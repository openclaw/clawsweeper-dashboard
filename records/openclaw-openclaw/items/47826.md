## Summary

Keep open. Current main and the latest release still lack the requested read-only Gateway REST endpoints for session, agent, and status queries; the linked implementation PR is closed unmerged, and the related session-management issue is adjacent but does not fully cover this polling-oriented read API.

## What This Changes

Expose lightweight authenticated Gateway HTTP REST endpoints for common read-only session and agent queries so integrations can avoid CLI cold-start and WebSocket handshakes.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over existing `agents.list`, `sessions.list`, and `sessions.get` semantics, with route/auth/schema tests and public docs, coordinated with the broader session-management REST tracker.

## Reproduction Assessment

Yes. Source inspection on current main shows no registered REST route for the requested sessions/agents/status query endpoints, and unmatched HTTP paths fall through to the 404 tail.

## Solution Assessment

Unclear until a maintainer confirms the public API shape. The narrow maintainable direction appears to be an authenticated REST wrapper over existing read RPC methods, not reopening the closed PR as-is because it already raised input-validation and CORS policy concerns.
