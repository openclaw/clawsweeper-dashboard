## Summary

Keep open. Current main and v2026.5.12 still lack the requested read-only Gateway REST endpoints for session, agent, and combined status queries; the earlier implementation PR is closed unmerged, and the related session-management issue is adjacent rather than a full replacement for this polling-oriented API.

## What This Changes

The issue requests lightweight authenticated Gateway REST endpoints for read-only session and agent queries so integrations can avoid slow CLI startup and WebSocket JSON-RPC handshakes.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over existing `agents.list`, `sessions.list`, and `sessions.get` semantics, with route/auth/schema tests and public docs, coordinated with the broader session-management REST tracker.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.12 shows no registered REST route for the requested sessions/agents/status query endpoints, and unmatched HTTP paths fall through to the 404 tail.

## Solution Assessment

Unclear until a maintainer confirms the public API shape. The narrow maintainable direction appears to be an authenticated REST wrapper over existing read RPC methods, not reopening the closed PR as-is because it already raised input-validation and CORS policy concerns.
