## Summary

Keep open. Current main still lacks the requested read-only Gateway REST endpoints for sessions, agents, and status; the related implementation PR is closed unmerged, and the broader session-management issue does not fully cover this read-only polling use case.

## What This Changes

Request to add lightweight authenticated Gateway HTTP REST endpoints for common read-only session and agent queries so web apps can avoid slow CLI cold starts or full WS RPC clients.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over `agents.list`, `sessions.list`, and `sessions.get`, enforce the existing operator read-scope boundary, add route/auth/schema tests, document the contract, and coordinate naming with #20934.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: current main has no registered route for the requested read-only REST endpoints, Control UI excludes `/api/sessions`, and unhandled HTTP paths fall through to 404.

## Solution Assessment

Unclear until a maintainer confirms the public API shape. The narrow maintainable direction appears to be an authenticated REST wrapper over existing read RPC helpers and scopes, not command parsing on HTTP chat paths or the closed PR as-is.
