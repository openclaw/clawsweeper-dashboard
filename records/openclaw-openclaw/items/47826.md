## Summary

Keep open: current main and v2026.5.22 still lack the requested lightweight GET REST endpoints for session, agent, and status-style queries; the adjacent admin HTTP RPC plugin is POST-only, default-off, and not an equivalent read-only REST surface.

## What This Changes

Expose lightweight authenticated HTTP REST GET endpoints on the Gateway for session, agent, and status-style read-only queries so integrations can avoid CLI and WebSocket overhead.

## Best Possible Solution

Choose one canonical authenticated Gateway HTTP query surface, then implement it over existing Gateway RPC/session semantics with route, auth, schema, docs, and tests for a deliberate data-exposure boundary.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or combined status queries, and unmatched /api paths are deliberately outside the Control UI fallback.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a single authenticated HTTP query contract over existing Gateway methods rather than a second overlapping control surface.
