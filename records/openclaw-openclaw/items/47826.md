## Summary

Keep open. Current main and v2026.6.6 still lack the requested lightweight read-only REST endpoints; the later admin HTTP RPC plugin is default-off and omits `sessions.*`, so it only partially covers adjacent agents/status access, and the direct implementation PR was closed unmerged.

## What This Changes

The issue requests bearer-authenticated Gateway HTTP endpoints for common read-only session and agent queries so integrations can avoid slow CLI startup and WebSocket RPC handshakes.

## Best Possible Solution

Define a focused authenticated Gateway HTTP query surface that reuses existing `sessions.list`/`sessions.describe`/`sessions.get`, `agents.list`, and status handlers or schemas, then document the chosen route shape and security posture.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 expose the needed session and agent query logic through Gateway RPC, but not through the requested REST routes; the only adjacent HTTP RPC plugin omits `sessions.*`.

## Solution Assessment

Unclear until maintainers confirm the product/security shape. The maintainable direction is a thin authenticated adapter or an intentional `admin-http-rpc` allowlist expansion, not a standalone copy of session business logic.
