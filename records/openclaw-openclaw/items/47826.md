## Summary

Keep open: current main and v2026.5.22 still lack the requested lightweight GET REST endpoints for sessions, agents, and combined status. The default-off admin HTTP RPC plugin is adjacent but POST-only and not an equivalent read-only REST surface, so the remaining work needs maintainer product and security review.

## What This Changes

The issue requests authenticated Gateway HTTP GET endpoints for common session, agent, and status queries so integrations can avoid slow CLI cold starts and WebSocket setup.

## Best Possible Solution

Choose one canonical authenticated Gateway HTTP query surface, map it to existing Gateway RPC/session semantics, document the auth and data-exposure boundary, and cover route/auth/schema behavior with focused tests.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or combined status queries, and /api paths fall through rather than being served by the Control UI.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a single authenticated HTTP query contract over existing Gateway methods, not a second overlapping control surface with ad hoc auth/CORS behavior.
