- `impact:session-state`: The requested endpoints expose session listings and session detail, which are session-state surfaces.
- `impact:security`: Exposing Gateway session and agent metadata over HTTP requires auth, origin, and field-minimization review.

## Summary

Keep open. Current main still does not provide the requested first-class GET REST endpoints, and the newer admin HTTP RPC plugin is only an adjacent opt-in POST RPC surface that omits `sessions.list` and `sessions.get`; the linked implementation PR is closed unmerged, while the related session-management issue remains adjacent rather than a replacement.

## What This Changes

The issue requests lightweight authenticated Gateway REST endpoints for read-only session and agent queries so integrations can avoid slow CLI cold-starts or WebSocket control-plane setup.

## Best Possible Solution

Adopt one canonical HTTP query surface: either document and expand admin HTTP RPC for trusted host automation, or add a narrow authenticated read-only REST API over the existing `agents.list`, `sessions.list`, and `sessions.get` semantics with docs and route/auth/schema tests.

## Reproduction Assessment

Yes, at source level. Current main has no requested GET route helper or request stage, unmatched paths fall through to 404, and the adjacent admin HTTP RPC plugin is POST-only, opt-in, and does not allow `sessions.list` or `sessions.get`.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is an authenticated wrapper over existing Gateway RPC semantics, with admin HTTP RPC considered explicitly so OpenClaw does not grow competing HTTP control-plane surfaces.
