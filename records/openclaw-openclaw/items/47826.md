## Summary

Keep open. Current main and v2026.6.10 still only partially satisfy this through the default-off admin HTTP RPC plugin: `status` and `agents.list` can be called when enabled, but the requested HTTP session list/detail/get surface is not exposed.

## What This Changes

Gateway should expose REST HTTP endpoints for common read-only session and agent queries to avoid slow CLI cold starts or WebSocket client setup.

## Best Possible Solution

Define one approved authenticated HTTP query surface that reuses existing Gateway `status`, `agents.list`, and `sessions.*` handlers, documents the security contract, and avoids duplicating session business logic.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 expose the underlying `sessions.*` and `agents.list` Gateway RPC methods, but neither the core HTTP route table nor the admin HTTP RPC allow-list exposes the requested read-only session list/detail methods over HTTP.

## Solution Assessment

Unclear until maintainers confirm the API/security contract. The narrow maintainable shape is either an intentional admin HTTP RPC allow-list expansion for read-only session methods or a thin dedicated REST adapter over existing handlers.
