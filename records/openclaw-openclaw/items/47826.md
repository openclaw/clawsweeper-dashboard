## Summary

Keep open: current main and the latest release still lack the requested lightweight GET REST endpoints for session, agent, and status queries. The adjacent admin HTTP RPC plugin is POST-only, default-off, and not an equivalent read-only REST surface, so this still needs maintainer product and security review.

## What This Changes

Request authenticated lightweight Gateway HTTP REST endpoints for common session, agent, and status queries so web integrations can avoid slow CLI polling and WebSocket client setup.

## Best Possible Solution

Choose one canonical authenticated Gateway HTTP query surface, then implement it over existing Gateway RPC/session semantics with route/auth/schema/docs/tests and a deliberate data-exposure boundary.

## Reproduction Assessment

Yes, at source level. Current main has no registered GET REST route for sessions, agents, or combined status queries, and unmatched /api paths are not handled by the Control UI fallback.

## Solution Assessment

Unclear until maintainers choose the public API and security boundary. The maintainable direction is a single authenticated HTTP query contract over existing Gateway methods rather than a second overlapping control surface.
