## Summary

Keep open. Current main and v2026.6.10 still only partially cover this through the default-off admin HTTP RPC plugin: `status` and `agents.list` can be called when that plugin is enabled, but the requested read-only session list/detail/get HTTP surface is not exposed.

## What This Changes

The issue requests lightweight authenticated HTTP endpoints on the gateway for common read-only session and agent queries so integrations can avoid slow CLI cold starts and WebSocket client complexity.

## Best Possible Solution

Define one approved authenticated HTTP query surface that reuses existing Gateway `status`, `agents.list`, and `sessions.*` handlers, documents the security contract, and avoids duplicating session business logic.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 expose the underlying Gateway session and agent RPC methods, but the HTTP route table and admin HTTP RPC allow-list do not expose the full requested read-only session query surface.

## Solution Assessment

Unclear until maintainers confirm the API/security contract. The narrow maintainable shape is either an intentional admin HTTP RPC allow-list expansion for read-only session methods or a thin dedicated REST adapter over the existing handlers.
