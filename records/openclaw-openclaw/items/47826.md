## Summary

Keep open. Current main still lacks the requested read-only Gateway REST query endpoints; the existing RPC handlers prove the data path exists, but exposing it over HTTP is a new public API that needs maintainer product/security approval.

## What This Changes

Expose lightweight authenticated read-only Gateway HTTP REST endpoints for sessions, agents, session details, and health/status queries so web integrations can avoid CLI or WebSocket overhead.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over the existing read-scoped RPC helpers, document it, and coordinate naming with the broader session-management API discussion.

## Reproduction Assessment

Yes. Source inspection on current main shows no `/api/v1/sessions`, `/api/v1/agents`, `/api/v1/status`, or `/api/v1/sessions/:key` route in the HTTP pipeline, and unhandled paths fall through to 404.

## Solution Assessment

Unclear until maintainers approve the public API shape. The narrow maintainable direction appears to be an authenticated REST wrapper over existing read-scoped RPC handlers, not command parsing on chat endpoints or adopting the closed PR wholesale.
