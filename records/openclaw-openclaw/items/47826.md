## Summary

Keep open. Current main and v2026.6.8 only partially address the request through the default-off admin HTTP RPC plugin: `status` and `agents.list` are reachable over HTTP when enabled, but `sessions.list`, `sessions.describe`, and `sessions.get` remain absent from the HTTP allow-list and there are no dedicated read-only session REST routes.

## What This Changes

The issue requests lightweight gateway HTTP endpoints for read-only session and agent queries so integrations can avoid slow CLI startup or WebSocket control-plane setup.

## Best Possible Solution

Keep this issue open and define an approved authenticated HTTP query surface that reuses existing gateway session/agent/status handlers without duplicating session business logic.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 expose the needed session query logic through Gateway RPC, but the shipped HTTP surface only partially overlaps through default-off admin HTTP RPC and omits `sessions.*`.

## Solution Assessment

Unclear until maintainers confirm the product/security contract. The narrowest maintainable direction appears to be either an intentional `admin-http-rpc` allow-list expansion for read-only session methods or a thin dedicated REST adapter over the same existing handlers.
