## Summary

Keep open. Current main still does not expose the requested read-only Gateway REST endpoints for sessions, agents, or combined status; the older implementation PR is closed unmerged, and the related session-management tracker is adjacent rather than a replacement for this polling-oriented API.

## What This Changes

The issue requests lightweight authenticated Gateway REST endpoints for common read-only queries so web apps can poll sessions and agents without the CLI or a WebSocket client.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over existing `agents.list`, `sessions.list`, and `sessions.get` semantics, with strict route/auth/schema tests and public docs coordinated with the session-management REST tracker.

## Reproduction Assessment

Yes. Source inspection on current main shows no route helper or request stage for the requested sessions, agents, or status REST endpoints, and unmatched paths fall through to the 404 tail.

## Solution Assessment

Unclear until maintainers approve the public API/auth shape. The narrow maintainable direction is an authenticated REST wrapper over existing read RPC methods, not the closed PR as-is because it already raised parsing and CORS concerns.
