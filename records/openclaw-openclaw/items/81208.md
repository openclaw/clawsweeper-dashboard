## Summary

Keep open: the PR targets a real Bedrock Mantle log-noise bug, but the proposed env-marker gate would also disable valid AWS default credential-chain discovery, and the supplied proof is test-only rather than a real OpenClaw runtime run.

## What This Changes

The PR adds an amazon-bedrock-mantle preflight that skips implicit discovery unless discovery is explicitly enabled, an explicit bearer token is set, or AWS env credentials are detected, plus regression tests and a changelog entry.

## Best Possible Solution

Keep the explicit discovery.enabled=false escape hatch, but suppress repeated no-credentials log noise without blocking AWS default credential-chain discovery.

## Reproduction Assessment

Yes. Source inspection plus dependency source shows the branch returns null before @aws/bedrock-token-generator can use fromNodeProviderChain when env markers are absent, even though current docs and code support default-chain credential sources.

## Solution Assessment

No. The env-marker preflight is broader than the bug because it treats missing AWS env markers as missing AWS credentials; a narrower cache, throttle, or explicit-disabled gate would preserve documented default-chain discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve AWS default-chain Mantle discovery:** `extensions/amazon-bedrock-mantle/discovery.ts:352`
  - body: This gate returns null whenever the env lacks the small set of markers recognized by resolveAwsSdkEnvVarName, but Mantle discovery is documented and implemented to mint bearer tokens through the AWS default credential chain. Valid shared-config, SSO, web-identity, instance-role, and task-role setups can work without those markers, so this would silently remove the implicit provider unless users force discovery.enabled=true.
