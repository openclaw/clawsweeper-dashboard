## What This Changes

The issue requests restoring no-real-key local Ollama behavior for OpenClaw running in an Orb VM against an Ollama server on the host.

## Best Possible Solution

Add a narrow local/self-hosted Ollama host classification for the Orb VM host alias, with regression tests proving it accepts `host.orb.internal` while public remote hosts still require real credentials.

## Reproduction Assessment

Yes, from source inspection: with `baseUrl: "http://host.orb.internal:11434"` and no real env/profile key, both the Ollama synthetic-auth predicate and the core local-marker predicate leave the provider without usable auth. I did not run a live Orb VM reproduction in this read-only review.

## Solution Assessment

No, the current workaround of adding agent-level auth does not restore the documented local no-real-key behavior. The maintainable fix is a narrow local-host policy update plus tests, not a broad relaxation for all dotted hostnames.
