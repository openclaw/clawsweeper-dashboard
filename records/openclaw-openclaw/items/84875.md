## Summary

Keep open: current main still treats `host.orb.internal` as a remote-looking host for both the core custom-provider marker path and the Ollama plugin synthetic-auth path, so the reported missing-key failure remains source-reproducible.

## What This Changes

The issue reports that local Ollama fails after upgrade in an Orb VM because `host.orb.internal` is reachable but no longer accepted as a no-auth local Ollama endpoint.

## Best Possible Solution

Add a narrow, tested local-host policy for the Orb host alias, and verify whether the setup `/api/tags` failure is part of the same path, while keeping public remote Ollama hosts credential-protected.

## Reproduction Assessment

Yes from source inspection: with `baseUrl: "http://host.orb.internal:11434"` and only the `ollama-local` marker, both local-auth predicates leave the provider without usable auth. I did not run a live Orb VM reproduction in this read-only review.

## Solution Assessment

No, current main is not the best end state because it excludes the reported VM host alias from local Ollama auth while the documented local/LAN behavior does not require a real bearer token. The safer solution is a narrow alias-aware local-host policy with regression tests, not a broad relaxation for arbitrary public remotes.
