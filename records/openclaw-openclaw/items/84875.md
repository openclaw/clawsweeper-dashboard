## Summary

Keep open: current main still treats `host.orb.internal` as remote for local Ollama auth, so the reported `ollama-local` provider config can still fall through to `No API key found` after upgrade. The related prior fixes covered loopback/private-LAN custom providers, but not this VM host alias.

## What This Changes

The issue reports a 2026.5.19 regression where OpenClaw running inside an Orb VM cannot use a host Ollama server at `http://host.orb.internal:11434` without a real API key.

## Best Possible Solution

Add narrow local/self-hosted Ollama recognition for known VM host aliases such as `host.orb.internal`, keep public remote hosts credentialed, and cover both provider-level marker auth and plugin synthetic auth with regression tests.

## Reproduction Assessment

Yes from source inspection: `host.orb.internal` is parsed as a dotted hostname that fails both current local-auth predicates, so `apiKey: "ollama-local"` is not accepted as local provider auth. I did not run a live Orb VM reproduction in this read-only review.

## Solution Assessment

No, current main is not the best end state because it excludes a common VM host alias from the local Ollama no-auth path. The maintainable repair is a narrow alias/local-host policy fix with tests, not a broad relaxation for all dotted hostnames.
