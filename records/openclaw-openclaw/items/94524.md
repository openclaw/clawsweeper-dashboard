- **[P1] Classify plugin producers before defaulting to actionable:** `src/auto-reply/reply/session-system-events.ts:143`
  - body: The new partition treats every event without `quarantineInPrompt` as actionable, but bundled plugin producers such as Slack subtype events and Telegram/Discord reactions still enqueue attacker-reachable event text without the flag. That leaves those events draining as `System:` prompt content after this PR; classify all such producers or make untrusted the safe default before merge.
  - confidence: 0.9
- **[P1] Settle the plugin-facing quarantine contract:** `src/infra/system-events.ts:47`
  - body: `enqueueSystemEvent` is exposed through the plugin runtime and SDK runtime subpath, so adding `quarantineInPrompt` creates a plugin-visible prompt-trust option. Without documented semantics, version-skew behavior, and compatibility tests, external plugins cannot know which events must be quarantined or what older hosts will do with the option.
  - confidence: 0.87
- **[P2] Include prompt trust in duplicate identity:** `src/infra/system-events.ts:135`
  - body: The PR stores a new prompt-trust bit on queued events, but duplicate detection still compares only text, context key, and delivery context. If an actionable and quarantined event collide on those fields, one can be dropped under the other trust classification; include `quarantineInPrompt` in the duplicate/equality path or prove such mixed-trust collisions cannot occur.
  - confidence: 0.78
