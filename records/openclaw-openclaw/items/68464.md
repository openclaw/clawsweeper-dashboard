## Summary

Keep this PR open, but it is not mergeable as-is: the hardening direction is credible, yet PR-head source inspection still shows two P2 correctness blockers, missing real behavior proof, and overlap with active subagent completion-delivery work.

## What This Changes

The PR adds persisted subagent completion delivery claims and stored user-facing payloads, switches completion announces toward queue-first routing, and hardens iMessage outbound sanitization/chunking.

## Best Possible Solution

Consolidate the hardening around one durable completion-delivery design, fix claim cleanup and iMessage normalization boundaries, align docs, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes for the PR-head defects: source inspection shows the persisted claim can be created before an already-aborted return that bypasses cleanup, and chunked iMessage text can be normalized again per send. The original intermittent production reports remain credible discussion evidence but were not live-reproduced here.

## Solution Assessment

No. The direction is appropriate, but this patch is not the best mergeable solution until the claim-finalization bug, iMessage normalization boundary, docs mismatch, rebase conflict, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Release the claim on early abort:** `src/agents/subagent-announce-dispatch.ts:200-204`
  - body: When announceId is present, beginPersistedAnnounceDeliveryClaim can create a fresh claim before this already-aborted return. Because this return occurs before the finally block, the claim remains in flight until lease expiry and later retries can be rejected as delivery-already-in-flight.
  - confidence: 0.92
- **[P2] Avoid normalizing each delivered iMessage chunk:** `extensions/imessage/src/send.ts:146`
  - body: Monitor delivery normalizes the full reply before splitting it into chunks, but this send path normalizes each chunk again. A long fenced code block split across chunks can get extra closing fences appended to intermediate or final chunks, corrupting the delivered formatting.
  - confidence: 0.87
- **[P3] Align docs with queue-first completion delivery:** `src/agents/subagent-announce-dispatch.ts:213-214`
  - body: This changes completion dispatch to queue-first, but the public subagent docs still describe direct agent delivery first with queue fallback afterward. Update the docs so released behavior matches the new delivery order.
  - confidence: 0.84
