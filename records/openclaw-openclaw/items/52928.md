## Summary

Keep open. Current main still does not provide the requested additive `agents.setDefault` Gateway RPC: the agent handlers, advertised method list, ADMIN scope table, protocol schema/types/validators, SDK client namespace, and public Gateway docs expose agent CRUD and file helpers but no default-agent mutation method. The linked #59070 Control UI fix confirms the default storage shape and config-edit workaround, but it does not satisfy external Gateway clients that need a dedicated RPC.

## What This Changes

Request an ADMIN-scoped `agents.setDefault` Gateway RPC that marks one configured agent as default, clears other defaults, rejects unknown agent IDs, and returns the new `defaultId`.

## Best Possible Solution

Add a narrow, additive core Gateway method that reuses `agents.list[].default`, validates that the target agent is configured, persists exactly one default entry, returns the resolved `defaultId`, and updates protocol schemas, types, validators, SDK, docs, tests, and changelog.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static inspection gives a high-confidence verification path: search the Gateway handler map, method list, ADMIN scope table, protocol schemas/types/validators, SDK namespace, docs, and tests for `agents.setDefault`, and current main has no such surface.

## Solution Assessment

Yes, the requested ADMIN-scoped RPC is the narrowest maintainable solution for external desktop/web clients that need a simple default-agent action. Raw `config.patch` can remain available for bulk edits, but it should not be the only client-facing API for this common mutation.
