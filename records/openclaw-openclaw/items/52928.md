## Summary

Keep open. Current main and v2026.5.22 still lack a dedicated `agents.setDefault` Gateway RPC; the existing UI path only stages `agents.list[].default` through the config form, so SDK/desktop clients still need raw config mutation for this action.

## What This Changes

Request adds an ADMIN-scoped Gateway RPC, `agents.setDefault`, to change the default configured agent without using raw `config.patch`.

## Best Possible Solution

Add `agents.setDefault` as an additive `operator.admin` Gateway RPC that validates a configured agent id, updates the canonical `agents.list[].default` flags, returns the new `defaultId`, and keeps protocol, SDK, docs, and focused tests in sync.

## Reproduction Assessment

Yes. Source inspection on current main shows no `agents.setDefault` handler, descriptor, protocol schema, or SDK helper, while `agents.update` is a closed schema that does not accept `default`.

## Solution Assessment

Yes. The requested additive RPC is the narrow maintainable path because it reuses the existing `agents.list[].default` storage contract instead of adding `agents.defaultId` or forcing every client to build raw config patches.
