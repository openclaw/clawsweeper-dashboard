## Summary

Keep open. Current main still does not implement the requested dedicated Gateway RPC: agent CRUD and file helpers exist, and the Control UI now has a config-edit workaround for default selection, but there is no `agents.setDefault` handler, advertised method, ADMIN scope entry, protocol schema/type/validator, SDK helper, public docs entry, or targeted test coverage.

## What This Changes

Request to add an ADMIN-scoped `agents.setDefault` Gateway RPC that sets exactly one configured agent as the default without requiring clients to use raw `config.patch`.

## Best Possible Solution

Add a small additive core Gateway method that reuses the existing `agents.list[].default` config representation, rejects unknown configured agents, persists exactly one default entry, returns `{ ok: true, defaultId }`, and keeps `config.patch` as the lower-level bulk-edit escape hatch.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. Static inspection gives a high-confidence verification path: current main lacks `agents.setDefault` across the handler map, advertised methods, scopes, protocol schemas/types/validators, SDK helper, docs, and tests.

## Solution Assessment

Yes, the requested additive ADMIN-scoped Gateway method is the narrowest maintainable solution for desktop/web clients that should not have to construct raw config merge patches. The current Control UI config helper is useful precedent for storage semantics, but it does not solve the external-client RPC gap.
