## Summary

Keep open. Current main and the latest release still lack a dedicated `agents.setDefault` Gateway RPC; the related Control UI bug only fixed the invalid `agents.defaultId` write path and explicitly left this RPC request open.

## What This Changes

The issue requests an additive Gateway RPC method for setting the default agent without requiring clients to construct a raw `config.patch`.

## Best Possible Solution

Add `agents.setDefault` as an additive `operator.admin` Gateway RPC that validates a configured agent id, updates the canonical `agents.list[].default` flags, returns the new `defaultId`, and keeps schema, SDK, docs, and focused tests in sync.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main exposes the existing agents CRUD methods and defaultId read model, but no `agents.setDefault` handler, schema, registry entry, or SDK helper. I did not need a live gateway run because the missing RPC surface is visible in the protocol and handler source.

## Solution Assessment

Yes, the requested additive RPC is the narrow maintainable solution. It should reuse the existing canonical `agents.list[].default` storage contract instead of adding `agents.defaultId` or teaching every client to build raw config patches.
