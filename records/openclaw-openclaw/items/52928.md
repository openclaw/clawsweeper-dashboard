## What This Changes

The issue requests a dedicated Gateway `agents.setDefault` ADMIN RPC that sets one configured agent as default without requiring clients to patch raw config.

## Best Possible Solution

Add an additive `operator.admin` RPC that validates a configured agent id, atomically rewrites the canonical `agents.list[].default` flags, returns the new `defaultId`, and updates protocol, SDK, docs, and focused tests together.

## Reproduction Assessment

Yes, source-reproducible. Current main has no `agents.setDefault` handler, descriptor, protocol schema, docs entry, or SDK helper, while `agents.update` is a closed schema that does not accept a default flag.

## Solution Assessment

Yes. The requested additive RPC is the narrow maintainable path because it keeps clients away from raw `config.patch` while reusing the canonical `agents.list[].default` storage contract.
