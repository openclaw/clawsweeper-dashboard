## Summary

Keep open. Current main and v2026.5.19 still lack a supported Telegram parse-mode config, while the Telegram plugin continues to document and implement HTML parse mode with plain-text fallback only after HTML parsing fails.

## What This Changes

The issue requests a Telegram parseMode setting with channel-level and optional per-account overrides.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default, defines per-account inheritance deliberately, and wires the selected mode through sends, captions, edits/previews, delivery helpers, docs, generated metadata, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, docs, release inspection, and live issue context give high confidence that current main still lacks a supported Telegram parse-mode config and assumes HTML formatting.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable solution is a deliberate Telegram plugin config/API addition that preserves current HTML behavior unless maintainers choose a documented migration or opt-in alternative.
