## Summary

Keep open. Current main still lacks a supported Telegram parse-mode config and continues to hardcode HTML across text, caption, edit, preview, and delivery paths; the related #10758 and #16121 threads are narrower custom-emoji/raw-formatting requests, not canonical replacements.

## What This Changes

Request to add a Telegram parse-mode config, including a channel default and optional per-account override, so outbound Telegram messages can use plain text, HTML, or Markdown-style rendering instead of a hardcoded mode.

## Best Possible Solution

Add a typed Telegram plugin config surface that preserves current HTML behavior by default unless maintainers intentionally change it, supports a safe no-parse path, and wires the setting through sends, captions, edits, previews, and delivery helpers with schema/types/metadata/docs/tests updated together.

## Reproduction Assessment

Yes. The missing feature is source-reproducible: the strict Telegram config schema/types omit parseMode and current outbound paths still set parse_mode: "HTML"; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the exact supported values, default, and escaping semantics. The narrow maintainable direction is a bundled Telegram plugin config option rather than a core-wide change, with raw HTML/MarkdownV2/custom emoji behavior treated as explicit design choices or follow-ups.
