## Summary

Keep open. Current main and the latest release still lack a supported Telegram parse-mode config, while schema/types/docs and the main Telegram delivery paths continue to assume HTML formatting; related custom-emoji and formatter issues are adjacent, not a replacement for this broader config decision.

## What This Changes

Request to add a Telegram parseMode configuration option, including possible top-level and per-account overrides, so Telegram output can use plain text, Markdown, or HTML instead of a hardcoded mode.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default unless maintainers deliberately choose a migration, defines account inheritance, and wires all delivery/preview paths with docs and tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, docs, release inspection, and live issue context give high confidence that current main still lacks a supported Telegram parse-mode config.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable path is a deliberate Telegram plugin config addition with explicit defaults, inheritance semantics, security boundaries, and upgrade proof.
