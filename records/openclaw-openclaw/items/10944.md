## Summary

Keep open. Current main still lacks a supported Telegram parse-mode config at the channel or account level, while the documented and tested Telegram delivery paths continue to use HTML formatting with fallback behavior; related custom-emoji, spacing, and reasoning-tag issues are narrower follow-ups rather than replacements for this broader config request.

## What This Changes

Add an optional Telegram `parseMode` configuration option, potentially with per-account override, so outbound Telegram messages are not forced through one hardcoded parse mode.

## Best Possible Solution

Add a Telegram-owned typed parse-mode setting with documented channel/account precedence, wire the resolved mode through sends, captions, edits, draft previews, and delivery helpers, then update schema, exported types, generated metadata/help, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection on current main gives high confidence that Telegram config has no `parseMode` field and delivery paths still send rendered output as HTML.

## Solution Assessment

Yes in direction, but not as an automatic repair. The narrow maintainable path is a Telegram plugin config addition after maintainers choose supported modes, defaults/backcompat behavior, account precedence, and escaping policy.
