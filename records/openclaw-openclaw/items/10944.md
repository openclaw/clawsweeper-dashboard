## Summary

Keep open. Current main and v2026.5.20 still lack a supported Telegram parse-mode config, while the Telegram send, caption, draft-preview, and lower-level delivery paths continue to assume HTML formatting with fallback only after formatting or delivery failures.

## What This Changes

The issue requests a Telegram parseMode configuration, including possible top-level and per-account overrides, so users can choose plain text, Markdown, or HTML behavior.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default unless maintainers choose a documented migration, defines account inheritance deliberately, and wires the selected mode through sends, captions, edits/previews, delivery helpers, generated metadata, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, docs, live issue context, related-item search, and shipped-release inspection give high confidence that current main still lacks a supported Telegram parse-mode config and assumes HTML formatting.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable solution is a deliberate Telegram plugin config addition with explicit defaults, inheritance semantics, security boundaries, and upgrade proof.
