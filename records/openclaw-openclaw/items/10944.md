## Summary

Keep open. Current `main` and `v2026.6.10` still do not expose the requested Telegram `parseMode` setting; the existing `richMessages` toggle solves an adjacent rich-message compatibility problem, but not the broader parse-mode/text-mode selection request.

## What This Changes

The issue requests a configurable Telegram parse-mode setting at channel and account scope so users can choose plain text, Markdown-style formatting, or HTML behavior.

## Best Possible Solution

Add a maintainer-approved Telegram formatting policy in the bundled Telegram plugin, likely as a typed parse/text mode config wired through send, edit, media-caption, draft-preview, outbound-adapter, docs, metadata, and tests, with `richMessages` interaction and escaping/security semantics explicit.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `channels.telegram.parseMode` and per-account `parseMode` are still absent from the strict Telegram config schema/types, while delivery paths still choose HTML or `richMessages` internally. I did not run a live Telegram rendering repro because the remaining request is a config/product surface, not a simple runtime crash.

## Solution Assessment

Unclear whether the exact proposed `none | Markdown | HTML` shape is the best permanent API. The safer solution is a maintainer-approved Telegram formatting policy that preserves current HTML defaults, keeps rich messages explicit, and only exposes modes whose escaping and client behavior are well defined.
