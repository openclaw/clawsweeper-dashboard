## Summary

Current main still has no supported top-level or per-account Telegram parse-mode setting: the documented and tested default remains Telegram-safe HTML with plain-text retry on parse failure. The request is still necessary, but the supported mode set and account-level granularity require maintainer product/security judgment, so it should remain open rather than enter an autonomous fix lane.

## What This Changes

Add a typed Telegram parse-mode setting, potentially with per-account overrides, so operators can choose HTML or unparsed/plain outbound messages instead of always using the fixed HTML pipeline.

## Best Possible Solution

Add one plugin-owned, typed formatting policy only after maintainers approve its narrow mode set and precedence; resolve it once per Telegram account and propagate it through every outbound surface, preserving HTML as the compatibility default and treating raw HTML/custom emoji as separate explicit work.

## Reproduction Assessment

Yes, at high confidence from current-main source and schema: the strict Telegram config exposes no parse-mode setting and outbound delivery continues to select HTML. No live Telegram run is needed to establish that the requested configuration capability is absent.

## Solution Assessment

Unclear as proposed: a centralized plugin-owned policy is the right boundary, but maintainers must first choose the supported modes, compatibility default, per-account precedence, and whether MarkdownV2 or raw HTML belongs in this issue.
