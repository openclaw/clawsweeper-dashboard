## Summary

Keep open. Current main and v2026.6.5 still expose no persistent Telegram parse-mode setting, while documented outbound behavior remains Telegram-safe HTML with plain-text fallback; maintainers must first define safe modes, defaults, inheritance, and cross-path semantics.

## What This Changes

Add configurable Telegram outbound formatting modes, potentially with per-account overrides, instead of always rendering messages through the HTML path.

## Best Possible Solution

Keep HTML as the compatibility default and define a plugin-owned typed formatting policy—starting with a safe plain-text opt-out if sufficient—then propagate it consistently through every Telegram send, caption, edit, preview, and fallback path.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.5 have no persistent Telegram parse-mode configuration and document a fixed HTML renderer with plain-text fallback. A live Telegram run was not needed to establish that the requested capability is absent.

## Solution Assessment

No, not exactly as proposed. A generic string selector would blur input rendering, Telegram API mode, escaping, and trust boundaries; maintainers should first define typed safe modes and one consistent implementation across all outbound surfaces.
