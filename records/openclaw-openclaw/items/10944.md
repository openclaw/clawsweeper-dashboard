## Summary

Keep open. Current main still does not expose a Telegram parseMode config at the channel or account level; the newer richMessages setting and open fallback PR are adjacent rendering/delivery work, not the requested plain/HTML/Markdown parse-mode contract.

## What This Changes

The issue requests configurable Telegram parse mode at the channel and possibly account level so users can choose plain text, HTML, or Markdown-style rendering instead of an internal hardcoded format.

## Best Possible Solution

Define one maintainer-approved Telegram formatting contract, then add typed Telegram plugin config, docs, metadata, and tests that preserve current defaults while wiring the chosen behavior through sends, captions, edits, previews, and durable delivery.

## Reproduction Assessment

Yes, source-level: current main has no Telegram parseMode key in the strict schema, exported config type, public Telegram docs, or user-facing config metadata. The exact client-rendering complaints still need live Telegram proof per chosen mode.

## Solution Assessment

Unclear as proposed: a parseMode knob is plausible, but the exact supported modes, default, per-account scope, and interaction with richMessages need maintainer direction. The maintainable solution is one documented formatting contract wired across every Telegram delivery path.
