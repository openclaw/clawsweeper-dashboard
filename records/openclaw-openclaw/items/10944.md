## Summary

Keep open. Current main and v2026.6.5 still provide no supported Telegram parse-mode control and consistently use HTML formatting with plain-text fallback; the remaining blocker is a maintainer decision about safe modes, defaults, inheritance, and whether narrower formatter fixes cover the motivating cases.

## What This Changes

The issue requests configurable Telegram output parse mode globally and optionally per account.

## Best Possible Solution

Maintainers should define the minimum safe Telegram output policy first—likely retaining HTML as the compatibility default and considering a plain/no-parse option—then route one resolved mode through the channel-owned formatter, all outbound surfaces, strict schema and types, generated metadata, public docs, and focused unit plus live Telegram proof.

## Reproduction Assessment

Yes for the capability gap: current-main and release schema, types, docs, and delivery paths consistently show no supported parse-mode selection. The reported emoji and spacing symptoms still need live Telegram proof before they determine the chosen modes.

## Solution Assessment

No, the proposed three-mode global-plus-account configuration is not yet proven to be the best shape; maintainers should first decide the minimum supported modes, safe default, inheritance contract, and whether narrower formatter fixes cover the motivating cases.
