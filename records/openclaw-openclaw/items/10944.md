## Summary

Current main still hardcodes Telegram-safe HTML across outbound delivery and exposes no supported top-level or per-account parse-mode setting. The request remains useful, but the supported modes and escaping contract need maintainer product judgment; the narrower custom-emoji and spacing reports do not supersede it.

## What This Changes

Add a configurable Telegram output parse mode, potentially with per-account overrides.

## Best Possible Solution

Keep rendered HTML as the safe default and, if maintainers approve a new setting, expose only explicitly supported modes through one Telegram-owned renderer contract used consistently by sends, captions, edits, retries, and previews.

## Reproduction Assessment

Yes for the missing capability: current strict schemas, documentation, and outbound code provide a high-confidence source reproduction showing no selectable parse mode. The claimed emoji and spacing symptoms are not independently established as current-main failures that specifically require changing parse mode.

## Solution Assessment

Unclear. A plain/no-parse option may be a reasonable narrow addition, but a general HTML/MarkdownV2 switch is not yet proven to be the safest or most maintainable answer to the cited formatting problems.
