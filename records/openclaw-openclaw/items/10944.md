## Summary

Keep open. Current main and the latest shipped release still do not expose a supported Telegram parse-mode config, while the Telegram plugin continues to assume HTML formatting across text sends, media captions, draft previews, and lower-level delivery helpers; related custom-emoji, spacing, and markdown-table issues are narrower and do not replace this config request.

## What This Changes

The issue requests a Telegram `parseMode` configuration surface, optionally per account, to choose plain text, Markdown, or HTML behavior instead of a hardcoded Telegram parse mode.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default, deliberately defines account inheritance, and wires the selected mode through sends, captions, edits/previews, delivery helpers, docs, generated metadata, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, docs, release inspection, related-item context, and live issue state give high confidence that current main still lacks a supported Telegram parse-mode config and assumes HTML formatting.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable solution is a deliberate Telegram plugin config/API addition that preserves current HTML behavior unless maintainers choose a documented migration or opt-in alternative.
