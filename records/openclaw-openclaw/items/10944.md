## Summary

Keep open. Current main still does not expose a supported Telegram `parseMode` setting at channel or account level, and the main send, caption, edit, draft-preview, and outbound-formatting paths still resolve Telegram delivery to HTML with fallback behavior rather than a user-selected parse mode. Related custom-emoji, literal-tag, and formatter-spacing reports are useful context, but none is a canonical replacement for this broader config request.

## What This Changes

Request adds optional Telegram channel/account `parseMode` configuration so deployments can choose plain text, Markdown-style formatting, or HTML instead of always using the current HTML delivery behavior.

## Best Possible Solution

Add a Telegram-owned typed parse-mode setting with documented precedence, preserve current HTML behavior by default unless maintainers choose a migration, and wire the resolved mode consistently through delivery, schema/types, metadata, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection on current main gives high confidence that no supported channel/account `parseMode` config exists and that Telegram delivery still uses HTML parse mode in the main paths.

## Solution Assessment

Yes in direction, but not as an automatic repair. The maintainable path is a Telegram plugin config addition after maintainers choose supported modes, defaults, account precedence, and escaping/trust policy.
