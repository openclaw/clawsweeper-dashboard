## Summary

Keep open. Current main and v2026.5.20 still lack a supported Telegram parse-mode config, while Telegram text, captions, delivery helpers, previews, docs, and metadata continue to assume HTML parse mode with plain-text fallback only after HTML handling fails. The linked custom-emoji and spacing issues are useful context but are narrower than this broader config request.

## What This Changes

Request a Telegram `parseMode` config, including possible per-account overrides, so operators can choose formatting behavior instead of hardcoded Telegram HTML handling.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default, deliberately defines account inheritance, and wires the selected mode through sends, captions, edits/previews, delivery helpers, docs, generated metadata, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, docs, release inspection, and live issue context give high confidence that current main still lacks a supported Telegram parse-mode config and assumes HTML formatting.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable solution is a deliberate Telegram plugin config/API addition that preserves current HTML behavior unless maintainers choose a documented migration or opt-in alternative.
