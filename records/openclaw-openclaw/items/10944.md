## Summary

Keep open. Current main and the latest shipped release still lack a supported Telegram parse-mode config, and the relevant Telegram send, caption, draft-preview, and lower-level delivery paths still assume HTML formatting with plain-text fallback only after HTML formatting failures.

## What This Changes

The issue requests a Telegram channel/account `parseMode` config so users can choose plain text, Markdown-style, HTML, or similar Telegram formatting behavior instead of the current hardcoded mode.

## Best Possible Solution

Add a typed Telegram plugin parse-mode config that preserves current HTML behavior by default unless maintainers choose a documented migration, defines account inheritance deliberately, and wires the selected mode through sends, captions, edits/previews, delivery helpers, generated metadata, docs, and focused tests.

Not applicable as a bug reproduction because this is a feature/config request. Source, tests, docs, shipped-release inspection, and related-item search give high confidence that current main still lacks a supported Telegram parse-mode config and assumes HTML formatting.

## Solution Assessment

Yes directionally, but not as an automatic narrow bug fix. The maintainable solution is a deliberate Telegram plugin config/API addition with explicit defaults, inheritance semantics, security boundaries, and upgrade proof.
