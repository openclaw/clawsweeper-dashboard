## Summary

Keep open. Current main and v2026.6.5 still provide no supported Telegram parse-mode setting, and no viable related issue or PR supersedes the request; maintainers must define a compatibility-safe formatting contract before implementation.

## What This Changes

Add configurable Telegram outbound parsing at channel and optionally per-account scope instead of always selecting Telegram-safe HTML.

## Best Possible Solution

Preserve HTML as the compatibility default, define one account-resolved formatting policy, add plain/unparsed output first if approved, and support MarkdownV2 only through a dedicated renderer with consistent handling across messages, captions, edits, previews, and fallbacks.

## Reproduction Assessment

Yes for the missing capability: current-main schema, docs, tests, and outbound callers directly show fixed HTML behavior with no selectable parse mode. Live Telegram validation is still required to prove the visual benefit and escaping behavior of each alternative mode.

## Solution Assessment

No, the original proposal is not yet the best contract because it assumes Markdown is current behavior, does not distinguish MarkdownV2, and leaves inheritance, compatibility defaults, fallbacks, and non-message surfaces undefined. A compatibility-preserving account-resolved policy is the safer design.

## Risks / Open Questions

- The proposed values are not a drop-in implementation contract: current behavior is HTML, while Telegram MarkdownV2 needs a dedicated renderer and substantially different escaping.

- Raw HTML and custom-emoji support introduce additional trust and entity-handling questions already represented by the narrower https://github.com/openclaw/openclaw/issues/16121.
