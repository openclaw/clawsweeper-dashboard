## Summary

Keep open. Current main still does not expose a Telegram `parseMode` setting at channel or account scope; the newer `richMessages` option fixes adjacent rich-message compatibility behavior but is not the requested plain/HTML/Markdown parse-mode contract.

## What This Changes

The issue requests a configurable Telegram message parse mode, including a top-level default and possible per-account override.

## Best Possible Solution

Define one maintainer-approved Telegram formatting contract, then add typed Telegram plugin config, docs, metadata, and tests that preserve current defaults while wiring the chosen behavior through sends, captions, edits, previews, and durable delivery.

## Reproduction Assessment

Yes, source-level: current main has no Telegram `parseMode` key in the strict schema, exported config type, public Telegram docs, or generated metadata. I did not run a live Telegram client proof for each requested mode in this read-only review.

## Solution Assessment

Unclear as proposed: a parse-mode knob is plausible, but the exact supported modes, default, per-account scope, and interaction with `richMessages` need maintainer direction. The maintainable solution is one documented formatting contract wired across every Telegram delivery path.
