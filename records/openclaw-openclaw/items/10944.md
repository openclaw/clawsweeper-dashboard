## Summary

Keep open. Current main and the latest release still do not expose a supported Telegram parseMode config, and the remaining work is a product/config decision rather than a cleanup-close candidate.

## What This Changes

Request to add Telegram parse-mode configuration, including possible per-account overrides, so users can choose plain text, Markdown-like rendering, or HTML behavior instead of current hardcoded formatting paths.

## Best Possible Solution

Maintainers should define the supported Telegram formatting contract first, then add typed Telegram config and docs/tests that preserve current defaults while consistently wiring chosen behavior across text, captions, edits, previews, and payload delivery.

## Reproduction Assessment

Yes, at source level: current main has no Telegram parseMode key in the strict schema or exported config type, and the outbound paths still choose rich/HTML behavior internally. I did not run a live Telegram bot repro because the missing config surface is clear from source and docs.

## Solution Assessment

Unclear as proposed: a parse-mode config knob is plausible, but the exact values, default, per-account scope, and interaction with Telegram rich messages need maintainer direction. The safer end state is a single documented Telegram formatting contract wired through every delivery path, not a one-path parse_mode toggle.
