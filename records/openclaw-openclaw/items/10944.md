## Summary

Keep open. Current main still has no supported Telegram parse-mode configuration at either the channel or account level, while text sends, media captions, edits, draft previews, and lower-level delivery helpers still force Telegram HTML formatting with plain-text fallback. The discussion correctly notes the original Markdown claim is outdated, but the broader configurability request remains valid; #10758 and #16121 are narrower custom-emoji/raw-HTML items rather than canonical replacements.

## What This Changes

The issue requests adding Telegram parseMode configuration at the top-level channel and per-account config surfaces so users can choose plain/no-parse, Markdown-style, or HTML behavior instead of a hardcoded Telegram parse mode.

## Best Possible Solution

Add a typed Telegram plugin config surface for parse-mode behavior, preserve the current HTML rendering as the compatibility default unless maintainers intentionally change it, support a safe plain/no-parse path, and wire the setting through sends, captions, edits, draft previews, and delivery helpers with schema/types/metadata/docs/tests updated together.

## Reproduction Assessment

Yes. The missing feature is reproducible by source inspection: strict Telegram config schema/types/metadata omit parseMode, and the current outbound Telegram paths still set parse_mode: "HTML" instead of reading config.

## Solution Assessment

Unclear for the exact proposal because the supported values and default are not settled. The maintainable direction is a bundled Telegram plugin config option with explicit escaping/trust semantics, with MarkdownV2/raw HTML/custom emoji behavior decided as part of maintainer review or tracked follow-ups.
