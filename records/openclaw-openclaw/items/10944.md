## Summary

Keep open. Current main still has no supported Telegram parse-mode config at the channel or account level, and the Telegram delivery paths continue to force HTML parse mode with plain-text fallback rather than exposing the requested configuration.

## What This Changes

Request to add Telegram parse-mode configuration, including a top-level setting and optional per-account override, instead of relying on a hardcoded Telegram parse mode.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface with typed top-level and account-level parse-mode resolution, then wire the resolved mode consistently through sends, captions, edits, and previews while documenting any deliberately unsupported modes.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection still confirms the current limitation because strict Telegram config has no parseMode field and delivery paths hardcode HTML.

## Solution Assessment

Yes in direction, but it needs maintainer scoping first. The maintainable path is a Telegram-owned config addition with consistent delivery wiring; raw HTML/custom emoji semantics should remain an explicit product/security decision or narrower follow-up.
