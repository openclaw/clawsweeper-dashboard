## Summary

Keep open. Current main still lacks a supported Telegram parse-mode config and continues to hardcode HTML formatting across text sends, media captions, edits, previews, and lower-level delivery helpers; the related custom-emoji issues are narrower and do not supersede this request.

## What This Changes

The issue requests a configurable Telegram parseMode setting, optionally with per-account overrides, so users can choose plain/no-parse, Markdown-style, or HTML rendering instead of the current hardcoded mode.

## Best Possible Solution

Add a bundled Telegram plugin config surface that preserves current HTML behavior by default unless maintainers choose otherwise, supports a safe no-parse path, and updates schema, exported types, generated metadata/help, docs, and focused Telegram/config tests together.

## Reproduction Assessment

Yes. The missing feature is source-reproducible from current main: the strict Telegram schema/types/docs omit parseMode and the send, preview, edit, and delivery paths still hardcode HTML parse mode; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the supported values, default, and escaping semantics. The narrow maintainable direction is a Telegram plugin config option, with raw HTML, MarkdownV2, and custom emoji behavior treated as explicit design choices or follow-ups.
