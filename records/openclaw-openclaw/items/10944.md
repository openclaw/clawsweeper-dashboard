## Summary

Keep open. Current main still has no supported Telegram parseMode config at the channel or account level; the newer richMessages option is adjacent but does not provide the requested plain/Markdown/HTML parse-mode contract.

## What This Changes

The issue requests configurable Telegram parseMode behavior at the channel and per-account level instead of hardcoded outbound formatting.

## Best Possible Solution

Define one supported Telegram formatting contract first, then add typed Telegram config, docs, metadata, and tests that preserve current defaults while wiring the chosen behavior through sends, captions, edits, previews, and durable delivery.

## Reproduction Assessment

Yes, source-level: current main has no Telegram parseMode key in the strict schema or exported config type, and outbound paths still choose HTML/rich behavior internally. I did not run a live Telegram repro because the missing config surface is clear from source and docs.

## Solution Assessment

Unclear as proposed: a parseMode knob is plausible, but the exact modes, default, per-account scope, and interaction with richMessages need maintainer direction. The maintainable solution is a single documented formatting contract wired across every Telegram delivery path.
