## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject non-local SystemRoot before spawning cmd.exe:** `src/infra/windows-encoding.ts:70-78`
  - body: `path.win32.isAbsolute()` returns true for UNC and extended-length roots, and the basename/parent check still accepts values like `\\server\share\Windows`. That lets a tainted environment steer the `chcp` probe to `\\server\share\Windows\System32\cmd.exe`; require a local drive root or use the existing trusted install-root resolver before spawning.
  - confidence: 0.92
- **[P1] Apply the same cmd.exe hardening to the memory SDK:** `packages/memory-host-sdk/src/host/windows-output-decoder.ts:143-150`
  - body: The package-local decoder repeats the same `SystemRoot` validation before calling `spawnSync(resolveTrustedWindowsCmdPath(), ...)`, so the SDK path would keep the executable hijack risk even if the core helper is fixed. Harden this copy as well, or share a package-safe local-root helper.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces two security-sensitive executable path derivations that still trust non-local `SystemRoot` roots before spawning `cmd.exe`.

Concerns:

- **[medium] UNC SystemRoot can redirect core cmd.exe probe:** `src/infra/windows-encoding.ts:70`
  - body: The core Windows encoding helper accepts UNC/device absolute roots and joins them with `System32\cmd.exe`, allowing a tainted environment to redirect the `chcp` probe to an attacker-controlled executable.
