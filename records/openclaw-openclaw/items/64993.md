## Label Justifications

- `P1`: This is a high-severity auth-boundary issue that can expose owner-scoped loopback tools to code holding a valid backend bearer.
- `impact:security`: The report is about authorization scope being derived from attacker-mutable loopback request headers after bearer authentication.

## Summary

Keep open. Current main and v2026.5.18 still derive MCP loopback session, account, message-channel, and inbound-event scope from mutable request headers after bearer authentication; only the owner bit has been token-bound, and the full fix remains in an open conflicting PR.

## What This Changes

The issue asks OpenClaw to bind MCP loopback authorization scope to bearer-token identity instead of trusting mutable request headers.

## Best Possible Solution

Land a rebased, security-reviewed MCP loopback fix that derives the full session/account/channel/inbound-event/owner tuple from server-side token identity, removes emitted scope headers where no longer needed, and keeps regression coverage for spoofed scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: authenticate with a valid loopback bearer, vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and current main threads those values into McpLoopbackToolCache and resolveGatewayScopedTools.

## Solution Assessment

Yes for the token-bound full-tuple direction. Current main’s owner-token mitigation is only partial because the rest of the loopback scope still comes from request headers.

Status: needs_attention

Summary: The reported MCP loopback auth-boundary issue remains because several scope fields still come from mutable request headers after bearer authentication.

Concerns:

- **[high] Bind MCP loopback scope to token identity:** `src/gateway/mcp-http.request.ts:173`
  - body: resolveMcpRequestContext still accepts session, account, message-channel, and inbound-event scope from inbound request headers, so code with a valid loopback token can present a different scoped context until the gateway records authoritative scope server-side.
  - confidence: 0.94
- **[medium] Stop emitting mutable scope headers to child backends:** `src/gateway/mcp-http.loopback-runtime.ts:37`
  - body: The generated loopback MCP client still sends scope headers sourced from backend environment values, preserving the client-controlled scope channel the issue asks to remove.
  - confidence: 0.9
- **[medium] Do not cache scoped tools by client-provided scope:** `src/gateway/mcp-http.runtime.ts:57`
  - body: McpLoopbackToolCache keys catalogs by the mutable session/account/channel tuple, so spoofed header values can select different cached tool catalogs under the same bearer class.
  - confidence: 0.88
