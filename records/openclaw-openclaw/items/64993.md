## Summary

Keep this security issue open. Current main mitigates the original owner-flag spoof by deriving owner status from owner/non-owner bearer tokens, but session key, account id, and message channel still come from mutable MCP loopback request headers; #67376 remains the active implementation path for token-bound scope.

## What This Changes

Bind MCP loopback scope to bearer-token identity so spawned backends cannot spoof session, account, channel, or owner scope through request headers.

## Best Possible Solution

Land a reviewed token-bound MCP loopback design that records authoritative per-backend scope server-side, resolves all scoped tool context from token identity, removes accepted/emitted scope headers and backend scope env vars, invalidates scoped caches on token revocation, and keeps regression coverage for owner spoofing plus session/account/channel isolation.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence for the remaining issue: use a valid non-owner loopback bearer while changing x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and current main still passes those header values into scoped tool resolution; the specific owner-header spoof no longer reproduces.

## Solution Assessment

Yes. Binding scope to bearer-token identity is the narrow maintainable fix for this trust boundary; current main is only a partial mitigation, and #67376 appears to target the right boundary but needs maintainer/security review.
