## Summary

Keep open. Current main still lets MCP loopback session, account, and message-channel scope come from mutable request headers after bearer authentication; only the owner-bit spoofing path has been partially mitigated, and the linked fix PR remains open and conflicting.

## What This Changes

Bind MCP loopback scope to bearer-token identity so session, account, channel, and owner scope cannot be spoofed through request headers.

## Best Possible Solution

Land a reviewed MCP loopback security patch that resolves the full session/account/channel/owner tuple from server-side token identity, removes emitted and accepted scope headers, invalidates scoped caches on revocation, and keeps regression coverage for owner and non-owner scope isolation.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: use a valid loopback bearer and vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values flow into resolveGatewayScopedTools through McpLoopbackToolCache.

## Solution Assessment

Yes for the token-bound-scope direction. Current main’s owner-token mitigation is only partial; the maintainable solution is to bind the full scope tuple to token identity and land the fix after security review.
