## Summary

Keep open. Current main still accepts session, account, and message-channel scope from mutable MCP loopback request headers after bearer authentication; the owner-header spoofing piece is partially mitigated, but the broader token-bound-scope fix remains open in https://github.com/openclaw/openclaw/pull/67376 and prior maintainer triage explicitly kept this as a real auth-boundary issue.

## What This Changes

The issue asks OpenClaw to bind MCP loopback session, account, message-channel, and owner scope to bearer-token identity instead of accepting mutable request headers.

## Best Possible Solution

Land a reviewed MCP loopback security patch that records authoritative per-backend scope server-side, resolves session/account/channel/owner context from token identity, removes accepted/emitted scope headers, invalidates scoped caches on revocation, and keeps regression coverage for owner spoofing plus session/account/channel isolation.

## Reproduction Assessment

Yes. Source inspection against current main gives a high-confidence path: use a valid loopback bearer and vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values reach resolveGatewayScopedTools; I did not run a live exploit in this read-only review.

## Solution Assessment

Yes for the token-bound-scope direction. The narrow maintainable solution is to bind the full scope tuple to bearer-token identity; https://github.com/openclaw/openclaw/pull/67376 is the active path but needs rework or security review before landing.
