## Summary

Keep open. Current main and the latest stable release still derive MCP loopback session, account, and message-channel scope from mutable request headers after bearer authentication; https://github.com/openclaw/openclaw/pull/69796 only fixed the owner-bit spoofing path, and the full token-bound-scope PR remains open and conflicting at https://github.com/openclaw/openclaw/pull/67376.

## What This Changes

The issue asks OpenClaw to bind MCP loopback scope to the bearer credential instead of accepting session/account/channel/owner scope from mutable request headers.

## Best Possible Solution

Land a rebased, security-reviewed MCP loopback fix that resolves the full session/account/channel/owner tuple from server-side token identity, removes emitted and accepted scope headers, invalidates scoped caches on revocation, and keeps regression coverage for scope isolation.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: use a valid loopback bearer and vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values flow into resolveGatewayScopedTools through McpLoopbackToolCache.

## Solution Assessment

Yes for the token-bound full-tuple direction, but not as current main stands. The current owner-token mitigation is only partial; the maintainable fix is to derive all loopback scope from token identity and stop accepting scope headers.
