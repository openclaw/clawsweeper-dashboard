## Summary

Keep open. Current main mitigates the owner-flag spoof by deriving owner status from owner/non-owner bearer tokens, but session key, account id, and message channel still come from mutable MCP loopback request headers; the open implementation path is https://github.com/openclaw/openclaw/pull/67376.

## What This Changes

The issue asks OpenClaw to bind MCP loopback session, account, channel, and owner scope to bearer-token identity instead of trusting mutable request headers.

## Best Possible Solution

Land a reviewed token-bound MCP loopback design that records authoritative per-backend scope server-side, resolves all scoped context from token identity, removes accepted/emitted scope headers and backend scope env vars, invalidates scoped caches on token revocation, and preserves regression coverage for owner spoofing plus session/account/channel isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: authenticate with a valid non-owner loopback bearer, change x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and current main still passes those header values into scoped tool resolution.

## Solution Assessment

Yes. Binding the full scope tuple to bearer-token identity is the narrow maintainable fix; current main is only a partial owner-status mitigation, and https://github.com/openclaw/openclaw/pull/67376 targets the right boundary but still needs review.
