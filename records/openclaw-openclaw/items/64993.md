## Summary

Keep this security issue open. Current main mitigates the exact owner-flag spoof by deriving owner context from owner/non-owner bearer tokens, but the broader token-bound scope requested here is still missing because session key, account id, and message channel continue to come from mutable MCP loopback request headers; #67376 remains the active implementation path.

## What This Changes

The issue asks OpenClaw to bind loopback MCP scope to the authenticated bearer token instead of trusting request headers for session, account, message channel, and owner context.

## Best Possible Solution

Land a reviewed token-bound MCP loopback design that records authoritative per-backend scope server-side, resolves all scoped tool context from token identity, removes accepted/emitted scope headers and backend scope env vars, invalidates scoped caches on token revocation, and keeps regression coverage for owner spoofing plus session/account/channel isolation.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence for the remaining issue: use a valid non-owner loopback bearer while changing x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and current main still passes those header values into scoped tool resolution. The specific owner-header spoof no longer reproduces because senderIsOwner comes from the matched bearer token.

## Solution Assessment

Yes. Binding scope to bearer-token identity is the narrow maintainable fix for this trust boundary; the current main changes are a partial mitigation, and #67376 appears to target the right boundary but needs maintainer/security review before this issue can close.
