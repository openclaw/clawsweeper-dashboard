## Summary

Keep open. Current main still accepts session key, account id, and message channel from mutable MCP loopback request headers after bearer authentication, and latest release v2026.5.7 contains the same path; https://github.com/openclaw/openclaw/pull/67376 is the open implementation path but still needs security review before landing.

## What This Changes

The issue reports an MCP loopback auth-boundary flaw where bearer-authenticated requests can spoof scoped tool context through mutable request headers.

## Best Possible Solution

Land a reviewed, narrow MCP loopback patch that records authoritative per-backend scope server-side, resolves all scoped context from token identity, removes accepted/emitted scope headers, and keeps regression coverage for owner spoofing plus session/account/channel isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: authenticate to /mcp with a valid non-owner loopback bearer, vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and current main passes those values into resolveGatewayScopedTools.

## Solution Assessment

Yes for the token-bound-scope direction. The narrow maintainable fix is to bind the full scope tuple to bearer-token identity; current main only partially mitigates owner-header spoofing, and the open PR should be reviewed or reworked before landing.
