## Summary

Keep open. Current main and the latest release still accept MCP loopback session, account, and message-channel scope from mutable request headers after bearer authentication; only the owner-bit spoofing part has been partially mitigated, and maintainer triage explicitly kept this as a real auth-boundary issue.

## What This Changes

The issue requests binding MCP loopback request scope to bearer-token identity instead of trusting mutable scope headers from spawned backends.

## Best Possible Solution

Land a reviewed MCP loopback security patch that resolves full session/account/channel/owner scope from server-side token identity, removes accepted and emitted scope headers, invalidates scoped caches on revocation, and keeps regression coverage for both owner spoofing and session/account/channel isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a valid loopback bearer and vary x-session-key, x-openclaw-account-id, or x-openclaw-message-channel, and those values reach resolveGatewayScopedTools through McpLoopbackToolCache.

## Solution Assessment

Yes for the token-bound-scope direction. The current main partial owner-bit mitigation is not enough; the narrow maintainable fix is to bind the full scope tuple to bearer-token identity and land it after security review.
