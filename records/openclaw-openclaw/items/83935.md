## Summary

Keep open: current main and the latest v2026.5.18 release still map the reported stale `openai-codex` missing-key failure to generic provider-auth guidance, while the closing PR that adds doctor-first recovery guidance remains open and unmerged.

## What This Changes

The issue asks post-update sessions with stale `openai-codex` route state to auto-repair or point users at `openclaw doctor --fix` instead of generic API-key setup guidance.

## Best Possible Solution

The desired end state is that stale post-update Codex route failures either avoid stale session pins or point users to doctor-first repair; https://github.com/openclaw/openclaw/pull/83937 is the current implementation candidate.

## Reproduction Assessment

Yes, at source level, though not through a fresh live upgrade in this review. Current main and v2026.5.18 return generic `openai-codex` missing-key guidance, while doctor code/docs and reporter output show the stale route state is repairable by doctor.

## Solution Assessment

Yes. The linked PR's narrow doctor-first guidance and stale auto-session cleanup reuse the existing doctor repair boundary; automatic startup repair would be a separate product choice.
