## Label Justifications

- `P1`: The report describes a post-update regression that can break a live Telegram agent session until manual repair.
- `impact:session-state`: The failure is tied to persisted session route fields such as `providerOverride` and `modelProvider` staying stale after update.
- `impact:auth-provider`: The user-visible symptom is provider auth routing to `openai-codex` and surfacing a missing-key error despite healthy Codex/OpenAI auth after doctor repair.

## Summary

Keep open: current main has the doctor repair path for stale Codex session routes, but the live chat failure formatter still maps `openai-codex` missing-key failures to generic gateway-auth guidance instead of the existing doctor recovery path.

## What This Changes

The issue requests better post-update handling when stale `openai-codex` session routing causes generic missing-key replies until doctor repairs the session state.

## Best Possible Solution

Post-update stale Codex session route state should produce the existing `openclaw doctor --fix` recovery guidance, or be repaired through the existing doctor-owned path before chat sessions resume if maintainers explicitly want startup mutation.

## Reproduction Assessment

No live upgrade reproduction was run, but the source-reproducible path is clear: doctor detects and repairs stale `openai-codex` session route fields, while chat missing-key formatting still emits generic auth setup guidance for `openai-codex`.

## Solution Assessment

Yes, a narrow fix should reuse the existing stale-route knowledge in user-facing failure handling; startup auto-repair should remain a separate maintainer choice if it would mutate session stores outside doctor.
