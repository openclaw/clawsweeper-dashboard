## Summary

Keep open. Current main and v2026.5.19 still lack a server-side listability guard for sessions.list; null updatedAt may now normalize to 0 in one path, but a store placeholder without sessionId can still be selected and serialized, and the linked fix PR is closed unmerged.

## What This Changes

The issue reports that sessions.list returns a malformed per-agent store placeholder row instead of only real sessions with identity and activity metadata.

## Best Possible Solution

Add a Gateway-side listability guard before row construction so invalid persisted placeholders are excluded from sessions.list, with regression coverage for both the utility path and RPC path.

## Reproduction Assessment

Yes, source-reproducible. A per-agent store entry keyed sessions can be canonicalized to agent:<id>:sessions, accepted by the current list filters for that agent, and serialized as a row without sessionId.

## Solution Assessment

Yes. A server-side guard before row construction is the narrow maintainable fix; UI-only filtering would leave the RPC contract malformed, and a null-only updatedAt filter would miss current normalization variants.
