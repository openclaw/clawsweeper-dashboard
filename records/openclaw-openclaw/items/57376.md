## Summary

Keep open. Current main still allows invalid per-agent store entries to pass through `sessions.list` and serialize as null-updated rows; the only linked fix PR is closed unmerged, so the reported UI/RPC bug still needs a clean fix.

## What This Changes

The issue requests that `sessions.list` exclude store-level placeholder entries that lack `sessionId` and `updatedAt` instead of returning them as session rows.

## Best Possible Solution

Add a server-side session-list listability guard so only valid `SessionEntry` rows with real session metadata are serialized, with regression tests for both utility and Gateway RPC paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a raw `sessions` entry in a per-agent store canonicalizes to `agent:<id>:sessions`, passes current filtering, and is serialized with `updatedAt: null` and no `sessionId`. I did not run a live RPC because the review is read-only.

## Solution Assessment

Yes. A server-side listability guard before row construction is the narrowest maintainable fix; Control UI filtering would leave the RPC contract leaking malformed rows, and the prior unmerged PR should be replaced cleanly rather than revived with its accidental log file.
