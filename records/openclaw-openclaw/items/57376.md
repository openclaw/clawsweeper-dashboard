## Summary

Keep open. Current main and the latest release still lack a server-side listability guard for `sessions.list`; current normalization may coerce the reported `updatedAt: null` to `0`, but a no-`sessionId` store placeholder can still flow into the RPC, and the linked fix PR was closed unmerged.

## What This Changes

The issue asks for Gateway `sessions.list` to exclude store-level placeholder rows like `agent:<id>:sessions` that lack a real `sessionId` and activity.

## Best Possible Solution

Add a Gateway-side session-list selection guard that excludes invalid persisted rows before row construction, with coverage for utility-level listing and the `sessions.list` RPC using a per-agent store placeholder plus a real session.

## Reproduction Assessment

Yes, source-reproducible. A per-agent store entry keyed `sessions` can be canonicalized to `agent:<id>:sessions`, accepted by the current list filters for that agent, and serialized as a row without `sessionId`.

## Solution Assessment

Yes. A server-side listability guard before row construction is the narrow maintainable fix; UI-only filtering would leave the RPC contract leaking malformed rows, and a null-only timestamp filter is too narrow for current normalization.
