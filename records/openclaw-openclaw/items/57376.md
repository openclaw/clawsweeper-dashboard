## Summary

Keep open. Current main still allows a store-level placeholder object to be loaded, canonicalized to an agent session key, selected by `sessions.list`, and serialized as a row without real session identity; the only direct fix PR is closed unmerged.

## What This Changes

The issue requests that `sessions.list` exclude store-level placeholder entries such as `agent:<id>:sessions` that have no `sessionId`, no origin, and no activity timestamp.

## Best Possible Solution

Add a Gateway-side listability guard before row construction, with regression coverage for both the session utility and the RPC combined-store path while preserving valid global, unknown, direct, group, and subagent rows.

## Reproduction Assessment

Yes from source inspection. Current main can load a raw per-agent `sessions` store key, canonicalize it to `agent:<id>:sessions`, pass it through `sessions.list` filters, and serialize it with `updatedAt: null` and no `sessionId`; I did not run a live RPC in this read-only review.

## Solution Assessment

Yes. A server-side listability guard before row construction is the narrow maintainable fix; UI-only filtering would leave the RPC contract malformed, and a broad store migration would be premature without identifying the writer.
