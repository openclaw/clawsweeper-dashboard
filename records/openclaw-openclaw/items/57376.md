## Summary

Keep open. Current main still allows malformed per-agent session-store placeholder rows through `sessions.list`, and the linked #57419 fix PR is closed without merging, so the reported API/UI leak remains source-reproducible.

## What This Changes

Report asks `sessions.list` to exclude store-level placeholder rows such as `agent:<id>:sessions` that have no `sessionId` and `updatedAt: null`.

## Best Possible Solution

Add a Gateway-side listable-session guard before row construction and cover a placeholder-plus-real-session store in unit/RPC regression tests.

## Reproduction Assessment

Yes. Source inspection is enough: a raw `sessions` key in an agent store canonicalizes to `agent:<id>:sessions`, survives `filterAndSortSessionEntries`, then `buildGatewaySessionRow` emits `updatedAt: null` with no `sessionId`.

## Solution Assessment

Yes. Server-side filtering before `buildGatewaySessionRow` is the narrowest maintainable fix; Control UI-only filtering would leave the RPC contract leaking malformed rows.
