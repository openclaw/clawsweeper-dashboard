## Summary

Keep open. Current main still lets malformed store placeholder rows flow through `sessions.list`: the handler passes the combined store directly into the list helper, the helper filters scope/search/activity but does not require a real `sessionId` or activity timestamp, and the row builder serializes `updatedAt` as `null` with an omitted `sessionId`, matching the reported phantom row. The linked implementation PR #57419 is closed and unmerged in the provided context, so the server-side bug remains actionable.

## What This Changes

The issue requests that `sessions.list` exclude store-level placeholder entries such as `agent:<id>:sessions` that lack a `sessionId`, origin/display metadata, and real activity.

## Best Possible Solution

Add a server-side validity guard at `listSessionsFromStore` or the combined-store normalization boundary so non-session placeholders are excluded before row construction, with regression coverage for an `agent:<id>:sessions` placeholder beside a real session. Preserve legitimate session rows, including global/unknown or subagent rows that are intentionally listable, and keep clients from needing their own workaround filters.

## Reproduction Assessment

Yes. A direct unit-level reproduction is to pass `listSessionsFromStore` a store containing `agent:sean:sessions` with no `sessionId` and no valid `updatedAt` alongside a real `agent:sean:*` session; current code retains the placeholder and builds a row with `updatedAt: null` and no `sessionId`. No tests were run because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix is server-side filtering or normalization before `buildGatewaySessionRow`; client-side filtering would hide only the Control UI symptom, and a suffix-only check would be weaker than a clear listable-session predicate.
