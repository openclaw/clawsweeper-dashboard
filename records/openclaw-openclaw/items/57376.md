## Summary

Keep open. Current main still permits the malformed store-placeholder path: per-agent raw `sessions` keys canonicalize to agent-scoped keys, list filtering does not require a valid session entry, and row construction serializes the reported null/undefined fields; the only linked fix PR is closed unmerged.

## What This Changes

The issue asks for `sessions.list` to stop returning store-level placeholder rows like `agent:<id>:sessions` that lack `sessionId` and `updatedAt`.

## Best Possible Solution

Add a server-side validity guard before row construction, with regression coverage for a malformed per-agent `sessions` placeholder plus a real session for the same agent.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a per-agent store record named `sessions` with an object value canonicalizes to `agent:<id>:sessions`, passes the current list filters, and is emitted with `updatedAt: null` and no `sessionId`.

## Solution Assessment

Yes. The narrow maintainable fix is server-side filtering or normalization before `buildGatewaySessionRow`; client-side Control UI filtering would leave the API contract leaking malformed rows.

## Repair Work Prompt

Fix the `sessions.list` phantom store-placeholder leak reported at https://github.com/openclaw/openclaw/issues/57376. The server should not return malformed store-level rows such as a per-agent raw `sessions` entry canonicalized to `agent:<id>:sessions` when the entry lacks a non-empty `sessionId` and finite `updatedAt`. Keep the fix in the Gateway/session-store listing boundary, not as a Control UI-only workaround, and avoid adding startup/load-time migrations. Add focused regression coverage for a malformed placeholder plus at least one real session for the same agent. Include a user-facing `CHANGELOG.md` fix entry. Relevant prior source work: closed unmerged https://github.com/openclaw/openclaw/pull/57419.
