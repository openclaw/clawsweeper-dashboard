## Summary

Keep open. Current main still permits malformed per-agent session-store placeholder rows through `sessions.list`, and the linked fix PR is closed unmerged, so the reported API/UI leak remains source-reproducible.

## What This Changes

The issue reports that `sessions.list` returns a store-level placeholder row such as `agent:<id>:sessions` with `updatedAt: null` and no `sessionId` or origin metadata.

## Best Possible Solution

Add a Gateway-side listable-session guard before row construction, preserve valid legacy/global rows, and cover placeholder-plus-real-session stores in unit/RPC regression tests with a changelog fix entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a raw per-agent `sessions` placeholder canonicalizes to `agent:<id>:sessions`, survives list filtering, and is serialized with `updatedAt: null` and no `sessionId`.

## Solution Assessment

Yes. Server-side filtering before `buildGatewaySessionRow` is the narrowest maintainable fix; Control UI-only filtering would leave the RPC contract leaking malformed rows.
