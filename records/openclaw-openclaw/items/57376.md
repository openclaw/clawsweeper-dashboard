- `impact:session-state`: The issue is about malformed persisted session state leaking into a client-visible sessions.list row.

## Summary

Keep open. Current main still allows a store-level placeholder object to be loaded, canonicalized to an agent session key, selected by sessions.list, and serialized as a row without a real session identity; the only direct fix PR is closed unmerged.

## What This Changes

Filter sessions.list so store-level placeholder keys such as agent:<id>:sessions do not appear as real session rows.

## Best Possible Solution

Add a server-side sessions.list listability guard before row construction, with regression coverage for utility-level listing and the Gateway RPC path.

## Reproduction Assessment

Yes. Source inspection on current main shows a raw per-agent store key can become agent:<id>:sessions, survive normalization as an object without sessionId or updatedAt, pass the current sessions.list filters, and serialize with updatedAt null and no sessionId.

## Solution Assessment

Yes. A Gateway-side listability guard before row construction is the narrow maintainable fix; UI-only filtering would leave the RPC contract malformed, and filtering only null updatedAt would miss normalized placeholder variants.
