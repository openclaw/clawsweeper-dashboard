- `impact:session-state`: The issue is about malformed persisted session state leaking into a client-visible `sessions.list` row.

## Summary

Keep open. Current `sessions.list` still has no server-side listability guard that requires a real session identity or activity before row construction, so a store-level placeholder keyed like an agent sessions store can still leak as a malformed row; the linked fix PR was closed unmerged.

## What This Changes

The issue requests filtering store-level placeholder entries out of `sessions.list` so only real sessions with identity and activity are returned.

## Best Possible Solution

Add a Gateway-side listability guard before `GatewaySessionRow` construction and cover both the utility-level listing and `sessions.list` RPC path with a store-level placeholder fixture.

## Reproduction Assessment

Yes. Source inspection on current main shows a raw per-agent `sessions` store key can canonicalize to `agent:<id>:sessions`, survive normalization as an object without `sessionId` or `updatedAt`, pass the current list filters, and serialize as a row with `updatedAt: null`.

## Solution Assessment

Yes. A server-side listability guard before row construction is the narrow maintainable fix; UI filtering would leave the RPC contract malformed, and filtering only `updatedAt: null` would be less robust than requiring a real session identity or activity signal.
