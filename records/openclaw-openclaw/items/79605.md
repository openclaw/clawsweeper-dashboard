## Summary

Keep open: current main still carries the Telegram partial-preview implementation, but no merged fix addresses the reported final-only behavior, and a maintainer explicitly said the related debounce-removal PR should close while this issue stays open for a debounce-preserving fix.

## What This Changes

The issue reports that Telegram `streaming.mode: "partial"` in OpenClaw 2026.5.7 sends only a final message instead of creating a live preview and editing it with `editMessageText`.

## Best Possible Solution

Restore Telegram partial preview edits for direct-model text replies while preserving the 30-character first-preview debounce, then verify with focused regression coverage plus a redacted live Telegram DM run or equivalent runtime logs.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection supports a conditional final-only path when no usable cumulative partial crosses the first-preview gate before finalization, but the exact direct-model breakpoint needs reporter or maintainer logs.

## Solution Assessment

No patch is proposed here. The related debounce-removal fix is not the best solution because a maintainer stated the 30-character first-preview debounce is intentional; the safer path is to preserve that behavior and repair where usable text deltas or finalization stop reaching the preview lane.
