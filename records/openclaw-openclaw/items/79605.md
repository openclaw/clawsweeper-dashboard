## Summary

Keep open: this is still a docs-backed Telegram preview-streaming regression with no merged fix on current main, and the related no-debounce fix was explicitly closed because this issue should stay open for a solution that preserves Telegram’s 30-character first-preview debounce.

## What This Changes

The issue asks OpenClaw to make Telegram `streaming.mode: "partial"` create and edit a live preview in DMs instead of only sending the final response.

## Best Possible Solution

Restore Telegram partial previews while preserving the intentional first-preview debounce, with focused regression coverage plus redacted live logs or a Telegram DM smoke showing preview `sendMessage` followed by `editMessageText` before final delivery.

## Reproduction Assessment

No, we do not yet have a high-confidence live reproduction path. Source inspection explains the final-only symptom when no usable partial text reaches the Telegram draft stream before final delivery, but the requested gateway logs are still needed to isolate the live direct-model path.

## Solution Assessment

No patch is proposed on this issue, and the related zero-threshold patch is not the best fix because maintainers want to preserve the 30-character push-notification debounce. The narrow solution should identify where usable deltas stop and restore preview edits without changing that product behavior.
