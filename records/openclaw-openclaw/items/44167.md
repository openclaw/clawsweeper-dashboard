## Summary

Keep open: current main has the retry-classifier portion but still lacks the exhausted multi-item batch split fallback, and this PR is the direct paired fix for the open linked issue. The branch is not mergeable as written because it conflicts with main, lacks real behavior proof, and regresses current generic batch-timeout retry classification.

## What This Changes

The PR changes memory-core batch embedding wrappers to use a retry helper that classifies transport errors separately and recursively splits failed multi-item text and structured batches, with focused policy tests.

## Best Possible Solution

Rebase or replace this with a narrow memory-core patch that preserves the current retry classifier, including generic `timed out`, adds split-after-exhausted-transport-retry fallback for text and structured batches, and includes regression coverage plus redacted real indexing proof.

## Reproduction Assessment

Yes, at source level: mock a multi-item embedding batch to keep throwing `TypeError("fetch failed")` after retries while single-item calls succeed. Current main sends both batch wrappers through `runMemoryEmbeddingRetryLoop`, which throws after exhaustion because no split fallback exists.

## Solution Assessment

No, not as written. Retry-plus-split is the right narrow direction, but this branch must preserve generic batch timeout retries and provide real behavior proof before merge.
