## Summary

Keep this PR open: the split-after-exhaustion behavior is still useful and not present on current main, but the branch is conflicting and the proposed predicate currently regresses retrying OpenClaw's own embedding timeout errors.

## What This Changes

The PR updates memory-core embedding batch calls to classify transient transport errors separately and recursively split multi-item batches after exhausted retries, with focused policy tests.

## Best Possible Solution

Rebase or replace this with a narrow current-main patch that preserves the shipped retry classifier, adds the exhausted transport batch-split helper for text and structured embeddings, and keeps focused regression coverage; maintainers can handle release-note polish at landing.

## Reproduction Assessment

Yes. A high-confidence source/unit reproduction is to mock a provider so multi-item embedding batches keep throwing `TypeError("fetch failed")` after retries while single-item batches succeed; current main retries the original full batch and then throws because no split helper exists.

## Solution Assessment

No, not as written. The split-helper direction is the right narrow fix, but the branch needs a current-main rebase and must keep existing `timed out` retry behavior before it is mergeable.

## Review Findings
