## Summary

Keep open: current main and v2026.5.12 already retry the named transport-shaped embedding errors, but they still do not split exhausted multi-item text or structured embedding batches; this PR contains useful split-after-exhaustion work, but the current branch is not merge-ready because it drops generic timeout retry coverage and lacks real behavior proof.

## What This Changes

The PR adds memory-core transport retry classification, a batch retry-with-split helper for text and structured embeddings, and policy tests for exhausted embedding transport failures.

## Best Possible Solution

Rebase or replace this with a focused memory-core patch that preserves current retry classification and timeout/AbortSignal behavior, adds split-after-retry-exhaustion fallback for text and structured batches, and includes regression coverage plus redacted real indexing proof.

## Reproduction Assessment

Yes, at source level. Mock a multi-item embedBatch or embedBatchInputs call so it exhausts retries with TypeError("fetch failed") while singleton calls succeed; current main routes those batches directly through runMemoryEmbeddingRetryLoop, which throws after exhausted retries with no split fallback.

## Solution Assessment

No, not as written. Retry-plus-split in the memory-core batch wrappers is the right narrow direction, but this branch must preserve generic timeout retry behavior and current abortable timeout handling, then provide real behavior proof before merge.
