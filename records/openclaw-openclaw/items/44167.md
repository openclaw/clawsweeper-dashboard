## Summary

Keep open: current main already shipped the retry-classifier subset through https://github.com/openclaw/openclaw/pull/76311, but this PR still contains the useful split-after-exhausted-transport-retries behavior; it is not mergeable as written because it drops generic batch-timeout retry coverage and lacks real behavior proof.

## What This Changes

The PR changes memory-core embedding retry policy to classify transport failures as retryable and recursively split text and structured multi-item embedding batches after exhausted transport retries.

## Best Possible Solution

Rebase or replace this with a narrow memory-core patch that preserves the current retry classifier, including generic `timed out`, adds split-after-exhausted-transport-retry fallback for text and structured batches, and includes regression coverage plus real indexing proof.

## Reproduction Assessment

Yes, at source level: current main retries transport-shaped errors but both batch wrappers still return the shared retry loop directly, so a mocked multi-item `TypeError("fetch failed")` that keeps failing after retries would throw instead of splitting.

## Solution Assessment

No, not as written. Retry-plus-split is the right narrow direction, but this branch must preserve generic batch timeout retries and provide real behavior proof before merge.
