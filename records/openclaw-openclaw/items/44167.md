## Summary

Keep open. Current main already retries the reported transport-shaped embedding errors through the merged retry-only work in https://github.com/openclaw/openclaw/pull/76311, but the PR still contains a meaningful split-after-exhausted-retries fallback that is not on main; it also needs a small timeout-regression fix and real behavior proof before merge.

## What This Changes

The PR changes memory-core embedding batch retry logic to classify transport failures, recursively split exhausted multi-item text and structured batches, update retry logging, and add policy tests.

## Best Possible Solution

Land or replace a current-main memory-core patch that preserves existing transport and timeout retry classification while adding exhausted multi-item split fallback for both text and structured embedding batches, with regression coverage and real indexing proof.

## Reproduction Assessment

Yes, at source level. Current main's text and structured batch wrappers call `runMemoryEmbeddingRetryLoop` directly, so a mocked multi-item `TypeError("fetch failed")` after retry exhaustion would throw instead of recursively splitting.

## Solution Assessment

No, not as written. Retry-plus-split is the right narrow direction, but the branch needs to preserve generic `timed out` retry behavior and add real behavior proof before merge.
