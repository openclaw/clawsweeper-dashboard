## Summary

Keep open: current main and v2026.6.8 still have the reported gap. Managed browser download endpoints exist, but the agent-facing browser tool schema, runtime barrel, and dispatcher do not expose `download` or `waitfordownload`, and open fixing PRs with closing syntax are still awaiting maintainer choice.

## What This Changes

The issue asks OpenClaw to notify agents when browser clicks trigger downloads, either by linking click results to download completion or by exposing agent-facing download/wait actions.

## Best Possible Solution

Keep this issue open while maintainers choose and land one browser-plugin fix that gives agents a reliable, documented download completion signal without duplicating or weakening the existing managed download constraints.

## Reproduction Assessment

Yes at source level: current main has managed `/download` and `/wait/download` routes, but the agent-facing browser tool action enum, runtime exports, and dispatch switch do not expose them or report click-triggered downloads. I did not run a live browser download because this review is read-only.

## Solution Assessment

Unclear as a product/API choice: both click-result notification and explicit download/wait actions could solve the user problem. The best fix is the maintainer-selected surface with regression coverage for host and node routing plus download path constraints.
