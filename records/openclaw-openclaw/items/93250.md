## Summary

Keep open: current main and the latest release still have the reported browser-download feedback gap, and two open PRs explicitly close this issue. The lower-level managed download routes exist, but the agent-facing Browser tool cannot call them and `act:click` still returns no download result.

## What This Changes

The issue asks for the browser tool to notify the agent when a click-triggered download completes, either by linking click results to download events or by exposing wait/download actions.

## Best Possible Solution

Land one reviewed browser-plugin fix that gives agents a single canonical download status/action path, preserves existing managed-download constraints, updates focused tests/docs, and then lets GitHub close this issue.

## Reproduction Assessment

Yes, at source level: current main saves unmanaged downloads in the browser layer but `act:click` returns no download payload and the Browser tool schema has no download or wait-for-download action. I did not run a live Chromium download in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape: current main is not sufficient, while the open PRs represent different plausible fixes. The best path is to review and land one canonical PR that keeps the existing download security constraints intact.
