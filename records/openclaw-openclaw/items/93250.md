## Summary

Keep open: current main still has the reported gap. The managed browser download endpoints exist, but the agent-facing browser tool does not expose `download` or `waitfordownload`, and `act:click` still returns no download result; https://github.com/openclaw/openclaw/pull/93307 and https://github.com/openclaw/openclaw/pull/93859 are open candidate fixes with closing syntax, so this issue should remain open until one lands or maintainers choose a different API shape.

## What This Changes

Report browser-triggered downloads to the agent, either by linking click results to saved downloads or by exposing wait/download actions that return the saved file path.

## Best Possible Solution

Land one reviewed browser-plugin fix that creates a single canonical agent-facing download status/action path, preserves existing download root and profile limitations, adds focused tests/docs, and closes duplicate PRs after the chosen PR merges.

## Reproduction Assessment

Yes, at source level: current main saves unmanaged downloads in the browser layer but `act:click` returns no download payload and the Browser tool schema has no download/wait action. I did not run a live Chromium download in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape: current main is not sufficient, while the open PRs represent different plausible fixes. The best path is to review and land one canonical PR that preserves the existing download security constraints.
