## Summary

Keep open: current main and v2026.6.8 still have managed browser download routes, but the agent-facing browser tool does not expose download completion for click actions or the explicit download/wait actions. Open candidate PRs now own the remaining work and need a maintainer API/security-boundary choice before this issue can close.

## What This Changes

The issue reports that browser click-triggered downloads are saved without being reported to the agent, causing repeated download attempts.

## Best Possible Solution

Land one reviewed canonical browser plugin fix that reports click-triggered download completion or exposes explicit download/wait actions through the existing managed download API, with focused tests and live proof, then close duplicate implementations after merge.

## Reproduction Assessment

Yes, at source level: current main auto-saves browser downloads and has managed download endpoints, but `act:click` returns no download payload and the browser tool schema/dispatcher omit download actions. I did not run a live Chromium download in this read-only review.

## Solution Assessment

Unclear: augmenting click results and adding explicit `download`/`waitfordownload` actions are both plausible, and maintainers need to choose the canonical API shape. The safest implementation should reuse the existing managed download API and preserve its path/profile restrictions.
