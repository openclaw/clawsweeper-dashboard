## Summary

Keep open: current main and the latest release still expose browser downloads through the control API/CLI, but the model-facing browser tool has no `download` or `waitfordownload` action and ordinary `act` clicks still return success without download metadata. The linked open PRs are implementation candidates, so the issue should remain visible until one is reviewed and landed or closed.

## What This Changes

The issue requests an agent-visible browser download completion/status path when a browser click saves a file, so the agent stops retrying successful downloads.

## Best Possible Solution

Review and land one current-main, proof-backed browser-tool download/waitfordownload or click-result notification implementation that reuses the existing managed download endpoints, preserves path/profile restrictions, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has managed download routes, but the model-facing browser tool schema and dispatcher do not expose them, and `act` clicks return success without download metadata.

## Solution Assessment

Yes in direction, but not yet landed. The maintainable owner boundary is to wire the agent tool or click result to the existing managed download endpoints, with maintainer acceptance of the agent-visible download capability.
