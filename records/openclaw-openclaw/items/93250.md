## What This Changes

The issue asks for browser click-triggered downloads to be surfaced to the agent, either by linking click results to download completion or by exposing wait/download actions.

## Best Possible Solution

Land one reviewed browser-plugin fix that creates a single canonical agent-facing download status/action path, preserves existing download root and profile limitations, adds focused tests/docs, and then closes the duplicate candidate PRs.

Yes, at source level: current main saves unmanaged downloads in the browser layer and has managed download routes, but the Browser tool schema and dispatcher do not expose download actions, and act:click returns no download payload. I did not run a live Chromium download in this read-only review.

Unclear until maintainers choose the API shape: current main is not sufficient, while the open PRs represent different plausible fixes. The best path is to review and land one canonical PR that preserves the existing download security constraints.
