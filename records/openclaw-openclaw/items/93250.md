## Summary

Keep open: current main and the latest release still have the reported agent-facing gap. Managed browser download endpoints exist below the tool, but the Browser tool schema/dispatcher do not expose download actions and `act:click` still returns no download metadata; https://github.com/openclaw/openclaw/pull/93307 and https://github.com/openclaw/openclaw/pull/93859 are open closing candidates, so this issue should wait for one reviewed API choice to land.

## What This Changes

The issue reports that browser-tool click-triggered downloads can be saved without notifying the agent, causing repeated attempts, and asks for click download feedback or a wait/download action.

## Best Possible Solution

Land one reviewed browser-plugin fix that creates a single canonical agent-facing download status/action path, preserves existing download root and profile limitations, adds focused tests/docs, and then close duplicate PRs after the chosen PR merges.

## Reproduction Assessment

Yes, at source level: current main saves unmanaged downloads and has managed download routes, but the Browser tool schema/dispatcher cannot expose them and `act:click` returns no download result. I did not run a live Chromium download in this read-only review.

## Solution Assessment

Unclear: current main is not sufficient, and the open PRs represent different plausible API shapes. The best path is for maintainers to choose and land one canonical browser-plugin fix that keeps the existing download security constraints.
