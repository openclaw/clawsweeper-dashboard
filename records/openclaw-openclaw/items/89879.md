## Summary

Keep open: the PR addresses the right lock writer, but it is not merge-ready because the fallback only catches the reported `ENOTSUP` code while known unsupported-hardlink paths can surface other errno codes, and the supplied proof is still monkey-patched rather than a real unsupported-hardlink filesystem run.

## What This Changes

The PR updates `writeUsageCostCacheLockAtomically` to reuse the lock payload and fall back to `fs.promises.writeFile(lockPath, payload, { flag: "wx" })` when `fs.promises.link()` fails with `ENOTSUP`.

## Best Possible Solution

Keep the lock-writer fallback, but use the repository's full unsupported-link error set, add focused fallback coverage, and require real unsupported-filesystem proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main publishes `.usage-cost-cache.json.lock` with `fs.promises.link`, and `acquireUsageCostCacheRefreshLock` only handles `EEXIST`, so an unsupported-link errno from the reported mount class escapes refresh. I did not run a live SMB/NFS/virtiofs mount repro in this read-only review.

## Solution Assessment

No, not as submitted. The fallback is in the right owner boundary, but the best fix should use the repo's broader unsupported-link error set instead of hard-coding only the observed `ENOTSUP` case.
