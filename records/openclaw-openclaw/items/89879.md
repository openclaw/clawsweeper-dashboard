## Summary

Keep open: the fallback is in the right lock helper, but it still only catches `ENOTSUP` while repo precedent and `link(2)` docs show other unsupported-hardlink errors, and the current proof is a monkey-patched `link()` run rather than a real unsupported filesystem.

## What This Changes

The PR changes the session cost/usage cache lock publisher to reuse the JSON lock payload and fall back to exclusive `wx` creation when `fs.promises.link()` reports `ENOTSUP`.

## Best Possible Solution

Broaden the lock-writer fallback to the known unsupported-hardlink error set, add focused regression coverage for fallback and contention behavior, then provide real unsupported-filesystem proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes at source level: current main publishes `.usage-cost-cache.json.lock` with `fs.promises.link`, and `acquireUsageCostCacheRefreshLock` only handles `EEXIST`, so unsupported-link errors from the reported mount class escape refresh. I did not run a live SMB/NFS/virtiofs mount repro in this read-only review.

## Solution Assessment

No, not as submitted. The fallback is in the right owner boundary, but the best fix should cover the known unsupported-link error set and add focused regression coverage before merge.
