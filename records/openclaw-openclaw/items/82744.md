## Summary

Keep open: the report remains a concrete current-main bug candidate because `/think` validation still uses the selected provider/model capability view before the later OpenAI Codex PI routing path, while OpenAI/OpenAI Codex policy declares `xhigh` for the reported GPT-5 models and the reporter provided fresh beta evidence for both canonical `openai/*` and explicit `openai-codex/*` configs.

## What This Changes

Fix runtime `/think` capability validation so supported OpenAI Codex-routed GPT-5 thinking levels are accepted for the effective serving target.

## Best Possible Solution

Make `/think` validation and stale thinking-level remapping use the same effective provider/model capability contract that the run will use, while keeping provider-owned thinking profiles and intentional catalog opt-outs intact.

Yes, source-reproducible with high confidence: the issue gives fresh live `/think xhigh` failures, and current source shows the directive gate checking selected provider/model capability before the later OpenAI PI-to-Codex routing context.
