## Summary

Keep open. Current main still lacks Feishu normal-reply sent-hook emission, so the PR is useful, but the branch is stale against current main and would drop the late-final streaming-card guard added after the PR was cut; it also lacks the required changelog entry.

## What This Changes

The PR adds Feishu normal-reply `message_sent` and internal `message:sent` hook emission, propagates session/group metadata from Feishu bot dispatch, exposes streaming-card message IDs, and adds dispatcher regression tests.

## Best Possible Solution

Land this PR or an equivalent Feishu-owned patch after rebasing the hook emission onto current main, preserving the late-final streaming-card guard and its tests, adding changelog credit, and keeping Drive/comment reply flows out of scope.

## Reproduction Assessment

Yes. The original bug is reproducible by configuring a Feishu `message:sent` hook and sending a normal bot conversation reply on current main, where the dispatcher sends directly without hook emission. The PR regression is source-reproducible with the current-main late-final test sequence: close a streaming card, then deliver a distinct late final with media.

## Solution Assessment

No, not as currently proposed. The Feishu reply dispatcher is the right fix boundary because it bypasses shared outbound delivery, but the branch must be rebased over current main's `streamingClosedForReply` guard and include the required changelog entry before it is the maintainable solution.
