## Summary

Keep open. This is a protected MEMBER-authored maintainer issue, and current main plus v2026.6.8 still have a source-backed final-fallback loss path while related open PRs cover only narrower slices.

## What This Changes

The issue asks OpenClaw to define and enforce a cross-channel contract so sanitized final fallback text is delivered or reported as an explicit delivery failure, not silently suppressed.

## Best Possible Solution

Keep this as the canonical maintainer-owned delivery contract, land or replace partial candidate PRs as slices, then add SDK/core conformance and diagnostics coverage across durable text-capable channels.

## Reproduction Assessment

Yes for a source-level reproduction: current main still suppresses final WhatsApp isError payloads and returns empty delivery identity for routed error payloads. I did not run a live cross-channel reproduction in this read-only review.
