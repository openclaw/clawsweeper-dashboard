## Summary

Keep open: the PR has a useful retry-backoff goal, but current main now treats terminal native-approval auth failures as disabled/no-retry, and this branch would turn `PAIRING_REQUIRED` into a periodic 300s retry without after-fix runtime proof.

## What This Changes

The PR changes native approval-handler bootstrap failures from a fixed 1s retry to exponential transient backoff, a 300s `PAIRING_REQUIRED` retry with diagnostics, and additional retry-state tests.

## Best Possible Solution

Rebase the transient backoff and diagnostic improvements onto current main while preserving terminal auth-failure pauses unless maintainers explicitly choose a new pairing-required retry policy.

## Reproduction Assessment

Yes, source-reproducible. Current main still uses a fixed 1s retry for non-terminal bootstrap failures, while current source and tests show `PAIRING_REQUIRED` terminal failures should not retry.

## Solution Assessment

No, not as written. The transient backoff is a narrow reliability improvement, but the branch's periodic `PAIRING_REQUIRED` retry conflicts with current main's terminal auth-failure pause.
