## Summary

Keep open. Current main partially addresses the production pairing-required retry storm by treating terminal native-approval auth failures as disabled instead of retrying, but it still uses a fixed 1-second retry for other bootstrap failures, so the PR's bounded backoff behavior is not fully implemented on main.

## What This Changes

The PR changes native approval handler bootstrap failures from a fixed 1-second retry loop to exponential transient backoff, a 300-second pairing-required backoff with clearer operator diagnostics, and targeted bootstrap tests.

## Best Possible Solution

Carry forward the useful bounded retry behavior as a rebased current-main patch that preserves the shipped terminal auth semantics unless maintainers explicitly prefer periodic pairing-required retry, adds the missing changelog entry, and keeps focused bootstrap/runtime tests for transient backoff and auth-pause behavior.

## Reproduction Assessment

Yes. The PR body gives a production PAIRING_REQUIRED retry-storm path, and current-main unit tests/source provide a high-confidence focused check: non-terminal failures still retry after exactly 1 second, while terminal PAIRING_REQUIRED errors no longer retry.

## Solution Assessment

Unclear. The PR is a narrow fix for transient backoff, but its 300-second pairing-required retry now overlaps a later shipped design that disables terminal auth retries; the safest solution is a rebased variant that reconciles those semantics explicitly.
