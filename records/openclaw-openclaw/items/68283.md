## Summary

Current main already fixes the production retry storm this PR targeted. The shipped implementation takes a narrower path than the PR: structured auth/pairing startup failures pause the Gateway client reconnect path and surface a terminal native-approval startup error, and the bootstrap does not schedule another retry for that terminal condition.

## What This Changes

The PR modifies native approval handler bootstrap startup failures to use exponential retry backoff, adds a 300-second pairing-required retry path with pairing diagnostics, and expands bootstrap retry tests.

## Best Possible Solution

Keep the shipped terminal-auth handling on main: pairing or auth startup failures that need operator action should pause instead of waking periodically, while retryable Gateway auth handoffs continue to wait for a successful hello.

## Reproduction Assessment

Yes. The PR body provides the production symptom, and current tests reproduce the core failure path with PAIRING_REQUIRED flowing through onReconnectPaused into a terminal startup error that the bootstrap does not retry.

## Solution Assessment

Yes. Current main’s no-retry terminal path is a narrower and safer fix for PAIRING_REQUIRED than a 300-second retry loop, because missing operator.approvals needs operator action rather than periodic reconnect attempts.
