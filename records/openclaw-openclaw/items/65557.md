## Summary

Keep open. Current main has a Control UI session model picker and a global `agents.defaults.models` allowlist, but it still lacks account-scoped provider/model allowlists, per-account defaults, wildcard policy, and account-aware server-side rejection; the related #7424 remains open but is narrower per-agent allowlist work, so it does not supersede this broader admin/account policy request.

## What This Changes

Request a session provider/model selector plus admin-managed per-account provider/model allowlists, per-account defaults, global fallback defaults, and server-side rejection for disallowed selections.

## Best Possible Solution

Define one server-side provider/model policy contract that composes global defaults, per-agent configuration, and account/user restrictions, then reuse it for catalog listing, session patching, slash/tool model changes, OpenAI-compatible overrides, admin UI, and docs.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. The source/docs verification path is high-confidence: current main exposes a session picker and global configured-model allowlist, but no account-scoped model policy or account-aware rejection path.

## Solution Assessment

Unclear because there is no proposed patch and the authorization model is still undefined. The narrow maintainable direction is a shared server-side policy resolver first, with UI filtering treated as presentation only.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because account-scoped provider/model allowlists would be an authorization boundary and must be enforced server-side.

- **[medium] Define server-side model authorization boundary:** `src/gateway/sessions-patch.ts:431`

  - confidence: 0.88

  - body: `models.list` currently accepts only a view and resolves a global visible catalog. Account-scoped allowlists need the server to derive trusted caller/account context before returning picker data.
