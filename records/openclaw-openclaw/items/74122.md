## Summary

Keep open: current main still uses a short config-reload debounce, drops watcher event paths before logging/evaluation, and does not provide native plugin runtime hot reload for non-config watched files.

## What This Changes

Request coalescing rapid gateway watcher changes into one reload, adding source-attributed reload logs, and avoiding full plugin/channel restarts for scoped non-config changes.

## Best Possible Solution

Split the work into a narrow gateway config reloader change for coalescing/source attribution, while tracking plugin runtime hot reload and targeted channel reload through #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms debounce and discarded watcher paths, so edits outside that short window can produce separate reload evaluations without trigger attribution. The non-config plugin/bootstrap portion is only partially reproducible from current main because the managed reloader watches the active config path and native plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No current-main behavior is not the best final solution for the reported operator problem. The safer path is to fix config reload coalescing/log attribution in the gateway reloader and handle plugin/channel lifecycle granularity in the existing broader follow-ups.
