## Summary

Keep open: current main still uses a 300 ms config reload debounce, watcher callbacks do not retain the triggering filesystem path, and the broader plugin/channel scoped reload parts remain split across adjacent open follow-ups.

## What This Changes

The issue requests coalescing rapid gateway watcher events into one reload, logging the triggering path, and avoiding full plugin/channel reloads for narrower file changes.

## Best Possible Solution

Scope the config reloader work separately from plugin hot-reload and targeted channel reload, then implement only the narrow coalescing and trigger-attribution changes in the gateway watcher path.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms default debounce and pathless watcher scheduling. Unclear for arbitrary non-config watched files because current managed gateway reload watches configSnapshot.path only.

## Solution Assessment

No: current main is not the final solution, and the issue should not be solved as one broad reload rewrite. The maintainable path is a focused config reloader change plus separate follow-through on #14438 and #72997.
