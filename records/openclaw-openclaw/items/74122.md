## Summary

Keep open. Current main still uses a 300 ms config reload debounce, discards the watcher event path before reload evaluation/logging, documents plugin runtime hot reload as unsupported, and restarts targeted channels by stop/start, so the reported operator problem is neither implemented nor fully superseded.

## What This Changes

The issue requests coalescing rapid gateway watcher edits into one reload, logging the triggering file, and avoiding full plugin/channel restart work for non-config changes.

## Best Possible Solution

Split the end state into maintainable reload work: add source-aware config reload coalescing and trigger logging in the gateway reloader, then keep plugin runtime hot reload under #14438 and targeted channel reload semantics under #72997.

## Reproduction Assessment

Yes for the config watcher portion: source inspection shows reload scheduling uses a 300 ms debounce and drops the event path, so saves spaced by seconds can schedule separate reloads with no triggering-file log. The plugin-code and broader non-config scoped reload portions are feature work rather than a fully supported current watcher path.

## Solution Assessment

No, current main is not the best final solution for the reported issue. The narrow maintainable first step is source-aware config reload coalescing/logging, while scoped plugin runtime and channel reload behavior needs separate lifecycle design.
