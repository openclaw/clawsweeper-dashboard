## Summary

Keep open: current main still has only a 300 ms config-reload debounce, does not retain chokidar event paths for reload attribution, and watches only the active config file; related open items cover plugin hot reload and targeted channel reload but not the whole reported watcher coalescing problem.

## What This Changes

The issue requests coalesced gateway watcher reloads, trigger-attribution logging, and scoped reloads for non-config/plugin/channel edits.

## Best Possible Solution

Split the gateway config reloader debounce and trigger-attribution work from the broader plugin hot-reload and targeted channel lifecycle follow-ups tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show the 300 ms default debounce, pathless watcher scheduling, and config-only watch path. The plugin/bootstrap-file portion is only partially reproducible from current main because the managed gateway reloader watches the active config file rather than arbitrary plugin or bootstrap files.

## Solution Assessment

No, current main is not the best final solution for the reported operator problem. The maintainable path is a narrow gateway config-reloader change plus separately scoped plugin-runtime and channel reload work.
