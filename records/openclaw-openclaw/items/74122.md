## Summary

Keep open. Current main still has only a short config-file debounce, drops the filesystem event path before reload evaluation/logging, and does not implement the broader scoped plugin/channel reload behavior requested here; related plugin hot-reload and channel reload work remains open in #14438 and #72997.

## What This Changes

The issue requests coalescing rapid watched-file changes into one reload, lighter scoped reloads for non-config changes, and log attribution for the file that triggered reload.

## Best Possible Solution

Split the work into maintainable pieces: add source-aware config reload coalescing and trigger logging in the gateway reloader, then align plugin runtime hot reload and targeted channel reload behavior with #14438 and #72997 rather than hiding those lifecycle decisions inside a debounce-only change.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows watcher events more than the current 300 ms debounce apart can schedule separate reload runs, and the event path is discarded before logging. The non-config scoped reload portion is only partially reproducible from current main because the gateway config reloader watches the active config path and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, current main is not the best final solution for the reported operator problem. The safest path is a narrow config coalescing/log-attribution patch plus separate maintainer-scoped decisions for plugin runtime and channel/account reload granularity.
