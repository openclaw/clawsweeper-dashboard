## Summary

Keep open: current main still has a short 300 ms config reload debounce, watcher events are scheduled without retaining the changed file path, and the broader non-config/plugin hot-reload behavior remains only partially covered by existing reload planning.

## What This Changes

The issue requests coalescing rapid watched-file changes into fewer gateway reloads, scoped refresh for non-config changes, and clearer reload-trigger logging.

## Best Possible Solution

Split the narrow gateway config reloader work for coalescing and trigger attribution from the broader plugin hot-reload and targeted channel reload follow-ups tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms default debounce and pathless watcher scheduling. The non-config watched-file portion is unclear on current main because the managed watcher is config-path scoped.

## Solution Assessment

No: current main is not the best final solution for the operator problem. The maintainable direction is a scoped config-reloader change plus separate maintainer-scoped work for plugin runtime and channel lifecycle semantics.
