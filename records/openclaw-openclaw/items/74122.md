## Summary

Keep open. Current main has configurable config-file debounce, but the default is still a short 300 ms window, watcher callbacks discard the filesystem event path before reload evaluation, and the broader scoped plugin/channel reload behavior remains unresolved and overlaps open lifecycle work in #14438 and #72997.

## What This Changes

The issue requests coalescing rapid watched-file edits into fewer reloads, adding reload trigger attribution in logs, and avoiding full plugin/channel restarts for smaller non-config changes.

## Best Possible Solution

Split the end state into maintainable pieces: add source-aware config reload coalescing and trigger attribution in the gateway reloader, then handle plugin runtime hot reload and targeted channel reload through the existing lifecycle tracks rather than folding all restart semantics into one watcher change.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows watcher events outside the current 300 ms debounce can schedule separate reload runs, and the event path is discarded before logging. The non-config scoped reload portion is more of a feature scope issue because current gateway config reload watches the active config path and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, current main is not the best final solution for the reported operator problem. The narrow maintainable path is config coalescing plus trigger logging first, with plugin runtime and channel reload granularity handled through the related lifecycle issues.
