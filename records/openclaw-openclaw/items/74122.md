## Summary

Keep open: current main only provides a short configurable config-reload debounce and still drops watcher source paths, while the broader plugin runtime and channel lifecycle pieces remain open follow-ups rather than implemented behavior.

## What This Changes

Request to coalesce rapid gateway watcher changes into fewer reloads, log the triggering file, and avoid full plugin/channel restarts for smaller watched-file changes.

## Best Possible Solution

Split the work: make a narrow gateway config reloader improvement for coalescing/source attribution, and keep plugin runtime hot reload plus targeted channel reload in their existing lifecycle follow-ups.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms debounce and discarded watcher paths, so edits outside that window can produce separate reload evaluations without trigger attribution. The non-config watched-file portion is only partially reproducible from current main because the managed reloader watches the active config file and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, treating this as one broad watcher fix is not the best path. The safer solution is to scope config debounce/log attribution separately from #14438 and #72997 lifecycle work.
