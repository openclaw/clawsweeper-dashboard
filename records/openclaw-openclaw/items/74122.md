## Summary

Keep open: current main partially covers config reload coalescing, but it still uses a 300 ms default debounce, does not carry the triggering filesystem path through watcher scheduling, and the broader scoped plugin/channel reload behavior remains split across open design work.

## What This Changes

Request to coalesce rapid watcher-triggered reloads, avoid full plugin/channel restarts for scoped non-config changes, and log which file or reason triggered a reload.

## Best Possible Solution

Split the config watcher debounce/log-attribution work from broader plugin source reload and channel/account-scoped reload design, using https://github.com/openclaw/openclaw/issues/14438, https://github.com/openclaw/openclaw/issues/72997, and https://github.com/openclaw/openclaw/issues/43935 for the larger reload semantics.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the config-watcher slice: the default debounce is 300 ms and watcher callbacks do not retain the triggering filesystem path; the non-config watched-file portion was not live-reproduced and current docs/source scope the managed watcher to the active config path.

## Solution Assessment

Unclear as one combined request. Longer/default coalescing and path attribution are plausible narrow Gateway changes, but scoped plugin and channel reload behavior needs maintainer scoping across existing reload-design issues before implementation.
