## Summary

Keep open: current main and the latest release still only partially address the report. The Gateway config watcher has a configurable debounce, but the default remains 300 ms, watcher events do not retain the triggering filesystem path for attribution, and the requested scoped plugin/channel reload behavior is still split across open design items rather than implemented here.

## What This Changes

The issue requests debounced watcher reloads, scoped reload behavior for non-config changes, and clearer reload-trigger diagnostics.

## Best Possible Solution

Split the narrow Gateway config-watcher debounce and trigger-attribution work from the broader scoped plugin/channel reload design, using the existing related issues for hot plugin reload and channel/account reload semantics.

## Reproduction Assessment

Partially. Source inspection gives a high-confidence path for the config-watcher slice: the default debounce is 300 ms and watcher callbacks do not retain the triggering filesystem path, but the non-config watched-file scope was not live-reproduced and current docs/source scope the managed watcher to the active config path.

## Solution Assessment

Unclear as one combined request. Longer coalescing and path attribution are plausible narrow Gateway changes, but scoped plugin/channel reload needs maintainer scoping across the existing reload-design issues before implementation.
