## Summary

Keep open: current main still has only a short 300 ms config reload debounce, watcher events do not retain the triggering filesystem path, and the broader scoped plugin/channel reload behavior remains unresolved in related open issues.

## What This Changes

Request to debounce rapid gateway watcher changes, avoid unnecessary full reloads for scoped or non-config changes, and log the file or reason that triggered each reload.

## Best Possible Solution

Split the config watcher coalescing and trigger-attribution work from the broader plugin hot-reload and targeted channel reload follow-ups tracked by #14438 and #72997.

## Reproduction Assessment

Yes, partially. The config watcher portion is source-reproducible from the 300 ms default, debounce scheduler, and pathless chokidar handlers; the non-config watched-file portion is unclear because current managed reload is scoped to the active config path.

## Solution Assessment

No. Current main is not the best final solution for the operator problem; the maintainable path is a narrow gateway config-reloader change plus separate maintainer-scoped plugin/channel reload work.
