## Summary

Current main still leaves the reported operator problem unresolved: config reloads only have a 300 ms default debounce, the watcher event path is not preserved for reload attribution, plugin runtime hot reload remains unsupported, and channel reloads still stop/start planned channels. The scoped-reload pieces overlap active follow-up work in #14438 and #72997, so this should stay open for maintainer scoping rather than be closed as implemented or superseded.

## What This Changes

The issue asks for rapid watched-file edits to be coalesced into one Gateway reload, non-config watched files to refresh only the affected subsystem, and reload logs to show the triggering file.

## Best Possible Solution

Implement source-aware config reload coalescing and trigger logging in the Gateway reloader, then keep plugin runtime hot reload and targeted channel reload behavior aligned with the dedicated open follow-ups instead of folding all lifecycle changes into a debounce-only patch.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows events more than 300 ms apart can schedule separate reload runs, and the chokidar event path is discarded before logging. The broader non-config scoped reload portion is only partially reproducible from current source because plugin runtime hot reload is documented as unsupported and targeted channel reload remains separate product work.

## Solution Assessment

No, solving the whole request as one patch is not the narrowest maintainable path. A safer direction is a first focused Gateway config-reloader change for coalescing and source attribution, with separate maintainer-scoped decisions for #14438 and #72997.
