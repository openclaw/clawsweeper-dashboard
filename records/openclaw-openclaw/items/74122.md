## Summary

Keep open: current main has a config reload debounce, but it defaults to 300 ms, watcher callbacks still do not retain the changed path, and the broader scoped plugin/channel reload behavior remains unresolved in related open follow-ups.

## What This Changes

Request coalesced gateway watcher reloads, reload reason logging, and scoped non-config/plugin/channel refresh instead of repeated full reloads during rapid edit sessions.

## Best Possible Solution

Split the narrow config reloader coalescing and trigger-attribution work from the broader plugin hot-reload and targeted channel reload designs tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher slice by source inspection: current main uses a 300 ms debounce and drops the changed path before scheduling. The broader non-config/plugin-file reproduction remains unclear because the managed gateway reloader watches the active config path while plugin/channel scoped reload is tracked separately.

## Solution Assessment

Unclear: longer debounce defaults and path logging are plausible, but bundling them with scoped plugin/channel reload is not the narrowest maintainable fix. The safer direction is to split the config watcher improvement from the existing scoped reload follow-ups.
