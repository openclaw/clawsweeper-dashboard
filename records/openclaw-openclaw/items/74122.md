## Summary

Keep open: current main still has only a short config-reload debounce, drops watcher event paths before logging, and does not cover the broader native plugin/runtime hot-reload request described here.

## What This Changes

The issue requests coalescing rapid watched-file edits into fewer gateway reloads, better reload-source logging, and lighter scoped reload behavior for non-config changes.

## Best Possible Solution

Split the work: improve gateway config watcher coalescing and trigger attribution here, while keeping native plugin hot reload and targeted channel reload in their existing follow-up issues.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms debounce and discarded watcher paths, so edits outside that short window can produce separate reload evaluations without source attribution. The non-config watcher portion is only partially reproducible from current main because the managed reloader watches the active config path.

## Solution Assessment

No current-main behavior is not the best final solution for the reported operator problem. A maintainable fix should be scoped to gateway config reload coalescing/logging first, with plugin/channel lifecycle granularity handled separately.
