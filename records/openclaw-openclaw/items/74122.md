## Summary

Keep open: current main still defaults config reload debounce to 300 ms, drops chokidar event paths before logging, and only watches the active config file path; existing config hot-reload planning covers some scoped channel/config cases but not the full coalescing, trigger-attribution, and plugin-runtime reload request.

## What This Changes

The issue asks for gateway watcher reloads to coalesce rapid file edits, avoid full reloads for narrower non-config changes, and log the triggering file.

## Best Possible Solution

Split the narrow gateway config reloader work for coalescing and trigger attribution from the broader plugin-runtime and channel lifecycle follow-ups already tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms default debounce and pathless watcher scheduling, so edits outside that short window can still produce separate reload evaluations without file attribution. The native plugin/bootstrap-file portion is only partially reproducible from current main because the managed reloader watches the active config path rather than arbitrary plugin or bootstrap files.

## Solution Assessment

No current-main behavior is not the best final solution for the reported operator problem. The maintainable path is a scoped config-reloader change plus separate maintainer-scoped work for plugin hot reload and targeted channel reload semantics.
