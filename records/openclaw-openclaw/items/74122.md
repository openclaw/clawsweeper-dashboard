## Summary

Keep open. Current main still does not satisfy the central request: config watcher events are only debounced by the existing short window, the filesystem event path is dropped before reload evaluation/logging, and the broader plugin/channel scoped-reload behavior remains separate lifecycle work tracked by related open issues.

## What This Changes

Request to coalesce rapid watched-file edits into a single gateway reload, avoid full plugin/channel restarts for non-config changes, and log the path or reason that triggered the reload.

## Best Possible Solution

Add source-aware config reload coalescing and trigger attribution in the gateway reloader, then handle plugin runtime hot reload and targeted channel reload through the existing lifecycle discussions rather than folding those decisions into a debounce-only change.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows watcher events outside the current 300 ms debounce can schedule separate reload runs, and the watcher does not retain the changed path. The non-config scoped reload portion is only partially reproducible from current main because the gateway config reloader watches the active config file while plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No; current main is not the best final solution for the reported operator problem. The safer path is a focused config coalescing/log-attribution change plus maintainer-scoped follow-through for plugin runtime and channel reload granularity.
