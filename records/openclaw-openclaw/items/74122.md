## Summary

Keep open: current main still has only a short config reload debounce and pathless watcher scheduling, while the broader non-config/plugin and targeted channel reload parts remain split across related open work.

## What This Changes

The issue requests debouncing rapid watched-file changes into one gateway reload, scoped reloads for non-config watched files, and reload logs that name the triggering file.

## Best Possible Solution

Split the core config-reloader debounce and trigger-attribution work from the broader plugin source reload and targeted channel reload efforts, keeping each behavior in its owning gateway/plugin/channel surface.

## Reproduction Assessment

Yes for the config watcher portion: source and tests show a 300 ms default debounce and pathless watcher scheduling. Unclear for non-config files because current source and docs show the managed reloader watches only the active config path.

## Solution Assessment

No: current main is not the final solution for the operator problem. The safer path is a scoped config-reloader improvement plus separate follow-through on #14438 and #72997 for plugin source and channel reload semantics.
