## Summary

Keep open. Current main still only has a short active-config watcher debounce, drops the filesystem event path before reload evaluation/logging, and does not implement the broader scoped plugin/channel reload behavior requested here.

## What This Changes

The issue requests coalescing rapid watcher events into one reload, lighter scoped reloads for non-config watched files, and reload logs that identify the triggering file.

## Best Possible Solution

Split the work into maintainable pieces: improve source-aware config reload coalescing and trigger logging in the gateway reloader, while handling plugin runtime hot reload and targeted channel refresh through the existing lifecycle discussions in #14438 and #72997.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the config watcher portion by source inspection: watcher events more than the current 300 ms debounce apart can schedule separate reloads, and the event path is discarded before logging. The broader non-config scoped reload portion is only partially reproducible from current main because this reloader watches only the active config path and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

Is this the best way to solve the issue? No, current main is not the best final solution for the reported operator problem. The safer direction is a narrow config coalescing/log-attribution change plus separate maintainer-scoped decisions for plugin runtime and channel reload granularity.
