## Summary

Keep open: current main has config reload debouncing, but the default remains 300 ms, watcher scheduling does not retain the triggering path, and the broader scoped non-config/plugin/channel reload behavior is still unresolved in related open follow-ups.

## What This Changes

Request debounce and trigger attribution for rapid gateway watcher reloads, plus more scoped reloads for non-config/plugin/channel edits.

## Best Possible Solution

Split the narrow gateway config reloader coalescing and trigger-attribution work from the broader plugin hot-reload and targeted channel reload follow-ups tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher slice by source inspection: default debounce is 300 ms and watcher callbacks do not carry the changed path. The non-config/plugin-file slice is unclear because the managed gateway reloader watches the active config file and plugin source/install changes are restart-backed.

## Solution Assessment

Unclear: the requested direction is plausible, but combining longer debounce defaults, path logging, and scoped plugin/channel reloads is not the narrowest maintainable fix without maintainer scoping. The safer path is to split config reloader work from the existing #14438 and #72997 follow-ups.
