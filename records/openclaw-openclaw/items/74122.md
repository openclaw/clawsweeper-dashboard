## Summary

Keep open: current main only debounces config watcher events inside the existing 300 ms default window, watcher callbacks do not retain the triggering filesystem path, and the scoped plugin/channel reload part still overlaps broader open reload work.

## What This Changes

Request to coalesce rapid gateway watcher changes, avoid full plugin/channel reloads for non-config changes, and log the file or reason that triggered reloads.

## Best Possible Solution

Split narrow config reloader coalescing and trigger-attribution work from the broader plugin hot-reload and targeted channel reload design tracked by #14438 and #72997.

## Reproduction Assessment

Yes for the config watcher slice by source inspection: current main has a 300 ms default debounce and pathless watcher callbacks. The broader non-config/plugin-file slice is only partly reproducible from source because the managed gateway watcher is config-path scoped while plugin source/install changes are restart-backed.

## Solution Assessment

Unclear: the requested direction is plausible, but combining longer default debounce, path logging, and scoped plugin/channel reload is not the narrowest maintainable change. The safer solution is to split the config reloader work from the broader reload feature issues.
