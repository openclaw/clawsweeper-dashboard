## Summary

Keep open. Current main still has only a short 300 ms config-reload debounce, drops the watcher event path before reload evaluation/logging, watches the active config file rather than plugin/bootstrap runtime files, and the broader plugin/channel lifecycle work remains represented by open related items.

## What This Changes

Request coalescing rapid watcher events into one reload, logging the file that triggered reload, and avoiding full plugin/channel reloads for non-config watched-file changes.

## Best Possible Solution

Split the work into maintainable pieces: improve source-aware config reload coalescing and trigger attribution in the gateway reloader, then handle plugin runtime hot reload and targeted channel reload through the existing lifecycle discussions in #14438 and #72997.

## Reproduction Assessment

Yes for the config-reload portion: source and tests show watcher events outside the 300 ms debounce window can produce separate reload evaluations, and the event path is discarded. The non-config plugin/bootstrap portion is only partially reproducible from current main because the managed reloader watches the active config path and native plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, current main is not the best final solution for the reported operator problem. It has configurable config debounce and some scoped config reload planning, but it does not provide the longer coalescing/source attribution or plugin runtime refresh behavior requested here.
