## Summary

Keep open: current main still has separate unaddressed watcher surfaces, with only a 300 ms config debounce, no watched-path attribution in config reload logs, no debounce in the dev `gateway:watch` process restarter, and no supported native plugin runtime hot reload.

## What This Changes

Request to coalesce rapid watched-file changes into one reload, add reload trigger visibility, and avoid full plugin/channel restarts for narrower non-config changes.

## Best Possible Solution

Split the narrow watcher debounce and trigger-attribution work from the broader plugin runtime and channel lifecycle follow-ups already represented by #14438 and #72997.

## Reproduction Assessment

Yes for the core watcher behavior: source and tests show 300 ms config debounce with pathless scheduling, plus `gateway:watch` restarts on each relevant event. The exact bootstrap/plugin-dist examples need scoping because not all of those paths are watched by the same reloader.

## Solution Assessment

No current-main behavior is not the best final solution. The maintainable path is a narrow coalescing/logging change for the relevant watcher(s), with separate product-scoped work for native plugin hot reload and targeted channel reload semantics.
