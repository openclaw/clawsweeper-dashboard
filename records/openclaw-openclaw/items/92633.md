## Summary

Keep open: current `main` and the latest release still run the primary memory/session branch before the wiki/supplement branch inside one `memory_search` deadline, and timeout errors still fall through to the generic embedding/provider warning. Live linked PRs now target the same work, so the issue should stay open until maintainers land or reject that implementation path.

## What This Changes

The issue asks OpenClaw to make `memory_search` with `corpus="all"` avoid timing out when individual memory, session, and wiki searches each succeed, return partial results where possible, and stop blaming healthy embedding provider configuration for all-corpus deadline failures.

## Best Possible Solution

Land a focused memory-core fix that starts independent all-corpus branches concurrently, then finish timeout-specific attribution or partial-result semantics before closing this issue.

## Reproduction Assessment

Yes, source-reproducible. Current tests model a stalled wiki supplement under `corpus="all"`, advance the 15s deadline, and assert the same generic unavailable timeout shape, though I did not run the live Windows setup.

## Solution Assessment

Yes for the main timeout cause: overlapping the independent memory/session and wiki/supplement branches is the narrow owner-local fix for additive latency. The warning/debug part should either land through the focused attribution PR after its blocker is fixed or be folded into the same memory-core fix before closure.
