## Summary

Keep open: current main and the latest release still use one 15s tool deadline across primary memory plus wiki/supplement search, and a stalled supplement branch returns the generic embedding/provider unavailable payload instead of partial results. An open PR now targets this exact behavior, but it has not merged.

## What This Changes

The issue requests that `memory_search` with `corpus="all"` return partial results and accurate timeout diagnostics when combined memory/session/wiki search exceeds the tool deadline while individual corpora succeed.

## Best Possible Solution

Land a focused memory-core fix that isolates primary-memory and supplement/wiki phases, returns available partial results with failed-phase debug, and classifies tool-deadline timeouts separately from provider failures.

## Reproduction Assessment

Yes. Current source and tests reproduce the shape by making the wiki supplement never settle under `corpus="all"`, advancing the 15s deadline, and receiving the same generic unavailable payload; I did not execute the live Windows setup in this read-only review.

## Solution Assessment

Yes. Isolating corpus phases and returning partial results with timeout-specific diagnostics is the narrow maintainable fix; simply increasing the global 15s deadline would be a weaker mitigation.
