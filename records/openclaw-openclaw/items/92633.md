## What This Changes

## Best Possible Solution

Review and land or close the two linked PRs as the canonical path, with the concurrency fix for the timeout cause and a corrected attribution/debug fix that does not mask provider-originated timeouts.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 run memory before supplement search under one 15s deadline, and existing tests show all-corpus stalls return the reported timeout shape. I did not run a live Windows/OpenAI-compatible backend repro in this read-only review.

## Solution Assessment

Yes for the active direction, but only as a split fix: the narrow concurrency PR is the right owner-local repair for additive latency, while attribution/debug needs the separate linked PR after its classifier issue is resolved. Partial-result timeout semantics should be explicitly accepted, deferred, or scoped into a follow-up rather than assumed.
