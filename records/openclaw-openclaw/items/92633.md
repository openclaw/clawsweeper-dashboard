## Summary

Keep open: current main and the latest release still use one `memory_search` tool deadline across the primary memory/session branch plus wiki/supplement aggregation, and timeout failures still flow through the generic embedding/provider warning. Two open PRs have closing references to this issue, so the issue should stay open until maintainers land or close those fix paths.

## What This Changes

The issue reports that `memory_search` with `corpus="all"` times out after 15s while `memory`, `sessions`, `wiki`, and the CLI search succeed individually, and asks for branch-isolated timeout handling plus accurate diagnostics.

## Best Possible Solution

Land a focused memory-core fix that runs independent memory/session and wiki/supplement work within the deadline, preserves cooldown semantics, returns safe partial results where possible, and reports timeout attribution by branch.

## Reproduction Assessment

Yes. Current source and existing tests provide a high-confidence source reproduction: a never-settling wiki supplement under `corpus="all"` advances the 15s deadline and returns the same generic embedding/provider unavailable payload; I did not run the reporter's live Windows setup.

## Solution Assessment

Yes. The best fix is in memory-core's all-corpus aggregation path, with branch-level concurrency or deadlines plus accurate timeout attribution; merely increasing the global 15s deadline would be a weaker mitigation.
