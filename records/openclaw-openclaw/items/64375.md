## Summary

Keep open. Current main still has the at-only deletion guard, and this PR is a narrow, directionally correct fix for the linked cron bug; it needs the required changelog entry before merge.

## What This Changes

The PR removes the at-schedule guard from cron deleteAfterRun handling and adds regression coverage for successful every/cron deletion plus recurring error retention.

## Best Possible Solution

Land this PR or an equivalent minimal guard removal after adding a single-line Unreleased Fixes changelog entry; track any stronger cron audit/tombstone behavior as a separate cross-schedule hardening follow-up.

## Reproduction Assessment

Yes. On current main, a successful every or cron job with deleteAfterRun true cannot return shouldDelete because applyJobResult requires schedule.kind === "at".

## Solution Assessment

Yes, with the changelog process fix. Removing only the schedule-kind guard is the narrowest maintainable behavior change because the timer and manual-run callers already delete and persist when shouldDelete is true, while errors and skipped runs stay retained.
