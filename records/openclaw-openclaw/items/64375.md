## What This Changes

The PR removes the schedule-kind guard from cron result deletion and adds regression tests for successful `every`, successful `cron`, and failed recurring jobs with `deleteAfterRun`.

## Best Possible Solution

Refresh the new tests to current `requestHeartbeat`, add redacted real cron-run proof, then land the minimal guard removal with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard makes `shouldDelete` false so the stored job is not removed.

## Solution Assessment

Yes after the fixture refresh and real proof. Removing only the schedule-kind guard matches the current common-field contract; durable audit or tombstone changes would be broader cron storage policy work for all schedule kinds.
