## What This Changes

The PR removes the at-only `deleteAfterRun` deletion guard in cron result handling and adds regression tests for successful `every`/`cron` self-deletion plus error retention.

## Best Possible Solution

Land the minimal guard removal with refreshed heartbeat fixtures and focused recurring success/error regression coverage once maintainers accept the recurring self-delete semantics.

## Reproduction Assessment

Yes, at source level. A successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the current at-only guard makes `shouldDelete` false and leaves the stored job in place.

## Solution Assessment

Mostly yes. Removing the schedule-kind guard is the narrow code fix if the common `deleteAfterRun` field contract is accepted, but the branch needs a fixture refresh and real behavior proof before merge.
