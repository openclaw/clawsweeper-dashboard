## Summary

Keep open: current main still has the recurring `deleteAfterRun` bug, and this PR is the narrow canonical fix path, but the external PR still needs after-fix real behavior proof before merge.

## What This Changes

The PR removes the `kind === "at"` restriction from cron delete-after-run handling and adds regression tests for successful `every`/`cron` deletion plus recurring error retention.

## Best Possible Solution

Land this PR or an equivalent minimal guard removal with focused regression coverage, while tracking broader cron audit/tombstone or session-cleanup hardening separately.

## Reproduction Assessment

Yes. Source inspection on current main shows a successful `every` or `cron` job with `deleteAfterRun: true` reaches `applyJobResult`, but the at-only guard keeps `shouldDelete` false so the stored job is not removed.

## Solution Assessment

Yes, once refreshed and proven. Removing only the schedule-kind guard is the narrowest maintainable fix because `deleteAfterRun` is already accepted as a common field and the timer/manual callers already handle deletion and persistence when `shouldDelete` is true.
