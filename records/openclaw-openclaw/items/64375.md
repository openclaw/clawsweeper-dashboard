The PR removes the at-only `deleteAfterRun` guard from cron result handling and adds regression tests for successful `every`/`cron` deletion plus recurring error retention.

## Best Possible Solution

## Solution Assessment

Mostly yes. Removing the schedule-kind guard is the narrow code fix if the common-field contract is accepted, but the PR needs a test refresh, real behavior proof, and maintainer acceptance of the upgrade behavior before merge.
