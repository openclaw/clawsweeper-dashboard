## Summary

Keep this PR open. Current main still ignores `deleteAfterRun` for successful recurring cron jobs because the deletion guard is `at`-only, and this PR is a narrow implementation candidate for the linked open bug. The code change looks functionally correct, but it needs the required changelog entry before merge.

## What This Changes

The PR removes the `at`-schedule guard from cron `applyJobResult` deletion and adds regression tests for successful `every`/`cron` deletion plus error retention.

## Best Possible Solution

Land this PR or an equivalent minimal guard change after adding the changelog entry and validating the current branch. If maintainers want stronger audit, tombstone, or run-log guarantees for self-deleting jobs, track that as a separate cross-schedule hardening follow-up instead of expanding this focused fix.

## Reproduction Assessment

Yes. On current main, a job with `schedule.kind` of `every` or `cron`, `deleteAfterRun: true`, and an `ok` result cannot return `shouldDelete` because `applyJobResult` requires `schedule.kind === "at"`.

## Solution Assessment

Yes, with one process fix. Removing only the schedule-kind guard is the narrowest code change for the reported behavior because existing timer and manual callers already remove and persist on `shouldDelete`, while failures and skipped runs remain retained.
