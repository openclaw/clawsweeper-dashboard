## Summary

Keep open: current main still shows a plausible conflict between injected MEMORY.md guidance and the TTS system hint, and maintainer follow-up is still waiting on gateway truncation logs and active TTS config before changing prompt policy.

## What This Changes

The issue asks OpenClaw to keep MEMORY.md TTS instructions authoritative over default model-driven TTS directives during Telegram voice sessions.

## Best Possible Solution

Define TTS directive precedence, then make the TTS prompt/config path honor the intended no-directives mode or document the local Piper setup through messages.tts instead of relying on MEMORY.md alone.

## Reproduction Assessment

No, not with high confidence. Source inspection shows MEMORY.md is injected and refreshed while the enabled TTS hint still advertises [[tts:...]], but there is no current-main Telegram/Kimi transcript or gateway log proving where the fallback starts.

## Solution Assessment

Unclear. A broad rule that arbitrary MEMORY.md text overrides runtime TTS policy is risky; the narrower maintainable path is to confirm policy and make the existing messages.tts.modelOverrides.enabled=false behavior align with the injected TTS hint if that is the desired mode.
