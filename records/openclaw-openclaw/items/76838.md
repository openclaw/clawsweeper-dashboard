## Summary

Keep open: current main shows a real prompt-policy tension rather than a cleanup-close case. `MEMORY.md` is injected, but the TTS voice section is also injected later and explicitly tells the model to use `[[tts:...]]` directives when TTS is enabled by default policy.

## What This Changes

The issue asks OpenClaw agents to keep following `MEMORY.md` voice/TTS instructions across a Telegram session instead of later emitting `[[tts:...]]` directive tags.

## Best Possible Solution

Define the intended precedence explicitly: either make TTS prompt generation honor an explicit no-directives policy, or document that local Piper-style flows should be configured through `messages.tts`/`tts-local-cli` with `modelOverrides.enabled=false` instead of relying on `MEMORY.md`.

## Reproduction Assessment

No high-confidence runtime reproduction is available. Source inspection does show a plausible current-main conflict because `MEMORY.md` is injected while the later Voice section can still encourage `[[tts:...]]` directives, but the issue lacks a deterministic Telegram/Kimi transcript or config proof.

## Solution Assessment

Unclear. Treating `MEMORY.md` as an override for the TTS system hint might solve this report, but the narrower maintainable path is an explicit TTS precedence/config decision and matching docs/tests.
