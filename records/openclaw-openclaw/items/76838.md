## Summary

Keep open: current main still has a plausible prompt-policy conflict, because workspace `MEMORY.md` is injected but the enabled TTS hint later tells the model to use `[[tts:...]]` directives, and the latest human follow-up asks for truncation/session diagnostics before changing behavior.

## What This Changes

The issue asks OpenClaw to keep honoring `MEMORY.md` instructions that forbid `[[tts:...]]` directives and require local Piper voice-file replies throughout a Telegram voice session.

## Best Possible Solution

Define TTS directive precedence in code/docs and, if approved, make the TTS hint avoid advertising directives when model directives are disabled.

## Reproduction Assessment

No. Source inspection shows `MEMORY.md` injection plus a later TTS directive hint, but no gateway logs, `/context` output, active config, or current-main Telegram/Kimi transcript proves where the reported fallback begins.

## Solution Assessment

Unclear. Treating arbitrary `MEMORY.md` text as authority over runtime TTS policy is fragile; the narrower maintainable path is config-driven precedence around `messages.tts.modelOverrides.enabled` if maintainers approve it.
