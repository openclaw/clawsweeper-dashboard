Keep open: current main and the latest shipped release still have a plausible conflict between workspace `MEMORY.md` guidance and the TTS runtime hint, and the issue discussion is waiting on gateway truncation logs, `/context`, active `messages.tts` config, and a transcript before choosing a policy fix.

## What This Changes

The issue requests that `MEMORY.md` instructions forbidding model TTS directives be followed consistently during Telegram voice interactions instead of falling back to `[[tts:...]]` commands.

## Best Possible Solution

Define TTS directive precedence explicitly, then make the existing `messages.tts.modelOverrides.enabled=false` path suppress directive prompting if that is the intended no-directives contract, while using logs and `/context` to rule out bootstrap truncation.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows `MEMORY.md` is loaded and refreshed while the TTS system hint still advertises directives after Project Context, but the report lacks the session logs, `/context`, config, and transcript needed to reproduce the fallback on current main.

## Solution Assessment

Unclear. A config-driven fix that makes the TTS prompt honor `messages.tts.modelOverrides.enabled=false` looks narrower than making arbitrary `MEMORY.md` text override runtime TTS policy, but maintainers still need to confirm that intended precedence.
