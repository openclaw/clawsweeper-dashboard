## Summary

Keep open: current main still shows a plausible prompt-policy conflict rather than a cleanup-close case. `MEMORY.md` is loaded and refreshed, but the TTS hint is appended after Project Context and still tells the model to use `[[tts:...]]`, while the thread is waiting on truncation logs, `/context`, active TTS config, and a transcript.

## What This Changes

The issue requests that agent sessions consistently honor `MEMORY.md` instructions that forbid model TTS directives and require local Piper audio for Telegram voice replies.

## Best Possible Solution

Define TTS directive precedence explicitly, then either make the TTS hint honor the existing no-model-directives config path or document local Piper setup through `messages.tts` rather than relying on `MEMORY.md` to override runtime guidance.

## Reproduction Assessment

No, we do not have a high-confidence end-to-end reproduction. Source inspection shows a real conflict path between injected `MEMORY.md` and the later TTS hint, but the reporter has not provided logs, `/context`, config, or a transcript showing the switch on current main.

## Solution Assessment

Unclear. Making arbitrary `MEMORY.md` text override runtime TTS policy is not obviously the best fix; the safer maintainable path is a maintainer-approved precedence rule, likely using the existing `messages.tts.modelOverrides.enabled=false` behavior.
