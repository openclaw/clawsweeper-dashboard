## Summary

Keep open: current main and the latest shipped release still rethrow non-Windows failures from the exec-approvals directory chmod path, while the last known good release did not chmod that directory. The broad configurable-permissions issue is related, but this issue has a narrower regression path that can be fixed in exec approvals without waiting for a full permissions policy.

## What This Changes

Fix exec approval persistence so Kubernetes fsGroup PVC mounts where the state directory is not owned by the container user do not make every exec tool call fail on directory chmod.

## Best Possible Solution

Make only the exec-approvals directory chmod tolerant of expected permission/unsupported filesystem errnos while preserving successful owner-only hardening, path safety checks, final file mode 0600, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.5.12` rethrow `EPERM` from the approvals-directory chmod before the exec approvals write can complete, and the reporter provided Kubernetes fsGroup logs for the same path; I did not run a live Kubernetes pod.

## Solution Assessment

Yes, with a narrower shape than the broad permissions-policy work. The best fix is a targeted errno-tolerant or ownership-aware chmod hardening path in exec approvals, not a global chmod skip or waiting for https://github.com/openclaw/openclaw/issues/56263.
