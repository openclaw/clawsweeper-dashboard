## Summary

Keep open: current main still has the reported non-Windows `chmodSync` rethrow in exec approvals, and the latest shipped release contains the same behavior while the last known good tag did not. The report is concrete, source-reproducible, and narrow enough for a focused fix.

## What This Changes

The issue reports that exec tool calls fail on Kubernetes fsGroup-mounted PVCs because exec approval persistence chmods the state-directory root and rethrows `EPERM`.

## Best Possible Solution

Make exec approvals directory hardening best-effort for known chmod permission/filesystem errors, keep destination and symlink guards intact, and add focused regression coverage for the fsGroup/EPERM case.

## Reproduction Assessment

Yes, from source inspection and reporter logs, though I did not run a live Kubernetes pod. Current main calls `chmodSync` on the approvals directory and rethrows non-Windows `EPERM` before the exec approval write path can continue.

## Solution Assessment

Yes, the best fix is a narrow change in `src/infra/exec-approvals.ts` that treats expected chmod permission/filesystem errors as best-effort hardening failures while retaining the existing directory, symlink, hard-link, and file permission checks.
