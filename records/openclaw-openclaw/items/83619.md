## What This Changes

The issue reports a 2026.5.12 regression where exec approvals persistence fails with chmod EPERM on fsGroup-mounted Kubernetes state directories, blocking every exec tool call before the command runs.

## Best Possible Solution

Make exec approvals directory hardening best-effort only for expected managed-volume chmod permission/filesystem errors, keep the existing approval-store safety checks intact, and add focused regression coverage for non-Windows EPERM on the directory chmod path.

## Reproduction Assessment

Yes, at source level. Current main still chmods the exec approvals directory and rethrows non-Windows EPERM before the approvals write can proceed, though I did not run a live Kubernetes PVC pod in this read-only review.

## Solution Assessment

Yes. A narrow errno-tolerant chmod hardening path in src/infra/exec-approvals.ts is the maintainable fix, provided the existing symlink, hard-link, temp-file, and file-mode protections remain enforced.
