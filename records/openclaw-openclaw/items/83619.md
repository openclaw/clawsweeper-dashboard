## Summary

Keep open: the reported regression is still present on current main, and an open linked fix PR uses closing syntax but has not merged. Current `ensureDir` still rethrows non-Windows chmod failures before exec approval persistence can continue, matching the Kubernetes fsGroup PVC failure mode described in the report.

## What This Changes

The issue reports that Kubernetes deployments with an fsGroup-mounted PVC fail every exec tool call because exec approvals directory hardening rethrows `EPERM` from `chmod` on `/home/openclaw/.openclaw`.

## Best Possible Solution

Land a focused fix that keeps symlink, hard-link, ownership, temp-file, and final file-mode protections intact while tolerating expected chmod permission/filesystem errors when the path remains accessible.

## Reproduction Assessment

Yes. Source inspection on current main shows the exact non-Windows `chmodSync` rethrow path, and the reporter provided a concrete Kubernetes fsGroup PVC setup where that syscall returns `EPERM`.

## Solution Assessment

Yes. The maintainable fix is to make this chmod hardening best-effort for expected managed-volume errors while retaining the existing directory and approval-file safety checks.
