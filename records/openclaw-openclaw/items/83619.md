## Label Justifications

- `P1`: The issue reports a shipped regression that blocks every exec tool call for affected Kubernetes deployments.
- `impact:security`: The failing path is filesystem permission hardening around the exec approvals store and the workaround would broaden container capabilities.
- `impact:crash-loop`: The failure makes the exec tool unavailable on every invocation in the affected runtime, causing an availability failure for agent shell workflows.

## Summary

Keep open: current main and the latest shipped release still rethrow non-Windows chmod failures in exec approvals, and the linked fix PR is open but not merged.

## What This Changes

Make exec approvals persistence tolerate Kubernetes fsGroup PVC chmod EPERM instead of failing every exec tool call.

## Best Possible Solution

Land a narrow security-reviewed fix, likely through https://github.com/openclaw/openclaw/pull/83695, that makes managed-volume chmod failures non-fatal while preserving exec approval store safety and adding regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main writes exec approvals through `ensureDir`, which rethrows non-Windows EPERM from `chmodSync` before the exec path can proceed. I did not run a live Kubernetes PVC pod in this read-only review.

## Solution Assessment

Yes, the maintainable direction is a narrow chmod-hardening fix in the approvals/state permission path. The linked PR is the right implementation vehicle, but it still needs security-safe review before this issue can close.
