## What This Changes

The issue reports that exec tool calls fail on Kubernetes fsGroup PVC mounts because exec approval persistence hard-fails when `chmod` on the OpenClaw state directory returns `EPERM`.

## Best Possible Solution

Land `https://github.com/openclaw/openclaw/pull/83695` or an equivalent narrow shared chmod-tolerance fix that verifies access after expected permission/filesystem errors while preserving the existing approval-store safety guards.

## Reproduction Assessment

Yes. Source inspection shows current `main` still rethrows non-Windows `EPERM` from the approvals directory chmod before approval writes continue, matching the reporter's Kubernetes fsGroup logs.

## Solution Assessment

Yes. The narrow maintainable fix is to make expected managed-volume chmod failures best-effort after verifying access, while keeping the existing directory, symlink, hard-link, temp-file, and file-mode checks intact.
