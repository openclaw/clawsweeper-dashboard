## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Target the current SQLite auth-profile store:** `src/agents/auth-profiles/paths.ts:32`
  - body: This adds sealed `auth-profiles.json` writes, but current main persists auth-profile secrets through the per-agent SQLite `auth_profile_store` table. Keeping a new JSON writer would miss the runtime path current users exercise after upgrade and create a competing credential source.
  - confidence: 0.9
- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: `saveSealedJsonFile` writes plaintext whenever `OPENCLAW_PASSPHRASE` is absent. Because the load path treats a sealed store without the passphrase as missing and later persistence can still run, this can overwrite an encrypted auth store with plaintext credentials instead of failing closed.
  - confidence: 0.95
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still reads the main auth store through `loadJsonFile`, so a main store written in the new sealed format is parsed as raw JSON and inheritance silently fails for subagents when a passphrase is configured.
  - confidence: 0.92
- **[P2] Record explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed format derives the AES key with Node's implicit `scryptSync` defaults and does not store KDF metadata, making the credential format harder to audit and upgrade safely.
  - confidence: 0.84
- **[P2] Write sealed credential files atomically with private mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: The writer creates or truncates the credential file and only then applies `chmod(0o600)`. Credential persistence should use the repo's atomic/private replace path so there is no plaintext permission window, partial-write loss, or symlink-following clobber path.
  - confidence: 0.86
