## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Block plaintext rewrites of sealed stores:** `src/agents/auth-profiles/store.ts:438`
  - body: When `loadCoercedStore` returns `null` because `auth-profiles.json` is sealed but `OPENCLAW_PASSPHRASE` is missing, this write path can still run after legacy/OAuth/CLI sync and `saveSealedJsonFile` will persist plaintext. That breaks the PR's fail-closed claim and can leak or replace previously sealed credentials.
  - confidence: 0.94
- **[P1] Preserve expectFinal timeout semantics:** `src/gateway/client.ts:526-530`
  - body: This timer is scheduled for every request, including `expectFinal` calls. If the gateway sends the expected `accepted` response and the final result takes longer than the default, the pending request now rejects even though current main intentionally leaves `expectFinal` unbounded unless the caller passes a timeout.
  - confidence: 0.91
- **[P2] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:416`
  - body: The subagent inheritance path still reads the main auth store with `loadJsonFile`, so once the main store is sealed it fails to parse and the subagent silently skips inherited credentials. Read the main store through the sealed-aware loader before cloning it.
  - confidence: 0.9
- **[P2] Persist KDF metadata with sealed files:** `src/infra/sealed-json-file.ts:39`
  - body: The new envelope records the cipher but not the scrypt parameters, while key derivation relies on Node's implicit defaults. Store explicit KDF parameters in the envelope and validate them on read so the format remains auditable and upgradeable.
  - confidence: 0.82
- **[P2] Write sealed stores with restrictive atomic semantics:** `src/infra/sealed-json-file.ts:113-114`
  - body: Direct `writeFileSync` followed by `chmodSync` creates a permissions window and follows existing symlinks for secret material. Use the repo's safe JSON write pattern or an atomic `0600` no-follow write path before making this a credential-store primitive.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes credential storage and introduces concrete fail-closed, KDF metadata, and restrictive-write issues.

Concerns:

- **[high] Sealed credentials can be downgraded to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: A sealed auth store that cannot be read without `OPENCLAW_PASSPHRASE` can still be overwritten by a later store write, and the new writer emits plaintext when the passphrase is absent.
  - confidence: 0.94
- **[medium] KDF parameters are implicit and not serialized:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed format uses `scryptSync(passphrase, salt, 32)` without explicit cost parameters or envelope metadata, making security posture and future compatibility harder to audit.
  - confidence: 0.82
- **[medium] Secret-store writes are not restrictive before content hits disk:** `src/infra/sealed-json-file.ts:113`
  - body: The writer uses direct `writeFileSync` and only calls `chmodSync` afterward, which can expose a permission window and can follow an existing filesystem link target.
  - confidence: 0.86
