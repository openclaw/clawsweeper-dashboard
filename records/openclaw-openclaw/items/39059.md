Full review comments:

- **[P1] Target the current SQLite auth-profile store:** `src/agents/auth-profiles/store.ts:526`
  - body: This branch seals auth-profiles.json, but current main persists auth profiles through the per-agent SQLite auth_profile_store row. A refresh that keeps this write target would either conflict or leave the canonical credential store unsealed.
  - confidence: 0.94
- **[P1] Prevent sealed-store downgrade without a passphrase:** `src/infra/sealed-json-file.ts:113`
  - body: saveSealedJsonFile falls back to plaintext whenever OPENCLAW_PASSPHRASE is absent. If an existing sealed auth store is loaded without the env var and a migration or external-CLI sync writes, this can replace encrypted credentials with plaintext instead of failing closed.
  - confidence: 0.95
- **[P1] Preserve expectFinal request waits:** `src/gateway/client.ts:520-524`
  - body: The new default timeout applies even when expectFinal is true, but current main deliberately lets expectFinal requests wait indefinitely unless a caller passes a timeout. This can break long-running agent/gateway calls that only receive an accepted ack before the final result.
  - confidence: 0.9
- **[P1] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:412-413`
  - body: The subagent inheritance path still reads the main store through loadJsonFile, so once the main auth-profiles.json is sealed, subagents cannot inherit it through this path. The read path needs to use the same sealed reader or be replaced by the current SQLite inheritance path.
  - confidence: 0.9
- **[P2] Record KDF parameters in the sealed envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The envelope stores algorithm, salt, IV, tag, and ciphertext but not the scrypt parameters used to derive the key. Without per-file KDF metadata, future tuning can create compatibility or audit ambiguity for credential files users need to keep decryptable.
  - confidence: 0.84
- **[P2] Write sealed stores atomically with private mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: The writer uses writeFileSync and then chmodSync, which leaves no atomic replace path and can briefly create a secret-bearing file before permissions are reasserted. Credential persistence should use the existing atomic/private write helper or an equivalent no-follow temp-and-rename path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes credential-at-rest handling and introduces concrete downgrade, write-safety, and key-derivation design concerns before merge.

Concerns:

- **[medium] Sealed auth stores can downgrade to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: saveSealedJsonFile writes plaintext when OPENCLAW_PASSPHRASE is absent, so an existing encrypted auth store can be overwritten in plaintext during later sync or migration writes.
  - confidence: 0.95
- **[medium] Credential file writes are not atomic/private at creation:** `src/infra/sealed-json-file.ts:113`
  - body: The sealed writer uses writeFileSync followed by chmodSync instead of creating a private temp file and atomically replacing the destination, leaving avoidable permission-window and durability risk for secret-bearing files.
  - confidence: 0.86
- **[low] KDF metadata is absent from the envelope:** `src/infra/sealed-json-file.ts:38`
  - body: The sealed format does not store scrypt parameters with the encrypted payload, making future tuning and compatibility harder to reason about for long-lived credential stores.
  - confidence: 0.84

  - body: Current main stores auth-profile secrets in SQLite, so sealing only auth-profiles.json does not protect the canonical auth-profile persistence path after a refresh.
