## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rework sealing around the SQLite auth store:** `src/agents/auth-profiles/paths.ts:4`
  - body: Current main now reads and writes auth profiles from each agent's `openclaw-agent.sqlite`, while this patch seals `auth-profiles.json` and legacy `oauth.json` runtime paths. Resolving the conflicts by carrying this JSON writer forward would miss the canonical credential store and conflict with the doctor-owned legacy import path.
  - confidence: 0.91
- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:435`
  - body: When a sealed `auth-profiles.json` cannot be read because `OPENCLAW_PASSPHRASE` is absent, this path can still decide to persist migrations/OAuth/CLI sync and call `saveSealedJsonFile` without a passphrase. That overwrites a sealed credential store with plaintext instead of failing closed.
  - confidence: 0.9
- **[P2] Use safe replacement for sealed credential writes:** `src/infra/sealed-json-file.ts:111`
  - body: `saveSealedJsonFile` writes directly to the destination and only chmods afterward. For credential material, this should use the repository's safe replace pattern or equivalent restrictive temp-file semantics so failures, symlinks, or chmod windows do not weaken the store.
  - confidence: 0.82
- **[P2] Store explicit KDF metadata in the envelope:** `src/infra/sealed-json-file.ts:38`
  - body: The sealed format derives the AES key with implicit `scryptSync` defaults and does not store KDF parameters in the envelope. That makes the format harder to audit or tune later without risking decryptability drift.
  - confidence: 0.76
