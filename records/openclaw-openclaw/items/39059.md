- **[P1] Block plaintext rewrites of sealed stores:** `src/agents/auth-profiles/store.ts:436-438`
  - body: When a sealed auth store exists but OPENCLAW_PASSPHRASE is missing, loadProtectedAuthJson returns undefined and this path can still merge OAuth/CLI credentials and call saveSealedJsonFile. Because the writer falls back to plaintext without a passphrase, a protected credential store can be silently downgraded and partially overwritten.
  - confidence: 0.95

  - body: Current main already leaves expectFinal requests unbounded by default, supports timeoutMs: null, and handles AbortSignal cancellation. This stale implementation applies a default timeout to expectFinal calls and lacks the current cancellation contract, so rebasing it would regress long-running gateway requests.
  - confidence: 0.9
- **[P2] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:412`
  - body: This fallback still reads the main auth-profiles.json with loadJsonFile, but the PR can now write that file as a sealed envelope. With OPENCLAW_PASSPHRASE enabled, subagents will fail to parse the main store and silently skip credential inheritance.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces credential-at-rest sealing, but the current implementation has concrete downgrade, key-derivation metadata, and sensitive-file persistence hardening problems.

Concerns:

- **[high] Prevent sealed-store plaintext downgrade:** `src/agents/auth-profiles/store.ts:438`
  - body: A missing passphrase can make the loader treat a sealed store as absent, after which a migration or sync path can write plaintext over the sealed credential file.
  - confidence: 0.95
- **[medium] Record stable KDF metadata:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed envelope records algorithm, salt, IV, tag, and ciphertext, but not explicit KDF parameters, which makes future tuning and compatibility auditing unsafe for a credential format.
