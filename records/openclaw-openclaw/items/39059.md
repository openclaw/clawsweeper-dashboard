## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent plaintext downgrade when the passphrase is absent:** `src/agents/auth-profiles/store.ts:438`
  - body: When an existing `auth-profiles.json` is sealed and `OPENCLAW_PASSPHRASE` is missing, `loadCoercedStore` returns `null`; this fallback can then merge legacy OAuth or external CLI credentials and call `saveSealedJsonFile` without a passphrase. That writes plaintext over the sealed file, so the advertised fail-closed behavior becomes a credential downgrade and possible data loss.
  - confidence: 0.91
- **[P1] Read inherited stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412`
  - body: This PR allows the main auth store to be written with the sealed prefix, but the subagent inheritance path still loads that main store with `loadJsonFile`. With `OPENCLAW_PASSPHRASE` enabled, raw JSON parsing fails and subagents silently stop inheriting credentials from the main agent.
  - confidence: 0.86
- **[P2] Use atomic private writes for sealed auth files:** `src/infra/sealed-json-file.ts:113-114`
  - body: The new credential writer opens the destination with plain `writeFileSync` and only applies `chmod(0o600)` afterward. For secret-bearing auth files this bypasses the existing fs-safe atomic/private write helpers, leaving a permissions window and weaker crash/TOCTOU behavior than the current credential-store hardening pattern.
  - confidence: 0.86
- **[P2] Store explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The v1 sealed file format derives keys with `scryptSync` but records no KDF parameters in the envelope. If this format ships, future tuning or compatibility checks cannot tell which work factor protected an existing file, so the format should include validated KDF metadata before becoming a credential-storage contract.
  - confidence: 0.78
