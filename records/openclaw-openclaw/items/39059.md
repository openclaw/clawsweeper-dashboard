## Summary

Keep open for maintainer/security handling: the PR targets real gateway/auth hardening work, but the branch is not merge-ready because the sealed auth path can downgrade encrypted credentials to plaintext, subagent auth inheritance still uses a raw JSON reader, and current main has since moved these surfaces to package gateway client and SQLite-backed auth storage.

## What This Changes

The branch adds GatewayClient request timeouts, optional OPENCLAW_PASSPHRASE sealing for JSON auth-profile/OAuth stores, transcript chmod hardening, tests, and a changelog entry.

## Best Possible Solution

Rework the credential hardening against current SQLite auth-profile storage with no silent downgrade path, safe atomic/permission semantics, explicit KDF metadata, and upgrade tests; keep the already-current gateway timeout behavior in the shared gateway-client package.

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth-profiles.json exists but OPENCLAW_PASSPHRASE is missing, loadCoercedStore returns null and this branch can still persist legacy/OAuth/CLI sync results through saveSealedJsonFile. With no passphrase, that writer emits plaintext, so an encrypted credential store can be silently replaced by plaintext instead of failing closed.
  - confidence: 0.93
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still calls loadJsonFile on the main auth store. After this PR, that main store may be written with the sealed prefix, so loadJsonFile fails to parse it and subagents silently stop inheriting credentials whenever OPENCLAW_PASSPHRASE is enabled.
  - confidence: 0.88
- **[P2] Write sealed credential files without chmod-after-write:** `src/infra/sealed-json-file.ts:113-114`
  - body: saveSealedJsonFile writes credential material to the destination and only then applies chmod. For a security-sensitive auth store, create the file with restrictive mode and use an atomic replacement path so there is no permissive creation window or direct target clobber path.
  - confidence: 0.82
