## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Target credential sealing at the SQLite store:** `src/agents/auth-profiles/paths.ts:32`
  - body: This new path seals auth-profiles.json, but current main stores active auth-profile secrets in each agent's openclaw-agent.sqlite auth_profile_store row. Merging this would harden a legacy import file while leaving the real credential store unsealed, so the encryption design needs to move to the SQLite persistence layer or a documented migration boundary.
  - confidence: 0.95
- **[P1] Block plaintext downgrades when the passphrase is absent:** `src/agents/auth-profiles/store.ts:438`
  - body: If a sealed store cannot be read because OPENCLAW_PASSPHRASE is missing, this migration/sync path can still decide to write and call saveSealedJsonFile without a passphrase. That helper then writes plaintext, so an operator who forgets the env var can overwrite sealed credentials with plaintext or partial replacement data.
  - confidence: 0.94
- **[P1] Use the sealed reader for inherited main auth:** `src/agents/auth-profiles/store.ts:412`
  - body: Subagent inheritance still reads the main store with loadJsonFile, which cannot parse the sealed envelope this PR writes. With OPENCLAW_PASSPHRASE set, subagents can silently fail to inherit credentials from the main store unless this path uses the same protected reader as the rest of the sealed auth load flow.
  - confidence: 0.88
- **[P2] Preserve expectFinal request timeout semantics:** `src/gateway/client.ts:520-524`
  - body: This branch applies a default timeout to every request unless a numeric override is supplied, but current main's packages/gateway-client deliberately leaves expectFinal requests without an automatic timeout by default and supports timeoutMs:null. Reintroducing the stale src/gateway/client.ts version can break long-running accepted/final gateway flows.
  - confidence: 0.86
- **[P2] Write sealed credentials atomically and privately:** `src/infra/sealed-json-file.ts:113-114`
  - body: The new helper writes sensitive payloads with writeFileSync and only then chmods the destination. Credential writes should use the existing atomic/private-at-create-time replacement path so a crash, permission window, symlink/clobber edge, or chmod failure cannot expose or corrupt auth material.
  - confidence: 0.89
- **[P2] Record explicit KDF parameters in sealed files:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed envelope derives the key with scryptSync but does not record explicit KDF parameters in the stored envelope. Persisted credential formats need stable, auditable work factors so future tuning does not create ambiguous or incompatible encrypted files.
  - confidence: 0.82
