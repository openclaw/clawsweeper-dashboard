## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent plaintext overwrite of sealed stores:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth-profiles.json is loaded without OPENCLAW_PASSPHRASE, the reader returns undefined, but this branch can still set shouldWrite from legacy OAuth or external CLI sync and then save plaintext over the sealed file. That violates the advertised fail-closed behavior and can leak or lose credentials.
  - confidence: 0.94
- **[P1] Use the sealed reader for inherited main auth:** `src/agents/auth-profiles/store.ts:412-413`
  - body: This fallback still reads the main auth store through loadJsonFile, so once the main store has been sealed, subagents cannot inherit those credentials. The fallback should use the same sealed-aware load path before deciding no main store exists.
  - confidence: 0.88
- **[P2] Write sealed credential files atomically with restricted mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: saveSealedJsonFile writes credential material with writeFileSync and only chmods afterward, replacing the existing fs-safe JSON writer with a permissions window and no atomic replace behavior. A credential-store hardening path should create/write the replacement with 0600 semantics from the start and avoid downgrading durability.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch changes credential storage and introduces a plaintext downgrade path plus raw credential-file writes that need security owner attention.

Concerns:

- **[high] Sealed auth store can be downgraded to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: The PR catches missing-passphrase reads and later writes through a plaintext fallback, so an encrypted auth store can be overwritten without the passphrase when migration or external CLI sync runs.
  - confidence: 0.94
- **[medium] Credential writer bypasses atomic restricted-mode storage:** `src/infra/sealed-json-file.ts:113`
  - body: The new sealed writer uses writeFileSync followed by chmod rather than an atomic 0600 replacement helper, creating a permissions and durability regression for secrets.
  - confidence: 0.8
- **[medium] KDF parameters are implicit and not recorded:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed envelope derives keys with scryptSync defaults and stores no KDF metadata, making future compatibility and security tuning harder to audit.
  - confidence: 0.72
