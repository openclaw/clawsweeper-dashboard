## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Target the current SQLite auth-profile store:** `src/agents/auth-profiles/paths.ts:32`
  - body: This adds sealed `auth-profiles.json` writes, but current main and v2026.6.9 persist auth-profile secrets through the per-agent SQLite `auth_profile_store` row and explicitly do not read legacy `auth-profiles.json` at runtime. Sealing the stale JSON path would miss the credential store current users exercise and create a competing source of truth.
  - confidence: 0.93
- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: `saveSealedJsonFile` writes plaintext whenever `OPENCLAW_PASSPHRASE` is absent. Because the load path treats a sealed store without the passphrase as missing and later persistence can still run, an encrypted auth store can be overwritten with plaintext credentials instead of failing closed.
  - confidence: 0.95
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412-413`
  - body: The subagent inheritance path still reads the main auth store through `loadJsonFile`, so a main store written in the new sealed format is parsed as raw JSON and inheritance silently fails for subagents when a passphrase is configured.
  - confidence: 0.91
- **[P1] Preserve current expectFinal timeout semantics:** `src/gateway/client.ts:520-524`
  - body: The PR computes a timeout for every request from `opts.timeoutMs ?? this.opts.requestTimeoutMs`, so `expectFinal` calls are capped by default. Current main deliberately leaves `expectFinal` requests unbounded unless the caller opts into a timeout, because accepted/final flows can be long-running.
  - confidence: 0.88
- **[P2] Record explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The sealed format derives the AES key with implicit `scryptSync` parameters and does not store KDF metadata in the envelope, making credential files harder to audit and migrate safely if parameters need to change.
  - confidence: 0.84
- **[P2] Write sealed credential files atomically with private mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: The writer creates or truncates the credential file and only then applies `chmod(0o600)`. Credential persistence should use the repo's atomic/private replace pattern so there is no permission window, partial-write loss, or symlink-following clobber path.
  - confidence: 0.87
