## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Target the current SQLite auth-profile store:** `src/agents/auth-profiles/store.ts:526`
  - body: This branch seals `auth-profiles.json`, but current main persists auth profiles in `openclaw-agent.sqlite` and tests assert the JSON file is not read at runtime. Refresh the design around the SQLite auth-profile store or it will leave current credentials unsealed while adding a stale parallel path.
  - confidence: 0.92
- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: `saveSealedJsonFile` writes plaintext whenever `OPENCLAW_PASSPHRASE` is absent. If a sealed store exists and a migration or external-CLI sync write runs without the env var, this can overwrite encrypted credentials with plaintext instead of failing closed.
  - confidence: 0.94
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:416`
  - body: The subagent inheritance path can now write a sealed store, but it still reads the main store with the plaintext JSON loader just above this line. With `OPENCLAW_PASSPHRASE` enabled, subagents silently miss inherited credentials because the sealed main store cannot be parsed as raw JSON.
  - confidence: 0.88
- **[P1] Preserve current expectFinal timeout semantics:** `src/gateway/client.ts:525-530`
  - body: This implementation applies the default timeout even when `expectFinal` is true. Current main deliberately leaves `expectFinal` requests unbounded unless a caller supplies a timeout, so this stale implementation can break long-running accepted-then-final gateway RPCs.
  - confidence: 0.87
- **[P2] Write sealed credential files atomically and privately:** `src/infra/sealed-json-file.ts:113-114`
  - body: The sealed writer opens the final path with `writeFileSync` and only then applies `chmod(0o600)`. Credential files should use the existing atomic/private write pattern so partial writes, permission windows, and symlink-style clobber risks are not introduced with the encryption primitive.
  - confidence: 0.84
- **[P2] Record explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The new sealed format derives keys with `scryptSync` defaults and stores no KDF metadata. For an auth-store encryption format, the work factors should be explicit and versioned in the envelope so future tuning does not create audit or decryptability ambiguity.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: Needs attention: the PR changes credential confidentiality semantics and introduces concrete downgrade, persistence-hardening, and KDF-design concerns.

Concerns:

- **[high] Sealed credential stores can downgrade to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: A sealed auth store read without `OPENCLAW_PASSPHRASE` can be treated as missing, and the writer then persists plaintext when the passphrase is absent, undermining the claimed fail-closed behavior.
  - confidence: 0.94
- **[medium] Sealed file writes are not atomic or private at creation:** `src/infra/sealed-json-file.ts:113`
  - body: The new credential writer uses direct final-path writes plus chmod-after-write instead of an atomic temp-file replacement with private creation mode, which creates durability and local filesystem attack concerns for auth material.
  - confidence: 0.84
- **[medium] KDF settings are implicit and not versioned:** `src/infra/sealed-json-file.ts:38`
  - body: The sealed JSON envelope omits explicit scrypt work factors, making the credential encryption posture harder to audit and harder to evolve safely across releases.
  - confidence: 0.81
