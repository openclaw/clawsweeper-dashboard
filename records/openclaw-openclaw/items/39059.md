Overall confidence: 0.93

Full review comments:

- **[P1] Target the SQLite auth-profile store:** `src/agents/auth-profiles/paths.ts:29`
  - body: This branch seals `auth-profiles.json`, but current main persists auth profiles in each agent's `openclaw-agent.sqlite` and even tests that legacy `auth-profiles.json` is not read at runtime. The credential hardening would miss the active store and revive a retired path instead of protecting current users.
  - confidence: 0.95
- **[P1] Block plaintext downgrades when passphrase is missing:** `src/agents/auth-profiles/store.ts:435`
  - body: When a sealed auth store exists and `OPENCLAW_PASSPHRASE` is absent, this load path can return an empty/coerced store and later call `saveSealedJsonFile` without a passphrase, overwriting encrypted credentials as plaintext. That contradicts the fail-closed behavior and can leak or lose credentials.
  - confidence: 0.94
- **[P1] Read sealed main auth when inheriting subagent credentials:** `src/agents/auth-profiles/store.ts:413`
  - body: The subagent inheritance block still reads the main auth store with the plain JSON loader while this PR can write that main store in sealed form. With a passphrase-enabled main store, subagents silently fail to inherit credentials instead of using the same protected reader.
  - confidence: 0.88
- **[P2] Preserve expectFinal timeout semantics:** `src/gateway/client.ts:517-523`
  - body: This timeout calculation gives `expectFinal` requests the default timeout unless callers override it, but current main's merged gateway implementation intentionally leaves `expectFinal` unbounded by default for long-running final-response flows. Reintroducing the default cap can make legitimate agent/chat requests fail after the timeout.
  - confidence: 0.9
- **[P2] Write sealed credentials atomically and privately:** `src/infra/sealed-json-file.ts:111-114`
  - body: `saveSealedJsonFile` writes sensitive credential payloads directly and only chmods afterward, which leaves an avoidable permissions window and non-atomic clobber behavior. Use the existing atomic/private write helper or an equivalent temp-file create-with-mode path before claiming auth-store sealing is safe.
  - confidence: 0.89
- **[P2] Store KDF parameters in the sealed envelope:** `src/infra/sealed-json-file.ts:38-40`
  - body: The sealed format derives keys with implicit `scryptSync` defaults but does not record KDF parameters in the envelope. That makes the security posture hard to audit and makes future tuning or compatibility changes risky for persisted credentials.
  - confidence: 0.82
