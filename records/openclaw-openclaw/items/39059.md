## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Target credential sealing at the SQLite store:** `src/agents/auth-profiles/paths.ts:32`
  - body: Current main stores active auth-profile secrets in `openclaw-agent.sqlite`, and tests now assert `auth-profiles.json` is not created or read at runtime. Sealing this legacy JSON path would leave the real credential store unprotected and risks reviving a retired runtime path.
  - confidence: 0.95
- **[P1] Block plaintext downgrades when passphrase is absent:** `src/agents/auth-profiles/store.ts:435`
  - body: When a sealed store cannot be read because `OPENCLAW_PASSPHRASE` is missing, this load path can still fall through to migration/sync writes and call `saveSealedJsonFile` without a passphrase. That helper writes plaintext, so an encrypted store can be overwritten with unsealed credentials.
  - confidence: 0.94
- **[P1] Use the sealed reader for inherited main auth:** `src/agents/auth-profiles/store.ts:413`
  - body: The branch can write sealed auth stores, but the subagent inheritance path still reads the main store through the plain JSON reader just above this save. With a passphrase-enabled main store, subagents can silently fail to inherit credentials.
  - confidence: 0.89
- **[P2] Preserve expectFinal request timeout semantics:** `src/gateway/client.ts:517-522`
  - body: This timeout calculation gives every request a default 30s timer even when `expectFinal` is true. Current main intentionally leaves expectFinal requests without an automatic timeout unless the caller passes `timeoutMs`, because long agent calls can send accepted/interim responses before the final result.
  - confidence: 0.88
- **[P2] Write sealed credentials atomically and privately:** `src/infra/sealed-json-file.ts:113-114`
  - body: `writeFileSync` followed by `chmodSync` creates the sensitive file before restrictive permissions are guaranteed and does not protect the old file from partial writes or clobber paths. Credential writes should use the existing private-at-create-time atomic replace pattern or an equivalent SQLite-layer transaction design.
  - confidence: 0.9
- **[P2] Record explicit KDF parameters in sealed files:** `src/infra/sealed-json-file.ts:38-39`
  - body: The sealed envelope does not record scrypt parameters, and the key derivation call does not pass explicit work factors. Persisted credential formats need auditable parameters in the envelope so future tuning does not break or silently change decryptability.
  - confidence: 0.82
