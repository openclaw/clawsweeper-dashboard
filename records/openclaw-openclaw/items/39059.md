## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/infra/sealed-json-file.ts:113`
  - body: When `OPENCLAW_PASSPHRASE` is missing, `saveSealedJsonFile()` writes plaintext even if the destination is already sealed. Combined with the missing-passphrase load path returning an absent store, a legacy/OAuth/external-CLI sync can replace encrypted credentials with plaintext instead of failing closed.
  - confidence: 0.95
- **[P1] Target the current SQLite auth-profile store:** `src/agents/auth-profiles/paths.ts:29`
  - body: This adds sealed `auth-profiles.json` writes, but current main and the latest release persist auth-profile secrets through the per-agent SQLite `auth_profile_store` row. Keeping a new JSON writer would miss the runtime path current users exercise after upgrade and create competing storage behavior.
  - confidence: 0.9
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still reads the main auth store through the raw JSON reader, so a main store written in the new sealed format cannot be inherited when a passphrase is configured. Either retarget this to the current SQLite store or make the inheritance read use the same protected reader.
  - confidence: 0.86
- **[P2] Record explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed format derives the AES key with Node's implicit `scryptSync` defaults and does not store KDF metadata. That makes the credential format harder to audit and upgrade safely because readers cannot know which cost parameters were used.
  - confidence: 0.84
- **[P2] Write sealed credential files atomically with private mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: The writer creates or truncates the credential file and only then applies `chmod(0o600)`. Credential persistence should use the repo's atomic/private replace path so there is no permission window, partial-write loss, or symlink-following clobber path.
  - confidence: 0.86
