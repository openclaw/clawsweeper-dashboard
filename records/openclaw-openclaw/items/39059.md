## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:435`
  - body: When a sealed `auth-profiles.json` cannot be read because `OPENCLAW_PASSPHRASE` is missing, this path can still set `shouldWrite` through migration or external CLI sync and then call `saveSealedJsonFile`. With no passphrase, that writer emits plaintext, so a sealed credential store can be silently replaced instead of failing closed.
  - confidence: 0.95
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:413`
  - body: The subagent inheritance path still reads the main auth store with the raw JSON reader while this PR can write the main store with the sealed prefix. That means subagents can silently stop inheriting credentials whenever `OPENCLAW_PASSPHRASE` is enabled.
  - confidence: 0.89
- **[P2] Write sealed credentials with an atomic private-file path:** `src/infra/sealed-json-file.ts:111-114`
  - body: `saveSealedJsonFile` writes credential material directly to the target and only then applies `chmod`, while also omitting explicit KDF metadata. For a credential format, use a current atomic/private writer and record the KDF parameters so permissions and decryptability are auditable.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive credential sealing but still has downgrade, inheritance, KDF, and filesystem-write concerns.

Concerns:

- **[high] Sealed auth store can be overwritten as plaintext:** `src/agents/auth-profiles/store.ts:435`
  - body: A missing passphrase is treated as an unreadable store, and a later migration or external CLI sync write can save plaintext over a previously sealed `auth-profiles.json`.
  - confidence: 0.95
- **[medium] Subagents can lose access to sealed inherited credentials:** `src/agents/auth-profiles/store.ts:413`
  - body: Subagent inheritance still reads the main store with `loadJsonFile`, so sealed main-store content cannot be parsed and inherited credentials disappear under `OPENCLAW_PASSPHRASE`.
  - confidence: 0.89
- **[medium] Credential writer creates permission and clobber windows:** `src/infra/sealed-json-file.ts:113`
  - body: The sealed writer uses direct `writeFileSync` plus chmod-after-write instead of an atomic mode-at-create replacement path for credential material.
  - confidence: 0.84
- **[medium] KDF parameters are implicit and absent from the envelope:** `src/infra/sealed-json-file.ts:38`
  - body: The sealed format uses `scryptSync` without explicit parameters or envelope metadata, weakening auditability and future decryptability for a stored credential format.
  - confidence: 0.74
