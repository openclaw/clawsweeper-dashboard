## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block plaintext rewrites of sealed auth stores:** `src/agents/auth-profiles/store.ts:435`
  - body: When `loadProtectedAuthJson` returns `undefined` because the store is sealed and `OPENCLAW_PASSPHRASE` is missing, this path can still set `shouldWrite` after OAuth or external-CLI sync and call `saveSealedJsonFile`. With no passphrase that helper writes plaintext over the sealed file, losing and exposing credentials instead of failing closed.
  - confidence: 0.94
- **[P2] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:413`
  - body: This block now saves inherited subagent stores through the sealed writer, but it still reads the main store with the raw JSON reader. Once the main store is sealed, `loadJsonFile` cannot parse the `openclaw-sealed-json-v1:` payload, so subagents silently skip credential inheritance whenever passphrase sealing is enabled.
  - confidence: 0.9
- **[P2] Persist explicit KDF metadata in sealed files:** `src/infra/sealed-json-file.ts:38`
  - body: The sealed format derives keys with `scryptSync(passphrase, salt, 32)` but does not encode the work factors in the envelope. At-rest credential encryption should use explicit parameters and store them with the ciphertext so future tuning or runtime changes do not create ambiguous security or decryptability behavior.
  - confidence: 0.84
- **[P2] Write sealed auth stores atomically with restrictive mode:** `src/infra/sealed-json-file.ts:112-113`
  - body: `writeFileSync` creates or truncates the target before `chmodSync(0o600)`, leaving a permissions window and following any replaced symlink at the final path. Sensitive auth-store writes should use an atomic temp-file or fd-based path that creates the file with restrictive mode and avoids symlink clobbering.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces credential-at-rest encryption code, but the current implementation has concrete downgrade, key-derivation, and sensitive-file write hardening problems.
