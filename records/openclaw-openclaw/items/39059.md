## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block plaintext rewrites of sealed stores:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth store exists but `OPENCLAW_PASSPHRASE` is missing, `loadProtectedAuthJson` returns `undefined`; if legacy OAuth or external CLI sync then sets `shouldWrite`, this line saves the replacement store through `saveSealedJsonFile` with no passphrase, which overwrites the sealed file as plaintext and can lose or expose credentials. The write path needs to fail closed or skip persistence when the existing destination is sealed and cannot be decrypted.
  - confidence: 0.94
- **[P1] Preserve expectFinal timeout semantics:** `src/gateway/client.ts:520-524`
  - body: This timeout selection applies the default timeout to every request, including `expectFinal` flows unless the caller passes an explicit override. Current main deliberately leaves `expectFinal` unbounded by default because accepted long-running agent calls can legitimately wait for the final response; this PR would make those flows fail after 30 seconds.
  - confidence: 0.9
- **[P2] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:412-413`
  - body: The subagent inheritance path still reads the main auth store with `loadJsonFile`, so a main `auth-profiles.json` written in the new sealed format will fail JSON parsing and skip inheritance. This should use the same sealed-aware reader as the other auth-store load paths.
  - confidence: 0.89
- **[P2] Persist KDF metadata with sealed files:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed envelope records the cipher fields but not the scrypt parameters used to derive the key. Without explicit KDF metadata, future tuning or compatibility decisions cannot reliably distinguish old sealed files from new ones, which is a poor contract for durable credential storage.
  - confidence: 0.8
- **[P2] Write sealed stores with restrictive atomic semantics:** `src/infra/sealed-json-file.ts:113-114`
  - body: The new sealed writer writes directly to the destination and only applies `0600` after the write succeeds. For credential files, create or replace through a restrictive temp file or equivalent safe writer so plaintext or ciphertext is never exposed through a permissive file-mode window and partial writes do not corrupt the store.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces credential-store encryption code, but the current implementation has concrete downgrade, KDF-contract, and file-write hardening issues.

Concerns:

- **[high] Sealed stores can be downgraded to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: A sealed auth store that cannot be read because the passphrase is missing can still be overwritten by migration or external credential sync, and `saveSealedJsonFile` writes plaintext when no passphrase is set.
  - confidence: 0.94
- **[medium] Sealed envelope lacks KDF parameters:** `src/infra/sealed-json-file.ts:39`
  - body: The durable encrypted file format does not store explicit scrypt parameters, making future audit, tuning, and compatibility handling ambiguous.
  - confidence: 0.8
- **[medium] Credential writes chmod after direct write:** `src/infra/sealed-json-file.ts:113`
  - body: The sealed writer writes directly to the target and then chmods it, which is weaker than creating/replacing credential files with restrictive permissions from the start.
  - confidence: 0.78
