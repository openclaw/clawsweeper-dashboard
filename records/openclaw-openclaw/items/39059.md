  - body: When a sealed `auth-profiles.json` cannot be read because `OPENCLAW_PASSPHRASE` is missing, this path can still merge OAuth or external CLI credentials and call `saveSealedJsonFile`; with no passphrase that helper writes plaintext over the sealed file instead of failing closed.
  - confidence: 0.94

  - body: The PR can now write the main auth store in sealed form, but subagent inheritance still reads it with `loadJsonFile`. Once the main store is sealed, this raw JSON read fails and subagents silently skip credential inheritance.
  - confidence: 0.9
- **[P2] Persist explicit KDF metadata in sealed files:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed format derives keys with `scryptSync(passphrase, salt, 32)` but does not encode the work factors in the envelope. Credential-at-rest encryption needs explicit parameters stored with the ciphertext so future tuning or runtime changes do not create ambiguous decryptability or security behavior.
  - confidence: 0.84
- **[P2] Write sealed auth stores atomically with restrictive mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: `writeFileSync` creates or truncates the target before `chmodSync(0o600)`, leaving a permissions window and following any replaced symlink at the final path. Sensitive auth-store writes should create a restrictive temp file or fd and atomically replace the target.
  - confidence: 0.82
