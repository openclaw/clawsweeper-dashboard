## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth-profiles.json exists but OPENCLAW_PASSPHRASE is missing, loadProtectedAuthJson returns undefined and this migration/sync path can still call saveSealedJsonFile. With no passphrase, that helper writes plaintext JSON over the sealed credential store, breaking the stated fail-closed behavior and risking credential exposure/loss.
  - confidence: 0.94
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:416`
  - body: This branch can write the main auth store in sealed form, but subagent inheritance still reads the main store through loadJsonFile nearby. A sealed main auth-profiles.json will fail raw JSON parsing, so subagents silently miss inherited credentials whenever the passphrase mode is enabled.
  - confidence: 0.9
- **[P1] Do not revive the legacy JSON auth-profile store:** `src/agents/auth-profiles/paths.ts:29`
  - body: Current main persists auth profiles in openclaw-agent.sqlite and explicitly tests that auth-profiles.json is not created or read at runtime. Wiring the new sealing mode to ensure/write auth-profiles.json is the wrong storage owner after rebase and would create a competing credential source of truth.
  - confidence: 0.86
- **[P2] Write sealed credentials with a private atomic path:** `src/infra/sealed-json-file.ts:113-114`
  - body: saveSealedJsonFile writes directly to the destination and chmods afterward, while also omitting explicit KDF metadata from the sealed envelope. For a credential-at-rest format, this should use a private temp-file/rename path with no plaintext permission window and an auditable KDF envelope before it is used for auth stores.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive credential sealing but still has concrete downgrade, filesystem-write, and crypto-envelope concerns.

Concerns:

- **[high] Sealed auth stores can be overwritten as plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: A missing passphrase is treated as an empty/unreadable store, and later write paths can persist through saveSealedJsonFile without a passphrase, replacing sealed credentials with plaintext.
  - confidence: 0.94
- **[medium] Sealed credential writes are not atomic/private:** `src/infra/sealed-json-file.ts:113`
  - body: The new writer uses direct writeFileSync followed by chmod, which leaves credential writes without an atomic replacement path and can create a permissions/symlink race in hostile or shared directories.
  - confidence: 0.88
- **[medium] KDF parameters are implicit and not stored:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed envelope calls scryptSync without explicit work factors and records no KDF metadata, making future compatibility and auditability weaker for an at-rest credential format.
  - confidence: 0.82
