- **[P1] Prevent plaintext overwrite when the sealed store is locked:** `src/agents/auth-profiles/store.ts:435-439`
  - body: When the sealed file cannot be read because the passphrase is absent, this path continues with an empty store and can set `shouldWrite` through OAuth, legacy, or CLI synchronization; `saveSealedJsonFile` then overwrites the encrypted file as plaintext.   - confidence: 0.99

  - body: The subagent fallback still uses `loadJsonFile`, so a sealed main `auth-profiles.json` fails JSON parsing and credentials are silently not inherited. Use the sealed-aware path or, after refreshing to current main, the canonical auth backend.

  - body: Writing directly to the destination truncates the only credential copy before the new bytes are complete and applies `0600` only afterward. Use the repository's atomic replacement helper with restrictive creation mode and durability options, and reject writes through links or otherwise preserve the current auth backend's filesystem guarantees.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: Credential sealing is unsafe as implemented because sealed data can be downgraded or lost, subagent auth can disappear, and secret files are not replaced atomically.

Concerns:

- **[high] Sealed credentials can downgrade to plaintext:** `src/agents/auth-profiles/store.ts:435`

  - confidence: 0.99
- **[high] Credential inheritance fails for sealed stores:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent fallback uses the plaintext JSON loader, so sealed main-store credentials are silently unavailable to the child agent.
  - confidence: 0.98

  - body: The helper truncates the destination before a complete durable replacement and applies restrictive permissions only after writing.
  - confidence: 0.96
