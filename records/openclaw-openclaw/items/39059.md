## Summary

Full review comments:

- **[P1] Target the SQLite auth-profile store:** `src/agents/auth-profiles/store.ts:526`
  - body: Current main persists active auth profiles in `openclaw-agent.sqlite` and explicitly does not read legacy `auth-profiles.json` at runtime. Sealing `authPath` here hardens a stale file target, leaves the active credential store unprotected, and reintroduces a competing source of truth.
  - confidence: 0.94
- **[P1] Block plaintext downgrades when passphrase is missing:** `src/agents/auth-profiles/store.ts:438`
  - body: If a sealed auth store cannot be read because `OPENCLAW_PASSPHRASE` is absent, this path can still set `shouldWrite` from legacy/OAuth/CLI data and call `saveSealedJsonFile` without a passphrase. That helper then writes plaintext, contradicting the fail-closed claim and risking credential exposure or loss.
  - confidence: 0.94
- **[P1] Read sealed main auth when inheriting subagent credentials:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still uses the plain JSON loader for the main auth store even though this PR can write that store in sealed form. With a passphrase-enabled main store, subagents silently fail to inherit credentials.
  - confidence: 0.88
- **[P2] Preserve expectFinal timeout semantics:** `src/gateway/client.ts:520-524`
  - body: Current main deliberately leaves `expectFinal` requests unbounded by default, with tests covering that contract. This timeout calculation applies the default timeout even for `expectFinal`, so legitimate long-running agent/chat requests can fail after the timeout.
  - confidence: 0.9
- **[P2] Write sealed credentials atomically and privately:** `src/infra/sealed-json-file.ts:113-114`
  - body: `saveSealedJsonFile` writes credential payloads directly and only chmods afterward, leaving an avoidable permission window and non-atomic clobber behavior. Use the existing atomic/private write path or equivalent create-with-mode temp rename before treating this as safe credential storage.
  - confidence: 0.89
- **[P2] Store KDF parameters in the sealed envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The sealed format derives keys with implicit `scryptSync` defaults and does not record KDF parameters. Persisted credentials need explicit, envelope-stored KDF metadata so strength and future decryptability are auditable and stable.
  - confidence: 0.82

## Security Review
