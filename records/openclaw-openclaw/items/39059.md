## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: If an existing sealed `auth-profiles.json` cannot be read because `OPENCLAW_PASSPHRASE` is missing, this path can still set `shouldWrite` after legacy/OAuth/CLI sync and call `saveSealedJsonFile()` with no passphrase. That overwrites the sealed credential store as plaintext, so the write must fail closed or be skipped until the store can be decrypted.
  - confidence: 0.93
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:416`
  - body: The branch writes auth stores with `saveSealedJsonFile()`, but the subagent inheritance fallback still reads the main store through `loadJsonFile()` just above this line. A passphrase-enabled main store will not parse as JSON, so subagents silently lose inherited credentials; use the sealed-aware reader for this fallback.
  - confidence: 0.9
- **[P1] Move sealing onto the current SQLite auth store:** `src/infra/sealed-json-file.ts:113`
  - body: Current main persists auth profiles in `openclaw-agent.sqlite` via `auth_profile_store`, but this PR adds a new JSON-file sealing primitive for `auth-profiles.json` and legacy `oauth.json`. After rebase, that would not protect the live credential store and risks creating a parallel legacy source of truth; the sealing design needs to cover the SQLite persistence/migration path.
  - confidence: 0.88
- **[P2] Create sealed credential files atomically with private mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: This writes sensitive credential material directly to the destination and only then applies `chmod(0o600)`. Use an existing atomic/private-mode replace path or an equivalent sync helper so plaintext or sealed credential files are never created with broader permissions or left partially written.
  - confidence: 0.84
- **[P2] Record KDF parameters in the sealed envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: `deriveKey()` relies on implicit `scryptSync` parameters while the v1 envelope stores no KDF metadata. Make the scrypt parameters explicit and persist them in the envelope so future tuning and decrypt compatibility are auditable.
  - confidence: 0.8
