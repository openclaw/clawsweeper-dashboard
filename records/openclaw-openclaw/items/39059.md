## Best Possible Solution

Split or redesign the branch: keep current-main gateway timeout semantics, carry only still-needed transcript permission hardening if maintainers want it, and handle credential-at-rest encryption through the canonical security design in https://github.com/openclaw/openclaw/issues/43794.

No. The gateway timeout part should defer to current main, and the credential sealing part needs a security-owned design for fail-closed writes, KDF metadata, atomic persistence, migration, recovery, and SecretRef compatibility before merge.

Overall confidence: 0.92

Full review comments:

- **[P1] Block plaintext rewrites of sealed stores:** `src/agents/auth-profiles/store.ts:436-438`
  - body: When the sealed auth store cannot be read because OPENCLAW_PASSPHRASE is missing, this path can still set `shouldWrite` from legacy/OAuth/external-CLI data and call `saveSealedJsonFile`; with no passphrase, that writer replaces the sealed file with plaintext. That violates the PR's fail-closed claim and can leak or lose credential data.
  - confidence: 0.94
- **[P1] Preserve expectFinal timeout semantics:** `src/gateway/client.ts:520-524`
  - body: This branch applies the default timeout to every request, including `expectFinal` requests. Current main intentionally keeps `expectFinal` flows unbounded unless a caller passes an explicit timeout, so carrying this code forward can abort long-running gateway operations after 30 seconds.
  - confidence: 0.9
- **[P2] Use the sealed reader for inherited main stores:** `src/agents/auth-profiles/store.ts:412-413`
  - body: The subagent inheritance fallback still reads the main auth store with `loadJsonFile`. Once the main store is sealed, that read cannot parse the new envelope and subagents silently stop inheriting credentials; use the sealed-aware reader introduced by this PR.
  - confidence: 0.91
- **[P2] Record KDF parameters in the sealed envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The new file format derives keys with implicit Node scrypt defaults and stores no KDF metadata. Once this format ships, future tuning or compatibility checks cannot tell which parameters protected an existing file, so the envelope should carry explicit KDF settings.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces credential-at-rest encryption code, but the implementation has concrete downgrade, KDF metadata, and sensitive-file persistence hardening problems.

Concerns:

- **[high] Sealed auth stores can downgrade to plaintext:** `src/agents/auth-profiles/store.ts:437`
  - body: A missing passphrase is converted into an undefined read, after which migration or external credential sync can persist the store through a writer that falls back to plaintext.
  - confidence: 0.94
- **[medium] KDF parameters are implicit and absent from the format:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed JSON format calls scrypt without explicit options and does not store KDF metadata, making security posture and future decryptability harder to audit or evolve.
  - confidence: 0.84
- **[medium] Sensitive file persistence is not atomic or mode-first:** `src/infra/sealed-json-file.ts:113`
  - body: The writer uses writeFileSync followed by chmod, which leaves a permissions window and does not address symlink or hardlink clobber concerns for credential files.
  - confidence: 0.82
- **[medium] Credential encryption policy is not settled:**
  - body: The PR adds an env-passphrase storage format without a documented key-management, recovery, migration, backup, SecretRef, and runtime plaintext contract for existing users.
  - confidence: 0.87
