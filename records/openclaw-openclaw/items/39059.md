Full review comments:

- **[P1] Block plaintext rewrites of sealed stores:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth-profiles.json cannot be read because OPENCLAW_PASSPHRASE is missing, this path can still set shouldWrite from legacy/OAuth/CLI sync and then save the store without a passphrase. saveSealedJsonFile falls back to plaintext in that case, so an encrypted credential store can be overwritten with partial plaintext credentials instead of failing closed.
  - confidence: 0.97
- **[P1] Preserve expectFinal request semantics:** `src/gateway/client.ts:520-524`
  - body: The PR applies the default 30s timeout to every request, including expectFinal calls. Long-running session/chat requests use expectFinal while waiting for a final response after an accepted ack, and current main explicitly avoids auto-timing those out unless the caller provides a timeout, so this can break normal agent turns that run longer than 30 seconds.
  - confidence: 0.92
- **[P2] Use sealed reads for inherited main stores:** `src/agents/auth-profiles/store.ts:412-413`
  - body: Subagent inheritance still reads the main auth store with loadJsonFile, but this PR can now write that main store in sealed form. With OPENCLAW_PASSPHRASE enabled, loadJsonFile cannot parse the sealed prefix, so subagents silently fail to inherit credentials from the main agent.
  - confidence: 0.9
- **[P2] Persist KDF parameters in sealed envelopes:** `src/infra/sealed-json-file.ts:38-39`
  - body: The new sealed format derives keys with scryptSync defaults but does not store the KDF name or parameters in the envelope. That makes the security posture hard to audit and leaves future decryptability tied to implicit runtime defaults, so the v1 format should record and validate explicit KDF metadata before shipping.
  - confidence: 0.82
- **[P2] Write sealed stores atomically with restrictive mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: saveSealedJsonFile writes the credential payload directly and only then chmods the file. For a secret store this leaves a permissions window and an in-place truncate/write path; use an atomic replacement helper or secure open mode so plaintext or sealed credential bytes are never created with broad permissions or clobbered in place.
  - confidence: 0.86
- **[P3] Remove the direct changelog edit:** `CHANGELOG.md:750`
  - body: CHANGELOG.md is release-owned in this repository, and normal PRs should carry release-note context in the PR body or commit message instead. This entry should be removed from the branch and left to release generation.
  - confidence: 0.88
