## Summary

Keep open: this maintainer-labeled security/auth PR still has useful hardening intent, but the current patch is not merge-ready because the sealed-store paths can downgrade credentials to plaintext, subagent auth inheritance can stop working, and current main has since moved the affected gateway/auth contracts.

## What This Changes

The branch adds gateway request timeouts, optional `OPENCLAW_PASSPHRASE` sealed JSON auth-store support for `auth-profiles.json` and `oauth.json`, transcript chmod reassertion, focused tests, and a changelog entry.

## Best Possible Solution

Refresh the branch onto current main, keep the existing gateway-client timeout implementation, and redesign credential sealing around the current SQLite auth-store/doctor migration path with no plaintext downgrade and covered main/subagent behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a sealed auth store, unset `OPENCLAW_PASSPHRASE`, then trigger a migration or external-CLI sync write and the PR path can save plaintext. The subagent inheritance issue is also source-reproducible because the fallback still calls `loadJsonFile()` on the main store.

## Solution Assessment

No, this is not yet the best way to solve the issue. The gateway timeout piece is now better owned by the current `packages/gateway-client` implementation, and credential sealing needs to be rebuilt around current SQLite auth persistence with fail-closed and upgrade tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent plaintext overwrite when the sealed store is locked:** `src/agents/auth-profiles/store.ts:438`
  - body: When `loadProtectedAuthJson()` returns `undefined` for a sealed store without `OPENCLAW_PASSPHRASE`, this path can still decide `shouldWrite` and call `saveSealedJsonFile()` without a passphrase. That rewrites a previously sealed credential store as plaintext during legacy/OAuth/external-CLI sync instead of failing closed.
  - confidence: 0.94
- **[P1] Read sealed main stores for subagent inheritance:** `src/agents/auth-profiles/store.ts:412`
  - body: This fallback still uses `loadJsonFile(mainAuthPath)`, but the PR can now write the main store as `openclaw-sealed-json-v1:` text. Subagents with no local store will skip inherited credentials whenever the main store is sealed, so the fallback should use the sealed-aware reader.
  - confidence: 0.9
- **[P2] Write sealed auth stores with safe replace semantics:** `src/infra/sealed-json-file.ts:113-114`
  - body: `saveSealedJsonFile()` writes credential-bearing content directly to the destination and only applies `0600` afterward. That creates a permissions/TOCTOU window and loses the atomic-write protection expected for auth stores; use the current durable auth-store owner or a safe temp-file replace path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch improves intended security posture, but the sealed-store implementation currently introduces concrete credential downgrade and filesystem-write concerns.

Concerns:

- **[high] Sealed credentials can be downgraded to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: A missing passphrase makes the sealed read return no store, after which migration or sync can persist the auth store without a passphrase and replace sealed credentials with plaintext.
  - confidence: 0.94
- **[medium] Credential writes are direct and chmod-after-write:** `src/infra/sealed-json-file.ts:113`
  - body: The new sealed JSON helper writes directly to the target path before applying restrictive permissions, which is weaker than the existing durable/atomic credential-write expectations.
  - confidence: 0.82
- **[low] KDF parameters are not explicit in the sealed envelope:** `src/infra/sealed-json-file.ts:39`
  - body: The encryption helper uses implicit `scryptSync` parameters and does not record KDF metadata, making the sealed format harder to audit or tune safely later.
  - confidence: 0.72
