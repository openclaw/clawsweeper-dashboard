Overall confidence: 0.93

Full review comments:

- **[P1] Target credential sealing at the SQLite store:** `src/agents/auth-profiles/paths.ts:32`
  - body: This branch creates and seals auth-profiles.json, but current main persists auth profiles in openclaw-agent.sqlite and explicitly does not read legacy auth-profiles.json at runtime. Merging this would miss the active credential store and reintroduce a retired path instead of protecting current users.
  - confidence: 0.95
- **[P1] Block plaintext downgrades when passphrase is absent:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed auth store exists and OPENCLAW_PASSPHRASE is missing, the loader returns undefined and later migration/sync paths can call saveSealedJsonFile without a passphrase, overwriting encrypted credentials as plaintext. That contradicts the claimed fail-closed behavior and can leak or lose credentials.
  - confidence: 0.94
- **[P1] Use the sealed reader for inherited main auth:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still reads the main auth store with loadJsonFile even though this PR can write that store in sealed form. With OPENCLAW_PASSPHRASE enabled, subagents can silently fail to inherit provider credentials.
  - confidence: 0.88
- **[P2] Preserve expectFinal request timeout semantics:** `src/gateway/client.ts:520-524`
  - body: This timeout calculation applies the default timeout to expectFinal requests unless the caller supplies an override, but current main intentionally leaves expectFinal unbounded by default for long-running final-response flows. Reintroducing the old cap can make legitimate agent requests fail after the timeout.
  - confidence: 0.9
- **[P2] Write sealed credentials atomically and privately:** `src/infra/sealed-json-file.ts:113-114`
  - body: saveSealedJsonFile writes credential payloads directly and only chmods afterward, leaving a permissions window and non-atomic clobber behavior. Credential sealing should use the existing atomic/private write pattern or be implemented in the active SQLite store with equivalent guarantees.
  - confidence: 0.89
- **[P2] Record explicit KDF parameters in sealed files:** `src/infra/sealed-json-file.ts:38-40`
  - body: The sealed format derives keys with implicit scryptSync defaults and does not store KDF parameters in the envelope. Persisted credential formats need explicit work factors so security posture and future decryptability are auditable.
  - confidence: 0.82
