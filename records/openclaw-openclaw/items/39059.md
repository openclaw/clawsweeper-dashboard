## Label Justifications

- `P1`: The proposed credential persistence path can expose or lose provider credentials during plausible startup and upgrade workflows.
- `merge-risk: 🚨 compatibility`: The new passphrase requirement and persistent file format can make existing stores unreadable and require undocumented operator recovery.
- `merge-risk: 🚨 auth-provider`: The changed readers and writers can suppress inherited provider credentials or replace the credential source of truth.
- `merge-risk: 🚨 security-boundary`: The patch creates an encryption boundary but permits plaintext downgrade and uses an unsafe in-place secret-file replacement path.

## Summary

Keep this maintainer-authored PR open for explicit security-owner handling, but do not merge the current branch: current main already implements request-level gateway timeouts, while the remaining auth-store work targets superseded persistence ownership and introduces credential downgrade and inheritance regressions. The credential-at-rest goal remains meaningful and is still tracked by https://github.com/openclaw/openclaw/issues/43794, so the branch is not appropriate for cleanup closure.

## What This Changes

The PR adds bounded gateway RPC timeouts, optional `OPENCLAW_PASSPHRASE` sealing for JSON auth stores, and permission reassertion after mirrored transcript writes.

## Best Possible Solution

Keep the package-owned gateway timeout implementation already on main, then design credential-at-rest protection at the current SQLite auth-profile boundary with security-owner approval, explicit migration and recovery commands, atomic durable writes, rotation/version metadata, and fail-closed tests for fresh installs and upgrades.

Full review comments:

- **[P1] Prevent plaintext overwrite when the sealed store is locked:** `src/agents/auth-profiles/store.ts:436-438`
  - body: When the sealed store cannot be read because the passphrase is absent, this path still allows OAuth, legacy, or external-CLI data to set `shouldWrite`; `saveSealedJsonFile` then writes plaintext over the existing encrypted credentials. Preserve the locked state and reject all persistence until the store can be decrypted or explicitly migrated.
  - confidence: 0.99
- **[P1] Read the sealed main store for subagent inheritance:** `src/agents/auth-profiles/store.ts:412`
  - body: This fallback still calls `loadJsonFile`, but the PR can now write the main store with a non-JSON sealed prefix. Subagents therefore silently skip inherited credentials whenever sealing is enabled; use the same protected reader and propagate locked-store failures safely.
  - confidence: 0.99
- **[P2] Replace secret files atomically with restrictive creation mode:** `src/infra/sealed-json-file.ts:113-114`
  - body: Writing the destination in place creates truncation, permission-window, and link-race hazards for credential data, with `chmod` occurring only after the write. Use the repository's atomic replacement primitive with mode `0600`, durability options, and link-safe destination handling.
  - confidence: 0.95
