## Summary

Keep this PR open for maintainer/security handling, but it is not merge-ready: the branch is dirty against current main, the auth sealing path can downgrade sealed credentials to plaintext, subagent auth inheritance still uses the plaintext JSON reader, and current main has already moved the relevant auth store to SQLite while the gateway timeout part is already implemented in the shared gateway client.

## What This Changes

The PR adds GatewayClient request timeouts, optional `OPENCLAW_PASSPHRASE` sealed JSON auth-store support, transcript chmod hardening, focused tests, and a changelog entry.

## Best Possible Solution

Rebase and redesign the credential hardening against current SQLite auth-profile storage with no plaintext downgrade path, explicit KDF/envelope choices, safe atomic permission semantics, and focused upgrade tests; keep or drop the gateway timeout portion based on the already-merged shared gateway-client implementation.

## Solution Assessment

No, this is not the best current solution as written. The gateway timeout portion is already implemented on current main, while auth sealing needs a current SQLite-backed design with fail-closed writes, explicit KDF metadata, and atomic/permission-safe persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent sealed stores from downgrading to plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: When a sealed `auth-profiles.json` cannot be read because `OPENCLAW_PASSPHRASE` is missing, this path can still set `shouldWrite` through legacy/OAuth/CLI sync and then call `saveSealedJsonFile`. With no passphrase, that writer emits plaintext, so a sealed credential store can be silently replaced instead of failing closed.
  - confidence: 0.95
- **[P1] Read inherited main stores through the sealed reader:** `src/agents/auth-profiles/store.ts:412`
  - body: The subagent inheritance path still calls `loadJsonFile` on the main auth store. After this PR, that main store may be written with the sealed prefix, so raw JSON parsing fails and subagents can silently stop inheriting credentials whenever `OPENCLAW_PASSPHRASE` is enabled.
  - confidence: 0.89
- **[P2] Write sealed credential files without direct chmod-after-write:** `src/infra/sealed-json-file.ts:113-114`
  - body: `saveSealedJsonFile` writes credential material directly to the target and only then applies `chmod`. For a security-sensitive auth store, use a current atomic replacement path with restrictive mode at creation time and explicit KDF/envelope metadata so there is no permissive creation or target-clobber window.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff aims to improve security, but the current auth sealing implementation introduces concrete credential downgrade, auth inheritance, and filesystem/KDF risks.

Concerns:

- **[high] Sealed auth store can be overwritten as plaintext:** `src/agents/auth-profiles/store.ts:438`
  - body: A missing passphrase is swallowed as an unreadable store, and a later migration/sync write can save plaintext over a previously sealed `auth-profiles.json`.
  - confidence: 0.95
- **[medium] Subagents can lose access to sealed inherited credentials:** `src/agents/auth-profiles/store.ts:412`
  - body: Subagent inheritance still reads the main store with `loadJsonFile`, so sealed main-store content cannot be parsed and inherited credentials disappear under `OPENCLAW_PASSPHRASE`.
  - confidence: 0.89
- **[medium] Credential writer creates permission and clobber windows:** `src/infra/sealed-json-file.ts:113`
  - body: The sealed writer uses direct `writeFileSync` plus chmod-after-write instead of an atomic mode-at-create replacement path for credential material.
  - confidence: 0.84
- **[medium] KDF parameters are implicit and absent from the envelope:** `src/infra/sealed-json-file.ts:39`
  - body: The sealed format uses `scryptSync` without explicit parameters or envelope metadata, weakening auditability and future decryptability for a stored credential format.
  - confidence: 0.74
