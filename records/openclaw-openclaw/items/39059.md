## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent plaintext overwrite of sealed stores:** `src/agents/auth-profiles/store.ts:438`
  - body: If a sealed auth store cannot be read because `OPENCLAW_PASSPHRASE` is missing, `loadCoercedStore` returns `null` and this write can persist through `saveSealedJsonFile` with no passphrase, replacing sealed credentials with plaintext. The write should fail closed or force the run read-only before any persistence occurs.
  - confidence: 0.96
- **[P1] Use the sealed reader for inherited auth:** `src/agents/auth-profiles/store.ts:416`
  - body: This inheritance path writes cloned auth through the sealed writer, but it still reads the main store above with `loadJsonFile`; once the main store is sealed, subagents will silently stop inheriting credentials. Read the main store through the same sealed-aware loader before cloning it.
  - confidence: 0.91
- **[P1] Write sealed credentials with safe atomic permissions:** `src/infra/sealed-json-file.ts:113-114`
  - body: `saveSealedJsonFile` writes credential material directly to the target and only then applies `chmod(0o600)`, following normal symlink behavior and creating a permissions window. Use the existing safe atomic writer or explicit secure open semantics before this becomes an auth-store protection boundary.
  - confidence: 0.88
- **[P2] Persist explicit KDF parameters in the envelope:** `src/infra/sealed-json-file.ts:38-39`
  - body: The new v1 sealed format derives keys with implicit Node scrypt defaults and stores no KDF metadata, which makes strength and future decryptability hard to audit once files exist. Put explicit KDF parameters in the envelope before shipping a persistent credential format.
  - confidence: 0.78
