## Summary

Keep open. Current main still has no implemented repair for the reported `</arg_value>>` suffix, the active `exec` and `read` paths still pass received string arguments through unchanged, and open PR #48792 already targets this issue with closing syntax.

## What This Changes

The issue reports that Windows Qwen/DashScope tool calls append `</arg_value>>` to `exec()` commands and `read()` paths, corrupting command execution and file reads.

## Best Possible Solution

Keep this issue open until PR #48792 or a replacement lands a narrow fix at the active tool-argument or provider stream boundary, with regression coverage for the reported Qwen/DashScope suffix and negative coverage for legitimate XML-like payload text.

## Reproduction Assessment

Yes for the local boundary behavior: if `exec` receives `command: "echo test</arg_value>>"` or `read` receives a suffixed `path`, current main forwards that string unchanged. The live Windows/DashScope emission itself was not reproduced in this read-only review.

## Solution Assessment

No current-main solution exists yet. The maintainable fix should be narrow to the malformed tool-argument boundary or affected provider stream path; record-wide stripping is riskier unless tests prove it preserves valid content.
