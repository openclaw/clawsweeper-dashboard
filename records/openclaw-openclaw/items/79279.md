## Summary

Keep open: current main still has the forced Codex runtime path that can carry non-OpenAI providers into the Codex app-server, and the linked fix PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to avoid routing MiniMax, Anthropic, or other non-OpenAI selections into the Codex app-server after Codex runtime migration.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79238 or an equivalent provider/model-scoped runtime fix that stops doctor and stale session state from pinning mixed-provider agents to Codex while preserving Codex for supported OpenAI routes.

## Reproduction Assessment

Yes, source-reproducible on current main: a cataloged MiniMax/Anthropic session model can be persisted while explicit Codex runtime forces the Codex harness and forwards that provider to app-server. I did not run the full live Docker/Codex/MiniMax scenario in this read-only review.

## Solution Assessment

Yes, the linked direction is the best current path: move runtime policy to provider/model scope and clean stale whole-agent/session Codex pins instead of adding provider-specific failure handling after app-server startup.
