## Summary

Keep open: current main still has a source path matching the reported Codex runtime/provider mismatch, and the related reports remain open rather than merged fixes.

## What This Changes

The issue reports that after doctor migration to `agentRuntime.id: "codex"`, non-OpenAI model refs such as MiniMax or Anthropic can still be selected and then fail inside Codex app-server instead of being routed or rejected earlier.

## Best Possible Solution

Enforce runtime/provider compatibility before model overrides or fallback dispatch reach Codex, using the existing harness support contract to reject explicit Codex plus non-supported providers with a clear error while keeping `auto` provider-aware.

## Reproduction Assessment

Yes, source-reproducible: current main can persist a non-OpenAI session model override while explicit Codex runtime forces the Codex harness and forwards that provider to app-server. I did not live-run the Docker/Codex/MiniMax setup in this read-only review.

## Solution Assessment

Unclear for one exact patch, but the maintainable direction is clear: validate compatibility at a shared gateway/runtime boundary before the app-server call, not by adding scattered special cases after failure.
