## Summary

Keep open. Current main partially respects `OPENCLAW_HOME` for central state/config and agent defaults, but the reported contract is still not true for the dev/git checkout path, installer bootstrap defaults, and conflicting public docs.

## What This Changes

The issue requests that `OPENCLAW_HOME` drive OpenClaw’s default state, config, git checkout, setup, skills, and related internal paths unless more specific path env vars are set.

## Best Possible Solution

Define the supported `OPENCLAW_HOME` contract for internal OpenClaw defaults, update dev/git checkout and installer bootstrap paths to use the existing effective-home helper unless explicit path overrides are set, add focused regression tests, and align docs with any intentional OS-home exceptions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with `OPENCLAW_HOME` set and `OPENCLAW_GIT_DIR` unset, `resolveGitInstallDir` still resolves from `os.homedir()`, and `scripts/install.sh` still initializes git/config/workspace bootstrap paths from `$HOME`.

## Solution Assessment

Yes, a narrow repair in the existing path-resolution seams is the best route. The safer boundary is to fix internal OpenClaw defaults and docs while leaving explicitly OS-home host behavior and the separate sandbox tilde work in #65536 alone.
