## Summary

Keep open. Current main and v2026.5.18 still only partially satisfy the documented `OPENCLAW_HOME` contract: state/config paths use the effective home, but git checkout, installer bootstrap defaults, and the explicit `.openclaw` nesting edge case still fall back to `$HOME`/`os.homedir()` or append another `.openclaw`; related open PRs cover subsets rather than a merged full fix.

## What This Changes

The issue requests that `OPENCLAW_HOME` determine OpenClaw state, config, git, installer, and related defaults unless more specific path environment variables are set.

## Best Possible Solution

Define the internal `OPENCLAW_HOME` contract in one place, update remaining git/update/install defaults to derive from it when explicit path env vars are absent, add the narrow state-dir basename guard, and document intentional OS-home exceptions.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set and more specific path env vars absent, current main still resolves default dev checkout and installer bootstrap paths from OS `$HOME`/`os.homedir()`, and `OPENCLAW_HOME=/home/user/.openclaw` still nests state/config under `.openclaw/.openclaw`.

## Solution Assessment

Yes for a narrow repair through the existing home/path seams. The maintainable path is to reuse the effective-home helper, add focused guards/tests, and document intentional OS-home exceptions rather than replacing every `HOME` or `os.homedir()` occurrence.
