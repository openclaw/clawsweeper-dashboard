## Summary

Keep open. Current main still only partially satisfies the documented `OPENCLAW_HOME` contract: state/config and some agent defaults use the effective home, but the dev/git checkout and installer bootstrap defaults still fall back to the OS account home, and the explicit `.openclaw` home edge case remains tracked separately but unmerged.

## What This Changes

The issue asks OpenClaw to consistently derive internal state, config, git checkout, installer, and related paths from `OPENCLAW_HOME` unless a more specific path env var is set.

## Best Possible Solution

Keep the effective-home helper as the contract, update the remaining internal git/update/install defaults to derive from it when explicit path env vars are absent, add a narrow state-dir basename guard, and document intentional OS-home exceptions.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set and more specific path env vars absent, current main still resolves the dev checkout from `os.homedir()` and installer bootstrap defaults from `$HOME`; I did not run a live install/update flow in this read-only review.

## Solution Assessment

Yes for a narrow repair in the existing path-resolution seams. A blanket replacement of every `HOME` or `os.homedir()` lookup would be risky because several host-account paths are likely intentional.
