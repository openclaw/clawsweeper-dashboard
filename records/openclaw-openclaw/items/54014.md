## Summary

Keep open: current main still only partially honors the documented OPENCLAW_HOME contract. State/config defaults now use the effective home in some paths, but the git checkout and installer/bootstrap defaults still fall back to the OS account home, and the existing open PRs cover narrower pieces rather than the broader issue.

## What This Changes

The issue asks OpenClaw to consistently derive internal state, config, git checkout, and related defaults from OPENCLAW_HOME unless more specific path environment variables are set.

## Best Possible Solution

Centralize the OPENCLAW_HOME contract for internal defaults, update the remaining git/update/install/bootstrap paths to use it when explicit path env vars are absent, preserve documented parent-home semantics, and document intentional OS-home exceptions.

## Reproduction Assessment

Yes, at source level. With OPENCLAW_HOME set and more specific path env vars absent, current main still derives some defaults from the effective home but continues to use OS HOME/os.homedir for git/install bootstrap paths and still appends .openclaw in the explicit state-dir-name edge case.

## Solution Assessment

Yes for a bounded repair through the existing home/path helpers. The maintainable path is to reuse resolveRequiredHomeDir, add focused state-dir basename guards and tests, and document intentional OS-home exceptions rather than sweeping every HOME or os.homedir occurrence.
