## Summary

Keep open. Current main and the latest release still only partially satisfy the documented `OPENCLAW_HOME` contract: state/config and some agent defaults use the effective home, but git checkout and installer bootstrap defaults still fall back to the OS account home, and the explicit `.openclaw` home edge case remains unresolved. Related PRs cover useful subsets, but none is a merged canonical fix for the remaining core behavior.

## What This Changes

The issue reports that `OPENCLAW_HOME` does not consistently drive OpenClaw state, config, git checkout, installer, skills, and related internal paths when more specific path environment variables are unset.

## Best Possible Solution

Define the internal `OPENCLAW_HOME` contract in one place, update the remaining git/update/install defaults to derive from it when explicit path env vars are absent, add the narrow state-dir basename guard, and document intentional OS-home exceptions.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set and more specific path env vars absent, current main still resolves default dev checkout from `os.homedir()` and installer bootstrap paths from `$HOME`; the `.openclaw/.openclaw` state/config edge case is also directly visible in the helper code.

## Solution Assessment

Yes for a narrow repair through the existing home/path seams. The safer path is to reuse the effective-home helper, add focused guards/tests, and document intentional OS-home exceptions rather than replacing every `HOME` or `os.homedir()` occurrence.
