## Label Justifications

- `P2`: This is a confirmed path-resolution bug with a limited but real impact on custom-home installs and service-account deployments.
- `impact:session-state`: The reported behavior can split state, agent directories, sessions, and workspace paths between `OPENCLAW_HOME` and the OS account home.
- `impact:auth-provider`: Config and credential-adjacent paths can resolve under the wrong home, affecting provider and secret setup discovery.

## Summary

Keep open. Current main and `v2026.5.18` still only partially honor `OPENCLAW_HOME`: state/config helpers use the effective home, but dev/git checkout and installer bootstrap defaults still resolve from the OS account home, and the related PRs are open rather than merged fixes.

## What This Changes

The issue asks OpenClaw to use `OPENCLAW_HOME` as the default base for state, config, and git paths unless more specific path environment variables are set.

## Best Possible Solution

Align the remaining internal git/update/install defaults with the existing effective-home helper, add a narrow `.openclaw` state/config guard where appropriate, and document any intentional OS-home exceptions.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_HOME` set and `OPENCLAW_GIT_DIR` or direct config/state overrides absent, current main still derives the git checkout and installer bootstrap config checks from `os.homedir()` or `$HOME`.

## Solution Assessment

Yes, a narrow repair through the existing effective-home path helpers is the best direction. A blanket rewrite of every home lookup would be unsafe because some OS-account paths are intentional.
