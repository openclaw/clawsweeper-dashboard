## Summary

## What This Changes

The PR adds a LINE webhook in-flight guard that replies busy and skips overlapping messages for the same LINE conversation/user key.

## Best Possible Solution

Handle only LINE events with a stable sender identity or make conversation-level throttling an explicit, tested maintainer choice, then require redacted real LINE proof before merge.

Yes, source-reproducible: the PR guards postbacks by `getLineSourceInfo`, while the pinned LINE SDK types allow group/room postbacks to omit `userId`. I did not run a live LINE repro in this read-only review.

## Solution Assessment

No. The message guard is a plausible mitigation, but applying it uniformly to postbacks is too broad until user-idless group/room sources and skip-vs-queue semantics are handled.

- **[P2] Preserve per-user keys for group postbacks:** `extensions/line/src/bot-handlers.ts:650-653`
  - body: This guard now wraps postbacks too, but the pinned LINE SDK types make `userId` optional for group/room sources and document it as only included in message events.
