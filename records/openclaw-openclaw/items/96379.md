## Summary

Keep open: the LINE guard is a useful mitigation, but the current PR is not merge-ready because it can suppress senderless group/room postbacks, still lacks real LINE proof, and needs maintainer agreement on dropping rather than queueing rapid events.

## What This Changes

Adds a process-local LINE in-flight guard that replies busy and skips overlapping message/postback events, wires it into the LINE bot, and adds handler tests.

## Best Possible Solution

Keep the guard in the LINE plugin only after it handles events without stable sender identity and maintainers accept the drop-vs-queue behavior with real LINE proof.

## Reproduction Assessment

Yes, source-level for the introduced defect: PR head keys group/room postbacks with `userId ?? "unknown"` while the LINE SDK type says group/room userId is optional and only included in message events. The original rapid-message symptom still lacks real LINE reproduction in the PR body.

## Solution Assessment

No. The LINE plugin is the right owner boundary, but a default-on skip guard across messages and postbacks is not the best merge shape until senderless postbacks and skip-vs-queue semantics are settled.
