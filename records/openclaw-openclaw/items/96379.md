## Summary

Keep open: this is a useful LINE UX mitigation, but it is not merge-ready because senderless group/room postbacks can be throttled under a shared `unknown` key, the PR changes delivery semantics by dropping rapid follow-ups, and real LINE behavior proof is still missing.

## What This Changes

The PR adds a process-local LINE per-user in-flight guard that sends a busy reply for overlapping message/postback events and wires it into the LINE bot with unit tests.

## Best Possible Solution

Keep the LINE plugin guard only for events with stable sender identity, or explicitly test and accept conversation-level throttling, then require redacted real LINE proof before merge.

## Reproduction Assessment

Yes, source-level: PR head builds group/room postback guard keys with `userId ?? "unknown"`, while the pinned LINE SDK types allow group/room sources to omit `userId`. I did not run a live LINE reproduction in this read-only review.

## Solution Assessment

No. The LINE plugin is the right owner boundary, but applying the skip guard uniformly to postbacks is too broad until senderless group/room sources and skip-vs-queue semantics are handled.
