## Summary

Keep open: current main does not already implement this LINE guard, but this PR is not merge-ready because it adds a message/postback suppression path without real LINE behavior proof, and group/room postbacks can collapse onto an `unknown` user key.

## What This Changes

The PR adds an in-memory LINE per-user in-flight guard, wires it into `createLineBot`, sends a busy reply for concurrent events, and adds mocked handler tests.

## Best Possible Solution

Handle only sources with a stable sender identity or explicitly define conversation-level postback throttling, add user-idless group/room postback coverage, and require redacted real LINE proof before merge.

## Reproduction Assessment

Yes, source-reproducible for the review finding: the PR guards postbacks by `getLineSourceInfo`, while the pinned LINE SDK types allow group/room postbacks to omit `userId`. I did not run a live LINE repro in this read-only review.

## Solution Assessment

No. The message guard is a plausible mitigation, but applying it uniformly to postbacks is too broad until user-idless group/room sources and skip-vs-queue semantics are handled.
