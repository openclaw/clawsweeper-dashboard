## Summary

Keep open: current main and v2026.6.9 still only promote parseable plain-text tool-call payloads with arguments, while the reported bare marker shape has no executable arguments and remains a provider/runtime policy gap rather than a completed fix.

## What This Changes

The issue reports non-Anthropic provider routes leaking tool-call intent as visible text such as `[tool: exec]` or `[tool:exec]` instead of producing structured `toolCall` content that the agent loop can execute.

## Best Possible Solution

Keep this issue open until maintainers choose an explicit recovery policy for bare leaked tool markers, backed by live provider proof or a maintainer-approved harness plus focused regression tests.

## Reproduction Assessment

Yes for the source-level gap, not for a live provider replay. Current main only promotes parseable text tool-call payloads with arguments, while the reported bare marker has no arguments to execute.

## Solution Assessment

Unclear. Directly executing a bare marker is not the best fix because it lacks arguments; the safer direction is a maintainer-chosen retry, fallback, blocking, forced-tool, diagnostic, or tool-surface policy.
