## Summary

Keep open: current main and the latest release still do not safely recover the reported spaced, no-argument `[tool: exec]` marker, and the current open invoke-style repair PR only covers a different partial shape while it remains blocked on sanitizer review.

## What This Changes

The issue reports non-Anthropic/API-compatible models emitting plain-text tool-call markers such as `[tool: exec]` instead of structured tool-use blocks, leaving turns stuck without executing tools.

## Best Possible Solution

Define a provider-runtime recovery policy that warns, retries, or falls back for bare leaked tool markers without executing no-argument text, then add focused parser/diagnostic coverage and live provider proof.

## Reproduction Assessment

Yes for the source-level gap: current main rejects `[tool: exec]` because of the space and rejects bare `[tool:exec]` because no JSON/XML arguments follow. I did not run a live external-provider replay, so the model frequency evidence remains the reporter's redacted trajectories and screenshots.

## Solution Assessment

Unclear until maintainers choose policy. Directly parsing a bare marker is not the best fix because there are no arguments; warning, retry, fallback, or provider guidance needs an explicit runtime decision.
