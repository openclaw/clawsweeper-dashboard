## Summary

Keep open: current main and the latest release still only promote parseable serialized tool-call text with arguments, while the reported bare `[tool: exec]` style marker has no executable arguments and is not safely recovered. The open invoke-style repair PR overlaps but does not own this provider-specific bare-marker policy gap.

## What This Changes

The issue reports that some non-Anthropic or compatibility-layer models emit plain text tool-call markers such as `[tool: exec]` instead of structured tool-call blocks, leaving no tool execution and a stalled session.

## Best Possible Solution

Keep this issue open until maintainers choose an explicit recovery policy for bare leaked tool markers, backed by live provider proof and focused regression coverage.

## Reproduction Assessment

Yes for the source-level gap, not for a live provider replay. Current main only promotes parseable text tool-call payloads, while the report's bare marker has no arguments to execute.

## Solution Assessment

Unclear: directly executing a bare marker is not the best fix because it lacks arguments. The safer direction is diagnostics plus retry, failover, forced-tool, or model guidance policy chosen by maintainers.
