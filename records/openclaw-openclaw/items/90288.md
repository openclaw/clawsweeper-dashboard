## Summary

Keep open: current main and v2026.6.8 still do not safely recover or diagnose the reported bare `[tool: exec]`-style marker. Complete serialized text tool calls with arguments are repaired, and an open PR targets a different invoke-style XML shape, but the central no-argument marker case still needs maintainer policy and live provider proof.

## What This Changes

Reported that non-Anthropic/OpenAI-compatible models can emit bare bracketed tool markers as assistant text instead of structured tool calls, leaving no tool execution.

## Best Possible Solution

Keep this issue open until maintainers choose a provider-runtime recovery policy for bare leaked tool markers, preferably diagnostics plus retry/failover or explicit operator guidance rather than executing no-argument assistant text.

## Reproduction Assessment

Source-reproducible but not live-reproduced: current main only promotes serialized text tool calls when a registered tool name and arguments are parseable, while the reporter's evidence shows bare markers without arguments. I did not run a live MiniMax or DeepSeek replay in this read-only review.

## Solution Assessment

Unclear: parsing a bare marker directly is not the best fix because it has no arguments. The safer direction is an explicit recovery and diagnostic policy, with live provider proof before changing retry or fallback behavior.
