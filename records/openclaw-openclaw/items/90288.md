## Label Justifications

- `P1`: Affected provider routes can leave tool-using agent turns stuck with no tool execution, which is a broken agent workflow for real users on the latest release.
- `impact:auth-provider`: The problem is tied to provider/model tool-call compatibility on `anthropic-messages` and `openai-completions` routes.

## Summary

Keep open: current main and v2026.6.1 only repair plain-text tool calls when the marker is followed by a complete JSON or XML-ish argument payload, so the reported bare marker/no-tool-call stall remains a meaningful provider-runtime bug.

## What This Changes

The issue reports that MiniMax and DeepSeek-compatible provider routes sometimes emit plain-text tool markers such as `[tool: exec]`, leaving the text visible while no tool executes.

## Best Possible Solution

Add a provider-safe recovery path for bare leaked tool markers, likely retry/failover or a clear diagnostic plus model-specific guidance, while keeping no-argument assistant text from being executed as tools.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: current code rejects `[tool: exec]` because of the space and rejects bare `[tool:exec]` because no JSON/XML arguments follow. The reporter supplied detailed trajectory data for the provider-side occurrence.

## Solution Assessment

Unclear: detection/retry is plausible, but direct parsing of bare text into tool calls is not the best safe fix because there are no tool arguments. Maintainers should choose a recovery policy that handles the stuck turn without blindly executing arbitrary assistant text.
