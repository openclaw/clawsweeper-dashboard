## Label Justifications

- `P1`: Affected provider/model routes can leave tool-using agent turns stuck with no tool execution, which is a broken user-facing agent workflow.
- `impact:auth-provider`: The failure is tied to provider/model tool-call compatibility on Anthropic Messages and OpenAI-compatible routes.

## Summary

Keep open: current main and v2026.6.6 recover complete serialized text tool calls with arguments, but the reported bare `[tool: exec]` marker still has no safe recovery path and can remain an unsupported provider/model tool-use failure. The remaining fix needs maintainer policy around retry, failover, diagnostics, or provider guidance rather than an automated parser change that executes no-argument assistant text.

## What This Changes

The issue reports MiniMax, DeepSeek, and other non-Anthropic/provider-compatible models emitting bare text tool markers instead of structured tool-use blocks, leaving tool calls unexecuted and sessions stuck.

## Best Possible Solution

Keep this issue open until maintainers choose a provider-runtime recovery policy that handles bare leaked tool markers with retry/failover or clear diagnostics and guidance, without executing no-argument assistant text as a tool call.

## Reproduction Assessment

Source-reproducible but not live-reproduced: current main rejects `[tool: exec]` because of the space and rejects bare `[tool:exec]` because no JSON/XML arguments follow. The provider-side frequency evidence comes from the reporter's trajectory data and screenshots.

## Solution Assessment

Unclear: detection and retry/failover are plausible, but direct parsing of a bare marker is not the best fix because there are no tool arguments. Maintainers should choose an explicit recovery and diagnostic policy before code changes.
