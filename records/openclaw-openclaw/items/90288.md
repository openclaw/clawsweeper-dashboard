## Label Justifications

- `P1`: Affected provider/model routes can leave tool-using agent turns stuck with no tool execution, which is a broken user-facing agent workflow.
- `impact:auth-provider`: The failure is tied to provider/model tool-call compatibility on Anthropic Messages and OpenAI-compatible routes.
- `impact:message-loss`: The intended tool-backed response can be replaced by a visible bare marker with no executed tool result.
- `impact:session-state`: The assistant transcript can imply a tool action while the actual tool-call ledger has no matching invocation.

## Summary

Keep open: current main and latest release still only promote complete serialized text tool calls with arguments, while the reported bare `[tool: exec]` marker remains unsupported and can still leave provider/model tool use stuck or visible as text. A merged related PR covers serialized payload recovery, but not this no-argument bare-marker failure.

## What This Changes

The issue reports MiniMax and DeepSeek-compatible provider routes emitting bare assistant text such as `[tool: exec]` instead of structured tool-use/tool-call blocks, leaving no tool execution and a stuck-looking turn.

## Best Possible Solution

Keep this open for a maintainer-owned provider-runtime policy: recover bare leaked tool markers with retry/failover or clear diagnostics and model/tool-surface guidance, without executing no-argument assistant text as a tool call.

## Reproduction Assessment

Source-reproducible but not live-reproduced: current main rejects `[tool: exec]` because of the space and rejects bare `[tool:exec]` because no JSON/XML arguments follow. The reporter supplied detailed trajectory data and screenshots for the provider-side occurrence.

## Solution Assessment

Unclear: retry/failover/detection is plausible, but direct parsing of a bare marker is not safe because there are no tool arguments. Maintainers should choose an explicit recovery policy instead of blindly executing matching text.
