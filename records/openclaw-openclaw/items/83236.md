## Summary

Keep open: the report is fresh, concrete, and describes a blocking gateway/channel workflow on the shipped macOS iMessage path. Current main has generic channel startup handoff changes, but I did not find high-confidence proof that iMessage startup is fully bounded or isolated; the related Telegram report at https://github.com/openclaw/openclaw/issues/81191 is useful precedent, not a terminal duplicate for this iMessage surface.

## What This Changes

This issue requests hardening iMessage channel `start-account` startup so the gateway event loop and TUI/session routing stay responsive after restart.

## Best Possible Solution

Land a bounded iMessage startup/readiness fix, or a shared channel-startup guard if the root cause is framework-wide, with regression coverage and real macOS proof that gateway RPC/TUI traffic remains responsive during iMessage startup.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter provided strong logs, and source inspection confirms the relevant iMessage startup path, but the macOS event-loop starvation itself still needs real-environment proof on current main.

## Solution Assessment

Unclear. Current main’s generic `start-account-handoff` change is useful but does not by itself prove the iMessage monitor startup path cannot starve the gateway event loop; a safer fix should bound or isolate iMessage startup work and verify the restart/TUI flow.
