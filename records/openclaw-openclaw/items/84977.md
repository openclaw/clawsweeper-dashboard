## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep Gemma tool text buffered until promotion:** `src/plugin-sdk/tool-payload.ts:168-185`
  - body: Adding Gemma parsing here makes final-message promotion possible, but LM Studio's stream buffering still only treats bracket and Harmony prefixes as possible plain-text tool calls. A response starting with `<|tool_call>` will fail `couldStillBePlainTextToolCall` on the first delta and flush raw tool syntax before the final `done` event can replace it, so the leak this PR targets can still happen. Please add the Gemma prefix to that buffering guard and cover the LM Studio stream path.
  - confidence: 0.92
- **[P2] Parse each Gemma tool call independently:** `src/plugin-sdk/tool-payload.ts:309`
  - body: Using `lastIndexOf(GEMMA_CALL_CLOSING)` consumes through the final closing marker in the remaining text. Gemma4 allows multiple calls concatenated without separators, and `parseStandalonePlainTextToolCallBlocks` expects each parser call to return the current block's end so it can advance to the next block; this will collapse multiple calls into one malformed call instead of emitting one tool call per block.
  - confidence: 0.9
- **[P2] Remove the production `any` from the parser:** `src/plugin-sdk/tool-payload.ts:323`
  - body: This production file is covered by `.oxlintrc.json`'s `typescript/no-explicit-any: error`, and the test-only override does not apply here. The new `let value: any` should fail lint; keep the match value typed as `string | undefined` or `unknown` and narrow it before assignment.
  - confidence: 0.95
