## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep Gemma tool text buffered until promotion:** `src/plugin-sdk/tool-payload.ts:63-64`
  - body: Adding the Gemma parser here is not enough for streaming LM Studio runs: `couldStillBePlainTextToolCall` only keeps bracketed and Harmony-looking text buffered, so a stream starting with `<|tool_call>` is flushed as normal text before the final `done` event can promote it. Add the Gemma prefix to the LM Studio buffering guard and cover the streaming path, otherwise users can still see the raw tool-call text in preview channels.
  - confidence: 0.9
- **[P2] Parse each Gemma tool call independently:** `src/plugin-sdk/tool-payload.ts:309`
  - body: This uses the last closing marker in the remaining text, so two adjacent Gemma calls are consumed as one block from the first opener through the final closer. That prevents `parseStandalonePlainTextToolCallBlocks` from emitting the later tool call and can corrupt the first call's arguments; stop at the current closing marker and let the outer loop parse the next block.
  - confidence: 0.91
- **[P2] Strip unpromoted Gemma blocks from visible text:** `src/plugin-sdk/tool-payload.ts:356-359`
  - body: The visible-text stripper still prefilters only bracketed and Harmony patterns, so Gemma text that is unsupported, unregistered, or not promoted by LM Studio returns unchanged and can still leak into user-visible replies. Wire the new prefix into the stripping path and add an assistant-visible-text regression.
  - confidence: 0.86
- **[P2] Remove the production any from the parser:** `src/plugin-sdk/tool-payload.ts:323`
  - body: The repo lint config treats `typescript/no-explicit-any` as an error for production code, so this added `any` should fail the lint gate. Keep the capture value as `string | undefined` or another narrowed type instead.
  - confidence: 0.96
