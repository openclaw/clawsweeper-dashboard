The PR adds Gemma `<|tool_call>call:...<tool_call|>` parsing to the shared tool-payload parser plus a unit test for one `exec` call.

## Best Possible Solution

Land a stricter Gemma parser that updates LM Studio buffering and visible-text stripping, parses one completed block at a time, preserves JSON-compatible argument values, stays lint-clean, and covers parser plus stream-promotion behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main does not parse the Gemma prefix, and LM Studio streaming does not buffer it before promotion. I did not run a live LM Studio/Gemma session in this review.

## Solution Assessment

No, not as currently written. The shared parser is the right general surface, but the maintainable fix must also cover LM Studio stream buffering, visible-text stripping, per-block parsing, typed arguments, and lint-clean code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update Gemma leak guards with the new prefix:** `src/plugin-sdk/tool-payload.ts:63-64`
  - body: Adding the Gemma tokens only to the shared parser leaves the existing leak guards unaware of the new format. LM Studio still flushes streaming text unless it starts with bracket/Harmony prefixes, and the plain-text strip fast path still skips Gemma-shaped text, so `<|tool_call>` can remain visible before final promotion or sanitizer cleanup. Please update those guards and add stream/sanitizer regressions.
  - confidence: 0.93
- **[P2] Parse only the current Gemma tool call:** `src/plugin-sdk/tool-payload.ts:309`
  - body: `lastIndexOf(GEMMA_CALL_CLOSING)` searches the whole remaining response and consumes through the final closing marker. When Gemma emits adjacent tool calls, the standalone parser will collapse them into one malformed block instead of advancing one completed call at a time.
  - confidence: 0.9
- **[P2] Preserve Gemma argument value types:** `src/plugin-sdk/tool-payload.ts:319-323`
  - body: The regex parser stores quoted and bare values as strings and stops bare values at commas or braces, so numeric, boolean, array, or nested-object tool arguments can be truncated or passed with the wrong type. Tool calls should produce JSON-compatible argument values or reject unsupported Gemma payloads instead of silently changing them.
  - confidence: 0.8
- **[P2] Remove the explicit any from production code:** `src/plugin-sdk/tool-payload.ts:323`
  - body: This production file is covered by `.oxlintrc.json`'s `typescript/no-explicit-any: error`, and CI's `check-lint` job is already failing on the new `let value: any`. Keep the value typed as `string | undefined` or `unknown` and narrow it before assignment.
  - confidence: 0.97
