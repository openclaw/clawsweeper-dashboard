## Summary

Keep open: the reported LM Studio Gemma tool-call leak is valid and the screenshots show the simple happy path improving, but the patch still does not parse the documented Gemma argument grammar safely and does not hide unpromoted Gemma blocks on the existing sanitizer path.

## What This Changes

The PR adds Gemma `<|tool_call>call:` parsing to the shared tool-payload parser, LM Studio stream buffering, and one parser unit test.

## Best Possible Solution

Land an end-to-end Gemma parser backed by the official chat-template grammar, with LM Studio promotion and user-visible stripping tests for simple, typed or nested, and unsupported blocks.

## Reproduction Assessment

Yes at source level: current main does not recognize the Gemma opener in the shared parser used by LM Studio promotion, and the PR screenshots show the same real LM Studio flow leaking before and executing after. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the current patch is not the best maintainable fix because it implements a flat regex subset instead of the Gemma argument grammar and does not update the existing visible-text cleanup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse Gemma arguments recursively:** `src/plugin-sdk/tool-payload.ts:319-323`
  - body: The Gemma template can emit nested mappings, arrays, booleans, numbers, and quoted strings, but this regex stops at commas/braces and stores every value as a string. A call with an object or numeric parameter would be promoted with corrupted arguments, so tool execution can receive the wrong payload.
  - confidence: 0.9
- **[P2] Strip unpromoted Gemma blocks from visible text:** `src/plugin-sdk/tool-payload.ts:63-64`
  - body: Adding the Gemma opener to parsing is not enough because `stripPlainTextToolCallBlocks` still returns early unless the text looks bracketed or Harmony-style. If a Gemma block is unsupported, unregistered, or fails promotion, the delivery/history sanitizer can leave tool-call text visible despite the documented policy that unsupported local-model tool requests are hidden.
  - confidence: 0.86
