## Summary

The PR should stay open because the docs topic is useful, but the proposed text still contradicts the current plugin logger contract and makes a command-specific stdout-corruption claim that current main does not support from source inspection.

## What This Changes

The PR adds a `docs/plugins/building-plugins.md` section advising plugin authors to use `api.logger` instead of `console.log` in `register()`.

## Best Possible Solution

Keep the logging guidance, but rewrite it to match the actual `OpenClawPluginApi` logger contract and describe stdout pollution as a general machine-readable CLI output risk unless a current command-specific reproduction is added.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a docs-only PR. The review findings are source-reproducible by comparing the proposed Markdown with the current SDK types and CLI schema path.

## Solution Assessment

No, not as written. The direction is maintainable, but the example should use required `api.logger.info`/`warn`/`error` calls and optional-chain only `debug`, while avoiding the unsupported `openclaw config schema` claim.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the required logger API in the example:** `docs/plugins/building-plugins.md:277-290`
  - body: The new docs say the logger property and its methods are optional and show `api.logger?.info?.(...)`, but `OpenClawPluginApi.logger` is required and `info`, `warn`, and `error` are required methods; only `debug` is optional. This would teach plugin authors the wrong SDK contract.
  - confidence: 0.96

Keep the stdout warning, but phrase it as a general machine-readable CLI output risk unless a current repro is added.
  - confidence: 0.84
