## Summary

Keep open: the docs topic is not already covered on main and is narrow, but the current patch documents the logger contract incorrectly, makes an unsupported command-specific `openclaw config schema` claim, and lacks required real behavior proof.

## What This Changes

The PR adds a logging best-practices section to `docs/plugins/building-plugins.md` telling plugin authors to use `api.logger` instead of `console.log` in `register()`.

## Best Possible Solution

Keep the logging guidance only after rewriting it to match the required logger API and framing stdout pollution as a general machine-readable CLI risk unless a current command-specific reproduction is added.

## Reproduction Assessment

Yes for the review findings: the mismatch is source-reproducible by comparing the added Markdown with `OpenClawPluginApi`, `PluginLogger`, `buildPluginApi`, and the config-schema source path. A runtime bug reproduction is not applicable because this is a docs-only PR.

## Solution Assessment

No, not as written. The direction is useful, but the example should use the required `api.logger.info`/`warn`/`error` contract and optional-chain only `debug`, while generalizing the `openclaw config schema` claim unless the author provides a current repro.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the required logger API in the example:** `docs/plugins/building-plugins.md:277-290`
  - body: The new example and explanation treat `api.logger` and `info` as optional, but `OpenClawPluginApi.logger` is required and `PluginLogger.info`/`warn`/`error` are required; only `debug` is optional. This teaches plugin authors the wrong SDK contract.
  - confidence: 0.96
- **[P3] Generalize the config-schema stdout warning:** `docs/plugins/building-plugins.md:281-297`
  - body: The new section says `console.log` in `register()` corrupts `openclaw config schema`, but current main builds that schema from runtime schema metadata and writes JSON through `writeRuntimeJson` without directly executing plugin `register()`. Keep the warning as a general machine-readable CLI stdout risk unless a current command-specific repro is added.
  - confidence: 0.84
