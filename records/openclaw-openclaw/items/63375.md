## Summary

Keep this PR open. The requested docs direction is useful and current main still lacks a logging best-practices section, but the proposed text has concrete accuracy issues against the current SDK logger contract and config-schema implementation path.

## What This Changes

The PR adds a docs/plugins/building-plugins.md section advising plugin developers to use api.logger instead of console.log because stdout writes can corrupt CLI JSON output.

## Best Possible Solution

Revise the docs addition into concise plugin logging guidance that recommends api.logger over stdout-writing console.log, shows required api.logger.info/warn/error calls and optional api.logger.debug?.(...), and phrases stdout corruption as a general machine-readable CLI output risk unless a current command-specific reproduction is added.

## Reproduction Assessment

Not applicable as a runtime reproduction for this docs-only PR. The review findings are high-confidence by comparing the proposed docs diff with the current OpenClawPluginApi and PluginLogger types plus the current config schema implementation path.

## Solution Assessment

No, not as written. The direction is maintainable, but the proposed wording should be narrowed to the actual SDK contract and current CLI schema behavior before merge.
