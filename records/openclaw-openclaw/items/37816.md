## Summary

Keep this issue open. Current main still does not provide the requested Telegram mode for durable, separate, Web UI-style activity messages. Telegram now has live preview edits, preview tool-progress lines, and `/reasoning` behavior, but those are preview/final-delivery paths rather than a persistent per-activity delivery contract. The related PR in the provided context was closed unmerged and did not implement the feature.

## What This Changes

Request an opt-in Telegram mode that persists Web UI-style agent activity, including reasoning, tool calls/results, and memory operations, as separate messages instead of only typing/final preview behavior.

## Best Possible Solution

Define an explicit opt-in Telegram activity visibility design with defaults unchanged, durable separate messages only for approved event classes, internal and `NO_REPLY` turn suppression, output redaction/truncation, coalescing/rate limiting, config schema/UI/docs updates, and focused Telegram dispatch tests. Reuse existing reply-option callbacks where sufficient; add a narrow generic SDK/channel event seam only if the current callbacks cannot represent the required durable events.
