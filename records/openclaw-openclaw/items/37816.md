## Summary

Keep open: current main has Telegram preview/progress drafts, reasoning visibility controls, and a narrow verbose/tool-progress fallback fix, but it still does not provide the requested durable per-step Telegram activity transcript. The related observer work remains open/draft and does not supersede this Telegram-specific UX and policy request.

## What This Changes

The issue requests an opt-in Telegram mode that renders agent activity as persistent per-step messages with Web UI-like visibility.

## Best Possible Solution

Define an explicit opt-in activity visibility design with redaction, NO_REPLY exclusion, retention rules, batching/rate limits, and a clear choice between a Telegram-owned persistent-message mode and a shared observer contract.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection shows current main supports edited preview/progress drafts and reasoning/verbose controls rather than durable per-step Telegram activity messages.

## Solution Assessment

No complete implementation is on main yet. The suggested config direction is plausible, but the maintainable solution needs product/security design and coordination with the broader observer-mode work before code changes.

## Risks / Open Questions

- Persistent Telegram activity messages could expose reasoning, tool arguments, command output, memory details, local paths, or private context without a clear redaction and retention policy.
- A per-step Telegram transcript could create message volume and Bot API rate-limit pressure unless batching, truncation, and opt-in controls are designed first.
- The architecture overlaps the open observer-mode tracker and draft activity dashboard PR, so maintainers should choose between a Telegram-owned mode and a shared observer contract.
