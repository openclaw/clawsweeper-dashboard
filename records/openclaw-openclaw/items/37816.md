## Summary

Keep open: current main has Telegram tool-progress preview/progress drafts and the narrow verbose fallback was fixed, but it still does not provide the requested durable per-step Telegram activity transcript or Web UI parity mode.

## What This Changes

Request a Telegram mode/config that renders each agent activity step as a persistent separate message while excluding NO_REPLY housekeeping.

## Best Possible Solution

Define an explicit opt-in activity visibility design with redaction, NO_REPLY exclusion, rate/volume limits, and either a Telegram-owned persistent-message mode or a shared observer contract.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection shows current main supports compact edited preview/progress drafts rather than durable per-step Telegram activity messages.

## Solution Assessment

No current implementation is complete. The suggested config direction is plausible, but the maintainable solution needs product/security design and coordination with the broader observer-mode work before code changes.

## Risks / Open Questions

- Persistent Telegram activity messages could expose reasoning, tool arguments, command output, memory details, or local paths without a clear redaction and retention policy.
- A per-step Telegram transcript could create message spam and Bot API rate-limit pressure unless the UX has batching, truncation, and opt-in controls.
- The right architecture overlaps https://github.com/openclaw/openclaw/issues/12831 and https://github.com/openclaw/openclaw/pull/60814, so maintainers should choose whether this is a Telegram-specific mode or a shared observer contract.
