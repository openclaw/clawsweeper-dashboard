## Summary

Keep this open. Current main has improved Telegram preview streaming, tool-progress preview lines, and a separate reasoning preview lane, but it still does not provide the requested durable, separate Telegram activity-message mode with Web UI-style visibility. The linked PR in the discussion was closed unmerged and did not implement the feature.

## What This Changes

Request an opt-in Telegram mode or flag that renders each agent activity step as persistent separate messages, mirroring Web UI activity visibility while excluding internal NO_REPLY housekeeping turns.

## Best Possible Solution

Keep this as explicit bundled Telegram product work: add an opt-in `channels.telegram` activity-message setting with defaults unchanged, preserve NO_REPLY/internal-run suppression, redact and truncate sensitive activity data, rate-limit/coalesce high-volume events, and update config schema, docs, UI/config metadata, and Telegram dispatch tests. Reuse existing reply callbacks where they can model the events, and add only a narrow generic channel/plugin activity seam if the current contract cannot express the missing activity types.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence source-level verification path: current config types, schema, docs, and Telegram dispatch only implement preview-edit streaming and do not expose a durable `full` or `activityMessages` mode.

## Solution Assessment

Not applicable to a proposed patch because this is an open feature request. The requested direction is plausible, but the maintainable solution needs maintainer product/security review before implementation because it intentionally broadens what Telegram receives.
