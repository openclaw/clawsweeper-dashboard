## Label Justifications

## Summary

Current main improves Telegram progress visibility but still does not provide durable, separate activity messages, and the merged Control UI work explicitly leaves this issue as distinct scope. The request is directionally aligned with channel improvements, but raw reasoning/tool parity would cross a sensitive disclosure boundary and needs an explicit product/security decision before implementation.

## What This Changes

Add an opt-in Telegram mode that keeps sanitized agent activity steps as persistent separate messages instead of only showing replaceable previews or the final answer.

## Best Possible Solution

Define a portable typed activity-summary contract that reuses the existing redaction and progress-event concepts, then let Telegram opt into durable delivery of approved summaries only; raw chain-of-thought, raw arguments, and unrestricted tool output should remain excluded.

## Reproduction Assessment

Not applicable: current main already provides temporary Telegram progress previews and reasoning delivery, while persistent per-step activity messages are a new capability rather than broken documented behavior.

## Solution Assessment

No, the proposed raw Web UI parity and Telegram-only flag are not yet the best solution; a sanitized portable activity contract with explicit durability and disclosure policy is safer and avoids duplicating channel-specific logic.

## Risks / Open Questions

- Publishing raw reasoning, tool arguments, outputs, file paths, or memory operations into Telegram would create durable third-party copies of potentially sensitive operational data.
- One persistent message per activity event could cause message spam, Telegram rate-limit pressure, and ordering or duplication problems alongside preview and final-reply delivery.
- The issue's suggested new streaming/config mode is not yet a settled contract; choosing channel-specific configuration before defining a portable sanitized activity surface could create competing behavior paths.
