## Summary

Keep open: this targets a real source-reproducible Skill Workshop UI agent-scope bug, but the branch still forces default/global sessions into an explicit UI-derived agent and lacks real Control UI proof, so it is not merge-ready.

## What This Changes

The PR adds Skill Workshop agent-id state, derives it from the current session/default agent, and sends it on proposal list, inspect, apply, and reject Gateway RPCs.

## Best Possible Solution

Land one canonical UI fix that sends agentId only for explicit agent-scoped sessions, leaves default/global sessions unscoped for Gateway fallback, adds focused coverage, and includes redacted real Control UI/Gateway proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 send Skill Workshop proposal requests from the UI without an agent scope while Gateway and CLI already support scoped workspaces. I did not run a live Control UI repro.

## Solution Assessment

No. Threading agentId through the existing Gateway contract is the right layer, but this implementation should omit agentId for default/global/unparsed sessions instead of forcing an explicit fallback value.
