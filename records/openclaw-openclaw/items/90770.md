## Summary

Keep open: this targets a real Skill Workshop UI bug, but the patch is not merge-ready because it forces an explicit UI-derived agentId for default/global sessions and still lacks live multi-agent Control UI proof.

## What This Changes

The PR adds UI state for a Skill Workshop agent id and sends it with proposal list, inspect, apply, and reject Gateway RPCs.

## Best Possible Solution

Land one canonical Control UI fix that sends `agentId` only for explicit agent-scoped sessions, leaves default/global sessions unscoped for Gateway fallback, adds focused coverage, and includes redacted real Control UI plus Gateway proof.

## Reproduction Assessment

Yes. The source path is high-confidence: current main sends unscoped Skill Workshop proposal RPCs from the UI while Gateway and CLI already support agent-scoped proposal workspaces.

## Solution Assessment

No. Threading `agentId` through the existing Gateway contract is the right layer, but this implementation should omit `agentId` for default/global sessions instead of forcing an explicit fallback value.

- **[P1] Preserve the default proposal fallback:** `ui/src/ui/app-settings.ts:143-145`
  - body: `resolveSkillWorkshopAgentId` always returns a non-null id, so default/global sessions send an explicit `agentId` instead of omitting it.
