## Summary

Keep open: current main and v2026.6.6 still send unscoped Skill Workshop proposal requests from the Control UI, but this PR is not merge-ready because it turns default/global sessions into explicit UI-derived agent routing and still lacks real Control UI plus Gateway proof.

## What This Changes

The PR derives a Skill Workshop agent id from the current Control UI session and threads it through list, inspect, apply, and reject Gateway proposal requests.

## Best Possible Solution

Land one canonical Control UI fix that passes `agentId` only for explicit `agent:<id>:...` sessions, leaves default/global sessions unscoped for Gateway fallback, adds focused coverage, and includes redacted live proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 send unscoped Skill Workshop proposal RPCs from the UI while Gateway and CLI already support agent-scoped proposal workspaces. I did not run a live Control UI plus Gateway repro.

## Solution Assessment

No. Threading the existing optional `agentId` contract through the UI is the right layer, but this implementation should omit `agentId` for default/global sessions instead of forcing an explicit fallback value.
