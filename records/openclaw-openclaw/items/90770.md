## Summary

Keep open: the linked bug is real and current main still lacks the UI agent scope, but this PR is not merge-ready because it always derives a non-null `agentId` for default/global sessions, which bypasses Gateway's configured-default fallback. The external proof is also test/source output only, not after-fix behavior from a real Control UI plus Gateway setup.

## What This Changes

The PR adds UI state for a Skill Workshop agent id and sends it with proposal list, inspect, apply, and reject RPCs.

## Best Possible Solution

Land one canonical Control UI fix that sends `agentId` only for explicit `agent:<id>:...` sessions, leaves default/global sessions unscoped for Gateway fallback, adds focused UI coverage, and includes redacted real Control UI plus Gateway proof.

## Reproduction Assessment

Yes, source-level: current main sends unscoped Skill Workshop proposal RPCs from the UI while Gateway and CLI already support agent-scoped proposal workspaces. I did not run a live Control UI plus Gateway repro.

## Solution Assessment

No. Threading `agentId` through the existing Gateway contract is the right layer, but this implementation should omit `agentId` for default/global sessions instead of forcing an explicit fallback value.
