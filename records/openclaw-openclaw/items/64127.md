## Summary

Keep open. The narrowed circuit-breaker scope is still not implemented on current main, and the latest branch still has blocking integration gaps that keep `quota_exhausted` from reliably reaching and staying in the disabled lane. The earlier session-key/type-guard feedback appears addressed, and the TUI/status-provider work was correctly split to #72557.

## What This Changes

The branch adds a `quota_exhausted` failover/auth-profile reason, routes it into long disabled backoff, emits an `agent:provider_tripped` hook, and updates periodic-limit classification tests.

## Best Possible Solution

Land the narrowed circuit breaker after carrying `quota_exhausted` through status precedence, auth-profile reason scoring, disabled-window routing, and quota-specific probe behavior, with targeted regression tests plus docs/changelog. Keep the TUI/status-provider API work separate in #72557.

## Reproduction Assessment

Yes. A focused test can feed a 429 signal with daily/weekly limit text through `classifyFailoverSignal`, then assert auth-profile disabled reason scoring and model-fallback probe behavior for `quota_exhausted`; current-main tests already prove the old periodic-limit path is still `rate_limit`.

## Solution Assessment

No, not as currently patched. The direction is narrow and maintainable, but the implementation must fix the 429 precedence, unavailable-reason allowlist, and quota probe policy before it solves the reported loop.
