## Summary

Keep open: current main still lacks this quota-exhaustion circuit-breaker behavior, and the PR is an active, useful implementation path, but the current patch is not merge-ready because it still needs real behavior proof, session/lane suspension preservation, public contract docs, and removal of unrelated generated artifacts.

## What This Changes

The PR adds `quota_exhausted` classification/backoff, an `agent:provider_tripped` internal hook and guard, related regression tests, plus generated protocol and e2e-script changes.

## Best Possible Solution

Land a rebased, narrowed quota circuit breaker that preserves session/lane suspension, documents the hook/reason/backoff contract, removes unrelated generated artifacts, passes CI, and leaves account-scoped sibling suppression to the dedicated follow-up.

## Reproduction Assessment

Yes at source level: current main still maps HTTP 429 and periodic usage-window quota text into `rate_limit`, and the discussion includes real incident context. I did not run a live exhausted-provider scenario, so this is not fully reproduced in a real provider setup.

## Solution Assessment

No, not as currently patched. The quota-circuit-breaker direction is plausible, but the branch should preserve lane/session suspension semantics, document the new API contract, remove generated artifacts, pass CI, and include real behavior proof before merge.
