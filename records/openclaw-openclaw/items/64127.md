## Summary

Keep open: current main still treats periodic quota-window failures as rate limits, and this active external PR addresses a real provider-loop problem, but the branch is not merge-ready because it still bypasses the lane-suspension path, lacks docs for a new plugin-visible contract, includes ignored generated output, and has no after-fix real behavior proof.

## What This Changes

The PR adds quota-exhaustion failure classification/backoff, an internal `agent:provider_tripped` hook, related regression tests, generated protocol artifacts, and a small e2e script change.

## Best Possible Solution

Land or replace this with a rebased, narrowed quota circuit breaker that preserves lane suspension semantics, documents the new reason and hook contract, removes generated artifacts, passes CI, and includes redacted after-fix behavior proof.

## Reproduction Assessment

Yes at source level: current main still maps HTTP 429 and periodic quota-window text into `rate_limit`, and the discussion includes real Gemini monthly-cap incidents. I did not reproduce it against a live exhausted provider account.

## Solution Assessment

No, not as currently patched. The direction is plausible, but the patch should preserve `suspend_lanes`, document the new API/behavior contract, remove generated artifacts, pass checks, and include real behavior proof before merge.
