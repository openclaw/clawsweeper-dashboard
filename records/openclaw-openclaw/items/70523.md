## Summary

Keep this PR open. Current main still lacks fallback-aware suppression for raw light/REM inline dreaming blocks, and this branch remains a relevant implementation candidate, but the proposed fix needs another pass because current main now routes cron-triggered narratives through a detached path that does not observe the new fallback result.

## What This Changes

The PR adds memory-core fallback-result plumbing, inline managed-block removal support, contamination detection reuse, regression tests, and a MiniMax FAQ entry to suppress raw dreaming metadata in daily memory files after request-scoped fallback.

## Best Possible Solution

Land a rebased memory-core-only fix that suppresses raw inline light/REM blocks whenever request-scoped fallback successfully writes the diary, including the detached cron path, while preserving separate reports, daily-note formatting, and phase events; split unrelated FAQ docs into a separate change.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce at code level: inline light/REM writes happen before narrative fallback, and the narrative API has no fallback-success signal; the PR adds focused tests for the heartbeat path but not the current cron-detached path.

## Solution Assessment

No. The PR's core direction is narrow and appropriate for the awaited heartbeat path, but it is not yet the best complete fix against current main because cron-triggered detached narratives ignore the new fallback result and can still leave raw inline output.
