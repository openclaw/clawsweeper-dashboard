## Summary

Keep open. Current main still has the race where a prefixed BlueBubbles reply lookup can join a bare in-flight fetch and return before its prefixed alias is cached, while the PR appears to make the narrow plugin-local fix with regression coverage and no blocking security or correctness findings.

## What This Changes

The PR updates BlueBubbles reply-context API fallback to cache validated `p:n/<guid>` aliases when they join an existing bare-message in-flight fetch, with a regression test and changelog entry.

## Best Possible Solution

Land the narrow BlueBubbles plugin fix after maintainer review and normal changed-gate validation, keeping the alias behavior local to the reply-context fallback rather than adding a core cache seam.

## Reproduction Assessment

Yes. On current main, start a bare `msg-race` fallback fetch and concurrently call `p:0/msg-race` for the same account; the prefixed caller returns the existing promise before alias preservation, leaving the exact prefixed cache key absent. The PR's added regression test models that path directly.

## Solution Assessment

Yes. Caching the validated part-index alias from a success continuation on the shared in-flight promise preserves single-request dedupe while recording the joining caller's alias, which is narrower than changing fetch keying or redesigning the reply cache.
