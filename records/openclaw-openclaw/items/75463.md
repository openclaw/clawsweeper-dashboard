## Summary

Keep this PR open. Current main still has the mixed bare/prefixed in-flight reply-context alias race, while the PR applies a narrow BlueBubbles-local fix with a direct regression test and no blocking review finding found in the diff.

## What This Changes

The PR updates BlueBubbles reply-context fallback to cache a validated `p:n/<guid>` alias when that caller joins an existing bare in-flight fetch, adds a concurrency regression test, and adds an Unreleased changelog fix note.

## Best Possible Solution

Land the narrow BlueBubbles alias-caching fix after normal PR review and gates, keeping the behavior inside the BlueBubbles plugin and preserving the existing reply-cache and typed-client fallback contracts.

## Reproduction Assessment

Yes. On current main, a bare `msg-race` fallback fetch can enter `inflight` first, then a concurrent `p:0/msg-race` call returns the shared promise before alias preservation, leaving the prefixed cache key absent; the PR's added test models this path directly.

## Solution Assessment

Yes. Adding a success continuation for validated prefixed callers that join an in-flight bare fetch preserves request dedupe while recording the missing alias, which is narrower than rekeying the in-flight map or redesigning the reply cache.
