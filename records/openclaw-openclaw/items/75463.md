## Summary

Keep open. Current main still has the in-flight dedupe race for mixed bare and prefixed BlueBubbles reply-context lookups, while the updated PR addresses that path with a narrow cache continuation, regression coverage, and the previously missing changelog entry. I found no blocking correctness or security issue in the proposed diff.

## What This Changes

The PR updates BlueBubbles reply-context fallback so a `p:n/<guid>` lookup that joins an existing bare in-flight fetch caches its prefixed alias, adds a concurrency regression test, and adds a changelog fix note.

## Best Possible Solution

Land the narrow BlueBubbles alias-caching fix after PR checks, keeping the behavior local to the BlueBubbles plugin and preserving the existing typed-client fallback and cache contracts.

## Reproduction Assessment

Yes. On current main, start a bare `msg-race` fallback fetch and concurrently call the same account with `p:0/msg-race`; the prefixed caller returns the existing promise before any alias write, leaving the exact prefixed cache key absent. The PR's added regression test models that path directly.

## Solution Assessment

Yes. Caching a validated part-index alias from a success continuation on the shared in-flight promise preserves single-request dedupe while recording the joining caller's alias, which is narrower than redesigning the reply cache or fetch keying.
