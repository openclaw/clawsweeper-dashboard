## Summary

Keep open. Current main still returns shared BlueBubbles fallback fetches before alias preservation, while this PR applies a narrow continuation-based alias cache write with focused regression coverage and no blocking functional or security finding found.

## What This Changes

This PR updates the BlueBubbles reply-context fallback to cache validated `p:n/<guid>` aliases for callers that join an existing bare-message in-flight fetch, adds regression coverage, and records the fix in the changelog.

## Best Possible Solution

Land the narrow BlueBubbles plugin fix after maintainer review and any needed rebase, keeping alias handling local to the reply-context fallback cache path.

## Reproduction Assessment

Yes. On current main, start a bare `msg-race` fallback fetch and concurrently call `p:0/msg-race` for the same account; the prefixed caller returns the existing promise before alias preservation, leaving the exact prefixed cache key absent.

## Solution Assessment

Yes. Recording a validated part-index alias from a success continuation on the shared in-flight promise preserves single-request dedupe and is narrower than changing fetch keying or redesigning the reply cache.
