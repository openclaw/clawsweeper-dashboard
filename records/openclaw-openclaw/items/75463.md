## Summary

Keep this PR open. Current main still has the mixed bare/prefixed in-flight race, and the PR is a narrow BlueBubbles-local fix with focused regression coverage and no blocking correctness or security findings found in this read-only review.

## What This Changes

The PR updates BlueBubbles reply-context fallback so a validated `p:n/<guid>` lookup that joins an existing bare in-flight fetch caches its prefixed alias, adds a concurrency regression test, and adds a changelog fix entry.

## Best Possible Solution

Land the narrow BlueBubbles alias-cache fix after normal PR checks, keeping the behavior inside the plugin-owned reply-context fallback and preserving the existing in-flight request dedupe contract.

## Reproduction Assessment

Yes. On current main, start a bare `msg-race` fallback fetch and concurrently call the same account with `p:0/msg-race`; the second call sanitizes to the same in-flight key and returns before the prefixed alias write path, leaving the exact prefixed cache key absent.

## Solution Assessment

Yes. The PR records only validated part-index aliases from the shared successful fetch result, which preserves single-request dedupe and is narrower than changing cache keying or redesigning the reply-context fallback.
