## Summary

Keep this PR open. Current main still has the mixed bare and prefixed in-flight reply-context alias race, and the PR is a narrow BlueBubbles plugin fix with focused regression coverage, changelog coverage, and no concrete correctness or security blocker found in the provided diff.

## What This Changes

The PR updates BlueBubbles reply-context API fallback so validated `p:n/<guid>` aliases are cached when their caller joins an existing bare-message in-flight fetch, and adds a regression test plus changelog entry.

## Best Possible Solution

Land the narrow BlueBubbles plugin fix after maintainer review and normal changed-gate validation, keeping the alias preservation local to the reply-context fallback rather than adding a core cache or channel-level seam.

## Reproduction Assessment

Yes. On current main, start a bare `msg-race` fallback fetch and concurrently call `p:0/msg-race` for the same account; the prefixed caller returns the shared in-flight promise before alias preservation, leaving the exact prefixed cache key absent. The PR's new regression test models that path directly.

## Solution Assessment

Yes. The PR preserves bare-id request dedupe while caching the joining caller's validated prefixed alias from a success continuation, which is narrower and more maintainable than changing fetch keying or redesigning the reply cache.
