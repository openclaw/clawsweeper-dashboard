## Summary

Current main and the shipped v2026.6.6 release already normalize the reported standalone Telegram `<code class="language-...">` leak before send, while preserving fenced `<pre><code class="language-...">` blocks; the broader open angle-bracket reasoning-tag truncation issue is a separate remaining path.

## What This Changes

The issue reports Telegram exposing literal `<code class="language-...">` HTML instead of rendering the text as a formatted code block.

## Best Possible Solution

Keep the shipped Telegram formatter behavior and close this narrow report; leave the distinct strict reasoning-tag truncation work tracked separately at https://github.com/openclaw/openclaw/issues/49104.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence source reproduction: raw `<code class="language-text">` input is normalized to `<code>` before Telegram HTML send.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix because it removes the unsafe standalone classed code tag while preserving OpenClaw's fenced-code language behavior.
