## Summary

Current main already covers the reported Telegram leak: raw standalone `<code class="language-...">` is normalized before send, valid fenced-code language blocks are preserved, and Telegram sends use HTML parse mode. The proof commit is on current main only; I could not verify it in the latest tagged release.

## What This Changes

The issue asks OpenClaw to stop leaking literal Telegram code-language HTML tags and render them as formatted Telegram code blocks.

## Best Possible Solution

Keep the current main formatter/send behavior and let the next release carry the Telegram normalization fix to users.

## Reproduction Assessment

No high-confidence current-main repro remains. Source and regression coverage show the reported raw code-language tag shape is normalized before Telegram send; I did not run live Telegram in this read-only review.

## Solution Assessment

Yes. The current fix is narrow: normalize unsupported standalone code-language attributes at the Telegram formatter boundary, preserve the supported `<pre><code class="language-...">` form, and send with HTML parse mode.
