## Summary

Keep open: current main still does not expose per-conversation identity to provider `createStreamFn` hooks, and this PR is the active focused implementation with sufficient runtime proof. The remaining blocker is maintainer acceptance of the additive provider SDK field names and local-use privacy boundary, not a concrete contributor-facing code defect.

## What This Changes

The PR adds optional `sessionId` and `sessionKey` to the provider `createStreamFn` context, threads them through agent-turn, compaction, `/btw`, and Google transport registration paths, and documents/tests the hook contract.

## Best Possible Solution

Land the additive context seam after maintainer API-contract sign-off, keeping one-off completion paths identity-free and documenting these identifiers as request-scoped local plugin state.

## Reproduction Assessment

Yes. Source inspection on current main shows session identity exists in the relevant agent flows but is not exposed through `ProviderCreateStreamFnContext`; I did not run the live GlueClaw two-conversation repro.

## Solution Assessment

Yes, pending maintainer API sign-off. Optional fields on the existing provider `createStreamFn` context are the narrowest maintainable seam for custom provider transports; plugin-side hashing or cached extra-param workarounds would be weaker.
