## Summary

Keep this open. Current main has useful lower-level controls, but it still documents a single trusted-operator gateway model and does not implement shared-gateway admin/user/guest RBAC, per-user credential isolation, or audit semantics; the latest maintainer comment also explicitly kept this tracker open.

## What This Changes

Request shared-deployment RBAC with admin/standard/guest roles, sensitive config protection, per-user credentials or environments, authentication/session controls, and audit logging.

## Best Possible Solution

Keep this as the broad product/security tracker. A complete direction should define caller identity, role or scope authorization for sensitive config and credential operations, per-user or per-tenant secret isolation, audit logging, channel/group identity mapping, documented trust boundaries, and regression coverage; if maintainers choose an external-wrapper or ClawHub-first path, document the missing or sufficient plugin/API surfaces explicitly.

## Reproduction Assessment

Not applicable as a feature/security-model request. Source and docs verification give a high-confidence current-main check: the shared-gateway RBAC fields and admin/user/guest roles are absent, while the documented model remains trusted-operator per gateway.

## Solution Assessment

Unclear until maintainers choose the product/security model. The safest solution is a maintainer-led design or RFC that splits core gateway identity/authorization seams from external wrapper or plugin work, rather than closing this or queueing one broad implementation PR.

## Risks / Open Questions

- Partial controls such as trusted-proxy allowUsers, operator scopes, config redaction, SecretRefs, per-agent auth stores, sandbox/tool policy, and sender-scoped group tool policy can be mistaken for full multi-user RBAC.
- Community wrappers mentioned in the discussion, including Pinchy, teamclaw, and openclaw-mission-control, are useful prior art but do not prove the OpenClaw core trust boundary or extension APIs are complete for this feature.
- The request spans security/product design across gateway identity, channel caller mapping, config/secrets APIs, tool authorization, and audit logging, so a narrow automated fix would likely choose the wrong boundary.
