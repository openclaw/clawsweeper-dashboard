## Summary

Keep open. Current main has useful groundwork: packaged docs are included, agents receive a local docs path in the system prompt, and `config.schema.lookup` exists for config fields. The requested native local-first docs retrieval surface is still missing: there is no Gateway/agent `docs.search`, `docs.get`, docs API lookup, or changed-since method, and the CLI docs search remains live-doc backed. The linked live-doc drift report #48920 is complementary evidence, not a duplicate closure path.

## What This Changes

The issue requests a release-packaged, version-matched OpenClaw documentation bundle plus first-class local-first docs retrieval tools for agent-guided onboarding.

## Best Possible Solution

Define a versioned docs manifest/search index shipped with releases, expose maintained local-first docs search/get/schema/API lookup to agents through a deliberate core/Gateway contract, keep live Mintlify as an explicit latest-docs fallback, and document/test installed-version versus latest-version behavior.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path for the current gap across package allowlisting, docs path resolution, prompt guidance, Gateway tool actions, Gateway RPC methods, and the live-backed CLI docs command.

## Solution Assessment

Unclear until maintainers choose the index and API contract. The local-first retrieval direction is appropriate, but prompt-only guidance or the current live `openclaw docs` command is not the narrowest complete solution for agent-guided onboarding.
