## Summary

Keep open. Current main has useful groundwork: the package includes the docs tree, agents can resolve a local docs path, the system prompt tells agents to consult local docs first, and `config.schema.lookup` gives path-scoped config field docs. The central request remains unimplemented because there is still no native local-first docs retrieval surface such as docs search/get/API/changed-since for agents, and the existing `openclaw docs` command is explicitly a live docs search. The related open live-doc drift report #48920 and the reporter's downstream scraped-wiki experience support the product problem rather than superseding it.

## What This Changes

Request to ship version-matched bundled docs with releases and expose local-first native docs retrieval tools or APIs for agent-guided onboarding.

## Best Possible Solution

Define a versioned docs manifest plus local search/get index shipped with releases, expose a maintained local-first retrieval contract through the agent or Gateway surface, keep live Mintlify as an explicit latest-docs fallback, and document/test the installed-version versus latest-version behavior.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. The high-confidence verification path is current-main source/docs inspection: packaged docs, path guidance, and config schema lookup exist, but native local-first docs search/get/API/changed-since methods do not.

## Solution Assessment

Unclear until maintainers choose the retrieval contract and bundle/index format. The local-first direction is appropriate, but current prompt-only guidance plus live `openclaw docs` search is not the complete solution requested here.
