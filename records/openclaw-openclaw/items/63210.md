- `P1`: Confirmed main-session length truncation can leave incomplete tool arguments or corrupted written output in real user workflows.
- `impact:data-loss`: The report centers on incomplete or corrupted file/tool output being accepted without a hard failure signal.
- `impact:session-state`: The affected behavior is agent turn/session state deciding whether a truncated assistant response is complete, failed, or recoverable.

## Summary

Keep open. Current main has adjacent mitigations, but it still does not treat provider `stopReason: "length"` as a main-session incomplete-output condition before tool dispatch, so the reported recovery gap remains.

## What This Changes

Detect provider output truncation in main interactive agent sessions, block unsafe truncated tool calls, and add bounded model-visible recovery/config behavior.

## Best Possible Solution

Define and implement a documented main-session truncation contract that treats provider `length` as incomplete output before tool dispatch, blocks unsafe truncated tool calls, and uses bounded recovery semantics aligned with subagent policy.

## Reproduction Assessment

Yes, at source level: current transports normalize provider truncation to `length`, while the main-session incomplete-turn and tool-result guards do not treat `length` as a recovery or fail-closed condition. A live provider cutoff replay was not run in this read-only review.

## Solution Assessment

Unclear as a final design. Detection and fail-closed handling before tool execution is the right boundary, but recovery modes, retry budgets, and circuit-breaker behavior need maintainer runtime-policy judgment before implementation.
