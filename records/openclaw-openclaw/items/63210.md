## Summary

Keep open. Current main still has the main-session truncation recovery gap: providers can surface length/incomplete termination, but the interactive WebSocket conversion and embedded-runner recovery logic do not consistently preserve that signal or block suspect length-truncated tool calls before dispatch.

## What This Changes

Request detection and recovery for main interactive agent turns that stop because of output length, especially when truncation affects tool-call arguments.

## Best Possible Solution

Add a focused Phase 1 main-session recovery path that preserves provider length/incomplete metadata, detects length-terminated turns with tool-call activity before dispatch, refuses suspect truncated tool execution, and injects concise model-facing recovery feedback.

## Reproduction Assessment

Yes. The issue is source-reproducible by synthesizing an OpenAI WebSocket response.completed with status "incomplete" plus function-call output, or a transport assistant message with stopReason "length" plus tool-call activity, then checking that current main does not preserve/recover before normal runner handling.

## Solution Assessment

Yes for a narrow first repair. Runtime-level detection before tool dispatch is the maintainable fix; forwarding larger maxTokens or relying on downstream schema rejection reduces some failures but does not protect valid-looking truncated arguments.
