## Summary

Keep open. Current main still has a real main-session recovery gap: several provider paths can surface or encode length/incomplete termination, but the interactive runner does not consistently preserve that signal through WebSocket conversion or block suspect length-truncated tool calls before dispatch.

## What This Changes

Request detection and model-facing recovery for main interactive agent turns that hit provider output length limits, especially before executing truncated tool-call arguments.

## Best Possible Solution

Add a focused Phase 1 main-session recovery path: preserve provider length/incomplete metadata through message conversion, detect length-terminated turns with tool-call activity before dispatch, refuse suspect truncated tool execution, and inject a concise recovery/continuation instruction for the model. Leave retry budgets and circuit breakers as a separate design follow-up.

## Reproduction Assessment

Yes. The issue has production confirmation, and a deterministic test can synthesize an OpenAI WebSocket response.completed with status "incomplete" plus function-call output, or a transport message with stopReason "length", then assert the main runner blocks suspect tool execution and adds recovery feedback.

## Solution Assessment

Yes for a narrow first repair. Runtime-level detection before tool dispatch is the maintainable fix; increasing maxTokens or relying on downstream schema rejection reduces some failures but does not protect valid-looking truncated arguments.
