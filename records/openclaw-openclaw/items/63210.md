## Label Justifications

- `P1`: Confirmed main-session length truncation can leave incomplete tool arguments or corrupted written output in real user workflows.
- `impact:data-loss`: The report centers on incomplete or corrupted file/tool output being accepted without a hard failure signal.
- `impact:session-state`: The affected behavior is agent turn/session state deciding whether a truncated assistant response is complete, failed, or recoverable.

## Summary

Keep open. Current main has useful adjacent mitigations, but it still lacks a main-session policy for provider `stopReason: "length"` that blocks unsafe truncated tool execution and gives bounded model-visible recovery feedback.

## What This Changes

Request detection and recovery for main interactive agent output truncation, especially when length-terminated tool-call arguments may be incomplete or corrupt.

## Best Possible Solution

Add a documented main-session truncation policy that treats provider `length` as incomplete output before tool dispatch, blocks unsafe truncated tool calls, prompts bounded recovery, and coordinates config semantics with the open subagent policy tracker.

## Reproduction Assessment

Yes, at source level: current transports normalize provider truncation to `length`, while the main-session incomplete-turn and tool-result guards do not treat `length` as a recovery or fail-closed condition. I did not run a live provider truncation replay in this read-only review.

## Solution Assessment

Unclear as a final design. The detection and fail-closed direction is sound, but recovery modes, retry budgets, and circuit-breaker behavior need maintainer runtime-policy judgment before implementation.
