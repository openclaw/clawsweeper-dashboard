## Summary

Keep open. Current main still carries the provider `length` signal, but the main-session incomplete-turn and pending-tool-call paths do not treat `stopReason: "length"` as an incomplete output condition before tool dispatch, and the issue discussion includes a confirming main-session truncation case.

## What This Changes

Request main interactive sessions to detect provider output truncation, preserve the provider stop reason, block or recover unsafe truncated tool calls, and add bounded recovery/retry policy.

## Best Possible Solution

Define a documented main-session truncation contract that treats provider `length` as incomplete output before tool dispatch, blocks unsafe truncated tool calls, and uses bounded recovery semantics aligned with subagent policy.

## Reproduction Assessment

Yes, at source level: current transports normalize provider truncation to `length`, while the main-session incomplete-turn and tool-result guards do not treat `length` as a recovery or fail-closed condition. A live provider cutoff replay was not run in this read-only review.

## Solution Assessment

Unclear as a final design. Detection and fail-closed handling before tool execution is the right boundary, but recovery modes, retry budgets, and circuit-breaker behavior need maintainer runtime-policy judgment before implementation.
