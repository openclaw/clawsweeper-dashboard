## Summary

Keep open. Current main has adjacent mitigations, but it still lacks the requested main-session `stopReason: "length"` policy that preserves the truncation signal, prevents unsafe truncated tool execution, and gives bounded model-visible recovery feedback.

## What This Changes

Request main interactive-agent recovery for provider output truncation, including preserved metadata, guarded tool execution, recovery prompts, and bounded config/retry policy.

## Best Possible Solution

Add a documented main-session truncation policy that treats provider `length` as incomplete output before tool dispatch, blocks unsafe truncated tool calls, prompts bounded recovery, and keeps retry/circuit-breaker behavior behind explicit runtime policy.

## Reproduction Assessment

Yes at source level: current transports normalize provider truncation to `length`, while the main-session incomplete-turn and tool-result guards do not treat `length` as a recovery or fail-closed condition. I did not run a live provider truncation replay in this read-only review.

## Solution Assessment

Unclear as a final design. The detection and fail-closed direction is sound, but the config modes, retry budgets, and circuit breaker need maintainer runtime-policy judgment before an automated fix is safe.
