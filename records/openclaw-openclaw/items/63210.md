## Summary

Keep open. Current main still has the reported main-session gap: several transports can surface provider length truncation, but the main interactive runner does not consistently preserve that signal through OpenAI WebSocket conversion or use it before tool dispatch to block suspect truncated tool calls and steer the model to retry.

## What This Changes

Add detection and model-facing recovery for main interactive agent sessions when provider output truncation interrupts text or tool-call parameters.

## Best Possible Solution

Main interactive sessions should preserve provider length metadata, treat length-terminated turns with tool-call activity as suspect before dispatch, refuse unsafe tool execution, and add concise model-facing recovery feedback. The broader retry/circuit-breaker configuration should remain a follow-up design item.

## Reproduction Assessment

Yes. The issue has a confirmed released-build example, and a deterministic test can synthesize an OpenAI WebSocket response.completed with status incomplete plus a function_call, or an Anthropic/OpenAI transport event with stopReason length, then assert the runner does not dispatch the suspect tool call and adds recovery feedback.

## Solution Assessment

Yes for the narrow Phase 1 path. Runtime-level detection before tool dispatch is the maintainable first fix; increasing maxTokens or relying on downstream schema rejection only reduces frequency and does not prevent valid-but-truncated tool arguments.
