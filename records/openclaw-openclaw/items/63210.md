## Summary

Keep open. Current main has adjacent mitigations, including OpenAI-compatible `maxTokens` forwarding, but it still does not implement the requested main-session `stopReason: "length"` policy that preserves truncation metadata, blocks unsafe tool execution, and gives bounded model-visible recovery feedback. The PathCourse comment points to an external inference-layer workaround/example, not an OpenClaw core fix.

## What This Changes

The issue requests main-agent detection and recovery for provider output truncation when `stopReason: "length"` can leave tool calls or text incomplete.

## Best Possible Solution

Add a documented main-session truncation policy that preserves the provider length signal, blocks unsafe truncated tool calls before execution, gives bounded model-visible recovery feedback, and keeps broader retry or circuit-breaker behavior behind explicit maintainer policy.

## Reproduction Assessment

Yes at source level: provider `length` is normalized, but the main-session incomplete-turn and tool-result guards do not treat `length` as an incomplete tool-call or recovery condition. I did not run a live provider truncation replay in this read-only review.

## Solution Assessment

Unclear as a final design. The detection and fail-closed direction is sound, but the requested config modes, retry budgets, and circuit breaker require maintainer runtime-policy judgment before implementation.
