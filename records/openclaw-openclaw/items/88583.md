## Summary

Keep open: the PR targets a real session-state corruption class, but the patch can make a valid follow-up lifecycle event look stale after preflight compaction rotates the session id, so it should not merge as-is.

## What This Changes

The PR adds an optional sessionId to agent lifecycle events/run context and skips session lifecycle persistence when the event owner sessionId differs from the current session row.

## Best Possible Solution

Refresh or register the lifecycle run context after any preflight/session rotation, then keep the sessionId guard with regression coverage for both reset-stale events and legitimate rotated follow-up runs.

## Reproduction Assessment

Yes, source-reproducible: followup-runner stamps the old run.sessionId into AgentRunContext before runPreflightCompactionIfNeeded, and that helper can later update the run to a new sessionId before lifecycle events are emitted.

## Solution Assessment

No, not as written. The sessionId guard is the right shape for reset-stale events, but it must also keep the run context in sync with legitimate pre-run session rotations.
