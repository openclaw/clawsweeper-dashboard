## Summary

Keep open: the PR has sufficient after-patch startup proof, but the current patch regresses the Teams monitor startup contract by turning listen failures into a successful null-app result. The claimed `expressApp.listen` root cause is not supported by Express 5.2.1 source, and hard-coded bind behavior overlaps with the separate configurable host-binding PR.

## What This Changes

This PR changes Microsoft Teams webhook startup to use an explicit Node HTTP server, bind to `0.0.0.0`, add startup error logging, and make small handler callback edits.

## Best Possible Solution

Preserve the Teams monitor fail-fast startup contract, keep improved startup logging only if startup errors still reject or rethrow, and leave configurable bind-host behavior to the dedicated host-binding path.

## Reproduction Assessment

Yes for the review finding: source inspection of the PR diff shows startup errors become a null-app success result, while current main and lifecycle coverage require rejection on listen failure. No high-confidence reproduction was established for the PR's claimed original startup failure.

## Solution Assessment

No: explicit HTTP server construction and clearer logging could be acceptable, but swallowing startup failures is not the best fix. The safer repair is to log and still reject or rethrow, with bind-host configurability handled by https://github.com/openclaw/openclaw/pull/63347.
