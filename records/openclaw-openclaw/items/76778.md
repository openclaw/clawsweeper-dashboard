## Summary

Keep open: current main still has only the internal hook-runner timeout seam and plugin-authored `timeoutMs`; there is no user-facing config schema, type, docs, or global-runner wiring for per-hook timeout overrides.

## What This Changes

The issue requests a schema-backed user config path for per-hook hook timeout overrides that populate the hook runner timeout maps.

## Best Possible Solution

Define a bounded config contract, wire it into global hook runner initialization, and document precedence against plugin-owned `timeoutMs`.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection confirms current main lacks a user config path while the internal runner override seam and plugin-owned `timeoutMs` exist.

## Solution Assessment

Unclear pending maintainer product choice: the runner seam makes the implementation feasible, but the public path, valid hook-name validation, min/max bounds, reload behavior, and void-vs-modifying scope need a deliberate config contract decision.
