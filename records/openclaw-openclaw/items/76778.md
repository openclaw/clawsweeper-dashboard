## Summary

Keep open: current main has the internal hook-runner timeout maps and plugin-author `timeoutMs`, but no schema, type, docs, or global-runner wiring that lets user config populate per-hook timeout overrides.

## What This Changes

The issue requests a user-configurable per-hook timeout map for plugin hook execution, such as `plugins.hookTimeouts.<hookName>` or a similar public config path.

## Best Possible Solution

If accepted, add a bounded schema-backed hook-timeout config contract, wire it into global hook runner initialization, update config/help/docs, and document precedence against plugin-owned `timeoutMs`; otherwise steer affected plugins toward their own bounded timeout config.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection confirms the internal runner seam and plugin-owned timeout path exist while the requested user config surface is absent on current main.

## Solution Assessment

Unclear pending maintainer product choice: the runner seam makes this mechanically feasible, but the public config path, validation bounds, and global-versus-plugin-owned scope should be decided before implementation.
