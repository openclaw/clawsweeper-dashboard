## Summary

Keep open: the shared delivery-profile RTL isolation is a plausible central fix with strong Telegram proof, but merging it as-is can make Discord leak internal `analysis:`/`commentary:` lines that contain RTL text because the new RLI/PDI prefix is added before Discord's post-sanitizer runs.

## What This Changes

The PR adds a shared RTL isolation helper, applies it to assistant-visible delivery text only, adds shared sanitizer tests, and replaces the TUI-local helper with the shared helper.

## Best Possible Solution

Keep the shared delivery-profile isolation approach, but first make Discord remove or recognize internal channel lines before RLI/PDI wrapping can hide their prefixes, then land only after maintainers accept the delivered-byte contract.

## Reproduction Assessment

Yes. The new Discord regression is also source-reproducible from the ordering of `sanitizeAssistantVisibleText` before `stripDiscordInternalChannelLines`.

## Solution Assessment

No, not as submitted. Reusing the shared delivery sanitizer is likely the right central seam for the RTL bug, but the PR must preserve downstream channel scrubbers such as Discord's internal-line filter before it is the best fix.
