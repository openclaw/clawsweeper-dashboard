## Summary

Keep open. Current main still has the no-retry fetch/cache path, and open PR #74553 already references this issue with closing syntax and targets the same narrow repair.

## What This Changes

The issue asks for bounded transient retry when media-understanding fetches Discord audio attachments so temporary CDN/network failures do not drop voice notes.

## Best Possible Solution

Land PR #74553 or an equivalent shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary, with regression coverage for transient failures and fail-fast carve-outs.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: route a Discord audio attachment URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one cache fetch call and fails before transcription starts.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, not Discord-specific retry logic, while preserving the existing fail-fast cases.
