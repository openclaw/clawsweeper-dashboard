## Summary

Keep open. Current main still has a source-reproducible single-shot remote media fetch path for Discord audio preflight, and open PR #74553 is already the linked unmerged fix path.

## What This Changes

Add bounded retry/backoff for transient remote media fetch failures before audio media-understanding transcription starts.

## Best Possible Solution

Review and land PR #74553 or an equivalent shared retry implementation at the fetchRemoteMedia and MediaAttachmentCache boundary while preserving fail-fast behavior for SSRF blocks, maxBytes, aborts/timeouts, and non-retryable HTTP responses.

## Reproduction Assessment

Yes. Source inspection shows a Discord audio URL reaches MediaAttachmentCache.getBuffer, which calls fetchRemoteMedia once with no general retry policy before transcription can start.

## Solution Assessment

Yes. The narrow maintainable solution is a shared bounded retry policy at the remote media fetch/cache boundary, and PR #74553 is already the active linked path for that fix.
