## Summary

Keep open. Current main still uses a single shared remote-media fetch for the Discord audio preflight path, and #74553 is an open, unmerged closing PR for the requested retry behavior.

## What This Changes

Add bounded retry/backoff for transient media-understanding remote attachment fetch failures before Discord audio transcription.

## Best Possible Solution

Land #74553 or an equivalent narrow shared retry implementation at the fetchRemoteMedia/MediaAttachmentCache boundary while preserving fail-fast handling for 4xx, SSRF blocks, caller aborts/timeouts, and maxBytes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main performs one cache fetch call before surfacing the failure.

## Solution Assessment

Yes. A shared retry policy at the remote media fetch/cache boundary is the narrowest maintainable fix because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific policy.
