## Summary

Keep open. Current main still uses the shared media-understanding attachment cache without a general transient retry policy, and open PR #74553 is already linked as the unmerged fix path.

## What This Changes

The issue requests bounded retries for transient remote media fetch failures before Discord audio transcription gives up.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the fetchRemoteMedia and MediaAttachmentCache boundary while preserving SSRF, maxBytes, 4xx, and caller-abort fail-fast behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: route a Discord audio URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one cache fetch call with no general retry policy.

## Solution Assessment

Yes. The best fix is a shared bounded retry policy at the remote media fetch/cache boundary, not Discord-specific retry logic, because the same cache serves audio preflight and other media-understanding consumers.
