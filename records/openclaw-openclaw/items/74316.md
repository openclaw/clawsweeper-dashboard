## Summary

Keep open. Current main still routes Discord voice-note audio through the shared media-understanding attachment cache without a general transient retry, and open PR #74553 is the linked fix path that should be reviewed or landed before this issue closes.

## What This Changes

Add bounded retry/backoff for transient remote media fetch failures before audio transcription so Discord voice notes are not dropped after one failed download.

## Best Possible Solution

Review and land PR #74553, or an equivalent shared retry implementation at fetchRemoteMedia and MediaAttachmentCache, while keeping the existing decision summary stable.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord audio URLs reach MediaAttachmentCache.getBuffer, which calls fetchRemoteMedia once without a retry predicate or multiple attempts.

## Solution Assessment

Yes. A bounded shared retry at the remote media fetch/cache boundary is the narrowest maintainable fix; Discord-specific retry logic would miss other media-understanding consumers of the same cache.
