## Summary

Keep open. Current main still performs a single shared media-understanding remote fetch for normal URLs, and the linked implementation PR #74553 is open/unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for bounded retry/backoff around transient remote media fetch failures so Discord voice-note audio reaches media-understanding transcription instead of being dropped after one failed download.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the `fetchRemoteMedia` / `MediaAttachmentCache` boundary, with coverage for transient fetch throws, 5xx responses, response-body read failures, and fail-fast behavior for 4xx, SSRF/policy blocks, caller aborts/timeouts, and `maxBytes`. Keep the existing `(X/Y)` decision summary as attachment success/total and expose retry detail separately if needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send a remote Discord audio attachment through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches `MediaAttachmentCache.getBuffer` and calls `fetchRemoteMedia` with no retry policy.

## Solution Assessment

Yes. A shared retry at the media fetch/cache boundary is the narrowest maintainable fix because it covers Discord audio preflight and normal media-understanding consumers without introducing Discord-specific retry behavior.
