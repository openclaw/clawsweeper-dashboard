## Summary

Keep open. Current main still routes Discord audio preflight through the shared media-understanding cache, and that cache performs a single normal fetchRemoteMedia call without a generic transient retry policy. PR #74553 is open, unmerged, and explicitly tracks this issue, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

Add bounded retry/backoff for transient remote media fetch failures in media-understanding so Discord audio attachments are not dropped before transcription.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the fetchRemoteMedia/MediaAttachmentCache boundary, with focused tests for transient fetch errors, transient body-read errors, 5xx responses, and fail-fast carve-outs plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: send a Discord audio attachment URL through preflight/media-understanding, make the shared remote fetch fail transiently, and current main reaches one MediaAttachmentCache.getBuffer fetchRemoteMedia call before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to solve this because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
