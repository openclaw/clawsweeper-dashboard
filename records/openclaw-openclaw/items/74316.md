## Summary

Keep open. Current main still has a source-level single-attempt remote media fetch path for Discord voice-note transcription, and PR #74553 is open, unmerged, and specifically targets this issue with the shared retry fix.

## What This Changes

The issue requests bounded transient retry for remote media attachment fetches used before Discord voice-note audio transcription starts.

## Best Possible Solution

Review and land PR #74553 or an equivalent shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary.

## Reproduction Assessment

Yes. Source inspection shows Discord audio URLs flow into shared transcription, and a transient `fetchRemoteMedia` failure before transcription is reachable with one default fetch attempt on current main.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, while preserving existing fail-fast carve-outs and stable attachment summary logging.
