## Summary

Keep open. Current main still routes Discord preflight audio and normal media-understanding remote attachments through a shared cache/fetch path that has no general transient retry, and the linked open PR is an implementation candidate rather than a reason to close the issue before merge.

## What This Changes

Add bounded transient retries to remote media fetches used by media-understanding so Discord voice-note downloads are not lost after a single transient failure.

## Best Possible Solution

Review and land PR #74553, or replace it with an equivalent narrow fix at the shared fetchRemoteMedia/MediaAttachmentCache boundary with focused tests for transient failures, fail-fast carve-outs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: give MediaAttachmentCache a remote attachment URL, make the shared fetch fail transiently, and current main reaches one fetchRemoteMedia call with one default guarded attempt before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to solve this because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
