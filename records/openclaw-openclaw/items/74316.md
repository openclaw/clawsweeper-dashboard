## Summary

Keep open. Current main still performs a single shared remote-media fetch for the Discord audio preflight path, and open PR #74553 is the unmerged implementation path, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Add bounded retry/backoff for transient shared media-understanding remote media fetch failures before audio transcription gives up.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the `fetchRemoteMedia`/`MediaAttachmentCache` boundary while preserving fail-fast behavior for 4xx, SSRF blocks, caller aborts/timeouts, and maxBytes.

## Reproduction Assessment

Yes. A high-confidence source-level repro is to route a remote Discord audio URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main performs one cache fetch call before surfacing the failure.

## Solution Assessment

Yes. A shared retry policy at the remote media fetch/cache boundary is the narrowest maintainable solution because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific recovery policy.
