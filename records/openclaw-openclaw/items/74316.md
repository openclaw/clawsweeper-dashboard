## Summary

Keep open. Current main still has single-shot remote media fetches for media-understanding audio, and open PR #74553 closes this issue but has not merged, so the issue should remain open until that implementation lands.

## What This Changes

Request bounded retry/backoff for transient remote media fetch failures before Discord audio attachments reach transcription.

## Best Possible Solution

Review and land PR #74553 or an equivalent shared retry implementation at the fetchRemoteMedia and MediaAttachmentCache boundary, while keeping the existing attachment summary stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send a Discord audio URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one default fetch attempt with no general retry policy.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared bounded retry policy at fetchRemoteMedia/MediaAttachmentCache, with 4xx, SSRF/policy, maxBytes, and caller abort failures kept fail-fast; the linked open PR is the active candidate for that shape.
