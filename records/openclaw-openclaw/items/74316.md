## Summary

Keep open. Current main still performs a single shared remote-media fetch before audio transcription, and PR #74553 is open and unmerged with closing syntax for the same retry fix.

## What This Changes

The issue asks for bounded transient retry for media-understanding remote media fetches so Discord voice-note attachments are not dropped after one fetch failure.

## Best Possible Solution

Land PR #74553 or an equivalent shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary while keeping the existing attachment success summary stable.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: pass a Discord audio attachment URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main rethrows after the single cache fetch before transcription starts.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, not Discord-specific retry logic, while preserving fail-fast carve-outs.
