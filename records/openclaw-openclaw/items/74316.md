## Summary

Keep open. Current main still has the single-shot media-understanding remote fetch path, and open PR #74553 is the linked unmerged implementation path for the same retry boundary.

## What This Changes

The issue requests bounded retry for transient remote media fetch failures before Discord audio transcription starts.

## Best Possible Solution

Land PR #74553 or an equivalent shared retry implementation at `fetchRemoteMedia` and `MediaAttachmentCache`, preserving fail-fast behavior for 4xx, SSRF blocks, aborts, and maxBytes failures.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: pass a Discord audio URL through media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one cache fetch call with no general retry policy.

## Solution Assessment

Yes. The narrowest maintainable fix is the linked shared fetch/cache retry path, not Discord-specific retry logic, while keeping existing fail-fast carve-outs and stable attachment summary logging.
