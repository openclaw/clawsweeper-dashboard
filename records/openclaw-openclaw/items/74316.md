## Summary

Keep open. Current main still routes Discord voice-note media through the shared media-understanding cache without a general transient retry, and PR #74553 is open, unmerged, and linked with closing syntax, so this issue should remain open until that fix lands or is replaced.

## What This Changes

Add bounded retry for transient remote media fetch failures in media-understanding so Discord audio attachments are not dropped before transcription starts.

## Best Possible Solution

Land PR #74553 or an equivalent shared bounded retry at the fetchRemoteMedia and MediaAttachmentCache boundary while keeping the existing attachment success summary stable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord preflight builds MediaUrls, transcribeFirstAudio runs audio media-understanding, MediaAttachmentCache.getBuffer calls fetchRemoteMedia, and a transient fetch failure or 5xx is not retried on current main.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, not Discord-specific retry logic, with existing fail-fast carve-outs preserved.
