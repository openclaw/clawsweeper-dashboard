## Summary

Keep open. Current main still routes Discord voice-note preflight through the shared media-understanding attachment cache without a general transient retry, and open PR #74553 already uses closing syntax for this issue but is not merged.

## What This Changes

Request bounded transient retry for remote media fetches so Discord audio attachments are not permanently dropped after one fetch failure.

## Best Possible Solution

Review and land PR #74553, or an equivalent shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Discord audio attachment URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main makes one cache fetch call and rethrows before transcription starts.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, not Discord-specific logic, while preserving fail-fast carve-outs.
