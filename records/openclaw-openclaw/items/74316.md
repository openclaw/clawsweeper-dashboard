## Summary

Keep open. Current main still routes Discord voice-note URLs through the shared media attachment cache without a bounded retry path, and open PR #74553 explicitly closes this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Add bounded retry/backoff to remote media fetches so transient Discord audio attachment fetch failures do not drop voice notes before transcription.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the fetchRemoteMedia/MediaAttachmentCache boundary, with focused coverage for transient fetch/read failures, 5xx retries, and the fail-fast carve-outs plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: route a Discord audio attachment URL through preflight/audio transcription and make the shared remote fetch fail transiently; current main reaches one MediaAttachmentCache.getBuffer fetchRemoteMedia call before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to solve this because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific retry policy.
