## Summary

Keep open. Current main still routes Discord voice-note URLs through the shared media-understanding attachment cache with a single default remote fetch, and PR #74553 is the open unmerged fix path.

## What This Changes

The issue requests bounded retry/backoff for transient remote media fetch failures before audio transcription, especially for Discord voice-note URLs.

## Best Possible Solution

Land PR #74553 or an equivalent shared retry implementation at the fetchRemoteMedia and MediaAttachmentCache boundary, with stable attachment summary logging and separate retry metadata if needed.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: pass a Discord audio URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one cache fetch call with no general retry policy.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared bounded retry policy at the remote media fetch/cache boundary, not Discord-specific retry logic, while preserving the existing fail-fast carve-outs.
