## Summary

Keep open. Current main still routes Discord voice-note URLs through the shared media-understanding cache with no general transient retry, and the linked PR #74553 is open and unmerged; the later discussion correctly narrows this to the fetch/cache boundary rather than the `(0/1)` log summary.

## What This Changes

Request bounded retries for transient remote media fetch failures in media-understanding so Discord voice-note transcription does not lose attachments before provider STT runs.

## Best Possible Solution

Land PR #74553 or an equivalent shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary, keeping the attachment success summary stable and exposing retry metadata separately if needed.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: send a Discord audio attachment URL through preflight/media-understanding and make the guarded fetch fail transiently or return 5xx; current main reaches one cache fetch call and rethrows before transcription starts.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the remote media fetch/cache boundary, not Discord-specific retry logic, while preserving the existing fail-fast carve-outs.
