## Summary

Keep open. Current main still has ordinary media-understanding URL fetches going through one guarded `fetchRemoteMedia` call, and the linked closing PR #74553 is open and unmerged.

## What This Changes

Request bounded retry/backoff for transient remote media fetch failures before Discord audio attachments reach transcription, while keeping 4xx, SSRF, maxBytes, and abort failures fail-fast.

## Best Possible Solution

The end state should be a shared bounded retry policy at the remote media fetch/cache boundary with tests for transient retries and fail-fast carve-outs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a remote Discord audio URL through preflight/transcription and make the guarded fetch fail transiently or return 5xx; current main makes one cache fetch before surfacing failure.

## Solution Assessment

Yes. A shared retry policy at `fetchRemoteMedia` or `MediaAttachmentCache` is the narrowest maintainable fix because it covers Discord audio preflight and other media-understanding URL consumers without adding Discord-specific behavior.
