## Summary

Keep open. Current main still has the reported single-shot remote media fetch path, and open PR #74553 already references this issue with closing syntax and targets the same shared fetch/cache boundary.

## What This Changes

Add bounded transient retry for media-understanding remote attachment fetches so Discord voice notes are not lost before transcription starts.

## Best Possible Solution

Review and land PR #74553 or an equivalent shared bounded retry at the remote media fetch/cache boundary, with regression coverage for transient failures and fail-fast carve-outs.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: route a Discord audio attachment URL through preflight/media-understanding and make the guarded remote fetch fail transiently or return 5xx; current main reaches one fetch path and rethrows before transcription starts.

## Solution Assessment

Yes. The narrow maintainable fix is a shared bounded retry in fetchRemoteMedia/MediaAttachmentCache rather than Discord-specific retry logic, while keeping known non-retryable classes fail-fast.
