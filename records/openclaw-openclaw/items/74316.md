## Summary

Keep open. Current main still lacks retry/backoff at the shared media-understanding remote fetch/cache boundary, and open PR #74553 is the paired closing implementation path rather than a reason to close this issue before merge.

## What This Changes

Request to add bounded retry/backoff for transient remote media fetch failures so Discord audio attachments are not permanently dropped before transcription.

## Best Possible Solution

Land PR #74553 or an equivalent narrow shared retry implementation at the fetchRemoteMedia/MediaAttachmentCache boundary, preserving fail-fast behavior for SSRF blocks, caller aborts, maxBytes, and unrecoverable HTTP responses while adding focused tests and the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: route a Discord audio attachment URL through preflight transcription, make the shared remote fetch fail transiently, and current main performs one MediaAttachmentCache.getBuffer -> fetchRemoteMedia call before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable solution because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy or changing provider transcription behavior.
