## Summary

Keep open: this is useful and proof-backed work, but the quarantine proxy currently redirects only lifecycle method calls while the already-resolved engine still reports the failed plugin's metadata. That can leave memory, compaction, and owner-scoped runtime behavior on the wrong path after fallback, and the protected maintainer label means the compatibility decision should remain in human review.

## What This Changes

The PR adds process-local quarantine and legacy fallback for failing non-legacy context engines, exposes quarantine state in health responses, expands kitchen-sink fixture coverage, and documents the failure-isolation contract.

## Best Possible Solution

Land only after the quarantine path makes effective engine identity, owner, and capabilities consistent with the legacy fallback for the active proxy and maintainers accept the documented fallback policy.

## Reproduction Assessment

Yes, from source inspection. A focused regression can resolve a non-legacy engine, trigger an `assemble()` failure, and then observe that the same proxy still reports plugin metadata while fallback methods are used.

## Solution Assessment

No, not as-is. The quarantine/fallback direction is plausible, but the active proxy must also present effective fallback metadata or the existing callers will keep making session-state decisions from the failed plugin.
