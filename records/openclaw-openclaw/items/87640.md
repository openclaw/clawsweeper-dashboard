## Summary

Keep open for normal maintainer review: the patch looks coherent and well covered, but this maintainer-labeled PR deliberately changes context-engine failure handling from fail-closed to legacy fallback, which is compatibility and session-state sensitive.

## What This Changes

The PR adds process-local quarantine for failing non-legacy context engines, reports quarantine state in CLI/Gateway health, expands kitchen-sink ClawHub fixture coverage, and updates context-engine docs/tests.

## Best Possible Solution

Land only after maintainers accept the new quarantine contract as the intended default for broken selected context-engine plugins, with the current health visibility and Testbox proof preserved in the merge record.

## Reproduction Assessment

Yes, source inspection and added tests show the current-main behavior: non-default context-engine resolution already falls back in some cases, while lifecycle failures previously propagated and could keep the active path on a failing engine. I did not execute the tests during this read-only review.

## Solution Assessment

Mostly yes: centralizing quarantine in the registry is the maintainable place for this failure-isolation behavior. The remaining decision is policy-level: maintainers need to accept legacy fallback as the default for selected plugin-engine failures.
