## Summary

Keep open: current main and v2026.6.8 still restart unhealthy channel accounts on the health-monitor cadence without a startup-progress or bounded restart-wave guard, and the linked fix PR was closed unmerged after maintainer feedback that the global event-loop gate was too broad.

## What This Changes

The issue reports a fixed-interval gateway restart cascade under Telegram multi-account churn when the channel health monitor restarts a stalled start phase during event-loop saturation.

## Best Possible Solution

Keep this issue open for a bounded gateway health-monitor fix, likely using restart scheduling or startup-progress state plus proof against the actual restart path.

## Reproduction Assessment

No high-confidence live reproduction is available; the report says it was observed once and not reproduced deterministically. Source inspection shows a credible current-main path from Telegram `connected:false` startup state through health policy to health-monitor restart cadence.

## Solution Assessment

No for the closed related PR: maintainers rejected the one-sample global event-loop gate as too broad and incomplete. A better solution should bound restart waves or carry startup-progress state while preserving recovery for genuinely dead channels.
