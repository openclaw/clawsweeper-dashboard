## Summary

Keep open: current main and v2026.6.9 still restart unhealthy channel accounts on the channel-health-monitor cadence without a bounded restart scheduler, startup-progress signal, or event-loop-aware guard, and the only linked fix PR was closed unmerged after maintainer feedback that its global event-loop gate was too broad.

## What This Changes

The issue reports a Telegram multi-account churn scenario where gateway channel health monitoring can restart-cascade on a fixed interval and produce repeated 502s while startup is stalled under event-loop saturation.

## Best Possible Solution

Keep this issue open for a maintainer-approved gateway health-monitor fix, likely a bounded restart scheduler or startup-progress signal with proof against the actual restart path.

## Reproduction Assessment

No high-confidence live reproduction is available; the issue says it was observed once and not reproduced deterministically. Source inspection shows a credible current-main path from Telegram `connected: false` startup state through health policy to health-monitor restart cadence.

## Solution Assessment

No for the closed related PR: maintainer feedback rejected the one-sample global event-loop gate as too broad and still vulnerable to restart waves. A better fix should bound restart scheduling or carry startup-progress state while preserving recovery for genuinely dead channels.
