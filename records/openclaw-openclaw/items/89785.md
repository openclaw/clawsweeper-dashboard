## Summary

Keep open: current main and the latest release still restart unhealthy channel accounts on the health-monitor cadence without a startup-progress or restart-wave guard, and the related attempted fix was closed unmerged after reviewer feedback that the global event-loop gate was too broad.

## What This Changes

The issue reports a Telegram multi-account churn scenario where gateway health monitoring appears to restart-cascade during saturated channel startup, causing repeated 502s and wedged sessions.

## Best Possible Solution

Keep the issue open and pursue a narrower gateway health-monitor fix, likely a bounded restart scheduler or startup-progress signal with regression coverage and proof against the actual restart path, instead of reviving the one-sample global event-loop gate.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows a credible current-main path: Telegram polling can remain connected:false past connect grace, readiness can fail, and the health monitor can restart accounts on the reported cadence.

## Solution Assessment

No for the closed related PR: the global event-loop gate was rejected as too broad and incomplete. A better fix should bound restart scheduling or carry startup-progress state so multi-account churn cannot queue a restart burst.
