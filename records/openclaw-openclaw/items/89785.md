## What This Changes

Investigate a reported gateway restart cascade where Telegram multi-account startup churn appears to trigger fixed-interval health-monitor restarts during event-loop saturation.

## Best Possible Solution

Keep this issue open for a bounded gateway health-monitor restart scheduler or startup-progress signal with proof against the actual restart path, while preserving recovery for genuinely dead channels.

## Reproduction Assessment

No high-confidence live reproduction is available; the issue says it was observed once and not deterministically reproduced. Source inspection shows a credible current-main path from Telegram `connected: false` startup state through health policy to health-monitor restarts on the monitor cadence.

## Solution Assessment

No for the closed related PR: maintainer feedback rejected the one-sample global event-loop gate as too broad and not sufficient for multi-account restart bursts. The better direction is a bounded restart scheduler or startup-progress signal that preserves recovery for truly dead channels.
