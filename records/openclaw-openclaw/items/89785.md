## Summary

Keep open: current main and v2026.6.8 still restart unhealthy channel accounts on the health-monitor cadence without a startup-progress or bounded restart-wave guard, and the linked attempted fix was closed unmerged after maintainer feedback that the global event-loop gate was too broad.

## What This Changes

The issue asks OpenClaw to prevent gateway health-monitor restart cascades during Telegram multi-account slow-start or churn conditions.

## Best Possible Solution

Keep this issue open for a bounded gateway health-monitor fix, likely using restart scheduling or startup-progress state plus regression coverage and proof against the actual restart path.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows a credible current-main path: Telegram polling can stay connected:false past connect grace, readiness can fail, and the health monitor can restart accounts on the reported cadence.

## Solution Assessment

No for the closed related PR: the global event-loop gate was rejected as too broad and incomplete. A better fix should bound restart scheduling or carry startup-progress state so multi-account churn cannot queue a restart burst.
