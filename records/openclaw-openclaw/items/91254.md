## Summary

Keeping this open: current main still has blind plugin text replacement surfaces and an open linked fork PR, but the exact reported bash tool-argument mutation and an `ocplatform` rewrite were not proven in OpenClaw main.

## What This Changes

Prevent plugin or rebrand text transforms from rewriting filesystem paths or bash/tool-call arguments in a way that can silently route file operations to the wrong directory.

## Best Possible Solution

Audit the text-transform contract across provider, CLI, and Codex boundaries, then either land a focused compatibility-safe fix with regression coverage or close the linked fork PR if the report belongs to a private/non-main runtime.

## Reproduction Assessment

No high-confidence current-main reproduction for the exact bash tool-call misrouting. Source inspection does show blind text replacements can rewrite prompt/output strings when a plugin registers a matching transform, but no `ocplatform` rule or normalized tool-call argument transform was found.

## Solution Assessment

Unclear. The safest path is to first define the supported plugin text-transform contract across CLI/provider/Codex surfaces; the linked fork PR is useful evidence but not clearly the best fix because it changes output handling only and adds a broad env escape hatch.
