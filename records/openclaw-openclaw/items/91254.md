## Summary

Keep open: current main and v2026.6.9 still apply blind plugin text replacements to CLI/provider prompt and output strings. The linked fork PR is useful candidate context, but the permanent transform boundary is compatibility-sensitive and still needs maintainer review.

## What This Changes

The issue reports that plugin/backend text transforms can rewrite legacy brand strings inside path-like prompt/tool/output text, silently sending file operations to a rebranded directory.

## Best Possible Solution

Land a maintainer-sponsored fix that preserves cosmetic transforms but prevents path/tool-bearing text from being rewritten across CLI/provider surfaces, with regression coverage and live proof for a dual-directory setup.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction of the exact macOS/Signal dual-directory setup is available. Source inspection does show that a registered replacement rewrites path-like strings through the same helper before CLI/provider transport and after output parsing.

## Solution Assessment

Unclear until maintainers choose the boundary. The linked path-aware fork PR is plausible review input, but it does not by itself settle input transforms, provider transforms, native tool semantics, or public plugin API compatibility.
