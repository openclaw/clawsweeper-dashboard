## Summary

Keep this open: current main still contains source-visible write paths where direct-message channel handlers pass `mainSessionKey` into `updateLastRoute` under isolated DM routing, and the related fix PRs are closed unmerged. The recent Signal release repro strengthens that this stale delivery-context class is still observable, not only a historical report.

## What This Changes

The issue requests that `session.dmScope: per-channel-peer` keep inbound direct-message last-route delivery context scoped to the channel-specific session instead of contaminating the main session.

## Best Possible Solution

Apply the existing inbound last-route policy consistently at each direct-message update source, keeping main-session writes only when `lastRoutePolicy` is `main` and adding focused regression coverage for representative affected channels.

## Reproduction Assessment

Yes, via source inspection on current main. Under `per-channel-peer`, routing produces a session-scoped key, but multiple direct-message channel paths still ask the recorder to write channel delivery context to `mainSessionKey`.

## Solution Assessment

Yes, the narrow maintainable fix is to prevent the bad write at the `updateLastRoute` payload source by using the existing route-policy helper. Read-side fallback hardening and legacy-session cleanup are useful mitigations but do not remove the record-stage contamination path.
