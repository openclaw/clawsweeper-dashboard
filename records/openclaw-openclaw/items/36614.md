## Summary

Keep this open: current main still has the write-side contamination path the maintainer comment called out, because several channel record payloads still target the main session while the recorder writes delivery context to that requested target.

## What This Changes

The issue asks for inbound DM last-route writes under `session.dmScope: per-channel-peer` to stay scoped to the isolated channel-peer session instead of updating `agent:main:main`.

## Best Possible Solution

Route inbound last-route writes through the existing `lastRoutePolicy` boundary so isolated DM sessions receive their own delivery context while shared-main routes still update the main session.

## Reproduction Assessment

Yes, from current-main source inspection. A `per-channel-peer` DM resolves to an isolated route session, but iMessage and other channel paths still pass `mainSessionKey` as the last-route target and `recordInboundSession` writes delivery context there.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because affected channel write paths bypass `resolveInboundLastRouteSessionKey`. The narrow maintainable fix is to use that helper or equivalent shared policy when constructing inbound record updates.
