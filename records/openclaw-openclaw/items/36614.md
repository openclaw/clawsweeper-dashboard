## Summary

Keep this open: current main still has direct-message write paths that target the main session for non-main DM scopes, so the write-side contamination described here is not fully fixed. The later session-initialization cleanup mitigates some stale-route fallout, and an open Slack-only PR covers one surface, but iMessage, Signal, Matrix, Mattermost, and similar paths still need the write target corrected at the source.

## What This Changes

Prevent direct-message channel `updateLastRoute` writes under isolated DM scopes from storing channel-specific delivery context on `agent:main:main`.

## Best Possible Solution

Use the existing inbound last-route policy when building direct-message route-update payloads so isolated DM scopes write to the isolated session and shared-main DM mode keeps the current owner-pin behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `session.dmScope: per-channel-peer`, affected channel monitors use an isolated route session but still pass the main session key into `recordInboundSession`, which then persists the channel delivery context to that target.

## Solution Assessment

Yes. Fixing the write target at channel payload construction, or through a shared typed helper used there, is narrower and more maintainable than relying on later session-initialization cleanup.
