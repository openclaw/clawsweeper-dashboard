## Summary

Keep this open: current main still has write-side last-route contamination paths where inbound per-channel-peer DMs can ask the session recorder to write channel delivery context onto the main session. The merged related work only covered read-side fallback behavior, while the direct write targets remain in iMessage, Signal, Matrix, Mattermost, LINE, and a Discord component path.

## What This Changes

The issue reports that per-channel-peer inbound direct messages still write channel-specific last-route delivery context onto the main session.

## Best Possible Solution

Centralize inbound DM last-route target selection on `resolveInboundLastRouteSessionKey` across affected channels, preserving main-session writes only when `route.lastRoutePolicy` is `main`.

## Reproduction Assessment

Yes, via source inspection. Current routing marks per-channel-peer direct messages as session-scoped, but affected inbound channels still pass the main session key to a recorder that persists the supplied delivery context to that target.

## Solution Assessment

Yes, preventing the bad write at the `updateLastRoute` payload source or a shared helper boundary is the narrow maintainable fix. Read-side fallback hardening does not remove the stored contamination path.
