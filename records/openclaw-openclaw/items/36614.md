## Summary

Keep this open: current main still has source-visible direct-message paths that write inbound channel delivery metadata to the shared main session while per-channel-peer routing produces an isolated session key, and the issue has a fresh release repro plus closed unmerged fix attempts.

## What This Changes

The issue reports that per-channel-peer inbound DMs can still contaminate the shared main session with channel-specific updateLastRoute delivery context.

## Best Possible Solution

Scope direct-message last-route writes through the existing route lastRoutePolicy helper across affected channel plugins, keeping main-session writes only for routes that intentionally collapse to main and adding regression coverage.

## Reproduction Assessment

Yes, via source inspection on current main. With per-channel-peer, route resolution produces an isolated session key, but multiple direct-message channel paths still pass mainSessionKey into updateLastRoute.

## Solution Assessment

Yes, the best fix is the narrow write-side correction at each updateLastRoute payload source using the existing resolveInboundLastRouteSessionKey policy. Read-side fallbacks reduce symptoms but do not stop the stored-session contamination reported here.
