## Summary

Keep this open: current main still has iMessage and several other inbound channel paths building direct-message `updateLastRoute` payloads with `mainSessionKey`, while `recordInboundSession` persists `deliveryContext` to that requested target. The related fix attempts are closed and unmerged, and the merged related PR only addressed the read-side WebChat stale-route leakage, not this write-side contamination.

## What This Changes

The issue asks OpenClaw to stop channel-specific inbound DM `updateLastRoute` writes from contaminating the main session when `session.dmScope` is `per-channel-peer`.

## Best Possible Solution

Centralize channel `updateLastRoute` target selection on `resolveInboundLastRouteSessionKey` across the affected inbound paths, preserving main-session writes only when `route.lastRoutePolicy` is `main`.

## Reproduction Assessment

Yes, via source inspection. With `session.dmScope: per-channel-peer`, routing produces a session-scoped key, but current iMessage and other direct-message paths still ask `recordInboundSession` to write channel delivery context to the main session.

## Solution Assessment

Yes, preventing the bad write at the `updateLastRoute` payload source is the narrow maintainable fix. Read-side fallback hardening and session cleanup do not eliminate the record-stage contamination path.
