## Summary

Keep open. Current main still permits the write-side contamination the report describes: scoped direct-message routes are classified for session-scoped last-route updates, but the central recorder will still write channel delivery metadata to a caller-supplied main-session target, and several channel paths still supply that main-session target for direct messages.

## What This Changes

The issue requests preventing per-channel-peer inbound channel messages from writing channel-specific delivery metadata onto the main session.

## Best Possible Solution

Finish the write-side routing fix by making inbound last-route target selection consistently honor route.lastRoutePolicy. Direct-message routes with lastRoutePolicy="session" should persist delivery metadata to the scoped inbound session, while lastRoutePolicy="main" should preserve the existing shared-main behavior; representative channel writers and direct updateLastRoute callers should use the same helper/policy boundary.

## Reproduction Assessment

Yes. A high-confidence static and unit-test reproduction exists on current main: per-channel-peer resolves DMs to scoped session keys with lastRoutePolicy="session", but iMessage and other direct-message writers still pass the main session to updateLastRoute, and the central recorder test currently expects that cross-target deliveryContext write.

## Solution Assessment

No proposed patch is currently open. The safest repair is not to suppress all cross-target updates blindly, but to route last-route writes through resolveInboundLastRouteSessionKey or an equivalent generic helper everywhere direct-message writers build updateLastRoute.
