## Summary

Keep this open: current main still has the write-side route contamination path for iMessage, and similar channel call sites still pass the main session as the direct-message last-route target instead of the isolated route session.

## What This Changes

The issue asks OpenClaw to stop per-channel-peer inbound DM route recording from writing channel-specific delivery context onto the shared main session.

## Best Possible Solution

Route inbound last-route writes through the existing session-scoped policy helper for isolated DMs, preserve main-session writes only when the resolved policy is `main`, and add regression coverage across the affected channel/session boundary.

## Reproduction Assessment

Yes, from a high-confidence source-level path rather than a live run. With `session.dmScope: per-channel-peer`, routing resolves the DM to an isolated session, the iMessage monitor still passes the main session to `updateLastRoute`, and the session store writes the channel delivery context to that target.

## Solution Assessment

No, the current path is not the best solution because it bypasses the existing `resolveInboundLastRouteSessionKey` policy helper. The narrow maintainable fix is to use that helper or an equivalent shared guard at the channel record boundary while keeping shared-main behavior intact.
