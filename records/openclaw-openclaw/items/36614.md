## Summary

Keep this open: current main still has a source-reproducible write-side route contamination path, with iMessage and several other channel paths passing the main session as the inbound DM last-route target while the central recorder writes delivery context to that requested target.

## What This Changes

The issue asks OpenClaw to stop per-channel-peer inbound channel messages from writing channel-specific updateLastRoute delivery context onto the main session.

## Best Possible Solution

Route inbound last-route writes through the existing lastRoutePolicy boundary, using the isolated session for per-channel-peer DMs and preserving main-session writes only when the resolved policy is main.

## Reproduction Assessment

Yes, from current-main source inspection. A per-channel-peer DM resolves to an isolated route session, but iMessage still sends updateLastRoute to the main session and recordInboundSession writes delivery context to that target.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it bypasses resolveInboundLastRouteSessionKey on affected channel write paths. The narrow maintainable fix is to use that helper or an equivalent shared policy at inbound record construction.
