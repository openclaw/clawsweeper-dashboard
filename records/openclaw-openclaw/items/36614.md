## Summary

Keep open: current main still has write-side direct-message paths that target the shared main session for `updateLastRoute` while per-channel-peer routing uses isolated session keys, and a fresh Signal repro on v2026.5.12 reports the same contamination class.

## What This Changes

The issue reports that per-channel-peer inbound direct messages write channel-specific last-route delivery metadata into the shared main session.

## Best Possible Solution

Apply the existing inbound last-route session-key policy consistently across remaining direct-message channel paths, with regression coverage proving isolated DM sessions do not write channel delivery metadata to `agent:<id>:main`.

## Reproduction Assessment

Yes, at source level. With `per-channel-peer`, route resolution produces an isolated session key, while multiple direct-message handlers still pass `mainSessionKey` into `updateLastRoute`.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing `resolveInboundLastRouteSessionKey` policy at the remaining write sites or an equivalent central contract, while preserving intentional main-session behavior.
