## Summary

Keep this open: current main still has direct-message record payloads, including iMessage, targeting the main session while `recordInboundSession` persists `deliveryContext` to the requested target before dispatch; the related fix PRs are closed/unmerged and no open canonical item supersedes this write-side bug.

## What This Changes

When `session.dmScope` is `per-channel-peer`, inbound channel DMs should keep last-route delivery context on the isolated channel/peer session instead of writing it to the main session.

## Best Possible Solution

Use the shared inbound last-route policy when building channel `updateLastRoute` payloads so session-scoped DMs write to their isolated session and only shared-main DMs write to `agent:main:main`, with focused regression coverage.

## Reproduction Assessment

Yes, via source inspection. With `session.dmScope: per-channel-peer`, iMessage resolves an isolated route session, then passes `mainSessionKey` to `updateLastRoute`, and the recorder writes channel `deliveryContext` to that main-session target before dispatch.

## Solution Assessment

Yes, preventing the write at the source is the narrow maintainable solution. Session-initialization cleanup is useful migration behavior, but it does not eliminate the record-stage contamination path.
