## Summary

Keep this PR open, but it is not merge-ready: the patch fixes the reported false negative by silently unpinning stored exact npm specs in a helper used by both dry-run and live updates, which conflicts with the current documented and tested pin contract.

## What This Changes

The branch changes `src/plugins/update.ts` so `resolveNpmUpdateSpecs` strips exact-version npm selectors to bare package names before update probes.

## Best Possible Solution

Preserve exact-version pin semantics by default, and solve the false negative through the explicit unpinned npm-spec path or a maintainer-approved latest-advisory mode with aligned docs, tests, and real CLI proof.

## Reproduction Assessment

Yes, at source level: current main passes the recorded exact npm spec into the update probe, and live npm metadata shows exact specs resolve to the pinned version while bare package names resolve to the current default. I did not run a full CLI reproduction because this review is read-only.

## Solution Assessment

No. The patch is a plausible root-cause fix, but it is not the best solution because it silently unpins stored exact versions in a helper shared by dry-run and live updates despite current docs and tests saying those pins remain sticky.
