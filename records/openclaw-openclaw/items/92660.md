## Summary

Keep this PR open, but the current patch is not merge-ready because it fixes the reported false-negative by silently changing the documented exact-version pin contract for id-based plugin updates.

## What This Changes

The branch changes `src/plugins/update.ts` so `resolveNpmUpdateSpecs` strips exact-version selectors from npm install specs before plugin update probes.

## Best Possible Solution

Preserve exact-version pin semantics by default, and solve the false-negative either through the existing explicit unversioned npm-spec path or a maintainer-approved latest-advisory mode with aligned docs and tests.

## Reproduction Assessment

Yes at source level: current main passes the recorded exact npm spec into the update probe, and live `npm view` shows that exact specs resolve to the pinned version while bare package names resolve to the current default. I did not run the full CLI because this review is read-only.

## Solution Assessment

No. The patch is a plausible root-cause fix, but it is not the best solution because it silently unpins stored exact versions for both dry-run and live updates despite current docs/tests saying those pins remain sticky.
