## Summary

Keep this PR open: current main still has the automatic reset/session-memory gap, but this branch is not merge-ready because it awaits potentially expensive hook work on the inbound reply path and lacks the required changelog entry. The same behavior is also being explored in #61675 and #71792, so a maintainer should choose the canonical hook contract before landing one implementation.

## What This Changes

The PR emits `session:idle_reset` and `session:daily_reset` internal hook events during stale session rollover, updates the bundled session-memory hook to handle them, and adds related docs and tests.

## Best Possible Solution

Land one canonical implementation that preserves session memory for automatic daily and idle rollovers with stable transcript access, bounded or off-path hook work, docs, tests, and a changelog entry, then close or supersede the competing implementations.

## Reproduction Assessment

Yes. Source inspection on current main shows stale sessions rotate/archive in `initSessionState`, while session-memory registers and handles only `command:new` and `command:reset`.

## Solution Assessment

No, not as-is. The direction is correct, but awaiting internal hook dispatch inside session initialization can put model-backed session-memory work on the reply-critical path; a bounded asynchronous path or stable transcript handoff is safer.
