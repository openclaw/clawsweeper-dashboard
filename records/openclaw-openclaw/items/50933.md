## Summary

Keep open. Current main still has the automatic daily/idle session-memory gap, but this branch should not merge as-is because it awaits the new automatic hook dispatch on the reply-critical session initialization path while related open PRs are exploring different event semantics.

## What This Changes

The PR emits `session:idle_reset` and `session:daily_reset` internal hook events for stale automatic rollovers, updates the bundled session-memory hook to handle them, and adds related docs and tests.

## Best Possible Solution

Land one canonical, focused fix that preserves session memory for automatic daily and idle rollovers with stable pre-reset transcript access, keeps expensive hook work bounded or off the inbound reply path, updates hook metadata/docs/changelog, and closes or supersedes the competing implementations once chosen.

## Reproduction Assessment

Yes. Configure and enable `session-memory`, let a stored session become stale by daily or idle policy, then send the next message; current main rotates and archives without any automatic internal reset hook while the hook metadata and handler remain command-only.

## Solution Assessment

No, not as-is. The direction is correct, but the automatic hook dispatch should not be awaited inline during session initialization; a bounded asynchronous path or stable transcript handoff would be safer, with maintainer choice on the event contract.
