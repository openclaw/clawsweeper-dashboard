## Summary

Keep open. Current main still has the automatic daily/idle rollover session-memory gap, and this PR remains a plausible implementation path, but the proposed patch still awaits reset-hook work in the inbound reply path and overlaps with newer open implementations that use different reset semantics.

## What This Changes

The PR emits automatic `session:daily_reset` and `session:idle_reset` hook events during stale session rollover, updates the bundled `session-memory` hook to handle them, and adds related docs and regression tests.

## Best Possible Solution

Land one focused canonical fix that gives automatic rollover the same memory-preservation guarantee as manual resets, passes stable previous-session transcript data, keeps expensive hook work off the reply-critical path, and updates bundled hook metadata, docs, and daily/idle regression coverage.

## Reproduction Assessment

Yes. Configure an enabled session-memory hook, let a session become stale by daily or idle policy, then send the next non-command message; current main rotates and archives the old transcript while session-memory remains gated to command reset/new events.

## Solution Assessment

No, not as-is. The direction is valid, but the patch should dispatch automatic reset memory work asynchronously or bound it with stable transcript access rather than awaiting expensive hook handlers inline during session initialization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Run automatic reset hooks off the reply path:** `src/auto-reply/reply/session.ts:685`

Dispatch this work asynchronously or otherwise bound it while preserving stable transcript access.
