## Summary

Keep open. Current main still prunes expired sandbox runtimes and registry entries without deleting the corresponding non-shared sandbox workspace directory, and the target-repo fix PRs linked from the discussion are closed unmerged.

## What This Changes

Sandbox prune should remove expired OpenClaw-managed workspace directories as well as the runtime container and registry entry.

## Best Possible Solution

Implement guarded cleanup in the main sandbox lifecycle so only the expired runtime’s OpenClaw-managed, non-shared, unused workspace is removed, with regression coverage for cleanup and preservation cases.

## Reproduction Assessment

Yes, at source level. An expired non-shared sandbox registry entry reaches `pruneSandboxContainers`, which removes the runtime and registry entry while the workspace directory created by `ensureSandboxWorkspaceLayout` has no deletion path.

## Solution Assessment

No for current main; it does not yet solve the leak. The best fix is a narrow backend-aware cleanup with guards for shared, reused, or still-active workspaces, not a new force-refresh mode or config option.
