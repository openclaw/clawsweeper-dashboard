## Summary

Keep open: current `origin/main` and the latest release still prune sandbox runtimes and registry rows without deleting the corresponding host workspace directory, and the live closing PR is open but conflicting and unmerged.

## What This Changes

Sandbox prune should remove expired sandbox workspace directories as well as runtimes and registry entries so stale workspaces do not leak disk space or preserve stale state.

## Best Possible Solution

Land a backend- and scope-aware cleanup that deletes only the owned expired workspace after confirmed runtime removal while preserving registry retry behavior on failures.

## Reproduction Assessment

Yes, at source level. Current `origin/main` creates non-shared sandbox workspaces under the configured root, then prune removes the runtime and registry row without deleting that host workspace directory.

## Solution Assessment

No, the bare `fs.rm()` suggestion is only a plausible fix. The best fix needs explicit ownership, scope, backend, and retry-safety checks so cleanup does not remove shared or still-active workspace data.
