## Summary

Keep this open: current main and the latest stable release still prune sandbox runtimes and registry entries without deleting the host sandbox workspace directory, and the only open closing PR is conflicting and not a safe ready-to-land owner path.

## What This Changes

Sandbox pruning should remove an expired sandbox's workspace directory as well as its runtime and registry entry.

## Best Possible Solution

Implement a current-main, backend-aware cleanup contract that deletes only the workspace owned by an expired regular sandbox after confirmed runtime removal, preserves shared/live/browser scopes, and adds focused prune and recreate coverage.

## Reproduction Assessment

Yes, at source level. Current main creates sandbox workspaces under the configured sandbox root and prunes expired runtimes and registry records without deleting those workspace directories; no live Docker or SSH prune was run in this read-only review.

## Solution Assessment

No, the bare fs.rm suggestion is only a plausible fix shape. The maintainable solution needs backend-aware ownership checks, shared/agent/browser guards, and regression coverage before deleting host workspaces.
