## Summary

Keep open: current main and the latest shipped release still prune expired sandbox runtimes and registry entries without deleting Docker sandbox workspace directories. The linked fork PR is an implementation candidate, and prior direct-deletion attempts exposed workspace data-loss hazards that need owner review before landing.

## What This Changes

Sandbox prune should remove expired Docker sandbox workspace directories along with runtimes and registry entries so stale workspaces do not leak disk or session state.

## Best Possible Solution

Add guarded backend-aware prune cleanup that removes only expired, OpenClaw-managed, unused workspace state after confirmed runtime removal, with regression coverage for delete and preserve cases.

## Reproduction Assessment

Yes, at source level. An expired Docker sandbox registry entry reaches pruneSandboxContainers, which removes the runtime and registry entry while the workspace created by ensureSandboxWorkspaceLayout has no deletion path.

## Solution Assessment

No for the naive direct recursive deletion shape tried in earlier PRs. The right direction is cleanup during prune, but it should be guarded by backend, scope, active-use, and stored workspace identity checks.
