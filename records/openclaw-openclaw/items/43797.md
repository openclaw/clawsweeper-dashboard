## Summary

This should stay open: current main and the latest release still remove the sandbox runtime and registry entry without removing the ordinary Docker workspace directory. There is an open linked fork PR, but it is conflicting and unmerged, so it is only a candidate fix path.

## What This Changes

The issue reports that sandbox pruning removes Docker containers but leaves expired sandbox workspace directories under the configured sandbox root.

## Best Possible Solution

Review or replace the linked fix with a narrow prune-owned change that removes only the expired non-shared sandbox workspace after backend runtime removal succeeds, with regression coverage for stale workspace cleanup and shared-scope preservation.

## Reproduction Assessment

Yes at source level: current prune code removes the runtime and registry entry, and the Docker backend only runs `docker rm -f`; neither path removes the ordinary sandbox workspace directory. I did not run a live Docker prune flow in this read-only review.

## Solution Assessment

Yes, a narrow prune-owned cleanup after successful runtime removal is the best fix shape. A new config option is not needed for the central bug because `openclaw sandbox recreate` already covers explicit manual refresh after configuration changes.
