## Summary

Keep open: current main and v2026.5.18 still remove Docker sandbox runtimes and registry entries without deleting the associated host workspace directory. The issue also has an open closing PR, so the best next step is review or replacement of that implementation rather than closing the report early.

## What This Changes

The issue asks for expired sandbox pruning to remove associated workspace directories as well as Docker runtimes and registry entries.

## Best Possible Solution

Add guarded backend-aware prune cleanup that removes only expired OpenClaw-managed unused workspace state after confirmed runtime removal, with regression coverage for delete and preserve cases.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches the Docker prune path, which removes the runtime and registry entry while the workspace created from `cfg.workspaceRoot` has no deletion path.

## Solution Assessment

No for a direct derived-path `fs.rm` patch. The maintainable fix should be backend-aware, tied to the stored workspace identity/scope, and guarded against active or shared workspace use.
