## Summary

Current main still prunes expired sandbox runtimes and registry entries without removing their non-shared workspace directories. The issue should remain open because a live closing PR owns the implementation, but that branch is conflicting and its post-removal deletion approach needs a race-safe lifecycle fix before landing.

## What This Changes

Remove expired non-shared sandbox workspace directories when their sandbox runtimes are pruned.

## Best Possible Solution

Integrate workspace cleanup into an ownership-safe scope teardown boundary, preserve shared workspaces, and add focused tests for session scope, multiple sessions sharing agent scope, failed runtime removal, and concurrent same-scope recreation.

## Reproduction Assessment

Yes at source level: on current main an expired ordinary sandbox reaches backend and registry deletion without any corresponding local workspace removal. A live filesystem reproduction was not run because this review is read-only.

## Solution Assessment

No, a plain post-removal `fs.rm` is not the best solution because it leaves a same-scope recreation race and must account for multiple registry entries sharing an agent workspace. Cleanup should be tied to verified scope ownership or an atomic lifecycle operation.
