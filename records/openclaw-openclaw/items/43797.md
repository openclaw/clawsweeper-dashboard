## Label Justifications

- `impact:session-state`: The report is about stale sandbox workspace state persisting across prunes and preventing updated workspace files from taking effect.

## Summary

Keep this open: current main and the latest release still remove sandbox runtimes and registry records without cleaning the Docker host workspace directory, and the only open referenced PR is a conflicting fork PR with an outdated cleanup shape.

## What This Changes

Make sandbox pruning clean up expired sandbox workspace directories in addition to runtime and registry entries.

## Best Possible Solution

Implement a current-main, backend-aware cleanup contract that deletes only the workspace owned by an expired regular sandbox after confirmed runtime removal, preserves shared/live scopes, aligns recreate behavior when appropriate, and adds focused prune/recreate regression coverage.

## Reproduction Assessment

Yes at source level. Current main creates sandbox workspaces under the sandbox workspace root and prunes expired runtimes/registry entries without deleting those Docker workspace directories; no live Docker or SSH run was performed in this read-only review.

## Solution Assessment

No, a bare recursive delete in the shared prune helper is not the best fix. The maintainable solution needs backend-aware ownership, regular-sandbox-only cleanup, shared/agent/browser/live-scope guards, and focused tests.
