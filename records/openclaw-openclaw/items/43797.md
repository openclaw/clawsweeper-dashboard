## Summary

Keep open: current main and the latest shipped tag still prune expired sandbox runtimes and registry entries without deleting the Docker sandbox workspace directory. The open fork PR is an implementation candidate, not a reason to close, and prior direct-delete attempts exposed data-loss hazards that need owner review.

## What This Changes

The issue requests that sandbox prune delete expired Docker sandbox workspace directories along with the runtime and registry entry.

## Best Possible Solution

Implement guarded backend-aware cleanup that removes only expired, OpenClaw-managed, non-shared and unused workspace state after confirmed runtime removal, with regression coverage for delete and preserve cases.

## Reproduction Assessment

Yes, at source level. An expired Docker sandbox registry entry reaches `pruneSandboxContainers`, which removes the runtime and registry entry while the workspace created by `ensureSandboxWorkspaceLayout` has no deletion path.

## Solution Assessment

No for the direct recursive deletion shape already tried. The maintainable fix is a guarded backend-aware cleanup path with active-use, browser, agent-scope, shared-workspace, and remote-backend protections.
