## Summary

This should stay open: current main and the latest release still prune Docker/runtime resources and registry rows without deleting the non-shared sandbox workspace directory. The linked fork PR is an unmerged, conflicting fix candidate, so it should be reviewed, adopted, replaced, or closed before opening another repair lane.

## What This Changes

The issue asks OpenClaw to remove expired sandbox workspace directories during sandbox prune instead of only removing Docker containers and registry entries.

## Best Possible Solution

Land or replace the existing linked fix candidate with a bounded sandbox lifecycle change that removes non-shared expired workspace directories after runtime removal and adds regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure non-shared sandbox prune, let an entry expire, and `maybePruneSandboxes` removes the runtime and registry entry without removing the workspace directory. I did not run a live Docker prune in this read-only review.

## Solution Assessment

Yes, the best fix appears to be a narrow lifecycle cleanup in the sandbox prune/remove path, preserving shared-scope safety and adding focused tests. A new config option for always-fresh workspaces would be broader product surface and is not needed for the reported prune leak.
