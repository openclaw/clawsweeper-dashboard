## Summary

Keep open: current main and v2026.5.12 still remove expired sandbox runtimes and registry entries without deleting Docker sandbox workspace directories. The issue is also paired with an open fork PR that uses closing syntax, so the issue should remain open until that PR or a safer replacement lands.

## What This Changes

The issue asks for sandbox prune to remove expired Docker workspace directories as well as runtimes and registry entries.

## Best Possible Solution

Add guarded backend-aware prune cleanup that removes only expired OpenClaw-managed unused workspace state after confirmed runtime removal, with regression coverage for delete and preserve cases.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `pruneSandboxContainers`, which removes the runtime and registry entry while the workspace created from `cfg.workspaceRoot` has no deletion path.

## Solution Assessment

No, a direct derived-path `fs.rm` is not the safest solution shape. The durable fix should be backend-aware, tied to stored workspace identity, and guarded against active/shared workspace use.
