## Summary

Keep open: current main and the latest release still prune expired Docker sandbox runtimes and registry entries without deleting the managed workspace directory, while the open linked fork PR is conflicting and not a clean target-repo landing path.

## What This Changes

The issue asks for sandbox prune to remove Docker-backed workspace directories along with expired containers so disk usage and stale workspace state do not accumulate.

## Best Possible Solution

Implement or replace the linked work with backend-aware cleanup that removes only confirmed unused OpenClaw-managed Docker per-scope workspaces after runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, removes the backend runtime and registry entry, while the Docker workspace created from `cfg.workspaceRoot` has no production deletion path.

## Solution Assessment

No for a direct `fs.rm` in prune alone. The safer fix should be backend-aware, tied to stored managed workspace identity, and guarded against shared, active, `rw`, or backend-owned workspace deletion.
