Keep open: current main and v2026.6.6 still prune sandbox runtimes and registry rows without deleting the matching host workspace directory, and the live closing PR remains open, unmerged, and dirty.

## What This Changes

The issue requests that sandbox prune remove expired sandbox workspace directories as well as the corresponding runtime containers and registry entries.

## Best Possible Solution

Review, repair, land, or replace the linked fix with a backend- and scope-aware cleanup that deletes only the owned expired workspace after confirmed runtime removal.

## Reproduction Assessment

Yes, at source level. Current main creates sandbox workspaces under the configured root, while prune removes only the runtime and registry row for expired ordinary sandbox entries.

## Solution Assessment

No, the bare `fs.rm()` suggestion is only a plausible fix. The best fix needs explicit ownership, scope, backend, and retry-safety checks so cleanup does not remove shared or still-active workspace data.
