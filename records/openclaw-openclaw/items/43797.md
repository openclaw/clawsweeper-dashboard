## Summary

Keep open. Current main still removes expired sandbox runtimes and registry entries without deleting the non-shared sandbox workspace directory, and open PR #43835 is already paired to this bug with closing syntax, so the issue should stay open until a fix merges.

## What This Changes

The issue asks for sandbox prune to delete expired sandbox workspace directories under `~/.openclaw/sandboxes/` along with removing the runtime containers and registry entries.

## Best Possible Solution

Centralize workspace cleanup in the sandbox removal path so prune and recreate delete only OpenClaw-owned, non-shared workspace directories after runtime removal, preserve shared scope, tolerate missing directories, and add focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with an expired non-shared sandbox registry entry, `maybePruneSandboxes` reaches `pruneSandboxContainers`, which removes the runtime and registry entry while the workspace directory created by `ensureSandboxWorkspaceLayout` is left untouched.

## Solution Assessment

No for current main; it does not solve the leak. The maintainable direction is a guarded workspace cleanup step, ideally by reviewing and landing PR #43835 or replacing it with an equivalent branch that also covers the recreate path.
