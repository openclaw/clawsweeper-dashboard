## Summary

Keep open: current main and the latest release still remove expired sandbox runtimes and registry entries without deleting the managed Docker workspace directory, and the linked cross-repo implementation PR remains open and conflicted rather than merged.

## What This Changes

The issue asks for sandbox prune to delete expired managed workspace directories in addition to removing runtimes and registry entries.

## Best Possible Solution

Review or replace the linked implementation with backend-aware cleanup that deletes only confirmed unused OpenClaw-managed per-scope workspaces after runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, removes the backend runtime and registry entry, while the workspace created from `cfg.workspaceRoot` has no production deletion path for Docker workspaces.

## Solution Assessment

No for the direct suggested `fs.rm` shape alone. The safer maintainable fix should be backend-aware, tied to stored managed workspace identity, and guarded against shared, active, or backend-owned workspace deletion.
