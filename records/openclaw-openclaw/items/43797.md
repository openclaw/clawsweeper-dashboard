## Summary

Keep open: current main and v2026.5.20 still prune sandbox runtimes and registry entries without deleting the managed Docker workspace directory, while the linked cross-repo fix PR remains open and conflicting.

## What This Changes

The issue requests that sandbox pruning delete expired per-scope workspace directories along with Docker containers and registry entries.

## Best Possible Solution

Review or replace the linked implementation with backend-aware cleanup that deletes only confirmed unused OpenClaw-managed per-scope Docker workspaces after runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, removes the backend runtime and registry entry, while the Docker workspace created from `cfg.workspaceRoot` has no production deletion path.

## Solution Assessment

No for the direct suggested `fs.rm` shape alone. The safer maintainable fix should be backend-aware, tied to stored managed workspace identity, and guarded against shared, active, rw, or backend-owned workspace deletion.
