- `impact:session-state`: Persisted sandbox workspace state can survive prune and drift from current agent configuration or session state.

## Summary

Keep open: current `main` and `v2026.5.18` still prune sandbox runtimes and registry entries without deleting the managed workspace directory, and live GitHub shows an open linked implementation PR that should be reviewed or replaced before this issue closes.

## What This Changes

Sandbox prune should remove the associated managed workspace directory when an expired sandbox runtime is pruned.

## Best Possible Solution

Add guarded backend-aware cleanup that removes only unused OpenClaw-managed per-scope workspaces after confirmed runtime removal, preserving shared/active/backend-owned state with delete and preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes` and removes the runtime plus registry entry, while the workspace created from `cfg.workspaceRoot` has no production deletion path.

## Solution Assessment

No for the direct derived-path `fs.rm` shape from earlier related PRs. The maintainable fix should be backend-aware, tied to stored workspace identity, and guarded against active or shared workspace use.
