## Summary

Keep open: current main and v2026.5.20 still remove sandbox runtimes and registry entries without deleting the Docker workspace directory, while an open conflicting closing PR and prior unsafe cleanup attempts make this a maintainer-reviewed fix rather than cleanup-close material.

## What This Changes

The issue asks for sandbox pruning to remove OpenClaw-managed workspace directories under the sandbox workspace root after expired sandbox runtimes are removed.

## Best Possible Solution

Implement backend-aware cleanup that deletes only confirmed unused OpenClaw-managed per-scope Docker workspaces after successful runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, which removes the runtime and registry entry while the Docker workspace created from `cfg.workspaceRoot` has no production deletion path.

## Solution Assessment

No for the direct suggested `fs.rm` shape alone. The safer solution needs a backend-aware cleanup boundary with active/shared/rw preservation and focused regression tests.
