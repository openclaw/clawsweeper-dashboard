## Summary

This should stay open: current main and v2026.6.8 still remove sandbox runtimes and registry rows without deleting the non-shared sandbox workspace directory. The linked fork PR is open but conflicting and unmerged, so it is a fix candidate for maintainer handling rather than proof that the issue is resolved.

## What This Changes

The issue requests that sandbox pruning remove expired non-shared sandbox workspace directories as well as container/runtime registry entries.

## Best Possible Solution

Add a narrow core sandbox cleanup path that deletes only non-shared per-scope sandbox workspaces after successful runtime removal, with regression coverage for prune and manual recreate behavior.

## Reproduction Assessment

Yes from source inspection: a stale registry entry reaches `removeRuntime` and registry deletion, but no code removes the corresponding non-shared workspace directory. I did not run a live Docker prune in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is core-owned cleanup of the per-scope sandbox workspace after successful runtime removal, plus tests; a new config option is not needed for the reported bug.
