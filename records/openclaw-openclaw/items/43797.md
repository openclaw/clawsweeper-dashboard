## Summary

Keep open: current `origin/main` and `v2026.6.6` still prune sandbox runtimes and registry rows without deleting the corresponding Docker sandbox workspace directory, and the linked cleanup PR is open but dirty and unmerged.

## What This Changes

Sandbox prune should remove expired sandbox workspace directories as well as Docker containers and registry entries.

## Best Possible Solution

Land a backend- and scope-aware cleanup that removes only the owned expired workspace after confirmed runtime removal while preserving registry retry behavior on failures.

## Reproduction Assessment

Yes, source-level. Current `origin/main` creates non-shared sandbox workspaces under `cfg.workspaceRoot`, then prune removes the runtime and registry row without deleting that host workspace directory; I did not run a live Docker prune because this review is read-only.

## Solution Assessment

No, a bare `fs.rm()` after prune is only a plausible fix. The best fix needs explicit ownership, scope, backend, and retry-safety checks so cleanup does not remove shared or still-active workspace data.
