## Summary

Keep open: current main and v2026.6.6 still remove expired sandbox runtimes and registry rows without deleting the matching Docker host workspace directory, and the open closing PR is still unmerged and conflicting.

## What This Changes

The issue reports that sandbox prune leaves expired sandbox workspace directories under the configured sandbox workspace root after removing Docker runtimes.

## Best Possible Solution

Review, repair, land, or replace the linked fix with backend- and scope-aware cleanup that deletes only owned expired workspaces and has focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main creates non-shared sandbox workspaces under the configured root while prune removes only the runtime and registry entry.

## Solution Assessment

No, the bare `fs.rm()` suggestion is only a plausible fix. The maintainable fix needs explicit ownership, scope, backend, and retry-safety checks before deleting a workspace directory.
