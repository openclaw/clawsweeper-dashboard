## Summary

Current main still removes expired sandbox runtimes and registry entries without deleting the corresponding Docker workspace directory, so the reported disk leak remains valid. The linked target-repo fix PRs are closed unmerged, and the only open closing reference is a conflicting fork PR rather than an OpenClaw target-repo PR.

## What This Changes

The issue requests that sandbox pruning remove expired workspace directories under `~/.openclaw/sandboxes/` in addition to removing Docker sandbox containers and registry entries.

## Best Possible Solution

Implement guarded workspace cleanup in the sandbox lifecycle so only an expired, OpenClaw-managed, non-shared and unused workspace is removed after confirmed runtime removal, with regression coverage for deletion and preservation cases.

## Reproduction Assessment

Yes, at source level. An expired sandbox registry entry reaches `pruneSandboxContainers`, which removes the runtime and registry entry while the workspace directory created by `ensureSandboxWorkspaceLayout` has no deletion path.

## Solution Assessment

No for current main. The right solution is a narrow backend-aware cleanup with active-use and shared-workspace guards, not a new config option or the unsafe recursive deletion shape from the closed PRs.
