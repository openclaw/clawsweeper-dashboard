## Summary

Keep open: current main and v2026.5.20 still remove expired sandbox runtimes and registry entries without deleting the managed Docker workspace directory, while the only open closing PR is an external conflicting branch.

## What This Changes

Clean up OpenClaw-managed sandbox workspace directories when sandbox prune removes expired runtimes.

## Best Possible Solution

Implement or replace the linked work with backend-aware cleanup that removes only confirmed unused OpenClaw-managed per-scope Docker workspaces after runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, removes the backend runtime and registry entry, while the Docker workspace created from `cfg.workspaceRoot` has no production deletion path.

## Solution Assessment

No for a direct `fs.rm` in prune alone. The safer fix should be backend-aware, tied to stored managed workspace identity, and guarded against shared, active, `rw`, or backend-owned workspace deletion.
