## Summary

Keep this open: current main and the latest release still prune sandbox runtimes and registry rows without deleting the corresponding host workspace directory, and the open closing PR is conflicting rather than a landed fix.

## What This Changes

The issue asks for sandbox prune to remove expired sandbox workspace directories as well as Docker containers and registry entries.

## Best Possible Solution

Implement an owned-workspace cleanup contract that deletes only expired, non-shared sandbox workspaces after confirmed runtime removal, while preserving registry retry behavior on failure.

## Reproduction Assessment

Yes, at source level. Current main creates non-shared sandbox workspaces under the configured root, then prune removes the runtime and registry row without deleting that host workspace directory.

## Solution Assessment

No. The bare `fs.rm()` suggestion is only a plausible implementation; the best fix needs explicit owned-workspace and scope checks so recursive deletion cannot remove shared or still-active data.
