## Label Justifications

- `P2`: This is a normal-priority sandbox cleanup bug with disk and stale-workspace impact limited to users who enable sandbox pruning.
- `impact:session-state`: Persisted sandbox workspace state can survive prune and drift from the current agent configuration or session state.

## Summary

Keep open: current main and the latest release still prune sandbox runtimes and registry entries without deleting the managed Docker workspace directory, and live GitHub shows an open linked implementation PR that should be reviewed or replaced before this issue closes.

## What This Changes

Make sandbox pruning remove the expired sandbox's managed workspace directory as well as the runtime and registry entry so stale sandboxes do not leak disk or state.

## Best Possible Solution

Review or replace the linked PR with guarded backend-aware cleanup that deletes only confirmed unused OpenClaw-managed per-scope workspaces after runtime removal, preserves shared/rw/active/backend-owned state, and adds delete plus preserve regression coverage.

## Reproduction Assessment

Yes, at source level. An expired registry entry reaches `maybePruneSandboxes`, removes the backend runtime and registry entry, while the workspace created from `cfg.workspaceRoot` has no production deletion path for Docker workspaces.

## Solution Assessment

No for a direct derived-path `fs.rm` alone. The safer maintainable fix should be backend-aware, tied to stored managed workspace identity, and guarded against shared, active, or backend-owned workspace deletion.
