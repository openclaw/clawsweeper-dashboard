## Summary

Keep this open: current main and the latest release still remove stale sandbox runtimes and registry rows without deleting the corresponding non-shared host workspace directory, and the only open closing PR is conflicting and not a safe canonical fix path.

## What This Changes

Sandbox prune should delete expired non-shared sandbox workspace directories as well as runtime containers and registry entries.

## Best Possible Solution

Implement a backend-aware workspace cleanup contract that deletes only expired, owned, non-shared regular sandbox workspaces after confirmed runtime removal, with coverage for shared/agent/browser safety.

## Reproduction Assessment

Yes, at source level. Current main creates non-shared sandbox workspaces under the configured root and prunes stale runtime containers plus registry rows without deleting those workspace directories.

## Solution Assessment

No, the bare `fs.rm()` suggestion is only a plausible implementation. The best fix needs explicit workspace ownership/scope checks and regression coverage before recursive deletion is safe.
