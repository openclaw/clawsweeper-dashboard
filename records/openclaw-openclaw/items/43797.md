## Summary

Keep open. Current main still removes expired sandbox runtimes and registry entries without deleting the corresponding non-shared workspace directory, and open PR #43835 is already linked as an implementation candidate, so this issue should remain open until a fix merges.

## What This Changes

The issue asks for sandbox pruning to delete expired workspace directories under `~/.openclaw/sandboxes/` in addition to removing sandbox runtimes and registry entries.

## Best Possible Solution

Centralize guarded workspace cleanup in the sandbox removal path so prune and recreate delete only OpenClaw-owned, non-shared sandbox workspace directories after runtime removal, with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: an expired non-shared sandbox registry entry reaches `maybePruneSandboxes`, which removes the runtime and registry entry while the workspace directory created by `ensureSandboxWorkspaceLayout` remains untouched.

## Solution Assessment

No for current main; it does not yet solve the leak. The best path is to review and land PR #43835 or replace it with an equivalent guarded cleanup that also covers the recreate path.
