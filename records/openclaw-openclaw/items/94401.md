## Summary

This PR should stay open: it fixes the central duplicate `delivery-mirror` case, but the current dedupe state can also suppress a legitimate later delivery-mirror reply that repeats the previous assistant text after a user turn.

## What This Changes

The PR changes session-memory transcript extraction to skip `delivery-mirror` assistant rows when their sanitized text matches the previous assistant text, and adds regression tests for duplicate mirrors plus standalone gateway-injected and delivery-mirror rows.

## Best Possible Solution

Keep the consumer-layer semantic filter, but make the mirror dedupe scoped to the immediately previous visible assistant output and add a regression for a repeated delivery-mirror reply after a user turn before landing this as the canonical fix.

## Reproduction Assessment

Yes. Current main source reads text from every assistant message row, so a raw thinking assistant row followed by a matching `delivery-mirror` row yields duplicate `assistant:` lines in the session-memory excerpt.

## Solution Assessment

No, not yet. The consumer-layer fix is the right boundary, but the implementation should compare only against the immediately previous visible assistant output rather than a stale assistant value that survives user turns.

- **[P2] Reset mirror dedupe after user turns:** `src/hooks/bundled/session-memory/transcript.ts:89-90`
  - body: `lastAssistantText` is only updated for assistant rows and is never cleared when a user row is emitted. Please reset the dedupe state on visible user output, or track the previous emitted role so only an immediately preceding assistant line can suppress a mirror duplicate.
