## Summary

Keep open: the requested Discord metadata feature is still absent from current main, but this PR is not merge-ready because it wires the field into a stale message-processing location and lacks real Discord proof.

## What This Changes

The PR adds an optional Discord sender client-status context field, populates it from the presence cache, and emits `sender_client_status` in user-role conversation metadata.

## Best Possible Solution

Rebase the feature into `buildDiscordMessageProcessContext`, emit the metadata only when at least one platform status exists, add focused tests, and attach redacted live Discord proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for live after-fix behavior: source inspection proves current main lacks the field and has the presence cache available, but validating the feature needs a real Discord bot with Presence Intent and desktop/mobile/web messages.

## Solution Assessment

Is this the best way to solve the issue? No, not as authored; the feature direction is narrow, but the code should move into the current context builder and avoid emitting empty status objects.
