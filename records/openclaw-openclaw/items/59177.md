## Summary

Keep open. Current main still lacks the requested Discord sender client-status metadata, and the source PR is useful, but the patch targets an older inline Discord payload path instead of the current `buildDiscordMessageProcessContext` builder.

## What This Changes

The PR wires cached Discord sender presence into `MsgContext` and renders `sender_client_status` in the user-role conversation metadata block.

## Best Possible Solution

Land a rebased version that resolves sender presence in the current Discord context builder, emits `sender_client_status` only when at least one platform key is present, keeps it in user-role metadata, and adds focused tests plus changelog.

## Reproduction Assessment

Partly. Static inspection reproduces the current-main gap because the requested field is absent and the active Discord context builder omits presence lookup; live reproduction needs a Discord bot with Presence Intent and real desktop/mobile/web client messages.

## Solution Assessment

No, not as currently authored. The feature direction is narrow and maintainable, but the implementation must move from the stale `message-handler.process.ts` payload location into `message-handler.context.ts` and avoid empty status objects.
