The PR adds an optional `SenderClientStatus` context field, reads Discord cached presence in the message handler, and emits `sender_client_status` in user-role conversation metadata.

## Best Possible Solution

Implement the feature in `buildDiscordMessageProcessContext`, serialize only non-empty client-status metadata in the user-role block, add focused tests, and attach redacted live Discord proof.

## Reproduction Assessment

No for live after-fix behavior: source inspection shows current main lacks `SenderClientStatus` while the presence cache and listener exist. Validating the feature requires a real Discord bot with `GuildPresences` enabled and real desktop/mobile/web messages.

## Solution Assessment

No, not as authored: the feature direction is narrow, but the patch should move from the stale inline payload site into `buildDiscordMessageProcessContext` and omit empty status objects.
