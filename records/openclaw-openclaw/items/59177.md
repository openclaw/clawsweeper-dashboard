## What This Changes

The PR adds optional Discord sender client-status metadata to `MsgContext`, serializes it in the user-role conversation-info block, and reads it from the Discord presence cache during message processing.

## Best Possible Solution

Use or extend the Discord `member-info` tool for client-status access unless maintainers explicitly accept automatic prompt metadata; if metadata is accepted, rebase it into `buildDiscordMessageProcessContext`, omit empty statuses, add focused tests, and attach redacted live proof.

After-fix validation still needs a real Discord bot with `GuildPresences` enabled.

## Solution Assessment

No, not as authored: the patch targets the stale context construction site, can serialize empty status metadata, and should first resolve whether client status belongs in automatic prompt metadata or the existing Discord tool path.
