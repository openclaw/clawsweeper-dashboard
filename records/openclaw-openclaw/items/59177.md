## Summary

Keep open: the requested Discord sender client-status metadata is still absent from current main and the feature remains useful, but this PR is stale against the current Discord context path and lacks real after-fix behavior proof.

## What This Changes

The PR adds `SenderClientStatus` to `MsgContext`, reads Discord `client_status` from the presence cache, and emits `sender_client_status` in user-role conversation metadata.

## Best Possible Solution

Rebase the feature into `buildDiscordMessageProcessContext`, emit `sender_client_status` only when at least one platform key exists, keep it in user-role metadata, and add focused tests plus redacted live Discord proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for live behavior: source inspection shows current main lacks the field and has the presence cache available, but live proof requires a Discord bot with Presence Intent and real desktop/mobile/web messages.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently authored; the feature direction is narrow, but the implementation must move to the current context builder and avoid empty `sender_client_status` objects.
