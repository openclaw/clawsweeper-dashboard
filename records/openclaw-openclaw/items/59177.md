## Summary

Keep open. Current main still lacks `SenderClientStatus`/`sender_client_status`, and the Discord presence cache plus `GuildPresences` listener make the feature feasible, but this PR is stale against the current Discord inbound pipeline because `ctxPayload` construction moved out of `message-handler.process.ts` into `message-handler.context.ts`.

## What This Changes

The PR wires cached Discord sender presence into `MsgContext` and renders `sender_client_status` in the user-role conversation metadata block.

## Best Possible Solution

Land a rebased version that resolves sender presence in the current Discord context builder, types the value from `discord-api-types`, emits `sender_client_status` only when at least one platform key is present, keeps it in the user-role metadata block, and adds focused tests plus the required changelog entry.

## Reproduction Assessment

Partly. The missing current-main behavior is reproducible by static inspection (`rg` finds no field, and the context builder omits it); live end-to-end validation would require a configured Discord bot with the Presence Intent enabled and messages from desktop/mobile/web clients.

## Solution Assessment

No, not as currently authored. The product direction is narrow and maintainable, but the PR needs to be moved onto the current `message-handler.context.ts` path and should avoid empty status objects before it is the best implementation.
