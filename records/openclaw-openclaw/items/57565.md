## Summary

Keep open: current main still violates the documented Mattermost `replyToMode: "off"` thread-delivery behavior, and this PR has the right core split but cannot merge as-is because it widens shared `ChatType`/SDK contracts and lacks inspectable real behavior proof.

## What This Changes

The PR separates Mattermost session routing from delivery routing with a new `messageThreadId`, updates reply/typing/tool-context paths, adjusts session thread classification, and carries the older missing-channel-type approach into shared chat types.

## Best Possible Solution

Land a fresh current-main Mattermost fix that keeps `effectiveReplyToId` for session forking, carries a separate actual thread root for delivery/tool context, coordinates with the Mattermost-local missing-channel-type work, and includes focused tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: docs promise off-mode replies stay in an existing Mattermost thread, while current main suppresses the only thread id and then uses that suppressed value for reply delivery.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but this branch should be rebased or replaced to avoid the shared `unknown` ChatType approach and to add inspectable real behavior proof.
