## Summary

Keep open: the Mattermost `replyToMode: "off"` thread-delivery bug is still source-reproducible on current main, and this PR has the right central split between session routing and delivery routing. It should not merge as-is because the branch is conflicting, lacks inspectable real behavior proof, and mixes the fix with a broad shared `ChatType` change that newer Mattermost-local work avoids.

## What This Changes

This branch separates Mattermost delivery thread roots from replyToMode session routing, updates inbound context/typing/delivery to use a new message thread id, adds Mattermost tool context, adjusts session reset classification, and includes missing-channel-type changes across Mattermost, Slack, core ChatType, and the plugin SDK.

## Best Possible Solution

Land a fresh or rebased Mattermost-local fix that carries the actual inbound thread root separately from session forking, keeps missing channel metadata local to Mattermost, coordinates with the related open PRs, and includes focused tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: docs promise off-mode replies stay in existing Mattermost threads, while current main suppresses that root and then uses the suppressed value for context, typing, and delivery; I did not run a live Mattermost server.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but missing-channel handling should stay Mattermost-local and this branch needs rebase plus inspectable proof.
