## Summary

Keep open: the Mattermost bug is valid on current main by source inspection, but this PR should not merge as-is because it broadens shared `ChatType` for a Mattermost-only missing-channel state and still lacks inspectable real behavior proof.

## What This Changes

The PR separates Mattermost session routing from delivery routing for `replyToMode: "off"` threads, propagates a new message thread id, adjusts tool context/session reset behavior, and broadens missing channel-type handling.

## Best Possible Solution

Land a fresh or rebased Mattermost-local fix that carries the actual inbound thread root separately from session forking, coordinates with https://github.com/openclaw/openclaw/pull/78530 and related thread-context work, and includes focused tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection shows current main documents off-mode replies staying in existing Mattermost threads, but the monitor suppresses the thread root and then uses that suppressed value for `MessageThreadId`, typing, and delivery.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but the shared `ChatType` change is too broad and should be replaced by a Mattermost-local missing-channel handling path.
