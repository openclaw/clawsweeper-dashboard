## Summary

Keep open: current main still loses the Mattermost thread root for existing threaded posts when `replyToMode` is `off`, but this PR should not merge as-is because it broadens shared chat typing, is conflicting/abandoned, and lacks inspectable real behavior proof.

## What This Changes

The PR separates Mattermost session routing from delivery thread context for `replyToMode: "off"`, updates reply/typing/tool-context paths, adjusts session reset classification, and adds shared `unknown` chat-type handling.

## Best Possible Solution

Land a fresh Mattermost-local fix that carries the actual inbound thread root separately from session forking, coordinates with https://github.com/openclaw/openclaw/pull/78530 for missing channel types, and includes focused tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: docs promise off-mode replies stay in an existing Mattermost thread, while current main suppresses that root and then uses the suppressed value for reply delivery; I did not run a live Mattermost server.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but this branch should be rebased or replaced to keep missing channel type state local to Mattermost and to provide inspectable real behavior proof.
