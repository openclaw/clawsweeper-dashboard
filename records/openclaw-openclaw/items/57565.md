## What This Changes

The PR separates Mattermost delivery thread metadata from session-routing `effectiveReplyToId`, uses that value for replies/typing/tool context, and adds a shared `unknown` chat-type path for missing Mattermost channel types.

## Best Possible Solution

Ship a narrow current-main Mattermost fix that carries the actual thread root separately for delivery, typing, and `MessageThreadId` while preserving off-mode session keys and current missing-channel-type fail-closed handling; coordinate any tool/cron inheritance leftovers with https://github.com/openclaw/openclaw/pull/52120 and https://github.com/openclaw/openclaw/pull/52236.

Yes, at source level: current main suppresses the Mattermost thread root when `replyToMode` is off and then reuses that suppressed value for `MessageThreadId`, typing, and final delivery. I did not run a live Mattermost server proof in this read-only review.

## Solution Assessment

No. Separating delivery thread root from session routing is the right direction, but this branch is not the best implementation because it conflicts with current main and duplicates missing-channel-type handling already merged in https://github.com/openclaw/openclaw/pull/84091.
