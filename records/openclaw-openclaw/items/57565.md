## Summary

Keep open: current main and v2026.5.22 still appear to have the Mattermost `replyToMode: "off"` thread-delivery bug, but this PR is not merge-ready because it is conflicting, lacks inspectable real-behavior proof, and carries stale shared chat-type/API changes that current main solved differently.

## What This Changes

The PR separates Mattermost delivery thread metadata from session `effectiveReplyToId`, routes replies, typing, and tool context through `messageThreadId`, adjusts session reset classification, and carries an older missing-channel-type `unknown` ChatType cascade.

## Best Possible Solution

Rebase a narrow Mattermost-only fix onto current main: keep shared `ChatType` closed, preserve `batched`, reuse current missing-channel guards, add `messageThreadId` for delivery/tool context, cover reset classification, and require redacted real Mattermost proof before merge.

## Reproduction Assessment

Yes from source inspection, not from a live Mattermost run: with `replyToMode: "off"` and an inbound `post.root_id`, current main computes `effectiveReplyToId` as `undefined` and then uses that value as the outbound thread root. The documented repro is configuring Mattermost with `replyToMode: "off"`, posting inside a thread, and observing the agent reply at the channel root.

## Solution Assessment

No for the PR as submitted: the `messageThreadId` split is the right fix direction, but the branch should be narrowed to current main instead of carrying the stale `unknown` ChatType cascade and old reply-mode assumptions.
