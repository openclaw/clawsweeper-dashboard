## Summary

Keep open: current main still appears to have the documented Mattermost `replyToMode: "off"` delivery bug, but this branch is not safe to merge as-is because it widens shared chat-type state for a Mattermost lookup failure and lacks inspectable real behavior proof.

## What This Changes

This PR separates Mattermost thread delivery state from replyToMode session routing, routes replies, typing, and tool context through a new message thread id, adjusts session reset classification, and widens chat-type handling for missing Mattermost channel types.

## Best Possible Solution

Land a fresh or rebased Mattermost-local fix that carries the actual inbound thread root separately from session forking, coordinates with https://github.com/openclaw/openclaw/pull/78530 and https://github.com/openclaw/openclaw/pull/52120, and includes focused tests plus redacted real Mattermost proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: docs promise off-mode replies stay in existing Mattermost threads, while current main suppresses that root and then uses the suppressed value for delivery. I did not run a live Mattermost server.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but missing channel-type handling should stay Mattermost-local and the branch needs rebase plus inspectable proof.
