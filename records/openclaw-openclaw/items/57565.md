## What This Changes

The PR changes Mattermost routing so `replyToMode: "off"` keeps channel/group sessions un-forked while still delivering replies, typing indicators, and tool context to the originating thread.

## Best Possible Solution

Land a fresh current-main Mattermost fix that keeps `effectiveReplyToId` for session forking, carries a separate actual thread root for delivery/tool context, and coordinates with the local missing-channel-type fix instead of widening shared `ChatType`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: docs promise off-mode channel/group replies stay in an existing thread, while current main suppresses the only thread id and then uses that suppressed value for reply delivery.

## Solution Assessment

No, not as a direct merge. Separating session routing from delivery routing is the right fix shape, but this branch needs rework to remove the superseded shared `unknown` ChatType approach and add inspectable real behavior proof.
