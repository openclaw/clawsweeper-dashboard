## Summary

Keep this PR open. Current main still lacks the PR's central separation between Mattermost session routing and delivery/thread-context routing for `replyToMode: "off"`, while the source branch is now dirty against main and still depends on open PR #57970.

## What This Changes

The branch adds a separate Mattermost `messageThreadId` for delivery, typing, inbound context, and tool context while keeping `effectiveReplyToId` for session routing, plus related unknown-channel-type, reset-classification, and test updates.

## Best Possible Solution

Land a fresh, current-main implementation that keeps `effectiveReplyToId` authoritative for session forking, adds a separate delivery/tool-context thread id for Mattermost channel/group thread replies when `replyToMode` is off, preserves direct-message non-threading and `batched` mode, and either lands or folds the unknown-chat-type dependency with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a Mattermost channel/group post with `post.root_id` and `replyToMode: "off"` yields `effectiveReplyToId: undefined`, and the outbound delivery paths still use that undefined value as the reply root; current tests also lack the proposed `messageThreadId` assertion.

## Solution Assessment

Yes conceptually, but not as a direct merge. Separating session routing from delivery routing is the narrow maintainable fix, while the current branch needs a rebase or replacement that preserves newer main behavior and resolves the open #57970 dependency.
