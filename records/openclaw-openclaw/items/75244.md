## Summary

Keep open. Current main still enforces the native plugin-tool contract that returned tools must have an object-valued `parameters` field, and tests explicitly assert that `inputSchema`-only tools are skipped with the same diagnostic. The external `openclaw-remnic` TypeBox failure remains plausible but is not proven by code in this checkout, so this needs maintainer API-contract and diagnostics direction rather than cleanup closure.

## What This Changes

The issue reports that third-party memory plugin tools using TypeBox schemas are rejected as malformed every turn and asks OpenClaw to improve diagnostics or schema-key/schema-shape handling.

## Best Possible Solution

Capture or safely log the failing third-party tool shape, then make a deliberate plugin-boundary decision: keep `parameters` as the only native contract with clearer diagnostics, or normalize a documented `inputSchema` alias at registration/tool-resolution time with SDK docs and focused contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external `openclaw-remnic` steps, but that plugin and its runtime tool object are not present in this checkout; current main only gives high-confidence reproduction for `inputSchema`-only native tools being skipped.

## Solution Assessment

Unclear. Diagnostic improvement is a safe first step, but accepting `inputSchema` or adding TypeBox-symbol handling is not clearly the best fix until maintainers inspect the failing runtime shape and choose whether to expand the native plugin contract.
