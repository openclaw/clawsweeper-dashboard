## Summary

Keep open. Current main still enforces the native plugin-tool contract that a tool must expose an object-valued `parameters` field, and it still drops malformed tools with a diagnostic instead of accepting `inputSchema`; the external TypeBox failure is concrete but not proven from a returned tool object in this checkout, so this needs maintainer plugin-API direction rather than cleanup closure.

## What This Changes

The issue reports that native plugin tools using TypeBox schemas under `parameters` are rejected as missing a parameters object, causing external memory tools to be dropped every agent turn.

## Best Possible Solution

Keep the native plugin-tool schema boundary explicit: capture the failing third-party tool object, improve diagnostics enough to reveal its runtime shape, then either preserve `parameters` as the only native contract or add a documented normalized alias with SDK docs and contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external reproduction steps and exact diagnostics, but the failing external plugin object is outside this checkout; current main only gives a high-confidence path for reproducing `inputSchema`-only native tools being skipped.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow safe first step, but accepting `inputSchema` or adding TypeBox-specific handling needs maintainer API-contract judgment once the failing object shape is known.
