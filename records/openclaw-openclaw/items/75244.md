## Summary

Keep open. Current main still enforces the documented native plugin-tool contract that `parameters` must be a non-array object, and current tests intentionally treat `inputSchema`-only native tools as malformed. The external failure may be real, but the TypeBox-specific root-cause theory is not proved by the dependency source, which returns ordinary object schemas that should pass the current guard.

## What This Changes

The issue requests that plugin tools using TypeBox schemas for `parameters`, especially memory tools from `openclaw-remnic`, stop being rejected as malformed and silently dropped from agent tool lists.

## Best Possible Solution

Keep this open until maintainers can either capture the failing plugin's actual returned tool object or choose a deliberate plugin-contract change. The end state should be a documented and tested contract: either plugin-side correction to return a JSON-schema-like `parameters` object, or an OpenClaw change that adds targeted diagnostics and any accepted alias behavior at the plugin boundary.

## Reproduction Assessment

Unclear. The issue gives concrete external reproduction steps, but the specific third-party plugin value is not present in this repository; current main only gives a high-confidence reproduction for `inputSchema`-only tools being skipped, not for a valid TypeBox `Type.Object()` assigned to `parameters` being rejected.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow maintainable first step, but accepting `inputSchema` or TypeBox-special cases as native plugin inputs needs maintainer API-contract judgment before an automated fix PR is safe.
