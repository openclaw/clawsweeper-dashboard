## Summary

Keep open. Current main still matches the documented native plugin-tool contract: native tools need a non-array object in `parameters`, `inputSchema`-only tools are intentionally treated as malformed, and malformed tool factories are skipped with diagnostics. The external `openclaw-remnic` failure may be real, but the failing returned tool object is not in this checkout, and the TypeBox source path inspected still appears to produce ordinary object schemas that should pass the current guard.

## What This Changes

The issue requests that native plugin tools using TypeBox object schemas in `parameters` stop being rejected and silently dropped, with possible fixes around diagnostics, `inputSchema` aliasing, or schema validation.

## Best Possible Solution

Capture the actual `openclaw-remnic` returned tool object at runtime and make the plugin-tool contract explicit from that evidence. If `parameters` is a valid JSON-schema-like object, fix the OpenClaw boundary and add regression coverage; if not, keep the native `parameters` contract and improve diagnostics/docs so plugin authors can correct their registrations.

## Reproduction Assessment

Unclear. The issue gives concrete external steps, but the failing plugin/version is outside this repository; current main only gives a high-confidence reproduction for `inputSchema`-only tools being skipped, not for a valid TypeBox `Type.Object()` assigned to `parameters` being rejected.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow maintainable first step, but accepting `inputSchema` or TypeBox-specific exceptions as native plugin inputs is not clearly the best fix without the failing runtime value and a maintainer API-contract decision.
