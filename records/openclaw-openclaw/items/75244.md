## Summary

Keep open. Current main still enforces the native plugin-tool contract as `parameters` plus an execute function, and current tests intentionally skip `inputSchema`-only plugin tools with the same malformed-tool diagnostic. The report is recent and has concrete external logs, but the failing `openclaw-remnic` tool object is not present in this repo, while the bundled memory plugin uses module-level TypeBox object schemas under `parameters`, so the remaining decision is a plugin API/diagnostics boundary call rather than a safe cleanup close.

## What This Changes

The issue asks OpenClaw to stop dropping `openclaw-remnic` memory tools whose TypeBox schemas are reported as malformed, improve diagnostics, and possibly accept or normalize `inputSchema` alongside `parameters`.

## Best Possible Solution

Keep one explicit native plugin schema contract: either preserve JSON-schema-like `parameters` and improve diagnostics enough to expose the bad third-party object shape, or intentionally add a normalized `inputSchema` alias with SDK docs, changelog, and plugin-boundary tests after maintainer approval.

## Reproduction Assessment

Unclear. The issue gives concrete external reproduction steps for `openclaw-remnic`, but that plugin and its failing returned tool object are not in this checkout; current main only gives high-confidence reproduction for `inputSchema`-only native tools being skipped.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow safe first step, but accepting `inputSchema` or adding TypeBox-specific handling is not clearly the best fix until maintainers inspect the captured failing object and decide whether to expand the native plugin contract.
