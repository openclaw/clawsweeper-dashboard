## Summary

Keep open. Current main still drops native plugin tools whose returned object lacks an object-valued `parameters` field, and existing tests intentionally cover the `inputSchema`-only rejection path. The reported external TypeBox failure is plausible and well described, but the exact `openclaw-remnic` runtime tool object is not in this checkout, so this needs maintainer plugin-contract and diagnostics follow-up rather than cleanup closure.

## What This Changes

The issue reports that third-party plugin memory tools using TypeBox schemas are rejected by native plugin-tool validation as missing `parameters`, causing the tools to be dropped on every agent turn without user-visible feedback.

## Best Possible Solution

Capture or safely summarize the failing third-party tool shape, then make a deliberate native plugin boundary decision: either keep `parameters` as the only supported contract with clearer diagnostics, or normalize a documented `inputSchema` alias at registration/tool-resolution time with SDK docs and focused contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external `openclaw-remnic` steps, but that plugin and its exact runtime tool object are not present here; current main only gives high-confidence evidence for `inputSchema`-only native tools being rejected.

## Solution Assessment

Unclear. A diagnostic improvement is a safe first step, but accepting `inputSchema` or TypeBox-symbol handling is not clearly the best fix until maintainers inspect the failing runtime shape and choose whether to expand the native plugin contract.
