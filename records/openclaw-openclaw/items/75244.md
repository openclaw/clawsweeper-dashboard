## Summary

Keep open. Current main still enforces the native plugin-tool contract around an object-valued `parameters` field, still drops malformed plugin tools at request-time resolution, and still has tests that treat `inputSchema`-only native tools as malformed. The reported external TypeBox failure is concrete and worth maintainer follow-up, but the failing `openclaw-remnic` tool object is not present in this checkout, so there is not enough proof to close as implemented, duplicate, or outside the repo.

## What This Changes

The issue requests fixing native plugin tool registration so TypeBox/object schemas supplied through plugin tools are not rejected and silently dropped, with better diagnostics and possible `inputSchema` normalization.

## Best Possible Solution

Keep the issue open until maintainers capture the failing third-party tool object, improve diagnostics enough to expose its runtime shape, and then either preserve the `parameters`-only contract with clearer guidance or add a documented compatibility normalization path with contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external steps using `openclaw-remnic` v9.3.82, but that plugin is outside this checkout; current main only gives a high-confidence reproduction for `inputSchema`-only native tools being skipped, not for TypeBox `parameters` objects being rejected.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow safe first step, but accepting `inputSchema` or adding TypeBox-specific handling is not clearly the best fix until the failing object shape is inspected and maintainers decide whether to expand the native plugin contract.
