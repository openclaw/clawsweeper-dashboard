## Summary

Keep open. Current main still follows the documented native plugin-tool contract that schemas live under `parameters`, skips malformed tools with diagnostics, and intentionally treats `inputSchema`-only native plugin tools as malformed. The external TypeBox failure may be real, but the actual returned `openclaw-remnic` tool object is not in this repo, and the TypeBox source path checked still returns an object shape that should satisfy the current guard.

## What This Changes

The issue reports that third-party plugin memory tools using module-level TypeBox `Type.Object()` schemas under `parameters` are rejected as malformed and dropped on every agent turn.

## Best Possible Solution

Maintain a single documented and tested plugin-tool schema boundary: either keep native plugins on JSON-schema-like `parameters` and improve diagnostics enough to identify bad third-party objects, or deliberately add a normalized `inputSchema` alias with SDK docs and contract tests after maintainer approval.

## Reproduction Assessment

Unclear. The issue gives concrete external reproduction steps for `openclaw-remnic`, but that plugin object is not available in this checkout; current main only gives a high-confidence reproduction for `inputSchema`-only native tools being skipped, not for a valid TypeBox object assigned to `parameters` being rejected.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow first step, but accepting `inputSchema` or adding TypeBox-specific validation is not clearly the best fix until maintainers decide whether to expand the native plugin contract and inspect the actual failing object.
