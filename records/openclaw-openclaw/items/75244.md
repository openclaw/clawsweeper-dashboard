## Summary

Keep open. Current main still enforces the native plugin-tool contract that `parameters` must be a non-array object and intentionally skips `inputSchema`-only tools, while the reported TypeBox failure depends on an external plugin value that is not captured in this repository. The narrow diagnostic improvement is plausible, but accepting `inputSchema` or TypeBox-specific fallbacks needs maintainer API-contract direction.

## What This Changes

The issue requests that plugin tool registration accept or better diagnose TypeBox/JSON-schema parameters so external memory tools are not silently dropped on each agent turn.

## Best Possible Solution

Keep this open until maintainers can inspect the actual failing tool object or choose an explicit native plugin contract change. The end state should be a documented, tested boundary: either plugin-side correction to return a JSON-schema-like `parameters` object, or an OpenClaw change that adds targeted diagnostics and any accepted alias behavior deliberately.

Unclear. ## Solution Assessment

Unclear. Better malformed-tool diagnostics would be a narrow maintainable first step, but accepting `inputSchema` or TypeBox-special cases as native plugin inputs needs maintainer API-contract judgment before a fix PR is safe.
