## Summary

Keep open. Current main still enforces the native plugin-tool contract that `parameters` must be a non-array object, and the existing tests intentionally reject `inputSchema`-only native tools. The external failure may be real, but the exact `openclaw-remnic` returned tool object is not present here, and the upstream TypeBox source paths checked return ordinary schema objects that should pass the current guard.

## What This Changes

The issue reports that third-party plugin tools using TypeBox object schemas under `parameters` are rejected as malformed and dropped each agent turn, and asks for better diagnostics or a deliberate schema-contract expansion such as accepting `inputSchema`.

## Best Possible Solution

Keep this open until maintainers either capture the failing plugin's actual returned tool object or choose a documented plugin-contract change. The end state should be a tested boundary: plugin tools return a JSON-schema-like `parameters` object, and OpenClaw either improves diagnostics only or deliberately normalizes any accepted alias such as `inputSchema`.

## Reproduction Assessment

Unclear. The issue provides concrete external reproduction steps, but this checkout can only verify the current `inputSchema`-only rejection path and the documented Type.Object-under-parameters contract, not the reported rejection of a valid TypeBox object assigned to `parameters`.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow maintainable first step, but accepting `inputSchema` or adding TypeBox-specific exceptions needs maintainer API-contract direction before it is clearly the best fix.
