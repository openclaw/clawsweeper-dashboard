## Summary

Keep open. Current main still has the reported rejection path for tools without a record-valued `parameters` field, but the TypeBox-specific root cause is not proved from repository evidence because the inspected TypeBox object factories return ordinary objects that should pass the current guard.

## What This Changes

The issue requests better handling and diagnostics for plugin tool schemas that are reported as `missing parameters object`, especially TypeBox-backed memory tools and possible `inputSchema` aliases.

## Best Possible Solution

Capture or add a fixture for the actual failing returned tool object, then either fix the plugin-side schema shape or make a deliberate OpenClaw contract change with diagnostics, normalization, docs, and focused tests for whatever schema aliases are supported.

## Reproduction Assessment

Unclear. The issue gives concrete external steps with `openclaw-remnic`, but this checkout can only confirm the `inputSchema`-only rejection path; it does not reproduce a TypeBox `Type.Object()` value failing the current `isRecord` guard.

## Solution Assessment

Unclear. Richer diagnostics would be a narrow maintainable first step, but accepting `inputSchema` or adding TypeBox-specific bypasses should wait for maintainer agreement because that changes the public plugin-tool contract.
