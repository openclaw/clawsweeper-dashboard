## Summary

Keep open. Current main still matches the reported failure mode for malformed native plugin tools: `parameters` must be a non-array object, `inputSchema`-only tools are intentionally skipped, and malformed tools are logged into plugin diagnostics. The issue’s external `openclaw-remnic` failure may be real, but the current repo does not contain the failing returned tool object, and TypeBox 0.34.49 source still appears to return an ordinary object schema that should pass the guard.

## What This Changes

The issue reports that third-party plugin tools using TypeBox schemas assigned to `parameters` are rejected as malformed and silently dropped from agent turns.

## Best Possible Solution

Capture the actual `openclaw-remnic` returned tool object at runtime and make the contract explicit from that evidence. If `parameters` is a valid JSON-schema-like object, fix the OpenClaw boundary and add regression coverage; if it is not, keep the native contract and improve diagnostics/docs so plugin authors can correct their registrations.

## Reproduction Assessment

Unclear. The issue gives concrete external steps, but the failing plugin/version is outside this repository; current main only provides a high-confidence reproduction for `inputSchema`-only tools being skipped, not for a valid TypeBox `Type.Object()` assigned to `parameters` being rejected.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow maintainable first step, but accepting `inputSchema` or TypeBox-specific exceptions as native plugin inputs is not clearly the best fix without the failing runtime value and a maintainer API-contract decision.
