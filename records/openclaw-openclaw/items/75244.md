## Summary

Keep open. Current main still enforces the native plugin-tool contract by dropping tools whose `parameters` value is not an object, while the public plugin guide recommends `Type.Object(...)` under `parameters`; the report is concrete enough for maintainer API-contract triage, but the failing `openclaw-remnic` returned tool object is not present in this checkout, so the exact runtime shape is still unproven.

## What This Changes

Fix native plugin tool schema validation so TypeBox-based memory tools from external plugins are not silently dropped and malformed-schema diagnostics expose the failing runtime shape.

## Best Possible Solution

Keep one explicit native tool-schema boundary: capture the failing third-party tool object, improve diagnostics enough to identify its runtime shape, then either preserve the `parameters`-only contract with clearer docs or add a normalized `inputSchema` alias with SDK docs and contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external reproduction steps with `openclaw-remnic`, but that plugin object is outside this checkout; current main only provides a high-confidence path for reproducing `inputSchema`-only native tools being skipped.

## Solution Assessment

Unclear. Better malformed-tool diagnostics are a narrow first step, but accepting `inputSchema` or adding TypeBox-specific handling is not clearly the best fix until maintainers inspect the captured failing object and decide whether to expand the native plugin contract.
