## Summary

Keep open. Current main still requires native plugin tools to return an object-valued `parameters` field and still drops malformed tools with diagnostics; the reported external `openclaw-remnic` TypeBox failure is plausible, but the exact failing runtime shape is not present in this checkout, so this needs maintainer API-contract and diagnostics direction.

## What This Changes

The issue asks OpenClaw to stop silently dropping plugin tools whose TypeBox `parameters` schema is rejected as missing, improve diagnostics, and consider normalizing `inputSchema` as an alias.

## Best Possible Solution

Add a safe diagnostic that captures the failing schema shape, then either keep the documented `parameters`-only contract with clearer docs or normalize a documented `inputSchema` alias in the plugin-tool resolver with focused contract tests.

## Reproduction Assessment

Unclear. The issue gives concrete external `openclaw-remnic` steps, and current tests reproduce the `inputSchema`-only rejection path, but this checkout does not contain the external plugin object needed to verify the reported TypeBox module-level schema rejection.

## Solution Assessment

Unclear. Better diagnostics are a narrow first step, but accepting `inputSchema` or TypeBox-symbol handling is not clearly the best fix until maintainers inspect the failing runtime object and decide whether to expand the native plugin contract.
