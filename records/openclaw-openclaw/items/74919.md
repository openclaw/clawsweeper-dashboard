## Summary

Keep open. Current main still waits for the embedded agent idle timeout before clearing pending tool results during cleanup, so the PR addresses a real session-lock latency path; the branch needs only narrow process cleanup before it is merge-ready.

## What This Changes

The PR adds an immediate-timeout path for embedded-runner pending-tool-result cleanup, passes that path during aborted attempts, and adds unit coverage that `waitForIdle` is bypassed for zero or negative timeouts.

## Best Possible Solution

Merge the abort cleanup fast path after adding the required Unreleased changelog entry and trimming the whitespace-only lines in the added test, while leaving the broader Windows latency investigation tracked separately unless later evidence proves this fully resolves it.

## Reproduction Assessment

Yes. On current main, a guarded session manager plus an agent whose `waitForIdle` never resolves reproduces the wait path: pending-tool-result cleanup waits for the timeout, and cleanup releases the session lock only afterward.

## Solution Assessment

Yes, with small process fixes. The proposed `timeoutMs <= 0` fast path reuses the existing clear-on-timeout behavior for abort cleanup while preserving the normal idle wait for successful completion.
