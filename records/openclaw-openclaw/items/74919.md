## Summary

Keep open. Current main still waits before abort cleanup can clear pending tool results and release the session lock, and this PR is a narrow fix for that path. It is not merge-ready as-is because the user-visible perf change still needs a changelog entry, and the provided patch contains whitespace-only lines in the new test block.

## What This Changes

The PR adds an abort-only immediate-timeout path for embedded agent pending-tool-result cleanup and unit coverage that zero or negative timeouts bypass `waitForIdle`.

## Best Possible Solution

Merge the abort cleanup fast path after adding the required changelog entry, trimming the whitespace-only test lines, and validating the targeted wait-before-flush test plus the changed gate; keep broader Windows startup latency tracking separate unless follow-up evidence shows this fully resolves it.

## Reproduction Assessment

Yes. On current main, a guarded session manager plus an agent whose `waitForIdle` never resolves reproduces the delay: `flushPendingToolResultsAfterIdle({ clearPendingOnTimeout: true })` waits for the timeout before clearing pending state, and cleanup releases the session lock only afterward.

## Solution Assessment

Yes, with small process fixes. The PR's `timeoutMs: 0` path is the narrow maintainable solution for aborted cleanup because it reuses existing clear-on-timeout semantics while preserving the normal idle wait for non-abort completion.
