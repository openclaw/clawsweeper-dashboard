## Summary

Keep open. The PR implements a plausible repeat-provider onboarding slice, but it is not merge-ready because the new test fixture fails type checking, the contributor proof is mock-only, and the new prompt bypasses the wizard localization path.

## What This Changes

The PR adds a repeat confirmation to interactive `openclaw onboard`, covers normal/custom/explicit-auth-choice paths in tests, and updates onboarding docs plus the changelog.

## Best Possible Solution

Fix the invalid test fixture and localize the new prompt, then prove the interactive onboarding loop in a real terminal session; broader role and fallback assignment can remain a follow-up if maintainers accept this first slice.

## Reproduction Assessment

Yes for the PR blocker: GitHub CI reproduces the type failure in `src/wizard/setup.test.ts`. For the linked feature itself, reproduction is not applicable because it is new onboarding UX, though source inspection confirms current main is single-pass.

## Solution Assessment

No, not yet. The repeat-loop approach is a reasonable narrow direction, but this patch needs a valid config fixture, localized prompt text, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a valid model config fixture:** `src/wizard/setup.test.ts:950`
  - body: The new custom-provider test returns `models: { default: "custom/model" }`, but `ModelsConfig` has no `default` property, so `check-test-types` fails with TS2353. Use a valid current config shape, such as `agents.defaults.model`/`agents.defaults.models` or provider model entries, so the test compiles.
  - confidence: 0.96
- **[P3] Localize the new onboarding prompt:** `src/wizard/setup.ts:608`
  - body: `setup.ts` routes existing onboarding prompts through `t("wizard.setup...")`, and the wizard ships aligned English, Simplified Chinese, and Traditional Chinese locale trees. This hard-coded prompt will stay English under non-English `OPENCLAW_LOCALE`; add a locale key in all wizard locale files and use `t(...)` here.
  - confidence: 0.82
