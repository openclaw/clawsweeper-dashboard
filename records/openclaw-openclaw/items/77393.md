- **[P1] Port Slack catch-up to the plugin path:** `src/slack/monitor/catch-up.ts:1`
  - body: Current main removed the `src/slack` shim directories and runs Slack monitor code from `extensions/slack`. Adding new production behavior under `src/slack/monitor` is the wrong owner boundary and either conflicts on rebase or reintroduces core channel-specific code that the current Slack plugin will not own.
  - confidence: 0.96
- **[P2] Keep TTL changes behind the canonical config path:** `src/auto-reply/reply/typing.ts:25`
  - body: This hard-codes much longer fallback typing durations without docs or config plumbing, while https://github.com/openclaw/openclaw/pull/72009 is the active configurable TTL path with proof. Existing users would get longer typing indicators without opt-in, so drop these defaults or rework them through the canonical TTL configuration decision.
  - confidence: 0.88
- **[P2] Page Slack catch-up API reads:** `src/slack/monitor/catch-up.ts:31-34`
  - body: The catch-up code reads only one `conversations.list` page and one 50-item history/replies page per channel. Slack documents these APIs as cursor-paginated and says fewer than the requested items may be returned before the end, so busy or large workspaces can silently miss the very mentions this feature is meant to replay. ([api.slack.com](https://api.slack.com/methods/conversations.list))
  - confidence: 0.9
- **[P2] Do not filter out old thread roots before catch-up:** `src/slack/monitor/catch-up.ts:112-113`
  - body: The second pass tries to find threads whose root is older but whose latest reply is new, but it loops over `history.messages` fetched with `oldest` set to the disconnect time. Slack's `oldest` filter returns only messages after that timestamp, so old roots with new mention replies are never examined and those thread mentions remain missed. ([api.slack.com](https://api.slack.com/methods/conversations.history))
  - confidence: 0.86
