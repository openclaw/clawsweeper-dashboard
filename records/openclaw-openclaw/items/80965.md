## Summary

Keep open: the resolver change is a focused bug fix with sufficient before/after proof, but the branch still leaves user-facing channel docs that describe populated `groups` as an allowlist without the new explicit-`open` qualification.

## What This Changes

This PR makes explicit `channels.<provider>.groupPolicy: "open"` treat `groups` entries as per-group overrides, adds regression tests, and updates groups/security docs plus the changelog.

## Best Possible Solution

Keep the resolver and regression-test direction, then finish a docs consistency sweep so every group allowlist reference distinguishes omitted or `"allowlist"` policy from explicit `"open"`.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `allowlistEnabled` is true for any non-empty `groups` map, and the PR body includes direct before/after resolver output for the same case.

## Solution Assessment

Mostly yes. Making explicit `open` authoritative is the narrowest maintainable source fix and preserves omitted-policy compatibility; the remaining issue is a docs consistency repair, not a resolver redesign.
