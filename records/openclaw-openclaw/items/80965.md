## Summary

Keep open: the resolver/test change matches the documented behavior and the revised PR body now includes direct resolver proof, but user-facing docs still contain unqualified groups-as-allowlist statements that become misleading after this broader explicit-open behavior.

## What This Changes

This PR makes explicit `channels.<provider>.groupPolicy: "open"` keep populated `groups` as per-group overrides, adds regression tests, and updates groups/security docs plus the changelog.

## Best Possible Solution

Land the narrow resolver/test fix after qualifying every user-facing groups-as-allowlist statement to apply only when `groupPolicy` is omitted or explicitly `"allowlist"`.

## Reproduction Assessment

Yes, by source inspection: current main sets `allowlistEnabled` for any non-empty `groups` dictionary, so explicit `groupPolicy: "open"` still blocks unlisted groups. I did not execute tests because this review is read-only.

## Solution Assessment

Mostly yes: making explicit `open` authoritative is the narrowest maintainable source fix and preserves omitted-policy compatibility. The remaining issue is a docs/security consistency sweep, not a resolver design problem.
