## Summary

Keep open: the resolver fix is the right narrow direction and the PR body now has sufficient before/after live output, but the PR still leaves user-facing docs that say populated `groups` always acts as an allowlist, which conflicts with the new explicit-open behavior and can mislead operators about group exposure.

## What This Changes

This PR changes the shared channel group-policy resolver so explicit `groupPolicy: "open"` keeps populated `groups` as per-group overrides, adds regression coverage, and updates group/security docs plus the changelog.

## Best Possible Solution

Land the resolver/test fix after qualifying every remaining groups-as-allowlist doc statement to apply only when `groupPolicy` is omitted or explicitly `"allowlist"`.

## Reproduction Assessment

Yes, by source inspection: current main sets `allowlistEnabled` for any non-empty `groups` dictionary, so explicit `groupPolicy: "open"` still blocks unlisted groups. I did not execute tests because this review is read-only.

## Solution Assessment

Mostly yes: making explicit `open` authoritative is the narrowest maintainable source fix and preserves omitted-policy compatibility. The safer merge path is to complete the docs/security wording sweep before landing.
