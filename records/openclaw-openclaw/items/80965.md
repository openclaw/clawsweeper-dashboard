## Summary

Keep open: the resolver/test direction matches the documented `groupPolicy: "open"` contract and the PR body now has usable before/after proof, but the docs sweep is still incomplete because several user-facing pages still say populated `groups` always act as an allowlist.

## What This Changes

This PR makes explicit `channels.<provider>.groupPolicy: "open"` keep populated `groups` entries as per-group overrides, adds resolver regression tests, and updates groups/security docs plus the changelog.

## Best Possible Solution

Land the resolver/test fix after qualifying every groups-as-allowlist statement to apply only when `groupPolicy` is omitted or explicitly `"allowlist"`.

## Reproduction Assessment

Yes. Source inspection on current main shows `allowlistEnabled` is true for any non-empty `groups` dictionary, so explicit `groupPolicy: "open"` still blocks unlisted groups; the PR body also includes before/after resolver output for that path.

## Solution Assessment

Mostly yes. Making explicit `open` authoritative is the narrowest maintainable code fix and preserves the omitted-policy compatibility path; the remaining problem is a docs/security consistency sweep.
