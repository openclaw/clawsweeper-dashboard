## Summary

Keep open: the resolver fix matches the documented `groupPolicy: "open"` contract and the PR body now has adequate before/after proof, but remaining user-facing docs still say populated `groups` always acts as a group allowlist, which would understate access after this behavior change.

## What This Changes

This PR makes explicit `channels.<provider>.groupPolicy: "open"` treat populated `groups` as per-group overrides, adds regression coverage, and updates groups/security docs plus the changelog.

## Best Possible Solution

Land the resolver/test fix after qualifying every remaining groups-as-allowlist statement to apply only when `groupPolicy` is omitted or explicitly `"allowlist"`.

## Reproduction Assessment

Yes. Source inspection on current main shows non-empty `groups` forces `allowlistEnabled`, and the PR body includes before/after resolver output for the repro config.

## Solution Assessment

Mostly yes. Making explicit `open` authoritative is the narrow source fix and preserves omitted-policy compatibility, but the documentation sweep is incomplete.
