## What This Changes

The PR makes explicit `channels.<provider>.groupPolicy: "open"` keep group allowlisting disabled when `groups` entries exist, with regression tests, docs, and a changelog entry.

## Best Possible Solution

Land the narrow resolver/test fix after adding real behavior proof and updating the security guide so all docs say `groups` is an implicit allowlist only when `groupPolicy` is omitted or explicitly allowlist.

## Reproduction Assessment

Yes by source inspection: current `origin/main` sets `allowlistEnabled` for any non-empty `groups` dictionary, so explicit `open` still blocks unlisted groups. I did not execute tests because this review is read-only.

## Solution Assessment

Mostly yes: making explicit `open` authoritative is the narrowest source fix and preserves omitted-policy compatibility. The PR should also update the security guide and add real behavior proof before merge.
