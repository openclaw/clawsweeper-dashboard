## Summary

Keep open: the resolver change is a valid narrow bug fix with sufficient copied runtime output, but the PR still leaves user-facing docs saying populated `groups` is always an allowlist, which conflicts with the new explicit-`open` behavior and can understate group admission.

## What This Changes

The PR changes shared channel group-policy resolution so explicit `groupPolicy: "open"` treats populated `groups` as per-group overrides, then adds regression tests, docs, and a changelog entry.

## Best Possible Solution

Land the resolver/test fix after qualifying every user-facing groups-as-allowlist statement to apply only when `groupPolicy` is omitted or explicitly `"allowlist"`.

## Reproduction Assessment

Yes, by source inspection: current main enables allowlist mode for any non-empty `groups` dictionary, so explicit `groupPolicy: "open"` still blocks unlisted groups. I did not execute tests because this review is read-only.

## Solution Assessment

Mostly yes: making explicit `open` authoritative is the narrowest maintainable source fix and preserves omitted-policy compatibility. The remaining blocker is a docs/security consistency sweep, not a resolver design issue.
