## Summary

Keep open: the resolver/test change matches the documented `groupPolicy: "open"` contract and the PR body now has sufficient before/after proof, but remaining user-facing docs still describe populated `groups` as an allowlist without the new explicit-open qualification.

## What This Changes

This PR changes shared group-policy resolution so explicit `groupPolicy: "open"` treats `groups` entries as per-group overrides, adds regression coverage, and updates related docs and the changelog.

## Best Possible Solution

Land the resolver and regression tests after completing a focused docs sweep that qualifies every populated-`groups` allowlist statement for omitted or explicit `"allowlist"` policy only.

## Reproduction Assessment

Yes. Source inspection shows current main sets `allowlistEnabled` for any non-empty `groups` dictionary, and the PR body includes direct before/after resolver output for the same repro config; I did not execute tests because this review is read-only.

## Solution Assessment

Mostly yes. Making explicit `open` authoritative is the narrow source fix and preserves omitted-policy compatibility, but the docs/security sweep needs to be finished before merge.
