## Summary

Keep open: the idle-CPU bug is plausible from current source and Chokidar’s contract, but this PR is not merge-ready because it drops production write-settling behavior on both watcher paths, is dirty against current main, and still lacks real macOS after-fix proof.

## What This Changes

The PR conditionally passes Chokidar `awaitWriteFinish` only when `VITEST` enables polling for the Gateway config reload and skills refresh watchers.

## Best Possible Solution

Keep the CPU fix direction, but rebase onto current main and replace Chokidar’s polling-based `awaitWriteFinish` with a compatibility-safe settling or retry path that preserves direct-save behavior, then add focused tests and redacted macOS proof.

## Reproduction Assessment

No high-confidence local macOS repro was run in this read-only review. Source inspection and upstream Chokidar docs support the reported mechanism: current main enables `awaitWriteFinish` on both watchers, and Chokidar polls file size while that option is active.

## Solution Assessment

No; this PR targets the likely polling cause, but removing `awaitWriteFinish` outright on native-event paths also removes chunked-write protection. The safer solution is a rebased non-polling settling or retry strategy that preserves current direct-save semantics.
