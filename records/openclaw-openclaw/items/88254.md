## Summary

Keep open: current main and latest release v2026.6.9 still use a single 900 ms Crestodian Gateway `/healthz` probe and can render the reported false startup banner, while the open fix candidate at https://github.com/openclaw/openclaw/pull/90142 has not merged.

## What This Changes

The issue reports that Crestodian can show a false `Gateway: not reachable` startup banner when its first short local Gateway health probe races service startup.

## Best Possible Solution

Land a refreshed equivalent of the linked PR that retries bounded local-loopback startup probe misses, keeps remote probes single-shot, and updates focused tests plus Crestodian docs wording.

## Reproduction Assessment

Yes at source level: current main performs one 900 ms `/healthz` probe and renders the reported not-reachable startup text when that probe fails. I did not live-reproduce the reporter's exact SSH/systemd first-login race.

## Solution Assessment

Yes for direction: a bounded local-loopback retry is narrower than raising every Gateway probe timeout globally. The issue should remain open until https://github.com/openclaw/openclaw/pull/90142 or an equivalent repaired implementation merges.
