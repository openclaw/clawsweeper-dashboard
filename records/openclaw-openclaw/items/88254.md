## What This Changes

The issue requests a longer or retried Crestodian Gateway startup probe so first launch does not show a false not-reachable banner.

## Best Possible Solution

Land a narrow local-loopback bounded retry in Crestodian overview with focused regression coverage and first-launch proof, or explicitly close the linked PR if maintainers choose not to change startup latency.

## Reproduction Assessment

No, not as a live systemd/SSH reproduction. Source inspection on current main shows the single 900 ms `/healthz` probe and the exact not-reachable startup banner path reported here.

## Solution Assessment

Yes, the requested bounded retry in the existing Crestodian overview probe path is the narrowest maintainable solution. It should preserve `/healthz` versus `/readyz` semantics and avoid adding config.
