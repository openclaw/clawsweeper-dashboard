Fix the Crestodian startup Gateway reachability false negative reported in https://github.com/openclaw/openclaw/issues/88254. On current main, `loadCrestodianOverview` calls `probeGatewayUrl` once and `probeGatewayUrl` defaults to a 900 ms `/healthz` probe; v2026.5.27 has the same shipped default. Implement a bounded retry/backoff or carefully adjusted default in `src/crestodian/probes.ts` and/or `src/crestodian/overview.ts` so a transient first-launch delayed `/healthz` response does not leave the startup banner as not reachable when a quick retry succeeds. Keep Gateway `/healthz` liveness and `/readyz` readiness semantics unchanged, do not add config or CLI flags, and do not change unrelated Crestodian operations. Add focused Vitest coverage for first-failure-or-timeout-then-success behavior plus existing timeout capping. Do not edit `CHANGELOG.md`.
