## Summary

Keep open. Current main still separates sender/conversation/history metadata into untrusted blocks, but the live inbound channel body is appended as ordinary user text, and the session/Dreaming ingestion path still does not preserve channel-origin provenance for later memory policy. Open PR #64720 is a relevant implementation candidate for live body wrapping, but it is still open and previously had trust-classification blockers, and it does not by itself settle memory provenance.

## What This Changes

Add structural untrusted-content wrapping for inbound channel message bodies and carry channel/source attribution through Dreaming memory ingestion and retrieval.

## Best Possible Solution

Merge a corrected live prompt-boundary implementation that uses explicit ingress trust/source classification and nonce-backed or escaped structured wrapping, then extend transcript/Dreaming ingestion so channel-originated user content retains provenance through session corpus staging, promotion records, and retrieval policy. The existing untrusted metadata blocks should stay, but the live body, thread bodies, media transcripts, and memory path need a consistent trust-boundary contract.

## Reproduction Assessment

Yes. A current-main source trace is enough: channel rawBody/BodyForAgent reaches get-reply-run, metadata is prefixed as untrusted JSON, the live body is appended as plain user text, and later session ingestion can stage transcript snippets into Dreaming memory without channel trust provenance.

## Solution Assessment

Unclear as originally proposed. The direction is right, but a simple fixed <untrusted-channel-message> tag is not sufficient; the maintainable fix should use escaped or nonce-backed delimiters, explicit ingress trust metadata, preserved command parsing, and memory provenance instead of prompt formatting alone.
