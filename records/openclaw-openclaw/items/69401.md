## Label Justifications

- `P1`: The issue covers a cross-channel prompt-injection and persistent-memory poisoning boundary that can affect real deployments and needs security review.
- `impact:security`: The reported gap is an untrusted content boundary that could let external channel messages influence agent instructions or tool behavior.
- `impact:session-state`: The issue explicitly concerns poisoned inbound channel text persisting into long-term memory and later session context.

## Summary

Keep open: current main still sanitizes inbound channel text without nonce-framing body/thread text as untrusted external content, and session-memory export/retrieval still lacks channel-origin trust provenance; https://github.com/openclaw/openclaw/pull/64720 is an open partial implementation path and https://github.com/openclaw/openclaw/issues/7707 tracks the broader memory trust side.

## What This Changes

Request structurally wrapping inbound channel message bodies as untrusted content and preserving channel-origin provenance through memory and Dreaming.

## Best Possible Solution

Land a framework-owned ingress trust boundary that wraps genuinely external channel body/thread/history text with the existing nonce-backed external-content markers and carries channel-origin trust/provenance through memory ingestion, Dreaming, and retrieval policy.

## Reproduction Assessment

Yes, by source inspection. Current main still normalizes and sanitizes channel BodyForAgent/thread text, then sends it into prompt assembly and session-memory export without nonce-backed external-content framing or durable channel trust metadata.

## Solution Assessment

Unclear as originally proposed. Structural wrapping is the right direction, but fixed XML-like tags are weaker than the existing nonce-backed helper, and the memory side needs framework-assigned provenance/trust rather than prompt text tags alone.
