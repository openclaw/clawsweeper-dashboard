## Summary

Keep open. Current main has useful untrusted metadata handling and a partial Discord-specific wrapper, but the generic channel body/thread prompt path and durable memory provenance path still do not implement the requested trust boundary; the related PR is open and partial, and the memory/provenance issues remain open design work.

## What This Changes

The issue requests structural untrusted-content wrapping and durable source provenance for inbound channel messages before they reach prompts or long-term memory.

## Best Possible Solution

Define a framework-owned ingress trust boundary that wraps channel body, thread, and history text with nonce-backed external-content framing or an equivalent structured model context, then persist channel-origin trust provenance through memory ingestion, Dreaming, and retrieval policy.

## Reproduction Assessment

Yes. Source inspection on current main shows the generic channel body is sanitized but not structurally wrapped before prompt assembly, while session-memory indexing strips inbound metadata instead of preserving channel trust provenance.

## Solution Assessment

Unclear as originally proposed. The trust-boundary direction is sound, but fixed XML-like tags are weaker than the existing nonce-backed external-content helper, and the memory side needs structured framework-assigned provenance rather than only prompt text tags.
