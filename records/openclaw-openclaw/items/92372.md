- `impact:session-state`: The affected agent can begin sessions without its expected daily memory context and waste turns on failed recovery reads.

## Summary

Keep this issue open: release `v2026.6.5` predates a current-main startup-context change that may avoid the failing reread, but there is no focused proof yet that heartbeat and incoming-message starts for an explicit secondary workspace receive that agent’s daily memory correctly. The earlier related closure is not reliable provenance because it cited an unrelated QQ Bot commit.

## What This Changes

Ensure secondary agents with explicit workspaces retain the `memory/` path when loading daily memory instead of attempting a workspace-root basename read that returns `ENOENT`.

## Best Possible Solution

Add a focused current-main regression scenario for a configured non-default agent workspace, covering both heartbeat and inbound session startup, and ensure the runtime injects the existing daily file as `memory/YYYY-MM-DD.md`; fix the workspace/prompt caller only if that scenario fails, without adding basename-to-memory guessing in the read tool.

## Reproduction Assessment

No, not yet at high confidence on current main. The issue provides a concrete `v2026.6.5` trajectory, but current main has since changed startup-memory handling and lacks an explicit secondary-agent heartbeat or inbound regression test.

## Solution Assessment

Unclear until the current-main multi-agent scenario is exercised. Runtime startup preloading from the selected workspace is the preferred design; changing general `read` semantics to infer a missing directory would be unsafe.
