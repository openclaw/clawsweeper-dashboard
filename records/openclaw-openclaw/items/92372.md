## Summary

Keep open. Current main does not show the read tool stripping `memory/`; runtime startup preloading preserves `memory/...` under the selected workspace, but there is still no focused current-main proof that secondary-agent heartbeat and inbound-message starts receive that daily memory correctly, and the linked open PR is not a viable core fix.

## What This Changes

The issue asks OpenClaw to preserve secondary-agent daily memory paths like `memory/YYYY-MM-DD.md` during multi-agent startup/heartbeat/inbound runs instead of producing bare-filename ENOENT reads.

## Best Possible Solution

Add focused current-main proof for a configured non-default agent workspace on heartbeat and inbound-message startup; if it fails, fix the startup-context/workspace routing path rather than adding basename guessing to the read tool.

## Reproduction Assessment

No high-confidence current-main reproduction yet. The issue provides concrete release trajectory evidence and source shows bare date reads still ENOENT, but current main needs a targeted secondary-agent heartbeat and inbound-route proof before deciding the runtime still drops context.

## Solution Assessment

Unclear until the current-main multi-agent route is exercised. The best fix, if still reproducible, is in startup-context/workspace routing rather than changing general read semantics.
