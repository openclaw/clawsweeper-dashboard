## Summary

Keep open. Current main preserves `memory/` in the startup daily-memory prelude and resolves configured secondary-agent workspaces, but the read tool still treats a bare date filename as a normal workspace-root read, so the reported ENOENT remains plausible if the secondary-agent run emits that bare path; there is still no focused current-main proof that heartbeat or inbound routing drops the prefix.

## What This Changes

The issue reports that a secondary agent with an explicit workspace can call `read` with `YYYY-MM-DD.md` instead of `memory/YYYY-MM-DD.md`, causing a workspace-root ENOENT.

## Best Possible Solution

Add focused current-main proof for a configured non-default agent workspace on heartbeat and inbound-message startup; if it fails, fix the prompt/workspace routing boundary rather than adding broad basename guessing to the read tool.

## Reproduction Assessment

No high-confidence current-main reproduction yet. Source shows a bare `YYYY-MM-DD.md` read would still resolve at the workspace root and ENOENT, but the reported secondary-agent heartbeat or inbound-message route has not been proven to emit that bare path on current main.

## Solution Assessment

Unclear until the current-main multi-agent route is exercised. Current startup-context generation preserves `memory/`, so the best fix is likely in prompt composition or workspace routing only if a focused repro confirms prefix loss.
