## Summary

Keep open. Current main still has no high-confidence proof that the secondary-agent heartbeat or inbound-message startup path preserves `memory/` in the model-emitted read call, and the only linked fix PRs were closed unmerged after touching an unrelated QA-lab Telegram surface.

## What This Changes

The issue reports that secondary agents with explicit workspaces can emit a bare daily-memory filename instead of `memory/YYYY-MM-DD.md`, causing the read tool to resolve the file at the workspace root and fail with ENOENT.

## Best Possible Solution

Get a focused current-main reproduction for a non-default agent with an explicit workspace on heartbeat and inbound-message startup; if it still fails, fix the startup-context or workspace-routing path rather than adding basename guessing to the generic read tool.

## Reproduction Assessment

No high-confidence current-main reproduction yet. The issue body gives concrete setup and trajectory evidence from `2026.6.5`, and source shows a bare filename would fail, but this review did not prove current main still emits that bare path on the secondary-agent heartbeat or inbound route.

## Solution Assessment

Unclear until the current-main multi-agent route is exercised. A fix belongs in the startup-context/workspace routing layer if that route drops the prefix; changing the read tool to infer `memory/` from bare date filenames would be a riskier workaround.
