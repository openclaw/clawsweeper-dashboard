## Summary

This PR should stay open for normal maintainer review: the current main branch still has the fallback synchronous install-records read in the plugin-loader cache context, while the PR narrowly memoizes that fallback and clears it with existing cache invalidation. The linked live Discord/VPS timing evidence supports the performance benefit, and I found no concrete blocking correctness or security issue in the one-file patch.

## What This Changes

Memoizes the plugin install-records fallback read in `src/plugins/loader.ts` and resets that memo when plugin loader or registry caches are invalidated.

## Best Possible Solution

Land the memoized fallback if focused loader-cache validation stays clean, keeping the newer `installRecords` threading as the preferred path and using the memo only when callers have no snapshot records.

## Reproduction Assessment

Yes. Source inspection on current main shows the fallback sync install-records read happens before cache reuse when `options.installRecords` is absent, and the PR discussion includes live before/after timing evidence; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Memoizing only the fallback read while preserving caller-supplied `installRecords` is a narrow maintainable fix; the recent main-branch snapshot threading reduces common reads but does not remove this remaining fallback hot path.
