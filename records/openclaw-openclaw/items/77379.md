## Summary

Keep open: the reported hot-path I/O problem is source-reproducible, but this patch adds a process-wide install-record memo that is not invalidated by the source-change path used after plugin install, update, uninstall, and repair flows.

## What This Changes

The PR memoizes installed plugin index records in `src/plugins/loader.ts` and clears that memo when the full plugin loader cache is cleared.

## Best Possible Solution

Keep the hot-path optimization, but centralize install-record memo invalidation for both full loader clears and source-change registry-cache clears, then add regression coverage and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: warm the proposed memo, write different persisted install records, call the registry-cache invalidation path used after plugin source changes, then resolve plugin tools/cache context again. The PR would keep using the stale memo because only `clearPluginLoaderCache` resets it.

## Solution Assessment

No, not yet. Memoization is a reasonable direction for the hot path, but the current implementation must invalidate on every installed-index source-change path, not only on full plugin loader clears.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear the install-record memo on source-change cache clears:** `src/plugins/loader.ts:248-257`
  - body: The new process-wide install-record cache is only reset by `clearPluginLoaderCache()`, but plugin install/update/uninstall refreshes call `clearPluginRegistryLoadCache()` after writing `plugins/installs.json`. If this memo has been warmed earlier in the same process, those source changes will keep using stale install records and cache keys until a full loader clear or restart. Please reset the memo from `clearPluginRegistryLoadCache()` as well, or centralize the reset with every installed-index source-change invalidation.
  - confidence: 0.9
