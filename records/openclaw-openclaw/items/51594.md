## Summary

This PR should stay open because current main still has the cron minute-field bug and the branch is a useful fix direction, but it is not merge-ready: it misses composite list entries, encodes unsupported numeric-prefix stepping, changes wildcard/every-minute timing without maintainer acceptance, and still lacks contributor after-fix real behavior proof.

## What This Changes

The PR adds a helper in src/cron/stagger.ts plus regression assertions in src/cron/stagger.test.ts so top-of-hour cron stagger detection treats step, list, range, range-step, and wildcard minute fields as including minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive list parsing, no unsupported 0/N syntax, an explicit maintainer decision on wildcard/every-minute timing, focused persisted-path regression coverage, and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the helper still rejects step, list, and range minute fields because it requires a literal "0", and the PR discussion adds a concrete production failure shape for step schedules.

## Solution Assessment

No, not as written. The direction is right, but the implementation should match Croner semantics, recurse through list entries, avoid unsupported 0/N syntax, and treat wildcard timing as an explicit maintainer compatibility choice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep wildcard schedules exact unless accepted:** `src/cron/stagger.ts:19-20`
  - body: Treating `* * * * *` as top-of-hour adds the default stable offset to every-minute jobs in create, normalize, or doctor repair paths, so existing every-minute schedules can silently stop firing on the exact minute. Unless maintainers explicitly accept that timing change with upgrade coverage, leave wildcard minute fields out of this helper or gate the behavior separately.
  - confidence: 0.9
- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: The comma branch checks only for a literal `"0"`, but Croner parses each list element as a full field expression. Valid schedules like `0-5,30 * * * *` or `*/15,45 * * * *` still fire at minute zero and would still miss default staggering after this patch.
  - confidence: 0.95
- **[P3] Drop unsupported numeric-prefix step syntax:** `src/cron/stagger.ts:23`
  - body: OpenClaw uses Croner with default parsing options, and `0/15` is not accepted as default Croner syntax. The helper and tests should not make that form part of OpenClaw's supported cron contract; use Croner-valid examples such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
