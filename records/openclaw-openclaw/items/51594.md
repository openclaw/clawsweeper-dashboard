## Summary

Keep open: the linked cron bug is still present on current main and in the latest release, but this branch is not merge-ready because it changes wildcard/every-minute timing, misses composite list syntax, encodes unsupported Croner syntax, and lacks after-fix real behavior proof.

## What This Changes

This PR adds a cron minute-field helper and tests so top-of-hour stagger detection treats step, list, range, range-step, and wildcard minute fields as including minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive list parsing, no unsupported 0/N syntax, explicit wildcard/every-minute handling, persisted-path regression coverage, and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `isRecurringTopOfHourCronExpr` still rejects step, list, and range minute fields that include zero because it only accepts a literal minute field of `0`; the PR discussion also includes production-impact schedules.

## Solution Assessment

No, not as written. The PR targets the right helper, but it should match Croner semantics, recurse through list entries, drop unsupported 0/N coverage, and either remove wildcard/every-minute default staggering or get explicit maintainer acceptance with upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep wildcard schedules exact unless accepted:** `src/cron/stagger.ts:19-20`
  - body: Returning true for `*` makes `* * * * *` receive generated default stagger during create, normalization, and doctor repair. Existing every-minute jobs that rely on exact minute-boundary timing would be silently shifted by a stable offset, so this needs to be removed or explicitly accepted with upgrade coverage before merge.
  - confidence: 0.88
- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Drop unsupported numeric-prefix step syntax:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy ranges are enabled. The helper and test should not encode unsupported OpenClaw cron syntax as a staggered case; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.87
