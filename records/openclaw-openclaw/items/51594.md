## What This Changes

This PR adds a minute-field helper and tests so default cron stagger detection treats step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive list parsing, unsupported syntax removed, focused regression coverage, and a single-line changelog entry.

## Reproduction Assessment

Yes. Current main source still literal-checks minute "0", so resolveDefaultCronStaggerMs("*/15 * * * *"), "0,30 * * * *", and "0-5 * * * *" take the no-default-stagger path despite Croner scheduling them at minute zero.

## Solution Assessment

No. The PR targets the right helper, but it should recurse through comma-separated entries, avoid unsupported 0/N syntax under OpenClaw's Croner defaults, and add the required changelog entry before merge.
