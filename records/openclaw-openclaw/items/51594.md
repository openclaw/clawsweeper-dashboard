## What This Changes

The PR adds a minute-field helper and helper tests so top-of-hour cron stagger detection treats selected step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Land one Croner-compatible minute-zero helper fix with recursive list parsing, no unsupported numeric-prefix syntax, explicit wildcard/every-minute handling, and persisted-path regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.5.18 only detect literal minute field `0`, while the linked issue and production comment provide concrete step/list/range schedules that fire at minute zero and miss default staggering.

## Solution Assessment

No. The PR targets the right helper, but it should recurse through comma-list entries, drop unsupported `0/15` handling, and settle or remove wildcard/every-minute default staggering before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step handling:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10.0.1 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. The helper and test should not encode unsupported OpenClaw cron syntax as a staggered case; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
