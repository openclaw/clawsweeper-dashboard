## Summary

## What This Changes

This PR adds a minute-field helper and tests so default top-of-hour cron staggering detects step, list, range, range-step, and wildcard minute fields that include minute zero.

## Best Possible Solution

Land one Croner-compatible minute-zero helper fix with recursive list parsing, no unsupported numeric-prefix syntax, explicit wildcard/every-minute handling, persisted-path regression coverage, and redacted real behavior proof.

## Reproduction Assessment

Yes, from source inspection. Current main still rejects step, list, and range minute fields that include minute zero because isRecurringTopOfHourCronExpr only accepts a literal minute field of 0, and the discussion includes concrete production-impact schedules.

## Solution Assessment

No, not as written. The PR targets the right helper, but it should follow Croner semantics by recursing through list entries, dropping unsupported 0/N coverage, and either removing wildcard/every-minute default staggering or getting explicit maintainer acceptance with upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep wildcard minute schedules exact unless accepted:** `src/cron/stagger.ts:19-20`
  - body: Returning true for `*` makes `* * * * *` receive generated default stagger during create, normalization, and doctor repair. Existing every-minute jobs that rely on exact minute-boundary timing would be silently shifted by a stable offset, so this needs to be removed or explicitly accepted with upgrade coverage before merge.
  - confidence: 0.87
- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step handling:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy ranges are enabled. The helper and test should not encode unsupported OpenClaw cron syntax as a staggered case; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
