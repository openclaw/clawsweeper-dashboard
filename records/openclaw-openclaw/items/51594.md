## Summary

Keep open: this is a valid cron runtime bug and current main plus the latest release still use literal minute-field equality, but this PR is not merge-ready because the helper misses composite comma-list entries, locks in Croner-invalid numeric-prefix steps, and still lacks after-fix real behavior proof.

## What This Changes

This PR adds a minute-field helper in `src/cron/stagger.ts` and helper assertions so top-of-hour stagger detection treats step, list, range, range-step, and wildcard minute fields as including minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive comma-list parsing, no unsupported numeric-prefix step coverage, focused persisted-path regression tests, and redacted after-fix proof, then close the linked bug after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveDefaultCronStaggerMs` still reaches literal minute-field equality checks, and the linked production comment gives concrete before-fix impact for step cron expressions.

## Solution Assessment

No. The PR targets the right helper, but it needs recursive comma-list parsing and Croner-compatible step semantics before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before evaluating range or step syntax inside each list item. Croner parses comma-separated entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but would still miss the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step support:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless `sloppyRanges` is enabled. The helper and new test should not lock in syntax that OpenClaw does not accept; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
