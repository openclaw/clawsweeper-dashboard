## Summary

Keep this PR open: the linked cron stagger bug is still present on current main and latest release, and the PR is useful source work, but the patch still has a blocking Croner-compatibility gap plus missing after-fix real behavior proof.

## What This Changes

The PR changes cron top-of-hour stagger detection to treat selected minute step, list, range, range-step, and wildcard expressions as including minute zero, with helper-level regression assertions.

## Best Possible Solution

Land one Croner-compatible minute-zero helper with recursive comma-list parsing, no unsupported numeric-prefix step coverage, persisted-path regression tests, and redacted after-fix proof, then close the linked bug once the fix reaches main.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveDefaultCronStaggerMs still reaches literal minute-field equality checks, while the linked issue and production comment give concrete step-expression examples that should receive the default stagger.

## Solution Assessment

No. The PR targets the right helper, but it should not merge until comma-separated entries recurse through Croner-valid range/step syntax and unsupported numeric-prefix step support is removed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step support:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10.0.1 rejects numeric-prefix stepping such as `0/15` unless `sloppyRanges` is enabled. This helper and its new test should not make unsupported OpenClaw cron syntax look supported; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
