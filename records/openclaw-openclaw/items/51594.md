## Summary

Keep open: the current bug is still present on main, and this PR is a useful fix candidate, but the proposed helper has Croner-compatibility gaps and the external contributor has not posted real behavior proof.

## What This Changes

This PR adds a cron minute-field helper and tests so top-of-hour default stagger detection recognizes step, list, range, range-step, and wildcard minute fields that include minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive comma-list parsing, unsupported 0/N coverage removed, focused regression coverage across helper/service/normalization/doctor paths, and contributor real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveDefaultCronStaggerMs still reaches literal minute-field equality checks, so expressions like */15 * * * * and 0,30 * * * * take the no-default-stagger path despite Croner parsing them as minute-zero schedules.

## Solution Assessment

No. The PR changes the right helper, but it should match Croner semantics by recursively parsing comma entries and avoiding numeric-prefix 0/N syntax under default OpenClaw options before it can merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28-29`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:24`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. This helper and its new test should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
