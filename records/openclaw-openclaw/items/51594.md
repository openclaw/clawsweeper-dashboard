## Summary

Keep open: this is a real docs-backed cron stagger bug and the PR targets the right helper, but the current patch still misclassifies composite minute lists, accepts Croner-unsupported numeric-prefix steps, and lacks external real-behavior proof.

## What This Changes

The PR adds a minute-field parser and tests so cron default stagger detection treats step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Land a Croner-compatible minute-zero helper with recursive list parsing, unsupported numeric-prefix syntax removed, focused regression coverage across default-stagger paths, and maintainer-owned changelog work during landing.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveDefaultCronStaggerMs still reaches literal minute-field equality checks, so expressions like */15 * * * * and 0,30 * * * * take the no-default-stagger path despite Croner parsing them as firing at minute zero.

## Solution Assessment

No. The PR changes the right helper, but it should follow Croner semantics by recursively parsing comma entries and avoiding 0/N syntax under default Croner options before it can merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:27`
  - body: `minuteFieldIncludesZero` returns from the comma branch before parsing range or step syntax inside each list element. Croner parses comma entries recursively, so `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this patch would still return false and skip the default stagger.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step handling:** `src/cron/stagger.ts:22`
  - body: OpenClaw constructs Croner with default options, and Croner 10.0.1 rejects numeric-prefix steps like `0/15` unless `sloppyRanges` is enabled. The helper and new test should not make `0/15` part of OpenClaw's supported cron contract; use `*/15` or `0-59/15` coverage instead.
  - confidence: 0.86
