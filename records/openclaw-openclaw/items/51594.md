## Summary

Keep open: current main and the latest stable release still have the strict minute-field equality bug, and the production report confirms real impact, but this PR still has concrete Croner-semantics blockers and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a minute-field helper and tests so top-of-hour stagger detection treats selected step, list, range, range-step, and wildcard cron minute fields as including minute zero.

## Best Possible Solution

Land one Croner-compatible minute-zero helper fix that covers supported step/list/range syntax across helper, service, normalization, and doctor migration paths, with every-minute wildcard behavior made explicit.

## Reproduction Assessment

Yes, from source inspection rather than live execution. Current main's helper only accepts literal minute field `0`, while the linked bug and production comment provide concrete step/list/range schedules that fire at minute zero and miss the default stagger.

## Solution Assessment

No. The PR targets the right helper, but it should recurse through comma-list elements, drop unsupported `0/15` coverage, and either remove or get explicit acceptance for wildcard/every-minute default staggering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28-29`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:24`
  - body: OpenClaw constructs Croner without `sloppyRanges`, and Croner 10.0.1 rejects numeric-prefix stepping such as `0/15` by default. The helper and new test should not encode unsupported OpenClaw cron syntax as a staggered case; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
