## What This Changes

This PR adds a minute-field parser and tests so cron top-of-hour stagger detection recognizes selected step, list, range, range-step, and wildcard minute fields instead of only literal `0`.

## Best Possible Solution

Land one Croner-compatible minute-zero parser scoped to the intended top-of-hour behavior, with helper plus create/normalize/doctor regression coverage and explicit removal or maintainer acceptance of wildcard scope.

## Reproduction Assessment

Yes, from source inspection. Current main and v2026.5.18 still reject step/list/range minute fields because `isRecurringTopOfHourCronExpr` only accepts literal minute `0`, and the linked discussion includes production before-fix impact.

## Solution Assessment

No, not as-is. The central fix direction is right, but the implementation needs Croner-compatible list and step semantics, explicit handling of wildcard compatibility, persisted-path coverage, and after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not default-stagger every-minute cron schedules:** `src/cron/stagger.ts:19`
  - body: Returning true for `*` makes every existing `* * * * *` cron without explicit `staggerMs` receive the default stagger through create/normalize/doctor paths. Current behavior keeps those schedules exact, and the docs only promise automatic staggering for recurring top-of-hour expressions, so this needs to be removed or explicitly accepted as an upgrade-visible timing change.
  - confidence: 0.9
- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: The comma branch only checks each list item for literal `0`, but Croner parses list entries recursively. Expressions like `0-5,30 * * * *` or `*/15,45 * * * *` still fire at minute zero and would continue missing the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step handling:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. The helper and test should not make unsupported OpenClaw cron syntax look valid; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.87
