## Summary

Keep open: the underlying Feishu multi-account webhook `EADDRINUSE` bug is still valid, but this PR is not mergeable because it conflicts with current main, lacks inspectable real behavior proof, and regresses current webhook auth, body-limit, routing, and lifecycle behavior.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, with per-account route dispatch and pooled cleanup for webhook mode.

## Best Possible Solution

Land a refreshed pooled-server implementation that preserves current raw-body auth, pre-auth limits, rate-limit normalization, deterministic route selection, lifecycle/error semantics, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts Feishu accounts through concurrent channel startup and each webhook account creates and listens its own HTTP server on the resolved host:port, so duplicate webhook ports can collide with `EADDRINUSE`.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable form because it must be rebased and updated to preserve current webhook auth, body-limit, rate-limit, routing, and lifecycle contracts.

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This validates signatures after JSON parsing with `JSON.stringify(payload)`, returns true without an encrypt key, and compares with `===`.   - confidence: 0.95
- **[P1] Propagate shared-server errors to joiners:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can stay pending or marked running with no live route, so their restart path is skipped.
  - confidence: 0.88
- **[P2] Preserve tightened webhook body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Keep current rate-limit key normalization:** `extensions/feishu/src/monitor.transport.ts:320`
  - body: The extracted handler builds the rate-limit key from `req.socket.remoteAddress` directly, bypassing current main's `resolveRequestClientIp` and loopback normalization. This can split one client across counters and regress existing rate-limit-key behavior.
  - confidence: 0.86
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first route with a matching token. Feishu accounts can inherit or share a verification token, so token-only payloads can be delivered to the wrong account handler in multi-account deployments.
  - confidence: 0.8
- **[P2] Return malformed JSON as invalid JSON:** `extensions/feishu/src/monitor.transport.ts:474-477`
  - body: Malformed JSON makes `resolveRoute` return no route and the caller maps that to 404. Signed malformed bodies should keep current account-handler behavior of returning 400 `Invalid JSON` instead of looking like unroutable webhooks.
  - confidence: 0.76
