## Summary

This PR should stay open because the reported Feishu multi-account webhook bind failure is still source-reproducible on current main, but the branch is not mergeable as-is: it lags current webhook security hardening, has lifecycle/routing regressions, and lacks observable after-fix proof.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, pooled route dispatch, and pool-aware cleanup for multi-account webhook mode.

## Best Possible Solution

Land a refreshed shared-server implementation that preserves raw-body signature checks, 64 KiB and 5 second pre-auth guards, per-account lifecycle semantics, unambiguous path/token/appId routing, focused multi-account tests, and release-note coverage.

## Reproduction Assessment

Yes, from source inspection. Current main starts enabled Feishu accounts into concurrent monitor promises, and each webhook account creates and listens its own http.Server on the resolved host:port, so two accounts sharing that endpoint have a clear EADDRINUSE path.

## Solution Assessment

No. Pooling by host:port is the right direction, but this PR head is not the best mergeable solution until it is rebased onto current main and preserves the existing webhook security, routing, and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch verifies Feishu signatures from JSON.stringify(payload), treats a missing encryptKey as valid, and compares with ===. Current main verifies the raw request body with safeEqualSecret before JSON parsing and fails closed, so merging this would regress signed non-canonical bodies and missing-key cases.
  - confidence: 0.95
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared HTTP server emits error, this handler clears state for every route but only rejects the creator account's monitor promise. Accounts that joined through joinExistingServer only resolve on abort, so they can remain marked running after their route and socket have been torn down.
  - confidence: 0.88
- **[P2] Keep current pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head keeps older 1 MiB and 30 second webhook limits, while current main uses 64 KiB and 5 seconds with regression coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve those current limits before routing or replaying requests.
  - confidence: 0.93
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: When multiple pooled accounts share a verification token, the token-only fallback returns the first matching route before considering request path. Payloads without app_id, including url_verification, can be delivered to the wrong account handler; route by path plus token or reject the ambiguity.
  - confidence: 0.8
- **[P2] Return invalid JSON as a bad request:** `extensions/feishu/src/monitor.transport.ts:501-507`
  - body: In the multi-account path, malformed JSON makes resolveRoute return undefined, which then becomes 404 Not Found. That changes invalid-body semantics from the current bad-request behavior and makes malformed payloads indistinguishable from unroutable requests.
  - confidence: 0.76
