## Summary

Keep open: this PR targets a real Feishu multi-account webhook bug, but the head is stale against current main and would regress current webhook authentication, body-limit hardening, and pooled lifecycle behavior.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, per-account route registration, pooled request dispatch, and pool-aware cleanup in monitor state and transport code.

## Best Possible Solution

Land a refreshed Feishu-owned shared webhook server implementation that preserves raw-body signature checks, 64 KiB / 5 s pre-auth guards, per-account lifecycle semantics, path/token/appId-safe routing, targeted multi-account tests, and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main starts enabled Feishu accounts concurrently and monitorWebhook creates/listens one http.Server per account on the resolved host:port, so two webhook accounts sharing that endpoint have a clear EADDRINUSE path.

## Solution Assessment

No. Pooling by host:port is the right direction, but this PR head is not the best mergeable solution until it is rebased onto current main and preserves the existing webhook security and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch verifies Feishu signatures from JSON.stringify(payload), uses a direct string comparison, and returns valid when encryptKey is absent. Current main verifies the raw request body with safeEqualSecret before JSON parsing and fails closed, so signed non-canonical bodies and missing-key cases would regress.
  - confidence: 0.95
- **[P1] Reject joined monitors when the shared server errors:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: The shared server error handler clears pooled state for every route but only rejects the creator monitor promise. Accounts that joined through joinExistingServer remain pending until abort, so they can stay marked running after their route and socket are gone.
  - confidence: 0.88
- **[P2] Keep current pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head keeps older 1 MiB / 30 s webhook limits, while current main uses 64 KiB / 5 s with regression coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve those current limits before routing or replaying requests.
  - confidence: 0.93
- **[P2] Route ambiguous token matches by path or reject them:** `extensions/feishu/src/monitor.transport.ts:519-531`
  - body: When multiple pooled accounts inherit the same verification token, the token-only fallback returns the first route before considering request path. Payloads without app_id, including url_verification, can be delivered to the wrong account handler and fail under a different encryptKey or path policy.
  - confidence: 0.78
