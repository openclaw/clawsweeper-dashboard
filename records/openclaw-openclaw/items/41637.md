## Summary

Keep open. Current main still has the one-server-per-account Feishu webhook failure mode, but this PR should not merge as-is because the head is stale relative to current webhook hardening and still has pooled lifecycle/routing bugs.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port so multiple accounts can share one listener and route requests by verification token or app_id.

## Best Possible Solution

Land a refreshed Feishu-owned pooled webhook listener that preserves current raw-body auth guards, has deterministic route cleanup/error propagation, and includes multi-account lifecycle/security tests plus a changelog entry.

## Reproduction Assessment

Yes. Configure two Feishu webhook accounts with the documented default host and port; current main creates and listens a separate http.Server per account, so the second listener can fail with EADDRINUSE.

## Solution Assessment

No. The pooled-listener direction is the right narrow fix, but this implementation should be rebuilt on current main's hardened webhook path and prove routing, cleanup, and server-error behavior with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve raw-body signature validation:** `extensions/feishu/src/monitor.transport.ts:345-369`
  - body: The new extracted handler reads and parses JSON before calling `isFeishuWebhookSignatureValid`, and the PR head verifies a parsed payload instead of the exact raw body. Current main intentionally authenticates the raw body before parsing, so this stale path can reject valid signed JSON with different whitespace/order and changes unauthenticated malformed JSON from an auth failure into a parse/routing result. Please rebase the pooling work onto the current `readWebhookBodyOrReject`/raw-body signature flow.
  - confidence: 0.91
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears `httpServers`, `botOpenIds`, and `botNames` for every route but only rejects the creator account's monitor promise. Joined account promises from `joinExistingServer()` only resolve on abort, so they can remain marked running after their shared listener has been torn down and never trigger the gateway restart path.
  - confidence: 0.88
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first matching route, but Feishu config allows multiple accounts to inherit the same top-level verification token. Payloads without an `app_id` can therefore be delivered to the wrong account handler, especially when accounts have different encrypt keys or setup paths. Require an unambiguous match or include another discriminator before choosing the first token match.
  - confidence: 0.8
