## Summary

Keep open. The duplicate-port Feishu webhook bug is real on current main, but this PR head is conflicting, lacks inspectable after-fix real behavior proof, and regresses current webhook security, body-limit, routing, and lifecycle contracts.

## What This Changes

The branch adds a Feishu webhook server pool keyed by host:port, per-account route registration, pooled cleanup, and buffered dispatch/replay for multi-account webhook mode.

## Best Possible Solution

Land a refreshed pooled-server implementation that preserves raw-body auth, current pre-auth limits, rate-limit normalization, deterministic routing, lifecycle/error semantics, focused regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts Feishu accounts concurrently and each webhook account creates and listens its own HTTP server on the resolved host:port, so duplicate webhook ports can collide with `EADDRINUSE`.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch needs a refresh against current Feishu webhook security, routing, body-limit, rate-limit, and lifecycle contracts before it is mergeable.
