## Summary

Keep open. The same-port Feishu webhook bug is still source-reproducible on current main, but this PR is not safe to merge because it predates current webhook auth hardening, widens pre-auth limits, has pooled lifecycle/routing regressions, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port with per-account routes, pooled cleanup, and body buffering/replay changes in `extensions/feishu/src/monitor.transport.ts` and `extensions/feishu/src/monitor.state.ts`.

## Best Possible Solution

Land a refreshed pooled-server implementation that keeps the host:port sharing idea while preserving current raw-body auth, pre-auth limits, rate-limit normalization, deterministic route/error semantics, focused tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main creates and listens a separate Feishu webhook `http.Server` for each account, and multi-account startup can run account monitors in one provider run. I did not run a live Feishu/Lark setup in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right fix direction, but this branch is not the best mergeable form because it regresses current webhook auth, body-limit, rate-limit, routing, and lifecycle contracts.

- **[P1] Preserve raw-body signature checks:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This helper validates signatures over `JSON.stringify(payload)`, returns true without an encrypt key, and uses direct string comparison.   - body: When the shared server emits `error`, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can remain pending or marked running with no live route, so restart handling is skipped.
  - confidence: 0.88
- **[P2] Preserve tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth resource limits.
  - confidence: 0.93
- **[P2] Keep current rate-limit key normalization:** `extensions/feishu/src/monitor.transport.ts:320`
  - body: The extracted handler builds the rate-limit key from `req.socket.remoteAddress` directly, bypassing current main's `resolveRequestClientIp` and loopback normalization. This can split one client across counters and regress existing rate-limit behavior.
  - confidence: 0.86
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first route with a matching token. In multi-account webhook mode, shared or inherited verification tokens can send a webhook to the wrong account handler when the payload lacks an app id.
