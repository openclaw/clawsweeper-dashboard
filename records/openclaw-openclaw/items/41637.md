## Summary

Keep open: the reported Feishu multi-account `host:port` collision is still source-reproducible on current main, but this PR head is stale against current webhook security hardening, has blocking ingress/lifecycle regressions, and still lacks observable real behavior proof.

## What This Changes

The PR changes Feishu webhook monitoring to pool one HTTP server per host:port and route shared-server requests to account handlers by token or appId.

## Best Possible Solution

Land a refreshed pooled-server implementation on current main that preserves raw-body signature auth, 64 KiB and 5 second pre-auth limits, per-account lifecycle/error settlement, unambiguous routing, focused multi-account tests, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows two enabled Feishu webhook accounts can both reach `monitorWebhook`, and each account currently creates and listens its own HTTP server on the resolved host:port; I did not run a live repro in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right direction, but this PR head is not the best mergeable solution until it is rebased and preserves the current webhook security, routing, and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch verifies signatures from `JSON.stringify(payload)`, treats a missing `encryptKey` as valid, and compares with `===`. Current main verifies the raw request body before JSON parsing and uses `safeEqualSecret`, so signed non-canonical bodies and missing-key cases would regress.
  - confidence: 0.95
- **[P1] Propagate pooled server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can remain marked running with no live route or socket, so restart/recovery will not run for them.
  - confidence: 0.88
- **[P2] Keep current pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head keeps older 1 MiB and 30 second webhook limits, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: When multiple pooled accounts share a verification token, the token-only fallback returns the first route before considering path. Payloads without `app_id`, including URL verification, can be delivered to the wrong account handler.
  - confidence: 0.8
- **[P2] Return malformed JSON as a bad request:** `extensions/feishu/src/monitor.transport.ts:475-477`
  - body: Malformed JSON makes `resolveRoute` return `undefined`, which becomes `404 Not Found` in the multi-account path. That changes invalid-body semantics and makes malformed payloads indistinguishable from unroutable requests.
  - confidence: 0.76
