## Summary

Keep open: current main still has the Feishu multi-account same-port bind problem, but this PR is not mergeable as-is because it conflicts with current main and regresses webhook security, lifecycle, and routing behavior added after the branch was cut.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, route registration, token/appId dispatch, body replay, and pool-aware cleanup across the Feishu monitor state and transport files.

## Best Possible Solution

Land a refreshed pooled-server implementation that coalesces same host:port Feishu webhook accounts while preserving raw-body signature auth, pre-auth limits, deterministic account routing, lifecycle/error semantics, focused regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Multiple enabled Feishu webhook accounts can reach monitorWebhook, and current main still creates and listens a separate HTTP server for each account on the resolved host:port; I did not run a live repro in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable solution until it is rebased and preserves current webhook security, routing, body-limit, and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch validates signatures from JSON.stringify(parsedPayload), returns true when encryptKey is absent, and compares with ===. Current main requires encryptKey, validates against the raw body before parsing, and uses safeEqualSecret, so signed non-canonical bodies and missing-key paths would regress.
  - confidence: 0.95
- **[P1] Propagate pooled server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits error after joiners are registered, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can stay pending or marked running with no live route or socket.
  - confidence: 0.88
- **[P2] Keep current pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first matching route before considering path or ambiguity. Accounts can inherit or share a verification token, so token-only payloads can be delivered to the wrong account handler in multi-account deployments.
  - confidence: 0.8
- **[P2] Return malformed JSON as 400:** `extensions/feishu/src/monitor.transport.ts:475-477`
  - body: Malformed JSON makes resolveRoute return undefined, and the caller maps every miss to 404. A signed malformed body should keep the existing account-handler behavior of returning 400 Invalid JSON.
  - confidence: 0.76
