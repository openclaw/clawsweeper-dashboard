Full review comments:

- **[P1] Preserve raw-body signature checks:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This helper validates `JSON.stringify(payload)`, returns true without an encrypt key, and compares with `===`. Current main requires an encrypt key, signs the raw request body before parsing, and uses `safeEqualSecret`, so this regresses Feishu webhook auth.
  - confidence: 0.96
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can remain pending or marked running with no live route, so restart handling is skipped.
  - confidence: 0.9
- **[P1] Rate-limit unroutable pooled requests before buffering:** `extensions/feishu/src/monitor.transport.ts:428-433`
  - body: The pooled dispatcher calls `applyBasicWebhookRequestGuards` without a rate limiter before it buffers the body, and route misses return 404 without ever hitting an account limiter. Invalid-token floods can consume socket and memory budget that current main throttles earlier.
  - confidence: 0.88
- **[P2] Preserve tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled path should preserve the current pre-auth resource limits.
  - confidence: 0.94
- **[P2] Keep current rate-limit key normalization:** `extensions/feishu/src/monitor.transport.ts:320`
  - body: The extracted handler builds the rate-limit key from `req.socket.remoteAddress` directly, bypassing current main's `resolveRequestClientIp` and loopback normalization. This can split one client across counters and regress existing rate-limit behavior.
  - confidence: 0.86
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first route with a matching token. In multi-account webhook mode, shared or inherited verification tokens can send payloads without an app id to the wrong account handler.
  - confidence: 0.82
- **[P2] Return invalid JSON as a bad request:** `extensions/feishu/src/monitor.transport.ts:473-477`
  - body: Malformed JSON makes `resolveRoute` return `undefined`, and the caller maps that to 404. Current account handlers return 400 for signed invalid JSON, so multi-account pooled mode hides parse failures as routing misses.
  - confidence: 0.78
