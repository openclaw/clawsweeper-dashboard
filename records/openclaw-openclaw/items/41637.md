## What This Changes

The PR adds a Feishu webhook server pool keyed by host and port, then dispatches pooled requests to per-account handlers by token/appId with pooled cleanup.

## Best Possible Solution

Refresh this into a narrow pooled-server patch on current main that preserves Feishu webhook auth, guard, rate-limit, path, and lifecycle contracts; add focused multi-account routing/error/cleanup tests; and require redacted real Feishu/Lark proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows monitorWebhook creates and listens a separate http.Server per Feishu webhook account while multi-account webhook configs can share the same default host:port.

## Solution Assessment

No, not as-is. Sharing one listener is the right fix direction, but this stale branch needs a rebase and a narrower implementation that preserves current auth, body-limit, rate-limit, path, routing, and lifecycle guarantees.

  - body: The extracted handler still calls the stale signature path with the parsed payload, while the PR head's helper accepts a missing encryptKey and signs JSON.stringify(payload).   - confidence: 0.93

  - body: The pooled path uses the branch's 1 MiB/30 second Feishu webhook limits while buffering unauthenticated multi-account requests. Current main tightened pre-auth webhook reads to 64 KiB/5 seconds, so this would widen the unauthenticated upload budget.
  - confidence: 0.9
- **[P1] Throttle unroutable pooled requests before buffering:** `extensions/feishu/src/monitor.transport.ts:428-433`
  - body: This new pooled path intentionally omits the rate limiter before body buffering and route lookup. Requests with invalid or unknown tokens can therefore consume the multi-account buffer and JSON parse path without any per-account rate-limit key ever being charged.
  - confidence: 0.88
- **[P1] Reject joined accounts on shared-server errors:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: On a shared server error, this handler deletes state for every route but only rejects the creator account's monitor promise. Joiner promises created by joinExistingServer only resolve on abort, so they can remain running with no live route and no restart signal.
  - confidence: 0.87
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-531`
  - body: The token-only fallback returns the first route with a matching token, and url_verification falls back to any route. Feishu accounts can inherit the same top-level verification token, so payloads without app_id can be delivered to the wrong account handler or fail against the intended account's path/secret.
  - confidence: 0.78
