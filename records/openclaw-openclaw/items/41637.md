## Summary

Keep this PR open: it targets a real Feishu multi-account webhook bind problem, but the current head is not mergeable because it lacks observable real behavior proof and regresses current webhook security, routing, and lifecycle behavior.

## What This Changes

The PR adds a Feishu webhook server pool so multiple accounts on the same host:port share one HTTP server and dispatch requests by verification token or appId.

## Best Possible Solution

Land a refreshed pooled-server Feishu webhook fix that preserves raw-body signature auth, current 64 KiB/5 second pre-auth limits, unambiguous routing, per-account lifecycle/error settlement, focused tests, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows two enabled Feishu webhook accounts can both reach `monitorWebhook`, and current main creates/listens a separate HTTP server per account on the resolved host:port; I did not run a live repro in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right direction, but this PR head is not the best mergeable solution until it is rebased and preserves current webhook security, routing, and lifecycle contracts.

  - confidence: 0.93
