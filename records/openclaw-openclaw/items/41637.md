## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port and dispatches pooled requests to per-account handlers by verification token or app_id, with pooled cleanup.

## Best Possible Solution

Refresh this as a narrow pooled-server implementation on current Feishu webhook contracts, preserving auth/body/rate-limit/path/lifecycle behavior and adding focused multi-account tests plus redacted real Feishu/Lark proof.

## Reproduction Assessment

Yes. Source inspection on current main shows each Feishu webhook account creates and listens its own http.Server while the documented default webhookPort is 3000, so same-host multi-account configs can collide.

## Solution Assessment

No, not as-is. Sharing one listener is the right direction, but this branch must be refreshed on current webhook auth, body-limit, rate-limit, routing, and lifecycle contracts before it is a maintainable fix.

  - body: Current main verifies the Feishu signature against the exact raw request body before parsing.   - body: In the multi-account path, the dispatcher buffers and parses the body before any account-level rate limiter runs. Invalid or unroutable webhook traffic can therefore consume socket, memory, and JSON parse work without the current pre-body throttle boundary.
  - confidence: 0.88
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears state for every joined route but rejects only the creator account's monitor promise. Joiner promises only resolve on abort, so those accounts can remain marked running with no live route or restart signal.
  - confidence: 0.84
