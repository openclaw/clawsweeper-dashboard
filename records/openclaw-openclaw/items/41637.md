## Summary

Keep this PR open. It targets a real Feishu multi-account webhook bug that still exists on current main, and the shared host:port pool is the right general direction, but the submitted head is stale against current main and still has blocking correctness and security issues in unauthenticated webhook ingress and pooled lifecycle handling.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, registers per-account routes, dispatches pooled requests by token/app_id, replays buffered bodies to the selected handler, and makes monitor cleanup pool-aware.

## Best Possible Solution

Keep the fix inside the Feishu plugin, but refresh the branch over current main with a pooled host:port listener that preserves raw-body signature validation, the 64 KiB / 5 s pre-auth body budget, per-account throttling semantics, joined-account error propagation, unambiguous route selection, and focused multi-account webhook tests.

## Reproduction Assessment

Yes. Configure two enabled Feishu accounts in webhook mode with the same webhookHost:webhookPort; current main starts accounts concurrently and each monitorWebhook call creates and listens its own http.Server, so the second bind can fail with EADDRINUSE.

## Solution Assessment

No. A Feishu-plugin-local shared server is the right boundary, but this PR head is not the best implementation until it is rebased and preserves current webhook security, lifecycle, routing, and validation contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve raw-body signature validation:** `extensions/feishu/src/monitor.transport.ts:365-369`
  - body: Current main validates the exact raw request body with safeEqualSecret before JSON parsing. This branch validates JSON.stringify of the parsed payload with direct equality, so valid signed payloads with different formatting can fail and constant-time comparison is lost.
  - confidence: 0.88
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the pooled server emits error, this handler clears state for every route but only rejects the creator monitor promise. Joined accounts can remain marked running after their shared listener and route have been removed.
  - confidence: 0.84
- **[P2] Rate-limit pooled requests before buffering:** `extensions/feishu/src/monitor.transport.ts:428-433`
  - body: In multi-account mode the dispatcher buffers and parses requests before route resolution, but the pre-buffer guard omits the existing rate limiter. Invalid-token floods can consume timeout and memory budget without being counted by the current per-account limiter.
  - confidence: 0.82
- **[P2] Reject ambiguous token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: After the token+appId pass, the token-only fallback returns the first matching route. Accounts can inherit the same verification token, so payloads without app_id can be delivered to the wrong account instead of being rejected as ambiguous.
  - confidence: 0.81
- **[P3] Return malformed JSON as an invalid body:** `extensions/feishu/src/monitor.transport.ts:475-478`
  - body: Malformed JSON makes route resolution fail and the dispatcher returns 404. Current webhook handling returns an invalid-body response for signed malformed JSON, so pooled mode changes error semantics and hides bad webhook bodies as unknown routes.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The PR changes unauthenticated Feishu webhook ingress and has concrete regressions around signature validation, throttling, routing isolation, and body-budget hardening.

Concerns:

- **[medium] Raw-body signature hardening can be lost:** `extensions/feishu/src/monitor.transport.ts:365`
  - body: The branch validates signatures using parsed JSON serialization and direct string equality, while current main validates the exact raw body with safeEqualSecret. The shared-server implementation must preserve the current raw-body contract.
  - confidence: 0.88
- **[medium] Unroutable pooled requests bypass throttling:** `extensions/feishu/src/monitor.transport.ts:428`
  - body: The pooled dispatcher buffers and parses before route resolution without using the existing rate limiter, and route misses return before per-account guards run. Invalid-token traffic can consume request resources without being counted.
  - confidence: 0.82
- **[medium] Ambiguous token-only routing can cross accounts:** `extensions/feishu/src/monitor.transport.ts:519`
  - body: The token-only fallback selects the first matching route when multiple accounts share a verification token, which can misdeliver webhook payloads across accounts when app_id is absent or ambiguous.
  - confidence: 0.8
- **[low] Stale body budget is broader than current main:** `extensions/feishu/src/monitor.state.ts:31`
  - body: The branch carries 1 MiB and 30 s webhook body limits, while current main uses 64 KiB and 5 s. A refresh should keep the tighter unauthenticated request budget unless maintainers intentionally change it.
  - confidence: 0.78
