## Summary

Keep this PR open because it targets a real Feishu multi-account webhook bind failure that is still visible on current main, but the branch is not mergeable as-is: it lacks real behavior proof and regresses current webhook auth, pre-auth limits, routing, and lifecycle/error handling.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port with per-account routes, body buffering/replay, and pooled cleanup in `monitor.transport.ts` and `monitor.state.ts`.

## Best Possible Solution

Land a refreshed pooled-server implementation that preserves current raw-body auth, pre-auth limits, rate-limit normalization, deterministic route selection, lifecycle/error semantics, regression coverage, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main starts Feishu accounts concurrently and each webhook account creates and listens its own HTTP server on the resolved host:port, so duplicate webhook ports can collide with `EADDRINUSE`. I did not run a live Feishu setup in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable form because it must be refreshed to preserve current webhook auth, body-limit, rate-limit, routing, and lifecycle contracts.
