## Summary

Keep open: current main still creates one Feishu webhook HTTP server per account, so the reported same-host:port bind failure remains valid, but this PR is not mergeable as-is because it lacks real behavior proof, is merge-conflicting, and regresses security-sensitive webhook ingress behavior.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, pooled per-account routes, token/appId dispatch, body replay, and pool-aware cleanup in the Feishu monitor transport and state files.

## Best Possible Solution

Land a refreshed pooled-server implementation that coalesces same-host:port Feishu webhook accounts while preserving raw-body signature auth, pre-auth limits, deterministic account routing, lifecycle/error semantics, focused regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts one monitor task per enabled Feishu account and each webhook account creates and listens its own server on the resolved host:port; I did not run a live repro in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable solution until it is rebased and preserves current webhook security, routing, body-limit, and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch verifies signatures from JSON.stringify(payload) after JSON parsing, returns true when encryptKey is absent, and compares with ===. Current main requires an encrypt key, verifies the raw body before parsing, and uses safeEqualSecret, so signed non-canonical bodies and missing-key paths would regress.
  - confidence: 0.95
- **[P1] Propagate pooled server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits error, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can stay pending or marked running with no live route or socket.
  - confidence: 0.88
- **[P2] Keep current pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first matching route before considering path or ambiguity. Accounts can inherit or share a verification token, so token-only payloads can be delivered to the wrong account handler in multi-account deployments.
  - confidence: 0.8
- **[P2] Return malformed JSON as invalid JSON:** `extensions/feishu/src/monitor.transport.ts:474-477`
  - body: Malformed JSON makes resolveRoute return undefined, and the caller maps every miss to 404. A signed malformed body should keep the existing account-handler behavior of returning 400 Invalid JSON instead of looking like an unroutable webhook.
  - confidence: 0.76
