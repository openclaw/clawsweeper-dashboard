## Summary

Keep this PR open: it targets a real Feishu multi-account webhook bug on current main, but the branch is stale/conflicting and currently regresses Feishu webhook security and lifecycle behavior, with no inspectable real behavior proof.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, per-account route registration, buffered token/appId dispatch with body replay, and pooled cleanup in Feishu monitor state.

## Best Possible Solution

Land a refreshed pooled-server implementation that coalesces same-host:port Feishu webhook accounts while preserving raw-body auth, current pre-auth limits, rate-limit normalization, deterministic account routing, lifecycle/error semantics, focused regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts every enabled Feishu account and each webhook account creates and listens its own HTTP server on the resolved host:port, so duplicate webhook ports can collide.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable solution until it is refreshed against current Feishu webhook security, routing, body-limit, rate-limit, and lifecycle contracts.

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch validates signatures after JSON parsing with JSON.stringify(payload), returns true when encryptKey is absent, and compares with ===. Current main fails closed, signs the raw body before parsing, and uses safeEqualSecret, so non-canonical signed bodies and missing-key paths would regress.
  - confidence: 0.95
- **[P1] Propagate shared-server errors to joiners:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits error, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts can remain pending or marked running with no live route or socket, so their restart path is skipped.
  - confidence: 0.88
- **[P2] Preserve tightened webhook body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Keep current rate-limit key normalization:** `extensions/feishu/src/monitor.transport.ts:320-327`
  - body: The extracted handler builds the rate-limit key from req.socket.remoteAddress directly, bypassing current main's normalizeFeishuWebhookRateLimitClient behavior for loopback and proxy-aware client resolution. This regresses the existing rate-limit-key tests and can split one client across multiple counters.
  - confidence: 0.86
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first route with a matching token before checking path or ambiguity. Accounts can inherit or share a verification token, so token-only payloads can be delivered to the wrong account handler in multi-account deployments.
  - confidence: 0.8
- **[P2] Return malformed JSON as invalid JSON:** `extensions/feishu/src/monitor.transport.ts:475-477`
  - body: Malformed JSON makes resolveRoute return undefined and the caller maps every miss to 404. Signed malformed bodies should keep the current account-handler behavior of returning 400 Invalid JSON instead of looking like unroutable webhooks.
  - confidence: 0.76
