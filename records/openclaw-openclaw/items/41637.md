## Summary

Keep open. The current-main Feishu webhook transport still creates one HTTP server per account, so the reported multi-account `EADDRINUSE` bug remains, but this PR branch is stale against the current webhook security path and has pooled-server lifecycle gaps that should block merge.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, routes shared-server requests by token/appId, and updates monitor cleanup for pooled servers.

## Best Possible Solution

Land a refreshed Feishu-owned shared webhook server implementation that preserves current raw-body signature checks, 64 KiB/5 s pre-auth guards, per-account lifecycle semantics, targeted multi-account tests, and a changelog entry.

## Reproduction Assessment

Yes. Configure two Feishu webhook accounts with the same `webhookHost:webhookPort`; current main still creates and listens on one `http.Server` per account, so the second account has a clear `EADDRINUSE` reproduction path.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is not the best mergeable solution until it is rebased onto current main, preserves existing webhook hardening, and fixes joined-account error propagation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the current raw-body signature guard:** `extensions/feishu/src/monitor.transport.ts:45-69`
  - body: This branch computes the Feishu signature from `JSON.stringify(params.payload)` and returns valid when `encryptKey` is absent, while current main verifies the raw request body with constant-time comparison before JSON parsing. Rebase the pool change on top of the current raw-body guard so signed payloads with non-canonical JSON do not fail and invalid bodies keep the current fail-closed behavior.
  - confidence: 0.95
- **[P1] Reject joined account tasks when the pooled server errors:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: The shared server `error` handler deletes state for every pooled route but only rejects the creator account's monitor promise. Any account that joined through `joinExistingServer` remains pending and can stay marked running even though its route and server state were removed, so server errors need to settle all joined monitor tasks or trigger their restart path.
  - confidence: 0.88
- **[P2] Preserve the tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The branch head still has the older 1 MiB / 30 s Feishu webhook body limits, but current main tightened pre-auth handling to 64 KiB / 5 s and has regression tests for oversized and slow unsigned bodies. The rebase must keep the current limits and apply them to the pooled buffering path.
  - confidence: 0.93
