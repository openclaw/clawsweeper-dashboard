## Summary

Keep open: current main still has the per-account Feishu webhook server collision this PR is trying to fix, but the branch is not mergeable as-is because it conflicts with current main, lacks inspectable real behavior proof, and regresses current Feishu webhook security and lifecycle contracts.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port with per-account routes, body buffering/replay, and pooled cleanup in the Feishu monitor transport/state files.

## Best Possible Solution

Land a refreshed pooled Feishu webhook server implementation that preserves current raw-body auth, pre-auth limits, rate-limit normalization, deterministic shared-server lifecycle/error handling, focused regression tests, and real Feishu/Lark proof.

## Reproduction Assessment

Yes. By source inspection, current main creates and listens a separate Feishu webhook http.Server for each account, and multi-account provider startup can run account monitors in one provider run; I did not run a live Feishu/Lark setup in this read-only review.

## Solution Assessment

No. Pooling by host:port is the right fix direction, but this branch is not the best mergeable form because it regresses current webhook auth, resource-limit, rate-limit, routing, and lifecycle contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve raw-body signature checks:** `extensions/feishu/src/monitor.transport.ts:365-369`
  - body: This path passes the parsed JSON payload into signature validation, whose helper signs `JSON.stringify(payload)`. Current main signs the exact raw request body before parsing and uses `safeEqualSecret`, so signed non-canonical payloads and fail-closed missing-key behavior would regress.
  - confidence: 0.95
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the shared server emits `error`, this handler clears state for every route but rejects only the creator account's monitor promise. Joined accounts can remain pending or marked running with no live route, so restart handling is skipped.
  - confidence: 0.89
- **[P2] Preserve tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should keep the current pre-auth resource limits.
  - confidence: 0.93
- **[P2] Keep current rate-limit key normalization:** `extensions/feishu/src/monitor.transport.ts:320`
  - body: The extracted handler builds the per-account rate-limit key from `req.socket.remoteAddress` directly, bypassing current main's `resolveRequestClientIp` and loopback normalization. This can split one client across counters and regress covered rate-limit behavior.
  - confidence: 0.86
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first route with a matching token. In multi-account webhook mode, shared or inherited verification tokens can send a webhook to the wrong account handler when the payload lacks an app id.
  - confidence: 0.8
