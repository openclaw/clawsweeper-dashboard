## Summary

Keep open: current main still has the same Feishu multi-account webhook bind problem, but this PR is not mergeable as-is because it predates current webhook hardening, has blocking ingress/security regressions, and lacks inspectable real behavior proof.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port with per-account route registration, token/appId dispatch, body replay, and pooled cleanup.

## Best Possible Solution

Land a refreshed pooled-server implementation that coalesces same-host:port Feishu webhook accounts while preserving raw-body auth, current pre-auth limits, deterministic account routing, lifecycle/error semantics, focused regression tests, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts one monitor per enabled Feishu account and each webhook account creates and listens its own HTTP server on the resolved host:port, so duplicate webhook ports can collide.

## Solution Assessment

No. Pooling by host:port is the right direction, but this branch is stale and not the best mergeable solution until it preserves current Feishu webhook security, routing, body-limit, and lifecycle contracts.

  - confidence: 0.88

  - body: The PR head resets Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds with coverage for oversized and slow unsigned bodies. The pooled buffering path should preserve the current pre-auth limits.
  - confidence: 0.93
- **[P2] Disambiguate token-only pooled routing:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: The token-only fallback returns the first matching route before considering path or ambiguity. Accounts can inherit or share a verification token, so token-only payloads can be delivered to the wrong account handler in multi-account deployments.
  - confidence: 0.8
