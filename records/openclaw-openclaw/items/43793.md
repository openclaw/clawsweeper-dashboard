## Summary

Keep open. The central TUI reconnect UX is still not implemented on current main, but the PR head is stale against the current backend contract, has a concrete SSH host-key verification regression, and lacks external real behavior proof.

## What This Changes

The PR adds TUI gateway disconnect guidance, Ctrl+R reconnect plumbing, gateway reconnect timer cleanup, macOS remote gateway probe UI, and a deterministic diffs cleanup test adjustment.

## Best Possible Solution

Keep the reconnect UX goal, but rebase onto current main, model manual reconnect as an optional backend capability or gateway-only path, and preserve the shared strict SSH helper in the macOS probe.

## Reproduction Assessment

Yes. Static inspection reproduces the blockers: current main's `TuiBackend` has no `reconnect` method while PR head calls `client.reconnect()`, and PR head weakens SSH host-key checking that current main keeps strict.

## Solution Assessment

No. The UX direction is reasonable, but this branch is not the best current implementation because it bypasses the backend abstraction and weakens remote SSH verification; the safer path is an optional/gateway-only reconnect capability plus strict SSH options.
