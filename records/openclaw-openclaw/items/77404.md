## Summary

Keep open: the prior QR-login proxy fix for #72547 is present and documented as shipped, but this report describes a newer linked-but-disconnected runtime state under a managed macOS Gateway service. Source inspection suggests shell `HTTP_PROXY`/`https_proxy` may not reach the LaunchAgent unless configured as managed proxy config or durable `OPENCLAW_PROXY_URL`, so this needs focused proxy/runtime logs before closing or queuing a code fix.

## What This Changes

Reports WhatsApp stays linked but disconnected and receives no messages after QR login when HTTP proxy environment variables are set on OpenClaw 2026.5.3-1.

## Best Possible Solution

Confirm the Gateway service proxy environment from logs/deep status, then either route the reporter to durable `proxy.enabled`/`proxy.proxyUrl` setup or add a focused diagnostic/docs improvement for WhatsApp linked-but-disconnected proxy cases.

## Reproduction Assessment

No. Source inspection explains a plausible current-main path where QR login uses shell proxy env but the installed Gateway service does not, but the reported linked-but-disconnected runtime failure is not reproduced without Gateway logs or deep status.

## Solution Assessment

Unclear. The existing #72547 QR-login fix appears correct for the original 408 pairing failure; this newer symptom likely needs service proxy configuration proof before deciding whether code, docs, or support is the best fix.
