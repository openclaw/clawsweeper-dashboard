## Summary

This is a service environment/configuration report rather than a remaining WhatsApp plugin bug: current main wires ambient HTTP(S) proxy env into the WhatsApp socket, while installed gateway services intentionally require managed proxy config or durable `OPENCLAW_PROXY_URL`.

## What This Changes

The issue reports that WhatsApp links through an HTTP proxy in 2026.5.3-1 but the managed gateway never reaches connected state or receives messages.

## Best Possible Solution

Keep the existing service proxy contract and guide affected users to configure the managed gateway proxy path, reopening only if durable proxy config fails with diagnostic proof.

## Reproduction Assessment

Yes, source-level. The reported split is reproducible from the code path: CLI login can see ambient proxy env, while an already installed LaunchAgent gateway will not receive ambient `HTTP_PROXY`/`HTTPS_PROXY` unless the managed proxy config or durable `OPENCLAW_PROXY_URL` path is used.

## Solution Assessment

Yes. The best resolution is not a repository code change from this report as written; configure `proxy.enabled` plus `proxy.proxyUrl`, or durable `OPENCLAW_PROXY_URL`, then reinstall/restart the gateway service.
