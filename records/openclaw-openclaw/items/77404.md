## Summary

Close as implemented on main: the WhatsApp socket factory now builds both the Baileys WebSocket agent and upload dispatcher from proxy environment variables, and the gateway listener uses that same factory for the connected runtime state.

## What This Changes

The issue reports that WhatsApp remains linked but not connected when HTTP_PROXY/HTTPS_PROXY are set after QR login.

## Best Possible Solution

Keep the current WhatsApp-owned proxy-aware socket implementation and ship it with the next OpenClaw/WhatsApp plugin release; reopen only with logs from a build containing 841eb81 if the proxy connection still fails.

## Reproduction Assessment

Yes, from source and the reporter's release evidence: the affected path is WhatsApp QR/runtime socket creation under env proxy settings. I did not live-connect WhatsApp, but current-main code and tests cover the proxy agent path that the report depends on.

## Solution Assessment

Yes. Fixing the WhatsApp socket factory is the narrow maintainable solution because both CLI login and the gateway listener call `createWaSocket`, keeping the behavior extension-owned and out of core.
