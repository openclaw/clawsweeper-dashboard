## Summary

Close: current main already wires WhatsApp Web sockets to proxy environment handling when the Gateway process has that environment, and the reported linked-but-not-connected state matches managed LaunchAgent service configuration rather than a source change.

## What This Changes

The issue asks OpenClaw to receive WhatsApp messages after QR login when HTTP/S proxy environment variables are set.

## Best Possible Solution

Configure the managed Gateway with `proxy.enabled`/`proxy.proxyUrl` or durable `OPENCLAW_PROXY_URL`, reinstall/restart the LaunchAgent, and open a new source bug only if that configured service still fails.

## Reproduction Assessment

No. Source inspection shows the WhatsApp code uses proxy env when the Gateway process has it, while the report only shows terminal proxy env plus a managed macOS LaunchAgent Gateway.

## Solution Assessment

Yes. The narrow maintainable solution is service-level proxy configuration, not another WhatsApp source change, unless logs from a correctly configured Gateway prove the socket still bypasses the proxy.
