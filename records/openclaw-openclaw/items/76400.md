## Summary

Keep open. The PR targets a real gateway starvation path, but the current patch silently drops oversized `chat.history` records before the existing placeholder/truncation behavior can run, and it is also missing the required changelog entry for a user-facing gateway fix.

## What This Changes

The PR adds a 256 KiB JSONL line guard in transcript parsing and usage fallback, delegates async title extraction to the bounded sync title reader, and adds focused session-utils tests.

## Best Possible Solution

Revise the oversized-line guard so multi-MB transcript lines do not get synchronously JSON-parsed while `chat.history` still surfaces the existing oversized-message placeholder, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `chat.history` can read a large recent tail window and call `parseTailTranscriptRecord()` before truncation, and the PR finding is source-reproducible because the new guard returns `null` before the existing placeholder replacement path can see the message.

## Solution Assessment

No. Skipping the expensive parse is the right direction, but the best fix should preserve the already-tested `chat.history` placeholder behavior for oversized records and add the required user-facing changelog entry.
