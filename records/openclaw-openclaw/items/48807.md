## Summary

Keep open: current main still restarts unhealthy running channels without an active-run drain, so the central fix remains useful, but this PR is not mergeable as-is because it lacks real behavior proof and still has source-level restart regressions.

## What This Changes

The PR adds a 30s active-run drain before health-monitor restarts, a 60s restart cooldown floor, fresh timing resolution, and expanded health-monitor tests.

## Best Possible Solution

Land a rebased gateway fix that preserves non-manual stop semantics, distinguishes drained/recovered/timeout outcomes, and includes redacted real Discord or gateway proof.

## Reproduction Assessment

Yes for a source-level path: current main can restart an unhealthy running account immediately by calling stopChannel, which aborts the channel task before any drain. The original live Discord dropped-reply symptom still needs after-fix real environment proof.

## Solution Assessment

No, not as-is. The affected module is the right place, but the patch must return an explicit drain outcome, continue restarts after drain timeout, and preserve stopChannel(..., { manual: false }).

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Treat drain timeouts as restartable:** `src/gateway/channel-health-monitor.ts:268-272`
  - body: If an account is initially unhealthy, then a run starts during the drain and remains active past 30s, drainActiveRuns returns without saying it timed out. The post-drain health check can then see healthy-busy state and skip the restart that the timeout is supposed to permit.
  - confidence: 0.91
- **[P2] Preserve non-manual stop semantics:** `src/gateway/channel-health-monitor.ts:291`
  - body: Current main calls stopChannel with { manual: false } for health-monitor restarts. This call omits that option, so stopChannel defaults to manual=true; stop timeouts can enter the manual-stop path and suppress recovery restart bookkeeping.
  - confidence: 0.88
