## Summary

Keep this PR open, but it is not merge-ready: the drain-window direction addresses a plausible gateway race, while the current branch still has two source-level restart regressions and only test-only proof.

## What This Changes

The PR changes the gateway channel health monitor to drain active runs before health-monitor restarts, enforce a restart cooldown floor, resolve fresh timing config each check, and expand health-monitor tests.

## Best Possible Solution

Keep the drain-before-abort direction, but return an explicit drain outcome, continue restarts after drain timeout, preserve non-manual health-monitor stop semantics, rebase on main, and require redacted real gateway/Discord proof.

## Reproduction Assessment

Yes, there is a high-confidence source-level path: current main aborts unhealthy running channels immediately, and PR head can treat a still-busy post-drain timeout as recovery. The original production dropped-reply symptom still needs real deployment proof after the fix.

## Solution Assessment

No, not as-is. The affected module is the right place, but the patch must distinguish drain timeout from true recovery and preserve `stopChannel(..., { manual: false })` before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Treat drain timeouts as restartable:** `src/gateway/channel-health-monitor.ts:268-272`
  - body: When a run starts after the initial unhealthy snapshot and remains active past the 30s drain window, `drainActiveRuns` returns with no timeout/drained status. The post-drain health check can then return `healthy` for `reason: "busy"` and skip the restart that the timeout is supposed to permit.
  - confidence: 0.91
- **[P2] Preserve non-manual stop semantics:** `src/gateway/channel-health-monitor.ts:291`
  - body: Current main calls `stopChannel` with `{ manual: false }` for health-monitor restarts. Omitting that option makes stop timeouts enter the manual-stop path, which can suppress the recovery restart path after the abort timeout.
  - confidence: 0.88
