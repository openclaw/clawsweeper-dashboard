## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the drain-before-abort behavior, while the PR has a concrete timeout bug: after the 30-second drain expires with a still-active recent run, the post-drain health check treats the account as healthy/busy and skips the restart instead of executing the advertised abort fallback.

## What This Changes

The PR changes the gateway channel health monitor and tests to add active-run draining before health-monitor restarts, a 60-second restart cooldown floor, and an optional fresh timing callback.

## Best Possible Solution

Keep the drain-before-abort direction, but make the drain helper report whether it drained, timed out, observed stale state, or stopped, and only treat post-drain busy health as recovery when the drain actually completed rather than timed out. Rebase onto current main and keep the fix localized to the health monitor/tests while preserving current transport-liveness policy.

## Reproduction Assessment

Yes. A focused reproduction is a running disconnected account with activeRuns > 0 and a recent lastRunActivityAt that remains active past DRAIN_WINDOW_MS; the PR then exits the drain deadline and skips restart because postDrainHealth returns healthy/busy.

## Solution Assessment

No, not as-is. The proposed drain/cooldown approach is the right narrow area, but the post-drain recheck currently defeats the timeout fallback; the safer fix is to distinguish timeout from recovery and add a regression test with fresh lastRunActivityAt still active after the drain window.
