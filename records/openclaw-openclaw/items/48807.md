## Summary

Keep open: the PR is aimed at a real gateway race, but the current head can still skip the promised timeout fallback for live active runs after the drain window expires, and it lacks the required changelog entry.

## What This Changes

The PR adds health-monitor active-run draining before restarts, a 60-second restart cooldown floor, fresh timing resolution, and related gateway monitor tests.

## Best Possible Solution

Keep the drain-before-abort direction, but make drainActiveRuns return an explicit outcome and only skip restart for stopped, removed, disabled, or genuinely recovered channels; timeout with still-live runs should continue to fallback restart.

## Reproduction Assessment

Yes. Source inspection gives a focused reproduction: a running disconnected account with activeRuns > 0, busy true, and recent lastRunActivityAt that remains active past DRAIN_WINDOW_MS will make the PR's post-drain health check return healthy/busy and skip restart.

## Solution Assessment

No, not as-is. The branch is in the right owner module and the drain/cooldown idea is narrow, but timeout must be represented separately from recovery and covered by a live-run regression test before this is the maintainable fix.
