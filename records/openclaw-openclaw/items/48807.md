## Summary

Keep open: current main still restarts unhealthy running channels without draining active runs, so the PR addresses a real gateway message-delivery risk, but the PR head is conflicting, lacks real behavior proof, and still has two restart-semantics blockers.

## What This Changes

The PR changes the gateway channel health monitor to poll active runs before recovery stops, enforce a minimum restart cooldown, refresh timing values per check, and add health-monitor regression tests.

## Best Possible Solution

Keep the drain/cooldown direction, but land it only after a rebase, explicit drain outcomes for drained/timed-out/stopped, preserved `{ manual: false }` recovery stops, and redacted real Discord or gateway proof that replies still deliver through a health-monitor restart.

## Reproduction Assessment

Yes at source level: current main restarts an unhealthy running account by immediately calling `stopChannel(..., { manual: false })` without draining active runs. The exact production Discord dropped-text symptom was not live-reproduced in this read-only review.

## Solution Assessment

No, not as-is: the approach is plausible, but the PR needs explicit drain-result semantics and the existing recovery-stop contract preserved before it is the narrow maintainable fix.
