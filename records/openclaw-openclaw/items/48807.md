## Summary

Keep this PR open, but do not merge it as-is: the branch addresses a real gateway restart/drop risk, yet the current diff can skip the promised drain-timeout restart when a live run remains busy, and it is missing the required user-facing changelog entry.

## What This Changes

The PR changes the gateway channel health monitor and tests to drain active runs before health-monitor restarts, enforce a 60-second restart cooldown floor, and resolve fresh timing each cycle.

## Best Possible Solution

Keep the drain-before-abort direction, but make drainActiveRuns return an explicit outcome and only skip restart for stopped, removed, disabled, or genuinely recovered channels; timeout with still-live runs should continue to the fallback restart.

## Reproduction Assessment

Yes. Source inspection gives a focused reproduction: a running disconnected account with activeRuns > 0, busy true, and recent lastRunActivityAt that remains active past DRAIN_WINDOW_MS will make the PR's post-drain health check return healthy/busy and skip the timeout restart.

## Solution Assessment

No, not as-is. The branch is in the right owner module and the drain/cooldown idea is narrow, but timeout must be represented separately from recovery and covered by a live-run regression test before this is the maintainable fix.
