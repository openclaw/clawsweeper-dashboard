## Summary

Keep this PR open, but do not merge it as-is. Current main still restarts unhealthy running channels without draining active runs, so the contribution addresses a real gap; however the latest PR head can skip the promised timeout fallback when a recent active run remains busy after the drain deadline.

## What This Changes

The PR changes the gateway channel health monitor and tests to add active-run draining before health-monitor restarts, a 60-second restart cooldown floor, and a fresh timing callback.

## Best Possible Solution

Keep the drain-before-abort direction, but make the drain helper return an explicit outcome such as drained, timed out, stale, or stopped. Only treat post-drain busy health as recovery when the run actually drained or the channel genuinely recovered, then add a regression test with fresh lastRunActivityAt still active after the drain window and include the required changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is a running disconnected account with activeRuns > 0, busy true, and a recent lastRunActivityAt that remains active past DRAIN_WINDOW_MS; the PR then exits the drain deadline and skips restart because postDrainHealth returns healthy/busy.

## Solution Assessment

No, not as-is. The drain/cooldown approach is the right narrow area, but the post-drain recheck currently defeats the timeout fallback; the safer fix is to distinguish timeout from recovery and cover that live-run timeout case in tests.
