## Summary

Keep open: this PR addresses a plausible gateway restart race, but the current branch is still not safe to merge because source inspection shows a live-run drain-timeout path that skips the fallback restart, the branch also needs to preserve current-main recovery stop semantics, and no real after-fix gateway proof is present.

## What This Changes

The PR adds a gateway health-monitor drain window before restarting running channels, a 60-second restart cooldown floor, fresh timing resolution, and related tests.

## Best Possible Solution

Keep the drain-before-abort direction, but make the drain return an explicit outcome, continue the restart on live-run timeout, preserve current-main stopChannel({ manual: false }) recovery behavior, and add focused regression plus redacted real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an account enters the restart path, a live run remains active with recent lastRunActivityAt through DRAIN_WINDOW_MS, and the post-drain busy health result skips stop/start.

## Solution Assessment

No, not as-is. The module and drain/cooldown direction are appropriate, but the branch must distinguish timeout from recovery and preserve current-main recovery stop semantics before it is the maintainable fix.
