## Summary

Keep open: current main still lacks the drain-before-abort behavior, but this external PR is not mergeable as-is because the branch can skip the fallback restart after a live-run drain timeout and changes health-monitor stops to manual semantics, with no real behavior proof yet.

## What This Changes

The branch adds a health-monitor drain window before restarting running channels, a 60-second restart cooldown floor, fresh timing resolution, and related tests.

## Best Possible Solution

Keep the health-monitor drain direction, but make the drain return an explicit outcome, continue the restart on live-run timeout, preserve stopChannel({ manual: false }), rebase onto current gateway health policy, and require redacted real gateway/Discord proof before merge.

## Reproduction Assessment

Yes, via source inspection rather than a live run: a running disconnected account with recent activeRuns enters the restart path, stays busy through DRAIN_WINDOW_MS, and then PR head treats the post-drain busy health result as recovery. The original dropped-reply production symptom still needs real gateway/Discord proof.

## Solution Assessment

No, not as-is. The module and drain-before-abort direction are appropriate, but the branch must distinguish drain timeout from recovery and preserve the current recovery-stop semantics before it is the maintainable fix.
