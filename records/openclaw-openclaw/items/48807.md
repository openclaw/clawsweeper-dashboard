The PR changes gateway health-monitor restart logic to drain active channel runs before restart, add a 60s cooldown floor, refresh timing values per check, and expand health-monitor tests.

## Best Possible Solution

Land a rebased gateway fix that drains only live active runs, treats drain timeout as a restartable outcome, preserves non-manual recovery stop semantics, and includes redacted real Discord or gateway proof.

## Reproduction Assessment

Yes for the source-level race: current main restarts an unhealthy running account by immediately calling `stopChannel(..., { manual: false })`, which aborts the channel task without draining active runs. The exact production Discord dropped-text symptom was not live-reproduced here.

## Solution Assessment

No, not as-is. The health monitor is the right boundary, but the branch needs explicit drain timeout handling, preserved non-manual stop semantics, a rebase, and real-environment proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return a timeout outcome before rechecking health:** `src/gateway/channel-health-monitor.ts:263-272`
  - body: After the 30s drain deadline, `drainActiveRuns` returns the same `void` value as a successful drain. If `activeRuns` is still recent, the post-drain `evaluateChannelHealth` sees the account as `busy`/healthy and `continue`s, so the unhealthy channel is not restarted even though the timeout fallback was supposed to abort and recover it. Have the drain helper report whether it drained, timed out, or stopped, and only let the recovered-health recheck suppress restart after a real drain/recovery.
  - confidence: 0.86
- **[P2] Preserve recovery stop semantics:** `src/gateway/channel-health-monitor.ts:291`
  - body: This replaces current main's `stopChannel(..., { manual: false })` recovery stop with a plain `stopChannel(...)`. Since `stopChannel` defaults `manual` to true, a stop timeout records manual-stop semantics instead of `restartPending` recovery state, which can suppress the intended health-monitor recovery path. Pass `{ manual: false }` here and update the new tests to assert it.
  - confidence: 0.94
