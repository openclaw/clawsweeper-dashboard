## Summary

Keep open: current main still restarts unhealthy running channels without draining active runs, so the central bug is not obsolete. The PR is not mergeable as-is because the drain timeout path can skip the intended restart, the health-monitor stop call loses non-manual semantics, and contributor real behavior proof is still absent.

## What This Changes

The PR adds a health-monitor active-run drain window, a 60s restart cooldown floor, a fresh timing callback, and expanded gateway health-monitor tests.

## Best Possible Solution

Land a rebased gateway fix that drains active runs without losing restart semantics, distinguishes drained/recovered/timeout outcomes, and includes redacted Discord or gateway proof.

## Reproduction Assessment

Yes for a source-level path: current main can call stopChannel(..., { manual: false }) immediately for an unhealthy running account without checking activeRuns. I did not live-reproduce the Discord dropped-reply symptom in this read-only review.

## Solution Assessment

No, not as-is. The affected module is the right place, but the patch needs explicit drain outcomes, timeout restart handling, and preserved manual:false stop semantics before real-environment proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Treat drain timeouts as restartable:** `src/gateway/channel-health-monitor.ts:268-272`
  - body: If an unhealthy account still has a recent active run when the 30s drain deadline expires, drainActiveRuns returns without saying it timed out. The post-drain health check can then see healthy/busy state and continue, skipping the restart that the timeout path is supposed to allow.
  - confidence: 0.91
- **[P2] Preserve non-manual stop semantics:** `src/gateway/channel-health-monitor.ts:291`
  - body: Current main passes { manual: false } for health-monitor stops so stop timeouts stay in recovery-restart state. This new call drops that option, so stopChannel defaults to manual=true and can mark the account manually stopped on timeout, changing recovery bookkeeping.
  - confidence: 0.88
