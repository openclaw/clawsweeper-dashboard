## Summary

Keep open: current main has an internal command-lane snapshot API, but it is not exposed through `session_status`, a `sessions_capacity` tool, `subagents list`, or a structured global-capacity error from `sessions_spawn`. The related open issues are adjacent, not a complete superseding tracker for this narrow capacity surface.

## What This Changes

The issue asks for a structured way for agents to inspect sub-agent concurrency capacity before calling `sessions_spawn`.

## Best Possible Solution

Add an additive read-only capacity surface backed by `getCommandLaneSnapshot(CommandLane.Subagent)`, and keep it clearly separate from the requester-scoped `maxChildrenPerAgent` gate.

## Reproduction Assessment

Yes. A current-main static reproduction is enough: inspect `session_status`, `subagents`, and `sessions_spawn` and compare them with the existing subagent lane snapshot API; no agent-facing output currently exposes the global limit, active count, queued count, or remaining slots.

## Solution Assessment

Yes, the narrow maintainable fix is to expose the existing queue snapshot as structured read-only data. Changing `sessions_spawn` to reject at global capacity would alter today’s queueing semantics and is riskier unless maintainers explicitly want that behavior.
