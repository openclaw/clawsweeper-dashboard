## Summary

Keep this issue open. Current main still has the global sub-agent lane limit and per-requester child limit, but no agent-readable capacity surface in `session_status`, `subagents`, or `sessions_spawn`; the related #57053 budget issue and #10467 multi-lane issue are adjacent rather than superseding this specific capacity-visibility request.

## What This Changes

The issue requests structured exposure of sub-agent concurrency capacity through `session_status`, a new `sessions_capacity` tool, or structured `sessions_spawn` capacity feedback.

## Best Possible Solution

Expose a read-only sub-agent capacity object from core, preferably on `session_status.details` as the narrowest existing agent-facing surface, backed by `getCommandLaneSnapshot(CommandLane.Subagent)` plus requester child-cap usage from the subagent registry. Document that global capacity queues rather than rejects unless maintainers deliberately choose a new `capacity_exceeded` spawn behavior.

## Reproduction Assessment

Yes for the capability gap: inspecting current main shows `session_status.details` and `subagents(action="list")` lack global lane capacity fields, while `sessions_spawn` still queues on `CommandLane.Subagent`. The existing depth-limit test also demonstrates that `maxConcurrent` is not a structured spawn rejection gate.

## Solution Assessment

Yes, a read-only capacity surface is the best next step. Extending `session_status` is narrower than adding a new tool, and it can use the current queue snapshot seam without changing spawn semantics or the broader multi-lane request tracked in #10467.
