## Summary

Keep this PR open: the feature is plausible, but the latest branch still has two source-reproducible correctness issues and the external contributor has not provided real after-fix behavior proof beyond tests.

## What This Changes

The PR adds an opt-in cron output-history flag that records recent isolated agent outputs and injects them into later isolated cron prompts.

## Best Possible Solution

Keep the feature behind its opt-in payload flag, but filter non-substantive control replies before history storage and normalize malformed outputHistory inputs consistently with existing cron payload booleans.

## Reproduction Assessment

Yes for the review findings: current source shows successful cron runs can return suppressed NO_REPLY/HEARTBEAT_OK outputText, and the PR records successful outputText without filtering it. The malformed boolean path is also source-reproducible because outputHistory is accepted only as a boolean in schema but not cleaned up when a model sends a string value.

## Solution Assessment

No. The feature direction is plausible, but this patch should first filter control-token replies from recentOutputs and align outputHistory normalization with the existing allowUnsafeExternalContent boolean handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter control-token replies before storing history:** `src/cron/service/timer.ts:346-355`
  - body: `applyJobResult` records every successful `result.outputText`, but the cron runner can return successful suppressed outputs such as `NO_REPLY` or heartbeat acknowledgements when delivery is intentionally skipped. With `outputHistory` enabled, the next run then sees control text as previous output instead of the last substantive result; skip or sanitize these control-token-only replies before appending to `recentOutputs`.
  - confidence: 0.87
- **[P2] Normalize malformed outputHistory values before validation:** `src/cron/normalize.ts:104-105`
  - body: Adding `outputHistory` as an agent-turn field means malformed model output like `outputHistory: "true"` now survives normalization and fails the strict gateway schema. Mirror the existing `allowUnsafeExternalContent` cleanup or coercion behavior so recoverable cron add/update calls do not become invalid-params errors because the flag was stringified.
  - confidence: 0.76
