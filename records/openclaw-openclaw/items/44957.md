## Summary

Keep open: current main does not implement cron output history, and the PR is useful but still needs changes because it can persist silent/control replies into future prompts and has normalization/changelog gaps.

## What This Changes

The PR adds an opt-in cron `outputHistory` payload flag that stores recent isolated-agent outputs in job state and injects them into later isolated cron prompts, with schema, tool, and test updates.

## Best Possible Solution

Land a rebased or replacement version that keeps the opt-in API but records only substantive outputs, preserves existing delivery/silent-token contracts, normalizes the new field consistently, and documents the user-facing behavior in the changelog.

## Reproduction Assessment

Yes. Source inspection shows a `NO_REPLY` or heartbeat-only run can finish `ok` with `outputText`, delivery suppression is handled separately, and the PR records any successful non-empty output into history.

## Solution Assessment

No, not as-is. The feature direction is maintainable, but the current patch should filter non-substantive control replies, normalize invalid `outputHistory` values, and add the required changelog entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter silent replies before saving history:** `src/cron/service/timer.ts:347-355`
  - body: The new recording gate appends any successful non-empty `outputText`, but current cron delivery suppresses `NO_REPLY` and heartbeat-only responses without clearing that field. A quiet/no-op run can therefore store control text as the previous output and inject it into the next prompt instead of preserving the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values like the same path strips invalid `allowUnsafeExternalContent`. Malformed model output such as `outputHistory: "true"` can survive normalization and fail schema validation instead of being ignored or recovered.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/tools/cron-tool.ts:263-264`
  - body: This is a user-facing cron feature that adds new tool/protocol/runtime behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing feat/fix/perf changes to have an entry in the active changelog section.
  - confidence: 0.86
