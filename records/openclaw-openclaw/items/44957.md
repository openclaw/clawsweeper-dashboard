## Summary

Keep this PR open: current main does not implement cron output history, and the branch is useful, but source inspection found two merge-blocking behavior gaps plus a required changelog omission.

## What This Changes

The PR adds an opt-in cron agent-turn `outputHistory` flag that records recent outputs in job state, injects them into isolated prompts, updates cron normalization/protocol/tool surfaces, and adds focused tests.

## Best Possible Solution

Repair the PR branch by filtering silent/control outputs before history writes, normalizing malformed `outputHistory` inputs consistently with existing cron payload fields, adding a changelog entry, then rerunning focused cron/protocol/tool tests and the changed gate.

## Reproduction Assessment

Yes. Source inspection shows high-confidence paths for the PR-introduced bugs: silent/control token text can flow into `result.outputText` and then `recentOutputs`, and stringified `outputHistory` values survive normalization until strict gateway schema validation rejects them.

## Solution Assessment

No. The feature direction is plausible, but the patch should not merge until it filters no-op control outputs, handles malformed boolean inputs consistently, and records the user-facing feature in the changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Filter silent control replies before recording history:** `src/cron/service/timer.ts:346`
  - body: `applyJobResult` now records any successful non-empty `result.outputText`, but cron output text is populated before delivery suppression. A no-op run that returns `HEARTBEAT_OK` or `NO_REPLY` can therefore be saved as prior output, so the next isolated prompt sees a control token instead of the last substantive result. Please skip those silent/control-only outputs when appending `recentOutputs`.
  - confidence: 0.88
- **[P2] Normalize non-boolean outputHistory values:** `src/cron/normalize.ts:104`
  - body: The normalizer adds `outputHistory` as an agent-turn hint, but it does not strip or coerce non-boolean values the way existing boolean fields do. Inputs such as `payload: { kind: "agentTurn", message: "...", outputHistory: "true" }` will reach the gateway schema and fail validation instead of being recovered or cleaned up. Please normalize this field for add/update paths before validation.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/tools/cron-tool.ts:261`
  - body: This is a user-facing cron feature, but the PR does not update `CHANGELOG.md`. Repository policy requires user-facing `feat` changes to record an Unreleased changelog entry before merge.
  - confidence: 0.94
