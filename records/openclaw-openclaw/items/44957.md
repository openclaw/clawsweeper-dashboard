## Summary

Keep open. Current main does not implement cron output history, and this PR is useful source work but is not ready to merge: the head is dirty against current main and the latest diff can record silent control replies as prompt history.

## What This Changes

The PR adds an opt-in cron `outputHistory` flag that stores recent isolated-agent outputs in job state and injects them into later cron prompts, with schema, tool, and test coverage updates.

## Best Possible Solution

Land a rebased or replacement version of the feature that keeps the opt-in API but filters non-substantive control replies before recording history, preserves current prompt/delivery contracts, normalizes the new field consistently, and documents the user-facing cron behavior in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an isolated cron run that returns only `NO_REPLY` or heartbeat acknowledgement text can be classified as `ok` with `outputText`, delivery is suppressed separately, and the PR then records that text into `recentOutputs`.

## Solution Assessment

No, not as-is. The feature direction is maintainable, but the current patch needs a current-main rebase plus output filtering, normalization cleanup, and changelog coverage before it is the narrow safe solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter silent replies before saving history:** `src/cron/service/timer.ts:348-355`
  - body: The new recording gate appends any successful non-empty `outputText`, but current cron delivery suppresses `NO_REPLY` and heartbeat-only responses without clearing that field. A quiet/no-op run can therefore store `NO_REPLY` or `HEARTBEAT_OK` as the previous output and inject control text into the next prompt instead of preserving the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values the way the same function strips invalid `allowUnsafeExternalContent`. A nested payload such as `outputHistory: "true"` survives normalization and fails the gateway schema instead of being ignored or coerced in the malformed-model recovery path this code supports.
  - confidence: 0.78

  - body: This is a user-facing `feat(cron)` change that adds new cron tool and protocol behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing feat/fix/perf changes to have an entry in the active changelog section.
