## Summary

Keep open: current main does not already contain this cron output-history feature, and the PR still has correctness issues plus missing external real behavior proof.

## What This Changes

The PR adds a per-job cron `outputHistory` flag that stores recent isolated-agent outputs and injects them into later isolated cron prompts.

## Best Possible Solution

Land a rebased implementation only after filtering silent/control outputs from history, normalizing invalid flag values consistently, documenting the state/protocol contract, and adding redacted real runtime proof.

## Reproduction Assessment

Yes, for the review findings: source inspection shows the PR records raw successful `outputText` even when delivery suppresses `NO_REPLY`/`HEARTBEAT_OK`, and leaves non-boolean `outputHistory` values to fail schema validation. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but the current patch should filter non-substantive control output and mirror existing normalization cleanup before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Skip silent control tokens before appending history:** `src/cron/service/timer.ts:350`
  - body: `applyJobResult` appends any successful `outputText`, but silent cron deliveries still return ok with `outputText` set to `NO_REPLY`, `HEARTBEAT_OK`, or trailing silent-token text after delivery suppression. With `outputHistory` enabled, the next run then sees control text as previous output instead of useful context, which defeats the feature for no-op monitoring runs. Filter silent/heartbeat-only output before writing `recentOutputs`.
  - confidence: 0.91
- **[P2] Drop non-boolean outputHistory values during normalization:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now uses boolean `outputHistory` as an agent-turn hint, but it never removes non-boolean values afterward. Inputs like `payload: { kind: "agentTurn", message: "...", outputHistory: "true" }` survive normalization and then fail the gateway schema, unlike `allowUnsafeExternalContent` which is stripped when malformed. Strip or coerce invalid `outputHistory` values before validation.
  - confidence: 0.86
