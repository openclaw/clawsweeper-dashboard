## Summary

Keep open: current main does not implement cron output history, and the PR still needs changes before merge because it records silent control-token outputs into future prompts and lacks contributor real-behavior proof.

## What This Changes

The PR adds an opt-in cron `outputHistory` agent-turn field, records recent isolated-agent outputs in job state, injects them into later prompts, and updates cron tool, gateway schema, and focused tests.

## Best Possible Solution

Land a rebased version that keeps the opt-in history API, records only substantive outputs, normalizes malformed field values consistently with adjacent payload fields, preserves the external-content safety wrapper, and includes real after-fix proof.

## Reproduction Assessment

Yes. Source inspection shows current cron can produce ok `outputText` for `NO_REPLY` or heartbeat-only suppressed delivery, while the PR records any successful non-empty `outputText` into history.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch should filter silent/control outputs, normalize invalid `outputHistory` values like adjacent payload fields, rebase on current cron changes, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter silent control tokens before recording history:** `src/cron/service/timer.ts:346-351`
  - body: The new gate appends any successful non-empty `outputText`, but current cron computes that text before `NO_REPLY` and heartbeat-only delivery suppression. Quiet no-op runs can therefore store `NO_REPLY` or `HEARTBEAT_OK` as prior output and inject control text into the next prompt instead of the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values the way the same normalizer strips invalid `allowUnsafeExternalContent`. Malformed model output such as `outputHistory: "true"` survives normalization and fails schema validation instead of being ignored or recovered.
  - confidence: 0.78
