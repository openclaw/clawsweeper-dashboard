## Summary

Keep open. Current main does not implement the cron output-history feature, and the PR is useful source work, but the patch still records non-user-visible control replies into history and leaves the new boolean field less tolerant than the surrounding cron normalizer behavior.

## What This Changes

The PR adds an opt-in cron agent-turn `outputHistory` flag, stores up to five truncated recent outputs in cron job state, injects them into isolated-agent prompts, and updates cron tool, service, and gateway protocol tests.

## Best Possible Solution

Keep the feature open as source work, but land it only after filtering silent/heartbeat control tokens before history persistence, aligning `outputHistory` normalization with existing optional boolean handling, rebasing onto current cron runner code, and adding the expected user-facing changelog/docs coverage.

## Reproduction Assessment

Yes. A cron isolated run that returns `NO_REPLY` or `HEARTBEAT_OK` can still complete with `status: "ok"` and `outputText` set to that control text; the PR's new `applyJobResult` branch would then persist it whenever `payload.outputHistory` is enabled.

## Solution Assessment

No. The overall design is reasonable, but the current patch should filter non-user-visible control replies before recording history and should drop or coerce malformed `outputHistory` values consistently with adjacent cron payload booleans.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P2] Filter silent control tokens before recording history:** `src/cron/service/timer.ts:345-348`
  - body: `outputText` can be `NO_REPLY` or `HEARTBEAT_OK` after the runner has suppressed user-visible delivery. Because this branch records every successful non-empty `outputText`, the next cron prompt can receive control text as prior output instead of the last substantive result. Filter the silent/heartbeat tokens before appending to `recentOutputs`.
  - confidence: 0.88

  - body: The new field is accepted only when it is already a boolean, but malformed model/tool output such as `outputHistory: "true"` is left in the payload and then rejected by the gateway schema. Mirror the existing optional-boolean cleanup used for `allowUnsafeExternalContent` so otherwise recoverable cron add/update calls do not fail validation.
  - confidence: 0.79
