## Summary

Keep open. Current main does not implement `outputHistory`, and the PR is useful but still has a source-backed correctness blocker around saving silent/control replies into prompt history plus smaller normalization and changelog gaps.

## What This Changes

The PR adds an opt-in cron `outputHistory` payload flag that records recent isolated-agent outputs in job state and injects them into later isolated cron prompts, with schema, tool, and test updates.

## Best Possible Solution

Land a rebased or replacement version that keeps the opt-in API but records only substantive outputs, normalizes invalid `outputHistory` values consistently, preserves the external-content safety boundary, and documents the user-facing behavior in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current isolated cron can finish `ok` with `outputText` for `NO_REPLY` or heartbeat-only suppressed delivery, while the PR records any successful non-empty `outputText` into history.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch should filter silent/control outputs, handle malformed `outputHistory` values like adjacent payload fields, and add the required changelog entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter silent replies before saving history:** `src/cron/service/timer.ts:346-350`
  - body: The new recording gate appends any successful non-empty `outputText`, but current cron derives that text before `NO_REPLY` and heartbeat-only delivery suppression. A quiet no-op run can therefore store `NO_REPLY` or `HEARTBEAT_OK` as prior output and inject control text into the next prompt instead of preserving the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values the way the same normalizer strips invalid `allowUnsafeExternalContent`. Malformed model output such as `outputHistory: "true"` can survive normalization and fail schema validation instead of being ignored or recovered.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/tools/cron-tool.ts:263-264`
  - body: This is a user-facing cron feature that adds new tool, protocol, runtime, and state behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing feat/fix/perf changes to have an entry in the active changelog section.
  - confidence: 0.86
