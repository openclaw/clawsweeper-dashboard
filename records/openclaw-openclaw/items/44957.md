## Summary

Keep open: current main does not implement this capability, and the PR is still useful, but it is not merge-ready because it is conflicted, lacks real behavior proof, and has source-backed correctness blockers around what gets persisted into cron prompt history.

## What This Changes

The PR adds an opt-in cron `outputHistory` flag that stores recent isolated-agent outputs in cron job state and injects them into later isolated prompts, with protocol/tool schema updates and tests.

## Best Possible Solution

Land a rebased, maintainer-approved opt-in cron memory mode that filters non-substantive output before storing, normalizes the new flag consistently, includes focused regression coverage, and has redacted real cron-run proof.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source inspection shows successful `outputText` can be persisted without filtering control-only replies, and non-boolean `outputHistory` is not cleaned up before strict schema validation. The underlying request is a new opt-in capability, not a current-main bug reproduction.

## Solution Assessment

Unclear as proposed: a per-job opt-in history flag is plausible, but the branch must be rebased and the persisted-memory contract needs maintainer approval. The safer implementation filters non-substantive output before persistence and treats the new flag like adjacent boolean fields in normalization.
