## Summary

Keep open. Current main does not implement this cron output-history feature and no canonical duplicate was found, but this PR is not merge-ready: the final head still has two source-reproducible correctness gaps and no real behavior proof beyond tests/CI.

## What This Changes

This PR adds a cron agent-turn `outputHistory` flag that stores recent isolated-run outputs in job state, injects them into later prompts, updates cron tool/protocol normalization, and adds tests.

## Best Possible Solution

If maintainers want this feature, keep it as a cron-owned option, fix the recording and normalization gaps, refresh validation against current main, and require redacted real cron-run proof before merge.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows exact control-token `outputText` can be recorded, and malformed string `outputHistory` values survive normalization. For the feature itself, reproduction is not applicable on current main because the capability is absent.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the PR needs the two correctness fixes, real behavior proof, and maintainer agreement on the new cron memory semantics before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter control tokens before recording output history:** `src/cron/service/timer.ts:354-356`
  - body: This appends any successful non-empty `outputText` to `recentOutputs`, but the isolated-run path can return exact control replies such as `NO_REPLY` or `HEARTBEAT_OK` as `outputText` before delivery suppression finishes. Quiet/no-op runs with `outputHistory` enabled will then inject those internal tokens into the next prompt instead of the last substantive result; skip known silent and heartbeat control tokens before appending.
  - confidence: 0.88
- **[P2] Drop invalid outputHistory values during normalization:** `src/cron/normalize.ts:105`
  - body: The PR adds `outputHistory` as a payload field, but `coercePayload` never removes non-boolean values the way it does for `allowUnsafeExternalContent`. A flattened/model-generated payload like `outputHistory: "true"` survives normalization and then fails gateway validation instead of being ignored or recovered, so cron add/update recovery still breaks for a common malformed boolean shape.
  - confidence: 0.82
