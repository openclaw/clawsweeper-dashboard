## Summary

Keep open. The PR adds a useful cron feature, but the current patch still has two source-traceable correctness issues in output recording and normalization, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds an `outputHistory` flag for isolated cron agent-turn jobs, stores recent run outputs in cron job state, injects them into later prompts, and updates cron schemas, normalization, tool handling, and tests.

## Best Possible Solution

Keep the cron output-history feature in the cron agent-turn/runtime boundary, but filter non-substantive control responses before persistence, normalize `outputHistory` with the same defensive cleanup as adjacent payload booleans, update focused tests, and require real cron-run proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the patch issues: successful `ok` runs can carry `HEARTBEAT_OK` or `NO_REPLY` in `outputText`, and malformed/model-produced `outputHistory: "true"` survives normalization until boolean-only schema validation rejects it.

## Solution Assessment

No, not as-is. The overall direction belongs in cron, but the current implementation should avoid persisting control tokens, clean up invalid `outputHistory` values like neighboring agent-turn booleans, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip control tokens before recording history:** `src/cron/service/timer.ts:348-351`
  - body: `applyJobResult` records any non-empty `ok` `outputText`, but isolated runs compute `outputText` before heartbeat/NO_REPLY delivery suppression. A no-op run can therefore store `HEARTBEAT_OK` or `NO_REPLY` as prior output, so the next prompt gets control text instead of the last substantive result.
  - confidence: 0.88
- **[P2] Prune non-boolean outputHistory during normalization:** `src/cron/normalize.ts:105`
  - body: Adding `outputHistory` to agent-turn kind inference without the same non-boolean cleanup used for `allowUnsafeExternalContent` lets values like `outputHistory: "true"` survive normalization and then fail the gateway boolean schema. That breaks the malformed/model-produced cron add/update recovery path this PR is extending.
  - confidence: 0.82
