## Summary

Keep open. Current main does not already implement this cron output-history feature, and this external PR still needs real behavior proof plus at least one blocking correctness fix before it is merge-ready.

## What This Changes

The PR adds an opt-in cron agent-turn `outputHistory` flag that stores recent isolated-agent outputs in job state, injects them into later prompts, updates schema/tool normalization, and adds focused tests.

## Best Possible Solution

Land a rebased version only after maintainers accept the opt-in API, control-token outputs are filtered, malformed flag values are normalized, and real cron behavior proof is attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current isolated cron can return `ok` with `outputText` for silent or heartbeat-suppressed delivery, while the PR records any successful non-empty `outputText` into `recentOutputs`.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch should filter silent/control outputs and normalize malformed `outputHistory` values before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter control replies before saving history:** `src/cron/service/timer.ts:345-350`
  - body: The new history gate appends any successful non-empty `outputText`, but current cron derives that text before `NO_REPLY` and heartbeat-only delivery suppression. A quiet no-op run can therefore store `NO_REPLY` or `HEARTBEAT_OK` as prior output and inject control text into the next prompt instead of preserving the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values the way the same normalizer strips invalid `allowUnsafeExternalContent`. Malformed model output such as `outputHistory: "true"` can survive normalization and fail schema validation instead of being ignored or recovered.
  - confidence: 0.78
