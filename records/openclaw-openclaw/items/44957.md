## Summary

Keep this PR open for maintainer review: current main does not implement the proposed isolated-cron output-history feature, but the PR is still a feature/API change with source-reproducible correctness findings and no real behavior proof.

## What This Changes

The PR adds an opt-in cron `outputHistory` agent-turn flag that stores up to five truncated recent outputs in job state and injects them into later isolated cron prompts, with tool, normalization, gateway schema, and test updates.

## Best Possible Solution

Decide whether isolated cron should get bounded prior-output memory; if accepted, implement it on current cron contracts, filter non-substantive outputs, align docs/protocol/tests, and require real-run proof.

## Reproduction Assessment

Yes. The blocking review findings are source-reproducible from PR head: the timer records every successful `outputText`, while the delivery path can return `ok` with silent reply text still present, and malformed `outputHistory` values survive normalization.

## Solution Assessment

No. The feature direction is plausible, but this patch should filter control-token outputs, normalize the new flag consistently with adjacent payload fields, rebase onto current cron contracts, and provide real behavior proof before merge.
