## Summary

Keep open. Current main still does not implement isolated cron output history, and this PR remains the active implementation path, but the proposed patch still has merge-blocking correctness issues around persisting silent/control replies and recovering malformed `outputHistory` tool booleans.

## What This Changes

The PR adds a per-job cron `outputHistory` flag that persists bounded recent isolated-agent outputs in cron state and injects them into later isolated cron prompts, with cron tool, normalization, Gateway schema, and test updates.

## Best Possible Solution

Land a refined version that stores only meaningful bounded agent outputs, filters silent/control tokens before persistence, keeps external-content boundaries intact, normalizes or drops malformed `outputHistory` values consistently with existing cron recovery, and updates public cron docs plus changelog coverage for the new capability.

## Reproduction Assessment

Yes. A focused unit path can reproduce the PR bugs by enabling `payload.outputHistory`, returning a successful isolated run whose final text is `NO_REPLY` or `HEARTBEAT_OK`, and by normalizing an agent-turn payload with `outputHistory: "true"`.

## Solution Assessment

No, not as currently proposed. The feature direction is reasonable and earlier review concerns were addressed, but the patch should filter non-substantive control replies and normalize or drop non-boolean `outputHistory` before strict Gateway validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter silent replies before recording history:** `src/cron/service/timer.ts:347-355`
  - body: `applyJobResult` appends any successful `outputText`, but isolated delivery can return `status: "ok"` while the output is still `NO_REPLY`, `HEARTBEAT_OK`, or stale parent control text after delivery suppression. Those internal tokens would be injected into the next run as prior output, replacing useful history with non-substantive control text.
  - confidence: 0.86
- **[P2] Normalize malformed outputHistory booleans:** `src/cron/normalize.ts:101-102`
  - body: The new normalization path only recognizes real booleans as `outputHistory` and leaves non-boolean values in the payload. Model/tool recovery commonly sees stringified booleans, so `{ kind: "agentTurn", message, outputHistory: "true" }` can survive normalization and fail the strict Gateway schema instead of being coerced or stripped like adjacent recovered fields.
  - confidence: 0.78
