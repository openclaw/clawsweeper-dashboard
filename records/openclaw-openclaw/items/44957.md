## Summary

This PR should stay open: it adds a new opt-in cron memory contract that is not present on current main, and the latest branch still needs contributor real-behavior proof plus fixes for source-reproducible review blockers before merge.

## What This Changes

The PR adds an opt-in cron `outputHistory` payload flag, stores recent isolated-agent outputs in cron state, injects them into later isolated-agent prompts, updates cron normalization/tool/protocol handling, and adds focused tests.

## Best Possible Solution

If maintainers want bounded prior-output memory for isolated cron, land a rebased implementation that filters non-substantive control replies, normalizes the new flag consistently, updates docs/protocol/tests, and includes redacted real cron-run proof.

## Reproduction Assessment

Yes for the PR review blockers: the control-token persistence path and malformed `outputHistory` normalization are source-reproducible from the PR diff and review context. Reproduction is not applicable to the underlying feature request because current docs intentionally define isolated cron as fresh-session behavior.

## Solution Assessment

No. The direction is plausible, but the current patch is not yet the best maintainable solution because it needs product approval, control-token filtering, malformed-value cleanup, docs alignment, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter control-token replies before recording history:** `src/cron/service/timer.ts:347-351`
  - body: This appends any successful `outputText` to `recentOutputs` when `outputHistory` is enabled, but isolated runs can produce silent/control replies such as `NO_REPLY` or heartbeat acknowledgements while still completing successfully. Those no-op tokens would then be injected into later prompts as prior output instead of substantive history.
  - confidence: 0.87
- **[P3] Drop malformed outputHistory values during normalization:** `src/cron/normalize.ts:105`
  - body: `outputHistory` is treated as an agent-turn hint only when it is already boolean, but malformed model/tool values such as `"true"` are left in the payload for schema validation to reject. The adjacent recovery paths are meant to tolerate flattened or imperfect model output, so this should be coerced or stripped consistently instead of turning add/update calls into invalid params.
  - confidence: 0.74
