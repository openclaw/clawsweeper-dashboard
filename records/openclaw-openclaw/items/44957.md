## Summary

Keep open: current main does not implement this cron output-history capability, and the PR is useful but not mergeable as-is because it can persist silent control tokens, leaves malformed `outputHistory` values unrecovered, lacks the required changelog entry, and is currently dirty against main.

## What This Changes

The PR adds an opt-in cron agent-turn `outputHistory` flag that stores recent outputs in cron job state and injects them into later isolated cron prompts, with protocol, tool, schema, and test updates.

## Best Possible Solution

Land a rebased version that keeps the opt-in API, records only substantive prior outputs, normalizes malformed payload values consistently with adjacent fields, preserves the external-content guard, and documents the user-facing behavior in the changelog.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `applyJobResult` records any successful non-empty `outputText`, while isolated cron can produce successful silent or heartbeat-suppressed outputs before delivery filtering.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch should filter control-token history writes, normalize invalid `outputHistory` values, add the changelog entry, and rebase onto current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Filter silent replies before saving history:** `src/cron/service/timer.ts:346-351`
  - body: The new recording gate appends any successful non-empty `outputText`, but isolated cron computes that text before silent and heartbeat-only delivery suppression. A quiet run can therefore save `NO_REPLY` or `HEARTBEAT_OK` into `recentOutputs`, causing the next prompt to see control text instead of the last substantive result.
  - confidence: 0.91
- **[P3] Drop invalid outputHistory payload values:** `src/cron/normalize.ts:105`
  - body: `coercePayload` now treats boolean `outputHistory` as an agent-turn hint, but it does not strip non-boolean values the way it strips invalid `allowUnsafeExternalContent`. Malformed model output such as `outputHistory: "true"` can survive normalization and fail schema validation instead of being ignored or recovered.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/agents/tools/cron-tool.ts:263-264`
  - body: This is a user-facing cron feature that adds a new tool/protocol/runtime/state behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to have an active changelog entry.
  - confidence: 0.86
