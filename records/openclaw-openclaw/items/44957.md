## Summary

Keep open. Current main does not implement cron output history, so this is not closeable as obsolete, but the latest PR head still has source-backed correctness issues and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds an outputHistory cron agent-turn flag that stores recent isolated-agent outputs and injects them into later isolated-agent prompts.

## Best Possible Solution

Land a rebased version that stores only substantive prior outputs, preserves robust cron tool normalization, keeps the protocol/tool/docs contract aligned, and includes redacted real setup proof.

## Reproduction Assessment

Yes for the review findings from source inspection: current cron paths can return status ok with HEARTBEAT_OK or NO_REPLY in outputText, and the PR writes any ok outputText into recentOutputs. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction is plausible, but this patch should filter non-substantive control output and normalize malformed outputHistory values before merge, then show real cron behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Filter control-token outputs before recording history:** `src/cron/service/timer.ts:344-348`
  - body: This records every successful `outputText`, but cron already treats `HEARTBEAT_OK` and `NO_REPLY` as control-only or silent replies. Those runs can return `status: "ok"` with the token still in `outputText`, so `outputHistory` would persist the token and inject it into the next prompt as if it were a substantive previous result. Please skip silent/heartbeat/control-only output or record only the normalized deliverable text.
  - confidence: 0.88
- **[P3] Drop invalid outputHistory values during normalization:** `src/cron/normalize.ts:102`
  - body: Adding `outputHistory` as an agent-turn hint means malformed model output like `payload: { kind: "agentTurn", message: "...", outputHistory: "true" }` keeps the string through normalization and then fails schema validation. The adjacent `allowUnsafeExternalContent` path deletes non-boolean values before validation; `outputHistory` should get the same cleanup or coercion so this optional recovery field does not turn otherwise usable cron calls into invalid params.
  - confidence: 0.74
