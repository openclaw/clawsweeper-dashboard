## Summary

Keep open: the requested cron output-history feature is not present on current main, but this PR is not merge-ready because it lacks real behavior proof, conflicts with current main, and still has source-backed prompt-state correctness blockers.

## What This Changes

The PR adds an opt-in cron `outputHistory` payload flag, persists recent isolated-run outputs in cron job state, injects them into later isolated prompts, and updates cron tool/protocol tests.

## Best Possible Solution

Land a rebased, maintainer-approved opt-in cron memory mode that filters non-substantive outputs, normalizes the new flag consistently, and includes redacted real cron-run proof plus focused regression coverage.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source inspection shows successful `outputText` is persisted without filtering control-only replies, and non-boolean `outputHistory` is not normalized before strict schema validation. The underlying request is a new opt-in capability, not a current-main bug reproduction.

## Solution Assessment

Unclear as proposed: a per-job opt-in history flag is plausible, but the branch must be rebased and the persisted-memory contract needs maintainer approval. The safer implementation filters non-substantive output before persistence and treats the new flag like adjacent boolean fields in normalization.
