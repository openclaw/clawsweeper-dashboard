## Summary

Keep open. Current main still does not have the proposed cron output-history contract, but this external feature PR should not merge as-is because it lacks real behavior proof and still has source-backed correctness issues around control-token recording and malformed payload normalization.

## What This Changes

The branch adds an `outputHistory` field and `recentOutputs` cron job state so isolated cron agent turns can record recent outputs and inject them into future prompts, with schema, tool, service, and test updates.

## Best Possible Solution

Land a rebased version that stores only substantive prior outputs, recovers common malformed `outputHistory` values consistently with adjacent agent-turn fields, preserves current cron/protocol contracts, and includes redacted real setup proof.

## Reproduction Assessment

Yes for the review findings from source inspection: the PR records any successful `outputText`, while cron can return ok with `NO_REPLY` or heartbeat-style control output. I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No. The feature direction is plausible, but this patch should filter non-substantive control output, normalize malformed `outputHistory` values, and show real cron behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter control-token replies before recording history:** `src/cron/service/timer.ts:347-351`
  - body: This appends any successful `outputText` to `recentOutputs`, but cron can return `status: "ok"` with silent/control replies like `NO_REPLY` or heartbeat acknowledgements still in `outputText`. Those tokens would be persisted and injected into the next prompt as if they were substantive previous output.
  - confidence: 0.88

  - confidence: 0.74
