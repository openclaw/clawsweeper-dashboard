## Summary

Keep open: this is a useful npm plugin compatibility fallback with sufficient Crabbox proof, but the latest head still accepts the first older candidate after only host/API checks, so a version with missing or invalid plugin entry metadata can be selected and then fail instead of scanning to an older valid release.

## What This Changes

The PR changes npm plugin install/update resolution so bare specs and `@latest` can scan older stable npm package versions compatible with the current OpenClaw plugin API and minimum host version, with docs and tests.

## Best Possible Solution

Merge after the resolver skips candidates that fail plugin entry metadata validation, regression coverage proves latest-incompatible/next-invalid/older-valid resolution, and the branch is refreshed cleanly against current main.

## Reproduction Assessment

Yes, from source inspection: use an npm registry history where `latest` is incompatible, the next older stable package lacks valid `openclaw.extensions`, and an even older stable package is valid and compatible. The PR currently returns the first older candidate after host/API compatibility checks, before later package-entry validation would reject it.

## Solution Assessment

No as-is; the fallback direction is maintainable, but it should use or mirror the plugin package entry validation needed after install before accepting a candidate. The safer repair is to skip invalid or non-plugin candidates and keep scanning older stable versions.
