## Summary

Keep open: the npm compatibility fallback is useful and has real E2E proof, but the candidate scan currently treats any older version without host/API compatibility errors as compatible. That can stop on a non-plugin or malformed older package version and fail the install instead of continuing to an older valid plugin release.

## What This Changes

The PR changes npm plugin install/update resolution so bare specs and `@latest` can fall back to the newest stable package compatible with the current OpenClaw host/plugin API, with docs and npm install tests.

## Best Possible Solution

Keep the new fallback behavior, but require fallback candidates to have valid OpenClaw plugin package entry metadata and continue scanning past invalid or non-plugin versions, with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: latest is host/API-incompatible, the next older stable version lacks valid `openclaw.extensions`, and an even older version is valid; the resolver returns the middle candidate and the install fails later.

## Solution Assessment

No, not as-is. The fallback belongs in the installer, but the candidate predicate should include plugin package shape before treating a version as the newest compatible release.
