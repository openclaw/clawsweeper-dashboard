## Summary

Keep open: the PR adds a useful npm compatibility fallback, but the new resolver can still stop on an older stable package that is not a valid OpenClaw plugin package, leaving an older valid compatible version untried.

## What This Changes

The PR changes npm plugin install/update resolution so bare specs and `@latest` can fall back to the newest stable package compatible with the current OpenClaw host/plugin API, with docs and tests.

## Best Possible Solution

Merge after the fallback skips candidates that would fail plugin package entry validation and after regression coverage proves latest-incompatible, next-invalid, older-valid resolution.

## Reproduction Assessment

Yes from source inspection: use an npm registry history where `latest` is incompatible, the next older stable package lacks valid `openclaw.extensions`, and an even older stable package is valid and compatible. The PR currently returns the first older candidate after host/API compatibility checks, before the later package-entry validation would reject it.

## Solution Assessment

No as-is; the fallback direction is maintainable, but it should use or mirror the plugin package validation needed after install before accepting a candidate. The safer repair is to skip invalid/non-plugin candidates and keep scanning older stable versions.
