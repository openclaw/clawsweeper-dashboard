## Summary

Keep open. Current main still rejects existing local `<qqmedia>` files outside OpenClaw-managed media roots, and the requested auto-mirroring would broaden a deliberate QQ Bot local-file upload boundary that has already drawn a security concern in the issue discussion.

## What This Changes

The issue asks QQ Bot to copy existing local files referenced by `<qqmedia>` into `~/.openclaw/media/qqbot/` before sending them instead of rejecting paths outside QQ Bot media storage.

## Best Possible Solution

Define a security-preserving QQ Bot staging rule: either keep requiring agents and tools to place outbound files under OpenClaw media storage, or add a constrained mirror helper limited to explicitly trusted OpenClaw-owned output roots with canonicalization, symlink/traversal protection, size checks, docs, and regression tests.

## Reproduction Assessment

Yes. The issue body gives concrete steps, and current-main source inspection shows an existing workspace file outside `~/.openclaw/media` / `~/.openclaw/media/qqbot` still reaches the resolver rejection path.

## Solution Assessment

No for unrestricted auto-mirroring. The maintainable fix needs a maintainer-approved allowlist of trusted source roots or a documented staging workflow so QQ Bot does not become an arbitrary local-file upload primitive.
