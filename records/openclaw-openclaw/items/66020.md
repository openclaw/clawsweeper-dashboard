## Summary

Keep open. Current main still rejects existing `<qqmedia>` local files outside the guarded QQ Bot/OpenClaw media roots, and the requested auto-mirroring would change a deliberate local-file upload boundary that already has maintainer security concern in the discussion.

## What This Changes

The issue asks QQ Bot to copy existing local `<qqmedia>` files outside `~/.openclaw/media/qqbot/` into QQ Bot media storage before sending.

## Best Possible Solution

Define a security-preserving staging rule for QQ Bot: keep the explicit media-root requirement, or add a constrained mirror helper limited to approved OpenClaw-owned generated-output roots with canonicalization, symlink/traversal protection, size checks, docs, and regression tests.

## Reproduction Assessment

Yes. Source tracing shows `<qqmedia>` content flows into `sendMedia` and `resolveOutboundMediaPath`, where an existing local file outside the approved media roots returns `Media path must be inside QQ Bot media storage`; the current tests also assert outside files are blocked.

## Solution Assessment

No for unrestricted auto-mirroring. A maintainable fix needs an explicit allowlist or staging contract rather than copying any existing local path into upload storage.
