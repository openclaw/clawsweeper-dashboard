## Summary

Keep open. Current main still rejects existing `<qqmedia>` local files outside the guarded OpenClaw/QQ Bot media roots, and the requested auto-mirroring would change a deliberate local-file upload boundary that needs maintainer security judgment.

## What This Changes

The issue requests QQ Bot to copy existing local `<qqmedia>` paths outside `~/.openclaw/media/qqbot/` into QQ Bot media storage before sending.

## Best Possible Solution

Define a security-preserving staging rule for QQ Bot: keep the explicit media-root requirement, or add a constrained mirror helper limited to approved OpenClaw-owned output roots with canonicalization, symlink/traversal protection, size checks, docs, and regression tests.

## Reproduction Assessment

Yes. Source tracing shows a `<qqmedia>` workspace path flows into `sendMedia` and `resolveOutboundMediaPath`; an existing local file outside `~/.openclaw/media` returns `Media path must be inside QQ Bot media storage`.

## Solution Assessment

No for unrestricted auto-mirroring. A maintainable fix needs an explicit allowlist or staging contract rather than copying any existing local path into upload storage.
