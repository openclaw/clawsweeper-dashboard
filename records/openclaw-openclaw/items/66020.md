## Summary

Keep open. Current main still rejects existing `<qqmedia>` local files outside guarded media roots; the requested auto-mirroring is a real UX improvement but would loosen a deliberate local-file upload boundary.

## What This Changes

The issue asks QQ Bot to auto-copy existing local `<qqmedia>` files from workspace paths into `~/.openclaw/media/qqbot/` and send the mirrored copy instead of failing.

## Best Possible Solution

Define a security-preserving staging rule for QQ Bot: either keep requiring files under OpenClaw media storage, or add a constrained mirror helper limited to maintainer-approved OpenClaw-owned output roots with canonicalization, symlink/traversal protection, size checks, docs, and regression tests.

## Reproduction Assessment

Yes. The source trace from `<qqmedia>` parsing to `resolveOutboundMediaPath` shows an existing workspace file outside `~/.openclaw/media` reaches the rejection path without needing a live QQ account.

## Solution Assessment

No for unrestricted auto-mirroring. A safer fix needs an explicit allowlist or staging contract rather than copying any existing local path into QQ Bot upload storage.
