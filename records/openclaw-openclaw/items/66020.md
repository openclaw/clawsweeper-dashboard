## Summary

Keep open. Current main still rejects existing `<qqmedia>` local files outside the guarded OpenClaw/QQ Bot media roots, and the requested auto-mirroring would change a deliberate local-file upload boundary that needs maintainer security judgment.

## What This Changes

Request QQ Bot outbound `<qqmedia>` delivery to copy existing local files outside `~/.openclaw/media/qqbot` into that media directory before sending.

## Best Possible Solution

Define a security-preserving staging rule for QQ Bot: keep the current media-root requirement, or add constrained mirroring only from approved OpenClaw-owned output roots with canonicalization, symlink/race protection, size checks, docs, and regression tests.

## Reproduction Assessment

Yes. Source tracing shows a `<qqmedia>` workspace path flows into `sendMedia` and `resolveOutboundMediaPath`; an existing local file outside `~/.openclaw/media` returns the reported media-storage error, and tests assert outside files are blocked.

## Solution Assessment

No for unrestricted auto-mirroring. A maintainable fix needs an explicit allowlist or staging contract rather than copying any readable host path into QQ Bot upload storage.
