## Summary

Keep open. Current main still rejects existing `<qqmedia>` local files outside the guarded OpenClaw media roots, and the requested auto-mirroring is valid UX work but changes a deliberate QQ Bot upload security boundary already called out in discussion.

## What This Changes

Request QQ Bot to copy existing local `<qqmedia>` files outside its media storage into `~/.openclaw/media/qqbot/` before sending.

## Best Possible Solution

Define a constrained QQ Bot staging contract that preserves the upload boundary: allow only approved OpenClaw-owned output roots or explicit user-approved files, canonicalize paths, block symlink/traversal escapes, enforce size/type checks, and update docs plus regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an existing `~/.openclaw/workspace/projects/...` file passed through `<qqmedia>` reaches `resolveOutboundMediaPath`, fails `resolveQQBotPayloadLocalFilePath`, and returns the reported storage-root error.

## Solution Assessment

No for unrestricted auto-mirroring. The maintainable path is a narrowly allowlisted staging helper or explicit approval flow, not copying any readable local path into QQ Bot upload storage.
