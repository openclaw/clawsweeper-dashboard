## Summary

Keep open: the OAuth-only restoration is useful and well proven, but the patch still changes mixed OAuth-plus-API-key OpenAI audio setups from API-key transcription to OAuth transcription unless maintainers explicitly accept that compatibility break.

## What This Changes

The PR adds `openai-audio-transcriptions` to the OpenAI bearer-auth allow-list and updates model-auth and media-understanding tests for OAuth-backed audio transcription.

## Best Possible Solution

Preserve shipped API-key precedence for mixed-auth setups while allowing OAuth for explicit OAuth-profile or OAuth-only OpenAI batch STT if maintainers accept that contract.

## Reproduction Assessment

Yes. Current-main source inspection shows OpenAI audio passes `openai-audio-transcriptions` into `resolveApiKeyForProvider`, where OAuth profiles are rejected for that model API before transcription runs; the linked issue provides the matching failed configuration and error.

## Solution Assessment

No as-is. The shared auth predicate is a plausible owner boundary, but this diff is too broad because it changes shipped mixed-auth API-key precedence instead of limiting OAuth restoration to explicit or OAuth-only audio cases.
