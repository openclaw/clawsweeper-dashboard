## Summary

Keep open. Current main still persists a single Telegram accepted-update watermark before handler completion and restores that same watermark after restart, so the confirmed restart/reload durability gap remains. Open PR #73993 specifically uses closing language for this issue, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a Telegram polling fix for direct-chat messages being silently skipped when update offset handling advances past in-flight messages.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local durability fix that separates fetched/accepted offset durability from completed-handler safety and includes restart regression coverage for an interrupted two-update polling batch.

## Reproduction Assessment

Yes for the restart durability race by source inspection: current main accepts and persists update IDs before handler completion, restores the same accepted watermark on restart, and grammY runner advances its polling offset after successful fetch batches. The no-crash variant remains unclear without logs tying it to a restart/reload or another narrowed path.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the Telegram plugin polling/offset state path, with completed-safe durability and focused restart regression tests. Closing this issue before the linked implementation merges would be premature.
