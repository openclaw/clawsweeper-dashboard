## Summary

Keep open. Current main largely supersedes the original Telegram offset-advance race with default isolated polling ingress and durable spool persistence, but this issue is still paired with an open closing PR and the new default path has a separate open initialization bug that can still prevent Telegram messages from being processed.

## What This Changes

The issue reports Telegram direct-chat messages being silently lost when polling offsets advance ahead of message processing during overlapping work or restart conditions.

## Best Possible Solution

Resolve the open Telegram PR path by closing or superseding https://github.com/openclaw/openclaw/pull/73993, land or otherwise address https://github.com/openclaw/openclaw/pull/81975, then close this issue against the merged isolated-ingress durability implementation.

## Reproduction Assessment

Unclear for the original offset race on current main: source inspection shows the default polling path now durably spools fetched updates before advancing its stored offset, but the new isolated-ingress path has a separate source-proven initialization failure tracked in https://github.com/openclaw/openclaw/issues/81973.

## Solution Assessment

Yes in direction, but not complete as a cleanup close. The durable isolated polling spool is the right maintainable fix for the offset race, while the current default path still needs the open bot initialization fix before this thread is safely closed.
