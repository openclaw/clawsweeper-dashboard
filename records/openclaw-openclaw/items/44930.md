## Summary

Keep open. Current main still has a single Telegram polling offset store and the linked fix PR is open and unmerged, while source inspection plus the grammY runner contract support the restart/reload offset race described in the discussion.

## What This Changes

Fix Telegram polling so messages received while another update or job is in progress are not silently skipped by offset advancement.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73993 with a Telegram plugin-local accepted/completed offset split, fetched-offset durability barrier, migration handling, and restart regression coverage.

## Reproduction Assessment

Yes for the restart/reload durability race by source inspection: current main still has one persisted offset while grammY can advance its polling offset after a successful fetch before handlers complete. The normal-operation no-crash variant remains unclear without logs tying it to restart, reload, dedupe, or media-group behavior.

## Solution Assessment

Yes in direction: the fix belongs in Telegram polling and offset-state durability, as the linked PR attempts. Closing this issue before that implementation merges would be premature.
