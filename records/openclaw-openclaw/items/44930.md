## Summary

Keep open. The current main branch has improved local Telegram watermarking, but it still lacks the fetched-offset durability barrier and accepted/completed restart state needed for the confirmed grammY polling offset race; the open closing PR should be reviewed, repaired, landed, or replaced before this issue closes.

## What This Changes

Report a Telegram polling bug where messages can be silently dropped when getUpdates offset confirmation gets ahead of in-flight message handling.

## Best Possible Solution

Land a clean Telegram plugin-local durability fix that records fetched polling offsets before grammY can confirm newer offsets, restarts from the completed-handler watermark, and includes focused restart regression coverage.

## Reproduction Assessment

Yes for the restart/reload durability race by source inspection: grammY advances its internal polling offset after a successful batch while OpenClaw has no durable fetched/completed offset split on current main. The normal-operation no-crash variant is still unclear without logs tying it to restart, reload, media-group coalescing, or dedupe behavior.

## Solution Assessment

Yes in direction: the maintainable fix belongs in the Telegram polling and offset-state path, as proposed by the linked PR or a clean replacement. Closing this issue before that implementation merges would be premature.
