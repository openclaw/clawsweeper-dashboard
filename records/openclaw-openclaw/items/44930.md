## Summary

Keep open. Current main still has the Telegram restart/reload durability gap, and an open linked implementation PR already uses closing syntax for this issue, so closing before that PR or an equivalent fix lands would be premature.

## What This Changes

Telegram direct-chat messages can be silently lost when polling offset durability advances past an in-flight update.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73993 or an equivalent Telegram plugin-local fix that separates fetched/accepted offset durability from completed-handler safety and includes restart regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the restart/reload durability race by source inspection: grammY can advance polling offset after fetching a batch while OpenClaw still has handler work in flight, and current main restores a single offset on restart.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: the fix belongs in the Telegram plugin polling and offset state path, with completed-safe durability and focused restart regression tests.
