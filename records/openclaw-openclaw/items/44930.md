## Summary

Keep open. Current main still has the Telegram polling durability gap: the plugin persists and restores a single accepted update watermark, while grammY runner's concurrent polling model can confirm offsets before OpenClaw middleware finishes handling the update. The issue is also paired with open, unmerged PR #73993 using closing language, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to prevent Telegram long-polling messages from being silently lost when update offset handling races with concurrent message processing.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local durability fix that separates accepted/fetched offsets from completed-safe state, reconciles in-flight updates across restart, preserves duplicate/replay guards, and adds regression coverage for an interrupted two-update polling batch.

## Reproduction Assessment

Yes for the restart durability race: current source and tests show an update can be accepted and persisted before its handler completes, then rejected from the restored accepted watermark after restart. The reporter's no-crash variant remains unclear without runtime logs or a narrower failing sequence.

## Solution Assessment

Yes, the best direction is a narrow Telegram polling durability fix in the plugin, not a core channel rewrite. Closing this issue before the linked implementation lands would be premature.
