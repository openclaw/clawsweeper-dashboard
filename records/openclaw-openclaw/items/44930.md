## Summary

Current main and the latest stable release use isolated spool-first Telegram polling, durably recording each fetched update before advancing the polling offset; this closes the reported acknowledgement-before-handler-completion loss window, while the earlier linked fix PR was closed unmerged after becoming obsolete.

## What This Changes

Prevent Telegram messages from being silently lost when polling offsets advance before accepted updates finish processing.

## Best Possible Solution

Keep the shipped isolated spool-first Telegram ingress path as the canonical delivery boundary and handle any distinct spool-claim or drain failures through their narrower reports.

## Reproduction Assessment

Yes at source level for the reported offset race: the former concurrent polling architecture could acknowledge updates while handlers remained in flight. The reporter’s exact no-restart observation was not independently reproduced, but the central acknowledgement-before-durability failure mode is concrete.

## Solution Assessment

Yes. Durably storing the complete update before advancing polling state closes the loss window more directly than a completed-offset watermark alone and provides a restart recovery path.
