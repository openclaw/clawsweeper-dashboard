## Summary

Keep open. Current main still restores a single accepted Telegram update watermark as the restart skip cutoff, and open PR #73993 specifically targets this issue but is not merged.

## What This Changes

Prevent Telegram polling from silently dropping messages when offsets advance past updates that have been accepted but not fully processed.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local durability fix that separates accepted/fetched offset persistence from completed-handler replay safety and adds restart regression coverage.

## Reproduction Assessment

Yes for the restart durability race by source inspection: accepted update IDs persist before handler completion, restart restores that accepted ID as the skip watermark, and grammY advances polling offsets after successful fetch batches. The no-crash variant still needs logs to prove the exact path.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the Telegram plugin polling/offset state path, not core. Closing this issue before PR #73993 or an equivalent fix lands would be premature.
