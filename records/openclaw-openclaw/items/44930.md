## Summary

Keep open. Current main still records a Telegram update as accepted before the middleware finishes, restores that same single persisted watermark after restart, and #73993 is an open unmerged PR with closing language for this issue.

## What This Changes

The issue reports silently dropped Telegram direct-chat messages when polling offset handling can advance past in-flight updates during concurrent processing or restart.

## Best Possible Solution

Land #73993 or an equivalent Telegram plugin-local durability fix that separates accepted/fetched offset state from completed-handler safety and adds restart regression coverage for interrupted polling batches.

## Reproduction Assessment

Yes for the restart durability race by source inspection: current main persists accepted update IDs before handler completion and restores that accepted watermark as the restart skip point. The no-crash variant is still unclear without logs tying it to restart, reload, stall recovery, or another narrowed path.

## Solution Assessment

Yes in direction: the narrow maintainable fix belongs in the Telegram plugin polling/update-offset path, with durable completed-safe state and focused regression tests. Keeping this issue open until #73993 or an equivalent fix lands is the right workflow.
