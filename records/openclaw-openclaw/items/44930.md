## Summary

Keep open. Current main still persists a Telegram accepted-update watermark before handler completion and restores that same single watermark after restart, so the confirmed polling durability gap is not implemented on main. The issue is also paired with open, unmerged PR #73993 using closing language, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports Telegram polling messages being silently dropped when update offset handling advances past an in-flight message.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local durability fix that separates accepted/fetched state from completed-safe state and adds restart regression coverage for an interrupted two-update polling batch.

## Reproduction Assessment

Yes for the restart durability race: current source and tests show an update can be accepted and persisted before its handler completes, then skipped from the restored accepted watermark after restart. The no-crash variant remains unclear without logs or a narrower failing sequence.

## Solution Assessment

Yes, the best direction is a narrow Telegram polling durability fix in the plugin rather than a core channel rewrite. Closing this issue before the linked implementation lands would be premature.
