Full review comments:

- **[P1] Do not persist dashboard tokens in the app bundle:** `launcher/src/main.rs:310-313`
  - body: `save_config` writes the gateway token into `OpenClaw.app/Contents/MacOS/openclaw-launcher.conf` and `start_webui` later reuses it to build tokenized browser URLs. Current dashboard behavior keeps tokens scoped to the browser session and avoids exposing SecretRef-managed tokens in logs, clipboard history, or launch arguments, so this adds a weaker parallel auth store.
  - confidence: 0.94
- **[P1] Reuse the managed gateway lifecycle:** `launcher/src/main.rs:239-250`
  - body: The launcher starts `node openclaw.mjs gateway` as an unmanaged child process whenever port 18789 is closed. Existing macOS and onboarding flows install or attach to a managed Gateway service, so this can create overlapping lifecycles with no service metadata, restart policy, profile handling, or log integration.
  - confidence: 0.9
- **[P1] Delegate updates to `openclaw update`:** `scripts/openclaw-installer.ts:338-365`
  - body: The new installer implements its own update path that can discard local changes, pull `origin main`, install dependencies, and build the checkout. That bypasses the existing updater's channel handling, package/source install detection, doctor repair, plugin convergence, service refresh, and downgrade safeguards.
  - confidence: 0.93
- **[P1] Use the existing macOS packaging pipeline:** `build-macos-app.sh:73-105`
  - body: This script creates a parallel `OpenClaw.app` and DMG with hard-coded bundle metadata and no call into the current signing/notarization/package scripts. Current macOS packaging already owns bundle identity, Sparkle metadata, signing, TCC stability, and DMG creation, so this duplicate path would drift immediately.
  - confidence: 0.91
