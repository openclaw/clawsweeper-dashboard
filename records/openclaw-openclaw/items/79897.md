## What This Changes

The issue reports that OpenAI-compatible llama.cpp streaming saves zero token usage because the final usage-only SSE chunk is missed.

## Best Possible Solution

Preserve llama.cpp usage visibility by fixing the OpenAI-completions stream handling or timings fallback so saved transcripts record real token counts, with focused regression coverage and real llama.cpp proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main always requests streaming usage, only records yielded usage chunks, and the issue includes llama.cpp logs plus direct curl evidence showing nonzero usage outside OpenClaw.

## Solution Assessment

Is this the best way to solve the issue? The issue's requested drain-or-fallback direction is the right boundary; the currently open gating-only PR is not sufficient because it can stop requesting the usage data without restoring saved transcript counts.
