## Summary

Current main already handles the reported llama.cpp/OpenAI-compatible streaming shape: it routes supported `openai-completions` embedded streams through OpenClaw's transport, requests streaming usage, drains the async stream after `finish_reason: stop`, and records a trailing usage-only chunk.

## What This Changes

The issue requests that OpenAI-compatible streaming sessions keep reading through llama.cpp's trailing usage-only SSE chunk and persist nonzero token usage.

## Best Possible Solution

Keep the current main transport behavior; if a backend still omits usage telemetry entirely, handle that separately under the existing fallback-estimation discussion at https://github.com/openclaw/openclaw/issues/73990.

## Reproduction Assessment

No failing current-main reproduction remains. Source inspection and regression coverage show the current OpenAI-compatible stream path reads past `finish_reason: stop` and processes the trailing usage-only chunk.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it uses the existing OpenClaw transport, requests provider usage, drains the stream to completion, and persists nonzero usage without a new config knob.
