## Summary

Keep open: current main still has a source-reproducible usage-loss path for llama.cpp-style OpenAI-compatible streams whose final usage arrives after `[DONE]`. The related closed PR was not merged, and the current compat gate does not add a raw-SSE or fallback path that would preserve late usage into saved assistant/session usage.

## What This Changes

Fix OpenAI-compatible streaming so llama.cpp final usage or timings metadata is saved instead of leaving transcript and `/status` usage at zero.

## Best Possible Solution

Fix the OpenAI-completions transport or ingestion boundary so late llama.cpp usage/timings reach normalized assistant and session usage, without broadly disabling streamed usage requests for local backends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main relies on SDK-yielded chunks, while OpenAI Node 6.37.0 skips SSE after `[DONE]`; the issue discussion supplies live llama.cpp evidence that useful usage exists but is not persisted.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping this open for a transport/parser or fallback fix; a request-flag gate alone is not enough because it can prevent affected local backends from emitting the metadata needed for `/status` and compaction.
