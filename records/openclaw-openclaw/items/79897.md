## Summary

Keep open: current main still has a source-reproducible OpenAI-compatible streaming usage loss for llama.cpp-style streams whose final usage/timings arrive after `[DONE]`. The closed related PR removed the request flag instead of preserving emitted metadata, while the newer live comment shows the backend can produce usable nonzero usage when the final `done` usage survives.

## What This Changes

The issue reports zero saved usage for llama.cpp/OpenAI-compatible streaming when final usage metadata is emitted after stream finalization.

## Best Possible Solution

Keep requesting streamed usage and fix the OpenAI-completions transport or ingestion boundary so post-DONE llama.cpp usage/timings reach normalized assistant/session usage with focused regression coverage and redacted live backend proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, though not from a live current-main run: current main depends on SDK-yielded chunks, OpenAI Node 6.37.0 skips SSE after `[DONE]`, and reporters supplied llama.cpp log/curl/live mitigation evidence.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping this open for a transport/parser fix; the closed request-gating PR is not the best fix because it can stop affected local backends from emitting the metadata `/status` and compaction need.
