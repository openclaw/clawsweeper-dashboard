## Summary

Current main already handles the reported OpenAI-compatible streaming usage path: it keeps consuming the stream after a stop chunk, records trailing usage-only chunks, sends `stream_options.include_usage`, and has focused regression coverage for that shape.

## What This Changes

The issue asks OpenClaw to persist llama.cpp OpenAI-compatible streaming token usage instead of saving 0/0/0 when usage arrives after the stop chunk.

## Best Possible Solution

Keep the current transport behavior that requests usage, drains the stream until close, and falls back to normalized llama.cpp timing metadata for local OpenAI-compatible backends.
