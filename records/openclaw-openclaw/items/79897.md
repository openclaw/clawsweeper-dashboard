## Summary

Keep open: current main still has a source-plausible llama.cpp/OpenAI-compatible streaming usage gap, and an open upstream PR already references this issue with closing syntax. That PR should be reviewed or revised before this issue closes.

## What This Changes

The issue requests nonzero saved token usage for llama.cpp OpenAI-compatible streaming so `/status` and compaction can see actual context usage.

## Best Possible Solution

Fix the OpenAI-completions streaming path to preserve llama.cpp usage, either by consuming the final usage/timings data or by a robust timings fallback, then land a focused regression test and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main’s parser depends on SDK-yielded chunks while OpenAI Node v6.37.0 skips post-`[DONE]` SSE data, and the reporter supplied llama.cpp logs plus direct curl evidence; I did not run a live llama.cpp server.

## Solution Assessment

Is this the best way to solve the issue? Not yet: current main is not fixed, and the open PR removes the usage request for false-compat backends rather than preserving the usage data that `/status` and compaction need.
