## Summary

Keep open. Current main already reads trailing OpenAI-compatible usage-only chunks, but the embedded-runner/session path still has credible zero-usage handling gaps when the final assistant snapshot is all zeros; the linked fork PR is not a landed target-repo fix.

## What This Changes

The issue asks OpenClaw to preserve llama.cpp OpenAI-compatible streaming token usage when usage arrives in a final SSE chunk after the stop chunk.

## Best Possible Solution

Preserve nonzero streamed or timing-derived usage over all-zero assistant snapshots in the embedded runner/session path, while keeping provider-reported telemetry authoritative and leaving missing-telemetry estimation to the separate local-model fallback work.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main reproduction path: all-zero assistant usage is treated as a defined value in runner metadata and transcript/status extraction even when nonzero attempt or subscription usage exists.

## Solution Assessment

No, stopping only at the issue's proposed SSE-consumption fix would not be the best solution because current main already consumes empty-choices usage chunks. The narrower fix is to treat zero assistant usage as missing when nonzero attempt/subscription usage is available, and cover that path with regression tests.
