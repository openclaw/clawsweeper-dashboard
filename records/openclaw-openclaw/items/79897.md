## Summary

Keep open: current main still has a source-backed llama.cpp/OpenAI-compatible streaming usage gap, and the related upstream fix attempt was closed unmerged. The reporter's logs plus the OpenAI SDK stream contract point to a real post-`[DONE]` usage loss path that still needs a focused repair.

## What This Changes

The issue reports zero saved transcript/session usage for llama.cpp OpenAI-compatible streaming when usage or timings arrive only after stream finalization.

## Best Possible Solution

Preserve streamed usage requests for local/custom OpenAI-compatible backends and fix the OpenAI-completions stream path so llama.cpp usage or timings emitted after `[DONE]` reach normalized transcript and session usage, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main depends on SDK-yielded chunks, OpenAI Node 6.37.0 skips data after `[DONE]`, and the reporter supplied llama.cpp logs plus direct curl evidence for a trailing usage chunk.

## Solution Assessment

Is this the best way to solve the issue? Keeping this open is the right path; the best fix is not to gate off `include_usage`, but to preserve the request and capture or recover llama.cpp usage/timings reliably.
