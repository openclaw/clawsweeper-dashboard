## Summary

Keep open: current main still has a source-reproducible OpenAI-compatible streaming usage loss for llama.cpp-style streams whose final usage/timings arrive after `[DONE]`; the issue comments add useful live evidence that llama.cpp can provide nonzero usage when the final metadata survives into persistence.

## What This Changes

The issue reports llama.cpp/OpenAI-compatible streaming sessions persisting zero usage because final SSE usage/timings metadata is dropped before session accounting.

## Best Possible Solution

Keep requesting streamed usage and fix the OpenAI-completions transport or ingestion boundary so post-DONE llama.cpp usage/timings reach normalized assistant/session usage without a new config knob.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main depends on SDK-yielded chunks, OpenAI Node 6.37.0 skips SSE after `[DONE]`, and reporters supplied llama.cpp curl/live mitigation evidence; I did not run a live current-main llama.cpp endpoint.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a transport/parser fix; the closed request-gating PR is not the right direction because it can stop affected local backends from emitting the metadata OpenClaw needs.
