## Summary

Keep open: current main still has a source-reproducible usage-accounting gap for llama.cpp-style OpenAI-compatible streams, and the linked fix PR is still open with an existing review concern that it may suppress usage instead of preserving it.

## What This Changes

The issue asks OpenClaw to preserve token usage for llama.cpp OpenAI-compatible streaming responses whose usage-only SSE data arrives after finalization.

## Best Possible Solution

Fix the OpenAI-compatible streaming path to capture llama.cpp usage after finalization or recover it from timings metadata, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main, the OpenAI SDK parser, and the issue’s curl/log evidence line up, but I did not run a real llama.cpp backend.

## Solution Assessment

Is this the best way to solve the issue? No for the current linked PR as reviewed: gating off `stream_options` avoids the dropped chunk but does not restore the token usage that `/status` and compaction need.
