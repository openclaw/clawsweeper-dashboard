## Summary

Keep open: current main still has a source-reproducible path where llama.cpp-style usage emitted after `[DONE]` is lost because OpenClaw records only chunks yielded by the OpenAI SDK iterator. The related one-line request-gating PR was closed unmerged and would not preserve the backend metadata that the reporter and follow-up commenter showed can be useful.

## What This Changes

The issue asks OpenClaw to preserve or recover OpenAI-compatible llama.cpp streaming usage so saved transcript usage, `/status`, and compaction accounting stop showing zero or unknown tokens.

## Best Possible Solution

Keep requesting streamed usage and add a focused OpenClaw-owned transport or ingestion fix so post-DONE llama.cpp usage/timings reach normalized assistant/session usage with regression coverage.

## Reproduction Assessment

Yes at source level, though not from a live current-main run: current main requests streaming usage, the OpenAI SDK skips SSE after `[DONE]`, and OpenClaw only records SDK-yielded usage chunks. The reporter and follow-up commenter supplied live llama.cpp evidence that final usage exists and can produce nonzero status when preserved.

## Solution Assessment

Yes: the maintainable direction is a narrow transport/parser or fallback repair, not disabling `stream_options.include_usage`. The repair should preserve current local-backend usage requests and only add the missing late-metadata path.
