## Summary

Keep open: current main still has a source-reproducible path where OpenAI-compatible llama.cpp usage emitted after `[DONE]` is skipped by the OpenAI SDK iterator before OpenClaw can persist it. The related fix PR at https://github.com/openclaw/openclaw/pull/79958 is closed and unmerged, and its request-gating approach would remove the usage request that the live llama.cpp evidence says is needed.

## What This Changes

Fix OpenAI-compatible llama.cpp streaming so final usage or timings emitted after stream finalization are persisted instead of saving zero token usage.

## Best Possible Solution

Keep requesting streamed usage and repair the OpenAI-compatible streaming transport or ingestion boundary so post-`[DONE]` llama.cpp usage/timings reach normalized assistant and session usage with focused regression coverage and redacted live backend proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main depends on SDK-yielded chunks while OpenAI Node v6.37.0 skips SSE after `[DONE]`; the report and follow-up comment provide live llama.cpp evidence, but I did not run a live current-main backend in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a targeted transport or ingestion fix; the closed request-gating PR is not the best solution because it can stop affected local backends from emitting the metadata OpenClaw needs.
