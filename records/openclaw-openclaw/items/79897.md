## Summary

Keep open: current main still has a source-reproducible OpenAI-compatible streaming usage loss for llama.cpp-style streams when the final usage metadata is emitted after `[DONE]`. The useful recent discussion narrows the bug to preserving final streamed usage into session persistence, and the closed related PR would have stopped requesting the metadata instead of fixing that boundary.

## What This Changes

The issue asks OpenClaw to preserve nonzero llama.cpp OpenAI-compatible streaming usage instead of saving `0/0/0` when usage arrives in a final SSE chunk.

## Best Possible Solution

Preserve `stream_options.include_usage` and fix the OpenAI-completions transport or ingestion boundary so llama.cpp final usage/timings reach normalized assistant/session usage with regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main depends on OpenAI SDK-yielded chunks, the pinned SDK skips SSE after `[DONE]`, and reporter/commenter evidence shows llama.cpp sends usable final usage that OpenClaw fails to persist.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a transport/parser or ingestion fix; the closed request-gating PR is not the best fix because it can prevent affected backends from emitting the metadata OpenClaw needs.
