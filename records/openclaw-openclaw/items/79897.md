## Summary

Keep open: current main still routes OpenAI-compatible Chat Completions streams through the OpenAI SDK async iterable, which cannot expose SSE events after `[DONE]`, while OpenClaw only updates usage from chunks yielded by that iterable.

## What This Changes

The issue reports that llama.cpp OpenAI-compatible streaming can emit the final usage metadata after the stop chunk, leaving OpenClaw to persist zero token usage for status and compaction.

## Best Possible Solution

Fix the OpenAI-compatible stream ingestion boundary so final llama.cpp usage or timings reach normalized assistant/session usage, without adding a config option or changing provider request-policy defaults.

## Reproduction Assessment

Yes at source level: llama.cpp’s reported late usage shape is plausible, and current main relies on the OpenAI SDK iterator that skips SSE messages after `[DONE]`. I did not run a live current-main llama.cpp endpoint.

## Solution Assessment

Yes, keeping this open for a stream-consumption or fallback repair is the best path. The merged request-shaping and alias fixes are useful adjacent work, but they do not consume usage that the SDK never yields.
