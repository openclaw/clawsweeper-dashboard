## Summary

Keep open: current main still has a source-reproducible path where llama.cpp usage emitted after `[DONE]` is lost. Later embedded routing fixes help requests reach OpenClaw's transport, but the remaining issue is that OpenClaw records usage only from chunks yielded by OpenAI SDK 6.37.0, whose SSE iterator ignores messages after `[DONE]`.

## What This Changes

The issue reports that llama.cpp OpenAI-compatible streaming usage arrives after stream finalization and is persisted as zero usage in OpenClaw sessions.

## Best Possible Solution

Preserve streamed usage requests and fix the OpenAI-compatible transport or parser boundary so llama.cpp final usage/timings reach normalized assistant and session usage without breaking standard providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main requests streamed usage, but usage is only consumed from SDK-yielded chunks, and OpenAI SDK 6.37.0 skips SSE data after `[DONE]`. I did not establish a live current-main llama.cpp run.

## Solution Assessment

Is this the best way to solve the issue? Yes: keep this open for a narrow transport/parser fix that preserves post-DONE llama.cpp usage, rather than gating off `stream_options.include_usage` or adding a new config policy.
