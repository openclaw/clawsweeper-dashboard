## Summary

Keep open: current main still has a source-plausible llama.cpp/OpenAI-compatible streaming usage loss when the final usage or timings chunk is sent after `[DONE]`. The linked upstream PR is closed unmerged and proposed removing the usage request rather than preserving the usage data that `/status` and compaction need; the newer live reproduction comment strengthens the report.

## What This Changes

The issue asks OpenClaw to preserve llama.cpp/OpenAI-compatible streaming token usage when final usage or timings metadata arrives after the stop chunk instead of saving zero usage.

## Best Possible Solution

Keep requesting streamed usage and make the OpenAI-completions transport surface post-DONE llama.cpp usage or timings into normalized assistant/session usage, with focused regression coverage and redacted live backend proof before close.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live current-main no: current main depends on SDK-yielded chunks while OpenAI Node 6.37.0 skips SSE frames after `[DONE]`, and the reporters supplied llama.cpp log/curl/live mitigation evidence.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a transport/parser fix; the closed request-gating PR is not the best solution because it can prevent affected local backends from emitting the metadata needed for `/status` and compaction.
