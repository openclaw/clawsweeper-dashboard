## Summary

Keep open: current main still has a source-reproducible llama.cpp/OpenAI-compatible streaming usage loss when usage metadata arrives after `[DONE]`. The closed related PR did not land in this repository and proposed gating off the usage request instead of preserving the metadata that `/status` and compaction need.

## What This Changes

Report that llama.cpp/OpenAI-compatible streaming persists zero usage because the final usage-only SSE chunk arrives after stream finalization and is dropped before transcript/session accounting.

## Best Possible Solution

Keep requesting streamed usage and make the OpenAI-completions ingestion path carry post-DONE llama.cpp usage or timings into normalized assistant/session usage with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main depends on SDK-yielded chunks while OpenAI Node v6.37.0 skips post-DONE SSE frames, and reporters supplied llama.cpp curl/log/live mitigation evidence; I did not run a live current-main llama.cpp server.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a stream-ingestion or fallback fix; the closed request-gating PR is not the best solution because it can prevent affected local backends from emitting usage metadata.
