## Summary

Keep open: current main still has a source-plausible llama.cpp/OpenAI-compatible streaming usage gap, and an open upstream PR already references this issue with closing syntax. The issue should stay open until that PR is reviewed, revised, replaced, or merged.

## What This Changes

The issue reports that OpenAI-compatible llama.cpp streaming saves token usage as 0/0/0 because final usage metadata is not consumed after stream finalization.

## Best Possible Solution

Preserve llama.cpp token usage in the OpenAI-completions streaming path by handling the final usage/timings metadata or a robust fallback, then land focused regression coverage and redacted real backend proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main processes only SDK-yielded chunks, OpenAI Node v6.37.0 suppresses SSE after [DONE], and the reporter supplied llama.cpp log plus direct curl evidence.

## Solution Assessment

Is this the best way to solve the issue? No for the current linked PR direction: omitting the usage request can prevent custom/local backends from emitting the metadata that /status and compaction accounting need, so the fix should preserve usage and repair stream consumption or fallback handling.
