## Summary

Close as implemented on current main. The current embedded stream resolver routes PI's OpenAI-compatible default streams through OpenClaw's boundary-aware transport, which always requests streaming usage and keeps parsing usage-only chunks after a stop choice; the key fix is on main at 6eae017dd6142abe10c550b6a24d85cb652d176b but is not proven shipped in v2026.5.7.

## What This Changes

The issue reports zero saved token usage for llama.cpp OpenAI-compatible streaming because OpenClaw appears to close the stream before the trailing usage-only SSE chunk is processed.

## Best Possible Solution

Keep the current generic transport fix on main and ship it in the next release rather than adding a llama.cpp-specific workaround or config knob.

## Reproduction Assessment

No, not against current main. Source inspection and existing regression coverage show the reported stop-then-usage-only stream shape is now handled, while the historical embedded-stream bypass was fixed by 6eae017dd6142abe10c550b6a24d85cb652d176b.

## Solution Assessment

Yes. Routing OpenAI-compatible default streams through the existing boundary-aware transport and draining the stream through usage-only chunks is the narrow maintainable fix without adding provider-specific llama.cpp policy.
