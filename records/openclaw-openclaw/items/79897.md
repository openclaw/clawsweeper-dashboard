## Summary

Keep open: current main still has a source-reproducible OpenAI-compatible streaming usage loss for llama.cpp-style streams. OpenClaw requests streamed usage, but the consumer only sees OpenAI SDK-yielded chunks, and OpenAI Node v6.37.0 skips SSE messages after `[DONE]`; the related one-line gating PR was closed without merge and would not preserve the metadata the reporter needs.

## What This Changes

The issue reports that OpenAI-compatible llama.cpp streaming loses final usage metadata, causing saved transcript usage and context status to remain zero or unknown.

## Best Possible Solution

Keep requesting streamed usage and repair the OpenAI-completions transport or ingestion boundary so final llama.cpp usage/timings survive into normalized assistant/session usage with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main depends on SDK-yielded chunks, OpenAI Node v6.37.0 skips post-DONE SSE messages, and reporters supplied llama.cpp log/curl/live mitigation evidence; I did not run a live current-main llama.cpp backend.

## Solution Assessment

Is this the best way to solve the issue? Yes to keeping it open for a transport/parser fix; the closed request-gating PR is not the best fix because it can stop affected local backends from emitting the metadata `/status` and compaction need.
