## What This Changes

The PR changes task session-registry maintenance to preserve both raw lowercased and slugified cron job-id segments, with a regression test for both cron-run session key shapes.

## Best Possible Solution

Keep the both-shape preserve set for pruning, but report the running cron job count from the filtered running jobs rather than from the expanded preserve-key set.

## Reproduction Assessment

Yes. Source inspection shows current main lowercases running job ids while main-session cron-run keys slugify the job segment, so a running non-slug job can have an old slugged run row that current maintenance prunes; I did not execute the harness in this read-only review.

## Solution Assessment

Yes for the central prune fix: preserving both current producer shapes at the task-maintenance boundary is the narrowest safe direction. The remaining improvement is to decouple the output count from the expanded preservation identifiers.
