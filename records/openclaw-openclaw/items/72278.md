## Summary

Keep open. Current main still has the reported product/API gap: SecretRefs and provider config are limited to env/file/exec, and batching stops at exact SecretRef/provider aliases, while the public docs still show external vault examples that model each secret as a separate exec provider.

## What This Changes

The issue requests a first-class external-secret SecretRef model that batches and memoizes backend lookups across shared vault/account contexts instead of treating each external secret as a separate exec provider.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory only for that pass, and docs/configure/apply guide users toward that pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two logical external vault secrets as separate exec provider aliases with id "value" and the resolver will group them separately by source/provider, even if the backend account/container/item lookup work is shared.

## Solution Assessment

Unclear as a concrete implementation. The issue’s direction is sound, but the best fix needs maintainer product and security design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
