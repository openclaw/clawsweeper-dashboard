## Summary

Keep open. Current main still has the reported product/API gap: SecretRefs support only env/file/exec providers, batching is bounded by exact source/provider aliases, and the public 1Password and Vault examples still model each external secret as its own exec command with `id: "value"`. The related reports are useful symptom evidence, but they do not supersede this generic external-backend design item.

## What This Changes

Requests a first-class, backend-generic SecretRef external secret provider model that batches and memoizes vault-backed lookups instead of configuring one exec command per secret.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized only for that pass, and docs/configure/apply guide users toward the batchable pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: source/schema/configure/docs expose only env/file/exec providers, resolver batching groups by exact source/provider alias, and the external-vault docs still show per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The generic direction is sound, but the best fix needs maintainer product and security design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
