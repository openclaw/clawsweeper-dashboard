## Summary

Keep open. Current main still exposes only env/file/exec SecretRef providers, batches only within one source/provider alias, and still documents per-secret external CLI examples, so the generic external-vault batching gap remains a core SecretRef design item rather than a cleanup close.

## What This Changes

Request a first-class generic SecretRef external-vault provider pattern that batches and pass-memoizes backend lookups instead of modeling each external secret as a separate exec provider command.

## Best Possible Solution

Keep this as core SecretRef design work: define a generic batchable external-backend provider pattern with pass-scoped memoization, configure/apply/audit/docs support, regression coverage, and memory-only secret handling guardrails.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure two external vault secrets as separate exec provider aliases with id "value", and resolveSecretRefValues groups them separately by source/provider so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer product and secops design; a 1Password-only workaround or plugin punt would not solve the generic supported SecretRef integration shape.
