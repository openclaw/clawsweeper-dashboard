## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the public contract and provider schema expose only env/file/exec, batching stops at exact source/provider groups, and the official docs still show 1Password/Vault as one exec provider command per concrete secret value; related #56217 is narrower crash-loop/rate-limit work and does not supersede this broader provider-contract issue.

## What This Changes

The issue requests a first-class, generic SecretRef external-backend pattern that batches and pass-memoizes lookup work across vault-backed providers instead of configuring one exec provider per secret.

## Best Possible Solution

Keep this as a core SecretRef design item: add a supported external-backend pattern where one provider represents a backend/account/context, ids carry backend paths, startup resolution batches per backend, repeated lookup work is memoized in memory for one pass only, and docs/configure/apply guide users to that shape.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id "value" and the resolver groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue’s direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
