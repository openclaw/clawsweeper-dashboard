## Summary

Keep open. Current main still has the reported SecretRef product/API gap: batching is limited to exact refs within one configured env/file/exec provider alias, while the supported docs and configure flow still allow external vault secrets to be modeled as separate exec providers with `id: "value"`, which prevents cross-secret backend memoization.

## What This Changes

The issue requests a first-class SecretRef integration model that batches and memoizes external-vault lookups generically across backend providers.

## Best Possible Solution

Keep this as a core SecretRef design item: one provider should represent a backend/account/context, ids should carry backend paths, startup resolution should batch per backend, and pass-scoped in-memory memoization plus docs/configure/apply coverage should make the efficient pattern the default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id "value" and the resolver groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, provider/plugin boundaries, configure/apply behavior, docs, and secret-handling guarantees.
