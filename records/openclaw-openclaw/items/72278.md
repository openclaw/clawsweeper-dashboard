## Summary

Keep open: current main and v2026.5.7 still expose only env/file/exec SecretRefs, batching is scoped to one source/provider alias, and the shipped docs still show per-secret 1Password and Vault CLI provider examples that bypass cross-secret backend memoization.

## What This Changes

The issue requests a generic SecretRef external secret-provider model that batches and memoizes backend lookups across vault-style providers.

## Best Possible Solution

Keep this open as a core SecretRef design item: define a supported batchable external-backend pattern with pass-scoped in-memory memoization, then update configure/apply/audit flows, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level: configure two external vault secrets as separate exec provider aliases with id "value" and current resolveSecretRefValues groups them separately by source/provider, so backend lookup work cannot be shared across aliases. I did not run live external vault traffic.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but maintainers need to choose the provider/config/API shape; a 1Password-only workaround or ClawHub-only resolver would not satisfy the requested generic supported pattern.
