## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the contract and provider schema only expose env/file/exec, batching stops at exact source/provider/id grouping, and docs still show external vault CLI examples as one exec provider per concrete secret value.

## What This Changes

The issue requests a first-class external SecretRef provider pattern that batches refs and memoizes shared vault/container/item lookups across backend providers.

## Best Possible Solution

Keep this as a core SecretRef design item: add a supported external-backend pattern where one provider represents a backend/account/context, ids carry backend paths, startup resolution batches per backend, repeated lookup work is memoized in memory for one pass only, and docs/configure/apply steer users to that shape.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id "value" and the resolver groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue’s direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
