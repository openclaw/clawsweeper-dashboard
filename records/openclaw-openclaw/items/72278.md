## Summary

Keep open. Current main still has the SecretRef product/API gap described here: core supports only env/file/exec providers, batching is limited to exact refs inside a single source/provider alias, and the official external CLI examples still model each secret as its own exec provider with `id: "value"`.

## What This Changes

Request first-class external SecretRef provider integration that batches and pass-scoped memoizes lookup work across shared vault/account backends instead of per-secret exec commands.

## Best Possible Solution

Keep this as a core SecretRef design item: add a supported backend/account-oriented external provider pattern, batch ids through that shared context, memoize intermediate backend lookups only for one resolution pass, and update configure/docs/tests so the efficient pattern is the default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with `id: "value"`; resolveSecretRefValues will place them in separate source/provider groups, so shared backend container/item lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the maintainable fix needs owner design across the SecretRef contract, exec compatibility, provider/plugin boundaries, CLI flows, docs, and secret-handling guarantees.
