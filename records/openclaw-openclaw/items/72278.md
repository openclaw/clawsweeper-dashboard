## Summary

Keep open: current main still exposes only env/file/exec SecretRef providers, groups resolution at the configured source/provider alias boundary, and still documents external CLI examples that can model each secret as its own exec provider. The adjacent pluggable-source work at https://github.com/openclaw/openclaw/pull/72548 remains open/conflicting and is not yet an implementation on main.

## What This Changes

Request a first-class SecretRef external-backend pattern that batches refs per backend/provider and memoizes shared vault/entity lookups during one startup resolution pass.

## Best Possible Solution

Define a supported batchable external-backend SecretRef path with one provider per backend/account, backend-location ids, pass-scoped memory-only memoization, aligned configure/apply/audit/docs behavior, regression tests, and secops review.

## Reproduction Assessment

Yes, source-level: configure two external vault secrets as separate exec provider aliases with id `value`; current `resolveSecretRefValues` groups them separately by source/provider alias, so backend lookup work cannot be shared across aliases. I did not run live 1Password, Vault, or cloud-secret traffic.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is reasonable, but maintainers need to choose the provider/config/API shape and security policy before code changes; a one-backend workaround or ClawHub-only answer would not satisfy this generic supported-pattern request.
