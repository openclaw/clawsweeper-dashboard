- `impact:auth-provider`: The affected path is SecretRef-backed credential/provider resolution for external secret providers such as 1Password, Vault, and cloud secret managers.

## Summary

Keep open: current main and v2026.5.18 still expose only env/file/exec SecretRef providers, group resolution at the configured source/provider alias boundary, and document external CLI examples that can model one provider per secret. Related provider-seam work is still open, so there is no merged implementation to close this against.

## What This Changes

The issue requests a first-class, generic SecretRef integration model for external vault providers that batches and memoizes backend lookups across all configured vault backends.

## Best Possible Solution

Define a supported batchable external-backend SecretRef model with one provider per backend/account, backend-location ids, pass-scoped memory-only caching, aligned configure/apply/audit/docs behavior, and regression coverage.

## Reproduction Assessment

Yes, source-level: configure two external vault secrets as separate exec provider aliases with id `value`; current main groups by source/provider alias and invokes each alias separately, so backend lookup work cannot be shared across aliases. I did not run live 1Password, Vault, or cloud-secret traffic.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is reasonable, but maintainers need to choose the provider/config/API shape and security policy before a safe fix can be built.
