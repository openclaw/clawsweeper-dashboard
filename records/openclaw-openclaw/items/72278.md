## Summary

Keep open. Current main still has the reported core SecretRef integration gap: the public contract exposes only env/file/exec providers, resolver batching is bounded by exact source/provider groups, and the official external-vault examples still model 1Password and Vault as per-secret exec commands. The related reports provide useful symptom evidence but do not supersede this broader design item.

## What This Changes

Request to add a first-class, generic external SecretRef provider pattern that batches and pass-memoizes vault/backend lookups instead of forcing each external secret into its own exec provider command.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The end state should be a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized only for that resolution pass, and docs/configure/apply guide users toward that pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: source/schema/configure/docs expose only env/file/exec providers, resolver batching groups by exact source/provider alias, and the external-vault docs still show per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is appropriate for a core SecretRef API problem, but maintainers still need to choose the provider contract, backwards-compatible config/protocol shape, and secret-cache boundaries.
