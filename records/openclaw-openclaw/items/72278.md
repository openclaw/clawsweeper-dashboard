## Summary

Keep open. Current main still has the reported SecretRef product/API gap: supported provider shapes are only env/file/exec, resolver batching is limited to exact refs within a source/provider alias, and docs still show external CLI examples as one secret per exec provider with id `value`.

## What This Changes

Request a first-class SecretRef pattern that batches and memoizes external vault lookups generically instead of modeling each external secret as a separate exec provider command.

## Best Possible Solution

Design a core SecretRef integration contract that preserves backend/account context, batches refs per external backend with pass-scoped in-memory memoization, keeps exec opt-in and fail-closed guarantees, and updates configure/docs/regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id `value`; resolveSecretRefValues creates separate source/provider groups, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is appropriate, but maintainers should choose the design because it touches schema, configure/apply behavior, exec compatibility, docs, and secret caching guarantees.
