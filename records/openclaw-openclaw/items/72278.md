## Summary

Close as a core-scope cleanup: current main already has the generic SecretRef batching/plugin integration contract needed for external vault backends, and remaining backend-specific memoization should live in reviewed provider plugins published through ClawHub rather than new OpenClaw core adapters.

## What This Changes

The issue requests a first-class SecretRef integration model that batches and memoizes external vault lookups across backends instead of requiring one exec provider command per secret.

## Best Possible Solution

Keep the core SecretRef exec/plugin integration contract stable, and publish backend-specific memoizing resolvers such as Vault, 1Password, AWS, Azure, GCP, Bitwarden, or pass as separately reviewed ClawHub plugins.

## Reproduction Assessment

Yes, source-level for the old limitation: if users model each backend secret as a separate exec provider with `id: "value"`, the resolver groups by `source/provider` and cannot collapse those separate providers into one backend call. I did not run a live 1Password or Vault backend during this read-only review.

## Solution Assessment

No for new core vendor adapters; the narrower maintainable solution is the shipped generic exec/plugin integration seam plus ClawHub-published backend plugins that can memoize their own backend lookups during one batched resolver invocation.
