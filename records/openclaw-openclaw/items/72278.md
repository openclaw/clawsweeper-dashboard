## Summary

Keep open: current main and the latest release still expose only env/file/exec SecretRef providers, group resolution at the configured source/provider alias boundary, and document external CLI examples that can model one provider per secret. The related pluggable-source PR remains open and conflicting, so there is no merged implementation to close this against.

## What This Changes

Request a first-class batchable SecretRef pattern for external vault providers with pass-scoped memoization and updated docs/configure flows.

## Best Possible Solution

Define a supported batchable external-backend SecretRef model with one provider per backend/account, backend-location ids, pass-scoped memory-only caching, aligned configure/apply/audit/docs behavior, and regression coverage.

## Reproduction Assessment

Yes, source-level: configure two external vault secrets as separate exec provider aliases with id value; current main groups them separately by source/provider alias, so backend lookup work cannot be shared across aliases. I did not run live 1Password, Vault, or cloud-secret traffic.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is reasonable, but maintainers need to choose the provider/config/API shape and security policy; a one-backend workaround or ClawHub-only answer would not satisfy the generic supported-pattern request.
