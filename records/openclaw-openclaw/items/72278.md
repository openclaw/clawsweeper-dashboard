## Summary

Keep open: current main still exposes only env/file/exec SecretRefs, batches within one source/provider alias, and documents external CLI examples that model one secret as one exec provider. This is a feature/config/security design request rather than a narrow bug-fix cleanup item.

## What This Changes

Request a first-class, generic SecretRef path for external secret backends that can batch and memoize backend lookups across vault providers.

## Best Possible Solution

Define a supported batchable external-backend SecretRef path with one provider per backend/account, backend-location ids, pass-scoped memory-only memoization, aligned configure/apply/audit/docs behavior, regression coverage, and secops review.

## Reproduction Assessment

Yes, source-level. Configure two external vault secrets as separate exec provider aliases with id "value" and current resolveSecretRefValues groups them separately by source/provider alias, so backend lookup work cannot be shared across aliases; I did not run live external vault traffic.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is reasonable, but maintainers need to choose the provider/config/API shape and security policy before a safe code change; a one-backend workaround or ClawHub-only answer would not satisfy the generic supported-pattern request.
