## Summary

Keep open. Current main still has only env/file/exec SecretRef providers, and the resolver only batches exact refs within one source/provider alias, so the generic external-vault batching/memoization gap described here remains a real core design item.

## What This Changes

The issue asks OpenClaw to provide a generic, first-class external SecretRef provider pattern that batches and memoizes vault-backed lookups instead of relying on per-secret exec commands.

## Best Possible Solution

Keep this as core SecretRef design work: define a supported batchable external-backend pattern with pass-scoped memory-only memoization, configure/apply/docs guidance, and regression coverage while preserving current secret-handling guardrails.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure two external vault secrets as separate exec provider aliases with id "value", and resolveSecretRefValues will group them separately by source/provider so shared backend lookup work cannot be batched or memoized by core.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is sound, but the best fix needs maintainer design; a 1Password-only workaround or ClawHub-only answer would not solve the generic supported SecretRef integration shape.
