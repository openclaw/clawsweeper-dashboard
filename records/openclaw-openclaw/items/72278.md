## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the schema and resolver expose only env/file/exec providers, batching is limited to exact refs inside one source/provider alias, and the public docs still show per-secret external CLI providers that bypass backend-level batching.

## What This Changes

The issue requests a first-class SecretRef integration model that batches and memoizes external vault lookups across backend providers instead of per-secret exec commands.

## Best Possible Solution

Keep this as a core SecretRef design item: one configured provider should represent one external backend/account context, ids should carry backend secret paths, startup resolution should batch per backend with pass-scoped in-memory memoization, and docs/configure/tests should make that efficient pattern the default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with `id: "value"`; `resolveSecretRefValues` groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design across the SecretRef config schema, exec compatibility, provider/plugin boundaries, configure/apply behavior, docs, and secret-handling guarantees.
