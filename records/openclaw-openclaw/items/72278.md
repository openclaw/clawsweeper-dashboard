## Summary

Keep open. Current main still has the reported core product/API gap: SecretRefs are limited to env/file/exec, provider schema has no external-backend/account abstraction, runtime batching only groups exact refs by source/provider alias, and docs still include per-secret external CLI provider examples that bypass cross-secret backend memoization.

## What This Changes

Request a first-class generic SecretRef external-backend integration that batches and pass-scope memoizes vault lookups across providers instead of modeling each external secret as its own exec provider.

## Best Possible Solution

Keep this as a core SecretRef design item. The end state should be a supported external-backend pattern where one provider represents a backend/account/context, ids carry backend secret paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory for that pass only, and docs/configure/apply steer users toward that pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two logical external vault secrets as separate exec provider aliases with id "value" and the resolver will group and execute them separately by source/provider alias even if the backend account/container/item lookup work is shared.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
