## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the public contract is env/file/exec only, resolution batches only exact refs inside one source/provider alias, and the docs/configure flow still support per-secret external CLI providers that prevent shared external-backend memoization.

## What This Changes

Request a first-class SecretRef external-backend pattern that batches and pass-scoped memoizes vault lookups across providers instead of modeling each secret as its own command.

## Best Possible Solution

Design a core SecretRef provider contract where one configured provider represents an external backend/context, ids carry backend paths, and startup resolution batches plus pass-scoped memoizes backend/entity lookups with docs/configure/apply coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure two external vault secrets as separate exec provider aliases with id "value"; current resolution groups them by distinct source/provider keys, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, provider/plugin boundaries, configure/apply behavior, docs, and secret-handling guarantees.
