## Summary

Keep open. Current main still has the generic external SecretRef batching gap: core batches only exact refs within one source/provider alias, while the public docs still show external CLI examples that model each backend secret as its own exec provider command.

## What This Changes

Request a first-class SecretRef integration pattern for external secret backends that batches and pass-scope memoizes backend lookups across vault/provider backends.

## Best Possible Solution

Keep this as a core SecretRef design item: define a supported batchable external-backend pattern, update configure/apply/audit/docs, add regression coverage, and preserve memory-only pass-scoped caching with explicit exec execution gates.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. Configure two external vault secrets as separate exec provider aliases with id "value" and resolveSecretRefValues will group them separately by source/provider alias, so shared backend lookup work cannot be batched or memoized by core.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the narrowest maintainable fix needs maintainer product/security design rather than a 1Password-only workaround or a generic plugin punt.
