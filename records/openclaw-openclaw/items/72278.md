## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the shipped schema and resolver only support env/file/exec providers, batching is limited to exact refs within a single source/provider alias, and the docs still show per-secret external exec commands that prevent backend-level batching or memoization.

## What This Changes

The issue requests a first-class, generic SecretRef pattern for external vault-backed secrets that batches and memoizes provider lookups across shared backend contexts.

## Best Possible Solution

Keep this as a core SecretRef design item: provider config should preserve one backend/account/context, ids should carry backend paths, startup resolution should batch per backend with pass-scoped in-memory memoization, and configure/docs/tests should make that efficient pattern the default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id "value" and resolveSecretRefValues will group them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, provider/plugin boundaries, configure/apply behavior, docs, and secret-handling guarantees.
