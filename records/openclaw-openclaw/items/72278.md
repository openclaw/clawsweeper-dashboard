## Summary

Close as ClawHub/plugin scope: current main now has the generic batched exec SecretRef path plus a manifest-level plugin preset contract, VISION.md routes optional external secret backends to plugins/ClawHub, and an external Vault plugin already demonstrates that path without a missing core API.

## What This Changes

The issue asks OpenClaw core to provide a first-class, generic SecretRef external-vault integration model that batches and memoizes backend lookups across provider backends.

## Best Possible Solution

Keep OpenClaw core’s generic SecretRef exec and pluginIntegration contract stable, and publish backend-specific Vault, 1Password, AWS, Azure, GCP, or similar providers through ClawHub with per-plugin memoization and security review.

## Reproduction Assessment

Yes at source level: current main shows exact-ref/provider batching and the remaining backend container/item memoization must happen inside the provider resolver or plugin. I did not run a live 1Password Connect reproduction.

## Solution Assessment

No for a bundled core provider abstraction; the narrower maintainable solution is the shipped pluginIntegration contract plus backend-specific ClawHub providers that batch ids and memoize backend lookups in memory.
