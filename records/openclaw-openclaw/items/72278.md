## Summary

Keep open: current main still exposes only env/file/exec SecretRefs, batches at the source/provider alias boundary, and still documents per-secret external CLI examples that can bypass cross-secret backend memoization. The related pluggable-source PR remains open and conflicting, so it does not yet settle the generic external-backend batching, configure/apply/audit, docs, and security-policy design.

## What This Changes

The issue requests a first-class external SecretRef provider model that batches and memoizes backend lookups across vault providers instead of forcing per-secret exec provider commands.

## Best Possible Solution

Define a supported batchable external-backend SecretRef path with one provider per backend/account, backend-location ids, pass-scoped memory-only memoization, aligned configure/apply/audit/docs behavior, regression tests, and secops review.

## Reproduction Assessment

Yes, source-level. Configure two external vault secrets as separate exec provider aliases with id "value" and current resolveSecretRefValues groups them separately by source/provider alias, so backend lookup work cannot be shared across aliases; I did not run live external vault traffic.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is reasonable, but maintainers need to choose the provider/config/API shape and security policy before code changes; a 1Password-only workaround or ClawHub-only punt would not satisfy this generic supported-pattern request.
