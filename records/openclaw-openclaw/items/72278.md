## Summary

Keep open. Current main still has the reported product/API gap: SecretRefs are limited to env/file/exec providers, core batching only works after refs already share one configured provider alias, and the public 1Password/Vault examples still model each external secret as its own exec command with `id: "value"`.

## What This Changes

Request a first-class generic SecretRef external-backend pattern that batches and memoizes external vault/entity lookups across backends instead of relying on per-secret exec providers.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, ids carry backend secret paths, startup batches active refs per backend, repeated backend lookups are memoized only within the resolution pass, and docs/configure/apply steer users toward that pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: schema/types/configure expose only env/file/exec, resolver batching groups by exact source/provider alias, and the external-vault docs still show per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation because there is no patch or accepted contract yet. The narrowest maintainable direction is likely an additive external-backend/exec-resolver pattern built on the existing batch resolver, with explicit cache lifetime and exec-consent guarantees.
