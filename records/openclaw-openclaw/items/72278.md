## Summary

Keep open. Current main still has the reported SecretRef product/API gap: SecretRef sources remain env/file/exec only, batching stops at exact source/provider aliases, and the docs still show external CLI-backed secrets as separate exec providers with id `value`.

## What This Changes

Request to add a generic external-secret-provider SecretRef pattern that batches and memoizes vault/backend lookups across supported providers.

## Best Possible Solution

Design a core SecretRef provider model/API where one provider represents a backend/account context, ids carry backend secret paths, and pass-scoped in-memory memoization is covered by configure/apply/docs/tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure two external secrets as distinct exec provider aliases with id `value`; current resolution groups them as separate `exec:<alias>` batches, so shared backend work cannot coalesce.

## Solution Assessment

Unclear as a concrete patch. The issue direction belongs in core rather than a plugin-only workaround, but the maintainable fix needs design across schema, resolver compatibility, CLI flows, docs, and secret-handling guarantees.
