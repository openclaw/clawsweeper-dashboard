## Summary

Keep open. Current main still has only env/file/exec SecretRef providers, and batching/memoization is bounded to exact SecretRef keys plus the configured source/provider alias; the public external-vault examples still model 1Password and Vault as one exec provider command per secret with `id: "value"`. The related reports are useful symptom evidence, but #70952 is closed and #56217/#55459 track narrower 1Password/macOS failure modes rather than this generic SecretRef integration gap.

## What This Changes

The issue requests a first-class, generic external SecretRef provider pattern that batches and memoizes backend lookups across vault providers instead of requiring one exec provider command per secret.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The end state should let one configured provider represent an external backend/account/context, use SecretRef ids for backend paths, batch active refs through that backend during startup/reload, reuse pass-scoped in-memory results for repeated container/item/entity/payload work, and update configure/apply/docs/tests while preserving the current secret-handling guarantees.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the schema/types expose only env/file/exec providers, the resolver groups by exact source/provider alias, and the public 1Password/Vault examples still use per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The generic direction is sound and is more maintainable than a 1Password-specific workaround, but maintainers still need to choose the supported contract and security boundaries before this is safe to implement.
