## Summary

Keep open. Current main still has only env/file/exec SecretRef providers and batches by exact source/provider alias, while the official external CLI examples still model 1Password and Vault as one exec provider per secret with `id: "value"`; this remains a core SecretRef design gap rather than a stale or ClawHub-only request.

## What This Changes

The issue asks for a first-class SecretRef integration model that lets external vault-style providers batch and memoize backend lookups across shared backend contexts.

## Best Possible Solution

Keep this as a core SecretRef design item: one configured provider should represent a backend/account/context, refs should carry backend paths, and startup resolution should batch per backend with pass-scoped in-memory memoization plus docs/configure/apply coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with `id: "value"`; the resolver groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, provider/plugin boundaries, configure/apply behavior, docs, and secret-handling guarantees.
