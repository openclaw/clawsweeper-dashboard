## Summary

Keep open. Current main still has the reported product/API gap: SecretRef sources and provider schema are limited to env/file/exec, resolver batching is bounded by exact source/provider aliases, and the public 1Password and Vault examples still model each external secret as its own exec command with `id: "value"`. The related 1Password reports are useful symptom evidence, but they do not supersede this generic external-backend design item.

## What This Changes

The issue requests a first-class SecretRef integration model that batches and pass-scope memoizes external vault lookups across backend providers instead of requiring one exec provider command per secret value.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory only for that pass, and docs/configure/apply guide users toward the batchable pattern by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external secrets as separate exec provider aliases, as the docs show for 1Password/Vault, and `resolveSecretRefValues` will group them separately by alias even if the backend account/container/item lookup work is shared.

## Solution Assessment

Unclear as a concrete implementation. The issue’s generic direction is sound, but the best fix needs maintainer product and security design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
