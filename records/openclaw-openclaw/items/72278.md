## Summary

Keep open. Current main still has the reported SecretRef product/API gap: supported sources remain env/file/exec, batching stops at exact source/provider groups, and the docs still show one external-vault CLI command per concrete secret value; related PR #72548 is open but not merged and does not close this issue.

## What This Changes

Request a first-class external SecretRef provider pattern that batches and pass-scoped memoizes lookups across vault-backed secret providers.

## Best Possible Solution

Keep this as a core SecretRef design item: provide a supported external-backend pattern where one provider represents a backend/account/context, ids carry backend paths, startup resolution batches per backend, repeated lookup work is memoized in memory for one pass, and docs/configure/apply guide users to that shape.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two vault secrets as separate exec provider aliases with id "value" and resolveSecretRefValues groups them separately by source/provider, so shared backend lookup work cannot be batched or memoized by OpenClaw.

## Solution Assessment

Unclear as a concrete implementation. The issue’s direction is sound, but the best fix needs maintainer design across config schema, exec compatibility, plugin/provider boundaries, configure/apply behavior, docs, and secret-handling guarantees.
