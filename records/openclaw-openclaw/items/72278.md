## Summary

Keep open. Current main still leaves the generic external SecretRef batching gap: the public contract and schema only model env/file/exec providers, resolver batching stops at exact refs within one source/provider alias, and the docs still show per-secret external CLI providers for 1Password and Vault.

## What This Changes

Requests a first-class, generic SecretRef external-backend pattern that batches and memoizes vault lookups across one provider context instead of modeling each secret as a separate exec provider.

## Best Possible Solution

Keep this as a core SecretRef design item: standardize a batchable external-backend provider pattern with pass-scoped memoization, configure/docs support, and regression coverage while preserving current secret-handling guardrails.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure two external vault secrets as separate exec provider aliases with id "value", and resolveSecretRefValues will group them separately by source/provider so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The issue direction is sound, but the best fix needs maintainer design; a 1Password-only workaround or plugin punt would not solve the generic supported SecretRef integration shape.
