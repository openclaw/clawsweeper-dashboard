## Summary

Keep open. Current main still has the reported SecretRef product/API gap: the contract and schema only expose env/file/exec providers, batching is limited to exact refs grouped by source/provider, and the docs still show external vault examples as one exec provider command per concrete secret value.

## What This Changes

The issue requests a generic SecretRef integration model that lets external secret backends batch and pass-scope memoize lookups across backend vault/entity providers.

## Best Possible Solution

Keep this as a core SecretRef design item: one provider should represent a backend/account/context, ids should carry backend paths, startup resolution should batch per backend, and repeated backend lookup work should be memoized in memory for one resolution pass with no disk cache or plaintext logging.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure two external vault secrets as separate exec provider aliases with id `value`, and the resolver groups them separately by source/provider so shared backend lookup work cannot be batched or memoized.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is valid, but the best fix should be coordinated with the pluggable SecretRef provider-source work and must settle compatibility, cache lifetime, docs/configure behavior, and exec/provider security semantics first.
