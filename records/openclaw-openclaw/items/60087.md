## Summary

Keep open. Current main still routes managed `gateway start` through `service.restart()` after the login-shell exec PATH probe, and the open PR that closes this issue has not merged.

## What This Changes

The issue reports that login-shell exec PATH probing can source `~/.profile`, rerun `openclaw gateway start`, and restart the managed gateway on every exec invocation.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75545 with a narrow fix that makes managed `gateway start` an ensure-running no-op after repair checks, while preserving explicit `gateway restart`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: gateway-host exec invokes the login-shell PATH probe, `gateway start` reaches `service.restart()`, and the Linux systemd adapter performs a restart; I did not run a live systemd/profile reproduction.

## Solution Assessment

Yes as a direction. Making managed `gateway start` idempotent after repair-required checks is the narrow maintainable fix; changing exec away from login-shell PATH discovery would be broader and could break PATH behavior.
