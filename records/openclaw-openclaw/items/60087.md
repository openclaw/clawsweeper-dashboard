## Summary

Keep open. Current main still contains the reported source-level failure chain: host exec performs a login-shell PATH probe before the exec runtime marker is applied, and managed `gateway start` still delegates to the restart path for installed or loaded services.

## What This Changes

The issue asks for exec tool invocations not to restart the Gateway when a shell startup file calls `openclaw gateway start`.

## Best Possible Solution

Fix the current implementation narrowly by making managed `gateway start` a no-op when the service is already running and by making the exec login-shell PATH probe unable to trigger profile side effects before the actual command starts.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `host=gateway` exec still reaches `shell -l -c env -0`, and a profile-sourced `openclaw gateway start` still reaches the service restart path for installed or loaded services.

## Solution Assessment

Yes for a repair PR, but not for current main. The narrow maintainable fix is two-sided: idempotent managed service start plus a side-effect-safe exec PATH probe, with tests proving the probe receives the protective context before shell startup files run.
