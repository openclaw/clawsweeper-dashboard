## Label Justifications

- `P1`: Affected users can have every exec tool invocation restart the gateway and drop active WebSocket sessions.
- `impact:crash-loop`: The report is about repeated gateway SIGTERM and restart behavior triggered by exec calls.

## Summary

Keep open. Current main and v2026.5.18 still route managed `gateway start` through restart after the login-shell exec PATH probe, and the open closing PR has not merged yet.

## What This Changes

The issue asks for exec tool calls not to restart the gateway when a login shell sources a profile entry that runs `openclaw gateway start`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75545 with a narrow fix that makes managed `gateway start` an ensure-running no-op only after repair checks pass, while preserving explicit `gateway restart`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: gateway exec invokes the login-shell PATH probe, `gateway start` reaches `service.restart()`, and Linux systemd restart maps to `systemctl --user restart`; I did not run a live systemd reproduction.

## Solution Assessment

Yes. Making managed `gateway start` idempotent after repair-required checks is the narrow maintainable fix; changing exec away from login-shell PATH discovery would be broader and could break PATH behavior.
