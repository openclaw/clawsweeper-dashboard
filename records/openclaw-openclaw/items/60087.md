## Summary

Keep open. Current main and the latest release still route `gateway start` through the service restart path, exec still performs the login-shell PATH probe, and the linked closing PR remains open/conflicting rather than landed.

## What This Changes

The issue asks for exec-triggered login-shell startup files to stop restarting a healthy running gateway when they call `openclaw gateway start`.

## Best Possible Solution

Land or replace the linked PR with a narrow fix that makes managed `gateway start` an ensure-running no-op only after repair checks pass, while preserving explicit `gateway restart`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: gateway exec calls the login-shell PATH probe, `gateway start` still reaches `service.restart()`, and the Linux service adapter maps restart to `systemctl --user restart`.

## Solution Assessment

Yes. Making managed `gateway start` idempotent for healthy already-running services is the narrowest maintainable solution; changing exec away from login-shell probing would be broader and risks PATH behavior.
