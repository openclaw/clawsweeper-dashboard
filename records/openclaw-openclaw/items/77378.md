## Summary

Keep open: current main still leaves Mattermost non-DM sessions without a routable persisted delivery context, and the duplicate `channel:`/`group:` session rotation part is not fully covered by the linked open PRs.

## What This Changes

The issue reports duplicate Mattermost sessions after reload, where an older session lacks `deliveryContext.to` and later flushes queued responses as duplicate posts.

## Best Possible Solution

Add a Mattermost-owned fix and regression tests that persist routable delivery context for group/channel sessions and handle legacy `channel:` versus `group:` entries for the same Mattermost channel id without replaying stale queued responses.

## Reproduction Assessment

No, not for the full duplicate flush: I did not live-run Mattermost reload plus LLM interruption. Source inspection does reproduce the core missing route persistence path for Mattermost non-DM sessions on current main.

## Solution Assessment

Unclear as a complete fix: persisting last-route metadata for non-DM Mattermost sessions is the narrow first repair, but legacy `channel:`/`group:` duplicate cleanup should be coordinated with the existing open Mattermost delivery-context PR.
