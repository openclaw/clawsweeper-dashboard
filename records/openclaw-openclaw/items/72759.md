## Summary

Keep open for maintainer review. The current branch uses the valid `off` mode, but it changes Telegram's public default without updating the resolver test, docs, config UI hints, or changelog, while current `main` still documents and tests `partial` as the default and has separate partial-mode finalization logic for the ghost-preview class of bugs.

## What This Changes

The PR changes Telegram preview streaming's unset fallback from `partial` to `off` in `extensions/telegram/src/preview-streaming.ts`.

## Best Possible Solution

Maintain a single explicit Telegram streaming contract: either keep `partial` as the default and rely on the current durable preview-finalization path, or deliberately change the default to `off` with matching resolver tests, docs, config UI hints, and a user-facing changelog entry.

## Reproduction Assessment

Yes for the PR impact: current source and tests show unset Telegram streaming resolves to `partial`, while the PR changes that fallback to `off`. The linked ghost-message report has concrete external reproduction, but I did not run a live Telegram bot verification in this read-only pass.

## Solution Assessment

No as submitted. `off` is a valid resolver value, but changing only the fallback leaves public contract surfaces contradictory and skips the product decision of whether final-only default behavior should replace the newer partial-mode finalization fixes.
