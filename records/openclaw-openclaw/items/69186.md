## Summary

Keep open. Current main still has final chat-event handling, manual Web Push controls, and the public `agent_end` hook, but none implements the requested opt-in browser-local completion chime for Control UI turns.

## What This Changes

Request an optional completion or success sound when an agent finishes a turn in the OpenClaw Control UI.

## Best Possible Solution

Add an off-by-default Control UI setting that plays a short built-in browser-local chime when the active chat run reaches a successful final state, with docs and focused UI/storage tests.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: emit a final `chat` event for the active run and the UI clears the run state, while settings and notification code have no completion-sound path.

## Solution Assessment

Yes. The narrowest maintainable solution is a Control UI-owned local chime tied to terminal chat events; the existing `agent_end` plugin hook and Web Push controls do not satisfy the background-tab audio cue.

## Repair Work Prompt

Implement the requested completion/success notification sound for Control UI agent turns. Add an off-by-default persisted `UiSettings` preference for a browser-local built-in chime, expose it in the Control UI settings/notifications area, and trigger it only when the active chat run reaches a successful `final` terminal state. Keep the implementation UI-owned; do not change `agent_end` plugin semantics, Web Push infrastructure, backend push APIs, or channel behavior. Catch and ignore rejected audio playback promises so browser autoplay policy failures do not break chat handling. Do not implement remote custom sound URLs unless maintainers explicitly approve that privacy/security scope. Add focused tests for settings persistence, config rendering, and final-event playback behavior, update `docs/web/control-ui.md`, and add a user-facing `CHANGELOG.md` entry.
