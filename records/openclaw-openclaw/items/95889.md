## What This Changes

The PR changes forced gateway restarts to use a 5s drain budget, wait on active task/run helpers, and pass the remaining budget into server close reply-drain handling.

## Best Possible Solution

Keep the bounded forced-restart drain direction if maintainers accept the 5s forced-restart delay tradeoff, rename the stale test description, and require redacted live/log proof that pending replies drain before shutdown.

## Reproduction Assessment

Yes from source and reported logs: current main's forced restart path returns 0ms, skips active-work drain, and then `server.close()` can receive no reply-drain budget. I did not run the live gateway scenario in this read-only review.

## Solution Assessment

Mostly yes; the patch uses the existing gateway restart drain owner boundary instead of adding a new config or retry path, but the real pending-reply behavior still needs live proof before merge.
