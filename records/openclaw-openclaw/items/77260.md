## Summary

Keep open: current main still suppresses text slash-command replies in group/channel `message_tool` mode, while only native command turns get the visible-reply bypass.

## What This Changes

The issue requests visible delivery for canned slash-command/system replies in group chats when `messages.groupChat.visibleReplies` is set to `message_tool`.

## Best Possible Solution

Add a shared command/system-reply visibility bypass in the auto-reply delivery path, preserving `message_tool_only` suppression for normal model final replies and send-policy denial.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Matrix group `/new` becomes a `CommandSource: "text"` channel turn, resolves to `message_tool_only`, returns a canned command reply, and dispatch skips delivery while `suppressDelivery` is true.

## Solution Assessment

Yes. The narrow maintainable fix is shared command/system reply handling in auto-reply dispatch or visibility policy, not a Matrix-specific workaround or a new config option.

## Repair Work Prompt

Address this issue by making canned command/system replies from text slash commands visible in group/channel `message_tool` mode. On current main, Matrix-style `ChatType: "channel"` plus `CommandSource: "text"` resolves to `message_tool_only`; command handlers such as `/new` return a fixed reply, but `dispatchReplyFromConfig` skips final delivery under `suppressDelivery`. Implement this generically in the shared auto-reply delivery/visibility path, not as a Matrix-only workaround and not by adding a config option. Preserve suppression for normal model final replies, preserve `sendPolicy: deny`, and keep native command behavior visible. Add regression coverage for a text `/new` or `/status` group/channel turn with `messages.groupChat.visibleReplies: "message_tool"`, plus coverage that normal model final replies remain private. Update docs/changelog for the user-facing behavior change.
