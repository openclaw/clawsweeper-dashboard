## Summary

Keep open: current main still treats text slash commands in group/channel chats as message-tool-only source turns, so their canned command replies can be processed but not delivered visibly.

## What This Changes

The issue asks for slash command/system replies in group chats to bypass `messages.groupChat.visibleReplies: "message_tool"` suppression.

## Best Possible Solution

Add a generic command/system-reply visibility bypass in the shared auto-reply delivery path, with regression coverage proving text slash command replies are visible while normal model final replies remain message-tool-only.

Yes. Source inspection gives a high-confidence reproduction path: a Matrix group/channel `/new` sets `CommandSource: "text"`, resolves `message_tool_only`, returns a canned command reply, and then dispatch suppresses final delivery.

## Repair Work Prompt

Fix the regression where text slash command/system replies in group/channel chats are silently dropped when `messages.groupChat.visibleReplies` is `message_tool`. Keep normal model final/block/preview replies message-tool-only, keep native command behavior visible, and keep `sendPolicy: deny` suppressing delivery. Implement the fix in the shared auto-reply delivery path rather than Matrix-specific code; add regression coverage for a text `/new` or `/status` group/channel command returning a canned reply that is delivered visibly under message-tool-only mode, plus coverage that ordinary model final replies remain suppressed. Update CHANGELOG.md because this is a user-facing fix; do not add a new config option.
