## Summary

Keep open: current main and v2026.6.8 still lack a `session.maintenance` control that caps or rotates one active transcript JSONL file, while the existing disk-budget and compaction paths only address adjacent pressure cases and the remaining fix changes retention semantics.

## What This Changes

The issue requests a bounded maintenance policy for individual transcript JSONL files so long-lived sessions cannot grow without limit and stall the gateway.

## Best Possible Solution

Choose one maintainer-approved active-transcript bounding policy, then align schema, docs, cleanup behavior, locking, and regression tests around that policy without adding hot-path directory scans.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and v2026.6.8 have no `session.maintenance` field or runtime path that caps an individual active transcript JSONL file. I did not live-reproduce the exact CPU 100% stall with a production-sized transcript in this read-only review.

## Solution Assessment

Unclear. A disabled-by-default `session.maintenance` transcript cap is plausible, but the best fix depends on a maintainer decision about raw archival versus semantic compaction, retention, and hot-path overhead.
