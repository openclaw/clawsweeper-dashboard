## Summary

Keep open. Current main still has no `session.maintenance` byte or line cap for individual active transcript `.jsonl` files; the shipped `agents.defaults.compaction.maxActiveTranscriptBytes` guard is useful but compaction-scoped, opt-in, and not the requested maintenance cleanup control.

## What This Changes

Request adding session maintenance controls to cap or rotate individual transcript `.jsonl` files so long-lived sessions cannot grow without bound.

## Best Possible Solution

Define a conservative, opt-in transcript maintenance policy, or explicitly document that OpenClaw only supports the compaction-based active transcript guard and not raw `.jsonl` line/byte caps.

## Reproduction Assessment

Yes, at source level. The issue provides concrete oversized transcript examples, and current main still has no public config field or maintenance runtime path that can cap an active referenced `.jsonl` transcript.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default `transcriptRotateBytes` / `transcriptMaxLines` style feature is plausible, but it needs to be reconciled with the shipped compaction byte guard and deprecated `rotateBytes` contract.
