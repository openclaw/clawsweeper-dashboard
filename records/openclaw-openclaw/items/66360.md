## Summary

Keep open: current main and v2026.6.8 have a compaction-based active transcript byte guard, but the requested `session.maintenance` raw transcript cap or explicit compaction-only product decision remains unresolved.

## What This Changes

The issue requests bounded maintenance for transcript `.jsonl` files so long-lived session transcripts cannot grow without a size or line cap and stall the gateway.

## Best Possible Solution

Maintainers should choose the supported direction: either add a narrow, explicit raw transcript maintenance policy with schema/docs/tests, or document and close around the existing compaction-only guard.

## Reproduction Assessment

Yes at source level for the missing `session.maintenance` transcript cap: current schema/runtime have no raw per-transcript size or line threshold, while the CPU stall itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. The current compaction-based guard is a real mitigation for active transcript reopen cost, but it is not the requested `session.maintenance` raw cap, so maintainers need to choose the product direction before implementation.
