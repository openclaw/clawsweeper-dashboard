## Summary

Keep open. Current main still lacks a `session.maintenance` size or line cap for active transcript `.jsonl` files; the shipped `agents.defaults.compaction.maxActiveTranscriptBytes` guard is useful but lives under agent compaction settings, requires `truncateAfterCompaction`, and is not the requested maintenance cleanup path.

## What This Changes

The issue requests a `session.maintenance` mechanism to cap, rotate, or tail-trim individual active transcript `.jsonl` files so long-running chat sessions cannot grow without bound.

## Best Possible Solution

Add an explicit, documented transcript maintenance path for active `.jsonl` files, with disabled-by-default settings, `warn`/`enforce` behavior matching existing maintenance, dry-run visibility in `openclaw sessions cleanup`, and regression coverage for header preservation, tail retention, archive cleanup, and concurrency failure paths.

## Reproduction Assessment

Yes. The issue provides concrete oversized transcript examples, and current-main source inspection reproduces the missing control path: the maintenance schema/resolver have no transcript cap and save-time maintenance never shrinks an active referenced transcript.

## Solution Assessment

Yes, a focused disabled-by-default maintenance control is the best fix if maintainers want this behavior under `session.maintenance`. The current compaction byte guard should remain, but it is a separate semantic compaction trigger rather than a general maintenance cap.
