## Label Justifications

- `P1`: The report describes gateway CPU saturation and unresponsiveness from unbounded active transcripts, and source inspection confirms the requested cap is still missing.

## Summary

Keep open: current main and v2026.6.8 still do not provide a `session.maintenance` policy that caps or rotates an individual active transcript JSONL file. Related disk-budget cleanup and opt-in post-compaction rotation help adjacent pressure cases, but they do not implement the requested raw transcript maintenance control and the remaining fix needs a maintainer decision about live transcript retention semantics.

## What This Changes

The issue requests a source-level cap or rotation policy for individual active session transcript JSONL files so long-lived sessions cannot grow without bound.

## Best Possible Solution

Choose one maintainer-approved active-transcript bounding policy, then align schema, docs, cleanup behavior, concurrency locking, and regression tests around that policy instead of leaving operators to run external cleanup cron jobs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no `session.maintenance` field or runtime path that caps an individual active transcript JSONL file. I did not live-reproduce the reporter's exact CPU 100% stall with a 100 MB production transcript.

## Solution Assessment

Unclear. A disabled-by-default `session.maintenance` transcript cap is plausible, but maintainers first need to decide whether raw archival, semantic compaction, disk-budget cleanup, or documentation is the supported policy for active transcript growth.
