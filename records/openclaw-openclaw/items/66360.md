## Label Justifications

- `P1`: The report describes a real long-running channel-session workflow where oversized transcript files made the gateway CPU-bound and unresponsive.
- `impact:crash-loop`: The affected problem class is process availability: unbounded transcript growth can hang or stall the gateway.

## Summary

Keep open. Current main and v2026.6.6 still do not provide a raw byte or line cap for active referenced transcript `.jsonl` files; existing disk-budget cleanup and `maxActiveTranscriptBytes` compaction address adjacent pressure but do not implement the requested `session.maintenance` policy.

## What This Changes

Add bounded maintenance for active transcript `.jsonl` files so long-lived sessions can be capped or rotated instead of growing without bound.

## Best Possible Solution

Maintainers should decide and document the supported policy, then either add an opt-in raw transcript cap with schema/docs/tests or explicitly declare compaction-based shrinking as the supported answer.

## Reproduction Assessment

Yes for the missing capability: current main and v2026.6.6 have no raw `session.maintenance` byte or line cap for active referenced transcripts. I did not reproduce the reported CPU saturation live.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw cap is plausible, but the best fix must be reconciled with `maxActiveTranscriptBytes`, successor-transcript compaction, disk-budget eviction, and history recovery semantics.
