## Label Justifications

- `P1`: The report describes a real long-running session workflow where unbounded transcripts pinned gateway CPU and made the gateway unresponsive.
- `impact:crash-loop`: The affected problem class is process availability: oversized transcript files can lead to gateway hangs or unresponsiveness.

## Summary

Keep open. Current main and the latest release still lack a raw `session.maintenance` byte or line cap for referenced transcript `.jsonl` files; the active-transcript compaction guard is useful but intentionally not raw archival and does not settle the requested maintenance policy.

## What This Changes

Request a per-transcript size or line cap/rotation policy for session `.jsonl` files so long-running sessions cannot grow without bound.

## Best Possible Solution

Maintainers should either define an opt-in raw transcript maintenance policy with schema, docs, and regression tests, or explicitly make compaction-based active transcript shrinking the supported direction and document that `session.maintenance` does not cap active raw JSONL files.

## Reproduction Assessment

Yes for the missing maintenance capability: current source and the v2026.5.18 tag have no `session.maintenance` raw transcript byte or line cap and no save-time rotation path for active referenced `.jsonl` files. I did not reproduce the reported 100% CPU saturation live.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw cap is plausible, but the maintainable solution must be reconciled with `maxActiveTranscriptBytes`, compaction successor transcripts, disk-budget cleanup, and history recovery semantics.
