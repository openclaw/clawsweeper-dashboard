## Summary

Keep open. Current main still lacks a `session.maintenance` byte or line cap for active transcript `.jsonl` files; the shipped compaction byte guard is useful but explicitly compaction-scoped, not raw JSONL archival or maintenance cleanup.

## What This Changes

Request to add a session-maintenance cap or rotation policy for individual transcript `.jsonl` files so long-lived chat sessions cannot grow without bound.

## Best Possible Solution

Maintainers should either define an opt-in raw transcript maintenance policy with schema, docs, and tests, or explicitly document that OpenClaw supports compaction-based active transcript shrinking instead of `session.maintenance` raw JSONL caps.

## Reproduction Assessment

Yes, at source level for the missing maintenance cap: current main has no public `session.maintenance` transcript byte/line threshold and no maintenance path that rotates or tail-trims an active referenced `.jsonl`. I did not reproduce the reported 100% CPU saturation live.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw transcript cap is plausible, but it needs to be reconciled with the existing compaction guard and session history recovery policy.
