## Summary

Keep open: current main and v2026.6.8 still lack a raw `session.maintenance` size/line cap for active transcript `.jsonl` files. Current code has adjacent mitigations through disk-budget cleanup and opt-in compaction-based transcript rotation, but the reported raw-cap behavior remains a maintainer product/config decision.

## What This Changes

The issue requests bounded maintenance for oversized session transcript `.jsonl` files so long-lived group/topic sessions cannot grow without limit and stall Gateway.

## Best Possible Solution

Maintainers should choose between documenting compaction-only support or adding an explicit disabled-by-default raw transcript maintenance policy with schema/docs, deterministic oversized-transcript fixtures, and no repeated full-tail scans on startup or hot writes.

## Reproduction Assessment

Yes, as a source-level reproduction path: current `session.maintenance` has no raw per-transcript cap, `rotateBytes` is ignored, and the append path continues writing to the same active `.jsonl`. I did not run a live oversized-file CPU reproduction in this read-only review.

## Solution Assessment

Unclear. A disabled raw cap is a plausible fix, but current main deliberately offers semantic compaction plus disk-budget cleanup, so maintainers need to decide whether raw active transcript archiving/truncation belongs in core policy or should be documented as intentionally unsupported.
