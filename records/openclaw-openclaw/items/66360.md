## Summary

Keep open. Current main still lacks a `session.maintenance` byte or line cap for individual active transcript `.jsonl` files; the shipped compaction byte guard is useful but is opt-in, compaction-scoped, and explicitly not raw JSONL archival or maintenance cleanup.

## What This Changes

The issue requests a session maintenance cap or rotation mechanism for individual transcript `.jsonl` files so long-lived sessions cannot grow without bound.

## Best Possible Solution

Maintainers should either define a conservative opt-in transcript maintenance policy with schema/docs/tests, or explicitly document that OpenClaw supports only compaction-based active transcript shrinking and not raw `.jsonl` caps.

## Reproduction Assessment

Yes, at source level. Current main has no public `session.maintenance` transcript byte/line cap and no maintenance path that rotates or tail-trims an active referenced `.jsonl`; I did not live-reproduce the CPU saturation.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw transcript cap is plausible, but it needs to be reconciled with the existing compaction guard and the data-retention/recovery policy.
