## Summary

Keep open: current main still lacks a `session.maintenance` policy that caps or rotates one raw active transcript JSONL file; the existing active byte guard is a compaction-based mitigation and the remaining raw-retention policy needs maintainer direction.

## What This Changes

Requests a session maintenance policy to cap, rotate, or otherwise bound active transcript JSONL files so long-running sessions cannot grow without limit and saturate gateway CPU.

## Best Possible Solution

Choose whether OpenClaw supports compaction-only behavior or adds a disabled-by-default raw `session.maintenance` transcript cap, then align schema, docs, runtime boundaries, and oversized-transcript regressions to that decision.

## Reproduction Assessment

Yes, source-reproducible: current `session.maintenance` schema and resolver lack a per-transcript cap while docs say `rotateBytes` is deprecated and ignored. I did not recreate the exact CPU 100% stall with a large transcript.

## Solution Assessment

Unclear: `agents.defaults.compaction.maxActiveTranscriptBytes` is a plausible mitigation, but it is not the same raw `session.maintenance` cap requested here. The best fix depends on whether maintainers want raw archival/line caps or compaction-only behavior.
