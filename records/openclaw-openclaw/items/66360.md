## Label Justifications

- `P1`: The report describes gateway CPU saturation and unresponsiveness in a real long-running session workflow, which can break agent/channel operation for affected users.
- `impact:crash-loop`: The observed failure mode is gateway availability collapse requiring manual file intervention and restart.
- `impact:session-state`: The affected data is raw session transcript storage and session maintenance policy for long-lived conversation history.

## Summary

Keep open: current main has shipped compaction-based active transcript mitigations, but `session.maintenance` still has no individual raw `.jsonl` transcript cap/rotation policy, and the remaining fix needs maintainer product direction because it can rewrite or archive persistent session history.

## What This Changes

The issue requests a bounded maintenance policy for raw transcript `.jsonl` files so long-lived sessions cannot grow without limit and stall the gateway.

## Best Possible Solution

Keep this issue open while maintainers choose the transcript lifecycle direction: either document compaction-only as intentional, or add a conservative core-owned transcript maintenance policy with deterministic oversized-file fixtures and startup/cleanup regression coverage.

## Reproduction Assessment

Yes at source level for the missing maintenance capability: current `session.maintenance` has no individual transcript byte/line cap and `rotateBytes` is deprecated/ignored. I did not reproduce the exact 100% CPU stall against a live oversized transcript.

## Solution Assessment

Unclear. The prior raw `transcriptRotateBytes`/`transcriptMaxLines` PR was plausible, but current main moved toward compaction-based active transcript shrinking, so maintainers need to decide whether raw archival belongs in `session.maintenance` or in the broader SQLite/transcript lifecycle plan.
