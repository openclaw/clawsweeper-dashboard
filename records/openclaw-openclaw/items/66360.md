## Summary

Keep open: current main and the latest release still lack a `session.maintenance` policy that caps a referenced active transcript JSONL file. Existing disk-budget cleanup and compaction paths are adjacent mitigations, but they do not provide the raw active-transcript bound requested here.

## What This Changes

Add or define session maintenance controls that cap or rotate active transcript `.jsonl` files so long-running sessions cannot grow until the gateway becomes unresponsive.

## Best Possible Solution

Maintainers should choose one active-transcript bounding policy, then align schema, docs, locking/concurrency behavior, and oversized-transcript regression coverage around that policy.

## Reproduction Assessment

Yes, at source level: current main has no `session.maintenance` field or runtime path that caps one active transcript JSONL file. I did not live-reproduce the exact CPU 100% stall.

## Solution Assessment

Unclear: a disabled-by-default raw transcript cap is plausible, but maintainers need to choose raw archival, line caps, semantic compaction, or docs clarifying that raw active transcripts are intentionally uncapped.
