## Summary

Keep open: current main and the latest stable release still lack a `session.maintenance` control that caps or rotates an individual active transcript JSONL file, and the existing disk-budget plus compaction paths are adjacent mitigations rather than the requested raw transcript bound.

## What This Changes

Request a native `session.maintenance` policy to cap, rotate, or otherwise bound transcript `.jsonl` files whose unbounded growth can make the Gateway saturate CPU and become unresponsive.

## Best Possible Solution

Choose a maintainer-approved active-transcript bounding policy, then implement schema, docs, runtime maintenance, concurrency safeguards, and oversized-transcript regression tests around that policy without hot-path full-directory scans.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 have no `session.maintenance` raw active transcript cap, though I did not live-replay the exact CPU 100% stall.

## Solution Assessment

Unclear as a final fix: a disabled-by-default raw cap is plausible, but maintainers must first decide whether raw archival, semantic compaction, SQLite lifecycle policy, or docs is the supported behavior.
