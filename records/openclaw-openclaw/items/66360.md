- `impact:crash-loop`: The affected problem class is process availability: unbounded transcript growth can hang or stall the gateway.

## Summary

Keep open. Current main and the latest release still lack a raw `session.maintenance` byte or line cap for active referenced transcript `.jsonl` files; `maxActiveTranscriptBytes` is a compaction trigger, not a raw transcript maintenance policy.

## What This Changes

Request to add bounded maintenance for transcript `.jsonl` files so oversized raw transcripts can be capped, rotated, or otherwise kept from unbounded growth.

## Best Possible Solution

Maintainers should choose and document the direction: either add an opt-in raw transcript maintenance policy with schema, docs, deterministic oversized-file tests, and startup/maintenance scan bounds, or explicitly make compaction-based shrinking the supported answer for active transcript growth.

## Reproduction Assessment

Yes for the missing capability: current source and the latest release have no raw `session.maintenance` byte or line cap for active referenced transcript files. I did not reproduce the reported 100% CPU saturation live.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw cap is plausible, but the best fix must be reconciled with `maxActiveTranscriptBytes`, successor-transcript compaction, disk-budget eviction, and history recovery semantics.
