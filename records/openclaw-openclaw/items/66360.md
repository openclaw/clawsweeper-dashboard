## Label Justifications

- `P1`: The issue reports a real gateway CPU saturation/unresponsive workflow caused by unbounded transcript files.

## Summary

Keep open: current main and latest release have an opt-in compaction-based active transcript byte guard, but still do not implement a raw `session.maintenance` cap for individual `.jsonl` transcripts. The source problem remains valid, while the next step needs maintainer direction because the fix crosses config, retention, and compaction policy.

## What This Changes

The issue requests bounded maintenance for individual raw session transcript `.jsonl` files so long-lived topic/group sessions cannot grow without limit and saturate the gateway.

## Best Possible Solution

Choose one maintainer-approved policy for oversized active transcript files, then align runtime behavior, schema/help/docs, and regression coverage around that single supported path.

## Reproduction Assessment

Yes, as a source-level reproduction path. Current `session.maintenance` has no raw per-transcript cap, long-lived human topic/group sessions are protected from normal maintenance deletion, and the existing byte-size mitigation is a separate opt-in compaction trigger.

## Solution Assessment

Unclear. A raw transcript cap is a plausible source fix, but the best solution needs maintainer direction because current main already has compaction and disk-budget mechanisms with different semantics.
