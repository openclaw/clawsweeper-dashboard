## Summary

Keep open. Current main still lacks a `session.maintenance` byte or line cap for individual active transcript `.jsonl` files; the newer active-transcript compaction guard is useful but opt-in, compaction-scoped, and explicitly not raw JSONL archival or maintenance cleanup.

## What This Changes

The issue requests a per-transcript size or line cap for active `.jsonl` session transcript files under `session.maintenance`.

## Best Possible Solution

Maintainers should either define a conservative opt-in raw transcript maintenance policy with schema, docs, and tests, or explicitly document that OpenClaw supports compaction-based active transcript shrinking rather than `session.maintenance` raw JSONL caps.

## Reproduction Assessment

Yes, at source level. Current main has no public `session.maintenance` transcript byte/line cap and no maintenance path that rotates or tail-trims an active referenced `.jsonl`; I did not live-reproduce the CPU saturation.

## Solution Assessment

Unclear until maintainers choose the product direction. A disabled-by-default raw transcript cap is plausible, but it needs to be reconciled with the existing compaction guard and the data-retention/recovery policy.
