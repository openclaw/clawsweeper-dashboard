## Summary

Keep open: current main and v2026.6.10 still do not provide a `session.maintenance` policy that caps one active transcript JSONL file; the existing disk-budget and compaction paths are adjacent mitigations, not the requested raw transcript bound.

## What This Changes

Request a session-maintenance policy to cap, rotate, or otherwise bound individual transcript `.jsonl` files so long-running sessions cannot grow without limit.

## Best Possible Solution

Maintainers should choose one active-transcript bounding policy, then align config schema, docs, locking/concurrency behavior, and oversized-transcript regression coverage around that policy.

## Reproduction Assessment

Yes, at source level: current main has no `session.maintenance` field or runtime path that caps one active transcript JSONL file. I did not live-reproduce the exact CPU 100% stall.

## Solution Assessment

Unclear: a disabled-by-default raw transcript cap is plausible, but maintainers need to choose raw archival, line caps, semantic compaction, or documentation as the supported policy.
