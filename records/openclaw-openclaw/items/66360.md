## Summary

Keep open: current main and the latest release still do not provide a `session.maintenance` cap or rotation policy for individual active transcript JSONL files. Main has related disk-budget cleanup and opt-in semantic compaction, but those are not the requested raw transcript maintenance control and still leave a maintainer product decision around retention, compaction, and live-session safety.

## What This Changes

The issue requests bounded maintenance for individual session transcript `.jsonl` files so long-running sessions cannot grow without limit and stall the gateway.

## Best Possible Solution

Choose a maintainer-approved active-transcript bounding policy, then align schema, docs, cleanup behavior, concurrency locking, and regression tests around that single policy instead of leaving operators to run external cleanup cron jobs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no `session.maintenance` field or runtime path that caps an individual active transcript JSONL file. I did not live-reproduce the reporter's exact CPU 100% stall with a 100 MB production transcript.

## Solution Assessment

Unclear. A disabled-by-default `session.maintenance` transcript cap is plausible, but maintainers first need to decide whether raw archival, semantic compaction, disk-budget cleanup, or documentation is the supported policy for active transcript growth.
