## Summary

Current main and the shipped v2026.4.29 release already implement the requested Discord behavior: an allowlisted guild with no per-guild channel map is admitted, while explicit channel maps still deny non-listed channels and user/role filters still apply. The related prior reports cover the separate configured-channel-map case, which current code keeps distinct from omitted or empty channel maps.

## What This Changes

The issue asks to restore or clearly diagnose Discord guild handling where `channels.discord.groupPolicy: "allowlist"` and an allowlisted guild has no `guilds.<id>.channels` block.

## Best Possible Solution

Keep the shipped Discord plugin behavior: the guild allowlist gates guild admission, omitted or empty per-guild `channels` means all channels in that guild, a non-empty channel map denies non-listed channels, and `users`/`roles` continue to gate senders. Users still on 2026.4.27 should upgrade to v2026.4.29 or newer.

## Reproduction Assessment

No for current main: the reported config path is covered by the Discord policy helper and regression test, and the current path returns allowed for an allowlisted guild without a channel map. I did not run a live Discord gateway, but the source and shipped test evidence exercise the exact gate.

## Solution Assessment

Yes: the current implementation is the narrow maintainable fix because it lives in the Discord plugin policy helper, preserves fail-closed behavior for explicitly configured channel maps, and avoids a core migration or startup repair for a plugin-owned policy distinction.
