## Summary

Current main and the latest shipped tag do not match the reported deny-all behavior. Discord guild preflight treats an omitted or empty `guilds.<id>.channels` map as no channel allowlist, so an allowlisted guild remains allowed and sender/mention checks run next; the related #54277/#54353 context covers the different case where a non-empty channel map should block unlisted channels.

## What This Changes

The issue reports that Discord guild configs using `groupPolicy: "allowlist"` without a per-guild `channels` block deny every channel instead of allowing channels subject to `users` and `requireMention`.

## Best Possible Solution

Keep the documented Discord semantics: omitted or empty per-guild `channels` means all channels in an allowlisted guild are eligible, while a non-empty channel map denies unmatched channels. If no replies still occur, the next investigation should target the next gate in preflight or outbound delivery with verbose Discord logs.

## Reproduction Assessment

Yes for the policy-level check: the deterministic helper path has `groupPolicy: "allowlist"`, an allowlisted guild, and `channelAllowlistConfigured: false` returning allowed, with tests covering that case. No live Discord gateway was run during this read-only review.

## Solution Assessment

Not applicable as a patch review. The current implementation is the narrow maintainable behavior because it preserves guild-level allowlists without requiring per-channel enumeration and still blocks unmatched channels once a non-empty channel map is configured.
