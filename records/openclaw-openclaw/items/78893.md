## Summary

This should stay open: current main still preserves redundant qqbot provider prefixes in the cron delivery path, and the qqbot parser can misroute `qqbot:qqbot:c2c:...` as a C2C ID instead of rejecting or normalizing it.

## What This Changes

The issue asks for qqbot cron delivery targets to be normalized and validated to a single canonical `delivery.to` format.

## Best Possible Solution

Keep this in core as a narrow bug fix: normalize and parse qqbot delivery targets through the plugin-owned target helpers, prevent repeated provider prefixes, and make cron previews display the resolved target without duplicating the channel.

## Reproduction Assessment

Yes, at source level: a job with `delivery.channel="qqbot"` and `delivery.to="qqbot:c2c:..."` can resolve to a preview label prefixed again, and `qqbot:qqbot:c2c:...` survives the qqbot parser as a malformed C2C ID. I did not run a live qqbot cron delivery path.

## Solution Assessment

Yes: the narrow maintainable fix is to add/adjust qqbot-owned target parsing/normalization and cover cron preview resolution, not to add hard-coded qqbot special cases in generic cron storage.
