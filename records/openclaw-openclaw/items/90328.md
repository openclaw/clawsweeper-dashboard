## Summary

Keep this PR open for maintainer review: current main does not expose runtime metadata on `models.list`, and the branch provides a coherent Gateway/WebUI implementation with sufficient proof and no blocking line-level findings. The remaining decision is compatibility and product scope because it adds a public `ModelChoice.agentRuntime` response field while native picker rendering still ignores that metadata.

## What This Changes

The PR adds optional `agentRuntime` metadata to `models.list` model choices, uses it in WebUI model picker labels, regenerates Swift protocol output, and adds focused Gateway/UI regressions.

## Best Possible Solution

Land this as an additive protocol/WebUI observability improvement only after maintainers accept the public `ModelChoice.agentRuntime` field and WebUI-first picker scope; otherwise extend the shared native picker display before merge.

## Reproduction Assessment

Yes, source-level: current main's `ModelChoiceSchema` has no runtime field, while OpenAI official routes resolve to the Codex runtime through `openAIProviderUsesCodexRuntimeByDefault` and `resolveAgentHarnessPolicy`. That makes the reported model picker ambiguity reproducible from the current protocol and runtime policy paths without needing a live account.

## Solution Assessment

Yes, with a maintainer scope decision: exposing optional runtime metadata from `models.list` is the clean boundary for the WebUI label because the Gateway already owns model catalog shaping and runtime policy lookup. The safer alternative, if native parity is required now, is to carry the same label through `OpenClawChatModelChoice` before merge.
