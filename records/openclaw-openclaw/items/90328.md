## Summary

Keep open: the runtime metadata feature is useful and the supplied visual/Gateway proof is strong, but the PR still changes explicit user-configured model aliases in the WebUI picker by appending a runtime suffix to labels that should remain authoritative.

## What This Changes

The PR adds optional `agentRuntime` metadata to `models.list`, propagates it through generated/client types, and decorates WebUI model picker labels with non-default runtime names.

## Best Possible Solution

Preserve explicit aliases exactly, decorate only generated/default catalog labels with runtime metadata, and keep the optional protocol field plus generated/native artifacts aligned after the UI repair.

## Reproduction Assessment

Yes for the review blocker: source inspection shows the PR appends runtime suffixes to the same display-name path that currently honors explicit aliases, and current tests establish aliases as picker labels. I did not run tests because this was a read-only review.

## Solution Assessment

No as-is. The gateway/protocol direction is a plausible narrow observability surface, but the UI implementation should preserve explicit aliases while adding runtime decoration only to generated catalog labels.
