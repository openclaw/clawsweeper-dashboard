## Summary

Keep open: the PowerShell ScheduledTasks direction is useful, and the author fixed the first bot-reported bugs, but the current branch still has blocking Windows fallback, quoting, cleanup, policy, and proof gaps.

## What This Changes

The branch changes `src/daemon/schtasks.ts` to manage Windows Scheduled Tasks through PowerShell first, set task lifetime and battery settings, alter non-admin install behavior, and retain `schtasks` fallback paths.

## Best Possible Solution

Land a narrower ScheduledTasks update that preserves the documented fallback and cleanup contracts, uses existing cmd-safe quoting, removes invalid fallback flags, adds focused regression coverage, and includes redacted real Windows proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows concrete fallback-launch, command-quoting, fallback-create, and uninstall-cleanup regressions against current docs/tests. I did not run a live Windows end-to-end repro in this read-only review.

## Solution Assessment

No. PowerShell ScheduledTasks may be the right mechanism for disabling the 72-hour task limit and setting battery behavior, but this branch is not the best merge path until it preserves existing contracts and the admin-only fallback policy is explicitly accepted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep the documented Startup-folder fallback:** `src/daemon/schtasks.ts:954-959`
  - body: This preflight throws on every non-admin Windows install before `schtasks` can fail into the documented Startup-folder fallback. Current docs and tests require access-denied task creation to install a per-user login item and start the gateway immediately, so this is a user-visible regression unless a maintainer accepts a policy change.
  - confidence: 0.9
- **[P1] Pass the env when launching fallback tasks:** `src/daemon/schtasks.ts:917-935`
  - body: Both new run fallback branches call `launchFallbackTaskScript(params.scriptPath)`, but that helper expects a `GatewayServiceEnv` and reads the task script plus inline env from it. Passing a file path string breaks the fallback path when the task run is accepted but no process starts.
  - confidence: 0.93
- **[P1] Await and pass the env for Startup fallback launch:** `src/daemon/schtasks.ts:1047`
  - body: The create fallback also passes `scriptPath` to `launchFallbackTaskScript` and does not await it, so the access-denied Startup-folder path can report success while launching the wrong path or dropping an async rejection. Keep the existing env-based, awaited launch contract here.
  - confidence: 0.91
- **[P1] Quote the task script for cmd.exe:** `src/daemon/schtasks.ts:977`
  - body: The PowerShell task action builds `cmd.exe` arguments as `/d /s /c '<scriptPath>'`. `cmd.exe` does not use single quotes as path quoting, so paths with spaces or command metacharacters can fail or be parsed as extra commands; reuse the existing cmd quoting semantics for create and update actions.
  - confidence: 0.9
- **[P2] Remove the idle flag from ONLOGON fallback:** `src/daemon/schtasks.ts:1027-1028`
  - body: The `schtasks` fallback creates `/SC ONLOGON` while also passing `/I 0`. Microsoft documents `/I` as valid only with `ONIDLE` and a 1-999 value, so fallback installs can fail instead of registering the gateway task.
  - confidence: 0.96
- **[P1] Continue uninstall cleanup after PowerShell removal:** `src/daemon/schtasks.ts:1105`
  - body: Returning immediately after `Unregister-ScheduledTask` skips deletion of the Startup-folder launcher and generated task script below. Machines that previously used the fallback can keep auto-starting after uninstall, so remove the task and still run artifact cleanup.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete local command-construction risk in the new PowerShell ScheduledTask action.

Concerns:

- **[medium] Use cmd-safe quoting for task action paths:** `src/daemon/schtasks.ts:977`
  - body: The PR embeds the task script path into `cmd.exe /d /s /c '<path>'`; because `cmd.exe` treats single quotes as literal characters, spaces and metacharacters in the path can alter parsing or run unintended commands. Use the existing cmd quoting helper before embedding the command string in PowerShell.
  - confidence: 0.88
