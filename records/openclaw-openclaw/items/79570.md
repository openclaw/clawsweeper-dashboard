## Summary

Keep open: current main still builds the reported trusted inbound metadata plus untrusted sender envelope and routes that prompt shape through the OpenAI/Azure Responses payload path. The Azure docs/onboarding path now points users toward `azure-openai-responses`, but that dedicated transport still reuses the same Responses parameter builder, so the report is not obsolete.

## What This Changes

The issue asks OpenClaw to make Azure OpenAI Responses usable instead of returning 0-token synthetic refusals when sender metadata is included.

## Best Possible Solution

Add a live-proven Azure Responses compatibility path that preserves OpenClaw's untrusted metadata safety boundary, and document `openai-completions` as the fallback if Azure cannot support the current Responses prompt shape.

## Reproduction Assessment

No. Source inspection confirms current main still emits the reported prompt shape into the Responses path, but this review did not run a redacted Azure OpenAI deployment to reproduce the 0-token refusal end to end.

## Solution Assessment

Unclear. The suggested sender-envelope opt-out may unblock some trusted deployments, but a safer maintainable fix is likely Azure-specific payload shaping or a documented fallback that keeps the prompt-injection boundary intact.
