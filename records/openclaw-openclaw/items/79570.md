## Summary

Keep open: current main still composes the reported trusted system metadata plus untrusted sender user block into the Responses request path, and I did not find a merged fix or canonical superseding implementation. The issue is tracked in the Microsoft/Azure tracker, but that tracker is not a resolving PR.

## What This Changes

Make Azure OpenAI work with the `openai-responses` adapter instead of returning synthetic 0-token refusals, or document/configure a safe fallback.

## Best Possible Solution

Add a live-proven Azure Responses compatibility path that preserves untrusted sender/context safety, and document `openai-completions` as the fallback until that path is available.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection confirms current main still emits the reported system/user envelope into the Responses payload path, but the Azure refusal needs a redacted live Azure run to prove end to end.

## Solution Assessment

Unclear: an opt-out for the sender envelope may unblock Azure, but it changes an intentional untrusted-metadata safety boundary. A safer solution should be Azure-specific payload shaping or an explicit documented fallback, backed by live Azure proof.
