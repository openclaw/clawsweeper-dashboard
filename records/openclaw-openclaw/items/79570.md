## Summary

Keep open. Current main still builds the trusted inbound system prompt plus `Sender (untrusted metadata)` user block that the report identifies, and the merged Azure Responses fix in https://github.com/openclaw/openclaw/pull/88946 addressed a different completed-output failure rather than this synthetic zero-token refusal.

## What This Changes

The issue reports Azure OpenAI returning synthetic zero-token refusals when configured through the `openai-responses` adapter, while the same deployment works through `openai-completions`.

## Best Possible Solution

Keep this open for maintainer-owned Azure Responses compatibility triage: prove whether the dedicated Azure route fixes the refusal, then either document that route or move provider-visible sender metadata out of user text without weakening the default untrusted-context protections.

## Reproduction Assessment

Yes, with external credentials: the issue gives concrete Azure config and a `hello` turn/control path, and source inspection shows current main still assembles the implicated prompt envelope. I did not rerun the live Azure case, so this is medium-confidence rather than locally reproduced.

## Solution Assessment

No, not as a simple sender-envelope opt-out. The best fix needs maintainer security judgment and should preserve the untrusted-context boundary while proving an Azure-compatible request shape or documenting the dedicated Azure provider route.
