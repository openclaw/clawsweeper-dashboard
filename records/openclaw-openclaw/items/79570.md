## Summary

Keep open: current main still composes the reported trusted inbound system block plus the user-role `Sender (untrusted metadata):` block and sends that prompt shape through the Responses transport. I did not find a merged or documented resolution for the specific 0-token Azure refusal.

## What This Changes

The issue asks OpenClaw to make the OpenAI Responses adapter usable with Azure OpenAI instead of returning synthetic 0-token refusals caused by the inbound sender metadata envelope.

## Best Possible Solution

Keep this open until maintainers choose and live-prove an Azure Responses-safe prompt shape or explicitly document `openai-completions` as the supported Azure fallback without weakening sender-context safety.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection confirms current main still emits the reported system/user metadata envelope into the Responses payload path, but proving the Azure refusal requires a redacted live Azure OpenAI run.

## Solution Assessment

Unclear: a sender-envelope opt-out may unblock the reporter, but it crosses a safety boundary. The narrower maintainable path is Azure-specific payload shaping or a documented fallback, backed by live Azure before/after proof.
