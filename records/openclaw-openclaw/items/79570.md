## Summary

Keep open. Current main and the latest release still assemble the trusted inbound system prompt plus `Sender (untrusted metadata)` user-context text that the report identifies, while the merged Azure Responses normalization fix addressed a different empty-output failure rather than this synthetic zero-token refusal.

## What This Changes

The issue asks OpenClaw to stop Azure OpenAI Responses turns from producing synthetic zero-token refusals when inbound sender metadata is included in the Responses request payload.

## Best Possible Solution

Keep this as maintainer-owned Azure Responses compatibility work: prove the dedicated Azure route against the current inbound-context payload, then either document that route clearly or move provider-visible sender metadata out of user text without weakening the default untrusted-context protections.

## Reproduction Assessment

Yes, with external Azure credentials: the issue gives concrete config, a `hello` turn, and a completions control path, and source inspection shows current main still assembles the implicated prompt envelope. I did not rerun the private Azure shield path, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No, not as a simple sender-envelope opt-out. The best fix needs maintainer security judgment and should preserve the untrusted-context boundary while proving an Azure-compatible request shape or documenting the dedicated Azure provider route.
