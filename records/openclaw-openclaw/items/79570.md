## Summary

Keep open. Current main and the latest release still carry the trusted inbound metadata prompt, the `Sender (untrusted metadata):` user block, and the shared Responses input conversion path; related Azure Responses fixes address output/auth/stall failures, not this synthetic refusal trigger.

## What This Changes

The issue reports that Azure OpenAI on the OpenAI-v1 Responses surface refuses every OpenClaw turn when the inbound sender metadata envelope is present, while the same deployment works through Chat Completions.

## Best Possible Solution

Keep this open until maintainers choose an Azure-safe inbound metadata representation or compatibility warning, then prove it with a redacted live Azure Responses canary while preserving sender trust separation.
