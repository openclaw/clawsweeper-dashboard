## Summary

Keep open: current main still makes the trusted inbound metadata system block and the untrusted sender/context block model-visible on normal reply turns, and I did not find a merged fix or canonical replacement that resolves the Azure Responses refusal report.

## What This Changes

The issue asks OpenClaw to make Azure OpenAI Responses usable by avoiding or documenting the sender/inbound metadata envelope that triggers synthetic 0-token refusals.

## Best Possible Solution

Add a live-proven Azure Responses compatibility path that preserves sender/context safety, with docs recommending `openai-completions` or the dedicated Azure provider until the safe Responses shape is verified.

No high-confidence live reproduction is available from this review. Source inspection does confirm the reported trusted-system plus untrusted-user metadata envelope still reaches the Responses model path on current main.

## Solution Assessment

Unclear: an opt-out may unblock Azure, but it would change an intentional prompt-injection metadata boundary. The safer path is Azure-specific payload shaping or a documented fallback, backed by redacted live Azure proof.
