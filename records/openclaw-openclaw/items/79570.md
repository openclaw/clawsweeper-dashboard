Keep open: current main still builds and sends the reported trusted-system plus untrusted-user metadata envelope through the Azure/OpenAI Responses payload path, and the merged Azure stream fixes address adjacent first-event stalls rather than the 0-token refusal trigger reported here.

## What This Changes

The issue reports that Azure OpenAI Responses returns synthetic 0-token refusals for OpenClaw’s inbound metadata prompt envelope while the completions adapter succeeds.

## Best Possible Solution

Design and live-prove an Azure Responses-compatible prompt/payload shape that preserves OpenClaw’s untrusted sender metadata boundary, with a documented completions fallback if no safe Responses shape is available.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this review. The source path clearly still emits the reported system/user envelope into the Responses payload, but proving Azure’s synthetic refusal requires a redacted live Azure run.

## Solution Assessment

Unclear: the requested sender-envelope opt-out may unblock some deployments, but it changes an intentional prompt-injection boundary. A narrower Azure-specific payload-shaping or documented fallback path should be chosen with live proof.
