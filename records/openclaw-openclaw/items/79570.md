## Summary

Keep open: current main still builds the reported trusted inbound metadata plus untrusted sender block and routes that prompt shape through OpenAI/Azure Responses; I did not find a merged fix or a canonical direct duplicate that makes the Azure refusal report obsolete.

## What This Changes

The issue asks OpenClaw to make Azure OpenAI Responses usable when the inbound sender metadata envelope triggers Azure-side synthetic refusals, or to document a safe fallback.

## Best Possible Solution

Add a live-proven Azure Responses compatibility path that preserves sender/context safety, and document `openai-completions` as the Azure fallback until a secure Responses payload shape is validated.

## Reproduction Assessment

No. Source inspection confirms current main still emits the reported system/user envelope into the Responses payload path, but a high-confidence end-to-end reproduction still needs a redacted live Azure OpenAI Responses run plus the `openai-completions` control.

## Solution Assessment

Unclear. A sender-envelope opt-out might unblock some single-tenant deployments, but the safer maintainer direction is Azure-specific payload shaping or an explicit documented fallback that does not weaken the default untrusted-metadata boundary.
