## Summary

Keep open: this is a focused attempt to fix a real preflight compaction timeout bug, but the current patch still has merge-blocking timeout-ownership risk and committed proof artifacts that should not become durable repository surface.

## What This Changes

The PR changes budget/preflight compaction to pass `AbortSignal.any([replyOperation.abortSignal, AbortSignal.timeout(resolveCompactionTimeoutMs(cfg))])` and adds one repro script plus evidence markdown under `scripts/repro`.

## Best Possible Solution

Land a narrow repair that keeps compaction timeout ownership in the safety wrapper, defines the explicit cancellation signal for preflight compaction, removes PR-only proof artifacts, adds focused regression coverage, and includes redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.9 pass `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper lets external aborts race the configured timeout. I did not run the reporter's slow-provider runtime scenario in this read-only review.

## Solution Assessment

No as submitted. The call site is the right area, but the safer fix should keep the compaction timeout in the existing safety wrapper or pass a dedicated cancellation signal rather than injecting a second timeout through the external-abort parameter.
