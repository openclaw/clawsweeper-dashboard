## Summary

Keep open: current main still lacks an outer catch around approval expiry delivery, and this PR is a narrow reliability hardening. The remaining issue is low priority but real: the new catch is in a shared exec/plugin approval handler while its log message is hard-coded to `exec approvals`.

## What This Changes

The PR adds a catch handler around approval expiry notification delivery and adds focused exec approval forwarder tests for expiry delivery cleanup and error handling.

## Best Possible Solution

Merge the reliability hardening after the expiry catch log uses `params.strategy.kind`, while keeping `pending.delete(requestId)` before delivery and retaining the focused regression coverage.

## Reproduction Assessment

No high-confidence natural runtime reproduction was established. Source inspection shows current main launches expiry delivery without an outer catch, but ordinary per-target delivery failures are already caught inside `deliverToTargets`, so the failing path is limited to unexpected errors escaping that helper.

## Solution Assessment

Yes, adding an outer catch while preserving delete-before-delivery semantics is the narrow maintainable fix. The catch should use `params.strategy.kind` because the helper is shared by exec and plugin approvals.
