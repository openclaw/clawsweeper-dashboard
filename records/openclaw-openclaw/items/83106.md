## Summary

Keep open for maintainer review: the PR is a focused exec-approval bug-fix attempt, but the stated unhandled delivery rejection path is not supported by current main because delivery errors are already caught inside the helper, and the patch moves expiry cleanup behind channel I/O where a slow or stuck delivery can retain stale pending state. It also needs after-fix real behavior proof before merge.

## What This Changes

The PR moves exec/plugin approval expiry cleanup to after expiry-message delivery and adds an outer catch that logs and deletes on delivery failure.

## Best Possible Solution

Keep expiry cleanup independent of channel delivery; if there is a real unhandled expiry-notification path, add focused regression coverage and logging without retaining expired approvals past their timeout.

## Reproduction Assessment

No. Source inspection of current main shows the reported delivery-rejection path is already caught and logged inside deliverToTargets, and the PR body does not provide a real failing runtime trace.

## Solution Assessment

No. Moving pending.delete after delivery is not the best fix because it makes cleanup depend on channel I/O; the safer path is to keep deletion before the await and add evidence-backed handling only for a concrete uncaught path.
