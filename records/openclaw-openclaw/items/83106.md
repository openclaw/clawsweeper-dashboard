## Summary

Keep open: current main still has the detached approval-expiry IIFE, and this PR is a focused reliability hardening with sufficient terminal proof. The remaining issue is a low-risk diagnostic bug: the new catch log says exec approvals even though the shared handler also serves plugin approvals.

## What This Changes

The PR attaches a catch/log handler to expiry delivery in the shared approval forwarder and adds exec-forwarder tests for expiry error handling and pending cleanup.

## Best Possible Solution

Keep the catch and delete-before-delivery ordering, but derive the expiry catch log prefix from `params.strategy.kind` and keep focused regression coverage for expiry cleanup/error handling.

## Reproduction Assessment

No high-confidence natural runtime reproduction was established. Source inspection shows current main runs expiry delivery in a detached IIFE without an outer catch, while ordinary per-target delivery failures are already caught and the PR proof uses fault injection for the outer catch path.

## Solution Assessment

Yes, adding an outer catch while preserving `pending.delete` before delivery is the narrow maintainable fix. The catch should use `params.strategy.kind` because the helper is shared by exec and plugin approvals.
