## What This Changes

The PR adds an outer catch/log handler to approval-expiry fire-and-forget delivery and extends the exec approval forwarder tests for expiry cleanup and error handling.

## Best Possible Solution

Keep the reliability hardening, preserve pending deletion before delivery, and change the new expiry catch log to derive the approval kind from params.strategy.kind.

## Reproduction Assessment

No high-confidence natural runtime reproduction was established. Source inspection shows current main runs expiry delivery in a detached async IIFE without an outer catch, while the PR body's live proof uses fault injection rather than a real expiry target outage.

## Solution Assessment

Yes, adding an outer catch while preserving pending.delete before delivery is the narrow maintainable fix. The catch should use params.strategy.kind because the helper is shared by exec and plugin approvals.
