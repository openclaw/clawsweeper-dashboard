Keep open: this is a focused reliability PR with sufficient terminal proof and no blocking correctness or security issue. One low-priority diagnostic issue remains because the new shared expiry catch log hardcodes exec approvals even though the helper also drives plugin approvals.

## What This Changes

The branch adds a `.catch()` log handler to the approval-expiry delivery IIFE and adds exec-approval forwarder tests for expiry logging and pending cleanup.

## Best Possible Solution

Land the catch with delete-before-delivery semantics after adjusting the shared expiry catch log to identify the active approval kind.

## Reproduction Assessment

No high-confidence natural runtime reproduction was established. Current-main source shows a detached expiry delivery IIFE without an outer catch, and the supplied after-fix proof exercises the catch with live gateway fault injection.

## Solution Assessment

Yes, adding an outer catch while preserving `pending.delete` before delivery is the narrow maintainable fix. The catch should use `params.strategy.kind` because the helper is shared by exec and plugin approvals.
