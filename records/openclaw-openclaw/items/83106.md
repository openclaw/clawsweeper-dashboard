## Summary

Keep this PR open for human review: the proposed fix has a blocking cleanup-ordering issue, and the supplied proof is a standalone simulation rather than after-fix OpenClaw behavior.

## What This Changes

The PR reorders exec approval expiry cleanup so `pending.delete(requestId)` runs after expiry delivery and adds an outer catch that logs and cleans up on rejection.

## Best Possible Solution

Keep expiry cleanup independent of channel delivery; if a real uncaught expiry-notification failure exists, add focused regression coverage and logging without retaining expired approvals past timeout.

## Reproduction Assessment

No. Current-main source inspection shows ordinary delivery failures are already returned as failed statuses and logged inside `deliverToTargets`, while the PR proof is a standalone ordering simulation rather than a failing OpenClaw runtime path.

## Solution Assessment

No. Moving cleanup behind channel I/O is not the best fix because it keeps expired approvals pending until delivery settles; the safer path is to keep deletion before the await and add evidence-backed handling only for a concrete uncaught path.
