## Summary

Keep open, but the current patch is not merge-ready: current `main` already catches ordinary per-target delivery failures, while the PR keeps expired approval state pending during async delivery and can accept a late resolve after timeout.

## What This Changes

The PR moves exec-approval expiry cleanup after expiry notification delivery, adds rejection logging, and adds forwarder regression tests for expiry delivery ordering and failures.

## Best Possible Solution

Keep expiry cleanup before channel I/O; if a real uncaught expiry-notification failure exists, add focused logging or coordinate with the broader delivery-failure fix without letting expired approvals remain resolvable.

## Reproduction Assessment

No for the claimed delivery-failure bug as stated: current source already catches and logs ordinary per-target delivery failures. Yes by source inspection for the PR-introduced late-resolution behavior, because the diff keeps the pending entry resolvable while expiry delivery awaits.

## Solution Assessment

No. The proposed ordering change makes approval expiry depend on channel I/O; the narrower maintainable solution is to keep deletion at timeout and add targeted handling only for a concrete uncaught failure path.
