## Summary

Keep open: current main and v2026.6.8 still lack `agents.list[].subagents.workspace`, and source intentionally keeps default same-agent subagents on the requester workspace. This remains a security-sensitive config/product decision rather than a cleanup close.

## What This Changes

The issue requests a per-agent `subagents.workspace` config option so default child subagents can use a separate workspace without defining and explicitly targeting another agent profile.

## Best Possible Solution

Define one opt-in child-workspace isolation contract, then implement it consistently across config schema/types, spawn metadata resolution, docs, and focused security tests while preserving existing default inheritance unless maintainers choose a migration.

## Reproduction Assessment

Yes at source level: omitted `agentId` resolves to the requester agent, and same-agent spawns preserve `ctx.workspaceDir`; current tests assert inherited requester workspace. I did not run a live reproduction because this review is read-only.

## Solution Assessment

Unclear: a per-agent `subagents.workspace` override is plausible, but maintainers still need to choose the isolation boundary and precedence rules. The supported workaround is explicit child agent profiles plus `agentId`, which does not provide the requested default-subagent ergonomics.
