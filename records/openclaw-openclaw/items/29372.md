## Summary

Keep this open. Current main still does not implement `agents.list[].subagents.workspace`: the per-agent config type and strict runtime/generated schemas omit the key, and native default/same-agent subagent spawns still persist the requester workspace. The related merged cross-agent workspace fix addresses target-agent isolation, but not this requested parent-configured workspace override.

## What This Changes

Request adding an `agents.list[].subagents.workspace` option so default or same-agent spawned subagents can use a parent-configured workspace instead of inheriting the requester workspace.

## Best Possible Solution

If maintainers approve the product/security direction, add a backwards-compatible `agents.list[].subagents.workspace` key, normalize it like other workspace paths, expose it in types, runtime schema, generated schema, help/docs, and update native omitted-`agentId` plus explicit same-agent workspace resolution to prefer the override while preserving the current cross-agent target-workspace behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: add `workspace` under `agents.list[].subagents` and current strict schemas have no accepted key, while the same-agent spawn test and resolver path show same-agent spawns continue using the requester workspace.

## Solution Assessment

Unclear. The requested config key would address the reported isolation use case, but changing workspace inheritance affects security boundaries and config semantics, so maintainer product/security review should approve the contract before implementation.
