## Summary

Keep open: current main and the latest release still do not implement a per-agent `subagents.workspace` override, and the shipped `requireAgentId` option is only a mitigation that forces explicit child profiles. This remains a security-sensitive public config and workspace/session-boundary product decision, not conservative cleanup closure.

## What This Changes

The issue requests a per-agent `subagents.workspace` config override so default spawned subagents can use an isolated workspace instead of inheriting the parent agent workspace.

## Best Possible Solution

Define one opt-in child-workspace isolation contract, then implement it consistently across config schema/types, native and ACP spawn resolution, docs, and security tests while preserving current default inheritance unless maintainers choose a migration.

## Reproduction Assessment

Yes at source level: current main and v2026.6.9 lack `subagents.workspace`, and same-agent/default spawn paths still inherit the requester workspace metadata. I did not run a live reproduction because this was a read-only review.

## Solution Assessment

Unclear: a per-agent `subagents.workspace` override is plausible, but maintainers still need to choose the isolation boundary and precedence rules. The current supported workaround is explicit child agent profiles plus `agentId` and optional `requireAgentId`, which does not provide the requested default-subagent ergonomics.
