## Summary

Keep open: current main still does not define a `subagents.workspace` config contract, and omitted-`agentId` native subagent spawns still use the requester agent/workspace path. This remains a new security-sensitive config/product decision rather than cleanup that can be closed.

## What This Changes

Add an `agents.list[].subagents.workspace` config option so same-agent/default subagents can run from a separate workspace without a dedicated agent profile.

## Best Possible Solution

If maintainers accept this isolation model, add a schema-backed trusted config override with clear precedence while preserving explicit target-agent workspace behavior, `requireAgentId`, and internal-only workspace override boundaries.

## Reproduction Assessment

Not applicable: this is a feature/config policy request, not a broken documented behavior. Source inspection confirms the requested key is absent and omitted-`agentId` same-agent spawns still use the requester workspace path.

## Solution Assessment

Unclear until maintainers decide the product and security policy. The requested knob is plausible, but explicit child agent profiles plus `requireAgentId` are the current isolation path, so a second workspace source should be intentional.
