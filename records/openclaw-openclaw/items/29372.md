## Summary

Keep open: current main and v2026.6.8 still lack an `agents.list[].subagents.workspace` option, and current source/tests preserve inherited requester workspace for default same-agent subagents. This remains a useful security-sensitive configuration/product decision, not a cleanup close.

## What This Changes

The issue requests a per-agent `subagents.workspace` config option so default child subagents can use a workspace distinct from their parent agent.

## Best Possible Solution

Define one opt-in child-workspace isolation contract, then implement it consistently across config schema/types, native and ACP spawn workspace resolution, docs, and focused security tests while preserving current default inheritance unless maintainers choose a migration.

## Reproduction Assessment

Yes at source level: omitted `agentId` resolves to the requester agent, and same-agent spawns preserve `ctx.workspaceDir`; current tests assert both omitted-agentId acceptance and inherited requester workspace. I did not run a live reproduction because this review was read-only.

## Solution Assessment

Unclear: a per-agent `subagents.workspace` override is plausible, but maintainers still need to choose the isolation boundary and precedence rules. The current supported workaround is explicit child agent profiles plus `agentId`, which does not provide the requested default-subagent ergonomics.
