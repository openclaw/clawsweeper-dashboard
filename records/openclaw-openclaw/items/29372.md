## Summary

Keep open: current main and the latest release still do not implement a per-agent `subagents.workspace` override, and the existing `requireAgentId` mitigation only forces explicit child profiles rather than giving default subagents a separate workspace.

## What This Changes

The issue requests a per-agent `subagents.workspace` config option so default subagents can use a workspace different from the parent agent workspace.

## Best Possible Solution

Define one opt-in child-workspace isolation contract and implement it consistently across config schema/types, native and ACP spawn resolution, docs, and security regression tests while preserving current inheritance by default unless maintainers choose a migration.

## Reproduction Assessment

Yes, at source level: current main strictly defines per-agent `subagents` without `workspace`, and omitted-agent same-agent spawns still derive child workspace from the requester context. I did not run a live spawn in this read-only review.

## Solution Assessment

Unclear: a per-agent `subagents.workspace` override is plausible, but maintainers still need to choose the isolation boundary and precedence rules. The existing supported path is explicit child agent profiles with `agentId` and optional `requireAgentId`, which is only a mitigation for this request.
