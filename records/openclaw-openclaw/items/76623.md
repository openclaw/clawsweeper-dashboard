## Summary

Keep open. Current main still lets doctor’s missing-configured-plugin install repair select the external `feishu` install candidate from `channels.feishu` even when an enabled alternative plugin owns the same channel; the related auto-enable fix path is separate.

## What This Changes

The issue asks doctor repair to avoid installing `@openclaw/feishu` when `@larksuite/openclaw-lark` is already enabled for `channels.feishu`.

## Best Possible Solution

Teach the install-repair candidate path to honor existing channel ownership and preferOver metadata, then cover the `channels.feishu` plus enabled `openclaw-lark` case with a regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `channels.feishu`, enable `plugins.entries.openclaw-lark`, leave `feishu` uninstalled, and the collector can still choose the `feishu` catalog candidate because the configured channel id matches.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing channel-owner/preferOver arbitration for install repair; a hard-coded `feishu` and `openclaw-lark` skip would only cover this instance.

## Repair Work Prompt

Fix the doctor missing-configured-plugin install repair bug where `channels.feishu` can cause `@openclaw/feishu` to be installed even though enabled `openclaw-lark` already owns the same channel. Keep the fix generic: make install candidate collection or install execution honor existing enabled channel owners and `channelConfigs.<channelId>.preferOver`/channel ownership metadata, matching the auto-enable intent. Add regression coverage for `channels.feishu` plus `plugins.entries.openclaw-lark.enabled=true` with no `plugins.entries.feishu`, no `plugins.installs.feishu`, and no installed `feishu` package; the repair must not call the installer or write a `feishu` install record. Do not add a new config option, do not move plugin-owned behavior into core except through generic metadata helpers, and avoid a hard-coded Feishu/Lark special case unless current metadata cannot express the relationship. Add a single-line `CHANGELOG.md` fix entry because this is user-facing doctor repair behavior.
