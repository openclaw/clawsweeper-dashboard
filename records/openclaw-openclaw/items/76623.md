## Summary

Keep open. Current main still lets the missing-configured-plugin install repair choose the official `feishu` catalog candidate solely because `channels.feishu` is configured, while the existing alternative-owner/preferOver handling lives in adjacent auto-enable and blocker paths rather than this install-repair path.

## What This Changes

The issue asks doctor repair to skip installing the built-in/community `@openclaw/feishu` plugin when an enabled `openclaw-lark` plugin already handles `channels.feishu`.

## Best Possible Solution

Extend the missing-configured-plugin install repair to use the same channel-owner arbitration as startup/auto-enable, then add a regression test for `channels.feishu` plus enabled `openclaw-lark` with no `feishu` install record.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `channels.feishu` and an enabled `plugins.entries.openclaw-lark`, leave `feishu` uninstalled, and the current candidate collector can still select the `feishu` catalog entry because `configuredChannelIds` contains `feishu`. I did not execute the repair command because the review had to keep the checkout read-only.

## Solution Assessment

Yes, the narrow maintainable solution is to make install repair honor existing channel ownership/preferOver metadata before installing channel-selected candidates. A hard-coded `feishu`/`openclaw-lark` guard would cover the report but would not match the existing generic plugin replacement contract.
