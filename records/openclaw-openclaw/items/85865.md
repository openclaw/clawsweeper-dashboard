- `merge-risk: 🚨 availability`: The proposed direct SIGTERM can let descendant processes continue running after cancellation, consuming resources and confusing supervisor state.
- `merge-risk: 🚨 security-boundary`: Because this is the command/tool execution cancellation boundary, a canceled command continuing in a descendant process weakens the expected stop guarantee.

## Summary

Keep open. The SIGTERM-first direction is useful and the PR supplies real Linux proof, but the implementation routes the first phase through a child-only SIGTERM and can lose the existing process-tree kill guarantee for descendants once the direct child exits.

## What This Changes

The PR changes process supervisor cancellation to send SIGTERM first, schedules SIGKILL after a 5s grace period, and updates supervisor tests for graceful settlement and forced escalation.

## Best Possible Solution

Preserve the SIGTERM-first cleanup window while keeping cancellation tree-aware: signal the same child process tree/group, respect detached:false safeguards, escalate the tree to SIGKILL after grace, and keep any unfinished drain-timeout work tracked separately.

## Reproduction Assessment

Yes. Current source shows timeout/manual cancellation enters `cancelAdapter`, and the PR's own live output demonstrates the direct-child SIGTERM/SIGKILL behavior; the remaining blocker is the untested descendant process-tree path.

## Solution Assessment

No. The proposed supervisor timer is the right shape, but the implementation should preserve process-tree cancellation instead of sending the first phase only to the direct child.
