## Summary

Keep open. Current main and v2026.5.18 still run BOOT.md with final reply delivery disabled, a noop boot runtime, and no startup destination route; the two external reports make this a real BOOT.md delivery-contract bug rather than cleanup closure.

## What This Changes

The issue asks for BOOT.md startup runs to deliver requested user messages instead of losing visible output under the current boot-md execution path.

## Best Possible Solution

Define the BOOT.md startup delivery contract, then align runBootOnce, the startup hook context, message-tool routing or inherited destination behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level: a non-empty BOOT.md reaches runBootOnce, which calls agentCommand with deliver:false, a noop runtime log, and no inherited startup route. I did not run an end-to-end macOS or Telegram restart in this read-only review.

## Solution Assessment

No, the suggested one-line deliver:true change is not proven as the best fix because startup still has no defined destination route. The maintainable solution is to decide explicit message-tool-only, configured startup destination, or inherited-route semantics first, then implement that contract with tests.
