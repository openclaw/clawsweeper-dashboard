## Summary

Keep open. Current main and v2026.5.7 still run BOOT.md with final delivery suppressed and no inherited startup channel/target route; the report is source-reproducible, but the proposed one-line `deliver:true` change needs a maintainer decision on BOOT.md startup delivery semantics.

## What This Changes

Request to make BOOT.md startup sends user-visible instead of being silenced by the boot-md run's `deliver:false` path.

## Best Possible Solution

Define and document the BOOT.md startup delivery contract, then align boot execution, docs, and regression tests around either explicit message-tool destinations or a deliberate configured/inherited startup destination.

## Reproduction Assessment

Yes, source-reproducible. The restart path maps to boot-md calling `runBootOnce` without routing context, `runBootOnce` using `deliver:false`, and tests asserting that silent final-delivery shape.

## Solution Assessment

No. The suggested one-line `deliver:true` change is not the best proven fix because the startup path still lacks a channel and target, so maintainers should choose the delivery contract before implementation.
