## Summary

Keep open. Current main and v2026.5.7 still run BOOT.md through boot-md with final reply delivery disabled and no inherited startup channel/target route; the report remains source-reproducible, but changing this safely needs a maintainer decision on the BOOT.md startup delivery contract.

## What This Changes

The issue asks for BOOT.md startup work to be able to deliver user-visible startup messages instead of running silently under `deliver:false`.

## Best Possible Solution

Define and document the BOOT.md startup delivery contract, then align boot execution, message-tool routing, final-reply delivery, and regression tests around that contract.

## Reproduction Assessment

Yes, source-reproducible for the current-main delivery gap: the boot-md startup path reaches `runBootOnce`, which calls `agentCommand` with `deliver:false` and no inherited channel or target. I did not run a live macOS/Telegram restart, so the proof is source-level rather than end-to-end.

## Solution Assessment

No. The suggested one-line `deliver:true` change is not the best proven fix because the startup path still lacks a defined channel/target route; maintainers should choose the delivery contract first.
