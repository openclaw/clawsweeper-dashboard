## Summary

Keep open. Current main and v2026.5.12 still run BOOT.md with final reply delivery disabled, a noop boot runtime log, and no inherited startup channel/target route; fixing the user-visible behavior needs a deliberate BOOT.md startup delivery contract rather than cleanup closure.

## What This Changes

The issue reports that BOOT.md startup messages are not delivered because the boot-md startup run calls the agent with delivery disabled.

## Best Possible Solution

Define and document the BOOT.md startup delivery contract, then align boot execution, message-tool routing or inherited destination behavior, final-reply delivery, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level: a non-empty BOOT.md reaches runBootOnce, which calls agentCommand with deliver:false, a noop output runtime, and no inherited startup route. I did not run a live macOS or Telegram restart, so this is not an end-to-end reproduction.

## Solution Assessment

No, the suggested one-line deliver:true change is not proven to be the best fix by itself because the startup hook still has no defined destination contract. Maintainers should choose explicit message-tool-only, configured startup destination, or inherited route semantics first, then implement the smallest matching code/docs/test change.
