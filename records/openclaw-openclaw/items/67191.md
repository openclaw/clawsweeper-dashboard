## Summary

Keep open. Current main and the latest release still run BOOT.md through `agentCommand` with `deliver:false` and without a startup channel/target context; the current prompt only tells BOOT.md to use explicit `channel` plus `target`, which does not resolve the documented/default startup-status workflow reported here.

## What This Changes

The issue requests that BOOT.md startup runs be able to deliver user-visible startup messages instead of discarding output from the boot-md hook.

## Best Possible Solution

Define one explicit BOOT.md startup contract: either provide a configured/default delivery channel and target through the boot run and message-tool context, or update the template/docs/validation to require explicit channel plus target for BOOT.md sends. Preserve NO_REPLY as final assistant reply suppression and add a regression test for the chosen gateway-startup behavior.

## Reproduction Assessment

Yes. The report gives concrete restart steps, and source review confirms the same path: boot-md calls runBootOnce without delivery context, runBootOnce passes `deliver:false`, and the existing boot test asserts that call shape.

## Solution Assessment

No. The suggested one-line switch to `deliver:true` is not proven best because the boot path still lacks a channel/target and BOOT.md is documented around message-tool sends plus NO_REPLY; maintainers should choose the startup delivery contract first.

## Risks / Open Questions

- A BOOT.md file that hard-codes a valid channel and target may already be able to send through the message tool, so the unresolved behavior is the default/documented startup delivery route rather than every possible message-tool send.

- The related open lifecycle issue #52767 may affect the final design if startup messages should not run on every gateway restart.
