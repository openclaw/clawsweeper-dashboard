## Summary

Keep open. Current main and the latest release still run BOOT.md through a silent boot command with no inherited channel, target, thread, or reply context; the one-line `deliver:true` suggestion is not enough because startup delivery still needs an explicit routing contract.

## What This Changes

The issue asks for BOOT.md startup runs to deliver user-visible messages instead of running silently through boot-md with `deliver:false`.

## Best Possible Solution

Define the BOOT.md startup delivery contract, then align boot execution, docs, and regression tests around either explicit message-tool targets or a deliberate configured/inherited startup destination.

## Reproduction Assessment

Yes, source-reproducible. The restart path maps to boot-md calling `runBootOnce` without routing context, `runBootOnce` using `deliver:false`, and tests asserting that silent command shape.

## Solution Assessment

No. The suggested one-line `deliver:true` change is not the best proven fix because the startup path still lacks a channel and target, so maintainers should choose the delivery contract before implementation.
