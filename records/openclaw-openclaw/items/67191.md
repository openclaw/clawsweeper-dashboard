## Summary

Keep open. Current main and v2026.5.7 still run BOOT.md with final reply delivery disabled and no inherited startup channel/target route; the report remains source-reproducible, but fixing it safely needs a maintainer decision on the BOOT.md startup delivery contract.

## What This Changes

The issue asks for BOOT.md startup runs to deliver user-visible messages instead of running silently with `deliver:false`.

## Best Possible Solution

Define and document the BOOT.md startup delivery contract, then align boot execution, message-tool routing, final-reply delivery, and regression tests around that contract.

## Reproduction Assessment

Yes, source-reproducible: the boot-md startup path reaches `runBootOnce`, which calls `agentCommand` with `deliver:false`, a noop output runtime, and no inherited channel/target route. I did not run a live macOS or Telegram restart, so this is not an end-to-end reproduction.

## Solution Assessment

No, the suggested one-line `deliver:true` change is not the best proven fix by itself because the startup path still lacks a defined destination contract. Maintainers should choose the BOOT.md delivery semantics first, then implement the smallest matching code and docs change.
