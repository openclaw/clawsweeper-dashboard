## Summary

Keep open: current main and the latest release still default implicit deliverable ACP/subagent task notifications to `done_only`, so Discord-originated work can remain silent until terminal status. The user-visible gap is source-backed, but the permanent fix still needs maintainer direction on a Discord-owned or generic visibility seam rather than a core Discord hardcode.

## What This Changes

Discord-originated ACP/subagent tasks without an explicit notify policy should surface progress/state updates instead of leaving the channel with only a typing indicator until terminal status.

## Best Possible Solution

Expose Discord subagent progress through a maintainer-approved generic task visibility seam or an explicit Discord-owned notify policy, preserving non-Discord defaults and keeping Codex-native subagent and per-tool visibility in their separate companion issues.

## Reproduction Assessment

Yes, source-level: ACP/subagent task creation passes requester origin and pending delivery without an explicit notify policy, while current defaulting keeps the policy at `done_only` and state-change delivery only fires for `state_changes`. I did not run a live Discord transport repro in this read-only sweep.

## Solution Assessment

No for the exact proposed core hardcode: the behavior goal is useful, but `requesterOrigin.channel === "discord"` inside the core task registry is not the narrowest maintainable boundary. A Discord-owned explicit policy or generic task-visibility seam is safer.
