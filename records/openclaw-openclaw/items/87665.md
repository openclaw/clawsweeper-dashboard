## Summary

Keep open: current main and the latest local release tag still default implicit deliverable task notifications to `done_only`, so Discord-originated ACP/subagent work can remain silent until terminal state. The reported visibility gap is source-backed, but the proposed Discord-specific core default needs maintainer boundary review because OpenClaw policy keeps core plugin-agnostic.

## What This Changes

The issue asks for Discord-originated background/subagent tasks without an explicit notifyPolicy to surface state-change progress instead of staying on terminal-only delivery.

## Best Possible Solution

Expose Discord subagent progress through a generic task notification policy or plugin-provided explicit notify policy, preserving core channel agnosticism and leaving non-Discord implicit defaults unchanged unless maintainers choose a broader policy.

## Reproduction Assessment

Yes from source inspection: Discord/ACP spawn paths create pending tasks with a requester origin but no explicit notify policy, and the registry default remains `done_only`. I did not run a live Discord channel repro in this read-only review.

## Solution Assessment

No for the exact proposed core hardcode: the outcome is right, but `requesterOrigin.channel === "discord"` inside the task registry is likely the wrong ownership boundary. A generic task visibility signal or Discord/plugin-owned explicit policy would fit OpenClaw better.
