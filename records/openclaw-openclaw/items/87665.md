## Summary

Keep open: current main and v2026.6.6 still default deliverable task records without an explicit notify policy to `done_only`, and Discord ACP/subagent paths still pass requester origin without an explicit `notifyPolicy`. The central visibility problem remains source-reproducible, but the exact fix needs maintainer direction because a Discord-specific default in core would cross the core/plugin owner boundary.

## What This Changes

Request to make Discord-originated subagent or ACP tasks without an explicit `notifyPolicy` emit state-change progress updates instead of only terminal notifications.

## Best Possible Solution

Route Discord-originated ACP/subagent visibility through a generic task notification seam or Discord-owned explicit `notifyPolicy`, with regression tests proving non-Discord defaults and explicit policies are preserved.

## Reproduction Assessment

Yes, at source level: current main still gives deliverable tasks without explicit `notifyPolicy` a `done_only` default, while Discord ACP/subagent paths carry requester origin without setting policy. I did not run a live Discord repro in this read-only pass.

## Solution Assessment

No for the exact proposed core hardcode: the behavior goal is valid, but `requesterOrigin.channel === "discord"` inside the core task registry is not the clean owner-boundary fix. A Discord-owned explicit policy or maintainer-approved generic seam is safer.
