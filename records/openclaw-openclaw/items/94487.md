## Summary

Keep open: the PR targets a real source-level cache-retention bug, but the patch is not merge-ready because it accepts the new `standard` alias for non-Anthropic cache-eligible paths and the supplied proof does not exercise the changed config/runtime behavior.

## What This Changes

The PR adds `cacheRetention: "standard"` to `"short"` normalization in the embedded agent prompt-cache resolver and adds focused resolver tests for Anthropic direct and Bedrock.

## Best Possible Solution

Land one canonical fix that either scopes `standard` to the Anthropic/Bedrock cache family or deliberately updates the config contract, docs, types, and tests for a broader alias, with real behavior proof for the changed path.

## Reproduction Assessment

Yes, source-level: on current main, `cacheRetention: "standard"` for an Anthropic Bedrock model passes the family gate but is not one of `none | short | long`, so the resolver returns `undefined` and the caller omits the configured value. I did not run a live Bedrock request in this read-only review.

## Solution Assessment

No, not yet: the central resolver is a plausible boundary, but this implementation is too broad because it normalizes `standard` for Google and prompt-cache-key providers too. The safer fix is a scoped alias or an explicit docs/types/product-contract update if maintainers want the alias globally.
