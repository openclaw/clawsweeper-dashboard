## Summary

Keep open. Current main and v2026.6.10 still use the flat migrated-value set against the global known-secret env-key list, so the reported same-value `.env` over-scrub remains real; the exact prior fix PR was closed unmerged because the durable fix needs maintainer-owned target-registry ownership design.

## What This Changes

The issue reports that `openclaw secrets apply` can delete a non-migrated provider `.env` credential when two known provider env vars share the same plaintext value.

## Best Possible Solution

Add explicit env-scrub ownership metadata to the secret target registry, then have apply pair each owned env key only with that target or provider's migrated values, with regression coverage for provider auth, provider headers, auth profiles, plugin secrets, and non-provider env refs.

## Reproduction Assessment

Yes. Source inspection plus the reporter's proof show current main pairs one flat migrated-value set with the global known-secret env-key list, which makes same-value non-migrated provider env lines eligible for deletion; I did not run a fresh write-path repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the exact ownership metadata contract. The current global value/key scrub is not the best solution, and the registry-owned target ownership direction is the cleanest boundary shown by the closed prior PR discussion.
