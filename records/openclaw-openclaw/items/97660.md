- `impact:auth-provider`: The deleted `.env` line is a provider credential source used by later model/provider authentication.
- `maturity:stable`: The issue affects the CLI secrets apply/configure path under the `cli-install-update-onboard-doctor` M4 Stable feature area.

## Summary

Keep open. Current main and v2026.6.10 still use a flat migrated-value set against the global known-secret env-key list, so migrating one provider can still scrub another provider's same-value `.env` credential; the prior exact fix PR was closed unmerged in favor of a maintainer-owned target-registry ownership design.

## What This Changes

The issue asks OpenClaw to stop `secrets apply` and `secrets configure` from deleting unrelated same-value provider `.env` credentials during SecretRef migration.

## Best Possible Solution

Add explicit env-scrub ownership metadata to the secret target registry, then have apply pair each owned env key only with that target or provider's migrated values, with regression coverage for provider auth, provider headers, auth profiles, plugin secrets, and non-provider env refs.

## Reproduction Assessment

Yes by source inspection and reporter proof, but not by a fresh local write-path run in this read-only review. Current main still pairs one flat migrated-value set with the global known-secret env-key list, which makes same-value non-migrated provider env lines eligible for deletion.

## Solution Assessment

Unclear until maintainers choose the exact ownership metadata contract. The current global value/key scrub is not the best solution, and the registry-owned target ownership direction is the cleanest boundary shown by the closed prior PR discussion.
