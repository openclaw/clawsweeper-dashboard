## Summary

Keep open. Current main and the latest release still use a flat migrated-value set against the global known-secret env-key list, so the reported same-value provider collision remains a real secrets migration bug; the earlier fix PR was closed unmerged because the durable ownership model needs maintainer/secops design.

## What This Changes

The issue reports that `openclaw secrets apply` and `secrets configure` can delete a non-migrated provider credential from `.env` when two provider env vars share the same plaintext value.

## Best Possible Solution

Add an explicit env-scrub ownership model for registered secret targets, pair each owned env key only with that target's migrated values, and cover provider auth, ancillary provider headers, auth-profile targets, plugin secrets, and non-provider env refs with focused regression tests.

## Reproduction Assessment

Yes by source inspection, but not by a fresh local run in this read-only review. Current main still pairs one flat migrated-value set with the global known-secret env-key list, which matches the reported over-deletion path.

## Solution Assessment

No, the current implementation is not the best solution because it derives deletion eligibility from global value/key matching. The requested registry-owned ownership direction is more maintainable than the closed apply.ts inference PR, but maintainers should confirm the exact metadata contract.
