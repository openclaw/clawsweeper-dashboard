## Summary

Keep open: current main and v2026.6.8 still lack tool-specific Slack assistant thread status, and this PR is a focused candidate, but the patch is not merge-ready because it duplicates the shared tool-display registry, misses supported/fallback tool names, can churn Slack status writes on start/update events, and lacks real Slack behavior proof.

## What This Changes

The PR adds a Slack-local map from tool names to human-readable status labels and calls `setSlackThreadStatus` from Slack `onToolStart` when a mapped tool starts while assistant thread status is active.

## Best Possible Solution

Derive Slack assistant thread status from the shared progress display model, dedupe unchanged status writes, add focused Slack dispatch tests for mapped/fallback and clearing behavior, and require redacted live Slack proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.8 set Slack assistant thread status to `is typing...` and clear it, while `onToolStart` already fires for tool events without updating assistant status text. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

No. The `onToolStart` seam is the right narrow hook, but this implementation should reuse the shared tool display/progress label path and dedupe status writes instead of adding a separate Slack-only registry.
