## Summary

Keep this PR open, but the proposed patch is not sufficient as the final fix. The fallback may help a package-root-without-main case, but the linked packaged runtime-deps failure still has a concrete unresolved path because `qrcode` is a runtime value dependency that is not mirrored into staged runtime deps and is still exempted by the drift guard as type-only.

## What This Changes

This PR changes the shared QR runtime loader to try `import("qrcode")` first and fall back to `qrcode/lib/index.js`, adds QR runtime regression tests, and adds a changelog entry for the WhatsApp/Feishu QR pairing failure.

## Best Possible Solution

Land a complete shared QR packaging fix: keep the loader fallback if maintainers want the extra resilience, but also mirror `qrcode` into packaged root runtime deps, remove or update the stale drift-guard exemption, and add staged runtime-deps coverage proving QR helpers resolve from packaged WhatsApp/Feishu installs.

## Reproduction Assessment

Yes. The linked regression gives a concrete staged packaged install error, and current main statically reproduces the missing-dependency shape: shared QR rendering imports `qrcode`, WhatsApp/Feishu reach it through the SDK, and the staged runtime-deps mirror does not include `qrcode`.

## Solution Assessment

No, not yet. The fallback is a reasonable defensive addition, but by itself it still requires the `qrcode` package to exist in the staged runtime-deps root; the safer complete fix is to guarantee that package via mirror metadata and guard coverage.
