## Summary

Keep open: current main and v2026.5.19 still map Slack `message_changed` and `message_deleted` to subtype system events, and implementing `ignoreSubtypes` is a new public Slack config decision rather than conservative cleanup.

## What This Changes

Request adds a `channels.slack.ignoreSubtypes` config key so Slack `message_changed` and `message_deleted` events can be dropped before waking sessions.

## Best Possible Solution

Add an owner-approved, opt-in Slack subtype filter in Slack config, schema, docs, and the event layer after Assistant DM safeguards, while leaving `thread_broadcast` as normal message input.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.19` still route `message_changed` and `message_deleted` through subtype system-event enqueue, and focused config searches show no `ignoreSubtypes` key.

## Solution Assessment

Unclear for the exact public API until an owner decides the config scope and event ordering. The maintainable direction is an opt-in event-layer filter placed after Slack Assistant `message_changed` safeguards.
