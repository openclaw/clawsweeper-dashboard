## Summary

Keep open: current main and v2026.6.8 still route ordinary Slack `message_changed` and `message_deleted` subtypes into the Slack system-event path, and the requested `ignoreSubtypes` Slack config surface does not exist. The remaining blocker is a maintainer product decision about whether edit/delete observability should stay default-only or gain an opt-in suppression knob.

## What This Changes

The issue requests a Slack channel-level `ignoreSubtypes` config key so `message_changed` and `message_deleted` events can be dropped before waking agent sessions.

## Best Possible Solution

Choose the Slack edit/delete event policy explicitly; if suppression is accepted, add a narrow opt-in Slack plugin config field for selected edit/delete subtypes with schema, docs, and regression tests while preserving valid Assistant DM and `thread_broadcast` handling.

## Reproduction Assessment

Yes, at source level: ordinary Slack edit/delete events resolve a subtype handler and enqueue a session-scoped system event on current main. I did not run a live Slack Socket Mode reproduction in this read-only review.

## Solution Assessment

Unclear. The proposed `ignoreSubtypes` key is a plausible narrow opt-in solution, but adding a public Slack config knob is not automatically the best path until maintainers confirm the desired edit/delete event policy.
