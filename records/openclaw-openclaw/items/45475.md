## Summary

Keep open. Current main still maps Slack `message_changed` and `message_deleted` through the subtype system-event path that can enqueue a routed session, while the requested opt-in `channels.slack.ignoreSubtypes` key is absent from the public Slack config types, strict schema, docs, and focused tests. The related `thread_broadcast` behavior has been fixed separately, and the closed unmerged edit/delete PRs were hard-coded suppression proposals rather than this configurable request.

## What This Changes

Add an opt-in Slack config key, `channels.slack.ignoreSubtypes`, so configured Slack edit/delete subtypes are dropped before waking an agent session.

## Best Possible Solution

Implement a Slack-owned, opt-in ignore list for supported metadata subtypes with unchanged defaults. The config should validate at the public `channels.slack` and per-account surfaces, flow through the resolved Slack monitor context, and short-circuit matching `message_changed`/`message_deleted` events before system-event authorization or enqueue; docs and generated config metadata should describe the behavior.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: current focused tests expect a `message_changed` event with open DM policy to enqueue a system event, and the proposed config key is absent from strict validation and docs.

## Solution Assessment

Yes. Filtering configured subtypes before `authorizeAndResolveSlackSystemEventContext(...)` and `enqueueSystemEvent(...)` is the narrowest maintainable fix; prompt instructions are fragile, and the closed hard-coded suppression PRs were broader than this opt-in request.
