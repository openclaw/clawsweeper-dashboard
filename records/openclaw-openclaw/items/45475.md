## What This Changes

Requests a Slack `channels.slack.ignoreSubtypes` configuration key so `message_changed` and `message_deleted` events can be dropped before they enqueue system events or wake sessions.

## Best Possible Solution

Decide the Slack-owned event suppression policy; if accepted, add a typed account/channel scoped subtype suppression list at the event layer with schema, docs, and tests proving ignored subtypes enqueue no system events.

## Reproduction Assessment

Yes, source-level. On current main, `message_changed` and `message_deleted` resolve to subtype handlers that enqueue system events before normal message handling; I did not run a live Slack workspace repro.

## Solution Assessment

Unclear pending maintainer direction. The proposed key is plausible and bounded, but the best solution depends on whether maintainers want another Slack config surface, a different default policy, or an existing system-event filtering path.
