## Summary

Keep open: current main and the latest release still route Slack `message_changed` and `message_deleted` through the subtype system-event path, and the requested `ignoreSubtypes` Slack config surface is not implemented; this needs maintainer/product approval because it adds public config and must preserve Slack Assistant edit handling.

## What This Changes

Request a `channels.slack.ignoreSubtypes` config option that drops selected Slack message subtypes before they enqueue system events or wake agent sessions.

## Best Possible Solution

Add an owner-approved, opt-in Slack subtype filter in Slack config, schema, docs, and the event layer while preserving valid Slack Assistant `message_changed` handling and leaving `thread_broadcast` as normal message input.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.12` still route `message_changed` and `message_deleted` through subtype system-event enqueue, while focused config searches show no `ignoreSubtypes` key.

## Solution Assessment

Unclear for the exact public API until an owner decides the config scope and event ordering. The narrow maintainable direction is an opt-in event-layer filter placed after Slack Assistant `message_changed` safeguards.
