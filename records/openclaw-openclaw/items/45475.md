## Summary

Keep this open: current main and the latest release still map Slack `message_changed` and `message_deleted` events into system events, and there is no `ignoreSubtypes` Slack config surface to suppress them before session wake.

## What This Changes

Add an opt-in `channels.slack.ignoreSubtypes` configuration key so selected Slack message subtypes such as `message_changed` and `message_deleted` are dropped at the monitor layer.

## Best Possible Solution

Add a narrow, opt-in Slack subtype ignore list in the Slack plugin config and short-circuit matching subtype system events before authorization/session resolution, while preserving default edit/delete behavior and the existing thread_broadcast message handling.

## Reproduction Assessment

Yes. Source inspection and existing tests provide a high-confidence reproduction path: a `message_changed` or `message_deleted` Slack message event reaches the subtype handler and is enqueued as a system event unless authorization blocks it.

## Solution Assessment

Yes. An opt-in event-layer filter is the narrowest maintainable solution; prompt instructions are fragile, and changing the default edit/delete behavior would be broader than the report asks for.
