## Summary

Keep open. The latest branch is a UI-only renderer slice, while current main still emits/stores live chat deltas as text-only strings, so reasoning-only live payloads cannot reach the new object-stream rendering path; it also widens reasoning visibility by falling back from `/reasoning` to `/think`.

## What This Changes

The PR updates Control UI chat item construction, grouped stream rendering, and grouped-render tests to preserve object-shaped assistant stream payloads and render reasoning-only streaming bubbles.

## Best Possible Solution

Land one consolidated end-to-end reasoning-stream implementation across gateway chat delta/final payloads, Control UI stream state and types, rendering, focused tests, and changelog while preserving the `/reasoning` visibility contract.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main emits text-only chat deltas and the Control UI stores `extractText(payload.message)` in a string `chatStream`, so a thinking-only payload cannot reach this PR's object-stream rendering path in real live chat.

## Solution Assessment

No. Renderer support is useful, but this is not the best standalone fix until structured payloads are plumbed through gateway and controller state and `/reasoning` visibility remains independent from `/think` effort.

  - body: Treating any object stream as renderable means a reasoning-only payload with `showReasoning=false` becomes a stream item even though `renderGroupedMessage` returns no visible bubble content. The previous reading indicator is skipped and the UI can show an empty assistant row until visible text arrives.
  - confidence: 0.88

  - body: `/think` controls model effort, while `/reasoning` controls whether reasoning is displayed and falls back to `off`. Falling back from a missing session reasoning level to `props.thinkingLevel` can show reasoning for sessions that never enabled reasoning visibility.
  - confidence: 0.9
