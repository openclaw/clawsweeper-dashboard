## Summary

This PR should remain open but is not merge-ready: the proposed Feishu classifier can silently discard legitimate final replies, bypass the existing no-visible-reply recovery path, and lacks real-channel proof.

## What This Changes

The PR adds Feishu-specific outbound-text heuristics and two regression tests to suppress presumed SOP noise while allowing memory-governance probe replies.

## Best Possible Solution

Have the producing workflow emit an explicit internal-only or intentional-silence outcome, keep ordinary Feishu final text content-agnostic, and ensure fallback accounting reflects whether visible content was actually sent.

## Reproduction Assessment

Yes from source inspection. On the branch, delivering a final payload containing `OK`, `好的`, `收到`, or another matched short value returns without a Feishu send after the generic dispatcher has already counted the final.

## Solution Assessment

No. The narrow maintainable fix is an explicit producer-owned internal-output state with fallback-aware delivery accounting, rather than inferring workflow semantics from message text inside the Feishu transport.
