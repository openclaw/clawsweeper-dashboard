## Label Justifications

- `P1`: The proposed Feishu send gate can silently suppress common valid final replies in an active channel workflow.
- `merge-risk: 🚨 message-delivery`: The patch directly prevents selected final messages from reaching Feishu and can bypass the existing no-visible-reply recovery behavior.

## Summary

This PR targets a real Feishu symptom, but the proposed transport-level content classifier can silently discard legitimate final replies and bypass the existing no-visible-reply recovery path; it also lacks after-fix proof from a real Feishu setup.

## What This Changes

The PR adds Feishu-specific final-text suppression heuristics plus regression tests for suppressing presumed internal noise while preserving a memory-governance probe response.

## Best Possible Solution

Prevent internal diagnostic output at its owning runtime boundary, or carry an explicit typed non-user-visible outcome into the delivery pipeline so Feishu does not infer product semantics from arbitrary reply text and suppressed output remains correctly represented in fallback accounting.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deliver a text-only final such as `OK` or `收到`; the new classifier rejects it before any Feishu send, while the shared dispatcher can still count the resolved callback as a final delivery.

## Solution Assessment

No. Suppressing arbitrary text by transport-local phrase and length heuristics is not the best fix; the producer should identify internal output explicitly, and the delivery pipeline should preserve accurate visible-delivery state.
