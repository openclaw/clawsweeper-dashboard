## Summary

Keep open: the admission-time guard targets a real source-reproducible cron delivery bug, but the updated head still over-rejects deterministic provider-prefixed `last` routes and blocks disabling ambiguous legacy jobs. The generated protocol output is now included, but the after-fix proof remains test-only rather than a real configured multi-channel Gateway run.

## What This Changes

The PR adds gateway cron validation and tests for ambiguous implicit announce routes, plus a generated Swift Gateway model refresh.

## Best Possible Solution

Keep the gateway admission guard, but mirror delivery-plan semantics exactly, skip disabled or otherwise non-schedulable outcomes, and require real multi-channel Gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: omitted isolated/current/session `agentTurn` delivery resolves to announce `last`, then multi-channel selection throws when no explicit or fallback channel exists; I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No. Admission-time validation is the right boundary, but this diff still diverges from delivery resolution and should skip validation once an update disables the resulting job.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve prefixed targets with last channel:** `src/gateway/server-methods/cron.ts:77-78`
  - body: This throws whenever `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` remains deterministic and should not be rejected.
  - confidence: 0.9
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:244-245`
  - body: Including `enabled` in the validation trigger makes `cron.update` validate `{ enabled: false }` against the resulting delivery even though disabled jobs are not schedulable. That blocks the safest remediation for existing ambiguous jobs; skip this guard when the patched job is disabled.
  - confidence: 0.87
