## Summary

Keep open: the admission-time guard is useful, but this PR still rejects deterministic provider-prefixed `last` routes and blocks disabling ambiguous legacy jobs; the supplied proof is also formatter/lint/Vitest output rather than a configured multi-channel Gateway cron run.

## What This Changes

The branch updates gateway cron add/update validation and cron validation tests to reject ambiguous implicit `last` announce routing in multi-channel cron jobs.

## Best Possible Solution

Keep the gateway admission guard, but mirror delivery-plan and failure-destination semantics exactly, allow disabling invalid legacy jobs, add focused regressions, and require real multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection. Omitted isolated/current/session `agentTurn` delivery resolves to announce `last`, and multi-channel selection fails when no explicit or fallback channel exists; I did not run a live Gateway in this read-only review.

## Solution Assessment

No. Admission-time validation is the right boundary, but this diff must preserve provider-prefixed `last` routes and skip validation once an update disables the resulting job.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:78-82`
  - body: This guard still throws as soon as `delivery.channel` is `last`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` and the same failure-destination shape are deterministic and should stay accepted.
  - confidence: 0.92
- **[P2] Skip delivery validation after disabling a job:** `src/gateway/server-methods/cron.ts:256-261`
  - body: Including `enabled` in the trigger means `cron.update` with `{ enabled: false }` still validates the resulting ambiguous delivery and rejects legacy jobs before they can be disabled. Disabled jobs are not schedulable, so return after applying the patch when the next job is disabled, while keeping the activation guard for `enabled: true`.
  - confidence: 0.88
