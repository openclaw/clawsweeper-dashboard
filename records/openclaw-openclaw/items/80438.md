## Summary

Keep open: the guard is useful, but the current patch is not merge-safe because it applies fail-fast admission to session-scoped implicit `last` routes that current runtime can resolve from stored session context, and the real behavior proof still names an older head after the latest force-push.

## What This Changes

The PR adds gateway `cron.add`/`cron.update` admission validation and regression tests to reject ambiguous implicit `last` announce delivery in multi-channel configurations while preserving provider-prefixed, explicit-channel, disabled, `mode:"none"`, webhook, and single-channel cases.

## Best Possible Solution

Narrow the guard to truly unresolved ambiguous routes, preserve resolvable session/current routing or document a maintainer-approved strict behavior change, then refresh current-head proof.

## Reproduction Assessment

Yes from source inspection: in a multi-channel config, a `session:*` or `current` agent-turn job with implicit announce delivery reaches the new validator before current-main session route resolution can prove a deterministic last route. I did not run tests because this was a read-only review.

## Solution Assessment

No: failing fast for genuinely ambiguous routes is the right direction, but treating all session-scoped implicit `last` delivery as ambiguous is broader than current runtime behavior supports. The safer fix is to preserve or validate resolvable session routes before rejecting.
