## Summary

Keep open: the PR targets a source-reproducible cron delivery bug, but the current patch over-rejects deterministic prefixed routes, blocks disabling ambiguous legacy jobs, and still has only test/lint proof rather than real configured Gateway proof.

## What This Changes

The branch adds gateway cron admission validation and tests to reject ambiguous announce and failure-destination delivery routes in multi-channel configurations.

## Best Possible Solution

Keep the admission-time guard, but align it with delivery resolution semantics, skip it for disabled resulting jobs, and add redacted real multi-channel Gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection shows omitted isolated/current/session agentTurn delivery defaults to announce channel "last", and multi-channel delivery selection later fails when no deterministic channel exists.

## Solution Assessment

No. Admission-time validation is the right boundary, but this patch needs narrower route detection and disabled-job handling before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve prefixed targets with last channel:** `src/gateway/server-methods/cron.ts:77`
  - body: This condition throws whenever `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` remains deterministic and should not be rejected.
  - confidence: 0.88
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:244`
  - body: Including `enabled` in the validation trigger makes `cron.update` validate `{ enabled: false }` against the old ambiguous delivery, and the validator ignores that the patched job is now disabled. That blocks the safest remediation for existing jobs; skip the guard when the resulting job is not schedulable.
  - confidence: 0.86
