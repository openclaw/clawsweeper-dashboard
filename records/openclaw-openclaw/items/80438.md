## Summary

Keep open: the admission-time guard addresses a source-reproducible cron routing bug, but the PR still rejects deterministic provider-prefixed `last` routes and blocks disabling invalid legacy jobs; the supplied proof is only test/check output.

## What This Changes

The PR adds gateway cron add/update validation and focused tests to reject ambiguous implicit `last` announce delivery in multi-channel cron jobs.

## Best Possible Solution

Keep the admission-time guard, but mirror the delivery-plan and failure-destination routing semantics exactly, skip validation for disabled outcomes, add focused regressions, and attach redacted configured multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection. Omitted delivery for isolated/current/session `agentTurn` jobs resolves to announce `last`, while current gateway validation only checks explicit delivery objects.

## Solution Assessment

No. Admission-time validation is the right boundary, but this diff diverges from the existing provider-prefixed `last` route contract and validates jobs even after an update disables them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:79-82`
  - body: Existing delivery resolution and docs allow `delivery.channel: "last"` with a provider-prefixed `delivery.to` to select that provider before falling back to last. This check throws as soon as `channel` is `last`, so `cron.add`/`cron.update` would reject deterministic routes like `{ channel: "last", to: "telegram:123" }`, including failure destinations.
  - confidence: 0.93
- **[P2] Skip delivery validation for disabled outcomes:** `src/gateway/server-methods/cron.ts:242-257`
  - body: The update guard now runs for `enabled` and `schedule` patches after applying the patch, but it never checks `nextJob.enabled`. A legacy ambiguous cron job cannot be disabled with `{ enabled: false }` because validation still rejects its delivery even though disabled jobs are unscheduled and update clears `nextRunAtMs`.
  - confidence: 0.9
