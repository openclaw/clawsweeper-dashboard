## Summary

Keep open: the underlying cron delivery bug is source-reproducible and admission-time validation is the right boundary, but this PR still rejects deterministic provider-prefixed `last` routes, blocks disabled legacy-job remediation, and only supplies test/check output as behavior proof.

## What This Changes

The PR adds gateway cron add/update validation and focused tests to reject implicit `last` announce routing in multi-channel cron jobs while preserving selected deterministic routes.

## Best Possible Solution

Keep the admission-time guard, but mirror `resolveCronDeliveryPlan` and failure-destination semantics, allow updates that leave invalid legacy jobs disabled, and attach redacted configured multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection. Current main resolves omitted isolated/current/session `agentTurn` delivery to announce `last`, and multi-channel target selection can fail without an explicit or provider-prefixed route.

## Solution Assessment

No. Admission-time validation is the right maintainable boundary, but this implementation diverges from existing provider-prefixed `last` delivery semantics and validates disabled update outcomes that should be allowed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve provider-prefixed last-channel routes:** `src/gateway/server-methods/cron.ts:79-80`
  - body: Current delivery planning resolves a provider-prefixed `delivery.to` before falling back to `last`, and current tests cover `channel: "last", to: "telegram:123"`. This guard throws as soon as `delivery.channel` is `last`, so it rejects deterministic primary and failure-destination routes that should stay accepted.
  - confidence: 0.93
- **[P2] Allow disabling ambiguous legacy cron jobs:** `src/gateway/server-methods/cron.ts:242-257`
  - body: Updates now validate whenever `enabled` is present and then validate the patched job without checking `nextJob.enabled`. A `{ enabled: false }` remediation for an ambiguous legacy job will still fail, even though disabled jobs are unscheduled and `cron.update` clears their next-run state.
  - confidence: 0.9
