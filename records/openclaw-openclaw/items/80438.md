## Summary

Keep open: the underlying cron delivery bug is source-reproducible and admission-time validation is the right boundary, but this PR still rejects documented provider-prefixed `channel: "last"` routes and can block disabling invalid legacy jobs. The supplied proof is still formatter/lint/Vitest output rather than a configured multi-channel Gateway cron run.

## What This Changes

Adds gateway cron add/update validation and focused tests for ambiguous implicit `last` announce delivery in multi-channel configurations.

## Best Possible Solution

Keep the admission guard, but make it mirror delivery-plan and failure-destination semantics, skip validation for disabled post-update jobs, add regressions for both cases, and require redacted configured multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection. Current main defaults isolated/current/session `agentTurn` jobs without delivery to announce `channel: "last"`, and multi-channel delivery is ambiguous unless an explicit channel or provider-prefixed route supplies determinism.

## Solution Assessment

No. Admission-time validation is the right boundary, but this implementation must preserve provider-prefixed `last` routes and allow disabling invalid legacy jobs before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve provider-prefixed last routes:** `src/gateway/server-methods/cron.ts:79-82`
  - body: Current delivery planning and docs allow `channel: "last"` with a provider-prefixed `to` such as `telegram:123` because the prefix selects the deterministic provider before fallback. This check throws as soon as `params.channel` is `last`, so it rejects existing valid primary and failure-destination routes in multi-channel configs.
  - confidence: 0.94
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:256-261`
  - body: The update guard validates after applying any `enabled` patch, but it never exits when the patched job is disabled. A user trying to remediate a bad legacy job with `{ enabled: false }` can still be rejected even though disabled jobs are unscheduled.
  - confidence: 0.91
