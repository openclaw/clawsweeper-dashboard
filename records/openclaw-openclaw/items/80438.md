## Summary

The PR targets a real gateway cron validation gap, but the current head still blocks a safe remediation path for legacy ambiguous jobs and the supplied proof is focused tests/lint rather than a configured multi-channel Gateway cron add/update run.

## What This Changes

The PR adds gateway cron add/update validation and focused tests to reject ambiguous implicit `last` announce delivery in multi-channel configs while preserving deterministic announce routes.

## Best Possible Solution

Keep the admission-time guard, but validate only enabled outcomes so operators can disable legacy ambiguous jobs, then require redacted real Gateway proof before merge.

## Reproduction Assessment

Yes, source-level. Current main resolves omitted isolated/current/session `agentTurn` delivery to announce/`last`, and the create path can persist that implicit announce delivery; I did not execute tests because this review is read-only.

## Solution Assessment

No as written. Admission-time validation is the right seam, but the update guard must allow disabling existing ambiguous jobs and the PR still needs real configured-Gateway behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Allow disabling legacy ambiguous cron jobs:** `src/gateway/server-methods/cron.ts:242-247`
  - body: The new update guard runs whenever `enabled` is present, then validates the patched job even when the result is `enabled: false`. Existing jobs created before this guard can have implicit `last` announce delivery; with this branch, an operator cannot disable such a job in a multi-channel config because `{ enabled: false }` is rejected before `context.cron.update` clears scheduling state. Gate this validation on the post-patch job being enabled, or otherwise allow disabling while still blocking activation.
