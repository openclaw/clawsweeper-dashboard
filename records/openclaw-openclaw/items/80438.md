## Summary

Keep open: the fail-fast cron routing direction is useful, but this head still blocks session-resolved `last` delivery paths and the posted real behavior proof is for an older force-pushed head.

## What This Changes

The PR adds Gateway `cron.add`/`cron.update` validation and tests to reject implicit `last` announce routes in multi-channel cron configurations while preserving explicit, provider-prefixed, disabled, webhook, and single-channel cases.

## Best Possible Solution

Keep the fail-fast guard for truly unresolved multi-channel implicit routes, but preserve routes that resolve through stored current/session context and require current-head real behavior proof before merge.

## Reproduction Assessment

Yes from source: this head rejects implicit `last` delivery for isolated/current/session agentTurn jobs in multi-channel configs before the session-route resolver can prove the route is deterministic. Current main code and tests show `sessionTarget`/`sessionKey` can resolve `last` through stored session context.

## Solution Assessment

No. Admission-time validation is the right layer for truly ambiguous jobs, but the best fix should reuse or mirror the existing delivery resolver so session-resolved routes stay valid.
