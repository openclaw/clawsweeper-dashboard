## Summary

Keep open: the branch targets a real source-reproducible cron delivery bug, but it still introduces two validation regressions and the after-fix proof is test-only, so it should not merge as-is.

## What This Changes

The PR adds gateway cron admission validation that rejects ambiguous announce delivery routes in multi-channel configurations while preserving deterministic delivery and failure-destination cases.

## Best Possible Solution

Keep the add/update admission guard, but mirror delivery-plan and failure-destination semantics exactly, allow disabling invalid legacy jobs, and require real multi-channel Gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection shows omitted isolated/current/session `agentTurn` delivery resolves to announce `last`, and multi-channel selection fails when no explicit or fallback channel exists.

## Solution Assessment

No. Admission-time validation is the right boundary, but this diff must preserve provider-prefixed `last` routes and skip delivery validation once an update disables the resulting job.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:79-82`
  - body: This guard still throws as soon as `delivery.channel` is `last`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` and the same failure-destination shape are deterministic and should stay accepted.
  - confidence: 0.92
- **[P2] Skip delivery validation after disabling a job:** `src/gateway/server-methods/cron.ts:256-261`
  - body: An update with `{ enabled: false }` still validates the resulting ambiguous delivery and rejects legacy jobs before they can be disabled. Disabled jobs are not schedulable, so return after applying the patch when the next job is disabled, while keeping the activation guard for `enabled: true`.
  - confidence: 0.88
