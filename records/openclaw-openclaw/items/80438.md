## Summary

Keep open: the PR targets a real, source-reproducible cron delivery ambiguity, but the proposed guard introduces blocking regressions around deterministic provider-prefixed routes and disabling legacy ambiguous jobs, and the PR body only includes test proof.

## What This Changes

The PR adds gateway cron validation that rejects ambiguous announce and failure-destination delivery before cron jobs are added or updated, with expanded validation coverage.

## Best Possible Solution

Land a narrower admission guard that rejects only enabled or newly schedulable ambiguous announce outcomes, preserves provider-prefixed deterministic targets, and includes real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows omitted delivery for isolated agent-turn cron jobs resolves to announce delivery with channel "last", and multi-channel target selection fails later when no deterministic channel exists.

## Solution Assessment

No. The PR is aimed at the right validation boundary, but the current diff over-rejects deterministic prefixed routes and blocks disabling ambiguous legacy jobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve provider-prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:77`
  - body: This throws as soon as `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current main resolves `channel="last"` with `to="telegram:123"` through `resolveTargetPrefixedChannel` before falling back to last, so this rejects a deterministic route the PR says should remain supported. Gate the throw on the absence of `hasDeterministicChannel`.
  - confidence: 0.88
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:247`
  - body: Adding any `enabled` patch to this validation set makes `cron.update` reject `{ enabled: false }` for an existing isolated agent-turn job with omitted delivery, because the validator ignores that the patched job is disabled. That blocks the safest remediation for legacy ambiguous jobs; validate enabled patches only when they keep or make the job schedulable.
  - confidence: 0.86
