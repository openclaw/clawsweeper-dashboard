## Summary

The PR should stay open: the admission guard targets a real source-reproducible cron bug, but the current head still over-rejects deterministic provider-prefixed targets and blocks disabled legacy-job remediation; its real behavior proof is also only formatter/lint/Vitest output.

## What This Changes

The PR adds gateway cron admission validation and tests for ambiguous announce delivery in multi-channel configs, plus a generated Swift protocol model refresh.

## Best Possible Solution

Keep the admission guard, but mirror delivery-plan channel resolution exactly, skip disabled or otherwise non-schedulable outcomes, and land with real multi-channel Gateway proof.

## Reproduction Assessment

Yes. Source inspection shows omitted isolated/current/session agentTurn delivery resolves to announce channel last, and multi-channel selection fails when multiple message channels are configured without an explicit or fallback channel; I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No. Admission-time validation is the right boundary, but this implementation still diverges from delivery resolution and validates disabled outcomes, so it is not the narrowest maintainable fix yet.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve prefixed targets with last channel:** `src/gateway/server-methods/cron.ts:77-78`
  - body: This still throws whenever `delivery.channel` or `delivery.failureDestination.channel` is `"last"`, even when `to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` remains deterministic and should be accepted.
  - confidence: 0.91
- **[P2] Skip announce validation for disabled outcomes:** `src/gateway/server-methods/cron.ts:240-245`
  - body: The update guard now runs for `enabled` and `schedule` patches, then validates the merged job without checking whether `nextJob.enabled` is false. That blocks disabling an ambiguous legacy job, or editing one that is already disabled, even though disabled jobs are not schedulable.
  - confidence: 0.88
