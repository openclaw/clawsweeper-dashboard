## Summary

Keep open: the PR targets a real source-reproducible cron routing bug, but current head still rejects deterministic provider-prefixed `last` routes and blocks disabling invalid legacy jobs; the supplied proof is still test/check output only.

## What This Changes

The branch adds gateway cron add/update validation and tests to reject implicit `last` announce delivery in multi-channel cron jobs while preserving selected deterministic routes.

## Best Possible Solution

Keep the gateway cron admission guard, but make it mirror delivery-plan and failure-destination routing exactly, allow disabling invalid legacy jobs, add focused regressions, and require redacted real multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection. Omitted delivery for isolated/current/session `agentTurn` jobs resolves to announce `last`, and multi-channel target selection can fail without an explicit or provider-prefixed route.

## Solution Assessment

No. Admission-time validation is the right boundary, but this implementation must preserve provider-prefixed `last` routes and skip delivery validation once an update disables the resulting job.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:79-82`
  - body: Current delivery planning resolves a provider-prefixed `delivery.to` before falling back to `last`, and current tests cover `channel: "last", to: "telegram:123"` as deterministic. This guard throws whenever `delivery.channel` is `last`, so it rejects accepted primary and failure-destination routes that should remain valid.
  - confidence: 0.93
- **[P2] Skip delivery validation after disabling a job:** `src/gateway/server-methods/cron.ts:242-257`
  - body: Because `enabled` triggers outcome validation and the patched job is validated without checking `nextJob.enabled`, `cron.update` with `{ enabled: false }` still rejects an ambiguous legacy job before it can be disabled. Disabled jobs are unschedulable, so the guard should block activation, not remediation.
  - confidence: 0.9
