## Summary

Keep open: the admission-time cron guard addresses a source-reproducible bug, but the current diff still rejects a deterministic provider-prefixed route and blocks disabling ambiguous legacy jobs; the real-behavior proof is also test-only.

## What This Changes

The branch adds cron add/update validation and tests for ambiguous implicit announce delivery, plus a generated Swift GatewayModels refresh.

## Best Possible Solution

Keep the gateway admission guard, but make it mirror delivery-plan semantics exactly, skip validation for disabled outcomes, and land it with real multi-channel Gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted isolated/current/session `agentTurn` delivery resolves to announce `last`, then multi-channel channel selection fails closed without an explicit or fallback channel.

## Solution Assessment

No. Admission-time validation is the right boundary, but this implementation diverges from current delivery resolution and should not validate disabled, non-schedulable outcomes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve prefixed targets with last channel:** `src/gateway/server-methods/cron.ts:77-78`
  - body: This throws whenever `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves the prefixed target before falling back to `last`, so `channel: "last", to: "telegram:123"` remains deterministic and should be accepted.
  - confidence: 0.9
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:244-245`
  - body: Including `enabled` in the validation trigger makes `cron.update` validate `{ enabled: false }` against the resulting delivery even though disabled jobs are not schedulable. That blocks the safest remediation for existing ambiguous jobs; skip this guard when the patched job is disabled.
  - confidence: 0.88
