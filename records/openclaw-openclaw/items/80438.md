## Summary

Keep open: the underlying cron delivery bug is source-reproducible and admission-time validation is the right boundary, but this branch still rejects deterministic provider-prefixed `last` routes, blocks disabling invalid legacy jobs, and has only test-based proof.

## What This Changes

The PR adds gateway cron add/update validation and focused tests to reject ambiguous implicit `last` announce delivery in multi-channel cron setups.

## Best Possible Solution

Land an admission guard only after it mirrors delivery resolution semantics, allows disabled-job remediation, adds focused regressions for those cases, and includes redacted configured multi-channel Gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main defaults omitted isolated/current/session `agentTurn` delivery to announce with `channel: "last"`, and unresolved last-route delivery reaches channel selection that throws in multi-channel configs.

## Solution Assessment

No. Admission-time validation is a maintainable boundary, but this implementation must preserve provider-prefixed `last` routes and skip delivery validation once an update disables the resulting job.

## Review Findings

- **[P2] Preserve prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:79-82`

  - confidence: 0.93

  - confidence: 0.9
