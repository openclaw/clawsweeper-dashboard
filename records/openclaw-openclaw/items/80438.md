## Summary

Keep open: this is a useful cron delivery bug-fix direction, but the current PR is not merge-ready because it still rejects session-backed `last` routes that current runtime and docs support, and the supplied real-behavior proof is stale for the current head.

## What This Changes

The PR adds gateway `cron.add`/`cron.update` validation and regression tests to reject implicit `last` announce delivery in multi-channel configurations while preserving explicit, provider-prefixed, disabled, webhook, and single-channel cases.

## Best Possible Solution

Keep the admission-time guard for truly unresolved multi-channel announce routes, but preserve routes that current runtime can resolve from an explicit channel, provider prefix, single configured channel, or stored session delivery context before rebasing and refreshing proof.

## Reproduction Assessment

Yes, source-level. Current main maps isolated/current/session `agentTurn` cron jobs without delivery to announce/last and resolves that through session history or channel selection; the PR blocker is also source-visible because the new guard rejects before that lookup.

## Solution Assessment

No, not as-is. Admission-time validation is a plausible fix layer, but the best fix must preserve resolvable session-backed `last` routes or make the stricter policy an explicit maintainer-approved compatibility break.
