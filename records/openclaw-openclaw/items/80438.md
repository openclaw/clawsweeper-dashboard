## Summary

Keep open: the underlying cron delivery bug is source-reproducible, but this PR still rejects a deterministic provider-prefixed `last` route, can block disabling legacy ambiguous jobs, and only supplies test/mock proof.

## What This Changes

The PR adds gateway cron validation and tests to reject ambiguous implicit `last` announce routes for multi-channel cron delivery, plus a generated Swift protocol refresh.

## Best Possible Solution

Keep the admission-time guard, but mirror delivery resolution semantics, skip the guard for disabled/non-schedulable outcomes, and require real multi-channel Gateway proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: omitted isolated/current/session `agentTurn` delivery resolves to announce `last`, then multi-channel selection fails when no explicit or session fallback channel exists. I did not run a live Gateway repro because this review was read-only.

## Solution Assessment

No. Admission-time validation is the right boundary, but this diff needs to preserve deterministic provider-prefixed routes and allow disabling ambiguous jobs before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve prefixed targets with last channel:** `src/gateway/server-methods/cron.ts:77-78`
  - body: This throws whenever `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current delivery planning resolves a prefixed `to` before falling back to `last`, so `channel: "last", to: "telegram:123"` remains deterministic and should not be rejected.
  - confidence: 0.88
- **[P2] Allow disabling ambiguous legacy jobs:** `src/gateway/server-methods/cron.ts:244-245`
  - body: Including `enabled` in the validation trigger makes `cron.update` validate `{ enabled: false }` against the old ambiguous delivery, and the validator ignores that the patched job is now disabled. That blocks the safest remediation for existing jobs; skip this guard when the resulting job is not schedulable.
  - confidence: 0.86
