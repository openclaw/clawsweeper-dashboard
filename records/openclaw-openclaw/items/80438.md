## Summary

Keep this PR open: it addresses a real source-reproducible cron delivery bug, but the current patch introduces two blocking behavior regressions and still lacks real after-fix proof from a configured multi-channel setup.

## What This Changes

The PR adds gateway cron add/update validation and tests to reject ambiguous implicit announce and failure-destination delivery when multiple message channels are configured.

## Best Possible Solution

Land a narrower admission guard that rejects only enabled or newly schedulable ambiguous announce outcomes, preserves provider-prefixed deterministic routes, and includes real multi-channel cron proof.

## Reproduction Assessment

Yes, source-reproducible. Current main resolves omitted isolated agentTurn delivery to announce channel "last", and multi-channel outbound selection later fails when no deterministic channel can be chosen.

## Solution Assessment

No. The PR targets the right validation boundary, but this implementation over-rejects documented provider-prefixed deterministic routes and blocks disabling ambiguous legacy jobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve provider-prefixed targets when channel is last:** `src/gateway/server-methods/cron.ts:77-78`
  - body: This throws whenever `delivery.channel` is `"last"`, even when `delivery.to` is provider-prefixed. Current main and the cron docs allow `channel="last"` with `to="telegram:123"` to resolve deterministically through the target prefix, so gate the throw on the absence of a deterministic prefixed channel.
  - confidence: 0.88
- **[P2] Allow disabling ambiguous legacy cron jobs:** `src/gateway/server-methods/cron.ts:244-245`
  - body: Including any `enabled` patch in this validation set makes `cron.update` reject `{ enabled: false }` for an existing ambiguous isolated agentTurn job, because the validator ignores that the patched job is disabled. Validate these patches only when the resulting job remains schedulable.
  - confidence: 0.86
