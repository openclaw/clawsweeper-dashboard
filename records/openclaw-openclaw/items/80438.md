## Summary

Keep open: the PR addresses a real current-main cron delivery gap, but the patch introduces a blocking update-path regression and still lacks after-fix real behavior proof from a configured multi-channel Gateway flow.

## What This Changes

The PR adds gateway cron add/update validation that rejects ambiguous implicit `last` announce delivery in multi-channel configs and expands cron validation tests for deterministic routes and failure destinations.

## Best Possible Solution

Keep the gateway admission guard, but allow safe remediation such as disabling legacy ambiguous jobs, preserve deterministic prefixed routes, and add redacted real multi-channel Gateway proof before merge.

## Reproduction Assessment

Yes, source-level. Current main shows omitted isolated/current/session agent-turn delivery can resolve to implicit `last`, and the PR's focused tests model the intended guard path, though I did not execute tests because this review is read-only.

## Solution Assessment

No as written. Admission-time validation is the right seam, but the update guard must allow disabling existing ambiguous jobs and the PR still needs real configured-Gateway behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow disabling legacy ambiguous cron jobs:** `src/gateway/server-methods/cron.ts:246-247`
  - body: The new update guard runs whenever `enabled` is present, then validates the patched job even if the result is `enabled: false`. Existing jobs created before this guard can have implicit `last` announce delivery; with this branch, an operator cannot disable such a job in a multi-channel config because `{ enabled: false }` is rejected before `context.cron.update` clears scheduling state. Gate this validation on the post-patch job being enabled, or otherwise allow disabling while still blocking activation.
  - confidence: 0.88
