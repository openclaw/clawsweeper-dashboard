## Summary

Keep open: the admission-time guard is on the right gateway surface, but the latest head still blocks a safe disable path for legacy ambiguous jobs and the external PR still lacks real configured-Gateway proof.

## What This Changes

The PR adds gateway cron add/update validation that rejects ambiguous implicit `last` announce delivery in multi-channel configs and expands cron validation tests for deterministic routes and failure destinations.

## Best Possible Solution

Keep the gateway admission guard, but gate validation on active or activating jobs so disabling legacy ambiguous jobs remains possible, then add redacted real multi-channel Gateway cron add/update proof before merge.

## Reproduction Assessment

Yes, source-level. Current main lets omitted isolated/current/session `agentTurn` delivery pass admission validation and later resolves it to announce channel `last`; I did not run tests because this was a read-only review.

## Solution Assessment

No as written. Admission-time validation is the right seam, but the PR should allow disabling already-persisted ambiguous jobs and still needs real configured-Gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow disabling legacy ambiguous cron jobs:** `src/gateway/server-methods/cron.ts:256-261`
  - body: The new update guard validates whenever `enabled` is present, then rejects the patched job even when the patch sets `enabled: false`. Existing jobs created before this guard can have implicit `last` announce delivery, so this prevents operators from disabling a bad job even though the service update path would clear scheduling state. Gate this validation on the post-patch job being enabled, or otherwise allow disabling while still blocking activation.
