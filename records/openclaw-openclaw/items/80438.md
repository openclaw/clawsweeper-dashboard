## Summary

Keep open: the Gateway admission guard is aimed at the right cron delivery boundary, but the latest head still exempts any nonempty session key even when no routable session delivery context exists, and the current-head real behavior proof is stale.

## What This Changes

This PR adds Gateway cron add/update validation that rejects ambiguous implicit or `last` announce delivery in multi-channel configs while preserving explicit, provider-prefixed, single-channel, and session-backed routes.

## Best Possible Solution

Tighten the session-backed exception to require an actually routable stored/current delivery context, then refresh current-head multi-channel Gateway proof before maintainers accept the fail-fast compatibility change.

## Reproduction Assessment

Yes. Source inspection shows current main defaults detached cron output to `announce`/`last` and then runtime delivery can fail multi-channel selection when no explicit or stored route exists.

## Solution Assessment

No, not yet. Gateway admission is the right boundary, but the current session-key exception is too broad and the real behavior proof must be refreshed for the latest head.
