## Summary

## What This Changes

The branch adds Gateway cron.add/cron.update validation for ambiguous implicit-last announce routing in multi-channel configs, expands cron validation tests, and changes the Docker E2E heartbeat cleanup helper.

## Best Possible Solution

Cover all detached output payloads at cron.add/update admission, prove or split the Docker helper fix, and get maintainer acceptance of the fail-closed multi-channel compatibility behavior before merge.

## Reproduction Assessment

Yes from source inspection: current main defaults isolated/current/session `agentTurn` and `command` jobs with omitted delivery to announce/last, while PR head only guards omitted `agentTurn` delivery. I did not execute tests in this read-only review.

## Solution Assessment

No: Gateway admission is the right boundary for this mitigation, but the current guard is too narrow for command payloads and the Docker process-group fix is incomplete. A safer version should cover the full detached-output surface and either correctly manage a process group or remove that automation change from this PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include command payloads in implicit delivery detection:** `src/gateway/server-methods/cron.ts:175-180`
  - body: The omitted-delivery branch only treats `agentTurn` as detached output, but current delivery planning defaults both isolated `agentTurn` and `command` jobs without delivery to announce/last. Multi-channel `cron.add` or `cron.update` for an isolated command job can still pass admission and reach the same implicit `last` runtime failure this PR is meant to prevent; include `payload.kind === "command"` and cover add/update cases.
  - confidence: 0.94
- **[P2] Start a process group before signaling one:** `scripts/lib/docker-e2e-logs.sh:58`
  - body: This sends to `-$command_pid`, but the helper still launches the command with plain `"$@" &`, so noninteractive shells do not create a child process group whose id equals `$command_pid`. The group kill therefore falls back to the old single-pid kill and can still leave descendants behind; start the command in its own process group or split this change out.
  - confidence: 0.86
