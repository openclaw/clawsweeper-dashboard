## Summary

Keep open. Current main loads zh-TW UI bundles, but Settings/Form labels are still resolved from English config schema/uiHints metadata outside the i18n lookup path, so this is a real localization gap that needs maintainer direction across schema metadata and generated locale contracts.

## What This Changes

Localize Traditional Chinese settings-page Config Schema form labels and descriptions instead of leaving JSON Schema metadata in English.

## Best Possible Solution

Define a first-class config metadata localization path keyed from English schema labels/help, with generated locale output and English fallback while preserving stable config schema/plugin contracts.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: with the Control UI set to zh-TW, Settings/Form field labels still come from English `uiHints` or schema `title`/`description` rather than i18n keys; I did not run a browser in this read-only review.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation needs maintainer choice between Control UI-side translation by schema key and a protocol/schema metadata localization contract; hand-editing generated zh-TW output is not durable.
