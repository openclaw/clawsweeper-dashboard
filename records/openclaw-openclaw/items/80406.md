## Summary

Keep open: current main still routes the settings form through English config-schema metadata, while Control UI locale strings are handled by a separate i18n map. The related localization threads are closed broad trackers, but this is a concrete remaining config-schema localization gap that needs maintainer direction.

## What This Changes

Request that Control UI settings/config schema field titles and descriptions participate in zh-TW localization instead of staying English from schema metadata.

## Best Possible Solution

Define a first-class localization contract for config schema metadata, including `FIELD_LABELS`, `FIELD_HELP`, section metadata, and plugin/channel `uiHints`, then feed it through the generated Control UI i18n pipeline with zh-TW coverage tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `config.schema` sends English `uiHints`/schema titles and the renderer consumes them without a locale lookup. I did not run a live browser session.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should choose the localization boundary before implementation: client-side translation keys, localized schema payloads, or static override maps all have different plugin and generated-locale implications.
