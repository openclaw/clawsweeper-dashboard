## Summary

Keep open: the bug is source-reproducible on current main, but this PR updates an older run-level helper path while active CLI prompt construction now happens in `prepareCliRunContext`; the external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds session-key-based minimal prompt mode selection to the CLI runner helper, forwards `sessionKey` from `runCliAgent`, moves prompt-mode resolution toward `routing/session-key`, and adds helper tests for cron/subagent heartbeat suppression.

## Best Possible Solution

Derive session-based prompt mode in the current CLI prepare/system-prompt path, preserve `systemPromptOverride`, add prepare-level cron/subagent regression coverage, and collect redacted real CLI-backend proof.

## Reproduction Assessment

Yes, source-reproducible. `createCronPromptExecutor` passes `params.runSessionKey` into `runCliAgent`, while current `prepareCliRunContext` builds the active CLI system prompt without deriving `promptMode` from that key; no live cron run was executed in this read-only review.

## Solution Assessment

No. The intended policy is correct, but the PR updates the old helper/call-site path instead of deriving prompt mode where the active CLI system prompt is prepared.
