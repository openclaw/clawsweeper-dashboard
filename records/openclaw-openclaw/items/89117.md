## Summary

Keep open: current main only supports the preview Gemini Embedding 2 id, and this PR touches the right Google provider boundary, but the patch still sends an upstream-unsupported task-type field on the GA path, leaves public config guidance stale, needs an explicit preview-to-GA upgrade story, and has no real Gemini proof.

## What This Changes

The PR adds the GA `gemini-embedding-2` model to the Google memory embedding provider, maps the preview id to the GA id, widens output-dimensionality validation, and shares normalization with multimodal checks.

## Best Possible Solution

Update the provider, tests, docs, and schema help so GA requests follow Google's current contract, then add maintainer-approved preview-to-GA upgrade behavior and redacted real Gemini proof before merge.

## Reproduction Assessment

Yes for the PR defect: PR-head source shows GA requests still include `taskType`, and the official Gemini docs say `gemini-embedding-2` does not support that parameter. I did not run a credentialed live Gemini request.

## Solution Assessment

No. Adding GA support in the Google provider is the right boundary, but this implementation is not the best fix until it handles the GA task-type contract, docs/help alignment, preview upgrade behavior, and real provider proof.

## Review Findings

Full review comments:

- **[P1] Stop sending taskType with Gemini Embedding 2:** `extensions/google/embedding-provider.ts:77`
  - body: Adding `gemini-embedding-2` to the v2 model set routes GA requests through the existing request helper, which still serializes `taskType` for query and batch calls. Google's current Gemini embeddings docs say `task_type` is not supported for `gemini-embedding-2`; omit it for GA or map task semantics into the documented prompt prefixes before sending the request. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/embeddings?utm_source=openai))
  - confidence: 0.9
- **[P1] Make preview-to-GA remapping upgrade-safe:** `extensions/google/embedding-provider.ts:240-241`
  - body: Normalizing `gemini-embedding-2-preview` to `gemini-embedding-2` changes `client.model`, and the adapter uses that value in `cacheKeyData`, so existing preview configs can hit a new index/cache identity without an explicit migration, alias, or reindex decision. Either preserve a deliberate compatibility identity, add a tested alias when embeddings are equivalent, or document/prove the required reindex behavior.
  - confidence: 0.84
- **[P2] Update public config guidance for GA dimensions:** `extensions/google/embedding-provider.ts:210`
  - body: The code now accepts `gemini-embedding-2` and any integer dimension from 128 through 3072, but the memory config reference and schema help still advertise only the preview id and 768/1536/3072. Update the user-facing docs/help with the implementation so users do not get stale configuration guidance.
  - confidence: 0.82
