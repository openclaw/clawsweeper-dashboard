## What This Changes

The PR changes Telegram fetch fallback classification so EADDRNOTAVAIL and EAFNOSUPPORT local socket failures skip sticky/pinned IP retry, adds message-token parsing for those errnos, and adds focused classifier tests.

## Best Possible Solution

Land the Telegram-local classifier only after real transport proof or maintainer override confirms local-socket errnos should fail fast while remote-reachability and code-less fallback behavior stay unchanged.

## Reproduction Assessment

Yes at source level: current main treats message-only `fetch failed` with no collected `.code` as retryable, so a message-only EADDRNOTAVAIL envelope reaches the existing fallback path. I did not reproduce a live kernel-level local socket outage.

## Solution Assessment

Mostly yes: the fix lives in the Telegram fetch classifier where both API and media retry paths share the decision, and it preserves message-only ECONNREFUSED fallback. The remaining question is proof/acceptance for the fail-fast transport behavior, not a better code layer.
