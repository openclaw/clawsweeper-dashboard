## Summary

Keep open: the linked Telegram fallback bug is still real on current main, but this PR should not merge as-is because the new message parser also changes message-only `EAFNOSUPPORT` from retrying to non-retrying without routing it through the local-failure guard or covering it in tests.

## What This Changes

The branch adds Telegram fetch error-message errno parsing so local socket allocation failures skip pinned-IP fallback while preserving existing remote-reachability fallback cases.

## Best Possible Solution

Scope the message parser to the proven `EADDRNOTAVAIL` case, or explicitly classify and test `EAFNOSUPPORT` before merging the Telegram fallback change.

## Reproduction Assessment

Yes, at source level: current main retries any code-less `fetch failed` envelope, so a message-only `EADDRNOTAVAIL` error follows the misleading pinned-IP fallback path. I did not reproduce a live host-wide socket outage, which would require kernel/network-extension failure conditions.

## Solution Assessment

No, not yet. The `EADDRNOTAVAIL` fix is in the right Telegram fetch classifier, but the patch also changes `EAFNOSUPPORT` behavior without explicit classification or coverage; the narrow fix is to remove that token or make it a fully tested local-failure code.
