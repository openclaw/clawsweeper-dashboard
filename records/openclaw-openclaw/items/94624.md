## Summary

Keep open: the PR targets a real Telegram transport bug and has useful after-fix terminal proof, but the new message-token parser can suppress the existing fallback for message-only `ECONNREFUSED` envelopes because it recognizes a code that is not retry-allowlisted.

## What This Changes

The PR updates Telegram fetch fallback classification to parse errno tokens from error messages, skip pinned-IP fallback for `EADDRNOTAVAIL`, log a local socket-allocation cause, and add focused regression tests.

## Best Possible Solution

Keep the `EADDRNOTAVAIL` fail-fast behavior, but align message-token parsing with the local-socket and retry allowlists so remote fallback behavior is not changed accidentally.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on current `main`, a message-only `fetch failed` envelope with no collected `.code` takes the fallback branch. I did not reproduce the host-wide `EADDRNOTAVAIL` outage live because that requires kernel/network-extension failure conditions.

## Solution Assessment

No as submitted: skipping pinned-IP fallback for `EADDRNOTAVAIL` is the right fix, but the new message parser should not also suppress fallback for non-allowlisted remote codes by accident. The safer solution is to keep parser tokens aligned with explicit local-socket and retry classifications.
