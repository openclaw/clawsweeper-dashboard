## Summary

Keep open. Current main and the latest release still have a source-level path where Discord-visible tool/progress payloads can overtake assistant text segments when block streaming is off; the related open PR only covers the block-streaming-enabled same-channel delivery barrier, not this buffered text path.

## What This Changes

The issue reports Discord responses rendering visible tool output before assistant text segments when one model response alternates text and tool calls.

## Best Possible Solution

Preserve production order for Discord automatic source delivery by routing text blocks and visible tool/progress payloads through one ordered path while keeping block-streaming defaults, progress previews, and message-tool-only suppression intact.

## Reproduction Assessment

Yes, at source level. Current main still buffers text-only block replies when block streaming is off, while visible tool/progress and final inline tool payload paths can reach Discord before the accumulated assistant text.

## Solution Assessment

No, defaulting Discord block streaming on is not the best fix because block streaming has separate fragmentation and UX tradeoffs. The safer repair is a narrow ordered-delivery fix for Discord automatic source delivery without changing global or per-channel defaults.
