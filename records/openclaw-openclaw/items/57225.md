## Summary

Keep open. Current main and v2026.5.22 still preserve the reported source-level path: with block streaming off, Discord-visible text-only blocks are deferred to final payload delivery while tool/progress output can be shown earlier; the merged block-reply barrier fixes an adjacent block-streaming-enabled path, not this disabled-streaming ordering bug.

## What This Changes

The issue asks OpenClaw to preserve Discord production order for assistant text segments interleaved with tool calls instead of delivering tool/progress output before deferred text.

## Best Possible Solution

Preserve production order for Discord automatic source delivery by routing visible text blocks and tool/progress payloads through one ordered delivery path, without changing block-streaming defaults or broader explicit-send policy.

## Reproduction Assessment

Yes, at source level. Current main still defers text-only blocks when block streaming is off, omits the pre-tool block flush callback on that disabled path, and assembles inline tool summaries before deferred answer text; this read-only review did not run live Discord proof.

## Solution Assessment

Yes for a narrow ordered-delivery repair; no to turning Discord block streaming on by default. The safer fix is to preserve production order in the existing delivery path while leaving default and explicit-send policy choices unchanged.
