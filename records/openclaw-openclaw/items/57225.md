## Summary

Keep open. Current main still has a source-level Discord delivery path where text-only assistant blocks are withheld until final delivery when block streaming is off, while visible tool/progress delivery can happen before that final text.

## What This Changes

Discord text segments interleaved with tool calls should be delivered in production order instead of batching the text after visible tool/progress messages.

## Best Possible Solution

Preserve production order for Discord automatic source delivery by sending or coalescing text blocks and visible tool/progress payloads through one ordered path, without globally enabling Discord block streaming or changing message-tool-only guild behavior.

## Reproduction Assessment

Yes, at source level. Current main has a high-confidence path where block streaming is off, text-only block replies stay buffered for final delivery, and visible tool/progress callbacks can run before that final text.

## Solution Assessment

No, defaulting Discord block streaming on is not the best fix. The safer solution is a narrow ordering repair for the affected Discord automatic-delivery path, because https://github.com/openclaw/openclaw/issues/57875 shows block streaming itself has fragmentation tradeoffs.
