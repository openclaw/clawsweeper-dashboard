## Summary

Keep open. Current main still leaves the default Discord path in final-only delivery: text-only block replies are buffered when block streaming is off, while tool/progress messages can still be delivered before the final accumulated text. The reporter's block-streaming workaround remains an opt-in configuration, and the related Discord fragmentation report means a blunt default switch would need care.

## What This Changes

Request that Discord preserve assistant text/tool-call production order for multi-segment responses instead of delivering tool outputs before all buffered text.

## Best Possible Solution

Preserve ordering at the Discord tool-boundary delivery layer without broadly changing global streaming defaults: pre-tool assistant text should be delivered or coalesced before the corresponding tool/progress output, while keeping existing Discord preview/block streaming modes explicitly configurable.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is Discord with streaming/blockStreaming unset, verbose tool/progress delivery enabled, and a model response shaped as text then tool then text; current code buffers text-only blocks until final delivery while tool callbacks can send earlier.

## Solution Assessment

No for the suggested blanket default of Discord block streaming on. A narrower Discord-scoped ordered flush or coalescing path would solve the observed ordering while respecting the documented rate-limit and fragmentation constraints.
