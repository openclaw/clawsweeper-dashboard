## Summary

Keep open. Current main and the latest release still have a source-level Discord path where block-streaming-off text segments are buffered for final delivery while visible tool/progress payloads and inline tool summaries can appear before the accumulated assistant text; the merged block-reply barrier fixes a related block-streaming-enabled path, not this disabled-streaming ordering bug.

## What This Changes

Issue requests preserving Discord-visible order for assistant text segments and tool output when multi-segment responses are interleaved with tool calls.

## Best Possible Solution

Preserve production order for Discord automatic source delivery by routing visible text blocks and tool/progress payloads through one ordered path, while keeping block-streaming defaults and broader explicit-send policy decisions unchanged.

## Reproduction Assessment

Yes, at source level. Current main still buffers text-only block replies when block streaming is off, wires no block flush callback for that disabled path, and then appends inline tool summaries before accumulated assistant text in final payload assembly.

## Solution Assessment

Yes for a narrow ordered-delivery repair; no for defaulting Discord block streaming on. The safer fix is to preserve production order in the existing Discord automatic delivery path without changing global/per-channel defaults or the broader explicit-send policy.
