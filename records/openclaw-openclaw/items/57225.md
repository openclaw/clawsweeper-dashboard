## Summary

Keep open. Current main and the latest release still have a source-level path where Discord text-only blocks can wait for final delivery while tool/progress payloads are sent first; newer progress-preview defaults mitigate some fresh configs but do not fully preserve production order for the reported block-streaming-off path.

## What This Changes

The issue asks OpenClaw to preserve Discord delivery order for assistant text segments interleaved with tool calls.

## Best Possible Solution

Preserve production order for Discord automatic source delivery by routing text blocks and visible tool/progress payloads through one ordered delivery path, while keeping progress drafts and message-tool-only suppression intact.

## Reproduction Assessment

Yes, at source level. Current main still buffers text-only block replies when block streaming is off, while tool/progress callbacks can send visible payloads before final text delivery.

## Solution Assessment

No, defaulting Discord block streaming on is not the best fix because block streaming has separate fragmentation tradeoffs. The safer fix is a narrow ordered-delivery repair for Discord automatic source delivery.
