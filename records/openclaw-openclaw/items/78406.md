## Summary

Keep open: current main still has a source-level Discord deploy-cache path where a secondary bot with the same command set can skip REST reconciliation before its application has any commands.

## What This Changes

Ensure native Discord slash commands deploy for each configured Discord account in multi-bot setups when native commands are enabled.

## Best Possible Solution

Scope the Discord command-deploy hash cache by application/client/account so each bot reconciles its own application commands while preserving same-application cache skips.

## Reproduction Assessment

Yes, source-reproducible: two Discord clients for different applications sharing `commandDeployHashStorePath` and identical command specs will let the first write `global:reconcile` and make the second return before any REST check. I did not run a live Discord deployment.

## Solution Assessment

Yes: the narrow maintainable fix is to include application/client/account identity in the deploy cache key or path, then add a regression test for two applications with the same command set.

## Repair Work Prompt

Fix Discord native slash-command deployment for multi-account bots. Current main uses one `discord/command-deploy-cache.json` and deploy-cache keys such as `global:reconcile` that omit applicationId/accountId, so a second Discord account with the same command set can skip all REST reconciliation and leave its application with zero commands. Keep config semantics unchanged. Scope the cache key or cache path by application/client/account identity, preserve same-application unchanged-deploy skips, and add regression coverage with two clients/app IDs sharing a hash store and identical commands where the second app still performs its first REST reconciliation. Add a user-facing CHANGELOG.md fix entry.
