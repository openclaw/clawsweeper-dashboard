## Summary

Keep open. Current main still lacks the requested configurable gateway/channel silence watchdog keyed to operator-visible delivery; existing mechanisms cover run-state telemetry, transport polling/reconnect watchdogs, model/provider idle timeouts, and outbound delivery diagnostics, but none implement the threshold/reset/repeat/max-beacon contract. The overlapping same-author #47591 remains open, so this should be coordinated rather than closed or split into an automated repair without maintainer defaults.

## What This Changes

The issue requests a configurable gateway/channel silence watchdog that sends runtime beacons during active work when no operator-visible output has been delivered for a configured interval.

## Best Possible Solution

Define a generic, documented runtime/channel watchdog seam tied to actual operator-visible delivery timestamps, with per-channel config/schema/docs, reset/cancel/repeat/max-beacon semantics, focused coverage for streaming and outbound delivery paths, and coordination with #47591’s delivery-confirmation design.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence check is source and docs inspection on current main, which shows adjacent liveness mechanisms exist but the requested watchdog contract is absent.

## Solution Assessment

Unclear until maintainer approval. The maintainable direction is a shared channel/runtime seam keyed to actual visible delivery, not per-channel ad hoc timers or model prompt rules.
