## Summary

Keep open. Current main still has plugin-only post-core update handling and no user-facing post-update script or hook contract; the latest reporter comment adds production evidence and explicitly defers implementation until the updater-time local-script security contract is defined.

## What This Changes

Add a user-facing `openclaw update` post-update hook that can run configured scripts after plugin sync with update context, timeout, JSON reporting, and non-fatal failure handling.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection gives a high-confidence current-main check that only plugin post-update processing exists and no user-facing update hook is exposed.

## Solution Assessment

Unclear: a post-update hook is a plausible direction, but maintainers need to decide timing, failure behavior, environment exposure, timeout, logging/redaction, and restart interactions before code is added.
