## Summary

Keep open. Current main still has only internal plugin post-update processing and documented Gateway/plugin lifecycle hooks; a user-facing post-update script hook would be a new updater contract with product, security, timeout, and restart-order decisions.

## What This Changes

Add a user-facing post-update hook for `openclaw update` that can run local scripts after plugin sync, receive update context, and report script results without failing the update.

## Best Possible Solution

Define or explicitly decline a supported update lifecycle hook with schema, docs, tests, security defaults, and coverage across CLI updates, package-manager updates, Gateway auto-updates, plugin sync, and restart sequencing.

## Solution Assessment

## Risks / Open Questions

- The desired contract is still a product/security decision: timing before or after plugin sync/restart, environment, timeout, failure policy, JSON shape, and opt-in semantics are all unresolved.
- Running arbitrary local scripts from the updater has concrete security and reliability implications because it would execute around package replacement and managed-service restart with the user's privileges.
- Adjacent open work on plugin convergence and plugin-updated hooks may change nearby update/plugin lifecycle code, so any design should coordinate with those surfaces rather than bolt on another post-core branch.
