## Summary

Keep open. Current main still has plugin-only post-update processing and no user-facing update hook schema, result shape, or documented event; the reporter also confirmed the production need and explicitly deferred any PR until the local-script security contract is designed.

## What This Changes

Add a user-facing post-update hook that can run local scripts after `openclaw update`, receive update context, report exit codes, respect a timeout, and warn rather than fail on script errors.

## Best Possible Solution

Define or explicitly reject a supported post-update lifecycle hook with schema, docs, tests, update-result typing, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection give a high-confidence verification path that current main has internal plugin post-update handling but no user-facing post-update hook.

## Solution Assessment

Unclear: a post-update hook is plausible, but the reporter and code evidence show the trust boundary, timing, output, and failure contract need maintainer design before implementation. Adding a directory convention or config key now would be premature without that contract.
