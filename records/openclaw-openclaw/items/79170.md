## Summary

Keep open. Current main has a stronger plugin-only post-core convergence path after the recent merged updater PR, but it still does not expose a user-facing post-update hook, script result shape, timeout/config surface, or security contract for running local code during update.

## What This Changes

The issue requests a user-facing post-update hook for running local scripts after `openclaw update` completes.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, JSON output semantics, and security rules across CLI package-manager/git updates and Gateway restart flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source, docs, live issue discussion, and the related merged updater PR give a high-confidence verification path that current main still has plugin-only post-update work and no user-facing script hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure semantics, timeout, declaration authority, environment exposure, output redaction, and restart interactions before code is added.
