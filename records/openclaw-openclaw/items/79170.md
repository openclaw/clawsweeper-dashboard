## Summary

Keep open. Current main still has plugin-only post-update handling and no documented update-complete hook, while the request would add a new updater contract for local script execution, ordering, timeout, restart, and JSON reporting semantics.

## What This Changes

The issue requests a user-facing post-update hook for `openclaw update`, either through config or a hook directory, so local patch/setup scripts can run automatically after updates.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, fresh-process context handling, bounded timeout/failure behavior, and security guidance across CLI and Gateway update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection confirms current main has internal plugin post-update work but no user-facing post-update/update-complete hook.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide timing, environment, timeout, failure behavior, JSON output, and restart interactions before implementation.
