## Summary

Keep open. Current main still has only internal plugin post-update processing and Gateway lifecycle hooks; a user-facing update hook would be a new updater contract with security and restart-order design choices.

## What This Changes

Request a user-facing post-update script hook or directory convention that runs after `openclaw update` so local patch/setup scripts can be reapplied automatically.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, and tests for CLI, package-manager, and Gateway auto-update flows, including security defaults and restart ordering.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection confirms current main has no user-facing post-update hook.

## Solution Assessment

Unclear: a config key or directory convention is plausible, but maintainers need to decide lifecycle timing, failure behavior, environment, timeout, permissions, and interaction with package replacement, plugin sync, and restart.
