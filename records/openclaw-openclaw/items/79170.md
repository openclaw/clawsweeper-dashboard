## Summary

Keep open. Current main still has only internal plugin post-update handling; a user-facing post-update script hook would add a new updater contract with restart ordering, timeout, JSON reporting, and local-script security semantics.

## What This Changes

The issue requests a user-facing post-update hook that can run local scripts after `openclaw update` and report their results.

## Best Possible Solution

Keep this open until maintainers define or reject a supported update lifecycle hook, then implement it with schema/types, docs, tests, JSON output, and security semantics across CLI and Gateway update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection on current main confirms the requested user-facing post-update hook surface is absent.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure behavior, environment, timeout, permissions, output redaction, and restart interactions before implementation.
