## Summary

Keep this open: current main and v2026.6.6 still only expose internal post-core plugin convergence, not a user-facing post-update hook, and the remaining work needs a maintainer-owned product and security contract before implementation.

## What This Changes

Request a user-facing post-update hook that can run local operator scripts after `openclaw update` completes and report hook outcomes without failing the update by default.

## Best Possible Solution

Define one explicit, security-reviewed post-update extension contract for update finalization, then implement it with schema/types, JSON output, docs, and tests covering environment, timeout, output redaction, failure handling, and restart behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give high-confidence proof of the capability gap: OpenClaw has internal post-core plugin convergence but no user-facing post-update hook.

## Solution Assessment

No. The proposed config key or directory convention is only a plausible shape; the best path is a maintainer-designed update lifecycle contract that defines the security and failure semantics first.

## Risks / Open Questions

- Executing arbitrary local scripts during or immediately after a package swap could expose credentials, process environment, working directories, stdout/stderr, or partially swapped install state without a defined trust and redaction contract.
- A direct config key or directory convention could create a second hook surface without deciding whether update finalization, internal hooks, plugin hooks, or an operator workflow owns the behavior.
- Failure behavior is still a product decision: warn-and-continue, block restart, require repair, or rollback all have different availability and operator-safety consequences.
