## Summary

Keep open. Current main still has internal plugin-only post-update handling and no user-facing post-update/update-complete hook; the requested behavior would add a new updater and local script execution contract rather than fix broken existing behavior.

## What This Changes

Request a user-facing post-update hook for `openclaw update` that can run configured scripts after plugin sync, report script results, respect timeouts, and warn without failing the update.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection gives a high-confidence verification path that current main has plugin-only post-update work and no user-facing update lifecycle hook.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure behavior, environment, timeout, permissions, and restart interactions before implementation.
