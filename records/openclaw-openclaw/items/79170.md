## Summary

Keep this issue open: current main and v2026.6.6 still expose only internal post-core plugin convergence, not a user-facing post-update hook, and the requested script surface needs a maintainer-owned security/product contract before implementation.

## What This Changes

Request a user-facing post-update hook or directory convention so local update follow-up scripts can run after `openclaw update`.

## Best Possible Solution

Define one explicit post-update extension contract for update finalization, including declaration owner, environment allowlist, timeout, output capture/redaction, failure semantics, JSON shape, docs, and tests before implementing the hook.

## Reproduction Assessment

Not applicable as a bug reproduction. The capability gap is high-confidence from current source, docs, config schema, and the latest release: there is internal plugin post-update convergence but no user-facing post-update hook.

## Solution Assessment

No, the proposed config key or directory convention is only a plausible shape. The best path is a maintainer-designed update lifecycle contract that composes with existing hook/plugin/update boundaries and defines security behavior first.

## Risks / Open Questions

- Executing arbitrary local scripts during or immediately after a package swap can expose secrets, mutate the installation, or restart services without a defined trust and redaction contract.
- A direct `hooks.postUpdate` or directory convention could add another config/runtime surface without deciding whether this belongs in update finalization, internal hooks, plugin hooks, or a narrower operator workflow.
- Failure handling is product-critical: warning, halting restart, rolling back, and JSON reporting have different operator and availability consequences.
