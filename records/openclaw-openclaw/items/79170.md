## Summary

Keep open. Current main still has internal plugin-only post-update handling, while the requested user-facing `openclaw update` hook would add a new updater, restart, JSON-output, timeout, and local-script security contract that maintainers have not defined yet.

## What This Changes

Request a user-facing post-update hook for `openclaw update` that can run local scripts after plugin sync, receive update context, report script results, and warn rather than fail on hook errors.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI update, package-manager/fresh-process resume, plugin sync, and Gateway restart flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection on current main show internal plugin post-update work but no user-facing update lifecycle hook.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide timing, failure behavior, environment exposure, output redaction, timeout, and restart interactions before implementation.
