## Summary

Keep this open: current main and v2026.6.6 still expose only internal post-core/plugin convergence, while the requested user-facing post-update hook remains missing and needs a maintainer-owned security contract before implementation.

## What This Changes

Request a user-facing post-update hook or hook directory that runs after `openclaw update`, receives update context, reports script results, honors timeouts, and does not fail the update on script failure.

## Best Possible Solution

Define one security-reviewed update-finalization extension contract, then implement the chosen surface with schema/types, JSON output, docs, and tests for environment, timeout, output redaction, failure handling, and restart behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give high-confidence proof of the capability gap: OpenClaw has internal post-core plugin convergence but no user-facing post-update hook.

## Solution Assessment

No, the proposed config key or directory convention is only a plausible implementation shape. The best path is a maintainer-designed update lifecycle contract that defines the security and failure semantics first.
