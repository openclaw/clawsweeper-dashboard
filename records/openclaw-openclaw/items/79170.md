## Summary

Keep open. Current main still has internal plugin-only post-update work, but no user-facing post-update hook; the requested local script/config surface would add a new updater lifecycle and a security contract that maintainers have not designed yet.

## What This Changes

Request a user-facing post-update hook that can run local scripts after `openclaw update`, receive update context, report exit codes, and respect timeout/failure policy.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across CLI package-manager/git updates and Gateway auto-update/restart flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source, docs, types, and live issue inspection provide a high-confidence verification path that current main has internal plugin post-update work but no user-facing post-update hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure handling, timeout, declaration authority, environment exposure, output redaction, restart interactions, and JSON reporting before code is added.
