## Summary

Keep open. Current main still has internal plugin-only post-update handling, while a user-facing post-update hook would add a new updater lifecycle and local-script security contract that has not been designed or implemented.

## What This Changes

Request a user-facing `openclaw update` post-update hook that can run local scripts after plugin sync, receive update context, report script results, respect timeouts, and warn rather than fail on script errors.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across shell update, package-manager update, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection provide a high-confidence verification path that current main has internal plugin post-update work but no user-facing post-update hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure handling, timeout, permissions, environment, output redaction, and restart interactions before code is added.
