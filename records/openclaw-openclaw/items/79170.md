## Summary

Keep open. Current main now has stronger internal plugin post-core convergence, but it still exposes no user-facing post-update hook, script result contract, config/schema surface, or security policy for arbitrary local code during updates.

## What This Changes

Add a supported post-update hook or lifecycle script surface after `openclaw update` so user tasks can run with update-result context, timeout handling, warnings, and JSON exit-code reporting.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across CLI package-manager/git updates plus Gateway restart and auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source, docs, types, related issue search, and live discussion provide a high-confidence verification path that current main has internal plugin post-update work but no user-facing post-update hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure handling, timeout, declaration authority, environment exposure, output redaction, and restart interactions before code is added.
