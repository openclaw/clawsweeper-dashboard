## Summary

Keep open. Current main still has internal plugin-only post-update handling, while the requested user-facing hook would add a new updater lifecycle, JSON/config surface, restart-order contract, and local-script security policy.

## What This Changes

Request a user-facing post-update hook for `openclaw update` so local scripts can run after core/plugin update work and report bounded results.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.
