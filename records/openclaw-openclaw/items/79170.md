## Summary

Keep open. Current main has strengthened internal plugin-only post-update convergence, but it still has no user-facing post-update hook, no script execution result shape, and no security contract for running operator-local code during update.

## What This Changes

The issue requests a user-facing post-update hook or hook directory that runs after `openclaw update` plugin sync and reports script outcomes without failing the update.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across package-manager, git, CLI, and Gateway restart update flows.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source, docs, tests, and live discussion give a high-confidence verification path that only internal plugin post-update handling exists today.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide trust boundaries, timing, failure semantics, redaction, and restart behavior before code is added.
