## Summary

Keep open. Current main still exposes only internal plugin post-update handling, and the latest reporter follow-up confirms this is a deliberate user-facing updater feature that needs security and lifecycle design before implementation.

## What This Changes

The issue requests a user-facing post-update hook for `openclaw update`, with update context, timeout handling, warning-on-failure semantics, and JSON result reporting.

## Best Possible Solution

Define or reject a supported update lifecycle extension contract, with schema, docs, tests, security rules, JSON output shape, and clear ordering across plugin sync and gateway restart/re-enable flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection confirm current main has internal plugin-only post-update handling and no user-facing post-update hook surface.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to settle lifecycle timing, failure behavior, timeout, environment exposure, output capture/redaction, and gateway restart interactions first.
