## Summary

Keep open. Current main still has internal plugin-only post-update handling, while a user-facing update hook would add a new updater contract with restart-order, timeout, JSON reporting, and local-script security decisions.

## What This Changes

Request a user-facing post-update hook so local patch or setup scripts can run after `openclaw update` with result context, timeout handling, and non-fatal reporting.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection confirms current main has internal plugin post-update work and no user-facing update lifecycle hook.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure behavior, environment, timeout, permissions, and restart interactions before implementation.
