## Summary

Keep open. Current main still only runs post-core plugin sync and reports `postUpdate.plugins`, while a user-facing updater hook would add new lifecycle, timeout, JSON, restart, and local-script security contracts.

## What This Changes

The request asks for a user-facing post-update hook or script directory that runs after `openclaw update` so local patches or custom tasks can be reapplied automatically.

## Best Possible Solution

Define or reject a supported update lifecycle hook with schema, docs, tests, JSON output, restart/fresh-process handling, and security semantics across CLI and Gateway update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection show internal plugin post-update work exists, but no user-facing update lifecycle hook is currently exposed.

## Solution Assessment

Unclear: the requested config entry or directory convention is plausible, but maintainers need to decide the updater lifecycle and local-script security contract before implementation.
