## Summary

Keep open. Current main still has internal plugin-only post-update processing, while the requested user-facing post-update hook would add a new updater, hook, JSON-reporting, timeout, and local-script security contract.

## What This Changes

Request a user-facing `openclaw update` post-update hook that can run local scripts after plugin sync and report script outcomes in update JSON output.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior. Source and docs inspection on current main confirms plugin-only post-update processing and no emitted update-complete hook.

## Solution Assessment

Unclear: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure behavior, environment, timeout, permissions, and JSON reporting before implementation.
