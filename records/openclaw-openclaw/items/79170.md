## Summary

Keep open. Current main has internal post-update doctor, plugin sync, completion refresh, and restart handling, but no user-facing post-update hook; adding one would be a new update-lifecycle execution surface that needs maintainer product and security design.

## What This Changes

The issue asks for a user-facing post-update hook or scripts directory so local patches and custom setup tasks can run automatically after `openclaw update`.

## Best Possible Solution

If maintainers want this capability, design a narrow opt-in updater extension point with schema/help/docs, bounded execution, clear before-restart ordering, and regression coverage across package and git update flows.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection do give high confidence that current main lacks a user-facing post-update hook or `post-update.d` contract.

## Solution Assessment

Unclear whether the proposed config key or directory convention is the best final shape. The maintainable path is a maintainer-approved design for a trusted, opt-in updater hook with explicit failure and security semantics.
