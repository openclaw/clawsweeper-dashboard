## Summary

Keep open. Current main still has internal plugin-only post-update handling, while the requested user-facing hook would add a new updater lifecycle and local-script execution contract that needs maintainer product and security design.

## What This Changes

The issue requests a user-facing post-update hook system for `openclaw update`, with script execution after plugin sync, update context, timeout behavior, warning semantics, and JSON output.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema, docs, tests, and security semantics across CLI, package-manager, and Gateway auto-update flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source and docs inspection gives a high-confidence verification path that current main has internal plugin post-update work but no user-facing update lifecycle hook.

## Solution Assessment

Unclear: a config entry or directory convention could solve the user problem, but maintainers first need to decide lifecycle timing, script trust, environment, timeout, output, failure, and restart semantics.
