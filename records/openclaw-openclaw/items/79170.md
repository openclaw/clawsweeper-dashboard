## Summary

Keep open. Current main still has internal plugin-only post-update handling, while the requested user-facing post-update hook would add a new updater lifecycle, config/script surface, and local-code execution security contract that is not implemented or designed yet.

## What This Changes

Request a user-facing post-update hook for `openclaw update` that runs local scripts after plugin sync with update context, timeout, warning-only failures, and JSON exit-code reporting.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across CLI package-manager/git updates and Gateway auto-update/restart flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source, docs, types, and live issue inspection provide a high-confidence verification path that current main has internal plugin post-update work but no user-facing post-update hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure handling, timeout, declaration authority, environment exposure, output redaction, and restart interactions before code is added.
