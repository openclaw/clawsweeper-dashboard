## Summary

Current main and v2026.6.6 still support only static binding selectors, while the related generic pre-routing hook remains open and its earlier PR closed unmerged. This issue should remain open for a maintainer product/API decision; likely related people are @steipete for routing/product review, @Squirbie for the binding contract, and @vignesh07 for resolver caching.

## What This Changes

Allow bindings to route inbound messages to different agents based on content patterns, optionally removing the matched prefix before agent dispatch.

## Best Possible Solution

Define a generic typed pre-routing plugin contract that receives normalized inbound content before route and session selection, returns a deterministic route decision, and scopes any body transformation explicitly; then implement content-prefix routing as plugin-owned policy or a thin documented binding surface if maintainers want it in core.

## Reproduction Assessment

Yes, at high confidence from source inspection: current main has no content selector in the strict binding schema and no message-body input in route resolution. This establishes the feature gap, though it is not a regression in an existing contract.

## Solution Assessment

No, the proposed binding-specific regex and stripping shape is not yet proven to be the best solution. A generic pre-routing plugin seam better matches the lean-core architecture, but maintainers must first define its precedence, safety, cache, and body-rewrite semantics.

## Risks / Open Questions

- A binding-only implementation could duplicate policy better owned by a generic pre-routing plugin contract and create inconsistent behavior across channel call sites.
- Regex matching needs explicit safety limits and deterministic normalization rules for captions, commands, mentions, attachments, and non-text messages.
- `stripMatch` needs a clear contract for whether it changes only the agent-facing prompt or also command parsing, raw body, reply context, logging, and hook payloads.
- Content-dependent routing must deliberately bypass or extend the current static route cache and define precedence against peer, parent-peer, role, mention, account, and channel routing.
