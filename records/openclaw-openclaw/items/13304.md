## Label Justifications

- `P2`: This is a normal-priority multi-agent routing capability request with real workflow impact but no current regression or emergency.
- `impact:session-state`: The requested matcher decides which agent/session receives a message, so absent support can put conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents can be routed through the default dispatcher workaround instead of reaching the intended handler directly.

## Summary

Keep open. Current main still exposes only static binding selectors, strict config validation does not admit `bindings[].match.content`, and the related pre-route hook work is an open adjacent API path rather than an implemented declarative binding matcher.

## What This Changes

The issue requests declarative content-based route binding matches so message prefixes or regexes can choose an agent and optionally strip the matched text before dispatch.

## Best Possible Solution

Settle the product/API direction first: either add a first-class validated `bindings[].match.content` contract with deterministic precedence, cache behavior, channel text plumbing, and body rewrite semantics, or land a generic pre-route hook and document the plugin/ClawHub implementation path.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection on current main shows `match.content`, content-aware route input, and rewritten body output are absent.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but the maintainable solution needs maintainer decisions on precedence, normalization, regex safety, route-cache behavior, and body-rewrite semantics; the related pre-route hook may also become the better extension point.
