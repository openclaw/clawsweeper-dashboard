## Summary

Keep open. Current main still lacks declarative content matching in agent bindings: the binding type and strict schema only accept static selectors, the resolver input/output and cache are static-route based, and related pre-routing hook work is open/separate rather than a shipped replacement for binding rules.

## What This Changes

The issue requests declarative regex or prefix content matchers in agent bindings so inbound messages can route directly to different agents and optionally strip the matched prefix.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe matching rules, content-aware cache behavior, representative channel text plumbing, explicit body-stripping semantics, and config/routing/docs tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request. Source inspection of current main gives high confidence that match.content, content-aware route input, and rewritten body output are absent.

## Solution Assessment

Unclear. Declarative match.content is plausible, but the maintainable solution needs maintainer API/product decisions on precedence, normalization, regex safety, route-cache behavior, and body-rewrite semantics before implementation.
