## Label Justifications

- `P2`: This is a valid multi-agent routing improvement with meaningful but non-urgent user impact and a broad API design surface.
- `impact:message-loss`: Without the requested routing decision, an inbound message can be handled by a different agent than the user intended.
- `impact:session-state`: Agent routing selects the isolated session and context that receive and retain the inbound turn.

## Summary

The requested capability is still absent on current main, and neither the closed unmerged pre-route PR nor the existing inbound hooks provide same-turn agent-route override. Keep this open for a maintainer product/API decision, preferably coordinated with the broader pre-routing hook request rather than implemented as an isolated regex field.

## What This Changes

Allow agent bindings to match inbound message content by pattern or prefix and optionally strip the matched text before agent processing.

## Best Possible Solution

Define a typed, deterministic pre-routing SDK contract that receives normalized inbound content and can return a same-turn agent/session decision; implement regex or prefix policy and optional agent-facing body transformation in a plugin unless maintainers explicitly decide this behavior is common enough for native bindings.

## Reproduction Assessment

Yes from source inspection: current bindings reject the proposed selector, the route resolver has no message-content input, and the content-aware inbound hook already has a canonical session and cannot override it. This confirms a feature gap rather than reproducing a broken existing contract.

## Solution Assessment

No definitive implementation is yet the best solution; adding regex and stripping directly to bindings would spread optional policy across core routing and channels, while a generic pre-route plugin seam better matches the repository architecture.

## Risks / Open Questions

- Native content matching would expand the public configuration surface and require explicit precedence against peer, parent-peer, role, account, channel, mention, and default routing.
- Regex safety, normalized channel text, route-cache behavior, and whether stripping changes only `BodyForAgent` or other command/reply/transcript fields remain unresolved.
- Implementing only the binding field would be one-sided because many channel/plugin callers resolve routes before their final agent-facing message body is prepared.
