## Label Justifications

- `P2`: This is a useful multi-agent routing capability with clear workflow value, but it is not a regression, emergency, or currently broken documented behavior.
- `impact:session-state`: The requested matcher determines which agent and session receives a message, so missing support can put conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents currently need a dispatcher workaround instead of directly reaching the intended handler.

## Summary

Keep open. Current `main` and the latest release still only support static binding selectors; the related pre-route-hook work is open and may inform the design, but it has not shipped and does not settle declarative binding content matchers.

## What This Changes

The issue requests a new `bindings[].match.content` matcher, with optional prefix/regex matching and matched-text stripping, so inbound messages can route to different agents by message text.

## Best Possible Solution

Settle the product/API direction first, then implement either a validated binding content matcher or a documented pre-route plugin path with deterministic precedence, safe matching, channel text plumbing, cache behavior, body rewrite semantics, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current `main` and v2026.5.20.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but maintainers need to decide precedence, normalization, regex safety, cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.

## Risks / Open Questions

- Maintainers still need to choose whether this belongs as declarative `bindings[].match.content`, a generic pre-route plugin hook, or both.
- `stripMatch` needs explicit semantics for agent-facing text, raw body, command handling, hook payloads, reply context, transcripts, and audit metadata.
- Regex/prefix matching needs normalization, length limits, and safe-regex policy before becoming a public config surface.
- Content-dependent routing conflicts with the current static route cache unless content rules bypass caching or the cache contract becomes content-aware.
