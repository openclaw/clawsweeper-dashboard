## Summary

Keep open. Current main still exposes only static binding selectors, strict validation rejects the proposed content matcher, representative routing call sites do not pass message text, and maintainer discussion already identified unresolved API/product decisions.

## What This Changes

The issue requests optional binding content matchers so inbound messages can route to different agents by regex or prefix and optionally strip the matched text before delivery.

## Best Possible Solution

Define a validated binding content matcher in the public config contract, specify deterministic precedence and regex limits, make routing cache/body flow content-aware, update representative channel/plugin paths, and document stripMatch semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed match.content config: strict validation has no such key, and route resolution cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but the exact design needs maintainer decisions on precedence, regex safety, cache behavior, normalization, and body rewriting before it is the safest implementation path.
