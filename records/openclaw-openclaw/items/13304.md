## Summary

Keep open: current main still only supports static binding selectors, and content-based routing would require new public config, resolver/cache behavior, and channel/plugin body-forwarding semantics. The related reports cover model routing or narrower mention routing, so they do not supersede this broader binding-level feature.

## What This Changes

Request adding a binding `content` matcher that can route messages to different agents by regex or prefix and optionally strip the matched text before agent delivery.

## Best Possible Solution

Define a validated binding content-matcher contract with deterministic precedence, safe pattern handling, content-aware cache behavior or bypass, normalized inbound-text propagation, documented stripped-body semantics, and focused config/resolver/channel coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability rather than broken behavior under an existing contract. Source inspection on current main shows the binding schema and resolver lack the requested content matcher and body-rewrite path.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but maintainers should first decide the public contract, precedence, cache behavior, regex safety, and stripping semantics versus a pre-route hook or narrower mention-routing design.
