## Summary

Keep open. Current main still lacks this feature: binding types, schema, docs, resolver inputs, route cache, and representative channel callers are static-only, while recent discussion already identifies product/API decisions still needed.

## What This Changes

The issue requests optional binding content matchers so inbound messages can route to different agents by regex or prefix and optionally strip the matched text before delivery.

## Best Possible Solution

Add a first-class validated binding content matcher only after maintainers settle precedence, regex limits, cache/body contracts, and channel/plugin plumbing, then document the behavior with regression coverage.

## Reproduction Assessment

Yes for the current limitation. Adding the proposed match.content object is source-reproducible as unsupported because strict config validation has no such key and resolveAgentRoute cannot receive message text.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but the proposed shape is not yet the safest implementation path until maintainers decide precedence, regex safety, cache behavior, normalization, and body rewriting.
