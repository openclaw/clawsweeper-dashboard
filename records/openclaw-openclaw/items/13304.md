## Label Justifications

- `P2`: This is a useful normal-priority routing capability with bounded user impact but cross-cutting API and product decisions.
- `impact:message-loss`: Content routing determines which agent receives an inbound message, so incorrect matching or precedence can misroute delivery.
- `impact:session-state`: The selected agent determines which session and conversational context own and persist the inbound turn.

## Summary

Current main still cannot route an inbound message by its content: bindings and route resolution remain metadata-only, while existing plugin hooks cannot redirect ordinary processing to another agent in the same turn. The broader pre-routing hook request remains open and may provide the right extension boundary, but it does not yet supersede this distinct declarative-routing request.

## What This Changes

Add content-pattern matching and optional matched-prefix stripping to agent bindings so inbound messages can select specialist agents.

## Best Possible Solution

Define a typed, deterministic pre-routing plugin contract that receives normalized inbound content and can select the current turn’s agent/session, implement pattern policy externally first, and add core binding syntax only if maintainers decide it is universal policy.

## Reproduction Assessment

Yes at source level: current main’s validated binding and resolver contracts have no content input or matcher, so the requested route cannot be configured. This is a capability gap rather than broken existing behavior.

## Solution Assessment

No, the proposed core regex-and-strip configuration is not yet the best proven solution; a generic plugin-owned pre-route contract better matches repository boundaries and avoids hard-coding optional policy.
