## Label Justifications

- `P2`: This is a useful multi-agent routing capability request with clear workflow value but no current regression or emergency.
- `impact:session-state`: The requested matcher determines which agent/session receives a message, so absent support can put conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents must currently be routed through a dispatcher workaround instead of directly reaching the intended handler.

## Summary

Keep open. Current main and the latest release still expose only static binding match fields, route resolution/cache do not carry message text or rewritten bodies, and the open pre-routing-hook work is adjacent rather than a shipped binding-level solution.

## What This Changes

The issue requests content-based bindings that route inbound messages to different agents using message prefixes or regex patterns and optionally strip matched text before dispatch.

## Best Possible Solution

Settle the product/API direction first, then add a validated binding content matcher or documented pre-route plugin path with deterministic precedence, safe matching, channel text plumbing, cache behavior, body rewrite semantics, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but maintainers need to decide precedence, normalization, regex safety, cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.

## Risks / Open Questions

- The public API direction is unsettled: maintainers need to choose a declarative `bindings[].match.content` contract, a generic pre-route plugin hook, or both.
- `stripMatch` needs explicit semantics for agent-facing text, raw body, command handling, transcripts, hook payloads, reply context, and audit metadata.
- Regex or prefix matching needs normalization, escaping, length limits, and safe-regex policy before becoming a public config surface.
- Content-dependent routing conflicts with the current static route cache unless caching is bypassed or keyed by normalized content for content rules.
