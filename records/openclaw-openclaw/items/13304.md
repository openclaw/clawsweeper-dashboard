## Label Justifications

- `P2`: This is a normal-priority routing capability request with real multi-agent impact but it requires product/API design before implementation.
- `impact:session-state`: The requested behavior decides which agent and session receives a message, so unresolved routing can put conversation state in the wrong agent scope.
- `impact:message-loss`: Without content-based routing, messages intended for specialist agents can be misrouted through a default dispatcher or fail to reach the intended handler directly.

## Summary

Keep open. Current main still only supports static binding selectors, and the related hook/model-routing work does not provide the requested declarative `bindings[].match.content` agent-routing contract.

## What This Changes

Request to add optional content or pattern matching to bindings so inbound messages can route to different agents and optionally strip matched prefixes before dispatch.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe matching rules, content-aware cache behavior, representative channel text plumbing, explicit body-stripping semantics, and config/routing/docs tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but the maintainable solution needs maintainer decisions on precedence, normalization, regex safety, route-cache behavior, and body-rewrite semantics before implementation.
