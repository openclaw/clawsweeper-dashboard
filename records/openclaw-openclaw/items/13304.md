## Label Justifications

- `P2`: This is a useful cross-channel routing enhancement with limited immediate urgency but meaningful API design scope.
- `impact:message-loss`: A content-routing decision can send an inbound message to the wrong agent and prevent the intended specialist from handling it.
- `impact:session-state`: Selecting a different agent changes the session and transcript that receive and retain the inbound turn.

## Summary

Current main still lacks content-aware binding selection, and the existing early plugin hook can inspect content but only claim/handle the event, not redirect it to another agent or session. The broader open pre-route hook request overlaps but is not an exact duplicate because it proposes stronger redirect and suppression authority; this issue should stay open for a maintainer product/API decision on the narrowest safe routing contract.

## What This Changes

Add content pattern matching and optional matched-prefix stripping to agent binding rules so inbound messages can select specialist agents by message text.

## Best Possible Solution

Define one typed, generic same-turn route-decision plugin seam that receives normalized inbound text and can select a validated agent/session target, with deterministic precedence, timeout/replay behavior, and explicit agent-facing body rewrite semantics; implement optional pattern routing in a plugin rather than adding regex policy to static core bindings.

## Reproduction Assessment

Yes at source level: current main’s strict binding schema has no content matcher, and the resolver receives no message text; the existing early claim hook cannot return a route override. This is a missing feature rather than a failing established behavior.

## Solution Assessment

No, the proposed direct regex matcher in core bindings is not yet the best maintainable solution. A narrow generic plugin route-decision contract better matches lean-core ownership and avoids duplicating content policy through channel routing paths.

- The broader pre-route proposal adds arbitrary session redirect and suppression authority, so merging the two requests without narrowing the contract could enlarge the plugin security boundary unnecessarily.
