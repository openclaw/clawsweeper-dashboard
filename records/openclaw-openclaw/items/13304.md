## Summary

Keep open. Current main still has only static binding selectors, strict config validation would reject a `match.content` object, and the resolver/channel call sites do not carry message text or return a stripped body; related hook work is separate and not a shipped replacement for declarative binding rules.

## What This Changes

Request to let `bindings` route inbound messages by message-content patterns or prefixes, optionally stripping the matched prefix before dispatching to the selected agent.

## Best Possible Solution

Define a first-class validated binding content matcher with documented precedence, safe matching rules, content-aware cache behavior, representative channel text plumbing, explicit body-stripping semantics, and focused config/routing tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request. Source inspection of current main and `v2026.5.12` gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent.

## Solution Assessment

Unclear. Declarative `match.content` is plausible, but the maintainable solution needs maintainer decisions on precedence, normalization, regex safety, route-cache behavior, and body-stripping semantics before implementation.
