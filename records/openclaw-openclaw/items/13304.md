- `impact:session-state`: The requested routing decision changes which agent session and context receive an inbound turn.
- `impact:message-loss`: Without content-aware routing, messages can be handled by the wrong specialist path and miss the intended agent workflow.

## Summary

Keep open: current main still has only static binding selectors, strict config validation rejects a content matcher, and the related pre-route hook work overlaps but does not implement declarative binding content matching or stripMatch semantics.

## What This Changes

The issue requests content-pattern matching in agent bindings so inbound messages can route to different agents and optionally strip matched prefixes before delivery.

## Best Possible Solution

Define one maintainer-approved routing API that covers content-aware agent selection, deterministic precedence, route-cache behavior, safe matching rules, and documented body-stripping semantics, or explicitly route this use case through a pre-route plugin hook once that API exists.

## Reproduction Assessment

Yes at source level: current main lacks a binding content matcher in the type/schema, resolver input, route cache, and tier selection. I did not run a live channel repro because this is a missing feature rather than a broken existing behavior contract.

## Solution Assessment

Unclear: the requested binding matcher is plausible, but VISION.md favors plugin seams for optional capability and the linked pre-route hook discussion may be the better owner boundary for some use cases.
