## Summary

Keep open. Current main still validates and resolves bindings from static channel/account/peer/guild/team/roles metadata only, and the related mention/pre-routing work does not provide the requested declarative content matcher or `stripMatch` behavior.

## What This Changes

Request to add declarative `bindings[].match.content` pattern or prefix matching with optional prefix stripping so inbound messages can route to different agents by message text.

## Best Possible Solution

Define a first-class validated binding content matcher with documented precedence, safe matching rules, cache behavior, representative channel text plumbing, optional body-stripping semantics, and focused config/routing tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config feature request. Source inspection gives high confidence current main lacks content match fields, content-aware route input, and body rewrite output.

## Solution Assessment

Unclear. A declarative `match.content` option may be a maintainable direction, but precedence, normalization, regex safety, cache behavior, and `stripMatch` semantics need maintainer API/product decisions first.

## Risks / Open Questions

- `stripMatch` needs an API decision on whether it rewrites only the agent-facing body or also raw body, command text, reply context, history, and channel metadata.
- Regex/prefix matching needs explicit normalization, length limits, and safe-regex behavior before accepting user config on hot inbound paths.
- Content-dependent routing conflicts with the current static route cache unless content rules bypass caching or the cache key/contract changes.
- Precedence must be designed against static binding tiers, runtime conversation bindings, ACP bindings, Discord mention/reply targeting, and the separate pre-routing hook proposal.
