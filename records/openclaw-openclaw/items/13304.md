## Summary

Keep open. Current main still exposes only static binding selectors, strict validation rejects a `match.content` object, and route resolution/cache/channel call sites do not carry message text or return a stripped body; the related issues either cover model routing or narrower Discord mention routing rather than superseding this feature.

## What This Changes

The issue requests content-pattern matching in agent bindings so messages can route to different agents based on prefixes or regexes.

## Best Possible Solution

Define a validated binding content matcher with deterministic precedence, safe pattern handling, content-aware route caching or bypass, normalized inbound-text propagation through representative channel paths, documented stripped-body semantics, and matching config/docs/test coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability rather than broken behavior under an existing contract. Source inspection on current main shows the binding schema and resolver lack the requested content matcher and body-rewrite path.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but maintainers still need to choose the public contract, precedence, cache behavior, regex safety, and stripping semantics versus a pre-route hook or narrower mention-routing design.
