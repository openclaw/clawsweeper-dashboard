## Label Justifications

- `P2`: This is a normal-priority multi-agent routing capability request with real workflow impact but no current regression or emergency.
- `impact:session-state`: The requested matcher decides which agent/session receives a message, so absent support can put conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents can be misrouted through a default dispatcher workaround instead of reaching the intended handler directly.

## Summary

Keep open. Current main still lacks a declarative `bindings[].match.content` contract, the resolver and route cache are static-metadata based, and the open pre-route hook work is only an adjacent possible implementation path rather than a shipped replacement for this feature.

## What This Changes

The issue requests optional regex or prefix content matchers in agent bindings, including optional matched-prefix stripping before dispatching to the selected agent.

## Best Possible Solution

Settle the product/API direction first: either add a validated `bindings[].match.content` contract with deterministic precedence, cache behavior, channel text plumbing, and body rewrite semantics, or land a generic pre-route hook and document the plugin/ClawHub implementation path.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection on current main confirms `match.content`, content-aware route input, and rewritten body output are absent.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but the maintainable solution needs maintainer decisions on precedence, normalization, regex safety, route-cache behavior, and body-rewrite semantics; the related pre-route hook may become the better extension point.

## Risks / Open Questions

- `stripMatch` needs a product/API decision on whether it rewrites only the agent-facing body or also raw command text, reply context, hook payloads, transcripts, and audit metadata.
- Content-dependent routing conflicts with the current static route cache unless the cache key includes normalized content, content rules bypass caching, or caching is partitioned by matcher type.
- Regex or prefix matching needs explicit normalization, length, escaping, and safe-regex rules before it becomes a public config surface.
- The open pre-route hook path may become the better plugin/ClawHub direction, but it is not merged and does not yet replace declarative binding content matchers.
