## Label Justifications

- `P2`: This is a normal-priority routing improvement with real multi-agent workflow impact but it needs product/API direction rather than urgent repair.
- `impact:message-loss`: Without content-based routing, messages in shared chats can be handled by the default or statically bound agent instead of the intended specialist agent.
- `impact:session-state`: Routing content to the wrong agent can attach conversation context to the wrong agent/session in multi-agent setups.

## Summary

Keep open: current main still lacks binding-level content matching and `stripMatch` behavior, and the open pre-routing hook work is only a partial adjacent path, not a merged replacement for the requested declarative binding API.

## What This Changes

The issue requests `bindings[].match.content` support so inbound message prefixes or regex patterns can route messages to different agents and optionally strip the matched text before the agent sees it.

## Best Possible Solution

Define one maintainer-approved routing surface: either a validated binding content matcher with deterministic precedence/cache/body-rewrite semantics, or a documented plugin hook path that explicitly replaces this config request.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main reproduction: adding `match.content` is rejected by the strict schema, and the resolver has no message text input or stripped-body output.

## Solution Assessment

Unclear until maintainers choose the API shape. A binding matcher is plausible, but the adjacent pre-routing plugin-hook design may be the better long-term owner for content-driven routing.
