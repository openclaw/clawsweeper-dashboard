## Summary

Keep open. Current main still lacks a validated `bindings[].match.content` contract, content-aware route input, and any route output for body stripping, while the related pre-route hook work is separate and still open rather than a shipped replacement for declarative binding rules.

## What This Changes

The issue requests content-pattern matching in agent bindings so prefixes or regexes can route inbound messages to different agents and optionally strip the matched text.

## Best Possible Solution

Define a first-class binding content matcher with strict schema/docs, deterministic precedence against existing static bindings and mention behavior, explicit safe matching rules, content-aware cache behavior, representative channel text plumbing, body-stripping semantics, and focused config/routing tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request. Source inspection on current main and `v2026.5.12` gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent.

## Solution Assessment

Unclear. Declarative `match.content` is a plausible direction, but the maintainable solution needs maintainer decisions on precedence, normalization, regex safety, cache behavior, and body-stripping semantics before implementation.
