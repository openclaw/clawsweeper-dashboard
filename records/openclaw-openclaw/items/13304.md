## Label Justifications

- `P2`: This is a normal-priority multi-agent routing improvement with real workflow impact but it is a feature/product decision, not an urgent regression.
- `impact:message-loss`: Without content-aware routing, messages intended for a specialist agent can be misrouted to the default agent or require forwarding through a middleman.
- `impact:session-state`: The routing decision determines which agent session receives and persists the inbound turn.

## Summary

Keep open. Current main and v2026.6.8 still support only static binding selectors; the related pre-routing hook work is overlapping but not a merged or product-approved replacement for binding-level content matching and strip semantics.

## What This Changes

The issue requests a new binding `content` matcher, including regex/prefix routing and optional prefix stripping, so inbound messages can route to different agents based on message text.

## Best Possible Solution

Define one maintainer-approved routing extension with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused routing/channel tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 show static binding types/schema, static route inputs/cache keys, and docs/callers that do not carry message text or return stripped body text.

## Solution Assessment

Unclear. A binding content matcher is plausible, but OpenClaw's plugin-boundary guidance and the active pre-routing hook discussion mean maintainers should settle the API/product shape before implementation.
