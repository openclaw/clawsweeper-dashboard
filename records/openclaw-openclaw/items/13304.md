## Summary

Keep open. Current main still supports static binding selectors only, strict config validation rejects the proposed content matcher, channel call sites do not pass message text into routing, and the maintainer discussion already identifies unresolved API/product decisions; related #38138 is narrower mention routing, while #10969 and #11220 concern model routing rather than agent binding selection.

## What This Changes

Add optional message-content regex or prefix matching to agent bindings so inbound messages can route to specialist agents and optionally strip the matched text before agent delivery.

## Best Possible Solution

Define a first-class validated binding content matcher in the public config contract, specify deterministic precedence with existing routing and mention behavior, update route resolution/caching and representative channel/plugin inbound paths, then document exact body-stripping semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed match.content config: strict validation has no such key, and route resolution cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A binding-level content matcher is a plausible user-facing surface, but the exact proposal needs maintainer decisions on precedence, regex safety, cache behavior, content normalization, and body rewriting before it is the safest implementation path.

## Risks / Open Questions

- Content matcher precedence is unresolved relative to peer, parent-peer, wildcard, guild/roles, account, channel, default, and narrower mention-gating behavior.
- Regex/prefix safety, normalization, and invalid-pattern handling need a public config contract rather than ad hoc channel behavior.
- stripMatch needs a product/API decision on whether it rewrites only the agent-facing body or also command text, raw body, reply context, grouped history, and transcript records.
- Route caching must become content-aware or bypass content-based bindings to avoid stale routing decisions.
