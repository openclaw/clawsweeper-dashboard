## Summary

Keep open. Current main still only supports static binding selectors, and adding content/prefix routing would be a new routing and config capability with unresolved precedence, regex safety, cache, normalization, and body-rewrite semantics.

## What This Changes

Add optional binding content matchers so inbound messages can route to different agents by regex or prefix, with optional matched-prefix stripping.

## Best Possible Solution

Define a first-class validated content matcher only after maintainers settle precedence, regex safety, cache and body contracts, then plumb normalized inbound text through representative channel/plugin routing paths and document the behavior with regression coverage.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows strict config validation rejects match.content, and the route resolver cannot receive message text or return a stripped agent-facing body.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but the proposed exact shape needs maintainer decisions before it is clearly the narrowest maintainable solution.
