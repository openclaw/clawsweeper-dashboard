## Summary

Keep open. Current main still only supports static binding selectors, and the recent discussion already identifies unresolved API/product decisions around precedence, regex safety, caching, normalization, and body stripping.

## What This Changes

Request adding optional content-based binding matchers so inbound messages can route to different agents by prefix or regex, with optional matched-text stripping.

## Best Possible Solution

Define a first-class validated binding content matcher only after maintainers settle precedence, regex limits, cache/body contracts, and channel/plugin plumbing, then document it with regression coverage.

## Reproduction Assessment

Yes for the current limitation. The proposed match.content object is source-reproducible as unsupported because strict config validation has no such key and resolveAgentRoute cannot receive message text or return a rewritten body.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but the proposed shape is not yet the safest implementation path until maintainers decide precedence, regex safety, cache behavior, normalization, and body rewriting.
