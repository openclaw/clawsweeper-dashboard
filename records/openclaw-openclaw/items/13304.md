## Label Justifications

- `P2`: This is a normal-priority routing improvement with real multi-agent workflow impact, but it is a feature/product decision rather than an urgent regression.
- `impact:message-loss`: Messages intended for a specialist agent can be routed to the default agent or require manual forwarding through a middleman.
- `impact:session-state`: The routing decision determines which agent session receives and persists the inbound turn.

## Summary

Keep open. Current main and v2026.6.8 still expose static binding selectors only; the overlapping pre-routing hook work is not a merged or product-approved substitute for a binding-level content matcher with stripMatch semantics.

## What This Changes

Request to add content or pattern matchers to bindings so inbound messages can route to different agents based on message text, with optional prefix stripping before agent dispatch.

## Best Possible Solution

Define one maintainer-approved routing extension with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused routing/channel tests.

## Reproduction Assessment

Yes at source level: current main's binding type/schema/resolver/cache take only static metadata, and representative channel callers do not pass body text. I did not run a live routing scenario in this read-only review.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but maintainers need to decide between config-level content matching and a plugin hook/API surface before implementation.

## Risks / Open Questions
