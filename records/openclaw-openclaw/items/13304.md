## Label Justifications

- `P2`: This is a concrete multi-agent routing improvement with clear workflow value, but it is not an urgent regression or broken documented behavior.
- `impact:message-loss`: Without content-based routing, messages intended for specialist agents must be manually dispatched and can reach the wrong handler.
- `impact:session-state`: Routing by content would determine which agent/session owns a conversation turn, so the feature directly affects session scoping.

## Summary

Keep open. Current main still exposes only static binding selectors and route resolution is not content-aware; the related pre-routing hook item may inform the design, but it does not yet implement declarative `match.content` routing in bindings.

## What This Changes

Request to allow binding rules to match inbound message content patterns or prefixes and optionally strip the matched text before dispatching to the selected agent.

## Best Possible Solution

Settle the routing API direction first, then implement either a validated binding content matcher or a documented pre-route plugin path with deterministic precedence, safe matching, channel text plumbing, cache behavior, optional body rewrite semantics, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but maintainers need to decide precedence, normalization, regex safety, cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.
