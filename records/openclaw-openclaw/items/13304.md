## Label Justifications

- `P2`: This is a valid multi-agent routing improvement with meaningful but non-urgent user impact and a broad API/config design surface.
- `impact:message-loss`: Without the requested routing decision, an inbound message can be handled by a different agent than the user intended.
- `impact:session-state`: Agent routing selects the isolated session and context that receive and retain the inbound turn.

## Summary

Keep open. Current main and the latest release still only support static binding selectors, while the related pre-routing hook work is either closed unmerged or still an open broader design discussion rather than an implemented replacement.

## What This Changes

The issue requests message-content matching in agent bindings, including regex or prefix matching and optional prefix stripping, so one chat can route messages to different agents based on text.

## Best Possible Solution

Define a maintainer-approved pre-routing/content-routing contract that receives normalized inbound content, has deterministic precedence and cache behavior, and either supports this as a plugin/API seam or explicitly promotes a validated binding matcher into core.

## Reproduction Assessment

Yes. Source inspection on current main shows the proposed selector is rejected by the binding schema, the resolver has no message-content input, and channel call sites pass only static route metadata.

## Solution Assessment

No definitive implementation is best yet. A direct regex field in bindings is plausible but not proven safer than a generic pre-routing plugin/API seam, given OpenClaw's core-lean/plugin-boundary guidance.

## Risks / Open Questions

- A native content matcher would expand the public config surface and needs explicit precedence against peer, parent-peer, role, account, channel, runtime conversation bindings, and mention behavior.
- The current route cache is keyed only by static route metadata, so content-dependent matches need a deliberate cache bypass or content-aware cache contract.
- stripMatch needs a product/API decision on whether it rewrites only the agent-facing body or also command text, reply context, grouped history, transcripts, and raw channel metadata.
- Regex matching needs normalization, length, and safety rules before it can run across channel/plugin inputs.
