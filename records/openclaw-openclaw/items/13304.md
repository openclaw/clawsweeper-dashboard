## Label Justifications

- `P2`: This is a normal-priority multi-agent routing capability request with real workflow value but no current regression or emergency.
- `impact:session-state`: The requested matcher decides which agent and session receives a message, so absent support can place conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents can be routed through a default dispatcher workaround instead of directly reaching the intended handler.

## Summary

Keep open. Current main still lacks declarative `bindings[].match.content` support, strict validation rejects that shape, route resolution/cache remain static, and the related pre-route hook work is still an open implementation path rather than shipped binding-level content routing.

## What This Changes

Request adding regex or prefix content matchers to agent bindings so inbound messages can route directly to specialist agents and optionally strip the matched prefix.

## Best Possible Solution

Settle the product/API direction first, then either add a validated `bindings[].match.content` contract with deterministic precedence, cache behavior, channel text plumbing, and body rewrite semantics, or land and document a generic pre-route hook/plugin path that fully covers this workflow.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but maintainers need to decide precedence, normalization, regex safety, route-cache behavior, and body-rewrite semantics, and the related pre-route hook may be a cleaner extension point.

## Risks / Open Questions

- `stripMatch` needs a product/API decision on whether it rewrites only the agent-facing body or also raw command text, hook payloads, transcripts, reply context, and audit metadata.
- Content-dependent routing conflicts with the current static route cache unless caching is partitioned by matcher type, bypassed for content rules, or keyed by normalized content.
- Regex or prefix matching needs explicit normalization, escaping, length, and safe-regex rules before becoming a public config surface.
- The open pre-routing hook proposal may become the preferred plugin path, but it is not merged and does not yet replace declarative binding content matchers.
