## Summary

Keep open: current main still supports only static binding selectors, and content-based binding routing would require a new public config contract plus resolver/cache/channel body semantics. The related pre-route hook work is adjacent but does not supersede the binding-level content matcher requested here.

## What This Changes

Add optional content pattern or prefix matching to `bindings[].match` so inbound messages can route to different agents by message text and optionally strip the matched prefix.

## Best Possible Solution

Define a validated binding content-matcher contract with deterministic precedence, safe pattern handling, content-aware cache behavior or bypass, normalized inbound-text propagation, documented stripped-body semantics, and focused config/resolver/channel coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability rather than broken behavior under an existing contract. Source inspection on current main gives high confidence that bindings still lack the requested content matcher and body-rewrite path.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but maintainers should first settle the public contract, precedence, cache behavior, regex safety, and stripping semantics versus an SDK pre-route hook or narrower mention-routing design.
