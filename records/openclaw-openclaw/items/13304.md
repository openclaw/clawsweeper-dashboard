## Label Justifications

- `P2`: This is a normal-priority routing improvement with real multi-agent workflow impact but it is a feature/product decision rather than an urgent regression.
- `impact:message-loss`: Without content-aware routing, messages intended for a specialist agent can be delivered to the default agent or require manual forwarding through a middleman.
- `impact:session-state`: The routing decision determines which agent session receives and persists the inbound turn.

## Summary

Keep open. Current main and v2026.6.6 still expose static binding selectors only; the config schema rejects a `match.content` object, the route resolver/cache do not carry message text or a rewritten body, and the related generic pre-route hook work at https://github.com/openclaw/openclaw/issues/81061 and https://github.com/openclaw/openclaw/pull/81198 does not supersede the binding-level product decision.

## What This Changes

Request adding content/prefix/regex matchers to agent bindings, including optional prefix stripping, so inbound messages can route directly to specialist agents by text.

## Best Possible Solution

Define one maintainer-approved routing extension: either a validated binding content matcher or a generic pre-route plugin API, with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused route/channel tests.

## Reproduction Assessment

Yes, at source level. Current main shows static binding types/schema, static route inputs/cache keys, and representative channel callers that do not carry message text or return a stripped body.

## Solution Assessment

Unclear. A binding content matcher is plausible, but OpenClaw’s plugin-boundary guidance and the related pre-route hook work mean maintainers should settle the API/product shape before implementation.
