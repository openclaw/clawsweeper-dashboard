## Summary

Keep open. Current main still supports only static binding selectors and no content matcher or stripped-body route output; the related Discord mention-routing item is narrower, while the model-routing items are closed and do not supersede this core routing/API feature request.

## What This Changes

Request declarative `bindings[].match.content` message-pattern routing, optionally stripping matched prefixes before dispatching to selected agents.

## Best Possible Solution

Define a first-class validated binding content matcher after maintainer API review, with deterministic precedence, safe matching, content-aware routing cache behavior, normalized text propagation through representative channel paths, and focused docs/tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing capability, not broken documented behavior. Source inspection does confirm the current limitation because binding types/schema, the resolver/cache, and channel call sites do not accept message text or return stripped body text.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but maintainers still need to choose the public contract versus a pre-route hook or channel-specific mention override before implementation.
