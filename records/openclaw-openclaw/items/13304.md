## Summary

Keep open. Current main still supports only static binding selectors, strict config validation rejects the proposed content matcher, and representative channel paths do not pass message text into route resolution or receive rewritten body text back. The related mention-routing request is narrower, while the older related reports are about model routing, so none supersedes this feature.

## What This Changes

Add optional content/pattern matching and optional matched-prefix stripping to agent bindings so messages in the same channel or peer can route to different agents based on text.

## Best Possible Solution

Define a first-class validated binding content matcher in the public config contract, specify deterministic precedence with existing static bindings and mention behavior, make route resolution and caching content-aware, pass normalized inbound text through representative channel/plugin routing paths, and document exactly what optional body stripping changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed match.content config: the strict binding schema has no such key, and the route resolver cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A content matcher on bindings is a plausible user-facing surface, but the exact implementation needs maintainer decisions on precedence, regex safety, cache behavior, content normalization, and stripMatch rewrite semantics before it is the safest fix.
