## Label Justifications

- `P2`: This is a normal-priority routing improvement with real multi-agent workflow impact but it needs product/API direction rather than urgent repair.
- `impact:message-loss`: Without content-based routing, messages in shared chats can be delivered to the default or statically bound agent instead of the intended specialist agent.
- `impact:session-state`: Routing different content to the wrong agent can attach conversation context to the wrong agent/session in multi-agent setups.

## Summary

Current main still lacks binding-level content matching or stripMatch semantics, and the open pre-routing hook work is only a possible adjacent design path, not a merged replacement for this requested config/API behavior.

## What This Changes

The issue requests adding optional message-content pattern matching to agent bindings so inbound messages can route to different agents and optionally strip the matched prefix before dispatch.

## Best Possible Solution

Define one maintainer-approved routing surface: either a first-class validated binding content matcher with deterministic precedence/cache/body-rewrite semantics, or a documented plugin hook path that explicitly replaces this config request.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main reproduction: adding match.content to a binding is rejected by the strict schema, and the resolver cannot inspect or rewrite message text.

## Solution Assessment

Unclear until maintainers choose the API shape; the requested binding matcher is plausible, but the adjacent pre-routing plugin-hook design may be the better long-term owner for content-driven routing.
