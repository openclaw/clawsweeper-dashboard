## Label Justifications

- `P2`: This is a normal-priority multi-agent routing improvement with real workflow impact, but it is a feature/API decision rather than an urgent regression.
- `impact:message-loss`: Without content-aware routing, messages intended for a specialist agent can be misrouted to the default agent or require a forwarding middleman.
- `impact:session-state`: The requested routing decision determines which agent session receives and persists each inbound turn.

## Summary

Keep open. Current main and the latest release still support only static binding selectors; the related pre-routing hook work overlaps but does not provide binding-level content matchers or stripMatch semantics, and the remaining API shape needs maintainer product direction.

## What This Changes

The issue requests optional binding content matchers so inbound messages can route to different agents based on message text patterns or prefixes.

## Best Possible Solution

Define one maintainer-approved routing extension with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused routing/channel tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 show static binding types/schema, static route inputs/cache keys, and docs/callers that do not carry message text or return stripped body text.

## Solution Assessment

Unclear. A binding content matcher is plausible, but OpenClaw's plugin-boundary guidance and the active pre-routing hook discussion mean maintainers should settle the API/product shape before implementation.
