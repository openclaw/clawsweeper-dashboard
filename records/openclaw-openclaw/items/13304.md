## Summary

Keep open. Current main still has only static binding selectors, and the existing maintainer review already called out unresolved API decisions around precedence, regex safety, cache behavior, normalization, and body rewriting.

## What This Changes

Request adding a binding-level content matcher so inbound messages can route to different agents by regex or prefix and optionally strip the matched prefix before delivery.

## Best Possible Solution

Define a first-class validated content matcher only after maintainers settle the routing contract, then plumb normalized inbound text through channel/plugin routing paths and document stripping behavior with regression coverage.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows strict config validation rejects match.content, and resolveAgentRoute cannot receive message text or return a stripped agent-facing body.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but the proposed regex and stripMatch shape is not yet clearly the narrowest maintainable solution without decisions on precedence, safety, caching, normalization, and body rewriting.
