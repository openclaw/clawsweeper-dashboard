## Summary

Keep open. Current main still exposes only static binding selectors, and related mention/pre_route work is open or narrower, so this remains a product/API routing design request rather than conservative cleanup.

## What This Changes

The issue requests a binding content matcher that routes inbound messages to different agents by regex or prefix and can optionally strip the matched text before agent dispatch.

## Best Possible Solution

Define a first-class validated content-routing contract only after maintainers settle precedence, normalization, safe matching, cache behavior, and body-rewrite semantics, then document it with focused routing/config/channel tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing capability, not broken documented behavior. Source inspection does confirm the current limitation because binding types/schema, route cache, and `resolveAgentRoute` do not accept message text or return a stripped body.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but a pre_route hook or mention-specific override may be the better boundary for some use cases, so maintainers need to choose the public contract first.

## Risks / Open Questions

- `stripMatch` needs a product/API decision on which text surfaces it rewrites: agent-facing body, raw body, reply context, grouped history, and channel metadata may differ.
- Content-dependent routing conflicts with the current static resolved-route cache unless cache keys, bypass rules, or matcher classes are redesigned.
- Regex/prefix matching needs normalization, safe-pattern, length, and precedence rules to avoid accidental broad routing or ReDoS-style hazards.
- Related mention-routing and pre_route proposals overlap but do not yet define whether binding content matchers, explicit mentions, or plugin hooks should own the public contract.
