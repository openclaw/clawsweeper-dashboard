## Summary

Keep open: current main still only supports static binding selectors, and content-based routing would need new public config, resolver/cache behavior, and channel body-rewrite semantics; the related items cover model routing or narrower Discord native-target routing rather than superseding this request.

## What This Changes

Request to let `bindings[].match` route agents by message content patterns, optionally stripping the matched prefix before the selected agent sees the message.

## Best Possible Solution

Define a validated binding content-matcher contract with deterministic precedence, safe pattern handling, content-aware cache behavior or bypass, normalized inbound-text propagation, documented stripped-body semantics, and matching config/docs/test coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability rather than broken behavior under an existing contract. Source inspection on current main shows the binding schema and resolver lack the requested content matcher and body-rewrite path.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but maintainers should first choose the public contract, precedence, cache behavior, regex safety, and stripping semantics versus a pre-route hook or narrower mention-routing design.
