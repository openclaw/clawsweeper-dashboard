## Label Justifications

- `P2`: This is a normal-priority routing improvement with real multi-agent workflow impact, but it is a feature/product decision rather than an urgent regression.
- `impact:message-loss`: Without content-aware routing, messages intended for a specialist agent can be delivered to the default agent or require manual forwarding through a middleman.
- `impact:session-state`: The routing decision determines which agent session receives and persists the inbound turn.

## Summary

Keep open. Current main and v2026.6.6 still expose static binding selectors only, and the related pre-route hook work could become a plugin-based path but does not yet supersede the binding-level routing/product decision.

## What This Changes

The issue requests a `bindings[].match.content` matcher with pattern matching and optional prefix stripping to route inbound messages to different agents based on message text.

## Best Possible Solution

Define one maintainer-approved routing extension, with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused route/channel tests.

## Reproduction Assessment

Yes, at source level. Current main shows static binding types/schema, static route inputs/cache keys, and representative channel callers that do not carry message text or return a stripped body.

## Solution Assessment

Unclear. A binding content matcher is plausible, but OpenClaw’s plugin-boundary guidance and the related pre-route hook work mean maintainers should settle the API/product shape before implementation.
