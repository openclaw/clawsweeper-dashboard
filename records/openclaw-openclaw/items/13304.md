## Label Justifications

- `P2`: This is a concrete multi-agent routing improvement with clear workflow value, but it is not a regression or broken documented behavior.
- `impact:message-loss`: Without content-based routing, messages intended for a specialist agent can be routed through the wrong handler or require manual relay.
- `impact:session-state`: The requested matcher decides which agent/session owns an inbound turn, directly affecting session scoping and conversation state.

## Summary

Keep open. Current main and the latest release still only support static binding selectors; no content matcher, content-aware route input, or body-rewrite output exists, and the related pre-routing hook work is adjacent but not a merged declarative binding solution.

## What This Changes

Request to let top-level agent bindings route by message text patterns or prefixes, optionally stripping the matched prefix before the agent sees the turn.

## Best Possible Solution

Settle the routing API direction first, then implement either a validated binding content matcher or a documented pre-route plugin path with deterministic precedence, safe matching, channel text plumbing, cache behavior, optional body rewrite semantics, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that match.content, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative match.content is plausible, but maintainers need to decide precedence, normalization, regex safety, cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.
