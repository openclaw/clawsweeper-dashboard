## Summary

Keep open. Current main still lacks a binding content matcher, strict config validation would reject the proposed shape, and the routing contract/cache/channel call sites remain static; the related Discord mention-routing and model-routing reports do not supersede this broader agent-binding feature.

## What This Changes

The issue requests an optional content pattern matcher on agent routing bindings, including optional matched-prefix stripping before the message reaches the selected agent.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe pattern handling, content-aware cache behavior, normalized inbound-text propagation through representative channel paths, and documented stripped-body semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new config/routing capability, not broken behavior under an existing contract. Source inspection gives high confidence that current main lacks the requested matcher and route body plumbing.

## Solution Assessment

Unclear. A binding-level matcher is plausible, but maintainers still need to choose the public contract versus a pre-route hook or narrower channel-specific mention behavior before implementation.

## Risks / Open Questions

- Binding content matching needs a maintainer decision on precedence relative to peer, parent-peer, role, account, channel, mention-gating, broadcast, and default routing.
- Regex or prefix matching needs explicit normalization, limits, and safety rules before becoming a public config surface.
- stripMatch needs a clear contract for whether it rewrites only agent-facing text or also command parsing, raw body, reply context, grouped history, transcripts, and hook payloads.
- The current static route cache would return incorrect results if content-sensitive rules were added without a cache-key change, cache bypass, or separate content-aware route path.
