## Summary

Keep open. Current main and the latest shipped release still expose only static binding selectors, while the related pre-route hook work covers plugin-level redirect/suppress hooks rather than declarative `bindings[].match.content` matching or `stripMatch` body rewrite semantics.

## What This Changes

The issue requests content-pattern matching in agent bindings so inbound messages can route to different agents by text prefix or regex, with optional stripping before dispatch.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe matching rules, content-aware cache behavior, representative channel text plumbing, optional body-stripping semantics, docs, and focused config/routing tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request. Source and shipped-release inspection give high confidence that current OpenClaw lacks `match.content`, content-aware route input, and body-rewrite output.

## Solution Assessment

Unclear. Declarative `match.content` may be a maintainable direction, but precedence, normalization, regex safety, cache behavior, and `stripMatch` semantics need maintainer API/product decisions before implementation.

## Risks / Open Questions

- `stripMatch` needs a product/API decision on whether it rewrites only the agent-facing body or also command text, raw body, reply context, grouped history, and channel metadata.
- Content-sensitive routing conflicts with the current static route cache unless the cache key, cache bypass, or matcher indexing model changes deliberately.
- Regex/prefix matching needs explicit normalization, length limits, and safe-regex rules to avoid surprising broad routes or ReDoS-style hazards.
- Precedence needs maintainer design across static bindings, mention gating, runtime focus/conversation bindings, ACP bindings, and channel-specific target overrides.
