## Label Justifications

- `P2`: This is a normal-priority multi-agent routing capability request with real workflow value but no current regression or emergency.
- `impact:session-state`: The requested matcher decides which agent and session receives a message, so absent support can place conversation state in the wrong agent scope.
- `impact:message-loss`: Messages intended for specialist agents can be routed through a default dispatcher workaround instead of directly reaching the intended handler.

## Summary

Keep open. Current main still only supports static binding selectors, strict config validation rejects the proposed content matcher shape, route resolution and caching do not carry message text, and the related pre-routing hook work is still open rather than a shipped binding-level solution.

## What This Changes

Request adding optional message-content pattern or prefix matching, with optional prefix stripping, to `bindings[].match` so one chat can route messages to different agents by text.

## Best Possible Solution

Settle the product/API direction first, then either add a validated `bindings[].match.content` contract with deterministic precedence, cache behavior, channel text plumbing, and body rewrite semantics, or land and document a generic pre-route hook/plugin path that fully covers this workflow.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that `match.content`, content-aware route input, and rewritten body output are absent on current main.

## Solution Assessment

Unclear for the exact API shape. Declarative `match.content` is plausible, but maintainers need to decide precedence, normalization, regex safety, route-cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.
