## Summary

Keep open. Current main and v2026.6.10 still support only static binding selectors; the open pre-routing hook work overlaps but is not a merged or product-approved replacement for binding-level content matching and stripMatch semantics.

## What This Changes

This issue requests content or prefix/regex matchers in routing bindings so inbound messages can route directly to different agents based on message text.

## Best Possible Solution

Define one maintainer-approved routing extension with deterministic precedence, normalized text, safe matching, cache semantics, body-stripping behavior, docs, and focused routing/channel tests.

## Reproduction Assessment

Yes at source level for the capability gap: current main's binding type/schema, resolver input/output, route cache, and representative channel callers accept only static selectors. This is not a failing bug repro, but it verifies the requested behavior remains absent.

## Solution Assessment

Unclear. A binding content matcher is plausible, but OpenClaw's plugin-boundary guidance and the active pre-routing hook discussion mean maintainers should settle the API/product shape before implementation.
