## Label Justifications

- `P2`: This is a concrete multi-agent routing improvement with clear workflow value, but it is a feature request rather than a regression in documented behavior.
- `impact:message-loss`: Without content-sensitive routing, messages intended for a specialist agent can be routed through the wrong handler or require manual relay.
- `impact:session-state`: The requested matcher decides which agent/session owns an inbound turn, directly affecting conversation and session state.

## Summary

Keep open. Current main and the latest release still only support static binding selectors; the requested declarative content matcher, content-aware route input, and body-rewrite output are still missing, and the related pre-route hook path remains open and unresolved.

## What This Changes

Request to add optional content pattern matching to agent route bindings, including regex or prefix matching and optional matched-prefix stripping before forwarding to the selected agent.

## Best Possible Solution

Settle the routing API direction first, then implement a validated content matcher or documented pre-route plugin path with deterministic precedence, safe matching, content-aware cache behavior, body rewrite semantics, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing/config capability request, not broken behavior under an existing contract. Source inspection gives high confidence that match.content, content-aware route input, and rewritten-body output are absent on current main.

## Solution Assessment

Unclear for the exact proposed API shape. Declarative match.content is plausible, but maintainers need to decide precedence, normalization, regex safety, cache behavior, body-rewrite semantics, and whether the broader pre-route hook should be the preferred extension point.
