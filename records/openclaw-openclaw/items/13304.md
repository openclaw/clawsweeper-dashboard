## Label Justifications

- `P2`: This is a normal-priority routing feature with concrete multi-agent workflow impact but no urgent regression.
- `impact:message-loss`: Without content-aware routing, group messages can be delivered to the default or statically bound agent instead of the intended specialist.

## Summary

Keep open: current main and latest release still support only static binding selectors, and the broader pre-route hook work is open, unmerged, and only partial overlap rather than a complete binding-level content matcher.

## What This Changes

The issue requests content-pattern matching in agent bindings so messages can route to different agents based on text prefixes or regexes.

## Best Possible Solution

Define one maintainer-approved content-aware routing contract, then update config validation, route resolution/cache semantics, representative channel inputs, docs, generated config surfaces, and focused tests together.

## Reproduction Assessment

Yes at source level: current main has strict static binding schemas and route inputs/cache keys that do not carry message text or rewritten bodies. I did not run a live channel repro because this is a missing feature/config contract, not broken behavior under an existing contract.

## Solution Assessment

Unclear. The proposed binding content matcher is plausible, but AGENTS.md and VISION.md make the product boundary important because a generic trusted pre-route plugin hook may be the cleaner long-term seam or a prerequisite.
