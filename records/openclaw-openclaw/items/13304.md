## Summary

Keep open. Current main still only supports static binding selectors and no content matcher or stripped-body route output; https://github.com/openclaw/openclaw/issues/38138 treats this as the canonical content-routing tracker, while the model-routing and Discord mention-routing items do not supersede it.

## What This Changes

The issue requests optional message-content pattern matching in agent bindings so inbound text can route to different agents and optionally strip the matched prefix before delivery.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe matching, content-aware cache behavior, normalized inbound-text propagation through representative channel paths, and documented stripped-body semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new routing capability, not broken documented behavior. Source inspection gives high confidence that current main lacks the requested binding type/schema fields, resolver input/output, cache key, and channel call-site body propagation.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but maintainers still need to choose the public contract versus a pre-route hook or narrower channel-specific mention override before implementation.

## Risks / Open Questions

- Binding content matching needs a maintainer decision on precedence relative to peer, parent-peer, role, account, channel, mention-gating, broadcast, and default routing.
- Regex or pattern matching needs safety limits and deterministic normalization before it becomes a config surface.
- `stripMatch` needs a clear contract for whether it rewrites only the agent-facing body or also command parsing, raw body, reply context, grouped history, transcripts, and hook payloads.
- The current static route cache would produce incorrect results if content-sensitive rules were added without a cache-key change or explicit bypass.
