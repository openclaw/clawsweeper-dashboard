## Summary

Keep open. Current main still only supports static binding selectors, strict config validation rejects a proposed `match.content` object, and resolver/channel call sites do not carry inbound message text or return rewritten content; this is a valid feature request that needs API/product design before implementation.

## What This Changes

Request adding optional content or pattern matchers to routing bindings so inbound messages can route to different agents and optionally strip matched prefixes.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe pattern handling, content-aware cache behavior, normalized inbound-text propagation through representative channel paths, documented stripped-body semantics, and focused config/routing tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this asks for a new routing/config capability rather than broken behavior under an existing contract. Source inspection gives high confidence that current main lacks the requested matcher and body-rewrite plumbing.

## Solution Assessment

Unclear. A binding-level content matcher is plausible, but maintainers still need to choose the public contract and precedence against alternatives like a pre-route hook or narrower channel-specific mention routing.
