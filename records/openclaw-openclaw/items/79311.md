## Best Possible Solution

Land only after maintainers sponsor the sidecar execution contract, the runtime loads canonical eligible skills, publish dedupe uses unique Feishu event identity, conflicts are resolved, and redacted live handler or trigger completion proof is posted.

## Solution Assessment

No. A Feishu-owned event runtime is a plausible layer, but this implementation is not the best fix until it uses canonical eligible-skill inputs, event-ID dedupe, current-main Feishu wiring, and a maintainer-approved sidecar contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor skill eligibility before loading subscribers:** `extensions/feishu/src/event.runtime.ts:145-146`
  - body: `startFeishuEventRuntime` loads subscriber specs from raw workspace/home/plugin roots, and the loader accepts any `SKILL.md` sidecar it finds. A skill disabled with `skills.entries.<key>.enabled = false`, blocked by requirements, blocked by bundled allowlists, or excluded by an agent skill filter can still import a handler or trigger an agent run from Feishu events; build subscribers from canonical eligible skill entries for the target agent instead.
  - confidence: 0.9
- **[P2] Deduplicate publish events by event ID:** `extensions/feishu/src/event.model.ts:387-393`
  - body: Bitable normalization chooses `record_id` or `field_id` before `event_id`, and the hijacker dedupes publish events on `normalizedEvent.sourceId`. Two legitimate updates to the same record or field can drop the second event for the dedupe TTL; use the unique Feishu event ID for dedupe and keep record and field IDs as subject tokens.
  - confidence: 0.88
