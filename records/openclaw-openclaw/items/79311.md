## Summary

Keep this PR open, but it is not merge-ready: the proposed Feishu subscriber runtime bypasses existing skill eligibility/settings, can silently dedupe away repeated publish events, lacks sufficient after-fix handler/trigger proof, and is currently conflicting with newer Feishu monitor work on main.

## What This Changes

This PR adds a Feishu event runtime that loads `feishu-event.subscribers.json` sidecars from skills, publishes normalized Feishu events to a topic bus, and dispatches matching handlers or agent triggers from monitor startup.

## Best Possible Solution

Land a rebased Feishu plugin implementation that uses canonical skill eligibility/install-policy boundaries, dedupes publish deliveries by unique event IDs, preserves current monitor runtime wiring, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, by source inspection: the event runtime scans raw skill roots without the canonical skill filters, and the hijacker dedupes publish events on `normalizedEvent.sourceId`. A focused regression can use a disabled skill sidecar plus two bitable events with the same `record_id` and distinct `event_id` values.

## Solution Assessment

No. The feature direction is useful, but the current implementation is not the best fix because it bypasses existing skill eligibility boundaries and uses subject IDs for event dedupe; the safer path is a canonical eligible-skill subscriber surface with event-id dedupe and upgrade/security proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Honor skill eligibility before loading subscribers:** `extensions/feishu/src/event.runtime.ts:146`
  - body: This path scans every configured workspace/global/extra skill root and loads subscriber sidecars directly, bypassing the existing skill eligibility pipeline. A skill disabled with `skills.entries.<key>.enabled = false` or excluded for a `targetAgentId` with `agents.list[].skills: []` can still run its handler or trigger when Feishu events arrive; build subscribers from canonical eligible skill entries for the target agent instead.
  - confidence: 0.9
- **[P2] Use event IDs for publish-event dedupe:** `extensions/feishu/src/event.model.ts:387-393`
  - body: The hijacker dedupes publish-path events on `normalizedEvent.sourceId`, but bitable normalization chooses `record_id`/`field_id` before `event_id`. That drops the second legitimate update to the same record or field for the Feishu dedupe TTL, so use the unique event id for dedupe and keep record/field ids as subject tokens.
  - confidence: 0.88
