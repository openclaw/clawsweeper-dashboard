## Summary

Keep open. This is a recent external feature PR with useful Feishu work, but current main does not implement the requested runtime and the PR still has blocking correctness/security issues plus insufficient real Feishu proof.

## What This Changes

The PR adds a Feishu declarative event runtime that loads skill sidecar subscriber specs, publishes normalized events through a topic bus, and dispatches handlers or agent triggers from monitor startup.

## Best Possible Solution

Land a Feishu-owned subscriber runtime only after it uses the canonical skill visibility contract, registers concrete Feishu SDK event types reliably, preserves per-event dedupe, documents the sidecar contract, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the raw skill-root scan, exact SDK event dispatch contract, and sourceId dedupe behavior are visible in the PR branch. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No. The runtime direction is plausible, but this patch is not the narrowest maintainable solution until it aligns with OpenClaw's skill visibility contract, exact Feishu SDK registration, and unique event dedupe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Respect skill visibility before executing subscribers:** `extensions/feishu/src/event.runtime.ts:145-147`
  - body: `startFeishuEventRuntime` scans raw skill roots and later imports handler files from those entries, bypassing the effective skill snapshot that enforces precedence and agent allowlists. A shadowed, disabled, or agent-disallowed skill could still receive Feishu events and execute host code, so derive subscribers from the canonical visible skill set or enforce the same filters first.
  - confidence: 0.89
- **[P2] Require exact event types for SDK registration:** `extensions/feishu/src/event.runtime.ts:204-209`
  - body: Subscriber specs can match only topics, categories, or subtypes, but the monitor only registers explicit `eventTypes` with the Feishu SDK. Because the SDK invokes handlers by exact parsed event type, those otherwise-valid subscribers never reach the topic bus unless the spec requires concrete event types or the runtime derives a concrete registration list.
  - confidence: 0.88
- **[P2] Deduplicate on the event delivery id:** `extensions/feishu/src/event.hijacker.ts:75-79`
  - body: The hijacker dedupes publish-path events with `normalizedEvent.sourceId`, but the normalizer chooses subject identifiers such as a bitable `record_id` before `event_id`. That treats later updates for the same record as duplicate deliveries, so keep a separate delivery id for dedupe and leave subject ids for routing/session hints.
  - confidence: 0.82
