## Summary

Keep this PR open for normal review. Current main does not implement declarative Feishu event subscribers, but the proposed feature has blocking correctness/security issues in skill gating, SDK event registration, dedupe behavior, and required proof/changelog coverage.

## What This Changes

The PR adds a Feishu event runtime that loads `feishu-event.subscribers.json` files from skills, publishes normalized Feishu events to a topic bus, and dispatches matching handlers or agent triggers from Feishu monitor startup.

## Best Possible Solution

Land a Feishu-owned event subscriber runtime only after it uses canonical skill visibility and precedence, registers concrete Feishu event types reliably, preserves per-event dedupe, documents the sidecar contract, includes a changelog entry, and has real Feishu proof showing handler and trigger dispatch.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the new loader scans raw skill roots outside the canonical skill snapshot, the SDK only invokes exact registered event types, and the hijacker dedupes on normalized `sourceId`. A live Feishu run was not attempted in this read-only review.

## Solution Assessment

No. A Feishu-owned event subscriber runtime is a plausible direction, but this implementation should be narrowed to the established skill contract, exact SDK registration model, and unique-event dedupe before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Respect skill visibility before executing subscribers:** `extensions/feishu/src/event.runtime.ts:145-147`
  - body: `startFeishuEventRuntime` scans raw skill roots and later imports subscriber handler files from those entries, bypassing the canonical skill snapshot that enforces precedence and agent allowlists. A lower-precedence, shadowed, or agent-disallowed skill can still receive Feishu events and execute host code, so build subscribers from the existing skill visibility contract or explicitly enforce the same filters first.
  - confidence: 0.89
- **[P2] Require exact event types for SDK registration:** `extensions/feishu/src/event.runtime.ts:204-209`
  - body: Subscriber specs can match only `topics`, `categories`, or `subtypes`, but this helper exposes only explicit `eventTypes` to `registerEventHandlers`. The Feishu SDK dispatches by exact registered type, so those otherwise-valid subscribers never reach the topic bus; require concrete event types or add a separate exact registration list.
  - confidence: 0.88
- **[P2] Deduplicate on the event id, not the subject id:** `extensions/feishu/src/event.hijacker.ts:75-79`
  - body: The hijacker checks duplicates using `normalizedEvent.sourceId`, while the normalizer uses subject identifiers such as a bitable `record_id` before `event_id`. That drops every later update for the same record as a duplicate, so keep a separate unique delivery id for dedupe and leave subject ids for routing/session hints.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `extensions/feishu/src/monitor.ts:48-51`
  - body: This is a user-visible `feat(feishu)` change, but the PR does not touch `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes before merge.
  - confidence: 0.94
