## Summary

Keep open: the observability attribution idea is useful and now has live happy-path proof, but the patch stores run attribution on process-local session state, so later same-session lifecycle events from paths that never clear `clientContext` can inherit and leak the previous caller's private attribution data.

## What This Changes

Adds an optional gateway/SDK `clientContext` request field, normalizes it, stores it on diagnostic session state, and forwards it as `DiagnosticEventPrivateData` to trusted diagnostics listeners, with generated protocol/API baseline updates.

## Best Possible Solution

Make attribution lifetime run- or turn-scoped, or clear it from every lifecycle ingress before emission, then have maintainers explicitly approve the trusted diagnostics API/privacy boundary and client compatibility story.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `clientContext` for a gateway run, then emit a later same-session `message.queued` or `session.state` through dispatch/lifecycle code that never clears it; the stale private data is still forwarded.

## Solution Assessment

No. The trusted private-data channel is a plausible direction, but storing a run attribution bag on shared session state is not the best fix because the lifetime is broader than the run being attributed.

Full review comments:

- **[P1] Scope clientContext to the run before forwarding it:** `src/logging/diagnostic.ts:912-915`
  - body: This forwards `state.clientContext` for every later lifecycle event in the same diagnostic session, but only the gateway `agent` handler clears that field when a later request omits it. A seeded gateway run followed by a normal channel/gateway event for the same session can therefore send the previous caller's private attribution bag to trusted diagnostics, misattributing spans and leaking stale identity data. Scope the bag to the run/turn or clear it from every ingress path before emitting.
