## What This Changes

The PR changes Gemini web_search so `day`/`pd` freshness becomes a query recency hint, while wider freshness and explicit date ranges still use Gemini `timeRangeFilter`, with matching cache, docs, and tests.

## Best Possible Solution

Land the day-only Gemini fallback after maintainers accept the compatibility tradeoff, keeping wider freshness and explicit date ranges as hard `timeRangeFilter` paths.

## Reproduction Assessment

Yes, at source/proof level. Current main builds a hard exact 24-hour Gemini `timeRangeFilter` for `day`/`pd`, and the PR body plus superseded PR comments provide de-identified live Gemini evidence that exact 24-hour hard windows are rejected.

## Solution Assessment

Yes, with a maintainer compatibility choice. The revised patch is the narrow provider-boundary mitigation: soften only endpoint-rejected `day`/`pd`, preserve hard wider ranges and explicit dates, and partition cache entries accordingly.
