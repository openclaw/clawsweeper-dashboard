## Summary

Keep open: the privacy mitigation is useful, but the patch still misses explicit route-to-channel deliveries and has no after-fix real behavior proof for the public channel surface.

## What This Changes

The branch adds a `suppressToolErrorDetails` flag through direct dispatch, queued followups, and embedded payload building so group/channel tool-failure warnings keep a compact warning while omitting command metadata and raw error suffixes.

## Best Possible Solution

Derive detail suppression from the actual external delivery target or route conversation type, cover direct and queued public-channel paths with focused tests, then add redacted real Telegram or equivalent channel proof before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: explicit webchat-to-channel route tests show no `ChatType`, while the PR derives redaction from `ctx.ChatType`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Payload-level redaction is a plausible layer, but the default must be derived from the actual public delivery target rather than only the inbound chat type, and the public-channel behavior still needs real proof.
