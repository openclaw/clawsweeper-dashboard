## Summary

Close: current main and docs make Gateway-start QMD refresh opt-in; the reporter's config leaves that path off, while boot and interval updates still arm once the long-lived manager opens.

## What This Changes

The issue asks OpenClaw to restore automatic QMD update/embed scheduling after upgrading, using `onBoot`, `interval`, and `embedInterval` without `memory.qmd.update.startup`.

## Best Possible Solution

Keep the lazy startup contract in place and direct users who need Gateway-start QMD refresh to opt into `memory.qmd.update.startup`; treat failures after that opt-in or after a real memory use opens the manager as a new actionable bug.

## Reproduction Assessment

No, not as a current-main bug. Source inspection gives a high-confidence explanation: with `memory.qmd.update.startup` omitted, Gateway startup policy is `off`, and the interval is only armed after the long-lived QMD manager opens.

## Solution Assessment

Yes. The current design is the narrow maintainable solution because it preserves lazy startup to avoid QMD startup hangs while retaining explicit `idle` and `immediate` modes for users who want boot refresh.
