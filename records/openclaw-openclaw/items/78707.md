## Summary

Keep open because current main still makes QMD gateway-start maintenance opt-in, so the reporter's 2026.5.4 configuration can plausibly leave update/embed timers unarmed after restart unless the long-lived memory manager is opened.

## What This Changes

The issue reports that QMD automatic update and embed scheduling stopped firing on boot or interval after upgrading to OpenClaw 2026.5.4.

## Best Possible Solution

Define the intended QMD maintenance contract, then either restore a documented always-on path for configured update intervals after gateway restart or make the lazy startup semantics and `memory.qmd.update.startup` migration path explicit in docs, doctor, and status output.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: with the reporter's config lacking `memory.qmd.update.startup`, gateway post-attach resolves QMD startup to off, while background interval/embed timers only exist after a full long-lived QMD manager opens.

## Solution Assessment

Unclear. Restoring old always-on QMD maintenance could fix the reported freshness regression, but it would reverse the current lazy-startup policy unless maintainers choose a narrower documented mode or migration path.
