## Summary

Keep open. Current main still has a Telegram group text-command path where a configured open group can route ordinary text but drop `/status` before any reply when the effective command sender allowlist has no matching entry, and PR #74894 is already open with closing syntax for this issue.

## What This Changes

The issue reports that Telegram group `/status` slash commands no-reply on OpenClaw 2026.4.27 while ordinary group messages in the same configured groups still reach the agent.

## Best Possible Solution

Review and land PR #74894, or replace it with an equivalent focused Telegram text-command authorization change, so configured open group and topic sessions deliver `/status` replies while disabled groups and explicit sender allowlists continue to fail closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: configure a Telegram group with a per-group `groupPolicy: "open"` override and no effective command sender allowlist, then ordinary text replies while `/status` is dropped before reply delivery.

## Solution Assessment

Yes for the direction, pending PR review. The narrow maintainable fix belongs in the Telegram command-authorization path rather than weakening the shared command gate, because the shared helper is a generic fail-closed primitive and the mismatch is Telegram-specific open-group treatment.
