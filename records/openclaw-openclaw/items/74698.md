## Repair Work Prompt

Fix the Telegram native command group authorization bug reported at https://github.com/openclaw/openclaw/issues/74698. In current main, `/status` in a configured/open Telegram group can pass normal group policy but still be rejected by `resolveTelegramCommandIngressAuthorization` because that helper hard-codes command group policy to `allowlist` and sees no command/group sender allowlist. Keep the fix within the Telegram plugin/auth boundary, preserve `commands.allowFrom` as the sole authorization source when configured, preserve disabled group/topic and group-not-allowed responses, and do not reintroduce DM pairing-store fallback for group authorization. Add regression coverage in `extensions/telegram/src/bot-native-commands.group-auth.test.ts` for at least: configured group with no sender allowlist authorizes `/status`, `groupPolicy: "open"` with no sender allowlist authorizes `/status`, and explicit `groupAllowFrom`/`commands.allowFrom` restrictions still reject unauthorized senders. Do not edit `CHANGELOG.md`; put release-note context in the PR body. Before merge, include real Telegram proof or a maintainer-approved equivalent showing the visible group `/status` reply path.
