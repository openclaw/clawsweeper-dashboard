## Summary

Keep open: current main still has the source-provable voice-call stale reaper path that can end an active Twilio streaming conversation, and the only linked fix PR was closed unmerged.

## What This Changes

The issue reports that Twilio inbound streaming conversation calls can be reaped as stale after about 120 seconds while still in `speaking`/`listening` activity.

## Best Possible Solution

Update the voice-call stale reaper to treat active conversation states, or an equivalent explicit activity marker, as non-stale while preserving cleanup for truly stuck pre-answer calls and adding focused regression coverage.

## Reproduction Assessment

Yes from source inspection: a `speaking` or `listening` call with unset `answeredAt` and age over `staleCallReaperSeconds` still reaches `endCall` on current main. The reporter’s redacted Twilio logs match that path at about 123 seconds.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the voice-call plugin reaper, either by skipping active conversation states or by using an equivalent call-activity signal, while leaving pre-answer stuck-call cleanup intact.

## Repair Work Prompt

Fix the voice-call stale reaper bug reported at https://github.com/openclaw/openclaw/issues/79121. On current main, `extensions/voice-call/src/webhook/stale-call-reaper.ts` skips only calls with `answeredAt` or terminal state, so active Twilio streaming conversation states such as `speaking` and `listening` with unset `answeredAt` can be ended after the 120-second default. Implement a narrow owner-local fix so genuinely active conversation calls are not reaped while preserving cleanup for pre-answer or stuck calls. Use closed https://github.com/openclaw/openclaw/pull/79124 as useful prior art, but re-evaluate the exact guard and avoid broad default/config changes unless necessary. Add focused regression coverage in `extensions/voice-call/src/webhook/stale-call-reaper.test.ts`; update webhook/media-stream tests if the activity signal changes, and update docs/config tests only if defaults or public behavior text changes. Add a user-facing changelog entry for landing. Do not move voice-call ownership into core.
