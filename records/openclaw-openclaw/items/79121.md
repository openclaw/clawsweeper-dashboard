## Label Justifications

- `P1`: This is a current-release channel workflow bug that can terminate otherwise active voice conversations after the default stale timeout.
- `impact:message-loss`: The issue drops an active voice-call conversation mid-turn by ending the call while transcripts and responses are still flowing.

## Summary

Keep open: current `main` and latest release still reap non-terminal voice-call records solely because `answeredAt` is unset, so the reported active `speaking` or `listening` Twilio conversation can still be ended at the stale timeout. The closed unmerged fix PR captured the likely repair shape, but there is no merged implementation yet.

## What This Changes

Prevent the voice-call stale-call reaper from ending active Twilio streaming conversations that are speaking or listening while `answeredAt` is unset.

## Best Possible Solution

Patch the voice-call stale reaper so active conversation states are not treated as stale pre-answer calls, while preserving cleanup for genuinely stuck setup states and adding regression coverage.

## Reproduction Assessment

Yes, at source level: an active call older than `staleCallReaperSeconds` with state `speaking` or `listening` and no `answeredAt` will pass the current reaper skip check and call `endCall`. I did not run a live Twilio reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow reaper-state guard plus regression tests; raising only the timeout would leave active conversations misclassified as stale.
