## Summary

Keep open: current main and the latest release still allow the voice-call stale reaper to end non-terminal calls solely because `answeredAt` is unset, which matches the reported active Twilio conversation being reaped while `speaking`. The bug is source-reproducible and narrow enough for a focused voice-call plugin fix.

## What This Changes

The issue reports that active Twilio voice-call conversation sessions can be ended by stale-call cleanup after about 120 seconds while transcripts and responses are still flowing.

## Best Possible Solution

Update the voice-call plugin so stale cleanup only ends truly stuck calls and cannot end live `speaking` or `listening` streaming conversations, with regression tests and docs/config alignment for the intended default.

## Reproduction Assessment

Yes, as a source-level reproduction: construct an active voice-call record older than `staleCallReaperSeconds` with state `speaking` or `listening` and no `answeredAt`; current `startStaleCallReaper` will call `endCall`. I did not run a live Twilio call in this read-only review.

## Solution Assessment

Yes, a narrow plugin repair is the best path: keep the stale cleanup for genuinely stuck calls, but make active conversation state or recent media/transcript activity protect the call from reaping. Adding a new user-facing knob is not needed unless maintainers choose to change the default separately.
