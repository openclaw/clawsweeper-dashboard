## Summary

Keep open: current main still reaps non-terminal voice-call records solely because `answeredAt` is unset, so the reported active Twilio `speaking` call can still be ended at the 120-second default threshold.

## What This Changes

Active Twilio streaming voice conversations should not be treated as stale when transcripts and AI responses are still flowing.

## Best Possible Solution

Fix the voice-call plugin so genuinely active conversation states are non-stale while preserving cleanup for pre-answer or stuck calls, with focused regression coverage and docs/config wording aligned to the final behavior.

## Reproduction Assessment

Yes, from source inspection without a live phone call: a Twilio streaming call can be in `speaking` or `listening` with `answeredAt` unset, and current main will reap it once its age exceeds `staleCallReaperSeconds`. The reporter's redacted logs match that path at 123 seconds.

## Solution Assessment

Yes: the narrow maintainable direction is to teach the owner-local stale reaper to skip active conversation states or otherwise mark accepted media-stream calls as active for reaper purposes. The closed unmerged PR shows a small viable shape, but the issue still needs a fresh current-main fix.
