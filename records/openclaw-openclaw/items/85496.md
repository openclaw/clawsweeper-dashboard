## Summary

Keep this PR open: the cleanup direction is useful, but the current patch unsubscribes the shared Codex app-server connection after a bound turn without adding a matching resubscribe path before later `turn/start` calls, so a second bound turn can miss its Codex notifications and time out.

## What This Changes

The PR adds best-effort Codex app-server `thread/unsubscribe` cleanup after bound conversation turns and expands conversation-binding regression tests.

## Best Possible Solution

Pair per-turn unsubscribe with an explicit per-turn subscription or resume before `turn/start`, add two-consecutive-turn coverage plus recovery-failure cleanup coverage, then land this PR and let it close https://github.com/openclaw/openclaw/issues/85458.

## Reproduction Assessment

Yes at source level: after this patch, one bound turn can unsubscribe the connection, and the next bound message still goes directly to `turn/start`; upstream Codex only attaches listeners on thread start/resume/fork, so the collector can miss turn notifications.

## Solution Assessment

No as proposed. Reusing the shared unsubscribe helper is the right cleanup primitive, but the PR must also reacquire a Codex thread subscription before each later bound `turn/start` or keep the long-lived subscription model.
