## Summary

Keep open. The PR targets the right cron service boundary for a real duplicate-name bug, but the patch still has two blocking regressions: raw pre-normalization comparisons can miss collisions, and duplicate-name failures would be rethrown by Gateway instead of returned as validation errors.

## What This Changes

The PR adds a cron name lookup helper, duplicate-name rejection in service add/update, duplicate-name service tests, and a changelog entry.

## Best Possible Solution

Land service-level uniqueness over normalized cron names and return collisions through the existing Gateway INVALID_REQUEST path, with regression coverage for service and server-method callers.

## Reproduction Assessment

Yes. Source inspection shows current main accepts duplicate cron names through add and update, and the PR diff exposes the Gateway error-mapping and whitespace-normalization failures without needing a live cron store.

## Solution Assessment

No. The service-layer boundary is right, but this implementation should normalize the candidate name before collision checks and make duplicate-name collisions use the existing Gateway validation response path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:323-325`
  - body: The new collision path throws a plain Error from the cron service. Gateway cron.add/update only convert TypeError and RangeError into INVALID_REQUEST responses and intentionally rethrow ordinary Errors, so CLI, RPC, and agent-tool callers will not receive the actionable duplicate-name error promised by the PR. Route this collision through a validation error path and add server-method coverage.
  - confidence: 0.9
- **[P2] Compare normalized cron names before checking duplicates:** `src/cron/service/ops.ts:321`
  - body: This lookup compares input.name before createJob trims and validates it, and update has the same issue with patch.name. A service caller can add " foo " after "foo"; findJobByName misses the raw value, then createJob stores both as "foo". Normalize the candidate name before add/update collision checks and cover whitespace names.
  - confidence: 0.86
