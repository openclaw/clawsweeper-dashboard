## Summary

Keep open. Current main still accepts duplicate cron names, and this PR targets the right service boundary, but it has two merge-blocking correctness issues around normalized-name comparison and gateway error mapping.

## What This Changes

The PR enforces unique cron job names in service add/update paths, adds duplicate-name service tests, and records the cron fix in the changelog.

## Best Possible Solution

Land service-level uniqueness over normalized cron names and return collisions through the existing gateway validation response path, with regression coverage for service and server-method callers.

## Reproduction Assessment

Yes. Source inspection shows current main accepts duplicate cron names through add and update, and the PR diff exposes the gateway error-mapping and whitespace-normalization failures without needing a live cron store.

## Solution Assessment

No. The service-layer boundary is right, but this implementation should normalize candidate names before collision checks and make duplicate-name collisions use the existing gateway validation response path.
