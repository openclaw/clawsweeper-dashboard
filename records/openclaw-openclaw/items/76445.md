## Summary

Keep open. The PR targets a valid cron duplicate-name bug, but the patch still misses normalized whitespace collisions and surfaces duplicate-name collisions as plain service errors that Gateway callers rethrow instead of returning validation responses.

## What This Changes

The PR adds a cron job name lookup helper, rejects duplicate names during service add/update, adds duplicate-name service tests, and adds a changelog entry.

## Best Possible Solution

Land normalized service-level uniqueness with Gateway validation-response coverage for CLI, RPC, and agent-tool callers, and leave optional replace or legacy duplicate-warning UX as separate maintainer-scoped follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. Current main adds and updates cron jobs without a sibling-name scan, while the PR-specific failures are visible from the diff plus current Gateway error mapping.

## Solution Assessment

No. The service-layer boundary is the right place to enforce uniqueness, but this implementation should compare normalized candidate names and route duplicate-name collisions through the existing Gateway `INVALID_REQUEST` validation path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:323-325`
  - body: The new collision path throws a plain `Error` from the cron service. Gateway cron.add/update only convert `TypeError` and `RangeError` into `INVALID_REQUEST` responses and intentionally rethrow ordinary errors, so RPC, CLI, and agent-tool callers will not receive the actionable validation response promised by the PR. Route this through a validation error path and add server-method coverage.
  - confidence: 0.9
- **[P2] Normalize cron names before duplicate checks:** `src/cron/service/ops.ts:321`
  - body: The lookup compares `input.name` before `createJob()` trims and validates it, and update has the same issue with `patch.name`. A caller can add `" foo "` after `"foo"`; `findJobByName` misses the raw value, then the service stores both as `"foo"`. Normalize the candidate name before add/update collision checks and cover whitespace names.
  - confidence: 0.86
