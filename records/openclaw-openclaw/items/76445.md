## Summary

Keep open. The PR targets a valid cron duplicate-name bug, but the current patch still misses whitespace-normalized collisions and throws duplicate-name failures as ordinary service errors that Gateway cron.add/update rethrow instead of returning INVALID_REQUEST responses.

## What This Changes

The PR adds a cron job-name lookup helper, rejects duplicate names in cron service add/update, adds duplicate-name service tests, and records the fix in CHANGELOG.md.

## Best Possible Solution

Land a service-level duplicate-name guard that compares normalized names, returns duplicate collisions through the existing Gateway validation path, and covers both service and server-method callers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main add/update allow duplicate stored names, and the PR diff shows the raw-name lookup and Gateway error-mapping regressions without needing a live cron store.

## Solution Assessment

No. The service-layer boundary is the right place, but this implementation should normalize the candidate before lookup and make duplicate-name failures participate in the existing INVALID_REQUEST response contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:323-325`
  - body: The new collision path throws a plain Error from the cron service. Gateway cron.add/update only convert TypeError and RangeError into INVALID_REQUEST responses and intentionally rethrow ordinary Errors, so RPC/CLI/tool callers will not receive the actionable duplicate-name response promised by the PR. Use the existing validation-error path and add server-method coverage.
  - confidence: 0.9
- **[P2] Compare normalized cron names before checking duplicates:** `src/cron/service/ops.ts:321`
  - body: This lookup compares input.name before createJob trims and validates it, and update has the same issue with patch.name. A caller can add " foo " after "foo"; findJobByName misses the raw value, then createJob stores both as "foo". Normalize first and cover whitespace names.
  - confidence: 0.86
