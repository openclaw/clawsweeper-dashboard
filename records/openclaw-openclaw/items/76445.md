## Summary

Keep open: current main still has the cron duplicate-name bug, but this patch is not merge-ready because it misses normalized-name collisions, throws through the wrong error path for Gateway callers, and lacks contributor real-behavior proof.

## What This Changes

The PR adds service-level cron job name lookup checks for add/update, duplicate-name service tests, and a changelog entry.

## Best Possible Solution

Land a corrected service-layer uniqueness guard that compares normalized candidate names and reports duplicate collisions through the existing Gateway INVALID_REQUEST path, with --replace or duplicate-list warnings handled as separate UX follow-ups if desired.

## Reproduction Assessment

Yes. Source inspection shows current main appends new cron jobs without a sibling-name check and applies update name patches before any collision check, while disabled jobs are hidden by default listing unless includeDisabled is true.

## Solution Assessment

No. The service-layer boundary is right, but this patch should compare normalized names and use the existing validation-error mapping path instead of raw-name lookups and plain Error throws.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Compare normalized cron names before checking duplicates:** `src/cron/service/ops.ts:322`
  - body: The new lookup checks raw input.name before createJob trims it, and the update path checks raw patch.name before applyJobPatch normalizes it. A caller can still rename a job to " foo " when "foo" already exists, then storage keeps both as "foo".
  - confidence: 0.88
- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:324-326`
  - body: The collision path throws a plain Error, but Gateway cron.add and cron.update only map TypeError and RangeError service failures to INVALID_REQUEST and rethrow ordinary errors. Use the validation error path so CLI, RPC, and agent-tool callers receive the actionable duplicate-name response promised here.
  - confidence: 0.9
