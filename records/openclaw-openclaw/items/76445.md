## Summary

Keep open. Current `main` still accepts duplicate cron names, and this PR targets the right service layer, but the branch is not mergeable as-is because it checks raw names before normalization, throws ordinary service errors that Gateway rethrows, and lacks after-fix real behavior proof from the contributor.

## What This Changes

The PR adds a cron service name lookup, rejects duplicate names on add/update, adds duplicate-name service tests, and adds a changelog entry.

## Best Possible Solution

Land a corrected service-layer uniqueness guard that compares normalized cron names and returns duplicate collisions through the existing Gateway `INVALID_REQUEST` path, while tracking `--replace` or duplicate-list warnings separately if maintainers want that UX.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: duplicate add reaches the unconditional `jobs.push`, and rename applies a normalized `patch.name` before any sibling-name collision check exists.

## Solution Assessment

No. The service-layer boundary is right, but this implementation should compare normalized candidate names and throw through the existing validation-error class that Gateway maps to `INVALID_REQUEST`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Compare normalized cron names before checking duplicates:** `src/cron/service/ops.ts:321`
  - body: The new lookup checks `input.name` before `createJob` trims and validates it, and the update path does the same with `patch.name` before `applyJobPatch` normalizes it. A caller can add `" foo "` after `"foo"`; the lookup misses the raw value, then storage keeps both as `"foo"`.
  - confidence: 0.88
- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:323-325`
  - body: The collision path throws a plain `Error`, but Gateway `cron.add` and `cron.update` only map `TypeError` and `RangeError` to `INVALID_REQUEST` and rethrow ordinary errors. Use the validation error path so CLI, RPC, and agent-tool callers receive the actionable duplicate-name response promised by the PR.
  - confidence: 0.9
