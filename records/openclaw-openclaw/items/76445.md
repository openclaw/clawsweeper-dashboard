## Summary

Keep open. The PR targets a valid cron bug, but the patch currently lets duplicate-name errors escape the Gateway as ordinary service failures and still compares raw service input before cron name normalization.

## What This Changes

The PR adds cron service duplicate-name checks for add/update, a new duplicate-name service test file, and a changelog entry for the user-facing cron fix.

## Best Possible Solution

Land a shared cron service uniqueness guard over normalized names and a Gateway validation response path, with regression coverage across service and server-method callers; keep replace/listing UX as explicit follow-up if not included.

## Reproduction Assessment

Yes. Source inspection shows current main accepts duplicate cron names through add and update, and the PR diff exposes a clear gateway error-mapping regression without needing a live cron store.

## Solution Assessment

No. The service-layer boundary is the right place to solve this, but this implementation should normalize the candidate name before collision checks and return duplicate-name collisions through the existing Gateway INVALID_REQUEST path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return duplicate-name collisions as validation errors:** `src/cron/service/ops.ts:323-325`
  - body: The new collision path throws a plain `Error` from the cron service. The Gateway `cron.add`/`cron.update` handlers only convert `TypeError` and `RangeError` into `INVALID_REQUEST` responses and intentionally rethrow ordinary `Error`s, so CLI, RPC, and agent-tool callers will not receive the actionable duplicate-name error promised by the PR. Please route this collision through a validation error path and add server-method coverage.
  - confidence: 0.9
- **[P2] Compare normalized cron names before checking duplicates:** `src/cron/service/ops.ts:321`
  - body: This lookup compares `input.name` before `createJob` trims and validates it. A service caller can add `" foo "` after `"foo"`; `findJobByName` misses the raw value, then `createJob` stores both as `"foo"`. Normalize the candidate name before add/update collision checks and cover whitespace names.
  - confidence: 0.86
