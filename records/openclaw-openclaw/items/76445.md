## Summary

Keep open: current main still allows duplicate cron names and no safer merged replacement was found, but this PR should not merge as submitted because its validation compares raw names before the cron service trims names for storage.

## What This Changes

The PR adds cron service-layer duplicate-name validation for job add/update paths, a new duplicate-name test file, and a changelog entry.

## Best Possible Solution

Enforce cron name uniqueness in the service against normalized stored names, keep focused service coverage for add and rename collisions, and let the existing gateway/CLI/agent callers surface the same service error.

## Reproduction Assessment

Yes from source inspection: current main `add` and `update` do not check sibling cron names before storing or renaming jobs. I did not run a live CLI repro because this review was read-only and could not create runtime state artifacts.

## Solution Assessment

No as submitted. The service layer is the right owner boundary, but the check must compare normalized candidate names and the branch must be refreshed before it is a safe merge path.
