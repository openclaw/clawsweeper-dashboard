## Summary

Keep open: the PR is a focused Telegram warning fix with sufficient terminal proof for the reported valid-default path, but its guard also suppresses the fallback warning when `defaultAccount` is configured to an invalid account id.

## What This Changes

The PR stores the normalized Telegram `defaultAccount`, uses it in the missing-default warning predicate, and adds a regression test for a multi-account `charles`/`hermes` setup.

## Best Possible Solution

Keep this resolver/test shape, but make suppression depend on the normalized configured default actually being selected and cover the invalid-default fallback case.

## Reproduction Assessment

Yes, by source inspection: current main resolves `defaultAccount` but still warns when that valid default is also the first sorted account id and there is no `accounts.default`. The PR body also supplies after-fix terminal proof for that resolver path.

## Solution Assessment

No. The patch is in the right file and fixes the reported valid-default case, but the guard should suppress only when the configured default is actually honored rather than when any non-empty `defaultAccount` is present.
