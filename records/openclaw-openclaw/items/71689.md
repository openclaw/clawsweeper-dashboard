## Label Justifications

- `P2`: Malformed local SQLite state can break durable task restore, but the affected surface is bounded to local persisted state recovery rather than a whole-runtime emergency.
- `impact:data-loss`: The issue concerns corrupted persisted SQLite task data and recovery choices that may replace the active database after preserving malformed state.
- `impact:session-state`: The affected `task_runs` and `task_delivery_state` rows are durable task/session workflow state restored during startup.

## Summary

Keep open. Current main and v2026.6.10 still close and rethrow malformed shared SQLite initialization while task-registry restore only logs the failure; both recovery PRs are closed unmerged and the remaining recovery policy needs maintainer choice.

## What This Changes

The issue asks OpenClaw to recover or safely handle task-registry restore failures when the SQLite state backing task runs is malformed.

## Best Possible Solution

Maintainers should choose a shared-state SQLite corruption policy, then land focused recovery that preserves malformed DB evidence and covers the task restore path with state/task regression tests.

## Reproduction Assessment

Yes, from source. A malformed shared state DB reaches `openOpenClawStateDatabase()` through `loadTaskRegistryStateFromSqlite()`, and current tests assert malformed DB initialization throws instead of recovering; I did not run a live startup repro in this read-only checkout.

## Solution Assessment

Unclear as a final product choice. The shared state opener is the likely owner boundary, but maintainers still need to choose availability-first quarantine/fresh-state behavior versus fail-closed or doctor-driven recovery.
