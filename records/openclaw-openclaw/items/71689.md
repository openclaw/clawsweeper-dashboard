## Summary

Keep open. Current main still has the reported warning-only restore behavior for malformed task-registry SQLite files: the store opens and reads `runs.sqlite` directly, and the registry catch path only logs the restore failure. The related #73517/#74406 maintenance-cache work is adjacent but does not cover malformed-image detection, quarantine, or rebuild.

## What This Changes

The issue requests automatic recovery when task registry restore fails because `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is a malformed SQLite database.

## Best Possible Solution

Add a narrow task-registry SQLite corruption fallback in the task-registry store path: detect malformed/corrupt SQLite errors during open or load, close any partial handle, preserve `runs.sqlite` plus `-wal` and `-shm` sidecars in a timestamped quarantine location, recreate a fresh registry database, and emit an operator-facing warning. Keep record salvage and the #73517 maintenance-cache work separate unless maintainers explicitly expand scope.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` exercises `ensureTaskRegistryReady()` through `loadTaskRegistryStateFromSqlite()` and currently reaches the warning-only catch in `restoreTaskRegistryOnce()`; the issue also includes `PRAGMA integrity_check` evidence from the affected database.

## Solution Assessment

Unclear for the suggested `.recover` detail because it may depend on sqlite CLI behavior and a product decision about salvaging private task records. The best first fix is quarantine-and-recreate with preserved sidecars and focused tests, leaving salvage as a follow-up if maintainers want it.
