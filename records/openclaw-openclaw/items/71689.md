## Summary

Keep open. Current main still opens and restores `tasks/runs.sqlite` directly and only logs restore failures, while the focused quarantine/recreate PR was closed unmerged for contributor queue reasons rather than because the fix landed.

## What This Changes

The issue requests corruption-aware task-registry recovery when malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` breaks gateway startup restore.

## Best Possible Solution

Implement a narrow availability-first recovery that classifies known SQLite corruption, preserves `runs.sqlite` plus WAL/SHM sidecars with a manifest, recreates a clean task store, and leaves salvage/root-cause prevention for follow-ups.

## Reproduction Assessment

Yes. I did not mutate the checkout to create a corrupt DB, but source inspection shows a malformed `runs.sqlite` follows the default SQLite open/load path into the warning-only restore catch, and the issue plus the closed PR proof include malformed-image and integrity-check evidence.

## Solution Assessment

Yes. Quarantine-and-recreate is the narrow maintainable availability fix; automatic `.recover`, `REINDEX`, payload caps, and root-cause writer/checkpoint work should be deliberate follow-ups.
