## Summary

Keep open: current main still only logs task-registry SQLite restore failures and has no malformed database quarantine, rebuild, or recovery path. The related #73517/#74406 path is a separate maintenance hot-loop fix and explicitly leaves broader malformed SQLite recovery to this issue.

## What This Changes

The issue reports gateway startup task registry restore failures when `tasks/runs.sqlite` is corrupted and asks for automated detection, snapshot/quarantine, recovery or rebuild, and clearer operator diagnostics.

## Best Possible Solution

Add a task-store-owned corruption fallback around `runs.sqlite` open/load: classify SQLite corruption errors, close any cached handle and WAL timer, snapshot/quarantine `runs.sqlite`, `runs.sqlite-wal`, and `runs.sqlite-shm` with private permissions, recreate a clean registry, and emit a clear operator diagnostic. Keep the #73517 maintenance hot-loop cache separate, and cover the malformed-image path with focused store tests, docs if operator behavior changes, and a changelog fix entry.

## Reproduction Assessment

Yes. The issue gives concrete steps using a corrupted `tasks/runs.sqlite`, current source shows that restore errors are only logged, and a temp `node:sqlite` smoke against an invalid DB reproduced the same `ERR_SQLITE_ERROR` class; the exact production btree corruption remains private.

## Solution Assessment

Yes, the requested store-level recovery direction is the narrowest maintainable fix. It should live in the task registry SQLite store rather than gateway startup or maintenance logic, with conservative quarantine/recreate behavior and no bundled dependency on an external recovery binary unless maintainers approve it.
