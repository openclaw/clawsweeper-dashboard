## Label Justifications

- `P2`: Malformed local SQLite state can break task-registry restore, but the blast radius is bounded to persisted local state recovery rather than a system-wide emergency.
- `impact:data-loss`: The issue concerns corrupted persisted SQLite task data and recovery choices that may replace the active DB after preserving the malformed store.
- `impact:session-state`: The affected task_runs and task_delivery_state rows are durable task/session workflow state restored on gateway startup.

## Summary

Keep open. Current main and v2026.6.10 still route task restore through the shared SQLite opener, which closes and rethrows malformed database initialization errors without quarantine, recovery, or retry; both linked recovery PRs are closed unmerged.

## What This Changes

Recover or quarantine malformed task-registry/shared-state SQLite so durable task restore no longer fails during gateway startup.

## Best Possible Solution

Maintainers should choose a shared-state SQLite corruption policy, then land focused recovery that preserves malformed DB evidence and covers the task restore path with state/task regression tests.

## Reproduction Assessment

Yes, from source. A malformed shared state DB reaches openOpenClawStateDatabase through loadTaskRegistryStateFromSqlite, and current tests assert malformed DB initialization throws instead of recovering; I did not run a live startup repro.

## Solution Assessment

Unclear as a final product choice. The shared state opener is the right likely owner boundary, but maintainers still need to choose availability-first quarantine/fresh-state behavior versus fail-closed or doctor-driven recovery.
