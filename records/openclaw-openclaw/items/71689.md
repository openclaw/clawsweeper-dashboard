## Summary

Keep open. Current main still routes task-registry startup restore through the SQLite store and only logs restore exceptions; it does not detect malformed database errors, quarantine `runs.sqlite` plus WAL/SHM sidecars, rebuild/recover a clean registry, or expose a task-registry restore-failed audit signal. The related #73517/#74406 work is useful but targets maintenance hot-loop cost, not malformed SQLite startup restore.

## What This Changes

The issue requests automatic recovery for task-registry startup restore failures when `tasks/runs.sqlite` is a malformed SQLite database.

## Best Possible Solution

Add task-registry-owned SQLite corruption handling in the store/restore/audit path: detect corrupt or not-a-database SQLite failures during open, schema setup, statement creation, and row load; close any partially opened handle; preserve the database plus WAL sidecars under a timestamped quarantine/snapshot; recover or reopen an empty clean registry when safe; and surface a task-registry restore-failed audit/operator diagnostic.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source still routes startup restore through the SQLite load path with only a catch-and-warn. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the task-registry store/restore/audit code; the adjacent #73517/#74406 maintenance-cache work addresses stale-record CPU cost and should not be treated as a malformed SQLite recovery fix.
