## Summary

Keep open: current main still routes task registry restore through the SQLite store and only logs restore failures. It does not classify malformed SQLite images, quarantine or rebuild `runs.sqlite`, retry with a clean/recovered store, or expose a task-registry restore failure through audit/diagnostics.

## What This Changes

The issue asks OpenClaw to recover gracefully when a corrupted `tasks/runs.sqlite` prevents durable task registry restore during Gateway startup.

## Best Possible Solution

Add task-registry-owned SQLite corruption handling around open, schema setup, statement creation, and snapshot load: close any partial handle, preserve `runs.sqlite` plus WAL/SHM sidecars under a timestamped quarantine, reopen a clean store or best-effort recovered store, and surface an operator/audit diagnostic for the restore failure.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR`, and current source inspection confirms startup still loads that SQLite file through a catch-and-warn restore path. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. A narrow task-registry store/restore/audit fix is the maintainable solution; the adjacent #73517 maintenance hot-loop report may share symptoms but does not directly handle malformed SQLite restore failure.
