## Summary

Keep open. Current main still opens and restores `tasks/runs.sqlite` directly and only logs restore failures, while the related #73517 fix shipped a maintenance hot-loop improvement rather than malformed SQLite image recovery.

## What This Changes

The issue requests recovery for gateway task registry startup failures when `tasks/runs.sqlite` is a malformed SQLite database.

## Best Possible Solution

Task registry storage should fail open by quarantining the malformed database plus sidecars, recreating a clean store, and warning operators, with tests for corrupt primary DB and sidecar preservation.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` follows `resolveTaskRegistrySqlitePath()` to `openTaskRegistryDatabase()` and `loadTaskRegistryStateFromSqlite()`, then falls into `restoreTaskRegistryOnce()`'s warning-only catch path.

## Solution Assessment

No for a `.recover`-first implementation without policy review. The narrow maintainable fix is corruption detection plus quarantine-and-recreate, leaving record salvage as an explicit later decision.

In `src/tasks/task-registry.store.sqlite.ts`, detect known SQLite corruption/malformed-image failures during open/schema/load, close any partial handles, preserve `runs.sqlite` plus `runs.sqlite-wal` and `runs.sqlite-shm` under a timestamped quarantine name or directory, recreate a fresh empty registry DB, and emit an operator-visible warning that includes the preserved path without exposing task contents.
