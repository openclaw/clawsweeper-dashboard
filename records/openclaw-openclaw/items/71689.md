## Summary

Keep open. Current main still opens and restores `tasks/runs.sqlite` directly and only logs restore failures, while the newer RCA comment adds useful corruption-shape evidence that is not covered by the closed #73517 maintenance hot-loop fix.

## What This Changes

Detect malformed/corrupt task registry SQLite on gateway restore, preserve the bad database, recreate a usable registry, and surface operator diagnostics.

## Best Possible Solution

Add an availability-first corruption fallback in the task registry SQLite store: classify malformed/corrupt SQLite failures, close any partial handle, quarantine `runs.sqlite` plus sidecars, recreate a fresh database, and emit a clear operator warning, with deeper payload/write-order RCA handled as follow-up if needed.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` follows `resolveTaskRegistrySqlitePath()` to `openTaskRegistryDatabase()` and `loadTaskRegistryStateFromSqlite()`, then falls into `restoreTaskRegistryOnce()`'s warning-only catch; the report and later RCA provide independent integrity-check evidence.

## Solution Assessment

Yes for the quarantine/recreate direction; unclear for the suggested `.recover` detail. The narrowest maintainable first fix is to restore gateway availability while preserving the bad files for diagnosis, not to invent a risky partial-salvage policy in the startup path.
