## Summary

Keep open. Current main still opens and restores the durable task registry directly from `tasks/runs.sqlite`; a malformed SQLite image still reaches the warning-only restore catch path without classification, quarantine, sidecar handling, recreation, retry, or operator-facing recovery.

## What This Changes

Issue requests a recovery path for malformed `tasks/runs.sqlite` so gateway startup can recover durable task registry state instead of repeatedly warning on SQLite corruption.

## Best Possible Solution

Add a narrow task-registry corruption fallback in the SQLite store: detect known malformed/corrupt SQLite open or restore errors, close any partial handle, move `runs.sqlite` plus `-wal` and `-shm` into a timestamped quarantine location, recreate a fresh registry database, and emit a clear operator warning while leaving record salvage for a separate decision.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should drive `ensureTaskRegistryReady()` through `loadTaskRegistryStateFromSqlite()` and hit the current warning-only catch path; the issue also includes independent SQLite integrity-check evidence.

## Solution Assessment

Unclear for the suggested `.recover` detail. The best first fix is quarantine-and-recreate with sidecar preservation and focused tests, because it restores gateway availability without inventing a risky partial-salvage policy.
