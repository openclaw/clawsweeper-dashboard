## Summary

Keep open. Current main still opens and restores the task registry SQLite database directly, and a malformed `runs.sqlite` still falls through to a warning-only restore failure without corruption detection, quarantine, rebuild, or operator-facing recovery.

## What This Changes

The issue requests automatic recovery when the durable task registry at `tasks/runs.sqlite` is malformed and gateway startup cannot restore it.

## Best Possible Solution

Add a narrow task-registry SQLite corruption fallback: classify malformed/corrupt SQLite errors during open or restore, close any partial handle, preserve `runs.sqlite` plus `-wal` and `-shm` sidecars in a timestamped quarantine location, recreate a fresh registry database, and emit a clear operator warning. Keep stale-record maintenance caching separate from this recovery path.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should exercise `ensureTaskRegistryReady()` through `loadTaskRegistryStateFromSqlite()` and hit the current warning-only catch; the issue also includes independent SQLite integrity-check evidence.

## Solution Assessment

Unclear for the proposed `.recover` detail. The narrowest maintainable first fix is quarantine-and-recreate with preserved sidecars and focused tests, leaving row salvage as a separate maintainer decision.
