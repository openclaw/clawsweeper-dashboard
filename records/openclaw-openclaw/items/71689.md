## Summary

Keep open. Current main still restores the task registry by directly opening `tasks/runs.sqlite` and logging a warning on restore failure, with no malformed-SQLite classification, sidecar quarantine, rebuild, retry, or operator-facing recovery path.

## What This Changes

Add a recovery path for malformed `tasks/runs.sqlite` so durable task registry startup can preserve the bad store, recreate usable state, and report the recovery clearly.

## Best Possible Solution

Add a narrow task-registry corruption fallback in the SQLite store: detect known malformed/corrupt SQLite open or restore errors, close any partial handle, move `runs.sqlite` plus `-wal` and `-shm` into a timestamped quarantine directory, recreate a fresh registry database, and emit a clear operator warning. Treat record salvage as a separate maintainer decision after the gateway can recover safely.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should drive `ensureTaskRegistryReady()` through `loadTaskRegistryStateFromSqlite()` and hit the current warning-only catch path; the issue also includes independent SQLite integrity-check evidence.

## Solution Assessment

Unclear for the proposed `.recover` detail. The narrowest maintainable fix is quarantine-and-recreate with preserved sidecars and focused tests, leaving data salvage policy for a separate review.
