## Summary

Keep open. Current main still opens and reads the task registry SQLite database directly and only logs `Failed to restore task registry` when restore throws; there is no corruption detection, quarantine, rebuild, or recovery path for a malformed `runs.sqlite`. The related #73517/#74406 work is about task-maintenance hot-loop caching and explicitly leaves this broader malformed-SQLite recovery behavior tracked here.

## What This Changes

The issue requests a recovery path when gateway startup restores the durable task registry from a malformed `tasks/runs.sqlite` SQLite image.

## Best Possible Solution

Add a narrow task-registry SQLite corruption fallback: detect malformed/corrupt SQLite errors during restore/open, close any partial handle, preserve `runs.sqlite` plus `-wal`/`-shm` sidecars in a timestamped quarantine location, recreate a fresh registry database, and emit a clear operator-facing warning. Keep the maintenance hot-loop cache tracked by #73517 separate.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should exercise `ensureTaskRegistryReady()` -> `loadTaskRegistryStateFromSqlite()` and hit the current warning-only catch in `restoreTaskRegistryOnce()`; the issue also includes `PRAGMA integrity_check` evidence from the affected database.

## Solution Assessment

Unclear for the proposed `.recover` detail; that may require a sqlite CLI dependency and policy choices. The narrowest maintainable first fix is quarantine-and-recreate with preserved sidecars and focused tests, leaving any record salvage as a follow-up maintainer decision.
