## Summary

Keep open. Current main still lacks malformed `tasks/runs.sqlite` recovery: restore failures are logged and the bad database remains in place, while the closed #73517 work only covers maintenance hot-loop performance on stale registries.

## What This Changes

Recover or rebuild the durable task registry when `tasks/runs.sqlite` is malformed instead of leaving repeated startup restore warnings.

## Best Possible Solution

Add task-registry SQLite corruption classification with preserve-before-mutate quarantine of `runs.sqlite` plus sidecars, recreate a fresh store for availability, and emit an operator-visible warning; keep deeper salvage and writer-root-cause diagnostics as follow-ups.

## Reproduction Assessment

Yes for a source-level reproduction path. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` reaches `openTaskRegistryDatabase()` and `loadTaskRegistryStateFromSqlite()`, then `restoreTaskRegistryOnce()` only logs the restore failure; the report and follow-up RCA include SQLite integrity evidence.

## Solution Assessment

Yes for the narrow quarantine-and-recreate first fix. The suggested `.recover` salvage and payload/write-path RCA work may be useful, but they should not block an availability-first repair that preserves the bad DB for later inspection.
