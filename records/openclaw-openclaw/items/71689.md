## Summary

Keep open. Current main still restores the durable task registry by loading the SQLite store and catches restore failures only as a warning; it does not classify malformed SQLite images, preserve/rebuild the bad database, or expose a restore-failed audit signal. The related maintenance hot-loop report is adjacent context, not a replacement for malformed `runs.sqlite` recovery.

## What This Changes

Add durable task-registry recovery for malformed or corrupt `tasks/runs.sqlite` so gateway startup can preserve, recover, or rebuild registry state instead of only logging restore failure.

## Best Possible Solution

Add task-registry-owned SQLite corruption handling around open, schema setup, statement creation, and row load: close any partial handle, preserve `runs.sqlite` plus WAL/SHM sidecars under a timestamped quarantine, reopen a clean store or best-effort recovered store, and expose an operator/audit diagnostic for the restore failure.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR`, and current source inspection confirms gateway startup still routes through the SQLite load path with only a catch-and-warn. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. A narrow task-registry store/restore/audit fix is the maintainable path; broad startup migrations or the adjacent #73517 maintenance-cache work would not directly address malformed SQLite restore failure.
