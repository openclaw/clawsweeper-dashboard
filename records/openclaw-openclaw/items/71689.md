## Summary

Keep open. Current main still opens and restores the task registry SQLite database directly and only logs restore failures; the closed #73517 fix covers stale-registry maintenance hot-looping, not malformed SQLite image recovery.

## What This Changes

The issue requests gateway task registry startup recovery when `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is a malformed SQLite database.

## Best Possible Solution

Add conservative task-registry SQLite corruption handling: detect known malformed/corrupt SQLite failures, close partial handles, quarantine the database plus sidecars, recreate a clean store, and emit an operator-visible warning.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` follows `resolveTaskRegistrySqlitePath()` to `openTaskRegistryDatabase()` to `loadTaskRegistryStateFromSqlite()` and lands in the warning-only restore catch; the report also includes independent SQLite integrity-check evidence.

## Solution Assessment

No for automatic `.recover` as the default first fix. The narrowest maintainable solution is corruption detection plus quarantine-and-recreate, with explicit sidecar preservation and clear operator reporting.
