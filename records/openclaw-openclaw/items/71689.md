## Summary

Keep open. Current main still opens and restores the durable task registry directly from `tasks/runs.sqlite`; malformed SQLite restore errors still land in a warning-only catch path without corruption classification, quarantine, fresh-database retry, or focused regression coverage.

## What This Changes

The issue asks for task registry startup recovery when `tasks/runs.sqlite` is a malformed SQLite database.

## Best Possible Solution

Add a narrow SQLite-store recovery path that detects malformed/corrupt task registry database errors, closes any partial handle, quarantines `runs.sqlite` plus `-wal` and `-shm`, recreates a fresh registry database, and emits a clear operator warning while leaving record salvage for a separate decision.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should drive gateway startup through `ensureTaskRegistryReady()` and `loadTaskRegistryStateFromSqlite()` into the current warning-only restore catch; the issue also includes SQLite integrity-check evidence.

## Solution Assessment

Unclear for the suggested `.recover` detail. The safest first repair is quarantine-and-recreate with sidecar preservation and regression tests, because that restores gateway availability without inventing partial-salvage semantics.
