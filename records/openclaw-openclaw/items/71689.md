## Summary

Keep open. Current main still opens and restores `tasks/runs.sqlite` directly and only logs restore failures; the related #73517 fix addresses maintenance hot-looping on stale registries, not malformed SQLite image recovery.

## What This Changes

Detect and recover from malformed durable task registry SQLite databases at gateway startup instead of only logging restore failures.

## Best Possible Solution

Add an availability-first corruption fallback in task registry SQLite storage: classify known corrupt/malformed SQLite failures, close partial handles, quarantine `runs.sqlite` plus sidecars, recreate a fresh database, and emit an operator-visible warning; defer record salvage unless maintainers explicitly want it.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` follows `resolveTaskRegistrySqlitePath()` -> `openTaskRegistryDatabase()` -> `loadTaskRegistryStateFromSqlite()` and falls into `restoreTaskRegistryOnce()`'s warning-only catch; the report also includes independent SQLite integrity-check evidence.

## Solution Assessment

Unclear for the suggested `.recover` detail. The narrowest maintainable first fix is corruption detection plus quarantine-and-recreate with sidecar preservation, because that restores gateway availability without inventing a risky partial-salvage policy.
