## Summary

Keep open. Current main still restores the task registry directly from the SQLite store and only logs restore failures; there is no corruption-specific detection, quarantine, sidecar preservation, recreate, or operator recovery path for a malformed `tasks/runs.sqlite`.

## What This Changes

The issue requests task-registry startup recovery when `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is malformed, so durable task restore does not fail with warning-only SQLite corruption errors.

## Best Possible Solution

Add a store-level task-registry SQLite corruption fallback: classify malformed/corrupt SQLite errors during open or snapshot load, close any partial handle, preserve `runs.sqlite` plus `-wal` and `-shm` sidecars in a timestamped quarantine location, recreate an empty registry database, and emit a clear operator-facing warning. Keep record salvage via `.recover` as a separate maintainer decision unless it can be done without new runtime dependencies.

## Reproduction Assessment

Yes. A malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` should exercise `ensureTaskRegistryReady()` through `loadTaskRegistryStateFromSqlite()` and currently reaches the warning-only restore catch; the issue also includes independent `PRAGMA integrity_check` corruption evidence.

## Solution Assessment

Unclear for the proposed `.recover` detail. The best first fix is a narrow SQLite store corruption classifier with quarantine-and-recreate behavior and focused tests, rather than adding a sqlite CLI salvage dependency in the startup path.
