## Summary

Keep open. Current main and latest release still restore tasks through the shared state SQLite opener, which closes and rethrows malformed database initialization errors without corruption classification, quarantine, recovery, or retry; the two linked recovery PRs are closed unmerged.

## What This Changes

The issue requests startup recovery for malformed persisted SQLite task-registry state instead of repeated task restore failure.

## Best Possible Solution

Land or sponsor a shared state DB corruption policy that preserves malformed DB evidence, makes the availability/data-retention tradeoff explicit, and covers malformed task-registry restore with focused state/task tests.

## Reproduction Assessment

Yes, from source. A malformed shared state DB reaches `openOpenClawStateDatabase()` through `loadTaskRegistryStateFromSqlite()`, and current tests assert malformed DB initialization throws rather than recovers; I did not run a live repro in this read-only review.

## Solution Assessment

No, not yet. The shared state opener is the right likely owner boundary, but maintainers still need to choose the permanent availability-first quarantine/fresh-state behavior versus fail-closed or doctor-driven recovery.
