## Summary

Keep open. Current main still lacks a built-in Accessible Control UI theme preset; the shipped surface is Claw, Knot, Dash, and one browser-local tweakcn import slot, which does not satisfy a first-class high-contrast color-vision-deficiency-safe preset. The broader open theme-studio issue is related but does not specifically cover this accessibility preset.

## What This Changes

Add a built-in Accessible theme preset to the Control UI theme picker using CSS variable overrides for high contrast and color-vision-deficiency-safe semantic colors.

## Best Possible Solution

Add a focused built-in Accessible theme family through the existing Control UI theme contract, first-paint bootstrap, Appearance and Quick Settings pickers, CSS variable blocks, targeted tests, docs, and a user-facing changelog entry while keeping the tweakcn custom import as a complementary local customization path.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. We do have a high-confidence current-main verification path: inspect the theme contract, bootstrap allowlist, picker lists, CSS theme blocks, docs, and tests, all of which omit Accessible.

## Solution Assessment

Yes, adding a built-in preset via the existing theme-family pattern is the narrowest maintainable solution. A browser-local tweakcn import is not an adequate substitute because it is not shipped, not reusable across users/devices, and not documented as an accessibility preset.
