## Summary

Keep open. Current main documents and implements a custom `toolsAllow` path for non-core memory plugins, but it still depends on manifest-discovered, enabled, non-colliding plugin tools; the report lacks the custom plugin manifest/config needed to prove a core bug versus an unsupported tool-ownership setup, and the remaining decision is broader memory-provider/tool-owner policy.

## What This Changes

The issue requests that Active Memory resolve recall tools from third-party/custom memory plugins instead of failing when the embedded sub-agent cannot materialize the configured memory tool allowlist.

## Best Possible Solution

Decide the intended memory-tool owner model for Active Memory, then either document the required unique `toolsAllow`/slot/manifest configuration or add a narrow manifest-aware memory recall resolver that preserves the hidden sub-agent's restricted tool surface.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this issue alone. Source inspection shows the embedded allowlist fails closed when no manifest-available plugin tool matches, but the custom plugin manifest/config is missing.

## Solution Assessment

Unclear. Parent-context inheritance is likely too broad; the safer direction is either explicit documentation of the existing manifest/unique-tool contract or a narrow memory-provider owner resolution path.
