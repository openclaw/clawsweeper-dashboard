## Summary

Keep open. Current main supports explicit custom Active Memory tool names only through manifest-discovered plugin tools and a single memory-slot owner; this report asks for broader third-party/hybrid memory ownership, but it lacks the custom plugin manifest/config needed to prove a core bug rather than a contract or product gap.

## What This Changes

The issue asks for Active Memory’s embedded recall sub-agent to work with third-party or custom memory backends instead of being effectively tied to the stock memory-core tool path.

## Best Possible Solution

Decide the intended Active Memory recall-tool ownership model, then either document the required memory slot/manifest/config setup or add a narrow memory-only resolver that can select a third-party recall owner without inheriting the parent agent’s whole tool context.

## Reproduction Assessment

No high-confidence current-main reproduction path yet. Source inspection shows plausible failure modes for manifest-discovery gaps or duplicate memory tool ownership, but the custom plugin manifest and effective plugin config are missing.

## Solution Assessment

Unclear. The broad parent-context inheritance suggested in the issue is riskier than necessary; the safer direction is maintainer-defined memory-tool ownership or a docs/config clarification around `contracts.tools`, `plugins.slots.memory`, and duplicate tool names.
