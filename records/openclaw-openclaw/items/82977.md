## Summary

Keep open. Current docs describe custom Active Memory recall tools as supported, but the source path still depends on embedded plugin-tool discovery through manifest contracts and plugin slot/enablement rules, so the report needs maintainer triage rather than cleanup closure.

## What This Changes

The issue asks Active Memory's hidden recall sub-agent to use third-party memory plugin tools instead of being limited to the stock memory-core context.

## Best Possible Solution

Decide the intended owner-selection model for Active Memory recall tools, then either document the required memory slot/manifest configuration or add a narrow plugin-aware recall-tool resolution path that preserves the hidden sub-agent's restricted tool surface.

## Reproduction Assessment

No high-confidence reproduction path yet. The issue includes the runtime error and environment, but not the custom plugin manifest/config needed to distinguish a core bug from an unsupported or misconfigured plugin contract.

## Solution Assessment

Unclear. The proposed parent-context inheritance is likely too broad; the safer direction is narrow memory-tool ownership resolution or a docs/config clarification around `contracts.tools`, `plugins.slots.memory`, and duplicate tool names.
