## Summary

Keep open. The discussion now has independent production evidence on v2026.5.12 and v2026.5.18 that Active Memory can lose plugin-backed memory recall in the embedded runner, but the comments also split the symptoms into registry materialization, model tool-calling capability, and provider-bridge behavior, so this needs maintainer/product triage rather than cleanup closure.

## What This Changes

The issue requests making Active Memory’s embedded recall runner work reliably with memory tools supplied by third-party or non-stock memory plugins instead of only stock memory-core-style tools.

## Best Possible Solution

Define the intended Active Memory memory-tool ownership model, then land a narrow embedded-runner/plugin-tool resolution fix plus diagnostics that distinguish missing registered tools from model/provider tool-call failures.

## Reproduction Assessment

Yes from source and reports, but not high-confidence in a single current-main local harness. The issue has multiple production reproductions and the source path clearly routes allowlisted memory tools through the embedded runner, but the remaining failures appear to involve at least registry, provider, and model-capability variants.

## Solution Assessment

Unclear. Parent-context inheritance would likely be too broad; the better solution is a narrow plugin-aware memory-tool materialization path with clearer diagnostics and provider/model capability checks.
