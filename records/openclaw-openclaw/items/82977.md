## Summary

Keep open. Current main still documents custom Active Memory recall tools as supported, the Active Memory path still runs a lightweight embedded sub-agent with an explicit memory-tool allowlist, and new production evidence shows the same failure with bundled memory-lancedb as well as third-party memory plugins.

## What This Changes

The issue asks for Active Memory embedded recall to materialize plugin-provided memory tools from third-party or alternate memory backends instead of only working reliably with the stock memory-core path.

## Best Possible Solution

Land or replace the focused plugin-tool materialization fixes so Active Memory can use the selected memory plugin's declared tools in the lightweight embedded runner while preserving explicit allowlists, plugin disablement, denylist, and manifest-contract boundaries.

## Reproduction Assessment

No live reproduction was run during this read-only review, but the issue now has concrete v2026.5.12 logs for both third-party and bundled memory-lancedb paths and current source still follows the reported embedded allowlist path. That is source-reproducible with medium confidence.

## Solution Assessment

Unclear. The issue is valid, but the parent-context inheritance proposal is too broad; the best fix is a narrow plugin-tool registry/materialization change or an explicit memory-slot contract decision that keeps the sub-agent tool surface restricted.
