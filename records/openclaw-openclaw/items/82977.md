## Summary

Keep open. Current main and v2026.5.12 support custom Active Memory recall tool names through `config.toolsAllow`, but the reported “standard memory tool name” case intersects with plugin manifest discovery, unique tool-name collisions, and memory-slot ownership, so this needs maintainer product/API triage rather than cleanup closure.

## What This Changes

The issue requests that Active Memory recall work with third-party memory or knowledge plugins instead of being effectively tied to stock memory-core tools.

## Best Possible Solution

Decide the supported third-party memory ownership model, then either document the existing unique-tool-name and memory-slot requirements clearly or add a narrow generic memory-provider resolution contract that preserves the recall sub-agent’s restricted tool surface.

## Reproduction Assessment

No high-confidence exact reproduction path yet. Source inspection shows the dependency on manifest-discovered, enabled, non-colliding plugin tools, but the issue does not include the custom plugin manifest/config or proof that the main agent’s visible tool is the intended third-party implementation.

## Solution Assessment

Unclear. Inheriting the parent agent’s full tool context is likely too broad; the safer maintainable direction is a narrow memory-provider/tool-owner contract or docs clarifying `contracts.tools`, unique tool names, and `plugins.slots.memory`.
