## Summary

Keep open: the issue now has credible latest-release Active Memory repro evidence with first-party `memory-lancedb`, and current `main` has partial related resolver fixes but not enough proof to close the broader custom-memory compatibility report.

## What This Changes

The issue asks Active Memory's embedded recall sub-agent to materialize tools from third-party or custom memory plugins instead of depending on the stock memory-core tool surface.

## Best Possible Solution

Define the supported Active Memory memory-tool owner model, then either make embedded recall resolve the selected memory plugin's declared tools safely or document the exact slot, manifest, and `toolsAllow` requirements with current-main proof.

## Reproduction Assessment

No high-confidence current-main reproduction path is established. The issue has strong v2026.5.12 production logs, including a bundled `memory-lancedb` repro, but current `main` also contains related plugin resolver fixes and no live current-main run was performed.

## Solution Assessment

Unclear. Broadly inheriting the parent agent tool context would be too wide; the safer solution is narrow selected-memory-plugin tool resolution, or a maintainer decision to document the supported slot/manifest configuration instead.
