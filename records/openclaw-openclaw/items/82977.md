## Summary

Keep open. Current `main` still routes Active Memory recall through a lightweight embedded agent with an explicit tool allowlist, and the latest discussion shows the same failure string can cover plugin-tool materialization, hidden-runner model capability, and provider bridge variants, so this needs maintainer triage rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make Active Memory recall work with memory tools provided by third-party or alternate memory plugins instead of only the stock memory-core path.

## Best Possible Solution

Define the intended Active Memory memory-tool ownership model, then either land a narrow embedded plugin-tool materialization fix with regression coverage or split the misleading diagnostic and provider-specific DeepSeek path into canonical follow-ups while preserving this issue as the umbrella until that decision is made.

## Reproduction Assessment

No high-confidence local reproduction was established in this review. The issue and comments provide credible production repro evidence, and current source confirms the embedded allowlist path, but the exact plugin manifest/config and provider setup are not available here.

## Solution Assessment

Unclear. Parent-context inheritance would likely broaden the hidden sub-agent surface too much; a narrower fix should preserve the restricted allowlist while correctly materializing eligible plugin-owned memory tools and improving diagnostics for model/provider no-tool-call cases.
