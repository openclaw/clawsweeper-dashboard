## Summary

Keep open. Current main still routes Active Memory through a restricted lightweight embedded run and plugin-tool materialization remains governed by manifest plus loaded-registry resolution; the new v2026.5.12 production reproduction and the related open PRs show this is not cleanup-closeable yet.

## What This Changes

The issue reports that Active Memory’s embedded recall agent cannot materialize allowlisted memory tools from third-party or non-stock memory plugins.

## Best Possible Solution

Keep the issue open until a narrow plugin-tool materialization fix lands that lets the hidden Active Memory sub-agent use only the configured memory tools from the selected or declared plugin, without inheriting the parent agent’s full tool context.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. The issue now has a second concrete production reproduction on v2026.5.12, and current source still shows the same restricted embedded Active Memory path and manifest-backed plugin-tool materialization boundary.

## Solution Assessment

Unclear as a final implementation. The broad parent-context inheritance suggestion would expand the hidden sub-agent’s tool surface, so the safer solution is a narrow plugin-tool registry/materialization fix or an explicit maintainer decision that a related open PR covers this memory-slot case.
