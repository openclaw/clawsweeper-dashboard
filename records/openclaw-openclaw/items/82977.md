## Summary

Keep open. Current main has custom Active Memory tool configuration and partial plugin-tool resolver safeguards, but the issue discussion now shows multiple unresolved failure modes around embedded plugin tool materialization, provider-specific tool bridges, and hidden-runner model capability, so this is not safe cleanup closure.

## What This Changes

Request that Active Memory's hidden recall sub-agent reliably use memory tools provided by third-party or non-default memory plugins instead of being tied to memory-core behavior.

## Best Possible Solution

Define the intended Active Memory memory-tool ownership contract, keep the hidden sub-agent's tool surface memory-scoped, and then fix or document the exact plugin/provider materialization paths that remain unsupported.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The issue has credible deployed reproductions on v2026.5.12 and v2026.5.18, and current source shows the same embedded toolsAllow and plugin-registry surfaces, but the exact failing current-main configuration still needs confirmation.

## Solution Assessment

Unclear. The proposed parent-context inheritance is too broad; the better path is a maintainer-owned memory-tool contract decision plus a narrow resolver/provider diagnostic fix that preserves the restricted sub-agent tool surface.
