## Summary

I found no safe close path yet: current docs still advertise custom Active Memory memory tools, current code still relies on the lightweight embedded runner plus plugin-tool resolver, and the newer comments add concrete live reproductions plus a narrower open DeepSeek/provider variant rather than showing this is fixed.

## What This Changes

The issue reports that Active Memory's embedded recall runner cannot reliably use memory tools provided by third-party or non-stock memory plugins.

## Best Possible Solution

Keep the issue open as the broad Active Memory/custom-memory compatibility tracker until maintainers decide the owner-selection model, then either land a narrow embedded-runner/plugin-tool materialization fix or split remaining provider-specific diagnostics into canonical linked issues.

## Reproduction Assessment

No current-main local reproduction was run in this read-only review, and the live reports are from v2026.5.12/v2026.5.18 before current-main registry changes. The issue nevertheless has credible deployment reproductions and current source still exercises the same lightweight embedded-runner/plugin-tool resolver path.

## Solution Assessment

Unclear. Parent-context inheritance is too broad for the hidden memory sub-agent; the maintainable direction is a scoped memory-tool/plugin ownership resolver plus clearer diagnostics for model-capability versus registry/provider failures.
