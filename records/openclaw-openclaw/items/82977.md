## Summary

Keep open. Current docs still promise custom Active Memory recall tools, current source still routes Active Memory through the lightweight embedded runner with a concrete `toolsAllow`, and the latest discussion adds current-release evidence plus narrower open follow-ups rather than proving this is fixed or safely duplicate-cleanable.

## What This Changes

The issue reports that Active Memory’s embedded recall runner cannot reliably use third-party or slot-selected memory plugin tools because the hidden runner can surface `No callable tools remain` even when the main runtime loads those tools.

## Best Possible Solution

Keep this open as the Active Memory/plugin-tool compatibility tracker, land or replace the narrow plugin-tool resolver fixes where applicable, and make a maintainer decision on model/provider diagnostics without broadening the hidden sub-agent beyond configured memory tools.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The source path and current-release comments make the failure source-reproducible with medium confidence, but the custom plugin manifest and provider/model split are still missing for a single definitive repro.

## Solution Assessment

No for the broad parent-context inheritance suggestion; that would expand the hidden recall runner's tool surface. The better solution is a narrow plugin-tool materialization and diagnostics path that preserves the configured memory-tool allowlist, plus docs or telemetry for model/provider-specific failures.
