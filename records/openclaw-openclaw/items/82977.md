## Summary

Keep open. The current source and latest release still route Active Memory through the lightweight embedded runner with an explicit memory-tool allowlist, while the discussion now shows multiple unresolved mechanisms rather than a single stale third-party-plugin misconfiguration.

## What This Changes

Active Memory should let its hidden recall sub-agent use memory tools from third-party or non-core memory plugins instead of only stock memory-core tools.

## Best Possible Solution

Split the observed failures into owned fixes: preserve explicit memory plugin tool materialization in embedded runs, clarify model/tool-call diagnostics, and land narrow provider-specific fixes without broadening hidden sub-agent access.

## Reproduction Assessment

No, not as a high-confidence local current-main reproduction. The issue has credible production logs and the current source still follows the same embedded toolsAllow/error path, but the exact custom plugin config and provider/model mechanism are not isolated in this checkout.

## Solution Assessment

Unclear. The safer direction is narrow resolver/provider diagnostics and tool-materialization fixes; inheriting the parent agent's whole tool context is likely too broad for a hidden Active Memory sub-agent.
