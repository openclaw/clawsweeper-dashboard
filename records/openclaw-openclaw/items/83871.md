## Summary

Keep this open: current main tracks active-run steering outcomes internally, but normal busy messages still do not expose accepted-steer vs fallback/follow-up status through diagnostics, `/queue` status, or a user/operator surface. The related docs and terminology issues are closed, but their comments explicitly preserve this remaining UX/diagnostics ask.

## What This Changes

Request to expose whether normal busy inbound messages were steered into the active run, deferred as follow-up, explicitly queued/collected, or interrupted.

## Best Possible Solution

Add one canonical busy-message outcome event or state record in the steering/follow-up path, surface it first through diagnostics and `/queue` or status output, and gate any chat acknowledgement behind existing verbose/status behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction: this is a feature and observability request. Source inspection gives high confidence that internal outcomes exist but no normal busy-message accepted-vs-fallback status surface is present.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the target surface. The maintainable direction is one canonical outcome signal reused by diagnostics and status surfaces rather than separate ad hoc acknowledgements.
