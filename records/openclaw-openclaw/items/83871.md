Keep open: current main documents default steering and tracks internal outcomes, but normal busy messages still do not expose a practical accepted-vs-fallback signal through diagnostics, `/queue` status, chat acknowledgements, or a cross-channel operator surface. The related docs/terminology items are closed, and broader status/soft-steer issues are adjacent rather than a clear superseding tracker for this specific outcome-visibility gap.

## What This Changes

Request a user/operator-visible status signal distinguishing accepted active-run steering from fallback or deferred follow-up queueing for busy messages.

## Best Possible Solution

Add one canonical busy-message outcome record or diagnostic event in the steering/follow-up path, surface it through diagnostics and `/queue` status first, and gate any chat acknowledgement behind existing verbose/status behavior.
