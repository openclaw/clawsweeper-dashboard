## Summary

Keep this open: current main tracks and documents steering versus follow-up behavior, but it still lacks a practical user/operator signal that says whether a normal busy-time message was accepted into the active run or fell back to a later turn. The adjacent docs issues are closed, and the broader busy-status issue at https://github.com/openclaw/openclaw/issues/25222 overlaps but does not fully own this steering-outcome diagnostic gap.

## What This Changes

Request to add a visible status or diagnostic signal that distinguishes accepted active-run steering, steering fallback to follow-up, explicit follow-up/collect queueing, and interrupt behavior for busy messages.

## Best Possible Solution

Add one canonical busy-message outcome event or state record in the steering/follow-up path, surface it through diagnostics and /queue status first, and gate any chat acknowledgement behind existing verbose/status controls.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection shows the internal accepted/rejected steering state exists, but current user/operator surfaces do not expose that normal busy-message outcome.

## Solution Assessment

Unclear until maintainers choose the target surface. The maintainable direction is a single canonical outcome signal reused by diagnostics/status surfaces rather than separate per-channel acknowledgements.
