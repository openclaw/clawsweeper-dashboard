## Summary

Keep open: current main tracks steering outcomes internally and documents steering/fallback behavior, but it still lacks a normal user/operator status surface that distinguishes accepted steering from fallback follow-up queueing.

## What This Changes

The issue asks OpenClaw to expose whether a busy inbound message was steered into the active run, deferred as a follow-up, explicitly queued, or interrupted.

## Best Possible Solution

Add one canonical busy-message outcome event or state record in the steering/follow-up path, surface it through diagnostics and `/queue` status first, and gate any chat acknowledgement behind existing verbose/status behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/UX diagnostics request. Source inspection gives high confidence that current main has internal outcomes but no normal busy-message accepted-vs-fallback status surface.

## Solution Assessment

Unclear until maintainers choose the target surface. The maintainable direction is one canonical outcome signal reused by diagnostics/status surfaces rather than separate ad hoc acknowledgements.
