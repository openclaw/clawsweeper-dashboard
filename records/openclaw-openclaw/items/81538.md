## Summary

Keep open. The report matches a source-level compatibility gap: isolated cron forwards the `toolsAllow` list, but managed web-search provider selection can miss the legacy `tools.web.search.apiKey` Brave key when the cron runtime lacks `BRAVE_API_KEY`.

## What This Changes

An isolated cron job with `toolsAllow: ["web_search"]` should be able to use Brave-backed `web_search` from Docker environment or persistent config.

## Best Possible Solution

Keep this open and fix managed `web_search` credential/provider resolution so isolated cron honors configured Brave credentials without relying on cron environment inheritance, with regression coverage for the legacy config path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main run was executed, but source inspection gives a clear reproduction path: Brave configured only via legacy `tools.web.search.apiKey` plus no visible `BRAVE_API_KEY` can be skipped by provider auto-detection.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow fix is in web-search credential/provider resolution and cron regression coverage, not a cron-specific environment workaround or a new config option.
