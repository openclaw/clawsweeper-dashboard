## Summary

Keep open: current main still has a source-level gap between the reported credential surfaces and web-search provider selection in isolated cron runs. The cron allowlist path appears intact, but runtime provider selection ignores the legacy `tools.web.search.apiKey` value that the Brave executor can still consume after Brave has already been selected.

## What This Changes

The issue reports that isolated cron runs cannot use `web_search` even with `toolsAllow`, Docker `BRAVE_API_KEY`, and persistent web-search config set.

## Best Possible Solution

Align runtime web-search provider selection with the documented credential contract, preserving the existing cron allowlist path and adding focused regression coverage for isolated cron plus Brave credential fallback.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show a concrete path: isolated cron forwards `toolsAllow`, but web-search provider selection ignores legacy `tools.web.search.apiKey` before Brave can consume it.

## Solution Assessment

Unclear until a regression is added. The narrowest maintainable fix is in credential/provider selection and tests, not a cron scheduler rewrite or a new config surface.
