## Label Justifications

- `P1`: Telegram media attachments can be skipped in real proxy-only/no-DNS environments while text delivery continues, breaking a live channel workflow.
- `impact:message-loss`: The issue is about inbound Telegram images/documents failing to download and being omitted from the agent message flow.

## Summary

Keep open. Current main and v2026.5.22 still leave Telegram Bot API file downloads on the strict media fetch path for env-proxy transports, so proxy-only or no-DNS environments can still skip inbound media while normal Telegram API calls use the env proxy.

## What This Changes

Telegram inbound media downloads fail in DNS/proxy-only environments because Bot API file fetches can still resolve api.telegram.org locally.

## Best Possible Solution

Adopt a maintainer-approved Telegram media fetch policy that safely uses trusted env-proxy handling for Bot API file downloads when HTTP(S) proxy env applies, or explicitly document that this path requires working local DNS.

## Reproduction Assessment

Yes, source-level. With HTTP(S) env proxy and unusable local DNS, normal Telegram API calls can use EnvHttpProxyAgent while media downloads pass env-proxy attempts into strict guarded fetch and can still resolve api.telegram.org locally; the issue logs show EAI_AGAIN media failures around successful sendMessage calls.

## Solution Assessment

Unclear until maintainer policy review. Hardcoded public DNS or alternate-DNS retries are not the best fix; the maintainable path is a narrow trusted env-proxy media policy with SSRF and proxy-bypass safeguards.
