## Summary

Keep open. Current main and the latest release still leave Telegram Bot API media downloads in strict guarded-fetch mode for env-proxy/no-DNS setups, so inbound images/files can fail while normal Telegram API calls still route through the proxy.

## What This Changes

The issue requests reliable Telegram bot media/image fetching when DNS or proxy-only environments cause Bot API file downloads to fail with MediaFetchError.

## Best Possible Solution

Have maintainers approve a Telegram media guarded-fetch policy that preserves hostname/SSRF checks while allowing trusted env-proxy/no-DNS file downloads, then add focused tests, docs, and redacted live proxy proof.

## Reproduction Assessment

Yes, at source level: configure Telegram with HTTP(S) env proxy and no usable local DNS, and the media path can still resolve api.telegram.org locally through strict guarded fetch. The May 3 comment adds sanitized runtime logs showing media failures while normal Telegram sends succeed through the proxy.

## Solution Assessment

Unclear. The likely code path is narrow, but the best fix requires maintainer product/security direction on whether Telegram media may use trusted env-proxy DNS or must keep strict local DNS pinning.
