## Summary

Keep open. Current main fixed the explicit-proxy media path, but env-proxy/no-DNS Telegram media downloads still fall through the strict guarded-fetch path that can resolve `api.telegram.org` locally before the proxy handles the target.

## What This Changes

The issue asks for Telegram bot media downloads to work when local DNS for `api.telegram.org` fails, especially in VPN, env-proxy, or no-DNS environments.

## Best Possible Solution

Adopt a maintainer-approved Telegram media policy that either uses trusted env-proxy guarded fetch for Bot API file downloads with SSRF/NO_PROXY regression coverage, or explicitly documents that env-proxy media downloads require working local DNS.

## Reproduction Assessment

Yes, as a source-level reproduction: configure Telegram with HTTP(S) env proxy and no usable local DNS, then media downloads still enter strict guarded fetch and can resolve `api.telegram.org` locally before the proxy handles the target. The issue also includes sanitized current-release `EAI_AGAIN` logs showing media failures while normal sends succeed through `EnvHttpProxyAgent`.

## Solution Assessment

Unclear. Hardcoding public DNS is not the right fix; the safer direction is a narrow trusted env-proxy media path or explicit documentation, but that is a maintainer SSRF/proxy-policy decision.
