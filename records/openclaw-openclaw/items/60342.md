- `impact:message-loss`: The issue is about Telegram images/documents failing to download and being skipped from the inbound message flow.

## Summary

Keep open. Current main still leaves Telegram env-proxy media downloads outside trusted proxy DNS mode, so no-DNS proxy environments can still fail file/image fetches while normal Telegram API calls use the env proxy successfully.

## What This Changes

The issue asks OpenClaw to make Telegram bot media/image downloads reliable when the gateway can reach Telegram through a proxy but local DNS for api.telegram.org is unavailable.

## Best Possible Solution

Adopt a maintainer-approved Telegram media policy that safely uses trusted env-proxy handling for Bot API file downloads with SSRF, NO_PROXY, and fallback coverage, or document that env-proxy media downloads require working local DNS.

## Reproduction Assessment

Yes, source-level: configure Telegram with HTTP(S) env proxy and no usable local DNS, then media downloads still take strict guarded fetch and can resolve api.telegram.org locally. The issue discussion also includes sanitized current-release logs showing media failures while normal sends succeed through EnvHttpProxyAgent.

## Solution Assessment

Unclear. The narrow code fix is likely in Telegram media proxy policy, but maintainers need to choose the SSRF/env-proxy contract; hardcoded public DNS is not the right solution.
