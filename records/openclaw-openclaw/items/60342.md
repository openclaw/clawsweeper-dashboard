## Summary

Keep open. Current main still routes Telegram media downloads through the Telegram transport retry predicate, but that predicate omits `ENOTFOUND` and `EAI_AGAIN`, so the reported DNS-shaped media failure can still fail before the dispatcher fallback attempts. Prior proxy and IPv4 media fixes are relevant progress, but they do not cover this exact DNS lookup path.

## What This Changes

The issue reports Telegram inbound image/media downloads failing with `MediaFetchError` caused by `getaddrinfo ENOTFOUND api.telegram.org` on macOS polling mode, and asks for reliable media fetches or DNS/proxy guidance.

## Best Possible Solution

Maintainers should choose one explicit policy for Telegram media DNS failures: either treat pre-connect DNS lookup failures as Telegram media fallback candidates with focused regression tests and preserved SSRF hostname guard behavior, or document that inbound media downloads require working host DNS or an operator-configured Telegram proxy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a `TypeError("fetch failed")` with nested `code: "ENOTFOUND"` will not satisfy `shouldRetryTelegramTransportFallback`, so `fetchRemoteMedia` will not advance to the fallback dispatcher attempts; I did not live-test the reporter's VPN DNS environment.

## Solution Assessment

Unclear. Hardcoding public DNS servers is not the best default; the maintainable choices are a narrow Telegram media fallback expansion for pre-connect DNS errors or explicit documentation of the host DNS/proxy requirement.
