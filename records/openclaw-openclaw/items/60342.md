## Summary

Keep open. Current main still has a source-reproducible Telegram media DNS/proxy gap: `ENOTFOUND`/`EAI_AGAIN` do not satisfy the media fallback predicate, and the fresh 2026.5.2 comment shows media downloads failing while normal Telegram sends still work through the proxy path.

## What This Changes

Request that Telegram Bot API media downloads recover from DNS/proxy lookup failures instead of failing as `MediaFetchError`.

## Best Possible Solution

Use the existing guarded-fetch proxy/fallback seams to make Telegram media downloads honor the documented proxy path for DNS-safe operator setups, with regression coverage; if maintainers decline that policy, document host DNS/proxy requirements explicitly.

## Reproduction Assessment

Yes at source level. A `TypeError("fetch failed")` with nested `code: "ENOTFOUND"` or `"EAI_AGAIN"` will not satisfy `shouldRetryTelegramTransportFallback`, so `fetchRemoteMedia` will not advance to later dispatcher attempts; the latest issue comment adds current-release log evidence for the same failure shape.

## Solution Assessment

Unclear. Hardcoding public DNS servers is not the best fix; the maintainable choices are a narrow Telegram media fallback/proxy-DNS change using existing SSRF guard seams, or explicit documentation that media downloads require working host DNS or an operator-configured trusted proxy path.
