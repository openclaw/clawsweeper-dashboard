## Summary

Keep open. Current main still does not classify Telegram Bot API DNS lookup failures as media transport fallback candidates, so the reported `MediaFetchError` path can still stop before the existing dispatcher fallback attempts are tried.

## What This Changes

The issue asks for Telegram inbound media/image downloads to recover when `api.telegram.org` DNS resolution fails under VPN or host DNS configuration.

## Best Possible Solution

Maintainers should choose a clear policy: either narrowly treat Telegram pre-connect DNS failures as fallback candidates with regression tests and preserved SSRF guards, or document that inbound media downloads require working host DNS or an operator-configured Telegram proxy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a `TypeError("fetch failed")` with nested `code: "ENOTFOUND"` will not satisfy `shouldRetryTelegramTransportFallback`, so `fetchRemoteMedia` will not advance to later dispatcher attempts.

## Solution Assessment

Unclear. Hardcoding public DNS is not the maintainable fix; the safer options are a narrow Telegram fallback expansion for `ENOTFOUND`/`EAI_AGAIN` if maintainers accept that policy, or explicit documentation of the host DNS/proxy requirement.
