## Summary

Keep open. Current main still leaves the Telegram Bot API file download path on strict guarded media fetch for env-proxy transports, so proxy-only or no-DNS environments can still skip inbound images/files even while normal Telegram API calls route through the env proxy.

## What This Changes

The issue reports Telegram bot image/media downloads failing with DNS errors in VPN or proxy-only environments while normal Telegram bot traffic can still work.

## Best Possible Solution

Adopt a maintainer-approved Telegram media policy that safely reuses trusted env-proxy handling for Bot API file downloads, or explicitly document that this media path requires working local DNS.

## Reproduction Assessment

Yes, source-level: configure Telegram with HTTP(S) env proxy and no usable local DNS, then media downloads can still use strict guarded fetch and resolve api.telegram.org locally. The May 3 issue comment adds sanitized runtime logs showing media failures while normal sendMessage calls succeed through the proxy.

## Solution Assessment

Unclear. The likely code path is narrow, but the best fix depends on a maintainer decision about SSRF checks and whether Telegram Bot API file downloads may trust env-proxy DNS the same way normal Telegram API calls do.
