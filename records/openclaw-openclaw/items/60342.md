## Summary

Keep open. Current main still has a Telegram media download gap where env-proxy or fallback transport can be configured, but media SSRF wrapping can still require local DNS and the Telegram fallback predicate still rejects `ENOTFOUND`/`EAI_AGAIN` before later dispatcher attempts are tried.

## What This Changes

The issue asks for Telegram inbound media/image downloads to keep working when local DNS cannot resolve `api.telegram.org` but Telegram traffic is otherwise reachable through proxy or fallback transport.

## Best Possible Solution

Telegram media downloads should preserve the existing SSRF allowlist while using the same trusted env/explicit proxy and fallback transport semantics as the rest of Telegram Bot API traffic.

## Reproduction Assessment

Yes, at source level: an undici `TypeError("fetch failed")` with nested `code: "EAI_AGAIN"` or `"ENOTFOUND"` does not satisfy `shouldRetryTelegramTransportFallback`, so `fetchRemoteMedia` stops before later dispatcher attempts. The fresh 2026.5.2 log comment matches that path for media downloads while normal Telegram sends still work.

## Solution Assessment

No, hardcoding public DNS servers is not the best fix. The maintainable repair is to reuse the existing trusted env-proxy/explicit-proxy guard modes and narrowly classify Telegram pre-connect DNS failures as fallback candidates without weakening SSRF checks.
