## Summary

Keep this open: current main fixed several adjacent Telegram media-download proxy bugs, but the May no-DNS/env-proxy datapoint still maps to a remaining source-level gap in the guarded media fetch path and needs maintainer/security direction before an automated repair.

## What This Changes

Telegram inbound media downloads should succeed when the gateway can reach Telegram through a proxy/VPN path but local DNS returns ENOTFOUND or EAI_AGAIN for api.telegram.org.

## Best Possible Solution

Define a maintainer-approved trusted proxy DNS policy for Telegram media downloads that preserves SSRF hostname checks, make the media path match documented Bot API proxy behavior, and add focused no-DNS/env-proxy regression coverage.

## Reproduction Assessment

Yes at source level, but not by a live Telegram run in this review. The reported EAI_AGAIN/ENOTFOUND path matches current main: Telegram media env-proxy dispatcher attempts still enter strict guarded fetch and require local DNS unless the proxy is an explicit trusted proxy.

## Solution Assessment

Unclear until maintainers choose the proxy trust policy. Hardcoding public DNS or blindly bypassing DNS pinning is not the best fix; the safer direction is a bounded Telegram media trusted-proxy path with tests and docs.
