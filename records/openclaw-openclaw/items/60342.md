## Summary

Keep open. Current main fixed the explicit-proxy variant, but the issue discussion includes a later env-proxy/no-DNS data point and the current Telegram media path still runs strict guarded fetch for env-proxy dispatcher policies, which can pre-resolve api.telegram.org locally and fail with ENOTFOUND/EAI_AGAIN before the proxy resolves the target.

## What This Changes

The issue asks OpenClaw to make Telegram bot media downloads resilient to DNS failures in polling mode when media fetches hit ENOTFOUND/EAI_AGAIN for api.telegram.org.

## Best Possible Solution

Adopt a maintainer-approved Telegram media policy that either uses trusted env-proxy guarded fetch for proxy-routed Bot API file downloads with SSRF/NO_PROXY regression coverage, or explicitly documents that env-proxy media downloads require working local DNS.

## Reproduction Assessment

Yes. Source inspection shows env-proxy Telegram media downloads still run through strict guarded fetch, and the issue has sanitized current-release logs showing EAI_AGAIN media failures while normal Telegram sends succeed through EnvHttpProxyAgent.

## Solution Assessment

Unclear. Hardcoding public DNS is not the right fix; the safer direction is a narrow trusted env-proxy media path with SSRF and NO_PROXY tests, but that needs an explicit maintainer network-policy decision.
