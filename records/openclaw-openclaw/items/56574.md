## Summary

Keep open: the plugin-registry/tool-context intent still has plausible value, but this PR should not merge as-is because PR head broadens live-test profile loading, stages live config from a stale env snapshot, is stale against current plugin loader/runtime surfaces, and lacks external real behavior proof.

## What This Changes

The PR adds same-plugin tool registration dedupe, threads per-plugin config into plugin tool factory context, relaxes gateway-bindable registry reuse, and updates live-test home/profile staging docs and tests.

## Best Possible Solution

Refresh or split the patch so only still-needed registry/SDK contract changes land on current plugin surfaces, preserve current main's live-test profile gating, and require redacted real-plugin or live-run proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks direct pluginConfig in the tool factory context, while PR head visibly loads .profile before the live gate and stages live state from a pre-profile snapshot; I did not run tests because this was a read-only review.

## Solution Assessment

No. The registry/tool-context idea may be useful after refresh, but this branch is not the narrowest maintainable solution because it mixes stale loader changes with live-test profile handling regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate profile loading to live test runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` unconditionally, so normal non-live runs can import missing vars from the developer's real `.profile`. That can leak provider keys/config paths into isolated tests and make the default test lane depend on host state. Keep profile loading behind live/real-home mode or an explicit option.
  - confidence: 0.92
- **[P2] Capture live env after loading the profile:** `test/test-env.ts:336-348`
  - body: `liveEnvSnapshot` is captured before `.profile` is loaded, but `stageLiveTestState` uses that snapshot to resolve `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are only exported from `.profile`, live staging copies the wrong config/auth state.
  - confidence: 0.9
