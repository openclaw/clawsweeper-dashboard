Keep open: current main still lacks the exact tool-registration dedupe and plugin tool `pluginConfig` threading, but this PR head is not safe to merge as-is because it also imports real profile env into ordinary tests and snapshots live staging env too early. ## What This Changes

The PR updates plugin registry/loading/tool resolution to dedupe repeated per-plugin tool registrations, pass per-plugin config to tool factories, reuse gateway-bindable registries for default requests, and revise live-test HOME/profile staging docs/tests.

## Best Possible Solution

Land a refreshed or split change on current main that keeps exact same-plugin tool dedupe and pluginConfig threading with focused regression tests and a changelog entry, while preserving current gateway compatibility and fixing live-test staging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path. Current main still appends duplicate plugin tool registrations and calls tool factories without `pluginConfig`, while PR head shows deterministic test-env regressions at lines 336-348.

## Solution Assessment

No, not as currently proposed. The registry/tool-context fix direction is maintainable, but the PR should be rebased or split so it does not carry credential-sensitive live-test staging regressions and does not duplicate current-main gateway reuse work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` for every test env install, before checking whether the run is live or explicitly using the real HOME. That imports missing variables from the developer's real `.profile`, including provider keys or config paths, into normal isolated tests and makes the default test lane host-dependent. Gate this behind live/real-home mode or an explicit test option.
  - confidence: 0.92
- **[P2] Snapshot env after loading the profile:** `test/test-env.ts:336-348`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState` later uses that snapshot for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are exported only from `.profile`, live staging copies the default state instead of the operator-selected config/auth state. Capture the snapshot after profile loading or pass the post-load env values to staging.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/plugins/registry.ts:313-340`
  - body: This PR changes user-visible plugin tool behavior by deduping registrations and exposing per-plugin config to tool factories, but it does not add a `CHANGELOG.md` entry. Repo policy requires user-facing fix changes to be recorded, so add one concise entry for the plugin tool fix.
  - confidence: 0.86

## Security Review

Summary: The PR broadens local secret exposure in ordinary tests by sourcing the real profile outside live-test mode.

Concerns:

- **[low] Profile env leaks into non-live tests:** `test/test-env.ts:338`
  - body: Unconditionally sourcing the real HOME profile in `installTestEnv` can load provider keys and config paths into default isolated tests, broadening local secret exposure and making tests use operator configuration unexpectedly.
