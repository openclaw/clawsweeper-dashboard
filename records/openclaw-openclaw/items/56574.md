## Summary

Keep open. Current main still lacks the core plugin fixes this PR is trying to provide, so the work is not obsolete; however, the PR head is stale and the added live-test HOME/profile staging introduces credential/config handling regressions that should block merge as-is.

## What This Changes

This PR changes plugin registry/loading/tool resolution to dedupe same-plugin tool registrations, pass per-plugin config into plugin tool factories, reuse compatible gateway-bindable registries for default runtime requests, and update live-test HOME/profile staging docs and tests.

## Best Possible Solution

Rebase or replace the PR on current main, keep the registry-level exact dedupe and pluginConfig tool-context threading with focused regression tests, align the registry compatibility change with the current runtimeSubagentMode/workspace tracking, and split or fix the live-test HOME/profile staging so profile env is loaded only for live/real-home runs and staging uses the post-profile environment.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main still appends duplicate plugin tool registrations and omits pluginConfig from tool factory context, while PR head unconditionally sources profile env and snapshots live env before profile loading.

## Solution Assessment

No as currently proposed. The plugin registry/tool-context direction is the right maintainable boundary, but the stale mixed test-env changes introduce credential/config handling regressions that should be gated, reordered, or split before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` for every test run before checking whether this is a live or real-home run. That imports missing variables from the developer's real `.profile`, including provider keys or `OPENCLAW_CONFIG_PATH`, so default isolated tests can become machine-dependent and touch real credentials/config. Gate this behind live/real-home mode or an explicit test option.
  - confidence: 0.92
- **[P2] Snapshot env after loading the profile:** `test/test-env.ts:336-338`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState` later uses that snapshot for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are only exported from `.profile`, live runs stage the default `.openclaw` state instead of the operator-selected config/auth state. Capture the snapshot after profile loading or pass the post-load env values to staging.
  - confidence: 0.9

## Security Review
