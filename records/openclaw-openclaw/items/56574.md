## Summary

Keep open. Current main still does not implement the central plugin registry/tool-context fixes, and this PR has useful work, but the head patch is not correct as-is because its live test environment changes can leak host profile state into ordinary tests and stage the wrong live config/auth state.

## What This Changes

The PR changes plugin registry/loading/tool resolution to dedupe same-plugin tool registrations, pass per-plugin config into plugin tool factories, reuse compatible gateway-bindable registries for default requests, and update live-test HOME/profile staging docs and tests.

## Best Possible Solution

Rebase or replace the PR on current main, keep the registry-level exact dedupe and pluginConfig tool-context threading with focused regression tests, preserve the safe gateway-method compatibility dimensions, and either fix or split the live-test HOME/profile staging changes before merge.

## Reproduction Assessment

Yes. The remaining current-main gaps and the PR-head blockers are reproducible by source inspection: current main still appends duplicate tool registrations and omits pluginConfig from factory context, while PR head unconditionally loads profile env and snapshots env before profile loading.

## Solution Assessment

No as currently proposed. The plugin registry/tool-context direction is the right boundary, and the earlier partial-overlap/gateway-method comments are addressed in PR head, but the mixed live-test staging changes introduce credential/config handling regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` for every test run before checking whether this is a live run. That imports missing variables from the developer's real `~/.profile`, including provider keys or `OPENCLAW_CONFIG_PATH`, so default isolated tests can become machine-dependent and touch real credentials/config. Gate this behind live/real-home mode or an explicit test option.
  - confidence: 0.92
- **[P2] Snapshot env after loading the profile:** `test/test-env.ts:336-338`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState` later uses that snapshot for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are only exported from `~/.profile`, live runs stage the default `~/.openclaw` instead of the operator-selected config/auth state. Capture the snapshot after profile loading or pass the post-load env values to staging.
  - confidence: 0.9

## Security Review
