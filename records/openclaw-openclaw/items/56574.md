Keep this PR open: the plugin fixes are still useful and not fully present on current main, but the branch has a concrete test-environment isolation regression that should block merge until repaired and rebased.

## What This Changes

The PR deduplicates repeated plugin tool registrations, passes per-plugin config into tool factories, adds compatible gateway-bindable registry reuse tests, and changes live-test HOME/profile staging docs and helpers.

## Best Possible Solution

Carry the pluginConfig and exact tool-registration dedupe fixes forward onto current main, while preserving the newer runtime cache design and live-test isolation safeguards already present there.

## Reproduction Assessment

Yes. Source inspection of PR head shows normal non-live installTestEnv calls still run loadProfileEnv(realHome), and the live staging env snapshot is taken before profile values are loaded.

## Solution Assessment

No. The plugin-side direction is maintainable, but this PR branch is not the best merge form until the test-env regression is fixed and the stale runtime/test-env changes are reconciled with current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate profile loading to live test runs:** `test/test-env.ts:338`
  - body: `installTestEnv()` now calls `loadProfileEnv(realHome)` even when `live` is false, so ordinary `pnpm test` runs import missing variables from the developer's `~/.profile` before the isolated HOME is created. That makes non-live tests host-dependent and can leak local provider/config env into code paths that should remain isolated; keep profile loading behind live, explicit real-home use, or an explicit test option.
  - confidence: 0.92
- **[P2] Snapshot live env after profile loading:** `test/test-env.ts:336-338`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState()` uses that snapshot to resolve `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are supplied only by `~/.profile`, the staged live HOME copies from the defaults instead of the active profile-backed config/auth state.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR broadens ~/.profile sourcing into normal test runs, which weakens test isolation and can expose local credentials/config to non-live tests.

  - body: PR head sources the user's profile before creating the isolated test HOME for every installTestEnv call, so arbitrary missing env vars from local profile files can affect ordinary test execution outside live-test mode.
