## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` unconditionally, so normal isolated test runs can import provider keys or config paths from the developer's real `.profile`. Gate this behind live/real-home mode or an explicit test option so default tests remain host-independent.
  - confidence: 0.92
- **[P2] Snapshot env after sourcing profile:** `test/test-env.ts:336-348`
  - body: `liveEnvSnapshot` is captured before `.profile` is loaded, but `stageLiveTestState` uses that snapshot for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are profile-only exports, live staging copies the default state instead of the selected config/auth state.
  - confidence: 0.9
- **[P3] Add the plugin fix changelog entry:** `src/plugins/registry.ts:313-340`
  - body: This PR changes user-visible plugin tool behavior by deduping registrations and exposing per-plugin config to tool factories, but it does not add a `CHANGELOG.md` entry. Repo policy requires user-facing fixes to be recorded.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces low-severity local secret/config handling regressions in the test environment setup.

Concerns:

- **[low] Profile env leaks into non-live tests:** `test/test-env.ts:338`
  - body: Unconditionally sourcing the real HOME profile in `installTestEnv` can load provider keys and config paths into default isolated tests, broadening local secret exposure and making tests depend on operator configuration.
  - confidence: 0.88
