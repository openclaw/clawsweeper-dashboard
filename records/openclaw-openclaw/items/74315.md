## Summary

Keep open. The PR is aimed at a valid model-selection bug, but the current diff only adds exact-first lookup in two provider-map helpers; current runtime/default/fallback paths still route selected refs through global provider normalization, so the linked `kimi-coding/k2p5` configuration can still collapse to `kimi/k2p5` before execution or status display.

## What This Changes

The PR adds case-insensitive exact provider-key lookup before alias-normalized fallback in `findNormalizedProviderValue` and `findNormalizedProviderKey`, with unit tests for those helper semantics.

## Best Possible Solution

Revise this PR so explicitly configured provider IDs such as `kimi-coding` survive default resolution, catalog/allowlist lookup, fallback candidate construction, runtime execution, and status display, while retaining `kimi` and `kimi-code` compatibility where provider-family aliases are intended. Add an end-to-end regression test for the linked `models.providers.kimi-coding` plus `agents.defaults.model.primary: kimi-coding/k2p5` configuration.

## Reproduction Assessment

Yes. The linked report provides a concrete config, and source inspection shows the exact provider can be recognized during config parsing but is later collapsed by `normalizeModelRef` in the agent command and fallback paths.

## Solution Assessment

No. Exact-first helper lookup is a useful part of the solution, but it is not the best complete fix because the runtime and catalog paths that determine the selected model still normalize `kimi-coding` back to `kimi`.
