## Summary

Keep open: current main and v2026.6.10 still forward only nested `args` or `input`, so flattened compact Tool Search `tool_call` arguments are still dropped. The issue should remain the canonical report while the open closing PRs are reviewed and one corrected, proof-backed fix is selected.

## What This Changes

The issue requests that compact Tool Search `tool_call` preserve or clearly reject flattened target-tool arguments emitted under `localModelLean`.

## Best Possible Solution

Land one narrow Tool Search normalization that preserves explicit `args` and `input`, safely handles flattened target fields, includes focused regression tests, and has redacted localModelLean runtime proof before closing this issue.

## Reproduction Assessment

Yes. Source-level reproduction is clear because current `readCallArgs` forwards only `params.args ?? params.input ?? {}`, so a payload like `{ id, command }` dispatches `{}`; I did not independently rerun the live Telegram/vLLM setup.

## Solution Assessment

Yes, the right solution layer is the compact Tool Search wrapper parser, not Telegram or a provider-specific workaround. The best path is to review and land one corrected linked PR with real localModelLean proof.
