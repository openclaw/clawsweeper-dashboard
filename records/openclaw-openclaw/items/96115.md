## Summary

Keep open: current `main` and the latest release still drop flattened compact Tool Search `tool_call` arguments, and the linked closing PR is open but unmerged.

## What This Changes

The issue reports that local-model lean compact Tool Search `tool_call` drops target-tool arguments when local models flatten them beside `id`, causing repeated missing-parameter failures.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96127 or an equivalent narrow Tool Search fix that preserves explicit `args`/`input` precedence and covers flattened wrapper arguments.

## Reproduction Assessment

Yes, at source level: current `readCallArgs` returns `params.args ?? params.input ?? {}`, so a payload like `{ id, command }` dispatches `{}`. I did not run the reported Telegram/vLLM session in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the Tool Search wrapper argument normalization, not Telegram or a provider-specific path. The linked PR targets that layer while preserving the existing nested `args` contract.
