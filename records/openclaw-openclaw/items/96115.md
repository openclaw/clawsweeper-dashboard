## Summary

Keep open: current main and v2026.6.9 still have the reported Tool Search wrapper behavior, where structured `tool_call` forwards only `args` or `input`, so flattened target parameters are silently dispatched as `{}` instead of producing a wrapper-shape error.

## What This Changes

The issue reports that localModelLean's compact structured Tool Search `tool_call` drops target-tool parameters when a local model flattens them beside the wrapper `id`.

## Best Possible Solution

Keep this issue open and land a narrow Tool Search wrapper fix that detects flattened target parameters before dispatch, returns a clear `tool_call` wrapper-shape error, and preserves existing nested `args`/`input` behavior.

## Reproduction Assessment

Yes at source level: current main's `readCallArgs()` forwards only `args` or `input`, so flattened target fields beside `id` are dropped before target execution. I did not run a live Telegram/vLLM session in this read-only review.

## Solution Assessment

Yes for the narrow repair path: reject flattened wrapper-level target parameters with a clear `tool_call` shape error before dispatch, and cover it with focused tests. Automatic normalization may be useful later, but it is a broader wrapper-contract decision than needed to stop the false missing-parameter failures.
