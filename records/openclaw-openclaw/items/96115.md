## Summary

Keep open: current `main` and the latest release still drop flattened compact Tool Search `tool_call` target arguments, while two open closing PRs are competing fix candidates rather than merged implementations.

## What This Changes

The issue reports that local-model lean compact Tool Search accepts flattened `tool_call` payloads but forwards `{}` to the selected target tool unless arguments are nested under `args` or `input`.

## Best Possible Solution

Land one narrow Tool Search normalization that preserves explicit `args` and `input`, safely handles flattened target args, includes focused regression coverage, and has redacted real localModelLean proof.

## Reproduction Assessment

Yes, at source level: a compact `tool_call` payload like `{ id, command }` reaches `readCallArgs`, which forwards `{}` because only `args` or `input` are read. I did not run the reported Telegram/vLLM session in this read-only review.

## Solution Assessment

Yes, the right fix layer is the Tool Search wrapper argument parser rather than Telegram or a provider-specific path. The issue should stay open until one corrected, proof-backed linked PR is merged.
