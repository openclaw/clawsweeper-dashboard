## What This Changes

The issue requests that compact Tool Search `tool_call` stop dropping target-tool arguments when local models emit them as flattened wrapper fields.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96127, or replace it with an equivalent narrow Tool Search normalization fix that preserves explicit `args`/`input` precedence and adds regression coverage.

## Reproduction Assessment

Yes, at source level: current `readCallArgs` forwards only `params.args ?? params.input ?? {}`, so flattened wrapper fields become `{}`. The issue discussion also includes a later live Telegram/local-model lean reproduction, but I did not run that live setup in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is in Tool Search wrapper argument normalization, not in Telegram or a provider-specific path, while preserving the existing nested `args` contract.
