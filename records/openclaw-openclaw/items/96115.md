## Summary

Keep open: current main and the latest release still drop flattened compact Tool Search `tool_call` arguments, and two open closing PRs are competing fix candidates rather than merged fixes.

## What This Changes

The issue reports that local-model lean compact `tool_call` accepts flattened target-tool fields but forwards only nested `args` or `input`, causing target tools to receive `{}` and fail with missing parameters.

## Best Possible Solution

Land one narrow Tool Search parser fix that preserves canonical `args`/legacy `input`, filters wrapper metadata, and includes focused regression coverage plus real local-model lean proof.

## Reproduction Assessment

Yes, at source level: current `readCallArgs` forwards only nested `args` or `input`, so a payload like `{ id, command }` dispatches `{}`. I did not run the reported Telegram/vLLM session in this read-only review.

## Solution Assessment

Yes, the right layer is Tool Search wrapper argument normalization rather than Telegram or provider-specific code. The remaining maintainer work is to choose or correct one open fix PR and require proof before merge.
