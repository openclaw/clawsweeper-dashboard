## Summary

Keep open. Current main still only partially handles the reported Google Chat Workspace Add-ons incompatibility: message add-on wrappers are accepted, but lifecycle wrapper variants still reject as invalid payload, and the public setup path still omits the required numeric appPrincipal guidance.

## What This Changes

The issue requests Google Chat Workspace Add-ons compatibility for app-url JWTs, numeric appPrincipal setup, native add-on wrapper parsing, and clearer setup guidance.

## Best Possible Solution

Keep this issue open and consolidate the remaining Google Chat Add-ons work: document app-url plus numeric appPrincipal discovery, add setup/status guidance for appPrincipal, extend add-on normalization for supported payload variants with tests, and preserve strict principal binding unless a maintainer explicitly changes that trust model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: submit an add-on body with commonEventObject.hostApp CHAT and only chat.addedToSpacePayload or chat.removedFromSpacePayload, and current main skips normalization because chat.messagePayload is absent before failing root type/space validation.

## Solution Assessment

No, current main is not a complete solution. The narrow maintainable path is plugin-local parser coverage plus docs/setup guidance; auto-accepting unbound add-on service accounts is a separate auth-policy decision.
