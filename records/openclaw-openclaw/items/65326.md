## Summary

Keep open. Current main only partially addresses the Google Chat Workspace Add-ons incompatibility: message add-on payloads and auth diagnostics exist, but lifecycle add-on wrappers still fail and public setup guidance still does not explain numeric appPrincipal discovery.

## What This Changes

Fix Google Chat Workspace Add-ons incompatibilities by documenting app-url/appPrincipal setup, unwrapping add-on event bodies, and clarifying appPrincipal discovery/auth handling.

## Best Possible Solution

Consolidate the open Google Chat PRs into plugin-local parser and setup/docs fixes while preserving explicit appPrincipal binding unless maintainers approve an auth-policy change.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: submit an add-on envelope with `commonEventObject.hostApp: "CHAT"`, `authorizationEventObject.systemIdToken`, and only `chat.addedToSpacePayload` or `chat.removedFromSpacePayload`; current main does not unwrap it and reaches `400 invalid payload`. I did not run a live gateway because this review is read-only.

## Solution Assessment

No, current main is not a complete solution. The narrow maintainable path is to normalize supported add-on payload variants in the Google Chat plugin, document/setup numeric appPrincipal discovery, and leave any relaxed principal trust model to maintainer review.
