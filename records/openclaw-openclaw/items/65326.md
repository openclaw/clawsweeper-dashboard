## Summary

Keep open. Current main partially addresses the Google Chat Workspace Add-ons incompatibility: app-url add-on auth is strict and now warns about bad or missing `appPrincipal`, and `messagePayload` add-on wrappers are accepted, but other reported wrapper variants still reject as invalid payload and the public docs/setup flow still omit numeric `appPrincipal` discovery guidance.

## What This Changes

Request to make the bundled Google Chat plugin work with Google Workspace Add-ons by documenting `app-url`/numeric `appPrincipal` setup and parsing add-on-wrapped event bodies without an external proxy.

## Best Possible Solution

Consolidate the Google Chat Add-ons work in the plugin: document `app-url` plus numeric `appPrincipal`/JWT `sub` discovery, add setup/status guidance for that field, extend add-on normalization for supported payload variants with tests, and preserve explicit principal binding unless maintainers intentionally change the auth policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: submit a Workspace Add-ons body with `commonEventObject.hostApp: "CHAT"`, `authorizationEventObject.systemIdToken`, and only `chat.addedToSpacePayload` or `chat.removedFromSpacePayload`; current main skips normalization because `chat.messagePayload` is absent and then returns 400 because root `type`/`space` are missing.

## Solution Assessment

Partially. The docs and parser suggestions are the narrow maintainable direction, but the suggested auto-discovery/auto-acceptance of add-on principals is not clearly the best fix because current code and tests deliberately require explicit principal binding.
