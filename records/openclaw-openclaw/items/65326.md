## Summary

Keep open. Current main still only partially handles the reported Google Chat Workspace Add-ons incompatibility: message Add-ons wrappers are accepted, but other supplied Add-ons payload variants still reject, and the public setup path still does not tell users how to configure the required numeric appPrincipal.

## What This Changes

The issue requests Google Chat Workspace Add-ons compatibility for app-url JWTs, numeric appPrincipal setup, native Add-ons wrapper parsing, and clearer setup guidance.

## Best Possible Solution

Keep this issue open and consolidate the remaining Google Chat Add-ons work with the open parser/auth PRs: document audienceType app-url plus numeric appPrincipal discovery, add setup wizard/status guidance for appPrincipal, broaden Add-ons normalization for supported payload variants with tests, and preserve strict principal binding unless a maintainer explicitly changes the trust model.

## Reproduction Assessment

Yes. A current-main static reproduction is clear: submit an Add-ons body with commonEventObject.hostApp CHAT and chat.addedToSpacePayload or removedFromSpacePayload, and the parser skips normalization because chat.messagePayload is absent, then fails root type/space validation as invalid payload.

## Solution Assessment

No, current main is not the complete solution. The narrow maintainable path is to keep appPrincipal binding strict, improve docs/setup guidance, and extend parser coverage for supported Add-ons payload variants rather than auto-accepting unbound Add-ons service accounts.
