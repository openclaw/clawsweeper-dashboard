## Summary

Keep open. Current main only partially handles the reported Google Chat Workspace Add-ons incompatibility: message wrappers and strict app-url/appPrincipal auth exist, but lifecycle wrapper variants still fail and public setup guidance still omits numeric appPrincipal discovery.

## What This Changes

Request native handling and documentation for Google Workspace Add-ons app-url JWTs, numeric appPrincipal binding, and wrapped Google Chat event bodies.

## Best Possible Solution

Keep this issue open and consolidate the remaining Google Chat Add-ons work: document app-url plus numeric appPrincipal discovery, add setup/status guidance, extend plugin-local add-on normalization for supported payload variants with tests, and preserve explicit principal binding unless maintainers approve a trust-model change.

## Reproduction Assessment

Yes. Submit an Add-ons body with `commonEventObject.hostApp: "CHAT"`, `authorizationEventObject.systemIdToken`, and `chat.addedToSpacePayload` or `chat.removedFromSpacePayload` but no `chat.messagePayload`; current main skips normalization and fails root `type`/`space` validation with `400 invalid payload`.

## Solution Assessment

No, current main is not a complete solution. The narrow maintainable path is plugin-local parser coverage plus docs/setup guidance; auto-discovering or auto-accepting principals is a separate auth-policy decision.
