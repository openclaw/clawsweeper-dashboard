## Summary

Keep open. The PR addresses a real gap on current main and the latest head fixed the earlier stale-test/manifest blockers, but it still needs public docs for the new knobs and maintainer security/product sign-off on retries for custom embedding endpoints.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` to memory-lancedb config/schema/UI hints, wires them into the direct OpenAI-compatible embeddings client, adds parser/manifest tests, and adds a changelog entry.

## Best Possible Solution

Land this as a scoped memory-lancedb change after documenting defaults/ranges/direct-client applicability and explicitly accepting or constraining the custom `baseUrl` retry trust model.

## Reproduction Assessment

Yes. On current main, configs containing `embedding.timeoutMs` or `embedding.maxRetries` are rejected by manifest/runtime validation, and the direct OpenAI-compatible client has no config-backed constructor timeout or retry setting.

## Solution Assessment

Unclear. The implementation is in the right owner plugin and uses supported OpenAI SDK options, but the final merge path should clarify docs/applicability and resolve the custom `baseUrl` retry security posture first.
