## Summary

Keep open: the central memory-lancedb timeout/retry feature is not on current main, but this PR is not merge-ready because it is stale against the lazy OpenAI runtime path, lacks real behavior proof, misses user docs/wording alignment, and needs a maintainer decision on retries for custom endpoints.

## What This Changes

The PR adds memory-lancedb `embedding.timeoutMs` and `embedding.maxRetries` config/schema/UI/test/changelog wiring and passes those values into the direct OpenAI-compatible embedding client.

## Best Possible Solution

Rebase the plugin-local config onto current main's lazy direct-client path, document the options, align fallback wording with behavior, settle the custom-endpoint retry policy, and add redacted real runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main rejects `timeoutMs` and `maxRetries` in both the runtime parser and manifest schema, and the direct client has no config-backed default timeout/retry knobs. I did not run live memory-lancedb traffic in this read-only review.

## Solution Assessment

No, not yet. Plugin-local timeout/retry wiring is the right direction, but this branch needs to preserve current main's lazy client construction, add docs, correct fallback wording, provide real behavior proof, and resolve the custom-endpoint retry trust model before merge.
