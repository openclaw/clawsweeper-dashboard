## What This Changes

## Best Possible Solution

Ship the direct-client knobs after the manifest/config UI and release note clearly say they apply to the direct OpenAI-compatible client, or deliberately extend the provider embedding contract with focused tests.

## Reproduction Assessment

Yes, at source/dependency level: current main lacks configurable OpenAI constructor timeout/retry options, and the PR body includes terminal blackhole proof for the direct client. The provider-scope mismatch is also source-reproducible because provider-auth configs fall through to a provider contract that only accepts `signal`.

## Solution Assessment

No, not quite: the direct-client implementation is the right narrow runtime path, but the public config copy and changelog must match that scope before merge. The larger alternative is an explicit provider-contract change with tests for provider-backed embeddings.
