## Summary

Keep open: current main still lacks these memory-lancedb timeout/retry knobs, and the latest PR head keeps the lazy OpenAI transport, but it still needs real behavior proof plus small docs/wording cleanup and maintainer security acceptance before merge.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` config/schema/UI support, parser tests, runtime wiring, and a changelog entry for memory-lancedb OpenAI-compatible embedding requests.

## Best Possible Solution

Merge plugin-local timeout/retry support after docs, fallback wording, real runtime proof, and custom-endpoint retry policy are aligned; keep the behavior scoped to the direct OpenAI-compatible path unless the shared provider adapter contract grows timeout/retry options.

## Reproduction Assessment

Yes for the source-level config gap: current main rejects `embedding.timeoutMs` and `embedding.maxRetries` and has no config-backed default timeout/retry knobs on the direct OpenAI-compatible memory path. I did not reproduce a live provider rate-limit storm.

## Solution Assessment

No, not yet. Plugin-local config is the right boundary and the latest head preserves the lazy transport, but the PR still needs docs/wording cleanup, real runtime proof, and maintainer acceptance of the custom-endpoint retry model.
