## Summary

Keep open. Current main still lacks the requested memory-lancedb timeout/retry configuration, and the latest PR head is useful, but it still needs public docs, fallback wording cleanup, and maintainer security/product sign-off for retry behavior on custom embedding endpoints before merge.

## What This Changes

This PR adds memory-lancedb `embedding.timeoutMs` and `embedding.maxRetries` config parsing, manifest schema/UI hints, tests, direct OpenAI-compatible client wiring, and a changelog entry.

## Best Possible Solution

Land a scoped memory-lancedb update that keeps the owner-local timeout/retry behavior, documents the new options and trust model, and makes the fallback wording match the actual OpenClaw defaults.

## Reproduction Assessment

Yes. On current main, adding `embedding.timeoutMs` or `embedding.maxRetries` to memory-lancedb config is rejected by the parser/manifest, and the direct OpenAI-compatible client has no config-backed constructor timeout or retry setting.

## Solution Assessment

Unclear. The PR targets the right plugin module and uses supported OpenAI SDK options, but it is not the best final shape until the public docs, fallback wording, and custom-baseUrl retry trust model are resolved.

## Security Review

Status: needs_attention

Summary: Security review needs maintainer attention because the diff exposes retry control on the existing arbitrary custom embedding endpoint path.

Concerns:

- **[medium] Clarify custom baseUrl retry trust model:** `extensions/memory-lancedb/config.ts:193`
  - body: Current main already permits a configured `embedding.baseUrl`, but this PR lets config raise retries up to 5 on that direct request path. Because that can increase repeated requests to arbitrary configured endpoints, maintainers should explicitly accept and document the trusted-local-config model or constrain the retry override when a custom baseUrl is used.
