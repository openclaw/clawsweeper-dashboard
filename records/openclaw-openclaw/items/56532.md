## Summary

Keep open. Current main still does not accept or wire configurable `memory-lancedb` embedding timeout/retry settings, so this PR is not obsolete; the latest diff also appears to have addressed the earlier stale assertion and manifest-schema blockers. The remaining blockers are review/cleanup items: security maintainer judgment for retries on the existing configurable `baseUrl` path, plus required docs/changelog coverage.

## What This Changes

The branch adds `embedding.timeoutMs` and `embedding.maxRetries` to the memory-lancedb config parser, manifest schema, UI hints, and tests, then passes them into the OpenAI-compatible embedding client.

## Best Possible Solution

Refresh and land a scoped memory-lancedb PR that preserves current main's bounded auto-recall request path, adds the config/parser/manifest/runtime wiring, documents the new knobs, includes a changelog entry, and explicitly resolves the `baseUrl` plus retry trust-model question before merge.

## Reproduction Assessment

Yes. On current main, a config containing `embedding.timeoutMs` or `embedding.maxRetries` is rejected by the plugin manifest/runtime parser, and the direct OpenAI-compatible client is constructed without configurable timeout or retry options.

## Solution Assessment

Unclear. The implementation is in the right owner module and uses supported OpenAI SDK options, but it is not the final best solution until maintainers decide whether custom `baseUrl` should allow configurable retries/env expansion and the release/docs surfaces are completed.
