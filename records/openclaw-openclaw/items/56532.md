## Summary

Keep open: current main still lacks config-backed memory-lancedb embedding timeout/retry support, and the PR is useful but should not land until the custom `baseUrl` retry trust model and small docs/wording issues are resolved.

## What This Changes

The PR adds memory-lancedb `embedding.timeoutMs` and `embedding.maxRetries` parsing, manifest/UI schema entries, OpenAI client wiring, config tests, and a changelog entry.

## Best Possible Solution

Land a scoped memory-lancedb update that keeps this plugin-owned behavior out of core, documents the new config knobs, fixes fallback wording, and explicitly handles the trusted-config policy for custom OpenAI-compatible endpoints.

## Reproduction Assessment

Yes. Source inspection on current main shows `timeoutMs` and `maxRetries` are absent from both runtime parsing and manifest schema, and the direct OpenAI-compatible client is built without config-backed timeout/retry options.

## Solution Assessment

Unclear. The plugin-local client wiring is narrow and uses supported OpenAI SDK options, but the final solution should first settle custom-endpoint retry policy and align docs/changelog wording with the actual defaults.

## Security Review

Status: needs_attention

Summary: The diff is not obviously unsafe, but it expands retry control on the existing arbitrary custom embedding endpoint path and needs an explicit maintainer trust-model decision.

Concerns:

- **[medium] Clarify custom baseUrl retry trust model:** `extensions/memory-lancedb/index.ts:334`
  - body: Current main already lets trusted plugin config choose `embedding.baseUrl` with env expansion, but this PR lets config raise direct-client retries above the SDK default. If plugin config is treated as untrusted input, that can amplify requests to arbitrary configured endpoints; maintainers should document the trusted-local-config model or constrain retry overrides when a custom `baseUrl` is used.
