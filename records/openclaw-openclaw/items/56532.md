## Summary

Keep open: current main still lacks the requested memory-lancedb timeout/retry config, and the PR is plausible but needs docs/wording cleanup plus maintainer decision on retry behavior for custom embedding endpoints.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` to memory-lancedb runtime config, manifest schema/UI hints, direct OpenAI-compatible client options, config tests, and the changelog.

## Best Possible Solution

Land a scoped memory-lancedb update that documents the timeout/retry knobs, aligns fallback wording with actual defaults and manifest rejection behavior, and explicitly handles retry policy for custom baseUrl endpoints.

## Reproduction Assessment

Yes. Source inspection on current main shows `timeoutMs` and `maxRetries` are absent from the allowed runtime keys and manifest schema, and the direct OpenAI-compatible client is constructed without those options.

## Solution Assessment

Unclear. The direct OpenAI client wiring uses supported SDK options and stays within the memory-lancedb plugin, but the final patch should resolve docs, fallback wording, and the custom-baseUrl retry trust model before merge.

## Security Review

Status: needs_attention

Summary: The diff needs maintainer security review because it adds configurable retry control to the existing arbitrary custom embedding endpoint path.

Concerns:

- **[medium] Clarify custom baseUrl retry trust model:** `extensions/memory-lancedb/index.ts:334`
  - body: Current main already lets trusted plugin config choose `embedding.baseUrl` with env expansion, but this PR allows config to raise direct-client retries up to 5. That can amplify requests to arbitrary configured endpoints if config is untrusted, so maintainers should explicitly accept and document the trusted-local-config model or constrain retry overrides when a custom baseUrl is used.
  - confidence: 0.78
