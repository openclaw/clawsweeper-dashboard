## Summary

Keep open: current main still lacks the requested memory-lancedb timeout/retry config, and the PR remains useful, but it cannot merge as-is because the branch is based on the older eager OpenAI client path, lacks real behavior proof, and needs a custom-endpoint retry policy decision.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` config/schema/UI/test wiring and passes those values into the direct OpenAI-compatible memory-lancedb embedding client.

## Best Possible Solution

Land a rebased plugin-local implementation that documents the new options, threads them through current main's lazy direct-client path, and explicitly resolves the trusted custom-endpoint retry policy with real runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main rejects `timeoutMs` and `maxRetries` in both the runtime parser and manifest schema, and the direct OpenAI-compatible client has no config-backed constructor timeout/retry options. I did not run live memory-lancedb traffic in this read-only review.

## Solution Assessment

Not yet. Plugin-local config and OpenAI SDK wiring are the right direction, but the branch should be rebased onto current main's lazy client construction, align docs and fallback wording, and settle the custom-endpoint retry policy before merge.
