## Summary

Keep open: the direct-client wiring and terminal proof look sound, but the public docs and manifest still present the timeout/retry budget as if provider-auth embeddings honor it, while the source only applies those knobs on the direct `embedding.apiKey` OpenAI-compatible path.

## What This Changes

The branch adds optional `memory-lancedb` `embedding.timeoutMs`/`embedding.maxRetries` config, manifest/docs/changelog coverage, and passes configured values into the direct OpenAI-compatible embedding client.

## Best Possible Solution

Align the public surface with the actual direct-client runtime path, or intentionally extend the provider embedding contract so provider-auth OpenAI configs honor the same timeout/retry budget with tests.

## Reproduction Assessment

Yes at source/dependency level: current main lacks operator-configurable OpenAI constructor timeout/retry in the direct memory-lancedb client, and OpenAI 6.38.0 defaults to a 600000ms timeout and two retries. The remaining PR mismatch is also source-reproducible because the added docs example omits `embedding.apiKey`, which routes to `ProviderAdapterEmbeddings` where the new fields are not used.

## Solution Assessment

Not yet. The direct-client implementation is a maintainable narrow path, but the docs and manifest need to stop promising the budget for provider-auth embeddings unless the provider embedding contract is deliberately extended and tested.
