## Summary

Keep open: current main still lacks the requested memory-lancedb timeout/retry config, and the PR targets the right plugin-owned direct OpenAI-compatible client path, but it still needs docs/wording cleanup plus maintainer review of the custom-baseUrl retry trust model before merge.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` to memory-lancedb config parsing, manifest schema/UI hints, direct OpenAI-compatible embedding client construction, config coverage, and the changelog.

## Best Possible Solution

Land a scoped memory-lancedb update that documents the new timeout/retry knobs, fixes fallback wording, and either documents or constrains retry behavior for custom OpenAI-compatible endpoints.

## Reproduction Assessment

Yes. Source inspection on current main shows `timeoutMs` and `maxRetries` are rejected by both runtime parsing and manifest schema, and the direct OpenAI-compatible client is constructed without config-backed timeout/retry options.

## Solution Assessment

Unclear. The direct client wiring is a narrow plugin-owned implementation using supported OpenAI SDK options, but the final patch should resolve docs, fallback wording, and the custom-baseUrl retry trust model first.

## Security Review

Status: needs_attention

Summary: The diff needs maintainer security review because it adds configurable retry control to the existing arbitrary custom embedding endpoint path.

Concerns:

- **[medium] Clarify custom baseUrl retry trust model:** `extensions/memory-lancedb/index.ts:334`
  - body: Current main already lets trusted plugin config choose `embedding.baseUrl` with env expansion, but this PR allows config to raise direct-client retries up to 5. That can amplify requests to arbitrary configured endpoints if config is untrusted, so maintainers should explicitly accept and document the trusted-local-config model or constrain retry overrides when a custom baseUrl is used.
  - confidence: 0.78
