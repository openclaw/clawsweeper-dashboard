## Summary

Keep open. Current main does not implement the requested memory-lancedb timeout/retry config, and this PR is still useful, but it should not be treated as cleanup-closeable or automatically repaired until maintainers resolve the custom baseUrl plus retry trust-model question and the public docs/changelog gaps.

## What This Changes

The PR adds `embedding.timeoutMs` and `embedding.maxRetries` to the memory-lancedb config parser, manifest schema/UI hints, OpenAI-compatible direct client construction, and config validation tests.

## Best Possible Solution

Land a scoped memory-lancedb change that keeps the existing bounded auto-recall behavior, adds direct OpenAI-compatible timeout/retry settings with manifest/runtime/test coverage, documents the new knobs publicly, adds a changelog entry, and explicitly decides how custom `baseUrl` plus retry configuration should be constrained or trusted.

## Reproduction Assessment

Yes. On current main, adding `embedding.timeoutMs` or `embedding.maxRetries` to memory-lancedb config is rejected by manifest/runtime validation, and the direct OpenAI-compatible client has no config-backed constructor timeout or retry settings.

## Solution Assessment

Unclear. The PR changes the right owner module and uses supported OpenAI SDK options, but it is not the final best solution until docs/changelog are added and maintainers decide whether retries on custom `baseUrl` should be allowed, capped differently, or documented as trusted local config only.
