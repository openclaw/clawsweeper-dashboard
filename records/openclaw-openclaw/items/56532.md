## Summary

Keep this PR open: the direct-client timeout/retry wiring is close and the supplied terminal proof is useful, but the latest docs still show a provider-auth config path that does not receive the new budget knobs and describe invalid values differently from the implementation.

## What This Changes

The branch adds optional `embedding.timeoutMs` and `embedding.maxRetries` config, manifest metadata, docs, changelog, and config tests for memory-lancedb, and passes the values into the direct OpenAI-compatible embedding client only when explicitly configured.

## Best Possible Solution

Keep the opt-in direct OpenAI-compatible client wiring, but align docs and manifest wording to that runtime scope unless maintainers intentionally extend the provider embedding contract with timeout/retry support and focused coverage.

## Reproduction Assessment

Yes at source/dependency level: current main constructs the direct OpenAI embedding client without a configurable constructor timeout/retry budget, and OpenAI 6.38.0 defaults to a 600000ms timeout with 2 retries. I did not run a full Gateway scenario in this read-only review.

## Solution Assessment

No, not quite: the opt-in direct-client wiring is a maintainable narrow fix, but the public docs and manifest must match that direct-client scope or the provider adapter contract needs an intentional extension with tests.
