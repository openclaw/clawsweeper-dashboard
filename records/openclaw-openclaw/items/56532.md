## Summary

Keep open: this is a useful, focused availability improvement for the direct `memory-lancedb` OpenAI-compatible client, and the supplied terminal proof is convincing. The branch still needs a small public-surface repair because the docs and manifest present the new budget knobs as general embedding controls even though provider-auth configs do not consume them.

## What This Changes

Adds optional `memory-lancedb` `embedding.timeoutMs` and `embedding.maxRetries` config, manifest/docs/changelog coverage, validation tests, and direct OpenAI-compatible client wiring.

## Best Possible Solution

Ship the direct-client timeout/retry knobs after the docs and manifest clearly scope them to `embedding.apiKey`/direct OpenAI-compatible clients, or deliberately extend the provider embedding contract with timeout/retry support and focused tests.

## Reproduction Assessment

Yes, at source/dependency level: current main lacks direct OpenAI constructor timeout/retry config, OpenAI 6.38.0 defaults to 600000ms and two retries, and the PR proof shows a configured direct client failing fast against a blackhole endpoint. The remaining docs/manifest mismatch is also source-reproducible by tracing the provider-auth example to `ProviderAdapterEmbeddings`.

## Solution Assessment

No, not quite: the direct-client implementation is a narrow maintainable solution and preserves SDK defaults for upgrades, but the public surface still overstates where the knobs apply. The safer fix is to scope docs/manifest to the direct client unless maintainers intentionally extend the provider embedding contract.
