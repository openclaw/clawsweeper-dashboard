## Summary

Keep open: current main still lacks the requested memory-lancedb timeout/retry config, but this PR is not merge-ready because it needs to be rebased onto the current lazy embedding-client implementation, documented, and backed by real behavior proof.

## What This Changes

This PR adds memory-lancedb `embedding.timeoutMs` and `embedding.maxRetries` parsing/schema/UI wiring, applies them to the direct OpenAI-compatible embedding client, adds config tests, and updates the changelog.

## Best Possible Solution

Land a rebased plugin-local implementation that threads timeout/retry through the current lazy direct-client path, documents defaults/ranges/scope, and explicitly resolves the custom-endpoint retry policy.

## Reproduction Assessment

Yes, by source inspection: current main rejects `timeoutMs` and `maxRetries` in both the runtime parser and manifest schema, and the direct OpenAI-compatible client has no config-backed retry/timeout constructor options. I did not run live memory-lancedb traffic in this read-only review.

## Solution Assessment

Not yet. Plugin-local config and OpenAI SDK wiring are the right direction, but the patch should be rebased onto current main's lazy client construction and should align docs, fallback wording, and custom-endpoint retry policy before merge.
