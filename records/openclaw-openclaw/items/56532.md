## Summary

Keep open. The requested memory-lancedb timeout/retry config is not on current main and the PR is useful, but it is not merge-ready because it is stale against the current lazy embedding runtime, lacks real behavior proof, needs docs/wording cleanup, and still needs a maintainer decision on retry policy for custom endpoints.

## What This Changes

The PR adds memory-lancedb `embedding.timeoutMs` and `embedding.maxRetries` config/schema/UI/test/changelog wiring and passes those values into the direct OpenAI-compatible embedding client.

## Best Possible Solution

Rebase the plugin-local timeout/retry config onto current main's lazy direct-client path, document the options, align fallback wording with behavior, settle the custom-endpoint retry policy, and add redacted real runtime proof.

## Reproduction Assessment

Yes, for the config gap: current main rejects `timeoutMs` and `maxRetries` in both runtime parsing and manifest validation, and the direct client has no config-backed default timeout/retry knobs. I did not run live memory-lancedb traffic or reproduce a rate-limit storm in this read-only review.

## Solution Assessment

No, not yet. Plugin-local wiring is the right direction, but the current branch should be rebased onto the lazy `clientPromise`/generic `post('/embeddings')` path and needs docs, wording, security-policy review, and real behavior proof before merge.
