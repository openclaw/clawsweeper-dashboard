## Summary

Keep open: the direct-client timeout/retry wiring is useful and proof-backed, but the public config hints still present these as generic embedding API controls even though provider-backed embeddings ignore them.

## What This Changes

Adds optional `memory-lancedb` `embedding.timeoutMs` and `embedding.maxRetries` config/schema/UI fields, forwards them to the direct OpenAI-compatible embedding client, and adds config tests plus a changelog entry.

## Best Possible Solution

Ship the direct-client timeout/retry knobs after the manifest/config UI and release note clearly say they require the direct OpenAI-compatible client, or deliberately extend the provider embedding contract with focused tests.

## Reproduction Assessment

Yes, at source/dependency level: current main lacks constructor-level OpenAI timeout/retry config, OpenAI 6.38.0 defaults to a 10-minute timeout with 2 retries, and the PR body includes terminal blackhole proof for the direct client. The remaining mismatch is source-reproducible by tracing provider-auth configs through `ProviderAdapterEmbeddings`.

## Solution Assessment

No, not quite: the direct-client implementation is the right narrow runtime path, but the public config copy must match that scope before merge. The alternative is a larger provider-contract change with tests for provider-backed embeddings.
