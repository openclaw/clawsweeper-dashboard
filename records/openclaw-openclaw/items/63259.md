## Summary

Keep open: current main still lacks the requested per-model probe flag, and the PR is directionally correct but still has a summary type mismatch plus missing user-facing docs/changelog coverage before merge.

## What This Changes

The PR adds `models status --probe --all` option plumbing, expands probe target selection to every configured candidate model, warns when `--all` is used without `--probe`, and adds focused route-parser and probe-target tests.

## Best Possible Solution

Land a refreshed version that preserves the current one-model-per-provider default, exposes `probeAll` consistently in the probe summary contract, and documents the new CLI flag with a changelog entry.

## Reproduction Assessment

Yes. The current-main gap is reproducible by reading the CLI/parser/probe-target path: no `--all` status flag is registered or forwarded, and target building selects one model per provider.

## Solution Assessment

No, not as-is. The implementation strategy is the right narrow path because it reuses the existing probe builder and execution flow, but the exported summary type and user-facing docs/release notes need to match the new surface.
