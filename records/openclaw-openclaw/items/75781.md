## Summary

Keep open. The linked bug appears valid, but this PR does not actually fix the current auth-none backend path because the changed helper is still gated by `sharedAuthOk`, which current main never sets for `authMethod: "none"`.

## What This Changes

The PR adds `"none"` to `shouldSkipLocalBackendSelfPairing`'s shared-secret auth method check and adds unit coverage for `authMethod: "none"` backend self-pairing cases.

## Best Possible Solution

Handle `authMethod: "none"` as a local-backend self-pairing satisfied state only inside the narrow backend/local/no-browser-origin bypass, with regression tests that reflect the real resolver output where `sharedAuthOk` is false. Avoid making `sharedAuthOk` globally true for auth-none because that value feeds broader device-identity policy.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: with `gateway.auth.mode: "none"`, auth resolves to method `none`, `sharedAuthOk` remains false, and a device-less backend connection still reaches the `device identity required` close path.

## Solution Assessment

No. Adding `"none"` to the shared-secret method list is not sufficient because the surrounding condition still requires `sharedAuthOk`; the safer fix is a local-backend-specific auth-none branch plus realistic tests.
