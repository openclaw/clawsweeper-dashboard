## Summary

Keep open. The linked bug is valid and this PR is paired with open #75780, but the patch is not merge-ready because it turns `auth.mode: "none"` into generic `sharedAuthOk`, affecting device-less operator WebSocket connections before the backend-locality gate constrains the bypass.

## What This Changes

The PR treats `auth.mode: "none"` as shared-auth-like for gateway backend WebSocket self-pairing and adds helper/auth-state tests for that mode.

## Best Possible Solution

Keep the no-auth allowance scoped to trusted local backend self-pairing, preserve generic device-identity policy for other operator clients, and cover the real handshake path with remote/proxy/browser negative cases plus a changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main returns auth method `none`, the PR makes that set `sharedAuthOk`, and current missing-device policy allows operator connections on that flag before backend self-pairing constrains the path; #75780 supplies the user-facing backend RPC failure steps.

## Solution Assessment

No. The intended fix is narrower than this PR: handle `none` at the backend-locality self-pairing boundary or equivalent scoped seam, not by reclassifying all no-auth WebSocket handshakes as shared-auth-like.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep none auth out of the global shared-auth bypass:** `src/gateway/server/ws-connection/auth-context.ts:118`
  - body: Adding `authResult.method === "none"` makes every successful no-auth WebSocket handshake set `sharedAuthOk`. `evaluateMissingDeviceIdentity` allows `role=operator` with that flag before the backend self-pairing decision scopes anything, so device-less non-backend or proxied operator clients can be admitted. Keep the no-auth allowance inside the local backend self-pairing path instead.
  - confidence: 0.9
- **[P2] Cover the resolver path for none-auth loopback:** `src/gateway/server/ws-connection/handshake-auth-helpers.ts:268`
  - body: The new helper test passes `locality: "shared_secret_loopback_local"` directly, but the real path gets that value from `resolvePairingLocality`, whose loopback shared-auth checks still only recognize token/password. If no-auth non-direct loopback backend clients should work, update or test the resolver/handshake path instead of only testing a preselected locality.
  - confidence: 0.82

  - body: This is a user-facing gateway fix for a documented auth mode and linked bug, but the PR does not add a `CHANGELOG.md` entry. Add a single-line Fixes entry under the active version before merge.
