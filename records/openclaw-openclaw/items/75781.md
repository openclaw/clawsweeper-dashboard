## Summary

Keep open: the PR still does not fix the linked auth-none backend failure because the missing-device decision ignores the helper skip and still reaches the `device identity required` close path.

## What This Changes

The PR adds a local backend self-pairing skip for `authMethod: "none"`, adds helper/auth-state tests, and inserts a placeholder changelog edit.

## Best Possible Solution

Add a scoped missing-device allowance for local, browser-origin-free `gateway-client` backend connects under `gateway.auth.mode: "none"`, preserve remote/browser negatives, cover the full handshake path, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: auth-none resolves with `sharedAuthOk: false`, then a device-less local backend connect reaches `reject-device-required` and closes with 1008.

## Solution Assessment

No. The helper-only change is not the best fix because the rejection happens in the missing-device policy; the allow signal needs to be threaded into that policy and tested through the real handshake path.
