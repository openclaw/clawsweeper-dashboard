## Summary

Keep open: current main still rejects the reported device-less `auth.mode: "none"` backend path, and this PR’s helper-only change still does not reach the missing-device decision that closes the socket.

## What This Changes

This PR makes local browser-origin-free `authMethod: "none"` backend clients skip self-pairing in the handshake helper, adds auth-state/helper tests, and inserts a blank line in `CHANGELOG.md`.

## Best Possible Solution

Thread a narrow local backend no-auth allow signal into the missing-device policy or handler, preserve the existing remote/browser-origin restrictions, add full handshake-path regression coverage, and add the required changelog entry.

## Reproduction Assessment

Yes, by source inspection. With `gateway.auth.mode: "none"`, auth resolves to method `none` with `sharedAuthOk: false`, so a device-less backend connect reaches `evaluateMissingDeviceIdentity` and closes with `1008: device identity required` before this helper skip can allow it.

## Solution Assessment

No. The helper-only change is not the best fix because it changes a skip signal that the missing-device gate does not consume; the narrow maintainable fix is to carry that scoped local-backend no-auth allowance into the gate and test the real connect path.
