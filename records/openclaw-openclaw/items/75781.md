## Summary

Keep open. The linked bug is valid, but the latest PR still does not make the no-auth backend path reachable because `auth.mode: "none"` keeps `sharedAuthOk` false while the changed helper remains gated on `sharedAuthOk`.

## What This Changes

The PR adds `authMethod: "none"` to the gateway backend self-pairing helper, adds no-auth auth-state/helper tests, and adds a gateway auth changelog entry.

## Best Possible Solution

Add a narrow no-auth allowance for browser-origin-free local backend `gateway-client` coordination, and cover auth resolution, locality, full handshake success, and remote/browser negative cases.

## Reproduction Assessment

Yes. The linked issue gives concrete user steps, and source inspection reproduces the decision chain: `auth.mode: "none"` resolves to method `none`, `sharedAuthOk` stays false, the helper remains gated by `sharedAuthOk`, and missing-device handling can close with `1008: device identity required`.

## Solution Assessment

No. The latest PR avoids the earlier broad shared-auth flag regression, but adding `none` to `usesSharedSecretAuth` is unreachable for the real no-auth state; the safer fix is a scoped backend-local no-auth seam in the missing-device/pairing policy.
