## Summary

Keep open. The related bug report is plausible, but this PR is not merge-ready because the proposed helper change does not make `gateway.auth.mode: "none"` satisfy the real `sharedAuthOk` guard in the WebSocket handshake, so the reported device-less backend path would still reject with `device identity required`.

## What This Changes

The PR changes the gateway WebSocket handshake helper to treat auth method `"none"` as shared-secret-like for local backend self-pairing skips and adds helper-level unit cases for that auth method.

## Best Possible Solution

Make `auth.mode: "none"` eligibility live at the auth-state or pairing-decision boundary that the real handshake uses, scoped to trusted local backend clients without browser-origin/proxy exposure, and validate it with an integrated backend connect or gateway-call test plus remote/browser-origin negatives.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `auth.mode: "none"` produces `authMethod: "none"` and `sharedAuthOk: false`, then the device-less backend path keeps `skipLocalBackendSelfPairing` false and falls through to `device identity required`.

## Solution Assessment

No. The helper-only change is not sufficient because it updates the auth-method half of the condition while leaving the actual `sharedAuthOk` input false; the safer fix is to align the real auth-state/pairing decision and test that path directly.
