## Summary

Keep open. The PR addresses the earlier helper-only gap by also setting `sharedAuthOk` for `auth.mode: "none"`, but that moves the fix to a generic auth-state flag consumed by the whole missing-device policy, not only the local backend self-pairing path. Because this widens gateway auth/device-identity bypass semantics, it needs maintainer/security-aware revision before merge.

## What This Changes

The PR changes gateway WebSocket auth handling to treat auth method `"none"` as shared-auth-like for backend self-pairing and adds helper/auth-state tests for that mode.

## Best Possible Solution

Keep the fix scoped to trusted local backend self-pairing: make the none-auth case eligible at the backend/locality decision boundary, preserve existing device-identity behavior for other operator clients, and cover the real handshake or gateway-call path plus remote/proxy/browser negatives.

## Reproduction Assessment

Yes. The source-level reproduction is clear from the PR diff: `auth.mode: "none"` becomes `sharedAuthOk: true`, then the current missing-device policy allows `role=operator` before checking whether the client is the local backend self-pairing case.

## Solution Assessment

No. The PR's second commit fixes the earlier guard input, but at a too-broad boundary; the safer solution is a backend-locality-specific none-auth allowance with integrated tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep none auth out of the global shared-auth bypass:** `src/gateway/server/ws-connection/auth-context.ts:118`
  - body: This makes every successful `auth.mode: "none"` WebSocket handshake set `sharedAuthOk`. The missing-device policy treats `role=operator && sharedAuthOk` as enough to skip device identity before the backend-specific self-pairing decision constrains anything, so device-less non-backend or proxied operator clients can be admitted with their requested scopes. Scope the none-auth allowance to the local backend self-pairing path and add negative remote/proxy/browser cases.
  - confidence: 0.9
- **[P2] Test the locality resolver for the none-auth loopback case:** `src/gateway/server/ws-connection/handshake-auth-helpers.ts:268`
  - body: The new helper case manually passes `shared_secret_loopback_local`, but the real path gets that value from `resolvePairingLocality`, whose shared-secret loopback branch still only recognizes token/password auth. If auth none is meant to work for non-direct loopback backend clients, update or test the resolver instead of only testing `shouldSkipLocalBackendSelfPairing` with a preselected locality.
  - confidence: 0.82
