## Summary

Keep open. The PR is not merge-ready because the added no-auth branch remains behind `sharedAuthOk`, while current auth resolution keeps `authMethod: "none"` from setting `sharedAuthOk`, so the linked backend device-identity failure can still reach the `device identity required` close path.

## What This Changes

The PR adds `authMethod: "none"` to `shouldSkipLocalBackendSelfPairing`, adds no-auth auth-state/helper tests, and adds a gateway auth changelog entry.

## Best Possible Solution

Add a narrow missing-device allowance for browser-origin-free local backend `gateway-client` coordination under `gateway.auth.mode: "none"`, without marking no-auth as shared auth globally, and cover full auth resolution, handshake success, and remote/browser negative cases.

## Reproduction Assessment

Yes. The linked issue gives concrete steps, and source inspection reproduces the path: `auth.mode: "none"` resolves to method `none`, `sharedAuthOk` remains false, the helper bypass is not reachable, and the missing-device handler can close with `1008: device identity required`.

## Solution Assessment

No. Adding `none` to `usesSharedSecretAuth` is not sufficient because the real no-auth state does not satisfy `sharedAuthOk`; the safer fix is a scoped local-backend no-auth seam in the missing-device/pairing policy.
