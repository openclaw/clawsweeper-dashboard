## Summary

Keep open. The PR still does not fix the reported no-auth backend RPC path because the helper change remains gated on `sharedAuthOk`, while current `auth.mode: "none"` resolution leaves `sharedAuthOk` false before the missing-device rejection.

## What This Changes

The PR adds `authMethod: "none"` coverage to the local backend self-pairing helper, adds gateway auth tests, and updates the changelog.

## Best Possible Solution

Add a narrow missing-device allowance for browser-origin-free local backend `gateway-client` coordination under `gateway.auth.mode: "none"`, without marking no-auth as shared auth globally, and cover full auth resolution plus remote/browser negative cases.

## Reproduction Assessment

Yes. The linked issue gives concrete steps, and source inspection on current main confirms `auth.mode: "none"` resolves to method `none`, leaves `sharedAuthOk` false, and reaches the `1008: device identity required` rejection path.

## Solution Assessment

No. Adding `none` to `usesSharedSecretAuth` is not sufficient because the real no-auth state does not satisfy the helper's `sharedAuthOk` gate; the safer fix belongs in a scoped local-backend no-auth missing-device policy.
