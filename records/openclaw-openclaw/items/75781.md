## Summary

Keep open. The latest PR revision avoids the earlier broad `sharedAuthOk` bypass, but it still does not fix the reported backend RPC path because `auth.mode: "none"` resolves with `sharedAuthOk=false` while the local backend skip still requires `sharedAuthOk` before it can help.

## What This Changes

The PR adds `auth.mode: "none"` handling to the gateway backend self-pairing helper, adds auth-state/helper tests, and adds a gateway auth changelog entry.

## Best Possible Solution

Keep `sharedAuthOk` scoped to real shared-secret/trusted-proxy auth, and add a narrow local-backend no-auth allowance in the handler or missing-device policy with full handshake regression tests and remote/browser negative cases.

## Reproduction Assessment

Yes. #75780 gives concrete user steps, and source inspection reproduces the decision chain: `auth.mode:none` returns method `none`, `sharedAuthOk` remains false, and device-less backend connects hit the missing-device rejection path.

## Solution Assessment

No. The current PR shape is narrower than the earlier unsafe global bypass, but it still requires `sharedAuthOk` in the backend skip path and therefore does not solve the actual no-auth backend connection failure.

- **[P1] Make the none-auth backend path bypass missing-device rejection:** `src/gateway/server/ws-connection/handshake-auth-helpers.ts:268-270`

In the real handler path, `evaluateMissingDeviceIdentity` runs before pairing and rejects device-less backend/operator connects when `sharedAuthOk` is false, so subagent and gateway RPCs under no-auth still close with `device identity required`. Scope the no-auth allowance into the local-backend missing-device decision or pass a real backend-local allow flag through the handler.
  - confidence: 0.93

That test cannot catch the reported failure; add coverage through the resolver/message-handler path or set the helper input to the reachable state.
  - confidence: 0.88
