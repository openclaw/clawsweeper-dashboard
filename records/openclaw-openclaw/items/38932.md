## Summary

Keep open. The docs addition is not already on current main and is potentially useful, but the proposed security-hardening guide still has a correctness issue: its forbidden-path verification can test an unrelated caller profile path instead of a directory actually denied by the guide, which can give users false confidence.

## What This Changes

The PR adds a native Windows no-Docker/WSL host-hardening fallback guide, plus docs navigation and cross-links from Windows and sandboxing docs.

## Best Possible Solution

Keep the docs direction only if maintainers want a native-Windows fallback, but make the verification target an explicit denied sibling path and keep the WSL2/Docker-first framing prominent.

## Reproduction Assessment

Yes for the docs defect: the PR source and a local PowerShell quoting probe show that the forbidden-path command can expand the caller's USERPROFILE before runas executes. Full Windows ACL behavior was not live-verified in this review.

## Solution Assessment

No, not as currently written. The narrow fix is to verify an explicit blocked sibling directory from the same parent used by the deny loop, with clear expected failure wording; the broader page still needs maintainer product/security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Verify a directory this guide actually denies:** `docs/gateway/windows-no-docker-hardening.md:158`
  - body: The forbidden-path check expands `$env:USERPROFILE` in the calling PowerShell session before `runas` starts the constrained user, so it can probe the caller's profile instead of a sibling directory denied in step 4. That path may fail for unrelated Windows profile ACL reasons, letting users think the hardening worked even if the deny loop was skipped or pointed at the wrong parent; use an explicit blocked path from the sibling-deny set and state the expected failure.
  - confidence: 0.91
