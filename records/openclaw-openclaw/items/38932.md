## Summary

Keep this PR open: current main still lacks the proposed Windows no-Docker hardening fallback, but the PR is not ready because its security-hardening verification can false-pass and the adjacent sandboxing guidance needs owner review.

## What This Changes

The PR adds a Windows native no-Docker/WSL hardening docs page, registers it in docs navigation, and cross-links it from Windows and sandboxing docs.

## Best Possible Solution

Keep the fallback only after the verification targets an explicitly denied path, the links are rebased onto current docs, and security docs owners approve the threat-model wording.

## Reproduction Assessment

Yes for the PR docs defect. Inspecting PR head is enough: the deny loop blocks sibling directories under the workspace parent, while the forbidden-path test checks `$env:USERPROFILE` instead of one of those denied paths.

## Solution Assessment

No, not as-is. The docs direction is narrow and plausible, but the verification should test an explicitly denied sibling path or a reader-supplied blocked path before this security-hardening guide is merged.
