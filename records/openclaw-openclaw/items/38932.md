## Summary

Keep open. The PR adds a unique native-Windows hardening fallback that is not on current main, but it should not merge as-is because the guide can falsely validate the ACL boundary and misses an elevated-shell prerequisite for the local-user and ACL steps.

## What This Changes

This PR adds a docs-only native Windows hardening fallback guide and links it from gateway sandboxing and Windows platform docs.

## Best Possible Solution

Rebase the guide onto the current Windows and sandboxing docs, keep Docker/WSL2 as the primary recommendation, and publish only a carefully scoped fallback that validates the actual ACL boundary.

## Reproduction Assessment

Yes for the PR review findings from source inspection: the latest head's forbidden-path sample checks a caller-expanded profile path instead of a denied sibling directory, and the setup path omits the elevated-shell prerequisite before user and ACL commands. I did not run the full Windows ACL workflow live.

## Solution Assessment

Unclear. A native Windows fallback guide could be useful, especially with the linked reference material, but the maintainable version should be rebased onto current docs and should not ship with a false-confidence verification step.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] State that setup needs elevated PowerShell:** `docs/gateway/windows-no-docker-hardening.md:58`
  - body: The guide starts the copy-paste path with only “In PowerShell,” but creating a local user and changing ACLs under the workspace parent require an elevated/admin shell on Windows. Readers following this from a normal shell will fail at `New-LocalUser` or the `icacls` steps before the hardening flow is usable.
  - confidence: 0.86
- **[P2] Verify a directory denied by the ACL step:** `docs/gateway/windows-no-docker-hardening.md:158`
  - body: The forbidden-path check probes `$env:USERPROFILE`, which is expanded by the caller before `runas` and is not one of the sibling directories denied in step 4. This can pass even when the deny loop was skipped or targeted incorrectly, so the test should use an explicitly blocked sibling path or a user-provided denied path.
  - confidence: 0.95
