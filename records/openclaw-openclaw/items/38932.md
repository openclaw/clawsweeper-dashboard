## Summary

Keep open: current main still lacks this fallback page, and PR head has a security-doc correctness issue because its forbidden-path check can pass without validating the ACL deny step.

## What This Changes

The PR adds a docs-only native Windows no-Docker/WSL hardening fallback page, registers it in docs navigation, and cross-links it from Windows and sandboxing docs.

## Best Possible Solution

Publish the fallback only after the verification targets an explicitly denied sibling or configured blocked path, links/navigation are rebased onto current docs, and secops approves the threat-model wording.

## Reproduction Assessment

Yes for the PR docs defect. Source inspection of PR head shows the deny loop blocks sibling directories under the workspace parent, while the verification command checks the caller-expanded user profile path instead of a path that the guide explicitly denied.

## Solution Assessment

No, not as-is. The docs direction is plausible, but the verification needs to test the hardening step itself and the branch should be rebased into the current docs structure before security-owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify a path the guide explicitly blocks:** `docs/gateway/windows-no-docker-hardening.md:158`
  - body: The forbidden-path check uses `$env:USERPROFILE` inside the outer `runas` command, so PowerShell can expand it to the launching user's profile path. That path may be inaccessible because of normal Windows profile ACLs even if the sibling-deny loop never ran, giving readers false assurance; capture an explicitly denied sibling path or require a `$BlockedPath` and test that instead.
  - confidence: 0.9
