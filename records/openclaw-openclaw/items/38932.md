## Summary

Keep open: current main does not include this native Windows no-Docker fallback, and the PR addresses a real docs gap, but the security verification snippet can give false confidence because it tests a profile path instead of a path denied by the guide.

## What This Changes

The PR adds a docs-only native Windows fallback hardening guide for running OpenClaw without Docker or WSL2 and links it from gateway sandboxing and Windows platform docs.

## Best Possible Solution

Publish a narrowly framed fallback only after the verification step proves the ACL boundary the guide creates, while keeping WSL2 and Docker sandboxing as the recommended paths.

## Reproduction Assessment

Yes, by source inspection of the final PR doc: the hardening step denies sibling directories, but the verification command checks `$env:USERPROFILE` instead of a denied sibling path. That makes the doc bug reproducible without needing to run the whole Windows flow.

## Solution Assessment

No, not yet. The guide should verify a known denied sibling path or an explicit `$BlockedPath`, then a maintainer should make the product/security call on publishing the fallback.
