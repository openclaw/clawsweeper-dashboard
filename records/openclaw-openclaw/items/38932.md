## Summary

Keep open. Current main does not contain the new Windows no-Docker hardening page, and the PR still has a blocking security-docs correctness issue: the forbidden-path verification tests a profile path instead of a path the guide explicitly denies, so users can get false confidence in the ACL boundary.

## What This Changes

This PR adds a docs-only native Windows hardening fallback guide for users without Docker/WSL and links it from gateway sandboxing, Windows platform docs, and docs navigation.

## Best Possible Solution

Publish the guide only after the verification step targets an explicitly denied sibling or configured blocked path, keeps Docker/WSL as the primary recommendation, and receives secops/docs approval.

## Reproduction Assessment

Yes for the PR defect: source inspection of the PR head shows the deny loop targets workspace-parent siblings while the forbidden-path test probes `$env:USERPROFILE`. I did not establish live Windows execution of the full hardening flow.

## Solution Assessment

No, not as-is. The docs direction is plausible, but the verification example should test a path the guide actually blocks and the security framing should be approved by the security/docs owners.
