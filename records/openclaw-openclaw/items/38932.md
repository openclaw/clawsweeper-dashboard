## Summary

Keep open: the guide is not on current main, and the PR still has a blocking docs-security correctness issue where the verification step can falsely validate the intended Windows ACL boundary.

## What This Changes

The PR adds a docs-only native Windows no-Docker/WSL host-hardening fallback guide and links it from gateway sandboxing, Windows platform docs, and docs navigation.

## Best Possible Solution

Publish the guide only after the forbidden-path test validates an explicitly denied path, WSL2/Docker remain the recommended path, and security/docs owners approve the threat model.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the guide denies workspace-parent siblings but tests `$env:USERPROFILE` instead of a denied sibling path. I did not establish live Windows execution of the full hardening flow.

## Solution Assessment

No, not as-is. The docs direction is plausible, but the verification example should test a path the guide actually blocks and the security framing should receive owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Verify an explicitly denied path:** `docs/gateway/windows-no-docker-hardening.md:158`

Point the check at an explicitly denied sibling path, or ask the reader to set a blocked path from the deny loop, so verification proves the hardening step.
