## What This Changes

This PR adds a native Windows host-hardening fallback docs page, registers it in docs navigation, and cross-links it from Windows and sandboxing docs.

## Best Possible Solution

Keep the fallback only if security docs owners approve it, rebase it onto current docs, and verify an explicitly denied sibling path or reader-supplied blocked path.

## Reproduction Assessment

Yes for the docs defect. Inspecting the PR head is enough: step 4 denies sibling directories, while the line 158 forbidden-path test checks `$env:USERPROFILE` instead of one of those denied paths.

## Solution Assessment

No, not as-is. The docs direction is narrow and plausible, but the verification must test the actual ACL boundary and the branch should be updated against current Windows docs.
