## Summary

Keep open: current main does not include this Windows no-Docker hardening guide, and the PR still has a security-doc correctness issue where the forbidden-path verification can give false assurance.

## What This Changes

The PR adds a docs-only native Windows no-Docker/WSL hardening fallback page, docs navigation entry, and cross-links from Windows and sandboxing docs.

## Best Possible Solution

Publish the fallback only after the verification targets an explicitly denied or user-configured blocked path, the docs are rebased into current navigation, and security-doc owners approve the threat-model wording.

## Reproduction Assessment

Yes for the PR docs defect. Source inspection of PR head shows the guide denies sibling directories under the workspace parent, while the forbidden-path test checks a user-profile path that may be inaccessible for unrelated reasons.

## Solution Assessment

No, not as-is. The docs direction is plausible, but the verification must test the hardening rule itself and the security wording needs owner review before this should merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify a path the guide explicitly blocks:** `docs/gateway/windows-no-docker-hardening.md:158`
  - body: The forbidden-path check uses `$env:USERPROFILE` inside the outer `runas` command, so PowerShell can expand it to the launching user's profile path. That path can be inaccessible because of normal profile ACLs even if the sibling-deny loop never ran, giving readers false assurance; capture an explicitly denied sibling path or require a `$BlockedPath` and test that instead.
  - confidence: 0.9
