## Summary

Keep open. Current main still lacks the native Windows no-Docker/WSL hardening fallback, so the PR is not obsolete, but the current PR head is not merge-ready because its forbidden-path verification can pass without proving the sibling-deny ACLs were applied.

## What This Changes

This PR adds a native Windows no-Docker/WSL host-hardening fallback docs page, registers it in the docs navigation, and links it from the Windows and sandboxing docs.

## Best Possible Solution

Keep the Windows host-hardening fallback only if security-doc maintainers want this documented core path, then make the forbidden-path check validate an explicitly denied sibling or reader-supplied blocked path and rebase the links into the current Windows and sandboxing docs.

## Reproduction Assessment

Yes for the docs defect. Static inspection of the PR head shows the ACL deny loop targets sibling directories, while the forbidden-path test probes `$env:USERPROFILE` instead of a denied sibling path.

## Solution Assessment

No, not as-is. The docs direction is narrow and plausible, but the security verification must test the actual hardening step and the branch should be reconciled with current main before merge.
