Keep open. Current main still lacks this native Windows fallback guide, but this PR is not merge-ready because the forbidden-path verification can false-pass, the linked proof is insufficient for security-hardening docs, and the threat-model wording needs owner review.

## What This Changes

This PR adds a docs page for native Windows host hardening without Docker/WSL, adds it to docs navigation, and cross-links it from gateway sandboxing and Windows platform docs.

## Best Possible Solution

Publish a corrected guide only after it verifies a path explicitly blocked by its own ACL step, includes redacted Windows proof, and receives secops/product sign-off.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the guide denies workspace-parent siblings but verifies `$env:USERPROFILE`, which can be inaccessible for unrelated reasons. The real Windows hardening flow still lacks sufficient after-fix proof.

## Solution Assessment

No, not as-is. The docs direction is plausible, but a maintainable security-hardening guide should verify an explicitly denied path and be backed by final Windows proof before publication.
