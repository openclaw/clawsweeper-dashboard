## Summary

Keep open: current main has Control UI/WebChat, PWA/Web Push, device pairing, DM session-scope isolation, and provider-backed Talk/WebRTC voice, but it still does not provide browser-held E2EE chat envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

Request native browser-based E2EE chat for OpenClaw instances, with browser-held keys and isolated encrypted channels instead of relying on third-party chat apps.

## Best Possible Solution

Keep this as a maintainer-led product/security architecture item covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, relay/fallback behavior, transcript semantics, Gateway protocol changes, Control UI UX, docs, and contract tests.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, not a bug report; source inspection verifies the current WebChat behavior rather than a failing reproduction path.

## Solution Assessment

Unclear as an implementation plan. A native E2EE browser channel could solve the stated goal, but WebRTC/relay behavior, key lifecycle, transcript storage, and plugin/core boundaries need maintainer design before implementation.

## Security Review

Summary: Security-sensitive issue; current main’s browser chat is authenticated Gateway RPC, not browser-held E2EE, so any implementation needs explicit security design.
