## Summary

Keep open: current main still provides authenticated Gateway/WebChat RPC with Gateway-visible message text, not the requested browser-held E2EE browser chat transport with encrypted envelopes, key negotiation, or per-device/per-agent encrypted channels.

## What This Changes

The issue proposes a native browser-based E2EE chat transport for OpenClaw with browser-held keys and per-agent/per-device isolation, without requiring a third-party chat app.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after maintainers settle the trust model.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as a concrete implementation plan. The requested direction could address the user problem, but the maintainable solution needs product and security architecture before code can be scoped.
