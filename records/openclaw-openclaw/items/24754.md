## Summary

Keep open: current main and v2026.6.10 still provide authenticated Gateway WebSocket WebChat, browser device identity/pairing, configurable DM session isolation, and Talk/WebRTC voice, but not the requested browser-held E2EE chat transport with encrypted per-device/per-agent browser channels. The live discussion already reached the same keep-open conclusion, and targeted GitHub searches did not find a canonical duplicate or replacement PR.

## What This Changes

The issue requests a native browser-based end-to-end encrypted chat transport with QR pairing, no third-party chat app, and per-agent/per-device isolated encrypted channels.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then implement protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source and release inspection confirm current main lacks encrypted browser chat fields and still routes chat.send plaintext through the Gateway.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is a useful starting point, but it is not the best complete solution until maintainers settle the threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership.
