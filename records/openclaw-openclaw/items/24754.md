## Summary

Keep open: current main has Gateway/WebSocket WebChat, Control UI chat, device pairing, DM session scoping, and Talk voice transports, but it still lacks browser-held E2EE browser chat with encrypted envelopes, browser key negotiation, and per-device/per-agent encrypted channels.

## What This Changes

Request a native no-third-party browser E2EE chat layer with QR/device pairing and per-agent/per-device encrypted channel isolation.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture request, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as a concrete implementation plan. The requested direction may solve the user problem, but WebRTC/WebCrypto, relay fallback, key lifecycle, and transcript policy need maintainer product and security design before code can be scoped.
