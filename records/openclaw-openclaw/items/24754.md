## Summary

Keep open: current main still has authenticated Gateway/WebSocket WebChat and Control UI chat, but it does not implement browser-held E2EE chat with encrypted message envelopes, key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

Request a native browser-based E2EE chat transport for OpenClaw that avoids third-party chat apps and isolates per-agent/per-device conversations.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture request, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

Unclear as a concrete implementation plan. The requested direction may solve the user problem, but the maintainable path needs product and security architecture before code can be scoped.
