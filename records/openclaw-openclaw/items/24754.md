## Summary

Keep open: current main has authenticated Control UI/WebChat, device pairing, Web Push, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but it still lacks the requested browser-held E2EE chat transport and per-device/per-agent encrypted browser channels.

## What This Changes

Request a native browser-based E2EE chat transport for OpenClaw instances with QR pairing, no third-party chat service, and per-device/per-agent encrypted channels.

## Best Possible Solution

Keep this as a maintainer-led product/security architecture item covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, relay fallback, transcript semantics, Gateway protocol changes, Control UI UX, docs, and contract tests.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection verifies that current browser chat is authenticated Gateway RPC rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. A native E2EE browser channel could solve the goal, but key lifecycle, relay/fallback behavior, transcript storage, and core/plugin boundaries need maintainer design first.
