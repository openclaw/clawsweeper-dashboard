## Summary

Keep open: current main still provides authenticated Gateway WebSocket WebChat, Control UI chat, device identity, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE browser chat transport with encrypted envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

The issue requests a native browser-based E2EE chat transport with no third-party chat provider, QR pairing, browser-held keys, and per-device/per-agent encrypted isolation.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, isolated routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture request, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as a concrete implementation plan. The requested direction may address real first-run and trust-boundary problems, but maintainers need to choose the product/security architecture before code can be scoped.
