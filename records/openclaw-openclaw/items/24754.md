## Summary

Keep open: current main still provides authenticated Gateway/WebSocket browser chat and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE browser chat transport with encrypted envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels. Prior review comments reached the same conclusion, and the latest source still routes browser chat as Gateway-visible plaintext.

## What This Changes

The issue requests a built-in browser-based, no-third-party E2EE chat transport with per-agent/per-device isolation and QR-style device pairing.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, isolated routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.
