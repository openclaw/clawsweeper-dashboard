## Summary

Keep open: current main and the latest release still implement WebChat as authenticated Gateway/WebSocket chat with Gateway-visible message text, not the requested browser-held E2EE browser chat transport with encrypted envelopes, key negotiation, or per-device/per-agent encrypted channels.

## What This Changes

Request a native browser-based E2EE chat layer for OpenClaw instances that avoids third-party chat apps and provides per-agent/per-device encrypted channel isolation.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.
