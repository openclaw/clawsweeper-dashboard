## Summary

Keep open: current main and the latest release still provide authenticated Gateway WebSocket WebChat, device auth/pairing, PWA/Web Push, and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE chat transport with encrypted message envelopes, browser key negotiation, or per-device/per-agent encrypted channels.

## What This Changes

Request a built-in browser-based E2EE chat transport with per-device/per-agent keys and no third-party chat app.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then implement protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is a useful starting point, but it is not the best complete solution until maintainers settle the threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership.
