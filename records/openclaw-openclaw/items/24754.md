## Best Possible Solution

Define a maintainer-approved threat model and additive encrypted browser transport contract covering key lifecycle, authorization binding, agent routing, transcript/storage semantics, offline delivery, and relay fallback, with generic Gateway/plugin seams first if an external implementation should precede core UI integration.

## Risks / Open Questions

- The proposal does not yet define key provisioning, recovery, rotation, revocation, or multi-device synchronization.
- Per-device keys must be securely bound to Gateway authorization and the selected agent without creating a second conflicting routing or identity system.
- Encrypted transcript visibility, server-side agent processing, local persistence, search, history reloads, and backup behavior require an explicit threat model.
- Direct WebRTC cannot guarantee connectivity across all NAT and network conditions, so relay ownership, metadata exposure, availability, abuse controls, and operating cost need a product decision.
