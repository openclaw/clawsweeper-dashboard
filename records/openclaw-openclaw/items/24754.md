## Summary

Keep open: current main still provides authenticated Gateway WebSocket browser chat, device identity, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but it does not implement the requested browser-held E2EE chat transport with encrypted envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

Request native browser-based, no-third-party E2EE chat with browser-held keys and per-agent/per-device channel isolation.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, isolated routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.
