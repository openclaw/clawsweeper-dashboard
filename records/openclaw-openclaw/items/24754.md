## Summary

Keep open: current main still uses authenticated Gateway WebSocket chat and provider-backed Talk, not browser-held E2EE chat with encrypted envelopes or per-device/per-agent browser keys.

## What This Changes

Request native browser-based E2EE chat built into OpenClaw, without third-party chat apps, with isolated encrypted channels per agent/device.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol, key lifecycle, UI, docs, and contract tests after maintainers settle routing, transcript visibility, and relay policy.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection of current main verifies that browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. The requested direction could solve the user problem, but the maintainable path needs product/security design before a code change can be scoped.
