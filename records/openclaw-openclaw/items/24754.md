## Summary

Keep open. Current main has WebChat/Control UI chat, browser device identity, secure DM session modes, and provider-backed Talk/WebRTC voice, but it still does not implement browser-held E2EE chat envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels; the existing issue comments reached the same conclusion.

## What This Changes

The issue proposes a native browser-based, browser-held E2EE chat transport for OpenClaw instances, with per-device and per-agent encrypted channels and no third-party chat provider.

## Best Possible Solution

A maintainer-approved Gateway/browser chat design should add encrypted envelopes, browser-held key negotiation and lifecycle, per-device/per-agent routing, transcript/storage semantics, fallback behavior, UI/docs, and contract tests; if plugin-first, define the missing Gateway/plugin seams first.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs inspection on current main: chat.send is still a plaintext-shaped Gateway RPC and no encrypted browser chat envelope/key-management fields exist.

## Solution Assessment

Unclear. The WebRTC/WebCrypto direction is plausible as a proposal, but the best maintainable path is security architecture and additive protocol/UI design first, not a narrow patch or ClawHub-only channel plugin.
