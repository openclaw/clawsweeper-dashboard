## Summary

Keep open: current main and v2026.6.6 still provide authenticated Gateway WebChat, Control UI chat, device pairing, DM session routing, and provider-backed Talk/WebRTC voice, but not browser-held E2EE chat envelopes, browser key negotiation, or per-device/per-agent encrypted browser chat channels.

## What This Changes

Request native browser-based E2EE chat built into OpenClaw, with no third-party chat app and per-agent/per-device encrypted channel isolation.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then add protocol, UI, key lifecycle, routing, transcript, relay, recovery, docs, and contract-test coverage under the chosen core or plugin boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source and release inspection give a high-confidence current-behavior check that the requested capability is absent.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is useful but not yet the best complete solution because it does not settle trust boundaries, key recovery, offline/relay behavior, transcript semantics, or plugin/core ownership.
