## Summary

Keep open: current main and v2026.6.8 still provide authenticated Gateway WebSocket WebChat, device auth/pairing, PWA/Web Push, DM session isolation settings, and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE browser chat transport with encrypted envelopes or per-device/per-agent browser keys.

## What This Changes

The issue proposes native browser-based E2EE chat for OpenClaw instances without requiring third-party chat apps, accounts, or installs.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then implement protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source and release inspection give a high-confidence current-behavior check that the requested capability is absent.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is a useful starting point, but it is not the best complete solution until maintainers settle the threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership.
