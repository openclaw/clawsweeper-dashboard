## Summary

Keep this issue open. Current main has browser Control UI/WebChat, WebCrypto-backed browser device identity, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but it still does not provide the requested browser-held E2EE chat transport, encrypted chat envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels. The existing maintainer/Codex comment also explicitly kept this open, and a current-main source pass matches that assessment.

## What This Changes

The issue requests a native browser-based E2EE chat layer for OpenClaw instances, without third-party chat apps, with per-device and per-agent encrypted channels.

## Best Possible Solution

Keep this as a maintainer-led product/security architecture item. The desired end state is an additive Gateway/browser chat design with encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, transcript/storage semantics, relay/fallback behavior, Control UI/WebChat UX, docs, and contract tests; if maintainers choose plugin-first delivery, define any missing Gateway/plugin seams before an external transport bypasses core auth or transcript policy.

## Reproduction Assessment

Not applicable as a feature request. The verification path is source/docs inspection against current main: the chat protocol and handlers still accept plaintext message strings, while docs describe Gateway WebSocket chat rather than browser-held E2EE transport.

## Solution Assessment

Unclear. The requested browser-native WebRTC/WebCrypto direction is plausible, but it is a starting architecture proposal rather than a proven narrow fix; the safer path is maintainer security design first, then a staged protocol/UI implementation.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because current browser chat is authenticated Gateway WebSocket chat where the Gateway sees message content, while the requested E2EE design would change encryption, routing, and storage trust boundaries.

Concerns:

  - confidence: 0.9

  - body: Current docs expose dmScope as a messaging-context boundary, not hostile multi-tenant isolation; an E2EE browser channel must not imply stronger per-agent or per-user isolation than Gateway auth, routing, and transcript storage actually enforce.
  - confidence: 0.82
