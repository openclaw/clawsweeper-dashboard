## Summary

Keep open: current main and the latest release still provide authenticated Gateway WebSocket WebChat, device/session tooling, and provider-backed Talk/WebRTC, but not the requested browser-held E2EE chat transport with encrypted message envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

The issue requests a native browser-based, end-to-end encrypted chat transport for OpenClaw with per-agent/per-device isolation and no third-party chat app.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, isolated routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture request, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. The direction addresses real first-run and trust-boundary concerns, but maintainers need to choose the protocol, key lifecycle, storage, routing, and relay architecture before code can be scoped.
