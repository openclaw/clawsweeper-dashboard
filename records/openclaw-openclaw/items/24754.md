## Summary

Keep open: current main and v2026.6.10 still provide authenticated Gateway WebSocket WebChat, device pairing, PWA/Web Push, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE browser chat transport with encrypted per-device/per-agent browser channels. Prior Codex/ClawSweeper comments reached the same conclusion, and current source plus live GitHub search did not find a canonical duplicate or implemented replacement.

## What This Changes

The issue requests a native browser-based end-to-end encrypted chat transport for OpenClaw instances, with no third-party chat provider and per-agent/device isolation.

## Best Possible Solution

Define a maintainer-approved encrypted browser-channel capability and threat model first, then implement protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source and release inspection confirm current OpenClaw lacks encrypted browser chat fields and still routes chat.send plaintext through the Gateway.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is a useful starting point, but it is not a complete solution until maintainers settle the threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership.
