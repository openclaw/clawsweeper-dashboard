## Summary

Keep this issue open for maintainer-led product and security architecture work. Current main and v2026.6.6 provide browser WebChat, device pairing, PWA/Web Push, configurable DM session isolation, and provider-backed WebRTC voice, but they still do not provide browser-held E2EE message envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels. The existing maintainer-side discussion reached the same conclusion, and no duplicate issue or implementation PR now owns the request.

## What This Changes

Proposes a native browser-based end-to-end encrypted chat transport with per-device and per-agent keys, browser pairing, and no third-party chat service.

## Best Possible Solution

Define the trust boundary and a generic encrypted-browser-channel capability first, then design an additive protocol covering per-device and per-agent key lifecycle, authenticated pairing, encrypted envelopes, routing, transcript semantics, offline and relay behavior, recovery, plugin integration, and contract tests before selecting a core or plugin implementation owner.

## Reproduction Assessment

Not applicable; this is a proposal for a new transport and security architecture rather than a failure of already-defined behavior. Source and release inspection do provide a high-confidence way to verify that the requested capability is currently absent.

## Solution Assessment

No, the WebRTC-first sketch is not yet the best complete solution because it does not define the trust boundary, offline delivery, relay fallback, key recovery, transcript semantics, or generic plugin capability. A threat model and protocol design should precede implementation.
