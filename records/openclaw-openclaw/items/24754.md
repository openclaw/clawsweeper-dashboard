## Summary

Keep open: current main and the latest release have authenticated Gateway WebSocket WebChat, browser/device pairing, PWA/Web Push, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but they still do not implement browser-held E2EE chat envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

Request a built-in browser-based E2EE chat transport with browser-held keys, QR pairing, no third-party chat app, and per-agent/per-device channel isolation.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then implement protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source, docs, and release inspection give a high-confidence current-behavior check that the requested capability is absent.

## Solution Assessment

No; the WebRTC/WebCrypto sketch is a useful starting point, but it is not the best complete solution until maintainers settle the threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership.
