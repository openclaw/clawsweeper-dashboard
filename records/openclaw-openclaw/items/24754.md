## Summary

Keep open: current main still provides authenticated Gateway/WebChat RPC and provider-backed Talk, but not the requested browser-held E2EE browser chat transport with encrypted message envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

Request native browser-based E2EE chat for OpenClaw instances without a third-party chat service, with isolated per-agent/per-device channels.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol, key lifecycle, UI, docs, and contract tests after maintainers settle routing, transcript visibility, and relay policy.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection of current main verifies browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. The requested direction could solve the user problem, but the maintainable path needs product/security design before a code change can be scoped.
