## Summary

Keep open: current main has authenticated Gateway WebSocket WebChat, Control UI chat, device identity, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but it still lacks the requested browser-held E2EE browser chat transport with encrypted envelopes, browser key negotiation, and per-device/per-agent encrypted browser channels.

## What This Changes

The issue proposes native browser-based end-to-end encrypted chat for OpenClaw, avoiding third-party chat providers while isolating per-agent/per-device conversations.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, isolated routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product and security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture request, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as a concrete implementation plan. The requested direction may solve real first-run and trust-boundary problems, but maintainers need to choose the product/security architecture before code can be scoped.
