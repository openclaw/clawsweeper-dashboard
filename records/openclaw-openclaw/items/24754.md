## Summary

Keep open: current main still provides authenticated Gateway/WebSocket browser chat with Gateway-visible message text, not the proposed browser-held E2EE browser chat transport with encrypted envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels.

## What This Changes

The issue proposes a native browser-based, browser-held E2EE chat transport for OpenClaw with per-device and per-agent encrypted channels, avoiding third-party chat providers.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol fields, browser-held key lifecycle, routing, transcript/storage semantics, relay policy, UI, docs, and contract tests after product/security review.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection confirms current browser chat is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as a concrete implementation plan. The requested direction may address the user problem, but the maintainable solution needs product and security architecture before code can be scoped.
