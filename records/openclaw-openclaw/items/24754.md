## Summary

Keep open: current main has authenticated Gateway WebSocket WebChat, Control UI, device identity, PWA/Web Push, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but not the requested browser-held E2EE chat transport or per-device/per-agent encrypted browser channels.

## What This Changes

The issue requests a native browser-based, no-third-party, end-to-end encrypted chat layer with QR pairing and per-agent/per-device channel isolation.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat design first, then implement protocol, UI, docs, and contract tests once maintainers settle key lifecycle, routing, transcript visibility, and relay policy.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection verifies the current browser chat path is authenticated Gateway RPC rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. The requested direction could solve the user problem, but it needs maintainer product/security design before a maintainable code change can be scoped.
