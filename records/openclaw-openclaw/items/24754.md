## Summary

Keep open: current main has Control UI/WebChat, device pairing, Web Push, DM session-scope hardening, and provider-backed Talk/WebRTC voice, but it still lacks browser-held E2EE chat envelopes, browser key negotiation, and per-device/per-agent encrypted browser chat channels. The recent maintainer/Codex discussion reached the same conclusion, and current source still matches it.

## What This Changes

The issue proposes a native browser-based E2EE chat layer for OpenClaw instances with QR pairing, no third-party chat provider, and per-agent/per-device channel isolation.

## Best Possible Solution

Keep this as a maintainer-led product/security architecture item covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, relay fallback, transcript semantics, Gateway protocol changes, Control UI UX, docs, and contract tests.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, not a bug report; source inspection verifies the current browser chat path is authenticated plaintext-shaped Gateway RPC rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. A native E2EE browser channel could solve the stated goal, but key lifecycle, relay/fallback behavior, transcript storage, and core/plugin boundaries need maintainer design before implementation.
