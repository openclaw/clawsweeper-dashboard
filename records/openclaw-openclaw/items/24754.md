## Summary

Keep open: current main still provides authenticated Gateway WebSocket WebChat and provider-backed browser Talk, but it does not implement browser-held E2EE chat, encrypted message envelopes, or per-device/per-agent encrypted browser channels. This remains a broad product/security architecture item, and recent issue discussion reached the same conclusion.

## What This Changes

Request native browser-based E2EE chat built into OpenClaw with no third-party app, no signup, and isolated per-device/per-agent encrypted channels.

## Best Possible Solution

Define an additive Gateway/browser E2EE chat architecture first, then implement protocol, UI, docs, and contract tests once maintainers settle key lifecycle, routing, transcript visibility, and relay policy.

## Reproduction Assessment

Not applicable. This is a feature/security architecture proposal, and source inspection verifies the current browser chat path is authenticated Gateway RPC with Gateway-visible message text rather than browser-held E2EE.

## Solution Assessment

Unclear as an implementation plan. The requested direction could solve the user problem, but the maintainable path needs product/security design before a code change can be scoped.
