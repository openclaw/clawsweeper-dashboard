## Summary

Keep open: current main and v2026.6.10 still provide authenticated Gateway WebSocket WebChat and browser Talk/WebRTC, but not browser-held E2EE chat with encrypted per-device/per-agent channels. The current chat contract still sends a plaintext string through the Gateway, so this remains product and security architecture work rather than conservative cleanup.

## What This Changes

Request a native browser-based E2EE chat layer with QR pairing and isolated per-device/per-agent encrypted channels, without requiring a third-party chat app.

## Best Possible Solution

Define a maintainer-approved encrypted-browser-channel capability and threat model first, then implement the protocol, UI, key lifecycle, routing, transcript, relay/recovery, docs, and contract tests under the chosen core or plugin boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new transport and security architecture proposal. Source and release inspection confirm current main lacks encrypted browser chat fields and still routes chat.send plaintext through the Gateway.

## Solution Assessment

No, not yet: the WebRTC/WebCrypto sketch is plausible, but the best solution needs an explicit threat model, key lifecycle, relay fallback, transcript semantics, and core/plugin ownership decision before implementation.
