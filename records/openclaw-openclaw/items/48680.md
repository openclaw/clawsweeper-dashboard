## Summary

Keep open. Current main still has a source-reproducible fallback gap for the reported provider business-denial shape: the fallback wrapper only fails a normal result when the result classifier returns a classification, and the current embedded classifier/CJK failover patterns still do not cover the reported `success:false` / `CE-011` / Chinese denial markers.

## What This Changes

The issue asks model fallback to keep trying remaining candidates when a provider returns a business rejection such as `success:false` / `CE-011`, instead of treating that attempt as `candidate_succeeded`.

## Best Possible Solution

Teach the embedded/failover classification boundary to recognize the reported CE-011 business-denial payload as failover-worthy, then verify the chain advances without weakening delivery and side-effect guards.

## Reproduction Assessment

Yes, source-reproducible with high confidence from current main and the issue logs. A non-GPT embedded result whose only signal is the reported unclassified `isError` text can still return `null` from the classifier, allowing the fallback attempt to be treated as success.

## Solution Assessment

Yes. The narrow maintainable fix is semantic provider-error classification for embedded results or shared failover patterns, not a broad retry of every surfaced error after fallback success is already committed.
