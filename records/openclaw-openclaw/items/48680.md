## Summary

Keep open. Current main still has the same failure shape: a nonthrowing embedded run with only an `isError:true` business-rejection payload can return a null fallback classification for non-GPT fallback models, after which the fallback loop records `candidate_succeeded` and stops.

## What This Changes

Treat provider business rejections such as HTTP-200 `success:false` / `CE-011` / model-access-denied payloads as failed model-fallback candidates so remaining fallback models are tried.

## Best Possible Solution

Land a focused failover classification fix with regression coverage for structured `success:false` / `CE-011` / `禁止访问` access-denial payloads, converting that semantic provider rejection into a failed fallback candidate before `candidate_succeeded` is observed while preserving the current side-effect and visible-reply safeguards.

## Reproduction Assessment

Yes. The issue provides concrete gateway logs, and current source gives a high-confidence code reproduction path: return a normal embedded result for a non-GPT fallback model with only an `isError:true` payload containing the reported business rejection text and no side effects; the classifier returns null and the fallback loop can mark the candidate succeeded.

## Solution Assessment

Yes. The narrowest maintainable fix is to extend the semantic failover classifier and embedded fallback regression tests for this provider-rejection shape; blanket retrying every error payload after a normal run would be riskier because it could duplicate replies or side effects.
