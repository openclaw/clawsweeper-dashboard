## Summary

Keep open. Current main still has the reported fallback path: an embedded run can return normally with only an unclassified `isError` payload, the result classifier returns no failure classification, and `runWithModelFallback` then records `candidate_succeeded` and stops the chain.

## What This Changes

Treat provider business rejections reported as agent `isError` payloads as failed fallback candidates so remaining fallback models are tried.

## Best Possible Solution

Add focused semantic-provider-error classification for error-only embedded results, with regression coverage for the reported `success:false` / `CE-011` / CJK denial payload while preserving visible-reply and side-effect guards.

## Reproduction Assessment

Yes, source-reproducible with high confidence from current main and the issue logs. A normal embedded result with only the reported unclassified `isError` payload can still return null from the classifier, allowing `candidate_succeeded` to be recorded.

## Solution Assessment

Yes. The narrow maintainable fix is in the embedded result or failover-pattern classification path, not a broad retry of every error payload after fallback success is already committed.
