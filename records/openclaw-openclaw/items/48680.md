## Summary

Keep open. Current main still has the reported source path: an embedded run that returns normally with only an unclassified error payload can pass the result classifier, after which model fallback records `candidate_succeeded` and stops.

## What This Changes

The issue asks OpenClaw model fallback to treat HTTP-200 semantic provider rejection payloads as failed fallback candidates so later fallback models are tried.

## Best Possible Solution

Add a focused semantic-provider-error classification path for error-only embedded results, with regression coverage for the reported `success:false` / `CE-011` / Chinese denial payload while preserving existing visible-reply and side-effect guards.

## Reproduction Assessment

Yes by source inspection and the issue logs. A normal embedded result with only the reported unclassified `isError` payload can still return null from the classifier, allowing `candidate_succeeded` to be recorded.

## Solution Assessment

Yes. The narrow maintainable fix is in the embedded result or failover-pattern classifier path, not in a broad fallback-loop retry of every error payload without the existing delivery, abort, and side-effect safeguards.

Fix openclaw/openclaw#48680. On current main, HTTP-200 semantic provider rejections can surface as normal embedded results with only an `isError` payload such as `{success:false, code:"CE-011", message:"当前ak因违规请求被禁止访问该模型..."}`; the embedded result classifier returns null, so `runWithModelFallback` records `candidate_succeeded` and skips remaining fallback models. Add a focused classifier/failover-pattern fix so this error-only structured/Chinese denial payload becomes a failed fallback candidate, while preserving guards for visible replies, direct/block replies, outbound delivery evidence, user aborts, deliberate silent replies, and tool side effects. Add regression coverage for the reported `success:false` / `CE-011` / `禁止访问` shape and keep neighboring OpenRouter 404, timeout, overloaded, billing, and incomplete-result tests passing. Because this is user-facing fallback behavior, add a concise `CHANGELOG.md` fix entry; do not change fallback configuration semantics, provider ownership, harness routing, or broad retry policy.
