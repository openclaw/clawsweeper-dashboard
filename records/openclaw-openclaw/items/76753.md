## Summary

Keep open: the PR targets a valid Discord liveness gap and is paired with the open related issue, but the latest diff still fires after Discord context work and bypasses the existing typing policy and delivery-target resolution, so it should not merge as-is.

## What This Changes

The PR adds a fire-and-forget Discord `sendTyping()` call after process-context creation and a changelog fix entry for Discord early typing parity with Telegram.

## Best Possible Solution

Implement a Discord-owned early typing path at the accepted-message boundary that respects effective typing mode and resolved delivery channel, with regression coverage for disabled typing, DMs, queue timing, and auto-thread guild targets.

## Reproduction Assessment

Yes from source inspection: current main queues accepted Discord messages and starts native typing only later through the reply-start pipeline, while the linked issue reports live 2-8s delays on slow pre-dispatch turns. I did not run a live Discord bot in this read-only sweep.

## Solution Assessment

No. The proposed call is too late for part of the named pre-dispatch work and bypasses the existing effective typing mode and resolved delivery target; the safer fix is to route an early cue through Discord-owned policy and target selection at the accepted boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move the cue before context work:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: The new call runs only after `await buildDiscordMessageProcessContext(...)`, but that helper already reads session state, builds pending guild history, may fetch thread starter context, and resolves auto-thread reply plans. Slow work there can still leave users with no native typing cue until this line runs, so the early cue needs to fire at an earlier accepted-message boundary or be limited to cases where that boundary is correct.
  - confidence: 0.87
- **[P2] Preserve typing policy and resolved target:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: This unconditional call bypasses the reply pipeline's typing policy and target resolution: `typingMode: "never"` or suppressed delivery can still hit Discord's typing endpoint, and auto-threaded guild replies can type in `messageChannelId` instead of the resolved delivery thread. Route the early cue through the same effective policy and channel selection used by the normal typing path.
  - confidence: 0.88
