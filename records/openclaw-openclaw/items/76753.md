## Summary

Keep open: current main still has the Discord typing gap, but this PR is not merge-ready because the new cue fires after context assembly and bypasses the existing typing policy and resolved delivery target.

## What This Changes

The PR adds a fire-and-forget Discord `sendTyping()` call after process context creation and adds a changelog fix entry for the Discord early typing issue.

## Best Possible Solution

Route the early Discord liveness cue through a small Discord-owned helper or controller path that respects effective typing mode and preserves the same delivery-channel semantics, with focused regression coverage for disabled typing, DMs, and auto-thread guild targets.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear timing path: current main starts Discord typing from the reply pipeline after process context and dispatch setup, and the related issue reports live DM delays; I did not run a live Discord timing check.

## Solution Assessment

Is this the best way to solve the issue? No. The proposed call is too late for part of the named pre-dispatch work and bypasses the existing typing policy and delivery-target resolution; the safer fix is to share or extend the existing typing path for the earliest correct boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move the cue before context work:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: The new call runs only after `await buildDiscordMessageProcessContext(...)`, but that helper already does the session/history/context and auto-thread planning work called out in the bug. Slow context assembly can still leave users with no native typing cue until this line runs, so the early cue needs to fire at an earlier accepted-message boundary or be limited to cases where that boundary is correct.
  - confidence: 0.86
- **[P2] Preserve typing policy and target semantics:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: This unconditional call bypasses the reply pipeline's typing policy and target resolution: `typingMode: "never"` can still hit Discord's typing endpoint, and auto-threaded guild replies can type in `messageChannelId` instead of the resolved delivery thread. Please route the early cue through the same effective policy and channel selection used by the normal typing path.
  - confidence: 0.88
