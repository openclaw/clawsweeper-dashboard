## Summary

Keep this PR open: the latest diff still adds the Discord typing cue after context assembly and outside the existing typing policy and delivery-target semantics, so it does not fully fix the reported timing gap and introduces regressions.

## What This Changes

The PR adds a fire-and-forget Discord `sendTyping()` call after message process-context creation and adds a changelog entry for #76417.

## Best Possible Solution

Implement the Discord liveness cue at the earliest accepted-message boundary for eligible conversations while preserving the same effective typing policy and resolved delivery target semantics as the normal reply pipeline.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, yes: current main starts Discord typing through the reply pipeline after process context and run setup, while #76417 reports the visible delay on slow DM turns.

## Solution Assessment

Is this the best way to solve the issue? No. The proposed direct call is too late for some named pre-dispatch work and bypasses existing typing-mode and target-resolution behavior; the safer fix is to route or gate the cue through those semantics.

- **[P2] Move the cue before context assembly:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: The new cue fires after `await buildDiscordMessageProcessContext(...)`, but that helper already builds history context, may fetch thread starter context, and awaits auto-thread planning.   - confidence: 0.87

  - body: This direct `sendTyping()` call is outside the reply pipeline's typing-mode resolution. A session configured with `typingMode: "never"` or otherwise suppressed typing would still hit Discord's typing endpoint, violating the documented no-typing mode.
  - confidence: 0.88
- **[P2] Use the resolved typing target for guild threads:** `extensions/discord/src/monitor/message-handler.process.ts:240`
  - body: The new call always uses `messageChannelId`, while the existing path derives the typing channel from `deliverTarget`. In auto-threaded guild flows the reply target can be a created thread, so this cue can type in the parent channel instead of the actual reply thread.
  - confidence: 0.86
