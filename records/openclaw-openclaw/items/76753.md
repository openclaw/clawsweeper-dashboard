## Summary

Keep open: this PR targets a real Discord liveness gap tracked by #76417, but the current diff fires after Discord process-context assembly and bypasses existing typing policy/target resolution, so it needs revision before merge.

## What This Changes

The branch adds a fire-and-forget Discord `sendTyping()` call after process context creation and adds a changelog fix entry for #76417.

## Best Possible Solution

Keep the Discord liveness improvement, but route the early cue through the same effective typing policy and delivery-channel semantics, with focused regression coverage for disabled typing, DMs, and auto-thread/guild targets.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear reproduction path for the timing class: current main starts Discord typing from the reply pipeline after process-context and dispatch setup, though no live Discord timing run was performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The proposed call is too late for context-assembly delays and is outside the existing typing policy and target-resolution path; the safer fix is a small shared helper or early controller path that preserves those semantics.

## Review Findings

Please route the early cue through the same effective policy and channel selection used by the normal typing path.
