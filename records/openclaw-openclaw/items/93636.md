## Summary

Keep open: current main and the latest release still have deleted-cwd crash paths, and this branch fixes several of them, but it is not merge-ready because the required-home fallback can redirect mutable OpenClaw state under the runtime executable directory and the external proof is still mock-only.

## What This Changes

The PR adds nullable `process.cwd()` handling across dotenv loading, PATH bootstrap, required-home resolution, is-main checks, and TUI runtime/local-shell paths, with regression tests and CI shard mapping.

## Best Possible Solution

Land the caller-specific nullable-cwd repair only after the no-home/no-cwd state-root path fails clearly or uses an explicit configured state root, with real deleted-cwd CLI/TUI proof.

## Reproduction Assessment

Yes, source-level. The linked report gives concrete deleted-cwd steps, and current main plus `v2026.6.9` still contain bare `process.cwd()` calls in CLI/TUI startup and local-shell paths; I did not run a live deleted-cwd TUI session in this read-only review.

## Solution Assessment

No, not yet. Caller-specific nullable cwd handling is the right direction, but the executable-directory required-home fallback should be replaced with a clear failure or explicit configured state root before merge.
