## Summary

Keep open. The nullable-cwd rework now covers the important CLI/TUI sibling paths, but the required-home fallback still redirects state/config paths to the runtime executable directory in a no-home/no-cwd edge case, and the external proof remains a stubbed `process.cwd()` probe rather than a real deleted-cwd CLI/TUI run.

## What This Changes

The PR adds caller-specific deleted-cwd handling for CLI dotenv loading, gateway-dispatch dotenv loading, runtime dotenv loading, PATH bootstrap, required-home fallback, is-main detection, TUI startup/auth/autocomplete, TUI local shell, tests, and CI test-plan sharding.

## Best Possible Solution

Land one caller-specific nullable-cwd fix that preserves dotenv/PATH/TUI trust semantics, refuses operator local-shell commands when cwd is gone, and fails clearly or uses an explicitly approved state base when no home source and no cwd exist.

## Reproduction Assessment

Yes, source-level. The linked issue gives concrete macOS deleted-cwd steps, and current main plus v2026.6.9 still have bare `process.cwd()` use in the reported CLI/TUI paths; I did not run a live deleted-cwd CLI/TUI smoke in this read-only review.

## Solution Assessment

No, not yet. The caller-specific nullable-cwd shape is the right direction for dotenv, PATH, TUI runtime, and local shell, but the required-home executable-directory fallback should be replaced or explicitly accepted by maintainers before merge.
