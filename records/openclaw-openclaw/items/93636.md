## Summary

Keep open: the branch is a plausible repair for the deleted-cwd crash, but it still has a concrete merge blocker in the no-home/no-cwd fallback and the external proof remains a stubbed `process.cwd()` probe rather than an actual deleted-cwd CLI/TUI run.

## What This Changes

The branch adds nullable cwd handling across CLI dotenv loading, PATH bootstrap, home-dir resolution, is-main, and TUI/local shell paths, with focused regression tests and test-plan routing.

## Best Possible Solution

Keep the caller-specific nullable-cwd approach, but replace the executable-directory state-home fallback with a clear failure or maintainer-approved explicit state-root behavior, then provide real deleted-cwd CLI/TUI proof.

## Reproduction Assessment

Yes, from source: current main still calls `process.cwd()` in required-home and TUI paths that the linked issue exercises, so a deleted cwd can throw `uv_cwd`. I did not run a live deleted-directory CLI/TUI repro in this read-only review.

## Solution Assessment

No, not yet: nullable cwd handling and local-shell refusal are the right shape, but using the executable directory as a state-home fallback is not the safest maintainable solution. A clear error or explicit configured state root is safer for persisted state paths.
