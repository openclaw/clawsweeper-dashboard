## Summary

Keep open: the report is fresh, source-backed, and current main still depends on each Feishu account's `message.mentions` payload for both group mention admission and mention-forward classification, with no exact merged fix or safe close target found.

## What This Changes

The issue reports that Feishu multi-account group messages that mention several bot accounts can omit the receiving bot's own mention from that account's event, causing OpenClaw to treat the message as a forward request instead of a direct mention.

## Best Possible Solution

Fix the Feishu plugin to build a canonical same-message mention view across multi-account deliveries or another provider-backed identity fallback, then cover the three-bot group scenario with focused regression tests.

## Reproduction Assessment

No live high-confidence reproduction is available from this review. The reporter provided concrete steps and logs, and source inspection shows a plausible failing path whenever Feishu omits the receiving account's self-mention from `event.message.mentions`.

## Solution Assessment

Unclear: a parser-only change may not recover a self-mention that Feishu removes from the receiving account's payload. The safer direction is an account-aware Feishu fix with tests that preserve existing false-positive guards.
