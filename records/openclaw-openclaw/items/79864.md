## Summary

Keep open: this is a plausible iMessage bug fix, but the draft PR currently introduces a local-file media bypass and still lacks real after-fix delivery proof for an imsg build that supports `send-rich --file`.

## What This Changes

The PR adds capability-probed iMessage reply attachments by detecting `imsg send-rich --file`, passing path or buffer attachments into `sendRichMessage`, and adding focused action tests.

## Best Possible Solution

Reuse the existing outbound attachment hydration/resolver for reply attachments, wait for or use a verified imsg build with `send-rich --file`, then land with redacted live macOS proof of a threaded attachment delivery.

## Reproduction Assessment

Yes, by source inspection: current main's iMessage `reply` action never consumes `media`, `filePath`, `path`, or `buffer`, so attachment-bearing replies cannot deliver an attachment through that path. I did not run a live macOS reproduction during this read-only review.

## Solution Assessment

No. Capability probing is a reasonable direction, but this implementation should route attachments through OpenClaw's existing media resolver/local-root checks, and the successful supported-imsg path still needs real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Run reply attachments through the media resolver:** `extensions/imessage/src/actions.ts:278`
  - body: This returns `media`/`path`/`filePath` values as trusted local paths and later passes them directly to `imsg --file`. The existing send/upload-file paths hydrate attachments through the media resolver so local-root, sandbox, and size checks run first; bypassing that lets an agent send any readable host file if it can name the path. Please reuse the existing attachment hydration/resolver and pass the resulting staged buffer/path into the reply flow.
  - confidence: 0.88
