## Summary

Keep this PR open: the bug is real and the feature-gated shape is plausible, but the current patch bypasses OpenClaw's validated outbound media path for reply attachments and lacks real supported-imsg delivery proof.

## What This Changes

The PR adds iMessage reply attachment plumbing through `imsg send-rich --file`, gated by a probed CLI capability, with fallback errors and new action tests.

## Best Possible Solution

Keep the feature gate, but validate and stage reply media through the existing outbound media access path before passing a file to imsg, then land with real supported-imsg delivery proof.

## Reproduction Assessment

Yes, source inspection and the PR's live trace give a high-confidence path: current main's reply action ignores attachment params and sends only text through `sendRichMessage`. I did not run a live macOS/iMessage reproduction in this read-only review.

## Solution Assessment

No, not as written. Feature-detecting `send-rich --file` is the right dependency boundary, but the patch should reuse validated media loading/staging and provide supported-imsg real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate reply attachment paths before sending:** `extensions/imessage/src/actions.ts:280-289`
  - body: This turns model-controlled `media`/`filePath`/`mediaUrls` strings directly into `--file` paths for imsg. Normal iMessage media sends first go through the outbound media loader, which enforces local roots, size limits, and host-read MIME checks; this path bypasses those guards and can send arbitrary readable host files. Please resolve/stage these inputs through the existing media access path before passing a file path to the runtime.
  - confidence: 0.88
- **[P2] Refresh cached CLI capability after imsg upgrades:** `extensions/imessage/src/probe.ts:218-232`
  - body: When a working bridge is probed before `send-rich --file` is installed, the new `sendRichSupportsAttachment: false` value is stored in the normal private-API status, and available statuses are cached with expiry `0`. A running gateway will keep rejecting reply attachments after upgrading imsg until restart/cache clear, so the capability needs its own TTL or force-refresh path.
  - confidence: 0.77
