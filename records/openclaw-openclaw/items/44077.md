## Summary

Keep open. The current sandbox mutation helper still creates replacement temp files with mode `0o600` and atomically replaces the target without capturing or reapplying the existing file mode, while sandboxed Write/Edit route through that bridge. The discussion’s narrowed root cause still matches current main; #20979 fixed a different UID/default-user problem, #44114 was closed unmerged, and the open #44260 appears adjacent rather than proven to fix this sandbox bridge path.

## What This Changes

Fix sandboxed file-tool Write/Edit so editing workspace files preserves sane existing permissions instead of rewriting files to owner-only mode and causing host-side EACCES.

## Best Possible Solution

Add a dedicated sandbox bridge fix that captures the existing target mode before replacement and applies it to the temp file before `os.replace()`, with a clear policy for new files. Cover the 0644-existing-file edit case in `fs-bridge-mutation-helper.test.ts`, keep the existing path-safety tests passing, and coordinate separately with #44260 if maintainers also want to change default modes for newly created user-facing files.
