## Summary

Keep open. Current main still routes sandbox Write/Edit/apply_patch writes through the sandbox mutation helper, whose atomic write path creates a 0600 temp file and replaces the target without preserving the existing file mode. The closed attempted fix was not merged, and the open related PR only changes host-side Write/Edit defaults, not the sandbox bridge path.

## What This Changes

The issue asks for sandbox file-tool edits on mounted workspaces to preserve sane file permissions instead of rewriting edited files to restrictive 0600 modes.

## Best Possible Solution

Add a focused sandbox mutation-helper fix that captures the existing target file mode before replacement, applies that mode to the temp file before fsync/replace, and covers both direct sandbox Write/Edit and sandbox apply_patch with regression tests. New-file default mode should be chosen deliberately and aligned with the host Write/Edit default-mode discussion without changing sensitive internal host write paths.

## Reproduction Assessment

Yes. The issue body gives concrete Docker sandbox steps, and current source independently shows a deterministic 0600 temp-file replace path; a focused helper test can reproduce 0644-to-0600 collapse without needing a full live Docker run.

## Solution Assessment

Yes, a narrow fix in src/agents/sandbox/fs-bridge-mutation-helper.ts is the best path. Changing only host fs-safe defaults, as in the related open PR, does not cover sandbox bridge writes or sandboxed apply_patch.
