## Summary

Keep open for normal maintainer review and exact-head validation. Current main still sends Discord REST FormData bodies directly to fetch except for the proxy-specific undici wrapper, while this PR applies the multipart boundary and Content-Type normalization at the REST client boundary with focused regression coverage and a changelog entry.

## What This Changes

The PR moves Discord REST FormData upload normalization into RequestClient, removes the proxy-specific undici FormData wrapper, adds multipart Content-Type regression tests, and records the Discord attachment upload fix in CHANGELOG.md.

## Best Possible Solution

Land the REST-level multipart normalization after maintainer review and exact-head validation, keeping the behavior scoped to Discord request serialization and retaining the new regression tests plus changelog entry.

## Reproduction Assessment

Yes. Current main shows attachment sends becoming FormData and reaching fetch without guaranteed REST-level multipart Content-Type normalization outside the proxy wrapper, and the PR body provides a live CONTENT_TYPE_INVALID failure plus focused fetch-level tests.

## Solution Assessment

Yes. Centralizing normalization in RequestClient is the narrowest maintainable fix because it covers direct, custom, and undici-backed Discord REST fetch paths without changing the send API, media loading policy, permissions, retries, or proxy configuration surface.
