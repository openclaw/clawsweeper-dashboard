## Summary

## What This Changes

The PR centralizes Discord REST `FormData` normalization into an explicit multipart body with a generated boundary, removes the proxy-specific undici `FormData` wrapper, and adds REST tests for custom fetch and undici fetch content types.

## Best Possible Solution

Land the REST-boundary multipart normalization after adding the required changelog entry and confirming the targeted Discord REST/proxy/send tests on the PR head.

## Reproduction Assessment

Yes. The PR body gives a concrete live Discord attachment failure, and current main shows a reproducible seam where `FormData` reaches custom/undici fetch without a guaranteed multipart Content-Type; the added tests exercise that seam directly.

## Solution Assessment

Yes, aside from the missing changelog. Centralizing multipart normalization in `RequestClient` is narrower and more maintainable than keeping a proxy-only wrapper because it covers both direct custom fetch and proxy fetch paths without changing Discord send semantics.
