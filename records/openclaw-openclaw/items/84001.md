## Summary

Keep open: current main and v2026.6.8 still route status service summaries through Windows service probes without the user-supplied timeout, and the directly relevant Windows schtasks/status fix PR was closed unmerged.

## What This Changes

The issue asks for Windows `openclaw status`, `status --json`, and JSON full-status paths to return or fail within the requested timeout instead of hanging while the diagnostic status path succeeds.

## Best Possible Solution

Thread the existing CLI timeout through status service summaries and Windows gateway/node service probes, keep the Startup-folder fallback behavior, and prove the four reported Windows status commands on a native scheduled-task install.

## Reproduction Assessment

No live current-main Windows repro was run. Source inspection shows the status service-summary path can still outlive the requested timeout because service probes ignore the CLI timeout and Windows schtasks uses fixed 15s/30s budgets.

## Solution Assessment

Yes, the requested behavior is the right fix boundary. The maintainable path is to revive or replace the narrow timeout-forwarding and task-scoped schtasks fix, then verify it on native Windows rather than closing this as already fixed.
