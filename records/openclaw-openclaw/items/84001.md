## What This Changes

The issue asks for `openclaw status` and `status --json --timeout 3000` on Windows to return or fail within the requested timeout instead of hanging while `status --all` succeeds.

## Best Possible Solution

Thread the existing CLI timeout through status service summaries and Windows gateway/node service probes, preserve Startup-folder fallback behavior, and prove the reported status commands on a native Scheduled Task install.

## Reproduction Assessment

No live current-main Windows reproduction was run. Source inspection shows the status timeout still stops at the service-summary boundary while Windows schtasks reads use fixed internal budgets, so the reported timeout contract remains source-reproducible.

## Solution Assessment
