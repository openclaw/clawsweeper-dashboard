## Summary

Keep open: current main still lacks the persistent structured taint/provenance propagation and risk-aware enforcement requested here; existing external-content markers, local web_fetch metadata, hooks, and exec approvals are useful building blocks, not the full security model.

## What This Changes

The issue requests a unified security hardening model with persistent external-content tainting, default denial of content-derived execution, exec risk classes, and provenance-aware secret and outbound-action guardrails.

## Best Possible Solution

Keep this as the core security architecture umbrella unless maintainers explicitly split it into canonical implementation issues with a shared contract for taint propagation and risk-aware enforcement.

## Reproduction Assessment

Yes for the source-level gap: current main shows marker-wrapped external content and narrow local metadata, while downstream hooks, approval payloads, persistence paths, and outbound events lack persistent taint and risk context. I did not need a full exploit run to verify the requested architecture is still absent.

## Solution Assessment

Yes, keeping this open for maintainer security design is the best current path. A narrow automated patch to one hook would not cover the required compatibility and enforcement work across plugins, approvals, transcripts, secrets, and outbound delivery.
