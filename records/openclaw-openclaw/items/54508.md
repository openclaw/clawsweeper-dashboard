## Summary

Keep open: current main and the latest shipped tag still have external-content wrappers, local web_fetch metadata, narrow inter-session provenance, generic tool-policy hooks, and exec approval analysis, but not the requested persistent taint model or provenance-aware enforcement across exec, secrets, persistence, and outbound actions.

## What This Changes

Request persistent taint/provenance tracking for external content plus default guardrails for content-derived execution, secret access, and outbound actions.

## Best Possible Solution

Keep this as the security architecture umbrella until maintainers land a coordinated provenance/taint contract or explicitly split it into canonical implementation issues that cover propagation and enforcement end to end.

## Reproduction Assessment

Yes for the source-level gap: current main and v2026.5.12 show marker wrapping, local metadata, and narrow input provenance, while the hook, approval, and outbound contracts lack persistent taint and risk context. I did not run a live exploit because the requested architecture is absent from source.

## Solution Assessment

Yes at the triage level: keeping this open for maintainer security design is the best current path. A narrow automated patch to one hook or approval type would not satisfy the cross-contract enforcement model requested here.
