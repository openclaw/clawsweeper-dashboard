## Summary

Keep open. Current main still has marker-based external-content wrapping, local web_fetch metadata, narrow user/inter-session provenance, hook blocking, exec approvals, and outbound hooks, but it does not implement the requested persistent taint propagation or provenance/risk-aware enforcement for execution, secrets, persistence, and outbound delivery. The related open issues are useful slices, not a clear canonical replacement for this cross-cutting security umbrella.

## What This Changes

The issue requests an end-to-end security model with persistent external-content tainting, default denial of content-derived execution, exec risk classes, and provenance-aware guardrails for secret access and outbound actions.

## Best Possible Solution

Keep this as the core security architecture umbrella unless maintainers intentionally split it into canonical implementation issues. The desired end state is structured provenance/taint metadata for external inputs and tool outputs, propagation through derived content and persistence, additive risk/provenance fields in hook and approval payloads, and core enforcement that prevents untrusted content from authorizing execution, secret access, or outbound actions without guarded review.

## Reproduction Assessment

Yes for the code-level gap: current-main source inspection shows external content becomes marker-wrapped text plus local payload metadata, while downstream hook, approval, replay, persistence, and outbound schemas lack the fields needed for persistent taint and risk-aware enforcement. A live exploit reproduction is not needed to verify that the requested architecture is still absent.

## Solution Assessment

Yes, keeping this open for maintainer security design is the best current path. A narrow automated patch would be risky because the maintainable solution needs additive contract design and compatibility review across plugins, approvals, transcripts, secrets, and outbound delivery.
