## Summary

Keep open: current main still has external-content text wrappers, local web_fetch metadata, narrow inter-session provenance, and generic approval hooks, but not the requested persistent taint model or risk-aware enforcement across execution, secrets, persistence, and outbound actions.

## What This Changes

The issue requests a security architecture for persistent external-content taint/provenance, default content-derived execution guardrails, exec risk classes, and provenance-aware secret/outbound action gating.

## Best Possible Solution

Keep this as the core security umbrella until maintainers either land a coordinated CaMeL/provenance design or explicitly split it into canonical implementation issues with a shared contract.

## Reproduction Assessment

Yes for the source-level gap: current main shows marker wrapping and narrow provenance fields, while the hook, approval, persistence, and outbound contracts lack persistent taint and risk context. I did not run a live exploit because the requested architecture is absent from source.

## Solution Assessment

Yes, keeping this open for maintainer security design is the best current path. A narrow automated patch to one hook would not cover the cross-contract enforcement model requested here.
