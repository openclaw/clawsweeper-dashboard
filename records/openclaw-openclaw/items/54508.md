## Summary

Keep open: current main and the latest release still provide marker wrapping, local tool-result metadata, narrow input provenance, hook blocking, exec approvals, and unsafe-content audit flags, but not the requested persistent taint model or provenance-aware enforcement across execution, secrets, persistence, and outbound actions.

## What This Changes

The issue requests persistent external-content taint tracking, default denial of content-derived execution, exec risk classes, and provenance-aware secret/outbound-action guardrails.

## Best Possible Solution

Keep this as the security architecture umbrella until maintainers either land a coordinated provenance/taint contract or explicitly split it into canonical implementation issues that cover propagation and enforcement end to end.

## Reproduction Assessment

Yes for the source-level gap: current main and v2026.5.18 show wrappers, local metadata, and narrow message provenance, while approval, hook, persistence, and outbound contracts lack persistent taint/risk context. I did not run a live exploit because this is an absent cross-cutting architecture rather than a single failing command path.

## Solution Assessment

Yes at the triage level: keeping this open for maintainer security design is the best current path. A narrow automated patch to one hook or approval type would not satisfy the requested end-to-end propagation and enforcement model.
