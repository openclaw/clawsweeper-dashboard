## Summary

Keep open. Current main still has marker-based external-content wrapping, immediate web_fetch metadata, narrow message provenance, hook blocking, exec approvals, and outbound hooks, but it does not implement persistent taint propagation or provenance/risk-aware enforcement across execution, secrets, persistence, and outbound delivery. The linked related items are useful slices, not a clear canonical replacement for this security umbrella.

## What This Changes

The issue requests a core security model for persistent external-content tainting, default denial of content-derived execution, exec risk classes, and provenance-aware guardrails for secrets and outbound actions.

## Best Possible Solution

Keep this as the core security architecture umbrella unless maintainers intentionally split it into canonical implementation issues. The target end state is structured provenance/taint metadata for external inputs and tool outputs, propagation through summaries and persistence, additive risk/provenance fields in hook and approval payloads, and core enforcement that blocks untrusted content from authorizing execution, secret access, or outbound actions without guarded review.

## Reproduction Assessment

Yes for the code-level gap: current-main source inspection shows external content becomes marker-wrapped text plus local payload metadata, while downstream hook, approval, replay, and outbound schemas lack the fields needed to reproduce persistent taint or risk-aware enforcement. A full exploit reproduction was not needed to verify the requested architecture is still absent.

## Solution Assessment

Yes, keeping this open for maintainer/security design is the best current outcome. A narrow automated patch would be risky because the maintainable solution needs additive contract design and compatibility review across plugins, approvals, transcript persistence, secrets, and outbound delivery.
