## Summary

Keep open. Current main still resolves Docker sandbox `pidsLimit` only from agent/global config and only emits Docker `--pids-limit` when that value is explicitly positive; the latest shipped tag has the same behavior. PR #44098 is still open and directly proposes the default-limit fix, so this issue should remain open until that PR or an equivalent maintainer-approved security fix lands.

## What This Changes

Request to add a default Docker sandbox `pidsLimit` and align docs/security guidance so sandbox containers receive a process cap when no agent or global limit is configured.

## Best Possible Solution

Land PR #44098 or an equivalent security-owned fix that adds a maintainer-approved default in `resolveSandboxDockerConfig`, preserves explicit global and per-agent overrides, proves `buildSandboxCreateArgs` emits the resolved default, and updates public sandbox/config/security docs if runtime behavior changes. Close this issue after that fix merges and ships.

## Reproduction Assessment

Yes. The source-level reproduction is direct: resolve Docker sandbox config with no agent/global `pidsLimit`, then build Docker create args; current main leaves `pidsLimit` undefined and therefore emits no `--pids-limit` flag.

## Solution Assessment

Yes, with maintainer approval of the value. Adding the default in the resolver is the narrowest maintainable fix because Docker arg generation already honors a positive resolved value; the WASM backend idea from the discussion is broader optional architecture, not a replacement for hardening the current Docker backend.
