## Summary

Keep open: current main still leaves unset sandbox Docker `pidsLimit` unresolved, and open PR #44098 directly tracks the fix with closing syntax. The issue should stay open until that PR, or an equivalent replacement, lands.

## What This Changes

The issue requests a default Docker sandbox `pidsLimit` plus clearer guidance for the security risk of leaving PID limits unset.

## Best Possible Solution

Land a maintained sandbox resource-default change that sets an agreed PID cap, preserves explicit overrides, defines opt-out semantics, updates docs/schema/tests/changelog, and coordinates broader CPU/memory defaults with #60827.

## Reproduction Assessment

Yes. Source inspection on current main is enough: unset `pidsLimit` resolves to `undefined`, and Docker create args omit `--pids-limit` unless the value is positive.

## Solution Assessment

Yes in implementation boundary, but not complete as a landed solution yet. The narrow resolver fallback belongs in the sandbox config path, with maintainer-approved default/opt-out semantics and public contract updates before merge.
