## Summary

Keep open: current main still leaves Docker sandbox `pidsLimit` unset by default, and the linked implementation PR that closes this issue is still open and unmerged. The WASM-backend comment is useful broader hardening context, but it does not resolve the Docker PID-cap default requested here.

## What This Changes

The issue requests a default Docker sandbox `pidsLimit` so containers receive a PID cap when no agent or global value is configured.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44098 or a narrow replacement that adds an approved PID cap, preserves explicit overrides, defines opt-out behavior, and updates schema, docs, tests, and changelog while leaving broader CPU/memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass that config to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`.

## Solution Assessment

Unclear until maintainer approval. The resolver fallback is the narrow implementation point, but the default value and Docker-default/unlimited opt-out contract need schema, docs, tests, and security-owner review.
