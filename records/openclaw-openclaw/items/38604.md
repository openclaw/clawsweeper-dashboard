## Summary

Keep open: current main and v2026.5.7 still leave Docker sandbox containers without a default PID cap, while the implementation PR that references this issue remains open and unmerged. The WASM-backend discussion is useful broader security context, but it does not remove the current Docker hardening gap.

## What This Changes

Add a default Docker sandbox `pidsLimit` so containers get a PID cap when no global or agent value is configured.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44098 or a narrow replacement that adds an approved PID cap, preserves explicit override and opt-out behavior, and updates schema, docs, tests, and changelog while leaving broader CPU/memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes. A safe source-level reproduction is to resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass that config to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`.

## Solution Assessment

Unclear as submitted. The resolver fallback is the narrow implementation point, but the default value and opt-out contract need maintainer approval plus schema/docs/test follow-through.
