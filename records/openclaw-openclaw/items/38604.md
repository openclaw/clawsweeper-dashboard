## Summary

Keep open: current main still resolves Docker sandbox `pidsLimit` only from agent/global config and omits `--pids-limit` when unset, while the open PR that closes this issue remains unmerged. The WASM-backend comment is useful broader hardening context, but it does not remove the current Docker sandbox PID-cap question.

## What This Changes

Request a default Docker sandbox `pidsLimit` so fork-bomb-style process spawning cannot run without an OpenClaw-enforced PID cap.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44098 or a narrow replacement that adds an approved PID cap, preserves explicit overrides, defines the opt-out behavior, and updates schema, docs, tests, and changelog while leaving broader CPU/memory defaults to https://github.com/openclaw/openclaw/issues/60827.

## Reproduction Assessment

Yes. A safe source-level reproduction is to resolve Docker sandbox config with no global or per-agent `pidsLimit`, then pass that config to `buildSandboxCreateArgs`; current main leaves the value undefined and emits no `--pids-limit`.

## Solution Assessment

Unclear as submitted. The resolver fallback is the narrow implementation point, but the default value and the Docker-default/unlimited opt-out contract need maintainer approval plus schema, docs, and tests.
