## Summary

Keep open. Current main and the latest shipped tag still leave Docker sandbox `pidsLimit` unset unless explicitly configured, and open PR #44098 directly references this issue with the proposed default-limit fix.

## What This Changes

Add a default Docker sandbox `pidsLimit` so sandbox containers get a process-count cap when neither global nor per-agent config sets one.

## Best Possible Solution

Land PR #44098 or an equivalent secops-reviewed change that adds the default in the resolver, preserves explicit overrides, proves Docker args include the resolved default, and updates docs if runtime behavior changes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve Docker sandbox config without global or per-agent `pidsLimit`, then build Docker create args; current main leaves the value undefined and emits no `--pids-limit` flag.

## Solution Assessment

Yes, pending maintainer approval of the value. Adding the fallback in `resolveSandboxDockerConfig` is the narrowest maintainable fix; the WASM-runtime idea raised in discussion is broader architecture work and does not replace hardening the current Docker backend.
