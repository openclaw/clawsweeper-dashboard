## Summary

Keep this PR open. Current main still lacks the requested PathGuard filesystem ACL capability, and the PR is the active implementation candidate, but the proposed patch is not safe to merge as-is because it leaves concrete filesystem-policy bypasses and generated config-schema drift.

## What This Changes

The PR adds a PathGuard filesystem policy layer with `tools.fs.allowedPaths` / `denyPaths`, structured policy denial results, media and apply_patch checks, and spawned-subagent filesystem policy ceilings.

## Best Possible Solution

Keep the PR open as source work, but require a current-main rebase and focused maintainer security review before landing or replacing it. The end state should be a coherent, generated-schema-backed filesystem ACL contract with deny-wins canonical PathGuard checks applied consistently to read/write/edit/apply_patch, local media, prompt-inline images, and spawned-subagent runs.

## Reproduction Assessment

Yes. The missing main feature is reproducible by inspecting current `ToolFsPolicy` and config schema, and the PR regressions have code-level reproductions: a denied prompt-inline image path is not checked against `allowedPaths`/`denyPaths`, and a relative deny glob over a symlinked directory can fail after realpath canonicalization.

## Solution Assessment

No. The product direction is plausible, but the current PR is not the best mergeable solution until it closes the remaining policy bypasses, regenerates config schema outputs, and rebases onto current main without dropping existing run-context behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Enforce PathGuard for prompt image loads:** `src/agents/pi-embedded-runner/run/attempt.ts:466`
  - body: The new `toolFsPolicy` is only passed into tool construction, but prompt-inline image loading still receives just `workspaceOnly`. A vision prompt can reference a local image path that should be blocked by `allowedPaths` or `denyPaths`, and `loadImageFromRef` will load it without the PathGuard check. Thread the effective `ToolFsPolicy` through `detectAndLoadPromptImages` / `loadImageFromRef` before calling `loadWebMedia`.
  - confidence: 0.91
- **[P1] Apply deny globs to symlinked relative paths:** `src/security/path-guard.ts:271-275`
  - body: Relative glob entries return `false` as soon as the canonical target is outside the workspace. That is correct for allow rules, but for deny rules it means `workspaceOnly=false` with `denyPaths: ["vendor/**"]` can miss `vendor` when it is a symlink to an external directory, so the deny rule is skipped after realpath resolution. Deny globs need to be evaluated against the canonicalized relative entry target instead of bailing out here.
  - confidence: 0.88
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:458-459`
  - body: The runtime schema now accepts `allowedPaths`, `denyPaths`, `tools.subagents.fs`, and `tools.sessions_spawn.fsPolicy`, but the generated base schema still exposes only `tools.fs.workspaceOnly`. This leaves `openclaw config schema`, UI schema consumers, and docs/audit tooling out of sync with the accepted runtime config, so operators cannot reliably discover or validate the new settings.
  - confidence: 0.86
