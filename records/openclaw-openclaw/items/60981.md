## Summary

## What This Changes

The PR adds a PathGuard filesystem ACL layer with allowed/deny path config, media and apply_patch enforcement, spawned-subagent policy ceilings, config schema/help updates, tests, and a minimatch dependency.

## Best Possible Solution

Rebase and repair the PR under maintainer security review, with one coherent generated-schema-backed filesystem ACL contract applied consistently to filesystem tools, prompt-inline/local media, apply_patch, and spawned-subagent runs.

Yes. The missing main feature is reproducible by inspecting current ToolFsPolicy and config types; the PR regressions are reproducible by following prompt image loading and PathGuard relative glob matching in the PR head.

## Solution Assessment

No. PathGuard is a plausible product direction, but this PR is not the best mergeable solution until the remaining policy bypasses, generated schema drift, and rebase risk are resolved with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce PathGuard for prompt image loads:** `src/agents/pi-embedded-runner/run/attempt.ts:466`
  - body: The new toolFsPolicy is only passed to createOpenClawCodingTools here, while prompt-inline image loading still calls detectAndLoadPromptImages with only workspaceOnly and loadImageFromRef has no fsPolicy parameter. A vision prompt can reference a local image path that should be blocked by allowedPaths or denyPaths and still load it, so thread the effective policy through prompt image detection before loadWebMedia.
  - confidence: 0.91
- **[P1] Apply deny globs to symlinked relative paths:** `src/security/path-guard.ts:271-275`
  - body: Relative glob entries still return false as soon as the canonical target is outside the workspace. That is correct for allow rules, but for deny rules it means workspaceOnly=false with denyPaths like vendor/** can miss vendor when it is a symlink to an external directory, so evaluate deny globs against the canonicalized entry target instead of bailing out here.
  - confidence: 0.88
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:458-459`
  - body: The runtime schema now accepts allowedPaths, denyPaths, tools.subagents.fs, and tools.sessions_spawn.fsPolicy, but the generated public schema still exposes only tools.fs.workspaceOnly and lacks the new subtrees. Regenerate the schema baseline so config schema consumers, UI surfaces, and docs/audit tooling see the new settings.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention: the PR is intended as filesystem hardening, but the current head still contains concrete policy bypasses before merge.

Concerns:

- **[high] Prompt-inline image reads bypass allow/deny policy:** `src/agents/pi-embedded-runner/run/attempt.ts:466`
  - body: Spawned or configured ToolFsPolicy is not applied to prompt image detection/loading, so local image paths embedded directly in a vision prompt can be read even when allowedPaths or denyPaths should block them.
  - confidence: 0.91
- **[high] Relative deny globs can miss symlinked targets:** `src/security/path-guard.ts:271`
  - body: For relative glob deny entries, PathGuard returns false when the canonical target is outside the workspace. A deny glob over a symlinked workspace path can therefore be skipped when workspaceOnly is disabled.
  - confidence: 0.88
