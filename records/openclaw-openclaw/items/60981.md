## Summary

Keep open: current main still only has the older workspaceOnly filesystem policy, while this PR adds a broad new ACL/security contract that is not mergeable as-is because PR-head source still shows policy bypasses, generated schema drift, red CI, and no real behavior proof.

## What This Changes

Adds PathGuard filesystem ACL enforcement for file tools, media loads, apply_patch, and spawned-subagent policy ceilings, with config/schema/help/test updates and a new minimatch dependency.

## Best Possible Solution

Land a narrower owner-reviewed filesystem ACL contract only after one generated-schema-backed policy is enforced consistently across file tools, prompt media, local media tools, apply_patch, and spawned subagents, with green CI and redacted real setup proof.

## Reproduction Assessment

Yes. The PR-head blockers are source-reproducible from code inspection: prompt image ingestion receives only workspaceOnly, relative deny globs return false for outside-workspace symlink targets, and the generated public schema lacks the runtime-added ACL fields.

## Solution Assessment

No. PathGuard is a plausible direction, but this branch is not the best mergeable solution until the bypasses, schema drift, red checks, and real behavior proof gap are resolved under security/design review.

  - body: Passing `toolFsPolicy` only into `createOpenClawCodingTools` leaves the prompt-inline image path below on `effectiveFsWorkspaceOnly` alone. A vision prompt can still reference a local image path that should be blocked by `allowedPaths`, `denyPaths`, or a spawned-session ceiling.

  - confidence: 0.88

  - confidence: 0.86
