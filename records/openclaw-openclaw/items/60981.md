## Summary

Keep open: current main still has only the existing `workspaceOnly` filesystem guard, and this broad security PR is not mergeable yet because PR-head source review still shows policy bypasses plus generated schema drift.

## What This Changes

The PR adds a PathGuard filesystem ACL with `allowedPaths`/`denyPaths`, media and `apply_patch` checks, spawned-subagent filesystem ceilings, config/schema updates, tests, and a new `minimatch` dependency.

## Best Possible Solution

Rebase and repair the branch under security/design review, with one generated-schema-backed ACL contract applied consistently to file tools, prompt/local media, `apply_patch`, and spawned subagents while keeping exec semantics explicit or split to a canonical follow-up.

## Reproduction Assessment

Yes. Current main's missing ACL surface is source-reproducible from the strict config/type/runtime policy code, and the PR-head blockers are source-reproducible by following prompt image loading, PathGuard relative glob matching, and generated schema output.

## Solution Assessment

No. PathGuard is a plausible direction, but this PR is not the best mergeable solution until the policy bypasses, generated schema drift, rebase risk, and real behavior proof gap are resolved under maintainer security review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce PathGuard for prompt image loads:** `src/agents/pi-embedded-runner/run/attempt.ts:466`
  - body: The new `toolFsPolicy` only reaches `createOpenClawCodingTools`; prompt-inline image ingestion still calls `detectAndLoadPromptImages` with only `workspaceOnly`. A vision prompt can therefore load a local image path that should be blocked by `allowedPaths`, `denyPaths`, or a spawned-session ceiling, so thread the effective policy through prompt image detection before loading local media.
  - confidence: 0.91
- **[P1] Apply deny globs to symlinked relative paths:** `src/security/path-guard.ts:271-275`
  - body: For relative glob entries, this branch returns false as soon as the canonical target is outside the workspace. That is correct for allow rules, but it lets deny rules like `vendor/**` miss `vendor` when it is a symlink to an external directory and `workspaceOnly` is false, weakening the new deny policy.
  - confidence: 0.88
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.agent-runtime.ts:458-459`
  - body: The runtime schema now accepts `tools.fs.allowedPaths`, `tools.fs.denyPaths`, `tools.subagents.fs`, and `tools.sessions_spawn.fsPolicy`, but the generated public schema still exposes only `tools.fs.workspaceOnly` and lacks the new subtrees. Regenerate the schema baseline so config consumers and docs/audit tooling see the new settings.
  - confidence: 0.86
