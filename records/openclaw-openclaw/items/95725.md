## Summary

Keep open for maintainer review: this is a small, well-proven executor-boundary fix, but it changes cron empty-output semantics broadly and needs explicit maintainer policy/canonical-PR selection before merge.

## What This Changes

The PR adds `allowEmptyAssistantReplyAsSilent: true` to both isolated cron runner handoffs and adds regression tests that assert the flag reaches the CLI and embedded runner mocks.

## Best Possible Solution

Maintainers should choose one documented cron silent-empty policy, land one canonical executor-boundary fix for that policy, and keep the runner-level opt-in contract explicit.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` omit the cron caller flag while the CLI and embedded runner contracts require it for empty assistant output. I did not run a new live cron tick in this read-only review.

## Solution Assessment

Partly: passing the existing flag at the cron executor boundary is the cleanest implementation layer, and this PR avoids the older shared-runner lane heuristic. The unresolved part is whether the product policy should be all-cron, delivery-gated, or per-job opt-in.
