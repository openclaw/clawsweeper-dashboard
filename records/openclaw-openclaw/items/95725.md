Keep open: this is a focused, plausible fix for a real current-main cron empty-response regression, but it still needs maintainer acceptance of the all-cron silent-empty policy plus stronger current-head true-empty runtime proof before merge.

## What This Changes

This PR adds `allowEmptyAssistantReplyAsSilent: true` to both isolated cron runner calls and adds a regression test asserting the flag is passed to CLI and embedded runner paths.

## Best Possible Solution

Land one canonical executor-boundary fix after maintainers accept the all-cron silent-empty policy and the branch has redacted current-head proof for truly empty or whitespace-only cron output.

## Reproduction Assessment

Yes, source-level. Current main omits `allowEmptyAssistantReplyAsSilent` at both cron runner handoffs, and the CLI and embedded runner contracts require the opt-in to avoid empty-response failure.

## Solution Assessment

Mostly yes. Passing the existing flag at the cron executor boundary is cleaner than a shared CLI lane heuristic and covers both runner paths; the unresolved question is whether maintainers accept unconditional all-cron silent success or want a narrower policy.
