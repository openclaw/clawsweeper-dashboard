## Summary

Keep open: current main still omits the cron silent-empty flag, while this branch passes it to both isolated cron runner paths and now has current-head Gateway proof for a truly empty assistant response ending `status: ok`. The remaining decision is maintainer acceptance of the cron-wide silent-empty policy and canonical branch selection across the related open PRs.

## What This Changes

Adds `allowEmptyAssistantReplyAsSilent: true` to the isolated cron CLI and embedded runner handoffs and adds a regression test covering both paths.

## Best Possible Solution

Land one executor-boundary fix that passes the silent-empty flag through both runner paths after maintainers accept the cron-wide empty-output policy, then close the superseded sibling PRs and linked issues.

## Reproduction Assessment

Yes. Source inspection shows current main's cron executor does not pass `allowEmptyAssistantReplyAsSilent`, while the CLI runner throws `empty_response` when that flag is absent and assistant text is empty.

## Solution Assessment

Yes as an implementation boundary, with product sign-off still needed. Passing the flag from the cron executor is narrower and cleaner than a shared CLI lane heuristic, but maintainers still need to accept whether all cron delivery modes should treat empty output as silent success.
