## What This Changes

This PR passes `allowEmptyAssistantReplyAsSilent: true` from the isolated cron executor to both CLI and embedded runner calls and adds a regression test for both handoffs.

## Best Possible Solution

Land one canonical executor-boundary fix after maintainers accept the all-cron silent-empty policy and the branch has redacted current-head proof for truly empty or whitespace-only cron output.

Yes, source-level: current main omits `allowEmptyAssistantReplyAsSilent` at both cron runner handoffs, and both runner contracts require the opt-in to avoid empty-response failure. I did not run a live current-main scheduled cron tick in this read-only review.

## Solution Assessment

Mostly yes: passing the existing flag at the cron executor boundary is the cleaner owner-boundary fix compared with a shared CLI lane heuristic, and it covers both runner paths. The remaining question is whether maintainers accept unconditional all-cron silent success or want a narrower delivery-gated or per-job policy.
