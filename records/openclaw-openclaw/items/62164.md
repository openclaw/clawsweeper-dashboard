## Summary

Keep open. The backend cache direction and split UI/Apple follow-ups are reasonable, but the current PR head still has branch-local blockers: scoped session lists are keyed by all templated agent stores, the public Gateway protocol docs do not describe the new hash contract, and the changelog includes an unrelated alpha-release note.

## What This Changes

The PR adds a gateway-side `sessions.list` result cache with `hash`/`lastHash` unchanged responses, config patch source/no-op fixes, subagent/transcript invalidation signals, generated Swift protocol models, tests, and changelog entries.

## Best Possible Solution

Keep the additive backend cache, but make the store fingerprint target set match `sessions.list` request scope, document the public protocol response/request contract, and keep the changelog limited to changes implemented by this PR.

## Reproduction Assessment

Yes. With templated per-agent session stores, call `sessions.list` with an `agentId`, then touch an unrelated agent store; the PR cache key changes even though current main's miss path loads only the requested agent's target set. The docs and changelog gaps reproduce by comparing the PR diff with `docs/gateway/protocol.md` and `CHANGELOG.md`.

## Solution Assessment

No, not yet. The backend approach is maintainable once the cache key is scoped to the same stores as the miss path and the public docs/changelog are corrected.
