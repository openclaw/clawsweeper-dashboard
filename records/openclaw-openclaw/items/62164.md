## Summary

Keep open: latest head fixed the earlier `lastHash` maxLength and clone fallback comments, and checks are green, but two merge blockers remain: scoped `sessions.list` cache keys still fingerprint unrelated templated agent stores, and the additive hash/unchanged protocol is not documented publicly.

## What This Changes

The PR adds a gateway `sessions.list` result cache with `hash`/`lastHash` unchanged responses, `config.patch` source no-op fixes, subagent/transcript invalidation signals, generated Swift protocol models, tests, and a changelog entry.

## Best Possible Solution

Keep the additive backend cache, but make store fingerprinting resolve the same agent-scoped target set as `loadCombinedSessionStoreForGateway` and add public Gateway protocol docs for `hash`, `lastHash`, and `{ unchanged: true }` before landing.

## Reproduction Assessment

Yes. With templated per-agent session stores, call `sessions.list` with `agentId`, then touch an unrelated agent store; the PR cache key changes because it fingerprints all stores while the miss path loads only the requested agent. The docs gap is reproducible by comparing the schema addition with `docs/gateway/protocol.md`.

## Solution Assessment

No, not yet. The backend approach is backward-compatible and maintainable, but the mergeable solution needs agent-scoped store fingerprints plus public protocol documentation.
