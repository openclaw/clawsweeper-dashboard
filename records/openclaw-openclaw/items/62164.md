## Summary

Keep open: the backend cache direction is useful and additive, and the earlier Greptile `lastHash` bound has been addressed, but the latest PR head still has two merge blockers around agent-scoped cache keys and missing public protocol documentation.

## What This Changes

The PR adds a gateway-side `sessions.list` result cache with `lastHash`/`hash`/`unchanged` RPC fields, config.patch source-shape fixes, cache invalidation signals, Swift protocol baselines, tests, benchmark coverage, and a changelog entry.

## Best Possible Solution

Keep the additive backend cache, but align store fingerprinting with the requested-agent miss path and document the public hash/unchanged contract before merge.

## Reproduction Assessment

Yes. On the PR head, configure templated per-agent stores, call `sessions.list` with `agentId`, then write an unrelated agent store; the cache key changes because it fingerprints all stores. The docs gap is reproduced by comparing the changed schema with `docs/gateway/protocol.md`.

## Solution Assessment

No, not yet. The cache/protocol approach is maintainable and backward-compatible, but it needs agent-scoped fingerprinting and public protocol docs to be the narrow mergeable solution.
