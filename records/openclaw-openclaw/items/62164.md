## Summary

Keep open. The latest PR head keeps the useful additive backend cache direction, but it still has an agent-scoped cache-key mismatch and does not document the new public `sessions.list` hash contract.

## What This Changes

The PR adds a gateway-side `sessions.list` result cache with `lastHash`/`hash`/`unchanged` protocol fields, config patch no-op/source-shape fixes, subagent and transcript invalidation signals, Swift protocol baselines, tests, a benchmark, and a changelog entry.

## Best Possible Solution

Keep the additive backend cache, but make store fingerprinting use the same requested-agent target set as the cache-miss loader and document the public hash contract before merge.

## Reproduction Assessment

Yes. On the PR head, configure templated per-agent session stores and call `sessions.list` with `agentId`; an unrelated agent store write changes the cache key because the PR fingerprints all stores. The docs gap is reproducible by comparing the PR schema with `docs/gateway/protocol.md`.

## Solution Assessment

No, not yet. The cache design is maintainable and backward-compatible, but the mergeable solution needs agent-scoped store fingerprinting plus public protocol documentation for the new request and response behavior.
