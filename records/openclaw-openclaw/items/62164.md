## Summary

Keep open. The PR’s backend direction is useful and the linked UI/Apple follow-ups are correctly split out, but the current branch still has two merge blockers: scoped `sessions.list` cache keys over-fingerprint unrelated agent stores, and the new public hash/unchanged protocol is not documented.

## What This Changes

This PR adds a gateway `sessions.list` result cache with `hash`/`lastHash` unchanged responses, config.patch no-op fixes, subagent/transcript invalidation signals, generated Swift protocol params, tests, and a changelog entry.

## Best Possible Solution

Keep the additive backend cache, but align store fingerprinting with the requested-agent target set and document the public hash/unchanged contract before merge.

## Reproduction Assessment

Yes. With templated per-agent session stores, call `sessions.list` with `agentId`, then mutate an unrelated agent store; the PR cache key changes because it fingerprints all stores while the miss path loads only the requested agent. The docs gap is reproducible by comparing the schema diff with `docs/gateway/protocol.md`.

## Solution Assessment

No, not yet. The backend cache approach is maintainable and backward-compatible, but the mergeable solution needs agent-scoped store fingerprints and public protocol documentation; the UI and Apple client adoption can remain follow-up work.
