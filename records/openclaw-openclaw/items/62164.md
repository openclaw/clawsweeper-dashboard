## Summary

Keep open: current main does not implement this cache/protocol work, and the PR is mostly on the right path, but PR head still misses a subagent generation bump for yielded/orphaned run mutations, so cached `sessions.list` rows can stay stale until TTL.

## What This Changes

The PR adds a gateway-side `sessions.list` result cache with `hash`/`lastHash` unchanged responses, config.patch source-shaped no-op handling, invalidation signals, docs, changelog, Swift models, and regression tests.

## Best Possible Solution

Fix the missed subagent-generation bumps and add targeted cache regression coverage, then land the additive backend cache while leaving UI and Apple `lastHash` adoption to follow-up PRs.

## Reproduction Assessment

Yes. Source comparison shows current main lacks `lastHash` and unchanged responses, while PR head adds them; the remaining stale-cache bug is source-reproducible because yielded subagent state mutates list-visible fields without bumping the generation used by the cache key.

## Solution Assessment

No, not yet. The additive protocol/cache direction is maintainable, but the subagent invalidation contract is incomplete for yielded/orphaned run mutations and should be repaired before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Bump subagent generation for yielded run state:** `src/agents/subagent-registry.ts:349-352`
  - body: Changing `persistSubagentRuns` so generation bumps are opt-in leaves bare persist calls after `markSubagentRunPausedAfterYield` without cache invalidation. Pass `{ bumpGeneration: true }` for these list-visible persist paths, including the ended-orphan reconciliation path, and add a cache regression.
