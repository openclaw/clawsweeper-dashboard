## Summary

Keep open. Current main still emits the empty-allowlist and untracked-provenance warning behavior for auto-discovered non-bundled plugins, and a maintainer already scoped this as a real UX/policy issue rather than stale cleanup. A related open PR addresses the populated-but-mismatched allowlist case, but it does not fully resolve this issue's first-run empty-allowlist remediation concern.

## What This Changes

The issue asks for plugin allowlist/provenance startup warnings to be less alarming and more actionable for first-time users, including clearer plugin ID guidance and better handling of bundled versus discovered plugins.

## Best Possible Solution

Keep this open and coordinate the warning UX work with #68389: land or rebase the populated-mismatch warning if maintainers want that path, then address this issue's first-run empty-allowlist remediation by keeping the security warning but adding exact plugin IDs, a concrete plugins.allow snippet, and links or command hints for plugin ID discovery.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path: a workspace/global discovered non-bundled plugin with plugins enabled and an empty allowlist still triggers the empty-allowlist warning, and a global loaded plugin without provenance still triggers the untracked-provenance warning.

## Solution Assessment

Unclear as a combined solution because the related PR only fixes the populated-mismatch case and intentionally preserves the empty-allowlist wording. The best narrow direction is still warning-copy/docs remediation while preserving trust semantics, not suppressing warnings or changing auto-load policy as part of this issue.
