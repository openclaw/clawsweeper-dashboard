## Summary

Keep open. Current main still emits the empty-allowlist and untracked-provenance warnings for activating plugin loads, and the maintainer discussion explicitly scoped this as unresolved UX/policy work rather than an obsolete report.

## What This Changes

The issue asks OpenClaw to make first-run plugin trust warnings clearer and less alarming, especially when `plugins.allow` is empty or users confuse channel IDs with plugin IDs.

## Best Possible Solution

Land a focused UX/docs change that keeps the warning but makes it actionable: include the discovered plugin IDs, show a minimal `plugins.allow` snippet, point to the plugin list/inspect commands, and keep broader loading-policy or channel-alias behavior as separate maintainer decisions.

## Reproduction Assessment

Yes. The current source tests provide a high-confidence reproduction path for the warning behavior: workspace/global auto-discovery with an empty allowlist still warns, and a loaded global plugin without provenance still produces the untracked-provenance warning.

## Solution Assessment

Yes for the narrow maintainer-scoped direction: improve warning copy, docs, and regression coverage while preserving warn-level trust semantics. No for suppressing the warning entirely or changing auto-load policy without a product/security decision.
