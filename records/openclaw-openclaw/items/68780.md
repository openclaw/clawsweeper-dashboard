## What This Changes

The issue asks for first-time plugin allowlist and untracked-provenance warnings to be less alarming and to tell users exactly how to trust discovered plugin IDs.

## Best Possible Solution

Land or revise the linked warning-copy/docs fix so the existing security warning keeps firing but includes exact plugin IDs, safe `plugins.allow` remediation, and CLI inspection guidance.

## Reproduction Assessment

Yes, at source level: with plugins enabled, an empty `plugins.allow`, and a workspace/global non-bundled plugin, current main emits the generic empty-allowlist warning; loaded untracked non-bundled plugins similarly get the generic provenance warning.

## Solution Assessment

Yes for the narrow copy/docs direction: preserve the warning and trust semantics, then add exact plugin IDs, safe allowlist guidance, and CLI inspection hints. Broader changes such as auto-mapping channel IDs, suppressing first-run warnings, or changing auto-load policy need separate maintainer product/security approval.
