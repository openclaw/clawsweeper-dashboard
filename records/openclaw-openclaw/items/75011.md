## Summary

Keep open. The documentation gap is real, but this PR still gives inaccurate operator recovery guidance for both paired-node command upgrades and stale node tokens, so it should not merge as-is.

## What This Changes

The PR updates `docs/gateway/pairing.md` with a Troubleshooting section for dropped node commands, missing pending requests, and stale tokens, plus small quote normalization changes.

## Best Possible Solution

Land a docs-only correction in `docs/gateway/pairing.md` that separates first-time node pairing from paired-node command upgrades, tells operators to reconnect or restart the node after approving newly requested command surfaces, and describes stale-token recovery as removing/re-pairing the node or creating and approving a fresh pending request.

## Reproduction Assessment

Yes. The issue is reproducible by reading the PR diff against current main and tracing connect-time reconciliation, WebSocket command snapshotting, approval/rejection state changes, and existing node-pairing tests.

## Solution Assessment

No, not as currently proposed. Adding troubleshooting to the pairing doc is the right shape, but the wording needs to match current gateway behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Align dropped-command guidance with reconnect gating:** `docs/gateway/pairing.md:206-211`
  - body: The new troubleshooting text says commands missing after approval were likely dropped during the pending state and can be retried once paired. Current main keeps allowlisted commands visible for first-time unpaired nodes, while paired nodes that reconnect with new commands keep only the previously approved command list for that live connection. Please distinguish first-time pairing from paired-node command upgrades and tell operators to reconnect or restart the node before retrying newly approved command surfaces.
  - confidence: 0.89
- **[P3] Correct stale-token recovery guidance:** `docs/gateway/pairing.md:222-223`
  - body: This says to reject and re-approve when a stale token is not accepted, but rejection only removes the pending request and does not issue a fresh token. Please point operators to removing/re-pairing the node or making the node create a new pending request, then approving that new request.
  - confidence: 0.78
