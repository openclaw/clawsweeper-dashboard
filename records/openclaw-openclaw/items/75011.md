## Summary

Keep open. The docs-only direction is useful, but the current patch still misstates gateway behavior: first-time unpaired node commands are kept visible after policy filtering, paired-node command upgrades stay limited to the previously approved command snapshot until reconnect, and rejection does not mint a replacement token.

## What This Changes

This PR adds a Troubleshooting section to docs/gateway/pairing.md for dropped commands, missing pending requests, and stale tokens, with small quote normalization changes.

## Best Possible Solution

Land a docs-only correction in docs/gateway/pairing.md that distinguishes first-time node pairing from paired-node command upgrades, tells operators to reconnect or restart the node after approving newly requested command surfaces, and describes stale or lost token recovery as remove/re-pair or create a new pending request before approval.

## Reproduction Assessment

Yes. The issue is reproducible by reading the PR diff against current main and tracing current connect-time reconciliation, WebSocket command snapshotting, approval/rejection state changes, and existing node-pairing tests.

## Solution Assessment

No, not as currently proposed. Adding troubleshooting to the pairing doc is the right repair shape, but the wording should be corrected before merge so it matches current gateway behavior and token lifecycle semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Align dropped-command guidance with reconnect gating:** `docs/gateway/pairing.md:206-211`
  - body: The new troubleshooting text says commands missing after approval were likely dropped during the pending state and can be retried once paired. Current main keeps allowlisted commands visible for first-time unpaired nodes, while paired nodes that reconnect with new commands keep only the previously approved command list for that live connection. Please distinguish first-time pairing from paired-node command upgrades and tell operators to reconnect or restart the node before retrying newly approved command surfaces.
  - confidence: 0.89
- **[P3] Correct stale-token recovery guidance:** `docs/gateway/pairing.md:222-223`
  - body: This says to reject and re-approve when a stale token is not accepted, but rejection only removes the pending request and does not issue a fresh token. Please point operators to removing/re-pairing the node or making the node create a new pending request, then approving that new request.
  - confidence: 0.78
