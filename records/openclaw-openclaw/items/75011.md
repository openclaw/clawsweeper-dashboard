## Summary

Keep open. The operator-docs gap is real, but this PR still contradicts current gateway behavior for first-time command visibility, paired-node command upgrades, and stale-token recovery; the remaining defects are narrow docs-only fixes.

## What This Changes

This PR adds a Troubleshooting section to docs/gateway/pairing.md for dropped-command, missing pending-request, and stale-token recovery scenarios, plus quote normalization.

## Best Possible Solution

Land a docs-only correction that documents current gateway semantics: keep WS device pairing separate from node.pair.* pending entries, distinguish first-time node pairing from paired-node command upgrades, require reconnect or restart before retrying newly approved command surfaces, and recover stale or lost tokens by removing/re-pairing or creating and approving a new pending request.

Yes. The issue and the PR defects reproduce by comparing the new Troubleshooting text with src/gateway/node-connect-reconcile.ts, src/infra/node-pairing.ts, the WS connect snapshot, and the existing node-pairing tests.

## Solution Assessment

No. Adding troubleshooting to docs/gateway/pairing.md is the right location, but this wording is not the best fix because it misstates current command visibility and token recovery; a narrow docs rewrite is the safer path.

- **[P2] Align command recovery guidance with gateway behavior:** `docs/gateway/pairing.md:209-214`
  - body: The new troubleshooting text says missing commands after approval were likely dropped during the pending state and only tells operators to re-trigger.
