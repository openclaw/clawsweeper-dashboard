## Summary

Keep open. Current main and v2026.6.6 include a delivery-time descendant drain and fallback for isolated cron subagents, but they still do not provide a reliable spawn/wait/aggregate primitive or a clear docs boundary saying that isolated cron fan-in orchestration is unsupported.

## What This Changes

The issue requests a cron-safe way to spawn parallel subagents, wait for all results, and aggregate them, or explicit documentation that isolated cron sessions do not support that pattern.

## Best Possible Solution

Maintainers should choose and document the support boundary: either add a durable cron-safe fan-in/continuation primitive that lets the parent aggregate results, or explicitly document that isolated cron jobs should use single-turn work, separate cron jobs, or no subagent fan-in.

## Reproduction Assessment

Yes for the source-level gap: current main has no sessions_await/barrier tool and docs still present sessions_yield as the wait primitive without an isolated-cron caveat.
