## Summary

Current main improves push-based completion and nested descendant fan-in, but it still exposes no deterministic barrier for an isolated cron root to await a declared child cohort and receive one structured aggregate. The report remains necessary, although its historical claim that the spawn note tells cron agents not to poll is outdated because current main suppresses that note for cron session keys.

## What This Changes

Add a reliable way for isolated cron orchestrators to spawn parallel subagents, wait for all required results, and continue to synthesize one final response.

## Best Possible Solution

Add one core, durable cohort abstraction that records expected child run IDs, terminal outcomes, timeout/partial policy, and a single continuation event, then make isolated cron completion remain nonterminal until that continuation finishes.

## Reproduction Assessment

Yes at the source-contract level: current main only offers non-blocking spawn plus next-event yield for cron roots, while deterministic aggregated fan-in is tested only for registered nested subagent runs. A live current-main cron execution was not performed in this read-only review.

## Solution Assessment

The requested capability is valid, but no single option in the issue is yet the proven best implementation; the safest design is a durable runtime-owned cohort continuation shared by cron and nested subagents, not polling, synchronous `sessions_send`, or a parallel ad hoc wait path.
