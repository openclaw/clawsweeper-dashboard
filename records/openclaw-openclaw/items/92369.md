- `P2`: The item concerns a bounded but important multi-agent cron workflow without threatening general runtime availability.
- `impact:session-state`: The parent run becomes inactive before child completion state can be deterministically aggregated into the scheduled workflow result.

## Summary

Current main does not provide a reliable isolated-cron fan-in contract: `sessions_yield` deliberately ends and deactivates the parent run, while later subagent completions arrive as separate messages. The existing implementation supports asynchronous continuation messaging, but it does not own the complete cron workflow through deterministic aggregation.

## What This Changes

Add reliable fan-out/fan-in orchestration for subagents launched from isolated cron sessions, or explicitly document that this workflow is unsupported.

## Best Possible Solution

Define one canonical core lifecycle in which isolated cron owns the workflow until all declared child prerequisites complete and the parent finishes synthesis, with structured timeout and partial-result semantics; if that lifecycle is intentionally unsupported, gate the misleading tool path in isolated cron and document the limitation explicitly.

## Reproduction Assessment

Yes at source level. The supplied isolated-cron flow reaches a tested `sessions_yield` contract that ends and deactivates the parent run, so it cannot synchronously retain ownership through fan-in; this read-only review did not execute a live cron job.

## Solution Assessment

No single requested option is yet proven best. Maintainers should first define cron continuation ownership, then extend the existing canonical yield/announce lifecycle rather than introducing a competing wait path by default.
