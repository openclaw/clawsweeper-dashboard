## Summary

Keep open: current main has cron descendant-drain and fallback-delivery mitigations, but it still lacks a model-facing barrier/await primitive and `sessions_yield` remains an end-turn pause rather than a durable cron spawn/wait/aggregate contract.

## What This Changes

The issue requests reliable isolated-cron subagent orchestration: spawn multiple subagents, wait for all required results, aggregate them, or document that this workflow is unsupported.

## Best Possible Solution

Decide and document one supported contract for isolated-cron subagent orchestration, then either codify the current limitations or design a narrow fan-in primitive that fits the existing agent runtime without a heavy orchestration layer.

## Reproduction Assessment

Yes at source level, but not via a live run here: current source and tests show no `sessions_await` or barrier tool, `sessions_spawn` remains non-blocking, and `sessions_yield` leaves the parent run idle after ending the turn.

## Solution Assessment

Unclear: the current descendant-drain/fallback path is a useful mitigation for stale cron delivery, but it is not the full requested spawn/wait/aggregate contract and the right permanent shape is a maintainer product/API decision.
