## Summary

Keep this open: current main and v2026.6.6 have a useful isolated-cron delivery fallback for descendant subagent output, but they still do not provide the requested model-facing way to spawn children, wait for all of them, and let the cron orchestrator aggregate results before finishing. The discussion from eightHundreds and icophy is consistent with the source: this needs maintainer product/API direction rather than cleanup closure.

## What This Changes

The issue requests reliable fan-out/fan-in support for isolated cron sessions that spawn subagents and need to wait for and aggregate child results.

## Best Possible Solution

Define the supported contract: either add a small core barrier/continuation primitive for isolated cron subagent workflows, or document that spawn-wait-aggregate orchestration is unsupported and recommend one-turn or staged-cron designs.

## Reproduction Assessment

Yes, at source level. Current tests show `sessions_yield` ends the parent run and cannot accept queued follow-up steering, while current source search found no replacement await/barrier primitive.

## Solution Assessment

No. Current main has a delivery-side descendant drain and fallback, but that is not the requested model-facing mechanism to spawn, wait for all child results, and continue orchestrator aggregation inside an isolated cron session.
