## Summary

Keep open: current main has best-effort descendant wait/fallback handling for isolated cron delivery, but source and docs still do not show a reliable first-class spawn-wait-aggregate contract for cron-isolated subagent orchestration.

## What This Changes

The issue requests a reliable cron-isolated subagent fan-out/fan-in path that can spawn multiple child runs, wait for them, and aggregate results.

## Best Possible Solution

Define one supported contract: document cron subagent fan-in as best-effort/unsupported, or land a narrow barrier/delegate API with cron, sessions_yield, and session-state proof.

## Reproduction Assessment

Yes from source for the missing capability: current main exposes no await/delegate/barrier tool and isolated cron only has a delivery-time descendant drain/fallback. I did not run the live cron scenario.

## Solution Assessment

Unclear; a blocking tool is plausible but adds core API and session-state risk, while documentation or a narrower cron continuation contract may be better. Maintainers should choose the product/API shape first.
