## Summary

Keep open: current main has cron delivery follow-up/fallback handling, but it does not provide the requested reliable model-facing spawn, wait, and aggregate mechanism, and the related barrier/await work is closed or adjacent rather than canonical.

## What This Changes

The issue requests reliable cron-isolated subagent fan-out/fan-in so an orchestrator can spawn workers, wait for completion, and aggregate their results.

## Best Possible Solution

Keep this open until maintainers choose one supported outcome: a first-class fan-in primitive, revised yield lifecycle semantics for cron-isolated runs, or explicit docs that cron-isolated parallel subagent orchestration is unsupported.

## Reproduction Assessment

Yes, source-reproducible: current tests assert the parent session is idle after `sessions_yield`, and current cron follow-up code only substitutes delivery output after the run. A live cron E2E was not run in this read-only pass.

Unclear. The issue gives multiple plausible solutions, and choosing between a barrier tool, changed yield lifecycle, or explicit unsupported documentation requires maintainer product direction rather than a narrow bug patch.
