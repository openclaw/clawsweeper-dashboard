## Summary

Keep open: current main has a cron delivery fallback that waits for descendant runs and can substitute descendant output, but it still does not provide the requested model-visible spawn/wait/aggregate primitive or a clear unsupported-boundary decision for isolated cron orchestration.

## What This Changes

The issue requests a reliable fan-out/fan-in mechanism for isolated cron sessions, or documentation that this orchestration pattern is unsupported.

## Best Possible Solution

Keep this as the canonical product/API decision point: either add a narrow first-class await/barrier primitive with focused cron/subagent lifecycle proof, or explicitly document that isolated cron supports only delivery fallback and not general multi-turn orchestration.

## Reproduction Assessment

Source-level only: current main clearly lacks `sessions_await` or a barrier primitive and still documents non-blocking spawn behavior. I did not establish a live failing isolated-cron run in this read-only review.

## Solution Assessment

No. The current delivery fallback is useful but not the best complete answer to the issue because it does not give the orchestrator a model-visible, structured wait-and-aggregate primitive or a documented unsupported boundary.
