## Label Justifications

- `P2`: The issue blocks a meaningful scheduled multi-agent workflow but is limited to advanced isolated-cron orchestration rather than the general runtime.
- `impact:session-state`: The failure is caused by the isolated parent run ending or being finalized without a durable continuation that associates all child completions and the final synthesis.

## Summary

Current main has a useful registry-backed cron follow-up path that drains active descendants and can combine their replies, but it does not resolve the reported `sessions_yield` lifecycle mismatch: the tool still ends the embedded turn, while cron does not consume the yielded state as a durable continuation. The issue remains necessary to choose and implement one reliable supported contract for isolated cron orchestration.

## What This Changes

Make isolated cron sessions reliably spawn parallel subagents, await all required completions, synthesize their results, and deliver one final outcome.

## Best Possible Solution

Use one cron-owned lifecycle path that records the spawned cohort, drains every required descendant, guarantees a final parent synthesis turn, and only then records and delivers the terminal cron outcome; either hide `sessions_yield` from isolated cron runs or make it participate in that same lifecycle rather than adding another parallel waiting mechanism.

## Reproduction Assessment

Yes at the source-contract level: `sessions_yield` ends the embedded turn and exposes yielded metadata, while isolated cron finalization does not consume that state as a durable continuation. This read-only review did not execute a live cron job, so the status is source-reproducible rather than directly reproduced.

## Solution Assessment

No single requested option is yet proven best. A new barrier tool would duplicate part of the existing registry-backed cron fan-in path; the narrower maintainable direction is to make cron and yielded-run lifecycle semantics converge on one explicit cohort and terminal-outcome contract.
