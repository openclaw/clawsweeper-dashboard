## Summary

Keep open: current main has internal cron descendant-drain and delivery fallback mitigations, but it still lacks the requested reliable cron-safe spawn/wait/aggregate primitive; the remaining work is a product/API decision, not an already-implemented fix.

## What This Changes

The issue requests a reliable fan-out/fan-in mechanism for isolated cron sessions to spawn multiple subagents, wait for all required completions, and aggregate their results.

## Best Possible Solution

Make a maintainer decision between a narrow supported barrier/gather API for run-scoped cron subagents and explicit docs that isolated cron only supports push-based completion/fallback, then implement that chosen path with focused cron/subagent tests.

## Reproduction Assessment

Yes at source level: current main exposes non-blocking sessions_spawn and turn-ending sessions_yield, but no sessions_await/barrier tool or wait-for-all result API; I did not run a live cron job.

## Solution Assessment

No, not yet: delivery fallback mitigates stale cron announcements, but the best maintainable solution still needs a maintainer-chosen API/lifecycle shape or docs explicitly saying this orchestration is unsupported in isolated cron.
