## Summary

Current main and v2026.6.6 drain descendant runs during cron delivery, but they do not provide a deterministic parent continuation that receives all child results and performs the requested aggregation. The issue should remain open for a maintainer decision on the lifecycle contract; the contributor’s correction of the earlier unrelated commit attribution is supported by direct history inspection. Likely related people: @tyler6204 (high confidence, introduced the cron descendant-gating and push-delivery flow) and Peter Steinberger (high confidence, later centralized the shared drain helpers).

## What This Changes

Make isolated cron orchestrators reliably spawn parallel subagents, await all required completions, and synthesize one aggregate result.

## Best Possible Solution

Extend the existing push-based yield/follow-up lifecycle so a parent can declare required child runs, receive one structured all-complete continuation, and retain cron timeout and delivery ownership through the final synthesis without blocking descendant lanes.

## Reproduction Assessment

Yes at source level with high confidence: the provided isolated cron configuration reaches a delivery-time descendant drain, while current code provides no deterministic cohort result or guaranteed parent synthesis continuation. A live current-main cron run was not executed because this review is read-only.

## Solution Assessment

No single proposed option is yet proven best. Repairing the existing documented sessions_yield/follow-up lifecycle is narrower and more maintainable than adding a parallel sessions_await API, but maintainers must define cohort, timeout, partial-result, and lane-ownership semantics.
