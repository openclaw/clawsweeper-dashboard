## Summary

Current main improves yielded-run continuation, but it still does not provide a deterministic multi-child barrier returning combined results, and isolated cron completion remains a separate lifecycle concern. The report is therefore still necessary; the single-turn and staged-cron workaround shared in the discussion is useful but does not satisfy the requested fan-out/fan-in contract.

## What This Changes

Add reliable parallel subagent spawning, wait-for-all continuation, and result aggregation for isolated cron sessions, or clearly document that the workflow is unsupported.

## Best Possible Solution

Define a lifecycle-owned grouped-child completion contract with structured all/partial/timeout outcomes, durable parent continuation, and clear restart/cancellation semantics; then make isolated cron remain nonterminal until the resumed parent completes synthesis, while documenting single-turn and staged-cron workarounds in the interim.

## Reproduction Assessment

Yes, at source-contract level: current main exposes immediate non-blocking spawn and turn-ending yield with later individual completion events, but no cohort-based wait-for-all response. A live cron job was not executed during this read-only review.

## Solution Assessment

No single alternative in the report is yet the best proven implementation. A cron-only `sessions_yield` exception would leave general fan-in unresolved; the maintainable direction is a lifecycle-owned grouped-completion primitive that cron can consume without blocking a concurrency lane.

## Risks / Open Questions

- The report combines two related but separable decisions: a general grouped-child barrier API and isolated cron ownership of resumed work through its final terminal outcome.
- A blocking wait implemented directly inside the current run could hold concurrency lanes and recreate nested-lane or timeout problems unless waiting releases the active execution slot and uses durable lifecycle state.
- Individual best-effort completion events require explicit partial completion, timeout, restart, cancellation, and duplicate-delivery semantics before they can safely back a deterministic barrier.
