## Summary

Keep open: current main still implements safe-bin executable trust at directory granularity only, and the requested path/hash trust would add a new security-sensitive exec trust/config contract that maintainers should design before implementation. The recent reporter follow-up adds production context and explicitly defers a PR until that contract is clarified.

## What This Changes

The issue requests per-binary safe-bin trust by resolved path, with optional SHA-256 pinning, instead of only trusting whole directories through `safeBinTrustedDirs`.

## Best Possible Solution

Decide the canonical exec trust surface first; if accepted, add typed config/schema, docs, audit or doctor guidance, and regression coverage for path, symlink, update, and hash-failure behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main supports directory-level safe-bin trust and a separate host executable allowlist, but no per-binary or hash safe-bin trust field.

## Solution Assessment

Unclear. The requested path/hash trust could reduce broad directory grants, but a quick hot-path field addition is not clearly the best fix until maintainers choose how it should interact with the existing executable allowlist and approval semantics.
