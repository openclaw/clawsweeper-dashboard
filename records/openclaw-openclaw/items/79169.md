## Summary

Keep open: current main still models safe-bin executable trust by directory only, while the requested path/hash trust would add a new security-sensitive config surface that overlaps with existing exec approval allowlists. The reporter also added fresh production context and is explicitly waiting for maintainer product/security direction before proposing code.

## What This Changes

Request a safe-bin config surface for trusting individual executables by absolute path and optionally SHA-256 hash instead of trusting whole directories.

## Best Possible Solution

Have maintainers choose the canonical trust surface, then either extend exec approval allowlists with the needed hash semantics or add a typed `safeBinTrustedBinaries` contract with schema, docs, audit/doctor guidance, and regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main supports directory-level safe-bin trust plus a separate host allowlist path, not per-binary or hash-pinned safe-bin trust.

## Solution Assessment

Unclear. A per-binary/hash config could solve the wrapper maintenance problem, but the narrow maintainable solution depends on whether exec approvals should remain the canonical path trust mechanism.
