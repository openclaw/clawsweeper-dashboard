## Summary

Keep open: current main still models safe-bin executable trust by trusted directory only, while the requested per-binary path/hash trust would add a new security-sensitive config surface that overlaps with the existing host allowlist.

## What This Changes

Request a `safeBinTrustedBinaries`-style config that trusts individual executable paths, optionally pinned by SHA-256, instead of trusting whole directories through `safeBinTrustedDirs`.

## Best Possible Solution

Decide whether the existing host allowlist is the intended per-path trust answer; if safe-bin scoped binary/hash trust is accepted, add a typed schema/docs/audit/doctor/tested contract.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main only supports directory-level safe-bin trust plus the separate host allowlist path.

## Solution Assessment

Unclear. A per-binary/hash config could solve the wrapper maintenance problem, but it changes exec trust semantics and overlaps with documented allowlist behavior, so product/security design should come before implementation.
