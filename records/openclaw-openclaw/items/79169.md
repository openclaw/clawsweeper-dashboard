## Summary

Keep open: current main still has directory-only safe-bin executable trust, and the production follow-up confirms the requested path/hash trust remains useful but needs maintainer product/security design because it changes exec trust semantics and overlaps the existing host allowlist.

## What This Changes

Add a safe-bin trust mechanism for specific executables by absolute path and optionally SHA-256 hash, instead of only trusting whole directories.

## Best Possible Solution

Decide the canonical exec trust surface first; if maintainers accept binary or hash-scoped trust, add typed config/schema/docs plus audit, doctor, runtime, and regression coverage around that contract.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main only supports directory-level safe-bin trust via safeBinTrustedDirs, with no per-binary path or hash field.

## Solution Assessment

Unclear. The requested path/hash trust would address the wrapper-maintenance and over-broad-directory problem, but the maintainable surface needs a product/security decision because the docs already position the host allowlist as explicit executable-path trust.
