## Summary

Keep open: current main still only supports directory-level safe-bin trust, while per-binary path or hash trust would add a new security-sensitive config surface rather than fix broken existing behavior.

## What This Changes

Request adds per-binary safe-bin trust by absolute path and optionally SHA-256 hash so operators do not need to trust an entire binary directory.

## Best Possible Solution

Maintainers should decide whether the existing exec approvals allowlist is the intended per-path trust answer; if not, design a typed safe-bin binary/hash trust contract across schema, runtime, docs, audit, doctor, and tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main only accepts directory-level `safeBinTrustedDirs` for safe-bin trust, while per-path executable trust is documented through the separate host allowlist.

## Solution Assessment

Unclear. A per-binary or hash field could address the wrapper maintenance problem, but it overlaps with the existing allowlist and should be designed as a security/product contract before any narrow code change.
