## Summary

Keep open: current main still models safe-bin trust by trusted directory only, while the requested path/hash trust would add a security-sensitive config surface that overlaps the existing host allowlist.

## What This Changes

The issue requests per-binary safe-bin trust by resolved executable path, with an optional SHA-256 pin, instead of trusting whole binary directories.

## Best Possible Solution

Decide whether the existing host allowlist is the intended per-path trust answer; if safe-bin scoped binary/hash trust is accepted, add a typed schema, runtime policy, docs, audit/doctor guidance, and tests for the new contract.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main supports directory-level safe-bin trust plus the separate host allowlist path, but no safe-bin-specific binary or hash trust field.

## Solution Assessment

Unclear. The requested config could reduce wrapper maintenance, but it changes exec trust semantics and overlaps documented allowlist behavior, so the maintainable path needs maintainer product/security design first.
