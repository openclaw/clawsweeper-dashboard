## Summary

Keep open. Current main still lacks a public image-tool media local-root allowlist: the explicit image tool resolves only workspace/default roots, the public media config schema/types do not expose a root field, and tests still preserve rejection of arbitrary external image paths. The newer managed-inbound media exemption helps the supporting OpenClaw media-store case, but it does not cover external iMessage attachment directories.

## What This Changes

Request a configurable image/media local-root allowlist so external channel attachment directories can be read by the image tool without copying files into the workspace.

## Best Possible Solution

Define and document a narrow opt-in local media root contract, or a channel-owned attachment-root propagation contract if maintainers prefer that boundary. The implementation should validate explicit non-filesystem-root directories, preserve managed inbound-media behavior and existing rejection coverage, integrate through the shared resolver/loadWebMedia path, and update schema/types/docs/tests together.

## Reproduction Assessment

Yes. A focused reproduction path is to call the explicit image tool with an absolute local image path outside the workspace/default roots and observe the existing rejection; current tests cover that behavior in src/agents/tools/image-tool.test.ts.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but the best fix needs maintainer/security judgment because it expands local read access and overlaps the narrower iMessage attachment-root approach in #39034.
