## Label Justifications

- `impact:security`: The requested capability changes which host filesystem paths a model-facing tool may read across agent trust boundaries.

## Summary

Current main now handles ordinary iMessage attachments through channel-scoped inbound roots, but deliberately excludes those roots when `tools.fs.workspaceOnly` is enabled. The supporting mixed-trust gateway use case therefore remains unresolved and requires an explicit filesystem-security product decision rather than stale cleanup.

## What This Changes

Add configurable additional local filesystem roots for image-tool reads so trusted workflows can process attachments stored outside OpenClaw-managed workspaces.

## Best Possible Solution

Keep inbound attachment access tied to matching channel and account context, then design an explicitly approved per-agent trusted-media capability or controlled staging path for mixed-trust gateways without weakening the default `workspaceOnly` boundary.

## Reproduction Assessment

Yes at source level: current main returns no channel inbound roots when `tools.fs.workspaceOnly` is enabled, directly matching the supporting mixed-trust report. The ordinary non-workspace-only iMessage case is now handled on main when invocation channel context is present.

## Solution Assessment

No, the proposed broad `tools.image.mediaLocalRoots` setting is not yet the safest maintainable design. A channel/account-bound staging flow or per-agent trusted-media capability would preserve isolation better than a gateway-wide image-tool allowlist.
