## Summary

Keep open: current main and v2026.6.10 still do not provide the requested configurable media local-root field, while the narrow iMessage attachment-root symptom is handled through channel-scoped inbound roots. The remaining work is a security/product decision because it expands model-facing host filesystem reads and overlaps the open tools.fs.roots design PR.

## What This Changes

Request adds an openclaw.json media local-root allowlist, such as tools.image.mediaLocalRoots, so image tools can read external attachment directories without copying files into the workspace.

## Best Possible Solution

Choose one reviewed filesystem/media-root contract, likely shared across image, PDF, prompt-image, outbound media, and host file tools, then wire schema, docs, tests, and security review through that canonical policy.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 have strict config schemas without tools.image.mediaLocalRoots, tools.mediaLocalRoots, or tools.fs.roots, and the image tool only receives default/workspace localRoots plus scoped inbound roots. I did not run a live iMessage or Docker gateway scenario in this read-only review.

## Solution Assessment

No for the exact image-only knob until maintainers choose the broader trust boundary. A shared filesystem/media-root policy is more maintainable than adding a separate image-tool-only allowlist that could drift from PDF, prompt-image, outbound media, and host file tools.
