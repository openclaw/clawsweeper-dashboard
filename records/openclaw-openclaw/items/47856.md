## Summary

Current main solves the original non-`workspaceOnly` iMessage case through channel-scoped inbound roots, but deliberately excludes those roots when `tools.fs.workspaceOnly` is enabled; the mixed-trust use case therefore remains a security-sensitive product decision. Likely related people: @vincentkoc (high confidence, current resolver and channel-root work), @jacobtomlinson (medium confidence, allowlist hardening), and @steipete (medium confidence, earlier shared-runtime ownership).

## What This Changes

Add configurable local filesystem roots for built-in media tools so trusted agents can process attachments stored outside their workspace.

## Best Possible Solution

Keep channel attachment access channel-owned; if trusted mixed-trust deployments need broader access, define one explicit per-agent filesystem/media authorization contract with documented precedence over `workspaceOnly` and focused security tests.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: with `tools.fs.workspaceOnly` enabled, media paths outside the normalized agent workspace are excluded and channel inbound roots are also disabled.

## Solution Assessment

No. The proposed `tools.image.mediaLocalRoots` shape is not yet the best proven solution because it duplicates channel trust and scopes shared filesystem authorization to one tool; maintainers must define ownership and `workspaceOnly` precedence first.
