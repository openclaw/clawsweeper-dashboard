## Summary

Keep open: current main and v2026.6.10 still do not expose a configurable media-root field for the image tool. Current code has a narrower iMessage attachment-root path through channel-scoped inbound roots, but the requested general local media allowlist remains a security-sensitive product/config decision and the nearest implementation path is still the open conflicting filesystem-roots PR.

## What This Changes

Request to add a `tools.image.mediaLocalRoots`-style config field so the image tool can read operator-approved local media roots such as iMessage attachment folders without manual copying.

## Best Possible Solution

Choose one canonical operator-controlled root policy, likely through the existing `tools.fs.roots` discussion or an explicitly scoped media policy, then land schema/types/docs/tests that cover image/PDF local media, iMessage inbound attachments, workspaceOnly behavior, and upgrade/security expectations.

## Reproduction Assessment

Yes, source-level: current tests and implementation show local image paths outside default/workspace roots are rejected, and current config schemas do not provide a mediaLocalRoots-style opt-in. I did not run a live iMessage reproduction in this read-only cleanup pass.

## Solution Assessment

No, not as a direct `tools.image.mediaLocalRoots` patch yet. The safer solution is for maintainers to choose a canonical filesystem/media root policy first, because the existing code deliberately separates channel inbound roots from broad host-local read roots.
