## Summary

Keep open. Current main partially addresses ordinary iMessage attachment handling through managed inbound media and channel-scoped attachment roots, but it still has no supported configurable local-media-root contract, and the remaining request is a security/product decision rather than a narrow bug fix.

## What This Changes

The issue requests an opt-in configuration field to extend local media roots for the image tool so trusted attachment directories can be read without copying files into the workspace.

## Best Possible Solution

Define one scoped, explicit local-media/filesystem root contract that preserves workspace-only isolation by default and makes trusted-root expansion schema-backed, documented, channel/agent-aware, and covered by regression tests.

## Reproduction Assessment

Yes, at source level. Current main has no public local-root config field, the image tool uses the shared resolver directly, and workspace-only mode disables channel inbound roots; I did not run live iMessage because this was a read-only review.

## Solution Assessment

No for the exact proposed image-only knob until maintainers settle the security contract. A shared scoped filesystem/media-root policy is safer than splitting the same read boundary across per-tool allowlists.
