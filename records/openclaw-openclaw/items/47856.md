## Summary

Keep open. Current main handles ordinary iMessage attachment paths when the image tool has matching channel/account context, and managed OpenClaw inbound media can be read under workspace-only policy, but there is still no supported configurable local-root contract and channel inbound roots are deliberately disabled when `tools.fs.workspaceOnly` is active.

## What This Changes

The issue requests configurable media local roots for the image tool so iMessage and similar attachment paths can be read without copying into the workspace.

## Best Possible Solution

Define one scoped, explicit local-media access contract that preserves workspace-only isolation by default and makes any trusted-root widening agent/channel-bound, schema-backed, documented, and regression-tested.

## Reproduction Assessment

Yes at source level: current media-tool code still has no public configured local-root field, returns only workspace roots when workspace-only is active, and disables channel inbound roots in that mode. I did not run tests because this review was read-only.

## Solution Assessment

No, the proposed broad `tools.image.mediaLocalRoots` field is not yet the safest design. A shared scoped filesystem/media-root policy would avoid splitting the same security boundary across per-tool knobs.
