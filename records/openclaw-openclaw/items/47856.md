## Summary

Keep open. Current main still does not expose a public image/media-tool local-root allowlist, and the existing resolver continues to allow only default/workspace roots while tests preserve rejection for external local image paths.

## What This Changes

Request an opt-in `tools.media.image.mediaLocalRoots`-style config field so the image tool can read configured external attachment directories such as iMessage attachments.

## Best Possible Solution

Define and document a narrow opt-in local media root contract, or a channel-owned attachment-root propagation contract, then wire it through the shared resolver and `loadWebMedia` with schema/type/docs/tests that preserve existing rejection behavior.

## Reproduction Assessment

Yes. A focused reproduction path is to call the explicit image tool with an absolute local image outside the workspace/default roots and observe the existing rejection; current tests cover that behavior in `src/agents/tools/image-tool.test.ts`.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but maintainers should choose the final boundary because it broadens local read access and overlaps the narrower iMessage attachment-root approach in #39034.
