## Summary

Keep open. Current main still does not provide a public image/media local-root allowlist for external attachment directories: the explicit image tool only uses workspace/default media roots, the public tools media schema and type contract have no `mediaLocalRoots` field, and existing tests still preserve rejection of non-workspace local paths. Managed inbound OpenClaw media is now allowed, which covers part of the supporting `media/inbound` case, but not external iMessage attachment paths.

## What This Changes

The issue requests a configurable image/media local-root allowlist so the image tool can read trusted attachment directories such as iMessage `Messages/Attachments` without manual copying.

## Best Possible Solution

Define a narrow, documented opt-in contract for image/media local roots or a channel-owned attachment-root propagation contract. The implementation should reject broad filesystem roots, preserve managed inbound-media behavior and existing outside-path rejection coverage, wire through the shared resolver/`loadWebMedia`, and update schema, types, docs, and focused tests together.

## Reproduction Assessment

Yes. A focused reproduction path is to call the explicit image tool with an absolute image path outside workspace/default media roots, such as a Messages attachment path; current tests already assert this class of path is rejected.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but a channel-owned root propagation path like the one explored in #39034 may be narrower for iMessage; maintainers should choose the boundary before implementation.
