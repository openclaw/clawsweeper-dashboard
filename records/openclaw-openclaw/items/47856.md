## Summary

Keep open: current main has fixed the narrow iMessage attachment-root path through channel-scoped inbound roots, but it still lacks the requested public configurable media local-root field. Adding that field would broaden model-facing local filesystem reads and overlaps the open `tools.fs.roots` design work, so this remains a maintainer product/security decision rather than a safe cleanup close.

## What This Changes

The issue requests a configurable image/media local-root allowlist so trusted attachment directories can be read without copying files into the workspace.

## Best Possible Solution

Resolve this as one shared filesystem/media-root policy: either land or narrow a reviewed `tools.fs.roots`-style config with schema, docs, tests, and security review, or document the current `tools.fs` plus channel inbound-root model as the supported boundary.

## Reproduction Assessment

Yes, source-level: current strict schemas do not expose `tools.image.mediaLocalRoots`, `tools.mediaLocalRoots`, or `tools.fs.localRoots`, and tests still reject arbitrary external local image paths. I did not run a live iMessage or Docker gateway scenario in this read-only review.

## Solution Assessment

No for the exact image-only knob until maintainers choose the security contract. The current inbound-root fix is the right narrow solution for iMessage attachments, while arbitrary host roots should be handled through a shared filesystem/media-root policy or documented as intentionally unsupported.
