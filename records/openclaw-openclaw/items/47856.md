## Summary

Current main fixes the original non-`workspaceOnly` iMessage symptom through channel-scoped inbound attachment roots, but it deliberately blocks those roots when `tools.fs.workspaceOnly` is enabled, so the mixed-trust use case remains a real product/security decision rather than stale work. Likely related people: Vincent Koc (high confidence, current trust-boundary implementation) and Jacob Tomlinson (medium confidence, earlier configuration-derived allowlist hardening).

## What This Changes

Add an operator-configurable allowlist of extra local roots for built-in image/media tools, including trusted-agent access outside the workspace.

## Best Possible Solution

Keep channel attachment roots as a separate inbound trust domain, and—if maintainers approve trusted-agent exceptions—add a narrowly named agent-scoped host-read policy with explicit `workspaceOnly` precedence, schema/help/docs, and tests proving both locked-down defaults and intentional opt-in access.

## Reproduction Assessment

Yes. On current main, setting `tools.fs.workspaceOnly` makes both media local roots and channel inbound roots exclude every non-workspace directory, which deterministically reproduces the mixed-trust limitation described in the comments.

## Solution Assessment

No, the proposed flat root merge is not yet the best solution. Current architecture intentionally separates broad host-local reads from channel-scoped inbound attachments, so an approved implementation should model a narrow trusted-agent exception instead of conflating those trust domains.
