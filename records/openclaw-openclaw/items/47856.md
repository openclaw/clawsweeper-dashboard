## Summary

Keep open. Current main fixes the iMessage/channel-inbound attachment symptom and managed inbound-media access, but it still does not provide a supported configurable local-media-root contract; the remaining work is a security/product decision about broadening host filesystem reads.

## What This Changes

Request to add an opt-in config field that extends image/media tool allowed local filesystem roots for attachment-heavy workflows.

## Best Possible Solution

Define one schema-backed local filesystem/media-root policy, preferably shared with the tools.fs.roots-style contract, while preserving scoped inbound channel roots and rejecting ad hoc per-image-tool allowlists.

## Reproduction Assessment

Yes, at source level for the current behavior: the strict config schema has no media-root allowlist field, and the image tool still rejects arbitrary outside-root local images. The iMessage-specific attachment path is separately fixed through scoped inbound roots on current main.

## Solution Assessment

No for the exact image-only knob until maintainers settle the security contract. A shared scoped filesystem/media-root policy is more maintainable than splitting the same read boundary across per-tool allowlists.
