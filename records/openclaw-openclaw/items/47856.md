## Summary

Keep open. Current main still lacks a public configurable local-root allowlist for image/PDF media tools, and the remaining work is a security-sensitive product/config decision rather than stale cleanup.

## What This Changes

The issue requests an opt-in configuration field that lets image and related media tools read local media from additional allowed roots, especially iMessage attachment directories.

## Best Possible Solution

Choose one canonical explicit-root policy, preferably through the shared filesystem/media-root contract, then wire it through media root resolution, schema/types/docs, image/PDF tests, and the iMessage attachment-root path without weakening managed inbound-media or workspaceOnly semantics.

## Reproduction Assessment

Yes, for the current limitation: source and tests show external local image/PDF paths are checked against default/workspace roots only. This is source-reproducible, but the requested allowlist is a new config capability rather than a documented behavior bug.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but maintainers need to choose between a per-media option, channel attachment-root inheritance, and the broader tools.fs.roots policy already under review.
