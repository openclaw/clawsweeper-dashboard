## Summary

Keep open. Current main still has no public image-tool local-root allowlist, and the managed inbound-media/iMessage attachment-root paths only cover adjacent ingestion and ACP/media-understanding flows, not explicit image-tool reads of external attachment directories.

## What This Changes

Add an opt-in config allowlist for additional image/media local roots so explicit image-tool calls can read iMessage attachments outside default or workspace roots.

## Best Possible Solution

Define and document a narrow opt-in local media root contract, or deliberately route channel-owned attachment roots into the explicit image/media tools, with schema/types/docs/tests preserving existing rejection behavior by default.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call the explicit image tool with an absolute image path outside the workspace/default roots and observe the current not-under-allowed-directory rejection, which is also covered by src/agents/tools/image-tool.test.ts.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but the best fix needs maintainer/security judgment because it expands local read scope and overlaps the narrower iMessage attachment-root approach in #39034.
