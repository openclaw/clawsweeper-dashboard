## Summary

Keep open: current main and v2026.6.10 still do not provide a configurable media-root field, while the narrow iMessage attachment-root path is only handled through channel-scoped inbound roots. The remaining work is a product and security-boundary decision, and the nearest implementation candidate is the still-open conflicting filesystem-roots PR.

## What This Changes

The issue requests a configurable media-root allowlist for the image tool so local attachment paths outside OpenClaw-managed directories can be read without copying them into the workspace.

## Best Possible Solution

Choose one canonical filesystem/media-root contract, likely by reviewing or narrowing https://github.com/openclaw/openclaw/pull/52951, and avoid adding an image-only root knob that would drift from PDF, outbound media, and prompt-image access.

## Reproduction Assessment

Yes. Source and tests show the image tool rejects non-workspace local image paths without a channel-scoped inbound root, and the config schema has no accepted configurable media-root field.

## Solution Assessment

No, not as an image-only field. The safer maintainable direction is a single per-agent filesystem/media-root contract that covers image, PDF, outbound media, and prompt-image paths consistently.
