## Summary

Keep open. Current main and the latest release still lack a supported configurable local-root contract for image/media tools; existing code only covers default/workspace roots plus channel-scoped inbound roots in limited contexts, so the remaining work needs maintainer product and security review.

## What This Changes

The issue requests an `openclaw.json` setting such as `tools.image.mediaLocalRoots` to extend image-tool allowed local filesystem roots for iMessage attachments and similar host media.

## Best Possible Solution

Define one scoped, explicit local media/filesystem root contract with schema, docs, tests, and security review, preserving workspace-only defaults while allowing deliberate operator-approved roots.

## Reproduction Assessment

Yes at source level: current code and tests show non-workspace image paths are rejected unless they match existing default/workspace or scoped inbound roots. I did not run tests because this review is read-only.

## Solution Assessment

No, the narrow proposed `tools.image.mediaLocalRoots` field is not clearly the best permanent shape. A shared scoped filesystem/media-root policy would avoid splitting the same security boundary across per-tool knobs.
