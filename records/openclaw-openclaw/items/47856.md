## Summary

Keep open: current main and v2026.6.8 solve the narrow iMessage attachment-root path through channel-scoped inbound roots, but they still do not expose the requested configurable media local-root field. The remaining work is a product/security decision because it broadens model-facing local filesystem reads and overlaps the open tools.fs.roots design PR.

## What This Changes

Request to add a configurable image/media local-root allowlist so the image tool can read iMessage and other attachment files outside OpenClaw-managed directories.

## Best Possible Solution

Resolve this through one reviewed filesystem/media-root policy, with schema, docs, tests, and security review, or document the current channel inbound-root plus tools.fs behavior as the intentional boundary.

## Reproduction Assessment

Yes, source-level: current strict schemas do not expose tools.image.mediaLocalRoots, tools.mediaLocalRoots, or tools.fs.roots, and current tests still reject ordinary external local image paths. I did not run a live iMessage or Docker gateway scenario in this read-only review.

## Solution Assessment

No for the exact image-only knob until maintainers choose the security contract. The narrow iMessage attachment symptom is already handled through inbound roots; the remaining arbitrary host-root access should be a shared filesystem/media-root policy or explicitly documented as unsupported.
