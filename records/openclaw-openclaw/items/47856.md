## Summary

Keep open. Current main and the latest release still lack a public configurable local-media root allowlist for the image tool, while the related iMessage attachment-root PR remains open and the broader media-root contract is security-sensitive.

## What This Changes

The issue requests a `tools.image.mediaLocalRoots`-style config option so the image tool can read trusted external attachment directories such as iMessage attachments without copying them into the workspace.

## Best Possible Solution

Define one narrow opt-in local media root contract with schema/types/docs, shared image/PDF/media-loader wiring, explicit root validation, and regression tests that preserve managed-inbound and `workspaceOnly` protections.

## Reproduction Assessment

Yes, at source level. Current main routes image/PDF local paths through fixed roots and strict config types/schemas expose no `mediaLocalRoots`-style option, while tests still reject non-workspace image paths.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but maintainers need to decide whether the canonical surface is media-tool roots, iMessage attachment-root inheritance, or a broader filesystem-root policy.
