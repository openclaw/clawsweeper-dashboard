## Summary

Keep open. Current main and the latest release still lack a public configurable local-media root allowlist for the built-in image/PDF media tools, while the overlapping iMessage-root and broader filesystem-root PRs remain open and need maintainer security/product review before they can supersede this discussion.

## What This Changes

The issue requests a configurable allowlist for additional local media roots so image/media tools can read paths such as iMessage attachments without manually copying files into the workspace.

## Best Possible Solution

Define one shared, opt-in local media/filesystem root contract with schema/types/help/docs, strict root validation, image/PDF/media-loader wiring, and tests that preserve managed-inbound and workspaceOnly protections.

## Reproduction Assessment

Yes, for the current limitation. Source and regression tests show outside-workspace local image/PDF paths are rejected and current schemas expose no public extra-root config; I did not run a live iMessage flow in this read-only review.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but maintainers need to choose whether the canonical contract is media-tool roots, iMessage attachment-root inheritance, or the broader tools.fs.roots policy.
