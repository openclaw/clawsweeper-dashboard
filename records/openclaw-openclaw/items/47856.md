## Summary

Keep open. Current main still lacks a public configurable media local-root allowlist: the media-tool resolver uses only workspace/default roots, the image tool passes that set directly to media loading, and the strict tools media/fs config schemas expose no root field. The remaining work is active but unresolved in https://github.com/openclaw/openclaw/pull/52951 and overlaps the narrower iMessage attachment-root work in https://github.com/openclaw/openclaw/pull/39034.

## What This Changes

The issue requests a configurable local media root allowlist so the image tool can read external attachment directories without copying files into the workspace.

## Best Possible Solution

Settle whether to land the shared tools.fs.roots contract from https://github.com/openclaw/openclaw/pull/52951 or a narrower media-only allowlist, then update schema/types/docs/tool root resolution and regression tests together.

## Reproduction Assessment

Yes, from source inspection. On current main, configure or imagine an external iMessage attachment path, call the explicit image tool with that path, and the tool validates against resolveMediaToolLocalRoots, which does not include any configured external media root; current tests also pin rejection of outside-root image paths.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but the best solution needs maintainer/security judgment because the broader tools.fs.roots PR and the narrower iMessage attachment-root PR solve overlapping parts of the same filesystem boundary.
