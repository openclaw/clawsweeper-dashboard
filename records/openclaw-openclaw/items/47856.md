## Summary

Keep open. Current main still lacks a public configurable media/local-root allowlist for the image/PDF media tools, and this issue is now the canonical open discussion for that product/security contract; the related implementation paths are still unmerged and overlap rather than fully supersede it.

## What This Changes

The issue requests an opt-in configuration field that lets image/media tools read local media from additional operator-approved roots such as iMessage attachment directories.

## Best Possible Solution

Define one shared, opt-in local media/filesystem root contract with narrow root validation, schema/types/docs, image/PDF/media loader wiring, and tests that preserve current managed-inbound and `workspaceOnly` protections.

## Reproduction Assessment

Yes, for the current limitation. Source and regression tests show outside-workspace local image/PDF paths are rejected and current schemas expose no public extra-root config; I did not run a live iMessage flow in this read-only review.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but maintainers need to decide whether the canonical contract is media-tool roots, iMessage attachment-root inheritance, or the broader `tools.fs.roots` policy.
