## Summary

Keep open: current main has the merged z.ai/Pi auto-compaction guard from https://github.com/openclaw/openclaw/pull/76056, but this recent report says the problem still reproduces after that release and shows prior turns missing from a session `.bak` file, not only from a provider request body. The remaining Feishu media/session rewrite path is credible but not isolated enough to close or auto-fix safely.

## What This Changes

Reports that consecutive Feishu DM turns on `zai/glm-5.1` lose the prior user/assistant exchange from the session JSONL when the second turn includes a DOCX/media attachment.

## Best Possible Solution

Add a deterministic regression for consecutive Feishu direct-message turns where the second turn carries a DOCX/media attachment on a z.ai-style GLM model, then fix the proven transcript rewrite, routing, or compaction boundary.

## Reproduction Assessment

No, not with high confidence. The report gives concrete Feishu/z.ai backup excerpts and steps, but the live Feishu DOCX path, full JSONLs, effective provider route, and same-session key were not verified against current main.

## Solution Assessment

Unclear. The existing z.ai auto-compaction guard is the right fix for the known provider-request context-loss class, but this report points to history already missing from a session backup, so the safer path is to isolate that rewrite or routing boundary first.
