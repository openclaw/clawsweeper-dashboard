## Summary

Keep open: the reported symptom is fresh on 2026.5.2 and appears distinct from the already-merged z.ai auto-compaction guard, because the issue shows session backups where visible user/assistant turns disappeared from the persisted JSONL rather than only from the provider request payload. Current source confirms the prior guard exists, but does not identify an intentional pruning, repair, or reset path that should remove those visible turns within 78 seconds.

## What This Changes

The issue reports that consecutive Feishu DM turns using `zai/glm-5.1` can silently drop prior user and assistant messages from the persisted session file.

## Best Possible Solution

Add a deterministic regression for the two-turn Feishu DM media-follow-up path that verifies the same session key and persisted JSONL retain prior visible turns, then fix the proven transcript rewrite or compaction boundary.

## Reproduction Assessment

No, not yet. The reporter provides concrete steps and credible backup excerpts, but I did not live-replay the Feishu plus z.ai gateway path and current source inspection does not expose a deterministic turn-dropping path.

## Solution Assessment

Unclear. The existing #76056 guard is the right fix for the known z.ai Pi auto-compaction provider-request bug, but this report points at persisted session-file loss after that fix shipped, so the smallest maintainable fix depends on isolating the new boundary first.
