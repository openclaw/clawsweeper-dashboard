## Summary

Keep open: current main already contains the shipped z.ai-style Pi auto-compaction guard, but this fresh report says the same symptom still occurs on 2026.5.2 with session backups showing visible turns disappearing within 78 seconds. The remaining trigger is not identified from source inspection, so closing as implemented would be unsafe.

## What This Changes

The issue reports that Feishu/Lark sessions using `zai/glm-5.1` lose prior user and assistant turns from the session transcript when a second media message arrives.

## Best Possible Solution

Add a focused regression around consecutive Feishu/Lark media turns on a z.ai-style provider, then fix the exact transcript rewrite or compaction path without broadening pruning/reset behavior for unrelated providers.
