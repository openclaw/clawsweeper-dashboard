## Summary

Keep open: the report is recent, post-2026.5.2, and includes concrete session-backup evidence that visible transcript history disappeared on disk after the known z.ai consecutive-turn guard had shipped. Current main contains the z.ai guard and a later transcript replay fix, but source inspection still does not identify an intentional reset, repair, maintenance, or cache-pruning path that should remove a fresh user/assistant exchange within 78 seconds.

## What This Changes

The issue reports that Feishu/Lark DM sessions using `zai/glm-5.1` lose prior user/assistant turns from the session JSONL when a second media/DOCX message arrives.

## Best Possible Solution

Add a deterministic regression for consecutive Feishu/Lark direct-message turns where the second turn carries a DOCX/media attachment on a z.ai-style GLM model, then fix the proven transcript rewrite, session routing, or compaction boundary without broadening unrelated pruning behavior.
