## Summary

Keep open: current main and the reported 2026.5.2 release already contain the prior z.ai auto-compaction guard, but the reporter supplies fresh post-release session backups showing visible conversation turns disappearing within 78 seconds. Source inspection does not identify an intentional reset/pruning path for that loss, and the backup model metadata conflicts with the stated z.ai default, so closing as implemented would be unsafe.

## What This Changes

The issue reports that consecutive Feishu DM turns using a z.ai/GLM provider can rewrite the session transcript so the first user/assistant exchange is lost before the second turn.

## Best Possible Solution

Add a focused regression for consecutive Feishu file-message turns on a z.ai-style GLM provider after #76056, then fix the exact transcript rewrite or compaction path without broadening pruning behavior for unrelated providers.

## Reproduction Assessment

No, not from the current checkout alone. The report provides concrete steps and session backup evidence, but I did not replay the live Feishu/@larksuite/openclaw-lark plus z.ai gateway path and source inspection leaves the remaining trigger ambiguous.

## Solution Assessment

Unclear. The existing #76056 guard is the right fix for the known Pi z.ai silent-overflow auto-compaction path, but this report reproduces on the shipped guard and the backup names point toward session-file repair, so the root cause should be isolated before patching.
