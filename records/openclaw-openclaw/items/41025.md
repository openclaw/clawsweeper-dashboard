## Summary

Keep open for maintainer review: current main still blocks remote markdown images and the linked user request remains open, but this branch is not merge-ready because it conflicts with current main, targets an older renderer shape, and has unresolved CSP, sanitizer, remote-media policy, and real-browser proof gaps.

## What This Changes

The PR changes Control UI markdown rendering and tests to render `http://` and `https://` markdown image URLs inline with `referrerpolicy`, plus a changelog note.

## Best Possible Solution

Make an explicit Control UI remote-media policy decision; if approved, implement it on the current markdown-it renderer with matching sanitizer, CSP, docs, tests, privacy mitigations, and browser proof.

## Reproduction Assessment

Yes, at source level: current main flattens non-data markdown image URLs to alt text, tests assert that behavior, and the served Control UI CSP blocks remote image loads. I did not run a live browser repro because this review is read-only.

## Solution Assessment

No. The current PR is not the best solution as written because it changes an older renderer path and does not align CSP, sanitizer allowed attributes, docs, or the remote-fetch privacy policy.

Full review comments:

- **[P2] Align CSP before enabling remote images:** `ui/src/ui/markdown.ts:149`
  - body: The new remote `<img>` branch can emit `http`/`https` sources, but current Control UI still serves `img-src 'self' data: blob:` and this PR does not update `src/gateway/control-ui-csp.ts`, so browsers will block the loads the change is meant to enable.
  - confidence: 0.94
- **[P2] Allow referrerpolicy through the sanitizer:** `ui/src/ui/markdown.ts:149`
  - body: The PR expects `referrerpolicy="no-referrer"` to survive, but the sanitizer uses an explicit `ALLOWED_ATTR` list without `referrerpolicy`; DOMPurify will strip the mitigation and the updated assertions will not hold.
  - confidence: 0.9
- **[P2] Resolve the remote image policy before default-on loads:** `ui/src/ui/markdown.ts:149`
  - body: Rendering arbitrary chat/model remote image URLs makes each viewer's browser fetch third-party resources, exposing IP address, user agent, and read timing; current docs and CSP intentionally prevent those fetches, so this needs an approved remote-media policy before default-on rendering.
  - confidence: 0.9
- **[P3] Correct the CSP claim in the changelog:** `CHANGELOG.md:15`
  - body: The changelog says this PR allows `https:` in the Control UI CSP `img-src` directive, but the diff does not modify the CSP helper, so the release note overstates the actual behavior.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff enables untrusted remote browser image fetches without an approved remote-media policy, matching CSP, or surviving referrer mitigation.

Concerns:

- **[medium] Remote image fetches leak viewer metadata:** `ui/src/ui/markdown.ts:149`
  - body: Rendered chat/model content could include tracking pixels that make the browser fetch a third-party URL, exposing viewer IP address, user agent, and read timing; `referrerpolicy` does not prevent the request itself.
  - confidence: 0.91
- **[low] Referrer policy mitigation is stripped:** `ui/src/ui/markdown.ts:149`
  - body: The PR emits `referrerpolicy="no-referrer"`, but the explicit DOMPurify allowed-attribute list does not include that attribute, so the intended Referer mitigation is removed unless the sanitizer allowlist changes.
  - confidence: 0.88
- **[low] Plain HTTP image sources weaken the policy:** `ui/src/ui/markdown.ts:45`
  - body: The proposed regex allows `http://` as well as `https://`; on HTTPS Control UI deployments this can be mixed-content blocked and is weaker than an approved secure remote-media allowlist or proxy policy.
  - confidence: 0.82
