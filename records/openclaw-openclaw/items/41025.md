## Summary

Keep open. The PR targets a real requested Control UI behavior and is linked to an open closing issue, but current main intentionally keeps remote markdown images text-only while CSP/docs reject remote image fetches, and this patch still needs product/security judgment plus real browser proof before merge.

## What This Changes

The PR changes Control UI markdown rendering/tests and the changelog so remote http/https markdown image URLs render inline with `referrerpolicy="no-referrer"`.

## Best Possible Solution

Make an explicit Control UI remote-media policy decision; if approved, land a coordinated renderer, CSP, sanitizer allowlist, docs, tests, privacy mitigation, and real browser proof change, otherwise keep the no-remote-fetch policy.

## Reproduction Assessment

Yes, at source level. Current main returns escaped alt text for non-data markdown image URLs, tests assert that behavior, and Control UI CSP/docs reject remote image fetches.

## Solution Assessment

No. A renderer/test/changelog-only change is not the best way to solve this because it conflicts with current CSP/docs/privacy policy and omits the sanitizer and CSP changes needed for the claimed behavior.

## Security Review

Status: needs_attention

Summary: The diff enables untrusted remote browser image fetches from chat content without an approved remote-media policy or working referrer mitigation.

Concerns:

- **[medium] Remote image fetches leak viewer metadata:** `ui/src/ui/markdown.ts:149`
  - body: Rendered chat/model content could include tracking pixels that make the browser fetch a third-party URL, exposing viewer IP address, user agent, and read timing; `referrerpolicy` does not prevent the request itself.
  - confidence: 0.91

  - body: The PR emits `referrerpolicy="no-referrer"`, but the explicit DOMPurify allowed-attribute list does not include that attribute, so the intended Referer mitigation is removed unless the sanitizer allowlist changes.
  - confidence: 0.88

  - body: The proposed regex allows `http://` as well as `https://`; on HTTPS Control UI deployments this can be mixed-content blocked and is weaker than a secure remote-media allowlist or proxy policy.
  - confidence: 0.82
