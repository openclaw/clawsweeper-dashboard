## Summary

Keep open, but this PR is not merge-ready: it targets a real requested UX change, yet current main deliberately blocks remote image fetches, the branch is dirty against main, and the latest patch still lacks matching CSP/docs policy and real browser proof.

## What This Changes

The branch changes Control UI markdown image rendering/tests and the changelog so http/https markdown image URLs emit inline `<img>` tags with `referrerpolicy` instead of falling back to alt text.

## Best Possible Solution

Decide the Control UI remote-media policy first; if approved, port the change to current main, align renderer, CSP, docs, changelog, tests, and privacy mitigations, then show real browser proof, otherwise keep the shipped no-remote-fetch behavior and continue policy discussion in https://github.com/openclaw/openclaw/issues/64607.

## Reproduction Assessment

Yes, at source level. Current main's markdown-it image renderer returns alt text for non-data image sources, tests assert that behavior, and the served CSP would block remote loads even if the renderer emitted an `<img>` tag.

## Solution Assessment

No. A renderer-only http/https allowance is not the best solution because current main intentionally prevents remote image fetches across renderer, CSP, docs, and tests; the maintainable path needs an approved remote-media policy and matching surfaces.

## Security Review

Status: needs_attention

Summary: Needs attention: the patch enables browser fetches for untrusted remote chat images while current main intentionally blocks that fetch class.

Concerns:

- **[medium] Remote image fetches leak viewer metadata:** `ui/src/ui/markdown.ts:149`
  - body: Allowing arbitrary remote image URLs in rendered chat/model content lets message content trigger third-party browser requests, exposing IP address, user agent, and read timing to the image host; `referrerpolicy` only removes the Referer header.
  - confidence: 0.91
- **[low] Plain HTTP image sources weaken the policy:** `ui/src/ui/markdown.ts:45`
  - body: The proposed regex allows `http://` as well as `https://`; on HTTPS Control UI deployments this can be blocked as mixed content or create a weaker remote-media policy than maintainers may intend.
  - confidence: 0.82

  - body: The changelog states that the PR allows `https:` in the Control UI CSP, but the PR does not change the CSP helper, which could mislead users and reviewers about the actual browser behavior.
