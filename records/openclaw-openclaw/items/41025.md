## Summary

Keep this PR open, but do not merge it as-is. Current main still blocks remote markdown images, and this PR is a focused implementation candidate for the linked open bug, but the renderer-only change conflicts with the served Control UI CSP, current docs, and the existing no-remote-fetch privacy posture.

## What This Changes

This PR changes the Control UI markdown renderer and tests so `http://` and `https://` markdown image URLs emit inline `<img>` elements instead of flattening to alt text, while continuing to flatten unsafe schemes.

## Best Possible Solution

Keep this PR as an implementation candidate, but require an explicit remote-media policy first. If maintainers approve remote images, revise the patch to update the renderer, CSP, docs, and tests together with mitigations such as HTTPS-only, `referrerpolicy="no-referrer"`, allowlisting, or a proxy; otherwise keep the current no-fetch behavior documented and route the broader media work through #64607.

## Reproduction Assessment

Yes. The source and tests give a high-confidence reproduction path: `toSanitizedMarkdownHtml("![Alt](https://example.com/img.png)")` is currently expected to return alt text without `<img>`, and the served CSP would block a remote image even if a renderer emitted one.

## Solution Assessment

No, the proposed renderer-only fix is not the best complete solution as-is. The narrower maintainable path is to decide the remote-fetch policy, then update the renderer, CSP, documentation, and tests consistently with the chosen mitigation model.
