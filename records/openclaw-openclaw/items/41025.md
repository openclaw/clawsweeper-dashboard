## Summary

Keep open for maintainer review: this PR is useful context for the linked remote-image request, but current main deliberately blocks remote image fetches through the renderer, CSP, docs, and prior hardening, so enabling remote http/https images is a product/security policy change rather than a narrow bug fix.

## What This Changes

The PR changes Control UI markdown rendering and tests so http/https markdown image URLs emit inline img elements instead of alt text.

## Best Possible Solution

Decide the Control UI remote-media policy first; if approved, implement it consistently across renderer, CSP, docs, tests, changelog, and privacy mitigations, otherwise keep the current no-remote-fetch contract and track broader inline media in #64607.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: toSanitizedMarkdownHtml("![Alt text](https://example.com/img.png)") is covered by tests that expect alt text without an img tag, and the served CSP would block remote image loads even if the renderer emitted them.

## Solution Assessment

No. A renderer-only change that allows both http and https image URLs conflicts with current CSP/docs/tests and changes operator-browser privacy behavior without an approved policy or mitigation.

## Security Review

Status: needs_attention

Summary: The diff has no workflow, dependency, lockfile, package-publishing, or supply-chain change, but it introduces browser remote-fetch behavior in chat and needs privacy/security review.

Concerns:

- **[medium] Arbitrary remote image fetches leak viewer metadata:** `ui/src/ui/markdown.ts:149`
  - body: The new http/https image path would let untrusted chat or model content make each viewer's browser request attacker-controlled image URLs, exposing network metadata and read timing to the image host. Current main's renderer, CSP, and docs prevent that fetch class.
