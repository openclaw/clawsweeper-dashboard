## Summary

Keep this PR open, but do not merge it as-is. It addresses a real open feature request, and a commenter explicitly asked not to let the feature die, but current main intentionally blocks remote image fetches through the markdown renderer, CSP, docs, and shipped hardening history; this PR only changes the renderer/test expectation and needs product/security approval plus a rebase onto the current renderer.

## What This Changes

The PR changes Control UI markdown rendering and tests so remote http/https markdown image URLs produce inline img elements instead of flattening to alt text.

## Best Possible Solution

Decide the Control UI remote-media policy first. If maintainers approve remote images, implement it against the current markdown-it renderer together with CSP, docs, changelog, and tests, preferably with a clear https/proxy/allowlist/referrer policy; otherwise keep the shipped no-remote-fetch behavior and use #64607 as the broader media-display design thread.

## Reproduction Assessment

Yes. Current main has a direct source/test reproduction: `toSanitizedMarkdownHtml("![Alt text](https://example.com/img.png)")` is expected to return alt text without an img tag, and the served CSP would still block remote image loads even if the renderer emitted them.

## Solution Assessment

No. A renderer-only change that allows both http and https remote images is not the best complete solution because it conflicts with current CSP/docs and changes operator-browser privacy behavior without an approved policy or mitigation.

## Security Review

Status: needs_attention

Summary: The diff has no workflow, dependency, lockfile, package-publishing, or supply-chain change, but it changes browser remote-fetch behavior in chat and needs privacy/security review.

Concerns:

- **[medium] Arbitrary remote image fetches leak viewer metadata:** `ui/src/ui/markdown.ts:149`
  - body: The new http/https image path would let untrusted chat or model content make each viewer's browser request attacker-controlled image URLs, exposing network metadata and read timing to the image host. Current main's renderer, CSP, and docs prevent that class of fetch.
  - confidence: 0.9
- **[low] HTTP image sources create mixed-content risk:** `ui/src/ui/markdown.ts:149`
  - body: The proposed remote-image regex allows plain http sources as well as https. On HTTPS Control UI deployments this can be blocked as mixed content or weaken the remote-media policy unless maintainers explicitly allow it.
