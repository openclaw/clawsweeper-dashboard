## Summary

Keep this PR open, but do not merge it as-is. Current main still intentionally blocks remote markdown image fetches in the Control UI renderer, CSP, tests, and docs, while this patch only changes the markdown renderer/tests and skips the required remote-media privacy/CSP/product decision.

## What This Changes

The PR changes Control UI markdown rendering and tests so `http://` and `https://` markdown image URLs emit inline `<img>` elements while unsafe schemes continue to flatten to alt text.

## Best Possible Solution

Decide the Control UI remote-media policy first. If maintainers approve remote images, implement it in the current markdown-it renderer together with CSP, docs, changelog, and tests; otherwise keep the shipped no-remote-fetch behavior and use #64607 as the broader media-display design thread.

## Reproduction Assessment

Yes. Current main has a direct source/test reproduction: `toSanitizedMarkdownHtml("![Alt text](https://example.com/img.png)")` is expected to return alt text without `<img>`, and the served CSP would still block `https:` images even if the renderer emitted them.

## Solution Assessment

No. A renderer-only change that allows both `http://` and `https://` is not the best complete solution because it conflicts with the current CSP/docs and changes operator-browser privacy behavior without an approved policy or mitigation.
