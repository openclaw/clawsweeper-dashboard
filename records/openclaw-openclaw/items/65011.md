## Summary

Keep open. Current main has native updater primitives, auto-update config, a config-page Updates section, update-available banner/action, and restart-sentinel verification, but it still lacks the requested first-class Update Manager with rich status/history, dry-run/progress, release-note links, safe-update controls, and post-update health summary.

## What This Changes

Request a first-class Control UI or onboarding Update Manager that surfaces native OpenClaw updates, auto-update configuration, status history, safe-update controls, and operator health feedback.

## Best Possible Solution

Keep this as an updater product umbrella and implement it in stages: add additive backend status/history/release-note/health contracts, then build a dedicated Control UI surface that edits existing update config and orchestrates the native updater without duplicating it.

## Reproduction Assessment

Not applicable as a runtime bug. The product gap is high-confidence by source inspection: current main has updater primitives and limited UI affordances, but no dedicated manager route or rich updater status contract.

## Solution Assessment

Yes for the product direction, but not as a single unsliced patch. The maintainable path is to keep the updater engine in core and expose it through additive gateway contracts plus staged Control UI work.
