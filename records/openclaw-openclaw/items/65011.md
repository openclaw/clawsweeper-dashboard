## Summary

Keep open. Current main has the native updater, auto-update config, a Control UI update banner/action, and restart-sentinel verification, but it still lacks the requested first-class Update Manager surface with full status, dry-run, release notes, persisted attempt/failure history, safe-update controls, and post-update health summary.

## What This Changes

The issue requests a first-class Control UI Update Manager that exposes native updater state, configuration, actions, release notes, safe-update controls, and health feedback for operators.

## Best Possible Solution

Keep this as the product umbrella for a staged Control UI Update Manager: add additive updater/status RPCs, expose the existing update config and actions in a dedicated Settings or Infrastructure surface, persist and display last attempt/success/failure state, link release notes before apply, and coordinate safe-update/health controls with the related auto-update and rollback issues.

## Reproduction Assessment

Not applicable as a runtime bug, but the product gap is reproducible by source inspection: current main has updater primitives and a banner/config affordance, while navigation and protocol code have no dedicated manager route or rich updater status contract.

## Solution Assessment

Yes for the product direction, but not as a single unsliced patch. The maintainable path is to surface and orchestrate the existing native updater through additive backend contracts and staged UI work rather than creating a second updater subsystem.
