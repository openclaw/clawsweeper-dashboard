## Summary

Keep open. Current main has native updater primitives, persisted auto-update check state, Control UI update banners/actions, restart-sentinel feedback, and docs, but it still lacks the requested first-class Control UI Update Manager with rich history, dry-run/progress, release-note links, safe-update controls, and post-update health summary.

## What This Changes

The issue requests a first-class Control UI/onboarding Update Manager that surfaces and orchestrates the native updater with safer rollout feedback.

## Best Possible Solution

Keep this as the Control UI/onboarding discoverability umbrella and implement it in stages: additive updater status/history/release-note/health contracts first, then a dedicated UI surface that edits existing update config and orchestrates the native updater without duplicating it.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence in the product gap: current main has CLI/config/banner primitives and restart-sentinel feedback, but no dedicated manager route or rich updater status contract.

## Solution Assessment

Yes for the product direction, but not as one unsliced patch. The maintainable path is to keep the updater engine in core, expose additive gateway contracts, and then build the Control UI manager on those contracts.
