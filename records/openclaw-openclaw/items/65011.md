## Summary

Keep open. Current main has native updater primitives, auto-update config, a Control UI update banner/action, and restart-sentinel verification, but it still lacks the requested first-class Update Manager surface with rich status, dry-run, release notes, persisted failure history, safe-update controls, and post-update health summary.

## What This Changes

The issue requests a first-class Control UI Update Manager that surfaces existing native updater and auto-update functionality with status, actions, safe-update controls, and operator feedback.

## Best Possible Solution

Keep this as the product umbrella for a staged Control UI Update Manager: add additive updater/status RPCs, expose the existing update config and actions in a dedicated settings or infrastructure surface, persist and display last attempt/success/failure state, link release notes before apply, and coordinate safe-update and health controls with the related updater issues.

## Reproduction Assessment

Not applicable as a runtime bug. The product gap is reproducible by source/docs inspection: current main has updater primitives and a banner/config affordance, while navigation and protocol code have no dedicated manager route or rich updater status contract.

## Solution Assessment

Yes for the product direction, but not as a single unsliced patch. The maintainable path is to surface and orchestrate the existing native updater through additive backend contracts and staged UI work rather than creating a second updater subsystem.
