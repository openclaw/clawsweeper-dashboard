## Summary

Keep open. Current main has native updater primitives, auto-update config, Control UI update execution, skipped/error feedback, and post-restart version verification, but it still does not provide the requested first-class Control UI Update Manager with full status, dry-run, release notes, safe-update controls, persisted attempt history, and health summary. The related open issues cover overlapping rollout and safety pieces rather than superseding this umbrella request.

## What This Changes

Add a first-class Control UI Update Manager that surfaces the existing native updater with discoverable settings, status, safe rollout controls, update actions, release notes, and post-update health feedback.

## Best Possible Solution

Keep this as the product umbrella for a staged Control UI Update Manager. The maintainable end state is a dedicated Settings or Infrastructure surface backed by additive updater/status RPCs, existing config schema writes for auto-update toggles, visible dry-run/update-now outcomes, release-note links, and health/safe-update orchestration coordinated with #12855 and #14526.

## Reproduction Assessment

Not applicable as a runtime bug, but the product gap is verifiable with high confidence: current main has no Update Manager route and the only UI/RPC surfaces are the banner/config Update button plus a narrow restart-sentinel update.status check.

## Solution Assessment

Unclear as a single implementation plan because the issue is an umbrella, but the requested direction is right: surface and orchestrate the native updater rather than creating a second updater. The safest path is to split the work into additive backend status contracts and UI slices before adding backup/rollback health controls.
