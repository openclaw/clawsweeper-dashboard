## Summary

Keep open. Current main has native updater primitives, auto-update config, a config-page Updates section, an update banner/action, and restart-sentinel feedback, but it still lacks the requested first-class Control UI Update Manager with rich status/history, dry-run/progress, release-note links, safe-update controls, and post-update health summary.

## What This Changes

Add a first-class Control UI/onboarding Update Manager that surfaces and orchestrates OpenClaw's native update and auto-update functionality.

## Best Possible Solution

Keep this as the Control UI/onboarding discoverability umbrella and implement it in stages: additive backend status/history/release-note/health contracts first, then a dedicated UI surface that edits existing update config and orchestrates the native updater without duplicating it.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence in the product gap: current main has CLI/config/banner primitives, but no dedicated manager route or rich updater status contract.

## Solution Assessment

Yes for the product direction, but not as one unsliced patch. The maintainable solution is to keep the updater engine in core and expose additive gateway contracts before building the Control UI manager.
