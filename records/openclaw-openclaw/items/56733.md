## Summary

Keep open. Current main has useful diagnostics, readiness detail, Feishu reconnect handling, and adjacent embedded-run abort cleanup, but it still does not provide a supported recovery path for the reported Gateway-alive/all-HTTP-timeout state or systemd watchdog/notify behavior for a frozen event loop.

## What This Changes

The issue reports an overnight WSL2 Gateway freeze where systemd keeps the process active while Feishu, provider, and other HTTP requests silently time out, and asks for event-loop monitoring, watchdog restart support, and less silent idle logging.

## Best Possible Solution

Keep this issue open until maintainers choose and implement a supported recovery contract: either a supervisor-level watchdog/notify path for Linux/WSL services, or a root-cause fix for the Feishu/plugin-runtime/HTTP stall class with regression coverage proving the Gateway recovers.

## Reproduction Assessment

No high-confidence local reproduction path is available. The report has detailed field evidence, but reproducing it appears to require the reporter's WSL2 mirrored-networking, systemd user service, Feishu WebSocket, multiple agents/providers, and overnight idle conditions.

## Solution Assessment

Unclear. The current diagnostics and channel-level fixes are useful, but they are not sufficient proof that the frozen Gateway can self-recover; a watchdog or root-cause fix needs maintainer reliability policy judgment.
