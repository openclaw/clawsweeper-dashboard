## Summary

Keep open: current main and the latest release include useful liveness diagnostics, readiness probes, and per-channel health restarts, but they still do not prove recovery for the reported process-alive gateway state where the event loop stops serving HTTP/channel work. Related open issues overlap with event-loop starvation, but none is a safe exact replacement for the whole-process liveness/supervisor gap.

## What This Changes

The issue requests detection and recovery for a gateway process that remains alive while the Node event loop freezes and all provider/channel HTTP requests time out.

## Best Possible Solution

Define the gateway liveness and supervisor contract, then prove on WSL2/systemd that a process-alive but unresponsive gateway is detected or restarted without unexpected channel/session loss.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact current-main WSL2/systemd overnight freeze. Source inspection does show the remaining recovery gap: liveness and channel monitors are in-process, and the systemd unit lacks an external watchdog contract.

## Solution Assessment

Unclear; current diagnostics and per-channel restarts are useful partial mitigations, but they are not the full solution for a hard-frozen gateway event loop. The maintainable path needs a maintainer decision on whole-process liveness versus narrower root-cause isolation.
