## Summary

Keep open. Current main has diagnostics and readiness detail for event-loop saturation, but it still does not prove recovery for a Gateway whose Node event loop stops serving HTTP while systemd reports the process running, and a newer open memory-core report shows the same starvation class is still active.

## What This Changes

The issue reports a live-but-frozen Gateway on WSL2/systemd where Feishu and LLM HTTP requests time out while the process remains active.

## Best Possible Solution

Keep this open until maintainers either land a root-cause nonblocking fix for the active starvation path or define a supported supervisor watchdog/notify recovery contract with proof that a stalled Gateway reports unhealthy or recovers.

## Reproduction Assessment

No high-confidence current-main reproduction path is available. The report has strong field evidence, but reproducing it appears to require WSL2 mirrored networking, systemd user service supervision, Feishu WebSocket activity, multiple agents/providers, and overnight idle conditions.

## Solution Assessment

Unclear. The diagnostics and readiness improvements are useful, but they are not sufficient proof that a fully frozen Gateway can self-recover; the durable fix likely needs maintainer reliability policy around watchdogs, process isolation, or specific blocking runtime paths.
