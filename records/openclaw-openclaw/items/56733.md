## What This Changes

The issue reports a gateway process that remains alive while the Node.js event loop becomes unresponsive, causing outbound provider and channel HTTP requests to silently time out until manual restart.

## Best Possible Solution

Define the gateway liveness and supervisor contract, then prove on WSL2/systemd that a process-alive but unresponsive gateway is either no longer reachable or is restarted without losing channel/session state unexpectedly.

## Reproduction Assessment

No, not as a live current-main reproduction. Source inspection gives a high-confidence remaining gap in recovery semantics, but the original WSL2/systemd overnight freeze has not been reproduced with current diagnostics enabled.

## Solution Assessment

Unclear; diagnostics and per-channel restarts are useful partial mitigations, but the best fix requires a maintainer decision on whole-process liveness versus root-cause isolation. A safe solution should include live WSL2/systemd proof and upgrade behavior for installed services.
