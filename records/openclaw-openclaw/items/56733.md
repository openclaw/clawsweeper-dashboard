## Summary

Keep open: current main and the latest v2026.6.8 release include useful liveness diagnostics, readiness probes, and channel-level restart loops, but they still do not prove recovery for the reported process-alive gateway state where the event loop stops serving HTTP and channel work. The remaining gap is whole-process liveness or supervisor behavior, so this needs maintainer direction and live WSL2/systemd proof rather than cleanup closure.

## What This Changes

The issue reports a process-alive gateway hang where the Node event loop stops serving provider HTTP and channel work after idle periods, leaving systemd unaware and requiring manual restart.

## Best Possible Solution

Define the gateway liveness and supervisor contract, then prove on WSL2/systemd that a process-alive but unresponsive gateway is detected or restarted without unexpected channel or session loss.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact current-main WSL2/systemd overnight freeze. Source inspection does show the remaining recovery gap: liveness and channel monitors are in-process, and the generated systemd unit lacks an external watchdog contract.

## Solution Assessment

Unclear; current diagnostics and per-channel restarts are useful partial mitigations, but they are not the full solution for a hard-frozen gateway event loop. The maintainable path needs a maintainer decision on whole-process liveness versus narrower root-cause isolation.
