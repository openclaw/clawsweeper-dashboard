## Summary

Keep open: current main and v2026.6.10 still have useful in-process liveness diagnostics, but they do not close the reported process-alive hard-freeze case because HTTP probes, readiness snapshots, maintenance ticks, and event-loop metrics all depend on the Gateway event loop running.

## What This Changes

The issue reports a WSL2/systemd Gateway that stays active as a process while the Node event loop stops serving provider, channel, and health traffic until manual restart.

## Best Possible Solution

Define the Gateway whole-process liveness contract, then land a proven supervisor or isolation fix that detects or recovers a process-alive unresponsive Gateway on WSL2/systemd without unexpected channel or session loss.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact current-main WSL2/systemd overnight freeze. Source inspection does show a remaining recovery gap: all current liveness, health, and readiness paths run inside the Gateway event loop, while generated systemd units still lack watchdog notification.

## Solution Assessment

Unclear. Current diagnostics and per-channel recovery are useful partial mitigations, but the best full fix needs maintainer direction on whole-process supervision versus root-cause isolation, plus live WSL2/systemd proof.
