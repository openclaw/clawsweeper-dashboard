## Summary

Keep open. Current main has stronger diagnostics, readiness detail, Feishu timeout/reconnect handling, and adjacent runtime fixes, but it still does not provide a supported recovery path for a Gateway process whose event loop stops serving HTTP while systemd sees the process as running.

## What This Changes

The issue reports an overnight WSL2/systemd Gateway freeze where the process remains alive but outbound LLM, Feishu, and health-related HTTP work silently times out until manual restart.

## Best Possible Solution

Keep this issue open until maintainers either add a supported supervisor-level watchdog/notify recovery contract for Linux/WSL services or land a root-cause fix with regression proof that a stalled Gateway recovers and probes report the right state.

## Reproduction Assessment

No high-confidence current-main reproduction path is available. The report has detailed field evidence, but reproducing it appears to require WSL2 mirrored networking, a systemd user service, Feishu WebSocket, multiple agents/providers, and overnight idle conditions.

## Solution Assessment

Unclear. The current diagnostics and channel-level mitigations are useful, but they are not sufficient proof that a frozen Gateway can self-recover; a watchdog or root-cause recovery path needs maintainer reliability policy judgment.
