## Summary

Keep open: current main and v2026.6.8 have useful liveness diagnostics, readiness details, HTTP probes, and channel-level restart loops, but those do not close the reported process-alive hard-freeze case because the detection paths still depend on the Gateway event loop running.

## What This Changes

The issue reports a process-alive Gateway freeze where the Node event loop stops serving HTTP/channel work and systemd does not restart it.

## Best Possible Solution

Define the Gateway whole-process liveness and supervisor contract, then prove on WSL2/systemd that a process-alive but unresponsive Gateway is detected or restarted without unexpected channel or session loss.

## Reproduction Assessment

No high-confidence live reproduction exists for the exact current-main WSL2/systemd overnight freeze. Source inspection does show a remaining recovery gap: liveness/readiness monitors run inside the Gateway event loop, and the generated systemd unit lacks an external watchdog-notify contract.

## Solution Assessment

Unclear; current diagnostics and per-channel restarts are useful partial mitigations, but they are not the full solution for a hard-frozen Gateway event loop. The maintainable path needs a maintainer decision on whole-process liveness versus narrower root-cause isolation.
