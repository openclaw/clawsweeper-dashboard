## Summary

Keep this PR open: the reported Codex shared-client race is credible and the isolation direction is narrow, but the patch currently bypasses the existing mocked client-factory seam and is likely to break the Codex app-server test suite before merge.

## What This Changes

This PR routes Codex app-server agent runs to per-run isolated clients when `agents.defaults.maxConcurrent > 1`, adds regression coverage, and adds a changelog fix entry.

## Best Possible Solution

Keep the isolated-client runtime behavior for concurrent Codex agent runs, but preserve an injectable client seam so unit and contract tests can exercise isolated and shared modes without starting real app-server subprocesses.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main defaults `agents.defaults.maxConcurrent` to 4, the shared-client key includes `agentDir`, and a different key clears and closes the existing app-server process. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The isolation approach is the narrow maintainable fix for the runtime race, but the implementation needs to honor the existing app-server client injection seam before it is merge-ready.
