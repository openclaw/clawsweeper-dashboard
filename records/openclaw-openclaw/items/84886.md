## Label Justifications

- `P1`: A beta-blocking Telegram replay window can duplicate agent turns and visible replies for real channel users.
- `impact:message-loss`: The reported failure class is duplicate message dispatch and duplicate visible Telegram replies from one inbound message.
- `impact:session-state`: A replayed inbound message can create multiple agent turns for the same Telegram message identity, drifting conversation state.

## Summary

Keep open: this is protected by the `beta-blocker` label, and current main still shows only update-level Telegram dedupe before `processInboundMessage()` rather than a durable account/chat/message dispatch ledger.

## What This Changes

The issue asks for durable Telegram message dispatch idempotency across isolated polling replay, restart, and recovery paths.

## Best Possible Solution

Add a Telegram-owned claimable persistent dispatch ledger keyed by account id, chat id, and Telegram message id after auth/policy checks and before reply-chain recording or agent dispatch, with commit/release semantics and regression coverage for isolated-spool replay.

## Reproduction Assessment

No high-confidence runtime repro script is present. The source path is clear: isolated spool replay can re-enter `bot.handleUpdate()`, and current Telegram handling lacks a durable message-id dispatch guard before agent dispatch.

## Solution Assessment

Yes, the suggested guard shape is the right maintainable direction. It should live in the Telegram plugin using the existing claimable persistent-dedupe contract rather than changing generic core dispatch policy.
