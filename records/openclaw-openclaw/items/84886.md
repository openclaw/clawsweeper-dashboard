## Summary

Keep open. Current main still lacks a durable Telegram message-level dispatch guard before agent handoff, and the protected `beta-blocker` label requires explicit maintainer handling rather than cleanup closure.

## What This Changes

The issue asks for durable Telegram inbound dispatch idempotency keyed by account, chat, and Telegram `message_id` across isolated polling replay and recovery.

## Best Possible Solution

Add a Telegram plugin-local claimable persistent dispatch guard after auth/policy checks and before reply-chain recording or `processInboundMessage`, with replay/restart coverage for commit and retry release semantics.

## Reproduction Assessment

No deterministic live replay script is available yet, so this is not a high-confidence live reproduction. Source inspection does show the replay-capable isolated polling path and the missing durable guard before `processInboundMessage`.

## Solution Assessment

Yes, the suggested guard is the narrow maintainable fix shape because existing channel contracts already support claim/commit/release dedupe. The implementation should stay inside the Telegram plugin and add regression coverage for replay, in-flight duplicate, successful commit, and retryable pre-dispatch release.
