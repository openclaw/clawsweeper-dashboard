## Summary

Keep open: current main still lacks the requested per-agent model request-timeout field, while existing provider-level timeout support may already be the intended ownership layer, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests a per-agent `model.requestTimeoutSeconds` config option for long-running scheduled `agentTurn` jobs that use sequential model fallback chains.

## Best Possible Solution

Maintainers should decide the timeout ownership first; if per-agent tuning is approved, add it through the canonical agent model schema/resolution path with docs and regression coverage, otherwise document provider timeout plus run/cron timeout as the supported answer.

## Reproduction Assessment

Partly. Source inspection confirms current main has no per-agent request-timeout field and only wires provider-level timeout into model requests; the linked Aegis issue supplies external production timing evidence, but I did not run a live isolated cron reproduction.

## Solution Assessment

Unclear. The per-agent override is plausible, but not proven best because OpenClaw already documents `models.providers.<id>.timeoutSeconds` and related open issues suggest fallback or cron timeout hierarchy may be the deeper fix.
