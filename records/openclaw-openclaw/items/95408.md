## Summary

Keep open: current main and v2026.6.10 still do not support the requested per-agent `model.requestTimeoutSeconds`, and the existing provider-level timeout plus open cron fallback work make this a product/config ownership decision rather than safe cleanup closure.

## What This Changes

The issue requests a per-agent model request-timeout override for complex isolated cron `agentTurn` jobs that use sequential provider fallback chains.

## Best Possible Solution

Maintainers should choose the timeout ownership model, then either add a canonical per-agent model timeout with schema/docs/tests or document provider timeout plus cron/run timeout and land the narrower cron fallback repair.

## Reproduction Assessment

Partly. Source inspection confirms current main lacks the requested per-agent model request-timeout field and only wires provider-level request timeout into model resolution, while the linked Aegis report supplies external production timing evidence rather than a local live repro.

## Solution Assessment

Unclear. A per-agent override is plausible, but existing provider-level timeout support and the open isolated-cron fallback repair path may be the narrower solution for the observed failures.
