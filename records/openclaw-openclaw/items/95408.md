- `P1`: The issue reports production scheduled agent workflows timing out before useful output is produced in a provider fallback chain.
- `impact:message-loss`: A timed-out cron-driven agent turn can fail to deliver the expected scheduled automation result.
- `impact:auth-provider`: The affected surface is model/provider timeout configuration and fallback routing across provider candidates.

## Summary

Keep open: current main and v2026.6.9 still lack the requested per-agent `model.requestTimeoutSeconds` field, while the existing provider-level timeout path covers only part of the reported cron fallback problem and the new timeout hierarchy needs maintainer product direction.

## What This Changes

Request a per-agent `model.requestTimeoutSeconds` config option so complex cron-driven `agentTurn` workflows can raise per-provider request ceilings without changing provider-wide defaults.

## Best Possible Solution

Maintainers should choose the timeout ownership and precedence first; if per-agent tuning is approved, add it through the strict agent model/config schema with docs and focused cron/fallback regression coverage, otherwise document the provider timeout plus run timeout path as the supported answer and keep broader per-model budgets on the canonical timeout tracker.

## Reproduction Assessment

Partly. Source inspection confirms the requested per-agent `model.requestTimeoutSeconds` field is absent and the current timeout path is provider/run scoped; the production cron failure itself was not live-reproduced in this read-only review.

## Solution Assessment

Unclear. A per-agent override is plausible, but maintainers need to decide whether this belongs in per-agent model config, provider config, per-model generation budgets, cron payload timeouts, or documentation before implementation.
