## Summary

Keep open: current main and the latest release still lack the requested per-agent `model.requestTimeoutSeconds` field, while the existing supported timeout owner is provider-level `models.providers.<id>.timeoutSeconds`; the remaining question is a maintainer product decision about whether to add another config surface for this cron/fallback workload.

## What This Changes

Request a per-agent model request timeout option for complex scheduled agent turns that use provider fallback chains.

## Best Possible Solution

Maintainers should first decide the timeout ownership model; if per-agent tuning is approved, implement it through the canonical agent model schema, resolver, docs, and regression coverage, otherwise document provider timeout plus run/cron timeout ceilings as the supported answer.

## Solution Assessment
