## Summary

Keep open: current main and the latest release still do not expose the requested per-agent `model.requestTimeoutSeconds` field. Adjacent provider-level and cron timeout fixes cover part of the failure family, but this specific config surface and precedence model still need maintainer product direction.

## What This Changes

The issue requests a per-agent model request-timeout option for complex isolated cron `agentTurn` jobs that use sequential provider/model fallback chains.

## Best Possible Solution

Maintainers should choose the canonical timeout scope and precedence; if per-agent tuning is approved, add it through the agent model schema/resolution path with docs and regression coverage, otherwise document provider timeout plus cron/run timeout as the supported answer.

## Reproduction Assessment

Partly. Source inspection confirms the requested per-agent model timeout field is absent and would be rejected, while the linked Aegis issue supplies external production timing evidence; I did not run a live isolated cron reproduction against current main.

## Solution Assessment

Unclear. The per-agent override is plausible, but maintainers need to decide whether timeout ownership belongs at agent, provider, per-model, cron payload, or a combination before this becomes a safe implementation.
