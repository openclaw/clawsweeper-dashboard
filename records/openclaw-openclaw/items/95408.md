## Summary

Keep open: current main and v2026.6.10 still lack the requested per-agent `model.requestTimeoutSeconds` surface, while the supported provider-level timeout path is only an adjacent workaround; adding a new timeout ownership layer needs maintainer product direction.

## What This Changes

The issue requests a per-agent `model.requestTimeoutSeconds` config option so complex cron-driven `agentTurn` workloads can use a longer per-provider timeout while walking a sequential model fallback chain.

## Best Possible Solution

Maintainers should choose the timeout ownership and precedence model first; if per-agent tuning is approved, add it through the canonical agent model schema/types/docs/resolution path with focused tests while preserving provider-level timeout and run-timeout ceilings.

## Reproduction Assessment

Partly. Source inspection confirms the requested per-agent `requestTimeoutSeconds` key is absent and that current timeout wiring is provider-level; the linked Aegis issue supplies production timing evidence, but I did not run a live isolated-cron reproduction on current main.

## Solution Assessment

Unclear. The per-agent override is plausible, but the safer path is to decide timeout ownership first because current docs, tests, and runtime already support provider-level timeout and related open issues cover broader timeout hierarchy work.
