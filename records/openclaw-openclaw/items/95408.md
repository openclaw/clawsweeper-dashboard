## Summary

Keep open: current main and v2026.6.9 still lack the requested per-agent `model.requestTimeoutSeconds` field, while the existing supported timeout knob is provider-level, so this remains a product/config precedence decision rather than a cleanup close.

## What This Changes

Request to add a per-agent `model.requestTimeoutSeconds` config override for model fallback chains so complex isolated cron `agentTurn` payloads can use a longer per-provider request ceiling.

## Best Possible Solution

Maintainers should choose the timeout ownership and precedence model first; if approved, add the canonical schema/types/docs/tests and runtime propagation in that chosen layer, otherwise document the provider/run timeout path or route the need through the broader per-model timeout tracker.

## Reproduction Assessment

Partly. Source inspection confirms current main rejects the requested per-agent model timeout field and only provider-level timeout feeds model request metadata; the linked Aegis issue supplies production timing evidence, but I did not run a live isolated cron reproduction.

## Solution Assessment

Unclear. The per-agent override is plausible, but not proven best because OpenClaw already supports provider-level request timeouts and related open issues cover per-model generation budgets and cron fallback timeout hierarchy.
