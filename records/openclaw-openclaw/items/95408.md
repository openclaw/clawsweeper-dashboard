## Label Justifications

- `impact:message-loss`: A timed-out cron-driven agent turn can fail to produce the expected scheduled automation output.

## Summary

Keep open: current main and the latest release still lack the requested per-agent model request-timeout field, while existing provider-level timeout support may already be the intended ownership layer. This needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

Request to add a per-agent `model.requestTimeoutSeconds` config option for long-running cron-driven agent turns that use sequential model fallback.

## Best Possible Solution

Maintainers should decide timeout ownership first; if per-agent tuning is approved, add it through the canonical agent schema/resolution path with docs and regression coverage, otherwise document provider timeout plus run/cron timeout as the supported answer.

## Reproduction Assessment

Partly. Source inspection confirms current main has no per-agent request-timeout field and wires request timeout through provider config; the linked Aegis issue supplies external production timing evidence, but I did not run a live isolated cron reproduction.

## Solution Assessment

Unclear. The per-agent override is plausible, but not proven best because OpenClaw already documents `models.providers.<id>.timeoutSeconds` and related timeout/fallback behavior may need a broader ownership decision.
