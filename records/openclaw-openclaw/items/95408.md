## Label Justifications

- `impact:message-loss`: A timed-out cron-driven agent turn can fail to produce the expected scheduled automation output.

## Summary

Keep open: current main and v2026.6.10 still do not implement per-agent `model.requestTimeoutSeconds`; the existing provider-level timeout path is real but leaves the requested timeout ownership and precedence as a maintainer product decision.

## What This Changes

The issue requests a per-agent `model.requestTimeoutSeconds` config option for complex isolated cron `agentTurn` jobs that use model fallback chains.

## Best Possible Solution

Maintainers should choose the timeout ownership model, then either document provider/run timeout as the supported answer or add a typed per-agent/per-model timeout contract with schema, runtime, docs, and regression coverage.

## Reproduction Assessment

Partly. Source inspection confirms current main rejects per-agent model timeout fields and only wires provider-level request timeout; the linked Aegis report supplies external production timing, but I did not run a live isolated cron reproduction.

## Solution Assessment

Unclear. The per-agent override is plausible, but the best fix depends on maintainer choice because OpenClaw already supports provider-level request timeout and has an open related per-model generation-timeout discussion.
