## Summary

Keep open: current main and v2026.6.9 still lack the requested per-agent model request-timeout field, while the existing supported timeout is provider-level and the linked production report shows a real cron/fallback pain point that needs maintainer timeout-policy direction.

## What This Changes

Request a new per-agent `model.requestTimeoutSeconds` config option so complex scheduled agent turns can raise the per-provider request ceiling without changing global defaults.

## Best Possible Solution

Maintainers should first choose the canonical timeout ownership and precedence; if per-agent tuning is approved, add it through the strict agent model schema, runtime model resolution, docs, doctor/config metadata as needed, and regression plus live cron proof.

## Reproduction Assessment

Partly, at source level: current main clearly lacks the requested per-agent request-timeout field and only wires provider-level timeout into model requests. The linked Aegis issue gives production timing evidence, but I did not run a live isolated cron reproduction.

## Solution Assessment

Unclear. The per-agent override is plausible, but the best fix depends on maintainer policy for provider timeout, whole-run timeout, per-model generation budget, and fallback-on-timeout semantics.
