## Summary

Keep open: current main and v2026.6.10 still do not expose the requested per-agent model request-timeout field; existing provider-level timeouts may be the intended layer, so maintainers need product direction rather than cleanup closure.

## What This Changes

Requests a per-agent `model.requestTimeoutSeconds` config field so complex isolated cron `agentTurn` fallback chains can give slow provider attempts a longer per-request ceiling.

## Best Possible Solution

Maintainers should choose the timeout ownership model, then either document provider/per-model timeout as the supported answer or add a typed per-agent request-timeout field with clear precedence, docs, and regression coverage.

## Reproduction Assessment

No for the full production cron failure: I did not run a live Aegis-style isolated cron payload on current main. Yes for the missing config contract: source inspection confirms the requested per-agent model request-timeout field is absent and provider-level timeout is the current wired path.

## Solution Assessment

Unclear. A per-agent override is plausible, but the best solution depends on maintainer choice between provider-level timeout, the broader per-model generation-timeout contract, and a new per-agent model timeout surface.
