## Summary

Keep open: this PR targets the right existing `session.message` path for the linked TUI live transcript bug, but it should not merge yet because the new live reload can still apply stale history after a session switch, and the external PR lacks real Gateway-to-TUI behavior proof.

## What This Changes

The PR adds a TUI handler for Gateway `session.message` events, reloads selected-session history on live transcript updates, and adds handler tests for reload coalescing and active-run deferral.

## Best Possible Solution

Land a narrow TUI `session.message` fix that guards both queued and in-flight history reloads by selected session/agent and includes redacted live TUI proof for the linked bug.

## Reproduction Assessment

Yes at source level: Gateway emits displayable transcript appends as `session.message`, while current main's TUI dispatcher does not handle that event. I did not run a live terminal reproduction in this read-only review.

## Solution Assessment

No as currently patched: consuming the existing Gateway event is the right owner boundary, but the implementation needs a stale-result guard for in-flight history reloads before it is the best fix.

- **[P2] Guard in-flight live history reloads:** `src/tui/tui-event-handlers.ts:151`
  - body: `runQueuedHistoryReload` verifies the selected session before starting `loadHistory()`, but `loadHistory` then awaits the gateway and later clears/rebuilds the chat log without rechecking that the selected session or agent is still the same.
