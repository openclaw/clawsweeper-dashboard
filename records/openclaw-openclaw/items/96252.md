## Summary

Keep open: the PR targets a real source-reproducible TUI live transcript gap, but the latest patch still has an in-flight session reload race and lacks real Gateway-to-TUI behavior proof.

## What This Changes

The PR adds TUI handling for Gateway `session.message` events so the selected session can refresh history on live transcript updates from another surface.

## Best Possible Solution

Land a narrow TUI `session.message` consumer only after stale in-flight reload results are guarded by selected session/agent and redacted real behavior proof shows the live TUI path working.

## Reproduction Assessment

Yes at source level: Gateway emits displayable transcript appends as `session.message`, while current main's TUI subscribes to session events but does not dispatch that event. I did not run a live terminal reproduction in this read-only review.

## Solution Assessment

No as currently patched: consuming the existing Gateway event is the right layer, but the implementation needs an in-flight stale-result guard and real Gateway-to-TUI proof before it is the best fix.
