## Summary

Keep open: current main and v2026.6.10 still fire-and-forget cron failure alerts, while this PR is the linked candidate fix with sufficient real behavior proof; the remaining maintainer decision is whether the bounded finalization wait is acceptable.

## What This Changes

The PR makes cron failure-alert senders return delivery outcomes, waits up to 10 seconds for per-job failure-alert delivery, persists the resulting status before finished events and run logs, and adds focused cron/gateway coverage.

## Best Possible Solution

Land the status fix if maintainers accept the bounded wait, or replace it with post-finish status persistence if cron finished events must not wait on alert transport.

## Reproduction Assessment

Yes, source-level. Current main computes failure-notification status before the async sendCronFailureAlert path can return and fire-and-forgets that send, matching the linked production report.

## Solution Assessment

Yes, if maintainers accept the bounded wait. The safer alternative is a post-finish persistence update that records the alert outcome without delaying finished events.
