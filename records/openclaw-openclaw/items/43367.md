## Summary

Keep open. Current main has mitigations for OAuth refresh contention, session-store hot paths, and subagent/task recovery, but concurrent `agents add` still has a source-reproducible stale-config conflict path with no add-level retry, and the remaining orchestration/cancellation slices are not fully shipped.

## What This Changes

Request stabilization of multi-agent orchestration across concurrent agent creation, concurrent runs/session locks, OAuth failover races, and detached child-work visibility/cancellation.

## Best Possible Solution

Keep this umbrella open until the config add concurrency fix lands, then split or close the remaining session-lock, failover, and detached-work pieces against narrower canonical work.

## Reproduction Assessment

Yes for the concurrent `agents add` conflict path from source inspection: racing adds that share a base hash can surface `ConfigMutationConflictError` without retrying or merging. The broader detached-work and session-death symptoms have strong reports, but not a single deterministic current-main reproduction harness here.

## Solution Assessment

No, current main is not yet the complete solution. The narrow maintainable path is to review or land the focused config retry/serialization work, then separately verify the remaining transcript-lock and detached-task behavior on current main.
