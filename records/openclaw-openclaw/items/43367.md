## Summary

Keep open. Current main has mitigations for OAuth refresh and session lock cleanup, but the concurrent `agents add` path still lacks retry/serialization around config conflicts, model fallback can still proceed after unclassified local errors, and open partial fix PRs are already tied to this issue.

## What This Changes

The issue asks OpenClaw to make multi-agent orchestration reliable across concurrent agent creation, isolated parallel runs, local lock failures, provider refresh races, and detached child work.

## Best Possible Solution

Keep this as the umbrella until maintainers either land the linked config-race fix and split the remaining lock/failover/orphan work into canonical issues, or land an end-to-end orchestration stability change with focused regression coverage.

## Reproduction Assessment

No, not for the full umbrella scenario. There is a strong source-level partial reproduction for concurrent `agents add` conflicts propagating without retry, but I did not establish the full detached-work/session-death spiral on current main.

## Solution Assessment

No, closing is not the best way to solve the issue. The narrow config-race PRs should be reviewed on their own, while the remaining session lock, fallback short-circuit, and child-work lifecycle pieces need maintainer scoping or separate fixes.
