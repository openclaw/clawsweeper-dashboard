## Summary

Keep open. Current main still only partially addresses the reported multi-agent orchestration failure cluster: OAuth refresh and some session-lock behavior have mitigations, but the concurrent agent config mutation path is still paired with an open implementation PR, and detached/subagent completion handling remains active in related open work.

## What This Changes

Stabilize multi-agent orchestration by making concurrent agent creation safe, preventing isolated parallel runs from tripping session/auth failover races, and giving operators clear control over detached child work.

## Best Possible Solution

Keep the issue open as the umbrella until maintainers review or replace #44997 for serialized config mutation, then explicitly track the remaining runtime pieces: session/run queue semantics, local lock/failover boundaries, provider auth noise boundaries, CLI-owned child-process cancellation/visibility, and subagent completion recovery with #44925 and #45200.

## Reproduction Assessment

Yes for the config-write portion: parallel non-interactive `openclaw agents add` calls against the same config exercise the inspected read-modify-write path. The detached child-work and multi-agent idle-race symptoms have strong production reports but still need a deterministic maintainer reproduction harness.

## Solution Assessment

No, current main is not yet the best complete solution. The narrowest path is to land or replace the focused config-lock PR, then handle subagent delivery/cancellation and run-queue behavior as separate, tested changes rather than closing the umbrella from partial mitigations.
