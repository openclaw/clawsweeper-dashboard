## Summary

Keep open. Current main still only partially addresses the reported multi-agent orchestration cluster: OAuth refresh has a real mitigation, but concurrent agent config mutation is still paired with unmerged #44997, session-store locking remains a shared `sessions.json` bottleneck, and subagent/detached-work handling is still covered by open follow-up work and fresh production reports.

## What This Changes

The issue requests stable multi-agent orchestration across concurrent agent creation, parallel agent runs, model failover, and cleanup or visibility for detached child work.

## Best Possible Solution

Keep this as the umbrella until maintainers land or replace #44997 for serialized config mutation, then split the remaining runtime work into focused fixes for session-store contention, run-queue semantics, failover boundaries, cancellation/visibility, and subagent completion recovery with #44925 and #45200 preserved as linked context.

## Reproduction Assessment

Partially yes: parallel non-interactive `openclaw agents add` calls against one config exercise the inspected stale-snapshot write path, and the latest comment gives a current-release session-lock reproduction shape. The detached child-work and idle-race pieces still need a deterministic maintainer harness.

## Solution Assessment

No, current main is not the complete solution. The narrowest maintainable path is to land or replace the focused config-lock PR first, then handle session scheduling, cancellation, and subagent delivery as separate tested runtime changes.
