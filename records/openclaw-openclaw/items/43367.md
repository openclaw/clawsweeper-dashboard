## Summary

Keep open. Current main only partially addresses the reported orchestration cluster: OAuth refresh races have a real mitigation, but concurrent agent creation is still not a serialized transaction, session metadata still funnels through a 10s monolithic store lock, and detached/subagent completion handling remains active in related open work.

## What This Changes

The issue requests reliable multi-agent orchestration across concurrent agent creation, parallel agent runs, provider failover, and detached child-work cleanup/visibility.

## Best Possible Solution

Keep this as the umbrella until the config-write, session-store contention, failover boundary, cancellation/visibility, and subagent completion-loss slices are either shipped or split into canonical narrower issues.

## Reproduction Assessment

Yes for the config-write and session-store-lock portions: the provided parallel `openclaw agents add` and concurrent agent/slash-command reports map directly to inspected current-main paths. The detached child-work and idle-race symptoms are credible but still need a deterministic maintainer harness.

## Solution Assessment

No, current main is not a complete solution. The narrow maintainable path is to land or replace the focused config and subagent PRs, then separately address session-store concurrency and CLI/gateway cancellation semantics.
