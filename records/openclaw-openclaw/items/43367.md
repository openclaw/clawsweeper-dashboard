## Summary

Keep open: current main still has a source-visible `agents add` config mutation race and the remaining orchestration failures are only partially addressed by separate merged or open work.

## What This Changes

The issue asks OpenClaw to make parallel agent creation and multi-agent execution reliable across config writes, session locks, OAuth failover noise, and detached child work.

## Best Possible Solution

Keep the issue open as the umbrella until maintainers land or reject the config-race PR, route session-store contention through the directory-store work, and keep subagent delivery/detached-work semantics tracked by narrower reliability issues.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the config overwrite race is source-reproducible on current main via stale `baseHash` conflict propagation, and recent comments give concrete session-lock/subagent reports. I do not have a single high-confidence live reproduction for the whole multi-agent cluster in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No single umbrella patch is the best path. The maintainable route is a sequence of narrow fixes plus maintainer decisions for durable session delivery, cancellation, and store architecture.
