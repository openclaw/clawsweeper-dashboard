## Summary

Keep open. Current main fixes part of the original config-add race and has auth/failover hardening, but the central multi-agent reliability problem remains: current source and the latest release still contain bounded announce give-up, requester wake rejection states, and several open narrower reports/PRs for delivery, session-lock, and recovery semantics.

## What This Changes

The issue requests reliable parallel multi-agent orchestration across concurrent agent creation, session locking, provider auth failover, and detached child-work cleanup.

## Best Possible Solution

Keep this as the umbrella until maintainers choose and prove a durable completion-delivery contract: concurrent agent creation stays transactional, requester wake or fallback delivery is exactly-once and user-visible, session locks do not strand parent lanes, provider auth errors do not mask local lock failures, and detached work has auditable cancel/recovery semantics.
