## Summary

Keep open: current main has addressed important config-mutation and OAuth-refresh slices, but the broader supported multi-agent workflow is still not proven stable. Source and live discussion still point to remaining session-state contention, finite subagent delivery give-up, timeout/cancellation semantics, and open related implementation work.

## What This Changes

The issue asks OpenClaw to make parallel multi-agent creation and execution reliable by preventing config overwrite races, session-lock failures, failover auth noise, and detached child work.

## Best Possible Solution

Keep this as a P1 umbrella and sequence focused owner-reviewed changes for session-state contention, durable subagent completion delivery, and child-run cancellation or recovery semantics.

## Reproduction Assessment

No, not for the full end-to-end multi-agent cluster; I did not run a live current-main stress setup. Source inspection and recent comments give medium-confidence reproduction for remaining slices such as whole-store session writes, finite subagent announce give-up, timeout finalization, and the open related PR at https://github.com/openclaw/openclaw/pull/45200.

## Solution Assessment

No single umbrella patch is the best way to solve this issue. The maintainable path is to keep the landed config and OAuth fixes, then split or sequence owner-reviewed work for storage contention, durable async completion delivery, and cancellation/recovery policy.
