## Summary

Keep open. Current main has real mitigations for OAuth refresh races, session-store hot-path writes, and task/cancel visibility, but the concurrent `agents add` path still has no retry/serialization around stale config snapshots, sub-agent announce is still documented as best-effort across gateway restarts, and open PRs still track partial fixes for this issue.

## What This Changes

The issue requests reliable multi-agent orchestration across concurrent agent creation, isolated parallel runs, provider failover, and cleanup/visibility for detached child work.

## Best Possible Solution

Keep this as the umbrella until the open config/subagent PRs are reviewed or replaced, then split any remaining scheduling, cancellation, and announce-durability gaps into narrower tracked bugs with focused repros.

## Reproduction Assessment

Yes for the config-add conflict slice: current source still reads one base config hash and propagates a mutation conflict without retry, matching the new reproduction comment. The full detached-work and idle-race cluster is source-plausible but not deterministically reproduced here end to end.

## Solution Assessment

No, current main is not a complete solution. The maintainable path is to land or replace the focused config retry/serialization PR, validate the session-store and transcript-lock mitigations under multi-agent load, and track subagent delivery/cancellation durability separately.
