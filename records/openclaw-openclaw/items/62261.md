## Summary

Keep this open. Current main still has the updater result-accounting path that can report an earlier failed dev-preflight lint candidate as the final update failure after a later candidate passed and the final update steps succeeded; related #63627 explains the lint/OOM pressure but does not supersede this updater bug.

## What This Changes

Dev-channel updates should succeed after preflight skips failing candidate commits and finds a later commit that passes build and lint.

## Best Possible Solution

Keep the dev preflight safety gate, but make failed candidate steps non-blocking once a later SHA has passed preflight and add a regression test plus changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can mock candidate A returning a failed or null-code `pnpm lint`, candidate B passing install/build/lint, and final rebase/build/doctor succeeding; current main will still classify candidate A's lint step as blocking.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/infra/update-runner.ts` result accounting, not in removing lint or changing the dev-channel preflight policy.
