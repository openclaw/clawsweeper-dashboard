## Summary

Keep this open. Current main still has a source-level path that matches the report: dev-channel preflight can try older candidate commits after a lint SIGKILL, but the final git update result still scans all recorded steps and treats an earlier failed preflight candidate as a blocking failure even after a later candidate passes and the update continues. The related #63627 comment is useful context for the lint/OOM side, but it does not supersede this update-runner result-accounting bug.

## What This Changes

The issue asks for `openclaw update --channel dev` to succeed instead of ending with a preflight lint error after switching to the dev channel on Linux.

## Best Possible Solution

Make dev preflight candidate failures non-blocking once a later candidate passes and the final update steps succeed, while preserving a hard `preflight-no-good-commit` failure when no candidate passes. Keep lint/build as the safety gate unless maintainers separately choose a lighter preflight policy for low-memory machines.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can mock `runGatewayUpdate` with candidate A returning `code: null` or non-zero for `pnpm lint`, candidate B passing lint/build, and final rebase/build/doctor succeeding; current main will still classify the earlier candidate failure as blocking through `findBlockingGitFailure`.

## Solution Assessment

Yes, the narrowest maintainable fix is in `src/infra/update-runner.ts`: exclude superseded dev-preflight candidate failures from the final blocking-step scan after a selected SHA is found. Removing lint entirely would be a broader product/safety change and is not necessary to fix this report.

## Repair Work Prompt

Fix the dev-channel updater bug from https://github.com/openclaw/openclaw/issues/62261. Current behavior: when `openclaw update --channel dev` tries multiple candidate commits, an earlier candidate can fail `preflight lint` with a non-zero/null exit, a later candidate can pass preflight and the update can continue through rebase/build/doctor, but the final result still reports failure because `findBlockingGitFailure(steps)` scans superseded preflight candidate failures. Keep the preflight safety model: do not remove lint/build and do not weaken `preflight-no-good-commit` when no candidate passes. Implement a narrow fix in `src/infra/update-runner.ts` so superseded dev-preflight candidate failures do not block once a selected SHA has passed and later update steps succeed. Add/adjust tests in `src/infra/update-runner.test.ts` for at least: first candidate lint exits null or non-zero, second candidate passes, final result is ok/rebases the second SHA; all candidates failing still returns the existing preflight failure path. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @tturnerdev` if this repo's active changelog section requires it. Do not change package-manager resolution, global update behavior, or unrelated lint memory policy in this PR.
