## Summary

Keep open. Current main still records failed dev-preflight lint candidates in the same step list that decides the final updater status, so a later successful candidate can still finish the update path and be reported as an error; related #63627 explains lint memory pressure but does not supersede this accounting bug.

## What This Changes

Fix dev-channel update result accounting so failed preflight attempts for superseded candidate commits do not fail the completed update after a later candidate passes.

## Best Possible Solution

Keep the dev preflight safety gate, but make failed candidate checks nonblocking once a later SHA has passed preflight, with a focused regression test and changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can mock candidate A returning a null-code or nonzero `pnpm lint`, candidate B passing install/build/lint, and final rebase/build/doctor succeeding; current main will still classify candidate A's lint step as blocking.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/infra/update-runner.ts` result accounting, not in removing lint or changing the dev-channel preflight policy.

## Repair Work Prompt

Fix the dev-channel updater bug reported in https://github.com/openclaw/openclaw/issues/62261. In `src/infra/update-runner.ts`, preserve the existing dev preflight install/build/lint safety gate, but make failures from superseded candidate commits nonblocking once a later candidate SHA has passed preflight and the final update steps succeed. Do not remove `pnpm lint`, relax the no-good-commit behavior, or change the separate lint/OOM policy tracked around #63627. Add a regression test in `src/infra/update-runner.test.ts` that mocks at least two dev candidates: the first has a failed or null-code `pnpm lint`, the second passes install/build/lint, and final rebase/install/build/ui/doctor/verify succeeds; the expected update result should be ok and should not report the first candidate's lint step as the final failure. Add a user-facing `CHANGELOG.md` fix entry for the updater behavior.
