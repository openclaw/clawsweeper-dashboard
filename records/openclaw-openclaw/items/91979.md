- `impact:data-loss`: The report concerns a silently missing persisted run-history record for a completed cron execution.

## Summary

Current main contains a credible persistence gap matching the report: scheduler state and finished-run history are separate writes, and the history write is fire-and-forget, so a failure can leave a completed job with updated last-run metadata but no history row. No provided related item owns this specific defect, so it should remain open for a focused core fix.

## What This Changes

Investigate why one completed isolated agent-turn cron job updated its last-run metadata but produced no entry in cron run history.

## Best Possible Solution

Persist the terminal history row within the canonical awaited cron completion boundary, coupled with job-state persistence, so a completed attempt cannot advance lastRunAtMs without a queryable history record.

## Reproduction Assessment

Yes at source level: the scheduler persists lastRunAtMs independently, while the gateway performs a fire-and-forget history append whose failure cannot prevent or roll back the state update. The reporter's exact failure trigger has not been recreated in a live gateway.

## Solution Assessment

Yes; enforcing one terminal persistence invariant at the cron owner boundary is narrower and more maintainable than documentation, retries, or a parallel fallback store.
