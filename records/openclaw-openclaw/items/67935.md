## Summary

Keep open. Current main still has the reported Control UI limitation: chat message group width is hard-coded in bundled CSS, there is no typed or validated gateway config field for chat width, and the bootstrap/UI state path carries no layout value. The related #72772 is broader ultrawide layout work, so it should not replace this narrower, actionable chat-width request.

## What This Changes

The issue asks for an upgrade-safe Control UI configuration or appearance setting to customize chat message maximum width instead of re-patching bundled CSS after every OpenClaw update.

## Best Possible Solution

Add a constrained Control UI chat-width setting that preserves the current `min(900px, 68%)` fallback, flows through the chosen typed setting surface into bootstrap or browser-local UI state, sets a CSS custom property on the shell/root, updates grouped chat CSS to consume it, and documents the supported value format.

## Reproduction Assessment

Yes. The issue is reproducible as a source-level capability gap on current main: the CSS is hard-coded and the config/schema/bootstrap/UI settings paths do not expose any supported chat-width override.

## Solution Assessment

Yes, if kept narrow. A validated width setting or browser-local Appearance control is the maintainable path; arbitrary custom CSS injection would be a broader and riskier solution than this issue requires.
