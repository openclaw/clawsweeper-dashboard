## Summary

Keep open. Current main still has the reported Control UI limitation: the chat message width is hard-coded in CSS, and the typed config, strict schema, bootstrap payload, UI settings, and public docs do not expose an upgrade-safe chat-width override. The related ultrawide layout issue is broader and should not replace this narrower request.

## What This Changes

Allow users to configure the Control UI chat message maximum width through OpenClaw configuration instead of patching bundled CSS after each upgrade.

## Best Possible Solution

Add a constrained Control UI chat layout setting that preserves the current `min(900px, 68%)` fallback, validates an allowed CSS width expression, flows through typed config/schema/help into the bootstrap JSON and UI state, sets a root CSS custom property, updates grouped chat CSS to consume it, and documents the supported format.

## Reproduction Assessment

Yes. The issue is reproducible as a source-level capability gap on current main: the CSS is hard-coded, the strict config schema rejects an equivalent setting, and no bootstrap/UI state path carries a layout width value.

## Solution Assessment

Yes, if kept narrow. A typed, validated width setting feeding a CSS custom property is more maintainable than post-upgrade bundle patching or arbitrary custom CSS injection.
