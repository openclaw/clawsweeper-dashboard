- `P1`: The PR addresses API-key exposure but currently risks breaking existing provider auth on upgrade.
- `merge-risk: 🚨 compatibility`: The diff changes the persisted models.json contract by deleting apiKey fields that current docs say may be used for agent provider auth.

## Summary

Keep open: the security goal is valid and the provided live-output proof is useful, but the patch strips credentials after the merge step that currently preserves documented agent models.json auth sources, so it can break existing custom-provider setups on upgrade.

## What This Changes

The PR adds a recursive models.json payload scrubber that removes apiKey keys and updates agent model-config tests to expect no apiKey in generated provider entries.

## Best Possible Solution

Strip prompt-facing generated apiKey data while preserving runtime auth through a tested migration or compatibility path for existing agent models.json credentials.

## Reproduction Assessment

Yes, source-level: current main serializes finalProviders into models.json, and the PR body includes live output showing the new strip path removes fake apiKey values. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written: stripping every apiKey after merge is too broad because it erases the documented existing models.json auth source. The safer solution needs a tested migration or compatibility path while still preventing prompt-facing secret persistence.
