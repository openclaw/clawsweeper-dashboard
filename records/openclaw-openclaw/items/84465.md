## Summary

Keep open: the security goal is valid, but the branch is not merge-ready because it strips a field that current generated-catalog validation, runtime auth lookup, and public docs still treat as part of the provider auth contract.

## What This Changes

The PR recursively removes `apiKey` fields from generated `models.json` payloads and updates agent models-config tests to expect provider and model keys to be omitted.

## Best Possible Solution

Land one canonical current-main sanitizer that removes prompt-visible plaintext provider secrets while preserving documented non-secret markers and migrating or explicitly retiring existing plaintext `models.json` credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes merged provider rows including documented `apiKey` markers, while the PR removes that field immediately before serialization.

## Solution Assessment

No. Serialization-boundary redaction is the right layer, but deleting every `apiKey` is broader than the current auth contract allows; the safer path is marker-aware redaction plus explicit upgrade policy.
