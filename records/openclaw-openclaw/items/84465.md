## Summary

Keep open: the security goal is valid, but this branch deletes `apiKey` fields that current merge/auth contracts and Pi model-registry compatibility still rely on for custom providers.

## What This Changes

This PR adds a recursive sanitizer that removes all `apiKey` fields from generated `models.json` providers before writing and updates model-config tests to expect no `apiKey` fields.

## Best Possible Solution

Use a provenance-aware sanitizer that keeps required non-secret `apiKey` markers and preserves or migrates existing `models.json`-owned credentials while still removing resolved plaintext secrets from newly generated output.

## Reproduction Assessment

Yes. Source inspection shows current main serializes provider data with apiKey values, and the PR defect is source-reproducible because the new helper strips every `apiKey` after merge-mode preservation.

## Solution Assessment

No. Deleting all `apiKey` fields is not the best fix because OpenClaw currently uses marker values and preserved merge-mode keys as compatibility contracts; the safer fix should scrub resolved plaintext without removing required markers or unmigrated existing credentials.
