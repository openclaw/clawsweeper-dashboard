## Summary

Keep open. The security goal is valid and the supplied proof shows `apiKey` omission, but the patch is incorrect because it removes every generated provider `apiKey` after the planner has already used that field for writability, so custom/provider-owned catalogs can become invalid or lose request auth; the branch is also conflicting with current main's root-plus-plugin catalog writer.

## What This Changes

The PR adds a recursive serialization helper that removes every `apiKey` field from generated `models.json` payloads and updates model-config tests to expect the field to be absent.

## Best Possible Solution

Land one maintainer-selected marker-aware redaction and migration implementation that removes plaintext secrets from every generated model catalog while preserving provider availability and upgrade credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: generate a custom provider with models and `apiKey`, strip the field at serialization, then load it through the model registry path that rejects model-bearing providers without `apiKey`.

## Solution Assessment

No. Serialization-boundary hardening is the right area, but deleting every `apiKey` is too broad; the safer fix is marker-aware plaintext redaction plus explicit migration or preservation for shipped upgrade credentials.
