## Summary

Keep open: current main and the latest release still exclude WhatsApp media from inbound debouncing and build only one media fact per inbound turn. The request fits the WhatsApp plugin surface, but it needs a product/config decision rather than strict bug-fix automation.

## What This Changes

The issue requests WhatsApp same-sender media/file burst coalescing so several rapid uploads become one agent turn with all media preserved.

## Best Possible Solution

Implement this inside the WhatsApp plugin with an explicit documented batching configuration, multi-media inbound facts, and regression plus live-channel proof that text debounce and Telegram album behavior remain unchanged.

## Reproduction Assessment

Yes by source inspection: current main routes media messages around the debounce predicate and buildWhatsAppInboundContext only supplies one media entry. Live WhatsApp proof is still needed before shipping a fix.

## Solution Assessment

Unclear until maintainers choose the config/default semantics. The likely maintainable path is plugin-owned WhatsApp media batching that reuses the existing multi-media inbound payload helpers rather than changing core first.
