## Summary

Keep open. The report describes a real cron classification problem, but this draft PR should not merge as written because it lets any payload text, including an earlier or quoted token, suppress a later fatal tool error while current main has newer failure and denial handling that an accepted design must preserve.

## What This Changes

The PR adds `===DONE_OK===` sentinel matching to cron payload outcome resolution and tests that final or payload text can make trailing error payloads non-fatal unless a run-level error exists.

## Best Possible Solution

Define a narrow cron success signal with explicit trust and precedence rules, rebase it onto the current resolver, preserve structured failure/denial behavior, and add targeted tests plus changelog/docs if the contract is accepted.

## Reproduction Assessment

Yes. A direct `resolveCronPayloadOutcome` call with a non-error result followed by a final `isError: true` payload reproduces current-main fatal classification, and the existing resolver test covers that behavior.

## Solution Assessment

No, not as written. A success sentinel may be a workable direction, but scanning all payload text is too broad; a trusted final signal or structured metadata would be safer.
