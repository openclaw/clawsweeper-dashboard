## Summary

Keep open. Current main still does not implement a `DONE_OK` cron success sentinel, and the PR targets a real cron false-error path, but the draft patch is not merge-ready because it trusts sentinel text from any payload and can hide later real failures.

## What This Changes

The PR adds `===DONE_OK===` detection to the isolated cron payload resolver and tests so final or payload sentinel text can make trailing error payloads non-fatal unless a run-level error exists.

## Best Possible Solution

Keep the draft open for maintainer review. If the success-sentinel direction is accepted, define the trusted success signal narrowly, preserve run-level/failure/denial precedence, add regression coverage for earlier or quoted sentinel text followed by a real failure, and document/changelog the cron contract.

## Reproduction Assessment

Yes. A focused call to `resolveCronPayloadOutcome` with a successful text payload followed by a final `isError: true` payload reproduces current-main fatal classification, and `run.ts` persists that fatal flag as `status: "error"`.

## Solution Assessment

No, not as written. A success sentinel may be a reasonable direction, but scanning every payload text is too broad; the safer implementation should trust only an intentional final success signal or equivalent structured metadata after maintainer agreement.
