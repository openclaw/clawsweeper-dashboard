## Summary

Keep open. The PR targets a plausible cron false-positive, but the patch is not mergeable as written because it lets any earlier or quoted `===DONE_OK===` text in any payload override a later real error, and the PR still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds whitespace-tolerant `===DONE_OK===` detection to cron payload outcome classification and resolver tests for sentinel, no-sentinel, and run-level-error cases.

## Best Possible Solution

Define a trusted final or structured cron success signal, preserve run-level/failure/denial precedence, and cover the accepted contract with focused resolver and runner regression tests plus redacted real-run proof.

## Reproduction Assessment

Yes, by source inspection. A `resolveCronPayloadOutcome` call with successful text followed by a later error payload remains fatal on current main, and `run.ts` maps that flag directly to cron `status`.

## Solution Assessment

No, not as written. A success sentinel may be viable, but scanning every payload text is broader than the trusted final signal needed to avoid masking later real failures.
