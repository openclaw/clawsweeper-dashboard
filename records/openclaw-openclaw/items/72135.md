## Summary

Keep open, but not ready to merge. Current main still lacks a native cron success sentinel, while this draft/conflicting PR scans untrusted payload text, lacks inspectable real behavior proof, and needs maintainer judgment on the success-token contract.

## What This Changes

The PR adds whitespace-tolerant `===DONE_OK===` detection to cron payload outcome classification and adds resolver tests for sentinel, no-sentinel, and run-level-error cases.

## Best Possible Solution

Define a trusted final or structured cron success signal, preserve run-level/failure/denial precedence, and cover the accepted contract with focused resolver and runner regression tests.

## Reproduction Assessment

Yes by source inspection. A `resolveCronPayloadOutcome` call with successful text followed by a later non-message error remains fatal on current main, and the runner maps that flag directly to cron `status`.

## Solution Assessment

No, not as written. A success sentinel may be viable, but scanning every payload text is broader than the trusted outcome signal needed to avoid masking later real failures.
