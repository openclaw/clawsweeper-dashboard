## Summary

Keep open: current main still lacks a native `===DONE_OK===` cron success sentinel, but this draft PR should not merge as written because it trusts arbitrary payload text and is stale against newer cron classifier changes.

## What This Changes

The PR adds `===DONE_OK===` sentinel detection to cron payload outcome classification and unit tests so final or payload sentinel text can make trailing error payloads non-fatal unless a run-level error exists.

## Best Possible Solution

Define a narrow cron success signal with explicit trust and precedence rules, rebase it onto the current resolver, and add focused tests plus changelog/docs if maintainers accept the contract.

## Reproduction Assessment

Yes by source inspection. A `resolveCronPayloadOutcome` call with non-error output followed by a final `isError: true` payload is covered by current tests as fatal, and `run.ts` records that flag as cron `status: "error"`.

## Solution Assessment

No, not as written. A success sentinel may be viable, but it should be limited to a trusted final signal or structured metadata and rebased onto the current failure/denial classifier.
