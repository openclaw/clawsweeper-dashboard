## Summary

Keep open. The PR targets a real source-reproducible cron classification path, but the branch is still draft/conflicting, lacks inspectable real behavior proof, and the proposed override trusts arbitrary payload text for a success token, which can mask a later real failure.

## What This Changes

The PR adds `===DONE_OK===` detection to cron payload outcome classification so a sentinel in final assistant text or payload text can make trailing error payloads non-fatal, with focused unit tests.

## Best Possible Solution

Define an explicit trusted cron success signal, preserve denial/failure/run-level-error precedence, rebase it onto the current resolver, and cover the accepted contract with regression tests plus docs/changelog during landing.

## Reproduction Assessment

Yes, by source inspection. Current main has a focused resolver test where earlier assistant output plus a trailing real error remains fatal, and `run.ts` records that flag as cron `status: "error"`.

## Solution Assessment

No, not as written. A native success signal may be viable, but the override should come from a trusted final signal or structured metadata rather than arbitrary earlier payload text.
