## Summary

Keep open. Current main still lacks a native `===DONE_OK===` cron success contract, but the draft should not merge as written because it trusts sentinel text from every payload and is missing the required user-facing changelog entry.

## What This Changes

The PR adds `===DONE_OK===` detection to cron payload outcome resolution and tests for treating sentinel-marked trailing error payloads as non-fatal unless a run-level error exists.

## Best Possible Solution

Define a narrow cron success signal with explicit trust and precedence rules, rebase it onto the current resolver, and document/changelog the user-visible classification change if maintainers accept the contract.

## Reproduction Assessment

Yes. A focused `resolveCronPayloadOutcome` call with a successful payload followed by a final `isError: true` payload reproduces current-main fatal classification, and the existing resolver test locks that behavior in.

## Solution Assessment

No, not as written. A success token may be a workable direction, but the current patch should trust only a final intentional signal or structured metadata rather than all payload text.
