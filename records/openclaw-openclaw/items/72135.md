## Summary

Keep open: current main still lacks a `===DONE_OK===` cron success contract and still classifies real trailing error payloads as fatal, but this draft PR should not merge as written because it trusts every payload text as a success override, lacks the required changelog entry, and is dirty against newer cron classifier changes.

## What This Changes

The PR adds `===DONE_OK===` matching to `resolveCronPayloadOutcome` and tests so final or payload sentinel text can make trailing cron error payloads non-fatal unless a run-level error exists.

## Best Possible Solution

Define a narrow cron success signal with explicit trust and precedence rules, rebase it onto the current resolver, preserve structured failure/denial behavior, and add targeted tests plus changelog/docs if maintainers accept the contract.

## Reproduction Assessment

Yes by source inspection. A focused `resolveCronPayloadOutcome` call with non-error output followed by a final `isError: true` payload matches the current regression test, and `run.ts` records that flag as an error status.

## Solution Assessment

No, not as written. A success sentinel may be viable, but this implementation should not trust arbitrary earlier payload text; a final trusted signal or structured metadata after maintainer agreement is safer.
