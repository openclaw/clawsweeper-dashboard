## Summary

Keep open. Current main still lacks a native `===DONE_OK===` cron success contract, so the PR addresses a real false-positive classification path, but the proposed implementation is not merge-ready because it trusts the sentinel anywhere in payload history and can suppress later real failures.

## What This Changes

The PR changes the isolated cron payload resolver and tests so `===DONE_OK===` in final assistant text or payload text makes trailing error payloads non-fatal unless a run-level error is present.

## Best Possible Solution

Keep the draft open. If maintainers accept the direction, rebase onto current main and implement a narrow cron success contract that trusts only an intentional final success signal or equivalent structured metadata while preserving structured failures, denial markers, run-level errors, and real trailing-error handling.

## Reproduction Assessment

Yes. A focused call to `resolveCronPayloadOutcome` with successful assistant output followed by a final `isError: true` payload reproduces current-main fatal classification, and `run.ts` persists that flag as `status: "error"`.

## Solution Assessment

No, not as written. The sentinel direction may be acceptable, but matching against every payload text is too broad because earlier or quoted sentinel text can suppress a later real failure; the safer fix is to constrain the trusted success signal and rebase it into the current classifier.
