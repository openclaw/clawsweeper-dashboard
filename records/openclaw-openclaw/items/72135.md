## Summary

Keep open. The reported cron false-positive path is source-reproducible and current main does not implement a native success sentinel, but this draft PR should not merge as written because it trusts arbitrary payload text and is now out of date with the current cron classifier.

## What This Changes

The PR adds `===DONE_OK===` sentinel detection to cron payload outcome classification and tests that trailing error payloads become non-fatal when the sentinel appears.

## Best Possible Solution

Define an explicit trusted cron success signal, preserve denial/failure/run-level-error precedence, rebase it onto the current resolver, and cover the accepted contract with focused regression tests plus docs/changelog during landing.

## Reproduction Assessment

Yes by source inspection. A `resolveCronPayloadOutcome` call with successful text followed by a final `isError: true` payload is fatal on current main, and `run.ts` records that flag as cron `status: "error"`.

## Solution Assessment

No, not as written. A success sentinel may be viable, but the override should come from a trusted final signal or structured metadata rather than arbitrary earlier payload text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restrict sentinel matching to a trusted final signal:** `src/cron/isolated-agent/helpers.ts:168-172`
  - body: Because this concatenates every payload's text, an earlier or quoted `===DONE_OK===` can make a later real tool failure non-fatal. Scope the override to an intentional final success signal or structured metadata, and add a sentinel-before-failure regression case.
