## Summary

Keep open. Current main still has the Telegram accepted-watermark restart gap, while this PR is an active, targeted implementation with no blocking correctness or security finding from this review; it should go through maintainer mergeability and validation rather than cleanup-close or automated replacement.

## What This Changes

The PR changes the Telegram plugin to persist fetched/accepted and completed update watermarks separately, restart polling from the completed watermark, log accepted-but-incomplete recovery ranges, export the expanded offset snapshot API, add regression tests, and add a changelog entry.

## Best Possible Solution

Land this PR or an equivalent Telegram plugin-local durability fix after maintainer review confirms mergeability on current main, preserving the duplicate/replay guards while storing accepted and completed restart state separately and keeping the getUpdates durability barrier.

## Reproduction Assessment

Yes for the restart durability race: current main accepts and persists update IDs before handler completion, restores that accepted watermark, and skips the same update after restart. The broader no-crash report remains unclear without runtime logs, but dependency source and current tests give a high-confidence reproduction path for the race this PR targets.

## Solution Assessment

Yes. Separating accepted/fetched state from completed-handler state inside the Telegram plugin is the narrowest maintainable fix; Greptile's suggested completed-watermark monotonic guard is reasonable defensive hardening, but the current PR call path already gates completed persistence through the tracker’s monotonic safe-completed request state, so I do not treat it as a blocking bug.
