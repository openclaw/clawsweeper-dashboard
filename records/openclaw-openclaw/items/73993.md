## Summary

Keep open. Current main still has the single accepted Telegram polling offset that can skip accepted-but-unfinished updates after restart, and this PR is the right local fix direction, but GitHub reports the branch dirty/unmergeable and not maintainer-editable, so it needs a clean replacement or rebase before it can land.

## What This Changes

The PR splits Telegram polling accepted and completed offset state, persists fetched `getUpdates` update IDs before returning batches to grammY, wires completed-watermark callbacks through the Telegram bot/session, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a clean Telegram plugin-local durability fix that preserves the accepted/completed watermark split, the fetched-offset persistence barrier, legacy state migration, and focused restart regression coverage.

## Reproduction Assessment

Yes for the restart durability race: current main accepts and persists update IDs before handler completion, restores the same accepted watermark on restart, and grammY runner 2.0.3 advances its internal offset after successful fetch batches. The no-crash variant remains unclear without logs tying it to a restart/reload or another narrowed path.

## Solution Assessment

Yes in direction, but not this branch as-is. A replacement or clean rebase that keeps the fix inside the Telegram plugin is safer than moving the behavior into core or merging the dirty head.
