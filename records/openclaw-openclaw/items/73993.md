Keep this PR open. Current main still has the Telegram single accepted-watermark behavior that can skip unfinished updates after restart, while the PR is a valid plugin-local implementation candidate and I did not find a concrete blocking correctness or security issue in the diff.

## What This Changes

The branch updates the Telegram plugin to persist fetched accepted offsets separately from completed handler watermarks, wires completed-offset persistence through polling and bot tracking, exports the expanded offset snapshot API, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land this PR or an equivalent Telegram plugin-local durability fix after maintainer review confirms mergeability on current main, preserving replay guards while storing completed-safe restart state and the getUpdates durability barrier.

## Reproduction Assessment

Yes for the restart durability race: current main accepts and can persist update_ids before handler completion, then skips those update_ids after restart from the accepted watermark. The broader no-crash report is still unclear without runtime logs or a narrower sequence.

## Solution Assessment

Yes for the implementation direction: separating fetched/accepted state from completed handler state inside the Telegram plugin is narrower and more maintainable than moving this into core. The branch still needs normal maintainer mergeability and validation handling before landing.
