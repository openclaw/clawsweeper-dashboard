## Summary

Keep open. Current main still has the Telegram polling durability gap, and this PR is a useful plugin-local fix direction, but the branch is dirty/uneditable and has one small defensive watermark issue before a clean replacement should land.

## What This Changes

The PR splits Telegram polling accepted/fetched offsets from completed-handler watermarks, persists fetched `getUpdates` IDs before grammY can advance, adds restart recovery logging and regression coverage, and updates the changelog.

## Best Possible Solution

Land a clean Telegram plugin-local replacement that preserves the accepted/completed watermark split, fetched-offset persistence barrier, legacy state migration, recovery logging, and focused restart regression coverage.

## Reproduction Assessment

Yes for the restart durability race: source inspection plus grammY runner 2.0.3 behavior show accepted update IDs can be persisted and Telegram-confirmed before handler completion, then skipped after restart. The no-crash report still needs runtime logs or a narrower sequence.

## Solution Assessment

Yes in direction, but not as a direct merge of this branch. The narrow maintainable path is a clean Telegram plugin replacement or rebase with the defensive completed-watermark guard added.
