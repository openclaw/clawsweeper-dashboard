## What This Changes

The issue requests CLI-runner session repair so fallback retries against resumed CLI sessions do not append duplicate user turns.

## Best Possible Solution

Add a CLI-owned fallback retry repair that refreshes or clears reusable CLI bindings between candidates, or transcript-confirms safe prompt handling, with timeout/rate-limit regression coverage.

Yes at source level. The fallback loop can reuse a captured session entry, the CLI path resumes the binding from that entry, and resume-mode execution sends prompt text again.

## Solution Assessment

Yes. The right fix boundary is CLI-owned session binding refresh/reset/reseed or backend-confirmed transcript repair; skipping prompt text blindly would be unsafe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54192. Current main can retry a CLI-backed fallback candidate with a captured session entry that still contains a reusable CLI binding, while resume-mode CLI execution sends prompt text again. Implement a bounded CLI-owned repair that prevents duplicate user turns on timeout/rate-limit fallback retries against resumed CLI sessions: refresh or clear the reusable CLI binding before later candidates, or use backend-confirmed transcript state to safely avoid re-submitting an already-persisted prompt. Do not make a global prompt-skip that can drop the task if the first attempt failed before persistence, and do not mix in media/partial-execution policy work from https://github.com/openclaw/openclaw/pull/57959. Add focused regression coverage for resumed CLI fallback retry behavior and update CHANGELOG.md under Fixes if the change is user-visible.
