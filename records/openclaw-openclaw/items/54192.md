Keep open. Current main still has the CLI fallback retry/session gap: fallback retries can reuse an existing CLI backend session while still sending the retry-marked prompt, and the explicit orphaned trailing-user repair remains embedded-runner-only.

## What This Changes

Add CLI-runner repair so fallback retries against resumed CLI sessions do not create duplicate user turns after rate limits or timeouts.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids reusing a backend session after the first attempt may have appended the prompt, while preserving the task through a fresh/reseeded CLI session or backend-confirmed transcript repair, with focused timeout and rate-limit regression coverage.

## Repair Work Prompt

Fix the CLI runner fallback retry duplicate-user-turn bug tracked by https://github.com/openclaw/openclaw/issues/54192. Observable problem: when a CLI provider fallback retry runs against an existing cliSessionId after a rate limit or timeout, OpenClaw can resume the same backend session and submit the retry-marked prompt again, creating duplicate user turns. Scope the fix to the CLI attempt/runner path; do not change embedded runner orphaned-user repair, broad model fallback trust policy, image stripping semantics, or unrelated session migrations. Implement a backend-aware repair that either resets/reseeds into a fresh CLI session after a possibly-mutating failed attempt or confirms the backend transcript state before avoiding resubmission. Preserve session_expired clearing behavior, auth profile/session binding metadata, OpenClaw transcript persistence, and fresh-session reseed behavior. Add focused tests for timeout and rate-limit fallback retries with a reused cliSessionId, plus a guard that the task is not dropped when the first attempt failed before persistence. Add a user-facing changelog Fixes entry if the PR changes shipped behavior; credit @mitchmcalister.
