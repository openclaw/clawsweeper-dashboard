## Summary

Keep open. Current main has adjacent Claude CLI session cleanup after failed reused turns, but the fallback loop still builds later fallback attempts from the captured session entry, and resume-mode CLI execution still sends prompt text whenever that stale CLI session id is supplied.

## What This Changes

The issue requests a CLI-runner repair so fallback retries against resumed CLI sessions do not append duplicate user turns.

## Best Possible Solution

Add a CLI-owned fallback retry repair that makes later fallback candidates use a refreshed or cleared CLI binding, or a backend-confirmed transcript repair, with focused timeout/rate-limit regression coverage.

## Reproduction Assessment

Yes, at source level: fallback attempts reuse the captured session entry, the CLI attempt path resumes the binding from that entry, and resume-mode CLI execution sends prompt text again. I did not run a live backend reproduction in this read-only review.

## Solution Assessment

Yes. The maintainable fix is a CLI-owned session reset/reseed or refreshed-entry handoff for fallback retries; skipping argv/stdin prompt text without transcript proof would be unsafe.
