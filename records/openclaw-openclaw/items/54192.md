## Summary

Keep open. Current main has adjacent Claude CLI cleanup, but fallback retries can still pass the pre-loop CLI session binding back into the CLI runner while resume-mode execution sends the prompt again; the specific same-run orphaned-user-message repair remains missing.

## What This Changes

The issue requests a CLI-runner repair so fallback retries against resumed CLI sessions do not append duplicate user turns.

## Best Possible Solution

Add a CLI-owned retry repair that makes subsequent fallback attempts use a fresh or safely reseeded CLI backend session, or only skips resubmission after backend transcript evidence confirms the prompt was already accepted.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the fallback loop reuses the captured session entry, the CLI attempt reads the resume binding from that entry, and resume-mode CLI execution still sends the prompt.

## Solution Assessment

Yes, the maintainable direction is a CLI-owned session repair/reset or transcript-confirmed resubmission guard; a simple argv/stdin prompt skip is not safe by itself.

Add a CLI-owned fallback retry repair at the agent-command/runAgentAttempt boundary so after a resumed CLI attempt fails with a fallback-eligible error, subsequent fallback attempts do not reuse the same backend CLI session and do not append the same user prompt again. Do not implement a blanket prompt skip unless backend transcript evidence confirms the prompt was accepted. Add focused regression coverage for a resumed Claude CLI attempt that fails with timeout or rate_limit and then falls back, asserting the next attempt does not use the old cliSessionId; include coverage that fresh/cold CLI timeouts do not clear unrelated state.
