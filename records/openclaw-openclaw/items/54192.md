## Summary

Keep open. Current main still sends fallback retry prompt text through the CLI runner while reusing an existing CLI backend session, and the explicit orphaned trailing-user repair remains limited to the embedded runner path. The prior discussion around #44188 is still relevant and correctly frames this as a CLI-session retry gap.

## What This Changes

The issue asks for CLI fallback retries against resumed CLI sessions to avoid duplicating the same user prompt in the backend session.

## Best Possible Solution

Add a CLI-owned fallback retry repair that prevents reusing a backend session after the first attempt may have appended the prompt, while preserving the original task through a fresh/reseeded CLI session or a backend-confirmed transcript repair. Cover timeout and rate-limit fallback retries with focused CLI regression tests.

## Reproduction Assessment

Yes. A focused mocked test can reproduce this by starting `runAgentAttempt` with an existing CLI session binding, making the first CLI-backed fallback candidate fail with timeout or rate-limit, and asserting the retry does not send the same prompt into the same `cliSessionId`.

## Solution Assessment

Unclear for the exact implementation, but the narrow CLI-owned repair direction is best. Directly porting the embedded session-leaf repair may not be sufficient because CLI transcript state is backend-owned; reset/reseed or confirmed repair is safer than unconditional prompt skipping.

## Repair Work Prompt

Fix the CLI runner fallback retry duplicate-prompt gap tracked by https://github.com/openclaw/openclaw/issues/54192. Current main can reuse an existing CLI backend session on fallback retry while sending a retry-marked copy of the original prompt, which can duplicate user turns in resumed CLI sessions. Keep the fix CLI-owned; do not change broad model fallback policy, do not add backend-specific hacks in core unless already represented by CLI backend metadata, and do not unconditionally skip prompt text because the first attempt may have failed before persisting the user turn. Prefer clearing/reseeding the reusable CLI session for fallback retries, or a backend-confirmed transcript repair that proves the prompt is already present before avoiding resubmission. Preserve the original task for the fallback model. Add focused regression coverage for timeout and/or rate-limit fallback retry with an existing `cliSessionId`, asserting the retry does not send the same prompt into the same backend session. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @mitchmcalister`.
