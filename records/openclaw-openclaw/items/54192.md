## Summary

Keep open. Current main still reuses the stored CLI backend session during model fallback retries while also sending a retry-marked copy of the original prompt, and the explicit orphaned trailing-user repair remains embedded-runner-only. The prior #44188 discussion correctly identified this as a CLI-session retry gap rather than a problem already fixed by the unified fallback safety work.

## What This Changes

The issue requests CLI fallback retry handling that prevents duplicate user turns when retrying against an existing resumed CLI session.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids reusing an already-mutated backend session for the same prompt while preserving the original task through a fresh/reseeded CLI session or a backend-confirmed transcript repair. Cover timeout and rate-limit fallback retries with focused CLI regression tests.

## Reproduction Assessment

Yes. A focused mocked test can reproduce this by giving `runAgentAttempt` an existing CLI session binding, making the first CLI attempt fail with `timeout` or `rate_limit`, then asserting the fallback retry does not send the same prompt into the same `cliSessionId`.

## Solution Assessment

Yes, a narrow CLI-owned repair is the right direction. Porting the embedded shape directly may not be enough because CLI transcript state lives in backend-owned sessions, so the safer fix is reset/reseed or transcript-confirmed repair rather than unconditional prompt skipping.
