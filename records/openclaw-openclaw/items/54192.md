Keep open. Current main still routes fallback attempts through the same CLI session binding while resume-mode CLI execution still submits prompt text, and the adjacent merged fallback fixes do not cover this CLI backend-session duplicate-turn repair.

## What This Changes

The issue requests a CLI-runner repair that prevents fallback retries in resumed CLI sessions from adding duplicate user turns.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids resubmitting an already-accepted current prompt to a reused backend CLI session while preserving the task through a fresh/reseeded session or transcript-confirmed repair.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI runner/fallback attempt path, not in broad fallback policy or a simple prompt skip unless backend transcript evidence confirms the prompt was already accepted.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54192. CLI fallback retries with an existing `cliSessionId`/`cliSessionBinding` can resubmit the same current prompt to a resumed backend CLI session after a fallbackable timeout or rate-limit failure. Implement a CLI-owned repair in the fallback attempt/CLI runner path that avoids duplicate backend user turns while preserving the task. Do not use a blanket prompt skip unless backend transcript/session evidence proves the first prompt was accepted; otherwise reset/reseed or branch the CLI session safely. Preserve existing `session_expired`, missing Claude transcript, auth-hash/session reuse, Claude-to-non-CLI context seeding, and embedded-runner orphan repair behavior. Add focused regression coverage for timeout/rate-limit fallback with a reused CLI session id and for the no-loss case where the first attempt failed before accepting the prompt. Add a single changelog `### Fixes` entry if the repair is user-visible.
