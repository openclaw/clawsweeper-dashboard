## Summary

Keep open. Current main still sends retry-marked prompt text into a reusable CLI backend session on fallback retry, while the explicit orphaned trailing-user repair remains embedded-runner-only.

## What This Changes

The issue requests CLI runner fallback retries against resumed CLI sessions to avoid duplicating the current user prompt in the backend transcript.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids reusing an already-mutated backend session for the same prompt while preserving the task through a fresh/reseeded session or transcript-confirmed repair.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: use an existing CLI session binding, make the first CLI attempt fail with `timeout` or `rate_limit`, then observe the fallback retry passing the retry-marked prompt into the same `cliSessionId`. ## Solution Assessment

Yes, the best direction is a narrow CLI-owned repair rather than broad fallback-policy changes. Porting the embedded leaf repair directly may not be sufficient because CLI transcript state lives in backend-owned sessions.

## Repair Work Prompt

Fix the CLI runner fallback retry duplicate-prompt bug for https://github.com/openclaw/openclaw/issues/54192. On a fallback retry after a timeout/rate-limit from a resumed CLI backend session, do not send the same current user prompt into the same backend `cliSessionId` again. Keep the fix CLI-owned: prefer a fresh/reseeded CLI session or backend-confirmed transcript repair, and stop if the design requires changing product fallback policy or adding a config option. Add focused regression coverage for an existing CLI session binding where the first CLI attempt fails with `timeout` or `rate_limit` and the fallback retry does not reuse the same backend session for the duplicate prompt. Preserve existing `session_expired`, missing-transcript, cross-provider image stripping, and embedded-runner orphaned-user-message behavior. Add a single-line changelog fix entry if the change is user-facing.
