## Summary

Keep open. Current main still has the CLI fallback retry gap: fallback attempts can reuse the captured CLI session binding while resume-mode CLI execution sends the prompt again, and the explicit orphaned-user-message repair remains embedded-runner-only.

## What This Changes

Request to add a CLI-runner repair so fallback retries against resumed CLI sessions do not append duplicate user turns.

## Best Possible Solution

Add a CLI-owned fallback retry repair that refreshes or clears stale CLI session bindings, or confirms the backend transcript before reuse, with focused timeout/rate-limit regression coverage.

## Reproduction Assessment

Yes, at source level: fallback attempts reuse the captured session entry, the CLI attempt path resumes the binding from that entry, and resume-mode CLI execution sends prompt text again. I did not run a live backend reproduction in this read-only review.

## Solution Assessment

Yes. The maintainable fix is a CLI-owned session reset/reseed or refreshed-entry handoff for fallback retries; skipping prompt text without transcript proof would be unsafe.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54192. Current main's fallback loop in src/agents/agent-command.ts passes a captured sessionEntryForAttempt into each runAgentAttempt; the CLI path in src/agents/command/attempt-execution.ts reads the CLI binding from that entry, and src/agents/cli-runner/execute.ts sends prompt text during resume. Add a CLI-owned fallback retry repair so, after a timeout/rate-limit/failover on a reused CLI session, the next fallback candidate does not resume the same stale backend session while submitting the prompt again. Prefer refreshing/clearing the CLI binding at the fallback boundary or using a backend-confirmed transcript repair; do not implement a blind prompt-skip that can drop the task. Add regression coverage for two CLI fallback candidates with an existing cliSessionId where the first reused attempt fails and the second attempt preserves the task without reusing the stale CLI backend session. Keep the fix in the CLI runner/session-store/fallback boundary and avoid provider-policy or plugin API changes. Add a changelog fix entry if this is treated as user-visible.
