## Summary

Keep open. Current main has adjacent Claude CLI session cleanup, but same-run fallback retry can still reuse the pre-loop CLI session entry while resume-mode CLI execution sends the prompt text again.

## What This Changes

Add a CLI runner fallback retry repair so resumed CLI backend sessions do not receive the same user prompt twice after timeout or rate-limit failover.

## Best Possible Solution

Add a CLI-owned retry repair that starts the next fallback attempt from a fresh or safely reseeded backend session, or skips resubmission only after backend transcript evidence proves the prompt was already accepted.

## Reproduction Assessment

Yes, at source level: the fallback loop reuses a captured session entry, the CLI attempt reads the resume binding from that entry, and resume-mode CLI execution still sends prompt text. I did not run a live backend reproduction in this read-only review.

## Solution Assessment

Yes, the maintainable fix direction is a CLI-owned session repair/reset or transcript-confirmed resubmission guard. A simple prompt-skip in argv/stdin handling is not safe without evidence that the backend accepted the first user turn.
