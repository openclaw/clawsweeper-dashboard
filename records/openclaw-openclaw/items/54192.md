## Summary

Keep open. Current main still has the CLI fallback retry gap: fallback retries can reuse the captured CLI session binding while resume-mode CLI execution sends the prompt text again, and no open PR currently closes this issue.

## What This Changes

The issue asks for a CLI runner retry repair so resumed CLI backend fallback retries do not append the same user prompt twice.

## Best Possible Solution

Add a CLI-owned retry repair that avoids resubmitting the same prompt into an already-used resumed backend session, using a fresh/reseeded CLI session or transcript-confirmed repair rather than a blind prompt skip.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the fallback loop can reuse a captured session entry, the CLI attempt reads the resume binding from that entry, and resume-mode CLI execution still sends the prompt. I did not run a live CLI backend repro in this read-only review.

## Solution Assessment

Yes, the maintainable direction is a CLI-owned session repair/reset or transcript-confirmed resubmission guard. A simple argv/stdin prompt skip is not safe by itself because the first attempt may not have persisted the user turn.
