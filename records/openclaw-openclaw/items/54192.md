## Summary

Keep open. Current main and the latest shipped release still reuse the stored CLI session binding on fallback retries while resume-mode CLI execution sends the prompt text again, and no closing PR currently owns this exact CLI retry repair.

## What This Changes

The issue requests CLI runner retry-session repair so resumed CLI fallback retries do not duplicate user turns.

## Best Possible Solution

Add a CLI-owned fallback retry repair that starts from a fresh or safely reseeded backend session, or skips resubmission only after backend transcript evidence proves the first prompt was accepted.

## Reproduction Assessment

Yes, at source level: the fallback loop reuses the captured session entry, the CLI attempt resumes the stored backend session, and resume-mode CLI execution still sends the prompt text. I did not run a live backend reproduction in this read-only review.

## Solution Assessment

Yes. The maintainable fix direction is a CLI-owned session repair/reset or transcript-confirmed resubmission guard; a simple argv/stdin prompt skip is not safe without proof that the backend accepted the first user turn.

## Repair Work Prompt

Fix the CLI runner fallback retry duplicate-user-turn bug tracked by https://github.com/openclaw/openclaw/issues/54192. In current main, fallback retries can reuse a stored CLI backend session while executePreparedCliRun still sends params.prompt in resume mode. Add a CLI-owned retry repair in the agent/CLI boundary: either reset/reseed the backend session before the next fallback attempt after the first attempt may have appended the prompt, or skip resubmission only when backend transcript evidence proves the prompt was already accepted. Do not implement a blind argv/stdin prompt skip, do not broaden embedded-runner behavior, and coordinate with the adjacent Claude CLI invalidation/reseed work in https://github.com/openclaw/openclaw/pull/81821 if it is still open. Add focused regression coverage for timeout/rate-limit fallback retry with an existing cliSessionId and assert the same prompt is not appended twice while the original task is preserved.
