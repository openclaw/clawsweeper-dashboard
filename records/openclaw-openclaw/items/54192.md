## Summary

Keep open. Current main still routes CLI fallback retries through a reusable CLI session while supplying prompt text on resume; the merged adjacent fixes cover embedded transcript dedupe and Claude-to-non-CLI context seeding, not this CLI backend duplicate-turn repair.

## What This Changes

The issue requests a CLI runner repair so fallback retries against resumed CLI sessions do not duplicate the user prompt in the backend session.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids resubmitting the same current prompt to an already-mutated CLI backend session, while preserving the task through a fresh/reseeded session or transcript-confirmed repair.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use an existing CLI session binding, make the first CLI attempt fail with `timeout` or `rate_limit`, and the fallback retry still passes prompt text into the same `cliSessionId`.

## Solution Assessment

Yes. The best direction is a narrow CLI-owned repair rather than broad fallback-policy changes; it should not be a simple prompt skip unless the backend transcript confirms the original prompt was already accepted.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54192. In the CLI runner path, fallback retries with an existing reusable CLI session can resubmit the same current prompt to the backend session after a timeout/rate_limit failure. Add a CLI-owned repair in `src/agents/command/attempt-execution.ts` and/or `src/agents/cli-runner/execute.ts` so fallback retries do not reuse an already-mutated backend session for the same prompt unless transcript-confirmed safe; prefer a fresh/reseeded session over a blind prompt skip. Preserve existing `session_expired` clearing, missing-transcript clearing, Claude CLI fallback-context prelude behavior, and embedded duplicate-persistence suppression. Add focused regression coverage for an existing CLI session binding where the first CLI attempt throws `FailoverError` with `timeout` or `rate_limit` and the fallback retry does not pass the same `cliSessionId` plus same prompt into the backend. Do not change broad fallback trust policy, provider output-mode defaults, unrelated session-store behavior, or plugin/channel code. Add a changelog entry if the fix is user-visible.
