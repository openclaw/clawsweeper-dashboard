## Summary

Keep open. Current main has an adjacent clear-after-error guard for reused Claude CLI sessions, but the fallback loop can still pass the same stale in-memory CLI session binding into the next fallback attempt, so the duplicate resumed-turn problem is not fully repaired.

## What This Changes

The issue requests CLI runner retry repair so fallback retries against an existing CLI backend session do not append the same user prompt twice.

## Best Possible Solution

Refresh or invalidate the CLI session binding before each fallback retry uses it, then add a regression where a resumed CLI attempt times out or rate-limits and the next fallback attempt does not reuse the same backend session id.

## Reproduction Assessment

Yes, by source inspection rather than a live run. A fallback retry can reuse the same captured sessionEntryForAttempt while the CLI runner still submits params.prompt in resume mode.

## Solution Assessment

No, current main is not sufficient yet. The new clear-after-error path updates persisted state, but the in-flight fallback loop can still pass the stale in-memory CLI binding into the next attempt.
