## Summary

This PR should stay open: it targets a real direct-agent and isolated-cron fallback persistence gap, but the current patch still misclassifies user live-switches that are consumed inside the fallback runner, and it is missing the required changelog entry.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, extends `updateSessionStoreAfterAgentRun` with an `isFromFallback` flag, and adds regression tests for fallback no-persist and live-switch behavior.

## Best Possible Solution

Use an explicit selection-source contract shared by direct agent and cron paths: automatic fallback remains transient, user live-switch targets persist even when `runWithModelFallback` handles the redirect internally, and the fallback runner exposes enough outcome metadata to avoid after-the-fact model/provider inference.

## Reproduction Assessment

Yes. Configure a primary plus a later fallback candidate, trigger `LiveSessionModelSwitchError` from the primary to that later candidate, and `runWithModelFallback` will consume the redirect before the PR's outer live-switch tracking records the target.

## Solution Assessment

No. The direction is right, but comparing final provider/model values after the run is not reliable enough; a safer fix should carry explicit fallback/live-switch outcome metadata from the fallback runner or persist the live-switch target before retry.
