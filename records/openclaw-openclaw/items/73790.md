## Summary

Keep open: the PR addresses a real fallback persistence bug, but the current patch still loses live-switch source metadata when the shared fallback runner redirects internally, and it can leave persisted sessions with a primary model paired to fallback contextTokens.

## What This Changes

The PR guards direct-agent and isolated-cron runtime model writes so automatic fallback models are not persisted, adds session-store and cron regression tests, and records the fix in the changelog.

## Best Possible Solution

Carry explicit selection-source metadata through runWithModelFallback, use it in both direct-agent and cron persistence, and preserve contextTokens whenever automatic fallback runtime model fields are not persisted.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a primary attempt that throws LiveSessionModelSwitchError targeting a later fallback candidate; runWithModelFallback redirects internally and returns that target with no outer catch, so the PR marks it as automatic fallback. A second focused case is a fallback result with a different context window, where the PR skips provider/model persistence but still writes fallback contextTokens.

## Solution Assessment

No. Final provider/model comparison is not the best fix because it loses whether the selected model came from automatic fallback or a user live switch; explicit source metadata plus treating contextTokens as part of the runtime snapshot is safer.

  - body: executeCronRun uses the same fallback runner and mutates liveSelection after the runner returns. If a live switch is handled inside the runner, comparing only final model/provider to the pre-run snapshot classifies the intentional switch like automatic fallback, so cron leaves the old runtime model persisted.
  - confidence: 0.84
- **[P2] Keep fallback contextTokens out of direct session state:** `src/agents/command/session-store.ts:152`
  - body: When isFromFallback is true, this guard skips provider/model persistence, but next.contextTokens was already populated from modelUsed before the guard. That can persist a fallback context window beside the preserved primary model and skew later status or compaction decisions.
  - confidence: 0.86
- **[P2] Keep fallback contextTokens out of cron session state:** `src/cron/isolated-agent/run.ts:811-817`
  - body: This fallback guard suppresses setSessionRuntimeModel, but the function still assigns sessionEntry.contextTokens from modelUsed immediately afterward. A fallback run can therefore leave cron sessions with primary provider/model fields and fallback contextTokens.
