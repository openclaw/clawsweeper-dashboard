## Summary

Keep this PR open: the reported sticky fallback-model persistence bug is still present on current main, and the PR is useful, but the patch still misclassifies user-initiated live switches that are handled inside the shared fallback runner and is missing the required changelog entry.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron runtime model persistence, extends the session-store update helper with an optional fallback flag, and adds regression tests.

## Best Possible Solution

Carry explicit selection-source metadata through the shared fallback runner so automatic fallback remains transient while user live switches persist, then wire direct-agent and cron persistence to that contract with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a primary model plus a later fallback candidate where the primary attempt throws `LiveSessionModelSwitchError` targeting that later candidate; `runWithModelFallback` consumes the redirect before the PR's outer tracking can record it.

## Solution Assessment

No. The PR direction is useful, but comparing final provider/model values is not the best fix because it loses whether the final selection came from automatic fallback or a user-requested live switch handled inside the fallback runner.
