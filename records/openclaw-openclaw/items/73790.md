## Summary

Keep open: the latest PR head still infers automatic fallback versus user live-switch from final provider/model values, but current `runWithModelFallback` can internally consume a live-switch redirect and return that target with no source signal. That leaves both direct-agent and cron persistence able to skip an intentional model switch.

## What This Changes

This PR changes direct-agent and isolated-cron session persistence so automatic fallback models are not stored as durable runtime models, preserves contextTokens when skipping persistence, adds regression tests, and updates CHANGELOG.md.

## Best Possible Solution

Add explicit selection-source metadata to `runWithModelFallback` results and have both direct-agent and cron persistence use that contract while preserving the no-persist/contextTokens behavior and live-switch regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a primary plus later fallback candidate, have the first attempt throw `LiveSessionModelSwitchError` targeting that later candidate, and the fallback runner returns the target with `attempts: []` while the PR guards classify it as fallback.

## Solution Assessment

No. The fallback no-persist direction is right, but final-value comparison is not the best contract; explicit fallback-result source metadata is the narrower maintainable fix.
