## Summary

Keep open: the PR handles the basic automatic-fallback persistence bug, but it still misclassifies live-session model switches that are consumed inside the shared fallback runner as transient fallbacks in both direct agent and cron paths.

## What This Changes

The PR guards direct-agent and isolated-cron session runtime model writes so automatic fallback models are not persisted, adds regression tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Carry explicit selection-source metadata through the shared fallback runner, then have direct-agent and cron persistence use that contract to persist user live switches while keeping automatic fallback transient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an inner attempt that throws LiveSessionModelSwitchError for a later configured fallback candidate; runWithModelFallback redirects internally and the PR then treats the successful target as automatic fallback.

## Solution Assessment

No. Comparing final provider/model values is not the narrowest maintainable fix because it loses whether the final selection came from automatic fallback or a user-requested live switch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve live switches handled inside fallback:** `src/agents/agent-command.ts:1204-1207`
  - body: `runWithModelFallback` can catch `LiveSessionModelSwitchError` from the inner `run` callback and jump to a later candidate without rethrowing to this outer catch. In that path `liveSwitchTargetModel` stays unset, so this expression marks the user-requested target as `isFromFallback: true` and `updateSessionStoreAfterAgentRun` skips persisting the switch.
  - confidence: 0.88
- **[P2] Preserve cron live switches handled inside fallback:** `src/cron/isolated-agent/run.ts:809-810`
  - body: `executeCronRun` uses the same fallback runner and mutates `liveSelection` after the runner returns. If the live switch is handled inside the runner, comparing the final model/provider only to the pre-run snapshot makes an intentional switch indistinguishable from automatic fallback, so cron leaves the old runtime model persisted.
  - confidence: 0.84
