## Summary

Keep open: the bug is real and the branch is useful, but current head still relies on final provider/model comparisons that misclassify internally handled live-session switches, and contributor real behavior proof is missing.

## What This Changes

The branch changes direct-agent and isolated-cron session persistence to avoid storing automatic fallback models as durable runtime models, preserves contextTokens, adds regression tests, and updates the changelog.

## Best Possible Solution

Add explicit selection-source metadata to runWithModelFallback results and have both direct-agent and cron persistence use that contract while preserving contextTokens behavior and live-switch/fallback regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a primary plus later fallback candidate, have the first attempt throw LiveSessionModelSwitchError targeting that later candidate, and runWithModelFallback returns the target with attempts: [] while the PR guards classify the final provider/model as fallback.

## Solution Assessment

No. The no-persist fallback direction is right, but final-value comparison is not the best contract; explicit fallback-result source metadata is the narrower maintainable fix.
