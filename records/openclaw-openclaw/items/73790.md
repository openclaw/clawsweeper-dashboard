## Summary

Keep open: the PR fixes the ordinary automatic-fallback persistence path and context-token drift, but current head still misclassifies user-initiated live switches that are handled inside the shared fallback runner, so direct-agent and cron sessions can skip persisting the requested switch.

## What This Changes

The PR guards direct-agent and cron session runtime persistence so automatic fallback model/provider/contextTokens are not stored durably, adds regression tests, and records the fix in the changelog.

## Best Possible Solution

Extend the shared fallback result with explicit selection-source metadata and have both direct-agent and cron persistence use that contract instead of inferring source from final provider/model equality.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the inner run callback throws LiveSessionModelSwitchError for a later configured fallback candidate, runWithModelFallback redirects internally and returns that target with attempts: [], and the PR callers then classify the target as fallback.

## Solution Assessment

No. The proposed guard is correct for ordinary automatic fallback but final-value comparison is not the best fix because it loses whether the successful selection came from automatic fallback or a user-requested live switch.
