## Summary

Keep this PR open: it targets a real current-main persistence edge, but the current patch still infers fallback-ness after the run and misses live-switch redirects that `runWithModelFallback` consumes internally. The prior Greptile comment was partly addressed, but the remaining direct-agent and cron redirect paths plus the missing changelog entry still block merge.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, extends `updateSessionStoreAfterAgentRun` with an optional `isFromFallback` flag, and adds regression tests for fallback no-persist and live-switch behavior.

## Best Possible Solution

Keep the fix focused on an explicit selection-source contract: automatic fallback must stay transient, user-initiated live switches must persist even when the fallback runner handles the redirect internally, and direct-agent plus cron paths should consume the same reliable outcome metadata.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a primary plus a later fallback candidate, have the primary attempt throw `LiveSessionModelSwitchError` targeting that later candidate, and observe `runWithModelFallback` consume the redirect before the PR's outer live-switch tracking sees it.

## Solution Assessment

No. The direction is useful, but the current patch is not the best safe fix because it infers fallback-ness from final provider/model comparisons; a safer approach should carry explicit fallback/live-switch outcome metadata from the fallback runner or persist the live-switch target before retry.
