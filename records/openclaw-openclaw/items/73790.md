## Summary

Keep open: the PR targets a real fallback-persistence bug, but the current head still misclassifies live switches consumed inside the fallback runner and is missing the required changelog entry.

## What This Changes

The PR guards direct-agent and isolated cron runtime-model persistence against automatic fallback selections and adds regression coverage for session-store and cron paths.

## Best Possible Solution

Carry explicit selection-source metadata through the shared fallback runner and use it at the direct-agent and cron persistence boundaries so automatic fallback remains transient while user live switches persist.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a primary model whose attempt throws `LiveSessionModelSwitchError` targeting a later fallback candidate; `runWithModelFallback` consumes that redirect before the PR's outer tracking sees it.

## Solution Assessment

No. Comparing final provider/model values is not the best fix because it loses whether the final selection came from automatic fallback or an explicit live-session switch.
