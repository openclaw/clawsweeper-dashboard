## Summary

Keep open: the branch fixed the context-token desync from the last review, but it still loses selection-source metadata when a live model switch is handled inside the shared fallback runner.

## What This Changes

The PR prevents automatic fallback models from being persisted as durable agent and cron session runtime models, adds regression tests, and records the fix in the changelog.

## Best Possible Solution

Extend the shared fallback result with explicit selection-source metadata and have both direct-agent and cron persistence use that contract instead of inferring source from final provider/model equality.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an inner attempt throws LiveSessionModelSwitchError for a later configured fallback candidate, runWithModelFallback redirects internally and returns that target with no fallback attempts, and the PR callers then misclassify the user switch as fallback.

## Solution Assessment

No. Comparing final provider/model values is not the best fix because it loses whether the successful selection came from automatic fallback or from a user-requested live switch; explicit metadata from the fallback runner is the narrower maintainable contract.
