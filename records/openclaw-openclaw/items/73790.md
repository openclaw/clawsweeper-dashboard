## Summary

Keep this PR open with changes requested. The underlying bug is valid, but the current patch still infers fallback-vs-live-switch state from final provider/model comparisons, so user-initiated live switches that are consumed inside the shared fallback runner can still be treated as automatic fallback and not persisted; the user-facing fix also needs a changelog entry.

## What This Changes

The PR adds guards around direct-agent and isolated-cron runtime model persistence, extends `updateSessionStoreAfterAgentRun` with an optional fallback flag, and adds regression coverage for fallback no-persist and live-switch behavior.

## Best Possible Solution

Use an explicit selection-source contract from the shared fallback runner through direct-agent and cron persistence: automatic fallback stays transient, user-initiated live switches persist even when handled inside fallback, and the behavior is covered by focused direct, cron, and fallback-runner tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a primary with a later fallback candidate, have the primary attempt throw `LiveSessionModelSwitchError` targeting that later candidate, and observe `runWithModelFallback` consume the redirect before the PR's outer tracking sees it.

## Solution Assessment

No. The current approach is not the best safe fix because it infers fallback state from final provider/model comparisons; the safer direction is to carry explicit fallback/live-switch outcome metadata from `runWithModelFallback` or persist the user switch before retry.
