## Summary

Keep open: the PR fixes the main automatic-fallback persistence path and preserves contextTokens, but the new guards still infer source from final provider/model values. That leaves user-initiated live switches handled inside `runWithModelFallback` misclassified as automatic fallback in both direct-agent and cron paths.

## What This Changes

The PR adds fallback no-persist guards for direct-agent/session-store and isolated cron, preserves contextTokens when skipping fallback persistence, adds regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Add explicit selection-source metadata to `runWithModelFallback` results and make both direct-agent and cron persistence use that contract instead of final provider/model comparison.

## Reproduction Assessment

Yes. Source inspection shows `runWithModelFallback` can consume `LiveSessionModelSwitchError` internally and return the switch target with `attempts: []`, so the PR's final-value guards have a concrete misclassification path.

## Solution Assessment

No. The PR direction fixes automatic fallback persistence, but final-value comparison is not the narrowest maintainable contract; explicit fallback-result source metadata is safer and covers both direct-agent and cron callers.
