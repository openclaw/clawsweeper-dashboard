## Summary

Keep open: the reported fallback persistence bug is real and the PR is useful, but the current diff still loses user-initiated live switches handled inside the shared fallback runner and can leave a preserved primary model paired with fallback context-window metadata.

## What This Changes

The PR changes direct-agent and cron session persistence to avoid durably writing automatic fallback model/provider selections, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Carry explicit selection-source metadata through runWithModelFallback and have direct-agent plus cron persistence treat provider, model, and contextTokens as one runtime snapshot: persist live switches, keep automatic fallback transient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an inner attempt that throws LiveSessionModelSwitchError for a later configured candidate; current model-fallback tests prove the runner redirects internally and returns the target without fallback attempts.

## Solution Assessment

No. Comparing final provider/model values is not the best fix because it loses whether the final selection came from primary execution, automatic fallback, or a user-requested live switch.

  - confidence: 0.86

  - confidence: 0.82
