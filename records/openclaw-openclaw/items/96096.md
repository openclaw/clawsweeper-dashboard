## Summary

Keep open for maintainer review: current main still lets explicit cron cloud stream-idle waits last for the full cron run timeout, while the current PR head narrows that behavior to a 60s cloud cap with local/self-hosted exceptions and I found no concrete line-level defect.

## What This Changes

The PR changes cron-triggered embedded agent LLM idle-timeout selection so remote/cloud stream stalls use a shorter cap while local or self-hosted provider routes can retain explicit longer cron timeouts, and documents that timeout behavior.

## Best Possible Solution

Land the focused timeout-policy change only if maintainers accept the 60-second cloud cron idle cap, while keeping broader cron budget exhaustion work separate.

## Reproduction Assessment

Yes, at source level: current main returns the full explicit cron run timeout from resolveLlmIdleTimeoutMs, so an opened-but-silent stream can use the same budget as the cron watchdog. I did not run a live provider that opens a stream and sends no chunks.

## Solution Assessment

Yes for the narrow stream-stall fallback bug: the patch changes the embedded stream-idle timeout at the owner boundary and relies on the existing timeout-classified fallback loop. It is not a complete answer to broader cron remaining-budget exhaustion, which should stay separate.
