## Summary

Keep open: the fix targets the right cron stream-stall/fallback path, but it introduces a compatibility blocker for hostname-based self-hosted provider cron jobs that previously relied on explicit job timeouts.

## What This Changes

The PR caps explicit cron cloud stream idle waits at 60s and adds focused helper tests for cron cloud, short cron, and loopback local-provider cron cases.

## Best Possible Solution

Preserve explicit cron timeouts for the local/self-hosted cases the PR promises to preserve, while keeping the new shorter cap for remote cloud cron stream stalls.

## Reproduction Assessment

Yes, source-level: current main gives explicit cron run timeouts the full idle watchdog, and the PR changes that behavior based on local-provider classification. The compatibility defect is source-reproducible from the helper’s documented hostname limitation; no live local-hostname provider was run.

## Solution Assessment

No as written: the approach is good for remote cloud stalls, but the local-provider preservation is too narrow for hostname-based self-hosted setups. The safer solution is to extend the preservation logic or make the breaking policy explicit with maintainer approval.
