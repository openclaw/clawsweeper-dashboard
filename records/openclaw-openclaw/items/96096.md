## Summary

Keep open: the PR targets a valid cron fallback bug, but the new 60s cron cloud idle cap applies before the code knows whether another fallback candidate actually remains, so strict or single-candidate cron jobs with longer explicit `timeoutSeconds` can start failing at 60s with no recovery path. The contributor proof is useful helper-level evidence, but it does not yet exercise an opened-but-silent stream through a cron fallback run.

## What This Changes

The PR caps cron-triggered cloud model stream idle waits at 60s while preserving shorter cron timeouts and detected local-provider explicit cron timeouts, with focused idle-timeout helper tests.

## Best Possible Solution

Apply the shorter stream-stall budget only when another fallback candidate can actually run, preserve explicit cron/provider timeout semantics for final or strict attempts, and add focused proof for a stalled stream advancing through cron fallback.

## Reproduction Assessment

Yes at source level: current code uses the explicit cron timeout as the idle ceiling, while the PR changes that to 60s for non-local cron model calls. I did not run a live opened-but-silent provider replay.

## Solution Assessment

No, not yet: the fix is in the right helper but is too broad because it caps final or strict attempts that cannot fall forward. The safer shape is to use the existing final-attempt/fallback state before shortening the idle budget.
