## Label Justifications

- `P2`: This is a clear public plugin API gap with limited blast radius, but it needs API ownership review rather than urgent runtime handling.
- `impact:session-state`: The requested cancellation API is about preventing stale in-progress runs from producing stale responses or drifting conversation/session state.

## Summary

Keep open: current main and v2026.5.22 still do not expose `api.runtime.agent.abort` on the public plugin runtime. Gateway abort plumbing and hook `runId` context now exist, but publishing a runId-based plugin abort API still needs an explicit ownership and authorization decision.

## What This Changes

Request a new public plugin runtime method, `api.runtime.agent.abort({ runId })`, so plugins can cancel in-progress agent runs.

## Best Possible Solution

Add a documented, additive plugin runtime cancellation seam backed by existing abort plumbing, with explicit run/session ownership rules, SDK mock support, docs, API coverage, and focused runtime/gateway tests.

## Reproduction Assessment

Yes. Static source inspection of current main and v2026.5.22 shows the runtime type, builder, SDK mock, and docs lack `api.runtime.agent.abort`, while hook run IDs and gateway abort plumbing exist.

## Solution Assessment

Unclear. The plugin runtime is the right boundary, but the proposed runId-only/admin-scope shape needs maintainer judgment on ownership and authorization before implementation.
