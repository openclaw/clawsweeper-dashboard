## Summary

Keep open. Current main still routes Google Chat thread placement through `replyToId` only, does not populate the shared `MessageThreadId` for queued follow-ups, and https://github.com/openclaw/openclaw/pull/74235 remains the active open implementation lane rather than shipped code.

## What This Changes

Preserve Google Chat thread/reply routing metadata across streamed block replies and queued follow-up replies so chunks remain in the inbound thread.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned fix, then close this issue with the merged commit and release proof.

## Reproduction Assessment

Yes. The issue and comments give concrete DM, threaded-space, long-reply, and concurrent-follow-up repro paths, and current main source still lacks the shared Google Chat thread context needed for those paths.

## Solution Assessment

Yes in direction. The maintainable fix is to carry a validated Google Chat thread resource through plugin-owned ingress, reply delivery, queued follow-up drain, and message-tool sends while keeping resource-shape validation at the Google Chat boundary.
