## Summary

Keep open. Current main still has the Google Chat thread target carried only through `replyToId`, while block-stream coalescing and queued follow-up routing do not preserve a separate Google Chat thread resource; open PR #74235 remains the active fix lane rather than a merged implementation.

## What This Changes

Google Chat replies should preserve inbound thread routing metadata across streamed block reply chunks and queued follow-up drains.

## Best Possible Solution

Land a consolidated Google Chat-owned fix that carries validated `spaces/.../threads/...` routing through webhook context, natural/block reply delivery, queued follow-up drain, and message-tool sends, with the queue schema mitigation handled in the same PR or a clearly linked follow-up.

## Reproduction Assessment

Yes. The issue gives concrete DM, threaded space, long-reply, and concurrent-message repro paths, and current source inspection shows the missing `MessageThreadId` plus replyTo-only delivery path that explains those symptoms.

## Solution Assessment

Yes in direction, but the exact implementation should stay additive and maintain the plugin/core boundary. The narrow maintainable path is to keep core generic while the Google Chat plugin validates and carries Chat thread resources across every delivery path.
