- `P1`: This is a confirmed Google Chat channel workflow bug that can misroute real user replies out of the intended thread.
- `impact:message-loss`: The reported failure misroutes streamed and queued Google Chat replies to the wrong visible conversation location.
- `impact:session-state`: The bug is caused by losing per-turn thread routing context across streamed replies and queued follow-up runs.

## Summary

Keep open. Current main and the latest release still carry Google Chat thread routing mainly through `replyToId` instead of a validated shared thread fact, while the active fix path remains the unmerged https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

The issue reports Google Chat streamed and queued replies losing thread metadata, causing reply chunks or follow-ups to post at the space top level instead of the inbound thread.

## Best Possible Solution

Land or replace the Google Chat-owned fix that carries a validated Chat thread resource through ingress, reply delivery, queued follow-up drain, and message-tool sends while keeping resource-shape normalization inside the plugin boundary.

## Reproduction Assessment

Yes, source-level. The issue gives concrete threaded-space, DM, long-reply, and concurrent-follow-up repro paths, and current main still lacks the shared Google Chat thread context needed to preserve those paths.

## Solution Assessment

Yes in direction, but it is not shipped yet. The maintainable solution is a Google Chat boundary fix that preserves a valid thread resource through existing reply, follow-up, and message-tool plumbing rather than broad channel-specific core policy.
