## Summary

Keep open. Current main still carries the Google Chat thread mainly as `replyToId` rather than a shared thread fact, and the active fix path is the unmerged https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

The issue reports Google Chat streamed and queued replies losing the inbound thread target and posting chunks or follow-ups at the space top level.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned fix that carries a validated thread resource through ingress, reply delivery, queued follow-up drain, and message-tool sends.

## Reproduction Assessment

Yes, source-level. The issue and comments give concrete threaded-space, DM, long-reply, and concurrent-follow-up repro paths, and current main still lacks the shared Google Chat thread context needed for those paths.

## Solution Assessment

Yes in direction, but it is not shipped yet. The maintainable fix is to keep Google Chat resource-shape normalization inside the plugin boundary while carrying a generic thread fact through reply, follow-up, and message-tool plumbing.
