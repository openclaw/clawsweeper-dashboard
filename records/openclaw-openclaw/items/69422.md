## Label Justifications

- `P1`: The report describes a real Google Chat channel workflow where replies from production deployments can land in the wrong thread or top-level space.
- `impact:message-loss`: The affected behavior is misrouted Google Chat reply delivery across streamed chunks and queued follow-ups.

## Summary

Keep open. Current main still routes Google Chat thread placement through `replyToId` rather than a durable Google Chat thread resource, and the active fix remains the open, conflicting https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

The issue reports that Google Chat streamed replies and queued follow-ups can lose the originating thread target and post to the space top level.

## Best Possible Solution

Rebase, review, and land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned fix that carries a validated thread resource through ingress, reply delivery, queued follow-up drain, and message-tool sends.

## Reproduction Assessment

Yes, with source-level confidence. The issue and comments provide concrete threaded-space, DM, long-reply, and concurrent-follow-up paths, and current main still lacks the shared Google Chat thread fact those paths need.

## Solution Assessment

Yes in direction, but it is not landed yet. The maintainable fix is to keep Google Chat resource-name normalization inside the plugin boundary while carrying a generic thread fact through reply, follow-up, and message-tool plumbing.
