## Label Justifications

- `P1`: The issue describes a reproducible Google Chat workflow where replies can land in the wrong thread or top-level space for real deployments.
- `impact:message-loss`: The affected behavior misroutes outbound channel messages away from the conversation thread where users expect them.

## Summary

Keep open. Current main still routes Google Chat thread state mostly through `replyToId`, while the broad implementation fix is still the open https://github.com/openclaw/openclaw/pull/74235 rather than shipped code.

## What This Changes

The issue reports Google Chat outbound replies losing their inbound thread target across streamed chunks and queued follow-up replies.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74235, or an equivalent Google Chat-owned fix that carries a validated thread resource through ingress, queued follow-up drain, message-tool sends, and reply delivery.

## Reproduction Assessment

Yes, source-level. The issue and comments give concrete threaded-space, DM, long-reply, and concurrent-follow-up paths, and current main still drops the distinct Google Chat thread fact before shared routing can use it.

## Solution Assessment

Yes in direction, but it is not shipped yet. The maintainable path is to keep Google Chat resource-shape normalization inside the plugin while carrying a generic thread fact through reply, follow-up, and message-tool plumbing.
