## Summary

Keep this open. Current main still lacks a separate Google Chat thread resource in the inbound turn context and still depends on `replyToId` for Google Chat delivery, block streaming, and queued follow-up routing; PR #74235 is the active draft fix candidate but is not merged.

## What This Changes

Preserve Google Chat thread/reply routing metadata across streamed block replies and queued follow-up replies so all outbound chunks stay in the inbound thread.

## Best Possible Solution

Land a consolidated Google Chat-owned fix that preserves validated `spaces/.../threads/...` thread resources through webhook context, natural/block reply delivery, queued follow-up drain, and message-tool sends; include the `messages.queue.byChannel.googlechat` schema mitigation or split it into a clearly linked follow-up.

## Reproduction Assessment

Yes. The issue gives concrete DM/space and concurrent-message repro steps, and current source inspection shows the missing `MessageThreadId` plus replyTo-only delivery path that explains those symptoms.

## Solution Assessment

Unclear for the open draft PR until maintainer review finishes. The narrow maintainable direction is to keep core generic while the Google Chat plugin validates and carries Chat thread resource names across every delivery path.
