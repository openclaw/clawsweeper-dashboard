## Summary

Keep open. Current main still lacks a durable Google Chat thread route fact for streamed replies and queued follow-ups, while the active implementation path remains the open, conflicting Google Chat thread-preservation PR.

## What This Changes

The issue asks OpenClaw to keep Google Chat streamed replies, continuation chunks, and queued follow-ups routed to the inbound Chat thread instead of leaking to the space top level.

## Best Possible Solution

Rebase, review, and land https://github.com/openclaw/openclaw/pull/74235 or an equivalent Google Chat-owned fix that carries a validated thread resource through ingress, reply delivery, queued follow-up drain, and message-tool sends.

## Reproduction Assessment

Yes, with source-level confidence. The issue and comments provide concrete threaded-space, DM, long-reply, and concurrent-follow-up paths, and current main still lacks the shared Google Chat thread fact those paths need.

## Solution Assessment

Yes in direction, but it has not landed. The maintainable fix is to keep Google Chat resource-name normalization inside the plugin boundary while carrying a generic thread fact through reply, follow-up, and message-tool plumbing.
