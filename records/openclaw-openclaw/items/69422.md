## Summary

Keep open. Current main still carries Google Chat thread routing through `replyToId` only and does not populate a generic thread context for queued follow-ups; https://github.com/openclaw/openclaw/pull/74235 is the active unmerged fix lane rather than shipped implementation.

## What This Changes

The issue asks for Google Chat streamed replies and queued follow-ups to preserve the inbound Chat thread target instead of falling back to top-level space messages.

## Best Possible Solution

Land a consolidated Google Chat-owned fix that carries a validated thread resource through webhook context, natural and block reply delivery, queued follow-up drain, and message-tool sends, with the queue schema workaround included or tracked separately.

## Reproduction Assessment

Yes. The issue and comments give concrete DM, threaded space, long-reply, and concurrent follow-up repro paths, and current main source shows Google Chat does not populate `MessageThreadId` even though follow-up routing depends on it.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to keep Google Chat resource-shape validation and fallback logic inside the plugin boundary while using the existing generic `MessageThreadId`/reply routing seams.
