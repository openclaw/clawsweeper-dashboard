## Summary

Keep open. Current main still routes the Google Chat thread resource through `replyToId` only, while block coalescing and queued follow-up drain do not preserve a separate Google Chat thread target; open PR #74235 remains the active fix lane rather than merged code.

## What This Changes

Google Chat streamed block replies and queued follow-up replies should preserve the inbound thread target instead of leaking chunks or drained replies to the space top level.

## Best Possible Solution

Land a consolidated Google Chat-owned fix that carries validated thread resources through inbound context, natural/block reply delivery, queued follow-up drain, and message-tool sends, with the `messages.queue.byChannel.googlechat` mitigation handled in the same PR or a linked follow-up.

## Reproduction Assessment

Yes. The issue gives concrete DM, threaded-space, long-reply, and concurrent-message repro paths, and current source inspection shows the missing `MessageThreadId`/thread carrier plus replyTo-only delivery path that explains them.

## Solution Assessment

Yes in direction, but it is not implemented on current main. The maintainable path is an additive Google Chat plugin fix plus any necessary generic core seam, without hard-coding bundled channel policy into core.
