## Summary

Keep open. Current main still does not implement `publicMode`, and this PR remains a useful implementation candidate, but the branch is not merge-ready: default-only `agents.defaults.publicMode` can still be skipped when no agent entry exists, streamed OpenResponses omits trusted proxy data, and the new owner header needs a rebase/design pass against the current compat HTTP `x-openclaw-scopes` model.

## What This Changes

This PR adds `publicMode` agent/default config, prompt and skills redaction, owner-trust header handling for OpenAI-compatible HTTP ingress, docs, and targeted tests.

## Best Possible Solution

Rebase the PR onto current main, fold public-facing agent trust into the existing compat HTTP auth/scope contract, then fix the remaining default-resolution and streaming parity bugs with focused tests and a changelog entry before normal maintainer review.

## Reproduction Assessment

Yes. The default-only bug reproduces by setting `agents.defaults.publicMode: true` without an `agents.list` entry and following the PR's `resolveAgentPublicMode()` path; the streaming mismatch is visible in the PR diff where the streamed `/v1/responses` call omits `trustedProxies`.

## Solution Assessment

No, not yet. The feature direction is reasonable, but the current branch is not the narrowest maintainable solution until it reconciles with main's `x-openclaw-scopes` owner model and makes default and streaming behavior consistent.

## Security Review

Status: needs_attention

Summary: The PR changes gateway owner/non-owner semantics and still has concrete trust-boundary gaps that should be fixed before merge.

Concerns:

- **[medium] Default-only publicMode is not enforced:** `src/agents/agent-scope.ts:153`
  - body: Operators can set `agents.defaults.publicMode: true` without listing the default agent, but the PR resolves public mode through `resolveAgentConfig()` and returns false when no entry exists. That can leave runtime and workspace metadata exposed despite the public-mode default.
  - confidence: 0.9
- **[medium] Streamed responses ignore trusted-proxy owner assertions:** `src/gateway/openresponses-http.ts:755`
  - body: The streamed OpenResponses call omits `trustedProxies`, so the new owner header cannot be honored for trusted upstreams in that path. This creates a security-boundary inconsistency between stream and non-stream requests.
  - confidence: 0.86
- **[medium] Owner header needs reconciliation with current scopes:** `src/gateway/http-utils.ts:24`
  - body: Current main derives compat HTTP owner semantics from shared-secret auth and `x-openclaw-scopes` / `operator.admin`. Adding a second owner header without a clear precedence relationship risks confusing deployments and weakening the gateway trust boundary documentation.
  - confidence: 0.78
