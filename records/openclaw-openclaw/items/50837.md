## What This Changes

The PR adds agent/default publicMode configuration, prompt and skills redaction, owner-trust HTTP header handling for OpenAI-compatible ingress, docs, and targeted tests.

## Best Possible Solution

Rebase onto current main and integrate public-facing agent mode as a security-reviewed config feature aligned with the compat HTTP scope contract, with default-only inheritance, streaming parity, regression coverage, and redacted real gateway proof.

Yes for the PR defects by source inspection of the PR head: default-only agents.defaults.publicMode is skipped when no agent entry exists, and streamed /v1/responses omits trustedProxies. There is no current-main bug reproduction because publicMode is a new feature absent from main.

## Solution Assessment

No, not yet. The feature direction is plausible, but the patch should first fit the current compat HTTP scope model, fix default-only inheritance and streaming parity, and provide real gateway proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Honor default-only public mode:** `src/agents/agent-scope.ts:155-156`
  - body: resolveAgentPublicMode() delegates to resolveAgentConfig(), which returns undefined before reading agents.defaults.publicMode when the target agent has no agents.list entry. A default-only single-agent config therefore skips prompt redaction and non-owner ingress defaults even though the operator enabled public mode.
  - confidence: 0.91
- **[P2] Pass trusted proxies to streamed responses:** `src/gateway/openresponses-http.ts:758`
  - body: The streamed /v1/responses path calls runResponsesAgentCommand() with publicMode but without trustedProxies, while the non-stream path passes trustedProxies. Trusted upstreams sending x-openclaw-sender-is-owner: true are downgraded only in streaming mode, so owner-only behavior diverges by response mode.
  - confidence: 0.88
