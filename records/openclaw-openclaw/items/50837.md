## Summary

Keep open: current main does not implement `publicMode`, and the PR still has blocking source-inspectable defects around default inheritance and streamed OpenResponses owner semantics, plus missing real behavior proof for the changed gateway behavior.

## What This Changes

Adds per-agent/default `publicMode` configuration, prompt redaction, HTTP owner-header handling, docs, and tests for public-facing agents.

## Best Possible Solution

Rebase onto current main and land a security-reviewed public-facing agent mode that honors defaults, keeps stream/non-stream behavior aligned, fits the compat HTTP scope model, and includes real gateway proof.

## Reproduction Assessment

Yes for the PR defects by source inspection of PR head: default-only `agents.defaults.publicMode` is skipped when no agent entry exists, and streamed `/v1/responses` omits `trustedProxies`. There is no current-main bug reproduction because `publicMode` is a new feature absent from main.

## Solution Assessment

No, not yet. The direction is plausible, but the patch needs default-only inheritance, streaming parity, reconciliation with current HTTP auth semantics, and real gateway proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Honor default-only public mode:** `src/agents/agent-scope.ts:155-156`
  - body: `resolveAgentPublicMode()` delegates to `resolveAgentConfig()`, which returns `undefined` when the target agent has no `agents.list` entry. A config with only `agents.defaults.publicMode: true` therefore skips prompt redaction and non-owner ingress defaults despite enabling public mode.
  - confidence: 0.91
- **[P2] Pass trusted proxies in streamed responses:** `src/gateway/openresponses-http.ts:748-759`
  - body: The streamed `/v1/responses` path calls `runResponsesAgentCommand()` without `trustedProxies`, while the non-stream branch passes it. Trusted upstreams sending `x-openclaw-sender-is-owner: true` are downgraded only for streamed requests, creating a stream/non-stream behavior mismatch.
  - confidence: 0.88

## Security Review
