## Summary

Keep open. Current main still has a source-reproducible diagnostic gap: embedding provider fetch failures can travel through the documented CLI command and be printed as `String(err)`, while pre-response remote embedding transport errors still miss provider context.

## What This Changes

The issue asks OpenClaw to make `openclaw infer embedding create --provider voyage` succeed on Node 24 or at least surface the underlying fetch cause instead of only `TypeError: fetch failed`.

## Best Possible Solution

Reuse the existing redacted cause-aware formatter at the CLI boundary and wrap pre-response remote embedding transport failures with provider context while leaving the guarded-fetch/SSRF dispatcher design intact.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused fetch failure from `embedBatch` reaches the default command catch and is printed with `String(err)`. The exact live Voyage transport failure still needs credential-backed Node 24 confirmation.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse existing redacted cause-aware error formatting in the CLI default catch path and add shared transport wrapping for pre-response embedding HTTP failures.
