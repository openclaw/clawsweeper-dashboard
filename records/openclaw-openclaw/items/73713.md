## Summary

Keep open. Current main still sends `openclaw infer embedding create` failures through the default CLI wrapper that prints `String(err)`, so nested `fetch` causes and `UND_ERR_*` codes can still be lost. The Voyage provider path uses the shared guarded remote embedding helper without a Voyage-local `fetchImpl`, which makes this a narrow cause-preservation/diagnostic bug rather than an implemented or duplicate cleanup item.

## What This Changes

The issue asks OpenClaw to audit and fix the Voyage `openclaw infer embedding create` path on Node 24 so embedding requests either succeed or surface the underlying `fetch`/Undici cause instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware while preserving the guarded fetch/SSRF architecture. The desired end state is provider-contextual errors that include nested fetch cause messages and stable codes after redaction; runtime dependency or Undici changes should wait for a live Node 24 smoke that proves the transport layer is still broken after diagnostics are visible.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause currently reaches `runtime.error(String(err))` on the embedding CLI path. Unclear for the live Voyage transport failure because this read-only review did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes, the narrow maintainable solution is to use the existing cause-aware formatter in the CLI wrapper and/or wrap pre-response remote embedding transport errors with provider context while preserving the original cause. Changing provider selection, credentials, SSRF policy, or Undici package pins would be premature until the hidden cause is visible.
