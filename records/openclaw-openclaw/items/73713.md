## Summary

Keep open. Current main still has the actionable diagnostic gap: `infer embedding create` lets provider embedding failures reach the default CLI wrapper, which prints `String(err)`, and the shared remote embedding HTTP helper still rethrows pre-response guarded-fetch failures before adding provider context. The related fetch-failure items are adjacent but do not supersede this narrower cause-preservation bug.

## What This Changes

Audit the `openclaw infer embedding create` Voyage embedding path and preserve nested `fetch` causes/codes instead of reporting only `TypeError: fetch failed`.

## Best Possible Solution

Keep the guarded-fetch and SSRF architecture, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware: show redacted provider context, nested cause messages, and stable error codes such as `UND_ERR_*` without changing credentials, provider selection, SSRF policy, or Undici pins before the real root cause is visible.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause reaches `embedding create` through the default CLI wrapper that prints `String(err)`. No high-confidence live reproduction of the Voyage/Node transport root was possible in this read-only review because no Voyage credential was available.

## Solution Assessment

Yes, the best next fix is narrow: use or extend the existing cause-aware formatter in the CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving original causes/codes. A dependency or transport-policy change would be premature until the underlying cause is visible.
