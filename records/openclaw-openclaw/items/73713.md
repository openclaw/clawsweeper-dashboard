## Summary

Keep open. Current main still leaves the actionable diagnostic bug unresolved: the embedding CLI can route provider failures through a `String(err)` wrapper, and the shared remote embedding HTTP path can let pre-response guarded-fetch transport exceptions escape without provider context or nested cause/code details.

## What This Changes

The issue asks OpenClaw to audit the `openclaw infer embedding create` Voyage path and surface nested fetch cause/code details instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware and code-aware while preserving the existing guarded fetch and SSRF architecture. Provider-contextual failures should include redacted nested cause messages and stable codes such as Undici `UND_ERR_*` values; dependency or transport changes should wait for live proof after diagnostics are visible.

## Reproduction Assessment

Yes for the actionable diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause still reaches the embedding CLI default error path that prints `String(err)`. Unclear for the live Voyage transport failure because this read-only pass did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes, the narrow maintainable fix is to use or extend the existing cause-aware formatter in the default CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving the original cause. Changing provider selection, credential lookup, SSRF policy, or Undici pins would be premature until the underlying cause is visible.
