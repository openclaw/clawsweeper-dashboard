## Summary

Keep open. Current main still has a source-reproducible diagnostics gap: embedding provider fetch errors can reach the documented CLI command and be printed with `String(err)`, while pre-response remote embedding transport failures still lack provider context.

## What This Changes

The issue asks for `openclaw infer embedding create` with Voyage to succeed when the same request works directly, or at least surface the underlying fetch cause such as `UND_ERR_*` instead of only `TypeError: fetch failed`.

## Best Possible Solution

Reuse the existing redacted cause-aware formatter at the CLI boundary and wrap pre-response remote embedding transport failures with provider context while keeping the guarded SSRF/runtime-fetch architecture shared.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused fetch failure from `embedBatch` reaches the default command catch and can be printed as `String(err)`. The exact live Voyage transport failure still needs credential-backed Node 24 confirmation.

## Solution Assessment

Yes. The narrow maintainable repair is to reuse cause-aware error formatting in the CLI default catch path and add shared provider-context wrapping for pre-response embedding HTTP failures, not a Voyage-only workaround or new config flag.
