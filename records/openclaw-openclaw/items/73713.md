## Summary

Keep open. Current main still leaves the reported diagnostic failure unresolved: the embedding CLI uses the default runtime wrapper that prints `String(err)`, while the Voyage provider and shared remote embedding helper can still let pre-response guarded-fetch transport exceptions escape without provider context or nested cause/code details.

## What This Changes

The issue requests auditing the `openclaw infer embedding create` Voyage embedding path on Node 24 and preserving the underlying fetch cause, including Undici error details, instead of reporting only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware and code-aware while preserving the existing guarded fetch and SSRF architecture. Provider-contextual failures should include redacted nested cause messages and stable error codes such as Undici `UND_ERR_*` values; transport or dependency changes should wait for live proof after diagnostics are visible.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause still reaches the embedding CLI default error path that prints `String(err)`. Unclear for the live Voyage transport failure because this read-only review did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes, the best next fix is narrow: use or extend the existing error formatter in the default CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving the original cause. Changing provider selection, credentials, SSRF policy, or Undici package pins would be premature until the underlying cause is visible.
