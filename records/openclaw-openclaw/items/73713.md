## Summary

Keep open. Current main still has the diagnostic bug: the embedding CLI path can collapse nested provider transport failures to `TypeError: fetch failed`, and the shared remote embedding helper does not wrap pre-response guarded-fetch exceptions with provider context.

## What This Changes

The issue asks for `openclaw infer embedding create --provider voyage` to succeed on the reported Node 24/Voyage setup and, when fetch fails, to surface nested fetch causes such as Undici error codes instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware while preserving the guarded fetch and SSRF design. The end state should report provider context plus nested fetch cause messages and stable codes, with Undici/runtime dependency changes considered only after the underlying cause is visible.

## Reproduction Assessment

Yes for the diagnostic gap: a `TypeError('fetch failed')` with a nested cause currently reaches `runtime.error(String(err))`, which loses the cause/code. Unclear for the live Voyage transport failure because this read-only review did not run the reporter's Node 24 credentialed smoke.

## Solution Assessment

Yes, the narrow maintainable repair is to use cause-aware formatting in the default CLI wrapper and/or wrap pre-response remote embedding transport errors with provider context while preserving the original cause. Changing provider auth, default selection, Undici pins, or SSRF policy would be premature without the surfaced cause proving that layer is still broken.
