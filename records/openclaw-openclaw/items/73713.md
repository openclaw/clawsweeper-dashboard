## Summary

Keep open. Current main still preserves the actionable diagnostic gap: the supported embedding CLI can route a caused provider fetch failure through a default error handler that prints only `String(error)`, and the shared remote embedding HTTP boundary still does not wrap pre-response guarded-fetch failures with provider context.

## What This Changes

The issue requests that `openclaw infer embedding create` succeed for valid Voyage embedding requests and, when fetch fails, surface the underlying fetch cause or `UND_ERR_*` details instead of only `TypeError: fetch failed`.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware with redacted provider context and stable nested codes when available.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused fetch failure can reach embedding create through a default catch path that prints only `String(error)`. The exact live Voyage transport failure still needs credential-backed confirmation.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extend the existing redacted cause-aware formatter in the CLI default catch path and wrap pre-response remote embedding transport errors while preserving the original cause/code; changing Undici pins or SSRF transport policy would be premature before the hidden cause is surfaced.
