## Summary

Keep open. Current main still leaves the actionable diagnostic bug unresolved: the embedding CLI uses the default command wrapper that prints `String(err)`, while the shared remote embedding HTTP path lets guarded-fetch transport exceptions escape without provider context or stable nested cause/code details. The open related memory rollback PR explicitly does not fix transport/provider failures, so it does not supersede this narrower CLI cause-preservation issue.

## What This Changes

Issue requests that `openclaw infer embedding create` succeed for a valid Voyage embedding request on Node 24 when the guarded/direct request succeeds, and that provider embedding failures preserve nested fetch causes such as Undici `UND_ERR_*` details.

## Best Possible Solution

Keep the existing guarded-fetch and SSRF architecture, then make the embedding CLI and shared remote embedding HTTP path cause-aware and provider-contextual. The fix should expose redacted nested cause messages and stable error codes such as Undici `UND_ERR_*` values before considering dependency pins, provider selection changes, or broader memory retry behavior.

## Reproduction Assessment

Yes for the actionable diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause still reaches the `embedding create` default CLI wrapper that prints `String(err)`. Unclear for the live Voyage transport failure because this read-only review did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes, the narrow maintainable solution is to use or extend the existing cause-aware formatter in the CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving the original cause/code. Changing credentials, SSRF policy, provider selection, or Undici pins would be premature until the hidden cause is visible.
