## Summary

Keep open. Current main still has the diagnostic gap described in the report: the embedding CLI uses the default runtime wrapper that prints `String(err)`, the embedding provider error is allowed to bubble from `embedBatch`, and the shared remote HTTP helper does not wrap pre-response transport failures with provider context or nested cause/code details. The Voyage path uses the shared remote embedding helper without a Voyage-local custom fetch implementation, so this remains a narrow cause-preservation bug rather than an implemented cleanup item.

## What This Changes

Audit the `openclaw infer embedding create` Voyage embedding path on Node 24 and surface nested fetch causes or `UND_ERR_*` details instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware while preserving the guarded fetch and SSRF architecture. The end state should show provider-contextual, redacted errors that include nested fetch cause messages and stable codes; dependency pins, provider selection, credentials, or SSRF bypasses should wait for a live smoke that proves the transport layer still fails after the hidden cause is visible.

## Reproduction Assessment

Yes for the diagnostic gap: by code inspection, a provider-thrown `TypeError("fetch failed")` with a nested cause reaches `runtime.error(String(err))` on the `embedding create` path. Unclear for the live Voyage transport failure because this read-only review did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing cause-aware formatter in the CLI wrapper and/or wrap pre-response remote embedding transport errors with the provider error prefix while preserving the original cause. Bypassing the guarded fetch path or changing Undici/package policy would be premature before the underlying cause is surfaced.
