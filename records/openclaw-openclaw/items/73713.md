## Summary

Keep open. Current main still leaves the actionable diagnostic gap unresolved: the infer embedding CLI uses the default command wrapper that flattens nested fetch causes, and the shared remote embedding HTTP path can still surface guarded-fetch transport failures without provider context or stable nested codes. The earlier ClawSweeper comment remains consistent with current main; related fetch/Undici reports cover adjacent transport behavior, but none supersedes this narrower cause-preservation issue.

## What This Changes

The issue requests an audit of the `openclaw infer embedding create` Voyage embedding path on Node 24 and a fix so provider fetch failures expose the underlying fetch cause or `UND_ERR_*` code instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware and code-aware while preserving the existing guarded-fetch and SSRF architecture. Provider-contextual failures should include redacted nested cause messages and stable codes such as Undici `UND_ERR_*`; dependency or transport changes should wait for live proof after diagnostics are visible.

## Reproduction Assessment

Yes for the actionable diagnostic gap: a mocked provider or guarded fetch rejection shaped like `TypeError("fetch failed")` with a nested cause still reaches the embedding CLI default error path that prints `String(err)`. Unclear for the live Voyage transport failure because this read-only review did not run Node 24 with a Voyage credential.

## Solution Assessment

Yes. The narrow maintainable fix is to use or extend the existing cause-aware formatter in the default CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving the original cause; changing provider selection, credential lookup, SSRF policy, or Undici pins would be premature until the underlying cause is visible.
