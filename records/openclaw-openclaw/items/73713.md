## Summary

Keep open. Current main still has the actionable diagnostic gap: the embedding CLI path lets provider transport errors reach the default CLI wrapper, which prints `String(err)` and drops nested `cause` details, while the shared remote embedding HTTP boundary still does not add provider context around pre-response guarded-fetch failures. The existing bot comment remains consistent with current source; related fetch-failure reports are adjacent, not a canonical replacement for this narrower cause-preservation bug.

## What This Changes

The issue asks for `openclaw infer embedding create` with the Voyage embedding provider to surface the underlying fetch cause, such as `UND_ERR_*`, instead of only `TypeError: fetch failed`.

## Best Possible Solution

Keep the guarded-fetch and SSRF architecture intact, but make the embedding CLI and shared remote embedding HTTP boundary cause-aware. The desired end state is a redacted provider-context error such as `voyage embeddings failed: fetch failed | invalid onRequestStart method | UND_ERR_INVALID_ARG`, without changing credentials, provider selection, Undici pins, or transport policy before the hidden cause is visible.

## Reproduction Assessment

Yes for the diagnostic gap: a unit-level provider failure shaped like `TypeError("fetch failed")` with a nested cause can be sent through `embedding create` and currently reaches the default CLI wrapper that prints `String(err)`. No live Voyage/Node 24 transport reproduction was performed in this read-only review.

## Solution Assessment

Yes, the narrow maintainable repair is to use or extend the existing cause-aware formatter in the CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving original causes/codes. A dependency pin or transport-policy change would be premature until the underlying cause is actually visible.
