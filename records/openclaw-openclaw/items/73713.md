## Summary

Keep open. Current main still has the actionable diagnostic gap: `infer embedding create` routes provider transport failures through the default CLI wrapper that flattens nested causes, and the shared remote embedding HTTP boundary does not add provider context around pre-response guarded-fetch failures.

## What This Changes

The issue requests auditing the Voyage embedding CLI transport path on Node 24 and preserving nested fetch causes for provider embedding failures.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware with redacted provider context and stable nested codes such as `UND_ERR_*`.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause reaches `embedding create` through the default CLI wrapper that prints `String(err)`. The exact Voyage/Node 24 transport failure still needs live credential-backed confirmation.

## Solution Assessment

Yes, the narrow maintainable solution is to reuse or extend the existing cause-aware formatter in the CLI default catch path and wrap pre-response remote embedding transport errors while preserving their original cause/code. Changing Undici pins or SSRF transport policy would be premature before the hidden cause is surfaced.
