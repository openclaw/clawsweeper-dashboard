## Summary

Keep open. Current main still has the actionable diagnostic gap: embedding creation provider failures flow through the default CLI wrapper that prints `String(err)`, and the shared remote embedding POST path does not add provider context around pre-response guarded-fetch transport failures. The related fetch-failure reports are adjacent but do not supersede this narrower CLI cause-preservation issue.

## What This Changes

The issue asks for `openclaw infer embedding create` on Voyage/Node 24 to avoid fetch/dispatcher mixing and surface nested fetch causes such as Undici codes instead of only `TypeError: fetch failed`.

## Best Possible Solution

Keep the existing guarded-fetch and SSRF architecture, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware. The end state should show redacted provider context plus nested cause messages and stable codes such as `UND_ERR_*`, without changing credentials, provider selection, SSRF policy, or Undici pins until the real underlying cause is visible.

## Reproduction Assessment

Yes for the actionable diagnostic gap: a provider rejection shaped like `TypeError("fetch failed")` with a nested cause reaches `embedding create` through the default CLI wrapper that prints `String(err)`. No high-confidence live reproduction of the Voyage/Node 24 transport root was performed in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to use or extend the existing cause-aware formatter in the CLI wrapper and wrap pre-response remote embedding transport errors with provider context while preserving original causes/codes. A dependency pin or transport policy change would be premature before the hidden cause is surfaced.

## Repair Work Prompt

Fix the diagnostic bug reported in https://github.com/openclaw/openclaw/issues/73713. Current main routes `openclaw infer embedding create` provider failures through `runCommandWithRuntime`, whose default catch path prints `String(err)` and drops nested `error.cause` details.
