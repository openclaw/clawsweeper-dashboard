## Summary

Keep open. Current main still leaves the actionable diagnostics gap: the documented embedding CLI can send caused provider fetch failures through a default `String(err)` error path, and the shared remote embedding HTTP helper still does not attach provider context to pre-response guarded-fetch failures.

## What This Changes

The issue asks OpenClaw to audit the Voyage embedding CLI fetch path on Node 24 and preserve underlying `fetch` cause or `UND_ERR_*` details when provider embedding requests fail.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware with redacted provider context and stable nested codes when available.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused provider transport failure can still flow through `embedding create` into a default CLI handler that prints only `String(err)`. The exact live Voyage transport failure still needs credential-backed confirmation.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extend existing redacted cause-aware formatting in the CLI default catch path and wrap pre-response remote embedding transport errors while preserving the original cause/code.
