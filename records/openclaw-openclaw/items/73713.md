## Summary

Keep open. Current main still has the actionable diagnostics gap: the documented embedding command propagates provider fetch failures into the default CLI catch, which stringifies the error and drops nested fetch causes, while pre-response remote embedding transport failures still miss provider context.

## What This Changes

The issue asks OpenClaw to make `openclaw infer embedding create --provider voyage` succeed on Node 24 or at least surface the underlying fetch cause instead of only `TypeError: fetch failed`.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the default CLI error path and shared remote embedding transport cause-aware with redacted provider context and preserved nested codes.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection plus a Node 24 check show a caused `TypeError("fetch failed")` reaches the default CLI stringification path and loses `UND_ERR_*` details. The exact live Voyage request failure was not credential-reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing redacted cause-aware formatter at the CLI boundary and wrap pre-response remote embedding transport failures while preserving the original cause.
