## Summary

Keep open. Current main still leaves the actionable diagnostic gap: the embedding CLI can flatten caused provider fetch failures to `String(err)`, and the shared remote embedding HTTP boundary still does not wrap pre-response guarded-fetch failures with provider context.

## What This Changes

The issue asks for the Voyage embedding CLI path to avoid Node 24 fetch/dispatcher failures when direct guarded fetch works and to surface nested fetch causes instead of only `TypeError: fetch failed`.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware with redacted provider context and stable nested codes when available.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused fetch failure can escape `withRemoteHttpResponse` and reach the embedding CLI default catch path that prints only `String(err)`. The exact live Voyage transport failure still needs Node 24 plus `VOYAGE_API_KEY` verification.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing redacted cause-aware formatter in the CLI error path and wrap pre-response remote embedding transport errors while preserving the original cause/code; changing Undici pins or SSRF transport policy would be premature.
