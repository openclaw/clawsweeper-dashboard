## Summary

Keep open. Current main still leaves an actionable diagnostic gap: the documented embedding CLI can route a caused provider fetch failure through default CLI error handling that prints only `String(error)`, and the shared remote embedding HTTP helper still lets pre-response guarded-fetch failures escape without Voyage/provider context.

## What This Changes

The issue asks OpenClaw to audit the Voyage embedding CLI fetch path on Node 24 and preserve underlying `fetch` cause or `UND_ERR_*` details when provider embedding requests fail.

## Best Possible Solution

Keep the guarded-fetch and SSRF design, then make the embedding CLI and shared remote embedding HTTP boundary cause-aware with redacted provider context and stable nested codes when available.

## Reproduction Assessment

Yes for the diagnostic gap: source inspection shows a caused fetch failure can reach embedding create through a default catch path that prints only `String(error)`. The exact live Voyage transport failure still needs credential-backed confirmation.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extend existing redacted cause-aware formatting in the CLI default catch path and wrap pre-response remote embedding transport errors while preserving the original cause/code.
