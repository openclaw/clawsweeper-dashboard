## Summary

Keep open. Current main still exposes the reported failure class: bundled MCP tools keep raw upstream schemas, LiteLLM has no provider schema hook, and the OpenAI-compatible serialization path does not recursively clean array-valued `type` fields for non-strict tool schemas.

## What This Changes

The issue asks OpenClaw to identify or sanitize MS365 MCP JSON Schemas so LiteLLM-routed OpenAI models do not reject tool calls with `Validation Error: "type"`.

## Best Possible Solution

Add focused regression coverage for an MS365-like bundled MCP schema with nested array-valued `type` fields, then normalize or clearly diagnose final effective tools for OpenAI-compatible routes without MS365-specific special casing.

## Reproduction Assessment

Yes for the source-level failure class: current main still lets raw bundled MCP schemas reach LiteLLM/OpenAI-completions serialization without recursive `type` array cleanup. The exact live MS365 tool responsible for the original 400 remains uncaptured.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed diff. The maintainable fix is a route-aware schema compatibility pass for final effective tools, not a MS365-only workaround or native-OpenAI-only helper change.
