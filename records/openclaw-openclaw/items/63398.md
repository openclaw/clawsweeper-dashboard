## Summary

Keep open. Current main still exposes the reported source-level failure class: bundled MCP/LSP tools are appended after provider schema normalization, LiteLLM registers no tool-schema compatibility hook, and the Gemini cleaner still has the array-items gaps called out in the latest comment; the related open PR only covers non-strict OpenAI required arrays.

## What This Changes

Request that OpenClaw sanitize MCP tool schemas from `@softeria/ms-365-mcp-server` before routing them through LiteLLM/OpenAI so provider validators do not reject JSON Schema `type` fields.

## Best Possible Solution

Normalize and diagnose the final effective tool set after bundled MCP/LSP materialization, and harden OpenAI/Gemini-compatible schema cleaners with MS365/workspace-mcp-like regression fixtures.

## Reproduction Assessment

Yes for the source-level failure class: current main still lets raw bundled MCP/LSP schemas enter final effective tools after the provider normalization pass, and the Gemini cleaner still preserves invalid array shapes. The exact live MS365/OpenAI 400 was not reproduced in this read-only review.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed diff. The narrow maintainable solution is a generic final-tool schema compatibility pass plus focused OpenAI/Gemini sanitizer fixes, not an MS365-only workaround; #66415 alone is too narrow for the current evidence.
