## Summary

Keep open. Current main still leaves the reported failure class unresolved: bundled MCP schemas are appended after provider normalization, LiteLLM registers no schema compatibility hook, and the OpenAI-family compat helper intentionally skips proxy/non-native OpenAI-completions routes; the newer Gemini/LiteLLM comment points to the same final-effective-tool gap, while open PR #66415 covers only a narrower OpenAI helper path.

## What This Changes

The issue reports OpenAI/LiteLLM rejecting MCP tool schemas from MS365 because unsupported JSON Schema `type` shapes reach provider validation.

## Best Possible Solution

Normalize or diagnose the final effective tool list after bundled MCP/LSP tools are added, using a provider-family-compatible path for LiteLLM OpenAI/Gemini routes and regression fixtures for nested `type` arrays, missing array `items`, and empty `items` schemas.

## Reproduction Assessment

Yes for the source-level failure class: a bundled MCP tool with nested `type: ["string", "null"]` or array schemas can still be assigned directly to `parameters` and appended to `effectiveTools` after the only provider normalization pass. The exact live MS365 tool responsible for the original HTTP 400 is still missing.

## Solution Assessment

No final solution is present on current main. The maintainable fix should cover final effective MCP/LSP tools and LiteLLM provider-family routing, not only native OpenAI helper normalization or an MS365-specific workaround.
