## Summary

Keep open. Current main still has the reported provider-schema failure class: raw bundled MCP tool schemas can reach the OpenAI-compatible transport with nested unsupported schema forms, LiteLLM has no provider schema hook, and a related open PR only covers a narrower OpenAI `required: []` case.

## What This Changes

The issue asks OpenClaw to make MCP tool schemas from `@softeria/ms-365-mcp-server` compatible with OpenAI-routed LiteLLM fallback instead of returning HTTP 400 schema validation errors.

## Best Possible Solution

Add regression coverage for MS365/workspace-like bundled MCP schemas, then normalize or diagnose final effective tools through a generic provider/model schema-compatibility path without MS365-specific special casing.

## Reproduction Assessment

Yes for the source-level failure class: current main still passes raw bundled MCP schemas into final effective tools while LiteLLM lacks a schema normalization hook. The exact live MS365 tool that produced the original HTTP 400 has not been isolated.

## Solution Assessment

No current complete solution is on main. The narrow open PR does not cover bundled MCP/LSP final-tool normalization or the newer LiteLLM/Gemini schema failures, so maintainers should choose whether to broaden it or split a focused follow-up.
