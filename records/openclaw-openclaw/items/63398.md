## Summary

Keep this open. Current main still has the reported schema path exposed: bundle MCP tools keep the upstream MCP `inputSchema`, provider normalization runs before those bundle tools are appended, LiteLLM has no provider tool-schema hook, and the OpenAI-family helper is intentionally scoped away from non-native/proxy OpenAI-completions routes.

## What This Changes

The issue asks OpenClaw to identify or sanitize MS365 MCP JSON Schemas so LiteLLM-routed OpenAI models do not reject tool calls with `Validation Error: "type"`.

## Best Possible Solution

Add a regression that feeds an MS365-like bundle MCP schema with nested array-valued `type` fields through the LiteLLM/OpenAI-completions serialization path, then normalize or diagnose those schemas after bundle tools are materialized and only for downstream routes that need OpenAI-compatible schema constraints.

## Reproduction Assessment

Yes for the source-level failure class: current main still allows raw bundle MCP schemas to be appended after provider normalization and sent through the LiteLLM/OpenAI-completions path without recursive `type` array cleanup. The exact live MS365 tool responsible for the original 400 is still uncaptured.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed diff. The likely maintainable solution is a route-aware schema compatibility pass for final effective tools, not a MS365-specific workaround or native-OpenAI-only strict helper change.
