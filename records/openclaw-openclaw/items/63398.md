## Summary

Keep this open. Current main still has the reported LiteLLM/OpenAI schema path exposed: bundle MCP tools keep raw MCP schemas, provider normalization runs before those tools are appended, LiteLLM registers no tool-schema normalization hook, and the OpenAI-family helper remains scoped to native OpenAI Responses/Codex routes rather than LiteLLM's OpenAI-completions proxy path.

## What This Changes

The issue asks OpenClaw to make MS365 MCP tool schemas compatible with OpenAI validation when those tools are routed through LiteLLM/OpenAI fallback models.

## Best Possible Solution

Add coverage that feeds an MS365-like bundle MCP schema with nested array-valued type fields through the LiteLLM/OpenAI completions serialization path, then normalize or diagnose that schema after bundle tools are materialized and only when the resolved downstream route needs OpenAI-compatible schema constraints.

## Reproduction Assessment

Yes for the source-level reproduction of the reported class: current main still allows raw bundle MCP schemas to reach LiteLLM/OpenAI after provider normalization. The exact live MS365 tool that triggered the original HTTP 400 is still uncaptured, so live end-to-end reproduction remains incomplete.

## Solution Assessment

Unclear for the related open PR because the provided PR summary targets empty required arrays, while this issue centers on OpenAI rejecting a type field such as nested type arrays. The maintainable fix should cover the full LiteLLM/OpenAI MCP schema path rather than only native OpenAI strict-tool helpers.
