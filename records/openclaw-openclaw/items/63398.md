## Summary

Keep open. Current main still has the reported failure class: bundled MCP/LSP tool schemas are appended after provider schema normalization, LiteLLM registers no tool-schema hook, and the existing OpenAI/Gemini cleaners do not cover all nested schema shapes described in the issue discussion.

## What This Changes

The issue reports provider-side rejection of raw MCP JSON Schemas from MS365/workspace MCP tools when routed through LiteLLM to OpenAI or Gemini-compatible models.

## Best Possible Solution

Normalize and diagnose the final effective tool list after bundled MCP/LSP materialization, add proxy-safe LiteLLM/provider-family schema cleanup without MS365-specific special casing, and cover the reported nested type-array and array-items schemas with regression tests.

## Reproduction Assessment

Yes for the source-level failure class: current main lets raw bundled MCP/LSP schemas enter effectiveTools after provider normalization, and the existing OpenAI/Gemini cleaners leave representative nested schema shapes untreated. The exact live MS365 offending tool has not been isolated.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed diff. The narrow maintainable solution is a generic final effective-tool schema compatibility pass plus provider/proxy regression coverage, not a MS365-only workaround.
