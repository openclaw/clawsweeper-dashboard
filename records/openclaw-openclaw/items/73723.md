## Summary

Keep open. Current main still has the same missing/unavailable Codex native hook relay path, hook readiness remains separate from live relay registration, the user-config opt-out discussed in the latest comment is not exposed by the Codex plugin schema, and open PR #73950 is an unmerged implementation candidate linked to this issue.

## What This Changes

The issue asks OpenClaw to repair or clearly report stale Codex native hook relay registrations after gateway restart or session recovery when ordinary hook readiness is green but PreToolUse still fails closed.

## Best Possible Solution

Land or replace PR #73950 with a narrow fix that makes stale or duplicate Codex native relay registrations recoverable or explicitly unhealthy, and decide whether to add a documented `nativeHookRelay.enabled=false` escape hatch as a separate unblock path without weakening relay security semantics.

## Reproduction Assessment

Unclear. The issue includes detailed live logs and current main still contains the exact missing-relay and unavailable-relay code paths, but this review did not have a deterministic local Telegram/Codex long-running session reproduction.

## Solution Assessment

Yes for the triage outcome: keeping this issue open is the right path because an open PR is already linked as the implementation candidate and current main does not prove the bug fixed. The code solution should stay focused on relay registry/rebinding/health behavior rather than treating generic hook readiness as sufficient.
