## Summary

Keep open: the patch targets the right OpenRouter music SSE parser, but silently skipping any malformed `data:` frame can let the stream complete with partial audio, and the PR only includes mocked/existing test output rather than real after-fix proof.

## What This Changes

The PR wraps `JSON.parse` in OpenRouter music SSE line parsing and skips malformed `data:` frames by returning `false`.

## Best Possible Solution

Make malformed OpenRouter music `data:` frames fail with a stable provider error unless the specific frame is proven ignorable, then cover that path with a focused regression test and redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence unit reproduction path: stream a valid audio chunk, then a malformed `data:` line, then `[DONE]`, and the PR branch can still return a non-empty result. I did not run tests because this review is read-only.

## Solution Assessment

No; catching the parser failure is the right location, but returning `false` is too broad for an audio stream because it can hide dropped data. A stable malformed-stream error, or a narrowly proven ignorable keepalive case, is safer.
