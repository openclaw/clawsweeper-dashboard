## Summary

Keep open: this is a valid OpenRouter music-streaming bug area, but the PR changes malformed `data:` frames from a fail-fast provider error into a silent skip, which can return partial audio if later chunks and `[DONE]` arrive. The contributor proof is also mock/test-only, so the PR needs revision and real behavior evidence before merge.

## What This Changes

Wraps `JSON.parse(data)` in the OpenRouter music-generation SSE parser with a catch path that returns `false` and skips malformed `data:` frames.

## Best Possible Solution

Handle malformed OpenRouter `data:` JSON as a controlled provider stream error, keep non-data/comment lines ignored, add a focused regression test, and include redacted real behavior proof for the updated path.

## Reproduction Assessment

Yes, at source level: current main calls `JSON.parse(data)` directly for OpenRouter music SSE `data:` frames, so malformed JSON throws from the stream parser. I did not run a live OpenRouter request in this read-only review.

## Solution Assessment

No. Catching the parse error is the right area, but returning `false` is not the best fix because it silently skips corrupt audio data; the maintainable fix is a controlled provider error with regression coverage.
