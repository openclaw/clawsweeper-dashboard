## Summary

Keep open for maintainer review: current main lacks reload metadata and the PR's additive wiring looks technically sound with sufficient live proof, but it changes the Gateway protocol/API surface.

## What This Changes

The PR adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, backed by the existing Gateway reload-rule matcher, with tests, protocol docs, and generated Swift model updates.

## Best Possible Solution

Land the additive lookup metadata if maintainers accept the protocol contract, keeping reload-kind resolution single-sourced from the existing planner and docs/generated clients aligned.

## Reproduction Assessment

Yes. Current source shows lookup returns no `reloadKind` while `buildGatewayReloadPlan()` classifies the same paths, and the linked issue includes both 2026.5.7 runtime evidence and a current-main lookup assertion.

## Solution Assessment

Yes, subject to maintainer API acceptance. Passing an optional resolver into `lookupConfigSchema()` reuses the existing matcher and avoids duplicating reload-rule tables or changing config patch behavior.
