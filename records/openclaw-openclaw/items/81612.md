## Summary

Keep open. Current main still lacks reload metadata in `config.schema.lookup`, the linked report has concrete runtime proof, and this PR is a focused additive implementation; the remaining decision is maintainer acceptance of the Gateway protocol/API surface and merge gates.

## What This Changes

The PR adds optional `reloadKind` metadata to config schema lookup results and child summaries, sourcing it from the existing Gateway config reload planner and updating tests, protocol docs, and generated Swift models.

## Best Possible Solution

Land the additive lookup metadata if maintainers accept the protocol surface, keeping reload behavior single-sourced from the existing planner and schema/protocol/client docs aligned.

## Reproduction Assessment

Yes. Source inspection shows current main cannot return reload metadata from `config.schema.lookup`, while the existing reload planner classifies the same paths; the linked issue also includes runtime proof on OpenClaw 2026.5.7.

## Solution Assessment

Yes, subject to maintainer acceptance of the API surface. Reusing the existing reload-rule matcher through an optional resolver is narrower than duplicating reload rules or adding a separate reload-rules endpoint for this path-scoped lookup use case.
