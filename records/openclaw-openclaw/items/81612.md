## What This Changes

The PR adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, wires it to the existing config reload planner, and updates tests, protocol docs, and the generated Swift model.

## Best Possible Solution

Land the additive lookup metadata if maintainers accept the API surface, keeping reload classification single-sourced from the existing reload planner and refreshing the branch against current main first.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current main source shows lookup results cannot include reload metadata, and the linked issue includes Docker proof on `openclaw@2026.5.7` showing the same field requires restart after patching.

## Solution Assessment

Is this the best way to solve the issue? Yes, subject to maintainer API acceptance. Passing an optional resolver into schema lookup reuses the existing reload planner and avoids a second rule table or a broader full-schema contract change.
