## What This Changes

The PR adds optional `reloadKind` metadata to `config.schema.lookup` results and child summaries, reusing the Gateway reload-rule matcher and updating protocol docs, tests, and generated Swift models.

## Best Possible Solution

Land the additive lookup metadata if maintainers accept the protocol surface, keeping reload behavior single-sourced from the existing reload planner and schema/protocol/client docs aligned.

## Reproduction Assessment

Yes. Current main can be checked by calling `config.schema.lookup` for `gateway.handshakeTimeoutMs` or `gateway`: the source shows lookup responses cannot include reload metadata even though the reload planner can classify those paths.

## Solution Assessment

Yes. Reusing the existing reload-rule matcher through an optional resolver is the narrowest maintainable path because it avoids duplicating rule tables and keeps the full schema contract unchanged.
