## Label Justifications

- `P2`: The report describes materially misleading Codex usage/cost accounting with limited blast radius to Codex Runtime usage surfaces.
- `impact:data-loss`: The issue is about Codex tool-loop usage data not being persisted into the transcript-derived usage cache, effectively dropping usage telemetry from local summaries.

## Summary

Keep open: current main still records Codex app-server last/current usage into assistant transcript entries and uses zero-usage mirrored tool-call records, while the usage-cost cache sums only transcript-visible usage. Upstream Codex exposes cumulative and last usage separately, so this looks like a real Codex usage-accounting gap rather than a duplicate or already-fixed report.

## What This Changes

Codex Runtime usage/cost accounting should stop undercounting tool-heavy sessions where mirrored toolUse records have zero usage and only high-level completion usage is nonzero.

## Best Possible Solution

Add a separate Codex run-usage or billing accumulator derived from app-server total-token deltas, while keeping context/session totalTokens based on last/current prompt usage and labeling any remaining transcript-only values clearly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a strong reproduction shape: Codex mirrored toolUse records are zero-usage, the projector records only last/current usage, and the cost cache sums transcript-visible usage; live replay still needs a sanitized transcript or Codex run to measure the multiplier.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the usage contract, because using cumulative Codex totals directly would solve cost undercounting but break context semantics; the safer direction is a separate per-turn aggregate/delta usage path plus clear UI/docs labels.
