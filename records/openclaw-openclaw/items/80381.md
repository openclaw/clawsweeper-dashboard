## Summary

Keep open: current main still leaves setup-time Docker CLI config writes exposed to host-style OpenClaw path environment, and an open linked PR already references this issue with closing syntax, so the issue should remain open until that PR lands or is closed.

## What This Changes

Windows/WSL Docker setup should avoid resolving host `/mnt/c/...` config paths inside the container during setup-time config writes.

## Best Possible Solution

Review and land the linked path-pinning PR, or replace it with equivalent container-local env pinning plus WSL-style regression coverage.

## Reproduction Assessment

No, not from a live current-main run in this review. The reporter gives concrete WSL steps, and current source inspection shows setup-time config writes can still inherit host `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` into container path resolution.

## Solution Assessment

Yes. Explicitly pinning container-local home/state/config/workspace env for setup-time Docker CLI runs and services is the narrow maintainable fix, and the issue should close only when the linked implementation merges.
