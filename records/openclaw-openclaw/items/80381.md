## Summary

Keep open: this is a fresh, plausible install-blocking WSL Docker regression, and current main still performs setup-time config writes at the exact reported `Docker gateway defaults` stage. The earlier macOS host-path fix addressed runtime `/Users` leakage, but it does not prove this setup-time `/mnt/c` failure is fixed.

## What This Changes

The issue reports a Docker setup regression on Windows/WSL where v2026.5.3 and later fail during setup-time gateway config sync with `EACCES: permission denied, mkdir '/mnt/c'`.

## Best Possible Solution

Make setup-time Docker config commands explicitly use container-local `HOME`, config, and state paths while leaving WSL `/mnt/c/...` values only as Compose bind-mount sources, then cover that with WSL-style regression proof.

## Reproduction Assessment

No, not from a live current-main run. Source inspection matches the reported failure stage and write path, but a real Windows/WSL Docker setup smoke or equivalent Compose environment proof is still needed.

## Solution Assessment

Yes, the narrow maintainable direction is a Docker setup boundary fix: pin prestart container-local config/state/home environment for setup writes and add regression coverage, rather than changing core config semantics or provider behavior.

## Repair Work Prompt

Fix the Windows/WSL Docker setup regression reported at https://github.com/openclaw/openclaw/issues/80381. Current main still runs `sync_gateway_config` through `run_prestart_cli` before gateway startup; make those setup-time config commands explicitly resolve container-local config/state/home paths so host paths such as `/mnt/c/Users/...` are used only as Compose bind-mount sources. Keep the fix scoped to Docker setup/Compose behavior; do not change general core config path semantics or add a new config option. Add a focused regression test in `src/docker-setup.e2e.test.ts` that exercises a WSL-style host `OPENCLAW_CONFIG_DIR=/mnt/c/...` and verifies the prestart command/container environment cannot write config under `/mnt/c`. Update Docker docs only if behavior or diagnostics change, and add a `CHANGELOG.md` fix entry because this is user-visible install behavior. Validate with the Docker setup test, shell syntax check, and a real Windows/WSL Docker setup smoke when available; if live WSL is unavailable, include explicit proof explaining the closest equivalent.
