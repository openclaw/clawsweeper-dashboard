## Summary

Keep open: this is a fresh external Docker/WSL install-blocking regression report, and current main still runs the setup-time config write at the reported failure point. The exact way `/mnt/c` becomes the in-container config path is not fully proven, so this should be fixed with targeted WSL/Docker proof rather than closed as already solved.

## What This Changes

The issue requests fixing Docker `setup.sh` on Windows/WSL when host config and workspace paths under `/mnt/c/...` make setup fail during gateway config sync with `EACCES: permission denied, mkdir '/mnt/c'`.

## Best Possible Solution

Reproduce the reporter's WSL/Docker setup or an equivalent prestart-CLI harness, then keep host `/mnt/c/...` values only as bind-mount sources while forcing setup-time config/state/home resolution to container-local paths and covering it with a WSL-style regression test.

## Reproduction Assessment

No, not with high confidence from a live run. Source inspection identifies the reported setup phase and the config write that would fail if `/mnt/c` leaks into `configPath`, but rendered Compose config means a real WSL/Docker smoke is still needed.

## Solution Assessment

Unclear until the leak is reproduced. The likely best fix is narrow Docker setup/prestart environment hardening, not a provider or onboarding policy change, but the patch should be driven by WSL proof.
