## Summary

Keep open: the PR addresses a real Telegram raw-update log leak on current main, but the proposed redaction table is still incomplete for Telegram update shapes that carry chat identifiers, and the maintainer-labeled security hardening needs explicit review before merge.

## What This Changes

The PR extracts Telegram raw-update formatting into a redacting formatter, wires the verbose raw-update logger through it, and adds a focused formatter regression test.

## Best Possible Solution

Keep the PR open and land a type-backed raw-update redactor that conservatively handles Telegram User, Chat, and private payload fields, with regression cases for message, callback, business/join, and reaction update shapes.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main serializes `ctx.update` directly in the verbose Telegram raw-update middleware, so private fields present in a Telegram update can be written to debug logs. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. Extracting a formatter is the right shape, but the current key-list approach is too narrow for known Telegram update types; broaden the redaction rules and tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Redact chat identifiers from all Telegram chat-shaped fields:** `extensions/telegram/src/raw-update-log.ts:22-37`
  - body: The new formatter only redacts `id` when the parent key is in a small allowlist, and it never redacts `user_chat_id`. Telegram's locked types include `business_connection.user_chat_id`, `chat_join_request.user_chat_id`, and reaction updates with `actor_chat: Chat`, so those raw updates can still write chat identifiers to `gateway/channels/telegram/raw-update` logs. Extend the key/parent coverage or detect Chat-shaped objects, and add regression cases for these update families.
  - confidence: 0.87
