## Summary

The requested capability is still necessary: current main continues to format reasoning with one fixed italic renderer used by Telegram and other delivery surfaces, while the only linked implementation PR closed unmerged and no newer viable canonical PR owns the feature. The issue should remain open for an explicit product decision because adding a public format setting requires portable channel semantics, precedence rules, and a carefully justified config surface.

## What This Changes

Add configurable presentation styles for visible reasoning blocks, selectable by channel or agent instead of always using italic Markdown.

## Best Possible Solution

Define a small typed, portable reasoning-presentation contract at the shared agent/channel boundary, retain the current italic rendering as the compatibility default, specify agent-versus-channel precedence, map only supported styles in each transport, and cover formatter, streaming, final delivery, config schema/help, and public docs end to end.

## Reproduction Assessment

Yes for the reported limitation: current-main source deterministically shows a one-argument fixed-italic formatter used by Telegram and sibling delivery paths. A live Telegram run is unnecessary to establish that no configurable presentation mode exists.

## Solution Assessment

No, the proposed raw per-channel or per-agent format string is not yet the best proven design. A typed portable presentation contract with explicit precedence and channel capability mapping is safer than leaking Telegram markup choices into shared runtime configuration.
