## Label Justifications

- `P2`: This is a real provider-routing bug that can send Z.ai model requests to the OpenAI SDK default host, but the blast radius is limited to the bundled Z.ai GLM-5 resolver path.
- `impact:auth-provider`: The issue affects provider routing and authentication behavior by resolving a Z.ai model without the Z.ai base URL, producing OpenAI-host authentication failures.

## Summary

This issue is still necessary: current main and the latest release still build synthesized Z.ai GLM-5 models with `baseUrl` coming only from explicit provider config or the GLM-4.7 template, and the open linked PR is the active fix candidate rather than a merged implementation.

## What This Changes

The issue requests that native Z.ai static catalog models such as `zai/glm-5-turbo` inherit the bundled provider `baseUrl` so OpenAI-compatible requests do not fall back to the OpenAI API host.

## Best Possible Solution

Review and land the provider-local manifest `baseUrl` fallback in the linked fix PR, preserving explicit provider and template endpoint precedence, then close this issue after the PR merges.

## Reproduction Assessment

Yes, this is source-reproducible: current main still lacks the manifest fallback, and the issue provides a packaged `openclaw@2026.6.8` resolver script showing `baseUrl` missing. I did not run a live API-key path in this read-only review.

## Solution Assessment

Yes. The maintainable fix belongs in the Z.ai provider dynamic resolver by falling back to the provider manifest base URL after explicit config and template base URLs, which is exactly the shape of the open fix candidate.
