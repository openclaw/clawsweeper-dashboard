## Label Justifications

- `P2`: This is a real provider-runtime routing bug with a confusing runtime failure, but it is scoped to Z.ai GLM-5 catalog model resolution.
- `impact:auth-provider`: The issue affects provider endpoint routing and can send a Z.ai-selected model through the OpenAI SDK default host with the wrong credential context.

## Summary

Keep open: current main and the latest release still have the Z.ai GLM-5 resolver path that can omit `baseUrl`, and the linked fixes at https://github.com/openclaw/openclaw/pull/94404 and https://github.com/openclaw/openclaw/pull/94461 are still open rather than merged.

## What This Changes

The issue asks OpenClaw to make Z.ai GLM-5 catalog model resolution carry the provider baseUrl so OpenAI-compatible requests go to Z.ai instead of OpenAI.

## Best Possible Solution

Land one narrow Z.ai plugin fix that preserves explicit provider/template endpoint precedence, falls back to the manifest/default Z.ai endpoint, adds focused resolver regression coverage, and then close this issue after merge.

## Reproduction Assessment

Yes. The issue includes a focused resolver snippet for v2026.6.8, and current source still shows the same missing fallback path; I did not execute it because this was a read-only review.

## Solution Assessment

Yes. The maintainable fix belongs in the Z.ai provider/model resolution path, not in a new config option or broad core fallback, because the manifest already owns the provider endpoint contract.
