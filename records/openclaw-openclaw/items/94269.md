- `impact:auth-provider`: The issue is about provider endpoint resolution and credentials for Z.ai models using the OpenAI-compatible transport.

## Summary

Keep this issue open: current `main` and the latest shipped release still have the Z.ai GLM-5 dynamic resolver path that can return a model without `baseUrl`, and the open linked PR is the active fix candidate rather than a reason to close before merge.

## What This Changes

The issue reports that bundled Z.ai static catalog models such as `zai/glm-5-turbo` resolve without the provider `baseUrl`, causing OpenAI-compatible runtime calls to default to OpenAI's API host.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94461 if its proof and checks are satisfactory, then let the merged closing PR close this issue.

## Reproduction Assessment

Yes. The reporter supplied a resolver-only npm reproduction, and source inspection shows current main can synthesize `zai/glm-5-turbo` without provider config or a template `baseUrl`; I did not run the script in this read-only checkout, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes for the current direction. The fix belongs in the Z.ai provider dynamic resolver because explicit provider config and template endpoints should keep precedence, with the manifest provider endpoint as the final provider-owned fallback.
