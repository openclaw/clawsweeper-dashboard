## Summary

Keep open: current main still has a credible source-level gap where configured Z.ai model resolution can depend on a static catalog model for transport data, while the reported shipped resolver output lacks the Z.ai `baseUrl` and can construct an OpenAI-compatible client without the Z.ai endpoint.

## What This Changes

The issue requests that bundled Z.ai static-catalog models inherit provider-level transport fields such as `baseUrl` during model resolution without requiring explicit provider config.

## Best Possible Solution

Fix the model resolver/static catalog merge path so manifest-backed Z.ai provider transport fields are inherited for configured agent model rows, while keeping Z.ai endpoint ownership in the bundled provider catalog.

## Reproduction Assessment

Yes from the reporter's npm release repro and current source inspection; I did not execute the repro in this checkout because dependencies are unavailable. The source path and tests show the expected provider `baseUrl` exists, while the reported configured sync resolver output can omit it.

## Solution Assessment

Yes: the narrow maintainable fix is in resolver/static-catalog transport merging, not a docs-only workaround or a new config option. Explicit provider config should remain an override, not the required path for bundled Z.ai models.
