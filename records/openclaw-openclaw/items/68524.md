## Summary

Keep open. Current main still sends raw iOS PhotosPicker bytes into the chat attachment path and rejects images over 5 MB before any resize, while https://github.com/openclaw/openclaw/pull/73710 is the open closing implementation candidate and is not merged.

## What This Changes

The issue requests client-side resizing and compression of iOS chat image attachments before transport to avoid failures and memory pressure from large camera-roll photos.

## Best Possible Solution

Land a focused iOS/OpenClawKit preprocessing fix that resizes and re-encodes camera-roll images off the MainActor before the final 5 MB gate, with broader media-limit configurability left to https://github.com/openclaw/openclaw/issues/67031.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an iOS PhotosPicker image Data payload over 5,000,000 bytes reaches addImageAttachment and is rejected before resize, while accepted payloads are later base64-encoded from the MainActor-backed view model.

## Solution Assessment

Yes for the direction, but it is not on main yet. The maintainable fix is an iOS chat attachment processing layer over JPEGTranscoder before the final size gate, and the open linked PR is the current candidate to review, repair, or replace.
