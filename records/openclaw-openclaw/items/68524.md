## Summary

Keep open. Current main still routes iOS chat composer image selections as raw bytes, rejects images over 5 MB before any resize, and base64-encodes accepted attachment bytes inside the MainActor view model. The issue is already paired with open PR #73710, which explicitly targets this composer path, so the maintainer cleanup path is PR review and device validation rather than closing this issue now.

## What This Changes

The issue asks for iOS chat composer photo attachments to be resized/compressed off MainActor before transport, with per-image and per-message size budgets for modern high-resolution camera images.

## Best Possible Solution

Review PR #73710 as the narrow iOS/OpenClawKit implementation, validate it on a real iPhone with several large Camera Roll photos, and keep broader gateway or provider-wide image-size policy work in separate issues such as #61041, #59913, and #67031.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: select or inject an image Data payload over 5,000,000 bytes into the iOS composer path and addImageAttachment rejects it before any resize; accepted attachments are later base64-encoded from stored raw Data in the MainActor view model.

## Solution Assessment

Yes for direction, pending PR review. A chat-specific processing layer over the existing OpenClawKit JPEGTranscoder is the narrow maintainable boundary for the iOS composer bug, while server/gateway resizing remains separate work.
