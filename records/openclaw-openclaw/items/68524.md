## Summary

Keep open. Current main still sends raw iOS PhotosPicker data into the chat attachment path, rejects images over 5 MB before any resize, and base64-encodes accepted attachment bytes from the MainActor-backed chat view model; open PR #73710 is the linked implementation candidate and should be reviewed before this issue closes.

## What This Changes

Request client-side iOS chat composer image resize/compression before transport so large modern phone photos do not fail or cause memory pressure.

## Best Possible Solution

Review and land or close PR #73710 as the narrow iOS/OpenClawKit fix, while leaving broader gateway and configurable media-limit work to #61041, #59913, and #67031.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path on current main: select or inject an iOS chat image Data payload over 5,000,000 bytes and the composer rejects it before any resize; accepted smaller payloads remain raw and are base64-encoded from the MainActor-backed view model.

## Solution Assessment

Yes for the direction, but it is not implemented on main yet. The narrow maintainable fix is an iOS chat attachment processing layer over the existing JPEGTranscoder before the final 5 MB safety gate, which is what PR #73710 is attempting.
