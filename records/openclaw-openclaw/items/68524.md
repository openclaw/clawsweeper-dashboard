## Summary

Keep open. Current main and the latest release still pass raw iOS PhotosPicker bytes into the chat attachment path, reject images over 5 MB before any resize, and base64-encode stored attachment bytes from the MainActor-backed view model; https://github.com/openclaw/openclaw/pull/73710 is the open closing implementation candidate and should be resolved before this issue closes.

## What This Changes

Resize and compress large iOS chat composer image attachments client-side before transport so modern camera-roll photos do not exceed memory and payload limits.

## Best Possible Solution

Land a focused iOS/OpenClawKit preprocessing fix that resizes and re-encodes camera-roll images off the MainActor before the final 5 MB gate, with https://github.com/openclaw/openclaw/pull/73710 as the current candidate.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: select or inject an iOS chat image Data payload over 5,000,000 bytes and addImageAttachment rejects it before resize; accepted payloads remain raw and are base64-encoded from the MainActor-backed view model.

## Solution Assessment

Yes for the implementation direction, but it is not on main yet. The narrow maintainable fix is an iOS chat attachment processing layer over JPEGTranscoder before the final size gate; the linked PR attempts that but still needs merge-conflict/proof cleanup.
