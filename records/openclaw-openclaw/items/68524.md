## Summary

Keep open. Current main still loads raw iOS PhotosPicker bytes into the chat view model, rejects image attachments above 5 MB before any client-side resize, and base64-encodes accepted attachment bytes from the MainActor-backed chat view model. The linked open PR #73710 explicitly closes this issue, so the correct cleanup is to review and validate that PR rather than closing the issue before merge.

## What This Changes

The issue requests client-side resize/compression for iOS chat composer image attachments, including off-MainActor processing and payload budgets for large camera-roll photos.

## Best Possible Solution

Review and validate PR #73710 as the narrow iOS/OpenClawKit implementation for this issue, while keeping broader gateway and configurable image-limit work in the separate related issues such as #61041, #59913, and #67031.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: select or inject an iOS chat image Data payload over 5,000,000 bytes and the composer path rejects it before resizing; smaller accepted payloads are stored raw and later base64-encoded from the MainActor-backed view model.

## Solution Assessment

Yes for the direction, pending PR review. The narrowest maintainable fix is to add an iOS chat attachment processing layer over the existing OpenClawKit JPEGTranscoder before the 5 MB safety gate, with physical-device validation before closing this issue.
