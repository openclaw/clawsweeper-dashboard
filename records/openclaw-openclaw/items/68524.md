## Summary

Keep open. Current main still sends raw iOS PhotosPicker bytes into the chat attachment path and rejects images over 5 MB before resize; https://github.com/openclaw/openclaw/pull/73710 is the open closing implementation candidate, so this issue should stay open until that PR is resolved.

## What This Changes

The issue asks for iOS chat composer image preprocessing so large camera-roll photos are resized, compressed, and encoded off the MainActor before transport.

## Best Possible Solution

Land a focused iOS/OpenClawKit preprocessing fix that resizes and re-encodes camera-roll images off the MainActor before the final 5 MB gate, with broader configurability left to the separate media-limits discussion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an iOS PhotosPicker image Data payload over 5,000,000 bytes reaches addImageAttachment and is rejected before any resize, while accepted payloads are base64-encoded from the MainActor-backed view model.

## Solution Assessment

Yes for the direction, but it is not on main yet. The maintainable fix is an iOS chat attachment processing layer over JPEGTranscoder before the final size gate, and the open linked PR is the current candidate to review or replace.
