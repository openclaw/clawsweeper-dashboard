## Summary

Keep open. Current main still sends raw iOS PhotosPicker bytes into the chat attachment path and rejects images over 5 MB before any resize, while the linked closing implementation PR remains open and unmerged.

## What This Changes

The issue asks for iOS chat image attachments to be resized/compressed off the MainActor before transport so 48MP photos do not exceed memory or provider payload limits.

## Best Possible Solution

Review and land or replace the linked iOS/OpenClawKit preprocessing PR with a focused fix that resizes and re-encodes camera-roll images before the final 5 MB transport gate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an iOS PhotosPicker image Data payload over 5,000,000 bytes reaches addImageAttachment and is rejected before any resize, while accepted bytes are later base64-encoded directly from the MainActor-backed view model.

## Solution Assessment

Yes for the direction, but it is not on main yet. Reusing the existing JPEGTranscoder through a chat-specific preprocessing layer is the narrow maintainable path; the open linked PR is the current implementation candidate to review or replace.
