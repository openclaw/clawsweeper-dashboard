## Summary

This remains the canonical open request for caller-directed tool-result release. Current main provides TTL-based in-memory pruning, deterministic persisted truncation for overflow recovery, compaction, and whole-session reset, but none gives users or agents targeted on-demand control; the narrower related overflow PR does not supersede that gap, and maintainers still need to choose ephemeral archive/recovery versus durable transcript rewrite semantics.

## What This Changes

Add a caller-directed way to remove an individual tool result from active agent context once it is no longer needed.

## Best Possible Solution

Define one core context-lifecycle contract that selects stable transcript entries, distinguishes ephemeral exclusion from durable deletion or recoverable archival, updates in-memory and persisted state atomically, and then exposes only the minimum maintainer-approved CLI or agent surface.

## Reproduction Assessment

Not applicable: this is a request for a new manual lifecycle capability, and current main's automatic TTL pruning behavior matches the documented contract.

## Solution Assessment

No, none of the proposed entry points is independently the best solution yet; maintainers should first define ephemeral, durable, and recoverable semantics behind one canonical core API, then choose the smallest public surface.

## Risks / Open Questions

- An in-memory-only clear can reappear after session reload, while a durable rewrite can permanently remove data used by history views, compaction, recovery, or auditing.
- A model-accessible clear operation needs strict selection and authorization semantics so it cannot discard the wrong tool result or erase state that later steps still require.
- Adding separate CLI, tool-call metadata, and agent-tool paths without one canonical lifecycle API would create competing context-state behavior.
