## Summary

Keep open. Current main and v2026.6.8 still provide automatic TTL/pressure pruning, transcript truncation, manual compaction, and session reset, but not caller-directed clear/archive/recover for a stable tool-result entry; the newer manual-free report was already closed into this issue, while the open overflow PR only addresses recovery ordering.

## What This Changes

Request a manual way to purge or archive no-longer-needed tool results from active context, potentially through a CLI command, agent-accessible function, or per-tool-call flag.

## Best Possible Solution

Define one canonical session/context-engine lifecycle operation for stable tool-result release or archival, then expose approved CLI, agent, and tool-call adapters over that operation with explicit auth, persistence, replay, and recovery semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main and v2026.6.8 expose automatic TTL/pressure pruning and manual compaction/reset, but no caller-directed release by stable tool-result identity in the context engine, session protocol, or docs.

## Solution Assessment

Is this the best way to solve the issue? No, not as separate CLI, agent function, and per-tool-call implementations. The maintainable path is one canonical context/session lifecycle operation with explicit persistence, recovery, authorization, and transcript-integrity semantics.
