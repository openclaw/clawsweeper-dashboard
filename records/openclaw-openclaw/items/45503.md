## Summary

Keep open. Current main and latest release v2026.6.6 still provide automatic cache-TTL pruning, pressure/overflow truncation, manual compaction, and whole-session reset, but not caller-directed clear/archive/recover for a stable tool-result entry; https://github.com/openclaw/openclaw/issues/82054 was already closed into this canonical design thread, while https://github.com/openclaw/openclaw/pull/81190 only covers overflow recovery ordering.

## What This Changes

Request a manual way to purge, archive, or recover no-longer-needed tool results from context, such as a CLI command, agent-callable operation, or clear-after-use flag.

## Best Possible Solution

Define one session/context-engine-owned lifecycle operation for stable tool-result release or archival, including persistence, recoverability, transcript integrity, and authorization semantics, then expose approved caller surfaces as thin adapters.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.6 expose automatic TTL pruning, pressure-driven truncation, compaction, and reset, but no caller-directed release by stable tool-result identity.

## Solution Assessment

No, not as independent CLI, agent function, and per-tool-call implementations. The maintainable path is one canonical session/context-engine lifecycle operation with explicit persistence, recovery, authorization, and transcript-integrity semantics.

## Risks / Open Questions

- The unresolved product choice is material: model-context-only masking, durable transcript rewrite, and recoverable archival have different replay, audit, compaction, restart, and storage behavior.
- An agent-callable clear operation could discard identifiers or state needed by later workflow steps unless eligibility, authorization, and recovery semantics are defined.
- Building separate CLI, agent-tool, and per-tool-call implementations would create competing context-lifecycle sources of truth unless they are adapters over one canonical operation.
