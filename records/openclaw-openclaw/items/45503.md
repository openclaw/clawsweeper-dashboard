## Summary

Keep open. Current main still has TTL/policy-driven pruning, broad manual compaction, and overflow truncation, but no user- or agent-addressable clear/archive/recover API for a specific completed tool result; a newer duplicate was closed back to this issue as the canonical design thread.

## What This Changes

The issue requests a user- or agent-controlled way to clear completed tool results from active context before automatic TTL pruning or compaction.

## Best Possible Solution

Design an additive prompt-view clear/archive/recover mechanism for completed tool results that preserves raw transcripts and provider pairing, works with context engines and compaction, and documents the supported user or agent API.

## Reproduction Assessment

Not applicable: this is a feature request, not broken behavior under an existing contract. The feature gap is high-confidence from source: pruning is cache-TTL driven, `/context` is inspect-only, `/compact` is session-scoped, and no targeted clear/archive/recover selector exists.

## Solution Assessment

Unclear. The proposed CLI command, agent function, and `clearAfterUse` flag are plausible entry points, but the maintainable solution needs maintainer API design around prompt-view archive/recover semantics rather than deleting transcript history.
