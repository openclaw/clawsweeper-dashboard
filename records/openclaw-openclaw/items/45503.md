## Summary

Keep open. Current main and the latest v2026.6.10 release still provide automatic TTL/pressure pruning, overflow truncation, manual compaction, reset, and delete, but not caller-directed clear/archive/recover for a stable tool-result entry; the duplicate manual-free report is already closed into this issue, while the open overflow PR addresses only automatic recovery ordering.

## What This Changes

The issue asks for caller-directed manual clearing or archiving of tool results from active context instead of relying only on automatic TTL, truncation, compaction, or session reset.

## Best Possible Solution

Define one canonical session/context-engine lifecycle operation for stable tool-result release or archival, then expose approved CLI, agent, and tool-call adapters over that operation with explicit auth, persistence, replay, and recovery semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main and v2026.6.10 expose automatic pruning, truncation, compaction, reset, and delete paths, but no caller-directed release by stable tool-result identity.

## Solution Assessment

Is this the best way to solve the issue? No, not as separate CLI, agent function, and per-tool-call implementations. The maintainable path is one canonical context/session lifecycle operation with explicit persistence, recovery, authorization, and transcript-integrity semantics.
