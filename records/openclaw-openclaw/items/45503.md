## Label Justifications

- `impact:session-state`: The requested feature changes which historical tool-result state is retained and supplied on subsequent turns.

## Summary

This remains necessary as the canonical request for caller-directed tool-result release. Current main supports automatic TTL pruning, overflow-driven truncation, compaction, and whole-session reset, but none provides authorized per-result lifecycle control; the related overflow PR only changes recovery ordering, and maintainers still need to choose masking, destructive clearing, or recoverable archival semantics.

## What This Changes

Add an authorized way for a user, agent, or tool caller to release a specific obsolete tool result from future active session context.

## Best Possible Solution

Define a single session-manager-owned operation that targets a stable tool-result identity, preserves transcript ancestry and tool-call/result validity, replaces released content with a minimal canonical marker in active context, and explicitly specifies whether original content remains recoverable; expose approved CLI or agent callers only as authorized thin adapters.

## Reproduction Assessment

Yes, at the source level. Current main exposes automatic TTL pruning, overflow-oriented truncation, compaction, and whole-session reset, and inspection found no targeted caller-directed tool-result release path.

## Solution Assessment

No, not as three independent implementations. The narrowest maintainable solution is one canonical session lifecycle API with explicit persistence, recovery, authorization, and transcript-integrity semantics, followed by whichever thin caller surfaces maintainers approve.
