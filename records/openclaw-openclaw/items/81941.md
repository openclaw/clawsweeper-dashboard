## Summary

Keep open: the report is recent, concrete, independently confirmed in https://github.com/openclaw/openclaw/issues/81944, and current main still has a split where normal OAuth profile resolution can accept a usable access token while Codex runtime/app-server refresh paths force OpenAI Codex token refresh. The merged work in https://github.com/openclaw/openclaw/pull/80738 is relevant history, but it intentionally preserved hard forced-refresh behavior and later comments report the same 401 `token_expired` pattern on a beta that includes that PR.

## What This Changes

The issue asks OpenClaw to stop failing Codex-backed agent turns after a fresh `openai-codex` OAuth login when status/probe paths report the OAuthRef-backed profile as healthy.

## Best Possible Solution

Keep this open for a maintainer-owned Codex auth review that decides whether runtime/app-server refresh must force token rotation or can reuse a still-valid OAuthRef access token, then add regression and live/redacted proof for both Codex harness and explicit PI paths.

## Reproduction Assessment

No, not at high confidence from this review alone. The current source path and two independent reporter logs strongly support the failure, but a real OpenAI Codex OAuth account is still needed to live-reproduce the exact current-main 401 `token_expired` response.

## Solution Assessment

Unclear. Reusing a valid access token may be the narrowest behavioral fix, but the recently merged hard-refresh contract and the pinned pi-ai refresh endpoint behavior need maintainer confirmation before changing runtime auth semantics.
