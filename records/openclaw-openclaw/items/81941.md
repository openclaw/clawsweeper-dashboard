## Summary

Keep open: the report is recent, concrete, independently confirmed, and maps to a current-main path where runtime turns answer Codex app-server refresh requests with a forced OpenAI Codex OAuth refresh even when status/probe paths can see a healthy OAuthRef-backed profile. The merged forced-refresh fix is relevant but does not prove this `token_expired` surface is fixed.

## What This Changes

The issue reports that OpenClaw 2026.5.12 fails Codex-harness agent turns with `401 token_expired` after fresh OpenAI Codex OAuth login while `models status` shows the OAuthRef-backed profile as valid.

## Best Possible Solution

Keep a focused Codex/OpenAI OAuth auth review open, decide the correct token-authority rule for app-server refresh, and add regression coverage for a fresh OAuthRef-backed profile that status/probe accepts but runtime refresh currently rejects.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The source path and two independent 2026.5.12 reports give medium-confidence reproduction: healthy non-forced status/probe resolution can diverge from the forced runtime refresh path that fails before model execution.

## Solution Assessment

Unclear. Reusing a valid access token may be the narrow behavioral fix, but https://github.com/openclaw/openclaw/pull/80738 intentionally enforced hard refresh and https://github.com/openclaw/openclaw/pull/73807 has unresolved token-authority review concerns, so the best fix needs maintainer/upstream-contract review first.
