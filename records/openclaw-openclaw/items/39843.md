## Summary

Keep open: current main and v2026.5.20 still have a source-level Google Chat path where a message-backed typing placeholder is created, exact NO_REPLY skips delivery, and no post-turn cleanup deletes the still-owned placeholder. The old target PR is closed unmerged; the remaining fork PR is not an upstream OpenClaw PR, so this remains an actionable repo bug.

## What This Changes

Google Chat should clear its message-backed typing placeholder when an agent reacts with an emoji and ends the turn with exact NO_REPLY.

## Best Possible Solution

Add a narrow Google Chat-owned cleanup that deletes any still-owned typingMessageName after a silent NO_REPLY turn, while preserving the normal first-reply edit path and keeping the message-tool placeholder follow-up separate.

## Reproduction Assessment

Yes, at source level: current main sends and stores typingMessageName before the channel turn, exact NO_REPLY returns before delivery, and the Google Chat monitor has no post-turn delete for the still-owned placeholder. I did not run a live Google Chat Workspace scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a Google Chat lifecycle cleanup after the current turn run or equivalent typing cleanup hook. It should not send dummy replies, add a config option, or expose a model-visible special edit API.
