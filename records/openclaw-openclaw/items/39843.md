## Summary

Keep open: current main and v2026.6.10 still create a Google Chat typing placeholder and only clear it from the delivery callback, while exact `NO_REPLY` final payloads are skipped before that callback runs. The open fork PR is a candidate fix, not an implemented fix, so maintainers still need to review, adopt, replace, or close it.

## What This Changes

The issue reports that Google Chat leaves its typing placeholder visible when an agent reacts with an emoji and ends the turn with `NO_REPLY`.

## Best Possible Solution

Land or adopt a narrow Google Chat plugin lifecycle fix that deletes or consumes `typingMessageName` when a turn settles without channel delivery, with focused regression coverage and real Google Chat proof.

## Reproduction Assessment

Yes, source-level: Google Chat creates and stores a message-mode typing placeholder, exact `NO_REPLY` final payloads are skipped before channel delivery, and the monitor does not currently pass a cleanup callback for the still-set placeholder. I did not run a live Google Chat scenario.

## Solution Assessment

Yes for the fix direction, but not implemented yet: the narrow maintainable solution belongs in the Google Chat plugin placeholder lifecycle, using the existing typing cleanup seam or equivalent post-turn cleanup rather than sending a visible fallback message.
