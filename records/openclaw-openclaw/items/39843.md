## Label Justifications

- `P2`: This is a normal-priority Google Chat delivery lifecycle bug with a visible stale chat artifact and channel-limited blast radius.
- `impact:message-loss`: The issue concerns channel message delivery state where a reply turn can complete without consuming the visible typing placeholder.

## Summary

Keep open: current main and v2026.6.10 still create a real Google Chat typing placeholder before dispatch and only clear it through the normal delivery callback, while exact `NO_REPLY` can normalize away before that callback runs. The linked fork PR is an open candidate fix, not implemented behavior.

## What This Changes

The issue reports that Google Chat leaves its typing placeholder visible when an agent reacts with emoji and then returns `NO_REPLY` instead of a text reply.

## Best Possible Solution

Land or adopt a narrow Google Chat-owned cleanup so silent no-delivery turns delete or consume the placeholder without sending a dummy visible message.

## Reproduction Assessment

Yes, source-level: Google Chat creates a message-mode typing placeholder before dispatch, exact `NO_REPLY` can normalize away before delivery, and the Google Chat call site does not wire cleanup for the still-set placeholder. I did not run live Google Chat in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the Google Chat plugin lifecycle: delete or consume the channel-owned placeholder when the run completes without a delivery callback. A dummy visible message would be worse UX, and generic core cleanup would not own the Google Chat message resource name.
