## Summary

Keep open: current main and v2026.6.10 still create a Google Chat typing placeholder before the inbound turn and only clear it inside the normal delivery callback; exact `NO_REPLY` finals can skip that callback. The open fork PR is a fix candidate, not an implemented fix, so this issue should stay open until maintainers adopt, replace, or close that candidate.

## What This Changes

The issue reports a Google Chat typing placeholder that remains visible when a turn ends with `NO_REPLY` after an emoji reaction.

## Best Possible Solution

Land or replace the narrow Google Chat placeholder cleanup so a silent no-delivery turn deletes or consumes `typingMessageName` without changing normal reply delivery.

## Reproduction Assessment

Yes, source-level: Google Chat creates a `typingMessageName` placeholder, exact `NO_REPLY` final payloads can be skipped before delivery, and current main has no post-run cleanup for the still-set placeholder. I did not run a live Google Chat Workspace repro.

## Solution Assessment

Yes for the fix direction, but it is not implemented on main: the maintainable solution is a Google Chat-owned placeholder lifecycle cleanup after silent no-delivery turns, with focused tests and real channel proof.
