## Summary

Keep open: current main and v2026.6.6 still create a real Google Chat typing placeholder before the turn and only clear it from the delivery callback, while exact NO_REPLY final payloads are dropped before delivery. The issue is still necessary, with an open cross-repo fix candidate that needs maintainer review and real Google Chat proof before this can close.

## What This Changes

The issue requests Google Chat to clear its typing placeholder when an agent returns NO_REPLY after sending an emoji reaction.

## Best Possible Solution

Keep this issue open until maintainers either adopt the open cross-repo fix or land an upstream Google Chat plugin cleanup that deletes or consumes the typing placeholder on silent NO_REPLY turns, with focused regression coverage and real Google Chat proof.

## Reproduction Assessment

Yes, from source inspection: Google Chat creates and stores a typing placeholder before the turn, exact NO_REPLY is dropped by the shared dispatcher before delivery, and Google Chat does not wire a cleanup callback for the still-set placeholder. I did not run a live Google Chat scenario.

## Solution Assessment

Yes for the direction, but not yet implemented on main: the narrow maintainable fix belongs in the Google Chat plugin's placeholder lifecycle, either through the existing typing cleanup seam or a local post-dispatch cleanup helper. The open cross-repo PR is a candidate, not proof that the issue can close.
