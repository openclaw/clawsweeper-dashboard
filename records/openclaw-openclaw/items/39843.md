## Summary

Keep open: current main and v2026.6.10 still create a Google Chat typing placeholder before the turn, while exact `NO_REPLY` finals are skipped before Google Chat delivery can edit or delete that placeholder. The open fork PR is a candidate fix, not an implemented fix, and needs maintainer review or replacement before this issue can close.

## What This Changes

The issue requests clearing Google Chat's message-based typing placeholder when an agent reacts with an emoji and then returns `NO_REPLY` without a text reply.

## Best Possible Solution

Review, adopt, or replace the open narrow fork PR with a Google Chat plugin cleanup that deletes or consumes `typingMessageName` on silent no-delivery turns, plus focused regression coverage and real Google Chat proof.

## Reproduction Assessment

Yes, source-level: Google Chat creates and stores a typing placeholder before the turn, exact `NO_REPLY` finals can skip delivery, and current main has no wired cleanup for the still-set placeholder. I did not run a live Google Chat scenario.

## Solution Assessment

Yes, the right fix boundary is the Google Chat plugin placeholder lifecycle, not a new config option or a model-visible workaround. The current open candidate is plausible but still needs maintainer review and real behavior proof before it becomes the merge path.
