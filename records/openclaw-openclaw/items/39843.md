## Summary

Keep open: current main and v2026.6.6 still create a real Google Chat typing placeholder before the turn, but exact `NO_REPLY` final payloads are skipped before Google Chat delivery can edit or delete that placeholder. The issue is still necessary, and the open cross-repo fix candidate needs maintainer review/adoption plus real Google Chat proof before this can close.

## What This Changes

The issue requests that Google Chat clear its message-based typing placeholder when an agent reacts with an emoji and then returns `NO_REPLY`.

## Best Possible Solution

Keep this issue open until an upstream Google Chat plugin cleanup deletes or consumes the typing placeholder on silent `NO_REPLY` turns, with focused regression coverage and real Google Chat proof.

## Reproduction Assessment

Yes, source-level: Google Chat creates and stores a typing placeholder before the turn, exact `NO_REPLY` is skipped before delivery, and the Google Chat monitor does not wire a cleanup callback for the still-set placeholder. I did not run a live Google Chat scenario.

## Solution Assessment

Yes for the fix direction, but it is not implemented on main: the narrow maintainable solution belongs in the Google Chat plugin placeholder lifecycle, either through the existing typing cleanup seam or a local post-dispatch cleanup helper.
