## Summary

Keep open: current main and the latest release still create a Google Chat typing placeholder before dispatch, while exact `NO_REPLY` final payloads are dropped before the Google Chat delivery callback can edit or delete that placeholder. The exact upstream PR attempts are closed unmerged, and the only open closing PR is in a fork, so the issue still needs maintainer follow-up rather than cleanup closure.

## What This Changes

The issue requests that Google Chat clear its visible typing placeholder when a turn ends with `NO_REPLY` after an emoji reaction.

## Best Possible Solution

Land a narrow Google Chat-owned cleanup for silent/no-delivery turns with regression coverage, either by adopting a suitable existing fix attempt or opening a clean upstream PR.

## Reproduction Assessment

Yes, source-reproducible: current main creates a Google Chat typing message before dispatch, exact `NO_REPLY` is dropped before delivery, and Google Chat has not wired a cleanup callback for that no-delivery path. I did not run a live Google Chat turn.

## Solution Assessment

Yes, a narrow Google Chat plugin fix is the best direction: reuse the existing typing cleanup lifecycle or add a local cleanup helper around `typingMessageName`, rather than changing global `NO_REPLY` suppression behavior.
