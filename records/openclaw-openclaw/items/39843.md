## Summary

Keep open: current main and v2026.5.22 still send and store a Google Chat message-backed typing placeholder before the turn, exact `NO_REPLY` can skip delivery entirely, and no post-turn cleanup deletes the still-owned placeholder. The target-repo fix PRs are closed; the open fork PR is useful prior art but is not a merged OpenClaw fix.

## What This Changes

The issue asks Google Chat to clear its message-backed typing indicator when an agent reacts with an emoji and then returns `NO_REPLY`.

## Best Possible Solution

Add a Google Chat-owned post-turn cleanup that deletes any still-owned `typingMessageName` after silent turns, while preserving normal delivery edits and avoiding dummy replies or new config.

## Reproduction Assessment

Yes, at source level: current main sends and stores `typingMessageName` before the turn, exact `NO_REPLY` returns before delivery, and the monitor exits without deleting a still-owned placeholder. I did not run a live Google Chat Workspace scenario.

## Solution Assessment

Yes. The narrow maintainable fix is Google Chat placeholder lifecycle cleanup after the turn run; sending dummy messages or adding a config option would be a noisier workaround.
