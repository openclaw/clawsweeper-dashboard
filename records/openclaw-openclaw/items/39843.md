## Summary

Keep open: current main and the latest release still leave the Google Chat typing placeholder owned by the delivery callback, while exact NO_REPLY final payloads are dropped before that callback can edit or delete the placeholder. The issue is still necessary because the upstream fix attempts are closed unmerged and the only open closing PR is a cross-repo fork that still needs maintainer review and real behavior proof.

## What This Changes

The issue reports a Google Chat typing placeholder that stays visible when an agent adds an emoji reaction and then returns NO_REPLY instead of a text reply.

## Best Possible Solution

Keep this issue open until maintainers either adopt the open fork PR or land a clean upstream Google Chat plugin fix that deletes or consumes the typing placeholder on silent NO_REPLY turns, with focused regression coverage and live Google Chat proof.

## Reproduction Assessment

Yes. Source inspection shows Google Chat creates and stores a typing placeholder before dispatch, exact NO_REPLY is dropped before delivery, and Google Chat does not wire a no-delivery cleanup callback for the still-set placeholder.

## Solution Assessment

Yes for the direction: the narrow maintainable solution is Google Chat-owned cleanup through the existing typing/dispatch lifecycle or a local `typingMessageName` cleanup helper. The current open fork PR is only a candidate until maintainers review it and require real Google Chat proof.
