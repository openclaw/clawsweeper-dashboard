## Summary

Keep open: current main and latest release still create a Google Chat message-backed typing placeholder before the turn, exact `NO_REPLY` can skip delivery entirely, and the monitor has no post-turn cleanup for the still-owned placeholder. The target-repo fix PRs are closed, and the open fork PR is useful prior art but not a merged OpenClaw fix.

## What This Changes

The issue asks OpenClaw to clear the Google Chat typing placeholder when an agent reacts with an emoji and then returns `NO_REPLY`.

## Best Possible Solution

Add Google Chat-owned post-turn cleanup that deletes any still-owned `typingMessageName` after silent turns, while preserving normal delivery edits and avoiding dummy replies or new config.

## Reproduction Assessment

Yes, at source level: current main sends and stores `typingMessageName`, exact `NO_REPLY` returns before delivery, and the monitor exits without deleting a still-owned placeholder. I did not run a live Google Chat Workspace scenario.

## Solution Assessment

Yes. The narrow maintainable fix is Google Chat placeholder lifecycle cleanup after the turn run; sending dummy messages or adding a config option would be a noisier workaround.
