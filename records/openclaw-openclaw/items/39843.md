## Summary

Keep open: current main and the latest release still create a Google Chat typing placeholder before the turn, while exact `NO_REPLY` final payloads are skipped before the Google Chat delivery callback can edit or delete that placeholder. The open fork PR is a fix candidate, not an implemented fix, and needs maintainer review/adoption plus real Google Chat proof before this issue can close.

## What This Changes

The issue asks Google Chat to clear its typing placeholder when an agent only reacts and returns `NO_REPLY`.

## Best Possible Solution

Land or replace a narrow Google Chat plugin lifecycle fix that deletes or consumes `typingMessageName` on silent no-delivery turns, with focused regression coverage and real Google Chat proof.

## Reproduction Assessment

Yes, source-level: Google Chat creates and stores a typing placeholder before the turn, exact `NO_REPLY` is skipped before channel delivery, and the current monitor does not clean up the still-set placeholder after the run. I did not run a live Google Chat scenario.

## Solution Assessment

Yes for the fix direction, but it is not implemented on main: the narrow maintainable solution belongs in the Google Chat plugin placeholder lifecycle, either through the existing typing cleanup seam or a local post-dispatch cleanup helper.
