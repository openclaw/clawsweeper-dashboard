## Best Possible Solution

Keep the aggregate cap, but make the live guard preserve or return the active post-rewrite entry id and cover that behavior with a focused regression test before maintainers accept the session-history truncation tradeoff.

## Reproduction Assessment

Yes for the PR defect by source inspection: the new aggregate path can call `rewriteTranscriptEntriesInSessionManager`, which branches before the first rewritten entry and re-appends the suffix with new ids, while the changed toolResult append path returns the old `appendResult.entryId`. ## Solution Assessment

No as submitted; the aggregate cap is the right direction, but the live append path needs to preserve or remap the returned entry id before merge. Maintainers also need to accept the deliberate persisted-history truncation behavior as the upgrade tradeoff.
