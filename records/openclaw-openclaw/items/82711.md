## Summary

Keep open: this is a recent external bug with detailed Telegram/runtime evidence, and current source still has a plausible duplicate path where repeated exec-approval completion handling can mint distinct follow-up idempotency keys and bypass Gateway agent dedupe. I found related exec-followup fixes, but no merged fix or canonical open item that covers this specific duplicate async-completion symptom.

## What This Changes

The issue reports async exec completions spawning multiple parallel agent follow-up sessions and delivering duplicate Telegram messages for one approved command.

## Best Possible Solution

Make async exec completion follow-up dispatch idempotent per approval/run while preserving the secure one-use runtime handoff, then prove one Telegram response per completion with focused regression coverage and live or Testbox evidence.

## Reproduction Assessment

Yes at source level: the reporter gives a concrete Telegram DM reproduction, and current source shows repeated follow-up dispatches can bypass Gateway dedupe when each handoff gets a new nonce idempotency key. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Unclear until patched, but changing only the prompt wording is not the best fix. The maintainable fix should preserve stable duplicate suppression for the follow-up while keeping the internal elevated handoff session-scoped and one-use.
