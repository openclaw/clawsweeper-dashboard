## Summary

Keep open. Current main still routes async exec completions through nonce-bearing follow-up `agent` requests with exact idempotency-key dedupe, and the linked open fix PR has not merged, so this issue should stay open until that PR is corrected, proven, and landed or replaced.

## What This Changes

The issue reports duplicate Telegram replies from multiple parallel agent sessions after one async exec completion.

## Best Possible Solution

Keep this issue open and review, fix, and prove https://github.com/openclaw/openclaw/pull/82717 or a replacement that collapses duplicate follow-up deliveries by stable approval id without breaking `expectFinal` or direct fallback behavior.

## Reproduction Assessment

Yes at source level, but not as a live Telegram reproduction in this review. Current main creates nonce-bearing exec follow-up idempotency keys and the gateway dedupes only exact `agent:${idempotencyKey}` entries, which matches the reported duplicate-continuation mechanism.

## Solution Assessment

Yes in direction, but not complete through the current linked PR. A backend idempotency guard keyed by stable approval id is the right narrow fix, but it must return or wait in a way that does not make duplicate `expectFinal` callers time out and fall back to another visible message.
