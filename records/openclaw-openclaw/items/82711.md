## Summary

Keep open. The report describes a fresh user-facing regression, and current source still routes approved async exec completions through a follow-up `agent` run where duplicate completion handling would need to be collapsed before new agent sessions are accepted.

## What This Changes

The issue requests that one async exec approval completion produce one Telegram follow-up reply instead of many duplicate replies from parallel sessions.

## Best Possible Solution

Add a per-approval completion guard so repeated async exec completion signals collapse before spawning follow-up agent runs, while preserving backend-only elevated handoff behavior.

## Reproduction Assessment

No, not with high confidence from this review alone. The issue gives detailed live steps and source inspection identifies the async exec follow-up path, but I did not run the Telegram DM flow against current main.

## Solution Assessment

Yes, a narrow backend idempotency fix is the right direction. Prompt wording changes alone would not prevent multiple accepted follow-up sessions if duplicate completion dispatch is the root cause.
