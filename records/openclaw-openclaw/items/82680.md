## Summary

Keep open. The report describes a real user-facing model override path, and current main still resolves explicit run overrides through a lower-level parser before checking the configured-model allowlist instead of using the alias-aware allowed-model resolver used elsewhere.

## What This Changes

The issue asks OpenClaw to accept an explicit canonical `--model provider/model` override even when the configured model has a nearby alias.

## Best Possible Solution

Resolve explicit run model overrides through the existing alias-aware allowed-model resolver, preserving canonical `provider/model` refs and the configured allowlist boundary.

## Reproduction Assessment

No, not as a live current-main repro. Source review and the reporter's concrete beta output point to the explicit override resolver and allowlist check, but the exact alias-substitution path still needs a focused test.

## Solution Assessment

Yes, the maintainable fix direction is narrow: use or mirror `resolveAllowedModelRef` for explicit overrides instead of changing global provider/model normalization semantics.
