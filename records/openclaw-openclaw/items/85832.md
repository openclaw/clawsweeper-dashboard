## Summary

Keep open: this is a focused browser MCP leak fix, but the current PR still has mock-only proof and a concrete Windows cleanup-ordering bug that can leave descendants behind.

## What This Changes

The PR adds default `--no-usage-statistics` for Chrome DevTools MCP and routes browser MCP session teardown through an owned-process-tree cleanup helper.

## Best Possible Solution

Land a browser-plugin cleanup that terminates owned Chrome MCP process trees cross-platform, preserves explicit usage-stat choices, and includes both regression coverage and redacted live process-count proof.

## Reproduction Assessment

Yes, at source level: current main still closes Chrome MCP sessions through `client.close()`, while the MCP SDK only terminates its direct stdio child and Chrome DevTools MCP can spawn a detached watchdog. I did not run a live multi-hour process-count reproduction in this read-only review.

## Solution Assessment

No, not yet: browser-plugin-owned cleanup is the right direction, but this patch needs the Windows ordering fix and live runtime proof before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Move Windows tree kill before closing stdio child:** `extensions/browser/src/browser/chrome-mcp.ts:601-602`
  - body: The Windows path ignores the collected descendants and runs `taskkill /T` only after `params.client.close()`. The MCP SDK close may terminate or wait out the direct child first, so by the time `taskkill` runs the root PID can be gone and child/helper processes are no longer in that tree. Run the Windows tree cleanup while the root child is still alive, or collect/kill descendants through a Windows process listing, and add a regression for that ordering.
  - confidence: 0.83
