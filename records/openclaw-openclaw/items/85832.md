## Summary

Keep open: this PR targets the right browser-plugin path for the linked P1 Chrome MCP process leak, but it is not merge-ready because the proof is mock-only and the Windows cleanup path can miss descendants after the MCP SDK has already closed the root child.

## What This Changes

The branch adds default `--no-usage-statistics`, routes Chrome MCP session closes through an owned process-tree cleanup helper, and adds a mocked regression for Linux-style child-first termination.

## Best Possible Solution

Land a browser-plugin cleanup that terminates owned Chrome MCP process trees cross-platform, preserves explicit usage-stat choices, and includes both regression coverage and redacted live process-count proof.

## Reproduction Assessment

Yes, at source level: current main still routes Chrome MCP teardown through `client.close()`, while the MCP SDK only owns the direct stdio child and Chrome DevTools MCP can create a detached watchdog. I did not run a live multi-hour process-count reproduction in this read-only review.

## Solution Assessment

No, not yet: the browser-plugin-owned cleanup direction is the right fix, but the proposed Windows ordering and lack of live runtime proof make this version incomplete for merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Move Windows tree kill before closing stdio child:** `extensions/browser/src/browser/chrome-mcp.ts:602`
  - body: The Windows path ignores the collected descendants and runs `taskkill /T` only after `params.client.close()`. The MCP SDK close waits for and may terminate the direct child first, so by the time `taskkill` runs the root PID can be gone and child/helper processes are no longer in that tree. Run the Windows tree cleanup while the root child is still alive, or collect/kill descendants through a Windows process listing, and add a regression for that ordering.
  - confidence: 0.83
