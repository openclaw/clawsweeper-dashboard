## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep broker prefixes from shadowing native channels:** `extensions/channel-broker/src/channel.ts:26-28`
  - body: These prefixes duplicate existing Slack, Discord, and Telegram channel prefixes. Current prefix ownership returns the first active plugin declaring a prefix (`src/infra/outbound/channel-target-prefix.ts:60`), so once `channel-broker` is active, explicit sends like `channel=telegram` with `to=telegram:...` can be treated as broker-owned and rejected or routed incorrectly. Keep only broker-owned prefixes here or make prefix resolution ownership-aware before claiming native channel prefixes.
  - confidence: 0.89
- **[P1] Resolve broker tokens before sending them:** `extensions/channel-broker/src/runtime.ts:65-66`
  - body: `outboundToken` is documented with an `op://`-style secret value, and the SDK runtime contract requires provider/channel execution to read resolved runtime config for credentials. This sends the raw configured value as `Authorization: Bearer ...`, so a SecretRef-style value would be sent literally instead of the actual token. Add the normal channel secret/runtime assignment path or document and enforce plaintext-only tokens before shipping.
  - confidence: 0.86
- **[P2] Thread cancellation into the broker fetch:** `extensions/channel-broker/src/runtime.ts:60`
  - body: `sendChannelBrokerText` accepts `signal`, but the runtime transport drops it before calling `fetch`. Cancelled or timed-out turns can keep waiting on the broker HTTP request until the network stack times out. Pass the signal through `sendBrokerOutboundRequest` and include it in the fetch init, with a focused abort test.
  - confidence: 0.82
