## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not claim native channel prefixes:** `extensions/channel-broker/src/channel.ts:172`
  - body: Registering the broker with `telegram`, `discord`, and `slack` target prefixes overlaps the existing native channel prefixes. Current core resolves a prefixed target by scanning registered plugin `targetPrefixes`, so `telegram:...`/`discord:...`/`slack:...` can stop belonging reliably to the native channels; keep broker routing behind an explicit broker prefix or selected `channel-broker` route until each migration is opt-in and proven.
  - confidence: 0.9
- **[P1] Resolve broker tokens before sending them:** `extensions/channel-broker/src/runtime.ts:66`
  - body: This sends `account.outboundToken` directly in the Authorization header, but the PR only stores raw config strings for `outboundToken`/`signingSecret` and does not use the existing SecretInput/SecretRef resolution surface. A configured secret marker would be sent literally to the provider and auth would fail; wire broker credentials through the normal secret contract and add focused coverage.
  - confidence: 0.86
- **[P2] Pass the send signal into broker HTTP:** `extensions/channel-broker/src/runtime.ts:60-70`
  - body: The channel send context passes `ctx.signal` down to `sendChannelBrokerText`, but the runtime request type and fetch call drop it. Cancellation, shutdown, or retry aborts can leave the broker HTTP send running and committing late, so pass the signal through `sendBrokerOutboundRequest` into fetch and cover the abort path.
  - confidence: 0.88
