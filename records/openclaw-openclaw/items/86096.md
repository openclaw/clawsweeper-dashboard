## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep broker prefixes from shadowing native channels:** `extensions/channel-broker/src/channel.ts:23-49`
  - body: This list advertises existing channel prefixes like `telegram`, `discord`, `slack`, `whatsapp`, and `signal` as broker-owned. Current target-prefix validation scans the active registry and treats the first matching prefix as the owner, while the native plugins already declare those prefixes. If the broker is configured or loaded before a native channel, selected native sends such as `channel=telegram` with `to=telegram:...` can fail as belonging to `channel-broker` or route through the wrong channel. Keep broker prefixes unique or add an explicit conflict-resolution/opt-in path before merge.
  - confidence: 0.87
- **[P2] Validate broker receipts before committing delivery state:** `extensions/channel-broker/src/runtime.ts:45-49`
  - body: `parseBrokerReceipt` accepts any object as a `BrokerReceiptV1`, then the send path indexes `receipt.messageIds` and records provider/platform/status metadata. A malformed or hostile provider response can crash the send or commit spoofed receipt data. Validate the protocol version, request id, provider id/platform, status, and `messageIds` array before returning the receipt.
  - confidence: 0.86
- **[P2] Do not publish inbound webhook support before the route exists:** `docs/channels/channel-broker.md:17-19`
  - body: The docs say V1 includes signed inbound HTTP webhooks, and the config accepts `signingSecret`, but this diff only implements outbound `/v1/outbound` delivery and stores the signing secret without any inbound route or verifier. Provider authors would build against a non-existent security surface. Either implement and test the inbound webhook/signature path or mark inbound webhooks as future work.
  - confidence: 0.82
