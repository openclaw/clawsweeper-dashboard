## What This Changes

This PR clears messaging-tool sent text, target, and media tracking arrays on every `compaction_end` event and updates the compaction handler test fixture.

## Best Possible Solution

Keep the small lifecycle fix if live proof confirms that normal post-compaction runs start with fresh messaging dedupe state while retry compaction still preserves deterministic side-effect accounting.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed `ctx.state.messagingToolSentTextsNormalized`, process a normal `compaction_end` with `willRetry: false`, then process a later `message_end` whose final text includes that stale normalized entry. I did not run a live harness because this review is read-only.

## Solution Assessment

Yes, clearing the per-run messaging sent arrays in `compaction_end` is the narrowest maintainable fix; it leaves same-turn dedupe behavior intact instead of weakening the duplicate matcher. A direct regression assertion would improve coverage but is not a blocking code finding by itself.
