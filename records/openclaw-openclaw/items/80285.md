## Summary

Keep open. The PR targets a valid source-visible stale messaging dedupe bug, but the current patch is not mergeable because it clears message-tool delivery evidence before compaction-retry side-effect accounting and still lacks real behavior proof.

## What This Changes

The PR clears messaging-tool sent text, target, and media tracking arrays during `compaction_end` and updates the compaction handler test fixture.

## Best Possible Solution

Preserve retry deterministic side-effect accounting while clearing stale messaging dedupe state at the normal compaction boundary, then add focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `ctx.state.messagingToolSentTextsNormalized`, process a normal `compaction_end` with `willRetry: false`, then process a later `message_end` whose final text includes that stale normalized entry.

## Solution Assessment

No. Clearing the per-run state is the right direction, but doing it before `resetForCompactionRetry()` breaks retry side-effect accounting; the clear should avoid the retry path or snapshot evidence first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve retry side-effect evidence before clearing:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:53-56`
  - body: The new clear runs before the `willRetry` branch. On retry, `resetForCompactionRetry()` currently reads these arrays first to set `state.hadDeterministicSideEffect`; clearing them here makes a retry after a successful message send look as if no messaging side effect happened, which can misclassify lifecycle/replay state. Move the clear out of the retry path or snapshot the evidence before clearing.
  - confidence: 0.94
