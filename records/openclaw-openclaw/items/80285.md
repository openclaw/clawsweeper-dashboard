## Summary

Keep open: the reported cross-run dedupe leak is plausible, but this patch is not correct as written because it clears messaging-tool delivery evidence before the compaction retry path snapshots it, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR clears messaging-tool sent text, target, and media tracking arrays during every `compaction_end` event and updates the compaction handler test fixture.

## Best Possible Solution

Keep the narrow lifecycle fix, but preserve retry-path delivery evidence before clearing per-run dedupe state and add regression coverage for both normal compaction cleanup and retry side-effect accounting.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `ctx.state.messagingToolSentTextsNormalized`, process a normal `compaction_end` with `willRetry: false`, then process a later `message_end` whose final text includes that stale normalized entry.

## Solution Assessment

No, not as written. Clearing in `compaction_end` is the right boundary, but the patch must either clear only after `resetForCompactionRetry` snapshots committed delivery evidence or move the non-retry cleanup into a helper that does not disturb retry accounting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve retry delivery evidence before clearing state:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:53-56`
  - body: On `willRetry: true`, these arrays are the evidence `resetForCompactionRetry()` uses to set `state.hadDeterministicSideEffect` before it clears retry-local state. Clearing them here first means a retry after `message`/`sessions_send` loses the committed-send marker, so later lifecycle/incomplete-turn handling can treat the run as having no deterministic side effect despite an external send. Snapshot the evidence before clearing, or defer this cleanup until after the retry reset has recorded it.
  - confidence: 0.92
