## Summary

Keep this PR open, but it should not merge as-is: the core bug report is source-reproducible, yet the proposed clearing runs before compaction-retry bookkeeping can preserve messaging-tool side-effect evidence, and the external PR still lacks real behavior proof.

## What This Changes

The PR clears messaging-tool sent text, target, and media tracking during `compaction_end` and updates the compaction handler test fixture, while the branch diff also includes unrelated release/workflow/version changes.

## Best Possible Solution

Keep the lifecycle fix, but revise it so normal compaction clears stale messaging dedupe state without clearing evidence before the retry reset records deterministic side effects.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed `messagingToolSentTextsNormalized`, process a normal `compaction_end` with `willRetry: false`, then process a later `message_end` whose final text contains the stale normalized sent text.

## Solution Assessment

No, not as currently written. Clearing the per-run arrays is the right direction, but the clearing must not run before `resetForCompactionRetry` captures messaging-tool delivery evidence on retry compactions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve retry delivery evidence before clearing state:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:84-93`
  - body: The new unconditional clears run before the `willRetry` branch. On overflow compaction retries, `resetForCompactionRetry()` currently inspects `messagingToolSentTexts`, `messagingToolSentMediaUrls`, and `messagingToolSentTargets` to set `hadDeterministicSideEffect`; clearing them first makes that evidence disappear, so a retry after `message(action=send)` or `sessions_send` can be treated as if no deterministic side effect happened. Move the new clear to the non-retry path, or otherwise preserve the evidence before clearing.
  - confidence: 0.92
