## Summary

Keep open. The linked bug is valid and the PR is on the right surface, but the current diff drops the selected profile on the default usage-auth path used by `/status`, so it does not yet fix the reported mismatch and still lacks real behavior proof.

## What This Changes

This PR adds preferred OAuth profile plumbing from status usage loading into provider usage auth resolution, regression tests for preferred and cross-provider profiles, and a changelog entry.

## Best Possible Solution

Thread the validated selected profile through both `resolveProviderAuths` branches while preserving fallback order, then land after redacted real `/status` or `session_status` proof from a multi-profile OAuth setup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the status auth label uses `sessionEntry.authProfileOverride`, while current usage auth still resolves OAuth credentials from provider order only.

## Solution Assessment

No. The intended direction is right, but this diff only applies the preferred profile in the credential-gated branch; the default branch used by status must also pass it into plugin and fallback auth resolution.
