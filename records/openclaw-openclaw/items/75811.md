## Summary

Keep open. Current main still exposes the reported exec policy fields in the model-facing tool schema and still derives effective host execution policy from tool-call arguments, so the model self-denial path remains present.

## What This Changes

The issue asks OpenClaw to remove or ignore model-controllable exec policy fields so operator/session policy remains authoritative at dispatch.

## Best Possible Solution

Implement an exec boundary that hides operator-owned policy fields from provider tool schemas, ignores or logs legacy model-supplied policy fields at dispatch, preserves authorized `/exec` and config/session policy as the source of truth, and updates exec docs plus focused schema/dispatch tests.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction exists by invoking the exec tool with permissive defaults and tool-call args containing `security: "deny"` or `security: "allowlist"`; current code consumes those args before host approval handling. The Hermes/vLLM stack from the report was not run in this read-only sweep.

## Solution Assessment

Yes: separating model-controllable command arguments from operator/session policy is the narrowest maintainable fix. The safer iteration is schema redaction plus dispatch normalization so older model outputs cannot still affect policy.
