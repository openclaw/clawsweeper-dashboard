## Summary

Keep open. Current main still exposes exec policy controls in the provider-visible schema and still folds model-supplied `security` and `ask` values into the effective host policy before denial handling.

## What This Changes

The issue requests splitting exec's model-controllable command arguments from operator-only policy fields such as `security`, `elevated`, `host`, `ask`, and `node`.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session `/exec` policy paths, and cover the boundary with schema and exec-policy tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is available by invoking the exec tool with otherwise permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current code derives restrictive policy from those args before host execution.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction for this bug.
