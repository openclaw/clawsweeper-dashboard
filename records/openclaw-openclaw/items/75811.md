## Summary

Keep open. Current main still exposes exec policy controls to the model-facing tool schema and consumes those values in effective exec policy; #76370 is the active unmerged fix path, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue asks OpenClaw to hide or ignore model-supplied exec policy controls so operator/session config remains authoritative.

## Best Possible Solution

Review and land #76370 or an equivalent patch that separates model-visible exec arguments from operator policy, preserves trusted config/session defaults, and adds schema plus dispatch regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `execSchema` exposes policy fields, `createExecTool` accepts them from tool-call args, and `params.security`/`params.ask` drive the effective policy before the denial paths.

## Solution Assessment

Yes. The narrow maintainable direction is to remove or redact operator-only fields from the model-facing schema and ignore model-supplied values at dispatch while preserving trusted config/default sources.

## Security Review

Status: needs_attention

Summary: Current main still lets model-visible exec arguments influence host exec policy, which can create denials that look operator-enforced.

  - body: The host approval helper derives host security from the supplied security value and throws the same `security=deny` denial family described in the report.
