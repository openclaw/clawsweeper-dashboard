## Summary

Keep open: current main still exposes and consumes model-supplied exec policy fields, and the linked PR is an unmerged implementation candidate with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to keep exec policy controls such as `security`, `ask`, `host`, `elevated`, and `node` operator-owned instead of model-controllable tool arguments.

## Best Possible Solution

Land a narrow fix that removes or ignores operator-only exec policy fields from model tool calls while preserving configured defaults, authorized `/exec` session overrides, and trusted chat-command overrides.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main exposes the policy fields in `execSchema` and then uses the supplied args to derive effective exec policy. I did not run the external vLLM/Hermes harness, so this is source-reproducible rather than live-reproduced here.

## Solution Assessment

Yes, the best direction is to remove or strip these operator-only fields at the model boundary and keep policy in config or authorized session state. The linked PR appears to target that narrow fix, so the issue should stay open for PR review rather than receive a separate repair job.

## Security Review

Status: needs_attention

Summary: Current main has a concrete exec policy-boundary weakness: model tool args can make effective exec policy more restrictive or alter allowed routing/elevation choices.

Concerns:

- **[medium] Keep exec policy out of model-controlled args:** `src/agents/bash-tools.exec.ts:1354`
  - body: The schema exposes policy fields and dispatch consumes them from tool-call args, so a model can self-deny commands with `security: "deny"` or `allowlist` and produce misleading downstream answers.
