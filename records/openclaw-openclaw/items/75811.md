## Summary

Keep open. Current main still exposes the reported exec policy fields in the model-facing tool schema and still reads model-supplied `security`, `ask`, `host`, `elevated`, and `node` values when deriving the effective exec target and host approval policy.

## What This Changes

The issue asks OpenClaw to separate model-controllable exec arguments from operator-controlled exec policy fields so generated tool calls cannot self-deny or otherwise override session/config policy.

## Best Possible Solution

Keep this open and implement an exec boundary that removes or redacts operator-owned policy fields from model-visible tool schemas, ignores or logs model-supplied policy fields at dispatch, and keeps authorized `/exec`, config, session, and host-approval paths as the source of policy truth.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction exists by creating the exec tool with permissive defaults and passing tool-call args containing `security: "deny"` or `security: "allowlist"`; current code will derive restrictive policy from those args before host execution. The reported Hermes/vLLM stack was not run live in this read-only sweep.

## Solution Assessment

Yes: separating the model-facing command schema from operator/session policy is the narrowest maintainable direction. The safer implementation is to both hide these fields from provider tool definitions and normalize/ignore them at the exec dispatch boundary so older model outputs cannot still affect policy.
