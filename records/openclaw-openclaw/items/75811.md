## Summary

Keep open. Current main still exposes exec policy fields in the model-facing schema and still folds model-supplied `security`, `ask`, `host`, `elevated`, and `node` values into exec dispatch, matching the reported self-denial path.

## What This Changes

The issue requests separating model-controllable exec arguments from operator-owned policy fields so generated tool calls cannot override or self-tighten exec policy.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session `/exec` policy paths, and cover the boundary with schema and exec-policy tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is available by invoking the exec tool with otherwise permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current code will derive restrictive policy from those args before host execution.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction for this bug.
