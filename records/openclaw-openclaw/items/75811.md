## Summary

Keep open. Current main still exposes exec policy controls in the model-facing schema and still consumes model-supplied `security`, `ask`, `host`, `elevated`, and `node` values during dispatch, so the reported self-denial path remains present.

## What This Changes

The issue requests that exec and analogous tool schemas stop exposing operator-only policy fields as model-controllable arguments.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema and exec-policy tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an exec call that supplies `security: "deny"` or `security: "allowlist"` reaches effective host policy through `params.security`, producing the same denial path described in the report.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with dispatch-time compatibility normalization, is the narrowest maintainable direction; a docs-only or error-message change would leave model-controllable policy intact.
