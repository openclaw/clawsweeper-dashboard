## Summary

Keep open. Current main still exposes exec policy controls to the model-facing tool schema and still folds model-supplied `security`, `ask`, `host`, `elevated`, and `node` values into exec dispatch, so the self-denial path described in the report remains present.

## What This Changes

Requests moving exec policy controls out of the model-facing `exec` schema and normalizing or ignoring model-supplied policy values at dispatch.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime inputs, keep operator policy in config/session/approval paths, normalize or reject model-supplied policy fields at dispatch, and cover the boundary with schema plus exec-policy tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a tool call with `security: "deny"` reaches `params.security`, becomes effective host policy, and produces the same `exec denied: host=gateway security=deny` path described in the report.

## Solution Assessment

Yes. Splitting model-controllable arguments from operator-controlled policy, with compatibility normalization at dispatch, is the narrow maintainable fix; a docs-only or error-message change would leave the policy boundary model-controllable.
