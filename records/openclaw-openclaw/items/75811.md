## Summary

Keep open. Current main still exposes exec policy/routing fields to the model-facing tool schema and still folds model-supplied `security`, `ask`, `host`, `elevated`, and `node` values into exec dispatch, so the reported self-denial path remains valid.

## What This Changes

The issue asks OpenClaw to separate model-controllable exec arguments from operator-controlled exec policy fields such as `security`, `ask`, `host`, `elevated`, and `node`.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema, OpenAI conversion, and exec-policy tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by invoking the exec tool with otherwise permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current dispatch normalizes those args into effective host policy before approval handling.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction; a broad tool-policy redesign is not needed to fix the confirmed exec bug.
