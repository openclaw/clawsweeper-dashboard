## Summary

Keep open. Current main still exposes exec policy fields in the model-facing schema and still consumes model-supplied `security`, `ask`, `host`, `elevated`, and `node` values during dispatch, so the reported self-denial path remains present.

## What This Changes

The issue asks OpenClaw to remove or normalize operator-only exec policy fields from the model-controllable tool-call surface.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema and exec-policy tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is available by invoking the exec tool with permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current code folds that value into effective host policy before denial handling.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction; a pure docs or error-message change would leave the boundary bug intact.
