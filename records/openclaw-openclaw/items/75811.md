## Summary

Keep open: current main still exposes and consumes model-supplied `exec` policy fields, and the open closing PR is the right implementation path rather than closing the issue before merge.

## What This Changes

The issue requests that model-facing `exec` schemas hide operator-controlled policy fields and that model-supplied values be ignored or normalized at dispatch.

## Best Possible Solution

Land or revise PR #76370 so only command/runtime fields are model-facing and exec policy defaults remain operator/session-owned, with regression coverage for ignored model-supplied policy fields.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main publishes the policy fields in `execSchema` and then consumes them during exec dispatch. I did not live-run the reporter's external vLLM/Hermes setup.

## Solution Assessment

Yes, hiding or ignoring model-supplied `security`, `ask`, `host`, `elevated`, and `node` is the narrow maintainable fix; the open PR appears aimed at that boundary and should be reviewed before this issue closes.
