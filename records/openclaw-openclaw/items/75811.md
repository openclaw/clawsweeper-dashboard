## Summary

Keep open. The earlier ClawSweeper assessment still matches current main: exec policy fields remain in the model-facing schema and model-supplied values still feed the effective host policy, so the self-denial path is still present and no close reason applies.

## What This Changes

The issue reports that model-facing exec tool schemas expose operator-controlled policy fields, allowing models to supply restrictive values that make exec deny itself.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime fields, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema plus exec-policy tests and docs/changelog updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: invoke the exec tool with otherwise permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current code folds that value into effective host policy before denial handling.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with a compatibility normalizer at dispatch, is the narrowest maintainable direction; docs or error-message changes would leave the boundary bug intact.
