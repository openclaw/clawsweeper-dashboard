## Summary

Keep open. Current main still exposes the reported exec policy fields in the model-facing tool schema and still consumes model-supplied `security`, `ask`, `host`, `node`, and `elevated` values while deriving the effective exec policy; adjacent exec-denial reports do not cover this model/operator boundary.

## What This Changes

The issue requests separating model-controllable exec arguments from operator-owned exec policy fields so model-generated tool calls cannot self-deny or alter host execution policy.

## Best Possible Solution

Keep the issue open and implement a boundary where provider-facing exec schemas expose only command execution inputs, while `/exec`, config, session defaults, and host approval files remain the source of policy truth. Dispatch should ignore or log model-supplied policy fields so older model outputs cannot still affect effective policy.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction exists by creating the exec tool with permissive gateway defaults and passing tool-call args containing `security: "deny"` or `security: "allowlist"`; current code normalizes those args into effective policy before host execution. The live Hermes/vLLM stack was not run in this read-only review.

## Solution Assessment

Yes: separating the model-facing schema from operator/session policy is the narrowest maintainable direction. The safer implementation is to both hide the policy fields from provider tool definitions and normalize or ignore them at dispatch.
