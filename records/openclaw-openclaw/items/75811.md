## Summary

Keep open. Current main still exposes exec policy controls in the model-facing tool schema and still folds caller-supplied `security`, `ask`, `host`, `elevated`, and `node` values into exec dispatch, matching the reported self-denial path.

## What This Changes

The issue asks OpenClaw to stop exposing operator-controlled exec policy fields such as `security`, `elevated`, `ask`, `host`, and `node` as model-controllable tool arguments.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema, adapter, and exec-policy tests plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is available by invoking exec with otherwise permissive defaults and tool-call args containing `security: "deny"` or `security: "allowlist"`; current main derives restrictive policy from those args before host execution.

## Solution Assessment

Yes. Splitting provider-visible command arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction; prompt guidance or error-message changes would leave model control over policy fields intact.
