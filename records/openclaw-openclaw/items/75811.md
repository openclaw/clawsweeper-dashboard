## Summary

Keep open. Current main still exposes exec policy controls to model-facing tool schemas and still folds model-supplied `security`, `ask`, `host`, `elevated`, and `node` values into dispatch policy, so the reported self-denial path remains present.

## What This Changes

The issue requests moving exec policy fields out of model-controllable tool arguments and enforcing them from operator/session policy instead.

## Best Possible Solution

Add a separate model-facing exec schema, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema and exec-policy regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an exec call that supplies `security: "deny"` or `security: "allowlist"` reaches effective host policy through `params.security`, producing the same denial modes described in the report.

## Solution Assessment

Yes. Splitting provider-visible tool arguments from operator/session policy, with compatibility normalization at dispatch, is the narrowest maintainable direction; changing only docs or denial text would leave model-controllable policy intact.
