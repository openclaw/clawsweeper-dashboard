## Summary

Keep open. Current main still exposes exec policy controls to model-facing tool schemas and still folds caller-supplied `security`, `ask`, `host`, `elevated`, and `node` values into dispatch decisions, so the reported self-denial path remains present.

## What This Changes

The issue asks OpenClaw to prevent model-generated `exec` tool calls from controlling operator policy fields such as `security`, `ask`, `host`, `elevated`, and `node`.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime inputs, normalize or ignore model-supplied policy fields at dispatch, preserve authorized config/session policy paths, and cover the boundary with schema and exec-policy tests plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an exec tool call that supplies `security: "deny"` or `security: "allowlist"` still reaches effective host policy and can produce the exact denial paths described in the issue.

## Solution Assessment

Yes. Splitting model-controllable command arguments from operator/session policy is the narrowest maintainable direction; changing only docs or error text would leave model-controllable policy intact.
