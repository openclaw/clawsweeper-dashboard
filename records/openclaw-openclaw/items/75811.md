## Summary

Keep this open: current main still exposes and consumes model-supplied exec policy fields, and the already-linked fix PR is open but unmerged, so the issue should remain open until that work lands or is replaced.

## What This Changes

The issue asks OpenClaw to remove or ignore model-controllable exec policy fields such as `security`, `ask`, `host`, `elevated`, and `node` from the model-facing exec boundary.

## Best Possible Solution

Land or replace the focused fix that removes operator-only exec controls from the model-facing schema and derives policy only from trusted config/session defaults.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main exposes the policy fields in `execSchema` and then applies model-supplied `security`, `ask`, `host`, and `elevated` values during exec dispatch. I did not run the external Hermes/vLLM harness, but the denial path for `security: "deny"` is clear from the code.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: remove operator-only fields from model-facing tool args and keep trusted `/exec`/chat-command policy as server-side defaults. This issue should stay open until #76370 merges or an equivalent fix lands.
