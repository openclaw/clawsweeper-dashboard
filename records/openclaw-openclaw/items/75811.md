## Summary

Keep open. Current main still exposes exec policy/routing fields in the model-facing tool schema and still folds model-supplied `security`, `ask`, `host`, `elevated`, and `node` values into exec dispatch, so the reported self-denial path remains valid.

## What This Changes

The issue asks OpenClaw to split model-controllable exec arguments from operator-controlled exec policy fields so models cannot self-impose denial or alter routing/approval behavior.

## Best Possible Solution

Add a model-facing exec schema limited to command/runtime arguments, normalize or reject model-supplied policy fields at dispatch, preserve authorized config/session `/exec` controls, and cover the boundary with schema, dispatch, docs, and changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is available by invoking `exec` with otherwise permissive defaults and args containing `security: "deny"` or `security: "allowlist"`; current code derives restrictive policy from those args before host execution.

## Solution Assessment

Yes. Splitting provider-visible exec arguments from operator/session policy, with a dispatch-time compatibility normalizer, is the narrowest maintainable direction; the exact reject-vs-ignore behavior needs maintainer/security review.
