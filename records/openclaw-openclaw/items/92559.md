- `P3`: The issue is a provider ergonomics feature request with limited blast radius rather than an urgent regression.
- `impact:auth-provider`: The request affects provider request-body routing and metadata for OpenAI-compatible backends.
- `impact:session-state`: The requested values would be derived from session/channel context and used for per-session provider isolation.
- `impact:security`: The request affects privacy isolation and would expose selected session metadata to provider request bodies if implemented.

## Summary

Keep open. Current main and v2026.6.6 still implement only static `extra_body` pass-through, while this request asks for dynamic session/channel-aware request metadata; related older PRs are closed or too narrow, and the open OpenRouter PR intentionally covers only `session_id`.

## What This Changes

The issue requests `{{session.*}}`-style template variables in `params.extra_body` so OpenAI-compatible provider request fields can vary per session or channel.

## Best Possible Solution

Choose one privacy-safe provider request-context design, then implement either generic `extra_body` interpolation or a provider-hook/session-identity seam that covers DeepSeek and OpenRouter without duplicating semantics.

## Reproduction Assessment

Yes for the current limitation from source inspection: configured `extra_body` is captured as a static sanitized record and merged into each OpenAI-compatible payload without session interpolation.

## Solution Assessment

Unclear. Direct templating in `extra_body` is plausible, but the best fix needs a maintainer decision on allowed session variables, escaping/secret boundaries, and alignment with the related OpenRouter session-id request.
