## Summary

Keep open: current main and v2026.6.6 only support static `extra_body` pass-through, while the requested session-scoped provider identifier remains unimplemented. The DeepSeek `user_id` use case is concrete, but adding request-time templating is a provider/config semantics decision, not conservative cleanup. ([api-docs.deepseek.com](https://api-docs.deepseek.com/zh-cn/quick_start/rate_limit))

## What This Changes

Allow `params.extra_body` values to use request-time session/channel template variables such as `{{session.channel}}` so one agent can send per-session provider identifiers.

## Best Possible Solution

If maintainers approve the product direction, add a narrow request-time resolver for vetted `extra_body` session fields, document the variable contract, and prove static values remain backward-safe.

## Reproduction Assessment

Not applicable as a feature request. Source and docs show `extra_body` is currently a static pass-through object, and repository search found no `{{session.*}}` template support for that path.

## Solution Assessment

Unclear. The requested direction is useful, but the best implementation should be a constrained session-metadata resolver with explicit privacy and compatibility rules rather than a broad template engine by default.
