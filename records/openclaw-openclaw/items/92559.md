## Summary

Keep open. Current main and v2026.6.9 still only support static `extra_body` pass-through; the related OpenRouter work is a narrower partial overlap, and the generic session/channel template request still needs maintainer product and security review.

## What This Changes

The issue requests per-session template expansion in provider `extra_body` values so fields like `user_id` can be derived from session/channel context at request time.

## Best Possible Solution

Design an explicit opt-in session-attribution/template seam with constrained, sanitized fields, provider-boundary docs, and focused tests before implementing generic request-body templating.

## Reproduction Assessment

No bug-style reproduction is applicable because this is an additive feature request. Source inspection gives high confidence that current main only passes static `extra_body` objects and has no session/channel template expansion path.

## Solution Assessment

Unclear as requested. Template variables are plausible, but maintainers should first choose the opt-in syntax, provider SDK boundary, and identifier-sanitization policy; the narrower OpenRouter PR should not be treated as the generic solution.
