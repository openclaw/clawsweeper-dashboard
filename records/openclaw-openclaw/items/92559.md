## Summary

Keep open: current main and the latest release only support static `extra_body` pass-through, while the requested session/channel-aware value resolution remains a real provider metadata feature request with privacy and API-design implications.

## What This Changes

The issue requests session-aware template variables in `extra_body` so a single agent can send per-session or per-channel provider request metadata such as `user_id`.

## Best Possible Solution

Design a request-scoped, opt-in provider metadata contract with sanitized session identity, then document and test how OpenAI-compatible `extra_body` or provider hooks consume it.

## Reproduction Assessment

Yes for the source-level capability gap: current main resolves `extra_body` only from static config and does not pass session/channel identity into that wrapper. This is a feature request, not a broken documented behavior repro.

## Solution Assessment

Unclear. `extra_body` templates are plausible, but the safer maintainable direction may be a narrower request-scoped provider identity/metadata contract rather than arbitrary templating in static config.
