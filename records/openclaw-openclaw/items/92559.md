## Summary

Keep open. Current main and the latest release support static `extra_body` pass-through, but they do not resolve session-aware template variables or expose session/channel context to the provider extra-param path; the remaining work needs maintainer product and privacy review.

## What This Changes

Request adds per-request session/channel template expansion for `extra_body`, such as resolving `{{session.channel}}` into a provider request field.

## Best Possible Solution

Design an explicit opt-in session-attribution/template seam that resolves a constrained, sanitized session context at request time, with provider SDK/docs/tests, rather than making all existing `extra_body` strings dynamic by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an additive feature request. Source inspection shows current main only passes static `extra_body` values and does not expose session/channel context to that path.

## Solution Assessment

Unclear as requested. Template variables are plausible, but maintainers should first choose the opt-in syntax, provider SDK boundary, and identifier-sanitization policy.

## Risks / Open Questions

- Resolving session/channel values into provider request bodies can leak stable channel, peer, agent, tenant, or task identifiers to third-party providers unless the design is explicit and sanitized.
- Treating existing string values in `extra_body` as templates could change literal config behavior unless templating is opt-in or tightly scoped.
- The right boundary is unsettled: this could be a core `extra_body` template feature, a provider SDK session-attribution seam, or a provider-owned opt-in wrapper.
