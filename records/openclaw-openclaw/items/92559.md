## Summary

Keep open. Current main supports static `extra_body` pass-through, but it does not resolve session/channel template variables there; the OpenRouter sibling issue and PR are narrower, and the earlier provider-metadata PRs are closed unmerged, so no cleanup close reason applies.

## What This Changes

The issue requests per-session template variables in provider `extra_body` so one agent can send different request metadata per channel/session.

## Best Possible Solution

Adopt a maintainer-approved, opt-in provider metadata design that exposes sanitized session attribution to provider hooks or request payloads without treating session keys as an auth boundary.

## Reproduction Assessment

Yes. Source inspection shows configured `extra_body` is merged statically into OpenAI-compatible payloads, and the provider extra-param/wrap contexts do not receive session/channel data for dynamic resolution.

## Solution Assessment

Unclear as proposed. Literal `extra_body` template expansion would satisfy the example, but a provider SDK seam or opt-in sanitized attribution object may be safer and easier to keep plugin-owned.
