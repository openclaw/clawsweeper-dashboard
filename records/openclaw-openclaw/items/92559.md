## Summary

Keep open. Current main still treats `extra_body` as static provider params and the provider hook/request context does not expose session or channel identity, so the requested session-aware interpolation remains real but needs maintainer design review rather than cleanup closure.

## What This Changes

The issue requests per-session template variables in provider `extra_body` values, such as resolving `{{session.channel}}` at request time.

## Best Possible Solution

Add a narrow documented request-time provider context for safe session identity, then support allowlisted interpolation or a provider-owned hook with tests for static literals, interpolated values, and OpenAI-compatible payload merging.

## Reproduction Assessment

Yes, for the missing capability: source inspection shows an `extra_body` object with a string like `{{session.channel}}` would be sanitized and assigned into the request payload unchanged. I did not run a live provider request, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as proposed. Template variables in `extra_body` are plausible, but the best fix should first settle whether session identity belongs in core request shaping, the provider plugin API, or a narrower provider/cache-key contract.
