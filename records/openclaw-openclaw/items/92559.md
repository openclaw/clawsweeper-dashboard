## Label Justifications

- `P2`: This is a normal-priority provider capability improvement with concrete cache isolation and privacy impact but no current core outage.
- `impact:auth-provider`: The request changes provider request-body identity fields used for provider-side routing, cache isolation, and attribution.
- `impact:session-state`: The reported problem is that multiple OpenClaw sessions or channels share provider-side cache identity when one agent uses static params.
- `impact:security`: The requested solution can intentionally expose session or channel identifiers to third-party provider logs, so privacy semantics matter.

## Summary

Keep open: current main supports only static `extra_body` pass-through, while the related provider-stream identity and header-templating threads do not implement config-level `extra_body` session variables.

## What This Changes

The issue requests session/channel-aware template variables in provider `extra_body` values so one agent can set per-session request body fields.

## Best Possible Solution

Define a narrow privacy-safe provider request identity or template contract, coordinated with the existing provider hook and header-templating issues, before implementing explicit `extra_body` placeholders with focused tests and docs.

## Reproduction Assessment

Yes from source for the absence of the requested feature: `extra_body` is sanitized and applied as static configured JSON, and docs define no template variables. There is no failing bug reproduction because the expected templating behavior is not an existing contract.

## Solution Assessment

No close decision is warranted. The proposed template syntax is plausible but not proven to be the best permanent API until maintainers choose between generic interpolation, a typed provider identity field, or a plugin hook seam.
