## Summary

Keep open for maintainer review: source inspection supports the reported memory-core bug and the narrowed patch, but the PR deliberately changes existing semantic-index outage behavior from silent fallback reindexing to a sync error.

## What This Changes

The branch updates memory-core to clear cached rejected embedding-provider initialization promises and to abort sync when an existing semantic index would be rewritten as FTS-only because no runtime provider is available.

## Best Possible Solution

Land the narrowed retry and semantic-index protection after maintainer acceptance of the fail-closed outage behavior, ideally with focused regression tests for retry, protected semantic indexes, and preserved fresh/FTS-only fallback.

## Reproduction Assessment

Yes, from source: current main can retain a rejected providerInitPromise and can full-reindex semantic metadata to fts-only when the runtime provider is null. I did not run a live provider outage in this read-only review.

## Solution Assessment

Yes: the two-part patch targets both failure modes and narrows the fail-closed guard to existing semantic metadata, preserving fresh and already-FTS-only fallback. The remaining question is maintainer acceptance of the compatibility tradeoff.
