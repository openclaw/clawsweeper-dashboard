## Summary

Keep this PR open: the rejected-promise retry fix is directionally correct, and the report includes useful real-world proof, but the new runSync guard is too broad and would break the supported default auto/FTS-only indexing path for users without an embedding provider.

## What This Changes

The PR clears cached failed memory embedding provider initialization promises and adds a runSync guard that aborts sync when a non-none memory provider is configured but unavailable.

## Best Possible Solution

Keep the retry fix, but narrow the sync abort to cases where provider loss would overwrite an existing semantic index, while preserving fresh/default FTS-only auto indexing and adding regression tests for both paths.

## Reproduction Assessment

Yes, partially from source: current main clearly caches a rejected providerInitPromise and can reindex provider:null as fts-only when meta is semantic. I did not establish the exact explicit-local-provider outage path because explicit providers throw before runSync, while auto no-provider returns provider:null.

## Solution Assessment

No: the retry fix is appropriate, but the runSync guard is not the best shape because it rejects supported default FTS-only indexing. The maintainable fix should protect existing semantic vectors without removing the current auto/FTS-only path.
