## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This rewrites the active `SessionEntry` to the target session but reapplies only the small `SessionHistoryItem.metadata` subset. Current main stores auth profile overrides, CLI/ACP runtime handles, exec/elevated policy, runtime model/fallback state, plugin slots, and other per-session fields on the same entry, and later turns read them directly. Switching can therefore resume one transcript under another session's credentials, runtime, or execution policy; restore or clear the full durable boundary before persisting the switch.
  - confidence: 0.88
- **[P1] Block switches when history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `/sessions` still builds targets from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0`, stale history remains listable and switchable, and this block pushes the current session then immediately evicts and archives it because the limit is zero. Return the disabled-history response before listing or resolving switch targets when the configured limit is `0`.
  - confidence: 0.86
- **[P2] Reference retained history transcripts during cleanup:** `src/config/sessions/types.ts:119-130`
  - body: The new `SessionHistoryItem` intentionally keeps old transcript paths switchable, but current disk-budget cleanup only marks each active entry transcript and compaction checkpoint files as referenced. With `session.maintenance.maxDiskBytes` enabled, retained history transcripts can be pruned as unreferenced while `/sessions` still advertises them, so switching back can reopen an empty or missing transcript.
  - confidence: 0.82
