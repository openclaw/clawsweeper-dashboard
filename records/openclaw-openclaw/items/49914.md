## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restore the complete session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: This rewrites the live `SessionEntry` to the target session and then reapplies only a small metadata subset. Fields such as auth profile overrides, CLI/ACP runtime handles, exec/elevated policy, runtime model/fallback state, usage mode, and cost can remain from the outgoing session, so the resumed transcript can run under the wrong credentials, backend, or execution policy.
  - confidence: 0.9
- **[P2] Protect retained history transcripts from cleanup:** `src/config/sessions/types.ts:257`
  - body: The new `sessionHistory` field intentionally leaves old transcripts switchable, but the current disk-budget reference counter only counts each active `entry.sessionId`. A deployment with session disk-budget enforcement can delete retained history transcripts while `/sessions` still lists them, making switch-back land in a missing or empty transcript.
  - confidence: 0.84
- **[P2] Honor disabled history before resolving switch targets:** `src/auto-reply/reply/commands-sessions.ts:258`
  - body: `historyLimit` is only read after targets are built from existing `sessionHistory`, so changing `session.historyLimit` to `0` can still expose and switch to stale retained sessions until another reset clears them. The disable setting should block or clear existing switch targets consistently.
  - confidence: 0.82
