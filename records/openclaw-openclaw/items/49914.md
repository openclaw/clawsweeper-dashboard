## Summary

Keep this PR open because current main still lacks chat-channel session switching, but the proposed patch is not ready to merge: the switch path creates a security-sensitive session-boundary leak by mutating one live `SessionEntry` and restoring only an allowlisted metadata subset.

## What This Changes

The PR adds a `/sessions` chat command, per-SessionKey LRU `sessionHistory`, `session.historyLimit` config/schema support, docs/generated config artifacts, and tests for chat session switching and reset history behavior.

## Best Possible Solution

Keep the feature direction, but implement chat switching as a real session boundary: either restore a complete audited per-session snapshot or keep retained sessions as separate store entries and switch a pointer, with lifecycle hooks, disk-budget references, docs, changelog, and tests aligned to that model.

## Reproduction Assessment

Yes. The PR-specific failures are source-reproducible: create two retained direct-chat sessions with different auth profile, exec/elevated policy, CLI/ACP handles, abort/fallback, fast/usage/cost, or plugin state, then switch through `handleSessionsListCommand`; only the metadata subset is restored and omitted fields survive from the outgoing session.

## Solution Assessment

No. The requested capability is useful, but this patch is not the narrowest maintainable solution because it mutates one live `SessionEntry` with a partial allowlist; a complete snapshot contract or separate-entry pointer model would be safer and easier to audit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore complete session-scoped state on switch:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This switch path rewrites the live `SessionEntry` into the target session and then reapplies only the small history metadata subset. Fields such as auth profile overrides, elevated/exec policy, CLI/ACP runtime handles, abort state, runtime model fields, fallback notices, fast/usage/cost state, and plugin-owned session slots can survive from the outgoing session, so resumed sessions may run under the wrong credentials, sandbox policy, or provider-side thread.
  - confidence: 0.92
- **[P2] Honor historyLimit=0 before resolving switches:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is checked only after targets are built from persisted `sessionHistory`. If an operator changes `session.historyLimit` to `0` while stale history exists, `/sessions` can still expose and switch to old sessions, then trim/archive the current one as the queue is forced to zero.
  - confidence: 0.87
- **[P2] Archive disabled-history entries before dropping them:** `src/auto-reply/reply/session.ts:650-651`
  - body: When history is disabled, this path clears the retained `sessionHistory` array but later archives only the immediately previous session. Older retained history entries are removed from state without the reset/archive lifecycle, leaving transcript files orphaned.
  - confidence: 0.84
- **[P2] Mark history transcripts as referenced during sweeps:** `src/config/sessions/types.ts:253-257`
  - body: The PR intentionally keeps `sessionHistory` transcripts switchable, but the disk-budget sweep still only treats active session artifacts and compaction checkpoints as referenced. With `session.maintenance.maxDiskBytes` enabled, a retained history transcript can be removed while `/sessions` still lists it.
  - confidence: 0.82
