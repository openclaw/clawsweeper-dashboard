Full review comments:

- **[P1] Restore the complete session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This switch rewrites the active `SessionEntry` and then reapplies only the small `SessionHistoryItemMetadata` subset. Fields such as `authProfileOverride`, CLI/ACP handles, elevated/exec policy, runtime model fields, abort state, fallback notices, and system prompt reports can remain from the outgoing session, so the next turn can run under the wrong credentials, runtime, or sandbox policy.
  - confidence: 0.88
- **[P2] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/session.ts:648-650`
  - body: The PR intentionally keeps old transcripts switchable in `sessionHistory`, but the existing disk-budget referenced set only protects the active entry and compaction checkpoints. With `session.maintenance.maxDiskBytes` enabled, cleanup can delete a retained transcript while `/sessions` still lists it, making the restored session empty or stale.
  - confidence: 0.83
- **[P2] Honor disabled history before resolving switch targets:** `src/auto-reply/reply/commands-sessions.ts:212-214`
  - body: `session.historyLimit=0` is documented as disabling switchable history, but this handler builds list/switch targets from persisted `sessionHistory` before checking the limit. If an operator lowers the limit to `0`, stale history can still be listed and selected, and the current session is then pushed and immediately evicted.
  - confidence: 0.81
