## Summary

Keep open: this is a useful chat-session feature PR, but the current branch still has blocking session-switching correctness issues and no required after-fix real chat proof.

## What This Changes

The PR adds a `/sessions` chat command, per-SessionKey LRU session history, `session.historyLimit`, metadata restoration, config/docs updates, and focused tests.

## Best Possible Solution

Finish this as a single well-tested session-transition path that restores or clears all per-session runtime state, protects retained transcripts from maintenance cleanup, wires the documented command forms, and includes redacted live chat proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head only matches `/sessions`, checks disabled history after building stale targets, and snapshots too little of `SessionEntry`. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is plausible, but the current implementation should not merge until the session-state contract is complete, disabled-history behavior is enforced, command UX matches the advertised API, and real chat proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore complete session state when switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` omits many fields that are still session-scoped, including auth profile overrides, CLI bindings, ACP state, runtime model/fallback state, abort cutoffs, cost, and exec/elevated policy. Because the switch mutates the single live `SessionEntry`, the resumed session can inherit the previous session's credentials, execution defaults, provider-side thread, or status state; please snapshot/restore or explicitly clear the full per-session state through one helper.
  - confidence: 0.9
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:148-150`
  - body: The PR body and linked feature request advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but the new handler only matches `/sessions`. The existing `/session` command remains the lifecycle handler on main, so users following the documented command form still get usage instead of switching.
  - confidence: 0.93
- **[P2] Reject switches when session history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: If `session.historyLimit` is changed to `0` while stored `sessionHistory` entries still exist, this code still resolves a target and then immediately evicts the current session from `nextHistory`. That makes old sessions switchable after the feature is disabled and can archive the conversation the user is currently in; return a disabled-history message before listing or switching stale targets.
  - confidence: 0.87
- **[P2] Protect retained history transcripts from disk cleanup:** `src/config/sessions/types.ts:119-130`
  - body: The PR intentionally keeps transcripts in `sessionHistory` switchable, but current disk-budget cleanup only marks active store entries and compaction checkpoints as referenced. Without adding history items to that referenced set, maintenance can delete a retained transcript while `/sessions` still lists it, so switching back opens an empty or missing conversation.
  - confidence: 0.78
- **[P3] Match ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is treated as a rendered row index before exact or prefix session-id matching. A valid session id prefix such as `1234` therefore can never select the matching session and instead jumps to row 1234 or reports not found, contradicting the advertised prefix behavior.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff can leak session-scoped auth and execution policy across switches because the history snapshot restores only a small subset of runtime state.

Concerns:

- **[medium] Execution and auth state can leak across switches:** `src/config/sessions/types.ts:81`
  - body: `SessionHistoryItemMetadata` excludes fields such as `elevatedLevel`, `execHost`/`execSecurity`, auth profile overrides, CLI bindings, and ACP state. After a switch, later turns can run with the previous session's credentials, sandbox, approval policy, or provider-side thread.
  - confidence: 0.88
