## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the full session state on switch:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This rewrites the live `SessionEntry` and then applies only `SessionHistoryItem.metadata`. Current main stores auth profile, exec/elevated policy, CLI/ACP runtime, pending delivery, fallback, cost, and plugin state on the same object; any field absent from metadata is inherited from the previous session, so `/sessions back` can resume under the wrong credentials/runtime or execution policy. Snapshot/restore or clear the complete current-main session-scoped contract before persisting the switch.
  - confidence: 0.9
- **[P2] Honor disabled history before resolving targets:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is checked only after a switch target is built from persisted `sessionHistory`. If an operator changes `session.historyLimit` to `0`, old sessions can still be selected and the current session is then evicted/archived during the switch, contradicting the documented disabled-history contract.
  - confidence: 0.87
- **[P2] Archive retained entries when disabling history:** `src/auto-reply/reply/session.ts:650-651`
  - body: When `historyLimit === 0` on reset, this drops the retained history array but later archives only `previousSessionEntry`. Older retained transcript files become untracked instead of going through the reset archive path, so disabling history can orphan files that were still switchable moments earlier.
  - confidence: 0.82
