## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: `/sessions` switches the visible `sessionId` and transcript, then restores only `SessionHistoryItem.metadata`. Fields such as auth profile overrides, CLI/ACP bindings, execution policy, runtime model/fallback state, response usage, cost, and system-prompt reports remain from the outgoing session, so the next turn can continue under the wrong credentials or backend conversation. Store and restore the full intended boundary, or clear session-boundary fields when the target history item lacks them.
  - confidence: 0.9
- **[P1] Protect retained history transcripts from cleanup:** `src/config/sessions/types.ts:257`
  - body: The new `sessionHistory` entries keep old transcripts switchable, but current disk-budget cleanup only treats each active `entry.sessionId` and compaction checkpoints as referenced. Deployments with `session.maintenance.maxDiskBytes` can delete a retained transcript while `/sessions` still lists it, so switching back can land in an empty or missing conversation. Include history transcript paths in cleanup ownership or archive them instead of leaving them as unreferenced primaries.
  - confidence: 0.88
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: `/sessions` builds targets from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0` after history exists, the command still lists and can switch to old entries, then immediately evicts and archives the current session. Check `historyLimit === 0` before listing or resolving switch targets so the documented disabled mode is enforced immediately.
  - confidence: 0.84
