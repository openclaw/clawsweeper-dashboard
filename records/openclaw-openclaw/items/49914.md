## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the full session state on switches:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` is what gets reapplied after the live entry is rewritten, but it omits session-scoped auth profile overrides, exec/elevated policy, fast/usage flags, runtime model/provider, fallback/cost/status, CLI handles, system prompt report, and ACP state. Switching can therefore resume one transcript while continuing under another conversation's credentials, provider thread, or execution policy.
  - confidence: 0.9
- **[P2] Honor the advertised singular switch command:** `src/auto-reply/reply/commands-sessions.ts:18`
  - body: The PR body advertises `/session <number>`, `/session <sessionId>`, and `/session back`, but this handler only matches `/sessions`. Users following the documented singular command still fall through to the existing lifecycle/settings handler instead of switching sessions.
  - confidence: 0.88
- **[P2] Check disabled history before resolving old targets:** `src/auto-reply/reply/commands-sessions.ts:213-219`
  - body: The switch handler builds targets from persisted history before checking `session.historyLimit`. If history is changed to `0` after entries already exist, stale sessions can still be listed or selected, and selecting one evicts the current session under the disabled-history path.
  - confidence: 0.86
- **[P2] Match session ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-221`
  - body: Numeric input is interpreted as a rendered list index before exact or prefix session-id matching. A session id with a numeric-only unique prefix such as `1234...` cannot be selected by its documented prefix and may even switch to entry 1234 if enough sessions exist.
  - confidence: 0.82
- **[P2] Archive cleared history when disabling retention:** `src/auto-reply/reply/session.ts:650-651`
  - body: When `session.historyLimit` is `0`, this branch clears `sessionHistory` but only archives the immediately previous session later. Older retained history entries are dropped from state without being archived, leaving their transcript files outside the normal reset-archive retention flow.
  - confidence: 0.8
- **[P2] Preserve retained history transcripts during cleanup:** `src/auto-reply/reply/session.ts:691-692`
  - body: The PR intentionally keeps historical transcripts switchable while they remain in `sessionHistory`, but disk-budget cleanup still treats only each active `sessionId` as referenced. With `session.maintenance.maxDiskBytes`, retained transcripts can be removed while `/sessions` still lists them, so switching back can reopen an empty/default transcript.
  - confidence: 0.78
