## Summary

Keep this PR open: current main still lacks the requested built-in chat session switching and the linked feature issue was closed as superseded by this PR, but this branch is not merge-ready because source-visible session-state, command UX, cleanup, security, and real-behavior-proof blockers remain.

## What This Changes

The branch adds a `/sessions` chat command, per-SessionKey LRU `sessionHistory`, `session.historyLimit` config/schema support, metadata restore on switch, command-list filtering, and focused tests.

## Best Possible Solution

Finish one explicit chat-session switch contract that restores or clears all per-session state, protects retained transcripts from cleanup, aligns the slash-command UX with the request, and is proven in a real chat channel.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head `97550cee172cb5ab8ac933075177a9345d432137` only handles `/sessions`, reads `historyLimit` after resolving history, and applies only `SessionHistoryItemMetadata`. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this implementation should not merge until the session-state contract is complete, disabled history is enforced before listing or switching, retained transcripts are protected, the command UX matches the linked request, and real chat proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore complete session state when switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` is the only state reapplied after the handler rewrites the live `SessionEntry`, but it omits auth profile overrides, CLI/ACP runtime handles, exec/elevated policy, fast/usage flags, runtime model/fallback fields, abort cutoffs, and cost. A resumed session can inherit credentials, execution policy, provider-side thread, or status/cost from the session the user just left.
  - confidence: 0.9
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:148-150`
  - body: The linked request and PR body advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but this new handler only matches `/sessions`. The existing `/session` handler remains the lifecycle settings handler, so users following the requested command form will not switch sessions.
  - confidence: 0.92
- **[P2] Reject disabled history before resolving stale targets:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: When `session.historyLimit` is changed to `0` while old history is still stored, this path builds targets from `sessionHistory` before reading the limit. That keeps old sessions listable and switchable after history is disabled and can archive the current conversation when it is pushed then immediately evicted.
  - confidence: 0.88
- **[P2] Keep retained history transcripts referenced during cleanup:** `src/config/sessions/types.ts:253-257`
  - body: Adding `sessionHistory` makes those transcripts intentionally switchable, but disk-budget cleanup still only treats active store entries and compaction checkpoints as referenced. A retained history transcript can be pruned while `/sessions` still lists it, so switching back can reopen a missing or empty conversation.
  - confidence: 0.82
- **[P3] Match ID prefixes before numeric row indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is interpreted as a rendered row index before exact or prefix session-id matching. A valid session-id prefix such as `1234` can never select the matching session and instead jumps to row 1234 or reports not found, contradicting the advertised prefix behavior.
  - confidence: 0.82
