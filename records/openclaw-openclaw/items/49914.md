## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore or clear the full session boundary:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: `/sessions <target>` rewrites the live `SessionEntry` to the selected session but only reapplies the small history metadata subset. Fields such as auth profile, CLI/ACP bindings, execution policy, fast/usage settings, abort/fallback state, runtime model, and prompt reports can survive from the outgoing session, so the next turn can continue the wrong backend conversation or run under the wrong credentials/policy.
  - confidence: 0.91
- **[P1] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: The PR keeps old transcripts switchable by storing them in `sessionHistory`, but it does not update the session maintenance/disk-budget reference collector. With `session.maintenance.maxDiskBytes` enforcement, those retained primary transcripts still look unreferenced and can be removed while `/sessions` continues listing them.
  - confidence: 0.86
- **[P2] Honor disabled history before exposing switch targets:** `src/auto-reply/reply/commands-sessions.ts:213`
  - body: The command builds list and switch targets from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0`, stale history remains visible and switchable until another reset, which contradicts the advertised disable behavior and can archive the current session during the attempted switch.
  - confidence: 0.88
