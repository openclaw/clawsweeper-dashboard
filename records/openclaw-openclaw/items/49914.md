## Summary

Keep open: this is the canonical branch for the closed chat-session switching request, but the current head still has blocking source-visible command wiring, session-state, disabled-history, and transcript-retention defects, plus missing real chat proof.

## What This Changes

The branch adds a `/sessions` chat command, per-SessionKey LRU `sessionHistory`, `session.historyLimit` config/schema entries, metadata restore on switch, command-list filtering, and focused tests.

## Best Possible Solution

Finish a single explicit chat-session switch path that restores or clears all per-session state, protects retained transcripts from maintenance cleanup, matches the documented command API, and ships with redacted live chat proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head only matches `/sessions`, resolves stale history before disabled-history handling, and restores only a small metadata subset of the live session entry. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is plausible, but this implementation should not merge until the session-state contract is complete, disabled history is enforced before listing or switching, retained transcripts are protected, the command UX matches the linked request, and real chat proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore complete session state when switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` is the only state reapplied after the switch mutates the live `SessionEntry`, but it omits auth profile overrides, CLI/ACP runtime handles, exec/elevated policy, fast/usage flags, runtime model/fallback fields, abort cutoffs, and cost. A resumed session can inherit credentials, sandbox or approval policy, provider thread, or status/cost from the session the user just left.
  - confidence: 0.9
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:148-150`
  - body: The linked request and PR body advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but this handler only matches `/sessions`. The existing `/session` handler remains the lifecycle settings handler, so users following the requested command form will not switch sessions.
  - confidence: 0.92
- **[P2] Reject disabled history before resolving stale targets:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: When `session.historyLimit` is changed to `0` while old history is still stored, this path builds targets from `sessionHistory` before reading the limit. That keeps old sessions listable/switchable after history is disabled and can archive the current conversation when it is pushed then immediately evicted.
  - confidence: 0.88
- **[P2] Keep retained history transcripts referenced during cleanup:** `src/config/sessions/types.ts:257`
  - body: Adding `sessionHistory` makes those transcripts intentionally switchable, but disk-budget cleanup still only treats active store entries and compaction checkpoints as referenced. A retained history transcript can be pruned while `/sessions` still lists it, so switching back can reopen a missing or empty conversation.
  - confidence: 0.82
- **[P3] Match ID prefixes before numeric row indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is interpreted as a rendered row index before exact or prefix session-id matching. A valid session-id prefix such as `1234` can never select the matching session and instead jumps to row 1234 or reports not found, contradicting the advertised prefix behavior.
  - confidence: 0.82
