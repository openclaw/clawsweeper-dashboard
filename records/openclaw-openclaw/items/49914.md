## Summary

Keep open. Current main still lacks the requested chat session switching feature, so the PR is not obsolete, but the proposed patch is not safe to merge because the switch path mutates one live session record while restoring only a partial metadata subset, leaving security-sensitive and runtime session fields able to bleed between conversations.

## What This Changes

The PR adds a `/sessions` chat command with per-SessionKey LRU `sessionHistory` retention, `session.historyLimit` config/schema support, command visibility filtering, docs/generated config artifacts, and targeted tests for session switching and reset history behavior.

## Best Possible Solution

Keep the feature direction, but make chat switching a real session boundary. The maintainable end state is either a complete, audited per-session snapshot/restore contract or a pointer-based store model where each retained session remains its own entry; history retention, cleanup, disk-budget references, lifecycle hooks, docs, changelog, and tests should all follow that same contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create two direct-chat sessions under one SessionKey with different auth profile, exec/elevated policy, CLI/ACP runtime handles, abort/fallback state, fast/usage/cost fields, then switch via the PR path; the target receives only the metadata subset and can inherit omitted fields from the outgoing session.

## Solution Assessment

No. The requested capability is useful, but this patch is not the best solution because it mutates one live `SessionEntry` and relies on a partial allowlist of restored fields; a complete boundary snapshot or separate-entry pointer model would be safer and easier to audit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore complete session-scoped state on switch:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This switch path rewrites the live `SessionEntry` into the target session and then reapplies only the small history metadata subset. Fields such as auth profile overrides, elevated/exec policy, CLI/ACP runtime handles, abort state, runtime model fields, fallback notices, and cost totals can survive from the outgoing session, so resumed sessions may run under the wrong credentials, sandbox policy, or provider-side thread.
  - confidence: 0.91
- **[P2] Honor historyLimit=0 before resolving switches:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is checked only after targets are built from persisted `sessionHistory`. If an operator changes `session.historyLimit` to `0` while stale history exists, `/sessions` can still expose and switch to old sessions, then trim/archive the current one as the queue is forced to zero.
  - confidence: 0.86
- **[P2] Archive disabled-history entries before dropping them:** `src/auto-reply/reply/session.ts:650-651`
  - body: When history is disabled, this path clears the retained `sessionHistory` array but later archives only the immediately previous session. Older retained history entries are removed from state without the reset/archive lifecycle, leaving transcript files orphaned.
  - confidence: 0.84
- **[P2] Mark history transcripts as referenced during sweeps:** `src/config/sessions/types.ts:257`
  - body: The PR intentionally keeps `sessionHistory` transcripts switchable, but the disk-budget sweep still treats only active session artifacts and compaction checkpoints as referenced. With `session.maintenance.maxDiskBytes` enabled, a retained history transcript can be removed while `/sessions` still lists it.
  - confidence: 0.82
