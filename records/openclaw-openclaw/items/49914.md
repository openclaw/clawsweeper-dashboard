## Summary

Keep open: current main still lacks built-in chat session switching, and this PR still has blocking session-state, command-syntax, disabled-history, transcript-retention, and real-proof gaps.

## What This Changes

The branch adds a `/sessions` chat command, per-SessionKey session-history retention, session metadata snapshots, `session.historyLimit` config/schema/docs, and tests for switching retained chat sessions.

## Best Possible Solution

Finish the feature on a fresh branch with a shared complete session snapshot/restore helper, aligned `/session` and `/sessions` command semantics, disabled-history guards, disk-budget awareness for retained transcripts, and redacted proof from a real chat channel.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR branch shows the command matcher, partial metadata restore, disabled-history switch path, and disk-budget reference gap directly. I did not run tests under the read-only review constraint.

## Solution Assessment

No. The feature direction is useful, but this implementation should not merge until it restores complete per-session state, honors the advertised command forms, blocks disabled-history switching, preserves retained transcripts through maintenance, and includes real chat proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore complete session state before switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` snapshots only a subset of `SessionEntry`, but the switch path mutates the live entry into a different session. Fields such as exec/elevated policy, auth profile overrides, CLI/ACP handles, runtime model state, fallback state, cost totals, and stop cutoffs can therefore leak into the resumed session; please snapshot/restore or explicitly clear the full per-session runtime state through one shared helper.
  - confidence: 0.89
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:144-150`
  - body: The PR body and linked feature request advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but this new handler only matches `/sessions`. The existing `/session` handler remains the lifecycle command, so users following the advertised syntax will not switch sessions.
  - confidence: 0.93
- **[P2] Reject switches when history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: If `session.historyLimit` is changed to `0` while old `sessionHistory` entries still exist, the handler still builds targets from those entries and only applies the limit after selecting one. That lets `/sessions <n>` archive the current conversation instead of reporting that switchable history is disabled.
  - confidence: 0.87
- **[P2] Keep retained history transcripts out of cleanup:** `src/config/sessions/types.ts:119-130`
  - body: The PR intentionally keeps history transcripts on disk until LRU eviction, but the disk-budget sweep only treats the active transcript for each store entry as referenced. With `session.maintenance.maxDiskBytes`, retained history transcripts can be deleted while `/sessions` still lists them, so switching back can reopen an empty or missing transcript.
  - confidence: 0.82
- **[P3] Match ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is interpreted as a rendered list index before exact or prefix session-id matching. A valid unique session-id prefix like `1234` can never select a session whose id starts with those digits, despite prefix matching being advertised.
  - confidence: 0.78
