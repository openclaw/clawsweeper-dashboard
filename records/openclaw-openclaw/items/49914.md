## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: This switch mutates the live `SessionEntry` to the target session and then reapplies only the small `SessionHistoryItem.metadata` subset. Session-scoped fields such as auth profile overrides, CLI/ACP runtime handles, execution policy, runtime model/fallback state, fast/usage settings, and cost can remain from the previous session, so the resumed transcript can run under the wrong credentials, backend, or policy.
  - confidence: 0.88
- **[P1] Reference retained history transcripts during cleanup:** `src/config/sessions/types.ts:257`
  - body: `sessionHistory` intentionally keeps old transcripts switchable, but the disk-budget and unreferenced-artifact sweep only treats the active entry transcript and compaction checkpoints as referenced. A deployment with session disk-budget enforcement can delete a retained history transcript while `/sessions` still lists it, making switch-back land in an empty or missing transcript.
  - confidence: 0.85
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: When an operator changes `session.historyLimit` to `0` after history already exists, `/sessions` still builds targets from `sessionHistory`, and switching can push the current session and immediately evict/archive it. The disable setting should block or clear existing switch targets consistently instead of leaving a transient path to old sessions.
  - confidence: 0.82
