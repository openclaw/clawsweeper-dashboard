## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restore or clear the full session boundary:** `src/config/sessions/types.ts:81-112`
  - body: The new history metadata only stores a small subset of `SessionEntry`, but later turns read auth profile overrides, runtime model/provider fields, CLI/ACP handles, exec/elevated settings, abort/fallback/cost state, and other session-scoped fields. After this handler mutates `sessionId`, the resumed transcript can run with the outgoing session's credentials or sandbox/tool policy, so the switch path needs a complete restore-or-clear contract with regressions.
  - confidence: 0.86
- **[P1] Protect retained history transcripts from cleanup:** `src/config/sessions/types.ts:253-257`
  - body: This PR intentionally keeps old transcripts switchable by storing them under nested `sessionHistory`, but the existing disk-budget cleanup only counts top-level active `entry.sessionId` references before deleting session artifacts. A retained history transcript can therefore be removed while `/sessions` still lists it, so maintenance ref counting needs to include history entries or the command must not advertise unprotected files.
  - confidence: 0.82
- **[P2] Honor disabled history before selecting targets:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is resolved only after the target is picked from persisted history. If an operator changes `session.historyLimit` to `0` while history entries already exist, `/sessions` still lists and switches to them, then the eviction loop archives the current session; return disabled before listing/switching or clear stale history first.
  - confidence: 0.84
