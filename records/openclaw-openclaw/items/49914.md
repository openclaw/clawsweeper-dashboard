## Summary

Keep this PR open, but it is not merge-ready: source review still shows correctness/security blockers in the session switch state contract, and the external PR has no attached real behavior proof for the visible chat flow.

## What This Changes

The branch adds a `/sessions` chat command, per-SessionKey LRU session history, `session.historyLimit`, metadata restore logic, config/schema updates, and focused tests for chat-session switching.

## Best Possible Solution

Finish this as a single session-transition path that restores or clears all per-session runtime state, protects retained transcript artifacts from maintenance cleanup, supports the documented command forms, and includes redacted live chat proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head only matches `/sessions`, checks disabled history after resolving stale targets, and snapshots too little of `SessionEntry`. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is plausible, but the current implementation should not merge until the session-state contract is complete, disabled-history behavior is enforced, command UX matches the advertised API, retained transcripts are protected, and real chat proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore complete session state when switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` carries only a subset of fields, but the switch mutates the single live `SessionEntry`. Fields such as auth profile overrides, CLI bindings, ACP state, elevated/exec policy, fast/usage flags, runtime model/fallback state, abort cutoffs, and cost can remain from the previous active session, so the resumed session may use the wrong credentials, execution policy, provider thread, or status state.
  - confidence: 0.9
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:148-150`
  - body: The PR advertises `/session <number>`, `/session <sessionId>`, and `/session back`, but this new handler only matches `/sessions`. The existing `/session` handler remains the lifecycle settings handler, so users following the documented switch form will get lifecycle usage instead of switching.
  - confidence: 0.93
- **[P2] Reject switches before resolving disabled history:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: When `session.historyLimit` is changed to `0` while stored history entries still exist, this path still resolves targets from `sessionHistory` and then evicts the current session from `nextHistory`. That keeps old sessions switchable after history is disabled and can archive the conversation the user is currently in; return a disabled-history response before listing or switching stale targets.
  - confidence: 0.88
- **[P2] Keep history transcripts referenced during maintenance:** `src/config/sessions/types.ts:119-130`
  - body: The PR makes `sessionHistory` transcripts intentionally switchable, but disk-budget cleanup only treats active store entries and existing checkpoint artifacts as referenced. A retained history transcript can be pruned while `/sessions` still lists it, so switching back can reopen a missing or empty conversation.
  - confidence: 0.8
- **[P3] Match ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is interpreted as a rendered row index before exact or prefix session-id matching. A valid session id prefix such as `1234` can never select the matching session and instead jumps to row 1234 or reports not found, contradicting the advertised prefix behavior.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch can leak session-scoped auth and execution policy across switches because the history snapshot restores only a small subset of runtime state.

Concerns:

- **[medium] Execution and auth state can leak across switches:** `src/config/sessions/types.ts:81`
  - body: `SessionHistoryItemMetadata` excludes fields such as auth profile overrides, CLI bindings, ACP state, `elevatedLevel`, and `execHost`/`execSecurity`/`execAsk`/`execNode`. After a switch, later turns can run with the previous session's credentials, sandbox, approval policy, or provider-side thread.
