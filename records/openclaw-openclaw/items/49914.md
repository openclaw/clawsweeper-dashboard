## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restore the full session boundary on switch:** `src/config/sessions/types.ts:260-285`
  - body: `buildSessionHistoryMetadata` snapshots only a subset of `SessionEntry`, but `/sessions <target>` mutates the live entry into another session and then reapplies only that subset. Fields such as `authProfileOverride`, CLI/ACP bindings, `fastMode`, `responseUsage`, exec/elevated policy, abort cutoffs, cost, fallback state, and runtime model can therefore carry over from the previous conversation, so the resumed session can run under the wrong credentials, provider thread, execution policy, or status state.
  - confidence: 0.89
- **[P1] Honor disabled history before resolving switch targets:** `src/auto-reply/reply/commands-sessions.ts:213-214`
  - body: The handler builds switch targets from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0` while history entries exist, `/sessions` can still list them and `/sessions 2` can switch, then immediately evict/archive the current session because the configured limit is zero.
  - confidence: 0.86
- **[P2] Protect retained history transcripts from cleanup:** `src/config/sessions/types.ts:253-257`
  - body: The new `sessionHistory` entries intentionally keep old transcripts switchable, but session disk-budget cleanup still only treats each active store entry's current transcript path as referenced. A retained `<oldSessionId>.jsonl` can be deleted as an unreferenced primary transcript while `/sessions` still lists it, causing switch-back to reopen a missing or empty transcript.
  - confidence: 0.82
