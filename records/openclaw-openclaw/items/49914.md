## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore full session state when switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: This mutates the current `SessionEntry` into the target session but only reapplies a small metadata subset. Fields such as `authProfileOverride`, CLI bindings, ACP state, runtime model fields, exec/elevated policy, fallback notices, `fastMode`, `responseUsage`, and `systemPromptReport` can remain from the outgoing session, so the resumed transcript can run under the wrong credentials, provider thread, or tool policy.
  - confidence: 0.93
- **[P1] Block stale history when history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: When an existing store already has `sessionHistory` and the operator sets `session.historyLimit` to `0`, this path still resolves and switches to retained entries, then pushes and immediately evicts the current session. That exposes old sessions after the feature is disabled and can archive the active transcript unexpectedly; check the limit before listing or target resolution.
  - confidence: 0.91
- **[P2] Archive retained entries when disabling history on reset:** `src/auto-reply/reply/session.ts:650-651`
  - body: The reset path clears `sessionEntry.sessionHistory` when `historyLimit` is `0` but later archives only `previousSessionEntry`. Older retained history entries are removed from the store without normal archive cleanup, leaving transcript files orphaned after an operator disables the feature.
  - confidence: 0.85
