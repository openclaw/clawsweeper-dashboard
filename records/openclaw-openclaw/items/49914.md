Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: `applyHistoryMetadata` restores only a small subset after this command rewrites `sessionId`. Fields such as auth profile override, CLI/ACP runtime handles, runtime model state, fallback notice state, estimated cost, and exec/elevated policy can remain from the outgoing session, so the resumed conversation may continue under the wrong credentials, backend thread, or sandbox posture.
  - confidence: 0.91
- **[P1] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: This stores old sessions in `sessionHistory` so their transcripts remain switchable, but disk-budget cleanup still protects only active store entries and compaction checkpoints. With disk-budget enforcement, retained history transcripts can be removed while `/sessions` still lists them, causing a switch to land in an empty or missing transcript.
  - confidence: 0.86
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: `/sessions` builds its list from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0`, stale entries remain visible and switchable until rollover, contradicting the disable contract and archiving the outgoing current session during the switch path.
  - confidence: 0.84
- **[P3] Match session IDs before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric input is treated as a rendered list index before exact or prefix session-id matching. A numeric-only session-id prefix such as `/sessions 1234` can never select a matching session even though the command advertises prefix matching; check exact/prefix matches before index fallback or require an explicit index marker.
  - confidence: 0.77
