## Summary

## What This Changes

The PR adds a `/sessions` chat command, per-SessionKey `sessionHistory` retention, session-switch metadata restoration, `session.historyLimit`, docs, schemas, and targeted tests for chat-channel session switching.

## Best Possible Solution

Finish this as a chat-command feature by wiring the promised command forms, switching complete per-session state through a single well-tested helper, and proving the flow in a real chat channel before merge.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P1] Restore full session state when switching history:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` only snapshots a small subset of `SessionEntry`, but switching mutates the single live entry into a different session. Fields such as `elevatedLevel`, `execHost`/`execSecurity`, auth profile overrides, CLI session bindings, ACP state, fallback state, stop cutoffs, and runtime model fields can therefore leak from the previous active session into the resumed one, causing later turns to run with the wrong credentials, execution policy, or provider-side thread. Please snapshot/restore or explicitly clear the full set of per-session runtime fields via a shared helper before allowing switches.
  - confidence: 0.88
- **[P2] Wire the advertised /session switch forms:** `src/auto-reply/reply/commands-sessions.ts:144-150`
  - body: The PR body and the linked feature request advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but this new handler only matches `/sessions`. The existing `/session` handler still treats any non-`idle`/`max-age` argument as usage, so users following the requested command cannot switch sessions.
  - confidence: 0.92
- **[P2] Reject switches when history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-265`
  - body: If an operator changes `session.historyLimit` to `0` while old `sessionHistory` entries are still stored, the handler still builds switch targets from that stale history. Selecting one then pushes the current session into `nextHistory` and immediately evicts it, archiving the conversation the user is currently in instead of reporting that switchable history is disabled.
  - confidence: 0.86
- **[P3] Match ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric-only input is interpreted as a list index before exact or prefix session-id matching. A valid session id prefix like `1234` can never select a session whose id starts with those digits; it jumps to row 1234 or reports not found, despite prefix matching being advertised.
  - confidence: 0.8

## Security Review

Status: not_applicable

Summary: The patch can leak session-scoped execution policy across switches because exec/elevated fields are not snapshotted or cleared.

Concerns:

- **[medium] Execution policy leaks across session switches:** `src/config/sessions/types.ts:81`
  - body: The PR snapshots only a subset of session metadata, while existing runtime code persists and later reads `elevatedLevel` and `exec*` from `SessionEntry`. After a switch, the resumed session can inherit the previous session's sandbox/approval defaults, which is a concrete security-sensitive behavior regression.
  - confidence: 0.86
