## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restore or clear the full session boundary:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This switch rewrites `sessionId` and then applies only `SessionHistoryItem.metadata`. Current `SessionEntry` also carries auth-profile overrides, CLI/ACP bindings, execution/elevation defaults, abort/fallback state, delivery context, runtime model fields, costs, and plugin state; those fields remain from the outgoing session and can route the resumed session through the wrong credentials, provider thread, or tool policy. Please move switching through a canonical restore/clear boundary or intentionally snapshot every session-scoped field.
  - confidence: 0.87
- **[P1] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/session.ts:648-649`
  - body: The PR keeps previous transcripts switchable by copying `sessionHistory`, but current disk-budget cleanup only treats active store entries and compaction checkpoints as referenced. With `session.maintenance.maxDiskBytes` or unreferenced-artifact pruning, an old `<sessionId>.jsonl` can be deleted while `/sessions` still lists it, so switching back lands in an empty or missing transcript. Add history entries to reference accounting, or prune the history entry before its file can be removed.
  - confidence: 0.84
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is checked only after targets are built, so a store that already has `sessionHistory` will still list and switch to those sessions after an operator sets `session.historyLimit` to `0`. In that state the switch path pushes the current session, immediately evicts it, and archives the conversation the user is leaving, which contradicts the documented disable behavior. Return a disabled-history response before listing or resolving targets when the limit is zero.
  - confidence: 0.78
