Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: This switches the live entry to the target session and then restores only the small `SessionHistoryItem.metadata` subset. Session-scoped auth profiles, CLI/ACP handles, execution/elevated policy, fast/usage settings, runtime model state, costs, fallback notices, and abort cutoffs are still read by later turns, so `/sessions back` can continue under the wrong credentials, provider-side thread, or sandbox policy.
  - confidence: 0.88
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:258`
  - body: When `session.historyLimit` is changed to `0` after history exists, this path still builds targets from `sessionHistory` and only applies the limit after selecting a target. That contradicts the documented disable behavior and can archive the current session as a side effect of a switch that should have been rejected.
  - confidence: 0.84
- **[P2] Reference retained transcripts during cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: The PR keeps old transcripts switchable by storing them in `sessionHistory`, but current disk-budget cleanup only marks the active entry transcript and compaction checkpoint files as referenced. A retained `<oldSessionId>.jsonl` can therefore be deleted as unreferenced while `/sessions` still advertises it, causing the switch to reopen an empty or missing transcript.
  - confidence: 0.82
