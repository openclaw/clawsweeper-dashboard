## Summary

Keep open. Current main still lacks the chat `/sessions` and switch-by-id/index/back behavior, so the PR is not obsolete; however, the supplied latest PR diff still has blocking correctness and security issues around partial in-place session-state restoration, disabled-history handling, transcript cleanup, and generated config drift.

## What This Changes

The PR adds chat `/sessions` and `/session <number|sessionId|back>` switching with per-SessionKey LRU `sessionHistory`, retained transcripts, `session.historyLimit` config/docs/schema updates, and targeted tests.

## Best Possible Solution

Keep the feature direction, but make chat session switching a complete session boundary: either persist and restore a full per-session snapshot or switch between separate stored entries by pointer. The final patch should hard-disable list/switch behavior when `session.historyLimit` is `0`, archive/retain history transcripts consistently, protect history artifacts during maintenance cleanup, fix generated docs/config baselines, and cover the security-sensitive state-boundary cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create two direct-chat sessions under one SessionKey with different auth, exec, CLI, ACP, abort, model, fallback, and cost fields, then switch with the new command path; the PR restores only a limited metadata subset, so omitted fields can persist from the outgoing session.

## Solution Assessment

No. The requested feature is valid, but this patch is not the narrowest safe implementation because it mutates one live `SessionEntry` and relies on a partial allowlist of restored fields; a complete snapshot or pointer-based model would be safer and easier to audit.
