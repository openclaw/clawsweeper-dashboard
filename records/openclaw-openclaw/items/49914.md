## Summary

Keep open. Current main still lacks chat-channel session switching, and this PR is the active implementation candidate, but the patch is not mergeable because the new switch path restores only a partial session snapshot and can bleed credentials, execution policy, and provider runtime handles across sessions.

## What This Changes

The PR adds `/sessions` and `/session` chat session switching, per-SessionKey LRU `sessionHistory` retention, `session.historyLimit`, docs/generated config updates, and targeted tests.

## Best Possible Solution

Keep the feature direction, but make session switching a complete state boundary: either persist and restore a full per-session snapshot or store each switchable session as its own entry and switch a pointer. The final patch should also hard-disable list/switch behavior when history is disabled, archive or retain every history transcript consistently, include history artifacts in maintenance cleanup, update public docs/changelog, and cover the state-boundary cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create two direct-chat sessions under one SessionKey with different auth, exec, CLI, ACP, abort, model, fallback, and cost fields, then switch with the new command path; the PR reapplies only a small metadata subset, so omitted fields can persist from the outgoing session.

## Solution Assessment

No. The requested feature is valid, but this patch is not the narrowest safe solution because it mutates a single live `SessionEntry` and relies on a partial allowlist of restored fields; a complete snapshot or pointer-based design would be safer and easier to audit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore complete session-scoped state on switch:** `src/auto-reply/reply/commands-sessions.ts:277-280`
  - body: This switch path mutates the live `SessionEntry` into the target session and then reapplies only the limited history metadata subset. Fields such as auth profile overrides, elevated/exec policy, CLI handles, ACP state, abort state, runtime model fields, fallback notices, and cost totals are neither saved nor cleared, so resumed sessions can inherit credentials, execution policy, or provider-side threads from the outgoing session.
  - confidence: 0.91
- **[P2] Gate switching when session history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `historyLimit` is checked only after the command has resolved targets from persisted history. If an operator changes `session.historyLimit` to `0` while stale history exists, `/sessions` can still expose and switch to old sessions, then trim the current one as the queue is forced to zero.
  - confidence: 0.86
- **[P2] Archive cleared history before dropping it:** `src/auto-reply/reply/session.ts:650-651`
  - body: When history is disabled, this path clears retained `sessionHistory` but only archives the immediately previous session. Older retained history entries are removed from state without going through the reset/archive lifecycle, leaving transcript files orphaned.
  - confidence: 0.84
- **[P2] Keep history transcripts referenced during disk sweeps:** `src/config/sessions/types.ts:257`
  - body: The PR intentionally keeps history transcripts switchable, but the disk-budget sweep still treats only top-level session artifacts as referenced. With `session.maintenance.maxDiskBytes` enabled, a retained history transcript can be removed as unreferenced while `/sessions` still lists it.
  - confidence: 0.82
- **[P2] Fix generated config baseline path count:** `docs/.generated/config-baseline.jsonl:1`
  - body: The JSONL baseline adds one `session.historyLimit` path record, but the metadata line changes `totalPaths` from 5593 to 5631. That makes the generated baseline internally inconsistent and likely to fail config-baseline drift checks.
  - confidence: 0.77
