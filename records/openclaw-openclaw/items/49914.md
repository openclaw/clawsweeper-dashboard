## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restore the complete session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: `applyHistoryMetadata` only reapplies a small metadata subset after rewriting the active `SessionEntry` to the target session. Fields such as auth profile overrides, CLI/ACP handles, exec/elevated policy, runtime model, abort state, fallback notices, usage mode, and cost remain from the outgoing session, so `/sessions back` can resume a transcript under the wrong credentials, provider-side thread, or execution policy.
  - confidence: 0.93
- **[P1] Block stale history when history is disabled:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: The command builds and renders switchable history before checking `session.historyLimit`, and the switch path only applies the limit after selecting a target. If an operator sets the new knob to `0` after history exists, `/sessions` can still expose old sessions and `/sessions <n>` can switch to them, contradicting the documented disable behavior and archiving the current session as an eviction side effect.
  - confidence: 0.9
- **[P2] Protect retained history transcripts from maintenance:** `src/config/sessions/types.ts:257`
  - body: Adding `sessionHistory` makes older transcripts intentionally switchable while they remain in the LRU queue, but the existing disk-budget cleanup only treats active `entry.sessionId` values as referenced. With `session.maintenance.maxDiskBytes` enabled, retained history transcripts can be removed while `/sessions` still lists them, so switching back can land in an empty or missing transcript.
  - confidence: 0.85
