## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Restore the full session-owned state on switch:** `src/auto-reply/reply/commands-sessions.ts:115-141`
  - body: `applyHistoryMetadata` only restores a small whitelist after changing `sessionId`. Current `SessionEntry` state includes auth profile overrides, execution policy, CLI/ACP runtime handles, runtime model/harness, fallback notice, abort, delivery, and plugin-owned fields; leaving those from the previous conversation can resume the target transcript under the wrong credentials, sandbox, provider thread, or runtime state. Rework the switch so it restores or clears the full durable session-owned contract, or avoid mutating one entry across sessions.
  - confidence: 0.88
- **[P2] Block switching when session history is disabled:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: The command resolves targets from persisted `sessionHistory` before enforcing `session.historyLimit`. If an operator changes the limit to `0` while old history entries remain, `/sessions <n>` can still switch to them and then archive the current session through the zero-limit eviction loop, contradicting the new disable knob. Return the disabled-history response before building or resolving switch targets when the effective limit is zero.
  - confidence: 0.82
