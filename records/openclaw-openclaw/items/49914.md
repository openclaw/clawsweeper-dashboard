## Summary

Keep open: this is a meaningful implementation candidate for chat-channel session switching, but the current patch is not merge-ready because it restores only a partial session snapshot, leaves disabled history switchable in one upgrade path, is currently conflicting with main, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `/sessions` chat command, per-session-key LRU session history, `session.historyLimit`, switch/preview handling, generated config schema updates, and focused unit coverage for chat session switching.

## Best Possible Solution

Rebase on current main, align the command contract with the open chat-session issue, and make switching restore or rebind a complete session boundary with upgrade, security, and real-chat proof before merge.

## Reproduction Assessment

Yes for the main review findings from source inspection: the PR switches sessions by mutating one `SessionEntry` and restoring only selected metadata, while current and PR-head code both store additional per-session auth, runtime, execution, and state fields on that same entry. ## Solution Assessment

No. A partial metadata snapshot is not the safest maintainable boundary for chat session switching; the better path is a current-main rebase plus a complete session-boundary restore or a binding/list primitive that keeps per-session state isolated.
