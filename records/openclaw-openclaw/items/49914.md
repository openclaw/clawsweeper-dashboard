## Summary

Keep open: current main still lacks built-in chat-channel session switching, and this PR is the canonical implementation candidate, but the patch has blocking session-state correctness/security issues and lacks attached real behavior proof.

## What This Changes

Adds a `/sessions` chat command, per-SessionKey LRU session history, `session.historyLimit`, tests, and generated config updates for chat-channel session switching.

## Best Possible Solution

Keep the feature work on this PR, but rebase it and make session switching restore an authoritative per-session state record, honor disabled history immediately, protect retained transcripts from cleanup, and document the final command syntax.

## Reproduction Assessment

Yes, from source inspection. The PR resolves switch targets from persisted `sessionHistory` before enforcing `historyLimit`, and the partial metadata snapshot leaves many current-main `SessionEntry` fields behind when switching sessions.

## Solution Assessment

No. The direction is useful, but the current patch is not the narrowest maintainable solution until it treats session switching as a full per-session state transition and updates cleanup/docs/proof around that contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore session-scoped runtime state on switch:** `src/config/sessions/types.ts:81-112`
  - body: `applyHistoryMetadata` only restores the fields listed in `SessionHistoryItemMetadata`, but the switch mutates the existing `SessionEntry` in place. Fields like `authProfileOverride`, `cliSessionIds`/`cliSessionBindings`, `model`/`modelProvider`, `fastMode`, `responseUsage`, `elevatedLevel`, `exec*`, `acp`, `estimatedCostUsd`, and fallback notice state can leak from the outgoing session into the resumed one, so a CLI-backed or auth-profile-specific conversation can continue on the wrong provider thread, credentials, or execution policy. Snapshot/restore or clear the full session-scoped runtime state, or persist switchable sessions as complete entries.
  - confidence: 0.9
- **[P2] Wire the advertised /session switch command:** `src/auto-reply/reply/commands-sessions.ts:18`
  - body: The new handler only accepts `/sessions` and `/sessions ...`, while the PR body and linked feature request promise `/session <number>`, `/session <sessionId>`, and `/session back`. Since `handleSessionCommand` is unchanged and still treats non-`idle`/`max-age` actions as usage errors, users following the advertised command cannot switch sessions. Either wire `/session <target>` into the switch path or align the PR/docs/tests around `/sessions <target>` only.
  - confidence: 0.92
- **[P2] Honor historyLimit=0 before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: If an operator changes `session.historyLimit` from a positive value to `0`, existing `sessionHistory` entries remain in the store until a rollover. This path still builds and displays those entries before checking the configured limit, and the switch path resolves them before applying the limit, so disabled history remains reachable and the current session can be immediately evicted/archived during the switch. Check `historyLimit === 0` before building switch targets or listing retained history.
  - confidence: 0.87
- **[P2] Keep retained history transcripts referenced during cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: This PR intentionally leaves transcripts for `sessionHistory` entries on disk so users can switch back, but the existing disk-budget sweep only marks each active `SessionEntry` transcript and compaction checkpoints as referenced. Under `session.maintenance.maxDiskBytes`, those retained history transcripts can look unreferenced and be deleted while `/sessions` still lists them. Include history transcript paths in the referenced-artifact set before retaining them as switchable.
  - confidence: 0.84
