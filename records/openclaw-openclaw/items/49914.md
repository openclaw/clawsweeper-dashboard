## Summary

Keep open: current main still lacks the requested chat-surface session list/switch command, so this PR is not obsolete, but the patch is not merge-ready because it mutates one SessionEntry while restoring only a partial session-state subset, leaves retained history transcripts outside disk-budget references, does not fully honor disabled history, has no inspectable real behavior proof, and is currently conflicting.

## What This Changes

The PR adds a `/sessions` chat command, per-SessionKey LRU session history, `session.historyLimit`, generated config/schema updates, and tests for listing/switching retained chat sessions.

## Best Possible Solution

Keep this PR as an implementation candidate only if maintainers want this command direction; before merge it should restore or isolate the full session boundary, update retention cleanup and disabled-history semantics, resolve the command-shape decision, rebase cleanly, and include redacted real chat proof.

## Solution Assessment

No. A chat-surface session switcher is plausible, but this implementation is too narrow for the session boundary and should either restore/clear the complete per-session state or use a design that does not mutate one live entry across multiple sessions.

Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:119-141`
  - body: `applyHistoryMetadata` restores only a small metadata subset after the command rewrites the live `SessionEntry` to the target session. Current main stores auth profile, CLI/ACP handles, abort cutoffs, execution/elevated policy, runtime model, fast/usage, fallback notice, and other session-scoped state on that same entry, so `/sessions 2` can resume one transcript while continuing under another session's credentials, provider thread, or tool policy. Restore/clear the complete boundary or use a per-session entry model before switching.
  - confidence: 0.9
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:258-264`
  - body: `session.historyLimit=0` is resolved only after the command has already built targets from persisted `sessionHistory`. If an operator disables history after entries exist, `/sessions` still lists them and `/sessions <n>` can switch to one while immediately evicting/archiving the current session, which contradicts the new disable contract.
  - confidence: 0.86
- **[P2] Protect retained history transcripts from cleanup:** `src/config/sessions/types.ts:257`
  - body: The new `sessionHistory` field keeps prior transcript files switchable, but current disk-budget maintenance only treats each active entry transcript and compaction checkpoint files as referenced. Add the retained history transcript paths to the referenced set, or deployments with `session.maintenance.maxDiskBytes` can delete files that `/sessions` still advertises.
  - confidence: 0.84
