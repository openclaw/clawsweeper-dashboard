## Summary

Keep this PR open as the canonical implementation path for chat-channel session switching, but it is not mergeable yet because the patch still has concrete correctness/security blockers and lacks inspectable real behavior proof.

## What This Changes

The PR adds a core chat `/sessions` command, per-SessionKey LRU session history, `session.historyLimit` config/schema/docs, transcript retention changes, command visibility filtering, and regression tests.

## Best Possible Solution

Define a complete core chat-session switching contract that restores or clears all session-scoped runtime, security, transcript, cleanup, and lifecycle state, then prove the visible chat flow in a real channel before merge.

## Reproduction Assessment

Yes, for review purposes by source inspection: current main lacks the built-in chat session-switching commands, and the PR's remaining failure paths are visible in the proposed handler and metadata snapshot. I did not run tests or a live chat flow because this review is read-only.

## Solution Assessment

No, not as-is. The feature direction matches the canonical request, but the mergeable solution needs complete session-state restore/cleanup semantics, documented command behavior that the implementation honors, disabled-history handling, and real chat proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the full session state on switches:** `src/config/sessions/types.ts:81-112`
  - body: `SessionHistoryItemMetadata` is what gets reapplied after the live entry is rewritten, but it omits session-scoped fields such as auth profile overrides, exec/elevated policy, fast/usage flags, runtime model/provider, fallback/cost/status, CLI handles, system prompt report, and ACP state. Switching can therefore resume one transcript while continuing under another conversation's credentials, provider thread, or execution policy.
  - confidence: 0.9
- **[P2] Honor the advertised singular switch command:** `src/auto-reply/reply/commands-sessions.ts:18`
  - body: The PR body and linked request advertise `/session <number>`, `/session <sessionId>`, and `/session back`, but this handler only matches `/sessions`. Users following the documented singular command still fall through to the existing lifecycle/settings handler instead of switching sessions.
  - confidence: 0.88
- **[P2] Check disabled history before resolving old targets:** `src/auto-reply/reply/commands-sessions.ts:213-258`
  - body: The switch handler builds targets from persisted history before checking `session.historyLimit`. If history is changed to `0` after entries already exist, stale sessions can still be listed or selected, and selecting one evicts the current session under the disabled-history path.
  - confidence: 0.86
- **[P2] Match session ID prefixes before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-221`
  - body: Numeric input is interpreted as a rendered list index before exact or prefix session-id matching. A UUID/session id with a numeric-only prefix such as `1234...` cannot be selected by its documented unique prefix and may even switch to entry 1234 if enough sessions exist.
  - confidence: 0.82
- **[P2] Preserve retained history transcripts during cleanup:** `src/auto-reply/reply/session.ts:691-692`
  - body: The PR intentionally keeps historical transcripts switchable while they remain in `sessionHistory`, but the existing disk-budget cleanup only treats the active `sessionId` as referenced. With `session.maintenance.maxDiskBytes` enforcement, retained transcripts can be removed while `/sessions` still lists them, so switching back can reopen an empty/default transcript.
  - confidence: 0.78
