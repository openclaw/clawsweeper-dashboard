## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore or clear the full session boundary:** `src/auto-reply/reply/commands-sessions.ts:115-141`
  - body: This switch mutates the live `SessionEntry` to the target session but only restores a small metadata subset. Fields such as auth profile overrides, runtime model identity, CLI/ACP handles, exec/elevated policy, fallback notices, costs, abort state, and pending delivery can remain from the outgoing session, so a resumed transcript can run under the wrong credentials, runtime, or security policy. Snapshot/restore the full session boundary or explicitly clear fields that must not carry across.
  - confidence: 0.9
- **[P1] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/commands-sessions.ts:263-264`
  - body: The PR keeps previous transcripts switchable by storing them in `sessionHistory`, but OpenClaw's disk-budget cleanup only counts active `entry.sessionId` transcript paths as referenced. With `session.maintenance.maxDiskBytes` enforcement, a retained history transcript can be deleted while `/sessions` still lists it, making switch-back land in an empty or missing transcript. Add history entries to the cleanup reference set or archive/remove them consistently.
  - confidence: 0.86
- **[P2] Honor disabled history before exposing switch targets:** `src/auto-reply/reply/commands-sessions.ts:258`
  - body: `buildSwitchableSessionList` is used before checking `session.historyLimit`, so stale persisted history remains listable and switchable after an operator sets the limit to `0`. In that state, switching pushes the current session into a zero-size queue and archives it even though switchable history is disabled. Gate list/switch behavior on the resolved limit or clear/archive stale history explicitly first.
  - confidence: 0.88
