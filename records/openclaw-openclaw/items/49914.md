## Review Findings

Overall correctness: not a patch

Overall confidence: 0.92

Full review comments:

- **[P1] Restore or clear the full session boundary:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: The switch path replaces `sessionId` and `sessionFile` and then applies only the small `SessionHistoryItem.metadata` subset. Fields such as auth profile overrides, CLI bindings, fast mode, response usage, execution/elevation settings, abort cutoffs, runtime model/fallback state, cost, system prompt report, skills, and ACP state can remain from the outgoing session, so a resumed transcript can run under the wrong credentials, backend, sandbox, queue, or usage state. Define the full switch boundary by restoring all session-scoped fields or clearing provider/runtime/security state the same way reset does, with regression coverage.
  - confidence: 0.93
- **[P1] Protect retained transcripts from cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: The PR makes old session files user-switchable by storing them only in `sessionHistory`, but the current disk-budget reference accounting only protects the active entry transcript and compaction checkpoints. A retained transcript can therefore be pruned as unreferenced while `/sessions` still advertises it as resumable. Include retained history files in cleanup accounting or remove/archive history entries consistently when their transcript is no longer retained.
  - confidence: 0.9
- **[P2] Honor disabled history before exposing switch targets:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: `handleSessionsCommand` builds the active-plus-history target list before it checks `session.historyLimit`, so persisted history from an earlier configuration can still be listed and switched after an operator disables session history. Check the effective limit before listing or switching retained entries, and cleanly archive or discard old retained entries when history is disabled.
  - confidence: 0.85
