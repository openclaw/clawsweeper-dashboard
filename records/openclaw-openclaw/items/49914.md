## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restore the full session boundary before switching:** `src/auto-reply/reply/commands-sessions.ts:280`
  - body: This switch mutates the live `SessionEntry` to the target session but reapplies only `SessionHistoryItemMetadata`. Current main has additional session-scoped auth profile, CLI/ACP handle, elevated/exec policy, runtime model, fallback, cost, abort, and status fields; after a switch, those can remain from the previous session and run the resumed transcript under the wrong credentials or execution policy.
  - confidence: 0.88
- **[P1] Protect retained history transcripts from cleanup:** `src/auto-reply/reply/session.ts:649`
  - body: The PR intentionally keeps old transcripts unarchived while their sessions remain in `sessionHistory`, but current disk-budget cleanup only marks active store entries and compaction checkpoints as referenced. With `session.maintenance.maxDiskBytes`, cleanup can remove a retained transcript that `/sessions` still lists, so switching back can reopen an empty or missing conversation.
  - confidence: 0.86
- **[P2] Honor disabled history before listing or switching:** `src/auto-reply/reply/commands-sessions.ts:170`
  - body: `/sessions` builds its list from persisted `sessionHistory` before checking `session.historyLimit`. If an operator changes the limit to `0`, stale entries remain visible and switchable until another rollover, contradicting the new disable contract and potentially archiving the current session during the switch path.
  - confidence: 0.82
- **[P3] Match session IDs before numeric indexes:** `src/auto-reply/reply/commands-sessions.ts:219-222`
  - body: Numeric input is treated as a rendered list index before exact or prefix session-id matching. A unique session-id prefix that is all digits, such as the start of a UUID, can never select that session even though the command advertises `sessionId prefix` selection.
  - confidence: 0.78
