## Summary

Keep open: the PR addresses a real doctor false-positive and includes useful live proof, but the patch would make doctor and doctor lint execute configured exec SecretRef providers without a doctor-specific opt-in or maintainer-approved policy.

## What This Changes

The PR resolves `gateway.auth.token` SecretRefs before emitting Gateway auth warnings in both doctor lint/core health checks and interactive doctor health, and adds a resolved SecretRef regression test.

## Best Possible Solution

Land a consolidated doctor fix that resolves env/file Gateway token SecretRefs after token auth is known to need a token, while handling exec SecretRefs through an explicit opt-in or an accurate unverified diagnostic approved by the auth/secrets owners.

## Reproduction Assessment

Yes. Source inspection plus the linked reporter evidence gives a high-confidence path: configure local token auth with a resolving `gateway.auth.token` SecretRef and run doctor; current main warns because the synchronous auth resolver drops SecretRef values.

## Solution Assessment

No. Resolving SecretRefs before warning is the right direction, but this implementation uses the full resolver for exec refs during doctor, which should be gated or explicitly approved before merge.
