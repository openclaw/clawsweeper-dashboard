## Summary

Keep open: the reported doctor warning is a real bug, but this patch is not safe to merge as-is because it makes normal doctor and doctor lint paths execute exec SecretRef providers without an explicit opt-in, conflicting with the repository's documented exec-consent posture.

## What This Changes

The PR changes doctor gateway-auth checks to resolve `gateway.auth.token` through the SecretRef-aware gateway token resolver before warning, and adds an e2e regression case for a resolved env SecretRef.

## Best Possible Solution

Land one doctor gateway-auth fix that covers lint and interactive doctor, resolves env/file SecretRefs by default, and either preserves the exec warning by default or adds an explicit, documented exec opt-in with tests for both modes.

## Reproduction Assessment

Yes. Source inspection on current main shows a SecretRef-backed `gateway.auth.token` is not materialized by `resolveGatewayAuth()`, so the doctor gateway-auth check reaches the configured-SecretRef warning path; I did not run the command because this review is read-only.

## Solution Assessment

No. The patch fixes the false warning for resolving SecretRefs, but unconditional exec resolution is too broad for doctor; the safer solution needs a maintained exec-consent boundary such as env/file-only default behavior or an explicit doctor exec opt-in.
