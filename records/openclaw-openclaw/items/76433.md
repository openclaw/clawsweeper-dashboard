## Summary

Keep open: current main and v2026.6.10 still gate keyed state to bundled or trusted official plugins. The narrower related issues cover official channel provenance/deployment failures, while this remains the broader community Plugin SDK access decision that needs security and product review.

## What This Changes

The issue requests exposing `api.runtime.state.openKeyedStore` to community plugins instead of limiting it to bundled or trusted plugins.

## Best Possible Solution

Define a maintainer-approved keyed-state capability or trust contract for community plugins with explicit opt-in, quotas, docs, parser/runtime tests, and fail-closed defaults.

## Reproduction Assessment

Yes, at source level: current main still rejects workspace and untrusted global plugin calls to `api.runtime.state.openKeyedStore`, and the runtime tests assert that rejection. I did not run tests because this is a read-only review.

## Solution Assessment

Unclear as proposed. The use case is concrete, but lifting the gate outright is not the best fix until maintainers choose the plugin trust, capability, quota, and warning policy.
