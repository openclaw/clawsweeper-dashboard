## Summary

Keep open. Current main still documents and enforces `api.runtime.state.openKeyedStore` as bundled-plugin-only, and exposing durable keyed state to community plugins is a new public Plugin SDK and trust-policy feature rather than an already implemented cleanup item.

## What This Changes

The issue requests lifting or redesigning the bundled-only restriction so community plugins can use `api.runtime.state.openKeyedStore` for durable, namespaced keyed state.

## Best Possible Solution

Define a deliberate public Plugin SDK durable-state contract for third-party plugins, including policy, quotas, compatibility expectations, docs, and regression tests, before relaxing the bundled-only guard.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence behavior check because current main rejects workspace-origin plugins and a runtime test asserts that rejection.

## Solution Assessment

Unclear until maintainer design. Removing the origin check is mechanically small, but durable persisted state for unbundled in-process plugins is a public SDK and trust-boundary decision that should be designed and documented explicitly.
