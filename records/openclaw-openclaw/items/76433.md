## Summary

Keep open: current main and v2026.6.6 still restrict `openKeyedStore` to bundled or trusted official plugins, the linked implementation PRs closed unmerged, and the remaining request is a public plugin SDK/security policy decision rather than stale cleanup.

## What This Changes

Request to expose `api.runtime.state.openKeyedStore` to community plugins, potentially through a manifest opt-in with stricter limits or startup warnings.

## Best Possible Solution

Define a maintainer-approved plugin-state capability or trust contract that preserves per-plugin isolation, quotas, documentation, parser coverage, and tests instead of unconditionally granting durable state access.

## Reproduction Assessment

Yes, at source level: a workspace or untrusted global plugin calling `api.runtime.state.openKeyedStore` still hits the registry gate and throws, and current tests assert that rejection. I did not run a live community plugin because this was a read-only review.

## Solution Assessment

Unclear as proposed: the use case is strong, but simply lifting the gate is not the best fix until maintainers choose the plugin trust, manifest capability, quota, and warning policy. A narrower explicit capability or provenance contract is safer than unconditional access.
