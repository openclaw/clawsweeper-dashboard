## Summary

Keep open: current main and latest release still gate keyed plugin state to bundled or trusted official installs, while the related open issues cover narrower official-plugin/container provenance gaps rather than broad community-plugin SDK access.

## What This Changes

The issue asks OpenClaw to expose `api.runtime.state.openKeyedStore` to community plugins, potentially through a manifest opt-in, quotas, and startup warning.

## Best Possible Solution

Define a maintainer-approved keyed-state capability or trust contract for community plugins, with explicit opt-in, quotas, documentation, parser coverage, and safe defaults.

## Reproduction Assessment

Yes, source-level: a workspace or untrusted global plugin calling `api.runtime.state.openKeyedStore` still hits the registry gate and current tests assert that rejection. I did not run a live community plugin in this read-only review.

## Solution Assessment

Unclear as proposed: the use case is strong, but lifting the gate outright is not the best fix until maintainers choose the plugin trust, capability, quota, and warning policy.
