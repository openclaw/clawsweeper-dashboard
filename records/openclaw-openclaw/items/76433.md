## Summary

Keep open: current main and v2026.6.10 still restrict keyed plugin state to bundled or trusted official installs, while the remaining request is a broader public Plugin SDK trust/capability decision for community plugins rather than a stale duplicate of the narrower official-channel deployment issues.

## What This Changes

Request to expose `api.runtime.state.openKeyedStore` to community plugins for restart-safe, plugin-isolated SQLite keyed state.

## Best Possible Solution

Define a maintainer-approved keyed-state capability or trust contract for community plugins with explicit opt-in, quotas, docs, parser/runtime tests, and fail-closed defaults.

## Reproduction Assessment

Yes, at source level. A workspace or untrusted global plugin calling `api.runtime.state.openKeyedStore` still hits the registry trust gate, and current runtime tests assert that rejection; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as proposed. The use case is concrete, but lifting the gate outright is not the best fix until maintainers choose the plugin trust, capability, quota, and warning policy.
