## Summary

Keep open. Current main intentionally keeps `api.runtime.state.openKeyedStore` behind a bundled-plugin origin check, and exposing durable keyed state to community plugins is a new Plugin SDK and trust-policy feature rather than an obsolete or external-only request.

## What This Changes

Expose the existing SQLite-backed `api.runtime.state.openKeyedStore` helper to community plugins, optionally through manifest opt-in, tighter limits, or startup warnings.

## Best Possible Solution

Add a deliberate public Plugin SDK contract for third-party durable keyed state, with documented trust policy, quotas, compatibility expectations, and regression tests; keep the current bundled-only guard until that design is accepted.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence behavior check because current main rejects workspace-origin plugins and a runtime test asserts that rejection.

## Solution Assessment

Unclear until maintainer design. Directly removing the origin check is mechanically small, but durable state for unbundled in-process plugins is a public SDK and trust boundary, so the maintainable solution needs explicit policy, docs, and tests.
