## Summary

Keep open: current main and v2026.6.6 still deny `openKeyedStore` to arbitrary community plugins, the two linked implementation PRs closed unmerged, and the newer open reports are narrower official-plugin deployment/provenance failures rather than a replacement for this plugin SDK policy decision.

## What This Changes

The issue asks OpenClaw to expose `api.runtime.state.openKeyedStore` to community plugins instead of limiting it to bundled or trusted official plugins.

## Best Possible Solution

Define a maintainer-approved plugin-state capability or trust contract that preserves per-plugin isolation, quotas, docs, parser coverage, and tests instead of unconditionally granting every community plugin durable state access.

## Reproduction Assessment

Yes at source level: a workspace or untrusted global plugin that calls `api.runtime.state.openKeyedStore` still hits the registry gate and throws, while current tests assert that rejection. I did not run a live community plugin because this was a read-only review.

## Solution Assessment

Unclear as proposed: the use case is strong, but simply lifting the gate is not the best fix until maintainers choose the plugin trust, manifest capability, quota, and warning policy. A narrower explicit capability/provenance contract is safer than unconditional access.
