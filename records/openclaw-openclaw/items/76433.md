## Summary

Keep open: current main and v2026.6.10 still gate `api.runtime.state.openKeyedStore` to bundled or trusted official installs, and the related open issues cover narrower official-channel provenance failures rather than this broader community-plugin SDK access request.

## What This Changes

The issue requests exposing `api.runtime.state.openKeyedStore` to community plugins instead of limiting it to bundled or trusted plugins.

## Best Possible Solution

Define a maintainer-approved plugin state capability contract that preserves safe defaults and documents the trust, quota, and migration expectations before lifting the runtime gate.

## Reproduction Assessment

Yes for the current limitation: source and tests show untrusted workspace/global plugins still throw when they call `api.runtime.state.openKeyedStore`. This is a requested SDK capability expansion rather than a broken existing contract.

## Solution Assessment

Unclear as an implementation path: lifting the gate directly is plausible but not necessarily the best fix until maintainers choose the trust and capability model for community plugins.
