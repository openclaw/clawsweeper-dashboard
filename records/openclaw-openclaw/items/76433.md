## Summary

Keep open. Current main still intentionally rejects `openKeyedStore` for non-bundled plugins, and the linked open PR #76508 is the implementation candidate, so this issue should wait for maintainer SDK/security review rather than being closed early.

## What This Changes

Expose `api.runtime.state.openKeyedStore` to community plugins, potentially through manifest opt-in and stricter limits.

## Best Possible Solution

Define and land a deliberate third-party durable-state SDK contract with policy, quotas, docs, and tests, then close this issue when #76508 or a successor merges.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence behavior check because current main rejects workspace-origin plugins and a runtime test asserts that rejection.

## Solution Assessment

Unclear until maintainer design. Directly removing the origin check is mechanically small, but manifest-gated durable state with quotas, docs, and tests is the safer maintainable direction for a public SDK surface.
