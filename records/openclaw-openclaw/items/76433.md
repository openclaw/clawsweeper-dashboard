## Label Justifications

- `P2`: This is a normal-priority plugin SDK feature request with clear source-level evidence but a bounded product/API blast radius.
- `impact:session-state`: The requested API is durable keyed state for restart-safe plugin deduplication, cursors, and recovery history.
- `impact:security`: Allowing community plugins to open host-managed durable state changes the plugin trust boundary.

## Summary

Keep open: current main and the latest release still restrict keyed plugin state to bundled or trusted official installs, while the issue asks for a broader community-plugin SDK capability. The linked community-access PRs are closed unmerged, and the newer Docker/channel reports are related trust-gate symptoms but do not replace this broader API/product decision.

## What This Changes

Expose `api.runtime.state.openKeyedStore` for community plugins so they do not need to reimplement restart-safe SQLite keyed state.

## Best Possible Solution

Define a maintainer-approved plugin-state access contract, likely a narrow manifest or provenance-based opt-in with quotas, tests, and docs, while preserving per-plugin isolation and the existing official-plugin trust boundary.

## Reproduction Assessment

Yes at source level: a workspace or untrusted global plugin record deterministically throws when calling `api.runtime.state.openKeyedStore`. The current tests also lock that rejection behavior, so a live community-plugin run is not needed to confirm the missing capability.

## Solution Assessment

Unclear as an implementation choice: direct gate removal is not enough because it changes the durable-state security boundary. The best path needs maintainer approval for manifest opt-in, trusted install provenance, operator trust, or another explicit policy before code changes.
