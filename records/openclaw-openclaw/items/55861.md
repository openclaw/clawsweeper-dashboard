## Summary

Keep open: the PR targets a real, still-relevant Control UI layout bug, but the current patch should not merge as-is because it widens a shared rail variable used by tablet drawer layouts and its regression test measures during a 200ms transition. The external PR also still lacks after-fix Safari proof.

## What This Changes

The PR widens the collapsed Control UI nav rail from 78px to 96px and adds a browser geometry test asserting content does not render under the collapsed nav rail.

## Best Possible Solution

Land or replace this with a desktop-scoped Safari-safe Control UI layout fix, settled browser geometry coverage, and redacted Safari proof while preserving tablet and mobile drawer behavior.

## Reproduction Assessment

No live high-confidence current-main reproduction was established in this read-only review. The linked issue provides concrete Safari/macOS steps and screenshots, and current main still exposes the collapsed-sidebar source path, so the issue is source-reproducible for triage.

## Solution Assessment

No, not as written. The maintainable fix should scope the wider rail to desktop or otherwise preserve tablet drawer behavior, and the geometry test should measure the final settled collapsed layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Scope the wider rail to desktop layouts:** `ui/src/styles/layout.css:9`
  - body: Changing the base `--shell-nav-rail-width` to 96px also changes the tablet collapsed drawer because `layout.mobile.css` reuses this variable under `max-width: 1100px`. Scope the 96px value to desktop, or override the tablet drawer back, so a Safari desktop fix does not widen the fixed overlay rail on tablet.
  - confidence: 0.93

  - body: The test compares rectangles after one `nextFrame()`, but `.shell` grid columns and `.shell-nav` width transition for 200ms. Wait for or disable the transition before measuring so the assertion covers the final collapsed layout.
  - confidence: 0.78
