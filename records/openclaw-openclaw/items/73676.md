Keep open. Current main and v2026.6.10 still contain unguarded `process.cwd()` calls on the reported CLI/TUI startup paths, and open closing PRs already own candidate fixes.

## What This Changes

The issue reports that launching `openclaw tui` from a deleted current working directory crashes with raw `ENOENT: uv_cwd` instead of a clear fallback or error.

## Best Possible Solution

Land one caller-specific nullable-cwd fix that covers CLI dotenv, gateway-dispatch/runtime dotenv, PATH/home fallback, and TUI runtime/local-shell paths, then close this issue with merge and release provenance.

## Reproduction Assessment

Yes, source-level. The issue gives concrete deleted-cwd steps, and current main still reaches bare `process.cwd()` calls on startup and TUI paths; I did not run a live deleted-cwd command in this read-only review.

## Solution Assessment

Yes for the issue direction. The maintainable fix is caller-specific nullable cwd handling rather than one generic HOME/temp fallback, because dotenv, PATH, required-home, and TUI operator-command paths have different trust semantics.
