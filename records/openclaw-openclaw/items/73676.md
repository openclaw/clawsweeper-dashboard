## Summary

Keep open. Current main and the latest release still have unguarded startup `process.cwd()` calls that can throw `ENOENT uv_cwd`, and the linked fixing PR is open rather than merged.

## What This Changes

The issue reports that OpenClaw CLI startup can crash with `ENOENT: no such file or directory, uv_cwd` when launched from a deleted current working directory.

## Best Possible Solution

Land or replace the linked fix with one canonical deleted-cwd guard for startup cwd resolution, dotenv loading, PATH bootstrap, and TUI startup, plus focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: delete the invoking cwd, then run a non-help CLI command so startup reaches direct `process.cwd()` calls in dotenv/home-dir handling. I did not run the repro because this review is read-only.

## Solution Assessment

Yes, the linked PR direction is the right shape: centralize safe cwd handling and route startup callers through it. The issue should remain open until that fix is merged or replaced by an equivalent current-main change.
