## Summary

Keep open. Current main and the latest release still have unguarded deleted-cwd `process.cwd()` paths in CLI startup and TUI runtime, and two open closing PRs already own candidate fixes.

## What This Changes

The issue asks OpenClaw CLI/TUI startup to handle a deleted current working directory instead of crashing with `ENOENT: uv_cwd`.

## Best Possible Solution

Land one caller-specific nullable-cwd fix that covers CLI dotenv, gateway-dispatch dotenv, runtime dotenv, PATH/home fallback, and TUI runtime paths, then close this issue with merge and release provenance.

## Reproduction Assessment

Yes, source-level. A deleted-cwd Node probe reproduced `ENOENT`/`uv_cwd`, and current main still reaches direct `process.cwd()` calls on the reported CLI/TUI paths.

## Solution Assessment

Yes for the issue direction. The maintainable fix is caller-specific nullable cwd handling rather than one generic HOME or temp fallback, because dotenv, PATH, required-home, and TUI spawn paths have different trust semantics.
