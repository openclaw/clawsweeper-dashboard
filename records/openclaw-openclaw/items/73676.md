## Summary

Keep open. Current main still contains unguarded CLI/TUI startup `process.cwd()` paths that can throw `ENOENT`/`uv_cwd`, and the linked closing implementation remains an open, unmerged pull request.

## What This Changes

The issue requests that CLI/TUI startup handle a deleted current working directory with a clear error or safe fallback instead of crashing with `uv_cwd`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74994, or an equivalent shared safe-cwd startup fix, with regression coverage for the early CLI/TUI cwd consumers.

## Reproduction Assessment

Yes, source-backed. The reporter's create/cd/delete/run path maps directly to unguarded `process.cwd()` calls in current-main CLI/TUI startup helpers, which can throw `ENOENT`/`uv_cwd` when the cwd has been removed.

## Solution Assessment

Yes as a direction. A shared guarded cwd helper across startup, dotenv, home-dir fallback, PATH bootstrap, and TUI workspace inference is the narrow maintainable fix compared with patching only one call site.
