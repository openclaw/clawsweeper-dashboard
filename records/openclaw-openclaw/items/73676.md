## Summary

Keep open. Current main and the latest release still have unguarded `process.cwd()` calls in CLI startup/dotenv paths, and the closed replacement attempt is no longer an open fix path.

## What This Changes

The issue reports that running `openclaw tui` from a deleted current working directory crashes with `ENOENT: uv_cwd` instead of handling the missing cwd.

## Best Possible Solution

Add one canonical deleted-cwd startup helper and route CLI dotenv, runtime dotenv, home-dir fallback, PATH bootstrap, and TUI cwd inference through it without using shared temp storage as a project-local cwd base.

## Reproduction Assessment

Yes, source-level. Delete the invoking cwd and run a non-help CLI command such as `openclaw tui`; current main reaches direct startup `process.cwd()` calls before the command can recover.

## Solution Assessment

Yes, the maintainable fix direction is a central deleted-cwd guard across startup cwd resolution. The prior branch's `os.tmpdir()` fallback should not be reused for project-local PATH discovery.
