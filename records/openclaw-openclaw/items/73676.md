## Summary

Keep open. Current main and the latest release still have unguarded startup `process.cwd()` call sites that can throw `ENOENT uv_cwd`, and the linked fixing PR is still open rather than merged.

## What This Changes

The issue asks OpenClaw CLI startup to tolerate a deleted current working directory instead of crashing with `ENOENT uv_cwd`.

## Best Possible Solution

Land PR https://github.com/openclaw/openclaw/pull/74994 or an equivalent focused fix that centralizes deleted-cwd handling across startup dotenv, home-dir, path bootstrap, and TUI paths with regression coverage.

## Reproduction Assessment

Yes. Node reproduces `ENOENT uv_cwd` after cwd deletion, and current main still calls `process.cwd()` unguarded in startup paths that run before `openclaw tui` can proceed.

## Solution Assessment

Yes. A shared safe-cwd helper or equivalent centralized guard across the startup call sites is the narrow maintainable fix; closing this issue before the linked PR merges would leave the reported crash present.
