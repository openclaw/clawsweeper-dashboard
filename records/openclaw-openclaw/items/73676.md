## Summary

Keep open. Current main still has unguarded `process.cwd()` calls on early CLI startup, dotenv, and home-dir fallback paths, and PR #74994 is an open linked implementation candidate rather than a merged fix.

## What This Changes

Handle deleted current working directories during CLI startup so `openclaw tui` reports or falls back cleanly instead of throwing `ENOENT: uv_cwd`.

## Best Possible Solution

Review and land PR #74994 or an equivalent shared safe-cwd startup fix with targeted regression coverage across main-module detection, CLI/runtime dotenv loading, and required home-dir fallbacks.

## Reproduction Assessment

Yes. The reporter's create/cd/delete/run path maps directly to current-main `process.cwd()` calls that execute during CLI startup and are known to throw `ENOENT: uv_cwd` when the process cwd has been removed.

## Solution Assessment

Yes, as a direction. A shared guarded cwd helper across the early startup consumers is the narrowest maintainable fix; only wrapping `shouldLoadCliDotEnv` would leave `isMainModule`, dotenv loaders, and home-dir fallbacks exposed.
