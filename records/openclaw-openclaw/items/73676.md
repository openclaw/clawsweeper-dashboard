## Summary

Keep open. Current main still has unguarded `process.cwd()` calls in early CLI startup, dotenv loading, and required home-dir fallbacks, and #74994 is an open closing implementation candidate rather than a merged fix.

## What This Changes

Handle a deleted current working directory during CLI startup so `openclaw tui` does not abort with `ENOENT: uv_cwd`.

## Best Possible Solution

Review and land #74994 or an equivalent shared safe-cwd startup fix with focused regression coverage for main-module detection, CLI/runtime dotenv loading, and required home-dir fallbacks.

## Reproduction Assessment

Yes. The reporter's create/cd/delete/run path maps directly to current-main startup code where `process.cwd()` is evaluated before or during CLI boot, which is the reported `ENOENT: uv_cwd` failure mode.

## Solution Assessment

Yes, as a direction. A shared guarded cwd helper across the early startup consumers is the narrowest maintainable fix; only wrapping `shouldLoadCliDotEnv` would leave `isMainModule`, dotenv loaders, and home-dir fallbacks exposed.
