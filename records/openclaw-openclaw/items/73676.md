## Summary

Keep open. Current main still has unguarded `process.cwd()` calls on the reported CLI startup path, and the issue is paired with open closing PR #74994, so it should close only after that PR or an equivalent fix lands.

## What This Changes

Handle deleted current working directories during CLI startup so `openclaw tui` reports or recovers cleanly instead of crashing with `ENOENT: no such file or directory, uv_cwd`.

## Best Possible Solution

Land PR #74994 or an equivalent shared safe-cwd startup fix that covers main-module detection, CLI/runtime dotenv loading, and required home-dir fallbacks with targeted deleted-cwd regression tests; then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. The reporter's create/cd/delete/run path maps directly to current main evaluating raw `process.cwd()` during entrypoint detection and CLI dotenv startup before command handling.

## Solution Assessment

Yes. A shared safe-cwd helper across the startup cwd callers is the narrowest maintainable direction because patching only one caller would leave other raw-cwd startup paths able to throw the same `uv_cwd` error.
