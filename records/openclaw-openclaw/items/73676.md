## Summary

Keep open. Current main still contains early CLI/TUI startup paths that call `process.cwd()` directly, and the explicit closing implementation remains open at https://github.com/openclaw/openclaw/pull/74994.

## What This Changes

The issue asks OpenClaw CLI startup, especially `openclaw tui`, to handle a deleted current working directory without surfacing an unhandled `ENOENT: uv_cwd` crash.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74994, or an equivalent shared safe-cwd startup fix, with regression coverage for the early CLI/TUI cwd consumers.

## Reproduction Assessment

Yes, source-backed. A deleted cwd makes Node `process.cwd()` throw `ENOENT uv_cwd`, and the reporter's `openclaw tui` path maps to raw `process.cwd()` calls still present in current-main CLI/TUI startup code.

## Solution Assessment

Yes as a direction. A shared guarded cwd helper across startup, dotenv, home-dir fallback, PATH bootstrap, and TUI workspace inference is the narrow maintainable fix compared with patching only one call site.
