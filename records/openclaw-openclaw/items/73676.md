## Summary

Keep open. Current main still has unguarded startup `process.cwd()` calls on the reported CLI startup path, and PR #74994 is an open, unmerged implementation candidate for the same issue. Repository policy says this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that OpenClaw CLI startup tolerate a deleted current working directory instead of crashing with `ENOENT: no such file or directory, uv_cwd`.

## Best Possible Solution

Review and land PR #74994 or an equivalent shared safe-cwd startup fix that covers main-module detection, CLI/runtime dotenv loading, and required home-dir fallbacks with targeted deleted-cwd tests; then close the issue after the fix reaches main.

## Reproduction Assessment

Yes. The reporter's create/cd/delete/run path maps to current main calling `process.cwd()` during entrypoint main-module detection and CLI dotenv probing before command handling can recover.

## Solution Assessment

Yes. A shared safe-cwd helper covering the startup call sites is the narrow maintainable direction; patching only one dotenv caller would leave other raw-cwd startup paths able to throw the same `uv_cwd` error.
