## Summary

Keep open. Current main still has unguarded startup `process.cwd()` calls on the reported path, and the issue is paired with open closing PR #74994 rather than a merged fix on main.

## What This Changes

Handle a deleted current working directory during CLI startup so commands such as `openclaw tui` do not crash with `ENOENT: no such file or directory, uv_cwd`.

## Best Possible Solution

Review and land PR #74994, or replace it with an equivalent shared safe-cwd startup fix that covers main-module detection, CLI/runtime dotenv loading, and home-dir fallback behavior with targeted deleted-cwd tests.

## Reproduction Assessment

Yes. The reporter's create/cd/delete/run sequence maps directly to current startup code that evaluates `process.cwd()` before command-specific recovery.

## Solution Assessment

Yes. A shared safe-cwd helper routed through all startup call sites is the narrow maintainable direction; guarding only one caller would leave other raw-cwd startup crashes in place.
