## Summary

Keep open. Current main still has unguarded early CLI/TUI `process.cwd()` calls that can throw `ENOENT uv_cwd`, and the issue is already paired with an open closing PR that should be reviewed or landed instead of closing this early.

## What This Changes

Handle deleted current working directories during CLI/TUI startup so raw `process.cwd()` `uv_cwd` errors do not crash OpenClaw.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74994, or an equivalent shared safe-cwd startup fix, with regression coverage for the early CLI/TUI cwd consumers.

## Reproduction Assessment

Yes, source-backed. A deleted cwd makes Node `process.cwd()` throw `ENOENT uv_cwd`, and current main's `openclaw tui` startup path still reaches direct `process.cwd()` calls before the intended fallback behavior exists.

## Solution Assessment

Yes as a direction. A shared guarded cwd helper across startup, dotenv, home-dir fallback, PATH bootstrap, and TUI workspace inference is narrower and more maintainable than patching only one call site.
