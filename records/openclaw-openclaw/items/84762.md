## Summary

Keep this PR open: it addresses the LiteLLM cron normalization bug in the right area, but the provider guard is too broad and currently breaks existing nested Google model normalization for OpenRouter/proxy refs; CI reproduces that regression in the model picker tests.

## What This Changes

The PR changes `src/config/model-input.ts` so config model refs only run Google preview alias normalization when the provider is `google`, `google-gemini-cli`, or `google-vertex`.

## Best Possible Solution

Keep the provider-scoped fix, but preserve nested `google/...` suffix normalization for proxy providers and add regression coverage for both `litellm/gemini-3-flash` and `openrouter/google/gemini-3-pro-preview`.

## Reproduction Assessment

Yes for source-level reproduction: current main shows the LiteLLM suffix can be Google-preview-normalized in config, and the PR CI reproduces the new OpenRouter nested-ref regression in `src/commands/model-picker.test.ts`. I did not run a live cron force-run in this read-only review.

## Solution Assessment

No: the proposed provider-only guard is not the best complete fix because it also disables intentional nested `google/...` normalization for proxy providers. The maintainable path is to skip bare non-Google Gemini suffixes while keeping explicit nested Google refs normalized.
