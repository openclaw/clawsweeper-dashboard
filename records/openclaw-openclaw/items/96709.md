## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid retrying already-sent batches:** `extensions/discord/src/outbound-payload.ts:116-126`
  - body: This wraps one `sendMessageDiscord` call in a delivery retry, but that call can now perform multiple Discord POSTs when `mediaUrls` has more than 10 entries. If a later batch gets a retryable 429/5xx after an earlier batch succeeded, the retry reruns from the beginning and duplicates the earlier attachments; move retry to each batch or make the overflow send idempotent.
  - confidence: 0.88
- **[P2] Wire mediaUrls through Discord actions:** `extensions/discord/src/send.outbound.ts:41`
  - body: The low-level option is added here, but the public Discord send action/runtime still read only scalar `mediaUrl`/`path`/`filePath` and pass only `mediaUrl` into `sendMessageDiscord`. The shared message-action runner already populates `mediaUrls`, so user-facing multi-attachment sends still collapse to the first file unless the Discord action path parses and forwards the array with tests.
  - confidence: 0.86
- **[P2] Preserve reply fanout across overflow batches:** `extensions/discord/src/send.shared.ts:495-496`
  - body: `sendDiscordMediaBatch` sends each overflow batch as a separate Discord message but passes the same `replyTo` into every batch. That bypasses the existing per-platform-send reply fanout behavior, so `replyToMode: first` can make all overflow messages reply to the original instead of only the first; apply the reply reference only to the intended batch or resolve it per platform send.
  - confidence: 0.78
