## Summary

Keep open. Current main still lacks a normal route-binding workspace override, and the linked implementation PRs are closed unmerged; this needs maintainer product/security approval rather than conservative cleanup.

## What This Changes

The issue requests an optional `bindings[].workspace` override so one agent can use different workspaces for different conversations.

## Best Possible Solution

If accepted, add a typed, documented normal-route `bindings[].workspace` override with explicit trust-boundary rules, propagation through route resolution and representative channel/outbound paths, stale-override clearing, schema/docs updates, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection gives high confidence the capability is absent because the strict route-binding schema rejects the key and runtime workspace resolution is still agent-scoped.

## Solution Assessment

Unclear until maintainer approval. The proposed precedence is plausible, but the maintainable solution needs explicit security/product decisions for workspace trust boundaries, persistence across interactions, and stale override clearing.
