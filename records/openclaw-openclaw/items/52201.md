## Summary

Keep open. Current main still does not implement binding-level workspace overrides for normal route bindings: the config type and strict schema exclude the requested field, resolved routes cannot carry a workspace payload, and reply/runtime setup still resolves the workspace from the matched agent rather than the matched binding. The prior implementation attempts are useful context but are closed unmerged, so there is no implemented or superseding fix to close against.

## What This Changes

The issue requests an optional `workspace` field on normal route bindings so one agent can use different workspaces for different conversations, with binding workspace taking precedence over agent workspace and defaults.

## Best Possible Solution

If maintainers accept the product shape, add a typed and documented `bindings[].workspace` field for normal route bindings, validate and trim it, thread it through resolved routes and inbound contexts, apply it consistently to bootstrap, memory, tool cwd, plugin loading, sandbox/media, and outbound interaction continuity, and clear it on all reroute paths.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The absence is high-confidence from source inspection: current schemas reject the proposed key, resolved routes cannot carry it, and workspace resolution remains agent-scoped.

## Solution Assessment

Unclear until maintainer approval. The proposed precedence order directly addresses the reported context bleed, but the safest implementation needs explicit decisions for workspace trust boundaries, outbound persistence, and reroute clearing before automation should change it.
