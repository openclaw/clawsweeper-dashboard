## Summary

Keep open. Current main still does not support a normal route-binding `workspace` override, and the related implementation PRs are closed unmerged; the remaining decision is a maintainer-approved product/security shape rather than conservative cleanup.

## What This Changes

Request an optional `bindings[].workspace` field so the same agent can use different workspaces for different conversations.

## Best Possible Solution

If accepted, implement a typed and documented normal-route `bindings[].workspace` override with trim/empty validation, route propagation, runtime workspace application, interaction-continuity handling, stale-override clearing, and regression coverage across routing plus representative channel/outbound paths.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. Source inspection gives high confidence the feature is absent because the strict route-binding schema rejects the key and runtime workspace resolution remains agent-scoped.

## Solution Assessment

Unclear until maintainer approval. The requested precedence is plausible, but the best implementation needs explicit product and security decisions around workspace trust boundaries and cross-channel persistence.
