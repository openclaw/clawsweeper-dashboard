## What This Changes

The issue requests cron execution observability through per-attempt logs, skipped/failure warnings, cron health reporting, and `openclaw cron list` annotations.

## Best Possible Solution

Define the cron observability contract, then add bounded structured outcome logs, cron-aware probe/readiness reporting, and operator-facing list warnings on top of the existing counters, diagnostics, alerts, and run history.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers define the health/readiness contract. The logging and list-warning pieces are narrow, but the probe/API shape should be coordinated before implementation.
