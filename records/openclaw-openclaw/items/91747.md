## Summary

Keep open: the PR is directionally right for the raw `/json/list` fallback, but it is not merge-ready because the sibling raw `/json/new` fallback still exposes an unchecked discovered `webSocketDebuggerUrl`, and the external PR currently has only harness-test proof.

## What This Changes

The PR adds CDP endpoint-policy validation for selectable per-tab WebSocket URLs returned by the browser plugin's raw `/json/list` fallback and adds focused fallback-path regression coverage.

## Best Possible Solution

Validate every normalized CDP WebSocket discovered from raw CDP HTTP responses before exposing it as `BrowserTab.wsUrl`, with coverage for both `/json/list` and `/json/new` and explicit maintainer acceptance of the strict-mode fail-closed behavior.

## Reproduction Assessment

Yes at source level: a raw `/json/new` response can return `webSocketDebuggerUrl: ws://169.254.169.254/...`, and current `openTab()` would normalize and expose it as `BrowserTab.wsUrl` without `assertCdpEndpointAllowed()`. I did not run tests because this review was read-only.

## Solution Assessment

No. Validating `/json/list` is the right direction, but the best fix applies the same policy check to all raw CDP-discovered WebSocket URLs before downstream routes can open them.
