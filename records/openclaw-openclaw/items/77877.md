## Summary

Keep open: the PR is a plausible gateway-side fix with sufficient real proof, but the latest head still has a fallback-path bug that can broadcast dead managed document links after transcript append failure, plus a smaller HTTP route-gating issue.

## What This Changes

The PR adds a managed outgoing document attachment module, HTTP route, chat promotion/cleanup integration, regression tests, and a changelog entry for serving assistant-generated documents as downloads.

## Best Possible Solution

Land a rebased gateway document download path after it strips unservable document links from append-failure fallbacks, gates the new HTTP stage by route prefix, and stays coordinated with the companion UI/media delivery PRs.

## Reproduction Assessment

Yes. Current main docs promise managed non-video attachment history, while source inspection shows the chat path only creates managed image blocks and the image module skips non-image MIME types.

## Solution Assessment

No. The sibling document route is a maintainable direction, but the current PR head should also strip document managed links on append failure and avoid loading the document route for unrelated HTTP paths.

- **[P2] Strip managed document links from fallback events:** `src/gateway/server-methods/chat.ts:460-468`
  - body: This adds document attachment blocks to assistantContent, but the append-failure fallback still strips only managed image blocks.   - confidence: 0.88

That means unrelated Control UI/API paths will import and run the managed document module before the Control UI stage; add a document route predicate and only register this stage for /api/chat/media/outgoing-doc/ paths.
  - confidence: 0.82
