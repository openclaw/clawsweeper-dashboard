## Summary

This is a credible WeCom delivery-loss bug, but the implicated `monitor.js` and queue code live in the external `WecomTeam/wecom-openclaw-plugin` package rather than in the OpenClaw source tree. OpenClaw currently only registers and pins that external package, and the same disabled-queue root cause is already tracked upstream, so this repository item should close as not actionable here.

## What This Changes

The issue reports concurrent WeCom single-chat messages from the same user causing both generated replies to remain stuck without `finish=true` delivery.

## Best Possible Solution

Keep the runtime fix in `WecomTeam/wecom-openclaw-plugin`, then update OpenClaw’s official external catalog only after a fixed WeCom package is published and integrity-pinned.
