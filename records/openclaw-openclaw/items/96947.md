## Summary

Keep open. The report matches current and v2026.6.10 source paths: OpenRouter dynamic Anthropic models still do not set the compat flag that lets `cacheRetention` reach the OpenAI-completions conversation cache-control path, while the OpenRouter post-wrapper only marks system/developer blocks, so there is no current-main fix or safer canonical duplicate to close against.

## What This Changes

The issue reports that after upgrading to v2026.6.10, OpenRouter-routed Anthropic Sonnet sessions with `cacheRetention: "long"` stopped writing conversation prompt-cache checkpoints and only cached the system prompt.

## Best Possible Solution

Confirm the outgoing current-main OpenRouter Anthropic payload and usage counters, then restore provider-owned conversation-level cache behavior with focused regression coverage and live OpenRouter cache proof.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a credible current-main reproduction path because OpenRouter Anthropic routes keep the system-marker wrapper but do not pass cache retention into the conversation-marker transport path without additional compat/capability proof.

## Solution Assessment

Unclear. The desired behavior is established by docs and prior telemetry, but maintainers should choose the repair after live OpenRouter payload and usage proof because the correct fix may be marker placement, cache-retention eligibility, or upstream affinity semantics.
