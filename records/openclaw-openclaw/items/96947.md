## Summary

Keep open. The report matches a current source-level gap: OpenRouter Anthropic config can still add the static system cache marker while `cacheRetention` is not propagated into the OpenAI-compatible transport path that would mark the moving conversation breakpoint.

## What This Changes

The issue reports a v2026.6.10 OpenRouter Anthropic cache regression where `cacheWrite` stays at zero and only the system-prompt prefix is cached.

## Best Possible Solution

Fix the OpenRouter Anthropic cache-retention propagation so the conversation breakpoint is marked again, then add focused request-payload coverage plus live OpenRouter cacheRead/cacheWrite proof.

## Reproduction Assessment

Yes, source-reproducible: current code can leave OpenRouter Anthropic `cacheRetention` out of the underlying `openai-completions` transport while still adding only the wrapper's system marker. I did not run a live OpenRouter session, so provider-side cache accounting still needs real proof.

## Solution Assessment

Unclear whether the best fix is to propagate `cacheRetention` into the existing transport path or to make the OpenRouter wrapper own conversation markers too. The end state should avoid a second competing cache-control path and prove both request payload shape and live usage metrics.
