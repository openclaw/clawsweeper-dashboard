## Label Justifications

- `P1`: The report describes a broken provider/auth routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is auth-profile and provider selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main and the latest release still encode a source-backed path where explicit OpenAI PI routing can prefer `openai-codex` auth/profile order ahead of the normal `openai` route, matching the reported API-key model shadowing. The related open items cover Pro-model availability and picker clarity, but they do not fully own this exact mixed OpenAI API-key plus Codex OAuth coexistence bug.

## What This Changes

The issue asks OpenClaw to keep `openai/gpt-5.5-pro` reachable through the OpenAI API-key route when an `openai-codex` OAuth profile is also configured and `agentRuntime.id: "pi"` is pinned.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI refs or explicitly selected normal `openai` API-key profiles prefer the OpenAI API path, while intentional PI plus selected `openai-codex` compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` before `openai` for explicit OpenAI PI routing, and the runner consumes that order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may break the documented PI plus Codex OAuth compatibility route. A capability-, auth-profile-, or model-aware routing rule is the safer maintainable direction.
