## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed OpenAI/Codex auth precedence path that can select `openai-codex` ahead of `openai` for explicit PI OpenAI routes, matching the reported API-key model shadowing. The related open issues cover Pro-model availability and picker clarity, but this coexistence routing bug is narrower and not fully owned by either related item.

## What This Changes

Route `openai/gpt-5.5-pro` and other OpenAI API-key models through the OpenAI API-key path when the `openai` provider and PI runtime are explicitly selected, even when an `openai-codex` OAuth profile is also configured.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI model refs or explicitly selected normal `openai` API-key profiles prefer the `openai` API path, while intentional PI plus selected `openai-codex` compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` before `openai` for explicit OpenAI PI routing, and the embedded runner consumes that provider order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may conflict with the currently tested PI plus Codex OAuth compatibility route. A model-, profile-, or capability-aware precedence rule is the safer maintainable direction.
