## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed auth-provider path where explicit OpenAI PI routing can prefer `openai-codex` OAuth over `openai` API-key credentials, and the existing tests intentionally preserve that compatibility behavior rather than distinguishing API-only OpenAI model refs.

## What This Changes

The issue reports that paid OpenAI API-only models can be routed through an `openai-codex` OAuth profile despite `agentRuntime.id: "pi"` being pinned to the `openai` provider.

## Best Possible Solution

Define a narrow routing contract so API-only `openai/*` model refs with explicit API-key intent prefer `openai` auth, while subscription-compatible OpenAI routes can still intentionally use Codex OAuth when configured.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` before `openai` for explicit OpenAI PI routing and the embedded runner consumes that order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may break the documented PI plus Codex OAuth compatibility route. A model- or capability-aware routing rule is the safer maintainable direction.
