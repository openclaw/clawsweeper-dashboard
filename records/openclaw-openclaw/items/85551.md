## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed path where an explicit OpenAI PI route can choose an `openai-codex` OAuth profile ahead of the OpenAI API-key route, matching the reported auth/provider shadowing. The related open issues cover Pro-model availability and picker clarity, but neither fully owns this exact coexistence precedence bug.

## What This Changes

The issue requests fixing OpenAI model/auth routing so `openai/*` models pinned to the PI runtime can use an OpenAI API-key route instead of being shadowed by an `openai-codex` OAuth profile.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI model refs or explicitly selected normal OpenAI API-key profiles prefer the OpenAI API path, while intentional PI plus Codex OAuth compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can select `openai-codex` auth for an explicit OpenAI PI route, and the runner consumes that profile order for request auth. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may break the documented PI plus Codex OAuth compatibility route. A model-, capability-, or auth-profile-aware routing rule is the safer maintainable direction.
