## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed path where an explicit OpenAI PI selection can prefer `openai-codex` auth/profile routing over `openai`, and the code does not distinguish API-only OpenAI model refs such as `gpt-5.5-pro` or `gpt-5.4-nano` from subscription-compatible routes.

## What This Changes

The issue asks OpenClaw to keep `openai/gpt-5.5-pro` with provider/model `agentRuntime.id: "pi"` on the OpenAI API-key route when an `openai-codex` OAuth profile is also present.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI model refs with explicit provider/model PI policy prefer `openai` API-key auth, while intentional PI plus Codex OAuth compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` before `openai` for explicit OpenAI PI routing and the runner consumes that order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may break the documented PI plus Codex OAuth compatibility route. A capability- or model-aware routing rule is the safer maintainable direction.
