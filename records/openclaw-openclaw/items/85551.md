## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed path where an explicit OpenAI PI route can consume Codex OAuth auth order ahead of the normal OpenAI API-key provider, matching the reported model shadowing. The related Pro-model availability and picker clarity issues are useful context, but neither fully owns this exact auth-precedence bug.

## What This Changes

The issue asks OpenClaw to keep `openai/gpt-5.5-pro` on the OpenAI API-key path when `agentRuntime.id: "pi"` is pinned, instead of routing through an `openai-codex` OAuth profile.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI model refs or explicitly selected normal OpenAI API-key profiles prefer the OpenAI API path, while intentional PI plus selected `openai-codex` compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` before `openai` for explicit OpenAI PI routing and the runner consumes that order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: making `agentRuntime.id: "pi"` an unconditional hard override would address the API-only model case but may break the documented PI plus Codex OAuth compatibility route. A capability-, auth-profile-, or model-aware routing rule is the safer maintainable direction.
