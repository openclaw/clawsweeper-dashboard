## Label Justifications

- `P1`: The report describes a broken auth/provider routing path that can block paid OpenAI API-only models for users who also keep Codex OAuth configured.
- `impact:auth-provider`: The affected behavior is provider/auth-profile selection between `openai` API-key credentials and `openai-codex` OAuth credentials.

## Summary

Keep open: current main still has a source-backed route where explicit OpenAI PI runs can prefer an `openai-codex` auth profile over the normal `openai` API-key path, matching the reported coexistence failure. Related open issues cover Pro-model availability and picker clarity, but neither fully owns this exact auth/provider precedence bug.

## What This Changes

The issue asks OpenClaw to keep `openai/gpt-5.5-pro` on the OpenAI API-key route when Codex OAuth is also configured and the model is pinned to PI, with related picker labeling clarity.

## Best Possible Solution

Define a narrow OpenAI/Codex routing contract so API-only OpenAI model refs or explicitly selected normal `openai` API-key profiles prefer the OpenAI API path, while intentional PI plus selected `openai-codex` compatibility remains supported for subscription-capable models.

## Reproduction Assessment

Yes at source level: current main can put `openai-codex` ahead of `openai` for explicit OpenAI PI routing, and the embedded runner consumes that order for auth selection. I did not establish a live failing request with real OpenAI/Codex credentials.

## Solution Assessment

Unclear: treating `agentRuntime.id: "pi"` as an unconditional hard override would address the API-only model case but could break the documented PI plus Codex OAuth compatibility route. A model-, capability-, or explicit-auth-profile-aware rule is the safer maintainable direction.
