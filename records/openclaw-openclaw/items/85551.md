- `impact:auth-provider`: The reported failure is specifically about OpenAI versus OpenAI Codex provider selection, OAuth profile choice, and API-key routing.

## Summary

Keep open: current main has a source-visible OpenAI PI auth-routing path that can prefer an eligible `openai-codex` OAuth profile over OpenAI API-key auth, but that behavior partly matches the recently merged Codex-order routing contract, so changing it needs maintainer direction rather than cleanup closure.

## What This Changes

The issue asks OpenAI API-only models pinned to `agentRuntime.id: "pi"` to route through OpenAI API-key auth instead of an installed `openai-codex` OAuth profile, with clearer picker grouping for the two auth routes.

## Best Possible Solution

Keep both routes explicit: preserve intentional Codex-auth PI compatibility, but make API-key intent unambiguous for API-only OpenAI models through `openai:default`/`auth.order.openai` or a clearly documented provider `auth: "api-key"` rule, with status/model-picker diagnostics showing which route will actually run.

## Reproduction Assessment

Yes, source-level: current main can include `openai-codex` OAuth profiles in the PI auth candidate set and can switch an OpenAI PI run to the `openai-codex` runtime provider before API-key fallback. I did not run a live mixed OAuth plus API-key account reproduction.

## Solution Assessment

Unclear: the requested hard PI override is not clearly the safest fix because current docs and https://github.com/openclaw/openclaw/pull/82605 intentionally support Codex-auth PI compatibility. The maintainable path is to decide and document when API-key intent beats eligible Codex auth, then add focused routing/status tests.
