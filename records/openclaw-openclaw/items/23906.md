## Summary

Keep open. Current `main` still does not expose a per-spawn tool selector or tool-less mode through `sessions_spawn`; the lower-level embedded runner has `disableTools`/`toolsAllow` primitives, but the public spawn schema, native spawn params, and gateway `agent` RPC do not carry that policy, and the related plugin/MCP precedence questions remain unsettled.

## What This Changes

Request a backward-compatible `sessions_spawn` tool override so spawned subagents can run with no tools or a chosen per-run tool subset.

## Best Possible Solution

Define one canonical additive per-spawn tool-policy API shared with #15032, carry it through the spawn schema, native params, gateway `agent` RPC, docs, and tests, then map tool-less runs to existing `disableTools` and subsets to `toolsAllow` with clear plugin/MCP precedence.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence proof that current `main` lacks the requested API path because the public schema, native spawn params, and closed gateway RPC schema carry no per-spawn tool policy.

## Solution Assessment

No current implementation is present. The maintainable direction is an additive API/protocol field that reuses the existing embedded `disableTools`/`toolsAllow` primitives after maintainers settle naming and precedence.
