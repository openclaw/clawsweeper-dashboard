## Summary

Keep open. Current main still does not expose a per-spawn tool selector on `sessions_spawn`; the lower-level embedded runner has `disableTools` and `toolsAllow`, but the public spawn tool, native spawn params, gateway `agent` RPC, and agent ingress path do not carry that policy. The issue discussion also has a recent maintainer review explicitly keeping this open because plugin/MCP tool resolution and API precedence still need a coordinated decision with related reports.

## What This Changes

Issue requests an optional `sessions_spawn` tool-policy override so each spawned subagent can receive all tools, a selected subset, or no tools.

## Best Possible Solution

Keep this open and converge on one additive per-spawn tool-policy contract across the related issues. A complete implementation should add the public `sessions_spawn` field, validate and document it, carry it through `SpawnSubagentParams`, the gateway `agent` RPC, and agent ingress, map tool-less runs to embedded `disableTools`, map subsets to `toolsAllow`, and define plugin/MCP tool-name resolution plus precedence against existing policy layers.

## Reproduction Assessment

Yes. Source-level inspection on current main gives a high-confidence reproduction of the missing capability: there is no accepted or forwarded per-spawn tool-policy field on the public schema, native spawn params, gateway RPC, or ingress path. The local-model crash itself was not live-reproduced, but the API gap that causes the requested workflow to be impossible is clear.

## Solution Assessment

Unclear whether the exact `tools: "none" | string[]` shape is the best final API, because the same decision must cover security isolation, plugin/MCP tool names, and existing policy precedence. Static `tools.subagents.tools` config is not an adequate substitute for this request because it is not per spawn and cannot express one-off tool-less local-model runs.
