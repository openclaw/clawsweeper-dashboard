## Summary

Keep open: current main still lacks a per-spawn tool selector or tool-less mode on `sessions_spawn`, and the recent maintainer review plus related issues show the remaining work needs a coordinated API/security decision rather than cleanup closure.

## What This Changes

Add a backward-compatible per-spawn tool policy override to `sessions_spawn`, including tool-less and allowlist modes for local or restricted subagents.

## Best Possible Solution

Converge on one additive per-spawn tool-policy contract, then carry it through the public tool schema, native spawn params, gateway RPC, agent ingress, embedded-run fields, docs, and regression tests.

## Reproduction Assessment

Yes for the missing capability: current-main source inspection shows no accepted or forwarded per-spawn tool-policy field on the public schema, native spawn params, gateway RPC, or agent ingress path. The reported local Qwen crash itself was not live-reproduced, but the API gap that makes tool-less spawns impossible is clear.

## Solution Assessment

Unclear whether the exact `tools: "none" | string[]` shape is the best final API. The maintainable solution should reuse the lower-level `disableTools`/`toolsAllow` primitives while defining security precedence and plugin/MCP name resolution before implementation.
