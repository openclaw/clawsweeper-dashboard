## Summary

Keep open. Current main still lacks a built-in memory embedding provider registration step on the cold `MemoryIndexManager` provider path, so this PR is not obsolete, but the submitted diff needs revision before merge because it forks the memory-host SDK bridge from the package-owned source and omits the required changelog entry; related #73769 is still an open overlapping implementation candidate.

## What This Changes

This PR lazily registers built-in memory embedding providers before `MemoryIndexManager` creates an embedding provider, adds an `ensureBuiltInMemoryEmbeddingProvidersRegistered` helper, and rewrites the source memory-host SDK `engine-embeddings` bridge.

## Best Possible Solution

Land one canonical cold-registry fix for local memory embeddings, keep memory-host SDK bridges package-owned, add focused regression coverage for the accepted entrypoint, and include a `CHANGELOG.md` fix entry crediting the reporter/contributor.

## Reproduction Assessment

Yes. The linked issue gives a concrete installed-package reproduction using `agents.defaults.memorySearch.provider = "local"` and `openclaw memory status --deep`, and current main still shows the manager path can reach `createEmbeddingProvider` without first seeding built-ins; I did not execute the live path in this read-only pass.

## Solution Assessment

No as submitted. Manager-level registration is a plausible narrow fix, but the bridge rewrite should be removed and the branch should add changelog/test coverage while maintainers reconcile the overlap with #73769.
