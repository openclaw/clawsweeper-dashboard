## Summary

Keep open. Current main still lacks the cold-registry registration before the memory manager creates an embedding provider, so the PR is not obsolete; however, the submitted patch needs maintainer-directed revision because it rewrites the package-owned source bridge, lacks the required changelog entry for a user-facing fix, and overlaps with #73769’s CLI-scoped registration/test approach.

## What This Changes

This PR adds lazy built-in memory embedding provider registration before `MemoryIndexManager` creates a provider and expands the source memory-host SDK embeddings bridge.

## Best Possible Solution

Land one canonical cold-registry fix for local memory embeddings, preserve the package-owned source bridge as a thin re-export, add targeted regression coverage for the accepted entry point, and include a `CHANGELOG.md` fix entry crediting the contributor.

## Reproduction Assessment

Yes, source-backed. The linked issue gives a concrete installed-package path using `openclaw memory status --deep` with `memorySearch.provider = "local"`, and current main still has a manager path that can call `createEmbeddingProvider` before built-ins are registered.

## Solution Assessment

Unclear as submitted. Registering before provider creation is the right class of fix, but the source bridge rewrite should be removed and maintainers should reconcile this manager-level approach with #73769’s CLI-focused patch and tests.
