## Label Justifications

- `P1`: The deterministic bug silently changes real cron fire times for existing users without an error or warning.
- `impact:data-loss`: Editing the expression deletes persisted timezone and stagger metadata from the job schedule.

## Summary

Current `main` and v2026.6.5 both replace the full schedule object when `--cron` is supplied, so the report remains valid. The clean, open pull request https://github.com/openclaw/openclaw/pull/92295 explicitly owns the fix; keep this issue open until that PR is merged or superseded by another landed fix.

## What This Changes

Preserve an existing cron job’s `tz` and `staggerMs` when editing only its cron expression, while allowing explicit timezone or stagger overrides.

## Best Possible Solution

Land one focused CLI fix that applies the established existing-cron merge semantics when `--cron` changes the expression, preserving omitted metadata and testing both preservation and explicit overrides.

## Reproduction Assessment

Yes at high confidence from source inspection: current `main` constructs a partial cron schedule from the current flags and replaces the stored schedule wholesale. The reporter also provides a concrete v2026.6.5 before/after CLI sequence, but this read-only review did not execute it.

## Solution Assessment

Yes. Reusing the existing cron metadata merge semantics at the CLI edit boundary is narrower and safer than changing generic service patch behavior; the linked PR is the active implementation candidate.
