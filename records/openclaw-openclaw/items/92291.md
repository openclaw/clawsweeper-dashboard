## Label Justifications

- `P1`: A routine cron edit can silently shift production firing times for timezone-pinned jobs, and the reporter found multiple affected jobs.
- `impact:data-loss`: The command silently discards persisted timezone and stagger schedule metadata.

## Summary

This is a valid released bug that remains present on current main; keep the issue open while the linked implementation at https://github.com/openclaw/openclaw/pull/92295 is reviewed and either landed or revised.

## What This Changes

Preserve an existing cron job’s timezone and stagger window when `cron edit --cron` changes only its expression.

## Best Possible Solution

Route cron-expression edits through one canonical cron-to-cron merge operation that preserves all omitted schedule metadata, honors explicit timezone and stagger replacements, and is covered at the CLI-to-RPC boundary.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.5 construct a replacement cron schedule from only the current flags and assign it wholesale, matching the supplied CLI reproduction.

## Solution Assessment

Yes in intended behavior, but the best implementation is a canonical cron schedule merge rather than duplicating preservation logic for the two currently known optional fields.
