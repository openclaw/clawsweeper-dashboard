## Label Justifications

- `P1`: An ordinary edit can silently move production cron execution to another timezone, and the reporter found multiple affected jobs.
- `impact:data-loss`: The CLI silently deletes persisted `tz` and `staggerMs` schedule metadata from existing jobs.

## Summary

Current `main` and v2026.6.5 still send a partial cron schedule as a full replacement when `--cron` is used, so omitted timezone and stagger metadata are lost. The report is valid, but it must remain open because the clean, focused closing PR at https://github.com/openclaw/openclaw/pull/92295 has not merged.

## What This Changes

Preserve an existing cron job’s `schedule.tz` and `schedule.staggerMs` when `cron edit --cron` changes only its expression.

## Best Possible Solution

Land a focused canonical same-kind cron merge that preserves omitted metadata, honors explicit `--tz` and `--stagger` overrides, and verifies both the `cron.update` payload and a real CLI round trip.

## Reproduction Assessment

Yes at source level. Current `main` deterministically resolves `--cron` into a schedule containing only current flags and sends it as a full replacement, matching the reported v2026.6.5 round trip.

## Solution Assessment

Yes at the behavioral level: omitted fields should survive a same-kind cron edit and explicit flags should win. The narrowest maintainable implementation should generalize or reuse the existing cron merge helper rather than maintain parallel preservation rules.
