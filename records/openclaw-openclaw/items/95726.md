## Summary

Keep open: this PR targets a real shipped memory migration regression, but it is not merge-ready because the branch does not compile and the migration can replace the existing per-agent database instead of importing into it.

## What This Changes

The PR adds a core legacy-state migration that detects `$OPENCLAW_STATE_DIR/memory/main.sqlite`, copies it over the per-agent SQLite file, rewrites legacy memory tables, and archives the legacy file.

## Best Possible Solution

Implement a doctor-owned sidecar import that attaches/imports legacy memory rows into the existing per-agent database, preserves or rebuilds cache, FTS, and vector rows, keeps non-memory agent tables intact, and includes upgrade proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.9 route built-in memory to the per-agent SQLite database while the linked issue provides field evidence of a populated legacy sidecar and an empty active store after upgrade.

## Solution Assessment

No. The correct fix should import legacy memory rows into the existing per-agent database through a migration owner, not replace that database or leave cache/vector preservation unproven.

## Review Findings

Full review comments:

- **[P1] Make the migration compile before merge:** `src/infra/state-migrations.ts:4477-4480`
  - body: The new migration calls `requireNodeSqlite().Database.open` and later calls bare `now()`, but the repo's SQLite wrapper exposes `DatabaseSync` and the clock is only available as `params.now`. CI is already failing prod/test type checks on these lines, so this branch cannot build as written.
  - confidence: 0.95
- **[P1] Do not replace the existing agent database:** `src/infra/state-migrations.ts:4544-4551`
  - body: When the target DB exists but has zero memory chunks, this code renames the whole per-agent database and copies the legacy memory DB over it. The per-agent DB also contains auth profile and cache tables, so a normal fresh or empty-memory agent can lose active auth/cache state during doctor migration.
  - confidence: 0.93
- **[P1] Preserve cache and vector rows before archiving:** `src/infra/state-migrations.ts:4602-4619`
  - body: The migration only copies legacy `meta`, `files`, and `chunks`; it does not migrate the embedding cache, FTS table, or sqlite-vec rows. Current search uses the vector table when available and sync skips unchanged file hashes, so users can still lose semantic recall or cache state after the legacy store is archived.
  - confidence: 0.86
