## Summary

Keep open: current main already normalizes message-tool targets enough to handle raw Discord IDs versus `channel:<id>`, but the cron agent-turn path still does not suppress automatic source replies when an explicit announce delivery target is set, so the duplicate-delivery report remains actionable.

## What This Changes

The issue asks cron `agentTurn` deliveries with an explicit Discord `delivery.to` target to produce exactly one visible Discord message when `visibleReplies` is automatic.

## Best Possible Solution

Make explicit cron announce delivery choose one visible delivery path by suppressing automatic source replies for that turn while preserving normalized message-tool matching and existing failure-destination behavior.

## Reproduction Assessment

Yes, at source level: a custom-session cron `agentTurn` with explicit announce delivery reaches the embedded runner without a `sourceReplyDeliveryMode` override, while current visible-reply policy can still select automatic delivery. I did not live-run the Discord setup.

## Solution Assessment

Yes, the narrow maintainable fix is to suppress automatic source delivery only for cron turns with an explicit fallback announce target, then rely on the existing normalized message-tool target matcher to avoid fallback duplicates when the agent sends directly.

Fix the duplicate cron Discord delivery bug for `agentTurn` jobs with explicit announce delivery targets. Keep the boundary narrow: when a cron run has an explicit `delivery.channel`/`delivery.to` fallback announce target, suppress automatic source-reply delivery for that embedded agent turn so the final reply is delivered once by either the message tool or cron fallback, not both. Preserve existing normalized message-tool target matching for raw Discord IDs versus `channel:<id>`, and do not add a new config option or change failure-alert behavior unless a focused test proves the existing `failureDestination` contract is broken.
