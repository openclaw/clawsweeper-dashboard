## Summary

Keep open: current main and v2026.6.10 still keep `web_fetch` proxy routing as an explicit opt-in with no matching operator diagnostic, and the open candidate PR should be reviewed or closed before this issue is resolved.

## What This Changes

Request auto-enabling, or at least warning about, `tools.web.fetch.useTrustedEnvProxy` when the Gateway process has HTTP(S) proxy environment variables set.

## Best Possible Solution

Review and either land the diagnostic-only candidate PR or choose another explicit security-reviewed product direction while keeping proxy routing opt-in by default.

## Reproduction Assessment

Yes for the current behavior from source and tests: `HTTP_PROXY` does not enable `EnvHttpProxyAgent` unless `useTrustedEnvProxy` is true, and no matching doctor/status/audit warning exists on current main. I did not run a live blocked-egress network reproduction in this read-only review.

## Solution Assessment

No for unconditional auto-enable as stated; the warning-only candidate is the safer maintainable direction because it preserves the explicit SSRF/DNS-pinning opt-in while improving operator feedback.
