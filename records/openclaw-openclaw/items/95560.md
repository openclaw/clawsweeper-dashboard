## Label Justifications

- `P2`: The issue affects a real but bounded web_fetch proxy workflow and needs product/security scoping rather than urgent outage handling.
- `impact:security`: Auto-enabling trusted env proxy can change the SSRF and DNS-pinning boundary for web_fetch.
- `impact:other`: The report also affects operator network routing and discoverability for users behind corporate or regional proxies.

## Summary

Keep open: current main and the latest release still keep web_fetch trusted-env proxy as an explicit opt-in with no startup warning, while auto-enabling it from HTTP_PROXY/HTTPS_PROXY is a security-sensitive product choice because NO_PROXY and SSRF behavior are still under active review.

## What This Changes

Request to auto-enable or warn about web_fetch trusted-env proxy when Gateway sees HTTP_PROXY/HTTPS_PROXY but tools.web.fetch.useTrustedEnvProxy is unset.

## Best Possible Solution

Keep explicit web_fetch proxy routing off by default until the NO_PROXY security policy is settled, and consider a conservative startup or doctor diagnostic that points users to the documented opt-in.

## Reproduction Assessment

Yes for the current behavior: source and tests show web_fetch remains direct when HTTP_PROXY is present unless useTrustedEnvProxy is explicitly true. I did not run a live blocked-network reproduction in this read-only review.

## Solution Assessment

No for unconditional auto-enable as stated; the safer path is a maintainer-approved warning or doctor diagnostic while keeping proxy routing explicit until NO_PROXY security behavior is decided.
