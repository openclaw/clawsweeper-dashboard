## Summary

Keep open: current main and v2026.6.9 still keep `web_fetch` trusted-env proxy routing as an explicit opt-in with no startup warning, and changing that default or adding a diagnostic is a security-sensitive product decision rather than a narrow bug cleanup.

## What This Changes

Request to make `web_fetch` notice `HTTP_PROXY`/`HTTPS_PROXY` by auto-enabling `tools.web.fetch.useTrustedEnvProxy` or warning when proxy env vars are present but the opt-in is disabled.

## Best Possible Solution

Keep `web_fetch` proxy routing explicit by default; if maintainers approve the UX direction, add a focused startup or doctor diagnostic that points operators to the documented opt-in without changing the security default.

## Reproduction Assessment

Yes for the current behavior: set `HTTP_PROXY` and leave `tools.web.fetch.useTrustedEnvProxy` unset; current source and tests show `web_fetch` stays on the DNS-pinned direct path. I did not run a live blocked-network reproduction in this read-only review.

## Solution Assessment

No for unconditional auto-enable as stated; the safer maintainable path is a maintainer-approved diagnostic while preserving explicit proxy routing until the NO_PROXY/security policy is settled.
