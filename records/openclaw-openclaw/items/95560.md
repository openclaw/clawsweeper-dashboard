## Summary

Keep open: current main and v2026.6.9 still keep `web_fetch` trusted-env proxy as an explicit opt-in and I found no startup diagnostic, while auto-enabling it from ambient proxy env remains a security/product default decision tied to the open NO_PROXY policy work.

## What This Changes

Request to auto-enable, or at least warn about, `tools.web.fetch.useTrustedEnvProxy` when the Gateway process has HTTP(S) proxy environment variables.

## Best Possible Solution

Keep trusted env proxy routing explicit by default until the NO_PROXY/security policy is settled, then add a maintainer-approved startup or doctor diagnostic that points users to the documented opt-in.

## Reproduction Assessment

Yes for the current behavior: source and tests show `web_fetch` remains direct when `HTTP_PROXY` is present unless `useTrustedEnvProxy` is explicitly true. I did not run a live blocked-network reproduction in this read-only review.

## Solution Assessment

No for unconditional auto-enable as stated; the safer path is a maintainer-approved warning or doctor diagnostic while keeping proxy routing explicit until NO_PROXY and SSRF policy are decided.
