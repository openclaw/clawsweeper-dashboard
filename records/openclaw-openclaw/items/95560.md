## Summary

Keep open: current main still keeps `web_fetch` proxy routing as an explicit opt-in and has no operator warning when HTTP(S)_PROXY is present, while the open linked PR is the active candidate for the safer diagnostic-only path.

## What This Changes

The issue requests auto-enabling `tools.web.fetch.useTrustedEnvProxy` when HTTP(S)_PROXY is present, or at minimum warning operators that `web_fetch` will ignore those proxy env vars without the opt-in.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/96172 as the diagnostic-only fix, while keeping proxy routing explicit and leaving broader NO_PROXY/proxy policy to the canonical related items.

## Reproduction Assessment

Yes, source-level. Current main tests and runtime code show `web_fetch` stays direct when HTTP_PROXY is set unless `useTrustedEnvProxy` is explicitly true; I did not run a live blocked-network reproduction.

## Solution Assessment

No for unconditional auto-enable because it changes the security boundary. The warning-only approach in the linked PR is the safer maintainable direction if maintainers accept the diagnostic UX.
