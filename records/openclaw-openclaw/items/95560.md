## Summary

Keep open: current main and v2026.6.10 still keep `web_fetch` trusted-env proxy as an explicit opt-in with no built-in proxy-env diagnostic, and https://github.com/openclaw/openclaw/pull/96172 is an open closing PR for the lower-risk warning path. This issue should close only after that PR lands or maintainers choose a different security/product direction.

## What This Changes

The issue requests automatically enabling, or at least warning about, `tools.web.fetch.useTrustedEnvProxy` when the Gateway process has HTTP(S) proxy environment variables set.

## Best Possible Solution

Keep proxy routing explicit by default and resolve this issue through the diagnostic-only warning path in the open candidate PR if maintainers accept that operator UX.

## Reproduction Assessment

Yes. Source and focused tests show current main keeps `web_fetch` on the direct pinned-DNS path when `HTTP_PROXY` is set unless `useTrustedEnvProxy` is explicitly true; I did not run a live blocked-network reproduction in this read-only review.

## Solution Assessment

No for unconditional auto-enable because it changes the security boundary by default. The diagnostic-only approach in https://github.com/openclaw/openclaw/pull/96172 is the safer candidate if maintainers accept the UX.
