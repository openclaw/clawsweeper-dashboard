## Summary

Keep open: this is a focused Discord bug fix with useful real-behavior proof, but the latest patch still has a blocking queued-follow-up policy mismatch that can keep verbose progress hidden in a configured preview-suppression path.

## What This Changes

The PR carries Discord tool-progress payload kind through reply delivery, relaxes message-tool-only progress suppression in verbose mode, and forwards queued follow-up progress events with new regression tests.

## Best Possible Solution

Keep the PR direction, but make queued follow-up progress use the same verbose-overrides-preview-suppression policy as direct dispatch and add the missing queued follow-up regression case.

## Reproduction Assessment

Yes, with source-level confidence: current main strips or suppresses explicit Discord progress on the direct and queued paths described by the linked issue, and the PR body supplies live Discord canary output for the intended fix. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No. The overall fix direction is narrow and maintainable, but the queued follow-up gate should share the direct dispatch suppression helper semantics so verbose mode still overrides preview suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve verbose progress when preview suppression is set:** `src/auto-reply/reply/followup-runner.ts:486-487`
  - body: `suppressDefaultToolProgressMessages` is set by Discord draft/progress preview, but existing dispatch policy only suppresses default progress when verbose is not enabled. This gate makes queued follow-ups drop all forwarded progress whenever that option is true, so a verbose Discord follow-up can still look silent in the preview-suppression path. Please match the direct dispatch policy and add the queued follow-up case with `suppressDefaultToolProgressMessages: true` plus `verboseLevel: "on"`.
