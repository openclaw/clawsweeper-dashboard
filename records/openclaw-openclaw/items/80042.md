## Summary

Keep open: this PR is a focused bug fix with sufficient real Discord proof, but the latest head still blocks queued follow-up progress whenever Discord progress-draft suppression is active, even when verbose progress is enabled. That leaves the queued/progress path source-reproducibly silent and should be fixed before merge.

## What This Changes

The PR passes Discord delivery kind through the sanitizer, forwards queued follow-up tool/progress events, carries tool progress detail into follow-up runs, and adds Discord/auto-reply regression tests.

## Best Possible Solution

Keep the PR direction, but make queued follow-up gates share the direct dispatch verbose-overrides-preview-suppression contract and verify the Discord progress-draft path.

## Reproduction Assessment

Yes, source-level: current main lacks queued follow-up progress forwarding, and the PR body supplies live Discord direct/queued canary output for the intended fix. The remaining blocker is source-reproducible with a queued follow-up using suppressDefaultToolProgressMessages=true and verboseLevel="on".

## Solution Assessment

No, not yet. The sanitizer and queued-forwarding approach is narrow, but queued follow-up progress gates need to reuse direct dispatch semantics instead of treating preview suppression as a hard block.
