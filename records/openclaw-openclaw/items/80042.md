## Summary

Keep open: the PR has a plausible direction and sufficient live Discord proof, but the latest head still uses the default-message suppression flag to block queued Discord progress callbacks, which can leave draft-stream queued follow-ups silent.

## What This Changes

The PR carries Discord reply kind into delivery sanitization, preserves tool-progress payloads, forwards queued follow-up progress/tool-result events, refreshes a Swift gateway model field, and adds regression tests.

## Best Possible Solution

Keep the PR direction, but make queued follow-up progress use a distinct callback-forwarding gate so Discord draft/verbose progress remains visible without weakening final-reply privacy or Telegram tool-progress suppression.

## Reproduction Assessment

Yes, source-level. Current main strips Discord tool-looking payloads at the delivery sanitizer and queued follow-ups only account for compaction, while the PR body supplies redacted live Discord output for the after-fix direct and queued canaries.

## Solution Assessment

No for the latest head. The approach is narrow, but queued callback forwarding must be split from default-message suppression and covered by a queued regression with `suppressDefaultToolProgressMessages: true`.
