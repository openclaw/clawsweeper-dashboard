## Summary

Keep open: the PR is a plausible fix with supplied live Discord proof, but the latest head still drops queued follow-up progress whenever Discord preview suppression is enabled, even when verbose mode should override that suppression.

## What This Changes

The PR carries Discord tool-progress payload kind through reply delivery, relaxes message-tool-only progress suppression, forwards queued follow-up progress events, and adds regression tests.

## Best Possible Solution

Keep the PR direction, but make queued follow-up progress use the same verbose-overrides-preview-suppression policy as direct dispatch and add the missing queued follow-up regression case.

## Reproduction Assessment

Yes, with source-level confidence: current main and the PR diff show the direct and queued Discord progress paths, and the linked report plus PR body describe real Discord verbose-mode canaries. I did not run a live Discord reproduction in this read-only pass.

## Solution Assessment

No. The overall fix direction is narrow, but the queued follow-up gate should share the direct dispatch suppression semantics so verbose mode still overrides preview suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve verbose progress under preview suppression:** `src/auto-reply/reply/followup-runner.ts:488-490`
  - body: `suppressDefaultToolProgressMessages` is set by Discord draft/progress preview, but direct dispatch only suppresses default progress when verbose mode is not enabled. This condition drops all queued follow-up progress whenever preview suppression is true, so a verbose Discord follow-up can still look silent. Please allow `verboseLevel !== "off"` to override this suppression and add the queued follow-up case with `suppressDefaultToolProgressMessages: true`.
