## Summary

Keep open for human review: the patch adds a direct browser Web Speech microphone path that conflicts with the current Talk/server-side audio boundary, still lacks browser proof of microphone recognition and draft updates, and adds new unlocalized composer labels.

## What This Changes

The branch adds a conditional Web Speech API dictation button to the Control UI chat composer, streams recognition results into the draft, resets dictation on cleanup, and adds chat view tests.

## Best Possible Solution

Decide the intended browser dictation and privacy boundary, then either implement dictation through Gateway transcription/Talk relay or explicitly approve and document browser Web Speech behavior with localized labels and real browser proof.

Not applicable: this is a feature PR rather than a bug report. Current-main source and tests show no browser Voice input button, while the PR diff adds one.

No. Direct browser SpeechRecognition is not the best maintainable path without product/privacy approval because current docs and changelog favor Gateway-mediated audio, and the new control labels bypass the locale surface.
