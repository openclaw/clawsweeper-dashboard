## Summary

Keep open, but this PR is not merge-ready: it adds a real Control UI dictation feature, yet it reintroduces direct browser Web Speech against the documented Gateway-mediated audio boundary, hardcodes UI labels, and the supplied proof only shows bundled asset markers rather than live browser dictation behavior.

## What This Changes

The PR adds a browser Web Speech API dictation control to the Control UI chat composer and tests supported/unsupported recognition behavior.

## Best Possible Solution

Decide the intended browser dictation UX and privacy boundary, then either implement dictation through the Gateway transcription/Talk relay with localized labels, tests, and docs, or explicitly approve and document browser Web Speech behavior.

## Reproduction Assessment

Not applicable: this is a feature PR rather than a bug report. Source inspection is high confidence that current main intentionally lacks the stale `Voice input` browser button while related open issues request a dictation UX decision.

## Solution Assessment

No. Direct browser SpeechRecognition is not the best maintainable path without a product/privacy decision because current docs and changelog favor Gateway-mediated audio, and the new control labels bypass the established locale surface.
