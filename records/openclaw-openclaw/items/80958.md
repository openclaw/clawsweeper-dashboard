## Summary

Keep open, but the patch is not merge-ready: the latest PR head still adds direct browser `SpeechRecognition`, uses hardcoded composer labels, and supplies only bundle-marker proof rather than live browser dictation proof.

## What This Changes

This PR adds a Web Speech API dictation button to the Control UI chat composer, streams recognition text into the draft, resets dictation on chat cleanup, and adds chat-view tests for supported and unsupported browsers.

## Best Possible Solution

Decide the intended dictation privacy and UX boundary, then implement it through the Gateway/Talk transcription path with localized labels, tests, and docs, or explicitly approve and document browser Web Speech behavior.

## Reproduction Assessment

Not applicable: this is a feature PR, not a broken existing behavior report. Source inspection shows current `main` lacks browser dictation and explicitly tests that the stale `Voice input` button is absent.

## Solution Assessment

No. Direct browser `SpeechRecognition` is not the narrowest maintainable solution without maintainer privacy/product approval, and the new labels bypass the established Control UI locale surface.
