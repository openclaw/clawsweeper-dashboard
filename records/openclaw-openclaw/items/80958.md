## Summary

Keep open for human review: this PR adds a new Control UI voice-input mode, but the patch has a concrete draft-overwrite bug, lacks real browser behavior proof, and reopens a browser SpeechRecognition product/privacy direction that current main already treats cautiously.

## What This Changes

The PR adds a Web Speech API dictation button to the Control UI chat composer, streams recognition text into the draft, stops dictation during chat cleanup, and adds mocked chat-view tests.

## Best Possible Solution

Decide whether Control UI should support browser-vendor Web Speech dictation; if yes, integrate it with the current chat/Talk UX, preserve live draft edits, document privacy expectations, and require real browser proof before merge.

## Reproduction Assessment

Yes for the review finding by source inspection: start dictation, change the draft while listening, then the next recognition result writes against the start-time baseDraft. For the overall PR, this is a new feature rather than a bug report.

## Solution Assessment

No, not as proposed: the implementation overwrites live draft changes and needs product/privacy approval for browser SpeechRecognition. A safer path would merge transcript text against the latest draft and only ship the browser dictation mode after real browser proof and maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve draft changes made during dictation:** `ui/src/ui/views/chat.ts:299-316`
  - body: The handler captures `baseDraft` only when dictation starts, then every recognition result writes `appendSpeechTranscript(baseDraft, ...)`. If the user types or the app updates the draft while recognition is running, the next interim/final result discards those edits and rewrites from the stale starting text; merge against the latest `props.getDraft?.()` value or otherwise track only the speech-owned suffix.
  - confidence: 0.88
