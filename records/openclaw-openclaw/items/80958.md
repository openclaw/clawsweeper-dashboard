## Summary

Keep open for maintainer product/security review. The PR is a focused source-level attempt to restore Control UI dictation, but it directly reintroduces browser SpeechRecognition after current history removed browser-native voice controls for bypassing configured Talk/TTS provider settings, and the external contributor has not provided real browser behavior proof.

## What This Changes

The PR adds a browser SpeechRecognition dictation button to the Control UI chat composer, appends recognized text into the draft, stops dictation on cleanup, and adds chat view tests for supported and unsupported browsers.

## Best Possible Solution

Decide the intended browser voice UX first; if dictation should return beside Talk, implement an explicit documented source-level path that respects OpenClaw audio privacy/provider policy and includes focused tests plus real browser proof.

## Reproduction Assessment

No for a current-main bug, because this is a feature PR rather than a broken existing behavior report. The review finding is source-reproducible from the diff: the patch directly creates a browser SpeechRecognition instance while current main removed that class of control.

## Solution Assessment

No as written. A source-level dictation implementation is the right place to work, but the direct Web Speech API path needs explicit product/security approval or a different implementation that stays within OpenClaw's configured voice/provider policy.
