## Summary

Keep open for normal PR review. The PR is a focused feature implementation, but it has a source-visible draft-overwrite bug, insufficient real browser behavior proof, and reintroduces direct browser SpeechRecognition after current main deliberately removed that path.

## What This Changes

The PR adds a Web Speech API voice-input button to the Control UI chat composer, streams recognition text into the draft, stops dictation during cleanup, and adds chat view tests.

## Best Possible Solution

Decide the accepted browser dictation UX in the canonical product issues, then implement it source-first with live-draft-safe transcript merging, explicit provider/privacy handling, localized labels, and matching docs/tests.

## Reproduction Assessment

Yes, for the blocking PR bug by source inspection: start dictation, edit the draft before a recognition result arrives, and the handler appends to the captured baseDraft instead of the latest getDraft() value. I did not run tests because this review was read-only.

## Solution Assessment

No. The source-level approach is closer than dist injection, but the current patch can overwrite draft edits and reintroduces a direct Web Speech path that current main removed pending product direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Append transcripts to the live draft:** `ui/src/ui/views/chat.ts:437`
  - body: The handler captures `baseDraft` once when dictation starts and then reuses it for every recognition result. If the user types or edits the composer while dictation is active, the next interim/final result will rebuild the draft from the stale starting text and drop those edits. Read `props.getDraft?.() ?? props.draft` inside `onresult` before appending the transcript, or otherwise merge against the latest draft.
  - confidence: 0.91
- **[P3] Localize the new dictation labels:** `ui/src/ui/views/chat.ts:1584-1585`
  - body: These title/aria strings are new user-visible Control UI copy, but they are hard-coded instead of going through the UI i18n source of truth. Add locale keys in `ui/src/i18n/locales/en.ts` and the related i18n wiring so non-English locales and the i18n sync pipeline can pick them up.
  - confidence: 0.78
