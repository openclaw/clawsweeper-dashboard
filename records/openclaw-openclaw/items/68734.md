## Summary

Keep open: current main still lacks the requested dynamic hook allowlists, and the PR has real-world need plus after-fix proof, but the current head introduces concrete P2 regressions around client tools, snapshot skills, and prompt mutations.

## What This Changes

The PR extends `before_prompt_build` hook contracts with available tool and skill catalogs plus optional `toolsAllow`/`skillsAllow`, then applies those allowlists in the embedded runner with focused tests.

## Best Possible Solution

Land a documented hook seam that exposes the complete model-visible tool catalog and snapshot-resolved skill catalog, then narrows through existing policy, session, and provider prompt-builder paths without losing current prompt mutations.

## Reproduction Assessment

Yes. Source inspection of current main shows the hook contract and call path only expose `prompt` and `messages`, so a plugin cannot observe tool or skill catalogs or return hook-owned allowlists today.

## Solution Assessment

No, not as written. The hook seam is plausible, but the PR needs fixes for complete tool catalog exposure, snapshot-backed skills, and prompt mutation/provider transform preservation before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Expose client-hosted tools to prompt hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are added separately through `clientToolDefs` into `allCustomTools` and sent to the model. A plugin that mirrors the exposed list cannot preserve those tools in `toolsAllow`, so `setActiveToolsByName()` can remove model-visible tools the hook never exposed.
  - confidence: 0.86
- **[P2] Reuse snapshot skills when filtering:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: In snapshot-backed runs this branch reloads current workspace skill entries and rebuilds from live disk state. Current `resolveSkillsPromptForRun()` returns the snapshot prompt when present, so this can inject newer skill content or drop snapshot-only skills during a run that should stay snapshot-consistent.
  - confidence: 0.8
- **[P2] Preserve prompt mutations when narrowing skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from `appendPromptParams` after applying hook system-prompt mutations. Returning `skillsAllow` with `systemPrompt` or prepend/append system context can lose those prompt edits and bypass the provider transform path used for the original prompt.
  - confidence: 0.9
