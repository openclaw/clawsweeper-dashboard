## Summary

Keep open. Current main still lacks this dynamic narrowing hook seam, and the PR has useful real-world support and after-fix log evidence, but the branch still has correctness issues around complete tool exposure, prompt rebuilding, and snapshot-safe skill filtering before it can merge.

## What This Changes

The PR extends `before_prompt_build` hooks so plugins can receive tool and skill catalogs and return `toolsAllow`/`skillsAllow` to narrow per-turn model-visible tools and skill prompt content.

## Best Possible Solution

Land one documented dynamic narrowing seam that only narrows existing policy-approved tools and skills, includes client-hosted tools, preserves skill snapshots and provider transforms, and is covered by focused tests plus real-run proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the requested hook catalogs and allowlists are absent, and PR-head source inspection shows the remaining review findings without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation must expose the complete model-visible surface and rebuild prompts through the normal snapshot, mutation, and provider-transform paths before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebuild the tool prompt after toolsAllow:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the session registry, but the installed system-prompt rebuild function returns the original prompt. A tools-only hook allowlist can therefore leave the OpenClaw Tooling section describing the full pre-filtered tool list; rebuild and reapply the prompt from the filtered tool set.
  - confidence: 0.83
- **[P2] Expose client-hosted tools to prompt hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` comes from `effectiveTools`, but client-hosted tools are added later through `clientToolDefs` into `allCustomTools` and sent to the model. A plugin that mirrors the exposed catalog cannot keep those tools, so hook narrowing can remove model-visible client tools the hook never exposed.
  - confidence: 0.87
- **[P2] Reuse snapshot skills when filtering:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: In snapshot-backed runs this branch reloads current workspace skill entries and rebuilds from live disk state. Current `resolveSkillsPromptForRun()` uses the snapshot prompt first, so this can inject newer skill content or drop snapshot-only skills during a run that should stay snapshot-consistent.
  - confidence: 0.84
- **[P2] Preserve prompt mutations when narrowing skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from `appendPromptParams` after hook system-prompt mutations have been applied. Returning `skillsAllow` with `systemPrompt` or prepend/append system context can lose those edits and bypass the normal provider transform path.
  - confidence: 0.9
