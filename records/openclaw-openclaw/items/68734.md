## Summary

Keep open. The feature is not implemented on current main and has credible user demand, but this branch is conflicting with current main and still has source-level blockers in the tool catalog, skill snapshot, and prompt rebuild paths.

## What This Changes

The PR adds `before_prompt_build` tool and skill catalog inputs plus `toolsAllow` and `skillsAllow` hook results, then wires attempt-phase filtering and tests for dynamic per-turn prompt surface narrowing.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes the complete model-visible catalogs and applies allowlists through canonical prompt/session assembly while preserving snapshots, provider transforms, and hook prompt mutations.

## Reproduction Assessment

Yes, for PR defects by source inspection rather than a current-main bug reproduction. Current main lacks the proposed fields, and the PR head shows catalog, prompt rebuild, and snapshot paths that would misbehave after merge.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation needs a rebase and should feed allowlists into the existing prompt/session pipeline instead of rebuilding older assembled state afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but the model-facing custom tool list later appends client-hosted tools into `allCustomTools`. A plugin that mirrors this catalog cannot keep client tools in its allowlist, so narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and filters those, instead of filtering the captured `skillsSnapshot`. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used by the normal prompt path.
  - confidence: 0.9
