## Summary

Keep open. Current main still lacks the dynamic hook allow-list capability, and the PR addresses a real plugin boundary/token-pressure problem supported by the discussion, but the current patch has blocking runner correctness issues around prompt rebuilding, client-hosted tools, and skill snapshots.

## What This Changes

The PR extends before_prompt_build hooks with availableTools/availableSkills inputs and toolsAllow/skillsAllow outputs, then applies those outputs in the embedded runner to narrow per-turn tools and the injected skills catalog.

## Best Possible Solution

Keep the PR open and land the dynamic narrowing seam after a rebase that composes with existing prompt transforms, exposes every model-visible tool to the hook, filters snapshot-resolved skill data, and adds focused tests plus docs/changelog coverage for the public hook contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source inspection shows the missing capability, and the PR-head bugs can be reproduced with a hook returning skillsAllow plus prompt mutations, a run with clientTools, or a snapshot-backed skills run.

## Solution Assessment

Is this the best way to solve the issue? No as written. The plugin-owned dynamic allow-list direction fits the OpenClaw boundary better than MCP-server-side workarounds, but the implementation needs safer composition with the existing runner and snapshot contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When skillsAllow is set, this block rebuilds the system prompt from appendPromptParams and applies it directly. That discards any systemPrompt/prependSystemContext/appendSystemContext already applied from the hook and skips the provider transform path, so a plugin combining skill filtering with prompt edits loses those edits for the turn.
  - confidence: 0.9
- **[P2] Include client tools in the hook-visible catalog:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: availableTools is built from effectiveTools only, but client-hosted tools are registered later through clientToolDefs/allCustomTools. A plugin deriving toolsAllow from availableTools cannot include those client tools, and setActiveToolsByName can then deactivate them unintentionally.
  - confidence: 0.86
- **[P2] Filter snapshot skills without reloading the workspace:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this branch reloads current workspace skill entries instead of filtering the session snapshot. That can inject newer on-disk skill content or drop skills that existed in the snapshot, breaking the snapshot consistency the current skills runtime preserves.
  - confidence: 0.8
- **[P3] Document the new hook contract:** `src/plugins/hook-before-agent-start.types.ts:30-38`
  - body: This adds public Plugin SDK event/result fields, but the PR does not update the plugin hook docs or changelog. Plugin authors would not see the new availableTools/availableSkills and toolsAllow/skillsAllow contract in the public docs, and this user-facing feature needs a changelog entry before merge.
  - confidence: 0.78
