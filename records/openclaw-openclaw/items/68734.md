## What This Changes

## Best Possible Solution

Land a rebased, documented hook/API implementation that exposes the complete model-visible catalog and applies allowlists through the current prompt/session assembly path without bypassing policy, provider transforms, or skill snapshots.

## Solution Assessment

No, not as written. The hook seam is plausible, but this branch needs maintainer API direction plus a rebase and fixes for client tools, prompt/provider rebuilds, and snapshot skill handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is built from `effectiveTools`, but client-hosted tools are added later through `clientToolDefs` and then sent to the model in `allCustomTools`. A plugin that mirrors the exposed catalog cannot retain those client tools, so hook narrowing can remove tools the plugin never had a chance to allow.
  - confidence: 0.88
- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the session after the system prompt and provider transform were built from the pre-filtered tool list. A tools-only allowlist can leave prompt/provider state advertising removed tools, so the allowlist needs to feed the normal prompt assembly path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skills from disk and rebuilds without the captured snapshot. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
