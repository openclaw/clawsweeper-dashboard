## Summary

Keep open: current main still lacks the requested prompt-hook tool/skill narrowing seam, and the discussion includes real deployment evidence that the feature would help MCP-heavy and skill-heavy setups. The PR is useful but not mergeable as-is because PR-head source inspection still shows correctness bugs around client tools, snapshot skills, and prompt rebuild composition.

## What This Changes

The PR adds before_prompt_build hook fields for available tool/skill catalogs and per-turn toolsAllow/skillsAllow narrowing, with embedded runner wiring and focused tests.

## Best Possible Solution

Rebase the feature onto current main, keep the hook-owned narrowing seam, include every model-visible tool in the hook contract, filter skills from already-resolved skill data, and rebuild prompts through the existing prompt composition/provider transform path with focused tests.

## Reproduction Assessment

Yes. Source inspection verifies current main lacks the requested hook fields, and PR-head edge cases are reproducible from the code paths for clientTools plus toolsAllow, skillsSnapshot plus skillsAllow, and skillsAllow combined with hook system-prompt mutations.

## Solution Assessment

No, not as written. The feature direction is plausible and supported by discussion, but the current patch needs rebase and correctness fixes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve hook prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When skillsAllow is set, this block rebuilds systemPromptText from appendPromptParams after hookResult.systemPrompt/prependSystemContext/appendSystemContext were already applied. That overwrites those hook mutations for the turn and skips the provider system-prompt transform path used by the original prompt.
  - confidence: 0.9
- **[P2] Expose client tools to prompt-build hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: availableTools is built from effectiveTools only, but client-hosted tools are registered separately through clientToolDefs/allCustomTools. A plugin deriving toolsAllow from availableTools cannot include those client tools, and setActiveToolsByName(hookResult.toolsAllow) can deactivate model-visible tools it never saw.
  - confidence: 0.86
- **[P2] Filter snapshot skills without live workspace reload:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this branch ignores the existing skills snapshot and reloads current workspace skill entries. That can inject newer on-disk skill content or drop skills that were present in the session snapshot, breaking the snapshot consistency the skills runtime preserves.
  - confidence: 0.78
