## What This Changes

This PR adds before_prompt_build hook fields for availableTools/availableSkills plus toolsAllow/skillsAllow, then wires those hook results into embedded attempt tool filtering and skills prompt rebuilding.

## Best Possible Solution

Land a rebased, maintainer-approved hook contract that exposes the complete model-visible tool and skill catalog, applies narrowing without dropping existing hook or provider prompt mutations, and covers client tools plus retry/session restoration behavior.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. The two blocking review issues are source-reproducible from the PR head: skillsAllow plus prompt mutations drops the mutations, and clientTools are registered outside the advertised availableTools list.

## Solution Assessment

No, not yet. The direction is promising, but the implementation should be rebased onto current main, preserve existing prompt composition and provider transforms, and include all model-visible tool sources before maintainers decide the API shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds a fresh system prompt from `appendPromptParams` and applies it after the earlier hook `systemPrompt`/`prependSystemContext`/`appendSystemContext` handling. A plugin that returns `skillsAllow` together with prompt guidance loses that guidance for the turn, and provider-specific prompt transforms from the normal path are skipped. Rebuild the narrowed skills prompt through the same composition/transform path used for the original system prompt.
  - confidence: 0.92
- **[P1] Include client-hosted tools in hook availableTools:** `src/agents/pi-embedded-runner/run/attempt.ts:1837-1844`
  - body: `availableTools` is only derived from `effectiveTools`, but client-hosted tools are converted into `clientToolDefs` and passed to the session as part of `allCustomTools`. A classifier that trusts the hook event cannot include those client tools, and the later `setActiveToolsByName(hookResult.toolsAllow)` call can remove them from the active tool surface. Include client tool names in the hook event or apply the hook after the final custom tool list is assembled.
  - confidence: 0.89
