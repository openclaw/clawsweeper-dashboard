## Summary

Keep open: current main does not already provide the proposed prompt-hook tool/skill narrowing API, and the PR has credible real-world motivation, including the MCP-heavy deployment data in discussion. The branch still has merge-blocking prompt/session correctness issues and needs maintainer API review before it can land.

## What This Changes

The branch adds `before_prompt_build` tool and skill catalog inputs plus `toolsAllow` and `skillsAllow` hook results, then filters attempt tools and the injected skill prompt catalog with focused tests.

## Best Possible Solution

Land a rebased hook implementation that exposes the complete model-visible catalog and applies allowlists through the current prompt/session assembly path while preserving provider transforms, hook mutations, policy, and captured skill snapshots.

## Reproduction Assessment

Yes, for PR defects by source inspection rather than a current-main bug reproduction. Current main lacks the proposed fields, and the PR head shows catalog, prompt rebuild, and snapshot paths that would misbehave after merge.

## Solution Assessment

No, not as written. The hook seam is plausible, but the patch should be rebased and should feed allowlists into the canonical prompt/session pipeline instead of rebuilding older assembled state afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but the session registers client-hosted tools separately through `allCustomTools`. A plugin that mirrors the catalog into `toolsAllow` cannot include those client tools, so narrowing can remove tools it was never shown.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and filters those, instead of filtering the captured snapshot content. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used by the normal prompt path.
  - confidence: 0.9
