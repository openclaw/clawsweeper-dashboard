## Summary

Keep open: the feature is useful and current main still lacks this dynamic allowlist hook surface, but the PR has source-level blockers around complete tool catalogs, prompt/provider rebuilds, and snapshot skill handling.

## What This Changes

The PR extends `before_prompt_build` so plugins can receive available tool and skill names and return per-turn `toolsAllow`/`skillsAllow` allowlists.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes the complete model-visible catalog and applies allowlists through the canonical prompt/session assembly while preserving policy, provider transforms, and captured skill snapshots.

## Reproduction Assessment

Yes, as a source-level PR-defect check rather than a current-main bug reproduction. Current main lacks the proposed hook fields, and the PR diff shows the remaining catalog, prompt rebuild, and snapshot issues.

## Solution Assessment

No, not as written. The hook seam is plausible, but this branch needs a rebase and fixes that feed allowlists through the existing prompt/session pipeline instead of patching state after assembly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` only lists `effectiveTools`, while client-hosted tools are added separately to the model-facing custom tools. A plugin that mirrors this catalog cannot retain those client tools, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` runs after the system prompt and provider transform were built from the pre-filtered tool surface. A tools-only allowlist can leave prompt/provider state advertising removed tools, so the allowlist needs to feed the normal prompt assembly path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skill entries from disk and rebuilds without the captured snapshot. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
