The PR extends `before_prompt_build` hooks with `availableTools`/`availableSkills` inputs and `toolsAllow`/`skillsAllow` outputs, then applies those allowlists in the embedded runner with focused tests and public hook type updates.

## Best Possible Solution

Land a rebased, documented hook implementation that feeds allowlists through the current canonical session and `buildAttemptSystemPrompt` paths, includes client-hosted tools in catalogs, filters captured skill snapshots, and covers those cases with regression tests.

## Reproduction Assessment

Yes, for the PR defects by source inspection. The PR head shows `availableTools` built before client tools, snapshot-backed skills reloaded from disk, and skill-filtered prompts rebuilt outside the current transform path.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation should be rebased and integrated with the current canonical tool/session and prompt assembly flow instead of patching older assembled state afterward.
