## Summary

Keep open. The feature is useful and not implemented on current main, with real-world motivation in the discussion, but this PR is currently conflicting and still has merge-blocking correctness issues around the complete tool surface, snapshot-backed skills, and prompt rebuilding.

## What This Changes

The PR extends prompt-build hooks so plugins can see available tool and skill catalogs and return per-turn `toolsAllow`/`skillsAllow` allowlists that narrow model-visible tools and injected skill prompt content.

## Best Possible Solution

Land one documented dynamic narrowing seam that exposes the complete model-visible tool catalog, handles snapshot-consistent skills, and reapplies filtering through the normal prompt, provider-transform, and session paths without broadening access.

## Reproduction Assessment

Yes. Source inspection of current main shows the prompt-build hook cannot receive tool or skill catalogs or return allowlists, and source inspection of the PR head shows the remaining review findings without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the patch must include client-hosted tools, keep snapshot-backed skill prompts consistent, and rebuild filtered prompts through the existing mutation/provider-transform path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebuild the tool prompt after toolsAllow:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the session registry, but OpenClaw has installed a constant system-prompt rebuild function, so a tools-only hook allowlist leaves the model-visible Tooling section built from the full pre-filtered `effectiveTools` list. Rebuild and reapply the OpenClaw system prompt from the filtered tool set.
  - confidence: 0.82
- **[P2] Expose client-hosted tools to prompt hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are added later through `clientToolDefs` into `allCustomTools` and sent to the model. A plugin that mirrors the exposed list cannot retain those tools, so `setActiveToolsByName()` can remove model-visible client tools the hook never exposed.
  - confidence: 0.87
- **[P2] Reuse snapshot skills when filtering:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: In snapshot-backed runs this branch reloads current workspace skill entries and rebuilds from live disk state. Current `resolveSkillsPromptForRun()` returns the snapshot prompt before live entries, so this can inject newer skill content or drop snapshot-only skills during a run that should stay snapshot-consistent.
  - confidence: 0.84
- **[P2] Preserve prompt mutations when narrowing skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from `appendPromptParams` after hook system-prompt mutations have already been applied. Returning `skillsAllow` with `systemPrompt` or prepend/append system context can lose those edits and also bypass the normal provider transform path.
  - confidence: 0.9
