## Summary

Keep open. Current main still does not expose tool or skill catalogs to `before_prompt_build`, but this PR is conflict-marked and its current head has merge-blocking prompt/catalog consistency issues that need maintainer API direction before landing.

## What This Changes

The PR extends prompt-build hooks with tool and skill catalog inputs plus per-turn `toolsAllow` and `skillsAllow` outputs, then applies those allowlists in the embedded runner.

## Best Possible Solution

Land one rebased, documented dynamic narrowing seam that exposes the complete model-visible catalog, preserves snapshot-backed skill prompts, and reapplies prompt/provider/session updates through the normal run path without broadening access.

## Reproduction Assessment

Yes. Source inspection of current main shows `before_prompt_build` cannot receive tool or skill catalogs or return allowlists, and source inspection of PR head shows the remaining merge-blocking defects without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the current branch must be rebased and fixed so client tools, snapshot skills, and hook/provider prompt mutations stay consistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` runs after the session was given a system-prompt rebuild function based on the pre-filtered prompt. A tools-only hook allowlist can therefore narrow active tools while leaving the Tooling section built from the full tool set; rebuild and reapply the system prompt from the filtered tools through the normal prompt path.
  - confidence: 0.84
- **[P2] Expose client-hosted tools to prompt hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived only from `effectiveTools`, but client-hosted tools are added later to `allCustomTools` and sent to the model. A plugin that mirrors the exposed list cannot retain those tools, so an allowlist can unintentionally remove model-visible client tools it never saw.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries and rebuilds from disk. Snapshot runs should stay tied to the captured snapshot; otherwise filtering can inject newer skill content or drop snapshot-only skills.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
