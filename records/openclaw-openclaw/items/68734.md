## Summary

Keep open. The dynamic narrowing feature is not implemented on current main and has real user value, but this PR still has correctness blockers around client-hosted tools, snapshot-backed skills, and prompt transform preservation before it can merge.

## What This Changes

The PR adds `availableTools`/`availableSkills` hook inputs and `toolsAllow`/`skillsAllow` hook results, then applies them in the embedded runner to narrow per-turn tool and skill prompt surfaces.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes complete model-visible catalogs and applies allowlists through the canonical session and prompt assembly path while preserving snapshots, provider transforms, hook mutations, policy, and regression tests.

## Reproduction Assessment

Yes, for the PR defects by source inspection. Current main lacks the proposed fields, and the PR head shows catalog, snapshot, and prompt rebuild paths that would misbehave after merge.

## Solution Assessment

No, not as written. The hook seam is plausible, but the branch should be rebased and feed allowlists into the existing prompt/session pipeline instead of rebuilding older assembled state afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are registered later through `allCustomTools`. A plugin that mirrors the exposed catalog cannot include those client tool names, so `setActiveToolsByName(...)` can remove tools the hook never had a chance to preserve.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and filters those, instead of filtering `params.skillsSnapshot`. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from `appendPromptParams` and applies the result directly. That can drop earlier hook system-prompt mutations and bypass the provider transform path used by normal prompt assembly.
  - confidence: 0.9
