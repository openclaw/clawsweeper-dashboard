## Summary

Keep open: current main still lacks this hook capability, and the PR has concrete P2 runtime blockers before it is mergeable. The feature direction is supported by real deployment evidence, but the branch needs current-main integration fixes around client-hosted tools, snapshot-backed skills, and system-prompt rebuilding.

## What This Changes

The PR extends `before_prompt_build` so plugins can receive available tool and skill names and return per-turn `toolsAllow` and `skillsAllow` allowlists that narrow the model prompt surface.

## Best Possible Solution

Rebase onto current main, expose the complete model-visible tool and resolved skill catalogs to hooks, apply allowlists through existing session and prompt-builder seams, and add focused coverage for client tools, snapshot skills, provider transforms, and combined hook prompt mutations.

## Reproduction Assessment

Yes. Source inspection on current main shows the hook receives only `prompt` and `messages` and the result type lacks allowlists; the PR-head blockers are traceable from the diff and current-main tool, skill, and prompt paths without running tests.

## Solution Assessment

No, not as written. The approach is plausible, but the implementation should preserve current hook aggregation, include client-hosted tools, filter snapshot skills without live reload, and rebuild narrowed prompts through the normal provider-transform path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Expose client tools to prompt-build hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1840`
  - body: `availableTools` is built from `effectiveTools`, but client-hosted tools are registered separately through `clientToolDefs` and included in `allCustomTools`. A plugin deriving `toolsAllow` from the exposed catalog cannot include those model-visible tools, and `setActiveToolsByName(...)` can remove tools the hook never saw.
  - confidence: 0.86
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skill entries before filtering. That can inject newer on-disk skill content or drop skills that existed in the session snapshot, breaking the snapshot consistency current main preserves.
  - confidence: 0.8
- **[P2] Preserve hook prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds `systemPromptText` from the original prompt params after hook system-prompt mutations were applied. Runs that return `skillsAllow` together with `systemPrompt`, prepend, or append system context can silently lose those same-turn prompt edits and provider transforms.
  - confidence: 0.9
- **[P2] Preserve current hook aggregation on rebase:** `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts:90-96`
  - body: The branch helper is based on an older return shape. Current main also merges drained next-turn injections, `agent_turn_prepare`, heartbeat contribution, and append context; keeping the branch body during rebase would drop those prompt mutations on affected runs.
  - confidence: 0.84
