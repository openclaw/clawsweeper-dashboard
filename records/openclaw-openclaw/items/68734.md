## Summary

Keep open: current main does not implement the proposed hook-owned tool/skill narrowing, and the PR has useful real-world support, but the branch still has concrete embedded-runner correctness blockers before it is mergeable.

## What This Changes

The PR adds before_prompt_build fields for availableTools, availableSkills, toolsAllow, and skillsAllow, then uses hook-returned allowlists to narrow per-turn tools and skill prompt content in the embedded runner.

## Best Possible Solution

Keep the hook-owned narrowing contract, but rebase it onto current main and route all model-visible tools plus already-resolved skill data through the existing prompt composition/provider transform path with focused regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the proposed hook fields, and the PR-head edge cases are traceable in the diff for client tools, snapshot skills, and skillsAllow prompt rebuilding.

## Solution Assessment

No, not as written. The direction is plausible and supported by real deployment evidence, but the branch needs correctness fixes and current-main integration before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve hook prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When skillsAllow is set, this rebuilds the system prompt from appendPromptParams after hookResult.systemPrompt and prepend/append system context have already been applied. That drops those hook mutations for the turn and skips the provider system-prompt transform path used by normal prompt construction.
  - confidence: 0.9
- **[P2] Expose client tools to prompt-build hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839-1840`
  - body: availableTools is built from effectiveTools only, but client-hosted tools are registered later through clientToolDefs/allCustomTools. A plugin deriving toolsAllow from availableTools cannot include those model-visible tools, and setActiveToolsByName can deactivate tools the hook never saw.
  - confidence: 0.86
- **[P2] Filter snapshot skills without live workspace reload:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this branch reloads current workspace skill entries instead of filtering the existing snapshot data. That can inject newer on-disk skill content or drop skills that existed in the session snapshot, breaking the snapshot consistency current skills runtime preserves.
  - confidence: 0.78
