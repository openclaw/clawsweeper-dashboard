## Summary

Keep open: the feature is useful and has real-world support, but the PR head is still not correct against the current prompt/session pipeline. Source review shows remaining blockers around client-hosted tools, prompt/provider rebuild semantics, and snapshot-backed skill filtering.

## What This Changes

The PR extends `before_prompt_build` hooks so plugins can see available tool and skill names and return per-turn `toolsAllow`/`skillsAllow` allowlists.

## Best Possible Solution

Land a rebased, documented hook/API implementation that exposes the complete model-visible catalog and applies allowlists through current prompt/session assembly without bypassing policy, provider transforms, or skill snapshots.

## Reproduction Assessment

Yes, as a source-level PR-defect check rather than a current-main bug reproduction. Current main lacks the proposed allowlist hook surface, and PR head source shows the remaining correctness issues without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but this branch needs maintainer API direction plus fixes for client tools, prompt/provider rebuilds, and snapshot skill handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` only lists `effectiveTools`, but client-hosted tools are added later to the model-facing custom tool list. A plugin that mirrors the exposed list cannot retain those client tools, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the active session after the system prompt/report/provider transform were built from the pre-filtered tool list. A tools-only allowlist can leave prompt state advertising removed tools, so the allowlist needs to feed the normal prompt assembly path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and rebuilds without the snapshot. Snapshot runs should stay tied to captured skill content; otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
