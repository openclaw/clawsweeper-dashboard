## Summary

Keep open. Current main still does not implement the proposed tool/skill allowlist hook contract, but this PR is conflicting and still has source-level correctness blockers around complete tool catalogs, snapshot skills, and prompt rebuild semantics.

## What This Changes

This PR adds `before_prompt_build` `toolsAllow` and `skillsAllow` hook outputs, then wires attempt-phase filtering so plugins can narrow model-visible tools and injected skill catalog entries per turn.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes the complete model-visible catalog and applies allowlists through the canonical prompt/session assembly path while preserving provider transforms, hook mutations, policy, and captured skill snapshots.

## Reproduction Assessment

Yes, as a source-level PR-defect check rather than a current-main bug reproduction. Current main lacks the proposed fields, and the PR head shows catalog, prompt rebuild, and snapshot paths that would misbehave.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation needs to be rebased onto current main and feed allowlists into the existing prompt/session pipeline rather than patching older assembled state afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but the model-facing custom tool list later appends `clientToolDefs` into `allCustomTools`. A plugin that mirrors this catalog cannot keep client-hosted tools in its allowlist, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and filters those, instead of filtering the captured `skillsSnapshot`. Snapshot runs should stay tied to the captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
