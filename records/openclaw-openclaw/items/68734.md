## Summary

Keep open: the requested dynamic tool/skill narrowing seam is not on current main, and the real-world MCP-heavy deployment comment supports that the feature solves a concrete prompt-size problem. The PR is not merge-ready because the branch still has source-confirmed correctness issues around client-hosted tools, prompt rebuilding, snapshot-backed skills, and prompt mutations.

## What This Changes

The PR adds `availableTools`/`availableSkills` hook inputs and `toolsAllow`/`skillsAllow` hook outputs so prompt-build plugins can narrow the model-visible tool and skill prompt surface per turn.

## Best Possible Solution

Land one rebased, documented dynamic narrowing seam that exposes the complete model-visible catalog, preserves snapshot-backed skills, and reapplies prompt/provider/session updates without broadening access.

## Reproduction Assessment

Yes. Source inspection of current main shows the hook cannot receive tool or skill catalogs or return allowlists, and source inspection of PR head confirms the merge-blocking defects without a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the patch must include client-hosted tools, rebuild filtered prompts through the normal path, and keep snapshot skill prompts consistent after rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Rebuild the system prompt after toolsAllow:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the session registry, but OpenClaw installed a constant system-prompt rebuild function from the pre-filtered prompt. A tools-only hook allowlist can still leave the Tooling section built from the full `effectiveTools` list, so rebuild and reapply the prompt from the filtered tool set.
  - confidence: 0.82
- **[P2] Expose client-hosted tools to prompt hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived only from `effectiveTools`, but client-hosted tools are later added to `allCustomTools` and sent to the model. A plugin that mirrors the exposed list cannot retain those tools, so the allowlist can unintentionally remove model-visible client tools it never saw.
  - confidence: 0.87
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: In snapshot-backed runs this branch reloads current workspace skill entries and rebuilds from live disk state. Snapshot runs should stay tied to the captured snapshot, otherwise the filter can inject newer skill content or drop snapshot-only skills.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params after hook system-prompt mutations have already been applied. Returning `skillsAllow` with `systemPrompt` or prepend/append system context can lose those edits and bypass the normal provider transform path.
  - confidence: 0.9
