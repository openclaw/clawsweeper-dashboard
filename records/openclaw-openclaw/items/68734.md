## Summary

Keep open: the feature is useful and the live logs plus the MCP-heavy deployment comment support the need, but this branch is not merge-ready against current main and still has source-level correctness gaps in the hook/session prompt path.

## What This Changes

The PR extends `before_prompt_build` so plugins can inspect available tool and skill names and return per-turn `toolsAllow` and `skillsAllow` filters.

## Best Possible Solution

Expose a documented, complete per-turn narrowing contract that uses the current model-visible tool and skill catalogs, preserves snapshot semantics, and routes all prompt rebuilds through the existing provider/session pipeline.

## Reproduction Assessment

Yes, as a source-level PR review path rather than a current-main bug reproduction. Current main lacks this hook allowlist API, and PR head shows the remaining defects in the proposed implementation without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the branch needs maintainer API direction plus fixes for client-hosted tools, prompt/provider rebuilds, and snapshot skill handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` only lists `effectiveTools`, but client-hosted tools are added separately to the model-facing custom tool list. A plugin that mirrors the exposed list cannot retain those client tools, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the active session tools after the OpenClaw system prompt was built from the pre-filtered tool list. A tools-only allowlist can leave the prompt describing tools that are no longer active, so the prompt needs to be rebuilt through the normal prompt/provider path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and rebuilds without the snapshot. Snapshot runs should remain tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
