## Summary

Keep open: current main still lacks hook-owned dynamic tool/skill narrowing, and the discussion includes useful real-world token-pressure evidence, but the PR head has concrete correctness blockers around client tools, snapshot-backed skills, and system-prompt rebuild semantics before it can merge.

## What This Changes

The PR extends prompt-build hook contracts and embedded runner wiring so plugins can return `toolsAllow` and `skillsAllow` to narrow per-turn tool schemas and skill prompt injection.

## Best Possible Solution

Rebase onto current main, keep the narrow-only plugin seam, expose every model-visible tool to hooks, filter skills from resolved snapshot data, rebuild through the existing prompt composition/provider transform path, and add focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested hook catalogs and allow-lists are absent, and PR-head failures are reproducible from the client-tool, snapshot-skill, and prompt-rebuild code paths.

## Solution Assessment

No as written. The direction fits the plugin boundary better than MCP-server workarounds, but this patch needs the catalog, snapshot, prompt-composition, docs/changelog, and rebase issues fixed before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Include client tools in hook-visible tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` only lists `effectiveTools`, but client-hosted tools are registered separately through `clientToolDefs`/`allCustomTools`. A plugin deriving `toolsAllow` from this event cannot preserve those tools, and `setActiveToolsByName(hookResult.toolsAllow)` can deactivate model-visible client tools that were never exposed to the hook.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this branch reloads current workspace skill entries instead of filtering the resolved snapshot data. That can inject newer on-disk skill content or drop skills that existed in the session snapshot, breaking the snapshot consistency preserved by the existing skills runtime.
  - confidence: 0.82
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds the system prompt from `appendPromptParams` after `systemPrompt`/`prependSystemContext`/`appendSystemContext` hook mutations were already applied, and it skips the provider system-prompt transform used by the original prompt. Runs combining skill filtering with prompt mutations or provider transforms will silently lose those changes for the turn.
  - confidence: 0.9
- **[P3] Document and changelog the hook contract:** `src/plugins/hook-before-agent-start.types.ts:30-84`
  - body: This adds public plugin hook fields, but the PR does not update plugin docs or `CHANGELOG.md`. OpenClaw policy requires user-facing feat/API changes to be documented and changeloged so plugin authors have a supported contract for the new event/result fields.
  - confidence: 0.76
