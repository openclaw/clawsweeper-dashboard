## Summary

Keep open. Current main does not implement the proposed hook allowlist contract, and the PR has real use-case support plus after-fix live logs, but the current patch still has blocking correctness issues around client tools, snapshot-backed skills, and prompt rebuild semantics.

## What This Changes

The PR extends `before_prompt_build` with available tool/skill catalogs and `toolsAllow`/`skillsAllow`, then wires those allowlists into embedded runner tool and skill prompt filtering with focused tests.

## Best Possible Solution

Land a documented hook/API seam that exposes the complete model-visible tool catalog and resolved skill catalog, narrows only through existing policy/session/prompt-builder seams, preserves snapshots/provider transforms/prompt mutations, and adds focused runner plus hook contract coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the hook receives only `prompt` and `messages` and has no allowlist fields; the PR-head blockers are traceable from the diff and current tool, skill snapshot, and prompt rebuild paths without running a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation should be rebased onto current main and fix complete tool catalog exposure, snapshot skill filtering, and skill prompt rebuild through the normal provider-transform and hook-mutation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Expose client-hosted tools to prompt-build hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1840`
  - body: `availableTools` is built from `effectiveTools`, but client-hosted tools are registered separately through `clientToolDefs` and included in `allCustomTools`. A plugin deriving `toolsAllow` from the exposed catalog cannot keep those model-visible tools, and `setActiveToolsByName(...)` can remove tools the hook never saw.
  - confidence: 0.86
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skill entries before filtering. That can inject newer on-disk skill content or drop skills that existed in the session snapshot, breaking the snapshot consistency current main preserves.
  - confidence: 0.8
- **[P2] Preserve hook prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds `systemPromptText` from the original prompt params after hook system-prompt mutations were applied. Runs that return `skillsAllow` together with `systemPrompt`, prepend, or append system context can silently lose those same-turn prompt edits and provider transforms.
  - confidence: 0.9
