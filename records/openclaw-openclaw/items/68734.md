## Summary

Keep open. Current main still lacks the requested hook-based dynamic narrowing, and the discussion shows a real MCP-heavy/skill-heavy need, but the PR head is not merge-ready because the new filtering paths can drop client-hosted tools, lose hook prompt mutations, and break skill snapshot consistency.

## What This Changes

The PR extends before_prompt_build hooks with tool and skill catalog inputs plus toolsAllow/skillsAllow outputs, then applies those outputs in the embedded runner to narrow per-turn tools and skill prompt injection.

## Best Possible Solution

Rebase the PR onto current main, keep the dynamic narrowing seam, include every model-visible tool in the hook contract, filter skills from the resolved snapshot data, rebuild prompts through the existing hook composition/provider transform path, and add focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. Source inspection reproduces the missing current-main capability, and PR-head failures can be exercised with clientTools plus toolsAllow, skillsSnapshot plus skillsAllow, or skillsAllow combined with systemPrompt/prependSystemContext/appendSystemContext.

## Solution Assessment

No as written. The feature direction fits the plugin boundary better than server-side MCP workarounds, but the current patch needs the tool catalog, snapshot, prompt composition, and rebase issues fixed before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve hook system prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When skillsAllow is set, this block rebuilds systemPromptText from appendPromptParams after hookResult.systemPrompt/prependSystemContext/appendSystemContext were already applied. That overwrites those hook mutations for the turn and also skips the provider system-prompt transform path used by the original prompt.
  - confidence: 0.9
- **[P2] Include client tools in the hook-visible catalog:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: availableTools is built from effectiveTools only, but client-hosted tools are registered later through clientToolDefs/allCustomTools. A plugin deriving toolsAllow from availableTools cannot include those client tools, and setActiveToolsByName(hookResult.toolsAllow) can deactivate them.
  - confidence: 0.86
- **[P2] Filter snapshot skills without reloading the workspace:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this branch ignores params.skillsSnapshot and reloads current workspace skill entries. That can inject newer on-disk skill content or drop skills that were present in the session snapshot, breaking the snapshot consistency the existing skills runtime preserves.
  - confidence: 0.78
