## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve empty toolsAllow as deny-all:** `src/agents/pi-embedded-runner/run/attempt.ts:1884`
  - body: This guard only applies hook filtering when `toolsAllow.length > 0`, so a plugin returning `toolsAllow: []` gets a passthrough full tool set. Current runtime `toolsAllow` treats an explicit empty array as no tools, and the PR describes the hook as matching that contract, so this can expose tools that a plugin intentionally denied for the turn. Apply the hook whenever `toolsAllow` is defined, including `[]`, and update the session with the filtered registered names.
  - confidence: 0.92
- **[P1] Keep hook system mutations when rebuilding skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1951`
  - body: When `skillsAllow` is set, this branch rebuilds from `appendPromptParams` and overwrites `systemPromptText`, which discards any `systemPrompt`, `prependSystemContext`, or `appendSystemContext` already applied above and bypasses the provider transform path used for the normal prompt. A plugin that narrows skills while also adding system guidance will silently lose that guidance for the turn. Rebuild from the current composed prompt or reapply the same hook composition and provider transform after filtering the skill catalog.
  - confidence: 0.9
- **[P1] Expose client tools before applying toolsAllow:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: The hook event exposes `availableTools` from `effectiveTools`, but client-hosted tools are registered separately in `allCustomTools`. A classifier that returns the list it was shown cannot include those client tools, and `setActiveToolsByName(hookResult.toolsAllow)` can remove them from the session for that turn. Include the client tool names in the hook event or only filter the same surface that the hook can observe.
  - confidence: 0.84
