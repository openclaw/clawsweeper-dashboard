## Summary

Keep open. Current main still lacks this hook capability, and the PR addresses a real plugin-owned prompt-surface gap with live log proof, but PR head still has merge-blocking correctness issues around client tools, prompt rebuilding, and snapshot/provider-safe skill filtering.

## What This Changes

This PR extends `before_prompt_build` hooks to expose tool and skill catalogs and lets plugins return per-turn `toolsAllow` and `skillsAllow` allowlists.

## Best Possible Solution

Land a rebased, documented hook/API path that exposes the complete model-visible catalog and applies allowlists through the current prompt/session pipeline without bypassing policy or snapshot semantics.

## Reproduction Assessment

Yes, as a source-level gap and PR-defect check rather than a bug reproduction. Current main lacks hook catalogs/allowlists, and PR head source shows the remaining correctness defects without needing a live gateway.

## Solution Assessment

No, not as written. The hook seam is plausible, but the current branch needs maintainer API direction plus fixes for client tools, prompt/provider rebuilds, and snapshot skill handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` only lists `effectiveTools`, but client-hosted tools are added separately to the model-facing custom tool list. A plugin that mirrors the exposed list cannot retain those client tools, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Rebuild the prompt after narrowing tools:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` narrows the active session tools after the OpenClaw system prompt was built from the pre-filtered tool list. A tools-only allowlist can leave the prompt describing tools that are no longer active, so the prompt needs to be rebuilt through the normal prompt/provider path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and rebuilds without the snapshot. Snapshot runs should remain tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
