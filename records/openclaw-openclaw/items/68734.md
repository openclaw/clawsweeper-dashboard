## Summary

Keep open: current main still lacks the proposed hook allowlist capability, and the PR addresses a real prompt-surface problem, but the branch is not mergeable as written because allowlists are applied after key prompt/session assembly paths and miss client-tool, snapshot-skill, and hook-mutation cases.

## What This Changes

The PR extends `before_prompt_build` so plugins can see tool and skill catalogs and return per-turn `toolsAllow` and `skillsAllow` allowlists.

## Best Possible Solution

Land a rebased hook implementation that exposes the complete model-visible catalog and applies allowlists through canonical prompt, provider-transform, session, and skill-snapshot assembly paths.

## Reproduction Assessment

Yes, as a source-level PR-defect check: current main lacks the proposed fields, while the PR head shows the catalog, prompt rebuild, and snapshot paths that would misbehave. This is not a current-main bug reproduction because the capability is not yet present on main.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation should feed allowlists into the normal prompt/session assembly path instead of patching active state after the prompt and provider transform are already built.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are appended later to the model-facing custom tool list. A plugin that mirrors the catalog cannot retain those client tools, and the later active-tool update can remove tools the plugin never saw.
  - confidence: 0.88
- **[P2] Feed tool allowlists through prompt assembly:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: The system prompt and provider transform are built from the pre-filtered tool list before `toolsAllow` is applied. Because the session prompt override freezes the generated prompt, narrowing tools here can leave removed tools advertised in prompt/provider state for the turn.
  - confidence: 0.86
- **[P2] Filter snapshot skills from the snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this loads current workspace skill entries and rebuilds from them instead of filtering the captured snapshot. That can inject newer skill content or drop snapshot-only entries, breaking the snapshot contract for the run.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can discard earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
