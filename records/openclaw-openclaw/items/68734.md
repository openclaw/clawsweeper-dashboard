The PR adds optional `toolsAllow` and `skillsAllow` outputs plus hook-visible tool and skill catalogs so `before_prompt_build` plugins can narrow the per-turn model tool surface and injected skills catalog.

## Best Possible Solution

Land one maintainer-approved dynamic narrowing contract that preserves the normal prompt pipeline, exposes the final model-visible tool catalog, handles per-turn session tools safely, and includes focused compatibility tests plus current-main proof.

Full review comments:

- **[P1] Preserve prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this block rebuilds the system prompt from `buildEmbeddedSystemPrompt(...)` and applies it directly. That discards any `hookResult.systemPrompt` or prepend/append system context applied above, and it skips the provider transform path used for the original prompt, so plugins or providers can silently lose guidance for that turn.
  - confidence: 0.9
- **[P2] Include client-hosted tools in hook allowlists:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is built from `effectiveTools`, but client-hosted tools are added separately into `allCustomTools`. A plugin that returns `toolsAllow` based on the event cannot include those hosted tools, and `setActiveToolsByName(...)` can remove them from the active session for the turn.
  - confidence: 0.86
