## Summary

Keep open: current main does not implement this dynamic hook narrowing capability, and this active PR is still the canonical implementation thread, but the branch is conflicting and has correctness blockers before it can merge.

## What This Changes

The PR extends prompt-build hooks with tool and skill catalogs plus `toolsAllow`/`skillsAllow` outputs, then applies those allowlists during embedded agent prompt/session assembly.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes complete model-visible tool and skill catalogs, then applies allowlists through the canonical prompt/session assembly path while preserving transforms, snapshots, and hook mutations.

## Reproduction Assessment

Yes for the PR defects as source-level checks: current main lacks the proposed fields, and the PR head shows type-list, catalog, snapshot, and prompt rebuild paths that would fail or regress. This is not a current-main bug reproduction.

## Solution Assessment

No, not as written. The hook seam is plausible, but the branch needs a rebase and should feed allowlists into the existing `buildAttemptSystemPrompt` and session-tool pipeline with explicit public semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep appendContext in mutation fields:** `src/plugins/hook-before-agent-start.types.ts:92-93`
  - body: While adding `toolsAllow` and `skillsAllow`, this list drops `appendContext` even though `PluginHookBeforePromptBuildResult` still declares it. The completeness assertion below resolves to `never`, so typechecking fails; if bypassed, legacy hook stripping would also leave `appendContext` in the model-resolve result.
  - confidence: 0.93
- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are added separately into `allCustomTools`. A plugin that mirrors this catalog cannot keep client tools in its allowlist, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs this reloads current workspace skill entries from disk and filters those, instead of filtering the captured `skillsSnapshot`. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
