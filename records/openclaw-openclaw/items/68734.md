## Summary

Keep this PR open but not merge-ready: the dynamic narrowing capability is useful and has live-log proof, but the branch changes a public plugin hook contract, overlaps an open alternate tool-filtering API, and still has source-backed prompt-pipeline and hosted-tool correctness issues.

## What This Changes

The PR adds optional `toolsAllow` and `skillsAllow` results plus `availableTools` and `availableSkills` inputs to `before_prompt_build`, wiring them into embedded agent tool selection and skill prompt rebuilding.

## Best Possible Solution

Land one maintainer-approved dynamic prompt-surface contract that preserves the normal prompt pipeline, exposes the complete model-visible tool surface, and has regression coverage for hook prompt mutations plus hosted/client tools.

Yes for the review findings: source inspection of the PR head shows the `skillsAllow` rebuild replacing the already-mutated prompt, and current-main tool setup shows client-hosted tools are added outside the `effectiveTools` list exposed to hooks.

## Solution Assessment

No. The feature direction is promising, but the implementation should preserve the existing prompt transformation pipeline, expose the complete tool catalog, and wait for a maintainer decision on the competing hook API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this rebuild path constructs a fresh system prompt and applies it directly, after any `systemPrompt`, `prependSystemContext`, or `appendSystemContext` from the hook was already applied. It also skips the normal provider transform path, so a plugin that narrows skills can silently lose its own prompt guidance or provider-specific system text for that turn.
  - confidence: 0.9
- **[P2] Include client-hosted tools in hook allowlists:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: The hook event exposes `availableTools` from `effectiveTools`, but client-hosted tools are registered separately through `clientToolDefs` and `allCustomTools`. A plugin deriving `toolsAllow` from this event cannot include those names, and the later `setActiveToolsByName(...)` call can remove them from the session.
  - confidence: 0.84
