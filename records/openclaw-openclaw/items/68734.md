## Summary

Keep open: current main still lacks the hook catalog/allowlist API and real-world discussion supports the need, but the PR head has prompt/session correctness issues that should be fixed before merge.

## What This Changes

The PR adds `before_prompt_build` tool and skill catalog allowlists, applies hook-derived `toolsAllow` in the attempt phase, and rebuilds the skills prompt when `skillsAllow` is returned.

## Best Possible Solution

Land a rebased, documented hook implementation that exposes the complete model-visible catalog and applies allowlists through canonical prompt/session assembly while preserving provider transforms, hook mutations, policy, and captured skill snapshots.

## Reproduction Assessment

Yes, as a source-level PR-defect check rather than a current-main bug reproduction. Current main lacks the proposed hook fields, and the PR head shows catalog, prompt rebuild, and snapshot paths that would misbehave.

## Solution Assessment

No, not as written. The hook seam is plausible, but the implementation should feed allowlists into the existing prompt/session pipeline before assembly instead of patching state afterward.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is built from `effectiveTools` only, but client-hosted tools are added separately into the model-facing custom tools. A plugin that mirrors this catalog cannot keep those tools in its allowlist, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Feed tool allowlists into prompt assembly:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` runs after the system prompt and provider transform were built from the pre-filtered tool surface. A tools-only allowlist can leave prompt/provider state advertising removed tools, so the allowlist needs to feed the normal prompt assembly path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from the captured snapshot:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skill entries from disk and rebuilds without the captured snapshot. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
