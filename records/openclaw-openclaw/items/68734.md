## Summary

Keep open. Current main still lacks this hook catalog/allowlist capability, and the PR addresses a real token-pressure problem, but the current branch is conflicting and still has prompt/session assembly defects that should block merge.

## What This Changes

The PR extends `before_prompt_build` so plugins can receive available tool and skill names and return per-turn `toolsAllow`/`skillsAllow` allowlists.

## Best Possible Solution

Rebase and implement allowlists through the canonical prompt/session assembly path, exposing the complete model-visible catalog while preserving provider transforms, hook mutations, policy, and captured skill snapshots.

## Reproduction Assessment

Yes, as a source-level PR review path rather than a current-main bug reproduction. Current main lacks the proposed hook fields, and the PR head shows catalog, prompt rebuild, and snapshot paths that would misbehave.

## Solution Assessment

No, not as written. The hook seam is plausible, but this branch applies allowlists after prompt/session assembly and rebuilds skill prompts outside the existing transformed path; the safer path is to feed the narrowed catalogs into canonical assembly before the model call.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include client-hosted tools in hook catalogs:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is built from `effectiveTools` only, but client-hosted tools are added separately into the model-facing custom tools. A plugin that mirrors this catalog cannot keep those tools in its allowlist, so hook narrowing can remove tools it never saw.
  - confidence: 0.88
- **[P2] Apply tool allowlists before prompt assembly:** `src/agents/pi-embedded-runner/run/attempt.ts:1891`
  - body: `setActiveToolsByName()` runs after the system prompt and provider transform were built from the pre-filtered tool surface. A tools-only allowlist can leave prompt, cache, or provider state advertising removed tools, so the allowlist needs to feed the normal prompt/session assembly path.
  - confidence: 0.84
- **[P2] Filter snapshot skills from captured snapshots:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this reloads current workspace skill entries from disk and rebuilds without the captured snapshot. Snapshot runs should stay tied to captured skill content, otherwise filtering can inject newer skills or drop snapshot-only entries.
  - confidence: 0.84
- **[P2] Preserve hook prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1950-1952`
  - body: When `skillsAllow` is set, this rebuilds from the original append-prompt params and applies it directly. That can drop earlier `systemPrompt` or prepend/append system-context hook mutations and bypass the provider transform used for the normal prompt.
  - confidence: 0.9
