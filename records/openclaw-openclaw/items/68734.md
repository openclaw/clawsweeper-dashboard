## What This Changes

The PR adds `availableTools`/`availableSkills` and `toolsAllow`/`skillsAllow` to `before_prompt_build`, then applies per-turn tool and skill prompt filtering in the embedded runner with focused tests.

## Best Possible Solution

Land a rebased, maintainer-approved dynamic prompt-surface hook that uses a complete policy-filtered tool catalog, filters skills from the run's captured skill source, and composes through `buildAttemptSystemPrompt` and provider transforms.

Overall confidence: 0.84

Full review comments:

- **[P2] Include client tools in the hook-visible catalog:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is derived from `effectiveTools`, but client-hosted tools are converted into `allCustomTools` separately. A plugin that returns a `toolsAllow` list based on this event has no way to keep a required client tool, and `setActiveToolsByName(...)` can remove it from the active session for the turn.
  - confidence: 0.86
- **[P2] Preserve prompt transforms when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When `skillsAllow` is set, this rebuilds `systemPromptText` from `buildEmbeddedSystemPrompt(...)` and applies it directly. That can drop the hook `systemPrompt`/prepend/append mutations applied above and skip the provider transform path, so the model can receive a different system prompt only on skill-filtered turns.
  - confidence: 0.88
- **[P2] Filter skill snapshots without reloading live workspace skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: The snapshot path reloads skills from the workspace instead of filtering the captured run snapshot. Snapshot-backed or sandboxed runs can therefore rebuild the prompt from newer or different skill data than the rest of the turn is using.
  - confidence: 0.82
