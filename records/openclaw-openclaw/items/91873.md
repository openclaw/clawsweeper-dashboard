## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no visible failure, interrupting real user messaging.
- `impact:message-loss`: When the Slack channel plugin does not load, Slack messages and replies can be silently missed until the operator notices the channel is down.
- `impact:other`: The concrete affected surface is plugin startup and upgrade compatibility for an external channel, which is not covered by the narrower impact labels.

## Summary

Keep open: current main still discovers installed external plugins but does not start a configured non-bundled Slack channel unless the owning plugin is explicitly trusted, so the reported latest-release upgrade break is not fixed on main. The right next step needs maintainer judgment because it sits between upgrade compatibility, operator-visible startup warnings, and plugin trust policy.

## What This Changes

A Slack upgrade report says an npm-installed configured Slack channel stops loading after v2026.6.5 unless `plugins.allow` includes `slack`.

## Best Possible Solution

Preserve already installed configured Slack channels across upgrade through a narrow trusted-install migration or compatibility path, and surface blocked configured channels clearly in startup/status or doctor output.

## Reproduction Assessment

Yes from source inspection: installed/global channel owners are discovered, but current startup only starts non-bundled configured channel plugins when activation is explicitly enabled. I did not run a live Slack upgrade repro.

## Solution Assessment

No: the manual `plugins.allow` workaround proves the trust gate, but it is not the best product fix for existing upgrades. The maintainable fix should be in plugin startup, doctor/update migration, or operator-visible warning policy.
