## Summary

Keep open: current main and the latest release still require a configured non-bundled Slack channel owner to be explicitly trusted before startup loads it, so the reported upgrade regression is not implemented or superseded. The linked open PR is not a viable fix because it changes unrelated release-secret automation rather than Slack/plugin startup.

## What This Changes

The issue reports a Slack Socket Mode channel disappearing after upgrade because an npm-installed Slack plugin no longer starts without `plugins.allow` or another explicit trust entry.

## Best Possible Solution

Preserve already installed configured Slack channels across upgrade through a narrow trusted-install migration or compatibility path, and surface blocked configured channels clearly in startup/status or doctor output.

## Reproduction Assessment

Yes from source: current main and `v2026.6.6` require a non-bundled configured channel plugin to be explicitly enabled or allowlisted before startup loads it. I did not run a live Slack package-upgrade repro.

## Solution Assessment

No: the manual `plugins.allow` workaround proves the trust gate, but it is not the best product fix for existing upgrades. The maintainable fix needs a maintainer choice between a trusted installed-channel migration/compatibility path and loud warning/documented breaking-change behavior.
