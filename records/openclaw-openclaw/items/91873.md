## Label Justifications

- `impact:message-loss`: When Slack does not load, inbound Slack messages and replies can be missed until the operator notices the channel is down.

## Summary

Keep open: current main and the latest release still require explicit trust for configured non-bundled/global channel plugins, and the only direct fix attempts for this Slack upgrade regression are closed unmerged PRs.

## What This Changes

This issue asks OpenClaw to preserve Slack/npm-installed plugin loading across the 2026.4.26 to 2026.6.5 upgrade, or surface/document the required `plugins.allow`/explicit trust step loudly.

## Best Possible Solution

Keep the fail-closed plugin trust boundary only if upgrade, doctor, startup, or status paths make blocked configured external channels impossible to miss; otherwise add a bounded compatibility migration or activation path for upgraded configured channels.

## Reproduction Assessment

Yes. Source inspection shows a configured global/non-bundled channel plugin without explicit trust is blocked from startup, and the startup logger does not surface that blocker; I did not run a live Slack upgrade.

## Solution Assessment

Unclear on exact fix shape. Automatic re-enablement would address the compatibility regression but needs security/product approval; a migration plus startup/status warning is the safer path if fail-closed trust remains intentional.
