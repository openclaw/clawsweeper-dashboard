## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no visible failure, interrupting real user messaging.
- `impact:message-loss`: When the Slack channel plugin does not load, Slack messages and replies can be silently missed until the operator notices the channel is down.
- `impact:other`: The issue also concerns plugin startup and upgrade compatibility for an external channel, which is outside the narrower impact labels.

## Summary

Keep open: current main and v2026.6.5 still require explicit trust for non-bundled/global configured channel owners, and the repair/warning path does not cover an installed Slack owner blocked by policy. The central upgrade break remains and needs maintainer judgment between upgrade compatibility and plugin trust policy.

## What This Changes

The issue requests preserving Slack plugin startup across the 2026.4.26 to 2026.6.5 upgrade or surfacing a clear migration/warning when configured Slack cannot load.

## Best Possible Solution

Preserve configured installed Slack channels on upgrade through a trusted-install migration or compatibility path, and make blocked configured channels visible in startup/status or doctor without widening untrusted plugin activation.

## Reproduction Assessment

Yes from source inspection: installed/global channel owners are discovered, but configured-channel startup and presence policy require explicit trust before loading them. I did not run a live Slack upgrade repro.

## Solution Assessment

Unclear until maintainers choose the trust-policy direction. The manual plugins.allow workaround identifies the gate, but the best product fix should be a narrow upgrade migration/compat path or loud warning/status/doctor behavior.
