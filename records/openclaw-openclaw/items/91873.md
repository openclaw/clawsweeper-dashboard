## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no visible failure, interrupting real messaging workflows.
- `impact:message-loss`: When the Slack plugin does not load, inbound Slack messages and outbound replies can be silently missed until the operator notices the channel is down.
- `impact:other`: The concrete affected surface includes plugin startup and upgrade compatibility for external installed channels, which is meaningful outside the narrower impact labels.

## Summary

Keep open: current main still discovers installed external plugins, but configured non-bundled channel owners such as Slack still need explicit plugin trust before startup will load them. The same failure shape is present in v2026.6.5, and the relevant fix/warning work is still open in unmerged PRs.

## What This Changes

The issue asks OpenClaw to preserve Slack external-plugin startup across upgrade without requiring manual `plugins.allow`, or document/migrate the break and warn loudly when a configured channel is not loaded.

## Best Possible Solution

Land a maintainer-approved upgrade-safe policy: either treat existing configured installed Slack as trusted through a narrow migration/startup rule, or preserve explicit trust while surfacing a clear startup/status/doctor warning and migration step.

## Reproduction Assessment

Yes from source inspection: installed/global channel owners are discovered, but current startup only starts non-bundled configured channel plugins when they are explicitly trusted. I did not run a live Slack upgrade repro.

## Solution Assessment

No: adding `plugins.allow` is a valid workaround, but it is not the best product fix for existing upgrades. The maintainable fix needs a startup trust decision plus operator-visible migration or warning behavior.
