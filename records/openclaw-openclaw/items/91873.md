## Summary

Keep open: current main still has a source-visible path where a configured installed Slack channel can be skipped unless the external plugin is explicitly trusted, and the related diagnostic work is open and narrower than this Slack upgrade report.

## What This Changes

The issue asks OpenClaw to preserve Slack channel startup across the 2026.4.26 to 2026.6.5 upgrade or surface a loud migration warning when installed Slack is skipped without plugins.allow.

## Best Possible Solution

Define the generic configured official external channel policy: either migrate or activate previously working installed Slack safely, or keep explicit trust and make doctor, gateway startup, or channel status loudly tell the operator which plugins.allow or plugins.entries repair is required.

## Reproduction Assessment

Yes, source-level: current main skips configured non-bundled channel plugins unless they are explicitly enabled, and plugins.allow is an explicit enablement source. I did not run the full live Slack upgrade path, so this is not a high-confidence live reproduction.

## Solution Assessment

Unclear: a Slack-only bypass would be too narrow, and unconditional auto-loading may weaken the external plugin trust boundary. The best fix should be a generic external-channel migration or diagnostic decision that covers Slack and the related Discord pattern.
