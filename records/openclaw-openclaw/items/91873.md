## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no operator-visible startup failure, interrupting real messaging workflows.
- `impact:message-loss`: When the Slack channel plugin does not load, inbound Slack messages and replies can be missed until the operator notices the channel is down.
- `impact:other`: The issue also affects plugin startup and upgrade compatibility for external channel plugins, which is a concrete maintainer-visible surface outside the narrower owned impact labels.

## Summary

Keep open: current main and v2026.6.8 still require explicit trust for global/config external channel owners, gateway startup does not emit a configured-channel blocker warning, and every linked fix PR is now closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve Slack channel loading after upgrade without a manual `plugins.allow` repair, or to loudly warn/document the breaking migration when a configured channel is blocked.

## Best Possible Solution

Choose a maintainer-approved compatibility or migration path so configured installed Slack channels are either preserved after upgrade or surfaced loudly in gateway startup/status/doctor before messages are missed.

## Reproduction Assessment

Yes, source-reproducible: current main filters untrusted global/config channel owners until explicit trust is present, while the startup log does not warn about blocked configured channels. I did not run a live Slack upgrade reproduction.

## Solution Assessment

No current implementation fully solves it. The best fix needs maintainer choice between a narrow upgrade-compatible trust/migration path and a warning/status path that makes blocked configured channels visible before message loss.
