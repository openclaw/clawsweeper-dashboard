## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no visible failure, interrupting real user messaging.
- `impact:message-loss`: When the Slack channel plugin does not load, Slack messages and replies can be silently missed until the operator notices the channel is down.
- `impact:other`: The concrete affected surface is plugin startup and upgrade compatibility for an external channel, which is not covered by the narrower impact labels.

## Summary

Keep open: current main and the latest release still gate non-bundled/global configured channel owners on explicit trust, and the only open linked PR does not touch the Slack/plugin-loading path.

## What This Changes

The issue reports that upgrading from 2026.4.26 to 2026.6.5 stopped loading an npm-installed Slack channel unless `plugins.allow` explicitly included `slack`.

## Best Possible Solution

Preserve configured installed Slack channels across upgrades through a narrow trusted-install migration or compatibility path, and surface blocked configured channels in gateway startup/status or doctor before users miss messages.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.6 both filter untrusted global/config channel owners unless they are explicitly trusted, and the tests assert the same behavior. I did not run a live Slack upgrade reproduction.

## Solution Assessment

No; the manual `plugins.allow` workaround proves the trust gate but is not the best product fix for an upgrade regression. Maintainers should choose a narrow upgrade-compatible trust, migration, or warning path instead of requiring users to discover a silent channel outage.
