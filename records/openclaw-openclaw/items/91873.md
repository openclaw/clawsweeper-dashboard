## Label Justifications

- `P1`: A latest-release upgrade can leave a configured Slack Socket Mode channel unloaded with no visible startup failure, interrupting real user messaging.
- `impact:message-loss`: When the Slack channel plugin does not load, inbound Slack messages and replies can be missed until the operator notices the channel is down.
- `impact:other`: The issue also affects external plugin startup and upgrade compatibility, which is a concrete maintainer-visible impact outside the narrower message-loss label.

## Summary

Keep open: current main and v2026.6.5 still require explicit trust for a non-bundled installed Slack channel plugin, so a configured Slack channel can remain unloaded without `plugins.allow` or `plugins.entries.slack.enabled=true`. The remaining fix needs maintainer judgment because it crosses upgrade compatibility, operator-visible warnings, and external plugin trust policy.

## What This Changes

Restore upgrade-safe Slack channel loading or add migration and loud warnings when a configured installed Slack plugin is blocked by the new `plugins.allow` trust gate.

## Best Possible Solution

Preserve configured installed official Slack channels across upgrade with a maintainer-approved trusted-install migration or compatibility path, and make startup/status/doctor clearly warn when a configured channel owner is blocked by plugin trust policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: with `channels.slack` configured and an installed global Slack plugin but no explicit trust marker, current startup filters the non-bundled owner; I did not run a live Slack upgrade.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the manual `plugins.allow` workaround proves the gate, but the durable fix should be a maintainer-approved migration or compatibility path plus operator-visible warnings rather than an unconditional return to implicit external-plugin loading.
