## Label Justifications

- `P1`: A released upgrade can stop an established Slack channel without an explicit startup failure, directly disrupting a real channel workflow.
- `impact:message-loss`: Slack messages are neither received nor delivered when the configured plugin is omitted from gateway startup.
- `impact:other`: The gateway can report ready while silently excluding a configured production channel, creating a significant operator-observability failure.

## Summary

Keep open: v2026.6.5 and current main both require explicit trust for global/config-origin channel plugins, so a previously working Slack installation can remain discovered but inactive after upgrade. Existing upgrade repair handles missing official plugin installations, but it does not resolve the trust migration or loud startup-warning behavior reported here.

## What This Changes

Preserve an existing Slack channel across upgrade when its plugin was previously installed without `plugins.allow`, and clearly warn when a configured channel is blocked from starting.

## Best Possible Solution

Add a narrow doctor/update migration that converts verifiable legacy managed Slack installation provenance into the current explicit trust shape, and make gateway startup/status clearly name every configured channel blocked by missing or untrusted plugin ownership without trusting arbitrary global code.

## Reproduction Assessment

Yes, at source level: v2026.6.5 and current main require explicit trust for global/config-origin channel owners, matching the reported `plugins.allow` workaround. A live Slack upgrade was not run, but release-to-main source parity makes the reported path high confidence.

## Solution Assessment

No, requiring affected users to discover and manually add `plugins.allow` is not the best solution. The narrow maintainable fix is a provenance-aware upgrade migration plus explicit blocked-channel startup diagnostics, while retaining the fail-closed policy for arbitrary plugins.
