## Label Justifications

- `P1`: An upgrade can leave a configured Slack Socket Mode channel unloaded with no obvious gateway failure, interrupting real user messaging.
- `impact:message-loss`: When the Slack channel plugin does not load, Slack messages and replies can be missed until the operator notices the channel is down.
- `impact:other`: The issue also concerns plugin startup and upgrade compatibility for an external channel, which is a maintainer-visible impact outside the narrower labels.

## Summary

Keep open: current main and v2026.6.6 still require explicit trust before an installed non-bundled Slack channel plugin is included in gateway startup, so the reported `plugins.allow` workaround remains the effective path. Doctor diagnostics partially identify the blocker, but the requested startup-time warning/upgrade compatibility is not fixed, and linked PR #92723 is unrelated to the Slack/plugin-loading path.

## What This Changes

The issue asks OpenClaw to preserve Slack channel loading across upgrade without manually adding `plugins.allow`, or provide a clear migration and loud warning when a configured channel plugin is blocked.

## Best Possible Solution

Maintainers should choose the upgrade contract: either migrate or explicitly trust previously installed official Slack channel plugins with startup/status warnings for blocked configured owners, or document the breaking trust requirement with a first-class migration path.

## Reproduction Assessment

Yes from source inspection: Slack is startup-lazy, and the gateway planner starts non-bundled configured channel plugins only when they are explicitly enabled or trusted. I did not run a live Slack Socket Mode upgrade because that requires private workspace credentials.

## Solution Assessment

No, this is not solved yet. The manual `plugins.allow` workaround proves the trust gate, but the best permanent fix needs a maintainer decision between compatibility activation, migration/doctor repair, and operator-visible warnings.
