## Label Justifications

- `impact:message-loss`: Attachments can be admitted or downloaded by one path and then rejected or left unavailable during sandbox staging.

## Summary

Keep open: current main and the latest release still enforce the fixed 5 MiB media-store and sandbox-staging cap, and the focused implementation PRs are closed unmerged. The remaining work is real, but it needs a maintainer decision on the supported config contract, upper bound, and precedence across store, staging, chat admission, and channel/tool limits.

## What This Changes

Request to make the fixed 5 MiB sandbox media staging/store limit configurable so larger attachments can reach sandboxed agents.

## Best Possible Solution

Define a bounded top-level media staging/store limit that keeps the 5 MiB default, documents precedence against narrower limits, and wires the same resolved value through media store defaults, sandbox staging, chat.send preflight, docs, schema help, and regression tests.

## Reproduction Assessment

Yes. Current source still aliases sandbox staging to the fixed 5 MiB media-store cap, and gateway tests cover a 6 MiB attachment that is rejected as over the sandbox staging limit; this read-only review did not run a live attachment transfer.

## Solution Assessment

Unclear as proposed. A configurable cap is a plausible end state, but maintainers need to choose the canonical schema, safe maximum, and precedence rather than reusing tools.media.* or adding an isolated mutable process-global setting.
