## Summary

Keep open. Current main still has a hardcoded 5 MB media-store and sandbox staging cap, the top-level media config has no shared max-bytes field, and open PR #40950 is the unmerged closing implementation path.

## What This Changes

The issue requests a user-configurable sandbox/media staging byte limit instead of the current hardcoded 5 MB `MEDIA_MAX_BYTES` cap.

## Best Possible Solution

Land PR #40950 or an equivalent narrow fix that adds a bounded `media.maxBytes` defaulting to 5 MB, threads the effective value through store, sandbox staging, chat preflight, schema/help/docs/tests, and leaves broader image-limit work to #67031.

## Reproduction Assessment

Yes. A sandboxed non-image attachment larger than 5 MB follows current-main code through `MEDIA_MAX_BYTES` in store, staging, and chat preflight, and existing tests assert the same rejection behavior.

## Solution Assessment

Yes for keeping this open. The narrow maintainable path is to review and land the linked implementation or an equivalent targeted patch; closing this issue before that PR merges would drop the active regression tracker.
