## Summary

Keep open: current main and the latest release still enforce a fixed 5 MiB media-store and sandbox-staging cap, and `media.maxBytes` is not part of the strict config schema. The merged managed-PDF fallback only covers a narrow PDF pass-through case, while the requested general configurable cap still needs maintainer product and security review.

## What This Changes

The issue requests making the `MEDIA_MAX_BYTES` media-store and sandbox-staging cap user-configurable instead of fixed at 5 MiB.

## Best Possible Solution

Define a bounded, documented media-store and sandbox-staging limit contract, preserve secure defaults, thread the resolved cap through store/staging/chat preflight and relevant callers, and add focused upgrade/docs/tests before closing this issue.

## Reproduction Assessment

Yes at source level: current main still binds media store, sandbox staging, and chat preflight to the fixed 5 MiB cap, and tests assert oversize staging/rejection behavior. I did not run a live gateway/container transfer in this read-only review.

Unclear. Configurability is a plausible solution, but the best implementation depends on a maintainer-approved config scope, hard maximum, and precedence across existing media limit settings.
