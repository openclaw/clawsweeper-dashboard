## Summary

Current main and release v2026.6.5 still enforce the fixed 5 MiB media-store cap during sandbox staging, so the reported workflow remains unresolved. The focused implementation PRs closed unmerged, and their discussion leaves this issue as the canonical place for a maintainer decision on a bounded configuration contract and limit precedence.

## What This Changes

Allow operators to raise the fixed 5 MiB media-store and sandbox-staging cap through supported configuration.

## Best Possible Solution

Add one bounded, lifecycle-owned staging/store limit that preserves the 5 MiB default, defines precedence against narrower limits, and reports rejection consistently across chat admission and actual sandbox staging.

## Reproduction Assessment

Yes, from current source with high confidence: sandbox staging passes a fixed 5 MiB limit into both safe-copy paths and explicitly documents the mismatch with the larger chat admission cap. This read-only review did not run a live attachment transfer.

## Solution Assessment

Unclear as proposed: operator configurability is a reasonable end state, but maintainers must choose the canonical owner, maximum bound, and precedence rather than adding an isolated tools.media or mutable process-global setting.
