## Summary

Keep open: current main and v2026.5.20 still hard-code the 5 MiB media-store, sandbox staging, and chat preflight caps, and the prior implementation PR was closed unmerged as superseded by this canonical issue.

## What This Changes

The issue asks for the sandbox/media staging 5 MiB `MEDIA_MAX_BYTES` cap to become configurable through OpenClaw configuration.

## Best Possible Solution

Define a bounded, documented media staging max-byte contract and thread it consistently through media store, sandbox staging, chat preflight, schema/types/help, docs, and regression coverage while preserving the 5 MiB default unless configured.

## Reproduction Assessment

Yes. Source inspection on current main shows fixed 5 MiB store, sandbox staging, and chat preflight caps, and existing gateway coverage asserts that a 6 MiB sandboxed PDF is rejected; I did not run a live gateway flow in this read-only review.

## Solution Assessment

Unclear. A bounded configurable cap is a plausible direction, but the public schema, safety ceiling, and precedence with existing media-understanding and channel limits need maintainer approval before implementation.
