## Summary

Keep this open: current main and the latest release still enforce the fixed 5 MiB media-store/sandbox staging cap, the config schema still has no `media.maxBytes`, and the explicit closing implementation remains the open, unmerged pull request at https://github.com/openclaw/openclaw/pull/40950.

## What This Changes

The issue requests a user-configurable sandbox/media-store byte limit instead of the hardcoded 5 MiB `MEDIA_MAX_BYTES` cap.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/40950 or a narrow replacement that adds a bounded schema-backed media staging limit and threads it through store, sandbox staging, gateway preflight, help/docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the fixed 5 MiB store, sandbox staging, and chat preflight caps, and gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

Unclear. A bounded `media.maxBytes` setting is plausible, but the schema shape, safety ceiling, and precedence with existing channel and media-understanding caps need maintainer review before this becomes the final contract.
