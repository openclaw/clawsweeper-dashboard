## Summary

Keep this open: current main and the latest release still enforce the fixed 5 MiB media-store/sandbox staging cap, the config contract still has no top-level `media.maxBytes`, and the explicit closing implementation remains the open, unmerged pull request at https://github.com/openclaw/openclaw/pull/40950.

## What This Changes

The issue asks to make the 5 MiB media store and sandbox staging limit user-configurable instead of hardcoded.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/40950 or a narrow replacement that adds a bounded schema-backed staging limit, wires it through store, sandbox staging, gateway preflight, docs/help, and regression coverage, and preserves stricter caller-specific caps.

## Reproduction Assessment

Yes. Source inspection on current main shows the fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

Unclear. A bounded `media.maxBytes` setting is plausible, but the final schema, safety ceiling, and precedence with existing channel and media-understanding caps need maintainer review before this becomes the contract.
