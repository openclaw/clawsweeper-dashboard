## Summary

Keep open: current main and the latest release still use a fixed 5 MiB media-store and sandbox staging cap, while the existing `tools.media.*.maxBytes` settings are media-understanding limits rather than a sandbox/store cap. The issue remains valid, but the requested fix adds a new config/resource-control surface that needs maintainer product and security approval before implementation.

## What This Changes

The issue asks for the sandbox/media-store staging byte cap to be configurable instead of fixed at 5 MiB.

## Best Possible Solution

Define one bounded canonical media-store/sandbox staging limit with safe defaults, explicit validation, schema/help/docs coverage, and tests that keep media-understanding limits distinct.

Yes, source-level: a sandboxed attachment between 5 MiB and the 20 MiB chat attachment parse cap is admitted by parsing but rejected or skipped by the `MEDIA_MAX_BYTES` staging path. I did not run live sandbox E2E in this read-only review.

Unclear. Configurability is a plausible fix, but the best implementation depends on a maintainer-approved config scope, hard maximum, and precedence across existing media limit settings.
