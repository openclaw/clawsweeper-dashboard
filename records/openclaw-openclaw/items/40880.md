## Summary

Keep open. Current main still hardcodes the 5 MB media-store and sandbox staging cap, the strict top-level media config still has no shared max-bytes field, and open PR #40950 is already the unmerged closing implementation path for this issue.

## What This Changes

The issue asks for the hardcoded 5 MB sandbox/media staging limit to become user-configurable, such as via a top-level media max-bytes config field.

## Best Possible Solution

Land PR #40950 or an equivalent narrow fix that adds a bounded `media.maxBytes` config with the existing 5 MB default, threads the effective value through media storage, sandbox staging, and chat preflight, and updates schema/help/docs/tests without folding in the broader image-limit work tracked by #67031.

## Reproduction Assessment

Yes. A sandboxed non-image attachment larger than 5 MB follows a visible current-main path through `MEDIA_MAX_BYTES` in media storage, sandbox staging, and chat preflight, and existing tests assert the same rejection behavior.

## Solution Assessment

Yes for keeping this open. The narrow maintainable path is review and landing of #40950 or an equivalent implementation; closing this before that PR merges would drop the active regression tracker.
