## Summary

Keep open: current main and v2026.6.6 still use a fixed 5 MiB media-store/sandbox staging cap, and the existing `tools.media.*.maxBytes` fields control media-understanding limits rather than the sandbox/media-store cap. The report remains valid, but adding configurability needs a maintainer product/security decision on config shape, precedence, and resource bounds.

## What This Changes

The issue requests making the sandbox media staging byte limit configurable instead of fixed at 5 MiB.

## Best Possible Solution

Define a bounded canonical media-store/sandbox staging limit config, keep the safe default, thread it through media persistence and sandbox prechecks, and update tests/docs so existing media-understanding limits remain distinct.

## Reproduction Assessment

Yes, source-level: a sandboxed attachment between 5 MiB and the 20 MiB chat parse cap is admitted by attachment parsing but blocked by the `MEDIA_MAX_BYTES` sandbox precheck/staging path. I did not run live sandbox E2E in this read-only review.

## Solution Assessment

Unclear. Configurability is likely the right direction, but the best fix must choose the canonical config scope, precedence, and hard maximum before changing this security/resource boundary.
