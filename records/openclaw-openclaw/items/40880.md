## Summary

Keep open. Current main still hardcodes the 5 MB media store, sandbox staging, and chat preflight cap; `media.maxBytes` is not accepted by config, and PR #40950 remains open as the unmerged closing path.

## What This Changes

The issue requests a user-configurable media staging/store byte cap for sandboxed attachments instead of the current 5 MB constant.

## Best Possible Solution

Review and land PR #40950, or an equivalent narrow replacement, so the cap is schema-backed, documented, bounded, and threaded through store, staging, chat preflight, and tests.

## Reproduction Assessment

Yes for the current limit behavior. Source inspection shows a sandboxed non-image attachment larger than 5 MB follows the fixed `MEDIA_MAX_BYTES` preflight/staging path, and current tests assert that rejection.

## Solution Assessment

Yes for keeping this issue open. Because PR #40950 is open and explicitly linked as the implementation path, the maintainable solution is to review and land that PR or a narrow equivalent before closing the report.
