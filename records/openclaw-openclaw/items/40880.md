## Summary

Keep this open because current main and the latest release still preserve the fixed 5 MiB media-store and sandbox-staging cap, while the specific closing implementation remains the open, unmerged pull request at https://github.com/openclaw/openclaw/pull/40950.

## What This Changes

The issue asks for the 5 MiB sandbox/media staging cap to become configurable through `openclaw.json` instead of requiring patched build output.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/40950 or a narrow replacement that adds a schema-backed media staging limit and threads it through store, sandbox staging, gateway preflight, help/docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

Unclear for the final implementation. A bounded `media.maxBytes` setting is plausible, but schema shape, ceiling, and precedence with existing channel and media-understanding caps need maintainer review before landing.
