## Summary

Keep open. Current main still hardcodes the 5 MB media store, sandbox staging, and chat preflight cap, `media.maxBytes` is not accepted by config, and #40950 remains the open unmerged closing implementation path.

## What This Changes

The issue asks for the hardcoded 5 MB sandbox/media staging cap to be configurable through OpenClaw config.

## Best Possible Solution

Land #40950, or an equivalent narrow replacement, so the limit is schema-backed, documented, bounded, and threaded through store, sandbox staging, chat preflight, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a sandboxed non-image attachment larger than 5 MB follows the fixed `MEDIA_MAX_BYTES` preflight and staging path, and a current test asserts that rejection with a 6 MB PDF.

## Solution Assessment

Yes for keeping this issue open. Because #40950 is open and explicitly linked as the implementation path, the maintainable solution is to review and land that PR or a narrow equivalent before closing the report.
