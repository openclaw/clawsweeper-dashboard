## Label Justifications

- `P1`: The reported behavior can terminate an active CLI-backed agent session during a supported long-running workflow and lose in-flight session state.
- `impact:session-state`: The issue is about a live agent session being killed before it can receive the Monitor completion event and continue.
- `impact:crash-loop`: The affected path forcibly terminates the CLI subprocess through the no-output watchdog and surfaces a timeout failure.

## Summary

Keep open: current main and v2026.5.19 still enforce CLI no-output watchdogs from CLI stdout/stderr activity with no pending Monitor or expected-idle signal, so the reported session-loss path remains source-proven.

## What This Changes

The issue asks OpenClaw to stop the CLI no-output watchdog from killing sessions that are intentionally idle while a Monitor task is pending.

## Best Possible Solution

Model verified pending Monitor/tool waits as bounded expected idle that pauses or heartbeats the no-output watchdog only while tool work is actually pending, while preserving the overall timeout and normal no-output protection.

## Reproduction Assessment

Yes, at source level. The issue gives concrete Monitor steps and matching watchdog logs, and current main still times silent CLI turns from stdout/stderr activity rather than pending Monitor state; I did not run the live 180-second wait.

## Solution Assessment

Unclear. Treating verified pending Monitor waits as expected idle is the right direction, but raising constants or disabling the watchdog broadly is only a mitigation and the maintainable liveness contract needs owner choice.
