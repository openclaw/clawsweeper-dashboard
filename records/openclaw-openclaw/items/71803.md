## Summary

Keep open. Current main still applies the CLI no-output watchdog without any pending Monitor/tool expected-idle signal, so a healthy but silent Claude CLI turn can still be terminated. The related timeout-cap PR only lengthens the window for #40982 and does not solve this Monitor-specific reliability bug.

## What This Changes

The issue asks OpenClaw to stop treating a silent CLI session as stalled while it is intentionally waiting for a long-running Monitor task.

## Best Possible Solution

Add an explicit expected-idle/proof-of-life contract for pending Monitor or MCP/client-tool waits in the CLI runner path, while preserving the overall request timeout and keeping the no-output watchdog active for CLIs that are truly stalled.

## Reproduction Assessment

Yes. The issue provides concrete Monitor reproduction steps and watchdog logs, and current main source shows silent CLI processes still hit the no-output watchdog without a pending Monitor/tool exemption.

## Solution Assessment

No current proposed fix is sufficient. Raising timeout constants may reduce false kills, but the maintainable fix is to distinguish verified pending tool work from a stalled CLI process.
