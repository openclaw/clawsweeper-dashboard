## Summary

Keep open. Current main still applies the CLI no-output watchdog to silent Claude CLI turns with no Monitor-aware expected-idle signal, and the related open timeout-cap PR only lengthens the window rather than fixing healthy pending Monitor/tool waits.

## What This Changes

The issue requests Monitor-aware CLI watchdog behavior so a healthy Claude CLI session is not terminated while it is idle waiting for long-running monitored work.

## Best Possible Solution

Add an explicit expected-idle/proof-of-life contract for pending Monitor, MCP, or client-tool waits in the CLI runner path, while preserving the overall request timeout and the no-output watchdog for CLIs that are truly stalled.

## Reproduction Assessment

Yes. The issue gives a concrete Monitor wait command plus matching watchdog logs, and current main source shows silent CLI processes still hit the no-output watchdog without a pending Monitor/tool exemption.

## Solution Assessment

Yes for the issue's Monitor-aware expected-idle direction; no for a timeout-only fix. Raising constants is a mitigation, but the narrow maintainable solution is to distinguish verified pending tool work from a stalled CLI process.
