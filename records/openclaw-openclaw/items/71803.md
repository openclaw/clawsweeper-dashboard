## Label Justifications

- `P1`: The issue describes a broken CLI-backed agent workflow that kills real long-running sessions and loses in-flight task context.
- `impact:session-state`: The reported failure destroys the active agent session before it can consume the Monitor completion and continue the transcript.
- `impact:crash-loop`: The affected code path terminates the CLI process on a no-output timeout even though the supervised Monitor task may still be healthy.

## Summary

Keep open: current main and the latest shipped release still enforce CLI no-output watchdogs from process or Claude live-session output, with no pending-Monitor or expected-idle signal, so the reported session-loss path remains source-proven. Related open PRs mitigate timeout caps, closed stdin, or orphan output, but none is merged and none fully resolves the Monitor wait contract.

## What This Changes

The issue asks OpenClaw to treat pending Monitor waits as expected idle so the CLI no-output watchdog does not kill long-running silent claude-cli sessions.

## Best Possible Solution

Introduce a bounded expected-idle signal for verified pending Monitor/tool waits that pauses or heartbeats the no-output watchdog only while tool work is pending, while preserving the overall timeout and normal stall protection.

## Reproduction Assessment

Yes, at source level: the issue provides concrete Monitor steps and matching watchdog logs, and current main still computes and enforces static no-output timeouts without pending Monitor state. I did not run a live 180-second Claude CLI wait.

## Solution Assessment

Unclear. Treating verified Monitor waits as expected idle is the right direction, but raising constants or broadly disabling the watchdog is only a mitigation, so the maintainable liveness seam needs maintainer choice.
