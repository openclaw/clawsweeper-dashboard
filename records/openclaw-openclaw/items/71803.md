## Label Justifications

- `P1`: The report describes real user sessions being killed mid-workflow after a common long-running Monitor wait, which is an urgent agent runtime regression class.
- `impact:crash-loop`: The user-visible failure is process/session termination from a watchdog timeout rather than graceful waiting.
- `impact:session-state`: The killed CLI process loses the in-flight agent context needed to consume the Monitor completion and continue the task.

## Summary

Keep open: current main and the latest release still use output-driven CLI no-output watchdogs with no pending-Monitor or expected-idle signal, and the related PRs are open mitigations or adjacent diagnostics rather than a merged fix for this Monitor wait contract.

## What This Changes

The issue asks for CLI no-output watchdog behavior to exempt or heartbeat deliberate idle waits while a Monitor task is pending, so long-running tool work does not kill the agent session.

## Best Possible Solution

Add a bounded expected-idle signal for verified pending Monitor/tool waits that pauses or heartbeats the no-output watchdog only while tool work is actually pending, while preserving overall timeouts and normal no-output protection.

## Reproduction Assessment

Yes, at source level. The issue gives concrete Monitor steps and matching watchdog logs, and current main still times silent CLI turns from stdout/stderr activity rather than pending Monitor state; I did not run the live 180-second wait.

## Solution Assessment

Unclear. Treating verified pending Monitor waits as expected idle is the right direction, but the exact liveness signal and timeout semantics need owner choice so the watchdog still catches genuinely stuck CLI processes.
