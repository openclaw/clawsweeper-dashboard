## Label Justifications

- `P1`: The reported behavior can terminate an active CLI-backed agent session during a supported long-running workflow and lose in-flight user context.
- `impact:session-state`: The issue is about an in-flight agent session being killed before it can consume the Monitor result and continue.
- `impact:crash-loop`: The affected path forcibly terminates the CLI subprocess through the no-output watchdog and surfaces a timeout failure.

## Summary

Keep open: current main and v2026.5.19 still use output-driven CLI no-output watchdogs with no pending Monitor or expected-idle signal, so the reported session loss remains source-proven and is not implemented by the related open watchdog work.

## What This Changes

The issue requests that CLI watchdogs stop terminating otherwise idle CLI sessions while a verified Monitor task is pending.

## Best Possible Solution

Add a verified pending Monitor/tool-wait expected-idle contract that pauses or heartbeats only the no-output watchdog while preserving overall timeouts and normal stall detection.

## Reproduction Assessment

Yes, at source level. The issue provides concrete Monitor steps and matching watchdog logs, and current main still times silent CLI turns from stdout/stderr activity rather than pending Monitor state; I did not run the live 180-second wait.

## Solution Assessment

Unclear. Treating verified pending Monitor waits as expected idle is the maintainable direction, but raising constants or disabling the watchdog broadly is only a mitigation and the signal boundary needs maintainer choice.
