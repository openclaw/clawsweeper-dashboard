## Label Justifications

- `P1`: The reported behavior can terminate an active CLI-backed agent session during a supported long-running workflow and lose in-flight session continuity.
- `impact:session-state`: The issue is about an in-flight agent session being killed before it can consume the Monitor result and continue.
- `impact:crash-loop`: The affected path forcibly terminates the CLI subprocess via the no-output watchdog and surfaces a timeout failure.

## Summary

Keep open: current main and the latest release still use output-driven CLI no-output watchdogs with no pending Monitor or expected-idle signal, so the reported session-loss path remains source-proven and needs maintainer product review.

## What This Changes

Change CLI liveness so a Claude CLI session waiting on a pending Monitor task is not killed solely for producing no stdout within the no-output watchdog window.

## Best Possible Solution

Model verified pending Monitor/tool waits as bounded expected idle that pauses or heartbeats the no-output watchdog only while tool work is actually pending, while preserving overall timeouts and normal no-output protection.

## Reproduction Assessment

Yes, at source level. The issue gives concrete Monitor steps and matching watchdog logs, and current main still times silent CLI turns from stdout/stderr activity rather than pending Monitor state.

## Solution Assessment

Unclear. Treating verified pending Monitor waits as expected idle is the right direction, but raising constants or disabling the watchdog broadly would be a weaker liveness contract.
