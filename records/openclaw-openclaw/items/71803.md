## Summary

Keep open: current main still uses output-driven CLI no-output watchdogs with no verified pending-Monitor expected-idle exemption, so the reported session-loss path remains plausible. The related open timeout-cap PR may reduce symptoms, but it does not implement the Monitor-aware contract requested here.

## What This Changes

The issue asks the CLI watchdog to treat pending Monitor waits as expected idle instead of killing silent Claude CLI sessions after the default no-output window.

## Best Possible Solution

Add a bounded expected-idle or heartbeat mechanism for verified pending Monitor/tool waits, while preserving the overall request timeout and normal no-output protection for unproven stalls.

## Reproduction Assessment

Yes. The issue includes concrete Monitor steps and matching watchdog logs, and current main still has source-level evidence that silent CLI turns are timed by stdout/stderr output rather than pending Monitor state.

## Solution Assessment

Unclear. Treating verified pending Monitor/tool waits as expected idle is the right direction, but simply raising constants as in the related open PR is not the narrow durable fix.
