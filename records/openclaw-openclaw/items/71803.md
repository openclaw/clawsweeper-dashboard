## Summary

Keep open. Current main still applies the CLI no-output watchdog to silent Claude CLI turns with no Monitor-aware expected-idle signal, and the open timeout-cap PR only lengthens the watchdog window rather than fixing the healthy pending-Monitor state.

## What This Changes

The issue asks for CLI sessions waiting on long-running Monitor/tool work to remain alive instead of being killed by the no-output watchdog after the default idle window.

## Best Possible Solution

Add an explicit expected-idle/proof-of-life contract for pending Monitor or MCP/client-tool waits in the CLI runner path, while preserving the overall request timeout and the no-output watchdog for CLIs that are truly stalled.

## Reproduction Assessment

Yes. The issue gives a concrete Monitor wait command and matching watchdog logs, and current main source/tests show silent CLI processes still hit the no-output watchdog without a pending Monitor/tool exemption.

## Solution Assessment

No current or related proposed fix is sufficient. Raising timeout constants may reduce false kills, but the maintainable fix is to distinguish verified pending tool work from a stalled CLI process.
