## Summary

Keep open. Current main still treats silent CLI and Claude live-session turns as no-output stalls, with no Monitor-aware or pending-tool expected-idle signal; the related open constant-raising PR would only make the failure less frequent, not fix the healthy idle state reported here.

## What This Changes

The issue requests that CLI no-output watchdog handling stop killing sessions that are intentionally idle while waiting for a long-running Monitor task.

## Best Possible Solution

Add a generic expected-idle or proof-of-life contract for verified pending Monitor/MCP/tool waits in the CLI runner path, while preserving the overall request timeout and no-output protection for truly stalled CLI processes.

## Reproduction Assessment

Yes. The issue provides a concrete Monitor wait command and matching watchdog logs, and current source/tests show silent CLI processes still trip the no-output timeout without a Monitor-aware exemption.

## Solution Assessment

Unclear. Treating verified pending Monitor work as expected idle is the right direction, but maintainers should choose whether that state is detected in the Claude live-session parser, a generic CLI tool-wait signal, or a bounded heartbeat mechanism.
