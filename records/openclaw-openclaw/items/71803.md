## Summary

Keep open: current main still enforces CLI no-output watchdogs from stdout/stderr activity, with no verified pending Monitor/tool-wait exemption. The related open PR only raises timeout constants, which may reduce symptoms but does not implement the expected-idle contract requested here.

## What This Changes

The issue asks for CLI no-output watchdogs to treat pending Monitor work as expected idle instead of killing the provider process after the default timeout.

## Best Possible Solution

Add a bounded expected-idle mechanism for verified pending Monitor/tool waits in the Claude live-session and/or generic CLI supervisor path, while preserving the overall request timeout and normal no-output protection when no pending work is proven.

## Reproduction Assessment

Yes. The issue gives concrete Monitor steps and matching watchdog logs, and current main still has source-level evidence that silent CLI turns are timed by stdout/stderr output rather than pending Monitor state.

## Solution Assessment

Unclear. Treating verified pending Monitor/tool waits as expected idle is the right direction, but the maintainable seam should be chosen deliberately rather than only raising timeout constants or disabling the watchdog globally.
