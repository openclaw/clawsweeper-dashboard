## Summary

Keep open: current main still has output-driven CLI no-output watchdogs with no pending-Monitor or expected-idle signal. The related open PRs cover timeout caps and orphan-output idle-close behavior, but neither implements this issue's Monitor wait contract.

## What This Changes

Request that the CLI no-output watchdog not terminate claude-cli sessions that are deliberately silent while waiting for a long-running Monitor task.

## Best Possible Solution

Add a bounded expected-idle signal for verified pending Monitor/tool waits in the Claude live-session or supervisor path while preserving overall timeouts and normal no-output protection when no pending work is known.

## Reproduction Assessment

Yes, at source level. The issue gives concrete Monitor steps and matching watchdog logs, and current main still times silent CLI turns from stdout/stderr activity rather than pending Monitor state; I did not run a live 180-second Monitor wait in this read-only review.

## Solution Assessment

Unclear. Treating verified pending Monitor/tool waits as expected idle is the right direction, but the maintainable seam and safety boundary need owner choice; raising constants alone only mitigates some cases.
