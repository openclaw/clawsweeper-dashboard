## Summary

Keep open. Current main still lacks per-conversation identity in the provider `createStreamFn` context, and this PR targets the narrow public provider-plugin seam, but it needs small cleanup before merge: the new `sessionId` JSDoc misdescribes the value as a filename, and the user-facing SDK addition needs the required changelog entry.

## What This Changes

The PR adds optional `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and forwards them from embedded attempt, compaction, and BTW provider stream registration paths.

## Best Possible Solution

Land the additive optional fields after correcting the identity wording, adding a one-line `CHANGELOG.md` entry under `Unreleased` / `Changes`, and adding focused passthrough assertions. Keep the linked `createStreamFn` issue open until this PR merges.

## Reproduction Assessment

Yes. Current-main source inspection shows session identity is available in conversation-aware paths but omitted from the provider `createStreamFn` context; #73488 and the linked glueclaw issue provide a concrete multi-conversation transcript-bleed scenario.

## Solution Assessment

Yes, with cleanup. Adding optional fields to the existing `createStreamFn` context is the narrow generic seam for custom provider transports; #54725 covers `prepareExtraParams` and does not give `createStreamFn` plugins this identity.
