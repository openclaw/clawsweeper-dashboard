## Summary

Keep this PR open. Current main still lacks `sessionId` and `sessionKey` in the provider `createStreamFn` context, and the linked glueclaw issue/PR provide concrete downstream evidence for the multi-conversation transcript-bleed problem; the proposed plumbing is narrow and additive but still needs small documentation, changelog, and test follow-up before merge.

## What This Changes

The PR adds optional `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and forwards them through embedded attempt, compaction, and BTW provider stream registration paths.

## Best Possible Solution

Land the additive optional fields if maintainers accept the API shape, but first correct the session-id wording, add a one-line active changelog entry with contributor credit, and add focused passthrough coverage while keeping one-off image/simple-completion paths without conversation identity.

Yes. Current-main source inspection shows the session identity exists in embedded, compaction, and BTW flows but is not included in the provider `createStreamFn` context; the linked glueclaw report gives a concrete concurrent-DM transcript-bleed scenario.

## Solution Assessment
