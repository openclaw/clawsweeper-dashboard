## What This Changes

Telegram polling startup diagnostics are reported as being routed through the error output path instead of the normal log channel.

## Best Possible Solution

Review and land a focused Telegram monitor fix that routes lifecycle diagnostics through `runtime.log` while keeping actual failure paths on `runtime.error`, then let the linked PR close the issue.

## Reproduction Assessment

Yes, at source level: current main routes `polling cycle started` through the monitor logger, and that logger is currently `runtime.error`. I did not run a live Telegram polling session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to split normal monitor diagnostics to `runtime.log` while preserving `runtime.error` for persistence failures and actual monitor errors; the linked PR appears to be the right review path.
