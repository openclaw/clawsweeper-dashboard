## Summary

Keep open. Current main still filters builtin memory extra paths to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, chunks non-session text as Markdown, and documents extra-path scanning as Markdown-only; the remaining decision is whether raw JSONL logs belong in builtin memory or in QMD, session memory, a converter, or a plugin workflow.

## What This Changes

The issue requests native JSONL support for builtin `memorySearch.extraPaths` so raw message hook logs can be indexed without converting them to Markdown.

## Best Possible Solution

Settle the product boundary, then either implement first-class JSONL extra-path support across discovery, parsing/chunking, watcher/read behavior, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin extra paths to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` paths, and reserves JSONL parsing for the separate session transcript source.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable fix is broader than adding an extension check and may instead be QMD, session-memory, converter, or plugin documentation.
