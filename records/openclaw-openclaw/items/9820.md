## Summary

Keep open. Current main and v2026.6.6 still exclude ordinary JSONL files from builtin `memorySearch.extraPaths`, while the JSONL support that exists is limited to opt-in session transcript indexing with a narrower redaction and line-mapping contract.

## What This Changes

The issue asks builtin `memorySearch.extraPaths` to admit and index JSONL raw message logs without converting them to Markdown.

## Best Possible Solution

Adopt one canonical contract: either implement schema-aware JSONL admission, extraction, redaction, line mapping, watching, retrieval, docs, and tests across builtin memory, or document session memory, QMD, or an external converter/plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level. Current main directly excludes plain JSONL from ordinary memory discovery, retrieval, and watching, while session transcript JSONL follows a separate line-mapped path.

## Solution Assessment

No, not as the proposed extension-list-only change. The maintainable solution needs a JSONL record-to-text and redaction contract plus consistent updates across admission, indexing, watching, retrieval, citations/docs, and tests, or a documented non-builtin alternative.
