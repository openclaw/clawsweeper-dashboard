## Summary

Keep open. Current main and the latest release still exclude ordinary JSONL files from builtin memory extra-path discovery and retrieval, while the existing JSONL handling is limited to session transcript indexing; the remaining question is a product contract for builtin JSONL versus a documented alternative.

## What This Changes

Request native JSONL ingestion for builtin `memorySearch.extraPaths` so raw message logs can be indexed without converting them to Markdown.

## Best Possible Solution

Adopt one canonical contract: either implement schema-aware JSONL admission, parsing, chunking, line mapping, watching, retrieval, docs, and focused tests across builtin memory, or document QMD/session-memory/converter-plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level. Current main directly excludes plain JSONL from ordinary memory discovery and retrieval, while session transcript JSONL follows a separate line-mapped `sessions` source path.

## Solution Assessment

No, not as the proposed extension-list-only change. The best solution must define JSONL record semantics and update discovery, indexing, watching, retrieval, citations/docs, and tests consistently, or document the supported alternative.
