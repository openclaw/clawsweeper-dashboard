## Summary

Keep open. Current main and the latest release still do not support arbitrary JSONL files through builtin `memorySearch.extraPaths`; JSONL parsing exists for the separate session transcript path, and maintainers still need to decide whether raw hook logs belong in builtin memory, QMD/session memory, or a plugin/converter workflow.

## What This Changes

The issue requests native JSONL support for builtin memory indexing of `memorySearch.extraPaths` so raw message-log files can be semantically searched without conversion.

## Best Possible Solution

Define the JSONL memory-file contract first, then either implement it consistently across builtin memory admission, parsing/chunking, watching, `memory_get`, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin `extraPaths` to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, and reserves JSONL parsing for the separate session transcript source.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable fix is broader than adding `.jsonl` to an extension list and may instead be a documented QMD/session-memory/converter/plugin workflow.
