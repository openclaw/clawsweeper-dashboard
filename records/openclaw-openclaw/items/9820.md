## Summary

Keep open. Current main still does not implement native JSONL ingestion for builtin `memorySearch.extraPaths`: discovery, `memory_get`, watcher filtering, docs, and tests remain Markdown/multimodal-oriented, while JSONL support is limited to the separate opt-in session-transcript source. The recent maintainer-side discussion also explicitly kept this open, and the current code still matches that assessment.

## What This Changes

The issue requests native JSONL support for `memorySearch.extraPaths` so raw message logs can be indexed without converting them to Markdown.

## Best Possible Solution

Add a documented native JSONL path for builtin `memorySearch.extraPaths`, including file admission, line-oriented chunking with source line mapping, `memory_get` behavior, watcher filtering, config help/docs, and focused regression tests. If maintainers choose not to support arbitrary raw JSONL in builtin memory, document the intended QMD/session-memory/converter-plugin workflow before closing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a `.jsonl` file or directory under `memorySearch.extraPaths`: current `listMemoryFiles` skips it unless it is `.md` or enabled multimodal media, `readMemoryFile` rejects non-`.md`, and the watcher ignores `.jsonl` file events.

## Solution Assessment

No, merely adding `.jsonl` to an extension list would be incomplete. The maintainable fix needs a line-aware JSONL indexing contract plus matching read/watch/docs/tests, while preserving the existing separate session-transcript JSONL source.
