## Summary

Current main and release v2026.6.5 still exclude arbitrary JSONL from builtin memory extra paths. The source gap is clear, but the issue needs a maintainer decision on whether hook-produced JSONL belongs in the bundled memory plugin or a converter, QMD, or external plugin workflow; the recent discussion correctly keeps that decision open.

## What This Changes

Add native JSONL ingestion and record-aware chunking for files configured through builtin memory extra paths.

## Best Possible Solution

Define the supported JSONL record schema and ownership at the memory plugin boundary, then either implement discovery, extraction, chunking, citations, retrieval, watching, tests, and docs as one coherent feature or document a supported converter, QMD, or external plugin workflow.

## Reproduction Assessment

Yes at source level: current main consistently excludes arbitrary JSONL from builtin extra-path discovery and retrieval and has no record-aware memory-file indexing path for it.

## Solution Assessment

No, the proposed extension allowlist alone is not the best solution because it omits the JSONL extraction contract, line and citation mapping, retrieval, watcher behavior, tests, documentation, and owner boundary.
