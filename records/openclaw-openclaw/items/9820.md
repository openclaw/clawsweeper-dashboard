## Summary

Keep open: current main still does not support arbitrary JSONL files through builtin `memorySearch.extraPaths`; the existing session-transcript JSONL and `oc-path` plugin paths are adjacent but do not solve raw extra-path memory ingestion, and the remaining work needs maintainer product/security direction.

## What This Changes

The issue requests native JSONL support for memorySearch extra paths so raw message-log JSONL files can be indexed without conversion.

## Best Possible Solution

Keep this open until maintainers either add a complete JSONL ingestion contract for builtin memory extra paths or document the supported converter/plugin/QMD alternative and close on that product decision.

## Reproduction Assessment

Unclear; a small extension-list patch is not the best complete solution because file admission, JSONL parsing/chunking, line mapping, watcher behavior, `memory_get`, docs, and redaction semantics all need one coherent contract.
