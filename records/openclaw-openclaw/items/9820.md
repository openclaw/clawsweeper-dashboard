## Summary

Keep open. Current main still does not support native JSONL ingestion through builtin memorySearch.extraPaths, and the recent maintainer-side discussion already identified this as a product/API follow-up rather than cleanup to close.

## What This Changes

Add native JSONL support to memorySearch.extraPaths so raw message-log JSONL can be indexed for semantic memory search without conversion to Markdown.

## Best Possible Solution

Adopt one documented path: either implement JSONL as a first-class builtin extra-path source across discovery, parsing/chunking, watching, exact reads, config/docs, and tests, or document the supported raw-log workflow outside builtin extra-path indexing.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is an enhancement request. Source inspection gives high confidence that arbitrary .jsonl extra paths remain unsupported on current main.

## Solution Assessment

Unclear. Adding JSONL support is plausible, but the proposed extension whitelist alone is not the best fix because JSONL chunking, line mapping, memory_get, watcher behavior, and docs need a coherent product decision.
