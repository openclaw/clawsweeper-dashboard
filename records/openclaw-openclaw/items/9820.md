## Summary

Keep open. Current main and the latest release still do not implement arbitrary JSONL ingestion through builtin `memorySearch.extraPaths`; the request is valid at source level but needs maintainer direction on the builtin-vs-QMD/plugin boundary before a safe implementation.

## What This Changes

The issue requests native JSONL support for memory indexing so raw message logs in `memorySearch.extraPaths` can be discovered, chunked, searched, and read without converting them to Markdown.

## Best Possible Solution

Define the JSONL memory-file contract first, then either implement it consistently across builtin memory admission, parsing/chunking, watching, `memory_get`, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin `extraPaths` to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, and reserves JSONL line mapping for the separate session transcript source.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable fix is broader than adding `.jsonl` to an extension list and may instead be a documented QMD/session-memory/converter/plugin workflow.
