## Summary

Keep open. Current main still excludes arbitrary `.jsonl` files from builtin `memorySearch.extraPaths` discovery, `memory_get`, and watcher/indexing behavior; the existing JSONL path is limited to OpenClaw session transcripts, and the live discussion already calls out the remaining product decision.

## What This Changes

Request native JSONL ingestion for the builtin memory indexer so raw message logs listed in `memorySearch.extraPaths` can be semantically searched without conversion to Markdown.

## Best Possible Solution

Define the memory product boundary, then either implement JSONL extraPaths across discovery, parsing/chunking with line mapping, watching, `memory_get`, docs, and tests, or document the supported alternative workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows plain `.jsonl` extraPaths are filtered out or rejected while session transcript JSONL uses a separate parser; I did not run a live filesystem repro in this read-only review.

## Solution Assessment

Unclear. Native JSONL support may be the right direction, but the proposed allow-list alone is not enough because the maintainable fix must define JSONL parsing, line mapping, watcher/read behavior, and docs, or choose an explicit alternative.
