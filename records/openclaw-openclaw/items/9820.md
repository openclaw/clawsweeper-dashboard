## Summary

Keep open. Current main and the latest release still do not support arbitrary JSONL files in builtin memorySearch.extraPaths; the existing JSONL support is limited to opt-in session transcript indexing with a separate sanitized parser.

## What This Changes

The issue requests native JSONL ingestion for memorySearch.extraPaths so raw message-log JSONL can be indexed without converting to Markdown.

## Best Possible Solution

Adopt one explicit path: implement schema-aware JSONL extraction, redaction, line mapping, watching, retrieval, docs, and tests across builtin memory, or document QMD, session-memory, or a converter/plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level. Current main excludes plain JSONL from builtin memory discovery, retrieval, and direct-file watching while session transcript JSONL follows a separate line-mapped parser.

## Solution Assessment

No, not as the proposed extension-list-only change. The maintainable solution needs a JSONL extraction/redaction/citation contract and consistent updates across admission, indexing, watching, retrieval, docs, and tests, or a documented non-builtin alternative.
