## Summary

Keep open. Current main still excludes arbitrary `.jsonl` files from builtin `memorySearch.extraPaths`, and the discussion already identifies this as a maintainer product/API decision rather than a cleanup close.

## What This Changes

The issue requests native `.jsonl` support for memory indexing through `memorySearch.extraPaths` so raw message logs can be searched without conversion.

## Best Possible Solution

Adopt one documented path: either implement JSONL as a first-class builtin extra-path source across discovery, parsing, watching, reads, docs, and tests, or document the supported raw-log workflow outside builtin extra-path indexing.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is an enhancement request. Source inspection gives high confidence that arbitrary `.jsonl` extra paths remain unsupported on current main.

## Solution Assessment

Unclear. Native JSONL support may be reasonable, but the proposed extension whitelist is not the best complete fix without JSONL parsing/chunking semantics, line mapping, `memory_get`, watcher behavior, docs, and tests.
