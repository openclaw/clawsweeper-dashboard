## Summary

Keep open. Current main and the latest release still treat builtin memory extra paths as Markdown plus enabled multimodal files, while JSONL parsing/line mapping is limited to session transcripts; accepting arbitrary raw JSONL logs needs a memory product decision.

## What This Changes

Add native JSONL support to builtin `memorySearch.extraPaths` so raw message logs can be indexed and searched without converting them to Markdown.

## Best Possible Solution

Define the memory product boundary, then either implement JSONL extraPaths across discovery, parsing/chunking with line mapping, watching, `memory_get`, docs, and tests, or document the supported alternative workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows a `.jsonl` file in `memorySearch.extraPaths` is skipped by discovery/watcher paths or rejected by `memory_get`, while JSONL parsing is limited to session transcript indexing.

## Solution Assessment

Unclear. Native JSONL support may be the right direction, but the proposed allow-list alone is not enough because the maintainable fix must define parsing, chunk boundaries, line mapping, watcher/read behavior, docs, and tests.
