## Summary

Current main still excludes ordinary JSONL files from builtin memory extra-path discovery and retrieval, while session transcript JSONL uses a separate structured indexing path. No canonical replacement issue or viable PR owns the request, and the issue discussion explicitly leaves the builtin-versus-alternative product decision unresolved.

## What This Changes

Add native JSONL ingestion to builtin memory search extra paths so raw message logs can be searched without Markdown conversion.

## Best Possible Solution

Adopt one canonical contract: either implement schema-aware JSONL admission, text extraction, line mapping, watching, retrieval, documentation, and focused tests across the builtin memory surface, or document session memory, QMD, or an external converter/plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level. Current main directly excludes plain JSONL from ordinary memory discovery and retrieval, while session transcript JSONL follows a separate line-mapped source path.

## Solution Assessment

No, not as the proposed extension-list-only change. The best solution must define JSONL record semantics and update discovery, indexing, watching, retrieval, citations, docs, and tests consistently.
