## Summary

Keep open. Current main still excludes arbitrary `.jsonl` files from builtin `memorySearch.extraPaths` discovery/read/watch paths, while the existing JSONL support is limited to session transcript indexing; this remains a product-scope feature request rather than a closed bug fix.

## What This Changes

The issue requests native `.jsonl` ingestion for builtin memory search extra paths so raw message hook logs can be indexed without conversion.

## Best Possible Solution

Define the product boundary, then either implement JSONL extraPaths across discovery, parsing/chunking, watching, `memory_get`, docs, and tests, or document the supported alternative workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows JSONL extraPaths are filtered out or rejected while session transcript JSONL uses a separate parser; I did not run a live filesystem repro in the read-only checkout.

## Solution Assessment

Unclear. Native JSONL support may be a good solution, but the proposed extension-list change is not sufficient because the maintainable fix must define JSONL parsing, line mapping, watcher/read behavior, and documentation, or choose an explicit alternative.
