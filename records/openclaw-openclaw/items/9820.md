## Summary

Keep open. Current main still does not support arbitrary `.jsonl` files in builtin `memorySearch.extraPaths`, and the remaining choice is a memory product/API decision rather than conservative cleanup.

## What This Changes

Request native JSONL ingestion for memory indexing so raw message-hook logs in `memorySearch.extraPaths` can be searched without converting them to Markdown.

## Best Possible Solution

Adopt one documented path: either implement JSONL as a first-class extra-path source across discovery, parsing, watching, reads, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes as a source-level behavior check: current main admits Markdown or enabled multimodal extra-path files and rejects ordinary `.jsonl` in discovery, watching, and `memory_get`.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native JSONL support may be the right direction, but the proposed extension whitelist is not enough without JSONL parsing, chunking, line mapping, watcher, read, docs, and privacy semantics.
