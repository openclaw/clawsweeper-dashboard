## Summary

Keep open. Current main still excludes plain JSONL from builtin memory extra paths, and recent maintainer-side discussion already kept this open because the remaining question is product scope rather than stale cleanup.

## What This Changes

The issue requests native JSONL support in builtin memory indexing so raw hook-generated message logs in `memorySearch.extraPaths` can be semantically searched without conversion to Markdown.

## Best Possible Solution

Either implement first-class JSONL extra-path support across admission, line-aware parsing/chunking, watcher/read behavior, docs, and tests, or document the supported QMD/session-memory/converter workflow as the intended path.

## Reproduction Assessment

Yes, at source level. Current main still filters builtin memory extra paths to Markdown or enabled multimodal files and rejects non-Markdown paths in `memory_get`; I did not run a live indexer because the checkout is read-only.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but QMD, session-memory, converter, or plugin workflows may be the intended supported path for raw logs.
