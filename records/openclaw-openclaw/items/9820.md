## Summary

Keep open. Current main still excludes arbitrary `.jsonl` files from builtin `memorySearch.extraPaths`, and the live discussion already kept this as a maintainer product/API decision rather than cleanup closure.

## What This Changes

The issue requests native JSONL ingestion for builtin memory indexing so raw message-hook logs in `memorySearch.extraPaths` can be searched without converting them to Markdown.

## Best Possible Solution

Adopt one documented path: implement JSONL as a first-class extra-path source across discovery, parsing, watching, reads, docs, and tests, or document QMD/session-memory/converter/plugin ingestion as the supported workflow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is an enhancement request, but source inspection gives high confidence that arbitrary `.jsonl` extra paths are still filtered or rejected on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native JSONL support may be the right product direction, but the proposed extension whitelist is not sufficient without clear JSONL parsing, chunking, line mapping, watcher, read, documentation, and privacy semantics.
