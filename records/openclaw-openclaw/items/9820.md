## Summary

Current main and release v2026.6.5 still limit builtin `memorySearch.extraPaths` to Markdown or enabled multimodal files. The source gap is concrete, but the issue should remain open for an explicit decision on whether arbitrary raw JSONL belongs in builtin memory or a plugin, QMD, or session-memory workflow.

## What This Changes

Add native JSONL ingestion for arbitrary files supplied through builtin memory search extra paths.

## Best Possible Solution

Choose one owner for raw-log ingestion, then either implement a complete builtin JSONL contract across discovery, parsing, chunk provenance, retrieval, watching, config help, docs, and focused tests, or document the supported plugin, QMD, session-memory, or converter workflow.

## Reproduction Assessment

Yes at source level: on current main a plain `.jsonl` extra path is deterministically rejected by file discovery and `memory_get` validation before it can participate in builtin indexing or retrieval.

## Solution Assessment

No, broadening the extension filter alone is not the best solution; a maintainable implementation must define JSONL record extraction and preserve line provenance consistently across indexing, citations, retrieval, and file watching.
