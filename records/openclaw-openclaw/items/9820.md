## Summary

Keep open. Current main and the latest release still do not support arbitrary JSONL files through builtin `memorySearch.extraPaths`; the existing JSONL handling is limited to session transcript indexing, and maintainers still need to decide whether raw hook logs belong in builtin memory, QMD/session memory, or a converter/plugin workflow.

## What This Changes

The issue requests native JSONL support for memory indexing from `memorySearch.extraPaths` so raw message logs can be searched without a converter.

## Best Possible Solution

Settle the builtin-versus-plugin/QMD boundary, then either implement JSONL extra-path support across admission, parsing/chunking, watching, reading, docs, and tests or document the supported converter/session-memory workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin extra paths to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, and reserves JSONL parsing for the separate session transcript path.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable solution is broader than adding `.jsonl` to an extension list and may instead be a documented QMD/session-memory/converter/plugin workflow.
