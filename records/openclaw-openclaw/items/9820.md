## Summary

Keep open. Current main still does not support native JSONL ingestion through builtin `memorySearch.extraPaths`, and the existing discussion already identifies this as a maintainer product/API follow-up rather than a cleanup close.

## What This Changes

Request native JSONL ingestion for `memorySearch.extraPaths` so raw message hook logs can be indexed without converting them to Markdown.

## Best Possible Solution

Adopt one documented path: either implement JSONL as a first-class builtin extra-path source across discovery, parsing/chunking, watching, exact reads, config/docs, and tests, or document the supported raw-log workflow outside builtin extra-path indexing.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is an enhancement request. Source inspection gives high confidence that arbitrary `.jsonl` extra paths remain unsupported on current main.

## Solution Assessment

Unclear. Native JSONL support is plausible, but the proposed extension whitelist is not the best complete fix because chunking, line mapping, `memory_get`, watcher behavior, and docs need a coherent product/API decision.
