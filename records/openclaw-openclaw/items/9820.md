## Summary

Keep open. Current main still excludes arbitrary `.jsonl` files from builtin `memorySearch.extraPaths`, and this is a new ingestion mode that needs maintainer product/API direction rather than cleanup closure.

## What This Changes

Request native builtin memory indexing support for `.jsonl` files in `memorySearch.extraPaths`, with JSONL lines treated as searchable chunks.

## Best Possible Solution

Adopt one documented path: implement JSONL as a first-class extra-path source across discovery, parsing, watching, reads, docs, and tests, or document QMD/session-memory/converter/plugin as the supported workflow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is an enhancement request; source inspection of current main gives high confidence that arbitrary `.jsonl` extra paths are still filtered or rejected.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native JSONL support may be reasonable, but the proposed extension whitelist is not enough without JSONL parsing/chunking semantics, line mapping, watcher behavior, `memory_get`, docs, tests, and privacy expectations.
