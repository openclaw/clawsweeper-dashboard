## Summary

Keep open. Current main still excludes plain JSONL from builtin memorySearch.extraPaths and memory_get, while the docs describe extra paths as Markdown-only; the remaining decision is product/API scope rather than stale cleanup.

## What This Changes

The issue requests native JSONL support in the builtin memory indexer so raw message-hook logs in memorySearch.extraPaths can be searched without a converter.

## Best Possible Solution

Either implement first-class JSONL extra-path support across admission, line-aware parsing/chunking, watcher/read behavior, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow as the intended path.

## Reproduction Assessment

Yes, at source level. Current main filters builtin memory extra paths to Markdown or enabled multimodal files and rejects non-Markdown paths in memory_get; I did not run a live indexer because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but QMD, session-memory, converter, or plugin workflows may be the intended supported path for raw logs.
