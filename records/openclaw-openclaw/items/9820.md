## Summary

Keep open. Current main still filters arbitrary `.jsonl` files out of builtin `memorySearch.extraPaths` discovery, read, and watch paths, while the existing JSONL handling is limited to the separate session transcript source; this remains a product-scope memory feature request rather than an implemented fix.

## What This Changes

Request native indexing of JSONL files supplied through `memorySearch.extraPaths` so raw message logs can be semantically searched without conversion.

## Best Possible Solution

Define the memory product boundary, then either implement JSONL extraPaths across discovery, parsing/chunking with line mapping, watching, `memory_get`, docs, and tests, or document the supported alternative workflow.

## Reproduction Assessment

Yes. Source inspection on current main shows plain `.jsonl` extraPaths are filtered out or rejected while session transcript JSONL uses a separate parser; I did not run a live filesystem repro in this read-only review.

## Solution Assessment

Unclear. Native JSONL support may be the right direction, but the proposed extension allow-list alone is not sufficient because the maintainable fix must define JSONL parsing, line mapping, watcher/read behavior, and docs, or choose an explicit alternative.
