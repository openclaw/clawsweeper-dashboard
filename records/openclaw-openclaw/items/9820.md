## Summary

Keep open. Current main and the latest release still exclude ordinary JSONL files from builtin `memorySearch.extraPaths`, while the existing JSONL handling is limited to the separate session-transcript indexing path and no canonical duplicate owns arbitrary JSONL extra-path ingestion.

## What This Changes

The issue requests native JSONL ingestion for builtin memory indexing so raw message-log files in `memorySearch.extraPaths` can be searched without converting them to Markdown.

## Best Possible Solution

Adopt one canonical contract: either implement schema-aware JSONL admission, text extraction, line mapping, watching, retrieval, docs, and tests across builtin memory, or document session memory, QMD, or an external converter/plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level. Current main directly excludes plain JSONL from ordinary memory discovery, retrieval, and watching, while session transcript JSONL follows a separate line-mapped path.

## Solution Assessment

No, not as the proposed extension-list-only change. The maintainable solution needs a JSONL record-to-text contract and consistent updates across admission, indexing, watching, retrieval, citations/docs, and tests, or a documented non-builtin alternative.
