## Summary

Keep open. Current main still does not provide native JSONL support for builtin memory extra paths: discovery remains Markdown/multimodal-focused, `memory_get` rejects non-Markdown files, ordinary indexing still chunks text as Markdown, and the prior review discussion leaves the builtin-vs-workflow product boundary unresolved.

## What This Changes

The issue requests first-class `.jsonl` ingestion for memory indexing so raw message hook logs in `memorySearch.extraPaths` can be semantically searched without converting them to Markdown.

## Best Possible Solution

Settle the builtin-versus-plugin/QMD product boundary, then either implement JSONL extra-path support across admission, parsing/chunking, watching, reading, docs, and regression tests or document the supported converter/session-memory workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin extra paths to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, and reserves JSONL parsing behavior for the separate session transcript path.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable solution is broader than adding `.jsonl` to an extension list and may instead be a documented QMD/session-memory/converter/plugin workflow.
