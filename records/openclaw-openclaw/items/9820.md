## Summary

Keep open: current main and the latest release still omit arbitrary JSONL support for builtin `memorySearch.extraPaths`, while the existing JSONL path is opt-in session transcript indexing with redaction and line mapping; adding raw log ingestion remains a product/security decision.

## What This Changes

Request native JSONL support for builtin `memorySearch.extraPaths` so raw message logs can be semantically indexed without converting them to Markdown.

## Best Possible Solution

Maintainers should decide whether raw JSONL belongs in builtin memory; if yes, implement a complete admission, parser/chunker, line-map, redaction, watcher, `memory_get`, docs, and test path, otherwise document the QMD/session-memory/converter-plugin workflow.

## Reproduction Assessment

Yes from source inspection: a `.jsonl` path under `memorySearch.extraPaths` is not admitted by `listMemoryFiles`, and `memory_get` still rejects non-`.md` paths. I did not run a filesystem repro because this review was read-only.

## Solution Assessment

Unclear for implementation: native JSONL support is plausible, but the best path needs maintainer choice between builtin support and QMD/session-memory/converter-plugin workflows plus a redaction and visibility contract.
