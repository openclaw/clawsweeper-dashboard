## Summary

Keep open. Current main still excludes `.jsonl` from builtin `memorySearch.extraPaths` discovery, exact reads, watcher admission, and docs; existing JSONL support is limited to session transcripts, and the recent maintainer-side review also kept this open.

## What This Changes

Request native JSONL support for builtin memory indexing so raw message logs in `memorySearch.extraPaths` can be searched without converting them to Markdown.

## Best Possible Solution

Add a documented native JSONL path for builtin `memorySearch.extraPaths` with line-aware chunking, matching exact-read and watcher behavior, docs, changelog, and focused tests; if that is out of scope, document the supported alternative before closing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a `.jsonl` file or directory in `agents.defaults.memorySearch.extraPaths`; current discovery, read, and watcher paths admit `.md` or enabled multimodal files only.

## Solution Assessment

No. The title's simple extension-list change would be incomplete; the maintainable solution needs line-aware JSONL chunking plus aligned file admission, `memory_get`, watcher filtering, docs, changelog, and tests.
