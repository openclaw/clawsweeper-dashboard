## Summary

Keep open: current main and v2026.6.8 still exclude ordinary JSONL from builtin memory extraPaths, while existing JSONL support is session-transcript-specific and has its own filtering, redaction, and line mapping.

## What This Changes

The issue requests native JSONL ingestion for builtin memory search extra paths so raw message-hook logs can be indexed without conversion.

## Best Possible Solution

Adopt one explicit path: implement safe native JSONL extraPaths support across discovery, parsing, indexing, retrieval, watching, docs, and tests, or document QMD/session-memory/converter plugin as the supported workflow.

## Reproduction Assessment

Yes, at source level: current main excludes plain JSONL from builtin memory discovery, retrieval, and watcher filtering, while session transcript JSONL uses a separate parser. I did not run a live repro because this is a read-only cleanup review.

## Solution Assessment

No, not as the issue body's extension-list-only sketch. The maintainable solution needs schema-aware extraction, redaction, line mapping, retrieval, watcher, docs, and tests, or an explicit documented alternative.
