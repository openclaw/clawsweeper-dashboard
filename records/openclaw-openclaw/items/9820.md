- `P2`: Native JSONL memory ingestion is a useful enhancement with limited blast radius and no urgent runtime regression.
- `impact:session-state`: The request concerns memory/search coverage for transcript-style logs and long-term context retrieval.

## Summary

Keep open. Current main and the latest release still exclude arbitrary JSONL from builtin memorySearch.extraPaths and memory_get, while the existing JSONL support is limited to session transcript indexing rather than raw extra-path ingestion.

## What This Changes

Request native JSONL ingestion for memorySearch.extraPaths so raw message-hook logs can be indexed without converting them to Markdown.

## Best Possible Solution

Either implement first-class JSONL extra-path support across admission, line-aware parsing/chunking, watcher/read behavior, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow as the intended path.

## Reproduction Assessment

Yes, at source level. Plain .jsonl files in extraPaths are excluded by listMemoryFiles and memory_get rejects non-.md paths; I did not run a live indexer because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but QMD, session-memory, converter, or plugin workflows may be the intended supported path for raw logs.
