## Summary

Keep open. Current main still excludes `.jsonl` from builtin `memorySearch.extraPaths` discovery, exact reads, watcher admission, and docs; the existing JSONL support is limited to session transcripts, and recent issue discussion already kept this open for the same gap.

## What This Changes

Request native `.jsonl` support in the builtin memory indexer so raw message logs in `memorySearch.extraPaths` can be semantically searched without conversion.

## Best Possible Solution

Add documented native JSONL support for builtin `memorySearch.extraPaths` across admission, line-aware parsing/chunking, `memory_get`, watcher filtering, docs, changelog, and focused tests; otherwise document the supported alternative workflow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a `.jsonl` file or directory in `agents.defaults.memorySearch.extraPaths`; current discovery, read, and watcher paths admit `.md` or enabled multimodal files only.

## Solution Assessment

No. The title's extension-list-only patch would be incomplete; the maintainable solution needs format-aware JSONL handling plus aligned read, watch, docs, changelog, and regression coverage.
