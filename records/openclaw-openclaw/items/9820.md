## Summary

Keep open. Current main still filters arbitrary `.jsonl` files out of builtin `memorySearch.extraPaths`, and the request is a new ingestion mode that needs maintainer product/API direction rather than cleanup closure.

## What This Changes

Request native JSONL support in builtin `memorySearch.extraPaths` so raw hook message logs can be semantically indexed without converting them to Markdown.

## Best Possible Solution

Adopt one documented path: implement JSONL as a first-class extra-path source across discovery, parsing, watching, reads, docs, and tests, or document QMD/session-memory/converter/plugin as the supported workflow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is an enhancement request; source inspection of current main gives high confidence that arbitrary `.jsonl` extra paths are still filtered or rejected.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native JSONL support may be reasonable, but the proposed whitelist is incomplete without JSONL parsing/chunking semantics, line mapping, `memory_get`, watcher behavior, docs, and tests.
