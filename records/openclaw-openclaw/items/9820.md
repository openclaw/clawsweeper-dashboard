## Summary

Keep open. Current main and the latest release still reject arbitrary `.jsonl` files from builtin `memorySearch.extraPaths`; the separate session-transcript JSONL path does not satisfy this requested extra-path ingestion behavior.

## What This Changes

The issue requests native JSONL ingestion for `memorySearch.extraPaths` so raw message logs can be indexed without converting them to Markdown.

## Best Possible Solution

Adopt one documented path: either implement JSONL as a first-class extra-path source across discovery, parsing, watching, reads, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Yes, as a source-level behavior check: current main admits Markdown or enabled multimodal extra-path files and rejects ordinary `.jsonl` in discovery, watching, and `memory_get`. I did not run a live indexing command because this read-only review should not generate index artifacts.

## Solution Assessment

Unclear. Native JSONL support may be the right product direction, but the proposed extension whitelist is not enough without JSONL parsing, chunking, line mapping, watcher/read behavior, docs, and privacy semantics.
