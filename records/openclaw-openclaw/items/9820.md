## Summary

Keep open. Current main and latest release still do not support arbitrary JSONL files through builtin `memorySearch.extraPaths`; the code only admits Markdown or enabled multimodal files there, while JSONL parsing is limited to the separate session transcript path.

## What This Changes

Add builtin memory-indexer support for `.jsonl` files supplied through `memorySearch.extraPaths` so raw message-hook logs can be searched without conversion.

## Best Possible Solution

Define the JSONL memory-file contract first, then either implement it consistently across builtin admission, parsing/chunking, watching, `memory_get`, docs, and tests, or document the supported QMD/session-memory/converter/plugin workflow.

## Reproduction Assessment

Yes, at source level. Current main filters builtin extra paths to Markdown or enabled multimodal files, rejects non-Markdown `memory_get` targets, and reserves JSONL parsing for the separate session transcript path.

## Solution Assessment

Unclear until maintainers decide the product boundary. Native JSONL support is plausible, but the maintainable solution is broader than adding `.jsonl` to an extension list and may instead be a documented QMD/session-memory/converter/plugin workflow.
