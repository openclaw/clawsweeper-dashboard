## What This Changes

The issue requests that `openclaw update` update the active custom `OPENCLAW_HOME`/pnpm install instead of installing a second copy under the default global prefix.

## Best Possible Solution

Keep the merged pnpm global-root implementation on main and let the next release after `v2026.5.7` carry it to affected users.

## Reproduction Assessment

No current-main failure reproduces from source inspection: `resolveGlobalInstallTarget` now derives the owning pnpm global root from the running package root and metadata. The focused regression test covers the Windows pnpm `--global-dir` update path.

## Solution Assessment

Yes. The merged solution is the narrow package-manager seam: infer valid pnpm global roots from package layout/metadata and pass `--global-dir`, without redefining `OPENCLAW_HOME` as a broad install-prefix contract.
