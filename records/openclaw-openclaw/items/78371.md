## Summary

Keep open: current main still resolves package-update targets from the active package manager's global root, so a custom Windows/pnpm OpenClaw root can be bypassed instead of updating the package tree the CLI is running from.

## What This Changes

Request that `openclaw update` respect a custom OpenClaw install location from `OPENCLAW_HOME` or the current invocation root instead of installing into another global prefix.

## Best Possible Solution

Preserve the actual running package root/custom global prefix during package updates, add Windows/pnpm regression coverage, and document any supported custom-prefix behavior if the fix exposes it.

## Reproduction Assessment

Yes, at source level: when the current package root differs from `<manager> root -g`/`openclaw`, current main will compute the update target from the manager global root instead of the running package root. I did not execute a live Windows update in this read-only review.

## Solution Assessment

Yes, the maintainable direction is a focused package-update target resolution fix plus regression tests, not a new CLI flag or general OPENCLAW_HOME policy change.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78371. `openclaw update` must update the OpenClaw package tree the CLI is actually running from when a package install lives under a custom Windows/pnpm/global prefix, instead of recomputing a different npm/pnpm global root and installing there. Keep `OPENCLAW_HOME` semantics as state/home; do not add a new config option or blindly treat every OPENCLAW_HOME as an install prefix. In the package update target code, derive or preserve an unambiguous current package root/prefix from `params.root`, argv1, and package-manager contracts; handle npm staged swaps and pnpm custom `globalDir`/`globalBinDir` safely. Add regression tests for a current package root like `D:\openclaw\node_modules\openclaw` where `npm root -g` or `pnpm root -g` would otherwise point elsewhere, plus existing npm prefix behavior. Add a user-facing CHANGELOG.md fix entry. Do not change unrelated update-channel, service restart, plugin sync, or installer behavior.
