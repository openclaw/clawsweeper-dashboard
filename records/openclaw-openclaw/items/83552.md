## Summary

Keep this PR open: it addresses a real harness-selection crash path, and the PR body includes useful runtime log proof, but the alias guard is too broad because it routes any known CLI alias to PI without checking that the alias belongs to the selected provider/model.

## What This Changes

The PR imports `isCliRuntimeAlias` and routes explicit CLI runtime aliases in `selectAgentHarnessDecision` to the built-in PI harness with a new selection reason.

## Best Possible Solution

Keep the CLI-alias crash fix, but validate CLI runtime aliases against their owning provider/model before selecting PI and preserve fail-closed behavior for invalid explicit runtime policies.

## Reproduction Assessment

Yes, from source inspection: current main resolves a model-scoped `agentRuntime.id: "claude-cli"` into a non-auto runtime and `selectAgentHarnessDecision` throws `MissingAgentHarnessError` because no harness with that id is registered. I did not run a live compaction repro in this read-only review.

## Solution Assessment

No, not as written: routing CLI aliases away from harness lookup is the right direction, but the guard should verify the alias belongs to the selected provider/model before bypassing the fail-closed path.
