## Summary

Keep open. The PR is aimed at a real source-backed crash path for built-in CLI runtime aliases, but the guard is narrower than the documented CLI-backend contract and the posted proof is still too thin for the failing real-behavior-proof gate.

## What This Changes

The PR adds CLI runtime alias handling in `src/agents/harness/selection.ts` so `claude-cli` and `google-gemini-cli` select the PI harness instead of throwing `MissingAgentHarnessError`.

## Best Possible Solution

Route every supported CLI backend alias to PI at the harness-selection boundary while preserving fail-closed behavior for embedded plugin harness ids, with focused regression coverage for built-in and custom CLI aliases.

## Reproduction Assessment

Yes, by source inspection. Current main resolves configured CLI runtime ids into the harness-selection path, where `claude-cli` is not a registered plugin harness and reaches `MissingAgentHarnessError`; the PR body also includes before/after WSL2 log snippets for the built-in case.

## Solution Assessment

No, not yet. Routing CLI aliases away from embedded harness lookup is the right direction, but the implementation should use supported CLI-backend detection rather than only the static built-in alias table.
