## Summary

Keep this PR open: the reported runtime crash is credible and the proof is sufficient, but the patch still has a routing-safety bug because it treats any known CLI alias as PI without confirming that the alias belongs to the selected provider/model.

## What This Changes

The PR adds CLI runtime alias handling to `selectAgentHarnessDecision` so `claude-cli` and `google-gemini-cli` select the PI harness instead of throwing `MissingAgentHarnessError`.

## Best Possible Solution

Keep the CLI-alias crash fix, but validate aliases through the provider-scoped runtime ownership map before selecting PI and preserve `MissingAgentHarnessError` for mismatches.

## Reproduction Assessment

Yes. Source inspection shows a model-scoped `agentRuntime.id: "claude-cli"` reaches `selectAgentHarnessDecision` as a non-auto runtime and throws because no harness has that id; the PR body and follow-up comment add real local repro/proof, but I did not run a live compaction repro in this read-only review.

## Solution Assessment

No, not as written. Routing CLI aliases away from plugin harness lookup is the right direction, but the branch should verify the alias belongs to the selected provider/model before bypassing the fail-closed path.
