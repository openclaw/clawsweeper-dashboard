## Label Justifications

- `P1`: The PR addresses a confirmed stable-session failure where configured CLI-runtime sessions can fail compaction or inbound agent runs.
- `merge-risk: 🚨 compatibility`: The proposed fallback changes the behavior of invalid explicit runtime configs from fail-closed to PI routing.
- `merge-risk: 🚨 auth-provider`: A mismatched CLI alias can alter provider/auth routing by running the direct PI provider path instead of the intended CLI backend.

## Summary

Keep open: the reported CLI-runtime compaction bug is real, but the patch should not merge until the new PI fallback is limited to provider-owned CLI aliases so invalid runtime config still fails closed.

## What This Changes

The PR adds a CLI runtime alias branch in `src/agents/harness/selection.ts` so explicit `claude-cli` or `google-gemini-cli` runtime policies select PI instead of throwing.

## Best Possible Solution

Keep the PI fallback for provider-owned CLI aliases, preserve `MissingAgentHarnessError` for mismatched explicit aliases, and add focused selection coverage for valid and invalid alias pairs.

## Reproduction Assessment

Yes. Current main’s selector throws for explicit unregistered `claude-cli`, and the PR body plus follow-up comment include real stable-runtime logs showing the before error and after-fix continuation.

## Solution Assessment

No. The direction is right for matching CLI aliases, but the current branch is too broad because it does not verify that the alias belongs to the selected provider/model before routing to PI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate CLI runtime ownership before PI fallback:** `src/agents/harness/selection.ts:174`
  - body: `isCliRuntimeAlias(runtime)` only checks the runtime string. With a mismatched policy such as an OpenAI model pinned to `claude-cli`, `resolveCliRuntimeExecutionProvider` would not choose a CLI backend, but this branch would return PI and allow the run to continue on the wrong provider/auth path instead of failing closed for invalid explicit config. Please require the alias to match the selected provider before returning PI and cover the mismatch case.
