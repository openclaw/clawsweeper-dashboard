## Summary

Keep open: current main still has the reported isolated-cron timeout path, and the linked open PR only improves the provider timeout hint instead of fixing the runtime behavior.

## What This Changes

The issue reports a regression where isolated cron jobs using `openai/gpt-4o-mini` hit the model idle watchdog after an upgrade, with confusing timeout guidance in the surfaced error.

## Best Possible Solution

Keep cron `timeoutSeconds` as the outer wall-clock deadline, avoid feeding it into the model idle watchdog for cron-triggered embedded runs, and land the provider-id error-message improvement as a companion fix.

## Reproduction Assessment

Yes, at source level. A positive isolated-cron `payload.timeoutSeconds` reaches `runEmbeddedPiAgent` as `params.timeoutMs`, then `resolveLlmIdleTimeoutMs` returns the capped run timeout before the cron no-default-timeout branch can run.

## Solution Assessment

No. Current main is not the desired end state, and the linked PR only fixes the hint text; the narrow maintainable fix is to separate cron outer runtime timeout from model idle-timeout selection.
