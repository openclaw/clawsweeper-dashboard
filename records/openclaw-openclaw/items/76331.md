## Summary

Keep open: current main still routes isolated-cron `timeoutSeconds` into the embedded model idle watchdog, and the related PR was closed unmerged after addressing only the confusing config-key hint.

## What This Changes

The issue reports a regression where isolated cron sessions using `openai/gpt-4o-mini` time out after the 2026.4.26 update, with a misleading model idle-timeout config hint.

## Best Possible Solution

Separate cron outer wall-clock `timeoutSeconds` from model idle-watchdog selection for cron-triggered embedded runs, then update the timeout hint as companion cleanup.

## Reproduction Assessment

Yes, at source level. A positive isolated-cron `timeoutSeconds` is converted to `timeoutMs`, forwarded into `runEmbeddedPiAgent`, then treated as explicit idle-timeout input before the cron branch can disable the idle watchdog.

## Solution Assessment

No. Current main is not the best end state; the narrow maintainable fix is to preserve cron `timeoutSeconds` as the outer run deadline while preventing it from shrinking or enabling the model idle watchdog.

## Repair Work Prompt

Fix the isolated cron model idle-timeout regression reported in https://github.com/openclaw/openclaw/issues/76331. Keep cron `timeoutSeconds` as the outer wall-clock deadline for cron-triggered embedded runs, but do not pass that cron deadline into `resolveLlmIdleTimeoutMs` in a way that enables or caps the model idle watchdog; cron with no model/provider idle timeout should continue to disable the idle watchdog. Add regression coverage for a positive isolated-cron `timeoutSeconds` path, keep the fix generic to cron/embedded-runner behavior, and do not add a new config option or OpenAI-specific special case. If updating the runtime error hint, substitute the actual provider id instead of the literal `<id>` placeholder, but treat that as companion cleanup rather than the core fix. Add a `CHANGELOG.md` fix entry if production behavior changes.
