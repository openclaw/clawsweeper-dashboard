## Summary

Keep open: current main still routes positive isolated-cron `timeoutSeconds` into the LLM idle watchdog before the cron disable branch, so the reported runtime timeout path remains. The related PR was closed unmerged and only addressed the error-message hint, not the cron timeout behavior.

## What This Changes

The issue asks for isolated cron runs with `gpt-4o-mini` to stop failing on model idle timeouts after the 2026.4.26 regression and for the misleading timeout config hint to be corrected.

## Best Possible Solution

Separate cron's outer `timeoutSeconds` deadline from model idle-timeout selection, preserve provider/model timeout knobs, add regression coverage for positive cron timeouts, and update the idle-timeout hint with a concrete provider key.

## Reproduction Assessment

Yes, at source level. A positive isolated-cron `payload.timeoutSeconds` becomes `timeoutMs`, is forwarded to `runEmbeddedPiAgent`, and is passed as `runTimeoutMs` to `resolveLlmIdleTimeoutMs`, which returns the capped run timeout before the cron no-watchdog branch.

## Solution Assessment

No. Current main is not the best end state, and the closed related PR only improved the hint text; the narrow maintainable fix is to stop treating cron's outer deadline as the model idle watchdog while preserving provider request timeouts.

Fix https://github.com/openclaw/openclaw/issues/76331.
