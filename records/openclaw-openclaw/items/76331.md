## Summary

Keep open: current main partially avoids the idle watchdog for cron runs with no explicit timeout, but a positive isolated-cron `timeoutSeconds` still flows into the embedded runner and is converted into a model idle timeout capped at 120s, matching the reporter's observation that increasing cron timeout does not prevent the model idle-timeout failure.

## What This Changes

Fix isolated cron jobs using `openai/gpt-4o-mini` so model idle timeouts do not break upgraded runs and timeout guidance points to actionable configuration.

## Best Possible Solution

Keep cron `timeoutSeconds` as the outer job runtime limit, use provider request timeout as the model idle watchdog only when configured, and update tests/docs/error guidance so OpenAI cron jobs are not told to edit auth profiles for request timing.

## Reproduction Assessment

Yes, at code level. A positive isolated cron `payload.timeoutSeconds` reaches `runEmbeddedPiAgent` as `timeoutMs`, and `resolveLlmIdleTimeoutMs` currently turns that into a capped 120s idle watchdog before the cron-specific disable branch can run.

## Solution Assessment

No, current main is not the best end state for this report. The narrower maintainable fix is to distinguish cron outer runtime timeout from model stream idle timeout while preserving provider-level request timeout semantics.
