## Summary

Keep open: current main still has the reported isolated-cron timeout path, and the linked open PR only improves the timeout hint instead of fixing the runtime behavior.

## What This Changes

Cron jobs with `sessionTarget: "isolated"` and `payload.timeoutSeconds` should not fail early with the model idle-timeout watchdog or point users at the wrong config surface.

## Best Possible Solution

Keep cron `timeoutSeconds` as the outer wall-clock deadline, avoid feeding it into the model idle watchdog for cron-triggered embedded runs, and land any error-message improvement only as a companion fix.

## Reproduction Assessment

Yes, at source level. A positive isolated-cron `payload.timeoutSeconds` reaches `runEmbeddedPiAgent` as `timeoutMs`, then `resolveLlmIdleTimeoutMs` converts it to the capped idle watchdog before the cron no-default-timeout branch can run.

## Solution Assessment

No. Current main is not the best end state, and #76422 only improves the hint; the narrow maintainable fix is to separate cron outer runtime timeout from model idle-timeout selection.
