## Summary

Keep open: current main still has the source-level timeout path where isolated cron `timeoutSeconds` can become the model idle watchdog before the cron no-default-idle branch runs, and the linked open PR only addresses the error-message placeholder.

## What This Changes

The issue reports a regression where isolated cron agent-turn jobs using `openai/gpt-4o-mini` hit the model idle watchdog after the 2026.4.26 update, with confusing timeout guidance.

## Best Possible Solution

Keep cron `timeoutSeconds` as the outer wall-clock deadline, keep cron-triggered embedded model idle behavior controlled by provider/request timeout instead of the job deadline, and land the clearer provider-id timeout hint as a companion fix.

## Reproduction Assessment

Yes, at source level. A cron `agentTurn` payload with a positive `timeoutSeconds` that differs from `agents.defaults.timeoutSeconds` is converted into `runTimeoutMs`, forwarded to `runEmbeddedPiAgent`, and selected by `resolveLlmIdleTimeoutMs` before the cron no-default-idle branch.

## Solution Assessment

No. Current main still conflates the cron outer deadline with the LLM idle watchdog, and #76422's message substitution is useful but not sufficient to restore isolated cron behavior.
