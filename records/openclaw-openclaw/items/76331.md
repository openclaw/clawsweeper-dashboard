## Summary

Keep open: current main still maps a positive isolated-cron timeout into the embedded runner's run timeout, and the idle-timeout resolver turns that into a capped model idle watchdog before its cron no-idle branch. The attached PR #76422 improves the placeholder in the error text but does not address the timeout handoff.

## What This Changes

The issue reports isolated cron jobs using openai/gpt-4o-mini failing with model idle timeouts after the 2026.4.26 update, plus confusing provider-timeout guidance.

## Best Possible Solution

Keep cron timeoutSeconds as the outer job deadline, let models.providers.<provider>.timeoutSeconds control provider/model idle timing, and add regression coverage for positive isolated-cron timeouts.

## Reproduction Assessment

Yes, source-level. A positive isolated agentTurn payload.timeoutSeconds reaches runEmbeddedPiAgent as params.timeoutMs, and resolveLlmIdleTimeoutMs converts that positive run timeout into a model idle watchdog before the cron disable branch can apply.

## Solution Assessment

No, current main is not the best solution, and #76422 only fixes the message placeholder. The narrow maintainable fix is to separate cron outer runtime timeout from the model idle watchdog while preserving provider request timeout semantics.
