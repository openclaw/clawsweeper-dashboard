## Summary

Keep open: current main still turns a positive isolated cron `timeoutSeconds` into the embedded runner `timeoutMs`, and the LLM idle-timeout resolver consumes that run timeout before the cron-specific disable branch, matching the reported 120s model-idle failure path.

## What This Changes

The issue requests restoring isolated cron runs with `openai/gpt-4o-mini` after an upgrade regression where jobs fail with a model idle-timeout message and confusing timeout-config guidance.

## Best Possible Solution

Keep cron `timeoutSeconds` as the outer job runtime limit, and let only provider request timeout or explicitly intended agent-level model timing control the LLM idle watchdog.

## Reproduction Assessment

Yes, at source level. A positive isolated cron payload timeout is forwarded into the embedded run and currently resolves to a capped model idle watchdog before the cron-specific disable branch can apply.

## Solution Assessment

No, current main is not the best end state. The narrower fix is to separate cron outer runtime timeout from model stream idle timeout while preserving `models.providers.<id>.timeoutSeconds` request-timeout semantics.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76331. Positive isolated cron `payload.timeoutSeconds` must remain the outer cron job runtime cap and must not implicitly become the model stream idle watchdog. Update the cron/embedded timeout handoff or `resolveLlmIdleTimeoutMs` so cron-triggered runs with a job-level timeout do not get the default-capped LLM idle watchdog unless a provider request timeout or intentionally configured agent-level timeout should apply. Preserve `models.providers.<id>.timeoutSeconds` semantics for provider HTTP/request and model idle timing, preserve non-cron behavior, and do not add a new config option. Add regression coverage for a cron trigger with a positive run timeout, update any affected docs/error guidance if the distinction changes, and add a user-facing `CHANGELOG.md` fix entry.
