## Summary

Keep open: current main still lets isolated-cron wall-clock `timeoutSeconds` flow into the model idle watchdog, and the related PR is closed unmerged after addressing only the error hint.

## What This Changes

The issue reports that isolated cron sessions using `openai/gpt-4o-mini` fail with the model idle-timeout watchdog after upgrading from 2026.4.24 to 2026.4.26, with a confusing timeout config hint.

## Best Possible Solution

Keep cron `timeoutSeconds` as the scheduler-owned outer deadline, prevent it from selecting the model idle watchdog for cron-triggered embedded runs, and land the provider-id error hint as a companion cleanup.

## Reproduction Assessment

Yes at source level. A positive isolated-cron `payload.timeoutSeconds` reaches `runEmbeddedPiAgent` as `timeoutMs`, then `resolveLlmIdleTimeoutMs` consumes it before the cron no-default-watchdog branch can run.

## Solution Assessment

No. Current main still conflates cron wall-clock timeout with model idle timeout, and #76422 only addressed the hint text; the narrow maintainable fix is to separate those timeout roles while preserving provider request timeout behavior.
