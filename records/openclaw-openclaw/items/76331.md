## Summary

Keep open: current main still has a source-level isolated-cron path where cron `timeoutSeconds` can become the model idle watchdog input, and the linked open PR only improves the timeout hint text rather than fixing the runtime behavior.

## What This Changes

The issue reports a regression where isolated cron jobs using `openai/gpt-4o-mini` fail with a model idle timeout after upgrading from 2026.4.24 to later releases, with confusing config-path guidance in the error.

## Best Possible Solution

Treat cron `timeoutSeconds` as the scheduler wall-clock deadline only, keep provider/model request timeout knobs as the model idle-watchdog inputs, and add regression coverage for positive isolated-cron timeouts.

## Reproduction Assessment

Yes, at source level. A positive isolated-cron `payload.timeoutSeconds` is resolved to `timeoutMs`, forwarded to `runEmbeddedPiAgent`, then passed into `resolveLlmIdleTimeoutMs` before the cron no-default-watchdog branch can run.

## Solution Assessment

No. Current main still preserves the timeout coupling, and #76422 only fixes the user-facing config-path hint; the runtime fix should separate cron outer timeout handling from model idle-timeout selection.
