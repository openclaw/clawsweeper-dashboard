## Summary

Keep open: the carrier-unwrapping direction is useful, but this PR is not merge-ready because the no-script-validated fallback is still raw-command based, carrier-unwrapped path-qualified interpreters can still bypass script target detection, and the external PR only provides test verification rather than real behavior proof.

## What This Changes

The PR changes agent exec script preflight to recursively inspect sudo/env/exec and shell-wrapper carrier chains, adds a fail-closed branch when no script was validated, and adds regression tests for readable carrier-wrapped Python/Node scripts.

## Best Possible Solution

Keep the recursive carrier-unwrapping hardening, repair the carrier-aware fallback and path-qualified interpreter gaps, add focused regressions for missing/outside/unreadable and absolute-interpreter carrier cases, and require redacted real exec/preflight proof before merge.

No: recursive carrier unwrapping is the right owner-boundary direction, but this implementation is not complete enough for the stated hardening. The narrow fix is to reuse carrier-aware context in the fallback and normalize unwrapped interpreter executables before target matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Make the no-script fallback carrier-aware:** `src/agents/bash-tools.exec.ts:1338`
  - body: When recursive extraction finds a script through `sudo`/`env` plus a shell wrapper but every candidate is missing, outside the workdir, or unreadable, `checkedAnyScript` stays false and this branch rechecks the raw command. That raw heuristic does not unwrap the carrier, so a command like `sudo -EH bash -lc 'python missing.py'` can still execute instead of failing closed.
  - confidence: 0.87
- **[P2] Normalize carried interpreter executable tokens:** `src/agents/bash-tools.exec.ts:591`
  - body: After unwrapping a carrier, this direct-target check still delegates to a parser that compares the raw executable token, so `sudo -u openclaw /usr/bin/python bad.py` or `/usr/bin/node bad.js` will not be recognized and the unsafe script is never preflighted. Reuse the basename/suffix normalization used by the carrier helpers before matching Python and Node executables.
  - confidence: 0.82
