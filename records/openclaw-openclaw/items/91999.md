## Label Justifications

- `P1`: An established terminal workflow can silently route subsequent turns to the wrong model after the user explicitly changes it.
- `impact:auth-provider`: The reported race replaces the exact user-selected provider/model with a stale Gemini selection.

## Summary

Keep open: current `main` has a credible session-store race where a model change persisted by `session_status` during an active turn can be overwritten by that turn’s stale end-of-run session metadata. The report describes broken documented behavior, no merged or open linked PR owns the fix, and adjacent reports cover different session-override problems.

## What This Changes

Preserve a model selected through `session_status(model=...)` during an active terminal turn when end-of-run session metadata is persisted.

## Best Possible Solution

Change shared end-of-run persistence to merge only fields owned by the completed run onto the latest stored session entry, preserving concurrent user model, auth-profile, and session-setting changes; cover the real `session_status` transaction and sibling `/model` or `sessions.patch` paths with focused regression tests.

## Reproduction Assessment

Yes at high source confidence: invoke `session_status(model=...)` during an active turn, then allow normal end-of-run persistence to merge its stale broad session entry. A focused automated reproduction should be added because this read-only review did not execute the runtime path.

## Solution Assessment

Yes, a concurrency-safe repair in the shared session-store boundary is the narrowest maintainable solution. A terminal-only workaround, provider-specific exception, or new configuration option would leave sibling callers and the underlying stale-write race unresolved.
