## Summary

Keep open. Current main has partial mitigations through phased assistant text suppression and `messages.visibleReplies`, but it still leaves direct chats in automatic visible-delivery mode by default, lacks the requested per-agent `directives.autoReply=false` contract, and has no shared public/internal assistant audience boundary. The related #74928 is the broader open tracker, but this issue still adds concrete Feishu/direct-chat context.

## What This Changes

The issue requests suppressing automatic channel delivery of assistant text when the same assistant response continues into tool calls, either through per-agent auto-reply control or final-only delivery.

## Best Possible Solution

Define a shared visible-delivery contract that separates internal/runtime assistant text from public channel text, then apply it consistently across provider stream parsing, partial previews, block streaming, final delivery, replay/history helpers, and channel dispatchers. If #74928 becomes canonical, preserve this issue’s Feishu/direct-chat reproduction context before closing this one as superseded.

## Reproduction Assessment

Partly. Source and tests give a high-confidence reproduction of the residual policy gap: direct chats default to automatic visible delivery, while message-tool-only mode suppresses automatic callbacks and finals only when explicitly active; the exact Feishu live mixed text-then-tool-call leak still needs a focused fixture or live channel check.

## Solution Assessment

No. The current `messages.visibleReplies` and phase-based commentary suppression are useful mitigations, but they are not the requested per-agent or final-only public delivery boundary and do not add the explicit public/internal audience contract tracked by #74928.
