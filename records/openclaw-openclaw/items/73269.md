## Summary

Keep open. Current main still only partially mitigates the report: group/channel rooms default to message-tool-only visible delivery, but direct/source chats still default to automatic visible replies, Feishu still sends automatic final/plain text in that mode, and the requested per-agent `autoReply=false` or explicit public/internal assistant-audience boundary is not present. Related #74928 is the broader boundary tracker, but this issue still carries concrete Feishu/direct-chat delivery context.

## What This Changes

The issue asks OpenClaw to stop auto-sending assistant text to channels when tool calls follow, either by adding per-agent auto-reply suppression or by delivering only final public assistant text after tool execution completes.

## Best Possible Solution

Define a shared visible-delivery contract that separates internal/runtime assistant text from public channel text, then apply it consistently across partial previews, block streaming, tool progress, final delivery, replay, and channel dispatchers. If maintainers choose #74928 as canonical, preserve this issue’s Feishu/direct-chat context there before closing it as superseded.

## Reproduction Assessment

Partly. Current source and tests provide a high-confidence reproduction of the residual policy gap: direct chats default to automatic delivery while `message_tool_only` suppresses automatic callbacks and final delivery; the exact Feishu mixed text-then-tool-call report still needs a focused fixture or live check.

## Solution Assessment

No. The current `messages.visibleReplies` mitigation is useful but not the complete requested fix because it is global/direct opt-in, not per-agent, and it does not add an explicit public/internal or final-only assistant-text boundary.
