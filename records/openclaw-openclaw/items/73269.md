## Summary

Keep open. Current main has useful mitigations through phased commentary suppression and global `messages.visibleReplies`, but it still has no per-agent `directives.autoReply=false` contract or shared public/internal assistant audience boundary, and direct chats still default to automatic visible delivery.

## What This Changes

The issue asks OpenClaw to stop automatically sending assistant text to channels when the same response also performs tool calls, either via per-agent auto-reply suppression or final-only delivery.

## Best Possible Solution

Define and document a shared channel-visible delivery contract that distinguishes public assistant text from internal/runtime narration, then apply it across provider stream parsing, transcript replay, visible extraction helpers, final delivery, and channel dispatchers. Once #74928 or a narrower PR carries that contract, preserve this issue's Feishu/direct-chat context in the canonical tracker and close this one as implemented or superseded.

## Reproduction Assessment

Partly. Source and tests provide a high-confidence reproduction of the remaining policy gap: direct chats default to automatic visible replies and no per-agent auto-reply directive exists, while the exact live Feishu mixed text/tool-call leak still needs a focused fixture or live channel check.

## Solution Assessment

No. The current phase and `messages.visibleReplies` mitigations reduce leaks, but they are not the requested per-agent or final-only visible-delivery contract and do not add the explicit public/internal audience boundary tracked by #74928.
