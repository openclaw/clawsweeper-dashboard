## Summary

Keep open. Current main still has only a partial visible-reply mitigation: group/channel turns default to message-tool-only delivery, but default direct/source chats still auto-deliver, Feishu still wires streaming partial callbacks in automatic mode, and there is no per-agent auto-reply or public/internal assistant audience boundary.

## What This Changes

The issue requests suppressing automatic channel delivery of assistant text when tool calls follow, either through per-agent auto-reply control or final-only/public visible-delivery semantics.

## Best Possible Solution

Define one shared visible-delivery contract that distinguishes public assistant text from internal/runtime text and applies consistently across preview, block, tool-progress, and final delivery. If #74928 becomes canonical, preserve this issue’s Feishu/direct-chat reproduction context there before closing this as superseded.

## Reproduction Assessment

Yes for the remaining policy gap. A default direct Feishu/source turn keeps automatic delivery and streaming partial callbacks active, while the existing tests show only `message_tool_only` mode suppresses those callbacks and final delivery.

## Solution Assessment

No. The current global `messages.visibleReplies` mitigation is useful, but it is not the complete fix because it is global/direct opt-in, not per-agent, and it does not add a public/internal or final-only assistant text boundary.
