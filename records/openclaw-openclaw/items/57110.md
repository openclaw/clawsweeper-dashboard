## Summary

Keep open. Current main and v2026.6.10 still have tokenless verified Tailscale Serve identity for the Control UI/WebSocket path and no optional secondary token/password-auth contract; the remaining work is security-sensitive product/config design, not already implemented or superseded.

## What This Changes

The issue requests opt-in secondary token/password authentication and clearer warnings/docs when Tailscale Serve identity is trusted for Gateway access.

## Best Possible Solution

Define a maintainer-approved layered Gateway auth contract, then update config types/schema/help, WebSocket auth enforcement, warning or audit messaging, public docs, and focused auth tests together.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 both accept verified Tailscale identity for ws-control-ui without token/password credentials when allowTailscale is enabled, and no live Tailscale runtime repro was run in this read-only sweep.

## Solution Assessment

Unclear. The requested opt-in flag is plausible, but maintainers should settle the permanent layered-auth contract instead of adding a one-off config key around one auth branch.
