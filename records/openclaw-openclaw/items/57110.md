## Summary

Keep open. Current main documents tokenless Tailscale Serve identity auth and a shared-secret workaround, but it still has no config, schema, or auth path that requires both verified Tailscale identity and token/password credentials; this remains a security-sensitive feature/design request rather than cleanup.

## What This Changes

Request an opt-in Gateway auth policy that requires token/password credentials in addition to verified Tailscale Serve identity, plus warnings and docs for the trust boundary.

## Best Possible Solution

Define an explicit Gateway auth policy for requiring both verified Tailscale Serve identity and shared-secret credentials, with schema/help metadata, docs, audit messaging, and regression tests across Control UI/WebSocket and HTTP behavior.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the Control UI/WebSocket auth path can accept verified Tailscale Serve identity without token/password when allowTailscale is enabled.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution needs a maintainer/security decision on the exact config contract and how it composes with existing token, password, trusted-proxy, Tailscale identity, and device-identity paths.
