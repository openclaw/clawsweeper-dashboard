## Summary

Keep open. Current main and v2026.6.8 still lack the requested optional layered Tailscale identity plus shared-secret auth contract; existing docs only offer disabling Tailscale identity and using token/password as the hardening path, so this remains security/config product work rather than stale cleanup.

## What This Changes

The issue asks for optional secondary token/password authentication and clearer warnings/docs when trusted Tailscale Serve identity is accepted without the built-in gateway credential layer.

## Best Possible Solution

Define a maintainer-approved layered Gateway auth contract, then implement schema/types/help metadata, auth enforcement, runtime warning or audit messaging, docs, and focused HTTP plus WebSocket tests.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the WS Control UI auth path accepts verified Tailscale identity without token/password credentials when allowTailscale is enabled.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix needs maintainer/security agreement on the permanent layered-auth contract rather than a one-off flag added directly to the current branch.
