## Summary

Keep open. Current main and the latest release still document and implement tokenless Tailscale Serve identity auth for Control UI/WebSocket when allowed, but there is no public Gateway config, schema, or auth path that requires both verified Tailscale identity and token/password credentials.

## What This Changes

The issue requests an opt-in Gateway policy to require token/password authentication in addition to Tailscale Serve identity, plus warnings and documentation for the trusted tailnet boundary.

## Best Possible Solution

Define an explicit layered Tailscale auth policy with config type/schema/help metadata, docs, warnings where appropriate, and regression coverage for Control UI WebSocket and HTTP Gateway behavior.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the Control UI/WebSocket auth path can accept verified Tailscale Serve identity without token/password credentials when allowTailscale is enabled.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution needs maintainer/security agreement on the exact public auth contract and how it composes with existing token, password, trusted-proxy, Tailscale identity, and device-identity paths.
