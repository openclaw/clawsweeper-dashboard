## Summary

Keep open. Current main documents and partially mitigates the Tailscale Serve trust boundary, but it still does not implement the requested identity-plus-token/password secondary-auth mode; the existing workaround disables Tailscale identity auth instead of layering it with a shared secret.

## What This Changes

Request optional secondary token/password authentication plus clearer warnings/docs for Tailscale Serve identity-based Gateway access.

## Best Possible Solution

Define an explicit, documented auth policy for Tailscale Serve that can require both verified Tailscale identity and a token/password on the intended surfaces, with schema/docs/audit coverage and clear compatibility behavior for existing `allowTailscale` deployments.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `authorizeWsControlUiGatewayConnect` with Tailscale forwarded headers, `allowTailscale: true`, and no `connectAuth` succeeds as `method: "tailscale"`; the existing auth tests assert that path.

## Solution Assessment

Unclear. The current workaround, `gateway.auth.allowTailscale: false` plus token/password mode, is maintainable but drops Tailscale identity; the requested layered mode needs a maintainer/security decision on the config contract and exact surfaces.
