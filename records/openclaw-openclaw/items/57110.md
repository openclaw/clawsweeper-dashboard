## Summary

Keep open. Current main documents the Tailscale Serve trust boundary and offers a workaround by disabling Tailscale identity auth, but it still lacks the requested identity-plus-token/password secondary-auth mode and no merged PR now owns that exact change.

## What This Changes

Add optional secondary token/password authentication, warnings, and docs for Tailscale Serve identity-based access.

## Best Possible Solution

Define an explicit Gateway auth policy that can require both verified Tailscale Serve identity and token/password credentials, with config schema, docs, audit messaging, and tests covering HTTP versus WS Control UI behavior.

## Reproduction Assessment

Yes. Source inspection and tests show the current WS Control UI path can accept verified Tailscale identity without token/password when allowTailscale is enabled; I did not run a live Tailnet E2E.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact config contract and whether it should compose with token mode, password mode, or trusted-proxy mode needs maintainer/security design first.
