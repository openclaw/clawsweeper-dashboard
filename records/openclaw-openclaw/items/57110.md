## Summary

Keep open. Current main and v2026.5.22 still lack an identity-plus-shared-secret Gateway auth contract for Tailscale Serve; the documented workaround disables Tailscale identity auth instead of layering token/password on top, so this needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue requests optional secondary token/password authentication plus clearer warnings/docs for Tailscale Serve identity-auth deployments.

## Best Possible Solution

Define an explicit layered Tailscale auth policy in core Gateway auth, with config type/schema/help metadata, docs, warnings where appropriate, and regression coverage for Control UI WebSocket and HTTP behavior.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the ws-control-ui auth path can accept verified Tailscale Serve identity without token/password credentials when allowTailscale is enabled.

## Solution Assessment

Unclear as a final design. The requested direction is plausible, but the existing workaround disables Tailscale identity rather than layering credentials, so maintainers need to choose the public auth contract before implementation.
