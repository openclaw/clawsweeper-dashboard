## Summary

Keep open. Current main documents the tokenless Tailscale Serve trust boundary and offers `gateway.auth.allowTailscale: false` as a hardening workaround, but it does not implement the requested mode where verified Tailscale identity and token/password auth are both required and preserved.

## What This Changes

The issue requests optional secondary token/password authentication on top of Tailscale Serve identity, plus clearer runtime/docs warnings about tokenless tailnet trust.

## Best Possible Solution

Define a maintainer-approved additive Gateway auth contract that can require shared-secret auth while preserving verified Tailscale identity for the Control UI/WebSocket Serve path, then update config types/schema/help, focused gateway auth tests, startup or runtime warnings, and public Tailscale/security docs. If maintainers do not want that combined mode, the docs should explicitly say `allowTailscale: false` is the supported defense-in-depth path and that Tailscale identity will not be retained as the user identity in that mode.

## Reproduction Assessment

Yes. The source-level reproduction is a WebSocket Control UI auth request with `gateway.auth.allowTailscale: true`, Tailscale forwarded headers, and no token/password: current tests and code accept it as `method: "tailscale"`; providing a shared secret instead takes the token/password path rather than requiring both.

## Solution Assessment

Unclear. The requested defense-in-depth direction is valid, but the proposed `auth.mode="trusted-proxy"` shape is probably not the best fit for current main because Tailscale Serve auth is controlled by `allowTailscale` and trusted-proxy currently rejects shared token configuration.
