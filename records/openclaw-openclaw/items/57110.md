## Summary

Keep open. Current main documents the Tailscale Serve trust boundary and the workaround of disabling Tailscale identity auth, but it still has no identity-plus-token/password secondary-auth contract; the related open work at https://github.com/openclaw/openclaw/issues/50630 and https://github.com/openclaw/openclaw/pull/50631 covers the separate `auth.mode="none"` Serve exposure, not this layered-auth feature.

## What This Changes

Add an opt-in Gateway auth setting that requires token/password authentication in addition to Tailscale Serve identity, plus clearer runtime/docs warning for tokenless tailnet trust.

## Best Possible Solution

Define an explicit Gateway auth policy for requiring both verified Tailscale Serve identity and shared-secret credentials, with schema/help metadata, docs, audit messaging, and regression tests across WS Control UI and HTTP behavior.

## Reproduction Assessment

Yes, at source level. Current main's WS Control UI auth can accept verified Tailscale Serve identity without connectAuth when `allowTailscale` is enabled, and focused tests assert that behavior; no live Tailnet E2E was run.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable fix needs a maintainer/security decision on the config contract and how it composes with existing token, password, trusted-proxy, and device-identity paths.
