## Summary

Keep open. Current main and v2026.6.6 still document and implement tokenless Tailscale Serve identity auth for Control UI/WebSocket when `allowTailscale` is enabled, but there is no config/schema/auth path that requires verified Tailscale identity plus a token/password.

## What This Changes

Request an opt-in secondary token/password authentication layer, warning, and docs hardening for Tailscale Serve identity-based gateway access.

## Best Possible Solution

Define a maintainer-approved layered Gateway auth contract, then implement the config type/schema/help metadata, auth enforcement, warning/audit/docs updates, and focused tests for Control UI WebSocket plus HTTP behavior.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the WS Control UI path can accept verified Tailscale Serve identity without token/password credentials when `allowTailscale` is enabled.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix needs maintainer/security agreement on the permanent auth contract rather than a one-off flag added directly to the current branch.
