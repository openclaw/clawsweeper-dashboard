## Summary

Keep open. Current main documents the Tailscale Serve trust boundary and the workaround of disabling Tailscale identity auth, but it still has no identity-plus-token/password secondary-auth contract; the linked no-auth Serve issue and PR cover a separate fail-closed startup gap.

## What This Changes

Requests optional secondary authentication for Tailscale Serve so verified Tailscale identity can be combined with token/password auth, plus clearer warnings and docs.

## Best Possible Solution

Define an explicit Gateway auth policy for requiring both verified Tailscale Serve identity and shared-secret credentials, with schema/help metadata, docs, audit messaging, and regression tests across WS Control UI and HTTP behavior.

## Reproduction Assessment

Yes, at source level. Current main has no secondary-auth config/schema field, and the Control UI/WebSocket auth path can accept verified Tailscale Serve identity without token/password when allowTailscale is enabled; no live Tailnet E2E was run.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution needs a maintainer/security decision on the exact config contract and how it composes with existing token, password, trusted-proxy, Tailscale identity, and device-identity paths.
