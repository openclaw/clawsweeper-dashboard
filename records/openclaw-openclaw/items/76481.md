## Summary

Keep open. Current main still lacks a gateway startup-planner path that activates the bundled plugin owning the configured TTS speech provider, and open PR #76540 explicitly closes this issue with a focused startup-planner fix, so the issue should remain open until that PR or an equivalent fix merges.

## What This Changes

Fix the 2026.5.2 regression where selecting `messages.tts.provider: "microsoft"` leaves the Microsoft speech plugin loaded/enabled in metadata but absent from the gateway startup plugin set, breaking TTS voice delivery.

## Best Possible Solution

Land PR #76540 or an equivalent manifest-contract-based startup-planner fix that activates only the configured speech-provider owner while honoring provider/plugin disablement and preserving lazy behavior elsewhere.

## Reproduction Assessment

Yes. The reporter supplied concrete 2026.5.2 steps/logs, and current main source still shows no configured speech-provider activation branch in the gateway startup planner; I did not run the live Telegram delivery path.

## Solution Assessment

Yes. The linked PR's direction is the narrow maintainable fix because it uses manifest-declared speech-provider ownership in startup planning instead of forcing all speech providers to load or changing synthesis/delivery behavior.
