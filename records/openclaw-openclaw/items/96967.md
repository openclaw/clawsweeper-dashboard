## Summary

Keep open: current main and v2026.6.10 still have the TUI `sessions.changed` `new`/`reset` handler unconditionally clearing tracked run state and loading history, while history replay re-adds assistant rows without run IDs. The earlier merged final-event guard at https://github.com/openclaw/openclaw/pull/88004 covers ordinary `chat.final` reloads, not this session-change path, and no open related PR is a proven canonical fix for this report.

## What This Changes

The issue reports that a Telegram-triggered assistant response appears twice in the TUI after a `sessions.changed` `reason: "new"` history reload replays the just-finalized assistant message.

## Best Possible Solution

Add a narrow TUI guard or reconcile path for post-final `sessions.changed` history reloads so a just-rendered assistant final is not duplicated or replaced, while preserving legitimate external `/new` and `/reset` clearing behavior.

## Reproduction Assessment

Yes at source level, but not as a live Telegram reproduction in this review. Current main and v2026.6.10 directly reload history on matching `sessions.changed` `new`/`reset` events, and history replay rebuilds assistant messages without run IDs; the exact duplicate timing still needs live proof.

## Solution Assessment

Unclear for the exact patch shape. The report points at the right failure class, but the best fix should not simply disable all `new`/`reset` reloads because those reloads are the shipped fix for external session resets.
