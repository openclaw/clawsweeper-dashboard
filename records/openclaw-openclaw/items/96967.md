## Summary

Keep open: current main and the latest release still have the TUI session-change history replay path that can duplicate a just-rendered assistant final, and the open linked PR is the active candidate fix but has not merged yet.

## What This Changes

The issue reports that openclaw-tui can render the same assistant reply twice when a Telegram-originated turn triggers a session history reload and a later live final event.

## Best Possible Solution

Review and land one canonical TUI fix, likely https://github.com/openclaw/openclaw/pull/96980 if maintainers accept it, then close this issue after the fix is merged or shipped.

## Reproduction Assessment

Yes, at source level: current main reloads history on matching `sessions.changed` events, history replay calls `finalizeAssistant` without a run ID, and live finals call it with a run ID. I did not run a live Telegram roundtrip in this read-only review.

## Solution Assessment

Yes for the current routing: keeping the issue open while the linked fix PR is reviewed is the right path. A fix should dedupe the replay/live-final interleaving without removing the external session reset reload behavior.
