## Summary

Keep open: this is a focused cron bug-fix PR, but source review shows the proposed guard is not on the timer path described by the PR, and the PR provides only code-review proof rather than real after-fix behavior proof.

## What This Changes

The PR adds an `isCronJobActive(job.id)` check to cron runnable-job collection so active jobs are skipped.

## Best Possible Solution

Reproduce the duplicate cron delivery on current main or the affected release line, then patch the actual timer/store scheduling path with focused regression coverage.

## Reproduction Assessment

No. The linked issue has user logs from app version 2026.4.15, but this review did not establish a current-main failing path; source inspection shows the PR's described re-arm window does not reach runnable-job collection while `state.running` is true.

## Solution Assessment

No. The added active-job check is not the narrow proven fix because the timer path returns before collection during active runs and clears active IDs before resetting `state.running`; the safer path is a repro-backed fix for the actual duplicate trigger mechanism.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Fix the actual timer re-arm path:** `src/cron/service/timer.ts:1175-1177`

A recheck timer firing while a job is active therefore will not reach this new branch, so the proposed change does not prove it prevents the reported duplicate execution.
