## Summary

Keep open: this PR attempts to address the duplicate cron report, but the added active-job check does not run in the timer re-arm window described by the PR, and the related bug remains better explained by the open canonical scheduler-reservation issue.

## What This Changes

The PR imports `isCronJobActive` and skips jobs already present in the in-memory active-job set during cron runnable-job filtering.

## Best Possible Solution

Reproduce the duplicate cron delivery on current main or the affected release, then fix the actual cron reservation path, likely around cross-process store locking, with focused regression coverage.

## Reproduction Assessment

Yes for the linked report at the source level, but not for this PR's stated timer re-arm explanation. Current source supports a two-process shared-store reservation race, while `state.running` prevents the same-process recheck path from reaching this new guard.

## Solution Assessment

No. The branch adds a same-process active-set check after an existing `runningAtMs` guard, but current timer flow does not reach that branch while a job is active; the maintainable fix should target the proven reservation path.
