## Summary

Keep this PR open with changes requested: the proposed active-job guard does not address the currently evidenced duplicate-trigger path, and the supplied proof only exercises a helper test rather than a real cron/timer run.

## What This Changes

The PR imports `isCronJobActive` into the cron runnable-job check and adds tests for the active-job helper functions.

## Best Possible Solution

Fix the linked scheduler bug in the cron store reservation path with cross-process-safe locking or atomic reservation, then add a scheduler-level regression and real cron-run proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the linked bug: two independent gateway/cron processes can race on the same store because the current lock and active-job Set are process-local. I did not run that live reproduction in this read-only checkout.

## Solution Assessment

No. The PR's process-local active-job guard is not the narrowest maintainable fix because current main already blocks same-process timer re-entry with `state.running` and persisted `runningAtMs`; the remaining credible bug is at cross-process store reservation.
