## Summary

Keep this PR open: the patch still does not address the linked duplicate-cron bug with high confidence. The added guard is process-local, the proposed root-cause explanation conflicts with current `main` timer flow, and the new test only exercises an existing helper rather than the scheduler path.

## What This Changes

The PR imports `isCronJobActive` into the cron runnable-job check and adds a helper-level active-job mark/clear test.

## Best Possible Solution

Fix cron job reservation and completion under a cross-process store lock, with a scheduler-level regression that reproduces the duplicate reservation path.

## Reproduction Assessment

Yes, by source inspection: run two independent gateway/cron processes against the same cron store and wake the same due job before either observes the other's persisted reservation. I did not execute that live path in this read-only review.

## Solution Assessment

No. A process-local `activeJobIds` check is not the narrow durable fix for the linked issue; the fix should protect the shared store reservation path and prove it at scheduler level.
