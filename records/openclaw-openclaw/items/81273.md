## Summary

Keep open: this PR targets a same-process active-job path that does not match the likely duplicate cron reservation failure, and it lacks after-fix real behavior proof.

## What This Changes

The PR imports `isCronJobActive` and skips runnable cron jobs whose ID is already marked active in the process-local active-job set.

## Best Possible Solution

Implement a repro-backed cross-process cron reservation fix around load/mark/persist and completion merge, with focused regression coverage, rather than merging the process-local active-job guard alone.

## Reproduction Assessment

Yes, source-reproducible: run two independent OpenClaw gateway/cron processes against the same cron store and wake the same due job before either observes the other's persisted reservation. I did not run that live reproduction in this read-only review, and the PR's specific same-process re-arm window is not reproduced by current source.

## Solution Assessment

No. The proposed active-job check does not run while `state.running` short-circuits timer ticks and cannot coordinate across separate gateway processes, so the narrower maintainable fix is a cross-process reservation guard with a targeted regression test.
