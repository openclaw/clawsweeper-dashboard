## Summary

Keep open: this patch is too narrow for the linked duplicate cron report because `activeJobIds` is process-local and current same-process timer execution already stays under `state.running` until all worker jobs complete; it also lacks after-fix real behavior proof.

## What This Changes

This PR adds a process-local `isCronJobActive(job.id)` guard to cron `isRunnableJob()` so jobs already marked active are skipped during collection.

## Best Possible Solution

Fix the persisted cron reservation path with a cross-process lock or equivalent transaction around load, collect, mark, and persist, then add regression coverage that models separate process-local locks.

## Reproduction Assessment

Yes, source-level: run two independent OpenClaw gateway or cron processes against the same cron store and wake the same due job before either observes the other's persisted `runningAtMs`. I did not execute that live path in this read-only review.

## Solution Assessment

No. The PR's process-local `activeJobIds` check does not cover separate gateway processes; the narrow maintainable fix is cross-process reservation locking around the persisted cron store reservation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard the persisted cron reservation across processes:** `src/cron/service/timer.ts:1175-1177`

Please fix the persisted reservation path with a cross-process lock or equivalent transaction, and cover that shape in a regression test.
