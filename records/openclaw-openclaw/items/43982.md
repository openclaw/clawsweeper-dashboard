## Summary

Keep open: the inline-image redaction is useful and source-reproducible, but this PR applies it to an older chat-history sanitizer location while current main routes display output through the shared projector. It also still lacks contributor-provided real behavior proof before merge.

## What This Changes

The PR strips inline image payload variants from gateway chat history responses and adds regression tests for redaction plus omitted-byte accounting.

## Best Possible Solution

Port the expanded inline-image redaction into `src/gateway/chat-display-projection.ts`, keep byte accounting accurate across stripped fields, cover `chat.history` plus live display paths, and leave durable transcript claim-checking to https://github.com/openclaw/openclaw/issues/1210.

## Reproduction Assessment

Yes, at source level: a message with `content: [{ type: "image", source: { type: "base64", data: "..." } }]` passed through the current projector keeps the nested image payload because current main only strips top-level `image.data` and audio `source.data`. I did not run a live Gateway/WebChat repro in this read-only review.

## Solution Assessment

No. The redaction logic is useful, but placing it only in `server-methods/chat.ts` is no longer the maintainable fix because current main centralizes display projection in `chat-display-projection.ts`.
