## Summary

Keep open: the PR targets a real gateway display bug, but the patch is attached to the old chat-history sanitizer instead of current main's shared display projector, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR strips inline image payloads from gateway chat-history content blocks for `source.data`, data-url `url`, and `image_url.url` shapes and adds regression tests for redaction and byte accounting.

## Best Possible Solution

Port the expanded inline-image redaction and accumulated byte accounting into `src/gateway/chat-display-projection.ts`, add shared-projector coverage for the affected shapes, and leave durable transcript claim-checking to the canonical storage issue.

## Reproduction Assessment

Yes, at source level: current main's projector strips only top-level `image.data` and audio `source.data`, while `chat.history`, live `session.message`, and session-history snapshots use that projector. I did not run a live Gateway/WebChat repro in this read-only review.

## Solution Assessment

No. The redaction logic is the right direction, but patching `server-methods/chat.ts` is no longer the maintainable fix because current main centralizes display projection in `chat-display-projection.ts`.
