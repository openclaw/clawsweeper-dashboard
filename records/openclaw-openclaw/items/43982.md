## Summary

Keep open: the redaction behavior is useful and still source-reproducible on current main, but this PR edits the old `server-methods/chat.ts` sanitizer while current display paths now run through the shared projector. It also still lacks after-fix real Gateway/WebChat behavior proof before merge.

## What This Changes

The PR strips inline image data payloads from additional gateway chat-history content-block shapes and adds regression tests for image payload redaction and byte accounting.

## Best Possible Solution

Port the expanded inline-image redaction into `src/gateway/chat-display-projection.ts`, keep omitted-byte accounting accurate across stripped fields, cover `chat.history`, live `session.message`, and session-history snapshots, and leave durable transcript claim-checking to https://github.com/openclaw/openclaw/issues/1210.

## Reproduction Assessment

Yes, at source level: a message with `content: [{ type: "image", source: { type: "base64", data: "..." } }]` passed through `projectChatDisplayMessage` or `projectRecentChatDisplayMessages` keeps the nested image payload because current main only strips top-level `image.data`. I did not run a live Gateway/WebChat repro in this read-only review.

## Solution Assessment

No. The redaction logic is useful, but placing it only in `server-methods/chat.ts` is no longer the maintainable fix because current main centralizes UI display projection in `chat-display-projection.ts`.
