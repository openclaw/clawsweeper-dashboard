## Summary

Keep open: the user-facing problem is valid, but this PR should not merge as-is because current main routes display history through the shared projector while the branch adds the new image redaction to the older server-methods sanitizer path. The external PR also has no real behavior proof beyond focused tests.

## What This Changes

The PR adds inline image data-url stripping and omitted-byte accumulation to gateway chat history sanitization plus regression tests.

## Best Possible Solution

Port the expanded inline-image redaction into `src/gateway/chat-display-projection.ts`, keep omitted-byte accounting accurate across stripped fields, cover chat.history/live/session-history display paths, and leave durable transcript claim-checking to https://github.com/openclaw/openclaw/issues/1210.

## Reproduction Assessment

Yes, at source level. On current main, a message containing `content: [{ type: "image", source: { type: "base64", data: "..." } }]` passed through `projectChatDisplayMessage` or `projectRecentChatDisplayMessages` keeps the nested image payload because only top-level `image.data` is removed.

## Solution Assessment

No. The redaction behavior is useful, but implementing it only in `server-methods/chat.ts` is no longer the narrowest maintainable solution because current main centralizes UI display projection in `chat-display-projection.ts`.
