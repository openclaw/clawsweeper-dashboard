## Summary

Keep open: current main still builds a Telegram `Conversation context` chat-window block from recent cached messages and passes it as runtime-only current-turn context to the model prompt, while the existing tests cover stale-boundary and intra-metadata dedupe but not dedupe against the persistent session transcript. This is a fresh user-reported regression, and the safe fix needs maintainer judgment on which Telegram chat-window messages are supplemental versus already represented in the session.

## What This Changes

Deduplicate or suppress Telegram conversation context blocks that repeat persistent session history, with an optional opt-out mentioned as one possible remedy.

## Best Possible Solution

Keep the issue open and design a transcript-aware Telegram context policy that preserves reply targets and ambient group context while avoiding reinjection of messages already represented in the active session/context-engine history.

## Reproduction Assessment

Yes, at source level: current main selects recent Telegram messages into `Conversation context`, renders that block, and prepends it as current-turn context while the transcript prompt is passed separately. I did not run a live Telegram setup, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear: a blanket disable or config toggle would reduce prompt pressure but could drop useful Telegram group/reply context. The maintainable fix is likely targeted dedupe against already assembled session history while keeping genuinely supplemental channel context.
