## Summary

Keep open. Current main still builds Telegram chat-window context from recent cached messages and prepends that context to the model prompt, while the same conversation turns can already be present in the persistent session history; the issue is source-provable and not covered by a merged fix or canonical duplicate in the provided context.

## What This Changes

The issue asks OpenClaw to stop duplicating Telegram conversation-context blocks against persistent session history to avoid prompt pollution, token waste, and dashboard clutter.

## Best Possible Solution

Deduplicate or omit Telegram chat-window entries already represented by the persistent session transcript, while preserving reply-target context that the transcript cannot reconstruct, and cover it with focused Telegram plus embedded-runner regression tests.

## Reproduction Assessment

Yes, at source level: the existing Telegram tests exercise multiple turns and current main then renders those previous messages into `currentTurnContext` before submitting the model prompt. I did not run a live Telegram account during this read-only review.

## Solution Assessment

Yes: the best fix is a narrow code change that respects the existing transcript/runtime-context split and filters Telegram context against session history. A new disable option should not be the first-line fix unless maintainers decide the product should expose a policy knob.
