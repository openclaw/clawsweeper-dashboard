- `impact:security`: The requested relay would let a remote OpenClaw instance mediate control of a user's signed-in local browser over extension/WebSocket/CDP paths.

## Summary

Keep open: current main still rejects the legacy Chrome extension relay driver and has no cross-machine browser-control replacement; the open browser-backend API issue is related, but this remains the canonical relay-specific regression with managed-hosting evidence.

## What This Changes

Restore or replace the removed Chrome extension relay so remote OpenClaw hosts can control a user's local Chrome across machines without client-side CLI or CDP setup.

## Best Possible Solution

Keep this issue as the relay-specific tracker until maintainers either design a hardened opt-in cross-machine browser relay/backend API or make an explicit retirement and migration decision, with https://github.com/openclaw/openclaw/issues/19289 carrying the generic driver-registration seam.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a browser profile with driver extension: current schema and routes reject it, the relay helper is a no-op, and a contract test asserts the unsupported-driver 400.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the best fix; a maintainer-approved browser backend/plugin API or an explicit retirement and migration decision is safer than an automated rollback.
