## Summary

Keep open: current main still intentionally rejects the legacy Chrome extension relay/profile driver, and the documented replacements do not provide the one-click cross-machine signed-in browser relay workflow described here. Resolving it needs an explicit maintainer browser strategy and security decision.

## What This Changes

Requests restoration of the removed Chrome extension relay/profile driver, or an optional plugin path for cross-machine signed-in browser control without user-side Node/CDP setup.

## Best Possible Solution

Keep the issue open until maintainers choose either a hardened opt-in relay owned by the browser plugin or an additive public browser backend/transport API for external relay implementations, with schema, docs, tests, migration behavior, and threat model aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure or create a profile with `driver: "extension"`; current main rejects it in schema/control routes, has a no-op relay startup helper, and has a contract test asserting the 400 response.

## Solution Assessment

Unclear. Reinstating the old core relay directly may not be the best long-term fix because OpenClaw’s direction favors plugin-owned optional capabilities, but current main does not yet expose the browser backend API needed for a clean external relay.
