## Summary

Keep open: current `main` and the latest release still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document host-local Chrome MCP, node-host, or remote-CDP alternatives rather than the reported one-click cross-machine extension workflow. The linked ClawHosters relay and FSB references are useful design context, but the remaining work needs browser-strategy, plugin/API, and security decisions.

## What This Changes

The issue asks OpenClaw to restore or replace the removed Chrome extension browser relay so remote OpenClaw hosts can control a user's local Chrome across machines without requiring local CLI or Node setup.

## Best Possible Solution

Keep this issue as the concrete cross-machine relay regression thread until maintainers either design a vetted browser driver/plugin API with an optional relay implementation or explicitly document that the Chrome extension relay workflow is no longer supported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current `main` rejects `driver: "extension"`, keeps relay startup as a no-op, and has a contract test asserting the unsupported-driver response; I did not run a live VPS plus Chrome extension setup in this read-only review.

## Solution Assessment

Unclear for exact restoration. Re-adding the old driver directly would not settle the remote browser-control trust boundary; the safer maintainable path is an owner-approved plugin/API or optional relay design with focused migration, docs, and security review.
