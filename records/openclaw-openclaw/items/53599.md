## Label Justifications

- `P1`: The report describes a shipped regression that blocks remote browser control for managed-hosting and cross-machine users after the extension relay removal.
- `impact:security`: The requested relay would control authenticated browser sessions across extension, gateway, and network boundaries, making browser credentials and operator authority central to the issue.

## Summary

Keep open: current main and v2026.6.8 still intentionally reject the legacy `driver: "extension"` path and keep the extension relay lifecycle as a no-op, while the reported one-click cross-machine browser relay workflow remains unmet by host-local Chrome MCP or remote CDP alternatives.

## What This Changes

The issue asks to restore or replace the removed Chrome extension browser relay so remote OpenClaw deployments can control a user's local Chrome across machines without local CLI setup.

## Best Possible Solution

Maintainers should choose the browser-control strategy explicitly: either keep the relay retired with clear migration guidance, or design a secure browser-backend/plugin API plus an optional relay implementation with migration, docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 reject `driver: "extension"`, keep the relay startup helper as a no-op, and include a contract test for the unsupported-driver response. I did not run a live cross-machine Chrome extension relay setup.

## Solution Assessment

No for a direct automatic restore. The issue is valid, but the best fix needs maintainer-approved browser strategy and security design, likely involving the browser-backend/plugin API tracked at https://github.com/openclaw/openclaw/issues/19289 rather than only restoring an old driver string.
