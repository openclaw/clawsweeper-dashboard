## Summary

Keep open: current main and the latest release still intentionally remove the Chrome extension relay path and reject `driver: "extension"`, while the documented replacements are host-local or require node/CDP setup and do not cover the reported one-click cross-machine managed-hosting workflow. The discussion adds concrete ClawHosters relay usage, another VPS/macOS confirmation, mac-to-Windows demand, and an FSB reference for a scoped extension-style bridge, so this should remain a maintainer product/security decision rather than be closed as duplicate or ClawHub-only.

## What This Changes

The issue reports that removing the Chrome extension browser relay left managed remote hosts without a one-click cross-machine browser-control replacement.

## Best Possible Solution

Maintainers should make an explicit browser strategy decision: either document that the one-click cross-machine extension relay is intentionally retired, or design a hardened opt-in relay/backend plugin API with scoped access, audit logs, consent, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes for the current limitation: source inspection shows `driver: "extension"` is rejected, relay startup is a no-op, tests assert the unsupported-driver response, and the latest release still has the same behavior. I did not run a live Chrome extension relay because current main intentionally lacks that runtime path.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and probably an explicit plugin/backend registration contract or a clear retirement decision.
