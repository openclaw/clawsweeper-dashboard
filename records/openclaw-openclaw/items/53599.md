## Label Justifications

- `P1`: The report describes a shipped-regression class workflow break for managed-hosting users who lost remote browser control after the extension relay removal.
- `impact:security`: The requested relay controls real browser sessions across a gateway boundary and can expose authenticated tabs, credentials, page content, and operator-level browser access.

## Summary

Keep this issue open: current main and the latest release still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document only host-local or node/remote-CDP alternatives, while the concrete cross-machine relay regression remains unresolved and security-sensitive.

## What This Changes

The issue requests restoration of, or a secure replacement for, the removed Chrome extension browser relay so remote OpenClaw hosts can control a user's browser across machines without local CLI setup.

## Best Possible Solution

Make an explicit browser strategy decision: either document that the legacy cross-machine extension relay is retired with supported alternatives, or design a secure browser-backend/plugin API and optional relay implementation that preserves remote existing-browser control with clear capabilities, tests, docs, and migration behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 reject `driver: "extension"`, keep the old relay startup helper as a no-op, and include a test asserting the unsupported-driver response. I did not run a live remote Chrome extension relay setup.

## Solution Assessment

No for a direct automatic restore. The problem is valid, but the maintainable fix needs a maintainer-approved browser strategy and security model, likely through a versioned browser-backend/plugin contract rather than simply widening the driver string or reviving the old relay stub.
