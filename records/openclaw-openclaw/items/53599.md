## Summary

Keep open: current main and latest release still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document only host-local Chrome MCP, node-host, or remote-CDP alternatives, so the reported one-click cross-machine workflow remains unresolved and needs product/security/API direction.

## What This Changes

The issue requests restoring or replacing the removed Chrome extension relay so remote OpenClaw deployments can control a user's local Chrome across machines without client-side CLI setup.

## Best Possible Solution

Maintainers should either document the one-click cross-machine extension relay as retired, or design a hardened opt-in relay/backend plugin API with scoped access, consent, audit logs, docs, migration behavior, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects `driver: "extension"`, keeps relay startup disabled, migrates legacy relay config to `existing-session`, and has a contract test asserting the unsupported-driver response. I did not run a live extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or a clear retirement decision.
