## Summary

Keep open. Current main still intentionally removes the Chrome extension relay path, rejects the legacy `extension` browser driver, and documents only host-local Chrome MCP, raw CDP, or node-host proxy alternatives; the managed-hosting cross-machine workflow described by the reporter and corroborating comments remains unserved and security-sensitive.

## What This Changes

Restore or replace the removed Chrome extension browser relay so a remote OpenClaw instance can control a user's local Chrome across machines without requiring local Node.js, CLI tools, open ports, or Chrome remote-debugging setup.

## Best Possible Solution

Keep this open until maintainers choose an explicit browser strategy: either a hardened opt-in Chrome extension relay owned by the browser plugin, or an additive public browser backend/transport API that lets an external plugin implement the relay without private internals, with schema, docs, tests, migration behavior, and a threat model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create or configure a profile with `driver: "extension"`; current main rejects it in schema/control routes, has a no-op relay startup helper, and has a contract test asserting the 400 response.

## Solution Assessment

Unclear. Reinstating the old core relay directly may not be the best long-term fix because OpenClaw's direction favors optional high-power capabilities through plugins, but current main lacks the browser backend API needed for a clean external relay.
