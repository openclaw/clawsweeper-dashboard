## Summary

Keep open: current main still removes the Chrome extension relay path and rejects `driver: "extension"`, while the available replacement paths do not cover the reported one-click cross-machine managed-hosting workflow. The related browser-backend API issue is relevant but does not by itself resolve this relay-specific regression and security/product decision.

## What This Changes

Request to restore the removed Chrome extension relay or add an optional/plugin gateway path so remote OpenClaw instances can control a user's local Chrome across machines without a local Node/CLI setup.

## Best Possible Solution

Make an explicit browser strategy decision: either document that the one-click cross-machine extension relay is intentionally retired, or design a hardened opt-in relay/backend plugin API with scoped access, auditability, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes for the current limitation: source inspection shows `driver: "extension"` is rejected, relay startup is a no-op, and tests assert the unsupported-driver response. I did not run a live Chrome extension relay because current main intentionally lacks that runtime path.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and probably an explicit plugin/backend registration contract.
