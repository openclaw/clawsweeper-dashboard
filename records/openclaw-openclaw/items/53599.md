## Label Justifications

- `P1`: The issue reports a confirmed regression that breaks an existing remote browser-control workflow for managed-hosting and split-machine users.
- `impact:security`: The requested relay would control real browser sessions across a gateway boundary and can expose authenticated tabs, credentials, page content, and operator-level browser access.

## Summary

Keep this issue open: current main and v2026.6.6 still intentionally reject `driver: "extension"`, keep the relay startup hook as a no-op, and document Chrome MCP/remote-CDP/node-host alternatives that do not fully replace the reported one-click cross-machine extension relay. The linked ClawHosters relay, FSB, and browser-relay projects are useful design input, but none is a current OpenClaw core replacement for this regression.

## What This Changes

The issue asks OpenClaw to restore or replace the removed Chrome extension browser relay so remote OpenClaw instances can control a user's local Chrome across machines without requiring local Node.js, open ports, or Chrome remote-debugging setup.

## Best Possible Solution

Make an explicit browser strategy decision: either document that the legacy cross-machine extension relay is retired with supported alternatives, or design a secure browser-backend/plugin API and optional relay implementation coordinated with https://github.com/openclaw/openclaw/issues/19289.

## Reproduction Assessment

Yes. Source inspection is enough: current main rejects `driver: "extension"`, keeps the extension relay startup hook as a no-op, and includes a contract test asserting the unsupported-driver response; I did not run a live remote Chrome extension relay setup.

## Solution Assessment

No for a direct automatic restore. The problem is valid, but the maintainable fix needs a maintainer-approved browser strategy and security model, likely through a versioned browser-backend/plugin contract rather than simply widening the driver string or reviving the old relay stub.
