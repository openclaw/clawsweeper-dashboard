## Label Justifications

## Summary

Keep open: current main and the latest release still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document node-host/remote-CDP/Chrome MCP alternatives that do not cover the reported one-click cross-machine extension workflow. The open browser backend API issue is related, but this issue remains the canonical relay-regression/product-security decision thread.

## What This Changes

The issue asks OpenClaw to restore or replace the removed Chrome extension browser relay for cross-machine remote browser control without requiring end users to install a node host or configure remote CDP.

## Best Possible Solution

Maintainers should either explicitly retire the one-click extension relay and document that migration limit, or design a hardened opt-in browser backend/plugin API plus relay implementation with consent, scoping, audit logs, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects `driver: "extension"`, keeps relay startup disabled, migrates legacy relay config to `existing-session`, and has a contract test asserting the unsupported-driver response. I did not run a live extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest or most maintainable solution; the safer path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or an explicit retirement decision.
