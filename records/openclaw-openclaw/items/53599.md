## Label Justifications

- `P1`: The report describes a shipped regression that blocks remote browser control for managed-hosting and cross-machine users after the extension relay removal.
- `impact:security`: The requested relay would control authenticated browser sessions across extension, gateway, and network boundaries, making browser credentials and operator authority central to the issue.

## Summary

Keep open: current main and v2026.6.10 still reject the legacy `driver: "extension"` path, migrate it to `existing-session`, and do not provide the one-click cross-machine Chrome extension relay workflow described by the reporter and follow-up users.

## What This Changes

The issue asks OpenClaw to restore or replace the removed Chrome extension browser relay for cross-machine remote browser control.

## Best Possible Solution

Maintainers should choose the browser-control strategy explicitly: keep the relay retired with clearer migration guidance, or design a secure browser-backend/plugin API plus an optional relay implementation with migration, docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: current main rejects `driver: "extension"`, migrates it away in doctor, and has a contract test for the unsupported-driver response. I did not run a live cross-machine Chrome extension relay setup.

## Solution Assessment

No for an automatic direct restore. The issue is valid, but the best fix needs maintainer-approved browser strategy and security design, likely involving the browser-backend/plugin API tracked at https://github.com/openclaw/openclaw/issues/19289 rather than only restoring an old driver string.
