## Label Justifications

- `P1`: The report describes a shipped regression that breaks remote browser control for managed-hosting users after the extension relay removal.
- `impact:security`: The requested relay would control authenticated browser sessions across gateway and extension boundaries, making browser access, credentials, and operator authority part of the issue.

## Summary

Keep open: current main and v2026.6.6 still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document Chrome MCP/existing-session or CDP/node-host alternatives that do not satisfy the reported one-click cross-machine extension relay workflow. The open browser-backend API issue is related architecture work, but it does not replace this concrete managed-hosting regression yet.

## What This Changes

Requests restoring or replacing the removed Chrome extension browser relay so remote OpenClaw hosts can control a user's local Chrome across machines without requiring Node/CLI or remote CDP setup on the user's machine.

## Best Possible Solution

Make an explicit browser strategy decision: either keep the legacy cross-machine extension relay retired with clear docs and supported alternatives, or design a secure browser-backend/plugin API plus optional relay implementation with migration, tests, docs, and security review.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 reject `driver: "extension"`, keep the relay startup helper as a no-op, and include a contract test asserting the unsupported-driver response. I did not run a live cross-machine Chrome extension relay setup.

## Solution Assessment

No for a direct automatic restore. The issue is valid, but the best solution needs maintainer-approved browser strategy and security design, likely involving the browser-backend/plugin API tracked at https://github.com/openclaw/openclaw/issues/19289 rather than only restoring an old driver string.
