## Summary

Keep open: current main and v2026.5.19 still intentionally remove the Chrome extension relay path, reject `driver: "extension"`, and migrate legacy relay config to Chrome MCP `existing-session`, while the reported one-click cross-machine managed-hosting workflow is not covered by the documented node-host/CDP alternatives. This should stay as a maintainer product and security decision, with https://github.com/openclaw/openclaw/issues/19289 as the related API-design thread rather than a replacement.

## What This Changes

Restore or replace the removed Chrome extension browser relay so remote OpenClaw deployments can control a user's local Chrome across machines without requiring a local Node CLI or manual CDP setup.

## Best Possible Solution

Maintainers should decide the browser strategy explicitly: document the one-click cross-machine extension relay as retired, or design a hardened opt-in relay/backend plugin API with scoped access, consent, audit logs, docs, migration behavior, and regression coverage.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main rejects `driver: "extension"`, keeps relay startup disabled, and has a contract test asserting the unsupported-driver response. I did not run a live Chrome extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or a clear retirement decision.
