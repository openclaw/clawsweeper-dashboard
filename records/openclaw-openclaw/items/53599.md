## Summary

Keep open: current main and v2026.5.22 still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document Chrome MCP/node-host/remote-CDP alternatives that do not cover the reported one-click cross-machine extension workflow. The ClawHosters relay and FSB comments are useful implementation context, while the open browser-backend plugin-interface issue is related but does not replace this concrete relay-regression and security/product decision thread.

## What This Changes

Request to restore or replace the removed Chrome Extension Relay and `driver: "extension"` path with a cross-machine browser-control option for remote hosted OpenClaw deployments.

## Best Possible Solution

Make an explicit browser strategy decision: either keep the legacy relay retired and document the unsupported cross-machine extension workflow clearly, or design a vetted browser driver/plugin API plus an optional relay implementation with security review, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: current main rejects `driver: "extension"`, keeps relay startup as an intentional no-op, and has a contract test asserting the unsupported-driver error. I did not run a live remote VPS and Chrome extension setup in this read-only review.

## Solution Assessment

Unclear for exact restoration: adding the old driver back directly is not enough because the remote browser-control trust boundary needs an owner-approved security and product shape. A vetted plugin/driver API plus an optional relay implementation is the safer maintainable path if maintainers want this workflow back.
