## Summary

Keep open: current main still rejects the removed extension relay driver and keeps relay startup as a no-op, while the documented replacements do not satisfy the reported one-click cross-machine managed-hosting workflow. The linked ClawHosters relay and FSB projects are useful external design context, and the related open relay/API issues are not yet an explicit maintainer consolidation signal.

## What This Changes

Restore or replace the removed Chrome extension relay so remote hosted OpenClaw instances can control a user's local Chrome across machines without client-side Node or CLI setup.

## Best Possible Solution

Keep this open until maintainers either consolidate it into a canonical relay/API design issue or define a hardened opt-in browser relay/backend API with auth, consent, scoping, docs, tests, and migration behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create or configure a browser profile with driver "extension": current main rejects it in config and the profile route, keeps the relay helper as a no-op, and has a contract test asserting the 400 response.

## Solution Assessment

Unclear. Restoring the old core relay verbatim is risky; a maintainer-approved optional browser plugin/backend API or hardened relay design is the safer path.
