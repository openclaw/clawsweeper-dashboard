## Summary

Keep open: current main still intentionally rejects the legacy Chrome extension relay driver, and the documented replacements do not cover the reported one-click cross-machine relay workflow. Related browser-backend and relay-restoration issues are useful context, but neither is a merged fix or clearly canonical enough to close this managed-hosting regression report without maintainer direction.

## What This Changes

The issue requests restoring or replacing the removed Chrome extension browser relay so remote OpenClaw instances can control a user's browser across machines without local Node.js, ports, or CLI setup.

## Best Possible Solution

Maintainers should either design a hardened opt-in browser relay/backend plugin path that covers remote hosted instances, or explicitly document that this cross-machine extension workflow is retired and point users to the supported replacement architecture.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a browser profile with driver extension: current main rejects it in config schema and profile creation, keeps relay startup disabled, and has a contract test asserting the unsupported-driver response.

## Solution Assessment

Unclear. Directly restoring the old core relay may not be the safest long-term fix; a hardened opt-in browser backend or relay plugin contract with consent, scoping, logging, and credential-submit guardrails is the more maintainable direction.
