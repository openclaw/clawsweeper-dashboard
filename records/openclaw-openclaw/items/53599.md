## Summary

Keep open: current main and v2026.5.22 still intentionally remove the Chrome extension relay, reject `driver: "extension"`, and document host/browser-node Chrome MCP, node-host, or remote-CDP alternatives, so the one-click cross-machine relay workflow remains unresolved and needs maintainer product/security/API direction.

## What This Changes

The issue asks for a cross-machine replacement or opt-in restoration path for the removed Chrome extension browser relay workflow.

## Best Possible Solution

Maintainers should either explicitly retire the one-click extension relay and document that migration limit, or design a hardened opt-in browser backend/plugin API plus relay implementation with consent, scoping, audit logs, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects `driver: "extension"`, keeps relay startup disabled, migrates legacy relay config to `existing-session`, and has a contract test asserting the unsupported-driver response. I did not run a live extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or a clear retirement decision.
