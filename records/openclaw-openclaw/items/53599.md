## Summary

Keep open: current main still intentionally rejects the legacy extension relay driver, and the documented replacements do not provide the cross-machine one-click browser relay workflow described here. The related open relay/API issues are useful context, but they do not fully replace this managed-hosting regression report.

## What This Changes

Restore or replace the legacy Chrome extension browser relay so remote OpenClaw servers can control a user's local Chrome across machines without host-local tooling.

## Best Possible Solution

Keep this open until maintainers choose a hardened opt-in relay owned by the browser plugin or an additive browser backend/transport API that supports external relay implementations with schema, docs, tests, migration behavior, and threat model aligned.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a profile with driver extension: current main rejects it in schema and control routes, has a no-op relay helper, and has a contract test asserting the 400 response.

## Solution Assessment

Unclear. Directly restoring the old core relay may not be the best long-term fix; the safer maintainable path is likely a hardened optional browser plugin or a public browser backend/transport API with security review.
