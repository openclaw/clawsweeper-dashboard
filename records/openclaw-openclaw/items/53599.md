## Summary

Keep open: current main and the latest release still intentionally remove the Chrome extension relay path, reject `driver: "extension"`, and migrate legacy relay config to Chrome MCP `existing-session`; the documented alternatives do not cover the reported one-click cross-machine managed-hosting workflow. This should remain a maintainer product and security decision, with https://github.com/openclaw/openclaw/issues/19289 as the related browser-backend API design thread rather than a replacement.

## What This Changes

Request to restore or replace the removed Chrome extension browser relay so remote OpenClaw hosts can control a user's local Chrome across machines without a local node host or remote CDP setup.

## Best Possible Solution

Maintainers should decide the browser strategy explicitly: document the one-click cross-machine extension relay as retired, or design a hardened opt-in relay/backend plugin API with scoped access, consent, audit logs, docs, migration behavior, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects `driver: "extension"`, keeps relay startup disabled, migrates legacy relay config to `existing-session`, and has a contract test asserting the unsupported-driver response. I did not run a live Chrome extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest solution; the maintainable path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or a clear retirement decision.
