## Summary

Keep open. Current main still intentionally removes the Chrome extension relay, rejects the legacy `extension` browser driver, and migrates legacy relay config to Chrome MCP `existing-session`; that path does not satisfy the reporter's cross-machine managed-hosting workflow or the corroborating comments.

## What This Changes

Restore or replace the removed Chrome extension browser relay so a remote OpenClaw instance can control a user's local Chrome without a local Node host or remote-CDP setup.

## Best Possible Solution

Keep this open until maintainers choose a deliberate browser strategy: either a hardened opt-in Chrome extension relay owned by the browser plugin, or an additive public browser profile-driver/transport API that lets an external plugin implement the relay without private internals, with schema, docs, tests, and a threat model.

## Reproduction Assessment

Yes. Without running a live browser, current source and contract tests give a high-confidence reproduction path: creating a profile with `driver: "extension"` is expected to return HTTP 400, and legacy config is normalized away from the relay.

## Solution Assessment

No for a direct legacy restore as-is. The narrow maintainable path needs a security-reviewed relay design or public browser-driver API; the current Chrome MCP migration is useful for host-local sessions but is not equivalent to the reported cross-machine one-click extension relay.
