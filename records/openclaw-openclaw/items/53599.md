## Summary

Keep open: current main and v2026.5.22 still intentionally remove the Chrome extension relay, reject the legacy browser profile driver, and document host-local Chrome MCP plus node-host/remote-CDP alternatives that do not satisfy the reported one-click cross-machine extension workflow.

## What This Changes

Restore or replace the removed Chrome extension relay so remote OpenClaw hosts can control a user's local Chrome without user-side Node, CLI, or remote-CDP setup.

## Best Possible Solution

Maintainers should either explicitly retire the one-click extension relay with clear migration-limit docs, or design a hardened opt-in browser backend/plugin API plus relay implementation with consent, scoping, audit logs, migration behavior, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main rejects driver "extension", keeps relay startup disabled, migrates legacy relay config to existing-session, and has a contract test asserting the unsupported-driver response. I did not run a live extension relay because the runtime path is intentionally absent.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the safest or most maintainable solution; the safer path needs maintainer-approved browser strategy, security review, and likely a browser backend/plugin registration contract or an explicit retirement decision.
