## Summary

Keep open: current main still rejects the legacy Chrome extension relay driver and keeps the relay startup helper disabled. The ClawHosters relay and FSB links are useful implementation signals, but the remaining work needs maintainer browser-strategy, Plugin SDK/API, and security direction before code cleanup or automation.

## What This Changes

Restore or replace the removed Chrome extension browser relay so remote OpenClaw instances can control a user's local Chrome across machines without requiring Node.js, CLI tooling, or remote-debugging tunnels on the user's device.

## Best Possible Solution

Keep this open until maintainers either design a hardened opt-in cross-machine browser relay/backend API or explicitly consolidate the managed-hosting use case into a canonical browser strategy issue.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a browser profile with driver extension: current schema and routes reject it, the relay helper is a no-op, and a contract test asserts the unsupported-driver 400.

## Solution Assessment

Unclear. Directly restoring the old core relay is not clearly the best fix; a maintainer-approved, hardened browser backend/plugin API or an explicit retirement/migration decision is safer than an automated rollback.
