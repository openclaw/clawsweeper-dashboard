## Summary

Keep open: current main still intentionally rejects the legacy Chrome extension relay driver, and the documented replacements do not provide the one-click cross-machine browser relay workflow reported here. The external ClawHosters relay and FSB MCP/extension work are useful implementation signals, but the remaining OpenClaw work still needs a maintainer browser/API/security decision rather than conservative cleanup closure.

## What This Changes

The issue requests restoring or replacing the removed Chrome extension browser relay so remote OpenClaw hosts can control a user’s local Chrome across machines without a local Node/CLI setup.

## Best Possible Solution

Keep this issue open until maintainers either design a hardened opt-in relay/browser backend API or explicitly consolidate the managed-hosting use case into a canonical browser-relay strategy issue.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a browser profile with driver extension: current main rejects it in schema and control routes, keeps the relay helper as a no-op, and has a contract test asserting the unsupported-driver 400.

## Solution Assessment

Unclear. Directly restoring the old core relay may not be the best long-term fix; a hardened optional browser plugin/backend API with scoped access, visible tab ownership, action logging, consent, and submit/credential guardrails is the safer maintainable direction.
