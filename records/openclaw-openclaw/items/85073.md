## Label Justifications

- `P1`: The PR touches a security-sensitive cron tool-policy path where restricted CLI cron runs can currently miss the intended fail-closed enforcement.
- `merge-risk: 🚨 compatibility`: Existing CLI-backed cron jobs with `toolsAllow` can change from running to failing closed once the allowlist reaches the CLI guard.

## Summary

Keep open: the CLI `toolsAllow` handoff is a real cron policy bug, but this branch also marks unattended cron runs as owner-authorized, which is a separate security/trust-boundary change and the external PR still has no real behavior proof.

## What This Changes

The PR forwards isolated cron `toolsAllow` into CLI runs, marks CLI and embedded cron runs as owner-authorized, and adds regression tests for both paths.

## Best Possible Solution

Land a narrow CLI `toolsAllow` handoff against current main with real cron proof, and leave scheduled cron owner identity unchanged unless maintainers explicitly approve that trust change.

## Reproduction Assessment

Yes. Source inspection on current main shows isolated cron forwards `toolsAllow` to the embedded runner but not to the CLI runner, and the CLI preparation layer already fails closed when that field is present.

## Solution Assessment

No. Forwarding `toolsAllow` into the CLI call is the narrow fix, but unconditional `senderIsOwner: true` is a separate trust-model change that should be removed or explicitly approved.
