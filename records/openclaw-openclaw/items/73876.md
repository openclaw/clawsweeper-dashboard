## Summary

Keep open. Current main still registers direct Moonshot only as `moonshot`, with no `moonshotai` provider alias, and an open unmerged PR already targets this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests support for `moonshotai/<model>` as a direct Moonshot API model reference while preserving existing `moonshot/<model>` configurations.

## Best Possible Solution

Preserve `moonshot` as the canonical provider/plugin id and add a backwards-compatible `moonshotai` direct model-catalog alias, with focused coverage proving `moonshotai/kimi-k2.6` plans through the existing Moonshot catalog/runtime/auth path.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrow maintainable path is to accept the OpenRouter-style slug while keeping existing `moonshot/...` configs and auth profiles stable.
