## Summary

Keep open. Current main still exposes direct Moonshot as `moonshot/...` with no `moonshotai` alias, and open closing PR #74946 is the active implementation candidate to review before this issue can close.

## What This Changes

Request direct Moonshot model refs copied as `moonshotai/<model>` to resolve to the existing direct Moonshot provider while preserving `moonshot/<model>`.

## Best Possible Solution

Land a reviewed alias fix that maps direct `moonshotai/<model>` to the existing Moonshot catalog while keeping `moonshot` as the canonical provider, plugin, auth profile, and documented config key.

## Reproduction Assessment

Yes. Source inspection shows `moonshotai/kimi-k2.6` parses as provider `moonshotai`, current main has no provider-id or manifest alias for that prefix, and the runtime unknown-model path reports the parsed provider/model.

## Solution Assessment

Yes. A backward-compatible `modelCatalog.aliases.moonshotai -> moonshot` fix is narrower than renaming the provider because existing `moonshot/...` configs, auth profiles, and docs stay valid.
