## Summary

Keep open. Current main still leaves `moonshotai` and `moonshot-ai` unaliased in runtime provider normalization, and the linked open PR only adds a manifest catalog-planning alias, so the reported direct model resolution failure remains unresolved.

## What This Changes

Accept OpenRouter-style direct Moonshot refs such as `moonshotai/kimi-k2.6` without breaking existing `moonshot/...` configs.

## Best Possible Solution

Update the active fix to add backward-compatible runtime aliases from `moonshotai` and `moonshot-ai` to canonical `moonshot`, with direct model-resolution regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection shows `moonshotai/kimi-k2.6` is parsed with provider `moonshotai`, while the direct Moonshot catalog is keyed under `moonshot`, matching the reporter's `Unknown model` logs.

## Solution Assessment

Yes. A provider-id alias is the narrowest maintainable fix because it preserves canonical `moonshot` configs, auth profiles, plugin ids, and docs; the catalog-only PR path is not sufficient for runtime resolution.
