## Summary

Keep open. Current main still exposes direct Moonshot as `moonshot/...` with no `moonshotai` alias, and the open closing PR #74946 is the right implementation candidate to review before closing this issue.

## What This Changes

Accept `moonshotai/<model>` as a direct Moonshot API alias so copied OpenRouter Moonshot refs can resolve without changing existing `moonshot/...` configs.

## Best Possible Solution

Land a reviewed alias fix that maps direct `moonshotai/<model>` to the existing Moonshot provider while keeping `moonshot` as the canonical plugin, auth profile, and config key.

## Reproduction Assessment

Yes. The source path is high-confidence: slash refs normalize the provider prefix through `normalizeProviderId`, current main has no `moonshotai` mapping, and the Moonshot catalog only exposes `moonshot/kimi-k2.6`; the reporter also provided matching runtime logs.

## Solution Assessment

Yes. Adding a backward-compatible alias is narrower and safer than renaming the provider, because existing `moonshot/...` configs and auth profiles should remain valid.
