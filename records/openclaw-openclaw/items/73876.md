## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is paired with open unmerged PR #74946 using closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that direct Moonshot model refs copied as `moonshotai/<model>` resolve to the bundled Moonshot provider while OpenRouter-routed refs remain under `openrouter/...`.

## Best Possible Solution

OpenClaw should accept `moonshotai/<model>` as a compatibility alias through the manifest catalog/provider resolution surface while keeping `moonshot` as the canonical plugin/provider id for auth profiles, docs, and existing configs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot` catalog rows.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable solution is to accept the OpenRouter-style slug while keeping existing `moonshot/...` auth profiles, plugin ids, and configs stable.
