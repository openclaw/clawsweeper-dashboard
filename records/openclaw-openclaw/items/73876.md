## Summary

Keep open. Current main still does not accept `moonshotai/<model>` as a direct Moonshot model reference, and the issue is already paired with open unmerged PR #74946 using closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Accept `moonshotai/<model>` as a backwards-compatible direct Moonshot API model reference while preserving existing `moonshot/<model>` configs.

## Best Possible Solution

Keep `moonshot` as the canonical provider/plugin id and add a compatibility alias for `moonshotai` through the existing manifest catalog alias path, with focused coverage proving `moonshotai/kimi-k2.6` resolves without duplicating provider auth/runtime ownership.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing keeps `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrow maintainable path preserves existing `moonshot/...` configs and accepts the OpenRouter-style slug for direct model refs.
