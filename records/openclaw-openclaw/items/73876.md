## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is already paired with open unmerged PR #74946 using closing syntax, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue requests accepting `moonshotai/<model>` as a direct Moonshot model reference, while preserving existing `moonshot/<model>` configs.

## Best Possible Solution

Keep `moonshot` as the canonical plugin/provider id and add a backward-compatible `moonshotai` direct-ref alias through the manifest catalog alias surface, with focused coverage proving `moonshotai/kimi-k2.6` resolves to the direct Moonshot catalog while existing `moonshot/...` configs continue to work.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable fix is to accept the OpenRouter-style slug while keeping existing `moonshot/...` auth profiles, docs, and plugin ids stable; #74946 is the active implementation candidate.
