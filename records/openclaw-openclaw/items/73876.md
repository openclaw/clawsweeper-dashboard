## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and open PR #74946 already targets this issue with GitHub closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Accept `moonshotai/<model>` as a direct Moonshot API model reference or otherwise reconcile Moonshot's direct provider naming with OpenRouter's MoonshotAI slug without breaking existing `moonshot/...` configs.

## Best Possible Solution

Add a Moonshot manifest catalog alias such as `modelCatalog.aliases.moonshotai.provider = "moonshot"`, with focused planner coverage, while keeping `moonshot` as the canonical plugin/provider/auth-profile id.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization does not alias it, and the Moonshot manifest has no direct catalog alias for that provider.

## Solution Assessment

No for a hard provider rename, because that would risk existing `moonshot/...` configs and auth profiles. Yes for the compatibility-alias approach targeted by PR #74946, which uses the existing manifest catalog alias seam.
