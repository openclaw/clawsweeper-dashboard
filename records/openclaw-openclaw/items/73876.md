## Summary

Keep open. Current main still does not accept `moonshotai/<model>` as a direct Moonshot model ref, and open PR #74946 remains the active unmerged implementation candidate for this issue.

## What This Changes

Accept `moonshotai/<model>` as a direct Moonshot provider alias while preserving existing `moonshot/<model>` configs and auth profiles.

## Best Possible Solution

Land #74946 or an equivalent Moonshot-owned manifest/catalog alias for `moonshotai` that keeps `moonshot` as the canonical provider id.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `moonshotai/kimi-k2.6` parses as provider `moonshotai`, but current main has no provider normalizer or Moonshot manifest alias that maps it to `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable solution is a Moonshot-owned alias that preserves existing configs and auth profile keys.
