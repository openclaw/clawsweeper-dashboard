## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is paired with open unmerged PR #74946 using closing syntax, so it should stay open until that PR lands or is replaced.

## What This Changes

Accept direct Moonshot model references using the OpenRouter-style `moonshotai/<model>` slug while preserving existing `moonshot/<model>` configurations.

## Best Possible Solution

Keep `moonshot` as the canonical plugin/provider id and add a backward-compatible catalog/provider alias so `moonshotai/<model>` resolves through the direct Moonshot catalog/runtime/auth path, with focused planner or model-resolution coverage and docs only where the accepted ref form becomes user-facing.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrow maintainable fix preserves existing `moonshot/...` configs while accepting the OpenRouter-style direct slug, which is the direction taken by open PR #74946.
