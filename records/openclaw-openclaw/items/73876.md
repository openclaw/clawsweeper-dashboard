## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is paired with open unmerged PR #74946 using closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to accept OpenRouter-style `moonshotai/<model>` refs for the direct Moonshot provider while preserving the routed `openrouter/moonshotai/<model>` form.

## Best Possible Solution

Add a focused `moonshotai` catalog/provider-ref alias that resolves through the existing Moonshot direct provider while keeping `moonshot` canonical, with tests proving the alias rows and existing `moonshot/...` refs both work.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. PR #74946's manifest-alias direction matches the existing planner seam and avoids breaking current `moonshot` configs.
