## Summary

Keep open. Current main still accepts direct Moonshot only as `moonshot/...`, does not normalize `moonshotai` or `moonshot-ai`, and the open closing PR #74946 remains unmerged and should be reviewed for runtime resolution coverage.

## What This Changes

Accept OpenRouter-style `moonshotai/...` direct Moonshot model refs without breaking existing `moonshot/...` configs.

## Best Possible Solution

Ship a backward-compatible runtime alias mapping `moonshotai` and `moonshot-ai` to canonical `moonshot`, with focused resolver tests, catalog/docs alignment as needed, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows `moonshotai/kimi-k2.6` parses as provider `moonshotai`, no normalization maps it to `moonshot`, and the loaded direct Moonshot catalog is keyed under `moonshot` only.

## Solution Assessment

Yes, with one caveat. A backward-compatible alias is safer than renaming the canonical provider, but the fix should include runtime provider-id normalization rather than relying only on manifest catalog rows.
