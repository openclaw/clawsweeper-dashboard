## Summary

Keep open. Current main still leaves `moonshotai/<model>` as an unknown direct provider path, and open PR #74946 is the active unmerged implementation candidate with closing syntax for this issue.

## What This Changes

Accept `moonshotai/<model>` as a direct Moonshot model reference while preserving existing `moonshot/<model>` configs.

## Best Possible Solution

Review and land #74946, or an equivalent Moonshot-owned manifest/catalog alias, so `moonshotai/<model>` resolves while `moonshot` remains canonical.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `moonshotai/kimi-k2.6` parses as provider `moonshotai`, and current main has neither a provider normalizer mapping nor a Moonshot manifest alias to resolve it to `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard rename. The narrow maintainable fix is to add a Moonshot-owned alias using the existing manifest catalog seam and leave existing `moonshot/...` configs untouched.
