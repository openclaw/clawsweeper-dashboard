## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and open PR #74946 is the active unmerged implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests that direct Moonshot model refs accept the OpenRouter/Moonshot AI slug form `moonshotai/<model>` while preserving existing `moonshot/<model>` configs.

## Best Possible Solution

Accept `moonshotai/<model>` through a Moonshot-owned manifest/catalog alias while keeping `moonshot` as the canonical provider id for existing configs and auth profiles.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `moonshotai/kimi-k2.6` parses as provider `moonshotai`, but current main has no provider normalizer or Moonshot manifest alias that maps it to `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable solution is a Moonshot-owned alias that preserves existing `moonshot/...` configs and auth profile keys.
