## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is paired with open unmerged PR #74946 using closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to accept Moonshot's OpenRouter-style `moonshotai/<model>` direct provider refs without breaking existing `moonshot/<model>` configuration.

## Best Possible Solution

Preserve `moonshot` as the canonical plugin/provider id and add a backwards-compatible `moonshotai` direct-ref alias through the existing catalog/provider resolution surface, with focused tests proving `moonshotai/kimi-k2.6` resolves to the direct Moonshot catalog while existing `moonshot/...` configs keep working.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable fix is to accept the OpenRouter-style slug while keeping existing `moonshot/...` auth profiles, docs, and plugin ids stable; #74946 is the active implementation candidate.
