## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is paired with open unmerged PR #74946 using closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests direct Moonshot model references copied as `moonshotai/<model>` to resolve to the canonical direct Moonshot provider while preserving existing `moonshot/<model>` configurations.

## Best Possible Solution

OpenClaw should accept `moonshotai/<model>` as a compatibility alias through the manifest catalog/provider resolution surface while keeping `moonshot` as the canonical plugin/provider id for auth profiles, docs, and existing configs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable fix is to accept the OpenRouter-style slug while keeping existing `moonshot/...` auth profiles, docs, plugin ids, and configs stable.
