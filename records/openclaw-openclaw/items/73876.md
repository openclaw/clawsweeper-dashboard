## Summary

Keep open. Current main still lacks a direct `moonshotai/<model>` compatibility path, and the issue is already paired with open unmerged PR #74946 using closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests direct Moonshot model refs such as `moonshotai/kimi-k2.6` to resolve consistently with OpenRouter's Moonshot organization slug while preserving existing `moonshot/...` configs.

## Best Possible Solution

Keep `moonshot` as the canonical provider/plugin id, add a backwards-compatible alias so `moonshotai/<model>` resolves through the direct Moonshot catalog/runtime/auth path, and leave stale `plugins.entries.moonshotai` as config cleanup rather than creating a second plugin id.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for a compatibility alias, no for a hard provider rename. The narrowest maintainable solution preserves existing `moonshot/...` configs while accepting the OpenRouter-style slug for direct model refs.
