## Label Justifications

- `P1`: This issue reports a sandbox security-boundary gap that can leave executable skill directories writable on first launch.
- `impact:security`: The affected behavior concerns sandboxing and whether agent-controlled commands can mutate skill code that should be read-only.

## Summary

Keep open: current main and the latest release still skip missing skill mount sources, so the first-launch read-only sandbox protection gap remains. The active fix path is the open closing PR at https://github.com/openclaw/openclaw/pull/94820, but it is unmerged and already has review/proof blockers.

## What This Changes

The issue reports that first-launch Docker rw sandboxes skip read-only skill overlays when host skills directories are absent, leaving `/workspace/skills` writable until a later restart.

## Best Possible Solution

Land a narrow sandbox fix that creates or materializes missing skill roots before read-only mount resolution while preserving canonical-root validation, adding missing-root and symlink-parent regression coverage, and providing first-launch Docker proof.

## Reproduction Assessment

Yes at source level: current main returns false for missing skill mount sources, which omits the read-only overlay before Docker container creation. I did not run a live Docker first-launch repro in this read-only review.

## Solution Assessment

No final fix is on main yet. The best path is to repair and land https://github.com/openclaw/openclaw/pull/94820, or replace it with an equally narrow fix that preserves canonical-root validation and proves the first-launch Docker behavior.
