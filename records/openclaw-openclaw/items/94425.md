## Label Justifications

- `P1`: The issue describes a sandbox security-boundary gap that can leave executable skill paths writable on first Docker sandbox launch.
- `impact:security`: The affected behavior concerns sandbox read-only mount enforcement for skill code and instructions.

## Summary

Keep this issue open: current main and the latest release still skip read-only skill overlays when the host skill source directories are missing, and an open closing PR is already attached as the candidate fix.

## What This Changes

Ensure Docker sandbox skill directories are read-only on first launch even when host `skills` and `.agents/skills` directories do not yet exist.

## Best Possible Solution

Land a safe fix that creates or materializes missing skill mount source directories before first container launch while preserving canonical-root and symlink validation, with regression tests and real Docker first-launch proof.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.9` return `false` for missing skill mount sources, so `resolveReadOnlyWorkspaceSkillMounts` omits the read-only overlays before Docker create. I did not run a live Docker first-launch repro in this read-only review.

## Solution Assessment

Yes on direction, but the exact fix still needs review: the end state should pre-create the missing skill directories without weakening canonical path validation or symlink guards.
