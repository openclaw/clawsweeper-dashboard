## Summary

Keep open. Current main and v2026.6.8 still protect only already-existing workspace skill roots; absent `skills` and `.agents/skills` roots remain normal writable workspace paths, and prior merged work explicitly preserved that absent-root behavior.

## What This Changes

The issue asks OpenClaw to protect `/workspace/skills` and `/workspace/.agents/skills` as read-only from the first sandbox launch even when the host skill directories do not exist yet.

## Best Possible Solution

Define and land a sandbox-owned policy that reserves workspace skill roots before first launch, applies matching read-only behavior across Docker and filesystem bridge surfaces, and updates tests/docs for the compatibility change.

## Reproduction Assessment

Yes, at source level. Current main filters read-only skill mounts to existing directories and keeps an explicit absent-root write test, so the first-launch missing-directory path is traceable without a live Docker run.

## Solution Assessment

Unclear as a direct `ensureAgentWorkspace` mkdir-only fix. Precreating reserved roots is a plausible narrow Docker fix, but the best solution must also settle the compatibility and bridge-policy behavior left by https://github.com/openclaw/openclaw/pull/85591.
