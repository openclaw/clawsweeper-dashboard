Keep open: current main and the latest release still filter out missing workspace skill mount sources, and an open linked PR is already the candidate fix for the same security-boundary issue. The issue should remain canonical until that PR, or a safer replacement, merges.

## What This Changes

The issue reports that missing host `skills` directories make first-launch Docker sandboxes skip read-only skill overlays, leaving executable skill paths writable until a later restart.

## Best Possible Solution

Keep this issue open as the canonical tracker, then land a sandbox fix that materializes missing skill mount source directories before first launch while preserving canonical-root validation and proving first-launch Docker behavior.

## Reproduction Assessment

Yes at source level: current main returns false for missing skill mount sources, and Docker/browser/remote filesystem paths consume that filtered mount list. I did not run a live Docker first-launch repro in this read-only review.
