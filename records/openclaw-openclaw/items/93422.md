## What This Changes

Request to add a WebChat/Control UI session-naming surface via `/label <name>` and/or named `/new <name>` so dashboard sessions show friendly names instead of UUID-like keys.

## Best Possible Solution

Settle the canonical session naming contract, then land either the focused named `/new` implementation or a maintainer-approved replacement that reuses the existing label/name path and keeps the session picker/CLI output consistent.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.9 route Control UI `/new` to plain `new-session` creation without passing the parsed name, and no local `/label` command is registered. This is a missing feature surface rather than a broken existing bug contract.

## Solution Assessment
