## Summary

Keep open: current main has the underlying label storage/display plumbing and the shared `/name <title>` command, but it still does not implement the requested Control UI `/label <name>` alias or `/new <name>` session-label workflow. The open linked pull request is the direct implementation candidate, so this issue should remain open until maintainers merge, narrow, or reject that product direction.

## What This Changes

Request to let WebChat/Control UI users assign human-readable session labels via `/label <name>` and/or `/new <name>` and see those labels in session selection and session JSON output.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/93587; merge that branch if the Control UI command surface is accepted, or fold the remaining scope into https://github.com/openclaw/openclaw/issues/55249 if maintainers prefer one broader naming contract.

## Reproduction Assessment

Yes for source-level confirmation: current main has label storage/display plumbing and `/name`, but Control UI `/new` still ignores trailing text and there is no `/label` dispatch case. This is a feature gap rather than a failing bug path.

## Solution Assessment

Unclear until maintainers decide product scope. Reusing the existing `label` field is the right storage boundary, but a browser-local `/label` alias and changed `/new <text>` semantics are product choices beside the shared `/name` command.
