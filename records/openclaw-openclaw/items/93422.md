## Summary

Keep open: current main has the `/name <title>` rename path and session label storage/display primitives, but it still does not implement named `/new` or a `/label` alias; the remaining work is now covered by open follow-up PRs that need maintainer review.

## What This Changes

The issue requests chat-side session naming through `/label <name>` and/or `/new <name>` so WebChat and Control UI sessions show a friendly persisted name instead of an opaque session key.

## Best Possible Solution

Review and reconcile https://github.com/openclaw/openclaw/pull/95700 with https://github.com/openclaw/openclaw/pull/95182, keeping session naming on one canonical label/title path and deciding separately whether `/label` should be an alias for `/name`.

## Reproduction Assessment

Yes, source-level: current main's Control UI `/new` path ignores trailing args, `/label` is not registered locally, and backend tests preserve `/new <tail>` as prompt handoff. This is a missing feature path, not a broken existing bug contract.

## Solution Assessment

No, not as a separate `displayName` or independent `/label` contract. The best path is to reuse the canonical session label/title primitive, preserve existing `/new <prompt>` behavior, and let maintainers settle the linked PR direction.
