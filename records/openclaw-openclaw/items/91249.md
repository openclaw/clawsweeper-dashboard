## Summary

Keep open: this PR targets a real current-main Windows Startup-folder fallback bug, but it is not merge-ready because the proof is still mocked/source-path only, the uninstall docs point at the wrong VBS location, and the VBS default compatibility tradeoff needs maintainer acceptance or mitigation.

## What This Changes

The PR changes the Windows Startup-folder fallback to write a hidden VBS launcher by default, updates fallback lifecycle tests, and revises gateway/uninstall docs.

## Best Possible Solution

Land a narrow VBS-default Startup fallback only after live Windows desktop proof, accurate cleanup docs for the actual Startup-folder entry, and explicit maintainer acceptance or fallback handling for WSH-blocked hosts.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.6 still write `start "" /min cmd.exe /d /c` for the default Startup-folder fallback, and the current default fallback test asserts that output. I did not establish a live Windows desktop login reproduction in this review.

## Solution Assessment

No, not yet. The runtime approach is a plausible narrow fix, but the best merge shape needs live Windows proof, corrected cleanup docs, and a maintainer decision on VBS-only compatibility versus an explicit fallback.
