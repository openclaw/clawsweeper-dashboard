## Summary

Keep open: the source approach is plausible, but the PR is not merge-ready because it weakens legacy `.cmd` fallback coverage, lacks live Windows proof for the visible-console claim, and changes a Windows startup fallback default with compatibility risk.

## What This Changes

The PR changes the Windows Startup-folder fallback to write a hidden `.vbs` launcher by default instead of a minimized `.cmd` launcher and updates focused fallback tests.

## Best Possible Solution

Promote VBS as the Startup-folder fallback default only with preserved legacy `.cmd` coverage, updated Windows docs, and redacted live Windows proof; coordinate with https://github.com/openclaw/openclaw/pull/89380 if maintainers want one combined Windows launcher-default decision.

## Reproduction Assessment

Yes, source-level: current main still writes the minimized cmd Startup fallback by default unless the hidden-launcher env flag is set, matching the linked Windows 11 report. I did not run a live Windows login-session reproduction in this read-only review.

## Solution Assessment

Not yet: making the existing VBS launcher the default is likely the right layer, but this PR weakens legacy coverage and lacks live Windows proof and docs. The best merge-ready path preserves shipped `.cmd` cleanup coverage while proving the new `.vbs` user-visible behavior.
