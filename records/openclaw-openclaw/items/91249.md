## Summary

Keep open: the runtime change is a plausible narrow fix for the Windows Startup-folder console flash, but the PR is not merge-ready because proof is still mocked/source-level only, the cleanup docs point users at the wrong file location, and the VBS default needs maintainer acceptance for Windows Script Host-constrained hosts.

## What This Changes

The PR changes the Windows Startup-folder fallback to write a hidden `.vbs` launcher by default, updates focused fallback tests, and edits Windows gateway/uninstall docs.

## Best Possible Solution

Land a narrow VBS-default Startup fallback only after native Windows proof, accurate Startup-folder cleanup docs, and maintainer acceptance or mitigation for WSH-blocked hosts.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` still write a minimized `cmd.exe` Startup-folder fallback by default. I did not reproduce the compositor flash in a live Windows desktop login session.

## Solution Assessment

No, not yet. The runtime path is plausible and narrow, but the best merge shape still needs native Windows proof, accurate cleanup docs, and a maintainer decision on VBS-only compatibility versus an explicit fallback.

- **[P2] Document the actual Startup-folder launcher path:** `docs/install/uninstall.md:117`

Please make the wording distinguish the Startup-folder login item from the state-dir task script so status and cleanup docs do not point at the wrong artifact.
