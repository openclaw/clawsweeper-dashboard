## Summary

Keep open. The reported bug is real and the PR is the right direction, but the branch predates current main's env-backed workspace default contract and the proposed canonicalizer would mishandle that path after rebase; the external PR also still lacks real behavior proof.

## What This Changes

The PR adds default workspace path canonicalization and applies it to setup, onboarding, and configure config writes with focused tests.

## Best Possible Solution

Land a rebased canonicalization that converts only built-in home/profile defaults to portable tilde paths, preserves custom and env-backed workspace defaults, and includes redacted saved-config proof from a real CLI flow.

## Reproduction Assessment

Yes, source-level. Current main resolves the documented default workspace to a host-specific path and setup/onboarding/configure write paths persist that value; the PR regression around `OPENCLAW_WORKSPACE_DIR` is also source-reproducible from current docs/source plus the proposed canonicalizer.

## Solution Assessment

No as submitted. Write-time canonicalization is the right fix direction, but it must not turn an env-backed `OPENCLAW_WORKSPACE_DIR` default into `~/.openclaw/workspace`, and the PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve OPENCLAW_WORKSPACE_DIR defaults:** `src/agents/workspace-default.ts:36`
  - body: Current main now treats `OPENCLAW_WORKSPACE_DIR` as the default workspace when no config is set, specifically for mounted/default deployments that should not be written into config. After this patch is rebased, this return path would canonicalize that env-backed absolute default to `~/.openclaw/workspace`, creating an explicit config value that overrides the env-mounted workspace on later runs. Skip canonicalization when the default came from `OPENCLAW_WORKSPACE_DIR`, or preserve that resolved path unchanged.
