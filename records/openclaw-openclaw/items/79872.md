## Summary

Keep open. The PR targets a valid config portability bug, but current main now documents and implements `OPENCLAW_WORKSPACE_DIR` as a default workspace source, and the branch would rewrite that env-backed default to `~/.openclaw/workspace`; it also still lacks contributor-provided real behavior proof.

## What This Changes

The PR adds write-time default-workspace canonicalization in setup, onboarding, and configure flows plus tests so recognized default workspaces are stored as `~/.openclaw/...` while custom absolute paths stay absolute.

## Best Possible Solution

Rebase the branch, keep write-time canonicalization for home/profile defaults, preserve `OPENCLAW_WORKSPACE_DIR`-derived defaults unchanged, add regression coverage for that case, and provide redacted saved-config or CLI proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default workspaces to host-specific absolute paths and setup/onboarding/configure persist them; the PR regression is also source-reproducible because the new canonicalizer would treat an env-backed default as a canonical home/profile default.

## Solution Assessment

No, not as written. Write-time canonicalization is the narrow maintainable fix direction, but it must not discard the documented `OPENCLAW_WORKSPACE_DIR` default and it still needs real behavior proof.
