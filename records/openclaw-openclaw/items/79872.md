## Summary

Keep open: this is a focused fix for a source-reproducible config serialization bug, but the PR needs a small code correction for the current `OPENCLAW_WORKSPACE_DIR` contract and still lacks after-fix real behavior proof.

## What This Changes

The PR adds write-time canonicalization so recognized default workspace paths are persisted as portable `~/.openclaw/...` values in setup, onboarding, and configure config writes.

## Best Possible Solution

Rebase the PR, keep canonicalization for home/profile-derived defaults, preserve env-backed `OPENCLAW_WORKSPACE_DIR` defaults, and add redacted real saved-config or CLI flow proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default workspaces to host-specific paths and setup/onboarding/configure persist those values directly; the PR regression around `OPENCLAW_WORKSPACE_DIR` is also source-reproducible from the current docs/tests and the proposed canonicalizer.

## Solution Assessment

No, not as written. Write-time canonicalization is the right direction, but the implementation must not rewrite an env-backed default workspace to `~/.openclaw/workspace`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve env-backed workspace defaults:** `src/agents/workspace-default.ts:34`
  - body: Current main lets `OPENCLAW_WORKSPACE_DIR` define the default workspace, and the docs say that env value is used when no explicit `agents.defaults.workspace` exists. Skip canonicalization when `OPENCLAW_WORKSPACE_DIR` is set, or persist that env-backed path unchanged.
