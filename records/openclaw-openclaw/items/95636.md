## Summary

Keep open for maintainer review, but the current patch is not merge-ready: it changes the documented and shipped telemetry content-capture default instead of adding an approved log-body opt-in or using the existing broad `captureContent: true` path.

## What This Changes

The PR flips `NO_CONTENT_CAPTURE.logBodies` to `true` in the diagnostics-otel service so default OTLP/stdout log exports use rendered log messages.

## Best Possible Solution

Keep log bodies disabled by default; if maintainers want searchable log bodies without broad content capture, add an approved narrow opt-in with schema, docs, tests, and redacted exporter proof.

## Reproduction Assessment

Yes for source-level review: current main and v2026.6.9 keep default log bodies as `"log"` unless boolean `captureContent: true` is configured. I did not run a live Loki or stdout exporter path in this read-only review.

## Solution Assessment

No. Flipping `NO_CONTENT_CAPTURE` is the wrong layer because current docs, schema, tests, and shipped behavior make rendered log bodies an explicit content-capture decision; the safer path is existing `captureContent: true` or a maintainer-approved granular opt-in.
