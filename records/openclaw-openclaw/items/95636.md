## Summary

Keep open, but this PR is not merge-ready: it changes a privacy-sensitive telemetry default that current code, tests, and docs intentionally keep behind `diagnostics.otel.captureContent: true`. The linked report was closed by its reporter as a configuration/default decision, and the companion PR targets a different logger extraction path rather than approving this default-policy change.

## What This Changes

The PR changes `NO_CONTENT_CAPTURE.logBodies` from `false` to `true` in the diagnostics-otel plugin default content capture policy.

## Best Possible Solution

Keep log bodies disabled by default; if searchable log bodies are desired, add a maintainer-approved narrow opt-in with schema/help/docs/tests and redacted real exporter proof.

## Reproduction Assessment

Yes for source-level reproduction: current main emits `body: "log"` unless `diagnostics.otel.captureContent` is boolean `true`, and tests encode that behavior. I did not run a live Loki collector in this read-only review.

## Solution Assessment

No. Flipping `NO_CONTENT_CAPTURE` is not the best fix because the current docs, schema, and tests make log bodies an explicit content-capture decision; the safer path is a maintainer-approved opt-in or reaffirming `captureContent: true`.
