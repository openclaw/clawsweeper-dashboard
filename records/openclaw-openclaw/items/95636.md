## Summary

This PR should stay open but is not merge-ready: current main and v2026.6.9 make rendered OTLP log bodies an explicit `diagnostics.otel.captureContent: true` privacy opt-in, while the patch exports them by default without docs/schema/tests, maintainer privacy direction, or real exporter proof.

## What This Changes

The PR changes the diagnostics-otel no-content capture policy so OTLP/stdout log records include rendered log message bodies by default.

## Best Possible Solution

Keep log bodies disabled by default; if maintainers want searchable log bodies without broad capture, add an approved narrow opt-in with schema, docs, tests, and redacted exporter proof.

## Reproduction Assessment

Yes for source-level review: current main and tests show default and granular capture produce `body: "log"`, while this PR flips the no-content default. I did not run a live Loki or stdout exporter path in this read-only review.

## Solution Assessment

No. Flipping `NO_CONTENT_CAPTURE` is the wrong layer because docs, schema, tests, and shipped behavior make rendered log bodies an explicit content-capture decision; the safer path is existing `captureContent: true` or a maintainer-approved granular opt-in.
