## Summary

Keep open for maintainer handling, but the patch is not merge-ready: it changes the documented telemetry content-capture/privacy default instead of adding or using an explicit log-body opt-in.

## What This Changes

The PR changes diagnostics-otel's no-content capture policy so OTLP log bodies are captured by default.

## Best Possible Solution

Keep log bodies disabled by default; if maintainers want searchable log bodies without broad capture, add an approved narrow opt-in with schema, docs, tests, and redacted exporter proof.

## Reproduction Assessment

Yes for source-level review: current main and tests show default and granular capture produce `body: "log"`, while broad capture emits a redacted message body. I did not run a live Loki or stdout exporter path in this read-only review.

## Solution Assessment

No. Flipping `NO_CONTENT_CAPTURE` is the wrong layer because docs, schema, tests, and shipped behavior make rendered log bodies an explicit content-capture decision; the safer path is existing `captureContent: true` or a maintainer-approved granular opt-in.
