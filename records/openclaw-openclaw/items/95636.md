## Summary

Keep open with a blocking finding: this PR changes a documented no-content telemetry default, so existing OTLP/stdout log users would start exporting rendered log bodies without the current `captureContent` opt-in.

## What This Changes

The PR flips `NO_CONTENT_CAPTURE.logBodies` to `true` in the diagnostics-otel plugin default content capture policy.

## Best Possible Solution

Keep log bodies disabled by default; if searchable OTLP bodies are desired, add a maintainer-approved opt-in with schema/help/docs/tests and redacted real-behavior proof.

## Reproduction Assessment

Yes for source-level reproduction: current main routes default log export through `NO_CONTENT_CAPTURE`, and `buildDiagnosticLogRecord` emits body `"log"` unless log-body capture is enabled. I did not run a live Loki collector in this read-only review.

## Solution Assessment

No. Flipping the no-content default is not the best fix because the current docs, schema, and tests make log bodies an explicit content-capture decision; a narrow opt-in or reaffirmed existing `captureContent: true` path is safer.
