## Summary

Keep open for maintainer handling, but the patch is not merge-ready: it flips a documented privacy default and would export rendered OTLP log bodies without the existing content-capture opt-in.

## What This Changes

The PR changes the diagnostics-otel default content-capture policy so OTLP log bodies are captured when `diagnostics.otel.captureContent` is unset or false.

## Best Possible Solution

Keep the current privacy gate unless maintainers approve a narrow documented log-body opt-in with redaction tests and collector proof.

## Reproduction Assessment

Yes for source-level behavior: current main returns `NO_CONTENT_CAPTURE` by default and emits the placeholder body unless log-body capture is enabled. I did not run a live collector/Loki reproduction in this read-only review.

## Solution Assessment

No. Changing the no-content-capture default is the wrong layer for a privacy-sensitive behavior; the safer solution is maintainer-approved policy, likely a narrow opt-in or keeping `captureContent: true` as the only body-export path.
