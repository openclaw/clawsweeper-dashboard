## Summary

Keep open. The request is concrete and current main does not ship a stable `/usr/local/bin/oap-healthcheck` executable or any visible `oap-sidecar` image surface, while the existing OpenClaw Docker image already has HTTP probe endpoints and Node-based health probes for `ghcr.io/openclaw/openclaw`. Because the requested image name, port, and release surface are not represented in this repo, this needs maintainer packaging/product clarification before a repair PR is safe.

## What This Changes

Request that the OAP sidecar image include a stable `/usr/local/bin/oap-healthcheck` executable for shell-free container health checks.

## Best Possible Solution

Define the intended image boundary first. If this targets the existing OpenClaw gateway image, add a stable executable health probe that wraps the existing `/healthz` or `/readyz` endpoints, update Docker/Kubernetes docs and manifest examples to use it, and cover it with Dockerfile/package-surface tests plus a Docker lane. If it targets a separate OAP sidecar image, route the work to the repository or workflow that actually builds and publishes that image.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a packaging feature request. Current-main verification is a source/docs scan showing no `oap-healthcheck` binary or `oap-sidecar` image surface, while existing container probes are inline Node commands against the gateway health endpoints.

## Solution Assessment

Unclear. A dedicated shell-free probe binary is a reasonable portability direction, but the repo first needs a maintainer decision on the image name, default port/path, implementation language, and whether this belongs to the current gateway Docker image or a separate OAP sidecar release surface.
