The request targets `ghcr.io/openclaw/oap-sidecar`, but current main has no OAP sidecar image, Dockerfile, workflow, docs, or binary surface to change; the repo-published container is `ghcr.io/openclaw/openclaw` and already documents a built-in Node-based `/healthz` healthcheck. This should be redirected to the repository or service that owns the OAP sidecar image rather than tracked as OpenClaw source work.

## What This Changes

Request shipping a dependency-free `/usr/local/bin/oap-healthcheck` executable in the `ghcr.io/openclaw/oap-sidecar` container for orchestrator health checks.

## Best Possible Solution

Leave the existing OpenClaw gateway image healthcheck as documented, and route OAP sidecar binary requests to the source repository or service that owns and publishes that sidecar image. If OpenClaw should add a new OAP sidecar product surface, that needs a separate maintainer-scoped product and release proposal with an owning source path.
