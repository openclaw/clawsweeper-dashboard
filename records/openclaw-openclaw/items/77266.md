## Summary

Keep open: current main has adjacent managed-npm install-record recovery and `openclaw update` repair, but the direct package-manager upgrade path can still trust a persisted plugin registry that omits recoverable external npm plugins, leaving them hidden from plugin and channel status surfaces.

## What This Changes

The issue reports that external npm plugins disappear from plugin/channel status after upgrading via a pnpm global OpenClaw update and require manual reinstall.

## Best Possible Solution

Teach the plugin registry snapshot and inspection paths to merge recovered managed-npm install records before trusting or deriving from persisted state, with regression coverage and a user-facing warning path but no package-manager execution during Gateway startup.

## Reproduction Assessment

Yes, source-reproducible: a managed npm root can contain plugin packages while the persisted registry/install ledger omits them, and the snapshot path can return or derive from that stale persisted state without loading recovered records.

## Solution Assessment

Yes, the narrow maintainable repair is registry/ledger recovery before persisted-state trust; running npm during Gateway startup or requiring users to reinstall plugins after package-manager upgrades would be broader and less safe.
