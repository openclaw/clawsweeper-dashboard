Keep open: current main has adjacent fixes for sibling npm installs and `openclaw update`, but the direct package-manager upgrade path can still trust an otherwise-current persisted plugin registry that omits recoverable managed-npm plugins.

## What This Changes

The issue requests that externally installed npm plugins remain visible and enabled after upgrading OpenClaw with `pnpm add -g openclaw@latest`.

## Best Possible Solution

Make persisted registry trust account for recovered managed-npm install records after package-manager upgrades, then add regression coverage so external plugins reappear without running package managers during Gateway startup.

## Reproduction Assessment

Yes, source-reproducible: create a managed npm root with plugin package entries but a current-looking persisted registry that omits them, then `loadPluginRegistrySnapshotWithMetadata` can return the stale persisted snapshot while registry inspection would detect source drift.

## Solution Assessment

No, the current partial fixes are not complete for this report; the safer fix is registry/ledger recovery and warning, not running npm during startup or package postinstall.
