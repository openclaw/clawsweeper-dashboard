## Summary

Keep open: current main has not implemented the central full-reply-to-thread behavior, and this PR is not mergeable as-is because it bypasses current announce/control-token safeguards and lacks after-fix real behavior proof.

## What This Changes

The branch adds a direct full-reply send to the announce target in `runSessionsSendA2AFlow`, moves announce generation into the requester session, suppresses duplicate announce sends, and passes `threadId`.

## Best Possible Solution

Define one policy-gated final-delivery path after ping-pong that filters non-deliverable control tokens, preserves scoped lanes and provenance, and ships with focused tests plus redacted real thread proof.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of PR head shows direct `latestReply.trim()` channel delivery while current main gates external delivery through announce suppression and non-deliverable-token checks. The original threaded ack complaint still lacks after-fix real setup proof in this PR.

## Solution Assessment

No. Direct raw sends are not the narrowest maintainable solution because they bypass the existing announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
