## Label Justifications

- `P1`: The PR touches an active sessions_send delivery workflow where a bad merge can misroute, drop, or leak visible cross-session/thread messages.
- `merge-risk: 🚨 message-delivery`: The diff changes what content is posted to an announce target and can suppress the existing announce send after a direct full-reply post.
- `merge-risk: 🚨 session-state`: The success path still runs an announce step in the requester session even when its output cannot be delivered, which can write synthetic announce context into session history.
- `merge-risk: 🚨 security-boundary`: Directly sending raw target replies can bypass `ANNOUNCE_SKIP` and announce-step filtering that currently act as a cross-session delivery boundary.

## Summary

This PR should stay open: the thread-delivery problem is plausible, but the branch is not merge-ready because it bypasses the announce safety gate, leaves an unused announce run in the success path, and still lacks real transport proof.

## What This Changes

The PR changes `runSessionsSendA2AFlow` to directly post the final target reply to an announce target, run the announce step in the requester session, suppress duplicate announce delivery after a successful direct post, and pass `threadId` to gateway sends.

## Best Possible Solution

Keep the current-main announce gate as the safety boundary, then implement full-reply thread delivery only after a clear policy decision and tests/proof showing final post-ping-pong content is delivered once without bypassing skip or non-deliverable filtering.

## Reproduction Assessment

Yes at source level, but not as a live reproduction in this review. The current-main announce path and PR-head direct-send path clearly show how duplicate suppression, skip handling, and final-reply delivery would behave.

## Solution Assessment

No. Posting the final reply after ping-pong addresses the earlier intermediate-reply concern, but the current patch still bypasses announce suppression/filtering and runs an unused announce step after successful direct delivery.
