## Summary

Keep open for maintainer handling. The PR targets a real A2A delivery pain point, but the branch is not safe to merge: it directly posts raw `latestReply` to the user-visible announce target, bypasses the announce/control-token policy, regresses scoped lanes/provenance, and has no after-fix real behavior proof.

## What This Changes

The PR changes `runSessionsSendA2AFlow` to send the final `latestReply` directly to the resolved announce target, run the announce step in the requester session, suppress duplicate announce sends, and pass `threadId`.

## Best Possible Solution

Define one policy-gated final A2A delivery path after ping-pong that can deliver the intended content only after control-token filtering and announce suppression, while preserving lanes, provenance, `threadId`, docs, tests, and real proof.

## Reproduction Assessment

Yes for the PR-introduced defects: PR-head source shows direct `latestReply.trim()` channel delivery and shared-lane requester announce behavior, while current main gates external delivery and scopes nested lanes. The original threaded ack complaint still lacks after-fix real setup proof.

## Solution Assessment

No. Direct raw full-reply sends are not the narrowest maintainable solution because they bypass the existing announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
