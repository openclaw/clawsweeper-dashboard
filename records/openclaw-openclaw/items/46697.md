## Summary

Keep open: the PR is not merge-ready because the branch publishes raw A2A replies directly to a user-visible target before the current announce suppression and control-token filters can approve delivery, and it still lacks after-fix real threaded setup proof.

## What This Changes

The PR changes `runSessionsSendA2AFlow` to send `latestReply` directly to the resolved announce target with `threadId`, run the announce step in the requester session, and suppress duplicate announce sends.

## Best Possible Solution

Define one policy-gated final A2A delivery path after ping-pong that filters control replies, honors announce suppression, preserves lanes/provenance/threadId, and is covered by tests plus real threaded proof.

## Reproduction Assessment

Yes for the PR-introduced defects: PR-head source shows direct `latestReply.trim()` channel delivery and shared-lane requester announce behavior, while current main gates delivery and scopes nested lanes. The original threaded ack complaint still lacks after-fix real setup proof.

## Solution Assessment

No. Direct raw full-reply sends are not the narrowest maintainable solution because they bypass the existing announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
