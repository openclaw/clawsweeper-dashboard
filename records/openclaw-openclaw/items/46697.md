## Summary

Keep open but block merge: the branch targets a real A2A thread-delivery gap, but it bypasses the announce suppression contract, can pollute requester session state, is stale against current main's hardened A2A path, and lacks after-fix real behavior proof.

## What This Changes

The branch changes `sessions_send` A2A to send the final `latestReply` directly to the announce target, run the announce step in the requester session, suppress duplicate announce sends, and pass `threadId` to the gateway send call.

## Best Possible Solution

Rebase onto current main and implement final-reply visible delivery through one policy-aware A2A delivery path that preserves `ANNOUNCE_SKIP`/non-deliverable suppression, correct target provenance, `threadId`, and live upgrade behavior.

## Reproduction Assessment

Yes, at source level. The PR head sends `latestReply` directly to `announceTarget` before the announce step can return `ANNOUNCE_SKIP`, and current main's helper/test contract shows that announce output is supposed to be policy-gated.

## Solution Assessment

No. The safer fix is to preserve the final-reply-after-ping-pong intent while keeping announce suppression, non-deliverable filtering, target provenance, hidden transcript handling, and current-main A2A boundary behavior intact.
