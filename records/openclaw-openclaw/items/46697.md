## Summary

Keep open. The PR still proposes a meaningful A2A delivery change, but the current head is not merge-ready because it bypasses current announce policy/control-token filtering and drops scoped-lane/provenance safeguards; contributor real behavior proof and maintainer product/security direction are still required.

## What This Changes

The PR changes `runSessionsSendA2AFlow` to send `latestReply` directly to the announce target after ping-pong, run announce generation in the requester session, suppress duplicate announce sends, and pass `threadId`.

## Best Possible Solution

Define one policy-gated final A2A delivery path after ping-pong that can deliver intended content only after control-token filtering and announce suppression, while preserving lanes, provenance, `threadId`, tests, docs, and real proof.

## Reproduction Assessment

Yes for the PR-introduced defects: PR-head source shows raw direct `latestReply` channel delivery and shared-lane requester announce behavior, while current main gates external delivery and scopes nested lanes. The original threaded ack complaint still lacks after-fix real setup proof.

## Solution Assessment

No. Direct raw full-reply sends are not the narrowest maintainable solution because they bypass the current announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
