## Summary

Keep open: the requested threaded full-reply behavior is not already on current main, but this PR head is not safe to merge because it adds a raw external delivery path that bypasses current A2A announce suppression, control-token filtering, and session-scoped lane behavior.

## What This Changes

The PR modifies `runSessionsSendA2AFlow` to send `latestReply` directly to the resolved announce target, move announce generation to the requester session, suppress duplicate announce sends, and pass `threadId`.

## Best Possible Solution

Define one policy-gated final-delivery path after ping-pong that filters non-deliverable control tokens, preserves thread routing plus session-scoped lane/provenance, and ships with focused tests and redacted real thread proof.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of PR head shows direct `latestReply.trim()` channel delivery while current main gates external delivery through announce suppression and non-deliverable-token checks. The original threaded ack complaint still lacks after-fix real setup proof in this PR.

## Solution Assessment

No. Direct raw sends are not the narrowest maintainable solution because they bypass the existing announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
