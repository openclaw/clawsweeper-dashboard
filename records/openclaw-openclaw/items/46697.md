## Summary

Keep open. Current main still sends the announce-step reply to the announce target rather than the target session's full final reply, but this PR is stale and the patch bypasses current announce-suppression and provenance guardrails.

## What This Changes

The PR adds direct delivery of the final A2A `latestReply` to the resolved announce target, then runs the announce step in the requester session and suppresses the announce send when direct delivery succeeds.

## Best Possible Solution

Implement a refreshed current-main fix that delivers the final post-ping-pong, deliverable target reply to the announce target while preserving current non-deliverable-token guards, per-session lanes, transcript suppression, provenance, tests, and changelog coverage.

## Reproduction Assessment

Yes, from source inspection. Current main posts the announce-step reply to the announce target, so the target session's full `latestReply` can be replaced by a short acknowledgement in the linked thread.

## Solution Assessment

No. The PR targets the right flow, but the proposed direct-send path bypasses announce suppression, runs an unused announce model step after successful delivery, and records inconsistent provenance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve announce suppression before direct delivery:** `src/agents/tools/sessions-send-tool.a2a.ts:104-110`
  - body: This direct send publishes `latestReply` to the announce target before the announce phase can return `ANNOUNCE_SKIP`, and this stale branch also lacks current-main non-deliverable token checks. That changes the existing contract from allowing the announce step to remain silent into always posting the target reply or control token to the linked thread.
  - confidence: 0.9
- **[P2] Skip the unused announce step after full delivery:** `src/agents/tools/sessions-send-tool.a2a.ts:141-144`
  - body: When `fullReplyPosted` succeeds, the later send branch is blocked by `!fullReplyPosted`, so this `runAgentStep` result cannot affect channel output. Running it anyway adds another nested model run and, unlike current main, omits `transcriptMessage: ""`, so the synthetic announce prompt can be written into the requester transcript.
  - confidence: 0.87
- **[P2] Use target channel in target provenance:** `src/agents/tools/sessions-send-tool.a2a.ts:148-149`
  - body: The announce step now marks `sourceSessionKey` as the target session while keeping `sourceChannel` as the requester channel. When requester and target channels differ, inter-session provenance is inconsistent; pass the target channel with the target session.
  - confidence: 0.82
