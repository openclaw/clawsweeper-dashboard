## Summary

Keep open: the PR targets a plausible threaded A2A delivery gap, but the branch is not merge-ready because it sends raw cross-session replies to an external channel before the existing announce/control-token policy can approve or suppress them, and it lacks after-fix real threaded proof.

## What This Changes

This PR modifies `runSessionsSendA2AFlow` to send `latestReply` directly to the resolved announce target after ping-pong, move announce generation into the requester session, suppress duplicate announce sends, and pass `threadId`.

## Best Possible Solution

Define one policy-gated final-delivery path after ping-pong that filters non-deliverable control tokens, preserves session-scoped lanes and provenance, and ships with focused tests plus redacted real thread proof.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection of PR head shows direct `latestReply.trim()` channel delivery while current main gates external delivery through announce suppression and non-deliverable-token checks. The original threaded ack complaint still lacks after-fix real setup proof in this PR.

## Solution Assessment

No. Direct raw sends are not the narrowest maintainable solution because they bypass the existing announce policy boundary; a single policy-gated final-delivery decision after ping-pong is safer.
