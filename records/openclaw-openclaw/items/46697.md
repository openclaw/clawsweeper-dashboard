## Summary

Keep this PR open, but not as a cleanup close: the intended A2A thread-content problem is still not implemented on current main, and the PR branch introduces correctness and security regressions around announce suppression and provenance.

## What This Changes

The PR changes the A2A `sessions_send` flow to post the target session's full reply directly to the announce target, then run a requester-side announce step and suppress duplicate announce delivery.

## Best Possible Solution

Define one final A2A delivery path that gates external posting on suppression and provenance checks, chooses the approved final content, sends at most one message, and includes focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is an A2A send with an announce target, non-empty `latestReply`, and an announce step returning `ANNOUNCE_SKIP`; the PR's direct-send block runs before that suppression can take effect, while current main suppresses the external send.

## Solution Assessment

No. Posting the raw reply directly before policy evaluation is not the narrowest maintainable fix; the safer direction is to keep one suppression-aware delivery path and only change the content it is allowed to send.

  - body: This sets `sourceSessionKey` to the target session while keeping `sourceChannel` as the requester channel. For cross-channel A2A, downstream prompt/provenance text can label the foreign target reply as coming from the wrong channel.
  - confidence: 0.8

  - body: When full-reply delivery succeeds, the later `fullReplyPosted` guard discards the announce result, but the patch still runs another agent step. That adds latency/cost and can record a synthetic announce turn with no deliverable output.
