## Summary

Keep open. Current main still lacks a hook/webhook-specific lifecycle policy, and the remaining work is a user-facing retention/config decision rather than conservative cleanup.

## What This Changes

Request configurable lifecycle handling to archive or remove completed webhook/hook sessions with optional delay and transcript retention.

## Best Possible Solution

Add an explicit hooks.sessionCleanup policy with conservative defaults, schema/help/docs coverage, runtime finalization for direct, mapped, and Gmail hooks, and regression tests for agent:<id>:hook:gmail:* cleanup semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Gmail or mapped hooks can resolve hook:gmail:<messageId>, run through isolated cron execution, persist as agent:<id>:hook:gmail:<messageId>, and bypass the cron-run-only reaper.

## Solution Assessment

Unclear until maintainers choose retention semantics. The requested direction is reasonable, but the maintainable shape is an explicit hooks.sessionCleanup policy rather than silently reusing subagent archive settings.
