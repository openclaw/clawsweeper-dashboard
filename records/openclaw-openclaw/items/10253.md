## Summary

Keep open: current main and v2026.6.8 still lack a hook-specific cleanup/archive policy, and the related open work covers cron `deleteAfterRun` cleanup or persistent hook-session reuse rather than this lifecycle decision.

## What This Changes

Request a configurable auto-cleanup and archival policy for hook/webhook sessions, including completion or delayed cleanup and transcript-retention behavior.

## Best Possible Solution

Define one maintainer-approved hook session lifecycle contract with safe defaults, explicit archive/delete/no-op modes, transcript-retention semantics, direct/mapped/Gmail hook coverage, precedence over existing maintenance, docs, schema, and focused gateway/cron tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches direct and mapped hook agent runs through isolated cron execution, stores non-`cron:` hook keys as ordinary agent session rows, and has only generic or cron-specific retention cleanup.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. A hook-specific policy is plausible, but the best fix must align destructive cleanup with existing session maintenance, cron retention, and persistent hook-session semantics.
