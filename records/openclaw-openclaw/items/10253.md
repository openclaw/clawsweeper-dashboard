## Summary

Current main still lacks the requested hook/webhook session cleanup policy. Generic session maintenance can eventually prune hook rows by age/count, but it does not provide on-complete cleanup, delayed cleanup, transcript retention, or completed-session visibility semantics.

## What This Changes

Request to add a configurable lifecycle policy for hook/webhook sessions so completed one-shot hook runs can be archived or removed with explicit transcript-retention behavior.

## Best Possible Solution

Define one core isolated task-session lifecycle policy covering hooks and cron-style one-shot runs, with explicit completion, delay, transcript archive/delete, list visibility, and precedence over generic session maintenance.

## Reproduction Assessment

Yes at source level: hook sessions are resolved to agent-scoped hook keys and only generic retention/cap maintenance can later prune them. I did not run a live Gmail hook because this is a read-only review.

## Solution Assessment

No for a standalone hook-only cleanup subsystem; the safer direction is a shared isolated task-session lifecycle with hook-specific config/docs folded into that policy.
