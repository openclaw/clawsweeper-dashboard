## Summary

Keep open. Current main still has retention-based cron/session maintenance and hook dispatch through isolated cron-style runs, but it has no hook/webhook-specific lifecycle policy for on-complete cleanup, delayed cleanup, transcript retention, or completed-session visibility; the broader ephemeral-session issue and prior stale PR are related but do not fully replace this hook/Gmail-specific product request.

## What This Changes

The issue requests a configurable lifecycle policy for hook/webhook-created sessions so completed runs can be archived, deleted, delayed, or retained with transcript controls.

## Best Possible Solution

Define an additive, opt-in hook lifecycle policy for direct /hooks/agent, mapped hooks, Gmail presets, generated hook:* keys, delayed cleanup, archive/delete behavior, transcript retention, and completed-session visibility, coordinated with the broader ephemeral-session design.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that current main cannot express or apply the requested hook-session lifecycle policy.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. The maintainable direction is an explicit hook cleanup policy or coordinated ephemeral-session design, not overloading generic session.maintenance, cron.sessionRetention, or subagent archive settings.
