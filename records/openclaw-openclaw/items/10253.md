## Summary

Keep open. Current main still lacks a hook/webhook-specific lifecycle policy; the existing generic session maintenance and cron run-session reaper do not provide on-complete hook cleanup, delayed hook cleanup, transcript-retention choices, or completed-hook visibility semantics.

## What This Changes

Add configurable lifecycle cleanup/archive behavior for webhook and hook sessions, including on-complete or delayed cleanup and transcript-retention controls.

## Best Possible Solution

Add an explicit hooks.sessionCleanup policy with conservative defaults, schema/help/docs coverage, runtime finalization for direct, mapped, and Gmail hooks, and regression tests for agent-scoped hook key cleanup semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Gmail or mapped hooks can resolve hook:gmail:<messageId>, run as isolated cron-style agent turns, persist as agent:<id>:hook:gmail:<messageId>, and bypass the cron-run-only reaper and sessions_list cron alias filter.

## Solution Assessment

Unclear until maintainers choose retention semantics. The requested direction is reasonable, but the maintainable solution is an explicit hooks.sessionCleanup policy rather than silently reusing subagent archive settings.
