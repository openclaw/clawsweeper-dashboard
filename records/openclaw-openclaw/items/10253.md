## Summary

Keep open: current main still lacks a hook/webhook-specific completion cleanup or archival policy, and the related cron cleanup PR is intentionally scoped to cron `deleteAfterRun`, not hook lifecycle semantics.

## What This Changes

The issue requests configurable auto-archive/delete behavior, delayed cleanup, transcript retention, and completed-session visibility handling for hook-triggered sessions such as Gmail webhooks.

## Best Possible Solution

Define one maintainer-approved hook/session lifecycle contract with clear defaults, precedence over existing retention maintenance, transcript archive/delete semantics, strict schema/help/docs coverage, and tests for direct hooks, mapped hooks, and Gmail session keys.

## Reproduction Assessment

Yes, at source level. Current main dispatches direct and mapped hook agent runs through isolated cron execution, stores non-`cron:` hook keys as ordinary agent session rows, and has only cron-retention or generic session-maintenance pruning rather than hook completion cleanup.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. A bare hook-only config knob is not obviously the best fix because destructive cleanup must align with existing cron and global session-maintenance owners.
