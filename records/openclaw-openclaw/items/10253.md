## Summary

Keep open. Current main still lacks a hook/webhook-specific lifecycle policy, and the existing cron/session maintenance paths do not provide on-complete or delayed hook-session archive/delete semantics with transcript-retention control.

## What This Changes

The issue requests configurable archive/delete/no-op cleanup for completed webhook or hook-created sessions, including delayed cleanup, transcript retention, and completed-session visibility behavior.

## Best Possible Solution

Add an explicit core hook-session lifecycle policy with conservative defaults, schema/help/docs coverage, and runtime finalization for direct /hooks/agent, mapped hooks, Gmail preset handling, and isolated-run completion, including tests for agent:<id>:hook:gmail:* persistence and cleanup semantics.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: dispatch a mapped Gmail hook that resolves to hook:gmail:<messageId>; current main persists it as agent:<id>:hook:gmail:<messageId> and only cron run aliases are reaped or hidden.

## Solution Assessment

Unclear until maintainers choose retention semantics. The requested direction is reasonable, but the safest implementation should be an explicit hooks.sessionCleanup policy rather than reusing subagents.archiveAfterMinutes implicitly.
