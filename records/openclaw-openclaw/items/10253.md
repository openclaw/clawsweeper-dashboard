## Summary

Keep open. Current main has generic session maintenance and cron-run retention, but still lacks the hook-specific lifecycle policy, on-complete or delayed cleanup, transcript-retention controls, and completed-hook visibility semantics requested here.

## What This Changes

The issue requests configurable cleanup or archival of completed hook/webhook sessions, including delayed cleanup and transcript retention policy.

## Best Possible Solution

Add an explicit hooks.sessionCleanup policy with default-safe schema, help/docs, and runtime finalization for direct, mapped, and Gmail hooks; define archive/delete/no-op modes, optional delay, transcript retention, and completed-session visibility while coordinating with existing maintenance settings.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Gmail or mapped hooks resolve hook:gmail:<messageId> or hook:<uuid>, dispatch as isolated cron-style turns, persist as agent:<id>:hook:* rows, and bypass both the cron-run reaper and the sessions.list cron-run filter.

## Solution Assessment

Unclear until maintainers choose retention semantics. The requested direction is reasonable, but the maintainable solution is an explicit hooks.sessionCleanup policy rather than silently reusing subagent archive settings.
