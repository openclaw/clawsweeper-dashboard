## Summary

Keep open: current main and v2026.6.8 still have hook session-key routing, generic session maintenance, and cron-specific cleanup, but no hook/webhook lifecycle policy for on-complete archive/delete, delayed cleanup, transcript retention, or completed-session visibility.

## What This Changes

The issue requests configurable cleanup or archival for hook/webhook-created sessions after completion.

## Best Possible Solution

Define one maintainer-approved hook session lifecycle contract with safe defaults, explicit archive/delete/no-op modes, transcript-retention semantics, direct/mapped/Gmail hook coverage, precedence over existing maintenance, docs, schema, and focused gateway/cron tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests through isolated cron execution, stores non-cron hook keys as ordinary agent-scoped session rows, and has no hook-specific cleanup/archive config surface.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. A hook-specific policy is plausible, but the best fix must align destructive cleanup with existing session maintenance, cron retention, cron delete-after-run, and persistent hook-session semantics.
