## Summary

Keep open: current main and the latest release still lack a hook/webhook-specific lifecycle policy for on-complete cleanup, delayed cleanup, transcript retention, or completed-session visibility. The merged related work only fixes cron `deleteAfterRun` cleanup, and the open persistent hook-session PR targets multi-turn reuse rather than cleanup/archive policy.

## What This Changes

The issue requests a configurable cleanup/archive lifecycle policy for completed webhook/hook sessions.

## Best Possible Solution

Define one maintainer-approved hook session lifecycle contract with safe defaults, explicit archive/delete/no-op or delay modes, transcript-retention semantics, direct/mapped/Gmail hook coverage, precedence over existing maintenance, docs, schema, and focused gateway/cron tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests through isolated cron execution, stores non-cron hook keys as ordinary agent-scoped session rows, and has no hook-specific cleanup/archive config surface; I did not run a live Gmail/webhook setup in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. A hook-specific cleanup policy is plausible, but the best fix must align destructive cleanup with existing session maintenance, cron retention, cron deleteAfterRun, and persistent hook-session semantics.
