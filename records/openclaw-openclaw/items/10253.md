## Summary

Keep open. Current main still has only retention-based cron/session cleanup and no hook/webhook-specific lifecycle policy for on-complete cleanup, delayed cleanup, transcript retention, or completed-session visibility; the broader ephemeral-session issue overlaps but does not supersede this Gmail/hook-focused request.

## What This Changes

Request a configurable lifecycle policy for webhook/hook-triggered sessions so completed hook runs can be archived or cleaned up without manual session-store edits.

## Best Possible Solution

Define an additive opt-in hook lifecycle policy for direct /hooks/agent, mapped hooks, Gmail presets, generated hook:* keys, delayed cleanup, archive/delete behavior, transcript retention, and completed-session visibility.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that current main cannot express or apply the requested hook-session lifecycle policy.

## Solution Assessment

Unclear until maintainers choose lifecycle semantics. The maintainable direction is an explicit opt-in hook cleanup policy, not overloading generic session.maintenance, cron.sessionRetention, subagent archive settings, or cron deleteAfterRun.
