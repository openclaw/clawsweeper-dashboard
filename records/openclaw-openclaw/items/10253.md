## Summary

Keep open. Current main has generic session maintenance that can eventually prune synthetic hook rows, but it still lacks the requested hook-specific lifecycle policy for on-complete or delayed cleanup, archive/delete mode, transcript retention, and completed-hook visibility; the related cleanup PRs I found are closed/unmerged, and the open persistent-session PR targets a different issue.

## What This Changes

Request a configurable lifecycle policy for webhook and hook-triggered sessions so completed hook sessions can be archived or removed with optional transcript retention.

## Best Possible Solution

Add an explicit hook cleanup policy, likely hooks.sessionCleanup or hooks.sessionRetention, with schema/help/docs and runtime finalization for direct, mapped, and Gmail hooks while coordinating with existing session maintenance.

## Reproduction Assessment

Yes. Source inspection on current main shows Gmail or mapped hook keys flow through isolated hook dispatch, persist as agent:<id>:hook:* rows, and do not have a hook-specific cleanup config or finalization path.

## Solution Assessment

Unclear until maintainers choose retention semantics. The requested direction is reasonable, but the maintainable solution should be an explicit hook cleanup policy rather than silently reusing subagent archive settings or relying only on generic session maintenance.
