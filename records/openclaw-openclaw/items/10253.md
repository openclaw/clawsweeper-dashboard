## Summary

Keep open: current main still lacks a hook/webhook-specific completion cleanup or archival policy, and the existing cleanup surfaces are only generic session maintenance or cron-specific delete-after-run behavior.

## What This Changes

Add configurable auto-cleanup, archival, delay, and transcript-retention policy for completed webhook/hook-triggered sessions.

## Best Possible Solution

Define one maintainer-approved hook session lifecycle contract with safe defaults, explicit archive/delete/no-op modes, delay semantics, transcript retention, direct/mapped/Gmail coverage, docs, schema, and focused gateway/cron tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent work through isolated cron execution, stores non-cron hook keys as ordinary agent session rows, and only has generic age/count/disk maintenance or cron-specific cleanup.

## Solution Assessment

Unclear until maintainers choose the lifecycle contract. A hook-specific policy is plausible, but the best fix must define safe defaults and precedence with existing session maintenance, cron retention, and persistent hook sessions.
