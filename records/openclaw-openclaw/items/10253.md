## Summary

Keep this issue open: current main still lacks a hook/webhook-specific lifecycle policy, and the active related cron cleanup PR only covers `deleteAfterRun` cron run sessions, not delayed hook cleanup, transcript-retention policy, or completed hook visibility.

## What This Changes

Request a configurable lifecycle policy for webhook/hook-triggered sessions so completed hook runs can be archived or removed with transcript-retention controls.

## Best Possible Solution

Define one maintainer-approved isolated hook/session lifecycle with clear defaults, precedence over existing retention maintenance, transcript archive/delete semantics, and focused hook/Gmail coverage before adding config surface.

## Reproduction Assessment

Yes at source level. Current main dispatches hook runs through isolated cron execution, persists non-cron hook keys as ordinary agent sessions, and has only cron-retention or generic session-maintenance pruning rather than hook completion cleanup.

## Solution Assessment

No, not as an unreviewed hook-only config knob. The maintainable direction is a shared isolated-session lifecycle decision that covers hooks, cron interactions, transcript retention, and completed-session visibility without creating conflicting cleanup owners.
