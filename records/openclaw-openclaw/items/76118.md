## Summary

Keep open. The main behavior fix is directionally correct for the linked startup crash, but the latest PR head still statically imports `./resolve.js` into `runtime.ts`, regressing the existing fast-path lazy resolver boundary before merge.

## What This Changes

The PR splits startup SecretRef resolution so config refs remain fail-fast while auth-store refs resolve per profile, downgrades stale auth-profile provider references to a warning, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the profile-scoped provider-error downgrade, but route the provider-error type guard through the existing lazy runtime-prepare boundary before landing.

## Reproduction Assessment

Yes. Current main can reproduce the issue by preparing a startup secrets snapshot with an auth-store `api_key.keyRef` or `tokenRef` whose SecretRef provider is absent, causing the batch resolver to throw before activation can continue.

## Solution Assessment

Mostly yes, but not as currently wired. Splitting config and auth-store resolution is the narrow maintainable fix, while the provider-error check should avoid eagerly importing the resolver module from `runtime.ts`.
