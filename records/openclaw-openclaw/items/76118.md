## Summary

Keep open for maintainer/secops review. The PR addresses the linked gateway-startup bug by isolating config versus auth-store SecretRef resolution while preserving hard failures for config refs and missing env-backed auth refs; I found no blocking correctness issue.

## What This Changes

The branch resolves auth-store SecretRefs per profile, warns on stale provider refs, updates lazy-boundary mocks, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Merge this PR after secops/codeowner review and normal branch freshness checks, keeping config SecretRefs strict while treating stale auth-profile providers as profile-scoped degradation.

## Reproduction Assessment

Yes. On current main, an auth store containing an `api_key.keyRef` or `tokenRef` whose provider is absent from `secrets.providers` flows into the startup-wide `resolveSecretRefValues` call and reproduces the reported required-secrets startup abort.

## Solution Assessment

Yes. Splitting config and auth-store assignments is the narrow maintainable fix because it keeps required config secret resolution fail-closed while limiting stale auth-profile provider failures to the affected profile.
