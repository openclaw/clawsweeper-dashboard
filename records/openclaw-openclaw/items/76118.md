## Summary

Keep open. The PR targets a real startup failure and the added tests cover the intended per-profile degradation, but the patch currently pierces the secrets runtime lazy boundary and still needs the required changelog entry before merge.

## What This Changes

The branch splits config SecretRef resolution from auth-store SecretRef resolution, downgrades per-profile auth-store resolution failures to a warning, and adds regression coverage.

## Best Possible Solution

Keep the per-profile auth-store degradation, but classify resolver errors through the already-loaded lazy runtime helper and add the Unreleased changelog entry.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: a stale auth-profile SecretRef enters the shared assignment batch and `resolveSecretRefValues` stops on the first provider/ref error before snapshot activation; the PR tests model that stale exec-provider case and a valid sibling profile.

## Solution Assessment

No. The behavioral direction is right, but statically importing resolver error classes in `runtime.ts` is not the narrowest maintainable implementation because it bypasses the existing lazy prepare boundary; expose the error check from `runtime-prepare.runtime.js` instead.
