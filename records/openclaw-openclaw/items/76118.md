## Summary

Keep open. The functional fix matches the reported current-main failure path, but the latest diff makes `src/secrets/runtime.ts` eagerly import the resolver module that the secrets fast path currently keeps behind `runtime-prepare.runtime.ts`; that lazy-boundary regression should be repaired before merge.

## What This Changes

This PR separates config SecretRef resolution from auth-profile SecretRef resolution, downgrades auth-profile provider-resolution failures to a warning, adds regression tests, and adds a changelog entry for #75814.

## Best Possible Solution

Keep the profile-scoped provider-error behavior, but route the provider-error type guard through the existing lazy runtime-prepare boundary so fast-path startup remains lean, then land after targeted tests and build.

## Reproduction Assessment

Yes. Current main can reproduce the bug by preparing a startup secrets snapshot with an auth-store api_key or token profile whose keyRef/tokenRef points at an unconfigured exec provider; the batch resolver throws before startup activation can continue.

## Solution Assessment

Mostly yes, but not as currently wired. Splitting config and auth-store resolution is the narrow maintainable fix, while the provider-error check should avoid an eager resolver import in `runtime.ts`.
