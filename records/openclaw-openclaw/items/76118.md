## Summary

Keep open: the PR addresses a real startup reliability bug, but the latest head still statically imports the SecretRef resolver from `runtime.ts`, regressing the existing lazy startup boundary.

## What This Changes

The PR separates config and auth-store SecretRef startup resolution so stale auth-profile exec providers warn per profile while config refs remain fail-fast, with regression tests and a changelog entry.

## Best Possible Solution

Keep the profile-scoped provider-error downgrade, but expose or use the provider-scoped error guard through the existing lazy runtime-prepare boundary before merge.

## Reproduction Assessment

Yes. Current main can reproduce the linked bug by preparing startup secrets with an auth-store `keyRef` or `tokenRef` whose exec provider is absent, which flows through the shared batch resolver into the startup-abort path.

## Solution Assessment

Mostly yes, but not as currently wired. Splitting config and auth-store resolution is the narrow maintainable fix; the provider-error check should stay behind the lazy runtime-prepare boundary.
