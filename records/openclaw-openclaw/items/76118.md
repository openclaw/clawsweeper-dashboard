## Summary

Keep open. The reported startup failure is real and the PR targets the right area, but the patch is not merge-ready because it conflicts with existing auth-store SecretRef behavior coverage, current node runtime CI is failing, and the user-facing fix needs a changelog entry.

## What This Changes

The PR separates config and auth-store SecretRef resolution, downgrades unresolved auth-profile refs to a new warning, and adds regression coverage for stale auth-profile refs.

## Best Possible Solution

Define the startup auth-profile SecretRef contract explicitly, keep stale dormant profiles profile-scoped and unavailable, preserve fail-fast behavior for genuinely required config surfaces, then align tests and changelog with that contract.

## Reproduction Assessment

Yes. Current main can reproduce the issue by loading an auth store with a `keyRef` or `tokenRef` whose SecretRef provider is absent and calling startup secret activation with default auth-store ref inclusion; the current code resolves that ref and throws before activation completes.

## Solution Assessment

No, not as-is. Splitting config and auth-store assignments is a plausible direction, but the PR must reconcile the existing missing-env auth-store test, document the user-facing fix in the changelog, and get maintainer agreement on the broader degrade-all-auth-profiles behavior.
