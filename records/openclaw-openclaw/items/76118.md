## Summary

Keep open. The PR addresses a real current-main failure mode, but the proposed degradation path can leave a shadowed plaintext auth credential usable after its explicit SecretRef fails, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR splits config SecretRef resolution from auth-store SecretRef resolution, downgrades per-profile auth SecretRef failures to a new warning, and adds regression coverage for stale auth-profile refs.

## Best Possible Solution

Keep the config SecretRef fail-fast behavior, but make auth-profile SecretRef degradation fail closed for the affected credential in every supported shape, then document the user-facing startup reliability fix in the changelog.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: an auth profile with a stale exec `keyRef` is collected into the same batch as config refs, the resolver throws for the missing provider, and startup activation wraps that as a required-secrets startup failure. The review finding is reproducible by adding a stale explicit `keyRef` plus a plaintext `key`/`token` and observing that the PR catch path leaves the plaintext usable.

## Solution Assessment

No. Splitting config and auth-store assignments is the right direction, but the catch path must also clear or disable the credential field owned by the failed auth ref so the profile cannot fall back to shadowed plaintext.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear shadowed credentials when auth refs fail:** `src/secrets/runtime.ts:423-429`
  - body: The new per-auth-store catch leaves the assignment target untouched after a SecretRef failure. Existing collectors only warn that `keyRef`/`tokenRef` override plaintext and rely on successful assignment to replace `profile.key`/`profile.token`; if a profile contains a stale explicit ref plus an old plaintext value, this branch now keeps that plaintext in the activated auth store, so the profile can still be selected instead of becoming unavailable. Clear the credential field, or add a clear-on-failure assignment hook, before downgrading the warning.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `CHANGELOG.md:8`
  - body: This is a user-facing gateway startup reliability fix, but the diff does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix/feat/perf changes to add a single-line entry under Unreleased, so please add one for the auth-profile SecretRef degradation behavior.
  - confidence: 0.9
