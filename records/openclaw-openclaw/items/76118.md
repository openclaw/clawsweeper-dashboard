## Summary

Keep open. Current main still has the reported startup-wide auth-store SecretRef failure mode, and this PR’s patch is a narrow fix: config SecretRefs remain fail-fast while provider-scoped auth-profile failures degrade to a warning; the exact head has green CI and I found no blocking review findings.

## What This Changes

The PR separates config SecretRef resolution from auth-store SecretRef resolution, downgrades missing auth-profile secret providers to a new warning, adds regression tests, and updates the changelog.

## Best Possible Solution

Merge this narrow runtime fix after normal maintainer review and mergeability checks, preserving fail-fast config secrets while keeping stale auth profiles unavailable rather than fatal.

## Reproduction Assessment

Yes. On current main, prepare a secrets runtime snapshot or startup activation with an auth-store api_key.keyRef/tokenRef whose provider is missing from `secrets.providers`; the missing provider throws before activation and startup wraps it as fatal.

## Solution Assessment

Yes. Splitting config assignments from auth-store assignments is the narrowest maintainable fix because it keeps required config secrets fail-fast while matching the existing request-time behavior that skips failed auth-profile candidates.
