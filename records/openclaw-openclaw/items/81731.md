## Summary

Keep open: current main and v2026.6.10 still use the `runAtMs + 1` cron-slot probe, the linked exact-second bug remains open, and this PR is the focused candidate fix with no blocking findings.

## What This Changes

The PR changes cron stale-future slot validation to probe one second after a candidate timestamp and adds a regression test for exact-second future cron slots.

## Best Possible Solution

Land this focused helper and regression test, or an equivalent probe-past-the-candidate-second fix, then close https://github.com/openclaw/openclaw/issues/81691.

## Reproduction Assessment

Yes, source-reproducible. Current main still routes future cron repair through the `runAtMs + 1` previous-run probe, and the pinned Croner 10.0.1 source operates at second precision for previous-run lookup.

## Solution Assessment

Yes. Probing one second past the candidate is the narrowest maintainable fix because it preserves the existing stale-future repair policy and avoids broader schedule normalization or caller-specific exceptions.
