## Summary

Keep open: current main and the latest release still use the 1ms previous-run probe, while this focused PR fixes the linked cron bug with a regression test and supplied Croner/recompute proof. The older related PR is closed unmerged, so it is not a safe superseding landing path.

## What This Changes

This PR changes cron stale-future slot validation from a +1ms to +1s previous-run probe and adds a regression test for an exact-second Honolulu cron slot multiple intervals in the future.

## Best Possible Solution

Land this narrow cron helper and regression-test fix, or an equivalent patch, then close https://github.com/openclaw/openclaw/issues/81691 after merge.

## Reproduction Assessment

Yes, from source inspection: current main asks Croner for the previous run at `slot + 1ms`, while Croner 10.0.1 strips milliseconds and computes previous runs at second granularity. That makes the reported exact-second future slot path source-reproducible without adding a new product mode.

## Solution Assessment

Yes. The +1_000 probe is the narrowest maintainable fix because existing staggered next/previous helpers already use one-second cursor steps, and the patch preserves the stale-slot repair policy instead of adding config or a second validation path.
