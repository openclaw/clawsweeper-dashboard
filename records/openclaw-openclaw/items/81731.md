## Summary

Keep open: current main and the latest release still use the stale +1 ms cron previous-run probe, while this PR is a narrow source-backed fix with focused regression coverage and sufficient terminal proof; I found no blocking findings.

## What This Changes

The PR changes cron stale-future schedule-slot validation from a +1 ms previous-run probe to a +1 s probe and adds a Honolulu exact-second regression test.

## Best Possible Solution

Land this PR or an equivalent one-second probe fix with the regression test, then close https://github.com/openclaw/openclaw/issues/81691 after merge.

## Reproduction Assessment

Yes, at source level: current main passes `runAtMs + 1` into Croner previous-run lookup, and Croner decrements previous-run searches by whole seconds after preserving the reference millisecond. I did not run tests in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix: moving only the private slot probe to +1 s matches Croner's second-granular behavior without changing public schedule APIs, config, storage, or broader cron evaluation semantics.
