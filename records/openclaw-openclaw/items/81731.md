## Summary

Keep open: current main and the latest release still use the old 1 ms cron-slot probe, while this open PR is the focused implementation path for the linked scheduler bug and has no blocking review findings.

## What This Changes

The PR changes cron stale-future slot detection to probe one second after a candidate timestamp and adds a regression test for preserving exact-second future cron slots.

## Best Possible Solution

Land this focused cron scheduler fix, or an equivalent narrow probe-past-the-candidate-second fix, then close https://github.com/openclaw/openclaw/issues/81691 after it reaches main.

## Reproduction Assessment

Yes. Current main reaches the stale-future repair helper with `runAtMs + 1`, and the pinned Croner 10.0.1 source shows previous-run enumeration strips milliseconds and works on second boundaries; I did not run tests in this read-only review.

## Solution Assessment

Yes. Moving the probe to `runAtMs + 1_000` is the narrowest maintainable fix because it preserves the existing repair policy while probing past the candidate second; broader schedule-wrapper changes would touch more behavior than needed.
