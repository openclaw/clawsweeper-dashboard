## Summary

Keep open: current main and the latest release still use the old exact-second cron probe, the linked issue remains open, and this PR is the focused candidate fix with sufficient proof and no blocking review findings.

## What This Changes

The PR changes cron stale-future slot validation to probe one second past candidate cron timestamps and adds a regression test for exact-second future cron slots.

## Best Possible Solution

Land this focused cron scheduler fix, or an equivalent narrow probe-past-the-candidate-second change with the regression test, then close https://github.com/openclaw/openclaw/issues/81691.

## Reproduction Assessment

Yes, source-level: current main and the latest release both still probe Croner at `runAtMs + 1`, and the pinned Croner source decrements previous-run searches by a second and clears milliseconds. I did not run tests because this review is read-only and the checkout lacks installed Croner.

## Solution Assessment

Yes. The `+ 1_000` probe is the narrowest maintainable fix because it preserves existing stale-future repair policy and only moves the previous-run lookup past the candidate second; broader schedule-wrapper changes would touch more cron behavior.
