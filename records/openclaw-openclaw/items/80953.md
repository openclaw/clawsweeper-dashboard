## Summary

Keep open because current main and the latest release still have the source-level string-model fallback-loss path, and an open linked PR explicitly targets this issue.

## What This Changes

The issue asks OpenClaw to warn or preserve fallback-bearing model config when a model setting is represented as a string that cannot carry fallback candidates.

## Best Possible Solution

Resolve the linked fix PR or a replacement so model-config write paths preserve fallback-bearing objects and string defaults produce an explicit diagnostic instead of silent fallback loss.

## Reproduction Assessment

Yes, at source level: a string model config returns `[]` from `resolveAgentModelFallbackValues()`, and fallback candidate construction uses that result directly. I did not run a live overload scenario in this read-only review.

## Solution Assessment

Yes for the direction: preserving fallback-bearing object configs at write boundaries and adding targeted diagnostics is the narrow maintainable solution. A warning-only patch should still be checked for the write paths that can clobber existing fallbacks.
