## Summary

Current main and the latest release cover the user-visible fallback-loss path: string-form default model config is documented as primary-only, command paths now preserve fallback-bearing object config when changing the primary, adding fallbacks to a string default converts it to object form, and doctor warns when per-agent model overrides would clobber default fallbacks.

## What This Changes

The issue reports that string-valued agent model config causes fallback model resolution to return an empty list without any warning.

## Best Possible Solution

Keep the documented string-primary semantics and the current object-preserving command/doctor behavior rather than auto-wrapping every string model ref at runtime.

## Reproduction Assessment

Yes, at source level: `resolveAgentModelFallbackValues()` returns `[]` for string model config. The current code and docs make that string-primary behavior explicit, while the command and doctor paths cover the fallback-loss scenario that made it harmful.

## Solution Assessment

Yes. Preserving object fallback config in model mutation paths and warning on per-agent clobbers is narrower and safer than changing runtime string semantics for all callers.
