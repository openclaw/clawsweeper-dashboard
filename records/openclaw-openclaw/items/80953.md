## Summary

Keep open: current main still has the source-level fallback-loss path, and an open linked PR already closes this issue but has not merged.

## What This Changes

The issue requests a warning or preservation guard when string-shaped model config disables configured model fallbacks.

## Best Possible Solution

Land the linked PR or an equivalent narrow patch that preserves fallback-bearing object configs at write boundaries and adds sanitized diagnostics for string primary-only configs.

## Reproduction Assessment

Yes, at source level: current main returns `[]` for string model input, and runtime fallback resolution consumes that helper for default fallbacks. I did not run a live `overloaded_error` failover scenario in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to preserve object model config shape when changing only `primary` and warn when string form removes fallback support. Auto-inheriting previous fallbacks inside the resolver would conflict with the documented primary-only string contract.
