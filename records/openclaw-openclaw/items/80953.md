## What This Changes

The issue reports that string-form `agents.defaults.model` makes `resolveAgentModelFallbackValues()` return `[]`, leaving no configured model fallback when fallback-bearing object config is replaced by a string.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80961 or a replacement that preserves fallback-bearing object config at write boundaries and adds sanitized diagnostics for string-form fallback loss.

## Reproduction Assessment

Yes. At source level, current main returns `[]` for string model input and the runtime fallback path consumes that helper directly; I did not run a live overload scenario in this read-only review.

## Solution Assessment

Yes for the direction. The maintainable path is to preserve object-form fallback config at write boundaries and add targeted, sanitized diagnostics; the linked PR should address its existing review blockers before landing.
