## Summary

Keep open: current main still has the source-level fallback-loss path for string-shaped default model config, and the linked fixing PR is open but unmerged.

## What This Changes

The issue requests warning or preservation behavior when string-shaped model updates would drop configured `agents.defaults.model.fallbacks`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80961 or an equivalent fix that preserves fallback-bearing object configs at write boundaries and emits sanitized diagnostics.

## Reproduction Assessment

Yes, at source level: a string `agents.defaults.model` returns `[]` from the fallback resolver, and the runtime fallback candidate builder consumes that result directly. I did not run a live overload scenario because this review is read-only.

## Solution Assessment

Partly: warning users and preserving fallback-bearing object configs at write boundaries is the narrow maintainable direction. Changing string-form reads to inherit unknown previous fallbacks would be a broader compatibility/product choice and is not needed for the bug fix.
