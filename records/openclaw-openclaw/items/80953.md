## Summary

Keep open: current main still has a source-reproducible path where an allowed agent-facing config mutation can replace an object-shaped model config with a string, after which fallback resolution returns an empty list. The raw string form is documented as primary-only, so the bug is the unsafe writer/model-switch path rather than the resolver alone.

## What This Changes

The issue reports that string-valued agent model configs can drop configured fallback models when a writer replaces `{ primary, fallbacks }` with a bare model string.

## Best Possible Solution

Keep string form as strict primary-only, but make primary-changing config writers preserve or canonicalize existing model objects with `fallbacks` and `timeoutMs`, with regression coverage for gateway config mutation and model switching.

## Reproduction Assessment

Yes by source inspection, but not by a live mutation run: `gateway config.patch` is allowed to change `agents.defaults.model`, and if that change writes a string, the runtime fallback resolver returns an empty list. I did not run a live gateway write because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve the existing object shape in primary-changing writers rather than changing the documented string-form fallback semantics everywhere.

## Repair Work Prompt

Fix the model fallback clobbering reported in https://github.com/openclaw/openclaw/issues/80953. Current main documents bare string model config as primary-only, so do not globally change `resolveAgentModelFallbackValues` to inherit fallback arrays. Instead, update the primary-changing config mutation/model-switch path so an attempted string assignment to an existing object-shaped `agents.defaults.model` or agent model preserves the current object metadata such as `fallbacks` and `timeoutMs` while changing only `primary`. Focus first on the agent-facing gateway `config.patch`/model-switch writer path because `openclaw models set` already preserves fallbacks via `applyDefaultModelPrimaryUpdate`. Add regression coverage showing an existing `{ primary, fallbacks }` config remains object-shaped with fallbacks after the writer changes the primary, and keep explicit `fallbacks: []` as an intentional no-fallback value. Avoid adding a new config option and avoid changing unrelated fallback classification behavior. Add docs only if the behavior or warning text changes.
