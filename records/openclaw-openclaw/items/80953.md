## Summary

Keep open: current main still has the source-level fallback-loss path for string-shaped default model config, and the issue is paired with an open fixing PR that uses closing syntax, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that string-shaped `agents.defaults.model` config drops fallback candidates without a warning when fallback-capable object config is replaced by a string.

## Best Possible Solution

Preserve fallback-bearing object model configs when updating only the primary model, and add sanitized diagnostics for string/default model shapes without changing the documented string-primary semantics.

## Reproduction Assessment

Yes, at source level: current main returns `[]` for string model input, and runtime fallback construction consumes that helper directly. I did not run a live overload scenario because this review is read-only.

## Solution Assessment

Yes for keeping this issue open: the linked open PR is the active implementation path, and the maintainable fix should preserve or warn at config write/diagnostic boundaries rather than making string configs inherit unknown previous fallbacks at read time.
