## Summary

Keep open because current main still has the source-level string-model fallback-loss path, and https://github.com/openclaw/openclaw/pull/80961 is an open, unmerged closing PR for this issue.

## What This Changes

The issue reports that string-form `agents.defaults.model` config causes fallback lists to be dropped because `resolveAgentModelFallbackValues()` returns an empty array for strings.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80961 or a replacement that preserves fallback-bearing object config at write boundaries and adds sanitized diagnostics for string-form fallback loss.

## Reproduction Assessment

Yes. Source inspection on current main shows string-form model config resolves to no fallbacks, and runtime fallback candidate construction consumes that helper directly; I did not run a live provider overload scenario.

## Solution Assessment

Yes for the direction. Preserving object-form fallback config when changing primaries and adding targeted, sanitized diagnostics is the narrow maintainable path without changing the documented string-primary/object-fallback semantics.
