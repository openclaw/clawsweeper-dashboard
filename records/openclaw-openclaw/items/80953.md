## Summary

Keep open: current main still has the source-level fallback loss for string-form default model config, the latest shipped release has the same behavior, and the linked implementation PR remains open with unresolved review blockers.

## What This Changes

The issue asks OpenClaw to avoid silently losing configured model fallbacks when a default model config is represented or rewritten as a string.

## Best Possible Solution

Land the linked PR or a replacement that preserves fallback-bearing object configs at write boundaries and adds sanitized diagnostics for string-form default model configs without changing documented string-primary semantics.

Yes, source inspection gives a high-confidence path: current main returns `[]` for string-form model input and the runtime fallback path consumes that helper for default model fallbacks. I did not run a live overloaded-model failure scenario.
