## Summary

Keep open because the checked-out main branch and latest release still have the source-level string-model fallback-loss path, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to stop silently losing model fallbacks when an agent model config is written as a string instead of the fallback-bearing object form.

## Best Possible Solution

End with model-write paths preserving fallback-bearing objects and with an explicit doctor/runtime diagnostic when string-form defaults cannot carry fallback chains.

## Reproduction Assessment

Yes, at source level: string-form model config returns [] from resolveAgentModelFallbackValues(), and runtime fallback construction consumes that result directly. I did not run a live overloaded-provider request in this read-only review.

## Solution Assessment

Yes for the direction: preserving object-form model configs at write boundaries and adding targeted diagnostics is the narrow maintainable fix. Broad auto-wrapping of string config would need more compatibility care than the current issue requires.
