## Summary

Keep open: current main and the latest release still contain the encrypted-content retry path, 429-to-rate_limit classification, and auth-profile cooldown path described by the report, while the broader replay-recovery PRs that may address the retry storm are still open and need maintainer coordination.

## What This Changes

The issue reports that the built-in openai-codex Responses transport repeatedly retries encrypted reasoning failures, causing long turn delays and false auth-profile rate-limit failover.

## Best Possible Solution

Coordinate and land the Responses replay recovery so invalid encrypted reasoning is not retried on every model call, then add live Codex OAuth multi-tool-turn proof that retry count, latency, and auth-profile cooldown behavior are fixed.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter supplied detailed logs, and current source plus v2026.5.22 source show the retry, 429 classification, and auth-profile cooldown paths needed for a source-level reproduction.

## Solution Assessment

Unclear for current main: the existing one-retry-per-request guard has not stopped the reported multi-call retry storm. The safer solution is the coordinated Responses replay-recovery path plus live proof before changing auth-profile rate-limit semantics.
