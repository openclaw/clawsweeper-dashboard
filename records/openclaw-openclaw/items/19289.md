## Label Justifications

- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current main and the latest release still expose a closed browser profile driver set and no plugin SDK browser-backend registration contract, while the discussion and related reports show this is a real core/plugin API design gap rather than implemented, duplicate, or ClawHub-only work.

## What This Changes

Request an additive browser backend/plugin interface so third-party MCP or custom adapters can register profile drivers without forking OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades all use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
