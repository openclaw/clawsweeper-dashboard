## Summary

Keep open. Current main still exposes a closed browser driver set with no Plugin SDK or runtime registration seam for third-party browser backends, and the existing maintainer comments plus related #53599/#55840/#52190 context point to a real core/plugin API design gap rather than a stale, duplicate, or ClawHub-only request.

## What This Changes

Requested a pluggable browser backend and profile-driver registration interface so third-party MCP/custom adapters can integrate without core forks.

## Best Possible Solution

Add an additive, versioned browser backend extension point in core and the Plugin SDK, with typed adapter registration, registered driver IDs through profile config/control/CLI surfaces, runtime dispatch through the registry, docs, and contract tests for MCP/custom adapters.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a browser profile with a non-built-in driver such as "extension"; current schema/control routes reject it, and the contract test asserts the 400 response.

## Solution Assessment

Unclear for the exact API shape until maintainer design happens, but the requested direction is valid. Replacing the entire browser plugin is not the narrowest solution because it cannot register a backend inside the existing profile/config/runtime path.
