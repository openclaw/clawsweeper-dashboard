## Summary

Keep open: current main and the latest release still expose a closed browser driver set with no Plugin SDK/browser backend registry for third-party MCP or custom adapters, and the related Browserbase, My Browser, and relay items are narrower demand signals rather than a generic interface implementation.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom drivers can register browser profile drivers without core patches.

## Best Possible Solution

Design a maintainer-approved, versioned browser backend/adapter extension point that registers driver ids through core and the Plugin SDK, updates config/schema/CLI/control routes, and adds docs plus security/contract tests for MCP/custom adapters.

## Reproduction Assessment

Yes, at source level: current main keeps browser driver config, schema, route validation, runtime resolution, CLI/client types, and Plugin SDK browser types closed to third-party driver ids. No live runtime setup is needed to prove the missing registration interface because current public contracts reject arbitrary drivers before dispatch.

## Solution Assessment

Yes for keeping this as a maintainer API-design issue; one-off built-in drivers are too narrow for the requested third-party backend seam. The best implementation needs a versioned registry/contract plus security review rather than an autonomous enum widening patch.
