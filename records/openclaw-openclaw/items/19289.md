## Summary

Keep open. Current main still has a closed browser driver set and no Plugin SDK/browser backend registration seam, while the issue discussion and related #52190/#53599/#55840 context show this is a real core/plugin API design gap rather than a stale, duplicate, or ClawHub-only request.

## What This Changes

Request an additive browser backend plugin interface so third-party MCP/custom drivers can be registered without modifying core browser code.

## Best Possible Solution

Add an additive, versioned browser backend extension point in core and the Plugin SDK, with typed adapter registration, registered driver IDs in profile config/control paths, runtime dispatch through the registry, docs, and contract tests for MCP/custom adapters.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a profile with a non-built-in driver such as "extension"; current schema/control routes reject it, and the contract test asserts that 400 response.

## Solution Assessment

Unclear for the exact API shape until maintainer design happens, but the direction is right. Whole-tool replacement through the browser plugin docs does not satisfy this request because it cannot register a backend inside the existing browser profile/runtime path.
