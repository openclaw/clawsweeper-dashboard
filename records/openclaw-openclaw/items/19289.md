## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current main and the latest release still expose a closed browser profile driver set and no Plugin SDK browser-backend registration contract, while the linked browser-relay and Patchright discussions show related demand but do not supersede this core/plugin API gap.

## What This Changes

Requests a public browser backend or driver registration interface so plugins, MCP engines, or custom adapters can register profile drivers without modifying OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades are still built-in-driver-only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract with a threat model is the maintainable path. Loosening the driver whitelist alone would not define runtime dispatch ownership, compatibility rules, or browser-control trust boundaries.
