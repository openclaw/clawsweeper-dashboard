## Summary

Keep open. Current main still has a closed browser profile driver set and no Plugin SDK or registry seam for third-party browser backends; the issue discussion and related relay reports support this as a core/plugin API design gap, not conservative cleanup closure.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom drivers can register without modifying OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire profile config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, client, CLI, runtime, SDK API, and tests remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. Full browser-plugin replacement can cover whole-tool alternatives, but it does not provide the requested stable backend registration seam for the existing browser profile surface.
