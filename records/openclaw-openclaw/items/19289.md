## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current main and v2026.5.20 still expose a closed browser driver set and only browser config facades in the Plugin SDK, so this remains a real core/plugin API design gap rather than implemented-on-main, duplicate, or ClawHub-only work.

## What This Changes

Request a browser backend registration interface so third-party MCP/custom browser drivers can bind without patching OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main and v2026.5.20 gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, CLI forwarding, and SDK facades use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
