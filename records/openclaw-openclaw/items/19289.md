## Summary

Keep open. Current main still limits browser profiles to built-in drivers, rejects arbitrary driver names, and exposes browser config facades but no browser backend registration API, so this remains a real core/Plugin SDK design gap rather than a cleanup close.

## What This Changes

Request a pluggable browser backend/driver registration API so third-party MCP or custom adapters can bind to browser profiles without forking core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route, CLI, runtime, SDK facade, and tests all remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. Full replacement of the bundled browser plugin is possible today, but it is not the requested stable backend registration seam for the existing browser profile surface.
