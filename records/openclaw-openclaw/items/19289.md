## Summary

Keep open: current main and v2026.5.22 still expose a closed browser profile driver set and no Plugin SDK browser-backend registration contract. The related discussion points to this as the canonical core/plugin API design gap, not something already implemented or fully handled by ClawHub.

## What This Changes

Request a pluggable browser backend interface so third-party MCP/custom browser drivers can register without OpenClaw core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades all use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.

- Closing this as ClawHub-only would leave plugin authors without the core/SDK registration seam needed to implement adapters cleanly inside the existing browser profile flow.
