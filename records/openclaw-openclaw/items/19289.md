## Summary

Keep this issue open. Current main still has a closed browser driver set across config, schema, client, route, runtime, and tests, while the Plugin SDK exposes browser facades but no browser-backend registration contract. The related Patchright and Chrome Extension Relay issues are useful demand signals, not replacements for the missing core/plugin API seam.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP or custom browser drivers can be registered without modifying core.

## Best Possible Solution

Design and implement an additive, versioned browser backend extension point in core and the Plugin SDK: define the typed backend contract, let plugins register driver ids, accept registered ids through config/schema/client/CLI/control routes, route runtime operations through the registry, and document/test the contract for MCP/custom adapters.

## Risks / Open Questions

- This is a cross-cutting public SDK, config, client, CLI, docs, and runtime contract; a narrow automated patch could freeze an unstable third-party API shape.
- Browser backend adapters would mediate browser automation, process launch or MCP attachment, SSRF policy, and auth boundaries, so the design needs explicit security review.
- Related #55840 may still need separate product handling because a generic backend interface would not automatically restore every removed Chrome Extension Relay capability.
