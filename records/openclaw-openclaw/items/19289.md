## Summary

Keep open. Current main still uses a closed browser profile driver set, rejects unknown driver ids, and exposes only browser config facades through the Plugin SDK; the related relay and Patchright reports are demand signals rather than a canonical replacement for this generic core/plugin API gap.

## What This Changes

Add a public browser backend registration interface so third-party browser drivers such as MCP/custom adapters can be selected without core changes.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route handling, runtime resolution, SDK API, and tests remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. The maintainable solution is an additive, versioned SDK/core seam, not merely loosening a whitelist or requiring whole-browser-plugin replacement.
