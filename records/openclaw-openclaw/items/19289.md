## Summary

Keep open. Current main still has no browser backend or driver registration seam: profile config, schema, CLI, client/runtime resolution, and route handling remain limited to built-in drivers, while the Plugin SDK exposes only whole-plugin replacement/facade helpers rather than a stable third-party browser backend contract.

## What This Changes

Request a public browser backend plugin interface so third-party MCP/custom adapters can register driver ids and bind into the bundled browser profile/tool surface.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire profile config/schema/help/client/CLI/control routes/runtime dispatch and docs around that contract with focused contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof of the current limitation because the relevant config, schema, route, CLI, client/runtime types, and tests accept only built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. Replacing the entire bundled browser plugin is possible today, but it is not the requested stable driver/backend registration seam for the bundled browser profile surface.
