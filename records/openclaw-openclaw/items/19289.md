## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current `main` and latest release still expose a closed browser profile driver set and whole-plugin replacement docs, not the requested in-profile third-party browser backend registration seam. The related relay and Patchright threads are supporting demand signals, but neither supersedes this broader Plugin SDK/core API design gap.

## What This Changes

Request a pluggable browser backend interface so third-party drivers, including MCP/custom adapters, can register without forking or patching OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades all use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
