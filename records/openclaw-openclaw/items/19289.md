## Summary

Keep open: current main and v2026.6.8 still expose only built-in browser profile drivers and no public browser-backend registry for third-party MCP/custom adapters. This remains a real core/plugin API design gap with security-sensitive browser-control implications, not a duplicate or ClawHub-only item.

## What This Changes

The issue requests a pluggable browser backend interface so external MCP/custom browser drivers can register without modifying OpenClaw core.

## Best Possible Solution

Define a maintainer-approved, additive browser backend Plugin SDK contract with explicit driver registration, runtime dispatch, security/capability rules, compatibility behavior, docs, and contract tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 keep browser driver config, schema, route validation, runtime resolution, and Plugin SDK surfaces closed to third-party driver ids. No live runtime setup is needed to prove the missing registration interface because public contracts reject arbitrary drivers before dispatch.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is plausible, but the maintainable solution should be an additive, security-reviewed Plugin SDK/core registry rather than ad hoc acceptance of new driver strings.
