## Summary

Keep open: current main still has no browser backend/driver registration seam for third-party MCP or custom adapters, and the remaining work is a cross-cutting core, browser plugin, Plugin SDK, config, and security design issue rather than a stale or ClawHub-only request.

## What This Changes

Request a public browser backend registration interface so third-party MCP/custom adapters can provide browser driver ids without forking core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, Plugin SDK docs, and tests still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. The unsafe shortcut would be to loosen the driver whitelist without defining the browser-driver trust boundary, runtime dispatch contract, and compatibility rules.
