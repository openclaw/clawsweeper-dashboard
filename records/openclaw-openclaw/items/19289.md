## Summary

Keep open: current main and v2026.6.5 still expose a closed browser-driver set with no Plugin SDK registration contract for third-party backends. Existing whole-plugin replacement is useful for standalone alternatives, but it does not satisfy the requested profile-level adapter seam; the related Patchright and extension-relay reports are demand signals rather than canonical replacements.

## What This Changes

Add a stable browser backend registration interface so third-party MCP or custom drivers can integrate without core forks.

## Best Possible Solution

Design an additive, typed, versioned browser-backend registry in core and the Plugin SDK, keep concrete third-party drivers externally distributed, and require capability/security, fresh-config, upgrade, and contract tests before adoption.

## Reproduction Assessment

Yes at source level: current main explicitly limits browser drivers in config types, schema, resolved profiles, and profile creation, so a plugin-defined backend cannot bind through a supported registry. This verifies a feature gap rather than reproducing broken established behavior.

## Solution Assessment

Yes in direction, but the proposed details need design: a narrow Plugin SDK registry is more maintainable than adding Patchright, relay, or MCP engines individually to core. Maintainers must still define capabilities, lifecycle, failure isolation, config semantics, and compatibility guarantees.
