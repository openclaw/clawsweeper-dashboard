## Summary

Keep open. Current main still has a closed browser-driver/profile contract and no Plugin SDK registration seam for third-party browser backends, while the discussion and related relay/Patchright issues show this is a real public API/design gap rather than cleanup-close material.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom adapters can register browser drivers without core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then broaden profile config/schema/help/client/CLI/control routes/runtime dispatch and docs around that contract with focused contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route, CLI, client/runtime, and SDK surfaces accept only built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. Replacing the whole bundled browser plugin is documented today, but it is not the requested stable driver/backend registration seam for the bundled browser profile surface.
