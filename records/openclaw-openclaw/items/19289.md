## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the core default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, and remote browser access boundaries.

## Summary

Keep open: current main and v2026.5.12 still use a closed browser driver set, and the Plugin SDK exposes browser config facades but no registration seam for third-party MCP/custom browser backends. The related relay and Patchright reports are useful demand signals, but they do not replace the public API and security design this issue is asking for.

## What This Changes

Request to add a pluggable browser backend/driver interface so third-party MCP or custom adapters can register without core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, Plugin SDK docs, tests, and v2026.5.12 still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. The unsafe shortcut would be to loosen the driver whitelist without defining the browser-driver trust boundary, runtime dispatch contract, and compatibility rules.
