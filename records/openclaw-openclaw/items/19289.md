## Summary

Keep open: current main and v2026.6.10 still expose a closed browser driver set with no Plugin SDK/core registry for third-party MCP or custom browser backends, and the open Browserbase/My Browser work is narrower than the requested generic adapter interface.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom drivers can register browser profile driver ids without forking core.

## Best Possible Solution

Design a maintainer-approved, versioned browser backend/adapter extension point that registers driver ids through core and the Plugin SDK, updates config/schema/CLI/control routes, and adds docs plus security and contract tests for MCP/custom adapters.

## Reproduction Assessment

Yes, at source level: current main keeps browser driver config, schema, route validation, runtime resolution, CLI/client types, and Plugin SDK browser types closed to third-party driver ids. No live runtime setup is needed because arbitrary drivers are rejected before backend dispatch.

## Solution Assessment

Yes for keeping this as a maintainer API-design issue; one-off built-in drivers are too narrow for the requested third-party backend seam. The implementation should be a versioned registry/contract with security review rather than an autonomous enum widening patch.
