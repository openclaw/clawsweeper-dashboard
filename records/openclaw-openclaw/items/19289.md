## Summary

Keep open. Current main still lacks a browser backend/driver registration API: the config, schema, CLI/client, runtime resolution, route, SDK surface, and tests remain limited to built-in browser drivers, while the related discussion treats this as a real Plugin SDK/API gap rather than a ClawHub-only plugin request.

## What This Changes

Request a stable browser backend/driver registration interface so third-party MCP or custom adapters can be selected by browser profiles without modifying core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then broaden profile config/schema/help/client/CLI/control routes/runtime dispatch and docs around that contract with focused contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because current main accepts only built-in browser drivers and exposes no backend registration hook.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. A ClawHub plugin can replace the bundled browser plugin wholesale today, but that is not the requested stable driver/backend registration seam inside the bundled browser profile surface.
