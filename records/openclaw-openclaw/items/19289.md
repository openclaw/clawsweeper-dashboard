## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current main and the latest release still restrict browser profiles to built-in driver ids and expose no Plugin SDK registry for external browser backends; the related relay and Patchright threads are useful demand signals, not replacements for this core/plugin API design work.

## What This Changes

Request an additive browser backend registration interface so plugins, MCP engines, or custom adapters can supply third-party browser driver ids without forking core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a browser-control security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, SDK facades, tests, and v2026.5.19 still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
