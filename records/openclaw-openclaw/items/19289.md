## Label Justifications

- `P2`: This is a meaningful but non-urgent extension gap affecting advanced browser integrations and third-party plugin developers.
- `impact:security`: A registered third-party backend could access authenticated browser sessions and sensitive page content, making the trust boundary central to the requested design.

## Summary

Keep this issue open: both current main and v2026.6.6 retain a closed browser-driver set, and the public Plugin SDK has configuration helpers but no profile-level backend registration contract. Related browser reports demonstrate demand but do not replace this general API-design issue.

## What This Changes

Add a stable browser backend registration interface so plugins can provide third-party MCP or custom drivers without core forks.

## Best Possible Solution

Define an additive, versioned browser-backend descriptor and runtime interface at the browser plugin and Plugin SDK boundary, with explicit capabilities, security policy, lifecycle ownership, validation, compatibility behavior, documentation, and contract tests.

## Reproduction Assessment

Yes, at source level: profile typing, route validation, runtime resolution, and plugin registration show no route for an arbitrary registered backend identifier. This reproduces a missing capability rather than a failure of an existing documented contract.

## Solution Assessment

The extension-point direction is appropriate, but merely widening the driver string is not the best complete solution. Maintainers should first define a narrow, versioned capability, lifecycle, compatibility, and security contract rather than exposing browser internals or adding another hard-coded backend.
