## Summary

Keep open: current main and the latest release still use a closed browser driver set and expose no Plugin SDK registration seam for third-party MCP/custom browser backends. The related relay reports are useful demand signals, but they do not replace the generic API and security design this issue is asking for.

## What This Changes

Request a pluggable browser backend interface so plugins, MCP engines, or custom adapters can register browser driver ids and bind runtime implementations without modifying core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, Plugin SDK docs, tests, and v2026.5.12 still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. The unsafe shortcut would be to loosen the driver whitelist without defining the browser-driver trust boundary, runtime dispatch contract, and compatibility rules.
