## Summary

Keep open: current main and v2026.6.5 still expose a closed browser-driver set and browser config facades, not a Plugin SDK backend registration contract. Whole-plugin replacement exists, but it does not satisfy the requested profile-level adapter seam for MCP/custom drivers.

## What This Changes

Request to add a stable browser backend/driver registration interface so plugins or MCP/custom adapters can supply profile drivers without patching core.

## Best Possible Solution

Design an additive, typed, versioned browser-backend registry in core and the Plugin SDK, keep concrete third-party drivers external, and require capability/security, fresh-config, upgrade, docs, and contract tests before adoption.

## Reproduction Assessment

Yes at source level: current main limits browser drivers in config types, schema, resolved profiles, CLI/control routes, and tests, so a plugin-defined backend cannot bind through a supported registry. This verifies a feature gap rather than broken established behavior.

## Solution Assessment

Yes in direction, but the exact contract needs maintainer design. A narrow Plugin SDK browser-backend registry is more maintainable than adding Patchright, relay, or MCP engines individually to core.
