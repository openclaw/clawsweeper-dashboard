## Summary

Keep open: current main still lacks a third-party browser backend registration contract, so this remains a real core/Plugin SDK design gap rather than a duplicate or ClawHub-only request.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom browser drivers can register without core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades all use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define trust boundaries, runtime dispatch, or compatibility rules.
