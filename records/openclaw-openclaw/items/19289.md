## Summary

Keep open. Current main still has a closed browser profile driver set and no Plugin SDK/core registration seam for third-party browser backends; the related relay and Patchright threads are useful demand signals but do not replace this generic API-design item.

## What This Changes

Request to add a pluggable browser backend/driver registration interface so third-party MCP or custom browser adapters can bind without forking core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route handling, runtime resolution, SDK API, and tests remain limited to built-in browser drivers.

## Solution Assessment

Yes at the design direction level: keeping this open for a versioned SDK/core seam is the maintainable path. A narrow whitelist loosening or forcing whole-browser-plugin replacement would not solve the requested third-party backend adapter contract cleanly.
