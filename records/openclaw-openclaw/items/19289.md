## Summary

Keep open. Current main still exposes a closed browser profile driver set and no SDK or registry seam for third-party browser backends; the related discussion supports this as a core/Plugin SDK design gap rather than a ClawHub-only replacement-plugin request.

## What This Changes

Request a pluggable browser backend interface so third-party MCP/custom adapters can register browser profile drivers without forking OpenClaw.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire profile config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route, CLI, runtime, SDK facade, and tests all remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. Full browser-plugin replacement can cover some alternatives, but it does not provide the requested stable backend registration seam for the existing browser profile surface.
