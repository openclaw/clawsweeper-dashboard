## Summary

Keep open: current main still lacks a browser backend/driver registration seam for third-party MCP or custom adapters, and the existing comments plus related relay reports support treating this as a core Plugin SDK/API and security design gap rather than stale cleanup or a ClawHub-only plugin request.

## What This Changes

Request to add a pluggable browser backend/driver registration interface so third-party MCP or custom adapters can be selected without modifying OpenClaw core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, Plugin SDK docs, and tests still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. The unsafe shortcut would be to loosen the driver whitelist without defining the browser-driver trust boundary, runtime dispatch contract, and compatibility rules.
