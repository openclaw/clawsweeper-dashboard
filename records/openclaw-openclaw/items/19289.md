## Label Justifications

- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, process launch, remote browser access, and navigation boundaries.

## Summary

Keep open: current main and v2026.5.20 still expose only built-in browser driver ids and no Plugin SDK/browser registry for third-party MCP or custom adapters, so this remains a real core/plugin API design gap rather than a ClawHub-only request or duplicate.

## What This Changes

The issue requests a pluggable browser backend interface so plugins can register third-party driver ids and bind MCP/custom browser adapters without forking core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, SDK facades, tests, and v2026.5.20 still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
