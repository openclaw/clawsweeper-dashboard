## Label Justifications

- `P2`: This is a real browser/plugin API gap for advanced automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect real browser profiles, signed-in sessions, process launch, remote browser access, navigation, downloads, and SSRF boundaries.

## Summary

Keep open: current main and the latest release still expose a closed browser profile driver set, route runtime dispatch only through built-in drivers, and provide Plugin SDK browser facades without a backend registration contract. The related relay and Patchright threads are demand signals, but neither supersedes this broader core/Plugin SDK design gap.

## What This Changes

Request to add a stable browser backend registration interface so third-party MCP/custom browser drivers can be selected without forking or patching core.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, tests, and SDK facades all use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
