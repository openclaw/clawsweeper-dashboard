## Summary

Keep this open: current main still has a closed browser driver set and no Plugin SDK/browser backend registration seam, while the maintainer discussion explicitly scopes this as a core/plugin API design gap rather than a ClawHub-only plugin request.

## What This Changes

Request a pluggable browser backend interface so third-party MCP/custom adapters can register browser drivers without OpenClaw core forks.

## Best Possible Solution

Add an additive, versioned browser backend/adapter extension point that lets plugins register driver ids, routes profile selection through that registry, documents the contract, and covers MCP/custom adapters with contract tests.

## Reproduction Assessment

Yes for the current limitation: source and contract tests show unknown browser driver ids are rejected and no browser-backend registration API exists. This is a feature/API gap rather than a broken existing contract.

## Solution Assessment

Yes, the requested direction is the right class of solution, but it needs maintainer design first. A standalone ClawHub plugin can replace the whole browser tool, but it cannot currently register a backend into the built-in profile driver/runtime flow.
