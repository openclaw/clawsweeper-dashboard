## Summary

Keep open: current main still lacks a browser backend registration seam, and the request remains a cross-cutting core/Plugin SDK design gap with security implications rather than a ClawHub-only plugin request.

## What This Changes

The issue requests a browser backend plugin interface so third-party MCP/custom browser drivers can register without OpenClaw core patches.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, Plugin SDK docs, and tests are still built around built-in driver ids.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. The unsafe shortcut would be to loosen the driver whitelist without defining the browser-driver trust boundary and runtime contract.

## Risks / Open Questions

- A browser backend registry would become a public SDK/config/runtime contract, so loosening driver validation without a versioned design could create API drift.

- The relay-specific regression remains tracked separately at https://github.com/openclaw/openclaw/issues/53599 and should not be collapsed into this generic API request without maintainer direction.
