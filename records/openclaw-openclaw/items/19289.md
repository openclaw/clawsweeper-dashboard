## Summary

Keep open: current main and v2026.6.6 still keep browser profile drivers closed and the public Plugin SDK lacks a browser-backend registration contract. Related browser issues are narrower demand signals, not replacements for this security-sensitive SDK design work.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP or custom browser drivers can register without forking core.

## Best Possible Solution

Define an additive, versioned browser backend Plugin SDK contract with explicit capability, lifecycle, security, compatibility, docs, and contract-test coverage.

## Reproduction Assessment

Yes, at source level: current main still has closed config, schema, CLI, route, runtime, docs, and SDK surfaces for browser profile drivers. No live runtime test is needed to prove the missing registration interface because the public contracts and route validation reject arbitrary driver IDs.

## Solution Assessment

Yes, the extension-point direction is appropriate, but it is not a small string/config change. The maintainable path is a maintainer-approved typed backend/adapter contract with security, lifecycle, and compatibility rules.
