## Summary

Keep open. Current main still has a closed browser driver set across config, schema, CLI/control routes, runtime resolution, and tests, while the Plugin SDK exposes browser helper facades but no browser-backend registration capability. The existing maintainer comment and related #52190/#53599/#55840 context point to a real core/plugin API design gap rather than a stale, duplicate, or ClawHub-only request.

## What This Changes

Request a pluggable browser backend interface so plugins, MCP engines, or custom adapters can register browser profile driver IDs without forking core.

## Best Possible Solution

Add an additive, versioned browser backend extension point in core and the Plugin SDK: define a typed adapter contract, let plugins register driver IDs, allow config/schema/help/client/CLI/control routes to accept registered IDs, route runtime dispatch through that registry, document the browser backend contract, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale.

## Reproduction Assessment

Yes. A high-confidence verification path is to create or configure a browser profile with a non-built-in driver such as "extension" or a custom ID; current main rejects it in schema/control routes, and the existing contract test asserts that rejection.

## Solution Assessment

Unclear for the exact API shape until maintainer design happens, but the direction is right: this belongs as a core/Plugin SDK seam because current plugin and ClawHub surfaces can replace the whole browser tool but cannot register a backend driver inside the existing browser profile/runtime path.
