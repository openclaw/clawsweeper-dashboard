## Label Justifications

- `P2`: This is a real feature/API gap for advanced browser automation, but it is not an urgent regression in the default browser workflow.
- `impact:security`: Browser backend registration would affect control of real browser profiles, signed-in sessions, remote browser access, navigation boundaries, and process launch behavior.

## Summary

Keep open: current main and v2026.5.20 still expose a closed browser driver set and only browser config/profile facades in the Plugin SDK, so this remains a real public API, product, and security design gap rather than an implemented or ClawHub-only request.

## What This Changes

Request an additive browser backend registration interface so third-party MCP or custom adapters can provide profile drivers without core modifications.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config, schema, CLI/help, control routes, runtime dispatch, docs, and contract tests around registered driver ids with an explicit browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main and v2026.5.20 gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, CLI forwarding, and SDK facades use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the current driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
