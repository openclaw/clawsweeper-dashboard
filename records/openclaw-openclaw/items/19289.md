## Summary

Keep open: current main and v2026.5.20 still have a closed browser driver set and no Plugin SDK registration seam for third-party browser backends, so this remains a real core/plugin API and security design gap rather than a ClawHub-only request or duplicate.

## What This Changes

Request to add a pluggable browser backend registration interface so third-party MCP or custom adapters can provide browser drivers without core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/help/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with a browser-control threat model.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main gives high-confidence proof of the current limitation because config, schema, route validation, runtime profile resolution, CLI forwarding, SDK facades, and v2026.5.20 still use built-in driver ids only.

## Solution Assessment

Yes at the direction level: a versioned SDK/core registration contract is the maintainable path. Loosening the driver whitelist alone would be unsafe because it would not define the trust boundary, runtime dispatch contract, or compatibility rules.
