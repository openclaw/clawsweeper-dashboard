## Summary

Keep open. Current main still has a closed browser profile driver set, rejects unknown driver ids, and has no Plugin SDK browser-backend registration seam; the related relay/Patchright issues are useful demand signals but do not replace this generic API design item.

## What This Changes

Request a pluggable browser backend interface so third-party MCP or custom drivers can register without core modifications.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, route handling, runtime resolution, SDK API, and tests remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. The maintainable solution is a versioned SDK/core seam, not merely loosening the driver whitelist or requiring whole-browser-plugin replacement.
