## Summary

Keep open: current main still treats browser profile drivers as a closed built-in set and exposes only browser config facades, so this remains a real core/Plugin SDK API design gap rather than a duplicate or ClawHub-only request.

## What This Changes

Request a pluggable browser backend interface so third-party MCP/custom adapters can register driver ids and bind to browser tools without core forks.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids with security review.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection on current main gives high-confidence proof of the limitation because config, schema, route validation, runtime profile resolution, Plugin SDK API, and tests are still built around built-in driver ids.

## Solution Assessment

Yes, keeping this open for a versioned SDK/core contract is the best current path. Merely accepting arbitrary driver strings or restoring one relay driver would not solve generic third-party backend registration safely.
