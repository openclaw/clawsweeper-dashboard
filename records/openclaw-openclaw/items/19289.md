## Summary

Keep open. Current main still has a closed browser driver set and no Plugin SDK/core registration seam for third-party browser backends, and the maintainer discussion already scoped this as API design work rather than ClawHub-only cleanup.

## What This Changes

Request a Plugin SDK/core browser backend registration interface so third-party MCP/custom adapters can be selected as browser profile drivers.

## Best Possible Solution

Design an additive, versioned browser backend contract in core and the Plugin SDK, then wire profile config/schema/CLI/control/runtime dispatch through registered driver ids with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an enhancement/API gap. Source inspection gives high confidence that current main rejects or normalizes arbitrary browser driver ids rather than dispatching through a backend registry.

## Solution Assessment

Yes, keeping this open for maintainer API design is the best path. A ClawHub plugin alone cannot satisfy the request until OpenClaw exposes a supported browser backend registration seam.
