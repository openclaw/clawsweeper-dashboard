## Summary

Keep open. Current main still has a closed browser profile driver set and only browser config facades in the Plugin SDK, while the live discussion and related relay issues support this as an unresolved core/plugin API design gap rather than cleanup closure.

## What This Changes

Request a core and Plugin SDK browser backend registration interface so third-party MCP/custom drivers can be selected from browser profiles without core patches.

## Best Possible Solution

Design an additive, versioned browser backend registration contract in core and the Plugin SDK, then wire profile config/schema/CLI/control routes/runtime dispatch/docs/tests around registered driver ids.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high-confidence proof of the current limitation because config, schema, CLI/control routes, runtime, SDK API, and tests remain limited to built-in browser drivers.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. Full browser-plugin replacement can cover whole-tool alternatives, but it does not provide the requested stable per-profile backend registration seam.
