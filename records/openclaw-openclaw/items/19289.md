## Summary

Keep open: current main and v2026.6.8 still expose only the built-in browser drivers and no public browser-backend registration surface. This remains a core/Plugin SDK design gap with security-sensitive browser-control implications, not a duplicate or ClawHub-only item.

## What This Changes

Request a pluggable browser backend interface so third-party MCP/custom browser drivers can register without core modifications.

## Best Possible Solution

Define an additive, versioned browser backend Plugin SDK contract with explicit driver registration, runtime dispatch, security rules, compatibility behavior, docs, and contract tests.

## Reproduction Assessment

Yes, at source level: current main rejects arbitrary browser driver ids in config/schema and the profile creation route before any runtime dispatch can occur. No live runtime setup is needed to prove the missing registration seam.

## Solution Assessment

Yes on direction, but not as a narrow automated patch. The maintainable solution is a maintainer-approved typed backend/adapter contract with security, lifecycle, compatibility, docs, and contract-test coverage rather than accepting new driver strings.
