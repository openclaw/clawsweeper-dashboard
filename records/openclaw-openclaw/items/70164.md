## Summary

Keep open: current `main` still has a source-visible path where Gateway emits `state:final` without a message when no accepted assistant text reaches the chat buffer, and the reporter's follow-up Gateway logs show that same no-assistant-event shape. Current `main` has related MiniMax thinking and data-only SSE fixes, but I did not find proof that they cover this MiniMax/WebSocket tool-result case or that another canonical item owns the remaining work.

## What This Changes

The issue asks OpenClaw Gateway WebSocket chat streams to include assistant/chat delta text instead of jumping from tool/lifecycle events directly to `state:final` without a message.

## Best Possible Solution

Trace one current MiniMax tool-result run from raw provider stream through embedded assistant event emission and Gateway WebSocket chat projection, then fix the layer that drops visible assistant text with focused regression coverage.

## Reproduction Assessment

No, not as a high-confidence current-main live reproduction. Source inspection shows the final-without-message path when no accepted assistant event reaches the buffer, and the reporter supplied credible Gateway logs, but the MiniMax failure path was not reproduced against current `main`.

## Solution Assessment

Unclear. The safest fix is to identify whether MiniMax transport parsing, embedded message filtering, or Gateway projection drops the visible text; synthesizing fallback text in Gateway finalization would risk masking the real layer.
