## Summary

Keep open: the PR targets a real source-reproducible diagnostics bug, but the current patch suppresses the sampler itself during startup grace and still lacks auditable after-fix real behavior proof.

## What This Changes

Adds a 60-second diagnostic heartbeat startup grace option, wires it from Gateway startup, and adds a unit test for suppressing liveness warnings during startup.

## Best Possible Solution

Gate liveness event and WARN emission during startup grace while still sampling every tick, update the regression test to assert sampling during grace, and add redacted after-fix runtime output.

## Reproduction Assessment

Yes. Source inspection and existing fake-timer tests show current main emits a liveness warning when queued diagnostic work exists and a liveness sample is returned; Gateway starts that heartbeat without a grace window.

## Solution Assessment

No. The intended internal grace window is narrow, but this patch should call the sampler during grace and suppress only event/warn emission so cold-start baselines are reset before the first post-grace tick.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep sampling during the grace window:** `src/logging/diagnostic.ts:958-961`
  - body: This skips `sampleDiagnosticLiveness` for the entire startup grace period. That helper resets the event-loop delay monitor and CPU/ELU baselines, so the first post-grace tick can still include the cold-start stall and emit the false warning this PR is trying to suppress. Call the sampler every tick and gate only event/warn emission while startup grace is active.
