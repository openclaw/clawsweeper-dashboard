## Summary

Keep this PR open: the bug and fix direction are credible, but the patch suppresses the liveness sampler itself during grace, which can preserve cold-start delay in the first post-grace sample and still emit the false warning. The real behavior proof also needs actual redacted terminal/log output because the body currently includes a claimed grace log line that is not implemented by the diff.

## What This Changes

Adds an internal 60s diagnostic heartbeat startup grace option, wires it from gateway startup, and adds a unit test for suppressing liveness warnings during startup.

## Best Possible Solution

Revise the heartbeat grace handling so liveness sampling and baseline resets continue every tick while warning/event emission is suppressed during startup, then add focused regression coverage and fresh redacted startup log proof.

## Reproduction Assessment

Yes, from source inspection: current main has no startup grace at the gateway heartbeat call, and the liveness warning path emits when a sample exists while queued or waiting diagnostic work is open. I did not run a live cold-start repro in this read-only review.

## Solution Assessment

No. The intended direction is narrow, but the implementation should call the sampler during grace and suppress only emission so the first post-grace tick does not include cold-start delay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sampling during the grace window:** `src/logging/diagnostic.ts:958-961`
  - body: This skips `sampleDiagnosticLiveness` for the whole startup grace period. That helper is what resets the event-loop delay monitor and CPU/ELU baselines, so the first post-grace tick can still include the cold-start stall and emit the false liveness warning this PR is trying to suppress. Call the sampler every tick and gate only the event/warn emission while `inStartupGrace` is true.
  - confidence: 0.88
