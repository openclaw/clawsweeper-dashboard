## Summary

Keep open: the PR targets a real source-reproducible diagnostic bug, but the latest head still skips liveness sampling during the startup grace window and the after-fix runtime proof is not auditable.

## What This Changes

The PR adds an internal diagnostic startupGraceMs option, passes a 60s grace window from gateway startup, and adds a liveness-warning regression test.

## Best Possible Solution

Run the liveness sampler on every heartbeat, suppress only liveness event/WARN emission during the startup grace window, cover that sampler contract in the test, and add redacted live startup output.

## Reproduction Assessment

Yes. Current main starts diagnostic liveness sampling at gateway startup with no grace gate, and a focused fake-timer test with queued diagnostic work can exercise the warning path from source inspection.

## Solution Assessment

No. The direction is right, but the current PR skips sampling during grace; the narrower maintainable fix is to always sample and gate only liveness event/WARN emission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep liveness sampling active during startup grace:** `src/logging/diagnostic.ts:962-964`
  - body: The grace branch returns `null` before calling `sampleLiveness`. `sampleDiagnosticLiveness` resets event-loop-delay metrics and CPU/ELU baselines, so startup delay can carry into the first post-grace sample and still produce the false warning this PR is trying to suppress. Always call the sampler and gate only event/WARN emission; update the test to assert the sampler ran during grace.
  - confidence: 0.95
