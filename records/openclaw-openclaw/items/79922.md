## Summary

Keep open: the live PR head still suppresses the liveness sampler during the grace window, so the first post-grace sample can inherit cold-start delay and reintroduce the false warning this PR is trying to avoid. The real behavior proof is also not auditable yet.

## What This Changes

Adds a diagnostic heartbeat `startupGraceMs` option, passes a 60 second gateway startup grace window, and adds a unit test for liveness warning suppression.

## Best Possible Solution

Keep the startup grace, but run the liveness sampler every heartbeat and suppress only liveness event/WARN emission during grace, with a regression assertion that sampling still occurs.

## Reproduction Assessment

Yes. Current main starts diagnostic liveness at gateway startup with no grace guard, and source inspection shows queued work plus a non-null liveness sample reaches the warning path; a focused fake-timer test can exercise it without the reporter's Windows host.

## Solution Assessment

No. The requested direction is right, but this patch gates before sampling; the safer fix is to sample every tick and gate only event/WARN emission while grace is active.
