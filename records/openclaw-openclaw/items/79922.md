## Summary

Keep open: this PR targets a real diagnostics bug, but the patch skips the liveness sampler during the grace window and lacks after-fix real behavior proof, so it should not merge as-is.

## What This Changes

The PR adds a diagnostic heartbeat startup grace option, passes a 60s grace from gateway startup, and adds a focused fake-timer liveness warning test.

## Best Possible Solution

Revise the patch so every heartbeat calls the liveness sampler during startup grace and only suppresses event/warning emission, then add redacted real cold-start proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main lacks a startup-grace gate, and this PR's changed ternary is enough to see that the sampler is skipped during grace instead of being called and suppressed.

## Solution Assessment

No. The direction is right, but the implementation should call the raw sampler unconditionally and discard its result only while startup grace is active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Call the liveness sampler during grace:** `src/logging/diagnostic.ts:958-961`
  - body: Inside the grace window this returns `null` before invoking the sampler. The real sampler resets the event-loop-delay histogram and CPU/ELU baselines each call, so a cold-start spike can carry into the first post-grace heartbeat and still emit the false warning this PR is meant to suppress. Call the sampler unconditionally, then ignore its result during grace.
  - confidence: 0.94
