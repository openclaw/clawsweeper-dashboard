## Summary

Keep open: the live head still suppresses liveness by skipping the sampler during grace, which can carry startup delay into the first post-grace sample, and the after-fix proof is narrative-only.

## What This Changes

The PR adds a `startupGraceMs` diagnostic heartbeat option, passes a 60 second grace window from gateway startup, and adds a liveness-warning regression test.

## Best Possible Solution

Keep the PR open, sample liveness every heartbeat, suppress only liveness event/WARN emission during startup grace, and require auditable redacted startup output before merge.

## Reproduction Assessment

Yes, source-reproducible: current main starts diagnostic liveness at gateway startup without a grace gate, and queued diagnostic work plus a warning liveness sample emits on the first heartbeat path. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is right, but the current PR gates before sampling; the narrower maintainable fix is to always call the sampler and gate only event/WARN emission during the grace window.
