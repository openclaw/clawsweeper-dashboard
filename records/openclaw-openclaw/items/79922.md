## Summary

Keep open: the PR targets a real current-main diagnostics bug, but the current diff skips the liveness sampler during the grace window, so the first post-grace sample can still report cold-start delay; it also lacks required real behavior proof from an external contributor setup.

## What This Changes

The PR adds a `startupGraceMs` option to diagnostic heartbeat startup, passes a 60-second grace window from gateway startup, and adds a focused liveness warning test.

## Best Possible Solution

Update the PR to always call the liveness sampler every heartbeat, suppress only liveness event/WARN emission during startup grace, add a regression assertion that the sampler runs during grace, and require redacted real startup proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main starts diagnostic heartbeat during gateway startup with no grace and existing fake-timer tests show a queued-work liveness sample warns on the first heartbeat. I did not run a live Windows cold-start reproduction in this read-only pass.

## Solution Assessment

No, this PR is not yet the best way to solve it because it suppresses warnings by skipping the sampler during grace, which misses the monitor reset/baseline update path. The narrow fix is the emission-boundary approach already described in https://github.com/openclaw/openclaw/pull/76401: sample every tick, then discard the sample only while grace is active.
