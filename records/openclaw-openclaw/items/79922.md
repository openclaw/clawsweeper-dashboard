## Summary

Keep open: the PR targets a valid diagnostics bug, but the latest head still skips liveness sampling during the grace window and the real behavior proof is not auditable.

## What This Changes

Adds a diagnostic heartbeat `startupGraceMs` option, passes a 60 second gateway startup grace window, and adds a liveness warning suppression test.

## Best Possible Solution

Keep the startup grace, but always run the liveness sampler on every heartbeat and suppress only liveness event/WARN emission while grace is active, with a test that proves sampling during grace.

## Reproduction Assessment

Yes. Source inspection shows current main starts diagnostic liveness at gateway startup without a grace window, and queued diagnostic work plus a non-null liveness sample reaches the WARN path; a fake-timer test can reproduce this without the reporter's Windows host.

## Solution Assessment

No. The requested direction is right, but this PR gates before sampling; the narrower maintainable fix is to sample every tick and gate only event/WARN emission.
