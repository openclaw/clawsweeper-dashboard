## Summary

Keep open: current main still has unbounded xAI OAuth response reads, and this PR is a narrow hardening that uses the existing bounded response reader without adding new dependencies or config. I found no blocking correctness or security finding, but maintainers should explicitly accept the new auth-provider response cap and the lack of committed regression coverage.

## What This Changes

The PR imports `readResponseWithLimit`, adds a 16 MiB xAI OAuth response cap, and applies it to the shared OAuth body reader plus the device-code polling JSON read.

## Best Possible Solution

Land this focused xAI OAuth hardening once maintainers accept the 16 MiB auth response cap; adding a focused oversized-body regression in `extensions/xai/xai-oauth.test.ts` would make the protection durable.

## Reproduction Assessment

Yes at source level: current main still calls `response.text()` and `response.json()` on the targeted xAI OAuth response paths, and the repository security rule identifies that pattern as an OOM risk. I did not run a current-main OOM repro because this review is read-only.

## Solution Assessment

Yes: using the existing SDK bounded response helper is the narrow owner-boundary fix, and preserving `readResponseBody` text keeps the current Cloudflare/error-body handling intact. `readProviderJsonResponse` is a viable helper for pure JSON reads, but it is not clearly better for the text-preserving shared OAuth reader.
