## Summary

Keep open: current main and v2026.6.10 still show the reported source shape, and https://github.com/openclaw/openclaw/pull/96799 is an open closing PR that should be reviewed before this issue closes.

## What This Changes

The issue asks QMD memory_search to distinguish guard/non-result outcomes from real zero-hit searches so interactive search only force-syncs on dirty, legitimate zero-hit results.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/96799 with a narrow memory-core fix that preserves public search compatibility while preventing forced sync for scope-denied, empty-query, no-collections, and clean-index zero-hit outcomes.

## Reproduction Assessment

Yes, from source inspection. Current main still force-syncs after any empty memory_search result while QMD returns [] for scope-denied, empty-query, and no-collection paths; I did not run a live QMD gateway reproduction.

## Solution Assessment

Yes at the issue level: discriminated internal search outcomes are the narrow maintainable direction because they preserve search() compatibility while letting memory_search avoid sync for guard and clean-index outcomes. The linked PR is the active candidate but still needs review for the exact API boundary.
