## Summary

Keep open: current main still has the ambiguous QMD empty-result contract and the unconditional zero-result forced-sync retry, and https://github.com/openclaw/openclaw/pull/96799 is an open closing PR that should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that QMD-backed `memory_search` treats scope/setup non-results as ordinary zero hits and can force a blocking search sync in the interactive path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96799 or an equivalent narrow fix that preserves `search()` compatibility while preventing QMD guard/setup and clean-index zero-hit outcomes from triggering blocking sync.

## Reproduction Assessment

Yes, at source level. Current main still returns `[]` for QMD scope/setup non-results and then force-syncs after any empty `memory_search` result; I did not run a live QMD Gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction. A discriminated internal QMD outcome is the maintainable fix because it preserves the existing `search()` result contract while letting the tool skip sync for guard/setup outcomes; the linked PR still needs normal API-boundary review.
