## Summary

Keep open: current main still has the source-reproducible empty-result forced-sync path, and the directly linked fix PR is still open rather than merged; the reporter also added useful downstream runtime verification that should be reviewed with that PR.

## What This Changes

The issue reports that QMD-backed memory_search treats legitimate zero hits, scope denial, empty queries, and missing collections as the same empty result and can then force a blocking QMD sync.

## Best Possible Solution

Keep this issue open, review and land https://github.com/openclaw/openclaw/pull/96799 or an equivalent focused fix, and coordinate with the broader https://github.com/openclaw/openclaw/issues/90023 cluster to avoid duplicate QMD zero-hit fixes.

## Reproduction Assessment

Yes from source inspection, though not from a live runtime repro in this read-only pass. Current main still force-syncs after any empty memory_search result while QMD returns [] for scope-denied, empty-query, and no-collections paths.

## Solution Assessment

Yes in direction: discriminating QMD search outcomes is the maintainable fix because it preserves the public search() hits contract while preventing guard/setup outcomes from triggering stale-index repair. The linked PR still needs normal maintainer review for its SDK/API boundary.
