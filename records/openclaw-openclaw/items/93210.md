## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist the anchor for retried payloads:** `apps/ios/Sources/Health/HealthExportService.swift:157-158`
  - body: When a 5xx/network failure saves a pending body, the matching new anchor is discarded. After that pending body later succeeds, this branch clears it and falls through to a fresh read using the old anchor, so the same HealthKit window can be uploaded again before the anchor advances; store the pending payload with its anchor or otherwise advance exactly once on retry success.
  - confidence: 0.9
- **[P1] Use background-readable export state:** `apps/ios/Sources/Health/HealthExportFileStore.swift:45`
  - body: The export can run from BGProcessingTask/HKObserver wakes, but `anchor.bin` and `pending.json` are written with complete file protection and read failures return nil. While locked, the app can treat existing pending data or anchors as absent and upload the wrong HealthKit window instead of safely deferring.
  - confidence: 0.89
- **[P2] Gate health background scheduling behind configuration:** `apps/ios/Sources/OpenClawApp.swift:108`
  - body: This schedules the Health Export BGProcessingTask on every launch, and again when the app backgrounds, even when the user never enabled Health Export. Keep disabled users on the existing background path and schedule Health Export only after a saved configuration, with scheduling avoided or canceled after clear where possible.
  - confidence: 0.86
