## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate the actual private destination:** `apps/ios/Sources/Health/HealthExportConfigStore.swift:42`
  - body: The guard accepts any host ending in `.ts.net`, but OpenClaw docs distinguish tailnet-only Serve from public Funnel, so this does not enforce the claimed own-private-network boundary before exporting HealthKit data. Require a stronger private endpoint proof, such as a paired gateway or tailnet identity check, or remove the anti-exfiltration claim and get product/security approval for the broader policy.
  - confidence: 0.9
- **[P2] Advance or drop the retried window before reading again:** `apps/ios/Sources/Health/HealthExportService.swift:154`
  - body: When a 5xx/network failure leaves a pending payload, the next successful retry calls `upload(..., advanceAnchor: nil)`, clears the pending body, and then immediately reads from the unchanged old anchor. That re-reads the same HealthKit window and posts it a second time; persist the matching anchor with the pending payload or make the retry path advance/skip before reading fresh samples.
  - confidence: 0.93
- **[P1] Skip background export when protected state is unavailable:** `apps/ios/Sources/Health/HealthExportFileStore.swift:56`
  - body: The file store returns `nil` when the protected anchor or pending payload cannot be read, and the new background/observer paths still call `reconcile`. A locked-device read failure can therefore look like a first run with no saved anchor or pending payload; treat protected-data unavailability as a stop/retry condition, or use a protection class suitable for the state that background exports require.
  - confidence: 0.84
