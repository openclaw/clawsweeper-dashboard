## Summary

Keep open: current main and the latest release still apply `toolFilter` against raw MCP `listTools()` names while `openclaw mcp probe` reports namespaced projected names, so the reported silent tool drop remains a valid source-reproducible bug. A fresh open candidate PR now references this issue, so maintainers should review that PR rather than queueing a duplicate repair.

## What This Changes

The issue reports that MCP per-server include/exclude filters silently hide all tools when users copy the namespaced tool names shown by `openclaw mcp probe`.

## Best Possible Solution

Review, adjust if needed, and land the linked candidate PR so the runtime accepts probe-visible namespaced tool names or otherwise emits a clear diagnostic when a non-empty include matches nothing.

## Reproduction Assessment

Yes, from source: current main passes raw `client.listTools()` names into `shouldExposeMcpTool`, while probe output is built from namespaced projected tool ids. I did not run the live MotherDuck credentialed flow in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to align the filter matcher with the names users see from probe and add a zero-match diagnostic; docs-only would still leave the runtime failure mode silent.
