## Summary

Keep open: current main and the latest release still apply MCP `toolFilter` entries to raw server tool names while `openclaw mcp probe` returns projected namespaced tool names, so the reported silent tool drop remains valid. The linked open PR is the active candidate fix, so this issue should stay open for that PR to land or be replaced.

## What This Changes

The issue reports that MCP per-server include/exclude filters can hide every server tool when users copy the namespaced tool names shown by `openclaw mcp probe`.

## Best Possible Solution

Land or replace the focused MCP filter fix so filter matching accepts the names users see from probe, preserves raw-name compatibility, and emits a diagnostic when a non-empty include exposes no tools.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a non-empty MCP include list with namespaced probe values, and current main filters raw `listTools()` names before projection; I did not run the credentialed MotherDuck flow in this read-only review.

## Solution Assessment

Yes. The best fix is a narrow runtime/docs/test change that aligns filter matching with the probe-visible naming contract, keeps raw-name filters working, and warns when a non-empty include matches nothing.
