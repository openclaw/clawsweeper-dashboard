## Summary

Keep open: the underlying gateway-auth security issue is still valid on current main, and this PR is directionally useful, but the patch is incomplete because managed service install/repair paths can still accept the invalid configuration before the new runtime guard runs.

## What This Changes

The PR adds a startup-time `resolveGatewayRuntimeConfig` guard and regression tests to reject Tailscale Serve with explicit `gateway.auth.mode: "none"` while preserving adjacent Tailscale auth cases.

## Best Possible Solution

Land a focused gateway-auth fix that rejects serve+none before runtime startup and managed service writes, preserves loopback-only none mode and valid Serve auth modes, adds regression coverage, and records the migration in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection on current main shows runtime config accepts serve+none, an existing test expects that resolution to succeed, and `authorizeGatewayConnectCore` grants mode-none callers without credentials.

## Solution Assessment

No. The runtime guard is necessary, but the narrow maintainable fix also blocks the same invalid configuration before service install/repair writes a daemon and includes the required changelog entry.
