## Summary

Keep open: the security issue is valid on current main, but this PR is not merge-ready because the rejection is runtime-only, install/preflight can still accept the same configuration, and after-fix real Tailscale Serve proof is missing.

## What This Changes

Adds a startup-time guard rejecting Tailscale Serve with `gateway.auth.mode: "none"` and adds focused runtime-config tests for Serve/Funnel auth combinations.

## Best Possible Solution

Centralize fail-closed validation for Serve plus explicit no-auth across runtime startup and install/onboarding preflight, while preserving trusted loopback-only no-auth and Serve with token/password coverage.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

No. The runtime startup guard is necessary, but the same policy should be enforced before install/onboarding writes or accepts a managed service plan.
