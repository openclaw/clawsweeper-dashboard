## Summary

This PR remains useful but is not ready to merge: it addresses a real gateway auth-boundary gap, but the branch only rejects the dangerous configuration at runtime startup, lacks contributor-supplied real Tailscale Serve proof, and is currently conflicting with main.

## What This Changes

The PR adds a startup-time rejection and focused unit coverage for `gateway.auth.mode="none"` combined with `gateway.tailscale.mode="serve"`.

## Best Possible Solution

Centralize a shared validation policy that rejects explicit no-auth with Tailscale Serve before runtime and install/onboarding service plans proceed, keep loopback-only no-auth valid, and land with redacted live Tailscale proof plus focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, then `authorizeGatewayConnectCore` returns `method: "none"` before any credential or Tailscale identity requirement is enforced.

## Solution Assessment

No, not as written. A runtime startup guard is directionally correct, but the safer fix rejects the same configuration in shared preflight before install/onboarding writes a service plan that later fails.
