## Summary

Keep open. This PR targets a real gateway auth exposure, but the patch still leaves managed install/onboard paths accepting the same invalid configuration and writing a service that will fail immediately at startup, so it should not merge as-is.

## What This Changes

The PR adds a startup-time runtime-config guard and regression tests to reject Tailscale Serve with explicit `gateway.auth.mode: "none"`, while preserving Funnel password enforcement and Serve with token auth.

## Best Possible Solution

Keep the focused runtime rejection, but add matching pre-start validation or normalization in daemon install/onboard/doctor service paths before a managed service is written. Preserve local loopback `auth.mode="none"`, allow Serve with real shared-secret auth or verified Tailscale identity where intended, and keep Funnel's password-only rule.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` in runtime config resolution, and the auth path then accepts `mode="none"` callers without credentials.

## Solution Assessment

No, not yet. The runtime guard is a necessary core boundary, but it should be paired with install/doctor/service validation or auth normalization so managed daemon setup cannot write a service that immediately fails under the new guard.
