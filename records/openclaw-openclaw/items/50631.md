## Summary

Keep open. The PR targets a real Gateway auth-boundary bug, but the branch is incomplete because it only rejects the unsafe Serve plus no-auth shape at runtime while install/onboarding preflight can still accept it, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds a `resolveGatewayRuntimeConfig` guard rejecting Tailscale Serve with `gateway.auth.mode: "none"` and adds runtime-config tests for Serve/Funnel auth combinations.

## Best Possible Solution

Centralize the Serve plus no-auth rejection so runtime startup and daemon install/onboarding preflight share the same policy, while preserving loopback-only no-auth and Serve with token/password as valid configurations.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

No. The runtime guard is necessary, but the patch should also reject the same configuration before managed service install/onboarding writes a daemon plan that will fail on boot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject serve+none before install flows consume it:** `src/gateway/server-runtime-config.ts:130-134`
