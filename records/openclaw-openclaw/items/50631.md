## Summary

Keep this PR open: the reported Serve + `auth.mode=none` bug is still present on current `main`, but the branch only adds a runtime startup guard and still misses the daemon install/onboarding preflight path called out in prior review.

## What This Changes

The PR adds a `resolveGatewayRuntimeConfig` guard rejecting Tailscale Serve with `gateway.auth.mode: "none"` and adds gateway runtime-config regression tests.

## Best Possible Solution

Share the Serve + none rejection through runtime and daemon install/onboarding preflight policy, preserving loopback-only `none` and Serve with real auth as valid configurations.

## Reproduction Assessment

Yes. Source inspection on current `main` shows Serve + auth mode `none` resolves successfully, an existing runtime-config test expects that configuration to pass, and `none` auth authorizes without credentials.

## Solution Assessment

No. The runtime guard is necessary, but the patch should also reject or normalize the same configuration before managed service install/onboarding writes a daemon plan.
