## Summary

Keep open: the underlying Gateway auth-boundary bug is still real on current main, but this PR should not merge as-is because it only blocks runtime startup, leaves managed install/onboarding preflight accepting the same configuration, and lacks external after-fix proof.

## What This Changes

Adds a startup-time runtime-config guard rejecting Tailscale Serve with `gateway.auth.mode: "none"` and adds focused runtime-config tests for Serve/Funnel auth combinations.

## Best Possible Solution

Centralize fail-closed Serve plus explicit no-auth validation across runtime startup and install/onboarding preflight, while preserving loopback-only no-auth and Serve with token/password, then land with focused regression tests and redacted real Serve proof.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

No. The startup guard is necessary, but the patch should also reject the same configuration before managed service install/onboarding writes a daemon plan that will fail on boot.
