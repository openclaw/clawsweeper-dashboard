## Summary

Keep open. The PR addresses a real gateway auth exposure, but the patch is not merge-ready because the new startup-only rejection leaves managed install, onboarding, and doctor service paths able to accept the same config and write a daemon that immediately fails on boot.

## What This Changes

The PR adds a `resolveGatewayRuntimeConfig` guard and tests to reject Tailscale Serve with explicit `gateway.auth.mode: "none"`, while preserving Funnel password enforcement and Serve with token auth.

## Best Possible Solution

Keep the focused runtime rejection, but add matching pre-start validation or normalization in daemon install, onboarding, and doctor service-repair paths before a managed service is written. Preserve trusted loopback-only `auth.mode="none"`, allow Serve with explicit shared-secret auth or verified Control UI/WebSocket Tailscale identity where intended, and keep Funnel password-only.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` in runtime config resolution, and the auth path then accepts mode-none callers without credentials.

## Solution Assessment

No, not yet. The runtime guard is a necessary core boundary, but by itself it is not the best complete fix because managed service install paths can still accept the invalid config and write a daemon that fails immediately under the new guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate serve+none before service install:** `src/gateway/server-runtime-config.ts:130-134`

Add install/doctor/service validation or normalize auth before writing the daemon.
