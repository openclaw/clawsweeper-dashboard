## Summary

Keep open: current main still has the source-level Tailscale Serve plus no-auth exposure, and this PR is a focused but incomplete fix because it only rejects at runtime while managed install/onboarding preflight can still accept the configuration; it also lacks real after-fix behavior proof.

## What This Changes

The branch adds a runtime-config guard that rejects `gateway.auth.mode: "none"` with Tailscale Serve and adds runtime-config tests for Serve/Funnel auth combinations.

## Best Possible Solution

Centralize validation so Tailscale Serve plus explicit no-auth is rejected before runtime and before managed service plans are written, while preserving loopback no-auth and Serve with token/password, with focused regression tests and real Serve proof.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

No. The runtime guard is necessary, but the patch should also reject the same configuration before managed service install/onboarding writes a daemon plan that will fail on boot.
