## Summary

Keep open: the runtime guard is useful, but the PR is incomplete because daemon install/onboarding can still accept the Serve + no-auth configuration, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a runtime-config guard that rejects Tailscale Serve with `gateway.auth.mode: "none"` and adds regression tests for Serve/Funnel auth combinations.

## Best Possible Solution

Share the Serve + none rejection through runtime and daemon install/onboarding preflight, preserving trusted loopback-only no-auth and Serve with real auth as valid configurations, with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none without credentials.

## Solution Assessment

No. The runtime guard is necessary, but the patch should also reject or normalize the same configuration before managed service install/onboarding writes a daemon plan.
