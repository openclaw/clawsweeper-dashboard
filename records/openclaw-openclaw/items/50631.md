## Summary

Keep open: the current-main vulnerability is source-reproducible and this PR moves in the right direction, but the patch is incomplete because managed install/repair paths can still accept the invalid configuration before writing a service that fails at startup; the user-facing fix also needs a changelog entry.

## What This Changes

The PR adds a `resolveGatewayRuntimeConfig` guard and tests to reject Tailscale Serve with explicit `gateway.auth.mode: "none"`, while preserving Funnel password enforcement and Serve with token auth.

## Best Possible Solution

Land a focused fix that rejects serve+none before both runtime startup and managed service install/repair paths, keeps trusted loopback-only none mode allowed, preserves valid Serve auth modes, adds regression coverage, and records the user-facing change in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveGatewayRuntimeConfig` accepts serve+none, `authorizeGatewayConnectCore` then accepts mode-none callers without credentials, and an existing runtime-config test expects serve+none resolution to succeed.

## Solution Assessment

No. The runtime guard is necessary, but the best fix also blocks or normalizes the same invalid configuration before daemon install, onboarding daemon install, and doctor service repair write a managed service.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Validate serve+none before service install:** `src/gateway/server-runtime-config.ts:130-134`

Add matching install/repair validation or normalize auth before writing the daemon.
