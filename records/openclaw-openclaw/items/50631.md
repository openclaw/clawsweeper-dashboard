## Summary

Keep open: the security issue is valid on current main, but this PR should not merge as-is because the new runtime guard leaves managed install and repair paths able to write a service that only fails on startup, and the user-facing security change lacks a changelog entry.

## What This Changes

The PR adds a runtime validation guard and tests to reject `gateway.auth.mode: "none"` when Tailscale Serve is active while preserving Serve with token auth.

## Best Possible Solution

Land a focused fix that rejects Serve + none consistently before runtime startup and before managed service writes, preserves loopback-only none mode, keeps valid Serve auth modes working, and records the migration in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection on current main shows runtime config accepts Serve + none, auth mode none authorizes without credentials, and an existing test expects that configuration to resolve successfully.

## Solution Assessment

No. The runtime guard is necessary, but the narrow maintainable fix also needs install/daemon/doctor repair preflight coverage so OpenClaw does not write a managed service that is known to fail at startup.
