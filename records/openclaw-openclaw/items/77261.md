## Summary

Current main already handles the reported Windows fsync failure in the shared media store, and Telegram inbound media goes through that helper before the user-facing download-failed reply path. The fix is proven on main but not traceable to a shipped release tag from this checkout.

## What This Changes

Windows Telegram inbound media should save successfully when `FileHandle.sync()` returns `EPERM` instead of replying that media download failed.

## Best Possible Solution

Keep the shared media-store EPERM tolerance on main and let the next release carry it; no Telegram-specific workaround is needed.

## Reproduction Assessment

No current-main failure reproduction is present: the original path is source-backed, but current main catches `EPERM` in `syncSavedMediaHandle` and has a focused regression test for that case. I did not run tests because this was a read-only review and `pnpm` is unavailable in the environment.

## Solution Assessment

Yes. Handling `EPERM` in the shared media store is the narrowest maintainable fix because Telegram and other channel plugins use `saveMediaBuffer`, and the change keeps fsync best-effort without adding a channel-specific policy knob.
