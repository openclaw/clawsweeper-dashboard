## Summary

Current main already fixes the Windows fsync `EPERM` failure in the shared media store, and Telegram inbound media uses that path, so this can close as implemented on main pending the next release pickup.

## What This Changes

The issue requests that Windows Telegram media ingestion tolerate `EPERM` from `FileHandle.sync()` when saving inbound media.

## Best Possible Solution

Keep the shared media-store EPERM guard on main and let the next release carry it, while treating narrower duplicate fsync reports or alternative PRs as superseded by that central fix.

## Reproduction Assessment

No, not as a current-main failure. Source inspection shows the historical Telegram media path now reaches a media-store guard that swallows only `EPERM` from `FileHandle.sync()`, and a regression test covers that behavior.

## Solution Assessment

Yes. Fixing the shared media-store fsync wrapper is the narrowest maintainable solution because Telegram, WebChat, and other inbound media paths share `saveMediaBuffer`, while non-EPERM sync errors still propagate.
