## Summary

Current main already handles the reported Windows `EPERM` media-store failure in the shared save path used by Telegram, with regression coverage and provenance from the earlier #76844 fix.

## What This Changes

Windows Telegram media ingestion should not fail when the post-write `FileHandle.sync()` call returns `EPERM`.

## Best Possible Solution

Keep the shared media-store EPERM handling on main and include it in the next stable or correction release so Telegram and WebChat media saves share the same fix.

## Reproduction Assessment

Yes for the reported pre-fix failure: source inspection shows Telegram media reaches `saveMediaBuffer`, and the regression test simulates `handle.sync()` rejecting with `EPERM`. I did not establish a failing current-main path because current main already handles that error.

## Solution Assessment

Yes: fixing the shared media-store fsync helper is the narrow maintainable solution because Telegram, WebChat, and other buffered media callers share `saveMediaBuffer`. A Telegram-only catch would leave the same Windows failure in other media ingestion paths.
