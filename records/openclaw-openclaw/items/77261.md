## Summary

Current main already fixes the Windows fsync EPERM media-save failure in the shared media store, so the Telegram symptom is covered without a Telegram-specific workaround.

## What This Changes

Windows Telegram inbound media should continue saving when the shared media-store fsync step returns EPERM instead of replying with a media download failure.

## Best Possible Solution

Keep the shared media-store EPERM tolerance on main and let the next release carry it, rather than adding a Telegram-specific workaround.

## Reproduction Assessment

Yes for the historical failure: the issue report and history identify Telegram media saving through saveMediaBuffer into FileHandle.sync on Windows, and current regression coverage simulates that EPERM path. Against current main, the failure should no longer reproduce because EPERM is handled as a best-effort sync miss.

## Solution Assessment

Yes. The current fix is the narrowest maintainable solution because it preserves atomic temp-file writes, ignores only EPERM from the best-effort fsync step, and still propagates other storage failures.
