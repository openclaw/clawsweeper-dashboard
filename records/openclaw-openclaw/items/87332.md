- `P2`: This is a normal-priority Telegram/plugin-state stability bug with real upgrade impact but a limited affected surface.
- `impact:data-loss`: The reported failure can leave Telegram cache migration entries unwritten or partially imported when plugin-state cap enforcement trips.

## Summary

Keep open. Current main still matches the reported shipped behavior: Telegram opens `telegram.message-cache` with a 1000-entry namespace budget but no TTL, persists rows without per-write TTL, and the core plugin-state store treats `expires_at = NULL` as live rows that count toward the global 1000-row plugin cap. No merged fix or canonical duplicate was found in the provided related context; the only current discussion is a contributor offering to investigate.

## What This Changes

The issue asks for Telegram message-cache plugin-state rows to expire or be bounded so they cannot fill the Telegram plugin’s 1000-row state cap and block doctor migrations.

## Best Possible Solution

Keep this open and land a focused Telegram/plugin-state fix that gives message-cache durable rows a bounded lifetime or smaller namespace budget with global-cap headroom, handles existing NULL-expiry rows, and protects doctor topic-name imports with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: fill `telegram.message-cache` with NULL-expiry rows near the plugin-wide cap, add any sibling Telegram namespace row, then a message-cache write or doctor plugin-state import can hit `PLUGIN_STATE_LIMIT_EXCEEDED`; I did not execute it because this review is read-only.

## Solution Assessment

Yes, the maintainable direction is a narrow Telegram-owned fix using the existing plugin-state TTL and max-entry contracts, plus migration coverage. Raising the global 1000-row cap alone would only move the failure point.
