## Summary

Keep open. Current main still exposes only an internal TUI event-handler watchdog override; normal TUI startup, CLI options, slash commands, docs, and config schema do not provide a user-facing threshold or disable control, and related #69026 and #69978 remain open rather than merged fixes.

## What This Changes

Request a user-facing configuration/control for the TUI streaming watchdog idle threshold so long-reasoning runs do not trigger the hardcoded 30-second warning.

## Best Possible Solution

Add a validated TUI-owned watchdog control, plumb it into createEventHandlers, document it, and cover default, configured, disabled, and warning behavior with focused tests.

## Reproduction Assessment

Yes. The current source and fake-timer tests provide a high-confidence path: an active run with no stream delta for the watchdog window triggers the warning/reset, and normal runTui does not override the 30-second default.

## Solution Assessment

Unclear on the exact UX. A user-facing TUI setting or command is the maintainable direction; provider/model timeout changes and #69026's simple threshold bump do not solve the configurability request.
