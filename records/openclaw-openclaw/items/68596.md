## Summary

Keep open: current main and the latest shipped tag still expose only an internal TUI `streamingWatchdogMs` seam with a 30-second fallback, while the public CLI/config/docs surface requested here remains absent and the active implementation PR is still unmerged.

## What This Changes

Requests a user-facing way to tune or disable the TUI streaming watchdog timeout instead of always using the current 30-second fallback.

## Best Possible Solution

Land a focused, maintainer-approved watchdog control that plumbs the chosen public setting into `createEventHandlers`, documents default and disabled behavior, and coordinates with the open related TUI watchdog issues.

## Reproduction Assessment

Yes for the current behavior. Source inspection and existing fake-timer tests show the 30-second watchdog still fires unless an internal `streamingWatchdogMs` value is passed, and normal TUI startup does not pass one.

## Solution Assessment

Unclear on the exact UX. A TUI-owned config/command/flag surface is maintainable, but maintainers still need to choose the public contract and decide whether broader default changes belong with this issue.
