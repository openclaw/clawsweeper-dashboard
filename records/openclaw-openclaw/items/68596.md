## Summary

Keep open: current main and the latest release still have only an internal TUI handler override for the streaming watchdog, with no user-facing config or CLI plumbing. The closed related issues and the closed unmerged implementation attempt do not replace this canonical request, while the open CLI watchdog issue is adjacent but covers a different timeout path.

## What This Changes

The issue requests a configurable threshold or disable switch for the TUI streaming watchdog notice that appears after quiet streaming periods.

## Best Possible Solution

Add a maintainer-approved TUI watchdog control under the chosen config surface, validate `0` and positive values, plumb it into `runTui` and `createEventHandlers`, update docs/tests, and keep the separate CLI backend no-output watchdog work tracked by https://github.com/openclaw/openclaw/issues/40982.

## Reproduction Assessment

Yes, at source level: current main still defaults the TUI watchdog to 30_000 ms and the real `runTui` callsite does not pass a configured override. I did not run an interactive TUI scenario because this was a read-only review, but the code path and existing handler tests are clear.

## Solution Assessment

Unclear as proposed: the need is real, but the issue’s suggested `gateway` or `acp.stream` paths may be the wrong owner for a terminal-only UI watchdog. A narrower `tui` config setting or explicit maintainer-chosen alternative would be the cleaner solution.
