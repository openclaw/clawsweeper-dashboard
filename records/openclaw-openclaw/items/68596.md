## Summary

Keep open. Current main still falls back to a 30-second TUI streaming watchdog with only an internal handler option, while normal TUI startup, CLI options, config schema, slash commands, and public docs expose no user-facing threshold or disable control; related #69026 and #69978 remain open rather than merged fixes.

## What This Changes

Request a user-facing way to configure or disable the TUI streaming watchdog idle timeout for long-silence reasoning, model, and tool runs.

## Best Possible Solution

Add a validated TUI-owned watchdog control, plumb it from the chosen public surface into createEventHandlers, document it, and cover default, configured, disabled, and warning behavior with focused tests.

## Reproduction Assessment

Yes. The current source and fake-timer tests give a high-confidence path: a run with no stream delta for DEFAULT_STREAMING_WATCHDOG_MS triggers the warning/reset path, and normal runTui does not override the 30-second default.

## Solution Assessment

Unclear on the exact UX. A TUI-owned setting or command is the maintainable direction, while provider/model timeout changes would not solve this client-side watchdog and #69026's simple threshold bump would not satisfy configurability.
