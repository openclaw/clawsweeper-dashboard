## Summary

Keep open. Current main still has only an internal TUI `streamingWatchdogMs` handler option with a 30-second fallback, while normal TUI startup, CLI/config schema, slash commands, and public docs expose no user-facing timeout or disable control; the discussion also has multiple affected-user confirmations and an unmerged implementation PR.

## What This Changes

Request a configurable or disable-able TUI streaming watchdog timeout so long-silence reasoning or tool runs do not get false stale-response notices.

## Best Possible Solution

Land a focused, maintainer-approved TUI watchdog control that plumbs the chosen public surface into `createEventHandlers`, documents default and disabled behavior, and coordinates with the adjacent watchdog issues and open implementation PR.

## Reproduction Assessment

Yes for the current behavior. Source inspection and existing fake-timer tests show the 30-second watchdog still fires unless an internal `streamingWatchdogMs` value is passed, and normal TUI startup does not pass one.

## Solution Assessment

Unclear on the exact UX. A TUI-owned config, flag, command, or combined surface is the maintainable direction, but maintainers still need to choose the public contract and decide whether the broader default changes in the open implementation PR belong with this issue.
