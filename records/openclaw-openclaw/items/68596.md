## Summary

Keep open. Current main still falls back to a 30-second TUI streaming watchdog with only an internal handler option, while normal TUI startup, CLI options, config schema, slash commands, and public docs expose no user-facing threshold or disable control. Related #69026 and #69978 remain open, and closed #75229 appears to be overlapping rather than a merged fix.

## What This Changes

The issue requests a user-facing way to configure or disable the TUI streaming watchdog timeout instead of relying on the hardcoded 30-second client threshold.

## Best Possible Solution

Add a validated TUI-owned watchdog control that can raise or disable the idle threshold, plumb it from the chosen public surface into `createEventHandlers`, document it on the TUI/config pages, and cover default, configured, disabled, and warning behavior with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The current handler defaults to 30,000ms, normal `runTui` passes no override, and the existing fake-timer tests exercise the same warning/reset path.

## Solution Assessment

Is this the best way to solve the issue? Unclear on the exact UX, but the best direction is a TUI-owned user-facing setting or command; changing provider/model timeouts would not address this client-side watchdog, and #69026's threshold bump alone would not make it configurable.
