## Summary

Keep open. Current main and the latest shipped release still only have an internal TUI `streamingWatchdogMs` handler option with a 30-second fallback; normal TUI startup, CLI/config surfaces, slash commands, and public docs still expose no user-facing timeout or disable control, while the open implementation PR remains unmerged and conflicting.

## What This Changes

Request to make the TUI streaming watchdog timeout configurable or disableable for slow or extended-reasoning models instead of using a fixed 30-second idle window.

## Best Possible Solution

Land a focused, maintainer-approved TUI watchdog control that plumbs the chosen public surface into `createEventHandlers`, documents default and disabled behavior, and coordinates with the related watchdog UX and quiet-run issues.

## Reproduction Assessment

Yes for the current behavior. Source inspection and existing fake-timer tests show the 30-second watchdog still fires unless an internal `streamingWatchdogMs` value is passed, and normal TUI startup does not pass one.

## Solution Assessment

Unclear on the exact UX. A TUI-owned public control is the maintainable direction, but maintainers still need to choose the public contract and decide whether broader default changes belong with this issue.
