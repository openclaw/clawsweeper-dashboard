## Summary

Keep open. Current main and the latest shipped tag still only have an internal TUI `streamingWatchdogMs` handler option with a 30-second fallback; normal TUI startup, CLI flags, config schema, and public docs still expose no user-facing timeout or disable control, and the active implementation PR remains unmerged.

## What This Changes

The issue requests a user-facing way to configure or disable the TUI streaming watchdog timeout for long quiet model runs.

## Best Possible Solution

Land a focused, maintainer-approved TUI watchdog control that plumbs the chosen public surface into `createEventHandlers`, documents default and disabled behavior, and coordinates with https://github.com/openclaw/openclaw/issues/69978 and https://github.com/openclaw/openclaw/issues/78360.

## Reproduction Assessment

Yes for the current behavior. Source inspection and existing fake-timer tests show the 30-second watchdog still fires unless an internal `streamingWatchdogMs` value is passed, and normal TUI startup does not pass one.

## Solution Assessment

Unclear on the exact UX. A TUI-owned config/command/flag surface is the maintainable direction, but maintainers still need to choose the public contract and decide whether the broader default changes in the open implementation PR belong with this issue.
