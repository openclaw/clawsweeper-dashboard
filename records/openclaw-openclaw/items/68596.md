## Summary

Keep open: current main and the latest release still expose only an internal TUI watchdog override, not a user-facing config, CLI, or command surface. The open related PR is a possible implementation path, but it is unmerged and carries broader default-policy decisions that need maintainer review.

## What This Changes

Request a configurable or disable-able TUI streaming watchdog timeout so long silent model or tool phases do not trigger misleading 30-second warnings.

## Best Possible Solution

Land or replace a narrow maintainer-approved TUI watchdog control that validates 0/positive values, plumbs it into `runTui`, updates docs/tests, and avoids unrelated watchdog default changes.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: normal TUI startup does not pass `streamingWatchdogMs`, so the handler falls back to 30 seconds. This is a feature/config request rather than a strict existing-contract bug.

## Solution Assessment

Unclear on the exact surface. The existing handler seam is the right low-level hook, but maintainers still need to choose the durable config, CLI, or command contract and keep the fix narrowly scoped to TUI behavior.
