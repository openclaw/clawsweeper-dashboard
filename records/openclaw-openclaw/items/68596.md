## Summary

Keep open: current main still only has an internal TUI watchdog option with a 30-second fallback, no public config/CLI/docs surface, and an open implementation PR is already attempting the central wiring but still needs human review/proof.

## What This Changes

Request to make the TUI streaming watchdog timeout configurable so long-reasoning or sparse-output runs do not hit the fixed 30-second warning/reset path.

## Best Possible Solution

Keep this canonical feature request open until a maintainer-approved TUI watchdog control lands with schema/docs/tests and is coordinated with https://github.com/openclaw/openclaw/issues/69978 and the quiet-active-run bug at https://github.com/openclaw/openclaw/issues/78360.

## Reproduction Assessment

Yes for the current behavior, though this is a feature/config request rather than a strict bug. Source inspection shows normal TUI startup cannot supply `streamingWatchdogMs`, so current sessions still use the 30-second handler fallback.

## Solution Assessment

Unclear on the exact product surface. A TUI-owned config or command surface is plausible, but maintainers still need to choose the contract and separate timeout policy from the related quiet-active-run behavior bug.
