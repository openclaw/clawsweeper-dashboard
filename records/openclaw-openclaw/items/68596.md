## Label Justifications

- `P2`: The issue affects real TUI users running long sparse-output model work, but the remaining work is a bounded config/UX improvement rather than a core outage.
- `impact:session-state`: The watchdog path can clear the TUI active-run state and show idle during quiet runs, which is session-state drift even when the backend may still be active.

## Summary

Keep open: current main and the latest shipped tag still expose only an internal TUI watchdog option, so users still cannot configure or disable the 30-second streaming watchdog through config, CLI, docs, or the normal TUI startup path.

## What This Changes

The issue requests a user-configurable TUI streaming watchdog timeout or disable setting for long-running quiet reasoning streams.

## Best Possible Solution

Add a maintainer-approved TUI watchdog control that carries a validated value from config and/or CLI into runTui, documents the behavior, treats 0 as disabled if that remains the contract, and keeps the separate quiet-active-run fix aligned with https://github.com/openclaw/openclaw/issues/78360.

## Reproduction Assessment

Yes for the current source state: normal TUI startup cannot pass streamingWatchdogMs, so sessions still fall back to the 30-second handler default. This is a feature/config request rather than a strict current-contract bug.

## Solution Assessment

Unclear on the exact surface. The existing handler seam is the right low-level hook, but maintainers still need to choose the durable public config or command contract and coordinate it with the quiet-active-run behavior fix.
