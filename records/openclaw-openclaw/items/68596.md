## Label Justifications

- `P2`: The issue affects TUI usability and confidence during sparse-output long-running model turns, but the remaining work is a bounded config/UX decision rather than a core outage.
- `impact:session-state`: The watchdog notice can make an active TUI run appear stale or uncertain while the backend may still be working, encouraging duplicate sends or context disruption.

## Summary

Keep open: current main and the latest release still expose only an internal TUI handler override for the streaming watchdog, with no user-facing config or CLI plumbing. Related duplicate issues have been closed back to this canonical thread, and the open implementation PR is still an unmerged candidate that needs maintainer direction on config namespace/default behavior.

## What This Changes

The issue requests a user-configurable TUI streaming watchdog timeout or disable switch so long reasoning/tool-call turns do not trigger the hardcoded no-output warning.

## Best Possible Solution

Add a maintainer-approved TUI watchdog control under the chosen config surface, validate `0` and positive values, plumb it into `runTui` and `createEventHandlers`, update focused docs/tests, and keep separate CLI backend no-output watchdog policy on its own canonical issue.

## Reproduction Assessment

Yes. Source inspection shows the missing public route: the handler accepts an internal `streamingWatchdogMs`, but production `runTui()` does not pass one and the strict config schema has no watchdog field.

## Solution Assessment

No final solution is on main yet. The requested capability is real, but the proposed `gateway` or `acp.stream` locations look like the wrong owner for a terminal-only UI watchdog unless maintainers explicitly choose that surface.
