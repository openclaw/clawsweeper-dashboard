## Summary

Keep open: the PR targets the right source-reproducible preflight compaction timeout bug, but the current patch drops user/RPC/restart aborts that arrive through the run's upstream abort signal and still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds a separate `ReplyOperation.explicitAbortSignal` and changes required preflight compaction to use that signal instead of the raw reply operation abort signal.

## Best Possible Solution

Derive a preflight cancellation-only signal that filters only lifecycle `TimeoutError` from upstream reply aborts while preserving user/RPC/restart cancellation, then prove it in a real slow-preflight setup.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction and the safety wrapper races that abort before the configured timeout; I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

No as written: separating lifecycle timeout from compaction timeout is the right direction, but the current implementation filters out all upstream aborts rather than only lifecycle timeout aborts and still lacks real runtime proof.
