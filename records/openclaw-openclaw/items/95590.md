## Label Justifications

- `P1`: The PR targets a real large-session compaction regression that can block agent turns and hold session work in active deployments.
- `merge-risk: 🚨 session-state`: The diff changes required preflight compaction while session context is near overflow, so wrong cancellation semantics can leave session state locked or stale longer than intended.
- `merge-risk: 🚨 availability`: Removing the abort signal can make stop, restart, shutdown, or drain wait for the compaction timeout instead of interrupting preflight work.

## Summary

Keep open: the PR targets the right source-level bug, but removing the abort signal entirely would also remove explicit stop/restart/drain cancellation from required preflight compaction, and the branch still lacks real slow-preflight runtime proof.

## What This Changes

The PR removes the reply-operation abort signal from required preflight compaction and adds a regression test asserting that no abort signal is passed.

## Best Possible Solution

Use a preflight cancellation source that excludes the normal reply lifecycle timeout, still propagates explicit stop/restart/shutdown/drain cancellation, keeps timeout ownership in `compactWithSafetyTimeout`, and proves the path with focused tests plus redacted real slow-backend logs.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races that external abort before the configured timeout. I did not run the reporter's slow-provider setup in this read-only review.

## Solution Assessment

No as written. Removing the raw reply signal is a plausible timeout-owner direction, but the best fix must preserve explicit cancellation while excluding only the lifecycle timeout and needs real runtime proof.
