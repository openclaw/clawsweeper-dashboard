## Summary

Keep open: the update fixes the earlier drop-all-cancellation problem, but the new explicit signal is too narrow because UI/API chat aborts flow through the upstream abort signal and would no longer interrupt required preflight compaction. The PR also still has only test/typecheck proof, not a real slow-preflight OpenClaw run.

## What This Changes

The branch adds `ReplyOperation.explicitAbortSignal`, routes it to required preflight compaction, and tests that upstream lifecycle aborts do not cap compaction while registry stop/restart cancellation still aborts the explicit signal.

## Best Possible Solution

Use a preflight cancellation signal that filters only the normal lifecycle timeout while still propagating user/RPC stop and restart aborts, then prove the slow-preflight path with redacted real runtime output.

## Reproduction Assessment

Yes at source level: current main passes `replyOperation.abortSignal` into required budget/preflight compaction, and the safety wrapper races caller aborts ahead of the configured timeout. I did not live-run the reporter's slow vLLM/Qwen setup.

## Solution Assessment

No as written. The new explicit signal is the right direction, but the implementation suppresses all upstream aborts rather than only lifecycle timeout, so it can miss user/RPC stop and restart cancellation.
