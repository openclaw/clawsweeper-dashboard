## Summary

Keep open. The earlier builder fix is present, but current and v2026.5.7 source still drop `sessionId` before the boundary-aware OpenAI Completions transport, so the runtime path can omit `prompt_cache_key` exactly as reported.

## What This Changes

The issue reports that `openai-completions` no longer emits `prompt_cache_key` in 2026.5.x despite `compat.supportsPromptCacheKey: true`, breaking prefix-cache reuse against oMLX and similar backends.

## Best Possible Solution

Pass the embedded run `sessionId` through the boundary-aware transport wrapper and add a focused regression test that exercises the full stream-resolution path, while preserving the existing compat/cache-retention gates.

## Reproduction Assessment

Yes, from source: `openai-completions` uses the boundary-aware transport path, that wrapper omits `sessionId`, and the completions builder only emits `prompt_cache_key` when `options.sessionId` is present. I did not execute a live backend repro, so this is source-reproducible rather than locally run.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate `sessionId` into the existing wrapper path and test that the runtime route reaches the already-shipped builder gate; changing config schema or compat defaults is unnecessary.
