## Summary

Keep open: current `main` still has the completions builder branch, but source tracing identifies a likely current regression where embedded boundary-aware OpenAI-compatible streams are wrapped without the run `sessionId`, so the builder never gets the value it needs to emit `prompt_cache_key`.

## What This Changes

The issue reports that `openai-completions` requests stopped emitting `prompt_cache_key` in 2026.5.x despite `compat.supportsPromptCacheKey: true`, breaking prefix caching for OpenAI-compatible backends.

## Best Possible Solution

Thread the embedded run `sessionId` through boundary-aware OpenAI-compatible stream wrappers so the existing completions builder can emit `prompt_cache_key`, then add focused regression coverage for that path.

## Reproduction Assessment

Yes, at source level: the embedded runner passes a run `sessionId`, but the boundary-aware OpenAI-compatible branch does not forward it into stream options, so the completions builder cannot emit `prompt_cache_key`. No live Docker/oMLX run was performed during this read-only review.

## Solution Assessment

Yes. Passing `sessionId` into the existing boundary-aware wrapper and adding a regression test is the narrowest maintainable fix; the existing builder and config contract should remain unchanged.
