## Summary

Keep open: current main has the completions builder support, but a current embedded-runner path can still drop `sessionId` before the builder, which explains why `prompt_cache_key` is absent for OpenAI-compatible completions in the reported Docker/oMLX setup.

## What This Changes

Report that OpenAI-compatible completions providers stopped emitting `prompt_cache_key` in 2026.5.x despite `compat.supportsPromptCacheKey` and `cacheRetention` being enabled.

## Best Possible Solution

Propagate the active run `sessionId` through the boundary-aware embedded OpenAI-compatible transport wrapper and add regression coverage that proves `openai-completions` receives it without changing the existing compat/cache-retention gates.

## Reproduction Assessment

Yes at source level, but not with a live Docker run. The current `openai-completions` boundary-aware path can call the builder without `options.sessionId`, and the builder requires that field before it emits `prompt_cache_key`.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the existing session id through `wrapEmbeddedAgentStreamFn` for boundary-aware transports; the builder and compat policy should stay as they are.
