## Summary

Keep open: the report is new, detailed, and describes a release Docker/oMLX regression that current source inspection does not explain. Current `main` and the v2026.5.7 source both contain the expected completions `prompt_cache_key` branch, so the remaining work is to verify the real packaged/runtime request path before declaring this fixed or writing a narrow patch.

## What This Changes

The issue reports that OpenAI-compatible completions requests stopped emitting `prompt_cache_key` in v2026.5.x despite `compat.supportsPromptCacheKey: true` and `cacheRetention: "long"`, regressing prefix-cache hits that worked in v2026.3.2.

## Best Possible Solution

Verify the actual v2026.5.7 Docker/runtime request path with cache-trace or redacted payload diagnostics, then fix the narrow layer that loses `compat.supportsPromptCacheKey`, `sessionId`, `cacheRetention`, or the built payload before it reaches oMLX.

No. The reporter gives a detailed same-config Docker/oMLX reproduction path, but I did not establish a high-confidence current-main reproduction and both current main and v2026.5.7 source emit the field under the documented gates.

## Solution Assessment
