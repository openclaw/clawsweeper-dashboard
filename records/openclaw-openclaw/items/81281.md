## Summary

Keep open: the builder branch exists, but current main still has a source-level runtime path where embedded boundary-aware `openai-completions` transports do not receive the run `sessionId`, so the cache-gated `prompt_cache_key` can be omitted exactly as reported.

## What This Changes

The issue reports a 2026.5.x regression where `openai-completions` requests no longer include `prompt_cache_key` despite `compat.supportsPromptCacheKey` and `cacheRetention` being configured, breaking prefix-cache hits that worked in 2026.3.2.

## Best Possible Solution

Propagate the embedded run `sessionId` into wrapped boundary-aware OpenAI-compatible transports, preserve the existing builder gates, and add a regression that proves an opted-in `openai-completions` embedded run reaches the payload builder with a session id.

## Reproduction Assessment

Yes at source level, but not by a live Docker run: the current wrapper path can call `buildOpenAICompletionsParams` without `options.sessionId`, and that builder then correctly omits `prompt_cache_key`. The reporter's release comparison supplies the real-world symptom for that path.

## Solution Assessment

Yes, the narrow maintainable fix is to carry the already-known run session id into the existing wrapped stream options. Reopening the builder-only PR direction is not enough because the builder already works when `sessionId` is present.
