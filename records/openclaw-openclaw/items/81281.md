## Summary

Keep open: current main still has a source-level cache-retention propagation gap for non-Anthropic/non-Google OpenAI-compatible completions providers, and an open PR with closing syntax is already the active fix path. The transport-level `prompt_cache_key` branch exists, so the remaining work is to resolve the linked PR and prove the packaged Docker/oMLX request path end to end.

## What This Changes

The issue reports that OpenAI-compatible completions providers stopped sending effective prompt-cache fields in 2026.5.x despite `compat.supportsPromptCacheKey: true` and `cacheRetention: "long"`.

## Best Possible Solution

Review and refine the linked fix PR so explicit `cacheRetention` reaches opted-in OpenAI-compatible completions providers without widening excluded provider families, then require a redacted Docker/oMLX payload trace before closing this issue.

## Reproduction Assessment

Source-level yes for the cache-retention propagation gap: current main drops explicit retention before non-Anthropic/non-Google openai-completions calls reach the transport wrapper. No high-confidence live reproduction of the exact Docker/oMLX `prompt_cache_key` absence was established in this read-only pass.

## Solution Assessment

No, current main is not the complete solution. The best path is to review and land a narrow version of the linked PR, with its Bedrock/provider-family guard preserved and a real outgoing-payload trace added.
