## Summary

Keep open: the builder gate is present, but current main still appears to drop the embedded run `sessionId` before the boundary-aware OpenAI Completions transport, so the reported missing `prompt_cache_key` has a plausible source-level root cause that the closed duplicate PR did not address.

## What This Changes

The issue reports that `openai-completions` requests in 2026.5.x omit `prompt_cache_key` despite `compat.supportsPromptCacheKey: true` and `cacheRetention: "long"`, breaking backend prefix-cache hits.

## Best Possible Solution

Propagate the embedded run `sessionId` through the boundary-aware stream wrapper, add regression coverage that an OpenAI-compatible completions fallback receives it, then verify the real payload includes `prompt_cache_key`.

## Reproduction Assessment

Yes by source inspection: the boundary-aware embedded transport path omits `sessionId`, and the completions builder suppresses `prompt_cache_key` when `options.sessionId` is absent. A live Docker/oMLX trace is still needed for end-to-end proof.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing wrapper contract and pass `sessionId` into the boundary-aware stream path, rather than changing the prompt-cache compat gate or adding new config.
