## Summary

Keep this PR open but not merge-ready: it targets a plausible stale Telegram image-ref failure, but the current diff disables existing prompt-local image refs for every embedded attempt and only supplies helper tests, not real Telegram/Azure behavior proof.

## What This Changes

The PR adds a prompt-image loading gate and declared attachment-block parsing, passes that gate as disabled from the embedded runner, reclassifies no-delivery external aborts as non-deliverable errors, and adds focused regression tests.

## Best Possible Solution

Preserve prompt-local image injection for explicit current-turn refs, suppress only refs proven to be replayed history, and verify both the Telegram/Azure stale-ref scenario and an explicit image prompt regression.

## Reproduction Assessment

Yes for the introduced regression by source inspection: the PR passes `includePromptRefImages: false` for all embedded attempts while current main loads prompt refs from the current prompt. The original Telegram/Azure failure is plausible but not proven with real runtime evidence in the PR.

## Solution Assessment

No: the stale-ref direction is reasonable, but disabling every prompt ref at the embedded runner boundary is too broad. The maintainable fix should distinguish current-turn image refs from replayed history and keep the documented prompt-local image path intact.
