## Summary

## What This Changes

The issue reports Windows node start/stop/restart disrupting WSL2 Gateway connectivity by treating the node's remote Gateway port as a local listener during Scheduled Task cleanup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85597, then close this issue after merge if no narrower follow-up remains.

## Reproduction Assessment

Yes, from source inspection: node services carry `node run --port` through the shared Windows Scheduled Task service, and current stop/restart cleanup treats that port as a Gateway listener port to reclaim. I did not run a live Windows/WSL2 VM reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to scope Gateway listener cleanup to Gateway services and match the installed node command for node lifecycle cleanup, which is the direction of the open linked PR rather than adding a WSL-specific workaround.
