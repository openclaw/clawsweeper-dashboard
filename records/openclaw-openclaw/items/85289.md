## What This Changes

The issue reports that Windows node service start/stop/restart can disrupt reachability to a WSL2-hosted Gateway by treating the node's remote Gateway port as local service-owned state.

## Best Possible Solution

Land or explicitly close the existing fix PR after maintainer review; when it merges, close this issue as implemented and keep both node-service and Gateway-service cleanup covered by regression tests.

## Reproduction Assessment

Yes. Current main source shows node service stop/restart reaches shared Scheduled Task Gateway port cleanup, parses the node `--port` as a local listener port, and can terminate or wait on unrelated busy listeners; I did not run a live Windows/WSL2 VM repro.

## Solution Assessment

Yes. The open PR's direction of scoping listener cleanup by existing service-kind metadata and matching the installed node command is narrower than adding WSL-specific PID filters or disabling Gateway cleanup globally.
