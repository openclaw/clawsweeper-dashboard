## Label Justifications

- `P1`: The report describes a current shipped Windows node lifecycle command breaking Gateway reachability for a real WSL2 setup, and source inspection confirms a plausible availability bug.

## Summary

Keep open: the reporter's follow-up overstates the verified-PID step, but current and shipped code still lets Windows node service lifecycle commands treat the node's remote Gateway port as a local Gateway listener port and then kill or wait on unrelated listeners.

## What This Changes

The issue asks for Windows `openclaw node start/stop/restart` to manage only the node service without disrupting a WSL2-hosted Gateway reached through localhost forwarding.

## Best Possible Solution

Use service-kind metadata to skip Gateway port reclaim/wait logic for node services while retaining it for Gateway services, with regression coverage for node start/stop/restart and existing Gateway cleanup.

## Reproduction Assessment

Yes from source inspection, though not by live VM execution. A Windows node service command carries `node run --port 18789`, and current Scheduled Task stop/restart/start-through-restart paths treat that port as a Gateway listener port to reclaim.

## Solution Assessment

Yes. The narrow maintainable fix is to scope Gateway listener cleanup to Gateway services using existing service-kind metadata, not to add a user-facing option or WSL-specific process-name filter.
