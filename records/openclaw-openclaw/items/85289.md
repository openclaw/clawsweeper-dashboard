## What This Changes

The issue reports that Windows node service start/stop/restart can disrupt WSL2 Gateway loopback connectivity by treating the node's remote Gateway port as a local Gateway listener to reclaim.

## Best Possible Solution

Ship a service-kind-aware daemon fix that skips Gateway listener reclaim and port-release waits for node services while preserving Gateway service cleanup and focused Windows Scheduled Task regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live WSL2 run. Current main lets node service stop/restart carry a `node run --port` Gateway connection port into shared Scheduled Task Gateway cleanup that can kill or wait on unrelated listeners on that port.

## Solution Assessment

Yes. The narrow maintainable path is to use existing service-kind metadata to avoid Gateway listener cleanup for node services, not to add WSL-specific PID-name filtering or a user-facing option.
