## Summary

Keep open: current main still has the source-reproducible Windows node service cleanup bug, and an open fixing PR with closing syntax is already paired with this issue, so this should remain open until that PR lands or is closed.

## What This Changes

The issue reports that Windows `openclaw node start/stop/restart` can disrupt WSL2 Gateway localhost connectivity by treating the node's remote Gateway port as a local listener to reclaim.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/85597; the desired end state is that Windows node services never reclaim or wait on the remote Gateway port while Gateway service cleanup remains intact.

## Reproduction Assessment

Yes from source inspection, though not by live VM execution in this review. Current main lets a Windows node service command carry `node run --port 18789` into Scheduled Task cleanup that treats that port as a Gateway listener to reclaim.

## Solution Assessment

Yes. The narrow maintainable solution is to use existing service-kind/installed-command context so node services skip Gateway listener cleanup while Gateway services retain their cleanup behavior.
