## What This Changes

Windows `openclaw node start/stop/restart` should not disrupt a WSL2-hosted Gateway reachable through the Windows loopback proxy.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85597 if maintainers accept its validation, then close this issue through that merged fix.

## Reproduction Assessment

Yes from source inspection, though not by live VM execution. A Windows node service can carry `node run --port 18789`, and current Scheduled Task stop/restart paths treat that as a Gateway listener port to reclaim.

## Solution Assessment

Yes. The narrow maintainable fix is to scope Gateway listener cleanup to Gateway services and use installed node command matching for node services, as the linked fix PR does.
