## Label Justifications

- `P1`: The report describes a current Windows node lifecycle command breaking Gateway reachability for an installed WSL2 setup, and source inspection shows a plausible shipped availability bug.
- `impact:crash-loop`: The affected behavior can make the Gateway unreachable from the Windows host and require a WSL2 restart, which is a process availability failure.

## Summary

Keep open: current and shipped Windows Scheduled Task code appears to reuse gateway listener cleanup for node services, so node stop/restart can treat the node's gateway connection port as a local listener and disturb whatever owns that port.

## What This Changes

The issue reports that Windows `openclaw node start/stop/restart` should manage only the node host service without disrupting a WSL2 Gateway reached through `127.0.0.1:18789`.

## Best Possible Solution

Gate Windows Scheduled Task port-reclaim logic to real Gateway services only, so Node service lifecycle never treats its gateway connection `--port` as a local listener to kill or wait on, and add regression coverage for both service kinds.

## Reproduction Assessment

Yes from source inspection, though not by live VM execution: a Windows node task command includes `node run --port 18789`, and current stop/restart cleanup treats that port as a Gateway listener to reclaim.

## Solution Assessment

Yes: the narrow maintainable fix is to separate Gateway-owned listener cleanup from Node service lifecycle, using the existing service-kind metadata rather than adding a user-facing option.
