## Summary

Keep open: current main and v2026.5.20 still route Windows node service lifecycle through shared Scheduled Task Gateway cleanup, so a node service can treat its remote Gateway connection port as a local listener to reclaim.

## What This Changes

The issue reports `openclaw node start/stop/restart` on a Windows host disrupting WSL2 Gateway reachability through localhost forwarding.

## Best Possible Solution

Use service-kind metadata to skip Gateway listener reclaim and port-release waits for node services while keeping the existing Gateway cleanup path and adding focused regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than live VM execution: a Windows node service stores `node run --port 18789`, and current stop/restart/start-through-restart paths treat that port as a Gateway listener port to reclaim.

## Solution Assessment

Yes. The narrow maintainable fix is to scope Gateway listener cleanup to Gateway services using existing service-kind metadata, not to add WSL-specific process-name filtering or a user-facing option.
