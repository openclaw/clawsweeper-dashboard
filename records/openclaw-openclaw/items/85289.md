## Summary

Keep open: current `main` still lets Windows node service lifecycle code interpret the node's remote Gateway connection port as a local Gateway listener port, so the reported WSL2 localhost-proxy disruption remains a plausible shipped availability bug.

## What This Changes

The issue reports that Windows `openclaw node start`, `stop`, or `restart` can disrupt WSL2 Gateway reachability when the node connects to a Gateway exposed through WSL2 loopback forwarding.

## Best Possible Solution

Scope Gateway listener reclaim, port-release waiting, and force-kill cleanup to actual Gateway services using service-kind metadata, while leaving node service start/stop/restart to manage only the node service process.

## Reproduction Assessment

Yes from source inspection, though not by live VM execution. A Windows node service command can carry `node run --port 18789`, and current Scheduled Task stop/restart paths treat that port as a Gateway listener port to reclaim.

## Solution Assessment

Yes. The narrow maintainable repair is to use existing service-kind metadata to skip Gateway port cleanup for node services, not to add WSL-specific PID filters or a new user-facing option.
