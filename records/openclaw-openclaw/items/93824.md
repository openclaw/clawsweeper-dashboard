## Summary

Keep open: current main and latest release still generate plain Telegram `<blockquote>` for Markdown blockquotes, while explicit raw `<blockquote expandable>` HTML is already preserved. The remaining work is the maintainer policy for generated collapsible quote authoring, not a cleanup close candidate.

## What This Changes

The issue requests generated Telegram rich-message blockquotes to emit the `expandable` attribute through a heuristic or explicit authoring signal.

## Best Possible Solution

Choose one maintainer-approved Telegram authoring policy for collapsible quote content, then implement that policy in the Telegram plugin with focused formatter/send coverage and live Telegram proof while preserving normal Markdown blockquotes by default unless maintainers explicitly choose otherwise.

## Reproduction Assessment

Yes, at source level: Markdown blockquotes flow through `renderTelegramHtml` and emit plain `<blockquote>`, while explicit raw `<blockquote expandable>` is preserved. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the authoring policy. The one-line universal default change is not the best fix because it changes all generated blockquotes; a safer path is an approved explicit marker, documented rich HTML path, or approved heuristic with tests and Telegram proof.
