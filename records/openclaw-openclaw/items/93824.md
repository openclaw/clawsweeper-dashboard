## Summary

Keep open: current main and v2026.6.8 still render generated Telegram Markdown blockquotes as plain `<blockquote>`, while only explicit raw `<blockquote expandable>` HTML is preserved; the remaining work needs maintainer policy on when generated quotes should become collapsible.

## What This Changes

The issue asks for Telegram rich-message Markdown blockquotes to emit an expandable/collapsible representation instead of always rendering as plain expanded blockquotes.

## Best Possible Solution

Maintainers should choose one explicit authoring policy for collapsible Telegram content, then add the narrow formatter/send-path implementation and live Telegram proof for that chosen behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Markdown blockquotes flow through `markdownToTelegramRichHtml` and the current renderer emits plain `<blockquote>`, while tests and v2026.6.8 source lock that output; I did not run a live Telegram send in this read-only review.

## Solution Assessment

No for the closed universal-emission fix. The source gap is real, but changing every generated quote to expandable is not the best default without maintainer approval; a chosen explicit marker, heuristic, or docs-backed `<details>` policy is safer.
