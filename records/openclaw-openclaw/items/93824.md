## Summary

Keep open: current main still preserves explicit `<blockquote expandable>` HTML but emits Markdown-generated blockquotes as plain `<blockquote>`, and the remaining generated-collapse behavior needs maintainer policy because current guidance points collapsible rich content to `<details>`.

## What This Changes

Request Telegram rich-message Markdown blockquotes to emit an expandable blockquote variant through a heuristic or explicit authoring signal.

## Best Possible Solution

Maintainers should choose one explicit authoring policy for collapsible Telegram blockquotes, then implement only that narrow path with formatter coverage, docs or prompt alignment, and Telegram Desktop proof.

## Reproduction Assessment

Yes, at source level: Markdown blockquotes flow through `markdownToTelegramRichHtml` and the shared renderer emits plain `<blockquote>`, while tests expect plain output. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Unclear as requested: the source gap is real, but choosing universal expansion, a length heuristic, or an explicit marker is a product/API decision. The safer direction is a maintainer-approved opt-in policy rather than changing every generated quote by default.
