## Summary

Keep open: current main still emits Markdown blockquotes as plain Telegram `<blockquote>` while only preserving explicit `<blockquote expandable>` HTML, and the remaining request needs maintainer direction because current guidance steers collapsible rich content to `<details>` instead of legacy expandable blockquotes.

## What This Changes

The issue requests that Telegram rich-message Markdown blockquotes gain an expandable/collapsible emission path instead of always rendering as static blockquotes.

## Best Possible Solution

Maintainers should choose one explicit authoring policy for collapsible Telegram content, likely `<details>` or a narrow opt-in marker, and only then add focused formatter/send-path coverage plus live Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Markdown blockquotes flow through `markdownToTelegramRichHtml` and the current renderer emits plain `<blockquote>`, while tests lock that output. I did not run a live Telegram repro because this read-only review did not execute runtime tests or channel sends.

## Solution Assessment

Unclear as requested: the source gap is real, but a length heuristic or new marker is a product/API choice, and current OpenClaw guidance already tells rich-text agents to use `<details>` for collapsible content. The best fix is not automation choosing a heuristic; it is maintainer approval of one explicit authoring surface and then a narrow implementation.
