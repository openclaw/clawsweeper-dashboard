## Summary

Keep open: current main and the latest release still emit plain generated Telegram blockquotes, while merged work only preserves explicit raw `<blockquote expandable>` HTML. The remaining work is a maintainer policy choice for the generated authoring path, not a conservative cleanup close.

## What This Changes

The issue requests Telegram rich-message/Markdown formatter support for emitting `<blockquote expandable>` from generated blockquotes so long quotes can collapse.

## Best Possible Solution

Choose one maintainer-approved Telegram authoring policy for collapsible quote content, then implement that policy with focused formatter/send coverage and live Telegram proof while preserving normal Markdown blockquotes unless maintainers explicitly choose otherwise.

## Reproduction Assessment

Yes, at source level. Current main routes Markdown rich messages through `renderTelegramHtml`, which emits plain `<blockquote>` for generated blockquote spans while preserving only explicit raw `<blockquote expandable>` HTML.

## Solution Assessment

Unclear until maintainers choose the authoring policy. The one-line global marker change is not the best fix because it changes all generated blockquotes; a safer path is an approved explicit marker, documented rich HTML/details path, or approved heuristic with tests and Telegram proof.
