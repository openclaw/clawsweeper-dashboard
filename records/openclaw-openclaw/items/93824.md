## Summary

Keep open: current main and the latest release still generate plain Telegram blockquotes from Markdown, while only preserving explicit user-supplied `<blockquote expandable>` HTML. The remaining work is a maintainer product decision about whether generated collapsible quotes should use `<details>`, an explicit marker, or a heuristic.

## What This Changes

Request that Telegram rich-message Markdown blockquotes emit collapsible `<blockquote expandable>` output via a heuristic or explicit authoring signal instead of only preserving user-supplied expandable HTML.

## Best Possible Solution

Maintainers should choose one explicit generated-collapsible policy for Telegram rich text, then implement that narrow path with formatter/send coverage and redacted Telegram proof while preserving normal blockquotes by default unless the policy says otherwise.

## Reproduction Assessment

Yes, source-level: Markdown blockquotes flow through the rich-message formatter and current main emits plain `<blockquote>` while tests lock that output. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Unclear as requested: the source gap is real, but choosing a length heuristic, marker, or `<details>`-only answer is a product policy decision. The best fix is not an automated unconditional formatter change.
