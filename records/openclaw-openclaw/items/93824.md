## Summary

Keep open: current main still emits generated Telegram Markdown blockquotes as plain `<blockquote>` and only preserves explicit `<blockquote expandable>` HTML. The merged related work solved raw HTML passthrough, while the generated-output proposals are closed unmerged and leave a maintainer policy choice.

## What This Changes

The issue requests a Telegram rich-message authoring path that emits expandable/collapsible blockquotes instead of always generating static blockquotes from Markdown quotes.

## Best Possible Solution

Keep the shipped explicit HTML passthrough, decide one maintainer-approved generated authoring policy, then implement that policy with focused formatter/send coverage, docs or prompt updates as needed, and live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main reproduction path: Markdown blockquotes still render through the shared Telegram formatter as plain `<blockquote>`, while explicit `<blockquote expandable>` HTML is only preserved when supplied directly.

## Solution Assessment

Unclear as requested. The source gap is real, but automatically emitting expandable blockquotes for all Markdown quotes is not the best settled fix because it changes existing quote semantics; maintainers should first choose an explicit marker, heuristic, or `<details>`-only policy.
