## Summary

Keep open: current main still has a source-backed skill SecretRef path where resolved skill `apiKey` values can enter `process.env`, while generic exec builds child environments from that sanitized parent environment without removing active skill keys.

## What This Changes

Prevent SecretRef-backed `skills.entries.<skill>.apiKey` values from being inherited by generic `exec` child processes.

## Best Possible Solution

Separate skill-owned secret delivery from generic exec inheritance so each skill can receive only its intended secrets while unrelated exec children get an active-skill-stripped environment, with regression tests and docs for both paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolved skill apiKeys are written to `process.env`, and generic exec builds child environments from that sanitized-but-not-skill-stripped parent env; I did not run a live `env` dump in this read-only review.

## Solution Assessment

Unclear as a one-line fix. Stripping active skill keys from generic exec would address the leak, but the current docs and tests intentionally inject skill env into the host run, so the safer fix needs a maintained skill-specific execution path or compatibility decision.
