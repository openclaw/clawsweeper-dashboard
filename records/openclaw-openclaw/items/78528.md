## Summary

Keep open: current main still has a distinct skill SecretRef path where resolved `skills.entries.<skill>.apiKey` values can be written into `process.env`, and generic host `exec` builds child environments from that sanitized environment without removing active skill keys. The related provider-key issue only covered `models.providers.*.apiKey`, not this skill-entry path, and the recent reporter follow-up plus assignment indicate active maintainer/security follow-up rather than cleanup closure.

## What This Changes

Ensure SecretRef-backed skill `apiKey` values are not exposed to generic `exec` child process environments while preserving intended skill execution.

## Best Possible Solution

Design a skill-scoped secret boundary where generic `exec` strips active skill-injected env keys, while skill-owned execution receives only the secrets it is entitled to, with regression coverage and docs for the boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: resolved skill apiKeys are written into `process.env`, and generic host `exec` builds its child env from that environment without active skill stripping; I did not run a live `env` command in this read-only review.

## Solution Assessment

Unclear as a one-line fix. Removing active skill keys from generic `exec` is likely necessary, but the maintainable solution needs owner/security judgment so skill-owned execution still receives its intended secrets without making them process-wide.
