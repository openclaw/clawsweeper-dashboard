## Summary

Keep open: current main still has a source-backed path where resolved `skills.entries.<skill>.apiKey` values are written to `process.env`, while generic host `exec` derives child environments from that process env without stripping active skill keys. The related provider-secret issue does not cover this skill-entry path, and the open skill-env work at https://github.com/openclaw/openclaw/pull/50960 pulls on the opposite product/security requirement rather than resolving this report.

## What This Changes

Prevent SecretRef-managed skill API keys from leaking into generic `exec` child process environments while preserving intentional skill env behavior.

## Best Possible Solution

Define and implement a scoped skill-secret execution boundary where generic `exec` does not inherit active skill SecretRef-derived keys, while skill-owned command paths can still receive their explicitly configured env values with tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: resolved skill apiKeys are written to `process.env`, and generic host `exec` builds child envs from that process env without stripping active skill keys; I did not run the live `env` command path in this read-only review.

## Solution Assessment

Unclear as a one-line fix. Stripping active skill keys from all generic exec children addresses the leak, but the maintainable solution must account for documented skill env injection and the open skill-env usability thread at https://github.com/openclaw/openclaw/issues/31583.
