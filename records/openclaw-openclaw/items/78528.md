## Summary

Keep open: current main still has a distinct skill SecretRef path where resolved `skills.entries.<skill>.apiKey` values can be written to `process.env`, and the generic exec spawn path sanitizes that environment without removing active skill keys. The related provider issue at https://github.com/openclaw/openclaw/issues/56441 does not cover this skill-entry path.

## What This Changes

Keep SecretRef-backed `skills.entries.<skill>.apiKey` values out of generic `exec` child process environments.

## Best Possible Solution

Add a maintained skill-scoped secret execution path so each skill can receive only its own configured secrets while generic exec receives a sanitized env with active skill SecretRef keys removed; cover both paths with regression tests and update the skills/security docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: resolved skill apiKeys are written to `process.env`, and generic exec builds child environments from that sanitized-but-not-skill-stripped process env. I did not run the live `env` command path in this read-only review.

## Solution Assessment

Unclear as a one-line fix: stripping active skill keys from all generic exec children would address the leak, but may break current prompt-based skills whose scripts rely on exec inheriting the configured primary env var. The maintainable fix should separate skill-owned command execution from generic exec inheritance.
