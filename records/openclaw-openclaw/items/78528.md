## Label Justifications

- `P1`: This is an urgent credential-boundary bug where skill-scoped secrets can be exposed to unrelated child processes spawned by agents.
- `impact:security`: The issue is about sensitive SecretRef-backed API keys crossing an unintended execution boundary.
- `impact:auth-provider`: The affected data is credential material configured through SecretRef-backed skill `apiKey` and env handling.

## Summary

Keep open: current main still has a source-backed path where resolved skill `apiKey` values are written into `process.env`, and generic host `exec` sanitizes that inherited env without stripping active skill keys. The open linked fix PR references this issue with closing syntax, so the issue should stay open until that PR lands or maintainers choose a replacement.

## What This Changes

The issue reports that SecretRef-backed `skills.entries.<skill>.apiKey` values leak from skill env injection into generic `exec` child process environments.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/82512 or a maintainer-approved replacement that strips active skill secrets from unrelated generic exec children while preserving intended skill-owned credential delivery.

## Reproduction Assessment

Yes, source-backed. Current main resolves skill `apiKey` into `process.env`, and generic host exec sanitizes inherited `process.env` without stripping active skill env keys.

## Solution Assessment

Unclear as a one-sided change. The best fix should define the scoped skill-secret boundary so intended skill commands still receive approved env credentials while unrelated generic exec children do not.
