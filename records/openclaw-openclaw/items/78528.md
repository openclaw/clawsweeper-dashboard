## Summary

Keep open: current main still has a distinct skill SecretRef path where resolved `skills.entries.<skill>.apiKey` values can be written into `process.env`, and generic `exec` builds child environments from that sanitized parent env without stripping active skill keys. The related provider-key report at https://github.com/openclaw/openclaw/issues/56441 covered a different provider config path and does not resolve this skill-entry leak.

## What This Changes

Prevent SecretRef-backed skill `apiKey` values from leaking into generic `exec` child process environments.

## Best Possible Solution

Add a maintained skill-scoped secret/env boundary so only skill-owned executions receive their resolved keys, while generic `exec` strips active skill-injected env keys and regression tests cover both flows.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: skill SecretRefs resolve into runtime config, `applySkillConfigEnvOverrides` writes the resolved primary env var to `process.env`, and generic `exec` passes a sanitized copy of that env to child processes without active skill-key stripping.

## Solution Assessment

Unclear as a one-line fix. Removing active skill keys from generic `exec` children addresses the leak, but the maintainable solution needs a skill-owned execution path or equivalent boundary so existing skill behavior is not accidentally broken.
