## Summary

Keep open: current main still has a source-visible path where resolved `skills.entries.<skill>.apiKey` values are written to `process.env`, and generic host `exec` sanitizes inherited env without stripping active skill keys. The related provider-secret issue is distinct, and the adjacent env-inheritance work in https://github.com/openclaw/openclaw/issues/31583 / https://github.com/openclaw/openclaw/pull/50960 means the fix needs maintainer security/design review rather than cleanup closure.

## What This Changes

The issue reports that SecretRef-managed skill `apiKey` values leak into generic `exec` child environments instead of staying skill-scoped.

## Best Possible Solution

Introduce a skill-scoped secret execution/env model so only the specific eligible skill invocation receives its configured secrets, while generic host `exec` strips active skill-secret keys and regression tests cover both paths.

## Reproduction Assessment

Yes, source-level: configure an active skill with a SecretRef-backed `apiKey` and a `primaryEnv`, then `applySkillEnvOverrides` writes the resolved value to `process.env` and host `exec` builds the child env from that process env. I did not run the live `env` command path in this read-only review.

## Solution Assessment

Unclear as a one-line sanitizer fix. Stripping active skill keys from every generic exec child addresses the leak, but the maintainable solution needs maintainer agreement on skill-scoped execution so eligible skill commands keep working without exposing secrets to unrelated exec calls.
