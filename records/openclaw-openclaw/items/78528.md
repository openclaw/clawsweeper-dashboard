## Label Justifications

- `P1`: A source-backed SecretRef boundary leak can expose skill-scoped credentials to unrelated agent-spawned subprocesses.
- `impact:security`: The issue is about credential isolation and preventing sensitive SecretRef values from leaking into generic child process environments.
- `impact:auth-provider`: The affected surface is `skills.entries.<skill>.apiKey` SecretRef resolution and auth/key delivery into runtime environments.

## Summary

Keep open: current main and the latest release still have a source-backed path where resolved skill `apiKey` SecretRefs are written into `process.env`, and generic host `exec` sanitizes that inherited process env without removing active skill keys. The linked fix PR is the right place to resolve this issue after review, but it is still open and currently marked conflicting.

## What This Changes

The issue reports that SecretRef-backed `skills.entries.<skill>.apiKey` values can be injected into the host process and inherited by generic `exec` child processes.

## Best Possible Solution

Keep this issue open until a maintainer-reviewed fix strips active skill SecretRef-derived keys from generic exec child environments while preserving or deliberately revising intended skill-owned env delivery.

## Reproduction Assessment

Yes, source-reproducible. Current main resolves active skill `apiKey` SecretRefs into `process.env`, and generic host `exec` builds child envs from that process env without an active-skill strip set; no live `env` dump was run in this read-only review.

## Solution Assessment

Unclear as a final shape. Stripping active skill keys from generic exec is the right mitigation direction, but maintainers still need to ensure it does not break intended skill-owned command env delivery or the related skill env inheritance work.
