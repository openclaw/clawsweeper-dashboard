## Summary

Keep open. Current main still exposes the requester session key to the A2A target prompt and runs reply/announce agent steps with `sourceTool: "sessions_send"`, without a scoped guard that prevents the target from calling `sessions_send` back to the requester.

## What This Changes

Prevent a target agent in an A2A `sessions_send` flow from calling `sessions_send` back to the requester and duplicating requester-channel messages.

## Best Possible Solution

Add an internal scoped tool-availability guard that hides or denies `sessions_send` during A2A target, reply, and announce runs while preserving inline reply delivery and the existing peer A2A lifecycle.

## Reproduction Assessment

Yes, by source inspection. Current main exposes the requester session key to the target prompt and has no scoped exclusion that removes `sessions_send` from target or nested A2A runs.

## Solution Assessment

Yes. A framework-level scoped guard is the narrowest maintainable fix; prompt-only guidance is unreliable, while private-delivery and dispatch-only modes are broader API work.
