## Summary

Keep open. Current main and the latest release still expose requester session identity to A2A target prompts, still run nested `sessions_send` reply/announce turns, and still have no scoped deny/stub preventing the target agent from calling `sessions_send` back to the requester.

## What This Changes

The issue reports duplicate requester-channel messages when an A2A target can call `sessions_send` back to the original requester during nested A2A handling.

## Best Possible Solution

Add a scoped internal `sessions_send` guard or error stub for A2A target/reply/announce nested runs, preserving inline result delivery and existing conversation semantics while returning a well-formed blocked-tool result if the target tries to reverse-call the requester.

## Reproduction Assessment

Yes for source-level reproduction: current main exposes the requester session key to the target prompt and runs nested A2A turns without a scoped `sessions_send` deny or stub. I did not run a live multi-agent channel scenario in this read-only review.

## Solution Assessment

Yes. A framework-level scoped capability guard is the narrowest maintainable fix; prompt-only instructions, hiding only `requesterSessionKey`, or adding dispatch-only mode would be brittle, incomplete, or broader product work.

Keep the fix bounded to `sessions_send` A2A target/reply/announce nested execution; do not add a public dispatch-only mode, do not remove the documented A2A conversation flow globally, and coordinate with but do not depend on https://github.com/openclaw/openclaw/pull/50720.
