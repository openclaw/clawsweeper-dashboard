## Label Justifications

- `P1`: The issue has multiple production confirmations and affects duplicate delivery in a core A2A messaging workflow.
- `impact:message-loss`: The reported failure can duplicate requester-channel messages through reverse `sessions_send` delivery and ping-pong reinjection.

## Summary

Keep open. Current main and v2026.5.20 still expose requester session context to A2A target prompts, run reply/announce nested turns with `sourceTool: "sessions_send"`, and construct `sessions_send` for normal non-embedded agent toolsets without a scoped reverse-call guard.

## What This Changes

The issue asks OpenClaw to prevent a target agent running under `sessions_send` A2A processing from calling `sessions_send` back to the requester and duplicating requester-channel output.

## Best Possible Solution

Add a scoped internal `sessions_send` deny/error stub for A2A target, reply, and announce nested runs, preserving inline result delivery and existing conversation semantics while blocking reverse calls back to the requester.

## Reproduction Assessment

Yes for source-level reproduction: current main exposes requester session identity to A2A target prompts and runs nested A2A turns without a scoped `sessions_send` deny or stub. I did not run a live multi-agent channel scenario in this read-only review.

## Solution Assessment

Yes. A framework-level scoped capability guard is the narrowest maintainable fix; prompt-only instructions, hiding only `requesterSessionKey`, or adding dispatch-only mode would be brittle, incomplete, or broader product work.
