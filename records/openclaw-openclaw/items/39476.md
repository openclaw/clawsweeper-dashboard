## Summary

Keep open. Current main still exposes the requester session in A2A prompt context and runs nested reply/announce turns through normal agent execution, so there is no source-level guard preventing a target agent from using `sessions_send` back to the requester.

## What This Changes

The issue asks OpenClaw to prevent A2A target sessions from calling `sessions_send` back to the requester and duplicating channel-visible replies.

## Best Possible Solution

Add a framework-level guard for nested A2A `sessions_send` processing so target/reply/announce turns cannot send back to the requester, while preserving normal explicit cross-session sends outside that nested flow.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls `sessions_send` to Agent B, current main exposes A's requester session in B's A2A context, and nested A2A turns run with the normal tool surface available.

## Solution Assessment

Yes. A scoped runtime guard or internal tool-policy/delivery-intent seam is the narrow maintainable fix; prompt-only instructions and the broader dispatch-only feature request do not close this reverse-send duplication path.
