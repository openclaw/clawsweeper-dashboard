## Summary

Keep open. Current main still exposes the requester session in A2A prompt context and runs nested reply/announce turns with normal agent tooling, so there is no framework guard preventing a target agent from calling `sessions_send` back to the requester.

## What This Changes

The issue asks OpenClaw to prevent A2A target agents from using `sessions_send` back to the requester and creating duplicate visible replies.

## Best Possible Solution

Add a framework-level guard for nested A2A `sessions_send` turns so target/reply/announce processing cannot call `sessions_send` back to the requester, while preserving ordinary explicit `sessions_send` outside nested A2A and existing parent-owned ACP/native subagent skips.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls `sessions_send` to Agent B, current main exposes A's requester session in B's A2A context and then runs nested reply/announce turns with normal tools available.

## Solution Assessment

Yes. A framework-level `sessions_send` denial or delivery-intent guard scoped to nested A2A processing is the narrowest maintainable direction; prompt-only instructions and the broader dispatch-only feature request do not close this reverse-send duplication path.
