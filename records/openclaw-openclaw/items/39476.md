## Summary

Keep open. Current main still exposes the requester session in A2A context and runs nested A2A reply/announce turns with normal tool availability, so there is no framework guard preventing a reverse `sessions_send`; the related fix PRs are closed unmerged and the newer related issues track adjacent delivery/dispatch semantics rather than replacing this bug.

## What This Changes

The issue asks for a guard that prevents a target agent from replying back with `sessions_send` during A2A processing, avoiding duplicate messages in the requester channel.

## Best Possible Solution

Add a framework-level nested-A2A tool guard, likely by carrying a scoped internal tool exclusion into target/reply/announce runs, so reverse `sessions_send` is blocked while existing conversation delivery and parent-owned skip behavior stay intact.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls `sessions_send`, Agent B receives A's requester session in A2A context, and current nested A2A runs still have normal tool availability rather than a scoped `sessions_send` denial.

## Solution Assessment

Yes. A scoped framework guard is the narrowest maintainable fix for this bug; prompt-only instructions and the broader dispatch-only feature request do not close the reverse-send duplication path.
