## Summary

Keep open. Current main still gives the A2A target the requester session key and still runs nested A2A reply/announce steps with `sourceTool: "sessions_send"` without a scoped denial of `sessions_send`; the closed broad PR is superseded, while the open private-delivery PR and dispatch-mode issue do not implement this narrow reverse-send guard.

## What This Changes

Prevent A2A target agents from calling `sessions_send` back to the requester and creating duplicate requester-channel messages.

## Best Possible Solution

Add an internal scoped tool-availability guard for A2A target and nested runs that blocks reverse `sessions_send` while preserving return-value delivery and the existing peer A2A lifecycle.

## Reproduction Assessment

Yes, by source inspection. Current main exposes the requester session key to the A2A target and has no scoped exclusion that removes `sessions_send` from the target or nested A2A tool surface.

## Solution Assessment

Yes. A framework-level scoped guard is the narrowest maintainable fix; prompt-only guidance is unreliable, and broader dispatch/private-delivery semantics belong to the separate open API discussions.
