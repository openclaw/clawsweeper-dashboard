## Summary

Keep open. Current main and the latest release still expose the A2A reverse-send path: the initial `sessions_send` call passes requester session context into the target run, and the nested reply/announce lifecycle still runs with `sourceTool: "sessions_send"` without a scoped guard that prevents the target from calling back.

## What This Changes

The issue asks OpenClaw to prevent an A2A target agent from calling `sessions_send` back to the requester and duplicating the reply already returned by the original tool call.

## Best Possible Solution

Add a scoped internal guard or deny/error stub for `sessions_send` during A2A target, reply, and announce nested runs while preserving normal top-level `sessions_send`, inline replies, and the existing parent-owned child-session skip.

## Reproduction Assessment

Yes for source-level reproduction: current main starts A2A runs with requester context and no scoped removal or denial of `sessions_send` from nested target processing. I did not run a live Discord or Telegram multi-agent setup.

## Solution Assessment

Yes. The narrowest maintainable fix is a framework-level scoped guard for `sessions_send` during nested A2A processing; prompt-only mitigation or hiding only the requester key would leave other reverse-send paths open.
