## Label Justifications

- `P1`: The issue has multiple production confirmations and affects duplicate delivery in a core A2A messaging workflow.
- `impact:message-loss`: The reported failure can duplicate requester-channel messages through reverse `sessions_send` delivery and ping-pong reinjection.
- `impact:session-state`: The bug depends on requester and target session identity crossing nested A2A turns and being reintroduced into session state.

## Summary

Keep open: current main and the latest release still expose requester session context to A2A target prompts, preserve the ping-pong/announce follow-up path, and do not scope-deny `sessions_send` inside nested A2A target runs.

## What This Changes

Requests a guard so an A2A target session cannot use `sessions_send` to call back to the requester and duplicate the response.

## Best Possible Solution

Add a scoped internal `sessions_send` deny/error stub for A2A target, reply, and announce nested runs while preserving inline result delivery and the documented conversation loop.

## Reproduction Assessment

Yes for source-level reproduction: current main exposes requester session identity to A2A target prompts and runs nested A2A turns without a scoped `sessions_send` deny or stub. I did not run a live multi-agent channel scenario in this read-only review.

## Solution Assessment

Yes. A framework-level scoped capability guard is the narrowest maintainable fix; prompt-only instructions, hiding only `requesterSessionKey`, or adding dispatch-only mode would be brittle, incomplete, or broader product work.
