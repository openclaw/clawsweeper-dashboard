## Summary

Keep open. Current main still exposes the requester session key to the A2A target and runs target/requester A2A steps without a scoped `sessions_send` denial, so the reverse-send duplicate path remains source-reproducible; the open dispatch-mode request is adjacent but does not replace this bug.

## What This Changes

Prevent A2A target sessions from using `sessions_send` back to the requester during `sessions_send` processing to avoid duplicate requester-channel delivery.

## Best Possible Solution

Add an internal A2A-scoped tool-availability guard that removes `sessions_send` from target and nested A2A turns while preserving the existing conversation-mode reply/announce lifecycle.

## Reproduction Assessment

Yes. Source inspection shows the target prompt receives `requesterSessionKey`, target/nested A2A runs use the normal tool surface that includes `sessions_send`, and no scoped deny is passed through the agent run path.

## Solution Assessment

Yes. A scoped framework-level tool guard is the narrow maintainable fix; removing `requesterSessionKey` alone reduces one path but does not prevent agent-id based reverse sends, and dispatch mode is a separate feature request.
