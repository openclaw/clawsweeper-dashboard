## Summary

Keep open. Current main still exposes the requester session key in the A2A target prompt and still runs nested A2A reply/announce turns with normal tool availability, so there is no framework guard preventing a target agent from calling `sessions_send` back to the requester.

## What This Changes

Prevent A2A target agents from using `sessions_send` back to the requester and causing duplicate requester-channel delivery.

## Best Possible Solution

Add a scoped framework-level guard that prevents reverse `sessions_send` during nested target A2A turns while preserving ordinary explicit `sessions_send`, existing conversation delivery, and parent-owned child skip behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: the target A2A prompt receives `requesterSessionKey`, and nested A2A target/reply turns still run without a scoped `sessions_send` denial.

## Solution Assessment

Yes. A narrow tool-availability guard is the best fix for this issue; prompt-only instructions and a broad dispatch-mode redesign do not reliably close the reverse-send duplication path.
