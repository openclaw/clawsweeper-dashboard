## Label Justifications

- `P1`: Multiple users report production duplicate delivery in a core A2A messaging workflow, and current source still exposes the failure path.
- `impact:message-loss`: The reported failure duplicates requester-channel messages through reverse `sessions_send` delivery and A2A reinjection.

## Summary

Keep open. Current main and v2026.5.20 still expose requester session context and run the A2A reply/announce lifecycle with `sourceTool: "sessions_send"`, while live issue context shows the prior fix PRs were closed unmerged and users confirmed the duplicate-delivery behavior after the original report.

## What This Changes

The issue asks OpenClaw to prevent an A2A target agent from using `sessions_send` back to the requester and duplicating requester-channel messages.

## Best Possible Solution

Add a scoped internal deny/error path for `sessions_send` during A2A target, reply, and announce nested runs while preserving inline result delivery and documented A2A conversation semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main constructs the A2A prompt with `requesterSessionKey`, executes nested reply/announce runs with `sourceTool: "sessions_send"`, and still exposes the general `sessions_send` tool in normal runs. I did not run a live multi-agent channel scenario.

## Solution Assessment

Yes. The narrowest maintainable fix is a framework-level scoped deny/stub for `sessions_send` during A2A target/reply/announce processing; prompt instructions, hiding only `requesterSessionKey`, or the separate dispatch-only feature would be incomplete or broader than this bug.
