## Label Justifications

- `P1`: Multiple users report production duplicate delivery in a core A2A messaging workflow, and current source still exposes the failure path.
- `impact:message-loss`: The reported failure duplicates requester-channel messages through reverse sessions_send delivery and A2A reinjection.

## Summary

Keep open. Current main and v2026.5.20 still expose requester session context and run the A2A reply/announce lifecycle without a scoped guard that removes or denies sessions_send in nested target runs; the closed fix PR was not merged, and adjacent private-delivery or dispatch proposals do not supersede this bug.

## What This Changes

The issue reports that an A2A target agent can call sessions_send back to the requester during nested processing, duplicating requester-channel messages.

## Best Possible Solution

Add a scoped internal sessions_send deny/error stub for A2A target, reply, and announce nested runs while preserving inline result delivery and documented A2A conversation semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main constructs the A2A prompt with requesterSessionKey and executes nested reply/announce runs with sourceTool sessions_send while the general tool factory still exposes sessions_send. I did not run a live multi-agent Discord or Telegram scenario.

## Solution Assessment

Yes. The narrowest maintainable fix is a framework-level scoped deny/stub for sessions_send during A2A target/reply/announce processing; prompt instructions, hiding only requesterSessionKey, or the separate dispatch-only feature would be incomplete or broader than this bug.
