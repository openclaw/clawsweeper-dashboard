## Label Justifications

- `P1`: The issue affects user-visible A2A message delivery in real multi-agent channel workflows.
- `impact:message-loss`: The failure mode is duplicate and misrouted supplemental message delivery from `sessions_send`.
- `impact:session-state`: The failure mode pollutes requester session state with A2A reply/announce messages.

## Summary

Keep open. Current main and the latest release still expose requester session identity to A2A target prompts and run nested reply/announce steps as `sessions_send`; the direct fix PRs are closed unmerged, and adjacent private-delivery or dispatch-mode items do not supersede this reverse-send duplicate-message bug.

## What This Changes

Prevent A2A `sessions_send` target agents from sending back to the requester and producing duplicate requester-channel messages.

## Best Possible Solution

Add an internal scoped tool guard that hides or denies `sessions_send` during A2A target, reply, and announce nested runs while preserving inline primary-reply delivery, control tokens, delivery metadata, and the documented A2A lifecycle.

## Reproduction Assessment

Yes for source-level reproduction: current main still exposes the requester session key in A2A prompts and runs nested A2A turns as `sessions_send` without a scoped deny guard. I did not run a live multi-agent channel reproduction in this read-only review.

## Solution Assessment

Yes. A framework-level scoped guard is the narrowest maintainable fix; prompt instructions, private delivery, dispatch mode, or graph loop detection are broader or less reliable alternatives for this specific bug.
