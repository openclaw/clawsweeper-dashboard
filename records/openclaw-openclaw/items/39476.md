## Summary

Keep open. Current main still exposes requester/target session context and runs nested A2A reply/announce turns with normal tooling, so no framework guard prevents reverse `sessions_send`; the related fix PRs were closed unmerged and #44309 tracks a broader dispatch-mode feature, not this bug.

## What This Changes

The issue reports duplicate requester-channel messages when an A2A target uses `sessions_send` back to the requester during an existing `sessions_send` flow.

## Best Possible Solution

Add a framework-level nested-A2A guard that blocks reverse `sessions_send` during target/reply/announce processing while preserving ordinary cross-session sends and the existing parent-owned subagent skips.
