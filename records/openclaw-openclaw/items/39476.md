## Summary

Keep open. Current `main` and v2026.5.12 still expose the requester session key to A2A target prompts and still run nested A2A reply/announce steps without a scoped denial of `sessions_send`; the closed fix attempt was not merged, and the open private-delivery PR does not replace this narrower reverse-send guard.

## What This Changes

The issue requests a core guard that prevents an A2A target agent from sending a reverse `sessions_send` back to the requester and duplicating user-visible messages.

## Best Possible Solution

Add an internal scoped tool-availability guard that hides or denies `sessions_send` during A2A target and nested reply/announce runs, while preserving inline reply delivery and the existing peer A2A lifecycle.

## Reproduction Assessment

Yes, by source inspection. Current `main` exposes the requester session key to the A2A target and has no scoped exclusion that removes `sessions_send` from target or nested A2A runs.

## Solution Assessment

Yes. A framework-level scoped guard is the narrowest maintainable fix; prompt-only guidance is unreliable, while private-delivery and dispatch-only modes are broader product/API work.
