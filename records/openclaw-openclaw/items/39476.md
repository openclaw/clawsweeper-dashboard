## Summary

Keep open. Current main still exposes requester session identity to A2A target prompts and still runs nested A2A reply/announce turns with `sourceTool: "sessions_send"`; I found no scoped deny/stub that blocks the target from calling `sessions_send` back to the requester.

## What This Changes

The issue asks OpenClaw to prevent A2A target agents from calling `sessions_send` back to the requester, which can duplicate requester-channel messages and reinject replies into the requester session.

## Best Possible Solution

Add a scoped internal `sessions_send` guard or stub for A2A target/reply/announce nested runs, preserving normal inline result delivery and documented A2A conversation semantics while returning a well-formed blocked-tool result if the target tries to reverse-call the requester.

## Reproduction Assessment

Yes for source-level reproduction: current main exposes the requester session key to the target prompt and runs nested A2A turns without a scoped `sessions_send` deny or stub. I did not run a live multi-agent channel scenario in this read-only review.

## Solution Assessment

Yes. A framework-level scoped capability guard is the narrowest maintainable fix; prompt-only instructions, hiding only `requesterSessionKey`, or adding dispatch-only mode would be either brittle, incomplete, or broader product work.
