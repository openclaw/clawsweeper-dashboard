## Summary

Keep open. Current main still gives the A2A target the requester session key and runs nested A2A requester/target steps without a scoped denial of `sessions_send`, so the reverse-send duplicate path remains source-reproducible.

## What This Changes

The issue asks for a framework-level guard so an A2A target handling `sessions_send` cannot call `sessions_send` back to the requester and duplicate the reply.

## Best Possible Solution

Add an internal scoped tool-availability guard for A2A target runs that blocks reverse `sessions_send`, while preserving existing conversation-mode reply/announce behavior unless a separate dispatch-mode change is approved.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: the target A2A prompt receives `requesterSessionKey`, and nested target/reply turns still run without a scoped `sessions_send` denial.

Yes. A narrow framework-level tool guard is the best fix for this bug; prompt-only instructions and broad dispatch-mode redesigns are either unreliable or larger than this report.
