## Summary

Keep open. Current main and v2026.5.22 still expose the reported A2A reverse-send path: target and follow-up runs receive requester context, nested A2A steps still use `sourceTool: "sessions_send"`, and the normal tool factory still includes `sessions_send` without a scoped nested-A2A guard.

## What This Changes

The issue reports duplicate requester-channel messages when an A2A `sessions_send` target can call `sessions_send` back to the requester during nested processing.

## Best Possible Solution

Add a scoped internal guard or deny/error stub for `sessions_send` during A2A target, reply, and announce nested runs while preserving normal top-level `sessions_send`, inline replies, thread-target rejection, and parent-owned child-session skips.

## Reproduction Assessment

Yes for source-level reproduction: current main starts A2A target/follow-up runs with requester context and no scoped removal or denial of `sessions_send` from nested A2A processing. I did not run a live multi-agent transport repro.

## Solution Assessment

Yes. The narrowest maintainable fix is a framework-level scoped guard for `sessions_send` during nested A2A processing; prompt-only mitigation or only hiding `requesterSessionKey` would leave other reverse-send paths open.
