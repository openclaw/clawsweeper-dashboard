## Summary

Keep open. Current main still exposes the requester session key in A2A prompts and does not apply a scoped `sessions_send` denial to nested A2A runs; the related exact fix PRs are closed unmerged and the open delivery/dispatch work is broader than this narrow duplicate-message guard.

## What This Changes

The issue asks OpenClaw to prevent A2A target sessions from using `sessions_send` back to the requester and causing duplicate requester-channel messages.

## Best Possible Solution

Add an internal scoped tool-availability guard that blocks `sessions_send` during A2A target/reply/announce runs while preserving inline return-value delivery and the existing conversation lifecycle.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: the target receives the requester session key in its A2A prompt, and nested A2A steps run with `sourceTool: "sessions_send"` without a scoped denial of the same tool.

## Solution Assessment

Yes. A framework-level scoped tool guard is the narrowest maintainable fix; prompt guidance and the broader dispatch/private-delivery API proposals do not reliably solve this exact duplicate-reply bug.
