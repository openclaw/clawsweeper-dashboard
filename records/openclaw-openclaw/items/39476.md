## Summary

Keep open. Current main and the latest release still expose the requester session in A2A context and still run `sessions_send` reply/announce steps without a scoped denial of `sessions_send`, while the linked private-delivery and dispatch-mode work do not replace this narrow reverse-send guard.

## What This Changes

The issue requests a core A2A guard that prevents a `sessions_send` target run from calling `sessions_send` back to the requester and duplicating channel output.

## Best Possible Solution

Add an internal scoped tool-availability guard for A2A target and nested runs that blocks reverse `sessions_send` while preserving inline reply delivery and the existing peer A2A lifecycle.

## Reproduction Assessment

Yes, by source inspection. Current main exposes the requester session key to the A2A target and has no scoped exclusion that removes `sessions_send` from target or nested A2A runs.

## Solution Assessment

Yes. A framework-level scoped guard is the narrowest maintainable fix; prompt-only guidance is unreliable, while private-delivery and dispatch-only modes are broader product/API work.
