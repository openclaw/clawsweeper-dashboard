## Summary

Keep open. Current main still exposes requester session context to nested A2A runs and runs reply/announce turns through normal agent tooling, so the reverse `sessions_send` duplicate-delivery path remains possible; the related fix PRs were closed unmerged and #44309 tracks a broader dispatch-only feature rather than this guard.

## What This Changes

Prevent nested A2A `sessions_send` target runs from calling `sessions_send` back to the requester and duplicating user-visible replies.

## Best Possible Solution

Add a framework-level nested A2A tool-policy guard so target/reply/announce runs cannot invoke `sessions_send` back to the requester, while preserving ordinary `sessions_send` behavior outside that flow.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls `sessions_send`, the target run receives requester session context, and nested A2A reply/announce turns run with the normal tool set still containing `sessions_send`.

## Solution Assessment

Yes. A scoped framework guard or delivery-intent deny for `sessions_send` during nested A2A is narrower and more reliable than prompt-only instructions, and it avoids bundling the broader dispatch-only feature request.
