## Summary

Keep this issue open. Current main still marks A2A target/follow-up turns as inter-session traffic from `sessions_send` while the target tool set still exposes `sessions_send`, so the reported reverse-call and duplicate-delivery path remains. The open closing PR at https://github.com/openclaw/openclaw/pull/91161 now owns the focused fix and should be reviewed rather than creating a second implementation.

## What This Changes

Prevent duplicate requester-channel messages when an A2A target reverses `sessions_send` back to the requester.

## Best Possible Solution

Land the linked provenance-scoped guard after proving that A2A target turns omit only `sessions_send`, ordinary turns restore it, unrelated inter-session sources remain unaffected, and a live cross-agent round trip produces exactly one requester-visible reply.

## Reproduction Assessment

Yes, at source level. A2A turns carry inter-session `sessions_send` provenance while current main still exposes the same tool to the target; this read-only review did not execute a live multi-agent channel reproduction.

## Solution Assessment

Yes. A per-turn provenance guard is the narrowest reliable solution because it removes only the reverse-call vector while preserving requester context, other session tools, unrelated inter-session flows, and normal-turn behavior.
