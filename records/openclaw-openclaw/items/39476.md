## Summary

Keep open. Current main still has the A2A reverse-send path: requester session details are exposed to the target prompt, nested A2A turns run as normal agent runs, and the normal tool construction still includes `sessions_send`, so there is no framework guard against a target calling back and causing duplicate user-facing delivery.

## What This Changes

The issue asks OpenClaw to prevent a target agent in an A2A `sessions_send` flow from calling `sessions_send` back to the requester and duplicating the response.

## Best Possible Solution

Add a framework-level guard scoped to nested A2A processing so target/reply/announce runs cannot call `sessions_send` back to the requester, while preserving normal explicit `sessions_send` and documented peer follow-up behavior outside that nested flow.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls `sessions_send` to Agent B, current main exposes A's requester session in B's A2A context and runs B with the normal tool list that still includes `sessions_send`.

## Solution Assessment

Yes. A narrowly scoped framework guard or internal per-run tool denial for nested A2A is the best direction; prompt-only workarounds are brittle, and the broader dispatch-only feature request does not close this reverse-send duplication path.
