## Summary

Keep open. Current main and latest release still expose the requester session key in A2A context and run nested A2A requester/target/announce steps without any scoped `sessions_send` denial, while the exact prior fix PR is closed unmerged and the open private-delivery work is broader and blocked.

## What This Changes

The issue requests preventing A2A target agents from calling `sessions_send` back to the requester and creating duplicate channel messages.

## Best Possible Solution

Add an internal scoped tool-availability guard for A2A nested runs that blocks reverse `sessions_send` while preserving return-value delivery and the existing conversation/announce lifecycle.

## Reproduction Assessment

Yes, by source inspection. The target receives the requester session key in A2A prompt context, and nested A2A runs carry `sourceTool: "sessions_send"` without any scoped exclusion of the `sessions_send` tool.

## Solution Assessment

Yes. A framework-level scoped tool guard is the narrowest maintainable fix for this bug; prompt-only guidance is unreliable, and public dispatch/private-delivery semantics belong to the separate broader work.
