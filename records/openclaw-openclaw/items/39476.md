## Label Justifications

- `P1`: The issue has multiple production confirmations and can duplicate cross-agent channel output in a core A2A messaging workflow.
- `impact:message-loss`: The reported failure duplicates or misroutes user-visible channel messages through reverse `sessions_send` delivery.
- `impact:session-state`: The bug is rooted in requester and target session context being exposed and reinjected across nested A2A turns.

## Summary

Keep open. Current `main` and the latest release still expose requester session identity to A2A target prompts and still run nested A2A reply/announce steps with `sourceTool: "sessions_send"`; I found no scoped deny/stub that would block the target from calling `sessions_send` back to the requester.

## What This Changes

The issue requests a framework-level guard so an A2A target cannot send a reverse `sessions_send` back to the requester and duplicate delivery in the requester channel.

## Best Possible Solution

Add an internal scoped tool guard, preferably a structured `sessions_send` block/stub that returns a well-formed tool error during A2A target/reply/announce nested runs, while preserving inline primary-reply delivery and documented A2A conversation semantics.

## Reproduction Assessment

Yes for source-level reproduction: current `main` exposes the requester session key in A2A prompts and runs nested A2A target/reply/announce turns without a scoped `sessions_send` deny or stub. I did not run a live multi-agent channel reproduction in this read-only review.

## Solution Assessment

Yes. A framework-level scoped capability guard is the narrowest maintainable fix; prompt instructions, hiding only `requesterSessionKey`, or adding dispatch-only mode are either brittle, incomplete, or broader product work.
