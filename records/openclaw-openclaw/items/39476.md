## Summary

Keep open. Current main and v2026.5.22 still expose the A2A reverse-send path: the target run gets requester session context, nested reply/announce steps run with `sourceTool: "sessions_send"`, and the normal tool factory still includes `sessions_send` without a scoped A2A guard.

## What This Changes

The issue asks OpenClaw to prevent A2A target agents from calling `sessions_send` back to the requester and duplicating requester-channel messages.

## Best Possible Solution

Add a scoped internal guard or deny/error stub for `sessions_send` during A2A target, reply, and announce nested runs while preserving normal top-level `sessions_send`, inline replies, thread-target rejection, and parent-owned child-session skips.

## Reproduction Assessment

Yes for source-level reproduction: current main starts A2A runs with requester context and no scoped removal or denial of `sessions_send` from nested target processing. I did not run a live Discord or Telegram multi-agent setup.

## Solution Assessment

Yes. The narrowest maintainable fix is a framework-level scoped guard for `sessions_send` during nested A2A processing; prompt-only mitigation or hiding only the requester key would leave other reverse-send paths open.
