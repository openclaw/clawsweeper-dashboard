## What This Changes

The issue reports duplicate A2A `sessions_send` delivery when the target agent can call `sessions_send` back to the requester during an inter-agent turn.

## Best Possible Solution

Land the linked provenance-scoped guard after proving that A2A target turns omit only `sessions_send`, ordinary turns restore it, unrelated inter-session sources remain unaffected, and a real cross-agent round trip produces one requester-visible reply.

## Reproduction Assessment

Yes, at source level: current A2A target turns carry `sourceTool: "sessions_send"` and requester provenance, while current tool construction still includes `sessions_send` for the target turn. I did not run a live multi-agent Discord or webchat reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR's per-turn provenance guard is the best narrow fix for the reverse-call vector because it removes only `sessions_send` during inter-agent `sessions_send` target processing. Broader one-way dispatch or waited-reply announce semantics should remain tracked by the separate related issues.
