## Summary

Keep open. Current main still matches the reported failure shape: A2A prompts expose the requester session, the follow-up and announce phases run as normal agent turns, and there is no per-run tool deny/allow contract that would stop nested A2A turns from invoking sessions_send back to the requester. Related #44309 tracks broader dispatch-only semantics, but it does not supersede this reverse-send guard.

## What This Changes

The issue requests a core A2A sessions_send guard so target A2A processing cannot call sessions_send back to the requester and duplicate visible channel replies.

## Best Possible Solution

Add a framework-level delivery intent or per-run tool policy scoped to nested A2A sessions_send turns, so target/reply/announce processing cannot call sessions_send back to the requester. Reduce requester session-key prompt exposure where possible, keep normal explicit sessions_send outside nested A2A working, preserve the owned ACP child skip, and add regression coverage for duplicate visible delivery.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: Agent A calls sessions_send to Agent B, current main exposes A's requester session in B's A2A context and then runs nested reply/announce turns with normal tools available; the existing sessions_send tests assert that ping-pong and announce path still runs.

## Solution Assessment

Yes. The issue's Option A direction, a framework-level sessions_send denial or delivery-intent guard for nested A2A processing, is the narrowest maintainable fix; prompt-only wording or relying on agents not to call back is not reliable enough.
