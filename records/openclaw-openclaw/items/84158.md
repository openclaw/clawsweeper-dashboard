## Label Justifications

- `P1`: A released Telegram isolated-ingress recovery path can still leave a real channel update stuck in processing state and delay user-visible Telegram handling.
- `impact:message-loss`: The affected update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.
- `impact:session-state`: The issue is about stale Telegram ingress spool state and handler lifecycle drift after a hung turn.

## Summary

Keep open: current main and the latest shipped tag still feed timeout recovery only from backlog-blocked lanes, so a lone active `.json.processing` claim can miss recovery when no later same-lane update exists. The closed follow-up PR had the right narrow shape but was closed unmerged because live Telegram proof was not available.

## What This Changes

Recover a Telegram isolated-ingress spooled handler that times out even when it is the only active `.json.processing` claim and no later same-lane update is queued.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handlers plus `blockedByLane`, add a focused lone `.json.processing` regression test, and prove the fix with targeted tests plus live Telegram/Crabbox proof when available.

## Reproduction Assessment

Yes, by source inspection rather than an executed test: claim one spooled Telegram topic update, let `handleUpdate` hang, leave no later same-lane `.json` update, advance past `spooledUpdateHandlerTimeoutMs`, and recovery receives an empty `blockedByLane` set.

## Solution Assessment

Yes; unioning active same-spool handler keys with `blockedByLane` is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart path.
