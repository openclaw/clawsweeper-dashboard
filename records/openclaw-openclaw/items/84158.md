## Summary

Keep open: current `main` and the latest release still route timed-out Telegram spooled handler recovery through backlog-derived `blockedByLane`, so a lone active `.json.processing` claim with no later same-lane update is still missed. The exact fix was sketched in closed unmerged https://github.com/openclaw/openclaw/pull/84194, while https://github.com/openclaw/openclaw/issues/84886 tracks a separate message-level idempotency gap.

## What This Changes

The issue asks Telegram isolated-ingress timeout recovery to include lone active spooled handlers even when no pending same-lane backlog exists.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from all active handlers for the same spool plus `drain.blockedByLane`, keep the existing tombstone/restart path, and add a regression test for a single claimed update with no later backlog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: claim one spooled Telegram update into `.json.processing`, leave no later same-lane pending `.json`, advance past `spooledUpdateHandlerTimeoutMs`, and timeout recovery receives an empty `blockedByLane`. I did not execute the test because this review was read-only.

## Solution Assessment

Yes, the suggested active-same-spool handler set unioned with `drain.blockedByLane` is the narrow maintainable fix because it reuses the existing fail-tombstone, reply-fence abort, and isolated-ingress restart behavior. The durable message idempotency work belongs in https://github.com/openclaw/openclaw/issues/84886, not this fix.
