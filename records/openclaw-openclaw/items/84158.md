## Summary

Keep open: current main and the latest release still route Telegram isolated-ingress timeout recovery through backlog-blocked lanes only, so the reported lone active `.json.processing` handler remains a valid narrow bug.

## What This Changes

The issue requests Telegram isolated-ingress timeout recovery for a single active spooled handler when no later same-lane update is queued.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handlers plus `drain.blockedByLane`, then add focused regression coverage for the no-backlog lone-handler timeout path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: claim one spooled Telegram topic update, leave no later same-lane pending `.json`, advance beyond `spooledUpdateHandlerTimeoutMs`, and timeout recovery receives an empty `blockedByLane` set.

## Solution Assessment

Yes. The suggested active-handler union is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart behavior instead of adding a second recovery path.
