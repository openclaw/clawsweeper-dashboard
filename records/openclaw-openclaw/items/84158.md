- `impact:message-loss`: The affected Telegram update can remain claimed instead of being delivered, retried, or failed cleanly.

## Summary

Keep open: current main and the latest release still route Telegram isolated-ingress handler timeout recovery through the backlog-blocked set, so a lone active claimed update with no later same-lane backlog is still not evaluated for timeout recovery.

## What This Changes

The issue asks for Telegram isolated-ingress timeout recovery to include lone active spooled handlers, not only handlers that block later same-lane backlog.

## Best Possible Solution

Patch Telegram polling-session timeout candidate selection to union active same-spool handler keys with `drain.blockedByLane`, then add focused no-backlog regression coverage and live/equivalent Telegram recovery proof before merge.

## Reproduction Assessment

Yes, by source inspection: start one spooled Telegram update whose handler never settles, leave no later same-lane pending update, advance past `spooledUpdateHandlerTimeoutMs`, and timeout recovery receives an empty `blockedByLane` candidate set. I did not execute that repro because this review was read-only.

## Solution Assessment

Yes, the suggested active-same-spool plus blocked-lane candidate union is the narrowest maintainable fix because it reuses the existing timeout tombstone, reply-fence abort, and restart path instead of adding a second recovery mechanism.
