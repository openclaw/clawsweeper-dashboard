- `impact:message-loss`: The affected Telegram update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.
- `impact:session-state`: The issue is about stale Telegram ingress spool state and active handler lifecycle drift after a hung turn.

## Summary

Keep open: current main still only evaluates Telegram spooled-handler timeout recovery from backlog-blocked lanes, and the linked open fix PR has not merged yet.

## What This Changes

Fix Telegram isolated ingress so a lone active `.json.processing` handler with no later same-lane backlog is still considered for timeout recovery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84194, or replace it with an equivalent narrow fix, then close this issue once the fix is on main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: claim one spooled Telegram update, leave no later same-lane pending update, advance past `spooledUpdateHandlerTimeoutMs`, and `drain.blockedByLane` remains empty so recovery is not invoked for that handler.

## Solution Assessment

Yes. Unioning active same-spool handler keys with `drain.blockedByLane` reuses the existing tombstone, reply-fence abort, and restart behavior without introducing a parallel recovery path.
