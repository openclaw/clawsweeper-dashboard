## Label Justifications

- `P1`: A released Telegram isolated-ingress recovery path can still leave real channel updates stuck in processing state and delay user-visible Telegram handling.
- `impact:message-loss`: The affected update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.
- `impact:session-state`: The issue is about stale Telegram ingress spool state and handler lifecycle drift after a hung turn.

## Summary

Keep open: current main still only sends backlog-blocked Telegram spool handlers into timeout recovery, so a lone active `.json.processing` claim without a later same-lane pending update can be missed.

## What This Changes

Recover Telegram isolated-ingress timeout handling for lone active spooled handlers when no pending same-lane backlog exists.

## Best Possible Solution

Update `polling-session.ts` to recover timed-out candidates from active same-spool handlers plus backlog-blocked handlers, add a no-backlog regression test, and keep message-level idempotency separate.

## Reproduction Assessment

Yes. Source inspection shows a single spooled update can be claimed into `.json.processing`, have its pending `.json` removed, and then reach timeout recovery with an empty `blockedByLane` set.

## Solution Assessment

Yes. Unioning active same-spool handler keys with `drain.blockedByLane` is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart path.
