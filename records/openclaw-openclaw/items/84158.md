- `impact:message-loss`: The affected update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.

## Summary

Keep open: current main still passes only `drain.blockedByLane` into timeout recovery, so a lone active `.json.processing` handler without a later same-lane pending update is not evaluated for timeout recovery.

## What This Changes

The issue reports that Telegram isolated-ingress timeout recovery misses a lone active spooled `.json.processing` handler when no same-lane backlog item exists.

## Best Possible Solution

Patch Telegram polling-session recovery to build timeout candidates from active handlers for the same spool plus `drain.blockedByLane`, then add focused regression coverage for the no-backlog lone-handler timeout path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start one spooled Telegram update whose handler never settles, leave no later same-lane update, advance past `spooledUpdateHandlerTimeoutMs`, and recovery receives an empty `blockedByLane`.

## Solution Assessment

Yes. The suggested union of active same-spool handler keys with `drain.blockedByLane` reuses the existing tombstone, reply-fence abort, and restart semantics without adding a new recovery policy.
