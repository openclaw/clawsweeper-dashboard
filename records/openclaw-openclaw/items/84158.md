- `impact:message-loss`: The affected update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.

## Summary

Keep open: current main still feeds Telegram isolated-ingress timeout recovery only from backlog-blocked lanes, so a lone active `.json.processing` claim with no later same-lane update can miss the timeout recovery path.

## What This Changes

Ensure Telegram isolated-ingress timeout recovery considers a lone active `.json.processing` handler even when no same-lane backlog exists.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handler keys plus `drain.blockedByLane`, add the lone-handler regression test, and keep the separate dispatch-idempotency work in https://github.com/openclaw/openclaw/issues/84886 out of scope.

## Reproduction Assessment

Yes, by source inspection: claim one spooled Telegram update, leave no later same-lane `.json` backlog, advance past `spooledUpdateHandlerTimeoutMs`, and current main calls timeout recovery with an empty `blockedByLane` set. I did not execute that repro because this review was read-only.

## Solution Assessment

Yes, the active-same-spool handler candidate set is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart behavior.
