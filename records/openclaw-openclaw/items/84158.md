## Summary

Keep open: current main and v2026.5.20 still pass only `drain.blockedByLane` into Telegram isolated-ingress timeout recovery, so a claimed lone `.json.processing` update is not evaluated. The closed fix PR at https://github.com/openclaw/openclaw/pull/84194 was not merged, and https://github.com/openclaw/openclaw/issues/84886 explicitly tracks a separate idempotency gap.

## What This Changes

Ensure Telegram isolated-ingress timeout recovery evaluates lone active spooled handlers even when no same-lane backlog exists.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handlers plus `drain.blockedByLane`, with focused regression coverage for a single claimed topic update and no later backlog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: claim one spooled Telegram update, remove the pending `.json`, leave no later same-lane update, advance past `spooledUpdateHandlerTimeoutMs`, and timeout recovery receives an empty `blockedByLane`. No live Telegram repro was run in this read-only review.

## Solution Assessment

Yes, the suggested active-handler plus `blockedByLane` candidate set is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and restart behavior while leaving durable message idempotency to the separate linked issue.
