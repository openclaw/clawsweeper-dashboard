## Summary

Keep open: current main still passes only backlog-blocked lane keys into Telegram spooled-handler timeout recovery, so the reported single active `.json.processing` claim with no later same-lane pending update remains a plausible missed watchdog path.

## What This Changes

The issue requests Telegram isolated-ingress timeout recovery for a single active spooled handler even when no later same-lane update is queued behind it.

## Best Possible Solution

Land a narrow Telegram polling fix that builds timeout candidates from active same-spool handlers unioned with `drain.blockedByLane`, with regression coverage for the no-backlog single-handler timeout path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: claim one spooled Telegram update whose handler never settles, leave no later same-lane pending update, advance past `spooledUpdateHandlerTimeoutMs`, and recovery receives no candidate from `blockedByLane`. I did not execute it because this review is read-only.

## Solution Assessment

Yes, the proposed union of active same-spool handler keys with `drain.blockedByLane` is the narrowest maintainable solution because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart behavior.
