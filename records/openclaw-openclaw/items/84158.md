## Summary

Keep open: current main still feeds timeout recovery only from backlog-blocked lanes, while a claimed lone `.json.processing` update has no pending `.json` backlog entry to put it in that set.

## What This Changes

The issue asks to extend Telegram isolated-ingress timeout recovery so a single active spooled handler with no later same-lane backlog can be failed to a tombstone and trigger ingress restart.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active handlers for the same spool plus `drain.blockedByLane`, then add a regression test for the no-backlog lone-handler timeout path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: start one spooled Telegram update whose handler never settles, leave no later same-lane update, advance past `spooledUpdateHandlerTimeoutMs`, and recovery receives an empty `blockedByLane`. I did not execute the test because this review was read-only.

## Solution Assessment

Yes, the suggested union of active same-spool handler keys with `drain.blockedByLane` is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and restart behavior.
