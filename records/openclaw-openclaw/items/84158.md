## Summary

Keep open: current main still feeds Telegram isolated-ingress timeout recovery only from backlog-blocked lanes, so a lone claimed `.json.processing` handler with no later same-lane update can avoid timeout recovery.

## What This Changes

The issue requests timeout recovery for a Telegram isolated-ingress spooled handler even when it is the only active claimed update and no later same-lane backlog exists.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handlers plus `drain.blockedByLane`, then add a no-backlog regression test and Telegram-focused validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start one spooled Telegram update whose handler never settles, leave no later same-lane update, advance past `spooledUpdateHandlerTimeoutMs`, and recovery receives an empty `blockedByLane`.

## Solution Assessment

Yes. The suggested union of active same-spool handler keys with `drain.blockedByLane` is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, and isolated-ingress restart behavior.
