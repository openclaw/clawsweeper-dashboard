## Summary

Keep open: current main still feeds timeout recovery only from `drain.blockedByLane`, and the direct fix attempt at https://github.com/openclaw/openclaw/pull/84194 was closed unmerged after live-proof access blocked the contributor. The linked https://github.com/openclaw/openclaw/issues/84886 is a separate message-id idempotency gap, not a replacement for this spool-claim recovery bug.

## What This Changes

The issue asks for Telegram isolated-ingress timeout recovery to include lone active spooled handlers even when no later same-lane update populates backlog-blocked lanes.

## Best Possible Solution

Patch `polling-session.ts` to build timeout candidates from active same-spool handler keys plus `drain.blockedByLane`, keep the existing tombstone/restart semantics, and track message-level dispatch idempotency separately in https://github.com/openclaw/openclaw/issues/84886.

## Reproduction Assessment

Yes by source inspection: start one spooled Telegram topic update whose handler never settles, leave no later same-lane update, advance past `spooledUpdateHandlerTimeoutMs`, and timeout recovery receives an empty `blockedByLane`. I did not execute the runtime repro because this review is read-only.

## Solution Assessment

Yes, the active same-spool handler key union is the narrowest maintainable fix because it reuses the existing timeout detection, tombstone, reply-fence abort, and restart path without adding a new config or policy surface.
