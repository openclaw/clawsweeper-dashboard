## Summary

Keep open: current main still routes Telegram isolated-ingress timeout recovery through backlog-blocked lanes only, and the open linked fix PR should resolve this issue only after it merges or is explicitly rejected.

## What This Changes

The issue reports a Telegram isolated-ingress bug where a lone claimed `.json.processing` update can miss timeout recovery when no same-lane backlog item exists.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84194; the desired end state is active same-spool handlers included in timeout candidates with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: claim one spooled Telegram update, remove its pending `.json`, let its handler exceed `spooledUpdateHandlerTimeoutMs`, and recovery receives no `blockedByLane` candidate.

## Solution Assessment

Yes. The narrow fix is to union active same-spool handler keys with `drain.blockedByLane` and reuse the existing tombstone/restart recovery path; the open linked PR follows that shape.
