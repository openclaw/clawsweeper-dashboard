## Summary

Keep open: current main and the latest release still use exact hidden-style predicates, so the reported near-invisible CSS cases remain a valid web_fetch prompt-injection hardening gap. The remaining work needs maintainer/security policy for color normalization and tiny-font thresholds before it is safe to automate.

## What This Changes

The issue asks for web_fetch HTML sanitization to remove near-invisible CSS payloads such as white or near-white text, tiny nonzero fonts, and matching foreground/background colors.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize explicit opaque colors, remove explicit same foreground/background color pairs, and apply a maintainer-approved tiny-font policy with targeted tests for both basic and Readability extraction paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass sanitizeHtml HTML containing style="color:white", style="font-size:1px", or matching color/background-color; current main has no predicate that removes those declarations.

## Solution Assessment

Unclear as proposed. The issue identifies the right gap, but the narrowest maintainable solution is a CSS declaration/color normalization path plus an explicit tiny-font threshold policy, not a short list of named-color regexes.
