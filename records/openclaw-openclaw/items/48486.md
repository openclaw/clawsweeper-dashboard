## Summary

Keep open: current main and v2026.5.12 still use exact inline-style predicates, so the reported near-invisible CSS sanitizer gap remains present. The prior discussion correctly points toward color normalization and explicit same foreground/background handling, but the tiny-font and near-white thresholds need maintainer/security policy before an automated repair is safe.

## What This Changes

Request to harden web_fetch HTML sanitization against near-invisible CSS, including near-white text, tiny nonzero fonts, and matching foreground/background colors.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque colors, remove elements whose explicit foreground and background colors normalize to the same value, and apply a maintainer-approved tiny-font threshold with targeted basic and Readability extraction tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass sanitizeHtml HTML containing style="color:white", style="font-size:1px", or matching color/background-color; current main has no predicate or color-comparison path that removes those declarations.

## Solution Assessment

Unclear as proposed. The issue identifies the right gap, but the narrowest maintainable fix is CSS declaration/color normalization plus an explicit tiny-font threshold policy, not a short list of named-color regexes.
