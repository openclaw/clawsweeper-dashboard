## Summary

The issue should stay open: current main still appears to leave the reported near-invisible CSS sanitizer gap in web_fetch, and the security-sensitive thresholds for color contrast and tiny text need maintainer policy rather than conservative cleanup closure.

## What This Changes

The issue requests web_fetch sanitizer hardening for near-white text, tiny nonzero font sizes, and matching foreground/background colors that can hide prompt-injection text from human reviewers.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque colors, remove elements whose explicit foreground and background colors normalize to the same value, and implement a maintainer-approved tiny-font threshold with targeted tests for both basic and Readability extraction paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass sanitizeHtml HTML with style="color:white", style="font-size:1px", or matching color/background-color; current main's regex predicates do not match those declarations, so the payload remains in extracted content.

## Solution Assessment

Unclear as proposed. The named-regex additions in the issue would be too narrow; the safer direction is a declaration/color normalizer plus maintainer-approved tiny-font policy to balance security and false positives.
