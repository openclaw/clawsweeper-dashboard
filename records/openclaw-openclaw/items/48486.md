## Summary

Keep open. Current main still has the reported web_fetch sanitizer gap: inline hidden-style handling is exact-pattern based and does not normalize opaque CSS colors, compare explicit foreground/background colors, or reject tiny nonzero font sizes. The follow-up discussion correctly points toward a CSS declaration/color-normalization fix, but the thresholds are security-sensitive and need maintainer policy judgment before a safe automated repair.

## What This Changes

The issue requests hardening `web_fetch` HTML sanitization against near-invisible CSS, including near-white text, tiny nonzero fonts, and matching foreground/background colors.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque colors, remove elements whose explicit foreground and background colors normalize to the same value, and apply a maintainer-approved tiny-font threshold. Cover both the basic and Readability extraction paths with targeted tests, and add a changelog/security note if maintainers classify this as user-facing hardening.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `sanitizeHtml` HTML containing `style="color:white"`, `style="font-size:1px"`, or matching `color`/`background-color`; none of those declarations match the current removal predicates, so the payload would remain in extracted content.

## Solution Assessment

No, not as a simple named-color regex patch. The maintainable direction is a small CSS declaration normalizer and explicit foreground/background comparison, plus a maintainer-approved tiny-font policy to avoid over-removing legitimate content.
