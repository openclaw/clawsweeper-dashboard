## Summary

Current main still leaves the reported web_fetch sanitizer gap open. The shared sanitizer remains exact-pattern based for inline CSS visibility, and the issue is security-sensitive enough that threshold choices should stay with maintainer/security review rather than cleanup closure.

## What This Changes

The issue requests stronger web_fetch HTML sanitization for near-invisible CSS patterns, including near-white text, tiny nonzero fonts, and matching foreground/background colors.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque colors, remove elements whose explicit foreground and background colors normalize to the same value, and apply a maintainer-approved tiny-font threshold. Cover both basic and Readability extraction paths with targeted tests, and add release notes if maintainers classify this as user-facing security hardening.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass sanitizeHtml HTML containing `style="color:white"`, `style="font-size:1px"`, or matching `color`/`background-color`; none of those declarations match the current removal predicates, so the payload remains eligible for extraction.

## Solution Assessment

No, not as a simple named-color regex patch. The maintainable fix is a small CSS declaration normalizer with explicit foreground/background comparison plus a maintainer-approved tiny-font policy.
