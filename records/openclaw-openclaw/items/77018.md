## Summary

Keep open; the latest head remains a security-sensitive patch with blocking target-scoping and inherited-proxy routing defects, and current checks are not green.

## What This Changes

The PR adds `proxy.loopbackMode`, rewires Gateway loopback proxy bypass through global-agent `NO_PROXY`, updates proxy schema/docs/changelog, and adds proxy/client coverage.

## Best Possible Solution

Keep `proxy.loopbackMode` only if GatewayClient receives a config-resolved URL, direct bypass is authorized solely for that target, inherited proxy-mode child CLIs bootstrap routing or fail closed, and CI is green.

## Reproduction Assessment

Yes. Source inspection of PR head shows the configured Gateway URL is computed but not carried into `GatewayClient`, and bypassed child CLI paths can inherit proxy mode without starting global-agent routing; I did not run tests because this review is read-only.

## Solution Assessment

No. `proxy.loopbackMode` is a plausible direction, but this implementation is not yet the narrow maintainable fix because exact configured-target enforcement, inherited child routing, and current CI are still incomplete.

Full review comments:

- **[P1] Carry the configured target into the client:** `src/gateway/client-bootstrap.ts:34-37`
  - body: `configuredConnection` is computed but never returned or passed to `GatewayClient`, and the client type has only a comment rather than a configured-url option. That means a per-call loopback URL can still authorize its own direct bypass instead of being compared with the configured Gateway URL promised by the docs.
  - confidence: 0.93

  - body: When a bypassed child CLI inherits `OPENCLAW_PROXY_LOOPBACK_MODE=proxy`, this branch avoids `NO_PROXY` but does not bootstrap global-agent or pass an explicit proxy agent to `ws`. Since those CLIs skip managed-proxy startup, the loopback WebSocket can still connect directly.
  - confidence: 0.9

  - confidence: 0.82
