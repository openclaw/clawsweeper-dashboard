## Summary

Keep open: the current PR head still has blocking managed-proxy policy gaps and a failing lint annotation, and the protected maintainer/human-review labels leave this with human maintainers.

## What This Changes

The PR adds `proxy.loopbackMode`, updates managed-proxy state/env/schema/docs/changelog, and changes GatewayClient loopback handling with targeted coverage.

## Best Possible Solution

Carry a config-resolved Gateway URL through GatewayClient callers, authorize direct bypass only for that exact target, and make inherited child `proxy` mode use an explicit proxy path or fail closed before landing.

## Reproduction Assessment

Yes. Source inspection of PR head shows the configured target is unused, exact-head lint fails on that unused value, and current main keeps local control-plane CLIs on proxy-bypass startup paths.

## Solution Assessment

No. `proxy.loopbackMode` is a plausible direction, but this patch is not yet the narrow maintainable fix because configured-target enforcement and inherited-child proxy routing remain incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Carry the configured target into the client:** `src/gateway/client-bootstrap.ts:34-37`
  - body: `configuredConnection` is computed but never used, which is already failing `check-lint`; more importantly, `GatewayClient` still only receives the actual connection URL, so a per-call loopback override can authorize its own direct bypass instead of being compared to the configured Gateway URL promised by the docs.
  - confidence: 0.91

That process can still open the loopback WebSocket directly instead of through the operator proxy.
  - confidence: 0.88

  - body: This PR adds a Sandbox/Windows changelog bullet for separate Docker bind-source work, but the branch does not include that change. Landing it here would publish an unrelated release note under this PR.
  - confidence: 0.78
