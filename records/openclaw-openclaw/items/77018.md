## Summary

This PR should stay open: current main still has the broad Gateway loopback bypass, but the latest PR head does not yet narrow it to an independently configured Gateway URL. The item is also maintainer-authored and maintainer-labeled, so it needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR adds `proxy.loopbackMode`, changes managed proxy loopback handling, and updates proxy docs, schema, changelog, and focused tests.

## Best Possible Solution

Carry an independently resolved configured Gateway URL through the Gateway bootstrap/call/client boundary, then enforce `gateway-only`, `proxy`, and `block` against that target with regression coverage.

## Reproduction Assessment

Yes. Source inspection is enough: current main bypasses any loopback `GatewayClient` URL, and the PR still supplies the caller URL as both actual and expected Gateway URL.

## Solution Assessment

No. The mode plumbing is a reasonable direction, but the bypass target must come from configuration or bootstrap state instead of the caller-provided connection URL.
