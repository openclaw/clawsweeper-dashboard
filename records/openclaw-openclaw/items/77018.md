## Summary

Keep this PR open: the latest head still leaves two proxy-policy blockers, and the PR is explicitly labeled for maintainer/human review.

## What This Changes

The PR adds `proxy.loopbackMode`, updates proxy config/schema/docs/changelog, and replaces temporary Gateway loopback HTTP-stack unpatching with global-agent `NO_PROXY` registration and tests.

## Best Possible Solution

Carry a config-resolved Gateway URL through bootstrap and `GatewayClient`, authorize default bypass only for that exact loopback target, and make inherited child `proxy` mode either bootstrap routing or fail closed before landing.

## Reproduction Assessment

Yes. Source inspection of PR head shows the configured target is unused, and current main keeps local control-plane CLI commands on proxy-bypass startup paths that can inherit `proxy` mode without global-agent bootstrap.

## Solution Assessment

No. `proxy.loopbackMode` is a plausible direction, but this patch is not yet the narrow maintainable fix because exact configured-target enforcement and inherited-child proxy routing remain incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

  - body: `configuredConnection` is computed but never returned or passed to `GatewayClient`, so a per-call loopback URL can still authorize its own direct bypass instead of being compared with the configured Gateway URL promised by the docs.

  - confidence: 0.89

  - confidence: 0.78
