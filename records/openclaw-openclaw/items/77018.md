## Summary

Keep this PR open: it is maintainer-labeled/human-review work and the current head still has security-sensitive proxy policy gaps that should block merge.

## What This Changes

This PR adds `proxy.loopbackMode`, changes Gateway loopback WebSocket proxy bypass behavior, threads Gateway URL context into related clients/tests, and updates proxy docs, schema, changelog, and coverage.

## Best Possible Solution

Carry a config-resolved Gateway URL through every GatewayClient caller, allow direct bypass only for that exact target in `gateway-only`, and make inherited `proxy` mode use a concrete proxy route or fail closed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a proxy-started parent can fork a `networkProxy: "bypass"` local control-plane CLI with inherited `OPENCLAW_PROXY_ACTIVE=1` and `OPENCLAW_PROXY_LOOPBACK_MODE=proxy`, and the child GatewayClient opens `ws` through unpatched Node HTTP.

## Solution Assessment

No. `proxy.loopbackMode` is a reasonable direction, but this head still does not enforce proxy-mode routing in inherited children or restrict default direct bypass to the configured Gateway URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route inherited proxy-mode loopback through the proxy:** `src/gateway/client.ts:308`
  - body: When a bypassed child CLI inherits `OPENCLAW_PROXY_LOOPBACK_MODE=proxy`, `startProxy()` has not bootstrapped `global-agent` in that process. Setting `directAgent` to `undefined` makes `ws` use Node's unpatched `http.request`, so loopback traffic can still go direct instead of through the managed proxy.
  - confidence: 0.9
- **[P1] Require the configured Gateway URL before direct bypass:** `src/gateway/client.ts:303-308`
  - body: `gateway-only` still derives the bypass from the actual loopback URL, while the computed configured URL is not stored or compared. Any caller-provided loopback GatewayClient URL can therefore clear the proxy env, contradicting the new docs promise that only the configured Gateway URL bypasses.
  - confidence: 0.87
- **[P2] Declare configuredGatewayUrl before passing it:** `extensions/google-meet/src/voice-call-gateway.ts:51`
  - body: The Google Meet plugin now passes `configuredGatewayUrl`, but `GatewayClientOptions` on this head only adds a comment and no property. Since the plugin SDK re-exports the core class, extension package type checks can reject this object literal and the client still cannot use the configured target.
  - confidence: 0.82
- **[P3] Remove the unrelated changelog entry:** `CHANGELOG.md:68`
  - body: This branch adds a Sandbox/Windows changelog bullet for different work even though the PR files do not include that fix. Landing this PR would publish an unrelated release note under this change.
  - confidence: 0.78
