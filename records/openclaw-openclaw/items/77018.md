## Summary

Keep open. This draft maintainer PR is pointed at the right proxy/Gateway hardening area, but the new exact-target check is currently fed a tautological `{ actualUrl: url, expectedGatewayUrl: url }`, so it does not yet enforce the configured Gateway URL boundary described by the PR and docs.

## What This Changes

The PR adds `proxy.loopbackMode`, threads active loopback mode through managed proxy state, changes Gateway loopback handling by mode, and updates proxy docs, schema, changelog, and focused tests.

## Best Possible Solution

Carry a separately resolved configured Gateway URL through the Gateway bootstrap/call/client boundary, then land the mode policy after maintainer review and exact-head validation.

## Reproduction Assessment

Yes. Source inspection is enough: current main bypasses any loopback GatewayClient URL, and the PR diff still compares the requested URL to itself rather than to an independent configured Gateway URL.

## Solution Assessment

No. The mode plumbing is a reasonable direction, but the target narrowing is incomplete until `expectedGatewayUrl` comes from configuration/bootstrap state rather than the same caller-controlled URL.
