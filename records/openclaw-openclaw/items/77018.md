## Summary

Keep open: latest PR head still leaves inherited proxy-mode Gateway clients able to connect directly, still scopes the default bypass from the actual per-call URL rather than a config-resolved target, and has an exact-head aggregate check failure under protected human-review handling.

## What This Changes

The branch adds `proxy.loopbackMode`, rewires Gateway loopback proxy bypass through global-agent `NO_PROXY`, and updates proxy docs, schema, changelog, and tests.

## Best Possible Solution

Keep the PR in human review until `GatewayClient` receives a config-resolved Gateway URL, direct bypass is limited to that target, inherited proxy-mode child CLIs bootstrap or explicitly use proxy routing, the unrelated changelog entry is removed, and exact-head CI is green.

## Reproduction Assessment

Yes. Source inspection of PR head shows `GatewayClient` registers `NO_PROXY` from the actual `url`, and bypassed `acp`/`chat`/`mcp`/`tui` command paths can inherit proxy mode without starting managed proxy routing.

## Solution Assessment

No. `proxy.loopbackMode` is a plausible direction, but this patch is not yet the narrow maintainable solution because exact configured-target enforcement and inherited child proxy routing are incomplete.
