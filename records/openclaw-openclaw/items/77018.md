## Summary

Keep open. This draft maintainer PR is addressing a real proxy-bypass boundary, but the current patch still makes the configured-target check caller-controlled, so it should not merge as-is.

## What This Changes

The PR adds `proxy.loopbackMode`, updates managed proxy loopback state, changes Gateway loopback bypass behavior, and adds proxy docs, schema, changelog, and focused tests.

## Best Possible Solution

Carry a separately resolved configured Gateway URL through the Gateway bootstrap/call/client boundary, then enforce `gateway-only`, `proxy`, and `block` modes against that independent target.

## Reproduction Assessment

Yes. Source inspection is enough: current main bypasses any loopback `GatewayClient` URL, and the PR patch still passes the same caller-provided URL as both actual and expected.

## Solution Assessment

No. The mode plumbing is a reasonable direction, but the target narrowing is incomplete until `expectedGatewayUrl` comes from configuration/bootstrap state rather than the same caller-controlled URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Pass the resolved configured Gateway URL separately, or make the bypass decision at a call site that already knows it.
