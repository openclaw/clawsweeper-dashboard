## Summary

Keep this PR open: it is a maintainer/human-review PR, and the latest head still leaves a security-sensitive gap where inherited `proxy.loopbackMode: "proxy"` can avoid the Gateway `NO_PROXY` bypass without actually bootstrapping proxy routing for bypassed child CLIs.

## What This Changes

The PR adds `proxy.loopbackMode`, replaces Gateway loopback bypass with global-agent `NO_PROXY` registration, and updates proxy docs, schema, changelog, and tests.

## Best Possible Solution

Keep the loopback policy direction, but ensure inherited child Gateway clients either bootstrap managed proxy routing or use an explicit proxy agent in `proxy` mode before this lands.

## Reproduction Assessment

Yes. Source inspection of PR head shows a high-confidence path: bypassed child CLIs inherit `OPENCLAW_PROXY_LOOPBACK_MODE=proxy`, skip `NO_PROXY` registration, and still construct a plain `ws` client without proxy bootstrap.

## Solution Assessment

No. `proxy.loopbackMode` is a plausible policy surface, but the current implementation is incomplete until `proxy` mode actually routes inherited child Gateway WebSockets through the managed proxy.
