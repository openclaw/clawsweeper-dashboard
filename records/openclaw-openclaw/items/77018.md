## Summary

Keep open for human review: the latest PR head still leaves security-sensitive managed-proxy loopback policy gaps, and this member PR has protected maintainer/human-review handling.

## What This Changes

The PR adds `proxy.loopbackMode`, GatewayClient loopback proxy policy handling, proxy docs/schema/changelog updates, and targeted Gateway/TUI/ACP/node-host coverage.

## Best Possible Solution

Carry a config-resolved Gateway URL through every GatewayClient caller, allow direct bypass only for that exact target in `gateway-only`, and make `proxy`/`block` modes use an explicit proxy path or fail closed with docs and changelog aligned.

## Reproduction Assessment

Yes, by source inspection. A bypassed local control-plane CLI can inherit `OPENCLAW_PROXY_ACTIVE=1` and `OPENCLAW_PROXY_LOOPBACK_MODE=proxy` without bootstrapping the proxy stack, and the configured-target bypass gap is visible because the computed configured connection is not passed into `GatewayClient`.

## Solution Assessment

No. `proxy.loopbackMode` is a reasonable direction, but this patch still needs exact configured-target enforcement and an inherited-child proxy path or fail-closed behavior before it is the narrow maintainable fix.
