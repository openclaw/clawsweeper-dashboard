## Summary

Keep this PR open: the exact configured-Gateway URL guard is now in the right direction, but the current head still has a security-sensitive proxy-mode gap for forked/local control-plane CLIs that inherit managed-proxy env without starting the proxy lifecycle.

## What This Changes

The PR adds `proxy.loopbackMode`, threads `configuredGatewayUrl` through Gateway client call sites, narrows the loopback bypass check, and updates schema, docs, changelog, and tests.

## Best Possible Solution

Keep the exact configured-Gateway URL guard, then make inherited `proxy` mode either use a concrete proxy agent/bootstrap path from inherited proxy env or fail closed, with regression coverage for a forked/bypassed local CLI.

## Reproduction Assessment

Yes. Source inspection is enough for the PR defect: a bypassed local control-plane CLI can inherit `OPENCLAW_PROXY_LOOPBACK_MODE=proxy` without running `startProxy()`, and the PR then creates a `ws` client with no direct agent and no proxy agent/bootstrap.

## Solution Assessment

No. The exact configured-URL guard is the right shape, but `proxy` mode must actually route through a managed proxy in inherited child processes instead of relying on ambient proxy env that Node and `ws` do not apply by themselves.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Route inherited proxy-mode loopback through the proxy:** `src/gateway/client.ts:315-316`
  - body: When a forked local control-plane CLI inherits `OPENCLAW_PROXY_LOOPBACK_MODE=proxy`, it skips `startProxy()`, so `global-agent` is never bootstrapped. This branch drops the direct loopback agent but still constructs `new WebSocket` with no proxy agent; `ws` delegates to Node `http.request`, which does not honor proxy env by itself, so the configured Gateway loopback still connects directly instead of through the managed proxy.
