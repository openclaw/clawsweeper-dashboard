## Summary

Keep open: the security-hardening direction is useful, but the patch filters manual recall after a search capped at the requested limit, so contaminated hits can hide clean memories; the external PR also only provides mocked test proof for a security-sensitive behavior change.

## What This Changes

The PR updates the memory-lancedb plugin to sanitize and guard `memory_recall` tool output, reuse recall filtering with auto-recall, strengthen prompt-injection capture filtering, and add focused regression tests.

## Best Possible Solution

Keep the guard/escape/sanitizer direction, but overfetch manual recall before filtering, add a regression where contaminated hits precede clean hits, and require redacted real behavior proof before merge.

## Solution Assessment

Not yet. Guarding and escaping at the plugin presentation boundary is the right direction, but the manual tool should overfetch before sanitizing and capping, matching the auto-recall invariant.

Full review comments:

- **[P2] Overfetch before filtering memory_recall hits:** `extensions/memory-lancedb/index.ts:1570`
  - body: `memory_recall` still asks LanceDB for only `limit` rows, then filters contaminated rows and slices to the same limit. If the top results are stale envelope/media sludge, clean memories just below that cutoff are never fetched, so the tool can return fewer or no memories even though safe matches exist; auto-recall already overfetches for this exact reason. Fetch a bounded overfetch count first, then sanitize and cap to the requested limit.
