## Summary

Current main implements substantial pieces of the requested experience, but not the central unified operator lifecycle: memory-core owns search, promotion, backfill, and recall workflows; memory-wiki owns provenance-rich ingestion; and memory-lancedb exposes store, recall, and destructive forget tools. The linked Engram Core project is useful external implementation context, but the remaining review/edit/conflict/tombstone semantics are still tracked by open sibling issues and require a maintainer product decision about the recommended memory plugin and API boundary.

## What This Changes

Add operator-facing remember, recall, review, edit, and forget commands plus ingestion helpers and provenance-aware output around the memory pipeline.

## Best Possible Solution

Treat the three Memory MVP issues as one sequenced product design: select the recommended active memory plugin, finalize conservative write and review/conflict state transitions, then expose a small plugin-owned operator surface with provenance and match explanations through generic core CLI routing only where needed.

Not applicable: this issue requests a new unified operator and ingestion surface rather than reporting a failure of an existing documented command contract.

## Solution Assessment

No, implementing the listed commands independently is not the best solution; the command surface should follow the canonical write, provenance, conflict, revision, and forgetting model selected for the recommended memory plugin.
