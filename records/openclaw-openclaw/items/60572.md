## Summary

Keep open. Current main still models memory as one global exclusive slot with additive supplements and a narrow dreaming coexistence exception, but it does not implement purpose-specific memory roles or per-agent memory slot overrides. The issue discussion describes active production composition needs, so this remains a valid architecture RFC rather than cleanup material.

## What This Changes

Request replacing the single `plugins.slots.memory` selector with multiple purpose-specific, per-agent memory slots for recall, compaction, capture, and user modeling.

## Best Possible Solution

Keep this issue open as the canonical multi-slot memory architecture RFC. The desired end state is a backward-compatible composable memory-role contract that preserves `plugins.slots.memory`, defines how additive prompt/corpus supplements and dreaming coexist, and only adds per-agent scoping if maintainers decide that belongs in the public config model.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a runtime bug. The current-main source check is high confidence: the config schema, runtime loader, docs, and tests still model one active memory slot rather than purpose-specific role slots.

## Solution Assessment

Unclear whether the exact four dotted slot names are the best final API. A maintainable solution should first design a generic role/capability registry or versioned memory composition contract, then map any user-facing shorthand onto that without breaking existing memory plugins.
