## Summary

Keep open. Current main still models memory as one active slot with additive supplement APIs and a narrow dreaming coexistence exception; it does not implement purpose-specific memory roles or per-agent memory slot overrides, and the issue discussion remains a real architecture request rather than cleanup material.

## What This Changes

The issue requests purpose-specific memory slots so separate plugins can own recall, compaction, capture, and user-modeling roles globally and per agent.

## Best Possible Solution

Keep this as the canonical memory-role composition RFC until maintainers define a backward-compatible public contract that preserves plugins.slots.memory, supports intentional additive memory layers, and decides whether per-agent slot scoping belongs in config.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/RFC. Source, schema, tests, docs, and related-thread search give high-confidence proof that current main still lacks purpose-specific memory role slots and per-agent memory slot overrides.

## Solution Assessment

Unclear whether the proposed four dotted slot names are the best final API. The maintainable path is a versioned memory capability/role contract with migration and docs, then a user-facing config shape that does not break existing memory plugins.
