## Summary

Keep open. Current main still treats memory as one exclusive active slot with additive supplement APIs and a narrow dreaming coexistence path, but it does not implement purpose-specific memory roles or per-agent memory slot overrides; the issue discussion shows active production composition needs rather than cleanup material.

## What This Changes

Request a backward-compatible architecture that replaces the single `plugins.slots.memory` choice with multiple purpose-specific memory roles and optional per-agent slot overrides.

## Best Possible Solution

Keep this as the canonical multi-slot memory RFC until maintainers design a backward-compatible memory-role composition contract that preserves existing `plugins.slots.memory`, clarifies additive supplements and dreaming coexistence, and decides whether per-agent slot scoping belongs in public config.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a runtime bug. The current-main source check is high confidence: config, schema, runtime loading, tests, and docs still model one active memory slot rather than purpose-specific role slots.

## Solution Assessment

Unclear whether the exact four dotted slot names are the best final API. A maintainable solution should start with a versioned memory capability/role contract and migration story, then expose user-facing shorthand without breaking existing memory plugins.
