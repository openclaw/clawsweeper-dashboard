## Summary

Keep open. Current main and the latest release still expose a single scalar memory slot plus additive supplement APIs, not the requested purpose-specific memory roles or per-agent memory slot overrides; the discussion also has concrete user reports showing this remains an architecture/product RFC rather than cleanup-ready obsolete work.

## What This Changes

Request purpose-specific and per-agent memory slots so recall, compaction, capture, and user-model providers can coexist instead of competing for one global `plugins.slots.memory` owner.

## Best Possible Solution

Keep this as the canonical memory-role composition RFC until maintainers choose a backward-compatible contract that preserves the primary memory slot while defining intentional additive memory layers and per-agent scoping.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/RFC. Source, docs, release inspection, and live issue comments give high-confidence evidence that current main still lacks purpose-specific memory role slots and per-agent memory slot overrides.

## Solution Assessment

Unclear whether the proposed four dotted slot names are the best final API. The maintainable path is a versioned memory role/capability contract with compatibility, migration, SDK docs, and tests rather than an automated one-file fix.
