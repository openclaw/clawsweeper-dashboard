## Summary

Keep open: current main and the latest release still expose a single scalar memory slot, while the open implementation PR that would address the request is unmerged and currently conflicting.

## What This Changes

The issue requests purpose-specific memory slots so recall, compaction, capture, dreaming, and user-model memory providers can coexist instead of competing for one global memory slot.

## Best Possible Solution

Review, rebase, and either land or replace the open memory role-slot implementation with a maintainer-approved contract that preserves legacy config through doctor, docs, runtime, and upgrade coverage.

## Reproduction Assessment

Yes. Source inspection shows the central architecture gap: the config schema only accepts `plugins.slots.memory` and `plugins.slots.contextEngine`, and runtime memory resolution loads one configured memory plugin id.

## Solution Assessment

Unclear. Multi-role slots are a plausible VISION-compatible plugin API direction, but the exact taxonomy, legacy `plugins.slots.memory` migration, and linked PR shape need maintainer product review before calling it the best solution.
