## Summary

Keep open. Current main still exposes one scalar memory slot plus additive supplement APIs, but it does not implement purpose-specific memory roles or per-agent memory slot overrides, and the discussion has concrete production reports that make this a real architecture RFC rather than cleanup material.

## What This Changes

The issue requests purpose-specific memory slots so multiple memory providers can participate in recall, compaction, capture, and user-model roles at the same time.

## Best Possible Solution

Keep this as the canonical memory-role composition RFC until maintainers define a backward-compatible public contract that preserves plugins.slots.memory, supports intentional additive memory layers, and decides whether per-agent slot scoping belongs in config.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/RFC. Source, schema, docs, tests, and related-thread inspection give high-confidence proof that current main still lacks purpose-specific memory role slots and per-agent memory slot overrides.

## Solution Assessment

Unclear whether the proposed four dotted slot names are the best final API. The maintainable path is a versioned memory capability/role contract with migration and docs, plus a user-facing config shape that does not break existing memory plugins.
