## What This Changes

The issue requests purpose-specific memory role slots and per-agent memory slot overrides so multiple memory-related plugins can compose instead of competing for one `plugins.slots.memory` owner.

## Best Possible Solution

Review, rebase, and either land or replace the open role-slot implementation with a maintainer-approved plugin/config contract that preserves legacy `plugins.slots.memory` through doctor, docs, runtime, and upgrade coverage.

## Reproduction Assessment

Yes, at source level. Current main and the latest release still expose a scalar `plugins.slots.memory` and resolve one active memory runtime plugin, so the requested role-slot architecture is absent without a live multi-agent setup.

## Solution Assessment

Unclear. Multi-role memory slots are a plausible plugin API extension, but the role taxonomy, per-agent semantics, legacy migration, and the current VISION.md one-memory-slot direction need maintainer approval before calling it the best solution.
