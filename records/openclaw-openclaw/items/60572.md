## Summary

Keep open. Current main still exposes one scalar memory slot plus additive supplement APIs, not purpose-specific memory roles or per-agent memory slot overrides, and the issue discussion includes concrete production reports that make this a real architecture RFC.

## What This Changes

Request purpose-specific memory slots and per-agent memory slot configuration so multiple memory providers can compose simultaneously.

## Best Possible Solution

Keep this as the canonical memory-role composition RFC until maintainers define a backward-compatible public contract that preserves plugins.slots.memory while supporting intentional additive memory layers.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/RFC. Source, docs, tests, live comments, and related-thread inspection give high-confidence proof that current main still lacks purpose-specific memory role slots and per-agent memory slot overrides.

## Solution Assessment

Unclear whether the proposed four dotted slot names are the best final API. The maintainable path is a versioned memory capability/role contract with migration and docs, plus a user-facing config shape that does not break existing memory plugins.
