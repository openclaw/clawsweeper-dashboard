## Summary

Keep this PR open: it targets a real TUI burst-send bug, but the patch still misses interleaved run events and the external-PR real behavior proof is mock/test output only.

## What This Changes

The PR changes TUI optimistic-send tracking from a boolean to a numeric pending counter across state, command/event handlers, tests, and the changelog.

## Best Possible Solution

Keep the counter direction only if `handleChatEvent` binds and decrements one pending count for every newly observed non-btw run regardless of active-run interleaving, with regression coverage for interleaved burst events and redacted live TUI proof.

## Reproduction Assessment

Yes, source-reproducible: the external NemoClaw report has concrete rapid-send steps and current main still has a single boolean optimistic marker consumed once. I did not execute a live TUI run in this read-only review.

## Solution Assessment

No. The counter approach is reasonable, but this implementation keeps binding under the active-run-empty branch, making the fix timing-dependent and leaving a stale positive counter possible after interleaved burst events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Bind pending runs even while another run is active:** `src/tui/tui-event-handlers.ts:365-370`
  - body: The counter is only consumed inside the `!state.activeChatRunId` branch. If run-2 or run-3 deltas arrive while run-1 is still active, those run ids are never tagged local and the counter stays positive, so a later unrelated run can be misclassified and the burst fix remains timing-dependent. Add coverage where all burst deltas arrive before any final, then bind/decrement on first observation of each pending non-btw run.
  - confidence: 0.84
