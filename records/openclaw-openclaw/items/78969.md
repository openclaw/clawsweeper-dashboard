## Summary

Keep open; the PR targets a real TUI transcript problem, but the latest head is not merge-ready because the new counter path is not reachable through the current production submit guards and the after-fix proof does not show a real TUI burst after the patch.

## What This Changes

This PR changes TUI optimistic-send state from a boolean to a numeric counter, tags first-seen run IDs as local while the counter is positive, updates TUI regression tests, and adds a changelog entry.

## Best Possible Solution

Reconcile the fix with current TUI submit semantics: either implement and prove a supported queued burst-send path that can accumulate pending local runs, or keep the busy guard and target the actual remaining history race with reachable-state tests.

## Reproduction Assessment

Unclear. The downstream report shows the old live symptom and the non-local final reload path is real, but current main and this PR head block rapid non-BTW submits before the pending counter can exceed 1, so I did not establish a reachable current-main TUI reproduction.

## Solution Assessment

No. The counter/local-binding approach may be suitable if TUI queues rapid sends, but this branch currently tests manually seeded state instead of the production submit path, so it needs a reachable fix boundary or explicit queued-send semantics.
