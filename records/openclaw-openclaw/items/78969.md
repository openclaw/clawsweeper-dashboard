## Summary

Keep open. The PR targets a real TUI bug, but the counter is still consumed only when no active run exists, so interleaved burst events can leave later runs untagged and still trigger history reloads; external real behavior proof is also only test/check output.

## What This Changes

The PR changes TUI optimistic-send state from a boolean to a counter, updates command/event handler tests, and adds a changelog entry for burst sends.

## Best Possible Solution

Keep the counter direction, but bind each first-observed pending normal run regardless of the current active run, add an interleaved burst-order regression test, and require redacted live TUI proof before merge.

## Reproduction Assessment

Yes, as a source-level reproduction: queue multiple normal TUI sends before gateway chat events, then let a later run finalize after the first clears `activeChatRunId`; current main consumes the boolean once and the later non-local final can reach `loadHistory`. I did not run a live TUI in this read-only review.

## Solution Assessment

No. A counter is the right direction, but this patch still binds pending runs only inside the no-active-run branch; the safer fix should tag each first-observed pending run and cover interleaved event ordering.
