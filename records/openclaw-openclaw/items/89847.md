## Summary

Keep open: current main and the latest release still preserve the reported failure-alert gap for best-effort cron failures and still derive `unknown` notification status without persisting an actual alert send result; related open PRs cover narrower status/thread/diagnostic slices rather than replacing this broader issue.

## What This Changes

The issue asks for cron failure alerts to deliver independently when isolated agent setup stalls in `runtime-plugins`, with clearer delivery-attempt status, Telegram thread routing, and more visible cron error rows.

## Best Possible Solution

Keep this issue as the canonical tracker for the runtime-stall failure-alert delivery/status contract, then land or revise the narrower status-persistence, thread-routing, and runtime-diagnostic PRs after maintainers decide the public cron semantics.

## Reproduction Assessment

Yes, source-reproducible: the reported best-effort cron shape currently returns before sending failure alerts, and errored runs can still persist a derived `unknown` notification status.

## Solution Assessment

No, not as one automatic cleanup fix. The durable fix should move failed-run alert delivery/status capture into a gateway-owned path, while explicit status vocabulary and `failureAlert.threadId` need coordinated API/product decisions.
