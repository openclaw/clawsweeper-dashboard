## Summary

Keep open: current main still has source-visible gaps where cron errors can leave failure-notification delivery as `unknown` or suppress `failureAlert` for best-effort jobs, and the related open items split rather than replace this alert-delivery/status work.

## What This Changes

The issue reports that cron `failureAlert` notifications can silently fail or remain `unknown` when an isolated cron stalls during `runtime-plugins`, leaving operators without the configured per-cron alert.

## Best Possible Solution

Keep this issue open for the distinct cron failure-alert delivery/status contract, while leaving runtime-plugin stall root cause to https://github.com/openclaw/openclaw/issues/87327, thread routing to https://github.com/openclaw/openclaw/issues/89848, and broader 2026.6.5 regression context to https://github.com/openclaw/openclaw/issues/92058.

## Reproduction Assessment

Yes at source level with medium confidence: current code still records `unknown` or `not-requested` failure-notification states before any async `failureAlert` send can persist a result. No live Telegram cron reproduction was run in this read-only review.

## Solution Assessment

Unclear as a final implementation shape: decoupling and persisting the gateway-level alert result is the right boundary, but the exact public status vocabulary and split from `failureDestination` or `threadId` need maintainer direction.
