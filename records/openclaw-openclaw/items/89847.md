## Summary

Keep open: current main and the latest release still have source-visible paths where configured cron failure alerts can be suppressed or recorded as `unknown`/`not-requested`, and the related open items cover adjacent runtime-stall or thread-routing work rather than replacing this alert-delivery/status bug.

## What This Changes

The issue asks OpenClaw to deliver cron `failureAlert` notifications from a gateway-level path when isolated runs stall before execution, make failure-notification statuses explicit, support topic routing, and improve cron-list visibility.

## Best Possible Solution

Define `failureAlert` as an independent gateway-owned failed-run notification path, persist durable status from the actual alert request outcome, and leave runtime-stall diagnostics plus Telegram `threadId` routing to their narrower related items.

## Reproduction Assessment

Yes. The reported config shape is source-reproducible: a best-effort cron error can persist `lastFailureNotificationDeliveryStatus: "unknown"` while `maybeEmitFailureAlert` returns before sending the configured alert.

## Solution Assessment

No, not as one bundled fix. The primary bug belongs in gateway-owned alert request/status persistence, while the requested status vocabulary and `failureAlert.threadId` additions need explicit public contract decisions or separate issues.
