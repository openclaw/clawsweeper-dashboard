## Label Justifications

- `P2`: The issue is a real cron observability/reliability bug with production impact, but it is bounded to scheduled-job alert delivery and status reporting rather than a core runtime outage.
- `impact:message-loss`: A configured failure alert can fail to reach the operator's Telegram alert surface, which is a channel message delivery loss.

## Summary

Keep open: current main still has a source-visible gap where pre-execution cron errors can leave failure notification delivery as `unknown`, while the actual `failureAlert` send is fire-and-forget and does not persist attempted/succeeded/failed state. The thread-routing part is now separately tracked by https://github.com/openclaw/openclaw/issues/89848, and the underlying runtime-plugins stall remains tracked by https://github.com/openclaw/openclaw/issues/87327, but this issue still owns a distinct alert-delivery/status bug.

## What This Changes

The issue asks OpenClaw to make cron `failureAlert` delivery reliable and observable when isolated agent setup stalls in the runtime-plugins phase.

## Best Possible Solution

Split the work: fix the cron/gateway failure-alert path to persist explicit send-attempt results for pre-execution failures, keep runtime-plugin stall diagnostics under https://github.com/openclaw/openclaw/issues/87327, and handle `failureAlert.threadId` through https://github.com/openclaw/openclaw/issues/89848.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source with medium confidence: the current timeout path can produce an error without a delivered value, and the state resolver records the failure notification as `unknown` while the alert sender does not write back a result. A live Telegram cron reproduction was not run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes for the primary boundary: failure-alert result tracking belongs in cron/gateway code, outside the isolated agent runtime that can stall. The status enum, thread routing, and CLI highlighting asks should be split or explicitly approved because they change public schema/config/UX.
