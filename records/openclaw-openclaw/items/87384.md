## Summary

Keep this open. Current main and the latest release still fall back from an empty inferred transcript file to CLI stdout, so the reported contamination path is real and not superseded by the related Sherpa empty-JSON fix.

## What This Changes

The issue asks OpenClaw to treat inferred file-output CLI audio transcripts as authoritative so empty or missing transcript files do not inject CLI progress stdout as transcript text.

## Best Possible Solution

Land a focused media runner fix that makes deterministically inferred transcript-file outputs authoritative for known file-output STT commands, then handle broader custom-wrapper `outputMode` policy separately if maintainers want it.

## Reproduction Assessment

Yes at source level: configure a known file-output CLI audio command, have it create an empty inferred `.txt` file and print progress to stdout, and current main falls through to `stdout.trim()`. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

Mostly yes: making known inferred file outputs authoritative is the narrowest repair for the existing bug. A generic custom-wrapper `outputMode` should be a separate maintainer choice if the known-command fix is not enough.
