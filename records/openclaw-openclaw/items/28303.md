## Summary

Keep open: current main and v2026.6.10 still expose `zh-CN` and `zh-TW` as Control UI and CLI wizard locale IDs. The request remains a valid but compatibility-sensitive product decision because the rename would need coordinated saved-setting migration and generated i18n/docs pipeline changes.

## What This Changes

The issue requests renaming Chinese locale identifiers from `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant` across Control UI, CLI wizard, generated locale files, docs metadata, and saved preferences.

## Best Possible Solution

Decide the canonical Chinese locale-ID contract, then migrate Control UI, CLI wizard, docs publish metadata, translation memory, and saved-setting handling in one coordinated change.

## Reproduction Assessment

Yes, from source: current main still defines Control UI and CLI wizard Chinese locales as `zh-CN` and `zh-TW`, and v2026.6.10 does too. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No, not as a simple rename. The maintainable solution needs a maintainer-approved canonical ID decision plus explicit migration or compatibility handling for saved browser settings, wizard locale normalization, and generated docs/i18n assets.
