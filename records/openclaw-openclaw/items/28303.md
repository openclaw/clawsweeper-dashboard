## Summary

Current main still uses `zh-CN` and `zh-TW` as the Control UI’s canonical locale, storage, loader, and generator identifiers, so the requested migration remains necessary if maintainers choose script-based tags. Keep this open for the existing product and compatibility decision: docs publishing already uses `zh-Hans`, but the Control UI contract is separate and a safe change must preserve saved preferences rather than perform only a file rename.

## What This Changes

Migrate the Control UI Chinese locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant`, including browser detection, generated locale artifacts, selector values, and saved-preference migration.

## Best Possible Solution

Use script tags as one coordinated canonical Control UI contract only after maintainer approval, normalize legacy stored region-tag values at the persistence boundary, map both region and script browser tags correctly, update generator inputs and registry wiring, regenerate all owned artifacts, and add upgrade plus fresh-detection coverage.

## Reproduction Assessment

Yes for the reported current condition: current-main source, generated artifacts, and tests deterministically show `zh-CN` and `zh-TW` as canonical Control UI identifiers. The issue does not report a broken runtime workflow requiring a live failure reproduction.

## Solution Assessment

No, a codebase-wide mechanical rename alone is not the best solution. The narrow maintainable path is a coordinated source-of-truth migration with legacy persisted-value normalization and tests for both upgrade and fresh browser detection.
