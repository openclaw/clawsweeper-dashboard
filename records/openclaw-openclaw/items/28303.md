## Summary

Keep open: current main and the latest release still expose `zh-CN` and `zh-TW` as the Control UI locale contract, and the related docs-only implementation was closed unmerged after maintainer feedback. This is a user-facing locale-contract enhancement that needs maintainer product approval before changing persisted locale IDs and docs publish behavior.

## What This Changes

Rename Chinese locale identifiers across the i18n system from `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant`, with migration for existing saved preferences.

## Best Possible Solution

If maintainers approve the contract change, make `zh-Hans` and `zh-Hant` canonical in the Control UI runtime, generator, tests, workflow, and docs while accepting or migrating legacy `zh-CN`/`zh-TW` at browser-detection and storage boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection gives high confidence that current main and `v2026.5.12` still expose and persist `zh-CN`/`zh-TW`, not canonical `zh-Hans`/`zh-Hant`, at the Control UI boundary.

## Solution Assessment

Yes directionally, but not as an automatic cleanup: a canonical rename with backwards-compatible alias normalization is the maintainable technical path. The docs publish shape and prior maintainer feedback on localized docs require explicit maintainer approval before implementation.
