## Summary

Keep open: current main and the latest release still expose `zh-CN` and `zh-TW` as the Control UI locale contract, while the exact runtime implementation PR is closed unmerged and maintainer i18n discussion makes this a product decision rather than safe cleanup.

## What This Changes

Request to rename the Chinese Control UI locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant`, including runtime imports, browser detection, persisted settings migration, picker labels, docs, and tests.

## Best Possible Solution

If maintainers approve the contract change, make `zh-Hans` and `zh-Hant` canonical for the Control UI runtime, generator, tests, workflow, and user-facing docs while accepting or migrating legacy `zh-CN`/`zh-TW` at browser-detection and storage boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection gives high confidence that current main and `v2026.5.7` still expose and persist `zh-CN`/`zh-TW`, not canonical `zh-Hans`/`zh-Hant`, at the Control UI runtime boundary.

## Solution Assessment

Yes directionally, but not as an automatic cleanup: a canonical rename with backwards-compatible alias normalization is the maintainable technical path. The docs route shape and prior i18n policy discussion require maintainer approval before implementation.
