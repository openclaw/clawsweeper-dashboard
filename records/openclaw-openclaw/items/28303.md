## Summary

Keep this issue open because current main still uses `zh-CN` and `zh-TW` as the Control UI runtime locale contract, generator inputs, workflow matrix, docs, saved-settings validation values, and focused test expectations. The only `zh-Hans`/`zh-Hant` implementation I found is the separate docs publishing mapping, which does not make those IDs accepted by the Control UI runtime.

## What This Changes

Request to rename Chinese i18n locale identifiers from `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant`, including migration for saved user preferences.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` the canonical Control UI locale IDs while keeping compatibility aliases for existing `zh-CN` and `zh-TW` saved values; update the runtime registry, browser detection, generator metadata, workflow matrix, docs, tests, and changelog.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/contract cleanup. The current behavior is high-confidence from source inspection: `zh-Hans`/`zh-Hant` are absent from the Control UI runtime surfaces and `zh-CN`/`zh-TW` remain asserted in source and tests.

## Solution Assessment

Yes, the narrow maintainable path is to make script IDs canonical in the Control UI runtime and normalize legacy region IDs at storage and browser-detection boundaries. A docs-only rename would not solve the runtime contract, and a hard rename without aliases would unnecessarily break saved settings.
