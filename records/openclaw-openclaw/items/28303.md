## Summary

Keep open: current main and v2026.5.19 still treat `zh-CN` and `zh-TW` as canonical UI and wizard locale IDs, while script-code inputs are only aliases in the wizard and docs publishing deliberately maps script-code Mintlify metadata onto region-coded generated directories. This is a compatibility and product-policy migration, not an already-implemented cleanup.

## What This Changes

The issue asks to change Chinese locale identifiers from region-coded `zh-CN`/`zh-TW` to script-coded `zh-Hans`/`zh-Hant` across UI, wizard, docs, imports, browser detection, and stored preferences.

## Best Possible Solution

If maintainers approve the contract change, migrate canonical IDs in one coordinated compatibility change across Control UI, wizard env handling, docs sync, docs, tests, and storage aliases while preserving legacy `zh-CN`/`zh-TW` inputs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source, docs, tests, shipped-release inspection, and closed related PRs give high-confidence proof that current main still treats `zh-CN` and `zh-TW` as canonical IDs.

## Solution Assessment

Unclear until maintainers set compatibility policy. The direction is reasonable, but the maintainable solution needs legacy aliases, stored preference migration, env handling, docs routing, and redirect policy rather than a direct rename.
