## Summary

Keep open: current `main` and latest release still use `zh-CN`/`zh-TW` across Control UI, CLI wizard, and docs-facing locale contracts. The request is valid, but the rename remains a compatibility-sensitive product decision because stored UI preferences, generated locale metadata, docs publish output, and wizard locale normalization need one coordinated migration boundary.

## What This Changes

The issue requests renaming Chinese locale identifiers from region-based `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant` across i18n surfaces, with migration for saved preferences.

## Best Possible Solution

Decide the canonical Chinese locale-ID contract first, then migrate Control UI, CLI wizard, docs publish metadata, generated i18n metadata, tests, docs, and saved settings in one coordinated change.

## Reproduction Assessment

Yes from source: the old locale identifiers remain in Control UI and wizard contracts, and script-code inputs are normalized back to those IDs. Latest release evidence also still shows `zh-CN` and `zh-TW`.

## Solution Assessment

No as a simple rename; the maintainable path needs a maintainer-approved canonical ID decision plus explicit migration or compatibility handling for saved settings and generated docs/UI assets.
