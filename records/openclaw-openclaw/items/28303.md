## Summary

Keep this open: current main and the latest release still use `zh-CN` and `zh-TW` as canonical Control UI locale, storage, loader, generator, and test values, while the direct implementation PRs are closed unmerged or only address adjacent docs/terminology work. ## What This Changes

Rename OpenClaw i18n locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant`, including runtime detection, lazy locale files, generated locale metadata, saved preference migration, and picker labels.

## Best Possible Solution

Make a maintainer-approved locale contract decision, then update the Control UI source-of-truth wiring and generator with a tested legacy-value migration, or explicitly keep `zh-CN`/`zh-TW` as the shipped canonical UI tags.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.6` shows `zh-CN` and `zh-TW` remain the canonical Control UI type, registry, loader, generator, saved-locale, docs, and test values.

## Solution Assessment

No, a codebase-wide mechanical rename alone is not the best solution. The maintainable path is a maintainer-approved source-of-truth migration that normalizes legacy stored values and regenerates the owned locale artifacts.
