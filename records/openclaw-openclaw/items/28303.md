## Summary

Keep open: current main and the latest release still use `zh-CN` and `zh-TW` as canonical locale IDs across the Control UI runtime, generated locale pipeline, docs publishing, and user-facing docs. The request is a low-priority public locale-contract enhancement, not an implemented cleanup or duplicate close candidate.

## What This Changes

Request to rename Chinese locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant`, including runtime files, detection, persistence migration, docs, and labels.

## Best Possible Solution

If maintainers approve the contract change, make `zh-Hans` and `zh-Hant` canonical across runtime, generator metadata, wizard APIs, workflow matrices, docs routing, and tests while preserving `zh-CN`/`zh-TW` aliases with storage migration.

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source and release inspection provide high-confidence proof that `zh-CN` and `zh-TW` remain canonical today.

## Solution Assessment

Yes directionally, but not as an autonomous cleanup. A canonical rename with backward-compatible alias normalization and migration is the maintainable path only after maintainers approve the public locale contract and docs-routing policy.
