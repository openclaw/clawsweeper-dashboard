## Summary

Keep open: current main still uses `zh-CN` and `zh-TW` as the Control UI locale contract, generator input, persisted-setting boundary, tests, workflow metadata, and docs copy. The closest exact implementation PR, https://github.com/openclaw/openclaw/pull/28354, was closed unmerged, and the open docs switcher issue at https://github.com/openclaw/openclaw/issues/73446 tracks a different Mintlify path-preservation problem.

## What This Changes

Request canonical script-based Chinese locale IDs, replacing `zh-CN`/`zh-TW` with `zh-Hans`/`zh-Hant` while preserving existing user preferences.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` canonical in the Control UI runtime, picker, generator, workflow, tests, and docs copy; normalize `zh-CN`/`zh-TW` as legacy aliases at browser-detection and storage boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection gives high confidence that current main still exposes and accepts `zh-CN`/`zh-TW`, not `zh-Hans`/`zh-Hant`, at the supported-locale boundary.

## Solution Assessment

Yes: the maintainable path is a canonical rename with backwards-compatible alias normalization. A docs-only rename would leave runtime behavior unchanged, and a hard rename without aliases would break saved preferences.
