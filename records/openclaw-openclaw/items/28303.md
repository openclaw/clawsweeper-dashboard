## Summary

Keep open: current main still uses `zh-CN` and `zh-TW` as the Control UI runtime, saved-settings, generator, workflow, docs, and test contract. The existing `zh-Hans`/`zh-Hant` work is limited to docs publishing metadata and does not make those IDs accepted by the Control UI.

## What This Changes

Requested rename of Chinese locale identifiers from region-based `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant` across the Control UI i18n contract, saved settings, docs, tests, and generator.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` canonical in the Control UI runtime and generator, normalize legacy `zh-CN`/`zh-TW` aliases at browser-detection and storage boundaries, and update docs, tests, workflow metadata, generated outputs, and changelog together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection gives high confidence that current main still rejects `zh-Hans`/`zh-Hant` at the supported-locale boundary and asserts `zh-CN`/`zh-TW` in runtime tests.

## Solution Assessment

Yes: the narrow maintainable solution is a canonical rename with backwards-compatible alias normalization. A docs-only rename would leave the Control UI rejecting the script IDs, while a hard rename without aliases would break existing saved browser settings.
