## Summary

Keep open: current main and the latest release still expose `zh-CN` and `zh-TW` as user-visible and persisted locale identifiers, while the related docs-only rename attempt was closed unmerged. This is a locale-contract enhancement that needs maintainer product approval before changing runtime IDs, storage migration behavior, and docs publish routing.

## What This Changes

The issue requests making `zh-Hans` and `zh-Hant` the canonical Chinese locale identifiers across runtime i18n, generated files, detection, UI labels, docs, and persisted preferences.

## Best Possible Solution

If maintainers approve the contract change, make `zh-Hans` and `zh-Hant` canonical across the Control UI runtime, generator, tests, workflow, docs, and any wizard-facing contract while accepting and migrating legacy `zh-CN` and `zh-TW` at storage and locale-detection boundaries.

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection confirms current main and `v2026.5.12` still expose and persist `zh-CN` and `zh-TW` at the relevant i18n boundaries.

## Solution Assessment

Yes directionally, but not as an autonomous cleanup. A canonical rename with backward-compatible alias normalization and storage migration is the maintainable path, but the public contract and docs-publish behavior need maintainer approval first.
