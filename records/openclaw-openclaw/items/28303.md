## Summary

Keep open: current main and v2026.6.5 still use `zh-CN` and `zh-TW` as canonical Control UI and wizard locale IDs, while docs only use `zh-Hans`/`zh-Hant` as publish metadata over region-coded output paths. The direct rename attempts were closed unmerged, so this remains a compatibility and product-policy decision rather than completed cleanup.

## What This Changes

Rename Chinese i18n locale identifiers from region-based `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant`.

## Best Possible Solution

If maintainers approve the contract change, migrate canonical IDs in one coordinated compatibility pass with legacy aliases, persisted-preference migration, docs publish/redirect policy, translation-memory continuity, and regression coverage; otherwise document the existing region IDs as intentional.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-code contract enhancement. Source, tests, release-tag inspection, and related PR discussion give high-confidence proof that current main still treats `zh-CN` and `zh-TW` as canonical IDs.

## Solution Assessment

Unclear until maintainers decide the locale compatibility policy. The requested direction is reasonable, but a direct rename is not the best complete fix unless it also handles legacy aliases, stored preferences, wizard env inputs, docs paths, redirects, and translation memory.
