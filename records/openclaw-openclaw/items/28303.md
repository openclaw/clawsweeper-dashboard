## Summary

Keep open because current main still uses `zh-CN` and `zh-TW` as the Control UI runtime, generator, docs, workflow, saved-settings, and test contract; the existing `zh-Hans`/`zh-Hant` mapping is limited to docs publishing and does not make those IDs accepted by the Control UI.

## What This Changes

Request to make `zh-Hans` and `zh-Hant` the canonical Chinese locale IDs across Control UI i18n while preserving or migrating existing `zh-CN` and `zh-TW` saved values.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` canonical in the Control UI runtime and generator, normalize legacy `zh-CN`/`zh-TW` aliases at storage and browser-detection boundaries, and update docs, tests, workflow metadata, generated locale outputs, and changelog together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale contract enhancement. Source inspection gives high confidence that current main still rejects `zh-Hans`/`zh-Hant` at the supported-locale boundary and asserts `zh-CN`/`zh-TW` in runtime tests.

## Solution Assessment

Yes, the best narrow solution is a canonical rename with backwards-compatible alias normalization. A docs-only rename would not affect the Control UI runtime, while a hard rename without aliases would break existing saved browser settings.
