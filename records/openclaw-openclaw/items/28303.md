## Summary

Keep open: current main still uses `zh-CN` and `zh-TW` as the Control UI runtime, storage, generator, workflow, docs, and test contract, so the requested script-based canonical IDs remain unimplemented.

## What This Changes

Request to make Control UI Chinese locale IDs script-based (`zh-Hans`/`zh-Hant`) and migrate legacy `zh-CN`/`zh-TW` saved values.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` canonical in the Control UI runtime and generator, normalize legacy aliases at browser-detection and storage boundaries, then update docs, tests, workflow metadata, generated outputs, and changelog together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a locale-contract enhancement. Source inspection gives high confidence that current main still rejects `zh-Hans`/`zh-Hant` at the supported-locale boundary and still asserts `zh-CN`/`zh-TW` in tests and docs.

## Solution Assessment

Yes: the narrow maintainable solution is a canonical rename with backwards-compatible alias normalization. A docs-only rename would leave the Control UI rejecting script IDs, while a hard rename without aliases would break existing saved settings.
