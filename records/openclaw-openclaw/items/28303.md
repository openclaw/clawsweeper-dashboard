## Summary

Keep open: current `main` and the latest release still use `zh-CN`/`zh-TW` as active Control UI and onboarding wizard locale identifiers, while docs publishing only partially uses script-based Mintlify language metadata. The remaining work is real, but it changes public locale IDs and persisted values, so it needs maintainer product/compatibility direction before a fix PR.

## What This Changes

The issue requests renaming Chinese locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant`, including generated locale files, detection, saved preference migration, and display labels.

## Best Possible Solution

Decide the canonical Chinese locale-ID contract across Control UI runtime, onboarding wizard, docs-publish metadata, and persisted preference migration; if maintainers accept the rename, implement it as one coordinated compatibility-aware change.

## Reproduction Assessment

Yes. Source inspection on current main shows the Control UI and onboarding wizard still declare, resolve, load, test, document, and persist `zh-CN`/`zh-TW` rather than canonical `zh-Hans`/`zh-Hant`.

## Solution Assessment

Unclear without maintainer direction. A direct rename is not enough; the best implementation needs an explicit compatibility plan for saved UI locale values, documented `OPENCLAW_LOCALE` values, generated UI i18n assets, and docs publish metadata.
