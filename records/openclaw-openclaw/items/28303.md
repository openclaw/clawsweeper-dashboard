Keep open: current main and the latest release still use `zh-CN` and `zh-TW` as the Control UI and CLI wizard locale contracts. The remaining work is a compatibility-sensitive product decision because the exact rename PR is closed unmerged, the docs pipeline only partially uses script-form language IDs, and the open zh-TW PR is terminology-only.

## What This Changes

Request to rename Chinese locale identifiers from `zh-CN`/`zh-TW` to `zh-Hans`/`zh-Hant` across i18n runtime wiring, generated assets, docs surfaces, and saved preference handling.

## Best Possible Solution

Decide the canonical Chinese locale-ID contract first, then migrate Control UI, CLI wizard, docs publish metadata, generated i18n assets, tests, and user-facing docs in one coordinated compatibility-aware change.

## Reproduction Assessment

Yes. Source inspection on current main shows the Control UI locale type/registry, saved-preference validation, CLI wizard locale map, tests, and docs still use `zh-CN` and `zh-TW`.

## Solution Assessment

No, not as a simple rename. The maintainable path is a maintainer-approved locale-ID contract with explicit migration and compatibility handling across UI, wizard, docs publish, and generated i18n artifacts.
