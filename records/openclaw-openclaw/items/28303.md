## Summary

Keep this issue open. Current main still exposes `zh-CN` and `zh-TW` as the Control UI runtime locale contract, generator inputs, workflow matrix, docs, saved-settings validation values, and focused test expectations; `zh-Hans` and `zh-Hant` are only present in the separate docs publishing path and are not accepted by the Control UI runtime.

## What This Changes

Request to rename Chinese i18n locale identifiers from `zh-CN`/`zh-TW` to script-based `zh-Hans`/`zh-Hant`, with migration for saved user preferences.

## Best Possible Solution

Make `zh-Hans` and `zh-Hant` the canonical Control UI locale IDs while keeping compatibility aliases for existing `zh-CN` and `zh-TW` saved values; update the runtime registry, browser detection, generator metadata, workflow matrix, docs, tests, and changelog without changing unrelated docs publishing internals unless the implementation requires it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/contract cleanup. The current behavior is high-confidence from source inspection: `zh-Hans`/`zh-Hant` are absent from the Control UI runtime surfaces and `zh-CN`/`zh-TW` remain asserted in source and tests.

## Solution Assessment

Yes, the narrow maintainable path is to make script IDs canonical in the Control UI runtime and normalize legacy region IDs at storage and browser-detection boundaries. A docs-only rename would not solve the runtime contract, and a hard rename without aliases would unnecessarily break saved settings.
