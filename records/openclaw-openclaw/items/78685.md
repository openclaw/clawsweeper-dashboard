## Summary

This should stay open: current main still has a source-level path where Telegram can inject a rewritten silent-reply fallback after an out-of-band `message.send` when direct source delivery remains in automatic mode.

## What This Changes

The issue reports that Telegram direct chats can receive an extra `Nothing further to report.` message after a successful `message.send` tool reply.

## Best Possible Solution

Treat successful messaging-tool delivery as a visible outcome for Telegram finalization so direct chats do not receive a synthetic silent-reply fallback after an out-of-band send.

## Reproduction Assessment

Yes, source-level: a direct automatic-mode turn with successful `message.send` and no final payload leaves Telegram's delivery summary empty and `suppressSilentReplyFallback` false, which reaches the rewritten `NO_REPLY` fallback path. I did not execute a live Telegram bot run.

## Solution Assessment

Yes, the narrow maintainable fix is to propagate or consult messaging-tool delivery evidence in the Telegram finalization path before injecting the silent fallback, rather than adding a new user-facing config switch.
