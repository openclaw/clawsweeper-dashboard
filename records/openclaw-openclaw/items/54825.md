## Summary

Keep this open. Current main still has user-visible billing and rate-limit fallback paths, WhatsApp has no suppress/custom/owner-only policy config, and the linked #69737 is related but narrower than this per-channel policy request.

## What This Changes

Request configurable per-channel suppression, custom text, or owner-only routing for provider/runtime error replies sent to WhatsApp and Telegram.

## Best Possible Solution

Add a shared outbound provider/runtime error-reply policy before channel delivery, with per channel/account/peer modes for silent suppression, custom text, normal reply, and owner-only routing while preserving full diagnostics in logs/operator surfaces.

## Reproduction Assessment

Yes. Source inspection plus existing tests give a high-confidence current-main reproduction path: pure billing fallback exhaustion in a WhatsApp-context run produces final billing text, and mid-turn rate-limit failures produce user-visible rate-limit text.

## Solution Assessment

No. Current main's sanitizer, WhatsApp isError suppression, and Telegram send-error hook are partial mitigations; the maintainable fix is a shared error-reply policy applied before any channel sends provider/runtime failure text.
