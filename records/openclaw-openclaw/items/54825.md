## Summary

Keep this open. Current main still has a source-reproducible path where billing fallback exhaustion is returned as ordinary final reply text before WhatsApp error-payload suppression can apply, WhatsApp has no accepted suppress/custom/owner-only config, and #69737 is related but narrower than this per-channel policy request.

## What This Changes

The issue requests per-channel controls to suppress, customize, or route provider/runtime error replies away from nontechnical WhatsApp and Telegram recipients.

## Best Possible Solution

Add a shared outbound provider/runtime error-reply policy before channel delivery, with per-channel/account/peer modes for suppression, custom text, normal reply, and owner-only routing while preserving full diagnostics in logs and operator surfaces.

## Reproduction Assessment

Yes. Source inspection on current main shows pure billing fallback exhaustion becomes an ordinary final reply payload without `isError`, and existing coverage exercises that WhatsApp-context path with mocked billing text.

## Solution Assessment

Unclear until maintainers choose the default behavior and owner-only routing semantics. The requested per-channel policy direction is plausible, but the maintainable fix should be a shared error-reply policy boundary rather than a WhatsApp-only string filter.
