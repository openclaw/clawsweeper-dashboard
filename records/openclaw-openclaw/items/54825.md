## Summary

Keep open. Current main has partial mitigations, but it still lacks the requested per-channel suppress/custom/owner-only provider-error policy, and related narrower issues/PRs do not cover the whole WhatsApp and Telegram behavior.

## What This Changes

Request per-channel controls to suppress, replace, or owner-route raw provider/runtime error replies on WhatsApp and Telegram.

## Best Possible Solution

Define a documented outbound provider/runtime error-reply policy before channel delivery, with per-channel or per-peer modes for normal reply, silent suppression, custom text, and owner-only routing while preserving raw diagnostics in logs and approved operator surfaces.

## Reproduction Assessment

Yes, source-level. Current main still builds final provider-failure reply text before channel policy, WhatsApp only filters payload.isError, and Telegram has only partial suppression controls.

## Solution Assessment

Unclear as an implementation, but the requested direction is valid. The maintainable path is a deliberate shared or channel-owned error-reply policy, not scattered text filters or silently changing only one channel default.
