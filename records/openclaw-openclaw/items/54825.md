## Summary

Keep this open. Current main has partial mitigations, especially WhatsApp suppressing text-only `isError` payloads, but the requested per-channel custom/suppress/owner-only provider-error policy is still not implemented and Telegram can still deliver error payload text.

## What This Changes

The issue requests suppressing, customizing, or owner-routing provider/runtime error messages before WhatsApp or Telegram users see them.

## Best Possible Solution

Add a documented outbound provider/runtime error-reply policy before channel delivery, with per-channel/account/peer modes for normal, silent suppression, custom text, and owner-only routing while preserving raw diagnostics in logs and approved operator surfaces.

## Reproduction Assessment

Yes. Source inspection and current tests show terminal billing/rate-limit failures still produce final error payload text, Telegram delivers `isError` payloads normally, and WhatsApp lacks the requested custom/owner-only config surface even though it now suppresses text-only error payloads.

## Solution Assessment

Yes in direction, but not yet as an implementation. The maintainable fix is a deliberate shared or channel-owned error-reply policy, not scattered sanitizer branches or reusing Telegram's delivery-error policy without clarifying semantics.
