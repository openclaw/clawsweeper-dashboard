## Summary

Keep this open. Current main still has source-backed paths that turn billing/rate-limit failures into user-visible reply text, WhatsApp has no suppress/custom/owner-only policy config, and the related #69737 tracks a narrower raw `errorMessage` fallback rather than this channel policy request.

## What This Changes

The issue requests a per-channel policy to suppress, customize, or owner-route raw provider/runtime error messages before they reach WhatsApp or Telegram users.

## Best Possible Solution

Add a shared outbound provider/runtime error-reply policy before channel delivery, with per channel/account/peer modes for normal reply, silent suppression, custom text, and owner-only routing.

## Reproduction Assessment

Yes. Source inspection plus existing tests give a high-confidence current-main path: billing fallback exhaustion and mid-turn rate-limit failures can become visible reply payloads, and WhatsApp lacks the requested policy config.

## Solution Assessment

No. The current partial mitigations are not the best endpoint; the maintainable solution is a shared outbound error-reply policy rather than one-off channel string filtering.
