## Summary

Keep this issue open. Current main still turns billing and rate-limit provider failures into user-visible final reply payloads, WhatsApp still has no suppress/custom/owner-only error policy, and Telegram’s existing error policy is applied to Telegram send failures rather than as a shared provider-error reply policy.

## What This Changes

The issue requests suppressible or configurable channel-level handling for provider/runtime error replies, especially on WhatsApp and Telegram, so non-technical recipients do not see internal billing or rate-limit text.

## Best Possible Solution

Implement a shared outbound provider/runtime error-reply policy before channel delivery. It should classify billing, rate-limit, auth, transport, and generic runtime failures; support per channel/account/chat/topic/DM modes for normal reply, silent suppression, custom text, and owner-only routing; preserve full diagnostics in logs/operator surfaces; and wire WhatsApp plus Telegram schema, docs, and tests consistently.

## Reproduction Assessment

Yes. Current tests and source provide a high-confidence reproduction path: pure billing fallback exhaustion produces billing text for a WhatsApp-context run, and mid-turn rate-limit failure produces an 'API rate limit reached' final reply; the latest issue comment adds live WhatsApp field evidence.

## Solution Assessment

No for current main. The existing sanitizer and Telegram send-error hook are partial mitigations, but the narrow maintainable fix is a shared outbound error-reply policy applied before any channel delivery, with channel-specific configuration only at the policy boundary.
