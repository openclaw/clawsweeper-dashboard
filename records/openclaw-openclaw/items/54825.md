## Summary

Keep open. Current main has partial mitigations, especially WhatsApp suppression for routed error payloads and Telegram delivery-error throttling, but it still lacks the requested per-channel suppress/custom/owner-only provider-error policy and the narrower raw-error fallback remains tracked separately.

## What This Changes

The issue asks for channel-safe handling of provider/runtime errors so WhatsApp and Telegram users do not receive raw or operator-focused billing/API failure text.

## Best Possible Solution

Add a documented outbound provider/runtime error-reply policy before channel delivery, with per-channel/account/peer modes for normal, silent suppression, custom text, and owner-only routing while preserving raw diagnostics in logs and approved operator surfaces.

## Reproduction Assessment

Yes. Source inspection shows provider and assistant failures still produce `isError` final payloads, WhatsApp only suppresses them without a configurable replacement/owner route, and Telegram coverage still asserts visible delivery for normal `isError` payload text.

## Solution Assessment

Yes in direction, but not yet as an implementation. The maintainable fix is a deliberate shared or channel-owned error-reply policy, not scattered sanitizer branches or silently changing Telegram's existing delivery-error controls.
