## Summary

Keep open: current main still lacks a native direct-parent-only send path for sub-agents. Announce-chain delivery, ACP-only parent streaming, and broad allowlist access to `sessions_send` do not enforce the requested parent-only, same-tree, no-chaining boundary.

## What This Changes

Request a native parent-scoped send path that lets sub-agents send progress updates or questions only to their direct parent session.

## Best Possible Solution

Add a narrow core parent-send path that derives the direct parent from trusted spawn/session metadata, enforces same-tree parent-only no-chaining behavior, and updates policy, resolver/send behavior, docs, and regression tests.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source inspection on current main gives high confidence that no native parent-only send tool or trusted parent alias exists while default sub-agent policy still denies `sessions_send`.

## Solution Assessment

Yes for a dedicated parent-scoped core API or trusted parent alias; no for broadly re-enabling raw `sessions_send`. The maintainable path should derive the direct parent server-side and enforce same-tree, parent-only, no-chaining behavior.

## Security Review

Status: needs_attention

Summary: Adding child-to-parent sends changes sub-agent cross-session authority and needs explicit same-tree targeting rules.

Concerns:

- **[medium] Derive the parent target server-side:** `src/agents/pi-tools.policy.ts:50`
  - body: Any implementation should derive the direct parent from trusted spawn/session metadata rather than accepting an arbitrary session key from the child; otherwise sub-agents could gain broader session-send authority than requested.
  - confidence: 0.92

  - body: The current visibility helper is requester-owned and descendant-oriented, so a child-to-parent send path must not turn tree visibility into broad transcript, sibling, ancestor, or cross-agent access.
  - confidence: 0.88
