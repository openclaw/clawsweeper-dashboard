## Summary

Current main still has no trusted child-to-direct-parent mid-run messaging path: native sub-agents are denied `sessions_send`, their dedicated control tool only lists runs, and completion announcements do not cover questions or progress during execution. The issue should remain open for an explicit product and security decision rather than treating the broader opt-in session-send configuration as equivalent.

## What This Changes

Add a trusted direct-parent-only mid-run messaging path for native sub-agents.

## Best Possible Solution

Expose one purpose-built parent-send operation backed by the canonical active sub-agent run record, reject missing or stale parent relationships, permit exactly one direct edge, and add authorization, nesting, reparenting, delivery, and documentation coverage.

## Reproduction Assessment

Yes, at source level. Current main unconditionally denies `sessions_send` to native sub-agents, offers no send action through `subagents`, and has no trusted `parent` session resolver.

## Solution Assessment

Yes in direction, but not yet in final API shape. A dedicated parent-only operation derived from trusted active-run metadata is safer and more maintainable than broadly re-enabling `sessions_send` or adding an unrestricted configuration flag.

## Risks / Open Questions

- The public API shape remains undecided: a dedicated tool and a reserved parent selector have different compatibility and discoverability consequences.
- Parent authority must come from the active canonical run record; using caller-provided session keys or broad visibility could expose siblings, ancestors, or another agent tree.
- Nested reparenting, completed runs, restarted runs, and stale registry records need explicit semantics so messages cannot reach a former parent.
- A separate mid-run path must coexist predictably with completion announcements and must not automatically forward messages farther up the hierarchy.
