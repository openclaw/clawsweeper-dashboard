## Label Justifications

- `P2`: This is a normal-priority feature request for sub-agent/session orchestration with limited blast radius.
- `impact:session-state`: The request changes how sub-agent sessions communicate with their parent session during an active run.
- `impact:security`: The requested parent-send path changes cross-session authority and must preserve same-tree boundaries.

## Summary

Keep open: current main and v2026.5.12 still lack a native direct-parent-only mid-run send path for sub-agents. Existing announce-chain delivery, ACP-only parent streaming, and broad opt-in access to raw `sessions_send` are adjacent primitives, but they do not provide the requested server-derived parent target with same-tree and no-chaining guarantees.

## What This Changes

Add a parent-scoped send path so sub-agents can send mid-run updates or questions only to their direct parent session.

## Best Possible Solution

Add a narrow core parent-send path that derives the direct parent from trusted spawn/session metadata, enforces same-tree parent-only no-chaining behavior, and updates policy, resolver/send behavior, docs, and regression tests.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source inspection on current main and v2026.5.12 gives high confidence that no native parent-only send tool or trusted parent alias exists while default sub-agent policy still denies `sessions_send`.

## Solution Assessment

Yes for a dedicated parent-scoped core API or trusted parent alias; no for broadly re-enabling raw `sessions_send`. Maintainers still need to choose the public shape and security rules before implementation.
