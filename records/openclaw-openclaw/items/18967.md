## Summary

Keep open: current main still denies `sessions_send` for sub-agents by default and does not provide a native direct-parent-only send tool or trusted `parent` resolver, while the issue is product/security-sensitive enough to remain with maintainers.

## What This Changes

The issue requests a parent-scoped mid-run send mechanism so sub-agents can message only the direct parent session without arbitrary cross-session access or upward chaining.

## Best Possible Solution

Design a dedicated parent-scoped send path that derives the direct parent from trusted spawn/session metadata, rejects sibling/ancestor/cross-agent targets, avoids automatic forwarding, and updates policy, resolver, docs, and regression tests together.

## Reproduction Assessment

Yes, from source inspection: current main still denies `sessions_send` to sub-agents by default, the send tool has no `sessions_send_parent`, and the resolver has no trusted `parent` alias. I did not run an end-to-end scenario because the requested capability is absent by design rather than a flaky runtime failure.

## Solution Assessment

Unclear until maintainers decide the API shape; a dedicated `sessions_send_parent` or a trusted `sessionKey: "parent"` resolver is the narrow maintainable direction, while a broad config flag that exposes unrestricted `sessions_send` would not meet the stated security constraints.
