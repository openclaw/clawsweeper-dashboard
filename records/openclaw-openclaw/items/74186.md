## Summary

Keep open. The PR targets a real current-main sandbox media URI bug and addresses the earlier missing host-path boundary check, but latest head still probes the inbound fallback by basename before applying the existing media URI authority and single-ID validation, and it is missing the required changelog entry.

## What This Changes

The PR adds early `media://` handling in `resolveSandboxedBridgeMediaPath`, resolves staged inbound fallback files through the sandbox bridge, and adds focused tests.

## Best Possible Solution

Land a narrow sandbox helper fix that detects `media://` before bridge resolution, probes staged fallback only for validated `media://inbound/<single decoded id>` references, returns raw URIs for downstream validation otherwise, covers edge cases, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: sandboxed image media goes through `resolveSandboxedBridgeMediaPath` before `loadWebMedia`, and POSIX path resolution corrupts `media://inbound/<id>` into a `/workspace/media:/inbound/<id>` path.

## Solution Assessment

No. Early media URI detection is the right direction, but the proposed fallback rewrite should reuse or mirror the existing `media://inbound/<decoded single id>` validation before `stat` or `resolvePath`.
