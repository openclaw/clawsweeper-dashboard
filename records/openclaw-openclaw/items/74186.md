## Summary

Keep open: the latest head fixed the initial workspace-boundary review point, but it still probes the inbound fallback for unsupported or nested `media://` URIs before preserving the existing inbound media URI contract.

## What This Changes

This PR adds early `media://` handling in `resolveSandboxedBridgeMediaPath` and tests sandboxed inbound fallback resolution.

## Best Possible Solution

Land a narrow sandbox helper fix that detects `media://` before bridge resolution, stages fallback only for validated `media://inbound/<single decoded id>` references, returns raw URIs or typed downstream errors otherwise, and includes the required changelog entry.

## Reproduction Assessment

Yes. Current main has a code-level reproduction path because sandboxed image inputs go through `resolveSandboxedBridgeMediaPath` before `loadWebMedia`; the PR-introduced validation bug can be reproduced by staging `media/inbound/foo.png` and passing `media://outbound/foo.png` or `media://inbound/a/foo.png`.

## Solution Assessment

No. Early detection is the right direction, but basename-based fallback probing is not the narrowest safe solution because it bypasses the existing host and decoded single-ID media URI checks.
