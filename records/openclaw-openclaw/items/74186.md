## Summary

Keep open. The current-main bug is real and the PR moves in the right direction, but PR head still probes the staged fallback by basename before preserving the existing media URI authority and single-ID validation, which can select unintended inbound files.

## What This Changes

The PR adds early `media://` handling to `resolveSandboxedBridgeMediaPath`, resolves staged inbound fallback files through the sandbox bridge, and adds focused tests.

## Best Possible Solution

Land a narrow sandbox helper fix that detects `media://` before bridge path resolution, accepts staged fallback only for validated `media://inbound/<single decoded id>` references, returns raw URIs for downstream validation otherwise, covers malformed/unsupported/nested cases, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: sandboxed image media goes through `resolveSandboxedBridgeMediaPath` before `loadWebMedia`, and POSIX path resolution corrupts `media://inbound/<id>` into a `/workspace/media:/inbound/<id>` path.

## Solution Assessment

No. Early media URI detection is the right direction, but basename-based fallback probing is not the narrowest safe fix because it bypasses the existing host and decoded single-ID checks; the helper should reuse or mirror that validation before `stat`/`resolvePath`.

## Review Findings

Full review comments:

- **[P2] Validate media URIs before probing inbound fallback:** `src/agents/sandbox-media-paths.ts:45-46`
  - body: The new branch probes `media/inbound/<basename>` for every `media://` URL. That lets inputs such as `media://outbound/foo.png`, `media://inbound/a/b.png`, or encoded nested IDs select a staged inbound file instead of following the existing `resolveInboundMediaReference` host and single-segment ID checks. Parse and validate `media://inbound/<decoded id>` before probing; otherwise return the raw URI for downstream validation.
  - confidence: 0.91

  - body: `new URL(filePath)` runs before the fallback `try`, so malformed `media://` inputs throw a raw URL parse error whenever an inbound fallback dir is configured. Guard the parse and fall through to the raw URI or the existing typed media-reference validation path.
  - confidence: 0.78
