## Summary

Keep open. This PR targets a real sandbox media URI bug on current main and the latest head added host workspace-boundary enforcement, but the patch still rewrites arbitrary `media://` references by basename before applying the existing media URI validation contract, so it should not merge yet.

## What This Changes

The PR adds early `media://` handling to `resolveSandboxedBridgeMediaPath`, attempts staged inbound fallback resolution through the sandbox bridge, returns raw URIs on fallback miss, and adds helper tests.

## Best Possible Solution

Land a narrow sandbox helper fix that detects `media://` before bridge path resolution, accepts staged fallback only for validated `media://inbound/<single decoded id>` references, returns raw URIs for downstream validation otherwise, covers malformed/unsupported/nested cases, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main has a code-level reproduction path through sandboxed media tools: `media://inbound/<id>` reaches the sandbox bridge resolver before media-store handling, where POSIX path resolution can corrupt it into a `media:/inbound` workspace path. The PR-introduced validation bug is also reproducible by staging `media/inbound/foo.png` and passing an unsupported or nested `media://` URI.

## Solution Assessment

No. Early media URI detection is the right direction, but basename-based fallback probing is not the safest fix because it bypasses the existing authority and decoded single-ID checks; the helper should reuse or mirror that parser before stat/resolve.
