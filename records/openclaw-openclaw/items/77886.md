## Summary

Keep open: current main and the latest release still hard-block agent-initiated protected config edits, and there is no first-class protected-config proposal or owner-approval protocol. The older broad approval request was closed in favor of this issue, https://github.com/openclaw/openclaw/issues/77439 tracks a distinct configurable allowlist policy, and the merged policy conformance work does not apply approved protected config changes.

## What This Changes

The issue requests an owner-approved, auditable flow for agent-proposed protected config changes such as enabling commands.mcp and registering MCP servers.

## Best Possible Solution

Design a core Gateway protected-config proposal protocol that keeps the fail-closed default, adds exact owner approval for promptable paths, preserves never-allow security boundaries, records audit metadata, and coordinates policy knobs with https://github.com/openclaw/openclaw/issues/77439.

## Reproduction Assessment

Yes at source level: current main rejects changed config paths outside the fixed agent allowlist, and guard coverage explicitly blocks mcp.servers through config.patch. I did not run a live Gateway session because the source and tests already match the reported failure path without writing to the checkout.

## Solution Assessment

Unclear until maintainer security design: an owner-approved flow is plausible and aligned with the problem, but the permanent solution must define approval scopes, never-allow paths, channel identity, audit records, and its relationship to configurable allowlist policy.
