## Summary

Keep open: current main and the latest release still hard-block agent-initiated protected config edits, and the requested owner-approved proposal/apply flow is a real security/product design gap rather than stale cleanup. The older broad approval request was already closed in favor of this issue, while https://github.com/openclaw/openclaw/issues/77439 separately tracks configurable trusted-agent allowlist policy.

## What This Changes

The issue requests a first-class owner-approved, auditable Gateway flow for protected config mutations such as enabling `/mcp` and registering MCP servers.

## Best Possible Solution

Design a core Gateway protected-config proposal protocol that keeps the fail-closed default, adds exact owner approval for promptable paths, preserves never-allow security boundaries, records audit metadata, and coordinates policy knobs with https://github.com/openclaw/openclaw/issues/77439.

## Reproduction Assessment

Yes at source level: current main rejects changed config paths outside the fixed agent allowlist, and guard coverage explicitly blocks `mcp.servers` via `config.patch`. I did not run a live Gateway reproduction, but the source and tests match the reported failure path.

## Solution Assessment

Unclear until maintainer security design: an owner-approved flow is plausible and aligned with the problem, but the permanent solution must define approval scopes, never-allow paths, channel identity, audit records, and the relationship to the configurable allowlist work.

## Risks / Open Questions

- The approval design must classify promptable, stronger-confirmation, and never-promptable paths so auth, exec, plugin-loading, hooks, browser, network, and MCP command surfaces do not become a blanket escalation path.
- Approvals need to bind the exact base config hash, patch/diff, changed paths, requester/session, expiry, and approver identity to avoid stale or swapped-patch application.
- Channel identity and owner proof need product/security design because an approval from a group chat is not equivalent to a local TUI, Control UI, or authenticated owner DM.
- MCP-specific review needs to distinguish harmless server config from command execution, env/secret references, cwd, executable provenance, tool discovery, and destructive tool exposure.
