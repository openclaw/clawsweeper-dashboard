## Summary

Keep open: current main still hard-blocks agent-initiated protected config changes, and the requested owner-approved path is a security-sensitive feature/product design rather than closeable cleanup.

## What This Changes

Request to add a first-class owner approval flow for agent-initiated protected config changes such as enabling `/mcp` and registering `mcp.servers` entries.

## Best Possible Solution

Design a core Gateway protected-config proposal and approval contract that binds approval to the exact patch, base hash, requesting session, owner identity, expiry, and risk class while keeping protected paths default-denied.

## Reproduction Assessment

Yes for the current limitation: source inspection shows agent-facing `gateway config.patch/apply` rejects `commands.mcp` and `mcp.servers` because they are outside the allowlist. I did not run a live Gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No implemented solution exists yet, and the best fix is not a simple allowlist expansion. A maintainable solution needs a first-class protected-config approval flow that preserves the model/operator boundary.
