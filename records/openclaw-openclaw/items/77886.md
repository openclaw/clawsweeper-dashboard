## Summary

Keep open: current main and v2026.6.10 still fail-closed on agent-initiated protected config edits, and no owner-approved protected-config proposal or config-approval protocol exists yet. This remains a security-sensitive product/design item, not stale cleanup.

## What This Changes

Request a first-class owner approval flow for agent-proposed protected config mutations, especially enabling `/mcp` and registering `mcp.servers`, with exact diff review, audit metadata, and safe apply/reload handling.

## Best Possible Solution

Design a core Gateway protected-config proposal protocol that keeps fail-closed defaults, classifies never-allow paths, prompts owners only for approved path classes, records audit metadata, and coordinates with the separate configurable allowlist policy in https://github.com/openclaw/openclaw/issues/77439.

## Reproduction Assessment

Yes, at source level: current main rejects changed paths outside the fixed agent allowlist, and tests explicitly block mcp.servers through config.patch. I did not run a live Gateway session because the review is read-only and the source/test path matches the reported failure.

## Solution Assessment

Unclear until maintainer security design. An owner-approved protected-config flow is plausible, but the final solution must define path classes, owner identity, audit records, patch binding, and how it differs from configurable allowlist policy.

- The approval design must separate promptable, stronger-confirmation, and never-promptable config paths so auth, exec, plugin loading, browser, network, hooks, and MCP command surfaces do not become a blanket escalation path.

- MCP-specific approval review needs to inspect command, args, cwd, env/SecretRef use, executable provenance, tool discovery, and destructive tool exposure before applying config.
