## Summary

Keep open: current main and v2026.6.8 still hard-block agent-initiated protected config edits, and there is no first-class protected-config proposal or owner-approval protocol. The older broad approval request was closed in favor of this issue, https://github.com/openclaw/openclaw/issues/77439 tracks a distinct configurable allowlist policy, and the merged policy conformance PR does not apply approved protected config changes.

## What This Changes

The issue requests a first-class owner approval flow for agent-proposed protected config changes, especially enabling `/mcp` and adding MCP servers without raw `openclaw.json` edits.

## Best Possible Solution

Design a core Gateway protected-config proposal protocol that keeps the fail-closed default, adds exact owner approval for promptable paths, preserves never-allow security boundaries, records audit metadata, and coordinates policy knobs with https://github.com/openclaw/openclaw/issues/77439.

## Reproduction Assessment

Yes at source level: current main rejects changed config paths outside the fixed agent allowlist, and guard coverage explicitly blocks `mcp.servers` through `config.patch`. I did not run a live Gateway session because this review is read-only and the source/tests match the reported failure path.

## Solution Assessment

Unclear until maintainer security design: an owner-approved flow is plausible and aligned with the problem, but the permanent solution must define approval scopes, never-allow paths, channel identity, audit records, and its relationship to configurable allowlist policy.
