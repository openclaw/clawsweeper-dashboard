## Summary

Keep open: current main still hard-blocks agent-initiated protected config changes, and the requested owner-approved path is a security-sensitive feature/product design rather than a stale or already-implemented cleanup item.

## What This Changes

Request an owner-approved, auditable Gateway flow for agents to propose protected config edits such as enabling `/mcp` and adding `mcp.servers` entries.

## Best Possible Solution

Design a core Gateway protected-config proposal and approval contract that binds approval to the exact patch, base hash, requesting session, owner identity, and risk class while keeping protected paths default-denied.

## Reproduction Assessment

Yes for the current limitation: source inspection and guard coverage show agent-facing `gateway config.patch/apply` rejects `commands.mcp` and `mcp.servers` because they are outside the allowlist. I did not run a live Gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No implemented solution exists yet, and the best fix is not a simple allowlist expansion. A maintainable solution needs a first-class protected-config approval flow that preserves the model/operator boundary.
