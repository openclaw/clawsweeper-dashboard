## Summary

Keep open: the PR addresses a real multi-agent Usage page regression, but it fixes the symptom by changing omitted `agentId` from the shipped effective/default-agent scope to all-agent gateway discovery, which reopens the session-data boundary that recent hardening intentionally narrowed.

## What This Changes

The PR changes `sessions.usage` so list requests without `agentId` return sessions from all configured agents and updates the gateway usage unit test to expect that behavior.

## Best Possible Solution

Keep omitted `agentId` scoped in the gateway, and fix the Control UI through an explicit selected-agent request or a maintainer-approved explicit all-agents contract with tests and real proof for the chosen behavior.

## Reproduction Assessment

Yes. Source inspection shows the Web UI sends `sessions.usage` without `agentId`, current main scopes that to the default agent, and the filter options are built from the returned sessions; I did not run a live multi-agent Control UI repro in this read-only pass.

## Solution Assessment

No. The PR fixes the visible symptom by broadening the server default, but the narrower maintainable path is to preserve the gateway scope boundary and make the UI or protocol request the intended agent scope explicitly.
