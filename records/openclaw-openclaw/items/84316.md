## Summary

Close as fixed on current main: the reported raw `[[tts]]` leak and missing voice delivery map to the message-tool/group delivery TTS bypass, and current main now runs tagged message sends through the shared TTS normalizer before Telegram or plugin delivery.

## What This Changes

The issue reports Telegram group sub-agent TTS directives being rendered as text with no voice message on OpenClaw 2026.5.12, plus a question about Telegram account-level TTS overrides.

## Best Possible Solution

Keep the current shared TTS normalization for both automatic replies and message-tool sends; if Telegram account-level TTS overrides are desired, track that as a narrower config/docs decision rather than keeping this delivery bug open.

## Reproduction Assessment

No live current-main reproduction was run. The 2026.5.12 failure is source-reproducible through the previously missing message-tool TTS normalization path, and current main now routes that path through the shared TTS pipeline.

## Solution Assessment

Yes. Reusing the shared `maybeApplyTtsToPayload` pipeline before message-tool/core/plugin delivery is the narrow maintainable fix because it handles directive stripping, synthesis, channel voice flags, and account/agent context without adding a Telegram-specific workaround.
