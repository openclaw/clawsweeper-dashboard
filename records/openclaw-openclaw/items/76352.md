## Summary

Keep open: current main still emits the reported warning when the initial Unix stale-PID `lsof` spawn returns ENOENT, and the linked open PR is the right active implementation candidate.

## What This Changes

The issue asks for gateway restart startup to avoid warning when `lsof` is missing during the stale PID scan.

## Best Possible Solution

Review and land PR #76364, or an equivalent narrow patch that treats unavailable `lsof` spawn errors as expected only in the initial scan while preserving warnings for real lsof failures.

## Reproduction Assessment

Yes. Source inspection shows a non-Windows `findGatewayPidsOnPortSync` call with `spawnSync` returning ENOENT reaches `restartLog.warn`, and the current test suite explicitly expects that warning.

## Solution Assessment

Yes. Mirroring the existing `pollPortOnce` ENOENT/EACCES/EPERM unavailable-lsof classification in the initial scan is the narrowest maintainable fix; the linked PR is the active candidate to review.
