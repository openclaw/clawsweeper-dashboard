## Summary

Keep open: current main still emits the noisy missing-lsof warning, and open PR #76364 is already linked as the implementation candidate, so this issue should not be closed before that fix merges or is rejected.

## What This Changes

Suppress the initial gateway restart stale-PID lsof warning when lsof is unavailable on minimal Linux systems.

## Best Possible Solution

Land #76364 or an equivalent narrow restart cleanup that treats missing or unavailable lsof as an expected initial-scan skip while preserving warnings for real lsof failures.

## Reproduction Assessment

Yes. Source inspection against current main shows `resolveLsofCommandSync` can return bare `lsof`, and the initial scan warns on the resulting ENOENT-style spawn error; the existing unit test currently asserts that warning.

## Solution Assessment

Yes. Differentiating ENOENT, EACCES, and EPERM in the initial scan is the narrowest maintainable fix, and the linked PR appears scoped to that behavior while leaving broader stale-PID fallback semantics alone.
