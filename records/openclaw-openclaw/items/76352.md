## Summary

Keep open: current main still emits the reported initial stale-PID `lsof` warning path, and an open linked fix PR already uses closing syntax for the same narrow repair.

## What This Changes

The issue asks for gateway restart cleanup to silently skip the initial stale-PID `lsof` scan when `lsof` is unavailable.

## Best Possible Solution

Review and land PR #76364, or an equivalent narrow patch that treats unavailable `lsof` spawn errors as expected while preserving warnings for real lsof failures.

## Reproduction Assessment

Yes. Source inspection shows a non-Windows `findGatewayPidsOnPortSync` call with a missing `lsof` binary reaches the warning path, and the current test suite explicitly expects that warning.

## Solution Assessment

Yes. Mirroring the existing `pollPortOnce` unavailable-lsof classification in the initial scan is the narrowest maintainable fix; the linked PR is the right implementation candidate to review before closing this issue.
