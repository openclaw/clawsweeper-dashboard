## What This Changes

The issue asks OpenClaw gateway restart startup to avoid warning when the initial stale-PID `lsof` scan runs on a system where `lsof` is not installed.

## Best Possible Solution

Refresh, prove, and land the focused linked PR or a narrow replacement that classifies permanent unavailable-`lsof` spawn errors in the initial scan while preserving warnings for unexpected failures.

## Reproduction Assessment

Yes, at source level: current main can resolve to bare `lsof`, Node returns ENOENT for a missing command, and the initial scan logs the reported warning for any spawn error. I did not run a full minimal-container gateway startup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to share the existing permanent unavailable-`lsof` classification with the initial scan, while keeping warnings for unexpected spawn errors and nonzero statuses.
