## Summary

Keep open: current main still emits the reported missing-lsof warning, and open PR #76364 is the active implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request suppressing the misleading WARN when gateway startup runs the initial Unix stale-PID lsof scan on systems where lsof is absent.

## Best Possible Solution

Review and land #76364, or an equivalent narrow change, so ENOENT/EACCES/EPERM from the initial lsof scan are treated as expected lsof-unavailable states while warnings remain for unexpected failures.

## Reproduction Assessment

Yes, from source inspection: mock the non-Windows findGatewayPidsOnPortSync spawnSync result with an ENOENT error and the current code logs the reported warning. The existing test suite already covers that path and currently expects the warning.

## Solution Assessment

Yes: sharing the permanent-unavailable classification already used by the later polling path is the narrowest maintainable fix. The open PR #76364 matches that direction by skipping ENOENT/EACCES/EPERM without suppressing unexpected lsof failures.
