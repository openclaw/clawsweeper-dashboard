## Summary

Keep open: current main still emits the reported warning when the initial Unix stale-PID lsof spawn returns an error, and linked PR #76364 is still the active implementation candidate.

## What This Changes

Suppress the misleading initial stale-PID lsof warning on minimal systems where lsof is unavailable while preserving gateway startup fallback behavior.

## Best Possible Solution

Review and land PR #76364, or an equivalent narrow patch that treats unavailable lsof in the initial scan like the later polling path while preserving diagnostics for real failures.

## Reproduction Assessment

Yes. Source inspection shows a non-Windows initial stale-PID scan with a missing `lsof` command reaches the warning path, and the current test suite explicitly expects that warning for an ENOENT-style error.

## Solution Assessment

Yes. Sharing the existing unavailable-lsof classification with the initial scan is the narrowest maintainable fix, and linked PR #76364 is the right implementation candidate to review before closing this issue.
