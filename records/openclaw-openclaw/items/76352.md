## Summary

Keep open: current main still emits the reported warning when the initial Unix stale-PID scan gets a spawn error from `lsof`, and an open linked PR already targets this issue with closing syntax.

## What This Changes

Request to silence the misleading gateway startup WARN when `lsof` is unavailable during the restart stale-PID scan.

## Best Possible Solution

Review and land #76364 if its diff matches the narrow ENOENT/EACCES/EPERM handling, or replace it with an equivalent focused fix; this issue should close only after that fix merges.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: make the non-Windows `findGatewayPidsOnPortSync` `spawnSync` call return an ENOENT error and current main logs the reported warning.

## Solution Assessment

Yes. Treating missing or inaccessible `lsof` as an expected unavailable state in the initial scan, while preserving warnings for real lsof failures, is the narrowest maintainable fix and matches the existing polling-path contract.
