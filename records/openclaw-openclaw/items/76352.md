Suppress the misleading initial stale-PID `lsof` warning when `lsof` is missing while preserving gateway startup and stale-process cleanup behavior.

## Best Possible Solution

Refresh, prove, and land the focused linked PR or a narrow replacement that treats ENOENT/EACCES/EPERM as expected unavailable-`lsof` states in the initial scan while preserving warnings for unexpected errors.

## Reproduction Assessment

## Solution Assessment

Yes, the narrow maintainable fix is to share the existing permanent unavailable-`lsof` classification with the initial scan rather than adding a separate binary preflight path. The linked PR is the right active path, pending refresh and real startup proof.
