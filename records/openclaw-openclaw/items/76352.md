## Summary

The reported behavior is still present on current main and in v2026.6.6: service-mode startup can call the initial stale-PID `lsof` scan, and that scan still warns for every spawn-level error including missing `lsof`. Keep this issue open because the linked closing PR is still open and unmerged, so the right next step is to review or land that PR rather than close the issue early.

## What This Changes

The issue asks OpenClaw to suppress the misleading restart stale-PID warning when `lsof` is not installed while preserving fallback behavior.

## Best Possible Solution

Land the focused linked restart fix after targeted validation and real no-`lsof` startup proof, preserving warnings for unexpected spawn failures and nonzero lsof exits.

## Reproduction Assessment

Yes from source inspection, with medium confidence: on Unix, `resolveLsofCommandSync()` can fall back to bare `lsof`, and `findGatewayPidsOnPortSync()` logs the reported warning for any spawn error including ENOENT. I did not run a live minimal Debian/Ubuntu/Alpine gateway start in this read-only review.

## Solution Assessment

Yes, the narrow fix shape is to classify missing or inaccessible `lsof` as an expected unavailable state in the initial scan while preserving warnings for unexpected failures. The open linked PR targets that boundary, so this issue should stay open until that PR lands or is replaced.
