## Summary

Keep open: current main still emits the reported warning when the initial Unix stale-PID lsof spawn returns ENOENT, and PR #76364 remains the linked implementation candidate rather than a merged fix.

## What This Changes

The issue asks for gateway restart cleanup to silently skip the initial lsof stale-PID scan when lsof is unavailable while preserving fallback behavior.

## Best Possible Solution

Land PR #76364 or an equivalent narrow change that classifies ENOENT, EACCES, and EPERM from the initial lsof scan as expected unavailable-lsof states while retaining warnings for real lsof failures.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-Windows initial stale-PID scan with spawnSync returning ENOENT reaches the warning path, and the existing unit test still asserts that warning.

## Solution Assessment

Yes. Reusing the existing unavailable-lsof classification from the later polling path is the narrowest maintainable fix, and the linked open PR is the right candidate to review before closing this issue.
