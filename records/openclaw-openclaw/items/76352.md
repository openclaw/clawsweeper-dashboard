## Summary

Keep open: current `main` still has the reported noisy initial `lsof` warning path, and the linked open PR #76364 already tracks the narrow fix with closing syntax.

## What This Changes

The issue requests that gateway restart stale-PID cleanup treat missing `lsof` as an expected fallback state instead of warning during the initial scan.

## Best Possible Solution

Review and land PR #76364, or an equivalent narrow patch that shares the unavailable-`lsof` classification with the initial scan while preserving warnings for unexpected failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on a non-Windows system where `spawnSync("lsof", ...)` returns ENOENT, `findGatewayPidsOnPortSync` still reaches the warning branch, and the current test suite expects that warning.

## Solution Assessment

Yes. Mirroring the existing `pollPortOnce` ENOENT/EACCES/EPERM unavailable-`lsof` handling in the initial scan is the narrowest maintainable fix, and the linked PR is the active implementation candidate.
