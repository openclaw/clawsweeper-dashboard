## Summary

Keep open: current main still warns on a missing `lsof` during the initial Unix stale-PID scan, and the issue is a narrow, source-reproducible logging bug rather than implemented or obsolete.

## What This Changes

Request that gateway restart stale-PID cleanup silently skip the initial `lsof` scan when the `lsof` binary is not installed.

## Best Possible Solution

Land a narrow restart cleanup fix that treats missing `lsof` in the initial stale-PID scan as an expected skip while preserving warnings for real lsof failures and existing stale-process semantics.

## Reproduction Assessment

Yes from source inspection: mock or run the non-Windows `findGatewayPidsOnPortSync` path with `spawnSync` returning ENOENT and it emits the reported warning. I did not run a live minimal OS gateway in this read-only review.

## Solution Assessment

Yes for the central cosmetic bug: differentiating ENOENT/EACCES/EPERM in the initial scan and returning quietly, optionally at debug level, is the narrowest maintainable fix. Adding a new stale-PID discovery fallback should be treated as separate scope unless maintainers explicitly want that behavior change.
