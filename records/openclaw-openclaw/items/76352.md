## Summary

Keep open: current main still reaches the reported warning path when the initial Unix stale-PID lsof spawn returns ENOENT, and #76364 is an open closing PR rather than a merged fix.

## What This Changes

Treat missing or unexecutable lsof during the initial gateway stale-PID scan as expected, skipping the misleading restart warning while preserving fallback behavior.

## Best Possible Solution

Review and land #76364, or an equivalent narrow patch that mirrors pollPortOnce unavailable-lsof handling in the initial scan while keeping warnings for unexpected lsof failures.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveLsofCommandSync can return bare `lsof`, and findGatewayPidsOnPortSync logs the reported warning when spawnSync returns an ENOENT-style error.

## Solution Assessment

Yes. The narrow maintainable fix is to classify ENOENT/EACCES/EPERM as unavailable lsof states in the initial scan, matching the later polling path without changing gateway cleanup policy.
