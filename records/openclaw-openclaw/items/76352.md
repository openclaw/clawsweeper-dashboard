## Summary

Keep open: current main still emits the reported initial lsof warning path, and PR #76364 is already open as the narrow closing implementation candidate.

## What This Changes

The issue asks gateway restart startup to silently skip the initial Unix lsof stale-PID scan when lsof is unavailable while preserving fallback behavior.

## Best Possible Solution

Review and land PR #76364, or an equivalent narrow patch that treats ENOENT/EACCES/EPERM from the initial lsof scan as expected unavailable-lsof states while preserving warnings for real lsof failures.

## Reproduction Assessment

Yes. Source inspection on current main shows non-Windows findGatewayPidsOnPortSync logs the reported warning whenever spawnSync returns an error, and the colocated test still expects that warning for an ENOENT-like initial scan failure.

## Solution Assessment

Yes. Mirroring pollPortOnce's ENOENT/EACCES/EPERM permanent-unavailable handling in the initial scan is the narrowest maintainable fix, with PR #76364 already open as the active candidate.
