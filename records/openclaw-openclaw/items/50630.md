## Summary

Keep open. Current main and the latest release still accept the reported Tailscale Serve plus explicit no-auth configuration, and the linked closing PR remains open, conflicting, and unmerged.

## What This Changes

Reject the unsafe combination of Tailscale Serve with explicit Gateway no-auth so Tailnet peers cannot reach the Gateway control plane without credentials.

## Best Possible Solution

Centralize a fail-closed validation that rejects `tailscale.mode: "serve"` with `gateway.auth.mode: "none"` in runtime and install/onboarding preflight, preserving loopback-only no-auth and adding regression/docs coverage.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none without credentials.

## Solution Assessment

Yes for the fail-closed direction. The maintainable fix should cover both gateway startup and shared install/onboarding preflight rather than only adding a runtime-only guard.
