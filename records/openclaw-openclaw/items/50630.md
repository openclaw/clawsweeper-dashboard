## Summary

Keep open. Current main and the latest shipped release still accept the reported Tailscale Serve plus explicit no-auth configuration, and the linked fixing PR is open, conflicting, and unmerged.

## What This Changes

The issue reports that `gateway.auth.mode="none"` combined with `gateway.tailscale.mode="serve"` can expose the Gateway through Tailscale Serve without credentials.

## Best Possible Solution

Review, rebase, prove, and land or replace https://github.com/openclaw/openclaw/pull/50631 with shared runtime and install/onboarding preflight validation that rejects explicit no-auth plus Tailscale Serve while preserving loopback-only no-auth.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

Yes, fail-closed validation is the right direction, but it should be shared across runtime startup and install/onboarding preflight so the dangerous config is rejected before service plans are accepted.
