## Summary

Keep open. Current main and the latest release still accept the reported Tailscale Serve plus explicit no-auth configuration, and the open fixing PR remains unmerged and conflicting.

## What This Changes

The issue asks OpenClaw to reject or otherwise protect `gateway.auth.mode="none"` when Tailscale Serve exposes the loopback Gateway to a Tailnet.

## Best Possible Solution

Harden Gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure, while preserving trusted loopback-only no-auth use and updating runtime-config tests plus public gateway/security docs for the final contract.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none without credentials.

## Solution Assessment

Yes, the fail-closed direction is the narrow maintainable fix. The existing linked PR may be the implementation path, but maintainers still need to resolve its conflict/proof state before landing it.
