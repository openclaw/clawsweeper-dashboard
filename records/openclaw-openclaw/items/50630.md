## Summary

Keep open. Current main still accepts the reported Tailscale Serve plus no-auth runtime configuration, `mode: "none"` still authorizes before verified Tailscale identity checks, and the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that `gateway.auth.mode="none"` combined with `gateway.tailscale.mode="serve"` exposes Gateway APIs to Tailnet peers without credentials.

## Best Possible Solution

Harden Gateway startup and install/onboarding preflight so Tailscale Serve cannot be combined with explicit no-auth, while preserving trusted local loopback no-auth and updating regression tests plus docs.

## Reproduction Assessment

Yes at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none without credentials.

## Solution Assessment

Yes. A startup-time rejection plus matching install/onboarding preflight is the narrow maintainable fix for this auth-boundary gap; the broader optional secondary-auth idea is separately tracked at https://github.com/openclaw/openclaw/issues/57110.
