## Summary

Current main still permits the reported `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` startup path: runtime validation only rejects non-password auth for Funnel, `mode:none` still authorizes before Tailscale identity handling, and an existing test still asserts this Serve/no-auth shape succeeds. This issue should remain open while the linked fix PR is reviewed because https://github.com/openclaw/openclaw/pull/50631 uses closing syntax and remains unmerged.

## What This Changes

Reject or otherwise harden the dangerous combination of Tailscale Serve exposure with Gateway auth mode set to `none`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/50631 so startup and any install/onboarding preflight reject explicit no-auth Gateway configuration whenever Tailscale Serve would expose it beyond loopback, while preserving trusted loopback-only local use and documenting the final trust model.

## Reproduction Assessment

Yes for source-level reproduction: current main still has a deterministic config path with `gateway.auth.mode="none"` and `gateway.tailscale.mode="serve"` that passes runtime validation, and request authorization returns success for mode none before credential checks. I did not run a live Tailnet end-to-end test in this read-only sweep.

## Solution Assessment

Unclear as a final product decision: rejecting Serve plus `mode:none` is the narrowest conservative hardening direction, but maintainers should confirm the intended Tailscale identity trust model and cover preflight paths before merging.
