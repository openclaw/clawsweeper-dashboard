## Summary

Keep open. Current main still accepts the reported Tailscale Serve plus `gateway.auth.mode="none"` startup shape, the auth path still returns success for mode none before Tailscale identity-header auth can run, and open PR #50631 remains an unmerged implementation candidate that references this issue with closing syntax.

## What This Changes

The issue requests closing the gateway auth-boundary gap where Tailscale Serve can expose a no-auth Gateway to Tailnet peers.

## Best Possible Solution

Harden Gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing HTTP contract, while preserving trusted loopback-only no-auth use and updating runtime-config tests plus public Gateway security docs.

## Reproduction Assessment

Yes. The deterministic reproduction is the config shape `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main still has a runtime-config test that asserts this shape starts successfully with loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` is the narrowest maintainable fix for the reported boundary gap, with tests proving Serve rejects mode none while loopback-only no-auth remains allowed.
