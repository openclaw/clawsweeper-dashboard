## Summary

Keep open. Current main still accepts the reported `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` startup shape, `auth.mode === "none"` still returns success before Tailscale identity-header auth can run, and open PR #50631 is still the unmerged implementation candidate for this same auth-boundary issue.

## What This Changes

The issue reports that combining `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` exposes Gateway APIs to Tailnet peers without a shared secret or verified HTTP identity path.

## Best Possible Solution

Harden Gateway startup so explicit no-auth mode cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing HTTP contract, while preserving trusted loopback-only no-auth usage and updating runtime-config tests plus public Gateway/Tailscale security docs for the final contract.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is the config shape `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape in runtime config and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` rejecting Serve plus no-auth is the narrowest maintainable fix for the reported boundary gap, with tests proving Serve rejects `mode: "none"` while loopback-only no-auth remains allowed.
