## Summary

Keep open. Current main still accepts the reported Serve plus no-auth startup shape, the auth path still accepts `mode: "none"` before Tailscale identity verification can run, and open PR #50631 remains the paired implementation candidate.

## What This Changes

Reject or otherwise harden the `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` configuration so Tailnet exposure does not bypass Gateway authentication.

## Best Possible Solution

Harden Gateway startup so explicit no-auth mode cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing HTTP contract, while preserving loopback-only no-auth and updating tests/docs for the final contract.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` rejecting Serve plus no-auth is the narrowest maintainable fix for the reported boundary gap, with tests proving Serve rejects `mode: "none"` while local loopback no-auth remains allowed.
