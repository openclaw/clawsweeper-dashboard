## Summary

Keep open. Current main and the latest release still accept explicit `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, the request auth path then accepts mode none without credentials, and the linked fixing PR remains open, conflicting, and unmerged.

## What This Changes

The issue reports a Gateway auth-boundary bug where Tailscale Serve plus explicit no-auth can expose Gateway APIs to Tailnet peers without credentials.

## Best Possible Solution

Review, rebase, prove, and land or replace https://github.com/openclaw/openclaw/pull/50631 with shared validation that rejects explicit no-auth plus Tailscale Serve before runtime or install flows accept it, while preserving trusted loopback-only no-auth use.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

Yes, fail-closed validation for explicit no-auth plus Tailscale Serve is the narrow maintainable direction. The accepted path should also cover shared preflight/install validation so operators do not get an install that fails only when the daemon starts.
