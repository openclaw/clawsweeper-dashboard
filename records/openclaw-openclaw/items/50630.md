## Summary

Keep open. Current main still accepts the reported Tailscale Serve plus no-auth startup shape, the auth path still returns success for mode none before verified Tailscale identity checks, and the linked fix PR remains open and unmerged.

## What This Changes

The issue reports that `gateway.auth.mode="none"` can be combined with `gateway.tailscale.mode="serve"`, exposing Gateway APIs to Tailnet peers without credentials.

## Best Possible Solution

Harden Gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure, preserve trusted loopback-only no-auth use, and update runtime-config tests plus public Gateway/Tailscale docs for the final contract.

## Reproduction Assessment

Yes. The source-level path is `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape in runtime config and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard rejecting Serve plus no-auth is the narrowest maintainable fix for this boundary gap; the broader optional secondary-auth design remains separately tracked at https://github.com/openclaw/openclaw/issues/57110.
