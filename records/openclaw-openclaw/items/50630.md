## Summary

Keep open. Current main still accepts the reported `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` startup shape, the HTTP auth path still treats mode none as successful before any Tailscale identity verification, and the linked fix PR is still open and unmerged.

## What This Changes

Reject or otherwise harden the gateway when explicit no-auth mode is combined with Tailscale Serve exposure to the tailnet.

## Best Possible Solution

Harden gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure, while preserving loopback-only no-auth use and updating runtime-config tests plus public gateway security docs for the final contract.

## Reproduction Assessment

Yes. The source-level reproduction is the config shape `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` rejecting Serve plus no-auth is the narrowest maintainable fix for this boundary gap; broader optional secondary auth belongs to the separate enhancement thread.
