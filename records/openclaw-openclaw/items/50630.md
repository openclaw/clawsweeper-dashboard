## Summary

Keep open. Current main and the latest release still accept explicit `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, the auth path still grants `method: "none"` before Tailscale identity or shared-secret checks, and the linked fixing PR remains open and unmerged.

## What This Changes

The issue requests blocking Tailscale Serve remote exposure when Gateway auth is explicitly disabled with `gateway.auth.mode="none"`.

## Best Possible Solution

Review, rebase, prove, and land or replace https://github.com/openclaw/openclaw/pull/50631 with shared validation that rejects explicit no-auth plus Tailscale Serve before runtime or install flows accept it, while preserving trusted loopback-only no-auth use.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

Yes. Fail-closed validation for explicit no-auth plus Tailscale Serve is the narrow maintainable direction, with matching config/install preflight and tests so operators do not discover the rejection only at daemon startup.
