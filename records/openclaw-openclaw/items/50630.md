## Summary

Keep open. Current main still accepts the reported `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` startup shape, the auth path still returns success for mode none before Tailscale identity-header auth can run, and open PR #50631 remains the unmerged implementation candidate for this same issue.

## What This Changes

The issue requests hardening Gateway startup so explicit no-auth mode cannot be combined with Tailscale Serve remote exposure to the Tailnet.

## Best Possible Solution

Harden Gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing HTTP contract, then update the runtime-config tests and public Gateway security/Tailscale docs for the final contract.

## Reproduction Assessment

Yes. The deterministic source-level reproduction is the config shape `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape in runtime config and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` rejecting Serve plus mode none is the narrowest maintainable fix for the reported boundary gap, with tests proving Serve rejects mode none while loopback-only no-auth remains allowed.
