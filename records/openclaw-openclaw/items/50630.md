## Summary

Keep open. Current main and the latest shipped tag still accept the reported Tailscale Serve plus explicit no-auth configuration, and the linked fixing PR remains open, conflicting, and unmerged.

## What This Changes

The issue reports that `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` starts and can grant credentialless Gateway access to Tailnet peers.

## Best Possible Solution

Land or replace the linked fix with shared validation that rejects explicit no-auth plus Tailscale Serve in runtime and install/onboarding preflight, while preserving trusted loopback-only no-auth and updating focused tests/docs.

## Reproduction Assessment

Yes, at source level. Current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, has a test asserting that shape succeeds, and then authorizes mode none before Tailscale identity or shared-secret checks run.

## Solution Assessment

Yes, the fail-closed direction is the narrow maintainable fix. The linked PR moves in that direction, but maintainers should ensure shared preflight coverage, upgrade messaging, and real Tailscale proof before landing.
