## Summary

Current main still accepts the reported Serve plus no-auth startup configuration, still authorizes mode none before verified Tailscale identity checks, and the open linked fix PR remains unmerged, so this issue should stay open.

## What This Changes

Reject or otherwise harden the configuration where `gateway.tailscale.mode="serve"` is combined with `gateway.auth.mode="none"`.

## Best Possible Solution

Harden Gateway startup so explicit mode none cannot be combined with Tailscale Serve remote exposure, preserve trusted loopback-only no-auth use, and update runtime-config tests plus user-facing Gateway/Tailscale security docs for the final contract.

## Reproduction Assessment

Yes. The source-level path is `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current main accepts that shape in runtime config and has a test asserting successful loopback binding.

## Solution Assessment

Yes. A startup-time guard rejecting Serve plus no-auth is the narrowest maintainable fix for this boundary gap; the broader optional secondary-auth design remains separately tracked at https://github.com/openclaw/openclaw/issues/57110.
