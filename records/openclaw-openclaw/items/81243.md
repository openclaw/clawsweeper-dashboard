## Summary

Keep open: this is a useful Discord feature PR, but the draft branch still has blocking public-tool issues for URL-only fetches and needs stronger after-fix proof for the public path; https://github.com/openclaw/openclaw/pull/81272 is an overlapping open implementation that maintainers should coordinate with.

## What This Changes

The PR adds a Discord message `fetch` action mapped to the existing `fetchMessage` runtime, updates action discovery/contracts/docs/tests, and adds a manual repro script.

## Best Possible Solution

Land one canonical implementation that exposes Discord `fetch` through the public message tool, treats URL-only fetches as targetless or URL-aware, advertises `url`/`messageLink`, preserves the messages gate and gateway execution, and includes focused public-runner coverage plus redacted live proof.

## Reproduction Assessment

Yes, for the blocking PR bug: source inspection shows `runMessageAction` calls `normalizeMessageActionInput` before Discord dispatch, and `fetch: "to"` makes a URL-only public call fail before the handler can parse the URL. I did not run live Discord proof in this read-only review.

## Solution Assessment

No. Reusing the existing Discord `fetchMessage` runtime is the right direction, but this branch should make URL fetch targetless or URL-aware, expose URL fields in the public schema, and coordinate with the overlapping implementation.
