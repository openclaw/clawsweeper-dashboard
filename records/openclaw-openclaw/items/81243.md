## Summary

Keep this PR open: the Discord fetch direction is useful and the real Discord proof is sufficient, but the latest target-alias fix is not exposed through the Discord channel plugin wrapper, so URL-only fetch can still fail before reaching the Discord handler.

## What This Changes

The PR adds a Discord `message` action `fetch` for retrieving one message by `channelId` plus `messageId` or by Discord message URL, with schema, shared action metadata, tests, skill docs, and a repro script.

## Best Possible Solution

Land this PR or a narrow equivalent after forwarding the Discord URL target alias through the channel plugin wrapper and proving both URL-only and direct-ID message-tool normalization paths.

## Reproduction Assessment

Yes for the feature gap: current main omits `fetch` from Discord discovery and the shared action union, while the internal Discord runtime already has `fetchMessage`. I did not rerun live Discord locally in this read-only pass.

## Solution Assessment

No, not quite: the implementation direction is narrow and reuses the existing Discord runtime, but the URL-only target alias must be exposed through the Discord channel plugin wrapper before the public message-tool path is complete.
