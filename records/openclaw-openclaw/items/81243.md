## Summary

Keep open: the feature is useful and current main does not expose Discord single-message fetch as a public `message` action, but this draft PR still breaks the promised URL-only public tool path and needs stronger public-path proof before merge.

## What This Changes

The PR exposes a Discord `message` action `fetch`, maps direct IDs or a URL to the existing `fetchMessage` runtime, advertises it under the messages gate, adds tests, a Discord skill note, and a manual repro script.

## Best Possible Solution

Revise this PR, or one canonical replacement, to expose Discord fetch through the public message tool with targetless URL handling, `url`/`messageLink` schema coverage, public-runner direct and URL tests, and redacted live proof.

## Reproduction Assessment

Yes. Source inspection shows `runMessageAction` normalizes before Discord dispatch, and the PR's `fetch: "to"` setting would make URL-only fetch without a current target fail before the Discord handler can parse the URL.

## Solution Assessment

No. Reusing the existing Discord fetch runtime is the right direction, but the public contract should make URL fetch targetless, expose URL fields in the tool schema, and prove the public path before merge.
