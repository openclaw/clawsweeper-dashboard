## Summary

Keep open: the feature is valid, but the current patch still makes `fetch` require a target, so the advertised URL-only Discord message fetch path can be rejected before the Discord handler parses the link.

## What This Changes

The PR adds a Discord message `fetch` action for channel/message IDs or Discord URLs, wires discovery/gateway/shared action contracts, relaxes `guildId`, updates tests, and documents the skill.

## Best Possible Solution

Expose Discord fetch through the existing runtime with both direct ID and URL forms passing public message-tool normalization, then land one canonical PR with redacted real proof for both paths.

## Reproduction Assessment

Not applicable as a current-main bug because this is a feature PR. The PR regression is source-reproducible: `fetch: "to"` makes a targetless URL-only call fail shared normalization before Discord URL parsing.

## Solution Assessment

No. Reusing the existing Discord runtime is the right direction, but the target-mode choice makes the documented URL-only path unusable; the safer shape is targetless fetch or explicit `url`/`messageLink` target satisfaction, plus schema coverage.
