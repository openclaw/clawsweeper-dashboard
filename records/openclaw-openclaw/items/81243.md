## Summary

Keep open: the latest head still has a source-reproducible blocker for the advertised URL-only Discord fetch path, and the supplied real proof only exercises a direct handler-level channel/message ID call while an overlapping implementation PR remains open.

## What This Changes

The PR adds a public Discord message `fetch` action wired to the existing `fetchMessage` runtime, updates shared action contracts/discovery/docs/tests, and adds a manual repro script.

## Best Possible Solution

Land one canonical PR that exposes Discord `fetch` through the public message tool, treats URL-only fetches as targetless, advertises `url`/`messageLink`, preserves messages gating and gateway execution, and includes focused public-runner coverage plus redacted live proof.

## Reproduction Assessment

Yes. Source inspection shows the public runner normalizes message actions before Discord dispatch, and the PR's `fetch: "to"` setting makes URL-only fetches without a current target fail before the Discord handler can parse the URL.

## Solution Assessment

No. Reusing the existing Discord fetch runtime is the right direction, but the branch should make URL fetch targetless or otherwise pass URL fields through normalization, expose the URL fields in the public schema, and prove the public path.
