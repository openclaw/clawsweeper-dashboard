## Summary

Keep open: this PR fixes the prior shared-action-contract omission, but the branch still makes URL-only Discord fetches target-required before the Discord URL parser can run, and the pasted CLI proof does not convincingly match the registered CLI/source path. There is also an overlapping open implementation at https://github.com/openclaw/openclaw/pull/81272, so maintainers should choose one canonical branch after the target-mode/proof issue is addressed.

## What This Changes

The PR exposes a Discord message `fetch` action that maps channel/message IDs or a Discord message URL to the existing `fetchMessage` runtime and updates action contracts, tests, and Discord skill docs.

## Best Possible Solution

Expose `fetch` as a targetless Discord message action that accepts `channelId` plus `messageId` or `url`/`messageLink`, advertises the URL fields in the public schema, preserves messages gating and gateway execution, and lands one canonical PR with credible redacted live proof.

## Reproduction Assessment

Yes. Source inspection shows current main has the internal Discord `fetchMessage` backend but no public `fetch` action, and this PR's URL-only path is source-reproducibly blocked by target-required normalization before Discord URL parsing.

## Solution Assessment

No. Reusing the existing Discord fetch runtime is the right direction, but this branch should make `fetch` targetless for URL-only calls and expose URL fields in the public tool schema before it is the best maintainable solution.
