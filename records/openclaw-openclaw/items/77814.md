## Summary

Keep open: current docs and code define Pi session JSONL as the durable transcript, but the reviewed Telegram delivery path can surface final assistant text through live reply payloads while relying on a separate Pi append path for persistence. The reporter provided concrete multi-session `toolResult -> user` gaps, and current `main` does not show an implemented guard or repair for that invariant.

## What This Changes

The issue reports intermittent Telegram direct-message agent turns where final assistant replies after tool calls are delivered to Telegram but missing from the active session JSONL.

## Best Possible Solution

Ensure every Telegram-delivered final assistant answer from a normal agent turn corresponds to a persisted model-visible assistant turn, with regression coverage for the tool-call sequence and a narrow fallback only if canonical Pi persistence cannot be confirmed.

## Reproduction Assessment

No, not as a live current-main reproduction in this read-only review. The reporter's JSONL gap examples plus the current source path give a credible source-level reproduction target: Telegram agent turn with tool calls, final delivery, then transcript inspection before the next user turn.

## Solution Assessment

Yes: keeping this open for a narrow persistence-invariant fix is the best path. The safer repair is to preserve Pi SessionManager as the canonical writer and add a deterministic guard/test, not to blindly mirror every Telegram final payload into JSONL.
