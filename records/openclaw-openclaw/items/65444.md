## Summary

Keep open. Current main still has targeted startup/import optimizations from related work, but it does not implement the requested lazy channel connection config, register/connect channel lifecycle split, or readiness/status semantics for registered-but-not-connected channels.

## What This Changes

The issue requests a default-on `channels.lazyConnect` option that defers channel transport startup until first use, with an opt-out path preserving eager startup.

## Best Possible Solution

Design an additive, capability-gated channel lifecycle that keeps registration cheap, lets channels declare whether transport startup can be deferred, exposes registered versus connected/running status, and preserves eager startup where required.

## Reproduction Assessment

Yes for the missing requested behavior: source inspection shows no consumed `channels.lazyConnect` option, startup still awaits channel start, and every registered channel plugin is still started through `startAccount`. I did not live-reproduce the reported Windows 50-second startup timing.

## Solution Assessment

Unclear for the exact proposal. Lazy channel connection is directionally aligned with the startup problem, but the maintainable solution should be additive, capability-gated, and backward-compatible rather than a blanket default lifecycle change for every channel.
