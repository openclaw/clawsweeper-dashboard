## Summary

Keep open: current main has startup handoff and lighter runtime-loading work, but it still lacks the requested `channels.lazyConnect` config, first-use channel connection lifecycle, and registered-versus-connected status/readiness contract.

## What This Changes

The issue requests a default-on `channels.lazyConnect` option that registers channel routing at startup but defers transport connection until first use, with an opt-out eager mode.

## Best Possible Solution

Define an additive, capability-gated channel lifecycle that can defer safe transports, preserves eager startup for channels that require it, exposes pending/registered/connected state, and updates config, SDK docs, and regression coverage together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested config and lifecycle are absent on current main and in the latest release tag, but I did not reproduce the reported Windows timing.

## Solution Assessment

Unclear for the exact default-on proposal. Lazy connection is directionally plausible, but the maintainable fix should be additive, capability-gated, and coordinated with the broader core-ready startup contract.
