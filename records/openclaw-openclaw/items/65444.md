## Summary

Keep open: current main still lacks the requested lazy channel connection config, channel register/connect lifecycle split, and registered-versus-connected readiness/status semantics; the overlapping core-ready issue does not fully supersede this channel-specific feature request.

## What This Changes

The issue requests a gateway channel lazy-connect lifecycle with an opt-out `channels.lazyConnect` config, deferred channel transport startup, and status/readiness semantics for registered versus connected channels.

## Best Possible Solution

Define an additive, capability-gated channel lifecycle that can defer transport startup where safe, exposes registered versus connected/running state, updates config/docs/tests, and preserves eager startup for channels that require it.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested behavior is absent, but I did not live-reproduce the reported Windows startup timing.

## Solution Assessment

Unclear for the exact proposal. Lazy connection is directionally aligned with the startup problem, but the maintainable solution should be additive, capability-gated, and backward-compatible rather than a blanket default lifecycle change for every channel.
