## Summary

Keep open. Current main has several startup/import optimizations, but it still does not implement lazy channel connection or a consumed opt-out config: channel sidecars remain part of startup readiness, the channel manager eagerly starts every registered channel account, and the public channel adapter still exposes the existing start/stop lifecycle.

## What This Changes

Request to add a default-on `channels.lazyConnect` lifecycle so channel plugins register cheaply at startup, connect on first use, and can opt out to eager startup with updated status/readiness states.

## Best Possible Solution

Add an explicit, documented lazy channel lifecycle that separates cheap registration/status from transport connection, lets each channel declare whether deferral is safe, preserves eager startup where needed, connects on first safe inbound/outbound use, and updates readiness/status/tests/docs around registered versus connected/running states.

## Reproduction Assessment

Yes for the missing feature on current main: source inspection shows no consumed `channels.lazyConnect`, startup still awaits `startChannels()`, and `startChannels()` still invokes `startAccount()` for every registered channel plugin. The exact Windows 50s timing would still require the reporter's environment.

## Solution Assessment

No. Current main's startup/import reductions are useful but are not the requested lazy-connect solution; the maintainable path needs an additive lifecycle/config/capability design rather than treating runtime import deferral as equivalent.
