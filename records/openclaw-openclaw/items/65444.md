## Summary

Keep open: current main still lacks the requested channel lazy-connect config, first-use transport connection, register/connect lifecycle split, and registered-versus-connected status semantics. Related work reduces adjacent startup cost or tracks broader core-ready behavior, but it does not replace this channel-specific product/API decision.

## What This Changes

The issue requests a default-on `channels.lazyConnect` mode that registers channel plugins at startup but defers transport connection until first use, with an opt-out config and registered-versus-connected status semantics.

## Best Possible Solution

Define an additive, capability-gated channel lifecycle that can defer transport startup where safe, exposes registered versus connected/running state, updates config/docs/tests, and preserves eager startup for channels that require it.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested config and lifecycle are absent, but I did not live-reproduce the reported Windows startup timing.

## Solution Assessment

Unclear for the exact default-on proposal. Lazy connection is directionally aligned with the startup problem, but the maintainable solution should be additive, capability-gated, and backward-compatible rather than a blanket lifecycle change for every channel.
