## Summary

Keep open: current main still starts channel plugins eagerly and lacks the requested core config, channel lifecycle split, and registered-versus-connected semantics; related startup work overlaps the readiness problem but does not implement first-use channel connection.

## What This Changes

Add default-on `channels.lazyConnect` behavior to defer channel transport connection until first use, with an opt-out eager mode and registered/connected status semantics.

## Best Possible Solution

Define an additive, capability-gated channel lifecycle that can defer transport startup where safe, expose registered versus connected/running state, and preserve eager startup for channels that require it.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested config and lifecycle are absent, but I did not live-reproduce the Windows startup timing.

## Solution Assessment

Unclear for the exact default-on proposal. Lazy connection is directionally aligned with the startup problem, but the maintainable path should be additive, capability-gated, and backward-compatible rather than a blanket lifecycle change across every channel.
