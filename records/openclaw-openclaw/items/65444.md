## Summary

Keep open. Current main still has the startup/import optimizations noted in the related work, but it does not implement a `channels.lazyConnect` option, a register/connect channel lifecycle split, or readiness semantics that allow normal gateway readiness before channel sidecars finish.

## What This Changes

The issue requests a default-on `channels.lazyConnect` option and split channel registration/connection lifecycle so gateway startup is not blocked by eager channel account startup.

## Best Possible Solution

Design an additive channel lifecycle that keeps registration cheap, lets each channel declare whether transport startup can be deferred, documents `registered` versus `running/connected` status, preserves eager startup where required, and covers readiness plus first inbound/outbound use in tests.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing behavior: there is no consumed `channels.lazyConnect` option, normal startup still awaits `startChannels()`, and `startChannels()` still starts every registered plugin account.

## Solution Assessment

Unclear for the exact proposal. Lazy channel connection is directionally aligned with the reported startup class, but the maintainable fix should be additive, capability-gated, and backward-compatible instead of applying a blanket default lifecycle change to every channel.
