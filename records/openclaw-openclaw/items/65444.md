## Summary

Keep open: current main still lacks the requested lazy channel connection config, channel register/connect lifecycle split, and registered-versus-connected readiness/status semantics; the related core-ready issue overlaps but does not fully own this channel-specific feature request.

## What This Changes

Request to add a default-on `channels.lazyConnect` option that defers channel transport connection until first use, with eager startup as an opt-out.

## Best Possible Solution

Define an additive, capability-gated channel lifecycle that can defer transport startup where safe, exposes registered versus connected/running state, updates config/docs/tests, and preserves eager startup for channels that require it.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms the requested config and lifecycle are absent, but I did not live-reproduce the reported Windows startup timing.

## Solution Assessment

Unclear for the exact default-on proposal. Lazy connection is directionally aligned with the startup problem, but the maintainable solution should be additive, capability-gated, and backward-compatible rather than a blanket lifecycle change for every channel.
