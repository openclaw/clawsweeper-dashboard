## Summary

Keep open. Current main has several startup/import optimizations, but it still does not implement the requested lazy channel connect lifecycle or `channels.lazyConnect` opt-out; channel sidecars still start registered channel accounts during gateway startup and readiness remains tied to sidecar completion.

## What This Changes

Request a default-on `channels.lazyConnect` gateway option that registers channel plugins at startup but defers transport connection until first channel use, with status distinguishing registered from connected.

## Best Possible Solution

Design an additive channel lifecycle that keeps plugin registration cheap, lets each channel declare whether account transport startup can be deferred, adds documented config/status semantics for registered versus running/connected, preserves eager startup where required, and covers gateway readiness plus first-use inbound/outbound behavior in tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no consumed `channels.lazyConnect` option, `startGatewaySidecars()` still awaits channel startup, and `startChannels()` still invokes every registered plugin account's `startAccount()` path.

## Solution Assessment

Unclear for the exact proposal. Lazy connection is directionally the right fix for the reported startup class, but the safest implementation should be capability-gated and backward-compatible rather than a blanket lifecycle split applied to every channel by default.
