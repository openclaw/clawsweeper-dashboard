## Summary

Keep open. Current main still classifies OpenAI-compatible `delta.content` as visible text before considering reasoning fields, and still classifies bare `reasoning_content`/`reasoning`/`reasoning_text` as `thinking` rather than provider-scoped visible text. Recent related work improves typed OpenRouter `reasoning_details`, Z.AI request-side thinking controls, and reasoning-only retry behavior, but it does not resolve the two mirrored Bailian/Z.AI classification cases reported here.

## What This Changes

The issue asks OpenClaw to make streamed `content` versus `reasoning_content` classification stable, including a safe fallback when provider output contains a visible answer only in reasoning fields.

## Best Possible Solution

Add provider-owned, explicitly documented compatibility metadata or stream normalization for known providers/models whose reasoning field can contain the user-visible final answer, with regression fixtures for the two mirrored samples and guards that keep true reasoning private by default.

## Reproduction Assessment

Yes for the parser-level behavior: a chunk with only `delta.reasoning_content` will deterministically produce a `thinking` block, and a chunk with non-empty `delta.content` will deterministically produce a text block. The exact live-provider nondeterminism still needs redacted provider fixtures or live confirmation.

## Solution Assessment

No, the issue's suggested global fallback is not the safest final fix because it can leak real reasoning. A provider-scoped compatibility path with explicit model metadata and tests is the maintainable solution.
