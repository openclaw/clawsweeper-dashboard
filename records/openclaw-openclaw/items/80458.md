## Summary

Keep open: current origin/main still has the reported one-entry raw-stream path, so phase-tagged commentary can bypass the already available final-answer filter before channel payload delivery.

## What This Changes

The issue reports that phase-tagged Codex assistant commentary can leak into channel-delivery payloads because raw streamed assistant text is preferred over the phase-filtered final answer.

## Best Possible Solution

Add a focused payload regression and make buildEmbeddedRunPayloads prefer the phase-filtered final answer whenever explicit phase metadata is present, while preserving raw media directives and unphased-provider behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: call buildEmbeddedRunPayloads with one non-empty assistantTexts value containing commentary plus final text and a lastAssistant whose textSignature blocks include final_answer; current main returns the raw streamed entry because the canonical fallback requires more than one streamed text.

## Solution Assessment

Yes. The narrow maintainable fix is to key off explicit phase metadata rather than streamed-text array length, using the existing final-answer extraction while preserving media-directive and legacy unphased paths.
