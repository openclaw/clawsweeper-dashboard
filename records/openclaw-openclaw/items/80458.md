## Summary

Keep open: current main has a partial guard for multi-entry streamed assistant text, but source inspection still shows a single raw streamed `assistantTexts` entry can bypass the phase-filtered `fallbackAnswerText` and deliver commentary alongside the final answer.

## What This Changes

Filter phase-tagged assistant replies in `buildEmbeddedRunPayloads` so channel-delivered payloads use `final_answer` text instead of raw streamed commentary.

## Best Possible Solution

Add a focused regression for a single raw streamed assistant text plus a phase-tagged `lastAssistant`, then make the payload builder prefer the phase-filtered final answer whenever explicit phase metadata exists while preserving existing unphased and media-directive behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence helper-level reproduction: call `buildEmbeddedRunPayloads` with one non-empty `assistantTexts` entry containing commentary plus final text and a `lastAssistant` whose text blocks include `phase: "final_answer"`. Current main returns the raw streamed entry because canonical fallback only activates for more than one streamed text.

## Solution Assessment

Yes, the suggested phase-metadata direction is the narrow maintainable fix, but it should not depend on the streamed-text array length. The safer rule is to use the already phase-filtered final answer when explicit phase metadata is present, with targeted exceptions for existing raw/media behavior.
