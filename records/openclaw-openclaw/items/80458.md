## Summary

Keep open: current main and the latest release still allow a single raw streamed assistant text to win over the phase-filtered final answer, and an open linked PR is already the implementation path for this issue.

## What This Changes

The issue reports that phase-tagged Codex/provider output can leak commentary-phase text into channel-delivery payloads because `buildEmbeddedRunPayloads` prefers raw streamed `assistantTexts`.

## Best Possible Solution

Land or replace the linked fix PR with a media-aware phase-filtered payload selection rule plus a focused regression for the single streamed-text case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: pass one raw streamed assistant text containing commentary plus final text and a phase-tagged `lastAssistant`; current main returns the raw streamed entry because canonical fallback only activates for more than one streamed text.

## Solution Assessment

Yes, the phase-filtering direction is the narrow maintainable fix, but it should use the existing media-aware final-answer source so attachments and reply directives are preserved.
