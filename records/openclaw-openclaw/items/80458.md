## Summary

Current main now fixes the reported channel-final payload leak: the embedded runner passes the current assistant into payload assembly and prefers the canonical structured final answer before falling back to raw streamed assistant text. The fix landed through the broader merged payload hardening PR and is present in the v2026.5.16-beta.5 prerelease, while the latest stable v2026.5.12 still had the old raw-stream precedence.

## What This Changes

The issue reports that phase-tagged provider streams can leak `commentary` text into channel delivery because `buildEmbeddedRunPayloads` preferred raw streamed `assistantTexts` over the filtered `final_answer`.

## Best Possible Solution

Keep the merged generic final-payload assembly fix and close the older overlapping contributor PR as superseded by https://github.com/openclaw/openclaw/pull/82850.

## Reproduction Assessment

No current-main failure remains: source inspection shows the old single-streamed-text precedence has been replaced by canonical final-answer selection. The old behavior is source-reproducible in v2026.5.12, where `nonEmptyAssistantTexts.length > 1` was still required before using the canonical fallback.

## Solution Assessment

Yes. The landed https://github.com/openclaw/openclaw/pull/82850 path is safer than the earlier phase-only branch because it prefers the structured final answer while preserving existing media-directive and stale-transcript boundaries.
