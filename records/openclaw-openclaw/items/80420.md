## Summary

Close as implemented: current main and release v2026.5.12 now preserve explicit cron `timeoutSeconds` through the embedded LLM idle watchdog, matching the root cause identified in the discussion. The related Codex cron compatibility fix also shipped, but the idle-watchdog equality bug here is covered by the timeout fix.

## What This Changes

The issue reports isolated cron jobs timing out with the model idle watchdog in OpenClaw 2026.5.7 even when a longer timeout is configured.

## Best Possible Solution

Keep the shipped v2026.5.12 implementation and ask for a new issue with fresh logs if cron timeouts persist after upgrading.

## Reproduction Assessment

Yes, source-level for the pre-fix bug. The old path collapsed explicit cron `timeoutSeconds` and agent defaults into the same number, then dropped `runTimeoutMs`, causing the cron idle watchdog to fall back to the implicit 120 second cap.

## Solution Assessment

Yes. Carrying the explicit per-run timeout signal from cron into the embedded runner is the narrow maintainable fix and avoids changing global timeout defaults or provider behavior.
