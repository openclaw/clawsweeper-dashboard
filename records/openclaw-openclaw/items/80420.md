## Summary

Keep open: current main still drops the explicit cron timeout when it numerically equals the agent default, and the active fix is still open at https://github.com/openclaw/openclaw/pull/79426.

## What This Changes

The issue reports cron-triggered isolated agent sessions timing out on OpenClaw 2026.5.7 even when an explicit long timeout is configured.

## Best Possible Solution

Land a narrow fix that carries the explicit cron timeout signal through the embedded runner, with regression tests and redacted real cron proof; keep broader idle-timeout configuration work in https://github.com/openclaw/openclaw/issues/78361.

## Reproduction Assessment

Yes, source-level. When cron `timeoutSeconds` equals `agents.defaults.timeoutSeconds`, current main collapses both to the same `timeoutMs`, drops `runTimeoutMs`, and lets the idle watchdog fall back to the implicit 120s cap.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix I found: carry the explicit cron timeout fact instead of inferring it from numeric inequality. The issue should stay open until that fix or an equivalent one merges and is proven.
