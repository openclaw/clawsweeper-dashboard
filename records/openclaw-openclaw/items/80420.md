## Summary

Keep open. Current main still has the cron equal-timeout signal-loss path, v2026.5.7 shipped the same heuristic, and the exact fix path is already under review in https://github.com/openclaw/openclaw/pull/79426 but has not merged yet.

## What This Changes

Cron-triggered isolated agent jobs should honor explicit per-run `timeoutSeconds` even when it equals `agents.defaults.timeoutSeconds`, instead of timing out under the LLM idle watchdog.

## Best Possible Solution

Land or revise the existing narrow PR so cron preserves explicit timeout provenance into the embedded runner, with regression coverage and redacted real cron-run proof before release.

## Reproduction Assessment

Yes, source-level. Set `agents.defaults.timeoutSeconds` and the cron payload `timeoutSeconds` to the same positive value; current main resolves both to the same `timeoutMs`, drops `runTimeoutMs`, and falls back to the capped implicit idle watchdog.

## Solution Assessment

Yes. The narrow maintainable fix is to carry an explicit cron timeout signal into the embedded runner; broader timeout policy or config-surface changes are not needed for this reported edge case.
