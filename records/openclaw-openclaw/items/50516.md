## Summary

Keep open. Current main still has the length-mismatch early return on the Nextcloud Talk webhook signature path, so the hardening is not implemented; the PR is useful but needs a small security fixup and rebase before it is merge-ready.

## What This Changes

The PR changes Nextcloud Talk signature verification to use Node `crypto.timingSafeEqual` with buffer comparisons and adds a new focused signature test file.

## Best Possible Solution

Keep the hardening work, but revise the verifier to use the shared `safeEqualSecret` helper or a LINE-style padded/distinct-buffer comparison, retain the focused signature tests, rebase the branch, and run the targeted Nextcloud Talk tests plus the normal changed gate before landing.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main behavior: a wrong-length signature returns before the XOR loop in `extensions/nextcloud-talk/src/signature.ts`, and the live webhook path calls that verifier from `extensions/nextcloud-talk/src/monitor.ts`.

## Solution Assessment

No. The PR direction is right, but the self-comparison dummy call is not the strongest local pattern; using `safeEqualSecret` or a distinct padded/dummy buffer would be the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Compare against a distinct dummy buffer:** `extensions/nextcloud-talk/src/signature.ts:32`
  - body: The mismatch branch calls `timingSafeEqual(expBuf, expBuf)`, so it compares the same Buffer object instead of exercising the two-buffer access pattern used by real signature checks. Use `safeEqualSecret`, padded buffers, or a distinct `Buffer.alloc(expBuf.length)` dummy so the length-mismatch path matches the intended timing-safe behavior.
