## Summary

Keep open for maintainer review: the patch is a plausible and likely correct fix for the cron upgrade data-loss path, with sufficient real behavior proof, but it intentionally changes upgrade/startup behavior by auto-writing SQLite cron rows, archiving legacy files, and allowing restored overdue jobs to run on startup.

## What This Changes

The PR adds legacy cron JSON/state/run-log auto-migration to doctor config preflight, detects legacy cron stores for read-only startup paths, and routes the gateway run fast path through shared CLI bootstrap.

## Best Possible Solution

Land this only after maintainer acceptance that startup preflight is the right migration boundary for preserving legacy cron jobs and that automatic catch-up/archiving are intended upgrade behavior.

## Reproduction Assessment

Yes. Current main reads cron runtime state from SQLite only and does not run cron migration in config preflight or the gateway fast path, while the linked user report describes the matching 2026.5.28 to 2026.6.1 loss path.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff. Moving the existing doctor-owned cron migration into config preflight and making gateway run use the same bootstrap is narrower than reintroducing runtime JSON fallback readers.
