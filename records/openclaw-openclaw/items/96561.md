## Summary

Keep open. Current main and the latest release still show the reported QMD CLI/tool wrapper overhead paths, and the protected `maintainer` label plus product choices around caching, grouping, concurrency, or warm-manager routing block cleanup closure.

## What This Changes

The issue asks to reduce repeated QMD memory-search manager, probe, and search overhead in the CLI and standard `memory_search` tool paths while preserving memory/session source semantics.

## Best Possible Solution

Keep this as the maintainer-owned QMD memory-search performance tracker and choose a bounded instrumentation plus optimization path that reduces repeated subprocess work without weakening collection rebind safety or memory/session diversity.

## Reproduction Assessment

Yes, at source level. Current main routes CLI memory search through `purpose: "cli"`, bypasses cached QMD manager reuse, validates collections during QMD initialization, and starts tool `debug.searchMs` after manager acquisition; I did not run live QMD timing in this read-only pass.

## Solution Assessment

Unclear. Reducing wrapper overhead is the right target, but maintainers need to choose the safe boundary among instrumentation, stable cache keys, one-call search, bounded concurrency, or warm-manager routing without losing rebind and source-diversity guarantees.
