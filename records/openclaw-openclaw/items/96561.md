## Summary

Keep open. Current main and v2026.6.10 still show the reported QMD CLI/tool wrapper overhead paths, no exact linked PR owns this issue, and the protected `maintainer` label requires explicit maintainer handling.

## What This Changes

The issue requests reducing QMD-backed memory search overhead in `openclaw memory search` and the `memory_search` tool by adding diagnostics and avoiding repeated setup/probe/search subprocess work while preserving memory/session behavior.

## Best Possible Solution

Keep this as the maintainer-owned QMD memory-search performance tracker and land a bounded instrumentation-first optimization that reduces repeated QMD wrapper work without weakening collection rebind safety or memory/session source diversity.

## Reproduction Assessment

Yes, at source level. Current main routes CLI memory search through transient QMD managers, validates collections during QMD initialization, and starts tool `debug.searchMs` after manager acquisition; I did not run a live QMD timing repro.

## Solution Assessment

Unclear. Reducing wrapper overhead is the right target, but maintainers need to choose the safe boundary among diagnostics, stable cache invalidation, grouped/concurrent QMD search, and warm-manager routing.
