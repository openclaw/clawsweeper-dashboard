## Summary

Keep open. The protected `maintainer` label prevents cleanup closure, and current `main` plus v2026.6.10 still match the source-level report: CLI search creates a transient QMD manager, collection/capability setup is per manager, and mixed memory/session QMD search remains split by source.

## What This Changes

The issue requests reducing repeated QMD setup/probe and sequential subprocess overhead in `openclaw memory search`.

## Best Possible Solution

Keep this as the maintainer-owned QMD CLI optimization tracker and choose a fix that safely reuses stable collection/capability facts or reduces subprocess count while preserving collection freshness and memory/session diversification.

## Reproduction Assessment

Yes at source level, but not with fresh live timing proof. Current main routes CLI memory search through a transient QMD manager that validates collections and probes QMD capabilities per manager instance, then searches mixed memory/session groups sequentially.

## Solution Assessment

Unclear. Caching or one-call search may be the right optimization, but maintainers need to choose the safe boundary because current docs intentionally split memory and sessions for source diversification.
