## What This Changes

The issue asks OpenClaw to reduce repeated QMD memory-search wrapper costs in `openclaw memory search` and the `memory_search` tool path while preserving source diversity and repair behavior.

## Best Possible Solution

Keep this issue as the canonical QMD wrapper-overhead tracker, review and land or close the partial cache/diagnostics PR, then split any remaining grouped-search and gateway-overhead work into narrower follow-ups.

## Reproduction Assessment

Yes, at source level. Current main routes CLI QMD search through transient managers, validates QMD collections during initialization, probes multi-collection support per manager instance, and starts tool `debug.searchMs` after manager acquisition; I did not run live QMD timing.

Unclear as a complete solution. The cache/diagnostics PR is a good bounded first step, but maintainers still need to choose the final cache invalidation, grouped-search, concurrency, and gateway timing strategy.
