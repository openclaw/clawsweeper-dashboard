## What This Changes

The issue requests reducing QMD-backed `openclaw memory search` and `memory_search` overhead by avoiding repeated manager setup/probes and sequential memory/session QMD subprocess searches.

## Best Possible Solution

Keep this as the maintainer-owned QMD memory-search performance tracker and choose a bounded fix that instruments manager acquisition, safely reuses stable QMD facts, or reduces subprocess count without weakening collection freshness or source diversity.

## Reproduction Assessment

Yes, at source level. Current main routes CLI memory search through a transient QMD manager that validates collections and probes QMD capabilities per manager instance, then searches mixed memory/session groups sequentially.

## Solution Assessment

Unclear. The issue outlines plausible fixes, but maintainers need to choose the safe boundary among cache invalidation, one-call search, bounded concurrency, or a warm manager path.
