- `impact:session-state`: The affected memory search path controls recall over durable memory and session transcript sources, so latency can degrade agent/session context retrieval.

## Summary

Keep open. The protected maintainer label prevents cleanup closure, and current source still matches the central CLI overhead report: `memory search` requests a one-shot CLI QMD manager, initialization validates collections, and mixed memory/session QMD search still runs source groups separately even though same-source fanout was already optimized.

## What This Changes

Reduce `openclaw memory search` latency by avoiding repeated transient QMD setup/probes and unnecessary sequential QMD subprocess work.

## Best Possible Solution

Keep this as the maintainer-owned QMD CLI optimization tracker and choose a fix that safely reuses stable collection/capability facts or reduces subprocess count without weakening collection freshness or memory/session diversification.

## Reproduction Assessment

Yes at source level, but not with live timing proof. Current main routes CLI memory search through a transient QMD manager that validates collections and probes capabilities per manager instance, and tests assert the CLI manager is not reused.

## Solution Assessment

Unclear. Caching or one-call search may be the right optimization, but maintainers need to choose the safe boundary because current docs intentionally split memory and sessions for source diversification.
