## Summary

Keep open: current main and the latest release still pass the raw Feishu reply target into streaming-card startup, and the three linked closing PRs are open but unmerged.

## What This Changes

The issue requests Feishu DM streaming-card replies to omit `replyToMessageId` so ordinary direct messages render as top-level Feishu messages without a reply label.

## Best Possible Solution

Review and land one linked Feishu dispatcher PR that passes `sendReplyToMessageId` into streaming startup, preserves topic/thread reply behavior, adds focused regression coverage, and closes this issue after merge.

## Reproduction Assessment

Yes. Source inspection reproduces the path: ordinary Feishu DMs set `skipReplyToInMessages`, but the streaming card start call still receives raw `replyToMessageId`, which makes the streaming helper choose Feishu reply mode.

## Solution Assessment

Yes, the best fix shape is narrow: reuse the already-computed `sendReplyToMessageId` for streaming-card startup rather than adding a new config workaround or changing non-streaming send behavior.
