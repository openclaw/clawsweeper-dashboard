## Summary

Keep open: the diagnostic direction is right and the latest proof shows a registration-path warning, but the PR is not mergeable because it exposes and documents a Plugin SDK `availability` contract that the plugin resolver only audits, not enforces.

## What This Changes

The PR adds an `onHiddenDiagnostic` planner callback, audits plugin tool descriptor availability during `resolvePluginTools`, preserves and documents plugin tool `availability`, and adds focused regression tests.

## Best Possible Solution

Keep the evaluator pure, surface unsupported diagnostics once at registration, and make plugin `availability` either a fully enforced documented SDK contract or an internal-only audit detail before merge.

## Reproduction Assessment

Yes. The original issue is source-reproducible on current main because `buildToolPlan` hides empty groups with `unsupported-signal` and no callback, and the PR-introduced gap is source-reproducible because `resolvePluginTools` returns tools before the audit and ignores the planner result.

## Solution Assessment

No. The registration-boundary warning is the right layer, but exposing a public `availability` field without enforcing it makes this too broad and incomplete as a mergeable fix.
