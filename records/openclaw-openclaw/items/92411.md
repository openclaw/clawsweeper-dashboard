## Summary

The planner-level callback works as demonstrated, but this PR does not fix the reported registration-time silence because current main has no production caller of `buildToolPlan`; the intended production owner must be established before adding this observer API.

## What This Changes

The PR adds an optional callback that emits each availability diagnostic associated with a tool hidden by `buildToolPlan`.

## Best Possible Solution

Identify the production tool-registration or runtime-planning owner, have that owner inspect the canonical hidden-plan result and report only malformed descriptor diagnostics, and prove the warning through the real registration path.

## Reproduction Assessment

Yes at the isolated planner level: empty groups produce `unsupported-signal` diagnostics that current main exposes only through `ToolPlan.hidden`. There is no current-main gateway-startup reproduction because no production path calls `buildToolPlan`.

## Solution Assessment

No; the callback is not connected to the registration behavior named by the report, and adding an exported observer before establishing its production owner creates an unnecessary parallel diagnostics path.
