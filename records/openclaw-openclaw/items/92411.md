## Summary

This PR is still useful and should stay open, but it needs one more code fix before merge: the new descriptor-level `plugin-enabled` availability check derives enabled plugin ids from `context.config` while the surrounding availability evaluation uses the active runtime config. The branch is also currently GitHub-conflicting, so the merge result needs refresh after the fix.

## What This Changes

The PR adds a hidden-tool diagnostic callback to the tool planner, evaluates plugin-authored tool availability during plugin tool resolution, exports availability types through plugin SDK entrypoints, documents the contract, and adds focused planner/plugin tests.

## Best Possible Solution

Keep the registration-boundary diagnostic and documented SDK availability contract, but make every descriptor availability fact use the same active runtime config source on cached and uncached paths before merging.

## Reproduction Assessment

Yes. Current main is source-reproducible: empty `allOf`/`anyOf` descriptors produce `unsupported-signal` diagnostics and `buildToolPlan` hides the tool without a registration-time callback or log path.

## Solution Assessment

No, not yet. The registration-boundary planner approach is the right shape, but the latest `plugin-enabled` integration must use the active runtime config consistently before this is the best mergeable fix.
