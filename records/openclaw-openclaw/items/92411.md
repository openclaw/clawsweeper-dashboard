## Summary

This PR should stay open because the diagnostic callback direction is useful, but the current branch publishes plugin tool availability as an SDK contract without making `resolvePluginTools` enforce it, and one documented `core` import path is broken.

## What This Changes

The PR adds an `onHiddenDiagnostic` planner callback, audits plugin tool descriptors during `resolvePluginTools`, preserves and documents plugin tool `availability`, and adds focused regression tests.

## Best Possible Solution

Keep the planner pure and merge only after plugin tool availability is either fully enforced as a documented SDK contract or confined to an internal registration-time diagnostic surface.

## Reproduction Assessment

Yes. Current main is source-reproducible because `buildToolPlan` records unsupported-signal diagnostics without a callback/logging path, and PR head is source-reproducible because `resolvePluginTools` returns tools before the audit and discards the planner result.

## Solution Assessment

No. The callback and registration-audit layer is plausible, but publishing plugin `availability` as an SDK contract without enforcement is not the best fix; either enforce it end to end or keep the public SDK surface out of this PR.
