## Summary

Keep open: the patch improves the queue state classification, but it is still blocked because a send-evidence `unknown_after_send` entry can replay the full batch when reconciliation returns `not_sent`, which is the reported duplicate-message failure mode.

## What This Changes

The PR hoists `platformSendStarted`, marks required mid-batch outbound failures with prior send evidence as `unknown_after_send`, and adds unit plus real SQLite queue coverage for that path.

## Best Possible Solution

Keep the state transition only after adding explicit partial-batch recovery semantics and tests for a send-evidence `unknown_after_send` entry whose reconciliation returns `not_sent`.

## Reproduction Assessment

Yes, at source level. Current main can leave partial send evidence on a queued batch, and recovery still replays `unknown_after_send` entries when reconciliation reports `not_sent`; no live Telegram repro was run in this read-only review.

## Solution Assessment

No. Marking send-evidence failures as `unknown_after_send` is useful, but the best fix needs recovery to distinguish partial-send evidence from ordinary unknown-send replay or persist per-payload evidence.
