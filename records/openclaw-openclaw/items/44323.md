## Summary

Keep this PR open for maintainer review rather than closing it. Current main has a newer gateway-aware `noteMemorySearchHealth` path and still does not include this PR's config-load validator, but the PR branch is not ready as-is because its added diagnostics can contradict current gateway-ready behavior, its per-agent validation gate is incomplete, and it mixes unrelated cron reliability plus agent lifecycle/watchdog modules into the memory-search change.

## What This Changes

The PR branch adds config-load and doctor diagnostics for `memorySearch.provider` with tests, and also adds cron reliability and agent lifecycle/watchdog runtime modules.

## Best Possible Solution

A refreshed narrow PR should extend the current `noteMemorySearchHealth`/doctor contribution path with structured output while preserving auto mode, custom provider ids, provider default models, QMD, gateway-probe readiness, and auth-store/env/provider credential fallback. Static config-load validation should be limited to exact contract-safe checks such as explicitly empty fields, and the cron/lifecycle/watchdog work should be split into separate proposals.

## Reproduction Assessment

Yes for the PR review findings: the diff itself shows a gateway-unaware diagnostic call added after the gateway-aware health check, and a per-agent validator hidden behind a defaults-only gate. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, this PR is not the best way to solve the issue as currently shaped. The safer direction is to build on current main's gateway-aware doctor diagnostics and add only narrow static validation where the runtime contract makes it exact, instead of duplicating checks and mixing unrelated runtime modules.

## Review Findings

Full review comments:

- **[P2] Align diagnostics with gateway readiness:** `src/commands/doctor.ts:328`
  - body: This call runs `checkMemorySearch` immediately after `noteMemorySearchHealth`, but the new checker does not receive `gatewayMemoryProbe`. For explicit OpenAI configs where the running gateway reports embeddings ready but the CLI process cannot resolve the key, current health emits a non-failing readiness note; this added call then emits `[FAIL] memorySearch configuration invalid`, giving contradictory doctor output for a working gateway. Fold the structured output into the existing probe-aware health path or pass the probe context.
  - confidence: 0.89
- **[P2] Validate per-agent configs without defaults:** `src/config/validation.memory-search.ts:117-118`
  - body: The exported validator returns before it reaches `agents.list` unless `agents.defaults.memorySearch` exists and is enabled. A config with only `agents.list[0].memorySearch.provider: "ollama"` and `remote.baseUrl: ""` therefore passes validation even though the new default-path check rejects the same value, leaving agent-specific bad config to runtime. Gate each config object independently instead of using the defaults object as the global switch.
  - confidence: 0.86
