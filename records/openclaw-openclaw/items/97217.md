## Summary

Keep this PR open: the Azure guarded-fetch hardening is useful, but the proposed non-OK wrapper eagerly reads error bodies before the OpenAI SDK can run its retry/cancel logic, which can turn retryable slow 429/5xx responses into stalls until the cap, stream end, or timeout.

## What This Changes

The PR injects resolved-base-url guarded fetch into both Azure OpenAI Responses SDK clients, exports the SSE buffer cap, and adds transport tests for JSON/SSE and non-OK body caps.

## Best Possible Solution

Keep the resolved-base-url guarded-fetch injection, but make the non-OK cap a lazy bounded response body so SDK retry/cancel semantics remain intact before maintainers decide the Azure rollout.

## Reproduction Assessment

Yes. Source inspection shows current main has no custom Azure Responses SDK fetch, and the PR wrapper reads non-OK bodies before returning to the OpenAI SDK, whose source retries/cancels retryable response bodies before calling response.text().

## Solution Assessment

No, not as submitted. SDK fetch injection with the resolved Azure base URL is the right seam, but the non-OK cap should be lazy rather than eagerly buffering before SDK retry logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Return a lazy bounded error body instead of eagerly reading it:** `src/llm/providers/azure-openai-responses.ts:216-222`
  - body: This wrapper reads every non-OK body before returning the `Response` to the OpenAI SDK. The SDK decides whether a non-OK response is retryable before it reads the body, and cancels retryable bodies without `response.text()`, so a slow or never-ending retryable 429/5xx stream now blocks the SDK from retrying until this loop reaches 64 KiB, the stream ends, or the request times out. Wrap the body in a bounded stream that enforces the cap when consumed, rather than pre-buffering it in `fetch`.
  - confidence: 0.9
