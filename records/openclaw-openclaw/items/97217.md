## Summary

Keep open. Current main still leaves Azure Responses SDK clients on the SDK default fetch and still returns non-OK guarded SDK responses before SSE sanitization, while this PR is a focused security/availability hardening with no blocking code findings; the remaining question is maintainer acceptance of the guarded-fetch compatibility and large-error-body truncation behavior before merge.

## What This Changes

The branch wires both Azure Responses SDK client paths through `buildGuardedModelFetch`, adds a lazy 64 KiB cap for non-OK guarded SDK response bodies, and adds regression tests for Azure JSON synthesis, non-OK body capping, and cancel-before-read behavior.

## Best Possible Solution

Land the centralized guarded-fetch/cap approach after maintainers accept the upgrade-sensitive transport behavior and the current-head CI gates finish.

## Reproduction Assessment

Yes. Current main source shows Azure Responses SDK clients lack a custom fetch and the shared sanitizer returns non-OK responses unchanged; the PR body also provides real local HTTP-server proof for the bounded/cancel behavior after the patch.

## Solution Assessment

Yes. A shared lazy cap plus Azure client fetch injection is the narrowest maintainable fix I found; per-provider eager wrappers were a worse fit because OpenAI SDK source cancels retryable non-OK bodies before retrying.
