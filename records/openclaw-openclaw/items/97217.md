## Summary

Keep open for maintainer review: the patch appears focused and correct, with sufficient live-output proof, but it intentionally changes Azure Responses networking and error-body behavior in a compatibility-sensitive provider path.

## What This Changes

The PR exports the SSE buffer cap constant, injects resolved-base-url guarded fetch into both Azure OpenAI Responses SDK constructors, adds a 64 KiB non-OK body cap wrapper, and adds focused transport tests.

## Best Possible Solution

Land the focused Azure guarded-fetch hardening after maintainers accept the stricter endpoint policy and error-body truncation behavior; add constructor-level coverage only if they want stronger regression proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main builds Azure Responses SDK clients without custom fetch, and OpenAI SDK v6.39.1 buffers final non-OK error bodies with response.text(); I did not run a live Azure endpoint in this read-only review.

## Solution Assessment

Mostly yes. SDK fetch injection is the existing narrow seam and using the resolved Azure base URL is the right owner boundary; a later shared non-OK cap may be cleaner but would broaden compatibility impact across all SDK providers.
