## Summary

Keep open for maintainer review: current main does not already solve this Azure Responses bounded-read path, the patch appears correct, and the remaining concern is the intentional compatibility and availability tradeoff from moving existing Azure SDK traffic onto guarded transport.

## What This Changes

The PR injects guarded fetch into both Azure OpenAI Responses SDK client constructors and adds a 64 KiB cap for non-OK response bodies, with focused transport tests.

## Best Possible Solution

Land the focused Azure guarded-fetch hardening after maintainers accept the stricter endpoint policy and error-body truncation behavior; add provider-level constructor coverage only if they want stronger regression proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main builds Azure Responses SDK clients without custom fetch, and OpenAI SDK 6.39.1 buffers non-OK error bodies with response.text(). I did not run a live Azure endpoint in this read-only review.

## Solution Assessment

Mostly yes. SDK fetch injection is the existing narrow seam and using the resolved Azure base URL is the right owner boundary; a shared non-OK cap would be cleaner later but would broaden compatibility impact across all SDK providers.
