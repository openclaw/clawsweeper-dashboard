## Summary

Keep this PR open: the Azure guarded-fetch wiring is a real security/availability fix, but the current non-OK wrapper eagerly consumes error bodies before the OpenAI SDK can run its retry or token-refresh logic.

## What This Changes

The PR exports the SSE buffer cap constant, injects `buildGuardedModelFetch` with the resolved Azure base URL into both Azure Responses SDK client paths, and adds a provider-local non-OK response body cap plus tests.

## Best Possible Solution

Keep the resolved-base guarded fetch injection, but make the non-OK cap a lazy bounded response body so the SDK can still retry or refresh from status and headers before reading error text.

## Reproduction Assessment

Yes. Source inspection shows `azureGuardedFetch` reads the non-OK body before returning a Response, while OpenAI SDK 6.39.1 normally decides retry/refresh from the Response first and cancels the body on retry.

## Solution Assessment

No, not as written. The SDK fetch hook and resolved Azure base URL are the right seam, but the non-OK cap should be implemented as a lazy bounded stream rather than an eager pre-SDK body read.
