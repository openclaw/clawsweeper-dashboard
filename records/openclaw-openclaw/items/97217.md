## Summary

Keep open: current main still lacks guarded fetch on Azure OpenAI Responses SDK clients, and the PR addresses a real unbounded-read hardening gap. It is not closeable as implemented or superseded; merge should wait for maintainer acceptance of the guarded-transport compatibility risk and ideally one narrow provider-level coverage improvement.

## What This Changes

The PR wires Azure OpenAI Responses SDK clients through `buildGuardedModelFetch`, adds a 64 KiB cap for non-OK response bodies, exports the shared SSE buffer cap constant, and adds focused guarded-fetch tests.

## Best Possible Solution

Land the Azure guarded-fetch hardening after maintainers accept the stricter transport rollout and either add provider-level wrapper coverage or consciously accept the current proof gap.

## Reproduction Assessment

Yes. Source inspection shows current main creates Azure Responses SDK clients without custom fetch, while OpenAI SDK v6.39.1 uses `ClientOptions.fetch` and reads non-OK error bodies with `response.text()`.

## Solution Assessment

Mostly yes. SDK fetch injection is the narrow existing seam and matches the transport-stream sibling, but provider-level coverage should exercise the actual `azureGuardedFetch` wrapper instead of copying its loop in a shared fetch test.
