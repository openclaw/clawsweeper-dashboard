## Summary

Keep this open. Current main still lacks native plugin-contributed Control UI pages: navigation/rendering are fixed to built-in tabs, the bootstrap contract has no plugin page or import-map metadata, the manifest contract has no `ui.pages`, and the current descriptor hook is a narrower data-only foundation rather than the requested page/tab loader.

## What This Changes

Request to let plugins declare native Control UI pages or tabs in `openclaw.plugin.json`, loaded at runtime as Custom Elements inside the existing Control UI shell.

## Best Possible Solution

Keep this as active design work for a versioned Control UI plugin-page/contribution API that decides between full plugin-owned pages, host-owned descriptor slots, or both, then documents the manifest/SDK declarations, asset serving, navigation grouping, gateway access, auth scopes, disablement, and security model.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Current-main source inspection gives a high-confidence absence proof across the manifest, bootstrap config, navigation shell, renderer, and existing descriptor API.

## Solution Assessment

Unclear. The proposed Custom Element loader may be a valid end state, but maintainers should first choose the Control UI trust boundary and whether #71736’s host-owned descriptor model is the safer first implementation.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-owned UI code or descriptors inside the authenticated Control UI surface.

Concerns:

- **[high] Define the Control UI plugin code trust boundary:**
  - body: Native plugin pages could share DOM state, theme variables, gateway client access, and operator authentication context with plugin-provided browser code. A design needs explicit CSP, asset-origin, dependency/runtime sharing, operator-scope, data-access, disable/unload, and trusted-vs-untrusted plugin rules before implementation.
