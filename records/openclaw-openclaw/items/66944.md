## Summary

Keep open. Current main still lacks native plugin page/tab contributions: navigation, render dispatch, bootstrap config, and plugin manifests remain fixed around built-in views, while the adjacent descriptor API is data-only and narrower than the requested plugin-owned page system. Recent issue discussion also adds live plugin-author demand, including Openclaw-Quest and a status-page plugin use case.

## What This Changes

Request a plugin UI extension system where plugins can declare native Control UI pages, routes, and browser-loaded Custom Elements through the plugin manifest or SDK.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed Control UI plugin-page API that is reconciled with the contribution-slots RFC while preserving the native page/tab use case.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but source inspection on current main confirms the requested manifest page declarations, bootstrap metadata, dynamic routing, and browser loader are absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native Custom Element tabs could address the UX gap, but maintainers should first choose the public API, rendering model, trust boundary, and relationship to the safer host-owned descriptor work in https://github.com/openclaw/openclaw/issues/71736.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would project plugin-owned browser UI into the authenticated Control UI.
