## Summary

Keep open. Current main and the latest release still have static Control UI navigation plus a narrow descriptor/action host-hook foundation, while the closest implementation path, https://github.com/openclaw/openclaw/pull/80388, is open and still needs maintainer API and security-boundary review.

## What This Changes

Add a plugin UI extension system so plugins can contribute native Control UI pages or tabs instead of relying on standalone HTTP pages or hardcoded core UI changes.

## Best Possible Solution

Choose and land a versioned plugin page or entry-point contract, likely by reviewing or narrowing https://github.com/openclaw/openclaw/pull/80388, with explicit browser trust, launch/auth, backend access, navigation grouping, docs, and UI/protocol tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request, and source inspection verifies current main lacks plugin page declarations, bootstrap metadata, dynamic native navigation, and a browser loader for plugin pages.

## Solution Assessment

Unclear until maintainers decide the browser trust model; a generic plugin entry-point/page seam is the right boundary, but arbitrary in-shell plugin JavaScript versus sandboxed iframe or data-only descriptors remains a product and security decision.
