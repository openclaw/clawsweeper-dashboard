## Summary

Keep open. Current main still lacks manifest-declared native plugin pages/tabs, plugin page bootstrap metadata, import-map delivery, and a browser-side Custom Element loader; the related open Control UI contribution-slots RFC is adjacent but explicitly does not close this native-page request.

## What This Changes

Request a plugin UI extension system so plugins can declare native Control UI pages/tabs and load plugin-owned Custom Elements inside the main shell.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed plugin page or panel API with safe asset loading, navigation grouping, gateway/auth access rules, disablement semantics, docs, and contract/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main lacks the requested manifest page declarations, bootstrap metadata, dynamic routing, and browser plugin loader.

## Solution Assessment

Unclear. Native Custom Element pages could solve the reported UX gap, but maintainers should first choose the rendering model, trust boundary, and relationship to the safer host-owned descriptor work in https://github.com/openclaw/openclaw/issues/71736.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-provided browser UI inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin-code trust boundary:**
  - body: Native plugin pages could share DOM state, bootstrap identity/config, Gateway client access, and operator scopes with plugin-provided browser code. The design needs explicit CSP, origin, dependency-sharing, permission, disablement, and revocation rules before implementation.
