## Summary

Keep open. Current main still has static Control UI navigation and an adjacent descriptor hook, but not manifest-declared native plugin pages, bootstrap page/import-map metadata, dynamic routing, or a browser-side plugin Custom Element loader; the related https://github.com/openclaw/openclaw/issues/71736 thread is adjacent design work, not a replacement for this native page/tab request.

## What This Changes

Add a plugin UI extension system so plugins can declare native Control UI pages or tabs from plugin metadata and have them load inside the main Control UI shell.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed Control UI plugin-page API, or explicitly fold it into a broader contribution-slot design while preserving native page/tab requirements and real plugin examples.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence absence proof across navigation, rendering, bootstrap config, manifest types/docs, and the current descriptor API.

## Solution Assessment

Unclear. A Custom Element loader may solve the reported UX gap, but maintainers should first choose the rendering and trust boundary and decide how it relates to host-owned descriptor slots in https://github.com/openclaw/openclaw/issues/71736.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-provided browser UI inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin-code trust boundary:**
  - body: Native plugin pages could share DOM state, theme variables, authenticated bootstrap context, gateway client access, and operator scopes with plugin-provided browser code. Any design needs explicit CSP, asset-origin, scope, disablement, dependency-sharing, and trusted-vs-untrusted plugin rules before implementation.
