## Summary

Keep open. Current main still lacks native plugin page/tab contributions; the existing descriptor surface is narrower, and the new Control UI entry-point PR is open but not merged.

## What This Changes

The issue requests a plugin UI extension system that lets plugins contribute native Control UI pages or tabs from plugin metadata or SDK registration.

## Best Possible Solution

Land or reject a maintainer-approved, versioned plugin UI entry/page API with explicit auth, scope, CSP/sandbox, disablement, docs, and contract/UI tests, likely coordinated with the open entry-point PR and the broader contribution-slots RFC.

## Reproduction Assessment

Yes, at source level: current main still has fixed Control UI navigation/rendering and no manifest or bootstrap shape for native plugin pages. This is a feature gap rather than a runtime bug reproduction.

## Solution Assessment

Unclear: the native page idea addresses a real UX gap, but maintainers still need to choose the API and browser trust model. The open entry-point PR is a plausible path, while the related RFC argues for safer host-owned descriptor surfaces for some cases.
