## Summary

Keep open. Current main still has static Control UI navigation plus a narrow data-only descriptor hook, but not manifest-declared native plugin pages, plugin page/import-map bootstrap metadata, dynamic tab routing, or a browser-side Custom Element loader; the related contribution-slots RFC is adjacent and explicitly does not settle the native page/tab design.

## What This Changes

Request a plugin manifest/SDK/UI system that lets plugins contribute native Control UI pages or tabs loaded as Custom Elements with shared theme and gateway context.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed Control UI plugin-page API that aligns with the broader contribution-slot RFC while preserving the native page/tab requirement.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; source inspection gives high confidence that current main lacks manifest page declarations, bootstrap plugin page metadata, dynamic routing, and a browser plugin loader.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native Custom Element tabs could solve the UX gap, but maintainers should first decide the public API, rendering model, plugin trust boundary, and relationship to the safer host-owned descriptor work in https://github.com/openclaw/openclaw/issues/71736.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would run plugin-provided browser UI inside the authenticated operator Control UI.
