## Summary

Keep open. Current main still has static Control UI tabs and only an adjacent descriptor hook; the requested plugin-owned native page/tab loader remains design work and overlaps with, but is not replaced by, #71736.

## What This Changes

Request to let plugins declare native Control UI pages/tabs in their manifest and load plugin Custom Elements inside the main UI shell.

## Best Possible Solution

Keep this as active design work for a versioned Control UI plugin-page/contribution API that defines manifest or SDK declarations, safe asset serving, navigation grouping, browser loading, gateway/client access, disablement, docs, and security rules.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence absence proof across navigation, rendering, bootstrap config, manifest docs/types, and the current descriptor API.

## Solution Assessment

Unclear. The requested Custom Element loader addresses a real plugin UX gap, but maintainers should first choose the trust boundary and decide how it relates to #71736's host-owned descriptor model.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it would place plugin-provided browser UI inside the authenticated Control UI surface.

Concerns:

- **[high] Define the Control UI plugin-code trust boundary:**
  - body: Native plugin pages could share DOM state, theme variables, authenticated bootstrap context, gateway client access, and operator scopes with plugin-provided browser code. The design needs explicit CSP, asset-origin, runtime-sharing, scope, disable/unload, and trusted-vs-untrusted plugin rules before implementation.
  - confidence: 0.92
