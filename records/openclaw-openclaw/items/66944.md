## Summary

Keep this open. Current main still has fixed built-in Control UI navigation and rendering, no `openclaw.plugin.json` `ui.pages` contract, no plugin page/import-map bootstrap metadata, and no browser-side loader for plugin Custom Elements. The merged host-hook work adds adjacent data-only UI descriptors, and the open RFC/follow-up PRs are useful context, but they do not implement or fully supersede this native plugin page/tab request.

## What This Changes

Request to let plugins declare native Control UI pages or tabs in `openclaw.plugin.json`, loaded at runtime as Custom Elements inside the existing Control UI shell.

## Best Possible Solution

Keep this as active design work for a versioned Control UI plugin-page/contribution API. The end state should define whether OpenClaw supports full plugin-owned Custom Element pages, host-owned descriptor slots, or both, then document manifest/SDK declarations, asset serving, navigation grouping, gateway/client access, auth scopes, unload/disable behavior, CSP/trust boundaries, and contract/UI coverage before plugin authors depend on it.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request, not a bug report. Current-main source inspection gives a high-confidence absence proof: the manifest, bootstrap config, navigation shell, UI renderer, and existing descriptor API have no native plugin page path.

## Solution Assessment

Unclear. The proposed full Custom Element loader may be a valid end state, but it is not proven to be the safest first implementation while #71736 is exploring host-owned descriptor slots that avoid arbitrary browser JavaScript in the operator UI.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would let plugin-owned UI code or descriptors enter the authenticated Control UI surface.

Concerns:

- **[high] Define the Control UI plugin code trust boundary:**
  - body: Native plugin pages could share DOM state, theme variables, gateway client access, and operator authentication context with plugin-provided browser code. A design needs explicit CSP, asset-origin, dependency/runtime sharing, operator-scope, data-access, disable/unload, and trusted-vs-untrusted plugin rules before implementation.
