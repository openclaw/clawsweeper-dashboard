## Summary

Keep open. Current main and the latest release have only the narrower plugin Control UI descriptor/session-action foundation, while the requested native plugin page/tab contribution model is still represented by an open RFC and an open implementation PR that needs maintainer API and security review.

## What This Changes

The issue requests a plugin UI extension system so plugins can contribute native Control UI pages or tabs from plugin metadata instead of only standalone pages.

## Best Possible Solution

Keep this issue open until maintainers accept and land a versioned plugin UI entry/page contract, preferably aligned with the open RFC so native pages, panels, actions, auth, disablement, and browser rendering do not splinter.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main check: plugin manifests have no ui.pages field, the public descriptor surface is limited to session/tool/run/settings metadata, and Control UI navigation is still built from fixed tabs. This is a missing feature rather than a broken existing behavior repro.

## Solution Assessment

Unclear. The need for an integrated plugin UI seam is real, but the proposed same-DOM Lit/import-map page model is not yet proven safer than a versioned host-owned or sandboxed entry-point contract with explicit auth, scopes, disablement, and security tests.
