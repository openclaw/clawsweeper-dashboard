## Summary

Keep open. Current main still has static Control UI navigation plus a narrow descriptor hook, but not manifest-declared native plugin pages, plugin page/import-map bootstrap metadata, dynamic tab routing, or a browser-side plugin Custom Element loader; the live Openclaw-Quest comments and the adjacent contribution-slots RFC show active design input, not a replacement for this request.

## What This Changes

The issue requests a plugin UI extension system that lets plugins declare native Control UI pages or tabs via plugin metadata and load browser Custom Elements inside the existing Control UI shell.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed Control UI plugin-page API, or explicitly fold it into a broader contribution-slot design while preserving the native page/tab requirements and concrete plugin examples.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives high confidence that current main lacks the requested manifest page declarations, bootstrap page/import-map metadata, dynamic routing, and browser plugin loader.

## Solution Assessment

Unclear. A Custom Element loader may solve the reported UX gap, but maintainers should first choose the rendering model, plugin trust boundary, and relationship to the safer host-owned descriptor slots in https://github.com/openclaw/openclaw/issues/71736.
