## Summary

Keep open. Current main still has static Control UI navigation and a narrow descriptor hook, but not manifest-declared native plugin pages, plugin page/import-map bootstrap metadata, dynamic tab routing, or a browser-side plugin Custom Element loader.

## What This Changes

The issue requests a plugin UI extension system where plugins can declare native Control UI pages/tabs from their manifest and load them at runtime inside the main shell.

## Best Possible Solution

Keep this as active design work for a versioned, security-reviewed Control UI plugin-page API, or explicitly fold it into the broader contribution-slot design while preserving the native page/tab requirement.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main lacks the requested manifest page declarations, bootstrap metadata, dynamic routing, and browser plugin loader.

## Solution Assessment

Unclear. A Custom Element loader could solve the reported UX gap, but maintainers should first choose the rendering model, plugin trust boundary, and relationship to the safer host-owned descriptor work in https://github.com/openclaw/openclaw/issues/71736.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-provided browser UI inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin-code trust boundary:**
  - body: Native plugin pages could share DOM state, theme variables, authenticated bootstrap context, Gateway client access, and operator scopes with plugin-provided browser code. Any design needs explicit CSP, asset-origin, scope, disablement, dependency-sharing, trusted-vs-untrusted plugin, and revocation rules before implementation.
