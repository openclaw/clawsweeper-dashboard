## Summary

Keep open. Current main still lacks native plugin page/tab contributions, and the adjacent descriptor API is narrower than this request; recent discussion also shows real plugin-author demand and unresolved API/security questions.

## What This Changes

The issue requests a plugin UI extension system that lets plugins contribute native Control UI pages or tabs from plugin metadata or runtime declarations.

## Best Possible Solution

Keep this as the canonical native plugin page/tab design track, reconciled with the descriptor-slot RFC and backed by an explicit security model before implementation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a runtime bug, but source inspection on current main confirms the requested manifest page declarations, bootstrap metadata, dynamic routing, and browser loader are absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Native plugin pages would address the UX gap, but maintainers should first choose the public API, rendering model, trust boundary, and relationship to the safer host-owned descriptor work in https://github.com/openclaw/openclaw/issues/71736.
