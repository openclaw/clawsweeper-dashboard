## Summary

Keep open: this reports broken existing WhatsApp multi-account cold-start behavior, and current main still routes configured accounts through the startup/auth path that matches the reported `readWebAuthState` crash surface. I found no current-main fix or canonical duplicate; the remaining gap is a live packaged cold-boot reproduction.

## What This Changes

The issue asks OpenClaw to stop crashing a secondary WhatsApp account with an explicit `authDir` on gateway boot and to register persistent inbound listeners for both accounts.

## Best Possible Solution

Patch the WhatsApp startup/runtime-loading path so linked default and secondary accounts both read auth state and register listeners after restart, with regression coverage for a secondary explicit `authDir`.

## Reproduction Assessment

No high-confidence live reproduction on current main yet. The reporter provides concrete v2026.5.2 steps, and current source maps those steps to `server-channels` account startup and WhatsApp `readWebAuthState`, but I did not run a cold packaged gateway with two linked WhatsApp accounts.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable fix should start in the WhatsApp plugin startup/runtime-loading path, adding a generic plugin-loader seam only if reproduction proves the packaged loader drops the runtime surface.
