## Summary

Keep open: the docs and current source define WhatsApp multi-account support with per-account `authDir`, and current main still cold-starts configured accounts through the WhatsApp auth check that calls `readWebAuthState` before listener startup. I did not find a merged fix, release note, or canonical duplicate; the remaining gap is reproducing the exact packaged gateway path that returns an undefined `readWebAuthState` surface.

## What This Changes

The issue requests that a secondary WhatsApp account with an explicit `authDir` survive cold gateway boot and register its persistent listener instead of crashing during auth-state reads.

## Best Possible Solution

Patch the WhatsApp plugin startup/runtime loading path so every configured account, including one with an explicit `authDir`, reads auth state through a valid runtime surface and starts its listener after restart, with regression coverage for default plus secondary cold boot.

## Reproduction Assessment

No, not with high confidence from this review alone. The reporter gives concrete steps and current source maps to the startup/auth path, but this pass did not run a packaged gateway cold boot or recreate the exact undefined `readWebAuthState` TypeError.
