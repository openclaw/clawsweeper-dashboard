## Summary

Keep open: this is a reported break in documented WhatsApp multi-account cold-start behavior, and current main still routes every configured account through the same gateway startup and WhatsApp auth-state path that matches the reported crash surface. Source inspection did not show an implemented fix or a canonical superseding item; the remaining gap is live packaged reproduction of the exact `readWebAuthState` TypeError.

## What This Changes

A secondary WhatsApp account with an explicit `authDir` should cold-start and register its inbound listener after gateway restart instead of crashing while reading auth state.

## Best Possible Solution

Patch the WhatsApp full-runtime or plugin-loader cold-start path so linked default and secondary accounts both read scoped auth state and register persistent listeners after restart, with regression coverage for a secondary explicit `authDir`.

## Reproduction Assessment

No high-confidence live reproduction on current main. The reporter provides concrete v2026.5.2 steps, and current source maps those steps to gateway account startup and `readWebAuthState`, but I did not execute a packaged gateway restart.

## Solution Assessment

Not applicable; no patch is proposed yet. The narrowest maintainable fix is likely in WhatsApp runtime/bootstrap loading, with a generic plugin-loader seam only if packaged reproduction proves the loader drops the runtime surface.
