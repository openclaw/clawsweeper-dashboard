## Summary

Keep open: this is a concrete report of broken existing WhatsApp multi-account cold-start behavior, and current main still routes each configured account through the startup auth check that reaches `readWebAuthState`; I found no merged/shipped fix or canonical duplicate in the local evidence.

## What This Changes

The issue reports that a WhatsApp secondary account with an explicit `authDir` pairs successfully in-session but crashes on gateway restart with an undefined `readWebAuthState` access.

## Best Possible Solution

Patch the WhatsApp plugin startup/runtime loading path so default and secondary accounts, including explicit `authDir` accounts, cold-start and register persistent listeners after restart, with a regression covering the packaged loader path.

## Reproduction Assessment

Unclear: the reporter gives concrete steps and source inspection maps those steps to the startup auth path, but I did not reproduce the current-main packaged gateway TypeError. Current source still exports `readWebAuthState`, which keeps the exact failing mechanism unresolved.

## Solution Assessment
