## Summary

Keep open: the report describes broken existing WhatsApp multi-account cold-start behavior, and current main still routes every configured account through the same startup/auth path that matches the reported crash surface. I did not find a current-main fix or canonical duplicate, and the exact packaged `readWebAuthState` TypeError still needs runtime reproduction before patching.

## What This Changes

The issue asks OpenClaw to cold-start both default and secondary WhatsApp accounts when the secondary account uses an explicit `authDir`.

## Best Possible Solution

Patch the WhatsApp full-runtime or plugin-loader cold-start path so linked default and secondary accounts both read auth state and register persistent listeners after restart, with regression coverage for a secondary explicit `authDir`.

## Reproduction Assessment

No high-confidence live reproduction on current main yet. The reporter provides concrete v2026.5.2 steps, and current source maps those steps to `server-channels` account startup and WhatsApp `readWebAuthState`, but the packaged cold-boot TypeError was not executed locally.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable fix is in the WhatsApp plugin startup/runtime-loading path or a generic plugin-loader seam if reproduction proves the packaged loader drops the runtime surface.
