## Summary

Keep open: this is a fresh report of broken existing WhatsApp multi-account cold-start behavior, and current main still routes configured accounts through the same gateway startup and WhatsApp lazy-runtime auth check that matches the reported `readWebAuthState` crash surface. I did not find a merged fix, shipped release note, protected duplicate, or current-main evidence strong enough to close it.

## What This Changes

The issue requests that WhatsApp secondary accounts with an explicit `authDir` start and register inbound listeners after gateway restart instead of crashing while the default account starts.

## Best Possible Solution

Fix the WhatsApp full-runtime cold-start path so every configured account, including a secondary account with explicit `authDir`, reads auth state and registers its persistent listener after gateway restart, with regression coverage for that account shape.

## Reproduction Assessment

No high-confidence live reproduction on current main yet. The reporter provided concrete v2026.5.2 steps, and source inspection maps those steps to the current gateway account startup and WhatsApp `readWebAuthState` call path, but the cold-boot TypeError was not executed locally.

## Solution Assessment

No patch is proposed yet. The narrow maintainable direction is to fix the WhatsApp full-runtime startup path, or a generic plugin-loader seam only if reproduction proves packaged loading drops the runtime export.
