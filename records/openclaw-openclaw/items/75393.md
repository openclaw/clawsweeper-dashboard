## Summary

Keep open: the underlying Copilot eager-input-streaming bug is valid, but this PR only patches the default helper and synthetic fallback paths while registered static and refreshed live catalog models remain uncovered; it also lacks after-fix real behavior proof.

## What This Changes

The branch adds `compat.supportsEagerToolInputStreaming: false` to `buildCopilotModelDefinition()` and the synthetic `resolveCopilotForwardCompatModel()` fallback for GitHub Copilot models.

## Best Possible Solution

Land a provider-owned Copilot fix that applies `supportsEagerToolInputStreaming: false` to every Anthropic Copilot model source, preserves it through catalog/config normalization where needed, and adds regression coverage for a registered Claude model.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the linked issue has concrete v2026.4.29 and v2026.5.4 rejection logs, and current main plus the pinned pi-ai contract show the metadata gap remains. I did not run a live Copilot request in this read-only review.

## Solution Assessment

No. The compat flag is the right narrow mechanism, but this PR applies it only to helper/fallback paths and misses registered static and live catalog models that cover the reported request path.
