## Summary

Keep open: this PR targets a real Copilot Claude schema bug, but the patch does not cover the registered static catalog or live catalog model rows used by the reported path, and the external PR still lacks after-fix real behavior proof.

## What This Changes

Adds `compat.supportsEagerToolInputStreaming: false` to the Copilot default model builder and the synthetic forward-compatible fallback model.

## Best Possible Solution

Land a Copilot provider fix that applies `supportsEagerToolInputStreaming: false` to every Anthropic Copilot model source, including static manifest rows, live `/models` mapping, dynamic fallback models, and catalog/config normalization if needed, with focused regression coverage and live proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: https://github.com/openclaw/openclaw/issues/75348 provides concrete commands and 400 rejection logs, while current main still leaves registered and live Copilot Claude rows without the compat flag that the pinned pi-ai package requires. I did not run a live Copilot account in this read-only review.

## Solution Assessment

No. The compat flag is the right narrow mechanism, but this PR applies it to helper/fallback paths that do not cover the reported registered catalog path or live catalog rows; the maintainable fix should set and preserve the flag where those model rows are built.
