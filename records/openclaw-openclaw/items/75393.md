## Summary

Keep open: the linked Copilot bug is valid and the compat flag is the right mechanism, but this PR still misses registered/static and live-catalog Copilot Claude models, so the reported `github-copilot/claude-*` path can remain broken.

## What This Changes

The PR adds `compat.supportsEagerToolInputStreaming: false` to the Copilot model helper and synthetic forward-compatible fallback.

## Best Possible Solution

Land a provider-owned Copilot fix that applies `supportsEagerToolInputStreaming: false` to every Anthropic Copilot catalog path, including static manifest rows, live `/models` rows, and dynamic fallback models, with regression coverage for a registered Claude model.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: https://github.com/openclaw/openclaw/issues/75348 gives a concrete v2026.4.29 command and rejection logs, and current main still resolves registered Copilot Claude rows without the required compat flag. I did not run a live Copilot account in this read-only review.

## Solution Assessment

No. The compat flag is the narrow right mechanism, but this PR applies it to helper/fallback paths that do not cover the reported registered static or refreshed live catalog models; the safer fix is to update the catalog rows or shared mapper and assert the registered model resolution.
