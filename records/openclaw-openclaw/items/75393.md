## Summary

Keep this PR open: the compat flag is the right mechanism, but the current patch does not reach the manifest-registered Claude model from the linked repro, and it lacks changelog/test coverage for the user-facing provider fix.

## What This Changes

The PR adds `compat.supportsEagerToolInputStreaming: false` to Copilot's default model helper and dynamic forward-compatible fallback model builder.

## Best Possible Solution

Land a provider-owned Copilot fix that marks every Anthropic Copilot model path with `supportsEagerToolInputStreaming: false`, including the manifest/static catalog and dynamic fallback, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The linked issue provides a concrete v2026.4.29 command and gateway error for `github-copilot/claude-haiku-4.5`, and current main shows that model is still manifest-registered without the compat override while pi-ai defaults to sending `eager_input_streaming`.

## Solution Assessment

No. The compat flag is the narrow mechanism, but this PR applies it to paths that do not cover the reported registered static model; the safer fix is to update the static catalog or route it through a shared helper, then assert a registered Claude model resolves with the flag.
