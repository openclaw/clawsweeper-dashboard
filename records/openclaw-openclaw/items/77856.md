## Summary

Keep this PR open, but it is not merge-ready: the reported Telegram mention path is plausible, yet the diff accepts any `tg://user` text link as a bot mention and the PR lacks observed after-fix behavior proof.

## What This Changes

The PR changes Telegram `hasBotMention` to accept `text_link` entities with `tg://user` URL variants and adds helper tests for those cases.

## Best Possible Solution

Accept `text_link` mentions only when the linked or displayed identity can be tied to the bot, then add a regression that fails on current main and include redacted real Telegram update or delivery proof.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows current main would miss a `text_link` whose display text is not `@botUsername`, but the PR does not include a real Telegram update or log proving that exact payload.

## Solution Assessment

No, the current diff is not the best fix because URL-scheme matching alone cannot identify the bot. The safer path is to compare the link text to `@botUsername` or pass the bot ID into the helper before accepting `tg://user?id=`.
