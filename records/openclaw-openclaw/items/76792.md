## Summary

Keep open. The report maps to a real doctor diagnostic edge: current main still emits the exact no-active-plugin note when CLI backend resolution is null before considering the gateway memory probe, even though doctor already fetches gateway memory status and docs say it should cross-reference that result.

## What This Changes

Make `openclaw doctor` stop reporting “No active memory plugin is registered” when the running gateway or deep memory status shows the active memory plugin is healthy.

## Best Possible Solution

Use gateway memory status or a targeted plugin bootstrap as doctor’s source of truth, and suppress or reword the in-process no-runtime note when runtime memory is known healthy.

## Reproduction Assessment

Unclear for a live current-main run because this review was read-only and did not start a configured gateway. Source inspection shows the reported message is emitted whenever CLI-side backend resolution returns null before the ready gateway probe is considered.

## Solution Assessment

No. The current implementation still has a misleading early return; the narrow maintainable fix is to add the null-runtime plus ready-gateway path and keep the existing warning only when no runtime and no healthy gateway evidence exist.
