## Summary

Keep open: the PR has a plausible Feishu/Lark topic-anchor fix and supplied real behavior proof, but the current branch still regresses the existing normal-group `root_id` contract by routing replies to the root instead of the triggering message.

## What This Changes

The PR updates Feishu plugin routing, docs, changelog, and tests to keep topic reply anchors across assistant replies, message-tool sends, media/cards, reactions, and subagent completion notices.

## Best Possible Solution

Keep the Feishu topic-anchor direction, but limit root/thread send targets to topic, direct-topic, or explicit `replyInThread` paths while preserving plain group `root_id` replies on the triggering message.

## Reproduction Assessment

Yes, source-reproducible: current main has a guarded normal-group `root_id` regression test and the PR branch changes that expectation to the root message. The topic bug itself also has supplied live Feishu/Lark after-fix proof in the PR body.

## Solution Assessment

No, not yet: splitting typing/current-message anchors from topic send anchors is the right boundary, but the current implementation broadens threaded root delivery to plain group `root_id` events. The safer repair is to exclude `groupSessionScope: "group"` root-only events from root-targeted threaded delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep normal-group root_id replies on the triggering message:** `extensions/feishu/src/bot.ts:1366`
  - body: In `groupSessionScope: "group"` events Feishu can include `root_id`, and current main intentionally replies to the triggering `messageId` because root replies can move the response into a hidden thread; see https://github.com/openclaw/openclaw/issues/32980. This predicate makes any `ctx.rootId` use the threaded root send target, so normal group replies would go back to `om_original_msg` instead of `om_quote_reply`.
