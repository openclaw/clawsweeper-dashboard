## Summary

Keep open. Current main still matches the report: Slack omits the typing controller and disables lifecycle reactions when the source reply delivery mode is message-tool-only, while the shared typing policy and docs expect typing to remain visible for group/channel tool-only turns.

## What This Changes

The issue requests restoring Slack in-flight feedback, especially assistant thread status, typing reactions, and configured status reactions, when group/channel visible replies use message-tool-only delivery.

## Best Possible Solution

Update the Slack plugin so message-tool-only source turns still wire typing callbacks and `typingReaction`, and let explicitly enabled Slack lifecycle status reactions operate without re-enabling automatic final/block/preview delivery. Keep preview and native text streaming gated unless maintainers make a separate product decision for visible draft output in tool-only rooms.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a Slack group/channel turn resolving to `message_tool_only`, dispatch passes no typing callbacks and disables the status reaction controller before the shared instant typing policy can take effect.

## Solution Assessment

Yes. Decoupling in-flight feedback from automatic visible reply delivery is the narrowest maintainable fix because it preserves the private final-reply semantics while restoring the documented progress signal; streaming should remain a separate decision.
