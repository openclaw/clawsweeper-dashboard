## Summary

Keep open: current main now hot-reloads explicit `agents.list` and binding changes, but it still does not implement workspace-owned agent config discovery. The remaining request adds a new config source and filesystem trust boundary, so it needs maintainer product and security direction rather than cleanup closure.

## What This Changes

The issue requests an `agents.autoDiscover` configuration option that scans external workspace directories for agent config files and loads them without manual central registration.

## Best Possible Solution

Keep `agents.list` as the canonical registry unless maintainers define an operator-controlled discovery or import design with trusted roots, deterministic conflict rules, validation, and session-removal semantics.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a report of broken existing behavior. Source inspection confirms current main only treats explicit config entries as canonical agent definitions.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because it adds a second config authority before trust, conflict, validation, and session lifecycle rules are defined. The safer current path is explicit agent registration with hot reload while maintainers decide whether discovery belongs in core.
