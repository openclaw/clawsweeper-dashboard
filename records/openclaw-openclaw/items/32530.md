## Label Justifications

- `P2`: This is a useful multi-workspace ergonomics feature with limited current-user urgency, but its design spans configuration, gateway lifecycle, and security boundaries.
- `impact:session-state`: Automatic addition, mutation, or removal of agents can change routing and the association of live and persisted sessions with agent identities.
- `impact:security`: Loading configuration discovered from external workspaces creates a new filesystem trust and configuration-injection boundary.

## Summary

Keep open for maintainer product and security design. Current main and release v2026.6.5 support explicit agent creation plus hot application of `agents.list` changes, which addresses the restart concern raised in discussion, but they still do not discover agent configurations from external workspaces. The remaining request needs deliberate trust-root, collision, removal, and live-session semantics rather than a straightforward config patch.

## What This Changes

Requests trusted automatic discovery and loading of agent configurations from configured external workspace directories.

## Best Possible Solution

Keep `agents.list` as the single canonical runtime registry and, if maintainers approve discovery, add an operator-controlled reconciliation layer with explicit trusted roots, deterministic precedence, safe path handling, atomic validation, and defined active-session behavior.

## Reproduction Assessment

Not applicable: this requests a new discovery mode rather than reporting broken behavior under an existing contract. Source, docs, and release inspection confirm that explicit registration remains the supported workflow.

## Solution Assessment

No, not as currently proposed. Merging arbitrary glob-discovered files directly into runtime configuration would create competing sources of truth; a validated reconciliation layer feeding the canonical agent registry is safer and more maintainable.
