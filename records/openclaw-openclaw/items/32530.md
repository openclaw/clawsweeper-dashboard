## Summary

Keep open: current main and the latest release support explicit agent registration, `$include`, and hot reload for `agents.list`, but they still do not implement workspace-owned agent config discovery. The remaining request would add a new config authority and filesystem trust boundary, so it needs maintainer product and security direction rather than conservative cleanup closure.

## What This Changes

The issue requests a new `agents.autoDiscover` configuration source that scans workspace directories for agent config files and dynamically registers them.

## Best Possible Solution

Define a maintainer-approved, operator-controlled discovery or import design with trusted roots, deterministic conflict handling, validation failure behavior, and session lifecycle semantics before adding a core config source.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a report of broken existing behavior. Source inspection confirms current main uses explicit `agents.list` as the canonical agent definition surface.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because trust, conflict, validation, persistence, and session lifecycle rules are undefined. The current safer path is explicit agent registration with hot reload while maintainers decide whether discovery belongs in core.
