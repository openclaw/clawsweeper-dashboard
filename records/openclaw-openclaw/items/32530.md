## Label Justifications

- `P2`: This is a meaningful multi-project administration improvement, but supported explicit agent registration and hot-reload paths already cover the current workflow.
- `impact:security`: Scanning workspace directories for agent config would introduce a new trusted filesystem configuration input.
- `impact:session-state`: Automatically adding, replacing, or removing agent IDs can affect session identity, routing, and per-agent state.

## Summary

Keep open: current main still uses explicit `agents.list` plus `agents.create`/`agents.update` as the canonical agent configuration path, and there is no external workspace-owned agent config discovery. The remaining request adds a new config authority, filesystem trust boundary, and session lifecycle behavior, so it needs maintainer product and security design before implementation.

## What This Changes

Request to add `agents.autoDiscover` so OpenClaw can scan configured workspace directories for agent config files and automatically load those agents.

## Best Possible Solution

Define a maintainer-approved, operator-controlled discovery or import design with trusted roots, deterministic conflict handling, validation failure behavior, and session lifecycle semantics before adding a core config source.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a report of broken existing behavior. Source inspection confirms current main uses explicit `agents.list` as the canonical agent definition surface.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because trust, conflict, validation, persistence, and session lifecycle rules are undefined. The safer current path is explicit agent registration with hot reload while maintainers decide whether discovery belongs in core.
