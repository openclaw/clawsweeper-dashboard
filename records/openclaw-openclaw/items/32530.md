## Summary

Keep open: current main and the latest release still use explicit `agents.list` plus `agents.create`/`agents.update`/`agents.delete` as the agent source of truth, and no `agents.autoDiscover` or workspace-owned agent config path exists. This remains a product and security design request, not a stale or already-implemented item.

## What This Changes

Add config-driven scanning of external workspace directories so agent config files can be discovered and merged without manual central registration.

## Best Possible Solution

Define an operator-controlled discovery or import design with trusted roots, deterministic precedence, validation failure behavior, persistence/removal semantics, and session lifecycle rules; otherwise keep explicit `agents.create` plus hot reload as the supported path.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a broken existing behavior report. Source inspection confirms current main uses explicit `agents.list` and agent mutation RPCs as the canonical agent definition surfaces.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because trust, conflict, validation, persistence, and session lifecycle rules are undefined. The safer current path is explicit agent registration plus hot reload while maintainers decide whether discovery belongs in core.

## Risks / Open Questions

- Automatic workspace scanning would add a new trusted filesystem configuration input; trusted roots, symlink handling, glob expansion, validation failures, and malicious workspace files need explicit rules.
- Merging discovered agents with explicit `agents.list` would create two authorities unless precedence, persistence, removal, duplicate IDs, and error handling are designed.
- Dynamic discovery or removal can change agent IDs, routing eligibility, auth/profile scope, transcript ownership, and active session behavior.
- The already-implemented hot `agents.list` path may solve enough of the operational pain that maintainers should decide whether discovery is worth the extra config authority.
