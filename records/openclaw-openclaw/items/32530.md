## What This Changes

Requests automatic discovery and loading of agent configurations from external workspace directories instead of explicit central registration.

## Best Possible Solution

Keep `agents.list` as the single canonical registry and use its existing hot-reload path for zero-restart management; if maintainers still want discovery, first specify an operator-controlled import or synchronization design with trusted roots, deterministic conflict rules, validation, removal/session semantics, and no runtime fallback configuration source.

## Risks / Open Questions

- Scanning globs and automatically trusting workspace-owned files would expand the configuration trust boundary and could permit unintended agent definitions from writable or newly mounted directories.
- Merging discovered entries with `agents.list` would create two configuration authorities unless precedence, duplicate IDs, removal behavior, validation failures, and persistence semantics are defined.
- Dynamic removal or replacement of discovered agents needs explicit session-state semantics so existing sessions are not silently reassigned, orphaned, or associated with a changed agent definition.
- A watch mode would add long-lived filesystem watcher lifecycle, resource, and consistency behavior beyond the narrower startup-discovery proposal.
