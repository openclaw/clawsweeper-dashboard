## Summary

Keep open: current main and v2026.6.8 still support explicit `agents.list`, `$include`, and hot reload for explicit agent updates, but they do not implement workspace-owned agent configuration discovery. The remaining request adds a new config authority and filesystem trust boundary, so it needs maintainer product and security direction rather than conservative cleanup closure.

## What This Changes

Request to add workspace-directory scanning so agent config files can be discovered and loaded without manually registering every agent in the central OpenClaw config.

## Best Possible Solution

Define a maintainer-approved, operator-controlled discovery or import design with clear trust roots, deterministic merge/conflict behavior, validation failure handling, and agent/session lifecycle semantics before adding a core config source.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a report of broken existing behavior. Source inspection confirms current main and v2026.6.8 use explicit `agents.list` as the canonical agent definition surface.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because trust, conflict, validation, persistence, and session lifecycle rules are undefined. The safer current path is explicit agent registration plus hot reload while maintainers decide whether discovery belongs in core.
