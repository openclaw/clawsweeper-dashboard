## Summary

Keep open: current main still has no external workspace agent auto-discovery contract. The active config types, strict schema, agent enumeration, workspace directory listing, gateway agent list filtering, CLI docs, and plugin SDK docs all remain centered on explicit `agents.list` entries plus the default agent; related hot-reload and disk-session discovery work does not provide trusted workspace scanning or a plugin seam that contributes agent definitions before enumeration.

## What This Changes

Request to add automatic discovery and loading of agent configuration files from configured external workspace directories so multi-project deployments do not need to manually update `agents.list`.

## Best Possible Solution

Keep the issue open for a maintainer design decision that defines either a first-class `agents.autoDiscover` contract or a documented plugin/API contribution point that runs before agent enumeration. The design should specify trusted roots, accepted filenames, schema validation, deterministic scan order, explicit precedence against `agents.list`, zero-restart behavior, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The absence is high-confidence from current-main inspection: the config schema rejects the proposed key, enumeration reads only `agents.list`, and gateway tests preserve explicit agent scope over disk-only agent directories.

## Solution Assessment

Unclear: the requested direction is plausible, but the exact `agents.autoDiscover` shape is not yet proven to be the narrowest maintainable solution. A safer design may be a generic agent-definition contribution seam if maintainers want this extensible beyond core config scanning.
