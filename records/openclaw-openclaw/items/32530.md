## Summary

Keep open: current main and v2026.6.8 support explicit `agents.list` management and hot-reload those changes, but they still do not implement workspace-owned agent config discovery. The remaining request adds a new config source and filesystem trust boundary, so it needs maintainer product and security direction rather than cleanup closure.

## What This Changes

Request automatic discovery and loading of agent configs from configured workspace directories instead of requiring every agent to be registered in `agents.list`.

## Best Possible Solution

Keep `agents.list` as the canonical registry until maintainers define an operator-controlled discovery or import design with trusted roots, deterministic conflict rules, validation, and session-removal semantics.

## Reproduction Assessment

Not applicable: this is a new discovery/configuration capability request, not a report of broken existing behavior. Source inspection confirms current main only treats explicit config entries as canonical agent definitions.

## Solution Assessment

No: the proposed glob-based merge is not yet the best solution because it adds a second config authority before trust, conflict, validation, and session lifecycle rules are defined. The safer current path is explicit agent registration with hot reload while maintainers decide whether discovery belongs in core.
