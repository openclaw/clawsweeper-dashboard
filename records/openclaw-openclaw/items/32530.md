## Summary

Keep open: current main and the latest shipped release still lack external workspace agent-configuration discovery. The existing hot reload and `agents.create` paths cover adjacent manual agent management, but trusted glob scanning, config merge precedence, and hot add/remove semantics still need maintainer product/security design.

## What This Changes

The issue requests an `agents.autoDiscover` configuration surface that scans workspace directories for agent config files and merges discovered agents into the agent registry.

## Best Possible Solution

Define a supported discovery contract for trusted roots, validation, ordering, precedence, and hot add/remove behavior, then implement it in the agreed core config or plugin/API surface.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection at current main shows no `agents.autoDiscover` schema/type path and no external workspace config scan in the agent enumeration paths.

## Solution Assessment

Unclear: `agents.autoDiscover` is plausible, but maintainers still need to decide the discovery contract, trust boundaries, precedence rules, and whether zero-restart hot-load or a plugin/API surface is the better long-term path.
