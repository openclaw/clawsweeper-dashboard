## What This Changes

## Best Possible Solution

Strip QMD-scoped env from the mcporter process and put per-agent QMD env only on qmd-owned/generated server spawns, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes from source inspection: current main passes the QMD-scoped env into mcporter, and mcporter 0.10.x source uses XDG_CONFIG_HOME for config candidates. I did not run a live gateway memory_search reproduction in this read-only review.

## Solution Assessment

No as submitted. Removing XDG_CONFIG_HOME is the right immediate direction, but retaining per-agent XDG_CACHE_HOME/QMD_CONFIG_DIR on the mcporter process is not the clean owner-boundary fix for the full leak.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not pass QMD-scoped env through mcporter:** `extensions/memory-core/src/memory/qmd-manager.ts:523-524`
  - body: This new `mcporterEnv` still sets `QMD_CONFIG_DIR` and the agent qmd `XDG_CACHE_HOME` on the mcporter process. mcporter v0.10+ consumes `XDG_CACHE_HOME` for its own cache path, and any server started by mcporter inherits this env, so the patch only fixes config discovery while leaving the broader QMD env leak in place. Keep the qmd-specific env on generated/local qmd server entries, or strip it from external mcporter calls.
  - confidence: 0.84
- **[P3] Make the XDG_CONFIG_HOME assertion host-env independent:** `extensions/memory-core/src/memory/qmd-manager.test.ts:4902`
  - body: The implementation builds `mcporterEnv` from `...process.env`, so a developer or CI host that already sets `XDG_CONFIG_HOME` will still pass that user value through and this assertion will fail. Clear/stub the env var in the test, or assert that it is not the agent-scoped qmd path.
  - confidence: 0.9
