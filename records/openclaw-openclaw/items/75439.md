## Summary

Keep open: the PR targets a valid current-main heartbeat runaway, but the current head still gates existing immediate wake-now and blocked-task follow-up paths too broadly.

## What This Changes

The PR adds a shared heartbeat cooldown helper, runner bookkeeping, regression tests, a reason-kind export, and a changelog entry to rate-limit loop-prone non-interval heartbeat wakes.

## Best Possible Solution

Keep the shared cooldown gate, but carry explicit immediate-wake intent for hook wake-now, cron wake-now fallback, and blocked-task follow-ups while leaving exec/spawn and cron next-heartbeat paths gated.

## Reproduction Assessment

Yes. Current main's exec-event runaway is reproducible with the scheduler test included in the PR; the PR-head regressions are reproducible by evaluating shouldDeferWake after a prior run with hook:wake, a cron wake-now fallback reason, or background-task-blocked.

## Solution Assessment

No. The centralized helper and runner seam are the right shape, but the current policy conflates loop-prone event wakes with existing user/operator wake-now contracts; explicit wake intent or narrower reason metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve hook and cron wake-now requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist still excludes hook:* and the cron wake-now fallback path. After any prior heartbeat, those requests hit shouldDeferWake, return not-due, and the targeted runner exits before runOnce, so /hooks/wake, hook-agent wakeMode now completions/errors, and busy main-session cron --wake now fall back to the next scheduled heartbeat instead of waking immediately. Keep cron next-heartbeat gated by adding explicit wake-now metadata or a distinct immediate reason.
  - confidence: 0.93
- **[P2] Allow blocked-task follow-ups through the gate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Task delivery uses background-task for terminal updates but background-task-blocked for the follow-up system event after a blocked command. Since only background-task is allowlisted here, the follow-up is deferred behind nextDueMs once the agent has run, breaking the push-driven task completion contract. Add coverage for background-task-blocked or carry explicit immediate intent from the task registry.
  - confidence: 0.9
