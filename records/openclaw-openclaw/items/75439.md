## Summary

This PR should stay open: the latest branch keeps the useful centralized exec-event cooldown and changelog, but the cooldown policy still defers existing immediate wake contracts for hook/cron wake-now paths and blocked-task follow-ups.

## What This Changes

The PR adds a shared heartbeat cooldown helper, runner bookkeeping, regression tests, and a changelog entry to prevent non-interval heartbeat wake loops from repeatedly firing within the configured cadence.

## Best Possible Solution

Keep the centralized cooldown gate, but limit it to loop-prone feedback reasons while preserving documented immediate hook, cron wake-now, and task follow-up delivery contracts with regression coverage.

## Reproduction Assessment

Yes. On the PR head, `shouldDeferWake` returns `not-due` for `hook:wake`, `cron:*`, and `background-task-blocked` after a prior run, and the targeted runner branch returns before `runOnce`.

## Solution Assessment

No. The shared helper is the right shape, but the current policy is broader than the existing wake-now contracts; the safer fix is an explicit immediate-reason policy or metadata that keeps loop-prone exec/spawn wakes gated.
