## Summary

Keep this PR open: the latest head fixes the original exec-event runaway shape, but the centralized gate is still too coarse for documented immediate wakes and one blocked-task follow-up path. The remaining issues are concrete source-level regressions introduced by the PR and should be fixed before merge.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, wires targeted and broadcast heartbeat wakes through it, adds scheduler/cooldown regression tests, and adds a changelog entry for the heartbeat runaway fix.

## Best Possible Solution

Keep the centralized cooldown seam, but make wake immediacy an explicit policy rather than inferring it from broad reason strings. Feedback-loop sources such as exec-event and ACP spawn updates should stay throttled, while documented wake-now paths and task completion/follow-up delivery should remain immediate with regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: after any heartbeat run records lastRunStartedAtMs and advances nextDueMs, call shouldDeferWake with hook:wake, queued cron:<id> wake-now fallback, or coalesced background-task-blocked and it returns not-due instead of allowing the immediate wake.

## Solution Assessment

No. The helper and shared runner call sites are the right direction, but the current reason-only allowlist still loses explicit wake-now intent; a safer fix should carry an immediate/cooldown policy through requestHeartbeatNow or encode exact immediate reasons without reopening exec-event feedback loops.

- **[P2] Preserve explicit wake-now requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist only includes `manual`, `wake`, and `background-task`, so `hook:wake` and queued `cron:<id>` fallbacks from `wakeMode: "now"` still use the event-driven path. After a prior heartbeat, those requests are skipped as `not-due` while `nextDueMs` is in the future, making documented `/hooks/wake` mode now and cron busy-fallback wake-now requests wait for the scheduled heartbeat. Carry explicit immediate intent through the request or encode exact wake-now reasons separately from feedback-loop reasons.
  - confidence: 0.92
- **[P2] Keep blocked task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery queues `background-task` and then a `background-task-blocked` follow-up for the same session; the wake coalescer keeps the later same-priority reason. Because this allowlist only treats `background-task` as immediate, the actual queued blocked follow-up is deferred as `not-due` after a recent heartbeat, so users can wait until the next interval for a task that needs action. Include the blocked follow-up reason or pass explicit immediacy from task delivery.
  - confidence: 0.88
