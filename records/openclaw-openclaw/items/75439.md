## Summary

Keep open: this is a valid fix direction for the exec-event runaway, but the latest head still defers documented immediate hook/cron wake-now and blocked-task follow-up paths and still lacks docs for the new SDK subpath.

## What This Changes

The PR adds a centralized heartbeat wake cooldown helper and runner bookkeeping/tests, exposes a `tool-descriptors` plugin SDK subpath, updates the changelog, and adjusts several plugin/test fixtures.

## Best Possible Solution

Keep the shared cooldown helper, but make wake policy reason-specific: throttle loop-prone exec/spawn/retry wakes while preserving documented immediate hook, cron, task, and manual paths, with SDK docs aligned.

## Reproduction Assessment

Yes. Current main reproduces the original bug with repeated targeted `exec-event` wakes within `nextDueMs`; the remaining PR regressions are source-reproducible by sending `hook:wake`, cron `wakeMode: "now"` fallback, or `background-task-blocked` after a prior heartbeat while `now < nextDueMs`.

## Solution Assessment

No. The centralized helper is the right shape, but the present policy is still too broad and defers documented immediate delivery paths; the narrow fix is to make the immediate-vs-loop-prone policy explicit and test each wake family.
