## Summary

Keep open: current main still has the heartbeat interval-bypass bug and this PR is the right repair shape, but the latest PR head still defers explicit wake-now and blocked-task follow-up paths after a prior heartbeat, which would regress existing runtime/docs contracts.

## What This Changes

The PR adds a shared heartbeat cooldown helper, wires targeted and broadcast heartbeat wakes through it, exports heartbeat reason kinds, adds cooldown/scheduler regression tests, and records a changelog fix.

## Best Possible Solution

Keep the centralized cooldown gate, but carry explicit wake-now/task-immediate intent separately or encode exact immediate reasons while leaving exec-event, ACP spawn, and ordinary feedback-loop wakes throttled, with regression tests for each preserved path.

## Reproduction Assessment

Yes. A source-level reproduction is clear: after a previous heartbeat records lastRunStartedAtMs and nextDueMs is still in the future, the PR helper returns not-due for hook:wake, cron:<id>, and background-task-blocked even when those wakes represent immediate follow-up work.

## Solution Assessment

No. The shared helper is maintainable, but the current raw-reason allowlist is too narrow; the safer fix is to preserve explicit immediacy through the wake request or exact immediate classifications plus targeted tests.
