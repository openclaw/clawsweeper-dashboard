## Summary

Keep open: current main still has the heartbeat exec-event cooldown bug, and this PR targets the right scheduler surface, but the patch would delay documented immediate wake paths and can convert retryable busy wakes into non-retryable cooldown skips. It also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds a heartbeat cooldown helper with tests, routes targeted and broadcast heartbeat wakes through shared per-agent cooldown/flood bookkeeping, and adds a regression test for repeated scoped exec-event wakes.

## Best Possible Solution

Land a centralized heartbeat cooldown that throttles self-feeding exec-event/external wake loops while preserving explicit wake-now and task-completion delivery contracts, keeping retryable busy skips retryable, and documenting the shipped user-facing fix in the changelog.

## Reproduction Assessment

Yes. The original bug has a source-level and unit-test reproduction: repeated scoped exec-event wakes hit runOnce repeatedly on current main because the targeted branch lacks a cooldown gate; the PR body includes the failing test shape. The review findings are also reproducible from the diff: wake/background-task reasons after a prior run hit the new nextDueMs gate, and requests-in-flight after pre-run bookkeeping poisons the retry.

## Solution Assessment

No. The shared cooldown helper is the right seam, but this implementation applies interval cooldown too broadly and records cooldown state before a heartbeat attempt is known to be non-retryable. A safer fix should distinguish immediate wake contracts from feedback-style exec wakes and record run-start bookkeeping only for real or non-retryable attempts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve immediate wake-now paths:** `src/infra/heartbeat-cooldown.ts:142-143`
  - body: `shouldDeferWake` now returns `not-due` for every non-manual wake after an agent has run while `nextDueMs` is in the future. That includes documented immediate paths such as `openclaw system event --mode now` (`wake`), hook wake-now (`hook:wake`), and task completion wakes (`background-task`, classified as `other`), so users can wait until the next scheduled heartbeat instead of getting the promised prompt delivery. Exempt explicit immediate-notification reasons, or carry an explicit immediacy flag while still throttling self-feeding exec wakes.
  - confidence: 0.93
- **[P2] Do not seed cooldown state on busy skips:** `src/infra/heartbeat-runner.ts:1828`
  - body: The runner records `lastRunStartedAtMs` before calling `runOnce`. If a targeted wake returns `requests-in-flight`, heartbeat-wake retries the same reason, but the retry can now be turned into `not-due` or `min-spacing` because the previous attempt was counted as a run even though no heartbeat ran. Move this bookkeeping after a non-retryable attempt, or roll it back for retryable busy skips.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `CHANGELOG.md:14`
  - body: This is a user-facing `fix` for runaway heartbeat scheduling, but the PR does not add an Unreleased/Fixes entry. Add a single-line `CHANGELOG.md` entry with allowed contributor credit before merging.
