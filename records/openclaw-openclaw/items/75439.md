## Summary

Keep open: current main still has the exec-event heartbeat runaway this PR targets, but the latest head still defers documented wake-now hook/cron delivery and blocked-task follow-up wakes after a prior run.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, runner bookkeeping and regression tests, a changelog entry, a public plugin SDK tool-descriptors export, generated SDK baseline hashes, and test mock adjustments.

## Best Possible Solution

Keep the centralized cooldown and flood guard for loop-prone exec/acp/default wakes, while preserving explicit hook/cron wake-now delivery and blocked-task follow-up immediacy with focused regression tests.

## Reproduction Assessment

Yes. Current main reproduces the original bug with repeated targeted exec-event wakes inside nextDueMs, and the PR regressions reproduce by evaluating hook:wake, cron:<id>, or background-task-blocked after a prior run while now < nextDueMs.

## Solution Assessment

No. The helper extraction is the right maintainable shape, but the immediate-reason policy is too narrow and should preserve existing delivery contracts without reopening the exec-event feedback loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve wake-now hook and cron requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The allowlist only includes `manual`, `wake`, and `background-task`, while `hook:*` and `cron:*` fall into the deferrable event-driven path after any prior run. `/hooks/wake` defaults to `mode: "now"`, and cron `wakeMode: "now"` falls back to `requestHeartbeatNow({ reason: "cron:<id>" })` when the lane is busy, so these user-requested deliveries can now wait until the next scheduled heartbeat whenever `every` is long.
  - confidence: 0.94
- **[P2] Keep blocked task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task follow-ups are emitted as `reason: "background-task-blocked"`, but that exact reason is not in the immediate allowlist. After the owner agent has already run, the new gate returns `not-due` until `nextDueMs`, so detached-work follow-ups no longer satisfy the documented immediate heartbeat wake for task completion.
  - confidence: 0.92
- **[P3] Document the tool-descriptors SDK subpath:** `package.json:1229-1232`
  - body: This adds `openclaw/plugin-sdk/tool-descriptors` as a public package export, but the plugin SDK subpath docs still jump from `tool-payload` to `tool-send`. Public SDK subpath additions are supposed to keep the docs aligned, so third-party plugin authors will not see this supported import surface.
  - confidence: 0.86
