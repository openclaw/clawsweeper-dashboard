## Summary

Keep open: the central cooldown direction is valid, but the exact PR head still defers documented immediate wake paths after a prior heartbeat run.

## What This Changes

The PR adds a shared heartbeat cooldown helper, wires targeted and broadcast heartbeat wakes through it, adds scheduler/cooldown regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the shared cooldown gate, but carry explicit wake-now intent separately or encode exact immediate reasons for hook wake-now, cron wake-now fallback, and blocked task follow-ups while keeping exec-event and ACP spawn feedback throttled.

## Reproduction Assessment

Yes. A source-level reproduction is clear: after a prior heartbeat records lastRunStartedAtMs and nextDueMs is in the future, the PR helper returns not-due for hook:wake, cron:<id>, and background-task-blocked even though those paths are intended immediate follow-ups.

## Solution Assessment

No. The helper shape is maintainable, but the current reason-only allowlist is too narrow; the safer fix is to preserve explicit immediacy through requestHeartbeatNow or add exact immediate reasons with targeted regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve wake-now fallback immediacy:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist omits `hook:wake` and queued `cron:<id>` wake-now fallbacks. `/hooks/wake` mode `now` enqueues `hook:wake`, and main-session cron `wakeMode: "now"` falls back to `cron:<id>` when direct heartbeat is busy; after a prior heartbeat these now hit the event-driven `not-due` path and wait for the scheduled interval instead of waking now. Carry explicit wake-now intent through the request or encode these exact immediate paths separately from feedback-loop reasons.
  - confidence: 0.94
- **[P2] Keep blocked task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery queues `background-task` and then `background-task-blocked` for the same session, and the coalescer keeps the later same-priority reason. Since only `background-task` is allowlisted here, the actual blocked follow-up is deferred as `not-due` after a recent heartbeat, so users can wait until the next interval for a task that needs action.
  - confidence: 0.91
