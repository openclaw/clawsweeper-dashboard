## Summary

Keep open. The heartbeat runaway bug is real and this PR has the right central-helper shape, but the latest head still defers documented immediate hook/cron wake-now and blocked task follow-up paths, and the new SDK subpath is not fully aligned with generated API/docs contracts.

## What This Changes

The PR adds a shared heartbeat cooldown helper and runner bookkeeping/tests for non-interval wakes, updates the changelog, and adds a `plugin-sdk/tool-descriptors` export with related test fixture adjustments.

## Best Possible Solution

Land a narrow cooldown policy that throttles loop-prone exec/spawn/retry wakes while preserving documented immediate hook, cron, system wake, and task follow-up delivery paths; if `tool-descriptors` remains public, align its docs and generated API hash.

## Reproduction Assessment

Yes. Current main reproduces the original bug by sending repeated targeted `exec-event` wakes within `nextDueMs`; the remaining PR regressions are source-reproducible by sending hook/cron wake-now or `background-task-blocked` after any prior heartbeat run.

## Solution Assessment

No. The central helper is the right maintainable direction, but the latest policy is too broad for documented immediate delivery paths and the SDK export needs the usual public-surface follow-through.
