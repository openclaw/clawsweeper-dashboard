## Summary

Keep this PR open. The central cooldown gate is the right direction for the exec-event runaway, but the latest head still treats several documented immediate wake paths as ordinary event-driven wakes, so merging it would delay webhook wake-now, cron wake-now fallback, and blocked task follow-up delivery until the next heartbeat interval.

## What This Changes

The branch adds a new heartbeat cooldown helper, wires targeted and broadcast heartbeat dispatch through it, records recent run starts for flood protection, adds cooldown/scheduler regression tests, and adds a changelog entry for the exec-event runaway fix.

## Best Possible Solution

Keep the centralized cooldown gate for feedback-prone sources, but carry explicit immediate intent for hook wake-now, cron wake-now fallback, and blocked task follow-ups through the wake layer or encode those exact safe reasons with regression tests. Exec-event and ACP spawn updates should remain gated.

## Reproduction Assessment

Yes. With a prior heartbeat run recorded and nextDueMs still in the future, the PR helper returns not-due for hook:wake, cron:<id>, and background-task-blocked because none are in the exact immediate allowlist.

## Solution Assessment

No. The shared gate and flood guard are maintainable, but the current reason-only allowlist is too narrow; the safer repair is to preserve explicit wake-now intent without reopening exec-event feedback loops.
