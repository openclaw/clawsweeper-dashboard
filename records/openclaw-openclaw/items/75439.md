## Summary

Keep open: the central cooldown shape addresses the real `exec-event` runaway, but the current PR branch still over-gates documented immediate wake paths and adds a public SDK subpath without updating the docs catalog.

## What This Changes

The PR adds a heartbeat cooldown helper with runner bookkeeping and tests, updates the changelog, exposes `openclaw/plugin-sdk/tool-descriptors`, and adjusts several plugin/test fixtures.

## Best Possible Solution

Keep the shared cooldown gate, but make the wake policy explicit enough to throttle loop-prone reasons while preserving documented `now` and task follow-up delivery contracts, with matching regression tests and SDK docs.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for repeated targeted `exec-event` wakes bypassing `nextDueMs`; the PR regressions are reproducible by sending `cron:*`, `hook:*`, or `background-task-blocked` after a prior heartbeat while `now < nextDueMs`.

## Solution Assessment

No. The extracted helper is a maintainable direction, but the current policy is too broad; a safer repair is to gate only loop-prone wakes and explicitly allow documented immediate wake reasons.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve cron and hook now wakes:** `src/infra/heartbeat-cooldown.ts:176-177`
  - body: `cron:*` and `hook:*` now fall into the generic event-driven cooldown path after any prior run. Cron `wakeMode: "now"` falls back to `requestHeartbeatNow({ reason: `cron:${job.id}` })` when the direct heartbeat is busy, and `/hooks/wake` defaults to `mode: "now"`, so this can leave already-queued user events waiting until the heartbeat interval expires.
  - confidence: 0.94
- **[P2] Keep blocked task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: `queueBlockedTaskFollowup` wakes with `reason: "background-task-blocked"`, but the immediate allowlist excludes that exact reason. After a prior heartbeat run, blocked follow-up notifications will be deferred to `nextDueMs` instead of being delivered promptly through the task completion path.
  - confidence: 0.91
- **[P3] Document the tool-descriptors SDK subpath:** `package.json:1233-1235`
  - body: This adds `openclaw/plugin-sdk/tool-descriptors` as a public package export, but the SDK subpath catalog still skips from `tool-payload` to `tool-send`. Public SDK subpaths should be documented alongside the export map so plugin authors can discover the supported import.
  - confidence: 0.88
