## Summary

Keep this PR open. It addresses a real bug on current main, but the latest head still gates documented immediate wake paths and blocked-task follow-ups behind the new cooldown, so it needs a narrow policy/test repair before maintainer merge review.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, routes targeted and broadcast heartbeat wakes through it, adds heartbeat cooldown and scheduler regression tests, and updates the changelog for the runaway exec-event fix.

## Best Possible Solution

Keep the shared cooldown gate, but pass or encode explicit wake intent separately from broad reason classification. Feedback-loop sources such as exec-event and ACP spawn updates should be throttled, while documented wake-now requests and task blocked follow-ups remain immediate and covered by targeted regression tests.

## Reproduction Assessment

Yes. A source-level repro is clear: after any heartbeat run records a recent start and nextDueMs is in the future, call the PR helper with hook:wake, queued cron:<id> wake-now fallback, or background-task-blocked and it returns not-due instead of allowing the immediate wake.

## Solution Assessment

No. The central helper and runner wiring are the right shape, but the latest reason-only allowlist is still too narrow; the safer fix is to carry explicit immediacy through requestHeartbeatNow or add exact immediate reasons with tests without reopening exec-event feedback loops.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve wake-now fallback immediacy:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist omits `hook:*` and queued `cron:*` wake-now fallbacks. `/hooks/wake` queues `hook:wake`, and main-session cron `wakeMode: "now"` falls back to `cron:<id>` when the direct heartbeat is busy; after a prior heartbeat these now hit the event-driven `not-due` path and wait for the scheduled interval instead of waking now. Carry explicit wake-now intent through the request or encode these exact immediate paths separately from feedback-loop reasons.
  - confidence: 0.93
- **[P2] Keep blocked task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery queues `background-task` and then `background-task-blocked` for the same session, and the coalescer keeps the later same-priority reason. Since only `background-task` is allowlisted here, the actual blocked follow-up is deferred as `not-due` after a recent heartbeat, so users can wait until the next interval for a task that needs action.
  - confidence: 0.9
