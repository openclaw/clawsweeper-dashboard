## Summary

Keep open: the PR targets a real current-main heartbeat cooldown gap, but the latest diff still breaks documented immediate wake paths for cron/hooks and blocked task follow-ups, with one SDK docs alignment issue remaining.

## What This Changes

The PR adds a centralized heartbeat wake cooldown/flood guard with runner bookkeeping and regression tests, plus a changelog entry, a public `plugin-sdk/tool-descriptors` subpath, and related test fixture updates.

## Best Possible Solution

Keep the centralized cooldown/flood guard, but narrow the policy so loop-prone wakes are gated while documented `now` and follow-up delivery reasons stay immediate, and keep any new SDK subpath documented with the export map.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for repeated targeted `exec-event` wakes bypassing `nextDueMs`; the remaining PR regressions are reproducible by issuing `cron:*`, `hook:*`, or `background-task-blocked` after a prior heartbeat while `now < nextDueMs`.

## Solution Assessment

No. The helper extraction is the right shape, but the current reason policy is too broad; the safer fix is an explicit immediate policy for cron/hook/task follow-up reasons plus focused regression tests and SDK docs alignment.
