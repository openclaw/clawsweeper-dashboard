## Summary

Keep open: the centralized cooldown shape addresses the exec-event runaway, but the latest PR head still delays documented wake-now and blocked-task follow-up paths, so it needs two narrow policy/test repairs before merge.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, runner bookkeeping, regression tests, and changelog text to gate loop-prone non-interval heartbeat wakes.

## Best Possible Solution

Keep the shared cooldown gate for loop-prone feedback reasons, but add explicit immediate semantics or metadata for hook and cron wake-now plus blocked-task follow-ups without making cron next-heartbeat immediate.

## Reproduction Assessment

Yes. Current main's exec-event runaway is reproducible with the scheduler test described in the PR, and the PR-head regressions are reproducible by calling shouldDeferWake after a prior run with hook:wake, a cron wake-now fallback reason, or background-task-blocked; the targeted runner returns before runOnce.

## Solution Assessment

No. The helper and runner seam are the right shape, and retryable busy bookkeeping looks fixed, but the policy is still too broad for existing wake-now and task contracts; narrow reason metadata and regression tests are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve hook and cron wake-now requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist still excludes hook:* and the cron wake-now path. Current /hooks/wake mode now enqueues hook:wake, hook agent wakeMode now enqueues hook:<job>, and main-session cron --wake now can either call runHeartbeatOnce directly or fall back to queued cron:<job>. After any prior heartbeat, this gate returns not-due before runOnce, so those user-requested wake-now events wait for the scheduled tick or, for direct cron wake-now, can mark the cron run skipped as not-due. Keep next-heartbeat cron behavior gated by distinguishing the wake-now path instead of treating all cron work the same.
  - confidence: 0.93
- **[P2] Keep blocked-task follow-ups immediate:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery uses a separate reason, background-task-blocked, for the follow-up system event after the terminal blocked message. This exact reason is not in the immediate set, so once an agent has run, the PR defers the "Task needs follow-up" wake behind nextDueMs even though the task notification path is documented and tested as a push-driven wake. Add coverage for background-task-blocked alongside background-task.
  - confidence: 0.9
