## Summary

Keep open: the central cooldown direction addresses the current exec-event runaway, but the PR-head reason policy still defers documented wake-now webhook/cron paths and blocked-task follow-up notifications after any prior heartbeat run.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, runner bookkeeping, regression tests, a changelog entry, generated SDK baseline updates, and a plugin SDK tool-descriptors export.

## Best Possible Solution

Keep the shared cooldown and flood guard for loop-prone exec/acp/default wakes, but preserve explicit wake-now webhook/cron reasons and blocked-task follow-up delivery with focused regression tests.

## Reproduction Assessment

Yes. The original bug is reproduced by repeated targeted `exec-event` wakes within `nextDueMs`; the PR regressions reproduce by calling `shouldDeferWake` after a prior run with `hook:wake`, `cron:<id>`, or `background-task-blocked` while `now < nextDueMs`.

## Solution Assessment

No. The shared helper is the right maintainable shape, but the current immediate-reason policy is too narrow and should preserve existing prompt-delivery contracts without reopening the exec-event feedback loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve wake-now hook and cron requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist only admits `manual`, `wake`, and `background-task`, so `hook:*` and `cron:*` fall through to the event-driven `now < nextDueMs` deferral after any previous run. `/hooks/wake` defaults to `mode: "now"`, and main-session cron uses `--wake now` plus a queued `cron:<id>` fallback when busy, so these user-requested wakes can wait until the next scheduled heartbeat whenever `every` is long.
  - confidence: 0.94
- **[P2] Preserve blocked task follow-up wakes:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The task registry emits `reason: "background-task-blocked"` for blocked follow-up messages, but this exact reason is not in the allowlist. After the owner agent has run once, the new gate returns `not-due` while `nextDueMs` is in the future, so detached-work follow-ups no longer get the documented immediate heartbeat wake.
  - confidence: 0.92
