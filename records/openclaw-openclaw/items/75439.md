## Summary

Keep open: the PR addresses a real current-main heartbeat runaway path, but the latest head still blocks documented cron and hook wake-now behavior by treating `cron:*` and `hook:*` as deferred event-driven reasons after any prior heartbeat run.

## What This Changes

The branch adds a centralized heartbeat cooldown helper, routes targeted and broadcast heartbeat wakes through it with per-agent run/flood bookkeeping, adds scheduler/cooldown regression tests, and updates the changelog.

## Best Possible Solution

Land the centralized cooldown only after it separates feedback-loop wakes from explicit wake-now intent, so `exec-event`, ACP spawn updates, retries, and non-immediate queued wakes are throttled while cron `wakeMode: "now"`, `/hooks/wake` mode now, and other explicit operator/system wake-now requests still reach `runOnce` immediately with regression coverage.

## Reproduction Assessment

Yes. The original bug has a source-level reproduction on current main: repeated session-scoped `exec-event` wakes reach `runOnce` within `nextDueMs`. The remaining PR regression can be reproduced by recording a prior heartbeat run, then issuing `hook:wake` or main-session cron `wakeMode: "now"` before `nextDueMs` and observing the PR helper return `not-due`.

## Solution Assessment

No. The shared cooldown helper is the right seam, but the current reason-only policy is too coarse because it cannot distinguish feedback events from explicit wake-now requests. A safer solution should carry an explicit immediacy/cooldown policy through the wake request or use exact wake-now reasons while keeping self-feeding `exec-event`/ACP paths gated.
