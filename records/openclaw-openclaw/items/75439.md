## Summary

Keep open: the cooldown helper is the right direction for the exec-event runaway, but the latest head still defers existing immediate hook, cron, and blocked-task follow-up wake paths after an agent has already run.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, runner bookkeeping, regression tests, a heartbeat reason type export, and a changelog entry for non-interval wake rate-limiting.

## Best Possible Solution

Keep the shared cooldown and flood guard, but carry explicit immediate-wake intent for hook wake-now, cron wake-now fallbacks, and blocked task follow-ups while continuing to gate loop-prone exec/acp wakes.

## Reproduction Assessment

Yes. Current main's exec-event runaway is covered by the PR's scheduler regression, and the PR-head regressions reproduce by evaluating shouldDeferWake after a prior run with hook:wake, cron:<job>, or background-task-blocked while now < nextDueMs.

## Solution Assessment

No. The centralized helper is the narrow maintainable shape, but the current immediate-reason policy is incomplete and delays documented push-driven wake paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve hook and cron wake-now requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist still excludes `hook:*` and `cron:*`. After any prior run, those reasons fall through to the event-driven path and defer while `now < nextDueMs`, delaying `/hooks/wake` mode `now`, hook-agent wakeMode `now` announcements/errors, and cron wake-now fallback delivery until the next scheduled heartbeat. Carry explicit immediate intent for these paths while keeping loop-prone exec/spawn wakes gated.
  - confidence: 0.93
- **[P2] Preserve blocked task follow-up wakes:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery queues a second system event with reason `background-task-blocked`, but only `background-task` is immediate here. Once an agent has run, that follow-up is treated as an event-driven wake and deferred behind `nextDueMs`, so the actionable blocked-task message can wait for the next scheduled heartbeat instead of following the push-driven task completion contract.
  - confidence: 0.91
