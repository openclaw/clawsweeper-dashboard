## Summary

Keep open. The current-main `exec-event` runaway is real and the centralized cooldown direction is useful, but latest PR head still delays documented wake-now and user-visible follow-up paths after a recent heartbeat.

## What This Changes

The PR adds a shared heartbeat cooldown helper, wires targeted and broadcast heartbeat dispatch through it, adds regression tests and changelog text, and includes small qa-lab/typecheck cleanup changes.

## Best Possible Solution

Keep the shared cooldown gate, but preserve explicit immediate intent for hook wake-now, cron wake-now fallback, and blocked-task follow-ups while leaving exec-event, ACP spawn, retry, and unknown feedback-loop reasons throttled.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `shouldDeferWake({ reason: "hook:wake" | "cron:<id>" | "background-task-blocked", lastRunStartedAtMs, now < nextDueMs })`, which returns `not-due` on the PR even though current source uses those reasons for immediate delivery paths.

## Solution Assessment

No. The centralized helper is the right maintainable shape, but the latest reason policy is too broad; the safer fix is an explicit immediate-intent policy plus regression tests for each preserved source path.

## Review Findings

Overall correctness: patch is incorrect
