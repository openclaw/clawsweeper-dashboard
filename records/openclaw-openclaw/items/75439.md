## Summary

Keep open: the PR targets a real current-main heartbeat runaway, but the latest head still gates documented immediate hook, cron, and blocked-task follow-up wakes, and it adds a public SDK subpath without the matching docs update.

## What This Changes

The PR adds a centralized heartbeat wake cooldown/flood guard with runner bookkeeping and regression coverage, plus a changelog entry, a public `tool-descriptors` SDK export, generated SDK baselines, and related test mock updates.

## Best Possible Solution

Keep the centralized cooldown seam for loop-prone exec/acp/default wakes, while preserving explicit user/requester-driven now paths and documenting any SDK subpath that remains in the branch.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for repeated targeted `exec-event` wakes bypassing `nextDueMs`, and the PR regressions are reproducible by evaluating `hook:wake`, `cron:<id>`, or `background-task-blocked` after a prior run while `now < nextDueMs`.

## Solution Assessment

No. The helper extraction and flood guard are the right shape, but the immediate-reason policy is too narrow; the safer fix is to keep documented now/follow-up reasons immediate while continuing to gate loop-prone exec/acp/default event wakes.
