## Summary

This PR should stay open: the centralized cooldown fixes the exec-event runaway shape, but the current wake policy still defers documented immediate cron, hook, and blocked-task follow-up paths after a prior heartbeat run.

## What This Changes

The branch adds a centralized heartbeat wake cooldown/flood guard with runner bookkeeping and regression tests, plus a changelog entry, a public `plugin-sdk/tool-descriptors` export, and related test mock updates.

## Best Possible Solution

Keep the centralized cooldown seam, but gate only loop-prone wake sources while preserving documented now/follow-up reasons and documenting any public SDK subpath that remains in the branch.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for repeated targeted `exec-event` wakes bypassing `nextDueMs`, and the PR regressions are reproducible by evaluating `cron:*`, `hook:*`, or `background-task-blocked` after a prior run while `now < nextDueMs`.

## Solution Assessment

No. The helper extraction and flood guard are the right direction, but the immediate-reason policy is too narrow; the safer fix is to keep explicit requester-driven now paths immediate while still deferring `exec-event`, ACP spawn updates, and unknown loop-prone events.
