## Summary

Keep open. The PR fixes a real current-main heartbeat cooldown gap, but the latest head still regresses existing immediate wake contracts for hook/cron wake-now paths and blocked task follow-ups, so it should not merge as-is.

## What This Changes

The PR adds a centralized heartbeat cooldown helper, runner bookkeeping, regression tests, a heartbeat reason type export, and a changelog entry for non-interval wake rate-limiting.

## Best Possible Solution

Keep the shared cooldown and flood guard, but carry explicit immediate-wake intent for hook wake-now, cron wake-now fallbacks, and blocked task follow-ups while keeping exec/acp loop-prone wakes gated.

## Reproduction Assessment

Yes. Current main's exec-event runaway is covered by the PR's scheduler regression test, and the PR-head regressions are reproducible by evaluating shouldDeferWake after a prior run with hook:wake, cron:<job>, or background-task-blocked.

## Solution Assessment

No. The centralized helper is the right shape, but the current allowlist conflates loop-prone events with existing wake-now contracts; exact immediate metadata or narrower reason handling is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve hook and cron wake-now requests:** `src/infra/heartbeat-cooldown.ts:70`
  - body: The immediate allowlist excludes `hook:*` and `cron:*`. After any prior heartbeat, those reasons fall through to `shouldDeferWake`, return a cooldown deferral while `now < nextDueMs`, and the runner skips before `runOnce`; that delays `/hooks/wake` mode `now`, hook-agent wakeMode `now` announcements/errors, and cron wake-now fallback delivery until the next scheduled heartbeat. Preserve explicit wake-now intent while keeping cron next-heartbeat and loop-prone exec/spawn wakes gated.
  - confidence: 0.93
- **[P2] Let blocked task follow-ups wake the session:** `src/infra/heartbeat-cooldown.ts:70`
  - body: Blocked task delivery queues a second system event with reason `background-task-blocked`, but only `background-task` is allowlisted here. Once the agent has run, that follow-up is treated as an event-driven wake and deferred behind `nextDueMs`, so the actionable blocked-task message can sit until the next scheduled heartbeat despite the push-driven task completion contract.
  - confidence: 0.9
