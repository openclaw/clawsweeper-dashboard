## Summary

Keep open: current main still has the forced-plugin harness support-check gap, so the PR targets a real bug, but the proposed patch is not merge-ready because it makes Codex claim OpenAI-family providers in auto mode and its PI fallback path conflicts with the current fail-closed runtime contract.

## What This Changes

The PR adds forced/session-pinned harness support checks, unsupported-candidate PI fallback or error behavior, broader Codex harness provider ids, focused tests, and a changelog entry.

## Best Possible Solution

Land the forced/session-pinned support check after rebasing onto current main, with explicit runtimes failing closed for unsupported providers and Codex `supports()` accepting OpenAI-family providers only when Codex runtime is explicitly requested.

## Reproduction Assessment

Yes. Current main source shows the forced-plugin branch returns the pinned harness without calling `supports()`, and the PR regression is statically reproducible because the widened provider set is checked without considering `requestedRuntime`.

## Solution Assessment

No as written. The selector-side support check is the right narrow fix, but the Codex provider widening needs runtime gating and the unsupported forced-runtime path should follow current main’s fail-closed contract.

## Review Findings

After rebasing, keep the support check but throw a clear unsupported-harness error instead of reintroducing PI fallback for explicit or session-pinned runtimes.
  - confidence: 0.86
