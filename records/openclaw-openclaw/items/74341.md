## Summary

Keep open. The selector-side guard addresses a real current-main bug, but the PR still widens Codex harness support in a way that lets auto mode claim `openai`/`openai-codex` routes, contradicting current docs and an existing regression test.

## What This Changes

The PR adds `supports()` validation for forced/session-pinned plugin harness selection, PI fallback/error behavior for unsupported candidates, broader Codex harness provider ids, focused tests, a status-test helper update, and a changelog entry.

## Best Possible Solution

Land the selector-side support check, but keep Codex auto mode conservative: allow `openai` and `openai-codex` only for an explicit Codex runtime selection, while preserving `codex/*` legacy auto support and adding tests for both paths.

## Reproduction Assessment

Yes. Current main reproduces the reported bug by inspection of the forced branch, and the PR regression is statically reproducible because the widened provider set makes `supports({ provider: "openai-codex", requestedRuntime: "auto" })` true while the existing test and docs require false.

## Solution Assessment

No as written. The `supports()` check in forced selection is the right narrow fix, but broadening the Codex default provider set without consulting `requestedRuntime` changes auto-mode routing outside the stated bug.
