## Summary

Keep open: the forced-harness `supports()` guard is the right bug-fix direction, but the current branch introduces a compatibility blocker by letting the Codex harness claim `openai` during auto-mode selection, and after-fix real gateway proof is still missing.

## What This Changes

The PR adds a forced-plugin harness support check, expands the Codex harness provider ids to include `codex`, `openai-codex`, and `openai`, updates harness/status tests, and adds a changelog entry.

## Best Possible Solution

Keep the forced `supports()` guard, but make the Codex harness claim `openai` and `openai-codex` only for forced/requested Codex runtime contexts, preserve auto-mode proxy compatibility, update the existing Codex harness contract test, and add redacted patched-gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main's forced-plugin branch bypasses `supports()`, and the PR thread includes production logs of unsupported providers reaching Codex; I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently written. The guard belongs in selection, but the Codex provider-id expansion must be context-aware so auto mode does not capture custom OpenAI-compatible providers.
