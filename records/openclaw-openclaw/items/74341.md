## Summary

Keep open: the forced-harness support check is a useful fix and the current-main bug is source-reproducible, but the branch still broadens Codex `supports()` unconditionally so auto mode can capture custom OpenAI-compatible providers, and the after-fix real behavior proof is still test-only.

## What This Changes

This PR adds a forced plugin-harness `supports()` check before selecting a pinned harness, broadens the Codex harness default provider ids, adds harness selection/Codex support tests, updates a status-test helper, and adds a changelog entry.

## Best Possible Solution

Keep the forced `supports()` guard, make Codex claim `openai` and `openai-codex` only for forced/requested Codex runtime contexts, preserve auto-mode proxy behavior, update both new and existing contract tests, and add redacted patched-gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main's forced-plugin branch bypasses `supports()`, and the PR discussion includes production logs plus a focused probe demonstrating unsupported providers reaching Codex before this patch.

## Solution Assessment

No. The forced `supports()` check is the right fix direction, but the current implementation also broadens Codex provider support in auto mode and should instead gate `openai`/`openai-codex` support to requested Codex runtime contexts.
