## Summary

Keep this PR open: it targets a real forced-harness selection bug, but the current diff also makes the Codex harness claim `openai` and `openai-codex` in auto mode, which conflicts with current docs and an existing regression test boundary.

## What This Changes

The PR adds forced/session-pinned agent harness `supports()` checks, PI fallback/error behavior for unsupported candidates, broader Codex harness provider ids, targeted tests, and a changelog entry.

## Best Possible Solution

Land the selector-side support check with a runtime-aware Codex support predicate, plus regression tests that keep `openai` and `openai-codex` out of auto mode while allowing them for explicit Codex runtime.

## Reproduction Assessment

Yes. The original bug reproduces by inspecting current main's forced branch, and the PR regression is statically reproducible because PR head makes `supports({ provider: "openai-codex", requestedRuntime: "auto" })` return supported despite the existing auto-mode contract.

## Solution Assessment

No as written. The forced-branch support check is the right narrow fix, but the Codex provider widening needs to be gated by explicit Codex runtime rather than changing auto-mode routing.
