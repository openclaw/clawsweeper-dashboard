## Summary

Keep this PR open: the underlying forced-harness `supports()` bypass is still present on current `main`, and the PR is directionally useful, but it needs to be adapted to the current fail-closed provider/model runtime contract and still needs after-fix real behavior proof.

## What This Changes

The PR gates forced plugin harness selection on `supports()`, expands the Codex harness provider set, adds targeted tests, and adds a changelog entry.

## Best Possible Solution

Keep the `supports()` gate and Codex provider-id alignment, but rebase it onto current main so explicit provider/model plugin runtimes fail closed and only auto/no-claim selection can choose PI.

## Reproduction Assessment

Yes, by source inspection and discussion. Current main’s forced plugin branch skips `supports()`, and PR comments include production traces plus a focused failing probe; I did not run tests in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No as written. The support check and Codex provider-id expansion are the right fix direction, but the unsupported explicit-runtime path should throw under the current fail-closed contract instead of routing to PI.
