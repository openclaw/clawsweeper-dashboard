## Summary

Keep open. The PR addresses a real current-main selector bug, but it still introduces a blocking contract regression by making the Codex harness claim `openai` and `openai-codex` in auto mode, contrary to current docs and tests.

## What This Changes

The PR adds forced/session-pinned harness `supports()` checks, PI fallback/error handling, broader Codex harness provider ids, targeted tests, and a changelog entry.

## Best Possible Solution

Land the selector-side support check, but keep Codex auto mode conservative: accept `openai` and `openai-codex` only when `requestedRuntime` is explicitly `codex`, and add regression assertions for auto mode before merging.

## Reproduction Assessment

Yes. Current main reproduces the reported bug by inspection of the forced branch that returns a plugin harness without `supports()`, and the PR regression is statically reproducible because the widened provider set makes `supports({ provider: "openai-codex", requestedRuntime: "auto" })` return supported.

## Solution Assessment

No as written. The forced-branch `supports()` check is the right narrow fix, but broadening the Codex default provider set without checking `requestedRuntime` changes auto-mode routing outside the stated bug.
