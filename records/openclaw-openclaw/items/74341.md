## Summary

Keep this PR open. The forced-harness bug is real and the selector-side support check is the right direction, but the PR head still widens Codex provider support in a way that makes auto mode claim `openai`/`openai-codex` routes, contradicting current docs and an existing test boundary.

## What This Changes

The PR gates forced/session-pinned plugin harness selection with `supports()`, adds PI fallback or clear errors for unsupported candidates, widens Codex harness provider ids, updates tests/status helpers, and adds a changelog entry.

## Best Possible Solution

Land the forced-harness support check with a runtime-aware Codex support predicate and regression tests that keep `openai` and `openai-codex` out of auto mode while allowing them for explicit Codex runtime.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main bypasses `supports()` in the forced branch, and the PR head makes `supports({ provider: "openai-codex", requestedRuntime: "auto" })` supported because the support predicate ignores `requestedRuntime`.

## Solution Assessment

No as written. The selector change is the narrow maintainable fix, but the Codex provider widening must be gated on explicit `requestedRuntime: "codex"` rather than changing auto-mode routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep Codex auto mode from claiming OpenAI routes:** `extensions/codex/harness.ts:8`
  - body: Changing the default provider set to include `openai` and `openai-codex` makes the Codex harness report support for those providers even when `requestedRuntime` is `auto`, because `supports()` only checks provider membership. Auto selection then routes configs that docs and existing tests keep on PI into the Codex app-server harness, so the provider widening needs to be runtime-aware.
  - confidence: 0.93

## Security Review

Status: needs_attention
