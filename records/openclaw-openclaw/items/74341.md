## Summary

## What This Changes

## Best Possible Solution

Keep the forced `supports()` guard, make Codex claim `openai` and `openai-codex` only in an explicit Codex runtime context, preserve custom OpenAI-compatible PI routing, update focused tests, and add redacted patched-gateway proof.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.5.22` shows forced plugin selection bypasses `supports()`, and the PR discussion includes production logs plus focused probes for unsupported providers reaching Codex.

## Solution Assessment

No, not as currently patched. The forced-branch support guard is the right direction, but `openai` and `openai-codex` support needs to be gated to explicit Codex runtime contexts so custom OpenAI-compatible configs are not captured by Codex auto-selection.
