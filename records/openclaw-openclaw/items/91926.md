## Label Justifications

- `P1`: A built-in agent restart workflow is reported to fail consistently on Windows and leaves the gateway unavailable until manual recovery.
- `impact:crash-loop`: The restart operation terminates the gateway but fails to establish a replacement process, causing process-level unavailability.

## Summary

Current main and v2026.6.5 still use the synchronous Windows scheduled-task stop-then-run sequence that matches the reporter’s detailed event-log failure, while a sibling restart path already has a detached handoff pattern that can guide a narrow fix.

## What This Changes

Fix Windows gateway restart from an agent exec session when Task Scheduler stops the task but silently fails to launch it again.

## Best Possible Solution

Move Windows CLI restart onto one lifecycle-owned detached relaunch mechanism that runs only after shutdown cleanup, verifies or observes task startup, and preserves the existing generated-script fallback without duplicating restart policy.

## Reproduction Assessment

Yes, at high confidence from source and supplied evidence: current main contains the exact synchronous stop-then-run path, and the report provides nine repeated attempts plus Task Scheduler event sequences. A fresh Windows Session 0 run was not executed in this Linux checkout.

## Solution Assessment

Yes, if implemented by consolidating with the existing detached Windows task-handoff strategy; adding another fixed delay, event-log-specific workaround, or parallel fallback path would be less maintainable.
