## Summary

Keep open. Current main still normalizes provider ids in the core contract, but the Usage/Control UI path still aggregates, filters, suggests, exports, and renders raw provider strings, so supported aliases such as `z-ai`, `z.ai`, and `aws-bedrock` can still display or group separately from canonical providers. The discussion usefully links related picker issues #47620 and #47621, but those closed issues do not resolve the remaining Usage surface.

## What This Changes

The issue asks for Control UI model-provider displays to use canonical normalized provider names instead of raw configured aliases.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for aggregate grouping, display labels, provider filters, query suggestions/query matching, and CSV export while preserving raw stored session/config fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a usage entry with provider `z-ai`, `z.ai`, or `aws-bedrock` flows into `sessions.usage` and the Usage UI with raw provider strings used for aggregate keys, filters, suggestions, badges, labels, and CSV export.

## Solution Assessment

Yes for the narrow repair direction. The maintainable fix is to call the existing canonical provider-id helper at Usage boundaries, not to add a separate UI alias table or silently rewrite stored session/config values.
