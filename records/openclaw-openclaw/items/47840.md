## Summary

Keep open: current main still exposes raw or lowercase-only provider values in the Control UI usage/session surfaces, and the only linked repair PR is closed unmerged. The remaining work needs a maintainer decision on the provider display contract before another fix lane.

## What This Changes

The issue asks for Control UI provider displays to show canonical normalized provider identifiers instead of raw config aliases such as `bailian` or `z-ai`.

## Best Possible Solution

Canonicalize effective provider identity at the Usage/session display and aggregate boundary using the approved provider alias contract, while preserving literal config keys only where users are editing configuration.

## Reproduction Assessment

Yes at source level: current main still lowercases or renders provider fields directly in Sessions, Usage filters, badges, CSV/source aggregate inputs, and aggregate keys. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No for the issue's per-render sketch: the maintainable fix is a single approved provider-display/aggregate boundary, not scattered calls in every render site. The exact UX still needs a maintainer choice between canonical-only labels and canonical labels with alias context.
