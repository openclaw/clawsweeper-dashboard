## Summary

Keep open: current main still exposes raw or lowercase-only provider values in Control UI session and usage paths, and the only linked repair PR closed unmerged. The remaining work needs a maintainer display-contract decision before another fix PR.

## What This Changes

The issue requests consistent normalized provider display in Control UI sessions and usage surfaces instead of showing raw configured provider aliases.

## Best Possible Solution

Canonicalize effective provider identity at the Usage/session API or aggregation boundary using the approved provider-id contract, then preserve or annotate literal config keys only in configuration-editing surfaces.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main still lowercases or renders provider fields directly in Sessions, Usage filters, badges, CSV, and aggregate keys; no browser run was performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No for the issue's per-render sketch: the safer fix is boundary canonicalization for derived runtime/usage identity plus an explicit UI rule for raw config keys.
