## Summary

Keep open. Current main still has a canonical provider normalizer, but the Usage/Control UI paths continue to use raw provider strings for aggregates, filters, suggestions, query matching, exports, and visible badges, so documented aliases such as `z.ai`/`z-ai` and `aws-bedrock` can still display or group separately from their canonical providers.

## What This Changes

Request canonical provider display in the Control UI instead of showing raw config/session provider aliases.

## Best Possible Solution

Reuse `normalizeProviderId` for Usage-derived display labels, aggregate keys, filters, suggestions, query matching, and export fields while preserving raw stored session/config values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Usage entry or session field with provider `z-ai`, `z.ai`, or `aws-bedrock` flows through the Usage API and Control UI with raw provider strings used for grouping, filtering, suggestions, matching, exports, and badges.

## Solution Assessment

Yes. The narrow maintainable fix is to apply the existing canonical provider-id helper at derived Usage API/UI boundaries, not to add a separate UI alias table or rewrite persisted raw session/config records.
