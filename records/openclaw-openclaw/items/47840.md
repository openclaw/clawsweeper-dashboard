## Summary

Current main still preserves raw provider aliases across Usage aggregation, filtering, export, and visible badges, and no merged or viable open PR owns the fix. The issue remains necessary, but the replacement should use plugin-owned alias metadata and needs a maintainer decision on canonical-only versus canonical-plus-configured-alias presentation.

## What This Changes

Make Control UI provider displays and related Usage grouping consistently use canonical provider identity instead of raw configured aliases.

## Best Possible Solution

Expose plugin-resolved canonical provider identity in prepared usage metadata, collapse aggregation and query behavior on it, and retain the configured alias only as optional secondary UI context if that presentation is approved.

## Reproduction Assessment

Yes at source level: current main keys Usage aggregates with raw provider strings and renders raw provider badges, while UI matching only lowercases rather than resolving aliases. A live browser reproduction was not performed during this read-only review.

## Solution Assessment

No, applying a global normalizer independently at render sites is not the best solution; canonical provider identity should be resolved through plugin-owned metadata before aggregation, filtering, export, and presentation.
