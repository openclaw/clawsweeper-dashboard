## Summary

Current main and v2026.6.5 still expose raw provider identifiers across Usage badges, CSV, filters, and aggregate inputs; the linked implementation closed unmerged for inactivity, so the issue remains valid, but the repair should follow current plugin-owned alias metadata rather than revive its older global-normalizer approach unchanged.

## What This Changes

Make Control UI provider displays and related Usage query/export/aggregation paths consistently use canonical provider identifiers instead of configured aliases.

## Best Possible Solution

Resolve provider aliases through the lifecycle-owned provider registry, carry the canonical id into Usage session and aggregate records, and render/query/export that prepared value while optionally retaining the configured alias only as secondary explanatory UI.

## Reproduction Assessment

Yes, at source level. Put a confirmed provider alias such as `z-ai` in a Usage session's provider field and current shipped code carries it unchanged into badges, CSV, filters, and aggregate inputs.

## Solution Assessment

No, the proposed per-render normalization is not the best fix. Canonical provider identity should be prepared once from provider-owned alias metadata before Usage aggregation and presentation, avoiding duplicated UI policy and inconsistent keys.
