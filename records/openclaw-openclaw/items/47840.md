## Summary

Keep open: current main and the latest release still expose raw or lowercase-only provider values in Control UI Usage/session display and aggregate paths, while the only linked implementation PR is closed unmerged. The remaining decision is the provider display contract, especially whether canonical labels should replace or annotate configured aliases.

## What This Changes

The issue requests consistent Control UI display of normalized model provider identifiers, such as showing canonical provider names instead of configured aliases like `bailian`, `z-ai`, or `aws-bedrock`.

## Best Possible Solution

Define the approved provider display and aggregate boundary for Usage/session surfaces, then land a focused fix that canonicalizes derived UI/query/export/aggregate values while preserving raw stored config/session fields.

## Reproduction Assessment

Yes. Source inspection shows current main still builds Usage provider options, query matches, CSV rows, badges, overview metadata, and usage aggregate keys from raw or lowercase-only provider strings, so alias variants can remain visible or split without a browser run.

## Solution Assessment

No for the issue's scattered per-render fix sketch. The maintainable solution is one approved provider identity/display contract at the Usage/session aggregate boundary, because server aggregates and exports can already be split before render code sees them.
