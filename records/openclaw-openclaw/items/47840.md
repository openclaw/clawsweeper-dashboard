## Summary

Current main still exposes unnormalized provider identities in Control UI session and usage paths. The issue should remain open because the earlier repair PR closed unmerged, and maintainers still need to decide how runtime-facing canonical identities should differ from literal configuration keys.

## What This Changes

Make Control UI provider labels consistently display canonical normalized provider IDs rather than raw configured aliases.

## Best Possible Solution

Canonicalize effective provider identity once at the usage/session aggregation or API boundary using the shared provider-ID contract, then use that value for filtering, exports, aggregates, and runtime-facing labels while deliberately preserving or annotating literal keys in configuration editors.

## Reproduction Assessment

Yes at source level: current-main Sessions renders `modelProvider` directly, and Usage filtering and presentation use raw or lowercase-only provider values rather than the shared canonical normalizer. A browser run was not performed during this read-only review.

## Solution Assessment

No, the proposed per-render normalization is not the best solution because provider identity also participates in aggregate keys, filters, suggestions, and exports. The safer boundary is shared normalized data for effective runtime and usage identity, plus an explicit UX rule for configuration-facing aliases.
