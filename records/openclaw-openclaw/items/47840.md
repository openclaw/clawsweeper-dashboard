## Summary

Keep open. Current main still renders raw provider fields in Control UI Usage/Sessions display, query suggestions, CSV export, and aggregate keys, while current provider-id normalization policy is not a simple “always alias to canonical” contract; maintainers should decide the supported display semantics before another fix PR is queued.

## What This Changes

The issue asks for Control UI provider displays to show normalized provider ids such as `zai` instead of raw config/provider aliases.

## Best Possible Solution

Define a supported provider display canonicalization contract, then apply it only to derived Control UI display, query, CSV, and Usage aggregate surfaces while preserving raw stored session/config fields.

## Reproduction Assessment

Yes. Source inspection on current main shows the affected Control UI Usage/Sessions paths rendering raw `modelProvider` or `providerOverride` values, while a narrower UI thinking helper already canonicalizes some provider aliases.

## Solution Assessment

Unclear. Normalizing supported aliases at derived display/query/aggregate boundaries is likely the right shape, but current main intentionally keeps the shared provider normalizer lowercase-only and the `bailian` example needs separate provider-direction approval.
