## Summary

Keep open. Current main still has a canonical provider normalizer, but the Usage/Control UI path continues to aggregate, filter, suggest, query, and render raw provider strings, so documented aliases such as `z.ai`/`z-ai` and `aws-bedrock` can still appear or split from their canonical providers.

## What This Changes

The issue requests consistent Control UI display of normalized model provider IDs instead of raw alias values from configuration or usage data.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for aggregate grouping, display labels, provider filters, query suggestions/query matching, and CSV/export-style output while preserving raw stored session/config fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Usage entry or session field with provider `z-ai`, `z.ai`, or `aws-bedrock` flows through `sessions.usage` and the Usage UI with raw provider strings used for aggregate keys, filters, suggestions, query matching, and badges.

## Solution Assessment

Yes. The narrow maintainable solution is to apply the existing canonical provider-id helper at Usage/API/UI derived boundaries, not to add a separate UI alias table or rewrite persisted raw session/config values.
