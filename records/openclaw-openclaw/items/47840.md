## Summary

Keep open. Current main still has a canonical provider normalizer, but the Usage/Control UI path continues to aggregate, filter, suggest, export, and render raw provider strings, so supported aliases such as `z-ai`, `z.ai`, and `aws-bedrock` can still appear or split from their canonical providers.

## What This Changes

The issue requests consistent Control UI display of normalized model provider names instead of raw provider aliases from config.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for aggregate grouping, display labels, provider filters, query suggestions/query matching, and CSV export while preserving raw stored session/config fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a usage entry with provider `z-ai`, `z.ai`, or `aws-bedrock` flows into `sessions.usage` and then through the Usage UI with raw provider strings used for aggregate keys, filters, suggestions, badges, labels, and CSV export.

## Solution Assessment

Yes for the narrow repair direction. The maintainable fix is to call the existing canonical provider-id helper at Usage/API/UI boundaries, not to add a separate UI alias table or silently rewrite stored session/config values.
