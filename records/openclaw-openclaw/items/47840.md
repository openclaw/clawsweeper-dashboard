- `P2`: This is a reproducible provider-display consistency bug with limited blast radius in the Control UI Usage surface.
- `impact:auth-provider`: The affected behavior is provider identity normalization and display for model/provider usage data.

## Summary

Keep open. Current main and the latest release still let raw provider aliases flow through Usage API/UI aggregation, filters, badges, and CSV/query surfaces, while the open draft PR https://github.com/openclaw/openclaw/pull/83005 is the active closing implementation path.

## What This Changes

The issue requests canonical provider ids in Control UI usage/provider displays and related usage aggregates instead of raw alias values from config or session data.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that applies normalizeProviderId at derived Usage API/UI display, filter, query, CSV, and aggregate boundaries while preserving raw stored config/session fields.

## Reproduction Assessment

Yes. Source inspection on current main shows documented aliases normalize in src/agents/provider-id.ts, but Usage API/UI paths still key and display provider values from raw modelProvider, providerOverride, and entry.provider strings.

## Solution Assessment

Yes. Reusing normalizeProviderId at derived Usage API/UI seams is the narrow maintainable fix; the separate question of whether bailian should become a canonical alias should not block the supported-alias display fix.
