## Summary

Keep open. Current main and the latest release still let raw provider aliases reach Usage API/UI aggregation, filters, badges, query suggestions, and CSV/display paths, and the open draft PR https://github.com/openclaw/openclaw/pull/83005 is the active closing implementation path rather than a landed fix.

## What This Changes

The issue asks the Control UI Usage/session provider display and aggregate surfaces to show canonical provider ids instead of raw configured aliases.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that applies normalizeProviderId at derived Usage API/UI display, filter, query, CSV, and aggregate boundaries while preserving raw stored config/session fields.

## Reproduction Assessment

Yes. Source inspection on current main shows supported aliases normalize in src/agents/provider-id.ts while Usage API/UI paths still key, filter, suggest, export, and display raw modelProvider, providerOverride, and entry.provider strings.

## Solution Assessment

Yes. Reusing normalizeProviderId at derived Usage/API/UI seams is the narrow maintainable fix; the separate question of adding or documenting a bailian alias should not block the supported-alias display fix.
