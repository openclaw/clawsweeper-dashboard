## Summary

Keep open. Current main still has a provider-id canonicalization contract, but the Usage/Control UI path continues to derive visible labels, provider filters, query suggestions, query matching, CSV export values, and aggregate keys from raw provider strings, so supported aliases such as `z.ai`/`z-ai` and `aws-bedrock` can still display or group separately from `zai` and `amazon-bedrock`.

## What This Changes

The issue requests Control UI provider displays and Usage-derived filters, searches, exports, and aggregates to show canonical provider ids instead of raw configured aliases.

## Best Possible Solution

Land a focused Usage-surface fix that applies the existing provider-id canonicalization contract to derived display labels, filters, suggestions, query matching, exports, and aggregate grouping while preserving raw persisted session/config values.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: a session or Usage entry with provider `z-ai`, `z.ai`, or `aws-bedrock` follows raw provider strings through the Usage API and Control UI display/filter/query/export paths. I did not run a browser session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse `normalizeProviderId` or a shared equivalent at derived Usage/API/UI boundaries, not to add a separate UI alias table or rewrite persisted raw session/config records.
