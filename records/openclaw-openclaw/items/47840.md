## Summary

Keep open. Current main still renders and aggregates provider display strings from raw `modelProvider`/`providerOverride` values in Sessions and Usage, the only linked fix PR was closed unmerged, and current provider-id normalization is not a broad alias contract.

## What This Changes

The issue requests consistent normalized provider labels in the Control UI instead of raw configured aliases such as `bailian`, `z-ai`, or `aws-bedrock`.

## Best Possible Solution

Define one Control UI provider display and Usage aggregation policy, then normalize derived labels, filters, query matching, CSV output, and aggregate keys without rewriting raw stored session or config fields.

## Reproduction Assessment

Yes, from source inspection: provider values flow directly from session/config fields into Sessions and Usage display paths. I did not run the UI because this review is read-only, but the code path is clear.

## Solution Assessment

Unclear. Applying the thinking-only alias helper everywhere may be too narrow or the wrong contract; maintainers should first choose the canonical display helper and whether normalization belongs only in derived UI/aggregate surfaces.
