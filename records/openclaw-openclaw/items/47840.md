Control UI should show normalized or canonical provider IDs instead of raw configured provider aliases in Sessions and Usage surfaces.

## Best Possible Solution

Define the provider-display canonicalization contract, then normalize derived Usage/Sessions display, filters, query suggestions, CSV, and aggregate keys without rewriting stored config or session fields.

Yes. Source inspection on current main shows the affected Usage/Sessions display, export, and aggregate paths still derive provider values from raw fields, and v2026.6.10 has the same patterns.

## Solution Assessment

Unclear. Canonicalizing derived UI/API surfaces is the right boundary, but the exact supported alias semantics, especially `bailian`, need maintainer confirmation before another automated fix.
