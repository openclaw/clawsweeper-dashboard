## Summary

## What This Changes

The issue asks the Control UI Usage/Sessions provider display and related filters/aggregates to show canonical provider IDs instead of raw configured aliases.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that canonicalizes derived Usage API/UI display, filter, query, CSV, and aggregate provider values with normalizeProviderId while preserving raw persisted fields.

## Reproduction Assessment

Yes. Source inspection on current main shows supported aliases normalize in src/agents/provider-id.ts while Usage API/UI paths still key, filter, suggest, export, and display raw modelProvider, providerOverride, and entry.provider strings.

## Solution Assessment

Yes. Reusing normalizeProviderId at derived Usage/API/UI seams is the narrow maintainable fix; adding a new bailian alias should remain a separate provider-contract decision.
