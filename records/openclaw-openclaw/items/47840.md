## What This Changes

The issue asks the Control UI Usage/provider surfaces to display and aggregate canonical provider ids instead of raw configured aliases.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that applies normalizeProviderId at derived Usage API/UI display, filter, query, CSV, and aggregate boundaries while preserving raw stored config/session fields.

## Reproduction Assessment

Yes. Source inspection shows the canonical provider normalizer and docs-backed alias contract exist, while current Usage API and Control UI paths still consume raw provider strings for display, filtering, export, and aggregates.

## Solution Assessment

Yes. The narrow maintainable path is to reuse normalizeProviderId at derived Usage/API/UI boundaries while preserving raw persisted session/config values; adding bailian as a new alias should be left out unless maintainers explicitly approve it.
