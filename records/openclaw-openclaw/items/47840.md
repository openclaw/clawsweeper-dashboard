## Summary

## What This Changes

The issue requests that Control UI provider displays use canonical normalized provider ids instead of raw config aliases.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that applies normalizeProviderId at derived Usage API/UI display, filter, query, CSV, and aggregate boundaries while preserving raw stored config/session fields.

## Reproduction Assessment

Yes. Source inspection shows the canonical alias normalizer and docs-backed alias contract exist, while current Usage API and Control UI paths still consume raw provider strings for display, filtering, export, and aggregates.

## Solution Assessment

Yes. The maintainable fix is to reuse normalizeProviderId at derived Usage/API/UI boundaries while preserving raw persisted fields; adding a new bailian alias should be handled separately if maintainers want it.
