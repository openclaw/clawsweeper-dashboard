- `impact:auth-provider`: The affected behavior is provider identity normalization and display for model/provider usage data.

## Summary

Keep open. Current main and v2026.5.22 still let raw provider aliases flow through Usage API/UI aggregation, filters, query suggestions, badges, and CSV/display surfaces, and the open draft PR https://github.com/openclaw/openclaw/pull/83005 is the active closing implementation path.

## What This Changes

The issue requests that Control UI Usage provider display and grouping use normalized provider IDs instead of raw config or session aliases.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83005 or a focused replacement that applies normalizeProviderId at derived Usage API/UI display, filter, query, CSV, and aggregate boundaries while preserving raw stored fields.

## Reproduction Assessment

Yes, source-level. Current main has a documented provider-id normalizer, but the inspected Usage API/UI paths still key, filter, export, and display provider values from raw strings without calling it.

## Solution Assessment

Yes, directionally. Reusing normalizeProviderId at derived Usage/API/UI seams is the narrow maintainable fix; adding a new bailian alias would be a separate provider-contract decision.
