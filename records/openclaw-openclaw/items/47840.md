## Summary

Keep open: current main and the latest release still expose raw or lowercase-only provider values in Control UI session/usage display and aggregation paths, while the only linked repair PR closed unmerged. The remaining work is valid, but the exact display contract needs maintainer direction before another fix PR.

## What This Changes

The issue asks for Control UI provider displays to show normalized provider identities consistently instead of raw configured aliases.

## Best Possible Solution

Define one provider display contract: canonicalize derived Usage/session display, filter, export, and aggregate identity with approved provider aliases, while preserving raw config keys only in configuration-editing surfaces.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main and v2026.6.6 still lowercase or render provider fields directly in Sessions, Usage filters, badges, CSV, and aggregate keys; no browser run was performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No for the issue's per-render Jp-only sketch; the safer fix is boundary canonicalization for derived runtime/usage identity plus an explicit rule for raw config-key display.
