## Summary

Keep open: current main and latest release still select the first stored GitHub Copilot profile in the implicit discovery/provider-auth path instead of honoring `auth.order`, and the only open linked repair PR is still unmerged and conflicting.

## What This Changes

The issue reports that GitHub Copilot auth selection ignores `auth.order` and uses the first stored auth profile instead.

## Best Possible Solution

Refresh or replace the open repair PR so current plugin discovery and provider-auth resolution use `resolveAuthProfileOrder()` for GitHub Copilot profile selection, with regression coverage for config order, stored order, and unresolved `tokenRef` fallback.

## Reproduction Assessment

Yes, at source level: current main still routes Copilot discovery and provider auth through raw profile listing and first usable profile selection while the auth-order contract says explicit order should constrain profile selection. I did not execute a live two-profile Copilot setup in this read-only pass.

## Solution Assessment

Yes, the requested direction is the right fix, but the active repair must target the current plugin-based discovery/provider-auth resolver rather than the obsolete pre-refactor `models-config.providers.ts` path.
