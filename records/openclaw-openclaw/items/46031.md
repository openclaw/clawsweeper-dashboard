## Summary

Keep open: current `main` and latest release `v2026.6.8` still select the first matching GitHub Copilot auth profile instead of the configured `auth.order`, and the linked repair candidate is open but conflicting and unmerged.

## What This Changes

The issue reports that GitHub Copilot implicit auth selection ignores `auth.order` and uses the first stored Copilot profile instead.

## Best Possible Solution

Land a focused fix that routes Copilot implicit discovery and provider auth through canonical `resolveAuthProfileOrder`, preserving env-token precedence and explicit provider config behavior.

## Reproduction Assessment

Yes. Source inspection shows current `main` still selects `profileIds[0]` for GitHub Copilot, and the focused current test still asserts first-profile selection; I did not run a gateway flow because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse `resolveAuthProfileOrder` in the Copilot implicit discovery/provider-auth path rather than creating a separate ordering rule.
