## Summary

Keep open: current main and the latest published release still select the first stored GitHub Copilot profile, and the open linked fix path at https://github.com/openclaw/openclaw/pull/48682 has not merged.

## What This Changes

The issue reports that GitHub Copilot implicit provider discovery ignores `auth.order` and uses the first stored Copilot auth profile instead.

## Best Possible Solution

Implement ordered Copilot profile selection in `extensions/github-copilot/auth.ts` or the existing order-aware SDK seam used by that helper, preserving env-token precedence, discovery disablement, explicit auth-order exclusions, and covering catalog plus embeddings.

## Reproduction Assessment

Yes. A high-confidence source path is to configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, and follow catalog discovery into `resolveFirstGithubToken`, which selects `profileIds[0]`.

## Solution Assessment

No landed solution exists yet. The narrow maintainable fix should live in the shared Copilot auth helper or an existing order-aware SDK auth seam; the open linked PR still needs maintainer repair, replacement, or closure with another fix path.
