## Summary

Keep open: current main and the latest shipped release still select the first stored GitHub Copilot profile in the Copilot helper, while the documented auth-order contract says explicit `auth.order` should control eligible profiles. An open fix PR references this issue with closing syntax, so this issue should stay open until that PR or a corrected replacement lands.

## What This Changes

Ensure implicit GitHub Copilot token selection honors `auth.order` instead of raw `auth.profiles` insertion order.

## Best Possible Solution

Update the Copilot plugin auth helper or plugin-facing SDK seam to use `resolveAuthProfileOrder`/`listUsableProviderAuthProfileIds` for catalog and embeddings while preserving env precedence and explicit `auth.order` exclusions.

## Reproduction Assessment

Yes. A high-confidence source path is to configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, and follow catalog discovery into `resolveFirstGithubToken`, which selects `profileIds[0]`.

## Solution Assessment

No landed solution exists yet. The narrow maintainable fix should live in the Copilot plugin helper or SDK auth seam rather than a broad core fallback that can bypass discovery-disablement or explicit auth-order exclusions.
