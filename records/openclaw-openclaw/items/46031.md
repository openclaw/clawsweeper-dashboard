- `impact:auth-provider`: The issue is about auth profile ordering and provider credential selection for GitHub Copilot.

## Summary

Keep open: current main still selects the first stored GitHub Copilot profile in the shared Copilot auth helper instead of applying `auth.order`, and an open linked PR still owns the fix path but is conflicting and previously reviewed as unsafe to merge as-is.

## What This Changes

The issue reports that implicit GitHub Copilot auth selection ignores `auth.order` and uses the first stored `github-copilot` profile instead.

## Best Possible Solution

Fix the shared Copilot plugin auth helper, or a narrow plugin-facing SDK seam it uses, so catalog and embeddings honor ordered eligible profiles while preserving env-token precedence, discovery disablement, and explicit auth-order exclusions.

## Reproduction Assessment

Yes. A high-confidence source path is to configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, and follow catalog discovery into `resolveFirstGithubToken`, which selects `profileIds[0]`.

## Solution Assessment

No landed solution exists yet. The narrow maintainable fix should live in the Copilot plugin helper or an existing order-aware SDK auth seam; the open linked PR is still conflicting and needs repair, replacement, or closure with another fix path.
