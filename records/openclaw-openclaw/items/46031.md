## Summary

Keep open. Current main still has order-blind Copilot token selection in the shared GitHub Copilot helper, and open PR #48682 references this issue with closing syntax, so this should stay open until that PR or an equivalent replacement is resolved.

## What This Changes

The issue reports that GitHub Copilot implicit provider selection ignores `auth.order` and uses the first stored `auth.profiles` entry instead.

## Best Possible Solution

The shipped behavior should use the existing order-aware auth profile resolution for GitHub Copilot token lookup, preserve environment-token precedence, and cover both catalog discovery and embeddings with regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure two stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer the non-first profile, and run implicit Copilot discovery or Copilot embeddings without env tokens. Current main reaches `profileIds[0]` from `Object.entries` order before any order-aware helper is used.

## Solution Assessment

Yes. The narrow maintainable fix is to have `resolveFirstGithubToken` use order-aware eligible profile ids, preserving env-token precedence and fallback behavior; because #48682 is open with closing syntax, this issue should remain open until that PR lands or is replaced.
