## Summary

Keep open. Current main still matches the reported GitHub Copilot auth-order bug: the shared Copilot token helper gets raw provider profiles and selects index zero, while the existing order-aware auth seam and docs define explicit `auth.order` semantics. An open fix PR already references this issue with closing syntax, so the maintainer path is to review, land, rework, or close that PR rather than close this issue early.

## What This Changes

The issue reports that implicit GitHub Copilot auth selection ignores `auth.order` and uses the first stored `auth.profiles` entry.

## Best Possible Solution

Use the existing order-aware auth seam from the GitHub Copilot plugin helper so both implicit catalog discovery and Copilot memory embeddings choose eligible profiles according to `auth.order`, with targeted regression coverage for env-token precedence, ordered profile choice, stale order, and unresolved token refs.

## Reproduction Assessment

Yes. Configure two stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer the non-first profile, and run implicit Copilot discovery or Copilot embedding setup with no env token; current main reaches `profileIds[0]` in `resolveFirstGithubToken`.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-owned: preserve environment token behavior, then iterate order-aware eligible Copilot profile ids via the existing auth-order seam. Because PR #48682 is open and references this issue, a duplicate automated repair branch is not the best path.
