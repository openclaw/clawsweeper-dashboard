## Summary

Keep open: current main and the latest shipped release still select the first stored GitHub Copilot auth profile in the shared Copilot helper, despite the documented auth-order contract. An open linked fix PR still references this issue and is currently conflicting, so the issue should remain open until that PR is repaired, replaced, or closed with a different fix path.

## What This Changes

The issue reports that GitHub Copilot auth selection ignores configured `auth.order` and instead selects the first stored `github-copilot` profile.

## Best Possible Solution

Fix the shared Copilot plugin auth helper, or a narrow plugin-facing SDK seam it uses, so catalog and embeddings honor ordered eligible profiles while preserving env-token precedence and explicit auth-order exclusions.

## Reproduction Assessment

Yes. A high-confidence source path is to configure multiple stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer a non-first profile, omit env tokens, and follow catalog discovery into `resolveFirstGithubToken`, which selects `profileIds[0]`.

## Solution Assessment

No landed solution exists yet. The narrow maintainable fix should live in the Copilot plugin helper or an existing order-aware SDK auth seam; the current linked PR appears stale against the moved plugin implementation.
