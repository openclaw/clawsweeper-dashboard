## Summary

Keep open: current `main` and the latest release still select the first stored GitHub Copilot token profile in the plugin catalog path instead of honoring `auth.order`, and the open linked PR is an implementation candidate rather than a close reason.

## What This Changes

The issue reports that implicit GitHub Copilot auth profile selection ignores `auth.order` and instead uses the first stored `auth.profiles` entry.

## Best Possible Solution

Make Copilot implicit discovery select stored tokens through the existing order-aware auth-profile path, with regression coverage for ordered profiles and env-token precedence; if the linked PR remains stale, adapt or replace it rather than adding a parallel config surface.

## Reproduction Assessment

Yes at source level: current `main` and `v2026.6.6` still route Copilot catalog token selection through `listProfilesForProvider(...)[0]` instead of the existing order-aware resolver. I did not run a live credential repro because this review is read-only and should not touch user credentials.

## Solution Assessment

Yes: the best fix is a narrow provider-auth selection change that reuses the existing `auth.order` contract and order-aware helper while preserving environment-token precedence. The open linked PR should be reviewed/adapted or replaced if its old-path diff no longer applies cleanly.
