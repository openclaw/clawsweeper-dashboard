## Summary

Keep open. Current main still selects the raw first stored GitHub Copilot profile in the shared plugin helper, and an open PR already references this issue with closing syntax, so cleanup closure would be premature.

## What This Changes

The issue reports that GitHub Copilot auth profile selection ignores existing auth.order configuration and uses the first stored profile instead.

## Best Possible Solution

Make the GitHub Copilot plugin token helper use the existing order-aware auth profile seam while preserving env-token precedence, discovery disablement, explicit auth.order exclusions, and focused coverage for catalog discovery plus embeddings.

## Reproduction Assessment

Yes, by source inspection: configure multiple stored github-copilot token profiles, set auth.order.github-copilot to prefer a non-first profile, omit env tokens, and run implicit Copilot discovery or Copilot embeddings; current main reaches profileIds[0].

## Solution Assessment

Yes. The narrow maintainable fix is to update the plugin-owned resolveFirstGithubToken path or a provider-auth SDK helper it uses, not a broad core fallback that can bypass plugin discovery controls.
