## Summary

Keep this PR open: the provider-local default flip is useful, but it should not merge until TTS-2 compatibility is made model-aware because OpenClaw still sends `temperature` for unpinned Inworld configs while upstream documents `temperature` as unsupported on Realtime TTS-2.

## What This Changes

The PR changes the bundled Inworld speech provider default to `inworld-tts-2`, updates model metadata, docs, tests, and changelog, and adds a static `X-User-Agent: openclaw` header to Inworld TTS and voices requests.

## Best Possible Solution

Make the Inworld provider's TTS-2 default model-aware for unsupported parameters, then land the provider-local metadata, docs, test, and changelog update after maintainer approval and green checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible: the changed default flows into the TTS request when no `modelId` is configured, and the existing temperature config path still serializes `temperature` for that request.

## Solution Assessment

Is this the best way to solve the issue? No as submitted: changing the provider-local default is the right surface, but the PR needs a narrow model-aware compatibility fix before making TTS-2 the unpinned default.

## Review Findings
