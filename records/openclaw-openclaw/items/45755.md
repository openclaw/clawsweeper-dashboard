## Summary

Keep open. Current main still resolves the configured identity emoji in the gateway, but the Control UI bootstrap contract, UI identity state, chat props, and chat sender footer only carry name/avatar, so the emoji is dropped before the assistant label renders; related #45898 is closed unmerged and only targeted the selector dropdown.

## What This Changes

The issue requests restoring the configured agent identity emoji beside the assistant name in the v2 Control UI chat transcript.

## Best Possible Solution

Propagate the existing optional identity emoji through the Control UI bootstrap/RPC identity contract, UI state, chat props, and streaming/persisted assistant footer rendering, with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `agents.list[].identity.emoji` or an `IDENTITY.md` emoji and render assistant chat groups on current main: the gateway resolves/returns emoji, but the UI identity and chat render paths discard it before `.chat-sender-name` is emitted.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the existing identity emoji through the Control UI identity contract and render it next to the assistant sender name; selector-dropdown-only work like #45898 does not address this transcript regression.
