## Summary

Keep open: the PR is useful and now has sufficient runtime proof, but the implementation snapshots prep timing at turn end, so `prepStages.totalMs` can include model/tool time after startup and mislead telemetry consumers.

## What This Changes

The PR exposes embedded runner prep-stage timing summaries on attempt results, embedded run metadata, and the `agent_end` plugin hook, with docs, changelog, and focused coverage.

## Best Possible Solution

Freeze the stream-ready prep-stage summary at the `stream-setup` boundary and reuse that stable summary for attempt results, run meta, and `agent_end` if maintainers accept the optional public telemetry field.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR exposing telemetry that current main does not publish. The review finding is source-reproducible from the tracker implementation and the PR's end-of-turn snapshot call sites.

## Solution Assessment

No as written: reusing the existing tracker and optional fields is narrow, but the snapshot must be captured at stream-ready time rather than at `agent_end` or function return.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Freeze prep timing before model execution:** `src/agents/pi-embedded-runner/run/attempt.ts:3341-3349`
  - body: The new hook snapshot is taken after the assistant turn has finished, but `snapshot().totalMs` is computed as elapsed time since tracker creation. On real runs with a slow model or tools, `event.prepStages.totalMs` and the returned `prepStages` can include non-prep runtime after `stream-setup`, mislabeling model time as startup latency. Capture the stream-ready snapshot immediately after `prepStages.mark("stream-setup")` and reuse it for the hook/result.
  - confidence: 0.92
