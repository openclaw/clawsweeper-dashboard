## Label Justifications

- `P1`: The PR targets a source-proven sandbox isolation bug that can make co-hosted OpenClaw instances reuse the wrong workspace runtime.
- `merge-risk: 🚨 compatibility`: The diff changes persistent non-shared sandbox identity across runtime names, registry rows, SSH roots, browser containers, and materialized skills workspaces.

## Summary

Keep open. The workspace-scoped sandbox identity approach is the right owner boundary and the PR body now includes useful live Docker proof, but the new `openclaw sandbox recreate --session` matcher can remove an unrelated workspace-qualified runtime whose raw session key only happens to have the same length.

## What This Changes

This PR adds resolved workspace hashes to non-shared sandbox scope keys and propagates them through Docker, SSH, browser, skills workspace identity, recreate filtering, docs, and regression tests.

## Best Possible Solution

Keep the context-level workspace-qualified scope-key design, fix the recreate matcher before merge, and have maintainers explicitly accept the one-time non-shared runtime identity reset.

## Reproduction Assessment

Yes. Current main and v2026.6.10 derive non-shared sandbox identity without workspace identity, and the canonical report gives a two-HOME same-agent Docker collision path; I did not run Docker in this read-only review.

## Solution Assessment

No, not in the current patch form. Computing the workspace-qualified key once in sandbox context is the best owner boundary, but `createSessionMatcher` must require the requested raw session prefix before accepting a workspace suffix.
