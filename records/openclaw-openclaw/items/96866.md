## Summary

Keep open: this is a source-proven sandbox isolation fix with sufficient Docker-path proof, but the current patch can still match and remove an unrelated workspace-qualified runtime during `openclaw sandbox recreate --session`.

## What This Changes

The branch adds workspace-qualified non-shared sandbox scope keys and propagates them through Docker, SSH, materialized skills workspaces, recreate filtering, docs, and regression tests.

## Best Possible Solution

Keep the context-level workspace-qualified scope-key design, fix the recreate matcher before merge, and have maintainers explicitly accept the documented one-time non-shared runtime identity reset.

## Reproduction Assessment

Yes. Current main and v2026.6.10 derive non-shared sandbox identity without workspace identity, and the canonical report gives a two-HOME same-agent Docker collision path; I did not run Docker in this read-only review.

## Solution Assessment

No, not in the current patch form. Computing the workspace-qualified key once in sandbox context is the right owner boundary, but `createSessionMatcher` must require the requested raw session prefix before accepting a workspace suffix.

The observable bug is that `createSessionMatcher` in `src/commands/sandbox.ts` accepts `:workspace:<8-hex>` after slicing by the requested session length without first proving the stored key starts with the requested raw session key. Update the matcher so `openclaw sandbox recreate --session <key>` matches the exact raw key or keys beginning with `${rawSessionKey}:workspace:<8-hex>` only.
