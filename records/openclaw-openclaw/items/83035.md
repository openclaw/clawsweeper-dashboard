## Summary

Keep open: this is a concrete crash report against the supported Linux gateway startup path, and current docs/package metadata still promise Node 24 or Node 22.16+. I could not reproduce it in an isolated npm install on Node v24.15.0, even with @openclaw/codex installed, so the next step is targeted debugging with the reporter's redacted stability bundle or exact NodeSource/systemd workspace rather than cleanup-close.

## What This Changes

The issue reports a gateway startup crash-loop on Linux where Node 22.22.2 and Node 24.15.0 fail inside Node's ESM-to-CJS translator before the gateway binds.

## Best Possible Solution

Identify the packaged module/import path that trips Node's translator, add a Linux packaged-startup regression smoke for that path, and only narrow the documented Node support matrix if the Node runtime cannot be worked around safely.

## Reproduction Assessment

No. The reporter supplied detailed crash logs, but an isolated openclaw@2026.5.12 plus @openclaw/codex@2026.5.12 startup on Node v24.15.0 reached ready, so we still need the redacted stability bundle or exact NodeSource/systemd workspace to reproduce the crash with high confidence.

## Solution Assessment

Unclear. The maintainable fix should target the specific packaged import/CJS dependency that triggers the translator assertion; changing docs or engine ranges is a fallback only after proving the runtime cannot be supported.
