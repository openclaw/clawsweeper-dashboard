## Summary

Keep open: current main still creates the WhatsApp Baileys socket with `syncFullHistory: false` and no explicit `shouldSyncHistoryMessage`, while Baileys v7 defaults that callback from `syncFullHistory`. The closed #66920 work addressed post-408 reconnect churn, but it does not fully supersede this linked-device relink/history-sync report.

## What This Changes

The issue asks OpenClaw's WhatsApp linked-device channel to receive and log group @mentions when DMs already work.

## Best Possible Solution

Keep the issue open and fix WhatsApp socket bootstrap so linked-device group participation/sender-key history sync is requested without enabling auto-replies to stale history.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source and dependency inspection give a clear path: current main leaves Baileys history-sync gating tied to `syncFullHistory=false`, so linked-device group events can fail before OpenClaw's `messages.upsert` handler logs anything.

## Solution Assessment

Yes, the best direction is a narrow WhatsApp plugin socket-option fix with regression coverage. Do not close this as #66920, because that landed reconnect-churn recovery while the missing Baileys history-sync override remains in current main.
