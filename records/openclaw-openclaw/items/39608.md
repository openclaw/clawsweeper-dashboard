## Summary

Keep open: current `main` still creates the WhatsApp Baileys socket with `syncFullHistory: false` and no explicit `shouldSyncHistoryMessage`, so the linked-device group-sync failure described here is not fixed by the merged reconnect recovery work. The issue is actionable from source and dependency contracts, but it still needs a focused WhatsApp plugin fix and regression coverage.

## What This Changes

The issue reports that WhatsApp linked-device group mentions never reach the gateway while direct messages continue working.

## Best Possible Solution

Update the WhatsApp plugin socket bootstrap to request the needed Baileys history-sync metadata for linked-device group delivery while preserving OpenClaw's existing protections against replying to stale history messages.

## Reproduction Assessment

No, not with live high confidence in this read-only review. Source and Baileys inspection provide a credible current-main path: group inbound can disappear before `messages.upsert`, while direct messages continue, because history-sync gating is still tied to `syncFullHistory: false`.

## Solution Assessment

Yes, the best direction is a narrow WhatsApp plugin socket-option fix with regression coverage around session creation and inbound history handling. Closing this as covered by #66920 would be premature because that shipped reconnect listener recovery, not the missing history-sync override.
