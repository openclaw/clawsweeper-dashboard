## Summary

Keep open. Current main and the latest stable release still have the permissive bearer fallback defaults described in the report; repository exposure is lower because the helper has no production callers and is not a documented SDK subpath, but the security-sensitive SDK-adjacent contract needs maintainer direction before changing or removing it.

## What This Changes

Request fail-closed bearer auth fallback defaults so fetch retries cannot attach tokens to arbitrary HTTP or unapproved URLs.

## Best Possible Solution

Decide whether this helper should remain supported; if it does, make token attachment fail closed with HTTPS-only defaults, an explicit attachment decision or allowlist, regression tests, and documentation.

## Reproduction Assessment

Yes. Source inspection and the checked-in unit test show that omitting `shouldAttachAuth` lets a 401 response trigger a retry with `Authorization: Bearer token-1`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer security/product judgment. Fail-closed defaults are the safer direction, but because the helper appears unused and undocumented as a public SDK subpath, deleting or internalizing it may be cleaner than promoting a breaking contract.
