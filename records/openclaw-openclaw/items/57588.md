## Summary

Keep open. Current main and the latest stable release still have the permissive bearer fallback defaults described in the report; the exposure is lower because the helper appears unused by production code and is not a documented SDK subpath, but this security-sensitive SDK-adjacent contract needs maintainer direction before changing or removing it.

## What This Changes

Request to make the bearer auth fallback helper fail closed so it does not attach bearer tokens to arbitrary 401/403 responses without HTTPS and caller-provided attachment restrictions.

## Best Possible Solution

Decide whether this helper should remain supported; if it does, make token attachment fail closed with HTTPS-only defaults, an explicit attachment decision or allowlist, regression tests, and documentation.

## Reproduction Assessment

Yes. Source inspection and the checked-in unit test show that omitting shouldAttachAuth lets a 401 response trigger a retry with Authorization: Bearer token-1; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer security/product judgment. Fail-closed defaults are safer if the helper stays, but because it is not a documented/exported SDK subpath, deleting or internalizing the helper may be narrower than promoting a breaking contract.
