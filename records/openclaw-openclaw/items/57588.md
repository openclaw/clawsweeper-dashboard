## Summary

Keep open. Current main and the latest stable release still have the permissive bearer fallback defaults described in the report; exposure appears lower because the helper is not documented/exported and has no production callers found, but this is a security-sensitive SDK-adjacent contract that needs maintainer direction before changing or removing it.

## What This Changes

The issue asks OpenClaw to make the bearer auth fallback helper fail closed so bearer tokens are not retried to arbitrary URLs by default.

## Best Possible Solution

Decide whether this helper should remain; if it does, make token retry require an explicit URL trust decision, default to HTTPS-only, add regression coverage, and document the supported contract.

## Reproduction Assessment

Yes. Source inspection plus the checked-in unit test show that omitting shouldAttachAuth lets a 401 response trigger a retry with Authorization: Bearer token-1; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer security/product judgment. Fail-closed defaults are safer if the helper stays, but because it is not a documented/exported SDK subpath, deletion or internalization may be narrower than promoting a breaking contract.
