## Summary

Keep open. Current main and latest release still carry the permissive bearer fallback defaults, and the source/test path reproduces the token-attachment concern; changing this SDK-adjacent auth helper needs maintainer security/product review rather than cleanup closure.

## What This Changes

The issue requests fail-closed defaults for `src/plugin-sdk/fetch-auth.ts` so bearer retries require HTTPS and an explicit URL trust predicate.

## Best Possible Solution

Decide whether this helper should remain; if it does, make its contract fail closed with explicit URL trust, HTTPS-by-default behavior, focused regression coverage, and matching SDK documentation.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. The current helper retries 401/403 responses with `Authorization` when no `shouldAttachAuth` predicate is supplied, and the checked-in test asserts that behavior.

## Solution Assessment

Unclear. Fail-closed defaults are likely safer if the helper remains, but deletion or internalization may be cleaner because `fetch-auth` is not a documented SDK subpath.
