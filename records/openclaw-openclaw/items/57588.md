## Summary

Keep open. Current main and v2026.5.7 still contain the permissive bearer fallback behavior; immediate shipped exposure is lower because the helper is unused and not a documented SDK entrypoint, but the security hardening decision is still real repo work rather than cleanup closure.

## What This Changes

The issue requests fail-closed defaults for `fetchWithBearerAuthScopeFallback` so bearer tokens are not retried to arbitrary 401/403 URLs by default.

## Best Possible Solution

Decide whether this helper should exist; if it remains, make token attachment fail closed with HTTPS-only behavior, an explicit attachment predicate or allowlist, focused tests, and matching SDK documentation if it becomes public.

## Reproduction Assessment

Yes. Source inspection and the checked-in unit test show that omitting `shouldAttachAuth` allows a 401 response to trigger a retry with `Authorization: Bearer token-1`; tests were not run because this review is read-only.

## Solution Assessment

Unclear pending maintainer judgment. Fail-closed behavior is the safer direction, but the helper is currently unused and not a documented SDK subpath, so deletion or a narrower internal pattern may be better than a breaking public contract change.
