## Summary

Keep open. Current main still has the permissive bearer fallback defaults described by the report, and the related #43005 redirect-header hardening does not cover this 401/403 retry path.

## What This Changes

Request to make `fetchWithBearerAuthScopeFallback` fail closed before retrying failed requests with bearer tokens.

## Best Possible Solution

Harden or retire this helper before treating it as a supported SDK surface: default to HTTPS-only, require an explicit attachment decision or allowlist, add regression tests, and document the intended contract.

## Reproduction Assessment

Yes. The existing unit test demonstrates the core behavior: with `shouldAttachAuth` omitted, a 401 response triggers a bearer-token retry and sends `Authorization: Bearer token-1`.

## Solution Assessment

Unclear pending maintainer security judgment. The requested fail-closed defaults are the safer direction, but because this is SDK-adjacent and potentially breaking, maintainers should choose between hardening the contract or removing the helper from the surface.
