## Summary

Keep open. Current main still matches the report: bearer fallback HTTPS enforcement is opt-in, the auth-attachment predicate is optional, and existing tests lock in retrying with a bearer token when `shouldAttachAuth` is omitted. The linked #43005 remains a related but separate fetch-guard redirect-header topic, not a canonical fix for this fallback helper.

## What This Changes

The issue requests fail-closed defaults for `fetchWithBearerAuthScopeFallback`, requiring HTTPS and an explicit attachment predicate or allowlist before bearer-token retries.

## Best Possible Solution

Keep this open as a focused SDK security-hardening item. Maintainers should decide the supported surface, then either remove the helper or make bearer fallback fail closed with HTTPS-by-default, an explicit host/predicate decision before attaching tokens, regression tests for HTTP and missing predicates, and docs if it becomes a supported plugin API.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction exists: call `fetchWithBearerAuthScopeFallback` with a `tokenProvider` and no `requireHttps` or `shouldAttachAuth`; current tests already expect a bearer retry when the predicate is undefined, and source only rejects HTTP when `requireHttps === true`.

## Solution Assessment

Unclear until maintainer security review picks the compatibility path. The fail-closed direction is the narrow security fix, but the safest implementation may be to harden the helper before publication or remove it if it is only an internal leftover.
