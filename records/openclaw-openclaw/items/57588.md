## Summary

Keep open. Current main and the latest release still have the permissive bearer fallback defaults described in the report; exposure is lower because the helper has no production call sites, but the SDK-adjacent contract still needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

Harden the fetch-auth bearer fallback so tokens are not attached to arbitrary 401/403 retry URLs without explicit HTTPS and attachment policy.

## Best Possible Solution

Decide whether this helper should be a supported SDK surface; if it remains, make token attachment fail closed with HTTPS-only defaults, an explicit attachment decision or allowlist, regression tests, and documentation.

## Reproduction Assessment

Yes. Source inspection and the checked-in unit test show that omitting `shouldAttachAuth` allows a 401 response to trigger a retry with `Authorization: Bearer token-1`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer security/product judgment. Fail-closed defaults are the safer direction, but because the helper appears unused and undocumented as a public SDK subpath, deletion or keeping the safer MSTeams-local pattern may be cleaner than promoting a breaking SDK contract.
