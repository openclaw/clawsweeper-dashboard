## Summary

Keep open. Current main and v2026.5.12 still carry the permissive fetch-auth bearer fallback defaults, and the source/test path clearly demonstrates auth retries without an explicit URL trust predicate; changing or removing this SDK-adjacent helper needs maintainer security/product judgment rather than conservative cleanup.

## What This Changes

Request to make `src/plugin-sdk/fetch-auth.ts` fail closed by default for bearer-token fallback retries, including HTTPS and URL-attachment restrictions.

## Best Possible Solution

Decide whether this helper should remain; if it does, require an explicit URL trust decision, default to HTTPS-only, add regression coverage, and document the supported contract.

## Reproduction Assessment

Yes. Source inspection shows the retry path only rejects when `shouldAttachAuth` exists and returns false, and the checked-in test omits `shouldAttachAuth` while expecting `Authorization: Bearer token-1` on the second fetch.

## Solution Assessment

Unclear. Fail-closed defaults are safer if the helper remains, but because it is not a documented SDK subpath and appears unused in production code, removal or internalization may be narrower than promoting a breaking contract.
