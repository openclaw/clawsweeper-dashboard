## What This Changes

The issue requests plugin slash-command dispatch for `openclaw agent` local/gateway paths and embedded TUI input.

## Best Possible Solution

Add a shared command-dispatch preflight for agent local/gateway ingress and embedded TUI that reuses the existing auto-reply command semantics, session metadata, authorization, and fallthrough behavior.

Yes. Source inspection is enough to reproduce the routing failure: plugin commands are dispatched from the auto-reply/chat path, while agent ingress and embedded TUI pass slash text directly to the agent runner.

## Solution Assessment

Yes, with scope control. The narrow maintainable fix is to reuse `handleCommands` or a shared wrapper before agent execution, while leaving unmatched slash/plain messages on the existing LLM path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78347. Plugin slash commands registered with `api.registerCommand()` should run when entered through `openclaw agent --local --message`, gateway-backed `openclaw agent --message`, and embedded TUI slash input. Reuse the existing auto-reply command dispatcher or a shared helper; do not create a separate plugin-command implementation. Preserve existing `chat.send` behavior, unmatched-command fallthrough to the LLM path, Gateway operator scopes, `senderIsOwner`/authorization semantics, session metadata, and model override boundaries. Add regression tests with a registered test plugin command for local agent, gateway agent, and embedded TUI paths, plus fallthrough coverage for plain/unmatched messages. Update `CHANGELOG.md` with a concise fix entry unless maintainers decide the change is internal-only. Stop and ask for maintainer review if the implementation would require a new config option, protocol-breaking Gateway change, or broader command authorization policy decision.
