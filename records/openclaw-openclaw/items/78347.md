## Summary

Keep open: current main still forwards `openclaw agent` local/gateway messages and unknown TUI slash input into agent execution, while plugin command execution remains wired through the auto-reply/channel command stack. Related work improved TUI autocomplete and tracks pass-through API semantics, but neither closes this missing dispatch boundary.

## What This Changes

Plugin-registered slash commands should dispatch from `openclaw agent` local/gateway paths and the TUI instead of being forwarded as ordinary agent prompts.

## Best Possible Solution

Add a shared pre-agent command-dispatch boundary for CLI local, Gateway agent RPC, and TUI input that reuses the existing plugin dispatcher while making authorization, session, channel, and reply context explicit.

## Reproduction Assessment

Yes. Source inspection shows current main forwards `--message` and TUI unknown slash input into agent execution paths, while plugin command dispatch is only present in the auto-reply/channel command stack; no live plugin was run because this review is read-only.

## Solution Assessment

Unclear as a final fix. Reusing the existing dispatcher is the right direction, but maintainers still need to decide where CLI, Gateway RPC, and local TUI supply authorization, session, channel, and reply context before agent execution.
