## Summary

Keep open: current main still has the plugin command dispatcher wired through auto-reply/chat/channel handling, while `openclaw agent` local/gateway runs and embedded TUI send slash-prefixed text into the agent path without invoking that dispatcher; the gateway-backed TUI path already appears covered through `chat.send`.

## What This Changes

The issue asks OpenClaw to dispatch plugin-registered slash commands from `openclaw agent` local/gateway runs and TUI input instead of sending them to the LLM.

## Best Possible Solution

Add a shared command-dispatch preflight for agent local/gateway ingress and embedded TUI that reuses existing auto-reply command semantics, session metadata, authorization, and unmatched-message fallthrough.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Current main defines plugin slash commands and dispatches them through auto-reply/chat, but the agent CLI/gateway and embedded TUI paths forward slash text to `agentCommandFromIngress`/`runAgentAttempt` without any dispatcher call.

## Solution Assessment

Yes, with scope control. The narrow maintainable fix is to reuse `handleCommands` or a shared helper before agent execution while preserving gateway scopes and leaving unmatched slash/plain messages on the existing LLM path.
