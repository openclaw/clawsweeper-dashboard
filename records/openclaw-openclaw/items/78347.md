## Summary

Keep open: current main still does not dispatch plugin-registered slash commands on the `openclaw agent` local or gateway-agent RPC paths, and local TUI still forwards unknown slash input into the embedded backend. Related work covers gateway TUI autocomplete and an open local-TUI shared-command fallthrough PR, but not the requested plugin-command execution path for `openclaw agent`.

## What This Changes

The issue requests dispatching plugin-registered slash commands from `openclaw agent` local/gateway paths and TUI instead of sending matching slash text to the model.

## Best Possible Solution

Add a shared pre-agent command dispatch boundary for `openclaw agent` local/gateway RPC and local TUI that reuses the existing plugin command dispatcher, preserves current `chat.send` behavior, and makes authorization/session context explicit.

## Reproduction Assessment

Yes, source-reproducible: current main passes prepared agent input directly to ACP or embedded agent execution, while plugin-command dispatch is wired through the auto-reply command handlers and selected channel/chat paths. I did not run a live plugin because this review is read-only.

## Solution Assessment

Unclear as a final fix: reusing `handleCommands` is plausible, but the best owner boundary must decide how CLI, gateway agent RPC, and local TUI supply authorization, channel, session, and reply context before agent execution.
