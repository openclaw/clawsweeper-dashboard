## Summary

Keep open: current `main` still routes `openclaw agent` local/gateway runs and embedded TUI sends through the agent runner without the existing plugin command dispatcher. The gateway TUI path appears to be better than reported because unknown TUI slash commands are forwarded to `chat.send`, which already goes through the auto-reply command pipeline.

## What This Changes

The issue requests plugin-registered slash commands to dispatch from `openclaw agent` local/gateway runs and from TUI slash input instead of being sent to the LLM.

## Best Possible Solution

Add one shared command-dispatch preflight for agent local/gateway ingress and embedded TUI that reuses the existing auto-reply command semantics, authorization, session metadata, and fallthrough behavior.

## Reproduction Assessment

Yes. Source inspection shows registered plugin commands are dispatched from the auto-reply/chat pipeline, while `openclaw agent` and embedded TUI pass slash text directly into `agentCommand`/`agentCommandFromIngress` without calling that dispatcher.

## Solution Assessment

Yes, with scope control. The narrow maintainable fix is to reuse the existing command dispatcher or a shared helper before agent execution, while preserving gateway scopes and leaving unmatched slash/plain messages on the existing LLM path.
