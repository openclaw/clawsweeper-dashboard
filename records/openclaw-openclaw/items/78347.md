## Summary

Keep open: current main and the latest release still route CLI/TUI slash text into agent execution, while plugin command dispatch remains owned by the auto-reply/channel command stack. The remaining work is real but crosses CLI, Gateway, TUI, and plugin-command authorization boundaries, so it needs maintainer direction before implementation.

## What This Changes

Requests plugin-registered slash commands to dispatch from `openclaw agent` local/gateway paths and TUI instead of being forwarded as normal agent prompts.

## Best Possible Solution

Add a shared pre-agent command-dispatch boundary for CLI local, Gateway agent RPC, and TUI input that reuses the existing plugin dispatcher while making authorization, session, channel, and reply context explicit.

## Reproduction Assessment

Yes, source-level. Current main still forwards `--message` and TUI unknown slash input into agent execution paths, while plugin command dispatch is only present in the auto-reply/channel command stack.

## Solution Assessment

Unclear as a final fix. Reusing the existing dispatcher is the right direction, but maintainers still need to decide where CLI, Gateway RPC, and local TUI supply authorization, session, channel, and reply context before agent execution.
