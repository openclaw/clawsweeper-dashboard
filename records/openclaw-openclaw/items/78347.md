## Summary

Keep open: current main only partially covers slash-command routing through Gateway chat.send, while the raw CLI/Gateway agent path and embedded TUI still forward slash text into agent runs without the plugin command dispatcher.

## What This Changes

Dispatch plugin-registered slash commands from openclaw agent local/gateway and TUI input instead of treating them as ordinary agent prompts.

## Best Possible Solution

Add one shared core command-dispatch seam for standalone slash input that aligns openclaw agent and embedded/gateway TUI with chat.send, with explicit auth context and tests for matched, pass-through, and non-matching messages.

## Reproduction Assessment

Yes, source-reproducible: current main shows the raw CLI/Gateway agent and embedded TUI paths bypass handleCommands while chat.send uses the command pipeline.

## Solution Assessment

Unclear: calling handleCommands in each entrypoint is plausible, but the best fix should probably be a shared dispatch seam with a maintainer-approved auth and reply contract.
