## Summary

Current main still matches the reported regression, and the issue should remain open because #78183 is an open, linked fix PR with closing syntax for this exact behavior.

## What This Changes

The issue reports that streaming provider HTTP 4xx responses with `text/event-stream` lose their JSON error body before provider error extraction can surface upstream details.

## Best Possible Solution

Review and land #78183 if its focused `!response.ok` sanitizer bypass and regression test pass maintainer checks, then close this issue as implemented on main.

## Reproduction Assessment

Yes from source inspection: a 400 response with `content-type: text/event-stream` and a plain JSON body flows through `buildGuardedModelFetch`, enters `sanitizeOpenAISdkSseResponse`, and loses chunks that lack `data:` before `createProviderHttpError` reads the body. I did not run a live Gemini request or local Vitest in this read-only review.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: skip the OpenAI SDK SSE sanitizer only for non-OK responses while leaving successful stream sanitization intact. The issue should remain open until that PR or an equivalent fix merges.
