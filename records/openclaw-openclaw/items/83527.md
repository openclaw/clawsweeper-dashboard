## Summary

Keep open: this is a focused Control UI bug fix, but the PR has blocking correctness gaps and only unit-test proof. The patch currently misses the `showToolCalls: false` readable path it is meant to improve and can show failed message-tool sends as normal assistant replies.

## What This Changes

The PR adds a `buildChatItems` projection that turns `message(action=send)` toolcall text into synthetic assistant bubbles, plus unit tests for projection, duplicate mirrors, non-send actions, and empty text.

## Best Possible Solution

Keep the fix in the Control UI chat projection path, but move it ahead of tool-call hiding/search filtering, gate bubbles on a matching successful send result or delivery mirror, dedupe narrowly, and add real Control UI proof.

## Reproduction Assessment

Yes at source level: current `normalizeMessage` turns assistant toolcall-only content into `toolResult`, and current `buildChatItems` has no message-send projection path. I did not run tests because this review was read-only.

## Solution Assessment

No, not yet: projecting message-tool sends in `buildChatItems` is the right boundary, but this implementation must run in readable mode and prove delivery success before creating a normal assistant bubble.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Project before hiding tool-call-only turns:** `ui/src/ui/chat/build-chat-items.ts:622-634`
  - body: `normalizeMessage` reclassifies assistant messages that contain only toolcall blocks as `toolResult`, but this projection runs after the `!showToolCalls` skip. With readable mode enabled, the exact `message(action=send)` turn this PR targets is skipped before the synthetic assistant bubble can be added; move the projection before this guard or exempt raw assistant message sends and add `showToolCalls: false` coverage.
  - confidence: 0.94
- **[P2] Gate bubbles on a successful send result:** `ui/src/ui/chat/build-chat-items.ts:663-672`
  - body: The new bubble is created from the toolcall arguments alone, without checking the paired `toolResult`. If a message send fails, is aborted, or returns an error, Control UI will still show the text as a normal assistant reply even though the user did not receive it; pair by `toolCallId` and require a successful delivery result or mirror before projecting.
  - confidence: 0.9
- **[P3] Apply search filtering to projected bubbles:** `ui/src/ui/chat/build-chat-items.ts:663-673`
  - body: The projection pushes synthetic bubbles before the existing `messageMatchesSearchQuery` filter runs. When chat search is active, an unrelated `message(action=send)` turn can leave a projected bubble in the results even though the source message is skipped; apply the same search predicate before injecting the bubble.
  - confidence: 0.82
