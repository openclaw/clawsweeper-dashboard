## Summary

Keep open: the fix direction matches the linked Control UI bug, but the patch currently projects send-tool intent as an assistant bubble without verifying that the send succeeded or targeted the current source conversation, and the supplied proof is unit-test output only.

## What This Changes

The branch adds a Control UI history projection in `buildChatItems` plus unit tests for `message(action=send)` projection, delivery-mirror de-duplication, non-send actions, and empty-text sends.

## Best Possible Solution

Keep the UI projection, but drive it from successful current-source delivery semantics or an equivalent matching tool-result check, add failure/off-target coverage, and require real Control UI/WebChat proof before merge.

## Reproduction Assessment

Yes at source level: current `normalizeMessage` classifies assistant messages containing only tool-call blocks as `toolResult`, and current `buildChatItems` has no projection path that turns `message(action=send)` text into an assistant bubble. I did not run a live Control UI reproduction in this read-only review.

## Solution Assessment

No, not as written. Projecting readable text in the UI is a reasonable narrow fix, but it should require the same successful/current-source delivery semantics as delivery mirrors or an equivalent matching tool-result check.
