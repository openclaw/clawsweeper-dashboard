## Summary

Keep open: current main still lacks the requested Control UI projection, but this branch is not merge-ready because it can render failed or off-target sends as assistant replies, bypasses chat search filtering, and only supplies unit-test output as proof.

## What This Changes

The PR adds a Control UI `buildChatItems` projection that turns unmirrored `message(action=send)` tool-call text into synthetic assistant bubbles, plus focused unit tests.

## Best Possible Solution

Keep the readable projection idea, but align it with delivery-mirror success/current-source semantics, preserve search filtering for projected bubbles, add regression coverage, and require real Control UI/WebChat proof before merge.

## Reproduction Assessment

Yes at source level: current `normalizeMessage` classifies assistant messages containing only tool-call blocks as `toolResult`, and current `buildChatItems` has no projection path that turns `message(action=send)` text into an assistant bubble. I did not run a live Control UI reproduction in this read-only review.

## Solution Assessment

No, not as written. A UI projection is a reasonable narrow fix, but it should require successful current-source delivery or equivalent mirror semantics and it must participate in chat search filtering.
