## Label Justifications

- `P1`: The report describes an upgrade regression that breaks embedded agent replies across real channel workflows for multiple users.
- `impact:session-state`: The failure is thrown by the embedded session-file takeover fence and directly concerns session JSONL ownership and transcript state.
- `impact:message-loss`: The channel can accept a user message and run model work, but the final reply can fail, drop, duplicate, or be replaced by an error.
- `impact:crash-loop`: A production report describes repeated takeover retries exhausting disk I/O until the VM became effectively unavailable.

## Summary

Keep open: current main still has the exact session-file fingerprint fence that throws after an unreconciled write during the prompt-release window, and an open closing PR already owns the broader same-process write fix.

## What This Changes

The issue reports embedded agent runs failing with `EmbeddedAttemptSessionTakeoverError` when legitimate same-process session writes change the JSONL while the prompt lock is released, causing dropped or failed replies across channels.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84250 or an equivalent ownership-aware fence fix, then track retry backoff and backup retention separately if they remain after the primary race is removed.

## Reproduction Assessment

Yes for a source-level reproduction: current main releases the prompt lock, saves an exact fingerprint, and throws if another same-session write changes the file before reacquire. I did not run a live Feishu/MiniMax/APFS reproduction in this read-only pass.

## Solution Assessment

Yes in direction: the maintainable fix is to make the fence ownership-aware for explicit OpenClaw-managed writes while preserving fail-closed external takeover detection. Current main only partially covers a delivery-mirror path, while the open closing PR targets the broader same-process write case.
