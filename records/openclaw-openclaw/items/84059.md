## Label Justifications

- `P1`: The issue describes a fresh upgrade regression that breaks embedded channel replies for real Feishu and Slack users.
- `impact:session-state`: The failure is thrown by the embedded session-file takeover fence and directly concerns transcript/session JSONL state.
- `impact:message-loss`: The channel accepts user or reaction events but the embedded run can fail before committing or delivering the final visible reply.

## Summary

Keep open: current main and v2026.5.18 still contain the reported fail-closed session-file fingerprint fence, and the discussion now has concrete Feishu, Slack, and cross-channel runtime evidence. The active related PR is useful but still open and does not by itself close the Feishu reaction/co-tenant write variant reported here.

## What This Changes

Prevent embedded agent turns from failing with `EmbeddedAttemptSessionTakeoverError` when legitimate session JSONL writes occur while the embedded prompt lock is released.

## Best Possible Solution

Keep the takeover fence, but make it ownership-aware: tolerate only verified OpenClaw-owned append-only transcript writes and separately serialize or defer same-session channel events such as Feishu reactions so active turns do not lose replies.

## Reproduction Assessment

Yes at source level: current main snapshots the session file at `releaseForPrompt()` and throws when the file fingerprint changes before the next `withSessionWriteLock()`. I did not run the live Feishu/Slack repro, but multiple comments provide concrete runtime logs and triggers that map to the inspected source path.

## Solution Assessment

Unclear as merged code because the related repair is still open. The maintainable direction is to combine the append-only runner-owned validation from https://github.com/openclaw/openclaw/pull/84046 with coverage for same-session channel/co-tenant writes such as the Feishu reaction trigger.
