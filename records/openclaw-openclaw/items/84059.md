## Summary

Keep open: current main still has the strict embedded session-file fingerprint fence, and an open fix PR with closing syntax is actively targeting this issue. The issue should stay open until a canonical session-lock fix merges.

## What This Changes

The issue reports a post-upgrade embedded runner failure where legitimate session-file writes during the released prompt-lock window trigger `EmbeddedAttemptSessionTakeoverError` and prevent channel replies.

## Best Possible Solution

Land one canonical ownership-aware session-lock fix that accepts legitimate lock-coordinated OpenClaw writes during prompt release while preserving fail-closed rejection of external session-file mutation, then close this issue after merge and release tracking.

## Reproduction Assessment

Yes, at source level: release one embedded attempt for prompt I/O, mutate the same session file while the fence is active, then call `withSessionWriteLock` and current main throws `EmbeddedAttemptSessionTakeoverError`. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear which open PR is the best final implementation because https://github.com/openclaw/openclaw/pull/84250, https://github.com/openclaw/openclaw/pull/84149, and https://github.com/openclaw/openclaw/pull/84289 overlap.
