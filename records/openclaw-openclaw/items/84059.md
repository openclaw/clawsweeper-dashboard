## Label Justifications

- `impact:session-state`: The failure is thrown by the embedded session-file takeover fence and directly concerns transcript/session file state.

## Summary

Keep open: current main still contains the exact fingerprint fence that throws on session-file changes after prompt-lock release, and the live issue is already linked to an open closing PR, https://github.com/openclaw/openclaw/pull/84250.

## What This Changes

This issue asks OpenClaw to stop false-positive embedded-session takeover failures when legitimate in-process transcript or session writes occur while the embedded prompt lock is released.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84250 or an equivalent ownership-aware fence fix, keep true external takeover detection fail-closed, and preserve this issue as the canonical tracker until that fix ships.

## Reproduction Assessment

Yes, at source level: current main snapshots the session-file fingerprint at `releaseForPrompt()` and throws on any fingerprint change before post-prompt write-lock reacquisition. I did not run a live channel reproduction, but the issue has multiple matching logs across Feishu, Slack, WebChat, Discord, and Telegram.

## Solution Assessment

Yes for the direction: the maintainable fix is to distinguish OpenClaw-owned in-process writes from true external takeovers, not to remove the fence or broadly ignore timestamp precision. The issue should stay open while the linked fix PR is reviewed or replaced.
