## Label Justifications

- `P1`: Multiple users report an upgrade regression where embedded channel turns fail before or after visible replies across Feishu, Slack, WebChat, and Discord.
- `impact:session-state`: The failure is thrown by the embedded session-file takeover fence and directly concerns transcript/session file state.
- `impact:message-loss`: The reported channels accept user input but the embedded run can fail before committing or surfacing the reply correctly.

## Summary

Keep open: current main still has the exact embedded session-file fence that throws on any fingerprint change after the prompt lock is released, and an open closing PR already targets the reported same-process write race.

## What This Changes

The issue reports embedded channel turns failing after upgrade because legitimate session/transcript writes during released prompt I/O are classified as session takeover.

## Best Possible Solution

Land or replace the linked ownership-aware session-fence fix, with regression coverage for trusted same-process writes and rejected external edits, then close this issue after the fix reaches main or a release.

## Reproduction Assessment

Yes, for a source-level reproduction: current main snapshots the session file at releaseForPrompt and throws if an append changes the fingerprint before withSessionWriteLock reacquires. I did not run a live channel reproduction in this read-only pass.

## Solution Assessment

Yes, the best repair direction is an ownership-aware fence that accepts legitimate same-process locked writes while preserving external takeover rejection. Relaxing timestamp precision alone would be a weaker fix because it could hide real file changes.
