## Summary

Keep open: current main still lacks a supported model/provider override for main-session cron jobs that use explicit `systemEvent` payloads, and the remaining choice is a product/API decision rather than stale cleanup.

## What This Changes

The issue requests a job-level model/provider override for cron jobs that run through `sessionTarget: "main"` with `payload.kind: "systemEvent"`.

## Best Possible Solution

Define one maintainer-approved cron override contract, validate it through the existing model-selection policy, and update protocol schema, normalization, persistence, main-session runtime, UI/docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: explicit `systemEvent` cron payloads cannot carry `model`, normalization deletes it, persistence does not round-trip it, and main-session runtime does not pass a model/provider override to heartbeat.

## Solution Assessment

Unclear. The need is valid, but the safe solution depends on maintainers choosing the supported API shape and provider/session precedence before implementation.
