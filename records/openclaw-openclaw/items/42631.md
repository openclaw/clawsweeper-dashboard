## Summary

Keep open. Current main and the latest release still lack a supported job-level model/provider override for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`; the source path is clear, but the fix changes cron/Gateway API semantics and override lifetime, so it needs maintainer product review.

## What This Changes

Request support for choosing a cron job's provider/model on stable main-session `systemEvent` runs without manually editing session metadata.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated job-scoped provider/model overrides on main-session system events, with explicit transient-versus-persisted semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `systemEvent` payloads cannot carry model/provider data, validation rejects extra fields, normalization strips attempted overrides, and the main-session enqueue/wake path has no override parameter. I did not run a live cron job because this is a read-only review.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public contract first because the proposed shapes differ in validation, provider routing, and whether overrides are transient or persisted.
