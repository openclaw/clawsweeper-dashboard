## Label Justifications

- `P2`: This is a clear cron/model capability gap with a source-reproducible path, but it is not an urgent outage or regression.
- `impact:session-state`: The workaround mutates session model metadata, and the correct fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior affects provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and the latest release still do not provide a supported job-level provider/model override for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`, and the only linked implementation PR was closed unmerged after review found runtime design problems.

## What This Changes

Allow cron jobs using main-session `systemEvent` payloads to choose a cheaper job-scoped provider/model without editing internal session metadata.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient job-scoped provider/model overrides on main-session `systemEvent` jobs, covering both wake modes without persisting cron-only state into later turns.

## Reproduction Assessment

Yes. Source inspection on current main shows `systemEvent` payloads accept only `kind/text`, Gateway validation rejects extra fields, normalization strips overrides, and main-session heartbeat execution has no model/provider carrier.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public field shape and transient-versus-persisted session semantics before implementation; the closed `payload.model` attempt was not safe as written.
