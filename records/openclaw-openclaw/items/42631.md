## Summary

Keep open: current main still lacks a supported job-level model override for `main` + `systemEvent` cron jobs, while the implemented model path is scoped to isolated `agentTurn` jobs and the remaining design affects main-session model persistence.

## What This Changes

The issue requests job-level model/provider selection for cron jobs that enqueue main-session system events.

## Best Possible Solution

Add an additive cron API/CLI/docs path that lets main-session `systemEvent` jobs declare a job primary model and applies it only to the intended cron-triggered turn, with regression coverage for validation, fallback behavior, and session persistence.

## Reproduction Assessment

Yes from source inspection: current main accepts `model` only on `agentTurn` payloads, rejects extra `systemEvent` fields, and does not forward model/provider data through the main-session cron execution path. I did not run a live cron because this was a read-only review.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation should be chosen by maintainers because mutating session-store overrides would reproduce the workaround while potentially changing later main-session behavior.

## Risks / Open Questions

- The product semantics are not settled: a main-session cron model could be a one-turn job primary, a persisted session override, or a dedicated-session override, and those have different effects on later interactive turns.
- Provider/model validation and fallback behavior need to stay aligned with the existing isolated cron `payload.model` contract if this is extended to main-session jobs.
- No live cron run was executed because this review was read-only, but the current schema and execution path are sufficient to show the requested capability is missing.
