## Label Justifications

- `P2`: This is a clear cron/model capability gap with limited blast radius, not an urgent outage or regression.
- `impact:session-state`: The reported workaround mutates session model metadata, and the correct fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior affects provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and v2026.5.20 still only support cron job model selection on agentTurn payloads, while main-session systemEvent jobs have no supported provider/model override carrier; the only linked implementation PR was closed unmerged after review found runtime and contract issues.

## What This Changes

Request support for per-job provider/model selection on cron jobs that use the stable main-session systemEvent path.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient provider/model overrides on main-session systemEvent jobs, covering both wake modes without persisting cron-only model state into later turns.

## Reproduction Assessment

Yes. Source inspection on current main shows model selection is accepted only on agentTurn cron payloads, explicit systemEvent payloads strip or reject model fields, and main-session execution has no override carrier.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public cron/Gateway field shape and transient-versus-persisted session semantics before implementation; the closed `payload.model` attempt was not safe enough to become the permanent contract.

## Risks / Open Questions

- Maintainers still need to choose the public field shape: `payload.model`, top-level cron `model`/`provider`, or a dedicated override block.
- The override lifetime is product-sensitive: persisting a cron-selected model into main-session metadata can leak into later interactive turns, while a transient carrier must survive both `wakeMode: "now"` and `wakeMode: "next-heartbeat"`.
- Provider/model allowlist validation, fallback policy, and auth-profile selection need to match the isolated cron model contract before this is safe to automate.
