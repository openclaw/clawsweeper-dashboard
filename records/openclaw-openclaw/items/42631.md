## Summary

Keep open: current main and the latest release still do not provide a first-class model/provider override for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`; the linked exact-fix PR is closed unmerged and the remaining API shape needs maintainer product judgment.

## What This Changes

The issue requests a supported cron job model/provider override for stable main-session system-event jobs.

## Best Possible Solution

Define one supported cron override contract for main-session system events, then implement it consistently across protocol schema, normalization, persistence, CLI/API, heartbeat forwarding, validation, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: a `systemEvent` cron payload with `model` is pruned by normalization, rejected by the public schema, persisted without `payload_model`, and executed without forwarding a heartbeat model override. I did not run a live cron/model provider job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The current manual session-store workaround is not a supported solution, but the best implementation needs one canonical contract and consistent fallback/provider validation instead of another hidden context-key side channel.
