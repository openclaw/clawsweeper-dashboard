## Summary

Keep open: current main and the latest release still do not support the explicit `sessionTarget: "main"` plus `payload.kind: "systemEvent"` model/provider override requested here. The closed linked PR was not merged, and the later merged cron model fix only handles implicit text payloads by promoting them to `agentTurn`; it explicitly leaves `systemEvent` pruning in place.

## What This Changes

The issue asks for cron jobs using `sessionTarget: "main"` plus `payload.kind: "systemEvent"` to support per-job model/provider selection without manual session metadata edits.

## Best Possible Solution

Land a maintainer-approved cron override contract that validates model/provider through the existing model-selection policy, applies it to each main-session cron run without manual session-store edits, and covers both `wakeMode` paths with protocol, normalization, persistence, runtime, docs, and regression tests aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence capability reproduction: `payload.model` on an explicit `systemEvent` is not accepted by the public protocol, is stripped by normalization, and is not propagated by the main-session executor.

## Solution Assessment

Unclear. `payload.model` may be the narrowest user-facing shape, but maintainers still need to choose between payload-level, top-level job fields, or an explicit session override block before implementation.
