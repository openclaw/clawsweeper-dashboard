## Summary

Keep open: current main and the latest release still do not provide a first-class model/provider override for cron jobs using `sessionTarget: "main"` with explicit `payload.kind: "systemEvent"`. The linked implementation PR is closed unmerged, and the later merged cron model fix only covers implicit text payloads promoted to `agentTurn`, while explicit `systemEvent` payloads still prune model fields and main-session execution passes only `heartbeat: { target: "last" }`.

## What This Changes

The issue requests a supported job-level model/provider override for stable main-session cron jobs that use systemEvent payloads.

## Best Possible Solution

Land a maintainer-approved cron override contract that validates model/provider through existing model-selection policy, applies it per main-session cron run without manual session-store edits, and aligns protocol schema, normalization, persistence, runtime, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence capability reproduction: explicit systemEvent payloads cannot carry `model`, normalization deletes it, persistence does not round-trip it, and main-session cron execution does not pass a model/provider override to heartbeat.

## Solution Assessment

Unclear. `payload.model` is a plausible narrow shape, but maintainers still need to choose whether the supported contract should be payload-level, top-level job fields, or an explicit session override block with provider semantics.
