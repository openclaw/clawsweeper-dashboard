## Summary

Keep open: current main still lacks a supported model/provider override for `main` + `systemEvent` cron jobs, and the only related implementation PR is closed unmerged. The remaining work is an additive cron/Gateway API decision, not a stale or already-fixed report.

## What This Changes

Allow main-session system-event cron jobs to choose a per-job provider/model without editing internal session metadata.

## Best Possible Solution

Add an additive cron protocol/CLI/docs contract for a validated job-scoped provider/model on main-session system events, keeping the override transient to the cron-triggered turn unless maintainers intentionally choose persisted session semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `systemEvent` payloads cannot carry model/provider data, normalization removes those fields, and the main-session enqueue/wake path has no override parameter.

## Solution Assessment

Unclear. The requested capability is useful, but maintainers should choose `payload.model` versus a top-level/session override contract and transient versus persisted semantics before implementation.
