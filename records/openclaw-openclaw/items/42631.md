## Label Justifications

- `P2`: This is a clear cron/model capability gap with a source-reproducible path, but it is not an urgent outage or regression.
- `impact:session-state`: The current workaround mutates session model metadata, and the correct fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior affects provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and the latest release still lack a supported job-level provider/model override for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`; the linked implementation PR was closed unmerged after review identified runtime design problems.

## What This Changes

The issue requests a supported way for main-session `systemEvent` cron jobs to choose a per-job provider/model without manually editing session-store metadata.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient job-scoped provider/model overrides on main-session `systemEvent` jobs, covering both wake modes without persisting cron-only state into later turns.

## Reproduction Assessment

Yes, by source inspection rather than a runtime failure. Current main accepts `model` only for `agentTurn`, rejects or strips it from `systemEvent`, and has no main-session cron override carrier.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public field shape and transient-versus-persisted session semantics before implementation; the closed `payload.model` attempt was not safe enough to merge as the permanent contract.
