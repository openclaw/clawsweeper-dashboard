## Summary

Keep open: current main and latest release still do not provide a supported model/provider override for explicit `systemEvent` cron jobs on the main session. The closest merged work only preserves implicit text-payload overrides by promoting them to `agentTurn`, while this request still needs a maintainer-approved cron override API and provider/session semantics.

## What This Changes

The issue requests first-class job-level provider/model override support for cron jobs that use `sessionTarget: "main"` with explicit `payload.kind: "systemEvent"`.

## Best Possible Solution

Land a maintainer-approved cron override contract that validates provider/model through existing model-selection policy, applies it per main-session cron run without manual session-store edits, and updates protocol schema, normalization, persistence, runtime, docs, and focused regression tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence capability reproduction: explicit systemEvent payloads cannot carry `model`, normalization deletes it, persistence does not round-trip it, and main-session cron execution does not pass a model/provider override to heartbeat.

## Solution Assessment

Unclear. `payload.model` is a plausible narrow shape, but maintainers still need to choose whether the supported contract should be payload-level, top-level job fields, or an explicit session override block with provider semantics.
