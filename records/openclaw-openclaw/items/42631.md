## Summary

Keep open: current main and the latest release still lack a supported model/provider override for `sessionTarget="main"` + `payload.kind="systemEvent"` cron jobs. The directly related PR was closed unmerged after review feedback, so this remains a product/API-shape decision rather than conservative cleanup.

## What This Changes

Requested job-level model selection for cron jobs that use the main session with system-event payloads.

## Best Possible Solution

Pick one first-class cron API shape for main-session system-event model overrides, then implement it consistently through cron persistence, Gateway protocol validation, model/provider authorization, heartbeat wake/run paths, CLI/tool display, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main-session cron execution has no supported field or call path for a system-event model/provider override. I did not run a live cron job because this read-only review could establish the gap from schema and execution code.

## Solution Assessment

No, the previous proposed implementation is not clearly the best fix because its `payload.model` approach had brittle next-heartbeat handling and was closed unmerged. The best path is a maintainer-approved API shape before implementation.
