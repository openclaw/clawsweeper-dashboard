## Label Justifications

- `P2`: This is a clear cron/provider-model capability gap with limited blast radius, not an outage or active regression.
- `impact:session-state`: The current workaround mutates session model metadata, and the intended fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior affects provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and v2026.5.22 still only support cron model selection on agentTurn/isolated-style payloads; main-session systemEvent jobs have no supported job-scoped provider/model carrier, and the prior exact implementation attempt was closed unmerged after review found runtime contract problems.

## What This Changes

Request support for selecting a provider/model per cron job when the job uses sessionTarget="main" with payload.kind="systemEvent".

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient job-scoped provider/model overrides on main-session systemEvent jobs across both wake modes.

## Reproduction Assessment

Yes. Source inspection is enough: current main strips or rejects model/provider fields on systemEvent cron payloads and the main-session execution path never carries a job-scoped provider/model override.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public field shape and transient-versus-persisted session semantics before implementation.
