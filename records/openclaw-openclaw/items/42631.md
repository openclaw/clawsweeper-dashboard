## Summary

Keep open: current main and v2026.6.10 still do not provide a first-class model/provider override for cron jobs using `sessionTarget: "main"` with explicit `payload.kind: "systemEvent"`. Adjacent merged work covers isolated/agentTurn payload model behavior and UI visibility, but the requested main-session systemEvent runtime/API contract remains missing.

## What This Changes

The issue requests first-class per-cron model/provider override support for cron jobs that run on the main session using explicit systemEvent payloads.

## Best Possible Solution

Define and land a maintainer-approved cron override contract that validates model/provider through existing model-selection policy, applies it per main-session cron run without manual session-store edits, and aligns protocol schema, normalization, persistence, runtime, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence capability reproduction: explicit systemEvent payloads cannot carry `model`, normalization deletes that field, the latest release has the same schema shape, and main-session cron execution does not pass a model/provider override to heartbeat.

## Solution Assessment

Unclear. `payload.model` is a plausible narrow shape, but maintainers still need to choose whether the supported contract should be payload-level, top-level job fields, or an explicit session override block with provider and fallback semantics.
