## Summary

Keep open: current main and the latest release still treat cron `payload.model` as an isolated `agentTurn`-only field, while the linked exact-fix PR is closed unmerged and the remaining API shape needs maintainer product judgment.

## What This Changes

The issue requests a supported job-level model/provider override for `sessionTarget: "main"` cron jobs that use `payload.kind: "systemEvent"`.

## Best Possible Solution

Define one supported cron model/provider override contract for main-session system events, then implement it consistently across protocol schema, normalization, persistence, CLI/API, heartbeat forwarding, validation, docs, and focused regression tests.

## Reproduction Assessment

Yes at source level: a `systemEvent` cron payload with `model` is pruned by normalization, rejected by the public schema, persisted without `payload_model`, and executed without forwarding a heartbeat model override. I did not run a live cron/model provider job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The current manual session-store workaround is not a supported solution, but the best implementation needs one canonical contract and consistent fallback/provider validation instead of another hidden context-key side channel.
