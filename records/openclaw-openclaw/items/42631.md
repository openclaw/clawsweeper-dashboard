## Summary

Keep open: current main still has no supported model/provider override for `main` + `systemEvent` cron jobs, and the related implementation PR is closed unmerged. The remaining question is product/API semantics, not whether the gap exists.

## What This Changes

Request a supported cron job model/provider override for main-session `systemEvent` jobs so recurring work can use a cheaper model without editing session metadata.

## Best Possible Solution

Add an additive cron API/CLI/docs path for a job-scoped primary model on main-session system-event jobs, with behavior that is limited to the cron-triggered turn unless maintainers intentionally choose persisted session semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `systemEvent` payloads reject/prune model data, and the main-session cron execution path forwards no model/provider override.

## Solution Assessment

Unclear. The requested capability is reasonable, but maintainers should choose the job-scoped versus persisted-session model semantics before implementation.
