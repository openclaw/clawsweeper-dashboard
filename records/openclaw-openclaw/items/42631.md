## Label Justifications

- `P2`: This is a clear and useful cron/model capability gap, but it is not an urgent outage or regression.
- `impact:session-state`: The workaround mutates session modelOverride/providerOverride metadata, and the real fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior changes provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and v2026.5.12 still lack a supported job-level provider/model override for main-session systemEvent cron jobs, and the only exact implementation attempt was closed unmerged after review found runtime design problems.

## What This Changes

The issue requests job-level model/provider selection for cron jobs that run on the main session using systemEvent payloads.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient job-scoped provider/model overrides on main-session systemEvent jobs, covering both wake modes without leaking state into later turns.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: systemEvent payloads accept only kind/text, Gateway validation rejects extra fields, normalization strips model/provider, and main-session execution never carries an override into heartbeat execution.

## Solution Assessment

Unclear. The requested capability is useful, but maintainers should choose the public field shape and transient-versus-persisted semantics before implementation; the closed payload.model attempt was not a safe merge path as written.
