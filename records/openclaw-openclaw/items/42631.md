## Summary

Keep open. Current main still does not provide a job-scoped model or provider override for cron jobs that run as main-session system events: the public cron schema only allows `text` on `systemEvent`, normalization removes `model`/`provider` from system-event payloads, and the main-session cron wake path only forwards text plus session/heartbeat targeting. The closed related PR #42708 is useful context but was not merged, so the requested capability remains open work.

## What This Changes

Request job-level model/provider selection for stable main-session `systemEvent` cron jobs without manually editing session-store model metadata.

## Best Possible Solution

Add an additive, job-scoped model contract for main-session system-event cron jobs, preferably reusing the existing cron model-ref format and allowlist validation, then thread the resolved override transiently through both direct `wakeMode="now"` heartbeat runs and queued wake requests. Update protocol schemas/types, normalization and patch behavior, cron runtime, CLI/agent tool/Control UI/macOS surfaces as applicable, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: create or normalize a main-session `systemEvent` cron payload with `payload.model`, and current main either rejects it at the strict protocol schema or normalizes it away before the runtime path, which then has no model/provider override field to pass forward.

## Solution Assessment

Unclear for a completed fix because no patch is merged. The requested direction is sound if implemented as a transient allowlist-validated per-job override; the closed PR #42708 should be treated as design context rather than the final approach because it attempted session-store mutation.
