## Summary

Keep open: current main and the latest release still do not provide a first-class model/provider override for cron jobs using `sessionTarget: "main"` with explicit `payload.kind: "systemEvent"`. The direct implementation PR is closed unmerged after review found an unsafe wake path, while the open UI issue/PR only cover existing `agentTurn` model selection/display.

## What This Changes

Allow main-session `systemEvent` cron jobs to specify a per-job model/provider override without manually editing session metadata.

## Best Possible Solution

Land a maintainer-approved cron override contract that validates model/provider through existing model policy and applies it per main-session cron run without manual session-store edits.

## Reproduction Assessment

Yes. Source inspection shows explicit `systemEvent` payloads cannot carry `model`, normalization deletes model/provider-like fields, and main-session cron execution does not pass any model/provider override into heartbeat.

## Solution Assessment

Unclear. `payload.model` is plausible, but the closed direct PR exposed wake-path risk, so maintainers still need to choose the durable API shape and provider/fallback semantics before implementation.
