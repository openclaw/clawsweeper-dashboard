## Summary

Keep open: current main and latest release still lack a supported model/provider override for main-session cron jobs that use explicit `systemEvent` payloads. The direct implementation attempt is closed unmerged, while adjacent merged work only covers `agentTurn`/implicit text payload model selection.

## What This Changes

Request first-class job-level model/provider selection for cron jobs using `sessionTarget: "main"` with `payload.kind: "systemEvent"`.

## Best Possible Solution

Land a maintainer-approved cron override contract that validates through existing model-selection policy and aligns protocol schema, normalization, persistence, runtime, UI/docs as needed, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence capability reproduction: explicit `systemEvent` cron payloads cannot carry `model`, normalization deletes it, persistence does not round-trip it, and the main-session runtime does not pass a model/provider override to heartbeat.

## Solution Assessment

Unclear. The need is valid, but maintainers still need to choose the supported API and precedence rules before implementation.
