## Label Justifications

- `P2`: This is a clear cron/model capability gap with limited blast radius, not an urgent outage or regression.
- `impact:session-state`: The current workaround mutates session model metadata, and the intended fix must avoid leaking cron-only model state into later main-session turns.
- `impact:auth-provider`: The requested behavior affects provider/model routing, allowlist validation, fallback policy, and possibly auth-profile selection for scheduled runs.

## Summary

Keep open. Current main and the latest release still do not provide a supported provider/model override for main-session systemEvent cron jobs, and the prior linked implementation PR was closed unmerged after review found runtime contract issues.

## What This Changes

The issue requests a supported job-level provider/model override for cron jobs that run through the main session with systemEvent payloads.

## Best Possible Solution

Add an additive cron protocol, CLI, normalization, docs, and Gateway execution contract for validated transient job-scoped provider/model overrides on main-session systemEvent jobs, covering both wake modes without persisting cron-only state into later turns.

## Reproduction Assessment

Yes, by source inspection rather than a runtime failure. Current main accepts job model selection only on agentTurn payloads, rejects or strips it from systemEvent payloads, and has no main-session cron override carrier.

## Solution Assessment

Unclear. The capability is useful, but maintainers should choose the public field shape and transient-versus-persisted session semantics before implementation; the closed `payload.model` attempt was not safe enough to merge as the permanent contract.
