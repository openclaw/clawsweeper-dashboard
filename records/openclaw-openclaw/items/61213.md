## Summary

Keep open. Current main still has the same source-level gap: route bindings can carry `comment`, but the matched comment is not propagated into the resolved route, session metadata, Gateway session rows, or Control UI session labels; the related Feishu display-name PR is open and does not close this binding-comment priority request.

## What This Changes

Request that Control UI session labels prefer the matched route binding comment, then the agent name, before falling back to OpenID/session-key identifiers.

## Best Possible Solution

Add a generic session display-label contract sourced from the matched route binding comment, preserve manual session labels/display names as overrides, and keep technical keys only as the final fallback for `sessions.list` consumers.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction. Configure a Feishu direct route binding with `comment: "税务师"`, receive a DM from an `ou_...` peer, and current main has no code path from the matched binding comment into route results, session metadata, Gateway rows, or the Control UI label.

## Solution Assessment

Yes. Solving this at the route/session metadata boundary is the narrow maintainable path; a UI-only formatter cannot know which binding matched and would leave non-UI `sessions.list` consumers inconsistent.
