## Summary

Keep open. Current main has partial friendlier session presentation through Feishu sender/group labels and Control UI agent identity display, but it still does not carry a matched route binding's `comment` into session metadata or make that comment the preferred Web Control UI session label. Related #73399 is useful Feishu display-name work, but it is open and scoped to sender/group/topic fallback behavior rather than the requested binding-comment priority.

## What This Changes

Request the Web Control UI to show recognizable session names by preferring `binding.comment`, then `agent.name`, before falling back to Feishu `ou_xxx` IDs or long session keys.

## Best Possible Solution

Add a Gateway-owned session display-label contract sourced from the matched route binding comment, preserve manual session labels as explicit overrides, fall back to agent identity/name and then existing display/key behavior, and have both the Control UI sessions table and chat selector consume the same row field.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a Feishu direct route binding with `comment: "税务师"`, receive a DM from `ou_...`, and observe that current route resolution, session metadata, `sessions.list`, and the Control UI have no path to display that binding comment.

## Solution Assessment

Yes. The maintainable fix belongs at the route/session-row metadata boundary; a UI-only formatter cannot know which binding matched and would leave other `sessions.list` consumers inconsistent.
