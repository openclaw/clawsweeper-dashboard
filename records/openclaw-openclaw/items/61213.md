## Summary

Keep open. Current main still accepts route binding comments but does not propagate the matched binding comment into route results, session metadata, Gateway session rows, or the Control UI label path; related #73399 is open Feishu display-name work but does not cover the requested binding-comment priority.

## What This Changes

The issue requests Web Control UI session labels that prefer `binding.comment`, then `agent.name`, before falling back to technical OpenID/session identifiers.

## Best Possible Solution

Add a Gateway-owned session display-label contract sourced from the matched route binding comment, preserve explicit manual session labels as overrides, then fall back to agent identity/name and existing technical labels for all `sessions.list` consumers.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a Feishu direct route binding with `comment: "税务师"`, receive a DM from an `ou_...` user, and observe that current route resolution, session metadata, Gateway rows, and Control UI rendering have no path to display that comment.

## Solution Assessment

Yes. Solving this at the route/session-row metadata boundary is the narrow maintainable approach; a UI-only formatter cannot know which binding matched and would leave other `sessions.list` consumers inconsistent.
