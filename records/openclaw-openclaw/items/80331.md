## Summary

Keep open: the PR addresses a real, source-reproducible memory extraPaths symlink traversal, but it introduces a missing-path regression and still needs real behavior proof beyond the unit-test wrapper before merge.

## What This Changes

The PR exports fs-safe realpath/symlink-parent helpers, gates memory extra-path directory reads with symlink-parent and realpath containment checks, and adds readMemoryFile regression tests.

## Best Possible Solution

Keep the symlink-parent and realpath hardening, but align the missing-path branch with fs-safe's not-found contract and add a non-directory-parent regression test before merge.

## Reproduction Assessment

Yes, source-reproducible: current main lexically authorizes extra-path directory reads and only rejects a symlink at the final path, so a symlinked intermediate directory can redirect the read. I did not run tests because this review is read-only and test commands create artifacts.

## Solution Assessment

No, not as written: the hardening direction is narrow and maintainable, but the new authorization helper needs to treat ENOTDIR as a missing path and cover that case before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

- **[P2] Treat ENOTDIR as missing for extra paths:** `packages/memory-host-sdk/src/host/read-file.ts:41`
  - body: When an existing non-directory component is used as a parent, such as `<extra>/note.md/child.md`, this `fs.lstat(absPath)` fallback throws `ENOTDIR`. The existing `statRegularFile()` path treats ENOTDIR as missing via fs-safe, but this new gate returns false first and turns a missing extra-path read into `path required`; please accept ENOTDIR here and add a regression test.
