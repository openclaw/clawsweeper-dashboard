## Summary

Keep open: current main still uses a Git-common-dir heavy-check lock with no per-worktree opt-in, and the request is a focused feature/config-policy change rather than a cleanup close candidate.

## What This Changes

Allow local operators to opt into per-worktree heavy-check locks while keeping the repo-common lock as the default.

## Best Possible Solution

Define a documented local-only lock-scope contract in `scripts/lib/local-heavy-check-runtime.mjs` that keeps repo-common locks by default, allows an explicit per-worktree scope, and adds focused wrapper tests for both scopes.

## Reproduction Assessment

Yes for the current behavior by source inspection: the lock helper resolves `--git-common-dir`, and both `check:changed` and most routed test runs acquire that helper. I did not run a live linked-worktree contention demo in this read-only review.

## Solution Assessment

Unclear until maintainers accept the policy: the proposed env opt-in is narrow and preserves the safe default, but it should be aligned with the active heavy-check lock tightening work before implementation.
