## Summary

Keep open. Current main still ignores provider-level OpenAI TTS timeout in chat synthesis, and this PR is the narrow functional fix, but it needs small documentation/process follow-up before merge.

## What This Changes

The PR preserves OpenAI speech provider `timeoutMs`, resolves non-telephony TTS attempt timeouts with explicit/provider/global precedence, and adds speech-core regression tests.

## Best Possible Solution

Merge the timeout propagation fix after adding an active-version changelog entry, documenting the OpenAI provider timeout field, and rerunning focused validation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `messages.tts.providers.openai.timeoutMs` higher than `messages.tts.timeoutMs`; chat synthesis still uses the global/request timeout and OpenAI normalization drops the provider timeout.

## Solution Assessment

Yes, with documentation/process follow-up. The proposed split is narrow and maintainable because speech-core owns generic explicit/provider/global precedence while the OpenAI plugin preserves its provider-owned config field without changing telephony behavior.
