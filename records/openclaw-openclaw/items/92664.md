## Summary

Keep open: current main still decodes the active read tool's text path as UTF-8, so the reported GBK bytes remain mojibake; the related Windows codepage helper makes a narrow repair plausible, and no duplicate or implemented-on-main owner was found.

## What This Changes

The issue asks for the read tool to correctly display GBK-encoded Chinese text files on Chinese Windows instead of returning mojibake.

## Best Possible Solution

Teach the active read text path to prefer strict UTF-8 and fall back to the Windows codepage only when UTF-8 is invalid on Windows, with GBK and UTF-8 regression tests.

## Reproduction Assessment

Yes at source level: current main reads text bytes and decodes them as UTF-8, and the reported GBK byte sample reproduces mojibake under that decode. I did not run a live Chinese Windows environment in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing Windows strict-UTF-8-then-codepage fallback for text reads rather than adding a new user-facing encoding parameter first. The repair should keep valid UTF-8 behavior unchanged and avoid expanding write/edit semantics.
