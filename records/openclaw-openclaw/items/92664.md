## Summary

Keep open: current main and v2026.6.6 still decode the active read tool text path as UTF-8, so the reported GBK bytes remain mojibake. The existing Windows codepage decoder makes this a narrow queueable repair, and I found no open PR or duplicate that owns the read-tool-specific work.

## What This Changes

The issue requests that the read tool correctly display GBK-encoded Chinese text files on Chinese Windows instead of returning UTF-8 mojibake.

## Best Possible Solution

Reuse or extend the existing strict-UTF-8-then-Windows-codepage decoder for the read tool text path, keeping valid UTF-8 unchanged and adding GBK regression coverage.

## Reproduction Assessment

Yes at source level: current main converts read-tool text buffers with UTF-8, and the reported GBK bytes reproduce mojibake under that decode. I did not run a live Chinese Windows environment in this read-only review.

## Solution Assessment

Yes for the bug boundary: the narrowest maintainable repair is to reuse the existing Windows codepage-aware fallback for invalid UTF-8 reads. Adding an `encoding` parameter first would expand the tool API and should be a separate maintainer choice if needed.
