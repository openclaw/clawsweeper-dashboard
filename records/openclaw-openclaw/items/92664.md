## What This Changes

The issue asks the read tool to correctly display GBK-encoded Chinese text files on Chinese Windows instead of returning UTF-8 mojibake.

## Best Possible Solution

Review the linked implementation PRs and land one UTF-8-preserving read-tool decoder fix with focused GBK coverage, then close this issue after the merge.

## Reproduction Assessment

Yes at source level, but not as a live Windows run: current main reads non-image buffers and calls `buffer.toString("utf-8")`, and a Node probe using the issue's GBK bytes mojibakes under UTF-8 while `TextDecoder('gbk')` recovers the text.

## Solution Assessment

Yes that a core read-tool fix is needed; the best shape is a UTF-8-preserving decoder or Windows-codepage fallback with focused GBK coverage, while the two open PRs need maintainer selection and repair before this issue can close.
