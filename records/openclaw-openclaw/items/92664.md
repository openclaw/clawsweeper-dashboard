## Summary

Keep open: current main still decodes the session read-tool text buffer with UTF-8, and v2026.6.6 has the same behavior. The issue now has several open linked PRs attempting the repair, so this should remain the canonical bug until one narrow fix lands.

## What This Changes

The issue reports that the session read tool decodes GBK text files as UTF-8 on Chinese Windows, producing mojibake instead of readable Chinese text.

## Best Possible Solution

Land one narrow PR that routes the read-tool text path through the existing strict-UTF-8-then-Windows-codepage decoder, with focused GBK regression coverage and no new encoding API unless maintainers choose that separately.

## Reproduction Assessment

Yes at source level: current main converts read-tool text buffers with UTF-8, and GBK byte sequences are not valid UTF-8. I did not run a live Chinese Windows repro because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing Windows codepage-aware decoder for invalid UTF-8 read buffers. Adding a model-facing encoding parameter would be a broader API choice, not necessary for the reported bug.
