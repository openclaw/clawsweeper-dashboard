## Summary

Keep this PR open for review rather than cleanup: the final head addresses the earlier paired-reconnect blocker, but it still needs real non-loopback gateway proof and an explicit maintainer decision on broadening `gateway.auth.rateLimit` beyond failed-auth attempts.

## What This Changes

Adds a node-pairing auth-rate-limit scope, wraps WebSocket node-pairing request creation in the auth limiter, preserves paired reconnects when upgrade pairing is throttled, and adds gateway regression tests.

## Best Possible Solution

Keep the first-time node-pairing throttle only if gateway owners intentionally want `gateway.auth.rateLimit` to cover authenticated pairing writes, then update the public contract and add redacted real non-loopback gateway proof before merge.

## Reproduction Assessment

Yes, at source level: current main calls `requestNodePairing` from WebSocket node connect without a limiter, and that function enters a serialized storage lock. I did not run live proof because this review is read-only.

## Solution Assessment

No, not yet: the first-time storage-lock throttle is a plausible fix location, but reusing the documented failed-auth limiter for successful node-pairing writes needs explicit contract/docs approval or a narrower gate.
