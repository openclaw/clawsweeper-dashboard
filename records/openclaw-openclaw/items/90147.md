## Summary

Keep open: the hardening idea is useful, but the patch is not merge-ready because the new limiter can reject paired-node reconnects that current main intentionally allows to continue on their already approved surface, and the external real-behavior proof is only test-harness output.

## What This Changes

Adds a node-pairing auth-rate-limit scope, wraps WebSocket node pairing requests in the gateway auth limiter, and adds concurrent node-pairing regression coverage.

## Best Possible Solution

Throttle first-time unpaired node pairing before the storage lock while preserving paired reconnects on their approved surface, and align docs/tests with any intentional gateway.auth.rateLimit behavior change.

## Reproduction Assessment

Yes, at source level: current main calls requestNodePairing from the WS connect path without a limiter, and the PR body describes a focused concurrent-connect harness. I did not run tests because this review is read-only.

## Solution Assessment

No. The first-time unpaired throttling location is plausible, but the same wrapper also fails closed for paired-node upgrade reconnects and broadens a documented failed-auth config surface without enough upgrade proof.
