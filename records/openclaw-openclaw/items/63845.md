## Summary

Keep this open. Current main documents useful inspection, truncation, pruning, compaction, skill prompt limits, local-model probes, and plugin/workflow extension surfaces, but it still does not answer the maintainer guidance being requested: tested high-count skill/tool ceilings, trigger-collision practices, workload-specific tuning, low-end local-model reference profiles, or a clear recommendation on fine-grained versus aggregated domain tools.

## What This Changes

The issue asks for maintainer best-practice guidance on running OpenClaw with 65+ skills/tools, large finance tool outputs, local models, and possible server-side aggregation tools.

## Best Possible Solution

Publish a maintainer-backed practical guide or maintainer answer for high-count skills/tools that states whether tested ceilings exist, gives SKILL.md description-collision practices, recommends inspection and tuning patterns, and explicitly describes when a hybrid aggregation plugin or workflow-backed tool is appropriate, including traceability, output caps, latency, and composability tradeoffs.
