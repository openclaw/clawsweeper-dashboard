## Summary

Keep open: the PR targets a real security-sensitive sandbox gap, but current main has not implemented macOS Seatbelt exec isolation and this branch still has correctness, security-boundary, proof, and product-design blockers.

## What This Changes

The PR adds a macOS `sandbox-exec` `sandboxProfile` option to `exec`, inline Seatbelt profiles, command wrapping, and profile-generation tests.

## Best Possible Solution

If Seatbelt support is accepted, implement it as a fail-closed, config-owned sandbox backend or backend capability under `agents.defaults.sandbox`, with docs, regression tests, and redacted macOS live proof.

## Reproduction Assessment

Yes for the PR defects from source review: the diff shows an allow-by-default restricted profile, a non-exported helper imported by the new test, an `anyOf`-style literal union schema, and unused wrapper locals. I did not run a live macOS sandbox canary in this read-only review.

## Solution Assessment

No. A model-selected per-call `sandboxProfile`, especially with `permissive`, is not the safest boundary for host command isolation; the existing sandbox backend/config contract is the narrower maintainable integration point.
