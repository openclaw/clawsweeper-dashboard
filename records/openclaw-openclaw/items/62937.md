Full review comments:

- **[P1] Export the helper before importing it in tests:** `src/agents/bash-tools.sandbox-exec.test.ts:3`
  - body: The new test imports `getSandboxProfileString` from `bash-tools.exec-runtime.js`, but the diff adds that helper as a private function. This makes the test/typecheck fail before it can validate the profile behavior.
  - confidence: 0.98
- **[P1] Make the default profile deny file access by default:** `src/agents/bash-tools.exec-runtime.ts:131-133`
  - body: The default profile starts with `(allow default)` and never adds a broad file deny before the narrow file-read/file-write allows. That does not support the PR's claimed `/tmp`/workspace-only filesystem boundary, so `sandboxProfile: "default"` would give users a false sense of isolation.
  - confidence: 0.9
- **[P1] Keep sandbox policy out of model-controlled parameters:** `src/agents/bash-tools.exec-runtime.ts:189-195`
  - body: Adding `sandboxProfile` to the exec tool schema lets the model choose `permissive` or omit sandboxing entirely. A security boundary needs to come from operator config or the existing sandbox backend path, not from an optional model-supplied argument.
  - confidence: 0.91
- **[P1] Remove the unused wrapper or wire its profile content:** `src/agents/sandbox-exec-wrapper.ts:147-156`
  - body: This new `src` file is included by core typecheck with unused-local enforcement, but `profileContent` is assigned and never read. The same helper also returns `-f -` without providing profile stdin, so the custom-profile path is not executable as written.
  - confidence: 0.86
