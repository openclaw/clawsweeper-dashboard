Keep open for now: this PR still has blocking correctness and security-boundary problems, no real macOS behavior proof, and it conflicts with current main's config-owned sandbox backend direction.

## What This Changes

The branch adds a macOS `sandbox-exec` profile parameter to `exec`, runtime wrapping, a Seatbelt wrapper module, and profile-generation tests.

## Best Possible Solution

If Seatbelt support is accepted, implement it as an operator-owned sandbox backend or capability under `agents.defaults.sandbox`, fail closed by default, document the config contract, and prove it with redacted macOS runtime output.

## Reproduction Assessment

Yes for the PR defects from source review: the diff shows an allow-by-default Seatbelt profile, a non-exported helper imported by the new test, provider-sensitive union schema shape, and typecheck-cleanliness issues. I did not run a live macOS sandbox canary in this read-only review.

## Solution Assessment

No. A model-selected per-call `sandboxProfile`, especially with `permissive`, is not the narrowest maintainable security boundary; the current sandbox backend/config contract is the safer integration point.
