## Summary

Keep open. Current main still has the read-only temp-file open before fsync, and this PR is the narrow code fix for the Windows media offload EPERM path; the remaining blocker is the required user-facing changelog entry already noted in the prior ClawSweeper review.

## What This Changes

The PR changes `writeSavedMediaBuffer` to reopen the just-written temp media file with `r+` before calling `handle.sync()`.

## Best Possible Solution

Land the narrow `r+` handle-mode fix with a single Unreleased Fixes changelog entry and avoid broad media-store rewrites.

## Reproduction Assessment

Yes, source-reproducible: WebChat or `chat.send` attachment offload reaches `saveMediaBuffer` and `writeSavedMediaBuffer`, where current main opens the temp file read-only before `handle.sync()`. Node/libuv's Windows path makes that read-only handle a plausible `FlushFileBuffers` EPERM failure; I did not run a live Windows repro.

## Solution Assessment

Yes for the code change: `r+` is the narrowest non-truncating mode that gives the just-written temp file a read/write descriptor before fsync. The PR still needs the required changelog entry for a user-facing Windows attachment fix.
