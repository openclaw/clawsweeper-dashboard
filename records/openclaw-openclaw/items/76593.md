## Summary

Keep open. The PR targets a real current-main Windows media offload failure path and the one-line code change is the right narrow direction, but it still needs the required user-facing changelog entry before landing.

## What This Changes

The PR changes `writeSavedMediaBuffer` to reopen the just-written temp media file with `r+` before calling `FileHandle.sync()`.

## Best Possible Solution

Land the `r+` media-store fix with an `Unreleased` / `Fixes` changelog entry and targeted media-store validation, without broadening media storage behavior.

## Reproduction Assessment

Yes, at source level: invoking `saveMediaBuffer` on Windows reaches `writeSavedMediaBuffer`, which opens the temp file read-only and then calls `FileHandle.sync()`. I did not run a live Windows repro in this checkout.

## Solution Assessment

Yes. Switching the post-write sync handle from `r` to `r+` is the narrowest maintainable fix because the temp file already exists, `r+` does not truncate, and Windows flush requires write access.
