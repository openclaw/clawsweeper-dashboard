The PR replaces the iMessage RPC client's readline stdout framing with an LF-only UTF-8 buffer and adds Vitest coverage for U+2028 and batched newline-delimited responses.

## Best Possible Solution

Keep the LF-only framing direction, but make the stdout reader removable and detach it in `stop()` before retry windows, then add redacted real macOS/imsg proof showing the U+2028 response resolves.

## Reproduction Assessment

Yes. Current main uses `node:readline` on iMessage RPC stdout, and a focused Node check shows raw U+2028 splits one JSON-RPC object into multiple unparseable line events; this review did not run a live macOS/imsg reproduction.

## Solution Assessment

No as submitted. LF-only framing is the right plugin-local fix shape, but the patch needs stdout listener cleanup in `stop()` and real after-fix proof before it is the best merge candidate.
