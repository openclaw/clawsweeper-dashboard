## Summary

Keep open: current main still has the reported Android stale-job reconnect race, but this PR needs changes because its inactive-job relaunch path can race with `disconnect()` and cancel the fresh run loop; it also needs the required changelog entry.

## What This Changes

The PR changes Android `GatewaySession.connect()` to relaunch `runLoop()` when the cached job is null or inactive and adds a `GatewaySessionInvokeTest` reconnect regression test.

## Best Possible Solution

Make `GatewaySession` job ownership explicit by snapshotting and clearing the old job synchronously in `disconnect()`, then land focused reconnect coverage and a single-line changelog entry with the Android fix.

## Reproduction Assessment

Yes. The high-confidence path is a rapid Android `connect()` -> `disconnect()` -> `connect()` sequence where the old job is inactive or pending cancellation while `job` still points at it; current source and the PR’s own added test target that path.

## Solution Assessment

No. Relaunching from `connect()` when `!existingJob.isActive` addresses the original stale reference, but without changing `disconnect()` it permits the pending cancellation coroutine to read and cancel the newly assigned job; snapshot-and-clear in `disconnect()` is the narrower maintainable fix.
