## Summary

Keep open: the PR targets a real Android reconnect race, but the branch still leaves job ownership unsafe because a pending `disconnect()` cleanup can cancel a newly launched run loop, and the external contributor has not posted after-fix Android runtime proof.

## What This Changes

The PR changes Android `GatewaySession.connect()` to relaunch `runLoop()` when the cached job is null or inactive and adds a reconnect regression test.

## Best Possible Solution

Make `GatewaySession` own the run-loop job by snapshotting and clearing the old job synchronously in `disconnect()`, then keep focused reconnect coverage for the rapid reconnect path.

## Reproduction Assessment

Yes, from source inspection: a rapid Android `connect()` to `disconnect()` to `connect()` path can leave `job` non-null while inactive or pending cleanup, and current main only checks `job == null`. I did not run tests because this review is read-only.

## Solution Assessment

No. Relaunching from `connect()` when `!existingJob.isActive` fixes one stale-reference case, but without snapshotting and clearing the old job in `disconnect()` the pending cleanup coroutine can cancel the fresh job.
