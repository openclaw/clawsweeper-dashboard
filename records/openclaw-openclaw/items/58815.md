## Summary

Keep open: this is a plausible Android GatewaySession bug fix, but the current PR is not mergeable because its relaunch guard can let a pending disconnect cleanup cancel the newly launched runLoop job, and the external contributor has not posted after-fix Android runtime proof.

## What This Changes

The branch changes Android GatewaySession.connect() to start runLoop() when the cached job is null or inactive and adds a Robolectric reconnect regression test.

## Best Possible Solution

Snapshot and clear the old job synchronously in disconnect() before async cancellation, keep focused reconnect coverage, and require redacted Android runtime proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main starts runLoop() only when job == null while disconnect() cancels and clears job asynchronously, so a rapid disconnect() -> connect() path can leave desired set without an active loop. I did not run Android tests in this read-only review.

## Solution Assessment

No. Adding !existingJob.isActive in connect() fixes one stale-job case, but without snapshotting and clearing job in disconnect(), pending cleanup can cancel the fresh job and recreate the stuck state.
