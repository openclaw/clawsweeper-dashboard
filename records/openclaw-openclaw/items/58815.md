## Summary

The PR is a focused Android reconnect fix that still looks useful, but it should not merge as-is because the new inactive-job branch can still leave `desired` set with no active `runLoop()` when pending `disconnect()` cleanup cancels the fresh job.

## What This Changes

The PR changes Android `GatewaySession.connect()` to relaunch `runLoop()` when the cached job is null or inactive and adds a `GatewaySessionInvokeTest` regression test for immediate disconnect-to-connect behavior.

## Best Possible Solution

Make `GatewaySession` own the run-loop job explicitly by snapshotting and clearing the old job synchronously in `disconnect()`, then keep focused reconnect coverage for the rapid reconnect path.

## Reproduction Assessment

Yes. The high-confidence source path is a rapid Android `connect()` -> `disconnect()` -> `connect()` sequence where the old `job` is inactive or pending cancellation while `job` still points at it; current main and the PR’s test target the same path.

## Solution Assessment

No. Relaunching from `connect()` when `!existingJob.isActive` fixes one stale-reference case, but without snapshotting and clearing the old job in `disconnect()` the pending cancellation coroutine can cancel the fresh job.
