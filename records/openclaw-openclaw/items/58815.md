## Summary

Keep open: the PR targets a real Android reconnect race, but the patch is not mergeable because pending disconnect cleanup can cancel the freshly launched runLoop job, and no after-fix Android runtime proof is posted.

## What This Changes

The PR changes Android GatewaySession to relaunch runLoop when the cached job is null or inactive and adds a Robolectric immediate disconnect-to-connect regression test.

## Best Possible Solution

Snapshot and clear the old job synchronously in disconnect() before async cancellation, keep focused reconnect coverage, and require redacted Android runtime proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main leaves job non-null until an async disconnect coroutine cancels and clears it, while connect() only starts runLoop when job is null. I did not run Android tests in this read-only review.

## Solution Assessment

No. Adding !existingJob.isActive in connect() addresses one stale-job case, but without snapshotting and clearing job in disconnect(), pending cleanup can cancel the fresh job and recreate the stuck state.
