## Summary

This PR should stay open because the reported Android reconnect bug is source-reproducible, but the proposed fix leaves a job-ownership race that can cancel the freshly relaunched runLoop and the external contributor has not posted after-fix Android runtime proof.

## What This Changes

The PR changes Android GatewaySession.connect() to relaunch runLoop() when the cached job is null or inactive and adds a Robolectric immediate disconnect-to-connect regression test.

## Best Possible Solution

Snapshot and clear the old job synchronously in disconnect() before async cancellation, keep focused reconnect coverage, and require redacted Android runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main leaves job non-null until async disconnect cleanup runs, while connect() only starts runLoop when job is null; I did not run Android tests in this read-only review.

## Solution Assessment

No. Adding !existingJob.isActive in connect() handles one stale-job case, but without snapshotting and clearing job in disconnect(), pending cleanup can cancel the fresh job and recreate the stuck state.
