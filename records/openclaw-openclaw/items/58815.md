## Summary

Keep open: the PR targets a real Android reconnect race, but the current diff is not mergeable because the unchanged async disconnect cleanup can cancel the newly launched job; it also lacks required after-fix Android runtime proof.

## What This Changes

The PR changes Android `GatewaySession.connect()` to relaunch `runLoop()` when the cached job is null or inactive and adds a Robolectric immediate disconnect-to-connect regression test.

## Best Possible Solution

Snapshot and clear the old job synchronously in `disconnect()` before async cancellation, keep focused reconnect coverage, and add redacted Android runtime proof.

## Reproduction Assessment

Yes, at source level. Current main leaves `job` non-null until async disconnect cleanup while `connect()` only launches when `job == null`; I did not run Android tests in this read-only review.

## Solution Assessment

No. Adding `!existingJob.isActive` in `connect()` handles one stale-job case, but the safer fix is to snapshot and clear the old job in `disconnect()` so pending cleanup cannot cancel a fresh `runLoop()`.
