## Summary

Keep open: the PR targets a real Android GatewaySession reconnect race, but the current diff leaves a pending-disconnect job ownership race that can cancel the newly launched runLoop, and the external PR still lacks after-fix Android runtime proof.

## What This Changes

The branch updates Android GatewaySession.connect() to relaunch runLoop() when the cached job is null or inactive and adds a Robolectric reconnect regression test.

## Best Possible Solution

Make GatewaySession snapshot and clear the old job synchronously in disconnect() before async cancellation, then keep focused reconnect coverage and require redacted Android runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main connect() only starts runLoop when job == null, while disconnect() cancels and clears job asynchronously, so a rapid disconnect() -> connect() path can leave desired set without an active loop.

## Solution Assessment

No. Adding !existingJob.isActive in connect() fixes one stale-job case, but without snapshotting and clearing job in disconnect(), pending cleanup can cancel the fresh job and recreate the stuck state.
