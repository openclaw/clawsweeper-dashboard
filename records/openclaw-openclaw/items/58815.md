## Summary

Keep open: current main still has the Android GatewaySession reconnect lifecycle gap, and this PR is useful source work, but the proposed patch is not mergeable as-is because it can let a pending disconnect coroutine cancel the freshly relaunched runLoop job. The remaining repair is narrow and suitable for an automated fix pass.

## What This Changes

The PR changes Android GatewaySession.connect() to relaunch runLoop() when the cached job is missing or inactive and adds a GatewaySessionInvokeTest reconnect case for immediate disconnect-to-connect behavior.

## Best Possible Solution

Revise GatewaySession lifecycle so disconnect synchronously snapshots and clears the old job before scheduling cancellation, cancel only that snapshot, keep reconnect behavior deterministic, and add a single-line user-facing changelog entry for the Android reconnect reliability fix.

## Reproduction Assessment

Yes. A rapid connect -> disconnect -> connect sequence exercises the stale-job path; the remaining PR bug follows from the interleaving where the delayed disconnect coroutine reads `job` after connect has assigned a fresh runLoop job.

## Solution Assessment

No. Relaunching from connect when `!existingJob.isActive` addresses the original symptom but is not the narrowest safe fix unless disconnect is also changed to snapshot and clear the old job before async cancellation.
