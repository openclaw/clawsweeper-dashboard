## Summary

Keep open: https://github.com/openclaw/openclaw/pull/75773 merged useful WhatsApp 428 retry recovery and an embedded Pi resource-loader wrapper, but the dependency source still shows `DefaultResourceLoader.reload()` calling package discovery with synchronous filesystem scans before OpenClaw's skip flags take effect, and there is no post-merge Windows WhatsApp proof that the first-inbound event-loop stall is gone.

## What This Changes

The issue reports a Windows WhatsApp regression where the first inbound message blocks the Node event loop during first agent startup and stalls subsequent model calls.

## Best Possible Solution

Retest current main or the next release on Windows 11 with a linked WhatsApp account; if `session-resource-loader` still blocks, fix the embedded Pi resource-loader boundary or upstream Pi discovery so reload does not synchronously scan unused resource trees while preserving OpenClaw inline extensions.

## Reproduction Assessment

No high-confidence live reproduction exists for current main. The reporter supplied a credible release-level Windows repro, and source inspection still shows the implicated `DefaultResourceLoader.reload()` path can perform synchronous discovery, but I did not run a linked WhatsApp Windows retest.

## Solution Assessment

Unclear. https://github.com/openclaw/openclaw/pull/75773 is a useful recovery mitigation, but the dependency source suggests the resource-loader change may not be the complete fix without live Windows proof or a narrower OpenClaw/upstream loader change.
