## Best Possible Solution

Keep the WhatsApp 428 reconnect mitigation, then make embedded-agent resource loading honor the no-discovery policy before synchronous package-manager and ambient resource discovery; leave broader plugin-registry async work to https://github.com/openclaw/openclaw/issues/78100 and https://github.com/openclaw/openclaw/pull/78639.

No live current-main Windows reproduction was run. Source inspection gives a medium-confidence reproduction path: embedded-agent startup calls DefaultResourceLoader.reload, which still enters package-manager resolution and auto-discovery before no-discovery options avoid later resource loading.

## Solution Assessment

Yes for keeping this open and routing a narrow repair. The best fix is not another WhatsApp retry change, but making the embedded resource-loader no-discovery policy skip ambient package/resource discovery before synchronous filesystem walks.
