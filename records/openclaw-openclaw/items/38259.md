## Summary

Keep open: current main and the latest release still assign raw parsed dotenv values without expanding `$VAR` or `${VAR}`, and the direct fix PRs were closed unmerged. The bug is source-reproducible, but the fix needs maintainer product/security direction because expansion changes credential and workspace-env trust boundaries.

## What This Changes

The issue requests expanding `$VAR` and `${VAR}` references in auto-loaded project/global `.env` files while preserving shell environment precedence.

## Best Possible Solution

Keep this open until maintainers decide the dotenv expansion trust boundary, then implement one shared tested policy across workspace, global, gateway.env, and service-env dotenv paths.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: both current workspace and global dotenv loaders assign the raw `dotenv.parse` value directly into `process.env`, matching the reported literal `${VAR}` behavior.

## Solution Assessment

Unclear. Expansion is a plausible fix, but the best implementation needs maintainer policy for workspace trust boundaries, provider credential aliases, and state-dir service env persistence.
