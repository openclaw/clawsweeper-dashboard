## Summary

Keep open. Current main and v2026.5.12 still assign parsed dotenv values literally, and the linked expansion PR was closed unmerged; the remaining work needs maintainer policy for expansion across trusted global and untrusted workspace dotenv boundaries.

## What This Changes

Request expansion of `${VAR}` and `$VAR` references when OpenClaw loads workspace and global `.env` files while preserving shell env precedence.

## Best Possible Solution

Define and document a bounded dotenv expansion contract after workspace key filtering, then implement it with focused coverage for precedence, escaping, cycles, missing values, and global versus workspace references.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion pass.

## Solution Assessment

Unclear for automation. Expansion is plausible, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
