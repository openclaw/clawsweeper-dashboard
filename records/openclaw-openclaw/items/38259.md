## Summary

Keep open. Current main and the latest release still load dotenv values literally, the existing config-string substitution path does not expand values inside `.env` files, and the related expansion PR is closed unmerged; the remaining decision is the security/product contract for expansion across workspace, global, and compatibility dotenv sources.

## What This Changes

The issue asks OpenClaw’s dotenv loading to expand `$VAR` and `${VAR}` references in loaded `.env` files while preserving shell env precedence.

## Best Possible Solution

Adopt a maintainer-approved dotenv expansion contract, likely with separate trusted-global and untrusted-workspace rules, then add focused regressions proving precedence, missing/cyclic references, escaped dollar literals, and workspace blocklist safety.

## Reproduction Assessment

Yes for current behavior: source inspection shows loaded `.env` values are parsed and assigned literally, so `${OPENAI_API_KEY}` remains unresolved. This is not a strict existing-contract bug repro because dotenv expansion is not documented as an OpenClaw `.env` contract today.

## Solution Assessment

Unclear until maintainers choose the expansion/security contract. The narrowest maintainable path is to define which dotenv sources may expand references, then implement that policy in the shared loader with tests around precedence and blocked keys.
