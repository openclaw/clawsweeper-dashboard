## Summary

Keep open: current main and v2026.6.10 still assign raw `dotenv.parse` values into the workspace/global dotenv environment without expanding `$VAR` or `${VAR}`. The merged state-dir service-env work only skips unresolved whole-value refs, so the central request remains and needs maintainer security/product direction.

## What This Changes

Request that OpenClaw expand `$VAR` and `${VAR}` references from workspace and global `.env` files while preserving shell environment precedence.

## Best Possible Solution

Define one documented dotenv expansion policy for workspace, global, gateway fallback, config env, and service-env paths, then implement it with coverage for shell precedence, blocked workspace keys, provider credentials, and unresolved refs.

## Reproduction Assessment

Yes. Source inspection is a high-confidence path: OpenClaw parses dotenv files and assigns the parsed values directly into `process.env`, matching the reported literal `${VAR}` behavior.

## Solution Assessment

Unclear. Expansion is plausible, but the best fix needs maintainer policy for workspace trust boundaries, provider credential aliases, and service-env persistence before implementation.
