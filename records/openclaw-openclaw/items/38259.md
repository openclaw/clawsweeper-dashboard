## Summary

Keep open. Current main and the latest release still load dotenv values literally, the matching expansion PRs are closed unmerged, and the remaining decision is how expansion should behave across credential and workspace trust boundaries.

## What This Changes

Request expansion of `${VAR}` and `$VAR` references while loading project and global `.env` files without overriding shell environment values.

## Best Possible Solution

Define and document source-scoped dotenv expansion semantics, then implement expansion after filtering with focused tests for precedence, escaping, cycles, missing variables, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` files and assigns parsed values directly to `process.env`, so a value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear for automation. Dotenv expansion may be the right product behavior, but the safest maintainable solution needs maintainer-defined trusted-global versus untrusted-workspace semantics before implementation.
