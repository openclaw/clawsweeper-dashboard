## Label Justifications

- `P2`: The issue affects provider credential aliasing and startup env behavior, but it has a limited, well-scoped dotenv surface and needs policy review rather than emergency handling.
- `impact:security`: Changing dotenv expansion can move credential material across trusted global and untrusted workspace environment boundaries.
- `impact:auth-provider`: The reported failure mode is provider-key aliasing that leaves auth variables literal and can break model/provider authentication.

## Summary

Keep open. Current main and v2026.5.12 still parse and assign dotenv values literally, and the right fix needs a maintainer-defined expansion contract across trusted global dotenv, gateway.env fallback, and untrusted workspace dotenv scopes.

## What This Changes

Request support for expanding `$VAR` and `${VAR}` references in loaded project/global `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define and document source-scoped dotenv expansion semantics, then implement expansion after key filtering with focused tests for precedence, escaping, cycles, missing variables, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` files and assigns parsed values directly to `process.env`, so a value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear as an implementation path until maintainers define the expansion contract. Adding expansion in the dotenv loader is plausible, but it must be source-scoped and preserve the untrusted workspace filtering boundary.
