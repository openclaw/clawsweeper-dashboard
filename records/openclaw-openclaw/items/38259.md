## Summary

Keep open. Current main and v2026.5.12 still load dotenv values literally, the expansion PR was closed unmerged, and the remaining work needs a maintainer policy decision for expansion across workspace, global, and shell credential boundaries.

## What This Changes

Expand `$VAR` and `${VAR}` references in loaded `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define and document source-scoped dotenv expansion semantics, then implement expansion after filtering with focused tests for precedence, escaping, cycles, missing vars, and credential-boundary cases.

## Reproduction Assessment

Yes. Source inspection on current main shows parsed dotenv values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear for automation. Expansion may be the right product behavior, but a blind expansion pass is not the best fix until maintainers define trusted-global versus untrusted-workspace semantics.
