## Label Justifications

- `P2`: The limitation can break provider-key alias configurations, but it affects a bounded setup pattern and needs design review rather than emergency action.
- `impact:security`: Interpolation could cross the deliberate trust boundary between untrusted workspace dotenv entries and protected shell/global secrets.
- `impact:auth-provider`: Unexpanded credential aliases remain literal and can cause provider authentication failures.

## Summary

Current main still leaves dotenv references literal, so the reported limitation remains; however, the loader now deliberately treats workspace `.env` as untrusted and blocks provider credentials and runtime-control variables, making blanket interpolation a security-boundary and product-semantics decision rather than a safe mechanical fix.

## What This Changes

Request expansion of `$VAR` and `${VAR}` references in automatically loaded project and global `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define and document a constrained interpolation contract that preserves shell precedence and escaped literals while preventing workspace files from resolving protected shell/global secrets; then add focused tests for same-file chains, missing values, cycles, quoting, and separate trusted-global versus untrusted-workspace behavior.

## Reproduction Assessment

Yes, from source with high confidence: current main parses dotenv files and directly assigns parsed strings without expansion, so the reported reference remains literal. No artifact-producing runtime test was run because this review is read-only.

## Solution Assessment

No, blanket recursive expansion is not yet the best solution because it can conflict with the newer workspace dotenv security boundary. A constrained, explicitly documented lookup policy is safer and more maintainable.
