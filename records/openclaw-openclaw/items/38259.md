## Summary

Keep open: current main and the latest release still load dotenv values literally, the linked implementation PR was closed unmerged, and a blanket fix needs maintainer product/security judgment because workspace `.env` files are intentionally untrusted.

## What This Changes

The issue requests `${VAR}` and `$VAR` reference expansion in OpenClaw-loaded `.env` files while preserving existing shell environment precedence.

## Best Possible Solution

Keep this open for a maintainer-approved dotenv expansion policy, likely separating trusted global state-dir dotenv from untrusted workspace dotenv and adding regression tests for precedence and secret-boundary bypasses.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: put a value such as `ALIAS=${BASE}` in a loaded `.env` file and the current loaders will assign the parsed literal string because there is no expansion step. I did not run a live startup reproduction in this read-only review.

## Solution Assessment

Unclear as a blanket change: current main does not solve the request, but applying expansion uniformly to workspace and global dotenv is not clearly the best solution because workspace dotenv is a documented security boundary. The safer direction is a maintainer-approved scoped policy with tests for shell precedence and secret-copy bypasses.
