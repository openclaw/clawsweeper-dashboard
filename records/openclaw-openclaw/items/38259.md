## Summary

Keep open. Current main and the latest release still load parsed dotenv values literally, the related implementation PR is closed unmerged, and expansion needs a maintainer decision around trusted global dotenv files versus untrusted workspace dotenv files.

## What This Changes

Request dotenv `.env` loading to expand `$VAR` and `${VAR}` references while preserving shell, CWD, and global env precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract after workspace key filtering, with tests and docs for precedence, same-file and cross-file references, missing values, escaping, cycles, and blocked workspace keys.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal on current main.

## Solution Assessment

Unclear as an automated repair. Expansion is a plausible improvement, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics instead of a narrow provider-key alias workaround.
