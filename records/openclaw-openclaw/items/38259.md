## Label Justifications

- `P2`: The request affects provider credential aliasing and startup env behavior, but the surface is bounded and needs policy review rather than emergency handling.
- `impact:security`: Changing dotenv expansion can move credential material across trusted global and untrusted workspace environment boundaries.

## Summary

Keep open. Current main and the latest release still load `.env` values literally, and adding expansion needs a maintainer security/product decision because it crosses trusted global, compatibility `gateway.env`, and untrusted workspace `.env` sources.

## What This Changes

Allow `.env` values such as `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` to expand `$VAR` and `${VAR}` references while preserving existing environment precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after key filtering with focused tests for precedence, escaping, cycles, missing references, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` entries and assigns parsed values directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. Adding expansion in the dotenv loader is plausible, but it must be source-scoped and preserve the untrusted workspace filtering boundary.
