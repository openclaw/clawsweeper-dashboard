## Summary

Keep open. Current main still loads `.env` values through `dotenv.parse` and assigns accepted values directly to `process.env`, while the existing docs/tests only cover config-string substitution after env loading; the related implementation PR #38254 is closed unmerged, so the requested behavior remains unresolved.

## What This Changes

Add expansion of `$VAR` and `${VAR}` references in loaded project/global `.env` values while preserving existing environment precedence.

## Best Possible Solution

Define documented dotenv-expansion semantics that preserve existing precedence and workspace trust boundaries, then implement the helper in the loader with regression coverage for expansion order, escaping, missing variables, cycles, and blocked-key interactions.

## Reproduction Assessment

Yes. By source inspection, the issue’s two-line `.env` case follows `readDotEnvFile` into raw `process.env[key] = value` assignments, so `${OPENAI_API_KEY}` is not resolved on current main.

## Solution Assessment

Unclear until maintainers confirm the exact workspace `.env` lookup policy. The narrow implementation belongs in the dotenv loader, but expansion from shell env in untrusted workspace files is a trust-boundary decision, not only a mechanical parser change.
