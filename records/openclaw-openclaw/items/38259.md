## Summary

Keep open. Current main and v2026.5.20 still parse dotenv values literally; adding expansion is a bounded but security-sensitive behavior choice because workspace, global, and compatibility env sources have different trust levels.

## What This Changes

Requests dotenv loading to expand `$VAR` and `${VAR}` references in project/global `.env` files while preserving shell-env precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after per-source key filtering with focused precedence, escaping, cycle, missing-reference, and credential-boundary tests.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` entries and assigns the parsed value directly to `process.env`, so an alias such as `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. A loader-level implementation is plausible, but it needs source-scoped semantics that preserve workspace filtering and existing precedence.
