## Summary

Keep open: current main and the latest release still load `.env` values as literal `dotenv.parse` strings, while the direct expansion PRs were closed unmerged and the later merged state-dir fix only skips unresolved service-env references. The request is useful, but it changes security-sensitive env and credential semantics, so it needs maintainer product/security direction before another fix PR.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion while loading workspace and global `.env` files without changing shell environment precedence.

## Best Possible Solution

Decide the dotenv expansion policy first, then implement one shared, tested helper that preserves process-env precedence and applies workspace security filtering before any expanded value can affect provider auth or runtime controls.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: workspace and global dotenv readers assign raw parsed values, and the latest release has the same behavior; I did not run a runtime repro because this review is read-only.

## Solution Assessment

Unclear. Direct expansion is plausible, but the best fix depends on maintainer policy for workspace trust, trusted global env, service env generation, and existing config substitution/SecretRef alternatives.
