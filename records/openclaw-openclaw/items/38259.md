## Summary

Keep open. Current main and v2026.5.12 still load parsed dotenv values literally, the related expansion PR was closed unmerged, and a correct fix needs explicit credential/trust-boundary semantics rather than a blind expansion pass.

## What This Changes

Request dotenv loading to expand `$VAR` and `${VAR}` references in project and global `.env` files while preserving shell env precedence.

## Best Possible Solution

Define and document bounded dotenv expansion semantics after workspace key filtering, then implement them with focused coverage for precedence and credential-boundary cases.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion pass.

## Solution Assessment

Unclear for automation. Expansion is plausible, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
