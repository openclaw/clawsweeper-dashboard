## Summary

Keep open. Current main and the latest release still assign parsed dotenv values literally, and the linked expansion PR was closed unmerged; the remaining work needs maintainer policy on expansion across workspace, global, gateway.env, and shell credential boundaries.

## What This Changes

The issue requests expanding `${VAR}` and `$VAR` references inside loaded `.env` files while preserving existing shell environment precedence.

## Best Possible Solution

Define source-scoped dotenv expansion semantics, document them, then implement expansion after filtering with focused coverage for precedence, escaping, cycles, missing vars, and credential-boundary cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: parsed dotenv values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear for automation. Expansion may be the right product behavior, but the best maintainable fix requires maintainer-defined trust-boundary and ordering rules before changing startup credential propagation.
