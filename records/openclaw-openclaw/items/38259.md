## Summary

Keep open. Current main and v2026.5.12 still load parsed dotenv values literally, the linked implementation PRs were closed unmerged, and adding expansion needs maintainer-defined trust-boundary semantics rather than a blind expansion pass.

## What This Changes

Request `.env` loading to expand `${VAR}` and `$VAR` references while preserving shell environment precedence.

## Best Possible Solution

Define and document bounded dotenv expansion semantics per source trust level, then implement expansion after filtering with focused tests for precedence, escaping, cycles, and credential-boundary cases.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion pass.

## Solution Assessment

Unclear for automation. Expansion may be the right product behavior, but a maintainable fix needs explicit trusted-global versus untrusted-workspace semantics before changing credential propagation.
