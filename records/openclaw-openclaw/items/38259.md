## Summary

Keep open. Current main and the latest release still load `.env` references literally; the request remains valid, but the fix needs a maintainer-defined security/product contract before changing workspace, global, and compatibility dotenv behavior.

## What This Changes

The issue requests expanding `$VAR` and `${VAR}` references when OpenClaw loads project and global `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after per-source key filtering with focused precedence, escaping, cycle, missing-reference, and credential-boundary tests.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` entries and assigns the parsed value directly to `process.env`, so an alias like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. Loader-level expansion is plausible, but an unscoped expansion pass would be risky unless it preserves workspace filtering, existing precedence, and documented trust boundaries.
