## Summary

Keep open. Current main and the latest release still load `.env` references literally, the linked implementation PRs are closed unmerged, and the remaining work needs maintainer policy around dotenv expansion across trusted global files and untrusted workspace files.

## What This Changes

Request expansion of `${VAR}` and `$VAR` references in loaded project and global `.env` files while preserving shell-env precedence.

## Best Possible Solution

Define and document a bounded dotenv expansion contract after workspace key filtering, then implement it with focused tests for precedence, cross-file references, escaping, cycles, missing values, and blocked keys.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal.

## Solution Assessment

Unclear as an automated repair. Expansion is a plausible improvement, but the best solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
