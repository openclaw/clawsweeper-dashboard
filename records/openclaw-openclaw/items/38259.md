## Summary

Keep open. Current main still loads dotenv values literally, and the prior fix attempts are closed unmerged, so the remaining work is a real dotenv-loader contract decision rather than cleanup-close material.

## What This Changes

The issue asks OpenClaw to expand `${VAR}` and `$VAR` references when loading project and global `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define a bounded expansion contract in `src/infra/dotenv.ts` after key filtering, with regression coverage for trusted global, untrusted workspace, same-file, chained, shell-precedence, missing-var, escaped-dollar, and blocked-key cases.

## Reproduction Assessment

Yes, source-reproducible: current main parses `.env` files and assigns the parsed values without an expansion step, so `${OPENAI_API_KEY}` remains a literal value. I did not run the Vitest repro because this review is read-only and the package manager/dependencies are unavailable in the checkout.

## Solution Assessment

Unclear. Dotenv expansion is plausible, but maintainers should first choose the trusted-global versus untrusted-workspace expansion rules; a narrow loader contract is safer than provider-specific alias handling.
