## Summary

Keep open. Current main and the latest release still load dotenv values literally, both related fix PRs are closed unmerged, and adding `$VAR` / `${VAR}` expansion would change the env-loading contract around security-filtered workspace `.env` files.

## What This Changes

The issue asks OpenClaw to expand `$VAR` and `${VAR}` references while loading project and global `.env` files without changing shell-environment precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract in `src/infra/dotenv.ts` after key filtering, with tests and docs for shell/CWD/global precedence, same-file and chained references, missing values, escaping, cycles, and blocked workspace keys.

## Reproduction Assessment

Yes, source-reproducible: current main parses `.env` files with `dotenv.parse`, stores parsed values unchanged, and assigns them directly to `process.env`. A line like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` therefore remains literal.

## Solution Assessment

Unclear. Expansion is a plausible improvement, but the best fix needs an explicit trusted-global versus untrusted-workspace policy instead of a provider-specific alias workaround.
