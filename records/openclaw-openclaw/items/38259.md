## Summary

Keep open. Current main and v2026.5.12 still assign parsed dotenv values literally, both expansion PRs are closed unmerged, and adding expansion safely needs maintainer-defined semantics across untrusted workspace dotenv, trusted global/gateway.env files, and shell credentials.

## What This Changes

The issue asks OpenClaw's `.env` loader to expand `${VAR}` and `$VAR` references while preserving shell environment precedence.

## Best Possible Solution

Define source-scoped dotenv expansion semantics, then implement expansion after filtering with focused docs and tests for precedence, escaping, cycles, missing refs, cross-file refs, and credential-boundary cases.

## Reproduction Assessment

Yes, by source inspection rather than an executed local repro. Current main parses `.env` content and assigns parsed values directly, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion layer.

## Solution Assessment

Unclear for automation. Expansion may be the right product behavior, but a blind expansion pass over all dotenv sources is not the best fix until maintainers define trusted-global versus untrusted-workspace semantics.
