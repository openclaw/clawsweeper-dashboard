## Summary

Keep open: current main and the latest release still load dotenv values literally, and the linked expansion PRs were closed without merging. The requested expansion is plausible, but it changes security-sensitive credential/env loading behavior and needs maintainer product/security review before implementation.

## What This Changes

Request that OpenClaw expand `$VAR` and `${VAR}` references in project and global `.env` files while preserving shell environment precedence.

## Best Possible Solution

Keep this open until maintainers decide the dotenv expansion policy, then implement one shared, tested helper that preserves process-env precedence and applies workspace security filtering before any value can affect provider auth or runtime controls.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: both workspace and global dotenv readers parse values and assign the raw parsed string, and the latest release has the same behavior. I did not run a runtime repro because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. The closed expansion PRs show a plausible direction, but the best fix needs a maintainer decision on whether expansion applies to workspace `.env`, trusted global `.env`, service env generation, or only a narrower trusted subset.
