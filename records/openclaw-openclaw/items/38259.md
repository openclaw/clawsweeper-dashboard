## Summary

Keep open. Current main still loads dotenv values literally, the latest release behaves the same, and both related expansion PRs are closed unmerged; the remaining decision is how to add expansion without weakening workspace `.env` trust boundaries.

## What This Changes

Request `$VAR` / `${VAR}` expansion when OpenClaw loads project and global `.env` files while preserving shell precedence.

## Best Possible Solution

Define a bounded expansion contract in the dotenv loader after key filtering, with tests and docs for shell/CWD/global precedence, same-file and chained references, missing values, escaping, cycles, and blocked workspace keys.

## Reproduction Assessment

Yes. Source inspection of current main shows `.env` files are parsed with `dotenv.parse` and the parsed string values are assigned directly to `process.env`, so an alias like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal.

## Solution Assessment

Unclear. Expansion is a plausible improvement, but the maintainable path needs an explicit trusted-global versus untrusted-workspace policy rather than a narrow provider-key alias workaround.
