## Label Justifications

- `P2`: The issue affects provider credential aliasing and startup env behavior, but the surface is limited and needs policy review rather than emergency handling.
- `impact:security`: Changing dotenv expansion can move credential material across trusted global, shell, and untrusted workspace environment boundaries.
- `impact:auth-provider`: The reported failure mode leaves provider-key aliases literal and can break model/provider authentication.

## Summary

Keep open: current main and the latest release still parse dotenv values literally, the linked fix PR is closed unmerged, and implementation needs a maintainer-defined expansion/security contract before a repair lane chooses semantics.

## What This Changes

The issue requests resolving `${VAR}` and `$VAR` references in project and global `.env` values while preserving shell environment precedence.

## Best Possible Solution

Define and document source-scoped dotenv expansion semantics, then implement expansion after workspace key filtering with tests for precedence, escaping, cycles, missing references, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` files and assigns parsed values directly to `process.env`, so a value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. Adding expansion in the dotenv loader is plausible, but it must be source-scoped and preserve the untrusted workspace filtering boundary.
