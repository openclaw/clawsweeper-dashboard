## Summary

Keep open. Current main still loads parsed dotenv values literally, and the related fix PR was closed without merging; however the requested expansion changes credential and workspace `.env` trust semantics, so it should not be auto-closed or blindly revived without maintainer scope.

## What This Changes

Request dotenv loading to expand `${VAR}` and `$VAR` references in project/global `.env` files while preserving process environment precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract in `src/infra/dotenv.ts`, preserving process-env precedence and adding regression/security coverage for global, CWD, same-file, chained, missing, and blocked-key cases.

## Reproduction Assessment

Yes, source-reproducible: the current loader parses dotenv files and assigns raw parsed values directly, so `${OPENAI_API_KEY}` remains literal. I did not run a temp-file repro because this review was read-only.

## Solution Assessment

Unclear. Expansion is a plausible direction, but the old closed PR predates later workspace `.env` hardening, so maintainers should first choose the trusted/global vs untrusted/workspace expansion rules.
