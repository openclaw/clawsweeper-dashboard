## Summary

Keep open. Current main and v2026.5.12 still load dotenv values literally; expansion remains useful but security-sensitive because workspace, global, and shell dotenv sources cross credential boundaries.

## What This Changes

The issue requests expanding `$VAR` and `${VAR}` references while loading project and global `.env` files without changing shell environment precedence.

## Best Possible Solution

Define source-scoped dotenv expansion semantics, then implement expansion after the appropriate filtering step with docs and regression coverage for precedence, escaping, cycles, missing references, and credential-boundary cases.

## Reproduction Assessment

Yes. Source inspection on current main shows parsed dotenv values are assigned directly into `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion layer.

## Solution Assessment

Unclear. Expansion is a plausible product direction, but a blind expansion pass is not the best fix until maintainers decide source visibility, filtering order, escaping, missing values, and cycle behavior.
