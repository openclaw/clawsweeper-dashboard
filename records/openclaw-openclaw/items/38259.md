## Summary

Keep open. Current main and the latest release still load dotenv references literally, and the remaining expansion contract crosses the trusted global dotenv versus untrusted workspace dotenv boundary.

## What This Changes

The issue requests expansion of `$VAR` and `${VAR}` references while loading project and global `.env` files without changing shell environment precedence.

## Best Possible Solution

Define and document a bounded dotenv expansion contract after workspace key filtering, then implement it with focused tests for precedence, escaping, cycles, missing values, and global versus workspace references.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: parsed `.env` values are assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal.

## Solution Assessment

Unclear as an automated repair. Expansion is a plausible improvement, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
