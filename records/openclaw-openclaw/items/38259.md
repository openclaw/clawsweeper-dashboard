## Label Justifications

- `P2`: The issue affects provider credential aliasing and startup env behavior, but the surface is bounded and needs policy review rather than emergency handling.
- `impact:security`: Changing dotenv expansion can move credential material across trusted global and untrusted workspace environment boundaries.
- `impact:auth-provider`: The reported failure mode leaves provider-key aliases literal and can break model/provider authentication.

## Summary

Keep open. Current main and v2026.5.19 still load `.env` values literally, and adding expansion remains security-sensitive because it crosses trusted global, `gateway.env`, and untrusted workspace `.env` scopes.

## What This Changes

The issue requests expansion of `${VAR}` and `$VAR` references in loaded `.env` values while preserving existing shell environment precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after key filtering with focused tests for precedence, escaping, cycles, missing references, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` files and assigns parsed values directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. Adding expansion in the dotenv loader is plausible, but it must be source-scoped and preserve the untrusted workspace filtering boundary.
