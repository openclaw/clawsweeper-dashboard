## Label Justifications

- `P2`: The issue can break provider credential aliasing, but the surface is bounded and needs policy/security review rather than emergency handling.
- `impact:security`: Changing dotenv expansion can move credential values across trusted global and untrusted workspace environment boundaries.
- `impact:auth-provider`: The reported failure mode leaves provider-key aliases literal and can cause provider authentication to fail.

## Summary

Keep open. Current main and v2026.5.20 still assign parsed dotenv values literally, the related implementation PRs are closed unmerged, and adding expansion needs a maintainer security/product decision because it crosses workspace, global, and compatibility dotenv trust boundaries.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion when loading workspace and global `.env` files while preserving process environment precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after per-source key filtering with focused precedence, escaping, cycle, missing-reference, and credential-boundary tests.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` entries and assigns the parsed value directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without an expansion layer.

## Solution Assessment

Unclear until maintainers define the expansion contract. A loader-level implementation is plausible, but it needs source-scoped semantics that preserve workspace filtering and existing precedence.
