## Summary

Keep open. Current main still parses and assigns dotenv values literally, the linked fix PR was closed unmerged, and the remaining work needs a maintainer decision on trusted global versus untrusted workspace `.env` expansion semantics.

## What This Changes

Allow OpenClaw-loaded `.env` files to expand `$VAR` and `${VAR}` references while preserving process environment precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract in `src/infra/dotenv.ts`, preserving process-env precedence and adding coverage for global, CWD, same-file, chained, missing, and blocked-key cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `.env` value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` is parsed and assigned unchanged because the loader has no expansion step.

## Solution Assessment

Unclear. Expansion is plausible, but maintainers should first choose trusted global versus untrusted workspace expansion rules and whether missing references stay literal, empty, or fail closed.
