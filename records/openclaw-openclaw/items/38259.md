## Summary

Keep open: current main and the latest release still load raw `dotenv.parse` values into the workspace/global dotenv environment without expanding `$VAR` or `${VAR}`. The related merged service-env PR only skips unresolved whole-value refs in the state-dir service-env path, so the central requested expansion behavior still needs maintainer security/product direction.

## What This Changes

Expand `$VAR` and `${VAR}` references in OpenClaw-loaded `.env` files while preserving existing environment precedence and credential safety.

## Best Possible Solution

Define one documented dotenv expansion policy for workspace, global, gateway fallback, and service-env paths, then implement it with regression coverage for shell precedence, blocked workspace keys, provider credentials, and unresolved-reference handling.

## Reproduction Assessment

Yes. Source inspection is a high-confidence reproduction path: the workspace and global dotenv loaders assign raw parsed values directly into `process.env`, matching the reported literal `${VAR}` behavior.

## Solution Assessment

Unclear. Expansion is a plausible solution, but the best fix needs maintainer policy for workspace trust boundaries, provider credential aliases, and state-dir service-env persistence before implementation.
