## Summary

Keep open: current main still loads workspace/global dotenv values as raw `dotenv.parse` output, and the merged state-dir service-env work only skips unresolved whole-value refs instead of expanding aliases. The central request remains product/security-sensitive because expansion can change credential and runtime-control materialization.

## What This Changes

Request dotenv loading to expand `$VAR` and `${VAR}` references in loaded `.env` files while preserving shell environment precedence.

## Best Possible Solution

Define one documented dotenv expansion policy across workspace, global, gateway fallback, config env, and service-env paths, then implement it with coverage for shell precedence, workspace blocklists, provider credentials, and unresolved refs.

## Reproduction Assessment

Yes. Source inspection is a high-confidence path: OpenClaw parses dotenv files and assigns parsed values directly into `process.env`, matching the reported literal `${VAR}` behavior.

## Solution Assessment

Unclear. Expansion is plausible, but the best fix needs maintainer policy for workspace trust boundaries, provider credential aliases, and service-env persistence before implementation.
