## Label Justifications

- `P2`: The issue can break provider credential aliasing, but the surface is bounded and requires policy/security review rather than emergency handling.
- `impact:security`: Dotenv expansion can change how credential values flow between trusted process/global env and untrusted workspace `.env` sources.
- `impact:auth-provider`: The reported literal alias can leave provider keys unresolved and cause provider authentication failures.

## Summary

Keep open. Current main and the latest release still load parsed dotenv values literally, the two expansion PRs are closed unmerged, and implementing this safely needs a maintainer decision on expansion semantics across workspace, global, and credential trust boundaries.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion for values loaded from workspace and global `.env` files while preserving existing process environment precedence.

## Best Possible Solution

Define and document source-scoped `.env` expansion semantics, then implement expansion after per-source key filtering with focused precedence, escaping, cycle, missing-reference, and credential-boundary tests.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` entries and assigns the parsed string directly to `process.env`, so a value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` stays literal without an expansion pass.

## Solution Assessment

Unclear until maintainers define the expansion contract. A loader-level implementation is plausible, but it needs source-scoped semantics that preserve workspace filtering, current precedence, and credential boundaries.
