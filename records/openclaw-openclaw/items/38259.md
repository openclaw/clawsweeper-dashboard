## Summary

Keep open. Current main still loads dotenv values literally, the two related fix PRs were closed unmerged, and adding expansion is a new env-loading contract that needs maintainer policy because workspace `.env` files are deliberately security-filtered.

## What This Changes

Allow `${VAR}` and `$VAR` references in project or global `.env` files to expand while preserving existing shell and dotenv precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract in `src/infra/dotenv.ts` after key filtering, with tests and docs for shell/CWD/global precedence, same-file and chained references, missing values, escaping, cycles, and blocked workspace keys.

## Reproduction Assessment

Yes, source-reproducible: current main parses `.env` files with `dotenv.parse` and assigns parsed values directly, so references like `${OPENAI_API_KEY}` remain literal. I did not run a live temp-file test because the review is read-only and source plus dependency behavior is explicit.

## Solution Assessment

Unclear. Expansion is a plausible product improvement, but the best implementation needs an explicit trusted-global versus untrusted-workspace policy instead of a provider-specific alias workaround.
