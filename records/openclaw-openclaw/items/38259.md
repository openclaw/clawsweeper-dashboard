## Label Justifications

- `P2`: The issue can break provider-key alias setups, but the affected surface is bounded and needs design review rather than emergency handling.
- `impact:security`: The requested interpolation behavior could cross the lower-trust workspace dotenv boundary into protected credentials or runtime controls if scoped incorrectly.
- `impact:auth-provider`: Literal credential aliases such as `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` can leave provider auth variables unusable at runtime.

## Summary

Keep open: current main and v2026.6.5 still load dotenv values literally, and the linked implementation PR is closed unmerged. The remaining work is a core env-loading product/security decision because workspace `.env` is now deliberately lower trust.

## What This Changes

The issue requests `$VAR` and `${VAR}` expansion when OpenClaw loads workspace and global `.env` files while preserving existing process-env precedence.

## Best Possible Solution

Define and document a constrained interpolation contract that preserves precedence and escaped literals while preventing workspace `.env` from resolving protected credentials or runtime-control values, then cover both trusted-global and untrusted-workspace cases in focused tests.

## Reproduction Assessment

Yes. Source inspection shows both dotenv loaders parse values and assign the parsed strings directly, so `${OPENAI_API_KEY}` remains literal without needing an artifact-producing runtime test.

## Solution Assessment

No. Blanket expansion is only a plausible fix; the best fix needs a maintainer-approved lookup policy that separates trusted global dotenv behavior from untrusted workspace dotenv behavior.
