## Summary

Keep open: current main and the latest release still load dotenv values literally, and the linked attempted fix at https://github.com/openclaw/openclaw/pull/38254 was closed unmerged. The requested behavior is real, but the project/global dotenv split is security-sensitive enough that maintainers should choose the expansion boundary before a new fix PR.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion when OpenClaw loads project and global `.env` files while preserving process-env precedence.

## Best Possible Solution

Define a narrow dotenv expansion contract that preserves shell precedence and the workspace trust boundary, then add focused loader tests and docs for global/workspace precedence, escaping, cycles, and credential safety.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.5` parse dotenv files and assign parsed values directly, so `${OPENAI_API_KEY}` remains literal. I did not run a mutating local repro command in this read-only review.

## Solution Assessment

Unclear. Expansion is a plausible fix, but applying it to both project and global `.env` files needs a maintainer security/product decision because workspace `.env` is intentionally lower trust.
