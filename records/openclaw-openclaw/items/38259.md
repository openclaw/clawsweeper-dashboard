## Summary

The issue remains necessary: current main and v2026.6.6 still assign parsed dotenv strings verbatim, while newer workspace `.env` hardening makes unrestricted process-environment interpolation unsafe without an explicit source-trust decision.

## What This Changes

Add `$VAR` and `${VAR}` interpolation to automatically loaded dotenv values while preserving existing environment precedence.

## Best Possible Solution

Define source-aware interpolation: trusted global dotenv files may resolve intentional aliases while preserving pre-existing values, but workspace dotenv must use a restricted lookup that cannot read blocked process secrets; cover precedence, chaining, cycles, escaping, missing references, and indirect-secret cases.

## Reproduction Assessment

Yes for the reported current behavior: both loaders assign parsed dotenv values verbatim, so `$VAR` and `${VAR}` references remain literal on current main. This is source-reproducible without running a write-producing test command.

## Solution Assessment

No, the closed PR should not be adopted unchanged because its unrestricted base-environment lookup predates workspace dotenv hardening. The best fix is a source-aware expansion contract approved for the current trust model.
