## Summary

Keep open. Current main and the latest release still load dotenv values literally, and the linked expansion PR was closed unmerged; the remaining work needs a maintainer decision on credential-boundary semantics.

## What This Changes

Request dotenv loading to expand `${VAR}` and `$VAR` references in loaded `.env` files while preserving shell environment precedence.

## Best Possible Solution

Define and document source-scoped dotenv expansion semantics, then implement expansion after filtering with tests for precedence, escaping, cycles, missing vars, and credential-boundary cases.

## Reproduction Assessment

Yes, at source level. Current main parses `.env` files and assigns parsed values directly to `process.env`, so a value like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal without another expansion layer.

## Solution Assessment

Unclear for automation. Expansion may be the right product behavior, but the safest maintainable solution needs maintainer-defined trusted-global versus untrusted-workspace semantics before implementation.
