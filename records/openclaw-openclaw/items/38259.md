## Summary

Keep open. Current main and v2026.5.12 still load dotenv references literally, and the useful closed proposal at https://github.com/openclaw/openclaw/pull/38254 was never merged; adding expansion still needs a clear trusted-global versus untrusted-workspace policy.

## What This Changes

The issue asks OpenClaw to expand `$VAR` and `${VAR}` references in project and global `.env` files while preserving environment precedence.

## Best Possible Solution

Define and document a bounded dotenv expansion contract after workspace key filtering, then implement it with focused tests for precedence, escaping, cycles, missing values, and global versus workspace references.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: parsed `.env` values are assigned directly to `process.env`, so the reported alias remains literal.

## Solution Assessment

Unclear for automation. Expansion is plausible, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
