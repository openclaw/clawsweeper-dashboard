## Summary

Keep open. Current main still loads parsed dotenv values literally, both related fix PRs were closed unmerged, and the remaining work needs a maintainer-approved expansion contract because workspace `.env` loading is security-filtered.

## What This Changes

The issue requests expanding `$VAR` and `${VAR}` references in OpenClaw-loaded `.env` files while preserving shell environment precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract in `src/infra/dotenv.ts` after key filtering, preserving process/CWD/global precedence and documenting same-file, chained, missing, escaped, and blocked-key behavior.

## Reproduction Assessment

Yes, source-reproducible: current main parses `.env` files with `dotenv.parse` and assigns the parsed values directly to `process.env`, so `${OPENAI_API_KEY}` remains literal. I did not run a live temp-file repro because this review is read-only.

## Solution Assessment

Unclear. The requested expansion is plausible, but the maintainable fix needs an explicit trusted-global versus untrusted-workspace policy rather than a provider-specific alias workaround.
