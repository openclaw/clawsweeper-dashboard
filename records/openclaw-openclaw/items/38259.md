## Summary

Keep open. Current main and the latest release still assign parsed dotenv values literally, and the only linked expansion PRs are closed unmerged; a safe fix still needs maintainer policy for the trusted global versus untrusted workspace dotenv boundary.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion when OpenClaw loads project and global `.env` files while preserving existing environment precedence.

## Best Possible Solution

Define and document a bounded dotenv expansion contract after workspace key filtering, then implement it with focused coverage for precedence, escaping, cycles, missing values, and global versus workspace references.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: parsed `.env` values are assigned directly to `process.env`, so the reported alias remains literal without an expansion pass.

## Solution Assessment

Unclear for automation. Expansion is plausible, but the maintainable solution needs explicit trusted-global versus untrusted-workspace semantics rather than a narrow provider-key alias workaround.
