## Summary

Keep open. Current main still parses loaded `.env` files with `dotenv.parse` and assigns accepted values directly to `process.env`, so dotenv-file aliases remain literal; existing docs/tests cover config-string substitution after env loading, not `.env` reference expansion. The related implementation PR #38254 is closed unmerged but remains useful prior art for expansion order and edge cases.

## What This Changes

Request dotenv-file expansion for `${VAR}` and `$VAR` references in project/global `.env` loading while preserving shell environment precedence.

## Best Possible Solution

Add a small, tested expansion helper in the dotenv loading path after key normalization/filtering and before assignment, using `process.env` plus earlier accepted dotenv entries as the lookup scope. Keep the existing precedence/blocklist contracts intact, document the behavior in the environment docs, and either share or intentionally separate the durable service env semantics with tests.
