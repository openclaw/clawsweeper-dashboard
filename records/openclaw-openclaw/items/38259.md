## Summary

Keep open. Current main and the latest release still load parsed dotenv values literally, the related implementation PR is closed unmerged, and adding expansion needs a clear policy for trusted global dotenv files versus untrusted workspace dotenv files.

## What This Changes

The issue requests `${VAR}` and `$VAR` expansion in project and global `.env` files while preserving existing shell environment precedence.

## Best Possible Solution

Define a bounded dotenv expansion contract after workspace key filtering, with tests and docs for shell/CWD/global precedence, same-file and chained references, missing values, escaping, cycles, and blocked workspace keys.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: dotenv values are parsed and assigned directly to `process.env`, so `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remains literal on current main.

## Solution Assessment

Unclear. Expansion is a plausible improvement, but the maintainable fix needs explicit trusted-global versus untrusted-workspace semantics instead of a narrow provider-key alias workaround.
