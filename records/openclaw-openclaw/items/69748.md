## Summary

Keep this open: current main and v2026.6.6 still use Discord `requireMention` plus `users`/`roles`, while this issue remains the canonical broader access-control design for sender-specific bypass, mention-only access, deny precedence, role/permission matching, and channel cascade semantics.

## What This Changes

The issue requests a new Discord guild/channel access model that replaces `requireMention` and flat sender allowlists with `bypass`, `mention`, and `deny` access lists, role/permission entries, and per-channel override semantics.

## Best Possible Solution

Approve one canonical Discord access-control contract, implement it inside the Discord plugin with deny-first tests, schema/docs updates, and a doctor migration from old shipped config shapes.

No live Discord repro was run, but source inspection gives a high-confidence way to show the limitation: configure current `users`/`roles` plus `requireMention` and there is no sender-specific bypass/mention/deny policy to exercise.

## Solution Assessment

Unclear until maintainers decide the contract; the proposal is coherent, but the best implementation should use one canonical schema and doctor migration rather than long-lived runtime compatibility for old fields.
