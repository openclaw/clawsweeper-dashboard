## Summary

Keep open. Current main still models Discord guild/channel ingress with `requireMention`, `users`, and `roles`; the amended deny-first `bypass`/`mention`/`deny` resolver with permission shortcuts, `inherit`, and wildcard cascade merging is not implemented and needs maintainer/security design.

## What This Changes

The issue requests a Discord access-control redesign that replaces `requireMention`/`users` with deny-first `bypass`/`mention`/`deny` lists supporting users, roles, permissions, and per-channel wildcard cascade overrides.

## Best Possible Solution

Implement this as a Discord plugin access resolver and config migration plan, with legacy-field compatibility, deny-first evaluation, wildcard/specific-channel cascade tests, docs, doctor/security-audit updates, and no core owner-specific policy baked in.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config redesign. The capability gap is source-verifiable because current schema, types, runtime, tests, and docs still use the legacy `requireMention`/`users`/`roles` model.

## Solution Assessment

Unclear. The amended cascade direction fits the existing wildcard allowlist model better than changing `groupPolicy`, but the access-control migration and security semantics need maintainer approval before this is the narrowest maintainable fix.
