## Summary

Keep this open: current main and v2026.6.8 still use Discord `requireMention` plus `users`/`roles`, while the requested bypass/mention/deny access model remains an unimplemented security-sensitive config redesign.

## What This Changes

Replace Discord guild/channel `requireMention` and sender allowlists with role/permission-aware `bypass`, `mention`, and `deny` access lists plus per-channel cascade semantics and migration.

## Best Possible Solution

Approve one canonical Discord access-control contract, then implement it inside the Discord plugin with strict schema/docs/generated metadata, deny-first runtime tests, and a doctor migration from shipped config shapes.

Yes, at source level: current config, schema, and runtime cannot express owner bypass plus everyone-else mention access or deny-first sender rules. No live Discord replay was run for this read-only review.

## Solution Assessment

Unclear until maintainers decide the permanent contract; the proposal is coherent, but the safest solution is one canonical Discord-owned schema with doctor migration, not long-lived runtime compatibility for old fields.
