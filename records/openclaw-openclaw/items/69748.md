- `impact:security`: The issue is about who can trigger a Discord-connected agent and which role/permission selectors authorize access.

## Summary

Keep open: current main and the latest release still use Discord guild/channel `requireMention` with `users`/`roles` allowlists, while the requested `bypass`/`mention`/`deny` model, deny-first precedence, permission shortcuts, and migration path are not implemented.

## What This Changes

Request to replace Discord guild/channel `requireMention` and `users` access with `bypass`/`mention`/`deny` lists, role/permission selectors, and per-channel cascade or override semantics.

## Best Possible Solution

Keep this as the canonical Discord access-control redesign until maintainers approve the schema, precedence, migration/doctor behavior, and security posture, then implement it in the Discord plugin with focused docs and tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection shows a guild config can choose only one `requireMention` boolean plus `users`/`roles` allowlists, so the owner-bypasses-mention while everyone-else-must-mention case cannot be represented today.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the direction addresses the real gap, but the final design needs maintainer approval and should use a canonical schema plus doctor migration rather than a long-lived runtime deprecation path for old keys.
