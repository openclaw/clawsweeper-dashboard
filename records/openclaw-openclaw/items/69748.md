## Summary

Keep open. Current main still exposes Discord guild/channel ingress through `requireMention`, `users`, and `roles`; the amended `bypass`/`mention`/`deny` cascade proposal is unimplemented and needs maintainer/security design because it can change who may trigger guild messages.

## What This Changes

Request to replace Discord guild/channel `requireMention` and sender allowlists with deny-first `bypass`/`mention`/`deny` access lists that support roles, permission shortcuts, wildcard cascade, per-channel overrides, and legacy migration.

## Best Possible Solution

Keep this as a single Discord plugin access-control design task: add a maintainer-approved resolver/schema and migration path, preserve documented wildcard allowlist behavior unless deliberately changed, and update runtime, docs, security audit, and regression tests together.

## Reproduction Assessment

Yes for the current-main capability gap by source inspection: strict Discord schemas and runtime only accept/use `requireMention`, `users`, and `roles`, while wildcard channel matching is direct-or-wildcard instead of a merge cascade.

## Solution Assessment

Unclear. The proposal addresses the per-user mention and wildcard-repeat pain, but the exact migration/defaults/permission shortcuts should be approved as a security-sensitive Discord access-control contract before implementation.
