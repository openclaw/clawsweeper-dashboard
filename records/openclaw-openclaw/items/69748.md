## Summary

Keep open. Current main still uses Discord `requireMention`, `users`, and `roles` with exact-or-wildcard channel matching, so the requested deny/bypass/mention access-list model remains unimplemented and needs maintainer security/product design before code work.

## What This Changes

The issue requests replacing Discord `requireMention` and flat sender lists with deny/bypass/mention access lists, role/permission entries, and channel-aware cascade overrides.

## Best Possible Solution

Keep this as an open Discord plugin design task: define the public access-list semantics, preserve intentional wildcard/channel behavior, add typed schema/runtime/docs/tests, and include a focused security audit of migration and precedence.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config redesign. Source inspection gives high confidence that current main lacks the proposed fields and still gates with `users`/`roles` plus one `requireMention` boolean.

## Solution Assessment

Unclear until maintainers approve the public config names, defaults, precedence, permission shortcuts, and migration path. The latest proposal is a plausible direction, but it changes security-sensitive Discord ingress semantics rather than being a narrow repair.
