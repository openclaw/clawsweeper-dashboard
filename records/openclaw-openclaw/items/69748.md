## Summary

Keep open. Current main still exposes Discord `users` and `roles` sender allowlists plus a single `requireMention` boolean, while the requested deny/bypass/mention access-list model with permission entries, deprecation behavior, and channel cascade semantics remains unimplemented and access-control sensitive.

## What This Changes

The issue requests a Discord guild/channel access-control redesign replacing `requireMention` and `users` with deny, bypass, and mention lists that support users, roles, permission shortcuts, and layered per-channel overrides.

## Best Possible Solution

Keep this as an open Discord plugin design task and implement it only after maintainers approve the access-list schema, deny precedence, wildcard cascade behavior, legacy migration/deprecation plan, docs, and focused security tests.

Not applicable as a bug reproduction because this is a feature/config redesign. Source inspection gives high confidence that current main lacks the requested fields and still uses `users`/`roles` plus boolean `requireMention`.

## Solution Assessment

Unclear until maintainers approve the public schema and authorization semantics. The amended cascade proposal is coherent, but changing Discord ingress access control is broader than a narrow repair and needs explicit product/security review.
