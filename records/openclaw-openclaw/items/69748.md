## Summary

Keep open. Current main still exposes Discord guild/channel access through `requireMention`, `users`, and `roles`, and the runtime still applies a single boolean mention gate after a sender allowlist check rather than evaluating deny-first `bypass`/`mention`/`deny` lists or permission shortcuts. The channel inheritance part also intentionally overlaps open #73387, so this needs maintainer design review rather than conservative cleanup.

## What This Changes

The issue requests replacing Discord `requireMention`/`users` access behavior with `bypass`, `mention`, and `deny` lists, role/permission entries, and group-policy-aware per-channel overrides.

## Best Possible Solution

Keep this as an open Discord plugin design task: add a maintainer-approved access resolver and config schema for deny-first `bypass`/`mention`/`deny` rules, preserve old `requireMention`/`users` behavior during a documented deprecation window, coordinate `groupPolicy`-aware channel inheritance with #73387, and update runtime, schema, doctor/audit, tests, and docs together.

## Reproduction Assessment

Yes for the current-main gap. A source-level check against current main shows the schema, runtime allowlist path, docs, and tests still use boolean `requireMention` plus `users`/`roles`, and still document/test unmatched configured channels as denied.

## Solution Assessment

Unclear. The proposal is coherent and concrete, but because it changes Discord access-control and migration semantics, the best implementation needs maintainer approval and coordination with #73387 before an automated fix attempts it.
