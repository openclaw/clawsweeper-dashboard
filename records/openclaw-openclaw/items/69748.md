## Summary

Keep open. Current main still uses the existing Discord `requireMention` plus `users`/`roles` allowlist model, and the amended proposal is a new security-sensitive access-control design rather than an already-implemented or superseded fix.

## What This Changes

The issue requests Discord `bypass`/`mention`/`deny` access lists with role/permission entries, legacy migration, and channel wildcard/specific-channel cascade semantics.

## Best Possible Solution

Keep this as an open Discord plugin design task: define a maintainer-approved access resolver and schema, preserve legacy fields through warnings/doctor migration, and update runtime, docs, tests, and security audit together.

## Reproduction Assessment

Yes for the current-main feature gap. Source, docs, and tests show the existing `requireMention` plus `users`/`roles` model and fallback-only wildcard matching; no live Discord run is needed to see the requested schema and cascade are absent.

## Solution Assessment

Unclear. The amended proposal fits the reported all-or-nothing access problem while respecting the existing wildcard allowlist model, but the deny/bypass/mention semantics, permission shortcuts, and migration need maintainer/security approval before implementation.
