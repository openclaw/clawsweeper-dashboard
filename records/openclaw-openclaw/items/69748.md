## Summary

Keep this open: current main and the latest release still use the existing Discord `requireMention` plus `users`/`roles` model, while the requested bypass/mention/deny precedence and channel cascade are new security-sensitive config behavior that needs maintainer product review.

## What This Changes

The issue requests a Discord guild/channel access model that replaces `requireMention` and `users` with bypass, mention, and deny lists, including role/permission entries and channel-layered overrides.

## Best Possible Solution

Land a maintainer-approved Discord access-control design that preserves safe defaults, defines one canonical schema and precedence model, adds doctor migration for old config, and updates Discord docs/tests together.

## Reproduction Assessment

No live Discord repro was run in this read-only review, but source inspection gives a clear way to reproduce the limitation: configure a guild/channel with `users` and `requireMention`, then observe there is no current config shape for one sender to bypass mention while others may only trigger by mention.

## Solution Assessment

Unclear; the bypass/mention/deny model is coherent, but the best fix depends on a maintainer decision about replacing versus extending `requireMention`/`users`/`roles`, channel wildcard cascade semantics, and migration policy.
