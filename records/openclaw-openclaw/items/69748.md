## Summary

Keep this open: current main and v2026.6.8 still use Discord `requireMention` plus `users`/`roles`, while the requested bypass/mention/deny access-list model remains an unimplemented, security-sensitive Discord config redesign.

## What This Changes

The issue requests replacing Discord guild/channel `requireMention` and flat sender allowlists with bypass/mention/deny lists, role/permission entries, and per-channel cascade semantics.

## Best Possible Solution

Approve one canonical Discord-owned access-control contract, then implement it in the Discord plugin with strict schema/docs/generated metadata, deny-first runtime tests, and a doctor migration from shipped config shapes.

## Reproduction Assessment

Yes, at source level: current schema and runtime cannot express owner bypass plus everyone-else mention access or deny-first sender rules. I did not run a live Discord replay in this read-only review.

Unclear until maintainers decide the permanent contract; the proposal is coherent, but the safest path is one canonical Discord plugin schema with doctor migration rather than long-lived runtime compatibility for old fields.
