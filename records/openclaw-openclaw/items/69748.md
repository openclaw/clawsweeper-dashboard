## Summary

Keep open. Current main still uses Discord `users`/`roles` allowlists plus a single `requireMention` boolean, and the reporter’s later comments narrowed this into a still-unimplemented access-list and wildcard cascade design that changes public config and ingress authorization semantics.

## What This Changes

The issue requests replacing Discord `requireMention`/`users` gating with `bypass`/`mention`/`deny` access lists, role and permission entries, and channel-level cascade overrides.

## Best Possible Solution

Keep this as an open Discord plugin design task: define the public access-list schema, preserve the intentional channel allowlist and wildcard semantics, add migration/docs/tests, and run focused security review on access widening.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config redesign. Source inspection gives high confidence that current main lacks the requested fields and still gates Discord guild senders with `users`/`roles` plus one `requireMention` boolean.

## Solution Assessment

Unclear until maintainers approve the public config names, defaults, precedence, permission shortcuts, and migration path. The amended cascade proposal is plausible, but it changes security-sensitive Discord ingress semantics rather than being a narrow repair.
