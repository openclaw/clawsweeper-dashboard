## Summary

Keep open: this is a fresh, detailed Slack thread data-loss report against the latest release, and current main still has the relevant idle-rollover and Slack thread dispatch paths without a focused regression proving the triggering message survives the rollover.

## What This Changes

A Slack thread message that arrives after the idle reset window should still be delivered to the newly rolled session instead of being dropped.

## Best Possible Solution

Preserve the existing idle reset semantics, but guarantee the inbound Slack thread event that triggers the rollover is recorded and dispatched into the new session with focused regression coverage.

## Reproduction Assessment

Unclear for a failing current-main run: the reporter supplied concrete v2026.5.4 timestamps and transcripts, and source inspection shows the stale thread rollover path, but this read-only pass did not live-reproduce the message loss.

## Solution Assessment

Yes: the narrowest maintainable direction is to keep configured idle rollover and fix the Slack/session pipeline so the current inbound message cannot be dropped during rollover; changing reset policy would be a broader product change.
