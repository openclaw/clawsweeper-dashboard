## Summary

Keep open: this remains a credible Slack thread data-loss bug. Current main still has the Slack thread idle-rollover path and the timestamp-only thread-history gate, and the related open PR targets outbound reply delivery rather than the inbound triggering-message loss reported here.

## What This Changes

The issue reports that an inbound Slack thread message arriving after idle session rollover is dropped instead of being delivered to the old or newly created session.

## Best Possible Solution

Preserve the triggering Slack thread event across idle rollover and align stale-thread history seeding with the same freshness decision used by session initialization, with focused regression coverage and live Slack proof if feasible.

## Reproduction Assessment

No high-confidence live reproduction was established. The reporter supplied concrete v2026.5.4 evidence, and current source still shows the idle rollover plus Slack thread dispatch/history-gate path, but this review did not run Slack Socket Mode or create a failing current-main test.

## Solution Assessment

Yes for the direction: preserving the triggering inbound event through the existing new-session path, and loading thread context when the prior session is effectively stale, is narrower than changing reset policy or adding a config knob. The first implementation step should be a failing regression that isolates where the message drops.
