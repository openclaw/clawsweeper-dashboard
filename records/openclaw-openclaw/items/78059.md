## Summary

Keep open: current main still has the stale-session rollover path plus a Slack thread-context gate that treats any stored timestamp as an existing thread, and no merged or closing PR preserves the inbound message that triggered the reset. The related open PRs cover adjacent compaction timestamp behavior or outbound Slack delivery, not this central inbound-loss case.

## What This Changes

The issue reports that the first Slack thread message after idle session rollover is dropped instead of being delivered to the newly rotated session.

## Best Possible Solution

Preserve the inbound Slack thread event through stale session rollover and make Slack thread-context seeding use the same freshness decision as session initialization, with focused regression coverage before any broader reset-policy change.

## Reproduction Assessment

No high-confidence live reproduction was established. The reporter provided concrete shipped-version evidence, and current source still shows a source-reproducible mismatch between stale-session rollover and Slack’s timestamp-only thread-context gate.

## Solution Assessment

Yes, the best fix direction is to preserve the triggering inbound event and align Slack thread prefetch with the stale-session freshness decision. Disabling idle reset or relying on the outbound Slack delivery PR would be broader or solve a different symptom.
