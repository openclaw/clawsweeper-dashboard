## Summary

This should stay open: the report describes a real existing Slack thread behavior, and current main still has the same idle-rollover/session-context boundary without a focused regression proving the triggering post-idle message is preserved.

## What This Changes

Slack thread messages that arrive after an idle session reset should be delivered to the old or new thread session instead of being dropped.

## Best Possible Solution

Preserve and dispatch the triggering Slack thread message across idle rollover, and add a regression test for a stale thread session receiving the first post-idle mention, including first-turn thread context when a new session is minted.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path for a maintainer with Slack socket-mode credentials: let a Slack thread session exceed the configured idle reset window, then mention the agent in that thread and inspect the old/new transcripts. I did not live-run it here, but the issue’s session evidence and current source match the stale-thread rollover path.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable fix is to keep the inbound Slack message attached to the post-rollover turn and align Slack thread context detection with session freshness, rather than disabling idle resets globally.
