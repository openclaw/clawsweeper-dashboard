## Summary

Keep open for maintainer merge review: the patch addresses a source-reproducible ClickClack allowlist bypass and I found no discrete code defect, but landing intentionally changes restrictive `allowFrom` configs to fail closed for non-listed senders.

## What This Changes

Adds a ClickClack inbound access resolver, routes gateway and inbound handling through `allowFrom`, derives `CommandAuthorized` from that decision, and adds focused regression tests.

## Best Possible Solution

Land the plugin-local shared-ingress enforcement once maintainers accept the intentional fail-closed `allowFrom` upgrade behavior, with release-note or docs follow-through during landing if they want that called out.

Yes from source inspection: current main dispatches non-bot ClickClack messages without resolving sender access and sets `CommandAuthorized` true. I did not run a live current-main ClickClack repro in this read-only review.

## Solution Assessment

Yes for the code direction: using the shared channel ingress resolver inside the ClickClack plugin keeps the authorization decision at the owning plugin boundary. The remaining question is maintainer acceptance of the deliberate fail-closed upgrade behavior for restrictive configs.
