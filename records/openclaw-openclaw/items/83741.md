## Summary

Keep this PR open: the code direction looks sound for the ClickClack allowlist bypass, but the current proof is still fixture/mock-based and the PR intentionally changes delivery for restrictive allowlists, so it needs human merge judgment rather than cleanup closure.

## What This Changes

The PR adds a ClickClack inbound access resolver, gates gateway and inbound handling on `allowFrom`, derives `CommandAuthorized` from the access decision, and adds focused regression tests.

## Best Possible Solution

Land the plugin-local shared-ingress enforcement after real ClickClack transport proof, or an explicit maintainer decision to accept the fixture proof, confirms allowed senders dispatch and denied senders are dropped before agent handling.

## Reproduction Assessment

Yes from source inspection: current main has an `allowFrom` config contract, but ClickClack gateway dispatch calls inbound handling without access resolution and inbound context sets `CommandAuthorized: true`. I did not run a live current-main ClickClack repro in this read-only review.

## Solution Assessment

Yes for the code direction: using the shared plugin SDK ingress resolver inside the ClickClack plugin is the right ownership boundary. The remaining question is proof and upgrade safety, not a different implementation shape.
