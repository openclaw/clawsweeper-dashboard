## Label Justifications

- `P1`: The PR fixes an authorization boundary in a channel workflow where configured ClickClack sender restrictions were not enforced before agent dispatch and command handling.
- `merge-risk: 🚨 compatibility`: Merging changes restrictive ClickClack `allowFrom` from effectively unenforced inbound behavior to fail-closed dispatch behavior.
- `merge-risk: 🚨 message-delivery`: Denied ClickClack senders will intentionally stop receiving model/reply dispatch, which is user-visible message delivery behavior.

## Summary

Keep this PR open for maintainer review rather than cleanup-close. The patch shape looks correct and the posted built-Gateway fixture proof is usable, but merging intentionally makes restrictive ClickClack `allowFrom` configs fail closed, so maintainers need to accept the compatibility and message-delivery impact.

## What This Changes

The branch adds a ClickClack access resolver, gates gateway and inbound handling on `allowFrom`, derives `CommandAuthorized` from that decision, and adds focused gateway/inbound tests.

## Best Possible Solution

Land the plugin-local shared-ingress enforcement once maintainers accept the intentional fail-closed `allowFrom` upgrade behavior, with release-note or docs follow-through during landing if they want that behavior called out.

## Reproduction Assessment

Yes from source inspection: current main dispatches non-bot ClickClack messages without resolving sender access and sets `CommandAuthorized` true. I did not run a live current-main ClickClack repro in this read-only review.

## Solution Assessment

Yes for the code direction: using the shared channel ingress resolver inside the ClickClack plugin keeps the authorization decision at the owning plugin boundary. The remaining decision is whether maintainers accept the deliberate fail-closed upgrade behavior for restrictive `allowFrom` configs.
