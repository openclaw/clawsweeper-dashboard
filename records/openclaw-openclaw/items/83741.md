## Summary

Keep open: the PR targets a real ClickClack authorization gap on current main, but the patch currently drops non-DM ClickClack messages under the existing `allowFrom` shape and has a failing test-type check.

## What This Changes

The PR adds a ClickClack inbound access resolver, gates gateway and inbound dispatch with sender access, derives `CommandAuthorized` from that access result, and adds focused gateway/inbound tests.

## Best Possible Solution

Keep the plugin-owned access resolver, but make ClickClack group/channel ingress use the existing `allowFrom` sender source and clear the test-type failure before maintainers accept the fail-closed upgrade behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows the gateway reaches inbound handling without resolving sender access, and inbound context creation hard-codes `CommandAuthorized: true`.

## Solution Assessment

No, not yet. The resolver is the right ownership direction, but it must preserve ClickClack group/channel `allowFrom` behavior and pass test-type validation before merge.

## Review Findings

Full review comments:

- **[P1] Preserve allowFrom for group ClickClack messages:** `extensions/clickclack/src/access.ts:60`
  - body: ClickClack currently exposes only `allowFrom`, but this resolver treats non-DM messages as `groupPolicy: "allowlist"` without passing `groupAllowFrom` or enabling fallback to `allowFrom`.   - confidence: 0.92

  - confidence: 0.99
