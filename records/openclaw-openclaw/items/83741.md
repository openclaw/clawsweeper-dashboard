## Label Justifications

- `P1`: The PR fixes an authorization boundary in an agent/channel workflow where configured ClickClack sender restrictions were not enforced before command handling.
- `merge-risk: 🚨 compatibility`: Merging changes restrictive ClickClack `allowFrom` from effectively advisory on inbound agent dispatch to fail-closed behavior for existing configs.
- `merge-risk: 🚨 message-delivery`: Non-allowlisted ClickClack senders will stop reaching model/reply dispatch after this patch, which is intended but user-visible.

## Summary

Keep open: this is a focused, source-reproducible security fix and the patch direction looks correct, but merging intentionally makes restrictive ClickClack `allowFrom` fail closed for inbound dispatch, so it still needs maintainer merge judgment.

## What This Changes

The PR adds a ClickClack inbound access resolver, gates gateway/inbound dispatch on sender access, derives `CommandAuthorized` from that decision, and adds focused gateway/inbound tests.

## Best Possible Solution

Land the plugin-local shared-ingress enforcement once maintainers accept the intentional fail-closed `allowFrom` upgrade behavior, with release-note or docs follow-through during landing if they want the behavior called out.

## Reproduction Assessment

Yes from source inspection: current main dispatches ClickClack non-bot messages without resolving sender access and sets `CommandAuthorized` true. I did not run a live current-main ClickClack repro in this read-only review.

## Solution Assessment

Yes for the code direction: using the shared channel ingress resolver inside the ClickClack plugin keeps the authorization decision at the owning plugin boundary. The remaining decision is whether maintainers accept the deliberate fail-closed upgrade behavior for restrictive `allowFrom` configs.
