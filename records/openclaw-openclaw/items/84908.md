## Summary

Keep open, but this PR is not ready to merge: current main already has the explicit Slack `replyBroadcast` primitive, while this branch adds a broader model-output directive on old/diverged Slack paths, forwards broadcast through media sends that current main intentionally rejects, and lacks real Slack behavior proof.

## What This Changes

The PR adds Slack reply-broadcast directive parsing, propagates `slackReplyBroadcast` through reply/outbound paths, forwards it to Slack sends/actions, and documents/tests the new directive.

## Best Possible Solution

Port the directive idea onto current main’s Slack plugin surfaces, keep the existing text/Block Kit only `replyBroadcast` contract, reject or ignore media/upload attempts explicitly, and coordinate any automatic mirroring with the open Slack broadcast policy discussion.

## Reproduction Assessment

Yes for the review findings at source level: the PR patch forwards `slackReplyBroadcast` into Slack media sends and strips the new tag regardless of `stripReplyTags`. No live Slack reproduction was provided for the user-visible directive behavior.

## Solution Assessment

No. The maintainable path is to build on current main’s `extensions/slack` implementation and existing `replyBroadcast` contract, not to merge the old `src/slack`/core-channel path with looser media semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep replyBroadcast out of media sends:** `src/channels/plugins/outbound/slack.ts:24`
  - body: Current main intentionally rejects `replyBroadcast` for Slack media/upload paths because the broadcast flag is a `chat.postMessage` thread-reply option, not a file-upload broadcast primitive. This patch forwards `slackReplyBroadcast` into `sendMedia`, so a media reply with the directive can have the tag stripped while the uploaded content is not actually broadcast, or only later text chunks are broadcast. Keep the directive limited to text/Block Kit thread replies and add coverage that media/upload attempts fail or are ignored explicitly.
  - confidence: 0.87
- **[P2] Honor stripReplyTags for the new directive:** `src/utils/directive-tags.ts:72`
  - body: `parseInlineDirectives` has a `stripReplyTags: false` mode that preserves reply tags while parsing them, but the new Slack broadcast tag is removed unconditionally here. Callers using the non-stripping parse mode will lose visible content only for this new directive, so the replacement should respect `stripReplyTags` like the existing reply tag branch or introduce an explicit option with tests.
  - confidence: 0.82
