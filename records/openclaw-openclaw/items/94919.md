- `impact:message-loss`: The operational fallback notice is not delivered to the originating user-visible output in subagent and async contexts.

## Summary

Keep open: current main still skips fallback notices for subagent-style preserved inter-session runs, and the provider request config still lacks a bounded per-provider connection policy. The issue is valid but needs maintainer product and security decisions before a safe implementation path is chosen.

## What This Changes

The issue asks OpenClaw to surface model fallback notices to the originating user for async Z.AI-triggered fallback and add a guarded per-provider connection policy to reduce those fallbacks.

## Best Possible Solution

Keep this open until maintainers define a surface-aware fallback-notice policy and a bounded guarded transport or retry policy that preserves the SSRF-protected provider fetch path.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main treats subagent announce provenance as session-state-preserving and the E2E suite asserts no `Model Fallback:` payload for that path. The live Z.AI endpoint failure was not reproduced because it needs the reporter's credentials and affected provider conditions.

## Solution Assessment

No as a direct implementation prescription: the visibility bug is real, but raw `fetchOptions` and unconditional async notice propagation need maintainer policy. The safer direction is a surface-aware notice contract plus whitelisted transport controls inside the existing SSRF-guarded path.
