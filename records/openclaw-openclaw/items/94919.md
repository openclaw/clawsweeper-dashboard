## Summary

Keep open: current main still suppresses fallback notices for subagent/inter-session provenance while the model-provider request config has no bounded connection-serialization policy. Related issues cover fallback visibility or provider socket failures separately, but none fully owns this combined Z.AI concurrency plus async visibility report.

## What This Changes

The issue reports invisible async model-fallback notices after Z.AI ECONNRESET failures and requests a guarded per-provider connection policy to reduce fallback frequency.

## Best Possible Solution

Keep this open for maintainer review, then split the durable fix into a surface-aware fallback-notice visibility policy and, only if approved, a generic bounded provider-transport connection policy that stays inside the SSRF-guarded fetch path.

## Reproduction Assessment

Yes for the OpenClaw notice/config gap: current source and an existing E2E test show subagent_announce fallback returns no Model Fallback payload, and the provider request schema lacks a connection-policy field. The Z.AI ECONNRESET concurrency behavior itself was not live-reproduced without provider credentials.

## Solution Assessment

No as a direct one-shot patch: an ad hoc Z.AI fetchOptions escape hatch would be the wrong first fix unless maintainers choose a generic guarded transport policy and resolve the shared fallback-notice visibility policy. The safer route is a maintainer-chosen surface policy plus bounded SSRF-preserving transport support if needed.
