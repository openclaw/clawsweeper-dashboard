- `impact:message-loss`: The operational fallback notice is not delivered to the originating user-visible output in subagent and async contexts.

## Summary

Keep open: current main and v2026.6.8 still suppress fallback notices for preserved inter-session/subagent provenance, and the provider request config still lacks a bounded per-provider connection policy. Related open issues overlap, but none replaces this combined Z.AI async-visibility plus guarded transport report.

## What This Changes

The issue requests user-visible model-fallback notices for async runs plus a guarded provider connection policy for Z.AI-style constrained endpoints.

## Best Possible Solution

Keep this open until maintainers define a surface-aware fallback-notice policy and a bounded provider transport or retry policy that stays inside the SSRF-guarded fetch path.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main skips fallback notice generation for `subagent_announce` provenance, and the E2E suite asserts no `Model Fallback:` payload for that path. The live Z.AI concurrency failure itself was not re-run because it needs provider credentials and affected endpoint conditions.

## Solution Assessment

No as a direct implementation prescription: the visibility bug is real, but the suggested connection knob and async notice propagation both need maintainer policy so they do not conflict with external-channel suppression, SSRF guard rules, or broader retry/backoff work.
