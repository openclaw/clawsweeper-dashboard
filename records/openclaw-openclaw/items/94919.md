## Label Justifications

- `P1`: The issue reports real async agent workflows silently switching from a configured Z.AI primary to a different fallback model without notifying the triggering user.
- `impact:message-loss`: The operational fallback notice/status message is not delivered to the originating user-visible output in subagent and async contexts.

## Summary

Keep open: current main and v2026.6.8 still suppress fallback notices for preserved inter-session/subagent provenance, and the provider request config still has no bounded per-provider connection policy. Related open issues cover notice visibility, SSRF dispatcher behavior, and retry policy, but none replaces this combined Z.AI async-visibility plus guarded transport report.

## What This Changes

The issue reports invisible model fallback notices in async/sub-agent contexts after Z.AI `ECONNRESET` fallback and requests a guarded per-provider connection policy to reduce those fallbacks.

## Best Possible Solution

Keep this open until maintainers define a shared fallback-notice visibility policy and, separately, a bounded provider transport or retry policy that stays inside the SSRF-guarded fetch path.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main skips fallback notice generation for `subagent_announce` provenance, and the E2E suite asserts no `Model Fallback:` payload for that path. The live Z.AI concurrency failure itself was not re-run because it needs the reporter's provider account and affected endpoint conditions.

## Solution Assessment

No as a direct implementation prescription: the visibility bug is real, but the suggested `fetchOptions` escape hatch and async notice propagation both need maintainer policy so they do not conflict with external-channel suppression, SSRF guard rules, or broader retry/backoff work.
