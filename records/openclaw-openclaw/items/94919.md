## Label Justifications

- `P1`: The issue reports real async agent workflows silently switching from a configured Z.AI primary to a different fallback model without notifying the triggering user.
- `impact:message-loss`: The operational fallback notice/status message is not delivered to the originating user-visible output in subagent and async contexts.
- `impact:auth-provider`: The affected behavior is provider routing and model choice after Z.AI model-fetch failures.

## Summary

Keep open: current main still skips fallback notice generation for inter-session subagent provenance, and the provider request config still lacks a bounded connection/serialization policy for strict provider endpoints. Related open issues cover notice visibility, SSRF dispatcher behavior, and retry policy, but none replaces this combined Z.AI async-visibility plus guarded-transport report.

## What This Changes

The issue asks OpenClaw to surface model-fallback notices to the originating user in async/subagent contexts and add guarded per-provider connection controls for strict Z.AI endpoints.

## Best Possible Solution

Keep this open until maintainers define a surface-aware fallback-notice policy and a bounded provider transport or retry policy that stays inside the SSRF-guarded fetch path.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main skips fallback notice generation for subagent_announce provenance, and the E2E suite asserts no Model Fallback payload for that path. The live Z.AI concurrency failure itself was not re-run because it needs provider credentials and affected endpoint conditions.

## Solution Assessment

No, not as a direct implementation prescription: the visibility bug is real, but the suggested fetchOptions escape hatch and async notice propagation both need maintainer policy so they do not conflict with external-channel suppression, SSRF guard rules, or broader retry/backoff work.
