## Summary

Keep open: current main and the latest release still have source-level async fallback-notice gaps and no per-provider connection-policy surface, while an open closing PR now owns the notice-propagation half and the transport/config half still needs maintainer product and security review.

## What This Changes

The issue requests reliable user-visible model-fallback notices for async Z.AI fallback runs plus a guarded per-provider connection policy to reduce Z.AI ECONNRESET-triggered fallbacks.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/95370 for async fallback-notice visibility, then split any guarded provider connection or retry policy into an explicit maintainer-approved transport/config follow-up.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main suppresses fallback notices for preserved subagent provenance and cron/ACP async boundaries do not preserve or relay fallback notices. The live Z.AI ECONNRESET rate itself was not rerun because it requires the reporter's provider account and endpoint conditions.

## Solution Assessment

No as a single automatic fix: the visibility bug has a focused open PR, but the per-provider fetchOptions/connection policy is a separate public config and SSRF-sensitive product decision. The safer path is to review the notice PR and handle guarded transport policy separately.
