## Summary

Keep open: current main and v2026.6.8 still have the async fallback-notice visibility gap, and the requested guarded per-provider connection policy is not implemented. The open linked PR at https://github.com/openclaw/openclaw/pull/95370 is a candidate for the notice-propagation slice, but it is not merged and deliberately leaves the Z.AI transport/config policy work open.

## What This Changes

This issue asks OpenClaw to surface model fallback notices to users in async runs after Z.AI fetch failures and to add a guarded per-provider connection policy to reduce those fallbacks.

## Best Possible Solution

Review and land or revise the linked async notice-propagation PR if proof and visibility policy are accepted, then track guarded provider connection or retry policy as separate maintainer-approved transport work.

## Reproduction Assessment

Yes for the OpenClaw source-level gap: current main emits fallback notices only outside preserved async provenance, the ACP parent relay ignores fallback lifecycle phases, and cron final-text collapse drops leading fallback notices. The live Z.AI endpoint failure was not rerun because it requires the reporter's account and affected endpoint conditions.
