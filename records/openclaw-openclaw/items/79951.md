## Summary

Keep open: the PR is a narrow and likely correct fix for a source-reproducible gateway restart config-loss bug, with no blocking code or security finding found, but the external contributor still has not supplied real behavior proof from an actual gateway restart path.

## What This Changes

The PR changes `src/cli/gateway-cli/run.ts` so the pre-read startup config snapshot is passed only to the first gateway start callback and omitted on later restart-loop starts.

## Best Possible Solution

Merge the first-start-only snapshot fix after real gateway/SIGUSR1 config-preservation proof is supplied, with focused regression coverage or equivalent maintainer validation before landing.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main pre-reads config once, reuses the same start callback across restart iterations, and server startup skips the disk read when that snapshot is injected. I did not run the Docker/SIGUSR1 flow live, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, the proposed first-start-only gate is the narrowest maintainable fix because it preserves the cold-start optimization while letting restart iterations use the normal disk snapshot loader. The merge path still needs real runtime proof and ideally a focused regression test.
