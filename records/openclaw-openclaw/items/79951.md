## Summary

Keep open. The code change is a narrow and likely-correct fix for the stale startup snapshot path, but the external PR's after-fix proof is only a closure simulation rather than an actual gateway restart/config-preservation run.

## What This Changes

The PR changes `src/cli/gateway-cli/run.ts` so `runGatewayCommand` passes `startupConfigSnapshotRead` only to the first `startGatewayServer` call inside the gateway loop.

## Best Possible Solution

Merge a first-start-only snapshot fix after adding focused regression coverage or equivalent maintainer validation, plus real runtime proof that an actual in-process restart preserves disk config changes.

## Reproduction Assessment

Yes. The current-main source path shows one pre-read snapshot closed over by a restart loop that calls the same start callback repeatedly, while server startup skips the disk read whenever that snapshot is supplied; I did not run the Docker/SIGUSR1 scenario live.

## Solution Assessment

Yes. Gating the pre-read snapshot to the first start is the narrowest maintainable fix because it preserves the cold-start optimization while making restart iterations use the normal disk snapshot loader; the merge path still needs real behavior proof and preferably regression coverage.
