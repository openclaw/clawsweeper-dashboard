## What This Changes

The branch changes `src/cli/gateway-cli/run.ts` so `startupConfigSnapshotRead` is passed only to the first gateway start callback.

## Best Possible Solution

Land the focused first-start-only snapshot fix after real restart proof shows post-boot config edits remain on disk, ideally with a regression test for the second start callback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main closes over the pre-read snapshot, the restart loop calls the same start callback again, and the startup loader skips disk when an initial snapshot is supplied. I did not execute the Docker/SIGUSR1 scenario in this read-only review.

## Solution Assessment

Yes, the PR's first-start-only gate is the narrowest maintainable fix because it preserves the cold-start optimization while forcing restart iterations through the normal disk snapshot loader. The remaining gap is proof and regression coverage, not a different code direction.
