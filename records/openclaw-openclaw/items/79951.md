## Summary

Keep open. Current main still has the stale `startupConfigSnapshotRead` reuse path, and this PR targets the right closure with a narrow fix, but it still needs real gateway restart proof before merge.

## What This Changes

The PR changes `src/cli/gateway-cli/run.ts` so `startupConfigSnapshotRead` is passed only to the first `startGatewayServer` call made by the in-process gateway restart loop.

## Best Possible Solution

Merge the first-start-only snapshot fix after adding focused regression coverage or equivalent maintainer validation, plus real runtime proof that an actual in-process gateway restart preserves disk config changes.

## Reproduction Assessment

Yes. Current main shows a high-confidence source reproduction path: one pre-read startup snapshot is closed over by a restart loop that calls the same start callback repeatedly, while server startup skips disk reread whenever that snapshot is supplied.

## Solution Assessment

Yes. Gating the pre-read snapshot to the first start is the narrowest maintainable fix because it preserves the cold-start optimization while making restart iterations use the normal disk snapshot loader.
