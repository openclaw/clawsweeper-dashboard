## Summary

Keep open: the PR addresses a real Fal routing/payload bug, but the current diff would regress existing Flux edits and applies model-specific Fal edit behavior as provider-wide capabilities.

## What This Changes

The PR modifies `extensions/fal/image-generation-provider.ts` to route GPT Image and Nano Banana edit requests to `/edit`, send `image_urls`, raise Fal edit image limits, and allow aspect-ratio overrides in edit mode.

## Best Possible Solution

Use endpoint/model-specific Fal edit handling: preserve Flux `/image-to-image` singular `image_url` behavior, use `/edit` with plural `image_urls` for GPT/Nano Banana, map sizing fields per Fal schema, then add regression tests and changelog.

## Reproduction Assessment

Yes, via source and schema inspection: current main routes GPT Image 2 input-image requests to `/image-to-image`, while Fal documents `/edit` with `image_urls`. The PR regression is also source-reproducible because Flux requires singular `image_url` but the diff sends `image_urls` for every edit request.

## Solution Assessment

No. The direction is right for the linked bug, but the blanket provider-wide payload, limit, and aspect-ratio changes are not the narrowest maintainable fix; the safer path is model-specific Fal edit routing and payload construction.

## Review Findings

Full review comments:

- **[P1] Keep Flux edits on `image_url`:** `extensions/fal/image-generation-provider.ts:347`
  - body: The diff changes every edit request to send `image_urls`, but the existing default `fal-ai/flux/dev/image-to-image` endpoint still requires singular `image_url`. That means existing Flux edits will omit the required field and fail even though the PR says Flux behavior is unchanged.
  - confidence: 0.94
- **[P2] Make edit sizing model-specific:** `extensions/fal/image-generation-provider.ts:205-210`
  - body: This branch returns `image_size` for all edit requests and advertises aspect-ratio support globally. GPT Image 2 uses `image_size`, but Nano Banana 2 documents `aspect_ratio`/`resolution`, while Flux image-to-image does not document those fields, so non-GPT edits get the wrong schema.
  - confidence: 0.87
- **[P2] Do not raise the Flux reference-image limit:** `extensions/fal/image-generation-provider.ts:280`
  - body: Raising the provider-level edit limit to 14 and only rejecting `>14` allows multi-image requests to existing Flux `/image-to-image`, but that endpoint accepts one `image_url`, not a list. Keep the higher multi-reference limit scoped to `/edit` models or enforce per-model limits before building the request.
  - confidence: 0.88

  - body: This is a user-facing `fix(fal)` change, and repo policy requires user-facing fixes to update `CHANGELOG.md`; the PR currently only changes provider code.
