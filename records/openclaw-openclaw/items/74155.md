## Summary

Keep open. Current main still lacks Slack-owned `fileId`/`messageId` schema guidance and still throws the generic `fileId` requirement, so the PR addresses a real gap, but the submitted patch has merge blockers in dependency metadata, action-list copy, lint, CI, and changelog coverage.

## What This Changes

The PR adds Slack message-tool schema descriptions for `fileId`, `messageId`, and `message_id`, adds a targeted `download-file` error when `messageId` is supplied without `fileId`, and updates Slack tests.

## Best Possible Solution

Land a Slack-owned schema/hint change that documents `fileId` versus `messageId` only for currently advertised Slack actions, keeps Slack package metadata accurate, and ships with green Slack-focused tests, lint, changed-gate proof, and a single Unreleased Slack fix changelog entry; keep downstream inbound `fileId` propagation separate.

## Reproduction Assessment

Yes. Call `handleSlackMessageAction` with `action: "download-file"`, `messageId` or `message_id`, and no `fileId`; current main reaches the generic `readStringParam(... "fileId", { required: true })` path, while the PR adds regression tests for that misuse shape.

## Solution Assessment

No, not as submitted. The Slack-owned schema contribution plus dispatcher hint is the narrow maintainable direction, but the patch must declare its runtime dependency, align every action reference with the actual Slack action surface, satisfy lint/CI, and include changelog coverage before merge.
