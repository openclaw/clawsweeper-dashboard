## Summary

Keep open. Current main has adjacent memory-flush, compaction, transcript-rotation, and queue dedupe protections, but they still do not cover the reported hook-timed, same-content duplicate delivery after a new id and roughly nine-minute gap; the provided related context does not identify a merged canonical fix for this exact behavior.

## What This Changes

The issue reports a bug where one user message is delivered again after memory flush and hook/auto compaction complete.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, session-rotation, and follow-up queue boundary so maintenance or hook-triggered compaction cannot convert the prior visible user prompt into a fresh user turn, while preserving legitimate queued follow-ups and intentional repeated prompts.

Yes for a synthetic regression: the issue gives enough timing and metadata to model one user prompt, memory flush, hook-triggered compaction, transcript/session rotation, and same-content re-delivery. Exact live reproduction would need a sanitized copy of the private trajectory or transcript.

## Solution Assessment

Yes, the best path is a narrow runtime fix rather than a broad product redesign. The existing 60-second compaction duplicate window and message-id-only queue dedupe are narrower than the reported nine-minute same-content duplicate, so the fix should target idempotency at the maintenance/session-boundary source.
