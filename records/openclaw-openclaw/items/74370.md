## Summary

Keep open. Current main has adjacent dedupe and compaction cleanup, but it still does not cover the reported hook-timed same-content user-message replay with a new id after roughly nine minutes.

## What This Changes

The issue reports that a user prompt is delivered a second time after memory flush and hook-triggered compaction, even though the user sent it once.

## Best Possible Solution

Add a focused idempotency guard at the maintenance, compaction, transcript-rotation, and follow-up queue boundary so hook-triggered maintenance cannot convert the previous visible user prompt into a fresh user turn, while preserving legitimate queued follow-ups and intentional repeated prompts.

## Reproduction Assessment

Yes for a synthetic regression: the issue gives enough timing, ids, and hook/compaction metadata to model one prompt, memory flush, hook-triggered compaction, transcript rotation, and same-content re-delivery. Exact live reproduction would need a sanitized transcript extract.

## Solution Assessment

Yes, a narrow runtime idempotency fix is the best path. Existing message-id caches, 60-second compaction duplicate pruning, and test-only transcript-hash helpers are too narrow for the reported nine-minute same-content replay.
