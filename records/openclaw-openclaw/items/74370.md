## Summary

Keep open. Current main still leaves the reported memory-flush plus hook-timed compaction duplicate-delivery path plausible: memory flush treats any compaction end as completion, adjacent runners already require `completed: true`, active-run queue dedupe is message-id based, and transcript duplicate pruning intentionally keeps distant repeats.

## What This Changes

The issue reports a same-content user message being delivered again with a new id after memory flush and hook-timed compaction, and asks for idempotent handling around compaction/replay.

## Best Possible Solution

Add a focused session-boundary idempotency guard in memory-flush compaction handling and queued follow-up replay, counting only completed compactions while preserving legitimate repeated prompts.

## Reproduction Assessment

Yes for a synthetic regression: the report supplies timing, ids, hook markers, same-content/new-id behavior, and current code has seams to simulate memory flush plus compaction-end events. Exact live reproduction would need a redacted transcript excerpt.

## Solution Assessment

Yes, the narrow maintainable direction is to align memory-flush compaction handling with the existing `completed: true` contract and add replay-boundary coverage. Global prompt dedupe would be less safe because users can intentionally send the same prompt again.
