## Summary

Keep open. Current main still has the adjacent weak spots from the prior ClawSweeper review: the memory-flush runner treats any compaction end event as completed, active-run followups still enqueue with message-id dedupe, and compaction duplicate pruning only covers a 60-second repeat window while the report describes about nine minutes.

## What This Changes

The issue reports that a single user message was delivered again after memory flush and hook-timed compaction/session rotation.

## Best Possible Solution

Add a narrow idempotency guard around memory-flush compaction/session rotation and active followup queueing so hook-triggered maintenance cannot turn the prior visible user turn into a fresh inbound turn, while preserving valid repeated user sends.

## Reproduction Assessment

Yes for a synthetic regression path: model one visible user prompt, a memory-flush run, hook-originated compaction/session rotation, and a same-content requeue with a different message id. Exact live reproduction would need a sanitized trajectory or transcript excerpt from the reporter.

## Solution Assessment

Yes, a focused runtime fix is the best direction. It should use the existing compaction completed contract plus a maintenance/session-boundary idempotency key rather than broad prompt suppression.
