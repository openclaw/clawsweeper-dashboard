## Summary

Keep open. Current main still matches the reported failure shape: the memory-flush path treats any compaction end event as a completed rotation signal, even though the embedded compaction event already carries a stricter `completed` flag that adjacent runner code honors.

## What This Changes

The issue reports duplicate delivery of the same user message after memory flush plus hook-timed compaction and asks for the memory-flush/compaction replay path to stop reinjecting the prior prompt.

## Best Possible Solution

Add a focused memory-flush compaction/session-rotation guard that honors only `completed: true` compaction end events and covers the same-content/new-id replay case with a synthetic regression test.

## Reproduction Assessment

Yes for a synthetic regression: drive the memory-flush embedded-agent callback with `stream: "compaction"`, `phase: "end"`, and `completed: false`, then assert it does not rotate the session or refresh queued followups. Exact live reproduction would need a redacted excerpt of the reporter's local transcript.

## Solution Assessment

Yes, a narrow runtime fix is the best direction. The safer alternative to global prompt dedupe is to make memory-flush session rotation use the existing completed-compaction contract while leaving legitimate repeated prompts alone.

## Repair Work Prompt

Fix openclaw/openclaw#74370. The observable bug is duplicate delivery of the same user prompt after memory flush plus hook-timed compaction: current main marks memory-flush compaction complete on any `compaction` `phase: "end"` event in `src/auto-reply/reply/agent-runner-memory.ts`, while the embedded compaction event exposes `completed` and adjacent follow-up code already requires `completed === true`. Implement a narrow guard so memory-flush session rotation, compaction-count increment, and queued-followup refresh happen only for completed compaction end events. Preserve overflow recovery where `willRetry: true` and `completed: true` still counts as a completed compaction. Do not add global prompt-based dedupe, do not widen the short duplicate-user-message pruning window as the primary fix, and do not change plugin/channel boundaries. Add regression coverage for `completed: false` or missing `completed` during memory flush not rotating or refreshing queued followups, plus a positive test that `completed: true` keeps existing rotation behavior. Add a one-line user-facing changelog fix entry.
