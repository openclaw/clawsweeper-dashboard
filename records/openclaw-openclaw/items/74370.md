## Summary

Keep open. Current main still has adjacent safeguards, but the reported memory-flush plus hook-timed compaction duplicate-delivery path is not fully covered: the memory-flush callback treats any compaction end event as a completed compaction, the active-run follow-up path dedupes by message id, and the existing duplicate-user-message pruning window is much shorter than the reported nine-minute redelivery.

## What This Changes

Prevent a hook-timed memory flush plus compaction from re-delivering a previously processed user message as a fresh turn.

## Best Possible Solution

Add a narrow idempotency guard at the memory-flush, compaction/session-rotation, and follow-up queue boundary so maintenance or hook-triggered compaction cannot turn the prior visible user prompt into a fresh user turn, while preserving legitimate repeated user messages.

## Reproduction Assessment

Yes for a focused synthetic regression: the issue provides enough timing, ids, and state markers to model one visible prompt, memory flush, hook-originated compaction/session rotation, and same-content re-delivery. Exact live reproduction would need a redacted transcript excerpt from the reporter.

## Solution Assessment

Yes, a focused runtime fix is the best direction. The safer path is to honor the existing completed compaction event contract and add an idempotency key around maintenance/session-boundary replay rather than applying broad prompt suppression.
