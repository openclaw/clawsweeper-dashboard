## Summary

Keep open. Current main still has adjacent protections for compaction, memory-flush metadata, transcript rotation, and queue dedupe, but they do not cover the reported hook-timed same-content redelivery after a new message id and roughly nine-minute gap.

## What This Changes

Prevent a prior user prompt from being re-delivered as a fresh message after memory flush and hook-triggered compaction.

## Best Possible Solution

Add regression-backed idempotency at the memory-flush, compaction, transcript-rotation, and follow-up queue boundary so maintenance or hook-triggered compaction cannot turn the prior visible user prompt into a new user turn, while preserving legitimate queued follow-ups and intentionally repeated prompts.

## Reproduction Assessment

Yes for a synthetic regression: model one user prompt, memory flush, hook-triggered compaction with modified files, transcript/session rotation, and same-content re-delivery with a new id. Exact live reproduction would need a sanitized copy of the private trajectory or transcript.

## Solution Assessment

Yes, a narrow runtime fix is the right direction. The current completed-only compaction event contract and existing queue/pruning helpers should be tightened at the maintenance boundary rather than adding a broad global prompt drop that could suppress intentional repeats.

## Repair Work Prompt

Fix the duplicate user-prompt delivery reported in https://github.com/openclaw/openclaw/issues/74370. The observable bug is: after a user sends one prompt, memory flush plus hook-triggered compaction/session rotation can cause the same prompt to be delivered again later as a fresh user message with a new id. Keep the fix narrow to memory flush, compaction/session rotation, and follow-up queue idempotency; do not redesign compaction, hooks, or channel routing globally. Start by adding a regression that models one user prompt, memory flush, hook-originated compaction with modified files, session/transcript rotation, and a same-content redelivery roughly nine minutes later. Current clues: `src/auto-reply/reply/agent-runner-memory.ts` marks any compaction `phase: "end"` as completion instead of requiring `completed === true`; duplicate-user-message pruning is only 60 seconds; active enqueue defaults to message-id dedupe; prompt-based dedupe exists but should not suppress intentional repeats. Preserve legitimate queued follow-ups and user-intended repeated prompts. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @allesgutefy`. Validate with the targeted tests listed in `workValidation`, then run the changed gate per repo policy.
