## Summary

Keep open. Current main still treats any memory-flush compaction `phase: "end"` event as successful, while the event producer and adjacent follow-up consumer use `completed === true`; the latest shipped release has the same loose consumer behavior and no merged duplicate or PR currently owns this exact replay bug.

## What This Changes

The issue reports duplicate delivery of a user message after memory flush and compaction.

## Best Possible Solution

Gate memory-flush session rotation and queued-followup refresh on `completed === true` compaction end events, with regression coverage for incomplete, aborted, and missing-completion events while preserving legitimate repeated prompts.

## Reproduction Assessment

Yes for a source-level synthetic path: a memory-flush run that receives `stream: "compaction"` with `phase: "end"` and `completed` false or missing still rotates and refreshes session state on current main. A full live replay would need the reporter's redacted transcript excerpt.

## Solution Assessment

Yes. The narrow maintainable fix is to honor the existing `completed === true` compaction contract in the memory-flush consumer instead of adding prompt-content dedupe that could block intentional repeated messages.
