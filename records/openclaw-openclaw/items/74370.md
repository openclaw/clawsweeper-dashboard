## Summary

Keep open. Current main still has the same adjacent gaps identified in the earlier ClawSweeper review: the memory-flush path treats any compaction end event as completed, normal active-run followup enqueueing remains message-id based, and compaction duplicate pruning only covers a short same-content window, so the reported nine-minute same-prompt re-delivery is still a plausible current-main bug.

## What This Changes

The issue reports that a single user message was delivered a second time after memory flush and hook-timed compaction/session rotation.

## Best Possible Solution

Add a narrow idempotency guard across memory-flush compaction/session rotation and followup enqueueing so maintenance or hook-triggered compaction cannot turn the previous visible user prompt into a fresh user turn, while preserving deliberate repeated user messages.

## Reproduction Assessment

Yes for a synthetic regression: the issue gives one original prompt, two compaction events including a hook-originated one, modified-file context, and a new duplicate message id about nine minutes later. Exact live reproduction would need a sanitized trajectory or transcript excerpt from the reporter.

## Solution Assessment

Yes, a focused runtime fix is the best direction. The safer approach is to key idempotency around maintenance/session-boundary replay and the existing compaction completion contract, not to globally suppress same-prompt user messages.
