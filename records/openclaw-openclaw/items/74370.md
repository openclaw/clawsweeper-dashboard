## Summary

Keep open. Current main still has only adjacent mitigations: silent memory flush can observe compaction end and refresh queued sessions, compaction duplicate-user pruning is limited to a short retry window, and active-run follow-up enqueue defaults to message-id dedupe. That leaves the reported nine-minute same-content, fresh-id re-delivery path unresolved enough for a focused repair PR.

## What This Changes

The issue requests that OpenClaw stop re-delivering a previous user prompt as a fresh turn after memory flush, hook-driven compaction, and session rotation.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, transcript-rotation, and follow-up queue boundary so maintenance or hook-triggered compaction cannot turn the prior user prompt into a fresh user turn, while preserving legitimate repeated prompts and queued follow-ups.

## Reproduction Assessment

Yes for a synthetic regression: the issue gives enough timing and metadata to model one user prompt, memory flush, hook-triggered compaction, transcript/session rotation, and same-content fresh-id re-delivery. Live reproduction is unclear without the private transcript or a sanitized trajectory export.

## Solution Assessment

Yes, a focused idempotency fix is the best direction. The current 60-second compaction duplicate window and message-id-only follow-up dedupe are narrower than the reported nine-minute same-content duplicate, so this does not need broad product redesign first.
