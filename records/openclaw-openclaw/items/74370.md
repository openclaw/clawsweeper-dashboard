## Summary

Keep open. Current main still has adjacent memory-flush, compaction, transcript-rotation, and queue dedupe protections, but they do not cover the reported hook-timed duplicate delivery of the same user prompt roughly nine minutes later with a new message id.

## What This Changes

The issue reports duplicate delivery of a single user message after memory flush and hook-timed compaction, with the same content reappearing as a fresh user turn about nine minutes later.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, session-rotation, and follow-up queue boundary so hook-triggered maintenance cannot convert the prior visible user prompt into a fresh user turn, while preserving legitimate queued follow-ups and intentional repeats.

## Reproduction Assessment

Yes for a synthetic regression: the report gives enough timing and metadata to model one user prompt, a memory flush, hook-triggered compaction, transcript/session rotation, and same-content re-delivery. Exact live replay would require a sanitized copy of the private session transcript.

## Solution Assessment

Yes, the best path is a narrow runtime fix rather than broad product redesign. Current main’s 60-second compaction duplicate window and message-id-only queue dedupe are narrower than the reported nine-minute same-content duplicate, so the fix should target idempotency at the maintenance/session boundary.
