## Summary

Keep open. Current main still leaves the reported hook-timed memory-flush plus compaction duplicate-delivery path plausible: memory flush treats any compaction end as completion, the active-run queue dedupes by message id, and transcript duplicate pruning only covers a short adjacent window.

## What This Changes

Prevent a memory flush plus hook-originated compaction/session-rotation sequence from re-injecting the prior user prompt as a new delivery.

## Best Possible Solution

Add a focused idempotency guard at the memory-flush compaction/session-rotation and queued follow-up boundary, honoring completed compaction events while preserving legitimate repeated user messages.

## Reproduction Assessment

Yes for a synthetic regression: the report supplies timing, ids, hook markers, same-content/new-id behavior, and the memory-flush plus compaction sequence to model. Exact live reproduction would need a redacted transcript excerpt from the reporter.

## Solution Assessment

Yes, a narrow runtime fix is the best direction. The safer alternative to global prompt dedupe is a session-boundary guard that uses the existing compaction completion contract and replay context.
