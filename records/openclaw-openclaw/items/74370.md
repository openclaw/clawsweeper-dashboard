## Summary

Keep open. Current main still matches the prior ClawSweeper analysis: the memory-flush path can treat any compaction end event as a successful rotation, even though adjacent runtime code already exposes and honors a stricter completed=true contract.

## What This Changes

Fix duplicate delivery of a single user message after memory flush and hook-timed compaction.

## Best Possible Solution

Add a session-boundary guard in memory-flush compaction handling so only completed compactions rotate or refresh queued follow-ups, while preserving legitimate repeated prompts.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74370. In `src/auto-reply/reply/agent-runner-memory.ts`, make the memory-flush compaction/session-rotation path treat compaction as completed only when a compaction end event has `data.completed === true`; preserve successful overflow/retry compactions where `willRetry` may be true but `completed` is true. Add focused regression coverage showing that an aborted or no-result `phase: "end", completed: false` memory-flush compaction does not increment/rotate the session or refresh queued follow-ups, while `completed: true` still does. Update existing tests that currently emit a bare `phase: "end"` to use the explicit completed contract. Do not add global prompt-content dedupe, do not change queue TTL/message-id semantics, and do not broaden compaction policy. Add a user-facing `CHANGELOG.md` fix entry unless maintainers classify this as internal-only.
