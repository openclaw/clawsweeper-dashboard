## Summary

Keep open. Current main still treats any memory-flush compaction `phase: "end"` event as completed, while the emitted compaction event and adjacent follow-up runner already require `completed === true`; the reported same-content/new-id replay remains a plausible current-main bug with a narrow repair path.

## What This Changes

Prevent a user message from being redelivered after memory flush and hook-originated compaction rotate or refresh session state.

## Best Possible Solution

Update memory-flush compaction handling to honor `completed === true` before rotating session state or refreshing queued followups, with regression coverage for missing/false completion and successful completion.

## Reproduction Assessment

Yes for a synthetic source-level path: current main still rotates and refreshes memory-flush session state after a compaction `phase: "end"` event even when `completed` is false or missing. Exact live reproduction would need the reporter’s redacted local transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `completed === true` compaction contract in the memory-flush path, not to add broad prompt-content dedupe that could block intentional repeated prompts.

## Repair Work Prompt

Fix the duplicate-delivery bug described in https://github.com/openclaw/openclaw/issues/74370. In the memory-flush runner, treat compaction as completed only when the compaction event is `phase: "end"` and `completed === true`, matching the embedded compaction event contract and the normal follow-up runner. Update the memory-flush rotation test that currently emits `phase: "end"` with no `completed` field, and add regression coverage for `completed: false` or missing not rotating/refreshing queued followups while `completed: true` still rotates. Do not add broad prompt-content dedupe that would reject intentional repeated prompts. Add a user-facing `CHANGELOG.md` fix entry if runtime behavior changes.
