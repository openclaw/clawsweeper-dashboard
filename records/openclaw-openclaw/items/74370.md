## Summary

Keep open. Current main still has adjacent safeguards, but the reported hook-timed memory-flush plus compaction duplicate delivery is not covered: the memory-flush path still treats any compaction end event as completion, prompt-based queue dedupe is not used for normal active-run followups, and the compaction duplicate-pruning window is much shorter than the reported nine-minute gap.

## What This Changes

The issue reports a same-content user message being re-delivered as a new turn after memory flush, hook-triggered compaction, and session rotation.

## Best Possible Solution

Add a narrow idempotency guard at the memory-flush, compaction/session-rotation, and followup queue boundary so maintenance or hook-triggered compaction cannot convert the prior visible user prompt into a fresh user turn, while preserving valid repeated user sends.

## Reproduction Assessment

Yes for a synthetic regression: the issue provides enough timing, ids, and state markers to model one visible user prompt, memory flush, hook-originated compaction/session rotation, and same-content re-delivery. Exact live reproduction would need a sanitized trajectory export or transcript excerpt.

## Solution Assessment

Yes, a focused runtime fix is the best direction. The safest implementation should use the existing completed compaction event contract and an idempotency key around maintenance/session-boundary replay rather than broad prompt suppression or a product-level redesign.
