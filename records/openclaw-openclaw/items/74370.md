## Summary

Keep open. Current main still has the reported duplicate-delivery risk around memory-flush compaction/session rotation: the memory-flush path treats any compaction end event as completion, active-run queue dedupe is still message-id based, and the existing duplicate transcript pruning intentionally does not cover the roughly nine-minute same-content replay described here.

## What This Changes

Reported duplicate delivery of a single user message after pre-compaction memory flush and hook-originated compaction/session rotation.

## Best Possible Solution

Add a focused idempotency guard at the memory-flush compaction/session-rotation and follow-up queue boundary, honoring completed compaction events while preserving legitimate repeated user messages.

## Reproduction Assessment

Yes for a synthetic regression: the report supplies enough timing, ids, hook markers, and same-content/new-id behavior to model one visible prompt followed by memory flush, hook-originated compaction, session refresh, and replay. Exact live reproduction would need a redacted excerpt of the reporter's transcript.

## Solution Assessment

Yes, a narrow runtime repair is the best direction. The safer fix is to honor the completed compaction contract and add session-boundary idempotency, not to globally suppress repeated prompt text.
