## Summary

Keep open. Current main still leaves the reported memory-flush plus hook-timed compaction duplicate-delivery path insufficiently guarded: the memory-flush runner treats any compaction end event as completion, active-run queue dedupe remains message-id based, and transcript duplicate pruning only covers a short adjacent window.

## What This Changes

Prevent a single user message from being re-delivered after memory flush and hook-timed compaction/session rotation.

## Best Possible Solution

Add a focused idempotency guard at the memory-flush compaction/session-rotation and follow-up queue boundary, honoring completed compaction events while preserving legitimate repeated user messages.

## Reproduction Assessment

Yes for a synthetic regression: the report supplies timing, ids, hook markers, same-content/new-id behavior, and the memory-flush plus compaction sequence. Exact live reproduction would need a redacted transcript excerpt from the reporter.

## Solution Assessment

Yes, a narrow runtime fix is the best direction. The safest implementation should honor the existing completed compaction event contract and add session-boundary idempotency rather than globally suppressing repeated prompt text.
