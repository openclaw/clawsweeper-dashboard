## Summary

Keep open; this is a useful and well-scoped hardening PR, but it is not merge-ready as submitted. The new `readProviderJsonResponse` import is a good source fix, but the shared provider HTTP test mock does not export or clear that helper, so Qwen/Alibaba provider tests that install the mock will fail until the mock contract is updated; maintainers should also accept the intentional 16 MiB oversized-JSON hard failure before merge.

## What This Changes

The PR changes DashScope video submit and poll success responses in `src/video-generation/dashscope-compatible.ts` to use `readProviderJsonResponse` with the shared 16 MiB provider JSON cap.

## Best Possible Solution

Keep the shared-helper source change, update the provider HTTP test mock export and cleanup path, run focused provider/helper tests, and land only after accepting the 16 MiB cap.

## Reproduction Assessment

Yes, from source inspection rather than local execution in this read-only review. Current main and `v2026.6.10` still use raw `response.json()` in both targeted DashScope success-body paths, and the PR body includes copied live output from a local `node:http` oversized-body proof.

## Solution Assessment

Mostly. Reusing `readProviderJsonResponse` is the narrowest maintainable source fix because the SDK already exposes a bounded JSON helper, but the branch must update `provider-http-test-mocks` before it is the best merge candidate.
