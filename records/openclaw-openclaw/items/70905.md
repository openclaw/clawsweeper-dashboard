## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on the outbound channel context, and the only related implementation PR was closed unmerged; the remaining decision is a public SDK seam with authorization semantics to settle.

## What This Changes

Expose an optional `agentId` on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the originating agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context, thread session id before mirror id only when present, document attribution-only semantics, and cover delivery plus SDK contract behavior after maintainer approval.

## Reproduction Assessment

Yes, by source inspection. Current main has upstream session and mirror agent ids, while `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the requested field.

## Solution Assessment

Yes as a likely direction, but only with maintainer sign-off. A narrow optional context field is cleaner than overloading display `identity`, provided the contract states that plugins must not treat it as authorization proof.
