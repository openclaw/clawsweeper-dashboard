## Label Justifications

- `P2`: This is a normal-priority plugin SDK enhancement with bounded blast radius but real outbound attribution impact for affected channel plugins.
- `impact:message-loss`: The issue reports that affected channel plugins can reject, misattribute, or silently drop non-inbound outbound sends without stable agent attribution.
- `impact:security`: The requested field exposes stable agent identity to plugins that may map it to remote authentication or authorization decisions.

## Summary

Keep open: current main and the latest release still omit `agentId` from `ChannelOutboundContext`. The request fits OpenClaw's plugin API direction, but it needs maintainer-approved trust semantics before implementation because it would expose stable agent identity to channel plugins.

## What This Changes

Expose an optional stable `agentId` on channel outbound send contexts so channel plugins can attribute sends to the producing agent.

## Best Possible Solution

Approve a narrow additive SDK contract for send-time agent attribution, then implement it with documented trust semantics, session/mirror precedence, public docs/API baseline updates, and focused outbound delivery tests.

## Reproduction Assessment

Yes, at source level: current main carries `session.agentId` and `mirror.agentId` upstream but `ChannelOutboundContext` and the adapter context builder do not expose it. I did not run a live plugin because the missing API surface is directly visible in source.

## Solution Assessment

Unclear until maintainers decide the trust contract. The optional field is a plausible narrow layer, but the best fix must define whether plugins receive attribution only or a verified sender-routing identity.
