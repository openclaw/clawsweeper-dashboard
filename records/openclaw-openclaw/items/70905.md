## Summary

Keep open. Current main still does not expose a plugin-facing `agentId` on `ChannelOutboundContext` or the outbound handler params, even though session and mirror agent metadata exist upstream; the only related implementation PR is closed unmerged and did not supersede this issue.

## What This Changes

Request to expose an optional `agentId` on `ChannelOutboundContext` so outbound channel plugins can attribute sends to the active agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context, thread session id before mirror id only when present, document attribution-only semantics, and cover it with delivery plus SDK contract checks after maintainer approval.

## Reproduction Assessment

Yes. A source trace on current main shows `OutboundSessionContext.agentId` and `OutboundMirror.agentId` exist upstream, while `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit `agentId`.

## Solution Assessment

Yes, with maintainer sign-off. A narrow optional field on the outbound adapter context is cleaner than overloading display `identity`, provided it is documented as attribution metadata and not treated as authorization proof.
