## Label Justifications

- `P2`: This is a normal-priority plugin SDK enhancement with bounded blast radius but real outbound attribution impact for affected channel plugins.
- `impact:security`: The requested field may be used by plugins as sender or remote-identity authority, so the trust boundary must be defined before exposing it.
- `impact:message-loss`: The report says affected channel plugins can reject, misattribute, or silently drop non-inbound outbound sends without stable agent attribution.

## Summary

Keep open: current main still omits `agentId` from the public channel outbound context, while upstream outbound session and mirror paths already carry an agent id. The request is a plausible plugin API extension, but it needs maintainer-approved trust/security semantics before implementation.

## What This Changes

Expose an optional stable `agentId` on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Approve a narrow additive SDK contract for send-time agent attribution, then implement it with documented trust semantics, session/mirror precedence, public docs/API-baseline updates, and focused outbound delivery tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `ChannelOutboundContext` and its adapter context builder do not expose `agentId`, while upstream `session.agentId` and `mirror.agentId` are already present in outbound delivery code.

## Solution Assessment

Unclear until maintainers decide the trust contract. The optional field is a plausible narrow layer, but the best fix must define whether plugins receive attribution metadata only or a verified sender-routing identity.
