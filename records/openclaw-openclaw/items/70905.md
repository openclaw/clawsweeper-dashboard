## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on outbound channel contexts, the only related implementation was closed unmerged, and the remaining decision is a public SDK contract/trust-boundary choice rather than conservative cleanup.

## What This Changes

The issue requests adding an optional stable agent id to `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

After maintainer approval, add a backward-compatible optional `agentId?: string` to the outbound channel context, thread session id before mirror id when present, document attribution-only semantics, and cover delivery plus SDK contract behavior.

## Reproduction Assessment

Yes, by source inspection. Current main has upstream session and mirror agent ids, while `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the requested plugin-facing field.

## Solution Assessment

Unclear until maintainers decide the trust semantics. A narrow optional context field appears maintainable, but it should ship with session-over-mirror precedence, absent-key semantics, and attribution-only documentation.
