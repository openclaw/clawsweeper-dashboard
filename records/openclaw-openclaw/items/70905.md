## Summary

Keep open. Current main still does not expose `agentId` on the plugin-facing outbound channel context, even though session and mirror agent metadata are already available in the outbound delivery path. The related implementation PR was closed unmerged, so the remaining work needs maintainer approval for the public plugin SDK contract rather than cleanup closure.

## What This Changes

Expose an optional stable agent id on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context and the internal handler params, populate it from session agent id first and mirror agent id second, omit the key when no agent is in scope, document that it is attribution metadata rather than authorization, and lock the contract with delivery tests plus SDK API baseline coverage.

## Reproduction Assessment

Yes. A source trace on current main reproduces the gap: session and mirror `agentId` are available upstream, but the public outbound adapter context type and context builder do not expose them to channel plugins.

## Solution Assessment

Yes, pending maintainer sign-off. Keeping stable routing metadata separate from display-only `identity` is the narrow maintainable direction, provided the implementation documents trust semantics and avoids present-but-undefined keys.
