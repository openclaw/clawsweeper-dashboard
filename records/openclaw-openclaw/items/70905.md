## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on `ChannelOutboundContext`, while the outbound pipeline already has session and mirror agent metadata available; the related implementation PR #73072 is closed unmerged, so there is no implemented or superseding fix to close against.

## What This Changes

Expose an optional stable agent id on outbound channel adapter context so channel plugins can attribute sends to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context and handler params, thread session agent id before mirror agent id, omit the key when no agent is in scope, document attribution-only semantics, and cover the behavior in delivery tests plus SDK baseline checks.

## Reproduction Assessment

Yes. A read-only source trace on current main reproduces the gap: session and mirror `agentId` are available upstream, but the public outbound adapter context type and context builder do not expose them to channel plugins.

## Solution Assessment

Yes, with maintainer sign-off. Keeping stable agent routing metadata separate from display-only `identity` is the narrow maintainable direction, provided future code treats it as attribution metadata rather than authorization proof.
