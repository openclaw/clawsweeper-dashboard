## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on outbound channel contexts, the only implementation PR was closed unmerged, and the remaining question is a public SDK identity/trust-boundary decision rather than conservative cleanup.

## What This Changes

Request to add an optional `agentId` to `ChannelOutboundContext` and thread existing outbound session or mirror agent ids into channel adapter sends.

## Best Possible Solution

Define a backward-compatible, attribution-only outbound `agentId` field with session-over-mirror precedence, explicit absent-key semantics, docs, and regression coverage after maintainer approval.

## Reproduction Assessment

Yes, by source inspection. Current main has upstream session and mirror agent ids, while `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the requested plugin-facing field.

## Solution Assessment

Unclear until maintainers decide the trust semantics. A narrow optional context field appears maintainable, but it should ship with attribution-only documentation, session-over-mirror precedence, and absent-key behavior locked by tests.
