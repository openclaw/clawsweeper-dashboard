## Label Justifications

- `P2`: This is a normal-priority plugin SDK enhancement with limited blast radius but real outbound attribution impact for affected channel plugins.
- `impact:message-loss`: The issue reports that affected channel plugins can reject, misattribute, or silently drop non-inbound outbound sends without stable agent attribution.

## Summary

Keep open. Current main and the latest stable tag still omit a plugin-facing `agentId` from outbound channel contexts, the only matching implementation PR was closed unmerged, and the remaining work needs an explicit SDK/trust-boundary decision before implementation.

## What This Changes

The issue requests adding optional `agentId` metadata to `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Have an owner approve the SDK contract, then add optional `agentId?: string` with attribution-only semantics, session-before-mirror precedence, absent-key behavior, docs/API baseline coverage, and focused outbound delivery tests.

## Reproduction Assessment

Yes, by source inspection. Current main carries agent ids in upstream session, mirror, and gateway send metadata, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the plugin-facing field.

## Solution Assessment

Unclear until the trust semantics are approved. A narrow optional context field appears maintainable, but it should ship as a documented SDK contract with explicit attribution-only or authorization semantics and absent-key behavior.
