## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on outbound channel contexts, the only related implementation was closed unmerged, and the remaining work needs an SDK contract and trust-boundary decision before code changes.

## What This Changes

Expose an optional `agentId` on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Have an owner approve the SDK contract, then add an optional `agentId?: string` with session-before-mirror precedence, absent-key behavior, attribution-only documentation, and focused delivery/API coverage.

## Reproduction Assessment

Yes, by source inspection. Current main carries agent ids in upstream session/mirror metadata, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the plugin-facing field.

## Solution Assessment

Unclear until the trust semantics are approved. A narrow optional context field appears maintainable, but it should ship with documented attribution-only meaning, session-over-mirror precedence, and absent-key behavior.
