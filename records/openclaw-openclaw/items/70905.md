## Label Justifications

- `P2`: This is a normal-priority plugin SDK improvement with a clear affected boundary and limited blast radius, but it needs maintainer contract review.
- `impact:message-loss`: The issue reports outbound sends for per-agent remote identities being rejected, unattributed, or losing attachments when plugins cannot identify the producing agent.
- `impact:security`: The requested field crosses agent identity and per-agent remote-auth semantics, so the trust boundary must be defined before exposure.

## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on `ChannelOutboundContext`, the related implementation closed unmerged, and exposing the field needs an SDK contract and agent-identity trust decision before implementation.

## What This Changes

Expose optional `agentId` on `ChannelOutboundContext` so outbound channel plugins can attribute sends to the producing agent.

## Best Possible Solution

Have an SDK owner approve the contract, then add optional `agentId?: string` with attribution-only semantics, session-before-mirror precedence, omit-when-absent behavior, docs/API updates, and focused delivery coverage.

## Reproduction Assessment

Yes, by source inspection. Current main carries agent ids in upstream session, mirror, and gateway-send metadata, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the plugin-facing field.

## Solution Assessment

Unclear until the SDK trust semantics are approved. A narrow optional context field appears maintainable, but it should ship with documented attribution-only meaning, session-over-mirror precedence, and absent-key behavior.
