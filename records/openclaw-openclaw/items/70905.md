## Label Justifications

- `P2`: This is a normal-priority plugin SDK improvement with a clear affected surface but limited blast radius and a required maintainer contract decision.
- `impact:message-loss`: The issue reports that channel plugins needing per-agent remote identity may reject or drop outbound sends when the producing agent id is unavailable.

## Summary

Keep open. Current main still lacks a plugin-facing `agentId` on outbound channel contexts, the only related implementation at https://github.com/openclaw/openclaw/pull/73072 was closed unmerged, and the remaining work needs an SDK contract and trust-boundary decision before implementation.

## What This Changes

Expose an optional stable `agentId` on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Approve the SDK contract, then add `agentId?: string` with attribution-only documentation, session-over-mirror precedence, absent-key behavior, and focused delivery/API coverage.

## Reproduction Assessment

Yes, by source inspection. Current main carries agent ids in upstream session and mirror metadata, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit a plugin-facing `agentId`.

## Solution Assessment

Unclear until the trust semantics are approved. A narrow optional context field appears maintainable, but it should ship with documented attribution-only meaning, session-over-mirror precedence, and absent-key behavior.
