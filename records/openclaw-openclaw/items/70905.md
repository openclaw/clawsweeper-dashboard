- `impact:message-loss`: The issue reports that affected channel plugins can reject, misattribute, or silently drop non-inbound outbound sends without stable agent attribution.

## Summary

Keep open. Current main and the latest stable release still omit a plugin-facing `agentId` from `ChannelOutboundContext`, and the only direct implementation attempt was closed unmerged; the remaining decision is the public SDK and agent-identity trust contract.

## What This Changes

The issue requests an optional `agentId` on outbound channel contexts so channel plugins can attribute sends to the originating agent.

## Best Possible Solution

Have maintainers approve the plugin SDK contract, then add a narrow optional `agentId?: string` with documented attribution semantics, session-before-mirror precedence, API baseline/docs updates, and focused outbound delivery tests.

## Reproduction Assessment

Yes, by source inspection. Current main carries agent ids in upstream session and mirror metadata, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the adapter context builder still omit the plugin-facing field.

## Solution Assessment

Unclear until the trust semantics are approved. A narrow optional context field appears maintainable, but it should ship as a documented SDK contract with explicit attribution-only or authorization semantics and absent-key behavior.
