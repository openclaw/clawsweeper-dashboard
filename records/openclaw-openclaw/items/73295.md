## Summary

Keep this PR open, but it is not mergeable as written: current main does not include the Discord queued-followup breaker, the branch still has concrete timer and abort-settlement correctness defects, and the external PR proof is only tests/CI with live Discord proof deferred.

## What This Changes

The PR adds a Discord-only queued-followup stuck-session breaker that aborts non-streaming active runs, prepends a recovery notice, resumes draining, exports the helper, and adds focused Vitest coverage.

## Best Possible Solution

Land a corrected recovery path that uses the existing settled abort-and-drain boundary, clears or replaces per-session timers, preserves queued state, and includes redacted live Discord proof.

## Reproduction Assessment

Yes for the PR defects: source inspection gives a high-confidence path for a stale timer to block or mis-time a later same-key session, and for immediate recovery drain to race the still-active reply-run registry. I did not run tests because this review was read-only.

## Solution Assessment

No, not as written. The direction is plausible, but the implementation should clear or replace stale timers and wait for abort settlement, or reuse the existing abort-and-drain/force-clear boundary before scheduling recovery work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh breaker timers for each queued session:** `src/auto-reply/reply/queue/circuit-breaker.ts:73-74`
  - body: The new timer map is keyed only by `queueKey` and returns when a timer already exists. A normal queue drain can empty and delete the followup queue without clearing this timer, so the next Discord session on the same key can inherit a stale timeout that fires on the wrong age or prevents a proper breaker from being armed.
  - confidence: 0.88

  - body: `abortActiveRun` only requests cancellation. The PR immediately prepends the recovery notice, marks the queue idle, and calls `scheduleDrain`, but the reply-run registry can still hold the same session key and make the recovery run hit `ReplyRunAlreadyActiveError`, leaving followups blocked behind the run that is still shutting down.
  - confidence: 0.9
