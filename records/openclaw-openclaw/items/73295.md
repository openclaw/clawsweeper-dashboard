## Summary

Keep open. The PR targets a real Discord queued-follow-up recovery gap that is not implemented on current main, but it should not merge as-is because the new breaker timer can be left stale across same-key sessions and recovery re-drains immediately after a signal-only abort; it also needs the required changelog entry.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker that arms a timer for queued follow-ups behind a non-streaming active run, aborts the run, prepends a recovery notice, restarts queue draining, exports the helper, and adds focused tests.

## Best Possible Solution

Land a corrected Discord follow-up recovery path that replaces or clears breaker timers per session, waits for abort settlement or force-clear before draining, preserves queued session state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The stale-timer failure is reproducible with fake timers by draining one Discord queue before the threshold and then enqueueing a later same-key stuck session; the abort race is reproducible by making abortActiveRun return true while the reply-run registry still reports that sessionKey active.

## Solution Assessment

No, not as written. The feature direction is plausible, but the implementation needs explicit timer lifecycle ownership and should reuse or mirror the existing abort-and-drain settlement boundary before restarting recovery work.
