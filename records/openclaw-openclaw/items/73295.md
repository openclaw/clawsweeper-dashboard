## Summary

Keep open: current main does not contain this Discord queue breaker, and the PR is not merge-ready. The earlier ClawSweeper and Greptile reviews correctly identify blocking timer-lifecycle and abort-settlement problems, and the external PR still has only mock/unit proof for a Discord runtime recovery change.

## What This Changes

The branch adds a Discord-only stuck-session circuit breaker that aborts non-streaming active runs after a threshold, prepends a recovery notice, drains queued followups, wires active-run helpers, and adds focused tests.

## Best Possible Solution

Land a corrected Discord follow-up recovery path that clears or replaces breaker timers per session, waits for abort settlement or force-clear before draining, preserves queued session state, and includes redacted real Discord behavior proof.

## Reproduction Assessment

Yes for the PR defects: fake timers can show a stale timer blocks a later same-key stuck session, and source inspection shows scheduleDrain can race while the reply-run registry is still active after abortActiveRun. I did not run tests because this review was read-only.

## Solution Assessment

No, not as written. The direction is plausible, but the implementation should replace or clear stale timers and reuse or mirror the existing abort-and-drain settlement boundary before scheduling recovery work.
