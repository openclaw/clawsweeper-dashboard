## Summary

Keep open. Current main has related conservative stuck-session diagnostics, but it does not include this Discord queue breaker; the PR is useful but not mergeable until the timer lifecycle, abort-settlement path, and changelog are fixed.

## What This Changes

The PR adds a Discord-only timer-based stuck-session circuit breaker for queued follow-ups, wires active-run helpers through the reply runner, exports the helper, and adds focused Vitest coverage.

## Best Possible Solution

Land a corrected Discord follow-up recovery path that owns breaker timer cleanup/replacement, waits for abort settlement or force-clear before draining, preserves queued session state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The stale-timer failure is reproducible with fake timers by draining one same-key Discord queue before the threshold and then enqueueing a later stuck session; the abort race is reproducible by making abortActiveRun return true while the reply-run registry still reports the sessionKey active.

## Solution Assessment

No, not as written. The feature direction is plausible, but the implementation should reuse or mirror the existing abort-and-drain settlement boundary and make timer lifecycle cleanup explicit.
