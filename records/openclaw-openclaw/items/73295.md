## Summary

Keep open. The PR targets a real Discord follow-up recovery gap, but the current head still has queue timer lifecycle and abort-settlement bugs that should block merge.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run streaming/abort helpers into reply execution, exports the helper, and adds focused queue tests.

## Best Possible Solution

Land a corrected Discord follow-up recovery path that owns timer cleanup/replacement, waits for abort settlement before draining, preserves queued session state, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The stale-timer bug is reproducible with fake timers by draining one same-key Discord queue before the threshold and enqueueing a later stuck queue; the abort race is reproducible by making abortActiveRun return true while the reply-run registry still reports the sessionKey active.

## Solution Assessment

No, not as written. The feature direction is narrow and useful, but it needs timer lifecycle handling and an abort-and-drain settlement boundary before it is the maintainable fix.
