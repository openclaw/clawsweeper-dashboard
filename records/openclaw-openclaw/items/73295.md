## Summary

Keep open. Current main still does not implement the Discord queued-followup stuck-session breaker this PR targets, but the proposed branch is not merge-ready because the breaker timer can go stale across sessions and the recovery drain can run before the aborted reply run has actually released the active-run registry.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run state and abort hooks through the reply runner, exports the queue helper, and adds focused Vitest coverage.

## Best Possible Solution

Land a corrected, narrowly scoped Discord follow-up recovery path that owns timer cleanup/replacement with the queue lifecycle, waits for abort settlement or uses the existing force-clear recovery helper before draining, refreshes queued session state when needed, and records the user-facing reliability fix in the changelog.

## Reproduction Assessment

Yes. The blocking behavior is reproducible by static inspection and focused fake-timer tests: drain one queued Discord session normally while the breaker timer remains registered, then enqueue a later same-key session; separately, make abortActiveRun return true while the reply-run registry still reports the session active and observe recovery drain hit the active-run guard.

## Solution Assessment

No, not as written. The feature direction is reasonable, but the implementation needs to be queue/session-lifecycle-aware and abort-drain-aware before it is the narrow maintainable fix.
