## Summary

Keep open. Current main does not contain this Discord-specific queued follow-up breaker, and the PR is useful source work, but it is not mergeable as written because timer and abort/drain lifecycle gaps can leave the queue unprotected or still blocked.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run abort helpers into the reply runner, exports the helper, and adds focused Vitest coverage.

## Best Possible Solution

Land a corrected, narrowly scoped Discord follow-up recovery path that owns timer cleanup/replacement, waits for abort settlement or force-clear before draining, preserves queued session state correctly, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The stale-timer failure is reproducible with fake timers by draining one same-key Discord queue before the threshold and then enqueueing a later stuck session; the abort race is reproducible by letting abortActiveRun return true while the reply-run registry still reports the sessionKey active.

## Solution Assessment

No, not as written. The feature direction is reasonable, but the implementation needs queue-lifecycle timer handling and abort-drain settlement before it is the narrow maintainable fix.

  - body: A breaker timer can survive a normal queue drain because this new map is not cleared from the drain cleanup path. A later Discord session on the same queueKey returns early instead of arming its own timer, so the fresh stuck session can be left unprotected once the stale timer no-ops.
