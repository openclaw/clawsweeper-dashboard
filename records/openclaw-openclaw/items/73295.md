## Summary

Keep this PR open. Current main still does not implement the Discord-specific active-run breaker the PR is trying to add, but the proposed patch is not safe to merge as written because its timer lifecycle can leave later Discord sessions unprotected and its recovery drain can race the still-active reply run.

## What This Changes

The branch adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run state and abort hooks through the reply runner, exports the queue helper, and adds focused Vitest coverage.

## Best Possible Solution

Land a corrected, narrowly scoped Discord follow-up recovery path that owns timer cleanup/replacement with the queue lifecycle, waits for abort settlement or uses the existing force-clear recovery helper before draining, refreshes queued session state if the active session is invalidated, and records the user-facing reliability change in the changelog.

## Reproduction Assessment

Yes. The blocking paths are reproducible by static inspection and with focused fake-timer tests: drain a queue normally while the breaker timer remains registered, then enqueue a later same-key Discord session; separately, make abortActiveRun return true while the reply-run registry still reports the session active and observe recovery drain hitting the existing active-run guard.

## Solution Assessment

No, not as written. The feature direction is reasonable, but the implementation needs to be queue/session-lifecycle-aware and abort-drain-aware before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh breaker timers for each new session:** `src/auto-reply/reply/queue/circuit-breaker.ts:72-74`
  - body: A breaker timer can survive a normal queue drain because nothing clears this new map from the drain cleanup path. A later Discord session on the same queueKey returns here instead of arming its own timer, so the fresh stuck session can be left unprotected once the stale timer no-ops.
  - confidence: 0.88
- **[P2] Wait for the abort before draining recovery work:** `src/auto-reply/reply/queue/circuit-breaker.ts:112`
  - body: abortActiveRun only signals cancellation; the reply-run registry can keep the same sessionKey active until the run unwinds. Calling scheduleDrain immediately can hit ReplyRunAlreadyActiveError and leave the queued follow-up blocked, so recovery should wait or use the existing abort-and-drain/force-clear path first.
  - confidence: 0.9

  - body: This adds user-visible Discord queue recovery behavior, but the PR does not update CHANGELOG.md. OpenClaw policy requires user-facing fixes/features to add a single-line entry with contributor credit before merge.
  - confidence: 0.82
