## Summary

Keep open. The bug is plausible and the PR targets the Telegram formatting boundary, but the current diff is not mergeable because it removes an existing helper, leaves another HTML send path uncovered, and still lacks live Telegram delivery proof for this transport-visible change.

## What This Changes

The PR adds a Telegram HTML sanitizer, wraps `sendTelegramText` HTML output with it, and adds format and delivery tests for unknown angle-bracket tags.

## Best Possible Solution

Preserve existing link attribute escaping, sanitize every Telegram `parse_mode: "HTML"` send boundary with a Bot API-aligned allow-list, and validate it with focused tests plus live Telegram delivery proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a full current-main live run in this review; source inspection and the linked report show raw HTML-mode paths can reach Telegram `parse_mode: "HTML"`, but the PR proof does not include a live Telegram send.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is reasonable, but this patch needs to preserve existing formatter behavior, cover the remaining HTML send boundary, align with Telegram's documented tag set, and provide live proof before merge.

- **[P1] Restore the HTML attribute escaping helper:** `extensions/telegram/src/format.ts:26`
  - body: The patch replaces `escapeHtmlAttr`, but `buildTelegramLink` still calls it when rendering link hrefs. That leaves `format.ts` with an undefined symbol and will fail typechecking/build; keep the helper or update the caller while adding the sanitizer.
  - confidence: 0.96

  - confidence: 0.9

  - confidence: 0.87
