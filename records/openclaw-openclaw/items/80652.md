## Summary

Keep open. The PR targets a plausible Telegram HTML delivery bug, but the current diff is not mergeable: it removes a helper still used by link rendering, only covers one of the HTML send boundaries, and its tag allow-list does not match Telegram's current HTML contract. It also still needs real Telegram behavior proof before merge.

## What This Changes

The PR adds a Telegram HTML sanitizer, applies it to `sendTelegramText`, and adds format and delivery tests for escaping unknown angle-bracket tags.

## Best Possible Solution

Preserve existing link attribute escaping, centralize sanitization across all Telegram `parse_mode: "HTML"` send boundaries, align the allow-list with the current Bot API, and prove the result with focused tests plus a live Telegram delivery check.

## Reproduction Assessment

Unclear for the full agent-response path without a live Telegram run. Source inspection does show raw `textMode: "html"` paths that can reach `parse_mode: "HTML"`, while the added `deliverReplies` test appears to exercise a path current main already markdown-escapes.

## Solution Assessment

No. This diff is not the best fix yet because it breaks the existing formatter helper, leaves another HTML send boundary uncovered, and uses a sanitizer allow-list that diverges from Telegram's documented HTML subset.

  - confidence: 0.87
