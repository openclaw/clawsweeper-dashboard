## What This Changes

The PR adds a Telegram HTML sanitizer, applies it to `sendTelegramText`, and adds format/delivery tests for escaping unknown angle-bracket tags.

## Best Possible Solution

Preserve existing link escaping, sanitize every Telegram `parse_mode: "HTML"` send boundary with a Bot API-aligned allow-list, and validate with focused tests plus real Telegram delivery proof.

## Reproduction Assessment

No for a full high-confidence live Telegram reproduction in this review. Source inspection shows raw HTML-mode text can still reach Telegram `parse_mode: "HTML"`, and the linked issue describes the user-visible truncation path.

## Solution Assessment

No. The direction is reasonable, but this patch needs to preserve existing formatter behavior, cover the remaining HTML send boundary, align the allow-list with Telegram's documented HTML support, and add live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restore the HTML attribute escaping helper:** `extensions/telegram/src/format.ts:26`
  - body: The patch replaces `escapeHtmlAttr`, but `buildTelegramLink` still calls it when rendering link hrefs. That leaves `format.ts` with an undefined symbol and will fail typechecking/build; keep the helper or update the caller while adding the sanitizer.
  - confidence: 0.96
- **[P2] Cover the direct HTML send boundary:** `extensions/telegram/src/bot/delivery.send.ts:114-116`
  - body: Wrapping only `sendTelegramText` leaves `sendMessageTelegram`'s explicit `textMode: "html"` path unchanged: `renderTelegramHtmlText` returns raw HTML and `send.ts` still sends it with `parse_mode: "HTML"`. Outbound `message.send` or adapter HTML deliveries can still hit the same Telegram parse failure.
  - confidence: 0.9
- **[P2] Match Telegram's supported HTML tags:** `extensions/telegram/src/format.ts:27-29`
  - body: The sanitizer allow-list omits supported HTML tags such as `span class="tg-spoiler"`, `tg-emoji`, and `tg-time`, while permitting `br`, which Telegram's HTML parse-mode list does not include. That can escape valid caller HTML and still pass unsupported markup through.
  - confidence: 0.87
