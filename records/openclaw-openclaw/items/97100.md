## Best Possible Solution

Land this after a maintainer explicitly accepts the public SDK freshness seam and Slack reset-session context behavior; otherwise keep the helper private or plugin-local with the same tests.

## Reproduction Assessment

Yes at source level: seed an existing Slack thread session row that is stale under daily or idle reset policy, then process a reply in that same thread. ## Solution Assessment

Yes functionally, if maintainers accept the public SDK seam: centralizing freshness keeps Slack aligned with session initialization and avoids duplicate reset logic. If that API direction is not accepted, the safer alternative is a private or Slack-local helper with the same behavior and tests.
