## Summary

## What This Changes

The PR adds a public session-entry freshness helper and uses it in Slack prepare so stale thread rows seed initial Slack thread context after reset.

## Best Possible Solution

A maintainer should explicitly accept the public SDK freshness seam and Slack reset-session prompt behavior, or ask for a private/plugin-local variant before merge.

## Reproduction Assessment

Yes at source level: seed an existing Slack thread session row that is stale under daily or idle reset policy, then process a reply in that same thread. Current main's Slack prepare path sees the row timestamp and skips history, while session initialization evaluates the row as stale and starts a new session.

## Solution Assessment

Yes functionally, if maintainers accept the public SDK seam: centralizing freshness keeps Slack aligned with session initialization and avoids duplicate reset logic in the plugin. If that public API direction is not accepted, the safer alternative is a private/internal helper with the same behavior.
