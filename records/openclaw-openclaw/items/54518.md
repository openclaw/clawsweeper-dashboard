## Summary

Keep open. Current main still makes host filesystem confinement optional for read/write/edit, source tests preserve outside-workspace access when `tools.fs.workspaceOnly` is unset or false, and the related PathGuard/local-user isolation work remains open and security-sensitive.

## What This Changes

The issue asks for per-agent filesystem confinement so agents cannot read or mutate files outside their own workspace.

## Best Possible Solution

Define and land an owner-reviewed confinement policy that aligns `tools.fs.workspaceOnly`, filesystem ACL semantics, media/apply_patch coverage, exec/process scope, sandbox behavior, plugin surfaces, docs, and upgrade tests.

## Reproduction Assessment

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so no live Feishu run is needed to prove the core contract gap.

## Solution Assessment

No for a quick fix. A one-line default flip would not cover exec/process/plugins/media/sandbox semantics and could break existing absolute-path workflows without an explicit upgrade policy.
