## Summary

Keep open. Current main and the latest shipped release still make workspace confinement opt-in, preserve unrestricted host read/write/edit behavior when `tools.fs.workspaceOnly` is unset or false, and the related filesystem-policy PRs remain open rather than merged.

## What This Changes

The issue requests a hard per-agent filesystem boundary so agents cannot read or modify files outside their own workspace.

## Best Possible Solution

Land an owner-reviewed per-agent confinement policy that aligns `tools.fs` defaults or ACLs, exec/sandbox semantics, plugin and media tool behavior, subagent inheritance, docs, doctor/audit warnings, and upgrade tests.

## Reproduction Assessment

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so a live Feishu run is not needed to prove the boundary gap.

## Solution Assessment

No for a quick fix. A default flip or narrow file-tool patch would not settle exec/process/plugins/media/sandbox behavior or existing-user compatibility without maintainer security and product review.
