## Summary

Keep open. Current main still treats an agent workspace as the default working directory plus opt-in guards, not as a hard per-agent confinement boundary; docs, config, runtime wiring, and tests continue to allow out-of-workspace file access when `tools.fs.workspaceOnly` is unset or false, and a complete fix needs security/product decisions across filesystem, sandbox, exec/process, media, MCP/plugin, and compatibility surfaces.

## What This Changes

The issue requests hard per-agent filesystem confinement so an agent cannot read or modify paths outside its own workspace.

## Best Possible Solution

Define a maintainer-approved confinement profile for agent/channel setups like the PM/Feishu bot: make workspace-scoped filesystem access explicit by default or during onboarding, deny or sandbox runtime tools where needed, cover plugin/MCP/media paths, make the active workspace visible to skills, and add regression coverage for cross-workspace reads/writes while preserving an intentional trusted-host escape hatch.
