## Summary

Keep open. Current main and the latest release still treat the workspace as a default working directory plus optional filesystem/sandbox guards, not as a hard per-agent confinement boundary, and the remaining work is security-sensitive policy spanning file tools, exec/process, plugins, media paths, and sandbox behavior.

## What This Changes

The issue requests hard per-agent filesystem confinement so agents cannot access or modify files outside their assigned workspace.

## Best Possible Solution

Define a maintainer-approved per-agent confinement policy that coordinates default filesystem scope, sandbox/default profile behavior, exec and plugin coverage, media path handling, skill workspace messaging, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `tools.fs.workspaceOnly` unset or false, current source and tests allow host read/write/edit paths outside the workspace; I did not run a live Feishu scenario because the source and test contract is explicit.

## Solution Assessment

No. The current implementation is not the best solution for the reported security expectation because it provides opt-in guardrails rather than a default per-agent confinement contract, and a one-line filesystem default flip would not cover runtime, plugin, media, sandbox, and compatibility behavior.
