## Summary

Keep open. Current main and the latest shipped tag still keep host filesystem confinement opt-in through `tools.fs.workspaceOnly`; unset or false preserves out-of-workspace read/write/edit behavior, while the related filesystem-policy PRs remain open and security/compatibility-sensitive.

## What This Changes

The issue asks for per-agent filesystem confinement so agents cannot read or mutate paths outside their own workspace.

## Best Possible Solution

Land an owner-reviewed per-agent confinement policy that aligns file tools, apply_patch, local media, exec/process semantics, sandbox mode, subagent inheritance, docs, warnings, and upgrade tests.

## Reproduction Assessment

Yes, source-reproducible. Current source and regression tests show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so a live Feishu run is not needed to prove the boundary gap.

## Solution Assessment

No for a quick fix. A default flip or narrow file-tool patch would not settle exec/process/plugins/media/sandbox behavior or existing-user compatibility without maintainer security and product review.
