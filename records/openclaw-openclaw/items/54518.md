## Summary

Keep open. Current main and v2026.5.12 still make per-agent workspace confinement opt-in rather than a hard boundary, and the related hardening PRs remain open with security/design blockers.

## What This Changes

The issue requests hard per-agent filesystem confinement so agents cannot read or modify paths outside their own workspace.

## Best Possible Solution

Define and land an owner-reviewed confinement policy that aligns `tools.fs.workspaceOnly`, sandbox defaults, file tools, media loads, exec/process coverage, plugin surfaces, docs, and regression tests.

## Reproduction Assessment

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit paths outside the workspace; no live Feishu run is needed to prove that contract.

## Solution Assessment

No. A one-line default flip is not the best complete fix because the security boundary spans defaults, compatibility, read/write/edit/apply_patch, media, exec/process, plugins, and sandbox behavior.
