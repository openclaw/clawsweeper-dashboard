## Summary

Keep open. Current main still makes host file-tool workspace confinement opt-in, tests preserve outside-workspace access when `tools.fs.workspaceOnly` is false or unset, and the related filesystem-policy and local-user isolation work remains open and security-sensitive.

## What This Changes

The issue asks OpenClaw to prevent an agent from reading or modifying files outside that agent's own workspace by default.

## Best Possible Solution

Define and land an owner-reviewed confinement policy that aligns `tools.fs.workspaceOnly`, filesystem ACL semantics, media/apply_patch coverage, exec/process scope, sandbox behavior, plugin surfaces, docs, and upgrade tests.

## Reproduction Assessment

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so a live Feishu run is not needed to prove the core boundary gap.

## Solution Assessment

No for a quick one-line fix. Flipping the default would not cover exec/process/plugins/media/sandbox semantics and could break existing absolute-path workflows without an explicit upgrade policy.
