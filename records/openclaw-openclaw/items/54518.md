## Summary

Keep open: current main and v2026.6.8 still leave host filesystem confinement opt-in, so read/write/edit can touch paths outside an agent workspace unless operators explicitly enable `tools.fs.workspaceOnly`; related security-policy issues and PRs overlap but do not safely replace this concrete workspace-confinement report.

## What This Changes

The issue requests enforced per-agent filesystem boundaries so agents cannot read or modify files outside their own workspace.

## Best Possible Solution

Keep this as the concrete workspace-confinement tracker while maintainers choose the security default, upgrade story, and exec/sandbox boundary, then land the chosen policy with docs and regression coverage.

## Reproduction Assessment

Yes at source level: unset `tools.fs.workspaceOnly` resolves to false, and tests intentionally allow host read/write/edit outside the workspace when that flag is unset or false. I did not run a live Feishu/minimax reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path: flipping `tools.fs.workspaceOnly` by default is plausible but compatibility-sensitive, and it does not settle exec or sandbox behavior. Maintainers should choose the security/product policy before implementation.
