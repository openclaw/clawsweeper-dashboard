## Summary

Keep open. Current main still treats each agent workspace as the default working directory plus opt-in guards, not as a hard per-agent confinement boundary; filesystem access outside the workspace remains allowed when `tools.fs.workspaceOnly` is unset or false, and sandboxing remains optional.

## What This Changes

The issue requests hard per-agent filesystem confinement so agents cannot read or modify paths outside their own workspace.

## Best Possible Solution

Define a maintainer-approved per-agent confinement profile for channel-bound agents like the PM/Feishu bot: make workspace-scoped filesystem access explicit, deny or sandbox runtime tools where needed, cover plugin/MCP/media paths, surface the active workspace clearly to skills, and add regression coverage for cross-workspace reads/writes while preserving intentional trusted-host workflows.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `tools.fs.workspaceOnly` unset or false, the host file tools are assembled without the workspace guard and current tests assert out-of-workspace read/write/edit behavior succeeds.

## Solution Assessment

No. Current main is not the best solution for the reported security expectation because it provides opt-in guardrails rather than default per-agent confinement; the safer fix needs a coordinated config, sandbox, filesystem, runtime-tool, docs, and test contract rather than a one-line default flip.
