## Summary

Keep open. Current main still treats the agent workspace as the default working directory plus optional guards, not as a hard per-agent confinement boundary, and changing that default is security-sensitive product policy across filesystem, runtime, sandbox, plugin, and media surfaces.

## What This Changes

Request that channel-bound agents be prevented from reading or modifying files outside their own configured workspace.

## Best Possible Solution

Define a maintainer-approved confinement profile for untrusted/channel-bound agents, then apply it consistently across filesystem, runtime, sandbox, plugin, media, docs, and security-audit surfaces with regression coverage.

## Reproduction Assessment

Yes. Source and current tests show that with tools.fs.workspaceOnly unset or false, host read/write/edit paths outside the workspace are allowed; I did not run the Feishu live path because the source contract already proves the remaining behavior.

## Solution Assessment

No. The current opt-in guardrails are not the best solution for the reported security expectation; maintainers need to choose the default confinement and compatibility policy before a narrow implementation is safe.
