## Summary

Keep open. Current main and v2026.5.19 still keep host filesystem confinement opt-in through `tools.fs.workspaceOnly`, preserve out-of-workspace host file behavior when unset or false, and the related filesystem-policy/security PRs are still open rather than merged.

## What This Changes

The issue requests hard per-agent workspace confinement so agents cannot read or modify files outside their own working space.

## Best Possible Solution

Land an owner-reviewed per-agent confinement policy that aligns file tools, exec/process semantics, plugin/media behavior, sandbox mode, subagent inheritance, docs, audit warnings, and upgrade tests.

## Reproduction Assessment

Yes, source-reproducible. Current source and regression tests show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so a live Feishu run is not needed to prove the boundary gap.

## Solution Assessment

No for a quick fix. A default flip or narrow file-tool patch would not settle exec/process/plugins/media/sandbox behavior or existing-user compatibility without maintainer security and product review.
