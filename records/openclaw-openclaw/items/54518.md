- `impact:security`: The issue is about whether agents can cross filesystem boundaries and access host or other-agent data.

## Summary

Keep open. Current main and v2026.5.18 still treat each agent workspace as the default cwd rather than a hard sandbox, and source/tests preserve unrestricted read/write/edit access unless `tools.fs.workspaceOnly` is explicitly enabled. Related hardening work remains open and security/product-sensitive, so this should not be swept closed.

## What This Changes

The issue asks OpenClaw to enforce per-agent workspace confinement so agents cannot read or modify other workspace or host files by default.

## Best Possible Solution

Define and land an owner-reviewed confinement policy that aligns `tools.fs.workspaceOnly`, defaults or migration behavior, file tools, apply_patch, local media, exec/process scope, plugin surfaces, sandbox docs, and regression proof.

## Reproduction Assessment

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit paths outside the workspace, and docs explicitly describe workspaces as cwd rather than a hard sandbox.

## Solution Assessment

No for a quick default flip alone. The maintainable fix needs a security/product decision that covers compatibility plus read/write/edit/apply_patch/media and exec or explicitly splits the exec boundary into a tracked follow-up.
