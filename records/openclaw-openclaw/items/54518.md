## Summary

Keep open: current main and v2026.6.10 still treat the agent workspace as a default cwd, not a hard filesystem boundary. Host read/write/edit remain able to touch paths outside the workspace unless operators enable `tools.fs.workspaceOnly` or sandboxing, and the related PRs/issues are partial, open, conflicting, or closed unmerged rather than a safe replacement.

## What This Changes

The issue asks OpenClaw to enforce per-agent workspace confinement so agents cannot read or modify files outside their assigned workspace.

## Best Possible Solution

Keep this as the concrete workspace-confinement tracker while maintainers choose the default confinement, upgrade compatibility, and exec/sandbox boundary, then land the chosen policy with docs and regression coverage.

## Reproduction Assessment

Yes from source: with `tools.fs.workspaceOnly` unset or false, host read/write/edit are built without the workspace guard and tests assert outside-workspace access succeeds. I did not run a live Feishu/minimax reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path: flipping `tools.fs.workspaceOnly` by default is plausible but compatibility-sensitive and does not settle exec/process access. The safer next step is a maintainer security/product decision that defines the default, migration, and scope.
