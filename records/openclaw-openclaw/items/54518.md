## Summary

Keep open. Current main still documents and tests workspace confinement as opt-in, host read/write/edit can touch outside paths when `tools.fs.workspaceOnly` is unset or false, and the related filesystem-policy PRs are open rather than shipped; the remaining work is security/product/compatibility review, not cleanup.

## What This Changes

The issue requests hard per-agent filesystem confinement so agents cannot access or modify files outside their own workspace.

## Best Possible Solution

Land an owner-reviewed confinement policy that aligns `tools.fs` defaults or ACLs, sandbox/exec semantics, plugin and media tool behavior, subagent inheritance, docs, doctor/audit warnings, and upgrade tests.

Yes. Source and tests on current main show unset or false `tools.fs.workspaceOnly` allows host read/write/edit outside the workspace, so a live Feishu run is not needed to prove the core boundary gap.

## Solution Assessment

No for a quick fix. A default flip or narrow file-tool patch would not settle exec/process/plugins/media/sandbox behavior or existing-user compatibility without maintainer security and product review.
