## Summary

Keep open: the related closed report covers raw tool-output handoff silence, but this newer report describes a distinct post-tool internal Codex shutdown path where current main can still fall back to the much longer terminal/supervisor timeout instead of a prompt terminal event.

## What This Changes

Investigate a Codex bridge stall where internal Codex `op.dispatch.shutdown` aborts a running task without sending `turn/completed`, `error`, or another terminal event to OpenClaw.

## Best Possible Solution

Add a narrowly scoped Codex app-server liveness or terminal-abort fallback, or consume an upstream Codex shutdown terminal event, so silent internal Codex aborts release the OpenClaw session without waiting for the broad supervisor timeout.

## Reproduction Assessment

Yes at source level, but not as a live end-to-end repro: current main has a clear path where post-tool raw assistant progress waits for the long terminal/supervisor fallback if no terminal event follows. The exact internal Codex `op.dispatch.shutdown` trigger still needs live Codex proof or a targeted harness simulation.

## Solution Assessment

Unclear. A Codex CLI terminal abort notification is the cleanest ownership boundary, but an OpenClaw fallback may be the practical safety net if Codex can silently abort while stdio remains open.
