## Summary

Keep open: current main covers the prior raw tool-output watchdog gap, but it still does not surface or quickly recover from a Codex internal shutdown that emits no terminal notification after post-tool assistant progress. The remaining behavior is a real Codex bridge liveness gap, not stale or already implemented.

## What This Changes

The issue requests handling for Codex CLI internal `op.dispatch.shutdown` aborts so the OpenClaw Codex bridge receives or derives a terminal event instead of leaving the session in `processing` until stuck-session recovery.

## Best Possible Solution

Have Codex emit a terminal shutdown/abort notification for this internal abort, or add a protocol-grounded OpenClaw liveness fallback with focused regression coverage for silent post-tool shutdowns.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not from a live Codex process in this pass, but source inspection gives a high-confidence bridge-side path: post-tool raw assistant progress can disarm the short completion watchdog, and a silent internal shutdown then leaves only the long terminal/supervisor fallback.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the report points to a real gap, but the cleanest fix appears to be a Codex-side terminal shutdown event, while an OpenClaw fallback needs owner judgment on timeout semantics.
