## Summary

Keep open. The PR addresses a valid Feishu current-main bug and includes sufficient live proof, but the latest diff still derives `replyInThread` from raw `replyToId`, so blank `replyToId` plus `threadId` can lose topic-thread delivery in the new outbound paths.

## What This Changes

This PR auto-threads Feishu `group_topic` and `group_topic_sender` `message(action="send")` replies against the inbound message and propagates `replyInThread` through text, card, and media sends with tests.

## Best Possible Solution

Land the Feishu-owned runtime fix after normalizing reply-mode derivation from the same trimmed `replyToId`/`threadId` decision and adding focused regression coverage for the blank-`replyToId` case.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: group context instructs `message(action=send)`, while Feishu action dispatch only threads explicit `thread-reply` calls.

## Solution Assessment

Yes, with one code repair. The action-handler/outbound propagation path is the narrow Feishu-owned fix because it preserves topic delivery even when the model follows the existing `send` instruction; the new reply-mode predicate needs normalization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Normalize replyToId before deriving replyInThread:** `extensions/feishu/src/outbound.ts:421`
  - body: `resolveReplyToMessageId()` treats whitespace `replyToId` as absent and falls back to `threadId`, but this new `replyInThread` check uses the raw `replyToId`. With `replyToId: " "` and a valid `threadId`, Feishu sends with the topic root as the reply target but omits `reply_in_thread`, so the reply can leave the topic-thread path. Derive the thread flag from the same trimmed reply-mode decision and cover this edge case for text/card/media.
  - confidence: 0.92
