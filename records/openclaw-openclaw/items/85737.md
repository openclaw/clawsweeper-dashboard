## Label Justifications

- `P1`: The PR changes live channel delivery policy and currently risks sending media when an existing deny-mode delivery boundary should block it.
- `merge-risk: 🚨 message-delivery`: The changed gate can wrongly deliver media-bearing tool results through a suppression path that currently blocks delivery.

## Summary

Keep open: the root-cause direction is credible, but the patch currently lets media payloads bypass `sendPolicy: deny` because the new exemption is applied to `shouldSuppressProgressDelivery()`, not only to message-tool-only source suppression. The PR also has unit-test-only proof for a Discord/Telegram runtime behavior change.

## What This Changes

The PR moves TTS payload processing before tool-result delivery suppression in `src/auto-reply/reply/dispatch-from-config.ts` and allows media-bearing tool results through message-tool-only suppression.

## Best Possible Solution

Process TTS before source-reply suppression, but exempt media only from message-tool-only source suppression when send policy allows delivery, with regression coverage for both TTS delivery and deny-mode media suppression.

## Reproduction Assessment

Yes, source inspection shows current `main` suppresses tool-result handling before TTS processing, and the linked issue provides live Discord and Telegram traces. I did not run a live reproduction in this read-only review.

## Solution Assessment

No as submitted. Moving TTS processing before message-tool-only suppression is the right direction, but the exemption must preserve `sendPolicy: deny` and needs focused regression and live behavior proof.
