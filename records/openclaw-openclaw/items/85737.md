## Summary

Keep open: the updated patch fixes the earlier direct media-delivery bypass, but it still moves TTS/media processing ahead of the `sendPolicy: deny` gate and the supplied proof does not exercise the changed dispatch path in a real channel.

## What This Changes

The PR moves TTS/media handling before tool-result source suppression and allows media-bearing tool results through `message_tool_only` suppression while adding a later `sendPolicyDenied` return.

## Best Possible Solution

Move the `sendPolicyDenied` return before TTS/media processing, keep the media exemption limited to message-tool-only source suppression, add focused deny-mode and message-tool-only media regression coverage, then verify a real Telegram or Discord TTS delivery path.

## Reproduction Assessment

Yes. The linked issue has concrete Discord and Telegram traces, and source inspection shows current main suppresses tool-result handling before TTS/media delivery in `message_tool_only` contexts; I did not run a live channel repro in this read-only review.

## Solution Assessment

No as submitted. Moving TTS processing before message-tool-only suppression is the right direction, but the deny gate should remain before TTS/media side effects and the changed channel path still needs real behavior proof.
