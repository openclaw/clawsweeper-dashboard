## Summary

Keep open. Current main still implements and documents the existing sandbox workspaceAccess contract, while this request changes the public sandbox filesystem/security model across backends rather than being fully covered by the narrower writable-none follow-ups.

## What This Changes

Requests refactoring `agents.defaults.sandbox.workspaceAccess` so `/workspace` always represents the session sandbox, `none` skips workspace mounts, `ro`/`rw` only control sandbox mount permissions, and the main workspace is exposed only through explicit binds.

## Best Possible Solution

Define the intended public workspaceAccess contract with sandbox/security maintainers first, then implement it consistently across layout resolution, Docker/SSH/OpenShell mounts, filesystem bridges, memory/bootstrap behavior, tests, and public docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: sandboxed runs with `workspaceAccess` set to `none` or `ro` still resolve read-only workspace mounts and the local/remote FS bridges reject write/edit operations with the reported read-only sandbox path error. I did not run a live Docker/Lark reproduction in this read-only review.

## Solution Assessment

Unclear. The requested direction is coherent, but it changes the public sandbox security contract; the maintainable path is to approve the semantics first, then make a coordinated backend/docs/test update rather than landing a narrow writable-none patch as the full solution.
