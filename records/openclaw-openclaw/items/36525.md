## Summary

Keep open. Current main still implements and documents the existing sandbox workspaceAccess contract, while this issue asks for a broader sandbox/security semantics change that is not fully covered by the narrower writable-none work in #37634 or open PR #46300.

## What This Changes

The issue requests refactoring agents.defaults.sandbox.workspaceAccess so sandbox /workspace always refers to a session sandbox, none omits the mount, ro mounts it read-only, rw mounts it writable, and the main workspace is never implicitly exposed.

## Best Possible Solution

Define the intended public workspaceAccess contract first, then either document the current model explicitly or implement the accepted new model coherently across layout resolution, all sandbox backends, filesystem bridges, memory/bootstrap behavior, tests, and public docs.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with sandboxing enabled and workspaceAccess set to none or ro, current mount builders and FS bridges mark /workspace read-only and reject write/edit operations with the reported read-only sandbox path error.

## Solution Assessment

Unclear. The requested direction may be a maintainable contract, but it is a sandbox/security semantics decision; the narrower #46300-style writable-none fix is not the full solution because it does not address rw mapping to the main workspace or /agent exposure.
