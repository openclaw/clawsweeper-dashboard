## Summary

Keep open. Current main still implements the existing sandbox workspace contract, and the requested change remains a broader sandbox/security semantics decision rather than something already shipped or safely closeable as a duplicate of the narrower writable-`none` work.

## What This Changes

The issue requests a semantic refactor of `agents.defaults.sandbox.workspaceAccess` so sandbox `/workspace` always refers to the per-session sandbox directory, `none` does not mount it, `ro` mounts it read-only, `rw` mounts it writable, and the main agent workspace is never implicitly exposed.

## Best Possible Solution

Have sandbox/security maintainers decide the intended contract first. If accepted, implement the new semantics coherently across sandbox layout resolution, Docker/browser/SSH/OpenShell mount builders, local and remote FS bridge write gates, memory/bootstrap behavior, tests, config/status output, and public sandbox docs; otherwise document the current contract and close only after that product decision.
