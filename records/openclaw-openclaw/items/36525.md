## Summary

Keep open because current `main` still has the reported sandbox write behavior and the broader requested `workspaceAccess` semantic change is not already implemented. The related open PR #46300 covers a narrower writable-`none` direction, but the issue also asks to stop using `rw` to expose the main workspace, which needs maintainer product/security judgment.

## What This Changes

Refactor `agents.defaults.sandbox.workspaceAccess` so `/workspace` is always the session sandbox with independent mount permissions and main workspace access requires explicit `docker.binds`.

## Best Possible Solution

Settle the sandbox workspace-access contract, then update Docker/OpenShell/browser mounts, FS bridge behavior, tests, and docs consistently, either by revising #46300 or replacing it with the broader approved design.

## Reproduction Assessment

Yes, source-reproducible on current `main`: with sandbox enabled and `workspaceAccess: "none"`, the sandbox workspace is selected but mounted and modeled read-only, so write/mkdir calls reject before Docker execution.

## Solution Assessment

Unclear. The requested direction is coherent, but it conflicts with the current documented meaning of `rw`, so maintainers should choose the contract before accepting code.
