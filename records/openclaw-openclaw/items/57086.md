## Summary

Keep open: current main and v2026.5.7 already supersede the gateway-agent part with a stronger freshness fix, but this branch also changes node-event dispatch and that proposed repair is incomplete. The external PR also lacks real behavior proof and is merge-conflicting, so it needs maintainer follow-up rather than cleanup closure.

## What This Changes

The branch suppresses forwarded `sessionId` values for gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a canonical `sessionKey` is present, with focused tests.

## Best Possible Solution

Keep the shipped gateway-agent freshness implementation, then repair or split node-event session freshness so id selection and store touches honor lifecycle timestamps before dispatch.

## Reproduction Assessment

Yes, at source level for the remaining review findings: stale keyed `voice.transcript` or `agent.request` rows can still be touched and forwarded with an old `sessionId` before downstream freshness selection. I did not run a live stale-session gateway flow in this read-only review.

## Solution Assessment

No. Current main's gateway-agent fix is the better central solution, while the branch's node-event change only deletes the ingress field; the safer fix is to evaluate/select the fresh session id before touching the store or defer that write to session resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve freshness before voice dispatch:** `src/gateway/server-node-events.ts:304`
  - body: Deleting the forwarded `sessionId` is not enough while this handler still selects `entry.sessionId` and queues a store touch before dispatch. For stale legacy rows without lifecycle/header timestamps, that touch can refresh `updatedAt` before `resolveSession` evaluates freshness, so resolve/select the fresh id first or defer the touch.
  - confidence: 0.8
- **[P2] Resolve freshness before agent-request dispatch:** `src/gateway/server-node-events.ts:436`
  - body: The `agent.request` path still chooses `entry.sessionId` and awaits `touchSessionStore` before handoff. Removing only the ingress field leaves stale legacy rows refreshed before freshness evaluation, so move freshness-aware selection before the touch or let the session resolver own the write.
  - confidence: 0.79
