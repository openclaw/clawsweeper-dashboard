## Summary

Keep open for maintainer follow-up: current main and v2026.5.7 already solve the central gateway-agent stale-session bypass, but this PR also carries node-event changes that are still incomplete and lacks real after-fix behavior proof.

## What This Changes

The PR suppresses forwarded `sessionId` values for gateway agent, `voice.transcript`, and `agent.request` dispatches when a canonical `sessionKey` is present, with focused test assertions.

## Best Possible Solution

Keep the shipped gateway-agent freshness implementation, then either repair the node-event handlers with freshness-aware id selection and rollover coverage or split that work into a fresh follow-up.

## Reproduction Assessment

Yes, at source level. A stale keyed `voice.transcript` or `agent.request` row that lacks reliable lifecycle/header timestamps can still be touched before downstream freshness evaluation, and current code shows the old `sessionId` is selected before dispatch.

## Solution Assessment

No. Suppressing the forwarded field is directionally useful, but the node-event paths need to resolve/select freshness before touching the store, or defer that touch to the session-resolution path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve freshness before voice dispatch:** `src/gateway/server-node-events.ts:304`
  - body: Deleting the forwarded `sessionId` is not enough while this handler still selects `entry.sessionId` and queues a store touch before dispatch. For stale legacy rows without lifecycle/header timestamps, that touch can refresh `updatedAt` before `resolveSession` evaluates freshness, so the old session may still be reused; resolve/select the fresh id first or defer the touch.
  - confidence: 0.8
- **[P2] Resolve freshness before agent-request dispatch:** `src/gateway/server-node-events.ts:436`
  - body: The `agent.request` path still chooses `entry.sessionId` and awaits `touchSessionStore` before handoff. Removing only the ingress field leaves stale legacy rows refreshed before freshness evaluation, so move freshness-aware selection before the touch or let the session resolver own the write.
  - confidence: 0.79
