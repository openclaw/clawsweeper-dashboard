## Summary

Keep open: the stale-session problem is real, current main still leaves the node-event paths exposed, and this PR is useful source work. The submitted branch is not merge-ready because it is dirty against current main and the node-event change removes the explicit ingress `sessionId` without fixing the earlier store touch that can still make a stale entry look fresh.

## What This Changes

This PR removes forwarded `sessionId` values from gateway agent and node-event ingress dispatch when a canonical `sessionKey` is present, and updates gateway/node-event tests to assert session-key-only handoff.

## Best Possible Solution

Rebase or replace the branch with a narrow node-event fix that evaluates freshness before touching the session store, or delegates session resolution before refreshing the old entry. Keep the current gateway-agent freshness guard, and add stale-entry regression tests for both `voice.transcript` and `agent.request` that prove a stale stored id is not reused or restored.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to seed a node-event session store entry with an old `updatedAt` and stored `sessionId`, trigger `agent.request` or `voice.transcript` by canonical `sessionKey`, and observe whether the downstream resolver rolls to a new id; the current and proposed code paths show why the pre-dispatch touch is material.

## Solution Assessment

No. Removing `sessionId` from the ingress options is directionally right, but it is not sufficient for node events while the handler still refreshes the stored entry before `resolveSession()` can evaluate staleness; freshness selection needs to move ahead of the touch or the touch needs to happen after resolution with the selected id.
