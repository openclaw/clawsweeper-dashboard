## Summary

Keep open. Current main already has a stronger gateway-agent freshness fix, but this PR still carries adjacent node-event scope, the node-event change is incomplete, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR suppresses forwarded `sessionId` values for gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a `sessionKey` is present, with focused gateway and node-event tests.

## Best Possible Solution

Keep the current main gateway-agent freshness implementation, then rework the node-event paths so freshness is resolved before store touches or the session resolver owns the write.

## Reproduction Assessment

Yes, at source level for the PR review findings: the node-event handlers select/touch the old session id before dispatch, while `resolveSession` gives an explicit `sessionId` priority over freshness. I did not run a live stale-session gateway flow in this read-only review.

## Solution Assessment

No. Suppressing the forwarded ingress field is not enough for node events because the stale id is selected and the store can be touched before downstream freshness selection; the narrower durable fix is freshness-aware selection before that write or delegating the write to session resolution.

## Review Findings
