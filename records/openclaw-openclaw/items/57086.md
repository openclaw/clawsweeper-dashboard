## Summary

Keep open because the branch still has merge-blocking review issues: current main already has a stronger gateway-agent freshness fix, while the added node-event changes only remove an ingress field and do not resolve the stale pre-dispatch store-touch path.

## What This Changes

The PR suppresses forwarded `sessionId` values for gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a `sessionKey` is present, with focused gateway and node-event tests.

## Best Possible Solution

Keep the current main gateway-agent freshness implementation, then rebase or replace the node-event part with a focused lifecycle-aware fix that resolves freshness before store touches or lets session resolution own the write.

## Reproduction Assessment

Yes, at source level for the remaining review findings: stale keyed `voice.transcript` or `agent.request` rows can still be touched with the old `sessionId` before downstream freshness selection. I did not run a live stale-session gateway flow in this read-only review.

## Solution Assessment

No. Current main's gateway-agent fix is the better central solution, while the PR's node-event change only deletes the ingress `sessionId`; the safer fix is to select freshness before touching the store or defer that write to session resolution.
