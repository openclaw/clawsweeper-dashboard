## Summary

Keep open: current main has the stronger gateway-agent freshness fix, but this PR still leaves the node-event paths incomplete and lacks contributor real behavior proof.

## What This Changes

The branch suppresses forwarded `sessionId` values for gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a `sessionKey` is present, with focused test updates.

## Best Possible Solution

Keep current main's freshness-aware gateway-agent implementation, then rework node-event handlers so freshness is resolved before any store touch or the session resolver owns that write.

## Reproduction Assessment

Yes, at source level: the node-event handlers select/touch the stored session id before dispatch, while legacy freshness can fall back to `updatedAt`. I did not run a live stale-session gateway rollover in this read-only review.

## Solution Assessment

No. Suppressing the forwarded `sessionId` is not sufficient for node events because the stale id and store write are chosen before downstream freshness resolution; the narrower durable fix is freshness-aware selection before that write or delegating the write to the resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

For legacy rows without lifecycle/header timestamps, that touch can refresh `updatedAt` before freshness is evaluated, so resolve freshness before the touch or defer the write.
  - confidence: 0.8

Deleting only the ingress field leaves stale legacy rows refreshed before freshness evaluation, so move freshness-aware selection before the touch or let session resolution own the write.
  - confidence: 0.79
