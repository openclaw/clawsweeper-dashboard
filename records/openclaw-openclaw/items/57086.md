## Summary

Keep open because current main already shipped a stronger gateway-agent freshness fix, but this PR still carries an incomplete node-event freshness change and lacks after-fix real behavior proof.

## What This Changes

The PR suppresses forwarded `sessionId` values for gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a `sessionKey` is present, with focused test updates.

## Best Possible Solution

Keep current main's shipped gateway-agent freshness implementation, then rework node-event handlers so freshness is resolved before any store touch or the session resolver owns that write.

## Reproduction Assessment

Yes, at source level for the remaining PR findings: the node-event handlers select and touch the stored session id before handoff, while freshness can fall back to `updatedAt`. I did not run a live stale-session rollover in this read-only review.

## Solution Assessment

No. Suppressing the forwarded `sessionId` is not sufficient for node events because the stale id and store write are chosen before downstream freshness resolution; the durable fix is freshness-aware selection before that write or delegating the write to the resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

  - body: Removing the forwarded `sessionId` is not enough while this handler still selects `entry.sessionId` and queues `touchSessionStore` before dispatch. For legacy rows without lifecycle/header timestamps, that touch can refresh `updatedAt` before freshness is evaluated, so resolve freshness before the touch or defer the write.
  - confidence: 0.8

Deleting only the ingress field leaves stale legacy rows refreshed before freshness evaluation, so move freshness-aware selection before the touch or let session resolution own the write.
  - confidence: 0.79
