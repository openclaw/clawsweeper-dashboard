## Summary

Keep open. Current main already fixed the gateway-agent half with a freshness-aware implementation, but this PR still carries meaningful node-event work and its node-event patch is incomplete because it drops the forwarded field without moving freshness selection ahead of the pre-dispatch store touch.

## What This Changes

The PR removes stored `sessionId` forwarding from gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a canonical `sessionKey` is present, with focused regression assertions.

## Best Possible Solution

Rebase on current main, keep the shipped gateway-agent freshness guard, and repair node-event handlers to evaluate or select freshness before any store touch or session-id reuse, with rollover tests.

## Reproduction Assessment

Yes, at source level. Seed a stale keyed legacy/no-header session row and trigger `voice.transcript` or `agent.request`; current node-event code selects and touches the stored session id before downstream freshness can own rollover.

## Solution Assessment

No, not as written. Removing the ingress `sessionId` is directionally useful, but the node-event paths need freshness-aware id selection before touching the store or should defer the touch until the gateway-agent/session path owns it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Resolve freshness before voice dispatch:** `src/gateway/server-node-events.ts:416-417`
  - body: Dropping the `sessionId` field from the voice ingress call is not enough while this handler still selects `entry?.sessionId` and queues `touchSessionStore()` before dispatch. For legacy rows without lifecycle/header timestamps, that touch can advance `updatedAt` and make downstream freshness treat the stale row as fresh; resolve/select the fresh id before the touch or defer the touch until session resolution owns it.
  - confidence: 0.8
- **[P2] Resolve freshness before agent-request dispatch:** `src/gateway/server-node-events.ts:589-590`
  - body: The `agent.request` path still chooses `entry?.sessionId` and awaits `touchSessionStore()` before handing off to `agentCommandFromIngress()`. Removing the ingress field leaves the stale row refreshed before freshness evaluation, so legacy/fallback rows can continue instead of rolling over; move freshness/id selection ahead of the touch or make the touch use the selected fresh session.
  - confidence: 0.79
