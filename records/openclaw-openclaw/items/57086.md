## Summary

Keep this PR open. Current main already has a different freshness-before-session-id guard for the gateway `agent` handler, but it does not cover the PR author's follow-up `server-node-events` scope, and the proposed node-event handoff needs repair so pre-dispatch session-store touches cannot mask stale lifecycle state.

## What This Changes

The PR branch removes stored `sessionId` forwarding from gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a canonical `sessionKey` is available, with regression assertions for those handoffs.

## Best Possible Solution

Keep the current main gateway-agent freshness guard, port the node-event fix in the same style by resolving stale lifecycle state before choosing or forwarding any session id, and add focused stale daily/idle rollover tests for both `voice.transcript` and `agent.request` rather than only asserting that the forwarded field is undefined.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a stale keyed session entry, trigger `voice.transcript` or `agent.request`, and observe current main forwarding the stored `sessionId`; `resolveSession()` then treats that explicit id as authoritative and does not roll the session.

## Solution Assessment

No, not as-is. The direction is useful, but the branch should be rebased onto current main and repaired so node-event pre-dispatch store touches cannot interfere with lifecycle freshness; the safer pattern is the current gateway handler's freshness-before-id selection plus node-event regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.77

Full review comments:

- **[P2] Resolve freshness before touching voice sessions:** `src/gateway/server-node-events.ts:307`
  - body: This removes `sessionId` from the voice transcript ingress call, but the handler still starts a session-store touch before dispatch. For rows that depend on fallback lifecycle timestamps, that touch can update bookkeeping before `resolveSession()` evaluates freshness, so a stale transcript can still be reused. Resolve/select the session id before the pre-dispatch touch, or defer the touch until the agent run owns the session update.
  - confidence: 0.78
- **[P2] Resolve freshness before touching agent requests:** `src/gateway/server-node-events.ts:439`
  - body: The `agent.request` path now omits `sessionId` when calling `agentCommandFromIngress()`, but it still awaits `touchSessionStore()` first. That pre-touch can make legacy or fallback-timestamp rows look freshly updated before downstream freshness checks run, leaving the stale session pinned despite the removed field. Move freshness/id selection ahead of this touch or make the touch use the selected fresh id.
  - confidence: 0.77
