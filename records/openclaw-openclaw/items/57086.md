## Summary

Keep open. Current main already fixes the original gateway `agent` stale-session path with a freshness-before-id selection, but this PR also tries to cover `voice.transcript` and `agent.request`; those node-event paths still need a freshness-first repair before the patch is mergeable.

## What This Changes

This PR removes stored `sessionId` forwarding from gateway `agent`, `voice.transcript`, and `agent.request` dispatches when a canonical `sessionKey` is present, with focused regression assertions.

## Best Possible Solution

Keep the current main gateway-agent freshness guard, rebase this PR, and fix the node-event paths by resolving daily/idle freshness before choosing or touching a session id, with rollover tests and a changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: create a stale keyed session row, trigger `voice.transcript` or `agent.request`, and the current node-event path selects/touches the stored `sessionId` before downstream freshness can choose a fresh one.

## Solution Assessment

No, not as-is. Removing `sessionId` from the ingress call is directionally useful, but the node-event handlers must evaluate freshness before pre-dispatch store touches so legacy/fallback lifecycle rows cannot be made fresh by bookkeeping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Resolve freshness before voice dispatch:** `src/gateway/server-node-events.ts:307`
  - body: Dropping `sessionId` from the voice ingress call is not enough while this handler still selects the stored id and queues a store touch before dispatch. For stale rows that fall back to `updatedAt`, that touch can make `resolveSession()` see the old row as fresh, so voice transcript runs can continue the expired session. Resolve/select the fresh id before the touch or defer the touch until after session resolution owns it.
  - confidence: 0.78
- **[P2] Resolve freshness before agent-request dispatch:** `src/gateway/server-node-events.ts:439`
  - body: The `agent.request` path still awaits `touchSessionStore()` after choosing `entry?.sessionId`, before handing off to `agentCommandFromIngress()`. That pre-touch can refresh legacy/fallback lifecycle rows and leave the stale session pinned even if the ingress `sessionId` field is removed. Move freshness/id selection before the store touch or make the touch use the selected fresh session.
  - confidence: 0.77
