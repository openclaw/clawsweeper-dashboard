## Summary

Keep open: current main still does not implement this PR's runtime `caffeinate` keep-awake behavior, and the branch still needs maintainer policy approval, real macOS proof, and child-process hardening before merge.

## What This Changes

The PR adds a macOS-only gateway run-loop helper that starts `caffeinate -i -w <pid>` and stops it when the gateway run loop exits.

## Best Possible Solution

If maintainers approve default keep-awake behavior, land a hardened macOS implementation with trusted `caffeinate` resolution, async error handling, focused run-loop coverage, and real macOS proof; otherwise steer users to explicit operator keep-awake helpers.

## Reproduction Assessment

Yes for the PR-introduced defect: the diff lacks a child `error` listener, and a Node probe showed missing executables emit `ENOENT` asynchronously after `spawn()` returns. The claimed macOS idle-sleep benefit is not proven by attached real behavior evidence.

## Solution Assessment

No, not as submitted. A keep-awake helper may be viable, but this implementation needs maintainer approval for default power policy plus child-process hardening and real macOS proof.
