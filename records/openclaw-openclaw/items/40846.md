## Summary

Keep open. The idea is focused, but this PR is not merge-ready: current main does not implement gateway-owned `caffeinate`, the branch now conflicts with the refactored gateway run loop, the helper can crash startup on spawn failure, and the default macOS power-policy change still needs real macOS proof and maintainer approval.

## What This Changes

The PR adds a macOS-only gateway helper that spawns `caffeinate -i -w <pid>` while the gateway loop is running and kills it during loop cleanup.

## Best Possible Solution

If maintainers want gateway-owned macOS keep-awake behavior, land a rebased implementation that hardens child-process error handling, avoids PATH-sensitive helper resolution, covers lifecycle behavior with focused tests, makes the power-policy choice explicit, and includes redacted real macOS proof; otherwise prefer the operator-helper direction in https://github.com/openclaw/openclaw/pull/61464.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Handle caffeinate spawn failures:** `src/cli/gateway-cli/run-loop.ts:35-39`
  - body: `spawn()` reports command execution failures through the returned child process, so this `try/catch` will not handle a missing or inaccessible `caffeinate`. Attach an `error` listener before treating the helper as started, otherwise macOS gateway startup can crash instead of degrading gracefully.
  - confidence: 0.93
- **[P2] Avoid PATH-sensitive caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:35`
  - body: This starts a gateway-managed helper by resolving `caffeinate` from PATH. Use a verified absolute system path such as `/usr/bin/caffeinate` or add an explicit resolution check, otherwise an altered launchd/login PATH can run an unintended binary during gateway startup.
  - confidence: 0.78
