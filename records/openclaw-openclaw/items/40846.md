## Summary

Keep open. Current main still does not start a macOS sleep-prevention helper, and the PR branch has blocking correctness and security issues in the Gateway startup path before it is mergeable.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <process.pid>` and cleans it up when the Gateway exits.

## Best Possible Solution

Revise before landing: after maintainer approval, start the helper only after Gateway lock ownership, resolve `/usr/bin/caffeinate` or `resolveSystemBin`, attach a child `error` listener before `unref()`, keep shutdown cleanup, and add focused tests plus docs/changelog.

## Reproduction Assessment

Yes for the patch-level defects: the diff shows missing child `error` handling and pre-lock startup, current main shows supervised lock retry behavior, and a Node probe confirms missing executable failures are asynchronous. The overnight macOS idle-sleep improvement itself was not reproduced here.

## Solution Assessment

No. The direction may be useful, but this is not the narrowest mergeable solution until command resolution, async spawn-error handling, lock-scoped lifecycle, tests, documentation, changelog, and maintainer approval are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child `error` event, so this `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate after lock ownership:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but supervised lock recovery can catch lock conflicts, wait, and retry in the same parent process. Start the helper only after this process owns the Gateway lock so it cannot keep the Mac awake while yielding to an existing Gateway.
  - confidence: 0.9
- **[P3] Document the macOS power change:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: This changes default Gateway power behavior on macOS, but the PR only edits code and current docs still present `caffeinate` as an operator choice. Add the required changelog entry and update Gateway/macOS docs if maintainers approve this default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the diff introduces a Gateway startup command-execution path that resolves `caffeinate` through PATH.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment choose which executable runs during Gateway startup. Use `/usr/bin/caffeinate` or the existing trusted system-bin resolver with explicit error handling to avoid PATH shadowing in dev shells, launchd wrappers, or service environments.
  - confidence: 0.84
