## Summary

Keep open. Current main still lacks a gateway keep-awake hook, and this PR is a plausible feature PR, but the submitted diff has a blocking spawn-error bug, missing changelog coverage, a small process-launch hardening concern, and needs maintainer approval for the default macOS power policy.

## What This Changes

The PR adds a macOS-only `caffeinate -i -w <pid>` helper to the gateway run loop and stops it when the gateway loop exits.

## Best Possible Solution

Land a maintainer-approved macOS keep-awake implementation with explicit lifecycle ownership, resilient child-process error handling, constrained executable resolution, focused tests, and changelog coverage.

## Reproduction Assessment

Yes for the blocking PR defect: the diff plus a local Node probe show failed spawns emit an asynchronous `error` event and do not throw synchronously. No for the long-duration macOS idle-sleep benefit; that behavior was not live-verified here.

## Solution Assessment

No, not as submitted. A macOS keep-awake helper may be reasonable, but the default power policy needs maintainer approval and the diff needs spawn hardening, tests, changelog coverage, and rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-40`
  - body: When `caffeinate` cannot be executed, `spawn()` emits a `ChildProcess` `error` event rather than throwing from the surrounding `try/catch`. With no listener, gateway startup can crash instead of degrading cleanly, so attach an `error` listener before `unref()` and keep cleanup tolerant of a failed launch.
  - confidence: 0.93
- **[P3] Add the required changelog entry:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: This changes visible macOS gateway behavior by preventing idle sleep while the gateway runs. OpenClaw policy requires user-facing feature/fix/perf changes to update `CHANGELOG.md`, but this PR only changes the run loop.
  - confidence: 0.86
