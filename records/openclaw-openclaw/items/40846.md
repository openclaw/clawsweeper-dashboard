## Summary

Keep open for maintainer review. Current main does not implement automatic macOS idle-sleep prevention, and this PR is useful source work, but the proposed patch is not mergeable as-is because it starts the helper before Gateway lock ownership and lacks async child-process error handling.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <process.pid>` and kills it when `runGatewayLoop` exits.

## Best Possible Solution

Revise before landing: create the macOS sleep-prevention helper only after Gateway lock ownership is established, use `/usr/bin/caffeinate` or a validated absolute resolver, attach a child `error` listener before `unref()`, keep cleanup tied to the run-loop lifecycle, add focused tests for spawn failure and lock-conflict cleanup, and document the default macOS power behavior if maintainers approve it.

## Reproduction Assessment

Yes for the patch-level defects: the PR diff shows the helper starts before lock acquisition and lacks a child `error` listener, current main retries supervised lock conflicts in the same parent process, and a Node probe confirms missing-executable spawn failures are asynchronous. The actual overnight idle-sleep improvement was not reproduced here.

## Solution Assessment

No. The direction may be useful, but this is not the best mergeable solution until command resolution, async spawn-error handling, cleanup scoping, tests, docs, and maintainer approval of the default macOS power behavior are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate after Gateway lock ownership:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but supervised lock recovery can catch lock conflicts, sleep, and retry inside the same parent process. Start the helper only after this process owns the Gateway lock so it cannot keep the Mac awake while yielding to an existing Gateway.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff introduces a new Gateway startup command-execution path using PATH-based executable resolution and incomplete child-process error handling.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because dev shells, launchd wrappers, or service environments can alter PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.82
