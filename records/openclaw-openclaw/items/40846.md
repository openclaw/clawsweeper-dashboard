## Summary

Keep open. Current main still does not provide automatic macOS idle-sleep prevention for the Gateway, so the contribution is not obsolete, but the PR is not mergeable as-is because it adds an unhandled child-process error path, starts the helper before Gateway lock ownership, and introduces a PATH-resolved startup command that needs security/product review.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <process.pid>` and calls its cleanup from `runGatewayLoop`'s `finally` block.

## Best Possible Solution

Revise before landing: start the macOS sleep-prevention helper only after Gateway lock ownership is established, resolve `/usr/bin/caffeinate` or use the trusted system-bin resolver, attach a child `error` listener before `unref()`, keep cleanup tied to run-loop shutdown, add focused tests, and update changelog/docs if maintainers approve making this default behavior.

## Reproduction Assessment

Yes for the patch-level defects: the PR diff shows the missing child `error` listener and pre-lock startup, current main shows supervised retries in the same parent process, and a Node probe confirms missing executable failures are asynchronous. The actual overnight macOS idle-sleep improvement was not reproduced here.

## Solution Assessment

No. The direction may be useful, but this is not the best mergeable solution until command resolution, async spawn-error handling, lock-scoped lifecycle, tests, docs/changelog, and maintainer approval of the default power behavior are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so this `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate after Gateway lock ownership:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but supervised lock recovery can catch lock conflicts, sleep, and retry inside the same parent process. Start the helper only after this process owns the Gateway lock so it cannot keep the Mac awake while yielding to an existing Gateway.
  - confidence: 0.9
- **[P3] Document the user-visible macOS power change:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: This changes default Gateway power behavior on macOS, but the PR only edits code and current docs still frame `caffeinate` as an operator choice. Add the required changelog entry and update Gateway/macOS docs if maintainers approve this default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Concerns:
