## Summary

Keep open: current main only partially overlaps through LaunchAgent/App Nap and Telegram timeout fixes; it still has no merged gateway-run-loop idle-sleep assertion, and this PR needs product approval, real macOS proof, and a spawn-error fix before merge.

## What This Changes

The PR adds a macOS-only gateway run-loop helper that starts `caffeinate -i -w <gateway pid>` and kills it when the gateway exits.

## Best Possible Solution

If maintainers want default keep-awake behavior, land a rebased and hardened macOS implementation with explicit `caffeinate` resolution policy, async error handling, focused run-loop coverage, and real macOS proof; otherwise keep this as an operator helper or documented workaround.

## Reproduction Assessment

Yes for the blocking PR defect: the diff plus a local Node probe show failed spawns emit an async `error` event outside the PR's `try/catch`. The long-duration macOS idle-sleep benefit itself is not proven by inspectable PR evidence.

## Solution Assessment

No, not as submitted. A keep-awake helper may be viable, but this version needs maintainer approval for the default power policy, child-process hardening, focused tests, and real macOS proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-40`
  - body: When `caffeinate` cannot be executed, `child_process.spawn()` emits an asynchronous `error` event instead of throwing, so this `try/catch` will not handle the failure. Attach an `error` listener before `unref()` or otherwise handle ENOENT/EACCES so gateway startup cannot crash on an optional helper failure.
  - confidence: 0.93
