## Summary

Keep open. Current main does not implement this behavior, and the PR is not merge-ready because the helper can crash the gateway on async spawn errors, lacks the required changelog entry, and changes default macOS power behavior that should get maintainer approval.

## What This Changes

This PR adds a macOS-only `caffeinate -i -w <pid>` sidecar to `runGatewayLoop` and cleans it up when the gateway exits.

## Best Possible Solution

Land a hardened macOS gateway sleep-prevention helper, if approved, with Darwin/no-op/error-path coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes for the blocking review finding: Node emits missing-executable failures on the child `error` event, and the PR patch has no listener. The underlying macOS sleep-prevention benefit was not live-verified here because the review environment is Linux.

## Solution Assessment

No. `caffeinate -i -w` is a plausible narrow macOS mechanism, but this patch needs async error handling, tests, a changelog entry, and maintainer sign-off on default idle-sleep prevention.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:34-40`
  - body: `child_process.spawn()` reports ENOENT and permission failures through an asynchronous child `error` event, not this synchronous `try/catch`. Without an `error` listener, a missing or inaccessible `caffeinate` path can terminate the gateway during startup instead of degrading with a warning.
  - confidence: 0.95
- **[P3] Add the required changelog entry:** `src/cli/gateway-cli/run-loop.ts:61`
  - body: This changes user-visible macOS Gateway power behavior, but the PR only modifies the run loop. OpenClaw policy requires user-facing feature/fix/perf changes to update `CHANGELOG.md`, so release notes would miss this behavior change.
  - confidence: 0.89
