## Summary

Keep open. Current main does not add the proposed run-loop keep-awake helper, the related launchd fix only partially overlaps the macOS idle/App Nap problem, and this PR still has a blocking child-process error-handling defect plus missing real behavior proof.

## What This Changes

The PR adds a macOS-only `caffeinate -i -w <pid>` helper to the gateway run loop and stops it when the loop exits.

## Best Possible Solution

If maintainers want this default behavior, land a hardened macOS keep-awake implementation with trusted `caffeinate` resolution, async spawn-error handling, focused run-loop coverage, and real macOS proof; otherwise direct users to explicit operator keep-awake helpers.

## Reproduction Assessment

Yes for the blocking PR defect: the diff plus a Node probe show failed spawns emit an async `error` event outside the PR's `try/catch`. The long-duration macOS idle-sleep benefit is not proven by the PR evidence.

## Solution Assessment

No, not as submitted. A keep-awake helper may be viable, but this implementation needs maintainer approval for the default power policy, child-process hardening, focused tests, and real macOS proof before merge.

## Review Findings
