## Summary

Keep open: current main has the launchd `ProcessType=Interactive` App Nap fix, but not this run-loop `caffeinate` behavior, and the PR still needs spawn-error hardening, real behavior proof, and maintainer approval for default macOS power behavior.

## What This Changes

The PR adds a macOS-only `caffeinate -i -w <gateway-pid>` helper to the gateway run loop and calls its cleanup from the loop finalizer.

## Best Possible Solution

If maintainers want default keep-awake behavior, land a hardened macOS implementation with trusted `caffeinate` resolution, async spawn-error handling, focused run-loop coverage, and real macOS proof; otherwise keep this as an explicit operator helper or documented workaround.

## Reproduction Assessment

Yes for the blocking PR defect: the diff plus a local Node probe show failed spawns emit an async `error` event outside the PR's try/catch. The long-duration macOS idle-sleep benefit itself is not proven by inspectable PR evidence.

## Solution Assessment

No, not as submitted. A keep-awake helper may be viable, but this version needs maintainer approval for the default power policy, child-process hardening, focused tests, and real macOS proof before merge.
