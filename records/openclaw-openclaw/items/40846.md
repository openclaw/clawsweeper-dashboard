## Summary

Keep this PR open: current main still does not have a gateway run-loop keep-awake helper, but the submitted implementation has a blocking async spawn-error defect, lacks contributor-supplied real macOS proof, and needs maintainer approval for default macOS power behavior.

## What This Changes

The PR adds a macOS-only `caffeinate -i -w <gateway pid>` child process to the gateway run loop and cleans it up when the loop exits.

## Best Possible Solution

If maintainers want default keep-awake behavior, land a hardened macOS implementation with trusted `caffeinate` resolution, async spawn-error handling, focused run-loop coverage, and real macOS proof; otherwise keep this as an explicit operator helper or documented workaround.

## Reproduction Assessment

Yes for the blocking PR defect: the diff plus a local Node probe show failed spawns emit an async `error` event outside the PR's `try/catch`. The long-duration macOS idle-sleep benefit itself is not proven by inspectable PR evidence.

## Solution Assessment

No, not as submitted. A keep-awake helper may be viable, but this version needs maintainer approval for the default power policy, child-process hardening, focused tests, and real macOS proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-40`
  - body: When `caffeinate` cannot be executed, `child_process.spawn()` emits an asynchronous `error` event instead of throwing, so the current `try/catch` does not handle the failure. Attach an `error` listener before `unref()` so the optional helper cannot crash gateway startup.
