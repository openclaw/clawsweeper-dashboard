## Summary

Keep open. Current main does not implement this gateway keep-awake behavior, and the PR is still potentially useful, but it should not merge as-is because the new child-process path can crash on spawn failure, the user-facing behavior lacks the required changelog entry, and the branch is stale against the current gateway loop.

## What This Changes

This PR adds a macOS-only `caffeinate -i -w <pid>` child process around `runGatewayLoop` and kills it from the loop cleanup path.

## Best Possible Solution

Rebase the idea onto the current gateway loop, add resilient child-process handling and focused tests, record the user-visible behavior in the changelog, and have a maintainer confirm the default macOS keep-awake policy before merge.

## Reproduction Assessment

Yes for the blocking PR defect: Node documents an async ChildProcess `error` event on spawn failure, and the PR's `spawnCaffeinate` has no `child.on("error")` listener. The macOS idle-sleep improvement itself was not live-verified in this Linux review environment.

## Solution Assessment

No, not as submitted. A macOS-only helper may be a reasonable implementation direction, but this patch needs spawn-error handling, changelog coverage, tests, a rebase, and maintainer confirmation of the default power behavior.

## Review Findings

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:33-37`
  - body: Node emits a ChildProcess `error` event when the executable cannot be spawned, and an EventEmitter `error` without a listener can crash the process. This code only wraps `spawn()` in `try/catch`, so a missing or restricted `caffeinate` can take down gateway startup instead of degrading gracefully; attach an `error` listener before `unref()` and cleanup return.
  - confidence: 0.91
