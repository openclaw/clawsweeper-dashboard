## Summary

Keep open: current main still lacks this gateway runtime keep-awake behavior, and the submitted PR is not merge-ready because it adds an unguarded macOS child-process launch, lacks the required changelog entry, and is stale against active gateway lifecycle work.

## What This Changes

The PR adds a macOS-only `caffeinate -i -w <pid>` helper to the gateway run loop and stops it when the gateway loop exits.

## Best Possible Solution

Land a maintainer-approved macOS keep-awake implementation with explicit lifecycle ownership, resilient child-process error handling, constrained executable resolution, focused tests, and changelog coverage.

## Reproduction Assessment

Yes for the blocking PR defect: the diff and Node child_process contract show failed spawns emit an `error` event that this code does not handle. The long-duration macOS idle-sleep benefit was not live-verified in this read-only review.

## Solution Assessment

No, not as submitted. A macOS keep-awake helper may be a reasonable direction, but the current diff needs child-process hardening, executable-resolution policy, tests, changelog coverage, and a rebase before maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-40`
  - body: When `caffeinate` cannot be executed, Node emits a `ChildProcess` `error` event rather than throwing from `spawn()`. With no listener, gateway startup can crash instead of degrading gracefully, so attach an `error` listener before `unref()` and keep cleanup tolerant of failed launch.
  - confidence: 0.91
- **[P3] Add the required changelog entry:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: This changes visible macOS gateway behavior by preventing idle sleep while the gateway runs. OpenClaw policy requires user-facing feature/fix/perf changes to update `CHANGELOG.md`, but this PR only changes the run loop.
  - confidence: 0.84

## Security Review
