## Summary

Keep open. Current main does not implement the macOS `caffeinate` helper, and the PR remains useful, but the patch needs rework before merge because it starts a new power-management helper before lock ownership, lacks async spawn-error handling, and introduces a PATH-resolved startup command.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <process.pid>` and cleans it up when the Gateway loop exits.

## Best Possible Solution

After maintainer approval, implement the helper as a lock-scoped macOS lifecycle step using a trusted `caffeinate` path, attach child `error`/exit handling before `unref()`, preserve shutdown cleanup, and add focused tests plus docs and changelog coverage.

## Reproduction Assessment

Yes for the patch-level defects: the diff shows the pre-lock helper startup and missing child `error` listener, current main shows supervised lock recovery, and a Node probe confirms missing executable errors are asynchronous. The requested overnight idle-sleep behavior itself was not live-reproduced here.

## Solution Assessment

No. The direction may be useful, but this PR is not the narrowest mergeable solution until command resolution, async spawn-error handling, lock-scoped lifecycle, tests, documentation, changelog, and maintainer approval are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child `error` event, so this `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate only after lock ownership:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper starts before `acquireGatewayLock()`, but supervised lock recovery can catch a lock conflict, find the existing Gateway healthy, and return. Start the helper only after this process owns the Gateway lock so a duplicate launch cannot keep the Mac awake while yielding to another Gateway.
  - confidence: 0.9
- **[P3] Document the macOS power change:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: This changes default Gateway power behavior on macOS, but the PR only edits code and current docs still present `caffeinate` as an operator choice. Add the required changelog entry and update Gateway/macOS docs if maintainers approve this default.
  - confidence: 0.82

## Security Review

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
