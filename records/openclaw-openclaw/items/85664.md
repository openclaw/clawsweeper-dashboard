## Summary

Keep open: the branch is a useful narrow HTTP read feature and the prior public hook-bypass concern appears addressed, but it still changes the default HTTP data-access boundary because `read` is appended for every HTTP tool-resolution path while it is not default-denied.

## What This Changes

The PR adds an internal raw coding-tools factory, wires the `read` coding tool into HTTP `/tools/invoke`, adds focused regression tests, and records the change in the changelog.

## Best Possible Solution

Expose `read` only through an explicit opt-in/default-deny path, keep the hook bypass internal, keep mutating coding tools absent, and add focused tests/docs for default 404 plus opt-in 200.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR appends `read` for every HTTP resolution, `read` is not in the HTTP default deny list, and absent tool policies are skipped. I did not run tests because this review is read-only.

## Solution Assessment

No as currently shaped: the narrow `read` wiring and internal hook bypass are maintainable, but the HTTP file-read surface should fail closed by default or require explicit opt-in before merge.
