## Summary

The dual-key opt-in is a reasonable security direction and the supplied live proof covers the intended built-in behavior, but the patch has two blocking shared-name defects: a plugin named `read` can shadow the opted-in built-in reader, and the audit suppresses warnings for every `read` name even when only a plugin implementation is reachable.

## What This Changes

The PR adds dual-key opt-in exposure of the built-in `read` coding tool to HTTP and SDK direct-invoke gateway surfaces, with config schema/help, security auditing, tests, and docs.

## Best Possible Solution

Keep the dual-key design, but make audit suppression conditional on the active built-in host-read opt-in and make the explicitly opted-in built-in `read` implementation win same-name collisions, with tests for both opt-in states.

## Reproduction Assessment

Yes at source level: registering a plugin tool named `read` exercises the existing shared-name selection and audit paths, reproducing both defects without an LLM or external service.

## Solution Assessment

No. Dual-key gating is a sound basis, but static name suppression and plugin-first collision handling do not safely implement the intended built-in host-read contract.
