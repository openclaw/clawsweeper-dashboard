## Summary

Keep open: the narrow runtime wiring is plausible and the supplied live proof is strong, but the operator-facing security wording still understates the file scope because `read` is not workspace-only unless `tools.fs.workspaceOnly` is enabled.

## What This Changes

The PR wires the coding `read` tool into HTTP `/tools/invoke` behind explicit `gateway.tools.allow`, adds an internal raw coding-tool factory, and updates focused gateway/security docs and tests.

## Best Possible Solution

Keep the narrow default-denied `read` implementation, but make every operator-facing warning describe the actual host file-read boundary and point to `tools.fs.workspaceOnly=true` when a deployment wants workspace-only reads.
