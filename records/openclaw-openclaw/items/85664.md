## Summary

Keep open for maintainer review: this is focused, unique feature work with strong supplied proof, but it expands the authenticated direct-invoke boundary to host file reads and needs explicit gateway/security owner acceptance before merge.

## What This Changes

This PR wires the raw coding `read` tool into the shared gateway direct-invoke resolver for HTTP `/tools/invoke` and SDK RPC `tools.invoke`, with default-deny, docs, audit, and regression-test updates.

## Best Possible Solution

Land only after gateway/security owners explicitly accept the host-file-read direct-invoke contract for both HTTP and SDK RPC, or revise the patch to require a distinct new opt-in or upgrade warning that cannot silently activate old inert `read` allow entries.

## Reproduction Assessment

Yes, source-level: current main resolves direct invoke through gateway tools only, while the PR appends raw coding `read` and supplies Docker live output for HTTP and RPC behavior. I did not run tests or the gateway because this review is read-only.

## Solution Assessment

Unclear until owner decision: the implementation is narrow and default-denied, but using the existing `gateway.tools.allow` entry can silently activate old inert configs and applies to SDK RPC too. The safer alternative is explicit owner approval or a distinct opt-in/upgrade warning.
