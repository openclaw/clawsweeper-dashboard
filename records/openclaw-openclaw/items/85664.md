## Summary

This PR remains necessary because current main does not expose the built-in `read` tool through Gateway direct invocation, and the broader write/edit PR depends on rather than supersedes it. The implementation is coherent and well-proven, but merging intentionally expands `operator.write` into configured host-filesystem read access, so it needs explicit Gateway/security owner approval.

Likely related people: `@ai-hpc` introduced the shared `tools.invoke` path (high confidence), and `@vincentkoc` recently maintained its hook/context handling (medium confidence).

## What This Changes

Adds a dual-opt-in path that exposes the built-in `read` coding tool through Gateway HTTP and SDK direct-invoke surfaces, with config, audit, documentation, and regression coverage.

## Best Possible Solution

Keep the shared resolver and canonical guarded reader with dual opt-in, then land only after owners explicitly approve `operator.write` as the authorization boundary and accept the documented outside-workspace behavior.

## Solution Assessment

Yes as an implementation shape: it reuses the shared resolver, canonical read factory, policy pipeline, and existing hook path rather than creating competing behavior. The permanent `operator.write` authorization boundary still requires maintainer product/security judgment.
