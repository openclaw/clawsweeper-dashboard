## Summary

Keep this PR open for a focused repair: the built-in host-read opt-in is carefully designed and strongly proven, but its name-wide deny implementation introduces a blocking upgrade regression for existing plugin tools named `read`.

## What This Changes

The PR adds dual-key opt-in exposure of the built-in `read` tool through Gateway HTTP and SDK direct-invoke surfaces, with configuration, auditing, tests, and documentation.

## Best Possible Solution

Keep the built-in reader default-off and require both `directInvoke.hostFsRead` and `gateway.tools.allow` when materializing that built-in tool, while leaving the generic default deny list unchanged so existing plugin tools retain their current policy behavior.

## Reproduction Assessment

Yes for the patch defect: comparing current main with the exact merge tree shows that the new name-wide deny removes every candidate named `read`, including existing plugin tools, unless operators add `gateway.tools.allow`.

## Solution Assessment

No as written: the dual-key product shape is conservative, but applying its second gate through a source-agnostic name deny is broader than necessary; source-specific built-in materialization achieves the security goal without changing plugin behavior.
