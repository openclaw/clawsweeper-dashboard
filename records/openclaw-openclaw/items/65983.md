## Label Justifications

- `P1`: The report describes untracked long-lived exec/Codex/MCP process trees that can keep consuming memory while OpenClaw reports no active sessions.
- `impact:session-state`: The affected behavior is ownership drift between the process supervisor, bash process registry, and user-visible session state.
- `impact:crash-loop`: The observed orphan process accumulation can drive memory growth and gateway/service availability pressure.

## Summary

Keep this open: current main still has in-memory-only process supervision and no real orphan reconciliation, and the formal closing PR is open, conflicting, and only covers the in-process/session-loss half of the report.

## What This Changes

Ensure background PTY exec process trees remain owned across gateway restart or session-state loss, or are reconciled and terminated instead of disappearing from `process list`.

## Best Possible Solution

Review and either land an updated successor to https://github.com/openclaw/openclaw/pull/66001 or replace it with a maintainer-approved lifecycle design that covers both in-process ownership drift and restart-era PTY orphan handling.

## Reproduction Assessment

Yes at the source level: current main loses supervisor and bash-process ownership on restart because both are in-memory and `reconcileOrphans()` is a no-op. I did not run a live PTY restart proof in this read-only review.

## Solution Assessment

Unclear as a final product path: the issue is valid, but maintainers still need to choose kill, reattach, or a narrower in-process reconciliation contract before a fix can be called best.
