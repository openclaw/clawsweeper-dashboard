## Label Justifications

- `P1`: Untracked long-lived PTY worker trees can keep consuming memory after OpenClaw loses ownership, affecting real agent/process workflows.
- `impact:session-state`: The core failure is drift between live process state and OpenClaw's in-memory session/run ownership.
- `impact:crash-loop`: The report describes orphan helper trees accumulating memory until process-level availability is at risk.

## Summary

Keep open: current main and the latest release still use in-memory-only supervisor/process tracking with a no-op orphan reconciliation hook, and the linked closing PR remains open and conflicting rather than merged.

## What This Changes

Report asks OpenClaw to prevent PTY-backed background exec runs from surviving gateway restart or lost session ownership as untracked process trees.

## Best Possible Solution

Refresh or replace the linked repair PR with ownership-aware reconciliation, an explicit cold-start stale PTY kill-or-reattach policy, and focused regression coverage for both preserved owned runs and orphan cleanup.

## Reproduction Assessment

Partially: source inspection gives high confidence that OpenClaw can lose exec/process ownership and report no sessions because tracking is memory-only and `reconcileOrphans()` is a no-op. I did not establish a live macOS restart reproduction proving the PTY tree survives independently.

## Solution Assessment

Unclear as a complete fix: the linked PR's in-process reconciliation is a reasonable smaller step, but the full issue needs an explicit maintainer decision on cold-start stale PTY runs and durable ownership metadata.
