## Summary

Keep this open: current main and the latest release still have the in-memory-only supervisor/session ownership gap, and the linked closing PR is open, conflicting, and explicitly only covers the in-process half of the requested restart/session-loss cleanup.

## What This Changes

The issue asks OpenClaw to keep background PTY exec process trees owned across gateway restart or session-state loss so they cannot survive as untracked orphan processes.

## Best Possible Solution

Land one lifecycle-owned process-supervisor fix, or an explicit split with a canonical follow-up, that handles lost in-process ownership and gateway-restart PTY orphans with focused tests and real PTY restart proof.

## Reproduction Assessment

Yes, at source level: current main uses in-memory Maps for background exec session ownership, process list reads those Maps, and reconcileOrphans is still a no-op. I did not run a live macOS or Linux PTY restart orphan scenario in this read-only review.

## Solution Assessment

Unclear: reconciling supervisor ownership is the right problem boundary, but the open PR is an in-process partial and maintainers still need to choose the durable restart policy. The safest path is to update or replace that PR rather than queue a second repair lane.
