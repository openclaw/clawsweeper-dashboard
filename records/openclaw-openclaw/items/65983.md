- `impact:session-state`: The central failure is drift between OpenClaw's tracked exec/session state and live OS process ownership.

## Summary

Keep open: current main still has memory-only background exec/session ownership, the latest release shipped the same restart-loss behavior, and https://github.com/openclaw/openclaw/pull/66001 is open, conflicting, and explicitly only a partial in-process mitigation.

## What This Changes

The issue asks OpenClaw to prevent background PTY exec process trees from surviving gateway restart or session-state loss as untracked orphan processes.

## Best Possible Solution

Define and land one core-owned supervisor lifecycle path that records active PTY background ownership durably, reconciles startup/session loss, and explicitly chooses kill versus reattach semantics with focused tests.

## Reproduction Assessment

Yes, at source level: current main keeps supervisor and process-session state in memory, and the docs explicitly say sessions are lost on restart. I did not run a live restart/orphan-process reproduction in this read-only review.

## Solution Assessment

Unclear as a final fix. Durable ownership and startup reconciliation belong in core, but maintainers still need to choose the kill-versus-reattach policy; the linked PR is only a partial in-process mitigation.
