## Label Justifications

- `P1`: Untracked long-lived process trees can accumulate helpers and degrade real gateway or agent availability.
- `impact:session-state`: The central failure is drift between OpenClaw's tracked exec/session state and live OS process ownership.
- `impact:crash-loop`: The reported orphan trees can drive memory growth and process-level availability pressure.

## Summary

Keep open: current main still uses process-local background exec and supervisor state, latest release provenance shows the same restart-loss model, and the linked fix PR remains open/conflicting while explicitly covering only the in-process half of the problem.

## What This Changes

Background PTY exec ownership should survive or reconcile gateway restart/session-state loss so stale OS process trees are killed or re-registered instead of becoming untracked.

## Best Possible Solution

Define one core-owned supervisor lifecycle path that records active PTY background ownership durably and reconciles startup/session loss after maintainers choose kill versus reattach semantics, then land https://github.com/openclaw/openclaw/pull/66001 or a replacement that matches that policy.

## Reproduction Assessment

Yes, at source level: current main keeps supervisor and process-session state in memory, and the docs explicitly say sessions are lost on restart. I did not run a live restart/orphan-process reproduction in this read-only review.

## Solution Assessment

Unclear as a final fix. Durable ownership and startup reconciliation belong in core, but maintainers still need to choose the kill-versus-reattach policy; the linked PR is only a partial in-process mitigation.
