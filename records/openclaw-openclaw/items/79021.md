## Summary

Keep open: current main still lacks a `refresh_token_reused` recovery path that clears or demotes the matching `lastGood` OAuth profile before fallback selection, and the only direct fix PR found was closed without merging.

## What This Changes

The issue asks OpenClaw to recover from terminal `openai-codex` OAuth refresh failures by clearing stale `lastGood` state and selecting another healthy OAuth profile when available.

## Best Possible Solution

Add a narrow auth-profile recovery path that treats terminal OAuth refresh failures as stale-profile signals, clears only the matching `lastGood` entry in the persisted owner and relevant runtime snapshot, then lets existing profile rotation choose another eligible OAuth profile before surfacing a re-auth hint.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection shows the reported path is still plausible because `refresh_token_reused` is wrapped without clearing `lastGood`, while the fallback helper can still prefer that stale profile pointer.

## Solution Assessment

Yes. A narrow runtime auth-profile fix is the best direction; it should reuse the existing owner-store, runtime snapshot, and profile-rotation contracts rather than changing doctor route rewrite behavior or adding a new setting.
