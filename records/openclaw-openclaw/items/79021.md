## Summary

Keep open: current main still has a source-level path where terminal OpenAI Codex OAuth refresh failures are wrapped without clearing the failed profile from lastGood, and the newer store-merge cleanup only covers invalidated child-agent overrides, not the reported same-store multi-profile stale lastGood case.

## What This Changes

The issue asks OpenClaw to recover from refresh_token_reused by demoting a stale openai-codex lastGood profile and selecting another usable same-provider OAuth profile.

## Best Possible Solution

Add a narrow auth-profile runtime cleanup that clears only the failed matching lastGood pointer on terminal openai-codex OAuth refresh failure, then relies on existing ordering/cooldown logic to try another eligible profile.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection gives a medium-confidence reproduction path: a stale selected OAuth profile can hit refresh_token_reused and flow through the wrapped failure path without clearing matching lastGood state.

## Solution Assessment

Yes, the best fix is a narrow runtime auth-profile cleanup using existing store locks and runtime snapshot contracts. It should not broaden into doctor route migration or the separate single-profile forced-refresh behavior tracked at https://github.com/openclaw/openclaw/issues/81941.
