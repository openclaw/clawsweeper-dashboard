## Summary

Keep open: current main still has a source-level path where an `openai-codex` OAuth `refresh_token_reused` failure is wrapped without clearing or demoting the failed profile from matching `lastGood` state, and the direct fix PR for that behavior was closed unmerged. A newer merged PR addressed a sibling forced-refresh stale-write path, but it does not implement the multi-profile stale-`lastGood` recovery requested here.

## What This Changes

The issue asks OpenClaw to recover from terminal `openai-codex` OAuth refresh failures by clearing stale `lastGood` state and selecting another valid same-provider OAuth profile when available.

## Best Possible Solution

Add a focused runtime recovery that clears or demotes only the failed profile's matching `lastGood` state on terminal `openai-codex` OAuth refresh failure, then lets existing profile ordering choose another eligible profile; keep broader canonical-owner and single-profile `oauthRef` questions separate.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The source-level path is clear from `resolveApiKeyForProfile`, `oauth-manager`, and `repair.ts`, and the issue includes concrete logs plus a manual recovery path.

## Solution Assessment

Yes, the narrow stale-`lastGood` cleanup is the best current fix direction for the original report. It should reuse the existing store lock/runtime snapshot contracts and avoid broad canonical-owner or doctor route-rewrite changes.
