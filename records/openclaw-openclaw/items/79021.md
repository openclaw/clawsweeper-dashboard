## Summary

Keep open: the report is concrete, current source only covers same-profile `refresh_token_reused` recovery and some main-store drift cases, and I did not find current-main coverage that demotes a terminally stale `lastGood` profile while selecting another fresh `openai-codex` OAuth profile from the same agent store.

## What This Changes

The issue asks OpenClaw to recover from stale OpenAI Codex OAuth profile selection when terminal refresh failure leaves `auth-state.lastGood.openai-codex` pointing at an expired profile.

## Best Possible Solution

Add a narrow auth-profile recovery path that treats terminal OAuth refresh failures as stale-profile signals, demotes or clears matching `lastGood`, and lets existing rotation choose another eligible `openai-codex` OAuth profile before surfacing the re-auth hint.

## Reproduction Assessment

No high-confidence live reproduction was run against current `main`. Source inspection shows the selected-profile refresh path can still throw without clearing `lastGood`, while failure marking records cooldown state but does not demote the stale profile pointer.

## Solution Assessment

Yes, a narrow runtime auth-profile fix is the best direction. It should reuse existing profile ordering and `advanceAuthProfile` rotation, not change the doctor route-rewrite behavior or add a new user-facing setting.
