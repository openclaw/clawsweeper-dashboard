## Summary

Keep open: current `main` still has a source-level path where terminal `openai-codex` OAuth refresh failure is wrapped without clearing a matching stale `lastGood` pointer. The later merged work at https://github.com/openclaw/openclaw/pull/80738 fixed forced-refresh stale-token reuse, but it did not add the multi-profile stale-`lastGood` recovery requested here.

## What This Changes

The issue requests recovery from stale OpenAI Codex OAuth `lastGood` profile selection after terminal refresh failures such as `refresh_token_reused`.

## Best Possible Solution

Add a narrow auth-profile recovery path that clears or demotes only the terminally failed matching `lastGood` entry, updates any relevant runtime snapshot, and then lets existing profile ordering choose another eligible same-provider OAuth profile.

## Reproduction Assessment

No live current-main gateway reproduction was run. Source inspection plus the reporter's logs give a high-confidence source path: terminal OAuth refresh failure can leave `lastGood` unchanged while the fallback helper can still consult that stale pointer.

## Solution Assessment

Yes, a narrow runtime auth-profile cleanup is the maintainable direction. It should reuse existing store locking, owner-store resolution, and runtime snapshot contracts rather than adding config, changing doctor route migration, or reworking provider routing.
