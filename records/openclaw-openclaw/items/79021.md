## Summary

Keep open: current `main` still has a source-level path where a terminal `openai-codex` OAuth refresh failure is wrapped without clearing a matching stale `lastGood` pointer, and the existing tests cover same-profile credential reload/retry rather than selecting another fresh same-provider profile from the same agent store. The direct fix attempt at https://github.com/openclaw/openclaw/pull/79029 was closed unmerged, so the central behavior remains actionable.

## What This Changes

The issue requests OAuth recovery that demotes a stale `openai-codex` `lastGood` profile after `refresh_token_reused` so another valid same-provider profile can be selected.

## Best Possible Solution

Add a small runtime auth-profile fix that clears only the failed profile's matching `lastGood` state on terminal OAuth refresh failure, then lets the existing profile ordering and rotation choose another eligible profile.

## Reproduction Assessment

No high-confidence live reproduction was run against current `main`. The source-level path is clear: terminal refresh failure can reach the wrapped error path while `lastGood` remains unchanged and fallback helpers can still consult that stale pointer.

## Solution Assessment

Yes, a narrow runtime auth-profile cleanup is the right direction. The safer fix is to reuse existing owner-store locking, runtime snapshot, and rotation contracts rather than adding a config option or changing doctor route-rewrite behavior.
