## Summary

Keep open: current main and the latest release still send opt-in Telegram rich text through `sendRichMessage`, and the direct content-aware fix candidate had positive Mantis proof but was closed unmerged by its author.

## What This Changes

Issue reports that Telegram rich-message delivery disables selected-text Quote/Quote & Reply while plain Telegram delivery preserves it.

## Best Possible Solution

Keep this issue open and have a maintainer either adopt/reopen the proven content-aware routing fix or explicitly document that `richMessages: true` trades selected-text Quote for native rich blocks.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 still send opt-in rich text through `sendRichMessage`, and Mantis proof for the closed candidate shows baseline main without selected-text Quote & Reply. I did not run a fresh live Telegram repro in this read-only issue review.

## Solution Assessment

Unclear as a product decision: content-aware fallback has positive Telegram proof and avoids a new config option, but it changes what `richMessages: true` means for ordinary text. Maintainers should choose whether to adopt that tradeoff or document rich mode as quote-incompatible.
