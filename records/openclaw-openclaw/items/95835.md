## Summary

Keep open: current main and v2026.6.10 still route `richMessages: true` Telegram text through `sendRichMessage`, while the proof-positive candidate for selected-text Quote was closed unmerged and no open PR now cleanly owns this issue.

## What This Changes

The issue requests that Telegram rich-message delivery preserve selected-text Quote/Quote & Reply, or otherwise have an accepted rich-mode fallback that keeps precise reply workflows usable.

## Best Possible Solution

Revive or sponsor a content-aware Telegram delivery fix with maintainer-approved rich-mode semantics and live Telegram proof, or explicitly document that selected-text Quote is unsupported when `richMessages: true` uses `sendRichMessage`.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 route `richMessages: true` text through `sendRichMessage`, while the quote-compatible path uses `sendMessage`. Reporter A/B evidence and the closed candidate PR's Mantis baseline/candidate proof further support the behavior.

## Solution Assessment

Unclear as a landed fix: content-aware fallback is the best proven direction so far, but it intentionally changes rich-mode delivery semantics and should be maintainer-approved before a replacement PR is queued or merged.
