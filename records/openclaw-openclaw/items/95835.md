## Label Justifications

- `impact:message-loss`: The affected rich-message shape suppresses selected quote context that users rely on for precise follow-up replies.

## Summary

Keep open: the quote-compatible default/plain Telegram path is implemented, but current main still sends opt-in `richMessages: true` durable text, bot replies, edits, and previews through Telegram rich-message APIs. The linked content-aware fallback PR had positive Mantis proof but was closed unmerged, so there is no current-main or shipped fix for the rich-enabled selected-text Quote behavior.

## What This Changes

The issue asks OpenClaw to preserve selected-text Quote/Quote & Reply for Telegram messages sent with `channels.telegram.richMessages = true`, or define a supported quote-compatible behavior for that rich-message mode.

## Best Possible Solution

Keep this issue open until maintainers choose and land a Telegram-plugin fix that preserves selected-text Quote for ordinary rich-enabled text, while documenting or testing the remaining behavior for true rich blocks that Telegram clients cannot quote.

## Reproduction Assessment

Yes at source level, with strong reporter A/B evidence, but not from a fresh live Telegram run in this review. Current main still routes `richMessages: true` text through `sendRichMessage`, while the plain/default path uses `sendMessage`.

## Solution Assessment

Unclear as a final implementation choice. Content-aware fallback is plausible and had Mantis proof in the closed PR, but maintainers still need to accept how `richMessages: true` should trade native rich rendering against quote-compatible delivery.
