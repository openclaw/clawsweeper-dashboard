- `impact:message-loss`: The affected behavior removes Telegram's selected-text quote/reply context for rich-delivered assistant messages.

## Summary

Keep open: current main and v2026.6.9 already provide the plain/default Telegram delivery fallback, but the rich-message opt-in path still sends `sendRichMessage` and the docs do not specifically warn that this can remove Telegram selected-text Quote/Quote & Reply.

## What This Changes

The issue asks OpenClaw to preserve Telegram selected-text Quote/Quote & Reply for rich Telegram messages, or clearly document the quote-compatible plain-delivery fallback.

## Best Possible Solution

Keep plain Telegram delivery as the supported quote-compatible path, document that `richMessages: true` can remove selected-text Quote/Quote & Reply, and require live Telegram proof before attempting a rich-message runtime workaround.

## Reproduction Assessment

Yes at source level, medium confidence: current main and v2026.6.9 route `richMessages: true` through `sendRichMessage` and false/unset through `sendMessage`. I did not independently live-reproduce the Telegram client selected-text menu behavior in this read-only review.

## Solution Assessment

Yes for the near-term fix: a narrow docs/config-help update is the safest maintainable solution because the quote-compatible fallback already exists. A code workaround should wait for live Telegram proof that rich messages can preserve selected-text Quote.
