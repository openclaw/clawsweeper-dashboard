## Summary

Keep open: current main and v2026.6.10 still send `richMessages: true` Telegram assistant text through `sendRichMessage`, while the quote-compatible path remains the standard `sendMessage` branch; the prior content-aware fix PR has proof but was closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve Telegram selected-text Quote/Quote & Reply for rich Telegram delivery, or use a quote-compatible fallback when rich delivery cannot preserve it.

## Best Possible Solution

Revive or replace the closed content-aware Telegram routing fix, with maintainer acceptance of the rich-mode tradeoff and live Telegram proof covering selected-text Quote plus true rich block rendering.

## Reproduction Assessment

Yes, at source level with strong reporter A/B evidence: current main and v2026.6.10 still send `richMessages: true` text through `sendRichMessage`, while the quote-compatible branch uses standard `sendMessage`. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

Unclear as a final product choice: content-aware fallback is the most concrete known repair and already had Telegram proof, but maintainers still need to accept how `richMessages: true` should trade rich rendering for quote-compatible delivery.
