## Summary

Keep open: current main still routes explicitly enabled Telegram rich text through `sendRichMessage`, the exact content-aware Quote fix PR is closed unmerged, and no merged replacement owns the remaining rich-enabled behavior.

## What This Changes

The issue asks OpenClaw to preserve Telegram selected-text Quote/Quote & Reply for `channels.telegram.richMessages = true`, or define a supported quote-compatible fallback for rich delivery.

## Best Possible Solution

Define the rich-mode contract: adopt a content-aware Telegram fallback for non-rich-block assistant text, keep true rich blocks on `sendRichMessage`, and document any remaining selected-quote limitation.

## Reproduction Assessment

Yes at source level: current main still sends rich-enabled text through `sendRichMessage`, while the quote-compatible path uses `sendMessage`, and related comments plus Mantis proof show the Telegram client behavior split. I did not run a fresh live Telegram repro in this read-only review.

## Solution Assessment

Unclear as an automatic fix: the content-aware fallback from the closed PR is the strongest candidate, but maintainers still need to accept the rich-mode presentation tradeoff or choose to document the limitation instead.
