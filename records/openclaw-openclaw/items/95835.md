Keep open: current main and v2026.6.9 already provide the quote-compatible default/off path, but this issue is about the remaining `richMessages: true` delivery path, and an open PR with closing syntax is already the active fix candidate.

## What This Changes

The issue reports that Telegram `richMessages: true` sends assistant text through rich-message delivery, which removes Telegram selected-text Quote/Quote & Reply compared with plain delivery.

## Best Possible Solution

Review and either land the linked content-aware Telegram delivery PR with real Telegram proof, or document that `richMessages: true` intentionally gives up selected-text Quote while keeping default/off delivery quote-compatible.

## Reproduction Assessment

Yes at source level, with strong reporter A/B evidence: current main sends `richMessages: true` text through `sendRichMessage`, while the quote-compatible path uses `sendMessage`. I did not run a live Telegram Desktop/mobile repro in this read-only review.
