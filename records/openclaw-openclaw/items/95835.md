## Label Justifications

- `P2`: This is a concrete Telegram workflow regression with a workaround and limited blast radius, not a crash, data loss, or core outage.
- `impact:message-loss`: The rich delivery path suppresses Telegram's selected quote reply affordance, which users rely on for precise follow-up messages.
- `impact:session-state`: Missing selected-text Quote prevents precise selected assistant text from becoming reliable follow-up context.

## Summary

Keep open: current `main` and latest release still route explicitly enabled Telegram rich text through `sendRichMessage`, and the only PR that targeted selected-text Quote fallback is closed unmerged.

## What This Changes

Fix Telegram `richMessages: true` delivery so selected-text Quote/Quote & Reply is preserved where possible, or settle a documented rich-mode fallback/tradeoff.

## Best Possible Solution

Land or sponsor an owner-approved Telegram delivery change that keeps rich-only blocks on rich APIs while sending quote-compatible plain/inline text through standard HTML, backed by redacted live Telegram proof; if maintainers reject that tradeoff, document that rich mode gives up selected-text Quote.

## Reproduction Assessment

Yes at source level: current `main` and `v2026.6.10` send explicitly enabled rich text through `sendRichMessage`, while default/off delivery uses standard `sendMessage`. I did not run a live Telegram client repro in this read-only review.

## Solution Assessment

No current solution is landed. The closed content-aware fallback PR was a plausible proved direction, but because it closed unmerged, maintainers need to choose whether to revive that tradeoff or define a different rich-mode Quote policy.
