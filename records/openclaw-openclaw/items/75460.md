## Summary

Close as ClawHub: Manifest is a useful optional LLM provider/router, but current main already has external provider-plugin surfaces for catalog metadata, auth choices, env-var discovery, install, and ClawHub/npm distribution. The PR also is not merge-ready as a bundled plugin because it creates a manifest-only extension directory without the required package/runtime entry files.

## What This Changes

This PR adds a new bundled `manifest` provider manifest with `MANIFEST_API_KEY` auth and a static `manifest/auto` catalog, plus Manifest provider documentation and provider-index entries.

## Best Possible Solution

Keep OpenClaw core unchanged and route Manifest to a separately maintained provider plugin published through ClawHub or npm, using the existing provider manifest/auth/catalog surfaces. Revisit core only if an actual missing plugin API seam is identified or maintainers explicitly decide to promote Manifest as an official bundled provider.

## Reproduction Assessment

Not applicable as a feature/provider addition. The verification path is source and diff inspection: current main has no Manifest provider, and the requested behavior maps to existing external provider-plugin APIs.

## Solution Assessment

No, merging this as another bundled provider is not the best path under current scope policy, and the branch is also missing bundled plugin package/runtime entry files. The narrower maintainable solution is an external provider plugin with ClawHub/npm distribution.
