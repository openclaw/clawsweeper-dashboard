## Summary

Keep open: the patch is aimed at the right bug, but it still misses the reported Slack result shape and broadens the middleware fail-closed boundary in ways that can preserve the wrong result after a post-processing failure.

## What This Changes

The PR adds a message-send-specific middleware fallback that returns the sanitized original result with a warning on tool-result middleware failure and adds two regression tests using top-level delivery identifiers.

## Best Possible Solution

Preserve a minimal safe success result only for proven `message.send` deliveries using the actual nested delivery result or receipt contract, while keeping generic fail-closed behavior for non-delivered, non-send, or unsafe results.

## Reproduction Assessment

Yes. A focused harness case with a Slack-shaped `details: { ok: true, result: { messageId, channelId, receipt } }` result followed by throwing or invalid middleware should reproduce the remaining failure; source inspection confirms the path and a reviewer reported reproducing it against the PR.

## Solution Assessment

No. The proposed direction is reasonable, but the implementation should build a minimal fallback from the actual delivery contract and keep fail-closed behavior for everything else.
