## What This Changes

The PR adds a `message.send` side-effect fallback in tool-result middleware and regression tests for thrown or invalid middleware after delivery.

## Best Possible Solution

Keep the side-effect-preserving fallback, but make it recognize the actual `message.send` receipt shapes and require a real delivered message identifier before returning success.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a real Slack/core `message.send` result stores the delivery receipt under `details.result`, then thrown or invalid middleware still falls through to the generic failure because the PR checks only top-level details.

## Solution Assessment

No. The direction is right, but the implementation must match the real nested result shape and require an actual delivery receipt; the current tests cover only a synthetic top-level shape.
