## Summary

Keep open. Current main still loses pending approval IDs on gateway restart and the user-facing `/approve` plus Telegram callback paths still surface or rethrow the raw approval-not-found failure; related fix attempts are closed unmerged, and the open overlapping PR only covers part of the UX and has its own blocker.

## What This Changes

The issue asks for stale Telegram approval callbacks after gateway restart to give clear expiry/restart guidance, with optional durable persistence for pending approvals.

## Best Possible Solution

Implement a narrow UX fix that translates approval-not-found failures into actionable re-request guidance in both generic `/approve` replies and Telegram inline callbacks, while leaving durable approval persistence for a separate design decision.

## Reproduction Assessment

Yes. Current source shows restart-created in-memory approval managers lose pending IDs, and current `/approve` plus Telegram callback tests still simulate `unknown or expired approval id` paths that expose or rethrow the raw failure.

## Solution Assessment

Yes for the immediate UX bug: mapping the existing approval-not-found classifier at user-facing reply/callback handlers is the narrowest maintainable fix. Persisting approvals across restarts is broader than needed for the confusing stale-button error.
