## Summary

Keep open: current main and latest `v2026.6.10` still keep pending exec/plugin approvals in process-local managers, so a gateway restart loses in-flight approval IDs. The merged Telegram fix at https://github.com/openclaw/openclaw/pull/82455 fixed the retry-loop symptom, but the remaining approval lifecycle and fallback-copy behavior needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to preserve pending approvals across gateway restarts or make stale approval callbacks and `/approve` attempts clearly explain restart expiry.

## Best Possible Solution

Adopt one explicit approval lifecycle: either expire restart-stale approvals with clear operator-facing copy on every approval surface, or persist pending approvals in SQLite with expiry, requester/reviewer binding, replay limits, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows gateway aux setup creates fresh `ExecApprovalManager` instances and `register()` stores pending approvals only in a `Map`, so resolving a stale ID after restart reaches the approval-not-found path.

## Solution Assessment

Unclear. Persistence is plausible, but the best fix requires maintainer/security choice between explicit restart invalidation UX and durable SQLite-backed pending approval replay.
