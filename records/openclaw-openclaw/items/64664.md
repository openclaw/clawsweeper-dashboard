## Summary

Keep open. Current main and latest release still keep pending approvals in process-local managers and still allow the `/approve` path to surface the raw stale-id text; the Telegram retry-wedge symptom is fixed, but the remaining gateway UX/taxonomy work is only covered by an open PR with compatibility follow-up, and durable persistence needs a security/product decision.

## What This Changes

The issue asks OpenClaw to handle stale approval callbacks after gateway restart more clearly, with optional durable persistence for pending approvals.

## Best Possible Solution

Land a reviewed stale-terminal approval taxonomy and consumer migration for the UX path, then make a separate maintainer security/product decision before adding durable approval persistence.

## Reproduction Assessment

Yes from source inspection. A pending approval lives in an in-memory `ExecApprovalManager`, gateway startup creates fresh managers, and current `/approve` coverage still expects raw stale-id text.

## Solution Assessment

Unclear as originally proposed. A narrow UX/taxonomy fix is the right near-term direction, but disk-backed approval persistence is not automatically the best fix without a replay-safe security design.
