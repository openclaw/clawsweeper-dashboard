## Summary

Keep open. Current main still uses process-local pending approval managers, still exposes the raw approval-not-found text through `/approve`, and still rethrows stale Telegram approval callback failures; the related implementation attempts are either closed unmerged or only partially cover the UX.

## What This Changes

The issue asks OpenClaw to preserve or gracefully explain stale approval IDs after a gateway restart, especially for Telegram inline approval buttons.

## Best Possible Solution

Implement a narrow UX fix that translates approval-not-found failures into actionable re-request guidance in both generic `/approve` replies and Telegram inline callbacks, while leaving durable approval persistence for a separate security-reviewed design.

## Reproduction Assessment

Yes, at source level. A gateway restart creates fresh in-memory approval managers, and current `/approve` plus Telegram callback paths still expose or rethrow the approval-not-found sentinel when the stale ID is submitted.

## Solution Assessment

Yes for the immediate UX bug: translating the existing approval-not-found classifier at the user-facing handlers is the narrowest maintainable fix. Persisting approvals across process restarts is broader and should be handled separately if desired.
