## Summary

Keep open. The Telegram retry-loop symptom was fixed and shipped through #82455, and current gateway errors now include structured stale-approval remediation, but current main still creates fresh in-memory approval managers on gateway startup, so gateway restart loses in-flight approval IDs. The remaining decision is whether approval state should become durable or intentionally expire on restart with consistent clear guidance across approval clients.

## What This Changes

The issue requests safer handling for approval callbacks and `/approve` attempts after gateway restart or expiry, including avoiding confusing stale-button errors and considering persistence for pending approvals.

## Best Possible Solution

Decide the restart-time approval contract, then either persist pending approvals in SQLite with expiry and replay protections or intentionally expire them on restart and make all approval surfaces show clear re-request guidance.

## Reproduction Assessment

Yes at source level: create an approval, restart the gateway so fresh managers are instantiated, then resolve the old ID and the gateway has no pending record; this read-only review did not run a live restart repro.

## Solution Assessment

No, current main only partially solves it: #82455 is the right narrow Telegram retry-loop mitigation, but the remaining restart-loss and generic stale-approval UX require a product/security decision or a broader fix.
