## Summary

Keep open: current main and `v2026.6.8` still keep pending exec/plugin approvals in process-local managers, so a gateway restart loses approval IDs. The merged Telegram fix at https://github.com/openclaw/openclaw/pull/82455 fixed the retry-loop/wedge symptom, but the remaining persistence versus explicit invalidation behavior needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to preserve or clearly invalidate pending exec/plugin approval callbacks after gateway restarts, especially for stale Telegram approval buttons.

## Best Possible Solution

Choose one explicit product/security direction: either intentionally expire restart-stale approvals with clear operator-facing copy on every approval surface, or persist pending approvals in SQLite with expiry, requester binding, replay limits, and regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main creates fresh process-local `ExecApprovalManager` instances and stores pending approvals only in `Map`s, while missing IDs resolve to `APPROVAL_NOT_FOUND`. I did not run a live restart because this review is read-only.

## Solution Assessment

Unclear: persistence is a plausible fix, but the best fix needs maintainer/security choice between explicit restart invalidation UX and durable SQLite-backed pending approval replay. A narrow copy-only fix would not address the reported state loss.
