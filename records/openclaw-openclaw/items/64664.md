## Summary

Keep open. Current main still stores pending approvals in process-local managers, and the user-facing /approve and Telegram inline callback paths still surface or rethrow the raw `unknown or expired approval id` failure instead of actionable restart/expiry guidance. The related narrow fix PRs are closed unmerged, so the issue remains a valid focused repair candidate.

## What This Changes

The issue requests that stale approval submissions after a gateway restart either survive restart or fail with clearer user guidance instead of a generic approval failure.

## Best Possible Solution

Implement the immediate UX fix by mapping approval-not-found failures to actionable expiry/restart guidance in both the generic /approve reply path and Telegram inline callback path, with focused regression tests. Keep durable pending-approval persistence as a separate product/storage design item.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: current source keeps approvals in in-memory maps recreated on gateway startup, and existing tests explicitly mock `unknown or expired approval id` in both /approve and Telegram callback paths while expecting the raw failure.

## Solution Assessment

Yes for the immediate issue. Translating the existing approval-not-found classifier at the user-facing surfaces is the narrowest maintainable fix; persisting approvals across restarts would be useful but is broader than needed to resolve the confusing stale-button error.
