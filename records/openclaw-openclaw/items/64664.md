## Summary

Keep open: current main and v2026.6.10 still keep pending exec/plugin approvals in process-local managers, so a gateway restart loses in-flight approval IDs. The merged Telegram fix resolved the retry-loop/wedge symptom, but the remaining durable-persistence versus explicit-restart-invalidation behavior still needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to handle stale approval IDs after gateway restart, either by persisting pending approvals or by making expired/restart-invalidated approval callbacks clearly recoverable.

## Best Possible Solution

Make the restart lifecycle explicit: either persist pending approvals in the canonical state database with strict expiry and identity bindings, or deliberately expire restart-stale approvals and surface clear re-request guidance across Telegram callbacks, /approve, and gateway clients.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a pending approval, restart the gateway so fresh in-memory managers are constructed, then resolve the stale ID and hit APPROVAL_NOT_FOUND; I did not run a live restart proof in this read-only review.

## Solution Assessment

Unclear. Persistence is plausible, but the best fix needs maintainer/security choice between durable SQLite-backed pending approvals and explicit restart invalidation UX; the merged Telegram PR only solved the retry-loop symptom.
