## Summary

Keep open. Current main fixed the Telegram infinite retry/spool wedge, but pending approval state is still process-local and the generic `/approve` path still surfaces the raw `unknown or expired approval id` failure instead of actionable recovery guidance.

## What This Changes

The issue asks for approvals lost on gateway restart to stop producing confusing stale Telegram approval-button and `/approve` errors, either through durable pending-approval state or clearer expiry/restart guidance.

## Best Possible Solution

Keep the PR 82455 terminal Telegram callback behavior, add actionable recovery copy for the remaining `/approve` stale-ID path, and have maintainers decide separately whether bounded secure approval persistence belongs in the gateway lifecycle.

## Reproduction Assessment

Yes, at source level. Creating an approval registers it only in the current process manager, while a stale `/approve` submission reaches tests and code paths that still return `unknown or expired approval id`; I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

Unclear for the full issue. The immediate copy fix is narrow, but durable approval persistence changes the gateway approval lifecycle and should be an explicit maintainer design decision rather than an automatic cleanup patch.
