## Label Justifications

- `P2`: The issue affects a real approval workflow after gateway restarts, but the blast radius is bounded to pending approvals and stale channel interactions.
- `impact:session-state`: Pending approval state is process-local and is lost across gateway restart, leaving visible approval messages out of sync with gateway state.

## Summary

Keep open. Current main and the latest release fixed the Telegram infinite retry symptom, but pending exec approvals are still held in process memory and several stale approval paths still either return the raw not-found text or silently clear/drop the callback without actionable restart guidance.

## What This Changes

The issue asks for pending approval actions to survive gateway restarts or at least show clear expiry/restart recovery guidance instead of a confusing stale approval error.

## Best Possible Solution

Keep this issue open until maintainers choose the restart semantics, then implement either bounded SQLite-backed approval persistence with secure expiry/replay rules or an explicit UX-only stale approval response if persistence is out of scope.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a pending exec approval, restart the gateway so the in-memory Map is lost, then resolve the old approval id through Telegram or /approve. I did not run a live gateway restart scenario in this read-only review.

## Solution Assessment

Unclear. A narrow recovery-message change is maintainable but incomplete; durable persistence may be the right end state, but it needs an explicit security and product decision about what can safely survive restart.
