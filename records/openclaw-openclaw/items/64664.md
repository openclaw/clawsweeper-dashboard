## Summary

Keep open. Current main fixed the Telegram infinite retry symptom, but pending approval IDs are still process-local and the generic `/approve` path can still expose the raw stale approval error instead of recovery guidance.

## What This Changes

The issue requests clearer stale approval handling after gateway restart and raises durable pending-approval persistence as a possible fix.

## Best Possible Solution

Keep the terminal Telegram callback handling already on main, add actionable stale-ID recovery copy for the remaining `/approve` path, and handle durable approval persistence as a separate maintainer-approved storage/security design.

## Reproduction Assessment

Yes, at source level. Gateway startup creates fresh approval managers and the manager stores pending records only in a private in-memory `Map`, so a restarted gateway cannot resolve old approval IDs; current `/approve` tests still assert raw not-found copy.

## Solution Assessment

Unclear for the full issue. A narrow stale-ID message fix is maintainable, but durable approval persistence changes approval capability lifetime across process restarts and needs maintainer design approval.
