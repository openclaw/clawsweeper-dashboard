## Summary

Keep open. Current main still stores pending approvals only in process-local managers and the manual `/approve` path still exposes the raw stale-id text; the Telegram retry-wedge symptom was fixed by a merged PR, and the open gateway-taxonomy PR is promising but still blocked on consumer migration.

## What This Changes

The issue reports that gateway restarts lose pending approval IDs, leaving stale Telegram approval buttons and manual approval retries to show confusing stale-id errors.

## Best Possible Solution

Land a reviewed stale-approval terminal UX/taxonomy fix with channel consumer migration, then make an explicit security/product decision before adding durable approval persistence.

## Reproduction Assessment

Yes. Source inspection shows pending approvals live in an in-memory manager and gateway startup creates fresh managers, while the current `/approve` test still expects the raw stale-id error text.

## Solution Assessment

No, not as a persistence-first change. A narrow stale-terminal UX/taxonomy fix is appropriate now, while durable restart persistence needs a separate replay-safe security design.
