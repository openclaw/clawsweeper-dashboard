## Summary

Keep open. Current main has fixed the Telegram retry-loop symptom and now attaches structured remediation to missing approval errors, but gateway restart still recreates empty in-memory approval managers, so in-flight approval IDs are still lost and the durable-vs-expire-on-restart contract remains unsettled.

## What This Changes

The issue asks OpenClaw to handle pending approvals after gateway restart so stale Telegram approval buttons do not leave users with confusing missing-approval errors.

## Best Possible Solution

Decide the restart-time approval contract, then either persist pending approvals in SQLite with expiry and replay protections or intentionally expire them on restart with consistent clear guidance across approval clients.

## Reproduction Assessment

Yes, source-reproducible: create/register an approval, restart the gateway so fresh ExecApprovalManager maps are created, then resolve the old ID and the gateway has no pending record. This review did not run a live restart repro.

## Solution Assessment

No, current main is only partial: the Telegram retry loop is fixed and backend errors have remediation details, but approval loss across restart remains. PR https://github.com/openclaw/openclaw/pull/92376 may improve stale classification, but the durable-vs-expire policy still needs maintainer direction.
