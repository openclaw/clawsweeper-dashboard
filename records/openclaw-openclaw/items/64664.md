## Summary

Keep open. Current main and v2026.6.6 still keep pending approvals in per-process managers, so a gateway restart drops pending IDs; Telegram retry-wedge handling is partly fixed, but the remaining stale UX and restart-persistence question is now covered only by unmerged related work with compatibility blockers.

## What This Changes

The issue asks OpenClaw to handle approval IDs lost after gateway restart by preserving pending approvals or showing clearer stale-approval guidance in Telegram and /approve flows.

## Best Possible Solution

Land or repair a bounded stale-terminal approval taxonomy with channel consumer migration, then decide separately whether replay-safe durable approval persistence belongs in SQLite.

## Reproduction Assessment

Yes from source inspection. Pending approvals live in ExecApprovalManager.pending and gateway startup creates fresh managers, so a restart drops the pending ID; I did not run a live restart in this read-only pass.

## Solution Assessment

Unclear as originally proposed. A narrow stale-terminal UX fix is appropriate, but durable restart survival needs a replay-safe SQLite design and explicit security/product approval rather than a simple JSON fallback.
