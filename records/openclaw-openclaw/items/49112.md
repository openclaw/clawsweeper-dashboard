## Summary

Keep open: current main still lacks gateway-lock doctor coverage, but this PR is not merge-ready because the added inspector can remove confirmed-live locks, uses weaker owner detection than gateway startup, wires into an obsolete doctor path, conflicts with main, and lacks real behavior proof.

## What This Changes

The PR adds a gateway lock inspector, a doctor note helper wired into the old doctor command path, fast-path mocks, and tests for reporting/removing stale gateway locks.

## Best Possible Solution

Land a rebased implementation that registers a gateway-lock doctor contribution, reuses gateway startup owner-status semantics, only removes dead/malformed or non-confirmed-live stale locks under --fix, and includes focused tests plus live proof.

## Reproduction Assessment

Yes, by source inspection. Current main registers session-lock doctor health but no gateway-lock contribution, and the PR's live-lock age-out and owner-status divergences are visible in the added inspector.

## Solution Assessment

No. The direction is useful, but the patch should be rebased into the doctor contribution pipeline, reuse the existing gateway owner-status contract, avoid removing confirmed-live locks, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:225-231`
  - body: The inspector marks any lock older than staleMs as stale and doctor --fix removes it. A healthy gateway can hold its singleton lock for far longer than 30 seconds, so this can delete an active guard and allow another gateway to start.
  - confidence: 0.94
- **[P2] Reuse the gateway owner-status contract:** `src/infra/gateway-lock.ts:208`
  - body: This only checks isPidAlive, while startup also checks port reachability, Linux start time, and argv shape. A recycled PID or unrelated process can look healthy to doctor even when startup would reclaim the stale lock.
  - confidence: 0.91
- **[P2] Register the check in doctor health contributions:** `src/commands/doctor.ts:226`
  - body: Current main routes doctor checks through runDoctorHealthContributions. Adding the helper to the old command body is stale against main and will not be part of the active contribution pipeline after rebase.
  - confidence: 0.88
- **[P3] Render unknown PID status as unknown:** `src/commands/doctor-gateway-locks.ts:27`
  - body: GatewayLockInspection allows pidAlive to be null, but the formatter treats any falsy value as dead. Unknown owner status should not be displayed as a confirmed-dead process.
  - confidence: 0.78
