## Summary

Keep this PR open, but it should not merge as-is. Current main still lacks gateway-lock doctor coverage, so the linked repair remains useful, but the branch is stale against the current doctor contribution flow and the proposed inspector can both delete active gateway locks and miss recycled or wrong-owner PIDs.

## What This Changes

The PR adds a gateway-lock inspection helper, doctor note wiring, fast-path mocks, and tests so `openclaw doctor` can report and optionally remove stale gateway lock files.

## Best Possible Solution

Add gateway-lock doctor health through the current contribution registry, backed by the same owner-status semantics used by gateway startup. Repair should remove dead owners immediately, use age only for unknown ownership, preserve confirmed-live locks, update user-facing docs and `CHANGELOG.md`, and cover live-old, recycled-PID, stale-without-fix, and stale-with-fix cases.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the doctor contribution registry and docs; the PR bug is reproducible from the diff by creating a lock with a live PID and old `createdAt`, then calling the new inspector with `removeStale: true`.

## Solution Assessment

No. The feature direction is right, but this patch is not the best solution because it reimplements weaker owner detection and wires into an older doctor flow; the safer path is to share or expose the startup owner-status helper and register the check through the current contribution pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:222-228`
  - body: The inspector treats `ageMs > staleMs` as stale even when `pidAlive` is true, and `doctor --fix` then removes the lock. A healthy gateway lock can live longer than 30 seconds, so this can delete an active lock and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse gateway owner status for inspection:** `src/infra/gateway-lock.ts:203-205`
  - body: `inspectGatewayLock` only calls `isPidAlive`, while startup also checks the port, Linux start time, and process argv. If a PID is recycled or belongs to another process, doctor can report `stale=no` even though startup would reclaim the lock.
  - confidence: 0.91
- **[P3] Render unknown PID status accurately:** `src/commands/doctor-gateway-locks.ts:25-27`
  - body: `pidAlive` is typed as `boolean | null`, but the formatter renders any falsy value as `dead`. If inspection cannot determine owner status, doctor should show `unknown` rather than telling operators the owner is confirmed dead.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/doctor-gateway-locks.ts:52`
  - body: This adds user-facing doctor diagnostics and `--fix` repair behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix/feat/perf changes to include a changelog entry.
  - confidence: 0.86
