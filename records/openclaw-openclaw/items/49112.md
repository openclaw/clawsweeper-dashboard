## Summary

Keep this PR open: current main still lacks gateway-lock doctor coverage, but the proposed patch is not mergeable as-is because it can delete live gateway locks, uses weaker liveness semantics than gateway startup, and wires into an outdated doctor command path.

## What This Changes

The PR adds gateway lock inspection, doctor output, `--fix` stale-lock removal, fast-path mocks, and focused tests for gateway-lock doctor diagnostics.

## Best Possible Solution

Implement gateway-lock doctor health as a registered doctor contribution backed by the same owner-status logic as gateway startup, preserving live locks while reporting and repairing dead or stale-unknown owners with tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no gateway-lock doctor contribution, and the PR defects are provable from the supplied diff against the existing gateway-lock owner-status contract.

## Solution Assessment

No. The direction is useful, but the implementation should reuse `resolveGatewayOwnerStatus`, only age out non-confirmed-live owners, and register through the current doctor contribution pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:222-228`
  - body: The inspector marks a lock stale whenever `ageMs > staleMs`, and `doctor --fix` removes stale locks. Healthy gateway locks normally live longer than 30 seconds, so this can delete an active gateway's lock and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse gateway owner status for inspection:** `src/infra/gateway-lock.ts:203-205`
  - body: `inspectGatewayLock` computes liveness with `isPidAlive` only, while gateway startup uses `resolveGatewayOwnerStatus` with port probing, Linux start-time matching, and argv validation. Recycled PIDs or unrelated processes can look healthy to doctor even though startup would reclaim the lock.
  - confidence: 0.91
- **[P2] Register the check through doctor contributions:** `src/commands/doctor.ts:223-224`
  - body: Current main runs doctor health checks through `runDoctorHealthContributions`, but the patch wires the gateway-lock check into the old command body. After rebase, this should be a doctor health contribution so the active pipeline runs it.
  - confidence: 0.88
- **[P3] Render unknown PID status accurately:** `src/commands/doctor-gateway-locks.ts:25-27`
  - body: `pidAlive` is typed as `boolean | null`, but the formatter treats any falsy value as dead. If inspection cannot determine liveness, doctor output should say unknown rather than implying the owner is confirmed dead.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/doctor-gateway-locks.ts:55-57`
  - body: This adds user-facing doctor diagnostics and `--fix` repair behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix, feat, and perf changes to include a changelog entry.
  - confidence: 0.86
