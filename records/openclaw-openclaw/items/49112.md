## Summary

Keep open: current main still lacks a gateway-lock doctor contribution, but this PR is not mergeable as-is because its inspector can delete active gateway locks and is wired into a stale doctor command path.

## What This Changes

The PR adds a gateway lock inspection helper, doctor output and repair wiring, fast-path mocks, and tests for stale gateway lock files.

## Best Possible Solution

Implement gateway-lock doctor health as a registered doctor contribution backed by the same owner-status logic as gateway startup, with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no gateway-lock doctor contribution, and the PR defects are provable from the diff against the existing gateway owner-status contract.

## Solution Assessment

No. The feature direction is right, but the patch should preserve confirmed-live locks, reuse startup owner-status semantics, and register through the current doctor contribution pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:222-223`
  - body: The inspector adds `age` as a stale reason whenever `ageMs > staleMs`, and `doctor --fix` removes stale locks. A healthy gateway lock older than 30 seconds is normal, so this can delete an active gateway's lock and allow a second instance to start.
  - confidence: 0.94
- **[P2] Reuse gateway owner status for inspections:** `src/infra/gateway-lock.ts:203-205`
  - body: `inspectGatewayLock` only checks `isPidAlive`, while gateway startup also checks port reachability, Linux start time, and process argv. Recycled PIDs or unrelated processes can look healthy to doctor even though startup would reclaim the lock.
  - confidence: 0.91
- **[P2] Register the check in doctor health contributions:** `src/commands/doctor.ts:223-224`
  - body: Current main runs doctor health through `runDoctorHealthContributions`, but this patch wires the check into the old command body. Move the gateway-lock check into the contribution registry so it runs in the active doctor pipeline after rebase.
  - confidence: 0.88
- **[P3] Render unknown PID status explicitly:** `src/commands/doctor-gateway-locks.ts:25-27`
  - body: `pidAlive` is typed as `boolean | null`, but the formatter renders any falsy value as `dead`. If owner status is unknown, doctor should say `unknown` rather than implying a confirmed-dead process.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/doctor-gateway-locks.ts:55-57`
  - body: This adds user-facing doctor diagnostics and repair behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fixes and features to include a changelog entry.
  - confidence: 0.86
