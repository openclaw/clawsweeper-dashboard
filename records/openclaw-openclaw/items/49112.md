## Summary

Keep open: the linked doctor gap is still real on current main, but this PR is not mergeable as-is because it targets the old doctor command body, can remove live gateway locks in repair mode, uses weaker owner detection than gateway startup, and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds a gateway-lock inspector, a doctor note helper, mocks, and tests so `openclaw doctor` can report and remove stale gateway lock files with `--fix`.

## Best Possible Solution

Land a rebased doctor health contribution that reports gateway lock state and repairs only dead, malformed, or non-confirmed-live stale locks using the gateway owner-status contract, with focused tests and redacted command-output proof.

## Reproduction Assessment

Yes. Source inspection shows current main runs doctor health contributions and has session-lock coverage, but no gateway-lock doctor contribution or inspector; I did not run `openclaw doctor` because this review is read-only.

## Solution Assessment

No. The feature direction is right, but the patch should be rebased into `src/flows/doctor-health-contributions.ts` and should mirror the gateway owner-status contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not delete live gateway locks:** `src/infra/gateway-lock.ts:221-227`
  - body: The inspector treats age alone as stale and removes the file in repair mode, even when the recorded PID is alive. Healthy gateway locks can live past the 30s threshold, so `openclaw doctor --fix` could remove an active singleton guard.
  - confidence: 0.94
- **[P2] Reuse the gateway owner-status contract:** `src/infra/gateway-lock.ts:204`
  - body: This reduces owner detection to `isPidAlive(pid)`, while gateway startup also checks port reachability, Linux start time, and gateway argv shape. A recycled PID or unrelated process can look healthy to doctor even when startup would reclaim the stale lock.
  - confidence: 0.91
- **[P2] Register the check in doctor health contributions:** `src/commands/doctor.ts:223`
  - body: Current main routes doctor checks through `runDoctorHealthContributions()`. Adding this helper to the old command body is stale after rebase and will not put the gateway-lock check into the active doctor pipeline.
  - confidence: 0.88
