## Summary

Keep open: current main still lacks gateway-lock doctor coverage, and this PR is a plausible vehicle, but the patch remains unsafe and stale against the current doctor contribution pipeline.

## What This Changes

The PR adds a gateway lock inspector, doctor note helper, fast-path mock, command wiring, and tests for stale gateway lock reporting and cleanup.

## Best Possible Solution

Land a rebased implementation that registers gateway-lock doctor health as a contribution, reuses the gateway startup owner-status contract, repairs only dead or stale-unknown/malformed locks under `--fix`, and adds focused tests/docs.

## Reproduction Assessment

Yes, by source inspection. Current main has session-lock doctor health but no gateway-lock doctor contribution/helper, and the PR diff exposes the owner-status and live-lock age-out problems without needing a runtime setup.

## Solution Assessment

No. The direction is maintainable, but the current patch should be changed to preserve startup lock semantics, register through the contribution registry, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:225-231`
  - body: The inspector adds `age` to stale reasons whenever the lock exceeds `staleMs`, and `doctor --fix` removes stale locks. A healthy gateway can hold its lock longer than 30 seconds, so this can delete an active singleton guard and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse the gateway owner-status contract:** `src/infra/gateway-lock.ts:208`
  - body: This only calls `isPidAlive`, but startup also checks port reachability, Linux start time, and process argv. Recycled PIDs or unrelated processes can therefore look healthy to doctor even though startup would reclaim the lock.
  - confidence: 0.91
- **[P2] Register the check in doctor health contributions:** `src/commands/doctor.ts:226`
  - body: Current main delegates doctor checks through `runDoctorHealthContributions`; wiring the helper into the old command body will not be part of the active pipeline after rebase. Add a contribution in `src/flows/doctor-health-contributions.ts` instead.
  - confidence: 0.88
- **[P3] Render unknown PID status as unknown:** `src/commands/doctor-gateway-locks.ts:27`
  - body: `pidAlive` is typed as `boolean | null`, but this formatter renders any falsy value as `dead`. Unknown owner status should not be shown as a confirmed-dead process.
  - confidence: 0.78
