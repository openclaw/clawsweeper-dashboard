## Summary

Keep open: the linked doctor coverage gap still exists on current main, but this PR is not mergeable as-is because it lacks real behavior proof and the patch can delete live gateway locks while wiring into an outdated doctor path.

## What This Changes

Adds a gateway-lock inspector, doctor output, `--fix` stale-lock removal, mocks, and tests for gateway lock health checks.

## Best Possible Solution

Land a rebased doctor health contribution that reports gateway lock state and repairs only dead, malformed, or non-confirmed-live stale locks using the existing owner-status semantics, with focused tests and redacted command-output proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has session-lock doctor coverage and gateway runtime stale-lock recovery, but no gateway-lock doctor contribution or helper; no live command was run because this review is read-only.

## Solution Assessment

No. The feature direction is right, but this patch should be rebased into `src/flows/doctor-health-contributions.ts` and should reuse the existing gateway owner-status contract before any `--fix` removal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve live gateway locks during repair:** `src/infra/gateway-lock.ts:221-227`
  - body: `inspectGatewayLock()` marks a lock stale solely because it is older than `staleMs`, even when `pidAlive` is true. A healthy gateway lock normally lives past 30 seconds, so `openclaw doctor --fix` could remove an active singleton guard and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse the gateway owner-status contract:** `src/infra/gateway-lock.ts:208`
  - body: The inspector reduces ownership to `isPidAlive(pid)`, while startup also checks port reachability, Linux start time, and gateway argv shape. A recycled PID or unrelated process can look healthy to doctor even when startup would reclaim the stale lock.
  - confidence: 0.91
- **[P2] Register the check in doctor health contributions:** `src/commands/doctor.ts:226`
  - body: Current main dispatches doctor checks through `runDoctorHealthContributions()`. Adding this helper to the old command body will not integrate the gateway-lock check after rebase; it needs a contribution in `src/flows/doctor-health-contributions.ts`.
  - confidence: 0.88
