## Summary

Keep open: current main still lacks gateway-lock doctor coverage for linked #49036, but the proposed patch needs changes before merge because it does not match current gateway lock ownership semantics or the current doctor contribution flow.

## What This Changes

The PR adds gateway lock inspection, doctor output and repair wiring, fast-path mocks, and tests so `openclaw doctor` can report and remove stale gateway lock files.

## Best Possible Solution

Implement gateway-lock doctor health as a registered doctor contribution backed by the same owner-status logic as gateway startup, preserving confirmed-live locks while repairing dead or stale-unknown owners and updating tests, docs, and changelog.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection shows current main has no gateway-lock doctor contribution, and the PR defects are provable from the diff against the existing gateway-lock owner-status contract.

## Solution Assessment

Is this the best way to solve the issue? No. The feature direction is right, but the patch should reuse startup owner-status semantics and register through the current doctor contribution pipeline instead of adding a weaker, stale integration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:222-228`
  - body: The inspector marks a lock stale solely because `ageMs > staleMs`, then `doctor --fix` removes it. Healthy gateway locks routinely live longer than 30 seconds, so this can delete an active gateway's lock and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse gateway owner status for inspection:** `src/infra/gateway-lock.ts:203-205`
  - body: `inspectGatewayLock` only checks `isPidAlive`, while startup also checks the port, Linux start time, and process argv. Recycled PIDs or unrelated processes can therefore look healthy to doctor even though startup would reclaim the lock.
  - confidence: 0.91
- **[P2] Register the check through doctor contributions:** `src/commands/doctor.ts:223-224`
  - body: Current main runs doctor health through `runDoctorHealthContributions`, but this patch wires the check into the old command body. Move the gateway-lock check into `src/flows/doctor-health-contributions.ts` so it runs in the active doctor pipeline after rebase.
  - confidence: 0.88
- **[P3] Render unknown PID status accurately:** `src/commands/doctor-gateway-locks.ts:25-27`
  - body: `pidAlive` is typed as `boolean | null`, but the formatter renders any falsy value as `dead`. If owner status is unknown, doctor should say `unknown` rather than implying a confirmed-dead process.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/doctor-gateway-locks.ts:55-57`
  - body: This adds user-facing doctor diagnostics and repair behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix, feat, and perf changes to include a changelog entry.
  - confidence: 0.86
