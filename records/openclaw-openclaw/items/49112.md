## Summary

Keep this PR open: the requested doctor coverage is still absent from current main, but the patch is not merge-ready because it can remove confirmed-live gateway locks and is wired into a stale doctor command path rather than the current contribution pipeline.

## What This Changes

The PR adds gateway lock inspection, doctor output and repair wiring, fast-path mocks, and tests so `openclaw doctor --fix` can report and remove stale gateway lock files.

## Best Possible Solution

Implement gateway-lock doctor health as a registered doctor contribution backed by the same owner-status logic as gateway startup, preserving live locks while repairing dead or stale-unknown owners with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no gateway-lock doctor contribution, and the PR defects are provable from the diff against the existing gateway-lock owner-status contract.

## Solution Assessment

No. The direction is useful, but the implementation should reuse `resolveGatewayOwnerStatus`, only age out non-confirmed-live owners, and register through the current doctor contribution pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not age out confirmed-live gateway locks:** `src/infra/gateway-lock.ts:222-228`
  - body: The inspector marks a lock stale solely because `ageMs > staleMs`, and `doctor --fix` removes stale locks. Healthy gateway locks can live longer than 30 seconds, so this can delete an active gateway's lock and allow a second gateway to start.
  - confidence: 0.94
- **[P2] Reuse gateway owner status for inspection:** `src/infra/gateway-lock.ts:203-205`
  - body: `inspectGatewayLock` only checks `isPidAlive`, while startup also checks port reachability, Linux start time, and process argv. Recycled PIDs or unrelated processes can therefore look healthy to doctor even though startup would reclaim the lock.
  - confidence: 0.91
- **[P2] Register the check through doctor contributions:** `src/commands/doctor.ts:223-224`
  - body: Current main runs doctor health through `runDoctorHealthContributions`, but this patch wires the check into the old command body. Move the gateway-lock check into the contribution list so it runs in the active doctor pipeline after rebase.
  - confidence: 0.88
- **[P3] Render unknown PID status accurately:** `src/commands/doctor-gateway-locks.ts:25-27`
  - body: `pidAlive` is typed as `boolean | null`, but the formatter renders any falsy value as `dead`. Unknown owner status should be shown as `unknown` rather than implying a confirmed-dead process.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/doctor-gateway-locks.ts:55-57`
  - body: This adds user-facing doctor diagnostics and repair behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fix, feat, and perf changes to include a changelog entry.
  - confidence: 0.86
