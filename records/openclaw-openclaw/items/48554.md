## Summary

Keep open. Current main still has the proxy-specific FormData gap: the proxy fetch wrappers pass caller bodies straight into `undici.fetch`, while the audio path builds standard `FormData` and supplies the proxy fetch as a custom fetch implementation. The issue is also paired with open PR #48556 using closing syntax, so the issue should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue requests that proxy-aware fetch normalize standard FormData bodies so audio transcription and batch uploads behind HTTP(S)_PROXY send multipart data instead of `[object FormData]`.

## Best Possible Solution

Land a narrow proxy-fetch fix that makes both `makeProxyFetch()` and `resolveProxyFetchFromEnv()` drop-in compatible with standard FormData bodies, preserves existing undici FormData instances, clears stale multipart content headers when rebuilding, and adds regression coverage for proxy-backed multipart uploads.

## Reproduction Assessment

Yes. The issue includes a focused Node/undici reproduction script, and current main still follows the same source path from standard FormData through a custom proxy fetch into `undiciFetch` without normalization.

## Solution Assessment

Yes. The proxy wrapper is the maintainable fix boundary because callers are correct to use standard FormData; the existing open PR or a narrow replacement should repair that boundary rather than changing audio or batch callers.
