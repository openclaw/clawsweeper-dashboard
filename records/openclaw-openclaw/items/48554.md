## Summary

Keep open. Current main still has the proxy-specific FormData gap: the proxy fetch wrappers pass caller bodies straight into `undici.fetch`, while the audio path builds standard `FormData` and supplies the proxy fetch as a custom fetch implementation; #48556 is the paired open fix path.

## What This Changes

The issue reports that proxy-backed audio transcription and batch uploads serialize standard `FormData` as `[object FormData]` instead of multipart data when `HTTP_PROXY` or `HTTPS_PROXY` is configured.

## Best Possible Solution

Land or replace #48556 with a narrow proxy-fetch fix that preserves standard `FormData` semantics for both proxy wrapper entry points, avoids double-converting undici `FormData`, clears stale multipart headers when rebuilding, and adds proxy-backed multipart regression coverage.

## Reproduction Assessment

Yes. The issue includes a focused Node/undici reproduction script, and current main still follows the same source path from standard `FormData` through a custom proxy fetch into `undiciFetch` without normalization.

## Solution Assessment

Yes. The proxy wrapper is the maintainable fix boundary because callers are correct to use standard `FormData`; the paired PR or a narrow replacement should repair that boundary rather than changing audio or batch callers.
