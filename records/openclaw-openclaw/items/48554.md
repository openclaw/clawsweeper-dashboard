## Summary

Keep open. Current main still has the proxy-specific multipart gap: the proxy fetch wrappers pass caller init directly into undici fetch, while the existing FormData normalizer is only on the runtime-dispatcher path and is bypassed for caller-supplied proxy fetch implementations. The issue is also paired with open PR #48556, so it should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

Audio transcription and batch upload FormData bodies should remain multipart when HTTP_PROXY or HTTPS_PROXY routes requests through the proxy fetch wrappers.

## Best Possible Solution

Review, rebase, land, or replace PR #48556 with a narrow proxy-fetch fix that preserves global FormData multipart semantics for both proxy wrapper entry points, avoids double-converting undici FormData, clears stale multipart body headers when rebuilding the body, and adds focused proxy-fetch regression coverage.

## Reproduction Assessment

Yes. The issue includes a focused FormData/undici reproduction script, and current main still follows the same source path from global FormData through resolveProxyFetchFromEnv into undiciFetch without normalization.

## Solution Assessment

Yes. Fixing the proxy-fetch wrapper is the narrowest maintainable boundary because callers are correct to use standard FormData and the proxy wrapper is the component swapping in undici under proxy configuration.
