## Summary

Keep open. Current main and v2026.5.7 still expose only transport-level Vertex `options.location` plus `GOOGLE_CLOUD_LOCATION`; per-model params are accepted, but the Google plugin has no hook that maps `params.location` into Vertex transport options, so mixed global/regional Vertex models under one project remains a valid feature gap.

## What This Changes

Request per-model Google Vertex location override so preview and GA Gemini models can use different Vertex endpoints with the same credentials.

## Best Possible Solution

Support and document `agents.defaults.models["google-vertex/<model>"].params.location`, forward it through a Google-plugin transport hook into Vertex options, and keep environment fallback behavior intact.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure two `google-vertex` model entries with different desired `params.location` values; current main reads those params but never maps `location` into Vertex transport options.

## Solution Assessment

Yes for the explicit override. The narrow maintainable path is a Google-plugin-owned `extraParamsForTransport` mapping for `params.location`, not a provider-specific top-level schema key or hard-coded preview defaults.
