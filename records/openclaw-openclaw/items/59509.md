## Summary

Keep open. Current main still resolves Google Vertex location from transport options or `GOOGLE_CLOUD_LOCATION`, and the existing per-model `params` seam is not yet mapped into the Google Vertex transport; the open closing PR should be reviewed or reworked rather than closing this issue early.

## What This Changes

Request support for a per-model Google Vertex location override so different `google-vertex` models can use different global or regional endpoints under one project and credential set.

## Best Possible Solution

Implement a Google-plugin-owned `params.location` transport patch for `google-vertex`, preserve the existing env fallback, document it, and cover mixed global/regional models with focused tests.

## Reproduction Assessment

Yes, source-level. Configure two `google-vertex` model entries with different `params.location` values; current main can read those params but never maps `location` into the Vertex transport path.

## Solution Assessment

Yes for the explicit override if it uses the existing provider hook seam. A new top-level model field or automatic preview-model defaults would be broader than needed.
