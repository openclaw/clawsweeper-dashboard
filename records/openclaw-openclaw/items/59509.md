## Summary

Keep this open. Current main has lower-level Google Vertex transport support for an option-level location and a generic provider transport-param seam, but the Google plugin still does not expose or forward a per-model Vertex location from model config, so the reported mixed global/regional endpoint use case remains valid.

## What This Changes

Request per-model Google Vertex location configuration so preview models can use the global endpoint while GA models use regional endpoints under the same project and credentials.

## Best Possible Solution

Implement this in the Google provider plugin by mapping `agents.defaults.models["google-vertex/<model>"].params.location` into the Vertex transport options, keeping the existing environment fallback, documenting the supported config shape, and adding focused tests for global and regional Vertex models coexisting under one project.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `google-vertex` models with different desired `params.location` values: current main accepts arbitrary `params`, but no Google hook forwards `location` into `resolveGoogleVertexLocation`, while a top-level model `location` is rejected by the generated schema.

## Solution Assessment

No current solution is in place. The best narrow fix is provider-owned mapping through `extraParamsForTransport`, not a provider-specific top-level key in the generic model-entry schema or hard-coded preview-model defaults.
