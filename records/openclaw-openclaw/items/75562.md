## Summary

Keep open. Current main still only has global/env-managed proxy routing for web-search calls; the core web-search schema, Brave and Perplexity plugin manifests, and provider runtimes do not expose or consume provider-scoped proxy settings. The request is valid, but it changes SSRF-protected egress and needs maintainer/security design rather than an automatic narrow fix.

## What This Changes

Request to add per-provider proxy configuration and env vars for web_search providers such as Brave and Perplexity so selected provider API calls can use an HTTP proxy without routing all web tools through it.

## Best Possible Solution

Add an approved provider-owned web-search proxy/request config surface, likely under each plugin's webSearch config plus optional provider env fallbacks, and route it through a shared guarded explicit-proxy helper that validates both the configured proxy and the final provider API target. Update the plugin manifests, config/schema docs, generated config metadata, provider tests, and coordinate the SSRF/private-network policy with https://github.com/openclaw/openclaw/issues/74357 before landing.

## Reproduction Assessment

Yes for the missing capability by source inspection rather than a live network run. Current schemas/manifests reject provider proxy fields, and Brave/Perplexity runtimes have no code path that reads BRAVE_PROXY_URL, PERPLEXITY_PROXY_URL, or webSearch.proxy before calling the shared guarded fetch helper.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the unsafe local workaround described in the issue is not the best fix; the maintainable path is a guarded provider-owned proxy transport seam rather than raw dispatcher use that bypasses SSRF checks.
