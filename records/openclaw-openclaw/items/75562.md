## Summary

Keep open. Current main still supports only global/process web-tool proxy routing for web search, while Brave and Perplexity plugin webSearch configs remain strict and do not expose a provider-scoped proxy field. The request is valid, but implementation is security-sensitive because it changes guarded web-search egress and overlaps the still-open SSRF policy split in #74357.

## What This Changes

Request provider-scoped proxy configuration and optional provider env vars for web_search providers so one provider such as Brave can use a proxy without routing all web tools or runtime traffic through global proxy settings.

## Best Possible Solution

Add a maintainer-approved plugin-owned web-search transport override, likely under each provider's webSearch config with a request.proxy-style shape, and route it through a shared guarded explicit-proxy helper that validates both the configured proxy and the final provider API target. Update plugin manifests, generated config/help metadata, provider docs, and focused tests, and resolve the private-network policy split from #74357 before landing.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas reject provider proxy fields, and the Brave/Perplexity runtimes never resolve or pass provider-specific proxy URLs.

## Solution Assessment

Unclear pending maintainer/security design. Provider-scoped routing is the right direction for the user need, but copying Telegram's raw makeProxyFetch pattern or using fetchWithRuntimeDispatcher directly would not be the best fix because web search must preserve fetchWithSsrFGuard protections.
