## Summary

Keep open. Current main has global env/managed proxy routing for web tools, but there is still no provider-scoped web-search proxy config or runtime path; the request is valid and overlaps security-sensitive SSRF/proxy policy work that should get maintainer design approval before implementation.

## What This Changes

Add a per-provider web-search proxy configuration surface so providers such as Brave and Perplexity can route only their API calls through a dedicated HTTP proxy instead of using global process proxy settings.

## Best Possible Solution

Add a provider-owned web-search request proxy setting under the plugin webSearch config surface, route Brave/Perplexity and any shared provider helper through a guarded explicit-proxy path that preserves SSRF checks, and document/test the policy together with the related trusted-network split.

## Reproduction Assessment

Not applicable as a classic runtime bug. The missing capability is high-confidence by source inspection: strict plugin schemas do not accept provider proxy fields, and the Brave/Perplexity runtimes never resolve or pass a provider-specific proxy URL.

## Solution Assessment

Unclear pending maintainer/security design. The direction is valid, but the safest implementation is a plugin-owned config plus guarded explicit-proxy helper, not the local workaround that bypasses SSRF protection or a Brave-only baseUrl workaround.
