## Summary

Keep open. Current main still lacks provider-scoped web-search proxy config or provider-specific proxy env vars; this is a security-sensitive feature/config change, not an implemented or stale bug report.

## What This Changes

The issue requests per-web-search-provider proxy configuration, such as `tools.web.search.brave.proxy` or `BRAVE_PROXY_URL`, so one search provider can use a proxy without affecting all web tools or channels.

## Best Possible Solution

Define an approved generic web-search transport config surface that reuses the guarded explicit-proxy path, keeps public provider API targets under strict SSRF policy, redacts proxy credentials, and updates provider manifests, docs, generated config metadata, and focused Brave/Perplexity coverage together.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction because this is a feature request. Source inspection gives high confidence the capability is missing: strict schemas reject provider `webSearch.proxy`, provider-specific proxy env vars were not found, and the shared helper has no explicit proxy parameter.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the maintainable solution is a generic guarded web-search proxy seam, not a Brave-only raw dispatcher patch or direct reuse of Telegram transport code.
