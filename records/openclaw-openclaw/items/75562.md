## Summary

Keep open. Current main supports global managed/env proxy routing for web tools, but it still has no per-provider web-search proxy config or runtime path, and the related items are adjacent rather than terminal: #27405 covered global HTTP_PROXY behavior, #19075 tracks a Brave baseUrl workaround, and #74357 tracks the SSRF policy split that this work should coordinate with.

## What This Changes

The issue requests per-provider proxy configuration for web search providers so one provider such as Brave can use a local HTTP proxy without routing every web tool through global proxy settings.

## Best Possible Solution

Add a provider-owned web-search proxy config surface, likely under plugins.entries.<provider>.config.webSearch.proxy or an approved equivalent request-proxy shape, and route it through a shared guarded explicit-proxy helper that preserves SSRF checks. Update plugin manifests, config docs, provider tests, and coordinate the private-network policy with #74357 before landing.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict core/plugin schemas do not accept proxy fields, and Brave/Perplexity web-search runtimes never resolve or pass a provider-specific proxy URL.

## Solution Assessment

Unclear pending maintainer/security design. The direction is valid, but the safest maintainable solution is provider-owned config plus a guarded explicit-proxy web-search helper, not a local patch that bypasses SSRF protection or a Brave-only baseUrl workaround.
