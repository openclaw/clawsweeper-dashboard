## Summary

Keep open. Current main has global/env proxy routing and Brave/Perplexity baseUrl workarounds, but it still lacks provider-scoped forward-proxy config for web_search providers; because this changes SSRF-protected egress, it needs maintainer/security design before implementation.

## What This Changes

Add provider-scoped proxy configuration for web_search providers so a single provider such as Brave or Perplexity can use a dedicated HTTP proxy without routing all web tools through it.

## Best Possible Solution

Add a maintainer-approved provider-owned web-search proxy transport that threads a guarded explicit-proxy dispatcher through the shared web-search guard, updates plugin manifests/docs/generated config metadata, and keeps public API targets under strict SSRF policy.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict Brave/Perplexity schemas reject webSearch.proxy, no BRAVE_PROXY_URL or PERPLEXITY_PROXY_URL path exists, and the shared web-search fetch wrapper has no explicit proxy parameter.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the narrow maintainable solution is a generic guarded provider-proxy seam rather than a Brave-only dispatcher patch or direct reuse of Telegram's channel transport.
