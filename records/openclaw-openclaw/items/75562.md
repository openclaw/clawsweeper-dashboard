## Summary

Keep open. Current main has global env/managed proxy routing for web tools, but it still has no provider-scoped web-search proxy config or runtime path. The request is valid, but it is security-sensitive because the implementation must preserve SSRF protections and coordinate with the open SSRF policy split in #74357.

## What This Changes

Request provider-scoped proxy configuration for web search providers so one provider, such as Brave or Perplexity, can route API calls through a proxy without routing all web tools through that proxy.

## Best Possible Solution

Add an approved provider-owned web-search proxy config surface, likely under plugins.entries.<provider>.config.webSearch.* with a request/proxy shape consistent with other provider transports, and route it through a shared guarded explicit-proxy helper that validates both the configured proxy and final provider API target. Update plugin manifests, generated config/docs, provider tests, and coordinate private-network policy with #74357 before landing.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas reject provider proxy fields, and Brave/Perplexity runtimes never resolve or pass a provider-specific proxy URL.

## Solution Assessment

Unclear pending maintainer/security design. The requested direction is valid, but the best implementation should be a guarded provider-owned proxy path, not a Brave-only local patch or a raw dispatcher path that bypasses SSRF checks.

## Risks / Open Questions

- Provider-scoped proxy config shape is not settled: the implementation could use a simple webSearch.proxy field, a request.proxy shape aligned with other provider transports, or another owner-approved schema.
- The design must coordinate with #74357 so adding explicit provider proxies does not keep broad private-network access for fixed public API providers.
- A local patch that uses raw fetchWithRuntimeDispatcher or makeProxyFetch could route traffic correctly while bypassing hostname policy, redirect handling, or proxy-host validation.
