## Summary

Keep open. Current main has global managed/env proxy routing and a Telegram account-level proxy pattern, but bundled web_search provider config and runtimes still do not expose or resolve a provider-scoped proxy URL; this also needs maintainer/security design because it changes web-search egress and SSRF behavior.

## What This Changes

Request a per-provider proxy configuration for web search providers, especially Brave and Perplexity, so one provider can use an HTTP proxy without routing all OpenClaw web traffic through it.

## Best Possible Solution

Add an approved provider-owned web-search proxy config surface, route provider proxy URLs through a guarded explicit-proxy web-search helper that preserves SSRF checks, and update provider manifests, docs, config tests, and Brave/Perplexity provider tests while coordinating the private-network policy with #74357.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas omit proxy fields, and Brave/Perplexity runtimes never resolve or pass a provider-specific proxy URL.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is valid, but the best implementation should be a shared guarded provider-proxy path rather than a local Brave-only patch or raw fetchWithRuntimeDispatcher bypass.
