## Summary

Keep open. Current main still only has global/env proxy routing for web-search traffic; Brave and Perplexity do not accept or resolve provider-scoped proxy config, and adding that surface should be treated as security-sensitive because it intersects with SSRF-protected egress and the open #74357 policy split.

## What This Changes

The issue requests provider-scoped web-search proxy configuration so a single provider such as Brave or Perplexity can route API calls through a local HTTP proxy without enabling a global proxy for all web tools.

## Best Possible Solution

Add an approved provider-owned web-search request/proxy config surface, route it through the existing guarded explicit-proxy machinery, and update the affected provider manifests, schema/help metadata, docs, SecretRef/redaction coverage, and provider tests. The implementation should be generic enough for Brave and Perplexity and should preserve stricter SSRF policy for fixed public provider endpoints while coordinating with #74357.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas reject the requested provider proxy fields, the Brave and Perplexity runtimes never read BRAVE_PROXY_URL or PERPLEXITY_PROXY_URL, and the shared web-search helper exposes no explicit-proxy parameter.

## Solution Assessment

Unclear pending maintainer/security design. The requested direction is reasonable, but the maintainable fix should use a guarded provider-owned proxy path rather than a Brave-only raw dispatcher or makeProxyFetch patch that bypasses SSRF checks.
