## Summary

Keep open. Current main still has global/env-managed proxy routing and Brave baseUrl overrides, but it does not implement provider-scoped forward-proxy config or BRAVE_PROXY_URL/PERPLEXITY_PROXY_URL for web-search providers; the remaining design is security-sensitive because it changes SSRF-guarded egress.

## What This Changes

Request to add per-web-search-provider proxy config and env overrides so providers like Brave or Perplexity can use a forward proxy without applying global proxy routing.

## Best Possible Solution

Add an approved web-search transport seam that reuses guarded explicit-proxy routing, preserves strict public-provider target policy, and updates provider manifests, docs, generated config metadata, and Brave/Perplexity tests together.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction; this is a feature request. The missing capability is high-confidence by source inspection: strict schemas reject provider webSearch.proxy, no provider-specific proxy env path was found, and the shared web-search helper has no explicit proxy parameter.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the safest implementation is a generic guarded provider-proxy helper rather than a Brave-only raw dispatcher patch or direct reuse of Telegram's channel-specific makeProxyFetch path.
