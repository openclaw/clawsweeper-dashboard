## Summary

Keep open. Current main has global/env-managed proxy routing and a Brave `baseUrl` workaround, but it still lacks provider-scoped forward-proxy configuration for web-search providers, and the remaining work needs maintainer/security design because it changes SSRF-protected egress.

## What This Changes

Add provider-scoped web-search proxy configuration so individual providers such as Brave or Perplexity can route API calls through a dedicated forward proxy.

## Best Possible Solution

Add an approved plugin-owned web-search proxy transport surface that reuses a shared guarded explicit-proxy path, keeps strict public-provider target policy intact, and updates manifests, docs, generated config metadata, and provider tests together.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict plugin schemas reject `webSearch.proxy`, no `BRAVE_PROXY_URL`/`PERPLEXITY_PROXY_URL` path exists, and the shared web-search fetch wrapper has no explicit proxy parameter.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the safest implementation is a generic guarded provider-proxy helper rather than a Brave-only raw dispatcher patch or direct reuse of Telegram's channel-specific transport.
