## Summary

Keep open. Current main has global/env-managed proxy routing and Brave/baseUrl-style endpoint overrides, but it still lacks provider-scoped forward-proxy config for web-search providers, and the remaining design is security-sensitive because it changes SSRF-guarded egress.

## What This Changes

Request per-provider proxy config and env vars for web-search providers so one provider API can use a forward proxy without routing all web tools or runtime traffic.

## Best Possible Solution

Add an approved plugin-owned web-search proxy transport seam that reuses guarded explicit-proxy routing, preserves strict public-provider target policy, and updates manifests, docs, generated config metadata, and provider tests together.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas reject provider webSearch.proxy, no BRAVE_PROXY_URL/PERPLEXITY_PROXY_URL path was found, and the shared web-search helper has no explicit proxy parameter.

## Solution Assessment

Unclear until maintainer/security design is approved. The requested direction is reasonable, but the safest implementation is a generic guarded provider-proxy helper rather than a Brave-only raw dispatcher patch or direct reuse of Telegram's channel-specific transport.
