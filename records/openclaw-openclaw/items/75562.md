## Summary

Keep open. Current main still only supports global/env managed proxy routing for web-search calls; there is no provider-scoped Brave or Perplexity proxy config, and adding one needs maintainer/security design because it touches SSRF-protected egress.

## What This Changes

Request provider-specific proxy configuration and environment variables for web_search providers such as Brave and Perplexity, modeled after Telegram's channel proxy setting.

## Best Possible Solution

Add an approved provider-owned web-search proxy/request config surface and route it through a shared guarded explicit-proxy helper that validates the configured proxy and the final provider API target. Update plugin manifests, config docs, generated surfaces, and provider tests, and coordinate the final policy with #74357 before landing.

## Reproduction Assessment

Not applicable as a classic runtime bug reproduction. The missing capability is high-confidence by source inspection: strict schemas reject provider proxy fields, and Brave/Perplexity runtimes never resolve or pass provider-specific proxy URLs.

## Solution Assessment

Unclear pending maintainer/security design. The requested direction is valid, but the best implementation should be a guarded provider-owned proxy path rather than a Brave-only raw dispatcher patch that bypasses SSRF checks.
