## Summary

Keep open: current main and the latest release still only provide separate managed `openclaw` profiles and Chrome MCP attach profiles, with no explicit import or clone flow. The request is useful, but copying signed-in browser state into an agent-controlled profile needs maintainer product and security approval before implementation.

## What This Changes

The issue requests an explicit one-shot CLI/API flow to copy an existing-session browser profile into a managed `openclaw` browser profile.

## Best Possible Solution

After approval, implement this in the bundled browser plugin as an explicit consent-gated one-time import that leaves the source untouched, resets the destination managed profile, strips transient browser files, relaunches normally, and documents that it is not live sync.

## Reproduction Assessment

Not applicable as a broken-behavior repro; this is a feature request for a new explicit import path. Source, docs, and latest-release inspection confirm the requested command/API does not exist today.

## Solution Assessment

Unclear until maintainer product and security review: an explicit one-shot import is safer than automatic copying, but maintainers still need to approve whether full-profile cloning is acceptable and what consent boundary it requires.
