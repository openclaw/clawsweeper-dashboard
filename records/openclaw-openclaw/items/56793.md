## Summary

Keep open: current main and the latest release still keep managed `openclaw` profiles separate from `existing-session` attach profiles, while importing signed-in browser state into managed storage needs a maintainer product and security decision.

## What This Changes

The issue requests an explicit one-shot command or API to import an existing browser session profile into a managed OpenClaw browser profile.

## Best Possible Solution

If maintainers accept the boundary, implement the flow inside the bundled browser plugin as an explicit one-shot import with destination stop/reset, source untouched, transient-file stripping, focused tests, and security-forward docs.

## Reproduction Assessment

Yes, at source level: current main has separate managed `openclaw` and `existing-session` paths plus profile create/delete/reset surfaces, but no import command or route. This is a missing feature surface, not a broken existing behavior reproduction.

## Solution Assessment

Unclear until maintainers decide the product and security boundary. The proposed explicit non-automatic import is the right shape if accepted, but it should not be queued as an autonomous fix before that decision.
