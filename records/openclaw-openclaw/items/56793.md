## Summary

Keep open. Current main and the latest shipped tag still keep managed browser profiles and existing-session attach profiles as separate paths, with no explicit one-shot import surface; the request is a plausible feature but needs maintainer product and security decisions because it would intentionally copy sensitive browser session state into an automation-managed profile.

## What This Changes

Request an explicit browser profile import command or API that copies an existing same-user browser profile into a managed OpenClaw browser profile without automatic startup copying.

## Best Possible Solution

If maintainers accept the product/security direction, implement this inside the browser plugin as an explicit opt-in import with source and destination validation, destination stop/reset handling, safe copy exclusions, managed-profile restoration, focused tests, and user-facing docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The source, docs, test-surface, release-tag, and GitHub-search checks show current OpenClaw supports managed profile lifecycle and existing-session attach paths, but not profile import or clone.

## Solution Assessment

Unclear. An explicit one-shot import is safer than automatic copying, but the maintainable solution depends on maintainer decisions about consent, exclusions, source/destination rules, and whether the public surface is CLI, API, or both.
