## Summary

Keep this open: current main has managed browser profiles and existing-session attach profiles, but no explicit one-shot import/clone path from a user browser profile into a managed OpenClaw profile. The request is plausible but security-sensitive because it would copy cookies, logged-in sessions, and other browser state into an agent-controlled profile, so it needs maintainer product/security review before automation.

## What This Changes

The issue requests an explicit one-shot browser profile import flow from an existing-session user profile into a managed OpenClaw profile.

## Best Possible Solution

Add this only after maintainer approval as an explicit browser-plugin command/API that stops or resets the destination, resolves the source profile intentionally, copies only safe same-user local data, strips transient Chrome runtime artifacts, relaunches as a normal managed profile, and documents that it is a one-time import rather than sync.

Not applicable: this is a feature request for a new explicit import path, not a report of broken existing behavior. Current-main source inspection confirms the requested command/API does not exist.

## Solution Assessment

Yes as a direction, but not as an automated fix yet: an explicit one-shot import is safer than automatic copying, while the exact source/destination lifecycle and security consent need maintainer approval.
