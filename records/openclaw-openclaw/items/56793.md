## Summary

Keep this open: current main and v2026.6.8 still separate managed `openclaw` profiles from Chrome MCP `existing-session` attach profiles, with no CLI/API path to import or clone one into the other. The request is useful but security-sensitive because it would copy logged-in browser state into an agent-controlled profile, so it needs maintainer product and security approval before implementation.

## What This Changes

Add an explicit one-shot import/clone flow from a user's existing browser session profile into the managed OpenClaw browser profile.

## Best Possible Solution

After maintainer approval, implement this in the bundled browser plugin as an explicit consent-gated one-time import that leaves the source untouched, resets the destination managed profile, strips transient browser files, relaunches normally, and documents that it is not live sync.

## Reproduction Assessment

Not applicable as a broken-behavior repro; this is a feature request for a new explicit import path. Source and docs inspection of current main and v2026.6.8 confirm the requested command/API does not exist.

## Solution Assessment

Unclear until maintainer product and security review: an explicit one-shot import is safer than automatic copying, but maintainers still need to decide whether full-profile cloning is acceptable and what consent boundary it requires.
