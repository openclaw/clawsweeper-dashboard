## Summary

Keep open: current main still separates managed `openclaw` browser profiles from `existing-session` attach profiles, and there is no explicit import or clone flow. The request is plausible, but copying signed-in browser state into managed agent storage needs maintainer product and security review first.

## What This Changes

Add an explicit one-shot CLI/API flow to import an existing-session browser profile into the managed `openclaw` browser profile without automatic startup copying.

## Best Possible Solution

If maintainers accept the boundary, add an explicit one-shot import that stops/resets only the managed destination, copies a narrowly defined safe subset, strips transient files, leaves the source untouched, and documents the security tradeoff.

## Reproduction Assessment

Yes at source level: current main has separate managed `openclaw` and `existing-session` profile paths plus create/delete/reset surfaces, but no import route or CLI command. This is not a runtime bug reproduction; it is a missing feature surface confirmed by code and docs.

## Solution Assessment

Unclear until maintainers decide the product and security boundary. The proposed explicit, non-automatic import is the right shape if accepted, but it should not be added silently or by an autonomous fix lane.

## Risks / Open Questions

- Copying a live signed-in browser profile can move cookies, account sessions, local storage, password-manager state, and sync-related data into agent-controlled managed storage.
- A naive copy can race browser locks or transient runtime files, producing a corrupt or misleading managed profile.
- Without explicit product and security rules, this could weaken the documented guarantee that the managed `openclaw` profile does not touch the personal browser profile.
