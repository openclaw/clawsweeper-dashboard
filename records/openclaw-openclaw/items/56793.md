## Summary

Keep open: current main and the latest release still expose separate managed and existing-session browser profile paths, but no explicit import/clone flow; the remaining blocker is a maintainer product and security decision about copying signed-in browser state into managed storage.

## What This Changes

The issue requests an explicit one-shot import flow that copies a user’s existing browser profile into an OpenClaw-managed browser profile without live sync.

## Best Possible Solution

If maintainers accept the boundary, implement the flow inside the bundled browser plugin as an explicit one-shot import with destination stop/reset, source untouched, transient-file stripping, focused tests, and security-forward docs.

## Reproduction Assessment

Yes, at source level: current main has separate managed openclaw and existing-session paths plus profile create/delete/reset surfaces, but no import command or route. This is a missing feature surface, not a broken existing behavior reproduction.

## Solution Assessment

Unclear until maintainers decide the product and security boundary. The proposed explicit non-automatic import is the right shape if accepted, but it should not be queued as an autonomous bug fix before that decision.
