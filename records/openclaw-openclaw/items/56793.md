## Summary

Keep open. Current main still lacks an explicit one-shot import/clone flow from an existing-session browser profile into a managed OpenClaw browser profile, and the requested state transfer needs maintainer product/security policy before implementation.

## What This Changes

The issue requests an explicit CLI or API flow to copy an existing-session browser profile into a managed OpenClaw browser profile without automatic startup sync.

## Best Possible Solution

If maintainers accept the feature, implement it in the bundled browser plugin as an explicit opt-in mutation with source/destination validation, destination stop/reset, documented exclusions, managed-profile redecorating, route protection, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The high-confidence check is source and docs inspection: current main implements managed profiles and existing-session attach profiles, but no supported import or clone path exists.

## Solution Assessment

Unclear. The proposed one-shot explicit import is safer than automatic copying, but the narrowest maintainable implementation depends on maintainer decisions about consent, copied state, exclusions, and API versus CLI surface.
