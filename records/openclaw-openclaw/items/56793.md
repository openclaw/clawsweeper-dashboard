## Summary

Keep open. Current main still lacks an explicit one-shot browser profile import/clone flow, and the request is a new security-sensitive browser feature that needs maintainer product policy before implementation.

## What This Changes

Add an explicit opt-in import flow that seeds a managed `openclaw` browser profile from an existing-session/user browser profile.

## Best Possible Solution

If maintainers accept the feature, implement it in the browser plugin as an explicit opt-in import with source/destination validation, destination stop/reset, documented exclusions, managed-profile restoration, focused tests, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence check is source and docs inspection showing create/delete/reset and existing-session attach paths exist, but no import/clone surface exists.

## Solution Assessment

Unclear. An explicit one-shot import is safer than automatic copying, but the maintainable implementation depends on maintainer decisions about consent, exclusions, source/destination rules, and CLI versus API surface.
