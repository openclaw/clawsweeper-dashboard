## Summary

Keep open. Current main still has no explicit one-shot import or clone flow from an existing-session Chrome MCP profile into a managed OpenClaw browser profile, and the request is product/security-sensitive because it would copy browser state that may include logged-in sessions into an agent-controlled profile.

## What This Changes

Add an explicit opt-in CLI or server API flow that seeds a managed OpenClaw browser profile from a same-user existing browser profile without touching the source profile.

## Best Possible Solution

If maintainers accept the direction, implement the flow in the bundled browser plugin as an explicit opt-in mutation: validate the source and destination profile types, stop or reset only the destination managed profile, copy with a documented exclusion list for transient and high-risk artifacts, redecorate the destination as a managed OpenClaw profile, protect the route like other persistent browser mutations, and update CLI/API docs plus focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The high-confidence check is source/docs inspection: current main documents and implements managed profiles plus existing-session attach profiles, but no import or clone path exists.

## Solution Assessment

Unclear. The proposed explicit one-shot import is safer than automatic startup copying, but the best implementation depends on maintainer decisions about consent, copied state, exclusions, and whether this belongs in CLI, server API, or both.
