## Summary

Keep open. Current main and the latest release still keep managed `openclaw` browser profiles and `existing-session` Chrome MCP attach profiles as separate paths with no explicit browser-profile import/clone surface; the request is a plausible feature but needs product and security review before implementation.

## What This Changes

Add an explicit one-shot flow to import a user's existing browser session/profile data into a managed OpenClaw browser profile.

## Best Possible Solution

If maintainers accept the direction, implement it in the browser plugin as an explicit opt-in import with strict source/destination validation, safe copy exclusions, destination reset handling, managed-profile restoration, focused tests, and public docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Current source, docs, tests, release-tag inspection, and related PR search show existing managed and existing-session browser profiles are supported, but import/clone is not.

## Solution Assessment

Unclear. An explicit one-shot import is safer than automatic copying, but the maintainable design depends on maintainer decisions about consent, exclusions, source/destination rules, and CLI/API shape.
