## Summary

Keep open: current main and v2026.6.10 still expose separate managed and existing-session browser profile paths, but no explicit one-shot import/clone flow; the remaining decision is a product and security boundary for copying signed-in browser state into managed storage.

## What This Changes

The issue requests an explicit CLI or API flow to copy an existing-session browser profile into a managed OpenClaw browser profile without automatic syncing.

## Best Possible Solution

If maintainers accept the boundary, implement this in the bundled browser plugin as an explicit one-shot import with destination stop/reset, source untouched, transient-file stripping, focused tests, and security-forward docs.

## Reproduction Assessment

Yes, at source level: current main has managed openclaw and existing-session attach paths plus create/delete/reset surfaces, but no import command or route. This is a missing feature surface rather than a broken existing behavior reproduction.

## Solution Assessment
