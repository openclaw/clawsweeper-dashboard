## Summary

This maintainer-labeled security-hardening PR still addresses a real gap in current main, but the current patch has blocking compatibility issues because it treats every disabled workspace plugin as untrusted and there is active related follow-up discussion for setup-entry error isolation.

## What This Changes

The PR adds a loader predicate and registry backstop to reject disabled workspace-origin channel setup registrations, plus loader and registry regression tests.

## Best Possible Solution

Land a narrow workspace trust guard that blocks untrusted workspace shadows before import and at registry registration while preserving trusted-disabled workspace setup-only metadata, then intentionally include or defer the related setup-entry try/catch resilience fix.

## Reproduction Assessment

Yes, from source inspection: current main can return a setup-only registration plan for an explicitly scoped disabled channel plugin before the normal disabled-plugin skip path, and the registry lacks a disabled-workspace backstop. I did not execute tests because this review is read-only.

## Solution Assessment

No, not in its current form: the loader and registry are the right boundary, but the current predicate should use the untrusted workspace disable reason instead of treating every disabled workspace plugin as untrusted.
