## Summary

Keep open. Current main and the latest release still omit the requested WhatsApp participant and quoted-reply fields from the gateway inbound log payload, even though the same function already computes and queues those fields for agent delivery; the only directly linked implementation PR was closed unmerged.

## What This Changes

Add already-computed WhatsApp participant identity and quoted-reply metadata to the gateway inbound log payload for inbound WhatsApp messages.

## Best Possible Solution

Add the scalar sender and quoted-reply fields that are already computed in `enqueueInboundMessage()` to the existing WhatsApp `inboundLogger.info()` payload, keep message delivery and mention gating unchanged, and cover the logger payload with a focused monitor-inbox regression test.

## Reproduction Assessment

Yes. A focused unit test can send a WhatsApp group quoted reply through the existing monitor-inbox harness and assert the logger payload includes `senderJid`, `senderE164`, `senderName`, and `replyTo*`; current main fails by source inspection because those keys are absent from the log object.

## Solution Assessment

Yes. The narrowest maintainable fix is to enrich the existing log object from `inbound.participantJid`, `inbound.senderE164`, `msg.pushName`, and `enriched.replyContext`, without changing access control, group mention gating, delivery behavior, or raw message logging.
