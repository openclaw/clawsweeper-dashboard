## Summary

Keep open. Current main still matches the reported failure path, and the linked implementation PR is open and unmerged, so this issue should remain open until that PR or a narrow successor lands.

## What This Changes

The issue reports that exec-spawned OpenClaw CLI commands inherit Gateway service environment markers and then fail or hang when they connect back to the Gateway over WebSocket.

## Best Possible Solution

Land #62204 or a narrow successor that strips Gateway service runtime markers at the host exec environment boundary, preserves OPENCLAW_SHELL=exec, and adds regression coverage for both credential precedence and clean child-process exit.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: Gateway service startup injects the reported markers, exec inherits process.env, the current sanitizer does not strip those markers, and Gateway credential resolution branches on OPENCLAW_SERVICE_KIND=gateway.

## Solution Assessment

Yes. Sanitizing internal Gateway service markers before host exec commands run is the narrowest maintainable fix; changing all CLI credential precedence would be broader and could alter legitimate Gateway-service behavior.
