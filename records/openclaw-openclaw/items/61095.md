## Summary

Keep open. Current main and the latest release still show the reported source path: Gateway service markers are exported into the process env, host exec inherits that env, the sanitizer does not strip the reported markers, and credential resolution still changes behavior when `OPENCLAW_SERVICE_KIND=gateway`; the open closing PR remains the implementation path but is unmerged and conflicting.

## What This Changes

The issue reports that agent exec-spawned OpenClaw CLI commands inherit Gateway service environment markers and then fail or hang during local gateway RPC.

## Best Possible Solution

Review, revise, and land https://github.com/openclaw/openclaw/pull/62204 or a narrow successor that strips exact internal runtime markers at the host exec boundary, preserves documented service controls and `OPENCLAW_SHELL=exec`, and proves the CLI command exits cleanly.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Gateway service env sets the markers, host exec inherits `process.env`, sanitizer policy leaves them intact, and credential resolution changes behavior when `OPENCLAW_SERVICE_KIND=gateway`.

## Solution Assessment

Yes, the host exec environment boundary is the right place to solve the issue. The current linked PR's broad prefix filter is not the best exact implementation because it also strips documented service controls.
