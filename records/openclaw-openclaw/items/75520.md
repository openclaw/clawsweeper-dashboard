## Summary

Keep open. The issue is paired with open PR #75521, which explicitly closes it, and current main still does not show the scoped-active to unscoped-request registry reuse described by the report. Main does include a hot-path mitigation that disables bundled runtime-dependency repair during fallback loads, so the exact latency should be re-profiled before final severity judgment.

## What This Changes

The issue asks OpenClaw to reuse the already-active scoped gateway plugin registry for ordinary unscoped embedded tool resolution instead of reloading plugins on every embedded message.

## Best Possible Solution

Review PR #75521 against current main and either land a narrow scoped-active to unscoped-request registry reuse with regression coverage, or close it with live/profile evidence that the current hot-path mitigation fully removes the user-visible reload cost while preserving loader compatibility.

## Reproduction Assessment

Yes. The source-level path is high confidence: gateway startup loads a scoped gateway-bindable registry, normal embedded commands do not pass gateway-bindable mode, and the active-registry compatibility check still does not relax scoped-active versus unscoped-request keys. The exact 20-40s latency needs fresh profiling on current main.

## Solution Assessment

Unclear until PR #75521 is reviewed. A loader-level compatibility path can be the right fix if it preserves workspace, config, activation metadata, runtime mode, and plugin-scope correctness; the current main mitigation alone does not prove that end state.
