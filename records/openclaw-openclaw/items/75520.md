## Summary

Keep open. Current main still has the reported cache-shape problem: regular embedded tool resolution does not prefer the already-active registry, while gateway startup activates a scoped gateway-bindable registry and the loader cache key includes plugin-id scope, so an unscoped tool request can fall through to a fresh plugin load and bundled runtime-dependency staging.

## What This Changes

The issue requests preventing embedded runs from rebuilding the plugin registry and re-staging bundled runtime dependencies during `core-plugin-tools` by safely reusing the active plugin registry for tool resolution.

## Best Possible Solution

Keep the issue open and land a narrow plugin-tool registry fix that avoids per-run unscoped reloads without losing tool registrations from plugins outside the startup scope; add focused coverage for scoped gateway startup plus regular embedded tool resolution, and coordinate with #75290 for any remaining factory-cost work.

## Reproduction Assessment

Yes for triage. The report includes concrete prep-stage and staging logs, and current source shows the scoped startup registry, unscoped tool request, scope-sensitive cache key, and fresh-load fallback needed to reproduce the behavior with a configured bundled plugin set.

## Solution Assessment

No for the literal proposed "prefer active registry in all cases" wording. The maintainable fix should reuse the active registry only when it preserves the expected plugin-tool surface, or otherwise ensure a compatible tool registry is loaded once and then reused across embedded runs.
