## Summary

Keep open. Current main still has the reported registry-reuse gap for embedded command runs that do not pass gateway-bindable mode into tool construction: the Gateway startup registry is scoped and active, but plugin tool resolution can fall through to an unscoped runtime load and restage bundled runtime dependencies.

## What This Changes

Reduce embedded-run prep latency by reusing the already-active plugin registry for plugin tool resolution instead of rebuilding and restaging bundled runtime dependencies per run.

## Best Possible Solution

Fix the embedded command/tool-resolution path so Gateway-hosted embedded runs reuse the active gateway-bindable plugin registry when that is the correct runtime surface, while preserving loader compatibility checks for explicit non-Gateway, snapshot, workspace, and config-different loads.

## Reproduction Assessment

Yes. The source-level path is high confidence: Gateway startup creates a scoped gateway-bindable active registry, command embedded runs omit the gateway-bindable flag, and plugin tool resolution then uses load options whose unscoped cache key cannot match the scoped startup registry.

## Solution Assessment

Unclear for the exact proposed fix. Reusing the active registry is the right direction for Gateway-hosted embedded tool resolution, but the safer implementation is a narrow gateway-bindable reuse or flag propagation rather than bypassing compatibility for every plugin tool call.
