## Summary

Keep open. Current main still has the scoped-active versus unscoped-request registry compatibility gap: Gateway startup activates a scoped gateway-bindable registry, while embedded plugin tool resolution can request an unscoped runtime registry whose cache key cannot match. The linked open PR #75521 is the right implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Avoid reloading the plugin registry for embedded message tool resolution when the active Gateway startup registry was loaded with a scoped plugin-id set.

## Best Possible Solution

Review and land PR #75521, or an equivalent narrow loader change, so an active scoped Gateway registry can satisfy an unscoped embedded tool request only when workspace, config, activation metadata, runtime mode, and the active loaded plugin-id set prove compatibility. Preserve explicit empty scopes, snapshot loads, and config/workspace-different loads as non-compatible.

## Reproduction Assessment

Yes. A focused source-level reproduction is to set an active gateway-bindable registry with a cache key built from `onlyPluginIds: ["demo"]`, then call `getCompatibleActivePluginRegistry` or `resolveRuntimePluginRegistry` with the same config/workspace but no `onlyPluginIds`; current main cannot match those cache keys and falls through to a fresh load.

## Solution Assessment

Yes for the proposed direction. The maintainable fix belongs in the loader compatibility seam, not a broad tool-level active-registry bypass, as long as it only relaxes the scoped-active to unscoped-request case when the active registry contents prove the same effective plugin surface.
