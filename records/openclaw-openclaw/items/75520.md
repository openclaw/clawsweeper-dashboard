## Summary

Keep open. The report is still paired with open PR #75521, and current main has partial hot-path mitigations but still does not show the scoped-active to unscoped-request registry reuse described here. The newer web-provider cache trace from jasonftl is also relevant and should be reviewed before deciding that the linked PR fully closes the latency regression.

## What This Changes

The issue reports a per-turn plugin registry reload/cache miss caused by gateway startup loading a scoped active registry while embedded tool and web-provider resolution later request unscoped or uncached plugin runtime data.

## Best Possible Solution

Review PR #75521 against current main and either land a narrow compatibility-preserving registry reuse, or replace it with an equivalent fix that makes embedded tool and web-provider runtime resolution reuse the already-compatible active registry without weakening workspace, config, activation, runtime-mode, or plugin-scope boundaries.

## Reproduction Assessment

Yes for the source-level cache-miss path: current main still loads a scoped gateway-startup registry and ordinary embedded tool resolution can request a different unscoped load context. The exact wall-clock latency needs a live multi-plugin gateway profile because partial mitigations have landed.

## Solution Assessment

Unclear until PR #75521 is reviewed with the newer web-provider cache trace. A narrow loader compatibility path or equivalent scope/cache propagation looks like the maintainable direction, but it should include regression coverage for both plugin tools and web-provider resolution.
