## Summary

Keep open. The issue is paired with unmerged PR #75521, and current main still does not implement the specific scoped-active registry to unscoped tool-request reuse described here; it only contains partial hot-path mitigations for runtime-deps repair and tool discovery cost.

## What This Changes

The issue reports that embedded message turns repeatedly reload plugin registries because gateway startup uses a scoped gateway-bindable plugin registry while regular plugin-tool resolution requests an unscoped/default snapshot.

## Best Possible Solution

Review PR #75521 against current main and either land a narrow compatibility-preserving registry reuse with regression coverage, or replace it with an equally narrow fix that covers both the plugin-tool registry miss and the web-provider cache-default path without weakening loader scope, activation, workspace, or runtime-mode contracts.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: gateway startup loads only the startup plugin ids in gateway-bindable mode, ordinary embedded command runs do not pass gateway binding, and current loader compatibility still treats scoped and unscoped cache keys as different.

## Solution Assessment

Unclear. Current main's partial mitigations reduce the worst runtime-deps/tool-discovery cost but do not prove the requested scoped-active to unscoped-request reuse; the proposed PR needs normal review, rebase/mergeability handling, and coverage for the exact compatibility contract.
