## Summary

Keep open. Current main still lacks the scoped-startup-registry to unscoped-embedded-tool reuse described by the report, and the issue is already paired with open PR #75521 using closing syntax. Main has a partial hot-path mitigation that skips bundled runtime-dependency repair during plugin-tool fallback loads, but that does not prove the registry reload/factory-cost problem is fully fixed.

## What This Changes

Reuse the already-active scoped gateway plugin registry for ordinary unscoped embedded message plugin-tool resolution to avoid repeated plugin reload and runtime dependency staging.

## Best Possible Solution

Review PR #75521 against current main and land a narrow compatibility-preserving registry reuse, or reject it with source and profiling evidence that the current hot-path mitigation fully solves the user-visible latency. The end state should preserve workspace, config, activation metadata, runtime mode, and plugin-scope compatibility, with regression coverage; any remaining per-run factory cost can stay tracked by #75290 or #75512.

Yes. Unclear whether the open PR is the best exact patch until it is reviewed on top of current main, but the requested direction is plausible and narrow.
