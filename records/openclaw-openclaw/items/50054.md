## Summary

Keep open: current main does not implement this Redis ACP locking feature, and the PR still has blocking correctness issues. The biggest problem is that the new default local lock path skips a contended ACP prompt even though current main already queues same-session turns and executes both.

## What This Changes

The PR adds Redis-backed ACP session locking, a local fallback lock manager, fail-closed Redis behavior, dispatch wiring, env documentation, and lock-manager tests.

## Best Possible Solution

Move the lock to a shared ACP turn boundary that preserves queue semantics, covers steer turns, and treats Redis ownership loss during renewal as a hard failure or cancellation instead of silent best-effort logging.

## Reproduction Assessment

Yes, source-reproducible. Current main has a manager-level queue contract that executes both same-session turns, while the PR’s default lock contention branch skips the second turn before it reaches that queue.

## Solution Assessment

No. The current PR is not the best solution because it wraps only one ACP entrypoint and changes queued same-session behavior into dropped work; the safer direction is a shared manager-level or turn-boundary lock that preserves existing semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve queued ACP turns on lock contention:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: Current `AcpSessionManager` queues same-session `runTurn` calls and executes both. This branch returns `acp_execution_locked` when the default local lock is contended, so a second prompt arriving while the first is running is marked skipped and lost in the default single-process deployment.
  - confidence: 0.93
- **[P2] Fail the turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:265-283`
  - body: In Redis mode, a failed renewal means the lock can expire while this turn continues. Another worker can then acquire the same session and start a concurrent turn, so renewal loss needs to cancel/fail the turn or otherwise preserve ownership.
  - confidence: 0.86
- **[P2] Apply the lock to steer turns too:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only in `tryDispatchAcpReply`, but `/acp steer` still calls `AcpSessionManager.runTurn()` directly. With Redis enabled, a steer command can overlap a normal ACP prompt across workers on the same session.
  - confidence: 0.84
- **[P2] Close Redis sockets when setup commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends AUTH/SELECT before returning the connection, while the caller only closes after `connectRedis()` resolves. Wrong credentials, an invalid DB, or a setup-command timeout rejects before ownership reaches the caller and leaks a socket on each skipped ACP turn.
  - confidence: 0.88
