## Best Possible Solution

Yes for the blocking PR regressions by source inspection: current main queues same-session runTurn calls, while the PR returns acp_execution_locked before that queue, leaves /acp steer outside the lease, and logs renewal loss without stopping the turn. ## Solution Assessment

No. The maintainable fix is a shared ACP turn lease that preserves existing queue semantics and covers prompt plus steer turns, with explicit lease-loss behavior and Redis cleanup/proof; adding a skip-before-queue lock only in dispatch-acp is not the right boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: Current main serializes same-session `runTurn()` calls in `AcpSessionManager`, so a second prompt waits and still runs. This branch checks the local fallback lock before that queue and returns `acp_execution_locked`, which means default single-process users can lose a same-session prompt instead of having it queued behind the active turn.
  - confidence: 0.93
- **[P2] Apply the lock at the shared ACP turn boundary:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lease is acquired only in `tryDispatchAcpReply`, but `/acp steer` calls `acpManager.runTurn()` through a separate production path. In Redis multi-worker mode, a steer command can still overlap a normal prompt for the same ACP session, leaving the core session-state race unsolved.
  - confidence: 0.88
- **[P2] Abort or stop leasing the turn when renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: The renewal loop treats `renew()` returning false or throwing as log-only and lets the turn continue. After the TTL expires, another worker can acquire the Redis lock and start a second same-session turn while the original one is still mutating runtime state.
  - confidence: 0.86
- **[P2] Close Redis sockets when setup commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` awaits AUTH and SELECT before returning the connection to the caller's `finally`. If either setup command rejects, the socket that was just opened is not closed, so repeated misconfigured ACP attempts can leak Redis connections while fail-closed dispatch keeps retrying.
  - confidence: 0.84
