## Summary

Keep open for maintainer consideration, but the patch should not merge as-is: current main does not already contain the Redis ACP lock feature, and the PR still has source-proven correctness and Redis cleanup blockers plus missing real behavior proof.

## What This Changes

This PR adds optional Redis-backed ACP session locking with local/fail-closed managers, env examples, and acquire/renew/release wiring around automatic ACP dispatch.

## Best Possible Solution

Implement distributed lease handling at a shared ACP turn boundary that preserves same-session queueing, covers prompt and steer turns, fails closed when a lease is lost, and guarantees Redis cleanup on every initialization path.

## Reproduction Assessment

Yes, for the blocking PR regressions by source inspection: current main queues same-session runTurn calls through the manager, while PR head returns acp_execution_locked before that queue and leaves /acp steer outside the lease. I did not run PR head or a live Redis-backed ACP setup.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the narrowest maintainable boundary because ACP already has a shared manager queue and steer is another production runTurn entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This branch returns acp_execution_locked when the local/default lock is held. Current AcpSessionManager.runTurn() serializes same-session turns and its regression test expects both concurrent turns to run, so the second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Apply the lease at every ACP turn entrypoint:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only in tryDispatchAcpReply(). /acp steer still calls acpManager.runTurn() directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort active turns after lease renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-284`
  - body: The renewal heartbeat logs false or throwing renewals and keeps the turn running. After the Redis TTL expires, another worker can acquire the same session while this process is still inside runTurn(), defeating the distributed lock.
  - confidence: 0.88
- **[P2] Close Redis sockets when initialization commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: connectRedis() sends AUTH and SELECT before returning, while the caller closes the socket only after connectRedis() resolves. Bad credentials, bad DB selection, or initialization timeout can leave the opened socket alive on each attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds a raw Redis socket path, and initialization failures can leak sockets during repeated ACP attempts.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before AUTH and SELECT, and failed initialization commands reject before the caller's cleanup finally is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
  - confidence: 0.84
