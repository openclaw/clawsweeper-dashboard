## Summary

Keep this PR open. Current main still has no distributed ACP session lock, and the branch is not merge-ready: it drops queued same-session turns in the default local mode, leaves `/acp steer` outside the lease, keeps long turns running after Redis renewal loss, and can leak Redis sockets when AUTH or SELECT fails. The author addressed the early per-command timeout and `REDIS_URL` fallback feedback, but the remaining review issues are substantive.

## What This Changes

The branch adds local, Redis, and fail-closed ACP session lock managers, wraps `tryDispatchAcpReply()` with acquire/renew/release behavior, documents Redis lock environment variables, and adds lock manager plus dispatch tests.

## Best Possible Solution

Revise the implementation around the shared ACP turn boundary so prompt dispatch and `/acp steer` use the same distributed lease, same-session overlap is queued or otherwise preserved instead of skipped, lease loss during long Redis-backed turns has a defined abort/fail-closed outcome, Redis sockets are closed on every initialization failure, and the final contract is covered by regression tests plus env/docs updates.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: current main has an ACP manager test proving same-session turns are queued, while the PR's new contended-lock branch and test explicitly skip the second turn before it reaches `runTurn()`.

## Solution Assessment

No. Wrapping only `tryDispatchAcpReply()` with a non-waiting lock is not the narrowest maintainable fix because ACP turns already have a shared manager boundary and `/acp steer` is a separate production entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This branch returns `acp_execution_locked` as soon as the session lock is held. With the default local backend, a second same-session prompt that current `AcpSessionManager.runTurn()` queues and executes is now dropped before it reaches the manager.
  - confidence: 0.94
- **[P2] Cover steer turns with the same session lease:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only in prompt dispatch, but `/acp steer` still calls `acpManager.runTurn()` through its own command path. In Redis mode, steer commands can overlap a normal prompt for the same session, leaving the cross-worker race this feature is meant to prevent.
  - confidence: 0.9
- **[P2] Fail the active turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:273-283`
  - body: When `renew()` returns false or throws, the heartbeat only logs and lets the turn continue. After the TTL expires, another worker can acquire the same session while the first worker is still running, so long turns lose the distributed-lock guarantee.
  - confidence: 0.88
- **[P2] Close sockets when Redis initialization commands fail:** `src/auto-reply/reply/session-lock-manager.ts:279-286`
  - body: `connectRedis()` sends AUTH and SELECT before returning the connection, but the caller only closes in a `finally` after `connectRedis()` resolves. Wrong credentials or an invalid DB reject these awaits and leak the just-opened socket on every ACP attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, install, or publishing changes were found, but the new raw Redis TCP/TLS path has a concrete connection cleanup issue.

Concerns:

- **[medium] Redis auth failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:279`
  - body: The Redis connection is opened before AUTH/SELECT, and failed initialization commands reject before the caller's cleanup `finally` is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets while ACP fails closed.
  - confidence: 0.84
