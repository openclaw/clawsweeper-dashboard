## Summary

Keep open: current main does not implement distributed ACP locking, but this PR should not merge as-is because the proposed non-waiting lock drops queued same-session prompts, misses `/acp steer`, keeps running after Redis lease loss, and can leak Redis sockets on initialization failure.

## What This Changes

The PR adds local, Redis, and fail-closed ACP session lock managers, wraps automatic ACP prompt dispatch with acquire/renew/release behavior, documents Redis lock env vars, and adds lock-manager plus dispatch tests.

## Best Possible Solution

Move lease handling to a shared ACP turn boundary that preserves existing same-session queueing, covers prompt and steer turns, treats lost Redis leases as abort/fail-closed, and closes sockets on every Redis initialization failure.

## Reproduction Assessment

Yes. The source-level path is high confidence: current main queues same-session ACP turns through AcpSessionManager, while PR head returns acp_execution_locked before the second turn reaches that queue.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the narrowest maintainable solution because ACP already has a shared manager boundary and `/acp steer` remains another production runTurn entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This returns `acp_execution_locked` when the local/default lock is held. Current `AcpSessionManager.runTurn()` serializes same-session turns and its regression test expects both concurrent turns to execute, so the second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Cover steer turns with the session lease:** `src/auto-reply/reply/dispatch-acp.ts:239`
  - body: The new lock is acquired only in `tryDispatchAcpReply()`. `/acp steer` still calls `acpManager.runTurn()` directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort the active turn when renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: The renewal heartbeat logs false or throwing renewals and keeps the turn running. After the Redis TTL expires, another worker can acquire the same session while this process is still inside `runTurn()`, defeating the distributed lock.
  - confidence: 0.88
- **[P2] Close Redis sockets on initialization failure:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends `AUTH` and `SELECT` before returning, while the caller closes the socket only after `connectRedis()` resolves. Bad credentials or DB errors therefore leave the opened socket alive on each attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds a raw Redis TCP/TLS path, and Redis initialization failures can leak sockets while ACP fails closed.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before `AUTH` and `SELECT`, and failed initialization commands reject before the caller's cleanup `finally` is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
  - confidence: 0.84
