## Summary

Keep open. The Redis-backed ACP locking idea is useful, but this branch is not merge-ready because source inspection shows blocking message-delivery/session-state regressions, an availability cleanup bug in the Redis socket path, missing real behavior proof, and a conflicting head.

## What This Changes

The PR adds optional Redis/local ACP session lock managers, ACP lock env knobs, `tryDispatchAcpReply` acquire/renew/release wiring, and focused lock/dispatch tests.

## Best Possible Solution

Move the distributed lease into the shared ACP turn boundary, preserve existing same-process queue semantics, cover prompt and steer turns, close Redis setup sockets on failure, and require redacted Redis-backed runtime proof for normal, contended, renewal-loss, and fail-closed behavior.

## Reproduction Assessment

Yes, for the blocking PR regressions by source inspection: current main queues same-session `runTurn` calls, while the PR returns `acp_execution_locked` before that queue, leaves `/acp steer` outside the lease, and logs renewal loss without stopping the turn. I did not run PR head or a live Redis-backed ACP setup in this read-only review.

## Solution Assessment

No. The PR adds the lease around one auto-reply entrypoint instead of the shared ACP turn boundary, which breaks existing queued prompt behavior and misses command-driven steer turns; the maintainable solution belongs in the common manager path with explicit upgrade/runtime proof.

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: `AcpSessionManager.runTurn()` already serializes same-session turns and still executes both requests, but this branch returns `acp_execution_locked` before entering that queue. In the default no-Redis path, a second same-session ACP prompt is now dropped instead of waiting behind the active turn.
  - confidence: 0.93
- **[P2] Apply the lease at the shared ACP turn boundary:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock only wraps `tryDispatchAcpReply`, while `/acp steer` still calls `acpManager.runTurn()` directly. With Redis enabled, a steer command can overlap a normal prompt on another worker, so the feature does not protect all same-session ACP turns.
  - confidence: 0.89
- **[P2] Stop the turn when lock renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:265-283`
  - body: Renew failures are logged but the active turn keeps running. If the Redis TTL expires after a missed renewal, another worker can acquire the same session while the first worker continues, defeating the distributed lock for long-running turns.
  - confidence: 0.86
- **[P2] Close Redis sockets when setup commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` runs AUTH and SELECT before returning the connection, but the caller closes the socket only after `connectRedis()` resolves. Bad credentials, an invalid DB, or a setup timeout therefore leave the newly opened socket alive on each failed ACP attempt.
  - confidence: 0.84
