## Summary

Keep open: the PR is a plausible ACP distributed-locking feature, but current main does not already implement it and the branch still drops queued turns, misses `/acp steer`, treats lost renewals as log-only, leaks Redis sockets on init failure, and lacks real behavior proof.

## What This Changes

The PR adds local, Redis, and fail-closed ACP session lock managers, wraps automatic ACP prompt dispatch with acquire/renew/release, documents new env vars, and adds tests.

## Best Possible Solution

Move lease handling to a shared ACP turn boundary that preserves same-session queueing, covers prompt and steer turns, aborts or fails closed on lost Redis leases, and closes Redis sockets on every initialization failure.

## Reproduction Assessment

Yes for the blocking PR regressions: source inspection shows current main queues same-session ACP turns through `AcpSessionManager`, while PR head returns `acp_execution_locked` before the second turn reaches that queue. I did not run a live Redis setup in this read-only review.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the best maintainable boundary because ACP already has a shared manager queue and `/acp steer` is another production `runTurn()` entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This returns `acp_execution_locked` when the local/default lock is held. Current `AcpSessionManager.runTurn()` serializes same-session turns and its regression test expects both concurrent turns to execute, so the second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Cover steer turns with the session lease:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only in `tryDispatchAcpReply()`. `/acp steer` still calls `acpManager.runTurn()` directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort active turns when lease renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: The renewal heartbeat logs false or throwing renewals and keeps the turn running. After the Redis TTL expires, another worker can acquire the same session while this process is still inside `runTurn()`, defeating the distributed lock.
  - confidence: 0.88
- **[P2] Close Redis sockets when initialization commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends `AUTH` and `SELECT` before returning, while the caller closes the socket only after `connectRedis()` resolves. Bad credentials or DB errors therefore leave the opened socket alive on each attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds a raw Redis TCP/TLS path and has a concrete socket-leak risk on Redis initialization failure.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before `AUTH` and `SELECT`, and failed initialization commands reject before the caller's cleanup `finally` is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
  - confidence: 0.84
