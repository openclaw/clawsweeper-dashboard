Full review comments:

- **[P1] Preserve queued ACP turns when the lock is held:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: Current main serializes same-session ACP turns inside AcpSessionManager, and its tests assert that concurrent requests both run. This new contended-lock return marks the second message skipped in the default local-lock path, so users can lose prompts that are currently queued and delivered.
  - confidence: 0.93
- **[P2] Lease the shared ACP turn path:** `src/auto-reply/reply/dispatch-acp.ts:239`
  - body: The new lock is acquired only in tryDispatchAcpReply, but /acp steer still calls acpManager.runTurn directly. In Redis mode a steer command can overlap a normal ACP prompt on the same session, leaving the cross-process session race this feature is meant to close.
  - confidence: 0.9
- **[P2] Stop the turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:265-283`
  - body: The heartbeat logs renew() failures and keeps the turn running. If the Redis TTL expires after a missed renewal, another worker can acquire the same session while the original long turn continues, defeating the distributed lock's session-state guarantee.
  - confidence: 0.87
- **[P2] Close Redis sockets when setup commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: connectRedis awaits AUTH and SELECT before returning the connection, while the caller only closes connections after connectRedis resolves. Bad credentials or an invalid DB index can therefore leave an opened TCP socket alive on every failed lock attempt.
  - confidence: 0.88
