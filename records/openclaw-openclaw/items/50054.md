## Summary

This PR is useful in intent but not merge-ready: current main already queues same-session ACP turns, while the PR’s default local lock path skips a contended turn before that queue can serialize it.

## What This Changes

The PR proposes Redis-backed distributed ACP session locking with local and fail-closed lock managers, dispatch acquire/renew/release wiring, env docs, and focused tests.

## Best Possible Solution

Put the lease at a shared ACP turn boundary that preserves queued prompt semantics, covers steer turns, and defines an explicit failure policy for lost Redis renewals before this lands.

## Reproduction Assessment

Yes for the blocking PR regressions by source inspection: current main queues same-session `runTurn` calls, while the PR returns `acp_execution_locked` before that queue can run. I did not run PR head or a live Redis-backed ACP setup.

## Solution Assessment

No; the current patch locks one dispatch entrypoint instead of the shared ACP turn boundary and changes default queued prompt behavior. A better solution should preserve main's queue, cover `/acp steer`, and define renewal-loss handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:239-247`
  - body: Current main serializes same-session `runTurn` calls and still executes both prompts, but this contended-lock branch returns before `runTurn` sees the second message. Because the PR uses `LocalSessionLockManager` by default when Redis is unset, a normal single-process user who sends another ACP prompt during an active turn now loses that prompt instead of having it queued.
  - confidence: 0.91
- **[P2] Apply the lease to `/acp steer` turns:** `src/auto-reply/reply/dispatch-acp.ts:230-236`
  - body: The new lock is only acquired in `tryDispatchAcpReply`, while `/acp steer` still calls `acpManager.runTurn` directly. In Redis mode, a steer command can still overlap with a normal prompt for the same session on another worker, so the distributed lock does not protect the shared session state it is intended to guard.
  - confidence: 0.86
- **[P2] Stop the turn when Redis lock renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:258-275`
  - body: Renewal failures are logged but the active turn continues. If the Redis lease expires during a long turn, another worker can acquire the same session and start a concurrent turn, defeating the distributed lock guarantee on the long-running turns that need it most.
  - confidence: 0.82
