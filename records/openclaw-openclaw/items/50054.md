## Summary

Keep this PR open but do not merge it as-is. Current main still lacks the proposed distributed ACP session lock surface, but the branch regresses same-session queuing, misses the `/acp steer` entrypoint, loses its Redis lease guarantee after renewal failure, and leaks Redis sockets on initialization errors.

## What This Changes

The branch adds local, Redis, and fail-closed ACP session lock managers, wraps automatic ACP prompt dispatch with acquire/renew/release behavior, documents Redis lock environment variables, and adds lock manager plus dispatch tests.

## Best Possible Solution

Implement the lease at a shared ACP turn boundary, preserve same-session queueing semantics, cover prompt and steer turns, fail closed when a Redis lease is lost, close sockets on every initialization failure, and add the required changelog/docs coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has a manager test proving same-session turns queue and both execute, while the PR's contended-lock branch explicitly skips the second dispatch before it reaches `runTurn()`.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the narrowest maintainable solution because ACP already has a shared manager queue boundary and `/acp steer` remains a separate production entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This branch returns `acp_execution_locked` as soon as the session lock is held. With the default local backend, a second same-session prompt that current `AcpSessionManager.runTurn()` queues and executes is now dropped before it reaches the manager.
  - confidence: 0.94
- **[P2] Cover steer turns with the same session lease:** `src/auto-reply/reply/dispatch-acp.ts:239`
  - body: The new lock is acquired only in automatic prompt dispatch, but `/acp steer` still reaches `acpManager.runTurn()` through its own command path. In Redis mode, steer commands can overlap a normal prompt for the same session.
  - confidence: 0.9
- **[P2] Fail the active turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: When `renew()` returns false or throws, the heartbeat only logs and lets the turn continue. After the TTL expires, another worker can acquire the same session while the first worker is still running, defeating the distributed-lock guarantee.
  - confidence: 0.88
- **[P2] Close sockets when Redis initialization commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends AUTH and SELECT before returning the connection, but the caller only closes in a `finally` after `connectRedis()` resolves. Wrong credentials or an invalid DB reject these awaits and leak the just-opened socket on every ACP attempt.
  - confidence: 0.89
- **[P3] Add the required changelog entry:** `.env.example:83-89`
  - body: This adds a user-facing ACP Redis locking option and behavior, but the branch does not update `CHANGELOG.md`. Repo policy requires user-facing `fix`, `feat`, and `perf` changes to include a changelog entry.
  - confidence: 0.86

## Security Review
