## Summary

Keep open: current main does not already provide ACP Redis session locking, but this PR should not merge as-is because source inspection confirms it drops queued same-session ACP turns, misses the `/acp steer` turn path, keeps running after lease-renewal loss, and can leak Redis sockets on initialization failure. The PR also lacks contributor real behavior proof for the new Redis-backed ACP behavior.

## What This Changes

This PR adds ACP session lock managers, optional Redis-backed locking with TTL environment variables, dispatch acquire/renew/release wiring, `.env.example` documentation, and lock-manager/dispatch tests.

## Best Possible Solution

Implement distributed lease handling at a shared ACP turn boundary that preserves same-session queueing, covers prompt and steer turns, fails closed when a lease is lost, and guarantees Redis cleanup on every initialization path.

## Reproduction Assessment

Yes for the blocking PR regressions: current main queues same-session turns through `AcpSessionManager.withSessionActor`, while PR head returns `acp_execution_locked` before the second prompt reaches that queue. Source inspection also identifies the uncovered steer entrypoint and Redis renewal/socket cleanup paths.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the best maintainable boundary because ACP already has a shared manager queue and `/acp steer` is another production `runTurn` entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This branch returns `acp_execution_locked` when the local/default lock is held. Current `AcpSessionManager.runTurn()` serializes same-session turns and its regression test expects both concurrent turns to execute, so the second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Apply the lease at every ACP turn entrypoint:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only in `tryDispatchAcpReply()`. `/acp steer` still calls `acpManager.runTurn()` directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort active turns after lease renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: The renewal heartbeat logs false or throwing renewals and keeps the turn running. After the Redis TTL expires, another worker can acquire the same session while this process is still inside `runTurn()`, defeating the distributed lock.
  - confidence: 0.88
- **[P2] Close Redis sockets when initialization commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends AUTH and SELECT before returning, while the caller closes the socket only after `connectRedis()` resolves. Bad credentials or DB errors therefore leave the opened socket alive on each attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds raw Redis TCP/TLS handling and currently leaks sockets when Redis AUTH or SELECT initialization fails.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before AUTH and SELECT, and failed initialization commands reject before the caller's cleanup `finally` is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
  - confidence: 0.84
