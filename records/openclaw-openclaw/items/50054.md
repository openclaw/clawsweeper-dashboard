## Summary

Keep open: current main still has no ACP Redis session-locking surface, but this PR is not merge-ready because it drops queued same-session ACP prompts, misses the `/acp steer` entrypoint, lets active turns continue after Redis lease renewal is lost, and can leak Redis sockets on initialization failures.

## What This Changes

The branch adds local, Redis, and fail-closed ACP session lock managers, wraps automatic ACP prompt dispatch with acquire/renew/release behavior, documents Redis lock environment variables, and adds lock-manager plus dispatch tests.

## Best Possible Solution

Put lease handling at a shared ACP turn boundary, preserve same-session queueing, cover prompt and steer turns, fail or abort active work when a Redis lease is lost, close Redis sockets on all initialization failures, and add the required changelog coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has a manager test proving same-session turns queue and both execute, while the PR's contended-lock branch skips the second dispatch before it reaches `runTurn()`.

## Solution Assessment

No. Wrapping only automatic prompt dispatch with a non-waiting lock is not the narrowest maintainable solution because ACP already has a shared manager queue boundary and `/acp steer` remains a separate production entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This returns `acp_execution_locked` when the local/default lock is held. Current `AcpSessionManager.runTurn()` queues same-session turns and its regression test expects both to run, so a second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Cover steer turns with the session lease:** `src/auto-reply/reply/dispatch-acp.ts:239`
  - body: The lock is acquired only in `tryDispatchAcpReply()`. `/acp steer` still calls `acpManager.runTurn()` directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort the active turn when renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-283`
  - body: The renewal heartbeat logs false/throw results and keeps the turn running. Once the Redis TTL expires, another worker can acquire the same session while this process is still inside `runTurn()`, defeating distributed locking.
  - confidence: 0.88
- **[P2] Close Redis sockets on initialization failure:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` sends AUTH and SELECT before returning the connection, while cleanup is only installed by the caller after `connectRedis()` resolves. Bad credentials or DB errors leak the opened socket on each ACP attempt.
  - confidence: 0.89
- **[P3] Add the required changelog entry:** `.env.example:83-89`
  - body: This adds user-facing ACP Redis locking env vars and behavior, but the PR does not update `CHANGELOG.md`; repo policy requires a changelog entry for user-facing fix, feat, and perf changes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a raw Redis TCP/TLS path, and Redis initialization failures can leak sockets while ACP fails closed.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before AUTH/SELECT, and failed initialization commands reject before the caller's cleanup `finally` is installed. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
