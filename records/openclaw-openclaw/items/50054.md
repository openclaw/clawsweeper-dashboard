## Summary

Keep open: the Redis-backed ACP locking direction is useful, but this PR currently drops queued ACP turns, misses the steer entrypoint, keeps running after lease-renewal failure, leaks Redis sockets on initialization errors, and lacks real behavior proof.

## What This Changes

The PR adds optional local/Redis ACP session lock managers, wraps automatic ACP prompt dispatch with acquire/renew/release, documents new env vars, and adds lock-manager plus dispatch tests.

## Best Possible Solution

Put distributed lease handling at the shared ACP turn boundary, preserve same-session queueing, cover prompt and steer runs, abort or fail closed when a Redis lease is lost, and close Redis sockets on every initialization failure.

## Reproduction Assessment

Yes for the blocking PR regressions by source inspection: current main queues same-session runTurn calls, while the PR returns acp_execution_locked before the second turn reaches that queue. No live Redis/ACP proof artifact was provided.

## Solution Assessment

No. Wrapping only tryDispatchAcpReply is not the narrowest maintainable solution because it drops queued local turns and misses runAcpSteer; the lease belongs at a shared ACP turn boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is contended:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: This returns acp_execution_locked when the local/default lock is held. Current AcpSessionManager.runTurn() serializes same-session turns and its regression test expects both concurrent turns to execute, so the second prompt is dropped before it can queue.
  - confidence: 0.94
- **[P2] Cover steer turns with the session lease:** `src/auto-reply/reply/dispatch-acp.ts:239`
  - body: The new lock is acquired only in tryDispatchAcpReply(). runAcpSteer() still calls acpManager.runTurn() directly, so Redis mode can overlap a steer turn with a prompt for the same session across workers.
  - confidence: 0.9
- **[P2] Abort the active turn when renewal fails:** `src/auto-reply/reply/dispatch-acp.ts:271-284`
  - body: The renewal heartbeat logs false or throwing renewals and keeps the turn running. After the Redis TTL expires, another worker can acquire the same session while this process is still inside runTurn(), defeating the distributed lock.
  - confidence: 0.88
- **[P2] Close Redis sockets on initialization failure:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: connectRedis() sends AUTH and SELECT before returning, while the caller closes the socket only after connectRedis() resolves. Bad credentials or DB errors therefore leave the opened socket alive on each attempt.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff adds raw Redis socket handling, and Redis initialization failures can leak sockets before caller cleanup runs.

Concerns:

- **[medium] Redis initialization failures can leak sockets:** `src/auto-reply/reply/session-lock-manager.ts:283`
  - body: The Redis socket is opened before AUTH and SELECT, and failed initialization commands reject before createRedisCommandRunner() installs its cleanup finally. A misconfigured or hostile Redis endpoint could accumulate open sockets during repeated ACP attempts.
  - confidence: 0.84
