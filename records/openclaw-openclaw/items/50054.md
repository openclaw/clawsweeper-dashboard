## Summary

Keep open: current main does not implement distributed ACP session locks, but this PR is not merge-ready because it conflicts with main, lacks real Redis-backed behavior proof, and source inspection shows blocking message-delivery, session-state, and availability regressions.

## What This Changes

The PR adds optional Redis-backed ACP session locking, local and fail-closed lock managers, ACP dispatch acquire/renew/release wiring, env examples, and focused lock/dispatch tests.

## Best Possible Solution

Move distributed locking to a shared ACP manager-level turn lease that preserves current local queue semantics, covers prompt and steer turns, fails or aborts on lease loss, closes Redis setup sockets, and includes upgrade-safe docs and real Redis proof.

## Reproduction Assessment

Yes, for the patch regressions by source inspection: the PR's contention branch returns before current main's same-session actor queue, /acp steer still calls runTurn directly, and renew loss is log-only. I did not run a Redis-backed ACP setup in this read-only review.

## Solution Assessment

No. The PR targets a real multi-worker reliability gap, but locking only tryDispatchAcpReply before the existing actor queue is not the safest boundary; a shared manager-level lease that preserves current queuing and covers steer turns is more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is held:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: In the default no-Redis path, `LocalSessionLockManager` returns `acquired: false` for a second same-session message and this branch records `acp_execution_locked` before `AcpSessionManager.runTurn`. Current main queues same-session turns and runs both, so this drops a prompt that should wait behind the active turn.
  - confidence: 0.93
- **[P2] Lease the shared ACP turn path:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is only acquired in `tryDispatchAcpReply`, but `/acp steer` still calls `acpManager.runTurn()` directly. In Redis mode, a steer command and a normal prompt for the same session can still overlap across workers, so the distributed lock needs to sit at the shared turn boundary or another wrapper used by both entrypoints.
  - confidence: 0.88
- **[P2] Stop the turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:265-283`
  - body: Renew failures are only logged while the turn continues. If the TTL expires during a long-running turn after a missed renewal, another worker can acquire the same session while the first worker is still active, defeating the distributed session-state guarantee.
  - confidence: 0.87
- **[P2] Close Redis sockets when setup commands fail:** `src/auto-reply/reply/session-lock-manager.ts:283-286`
  - body: `connectRedis()` awaits `AUTH` and `SELECT` before returning the connection, but the caller only closes the socket after `connectRedis()` resolves. A bad password, invalid DB, or setup timeout therefore leaves the opened socket alive on every fail-closed ACP attempt.
  - confidence: 0.86
