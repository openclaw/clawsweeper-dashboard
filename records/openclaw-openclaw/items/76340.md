## Summary

Keep open. The report is a credible performance regression with concrete old-version timing data, and current main has several adjacent mitigations, but there is not yet high-confidence proof that the reporter's Docker/Feishu/plugin-count event-loop stall is fixed or isolated to one remaining hot path.

## What This Changes

The issue reports a post-v2026.4.23 performance regression where upgraded Docker deployments hit multi-second event-loop stalls, slow status commands, Feishu/Lark probe timeouts, and stale runtime-dependency errors.

## Best Possible Solution

Capture a current-main Docker timing trace for the 14-plugin and high-plugin-count cases, then land a focused hot-path fix with regression coverage for the specific blocking operation.

## Reproduction Assessment

Unclear. The reporter provides concrete v2026.4.27/v2026.4.29 versus v2026.4.23 timing and event-loop logs, but this review did not establish a failing current-main Docker reproduction or isolate the exact current hot path.

## Solution Assessment

Unclear. The recent sessions.list, transcript-read, status-fast-path, and runtime-deps cleanup work is directionally right, but the maintainable fix needs current-main profiling to choose between session listing, transcript update, plugin discovery, and Feishu probe paths.
