## Summary

Keep open. Current main already covers the logging-config part of the PR through a newer direct config reader and rolling log resolution, but the local memory embedding provider still lacks the VRAM retry/CPU-only fallback behavior. The PR is useful source work, but it is not mergeable as-is because it is stale against the current package layout and still has blocking embedding-resource regressions.

## What This Changes

This PR changes logging config resolution and local memory embedding initialization by adding direct logging config reads, `YYYY-MM-DD` log path handling, GPU-fit retry, temporary CPU fallback, and targeted tests.

## Best Possible Solution

Port the embedding resilience behavior into the current `packages/memory-host-sdk` implementation, keep the existing logging fix on main, preserve documented `local.contextSize`, serialize concurrent VRAM recovery, dispose temporary CPU resources on every failure path, and add focused tests plus a user-facing changelog entry.

Yes for a code-review reproduction path: current main has a single `ensureContext()` path that rethrows local llama initialization failures, and the PR's mocked tests exercise `loadModel()` throwing an `available VRAM` error. A live hardware reproduction was not performed in this read-only sweep.

## Solution Assessment

Full review comments:

- **[P2] Preserve local context size when creating embeddings:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: The PR stops passing `local.contextSize` into `createEmbeddingContext()`. That makes the documented default of 4096 and user overrides ineffective, which can push constrained local setups back toward the runtime/model default and increase the same memory pressure this PR is meant to handle.
  - confidence: 0.87
- **[P2] Clean up CPU fallback when context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If `cpuModel.createEmbeddingContext()` rejects, this helper throws before it returns the `dispose` callback, so the temporary CPU model and llama runtime are leaked. Wrap CPU context creation so `cpuModel` and `cpuLlama` are disposed on that failure path too.
  - confidence: 0.84
