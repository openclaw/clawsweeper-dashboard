## Summary

Keep open: current main covers much of the logging side, but the active local embedding provider still lacks the requested VRAM retry and CPU fallback, and this PR needs a rebase/port, correctness fixes, and real behavior proof before merge.

## What This Changes

The PR changes logging config/file resolution and local memory embedding initialization/tests to expand dated log paths, retry VRAM-style GPU initialization failures, and fall back to a temporary CPU embedding context.

## Best Possible Solution

Port the embedding resilience to packages/memory-host-sdk while preserving local.contextSize and full cleanup semantics, and handle any remaining logging placeholder work through the narrower logging path already on main or the dedicated logging PR.

## Reproduction Assessment

Yes for source-level review: current main's active local embedding provider rethrows initialization errors, and the PR tests model a VRAM-style loadModel failure. I did not establish live GPU/node-llama-cpp runtime reproduction in this read-only pass.

## Solution Assessment

No, not as submitted. The maintainable fix is a rebased package-level patch that preserves the current context-size contract, serializes VRAM recovery, cleans up all CPU fallback failure paths, and includes real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve current local context size:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: Current main defaults local embeddings to `{ contextSize: 4096 }` and passes configured numeric or `"auto"` values into `createEmbeddingContext`. This patch creates the GPU embedding context with no options, so rebasing or porting it as-is would drop the VRAM-bounding contract it needs to preserve.
  - confidence: 0.9
- **[P2] Clean up CPU resources when context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: `createTemporaryCpuContext` only returns its disposer after `cpuModel.createEmbeddingContext()` succeeds. If context creation rejects, the temporary CPU model and llama runtime created above are left alive, so wrap that call and dispose both resources on the failure path.
  - confidence: 0.87
