## Summary

Keep open. The logging half is already covered on current main, but the active local embedding implementation still lacks VRAM retry and temporary CPU fallback, and this PR needs rework before merge because it targets the old source path and regresses current context-size and cleanup contracts.

## What This Changes

This PR changes logging config/file resolution and local memory embedding initialization by adding direct logging config reads, `YYYY-MM-DD` log path handling, GPU-fit retry, temporary CPU fallback, and targeted tests.

## Best Possible Solution

Port the useful embedding resilience into the current `packages/memory-host-sdk` implementation, preserve `local.contextSize`, serialize VRAM recovery, dispose temporary CPU resources on every success and failure path, leave the logging behavior already on main intact, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes for source-level review: current main rethrows local llama initialization failures, and the PR's mocked tests demonstrate the intended `available VRAM` retry/fallback path. A live GPU reproduction was not performed in this read-only sweep.

## Solution Assessment

No, not as submitted. The maintainable fix is a current-main rework or replacement PR against the package source that keeps the existing context-size contract and closes the CPU fallback lifecycle gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve local context size when creating embeddings:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: The PR stops passing `local.contextSize` into `createEmbeddingContext()`. Current main documents and tests the default of 4096 plus user overrides, so dropping this can make constrained local setups use the runtime/model default and reintroduce memory pressure.
  - confidence: 0.87
- **[P2] Clean up CPU fallback on context creation failure:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If `cpuModel.createEmbeddingContext()` rejects, this helper throws before returning the cleanup callback, leaving the temporary CPU model/runtime undisposed. Wrap CPU context creation so the model and llama instance are disposed on that failure path too.
  - confidence: 0.84
