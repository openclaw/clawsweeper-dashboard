## Summary

Keep open. The logging half is largely covered on current main, but the local embedding fallback is still absent from the active package implementation and the PR head still has correctness blockers around context-size preservation and CPU fallback cleanup.

## What This Changes

The PR changes logging config/file resolution and local memory embedding initialization/tests to add configured log path handling, date placeholder expansion, GPU retry, and CPU fallback.

## Best Possible Solution

Port the embedding resilience into packages/memory-host-sdk, preserve local.contextSize in GPU-fit and createEmbeddingContext calls, serialize VRAM recovery, and dispose temporary CPU resources on every failure path while keeping current main's logging implementation.

## Reproduction Assessment

Yes for source-level review: current main's active local embedding provider still rethrows local node-llama initialization failures without VRAM retry or CPU fallback. I did not run a live GPU/VRAM reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable fix is a current-main package patch that preserves the context-size contract and closes the CPU fallback lifecycle gaps; the PR's logging changes are mostly superseded by current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the configured embedding context size:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: The PR creates the GPU embedding context without passing the current `local.contextSize` value. Current main documents and tests that this value defaults to 4096 and is passed to node-llama-cpp, so dropping it can make constrained local setups use the runtime/model default and reintroduce memory pressure.
  - confidence: 0.9
- **[P2] Dispose CPU fallback resources if context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If `cpuModel.createEmbeddingContext()` rejects, this helper throws before returning the disposer, leaving the temporary CPU model and llama runtime alive. Wrap context creation so `cpuModel` and `cpuLlama` are disposed on that failure path too.
  - confidence: 0.88
