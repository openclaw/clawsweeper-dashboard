## Summary

Keep open: current main covers much of the logging config/file-resolution problem, but the active local embedding provider still has no VRAM retry or temporary CPU fallback, and this PR still needs rework plus real runtime proof before merge.

## What This Changes

The PR changes logging config/file resolution and local memory embedding initialization to expand dated log paths, retry GPU VRAM failures, temporarily fall back to CPU, and add focused tests.

## Best Possible Solution

Port the embedding resilience to packages/memory-host-sdk while preserving local.contextSize, serialized VRAM recovery, full CPU cleanup, and focused tests; keep current main's logging implementation or land a separate narrow logging-placeholder fix.

## Reproduction Assessment

Yes for source-level review: current main's active local embedding provider rethrows initialization errors, and the PR tests model the VRAM failure path. I did not establish a live GPU/node-llama-cpp reproduction in this read-only pass.

## Solution Assessment

No, not as submitted. The maintainable fix is a rebased current-main package patch that preserves the context-size contract and closes CPU fallback lifecycle gaps, with separate handling for any remaining logging placeholder behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the embedding context size:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: Current main's local memory contract passes local.contextSize, defaulting to 4096, into createEmbeddingContext. This patch creates the GPU embedding context with no options after hardcoding a 24-token fitContext on model load, so configured/default sizing is dropped and can recreate the VRAM pressure this change is meant to avoid.
  - confidence: 0.9
- **[P2] Clean up CPU resources when context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If cpuModel.createEmbeddingContext() rejects, this helper throws before returning the disposer, leaving the temporary CPU model and llama runtime alive. Wrap context creation so both resources are released on that failure path.
  - confidence: 0.87
