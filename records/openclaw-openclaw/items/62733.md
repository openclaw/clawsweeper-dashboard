## Summary

Keep open: the logging half is already covered on current main, but the local embedding VRAM fallback remains useful and is not implemented in the active package path. This PR should not merge as-is because it targets the stale `src/memory-host-sdk` location, drops the embedding `contextSize` contract, has a CPU fallback cleanup gap, and lacks real behavior proof.

## What This Changes

The PR changes logging config/file resolution and local memory embedding initialization/tests to add config reads, date-expanded log file paths, GPU retry, CPU fallback, and retryable model-path resolution.

## Best Possible Solution

Port only the embedding resilience to `packages/memory-host-sdk`, preserving `contextSize`, serialized VRAM recovery, full CPU cleanup, focused tests, and real node-llama-cpp proof while keeping current main's logging implementation.

## Reproduction Assessment

Yes for source-level review: current main's active local embedding provider rethrows initialization errors, and the PR tests model the VRAM failure path. I did not establish a live GPU/node-llama-cpp reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable fix is a current-main package patch that preserves the context-size contract and closes CPU fallback lifecycle gaps instead of landing the stale branch wholesale.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the embedding context size:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: Current main exposes `local.contextSize` with a 4096 default and passes it to `createEmbeddingContext`. This patch creates the GPU embedding context with no options, so custom and default sizing are lost and the runtime can recreate the VRAM pressure the change is meant to avoid.
  - confidence: 0.9
- **[P2] Clean up CPU resources when context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If `cpuModel.createEmbeddingContext()` rejects, `createTemporaryCpuContext` throws before returning the disposer, leaving the temporary CPU model and llama runtime alive. Wrap context creation so both resources are released on that failure path.
  - confidence: 0.87
