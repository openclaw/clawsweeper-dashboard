## Summary

Keep open: the logging half is already covered on current main, but the embedding fallback is still meaningful and the PR is not mergeable/correct as submitted.

## What This Changes

The PR changes logging config/file resolution and local memory embedding initialization/tests to add active config reads, date-expanded log files, GPU retry, and temporary CPU fallback.

## Best Possible Solution

Keep current main's logging implementation and port only the embedding resilience to packages/memory-host-sdk with contextSize preserved, serialized VRAM recovery, full CPU cleanup, tests, and real runtime proof.

## Reproduction Assessment

Yes for source-level review: current main rethrows local llama initialization failures, and the PR tests model the available-VRAM retry/fallback path. I did not run live GPU hardware proof in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable fix is a current-main package patch that preserves the context-size contract and closes the CPU fallback lifecycle gaps while leaving the already-fixed logging path alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the configured embedding context size:** `src/memory-host-sdk/host/embeddings.ts:192`
  - body: Current main exposes local.contextSize with a 4096 default and passes it to node-llama-cpp. This path creates the GPU embedding context with no options, so node-llama-cpp falls back to its auto context size and can recreate the VRAM pressure the PR is trying to avoid.
  - confidence: 0.9
- **[P2] Dispose CPU fallback resources if context creation fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If cpuModel.createEmbeddingContext() rejects, this helper throws before returning the disposer, leaving the temporary CPU model and llama runtime alive. Wrap context creation so cpuModel and cpuLlama are released on that failure path too.
  - confidence: 0.87
