## Summary

Keep open for maintainer/provider-compat follow-up: the fresh Qwen3.6/llama.cpp Docker report matches the closed Qwen XML tool-call parse reports, while current docs and code already treat raw XML tool-call text as provider/template incompatibility rather than a completed OpenClaw tool call. The LM Studio counter-data is useful because it shows Qwen3.6 can work when the server converts Qwen XML into structured OpenAI tool calls.

## What This Changes

The issue asks OpenClaw to avoid dashboard initialization parse errors when Qwen3.6 27B is served through llama.cpp and emits XML-style tool-call text.

## Best Possible Solution

Keep this as the live Qwen3.6/llama.cpp compatibility thread and decide whether OpenClaw should add clearer llama.cpp/Qwen diagnostics or docs around the existing `localModelLean`, `supportsTools: false`, and `tool_choice` knobs.

## Reproduction Assessment

No high-confidence live reproduction was established because this review did not run a Qwen3.6 27B llama.cpp server. The issue logs, related reports, docs, and current tool-call diagnostic path make the failure class source-reproducible at medium confidence.

## Solution Assessment

Unclear. The current direction of not executing raw XML tool text is the safest baseline, but maintainers still need to decide whether the best OpenClaw-side response is docs, a doctor diagnostic, or a narrow provider-compat guard.
