## Summary

Keep open: this is a focused attempt at a real gateway approval-routing bug, but the patch is not merge-ready because it calls a nonexistent gateway context API and still falls back to the unfiltered global broadcast path that the linked report is trying to remove. The external PR also has no after-fix real behavior proof.

## What This Changes

The branch changes two gateway plugin approval request fallback paths to try admin-only delivery when normal recipient resolution is unavailable.

## Best Possible Solution

Replace the unfiltered fallback with the existing approval-client lookup filtered to admin clients or a no-route expiry path, compare against the open turn-source routing candidate, then add focused regression coverage and redacted real Telegram/gateway proof.

## Reproduction Assessment

Yes at source level: current v2026.6.8 and current main contain the unfiltered fallback, and PR head still reaches `context.broadcast(...)` when the proposed admin lookup is unavailable or empty. I did not run the live two-bot Telegram scenario, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No. The current patch is not the best fix because it adds an undefined context method and preserves the unsafe global fallback; the safer fix is to use the existing approval-client lookup with an admin filter or expire with no route, while coordinating with the open turn-source routing candidate.
