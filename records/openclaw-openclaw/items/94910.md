## Summary

Keep open: the reported approval leak is real, but this PR is not merge-ready because it calls an undefined gateway context API, changes only a fallback path that live gateway contexts usually skip, and has no after-fix real behavior proof.

## What This Changes

The PR changes plugin approval request fallbacks in shared gateway approval handling and node-invoke policy handling from unconditional broadcast toward an admin-client broadcast attempt.

## Best Possible Solution

Either supersede this branch with the focused route-forwarding path or rework it to use the existing approval-client lookup and recipient visibility contract, then prove that only the originating agent's Telegram bot receives the approval prompt.

No. The submitted fallback-only patch is not the best fix because it uses a non-existent context method and misses the live recipient-filter/turn-source path where the leak can still occur.
