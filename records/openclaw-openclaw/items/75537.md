## Summary

Keep this PR open. Current main still limits Swift stored-device-token retry to loopback, while the PR narrowly extends the trusted retry path to pinned `wss://` sessions; the item is also maintainer-labeled and member-authored, so it should stay in normal maintainer review rather than cleanup close.

## What This Changes

The PR adds a Swift trust-provider protocol for device-token retry, makes `GatewayTLSPinningSession` opt in when a pinned fingerprint is configured, lets `GatewayChannelActor` retry on trusted pinned `wss://` sessions, and adds a macOS regression test for stale shared-token recovery.

## Best Possible Solution

Review and land this PR through the normal maintainer path if the trust-model change is accepted, keeping retry limited to loopback or explicitly pinned `wss://` sessions and retaining the regression coverage.

## Reproduction Assessment

Yes. Current main can be checked by the loopback-only `isTrustedDeviceRetryEndpoint()` implementation, and the PR adds a focused macOS test that reproduces stale shared-token recovery on a pinned remote gateway.

## Solution Assessment

Yes. The approach is narrow: it mirrors Node/Android behavior while requiring both `wss://` and an explicit TLS-pinning trust provider, instead of allowing all remote secure endpoints to replay stored device tokens.
