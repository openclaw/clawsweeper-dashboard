## Summary

Keep open: this is a plausible bug-fix direction, but the patch only handles the later env-marker guard and does not cover the earlier process-tree guard that a normal child spawned by the live gateway can still hit.

## What This Changes

The PR clears gateway service marker env vars from the auto-update child override, adds a regression test for that override, and adds a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff that also satisfies the process-tree safety guard, then cover that path with regression tests and redacted managed-service runtime proof.

## Reproduction Assessment

Yes, from source inspection: a managed gateway spawns the updater as a normal child, and `updateCommand` checks gateway process ancestry before the env-marker guard. I did not live-run launchd/systemd in this read-only review.

## Solution Assessment

No. Clearing the inherited marker keys is valid for the later env guard, but it is not sufficient for the documented service auto-update path because the earlier process-tree guard can still abort first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route auto-update outside the process-tree guard:** `src/infra/update-startup.ts:285-286`
