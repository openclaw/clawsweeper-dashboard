## Summary

Keep open: source inspection shows the patch clears the later env-marker guard, but a real managed-service `openclaw update --yes` still reaches the earlier gateway process-tree guard before package install; the supplied proof does not exercise that path.

## What This Changes

The PR clears gateway service marker environment variables from the auto-update child process env override, adds regression coverage, and records the fix in the changelog.

## Best Possible Solution

Implement a service-safe auto-update handoff that either runs outside the gateway ancestry check or explicitly handles the auto-update path without weakening the manual-update safety guard, then cover it with focused regression tests and redacted managed-service proof.

## Reproduction Assessment

Yes, from source inspection and the linked logs: a package auto-update spawned by the managed gateway inherits service context, and a real `--yes` update reaches the service preflight before package install. I did not live-run launchd/systemd in this read-only review.

## Solution Assessment

No. Clearing the marker env keys is a valid piece of the fix, but it is not the complete solution because the earlier process-tree guard can still abort the managed-service child before the env-marker guard matters.
