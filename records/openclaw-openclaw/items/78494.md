## Summary

Keep open. The bug is real, but this PR is not ready: the proposed `undefined` env override does not delete inherited service markers under the current env-merge contract, and the non-dry-run service update path still has the active gateway ancestry guard to resolve.

## What This Changes

The PR adds service marker keys to the auto-update child env override, adds a regression test for that override, and adds a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff, or a narrowly trusted auto-update path, that truly removes inherited service markers, preserves the manual package-update guard, and includes regression coverage plus redacted launchd/systemd proof.

## Reproduction Assessment

Yes. Source inspection shows the PR's `undefined` override does not remove inherited env markers, and the existing non-dry-run package path can still block gateway-owned child updates through the ancestry preflight.

## Solution Assessment

No. The proposed env override is not an effective deletion mechanism under current `resolveCommandEnv` behavior, and it does not address the earlier service ancestry block that applies to the real `openclaw update --yes` path.
