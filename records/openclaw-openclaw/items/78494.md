## Summary

Keep open: the env strip is mechanically correct, but it does not address the earlier managed-service ancestry block that a child spawned by the live gateway still hits before package install. The supplied proof shows dry-run/env behavior, not the patched non-dry-run managed-service update path.

## What This Changes

The PR clears gateway service marker env vars from the auto-update child spawn, adds a regression test, and adds a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff outside the live gateway process tree, or explicitly model a trusted auto-update path in `openclaw update`, with regression coverage for both marker env and ancestry-guard behavior plus redacted launchd/systemd proof.

## Reproduction Assessment

Yes. Source inspection shows the scheduled gateway auto-update spawns a non-dry-run `openclaw update --yes` child from the live gateway, and the package update path blocks when the gateway runtime PID is an ancestor; the provided logs support the same failure mode.

## Solution Assessment

No. Clearing the service marker env vars is a useful partial fix, but it leaves the earlier ancestry guard intact; the safer fix is to route auto-update through a service-safe handoff or a dedicated trusted path covered by tests and live proof.
