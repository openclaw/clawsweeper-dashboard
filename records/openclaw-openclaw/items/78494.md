## Summary

Keep open: the branch is a focused bug-fix attempt, but the proposed `undefined` env override does not remove inherited service markers under current `resolveCommandEnv` semantics, and the real non-dry-run package update path can still be stopped by the gateway ancestry preflight.

## What This Changes

The PR changes the auto-update child process env override to clear gateway service markers, adds a regression test for that override, and adds a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff that actually removes the inherited service markers and explicitly handles trusted auto-update through the ancestry preflight while preserving the manual in-service package-update refusal.

## Reproduction Assessment

Yes. Source inspection shows the inherited env markers survive the proposed override, and current tests/source show the gateway ancestry preflight blocks package updates from a gateway-owned child before package install.

## Solution Assessment

No. The current PR is not the best fix because `undefined` is not a deletion marker in `resolveCommandEnv`, and the branch does not route the real auto-update child past the existing ancestry guard safely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use a real marker deletion path:** `src/infra/update-startup.ts:285-286`
  - body: Setting these keys to `undefined` does not remove inherited values. `runCommandWithTimeout` resolves the child env by copying `process.env` first, and `assignChildEnvValue` ignores `undefined` override values, so a gateway-launched auto-update child will still inherit `OPENCLAW_SERVICE_MARKER` and `OPENCLAW_SERVICE_KIND`.
  - confidence: 0.97

  - body: Even after marker deletion is made real, `openclaw update --yes` still runs `maybeStopManagedServiceBeforePackageUpdate` before package install. A child spawned by the live gateway has the gateway PID in its ancestry, so the existing preflight can still return the process-tree block before the update reaches the install path.
  - confidence: 0.91
