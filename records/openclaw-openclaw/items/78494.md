## Summary

Keep open: the PR addresses a real updater bug, but the proposed env-marker strip is incomplete because current source still runs the package-update process-tree preflight before the marker-env guard, so a gateway-spawned updater can still be blocked before package install.

## What This Changes

The PR clears gateway service marker variables from the auto-update child process env override and adds regression coverage plus a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff that handles the auto-update child path before the ancestry block while preserving the manual in-service package-update guard, then cover both update-startup and update-command behavior with focused tests and redacted managed-service proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the managed gateway spawns the auto-update child, the package update path inspects the managed service runtime, and the ancestor block runs before the marker-env guard. I did not live-run launchd or systemd in this read-only review.

## Solution Assessment

No, this is not yet the best complete fix. Clearing marker keys is necessary for the later env guard, but the service runtime ancestry preflight still needs an auto-update-safe route or explicit handling before package install.
