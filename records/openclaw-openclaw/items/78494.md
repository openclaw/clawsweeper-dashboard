## Summary

Keep this PR open but do not merge as-is: it strips the inherited service-marker env vars, but the real non-dry-run package update still hits the managed-service ancestry guard before the env-marker guard would matter.

## What This Changes

The PR clears `OPENCLAW_SERVICE_MARKER` and `OPENCLAW_SERVICE_KIND` from the auto-update child env override, adds a focused regression test, and adds a changelog entry.

## Best Possible Solution

Implement a service-safe auto-update handoff, or an explicit trusted auto-update path, with regression coverage for both marker env and ancestry behavior while preserving the manual in-service package-update guard.

## Reproduction Assessment

Yes, at source level. The gateway scheduler spawns `openclaw update --yes`, current code blocks package updates when the managed gateway PID is an ancestor, and current tests assert that ancestry block; I did not run a live launchd/package update in this read-only review.

## Solution Assessment

No. Clearing inherited marker env vars is a useful partial fix, but it does not address the earlier ancestry preflight that still blocks a real gateway-spawned non-dry-run update.
