## Summary

Keep open: current main covers explicit `/stop` and `/subagents kill` cascades plus restart-aborted recovery, but I did not find a general parent-terminal hook that recursively cancels, resumes, or marks depth-2 descendants when a depth-1 subagent dies abruptly.

## What This Changes

The issue reports that nested subagent descendants can remain `stale_running` when their parent subagent dies mid-run.

## Best Possible Solution

Add narrow lifecycle reconciliation so descendants of a terminal or lost subagent parent are recursively cancelled, resumed, or marked lost without implementing broader chain tracking.

## Reproduction Assessment

Yes, from source inspection and the production evidence in the report rather than a live run. Explicit cascade and restart recovery paths exist, but the parent-terminal path does not connect an abruptly dead depth-1 subagent to still-running depth-2 descendants.

## Solution Assessment

No, current main is only a partial solution because it relies on explicit stop/kill paths, marker-based restart recovery, and a generic stale-run sweeper. The narrower maintainable fix is to hook parent terminal/lost reconciliation into existing subagent registry/control paths and cover depth-2 descendants with regression tests.
