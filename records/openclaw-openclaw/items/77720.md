- `impact:session-state`: The affected state is orphaned subagent session and task lineage that remains running or stale_running after the parent disappears.

## Summary

Keep open: current main and v2026.6.10 have explicit descendant cascade kill for intentional subagent control/admin kills, but the lost-parent/restart/orphan paths still do not automatically terminate, mark, or otherwise settle active depth-2 descendants when an intermediate subagent parent dies mid-run.

## What This Changes

The issue requests lifecycle handling so child subagent sessions are terminated or marked orphaned when their parent subagent dies abruptly.

## Best Possible Solution

Add one maintainer-approved lifecycle reconciliation path that reuses existing cascade kill, orphan recovery, and task maintenance primitives to settle active descendants when a parent subagent disappears.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main has depth-2 lineage and stale-running diagnostics, but lost-context/restart recovery does not traverse descendants; I did not run a live gateway kill/OOM/restart reproduction.

## Solution Assessment

Is this the best way to solve the issue? The requested direction is reasonable, but the exact fix is not settled because maintainers need to choose kill, orphan/lost, or resume semantics before implementation.
