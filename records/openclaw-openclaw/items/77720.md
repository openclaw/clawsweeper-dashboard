## Label Justifications

- `P1`: The issue reports production multi-agent descendants left running indefinitely after parent death, making depth-2 orchestration unsafe.
- `impact:session-state`: The affected state is orphaned subagent session/task lineage that remains `running` or `stale_running` after the parent disappears.

## Summary

Keep open: current main has explicit cascade kill and stale-task cleanup paths, but I still do not see a current-main path that automatically terminates or marks depth-2 descendants when an intermediate subagent parent is lost mid-run. The linked open task-maintenance PR is narrower and does not own the parent-death lifecycle decision.

## What This Changes

The issue reports that depth-2 subagent descendants can remain `stale_running` after their depth-1 parent dies or the gateway restarts mid-run.

## Best Possible Solution

Add one maintainer-approved lifecycle reconciliation path that settles active descendants and task state when a parent subagent dies, reusing existing cascade kill, orphan recovery, and task maintenance primitives instead of adding another stale-task workaround.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, with concrete depth-2 steps and production stale-running evidence, but I did not run a live gateway kill/OOM/restart reproduction on current main.

## Solution Assessment

Is this the best way to solve the issue? No current implementation is sufficient; the maintainable fix is to connect parent-death detection to existing descendant kill/orphan/recovery and task reconciliation primitives after maintainers choose the terminal semantics.
