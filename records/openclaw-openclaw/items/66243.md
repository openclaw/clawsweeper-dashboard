## Summary

Keep this PR open: the shared proxy-fetch behavior is not fully implemented on current main, and the production Discord multipart failure has credible supporting context. However, the proposed branch is not ready as-is because the request-like normalizer has a concrete security bug and the branch is stale against current env-proxy handling.

## What This Changes

The PR branch updates `src/infra/net/proxy-fetch.ts` and its tests to convert global `FormData` to undici `FormData`, flatten `Request` or request-like inputs, strip selected hop-by-hop/framing headers, and preserve signal/header override semantics.

## Best Possible Solution

Rebase the useful shared FormData normalization onto current main, preserve `resolveEnvHttpProxyAgentOptions(env)` behavior, remove or harden request-like duck typing, and keep header stripping aligned with proxy security expectations before merge.

## Reproduction Assessment

Yes for the FormData proxy bug: current `proxy-fetch.ts` lacks conversion, TC500 reports a production Discord repro, and undici’s FormData source shows multipart handling depends on undici-recognized FormData. The Request-flattening part is less proven because undici v8 already accepts `Request` inputs.

## Solution Assessment

No, not as written. The shared FormData conversion is a plausible fix, but the safest implementation should avoid broad request-like duck typing, preserve current env proxy options, and add focused security/compatibility tests.

## Review Findings

Full review comments:

- **[P2] Require own request-like URL properties:** `src/infra/net/proxy-fetch.ts:60`
  - body: The request-like guard uses `"url" in input`, so an object with an inherited `url` property can be treated as a proxy request and steer the outbound URL. Require an own `url` property, or avoid duck-typing and let undici handle real `Request` objects.
  - confidence: 0.9
- **[P2] Preserve current env proxy option handling:** `src/infra/net/proxy-fetch.ts:280-284`
  - body: Current main builds `EnvHttpProxyAgent` with `resolveEnvHttpProxyAgentOptions(env)`, including ALL_PROXY/all_proxy support. This branch is still on the older HTTP(S)-only gate and default constructor, so a rebase that keeps this function body would regress all-proxy setups and supplied-env tests.
  - confidence: 0.82
- **[P2] Strip Proxy-Connection with hop-by-hop headers:** `src/infra/net/proxy-fetch.ts:69-80`
  - body: The new strip set removes standard hop-by-hop/framing headers but still lets `Proxy-Connection` through the proxy path. Add it to the strip list, and consider removing headers named by the `Connection` header before forwarding.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch touches proxy request normalization and has concrete header/request-like input safety issues to address before merge.

Concerns:

- **[medium] Inherited URL request-like detection:** `src/infra/net/proxy-fetch.ts:60`
  - body: Using the `in` operator for `url` allows inherited prototype properties to redirect a normalized proxy request; this is a prototype-pollution gadget in a network egress path.
  - confidence: 0.9
