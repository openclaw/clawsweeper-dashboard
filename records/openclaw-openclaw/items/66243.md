## Summary

Keep open: the PR addresses a real shared proxy/FormData gap, but current main only fixed the Discord-specific and runtime-fetch paths, and this branch still has security and env-proxy compatibility blockers before it can merge.

## What This Changes

The PR updates shared proxy fetch code and tests to convert global FormData to undici FormData, flatten Request or request-like inputs, strip selected hop-by-hop headers, and preserve signal/header override semantics.

## Best Possible Solution

Carry forward the shared FormData normalization into current proxy-fetch while preserving resolveEnvHttpProxyAgentOptions(env), avoiding broad request-like duck typing unless a real caller needs it, and aligning header stripping with proxy security expectations. Coordinate the remaining shared fix with the open #48556 attempt and keep the already-merged Discord-specific fix from #71383.

## Reproduction Assessment

Yes for the FormData proxy bug: current shared proxy-fetch still lacks conversion, TC500 reported a production Discord repro, and undici v8.1.0 source shows multipart handling depends on undici-recognized FormData. The Request-flattening part is less proven because undici v8.1.0 already accepts Request inputs.

## Solution Assessment

No, not as written. A narrow shared FormData conversion is plausible, but this branch should preserve current env-proxy options, harden or remove request-like duck typing, and add focused security/compatibility coverage before merge.

## Security Review

Status: needs_attention

Summary: The diff touches proxy request normalization and still has concrete request-like detection and hop-by-hop header hardening issues.

  - body: Using the `in` operator for `url` allows inherited prototype properties to redirect a normalized proxy request; this is a prototype-pollution gadget in a network egress path.
