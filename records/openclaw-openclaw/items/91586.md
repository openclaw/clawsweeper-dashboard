## Label Justifications

- `P1`: The PR targets a package update workflow that can strand post-core plugin convergence and gateway restart for real users after a verified install.
- `merge-risk: 🚨 compatibility`: Existing users who relied on post-install doctor failures stopping the update will now see those completed failures treated as advisory warnings.
- `merge-risk: 🚨 availability`: The changed path can continue toward gateway restart after a completed doctor failure, and the managed restart leg is not covered by the supplied real proof.

## Summary

Keep open for normal maintainer review: the patch matches the package-update bug path and I found no blocking code finding, but it intentionally changes update failure semantics and the managed gateway restart leg is still not live-proven.

## What This Changes

The branch turns completed nonzero package post-install doctor results into advisory warning steps, renders their stderr as warnings, and adds update-cli regression tests for advisory versus killed doctor runs.

## Best Possible Solution

Land the narrow advisory behavior only if maintainers accept completed doctor failures as non-fatal after verified package installs; otherwise require a stricter classifier or managed-restart proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main returns the post-install doctor step directly and the package update helper treats a nonzero post-verify step as failed, stopping before post-core convergence. I did not run the updater locally because this was a read-only review.

## Solution Assessment

Yes, with maintainer risk acceptance: normalizing only the post-install doctor result after package verification is narrower than weakening install verification or post-core convergence. The safer alternative is to request managed restart proof or classify only known advisory doctor failures before merge.
