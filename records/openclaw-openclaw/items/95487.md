## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate stripping behind approved TTS policy:** `packages/speech-core/src/tts.ts:1211`
  - body: This rewrites `params.text` for every provider path, including explicit tool, stream, and telephony calls. Current `TtsConfig`/`TtsConfigSchema` have no opt-in or stored preference, and the canonical request is still a product-decision item, so upgrading users would silently stop hearing emoji they may intentionally ask TTS to speak.
  - confidence: 0.91
- **[P2] Do not fall back to emoji-only input:** `packages/speech-core/src/tts.ts:1211`
  - body: When stripping removes the whole utterance, `inputText` falls back to `params.text`, so emoji-only explicit TTS still sends the original emoji to the provider. Return a no-op or clear empty-after-filter error before provider selection instead of synthesizing the text this filter is meant to remove.
  - confidence: 0.9
- **[P2] Cover non-pictographic emoji sequences:** `packages/speech-core/src/tts.ts:1184`
  - body: The regex only starts matches with `Extended_Pictographic`, so regional-indicator flags, keycap emoji, and tag sequences are left in provider text even though the PR claims complex flag handling. Add coverage for those emoji classes or narrow the behavior and tests.
  - confidence: 0.88
- **[P2] Keep spoken metadata aligned with audio:** `packages/speech-core/src/tts.ts:1211`
  - body: Filtering inside `prepareSpeechSynthesis` changes only provider input. `maybeApplyTtsToPayload` still records `spokenText` from the unfiltered `textForAudio`, and the TTS tool transcript uses the original tool text, so transcript/status metadata can disagree with the audio output.
  - confidence: 0.86
