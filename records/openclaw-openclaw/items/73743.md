## Summary

Keep open. Current main still allows timeout-killed subprocesses to reach plugin callers with `code: null`, so the Kitchen cache-poisoning path remains plausible, and the broader macOS recipes/Kitchen fan-out regression still needs live plugin profiling.

## What This Changes

The issue asks OpenClaw to address slow CLI/plugin subprocess startup under macOS launchd fan-out and prevent recipes/Kitchen timeout output from being cached as successful command output.

## Best Possible Solution

Normalize timeout and no-output-timeout subprocess results for plugin callers, then profile the recipes/Kitchen startup path under launchd before choosing activation, worker, or scheduling changes.

## Repair Work Prompt

Fix the cache-poisoning sub-bug from https://github.com/openclaw/openclaw/issues/73743. In `src/process/exec.ts`, ensure `runCommandWithTimeout` returns a numeric non-zero code, preferably `124`, whenever `termination` is `timeout` or `no-output-timeout`, even when the child exits from SIGKILL and `resolveProcessExitCode` would otherwise be `null`. Preserve the Windows shim behavior for non-timeout clean-close cases. Update focused tests in `src/process/exec.test.ts` and `src/process/exec.no-output-timer.test.ts` to assert numeric timeout codes rather than only `not.toBe(0)`, and add or adjust plugin-runtime coverage if needed to prove plugin callers receive the hardened result. Add a user-facing `CHANGELOG.md` fix entry. Do not modify external recipes/Kitchen plugins, launchd scheduling, plugin activation policy, dependency pins, or broad startup architecture in this PR; call out that startup/fan-out profiling remains follow-up work.
