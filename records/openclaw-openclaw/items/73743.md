## Summary

Keep open. Current main still allows timeout-killed subprocesses to reach plugin callers with `code: null`, so the downstream cache-poisoning path remains actionable; the broader macOS recipes/Kitchen startup fan-out also is not proven fixed from this checkout.

## What This Changes

The issue reports slow OpenClaw plugin CLI subprocess startup under macOS launchd fan-out and downstream Kitchen cache poisoning when timed-out subprocesses return partial stdout as success.

## Best Possible Solution

Keep this as an active reliability issue until timeout-killed subprocesses always report a numeric non-zero failure to plugin callers, then profile the recipes/Kitchen command and worker path on macOS to pick the smallest startup or scheduling fix.

## Repair Work Prompt

Fix the core timeout-result contract behind the reliability report at https://github.com/openclaw/openclaw/issues/73743. Keep the change narrow: in `src/process/exec.ts`, ensure `runCommandWithTimeout` returns a numeric non-zero `code` for `termination === "timeout"` and `termination === "no-output-timeout"` even when Node reports null exit code/signal state after a timeout kill. Preserve the existing Windows shim success behavior for non-timeout cases and do not change plugin CLI startup/loading policy in this PR. Add focused regression tests so timeout and no-output-timeout cases assert a numeric non-zero code, preferably `124` when no better child exit code is available, and keep plugin runtime forwarding behavior covered. Add a single-line `CHANGELOG.md` fix entry because this is user-facing reliability behavior. Do not attempt to solve macOS launchd jitter, recipes/Kitchen worker design, or plugin startup profiling in this repair PR.
