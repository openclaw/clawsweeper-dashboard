## Summary

Keep open. Current main still lets shared subprocess timeouts surface to plugin callers with `termination: "timeout"` or `"no-output-timeout"` while `code` can remain `null`, so the reported cache-poisoning precondition is still present even though recent lazy CLI registration work may reduce some startup fan-out cost.

## What This Changes

The issue requests fixing a macOS OpenClaw CLI startup and launchd fan-out regression where slow recipes/Kitchen subprocesses time out and cache partial output as success.

## Best Possible Solution

Harden shared subprocess timeout results first, then profile the macOS recipes/Kitchen launchd path and make measured startup or worker changes based on that data.

## Reproduction Assessment

Yes at report/source level. The issue gives a concrete macOS launchd reproduction path, and current source verifies the nullable timeout-result precondition; I did not live-run the 21-plist macOS setup.

## Solution Assessment

Unclear for the full performance regression. The narrowest maintainable first fix is to make timeout and no-output-timeout subprocess results unambiguously fail for plugin callers, while startup slimming and launchd scheduling wait for profiling evidence.
