## Summary

Keep open. Current main has relevant startup mitigations, but it does not prove the reported macOS recipes/Kitchen regression is fixed, and the in-repo subprocess timeout path can still return a null code on timeout, which matches the cache-poisoning failure mode described in the issue.

## What This Changes

The issue requests fixes for slow OpenClaw CLI subprocess startup under macOS launchd fan-out and safer timeout handling so partial CLI output is not cached as successful recipe data.

## Best Possible Solution

Keep the issue open as a reliability investigation. The immediate in-repo end state should make timeout and no-output-timeout subprocess results unambiguously failed for numeric-code callers, while maintainer/plugin follow-up profiles the recipes CLI path and decides whether descriptor-backed, gateway-backed, or scheduling changes are needed.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with concrete environment, timings, launchd shape, process snapshots, and cache observations; current-main source review also reproduces the core timeout-result ambiguity without needing the external recipes plugin.

## Solution Assessment

Unclear as a full fix. A narrow core timeout-code hardening is the safest automated repair, but the broader startup regression likely needs maintainer-led profiling of the recipes plugin/app path and launchd deployment behavior.

## Repair Work Prompt

Fix the in-repo timeout-result hardening for https://github.com/openclaw/openclaw/issues/73743 only. In `src/process/exec.ts`, make `runCommandWithTimeout` return a stable nonzero numeric `code` for both `termination: "timeout"` and `termination: "no-output-timeout"` even when Node reports a null/undefined exit code after the timeout SIGKILL path. Prefer the existing timeout convention `124`; preserve partial stdout/stderr capture plus the existing `termination` and `noOutputTimedOut` fields. Add focused tests in `src/process/exec.test.ts` and `src/process/exec.no-output-timer.test.ts` that timeout and no-output-timeout cases with null exit state produce `code: 124`, not `null`. Update Windows coverage only if the shared helper behavior requires it, and keep plugin runtime forwarding behavior compatible. Add a single-line `CHANGELOG.md` Fixes entry thanking `@rjdjohnston`. Do not change external recipes/Kitchen plugin behavior, launchd scheduling, plugin CLI activation planning, runtime-deps staging, or add startup profiling in this PR.
