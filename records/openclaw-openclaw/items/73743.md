## Summary

Keep open. Current main still allows timeout-killed subprocess results to reach trusted plugin callers with `code: null`, so the cache-poisoning failure mode remains actionable; the broader recipes/Kitchen macOS fan-out path also lacks source or live-profile proof of being fixed.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup on macOS under launchd fan-out, plus downstream ClawKitchen cache poisoning when timed-out subprocess output is treated as success.

## Best Possible Solution

Keep this as an active reliability issue: first harden timeout-killed subprocess results to a numeric non-zero failure for plugin callers, then profile the recipes/Kitchen command and worker path on macOS before choosing startup slimming, a lighter worker RPC, or launchd staggering.

## Reproduction Assessment

Yes at report level. The issue includes concrete commands, timings, environment, process snapshots, and cache symptoms; this checkout independently verifies the shared timeout-result path but does not live-reproduce the external macOS recipes/Kitchen setup.

## Solution Assessment

Unclear for the full issue. Core timeout-result hardening is the narrowest safe first fix, while the startup fan-out component still needs plugin/live-host profiling before selecting a maintainable implementation.

## Repair Work Prompt

Fix the first-stage reliability bug from https://github.com/openclaw/openclaw/issues/73743: `runCommandWithTimeout` can return `code: null` after timeout or no-output-timeout kills, and `runtime.system.runCommandWithTimeout` forwards that raw result to trusted plugins, allowing callers to treat partial stdout as success. Scope the fix to timeout/no-output-timeout subprocess results: ensure those terminations return a numeric non-zero code, preferably `124` when no numeric code is available, while preserving ordinary non-timeout signal semantics. Add regression coverage showing timeout and no-output-timeout kills do not return null, and add plugin runtime coverage if the runtime boundary remains direct. Do not attempt to redesign recipes/Kitchen startup, worker-tick execution, launchd jitter, or plugin CLI activation in this PR.
