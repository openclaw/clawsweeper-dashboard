## Summary

Keep this open: current `main` and the latest shipped release still allow timeout/no-output-timeout subprocess results to reach plugin callers with `code: null`, so the cache-poisoning precondition described in the issue remains source-reproducible even though the broader macOS launchd fan-out was not live-reproduced here.

## What This Changes

The issue reports slow `openclaw` CLI subprocess startup on macOS that compounds under launchd fan-out and can poison ClawKitchen subprocess caches after timeouts.

## Best Possible Solution

Harden plugin-facing timeout and no-output-timeout subprocess results first, then profile the macOS recipes/Kitchen launchd startup path before changing worker scheduling or CLI architecture.

## Reproduction Assessment

Yes at source level: the report gives a concrete macOS launchd reproduction path, and current source plus latest release code still allow nullable timeout results to reach plugin callers. I did not live-run the 21-plist macOS setup or the external recipes/Kitchen plugins.

## Solution Assessment

Yes for the narrow first repair: making timeout and no-output-timeout subprocess results unambiguously fail for plugin callers is the smallest maintainable fix for the cache-poisoning path. Worker RPC changes, launchd jitter, and broader CLI startup architecture should wait for measured current-main evidence.
