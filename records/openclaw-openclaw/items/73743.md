## What This Changes

The issue reports a regression where slow `openclaw` CLI subprocess startup and launchd fan-out make workflow ticks stack up, causing ClawKitchen subprocess cache entries to store partial timeout output as successful recipe data.

## Best Possible Solution

Make plugin-facing subprocess timeout results unambiguously failure-valued, then use a real macOS recipes/ClawKitchen setup to profile whether core needs a narrower CLI plugin-load or worker-tick fast path.

## Repair Work Prompt

Fix the source-backed timeout-poisoning path from https://github.com/openclaw/openclaw/issues/73743. In current main, `src/plugins/runtime/runtime-system.ts` exposes `src/process/exec.ts` `runCommandWithTimeout` directly; timeout and no-output-timeout results can still return `code: null`, so external plugin callers that default non-number codes to 0 can cache partial stdout as success. Make timeout and no-output-timeout results exposed through `runtime.system.runCommandWithTimeout` unambiguously failure-valued without breaking abort/signal semantics; prefer a narrow change in `src/process/exec.ts` or a plugin-runtime adapter rather than adding config. Add focused Vitest coverage showing global timeout and no-output timeout return a non-zero numeric code plus termination metadata through the plugin runtime. Do not attempt the broader macOS launchd/recipes startup profiling or add jitter/scheduler features in this PR. Do not edit `CHANGELOG.md`.
