## Summary

Current `main` still leaves the timeout-result hardening path open: a timeout or no-output timeout can produce `termination: "timeout"`/`"no-output-timeout"` with `code: null`, and the plugin runtime still exposes that raw result to plugins. The broader macOS launchd startup regression is not proven fixed by the lazy plugin CLI work on `main`, and the external Kitchen/recipes plugins are not present in this checkout to verify their cache behavior directly.

## What This Changes

The issue reports slow OpenClaw CLI startup under macOS launchd workflow fan-out and downstream Kitchen cache poisoning when timed-out `openclaw recipes list` subprocess output is cached as success.

## Best Possible Solution

First harden plugin-facing timeout/no-output-timeout subprocess results so they cannot be interpreted as success, then separately profile the macOS recipes/Kitchen launchd startup path before changing worker scheduling or CLI architecture.

## Reproduction Assessment

Yes at source level: the report gives a concrete macOS launchd reproduction path, and current source still verifies the nullable timeout-result precondition. I did not live-run the 21-plist macOS setup or external Kitchen/recipes plugins in this checkout.

## Solution Assessment

Yes for the narrow first repair: make timeout and no-output-timeout subprocess results unambiguously fail for plugin callers before taking on the broader performance profiling. Worker RPC changes, launchd jitter, or scheduling guidance should wait for measured current-main evidence.
