## Summary

Keep open. Current main includes useful startup and runtime-dependency mitigations, but it does not prove the reported macOS recipes/Kitchen fan-out regression is fixed, and the core subprocess timeout path can still surface a null code on timeout, preserving the caller misclassification risk described in the issue.

## What This Changes

The issue requests reducing slow `openclaw` CLI subprocess startup and launchd worker fan-out, plus preventing ClawKitchen from caching partial timeout output as successful recipe JSON.

## Best Possible Solution

Keep this as an active reliability investigation: first harden core subprocess timeout results so timeout kills always look like numeric failures to plugin callers, then separately profile and lighten the recipes/Kitchen worker/list startup path and deployment fan-out behavior.

## Repair Work Prompt

Fix the core subprocess timeout-result misclassification described in https://github.com/openclaw/openclaw/issues/73743. In `src/process/exec.ts`, ensure `runCommandWithTimeout` returns a nonzero numeric code, preferably 124, whenever `termination` is `timeout` or `no-output-timeout` and the resolved child code is null or 0. Preserve normal exit/signal behavior and the Windows shim success semantics when no timeout kill was issued. Add regression coverage that fails on current main: mocked global timeout and no-output timeout cases with null child codes must not return null, and should return the chosen numeric timeout code. Update adjacent tests if necessary. Do not attempt to solve recipes/Kitchen startup, launchd jitter, or plugin CLI architecture in this PR. Add a single-line `CHANGELOG.md` fix entry under the active version crediting `Thanks @rjdjohnston`.
