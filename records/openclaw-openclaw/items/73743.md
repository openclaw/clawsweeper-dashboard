## Summary

Keep open. Current main still allows timeout-killed subprocesses to surface `code: null` through the plugin runtime, so the Kitchen cache-poisoning path remains plausible, and the macOS launchd startup fan-out regression still needs profiling rather than cleanup closure.

## What This Changes

The issue reports slow macOS `openclaw` CLI subprocess startup that compounds under launchd workflow fan-out and can poison a plugin subprocess cache after timeouts.

## Best Possible Solution

Normalize timeout and no-output-timeout subprocess results to an explicit failed outcome for plugin callers, then profile the recipes/Kitchen startup path under the reported launchd fan-out before choosing plugin activation, worker RPC, or scheduling changes.

## Repair Work Prompt

Fix the in-repo subprocess timeout contract for https://github.com/openclaw/openclaw/issues/73743. Make `runCommandWithTimeout` return an explicit failed numeric code and/or typed timeout signal for `termination: "timeout"` and `termination: "no-output-timeout"` even when Node reports no child exit code, so plugin callers cannot accidentally treat timeout-killed subprocesses as success by defaulting `null`/`undefined` to `0`. Keep the change in the shared process helper and its plugin-runtime contract; do not attempt recipes/Kitchen startup architecture, launchd scheduling, or external plugin rewrites in this PR. Add focused regression tests for both global timeout and no-output timeout missing-code cases, preserve Windows clean-close behavior, and add a CHANGELOG fix entry unless maintainers classify the final patch as purely internal.
