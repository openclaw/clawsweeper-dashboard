## Summary

Keep open. Current main still has a concrete subprocess timeout-result path that can return `code: null` after a timeout kill, and plugin callers receive that raw result; the broader macOS recipes/Kitchen launchd fan-out behavior is not proven fixed by current code.

## What This Changes

The issue reports a regression where slow `openclaw` CLI subprocess startup plus launchd fan-out makes recipes worker ticks pile up and lets Kitchen cache partial timeout output as a successful recipes response.

## Best Possible Solution

Keep this as an active reliability issue: first land a focused core fix so subprocess timeouts always surface as numeric non-zero failures to plugin callers, then profile and optimize the recipes/Kitchen startup path and launchd scheduling separately if the fan-out remains.

## Reproduction Assessment

Yes. The issue has a high-confidence report-level reproduction with commands, timings, environment, process snapshots, and cache observations; current source inspection also verifies the narrow timeout-code misclassification path without needing a live macOS run.

## Solution Assessment

Unclear for the full issue. The narrow core timeout hardening is the safest first repair, while the startup-cost and launchd-jitter pieces need targeted profiling around the external recipes/Kitchen plugin workflow.
