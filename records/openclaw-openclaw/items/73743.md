## Summary

Keep open. Current main still allows timeout-killed subprocesses to surface through the plugin runtime with `code: null`, so the cache-poisoning failure mode described by the report is not fixed; the broader recipes/Kitchen macOS startup fan-out also cannot be proven resolved from this checkout because that plugin surface is not present in the current tree.

## What This Changes

The issue asks OpenClaw to reduce slow CLI/plugin startup and launchd workflow fan-out for recipes/Kitchen, and to prevent timeout-killed subprocess output from being cached as a successful recipes list.

## Best Possible Solution

Keep this as an active reliability issue until timeout-killed subprocesses always report a numeric non-zero failure to plugin callers, then profile the recipes/Kitchen command and worker path on macOS to choose the smallest startup fix, such as command-specific activation narrowing, a lighter worker execution path, or documented launchd staggering if fan-out remains necessary.

## Reproduction Assessment

Yes at report level. The issue provides concrete commands, timings, environment, process snapshots, and cache symptoms; this checkout verifies the shared timeout-result path but does not live-reproduce the external macOS recipes/Kitchen setup.

## Solution Assessment

Unclear for the full issue. Core timeout-result hardening is the narrowest safe first fix, while startup slimming and launchd jitter need plugin/live-host profiling before selecting a maintainable implementation.
