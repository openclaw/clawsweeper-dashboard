## Summary

Keep open. Current main still allows timeout-killed subprocesses to return `code: null` through the plugin runtime, so the Kitchen cache-poisoning path remains plausible, and the broader recipes/Kitchen startup regression still needs live macOS profiling.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup and launchd worker fan-out causing ClawKitchen recipe cache poisoning after subprocess timeouts.

## Best Possible Solution

Normalize timeout and no-output-timeout subprocess results for plugin callers, then profile the recipes/Kitchen startup path under launchd before choosing activation, worker RPC, or scheduling changes.

Yes for the shared timeout-result precondition: source inspection shows timeout-killed commands can still surface `code: null` to plugin runtime callers. The full macOS launchd fan-out reproduction remains external to this checkout but is well specified in the report.

## Solution Assessment

Unclear for the full issue. The narrowest maintainable first repair is core timeout-result hardening; startup slimming, worker RPC changes, or launchd jitter should wait for profiling from the affected plugin and host setup.

## Repair Work Prompt

Fix openclaw/openclaw#73743 narrowly by hardening shared subprocess timeout results. Current main can return `code: null` from `runCommandWithTimeout` when `termination` is `timeout` or `no-output-timeout`, and `runtime.system.runCommandWithTimeout` forwards that raw result to plugin callers. Update `src/process/exec.ts` so timeout and no-output-timeout results always carry a nonzero numeric code, preserving the Windows success shim only for non-timeout exits. Add or update focused tests so timeout and no-output-timeout paths assert a numeric nonzero code, and keep plugin runtime forwarding behavior covered. Do not attempt broad macOS launchd jitter, recipes worker RPC, plugin startup optimization, or external Kitchen plugin changes in this PR.
