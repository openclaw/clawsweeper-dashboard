## Summary

Keep open. Current main has adjacent plugin-startup mitigations, but it still does not prove the reported macOS recipes/Kitchen launchd fan-out regression is fixed, and the shared subprocess timeout path can still surface a null exit code to plugin callers after timeout kills.

## What This Changes

The issue asks OpenClaw to reduce slow `openclaw recipes ...` subprocess startup and launchd fan-out, and to prevent Kitchen from caching timed-out partial CLI output as a successful `recipes list` result.

## Best Possible Solution

Keep this as an active reliability issue: first land a small core subprocess timeout hardening patch so timeout kills always produce numeric non-zero failures for plugin callers, then profile and optimize the recipes/Kitchen command path and launchd scheduling behavior with the external plugin sources and a macOS reproduction host.

## Reproduction Assessment

Yes. The report gives a high-confidence reproduction path with specific commands, versions, timings, process snapshots, and cache observations; this read-only checkout verifies the relevant core timeout surface but cannot live-measure the external macOS recipes/Kitchen environment.

## Solution Assessment

Unclear for the full issue. The narrowest safe first step is to harden `src/process/exec.ts` and plugin-runtime tests against nullable timeout codes; startup profiling, lightweight worker execution, and launchd jitter need separate follow-up after that repair.

## Repair Work Prompt

Fix the core subprocess timeout result misclassification behind https://github.com/openclaw/openclaw/issues/73743. Ensure `runCommandWithTimeout` returns a numeric non-zero code, preferably 124, whenever `termination` is `timeout` or `no-output-timeout`, even when the child close/exit code and signal resolve to null. Add focused regression tests proving both overall timeout and no-output timeout with null child code cannot produce `code: null`, and update plugin runtime coverage if needed so `runtime.system.runCommandWithTimeout` callers receive the hardened result. Do not attempt broad CLI startup refactors, recipes/Kitchen plugin rewrites, launchd jitter policy, or external plugin changes in this PR. Add a single-line `CHANGELOG.md` fix entry thanking @rjdjohnston. Validate with the targeted process/plugin runtime tests, targeted oxfmt check, and the changed gate per repo policy.
