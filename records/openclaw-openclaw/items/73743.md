## Summary

Keep open. Current main has relevant startup/runtime-deps mitigations, but it does not prove the reported macOS recipes/Kitchen regression is fixed, and the core subprocess timeout path can still return a null numeric code on timeout, preserving the caller-misclassification risk described in the report.

## What This Changes

The issue requests investigation and repair for slow `openclaw recipes` CLI startup under macOS launchd fan-out, plus hardening so timed-out subprocess output cannot be cached as successful Kitchen recipe data.

## Best Possible Solution

Keep this as an active reliability investigation: harden core subprocess timeout results so timeout kills always look like failures to plugin callers, then profile the external recipes CLI path on macOS and move heavy worker/list operations toward descriptor-backed or Gateway-backed lightweight execution where the plugin API already supports it.

Yes. The issue has a high-confidence report-level reproduction path with concrete environment, commands, timings, process snapshots, and cache observations; this checkout can verify the relevant core surfaces but cannot remeasure the external recipes/Kitchen flow.
