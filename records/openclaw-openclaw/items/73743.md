## Repair Work Prompt

Fix the shared subprocess timeout result contract for this issue. On current main, `runCommandWithTimeout` can return `code: null` for `termination: "timeout"` or `"no-output-timeout"`, and plugin callers receive that through `runtime.system.runCommandWithTimeout`, which lets callers treat timeout-killed subprocesses as success if they default nullish codes to zero. Change the core helper so timeout and no-output-timeout results always have a non-zero numeric code, preferably `124`, while preserving existing Windows clean-close behavior for non-timeout exits. Add or update focused tests in `src/process/exec.test.ts`, `src/process/exec.no-output-timer.test.ts`, `src/process/exec.windows.test.ts`, and plugin runtime forwarding tests as needed. Include a `CHANGELOG.md` fix entry because this affects user-visible plugin/CLI reliability. Do not attempt the broader recipes/Kitchen startup architecture, worker RPC, launchd jitter, or plugin activation refactor in this PR; leave that for profiling after the contract is hardened.
