## Summary

Keep open. Current main contains related CLI/runtime-dependency startup mitigations, but it does not prove the reported macOS Recipes/Kitchen regression is fixed, and the core subprocess timeout path can still surface a timeout with a nullable exit code that callers may misclassify as success.

## What This Changes

The issue asks OpenClaw to investigate and fix slow CLI subprocess startup and launchd worker fan-out that causes Recipes/Kitchen subprocess timeouts and cache poisoning after the 2026.4.26 upgrade.

## Best Possible Solution

Keep this as an active reliability investigation: first harden the core subprocess timeout contract so timeout kills always look like failures to plugin callers, then profile the Recipes CLI path on macOS and move heavy worker/list operations toward a lightweight descriptor-backed or Gateway-backed execution path where the plugin API supports it.

Yes, at report level. The issue gives concrete commands, environment, timings, process snapshots, and cache observations; this checkout can verify the core timeout surface but cannot live-reproduce the external macOS Recipes/Kitchen fan-out path.

## Solution Assessment

## Repair Work Prompt

Fix the core subprocess timeout misclassification described in https://github.com/openclaw/openclaw/issues/73743. In `src/process/exec.ts`, make `runCommandWithTimeout` return a non-zero numeric code, preferably `124`, for `termination === "timeout"` and `termination === "no-output-timeout"` even when `resolveProcessExitCode(...)` returns null. Preserve the existing signal, `killed`, `termination`, Windows shim, and no-output-timeout behavior. Update focused tests so timeout and no-output-timeout cases assert the numeric failure code rather than only `not.toBe(0)`. Add a user-facing `CHANGELOG.md` fix entry with reporter attribution. Do not attempt the broader Recipes/Kitchen startup redesign, launchd jitter policy, or plugin-specific worker RPC changes in this PR.
