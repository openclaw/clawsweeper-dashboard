## Summary

Keep open. Current main still lets timeout-killed subprocesses surface through the plugin runtime with `code: null`, so the timeout-output cache-poisoning path remains; lazy plugin CLI activation exists, but it is not enough evidence that the macOS recipes/Kitchen fan-out regression is fixed.

## What This Changes

The issue asks OpenClaw to reduce slow macOS CLI subprocess startup and launchd worker fan-out, and to prevent Kitchen from caching timeout-truncated `recipes list` output as successful JSON.

## Best Possible Solution

Keep the issue active until timeout-killed subprocesses always report a numeric non-zero failure to plugin callers, then profile the recipes/Kitchen macOS command path to select an evidence-backed startup or scheduling fix.

Yes at report level. The issue gives concrete commands, timings, environment, process snapshots, and cache symptoms; current source inspection verifies the shared timeout-result path but does not live-reproduce the external macOS launchd setup.
