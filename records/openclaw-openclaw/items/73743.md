## Summary

Keep open. Current main has relevant CLI and plugin runtime-deps startup mitigations, but they do not prove the reported macOS recipes/Kitchen fan-out regression is fixed, and the core subprocess timeout result can still expose a null exit code to plugin callers.

## What This Changes

The issue asks OpenClaw to reduce slow `openclaw recipes` subprocess startup under launchd fan-out and prevent ClawKitchen from caching timed-out partial CLI output as a successful recipes response.

## Best Possible Solution

Keep this as an active reliability investigation: first harden `runCommandWithTimeout` so timeout kills always look like numeric failures to plugin callers, then profile the recipes plugin CLI path on macOS and move list/worker-tick work toward descriptor-backed or Gateway-backed lightweight execution where feasible.

## Reproduction Assessment

Yes. The report gives a high-confidence reproduction path with concrete environment, launchd shape, commands, timings, process snapshots, and cache observations; this checkout can verify the core timeout and CLI-loading surfaces but cannot run the external macOS recipes/Kitchen fleet.

## Solution Assessment

Unclear as a full fix. Current startup mitigations are directionally useful but incomplete for this issue; the narrowest maintainable in-repo first step is subprocess timeout-result hardening, with separate profiling for recipes startup and launchd fan-out.

## Repair Work Prompt

Fix the narrow subprocess timeout-result bug described by https://github.com/openclaw/openclaw/issues/73743. In `src/process/exec.ts`, make `runCommandWithTimeout` return a numeric non-zero exit code, preferably `124`, for both `termination: "timeout"` and `termination: "no-output-timeout"` when this process issued the timeout kill and the resolved process code is `0` or `null`. Preserve the existing Windows shim behavior for non-timeout successful commands. Update the focused tests so timeout and no-output-timeout cases assert the exact numeric failure code instead of only `not.toBe(0)`, including `src/process/exec.test.ts` and `src/process/exec.no-output-timer.test.ts`; keep Windows coverage passing in `src/process/exec.windows.test.ts` if touched. Add a single-line user-facing changelog entry under the active Fixes section that references this issue without claiming the broader macOS recipes startup regression is fully fixed. Do not change plugin CLI startup policy, recipes/Kitchen behavior, launchd scheduling, runtime-deps installation policy, or external plugin APIs in this repair PR.
