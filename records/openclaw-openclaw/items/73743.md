## Summary

Keep open. Current main still allows timeout-killed subprocesses to reach plugin callers with `termination: "timeout"` or `"no-output-timeout"` and `code: null`, so the Kitchen cache-poisoning path remains plausible while the broader macOS launchd startup/fan-out regression has not been proven fixed.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup under macOS launchd fan-out, causing recipes/Kitchen subprocess timeouts to be cached as successful partial output.

## Best Possible Solution

Make timeout and no-output-timeout subprocess results unambiguously fail for plugin callers first, then profile the recipes/Kitchen launchd startup path before choosing any worker or scheduling redesign.

## Repair Work Prompt

Fix the narrow subprocess timeout failure path for https://github.com/openclaw/openclaw/issues/73743. On current main, `runCommandWithTimeout` can return `termination: "timeout"` or `"no-output-timeout"` with `code: null`, and `runtime.system.runCommandWithTimeout` forwards that raw result to plugins. Change the shared process/runtime contract so timeout and no-output-timeout results are unambiguously failed for plugin callers, preferably by returning a stable non-zero numeric code such as 124 for timeout-killed runs even when Node reports no exit code. Add focused regression coverage for overall timeout, no-output timeout, and Windows timeout behavior; update plugin runtime tests if the exposed shape changes. Do not implement launchd jitter, recipes worker RPC redesign, plugin cache changes outside this repo, or speculative CLI startup profiling fixes in this PR. Add a concise `CHANGELOG.md` fix entry because this affects user-visible plugin subprocess behavior.
