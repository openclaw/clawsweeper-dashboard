## Summary

Close as implemented on current main. The reported disabled-state path is now covered at both boundaries: `active-memory` is a standalone plugin rather than a `memory-core` submodule, entry-level `enabled:false` disables plugin activation, and the hook refreshes live config before returning without launching embedded recall when `config.enabled:false`; broader enabled-recall timeout work can stay in the existing Active Memory timeout tracker.

## What This Changes

Make Active Memory skip all blocking recall when the `active-memory` plugin entry or plugin config is disabled, even while `memory-core` remains enabled.

## Best Possible Solution

Keep the current main behavior: `active-memory` remains a separate plugin with entry-level disablement and a live `config.enabled` hook guard; broader timeout/fail-open behavior should remain with the existing Active Memory timeout tracker.

## Reproduction Assessment

No current failing reproduction remains. Source inspection and checked-in tests show the reported disabled config returns before any embedded Active Memory recall run is started.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this report because it honors both plugin-entry disablement and the runtime global pause without moving Active Memory into `memory-core`.
