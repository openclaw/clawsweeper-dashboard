## Summary

Keep open: current main still leaves non-image plugin runtime tools on the 30s Codex dynamic-tool default unless the tool call itself carries `timeoutMs`, while the public plugin SDK, runtime registration options, manifest metadata, and plugin-entry config do not expose a host-level per-tool timeout contract.

## What This Changes

The issue asks OpenClaw to let plugin runtime tools declare or configure a longer Codex dynamic-tool watchdog timeout instead of being limited to 30s.

## Best Possible Solution

Add a generic plugin-tool timeout contract that external plugins can use, honor it in the Codex dynamic-tool bridge with the existing 600s cap, document it, test it, and preserve the 30s default for tools without a timeout.

## Reproduction Assessment

Yes at source level, but not by executing the external plugin in this review. Current main resolves non-image plugin dynamic tools to call-argument `timeoutMs` or the 30s default, and the reporter supplied a concrete 2026.5.18 timeout log for the affected plugin.

## Solution Assessment

Unclear until maintainers choose the public contract shape. The current implementation is not the best end state for long-running plugin tools; the maintainable fix should be a generic plugin-facing timeout contract rather than another tool-name special case.
