- `impact:session-state`: The reported timeout returns failure to the parent turn while the plugin's internal delegation can continue and affect later tool/run state.

## Summary

Keep open: current main still leaves non-image plugin runtime tools on the 30s Codex dynamic-tool default unless the model supplies a per-call `timeoutMs`, and neither `defineToolPlugin` nor `api.registerTool` exposes a plugin-owned timeout hint.

## What This Changes

Request a plugin runtime tool timeout contract so long-running tools registered through `api.registerTool` or `defineToolPlugin` can exceed the Codex dynamic-tool 30s default.

## Best Possible Solution

Add a generic plugin-tool timeout contract, honor it in the Codex dynamic-tool bridge up to the 600s cap, document the precedence, and cover default plus long-running plugin-tool behavior with focused tests.

## Reproduction Assessment

Yes, at source level. Current main routes non-image plugin dynamic tools through the 30s fallback unless the model supplies `timeoutMs`, and the reporter supplied a concrete 2026.5.18 timeout log for the affected plugin.

## Solution Assessment

Unclear until maintainers choose the public contract shape. The maintainable fix should be generic plugin-facing timeout metadata, operator override config, or both, not another tool-name special case.
