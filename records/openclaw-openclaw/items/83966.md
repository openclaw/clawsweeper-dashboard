## Summary

Keep open: current main and the latest release still only provide a generic Codex dynamic-tool timeout plus a few hard-coded tool defaults, while public plugin runtime tools cannot declare or configure their own watchdog budget.

## What This Changes

Allow plugin runtime tools to declare or configure a Codex dynamic-tool watchdog timeout above the generic default.

## Best Possible Solution

Add one maintainer-approved generic plugin tool timeout contract, preferably plugin-owned metadata honored by the Codex bridge up to the existing 600s cap, with docs and focused SDK/bridge tests instead of more tool-name special cases.

## Reproduction Assessment

Yes at source level: plugin tools cannot declare a timeout in defineToolPlugin, generated metadata, or registerTool, and the Codex bridge falls back to 90s unless the call itself or a built-in tool special case supplies a timeout. I did not run the external lossless-claw live repro.

## Solution Assessment

No, current main is not the best final solution because raising the generic default and special-casing built-in tools still leaves plugin-owned long-running tools without a stable contract. The maintainable fix is a generic SDK/manifest/runtime timeout seam or an explicitly approved config contract.
