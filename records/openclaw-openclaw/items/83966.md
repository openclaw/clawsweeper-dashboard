## Summary

Keep open: current main still leaves non-image plugin runtime tools on the 30s Codex dynamic-tool bridge default unless the model sends a `timeoutMs` argument, and the plugin SDK/manifest contract has no host-level per-tool timeout metadata. Likely related people are `steipete` (medium, recent Codex dynamic-tool work and related timeout triage), Kevin Lin (medium, recent Codex app-server/plugin invocation work), Josh Avant (medium, recent Codex runtime work), and nitinjwadhawan (low, current checkout blame baseline for the exact timeout resolver).

## What This Changes

Request a public plugin-tool timeout contract so Codex dynamic tools registered by plugins can exceed the 30s bridge default up to the existing 600s cap.

## Best Possible Solution

Add a generic plugin-tool timeout contract that is available to external plugins, honored by the Codex dynamic-tool bridge, clamped to 600s, documented, and covered by regression tests while preserving the 30s default for tools without a timeout.

## Reproduction Assessment

Yes at source level, but not by executing the external plugin in this review. Current main resolves non-image plugin dynamic tools to call-argument `timeoutMs` or the 30s default, and the reporter supplied a concrete 2026.5.18 timeout log for the affected plugin.

## Solution Assessment

Unclear until maintainers choose the public contract shape. The current implementation is not the best end state for long-running plugin tools; the maintainable fix should be a generic plugin-facing timeout contract rather than another tool-name special case.
