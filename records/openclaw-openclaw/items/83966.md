## Summary

Keep open: current main and v2026.6.6 partially address the original 30s pain by using a 90s generic Codex dynamic-tool watchdog, but plugin runtime tools still cannot declare a host-level timeout through `defineToolPlugin`, generated tool metadata, or `api.registerTool(...)`, and no canonical duplicate now owns that plugin SDK/config decision.

## What This Changes

The issue requests a plugin-facing per-tool timeout contract or documented host configuration so long-running plugin runtime tools can use the existing 600s Codex dynamic-tool cap.

## Best Possible Solution

Design one generic plugin-tool timeout contract, clamp it to the existing 600s cap, document it for plugin authors/operators, and cover the Codex bridge plus SDK metadata path with focused tests.

## Reproduction Assessment

Yes, at source level. Current main still has no plugin-authored timeout metadata or registration option, and a non-special-cased plugin tool can only get a longer Codex watchdog if the model sends `timeoutMs` in call arguments.

## Solution Assessment

Unclear until maintainers choose the public contract shape. The current 90s default is a useful partial fix, but the maintainable solution should be a generic plugin-facing timeout contract rather than another tool-name special case.
