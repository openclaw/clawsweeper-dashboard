## Label Justifications

- `P2`: The issue reports a real plugin runtime limitation with source-confirmed behavior, but it is bounded to Codex dynamic tools and needs public API design rather than emergency handling.
- `impact:session-state`: The reported timeout makes the parent turn lose the tool result while plugin work can continue in the background, creating tool/run state drift.

## Summary

Keep open: current main and v2026.5.18 still only honor per-call `timeoutMs` plus the `image_generate` and media `image` special cases, while plugin SDK tool metadata and `api.registerTool` options do not expose a per-tool default timeout. This is a real public SDK/config gap, but it needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue requests a public way for plugin runtime tools to declare or configure longer Codex dynamic-tool watchdog timeouts.

## Best Possible Solution

Add a generic, documented plugin-tool timeout contract that the Codex dynamic-tool bridge honors up to the existing 600000 ms cap, with SDK/manifest docs and regression coverage for both `defineToolPlugin` and direct `api.registerTool` paths.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.5.18 route plugin dynamic tools without a generic timeout metadata/config path, so calls without a positive `timeoutMs` argument fall back to the 30000 ms watchdog unless they are the two special-cased image tools.

## Solution Assessment

Yes on direction, but not as a one-off plugin workaround. The maintainable fix is a public SDK/manifest/config contract that the Codex bridge consumes, with docs and tests, and maintainers need to choose that contract before implementation.
