## Summary

Keep open: current main and v2026.6.8 still only support per-call `timeoutMs`, image/media/message special cases, and the 90s generic Codex dynamic-tool default; plugin runtime tools still cannot declare a host-level timeout through the SDK or `api.registerTool`, and no canonical related PR owns that generic plugin contract decision.

## What This Changes

The issue asks for plugin runtime tools to declare or configure a Codex dynamic-tool timeout above the current default watchdog.

## Best Possible Solution

Design one generic plugin-tool timeout contract, clamp it to the existing 600s cap, document it, and cover both static `defineToolPlugin` metadata and direct runtime registration with focused Codex bridge tests.

## Reproduction Assessment

Yes, at source level. Current main still has no plugin-authored timeout metadata or registration option, so a non-special-cased plugin tool cannot declare a host-level timeout except by putting `timeoutMs` in model-call arguments.

## Solution Assessment

Unclear until maintainers choose the public contract shape. A generic plugin-facing timeout contract is more maintainable than another tool-name special case, but whether it belongs in metadata, operator config, or both is a product/API decision.
