## Summary

Keep this PR open. Current main still lacks an inbound Slack media path that can call `files.info` when file event payloads arrive without private download URLs, so the PR is still addressing real unimplemented behavior; however, the patch needs maintainer follow-up because it only refreshes metadata when URLs are absent and the branch is behind current Slack media changes.

## What This Changes

The PR adds an optional Slack WebClient to inbound Slack media resolution, threads it through message and thread-starter preparation, and adds tests for a `files.info` fallback when Slack file objects lack private URLs.

## Best Possible Solution

Update this PR or a narrow successor so the Slack plugin passes the WebClient into inbound direct, forwarded attachment, and thread-starter media resolution, refreshes metadata for missing URLs, and either retries stale/failed event URLs with fresh `files.info` metadata or explicitly scopes that as a separate maintainer-approved follow-up. Keep the existing Slack URL allowlist, authenticated redirect handling, SSRF policy, size/time limits, and add focused regression tests plus useful verbose diagnostics.

## Reproduction Assessment

Yes. A focused unit-level reproduction is visible from current main: call `resolveSlackMedia` with a file that has an `id` but no `url_private_download` or `url_private`, and it returns `null`; the inbound preparation path also lacks any client parameter that could call `files.info`.

## Solution Assessment

No, not as-is. The missing-URL fallback is the right narrow direction, but the best maintainable fix should be rebased onto current Slack media code and should address or consciously defer the stale event URL retry gap flagged in review discussion.
