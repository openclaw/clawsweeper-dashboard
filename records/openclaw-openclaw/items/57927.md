## Summary

This PR should stay open because the proposed sweep can still delete start data for an active long-running tool before its end handler runs. The duplicate utility imports are low-risk cleanup, but the tool-state sweep has a blocking session/message-state correctness risk and the external PR still lacks real behavior proof.

## What This Changes

The branch replaces local `clamp` and `isRecord` helpers with shared utility imports and adds a timestamp-based idle sweep for `toolStartData` in embedded tool execution tracking.

## Best Possible Solution

Track active in-flight tool calls per run, sweep only runs with no active calls after the idle window, and add focused regression coverage plus redacted runtime proof for both abandoned-run cleanup and long-running active tool preservation.

## Reproduction Assessment

Yes for the proposed-patch defect from source inspection: start one tool, let it run beyond the five-minute idle threshold, then start another tool so the sweep deletes the first tool's start record before its end handler runs. I did not run a live scenario because this review is read-only.

## Solution Assessment

No: the duplicate utility cleanup is fine, but timestamp-only run activity is not the best way to fix orphaned `toolStartData` because active tools do not heartbeat while running. The safer solution is to track active in-flight calls and sweep only inactive runs.
