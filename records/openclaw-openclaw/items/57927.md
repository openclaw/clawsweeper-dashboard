## Summary

Keep open: the utility cleanup is low risk, but the proposed `toolStartData` sweep can evict metadata for an active long-running tool call, and the external PR still lacks real after-fix behavior proof.

## What This Changes

This PR imports shared utility helpers and adds a five-minute run-idle sweep for orphaned `toolStartData` entries.

## Best Possible Solution

Keep the still-applicable utility cleanup, replace timestamp-only sweeping with an active-call-safe orphan cleanup, add focused regression coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: start one tool, leave it in flight for more than five minutes, trigger another tool start so the sweep runs, then finish the first tool and its start-data-derived args and duration are gone. I did not run a live repro because this review is read-only.

## Solution Assessment

No. Timestamp-only `lastRunActivity` is not a safe cleanup signal for active long-running tools; the narrower maintainable fix is to preserve records while a run has in-flight calls, using the existing active-count pattern or an equivalent guard.
