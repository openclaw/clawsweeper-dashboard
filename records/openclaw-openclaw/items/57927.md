## Summary

Keep open: current main does not implement the orphan cleanup, and the PR’s run-idle sweep can delete start data for an active long-running tool call; one utility hunk is also stale on current main.

## What This Changes

The PR imports shared utility helpers in two call sites and adds run-idle cleanup for module-level `toolStartData` records.

## Best Possible Solution

Implement orphan cleanup with an explicit in-flight count or equivalent guard per run, preserve the still-valid `clamp` cleanup, drop the stale doctor hunk, and add focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Start one tool call, advance past the five-minute idle window without ending it, trigger another tool start so the sweep runs, then end the first call and observe that `startData`-derived duration and args are missing.

## Solution Assessment

No. The shared utility cleanup is reasonable where it still applies, but `lastRunActivity` alone is not a safe cleanup signal because an active long-running call emits no intermediate activity.
