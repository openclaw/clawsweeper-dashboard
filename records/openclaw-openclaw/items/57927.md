## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks an orphan sweep for `toolStartData`, so the memory-leak cleanup remains relevant; however, the proposed run-idle sweep can delete state for an active tool call that runs longer than five minutes, which would degrade `after_tool_call` duration/params and cron/messaging bookkeeping.

## What This Changes

This PR replaces two local utility helpers with shared imports and adds a `lastRunActivity`-based sweep for orphaned `toolStartData` entries.

## Best Possible Solution

Revise the orphan cleanup so it tracks active in-flight tool calls per run and only sweeps runs with no outstanding calls; keep the harmless utility dedupe where it still applies, drop stale doctor code, and add focused regression coverage for both abandoned and long-running-active tool calls.

## Reproduction Assessment

Yes. A deterministic code path reproduces the PR bug: start one tool, wait longer than the five-minute idle threshold without ending it, trigger another tool start so the sweep runs, then end the first tool and observe missing `startData`.

## Solution Assessment

No. The shared utility imports are a reasonable cleanup, but the `lastRunActivity`-only sweep is not the best fix because it treats an active long-running tool as an idle run; an in-flight count or equivalent active-call guard is the safer maintainable approach.
