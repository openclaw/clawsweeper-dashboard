## Summary

Keep open: the PR has a useful cleanup/leak-fix direction, but the current sweep can delete metadata for active long-running tools and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR imports shared `clamp` and `isRecord` helpers and adds a five-minute run-idle sweep for orphaned `toolStartData` records.

## Best Possible Solution

Refresh the branch against current main, keep the still-applicable utility cleanup, implement orphan cleanup with explicit in-flight-call protection or an equivalent active-call guard, add focused regression coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: start one tool, leave it in flight for more than five minutes, trigger another tool start so the sweep runs, then finish the first tool and observe its start-data-derived args and duration are gone. I did not run a live repro because this review is constrained to read-only inspection.

## Solution Assessment

No. Timestamp-only `lastRunActivity` is not a safe cleanup signal for active long-running tools; the narrower maintainable fix is to preserve records while a run has in-flight calls, using the existing active-count pattern or an equivalent guard.
