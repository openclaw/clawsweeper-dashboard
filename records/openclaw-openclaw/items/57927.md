## Summary

Keep open: the utility dedupe is mostly reasonable, but the proposed `toolStartData` run-idle sweep can evict active long-running calls and the external PR still lacks real behavior proof.

## What This Changes

This PR imports shared `clamp`/`isRecord` helpers and adds a five-minute run-idle sweep for orphaned `toolStartData` records.

## Best Possible Solution

Update the branch to guard orphan cleanup with explicit in-flight tracking or an equivalent active-call check, keep only the utility cleanup that still applies, add focused regression coverage, and provide real runtime proof.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: start one tool, wait beyond the five-minute window without ending it, trigger another tool start so the sweep runs, then finish the first tool and observe missing start-data-derived args and duration.

## Solution Assessment

No. `lastRunActivity` alone is not a safe cleanup signal because an active long-running tool emits no intermediate activity; the narrower maintainable fix is to preserve records while a run has in-flight calls.
