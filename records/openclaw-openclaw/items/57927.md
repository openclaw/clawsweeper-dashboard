## Summary

Keep open: the cleanup direction is useful, but the branch is not merge-ready because the new idle sweep can delete active long-running tool records, real behavior proof is missing, and the branch is now conflicting with current main.

## What This Changes

The PR imports shared `clamp`/`isRecord` helpers and adds a five-minute per-run idle sweep for orphaned `toolStartData` entries.

## Best Possible Solution

Rebase the branch, keep only the duplicate-helper cleanup that still applies, and replace timestamp-only idle detection with in-flight per-run tracking plus focused tests and real runtime proof.

## Reproduction Assessment

Yes for the proposed-patch defect from source inspection: start one tool, let it run beyond five minutes, then start another tool so the sweep deletes the first tool's start record before its end handler runs. I did not run a live scenario in this read-only review.

## Solution Assessment

No: importing shared helpers is fine, but timestamp-only run activity is not the best cleanup mechanism. The safer fix is to track in-flight tool calls per run and sweep only runs with no outstanding calls.
