## Summary

Keep open: the utility deduplication is low risk, but the proposed `toolStartData` sweep can evict metadata for active long-running tools and needs contributor revision plus real behavior proof before merge.

## What This Changes

The PR replaces local `clamp`/`isRecord` helpers with shared utilities and adds a five-minute per-run idle sweep for `toolStartData`.

## Best Possible Solution

Keep the utility cleanup where it still applies, and implement orphan cleanup with an active-call-safe guard plus focused regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes by source inspection for the PR-introduced defect: start one tool, leave it in flight for more than five minutes, then trigger another tool start and the sweep can delete the first tool's start data before its end event. I did not run live proof in this read-only review.

## Solution Assessment

No. Timestamp-only run activity is not safe for quiet long-running calls; the narrower maintainable fix is to avoid sweeping runs with active tool executions, using `countActiveToolExecutions` or an equivalent in-flight guard.
