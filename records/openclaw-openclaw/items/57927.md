## Summary

Keep open: the utility dedupe looks low-risk, but the new orphan sweep can delete start metadata for an active long-running tool, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR replaces local `clamp` and `isRecord` helpers with shared utilities and adds a five-minute per-run idle sweep for `toolStartData`.

## Best Possible Solution

Keep the utility cleanup where it still applies, rework orphan cleanup to skip runs with active tool executions, add focused regression coverage, and require redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced defect: start one tool, leave it in flight for more than five minutes, then trigger another tool start so the sweep can delete the first tool's start data before its end event. I did not run live proof in this read-only review.

## Solution Assessment

No. Timestamp-only run activity is not safe for quiet long-running calls; the narrower maintainable fix is to avoid sweeping runs with active tool executions, using `countActiveToolExecutions` or an equivalent in-flight guard.
