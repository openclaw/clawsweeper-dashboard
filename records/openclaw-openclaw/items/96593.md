## What This Changes

The issue reports that WebChat jumps away from the newest message after rendering audio attachments and asks for media replies to remain pinned at the bottom like text replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96628 or an equivalent WebChat fix that re-runs the existing scroll scheduler after media metadata loads while preserving user scroll-up.

## Reproduction Assessment

Yes, source-reproducible: current main renders assistant audio/video controls after async availability without a media metadata hook back into chat scrolling. I did not run a live browser reproduction.

## Solution Assessment

Yes, the best fix direction is to reuse the existing WebChat scroll scheduler from the media-render boundary rather than add a setting or new scroll mode. The linked PR is the active implementation candidate, but it still needs proof before merge.
