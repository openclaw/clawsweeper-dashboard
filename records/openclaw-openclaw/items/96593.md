## Summary

Keep open: current main still renders WebChat audio attachments through an async availability/media-control path, but that later layout change is not tied back to the existing chat scroll scheduler. The latest known related merged PRs made audio visible and improved general scroll locking, but neither owns this audio-render-after-scroll gap.

## What This Changes

The issue requests that WebChat keep the viewport pinned to the latest message after WAV/MP3 audio attachment rendering, matching plain-text reply behavior.

## Best Possible Solution

Update the existing WebChat attachment render and auto-scroll boundary so audio attachment availability or metadata layout changes re-run near-bottom auto-follow without overriding a user who intentionally scrolled up.

## Reproduction Assessment

Yes, at source level: MEDIA audio replies reach the async attachment renderer, and current main does not tie later attachment availability or audio metadata layout changes back to the chat scroll scheduler. I did not run a live browser reproduction.

## Solution Assessment

Yes: the best fix is in the existing WebChat scroll/render path, not a new setting. It should preserve the current follow-lock behavior while following bottom-position layout changes caused by media attachments.
