## Summary

Keep open: current main still renders WebChat audio attachments through an async availability/media-control path that can change layout after the normal chat scroll scheduler runs, and no merged canonical fix covers that scroll-follow gap.

## What This Changes

The issue asks WebChat to keep the conversation pinned to the newest message after WAV/MP3 MEDIA audio attachments render.

## Best Possible Solution

Update the existing WebChat scroll-follow boundary so media attachment availability or audio metadata layout changes re-run near-bottom auto-follow without overriding a user who intentionally scrolled up.

## Reproduction Assessment

Yes, source-reproducible: MEDIA audio replies reach the async attachment renderer, and current main does not tie the later audio availability or media-control layout change back to the chat scroll scheduler. I did not run a live browser reproduction.

## Solution Assessment

Yes, the best fix is in the existing WebChat scroll/render path rather than a new setting. It should preserve the current user-scroll lock while following bottom-position layout changes from media attachments.
