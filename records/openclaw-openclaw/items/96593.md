## Summary

Keep open: the report describes a current WebChat UI bug, and current main still lacks a scroll-follow hook for post-render audio attachment availability or media metadata layout changes.

## What This Changes

WebChat should remain at the newest message after an assistant reply renders WAV/MP3 audio attachments from MEDIA directives.

## Best Possible Solution

Teach WebChat to re-run the existing near-bottom auto-follow behavior when assistant media attachments become available or media controls finish layout, without overriding a user who has intentionally scrolled up.

## Reproduction Assessment

Yes, source-reproducible: a WebChat assistant reply containing a MEDIA WAV/MP3 attachment reaches the audio attachment renderer, and later attachment/media layout updates are not tied back to the scroll scheduler. I did not run a live browser reproduction.

## Solution Assessment

Yes, the best fix is in the existing WebChat scroll/render path, not a new setting: preserve the current user-scroll lock while following bottom-position layout changes from media attachments.

Likely places: ui/src/ui/chat/grouped-render.ts, ui/src/ui/views/chat.ts, ui/src/ui/app-scroll.ts, ui/src/ui/app-lifecycle.ts, and adjacent tests.
