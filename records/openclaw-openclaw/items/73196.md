## Summary

Keep open: the startup-task isolation is plausible, but the branch also widens shared plugin registry loading to non-strict behavior that current source/tests treat as a strict contract, and the external PR still lacks real after-fix startup proof.

## What This Changes

The PR replaces gateway startup `Promise.all` with `Promise.allSettled` logging and changes the shared runtime registry loader to pass `throwOnLoadError: false`.

## Best Possible Solution

Keep the startup failure isolation, but preserve strict shared registry loading, make any non-strict handling gateway-startup-scoped or explicit, preserve error stacks, and add focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main still uses `Promise.all(startupTasks)`, while the shared runtime loader and tests assert strict `throwOnLoadError: true` behavior. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. `Promise.allSettled` is a reasonable narrow startup hardening, but flipping the shared runtime registry loader to non-strict behavior is broader than the reported gateway startup failure mode.
