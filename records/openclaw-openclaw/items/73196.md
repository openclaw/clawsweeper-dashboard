## Summary

Keep open. Current main does not implement this PR, but the proposed patch is broader than the stated gateway startup hardening because it changes the shared runtime registry loader from strict to non-strict for CLI, node-host, and message-helper callers that still have tests asserting strict plugin-load failures.

## What This Changes

The PR changes gateway startup maintenance to log rejected startup tasks after `Promise.allSettled` and changes the shared runtime plugin registry ensure path from strict plugin-load failures to graceful error records.

## Best Possible Solution

Keep gateway startup resilience scoped to the gateway startup path, preserve strict shared registry loading for existing callers, add focused tests for rejected startup tasks and the strict loader contract, improve the new rejection log detail, and add the required single-line changelog entry.

## Reproduction Assessment

Yes for the review findings. The source and focused tests show current strict `throwOnLoadError: true` expectations, while the provided PR diff flips that shared option and omits both test updates and `CHANGELOG.md`.

## Solution Assessment

No as written. `Promise.allSettled` can be a reasonable narrow startup hardening, but changing the shared `ensurePluginRegistryLoaded` default to non-strict is broader than the gateway startup failure mode and conflicts with existing contract tests.
