## Summary

Keep open: the PR addresses a plausible gateway startup crash path, but the branch globally relaxes shared plugin registry load failures and lacks after-fix real behavior proof, so it is not ready to merge.

## What This Changes

The PR changes gateway startup maintenance to log rejected startup tasks after `Promise.allSettled` and changes the shared runtime plugin registry loader from strict plugin-load failures to graceful error records.

## Best Possible Solution

Keep the gateway startup isolation, but scope graceful plugin-load handling to gateway bootstrap or an explicit option while preserving strict shared registry behavior and stack-preserving diagnostics.

## Reproduction Assessment

Yes, by source inspection for the review findings: current main still uses `Promise.all(startupTasks)`, while tests and source show shared registry loading is intentionally strict. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. `Promise.allSettled` is a reasonable narrow startup hardening, but flipping the shared runtime registry loader to `throwOnLoadError: false` is broader than the reported gateway startup failure mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve strict registry load errors outside startup:** `src/plugins/runtime/runtime-registry-loader.ts:149`
  - body: `ensurePluginRegistryLoaded` is shared by CLI, message-helper, agent-harness, and node-host paths, and current tests assert it forwards `throwOnLoadError: true`. Changing the shared flag to `false` lets broken plugins be carried forward in callers that currently expect an early `PluginLoadFailureError`; scope graceful handling to gateway startup or an explicit option.

  - confidence: 0.78
