## Summary

Keep open. The gateway startup `Promise.allSettled` direction is useful, but the PR also changes the shared runtime registry loader from strict failures to silent error records, which affects CLI/message/node-host callers and contradicts existing tests; the prior Greptile and ClawSweeper comments correctly identify the logging and scoping issues.

## What This Changes

The PR changes gateway startup maintenance to use `Promise.allSettled` and switches shared runtime plugin registry loading from throwing on plugin load errors to carrying error records.

## Best Possible Solution

Keep gateway startup resilience, but scope any non-strict plugin-load handling to gateway startup or an explicit option while preserving strict shared loader behavior, then add stack-preserving logs, focused tests, and a single-line changelog entry.

## Reproduction Assessment

Yes for the review findings. Current main shows `Promise.all(startupTasks)` can reject gateway bootstrap, while existing runtime-loader and CLI tests assert `throwOnLoadError: true` for the shared loader contract the PR changes.

## Solution Assessment

No as written. `Promise.allSettled` is a reasonable narrow startup hardening, but flipping the shared registry loader to `throwOnLoadError: false` is broader than the reported gateway startup failure mode.
