## Summary

Keep this PR open: current main still lacks the requested Docker workspace mount propagation option, but the proposed feature is incomplete because it only reaches the main sandbox container and leaves browser-container workspace mounts, public config docs/help, and changelog follow-through unresolved.

## What This Changes

The PR adds a `sandbox.docker.workspaceMountPropagation` config option, schema/type plumbing, Docker workspace bind formatting, resolver wiring, and sandbox mount/config-hash tests for `rprivate`, `rslave`, and `rshared`.

## Best Possible Solution

Land this as one coherent Docker sandbox config feature only after browser-container propagation parity, public docs/help/labels, changelog coverage, and security-owner approval are all aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on the PR head, configure `sandbox.docker.workspaceMountPropagation` as `rslave` or `rshared` and inspect Docker create args; the main sandbox mount gets the option while the browser container path still calls the shared mount helper without it.

## Solution Assessment

No. The direction is plausible, but the maintainable solution needs propagation parity for sandbox browser containers and public config documentation/metadata before this should merge.
