The PR adds a `sandbox.docker.workspaceMountPropagation` Docker sandbox config field, schema/generated-schema/type plumbing, main-container mount wiring, and sandbox mount/config-hash test updates.

## Best Possible Solution

Coordinate or fold #75678 into this work, apply the option consistently to main and browser sandbox containers, correct the propagation wording, add config help/labels/docs/changelog, and land after maintainer approval of the sandbox policy.

Yes. On the PR head, configure `sandbox.docker.workspaceMountPropagation` as `rslave` or `rshared` and inspect Docker create args: the main sandbox path receives the option while the browser container path still calls the shared mount helper without it.

## Solution Assessment

No. The direction is plausible, but the narrow maintainable solution needs browser-container parity, accurate user-facing documentation for Docker propagation semantics, and the usual public config metadata before merge.
