## Summary

Keep open. Current main still lacks the requested Docker workspace mount propagation config, so the PR is not obsolete, but the proposed patch remains incomplete because the propagation mode is wired only for the main sandbox container and not the sandbox browser path. The feature also touches isolation-sensitive Docker mount semantics, so it should get maintainer review before merge.

## What This Changes

The PR adds a `sandbox.docker.workspaceMountPropagation` config field with type/schema/generated-schema plumbing, resolver wiring, Docker workspace mount formatting changes, and sandbox mount/config-hash test updates.

## Best Possible Solution

Finish the feature by applying the selected propagation mode consistently to both main and browser sandbox workspace mounts, keep `rprivate` as the explicit default-compatible value, add the required public config docs/changelog/metadata, and have maintainers approve the `rslave`/`rshared` sandbox policy before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to set `workspaceMountPropagation` to `rslave` or `rshared` on the PR head and inspect Docker create args for main versus browser containers; the browser create path still omits the new helper argument.

## Solution Assessment

No. The PR is a useful direction, but it is not the best complete solution until browser containers receive the same mount propagation and the new public config surface has docs/changelog/config metadata plus maintainer policy approval.
