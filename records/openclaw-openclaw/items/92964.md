## Summary

Keep open: the linked session-state bug is valid, but this PR is not mergeable because the handler now reads `p.agentId` while the `sessions.describe` protocol schema still rejects that field and the PR CI fails on the resulting type error.

## What This Changes

The PR changes the `sessions.describe` Gateway handler to pass `p.agentId` into session target resolution.

## Best Possible Solution

Add optional `agentId` to the `sessions.describe` protocol, validate it with the existing requested-agent helper, pass the resolved agent through target and row projection, and cover scoped global/default behavior with focused Gateway protocol tests plus real RPC proof.

## Reproduction Assessment

Yes from source and CI: current main rejects `agentId` in `sessions.describe`, while the PR's changed handler line fails type checks because that protocol field does not exist. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No. The one-line handler pass-through is too narrow because it reads a non-existent protocol field and skips the sibling requested-agent validation/row-scoping pattern.
