## Summary

Keep open: current main and the latest release still treat explicit OpenAI provider config as an opt-in to the direct Images API route, so preferring Codex OAuth in that mixed setup is a compatibility-sensitive product decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to prefer or clearly diagnose Codex OAuth image generation for `openai/gpt-image-2` when explicit OpenAI direct provider config and an API key are also present.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Partly: current source and tests reproduce the direct-route selection with explicit OpenAI provider config while Codex OAuth exists. The exact live 401 missing-scope response remains reporter-provided and was not live reproduced here.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct config conflicts with current docs and tests. A diagnostic-only improvement is the safer narrow path unless maintainers choose a compatibility-sensitive precedence change.
