## Summary

Keep open: current main and the latest release still intentionally preserve explicit direct OpenAI provider config as an opt-in to the direct Images API route, so changing precedence toward Codex OAuth or adding direct-route diagnostics remains a maintainer product/compatibility decision.

## What This Changes

The issue asks OpenAI image generation to prefer Codex OAuth, or at least clearly log and explain direct API-key routing, when `openai/gpt-image-2` is configured alongside explicit OpenAI provider config.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

No high-confidence live reproduction was run here: source and tests reproduce the explicit-direct route selection, but the exact missing-scope 401 requires a live OpenAI API key/account state that this read-only review did not exercise.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct config conflicts with current docs and tests; a diagnostic-only improvement is the safer narrow fix unless maintainers choose a compatibility-sensitive precedence change.
