## Summary

Keep open: current main and the latest release still preserve explicit OpenAI provider config as the direct Images API route, so the remaining request is a maintainer decision about route precedence or a focused diagnostic improvement.

## What This Changes

The issue asks OpenClaw to prefer or clearly diagnose the Codex OAuth image-generation route when `openai/gpt-image-2`, explicit OpenAI Platform provider config, `OPENAI_API_KEY`, and a valid OpenAI OAuth profile coexist.

## Best Possible Solution

Maintainers should choose the route contract explicitly: either keep explicit provider config as direct-route intent and add a clear missing-scope diagnostic, or change precedence with upgrade-safe docs and tests for both direct API-key and Codex OAuth setups.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source and tests show current main still selects the direct route for explicit OpenAI provider config, but I did not run a live OpenAI request to reproduce the exact 401 scope response.

## Solution Assessment

Is this the best way to solve the issue? Unclear: preferring OAuth over explicit direct config is plausible but compatibility-sensitive, while a route-aware diagnostic may solve the confusion without changing configured route precedence.
