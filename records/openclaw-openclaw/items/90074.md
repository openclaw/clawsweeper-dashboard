## Summary

Keep open: current main and the latest release still preserve explicit direct OpenAI provider config as an opt-in to the direct Images API route, so preferring Codex OAuth in this mixed setup remains a maintainer product and compatibility decision.

## What This Changes

The issue requests that OpenAI image generation prefer or clearly diagnose Codex OAuth routing for `openai/gpt-image-2` when explicit direct OpenAI provider config and `OPENAI_API_KEY` are also present.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change precedence, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Partly: source and tests reproduce the reported route selection with explicit direct OpenAI config, but this read-only review did not reproduce the live 401 missing-scope response against OpenAI.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct provider config conflicts with current docs and tests; a diagnostic-first change is the safer narrow fix unless maintainers choose a compatibility-sensitive precedence change.
