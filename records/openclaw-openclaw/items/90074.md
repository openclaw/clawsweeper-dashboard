## Summary

Keep open: current main and the latest release still intentionally route explicit non-ChatGPT OpenAI provider config through the direct Images API path, so the remaining request is a maintainer product/compatibility decision or a focused diagnostic improvement, not conservative cleanup.

## What This Changes

The issue requests that OpenAI image generation prefer Codex/ChatGPT OAuth for gpt-image-2 when OAuth is available, or at least make direct API-key routing failures clearer.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Partly: source and tests show the route choice with high confidence because explicit direct OpenAI provider config prevents Codex Responses routing. The exact live 401 missing-scope response was not reproduced in this read-only review.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct config is not currently the best proven fix because it conflicts with shipped docs and tests. A diagnostic-only improvement may solve the confusion without changing route precedence, unless maintainers choose a compatibility-sensitive policy shift.
