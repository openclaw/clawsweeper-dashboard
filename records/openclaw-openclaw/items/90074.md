## Summary

Keep open: current main and the latest stable release still intentionally treat explicit non-ChatGPT OpenAI provider config as the direct Images API route, so the requested OAuth preference is a product/compatibility decision rather than an already-fixed bug. The narrower useful remaining work is deciding whether to change that precedence or add clearer diagnostics when direct routing fails while Codex OAuth is also available.

## What This Changes

The issue asks OpenClaw to prefer Codex OAuth, or at least explain the selected direct route clearly, for OpenAI image generation when explicit OpenAI provider config currently selects the direct Platform Images API route.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Partly: the route choice is high-confidence from current source and tests, because explicit direct OpenAI provider config prevents Codex Responses routing and sends image generation to the direct Images API. The exact live 401 missing-scope response was not reproduced here.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct config is not currently the best proven fix because it conflicts with shipped docs and tests. A diagnostic-only improvement may solve the confusion without changing route precedence, unless maintainers choose a compatibility-breaking policy shift.
