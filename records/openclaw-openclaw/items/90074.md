## Summary

Keep open: current main and the latest release still preserve explicit direct OpenAI provider config as an opt-in to the direct Images API route, so changing that precedence is a maintainer product/compatibility decision rather than conservative cleanup.

## What This Changes

Request to make OpenAI image generation prefer Codex/ChatGPT OAuth, or at least log clearer diagnostics, when `openai/gpt-image-2` is selected while explicit direct OpenAI provider config and an `OPENAI_API_KEY` are present.

## Best Possible Solution

Preserve the shipped explicit-direct route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Partly: source and tests reproduce the route selection with explicit direct OpenAI config, but the exact live 401 missing-scope response was not reproduced in this read-only review.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit direct config conflicts with current docs and tests; a diagnostic-only improvement is the safer narrow fix unless maintainers choose a compatibility-sensitive precedence change.
