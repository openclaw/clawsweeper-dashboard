## Summary

Keep open: current main and the latest release still treat explicit non-ChatGPT `models.providers.openai` config as a direct OpenAI Images API route, so the requested OAuth-precedence change or clearer direct-route diagnostic remains unresolved and compatibility-sensitive.

## What This Changes

Request OpenAI image generation to prefer Codex/ChatGPT OAuth, or at least produce clearer direct-route diagnostics, when explicit OpenAI provider config and `OPENAI_API_KEY` are also present.

## Best Possible Solution

Keep the explicit direct-route contract unless maintainers intentionally change it, and add focused direct-route/auth diagnostics for direct OpenAI image failures when Codex OAuth is also available.

## Reproduction Assessment

Yes for the route choice: source and tests show that explicit `models.providers.openai.api` direct config sends the request to `/images/generations` and avoids the Codex Responses route. The exact live 401 missing-scope response was not reproduced in this read-only review.

## Solution Assessment

Unclear: automatically preferring OAuth over explicit config conflicts with current docs and tests, so a diagnostic-only improvement may be the narrower maintainable fix unless maintainers choose a compatibility-sensitive route-policy change.
