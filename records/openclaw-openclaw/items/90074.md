## Summary

Keep open: current main and v2026.6.1 intentionally treat explicit non-ChatGPT OpenAI provider config as the direct Images API route, so the requested automatic OAuth preference conflicts with the documented contract. The remaining useful work is a maintainer decision about whether to change that precedence or add a narrow diagnostic that names the selected direct route when Codex OAuth is also available.

## What This Changes

The issue requests that OpenAI image generation prefer or clearly explain Codex OAuth routing when direct OpenAI provider config and an OAuth profile both exist.

## Best Possible Solution

Preserve explicit provider config as the direct-route contract unless maintainers intentionally change policy, and add focused route/error diagnostics for direct OpenAI image failures when a Codex OAuth profile is also available.

## Reproduction Assessment

Yes for the route choice from source: explicit non-ChatGPT OpenAI provider config prevents the Codex Responses route and falls through to the direct Images API path. The exact OpenAI 401 scope response was not live-reproduced in this review.

## Solution Assessment

No for automatically preferring OAuth over explicit direct config without maintainer approval, because current docs and tests make explicit provider config the direct-route signal. The narrower maintainable fix is to make direct-route failures self-explanatory and leave route precedence unchanged unless maintainers choose a compatibility-breaking policy shift.
