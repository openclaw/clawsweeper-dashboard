## Summary

Keep open: the PR is not obsolete and appears to be the right narrow fix for a shipped transcript-migration data-loss window, but it intentionally changes the embedded session takeover fence from same-inode to content-based acceptance for this rewrite path, so maintainer owner sign-off is still needed before merge.

## What This Changes

The PR makes the legacy linear-to-parent-linked transcript migration use an fs-safe sibling-temp atomic rename, adds regression coverage, and updates the embedded attempt fence to accept that benign inode-replacing migration with a re-stat guard.

## Best Possible Solution

Land the atomic migration fix after session/embedded-runner owner sign-off, and keep the broader non-atomic SessionManager rewrite follow-up separate from this focused PR.

## Reproduction Assessment

Yes, from source and supplied terminal proof. Current main and v2026.6.6 rewrite the migrated transcript through writeJsonlLines/fs.writeFile, and the PR body shows an on-disk ENOSPC-at-rename run preserving the original bytes and succeeding on retry.

## Solution Assessment

Yes, this is the best narrow fix for the migration itself: the whole-file rewrite is the failure point, and fs-safe sibling temp plus atomic rename preserves successful output while making failure retryable. The remaining question is maintainer acceptance of the necessary embedded-fence boundary change.
