## Summary

Keep open: current main appears to satisfy the backend cursor part of the report, but the picker still pages and counts raw `sessions.list` rows while hiding inactive subagent rows afterward, so a subagent-heavy agent can still show a misleading count and require paging through hidden rows.

## What This Changes

The issue asks for the Control UI chat session picker to remain navigable when an agent has many `spawn-child` sessions and for `sessions.list` pagination to return usable offsets.

## Best Possible Solution

Make the chat picker page and count over the same navigable session set it renders, preferably with a focused server-side filter or equivalent projection that excludes inactive subagent child rows for picker requests without changing the general Sessions view or spawned-child drilldowns.

## Reproduction Assessment

Yes from source inspection for the remaining UI problem: current main hides subagent rows after receiving a paginated raw result and displays the raw `totalCount`, which matches the reported `1 / many` dead-end shape. I did not run a live Control UI reproduction because this review is read-only.

## Solution Assessment

No complete fix is present yet: current main appears to solve the invalid backend cursor contract, but the best remaining fix is to align picker paging/counting with the rows the picker can actually display.
