## Summary

Keep open for normal PR review, but the current patch should not merge yet: it introduces a security-sensitive cron transport override surface and has a concrete header-merge regression in the cron path.

## What This Changes

The PR adds agent- and cron-job-level provider request overrides, threads them into model resolution, persists cron payload providers, and adds focused tests.

## Best Possible Solution

Keep the useful per-agent direction, fix cron overrides through the shared request-merge path, remove unrelated generated drift, document the public config surface, and only expose job-scoped transport overrides after an explicit security/product decision.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The PR body does provide a high-confidence after-fix proof path with copied mock-server output showing merged headers reaching a provider-compatible endpoint.

## Solution Assessment

No. The general model-resolution direction is plausible, but cron job overrides should use the existing merge helper or a prepared job request value, and unsafe transport fields need explicit maintainer approval or narrower scoping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate job-scoped transport overrides before merge:** `src/cron/types.ts:167`
  - body: `payload.providers` can now carry `request.auth`, `proxy`, `tls`, and `allowPrivateNetwork` on a mutable cron job payload. Existing provider request controls are operator-level config, so this turns a scheduled job patch into provider credential and egress policy control; please restrict the job surface to approved fields or add an explicit trust/validation boundary before exposing it.
  - confidence: 0.87
- **[P2] Preserve agent headers when applying job overrides:** `src/cron/isolated-agent/run.ts:511`
  - body: When the same provider has both agent and job `request.headers`, this shallow merge replaces the entire agent header map with the job header map. That loses non-conflicting agent headers before `mergeModelProviderRequestOverrides` can merge them, so the claimed global -> agent -> job precedence only works in tests that bypass the cron preparation path.
  - confidence: 0.92
- **[P3] Drop the unrelated A2UI hash change:** `extensions/canvas/src/host/a2ui/.bundle.hash:1`
  - body: This provider-header feature does not touch Canvas/A2UI sources, but the PR changes only the generated bundle hash. That can make bundle regeneration or package drift checks noisy or wrong; please remove it from this PR or pair it with the corresponding Canvas generated asset in a separate change.
  - confidence: 0.8
