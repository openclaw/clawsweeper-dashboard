## Summary

Keep open. This is a plausible feature PR, but the latest head still has a source-visible compile blocker, the public cron RPC schema still rejects the new cron payload field, and the persisted header surface needs an explicit secrecy/redaction policy before merge.

## What This Changes

The PR adds agent-level provider request overrides and cron job provider header overrides with global-to-agent-to-job merge precedence, config/schema typing, resolver wiring, and focused tests.

## Best Possible Solution

Align the additive cron protocol contract, remove the compile blocker, define the persisted-header secrecy policy, then prove a full Gateway cron create/update/run path against a mock provider.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head has an unused import under noUnusedLocals, and current main's cron RPC schema rejects payload.providers before cron persistence. Full runtime reproduction was not run in this read-only review.

## Solution Assessment

No. The agent-level resolver direction may be viable, but cron job headers need the public protocol schema, docs/client contract, validation, and secrecy policy handled together rather than only internal cron/service types.

## Security Review

Status: needs_attention

Summary: The PR exposes persisted cron job request headers without a clear redaction or SecretRef policy.

Concerns:

- **[medium] Define cron header secrecy policy:** `src/cron/types.ts:206`
