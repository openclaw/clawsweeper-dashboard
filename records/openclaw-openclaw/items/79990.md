Keep open: this is a plausible feature PR, but the current head still has blocking source-inspected issues. ## What This Changes

The PR adds agent-level provider request overrides, cron job payload provider headers, merge precedence across global/agent/job config, and focused resolver and cron patch tests.

## Best Possible Solution

Land this only after the branch compiles, the additive Gateway cron contract and docs are aligned, cron header validation/redaction policy is approved, and a full Gateway cron create/update/run path is proven against a mock provider.

Yes for the review findings by source inspection: PR head has an unused import under `noUnusedLocals`, and the new internal cron payload field is absent from the Gateway schema that rejects extra payload properties.

## Solution Assessment

No. The resolver merge direction may be viable, but the current PR is not the narrowest maintainable solution until compile, protocol, docs/client contract, and cron header secrecy policy are handled together.

## Security Review

Status: needs_attention

Summary: The PR introduces a persisted cron payload header surface that needs an explicit validation and redaction policy before it is exposed through Gateway cron APIs.

Concerns:

- **[medium] Define cron header secrecy policy:** `src/cron/types.ts:196`
  - body: `payload.providers.*.request.headers` can carry credential-like headers such as `Authorization` or `Cookie`, while cron jobs are persisted and returned by `cron.list`/`cron.get`. Add a maintainer-approved denylist/redaction/SecretRef policy before exposing this field through the public cron contract.
  - confidence: 0.84
