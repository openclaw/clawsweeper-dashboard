## Summary

Keep open: the PR still has blocking correctness and security-policy issues in the cron half of the feature. The agent-level provider override plumbing looks plausible, but the advertised cron payload field is not accepted by the public Gateway schema and persisted cron headers still need a defined secrecy/redaction policy.

## What This Changes

The PR adds agent-level provider request overrides, cron job provider header overrides, merge precedence plumbing, and tests for provider request resolution and cron payload patching.

## Best Possible Solution

Land this only after adding the public cron protocol schema/generated client/docs follow-through, defining cron header redaction or SecretRef-only policy, and proving Gateway cron create/update/run behavior end to end.

## Reproduction Assessment

Yes. Source inspection of PR head shows `payload.providers` added to internal cron types/service code while the public cron agentTurn payload schema remains closed without `providers`, and Gateway cron add/update validate through that schema before persistence.

## Solution Assessment

No. Agent-level provider request overrides are a plausible part of the solution, but cron job headers need a public Gateway contract plus a secrecy policy before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:205`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `cronAgentTurnPayloadSchema` still has no `providers` field and is closed with `additionalProperties: false`. `cron.add` and `cron.update` validate through that schema before persistence, so supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.95
- **[P2] Protect persisted cron headers before exposing them:** `src/cron/types.ts:205`
  - body: `payload.providers.*.request.headers` accepts arbitrary header names and values, then persists those values in cron jobs returned by `cron.list` and `cron.get`. Headers such as `Authorization` or `Cookie` need redaction, a denylist, or a SecretRef-only policy before this mutable cron payload field is exposed.
  - confidence: 0.87
