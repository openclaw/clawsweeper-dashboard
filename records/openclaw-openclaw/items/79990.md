## Summary

Keep open: this is a plausible feature PR, but the cron job surface is incomplete because PR head adds internal `payload.providers` while the Gateway cron schema still rejects that field before the service merge path can run. It also needs full-path cron proof and a maintainer-approved header exposure policy before merge.

## What This Changes

The PR adds per-agent provider request overrides, cron job payload provider headers, merge precedence plumbing, cron payload persistence, and resolver/cron tests.

## Best Possible Solution

Align the agent/cron config types, Zod config schema, Gateway protocol schema, generated clients, docs, tests, and cron header validation/redaction policy, then prove a Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron `payload.providers`, while the Gateway cron payload schemas still reject unknown agentTurn payload fields. This is a proposed-patch failure path, not a current-main bug.

## Solution Assessment

No. The per-agent resolver direction may be viable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve the header validation/redaction policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds job-scoped `payload.providers` to cron payload types and the service merge path, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. Supported `cron.add` and `cron.update` calls carrying these headers will be rejected by validation before they reach `context.cron.add/update`, so the advertised per-job header feature cannot be configured through the RPC surface.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The PR introduces a persisted cron payload header surface that needs explicit validation and redaction policy before it is exposed through the public cron API.

Concerns:

- **[medium] Define cron header exposure policy:** `src/cron/types.ts:196`
  - body: `CronJobProviderRequest.headers` is headers-only, but arbitrary header names can still carry credentials such as Authorization or Cookie, and cron payloads are persisted and returned by job APIs. Validate or deny sensitive header names, or document the maintainer-approved redaction and exposure policy before exposing this through the public cron API.
  - confidence: 0.82
