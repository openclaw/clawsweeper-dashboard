## Summary

Keep this PR open, but it is not merge-ready: the agent-level provider override direction looks plausible, while the cron header surface is currently unreachable through the public Gateway schema and needs a clear secrecy policy before persisted headers are exposed.

## What This Changes

Adds agent-level provider request overrides, cron payload provider header overrides, merge precedence wiring, and focused tests for provider request and cron patch behavior.

## Best Possible Solution

Land the feature only after defining the cron header secrecy policy, adding the public Gateway protocol schema and generated client/docs follow-through, and proving cron create/update/run behavior end to end.

## Reproduction Assessment

Yes. Source inspection of PR head shows payload.providers added to internal cron payload types while the public cron agentTurn payload schema remains closed without providers, and cron.add/update validate through that schema before persistence.

## Solution Assessment

No. Agent-level provider request overrides are a plausible part of the solution, but cron job headers need a public Gateway contract plus a redaction, denylist, or SecretRef-only policy before this is safe to merge.

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:205`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `cronAgentTurnPayloadSchema` still has no `providers` field and is closed with `additionalProperties: false`. `cron.add` and `cron.update` validate through that schema before persistence, so supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.95

  - body: `payload.providers.*.request.headers` accepts arbitrary header names and values, then the PR persists those values in cron jobs returned by `cron.list` and `cron.get`.   - confidence: 0.87
