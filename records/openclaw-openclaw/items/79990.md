## Summary

Keep this PR open, but it still needs changes before merge: the branch adds the internal cron payload field while leaving the public Gateway cron schema unwired, so supported cron.add/cron.update callers cannot configure the advertised job-level headers.

## What This Changes

The PR adds agent-level provider request overrides and cron-job-level provider header overrides, with global-to-agent-to-job precedence plumbing and focused resolver/cron persistence tests.

## Best Possible Solution

Land this only after the additive config/protocol contract, validators, docs, tests, generated clients, and cron header validation policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron payload.providers, while the Gateway protocol schemas used by cron.add and cron.update still reject unknown agent-turn payload fields. This is not a current-main bug reproduction; it is a proposed-patch failure path.

## Solution Assessment

No. The per-agent resolver direction may be acceptable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve validation for mutable job-scoped headers.
