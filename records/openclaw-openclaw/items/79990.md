## Summary

Keep open: the feature is plausible, but the PR still does not wire the new cron `payload.providers` field through the public Gateway cron schema/validators, and it needs an explicit security policy plus stronger full-path proof before merge.

## What This Changes

The PR adds per-agent provider request overrides and cron job payload provider headers with global-to-agent-to-job merge precedence, plus resolver and cron persistence tests.

## Best Possible Solution

Land this only after the config/protocol contract, validators, generated clients, docs, tests, and cron header validation/redaction policy are aligned, then prove a Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: the branch adds internal cron `payload.providers`, while current Gateway schemas still reject unknown agent-turn payload fields before cron add/update reaches the service. This is a proposed-patch failure path, not a current-main bug.

## Solution Assessment

No. The per-agent resolver direction may be viable, but the cron job feature is incomplete until the public protocol boundary accepts and validates the headers-only field and maintainers approve the persistence/redaction policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:197`
  - body: The PR adds job-scoped `payload.providers` to cron payload types and service merge code, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. Supported `cron.add` and `cron.update` calls carrying these headers will be rejected by validation before they reach the service, so the advertised per-job header feature cannot be configured through the RPC surface.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff adds a mutable, persisted cron payload header surface without an approved validation/redaction policy for credential-like headers.

Concerns:

- **[medium] Define cron header persistence policy:** `src/cron/types.ts:197`
  - body: `CronJobProviderRequest.headers` is headers-only, but arbitrary header names can still carry secrets such as `Authorization` or `Cookie`, and cron payloads are persisted and returned by job APIs. Validate or deny sensitive names, or document the maintainer-approved redaction and exposure policy before exposing this through the public cron API.
  - confidence: 0.82
