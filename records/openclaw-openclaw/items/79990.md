## Summary

Keep open. The feature direction is plausible, but the current PR still leaves the public cron create/update contract unable to accept `payload.providers`, and job-scoped arbitrary provider headers need explicit maintainer policy/security review before merge.

## What This Changes

The PR adds agent-level provider request overrides, cron job header overrides, merge precedence plumbing, cron payload persistence, and focused tests.

## Best Possible Solution

Land this as an additive cron/config contract only after the protocol schemas, generated client artifacts, docs, validator tests, and header validation policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: the PR adds internal cron `payload.providers`, while the Gateway schemas used by `cron.add` and `cron.update` still reject unknown agent-turn fields. Not applicable as a current-main bug report.

## Solution Assessment

No. The per-agent resolver direction is plausible, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve the policy for mutable job-scoped headers.

## Security Review

Status: needs_attention

Summary: The diff has no unrelated supply-chain churn, but job-scoped arbitrary provider headers are a security-sensitive public policy surface.

Concerns:

- **[medium] Define policy for arbitrary job provider headers:** `src/cron/types.ts:177`

That may be acceptable, but it needs maintainer-approved validation boundaries before becoming a public cron API.
