## Summary

Keep this PR open, but it is not merge-ready: the useful per-agent/provider direction is plausible, while the cron job feature currently stops at internal types and does not reach the public Gateway cron RPC contract.

## What This Changes

The PR adds agent-level provider request overrides and cron job-level provider header overrides with global-to-agent-to-job precedence.

## Best Possible Solution

Land this only after the additive config/protocol contract, validators, docs, generated clients, and cron header validation/redaction policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron payload.providers, while current main Gateway schemas still reject unknown agentTurn payload fields before cron.add or cron.update reaches the service.

## Solution Assessment

No. The per-agent resolver approach may be viable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve the header validation/redaction policy.
