## Summary

Keep open: this feature PR still has a correctness blocker because cron job provider headers are added to internal cron types and service paths, but not to the Gateway RPC schema that validates supported cron.add and cron.update requests. The mutable cron header surface also needs maintainer-approved policy and stronger full-path proof before merge.

## What This Changes

The PR adds agent-level provider request overrides, cron job payload provider headers, global-to-agent-to-job merge precedence, cron payload persistence, and focused resolver/cron tests.

## Best Possible Solution

Land this only after the additive config/protocol contract, validators, docs, generated clients, and cron header validation/redaction policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron payload.providers, while the Gateway schemas still reject unknown agentTurn payload fields before cron.add or cron.update reaches the service. This is a proposed-patch failure path, not a current-main bug.

## Solution Assessment

No. The per-agent resolver direction may be viable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve header validation/redaction policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds job-scoped payload.providers to cron payload types and the service merge path, but the Gateway cron payload schema still has additionalProperties: false and no providers property. Supported cron.add and cron.update calls carrying these headers will be rejected by validation before they reach context.cron.add/update, so the advertised per-job header feature cannot be configured through the RPC surface.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The PR exposes a mutable, persisted cron payload header surface without a validated policy for auth-like header names or redaction.

Concerns:

- **[medium] Define cron header policy:** `src/cron/types.ts:196`
  - body: CronJobProviderRequest.headers intentionally excludes auth/proxy/TLS fields, but arbitrary headers can still carry credentials such as Authorization or Cookie, and cron payloads are persisted and returned by job APIs. Validate or deny sensitive header names, or document the maintainer-approved redaction and exposure policy before exposing this through the public cron API.
  - confidence: 0.82
