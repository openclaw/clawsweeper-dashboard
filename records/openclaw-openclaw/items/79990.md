## Summary

Keep open: this is a useful feature PR, but the cron-job part is still blocked because PR head adds internal `payload.providers` storage without adding the public Gateway cron schema field, and the persisted-header secrecy policy is unresolved.

## What This Changes

The PR adds per-agent provider request overrides, cron payload provider header overrides, merge precedence wiring, and focused tests for provider request and cron patch behavior.

## Best Possible Solution

Land this only after adding the public cron protocol schema/client/docs follow-through, defining cron header redaction or SecretRef-only policy, and proving the agent and cron Gateway paths end to end.

## Reproduction Assessment

Yes for the blocking PR behavior: source inspection of PR head shows `payload.providers` added to internal cron types while the public cron schema remains closed without that field. I did not run a live Gateway repro because this read-only review cannot start artifact-producing test lanes.

## Solution Assessment

No. Agent-level provider request overrides are a plausible direction, but the cron-job surface needs a public Gateway contract and a credential-handling policy before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:205`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `cronAgentTurnPayloadSchema` still has no `providers` field and is closed with `additionalProperties: false`. Since `cron.add` and `cron.update` validate against that schema before persistence, supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.95
- **[P2] Protect persisted cron headers before exposing them:** `src/cron/types.ts:11`
  - body: `payload.providers.*.request.headers` accepts arbitrary header names and values, then the stored cron job is returned by `cron.list` and `cron.get`. Headers such as `Authorization` or `Cookie` need redaction, a denylist, or SecretRef-only values before this mutable cron payload field is exposed.
  - confidence: 0.87
