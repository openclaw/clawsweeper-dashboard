## Summary

Keep open: the PR is useful, but the cron-job portion is still incomplete because it adds `payload.providers` to internal cron types/service code while the public Gateway cron RPC schema remains strict and does not accept that field.

## What This Changes

The PR adds per-agent provider request overrides and cron-job provider header overrides with global-to-agent-to-job merge precedence plus focused tests.

## Best Possible Solution

Land only after the additive protocol/schema, validators, docs, generated client surface, tests, and cron header policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron `payload.providers`, while the Gateway cron schema still rejects unknown agent-turn payload fields. This is a proposed-patch failure path, not a current-main bug reproduction.

## Solution Assessment

No. The per-agent resolver direction may be viable, but the cron feature is not complete until the public protocol boundary accepts the headers-only field and maintainers approve validation/redaction policy.
