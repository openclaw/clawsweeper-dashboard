Full review comments:

- **[P1] Wire cron providers through the Gateway protocol schema:** `src/cron/types.ts:249-250`
  - body: This adds `payload.providers` to the runtime cron payload type, but supported Gateway calls are validated against `packages/gateway-protocol/src/schema/cron.ts`, whose `agentTurn` payload schema has `additionalProperties: false` and no `providers` field. A client sending the new per-job headers through `cron.add` or `cron.update` is rejected before the service code runs, so the advertised cron feature is not usable through the public API.
  - confidence: 0.95
- **[P1] Persist cron provider headers through SQLite:** `src/cron/service/jobs.ts:935`
  - body: The patch stores `providers` only on the in-memory cron payload, while `bindPayloadColumns` and `payloadFromRow` still serialize and reconstruct the fixed set of payload columns without `providers`. Any job-level headers that do get created will be lost after the cron store is written and reloaded, contradicting the PR's persistence claim.
  - confidence: 0.93
- **[P1] Define redacted cron header readback before exposing providers:** `src/cron/types.ts:250`
  - body: `CronJobProviderRequest.headers` accepts `SecretInput`, and the Gateway cron methods return job objects directly from list/get/add/update. Once the protocol accepts this field, plaintext header values can be echoed back unless this PR adds a clear SecretRef-only or redacted response policy with tests.
  - confidence: 0.9
