## Summary

Keep open: the PR still has two merge-blocking issues in the cron header path. The agent-level provider request plumbing is plausible, but the advertised cron job feature is not reachable through the public Gateway contract and the persisted header secrecy policy is unresolved.

## What This Changes

The branch adds per-agent provider request config and cron payload provider header overrides, wires them into embedded model and cron resolution, and adds provider, cron, and session-send tests.

## Best Possible Solution

Land this feature only after defining the cron header secrecy policy, adding the public Gateway cron schema and generated client/docs follow-through, and proving cron create/update/run behavior end to end.

## Reproduction Assessment

Yes for the PR defects. Source inspection on PR head shows internal `payload.providers` support was added while the public cron schema remains closed without that field, so supported `cron.add` and `cron.update` reject the advertised job-level header path.

## Solution Assessment

No. Agent-level provider request overrides are a reasonable part of the solution, but cron job headers need a public Gateway contract plus a redaction, denylist, or SecretRef-only policy before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:200`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `CronPayloadSchema` and `CronPayloadPatchSchema` still use a closed agentTurn schema with no `providers` field. Public `cron.add` and `cron.update` validate through that schema before persistence, so supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.96
- **[P2] Protect cron header values before returning jobs:** `src/cron/types.ts:200`
  - body: `payload.providers.*.request.headers` can now store arbitrary SecretInput header values, but existing `cron.list` and `cron.get` return the raw job payload to `operator.read` callers. Before exposing this mutable cron payload field, add a clear policy such as SecretRef-only storage, sensitive-header denial, or read-time redaction so headers like `Authorization` or `Cookie` are not disclosed.
  - confidence: 0.9
