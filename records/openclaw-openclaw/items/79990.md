## Summary

Keep open. The latest head fixes the earlier unused-import blocker, but the cron half of the feature is still blocked because `payload.providers` is only added to internal cron types/service code; the public Gateway cron schema still rejects it, and persisted request headers need a secrecy policy before exposure.

## What This Changes

The PR adds per-agent provider request overrides and per-cron-job provider header overrides with global-to-agent-to-job precedence, resolver wiring, config typing, cron persistence changes, and focused tests.

## Best Possible Solution

Keep the PR open for a maintainer-approved cron protocol and header secrecy design, then add schema/client/docs/test coverage and full Gateway cron proof.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: PR head adds `payload.providers` to cron types while the Gateway cron schema remains closed to that field, and cron get/list return persisted jobs. I did not run tests because this read-only review cannot create artifacts.

## Solution Assessment

No. The agent-level resolver direction looks viable, but cron job headers should land as a full protocol contract with schema/client/docs/tests and a secrecy policy, not as an internal payload-only field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:205`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but the Gateway `cronAgentTurnPayloadSchema` still has `additionalProperties: false` and no `providers` field. `cron.add` and `cron.update` calls that use the new field will be rejected before cron persistence, so the advertised cron feature is not reachable through the public protocol.
  - confidence: 0.94
- **[P2] Protect persisted cron headers before exposing them:** `src/cron/types.ts:205`
  - body: `payload.providers.*.request.headers` can contain credential-like headers such as `Authorization` or `Cookie`, and the PR persists those values in cron jobs that are returned by `cron.list` and `cron.get`. Please define a redaction, denylist, or SecretRef-only policy before exposing this mutable cron payload field.
  - confidence: 0.86
