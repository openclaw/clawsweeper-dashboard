## Summary

Keep this PR open: the agent-level resolver direction looks plausible, but the cron-job header feature is still not reachable through the public Gateway cron contract and it exposes persisted header values without an approved secrecy policy.

## What This Changes

The PR adds per-agent provider request overrides, per-cron-job provider header overrides, merge precedence wiring, cron persistence changes, and focused tests.

## Best Possible Solution

Keep the PR open for a full Gateway cron protocol contract plus a header secrecy policy, then update schema/client/docs/tests and provide full cron create/update/run proof.

## Reproduction Assessment

Yes for the blocking findings by source inspection: PR head adds `payload.providers` to cron internals while the public cron schema remains closed to that field, and cron get/list return stored jobs. I did not run tests because this is a read-only review.

## Solution Assessment

No. Agent-level request overrides are a plausible direction, but cron job headers should land through the Gateway protocol schema with docs/tests and a clear redaction, denylist, or SecretRef-only policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:205`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `cronAgentTurnPayloadSchema` still has no `providers` field and is closed with `additionalProperties: false`. `cron.add` and `cron.update` validate through that schema before persistence, so supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.95
- **[P2] Protect persisted cron headers before exposing them:** `src/cron/types.ts:205`
  - body: `payload.providers.*.request.headers` accepts arbitrary header names and values, then the PR persists those values in cron jobs returned by `cron.list` and `cron.get`. Headers such as `Authorization` or `Cookie` should have a redaction, denylist, or SecretRef-only policy before this mutable cron payload field is exposed.
  - confidence: 0.87
