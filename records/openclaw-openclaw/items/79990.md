## Summary

Keep open: the follow-up commits address the earlier broad transport, shallow header merge, and unrelated hash findings, but the public cron RPC schema still rejects the new `payload.providers` field, so job-level headers cannot be configured through `cron.add` or `cron.update`. The provided proof is useful helper-level terminal output, but it does not exercise the full gateway cron create/update/run path where this mismatch lives.

## What This Changes

The PR adds agent-level provider request overrides, cron job payload provider header overrides, resolver merge plumbing, cron payload persistence, and focused tests.

## Best Possible Solution

Update the cron protocol schemas, generated protocol/UI types, docs, and validator tests so headers-only provider overrides round-trip through cron create/update/list, then add full gateway cron proof for the behavior.

## Reproduction Assessment

Not applicable as a current-main bug report; yes for the PR finding. Source inspection shows `cron.add` and `cron.update` still validate payloads against schemas that reject `payload.providers` before the cron service can persist it.

## Solution Assessment

No. The per-agent resolver direction is plausible, but the cron job feature is incomplete until the public protocol/schema boundary accepts the new headers-only field and maintainers approve the job-scoped header policy.
