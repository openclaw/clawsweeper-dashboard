## Summary

Keep open: the feature is useful, but the PR still has blocking cron contract and secret-handling issues. Job-level headers are added to internal cron payload types while the public Gateway cron schema still rejects them, and the patch introduces persisted header values without a redaction, denylist, or SecretRef-only policy.

## What This Changes

The PR adds agent-level provider request overrides, job-level cron provider header overrides, merge precedence wiring, and tests across agent/provider/cron paths.

## Best Possible Solution

Land this only after the public cron protocol contract, generated/client/docs follow-through, cron header secrecy policy, and end-to-end Gateway cron proof are aligned.

## Reproduction Assessment

Yes. Source inspection of PR head shows `payload.providers` added to internal cron payload types while the public cron agentTurn payload schema remains closed without `providers`, and Gateway cron add/update validate through that schema before persistence.

## Solution Assessment

No. Agent-level provider request overrides are plausible, but cron job headers need a public Gateway contract plus a redaction, denylist, or SecretRef-only policy before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Add providers to the cron protocol schema:** `src/cron/types.ts:199-200`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but `cronAgentTurnPayloadSchema` still has no `providers` field and is closed with `additionalProperties: false`. `cron.add` and `cron.update` validate through that schema before persistence, so supported Gateway calls with job-level headers are rejected and the advertised cron feature is unreachable.
  - confidence: 0.95
- **[P2] Protect persisted cron headers before exposing them:** `src/cron/types.ts:200`
  - body: `payload.providers.*.request.headers` accepts arbitrary header names and `SecretInput` values, then the service persists those values in cron jobs returned by `cron.list` and `cron.get`. Headers such as `Authorization` or `Cookie` need redaction, a denylist, or SecretRef-only handling before this mutable cron payload field is exposed.
  - confidence: 0.88
