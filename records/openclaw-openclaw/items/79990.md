## Summary

Keep open: this is a useful feature direction, but PR head still leaves the advertised cron job header surface unreachable through Gateway validation and misses configured inline provider models; the mutable cron header policy also needs maintainer approval.

## What This Changes

The PR adds agent-level provider request overrides, cron job payload provider headers, merge precedence, cron payload persistence, and resolver/cron tests.

## Best Possible Solution

Land a narrower complete version only after the additive cron protocol/schema/docs/client contract, inline model-resolution branch, tests, full Gateway cron proof, and cron header validation/redaction policy are aligned.

## Reproduction Assessment

Yes for the PR findings by source inspection: PR head adds internal cron `payload.providers` while the Gateway schema rejects unknown payload fields, and configured inline provider models return before the new request-override merge path. This is a proposed-patch failure path, not a current-main bug.

## Solution Assessment

No. The per-agent resolver direction is plausible, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field, inline configured models are covered, and maintainers approve the header security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds job-scoped `payload.providers` to cron payload types and the service merge path, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. Supported `cron.add` and `cron.update` calls carrying these headers will be rejected by validation before they reach `context.cron.add/update`, so the advertised per-job header feature cannot be configured through the RPC surface.
  - confidence: 0.94
- **[P2] Apply overrides to inline provider models:** `src/agents/pi-embedded-runner/model.ts:713`
  - body: When a configured provider model already has an API, `resolveExplicitModelWithRegistry` returns the `inlineMatch` directly instead of running the new provider-request merge. That leaves documented `models.providers.*.models[]` custom models with only global request headers, so `agents.list[].providers...` and cron job headers are ignored for this model shape.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a persisted, mutable cron payload header surface without a validated policy for auth-like header names or API redaction.

Concerns:

- **[medium] Define cron header policy:** `src/cron/types.ts:196`
  - body: `CronJobProviderRequest.headers` intentionally excludes auth/proxy/TLS fields, but arbitrary headers can still carry credentials such as Authorization or Cookie, and cron payloads are persisted and returned by job APIs. Validate or deny sensitive header names, or document the maintainer-approved redaction and exposure policy before exposing this through the public cron API.
  - confidence: 0.84
