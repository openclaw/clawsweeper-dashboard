## Summary

Keep open: the feature is plausibly useful, but the patch is not mergeable as-is. The cron-job header surface is added to internal cron types and service merging without updating the Gateway cron payload schema, the PR head leaves an unused type import under `noUnusedLocals`, and the security/proof gaps around mutable persisted headers still need maintainer handling.

## What This Changes

The PR adds per-agent provider request overrides and per-cron-job provider header overrides with global-to-agent-to-job merge precedence, cron payload persistence, and resolver/cron tests.

## Best Possible Solution

Align the agent config and cron protocol surfaces with validators, docs, tests, and generated clients; define cron header validation/redaction policy; remove the build break; then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR findings by source inspection: PR head adds internal `payload.providers`, while the Gateway cron schema still rejects unknown agent-turn payload properties, and the same file has an unused import under `noUnusedLocals`. This is proposed-patch review, not a current-main bug reproduction.

## Solution Assessment

No. The per-agent resolver direction may be viable, but the cron-job feature is incomplete until the public protocol boundary, generated clients/docs, header security policy, and full-path proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds job-scoped `payload.providers` to cron payload types and the service merge path, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. Supported `cron.add` and `cron.update` calls carrying these headers fail validation before reaching `context.cron.add/update`, so the advertised per-job header feature cannot be configured through the public RPC surface.
  - confidence: 0.95
- **[P2] Remove the unused cron provider request import:** `src/cron/types.ts:3`
  - body: After narrowing cron job overrides to `CronJobProviderRequest`, `ConfiguredModelProviderRequest` is imported but never referenced. Core typecheck uses `noUnusedLocals`, so this leaves the PR with a straightforward build/typecheck failure until the import is removed.
  - confidence: 0.98

## Security Review

Status: needs_attention

Summary: The diff introduces a mutable, persisted cron payload header surface that needs an explicit validation and redaction policy before merge.

Concerns:

- **[medium] Define cron header validation and redaction policy:** `src/cron/types.ts:196`

Add a maintainer-approved allow/deny/redaction policy before exposing this through the public cron API.
