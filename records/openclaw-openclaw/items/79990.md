## Summary

Keep open: the per-agent direction is plausible, but the cron-job feature is not mergeable because the public Gateway cron schema still rejects `payload.providers`, the proof only exercises helper-level header merging, and the mutable cron header security policy is unresolved.

## What This Changes

The PR adds per-agent provider request overrides and per-cron-job provider request headers with global-to-agent-to-job precedence and focused resolver/cron tests.

## Best Possible Solution

Land this only after the agent config, cron protocol schema/runtime validators, docs/generated clients, and cron header validation/redaction policy are aligned, then prove a full Gateway cron create/update/run path against a mock provider.

## Reproduction Assessment

Yes for the proposed-patch failure path by source inspection: the PR adds internal `payload.providers`, while the Gateway cron schema remains closed and validation runs before cron service persistence. This is not a current-main bug reproduction; it is a PR correctness issue.

## Solution Assessment

No, not as-is. The per-agent resolver direction may be maintainable, but the cron job feature needs protocol wiring, docs/generated artifacts, security policy, and full-path proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds job-scoped `payload.providers` to cron payload types and the service merge path, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. Calls to `cron.add` or `cron.update` carrying these headers will be rejected by validation before reaching the cron service, so the advertised per-job header feature cannot be configured through the public RPC surface.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a mutable persisted cron header surface without a validated policy for auth-like header names, redaction, or API exposure.

Concerns:

- **[medium] Define cron header policy:** `src/cron/types.ts:196`
  - body: `CronJobProviderRequest.headers` intentionally excludes auth/proxy/TLS fields, but arbitrary headers can still carry credentials such as `Authorization` or `Cookie`, and cron payloads are persisted and returned by job APIs. Validate or deny sensitive header names, or document the maintainer-approved redaction and exposure policy before exposing this through the public cron API.
  - confidence: 0.82
