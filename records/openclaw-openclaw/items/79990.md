## Summary

Keep open: the PR still has a blocking cron protocol gap. PR head adds internal `payload.providers` support, but the Gateway cron schema still rejects that field before the cron service/runtime changes can use it.

## What This Changes

The PR adds per-agent provider request overrides, cron job-level provider headers, global-to-agent-to-job merge precedence, and resolver/cron persistence tests.

## Best Possible Solution

Land this only after the additive config/protocol contract, validators, docs, generated clients, tests, full-path proof, and cron header validation/redaction policy are aligned.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron `payload.providers`, while the Gateway cron schema still rejects unknown agent-turn payload fields before `cron.add` or `cron.update` reaches the service. This is a proposed-patch failure path, not a current-main bug.

## Solution Assessment

No as-is. The per-agent resolver direction may be viable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve validation/redaction policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire providers through the cron RPC schema:** `src/cron/types.ts:196`
  - body: The PR adds `payload.providers` to the internal cron payload type and service merge path, but the Gateway cron payload schema still has `additionalProperties: false` and no `providers` property. ## Security Review

Status: needs_attention

Summary: The diff adds a mutable persisted cron header surface without an approved validation or redaction policy for auth-like header names.

Concerns:

- **[medium] Define cron header validation and redaction:** `src/cron/types.ts:196`

Add validation/deny-listing or document and implement redaction before exposing this through the public cron API.
