Full review comments:

- **[P1] Add cron providers to the Gateway protocol schema:** `src/cron/types.ts:250`
  - body: This adds `payload.providers` to the cron runtime types/service, but the supported Gateway path still validates cron.add/update against `packages/gateway-protocol/src/schema/cron.ts`, whose agent-turn payload schema has `additionalProperties: false` and no `providers` field. As a result, users cannot create or patch the new per-job headers through the public Gateway API.
  - confidence: 0.93
- **[P1] Enforce header-only cron provider requests at runtime:** `src/cron/isolated-agent/run.ts:563-565`
  - body: The PR says cron payload provider overrides are headers-only, but this merge spreads the entire job `request` object into agent provider config. Because persisted/plugin-created cron payloads are runtime data, any `auth`, `proxy`, `tls`, or `allowPrivateNetwork` keys that reach this path would be sanitized as normal model-provider request overrides and could change transport policy.
  - confidence: 0.88
- **[P1] Define redacted cron header readback before exposing providers:** `src/cron/types.ts:250`
  - body: `payload.providers.*.request.headers` uses `SecretInput`, but cron handlers return job objects directly for add/get/list/update flows. Once the protocol accepts this field, raw provider headers or tenant identifiers can be echoed back unless the PR makes cron headers SecretRef-only or redacts them from public job readback.
  - confidence: 0.82
