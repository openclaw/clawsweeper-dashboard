## Summary

Keep open: this active feature PR has a plausible direction, but it is not merge-ready because the cron override is still internal-only at the Gateway protocol boundary, and the job-scoped header policy plus full cron-path proof need follow-up.

## What This Changes

The branch adds agent-scoped provider request overrides, cron job-scoped provider header overrides, merge precedence plumbing, and focused resolver/cron persistence tests.

## Best Possible Solution

Land this only after the additive config/protocol contract, validators, generated clients, docs, tests, and cron header validation policy are aligned, then prove a full Gateway cron create/update/run against a mock provider.

## Reproduction Assessment

Yes for the PR finding by source inspection: PR head adds internal cron payload.providers, while the Gateway protocol schemas used by cron.add and cron.update still reject unknown agent-turn payload fields. This is not a current-main bug reproduction; it is a proposed-patch failure path.

## Solution Assessment

No. The per-agent resolver direction may be acceptable, but the cron job feature is incomplete until the public protocol boundary accepts the headers-only field and maintainers approve validation for mutable job-scoped headers.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Wire providers into the cron RPC schemas:** `src/cron/types.ts:177`
  - body: The PR adds job-scoped `payload.providers` to the cron payload type and service merge path, but the Gateway protocol schema for agent-turn cron payloads still has `additionalProperties: false` and no `providers` property.
