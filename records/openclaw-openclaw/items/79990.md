## Summary

Keep open. The PR has useful per-agent provider request plumbing, but current head still exposes cron job payload transport overrides without an approved trust-boundary decision, still drops non-conflicting agent headers on the cron preparation path, and still includes unrelated generated Canvas hash drift.

## What This Changes

The branch adds agent and cron payload provider request override fields, threads request overrides into PI model resolution, persists cron payload providers, and adds focused tests.

## Best Possible Solution

Keep the per-agent direction, route cron job overrides through the shared request merge path or a prepared job request value, restrict or approve the job-scoped transport surface, remove unrelated generated drift, and document the new public config surface before merge.

## Reproduction Assessment

Not applicable as a feature PR rather than a current-main bug report. The remaining correctness issue is source-reproducible from the PR diff: the cron preparation path shallow-merges request.headers before the shared merge helper can preserve agent headers.

## Solution Assessment

No. The per-agent operator config direction is plausible, but cron job overrides need an explicit security/product boundary and should be merged through the existing request helper or passed as a prepared job-scoped request instead of mutating agent config with a shallow nested merge.
