## Summary

Keep open. Current main and the latest release still emit a localized natural-language `Current time:` line plus `Reference UTC:`, and the heartbeat path passes the heartbeat `startedAt` value into that shared line without a separate trigger timestamp; the ISO-specific linked PR was closed unmerged while the merged UTC-split PR only partially addressed the ambiguity.

## What This Changes

Request to make heartbeat current-time prompt context ISO 8601 and distinguish the current/processing time from the heartbeat trigger timestamp.

## Best Possible Solution

Choose a single model-visible timestamp contract for cron-style prompt inserts, then update the shared current-time helper and all affected caller tests together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the heartbeat body is built through `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)`, and that helper still emits the non-ISO localized `Current time:` block with no separate trigger timestamp. I did not run a live heartbeat because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the desired prompt-time contract. The maintainable implementation path is likely the shared helper, but the exact fields and whether `Current time` should mean processing time, trigger time, or both is a product decision.
