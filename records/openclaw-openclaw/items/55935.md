## Summary

Keep open. Current main and the latest release still build heartbeat prompt bodies with the shared non-ISO `Current time` / `Reference UTC` block using the heartbeat `startedAt` timestamp, and the related merged PR only split UTC onto its own line rather than adding ISO 8601 or a separate trigger timestamp.

## What This Changes

Request to change heartbeat-injected `Current time` prompt context to ISO 8601 and optionally expose a separate heartbeat trigger timestamp.

## Best Possible Solution

Adopt one shared prompt-time contract that keeps stable system prompts cacheable, uses machine-readable ISO timestamps where exact time is intentionally injected, and separates current processing time from heartbeat trigger time with focused coverage.

## Reproduction Assessment

Yes, from source inspection: heartbeat runs pass `startedAt` into the shared current-time helper, and that helper emits natural-language `Current time` plus non-ISO `Reference UTC` with no separate trigger timestamp. I did not run a live heartbeat in this read-only review.

## Solution Assessment

Unclear. The requested ISO format and trigger timestamp are plausible, but the best fix needs maintainer agreement on the shared heartbeat/cron/reset/compaction prompt-time contract and cache boundary.
