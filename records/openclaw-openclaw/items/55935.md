## Summary

Keep open. Current main and the latest release still emit the shared cron/heartbeat `Current time:` line in the old locale-style format with a non-ISO UTC suffix, heartbeat still uses `startedAt` for that line, and there is no separate heartbeat trigger timestamp. The closed implementation PR #55957 did not merge, while the currently open related PRs either touch adjacent surfaces or only partially address the prompt-format problem.

## What This Changes

Request to change heartbeat time context to an ISO 8601 `Current time` value and optionally add a distinct `Heartbeat triggered at` timestamp.

## Best Possible Solution

Adopt a shared ISO 8601 local timestamp with explicit offset for cron-style prompt time lines, keep any UTC anchor clearly secondary, and add a heartbeat-only trigger timestamp only if maintainers want to distinguish scheduler/run-start time from the agent processing-time context. Update the shared helper consumers and timestamp-detection tests together.

## Reproduction Assessment

Yes. The current-main source and existing tests give a high-confidence reproduction path: `resolveCronStyleNow` deterministically builds the old non-ISO line, and heartbeat passes `startedAt` into that helper without adding any separate trigger timestamp.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the shared helper contract and its tests, with any `Heartbeat triggered at` addition kept in the heartbeat path rather than leaking into all cron-style consumers. PR #42654 is not sufficient as-is because it still leaves `Current time` in locale prose instead of ISO 8601.
