## What This Changes

The issue reports Feishu Bitable table-level operations returning HTTP 400 after replacing the Feishu bot/app while read-oriented Feishu operations still work.

## Best Possible Solution

Keep the issue open until a Feishu owner can collect the exact API error/log_id and choose the plugin-owned fix: clearer permission/setup diagnostics if Feishu is rejecting the resource, or an explicit user-token auth design if tenant-token Bitable access is insufficient after app replacement.

## Reproduction Assessment

No for the full HTTP 400: a live Feishu tenant, exact response body, and log_id are still needed. Source inspection does reproduce the relevant current-main path: all reported Bitable table operations use the appId/appSecret SDK client, which defaults to tenant access token unless a user token is explicitly supplied.

## Solution Assessment

Unclear. A narrow diagnostics/docs fix may be best if Feishu is rejecting permissions or resource sharing, but changing Bitable table operations to use user access tokens would be a product/auth-surface decision rather than a mechanical bug fix.
