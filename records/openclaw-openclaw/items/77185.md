## Summary

Current main and release v2026.5.3 contain the generic startup-load-plan fix for selected external context-engine plugins, so the reported 2026.5.2 lossless-claw fallback is already implemented and shipped.

## What This Changes

Report that lossless-claw remains unregistered as the selected context engine on Windows npm global OpenClaw 2026.5.2, with duplicate plugin-id diagnostics from stale install paths.

## Best Possible Solution

Keep the shipped generic startup-plan implementation and direct any remaining duplicate-source shadowing after update to the plugin doctor/registry repair path with fresh diagnostics.

## Reproduction Assessment

Yes for the historical failure path: a selected external context-engine plugin that was not loaded during gateway startup would reach resolveContextEngine as unregistered and fall back to legacy. I did not run a live Windows npm gateway, but current source shows that selected slot is now included in startup plugin ids before engine resolution.

## Solution Assessment

Yes. Loading the selected plugins.slots.contextEngine owner through the generic gateway startup plan is the narrow maintainable fix, avoids a lossless-claw special case, and matches the documented plugin slot contract.
