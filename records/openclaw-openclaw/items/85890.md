## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` path when exec approvals writes its store, but the existing tests and changelog show this is a security-sensitive boundary rather than a safe automatic cleanup close.

## What This Changes

The issue requests exec approvals support for `~/.openclaw` as a symlink, by allowing the root-child symlink parent and following the directory symlink for the final directory check.

## Best Possible Solution

Define whether an immediate-child `~/.openclaw` symlink is a supported operator-owned state layout; if yes, update the guard with focused tests that allow only that case while preserving rejection for deeper or unsafe symlink traversal.

## Reproduction Assessment

Yes, by source inspection: a symlinked `~/.openclaw` reaches `assertNoSymlinkParentsSync` without `allowRootChildSymlink`, and the later `lstatSync` directory check would reject the symlink too. I did not run a live temp-home repro because this review is read-only.

## Solution Assessment

Unclear as proposed: adding `allowRootChildSymlink` and switching to `statSync` is mechanically plausible, but it would reverse an existing tested policy that rejects symlinked `.openclaw` components. The best fix needs an explicit security/product decision and regression tests for allowed and rejected symlink layouts.
