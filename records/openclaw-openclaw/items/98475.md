## Summary

Keep open. The PR threads the sandbox container workdir through the right send/reply media paths, but it now lets raw configured workdir strings drive media matching; current config accepts `docker.workdir` as an unnormalized string, so existing `/workspace/` setups can stop mapping `/workspace/...` media, and the external PR still lacks real OpenShell media-transfer proof.

## What This Changes

The PR adds an optional sandbox container workdir through message-tool, outbound attachment normalization, media-list normalization, and reply-media path normalization so non-/workspace sandbox paths can map back to the host sandbox root.

## Best Possible Solution

Canonicalize the container workdir at the sandbox media mapping boundary, cover default `/workspace`, `/workspace/`, and OpenShell `/sandbox` through message-action and reply-media tests, then merge the single complete candidate after real behavior proof.

Yes, from source: current main hard-codes `/workspace` in sandbox media mapping while OpenShell defaults to `/sandbox`, and the PR head shows the new raw workdir matching edge case. I did not run the live Kubernetes/OpenShell flow.

## Solution Assessment

No, not yet. Threading `containerWorkdir` through the production send/reply paths is the right fix shape, but the mapper should canonicalize the workdir and the PR needs real OpenShell media-transfer proof before merge.
