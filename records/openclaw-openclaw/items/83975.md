## Label Justifications

- `P2`: The bug breaks self-hosted or reverse-proxied ClawHub deployments with a path prefix, but the affected surface and fix are narrow.

## Summary

This issue should stay open: current main still builds ClawHub request URLs with root-relative API paths, so a configured non-root `OPENCLAW_CLAWHUB_URL` or `CLAWHUB_URL` pathname is dropped before the request is sent. The bug is narrow, source-reproducible, and suitable for a focused fix plus regression coverage.

## What This Changes

The issue requests that OpenClaw preserve non-root path prefixes from `OPENCLAW_CLAWHUB_URL` and `CLAWHUB_URL` when constructing ClawHub API request URLs.

## Best Possible Solution

Update the shared ClawHub URL builder to join normalized base URLs and API paths without discarding an existing base pathname, then add regression coverage for both ClawHub URL environment variables with a prefixed base URL.

## Reproduction Assessment

Yes from source inspection: current main passes leading-slash ClawHub API paths into a `new URL()` pattern that discards a configured base pathname. I did not run a live reverse-proxy ClawHub instance during this read-only review.

## Solution Assessment

Yes: fixing the shared `src/infra/clawhub.ts` URL construction is the narrowest maintainable solution because every affected ClawHub endpoint already flows through that helper.
