## Summary

Keep open: current main still lacks a provider-scoped dynamic visibility policy, and the Discord picker path can expose only the first provider page unless the desired provider is already the active/default provider. The requested direction adds a new config and picker policy contract, so it needs maintainer product review rather than conservative cleanup closure.

## What This Changes

Request provider-level model visibility filtering so `/model` and `/models` can show dynamically discovered models only for selected providers.

## Best Possible Solution

Define an additive provider-scoped visibility policy, or choose provider search/pagination instead, then apply the chosen contract through the shared visible catalog so text commands, Gateway clients, and Discord stay aligned while preserving exact `agents.defaults.models` precedence.

## Reproduction Assessment

Yes, by source inspection: the shared catalog path has no provider-scoped dynamic allowlist, and the Discord picker starts at provider page 1 while rendering only that page's provider options. I did not run a live Discord interaction.

## Solution Assessment

Unclear: the proposed provider-level config is plausible, but maintainers should choose between a new shared visibility contract and a Discord picker UX fix such as searchable or navigable provider pages.
