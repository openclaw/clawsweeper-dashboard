## Summary

Keep open. The idea is useful, but this PR is stale against the current memory provider architecture: it adds a public config key while most request/parse changes land in retired `src/memory` files and do not reach the active memory-host SDK or bundled provider plugins.

## What This Changes

Adds an optional `memorySearch.remote.encodingFormat` config field and attempts to pass `encoding_format` through remote embedding requests, including base64 response decoding for direct and batch paths.

## Best Possible Solution

Implement the feature, if desired, through the active memory-host SDK and provider-plugin contract with explicit provider compatibility rules, generated config schema/help/docs, direct and batch request coverage, base64 decoding, and cache or reindex implications handled deliberately.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `agents.defaults.memorySearch.remote.encodingFormat` on current main and the strict schema rejects it; applying this PR as-is still misses the active resolver, SDK provider contract, and OpenAI/Voyage plugin paths.

## Solution Assessment

No. The proposed patch is not the best current fix because it updates retired memory runtime files and a partial config surface instead of the active provider-plugin architecture and generated public config surfaces.
