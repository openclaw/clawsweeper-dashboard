## Summary

Keep this PR open, but it is not merge-ready: the requested capability is still not implemented on current main, while the branch now targets removed `src/memory` implementation files and lacks after-fix real behavior proof.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config key and tries to send/decode `encoding_format` for direct and batch remote embedding requests.

## Best Possible Solution

Decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through active config schema/types/help, memory-host SDK contracts, OpenAI-compatible provider adapters, direct and batch parsers, docs, tests, and redacted live-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows current main's active embedding paths do not carry `encodingFormat`, while the branch edits stale `src/memory` files. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses the active memory-host/provider surfaces, does not preserve the option through resolved config, and lacks real behavior proof.
