## Summary

Keep open: the feature may be useful, but this PR is not mergeable as-is because current main moved memory embedding runtime into the memory-host SDK and provider plugins while the branch still edits stale `src/memory` paths. It also lacks real-provider or end-to-end memory-search proof and needs maintainer API/product judgment before a replacement implementation.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config field and tries to pass/decode `encoding_format` for remote embedding request and batch paths.

## Best Possible Solution

Maintainers should first decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through the active schema, resolver, memory-host SDK contracts, provider adapters, direct and batch parsers, docs, focused tests, and real-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection against current main shows active embedding/config paths do not carry `encodingFormat`, while the branch edits stale `src/memory` files. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses active memory-host/provider surfaces, does not preserve the option through resolved config, and lacks real behavior proof.
