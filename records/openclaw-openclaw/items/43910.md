## Summary

Keep open for maintainer review: the requested configurable embedding encoding is not implemented on current main, but this PR is not mergeable as-is because it edits stale `src/memory` implementation files while the active embedding runtime now lives in `packages/memory-host-sdk` and provider plugins. It also still lacks contributor proof from a real OpenAI-compatible embedding request or end-to-end memory search.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting and attempts to send and decode `encoding_format` for direct and batch remote embedding requests.

## Best Possible Solution

Decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through the active config schema/types/help, memory-host SDK contracts, OpenAI-compatible provider adapters, direct and batch parsers, docs, tests, and redacted live-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows current main's active embedding paths do not carry `encodingFormat`, while the branch edits stale `src/memory` files. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses the active memory-host/provider surfaces, does not preserve the option through resolved config, and lacks real behavior proof.
