## Summary

Keep open: the configurable embedding encoding-format capability is not implemented on current main, and the PR still points at a plausible feature, but this branch is stale and incorrect against the current memory-host SDK/provider architecture and lacks real behavior proof.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config key and passes/decodes `encoding_format` for remote embedding direct and batch paths in the old `src/memory` implementation.

## Best Possible Solution

Maintainers should first decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through the active schema, resolver, memory-host SDK contracts, provider adapters, direct and batch parsers, docs, focused tests, and real-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection on current main shows the active config/provider paths omit `encodingFormat`, while the PR diff edits removed or stale `src/memory` embedding files. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses active memory-host/provider surfaces, does not preserve the option through resolved config, and lacks real behavior proof.
