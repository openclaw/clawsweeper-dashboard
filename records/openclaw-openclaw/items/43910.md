## Summary

Keep open. The feature is plausible, but this PR is not merge-ready on current main: it adds the config key and request/response handling mostly in the retired `src/memory` implementation, while the active plugin-backed memory paths still do not accept, resolve, send, or decode `encodingFormat`. Current `memory-lancedb` documentation also shows an intentional provider-compatibility stance around omitting `encoding_format`, so this needs maintainer product/rebase judgment rather than cleanup closure.

## What This Changes

This PR adds optional `memorySearch.remote.encodingFormat` support for float/base64 embedding requests, base64 response decoding, and related config/schema/type labels across the old remote memory embedding paths.

## Best Possible Solution

If maintainers want this capability, implement it on the current memory-host SDK and provider-plugin paths with a clear provider-compatibility policy, generated config schema/help/docs, direct and batch OpenAI-compatible request coverage, robust base64 float32 decoding, cache/reindex implications, and focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `agents.defaults.memorySearch.remote.encodingFormat` and trace current main plus this PR diff through schema, `resolveMemorySearchConfig`, `MemoryEmbeddingProviderCreateOptions`, and the active OpenAI direct/batch providers; the value is not carried through the active paths.

## Solution Assessment

No. This is not the best current fix because it is based on retired `src/memory` files and does not update the active SDK/plugin plumbing or settle the documented provider-compatibility tradeoff.
