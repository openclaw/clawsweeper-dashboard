## Summary

Keep open: the feature direction is plausible, but this PR is not ready because it still drops the new config in the memory-search merge path, targets pre-refactor `src/memory` implementation files, and has only mock/code-review proof for the real embedding behavior.

## What This Changes

The branch adds `memorySearch.remote.encodingFormat` schema/types/labels, conditionally sends `encoding_format`, and decodes base64 embeddings in direct and batch memory embedding paths.

## Best Possible Solution

Rebase onto current `main`, implement the option through the active memory-search config merge and memory-host-sdk provider/parser paths, keep omit-by-default behavior, and add focused config/request/decode tests plus redacted real-provider proof.

## Reproduction Assessment

## Solution Assessment

No. The direction is reasonable, and the later commits address the initial base64 decode criticism in old files, but the PR must move to current active paths and preserve the documented provider-compatibility policy before it is a maintainable fix.
