## Summary

This PR should stay open for maintainer/contributor follow-up rather than cleanup-close: the feature request is still not implemented on current main, but the branch is stale and currently patches removed `src/memory` paths instead of the active memory-host SDK/provider-plugin surfaces.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting for `float` or `base64` embedding requests and response parsing in the old memory embedding implementation.

## Best Possible Solution

Decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, port it through the active resolved config, memory-host SDK contracts, provider adapters, docs/help text, tests, and redacted live-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the branch patches stale `src/memory` paths and current main's active config/provider paths do not carry `encodingFormat`. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it does not port through current active memory-host/provider surfaces or include real behavior proof.
