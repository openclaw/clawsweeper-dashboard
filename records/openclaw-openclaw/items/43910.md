## Summary

Keep open: configurable embedding `encoding_format` is still not implemented on current main and the related hardcoded-float PR was closed in favor of this configurable direction, but this branch is not merge-ready because the new option is dropped before it reaches the client and the branch targets the old `src/memory` layout.

## What This Changes

The PR adds a `memorySearch.remote.encodingFormat` config key and threads `encoding_format` plus base64 decoding through remote embedding requests and batch outputs.

## Best Possible Solution

Implement an opt-in encoding format on the current memory host SDK/OpenAI-compatible adapter path, propagate it through resolved memory config, preserve omit-by-default behavior, and cover direct plus batch requests with regression tests.

## Reproduction Assessment

Yes for the review finding: source inspection of PR head shows the new value is read in the embedding client but omitted from the resolved `remote` object built by `src/agents/memory-search.ts`. I did not live-run the branch because this review is read-only.

## Solution Assessment

No. The direction is plausible, but the patch should be rebased to the current `packages/memory-host-sdk` and provider extension paths, propagate the config end to end, and use the existing DataView-style base64 validation pattern.
