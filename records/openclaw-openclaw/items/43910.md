## Summary

Keep open: the feature is not implemented on current main, and this PR still has merge-blocking correctness gaps because it edits stale memory files instead of the active memory-host SDK and provider-plugin paths. It also still lacks after-fix proof from a real provider or end-to-end memory-search run.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config key and attempts to send and decode `encoding_format` for remote embedding request and batch paths.

## Best Possible Solution

Maintainers should first decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, rewrite it through the active schema/types/help, resolver, memory-host SDK contracts, provider adapters, direct and batch parsers, docs, focused tests, and real-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection against current main shows active embedding/config paths do not carry `encodingFormat`, while the branch edits stale `src/memory` files. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses active memory-host/provider surfaces, does not preserve the option through resolved config, and lacks real behavior proof.
