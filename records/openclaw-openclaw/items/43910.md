## Summary

Keep open. Current main still lacks configurable `memorySearch.remote.encodingFormat`, and this PR is stale against the active memory-host/provider-plugin architecture, so it would not make the setting work for normal config-driven memory embeddings.

## What This Changes

This PR adds optional `memorySearch.remote.encodingFormat` support, threads it through old remote memory embedding request and batch code, and decodes base64 embedding responses.

## Best Possible Solution

Implement this only on the current memory-host SDK and provider-plugin paths, with an explicit provider-compatibility policy, generated schema/help/docs, direct and batch request coverage, robust base64 decoding, and cache or reindex implications handled deliberately.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `agents.defaults.memorySearch.remote.encodingFormat` on current main and the strict schema rejects it; applying this PR as-is would still miss the active resolver, provider contract, OpenAI plugin, and shared parser paths.

## Solution Assessment

No. This is not the best current fix because it updates retired `src/memory` files and a partial schema surface instead of the active provider-plugin architecture and public generated config surfaces.
