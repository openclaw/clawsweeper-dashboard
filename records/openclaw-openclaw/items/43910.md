## Summary

Keep open, but do not merge this branch as-is: current main does not implement a generic configurable embedding `encodingFormat`, and this PR still targets removed/stale `src/memory` paths instead of the active memory-host SDK and provider-adapter surfaces.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting and attempts to send/decode `encoding_format` for direct and batch remote embedding requests in old memory embedding files.

## Best Possible Solution

Decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through the active resolved config, SDK contracts, provider adapters, direct and batch parsers, docs, tests, and redacted live-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the branch patches stale `src/memory` paths while current main's active config/provider paths do not carry `encodingFormat`. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it misses the active memory-host/provider surfaces and still lacks real behavior proof.
