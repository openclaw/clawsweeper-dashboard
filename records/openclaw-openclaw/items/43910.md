## Summary

Keep this PR open, but the current branch is not merge-ready: it adds a useful-looking public config knob on stale memory paths while current main routes embedding work through the memory-host SDK and provider plugins.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting and attempts to send `encoding_format` plus decode `float` or `base64` embedding responses.

## Best Possible Solution

Decide the provider compatibility policy, then implement the option in the active SDK/provider-plugin paths with generated config/help/docs updates and direct, batch, and base64 parser coverage.

## Reproduction Assessment

Yes. Source inspection is enough: current main does not accept or propagate `memorySearch.remote.encodingFormat`, and the PR's request/decoder changes target removed `src/memory` paths rather than the active SDK and plugin files.

## Solution Assessment

No. The maintainable fix is not the current branch as-is; it needs an active-path implementation plus generated config/docs and tests after the provider compatibility policy is settled.
