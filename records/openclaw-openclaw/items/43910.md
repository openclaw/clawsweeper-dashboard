## Summary

Keep this PR open for maintainer judgment, but it is not mergeable as-is: it adds a new configurable embedding encoding surface while editing stale `src/memory` paths that no longer exist on current main, and current memory-lancedb compatibility work intentionally omits `encoding_format` for providers that reject it.

## What This Changes

The PR adds optional `memorySearch.remote.encodingFormat` config and threads `encoding_format` through old remote embedding request, response, and batch code.

## Best Possible Solution

Decide whether OpenClaw should expose a generic embedding encoding-format knob; if yes, implement it through the active memory-host SDK, provider/plugin config contracts, docs/help text, tests, and real-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the branch patches removed `src/memory` paths, while current main's schema/resolved config and active providers do not carry `encodingFormat`. The requested capability itself is a new config feature, not a broken existing-behavior reproduction.

## Solution Assessment

No. The direction may be useful, but this patch is not the maintainable implementation because it does not target current active memory/provider surfaces and it lacks real behavior proof.
