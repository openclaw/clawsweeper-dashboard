## Summary

Keep open: the configurable embedding format request is not implemented on current main, and this branch is still not merge-ready because it targets removed memory implementation files, misses active config/provider surfaces, and lacks real behavior proof.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting and threads float/base64 request and response handling through old remote embedding config/types and direct/batch paths.

## Best Possible Solution

Decide whether OpenClaw should expose a generic encoding-format knob; if yes, implement it through active resolved config, memory-host SDK contracts, provider plugin direct/batch adapters, docs, tests, and redacted live-provider proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows current main lacks the branch's `src/memory` implementation paths and the active config/provider paths do not carry `encodingFormat`. The original request is a new configurable capability, not a broken existing-behavior bug reproduction.

## Solution Assessment

No. The direction may be useful, but this branch updates stale paths and needs a maintainer decision plus a port through the active memory-host/provider surfaces before it is the maintainable fix.
