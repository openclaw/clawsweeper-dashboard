## What This Changes

This PR changes `stopChannel` so a plugin `stopAccount` fallback receives `AbortSignal.abort()` instead of a fresh never-aborted signal when no account AbortController exists.

## Best Possible Solution

Refresh the branch onto current main, keep the already-aborted fallback if maintainers accept the plugin lifecycle semantics, add focused gateway coverage for explicit stop with no controller, and provide redacted terminal or log proof from a real shutdown/reload path.

## Reproduction Assessment

Yes, source-reproducible: an explicit `stopChannel(channel, accountId)` call with no stored abort/task but with a plugin `stopAccount` reaches the fallback and passes a fresh non-aborted signal on current main. I did not run tests because this cleanup review is read-only.

## Solution Assessment

Yes for the code shape, but not yet merge-ready: `AbortSignal.abort()` is the narrowest way to match the active-controller path after `abort?.abort()`. The final PR still needs branch refresh, focused coverage, maintainer acceptance of the plugin lifecycle semantics, and real behavior proof.
