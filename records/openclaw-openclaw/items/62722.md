## What This Changes

The PR changes `stopChannel` so the `stopAccount` fallback receives `AbortSignal.abort()` instead of a fresh never-aborted signal when no stored account controller exists.

## Best Possible Solution

Land this fallback change on top of current `stopChannel` after the branch is refreshed and a focused regression test or equivalent proof covers the no-controller stop hook path.

## Reproduction Assessment

Yes, source-reproducible: current main calls `stopAccount` with a fresh non-aborted fallback signal when no controller exists, while abort-aware helpers only complete immediately when the signal is already aborted. I did not run a live gateway/plugin repro because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

Yes. Replacing the fallback with `AbortSignal.abort()` is the narrow maintainable fix because the existing-controller path already calls `abort?.abort()` before handing the signal to the stop hook.
