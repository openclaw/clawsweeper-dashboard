## What This Changes

The PR changes `stopChannel` so `stopAccount` receives `AbortSignal.abort()` when there is no stored per-account controller.

## Best Possible Solution

Land this narrow gateway fallback fix after the contributor adds redacted real behavior proof and the branch receives fresh relevant validation; a focused regression test for the no-controller `stopAccount` signal would make the behavior durable.

## Reproduction Assessment

Yes, from source inspection. On current main, a configured account with `stopAccount` but no stored controller receives a fresh non-aborted signal, and abort-aware cleanup such as `waitUntilAbort` would wait instead of completing immediately.

## Solution Assessment

Yes, the proposed code change is the narrow maintainable fix because `AbortSignal.abort()` makes the no-controller fallback match the already-aborted primary stop path. Merge readiness depends on proof and validation, not a different code approach.
