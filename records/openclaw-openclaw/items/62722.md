## Summary

Keep open: the gateway shutdown bug still exists on current main and the one-line PR looks like the right code fix, but this external PR needs real behavior proof before merge and its PR checks are not clean.

## What This Changes

The branch changes `src/gateway/server-channels.ts` so `stopChannel` passes `AbortSignal.abort()` to `stopAccount` when no stored `AbortController` exists.

## Best Possible Solution

Land this narrow gateway fix after the contributor adds real behavior proof and the relevant checks are green, ideally with a focused regression test for the no-controller `stopAccount` signal fallback.

## Reproduction Assessment

Yes from source inspection, though I did not run a live gateway reproduction in this read-only review. Current main aborts an existing controller before `stopAccount`, but the no-controller fallback passes a fresh non-aborted signal into the same stop hook.

## Solution Assessment

Yes, the proposed code change is the narrow maintainable fix: `AbortSignal.abort()` makes the fallback match the already-aborted primary path. Merge readiness still depends on real behavior proof and clean validation rather than more code churn.
