## Summary

Keep open: current main and the latest release still pass a fresh non-aborted fallback signal to `stopAccount` when no account controller exists, and the one-line fix is the right layer, but this external PR is not merge-ready while it conflicts with main and lacks after-fix real behavior proof.

## What This Changes

The PR changes the gateway `stopChannel` fallback from a fresh `AbortController().signal` to `AbortSignal.abort()` when calling a plugin `stopAccount` without an active controller.

## Best Possible Solution

Refresh the branch onto current main, keep the `AbortSignal.abort()` fallback, add focused gateway lifecycle coverage if feasible, and attach redacted real shutdown proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main's explicit `stopChannel(channel, accountId)` path can call a plugin `stopAccount` with no stored AbortController and therefore passes a fresh non-aborted fallback signal.

## Solution Assessment

Yes for the code shape, but not yet for merge readiness. The gateway manager owns construction of `ChannelGatewayContext`, and `AbortSignal.abort()` aligns the no-controller fallback with the already-aborted primary path without adding API or config surface.
