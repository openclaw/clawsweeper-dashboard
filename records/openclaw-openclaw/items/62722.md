## Summary

Keep open: current main and the latest release still pass a never-aborted fallback signal to `stopAccount`, and this PR is a focused gateway-layer fix, but the external PR still lacks required after-fix real behavior proof before merge.

## What This Changes

The PR changes `src/gateway/server-channels.ts` so `stopChannel` uses `AbortSignal.abort()` when calling `stopAccount` without an existing account AbortController.

## Best Possible Solution

Merge the small gateway-layer signal fix after real behavior proof and, ideally, a focused regression test for explicit `stopAccount` without an active controller.

## Reproduction Assessment

Yes, source-reproducible. Current main's explicit stop path with a plugin `stopAccount` and no stored AbortController passes a fresh non-aborted signal, so a cleanup hook checking `signal.aborted` would observe the wrong lifecycle state.

## Solution Assessment

Yes. The gateway manager is the owner that constructs `ChannelGatewayContext`, and `AbortSignal.abort()` matches the already-aborted signal used when a running controller exists without adding API or config surface.
