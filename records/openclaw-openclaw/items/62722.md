## Summary

Keep open: current main and v2026.6.9 still pass a never-aborted fallback signal to `stopAccount` when no controller exists, so this PR addresses a real remaining gateway lifecycle gap. ## What This Changes

The PR changes `stopChannel` so `stopAccount` receives `AbortSignal.abort()` instead of a fresh never-aborted signal when no account AbortController is stored.

## Best Possible Solution

Land a refreshed one-line fix with focused regression coverage and after-fix gateway/plugin stop proof, keeping the no-controller `stopAccount` fallback aligned with the already-aborted active-controller path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call `stopChannel(channel, accountId)` for a configured account with a `stopAccount` hook but no stored task/controller, and current main passes a non-aborted fallback signal. I did not prove a bundled production hang because the inspected bundled stop hooks do not currently consume the signal.

## Solution Assessment

Yes, the code shape is the narrowest maintainable fix: `AbortSignal.abort()` makes the no-controller fallback match the active-controller path after `abort?.abort()`. It still needs a rebase, regression coverage, and real behavior proof before merge.
