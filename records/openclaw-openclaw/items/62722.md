## What This Changes

The PR changes `stopChannel` so the no-controller `stopAccount` fallback receives `AbortSignal.abort()` instead of a fresh non-aborted signal.

## Best Possible Solution

Land this narrow gateway fallback fix after contributor proof and fresh validation; a focused regression test for the no-controller `stopAccount` signal would make the behavior durable.

## Reproduction Assessment

Yes, from source inspection. On current main, `stopAccount` can receive a fresh non-aborted fallback signal when no controller is stored, and abort-aware cleanup such as `waitUntilAbort` would keep waiting.

## Solution Assessment

Yes, the proposed code change is the narrow maintainable fix because `AbortSignal.abort()` makes the no-controller fallback match the already-aborted primary stop path; merge readiness depends on proof and validation, not a different code approach.
