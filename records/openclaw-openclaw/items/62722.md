## What This Changes

The PR changes `src/gateway/server-channels.ts` so the `stopChannel` fallback passes an already-aborted signal to `stopAccount` when no stored AbortController exists.

## Best Possible Solution

Land a refreshed gateway-layer fix that keeps the fallback signal already aborted, adds focused regression coverage for the no-controller `stopAccount` path, and includes redacted terminal or log proof from a real shutdown flow.

## Reproduction Assessment

Yes, source-reproducible. Current main's explicit stop path with a plugin `stopAccount` and no stored AbortController passes a fresh non-aborted signal, so cleanup code checking `signal.aborted` would observe the wrong lifecycle state.

## Solution Assessment

Yes, the proposed layer is the best fix shape. The gateway manager owns construction of `ChannelGatewayContext`, and `AbortSignal.abort()` aligns the no-controller fallback with the already-aborted primary path without adding API, config, or plugin-specific behavior.
