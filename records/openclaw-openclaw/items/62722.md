## Summary

Keep open: current main still passes a fresh non-aborted fallback signal into the gateway stop hook, and this PR is a narrow plausible fix, but it still lacks external real-behavior proof and the live branch is conflicting/dirty against main.

## What This Changes

This PR changes the gateway channel stop fallback to pass `AbortSignal.abort()` to `stopAccount` when no stored controller exists.

## Best Possible Solution

Land the fallback change after the branch is rebased, a focused regression test or equivalent proof covers the no-controller stop hook path, and redacted real behavior proof shows shutdown cleanup no longer waits on a never-aborted signal.

## Reproduction Assessment

Yes, source-reproducible: current main calls `stopAccount` with a fresh non-aborted fallback signal when no controller exists, while abort-aware helpers only complete immediately when the signal is already aborted. I did not run a live gateway/plugin repro.

## Solution Assessment

Yes. Replacing the fallback with `AbortSignal.abort()` is the narrow maintainable fix because the existing-controller path already calls `abort?.abort()` before handing the signal to the stop hook.
