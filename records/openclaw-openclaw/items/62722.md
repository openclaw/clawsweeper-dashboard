Keep this PR open: current main still has the non-aborted fallback, and the proposed one-line change is the narrow fix. Merge readiness is still blocked by missing external real behavior proof, a current merge conflict, and stale failing CI metadata.

## What This Changes

The PR changes `src/gateway/server-channels.ts` so `stopChannel` passes `AbortSignal.abort()` to `stopAccount` when no stored per-account `AbortController` exists.

## Best Possible Solution

Land the narrow gateway fallback fix after the contributor adds redacted real behavior proof and the PR is rebased with green relevant checks; a focused regression test for the no-controller `stopAccount` signal would make the behavior durable.

## Reproduction Assessment

Yes, from source inspection. On current main, a configured account with `stopAccount` but no stored controller receives a fresh non-aborted signal, and an abort-aware cleanup helper such as `waitUntilAbort` would wait instead of completing immediately.

## Solution Assessment

Yes, the proposed expression is the narrow maintainable fix because `AbortSignal.abort()` makes the no-controller fallback match the already-aborted primary stop path. Merge readiness still depends on proof, rebase, and validation rather than a different code approach.
