## What This Changes

The PR changes `stopChannel` so `stopAccount` receives `AbortSignal.abort()` instead of a fresh non-aborted signal when no stored channel abort controller exists.

## Best Possible Solution

Refresh the branch or replace it with the same narrow code change, add focused regression coverage for the no-controller `stopAccount` signal, and require redacted real shutdown/plugin proof before merge.

## Reproduction Assessment

Yes at source level: call `stopChannel(channel, accountId)` for a configured plugin account with a `stopAccount` hook and no stored abort/task, and current main passes a fresh non-aborted fallback signal. I did not run a live plugin repro in this read-only review.

## Solution Assessment

Yes, once rebased, `AbortSignal.abort()` is the narrowest maintainable fix because it makes the fallback match the primary path after `abort?.abort()`. Broader stop timeouts and health-monitor recovery are adjacent lifecycle work, not a replacement for this signal semantic.
