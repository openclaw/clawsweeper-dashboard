## Summary

Keep open: current main and v2026.6.10 still pass a fresh non-aborted signal to `stopAccount` when no account controller exists. The one-line fix is directionally correct, but the branch is conflicted against current main and lacks real behavior proof for the plugin-facing stop semantics.

## What This Changes

The PR changes the gateway `stopChannel` fallback so `stopAccount` receives `AbortSignal.abort()` when there is no stored account `AbortController`.

## Best Possible Solution

Rebase the `AbortSignal.abort()` fallback onto current main, add focused `server-channels` regression coverage for a `stopAccount` hook with no active task, and include redacted real gateway or plugin proof before merge.

## Reproduction Assessment

Yes at source level: call `stopChannel(channel, accountId)` for a configured plugin account with a `stopAccount` hook but no stored task/controller, and current main passes a non-aborted fallback signal. I did not run a live plugin repro in this read-only review.

## Solution Assessment

Yes, the proposed code shape is the narrowest fix once rebased: `AbortSignal.abort()` makes the no-controller fallback match the already-aborted controller path after `abort?.abort()`. Broader stop timeouts or shutdown redesigns are separate lifecycle work.
