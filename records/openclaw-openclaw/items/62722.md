## Summary

Keep open: current main and the latest release still pass a never-aborted fallback signal to `stopAccount`, and this PR is the canonical narrow fix for that signal inconsistency. It is not merge-ready because the branch conflicts with current main, lacks real behavior proof, and changes a plugin lifecycle context semantic that maintainers should consciously accept.

## What This Changes

The PR changes `stopChannel` so the `stopAccount` fallback receives `AbortSignal.abort()` when there is no stored account `AbortController`.

## Best Possible Solution

Keep the narrow fix, refresh it onto current main, add a focused regression test for the no-controller `stopAccount` signal, and require redacted real shutdown proof before merge.

## Reproduction Assessment

Yes at source level: an explicit `stopChannel(channel, accountId)` for a configured plugin with `stopAccount` but no stored task/controller reaches the fallback and current main passes a non-aborted signal. I did not run a live plugin shutdown in this read-only review.

## Solution Assessment

Yes, the proposed `AbortSignal.abort()` fallback is the narrowest owner-boundary fix for making the no-controller path match the already-aborted primary path. It still needs rebase, regression coverage, and real behavior proof before merge.
