## Label Justifications

- `P1`: The report affects a real Telegram automation workflow and can expose raw commands, local paths, and provider/runtime failure text in a user-facing chat.
- `impact:security`: Raw command lines, local filesystem paths, and provider errors in chat are an information-disclosure concern.
- `impact:message-loss`: Internal diagnostic and failure payloads are being routed as user-visible channel messages instead of staying in diagnostics or agent-owned summaries.

## Summary

Keep open: current main still has source-backed paths where fatal Codex/tool errors and cron failure alerts can become Telegram-visible text, including raw command, path, provider, or failure details; the issue is actionable but needs maintainer security/product policy before implementation.

## What This Changes

The issue asks OpenClaw to stop sending raw tool/runtime failure details as standalone Telegram messages from automated Codex-backed agent and cron runs.

## Best Possible Solution

Centralize a channel-safe failure policy that keeps detailed tool/provider errors in diagnostics and session state, delivers final assistant text when available, and sends only concise redacted actionable cron or fatal-run summaries to Telegram.

## Reproduction Assessment

Yes, at source level. Current helpers, tests, Codex protocol/source types, cron failure-alert code, and Telegram outbound delivery show raw failure text can still become a channel payload; I did not run a live Telegram repro.

## Solution Assessment

No, this is not ready for an automatic fix because the behavior boundary is a security/product decision. The maintainable solution is a centralized redaction and actionability policy across cron isolated-agent outcomes, fatal fallback text, and channel alert copy.
