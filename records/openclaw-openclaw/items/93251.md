## Summary

Keep open: current main and v2026.6.8 still allow raw or raw-derived Codex tool metadata, fatal cron error payloads, and cron failure-alert text to reach Telegram; the merged Feishu fix is useful related context but does not resolve this Telegram/cron safe-delivery policy.

## What This Changes

The issue asks OpenClaw to stop delivering raw Telegram tool and cron failure notices from automated Codex-backed agent runs and replace them with internal diagnostics or channel-safe summaries.

## Best Possible Solution

Adopt a central channel-safe failure policy for Telegram and cron delivery: keep raw command/provider details in diagnostics, let recovered final assistant text win, and send only concise redacted actionable summaries or thresholded alerts to chat.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current payload, Codex projector, Telegram outbound, cron helper, and cron alert code still preserve paths where raw or raw-derived failure text can be delivered. I did not run a live Telegram direct-chat reproduction.

## Solution Assessment

No, closing is not the best way to solve this issue because current main only partially mitigates recovered cron warnings and the Feishu fix is channel-specific. The safer fix is a maintainer-approved channel-safe failure policy rather than a Telegram-only string scrub or a broad new verbose toggle.
