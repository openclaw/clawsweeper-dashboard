## Summary

Keep open: current main still has source-backed paths that can surface tool-failure text and cron failure-alert details into chat, while the active related PR only covers part of the direct-chat progress leak and is not merged.

## What This Changes

The issue asks OpenClaw to stop delivering raw Telegram-visible tool/runtime failure text by default and keep detailed failures in diagnostics or agent-owned summaries.

## Best Possible Solution

Define and land a maintainer-approved channel-safe failure-notification boundary that keeps detailed failures in diagnostics/session state, preserves actionable terminal failures, and emits only redacted structured chat alerts when user action is needed.

## Reproduction Assessment

Yes, at source level: current main can still construct visible `⚠️ ... failed` tool payloads and cron failure-alert `Last error` messages for channel delivery. I did not run a live Telegram Desktop/Bot API reproduction or inspect the private diagnostics bundle.

## Solution Assessment

Unclear as an implementation plan: the requested guard is directionally right, but maintainers need to decide which failure details can ever be chat-visible and how that interacts with real mutating failures, cron alerts, and PR https://github.com/openclaw/openclaw/pull/89975.
