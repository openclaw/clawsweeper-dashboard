## Label Justifications

- `P1`: The report describes a recurring real Telegram automation workflow exposing internal failures to a user-facing chat.
- `impact:security`: Current paths can expose raw command output, local paths, and provider/runtime errors outside internal diagnostics.
- `impact:message-loss`: Diagnostic failure payloads can be routed as user-visible channel messages instead of a clean assistant-owned summary.

## Summary

Keep open: current main and the latest shipped tag still contain source-backed paths that can turn Codex/tool failures and cron failure alerts into chat-visible raw failure text, so this is not implemented or superseded.

## What This Changes

The issue asks OpenClaw to stop delivering raw tool/command and cron failure details as standalone Telegram messages from automated Codex-backed agent runs by default.

## Best Possible Solution

Keep this issue open as the concrete Telegram/Codex reproduction, then define a shared cron/agent/channel failure boundary that keeps raw details in diagnostics and sends only redacted, actionable chat text when a notification is necessary.

## Reproduction Assessment

Yes, at source level. Current source/tests show Codex command failures become `lastToolError.error`, embedded warning payloads can be user-visible, and cron alerts include raw `Last error` text; I did not run a live Telegram repro.

## Solution Assessment

No, current main is only a partial mitigation. The best fix is a shared cron/agent/channel failure policy that redacts or suppresses raw chat-visible details while preserving internal diagnostics and necessary user-action alerts.
