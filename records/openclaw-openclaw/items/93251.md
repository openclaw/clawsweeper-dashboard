## Summary

Keep open: current main only partially suppresses recovered cron tool-warning noise, while fatal Codex/tool payloads and cron failure alerts can still preserve raw command, path, provider, or runtime error text for Telegram delivery.

## What This Changes

The issue asks OpenClaw to keep raw automated-run tool and cron failure details internal by default and send Telegram users only agent-owned or channel-safe failure text.

## Best Possible Solution

Keep raw command/provider/tool failure details in diagnostics/session state, and send Telegram only a concise redacted actionable failure summary or thresholded alert approved by maintainers.

## Reproduction Assessment

Yes, at source level: current helper and failure-alert tests/code show recovered warnings are only partially handled while fatal error and cron alert paths still preserve raw deliverable text. I did not run a live Telegram chat reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial fix. The best solution is a shared cron/Codex failure projection policy before channel delivery, not a Telegram-only sanitizer or a new broad verbose toggle.
