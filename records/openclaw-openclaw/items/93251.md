## Label Justifications

- `P1`: The report affects a real Telegram automation workflow and can expose raw commands, local paths, and provider failure details to a user-facing chat.
- `impact:security`: Raw command lines, local paths, and provider errors in chat are an information-disclosure concern.
- `impact:message-loss`: Internal diagnostic/error payloads are being routed as user-visible channel messages instead of staying in the agent diagnostics path.

## Summary

Keep open: current main has partial protection for recovered Telegram cron tool warnings, but the remaining report is still meaningful because OpenClaw can still build user-visible tool-warning payloads and cron failure-alert bodies that include raw failure text for channel delivery.

## What This Changes

The issue asks OpenClaw to stop delivering raw Telegram tool and cron failure details from automated agent runs as standalone chat messages by default.

## Best Possible Solution

Add a channel-safe failure-notification boundary for automated agent runs: keep detailed tool/provider errors in diagnostics and cron run records, deliver final assistant text when available, and use only concise redacted actionable failure summaries for configured chat alerts.

## Reproduction Assessment

Yes, at source level. Current tests and source show tool-warning payloads and cron failure-alert messages that can include raw failure text; I did not run a live Telegram Desktop reproduction or inspect the private diagnostics bundle.

## Solution Assessment

No, closing this issue is not the best way to solve it. The narrow maintainable path is to tighten cron/agent delivery policy around raw failure payloads while preserving internal diagnostics, final assistant recovery, and configured failure notifications.
