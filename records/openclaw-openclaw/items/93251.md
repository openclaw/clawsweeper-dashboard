## Summary

Keep open: current main has partial recovery for some recovered cron/tool warnings, but source and tests still show Telegram/cron paths can deliver fatal tool, provider, runtime, or raw cron failure text to chat. The related Feishu fix and the narrower open cron-classifier issue do not cover this broader Telegram-safe failure policy.

## What This Changes

The issue asks OpenClaw to stop delivering raw tool, Codex, and cron failure text as standalone Telegram messages and to keep diagnostics internal or channel-safe.

## Best Possible Solution

Centralize channel-safe failure delivery so raw command/path/provider details stay in diagnostics or session state, recovered final assistant text wins, and only concise redacted actionable or thresholded failures reach chat.

## Reproduction Assessment

Yes at source level: current helper tests and failure-alert code preserve deliverable fatal tool/provider error text, and Telegram outbound sends resolved payload text without a sanitizer. I did not run a live Telegram chat reproduction in this read-only review.

## Solution Assessment

No, closing is not the best solution because current main still has multiple raw failure delivery paths. A Telegram-only sanitizer or new verbose flag would be too narrow; the maintainable fix belongs at the central payload, cron failure, and channel-safe delivery policy boundary.
