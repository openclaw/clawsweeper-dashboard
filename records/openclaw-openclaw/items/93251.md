## Label Justifications

- `P1`: The report affects a real Telegram automation workflow and can expose raw commands, local paths, and provider failure details to user-facing chat.
- `impact:security`: Raw command lines, local paths, and provider/runtime errors in chat are an information-disclosure concern.
- `impact:message-loss`: Internal diagnostic/error payloads can be misrouted as user-visible channel messages instead of staying in diagnostics or agent-owned summaries.

## Summary

Keep open: current main still has source-reproducible paths that can turn raw tool/provider failure text into Telegram/channel-visible cron output, while the existing Telegram final-text preference only covers part of the report.

## What This Changes

The issue requests suppressing or redacting raw tool and cron failure details before automated agent-run failures are delivered to Telegram or other chat channels.

## Best Possible Solution

Add a channel-safe failure notification boundary for automated agent runs: keep detailed tool/provider errors in diagnostics and cron run records, prefer final assistant text when present, and send only concise redacted actionable chat alerts when policy says notification is required.

## Reproduction Assessment

Yes, at source level. Current main still contains tests and source paths that produce raw tool-warning and raw `Last error` alert text for channel delivery; I did not run a live Telegram reproduction.

## Solution Assessment

No, closing is not the best way to solve the issue. The maintainable path is a maintainer-approved channel-safe failure policy that preserves diagnostics and configured alerts without leaking raw operational details by default.
