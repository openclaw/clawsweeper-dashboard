## Label Justifications

- `P1`: The report affects a real Telegram automation workflow and can expose raw commands, local paths, and provider failure details to user-facing chat.
- `impact:security`: Raw command lines, local paths, and provider/runtime errors in chat are an information-disclosure concern.
- `impact:message-loss`: Internal diagnostic failure payloads are being misrouted as user-visible channel messages instead of staying in diagnostics or being summarized by the agent.

## Summary

Keep open: current main and the latest release still have source-backed paths that can turn tool/runtime failures into channel-visible warning payloads or cron failure-alert text, and the issue is protected by security/product-review labels rather than being safe cleanup.

## What This Changes

The issue asks OpenClaw to stop delivering raw tool, command, path, and provider failure details as standalone Telegram chat messages from automated agent runs.

## Best Possible Solution

Add a shared channel-safe failure notification boundary for automated agent runs: keep raw tool/provider details in diagnostics and run records, deliver final assistant text when available, and send only concise redacted actionable summaries through chat alerts.

## Reproduction Assessment

Yes at source level. Current source and tests show visible tool-warning payloads and raw cron failure-alert bodies can still be produced; I did not run a live Telegram Desktop reproduction.

## Solution Assessment

No, closing or treating this as a Telegram-only cleanup is not the best solution. The maintainable fix is a shared delivery policy around raw failure details across Codex projection, embedded-runner payloads, cron output, and channel alerts.
