## Label Justifications

- `P1`: The issue affects a real Telegram automation workflow and can expose raw commands, local paths, and provider failure text into a user-facing chat.
- `impact:security`: The reported and source-backed paths can disclose raw command lines, local paths, and provider/runtime error details outside diagnostics.
- `impact:message-loss`: Internal diagnostic/error payloads can be routed as user-visible channel messages instead of the intended final assistant-owned response.

## Summary

Keep open: current main and the latest release still have source-backed paths that can send raw tool-warning text or raw cron failure-alert detail to Telegram/chat delivery, and the nearby open PR only covers one direct-progress slice rather than this automated cron/failure-alert report.

## What This Changes

Prevent Telegram automated-agent and cron delivery from sending raw tool or provider failure text as standalone chat messages while keeping detailed diagnostics internal.

## Best Possible Solution

Define and implement a channel-safe failure-notification boundary for automated agent runs: detailed tool/provider failures stay in diagnostics and run records, while chat receives only final assistant text or concise redacted actionable alerts.

## Reproduction Assessment

Yes, at source level. Current source and tests show raw failed-tool and cron failure-alert payloads can still be selected for channel delivery, although I did not run a live Telegram replay.

## Solution Assessment

No, closing or queueing an automatic fix is not the best path yet. The maintainable fix needs maintainer product/security policy for which failure details are safe in chat, then focused cron, runner, and channel coverage.
