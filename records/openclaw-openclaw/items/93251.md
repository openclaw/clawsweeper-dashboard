## Summary

Keep open: current main and the latest release still have source-backed paths that can send raw tool or cron failure details into Telegram/chat delivery, and the related open PRs only cover narrower cron recovery cases rather than the full failure-disclosure problem.

## What This Changes

The issue asks OpenClaw to stop delivering raw internal tool, command, and cron failure details as standalone Telegram messages from automated agent runs.

## Best Possible Solution

Add a policy-backed channel-safe failure boundary for automated agent runs: keep detailed tool/provider errors in diagnostics and cron state, deliver final assistant text when available, and use concise redacted actionable summaries for configured chat alerts.

## Reproduction Assessment

Yes, at source level: current main still has code paths and tests that can construct user-visible tool-warning payloads and cron failure-alert bodies with raw failure text. I did not run a live Telegram reproduction or inspect the private diagnostics bundle.

## Solution Assessment

No, closing this issue is not the best way to solve it because current main only partially mitigates recovered warnings. The narrow maintainable path is to define and implement a channel-safe failure-disclosure policy while preserving internal diagnostics and genuine actionable alerts.
