## Summary

Keep open: current main still rejects token-only password-mode handshakes, and this PR is a narrow valid fix, but the external PR still lacks after-fix real behavior proof and is currently conflicting/diverged from main.

## What This Changes

The PR makes password-mode gateway auth fall back from `connectAuth.password` to `connectAuth.token`, adds Rabbit/Funnel regression edge tests, and adds a changelog fix entry.

## Best Possible Solution

Repair the branch against current main, keep the fallback/tests/changelog narrow, require redacted real handshake proof, and land after the gateway changed checks pass.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, password mode passes only `connectAuth?.password` to `authorizePasswordAuth`, so `connectAuth: { token: "secret" }` deterministically returns `password_missing` before the PR fallback.

## Solution Assessment

Yes. The proposed `connectAuth.password ?? connectAuth.token` fallback is the narrowest maintainable fix because it stays inside password mode, preserves password precedence, and still compares against the configured shared password; device-specific Rabbit bootstrap tokens can remain a separate follow-up.
