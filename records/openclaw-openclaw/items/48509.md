## Summary

Keep open: current main still has only the generic tool and transcript/message hook surfaces, not a dedicated durable persistence-write hook or equivalent durable-write policy context. The linked CaMeL/provenance/security-policy issues remain open design work, so this is not stale cleanup or already implemented.

## What This Changes

Request an additive Plugin SDK hook that can inspect or block durable state writes such as memory, file, cron, config, and preferences before they cross session boundaries.

## Best Possible Solution

Define a versioned Plugin SDK durable-write policy contract with explicit target taxonomy, provenance/redaction, veto or mutation semantics, trust model, and coverage tests across representative persistent write paths.

## Solution Assessment

Unclear. A before_persistence_write hook is plausible, but maintainers should settle the durable-write contract, target taxonomy, provenance coverage, and trust semantics before landing a long-lived SDK event.
