## Summary

Keep open: current main and the latest release still expose only generic tool, transcript, message, cron, and session hooks, not a dedicated pre-write hook for durable state changes. This remains a coherent public Plugin SDK and security-policy design item that needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests a `before_persistence_write` plugin hook for blocking or auditing durable writes to memory, files, cron, config, and preferences.

## Best Possible Solution

Keep this as a maintainer-owned Plugin SDK/security design item, then implement one narrow documented durable-write contract across hook types, runner dispatch, selected writer boundaries, docs, and focused tests once the API and security semantics are approved.

## Reproduction Assessment

Yes, at source level: current main exposes `before_tool_call`, `tool_result_persist`, and `before_message_write`, but no `before_persistence_write` hook name or durable-write event type. No runtime execution is needed because this is a public API gap, not a flaky behavior report.

## Solution Assessment

Unclear pending maintainer API design: a dedicated hook is plausible, but maintainers still need to choose the taxonomy, blocking semantics, payload redaction, and relation to broader provenance issues.
