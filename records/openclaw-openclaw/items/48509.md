## Summary

Keep open: current main still lacks a durable persistence-write plugin hook or equivalent pre-write policy surface, and the adjacent CaMeL/provenance issues are open design context rather than superseding implementations.

## What This Changes

The issue requests an additive plugin hook that can classify and optionally block durable writes to memory, files, cron jobs, config, and preferences before those writes cross a session boundary.

## Best Possible Solution

Define a versioned additive SDK policy surface for durable writes, with stable target, path, session, provenance, and veto semantics documented and tested across representative persistent write paths.

## Reproduction Assessment

Not applicable as a runtime bug. Source and docs inspection on current main is high-confidence: the public hook registry and before_tool_call dispatch still lack a dedicated durable-write pre-veto surface.

## Solution Assessment

Unclear. A dedicated before_persistence_write hook is plausible, but maintainers should choose the durable-write policy contract and coverage model before implementation starts.
