## Summary

Keep open: current main still lacks a dedicated durable persistence-write hook or equivalent durable-write policy context, and the related CaMeL/provenance/security-policy threads remain open design work rather than merged coverage.

## What This Changes

Requests an additive Plugin SDK hook that fires before durable state writes such as memory, file, cron, config, and preference changes, carrying target metadata and optional veto semantics.

## Best Possible Solution

Define a versioned Plugin SDK durable-write policy contract with explicit target taxonomy, provenance, veto or mutation semantics, trust model, and coverage tests across representative persistent write paths.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/API request. Source inspection gives high confidence that current main has no durable pre-write hook or equivalent persistence context.

## Solution Assessment

Unclear. A before_persistence_write hook is plausible, but maintainers should choose the durable-write contract, target taxonomy, provenance coverage, and trust semantics before landing one narrow SDK event.
