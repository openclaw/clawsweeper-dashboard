## Summary

Keep this open. Current main still lacks a public durable persistence-write hook or equivalent pre-write context/veto, and the remaining design choices affect the plugin SDK, security policy, and several write paths rather than being a narrow cleanup close.

## What This Changes

Request an additive plugin hook, or equivalent `before_tool_call` context, that lets plugins classify, audit, and block durable state writes before memory, file, cron, config, or preference changes are persisted.

## Best Possible Solution

Design an additive, versioned plugin-SDK persistence-write policy surface with stable target/path/session/provenance metadata and at least blocking semantics, then wire it through the concrete durable write owners with docs and contract tests. Coordinate the API with #48503, #48517, and #48519 so action classification, session isolation, and persistence provenance fit together.

## Reproduction Assessment

Not applicable as a runtime reproduction; this is a feature/API gap. Static inspection of current main gives a high-confidence verification path: the hook registry/docs and representative memory, file, cron, and config write paths do not expose a durable pre-write hook.

## Solution Assessment

Unclear whether the exact proposed separate hook is the best final API. A durable-write hook is plausible, but maintainers should first choose between a new hook and enriched `before_tool_call` context, and define target/provenance/mutation semantics before implementation.
