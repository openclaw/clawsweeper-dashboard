## Summary

Keep this open: current main still has generic before_tool_call and transcript/message persistence hooks, but no dedicated pre-write plugin hook for durable state changes; the request is a coherent plugin SDK/security design item that needs maintainer product and security judgment rather than cleanup closure.

## What This Changes

The issue requests a new plugin SDK hook that fires before durable state changes such as memory, file, cron, config, or preference writes.

## Best Possible Solution

Keep this as a maintainer-owned plugin SDK/security design item and resolve it together with the related provenance and hook-context proposals before adding any public API.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: the current plugin hook surface and hook manager expose before_tool_call, message, and transcript hooks but no durable-state pre-write hook, while built-in memory/file write paths persist state without such a hook.

## Solution Assessment

Is this the best way to solve the issue? Unclear: a dedicated hook is plausible, but maintainers still need to choose whether durable-write classification belongs in a new hook, enriched before_tool_call metadata, or a broader provenance/enforcement primitive.
