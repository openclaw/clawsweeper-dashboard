## Summary

Keep open: current main still lacks a dedicated durable persistence-write hook or equivalent pre-write policy contract, and the linked CaMeL/provenance/security-policy threads remain open design context rather than merged coverage.

## What This Changes

Request an additive Plugin SDK hook that fires before durable state writes such as memory, files, cron jobs, config, and preferences so plugins can block or audit cross-session persistence.

## Best Possible Solution

Define a versioned Plugin SDK policy surface for durable writes with explicit target, provenance, veto, coverage, trust, and documentation semantics across representative persistent write paths.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/API request. Source inspection gives high confidence that current main has no durable pre-write hook or equivalent persistence context.

## Solution Assessment

Unclear. A before_persistence_write hook is plausible, but maintainers should choose the durable-write contract, taxonomy, coverage model, and trust semantics before landing one narrow SDK event.
