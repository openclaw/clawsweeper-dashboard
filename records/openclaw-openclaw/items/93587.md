## What This Changes

The PR changes Control UI chat-session creation so fallback or non-persisted visible sessions are not sent as `parentSessionKey` when starting a new dashboard chat.

## Best Possible Solution

Land the narrowed parent guard if maintainers accept the session-lineage boundary, while leaving broader session naming UX to the merged `/name` direction and related label work.

## Reproduction Assessment

Yes, source-reproducible: current main sends the visible session key as `parentSessionKey` except for `unknown`, and the gateway rejects supplied parents that lack `sessionId`. I did not run tests because this review was read-only.

## Solution Assessment

Yes, with maintainer acceptance of the session-state boundary. The Control UI create-session boundary is the narrowest maintainable layer because it can distinguish placeholder rows before building the gateway request while preserving persisted parent behavior.
