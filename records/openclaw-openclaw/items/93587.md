## Summary

Keep this PR open: the narrowed Control UI parent resolver appears to fix a real fallback-session bug without reintroducing the broader naming surface, and I found no blocking correctness or security finding. It still carries session-state merge risk because maintainers need to accept when fallback/non-persisted rows stop participating in lineage.

## What This Changes

The PR changes Control UI chat-session creation so fallback or non-persisted visible sessions are not sent as `parentSessionKey`, with focused regression tests for fallback, stale, persisted, and slash-command behavior.

## Best Possible Solution

Land the narrowed parent guard if maintainers accept the empty-list fallback semantics, while leaving broader session naming UX to the merged `/name` path and related label work.

## Reproduction Assessment

Yes, source-reproducible: current main sends fallback `main` or `agent:main:main` as `parentSessionKey`, and the gateway rejects supplied parents whose store entry lacks `sessionId`. I did not run tests because this review was read-only.

## Solution Assessment

Yes, with maintainer acceptance of the session-state boundary. The Control UI create-session boundary is the narrowest maintainable layer because it can distinguish placeholder rows before building the gateway request while preserving real persisted parent behavior.
