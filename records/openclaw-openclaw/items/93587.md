## What This Changes

The PR adds persisted session `displayName` support to gateway create/patch, protocol/native models, CLI JSON rows, and Control UI `/new <name>` plus `/label <name>` slash-command handling.

## Best Possible Solution

Have maintainers explicitly choose the durable session naming contract; if `displayName` is accepted for presentation names, land this additive WebChat/Control UI path and keep the broader CLI/programmatic naming scope tracked separately.

## Reproduction Assessment

Yes. Source inspection reproduces the missing capability on current main: there is no Control UI `/label` dispatch, `/new` does not pass a display-name payload, and the gateway create/patch schemas do not accept `displayName`.

## Solution Assessment

Mostly yes, if maintainers want `displayName` as a presentation-name field: the PR writes through the gateway/session metadata boundary instead of a UI-only cache. If maintainers want one canonical user-set name, the safer alternative is to align this with the existing `label` contract before merge.
