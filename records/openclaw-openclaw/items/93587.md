## Summary

Keep this PR open, but the current patch still has one session-parent correctness gap before merge: the new resolver treats `updatedAt` as enough proof that a row is parentable even though the Gateway accepts parent sessions only when the loaded entry has `sessionId`.

## What This Changes

The branch adds a Control UI parent-session resolver for new chat creation, focused regression tests for fallback/non-persisted parent handling, slash-command tests keeping `/new` and `/label` behavior unchanged, and an import-order-only app change.

## Best Possible Solution

Align Control UI parent eligibility with the Gateway `sessionId` parent contract, then land the narrowed fallback-parent fix while leaving naming surfaces to https://github.com/openclaw/openclaw/issues/93422 and the merged `/name` direction.

## Reproduction Assessment

Yes at source level. Current main sends the previous visible session key as `parentSessionKey` except for `unknown`, and Gateway rejects parent entries without `sessionId`; the PR body also supplies after-fix live output, but I did not run a local repro in this read-only review.

## Solution Assessment

No, not quite. The resolver is the right layer, but matching-row parent eligibility should require `sessionId` so the UI contract matches Gateway validation.
