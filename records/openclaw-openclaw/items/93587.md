## Summary

Keep open: current main still sends the visible Control UI session key as `parentSessionKey` except for `unknown`, while this PR adds a focused guard for fallback and non-persisted rows with matching regression coverage and sufficient real behavior proof. The remaining issue is maintainer acceptance of the session-lineage semantics, not a close or automated-repair reason.

## What This Changes

The PR changes Control UI chat-session creation to omit fallback or non-persisted visible session keys from `parentSessionKey` when starting a new dashboard chat.

## Best Possible Solution

Land the narrowed parent guard if maintainers accept that only persisted rows with `sessionId` can be reused as parents, while leaving broader naming UX to https://github.com/openclaw/openclaw/issues/93422 and https://github.com/openclaw/openclaw/pull/95700.

## Reproduction Assessment

Yes. Source inspection shows current main sends the visible session key as `parentSessionKey` except for `unknown`, while the gateway rejects supplied parents without `sessionId`; I did not rerun the browser flow in this read-only review.

## Solution Assessment

Yes. The Control UI create-session boundary is the narrowest maintainable layer because it can distinguish UI placeholders from persisted rows before building the gateway request; making the gateway accept missing parents would weaken the parent contract.
