## Summary

Keep open. The bug still exists on current main and this PR targets the right gateway validation boundary, but the patch is not merge-ready because it loads full channel plugins for a static account-list check, the gateway runtime-topology check is failing, the branch conflicts with current main, and contributor real behavior proof is missing.

## What This Changes

The PR adds gateway cron.add/cron.update validation for explicit announce delivery and failure-destination account IDs, plus server tests for valid and invalid account cases.

## Best Possible Solution

Land a focused gateway-side guard that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates only announce channels in the gateway, and persists after that incomplete guard.

## Solution Assessment

No. Gateway validation before persistence is the right boundary, but this implementation should be ported to current main and avoid getChannelPlugin-based full plugin loading in favor of read-only account metadata.
