## Summary

Keep open: this PR targets a real cron validation bug, but the current patch is not merge-ready because it validates accounts by loading full channel plugins in a gateway validation path, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds cron.add/update gateway validation and tests to reject explicit announce delivery and failure-destination account IDs that are not configured for the resolved channel.

## Best Possible Solution

Land focused gateway validation that resolves allowed accounts from read-only manifest/config metadata, covers primary and failure destinations, and preserves implicit last/omitted-channel, normalized IDs, and unlisted default-account compatibility.

## Reproduction Assessment

Yes, from source inspection: the CLI forwards --account, the protocol only shape-checks it, gateway validation remains channel-only, and runtime preserves explicit accountId precedence. I did not run a live cron create because this review is read-only.

## Solution Assessment

No. Gateway validation is the right boundary, but this implementation should use read-only configured-account metadata instead of getChannelPlugin-based plugin loading.
