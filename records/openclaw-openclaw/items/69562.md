## Summary

Keep open. The bug is valid and the PR targets the right gateway mutation boundary, but the branch still loads full channel plugins from gateway validation, is dirty against current main, has a failing gateway watch topology check, and lacks contributor real behavior proof.

## What This Changes

The PR adds gateway cron.add/cron.update validation for explicit announce delivery and failure-destination account IDs, with server tests for rejected and accepted account cases.

## Best Possible Solution

Land a focused gateway-side guard that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates only announce channels in the gateway, and persists after that incomplete guard.

## Solution Assessment

No. Gateway validation before persistence is the right boundary, but this implementation should avoid getChannelPlugin-based full plugin loading and be ported onto current main's read-only channel resolution path.
