## Summary

Keep open. The underlying bug is still valid on current main, but this PR is not ready to merge because it adds account validation through full channel plugin loading in a gateway hot path and still lacks contributor real behavior proof.

## What This Changes

The PR adds gateway cron.add/cron.update validation for explicit announce delivery and failure-destination account IDs, plus server tests for rejected and accepted account cases.

## Best Possible Solution

Land a focused gateway-side guard that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates only announce channels in the gateway, then persists after that guard; I did not create live cron jobs in this read-only review.

## Solution Assessment

No. Gateway validation before persistence is the right boundary, but this implementation should avoid getChannelPlugin-based full plugin loading and instead resolve allowed accounts through read-only configured-account metadata or a config-derived helper.
