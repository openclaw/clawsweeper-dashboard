## Summary

Keep open. The underlying bug still exists on current main and this PR is useful implementation work, but it needs maintainer port/review because the account-validation path is stale against the current read-only channel presence and plugin-cache boundaries.

## What This Changes

The PR adds gateway cron.add/update validation for explicit announce delivery and failure-destination account IDs, plus server tests for invalid IDs, unlisted defaults, normalized IDs, and omitted-channel behavior.

## Best Possible Solution

Land one focused gateway-side validation change that rejects unknown explicit account IDs for cron.add and cron.update, including failure destinations, while preserving omitted account IDs, omitted-channel/last behavior, normalized IDs, unlisted/default account compatibility, and current main's read-only channel/plugin boundary. Add targeted regression tests and a user-facing changelog entry, then resolve the linked bug.

## Reproduction Assessment

Yes. The linked bug gives concrete CLI steps, and current main source shows --account is forwarded into delivery.accountId, only shape-checked by the protocol, and then persists after channel-only gateway validation.

## Solution Assessment

Unclear. Gateway add/update validation is the right boundary, but this PR should be ported onto current main and use a lightweight configured-account resolver rather than full channel plugin lookup before merge.
