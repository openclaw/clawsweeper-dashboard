## Summary

Keep open. The PR targets a real cron validation gap also tracked by https://github.com/openclaw/openclaw/issues/69513, but the current branch still has a blocking gateway runtime-topology regression from full channel plugin loading and lacks contributor-provided real behavior proof.

## What This Changes

Adds gateway cron.add/cron.update validation and server tests for explicit announce delivery and failure-destination account IDs, including default-account, normalized-ID, and omitted-channel compatibility cases.

## Best Possible Solution

Land a focused gateway-side guard that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving current implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account, protocol validation only shape-checks it, gateway validation only checks channels, and cron.add persists after that guard; no live cron creation was run in this read-only review.

## Solution Assessment

No. Gateway validation is the right boundary, but this implementation should avoid getChannelPlugin-based full plugin loading and resolve allowed accounts from read-only metadata or config-derived helpers instead.
