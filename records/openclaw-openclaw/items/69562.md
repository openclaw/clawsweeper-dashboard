## Summary

Keep open. The bug is real and this PR targets the right gateway validation boundary, but the current branch validates by loading full channel plugins, which violates the gateway read-only runtime boundary and is reflected by a failing runtime-topology check; contributor real behavior proof is also missing.

## What This Changes

The PR adds gateway cron.add/cron.update validation and server tests for explicit announce delivery and failure-destination account IDs.

## Best Possible Solution

Land focused gateway validation that resolves allowed accounts from read-only manifest/config metadata, covers primary and failure destinations, and preserves implicit last/omitted-channel, normalized-ID, and unlisted default-account compatibility.

## Reproduction Assessment

Yes, from source inspection: the CLI forwards --account, protocol validation only shape-checks it, gateway validation remains channel-only, and runtime preserves explicit accountId precedence. I did not run live cron creation because this review is read-only.

## Solution Assessment

No. Gateway validation is the right boundary, but this implementation should use read-only configured-account metadata instead of getChannelPlugin-based plugin loading.
