## Summary

Keep open: current main still has the cron accountId validation gap and this PR targets a real bug, but the branch is conflicting, lacks after-fix real behavior proof, and still uses full channel plugin loading from gateway validation.

## What This Changes

This PR adds gateway cron.add/cron.update validation and integration tests for explicit announce delivery and failure-destination account IDs.

## Best Possible Solution

Port or replace the branch with a focused gateway-side guard that validates explicit cron announce account IDs through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates channels but not account membership in the gateway, and then persists the job.

## Solution Assessment

No. Gateway validation before persistence is the right boundary, but this implementation should be ported to current main and should use read-only configured-account metadata instead of getChannelPlugin-based full plugin loading.
