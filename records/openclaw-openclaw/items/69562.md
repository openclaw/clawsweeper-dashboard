## Summary

Keep open: current main still lacks cron announce account ID validation, and this PR targets the right bug but cannot merge as-is because it loads full channel plugins from gateway validation, failing the gateway runtime-topology check; contributor real behavior proof is also missing.

## What This Changes

The PR adds gateway cron.add/cron.update validation and tests for rejecting explicit announce delivery and failure-destination account IDs that are not configured for the resolved channel.

## Best Possible Solution

Land a focused gateway guard for explicit announce account IDs on primary and failure destinations, implemented through read-only plugin/config account metadata and covered by regression tests plus runtime-topology proof.

## Reproduction Assessment

Yes. Source inspection plus the linked live report show the CLI forwards --account to delivery.accountId, protocol validation only shape-checks it, gateway validation remains channel-only, and cron.add persists after that guard; I did not run the live cron command because this review is read-only.

## Solution Assessment

No. Gateway-side validation is the right boundary, but this implementation should resolve allowed accounts from read-only plugin/config metadata instead of getChannelPlugin() so it does not materialize bundled plugin runtime in the gateway hot path.
