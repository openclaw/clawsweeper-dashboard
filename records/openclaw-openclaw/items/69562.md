## Summary

Keep open. The linked bug is valid on current main, but this PR is not merge-ready because its account validation uses full channel plugin lookup in a gateway hot path and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds gateway cron.add/update validation for explicit announce delivery and failure-destination account IDs, plus server tests for invalid IDs, unlisted defaults, normalized IDs, and omitted-channel behavior.

## Best Possible Solution

Land focused gateway-side validation that rejects unknown explicit announce account IDs for cron.add and cron.update, including failure destinations, using a lightweight read-only configured-account resolver and preserving omitted-channel/last, normalized IDs, and unlisted default-account compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account into delivery.accountId, only shape-checks it, validates channels but not account membership, and preserves explicit accountId precedence during delivery resolution.

## Solution Assessment

No. Gateway validation is the right boundary, but this PR's current getChannelPlugin-based implementation can materialize full channel plugins in a gateway hot path; the safer solution is a read-only account resolver aligned with the current plugin/cache boundary.
