## Summary

Keep open. The bugfix direction is valid, but this PR is not merge-ready because it lacks real behavior proof and its account validator uses full channel plugin loading from gateway cron validation, with a failing runtime-topology check on the PR head.

## What This Changes

The PR adds gateway cron.add/cron.update validation for explicit announce delivery account IDs and regression tests for primary and failure-destination account IDs.

## Best Possible Solution

Port or replace the branch with a focused gateway-side guard that validates explicit cron announce account IDs through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, unlisted-default, and target-prefix compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates announce channels in the gateway, and then persists the job without account membership validation.

## Solution Assessment

No. Gateway validation before persistence is the right boundary, but this implementation should be ported to current main and should use read-only configured-account metadata instead of getChannelPlugin-based full plugin loading.
