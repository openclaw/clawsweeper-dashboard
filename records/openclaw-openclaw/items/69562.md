## Summary

Keep open. The underlying cron account-validation bug is still present on current main, but this PR is not merge-ready because it lacks real behavior proof, is conflicting against main, and routes gateway validation through full channel plugin loading instead of read-only configured-account metadata.

## What This Changes

The branch adds gateway cron.add/cron.update validation that rejects explicit announce delivery and failure-destination account IDs not configured for the resolved channel, plus regression tests.

## Best Possible Solution

Port or replace the branch with a focused gateway-side guard that validates explicit cron announce account IDs through read-only configured-account metadata while preserving implicit last, default-account, normalized-ID, and unlisted-default compatibility.

## Reproduction Assessment

Yes, from source inspection. Current main forwards --account to delivery.accountId, shape-checks it in protocol validation, validates channels but not account membership in the gateway, and then persists the job.

## Solution Assessment

No, not as submitted. Gateway validation before persistence is the right boundary, but this implementation should use read-only configured-account metadata instead of getChannelPlugin-based full plugin loading.
