## Summary

Keep open. Current main still has task status/progress fields plus several independent renderers, but it does not provide the requested shared compressed operational-summary contract with current state, last successful checkpoint, blocker, and next action.

## What This Changes

Add a shared deterministic operational summary layer for task status and reuse it across task CLI output, task summary views, and auto-reply/status rendering.

## Best Possible Solution

Add an additive TaskOperationalSummary contract in the task subsystem, derive it deterministically from structured task state/events, expose it through task-domain views, and make CLI, /tasks, /status, session_status, and notification rendering consume it while retaining existing raw fields.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an enhancement request. There is a high-confidence verification path by source and docs inspection: current main exposes raw fields and independent formatting, not the requested shared operational summary contract.

## Solution Assessment

Yes. A shared additive builder is the narrowest maintainable direction if it stays deterministic, sanitized, and compatibility-preserving; per-surface copy changes would keep the inconsistency the issue is asking to remove.
