## Summary

Keep open. Current main still exposes raw task status, progress, terminal, and delivery fields plus several independent renderers, but it does not provide the requested shared compressed operational-summary contract with current state, last successful checkpoint, blocker, and recommended next action.

## What This Changes

The issue requests a compact shared task operational-summary contract reused by task CLI, task summary views, and auto-reply/status rendering.

## Best Possible Solution

Add an additive TaskOperationalSummary type and deterministic builder in the task subsystem, expose it through task-domain views, and make CLI, /tasks, /status, session_status, and task notification/status rendering consume that shared contract while retaining existing raw fields.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an enhancement request. There is a high-confidence verification path by source and docs inspection: current main has raw status/detail fields and independent formatting, not the requested shared operational summary.

## Solution Assessment

Yes. A shared additive builder is the narrow maintainable direction if it stays deterministic and backward compatible; per-surface copy changes would preserve the inconsistency this issue is trying to remove.
