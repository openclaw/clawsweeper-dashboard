## Summary

Keep open. Current main still lacks the requested shared compressed operational-summary contract: task records and plugin DTOs expose raw state/progress/terminal fields, while CLI, chat, status, and session_status each assemble their own task phrasing.

## What This Changes

Add a deterministic shared task operational-summary builder with current state, last successful checkpoint, current blocker, and recommended next action across task CLI, task views, and auto-reply/status surfaces.

## Best Possible Solution

Add an additive TaskOperationalSummary type and deterministic builder in the task subsystem, expose it through task-domain views, and have CLI list/show, /tasks, /status, session_status, task notifications, and plugin runtime task views consume the shared summary while preserving existing raw fields.

Not applicable as a runtime reproduction because this is an enhancement request. Source, docs, and test inspection provide a high-confidence verification path: current main has raw task fields and independent renderers, not the requested shared operational-summary contract.

## Solution Assessment

Yes. A shared additive builder is the narrow maintainable direction because it centralizes deterministic phrasing without changing the underlying task lifecycle or breaking existing JSON/plugin fields.
