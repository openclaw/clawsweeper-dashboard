## Summary

The PR correctly identifies that setup latency should not silently shrink the provider’s request timeout, but resetting the inner timer without propagating the outer Codex watchdog signal can leave image work running after Codex has already timed the tool call out. The PR is maintainer-authored and protected, so it must remain open; the cancellation defect is narrow and should be repaired before merge.

## What This Changes

The PR gives image runtime setup and provider requests separate timeout phases, adds phase-specific errors and tests, and documents that preparation time does not reduce the provider request budget.

## Best Possible Solution

Propagate the dynamic-tool caller’s AbortSignal through the image tool into image setup and provider execution, compose it with phase-specific timers, and keep the full provider request budget plus distinct setup/request errors without detached work after the outer deadline.

## Reproduction Assessment

Yes. The branch’s own fake-timer scenario establishes 400 ms of setup followed by a fresh 1,000 ms hanging request, while the Codex wrapper’s configured outer watchdog fires at 1,000 ms from tool start.

## Solution Assessment

No, not yet. Preserving the full provider request budget is the right direction, but the best fix must also propagate outer cancellation so the request cannot outlive the Codex-visible tool deadline.
