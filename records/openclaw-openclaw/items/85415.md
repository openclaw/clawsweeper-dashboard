## Summary

This PR remains necessary because current main still subtracts setup latency from the provider-request timeout, contrary to the existing per-request help text. The patch is focused and has no definite correctness defect, but it changes the effective maximum operation duration and, despite the PR’s “Real behavior proof” section, provides only mocked test evidence; protected maintainer handling, an explicit timeout-contract decision, and real provider proof are required before merge.

## What This Changes

The PR gives image runtime setup and provider requests separate full timeout windows, adds phase-specific timeout errors, and updates focused tests and documentation.

## Best Possible Solution

Retain the phase-specific diagnostics and full provider-request timeout only after maintainers explicitly confirm that `timeoutSeconds` is a per-phase/per-request limit rather than a total operation deadline, then demonstrate the resulting abort and error behavior with a real image provider.

## Reproduction Assessment

Yes, at source level. Current main deducts setup latency from the request timeout, while the branch’s fake-timer test demonstrates a fresh full request budget after slow setup; no real provider reproduction was supplied.

## Solution Assessment

Unclear. Separating setup and request errors is maintainable and matches the existing per-request wording, but maintainers must decide whether setup and every retry should each receive a fresh timeout before this becomes the supported contract.
