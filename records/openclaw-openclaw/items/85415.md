## Summary

Keep this PR open for maintainer review. The patch is focused and appears correct, but it intentionally codifies per-phase image timeout semantics, which has upgrade-visible compatibility and availability implications for operators who expected a strict whole-operation wall-clock cap.

## What This Changes

The PR changes image-understanding timeout handling so setup and provider request phases use the full configured timeout independently, updates phase-specific errors, adds focused tests, and clarifies the docs.

## Best Possible Solution

Land the focused fix if maintainers accept per-phase image timeout semantics; otherwise preserve the full provider-request timeout but add an explicit outer operation watchdog for deployments that need a strict wall-clock cap.

## Reproduction Assessment

Yes, at source level: current main subtracts setup latency from the provider request timeout and clamps exhausted budgets to 1ms. I did not run tests because this review was read-only.

## Solution Assessment

Mostly yes: the patch aligns the implementation with the existing per-request help text and adds targeted regression tests. The remaining decision is whether maintainers also want a separate whole-operation wall-clock cap.
