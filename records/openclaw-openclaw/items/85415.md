## Summary

This PR should remain open because its clearer phase-specific errors address a real current-main problem, but the implementation changes one configured deadline into two sequential full deadlines. Direct Codex source inspection confirms the dynamic-tool handler waits for OpenClaw’s response without imposing an outer timeout, so slow setup followed by a stalled provider request can now consume nearly twice the configured duration. The PR is also maintainer-authored and protected by the `maintainer` label, preventing automated cleanup.

## What This Changes

The PR gives image-description setup and provider requests separate full timeout windows, adds phase-specific timeout errors and regression tests, and clarifies timeout semantics in three docs pages.

## Best Possible Solution

Retain the useful setup/request-specific error messages, but derive both phase timers from one absolute configured deadline and document that single owner; then prove the real provider path with redacted timing, abort, and error output.

## Reproduction Assessment

Yes. Current main visibly subtracts setup latency from the provider request budget, while the PR’s focused test demonstrates the proposed semantic difference; an actual provider reproduction was not supplied.

## Solution Assessment

No. Distinct phase diagnostics are useful, but reusing the entire configured timeout for two sequential phases changes the total-duration contract without an outer Codex deadline.
