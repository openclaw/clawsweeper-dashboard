## Summary

This PR addresses a real mismatch in the direct image path, but it is not merge-ready: the patch gives one configured timeout to setup and again to the provider while Codex already derives an outer dynamic-tool timeout from that same setting, and the supplied proof is mocked only. Because the author is a repository member, the PR is maintainer-protected and should remain open for the explicit timeout-owner decision and live proof.

## What This Changes

The PR stops subtracting image runtime setup latency from the provider request timeout, adds separate setup/request timeout errors and focused tests, and clarifies three timeout documentation passages.

## Best Possible Solution

Make one deadline owner authoritative for the complete image-tool invocation, pass its remaining budget and abort signal through setup and provider execution, and keep phase-specific errors by recording which phase owns the expiry; prove both direct and Codex-harness behavior against a real provider.

## Reproduction Assessment

Yes at source level: current main subtracts setup elapsed time before provider completion, and the branch's focused fake-timer test demonstrates the resulting reduced request budget. The outer Codex interaction and a real provider failure remain untested.

## Solution Assessment

No, not yet: phase-specific diagnostics are valuable, but independently restarting the same configured budget duplicates timeout ownership and does not provide consistent semantics across direct and Codex-wrapped callers.
