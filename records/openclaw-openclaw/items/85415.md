## Summary

The phase-specific timeout errors are useful, but the PR restarts the full image timeout after setup without composing the outer Codex dynamic-tool cancellation signal. That lets the outer watchdog return a timeout while the image provider request remains active, and the supplied proof covers mocked timing rather than real cancellation cleanup.

## What This Changes

The PR gives image runtime setup and provider requests separate timeout phases, adds phase-specific errors and focused tests, and clarifies in three docs pages that setup time does not reduce the provider request budget.

## Best Possible Solution

Preserve the clearer setup/request errors and full provider request budget, but thread the outer dynamic-tool AbortSignal through image setup and provider execution and prove that a slow-setup, hanging request is terminated when the outer watchdog fires.

## Reproduction Assessment

Yes, from source inspection: slow setup is followed by a fresh provider timer while the outer Codex watchdog still measures from tool start, and the image API does not consume the caller signal. The PR does not provide a live provider reproduction.

## Solution Assessment

No, not yet. Splitting the phase messages is reasonable, but the best fix must compose caller cancellation so the separate timers cannot leave provider work running after the tool call has ended.
