## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting any deferred configured channel plugin, so unrelated startup plugins can still run registration twice across the setup-runtime load and the post-listen full-runtime reload.

## What This Changes

The issue asks OpenClaw to initialize each plugin only once per gateway restart instead of running duplicate register/load cycles.

## Best Possible Solution

Narrow the post-listen reload to deferredConfiguredChannelPluginIds and merge only those promoted records into the attached registry, with regression coverage for a mixed deferred and non-deferred startup scope.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to start the Gateway with one non-deferred startup plugin that increments a register counter plus one configured deferred channel plugin; current main loads startupPluginIds in setup-runtime mode and later reloads startupPluginIds again in full-runtime mode.

## Solution Assessment

Yes. The narrow maintainable fix is to pass only deferredConfiguredChannelPluginIds into the post-listen full-runtime reload and add a regression test proving unrelated startup plugins are not re-registered.
