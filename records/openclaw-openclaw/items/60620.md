## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting a deferred configured channel plugin, so a non-deferred startup plugin can still register once during setup-runtime startup and again during the post-listen full-runtime reload.

## What This Changes

The issue asks OpenClaw to initialize each plugin once per gateway restart instead of running duplicate register/load cycles.

## Best Possible Solution

Scope the post-listen reload to deferredConfiguredChannelPluginIds and preserve the already attached non-deferred startup plugin runtime, with regression coverage for mixed deferred and non-deferred startup plugins.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to boot the Gateway with one non-deferred startup plugin and one configured deferred channel plugin: current main loads startupPluginIds in setup-runtime mode, then reloads startupPluginIds again after listen.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass only deferredConfiguredChannelPluginIds into the post-listen full-runtime reload and add a regression test proving unrelated startup plugins are not re-registered.
