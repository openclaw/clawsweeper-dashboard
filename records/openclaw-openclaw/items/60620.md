## Summary

Keep open. Current main still reloads the whole startup plugin scope after detecting deferred configured channel plugins, so unrelated startup plugins can still be reactivated during the post-listen full-runtime reload.

## What This Changes

The issue asks OpenClaw to initialize each plugin only once per gateway restart instead of running duplicate registration/load cycles.

## Best Possible Solution

Keep the issue open and fix the deferred configured-channel path so only deferred channel plugins are upgraded to full runtime after listen, while sidecar, memory, provider, and other non-deferred startup plugins keep their original single registration.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to boot the Gateway with a non-deferred startup plugin that increments a register counter plus a configured deferred channel plugin; current main loads startupPluginIds once in setup-runtime mode and then reloads startupPluginIds again after listen.

## Solution Assessment

Yes. The narrowest maintainable fix is to scope the deferred reload to deferredConfiguredChannelPluginIds, or equivalently merge only deferred channel full-runtime records, rather than reworking loader-wide cache semantics.
