## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting any deferred configured channel plugin, so a non-deferred startup plugin can still be activated once during setup-runtime startup and again during the post-listen full-runtime reload.

## What This Changes

The issue asks OpenClaw to initialize each plugin only once per gateway restart instead of running duplicate register/load cycles.

## Best Possible Solution

Limit the post-listen full-runtime reload to deferredConfiguredChannelPluginIds and retain or merge the already attached non-deferred plugin registry state.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a gateway startup plan with one non-deferred startup plugin plus one configured deferred channel plugin; current main loads startupPluginIds once in setup-runtime and again after listen in full-runtime.

## Solution Assessment

Yes. The best fix is a narrow gateway/plugin-loader repair that reloads only deferred channel plugin ids and avoids plugin-specific idempotency guards.
