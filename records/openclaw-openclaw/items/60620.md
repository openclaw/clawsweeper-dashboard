## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting any deferred configured channel plugin, so unrelated startup plugins can still run registration twice across setup-runtime startup and post-listen full-runtime reload.

## What This Changes

Ensure gateway restart initializes each plugin once instead of running duplicate registration/load cycles.

## Best Possible Solution

Narrow the post-listen reload to deferredConfiguredChannelPluginIds, merge the promoted deferred records into the existing registry, and add mixed deferred/non-deferred startup regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway startup plan with one non-deferred startup plugin plus one configured deferred channel plugin; current main loads the whole startupPluginIds in setup-runtime and later reloads the same whole startupPluginIds in full-runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to pass only deferredConfiguredChannelPluginIds into the post-listen reload and preserve the already-attached non-deferred plugin records, rather than adding plugin-specific idempotency workarounds.
