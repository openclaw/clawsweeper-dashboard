## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting a deferred configured channel plugin, so unrelated startup plugins can still run registration twice across the setup-runtime load and post-listen full-runtime reload.

## What This Changes

The issue asks for each plugin to initialize once per Gateway restart instead of running a duplicate register/load cycle.

## Best Possible Solution

Scope the post-listen reload to deferredConfiguredChannelPluginIds and merge only those full-runtime records into the attached registry, leaving non-deferred startup plugins on their original single registration.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to boot the Gateway with one non-deferred startup plugin that increments a register counter plus one configured deferred channel plugin; current main loads startupPluginIds in setup-runtime mode, then reloads startupPluginIds again after listen.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass only deferredConfiguredChannelPluginIds into the post-listen full-runtime reload, with a regression test proving unrelated startup plugins are not re-registered.
