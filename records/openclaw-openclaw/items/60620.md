## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting any deferred configured channel plugin, so non-deferred startup plugins can still run registration twice during one Gateway restart.

## What This Changes

The issue asks Gateway startup to initialize each configured plugin only once per restart instead of replaying registration during deferred plugin reload.

## Best Possible Solution

Scope the post-listen reload to deferredConfiguredChannelPluginIds and preserve or merge the already attached non-deferred plugin registry state.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway startup plan with one non-deferred startup plugin plus one configured deferred channel plugin; current main loads the whole startupPluginIds scope once in setup-runtime and again after listen.

## Solution Assessment

Yes. The narrow maintainable fix is to reload only deferredConfiguredChannelPluginIds after listen and keep existing non-deferred registry records instead of adding plugin-specific idempotency workarounds.
