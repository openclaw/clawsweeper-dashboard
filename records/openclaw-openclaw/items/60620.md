## Summary

Current main and the shipped v2026.5.2 lifecycle no longer take the duplicate setup-runtime plus full-runtime plugin registration path on normal Gateway restart, so this can close as implemented.

## What This Changes

The issue reports duplicate plugin registration/load cycles during Gateway restart, especially when deferred channel plugins are present.

## Best Possible Solution

Keep the shipped post-bind startup plugin runtime lifecycle; any cleanup of the now-inactive deferred reload branch can be handled separately if maintainers want to simplify it.

## Reproduction Assessment

No. The old OC 3.28 logs showed a real duplicate cycle, but current main’s normal Gateway startup does not run the early runtime load that made the later full-runtime reload duplicate plugin registration.

## Solution Assessment

Yes. Loading startup plugin runtime once after the listener is bound is a narrower maintained lifecycle fix than adding plugin-specific idempotency guards for every affected plugin.
