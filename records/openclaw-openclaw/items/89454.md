## Summary

Keep open: this is a member-authored Feishu regression-fix PR with useful unique work and no definite line-level correctness finding, but it is currently conflicting with `main` and needs maintainer acceptance of the auth-provider timing tradeoff before merge.

## What This Changes

The PR resolves Feishu outbound `appId`/`appSecret` SecretRefs through the existing SDK secret resolver before outbound sends, message-tool sends, and reply-dispatcher delivery use the strict Feishu runtime account resolver.

## Best Possible Solution

Refresh the branch, then either land this scoped Feishu repair after owners accept send-time credential resolution or move the same invariant to a single client-build/runtime-snapshot boundary before closing https://github.com/openclaw/openclaw/issues/89338.

## Reproduction Assessment

Yes, at source level: current `main` sends Feishu media through `sendMediaFeishu`, which calls the strict runtime account resolver and throws on unresolved exec/keychain `appSecret` SecretRefs before upload. I did not run a live Feishu tenant or Keychain reproduction in this read-only review.

## Solution Assessment

This is an acceptable bounded bug repair, but not conclusively the best final architecture. The cleaner owner-boundary alternative is to ensure Feishu client creation receives an active resolved runtime credential snapshot once, rather than resolving SecretRefs at each outbound entry point.
