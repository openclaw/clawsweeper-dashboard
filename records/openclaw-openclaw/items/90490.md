## What This Changes

The PR adds periodic gateway retry recovery for restart-continuation session deliveries, wires a shutdown stop handle, and adds lifecycle tests for startup and repeated recovery.

## Best Possible Solution

Land after required checks if maintainers accept the 60s queue-level retry behavior, keeping recovery in gateway scheduled services and relying on the existing backoff, claim, retry-budget, and idempotency protections.

## Reproduction Assessment

Yes. Current main schedules restart-continuation recovery once after startup, while failed session-delivery entries retain retry metadata and backoff for a later recovery pass that is not scheduled during normal uptime.

## Solution Assessment

Yes. The gateway scheduled-services owner is the narrow lifecycle boundary, and reusing the existing session-delivery recovery contract is cleaner than adding a Telegram-specific retry path or a second queue implementation.
