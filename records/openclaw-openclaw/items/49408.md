## Summary

Current main still does not support the reported external symlink workflow: symlinked skill directories are discovered, but any target outside the configured skill root is rejected by realpath containment, so this needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

Request to load skills whose directories under `~/.openclaw/skills` are symlinks to external folders.

## Best Possible Solution

Maintainers should choose and document the boundary: either support external managed-skill symlinks through an explicit opt-in model with loader, audit, docs, and tests aligned, or document them as unsupported and direct users to `skills.load.extraDirs`.

## Reproduction Assessment

Yes, source-reproducible. Current main discovers symlinked directories but rejects candidates whose realpath falls outside the configured skill root, and the existing symlink-escape test asserts the same skip path.

## Solution Assessment

Unclear. The original `fs.stat`-style fix is no longer sufficient because symlink directories are already detected; the maintainable solution depends on the chosen trust model.
