## Summary

Close as duplicate/superseded: current main has an explicit opt-in path for trusted external skill symlink targets, and the remaining default managed-root symlink loading behavior is already tracked by the open canonical issue and fix PR.

## What This Changes

Report asks OpenClaw to load skill directories symlinked from `~/.openclaw/skills` to external folders instead of silently omitting them.

## Best Possible Solution

Keep the remaining default managed-skill symlink policy in https://github.com/openclaw/openclaw/issues/44051 and land https://github.com/openclaw/openclaw/pull/59219 or an equivalent reviewed fix; use `skills.load.allowSymlinkTargets` for intentional external skill repos until then.

## Reproduction Assessment

Yes. Source inspection shows a symlinked skill under `~/.openclaw/skills` whose realpath leaves that root is skipped unless `skills.load.allowSymlinkTargets` includes the target, and the canonical issue has recent live reports of the same shape.
