## Summary

Current main still does not satisfy the reported external symlink workflow. Skill discovery now notices symlinked directories, but the loader realpaths every candidate and rejects targets outside the configured skill root; docs point shared external folders to `skills.load.extraDirs`, and the related audit-boundary PRs were closed unmerged, so this needs maintainer policy/security judgment rather than cleanup closure.

## What This Changes

The issue requests loading skill directories that are symlinked into the managed skills root from external filesystem locations.

## Best Possible Solution

Maintainers should choose and document the intended boundary: either add an explicit opt-in trust model for external symlinked managed skills with loader, audit, docs, and regression tests aligned, or clearly document root-escaping symlinks as unsupported and direct users to `skills.load.extraDirs` or normal skill installs.

## Reproduction Assessment

Yes. The issue's filesystem steps map directly to current source behavior: a symlinked directory is discoverable, then rejected once its realpath falls outside the configured root; existing tests exercise the same external-symlink rejection class.

## Solution Assessment

Unclear. The original one-line `fs.stat`-style fix is no longer the best fit because current main deliberately enforces realpath containment; the maintainable fix depends on the chosen trust model.
