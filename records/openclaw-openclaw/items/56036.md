## Summary

Keep open. Current main still lacks hook-context threading for the unwrapped `splitSdkTools()` → `toToolDefinitions()` fallback path, so the PR is not obsolete; however, the PR branch is currently dirty against main and the patch has blocking review issues in the added test import and the newly constructed hook context.

## What This Changes

The PR adds optional `HookContext` threading through `toToolDefinitions()` and `splitSdkTools()`, passes session metadata from embedded attempt and compaction paths, and adds a regression test for `before_tool_call` hook context delivery.

## Best Possible Solution

Refresh this PR on current main, keep the optional `HookContext` threading through `toToolDefinitions()` and `splitSdkTools()`, fix the added test to use the repo’s declared `typebox` import, build fallback hook context from the effective session key, include the resolved loop-detection config for parity with client tools, and keep #55742 separate unless maintainers intentionally coordinate the two plugin context patches.

## Reproduction Assessment

Yes. The current-main reproduction path is static and focused: an unwrapped tool executed through `splitSdkTools()` reaches `toToolDefinitions()` and calls `runBeforeToolCallHook()` without `ctx`, while the adjacent client-tool path forwards context.

## Solution Assessment

No, not as submitted. The direction is the right narrow fix, but the branch needs a rebase plus concrete corrections for the test import, effective session key, and loop-detection parity before it is the best maintainable solution.
