## Summary

Keep open. Current main still has the fallback `toToolDefinitions()` hook-context gap, so this PR is not obsolete; however, the branch needs a refresh because its public patch no longer applies cleanly to current main.

## What This Changes

This PR adds optional `HookContext` plumbing from embedded-runner call sites through `splitSdkTools()` and `toToolDefinitions()` so fallback `before_tool_call` hook dispatch can receive `agentId` and session metadata, plus a focused regression test.

## Best Possible Solution

Refresh this PR or create a narrow replacement that threads `HookContext` through the current `toToolDefinitions()` fallback path, preserves focused regression coverage, adds the required changelog entry, and makes an explicit decision on `loopDetection` parity with client tools.

## Reproduction Assessment

Yes. A high-confidence reproduction path is the current-main code path `splitSdkTools()` -> `toToolDefinitions()` -> `runBeforeToolCallHook()`: the call has no `ctx`, while `runBeforeToolCallHook()` only exposes `agentId` to plugins when `args.ctx.agentId` is present.

## Solution Assessment

Mostly yes, but not as the branch currently stands. HookContext threading is the narrow maintainable fix and matches the client-tool path, but the patch must be refreshed against current main and should explicitly include or reject loop-detection parity.
