## Summary

Keep this PR open: it targets a real claude-cli chat continuity bug and now has useful live proof, but the patch is too broad because it would let an existing Claude live process ignore stable system-prompt changes that currently force a restart.

## What This Changes

The PR exports `buildClaudeLiveFingerprint`, removes `context.systemPrompt` from the Claude CLI live-session fingerprint, and adds unit coverage for fingerprint stability across volatile system-prompt changes.

## Best Possible Solution

Add a stable live-session prompt fingerprint that excludes per-turn inbound metadata but still rotates the Claude live process when stable system-prompt inputs change.

## Reproduction Assessment

Yes, source-level: current main hashes the full `context.systemPrompt` into the live-session fingerprint while chat code injects volatile inbound metadata into that prompt and separately prepares a static subset. The PR body also now includes redacted live Telegram logs showing restarts before the patch and no matching restart logs after the runtime patch; I did not rerun live Telegram in this read-only review.

## Solution Assessment

No. Removing the whole `systemPromptHash` is broader than the bug boundary; the safer fix is to hash a stable prompt representation or exclude only volatile chat metadata while keeping real system-prompt changes session-invalidating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep stable system-prompt changes in the fingerprint:** `src/agents/cli-runner/claude-live-session.ts:276`

Please hash a stable prompt representation or exclude only the volatile chat metadata instead.
