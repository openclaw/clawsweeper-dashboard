## Summary

Keep open: current main and latest v2026.6.8 still return Twilio realtime TwiML with only a stream URL, then create and connect the realtime provider bridge only after the Media Stream `start` frame. The requested provider-WebSocket prewarm remains useful, but it needs maintainer product/resource review because pre-answer provider sessions can consume quota or billable connection time for no-answer, busy, failed, or abandoned calls.

## What This Changes

The issue asks OpenClaw voice-call to pre-warm realtime provider WebSockets during the TwiML webhook phase so first-turn voice latency is not paid after the Twilio Media Stream starts.

## Best Possible Solution

Design a maintainer-approved Voice Call prewarm lifecycle in the plugin that starts provider connection during TwiML generation, attaches it safely on stream start, keeps lazy fallback, and cleans up unused sessions on terminal or abandoned calls with live OpenAI and Gemini proof.

## Reproduction Assessment

Yes for the source-level timing path: current main and v2026.6.8 return TwiML with only a stream URL, then create and connect the provider bridge after the WebSocket `start` frame. I did not run live Twilio calls, so the exact latency measurements remain reporter-supplied.

## Solution Assessment

Unclear as an immediate patch: prewarming is the plausible performance direction, but the best solution needs maintainer approval on default versus opt-in behavior, provider quota exposure, cleanup semantics, and live Twilio/OpenAI/Gemini proof.
