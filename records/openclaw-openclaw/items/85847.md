## Summary

Keep open: current main and latest release v2026.6.9 still create/connect the realtime provider bridge after the Twilio Media Stream `start` frame, not during the TwiML webhook. The remaining work is real but needs maintainer product review because pre-answer provider sessions can consume quota or resources for calls that never attach media.

## What This Changes

The issue requests pre-warming realtime provider WebSocket sessions during the TwiML webhook phase to reduce first-turn voice-call latency.

## Best Possible Solution

Design a maintainer-approved Voice Call plugin prewarm lifecycle that starts provider setup during verified TwiML generation, attaches the telephony stream on `start`, and reliably closes or falls back on no-answer, failure, timeout, or provider connection errors.

## Reproduction Assessment

Yes for the source-level timing path: current main constructs and connects the provider bridge after the telephony WebSocket start frame. I did not reproduce the reporter's exact latency numbers with live Twilio calls.

## Solution Assessment

Unclear as an immediate patch: provider prewarming targets the right latency source, but the safest solution needs maintainer decisions on default behavior, quota exposure, cleanup, fallback, and live OpenAI/Gemini proof.
