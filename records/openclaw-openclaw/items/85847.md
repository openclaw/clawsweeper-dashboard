## Summary

Keep open: current main and the latest shipped release still return realtime TwiML first and only create/connect the realtime provider bridge after Twilio's stream `start` frame, so the reported latency path is real source-level behavior and not already implemented. The fix should go through maintainer design because pre-answer provider sessions affect quota, cleanup, and default behavior.

## What This Changes

Request pre-warming the Voice Call realtime provider WebSocket during the TwiML webhook response so first caller audio does not wait for provider session startup.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path that can start provider bridge setup during TwiML handling, preserve lazy fallback, buffer early media safely, and close unused prewarmed sessions on terminal call states.

## Reproduction Assessment

Yes for the source-level timing path: current main returns realtime TwiML first, then creates and connects the provider bridge only after Twilio sends the stream `start` frame. I did not run a live carrier call in this read-only review.

## Solution Assessment

Unclear as written: prewarming is the right performance direction, but the best implementation needs maintainer choice on default versus opt-in behavior, provider quota exposure, cleanup, and live proof for both OpenAI and Google.
