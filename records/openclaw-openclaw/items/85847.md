## Summary

Keep open: current main and v2026.6.8 still create/connect the realtime provider bridge after the Twilio Media Stream `start` frame, not during the TwiML webhook. The adjacent pre-answer context issue overlaps in lifecycle timing but does not own provider WebSocket prewarming, and the remaining work needs maintainer product review because pre-answer provider sessions can consume quota or resources for calls that never attach media.

## What This Changes

Request to prewarm realtime provider WebSocket sessions during Twilio TwiML webhook handling so first caller audio does not wait for provider connection setup.

## Best Possible Solution

Design a maintainer-approved Voice Call prewarm lifecycle in the plugin webhook path that starts provider connection during TwiML generation, safely attaches or falls back on stream start, and proves cleanup for unanswered or abandoned calls.

## Reproduction Assessment

Yes for the source-level timing path: current main returns TwiML with only a stream URL, then creates and connects the provider bridge after the WebSocket `start` frame. I did not run live Twilio calls, so the exact latency numbers remain reporter-supplied.

## Solution Assessment

Unclear as an immediate patch: provider prewarming is the right performance direction, but the safest solution needs maintainer decisions on default versus opt-in behavior, quota exposure, cleanup, and live OpenAI/Gemini proof.
