## Summary

Keep open: current main and the latest shipped tag still create/connect the realtime provider bridge only after the telephony stream reaches the WebSocket `start` frame, so the reported first-turn latency path is source-backed and not already implemented; the pre-answer provider-session behavior needs maintainer choice because it can consume provider quota for unanswered or abandoned calls.

## What This Changes

Request prewarming the Voice Call realtime provider WebSocket during the TwiML webhook response instead of waiting for the Twilio Media Stream WebSocket start frame.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path that starts provider setup during TwiML handling, preserves the current lazy fallback, buffers early audio safely, and closes unused prewarmed sessions on terminal call outcomes.

## Reproduction Assessment

Yes for the source-level timing path: current main returns realtime TwiML first, then constructs and connects the provider bridge only after the telephony WebSocket `start` frame. I did not run a live carrier reproduction in this read-only review.

## Solution Assessment

Unclear as written: prewarming is the right performance direction, but maintainers need to choose default versus opt-in behavior, provider quota exposure, cleanup semantics, and live proof expectations for OpenAI and Google.
