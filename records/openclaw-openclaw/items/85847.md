## Summary

Keep open: current main and v2026.6.9 still create and connect the realtime provider bridge after the Twilio Media Stream `start` frame, not during TwiML generation. The requested pre-answer provider prewarm lifecycle remains missing and needs maintainer product review because it can consume provider sessions for calls that never attach media.

## What This Changes

The issue requests pre-warming realtime provider WebSockets during the TwiML webhook so voice-call first-turn latency is lower when Twilio media connects.

## Best Possible Solution

Design a maintainer-approved Voice Call plugin prewarm lifecycle that starts provider setup during verified TwiML generation, attaches the media stream on `start`, and reliably closes or falls back on no-answer, timeout, provider failure, and abandoned-call paths.

## Reproduction Assessment

Yes for the source-level timing path: current main constructs and connects the provider bridge after the telephony WebSocket `start` frame. No for the exact latency measurements: I did not run fresh live Twilio/OpenAI/Gemini calls.

## Solution Assessment

Unclear as an immediate patch: provider prewarming targets the right latency source, but the safest solution needs maintainer decisions on default behavior, quota exposure, cleanup, fallback, and live OpenAI/Gemini proof.
