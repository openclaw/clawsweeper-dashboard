## Summary

Keep open: current main and latest release v2026.6.9 still create and connect the realtime provider bridge after the Twilio Media Stream `start` frame, not during the TwiML webhook. The issue remains useful, but it needs maintainer product review because pre-answer provider sessions can consume quota or connection slots for calls that never attach media.

## What This Changes

Request pre-answer realtime provider WebSocket prewarming for voice-call sessions so first-turn audio is not delayed by provider session setup after Twilio media starts.

## Best Possible Solution

Design a maintainer-approved Voice Call plugin prewarm lifecycle that starts provider setup during verified TwiML generation, attaches the carrier stream on `start`, falls back to lazy setup on failure, and closes unused provider sessions on no-answer, failure, timeout, or hangup with live Twilio/OpenAI/Gemini proof.

## Reproduction Assessment

Yes for the source-level timing path: current main constructs and connects the provider bridge only after the carrier WebSocket `start` frame. A live Twilio/OpenAI/Gemini reproduction is still needed to verify the exact latency magnitude.

## Solution Assessment

Unclear as an immediate patch: prewarming targets the right latency source, but the best implementation needs maintainer decisions on default behavior, quota exposure, cleanup ownership, fallback, and live proof.
