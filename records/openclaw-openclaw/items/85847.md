## Summary

Keep open: current main still creates and connects the realtime provider bridge only after the Twilio realtime stream WebSocket receives its first `start` frame, so the reported latency path is plausible and not already covered; the fix needs maintainer design because pre-answer provider sockets create cleanup, quota, and default-behavior tradeoffs.

## What This Changes

Request to pre-warm the OpenAI/Gemini realtime provider WebSocket during the Voice Call TwiML webhook so first-turn voice latency is not paid after media stream attachment.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path in the plugin-owned webhook/realtime handler that starts the provider bridge during TwiML handling, preserves the current lazy fallback, buffers early media safely, and closes unused prewarmed sessions on terminal/no-answer paths.

## Reproduction Assessment

Yes for the source-level timing path: current main returns realtime TwiML first, then constructs and connects the provider bridge only after the telephony WebSocket `start` frame. I did not run a live carrier reproduction in this read-only review.

## Solution Assessment

Unclear as written: prewarming is the right performance direction, but the best implementation needs a maintainer decision on default versus opt-in behavior, provider quota exposure, cleanup on unanswered calls, and live proof for both OpenAI and Google.
