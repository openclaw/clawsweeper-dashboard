## Summary

Keep open: current main and latest v2026.6.6 still return Twilio realtime TwiML before creating the provider bridge, then create/connect the provider bridge only after the Media Stream `start` frame. No merged fix or canonical duplicate owns this provider-WebSocket prewarm request, and the remaining decision is product-sensitive because pre-answer provider sessions can consume quota or resources for no-answer, busy, failed, or abandoned calls.

## What This Changes

Request pre-warming the OpenAI/Gemini realtime provider WebSocket during the TwiML webhook so first-turn voice-call latency is not paid after Media Stream upgrade.

## Best Possible Solution

Design a maintainer-approved Voice Call prewarm lifecycle in the plugin webhook handler that starts provider connection during TwiML generation, attaches it safely on stream start, keeps lazy fallback, and cleans up unused sessions on terminal statuses.

## Reproduction Assessment

Yes for the source-level timing path: current main and v2026.6.6 return TwiML with only a stream URL, then create/connect the provider bridge after the WebSocket `start` frame. I did not run live Twilio calls, so the exact latency measurements remain reporter-supplied.

## Solution Assessment

Unclear as an immediate patch: provider prewarming is the right performance direction, but the safest solution needs a maintainer decision on default/opt-in behavior, cleanup, quota exposure, and live OpenAI/Gemini proof.
