## Summary

Keep open: source inspection confirms current main and v2026.5.20 still return TwiML without creating the realtime provider bridge, then create and connect that bridge only after the telephony stream start frame. This is a valid Voice Call performance/design request, but pre-answer provider sessions need maintainer decisions around default behavior, quota exposure, and terminal-call cleanup before automation.

## What This Changes

Request prewarming Voice Call realtime provider WebSocket sessions during TwiML webhook handling so first caller audio does not wait on provider session setup after the Twilio Media Stream starts.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path in the plugin-owned webhook that starts the provider bridge during TwiML handling, preserves the lazy fallback, buffers early media, and closes unused prewarmed sessions on no-answer, busy, failed, or abandoned calls.

## Reproduction Assessment

Yes for the source-level timing path: current main and v2026.5.20 return realtime TwiML first, then construct and connect the provider bridge only after the media stream `start` frame. I did not run a live carrier reproduction in this read-only review.

## Solution Assessment

Unclear as written: prewarming is the right performance direction, but the maintainable implementation needs a decision on default versus opt-in behavior, provider quota exposure, terminal-call cleanup, and live proof for both OpenAI and Google.
