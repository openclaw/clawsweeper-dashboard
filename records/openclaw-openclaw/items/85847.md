## Summary

Keep open: current main still creates and connects the realtime provider bridge only after the Twilio/Telnyx media stream `start` frame, so the reported first-turn latency path remains source-reproducible and is not already owned by a merged fix; the remaining decision is how to prewarm without surprising operators with unused provider sessions or quota use.

## What This Changes

The issue requests prewarming the Voice Call realtime provider bridge during the TwiML webhook so first media arrives after provider session setup instead of starting setup on media-stream start.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path that starts provider setup during TwiML handling, keeps the current lazy fallback, buffers early media safely, and closes unused prewarmed sessions on terminal call states.

## Reproduction Assessment

Yes for the source-level timing path: current main returns TwiML first, then constructs and connects the provider bridge only after the media-stream `start` frame. I did not run a live Twilio call in this read-only review.

## Solution Assessment

Unclear as an automated fix: provider prewarming is the right performance direction, but the best implementation needs a maintainer choice on default versus opt-in behavior, quota exposure, cleanup, and live proof across OpenAI and Google.
