## Summary

Keep open. Current main still lacks an opt-in shared session grouping surface for selected channel/thread chats: routing uses only DM session controls, non-DM peers still resolve to channel-specific keys, the session schema is strict without channelGroups/threadIsolation-style fields, and docs/tests still describe isolated channel and thread sessions. The recent maintainer comment also treats this as the canonical tracker rather than a duplicate of the narrower related issues or the closed unmerged PR.

## What This Changes

The issue requests configurable shared sessions across selected Discord/Slack group, channel, and thread chats while keeping unlisted conversations isolated and replies routed to the originating chat.

## Best Possible Solution

Keep this as the canonical product/design tracker for selective shared sessions. The maintainable end state is an additive, opt-in routing/config contract with isolated defaults, explicit channel/thread membership, clear thread inheritance semantics, grouped queue serialization, originating-channel reply delivery, sandbox policy alignment, schema/types/help/docs updates, and regression coverage across routing, Discord, Slack, and session-sensitive behavior.

## Reproduction Assessment

Yes for the current limitation: static source and existing tests show two non-DM channel peers still resolve to distinct channel-specific session keys, and Slack thread replies still append a thread-specific suffix. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Unclear until maintainers settle the exact config shape and safety rules. The requested direction is plausible, but the safest implementation should be a generic session-routing seam rather than Discord/Slack-only special cases or a blunt global-session workaround.
