## Summary

Keep open: current main still has no opt-in surface for selectively sharing chosen group/channel/thread sessions, and the directly linked implementation PR is closed unmerged. Related open items cover narrower or adjacent session-unification models, so they are useful design inputs rather than a safe replacement for this broad tracker.

## What This Changes

The issue requests an opt-in configuration that maps selected group, channel, and thread chats to one shared agent session while preserving isolated defaults and origin-specific replies.

## Best Possible Solution

Define one additive canonical session-group routing contract with isolated defaults, explicit membership, thread inheritance rules, origin-preserving delivery, grouped queue serialization, and documented trust/sandbox semantics, then align narrower trackers to that contract.

## Reproduction Assessment

Yes, at source level. Current main deterministically assigns separate keys to non-DM peers and thread replies, and the strict session schema cannot express selective grouping.

## Solution Assessment

Unclear as proposed: channelGroups is plausible, but the best fix needs a maintainer-approved abstraction that reconciles group-to-main, named groups, identity anchoring, thread inheritance, reply routing, queueing, and sandbox policy.
