## Summary

Current main and v2026.6.5 still isolate non-DM peers and thread replies, and no config surface selectively maps chosen channels or threads to a named shared session. The open group-scope PR only proposes folding groups into the main session and currently has unresolved correctness, security-documentation, and real-proof blockers, so it is not a safe canonical replacement for this broader tracker.

## What This Changes

Add an opt-in way to map selected group, channel, and thread chats to one shared agent session while preserving origin-specific reply routing.

## Best Possible Solution

Define one additive routing contract that lets explicit bindings or peer selectors target a main or named shared session, preserves isolated defaults and origin-specific delivery metadata, makes thread inheritance explicit, and documents that grouping changes context and sandbox trust boundaries.

## Reproduction Assessment

Yes at source level: on current main, selected non-DM peers cannot share a session key and channel threads suffix separate keys, while strict global and binding schemas provide no grouping control. This is established behavior rather than a failing existing contract, so no runtime bug reproduction is applicable.

## Solution Assessment

The requested capability is valid, but the proposed channelGroups shape is not yet proven to be the best permanent API. Maintainers should first choose a canonical binding/session-target model that also covers the focused group-to-main use case without introducing parallel configuration sources.

## Risks / Open Questions

- Sharing a transcript/session across chats can expose one channel's conversation and tool context to members of another channel; session grouping must not be treated as authorization.
- The sandbox.mode="non-main" interaction raised in discussion means folding trusted groups into main or another shared key can change tool-execution isolation, not only conversational continuity.
- Thread inheritance, history fetching, reply targeting, queue serialization, and outbound-only route matching must remain consistent across Discord, Slack, and other channel implementations.
- A new global and/or per-binding configuration contract would be upgrade-sensitive public surface and needs one canonical ownership model rather than overlapping groupScope, channelGroups, and threadIsolation knobs.
