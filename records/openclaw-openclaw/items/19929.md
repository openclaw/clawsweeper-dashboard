## Summary

Keep open. Current main still lacks selective channel/thread session grouping: routing, strict session config, docs, and regression tests still preserve isolated non-DM channel sessions, while the prior implementation PR #20078 was closed unmerged and a maintainer comment treats this as the canonical design tracker.

## What This Changes

Request an opt-in session configuration that maps selected Discord/Slack channels and threads to a shared session key while preserving isolated defaults and origin-channel reply routing.

## Best Possible Solution

Add a backward-compatible, explicit channel/thread grouping design in routing, config schema/types/help, docs, and regression tests, with isolated defaults and clear sandbox/trust-boundary semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence current behavior: non-DM peers still resolve to channel/peer-specific keys and the strict session schema/docs expose no selective grouping config.

## Solution Assessment

Unclear until maintainers settle the product design. The maintainable direction is an additive opt-in config surface with explicit channel/thread membership, not a silent default change or a blunt global session collapse.

## Risks / Open Questions

- The exact product shape is still unsettled: global vs per-agent groups, explicit thread inheritance, and interaction with existing `bindings` and `identityLinks` need maintainer design review.
- Shared sessions intentionally expand live context and queue serialization across chats, so sandbox/tool-policy and trust-boundary behavior must be specified before implementation.
- A session-key-only implementation could miss Slack thread history keys, Discord parent-thread context, reply routing, and transcript write-lock behavior.
