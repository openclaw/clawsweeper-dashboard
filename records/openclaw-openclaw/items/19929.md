## Summary

Keep open: current main and the latest v2026.6.8 release still lack an opt-in surface for selectively sharing chosen group/channel/thread sessions, and the newer open groupScope PR only covers a narrower group-to-main path for a related issue.

## What This Changes

Request an opt-in configuration model that maps selected group, channel, or thread chats to a shared agent session while preserving isolated defaults and origin-specific reply routing.

## Best Possible Solution

Keep this as the broad shared-session tracker and coordinate it with the narrower groupScope PR, then design one opt-in routing contract with isolated defaults, explicit channel/thread membership, origin-preserving reply routing, grouped queue serialization, schema/types/docs updates, and security audit guidance.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives isolated keys for non-direct peers and thread suffixes, and the strict config schemas cannot express the requested selected channel/thread grouping surface.

## Solution Assessment

Unclear as a final product path. The proposed channelGroups shape is plausible, but maintainers should choose whether the permanent abstraction is channelGroups, groupScope, named inbound session routing, or an extension of conversation bindings before implementation.
