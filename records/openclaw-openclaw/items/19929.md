## Summary

Keep open: current main and the latest release still lack an opt-in shared-session surface for selected group/channel/thread chats, the linked implementation PR is closed unmerged, and related open issues are design inputs rather than a safe replacement.

## What This Changes

The issue requests an opt-in configuration to route selected Discord/Slack channels and threads into one shared session while leaving unlisted chats isolated and replies routed to the origin chat.

## Best Possible Solution

Define an additive opt-in shared-session contract with isolated defaults, explicit channel/thread membership, optional thread inheritance, preserved origin reply routing, grouped queue serialization, schema/help/docs updates, and focused Discord/Slack/security regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: configured channel/thread grouping cannot be expressed because current routing builds distinct non-DM channel/session keys and the strict session schema has no grouping option.

## Solution Assessment

Unclear. A `channelGroups`-style option is plausible, but maintainers still need to settle the product/security model across grouped channels, threads, DMs, queueing, and sandbox policy before implementation.
