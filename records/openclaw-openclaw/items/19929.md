## Summary

Keep open. Current main still lacks an opt-in shared-session surface for selected group/channel/thread chats; the existing code, docs, and tests preserve isolated channel/thread session behavior, and the prior implementation PR was closed unmerged while related issues are design inputs rather than replacements.

## What This Changes

The issue requests opt-in configuration to route selected group/channel/thread chats for the same agent into one shared session while keeping unlisted chats isolated.

## Best Possible Solution

Design an additive, explicit session grouping contract with isolated defaults, deterministic origin-channel reply routing, grouped queue serialization, thread inheritance policy, config/schema/help/docs updates, and regression coverage across routing, Discord, Slack, and sandbox-sensitive behavior.

## Reproduction Assessment

Yes, for the missing capability from source inspection: current main has no config/schema/route path for selected channel or thread groups, and non-DM peers still resolve to channel-specific session keys. Live channel execution was not needed to confirm the requested surface is absent.

## Solution Assessment

Unclear as an implementation choice because this is still a product/config design request. The best path is a narrow additive design that preserves isolated defaults and resolves thread, sandbox, and reply-routing semantics before coding.
