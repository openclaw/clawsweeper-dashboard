## Summary

Keep open. Current main still lacks selective shared session grouping for chosen group/channel/thread chats, and recent maintainer/bot reviews treat this as the broad canonical tracker rather than a duplicate of narrower related requests.

## What This Changes

Requests an opt-in config surface that maps selected group/channel/thread chats to a shared session key while keeping unlisted chats isolated and replies routed to the originating chat.

## Best Possible Solution

Design an additive core session grouping contract with isolated defaults, explicit channel/thread membership, deterministic origin-channel replies, grouped queue serialization, schema/help/docs updates, and Discord/Slack regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks the requested config/schema/routing surface and still resolves non-DM peers to channel-specific keys.

## Solution Assessment

Unclear as a final implementation because the exact config shape and trust-boundary policy need maintainer design. The maintainable direction is an opt-in core routing contract that preserves existing isolation unless explicitly configured.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because shared sessions can intentionally move live context and tool-policy assumptions across chat trust boundaries.
