## Summary

Keep this issue open. The requested selective shared-session capability remains absent: non-DM peers and threads still receive isolated keys, the strict session config has no grouping control, and the directly linked implementation PR closed without merging. Related open issues cover narrower or different session-unification models rather than safely replacing this broad tracker.

## What This Changes

Requests opt-in shared session keys for selected group, channel, and thread chats while preserving reply routing and isolated defaults.

## Best Possible Solution

Define one additive peer-group routing contract with explicit membership, trust and sandbox semantics, optional thread inheritance, origin-preserving delivery, grouped queue serialization, and isolated defaults, then align narrower related trackers to that contract.

## Reproduction Assessment

Yes at source level: current main deterministically assigns distinct keys to non-DM peers and thread replies, and the strict schema cannot express selective grouping. This confirms the capability gap without requiring a runtime failure reproduction.

## Solution Assessment

No, the proposed `channelGroups` shape is only one plausible design. The best fix needs a maintainer-approved canonical abstraction that reconciles binding-level, group-to-main, identity, thread, reply-routing, queue, and sandbox requirements.
