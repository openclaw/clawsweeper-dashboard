## Summary

Keep open: current main and the latest release still isolate non-DM group/channel peers, the strict session config has no selected channel/thread grouping surface, and the related open work covers narrower or adjacent routing designs rather than replacing this broader request.

## What This Changes

Request an additive config and routing surface so selected group/channel/thread chats can share one session while replies still return to the originating chat.

## Best Possible Solution

Keep this as the broad canonical tracker until maintainers approve one additive, opt-in shared-session routing contract with schema/types, docs, security guidance, and Discord/Slack/thread regression coverage.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives isolated keys for non-direct peers and strict config cannot express selected channel/thread grouping.

## Solution Assessment

Unclear as a final product path. A channelGroups-style shape is plausible, but maintainers should reconcile it with groupScope, named inbound session routing, identity routing, reply-origin metadata, and sandbox behavior before implementation.
