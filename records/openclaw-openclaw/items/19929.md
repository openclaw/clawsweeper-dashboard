## Summary

Current main and v2026.6.5 still intentionally isolate non-DM channel peers and thread conversations, while the requested selective grouping config does not exist. The closed implementation PR was not merged, and the related open issues address broader or different unification models, so this issue remains the best canonical tracker and needs maintainer product/security design rather than cleanup.

## What This Changes

Requests opt-in shared session keys for selected group, channel, and thread chats while preserving isolated defaults and origin-specific reply routing.

## Best Possible Solution

Define one additive route-level grouping contract with isolated defaults, explicit per-agent channel/thread membership, optional thread inheritance, immutable origin metadata for reply delivery, grouped queue serialization, and documented sandbox/trust behavior across Discord and Slack.

## Reproduction Assessment

Yes, by source inspection: current main deterministically generates isolated non-DM and thread session keys, and the strict schema has no selective grouping option. This reproduces the missing capability, not a violation of an existing documented contract.

## Solution Assessment

Unclear as a final API: the requested `channelGroups` concept addresses the need, but maintainers should compare a generic binding/route policy because it may express per-agent membership and trust semantics more cleanly than a global channel-only map.

## Risks / Open Questions

- A grouped session can expose one chat's transcript or tool-derived context to members of another chat unless membership, authorization, and audit semantics are explicit.
- Treating selected groups as main or trusted sessions can change `sandbox.mode: non-main` behavior and widen host-tool access across a trust boundary.
- Thread inheritance and origin reply routing must be designed together so a shared queue does not lose the source channel/thread or send a response to the wrong destination.
- The reported shared-workspace overwrite concern is plausible, but session grouping alone should not be treated as a general transactional fix for concurrent file writes.
