## Summary

Keep open: current main and the latest release still lack an opt-in surface for selectively sharing chosen group/channel/thread sessions, and the open related PR only covers a narrower group-to-main path that is not merged or proof-cleared.

## What This Changes

The issue requests an opt-in configuration that lets selected Discord/Slack channels or threads share one agent session while preserving isolated defaults and replying back to the originating chat.

## Best Possible Solution

Define one additive canonical session-group routing contract with isolated defaults, explicit membership, thread inheritance rules, origin-preserving delivery, grouped queue serialization, and documented trust/sandbox semantics, then align narrower trackers to that contract.

## Reproduction Assessment

Yes, at source level. Current main deterministically assigns separate keys to non-DM peers and thread replies, and the strict session schema cannot express selective group/channel/thread sharing.

## Solution Assessment

Unclear as proposed: channelGroups is plausible, but the best fix needs a maintainer-approved abstraction that reconciles named groups, group-to-main, canonical inbound sessions, identity anchoring, thread inheritance, reply routing, queueing, and sandbox policy.
