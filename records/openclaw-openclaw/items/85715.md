Keep this PR open, but it is not merge-ready: the central DSML recovery idea is useful, yet the current branch can promote malformed text into executable empty-argument tool calls, can duplicate work if native tool calls arrive later in the stream, and still lacks real proof of a live DSML response being recovered into an executed tool call. ## What This Changes

Adds DeepSeek/Foundry DSML text recovery in the OpenAI completions stream and skips the DeepSeek V4 thinking wrapper for the Microsoft Foundry fallback path, with focused parser and transport regression tests.

## Best Possible Solution

Centralize DSML capture and strict parsing in the DeepSeek filter, emit synthetic calls only after the stream has ruled out native tool_calls, preserve every executable wrapper kind already stripped by current main, and require redacted live proof of a recovered DSML tool-use turn before merge.
