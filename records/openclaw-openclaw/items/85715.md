## Summary

Keep this PR open, but not merge-ready: current main still only filters leaked DeepSeek DSML text, while this branch adds useful recovery work with blocking native-call precedence, DSML-kind coverage, proof, and security-boundary concerns.

## What This Changes

The PR adds DeepSeek/Foundry DSML text-to-tool-call recovery in the OpenAI completions stream and excludes Microsoft Foundry DeepSeek V4 from the generic DeepSeek `thinking` payload wrapper.

## Best Possible Solution

Land a narrower recovery path that shares the DSML filter’s wrapper capture, defers executable recovery until the stream is complete or native calls are ruled out, preserves provider request-shaping compatibility, and includes redacted live proof of an actual recovered DSML tool call executing.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. Source inspection does show current main strips DeepSeek DSML blocks without creating tool calls, and the related PR body includes copied live evidence of DSML markup being returned as assistant text.

## Solution Assessment

No, the current PR is not yet the best way to solve it. The recovery should be owned by the DSML filter/capture contract, delayed until native tool calls are ruled out, and proven against a real provider response before merge.
