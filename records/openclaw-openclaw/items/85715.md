## What This Changes

The PR replaces DeepSeek DSML text filtering with a parser that emits synthetic OpenAI-completions tool calls, and changes DeepSeek/opencode-go request shaping to omit the DeepSeek-style thinking field.

## Best Possible Solution

Land one canonical DeepSeek recovery path that parses the observed invoke/parameter DSML shape, marks recovered calls as toolUse, suppresses recovery when native tool_calls are present, preserves provider request-shaping compatibility, and includes redacted live DSML recovery proof.

Full review comments:

- **[P1] Set toolUse for recovered DSML calls:** `src/agents/openai-transport-stream.ts:2588`
  - body: DeepSeek's failure mode returns DSML in `content` with `finish_reason: "stop"` and no native `tool_calls`. This helper adds a synthetic `toolCall`, but never flips `output.stopReason` to `toolUse`; downstream runner metadata and tool execution depend on that stop reason, so the recovered call can still be treated as a terminal text turn instead of being executed.
  - confidence: 0.91
- **[P1] Parse the observed invoke/parameter DSML shape:** `src/agents/deepseek-dsml-parser.ts:116-119`
  - body: The live reproduction captured in https://github.com/openclaw/openclaw/pull/86032 uses `<｜DSML｜invoke name="session_status">` plus `<｜DSML｜parameter ...>` inside `<｜DSML｜tool_calls>`. This parser only recognizes `<tool_name>/<arguments>` or JSON payloads, so the actual observed provider output still returns null here and gets dropped.
  - confidence: 0.88
- **[P1] Suppress DSML recovery when native tool calls are present:** `src/agents/openai-transport-stream.ts:2608`
  - body: The completions loop processes `choiceDelta.content` before `choiceDelta.tool_calls`, so a chunk containing both DSML-looking text and native tool_calls will emit a synthetic call here and then the real native call. That can duplicate tool execution; native tool_calls should take precedence and suppress synthetic recovery for that response.
  - confidence: 0.86
