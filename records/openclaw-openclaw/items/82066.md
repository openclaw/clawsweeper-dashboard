## Summary

Current main already implements the intended blocked-tool-call recovery through the merged gateway diagnostics fix, and that implementation is safer than this branch because it gates abort recovery on actual tool/progress staleness rather than session age.

## What This Changes

This PR makes stale `blocked_tool_call` diagnostics request stuck-session recovery with `allowActiveAbort` after `diagnostics.stuckSessionAbortMs` and adds focused tests.

## Best Possible Solution

Close this branch and keep the merged https://github.com/openclaw/openclaw/pull/82369 implementation on main; broader tool timeout policy should stay in a separate canonical discussion if it remains needed.

## Reproduction Assessment

Yes, at source level: before the merged replacement, stale native tool calls could be classified as `blocked_tool_call` with `recoveryEligible: false`, and linked live logs showed `recovery=none`. I did not run a live reproduction in this read-only review.

## Solution Assessment

No for this branch: the current-main implementation from https://github.com/openclaw/openclaw/pull/82369 is the safer solution because it checks actual tool and progress staleness before aborting, while this branch uses session age.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate recovery on tool staleness, not session age:** `src/logging/diagnostic.ts:478`
  - body: This helper makes a blocked tool call recoverable when the session `ageMs` crosses the abort threshold, even if the active tool and last progress only recently crossed the warning threshold. A session can be old before a native tool call starts, so this can abort relatively recent tool work; current main avoids that by checking `activeToolAgeMs` and `lastProgressAgeMs` against the abort threshold.
  - confidence: 0.88
