## Summary

Keep open: current main still rejects or delays steer-mode messages during active non-streaming tool work, and multiple open implementation PRs already target this cluster.

## What This Changes

The issue requests that `steer` and `steer-backlog` messages be injected into active main-session turns during tool execution instead of waiting behind the channel/session queue until the turn completes.

## Best Possible Solution

Land one focused active-run steering fix that lets eligible `steer` and `steer-backlog` messages reach tool-boundary steering during active non-streaming work while preserving serialization for normal followup runs.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with `messages.queue.mode: "steer"` or `"steer-backlog"`, send a same-session Discord or Telegram message during active tool execution; current main either holds the message behind keyed inbound serialization or rejects steering through streaming-only gates.

## Solution Assessment

Unclear whether any open PR is best as-is. The maintainable direction is lifecycle-aware active-run steering plus channel queue handling that short-circuits only eligible steer messages before they wait behind the active run.
