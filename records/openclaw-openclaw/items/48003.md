## Label Justifications

- `P1`: Multiple users report a current broken Discord, Telegram, WebChat, and Codex-harness workflow where steering reaches a later turn instead of the active run.
- `impact:message-loss`: Steer messages are not delivered to the intended active turn and are effectively misdelivered as later follow-ups.
- `impact:session-state`: The missed steer changes what context the active session sees before the next model/tool boundary.

## Summary

Keep this open. Current main and the latest release still have the streaming-only steering gates that cause active tool-phase steer messages to be deferred, and the issue remains paired with open, unmerged fix candidates rather than implemented on main.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` and `/steer` messages are queued behind active main-session turns instead of being injected at the next supported tool/model boundary.

## Best Possible Solution

Keep this as the canonical active-run steering regression and resolve it by landing or replacing the focused lifecycle-aware steering fix, with the admission-lane fix reviewed alongside it if maintainers decide both layers are required.

## Reproduction Assessment

Yes, at source level. With queue mode `steer`, send a second message while the active run is in a non-streaming tool phase; current main's `isStreaming` gates reject or defer steering before it can reach the active runtime.

## Solution Assessment

No, current main and the latest release are not the solution yet. The narrow maintainable path is lifecycle-aware steering for active non-streaming phases, with admission-layer changes reviewed separately or together if required.
