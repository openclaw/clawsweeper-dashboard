## Label Justifications

- `P1`: This is a high-confidence regression in active steering that affects real Discord and Telegram main-session workflows.
- `impact:session-state`: The failed steering path keeps same-session user input out of the active agent context at the documented runtime boundary.
- `impact:message-loss`: Messages intended for the active turn are delayed or rerouted into later followup turns instead of being delivered to the running session.

## Summary

Keep open. Current main and the latest release still gate same-turn steering on streaming state, so active main-session runs in a tool phase can reject steering and let the inbound message fall through to later followup behavior; the focused open fix path remains the right maintainer action.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` queues mid-turn main-session messages until the active run finishes instead of injecting them at tool/model boundaries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365, or a focused successor, so active non-streaming embedded and reply runs can accept steering without weakening queue or compaction protections.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode: "steer"`, keep the same session active during a tool phase where the run is active but `isStreaming === false`, then send another inbound message; current main skips or rejects steering and falls back to followup behavior.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate, as proposed in the open replacement PR, is the narrow maintainable fix because it matches the documented runtime-boundary contract while preserving queue and compaction protections.
