## Label Justifications

- `P1`: This is a high-confidence regression in active steering that affects real Discord and Telegram main-session workflows.
- `impact:session-state`: The failed steering path keeps same-session user input out of the active agent context at the documented runtime boundary.
- `impact:message-loss`: Messages intended for the active turn are delayed or rerouted into later follow-up turns instead of being delivered to the running session.

## Summary

Keep open. Current main and v2026.5.22 still have the stream-only steering gates that reject or skip same-turn steering during active non-streaming tool phases, and the focused replacement PR is open but not merged.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` messages for main sessions are serialized until the active run completes instead of being injected into the running turn at tool boundaries.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365, or a focused successor, so active non-streaming embedded and reply runs can accept steering at runtime boundaries while preserving compaction and queue protections.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode` to `steer`, keep a session active during a tool phase where `isActive` is true and `isStreaming` is false, then send another inbound message; current main skips or rejects steering and falls back to later delivery.

## Solution Assessment

Yes. The narrow maintainable fix is to replace stream-only acceptance with lifecycle-aware active/stopped checks in the runner, embedded queue, and reply-run registry, which is the direction already captured by the open focused PR.
