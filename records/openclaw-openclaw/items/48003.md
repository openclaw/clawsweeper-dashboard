## Summary

Keep open: current main still has the reported steer-to-followup behavior shape, and the focused fix candidates are still open or unmerged.

## What This Changes

Messages sent with `messages.queue.mode: "steer"` should reach the active main-session run during tool-boundary windows instead of waiting for the current turn to finish.

## Best Possible Solution

Land or replace the focused steer fix so active runs accept lifecycle-safe steering during tool-boundary windows while preserving per-session run serialization and deliberate followup fallback.

## Reproduction Assessment

Yes, by source inspection and reported steps: run a long tool-using Discord or Telegram turn with `messages.queue.mode: "steer"`, then send another message while the run is active. Current main still gates steering on `isStreaming` and serializes same-session inbound work, so the message can fall through as a followup.

## Solution Assessment

Yes, the narrow maintainable direction is lifecycle-aware steering plus channel delivery that reaches the steering helper before same-session serialization blocks it. The final patch should keep the queue safety properties and add regression coverage around active non-streaming runs.
