## Summary

Keep open. Current main still does not satisfy the documented steer behavior: same-session channel work can remain serialized before steering is attempted, and the core steering helpers still reject active but non-streaming runs. The comments and related PRs contain useful reproduction and fix context, but the active fix work has not landed yet.

## What This Changes

`steer` and `steer-backlog` should inject inbound messages into active main-session turns at runtime/tool boundaries instead of processing them only after the current turn completes.

## Best Possible Solution

Land a focused, coordinated fix that makes active-run steering lifecycle-aware, preserves intentional serialization for non-steer work, adds channel delivery bypasses where inbound middleware blocks steer/interrupt delivery, and aligns docs/tests with the final runtime boundary.

## Reproduction Assessment

Yes, at source level. Current main documents active-run steering, but the code still serializes same-key inbound work and gates steer delivery on `isStreaming`, so active non-streaming tool phases are provably routed to followup instead of injection; no live channel run was performed in this read-only pass.

## Solution Assessment

No close action is appropriate. The best fix direction is the lifecycle-aware core change in the open replacement PR plus narrow channel bypass work for delivery layers like Telegram/Discord, reviewed and landed rather than closing the tracker early.
