## Summary

Keep open: current main and the latest release still have stream-only steering gates, and the two open same-repo fix candidates are unmerged review targets rather than implemented fixes.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` messages are delayed behind active main-session turns instead of being injected into the running turn at tool/model boundaries.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73365 plus the needed admission/spool work from https://github.com/openclaw/openclaw/pull/89238, or replace them with one current-main successor that lets eligible same-session messages reach active-run steering while preserving route, sender, fallback, and diagnostics behavior.

## Reproduction Assessment

Yes, source-level. Current main still gates embedded and reply-run steering on `isStreaming()`, while the issue comments provide repeated Discord, Telegram, Codex, and WebChat observations during active tool phases.

## Solution Assessment

No single open PR is proven to be the best complete fix yet. The right end state is clear, but maintainers should review or combine the lifecycle-aware injection work with the admission-path work so messages can reach steering before the active run ends.
