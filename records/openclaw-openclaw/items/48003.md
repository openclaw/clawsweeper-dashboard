## Label Justifications

- `P1`: This is a high-confidence regression in active-run steering that affects real Discord, Telegram, WebChat, and Control UI workflows.
- `impact:session-state`: Messages intended for the active turn are kept out of the running session context and can become later follow-up state.
- `impact:message-loss`: User messages are delayed or rerouted away from the intended active-turn delivery path, which is message-delivery loss for steer mode.

## Summary

Keep open. Current main and the latest release still contain the stream-only steering gates and same-session admission behavior that can keep steer-mode messages from reaching active main-session runs; the remaining work is already covered by open implementation PRs rather than ready to close.

## What This Changes

The issue asks for `messages.queue.mode: "steer"` to inject same-session messages into an active main-session turn at tool/model boundaries instead of processing them only after the run finishes.

## Best Possible Solution

Land a reviewed successor that coordinates the lifecycle-aware injection work in https://github.com/openclaw/openclaw/pull/73365 with the same-session admission path in https://github.com/openclaw/openclaw/pull/89238, preserving deliberate followup fallback and clear diagnostics.

## Reproduction Assessment

Yes. A high-confidence source path exists: with queue mode `steer`, a same-session inbound message during an active tool phase can be held by admission and then the runner/queue gates only accept `isStreaming`, so non-streaming active phases fall through to later delivery.

## Solution Assessment

Yes for the tracking direction. The narrow maintainable fix is to let eligible same-session turns reach queue policy and then use lifecycle-aware active/stopped injection instead of stream-only acceptance.
