## Label Justifications

- `P1`: This is a high-confidence regression in active steering that affects real Discord and Telegram main-session workflows.
- `impact:session-state`: The failed steering path keeps same-session user input out of the active agent context at the documented runtime boundary.
- `impact:message-loss`: Messages intended for the active turn are delayed or rerouted into later followup turns instead of being delivered to the running session.

## Summary

Keep open. Current main and the latest release still gate same-turn steering on streaming state, so active main-session runs in a tool phase can reject or skip steering and degrade the inbound message into a later followup; the open focused PR remains the right implementation path.

## What This Changes

The issue asks for steer-mode inbound messages to reach active main-session runs during tool execution instead of waiting behind the per-session run queue.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365, or a focused successor, so active non-streaming embedded and reply runs can accept steering without weakening queue or compaction protections.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure messages.queue.mode to steer, keep the same session active during a tool phase where isActive is true and isStreaming is false, then send another inbound message; current main skips or rejects steering and queues followup behavior.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate, as proposed in the open focused PR, is the narrow maintainable fix because it matches the documented runtime-boundary contract while preserving queue and compaction protections.
