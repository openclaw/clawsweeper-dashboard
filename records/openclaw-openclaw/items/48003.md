## Label Justifications

- `P1`: The issue is a high-confidence regression in active steer delivery that affects real Discord and Telegram workflows.
- `impact:session-state`: The failed steering path prevents same-session user messages from entering the active agent context at the documented runtime boundary.
- `impact:message-loss`: User messages intended for the active turn are delayed or rerouted into later followup turns instead of being delivered to the running session.

## Summary

Keep open. Current main and the latest shipped release still gate steer delivery on streaming state, so active tool-execution runs with `isStreaming === false` can skip or reject steering and degrade to followup behavior; the open focused replacement PR remains the right implementation candidate, not proof that the issue is fixed.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` queues same-session messages until the active main turn finishes instead of injecting them during active non-streaming tool-execution phases.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/73365` or a focused successor that adds lifecycle-aware active/stopped steering for embedded, reply, and main-session runs without weakening serialization or compaction guards.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode: "steer"`, keep a same-session run active during tool execution while `isStreaming` is false, then send another inbound message; current main either skips the steer call or rejects it as non-streaming and queues followup behavior.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate is the narrow maintainable direction because it matches the documented runtime-boundary contract while preserving existing serialization and compaction protections.
