## Summary

Keep open. Current main still has the reported steer-to-followup failure shape, and related open implementation PRs need review before this issue can close.

## What This Changes

Fix `steer` and `steer-backlog` so inbound messages during active main-session runs are injected at the next runtime boundary instead of waiting as followups.

## Best Possible Solution

Land or replace one lifecycle-aware steering fix that lets eligible messages reach active runs during non-streaming tool phases, keeps deliberate followup fallback, and aligns tests/docs with the final boundary.

## Reproduction Assessment

Yes, source-reproducible. Current main still gates steering on `isStreaming`, rejects non-streaming active handles, and serializes Discord same-session inbound work behind `KeyedAsyncQueue`.

## Solution Assessment

Unclear as proposed. The message-handler bypass addresses one channel queue, but the durable fix also needs lifecycle-aware embedded and reply-run injection; the open PRs should be reviewed or consolidated.
