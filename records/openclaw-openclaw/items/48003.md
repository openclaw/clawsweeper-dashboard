## Summary

Keep open. Current main still gates queue-mode steering on streaming state and rejects active non-streaming runs, while recent reporter comments reproduce the same followup-only behavior on Telegram/Codex releases.

## What This Changes

The issue requests that `messages.queue.mode: "steer"` inject inbound messages into active main-session runs at runtime/tool boundaries instead of serializing them until the turn completes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365 or a focused successor that accepts steer messages during active non-streaming embedded, reply, and main-session runs without weakening serialization or compaction guards.

## Reproduction Assessment

Yes. Configure `messages.queue.mode: "steer"`, keep a same-session run active during tool execution while streaming is false, then send another inbound message; current main either does not enter the steer path or rejects it as non-streaming and queues followup work.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate is the narrow maintainable fix because the documented behavior already promises steering at runtime boundaries; docs-only clarification or stale cleanup would leave the shipped regression intact.
