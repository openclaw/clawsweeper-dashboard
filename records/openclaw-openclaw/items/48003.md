## Summary

Keep this issue open. Current main still gates steer injection on streaming state, and an open maintainer PR already targets the active/non-streaming run fix for the same reported behavior.

## What This Changes

`messages.queue.mode: "steer"` should inject inbound messages into an active main-session run at runtime boundaries instead of waiting until the turn completes.

## Best Possible Solution

Review and land, or intentionally replace, https://github.com/openclaw/openclaw/pull/73365 so core embedded/reply steer injection accepts active non-stopped runs with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main still requires `isStreaming` before steering and the existing run-registry test explicitly expects active non-streaming handles to fail with `not_streaming`.

## Solution Assessment

Yes, the maintainable fix direction is to make the embedded and reply-run injection gates lifecycle-aware rather than adding a channel-only workaround. The open maintainer PR is the current focused candidate for that approach.
