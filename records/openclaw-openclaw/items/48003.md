## Label Justifications

- `P1`: This is a high-confidence regression in active steering for real Discord and Telegram workflows.
- `impact:session-state`: The failed steering path keeps same-session user input out of the active agent context at the documented runtime boundary.
- `impact:message-loss`: Messages intended for the active turn are delayed or rerouted into later followup turns instead of being delivered to the running session.

## Summary

Keep open. Current main and the latest release still gate same-turn steering on streaming state, so a main-session run that is active but executing tools can miss the steering path and fall back to queued followup behavior; the open replacement PR remains the right fix path.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` does not inject same-session messages into active main runs during non-streaming tool execution.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365, or a focused successor, so active non-streaming embedded and reply runs can accept steering without weakening serialization or compaction guards.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode: "steer"`, keep the same session active during a tool phase where `isActive === true` and `isStreaming === false`, then send another inbound message; current main skips or rejects steering and queues followup behavior.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate, as proposed in the linked replacement PR, is the narrow maintainable fix because it matches the documented runtime-boundary contract while preserving existing queue and compaction protections.
