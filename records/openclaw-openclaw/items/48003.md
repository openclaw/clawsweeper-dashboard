## Summary

Keep open. Current main still matches the reported failure path: steer-mode delivery is gated on streaming state, active non-streaming embedded/reply runs reject queued steer messages, and Discord inbound work is still serialized by session key before it can reach the reply pipeline. The right path is maintainer review of the existing implementation lane, not cleanup closure.

## What This Changes

The issue requests that steer-mode messages sent during an active Discord/main-session turn be injected at tool boundaries instead of waiting behind the per-session run queue or degrading to followup.

## Best Possible Solution

Land #73365 or an equivalent narrow fix that preserves normal same-session run serialization while allowing steer-mode messages targeting an active run to reach the steering queue during the full agent loop, including tool execution. The fix should cover embedded Pi handles, reply-run backends, and Discord same-session delivery, with focused regression tests for each surface.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: with `messages.queue.mode: "steer"`, send a Discord message during an active main-session tool-execution window; current main either holds the inbound job behind the same session queue or rejects steering through streaming-only guards when the run is active but not streaming.

## Solution Assessment

Unclear whether the open implementation PR is sufficient until maintainers verify it covers both the lifecycle gate and the Discord delivery path. The maintainable solution direction is narrow: keep the serialization benefits, but make only steer-mode messages for an active run bypass or enter the steering path safely.
