## Label Justifications

- `P1`: The issue describes a real active agent/channel workflow regression where user steering messages during long runs are delayed until a later turn.
- `impact:message-loss`: Messages intended for the active turn are not delivered to that turn and instead degrade to delayed follow-up behavior.

## Summary

Keep open: current main and latest release still gate steer delivery on streaming state and reject active non-streaming/tool phases, while the linked implementation PRs are open and unmerged.

## What This Changes

Reported regression where `messages.queue.mode: "steer"` messages for an active main session wait until the active turn finishes instead of being injected at the next active-run boundary.

## Best Possible Solution

Land or replace the focused lifecycle-aware steering and admission fixes (#73365/#89238), with tests and real channel proof for active non-streaming/tool-boundary steering across reply, embedded-run, Codex, and channel admission paths.

## Reproduction Assessment

Yes, source-reproducible: current main only attempts immediate steering when `isStreaming` is true, and both embedded/reply queue paths reject active non-streaming handles; issue comments add live Discord, Telegram, WebChat, and Codex-harness timelines.

## Solution Assessment

No implemented solution yet; the lifecycle-aware direction in #73365 plus the admission fix in #89238 looks like the right shape, but both remain open and need coordinated maintainer review or replacement.
