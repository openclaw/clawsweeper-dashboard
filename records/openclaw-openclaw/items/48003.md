## Summary

Keep open. Current main still gates steer delivery on streaming state, rejects active non-streaming embedded/reply runs, and the latest issue discussion includes fresh Telegram/Codex-harness reproductions on shipped 2026.5.x releases.

## What This Changes

Fix steer and steer-backlog queue handling so same-session messages sent during an active main-session run are injected at runtime boundaries instead of waiting for a followup turn.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73365 or a focused successor that makes active non-streaming embedded and reply runs injectable at runtime boundaries, with regression coverage for Pi, reply-run, and channel ingress paths.

## Reproduction Assessment

Yes, as a high-confidence source reproduction. Configure `messages.queue.mode: "steer"`, keep the same session active while `isStreaming` is false during tool execution, then send another inbound message; current source rejects or bypasses steering and falls back to followup.

## Solution Assessment

Yes. Keeping this open while reviewing the focused lifecycle-aware fix is the narrow maintainable path because current main has not fixed the regression and open PRs already target the implicated files.
