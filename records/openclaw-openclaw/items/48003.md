## Summary

Keep open. Current main still rejects active non-streaming steer delivery, and recent release comments reproduce the same followup-only behavior on Telegram/Codex instead of showing the regression is fixed.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` messages wait behind active main-session turns instead of being injected at tool/runtime boundaries.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73365 or a focused successor that lets active non-streaming embedded, reply, and main-session runs accept steering at runtime boundaries without weakening compaction or per-session serialization guards.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode: "steer"`, keep a same-session run active during tool execution while `isStreaming` is false, then send another inbound message; current main either never calls the steer path or rejects it as `not_streaming` and queues a followup.

## Solution Assessment

Yes. The maintainable direction is a focused lifecycle-aware fix that preserves serialization but lets steer messages reach active non-streaming runs; a docs-only clarification or stale close would not address the shipped behavior.
