## Label Justifications

- `P1`: This is a high-confidence regression in active steering that affects real Discord and Telegram main-session workflows.
- `impact:session-state`: The failed steering path keeps same-session user input out of the active agent context at the documented runtime boundary.
- `impact:message-loss`: Messages intended for the active turn are delayed or rerouted into later followup turns instead of being delivered to the running session.

## Summary

Keep open. Current main and the latest release still gate same-turn steering on streaming state, while the documented runtime boundary says mid-run messages should be delivered at the next tool/model boundary; the active fix path is the open replacement PR https://github.com/openclaw/openclaw/pull/73365.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` messages wait for the current main-session turn to finish instead of being injected into active non-streaming/tool phases.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73365, or a focused successor, so active non-streaming embedded and reply runs can accept steering without weakening queue, compaction, or direct-injection protections.

## Reproduction Assessment

Yes. A high-confidence source path exists: set `messages.queue.mode` to `steer`, keep the same session active in a tool phase where `isActive` is true and `isStreaming` is false, then send another inbound message; current main skips or rejects steering and falls back to later processing.

## Solution Assessment

Yes. The narrow maintainable fix is lifecycle-aware active/stopped injection in the existing embedded and reply steering paths, with queue and compaction protections preserved as in the open replacement PR.
