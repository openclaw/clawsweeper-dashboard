- `P1`: This is a well-supported regression in active agent/channel workflows where real user messages wait behind long turns instead of steering the active session.
- `impact:session-state`: The bug changes when user text enters the active session transcript/context, causing active-run state to diverge from user intent.
- `impact:message-loss`: Messages are not permanently lost, but they are delivered to the wrong turn timing and degrade from same-turn steering to delayed follow-ups.

## Summary

Keep open: current main still has source-level paths where same-session inbound work can be serialized before it reaches active-run steering policy, and the issue has an open related repair PR rather than a merged fix. The report is strong enough to keep as the canonical bug instead of closing as stale or implemented.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` degrades same-session messages into delayed follow-up turns instead of injecting them into the active main session at model/tool boundaries.

## Best Possible Solution

Land a focused fix set that lets same-session visible messages reach active-run queue policy during an active run and makes active embedded-run steering accept the full agent-loop lifecycle, with channel regression coverage for the affected ingress paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send a same-session message while an active turn owns the channel/session lane, and the later message can wait behind that lane before steering policy runs. I did not run a live Discord/Telegram/WebChat repro in this read-only review.

## Solution Assessment

No, current main is not the final solution. The best path is to review and land or replace the open repair work so active messages reach queue policy early enough and active-run steering uses the correct lifecycle boundary.
