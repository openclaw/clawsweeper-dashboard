## Summary

Keep open. Current main and the latest shipped release still gate steer delivery on streaming state, so active tool-execution runs with `isStreaming === false` can skip or reject steering and degrade to followup behavior; the linked open PRs are implementation candidates, not proof that the issue is fixed.

## What This Changes

Allow `steer` queue mode to inject incoming messages into an active main session during tool-execution boundaries instead of processing them as later followup turns.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/73365` or a focused successor that adds lifecycle-aware active/stopped steering for embedded, reply, and main-session runs without weakening serialization or compaction guards.

## Reproduction Assessment

Yes. A high-confidence source path exists: configure `messages.queue.mode: "steer"`, keep a same-session run active during tool execution while streaming is false, then send another inbound message; current main either skips the steer call or rejects it as non-streaming and queues followup behavior.

## Solution Assessment

Yes. A lifecycle-aware active/stopped gate is the narrow maintainable direction because it matches the documented runtime-boundary contract while preserving the existing serialization and compaction protections.
