## Summary

Keep open: current main still has streaming-only steering gates and same-session Discord inbound serialization, so the reported mid-turn steer regression remains valid while PR #73365 is the active implementation lane.

## What This Changes

Allow `messages.queue.mode: "steer"` to inject Discord/main-session messages into an active run at tool boundaries instead of waiting behind per-session queued work.

## Best Possible Solution

Update or land the active fix so steer-mode messages targeting an active run can reach the steering queue during non-streaming tool execution while preserving normal per-session serialization for non-steer work, with focused Discord and Pi/reply regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: with `messages.queue.mode: "steer"`, send a Discord message during an active main-session tool execution window. Current main either keeps it behind the same session queue or rejects steering through streaming-only guards.

## Solution Assessment

Unclear for the open implementation path. PR #73365 is the right lifecycle direction, but the fix needs to cover the Discord inbound queue path as well as the `isStreaming` gates to solve this issue completely.
