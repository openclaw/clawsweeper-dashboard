## What This Changes

`messages.queue.mode: "steer"` should inject mid-turn user messages into active main sessions at the next runtime boundary instead of delaying them behind the active session queue.

## Best Possible Solution

Keep this issue open until a clean current-main fix lands for both active-run injectability and any required admission bypass, with regression coverage and docs aligned for Discord, Telegram, WebChat, and Codex steering behavior.

## Reproduction Assessment

Yes, at source level: current main computes active and streaming state separately but only attempts same-turn steering when streaming is true, and both embedded/reply registries reject active non-streaming handles. The issue also has real-channel reproductions on Discord, Telegram, WebChat, and Codex paths, but I did not run live proof in this read-only review.

## Solution Assessment

No, current main is not the best solution because it still falls back to queued follow-up in the reported active non-streaming window. The best fix direction is a lifecycle-aware injectability change coordinated with the admission-layer fix and Codex/channel contracts.
