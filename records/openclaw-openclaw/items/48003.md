## Summary

Keep open. Current main and the latest release still gate steer delivery on streaming and same-session active-run admission, while the related repair PRs are open and unmerged; this remains the canonical tracker for the active-run steer regression.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` does not inject follow-up messages into an already-running main session at tool boundaries and instead processes them after the active turn completes.

## Best Possible Solution

Coordinate or replay the lifecycle-aware steer injection and same-session admission fixes onto current main, preserve session/thread routing, add focused regression coverage and docs alignment, then close this issue after the merged fix is verified.

## Reproduction Assessment

Yes. Source inspection on current main shows the reply, embedded-run, reply-run, and admission gates that can keep active non-streaming turns from steering, and the issue thread has repeated Discord, Telegram, and WebChat reproductions; no live repro was run in this read-only cleanup review.

## Solution Assessment

No, not as a single pre-queue bypass only. The maintainable fix needs both active lifecycle-aware injection after a message reaches the steering path and an admission/ingress path that lets eligible same-session turns reach queue policy while the run is still active.
