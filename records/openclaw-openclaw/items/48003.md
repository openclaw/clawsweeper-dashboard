## Summary

Keep open: current main still has the same source-level failure shape for active non-streaming steering, and the active implementation work at https://github.com/openclaw/openclaw/pull/73365 plus https://github.com/openclaw/openclaw/pull/75226 is open and unmerged.

## What This Changes

The issue reports a regression where steer and steer-backlog inbound messages are serialized or rejected during active main-session tool execution and become followup turns instead of being injected mid-turn.

## Best Possible Solution

Reconcile or replace the active implementation PRs with a focused fix that lets eligible steer-mode messages reach active-run steering during tool execution while preserving normal serialization for non-steer work, then add regression coverage for the lifecycle gate and channel queue path.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with `messages.queue.mode: "steer"` or `"steer-backlog"`, send a Discord same-session message during active tool execution; current main either holds the message behind same-key inbound serialization or rejects steering through streaming-only gates.

## Solution Assessment

Unclear whether the currently open PRs are complete. The maintainable direction is lifecycle-aware steering plus a channel queue path that can short-circuit eligible steer messages before they wait behind the active run.
