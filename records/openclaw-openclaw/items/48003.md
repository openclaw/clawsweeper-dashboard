## Summary

Keep open. Current main still does not satisfy the documented active-run steer behavior: same-session visible turns can wait at reply admission before queue policy runs, and the downstream steering helpers still require streaming before injection, while the two open implementation PRs are not merged.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` delays mid-turn user messages until the active main-session turn completes instead of injecting them into the active run at the next steering boundary.

## Best Possible Solution

Keep this issue as the canonical tracker until the active-run queue-admission and lifecycle-aware steering fixes are landed or replaced together, with regression coverage and live channel proof showing messages reach the active run instead of becoming delayed followups.

## Reproduction Assessment

Yes, at source level. Current main still has stream-only injection gates and a reply-admission path that can hold same-session visible turns before queue policy runs; multiple issue comments add real-channel timelines on Discord, Telegram, WebChat, and Control UI.

## Solution Assessment

Yes on the fix direction, but not yet on a final merge artifact. The maintainable solution is coordinated admission plus lifecycle-aware injection, reviewed against current main and Codex `turn/steer` contracts, rather than another separate issue or standalone repair PR.
