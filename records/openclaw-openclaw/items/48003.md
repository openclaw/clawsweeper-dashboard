## Summary

Keep open. Current main still gates steer delivery on streaming state, rejects active non-streaming embedded/reply runs, and the latest discussion adds a fresh Telegram/Codex-harness reproduction on v2026.5.7 rather than showing the problem is fixed.

## What This Changes

The issue asks for `steer` and `steer-backlog` messages to reach active main-session turns during tool or non-streaming phases instead of waiting as followup turns.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73365 or a focused successor that makes active non-streaming embedded and reply runs injectable at runtime boundaries, with regression coverage for Pi, reply-run, and channel ingress paths.

## Reproduction Assessment

Yes. Configure `messages.queue.mode: "steer"`, keep a same-session run active while `isStreaming` is false during tool execution, then send another inbound message; current source still rejects or bypasses steering and falls back to followup.

## Solution Assessment

Yes. Keeping this issue open while reviewing the focused lifecycle-aware fix is the right path because current main has not fixed the regression and open PRs already target the implicated files.
