## Summary

Keep open. Current main still does not satisfy the documented steering behavior for same-session inbound messages: Discord and Telegram still serialize normal messages before active-run steering can happen, and the focused target-repo fix work is still open and unmerged.

## What This Changes

The issue reports that `steer` and `steer-backlog` messages are queued as followups instead of being injected into active main-session runs at runtime boundaries.

## Best Possible Solution

Land or replace a focused lifecycle-aware steering fix that lets eligible same-session inbound messages reach the active steering queue before the run finishes, preserves deliberate followup fallback, and aligns the queue docs and regression tests.

## Reproduction Assessment

Yes, with source-level confidence and reporter-confirmed live steps: configure `messages.queue.mode` as `steer` or `steer-backlog`, start a long tool-using Discord or Telegram turn, send another normal message, and current main's serialized queues/fallback policy explain why it becomes a followup instead of active steering. I did not run a live channel reproduction during this read-only pass.

## Solution Assessment

Yes, keeping this issue open is the right tracking outcome; the maintainable fix belongs in a focused PR such as https://github.com/openclaw/openclaw/pull/73365, reconciled with https://github.com/openclaw/openclaw/pull/75226. Closing as implemented would be premature because neither fix is merged into current main.
