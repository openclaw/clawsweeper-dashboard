## Label Justifications

- `P1`: The issue is a documented active-run steering regression with multiple current user reports across Discord, Telegram, WebChat, and Codex-harness workflows.
- `impact:session-state`: Messages intended to enter the active run context can instead be associated with a later turn, changing session and transcript state.
- `impact:message-loss`: User steering messages can miss the active turn and silently degrade into delayed follow-up prompts.

## Summary

Keep open: current main and the latest release still gate steer injection on streaming state, reject active non-streaming run handles, and have open unmerged PRs for the lifecycle/admission fix path.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` messages for main sessions wait behind active turns instead of being injected into the active run at tool/model boundaries.

## Best Possible Solution

Ship a lifecycle-aware steering path that lets eligible same-session messages reach active-run queue policy and injection during active tool execution, while preserving route/session isolation and deliberate followup fallback for rejected steering.

## Reproduction Assessment

Yes, at source level: current main only attempts same-turn steering when `isStreaming` is true, and both embedded and reply-run queue helpers reject active non-streaming handles. The issue discussion also contains multiple real-channel reproductions, but I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear for the exact final patch shape, but current main is not the solution. The best-looking direction is to review and land https://github.com/openclaw/openclaw/pull/73365 plus the admission-fence work in https://github.com/openclaw/openclaw/pull/89238, or replace them with one clean successor that covers both paths.
