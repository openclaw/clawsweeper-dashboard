## Label Justifications

- `P1`: Multiple users report a regression where active-run steering fails across Discord, Telegram, WebChat, and Codex harness sessions.
- `impact:session-state`: The user guidance lands in a later turn instead of the active session context that should receive it.
- `impact:message-loss`: Messages are not permanently lost, but they are delivered to the wrong turn rather than being steered into the active run.

## Summary

Keep this open. Current main and the latest release still contain the active-run steering gates that reject or defer non-streaming tool-boundary messages, and the remaining work is already paired with open fix candidates rather than implemented on main.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` for main sessions queues mid-turn messages until the active run completes instead of injecting them at supported active-run boundaries.

## Best Possible Solution

Keep this issue as the canonical active-run steering regression and resolve it by reviewing and landing a focused lifecycle-aware steering fix, plus the admission-lane fix if maintainers decide both layers are required.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with queue mode `steer`, send a second message while the active run is in a non-streaming tool phase, and current main's `isStreaming` gates reject or defer steering. I did not run a live channel scenario in this read-only pass.

## Solution Assessment

No current main or shipped implementation is the best solution yet. The best fix is a focused lifecycle-aware steering change that keeps compaction/stopped-run guards while allowing active non-streaming tool phases, with admission-lane changes reviewed separately or alongside it.
