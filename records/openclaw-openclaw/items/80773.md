## Summary

Keep open for maintainer review: the requested CLI parity is valid and the PR supplies real terminal proof, but the implementation uses a custom dry-run validation path that bypasses plugin-aware raw config validation, so it can report success for a config that the real write path would reject.

## What This Changes

The branch adds `--dry-run`, `--json`, and `--allow-exec` to `openclaw config unset`, with dry-run validation, focused CLI tests, and config CLI documentation.

## Best Possible Solution

Land one canonical unset dry-run implementation that reuses the shared plugin-aware dry-run validation contract, preserves existing unset guards, updates docs, and then closes https://github.com/openclaw/openclaw/issues/80721.

## Reproduction Assessment

Yes. Current main is source-reproducible: `config unset` exposes only the path argument and action, while the linked issue includes released help output showing the same missing flags.

## Solution Assessment

No. Adding the flags is the right direction, but this patch should not introduce a separate schema-validation path that skips external plugin metadata; the safer fix is to reuse the shared dry-run operation path or the plugin-aware raw validator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use plugin-aware validation for unset dry-runs:** `src/cli/config-cli.ts:2020-2022`
  - body: `config unset --dry-run` now validates with `validateConfigObjectRaw`, which only covers base and bundled-channel validation. `config set/patch --dry-run` and real config writes use plugin-aware raw validation, so an unset that violates an installed external plugin schema can report dry-run success and then fail when actually written. Please use the shared dry-run path or `validateConfigObjectRawWithPlugins` and add a regression with an external plugin schema.
  - confidence: 0.87
