## Summary

Keep this PR open: the requested config unset dry-run parity is valid and recent, but the current patch still misses the existing SecretRef dry-run contract and needs contributor real-behavior proof for the latest head.

## What This Changes

The PR adds `--dry-run` and `--json` handling to `openclaw config unset` plus focused config CLI tests.

## Best Possible Solution

Route unset through the shared config mutation dry-run semantics, or an equivalent helper, so schema validation, SecretRef fallout checks, skipped exec metadata, JSON output, and `--allow-exec` behavior stay consistent across config commands.

## Reproduction Assessment

Yes. Current main source shows `config unset` has no dry-run or JSON options and writes via `replaceConfigFile` after removing the path; the linked issue also includes v2026.5.7 help output showing the parity gap.

## Solution Assessment

No. The requested direction is maintainable, but this implementation duplicates dry-run logic and misses the established exec SecretRef skip/opt-in behavior plus provider-map fallout validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Skip exec refs during unset dry-runs:** `src/cli/config-cli.ts:1820-1823`
  - body: This branch sends affected refs directly to `collectDryRunResolvabilityErrors` while `config unset` has no `--allow-exec` option. If a user dry-runs a nested edit under an exec provider, the provider remains configured in `nextConfig`, so resolving the affected refs can spawn the configured command even though sibling dry-runs skip exec refs by default.
  - confidence: 0.9
- **[P2] Validate provider-map removals for dangling refs:** `src/cli/config-cli.ts:1817-1819`
  - body: When the unset path is `secrets.providers`, `providerAlias` is undefined and the dry-run skips SecretRef fallout checks entirely. Removing the provider map can therefore report success even though existing refs now point at missing providers.
  - confidence: 0.84
