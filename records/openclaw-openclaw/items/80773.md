## Summary

Keep this PR open: the requested CLI parity is valid and source-reproducible, but the patch still has two correctness blockers and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds `--dry-run` and `--json` options to `openclaw config unset` plus three CLI tests for no-write and structured dry-run output behavior.

## Best Possible Solution

Implement unset dry-run through the existing validated config mutation semantics, reject JSON output outside dry-run mode, and merge only after focused tests plus real CLI output proof are present.

## Reproduction Assessment

Yes. The missing capability is source-reproducible because current main registers `config unset` without `--dry-run` or `--json`, and the linked issue includes released help output showing only `-h/--help`.

## Solution Assessment

No. The direction is right, but this patch is not the best implementation because it reports dry-run success before post-change validation and accepts a standalone JSON flag that is then ignored during mutation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate dry-run unsets before reporting success:** `src/cli/config-cli.ts:1798-1804`
  - body: The dry-run branch returns success immediately after removing the path from a cloned config. Normal writes go through write-time validation, and the linked parity request expects the resulting config to be schema-validated, so this can say an unset is safe even when the post-change config would be rejected.
  - confidence: 0.92
- **[P2] Reject standalone JSON mode for unset:** `src/cli/config-cli.ts:2060-2061`
  - body: `--json` is described as a dry-run result, but when it is passed without `--dry-run`, `runConfigUnset` ignores it and still mutates config with the normal human message. Match `config patch` by rejecting `--json` unless dry-run is set.
  - confidence: 0.87
