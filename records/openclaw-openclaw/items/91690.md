## Summary

Keep open: current main still clones and persists replacement session entries directly, while this PR adds a narrow guard and regression coverage for preserving non-empty human-facing session titles. The patch looks functionally sound, but the exported plugin-runtime `upsertSessionEntry` semantics change is compatibility-sensitive and should get maintainer/API-owner signoff before merge.

## What This Changes

The PR changes `upsertSessionEntry` to preserve existing non-empty `label` and `displayName` values when replacement metadata omits or empties them, and adds focused session-store regression tests.

## Best Possible Solution

Land the store-boundary preservation fix only after maintainers explicitly accept or document that `upsertSessionEntry` preserves stable human titles and that intentional clears use the patch/replace path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main clones and persists replacement metadata without merging existing title fields, so an upsert that omits `label`/`displayName` drops them. I did not run tests because this review was read-only.

## Solution Assessment

The store-boundary guard is the narrowest fix for the observed title wipe, but merge readiness depends on maintainer approval of the exported helper semantics. A safer final state is to document or test the intentional clear path before landing.
