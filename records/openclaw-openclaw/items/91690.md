## Summary

Keep this PR open: the source-level bug is still present on current main and in v2026.6.5, and the branch is a narrow, tested fix. The remaining blocker is maintainer/API-owner acceptance of the exported `upsertSessionEntry` behavior change.

## What This Changes

The PR changes session-store upserts to preserve existing non-empty `label` and `displayName` values when replacement metadata omits or empties them, with regression tests for preserve, replace, and intentional clear paths.

## Best Possible Solution

Land this after maintainer/API-owner acceptance that title preservation is the intended exported upsert contract and `patchSessionEntry` replacement remains the explicit clear path.

## Reproduction Assessment

Yes from source inspection, without executing a local repro: current main and v2026.6.5 persist the cloned replacement entry directly, so omitted or empty title fields can erase stored human titles. The linked issue and prior PR describe the same deterministic upsert shape.

## Solution Assessment

Yes, with maintainer contract acceptance: the store boundary is the narrowest durable fix because callers should not have to re-merge stable titles during runtime refresh writes. The safer alternative if compatibility concerns surface is to add an explicit upsert clear mode or documentation before merge.
