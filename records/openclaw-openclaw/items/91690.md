## Summary

## What This Changes

The PR changes session-store upserts to retain existing non-empty `label` and `displayName` values when replacement entries omit or carry empty title fields, and adds focused regression tests.

## Best Possible Solution

Land the store-bound fix only after maintainers confirm that human title fields are protected metadata for `upsertSessionEntry` and that intentional title clearing should use validated patch/update paths.

## Reproduction Assessment

Yes, by source inspection: current main persists the replacement entry directly in `upsertSessionEntry`, and the PR's tests model the omitted/empty title replacement shape. I did not execute tests because this was a read-only review.

## Solution Assessment

Conditionally yes: the store boundary is the narrowest place to protect stable titles from incomplete runtime refreshes. The remaining question is API policy, because the same helper is plugin-visible and previously behaved like a whole-entry replacement.
