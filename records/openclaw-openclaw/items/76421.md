## Summary

Keep open: current main has newer `sessions.list` event-loop mitigations, but I did not find proof that the reported compaction-induced Gateway stall is fixed or superseded; compaction still runs through the Gateway-side agent runtime path.

## What This Changes

The issue asks for Gateway session RPCs such as `sessions.list`, `sessions_send`, and `sessions_history` to remain responsive when large transcript compaction stalls the Node event loop.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so long compactions cannot starve cheap Gateway RPCs, with regression coverage that proves a concurrent lightweight RPC completes during a large compaction workload.

## Reproduction Assessment

No, not with high confidence against current main. The report has concrete v2026.5.2 logs and the source path still shows Gateway-side compaction work, but I did not run a live large-transcript case that stalls the event loop past the 10s RPC timeout.

## Solution Assessment

Unclear. The current async/lightweight `sessions.list` work is a useful partial mitigation, but the reported root cause likely needs a maintainer decision around compaction isolation or cooperative yielding plus Gateway RPC timeout/backpressure behavior.
