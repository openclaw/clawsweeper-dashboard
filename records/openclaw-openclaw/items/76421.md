## Summary

Keep open: current main has useful `sessions.list` event-loop mitigations, but compaction itself still runs through the Gateway-side embedded Pi compaction path and I did not find a current-main fix that proves cheap session RPCs stay responsive during a large compaction stall.

## What This Changes

The issue asks for Gateway session RPCs to remain responsive when large transcript compaction blocks the event loop in v2026.5.2.

## Best Possible Solution

Keep this open and make the compaction/runtime boundary protect cheap Gateway RPCs during large transcript compaction, with a regression test that simulates heavy compaction while a concurrent session RPC completes promptly.

## Reproduction Assessment

No, not as a high-confidence live reproduction against current main. The source path remains clear enough to treat as source-reproducible: compaction still executes in the Gateway-side embedded runner while session RPC callers retain 10s default timeouts.

## Solution Assessment

Unclear. The current lightweight `sessions.list` work is a good partial mitigation, but it does not solve the reported root cause unless compaction itself stops starving the Gateway event loop or the RPC policy explicitly handles that stall.
