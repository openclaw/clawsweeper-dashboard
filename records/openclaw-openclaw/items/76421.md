## Summary

Keep open: current main has adjacent Gateway timeout and `sessions.list` mitigations, but I did not find proof that compaction-induced Gateway event-loop starvation is fixed or superseded. The reported `v2026.5.2` failure class still maps to current code that runs compaction work in the Gateway-side runner flow while Gateway RPC callers retain a 10s default timeout.

## What This Changes

The issue asks for Gateway session RPCs such as `sessions.list`, `sessions_send`, and `sessions_history` to remain responsive when large transcript compaction stalls the event loop.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so long transcript compaction cannot starve cheap Gateway RPCs, with regression coverage proving event-loop responsiveness under a large compaction workload.

## Reproduction Assessment

No, not with high confidence against current main. The report has concrete `v2026.5.2` logs and current source still shows the same Gateway-process compaction and 10s RPC timeout shape, but I did not run a live large-transcript stall reproduction.
