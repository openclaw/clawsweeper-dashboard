## Summary

Keep open: current main has useful `sessions.list` event-loop mitigations, but the compaction path still appears to preprocess and invoke compaction in the Gateway-side flow, and the default Gateway RPC timeout still matches the reported 10s failure mode.

## What This Changes

The issue asks OpenClaw to prevent compaction-induced Gateway event-loop stalls from causing `sessions.list`, `sessions_send`, `sessions_history`, status, and probe RPCs to time out.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so long compactions cannot starve cheap Gateway RPCs, with a regression that exercises a large compaction workload and concurrent lightweight RPC.

## Reproduction Assessment

No, not with high confidence against current main. The v2026.5.2 logs are concrete and current source still shows same-process compaction plus a 10s Gateway timeout, but I did not establish a live large-transcript reproduction.

## Solution Assessment

Unclear. The current async lightweight `sessions.list` work is a good partial mitigation, but the reported root cause likely needs maintainer direction on compaction isolation or cooperative yielding and RPC timeout/backpressure semantics.
