## Summary

Keep open: current main has newer adjacent `sessions.list` event-loop mitigations, but they address list hydration and row construction rather than proving that large compactions can no longer starve Gateway RPCs for longer than the 10s call timeout.

## What This Changes

The issue reports v2026.5.2 Gateway WebSocket RPC timeouts during large-transcript compaction and asks for non-blocking compaction, CPU-impact limits, or separate timeout/queue handling.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so large compactions cannot starve cheap Gateway RPCs, with regression coverage for a concurrent RPC during a large compaction workload.

## Reproduction Assessment

No, not with high-confidence live proof against current main. The report has concrete v2026.5.2 logs and source inspection still shows in-process Gateway-side compaction plus a 10s Gateway timeout, but I did not create a failing large-transcript current-main repro.

## Solution Assessment

Unclear. The async `sessions.list` and lightweight-row changes are useful partial mitigations, but the reported root cause likely needs a maintainer choice between compaction isolation, cooperative yielding, and Gateway RPC timeout/backpressure semantics.
