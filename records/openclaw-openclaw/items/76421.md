## Summary

Keep open: current main has adjacent Gateway timeout mitigations for `sessions.list`, but the reported compaction-induced event-loop stall is not proven fixed or superseded.

## What This Changes

The issue reports that large-transcript compaction in v2026.5.2 can block the Gateway event loop long enough for `sessions.list`, `sessions_send`, `sessions_history`, and status probes to hit 10s timeouts.

## Best Possible Solution

Keep the issue open and change the compaction/runtime boundary so large compactions cannot starve cheap Gateway RPCs, with a regression that exercises a concurrent Gateway call during a large compaction workload.

## Reproduction Assessment

No high-confidence live reproduction against current main. The v2026.5.2 logs are concrete and current source still shows Gateway-side compaction work plus 10s RPC timeouts, but I did not create a failing large-transcript stall locally.

## Solution Assessment

Unclear. The current async `sessions.list` and checkpoint-copy changes are useful partial mitigations, but the reported root cause likely needs a maintainer decision between worker/process isolation, cooperative yielding, and RPC timeout/backpressure semantics.
