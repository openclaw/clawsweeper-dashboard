## Summary

Keep open: current main has useful adjacent Gateway mitigations, but I did not find a fix that isolates or cooperatively schedules compaction so cheap Gateway RPCs stay responsive during a large transcript compaction stall.

## What This Changes

The issue reports that large transcript compaction can block the Gateway event loop long enough for `sessions.list`, `sessions_send`, `sessions_history`, and status probes to hit Gateway timeouts.

## Best Possible Solution

Keep this open until compaction/runtime ownership is changed so long compactions cannot starve cheap Gateway RPCs, with regression coverage that proves a concurrent health or sessions RPC completes while a large compaction is underway.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. The report has concrete v2026.5.2 logs, and source inspection still shows compaction work sharing the Gateway process, but I did not run a large transcript workload that reproduces the 10s timeout today.

## Solution Assessment

Unclear. The existing async `sessions.list` and client readiness changes are good mitigations, but they are not the narrowest complete fix for compaction-induced event-loop starvation; maintainers should choose between compaction isolation, cooperative scheduling, and Gateway backpressure semantics.
