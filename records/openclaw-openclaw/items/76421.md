## Summary

Keep open: current main has adjacent Gateway timeout mitigations, including async `sessions.list` transcript hydration, but I did not find enough evidence that the reported compaction-induced event-loop stall is fixed or superseded.

## What This Changes

The issue reports that large-session compaction in v2026.5.2 blocks the Gateway event loop long enough for `sessions.list`, `sessions.send`, and `sessions.usage` RPCs to hit the 10s timeout.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so long compactions do not starve cheap Gateway RPCs, with regression coverage that proves event-loop responsiveness under a large transcript.

## Reproduction Assessment

No, not against current main with high confidence. The report has concrete v2026.5.2 logs and current source still shows compaction work running in the Gateway process, but I did not establish a live failing transcript that stalls the event loop beyond 10 seconds.

## Solution Assessment

Unclear. The current async `sessions.list` work is a useful partial mitigation, but the reported root cause likely needs a maintainer decision around compaction isolation/cooperative yielding and Gateway RPC timeout/backpressure semantics.
