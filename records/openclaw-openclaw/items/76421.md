## Summary

Keep open: current main has useful adjacent mitigations for slow `sessions.list`, but I did not find evidence that the reported compaction-induced Gateway event-loop starvation is fixed or superseded.

## What This Changes

The issue asks OpenClaw to prevent Gateway WebSocket RPCs such as `sessions.list`, `sessions_send`, and `sessions_history` from timing out when large transcript compaction stalls the Node event loop.

## Best Possible Solution

Keep this open and fix the compaction/runtime boundary so long compactions cannot starve cheap Gateway RPCs, with regression coverage for concurrent RPC responsiveness under a large transcript.

## Reproduction Assessment

No, not with a high-confidence live current-main reproduction. The v2026.5.2 logs are concrete and current source still shows compaction work running in-process, but I did not run a large transcript that proves the 10s timeout on current main.

## Solution Assessment

Unclear. The current async `sessions.list` work is a useful partial mitigation, but the reported root cause needs a maintainer decision around compaction isolation or cooperative yielding plus Gateway RPC timeout/backpressure semantics.
