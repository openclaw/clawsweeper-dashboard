## Summary

Keep open: current main and the latest release still lack `model.usage` emission on the HTTP ingress path, and an open linked fix PR is the active implementation candidate.

## What This Changes

HTTP ingress for `/v1/responses` and `/v1/chat/completions` should emit `model.usage` diagnostics so token and cost exporters see API traffic.

## Best Possible Solution

Land a reviewed fix at the shared ingress finalization boundary, or through a shared usage-emission helper, so both OpenAI-compatible HTTP endpoints emit exactly one `model.usage` event with token and cost fields.

## Reproduction Assessment

Yes. Source inspection on current main shows HTTP ingress reaches `agentCommandFromIngress`, while only channel reply and cron sibling paths emit `model.usage`; I did not run a live HTTP request in this read-only review.

## Solution Assessment

Yes. Emitting from the shared ingress finalization path, preferably via a small shared helper with the existing sibling event shape, is the narrow maintainable fix; the linked open PR should be reviewed before closure.
