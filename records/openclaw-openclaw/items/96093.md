## Summary

Keep open: current main still maps HTTP ingress usage into API responses and session state, but it does not emit the established `model.usage` diagnostic event for the shared `/v1/responses` and `/v1/chat/completions` ingress path, so exporters can miss token/cost telemetry for that traffic.

## What This Changes

The issue asks OpenClaw to emit the existing `model.usage` diagnostic for OpenAI-compatible HTTP API traffic so diagnostics exporters receive token and cost data.

## Best Possible Solution

Add one shared core `model.usage` emission helper for completed agent-run usage and call it from HTTP ingress plus the existing reply and cron surfaces, with regression coverage for both OpenAI-compatible endpoints.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: invoke either OpenAI-compatible HTTP endpoint with diagnostics enabled and provider usage in `agentMeta`; current main returns response usage but has no `model.usage` emission on the ingress path. I did not run a live HTTP repro in this read-only sweep.

## Solution Assessment

Yes, the best fix is to emit the existing diagnostic event at shared ingress finalization using the already-available agent metadata, preferably through a shared helper rather than exporter-specific logic or a new config option.
