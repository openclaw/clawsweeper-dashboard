## Summary

Keep open: current main still has provider/request timeout plumbing but not the requested user-facing idle-timeout controls at agent or cron scope, and recent corroborating comments plus related open timeout work show this needs maintainer scoping rather than cleanup.

## What This Changes

Expose model stream idle timeout as configurable agent default, per-agent, and per-cron settings instead of relying on the current 120s default/provider-timeout behavior.

## Best Possible Solution

Define one public timeout contract and precedence order for model idle, provider request, agent run, and cron wall-clock timeouts, then update schema, CLI, docs, and focused regressions around that chosen contract.

## Reproduction Assessment

Yes for the source-level request: current schemas, cron CLI/payload types, docs, and the resolver do not expose the requested `idleTimeoutMs` surfaces. No live Gemini stall was replayed, so the provider-specific runtime symptom remains based on reporter evidence and source-path analysis.

## Solution Assessment

Unclear. A dedicated idle-timeout hierarchy would address the requested escape hatch, but maintainers should first decide whether that is preferable to extending the existing provider request-timeout contract and landing narrower cron/Gemini fixes.
