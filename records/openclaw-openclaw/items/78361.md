## Summary

Keep open: current main still lacks the requested user-facing idle-timeout controls, and the existing provider timeout path plus open cron-timeout PR do not settle the public config contract.

## What This Changes

Expose model stream idle timeout as configurable agent default, per-agent, and per-cron-job settings instead of relying on the fixed/default watchdog behavior.

## Best Possible Solution

Choose one public timeout contract for stream silence, document its precedence against provider request and run timeouts, then update schema/types, cron CLI/payload handling, docs, and focused regression coverage around that contract.

## Reproduction Assessment

Yes for the source-level config gap: current schema, types, docs, and cron CLI do not expose the requested idle-timeout knobs, while the resolver still falls back to the 120s default when provider or explicit cron timeout inputs do not apply. No live Gemini stall was replayed.

## Solution Assessment

Unclear. A dedicated idle-timeout hierarchy is plausible, but maintainers should first decide whether to extend the existing provider timeout contract or add a separate agent/cron idle watchdog setting.
