## Summary

Close as a duplicate/superseded support report: the logged `sourceReplyDeliveryMode: message_tool_only` path is the documented Discord guild-channel default, and the same generated-but-not-posted symptom was already resolved as guidance in the prior closed Discord issue.

## What This Changes

The issue reports that Discord guild-channel replies are generated in OpenClaw but suppressed with `sourceReplyDeliveryMode: message_tool_only` instead of being sent back to Discord.

## Best Possible Solution

Follow the existing documented path: use a tool-call-reliable model for Discord guild sessions, or set `messages.groupChat.visibleReplies: "automatic"` when legacy automatic posting is desired.

## Reproduction Assessment

Yes for the expected no-auto-post path: current source and tests show Discord guild/channel turns default to `message_tool_only`, and that policy suppresses automatic final-reply delivery while the agent still processes the message. I did not run a live Discord bot because the reported log line maps directly to this source path.

## Solution Assessment

Yes: closing in favor of the existing documented/support guidance is the narrowest maintainable outcome. Automatically posting private final assistant text when the model skips the message tool would change the group-channel reply policy.
