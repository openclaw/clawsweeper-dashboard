## Summary

The report matches OpenClaw's documented Discord guild/channel visible-reply policy rather than a Discord send failure: group/channel final answers stay private unless the model uses `message(action=send)`, or the operator opts back into automatic visible replies.

## What This Changes

The issue reports that Discord guild-channel replies are generated and visible in the OpenClaw UI but are not delivered back to Discord because delivery is suppressed as `message_tool_only`.

## Best Possible Solution

Keep the current documented policy and route this as operator configuration/model selection: use a tool-call-reliable model for Discord guild rooms or set `messages.groupChat.visibleReplies: "automatic"` when automatic final replies are desired.

## Reproduction Assessment

Yes, source inspection reproduces the reported suppression path: a Discord guild message is treated as a channel turn and defaults to `sourceReplyDeliveryMode: "message_tool_only"`. I did not run a live Discord bot, but the log line and screenshots match the documented current behavior.

## Solution Assessment

Yes. The narrow maintainable answer is the existing config/model guidance, not a Discord delivery-code change; a new source bug would need evidence that `message(action=send)` ran successfully or `didSendViaMessagingTool: true` was recorded but Discord delivery still failed.
