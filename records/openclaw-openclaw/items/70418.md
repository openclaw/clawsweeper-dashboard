## Summary

Keep open. Current main has prompt-cache retention, cache-ttl pruning, heartbeat keep-warm guidance, Anthropic cache-control shaping, and Google cachedContents TTL refresh, but it still lacks the proposed opt-in silent Anthropic cache warmer and the remaining decision is product/architecture policy rather than conservative cleanup.

## What This Changes

Add an opt-in per-agent Cache TTL Warmer that refreshes Anthropic prompt-cache prefixes during idle periods without transcript, memory, or channel side effects.

## Best Possible Solution

Keep the design issue open until maintainers choose core vs heartbeat vs plugin/API ownership; an approved implementation should be explicit opt-in, reuse the exact stable conversation request prefix, discard the response, avoid tool execution and all transcript/session/channel/memory writes, document cost, and include focused unit plus live Anthropic validation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/design proposal. The absence check is high-confidence from current source, schema, docs, runtime, and synonym searches showing no `cacheWarmer` surface on main.

## Solution Assessment

Unclear because there is no patch and the ownership boundary is unsettled. A dedicated warmer may be the right shape, but the safest path depends on maintainer policy for paid background calls and whether core should expose a shared plugin/runtime seam instead.
