## Summary

Keep open. Current main has prompt-cache retention, cache-ttl pruning, heartbeat keep-warm guidance, Anthropic cache-control request shaping, and Google cachedContents TTL refresh, but it does not have the proposed opt-in `cacheWarmer` config/schema/runtime/docs surface or a silent no-transcript/no-channel Anthropic TTL warmer. The remaining work is a maintainer design decision because it would add paid background provider calls and needs policy around cost, scheduling, state isolation, and core-vs-plugin ownership.

## What This Changes

Requests an opt-in per-agent Cache TTL Warmer that silently sends minimal Anthropic-compatible requests to refresh prompt-cache TTL during idle periods independently of heartbeat semantics.

## Best Possible Solution

Keep the design issue open. If maintainers approve the direction, add an explicit opt-in cache-maintenance path or shared plugin/API seam that reuses the same stable prefix/tools/system/messages through the cache boundary, appends only a bounded marker, consumes and discards the provider response, avoids tool execution and all transcript/session/channel/memory writes, documents cost and provider limitations, and includes focused unit tests plus live Anthropic validation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/design proposal. The absence check is high-confidence from current source, schema, docs, runtime, and synonym searches showing no `cacheWarmer` surface on main.

## Solution Assessment

Unclear because there is no proposed patch and the ownership boundary needs maintainer judgment. A dedicated warmer may be the right product shape, but it should not be queued until maintainers decide whether core should own paid background model calls, extend heartbeat behavior, or expose a plugin capability seam.
