## Summary

Keep open. Current main does not include the proposed session-store recovery helper or `.prev` backup path, and the PR branch still has blocking correctness issues plus stale gateway-session changes that would regress current main behavior if applied as-is.

## What This Changes

The PR changes session-store loading to throw on corrupt/invalid JSON, writes `.prev` backups before atomic saves, adds transcript/back-up based recovery for gateway session stores, and wires that recovery into `sessions.list` and session resolution with focused tests.

## Best Possible Solution

Update this work on top of current main only if maintainers want the stricter corrupt-store contract. The end state should preserve current gateway request-context behavior, async session listing, deleted-agent guards, and model defaults while adding narrowly tested recovery for corrupt gateway session stores, `.prev` restore, and transcript fallback.

## Reproduction Assessment

Unclear for the original user-facing bug because the PR body provides no concrete reproduction steps or logs. Yes for the review findings: code inspection of the PR head and current main shows the pre-recovery target lookup, the `<=` threshold, and stale `sessions.list`/deleted-agent behavior.

## Solution Assessment

No. The recovery direction may be useful, but this PR is not the best mergeable solution until it is rebased onto current main, recovers before store-target lookup, avoids steady-state transcript scans, and preserves the current session-list and deleted-agent contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recover before resolving the target store:** `src/gateway/sessions-resolve.ts:83`
  - body: In the key branch, `resolveGatewaySessionStoreTarget(...)` runs before the new recovery block, but that resolver reads session stores internally. Since this PR makes malformed stores throw, a corrupt store can fail before recovery is attempted, leaving key-based resolution broken for the case the PR is trying to repair.
  - confidence: 0.9
- **[P2] Use a strict minimum-entry threshold:** `src/gateway/session-store-recovery.ts:172`
  - body: Both normal callers pass `minStoreEntries: 1`, but the `<=` check sends a healthy one-session store through transcript recovery on every list/resolve call. Use `<` so recovery only runs when the store has fewer entries than the requested minimum.
  - confidence: 0.95
- **[P2] Preserve the current sessions.list path:** `src/gateway/server-methods/sessions.ts:560-565`
  - body: This stale handler uses `loadConfig()` and `listSessionsFromStore(...)`, but current main uses `context.getRuntimeConfig()`, a bounded model-catalog load, and `listSessionsFromStoreAsync(...)`. Rebase this hunk so recovery wraps the current path instead of dropping runtime config, effective model defaults, and async yielding for large lists.
  - confidence: 0.87
- **[P2] Keep deleted-agent session checks:** `src/gateway/sessions-resolve.ts:186`
  - body: Current main rejects resolved sessions whose owning agent no longer exists. The PR head's new sessionId and label recovery paths return the matched key directly, so a stale recovered entry can become usable again after rebase unless every successful resolve path keeps `validateSessionAgentExists`.
  - confidence: 0.84
