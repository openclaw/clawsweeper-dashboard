## Best Possible Solution

Track the remaining work in https://github.com/openclaw/openclaw/issues/77783 and land a fresh, narrow loader/doctor recovery PR that preserves current gateway session-list and resolve contracts.

## Reproduction Assessment

Yes for the remaining product issue: current main catches session-store read/parse failures and returns the initialized empty store, and the canonical issue provides a concrete 0-byte Linux incident. The PR body itself does not provide repro steps or after-fix proof.

## Solution Assessment

No. This stale branch is not the best fix because it conflicts with current session-list and resolve behavior; a fresh canonical-issue PR should put recovery at the shared session-store/doctor boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recover before resolving the target store:** `src/gateway/sessions-resolve.ts:83`
  - body: In the key branch, `resolveGatewaySessionStoreTarget(...)` runs before the recovery block, but that resolver can read the session store. A malformed store can therefore throw before recovery is attempted, leaving key-based resolution broken for the corruption case this PR is meant to repair.
  - confidence: 0.9
- **[P2] Use a strict minimum-entry threshold:** `src/gateway/session-store-recovery.ts:172-173`
  - body: Both normal callers pass `minStoreEntries: 1`, but the `<=` check sends a healthy one-session store through transcript recovery on every list/resolve call. Use `<` so recovery only runs when the store has fewer entries than requested.
  - confidence: 0.95
- **[P2] Preserve the current sessions.list path:** `src/gateway/server-methods/sessions.ts:560-565`
  - body: This stale handler uses `loadConfig()` and synchronous `listSessionsFromStore(...)`, while current main uses request-context config, bounded model-catalog loading, diagnostics, and `listSessionsFromStoreAsync(...)`. Rebase the recovery around the current path instead of dropping those behaviors.
  - confidence: 0.87
- **[P2] Keep deleted-agent session checks:** `src/gateway/sessions-resolve.ts:186`
  - body: Current main rejects resolved sessions whose owning agent no longer exists. The new recovery paths must preserve that guard for every successful sessionId and label resolution path so recovered stale rows cannot become usable again.
  - confidence: 0.84
