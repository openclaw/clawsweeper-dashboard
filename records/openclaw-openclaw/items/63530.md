## Summary

Keep open. Current main still does not implement the PR's intended resilient session-store recovery, and the PR remains a plausible implementation path, but it is not merge-ready because the proposed recovery misses the key-target lookup failure path, can run transcript recovery for healthy one-session stores, and needs rebasing against newer sessions.list behavior on main.

## What This Changes

The PR changes session-store loading to throw on corrupt or invalid JSON, writes `.prev` backups before atomic session-store saves, adds gateway recovery from backups/transcripts, and wires that recovery into `sessions.list` and `sessions.resolve` with focused tests.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The end state should define the corrupt-session-store contract, preserve current main's session listing behavior, recover before any key-target lookup can throw, avoid unnecessary transcript scans, and add focused regression coverage for `.prev` restore, transcript fallback, `sessions.list`, and key/sessionId/label resolution.

## Reproduction Assessment

Unclear for the original user-facing bug because the PR body is still template-only and provides no repro steps or logs. Yes for the review findings: code inspection shows `resolveGatewaySessionStoreTarget` reads stores before the new try/catch, and the `<= minStoreEntries` check triggers recovery when a store has exactly one entry.

## Solution Assessment

No, not as proposed. The recovery direction may be useful, but this patch is not the narrowest maintainable solution until it handles the pre-catch target lookup, uses a strict minimum-entry threshold, and is rebased onto current main's gateway sessions flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Recover before resolving the target store:** `src/gateway/sessions-resolve.ts:83-89`
  - body: In the key branch, `resolveGatewaySessionStoreTarget(...)` runs before this recovery block, but that resolver reads session stores internally. A malformed store can therefore throw before recovery is attempted, leaving key-based `sessions.resolve` broken for the corrupt-store case this PR is trying to fix.
  - confidence: 0.88
- **[P2] Use a strict minimum-entry threshold:** `src/gateway/session-store-recovery.ts:171-172`
  - body: Both new normal callers pass `minStoreEntries: 1`, but the `<=` check sends a healthy one-session store through transcript recovery on every list/resolve call. Use `<` so recovery only runs when the store has fewer entries than the requested minimum.
  - confidence: 0.95
