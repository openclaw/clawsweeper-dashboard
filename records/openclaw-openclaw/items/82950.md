## Summary

Keep this PR open: current main and the latest release still compile stored exec approval `argPattern` values directly, and the PR is a focused security hardening patch with sufficient terminal proof. The remaining blocker is maintainer security/compatibility acceptance because unsafe stored patterns will fail closed after upgrade.

## What This Changes

The PR imports the shared safe-regex nested-repetition detector, rejects unsafe exec approval `argPattern` values before `RegExp` construction, warns on rejection, and adds regression coverage for unsafe and safe allowlist patterns.

## Best Possible Solution

Land this PR or an equivalent narrow guard after maintainer security/compatibility review explicitly accepts the fail-closed behavior for unsafe stored `argPattern` entries.

## Reproduction Assessment

Yes. Source inspection shows stored `argPattern` values still reach `new RegExp(argPattern).test(argsString)` on current main and v2026.6.8; I did not run a deliberate hang case in this read-only review.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original `argPattern` is the narrowest maintainable fix and preserves whitespace semantics better than switching this matcher to `compileSafeRegex`.
