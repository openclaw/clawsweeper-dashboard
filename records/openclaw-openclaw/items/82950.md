## Summary

Keep this PR open for maintainer merge review: the focused guard and tests address the source-reproducible ReDoS path, and I found no blocking correctness findings. The remaining decision is whether maintainers accept the fail-closed compatibility and command-approval security-boundary behavior for unsafe stored `argPattern` entries.

## What This Changes

The PR adds the shared nested-repetition regex guard to exec approval `argPattern` matching and adds regression coverage for unsafe and safe allowlist patterns.

## Best Possible Solution

Land this PR or an equivalent narrow guard after maintainer security review explicitly accepts the fail-closed behavior for unsafe stored `argPattern` entries.

## Reproduction Assessment

Yes. Current main and `v2026.6.8` compile stored `argPattern` values with `new RegExp(argPattern)` from the allowlist path, so the ReDoS risk is source-reproducible; I did not execute a hang case in this read-only review.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original `argPattern` is the narrowest maintainable fix and preserves whitespace semantics better than switching the matcher to `compileSafeRegex`.
