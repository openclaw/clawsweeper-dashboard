## Label Justifications

- `merge-risk: 🚨 compatibility`: Existing hand-authored exec approval `argPattern` entries that rely on nested repetition will stop matching after this change.

## Summary

Keep open: current main still compiles exec approval `argPattern` with raw `new RegExp`, and this PR adds a focused guard plus regression coverage for that ReDoS surface. I found no blocking patch defect, but maintainers should explicitly accept the compatibility-sensitive fail-closed behavior for legacy hand-authored patterns.

## What This Changes

The PR adds a `hasNestedRepetition` guard before exec approval `argPattern` regex compilation and adds regression tests for unsafe and safe allowlist patterns.

## Best Possible Solution

Land the guard if maintainers accept the fail-closed compatibility tradeoff, keeping safe `argPattern` semantics unchanged by compiling the original pattern text.

## Reproduction Assessment

Yes. Source inspection on current main shows `matchArgPattern` compiling config-provided `argPattern` with `new RegExp(argPattern)`, and the PR proof exercises the guard against nested repetition in a production build.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff. Calling `hasNestedRepetition` before `new RegExp` is narrower than switching to `compileSafeRegex`, because it avoids trimming and preserves existing safe `argPattern` semantics.
