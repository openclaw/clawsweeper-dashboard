## Summary

Keep this PR open: current main and the latest release still compile exec approval `argPattern` values directly, and this PR is the canonical focused fix, but the fail-closed allowlist behavior needs explicit maintainer acceptance before merge.

## What This Changes

The PR adds a `hasNestedRepetition` guard and warning before exec approval `argPattern` regex compilation, plus regression tests for unsafe and safe allowlist patterns.

## Best Possible Solution

Land this PR or an equivalent narrow guard after maintainers explicitly accept fail-closed behavior for unsafe stored `argPattern` entries; add docs or doctor discoverability first if they want less upgrade surprise.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.8` shows stored `entry.argPattern` reaches `new RegExp(argPattern).test(...)` through `matchAllowlist`; I did not run a deliberate hang case in this read-only review, and the PR body supplies terminal proof.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original untrimmed pattern is narrower than switching this matcher to `compileSafeRegex` and still protects loaded config; maintainer acceptance is needed for the fail-closed compatibility tradeoff.
