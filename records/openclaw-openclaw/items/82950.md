## Summary

Keep this PR open for maintainer review: current main still compiles exec approval `argPattern` values with raw `new RegExp`, while the PR adds the existing nested-repetition guard and focused regression coverage. I found no blocking code defect, but the fail-closed behavior for unsafe legacy allowlist patterns is compatibility-sensitive and needs explicit maintainer acceptance before merge.

## What This Changes

The PR adds a `hasNestedRepetition` check before exec approval `argPattern` regex compilation and adds regression tests for unsafe and safe patterns.

## Best Possible Solution

Land the guard if maintainers accept the fail-closed compatibility tradeoff, keeping safe pattern semantics unchanged by compiling the original untrimmed `argPattern` text.

## Reproduction Assessment

Yes. Current main shows raw `new RegExp(argPattern)` on config data, and the PR proof demonstrates the guard rejecting known nested-repetition patterns from a built production bundle.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff. Calling `hasNestedRepetition` before `new RegExp` is narrower than switching to `compileSafeRegex`, because it avoids trimming and preserves existing safe `argPattern` semantics.
