## Summary

Keep open: this is a focused security hardening PR for the linked ReDoS issue, but the current patch silently turns unsafe stored `argPattern` entries into ordinary allowlist misses or fallbacks. That needs an operator-visible diagnostic path, or explicit security-owner acceptance of silent fail-closed behavior, before merge.

## What This Changes

The branch adds a `hasNestedRepetition` guard before exec approval `argPattern` regex compilation and expands tests for unsafe and safe patterns.

## Best Possible Solution

Land the guard with maintainer-approved operator behavior: surface rejected unsafe `argPattern` diagnostics with docs/tests, or explicitly accept and document silent fail-closed semantics for stored policies.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.1` compile stored `argPattern` with `new RegExp`, and a direct Node check of `(a+)+$` against a crafted non-match timed out under a 2-second guard. I did not run a full live agent approval session with malicious stored config.

## Solution Assessment

No, not yet: the guard is in the right hot path and preserves untrimmed regex semantics, but silently returning `false` is not proven to be the best shipped operator behavior for persisted allowlist policy.
