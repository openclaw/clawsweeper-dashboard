## Summary

This PR should stay open for maintainer review: the patch is a narrow security hardening for a real exec-approval ReDoS surface, with no blocking code finding, but it intentionally makes unsafe legacy argPattern entries fail closed.

## What This Changes

The PR adds a hasNestedRepetition guard before compiling exec approval argPattern regexes and adds regression coverage for unsafe and safe patterns.

## Best Possible Solution

Land the narrow guard and regression coverage as fail-closed security hardening after maintainers accept the compatibility impact, with release or docs guidance if they want operators to audit legacy argPattern entries.

## Reproduction Assessment

Yes. Current main compiles configured argPattern strings with new RegExp before testing arguments, and documented allowlist entries can contain hand-authored regexes; I did not run a live hang because this review kept the checkout read-only.

## Solution Assessment

Yes. Validating with hasNestedRepetition before compiling the original argPattern is the narrowest maintainable code fix and reuses an existing local helper; the remaining question is maintainer acceptance of fail-closed legacy patterns.
