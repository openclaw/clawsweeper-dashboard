## Summary

Keep open: the patch appears correct and well proven, but it changes exec approval allowlist matching at a security boundary, so maintainer security and compatibility acceptance is still needed before merge.

## What This Changes

The PR adds a nested-repetition guard and warning before exec approval `argPattern` regex compilation, plus regression tests for unsafe and safe allowlist patterns.

## Best Possible Solution

Land this PR or an equivalent narrow runtime guard after maintainers explicitly accept fail-closed behavior for unsafe stored `argPattern` entries.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.8` shows stored `entry.argPattern` reaches `new RegExp(argPattern).test(...)`; I did not run a deliberate hang case in this read-only review.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original untrimmed pattern is narrower than switching this matcher to `compileSafeRegex` and still protects loaded config; maintainers still need to accept the fail-closed compatibility tradeoff.
