## Summary

Keep open: the patch is useful and appears correct, but it is not on current main and it intentionally changes stored exec approval `argPattern` matching to fail closed for unsafe regexes, so maintainer security/compatibility acceptance is still needed before merge.

## What This Changes

The PR adds a nested-repetition guard before compiling exec approval `argPattern` regexes and expands exec-command-resolution tests for unsafe and safe patterns.

## Best Possible Solution

Land this PR or an equivalent narrow guard after maintainers explicitly accept fail-closed behavior for unsafe stored `argPattern` entries; add docs or doctor discoverability first if they want less upgrade surprise.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.9` shows stored `entry.argPattern` reaches `new RegExp(argPattern).test(...)` through `matchAllowlist`; I did not run a deliberate hang case in this read-only review, and the PR body supplies terminal proof.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original untrimmed pattern is narrower than switching this matcher to `compileSafeRegex` and still protects loaded config; maintainer acceptance is needed for the fail-closed compatibility tradeoff.
