## Summary

Keep this PR open for maintainer review: the patch is narrow and well-proven, but it intentionally changes exec approval allowlist matching by failing closed for stored unsafe `argPattern` entries.

## What This Changes

The branch adds a `hasNestedRepetition` guard and warning before exec approval `argPattern` regex compilation, plus regression tests for unsafe and safe allowlist patterns.

## Best Possible Solution

Land this PR or an equivalent narrow guard after maintainers explicitly accept fail-closed behavior for unsafe stored `argPattern` entries; add docs or doctor discoverability first if they want less upgrade surprise.

## Reproduction Assessment

Yes. Source inspection shows stored `entry.argPattern` reaches `new RegExp(argPattern).test(...)` through `matchAllowlist`; the PR body supplies built-bundle terminal proof, though I did not run a deliberate hang case in this read-only review.

## Solution Assessment

Yes. Guarding with `hasNestedRepetition` immediately before compiling the original untrimmed pattern is narrower than switching this matcher to `compileSafeRegex`, while still protecting loaded config; maintainer acceptance is needed for the fail-closed compatibility tradeoff.
