## Summary

Keep open because current main and v2026.5.22 have useful failure-class and cooldown foundations, but they still do not implement the requested provider-wide auth-broken quarantine, network short quarantine, or normal `/status` provider-health surface.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits/network/invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health contract, apply it consistently in fallback selection, and expose a redacted provider-health snapshot in normal status while keeping deeper auth diagnostics in `models status`.

Not applicable as a strict bug reproduction because this is a feature and policy request. Source inspection gives high confidence that current main has partial failure-class support but not the requested provider quarantine and normal status provider-health behavior.

## Solution Assessment

Unclear until maintainers settle the hard-auth versus transient-refresh boundary, quarantine scope/duration, network handling, invalid-request fail-fast behavior, and redacted status contract. The requested direction is reasonable, but it is not a narrow mechanical bug fix.
