## Summary

Keep open. Current main has useful failure-class and cooldown foundations, but the requested hard-auth provider quarantine and provider-health visibility in normal status output are still not fully implemented, and the open related PR only covers a narrower session-scoped auth skip/recovery path.

## What This Changes

The issue requests failure-class-aware model/provider fallback with auth-broken quarantine, distinct rate-limit/network/invalid-request handling, and provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model, apply it consistently in fallback selection and auth-state persistence, and expose a redacted provider-health snapshot in status while keeping deeper diagnostics in models status.

## Reproduction Assessment

Yes, at source level: current main shows failure-class classification and profile cooldown/skip foundations, while provider-wide hard-auth quarantine and normal status provider-health output remain absent. I did not replay a live Anthropic OAuth outage.

## Solution Assessment

Unclear until maintainers choose the provider-health policy. The current foundations are useful, but the remaining solution needs an explicit contract for hard auth, rate-limit retry timing, network failure handling, invalid-request boundaries, and redacted status output.
