## Summary

Keep open. Current main has failure classification plus cooldown-aware fallback, but it still lacks the requested provider-level auth quarantine policy, invalid-request fail-fast behavior, network short quarantine, and normal status provider health surface.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits and network failures differently, fails fast on invalid requests, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate limit, network, billing/quota, schema, and invalid-request failures; apply that state consistently in fallback selection; and expose a redacted provider health snapshot in normal status while keeping deeper diagnostics in models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and some cooldown skipping exist, but the requested provider-level auth quarantine, network short quarantine, invalid-request fail-fast, and normal status provider health are absent. A live Anthropic OAuth outage is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining hard-auth signals, quarantine windows, invalid-request semantics, and status redaction need maintainer agreement. The safer path is a maintainer-approved policy first, followed by a narrow implementation across fallback, auth-profile health, status output, docs, and tests.
