## Summary

Keep open: current main and v2026.6.10 have failure-class foundations, profile cooldowns, an opt-in session skip cache, and auth details in models status, but they still do not provide the requested default hard-auth provider quarantine, invalid-request fail-fast policy, or provider-health block in normal status.

## What This Changes

Request default failure-class-aware provider fallback that quarantines hard auth failures, handles rate-limit/network/invalid-request classes differently, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, and invalid-request outcomes, applies it consistently in fallback, and exposes redacted status.

## Reproduction Assessment

Yes, at source level: current main still has default-off session skip behavior, profile-state-only provider skips, and no provider-health block in normal status; no live expired-token run was performed.

## Solution Assessment

No automated fix path is best yet: the direction is sound, but hard-auth classification, default quarantine, invalid-request fail-fast, and redacted status semantics need maintainer product judgment.
