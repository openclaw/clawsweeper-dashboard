## Summary

Current main now skips providers whose profiles are all persistently auth-broken and avoids repeatedly probing failed primaries through sticky fallback state, but meaningful requested behavior remains: ordinary auth failures still use short cooldowns, invalid-request failures can still advance fallback, and general `/status` lacks a provider-health/quarantine view. The open Codex incident and log-spam report cover narrower symptoms, so this issue remains the appropriate cross-provider product-policy thread.

## What This Changes

Add failure-class-aware provider fallback that quarantines auth-broken providers, handles transient failures differently, and exposes provider health to operators.

## Best Possible Solution

Extend the existing canonical auth-profile usage state and fallback decision path with an explicit, maintainer-approved failure policy and a redacted operator health projection; do not add a parallel provider quarantine cache or unconditional first-401 lockout.

## Reproduction Assessment

Yes at source level: current code and focused tests expose the exact profile-unusable and provider-skip decision path behind repeated auth-failure latency. This read-only review did not run a live provider outage, so it is not marked fully reproduced.

## Solution Assessment

No, not as the literal fixed table proposed. The maintainable solution is to build on canonical profile health after provider/Codex recovery is exhausted, with reason-specific scope and recovery semantics plus a redacted status projection.
