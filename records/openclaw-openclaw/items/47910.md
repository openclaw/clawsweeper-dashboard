## Summary

Keep open. Current main and the latest release have a failure-class foundation, profile cooldowns, and same-provider auth skip once profiles are already unavailable, but they still do not implement the requested provider-level auth-broken quarantine, network short quarantine, or normal `/status` provider-health surface; the hard-auth versus transient-refresh boundary remains a maintainer product decision.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, handles rate limits, network errors, and invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model with hard-auth signal boundaries, quarantine scope and duration, network handling, Retry-After treatment, and redacted status output before implementing the remaining behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/policy request. Source inspection gives high confidence that current main has partial failure-class foundations but not the requested provider quarantine and normal status health surface.

## Solution Assessment

Unclear until maintainers settle the hard-auth versus transient-refresh boundary, provider-versus-profile scope, quarantine durations, and redacted status contract. The requested direction is reasonable, but implementing it safely requires a product/security policy decision first.
