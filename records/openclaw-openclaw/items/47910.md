## Summary

Keep open. Current main has a failure-class-aware fallback foundation, but it still does not implement the full requested provider quarantine policy, network short quarantine, invalid-request fail-fast across model fallbacks, or provider quarantine/health in the normal status surface.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits and network errors differently, fails fast on invalid requests, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, and network failures, then apply it consistently in fallback selection and expose a redacted provider health snapshot in status.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature and policy request. Source inspection does give high-confidence evidence that current main only partially covers the requested behavior.

## Solution Assessment

Unclear. The requested direction is plausible, but the hard-auth classification boundary, invalid-request fail-fast policy, and redacted status contract need maintainer product/UX judgment before implementation.
