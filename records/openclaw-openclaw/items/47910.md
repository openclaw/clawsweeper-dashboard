## Summary

Keep open: current main and v2026.6.8 have useful failure-class handling plus an opt-in session skip cache, but the requested default provider quarantine policy and redacted provider-health status surface are still not implemented.

## What This Changes

Request provider fallback behavior that quarantines auth-broken providers by failure class and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model, apply hard-auth, rate-limit, network, and invalid-request policy consistently in fallback selection, and expose a redacted provider-health summary in status while keeping deeper diagnostics in models status or doctor.

## Reproduction Assessment

Yes for the source-level gap: current main shows default-off, session-scoped auth skip behavior and no normal status provider-health block. I did not run a live expired-token repro because that requires real provider credentials.

## Solution Assessment

No, not as an automated fix path. The direction is valid, but the hard-auth classification boundary, default quarantine behavior, and redacted status contract need maintainer product judgment first.
