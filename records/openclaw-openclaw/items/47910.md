## Summary

Keep open. Current main has a failure-class-aware fallback foundation and already skips providers whose auth profiles are all marked auth/auth_permanent-unavailable, but the full requested provider-wide auth quarantine, hard-auth policy contract, and normal status provider-health surface are still not implemented; the open related PR only covers a narrower session-scoped auth skip and re-login path.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits, network errors, and invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model, apply it consistently in fallback selection and auth-state persistence, decide the invalid-request fail-fast boundary, and expose a redacted provider-health snapshot in status while keeping detailed auth diagnostics in models status.

## Reproduction Assessment

Yes, at source level: current main shows failure classes, profile cooldowns, and auth/profile-state skips, while provider-wide hard-auth quarantine and normal status provider-health output remain absent. I did not replay a live Anthropic OAuth outage.

## Solution Assessment

Unclear until maintainers choose the provider-health policy. The existing foundations are useful, but the remaining solution needs an explicit contract for hard auth, rate-limit retry timing, network failure handling, invalid-request boundaries, and redacted status output.
