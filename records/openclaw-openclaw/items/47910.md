## Summary

Keep open. Current main has failure classification and cooldown-aware fallback foundations, but it still does not implement the requested provider-level auth quarantine, invalid-request fail-fast policy, network short quarantine, or provider health/quarantine output in normal status.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits/network/invalid requests differently, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing/quota, schema, and invalid-request failures; apply it consistently in fallback selection; and expose a redacted provider health snapshot in status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown behavior exist, but provider-level auth quarantine, network short quarantine, invalid-request fail-fast, and normal status provider health are absent. A live Anthropic OAuth outage is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining hard-auth signals, quarantine windows, invalid-request semantics, and status redaction contract need maintainer agreement first.
