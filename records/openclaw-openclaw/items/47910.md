## Summary

Keep open. Current main and v2026.5.18 include a failure-class-aware fallback foundation and profile-state auth skip behavior, but they still do not implement the requested provider-level auth-broken quarantine, network short quarantine, invalid-request fail-fast policy, or normal status provider-health surface; the remaining choices are provider/auth policy decisions rather than cleanup-close material.

## What This Changes

The issue requests failure-class-specific provider fallback, including auth-broken provider quarantine, Retry-After handling, network cooldown, invalid-request fail-fast, and provider health in status output.

## Best Possible Solution

Define and document a provider/profile health contract that separates hard auth, transient refresh, rate limit, network, and invalid-request outcomes; then implement fallback selection and redacted status output against that contract with upgrade-safe tests.
