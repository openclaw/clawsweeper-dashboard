## Summary

Keep open: current main and v2026.6.10 have useful failure-class foundations, an opt-in session skip cache, and models-status auth state, but they still do not implement the requested default hard-auth provider quarantine or provider-health block in normal status.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, handles rate limits, network failures, and invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model, apply hard-auth, rate-limit, network, and invalid-request policy consistently in fallback selection, and expose a redacted provider-health summary in status while keeping deeper diagnostics in models status or doctor.

## Reproduction Assessment

Yes, for the source-level gap: current main shows default-off, session-scoped auth skip behavior and no normal status provider-health block. I did not run a live expired-token repro because that requires real provider credentials.

## Solution Assessment

No, not as an automated repair path. The direction is valid, but the hard-auth classification boundary, default quarantine behavior, and redacted status contract need maintainer product judgment first.
