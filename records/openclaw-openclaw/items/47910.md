## Summary

Keep open: current main and the latest release have useful failure-class and auth-profile cooldown foundations, but the requested provider-wide auth quarantine, network short quarantine, invalid-request policy, and normal `/status` provider-health surface are still missing. The open related PR is adjacent remediation work, not a full implementation of this issue.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits, network failures, and invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health contract, apply it consistently in fallback selection, decide invalid-request and network handling, and expose a redacted provider-health snapshot in normal status while keeping deeper diagnostics in `models status`.

No strict bug reproduction is applicable because this is a feature and policy request. Source inspection gives high confidence that current main has partial failure-class support but not the requested provider quarantine and `/status` provider-health behavior.

## Solution Assessment

Unclear until maintainers settle the hard-auth versus transient-refresh boundary, quarantine scope/duration, network handling, invalid-request behavior, and redacted status contract. The requested direction is reasonable, but it is not a narrow mechanical bug fix.
