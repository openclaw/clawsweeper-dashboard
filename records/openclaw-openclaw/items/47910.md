## Summary

Keep open. Current main has the failure-class foundation, terminal format handling, and profile-state auth skipping, but it still lacks the requested provider-level auth quarantine, network short quarantine, and normal `/status` provider-health surface; the remaining work needs maintainer policy decisions, not cleanup closure.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits and network failures differently, fails fast on invalid requests, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health contract, apply it consistently in fallback selection, and expose a redacted provider-health snapshot in `/status` while keeping deeper auth diagnostics in `models status`.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature and policy request. Source inspection gives high confidence that current main has partial failure-class support but not the requested provider quarantine and `/status` provider-health behavior.

## Solution Assessment

Unclear until maintainers settle the hard-auth versus transient-refresh boundary, quarantine scope/duration, network handling, and redacted status contract. The requested direction is reasonable, but it is not a narrow mechanical bug fix.
