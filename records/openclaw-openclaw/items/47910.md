## Summary

Current main has failure classification and an opt-in session-scoped skip for recently auth-failing non-primary model candidates, but it does not satisfy the issue’s central provider/profile-wide quarantine and operator-visible health request. The remaining behavior overlaps with, but is broader than, the open Codex-specific incident at https://github.com/openclaw/openclaw/issues/86215 and the log-throttling report at https://github.com/openclaw/openclaw/issues/56979, so this issue should remain the cross-provider product-policy thread.

## What This Changes

Requests failure-class-aware provider fallback that quarantines auth-broken providers, treats transient and invalid-request failures differently, and exposes quarantine state to operators.

## Best Possible Solution

Make auth-profile usage state the canonical owner of credential health, propagate a closed exhausted-auth/transient classification into model fallback, define whether primary attempts may be skipped, and render that same state in status output without adding a parallel cache.

## Reproduction Assessment

Yes at source level: current main’s known-bad state is opt-in, session-scoped, restart-cleared, and keyed to a provider/model candidate, so it cannot provide the requested provider-wide fleet quarantine. A live latency reproduction was not run because it requires intentionally broken provider credentials.

## Solution Assessment

No, the proposed fixed provider-level TTL table is not yet the best maintainable shape. The safer direction is to distinguish exhausted credential recovery from transient refresh, store health at the affected profile or route boundary, and have fallback and status consume that canonical state.
