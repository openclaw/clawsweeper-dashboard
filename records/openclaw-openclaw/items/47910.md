## Summary

Keep open. Current main and the latest release have strong failure-class-aware fallback foundations, including persistent auth skip once all profiles are already marked unavailable and terminal handling for ordinary format failures, but the requested provider-wide hard-auth quarantine policy and normal status provider-health contract are still not fully implemented.

## What This Changes

The issue requests failure-class-aware model provider fallback that quarantines auth-broken providers, handles rate limits/network/invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Keep this as the canonical provider-health/quarantine policy discussion, then implement the accepted generic contract across failure classification, auth-state persistence, fallback selection, status output, docs, and regression tests.

## Reproduction Assessment

Yes, at source level. Current main shows profile-scoped cooldown/disabled handling and partial status observability, while provider-wide hard-auth quarantine and normal status provider-health output remain absent; I did not replay a live Anthropic OAuth outage.

## Solution Assessment

Unclear until maintainers choose the provider-health policy. The current profile-state foundation is useful, but the remaining solution needs an explicit contract for hard auth, Retry-After persistence, network failures, invalid request boundaries, and redacted status output.
