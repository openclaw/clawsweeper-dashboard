## Label Justifications

- `P2`: This is a normal-priority Telegram cron topic delivery safety fix with limited blast radius but meaningful merge-readiness blockers.
- `merge-risk: 🚨 compatibility`: The PR changes existing Telegram topic cron jobs from accepted delivery to fail-closed errors when provider proof is missing or the target form is not core-parseable.
- `merge-risk: 🚨 message-delivery`: The PR changes when OpenClaw marks Telegram cron delivery successful and can surface failures after one or more messages were sent.

## Summary

Keep open: the topic-safety goal is useful, but this PR is not merge-ready because it moves Telegram proof validation into cron core, breaks supported non-numeric Telegram targets, and lacks live Telegram proof.

## What This Changes

The PR adds Telegram provider delivery-proof metadata and makes cron Telegram topic delivery fail when provider chat/thread proof is missing or mismatched.

## Best Possible Solution

Keep the fail-closed topic-safety goal, but move proof validation to a Telegram-owned send/result contract or a generic channel proof hook that uses resolved chat/thread identity, then require redacted live Telegram proof before merge.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection and the PR's own head test show the new checker rejects `@openclaw_ops` with `threadId`, while current docs and plugin parsing support non-numeric Telegram targets.

## Solution Assessment

No. Fail-closed provider proof is directionally right, but this is not the best fix because cron core re-parses Telegram target grammar instead of using Telegram-owned resolved delivery identity.
