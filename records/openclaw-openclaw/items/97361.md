## Label Justifications

- `P2`: This is a normal-priority Telegram cron topic delivery safety fix with limited blast radius but meaningful merge-readiness blockers.
- `merge-risk: 🚨 compatibility`: The PR changes existing Telegram topic cron jobs from accepted delivery to fail-closed errors when provider proof is missing or the target form is not core-parseable.
- `merge-risk: 🚨 message-delivery`: The PR changes the conditions under which OpenClaw marks a Telegram cron delivery as successful and can surface failures after one or more messages were sent.

## Summary

Keep open: the PR is aimed at a real Telegram cron topic safety problem, but it is not merge-ready because the proof check is implemented in cron core with Telegram-specific parsing that can reject supported targets, and the contributor has not provided live Telegram proof.

## What This Changes

The branch adds Telegram provider delivery-proof metadata to send results, aggregates multi-send proof in the Telegram outbound adapter, and makes cron Telegram non-General topic delivery fail when provider topic proof is missing or mismatched.

## Best Possible Solution

Keep the fail-closed topic-safety goal, but move proof validation to a Telegram-owned send/result contract or a generic channel proof hook that uses resolved chat/thread identity, then require redacted live Telegram proof before merge.

## Reproduction Assessment

No live high-confidence reproduction is present. Source inspection shows current main lacks provider topic-proof validation, and the PR adds mocked tests for missing and mismatched proof cases.

## Solution Assessment

No. The fail-closed behavior is directionally right, but this implementation is not the best fix because cron core re-parses Telegram target grammar instead of using Telegram-owned resolved delivery identity.
