## Summary

Keep this PR open: the topic-proof safety goal is useful, but the patch has a blocking owner-boundary compatibility defect and still lacks contributor-provided live Telegram proof before merge.

## What This Changes

The PR surfaces provider-confirmed Telegram send proof metadata, aggregates multi-send proof, and makes cron Telegram non-General topic delivery fail when provider chat/thread proof is missing or mismatched.

## Best Possible Solution

Keep provider-confirmed topic proof, but validate it at the Telegram-owned routing/send boundary or a generic channel proof hook that receives resolved chat and thread identity, then require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes for the patch defect: source inspection shows a current cron delivery with `to: "@mybot"` and `threadId: 42` is accepted/stored today, but the PR's cron-core parser cannot derive an expected chat id and throws after a send result returns. No live current-main Telegram reproduction was run for the original safety bug.

## Solution Assessment

No. The fail-closed safety goal is reasonable, but this is not the best fix because cron core re-parses Telegram target grammar instead of using the Telegram plugin's resolved chat/thread identity.
