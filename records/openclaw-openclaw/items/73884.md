## Summary

Keep this PR open: current main still allows and clamps Telegram polling stall thresholds at 30000 ms, so the focused 60000 ms floor remains useful, but the external PR is not merge-ready because it still lacks real Telegram proof and its added regression test does not actually advance the watchdog clock.

## What This Changes

The PR raises Telegram `pollingStallThresholdMs` lower-bound handling from 30000 ms to 60000 ms across runtime clamping, config validation, generated metadata, docs/UI hints, and focused tests.

## Best Possible Solution

Refresh and land the 60000 ms lower bound across runtime, schema, metadata, docs/UI hints, and tests, with the watchdog-clock assertion fixed and redacted live Telegram/log proof added.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and clamps to 30000 ms while grammY long polling defaults to 30 seconds and the watchdog restarts when elapsed `getUpdates` liveness exceeds the configured threshold.

## Solution Assessment

Yes, with refresh and proof. Raising the existing lower bound to 60000 ms across runtime and public config surfaces is the narrow maintainable fix; the regression test should explicitly advance the watchdog clock.
