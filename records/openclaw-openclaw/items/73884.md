## Summary

Keep open: current main still accepts and clamps a 30000 ms Telegram polling-stall threshold even though the dependency long-poll window is 30 seconds and OpenClaw now guards getUpdates at 45 seconds. The PR is directionally narrow, but it needs a conflict refresh, contributor-owned real Telegram/log proof, and a small test fix before merge.

## What This Changes

The PR raises Telegram pollingStallThresholdMs from 30000 to 60000 ms across runtime clamping, config validation, generated metadata, docs, UI hints, and focused tests.

## Best Possible Solution

Refresh and land the 60000 ms lower bound across runtime, schema, metadata, docs, UI hints, and tests, with an effective regression test and redacted live Telegram/log proof.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and clamps 30000 ms, the watchdog restarts when elapsed time exceeds that threshold, and the pinned grammY runner uses a 30-second getUpdates long-poll timeout while OpenClaw guards getUpdates at 45 seconds.

## Solution Assessment

Yes, with refresh and proof. Raising the existing lower bound to 60000 ms across runtime and public config surfaces is the narrow maintainable fix, but the added regression test should actually advance time and the external PR needs live proof.
