## Summary

Keep open: current main still accepts and clamps a 30000 ms Telegram polling stall threshold, so the PR’s core bug fix remains useful, but it needs a conflict refresh, contributor real behavior proof, and one low-priority regression-test correction before merge.

## What This Changes

The PR raises Telegram pollingStallThresholdMs from 30000 to 60000 ms across runtime clamping, config validation, generated metadata, docs/UI hints, and focused tests.

## Best Possible Solution

Refresh and land the 60000 ms lower bound across runtime, schema, metadata, docs, UI hints, and tests while preserving current getUpdates liveness behavior and adding redacted live Telegram/log proof.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts 30000 ms, clamps runtime thresholds to that value, and the watchdog restarts when getUpdates liveness exceeds the threshold while grammY’s long-poll default is 30 seconds and OpenClaw’s getUpdates request guard is 45 seconds.

## Solution Assessment

Yes, with refresh and proof. Raising the existing lower bound to 60000 ms across runtime and public config surfaces is the narrow maintainable fix; the safer merge path is to repair the regression test and preserve the newer liveness refactors.
