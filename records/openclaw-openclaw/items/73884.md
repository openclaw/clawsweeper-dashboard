## Summary

Keep open: this PR addresses a still-present, source-proven Telegram polling configuration/runtime bug. It is not superseded by the merged getUpdates-liveness fix, but it needs a conflict refresh, redacted live Telegram proof, and one regression-test adjustment before merge.

## What This Changes

The PR raises the Telegram polling stall threshold lower bound from 30000 to 60000 ms across runtime clamping, config validation, generated metadata, docs, UI hints, and focused tests.

## Best Possible Solution

Refresh and land the 60000 ms lower bound across runtime, schema, metadata, docs/UI hints, and tests, preserving merged getUpdates-liveness changes and adding real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main accepts and clamps to 30000 ms while grammY long polling is 30 seconds and OpenClaw's getUpdates request guard is 45 seconds, so a low configured threshold can cross the watchdog window during normal long-poll jitter. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

Yes, with refresh and proof: raising the existing lower bound to 60000 ms across runtime and public config surfaces is the narrow maintainable fix. The regression test should explicitly advance the watchdog clock before invoking it.
