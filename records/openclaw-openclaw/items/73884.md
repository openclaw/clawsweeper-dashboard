## Summary

Keep open: current main still permits the unsafe 30000 ms Telegram polling stall threshold, and this PR is a plausible narrow fix, but it needs a conflict refresh, redacted real Telegram proof, and a small regression-test repair before merge.

## What This Changes

The PR raises the Telegram pollingStallThresholdMs lower bound from 30000 to 60000 ms across runtime clamping, validation, generated metadata, docs/UI hints, and focused tests.

## Best Possible Solution

Refresh and land the 60000 ms lower bound across runtime, schema, metadata, docs/UI hints, and tests, with the regression test clock fixed and redacted live Telegram/log proof added.

## Reproduction Assessment

Yes, source-reproducible. Current main accepts and clamps to 30000 ms while grammY long polling is 30 seconds and OpenClaw's getUpdates request guard is 45 seconds, so normal long-poll jitter can cross the watchdog threshold.

## Solution Assessment

Yes, with refresh and proof. Raising the existing lower bound to 60000 ms across runtime and public config surfaces is the narrow maintainable fix; the regression test should advance the watchdog clock explicitly.
