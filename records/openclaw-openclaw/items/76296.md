Full review comments:

- **[P1] Move workflow inputs out of the shell script:** `.github/workflows/android-apk-build.yml:127-128`
  - body: The `security-scm-fast` run already fails `zizmor` high-confidence template-injection on these direct `${{ github.event.inputs.* }}` expansions. Put the values in `env`, validate the closed set, and read shell variables so a dispatch input cannot become shell syntax.
  - confidence: 0.94
- **[P1] Gate canonical APK artifact publishing:** `.github/workflows/android-apk-build.yml:22-30`
  - body: The docs frame this as fork setup, but the workflow also runs on pushes to `main` and uploads a default thirdParty debug APK artifact. Remove the push trigger, gate it to forks, or make this an explicitly approved official APK distribution lane with provenance and release policy.
  - confidence: 0.9
- **[P1] Keep fork upkeep automation out of the canonical repo:** `.github/workflows/upstream-watch.yml:5-10`
  - body: This scheduled workflow grants `issues: write` for fork-behind notifications but has no repository guard, so merging it installs daily issue-writing automation in `openclaw/openclaw`. Add a job-level fork-only condition or keep this workflow out of core.
  - confidence: 0.87
- **[P2] Install the SDK platform the Android app compiles against:** `.github/workflows/android-apk-build.yml:73`
  - body: Current main sets `compileSdk = 37` and the existing Android CI installs `platforms;android-37.0`, but this new workflow installs only `platforms;android-36`. After merging with current main, the APK workflow can fail before producing artifacts; update it to the current SDK platform.
  - confidence: 0.86
