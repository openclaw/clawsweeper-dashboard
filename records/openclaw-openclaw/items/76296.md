## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks official APK artifact publishing and the work relates to the open #9443 request, but the patch has blocking workflow/security issues, mixes fork-maintenance automation into the canonical repo, and has only partial Android build proof.

## What This Changes

The PR adds a manual Android APK build workflow, a scheduled upstream fork-watch workflow, and fork-oriented APK build documentation.

## Best Possible Solution

Split fork-update automation out, keep #9443 as the canonical Android APK release tracker, and land only a maintainer-approved artifact workflow that scopes signing secrets tightly, validates play/third-party debug and release paths, and updates changelog/docs.

## Reproduction Assessment

Yes for the blocking workflow bug: the upstream-watch here-doc terminator is indented, and a matching `bash -n -c` probe fails before execution. APK release behavior is only partially reproducible from the PR because the reported verification covers third-party debug only.

## Solution Assessment

No. The maintainable path is a scoped Android release workflow tied to the existing release policy discussion, not a mixed fork helper plus signing/artifact workflow with job-wide secrets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope signing secrets to the release step:** `.github/workflows/android-apk-build.yml:45-51`
  - body: These signing secrets are defined in the job-level environment, which makes them available to every step in the job, including third-party setup/cache actions and debug builds. Move the secret env values onto the release-signing step only so APK signing material is not exposed to unrelated actions.
  - confidence: 0.91
- **[P2] Close the heredoc before updating the issue:** `.github/workflows/upstream-watch.yml:79`
  - body: The closing `EOF` for this `cat <<EOF` block is indented, so Bash does not recognize it as the here-doc terminator. When the fork is behind, the step fails parsing instead of creating or updating the issue; put `EOF` at column 1 or use a deliberate tab-stripping heredoc.
  - confidence: 0.96

  - body: This adds a user-facing Android APK build path and documentation, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing feature changes to be recorded before merge.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff introduces Android signing-secret handling, downloadable APK artifacts, new third-party workflow actions, and scheduled issue-writing automation.

Concerns:

- **[high] Do not expose signing secrets to every APK job step:** `.github/workflows/android-apk-build.yml:45`
  - body: The APK workflow sets signing secrets at job scope, so all steps and actions in the job can receive them when release secrets are available. Signing secrets should be scoped to the release-signing step only, and preferably only after release policy approval.
  - confidence: 0.91
- **[medium] Avoid adding fork issue automation to the main repo:** `.github/workflows/upstream-watch.yml:8`
  - body: The upstream-watch workflow grants `issues: write` and runs daily to manage fork update issues. That mutating automation is unrelated to Android APK artifacts and should not be added to the canonical repo without explicit maintainer approval.
  - confidence: 0.84
- **[medium] Gate APK artifact distribution on release policy:** `.github/workflows/android-apk-build.yml:170`
  - body: The workflow uploads downloadable APK artifacts while the official APK release request remains open for decisions on signing, flavor support, distribution channel, and release ownership.
  - confidence: 0.82
