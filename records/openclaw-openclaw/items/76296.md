## Summary

Keep open. Current main does not already provide this APK artifact workflow, but this PR should not merge as-is because the new workflow exposes Android signing secrets job-wide and the new fork-watch workflow adds scheduled issue-writing automation to the upstream repository; the PR head also has a failing workflow security audit.

## What This Changes

The PR adds Android APK build and fork upstream-watch GitHub workflows plus a short Android APK build guide.

## Best Possible Solution

Keep APK artifact work aligned with the existing Android CI/release path, with release signing secrets scoped only to the signing step or a protected environment and fork-maintenance automation kept out of the canonical upstream workflow set unless explicitly approved.

## Reproduction Assessment

Not applicable as a user bug; this is an infrastructure PR. The high-confidence review path is static workflow inspection plus current-main comparison, and the PR head already shows a failing workflow security audit.

## Solution Assessment

No. A GitHub APK workflow may be a reasonable direction, but this implementation is not the safest maintainable version because it broadens signing-secret exposure and adds fork-update issue automation to the upstream repo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restrict release signing secrets:** `.github/workflows/android-apk-build.yml:48-51`
  - body: The workflow exports the Android signing passwords and key alias at job scope, so every step in the job, including third-party setup/cache actions and debug builds, receives them. Move these secrets to the release signing step only, or gate release signing behind a protected environment before this can safely run in the upstream repository.
  - confidence: 0.92
- **[P2] Guard the fork watcher on canonical runs:** `.github/workflows/upstream-watch.yml:5-10`
  - body: This fork-maintenance workflow would be installed in `openclaw/openclaw` with a daily schedule and `issues: write`. Add an explicit repository guard or leave this workflow out of the upstream repo so the canonical repository cannot open/comment fork-behind issues from its own scheduled runs.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive workflow changes that should not merge until signing-secret scope and issue-writing automation are tightened.

Concerns:

- **[high] Android signing secrets are exposed to every job step:** `.github/workflows/android-apk-build.yml:48`
  - body: Job-level `OPENCLAW_ANDROID_*` secret environment variables are inherited by all steps, including third-party actions that do not need signing material. This unnecessarily broadens exposure of release keystore credentials.
  - confidence: 0.92
- **[medium] Fork watcher adds scheduled issue-writing automation:** `.github/workflows/upstream-watch.yml:5`
  - body: The new upstream-watch workflow grants `issues: write` on a daily schedule for fork maintenance. In the canonical repo this is an unnecessary write-capable automation surface unless maintainers explicitly want it and guard it.
  - confidence: 0.8
