## Summary

Keep open: this targets a valid linked heartbeat privacy bug and adds useful focused test coverage, but the new guard can fail existing empty heartbeat turns by reading payload metadata before proving a payload object exists. The external PR also still needs after-fix real behavior proof before merge.

## What This Changes

The PR imports reply-payload metadata, treats heartbeat tool-mode raw text without a `heartbeat_respond` payload as empty, and adds a Telegram heartbeat regression test for suppressing raw text.

## Best Possible Solution

Land one canonical fix for https://github.com/openclaw/openclaw/issues/94053 that suppresses raw non-tool heartbeat replies, preserves marked system/error notices, keeps empty replies on the existing quiet path, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source-reproducible: current main sets `sourceReplyDeliveryMode: "message_tool_only"` for heartbeat response-tool mode, then raw payloads without `heartbeat_respond` fall through to normalization and channel delivery. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not as submitted. Suppressing raw non-tool heartbeat replies in the heartbeat runner is the right boundary, but the guard must not call the metadata helper on `undefined` and should cover the no-payload path plus real behavior proof.
