## Summary

Keep open. Current main still drops the originating channel route before `plugin.approval.request`, while the gateway already requires those fields to avoid the no-route fast path; the linked repair PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to pass originating channel route metadata into `plugin.approval.request` for `before_tool_call` plugin approvals.

## Best Possible Solution

Carry trusted turn-source fields from `createOpenClawCodingTools` into the hook context and forward them to `plugin.approval.request`, with regression coverage for ordinary and trusted-policy approval paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: channel metadata enters the tool factory, is not included in hook context, the plugin approval request sends no route fields, and the gateway returns `decision: null` when no other approval route exists.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate trusted runtime route context through the hook approval path, not to read route fields from model-generated tool params.

## Repair Work Prompt

Fix the plugin approval routing bug tracked by https://github.com/openclaw/openclaw/issues/74003. In current main, `createOpenClawCodingTools` receives the originating channel/account/target/thread values, but `wrapToolWithBeforeToolCallHook` only gets agent/session/run/trace/loop context, and `requestPluginToolApproval` calls `plugin.approval.request` without `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, or `turnSourceThreadId`. Add a narrow trusted-context path for those route fields, populate it from the existing `createOpenClawCodingTools` options, and forward it to `plugin.approval.request` for both trusted-policy and ordinary `before_tool_call` `requireApproval` paths. Do not read route fields from model-generated tool params. Add regression coverage proving route fields are forwarded when present and omitted cleanly when absent, including the trusted-policy path. Add a user-facing `CHANGELOG.md` fix entry. Keep the change inside the agent/gateway approval contract; do not change Telegram duplicate-delivery fallback behavior from #75749 in this PR.
