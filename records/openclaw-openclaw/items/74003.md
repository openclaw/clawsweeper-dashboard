## Summary

Keep this open. Current main still drops the trusted turn-source route before before_tool_call plugin approvals call plugin.approval.request, so the gateway can still take the no-route path; the linked repair PR #74150 is closed and unmerged, while #75749 is a related but distinct duplicate-delivery follow-up.

## What This Changes

Forward the originating channel/account/chat/thread route from before_tool_call plugin approvals into plugin.approval.request so the gateway can route the approval prompt.

## Best Possible Solution

Carry trusted runtime route fields from createOpenClawCodingTools into HookContext and forward them as turnSource* fields for both trusted-policy and ordinary before_tool_call approval requests, while preserving fail-closed behavior when no route exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated run supplies route data to createOpenClawCodingTools, the wrapper drops it, plugin.approval.request is called without turnSource* fields, and approval-shared returns decision:null when no other approval route is available.

## Solution Assessment

Yes, with the reporter's corrected trust boundary. The maintainable fix is to pass trusted runtime route context through the hook wrapper, not to read model-generated tool params for routing metadata.

## Repair Work Prompt

Fix this issue in one focused PR. Current main still drops trusted route metadata before before_tool_call plugin approvals call plugin.approval.request, causing no-route failures from channel-originated runs.

Expected boundary:
- Add trusted route fields to the before_tool_call hook wrapper context: turnSourceChannel, turnSourceTo, turnSourceAccountId, and turnSourceThreadId.
- Populate them from createOpenClawCodingTools options using messageProvider, messageTo, agentAccountId, and messageThreadId.
- Forward those fields from requestPluginToolApproval to plugin.approval.request for both trusted-policy requireApproval and ordinary hook requireApproval flows.
- Do not read route metadata from model-generated tool params or hook-returned params.
- Preserve existing fail-closed behavior when no route/client/forwarder exists.
- Add a user-facing CHANGELOG.md fix entry unless maintainers decide this is internal-only.
