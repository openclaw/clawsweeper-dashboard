## Summary

Keep open. Current main still builds before_tool_call plugin approval requests without trusted turn-source routing metadata, so the gateway can still return the reported no-route null decision; the only linked repair PR is closed and unmerged, and the reporter's correction about not trusting model tool params remains important.

## What This Changes

Forward trusted originating-channel routing metadata into before_tool_call plugin approval requests so approval prompts can route back to the source chat.

## Best Possible Solution

Carry forward the useful intent from #74150, but implement it through trusted runtime context: extend HookContext and createOpenClawCodingTools to carry the same source-route data used by exec approvals, forward it into plugin.approval.request, add regression coverage for routed and unrouted plugin approvals, and preserve the gateway's fail-closed no-route behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: before_tool_call requireApproval reaches requestPluginToolApproval without turnSource* fields, then the gateway no-route branch returns decision:null and the agent maps that to the reported error string.

## Solution Assessment

Yes, with the corrected trust boundary. The original args.params/baseParams direction is not the best fix because those params are model/tool input; the maintainable path is trusted runtime route context propagated into hook ctx and forwarded from there.
