## Summary

Keep this open. Current main still drops trusted turn-source routing metadata before before_tool_call plugin approvals call plugin.approval.request, while the gateway already supports those fields and intentionally returns decision:null when no approval route is available. The reporter's correction is material: the fix should use trusted runtime context, not model/tool parameters; the linked repair PR is closed and unmerged.

## What This Changes

The issue requests that before_tool_call plugin approval requests include trusted originating-channel route metadata so plugin.approval.request can route prompts back to the source chat.

## Best Possible Solution

Extend the internal hook/tool wrapper context with trusted route metadata from the agent runtime, forward those fields into plugin.approval.request for both trusted-policy and ordinary before_tool_call approval paths, and add regression coverage proving routed requests are accepted while missing routes still fail closed.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: a channel-originated run passes route data into createOpenClawCodingTools, the wrapper drops it, requestPluginToolApproval sends no turnSource* fields, and the gateway returns decision:null when no other approval route exists.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to propagate trusted runtime route context through HookContext into plugin.approval.request; reading route fields from tool params or hook-returned params would be less safe because those values are model/plugin controlled.
