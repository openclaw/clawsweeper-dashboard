## Summary

Keep this open. Current main still drops trusted turn-source routing metadata before before_tool_call plugin approvals call plugin.approval.request, while the gateway already depends on those fields to avoid no-route expiry; the linked repair PR #74150 is closed unmerged.

## What This Changes

The issue reports that before_tool_call plugin approvals lose originating-channel routing metadata before calling plugin.approval.request, causing channel approval prompts to fail with no approval route.

## Best Possible Solution

Carry trusted runtime turn-source fields through HookContext from createOpenClawCodingTools into plugin.approval.request for both trusted-policy and ordinary hook approval paths, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: route context is available when tools are created, the hook context drops it, requestPluginToolApproval omits turnSource* fields, and the gateway returns decision:null when no other approval route is present.

## Solution Assessment

Yes, with the corrected source of truth. The narrow maintainable fix is to propagate trusted runtime route context through the hook context and approval request, not to read route fields from model-generated params.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74003. Current main loses trusted turn-source routing metadata before before_tool_call plugin approvals call plugin.approval.request, so channel-originated approvals can fail with no approval route. Implement the fix in the agent/hook boundary: carry trusted runtime source fields from createOpenClawCodingTools options into HookContext, then forward them from requestPluginToolApproval to plugin.approval.request for both trusted-policy requireApproval and ordinary before_tool_call requireApproval paths. Use the runtime context as the source of truth: turnSourceChannel should come from the message provider/channel value, turnSourceTo from the originating destination, turnSourceAccountId from the agent/account id, and turnSourceThreadId from the originating thread id where available. Do not read these values from args.params or hook-modified/model-generated tool params. Add focused regression coverage proving plugin.approval.request receives the fields when tools are wrapped with route context and omits them when unavailable. Preserve existing fail-closed approval behavior and do not change gateway routing policy. Add a user-facing CHANGELOG.md fix entry; mention plugin approvals, not extensions.
