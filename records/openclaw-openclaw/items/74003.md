## Summary

Keep this open. Current main still drops trusted turn-source routing metadata before `before_tool_call` plugin approvals call `plugin.approval.request`, so the gateway can still return `decision: null` and the agent maps that to the reported `Plugin approval unavailable (no approval route)` failure. The reporter's correction is important: the fix should use trusted runtime context, not model/tool params, and the linked repair PR #74150 is closed and unmerged.

## What This Changes

The issue requests forwarding channel-origin route metadata into `before_tool_call` plugin approval requests so the gateway can deliver approval prompts back to the originating chat.

## Best Possible Solution

Extend the internal hook/tool wrapper context with trusted `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId` from the agent runtime, forward those fields into `plugin.approval.request` for both trusted-policy and ordinary `before_tool_call` approval paths, and add regression coverage proving routed requests are accepted while missing routes still fail closed.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: a channel-originated run passes route data into `createOpenClawCodingTools`, the wrapper drops it, `requestPluginToolApproval` sends no `turnSource*` fields, and the gateway returns `decision: null` when no other approval route exists.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to propagate trusted runtime route context through `HookContext` into `plugin.approval.request`; reading route fields from tool params or hook-returned params would be less safe because those values are model/plugin controlled.
