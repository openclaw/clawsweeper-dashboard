## Summary

Keep this open. Current main still accepts routing metadata at tool creation time but does not carry it into the before_tool_call approval request, while the gateway already requires a turn-source route to avoid the reported `decision: null` no-route path.

## What This Changes

The issue requests forwarding originating channel route metadata into `plugin.approval.request` for before_tool_call plugin approvals.

## Best Possible Solution

Carry trusted originating route fields from `createOpenClawCodingTools` into the hook context and then into `plugin.approval.request`, with regression coverage for ordinary hook and trusted-policy approval paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: route inputs enter tool creation, the hook wrapper drops them, the plugin approval request sends no turn-source fields, and the gateway returns `decision: null` when no other route is available.

## Solution Assessment

Yes. The corrected direction is to read trusted runtime context, not model-generated tool params, and forward only those route fields through the existing plugin approval gateway contract.

## Repair Work Prompt

Fix the before_tool_call plugin approval no-route bug tracked by https://github.com/openclaw/openclaw/issues/74003. Current main accepts trusted route inputs in `createOpenClawCodingTools` but drops them before `plugin.approval.request`. Add route fields to the hook context from trusted runtime options (`messageProvider`, `messageTo`, `agentAccountId`, `messageThreadId`) and forward them to `plugin.approval.request` as `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId`. Do not read these fields from model-generated tool params. Cover both ordinary `before_tool_call` `requireApproval` and trusted-policy approval paths, and add a create-tools regression proving route inputs reach the wrapper/request. Keep the fix in agent approval plumbing and existing gateway contracts; do not change Telegram-specific delivery behavior or attempt to solve #75749 in this PR. Add a single-line user-facing changelog fix entry unless maintainers decide it is covered by another pending changelog entry.
