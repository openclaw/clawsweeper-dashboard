## Summary

Keep this open. Current main still drops the trusted turn-source routing context before before_tool_call plugin approvals are requested, while the gateway already supports those route fields and intentionally returns a null decision on unroutable approvals. The reporter's correction is important: the fix should use trusted runtime context, not model/plugin-controlled tool params; the related repair PR is closed and unmerged.

## What This Changes

The issue requests that before_tool_call plugin approval requests carry trusted originating-channel route metadata so plugin.approval.request can prompt the user in the source chat.

## Best Possible Solution

Extend the internal hook/tool wrapper context with trusted route metadata from the agent runtime, forward those fields into plugin.approval.request for both trusted-policy and ordinary before_tool_call approval paths, and add regression coverage proving routed requests are accepted while missing routes still fail closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated run passes message route data into createOpenClawCodingTools, the wrapper drops it, requestPluginToolApproval sends no turnSource* fields, and the gateway returns decision:null when no other approval route exists.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate trusted runtime route context through HookContext into plugin.approval.request; reading route fields from tool params or hook-returned params would cross the trust boundary because those values are model/plugin controlled.
