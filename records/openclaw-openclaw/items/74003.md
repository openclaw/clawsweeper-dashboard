## Summary

Keep this open. Current main still drops trusted turn-source routing metadata before `before_tool_call` plugin approvals call `plugin.approval.request`, while the gateway already depends on those fields to avoid the reported no-route failure and the linked repair PR #74150 is closed unmerged.

## What This Changes

Forward trusted turn-source routing metadata from channel-originated tool runs into `before_tool_call` plugin approval requests so prompts route back to the originating chat.

## Best Possible Solution

Carry the originating channel route fields through the agent hook context into `plugin.approval.request`, with regression coverage for both trusted-policy and ordinary hook approval paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: channel route options enter `createOpenClawCodingTools`, the hook wrapper drops them, the plugin approval request sends no turn-source fields, and the gateway returns `decision: null` when no other approval route is available.

## Solution Assessment

Yes, with the reporter’s corrected root cause. The narrow maintainable fix is to use trusted runtime context from `createOpenClawCodingTools`, not `args.params` or hook-adjusted params.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74003. Current main drops trusted turn-source routing metadata before `before_tool_call` plugin approvals call `plugin.approval.request`, causing `Plugin approval unavailable (no approval route)` for channel-originated plugin approvals. Do not read route values from model-generated tool params, `args.params`, hook event params, or plugin-modified params. Extend `HookContext` in `src/agents/pi-tools.before-tool-call.ts` to carry trusted `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId`; populate those fields in `src/agents/pi-tools.ts` from `createOpenClawCodingTools` options (`messageProvider`, `messageTo`, `agentAccountId`, `messageThreadId`); and include them in the `plugin.approval.request` payload for both trusted-policy `requireApproval` and ordinary hook `requireApproval` paths. Add focused regression tests proving the fields are forwarded and the existing no-route behavior remains unchanged when they are absent. Add a single-line user-facing `CHANGELOG.md` fix entry. Treat closed unmerged PR #74150 as useful context but prefer the corrected runtime-context trust boundary from the reporter comment. Do not change gateway delivery/fallback policy except for tests needed to prove the existing contract.
