## Summary

Keep open. Current `main` still has the reported routing gap: chat route inputs reach tool construction, but the `before_tool_call` hook context and `plugin.approval.request` payload do not carry `turnSource*`, so the gateway can still return `decision: null` for no approval route.

## What This Changes

The issue asks OpenClaw to forward trusted originating-channel route metadata from `before_tool_call` plugin approvals into `plugin.approval.request`.

## Best Possible Solution

Thread trusted originating route metadata through the agent hook context and into `plugin.approval.request`, with coverage for routed chat approvals and route-less fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: route options enter `createOpenClawCodingTools`, the hook context drops them, `requestPluginToolApproval` sends no route fields, and the gateway returns `decision: null` when no other approval route exists.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate trusted runtime route metadata through `HookContext` or an equivalent trusted structure, not to read `turnSource*` from model-controlled tool params.
