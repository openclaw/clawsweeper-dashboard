## Summary

Keep this open. Current main still drops trusted turn-source routing metadata before `before_tool_call` plugin approvals call `plugin.approval.request`, while the gateway route/fail-closed logic already depends on those fields and the linked repair PR #74150 is closed unmerged.

## What This Changes

Request forwarding originating channel, chat, account, and thread routing metadata into plugin approval requests created by `before_tool_call` `requireApproval`.

## Best Possible Solution

Carry trusted runtime turn-source fields through `HookContext` from `createOpenClawCodingTools` into `plugin.approval.request` for both trusted-policy and ordinary hook approval paths, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated run supplies route context to tool creation, the hook wrapper drops it, `requestPluginToolApproval` sends no turn-source fields, and the gateway returns `decision: null` when no other route is available.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to propagate trusted runtime route context through the hook context and approval request, not to read route fields from model params or hook-modified params.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74003. Current main drops trusted channel/chat/account/thread routing metadata before `before_tool_call` `requireApproval` calls `plugin.approval.request`, so channel-originated plugin approvals can fail with `Plugin approval unavailable (no approval route)`. Implement the fix in the agent owner path only: add trusted turn-source fields to `HookContext`, populate them in `createOpenClawCodingTools` from runtime inputs (`messageProvider`, `agentAccountId`, and the appropriate target/thread fields such as `messageTo`/`currentChannelId` and `messageThreadId`/`currentThreadTs`), and forward them from `requestPluginToolApproval` to `plugin.approval.request` for both trusted-tool-policy and ordinary `before_tool_call` approval paths. Do not read route fields from model-generated tool params, hook-returned params, or plugin-controlled data. Add focused regressions proving the wrapped tools forward route context and the approval RPC receives the four `turnSource*` fields when present while omitting them when absent. Add a single-line user-facing `CHANGELOG.md` fix entry; do not change unrelated approval delivery/fallback behavior from #75749 in this PR.
