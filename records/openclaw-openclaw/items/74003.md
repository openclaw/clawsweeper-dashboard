## Summary

Keep open. Current main still has the reported routing bug: turn-source metadata is available before tool wrapping, but the `before_tool_call` approval path does not carry it into `plugin.approval.request`, and the gateway still returns `decision: null` when no approval route exists. The linked repair PR #74150 is closed unmerged, and #75749 is adjacent fallback behavior rather than a replacement tracker.

## What This Changes

The issue asks OpenClaw to forward originating channel route metadata from `before_tool_call` plugin approval flows to `plugin.approval.request` so chat-channel approvals can be delivered.

## Best Possible Solution

Carry the originating channel route fields through the agent hook context into `plugin.approval.request`, with regression coverage for both trusted-policy and ordinary `before_tool_call` approval paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: channel metadata enters `createOpenClawCodingTools`, the hook wrapper drops it, the plugin approval request sends no route fields, and the gateway fails closed with `decision: null` when no other route exists.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate trusted runtime context from `createOpenClawCodingTools` into the hook approval request, not to read route fields from model-controlled tool params.
