## Summary

Current main still matches the reported bug. The agent-side before_tool_call approval path drops trusted turn-source routing metadata before calling the gateway, while the gateway and docs already support routed plugin approvals; the linked repair PR is closed and unmerged, and the reporterâ€™s corrected trust-boundary note is material.

## What This Changes

Forward trusted originating-channel routing metadata into `plugin.approval.request` for `before_tool_call` plugin approvals so channel-originated approvals can be routed back to the requester.

## Best Possible Solution

Extend the internal hook wrapper context with trusted route metadata from the agent runtime, forward those fields into `plugin.approval.request` for trusted-policy and ordinary before_tool_call approvals, and add regression coverage for routed acceptance plus fail-closed no-route behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: channel-originated runs pass route data into `createOpenClawCodingTools`, the hook wrapper drops it, `requestPluginToolApproval` sends no turn-source fields, and the gateway immediately returns `decision: null` when no other approval route is available.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to propagate trusted runtime context through `HookContext`; reading `args.params`, `baseParams`, or hook-returned params would be the wrong solution because those values are model/plugin controlled.

## Repair Work Prompt

Fix openclaw/openclaw#74003. Current main drops trusted originating-channel metadata before before_tool_call plugin approvals call `plugin.approval.request`, causing the gateway to return `decision: null` / "Plugin approval unavailable (no approval route)" when no other approval client is available. Implement a narrow fix by adding trusted route fields to the internal hook context passed from `createOpenClawCodingTools` into `wrapToolWithBeforeToolCallHook`, sourcing them from runtime options such as `messageProvider`, `messageTo`, `agentAccountId`, and `messageThreadId` or the existing channel/thread equivalents used for exec approval routing. Forward `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId` into `plugin.approval.request` for both trusted-policy and ordinary before_tool_call approval paths. Do not read these values from model-generated tool params, `baseParams`, or hook-returned params, and do not relax fail-closed no-route behavior. Use #74150 only as context; its trust-boundary assumption was corrected by the reporter. Add regression tests proving routed plugin approval requests include the trusted fields and stay accepted, while missing route metadata still fails closed. Add a user-facing CHANGELOG.md fix entry with non-bot credit if policy requires it.
