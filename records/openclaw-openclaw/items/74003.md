## Summary

Keep open. Current main still drops trusted turn-source routing metadata in the before_tool_call plugin approval path, while the gateway already supports routed plugin approvals and returns the reported no-route result when no route is present; the linked repair PR was closed unmerged after the params/baseParams approach was identified as unsafe.

## What This Changes

Route before_tool_call plugin approval prompts back to the originating channel by passing trusted turn-source metadata to plugin.approval.request.

## Best Possible Solution

Thread trusted originating route metadata from createOpenClawCodingTools or equivalent run context into HookContext, forward only that trusted metadata to plugin.approval.request, and add routed plus route-less regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has route options before tool wrapping, drops them from HookContext and plugin.approval.request, and the gateway returns decision: null when no approval client, forwarder, or turn-source route exists.

## Solution Assessment

Yes, if the fix uses trusted runtime context. The original args.params/baseParams direction is not the best solution because those values are model/tool-call controlled and can affect approval routing.
