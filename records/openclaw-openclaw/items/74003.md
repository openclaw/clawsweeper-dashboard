## Summary

Keep this open. Current `main` still drops trusted turn-source routing metadata before `before_tool_call` plugin approvals call `plugin.approval.request`, so the gateway can still return `decision: null` and the agent maps that to the reported no-route failure; the linked repair PR #74150 is closed and unmerged.

## What This Changes

The issue asks OpenClaw to forward trusted originating-channel turn-source metadata into `plugin.approval.request` for `before_tool_call` plugin approvals.

## Best Possible Solution

Propagate trusted `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId` through the hook/tool wrapper context into both trusted-policy and ordinary `before_tool_call` approval requests, while preserving the current fail-closed behavior when no route exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated run passes route data into `createOpenClawCodingTools`, the wrapper drops it, `requestPluginToolApproval` sends no turn-source fields, and `approval-shared` returns `decision: null` when no other route is available.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to carry trusted runtime route context through `HookContext`; reading these fields from tool params or hook-returned params would be spoofable.
