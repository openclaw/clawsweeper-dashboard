## Summary

Keep open. Current main still has the reported before_tool_call plugin approval routing gap, and the only linked fix PR was closed unmerged after its params-based approach was identified as unsafe.

## What This Changes

The issue asks OpenClaw to forward trusted turn-source routing metadata from before_tool_call plugin approvals into plugin.approval.request so chat approvals route back to the originating conversation.

## Best Possible Solution

Thread trusted originating route metadata from the run/tool-construction context into HookContext or an equivalent trusted structure, then forward only that trusted metadata to plugin.approval.request with regression coverage for routed and route-less callers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: route options exist at tool construction, the hook context drops them, requestPluginToolApproval sends no turnSource* fields, and the gateway returns decision: null when no other approval route exists.

## Solution Assessment

Yes, but only if the repair uses trusted runtime context. The closed param-based PR should not be reused because its discussion identified model/tool-call spoofing risk when route metadata is read from baseParams.
