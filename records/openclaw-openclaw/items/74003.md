## Summary

Keep this open. Current main still omits trusted turn-source routing metadata from the before_tool_call plugin approval request, while the gateway already accepts those fields and fail-closes with the reported null decision when no approval route is available. The linked repair PR #74150 is closed and unmerged, and the reporter's correction matters: the fix should use trusted runtime context, not model/tool parameters.

## What This Changes

The issue reports that before_tool_call requireApproval plugin approvals are sent to plugin.approval.request without trusted turn-source route fields, causing chat-channel approval prompts to fail with no approval route.

## Best Possible Solution

Extend the internal hook/tool wrapper context with trusted route metadata from the agent runtime, forward those fields into plugin.approval.request for both trusted-policy and ordinary before_tool_call approval paths, and add regression coverage proving routed requests work while unrouted requests still fail closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a before_tool_call requireApproval reaches requestPluginToolApproval without turnSource* fields, then handlePendingApprovalRequest returns decision:null when no approval client or forwarder is available, producing the reported error.

## Solution Assessment

Yes, with the corrected trust boundary. The narrow maintainable fix is to propagate trusted runtime route context from createOpenClawCodingTools into HookContext and then into plugin.approval.request; reading args.params or hook-returned params would be less safe because those are model/plugin-controlled.
