## Summary

Keep open. Current main still matches the corrected routing bug: route metadata reaches tool construction, but the before-tool-call approval context and `plugin.approval.request` payload do not carry it, so the gateway can return `decision: null` when no other approval route exists. The linked fix PR #74150 is closed unmerged, and #75749 tracks adjacent duplicate-delivery fallback behavior rather than replacing this report.

## What This Changes

The issue reports that `before_tool_call` plugin approvals do not receive origin-channel routing metadata, so approval prompts cannot be routed back to channels like Telegram.

## Best Possible Solution

Propagate server-owned origin route fields through the before-tool-call hook context into `plugin.approval.request` for both trusted-policy and ordinary hook approvals, with regression tests proving the fix does not read model-controlled tool params.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: channel route data is available when tools are built, the hook approval context drops it, the plugin approval request sends no route fields, and the gateway returns `decision: null` when no other route exists.

## Solution Assessment

Yes. The narrow maintainable fix is to carry trusted runtime route context through `HookContext` and into `plugin.approval.request`; #74150 is useful prior work, but the reporter's correction means the repair should avoid model-generated params.
