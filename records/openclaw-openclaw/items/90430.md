## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Resolve the same transcript path used for recovery reads:** `src/agents/main-session-restart-recovery.ts:602`
  - body: This builds the append target with raw `entry.sessionFile`, bypassing the containment and stale-metadata fallback in `resolveSessionFilePath`. Recovery reads already go through the transcript candidate resolver, so a stale/outside `sessionFile` can make this append or create the notice at a different path while WebChat reconnects to the canonical transcript and still misses the notice. Resolve the transcript path through the existing session helper, including the no-`sessionFile` fallback, before injecting the message.
  - confidence: 0.91
- **[P2] Check the transcript append result before reporting success:** `src/agents/main-session-restart-recovery.ts:316-322`
  - body: `appendInjectedAssistantMessageToTranscript` catches append errors and returns `{ ok: false, error }`, and sibling WebChat callers check that result. This code ignores the result, logs the notice as persisted, and returns `true`, so filesystem or lock failures still leave the failed session with no visible recovery notice and no failure signal. Check `ok` and return/log failure when the append fails.
  - confidence: 0.88
