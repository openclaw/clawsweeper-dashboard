## Summary

Keep open. Current main still has the reported outage surface: the shared keepalive loop has no tick-level failure breaker, Telegram typing still inherits the normal outbound retry runner, and the dedicated sendChatAction guard only suspends repeated 401s. The discussion narrowed the root cause correctly, and open related work needs maintainer scope triage before creating another repair branch.

## What This Changes

The issue requests bounding Telegram typing keepalive and sendChatAction failures during network outages so best-effort typing indicators cannot stall or crash the gateway.

## Best Possible Solution

Land one canonical runtime fix that treats typing indicators as best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply or enter a short cooldown without weakening normal message delivery retries, preserve existing 401 suspension behavior, and add regression coverage for both shared typing lifecycle and Telegram sendChatAction transient-failure behavior.

## Reproduction Assessment

Yes. The issue gives concrete network-outage steps and production log signatures, and current main still contains source paths that can repeatedly invoke expensive Telegram typing failures under outage conditions.

## Solution Assessment

Yes for the refined direction, but unclear whether the current open PR alone is sufficient. The maintainable fix should combine a bounded typing keepalive policy with Telegram-specific cheap retry or cooldown behavior, rather than only adding one generic catch or only suppressing one handler path.
