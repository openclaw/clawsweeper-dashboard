## Summary

Keep open. Current main still leaves the reported outage surface: the shared keepalive loop has no tick-level breaker, direct Telegram typing still uses the normal retry runner, and the existing sendChatAction guard only suspends repeated 401s while transient network failures are still handled by unmerged work.

## What This Changes

The issue requests bounded, best-effort handling for Telegram typing/sendChatAction failures so network outages cannot keep retrying typing indicators until the gateway stalls or crashes.

## Best Possible Solution

Land one canonical runtime fix that treats typing indicators as best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply or enter transient cooldown, preserve 401 suspension and normal message delivery retries, and add focused regression coverage.

## Reproduction Assessment

Yes. The issue provides concrete network-outage steps and production log signatures, and current main still has source paths that can repeatedly invoke expensive Telegram typing/sendChatAction failures under outage conditions.

## Solution Assessment

Unclear. The open transient-cooldown PR is directionally useful, but the maintainable fix should deliberately cover the Telegram-specific retry/cooldown path and decide whether the shared keepalive lifecycle also needs a tick-level breaker.
