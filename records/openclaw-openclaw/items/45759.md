## Summary

Keep open. Current main still leaves the reported Telegram typing outage surface: the generic keepalive loop has no tick-level breaker, Telegram typing still inherits the normal retry runner, and the existing sendChatAction guard only suspends repeated 401s rather than transient network failures.

## What This Changes

Bound Telegram typing keepalive/sendChatAction failures so network outages do not keep retrying cosmetic typing work until the gateway stalls or crashes.

## Best Possible Solution

Land one canonical runtime fix that treats typing indicators as best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply or enter transient cooldown, preserve 401 suspension and normal message delivery retries, and add focused regression coverage.

## Reproduction Assessment

Yes. The issue gives concrete network-outage steps and production log signatures, and current main still has source paths that can repeatedly invoke expensive Telegram typing/sendChatAction failures under outage conditions; I did not run a live outage test in this read-only review.

## Solution Assessment

Unclear whether the open transient-cooldown PR alone is sufficient. The narrow maintainable fix should cover both the Telegram-specific retry/cooldown path and the shared keepalive lifecycle edge without weakening reliable delivery retries for real messages.
