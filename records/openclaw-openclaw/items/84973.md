## Summary

## What This Changes

The PR wraps the Discord outbound `sendMedia` voice-send branch in a try/catch so failed voice delivery falls through to regular text/media delivery.

## Best Possible Solution

Retarget the fix to the `sendPayload`/`createDiscordPayloadSendContext` path or the CLI outbound dependency proxy so `discordVoice` cannot block Discord cron auto-TTS delivery, then prove the cron workflow.

## Reproduction Assessment

Yes, at source level. Current main applies TTS to cron payloads, marks Discord voice-note output as `audioAsVoice`, and dispatches those payloads through `sendPayload`, where `discordVoice` is still resolved before runtime fallback.

## Solution Assessment

No. The patch modifies `sendMedia`, but the reported cron auto-TTS failure goes through `sendPayload`; the safer fix is to stop the CLI deps proxy from manufacturing `discordVoice` or handle that fallback in the payload path.
