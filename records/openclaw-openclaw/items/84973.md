## Best Possible Solution

Retarget the fallback to `sendDiscordOutboundPayload`/`createDiscordPayloadSendContext` or the CLI dependency proxy so cron `audioAsVoice` delivery falls back only for unavailable voice adapters, with focused regression coverage and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main applies TTS before cron delivery, carries `audioAsVoice` into durable delivery, and Discord routes those payloads through `sendPayload`, where `sendVoice` is still attempted before any text/media fallback.

## Solution Assessment

No. The patch changes `sendMedia`, but the reported cron announce failure follows `sendPayload`; a maintainable fix needs to retarget that path and avoid swallowing unrelated voice-send errors.
