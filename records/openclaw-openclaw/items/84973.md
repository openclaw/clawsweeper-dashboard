## Summary

Keep open: the PR is aimed at the right Discord `sendPayload` path, but it is not merge-ready because the fallback still carries `audioAsVoice` into the generic media helper, which re-enters Discord `sendMedia` and retries voice delivery. It also swallows all exhausted voice-send errors, and the supplied proof does not show the real Discord cron workflow after the patch.

## What This Changes

The PR modifies `extensions/discord/src/outbound-payload.ts` so Discord `audioAsVoice` payload delivery catches voice-send failure and falls through toward text plus media delivery.

## Best Possible Solution

Make the payload fallback explicitly bypass voice delivery by clearing `audioAsVoice` or sending text/media directly, narrow fallback to the unavailable-adapter condition, add focused regression coverage, and then provide redacted real Discord cron proof.

## Reproduction Assessment

Yes, at source level: current durable delivery routes `audioAsVoice` cron payloads through `sendPayload`, and Discord `sendPayload` calls the voice sender for the first media URL. I did not run a live Discord cron repro in this read-only review.

## Solution Assessment

No. Moving the fix to `sendPayload` is the right direction, but the patch needs to bypass `audioAsVoice` on fallback and catch only the unavailable-adapter case.
