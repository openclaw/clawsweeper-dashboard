## Summary

Keep open: the linked Discord cron/TTS bug is valid, but this PR changes the `sendMedia` fallback path while durable `audioAsVoice` cron delivery goes through `sendPayload`, and the new catch would also hide non-unavailable voice-send failures.

## What This Changes

Wraps the Discord `sendMedia` `audioAsVoice` branch in a catch so voice-send failures fall through to text/media attachment delivery.

## Best Possible Solution

Move the fallback to the Discord payload delivery path used by durable cron delivery, narrow it to the unavailable voice-adapter condition, add focused regression coverage, and require redacted real Discord/log proof.

## Reproduction Assessment

Yes, at source level: cron TTS payloads flow into `sendDurableMessageBatch`, durable delivery routes `audioAsVoice` payloads through `sendPayload`, and Discord `sendPayload` directly calls `sendVoice` without fallback. I did not run a live Discord repro because this was a read-only review.

## Solution Assessment

No. The PR patches `sendMedia`, but the reported cron path uses `sendPayload`; the narrower maintainable fix should live in the Discord payload send path and catch only the unavailable voice-adapter case.
