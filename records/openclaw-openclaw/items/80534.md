## Summary

The MCP `tools.allow` prompt-leak fix is already implemented on current main and shipped; this PR should not merge because it repeats that path with a weaker boundary and also carries unrelated gateway skill-install protocol/security changes.

## What This Changes

The branch adds an MCP system-prompt tool filter in the embedded compaction path and tests, while also changing `skills.install` unsafe-override behavior across gateway schema/server code, UI, macOS, docs, and changelog.

## Best Possible Solution

Keep the shipped bundled-tool final-policy implementation and handle any future gateway unsafe-install policy change as a separate security/protocol PR.

## Reproduction Assessment

Yes, source-based: the reported prompt path is the `effectiveTools` assembly used for compaction/system-prompt construction, and current main filters bundled MCP/LSP tools before that list reaches the prompt. I did not run a live MCP server, but the current tests cover bundled-tool allowlist filtering.

## Solution Assessment

No, this PR is not the best way to solve it now. The shipped main implementation keeps the final policy pass scoped to bundled MCP/LSP tools, while this branch re-runs that policy over the combined tool list and also mixes in unrelated gateway security/protocol changes.

## Review Findings

- **[P2] Do not re-run final policy over core tools:** `src/agents/pi-embedded-runner/effective-tool-filter.ts:40-43`
