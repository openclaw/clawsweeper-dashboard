## Summary

Keep open: the patch targets a real current-main gateway dispatch bug, but current main does not yet fix it and the overlapping replacement PR is still unmerged, so this needs maintainer selection rather than cleanup closure.

## What This Changes

The branch changes `handleGatewayRequest` to always build the per-request gateway method registry from the live active plugin registry instead of honoring an optional cached `methodRegistry` snapshot.

## Best Possible Solution

Land one canonical live-registry dispatch fix with stale-registry regression coverage and redacted live CLI proof that a plugin-registered method succeeds through `openclaw gateway call`.

## Reproduction Assessment

Yes. Source inspection shows current main can pass a stale startup `methodRegistry` into `handleGatewayRequest`, while plugin `registerGatewayMethod` writes handlers and descriptors into the live active registry that the cached lookup skips.

## Solution Assessment

No, not as a complete landing shape. The runtime change is the right layer, but the best fix should include focused stale-registry regression coverage and live CLI proof; https://github.com/openclaw/openclaw/pull/94154 currently has the test but is still unmerged.
