## Summary

Keep this PR open, but it is not merge-ready: the reported gateway bug is valid, yet this branch unconditionally drops the caller-supplied dispatch registry that current main now uses for plugin-method authorization and dispatch, and the contributor proof is still only tests plus source analysis.

## What This Changes

The branch changes `handleGatewayRequest` to ignore the optional supplied `methodRegistry` and rebuild gateway method lookup from the live active plugin registry for every request.

## Best Possible Solution

Land one conflict-free gateway dispatch fix that uses the live registry when the attached registry is stale, preserves attached-registry dispatch when it is newer, adds regression coverage for both cases, and includes redacted live CLI proof.

## Reproduction Assessment

Yes. Source inspection shows the reported stale-startup registry bug, and current main also has an attached-registry regression test that this PR's unconditional live-only lookup would violate after rebase.

## Solution Assessment

No. The runtime layer is the right place to fix the bug, but the best fix must preserve the supplied attached registry when it contains the method and fall back to the live registry only for stale-miss cases.
