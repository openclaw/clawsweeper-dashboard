## Summary

Keep open. Current main still has the OpenAI-compatible HTTP hardcoded-prefix behavior, but this PR is not ready to merge because it derives the new prefix from the agent default before applying `x-openclaw-model` overrides and it lacks the required changelog entry for a user-visible, backward-incompatible session-key namespace change.

## What This Changes

The PR derives OpenAI-compatible HTTP generated session-key prefixes from the selected agent’s configured default provider, threads a pre-resolved agent id into gateway request context, and adds provider-prefix tests.

## Best Possible Solution

Maintainers should choose the intended generated-key namespace; if provider-specific keys are desired, derive the prefix from the final accepted backend model selection, keep explicit `x-openclaw-session-key` authoritative, cover default and override paths, and document the change in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Static inspection reproduces the main behavior because `/v1/chat/completions` still passes `sessionPrefix: "openai"`; the PR defect is reproduced by an Anthropic-default agent with `x-openclaw-model: openai/gpt-5.4`, which would run the OpenAI override under an Anthropic-prefixed generated key.

## Solution Assessment

No. The PR is a focused attempt, but the prefix is computed before backend model override resolution and the change needs an explicit changelog note plus maintainer confirmation on the backward-incompatible namespace.
