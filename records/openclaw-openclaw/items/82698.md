## Summary

Keep open. The linked current-main bug is valid and the branch is close, but the patch leaves a stale include-add suppression that can swallow the real recreate event after an included config file is deleted and restored.

## What This Changes

The PR adds gateway config include-file watching, passes the startup snapshot into the reloader, updates include watch paths after accepted snapshots and in-process writes, and extends regression tests.

## Best Possible Solution

Keep the include-watch approach, but remove or safely expire add-event suppression under Chokidar v5 and add a regression test for deleting and recreating an included config file before landing.

## Reproduction Assessment

Yes. Current main only watches the root `watchPath` while snapshot reads resolve `$include`, so an include-only edit has no watched path to schedule reload; the PR-specific blocker is also source-reproducible from Chokidar v5 `ignoreInitial` handling.

## Solution Assessment

No. The overall design is the right narrow fix, but this implementation assumes dynamic `watcher.add()` will clear the suppression through an add event, which Chokidar v5 suppresses with `ignoreInitial: true`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear include add suppressions before real add events:** `src/gateway/config-reload.ts:185-188`
  - body: With Chokidar 5 and `ignoreInitial: true`, adding an existing include file with `watcher.add()` does not emit the initial `add` event, so the suppression inserted here stays armed. If that include file is later deleted and recreated, the real recreate `add` event is swallowed and no reload runs, leaving the gateway stale until another watched change happens. Remove this suppression or make it expire, and add a delete/recreate regression test.
  - confidence: 0.89
