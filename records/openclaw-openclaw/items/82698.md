## Summary

Keep open: the PR targets a real source-proven gateway config hot-reload bug and supplies after-fix terminal proof, but the latest diff still has a startup include-watch race that should be fixed before merge.

## What This Changes

The PR adds include-path tracking to the gateway config reloader, passes the startup snapshot through managed startup, and adds regression tests for include watching and root-hash dedupe.

## Best Possible Solution

Keep the include-watch design, but serialize or generation-guard startup recursive sync so stale startup results cannot replace a newer validated watched set.

## Reproduction Assessment

Yes. Current main only watches the root config path, while $include resolution happens during snapshot reads, so editing only an included file has no watched event path; the PR finding is source-reproducible by interleaving the unawaited startup recursive sync with an early valid reload.

## Solution Assessment

No, not yet. The PR’s overall direction is the narrow maintainable fix, but the unguarded startup recursive sync needs to be serialized or made stale-result-safe before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Serialize the startup include-watch sync:** `src/gateway/config-reload.ts:505`
  - body: The startup path kicks off `syncIncludeWatchPaths(opts.initialSnapshot, "startup")` without awaiting it or checking that its result is still current. If the root config changes while that recursive startup scan is still reading include files, a valid reload can install the new watched set and then the stale startup scan can replace it and unwatch the new include path, recreating the startup race called out on the earlier include-watch PR. Track and await the startup sync, or discard stale sync results before applying them.
