## Summary

Keep open. The PR is a focused fix for a source-proven config reload bug and includes useful real behavior proof, but the proposed startup include-watch sync can still apply stale include paths after a later reload has already synced the current config.

## What This Changes

The PR threads the startup config snapshot into gateway config reloading, dynamically watches `$include` paths with Chokidar, bypasses include-only hash dedupe, and adds regression tests.

## Best Possible Solution

Keep the dynamic include-watch approach, but make include-watch synchronization ordered or generation-aware so stale startup scans cannot replace a newer valid watch set.

## Reproduction Assessment

Yes. Current main watches only the active root config path, so an edit to a `$include`-referenced file has no watched event path to schedule a reload; the PR also supplies after-fix terminal proof for that path.

## Solution Assessment

No, not yet. The proposed direction is the right narrow fix, but the fire-and-forget startup recursive sync needs stale-result protection before this is the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Serialize the startup include sync:** `src/gateway/config-reload.ts:508`
  - body: The recursive startup sync is launched fire-and-forget from the startup snapshot. If a user changes the root config while that scan is still reading nested includes, the real reload can sync the new include set first, and then the stale startup sync can apply the old set and unwatch the new include file. Guard this with a generation token, cancellation, or a shared sync queue so older scans cannot replace newer watch state.
