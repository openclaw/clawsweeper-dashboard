## Summary

Keep open. The PR is a focused fix for a real gateway config hot-reload bug and its real behavior proof is sufficient, but the latest head still has a source-level startup include-watch race and the PR is currently draft.

## What This Changes

The branch adds include-path tracking to the gateway config reloader, passes the startup config snapshot into managed startup, and adds regression tests for include watching and root-hash dedupe.

## Best Possible Solution

Keep the include-watch design, but make startup recursive include synchronization stale-result-safe before this PR leaves draft or lands.

## Reproduction Assessment

Yes, source-level. Current main watches only the root config path while $include resolution happens inside snapshot reads, so editing only an included file has no watched event path; the PR-specific race is visible from the unawaited startup recursive sync.

## Solution Assessment

No, not yet. The PR’s direction is the narrow maintainable fix, but the startup include-watch synchronization needs an await, serialization, or generation guard so stale recursive results cannot replace a newer watched set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Serialize startup include-watch sync:** `src/gateway/config-reload.ts:508`
  - body: The startup path applies direct include paths, then fires `syncIncludeWatchPaths(opts.initialSnapshot, "startup")` without awaiting or generation-guarding it. If the root config changes while that recursive scan is still reading old include files, a valid reload can install the new watched set and then the stale startup scan can apply its old set, unwatching the new include path and losing future include-only reloads. Track/await the startup sync or discard stale results before applying them.
  - confidence: 0.86
