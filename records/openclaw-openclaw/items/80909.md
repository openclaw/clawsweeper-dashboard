## Summary

Keep open: the current evidence points to sandbox tool allowlist filtering, not a generic provider serializer drop, and current main still lacks a targeted diagnostic/docs guard for this real config shape.

## What This Changes

Configured MCP and native plugin tools are reported as absent from provider-bound `tools[]` when a sandboxed setup uses a restrictive `tools.sandbox.tools.alsoAllow`.

## Best Possible Solution

Add a focused current-main diagnostic/docs/test fix that explains and detects sandbox filtering of MCP/plugin tools, with targeted guidance to allow `bundle-mcp`, exact server globs, plugin ids, or `group:plugins` rather than broadening sandbox defaults silently.

## Reproduction Assessment

Yes for a source-level/current-main path: a sandboxed `coding` run with `tools.sandbox.tools.alsoAllow` that omits `bundle-mcp` or `group:plugins` will materialize MCP tools and then filter them before `customTools`. I did not run the provider-body harness in this read-only review, but the issue discussion includes a current-main local provider-body reproduction of the same shape.

## Solution Assessment

No, the original provider-serialization framing is no longer the best repair target. The narrow fix should make the sandbox allowlist gate visible through docs, doctor or sandbox diagnostics, and regression coverage while preserving the existing explicit sandbox policy semantics.
