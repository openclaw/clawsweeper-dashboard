## Summary

Keep open: the issue has fresh stable-release wire evidence that conflicts with the existing static/customTools proof, and the latest comments broaden the suspected drop from MCP-only to native plugin tools. Current main still needs a provider-body reproduction for the OpenAI-compatible chat-completions path before this can be closed or safely patched.

## What This Changes

The issue reports that configured MCP server tools, and later arbitrary plugin `api.registerTool()` tools, do not reach the outbound OpenAI-compatible `tools[]` request payload despite healthy registration and direct MCP/tool checks.

## Best Possible Solution

Add provider-boundary coverage that captures the actual OpenAI-compatible chat-completions payload for both a configured stdio MCP server and a minimal native `api.registerTool()` plugin, then fix the first layer that drops those tools while preserving `minimal` and explicit deny behavior.

The current tests prove only the forced-Pi/customTools boundary, not the reported OpenRouter chat-completions payload.

## Solution Assessment

No. The existing test-only/customTools proof is not enough to resolve the issue; the narrow maintainable path is to add a real or fake-provider chat-completions request-boundary test plus a native plugin fixture before changing runtime policy.
