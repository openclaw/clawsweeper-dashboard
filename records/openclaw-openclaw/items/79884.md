## Summary

Keep open: current main still routes the WeCom wizard install through a catalog pin for `@wecom/wecom-openclaw-plugin@2026.4.23` and the npm installer rejects an existing package directory in install mode, matching the reported failure. The remaining fix should be handled deliberately because it intersects the official external npm integrity/provenance policy.

## What This Changes

The issue asks the configuration wizard to recognize an already-installed newer WeCom plugin instead of trying to install the older catalog-pinned version and failing on the existing package directory.

## Best Possible Solution

Preserve exact-integrity trust for official external plugins while making wizard setup reuse or repair a valid already-installed WeCom package instead of attempting a stale pinned reinstall.

## Reproduction Assessment

Yes, source-backed: choose WeCom in the wizard with `/home/node/.openclaw/npm/node_modules/@wecom/wecom-openclaw-plugin` already present, and the current install-mode path reaches the tested `plugin already exists` failure. I did not run the Docker wizard live in this read-only review.

## Solution Assessment

Unclear as an automated fix: the narrow UX bug is real, but the safest implementation needs maintainer choice between repinning the official WeCom catalog and validating/reusing an installed newer package without bypassing integrity provenance.
