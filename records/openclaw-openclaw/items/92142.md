## Summary

Keep this open. Current main and v2026.6.5 still forward iMessage final-reply metadata into outbound send without honoring the existing iMessage private-action gate, and targeted GitHub searches found no duplicate or merged fix.

## What This Changes

iMessage final delivery should avoid native/threaded reply metadata when reply actions or private API support are disabled, falling back to a normal top-level send.

## Best Possible Solution

Gate iMessage reply metadata at the plugin send/final-delivery boundary using the existing actions/private-API contract, and leave the separate TTS override request for its own issue.

## Reproduction Assessment

Yes for source reproduction: current main forwards payload.replyToId into iMessage final delivery, and sendMessageIMessage serializes it as reply_to without checking actions.reply. I did not establish a live SIP-enabled macOS repro in this runner.

## Solution Assessment

No: current main is not the best solved state because normal final delivery can still request SIP-gated threaded reply behavior. The narrow maintainable fix is to strip or gate reply metadata in the iMessage plugin path before adding a new replyToMode config surface.
