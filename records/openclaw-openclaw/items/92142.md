## What This Changes

Prevent iMessage final delivery from requesting a native threaded reply when `channels.imessage.actions.reply` is disabled.

## Best Possible Solution

Land the focused plugin-local action gate from the linked PR, verify that its real send path omits `reply_to`, and track capability-driven fallback or per-channel TTS policy as separate work requiring their own contracts and product review.

## Reproduction Assessment

Yes at source level: current main maps `replyToId` to `reply_to` without honoring `actions.reply`, matching the supplied SIP error and fake-client parameter check. This review did not independently execute the flow on a SIP-enabled Mac.

## Solution Assessment

Yes for the reported configured-disable case: enforcing the existing action setting inside the owning iMessage sender is narrower and more maintainable than adding another reply-mode setting. Automatic fallback based on bridge capabilities would be separate work because its upstream contract was not established.
