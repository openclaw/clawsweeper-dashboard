## Summary

Keep this open. Current main and the latest stable release still have the iMessage final-delivery path that forwards reply metadata into outbound send even when the iMessage private reply action is disabled, and no canonical duplicate or merged fix was found.

## What This Changes

The issue reports that iMessage final replies pass native reply metadata to `imsg` even when reply actions are disabled, causing SIP-gated delivery failures instead of degrading to a top-level send.

## Best Possible Solution

Fix iMessage final delivery to honor the existing iMessage reply action/private-API boundary before forwarding reply metadata, and split the separate iMessage TTS override request into its own product/config discussion if maintainers want to pursue it.

## Reproduction Assessment

Yes for source reproduction: current code forwards `payload.replyToId` to `sendMessageIMessage`, and the send path serializes it as RPC `reply_to` without checking `actions.reply`. I did not establish a live SIP-enabled macOS repro in this runner.

## Solution Assessment

No, current main is not the best solved state because the existing iMessage action/private-API boundary is not honored by normal final delivery. The maintainable fix is to gate or strip reply metadata at the iMessage send/delivery boundary instead of adding a new config surface first.
