## Summary

Keep this open. Current main and v2026.6.5 still route iMessage final-reply metadata into outbound send without honoring the existing iMessage reply action gate, so the reported SIP-gated delivery failure remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop sending iMessage final replies as native/threaded replies when private iMessage reply actions are disabled or unavailable.

## Best Possible Solution

Honor the existing iMessage private reply action boundary in the send or final-delivery path so disabled reply actions degrade to top-level iMessage sends without adding a new config surface first.

## Reproduction Assessment

Yes for source reproduction: current main forwards iMessage final reply metadata from monitor delivery to the send layer, and send serializes it as RPC `reply_to` without checking `actions.reply`. A live SIP-enabled macOS run was not performed in this runner.

## Solution Assessment

No, current main is not the solved state because the existing iMessage action/private-API boundary is not honored by final delivery. The narrow maintainable fix is to strip or gate reply metadata at the iMessage send/delivery boundary before considering any new config option.
