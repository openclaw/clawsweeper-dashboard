## Summary

Current main still forwards iMessage reply metadata without consulting the existing `actions.reply` setting, and the resulting native reply path is SIP-gated by imsg. Keep the issue open because the clean linked closing PR is still under review and has not merged.

## What This Changes

Make iMessage final delivery omit `reply_to`/`replyToId` when `channels.imessage.actions.reply` is disabled so delivery falls back to a top-level message.

## Best Possible Solution

Land the linked sender-boundary fix with focused regression coverage, preserve reply behavior when the action is enabled, and track any per-channel TTS override as a separate product/config decision.

## Reproduction Assessment

Yes at source level: current main unconditionally maps the supplied reply ID to imsg's `reply_to`, while upstream imsg confirms that native reply injection is refused with SIP enabled. A live macOS reproduction was not run in this read-only review.

## Solution Assessment

Yes for the central bug: honoring the existing action setting at the iMessage sender boundary is the narrowest maintainable fix and avoids adding another config mode. The separate TTS request is not part of this repair.
