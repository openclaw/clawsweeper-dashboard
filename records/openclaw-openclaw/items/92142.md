## Summary

Keep this open. Current main and v2026.6.5 still forward iMessage final-reply metadata into outbound send without honoring the existing iMessage reply action/private-API boundary, and no duplicate or merged fix was found.

## What This Changes

The issue requests that iMessage final response delivery degrade to normal top-level sends instead of passing reply metadata when iMessage threaded reply actions are disabled or unavailable.

## Best Possible Solution

Strip or gate iMessage reply metadata at the plugin send/final-delivery boundary when threaded reply is disabled or unavailable, using existing action/private-API surfaces rather than adding a new config option first; handle the TTS schema note separately.

## Reproduction Assessment

Yes for source reproduction: current main forwards payload.replyToId into iMessage final delivery, and sendMessageIMessage serializes it as reply_to without checking actions.reply. I did not establish a live SIP-enabled macOS repro in this runner.

## Solution Assessment

No: current main is not the best solved state because the existing iMessage action/private-API boundary is not honored by normal final delivery. The narrow maintainable fix is to strip or gate reply metadata at the iMessage delivery/send boundary, not to add a new replyToMode surface first.
