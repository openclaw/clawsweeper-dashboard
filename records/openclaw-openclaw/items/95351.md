## Summary

Keep open: current main still has only core-owned Claude/Gemini JSONL parsing for CLI backend tool events, and there is no public `CliBackendPlugin` hook for third-party JSONL schemas or display-only tool-card events. This is useful, but it changes the plugin SDK/runtime contract and needs maintainer API design before implementation.

## What This Changes

The issue requests a generic JSONL parser or display-only tool-event hook so third-party `CliBackendPlugin` backends can emit native tool-card events without impersonating Claude's stream format.

## Best Possible Solution

Design a narrow public CLI backend streaming hook or blessed harness path that maps backend JSONL into sanitized assistant, commentary, session, result, and display-only tool events, with docs and focused parser/runner tests.

## Reproduction Assessment

Yes for the source-level capability gap: current main has finite built-in JSONL dialect dispatch and no plugin-owned parser or display-only tool-event hook. I did not run a failing user scenario because this is a feature/API request rather than broken documented behavior.

## Solution Assessment

Unclear: a generic display-only parser hook fits the plugin boundary if maintainers want third-party CLI parity, but the exact SDK shape and whether this belongs in `CliBackendPlugin` or an agent-harness path needs maintainer design approval.
