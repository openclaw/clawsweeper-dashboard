## Summary

Keep open: current main partly broadened JSONL support with built-in Gemini parsing, but `CliBackendPlugin` still has no generic plugin-owned JSONL event parser or display-only tool-event hook for third-party CLI backends.

## What This Changes

The issue requests a public CLI backend extension point that maps third-party JSONL stream records into assistant, thinking, session, result, and native tool-card events.

## Best Possible Solution

Design a narrow plugin SDK/runtime hook that lets CLI backends normalize JSONL stream records into existing assistant, commentary, session, result, and sanitized display-only tool events, with docs and focused parser/runner tests.

## Reproduction Assessment

Yes for the source-level capability gap: current main has no plugin-owned JSONL parser hook and only accepts built-in dialect literals. This is a feature/API request, not a broken existing documented behavior.

## Solution Assessment

Unclear: a generic display-only parser hook is the right product layer if maintainers want third-party CLI parity, but the exact SDK shape and event contract need maintainer design approval.
