## Summary

Keep open: current main still exposes only built-in Claude/Gemini JSONL dialect parsing, so third-party `CliBackendPlugin` backends still lack the requested generic parser or display-only tool-event hook. This is a valid plugin SDK/runtime feature request that needs maintainer API design rather than cleanup closure.

## What This Changes

Requests a generic `CliBackendPlugin` JSONL parsing or tool-event hook so third-party CLI backends can surface native tool cards without emulating Claude or Gemini wire formats.

## Best Possible Solution

Design a narrow plugin SDK/runtime hook that normalizes backend-owned JSONL records into existing assistant, commentary, session, result, and sanitized display-only tool events, with docs and focused parser/runner tests.

## Reproduction Assessment

Yes for the source-level capability gap: current main has no plugin-owned JSONL parser hook and explicitly treats generic JSONL as not supporting correlated tool events. This is a feature/API request, not broken documented behavior.

## Solution Assessment

Unclear until maintainers choose the API shape: a generic display-only parser hook fits the plugin boundary, but the normalized event contract and safety semantics need product/API approval.
