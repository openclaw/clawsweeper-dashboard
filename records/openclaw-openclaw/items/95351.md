## Summary

Keep open: current main still has only built-in Claude/Gemini JSONL dialect handling, so third-party `CliBackendPlugin` backends do not have the requested generic parser or display-only tool-event hook.

## What This Changes

Request to add a generic plugin-owned JSONL parsing/tool-event hook for `CliBackendPlugin` backends so non-Claude/Gemini CLIs can emit native display-only tool cards.

## Best Possible Solution

Design a narrow plugin SDK/runtime hook that lets CLI backends normalize JSONL records into existing assistant, commentary, session, result, and sanitized display-only tool events, with docs and focused parser/runner tests.

## Reproduction Assessment

Yes for the source-level capability gap: current main has no plugin-owned JSONL parser hook and only accepts built-in dialect literals. This is a feature/API request, not a broken existing documented behavior.

## Solution Assessment

Unclear: a generic display-only parser hook fits the plugin boundary if maintainers want third-party CLI parity, but the exact SDK shape and event contract need maintainer design approval.
