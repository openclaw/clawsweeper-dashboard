## Summary

Keep open: current main and v2026.6.9 still lack a plugin-owned JSONL parser/tool-card hook, and the open linked PR is only a candidate implementation that still needs maintainer API/product review before this issue can close.

## What This Changes

Request a public `CliBackendPlugin` JSONL parsing or display-only tool-event hook so third-party CLI backends can feed native tool cards without impersonating built-in Claude/Gemini dialects.

## Best Possible Solution

Keep this issue as the canonical API-design tracker, then either sponsor and land the linked implementation after maintainer approval or replace it with a narrower approved parser-event contract.

## Reproduction Assessment

Yes for a source-level capability gap: current main has no `parseJsonlEvent` hook, only built-in Claude/Gemini JSONL dialects, and no plugin-owned path into native tool-card callbacks. This is a feature/API request rather than broken existing documented behavior.

## Solution Assessment

Unclear as a final product/API choice: a backend-owned parser hook fits the plugin boundary, but the public normalized event contract needs maintainer approval. The issue should remain open until an approved implementation merges or maintainers choose a different API shape.
