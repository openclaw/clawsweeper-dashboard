## What This Changes

The PR adds a cron runnable-selection guard plus restart catch-up regression coverage so stale `nextRunAtMs` slots are not executed again when `lastRunAtMs` is later than the slot.

## Best Possible Solution

Port the strict stale-slot guard onto the current `isRunnableJob` path, keep focused restart regression coverage, collect redacted restart-dedupe proof, and run fresh targeted cron plus changed-surface validation.

## Reproduction Assessment

Yes, by source inspection. Seed a stored cron job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot, then start cron: current main calls startup catch-up before maintenance repair and treats the due `nextRunAtMs` as runnable.

## Solution Assessment

Yes, after rebase and proof. Guarding due-slot runnable selection with strict `lastRunAtMs > scheduledAtMs` is the narrow fix and preserves same-millisecond retry slots.
