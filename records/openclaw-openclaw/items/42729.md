## What This Changes

The PR adds a cron timer guard to skip stale already-executed scheduled slots after gateway restart and adds a restart regression test for that state.

## Best Possible Solution

Port the stale-slot guard onto current `isRunnableJob`, keep focused restart coverage for the past-due already-executed slot, and validate with targeted cron tests plus redacted real restart proof.

## Reproduction Assessment

Yes, by source inspection: seed an enabled cron job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot, then start the gateway so `runMissedJobs` sees the stale due timestamp before maintenance advances it. I did not execute it because this review is read-only.

## Solution Assessment

Yes, after a current-main port: guarding only the due-`nextRunAtMs` fast path against an already-executed stale slot is the narrowest maintainable fix while preserving missed-slot replay and retry behavior.
