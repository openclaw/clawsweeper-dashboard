## Summary

Keep this PR open. Current main covers the future-`nextRunAtMs` restart shape, but `isRunnableJob` still treats a past-due stale `nextRunAtMs` as runnable before consulting `lastRunAtMs`; this PR addresses that remaining scheduler gap, though it needs a current-main port and contributor real behavior proof before merge.

## What This Changes

The PR adds a strict cron stale scheduled-slot guard in `src/cron/service/timer.ts` and a restart catch-up regression test for a past-due slot already covered by `lastRunAtMs`.

## Best Possible Solution

Port the strict stale-slot guard onto current `isRunnableJob`, keep focused restart regression coverage, collect redacted real restart-dedupe proof, then run targeted cron tests and changed-surface validation.

## Reproduction Assessment

Yes. By source inspection, seed a stored enabled cron job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot; startup catch-up currently treats the stale due `nextRunAtMs` as runnable before maintenance repair runs.

## Solution Assessment

Yes, after port and proof. The strict `lastRunAtMs > scheduledAtMs` guard is narrow, preserves same-millisecond retry slots, and matches the stale-slot invariant already used by the maintenance repair path.
