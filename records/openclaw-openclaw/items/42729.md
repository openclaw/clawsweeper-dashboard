## Summary

Keep open. Current main still has the past-due `nextRunAtMs` fast path that can collect an already-executed stale slot before startup maintenance repairs it, while this PR contains a focused fix and test for that remaining case; merge is blocked by a dirty/conflicting head and missing real behavior proof.

## What This Changes

The PR adds a strict stale scheduled-slot guard to cron `isRunnableJob` and a restart regression test for a persisted due `nextRunAtMs` older than `lastRunAtMs`.

## Best Possible Solution

Port the strict stale-slot guard onto current `isRunnableJob`, keep focused restart coverage for the past-due already-executed slot, and validate it with targeted cron tests plus redacted real restart proof.

## Reproduction Assessment

Yes, by source inspection: seed an enabled cron job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot, then start the gateway so `runMissedJobs` sees the stale due timestamp before maintenance repair runs. I did not execute it because this review is read-only.

## Solution Assessment

Yes, after a current-main port: guarding only the due-`nextRunAtMs` fast path against an already-executed stale slot is the narrowest maintainable fix while preserving missed-slot replay and same-millisecond retry behavior.
