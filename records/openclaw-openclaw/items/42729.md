## Summary

Keep open: current main still treats any due `nextRunAtMs` as runnable before checking whether that stale slot was already executed, and startup catch-up sees that state before maintenance advances it. The PR remains a plausible focused fix, but it needs real behavior proof and a conflict port before merge.

## What This Changes

The PR adds a stale already-executed slot guard to cron `isRunnableJob` and a restart catch-up regression test for a past-due `nextRunAtMs` with a later `lastRunAtMs`.

## Best Possible Solution

Port the strict stale-slot guard onto current `isRunnableJob`, keep focused restart coverage for the past-due already-executed slot, then validate with targeted cron tests and redacted real restart proof.

## Reproduction Assessment

Yes, by source inspection: seed an enabled cron job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot, then start the gateway so `runMissedJobs` sees the stale due timestamp before maintenance repair runs. I did not execute it because this review is read-only.

## Solution Assessment

Yes, after a current-main port: gating only the due-`nextRunAtMs` fast path on an already-executed stale slot is the narrowest maintainable fix while preserving missed-slot replay and same-millisecond retry behavior.
