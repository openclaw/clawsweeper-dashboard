## What This Changes

The PR adds a strict `lastRunAtMs > nextRunAtMs` guard in cron runnable selection and a restart catch-up regression test for stale already-executed slots.

## Best Possible Solution

Port the strict stale due-slot guard into current cron timer code, preserve same-millisecond retry behavior, keep focused restart coverage, and land after real proof plus current validation.

## Reproduction Assessment

Yes, by source inspection. On current main, startup calls `runMissedJobs` before the expired-slot maintenance repair, and `isRunnableJob` selects a past-due `nextRunAtMs` without first checking whether `lastRunAtMs` already passed that slot.

## Solution Assessment

Yes, after rebase and proof. Guarding the runnable-selection hot path is the narrowest maintainable fix for the remaining stale past-due slot case, and the strict `>` comparison preserves same-millisecond retry behavior.
