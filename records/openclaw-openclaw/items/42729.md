## Summary

Keep open: current main still treats a past-due `nextRunAtMs` as runnable before checking whether `lastRunAtMs` is later than that scheduled slot, so this PR is not obsolete. The patch is a plausible narrow fix, but it needs a current-main port, redacted real restart or seeded-runtime proof, and a small test-helper update before merge.

## What This Changes

The PR adds a strict `lastRunAtMs > nextRunAtMs` guard in cron's runnable check and a restart catch-up regression test for an already executed stale due slot.

## Best Possible Solution

Port the stale-slot guard onto current `isRunnableJob`, keep strict `>` to preserve same-millisecond retry slots, retain missed-slot replay coverage, and prove the restart behavior in a redacted real or seeded runtime.

## Reproduction Assessment

Yes, by source inspection: seed an enabled cron or every job with `nextRunAtMs` in the past and `lastRunAtMs` later than that slot, then start restart catch-up so `isRunnableJob` returns runnable before checking the slot history. I did not execute it because this review is read-only.

## Solution Assessment

Yes after a current-main port: guarding only the due-`nextRunAtMs` fast path is the narrowest maintainable fix while preserving missed-slot replay. The port must keep strict `>` and update the regression test for the current heartbeat helper.
