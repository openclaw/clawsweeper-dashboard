## Best Possible Solution

## Reproduction Assessment

Yes, by source inspection. On current main, startup invokes `runMissedJobs` before expired-slot maintenance repair, and `isRunnableJob` accepts a past-due `nextRunAtMs` without checking whether `lastRunAtMs` already passed that persisted slot.

## Solution Assessment

Yes, after rebase and proof. Guarding runnable selection with a strict `lastRunAtMs > scheduledAtMs` check is a narrow fix for the stale-slot case while preserving same-millisecond retry behavior.
