## Summary

The PR adds a root `models.json` header side parser, materializes sanitized catalog headers onto discovered runtime models, adds resolver tests, and documents `ctx.model.headers` for provider wrappers.

## Best Possible Solution

If maintainers accept static catalog header visibility, implement it in the current registry/request-config resolver path for root and generated plugin catalogs, preserve SecretRef stripping, and update SDK docs against that single path.

Yes for source-level reproduction: current main stores catalog provider/model headers in registry request-auth maps while parsed model rows keep `headers` undefined, and the wrapper receives the resolved `model`. I did not execute a failing runtime command in this read-only review.

## Solution Assessment

No, the submitted patch is only a plausible fix. The best fix should reuse the current registry/request-config loader instead of adding a second root-only parser, and maintainers should confirm the static-header trust boundary.
