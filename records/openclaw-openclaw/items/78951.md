## Summary

Keep open: the branch targets a real source-level provider-wrapper gap, but it still adds a stricter raw JSON parser for `models.json` than PI uses and the external PR lacks inspectable after-fix real behavior proof.

## What This Changes

The branch materializes sanitized static provider/model headers from generated `models.json` onto discovered PI runtime models, adds resolver coverage, and documents `ctx.model.headers` for provider wrappers.

## Best Possible Solution

Use PI-compatible `models.json` parsing or registry-owned request-header data to materialize sanitized headers, preserve SecretRef stripping, add JSONC/trailing-comma coverage, and attach redacted real terminal/log proof.

## Reproduction Assessment

Yes, as a source-level reproduction: PI stores provider/model request headers from `models.json` separately while generated custom models have `headers` undefined, and OpenClaw passes that model object into provider `wrapStreamFn`. I did not run a failing command because this review is read-only.

## Solution Assessment

No, not yet: exposing sanitized catalog headers is the right narrow boundary, but the patch should reuse PI-compatible parsing or registry data instead of silently dropping headers for JSONC `models.json` files.
