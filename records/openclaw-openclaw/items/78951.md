## Summary

Keep open: this is a plausible focused bug fix for a source-visible provider-wrapper gap, but the current patch should not merge because it reads `models.json` more narrowly than PI itself and still lacks inspectable real behavior proof.

## What This Changes

The PR materializes sanitized non-SecretRef generated catalog headers onto resolved PI models for provider `wrapStreamFn`, with resolver tests and SDK docs.

## Best Possible Solution

Reuse PI-compatible parsing or registry data for request headers, include provider/model/modelOverride header sources with SecretRef stripping, add regression coverage for those cases, then attach redacted real behavior proof.

## Reproduction Assessment

Yes, as a source-level reproduction: PI stores generated catalog request headers separately while emitted model objects can have `headers` unset, and current main passes that model object into `wrapStreamFn`. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: materializing sanitized catalog headers is the right narrow direction, but the patch should match PI's `models.json` parsing and header-source contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor JSONC models.json when materializing headers:** `src/agents/pi-model-discovery.ts:123`
  - body: PI accepts `models.json` after stripping line comments and trailing commas, then stores request headers for auth. This new side read uses raw `JSON.parse` and silently returns no header config for the same supported file, so `ctx.model.headers` stays missing while PI can still resolve and authenticate the model.
  - confidence: 0.87
- **[P2] Read modelOverride headers too:** `src/agents/pi-model-discovery.ts:140-152`
  - body: PI stores `providerConfig.modelOverrides[*].headers` as per-model request headers, but those headers are not copied onto the model object. This reader only scans `providerConfig.models`, so supported modelOverride headers still will not reach provider wrapper context.
  - confidence: 0.82
