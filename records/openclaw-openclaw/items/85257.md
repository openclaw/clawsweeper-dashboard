## Summary

Keep open: the durable WhatsApp handoff is the right direction for the linked message-loss bug, but the patch still has a blocking queue-update race that can replay already-handled batched messages.

## What This Changes

This PR adds an authDir-backed WhatsApp inbound queue, replays pending inbound records, delays read receipts until finalization, and updates related monitor tests.

## Best Possible Solution

Keep the durable handoff design, serialize all per-file queue updates, add focused batched/retry/restart monitor coverage, and make the authDir payload-retention boundary explicit before merge.

## Reproduction Assessment

Yes. Current main has a source-level ack-before-process path, and the PR's blocking race is source-reproducible from `Promise.all` over same-file durable queue read-modify-write operations with a re-entrant file lock.

## Solution Assessment

No, not as written. The durable queue approach is the right fix shape, but queue updates must be serialized or batched so the recovery mechanism cannot create duplicate delivery after restart.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Serialize durable queue updates before finalizing batches:** `extensions/whatsapp/src/inbound/monitor.ts:307-314`
  - body: `finalizeInboundEntries` completes every durable record in a batch through `Promise.all`, but each completion reads and rewrites the same `inbound-queue.json`. Since the plugin file lock is re-entrant in-process, two completions can read the same pending snapshot and the last write can leave another processed record pending; after restart that stale record can replay and send a duplicate reply. Serialize these queue mutations or merge the batch into one store update, and apply the same rule to the retry branch.
  - confidence: 0.91
