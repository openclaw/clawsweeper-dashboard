## Label Justifications

## Summary

Keep open: this PR targets a real P1 WhatsApp message-loss path and includes useful authDir proof, but the proposed patch has a blocking durable-store lost-update race when batched records complete in parallel.

## What This Changes

The PR adds an authDir-backed durable WhatsApp inbound queue, wires it into the WhatsApp inbox monitor before dispatch/replay, delays read receipts until finalization, and updates related monitor behavior tests.

## Best Possible Solution

Keep the durable queue approach, but serialize all queue read-modify-write mutations per store file and add a batched-completion regression test before merging.

## Reproduction Assessment

Yes at source level: current main marks WhatsApp messages read before handler completion and uses only memory-backed inbound dedupe, matching the linked ack-before-process loss mode. The PR-specific race is also source-reproducible from the parallel completion writes plus the re-entrant file-lock contract.

## Solution Assessment

No: the durable queue is the right owner-boundary fix, but this implementation needs serialized store mutations before it safely solves delivery loss.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Serialize durable queue completion writes:** `extensions/whatsapp/src/inbound/monitor.ts:307-313`
  - body: `finalizeInboundEntries` completes durable records with `Promise.all`, but each completion does a read-modify-write of the same `inbound-queue.json`. Because the plugin SDK file lock is re-entrant in-process, concurrent completions can read the same pending snapshot and the last writer can restore a completed record; restart replay can then duplicate inbound handling or replies.
