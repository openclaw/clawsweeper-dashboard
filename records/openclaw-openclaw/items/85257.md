## Label Justifications

- `P1`: The PR targets a real WhatsApp channel workflow where acknowledged inbound messages can be lost across a gateway crash before handler completion.
- `merge-risk: 🚨 message-delivery`: Merging changes WhatsApp inbound acknowledgement/finalization semantics and currently leaves a completed-duplicate path that can suppress read-receipt retries.

## Summary

Keep open: the durable WhatsApp handoff is the right direction, but the latest head still has a patch-level delivery regression where completed durable duplicates cannot retry failed read receipts.

## What This Changes

The PR adds an authDir-local WhatsApp inbound durable queue, replays pending inbound records on monitor attach, delays read receipts until finalization, serializes queue writes, and adds focused monitor/queue tests.

## Best Possible Solution

Keep the durable queue design, but preserve read-receipt retry behavior for completed duplicates before landing with targeted WhatsApp tests and explicit maintainer acceptance of authDir retention and fail-closed delivery semantics.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path where readMessages runs before enrichment, dedupe finalization, and onMessage delivery; the PR finding is also source-reproducible from the completed-duplicate early return.

## Solution Assessment

No, not yet. Durable queueing before handler dispatch is the right WhatsApp-owned boundary, but the implementation should keep a read-receipt retry path for completed duplicates before it is merge-ready.
