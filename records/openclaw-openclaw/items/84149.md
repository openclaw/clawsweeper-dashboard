## Summary

Keep this PR open: it targets a valid P1 session-lock regression, but the proposed epoch check can mask an external session-file mutation when any legitimate co-tenant lock acquisition also happened in the same prompt I/O window, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a process-local session write epoch counter and uses it in the embedded attempt session fence so same-process co-tenant writes during the prompt I/O window can be accepted, with new session-lock tests.

## Best Possible Solution

Keep the co-tenant progress fix, but make the fence prove or refresh only lock-coordinated file changes while still failing closed for any external mutation in the same prompt window.

## Reproduction Assessment

Yes. The linked issue has a clear source-level path on current main, and the PR bug is also source-reproducible: one extra same-process lock acquisition before reacquire makes currentEpoch exceed fenceWriteEpoch + 1 even if an external append also changed the file in that window.

## Solution Assessment

No. A process-local acquisition epoch is a useful signal, but this implementation treats acquisition count as ownership proof for all file changes and needs a safer lock-mediated generation or fingerprint refresh that cannot mask external mutation.
