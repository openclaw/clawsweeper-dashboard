## Summary

Keep open: this targets a real P1 session-lock regression, but the epoch shortcut knowingly weakens the takeover fence in mixed prompt windows and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a process-local session write epoch so co-tenant session-file writes during the prompt lock release can be accepted, plus focused session-lock tests.

## Best Possible Solution

Allow coordinated co-tenant progress while preserving takeover detection for unrelated same-window mutations, likely through append/content validation or another lock-mediated generation boundary reconciled with https://github.com/openclaw/openclaw/pull/84046.

## Reproduction Assessment

Yes by source inspection: current main records a prompt-window fingerprint and throws on any changed stat when reacquiring, while event and hook writers can legitimately use the same write lock during that window. I did not run a live heartbeat, cron, or channel-ingress reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the false-positive class, but the epoch only proves another in-process acquisition happened; it does not prove the changed bytes are coordinated, so append/content validation or an equivalent stronger boundary is safer.
