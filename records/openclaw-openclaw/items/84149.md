## Summary

Keep open: current source supports the linked session-stall bug, but this PR is not merge-ready because the epoch shortcut intentionally accepts a mixed-window external mutation case and no after-fix real behavior proof is present.

## What This Changes

The PR adds a process-local session write epoch, uses it to accept same-process co-tenant transcript writes during the prompt I/O lock-release window, and adds focused session-lock tests.

## Best Possible Solution

Allow co-tenant progress with a lock-mediated generation or fingerprint design that still detects uncoordinated writes in the same prompt window, backed by focused regression coverage and real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main records a fingerprint in `releaseForPrompt()` and throws on any changed fingerprint during reacquire, while registered co-tenant hooks can append through the same session write lock during that window. I did not run a live shared-session heartbeat or channel-ingress reproduction in this read-only review.

## Solution Assessment

No. The target behavior is right, but this implementation accepts the same-window external-bypass case and needs either a safer lock-mediated design or an explicit maintainer decision to own that tradeoff.
