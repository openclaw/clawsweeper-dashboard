## Summary

Keep open: the reported session-lock regression is real enough by source inspection, but this PR is not merge-ready because the epoch shortcut weakens same-window takeover detection and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a process-local session write epoch, uses it to re-baseline the embedded prompt fence after co-tenant lock acquisitions, and adds session-lock tests for co-tenant and external write cases.

## Best Possible Solution

Allow coordinated co-tenant progress while preserving detection of same-window external mutations, likely by validating the appended session content or another lock-mediated generation boundary, and reconcile this with https://github.com/openclaw/openclaw/pull/84046 before merge.

## Reproduction Assessment

Yes by source inspection: current main snapshots the session-file fingerprint at `releaseForPrompt()` and throws on any stat mismatch when the controller reacquires, while installed event and hook writers can use the same write lock during the prompt window. I did not run a live heartbeat, cron, or channel-ingress reproduction in this read-only review.

## Solution Assessment

No. The PR addresses the false-positive class, but a process-local acquisition epoch is not the best safety boundary because it cannot distinguish coordinated bytes from unrelated external mutation in the same window; append/content validation like the related fence design is safer.
