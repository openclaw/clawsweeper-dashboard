## Summary

Close as cannot reproduce: current main and the v2026.5.7 release source both route non-DM Google Chat spaces as group peers, and the shared session-key builder only falls back to the agent main session for direct-message peers. The earlier same-author report at https://github.com/openclaw/openclaw/issues/79395 was already closed with the same source/release finding, while the remaining open Google Chat thread metadata problem is tracked separately.

## What This Changes

The issue reports that Google Chat group messages create `agent:<agentId>:main` instead of `agent:<agentId>:googlechat:group:<spaceId>`, causing replies to route through WhatsApp.

## Best Possible Solution

Keep the current Google Chat group routing implementation; if a deployment still logs a main session key, handle it as a new runtime/package-provenance report with raw routing logs, exact installed artifact, and commit/version proof.

## Reproduction Assessment

No. The live steps require external Google Chat and WhatsApp credentials, but the current and v2026.5.7 source path contradicts the claimed failure: non-DM Google Chat spaces are passed as group peers, and group peers cannot build the agent main session key through the shared builder.
