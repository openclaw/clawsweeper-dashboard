## Label Justifications

- `P2`: This is a bounded user-facing export bug with misleading session artifacts, but no evidence of a core runtime outage or security emergency.
- `impact:data-loss`: The generated export silently omits assistant replies, tool calls, usage, and cost data from the session artifact.
- `impact:session-state`: The issue is about drift between OpenClaw's session transcript and backend-owned transcript state for the same conversation.

## Summary

This should stay open: current main and v2026.6.6 still export only the OpenClaw transcript without a backend-delegated warning, and https://github.com/openclaw/openclaw/pull/90867 is the active closing implementation candidate.

## What This Changes

The issue asks `/export-session` to stop producing misleading CLI/ACP-backed exports that contain only user messages and no assistant, tool, usage, or cost data.

## Best Possible Solution

Review and land a focused fix that at minimum warns in the command reply and exported HTML when the OpenClaw transcript only contains user messages, leaving full backend transcript merging as separate product work if needed.

## Reproduction Assessment

Yes for a source-level path: current main exports exactly the entries read from OpenClaw's session file, and a CLI/ACP user-only transcript will be encoded and reported as successful without a backend warning. I did not run a live CLI/ACP backend reproduction in this read-only review.

## Solution Assessment

Yes for the minimum fix path: warning in the command reply and exported HTML is the narrowest maintainable mitigation. Full backend transcript merging would be broader and can remain separate if maintainers want richer exports later.
