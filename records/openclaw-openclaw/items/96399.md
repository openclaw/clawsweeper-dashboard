## Summary

No blocking findings. This PR fixes a source-reproducible byte-vs-character limit mismatch in the shared parser and has sufficient terminal proof, with a small compatibility decision around intentionally stricter non-ASCII payload caps before merge.

## What This Changes

The branch adds UTF-8 byte counting to tool-call repair JSON/XML-ish payload limits and covers non-ASCII cases with regression tests.

## Best Possible Solution

Merge this parser-boundary fix after maintainers accept the stricter byte-cap semantics and required CI is green.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.10` compare UTF-16/string offsets against `maxPayloadBytes` in the JSON and XML-ish parser paths. A multi-byte payload can therefore exceed its UTF-8 byte cap while still passing current code.

## Solution Assessment

Yes. The shared parser package owns the `maxPayloadBytes` contract consumed by SDK and provider repair paths, so fixing it there is narrower and cleaner than duplicating byte-limit policy in downstream wrappers.
