## Summary

Keep open for maintainer review: the patch appears correct and well-proven, but it intentionally tightens shipped parser behavior for non-ASCII repaired tool-call payloads, so maintainers should accept that compatibility tradeoff before merge.

## What This Changes

The PR changes the shared tool-call repair parser to enforce `maxPayloadBytes` using UTF-8 byte counts for JSON and XML-ish plaintext tool-call payloads, with focused non-ASCII regression coverage.

## Best Possible Solution

Merge the shared parser fix after maintainers accept the stricter byte-cap semantics and required CI is green.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.10` compare string offsets against `maxPayloadBytes` in the JSON and XML-ish parser paths, so multi-byte payloads can exceed the intended UTF-8 byte cap.

## Solution Assessment

Yes. The shared repair parser owns the `maxPayloadBytes` contract used by SDK and provider repair paths, so fixing byte accounting there is narrower than adding downstream wrapper checks; the remaining question is maintainer acceptance of stricter behavior.
