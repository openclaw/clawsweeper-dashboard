## Summary

Keep this PR open as a valid focused bug fix. Current main and the latest release still count `maxPayloadBytes` with UTF-16/string offsets, while the branch moves JSON and XML-ish parser checks to UTF-8 byte accounting; the earlier proof blocker is addressed by copied live parser output in the PR body, with only a transient dependency-guard rerun still needed.

## What This Changes

The branch changes tool-call repair to count JSON and XML-ish `maxPayloadBytes` limits with UTF-8 byte length and adds non-ASCII regression tests.

## Best Possible Solution

Merge this parser-boundary fix after dependency-guard reruns cleanly, keeping UTF-8 byte-limit enforcement centralized in `packages/tool-call-repair`.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.10` compare UTF-16/string offsets against `maxPayloadBytes` in the JSON and XML-ish parser paths. A multi-byte payload can therefore exceed its UTF-8 byte cap while still passing current code.

## Solution Assessment

Yes, this is the best fix shape: the shared parser package owns the byte-limit contract used by SDK and provider repair paths. A downstream wrapper fix would duplicate limit policy and leave sibling parser callers inconsistent.
