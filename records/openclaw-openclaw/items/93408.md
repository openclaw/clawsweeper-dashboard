## Summary

Keep open: current main and the latest release still drop WhatsApp inbound access denials after verbose-only denial logs, and the requested default-visible signal is not implemented. The remaining choice is product/security shape, not whether the source report is valid.

## What This Changes

Request a default-visible, non-spammy WhatsApp diagnostic for inbound messages blocked by DM/group access policy without changing the access decision.

## Best Possible Solution

Add a WhatsApp-owned redacted, rate-limited denial diagnostic or agreed diagnostic event for blocked access decisions, while preserving current fail-closed authorization behavior and coordinating with the broader unauthorized-inbound audit direction.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, every relevant WhatsApp denial path logs through `logWhatsAppVerbose`, and the monitor returns before routing blocked messages. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as proposed: the observability goal is sound, but maintainers should choose the permanent shape, especially info log versus structured diagnostic event and the redaction/rate-limit contract.
