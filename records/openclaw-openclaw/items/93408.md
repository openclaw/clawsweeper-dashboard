## Summary

Keep open: current main and the latest release still gate WhatsApp access-denial logs behind `verbose`, so the requested default-visible denial signal is not implemented. The report is valid, but the remaining work needs a maintainer security/product decision on redaction, rate limiting, and whether to solve it as a WhatsApp log or shared unauthorized-inbound audit surface.

## What This Changes

The issue requests a default-visible, non-spammy signal when WhatsApp inbound access control drops DM or group messages that are currently only logged in verbose mode.

## Best Possible Solution

Add a WhatsApp-owned redacted, rate-limited denial diagnostic or an agreed shared unauthorized-inbound security event while preserving the current fail-closed access decisions.

## Reproduction Assessment

Yes, at source level: with `verbose` unset or false, the WhatsApp DM/group denial branches call a helper that returns before logging and then return blocked access before routing. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a product choice: a rate-limited info log would solve the immediate operator visibility gap, but maintainers should choose whether the permanent shape is a WhatsApp-local diagnostic or a shared structured unauthorized-inbound audit event.
