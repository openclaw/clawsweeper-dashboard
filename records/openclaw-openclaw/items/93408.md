## Summary

Keep open: current main and v2026.6.8 still route WhatsApp access-control denial messages through the verbose-only log helper and drop the inbound before dispatch. The issue is source-reproducible, but the fix needs maintainer product/security direction on redaction, rate limiting, and whether the signal is WhatsApp-local or a shared unauthorized-inbound audit event.

## What This Changes

The issue asks for default-visible observability when WhatsApp inbound messages are denied by allowlist or policy checks instead of only logging those drops when verbose mode is enabled.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves fail-closed authorization, or explicitly route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
