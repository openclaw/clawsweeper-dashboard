## Summary

Keep open: current main and v2026.6.8 still route WhatsApp access-control denial messages through the verbose-only logger, so default-config allowlist/policy drops remain silent. The report is source-reproducible, but the permanent fix needs maintainer security/product direction on redaction, rate limiting, and whether this should be a WhatsApp log or a shared unauthorized-inbound audit/status event.

## What This Changes

Request default-visible, non-spammy diagnostics for WhatsApp inbound access-control denials without changing the authorization decision.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic using existing ingress reason metadata, or route the work into a shared unauthorized-inbound security/audit surface while preserving current fail-closed authorization behavior.

## Reproduction Assessment

Yes, source-reproducible: default verbose is falsy, the relevant WhatsApp denial branches only call logWhatsAppVerbose, and the monitor returns before dispatch when access is blocked. No live WhatsApp account was run in this read-only review.

## Solution Assessment

Unclear as a final implementation: the visibility goal is sound, but maintainers need to choose between a WhatsApp-owned rate-limited log and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
