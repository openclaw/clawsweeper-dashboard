## Summary

Keep open: current main and the latest release still gate WhatsApp allowlist/policy denial diagnostics behind `verbose`, so default-config access-control drops remain silent. The issue is source-reproducible, but the durable fix needs maintainer product/security direction on redaction, rate limiting, and whether the signal is WhatsApp-local or a shared unauthorized-inbound audit/status event.

## What This Changes

The issue requests a default-visible, non-spammy diagnostic for WhatsApp inbound access-control drops without changing authorization decisions.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves fail-closed authorization behavior, or deliberately route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
