## Summary

Keep open: current main and v2026.6.8 still route WhatsApp access-control denials through verbose-only logging before dropping the inbound message. The remaining decision is security/product shape, not source uncertainty: maintainers need to choose redaction, rate limiting, and WhatsApp-local log versus shared unauthorized-inbound audit/status handling.

## What This Changes

The issue requests default-visible, rate-limited observability for WhatsApp inbound access-control denials that are currently silent unless verbose logging is enabled.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic using existing ingress reason metadata, or explicitly route the work into a shared unauthorized-inbound audit/status surface while preserving the current access decisions.

## Reproduction Assessment

Yes. At source level, the relevant WhatsApp denial branches only call logWhatsAppVerbose, and the monitor returns before dispatch when access is blocked; no live WhatsApp account was run in this read-only review.

## Solution Assessment

Unclear as a final implementation. A rate-limited info log is plausible, but a structured shared unauthorized-inbound audit event may be the better long-term shape because related Telegram work raises the same security-observability need.
