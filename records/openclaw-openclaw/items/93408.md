## Summary

Keep open: current main and the latest release still make the reported WhatsApp access-denial diagnostics verbose-only, and the requested default-visible signal is not implemented. The remaining blocker is choosing the security/privacy and rate-limit shape, not proving the source report.

## What This Changes

The issue requests a default-visible, non-spammy diagnostic when WhatsApp inbound messages are denied by DM or group access policy.

## Best Possible Solution

Add a WhatsApp-owned redacted, rate-limited denial diagnostic or agreed diagnostic event for blocked access decisions, while preserving current fail-closed authorization behavior and coordinating with https://github.com/openclaw/openclaw/issues/77546.

## Reproduction Assessment

Yes, at source level: default `verbose` suppresses every relevant WhatsApp access-denial log, and blocked messages return before routing. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as proposed: the observability goal is sound, but maintainers should choose the durable shape, especially info log versus structured diagnostic event and the redaction/rate-limit contract.
