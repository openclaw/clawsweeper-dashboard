## Summary

Keep open: current main and the latest release still drop WhatsApp DM/group policy denials before dispatch with only verbose-gated diagnostics, so default-config operators still cannot see why inbound messages were ignored. The requested visibility is valid, but the final shape needs maintainer product/security direction on redaction, rate limiting, and whether this belongs in WhatsApp logging or a shared unauthorized-inbound audit/status path.

## What This Changes

Add default-visible, privacy-conscious diagnostics when WhatsApp inbound messages are denied by DM/group policy or sender allowlists.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves the existing fail-closed authorization behavior, or deliberately route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, source-reproducible: configure WhatsApp so `dmPolicy` or `groupPolicy` denies an inbound message while `verbose` is false, and the denial returns before downstream processing with only verbose-gated log calls. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
