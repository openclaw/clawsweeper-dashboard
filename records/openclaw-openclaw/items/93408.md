## Summary

Keep open: current main and v2026.6.8 still only emit WhatsApp access-denial diagnostics through verbose logging, so default-config policy and allowlist drops remain silent. The issue is valid, but the fix needs maintainer product/security direction on redaction, rate limiting, and whether this stays WhatsApp-owned or becomes a shared unauthorized-inbound audit surface.

## What This Changes

Request default-visible, non-spammy observability for WhatsApp inbound messages dropped by DM/group policy or sender allowlists.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves fail-closed authorization, or explicitly route the work into a shared unauthorized-inbound audit/status event.

## Reproduction Assessment

Yes, at source level: default verbose is falsy, WhatsApp denial branches only call logWhatsAppVerbose, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
