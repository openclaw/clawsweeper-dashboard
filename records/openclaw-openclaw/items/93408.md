## Summary

Keep open: current main and the latest release still gate WhatsApp access-control denial diagnostics behind `verbose`, so default-config allowlist and disabled-policy drops remain silent. The report is valid, but the permanent shape needs maintainer product/security direction on redaction, rate limiting, and whether this stays WhatsApp-local or becomes a shared unauthorized-inbound audit/status event.

## What This Changes

Request default-visible, non-spammy WhatsApp diagnostics when inbound messages are denied by DM or group access-control policy.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves the existing fail-closed authorization behavior, or explicitly route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
