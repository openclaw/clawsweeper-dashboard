## Summary

Keep open: current main and the latest release still gate WhatsApp access-denial diagnostics behind `verbose`, so default-config allowlist and policy drops remain silent. The report is source-reproducible, but the fix needs maintainer security/product direction on redaction, rate limiting, and whether the signal is WhatsApp-local or a shared unauthorized-inbound audit event.

## What This Changes

Request a default-visible, bounded WhatsApp diagnostic or audit signal when inbound sender or group policy denies a message instead of logging only in verbose mode.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves fail-closed authorization, or route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
