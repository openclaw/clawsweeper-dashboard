- `P2`: This is a source-proven WhatsApp observability and security improvement with limited channel-specific blast radius, not an urgent outage or authorization bypass.
- `impact:security`: The issue concerns visibility and metadata handling at the WhatsApp inbound sender and group-policy authorization boundary.
- `impact:message-loss`: Denied WhatsApp messages are dropped before dispatch and currently have no default-visible reason for operators.

## Summary

Keep open: current main and the latest release still gate WhatsApp inbound denial diagnostics behind `verbose`, so default-config allowlist and policy drops remain silent. The request is valid, but the durable fix needs maintainer product/security direction on redaction, rate limiting, and whether the signal should be WhatsApp-local or a shared unauthorized-inbound diagnostic event.

## What This Changes

Request a default-visible, non-spammy diagnostic for WhatsApp inbound messages dropped by DM/group access-control policy or allowlists.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses existing ingress reason metadata and preserves the fail-closed authorization behavior, or explicitly route this into a shared unauthorized-inbound security event surface.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers need to choose between a WhatsApp-owned rate-limited diagnostic and a structured shared unauthorized-inbound security event with explicit redaction rules.
