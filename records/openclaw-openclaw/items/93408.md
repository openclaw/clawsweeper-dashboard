## Label Justifications

- `P2`: The issue reports a bounded WhatsApp access-control observability gap with security and dropped-message diagnosis impact.
- `impact:security`: The silent denials hide allowlist and disabled-policy decisions that operators use as a security boundary.
- `impact:message-loss`: Blocked inbound messages are intentionally dropped before reaching the agent, and the operator receives no default-visible reason.

## Summary

Keep open: current main and the latest release still only emit the relevant WhatsApp access-denial messages through verbose-gated logging, and the requested default-visible denial signal has not been implemented. The remaining blocker is maintainer product/security direction for the redaction, rate-limit, and info-log versus structured-event shape.

## What This Changes

The issue asks for default-visible, rate-limited or structured diagnostics when WhatsApp inbound messages are denied by access control.

## Best Possible Solution

Add a WhatsApp-owned redacted, rate-limited denial diagnostic or agreed diagnostic event for blocked access decisions, while preserving current fail-closed authorization behavior and coordinating with the broader unauthorized-inbound audit direction.

## Reproduction Assessment

Yes, at source level: current main only emits the relevant WhatsApp denial messages through a verbose-gated helper, and the monitor drops blocked messages before routing. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as proposed: the observability goal is sound, but maintainers should choose the permanent shape, especially info log versus structured diagnostic event and the redaction/rate-limit contract.
