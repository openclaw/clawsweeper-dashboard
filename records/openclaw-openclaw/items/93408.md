## Summary

Keep open: current main and latest release still gate WhatsApp access-control denial diagnostics behind verbose logging, so default-config allowlist and policy drops remain silent. The issue is valid, but the fix needs maintainer product/security direction on redaction, rate limiting, and whether this should be WhatsApp-local logging or a shared unauthorized-inbound audit/status event.

## What This Changes

Add default-visible observability for WhatsApp inbound messages dropped by allowlist or access-control policy without changing the authorization decision.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic that reuses shared ingress reason metadata, or deliberately fold the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, at source level: default gateway verbose is false, current WhatsApp denial branches only call the verbose-gated helper, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
