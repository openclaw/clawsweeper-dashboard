## Summary

Keep open: current main and the latest release still gate WhatsApp access-denial logs behind `verbose`, so default-config allowlist and policy drops remain silent. The issue is valid, but the permanent fix needs maintainer product/security direction on redaction, rate limiting, and whether this belongs as a WhatsApp log or a shared unauthorized-inbound audit signal.

## What This Changes

Request a default-visible, non-spammy diagnostic for WhatsApp inbound messages denied by sender allowlist or DM/group policy checks.

## Best Possible Solution

Define a redacted, bounded denial diagnostic for WhatsApp sender/policy blocks, preferably reusing shared ingress reason metadata and preserving the existing fail-closed authorization behavior.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited info log and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
