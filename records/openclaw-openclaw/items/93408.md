## Summary

Keep open: current main and v2026.6.9 still gate WhatsApp DM/group access-denial diagnostics behind verbose logging before dropping the message, and the fix needs maintainer security/product direction on redaction, rate limiting, and WhatsApp-local logging versus a shared unauthorized-inbound audit surface.

## What This Changes

Add a default-visible, rate-limited or structured diagnostic for WhatsApp inbound access-control denials without changing the authorization decision.

## Best Possible Solution

Define a redacted, rate-limited WhatsApp denial diagnostic using shared ingress reason metadata, or route it into a maintainer-approved shared unauthorized-inbound audit/status surface while preserving existing fail-closed behavior.

## Reproduction Assessment

Yes, at source level. Current main shows WhatsApp access-control denial paths return before dispatch and only log through a verbose-gated helper; I did not run live WhatsApp traffic in this read-only review.

## Solution Assessment

Unclear as a final implementation. The visibility goal is valid, but maintainers should choose between a WhatsApp-owned rate-limited info diagnostic and a structured/shared unauthorized-inbound audit surface with explicit redaction rules.
