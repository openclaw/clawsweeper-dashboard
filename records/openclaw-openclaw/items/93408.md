## Summary

Keep open: current main and v2026.6.8 still gate WhatsApp access-denial diagnostics behind `verbose`, so default-config allowlist and policy drops remain silent. The issue is valid, but the durable fix needs maintainer product/security direction on redaction, rate limiting, and whether this is a WhatsApp-local log or shared unauthorized-inbound audit signal.

## What This Changes

The issue requests default-visible, bounded observability when WhatsApp inbound access control drops DMs or group messages.

## Best Possible Solution

Define a bounded unauthorized-inbound diagnostic for WhatsApp sender and policy blocks, preserving the fail-closed access decision while redacting metadata and suppressing repeated denials by account, conversation, and reason.

## Reproduction Assessment

Yes, at source level: the reported denial paths are reachable in `checkInboundAccessControl`, each reported log line is guarded by `logWhatsAppVerbose(params.verbose, ...)`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: a rate-limited info log is plausible, but maintainers should choose between a WhatsApp-owned diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
