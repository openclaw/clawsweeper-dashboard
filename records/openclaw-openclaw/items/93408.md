- `impact:security`: The issue concerns visibility into WhatsApp authorization denials at the inbound sender and group-policy boundary.

## Summary

Keep open: current main and the latest release still route WhatsApp allowlist and policy denial messages through the verbose-only logger, so default-config drops remain silent. The issue is source-reproducible, but the permanent fix needs maintainer security/product direction on redaction, rate limiting, and WhatsApp-local logging versus a shared unauthorized-inbound audit signal.

## What This Changes

Requests default-visible, bounded WhatsApp inbound access-denial diagnostics for allowlist and policy drops without changing authorization decisions.

## Best Possible Solution

Define a redacted, bounded denial diagnostic for WhatsApp sender and policy blocks, preferably reusing shared ingress reason metadata while preserving the existing fail-closed authorization behavior.

## Reproduction Assessment

Yes, at source level: default `verbose` is falsy, the relevant WhatsApp denial branches only call `logWhatsAppVerbose`, and the monitor returns before dispatch when access is blocked. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited info log and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
