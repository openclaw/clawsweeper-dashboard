## Summary

Keep open: current main and v2026.6.9 still route WhatsApp DM/group access-control denials through verbose-only logging before returning a blocked result, so default-config operators still lack a visible reason for policy drops. The problem is valid, but the fix needs maintainer direction on privacy, rate limiting, and whether this should be WhatsApp-local logging or a shared unauthorized-inbound audit/status path.

## What This Changes

The issue requests default-visible, rate-limited or structured diagnostics when WhatsApp inbound messages are denied by WhatsApp DM/group access policy instead of only verbose-gated logs.

## Best Possible Solution

Define a redacted, bounded WhatsApp denial diagnostic that reuses shared ingress reason metadata and preserves current authorization behavior, or deliberately route the work into a shared unauthorized-inbound audit/status surface.

## Reproduction Assessment

Yes, source-reproducible: configure WhatsApp so `dmPolicy` or `groupPolicy` denies an inbound message while `verbose` is false, and the denial returns before downstream processing with only verbose-gated log calls. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Unclear as a final implementation: the observability goal is sound, but maintainers should choose between a WhatsApp-owned rate-limited diagnostic and a structured/shared unauthorized-inbound audit event with explicit redaction rules.
