## Summary

Keep open. Current main and the latest release still use strict minScore-gated dreaming promotion with no wildcard or serendipity lane, and the requested random candidate policy is a memory-product enhancement rather than cleanup eligible for closure.

## What This Changes

The issue requests a wildcardRatio-style deep-phase promotion lane so sub-threshold but gate-passing short-term memories can occasionally be promoted.

## Best Possible Solution

If maintainers want this behavior, add a bounded and explainable wildcard lane in memory-core that preserves hard gates, threads policy through ranking and apply, records promotion reasons, and documents the chosen default/config surface.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source, docs, and tests show the current behavior still strictly threshold-gates deep promotion candidates.

## Solution Assessment

Unclear. A bounded wildcard lane may fit the request, but the default randomness, config surface, and apply-time policy need maintainer product judgment before implementation.
