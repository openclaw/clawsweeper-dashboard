## Summary

Keep open. Current main and v2026.5.19 still use strict threshold-gated deep promotion with no wildcard/serendipity lane, and the requested random sub-threshold promotion policy is a product/config decision rather than conservative cleanup.

## What This Changes

The issue requests adding a configurable wildcard lane so some sub-threshold deep dreaming candidates can still be sampled into promotion ranking.

## Best Possible Solution

If maintainers want this behavior, add a bounded and explainable wildcard lane in memory-core that preserves hard gates, threads policy through ranking and apply, records promotion reasons, and documents the chosen default/config surface.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an enhancement request. Source, docs, release inspection, and tests show the current behavior still strictly threshold-gates deep promotion candidates.

## Solution Assessment

Unclear. A bounded wildcard lane may fit the request, but default randomness, config surface, apply-time policy, and explanation requirements need maintainer product judgment before implementation.
