## Summary

Keep open: the linked credential-exposure bug is still real on current main and v2026.5.22, but this draft only wires `connect`, has no tests or real macOS proof, and the new env-source path can silently fall back to stored credentials.

## What This Changes

Adds a shared Swift macOS CLI secret-input parser and wires `openclaw-mac connect` to accept password stdin/file/env flags while retaining inline `--password` with a warning.

## Best Possible Solution

Finish one safe-secret contract across `connect`, `configure-remote`, and `wizard`, make explicit secret sources fail when missing or unreadable, preserve legacy inline flags with warnings, and add focused Swift tests plus real macOS CLI proof before merge.

## Reproduction Assessment

Yes. Current-main source shows `connect` and `configure-remote` taking `--password` from `CommandLine.arguments`; a long-running invocation with that flag exposes the secret through argv/process listing without needing a live macOS run to understand the path.

## Solution Assessment

No. The direction is right, but this draft is not the best complete fix until it covers the remaining password-taking commands, handles missing explicit secret sources safely, and proves the macOS CLI behavior with tests or real output.
