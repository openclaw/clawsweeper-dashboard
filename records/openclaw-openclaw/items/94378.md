## Summary

Keep open: current main still fails strict mixed valid/invalid image responses, and the PR is a focused plausible fix, but the changed helper is SDK-exported so partial-success strict parsing needs maintainer acceptance before merge.

## What This Changes

The PR changes strict OpenAI-compatible image response parsing to skip malformed entries when at least one valid image remains and adds regression tests for mixed and all-invalid batches.

## Best Possible Solution

Land this narrow mixed-entry parser behavior only after maintainers accept partial-success strict parsing, while keeping URL/object/base64url proxy-response variants on separate review paths.

## Reproduction Assessment

Yes at source level. Current main throws inside the strict parser loop on the first invalid entry when malformedResponseError is set, and the production OpenAI-compatible provider supplies that option.

## Solution Assessment

Yes for the mixed b64_json batch behavior: deferring the strict error until after the loop is the narrow shared-helper fix. The remaining question is maintainer acceptance of the exported strict-mode compatibility change, not a code-shape defect.
