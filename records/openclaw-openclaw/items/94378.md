## Summary

Keep open: current main and the latest shipped tag still fail fast on the first invalid strict-mode image entry, and this PR contains a narrow unique fix for mixed valid/invalid `b64_json` batches. The patch looks correct for that behavior, but merging changes an SDK-exported parser contract from fail-fast to partial success, so maintainer acceptance is still required.

## What This Changes

The PR changes `parseOpenAiCompatibleImageResponse()` to skip invalid per-entry `b64_json` values in strict mode, throw only when a non-empty response has no parsable images, and adds regression tests for mixed and all-invalid batches.

## Best Possible Solution

Land this narrow mixed-entry parser behavior only after maintainers accept partial-success strict parsing, while keeping URL/object/base64url proxy-response variants on their separate review paths.

## Reproduction Assessment

Yes at source level. Current main and `v2026.6.8` throw inside the strict parser loop on the first invalid entry, and the PR body includes before/after terminal output for a mixed valid/invalid batch.

## Solution Assessment

Yes for the mixed `b64_json` batch behavior: deferring the strict error until after the loop is the narrow shared-helper fix. The remaining question is maintainer acceptance of the exported strict-mode compatibility change, not a code-shape defect.
