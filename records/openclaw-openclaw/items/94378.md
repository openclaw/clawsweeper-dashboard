## Summary

Keep open. The patch is a narrow source-level fix for mixed valid/invalid OpenAI-compatible `b64_json` arrays, but the external PR still has only parser/unit-test proof and no real proxy or CLI image-generation run showing the changed behavior.

## What This Changes

The PR changes `parseOpenAiCompatibleImageResponse()` to skip invalid entries in strict mode, throw only when a non-empty response yields no parsed images, and adds regression tests for mixed and all-invalid batches.

## Best Possible Solution

Land the narrow mixed-entry parser fix after redacted real proxy or CLI proof, or after explicit maintainer acceptance of parser-boundary validation, while keeping URL-backed, object-format, and URL-safe base64 work on the adjacent PRs.

## Reproduction Assessment

Yes at source level: current main throws during strict per-entry parsing when any entry is invalid, so a mixed response cannot return its valid images. I did not run artifact-creating tests or a live proxy flow in this read-only review.

## Solution Assessment

Yes for the mixed `b64_json` array behavior: deferring the strict error until after the loop is the narrowest fix I found. It is not the complete NewAPI URL/object/base64 compatibility fix, which remains adjacent work.
