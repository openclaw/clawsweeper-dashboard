## Summary

Keep open: the narrow parser patch looks correct for mixed valid/invalid OpenAI-compatible `b64_json` arrays, but this external PR still lacks real behavior proof from a live proxy or CLI image-generation run showing the changed path after the patch.

## What This Changes

The PR changes `parseOpenAiCompatibleImageResponse()` to skip invalid entries in strict mode, throw only when a non-empty batch yields no parsed images, and adds strict-mode regression tests.

## Best Possible Solution

Land the narrow mixed-entry parser fix after real behavior proof, or after explicit maintainer acceptance that parser-boundary validation is sufficient for this external PR.

## Reproduction Assessment

Yes at source level: current main throws in strict mode when any `data[]` entry is not parsable, so a mixed batch cannot return its valid entries. I did not run artifact-creating tests or a live provider flow during this read-only review.

## Solution Assessment

Yes for mixed `b64_json` arrays: deferring the strict error until after the loop is the narrowest maintainable parser fix. URL-only responses, object-format data, and URL-safe base64 are adjacent work on separate PRs.
