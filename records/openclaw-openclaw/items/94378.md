## Summary

This focused PR is still useful because current main and the latest release still fail fast on the first invalid strict-mode image entry; the remaining blocker is maintainer acceptance of changing an SDK-exported parser helper from fail-fast strict parsing to partial success for mixed valid/invalid batches.

## What This Changes

The PR changes `parseOpenAiCompatibleImageResponse()` to skip invalid entries in strict mode and throw only when a non-empty response yields no parsed images, with regression coverage for mixed and all-invalid batches.

## Best Possible Solution

Land the narrow mixed-entry parser behavior after maintainers accept partial-success strict parsing, and keep adjacent proxy response cases on their existing PRs.

## Reproduction Assessment

Yes at source level: current main throws inside the strict parser loop on the first invalid entry, and the PR body includes before/after terminal output for a mixed batch. I did not run the repro in this read-only review.

## Solution Assessment

Yes for the mixed `b64_json` batch behavior: deferring the strict error until after the loop is the narrowest shared-helper fix found. It is not the complete proxy compatibility fix for URL-backed or object-format responses.
