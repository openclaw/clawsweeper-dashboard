## Summary

Keep open: current main and v2026.6.8 still fail a strict mixed valid/invalid `data[]` batch before later valid images are processed, and the patch is a narrow source-correct parser fix with focused tests. The remaining blocker is that the external PR proof is still test-only, so it needs real behavior proof from an OpenAI-compatible image-generation run or explicit maintainer acceptance of parser-boundary proof before merge.

## What This Changes

The PR changes the shared OpenAI-compatible image response parser to skip invalid `data[]` entries in strict mode, throw only when a non-empty batch yields no images, and adds regression tests.

## Best Possible Solution

Land the narrow mixed-entry parser fix after real behavior proof or explicit maintainer acceptance of parser-boundary validation, while keeping URL-only response downloads on https://github.com/openclaw/openclaw/pull/81443.

## Reproduction Assessment

Yes at source level: current main throws in strict mode when a bad `data[]` entry appears before later valid `b64_json` entries. I did not run a live proxy reproduction because this read-only review cannot run artifact-creating tests or live provider flows.

## Solution Assessment

Yes for mixed `b64_json` batches: deferring the strict error until after the parser sees whether any entry was valid is the narrowest maintainable fix. URL-only response fetching is a different guarded-download problem and should stay on https://github.com/openclaw/openclaw/pull/81443.
