## Summary

Keep open. The PR targets a real Feishu group activation bug, but the current patch can trap a group in mention mode because unmentioned `/activation always` is blocked before the shared command handler can update persisted session state; it also still has only source/mock proof.

## What This Changes

Adds Feishu session `groupActivation` lookup before group admission and new unit plus `handleFeishuMessage` regression tests for activation overriding static `requireMention`.

## Best Possible Solution

Keep the Feishu session-activation precedence, but make the first group gate command-aware, avoid forced uncached session reads on the hot path, align docs, and add redacted live Feishu proof.

## Reproduction Assessment

Yes. The linked Feishu bug is source-reproducible because current main writes `groupActivation` but Feishu group admission never reads it; the PR's escape-path bug is also source-reproducible from the new override running before command authorization facts exist.

## Solution Assessment

No. Reading session activation in Feishu admission is the right boundary, but this implementation must preserve the `/activation always` recovery path and avoid an uncached group-message session read before it is the best fix.
