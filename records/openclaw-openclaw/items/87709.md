## Summary

Keep open. The Feishu bug is valid and the PR is pointed at the right admission boundary, but the current branch is not merge-ready against current main because it imports a stale session API, reads activation before binding route rewrites, can trap users in mention mode, adds an uncached hot-path store read, still needs docs alignment, and has only mock/source proof.

## What This Changes

The PR adds Feishu group activation helpers and tests, and changes `handleFeishuMessage` to apply persisted `groupActivation` over static mention-required config before group admission.

## Best Possible Solution

Repair the Feishu plugin so effective activation is read from the same finalized route/session used for dispatch, let authorized activation commands bypass mention gating, use the current session-entry seam without an uncached hot-path whole-store read, align docs, and require redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. Current main persists `/activation mention` to `groupActivation`, while Feishu group admission still derives the hard mention gate from config-only `requireMention` before dispatch.

## Solution Assessment

No. Reading session activation in Feishu admission is the right boundary, but this branch is not the best fix until it uses the current session API, reads after route binding finalization, preserves activation-command recovery, avoids the uncached hot-path read, and updates docs/proof.

  - body: This lookup happens before Feishu applies configured/runtime conversation binding route rewrites later in `handleFeishuMessage`. Those rewrites can replace `route.sessionKey`, so bound group/topic conversations will read activation from the default route but persist commands to the bound session, leaving `/activation mention` ineffective for that path.
  - confidence: 0.87
- **[P1] Let activation commands escape mention mode:** `extensions/feishu/src/bot.ts:732-735`
  - body: After this override turns stored `mention` into `requireMention: true`, Feishu still calls the first group gate with `allowTextCommands: false`. An unmentioned authorized `/activation always` message will be skipped before the shared activation handler can persist the switch back, so users can get stuck in mention mode unless they mention the bot.
  - confidence: 0.92
- **[P2] Avoid uncached session loads in group admission:** `extensions/feishu/src/group-activation.ts:24`
  - body: This helper forces `loadSessionStore(..., { skipCache: true })` for every Feishu group message before admission. That bypasses the normal session-store cache on a hot path; use the current cached session-entry helper or carry the prepared activation fact forward unless there is a tested freshness contract requiring a disk read here.
  - confidence: 0.86
- **[P2] Align Feishu activation docs:** `extensions/feishu/src/bot.ts:703-708`
  - body: The PR makes Feishu honor `/activation mention|always`, but current user-facing group docs still say other surfaces ignore `/activation` and the Feishu page only documents static `requireMention`. Please update the docs or get an explicit maintainer decision to leave that user-visible mismatch for follow-up.
  - confidence: 0.8
