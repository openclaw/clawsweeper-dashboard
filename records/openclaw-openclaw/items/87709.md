## Summary

Keep open. The PR targets a valid Feishu session-activation bug, but the current patch can filter an authorized unmentioned `/activation always` before the command handler can switch the group back, adds an uncached session-store read to the group-message hot path, leaves Feishu activation docs inconsistent, and still lacks real Feishu behavior proof.

## What This Changes

The PR changes Feishu group admission to read persisted session `groupActivation` and apply it over static `requireMention` config, with new regression tests.

## Best Possible Solution

Keep the intended Feishu session-activation precedence, but make the first admission gate command-aware, use a cached single-entry/prepared activation read instead of forced uncached whole-store loading, align Feishu/group docs, and require redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, source-level. The linked issue steps match current main, where `/activation mention` persists `groupActivation` but Feishu group admission still derives the hard mention gate only from config.

## Solution Assessment

No. Reading Feishu session activation at group admission is the right boundary, but this implementation is not the best fix until it preserves authorized activation-command recovery, avoids a forced uncached hot-path read, and aligns the docs.
