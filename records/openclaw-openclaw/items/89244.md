## Best Possible Solution

Keep the warning at the memory-core runtime watcher boundary, avoid config-time filesystem scans, and cover both newly created and extended chokidar watcher paths before merge.

## Reproduction Assessment

Yes for the PR defect by source inspection: the built-in fallback test shows native watcher failure adds the memory directory to an existing chokidar watcher, while the new pressure check only runs when creating a new watcher. I did not run a live FD-pressure repro in this read-only review.

## Solution Assessment

No: moving the warning out of config validation is the right layer, but the implementation is not complete until the existing-watcher fallback add path also checks pressure. The safer repair is one shared runtime pressure helper used after both watcher creation and watcher extension.
