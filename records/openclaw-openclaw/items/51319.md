## Summary

Keep open: this is a focused bug fix on the right code path, but the current PR head still fails CI because the new Darwin regression test does not mock the imported subprocess binding, and the external-contributor real behavior proof gate is still unmet.

## What This Changes

The PR adds macOS zombie-process detection to `isPidAlive()` using `ps -o state= -p` and adds a Darwin regression test.

## Best Possible Solution

Keep the shared `isPidAlive()` approach, fix the mockable subprocess boundary so CI passes, preserve first-character `Z` detection, and attach redacted real macOS stale-lock recovery proof before merge.

## Reproduction Assessment

Partially. Source inspection shows current main skips zombie detection outside Linux while gateway and session locks depend on `isPidAlive()`, and CI gives a high-confidence reproduction of the PR-introduced regression test failure, but I do not have live macOS gateway zombie proof here.

## Solution Assessment

Not yet. The implementation direction is narrow and supported by the macOS `ps` state contract, but the current patch must fix its subprocess mock/import mismatch and provide real macOS behavior proof before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Mock the spawned ps call used by pid-alive:** `src/shared/pid-alive.ts:1`
  - body: `pid-alive.ts` imports `spawnSync` directly, but the new test spies on `childProcess.spawnSync` after `isPidAlive` has already captured its own binding. Use a namespace import in the module or a module mock/dynamic import in the test so the regression actually exercises the mocked zombie state.
