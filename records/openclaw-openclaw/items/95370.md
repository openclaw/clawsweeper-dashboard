## Summary

The PR targets a real async fallback-notice gap, but it should stay open because the current fallback-step notice path can announce the last failed fallback as the selected model in multi-hop chains and then fail to dedupe the later canonical fallback notice.

## What This Changes

The PR adds ACP parent fallback/fallback_step notice relay and preserves leading fallback notice payloads when cron/isolated final text collapses streamed output.

## Best Possible Solution

Format user-facing async fallback notices from the canonical selected-to-active transition and dedupe fallback_step/fallback lifecycle notices before maintainer acceptance of expanded visibility.

## Reproduction Assessment

Yes, from source: current main ignores fallback lifecycle notices in the ACP parent relay and cron final-text collapse drops leading fallback notices; I did not run the live Z.AI or hosted cron setup.

No, not yet; the owner boundary is plausible, but fallback_step uses the last failed candidate as the selected model in multi-hop chains and can conflict with the final fallback notice.
