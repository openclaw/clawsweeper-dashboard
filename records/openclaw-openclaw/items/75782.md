## Summary

Keep open. Current main still has the same source-backed embedded-run auth boundary: the runner awaits auth initialization before marking the `auth` startup stage, and API-key profiles still call through provider runtime auth preparation; the existing lookup cache does not prove the reported 10-15s bucket is fixed.

## What This Changes

The issue asks OpenClaw to explain and reduce the embedded-run startup `auth` stage that blocks for 10-15s on an API-key-only model profile.

## Best Possible Solution

Add fine-grained auth substage timings first, then optimize the proven hot path while preserving provider `prepareRuntimeAuth` semantics.

## Reproduction Assessment

No high-confidence runtime reproduction for the 10-15s wall time. Source inspection confirms the measured `auth` bucket still includes API-key profile resolution plus provider runtime auth preparation on current main, but the production latency needs the reporter's setup or added substage timings.

## Solution Assessment

No for the suggested blanket skip/config flag. The narrower maintainable solution is instrumentation followed by a targeted cache, fast path, or parallelization only where the provider hook contract proves it is safe.
