## Summary

Keep open. Current main still has a sequential embedded-run startup `auth` bucket that covers auth-store/profile resolution plus `initializeAuthProfile()`, and that controller still resolves the API key and invokes provider runtime auth preparation even for normal API-key credentials. The report is fresh, has repeated production trace samples, and I found no current-main or shipped-release evidence that the specific `auth` stage latency is fixed.

## What This Changes

The issue reports a 10-15s embedded-run startup `auth` stage and asks for the stage contents plus a safe fast path, skip option, or parallelization for API-key-only installs.

## Best Possible Solution

Add fine-grained timing inside the embedded-run auth block, then optimize the proven hot subpath with a hook-aware fast path that avoids unnecessary runtime/plugin hydration when no provider auth hook can apply while preserving providers that intentionally transform API-key credentials.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated production timings, and source confirms what the `auth` bucket measures, but reproducing the 10-15s wall time on current main requires the reporter's provider/config/profile setup or equivalent instrumentation.

## Solution Assessment

No current fix is proposed. A blanket API-key skip or `OPENCLAW_SKIP_MODEL_AUTH_PREHEAT` is not clearly the best solution because current tests show API-key credentials can validly pass through provider runtime auth; substage timing followed by a provider-hook-aware optimization is safer.
