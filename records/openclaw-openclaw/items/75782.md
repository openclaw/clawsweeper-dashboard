## Summary

Keep open. Current main still lacks source-backed proof that the embedded-run `auth` latency is fixed: the startup bucket still waits for auth initialization, API-key credentials still pass through provider runtime auth preparation, and the reporter's follow-up narrows the production setup without proving the hot subpath is resolved.

## What This Changes

The issue requests diagnosis and reduction of a 10-15s embedded-run startup `auth` stage for API-key-only model auth setups.

## Best Possible Solution

Add fine-grained auth substage timings first, then optimize the proven hot path while preserving provider `prepareRuntimeAuth` semantics.

## Reproduction Assessment

Unclear for the 10-15s wall time. Source inspection confirms the measured bucket boundary and the API-key provider-runtime hook path on current main, but I did not reproduce the production latency without the reporter's provider/config/profile setup.

## Solution Assessment

No for the blanket skip/config direction. The maintainable path is substage instrumentation followed by a targeted cache, fast path, or parallelization only where provider hook contracts prove it is safe.
