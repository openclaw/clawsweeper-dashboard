## Summary

Keep open. The report is fresh, has a maintainer-side staging reproduction, and current main still routes API-key model auth through the embedded-run `auth` bucket and provider runtime auth hooks without proof that the 8-15s latency is fixed.

## What This Changes

The issue asks OpenClaw to diagnose and reduce the embedded-run startup `auth` stage latency for API-key-only model auth profiles.

## Best Possible Solution

Add fine-grained auth substage timing first, then optimize the measured hot path with a hook-aware fast path or cache that preserves provider runtime auth semantics.

## Reproduction Assessment

Yes for the shipped 2026.4.29 behavior: the issue gives trace commands and repeated timings, and the staging comment partially reproduced 8-9s `auth` on live embedded `chat.send`. Current main still needs the same staging setup or new substage instrumentation to quantify the exact remaining cost.

## Solution Assessment

No for a blanket API-key skip as proposed, because API-key auth can be provider-transformed today. The narrow maintainable path is instrumentation followed by a targeted cache/fast path for the proven expensive suboperation.
