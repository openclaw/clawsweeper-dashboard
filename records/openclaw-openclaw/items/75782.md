## Summary

Keep open. Current main still measures `auth` after a sequential `initializeAuthProfile()` await, and the API-key path still invokes provider runtime auth hooks; `v2026.5.2` includes broad plugin/runtime latency reductions, but I did not find source, tests, release notes, or reporter confirmation proving this specific 10-15s auth bucket is fixed.

## What This Changes

The issue requests diagnosis and optimization of a 10-15s embedded-run startup `auth` stage even when only API-key model auth profiles are configured.

## Best Possible Solution

Add auth substage timings around store load, profile ordering, `getApiKeyForModel`, runtime plugin lookup, and `prepareRuntimeAuth`, then optimize only the proven hot subpath while preserving provider hook semantics.

## Reproduction Assessment

Unclear. The issue has concrete production traces and current source confirms the measured bucket boundary, but a high-confidence current-main reproduction still needs the reporter's provider/config/profile setup or added substage instrumentation.

## Solution Assessment

No for the blanket skip path. The maintainable path is instrumentation first, followed by a targeted cache or fast path only where the provider contract proves it is safe.
