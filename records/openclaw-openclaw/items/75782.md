## Summary

Keep open: current main and v2026.6.10 still put auth-store setup, credential resolution, and provider runtime-auth preparation before the embedded-run `auth` stage is marked, and no related merged PR proves the API-key-only/custom-provider latency case is fixed.

## What This Changes

The issue asks OpenClaw to explain and reduce a repeated 10-15s embedded-run `auth` startup stage that persists even with only an API-key auth profile configured.

## Best Possible Solution

Keep this as the generic embedded-run auth-stage latency tracker, add maintainer-guided timing around auth-store load, `getApiKeyForModel`, and `prepareProviderRuntimeAuth`, then land a measured cache or fast path that preserves provider runtime-auth and profile-failover semantics.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The reporter supplied production traces, and current source still maps the timing bucket to per-run auth-store setup plus auth initialization; the exact slow substep needs instrumentation.

## Solution Assessment

Unclear: a blanket API-key skip is not the best fix and was withdrawn by the reporter because provider runtime-auth transforms are active contracts. The best path is measured substage timing followed by a cache or fast path that preserves auth semantics.
