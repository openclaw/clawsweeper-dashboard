## Summary

Keep open. Current main still has a sequential embedded-run startup `auth` bucket that covers auth-store/profile work and `initializeAuthProfile()`, and although current main now has a provider-runtime plugin lookup cache, I did not find proof that it fully fixes this issue's reported 10-15s auth-stage latency.

## What This Changes

The issue asks OpenClaw to explain and optimize the embedded-run startup `auth` stage, which reportedly blocks for 10-15 seconds even with only API-key model auth profiles configured.

## Best Possible Solution

Add fine-grained timing inside the embedded-run auth block, then optimize the proven hot subpath with a hook-aware fast path that preserves providers that intentionally transform API-key credentials.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated production timings, and source confirms the `auth` bucket boundary, but reproducing the 10-15s wall time on current main requires the reporter's provider/config/profile setup or equivalent instrumentation.

## Solution Assessment

Unclear for the requested skip path. Current main's provider-runtime cache is a safe partial improvement, but a blanket API-key skip is not clearly correct because API-key credentials can legitimately pass through runtime auth hooks.
