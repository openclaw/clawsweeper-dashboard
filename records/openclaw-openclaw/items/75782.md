## Summary

Keep open. Current main still awaits auth initialization inside the embedded-run startup `auth` bucket, and API-key profiles still pass through provider runtime auth preparation; recent plugin-runtime lookup caching is useful but does not prove the reported 10-15s production bucket is fixed.

## What This Changes

The issue asks OpenClaw to identify and reduce the embedded-run startup `auth` bucket that remains 10-15s on an API-key-only model auth profile setup.

## Best Possible Solution

Add fine-grained auth substage timing first, then cache, reuse, or parallelize only the proven hot path while preserving provider `prepareRuntimeAuth` semantics.

## Reproduction Assessment

Unclear for the exact 10-15s wall time because I did not run the reporter's custom gateway/profile setup. Source inspection does confirm the current-main sequential bucket and API-key provider-runtime hook path that the report points at.

## Solution Assessment

No for a blanket skip/config flag. The safer solution is substage instrumentation followed by a targeted cache/reuse or parallelization change where the provider runtime-auth contract proves it is safe.
