## Summary

Keep open. Current main still has a sequential embedded-run startup `auth` bucket that covers profile-store/order work and `initializeAuthProfile()`, and the API-key path still awaits provider runtime auth hooks; the new reporter follow-up adds useful production facts but does not show the 10-15s bucket is fixed.

## What This Changes

Investigate and reduce the embedded-run startup `auth` stage so API-key-only model auth no longer blocks synchronously for 10-15 seconds per turn.

## Best Possible Solution

Add auth substage timing around store load, profile ordering/provider preference, `getApiKeyForModel()`, and `prepareProviderRuntimeAuth()`, then optimize the measured hotspot without bypassing provider-owned API-key transforms.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated production timings, and source confirms the `auth` bucket boundary; reproducing the 10-15s wall time requires the reporter's provider/config/profile setup or equivalent instrumentation.

## Solution Assessment

Unclear for a final performance fix. A blanket API-key skip is not the best approach because API-key credentials can legally pass through provider runtime hooks, while instrumentation-first is the narrowest maintainable next step.
