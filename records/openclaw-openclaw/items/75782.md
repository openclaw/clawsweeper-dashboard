## Summary

Keep open. Current main still awaits auth initialization before marking the embedded-run startup `auth` bucket, and the API-key path still invokes provider runtime auth; the reporter narrowed the production case, but there is not yet enough substage timing to safely optimize or close this as fixed.

## What This Changes

The issue asks OpenClaw to explain and reduce the embedded-run startup `auth` bucket that remains 10-15s with a single API-key provider profile.

## Best Possible Solution

Add fine-grained auth substage timings first, then optimize the proven hotspot while preserving provider `prepareRuntimeAuth` semantics.

## Reproduction Assessment

No high-confidence local reproduction for the 10-15s wall time. Source inspection does confirm the current-main startup `auth` bucket boundary and the API-key provider-runtime hook path that could contain the reported stall.

## Solution Assessment

No for the withdrawn blanket skip/config direction. The narrow maintainable fix is substage instrumentation followed by a targeted cache, fast path, or parallelization only where the measured provider/auth contract proves it safe.
