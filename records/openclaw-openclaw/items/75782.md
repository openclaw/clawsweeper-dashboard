## Summary

Keep open. Current main still has a sequential embedded-run startup `auth` bucket that includes auth-store/profile work and `initializeAuthProfile()`, and I did not find proof that the reported 10-15s auth latency is fixed or superseded by the related performance trackers.

## What This Changes

The issue asks OpenClaw to explain and optimize the embedded-run startup `auth` stage when API-key-only model auth is configured.

## Best Possible Solution

Add fine-grained timing inside the embedded-run auth block, then optimize the measured hot subpath with a hook-aware fast path or cache that preserves provider runtime auth transforms.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated production timings, and source confirms the timing bucket, but I did not reproduce the 10-15s wall time without the reporter's provider/config/profile setup.

## Solution Assessment

No for a blanket API-key skip as proposed. A narrower solution should first instrument the auth substages, then skip or cache only the proven no-op work while keeping provider `prepareRuntimeAuth` contracts intact.
