## Summary

Keep open. Current main still awaits the embedded-run auth controller before marking the startup `auth` stage, and the API-key path still runs through provider runtime auth hooks; the reporter's added production details make the issue more actionable but do not prove the 10-15s bucket is fixed on main.

## What This Changes

The issue requests diagnosis and optimization of a 10-15s synchronous embedded-run `auth` startup stage even when only API-key model auth profiles are configured.

## Best Possible Solution

Add fine-grained timings inside the embedded-run auth block, then optimize only the proven hot subpath while preserving provider `prepareRuntimeAuth` semantics.

## Reproduction Assessment

Unclear. The issue has concrete production traces and source confirms the measured bucket boundary, but a high-confidence local reproduction of the 10-15s wall time still requires the reporter's provider/config/profile setup or equivalent instrumentation.

## Solution Assessment

No for the blanket skip path. The safer direction is substage instrumentation followed by a targeted cache or fast path that only bypasses runtime hooks when the provider contract proves that is safe.
