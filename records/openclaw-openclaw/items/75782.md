## Summary

Keep open. Current main still has a synchronous embedded-run `auth` bucket that includes auth-store/profile resolution plus runtime auth preparation, and the report is fresh with repeated traces; there is not enough evidence that a shipped or main-only fix already addresses this specific stage.

## What This Changes

The issue asks OpenClaw to identify and reduce the synchronous embedded-run `auth` startup stage that reportedly takes 10-15 seconds even with only API-key model auth profiles configured.

## Best Possible Solution

Add fine-grained timing inside the embedded-run auth block, then optimize the proven hot subpath: likely by avoiding unnecessary provider runtime/plugin hydration when no runtime auth hook can apply, while preserving existing provider hooks that intentionally transform API-key credentials.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated production timings, and source confirms what the `auth` bucket measures, but reproducing the 10-15 second wall time on current main requires the reporter's provider/config/profile setup or equivalent targeted instrumentation.

## Solution Assessment

No current fix is proposed yet. A blanket `OPENCLAW_SKIP_MODEL_AUTH_PREHEAT` or API-key-only skip is not clearly the best path because current tests show API-key credentials can validly pass through provider runtime auth; the safer first step is narrower substage timing and then a hook-aware fast path.
