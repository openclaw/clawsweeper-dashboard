## Summary

Keep open. Current main still has a sequential embedded-run startup `auth` bucket around auth-store/profile selection plus `initializeAuthProfile()`, and the later staging comment partially reproduced an 8-9s auth bucket after other mitigations, so there is not enough evidence for implemented-on-main or duplicate closure.

## What This Changes

Request to investigate and reduce the synchronous embedded-run startup auth stage, especially for API-key-only model profile setups.

## Best Possible Solution

Add fine-grained timing inside the auth block, then optimize the measured hot subpath while preserving provider runtime-auth hooks and API-key transformation behavior.

## Reproduction Assessment

Unclear but partially reproduced. The issue gives concrete trace commands and production timings, the source confirms the bucket boundary, and a later staging report reproduced an 8-9s auth bucket, but local reproduction against current main needs a matching live provider/profile setup.

## Solution Assessment

Unclear for the requested skip knob. A blanket API-key-only bypass is not the best proven fix because provider plugins can legitimately transform API-key credentials; substage instrumentation should come before changing the auth contract.
