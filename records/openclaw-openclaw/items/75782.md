## Summary

Keep open. Current main still has an aggregate sequential embedded-run `auth` startup bucket around auth-store/profile selection and runtime-auth preparation, and the fresh staging comment partially reproduces the expensive bucket rather than proving it is fixed.

## What This Changes

The issue reports that embedded-run startup spends 10-15 seconds in the `auth` stage even when only API-key auth profiles remain configured.

## Best Possible Solution

Add fine-grained auth substage timing first, then optimize the proven hot subpath with a hook-aware fast path that preserves provider `prepareRuntimeAuth` behavior.

## Reproduction Assessment

Yes for the reported production/staging class, but not from this read-only checkout. The issue gives concrete trace steps and the latest staging comment reports 8-9s `auth` buckets on live embedded `chat.send` runs against 2026.4.29.

## Solution Assessment

No for the requested blanket skip knob as the best fix. The safer path is instrumentation followed by a contract-preserving optimization, because current tests and docs show API-key credentials may still require provider runtime auth preparation.
