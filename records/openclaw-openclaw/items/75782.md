## Summary

Keep open. Current main still has a sequential embedded-run `auth` startup bucket that covers auth-store/profile resolution and provider runtime auth preparation, and the fresh staging comment partially reproduced the remaining 8-9s auth cost after other startup mitigations.

## What This Changes

Investigate and reduce the embedded-run startup `auth` stage, which is reported to take 10-15s per fresh run even with only API-key auth profiles.

## Best Possible Solution

Add fine-grained auth substage timing first, then optimize the proven hot path with a hook-aware fast path that preserves providers which transform API-key credentials.

## Reproduction Assessment

Unclear. The issue gives concrete trace commands and repeated timings, and poison partially reproduced 8-9s on staging, but the exact slow substage still needs instrumentation or an equivalent current-main live setup.

## Solution Assessment

No for a blanket API-key skip as the first fix. The maintainable direction is targeted instrumentation followed by a provider-contract-preserving optimization once the hot subpath is known.
