## Summary

Keep this open. Current main still sends `openai-completions` chat through OpenClaw's strict guarded model fetch, where ambient `HTTPS_PROXY` is ignored unless an explicit provider `request.proxy` override or managed proxy lifecycle is active, so the proxy-required sandbox case remains unresolved and security-sensitive.

## What This Changes

The issue requests that `openai-completions` `/v1/chat/completions` requests honor `HTTPS_PROXY` in a proxy-only sandbox instead of failing before reaching the proxy.

## Best Possible Solution

Add a narrowly scoped provider transport policy or diagnostic for trusted configured LLM endpoints while preserving strict DNS-pinned guarded-fetch defaults for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure an `openai-completions` model with `HTTPS_PROXY` set but no `models.providers.*.request.proxy` and no `OPENCLAW_PROXY_ACTIVE`; the chat path enters strict guarded fetch and will not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. A global ambient-env proxy switch would be too broad for this security-sensitive path; the maintainable direction is targeted provider transport handling, explicit config guidance, or a diagnostic.
