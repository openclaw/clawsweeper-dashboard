## Summary

Keep this open. Current main still has a credible source-level path where `openai-completions` chat requests use OpenClaw's guarded model fetch instead of the ambient Undici env-proxy dispatcher, and the issue's links to #42311, #56841, and #46306 point to closely related proxy-routing regressions rather than an already-shipped fix.

## What This Changes

The issue requests that `openai-completions` chat requests reach `api.openai.com` through configured HTTPS proxy routing in proxy-required sandbox environments.

## Best Possible Solution

Add a narrowly scoped provider-transport policy or diagnostic so trusted configured LLM endpoints can use intended proxy routing while strict guarded fetch remains DNS-pinned for arbitrary or untrusted URLs.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` set but without provider `request.proxy` or managed proxy lifecycle; the chat path enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. The safest direction is targeted provider-transport handling or a clear diagnostic/config path, not a global relaxation of strict SSRF guarded fetch.
