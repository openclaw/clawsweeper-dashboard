## Summary

Keep this issue open. Current `main` still has a credible `openai-completions` chat path that can bypass ambient `HTTPS_PROXY`: the OpenAI SDK is given OpenClaw’s guarded fetch, and that guard remains in strict mode unless managed proxy routing or an explicit provider proxy override is active.

## What This Changes

The issue reports that `openai-completions` chat requests to OpenAI can bypass `HTTPS_PROXY` in a proxy-required sandbox while other provider traffic uses the proxy.

## Best Possible Solution

Add a narrow provider-transport policy or diagnostic that makes configured trusted LLM endpoints use intended proxy routing while preserving strict SSRF guard behavior for arbitrary URLs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: configure `openai-completions` with ambient `HTTPS_PROXY`, no `proxy.enabled` lifecycle, and no `models.providers.<id>.request.proxy`; the chat request enters strict guarded fetch and does not opt into EnvHttpProxyAgent.

## Solution Assessment

No current implemented solution is present. The safest direction is targeted provider transport handling or clearer managed-proxy/request-proxy guidance, not a broad relaxation of strict guarded-fetch defaults.
