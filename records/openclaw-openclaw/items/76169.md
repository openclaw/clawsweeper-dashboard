## Summary

Keep this open. Current main still has a credible `openai-completions` chat path that bypasses ambient `HTTPS_PROXY` unless a provider request proxy override or the managed proxy lifecycle is active, and the reporter's proxy-log evidence narrows this beyond the localhost-only verification used for #56841.

## What This Changes

The issue asks for `openai-completions` chat-completion requests to honor proxy-required `HTTPS_PROXY` routing instead of attempting direct egress.

## Best Possible Solution

Add scoped provider-transport handling or diagnostics so trusted model-provider endpoints route through the intended proxy path while preserving strict SSRF behavior for arbitrary guarded fetches.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` set but no `models.providers.<id>.request.proxy` and no managed `OPENCLAW_PROXY_ACTIVE`; the chat request enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented solution is present on current main. The safest direction is a targeted provider-transport policy or explicit diagnostic/workaround guidance, not a global relaxation of strict guarded fetch.
