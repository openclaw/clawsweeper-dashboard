## Summary

Keep this issue open. Current main still has a source-level path matching the report: `openai-completions` chat uses OpenClaw's guarded fetch in strict mode, and ambient `HTTPS_PROXY` is only used when trusted env-proxy mode or the managed proxy lifecycle is active.

## What This Changes

The issue asks OpenClaw's `openai-completions` chat transport to honor proxy-only sandbox egress via `HTTPS_PROXY` instead of attempting direct guarded-fetch routing.

## Best Possible Solution

Add a targeted model-provider transport policy or diagnostic for trusted LLM endpoints so intended proxy routing works without relaxing strict guarded-fetch defaults globally.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` set, no `models.providers.<id>.request.proxy`, and no `OPENCLAW_PROXY_ACTIVE`; the chat request enters strict guarded fetch and does not opt into EnvHttpProxyAgent.

## Solution Assessment

No current implemented solution is present for the reported default path. The safest direction is targeted provider-transport handling or clear managed-proxy/request-proxy guidance, not a broad relaxation of `fetchWithSsrFGuard` strict mode.
