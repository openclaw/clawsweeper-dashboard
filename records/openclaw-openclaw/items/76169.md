## Summary

Keep this open. Current main still sends `openai-completions` chat requests through the guarded provider fetch path; without explicit provider `request.proxy` or the managed proxy lifecycle, strict guarded fetch does not adopt ambient `HTTPS_PROXY`, so the proxy-only sandbox failure remains credible.

## What This Changes

The issue reports that OpenAI-compatible Chat Completions requests configured with `api: "openai-completions"` bypass `HTTPS_PROXY` in a proxy-required sandbox while other OpenClaw transports route through the proxy.

## Best Possible Solution

Add a narrowly scoped provider transport policy or diagnostic so trusted configured LLM endpoints use intended proxy routing while arbitrary guarded URL fetches keep strict DNS-pinned defaults.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` but no provider `request.proxy` and no managed proxy lifecycle; the chat path enters strict guarded fetch, which does not opt into ambient env-proxy dispatch.

## Solution Assessment

Unclear for the final product behavior. A global relaxation of strict guarded fetch is not the best fix; the safer direction is targeted provider transport handling, explicit config guidance, or a clear diagnostic for proxy-only deployments.
