## Summary

Keep this issue open. Current main still has a credible `openai-completions` chat path that can bypass ambient `HTTPS_PROXY`: the OpenAI SDK is given OpenClaw's guarded fetch, and that guard remains in strict direct/DNS-pinned mode unless explicit provider proxy configuration or the managed proxy lifecycle is active.

## What This Changes

Investigate `openai-completions` chat-completions failures in proxy-required sandboxes where `/v1/chat/completions` misses `HTTPS_PROXY` while other provider traffic uses the proxy.

## Best Possible Solution

Add a narrow model-provider transport policy or diagnostic that makes configured trusted LLM endpoints use proxy routing when intended while preserving strict guarded-fetch behavior for arbitrary URLs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: configure `openai-completions` with `HTTPS_PROXY` set, no `models.providers.<id>.request.proxy`, and no managed proxy activation; chat requests enter strict guarded fetch without EnvHttpProxyAgent.

## Solution Assessment

No current implemented solution is present. The safest direction is targeted provider-transport handling or clear managed-proxy/request-proxy guidance, not a broad relaxation of `fetchWithSsrFGuard` strict mode.
