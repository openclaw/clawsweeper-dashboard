## Summary

Keep this open. Current main still routes `openai-completions` chat through OpenClaw's guarded model fetch path, where ambient proxy env is not honored unless explicit provider proxy configuration or managed proxy activation is present, so the proxy-required sandbox case remains unresolved.

## What This Changes

The issue reports that `openai-completions` chat requests to `https://api.openai.com/v1` bypass `HTTPS_PROXY` in a proxy-required sandbox while other provider, channel, and embedding requests reach the proxy.

## Best Possible Solution

Define a narrow provider transport policy or diagnostic for configured LLM provider endpoints while preserving strict DNS-pinned guarded fetch defaults for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure an `openai-completions` provider with `HTTPS_PROXY` set but no `models.providers.*.request.proxy` and no `OPENCLAW_PROXY_ACTIVE`; the chat path enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. A broad global ambient-proxy change is not the best solution for this security-sensitive path; the safer direction is targeted provider transport handling, explicit config guidance, or a diagnostic.
