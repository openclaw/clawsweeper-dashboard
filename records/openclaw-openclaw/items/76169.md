## Summary

Keep this issue open. Current main still sends `openai-completions` chat through OpenClaw's guarded fetch, which remains in strict/direct routing unless the managed proxy lifecycle or an explicit provider proxy override is active, so the reporter's ambient `HTTPS_PROXY` bypass report is credible.

## What This Changes

The issue asks OpenClaw to make OpenAI-compatible chat-completions requests honor `HTTPS_PROXY` in proxy-required sandbox environments.

## Best Possible Solution

Add a narrow provider-transport policy or diagnostic that makes configured trusted LLM endpoints use intended proxy routing while preserving strict SSRF guard behavior for arbitrary URLs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: configure `openai-completions` with ambient `HTTPS_PROXY`, no managed proxy lifecycle, and no `models.providers.<id>.request.proxy`; the chat request enters strict guarded fetch without trusted env-proxy mode.

## Solution Assessment

No current implemented solution is present. The safest path is targeted provider transport handling or clearer managed-proxy/request-proxy guidance, not a broad relaxation of strict guarded-fetch defaults.
