## Summary

Keep this open. Current main still has a credible source-level path where `openai-completions` chat requests use OpenClaw's strict guarded fetch instead of the ambient Undici global env-proxy dispatcher, so the reported proxy-required sandbox failure is not implemented away.

## What This Changes

Request that `openai-completions` chat requests to `api.openai.com` honor required `HTTPS_PROXY` routing instead of using strict direct guarded fetch in proxy-only sandboxes.

## Best Possible Solution

Add a narrowly scoped provider transport policy or diagnostic so trusted configured LLM endpoints can use intended proxy routing while strict guarded fetch remains DNS-pinned for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` set but no provider `request.proxy` and no managed proxy lifecycle; the chat path enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. The safest direction is targeted provider-transport handling or a clear diagnostic/config path, not a global relaxation of strict SSRF guarded fetch.
