## Summary

Keep this open. Current main still has a credible ambient-proxy bypass for `openai-completions` chat completions: the OpenAI SDK client uses OpenClaw's guarded fetch, and guarded fetch does not honor ambient `HTTPS_PROXY` in strict mode unless the managed proxy lifecycle is active or a provider request proxy override is configured.

## What This Changes

Request that `openai-completions` chat completions honor `HTTPS_PROXY` in a proxy-required sandbox instead of attempting direct egress and falling back after a connection error.

## Best Possible Solution

Design a narrow model-provider transport change or explicit configuration path that routes trusted LLM endpoints through the required proxy while preserving guarded-fetch DNS-pinning and SSRF protections.

## Reproduction Assessment

Yes. The report gives a concrete sandbox/proxy-log reproduction, and current main shows a code-level path where `openai-completions` chat uses strict guarded fetch instead of the global proxy dispatcher unless managed proxy or provider proxy override is active.

## Solution Assessment

No implemented solution is present yet. The safest direction is a targeted provider-transport or documentation/configuration fix, not a blanket relaxation of strict guarded-fetch defaults.
