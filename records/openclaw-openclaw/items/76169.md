## Summary

Keep open. Current main still sends `openai-completions` chat through OpenClaw's guarded fetch, and that path remains strict/direct unless an explicit provider proxy override, trusted proxy mode, or the managed proxy lifecycle is active.

## What This Changes

The issue asks for OpenAI-compatible chat completions to honor `HTTPS_PROXY` in proxy-required sandboxes instead of failing before reaching the proxy.

## Best Possible Solution

Add a scoped provider-transport policy or diagnostic so configured trusted LLM endpoints can route through env proxy when intended, while strict guarded-fetch behavior remains the default for arbitrary URLs and bypass edge cases.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: configure `openai-completions` with `HTTPS_PROXY` set, no `models.providers.<id>.request.proxy`, and no `OPENCLAW_PROXY_ACTIVE`; the chat path reaches strict guarded fetch without trusted env proxy mode.

## Solution Assessment

Unclear. The requested behavior is valid, but the best fix needs a narrow provider-transport policy, similar to the guarded media helper, rather than a broad relaxation of `fetchWithSsrFGuard` strict mode.
