## Summary

Keep this issue open. Current main still has a credible `openai-completions` chat path that can bypass ambient `HTTPS_PROXY`: the OpenAI SDK is passed OpenClaw's guarded fetch, and that guard remains in strict direct/DNS-pinned mode unless an explicit provider proxy override, trusted-env mode, or the managed proxy lifecycle is active.

## What This Changes

The issue reports that `openai-completions` chat requests to `https://api.openai.com/v1/chat/completions` fail in a proxy-required sandbox because they do not reach the configured `HTTPS_PROXY`.

## Best Possible Solution

Add a narrow model-provider transport policy or diagnostic that makes configured trusted LLM endpoints use proxy routing when intended, while preserving strict guarded-fetch behavior for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure `openai-completions` with `HTTPS_PROXY` set, no `models.providers.<id>.request.proxy`, and no `OPENCLAW_PROXY_ACTIVE`; the chat request enters strict guarded fetch and does not opt into EnvHttpProxyAgent.

## Solution Assessment

No current implemented solution is present. The safest direction is targeted provider-transport handling or clear managed-proxy/request-proxy guidance, not a broad relaxation of `fetchWithSsrFGuard` strict mode.
