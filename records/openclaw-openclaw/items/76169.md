## Summary

Keep this open. Current main still has a credible provider-egress proxy gap: native `api.openai.com` `openai-completions` configs are now normalized to Responses, but both Responses and remaining Completions model requests use the same strict guarded fetch unless explicit provider proxy config or the managed proxy lifecycle is active.

## What This Changes

The issue asks OpenAI-compatible model-provider requests to honor `HTTPS_PROXY` in proxy-required sandboxes instead of bypassing the egress proxy.

## Best Possible Solution

Add a narrow model-provider transport policy or diagnostic that makes trusted provider endpoints use proxy routing when intended, while preserving strict SSRF behavior for arbitrary URLs.

## Reproduction Assessment

Yes, at source level for the proxy-routing class. Configure a model provider with ambient `HTTPS_PROXY`, no `models.providers.<id>.request.proxy`, and no managed proxy activation; the request enters strict guarded fetch instead of trusted env-proxy mode.

## Solution Assessment

No implemented solution is present for the default ambient-proxy expectation. The safest direction is targeted provider-transport handling, diagnostics, or explicit `request.proxy` guidance rather than relaxing strict guarded fetch globally.
