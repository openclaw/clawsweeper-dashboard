## Summary

Keep this issue open. Current main still has a credible env-only proxy bypass path for `openai-completions`: the OpenAI chat-completions transport installs OpenClaw's guarded fetch, and guarded fetch intentionally ignores ambient `HTTPS_PROXY` in strict mode unless the managed proxy lifecycle has set `OPENCLAW_PROXY_ACTIVE=1`.

## What This Changes

The issue requests that `openai-completions` chat requests honor `HTTPS_PROXY` in proxy-required sandbox environments.

## Best Possible Solution

Keep the issue open and design a narrow model-provider transport fix that lets configured LLM endpoints use ambient proxy routing when appropriate without changing the default guarded-fetch behavior for untrusted URLs.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: configure an `openai-completions` model with only ambient `HTTPS_PROXY`, leave `OPENCLAW_PROXY_ACTIVE` unset, and the chat-completions request goes through strict guarded fetch with a direct pinned dispatcher rather than EnvHttpProxyAgent.

## Solution Assessment

No implemented solution is present yet. The safest direction is a targeted provider-transport change or explicit managed-proxy requirement, not a broad change to `fetchWithSsrFGuard` strict-mode defaults.
