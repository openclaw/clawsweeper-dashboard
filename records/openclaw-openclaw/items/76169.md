## Summary

Keep this open. Current main still routes `openai-completions` chat through OpenClaw's guarded model fetch, where ambient `HTTPS_PROXY` is intentionally ignored unless an explicit provider proxy override or managed proxy lifecycle is active, so the reporter's proxy-required sandbox case remains a real policy gap rather than an implemented fix.

## What This Changes

Requests that `openai-completions` chat requests to `https://api.openai.com/v1` honor ambient `HTTPS_PROXY` in proxy-required sandboxes instead of taking the direct guarded-fetch path.

## Best Possible Solution

Add a narrowly scoped provider transport policy or diagnostic for trusted configured LLM endpoints while preserving strict DNS-pinned guarded fetch defaults for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure an `openai-completions` model with `HTTPS_PROXY` set but no `models.providers.*.request.proxy` and no `OPENCLAW_PROXY_ACTIVE`; the chat path enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. A global switch to ambient env proxy would be too broad for this security-sensitive path; the maintainable direction is targeted provider transport handling, explicit config guidance, or a diagnostic.
