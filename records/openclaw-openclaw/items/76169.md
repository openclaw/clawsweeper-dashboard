## Summary

Keep this open. Current main still sends `openai-completions` chat through guarded model fetch, and that path intentionally does not use ambient `HTTPS_PROXY` unless an explicit provider proxy override or managed proxy lifecycle is active, so the reported proxy-required sandbox gap is not implemented on main.

## What This Changes

The issue requests that `openai-completions` chat requests to OpenAI route through `HTTPS_PROXY` in proxy-required sandboxes instead of failing with direct-egress network errors.

## Best Possible Solution

Add a narrowly scoped provider transport policy or diagnostic for trusted configured LLM endpoints while preserving strict guarded-fetch defaults for arbitrary URLs.

## Reproduction Assessment

Yes, at source level. Configure an `openai-completions` model with `HTTPS_PROXY` set but no `models.providers.*.request.proxy` and no `OPENCLAW_PROXY_ACTIVE`; the chat path enters strict guarded fetch and does not opt into env-proxy dispatch.

## Solution Assessment

No implemented fix is present on current main. A global ambient env-proxy switch would be too broad for this security-sensitive path; the safer direction is targeted provider transport handling, explicit config guidance, or a diagnostic.
