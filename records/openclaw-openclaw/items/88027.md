## Summary

Keep open: current main and the latest release still serialize Feishu post replies as one `md` element after producing `<at user_id="...">name</at>` text, so there is no implemented-on-main fix. The remaining blocker is proof and fix shape, because current Feishu docs describe both native post `at` nodes and `md` mention syntax, while prior attempted PRs show that reparsing arbitrary body text into live mentions can create message-delivery risk.

## What This Changes

The issue reports that Feishu post-mode outbound @ mentions render as plain text and asks to emit native post `at` elements while avoiding a suspected bot-name cache collision.

## Best Possible Solution

First capture a current live Feishu/Lark post-mode repro; if it still fails, emit native post `at` elements from structured mention targets and leave cache behavior alone unless a current collision is reproduced.

## Reproduction Assessment

Yes for the payload path, but not for a high-confidence live reproduction. Current main still builds textual `<at user_id="...">name</at>` mentions and wraps the whole post body in one `md` element, while the actual Feishu client rendering still needs redacted live proof.

## Solution Assessment

No, the proposed broad `splitAtElements(messageText)` shape is not yet proven to be the best fix. A safer fix should carry structured mention targets into native post `at` elements and only parse arbitrary body tags if maintainers explicitly accept the ping-safety tradeoff after live proof.
