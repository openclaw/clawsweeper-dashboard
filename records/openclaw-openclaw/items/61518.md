## Summary

Current main already implements the useful proxy-only web_fetch support through an explicit trusted env-proxy config that shipped, while this PR's automatic strict-mode proxy path is superseded by that safer boundary.

## What This Changes

This PR makes strict web_fetch auto-select an env-proxy dispatcher from HTTP(S)_PROXY, changes web-fetch tests to expect EnvHttpProxyAgent, and bundles cron NO_REPLY/forum-topic delivery docs and tests.

## Best Possible Solution

Keep the shipped explicit trusted env-proxy path for proxy-only web_fetch deployments, preserving strict DNS pinning by default and applying HTTP(S)_PROXY/NO_PROXY per redirect hop.

## Reproduction Assessment

Yes for the PR review concern. From the PR head source, `dispatcherPolicy` is selected from the initial URL before `fetchWithSsrFGuard` follows redirects, so an http URL with only HTTPS_PROXY and an https redirect keeps the original direct policy for the redirected hop.

## Solution Assessment

No. The PR's automatic ambient proxying in strict web_fetch is not the best solution; current main's explicit `tools.web.fetch.useTrustedEnvProxy` path is narrower, documented, and uses per-hop proxy selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose proxy policy per redirect hop:** `src/agents/tools/web-guarded-fetch.ts:53-56`
  - body: This derives `dispatcherPolicy` from `rest.url` before the guarded fetch loop handles redirects. If the first request is `http://` with only `HTTPS_PROXY` set and it redirects to `https://`, the HTTPS hop keeps the original direct policy and bypasses the configured proxy; the proxy decision needs to happen per hop or use the existing per-target helper.
  - confidence: 0.88
