## Summary

Keep open; this PR should not merge as-is because it changes the strict web_fetch SSRF/proxy boundary and chooses proxy mode from only the initial URL, while current main intentionally keeps strict web_fetch DNS-pinned unless an explicit trusted or managed proxy path is active. The cron NO_REPLY/forum-topic side work is already represented on current main, so the branch also needs scope cleanup before any renewed review.

## What This Changes

The PR makes strict web_fetch opt into env-proxy dispatch from HTTP(S)_PROXY, updates web-fetch tests, and also adds cron NO_REPLY/forum-topic delivery suppression docs and tests.

## Best Possible Solution

Keep strict web_fetch DNS-pinned by default; if proxy-only web_fetch support is desired, add an explicit trusted or managed proxy path that applies HTTP_PROXY, HTTPS_PROXY, and NO_PROXY per redirect hop and documents the proxy trust boundary.

## Reproduction Assessment

Yes. The review concern is reproducible from source: with the PR logic, start from an http:// URL with only HTTPS_PROXY set and redirect to https://; the dispatcherPolicy chosen before the fetch loop remains direct for the HTTPS hop.

## Solution Assessment

No. The PR is not the best fix because it auto-upgrades the untrusted strict web_fetch path from ambient environment state and only evaluates the initial URL; a maintainable fix should use an explicit trusted/managed proxy seam with per-hop policy checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Choose proxy policy per redirect hop:** `src/agents/tools/web-guarded-fetch.ts:53-57`
  - body: This derives `dispatcherPolicy` from `rest.url` before entering `fetchWithSsrFGuard`, but redirects are evaluated later inside that loop. If the first request is `http://` with only `HTTPS_PROXY` set and it redirects to `https://`, the HTTPS hop keeps the original direct policy and bypasses the configured proxy. Move the proxy decision into the guarded fetch loop or use the per-hop `shouldUseEnvHttpProxyForUrl` path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR changes strict web_fetch proxying in a way that weakens the current DNS-pinned SSRF contract without an explicit trusted-proxy boundary.

Concerns:

- **[medium] Preserve strict web_fetch DNS pinning:** `src/agents/tools/web-guarded-fetch.ts:53`
  - body: The patch makes strict web_fetch choose EnvHttpProxyAgent whenever protocol-specific proxy env is present. Proxy dispatch can move target resolution to the proxy path, so this should be gated by an explicit trusted or managed proxy mode with documented SSRF responsibilities instead of ambient HTTP(S)_PROXY alone.
