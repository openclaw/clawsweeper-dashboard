## Summary

Keep this PR open, but do not merge it as-is. The web-fetch proxy problem is real, yet the patch changes the strict `web_fetch` path in a security-sensitive area and still has a concrete redirect bug: the env-proxy decision is made from the initial URL, while redirects can change protocol and should be evaluated per hop. The bundled cron changes are now covered on current `main`, so the remaining useful work needs a narrower maintainer-reviewed proxy design.

## What This Changes

The PR makes strict `web_fetch` choose an env-proxy dispatcher when HTTP(S)_PROXY is configured, updates web-fetch tests, and also adds cron `NO_REPLY` and Telegram forum-topic delivery suppression docs/tests.

## Best Possible Solution

Keep strict `web_fetch` DNS-pinned by default. If proxy-only web_fetch support is desired, implement a narrow explicit trusted or managed proxy path that evaluates HTTP_PROXY, HTTPS_PROXY, and NO_PROXY per redirect hop, documents the proxy trust boundary, and drops the stale cron side changes.

## Reproduction Assessment

Yes. The source path is enough to reproduce the review concern: apply the PR logic to an `http://` web_fetch with only HTTPS_PROXY set, then redirect to `https://`; the fixed dispatcherPolicy remains direct across the redirect loop.

## Solution Assessment

No. The current PR is not the best fix because it auto-upgrades the strict untrusted web_fetch path from the initial URL scheme; a maintainable fix should preserve the strict default and route proxy-only cases through an explicit trusted/managed proxy seam with per-hop policy checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Choose proxy policy per redirect hop:** `src/agents/tools/web-guarded-fetch.ts:53-57`
  - body: This decides env-proxy mode from only the initial URL. If a request starts on `http://` with only HTTPS_PROXY configured and then redirects to `https://`, `fetchWithSsrFGuard` keeps the fixed direct dispatcherPolicy for the HTTPS hop, so the redirected request bypasses the configured proxy and still fails in proxy-only environments. Move the env-proxy decision into the guarded fetch loop or use a per-hop helper such as `shouldUseEnvHttpProxyForUrl`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes strict web_fetch to route through ambient EnvHttpProxyAgent, which is a security-sensitive change to the SSRF/DNS-pinning contract.

Concerns:

- **[medium] Preserve strict web_fetch DNS pinning:** `src/agents/tools/web-guarded-fetch.ts:53`
  - body: The PR flips untrusted `web_fetch` from the current pinned direct dispatcher to EnvHttpProxyAgent whenever HTTP_PROXY/HTTPS_PROXY is present. Undici proxy dispatch can delegate target resolution to the proxy, so this weakens the current strict-path invariant unless it is gated by an explicit trusted/managed proxy mode and documented proxy trust boundary.
