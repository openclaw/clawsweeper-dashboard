## Summary

Current main already ships the useful web_fetch proxy support through an explicit trusted env-proxy opt-in, while this PR's ambient strict-mode proxy change is superseded by that safer boundary and should not be merged as-is.

## What This Changes

The PR changes strict web_fetch to auto-select an env-proxy dispatcher from HTTP(S)_PROXY, updates web_fetch tests, and bundles cron NO_REPLY suppression docs/tests/runtime changes.

## Best Possible Solution

Keep the shipped explicit tools.web.fetch.useTrustedEnvProxy path, preserving strict DNS pinning by default while allowing operator-controlled proxy-only deployments to opt in.

## Reproduction Assessment

Yes for the PR review concern from source inspection: the branch chooses dispatcherPolicy from the initial URL before fetchWithSsrFGuard follows redirects. I did not run tests because pnpm is not installed in this worker.

## Solution Assessment

No. The PR's ambient automatic proxying is not the narrowest maintainable solution; current main's explicit trusted env proxy config is safer, documented, tested, and shipped.
