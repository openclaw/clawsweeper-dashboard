## Summary

Keep open: current main does not implement a command-level `chat.send` byte cap, and this PR is a useful hardening direction, but it needs revision before merge because it enforces size after NFC normalization, presents a character-count schema guard beside a byte-limit policy, lacks real behavior proof, and conflicts with current main.

## What This Changes

The PR adds a 512 KiB UTF-8 byte limit for gateway `chat.send` messages, plus schema metadata, sanitizer tests, and gateway protocol documentation.

## Best Possible Solution

Land a revised gateway hardening change that defines raw-vs-normalized byte policy, treats schema `maxLength` as character metadata or removes it, covers decomposed Unicode, includes real gateway/client proof, and documents upgrade impact.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection and a focused byte-count probe show decomposed Unicode can be over 512 KiB on input and under 512 KiB after NFC normalization. This is not a current-main bug reproduction because current main has no command-level `chat.send` byte cap.

## Solution Assessment

No. The hardening direction is reasonable, but the patch should enforce the intended byte representation before accepting the message and make clear that schema `maxLength` is a character constraint rather than byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure raw chat.send bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new cap checks `Buffer.byteLength(normalized, "utf8")` after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB inbound limit; check raw message bytes first or explicitly define a normalized-size policy.
  - confidence: 0.91
- **[P3] Clarify schema maxLength is character-based:** `src/gateway/protocol/schema/logs-chat.ts:41`
  - body: `Type.String({ maxLength })` compiles to JSON Schema `maxLength`, and Ajv validates characters via `ucs2length`, not UTF-8 bytes. Leave it as schema metadata with an explicit comment or remove it as byte enforcement to avoid future drift.
  - confidence: 0.84
