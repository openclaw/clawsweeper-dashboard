## Summary

Keep open. The PR targets a real missing guard because current main still has no `chat.send` message-size cap, but the proposed implementation measures bytes after NFC normalization, so an oversized decomposed Unicode payload can shrink under the limit and be accepted. The schema addition also uses JSON Schema `maxLength`, which is character-count validation rather than byte enforcement.

## What This Changes

The PR adds a documented 512 KiB UTF-8 `chat.send` message limit in the gateway sanitizer, mirrors it with a TypeBox schema `maxLength`, and adds focused sanitizer tests and protocol docs.

## Best Possible Solution

Revise the PR so the 512 KiB policy is enforced against the incoming `chat.send` string bytes before NFC normalization and before other processing. Keep the schema either clearly documented as an advisory character-count layer or remove/rework it, then add regression coverage for ASCII boundary, multi-byte characters, and decomposed Unicode that shrinks under NFC.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by reading the schema, sanitizer, and handler, and the PR bug is reproducible with a focused byte check where decomposed Unicode is 600000 bytes before NFC but 400000 bytes after NFC.

## Solution Assessment

No. The requested direction is sound, but this patch is not the best implementation because the DoS guard runs after normalization and the schema layer is character-count based; raw-byte enforcement with clearer schema/docs is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new limit measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it would be accepted despite the documented 512 KiB request-body limit. Check incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify the schema is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: `maxLength` on JSON Schema strings constrains Unicode code points, not UTF-8 bytes, while this PR advertises a byte limit. Multi-byte strings can pass this schema above 512 KiB and rely on the sanitizer, so the comment should state this is advisory/ASCII-aligned or the schema guard should be removed/reworked. ([learnjsonschema.com](https://www.learnjsonschema.com/2020-12/validation/maxlength/))
  - confidence: 0.82
