## Summary

Keep open: current main does not implement a `chat.send` byte cap, and this branch still has a merge-blocking normalized-byte check plus missing real behavior proof for an external PR.

## What This Changes

The branch adds a 512 KiB UTF-8 byte limit for gateway `chat.send` messages in sanitizer/schema/docs and adds sanitizer boundary tests.

## Best Possible Solution

Land a revised hardening patch that enforces the raw incoming UTF-8 byte cap before normalization, treats schema `maxLength` as advisory or documents its character semantics, and includes maintainer-approved policy plus real behavior proof.

## Reproduction Assessment

Yes for the PR defect. Source inspection of the diff plus a focused Node byte-count check shows a decomposed Unicode payload can exceed 512 KiB on input, shrink under the cap after NFC normalization, and pass the proposed sanitizer.

## Solution Assessment

No. The hardening direction is useful, but the implementation should measure raw incoming bytes before NFC normalization and should not imply JSON Schema `maxLength` is UTF-8 byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure raw message bytes before normalization:** `src/gateway/server-methods/chat.ts:235-238`
  - body: The new cap is computed from `normalized`, after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB request-body cap; measure raw incoming bytes before normalization or explicitly define and test a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength is character-based:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `maxLength` on a JSON Schema string is character-based, while this PR describes a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so the schema/comment should not present this as byte enforcement.
  - confidence: 0.84

## Security Review
