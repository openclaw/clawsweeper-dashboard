## Summary

Keep this PR open: current main still lacks the proposed command-level chat.send cap, but the branch should not merge as-is because it measures bytes after NFC normalization, overstates the schema layer as a byte guard, and lacks real gateway/client proof.

## What This Changes

The PR adds a 512 KiB UTF-8 byte cap for gateway chat.send messages, mirrors it with a protocol schema maxLength, adds sanitizer tests, and documents the limit.

## Best Possible Solution

Land a revised gateway hardening change that enforces raw inbound UTF-8 bytes before normalization, clarifies schema character semantics, updates protocol docs/tests, and includes real gateway/client proof.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection plus a focused Node byte-count check shows decomposed Unicode can exceed 512 KiB on input, shrink below the cap after NFC normalization, and pass the proposed sanitizer. This is not a current-main bug reproduction because current main has no chat.send message byte cap yet.

## Solution Assessment

No. The hardening direction is useful, but this patch should measure raw incoming UTF-8 bytes before normalization and avoid presenting JSON Schema maxLength as byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure raw chat.send bytes before normalization:** `src/gateway/server-methods/chat.ts:235-236`
  - body: The new cap checks `Buffer.byteLength(normalized, "utf8")` after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB request-body cap; measure the raw message bytes first or explicitly define a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength character semantics:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `Type.String({ maxLength })` is a JSON Schema character-count constraint, not a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so the schema/comment should not present this as byte-level enforcement.
  - confidence: 0.84

## Security Review
