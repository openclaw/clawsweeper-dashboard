## Summary

Keep open: current main still lacks a `chat.send` byte cap, and this PR has a blocking correctness/security finding because the proposed limit is checked after NFC normalization.

## What This Changes

The PR adds a 512 KiB UTF-8 byte limit for gateway `chat.send` messages in the sanitizer, mirrors it with a TypeBox schema bound, adds sanitizer tests, and documents the protocol limit.

## Best Possible Solution

Keep the hardening direction, but enforce the limit on raw incoming UTF-8 bytes before normalization, treat schema `maxLength` as advisory character validation or clarify it, and ship the protocol change with tests, docs, and changelog aligned.

## Reproduction Assessment

Yes for the PR defect. Source inspection plus a focused Node byte-count check shows a decomposed Unicode input can exceed 512 KiB on the wire, shrink below the cap after NFC normalization, and pass the proposed sanitizer.

## Solution Assessment

No. The direction is useful, but the current patch is not the best implementation because a DoS/request-size guard should measure raw incoming bytes before normalization and should not imply JSON Schema `maxLength` is byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:235-238`
  - body: The new byte cap measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KiB on input but about 400 KiB after NFC, so it would pass despite the documented 512 KiB request-body limit. Measure the incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify schema maxLength is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `maxLength` on JSON Schema strings constrains characters, while this PR advertises a UTF-8 byte limit. Multi-byte strings can exceed 512 KiB and still pass this schema before the sanitizer rejects them, so the comment should state this is only an advisory character-count/ASCII-aligned layer or the schema guard should be reworked.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `docs/gateway/protocol.md:183-185`
  - body: This PR changes user-facing gateway behavior by rejecting oversized `chat.send` messages, but the diff does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing behavior and protocol changes before merge.
  - confidence: 0.86
