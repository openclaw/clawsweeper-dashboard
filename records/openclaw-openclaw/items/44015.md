## Summary

Keep this PR open, but it is not merge-ready. Current main still has no `chat.send` byte cap, while the PR's main guard measures post-NFC bytes, allowing some oversized raw UTF-8 payloads through; the external PR also needs real behavior proof before merge.

## What This Changes

The PR adds a 512 KiB UTF-8 byte cap to gateway `chat.send` input sanitization, mirrors it with a TypeBox schema `maxLength`, adds sanitizer tests, and documents the protocol limit.

## Best Possible Solution

Land a revised hardening patch that enforces the raw incoming UTF-8 byte cap before normalization, treats schema `maxLength` as advisory character validation or removes it, documents the accepted policy, and includes after-fix real behavior proof.

## Reproduction Assessment

Yes for the PR defect. Source inspection plus a focused Node byte-count check shows a decomposed Unicode payload can exceed 512 KiB on input, shrink under the cap after NFC normalization, and pass the proposed sanitizer.

## Solution Assessment

No. The hardening direction is useful, but the implementation should measure raw incoming bytes before NFC normalization and should not imply JSON Schema `maxLength` is UTF-8 byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure raw message bytes before normalization:** `src/gateway/server-methods/chat.ts:235-238`
  - body: The new cap is computed from `normalized`, after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KiB on input but about 400 KiB after NFC, so it passes despite the documented 512 KiB request-body cap; measure the incoming string bytes before normalization, or explicitly document and test a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `maxLength` on a JSON Schema string is character-based, while the PR describes a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass this schema before the sanitizer sees them, so the comment should state this is only an advisory character/ASCII-aligned guard or the schema layer should be reworked.
  - confidence: 0.84
