## Summary

Keep open: current main still lacks the requested command-level `chat.send` message cap, but this PR has a blocking raw-byte normalization gap, a schema wording mismatch, and missing real behavior proof before merge.

## What This Changes

The PR adds a 512 KiB UTF-8 byte cap for `chat.send` messages in gateway sanitization, protocol schema metadata, tests, and gateway protocol docs.

## Best Possible Solution

Land a revised gateway hardening change that enforces the chosen raw-vs-normalized byte policy, documents schema character semantics, updates focused tests/docs, and includes redacted real gateway/client proof.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection plus a focused Node byte-count check shows decomposed Unicode can exceed 512 KiB on input, shrink below the cap after NFC normalization, and pass the proposed sanitizer. This is not a current-main bug reproduction because current main has no `chat.send` message byte cap yet.

## Solution Assessment

No. The hardening direction is useful, but the patch should either check raw incoming UTF-8 bytes before normalization or explicitly document a normalized-size policy, and the schema/docs should not present character `maxLength` as byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure raw chat.send bytes before normalization:** `src/gateway/server-methods/chat.ts:235-236`
  - body: The new cap checks `Buffer.byteLength(normalized, "utf8")` after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB inbound limit; check raw message bytes first or define a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength character semantics:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `Type.String({ maxLength })` is a JSON Schema character-count constraint, not a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so this schema/comment should not read as byte-level enforcement.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds a useful DoS guard, but its current placement and schema wording leave concrete size-enforcement gaps.

Concerns:

- **[medium] Check payload size before NFC normalization:** `src/gateway/server-methods/chat.ts:235`
  - body: The intended size guard measures `Buffer.byteLength(normalized, "utf8")`; decomposed Unicode can shrink during NFC normalization, allowing an oversized raw request body to pass the proposed 512 KiB cap.
  - confidence: 0.88

  - body: The protocol schema uses JSON Schema `maxLength`, which is character-based rather than UTF-8-byte-based, so it is not a first-line 512 KiB byte-size guard for multi-byte content.
  - confidence: 0.82
