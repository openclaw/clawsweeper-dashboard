## Summary

Keep open: the hardening request is not implemented on current main, and this PR still has a correctness/security issue in where it measures bytes plus missing real behavior proof.

## What This Changes

The PR adds a 512 KiB UTF-8 byte limit for gateway `chat.send` messages, updates the protocol schema/docs, and adds sanitizer tests.

## Best Possible Solution

Land a revised gateway hardening change that enforces a clearly defined raw UTF-8 inbound message limit before normalization, documents schema character semantics, updates tests/docs, and includes redacted real gateway/client proof.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection plus a focused Node byte-count check shows decomposed Unicode can exceed 512 KiB on input, shrink below the cap after NFC normalization, and pass the proposed sanitizer. This is not a current-main bug reproduction because current main has no `chat.send` message byte cap yet.

## Solution Assessment

No. The hardening direction is useful, but this patch should measure raw incoming UTF-8 bytes before normalization and avoid presenting JSON Schema `maxLength` as byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure raw chat.send bytes before normalization:** `src/gateway/server-methods/chat.ts:235-236`
  - body: The new cap checks `Buffer.byteLength(normalized, "utf8")` after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB request-body cap; measure the raw message bytes first or explicitly define a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength character semantics:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `Type.String({ maxLength })` is a JSON Schema character-count constraint, not a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so the schema/comment should not present this as byte-level enforcement.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds a useful DoS guard, but its current placement allows oversized decomposed-Unicode input to pass the documented raw byte limit.

Concerns:

- **[medium] Check payload size before NFC normalization:** `src/gateway/server-methods/chat.ts:235`
  - body: The intended size guard measures `Buffer.byteLength(normalized, "utf8")`; decomposed Unicode can shrink during NFC normalization, allowing an oversized raw request body to pass the proposed 512 KiB cap.
  - confidence: 0.88
- **[low] Do not treat schema maxLength as byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:39`
  - body: The protocol schema uses JSON Schema `maxLength`, which is character-based rather than UTF-8-byte-based, so it is not a first-line 512 KiB byte-size guard for multi-byte content.
  - confidence: 0.82
