## Summary

Keep open. Current main still has no `chat.send` message-size cap, and this PR is a useful narrow hardening candidate, but the proposed patch measures bytes after NFC normalization and therefore does not enforce the documented incoming 512 KiB request-body limit for decomposed Unicode input.

## What This Changes

The PR adds a documented 512 KiB UTF-8 `chat.send` message limit in the gateway sanitizer, mirrors it with a TypeBox schema `maxLength`, and adds focused tests plus protocol docs.

## Best Possible Solution

Revise the PR so the 512 KiB policy is enforced against the incoming raw string bytes before normalization, and either document the schema as an advisory character-count layer or rework/remove it; add a decomposed-Unicode regression case.

## Reproduction Assessment

Yes. The PR bug is reproducible with a focused byte check: `"e\u0301".repeat(200000)` is 600000 bytes before NFC and 400000 bytes after NFC, so the proposed normalized-byte check would accept an oversized raw input.

## Solution Assessment

No. The direction is sound, but the proposed implementation is not the best fix because the DoS guard runs after normalization and the schema layer is character-count based; raw-byte enforcement plus clearer schema/docs is narrower and more accurate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new limit measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it would be accepted despite the documented 512 KiB request-body limit. Check incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify the schema is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: `maxLength` on JSON Schema strings constrains characters, not UTF-8 bytes, while this PR advertises a byte limit. Multi-byte strings can pass this schema above 512 KiB and rely on the sanitizer, so the comment should state this is advisory/ASCII-aligned or the schema guard should be removed or reworked.
  - confidence: 0.82
