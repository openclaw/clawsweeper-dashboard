## Summary

Keep open: current main still lacks this `chat.send` size guard, and the PR has an actionable correctness issue because it checks byte length after NFC normalization, which can accept oversized raw decomposed-Unicode input.

## What This Changes

The PR adds a documented 512 KiB UTF-8 `chat.send` message limit in the gateway sanitizer, mirrors it with a TypeBox schema bound, and adds focused tests plus protocol docs.

## Best Possible Solution

Revise the PR so the server enforces the 512 KiB policy against raw incoming string bytes before normalization, then keep schema and docs wording aligned with that server behavior.

## Reproduction Assessment

Yes. Source inspection and the focused byte check give a high-confidence reproduction for the PR defect: a decomposed payload can be over 512 KiB before NFC normalization but under the limit after normalization, so the proposed check would accept it.

## Solution Assessment

No. The direction is sound, but the current implementation is not the narrowest maintainable fix because a request-size DoS guard should measure raw incoming bytes before normalization and should avoid presenting schema `maxLength` as byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:235-238`
  - body: The new byte cap measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KiB on input but about 400 KiB after NFC, so it would pass despite the documented 512 KiB request-body limit. Measure the incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify that schema maxLength is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `maxLength` on JSON Schema strings constrains characters, while this PR advertises a UTF-8 byte limit. Multi-byte strings can exceed 512 KiB and still pass this schema before the sanitizer rejects them, so the comment should state this is only an advisory character-count/ASCII-aligned layer or the schema guard should be reworked.
  - confidence: 0.82
