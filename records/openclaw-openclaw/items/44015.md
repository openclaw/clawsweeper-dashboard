## Summary

Keep open: current main still has no command-level `chat.send` byte cap, and this PR is not merge-ready because it adds a compatibility-affecting gateway limit without real behavior proof and still has byte-policy defects.

## What This Changes

The PR adds a 512 KiB UTF-8 `chat.send` message limit in the gateway sanitizer/schema, tests boundary and multibyte cases, and documents the new cap.

## Best Possible Solution

Land a revised gateway hardening change that defines the raw-vs-normalized byte policy, treats schema `maxLength` as character metadata or removes it, covers decomposed Unicode, and includes real gateway/client proof plus upgrade notes.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection and a focused byte-count probe show decomposed Unicode can be over 512 KiB on input and under 512 KiB after NFC normalization. This is not a current-main bug reproduction because current main has no command-level `chat.send` byte cap.

## Solution Assessment

No. The hardening direction is reasonable, but the patch should enforce the intended byte representation before accepting the message and make clear that schema `maxLength` is a character constraint rather than byte enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure raw chat.send bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new cap checks `Buffer.byteLength(normalized, "utf8")` after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB inbound limit; check raw message bytes first or explicitly define a normalized-size policy.
  - confidence: 0.91
- **[P3] Clarify schema maxLength is character-based:** `src/gateway/protocol/schema/logs-chat.ts:41`
  - body: `Type.String({ maxLength })` is a JSON Schema character-count constraint, not a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so this schema/comment should not read as byte-level enforcement.
  - confidence: 0.84
