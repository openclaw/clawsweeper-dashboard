## Summary

Keep open: current main still has no `chat.send` body byte cap, and this external PR is useful but not merge-ready because it measures the cap after NFC normalization, treats schema `maxLength` as a byte guard, and lacks real behavior proof.

## What This Changes

The PR adds a 512 KiB `chat.send` message limit in gateway sanitization, protocol schema, tests, and gateway protocol docs.

## Best Possible Solution

Land a revised gateway hardening change that enforces the cap on raw inbound UTF-8 bytes before normalization, documents schema character semantics, includes regression coverage, and has maintainer approval for the new protocol limit.

## Reproduction Assessment

Yes for the PR defect: source inspection plus a focused Node byte-count check shows decomposed Unicode can exceed 512 KiB on input, shrink below the cap after NFC normalization, and pass the proposed sanitizer. Current main does not yet enforce any `chat.send` message byte cap.

## Solution Assessment

No. The hardening direction is useful, but the implementation should measure raw incoming UTF-8 bytes before normalization and should clarify that schema `maxLength` is character-based rather than byte-based.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure raw chat.send bytes before normalization:** `src/gateway/server-methods/chat.ts:235-238`
  - body: The new cap is computed from `normalized`, after `message.normalize("NFC")`. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it can pass despite the documented 512 KiB request-body cap; measure raw incoming bytes before normalization or explicitly document and test a normalized-size policy.
  - confidence: 0.9
- **[P3] Clarify schema maxLength character semantics:** `src/gateway/protocol/schema/logs-chat.ts:35-39`
  - body: `Type.String({ maxLength })` becomes JSON Schema `maxLength`, which is character-based, while this PR describes a UTF-8 byte limit. Multi-byte messages can exceed 512 KiB and still pass schema validation before the sanitizer sees them, so the schema/comment should not present this as byte enforcement.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive because it adds a DoS guard, and the current diff weakens that guard by measuring bytes after normalization.

Concerns:

- **[medium] Check payload size before NFC normalization:** `src/gateway/server-methods/chat.ts:235`
  - body: The intended DoS guard measures `Buffer.byteLength(normalized, "utf8")`; decomposed Unicode can shrink during NFC normalization, allowing an oversized raw request body to pass the proposed 512 KiB cap.
  - confidence: 0.88
- **[low] Do not treat schema maxLength as byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:39`
  - body: The protocol schema uses JSON Schema `maxLength`, which counts characters rather than UTF-8 bytes, so it is not a first-line 512 KiB byte-size guard for multi-byte content.
  - confidence: 0.82
