## Summary

Keep this open. Current main still has the requested single-file behavior: runtime usage metadata is modeled on allowlist entries and normal gateway, node-host, and macOS allowlist execution can still save `lastUsed*` data back into `exec-approvals.json`; the separate `exec-runtime.json` store does not exist.

## What This Changes

The issue requests a backward-compatible split where immutable exec allowlist policy stays in `exec-approvals.json` and mutable usage tracking moves to a dedicated `exec-runtime.json` file.

## Best Possible Solution

Add a reviewed core hardening change that introduces a backward-compatible `exec-runtime.json` usage store, stops ordinary gateway, node-host, and macOS allowlist execution from writing runtime metadata to `exec-approvals.json`, preserves legacy approvals reads, and updates protocol, CLI, Control UI, macOS UI, docs, and tests to show merged legacy/runtime usage without broadening exec policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no `exec-runtime.json` surface, and `recordAllowlistUse` plus gateway/node/macOS call sites still write usage fields through the approvals store.

## Solution Assessment

Unclear. The requested split is a plausible end state, but the exact implementation should be reviewed by maintainers/secops because it changes a critical storage contract across multiple exec approval surfaces.
