## Summary

Keep open. Current main and release v2026.6.6 still store and update exec usage metadata inside `exec-approvals.json`; related open PRs cover narrower stale-write guards, not the requested runtime-store split, and this needs maintainer/security product review.

## What This Changes

Move mutable exec usage metadata such as `lastUsedAt` and `lastUsedCommand` out of `exec-approvals.json` into a separate runtime store so the approvals file remains policy-only.

## Best Possible Solution

Define and land a maintainer-approved storage contract that separates or transactionally guards runtime usage metadata, preserves legacy `lastUsed*` reads or migration, and updates gateway, node-host, macOS, CLI, protocol, docs, and tests without broadening exec policy.

## Reproduction Assessment

Yes for the remaining storage-contract problem: current source, tests/docs schema, and the latest release still store and write `lastUsed*` metadata through `exec-approvals.json`. I did not run a live multi-container restart or race reproduction in this read-only review.

## Solution Assessment

No as the proposed two-line gateway-only change; the direction is sound, but the best fix needs a reviewed cross-surface storage and migration contract. The open related PRs are narrower stale-write mitigations and do not make this request obsolete.
