## Summary

Keep open: current main still stores and updates exec usage metadata inside `exec-approvals.json`, while the requested `exec-runtime.json` split remains a security-sensitive storage-contract change that needs maintainer/secops design review.

## What This Changes

Separate mutable exec usage tracking fields from immutable exec allowlist rules by writing last-used metadata to a dedicated runtime file.

## Best Possible Solution

Introduce a reviewed, backward-compatible exec runtime usage store and update gateway, node-host, macOS, protocol, CLI/UI, docs, and regression tests while preserving fail-closed exec policy enforcement.

## Reproduction Assessment

Yes for the remaining storage-mixing behavior: current source, docs, protocol schemas, UI, macOS code, and tests still model last-used metadata on approval entries and write it through the approvals store. I did not live-reproduce the multi-container restart/corruption scenario in this read-only review.

## Solution Assessment

Yes in direction, but not as a gateway-only two-line change. The maintainable fix needs a cross-surface storage-contract migration with backward-compatible legacy reads and reviewed security semantics.
