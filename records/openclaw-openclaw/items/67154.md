## Summary

Keep open. Current main and the latest release still model and write last-used exec metadata inside `exec-approvals.json`; related atomic-write and stale-writeback work addresses narrower races, not the requested runtime-store split.

## What This Changes

Requests moving mutable exec last-use tracking into `exec-runtime.json` while leaving `exec-approvals.json` as the allowlist/policy file.

## Best Possible Solution

Introduce a reviewed, backward-compatible exec runtime usage store and update gateway, node-host, macOS, CLI/UI, protocol schemas, docs, and regression coverage without broadening host-exec policy.

## Reproduction Assessment

Yes for the remaining storage-mixing behavior: current source, tests, docs, protocol schema, and latest release evidence all show last-used metadata embedded in and written back to `exec-approvals.json`. I did not run a live multi-container restart/corruption race in this read-only review.

## Solution Assessment

Yes in direction, but not as a gateway-only two-line change. The maintainable solution needs a cross-surface storage-contract change with migration/display semantics and security review.
