## Label Justifications

- `P2`: This is a valid core hardening improvement with real operator impact, but it needs storage-contract design rather than emergency runtime intervention.
- `impact:data-loss`: The issue concerns mutable runtime writeback to the approvals file, which can lose or overwrite allowlist state in concurrent or restart-sensitive deployments.
- `impact:security`: Exec approvals are a host command-execution security boundary, and moving metadata persistence changes enforcement and audit behavior.

## Summary

Keep open. Current main and the latest release still keep exec usage metadata in `exec-approvals.json`; related work covers narrower writeback races, but not the requested runtime-store split.

## What This Changes

Request to move mutable exec usage fields such as `lastUsedAt` and `lastUsedCommand` out of `exec-approvals.json` into a dedicated `exec-runtime.json` store.

## Best Possible Solution

Add a reviewed, backward-compatible `exec-runtime.json` usage store, stop execution paths from writing usage metadata into `exec-approvals.json`, and update CLI/UI/protocol/macOS/docs to merge legacy and new usage metadata without broadening exec policy.

## Reproduction Assessment

Yes, for the remaining storage-contract problem: current source, tests, docs, protocol schema, macOS code, and release `v2026.5.18` still store last-used metadata inside `exec-approvals.json`. I did not run a live multi-container restart or corruption race in this read-only review.

## Solution Assessment

Yes in direction, but not as a two-line gateway change. The maintainable fix needs a reviewed cross-surface storage contract with migration, legacy metadata merge semantics, display updates, and security review.
