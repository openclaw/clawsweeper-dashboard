## Summary

Keep open. Current main and the latest release still store `lastUsedAt` and `lastUsedCommand` directly on exec allowlist entries and still save those updates back to `exec-approvals.json`; the related open PRs cover narrower stale-write guards, not this requested storage split.

## What This Changes

Separate mutable exec runtime usage tracking from the persisted exec approval allowlist by moving last-used metadata out of `exec-approvals.json`.

## Best Possible Solution

Define a maintainer/secops-approved exec usage metadata contract, then update gateway, node-host, CLI/UI/docs/tests so mutable last-used data no longer requires ordinary execution paths to rewrite the approval policy file.

## Reproduction Assessment

Yes, at source level. Current main still defines `lastUsed*` on `ExecAllowlistEntry` and writes those fields through `recordAllowlistUse` back into `exec-approvals.json`; I did not run a live multi-container restart/corruption race in this read-only review.

## Solution Assessment

No for the literal two-line `exec-runtime.json` sidecar as stated. The maintainable fix needs an approved storage, migration, and display contract across all exec approval surfaces.
