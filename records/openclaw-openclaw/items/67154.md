## Summary

Keep open. Current main still mixes exec allowlist policy with mutable usage metadata and still writes allowlist use back to `exec-approvals.json`; the related atomic-write mitigation reduces corruption risk but does not implement the requested two-file storage split.

## What This Changes

Move mutable exec usage metadata such as `lastUsedAt` and `lastUsedCommand` from `exec-approvals.json` into a dedicated `exec-runtime.json` store while preserving existing allowlist reads.

## Best Possible Solution

Introduce a backward-compatible `exec-runtime.json` usage store, keep `exec-approvals.json` as stable policy, merge legacy metadata for display only, and preserve fail-closed host-exec enforcement across gateway, node, and macOS implementations.

## Reproduction Assessment

Yes for the remaining storage-mixing behavior: a focused check can exercise an allowlist hit through `recordAllowlistMatchesUse` and observe `lastUsedAt`/`lastUsedCommand` written back to `exec-approvals.json`. The older truncation race from the related closed bug was not re-run because current main already uses temp-file rename writes.

## Solution Assessment

Yes in direction, but not as a two-line gateway-only change. A separate runtime store is the maintainable end state, provided the implementation includes reviewed migration, display merge behavior, protocol/UI updates, and native parity.
