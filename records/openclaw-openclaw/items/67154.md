## Summary

Keep this open. Current main still stores exec allowlist usage metadata in `exec-approvals.json` and still writes that file during normal allowlist use; the atomic-write mitigation addresses part of the related corruption class, but the requested `exec-runtime.json` split is not implemented.

## What This Changes

The issue requests moving mutable exec usage metadata such as `lastUsedAt` and `lastUsedCommand` from `exec-approvals.json` into a dedicated `exec-runtime.json` store.

## Best Possible Solution

Introduce a maintainer-reviewed, backward-compatible `exec-runtime.json` usage store, stop normal execution paths from writing usage metadata to the allowlist file, and update CLI/UI/docs/tests to read legacy and runtime metadata without broadening exec policy.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: current main still calls `recordAllowlistMatchesUse` after gateway and node allowlist execution, and that helper writes `lastUsedAt` and `lastUsedCommand` back through `saveExecApprovals`.

## Solution Assessment

Unclear as proposed. The two-file direction is likely the right end state, but the safe implementation is broader than a two-line gateway-only change because usage metadata, durable approvals, node-host execution, CLI display, docs, and protocol/UI consumers all share the current approvals schema.
