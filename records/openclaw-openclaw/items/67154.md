## Summary

Keep open. Current main and the latest shipped release still store and update mutable exec usage metadata inside `exec-approvals.json`; the related atomic-write fix and stale-writeback PR address narrower races but do not implement the requested runtime-store split.

## What This Changes

The issue requests moving `lastUsedAt`, `lastUsedCommand`, and related usage metadata out of `exec-approvals.json` into a dedicated runtime store such as `exec-runtime.json`.

## Best Possible Solution

Introduce a reviewed, backward-compatible exec runtime usage store and update gateway, node-host, macOS, CLI/UI, protocol schemas, docs, and regression tests without broadening host-exec policy.

## Reproduction Assessment

Yes for the remaining storage-mixing behavior: current source, docs, and tests show last-used metadata modeled on allowlist entries and written back through `saveExecApprovals`. I did not run a live multi-container restart or corruption race in this read-only review.

## Solution Assessment

Yes in direction, but not as a two-line gateway-only change. The maintainable solution needs a reviewed cross-surface storage-contract change with backward-compatible reads, migration/display semantics, and macOS/node/gateway parity.
