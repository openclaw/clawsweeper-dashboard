## Summary

Keep open. Current main and v2026.6.10 still store last-used exec metadata inside `exec-approvals.json`; the related open PRs cover narrower stale-write/freshness guards, so the requested storage split remains a maintainer/security product decision.

## What This Changes

The issue asks to separate mutable exec usage metadata from the exec approval allowlist by moving last-used fields out of `exec-approvals.json`.

## Best Possible Solution

Keep this issue open until maintainers approve a storage contract that separates or transactionally guards exec usage metadata without broadening exec approval policy or adding unowned JSON sidecar state.
