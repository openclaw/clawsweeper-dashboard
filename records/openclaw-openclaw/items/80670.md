## Summary

Keep open: the PR is a plausible fix for real WebChat transcript latency, but the current patch introduces a blocking attachment-path regression by awaiting transcript media persistence before dispatch, and the matching gateway-methods CI shard is failing with attachment-send timeouts.

## What This Changes

The PR eagerly writes WebChat `chat.send` user turns into the active session transcript, threads a suppression entry id through the reply/Pi runner path, and adds docs, changelog, and gateway/Pi tests for rollover, hooks, reset commands, and duplicate prevention.

## Best Possible Solution

Keep the eager transcript file append for non-busy WebChat turns, but keep media persistence and other slow transcript decoration off the dispatch critical path so ACK-to-dispatch latency does not regress for attachment sends.

## Reproduction Assessment

Yes for the PR blocker: the completed GitHub Actions gateway-methods shard reproduces it with two attachment-send test timeouts, and the diff shows the new pre-dispatch await waits on `persistedImagesPromise`. The original WebChat latency report is supported by the PR body’s redacted live output, but I did not run a local live reproduction.

## Solution Assessment

No, not yet. The eager persistence direction is reasonable, but the implementation should avoid awaiting attachment media persistence before dispatch; a narrower fix would persist the text turn promptly and attach or rewrite media metadata asynchronously or later in the existing post-dispatch path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Keep eager persistence off the attachment dispatch path:** `src/gateway/server-methods/chat.ts:2583`
  - body: This new pre-dispatch await calls `persistUserTranscriptUpdate("eager")`, which awaits `persistedImagesPromise` before `dispatchInboundMessage` runs. For attachment sends, media persistence can be slow or intentionally pending, so the handler never reaches dispatch until saving finishes; the current CI shard is timing out in the existing attachment-send tests that hold `saveMediaBuffer` open. The eager text append needs to avoid waiting on transcript media persistence before dispatch, or keep the media rewrite on a later/background path.
  - confidence: 0.93
