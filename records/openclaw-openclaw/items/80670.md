## Summary

Keep open: the bug-fix direction is plausible and the supplied live-output proof is sufficient, but the current PR head introduces a blocking attachment-send regression by awaiting media persistence before dispatch, with the gateway-methods CI shard failing on that path.

## What This Changes

The PR eagerly persists WebChat `chat.send` user turns to the active session transcript before agent-lane dispatch and threads suppression metadata through the Pi/reply path to avoid duplicate user-turn writes.

## Best Possible Solution

Keep the eager transcript append for eligible WebChat text turns, but move attachment media persistence and media-path transcript decoration off the pre-dispatch critical path so attachments cannot delay dispatch.

## Reproduction Assessment

Yes for the PR blocker: the source path shows pre-dispatch eager persistence awaiting `persistedImagesPromise`, and CI reports the gateway-methods shard failing on attachment-send tests that hold media saving open. The original WebChat latency bug is supported by the PR body's redacted live output, but I did not run that live setup locally.

## Solution Assessment

No, not yet. The current implementation fixes the right symptom for text-only turns, but attachment sends need a narrower design that persists the visible user turn promptly without awaiting media save before dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Keep eager persistence off the attachment dispatch path:** `src/gateway/server-methods/chat.ts:2583`
  - body: This new pre-dispatch await calls `persistUserTranscriptUpdate("eager")`, which awaits `persistedImagesPromise` before `dispatchInboundMessage` runs. Attachment sends can therefore stall dispatch until media saving finishes; the current gateway-methods CI failure is on existing attachment tests that intentionally hold `saveMediaBuffer` open. Persist the text turn without waiting for transcript media persistence, or keep the media rewrite on a later/background path.
