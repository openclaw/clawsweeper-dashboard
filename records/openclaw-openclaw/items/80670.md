## Summary

Keep this PR open: it targets a valid WebChat latency bug, but the current branch still skips eager transcript persistence in the queued same-session path where an earlier run is active.

## What This Changes

The PR adds eager WebChat user-turn transcript persistence before agent-lane dispatch, threads duplicate-write suppression through reply/Pi run parameters, and updates related docs, changelog text, and regression tests.

## Best Possible Solution

Keep the Gateway-owned eager transcript write, but allow it for queued same-session WebChat turns while passing entry-scoped suppression to Pi so chat.history updates immediately without duplicate user entries.

## Reproduction Assessment

Yes. The issue is source-reproducible with high confidence from the serialized same-session follow-up path, and the PR body includes local slow-path logs; I did not execute a live Gateway reproduction in this read-only pass.

## Solution Assessment

No. The overall owner boundary is reasonable, but the current implementation's busy-session guard prevents the central queued-run fix and the active-run test should assert immediate eager persistence instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow eager persistence while a prior run is active:** `src/gateway/server-methods/chat.ts:2328-2334`

That is the slow path this PR is meant to fix: the new chat.send sits behind the serialized session lane, so chat.history stays stale until the prior run releases unless the eager write still happens for queued same-session turns.
