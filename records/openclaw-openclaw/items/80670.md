## Summary

Keep open: the fix has convincing real-environment proof and the gateway/Pi boundary is directionally sound, but the current head still blocks attachment sends by awaiting media persistence before dispatch; the completed gateway-methods CI job failed on that path.

## What This Changes

The PR eagerly persists WebChat user turns before agent dispatch, threads duplicate-write suppression through reply/Pi paths, and adds WebChat/Pi tests, docs, and a changelog entry.

## Best Possible Solution

Keep the eager text transcript append, but decouple attachment media persistence from the pre-dispatch path and leave media enrichment/rewrite on the existing later path.

## Reproduction Assessment

Yes for the PR blocker: the gateway-methods CI job failed on the PR head, and source inspection shows the new pre-dispatch await waits on attachment media persistence. The original WebChat latency bug is supported by the PR body's redacted live output, but I did not run a local live reproduction.

## Solution Assessment

No, not yet. The eager persistence direction is maintainable, but attachment sends must still reach dispatch without waiting for slow media persistence.
