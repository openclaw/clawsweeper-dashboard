## Summary

Keep open: the PR targets a real WebChat latency problem, but the patch currently skips eager persistence exactly when another same-session run is active, which is the slow queued-lane case the PR says it fixes.

## What This Changes

The branch adds eager WebChat user-turn transcript persistence, active-session duplicate-suppression plumbing through reply/Pi runners, docs/changelog updates, and gateway/Pi regression tests.

## Best Possible Solution

Keep the Gateway-owned eager transcript write, but allow it for queued same-session WebChat turns while passing entry-scoped suppression to Pi so the durable history updates immediately without duplicate user entries.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main queues same-session follow-ups behind an active run, and the PR body includes local slow-path logs. I did not execute a live Gateway reproduction in this read-only pass.

## Solution Assessment

No. The overall owner boundary is reasonable, but the current implementation's busy-session guard prevents the central queued-run fix and the active-run test should assert immediate eager persistence instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Allow eager persistence while a prior run is active:** `src/gateway/server-methods/chat.ts:2328-2334`
  - body: This guard makes the eager transcript write return whenever another run for the same session is active. That is the slow path described by this PR: the new `chat.send` sits behind the serialized session lane, so `onAgentRunStart` and the fallback write do not run until the lane releases, leaving `chat.history` stale. The added active-run test currently expects no write for this case, so the latency regression remains.
