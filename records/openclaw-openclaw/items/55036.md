## Summary

Keep open. Current main still intentionally suppresses commentary-phase assistant text before it can reach Discord partial, block, or final delivery, while Discord draft previews only receive text that upstream emits through `onPartialReply` or explicit tool/progress callbacks. The related closed final-result latency report is adjacent but does not cover this early-commentary durability request.

## What This Changes

The issue requests that Discord commit early assistant acknowledgement/commentary as an immediate durable visible message so a gateway restart cannot lose it before the final reply.

## Best Possible Solution

Define an explicit safe user-visible status/progress contract, keep raw `commentary` suppressed, and route only approved status text through embedded subscribe and Discord draft/dispatcher delivery with restart-sensitive regression coverage.

## Reproduction Assessment

Yes. A source-level/unit reproduction is high-confidence: emit a commentary-phase assistant message before tool use and current tests show no partial reply, block reply, or assistant text is materialized, which means Discord has nothing durable to commit before restart.

## Solution Assessment

No, not as raw commentary forwarding. The maintainable fix is a narrower approved-status lane that preserves the shipped planning-text suppression boundary while giving Discord an immediate durable preview/update path.
