## Summary

Keep open. Current main still suppresses commentary-phase assistant text before Discord delivery, while the newer progress-draft path only covers approved tool/work status rather than the requested durable early commentary/status commit.

## What This Changes

Requests that Discord commit early assistant commentary/status text immediately and durably instead of buffering or losing it across gateway restarts.

## Best Possible Solution

Define an explicit safe status/progress contract and persist or stream only approved user-visible status text through embedded subscribe and Discord delivery, while keeping raw commentary suppressed.

## Reproduction Assessment

Yes for the observed current behavior: emit commentary-phase assistant text before tool use and current main returns before onPartialReply, onBlockReply, assistantTexts, or Discord draft delivery. I did not run a live Discord restart, but the source and tests give a high-confidence reproduction path for the loss window.

## Solution Assessment

No, forwarding raw commentary is not the best fix because it would conflict with the current internal-commentary suppression boundary. The better direction is a narrow approved-status lane that can be durable without exposing internal planning text.
