## Summary

Keep open: the fix is on the right path, but the new alias-aware cap lookup is scoped to the Anthropic 1M branch, so canonical Claude CLI routes for non-1M Anthropic models can still miss `models.providers.anthropic` `contextTokens` caps.

## What This Changes

The branch updates context-token resolution and session-store coverage so Claude CLI Anthropic routes prefer configured `contextTokens` caps over the runtime 1M window.

## Best Possible Solution

Apply the runtime-alias-aware configured `contextTokens` lookup to the general provider/model config scan, while keeping the Anthropic 1M `contextWindow` fallback only when no explicit cap exists.

## Reproduction Assessment

Yes, by source inspection: current main uses Claude CLI runtime metadata for session budget lookup, while docs place canonical Anthropic caps under the Anthropic provider. I did not run a live Claude CLI compaction session.

## Solution Assessment

No. The PR handles the Opus/Sonnet 1M override case, but the alias-aware lookup should also cover non-1M Anthropic models routed through `claude-cli`.
