## Summary

Keep open: current main and the latest release still intentionally compact some home paths to `~/...` in OpenClaw-owned display and prompt surfaces, while the only linked fix PR was closed unmerged. Because current tests and docs also accept home-relative paths, this needs a maintainer display-contract decision rather than cleanup closure or an automatic strict bug fix.

## What This Changes

The issue requests that agent-visible file paths use expanded absolute paths instead of `~/...` shorthand.

## Best Possible Solution

Maintainers should define the path display contract first, then implement the narrowest change for the chosen user-facing surface while preserving tool/path compatibility that intentionally accepts `~/...`.

No, not as a high-confidence live reproduction. Source inspection shows OpenClaw still shortens home paths in tool metadata and skill prompt surfaces, but the issue lacks an exact prompt, channel, transcript, or command showing a current final agent response that breaks copy/paste.

## Solution Assessment

Unclear. A blanket post-processor or new environment variable is not clearly the best fix because current code, tests, and docs intentionally use `~/...`; the safer path is a maintainer decision about which output surfaces should prefer expanded paths.
