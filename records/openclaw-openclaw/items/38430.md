## Summary

Current main intentionally compacts home-directory paths in agent tool displays and many generated diagnostics, so the reported behavior remains present. However, making every agent-visible path absolute is a user-facing output-policy change rather than a narrowly defined bug, and the linked environment-variable PR closed unmerged without establishing that a global switch was the right boundary.

## What This Changes

Change agent-visible filesystem references from `~` or `$OPENCLAW_HOME` shorthand to fully expanded absolute paths.

## Best Possible Solution

Define a narrow agent-visible path contract, then update the responsible tool-display or prompt boundary while retaining compact paths for unrelated local diagnostics unless maintainers explicitly choose a repository-wide convention.

## Reproduction Assessment

Yes at the source level: current agent tool-display code and shared display helpers deterministically compact effective-home paths. The report still lacks a specific model, prompt, and channel transcript showing which user-visible response surface motivated the broader request.

## Solution Assessment

No, the proposed global environment variable or blanket response post-processing is not the best established solution. It would add configuration and affect unrelated displays; maintainers should first choose the intended agent-visible boundary and privacy tradeoff.
