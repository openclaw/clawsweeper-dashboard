## Summary

Keep open: current main and the latest release still deliberately compact home paths to `~` in shared helpers, tool summaries, skill prompts, media directives, approval displays, and clients. This is not closeable as implemented; the remaining decision is a maintainer path-presentation policy, not a narrow stale cleanup.

## What This Changes

The issue asks OpenClaw to stop showing `~/...` shorthand in agent file-path responses and use full absolute paths instead.

## Best Possible Solution

Define a per-surface path-presentation contract for assistant prose, tool/client summaries, skill prompt locations, media directives, approval UI, and sandbox paths, then update only the surfaces that should stop using home-relative display.

## Reproduction Assessment

No, not for the full end-to-end agent-prose behavior. Source inspection does show deterministic `~` compaction or preservation in the shared helpers, tool summaries, skill prompts, media handling, approval display, and native/web clients.

## Solution Assessment

Unclear. A targeted maintainer-defined path-presentation policy is safer than reviving the closed env-var PR or applying a blanket output post-processor across surfaces that currently rely on `~`.
