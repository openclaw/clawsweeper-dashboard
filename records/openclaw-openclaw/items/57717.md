The PR adds Slack markdown-table extraction, Block Kit table attachment/fallback sending, related tests, and changes table-mode defaults/config metadata.

## Best Possible Solution

Port the Slack table send path onto current main while preserving plugin-metadata defaults, scoping block mode to attachment-capable sends or adding slash attachment support, fixing truncation indicators, and adding redacted live Slack proof.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR head returns block mode for all Slack callers while slash replies still send text-only responses, and block-mode IR collection removes table text unless metadata is consumed. I did not run live Slack in this read-only review.

## Solution Assessment

No. The Slack send-path direction is plausible, but the branch needs a current-main rebase, scoped default behavior, row-loss accounting, and real Slack proof before this is the maintainable merge path.

Full review comments:

- **[P1] Keep block mode out of text-only Slack replies:** `src/config/markdown-tables.ts:69-72`
  - body: `resolveMarkdownTableMode` now returns `"block"` for Slack globally, but slash-command replies still render through `markdownToSlackMrkdwnChunks` and call `respond({ text })` without consuming table metadata.   - confidence: 0.91
- **[P1] Preserve registry-backed table defaults:** `src/config/markdown-tables.ts:17-22`
  - body: Current main derives `DEFAULT_TABLE_MODES` from channel plugin metadata and invalidates it with the registry version; this branch reintroduces a static map. Rebasing this as-is would drop defaults owned by plugin metadata and create a second source of truth for channel table behavior.
  - confidence: 0.88
- **[P2] Report the row evicted for column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: For a table already at `SLACK_MAX_TABLE_ROWS` with only columns truncated, this branch pops the last row to make room for the `+N more columns` indicator but does not include `+1 more rows`. That silently drops a data row in the delivered table, so the indicator should either report the row loss or choose a fallback that preserves all rows.
  - confidence: 0.86
