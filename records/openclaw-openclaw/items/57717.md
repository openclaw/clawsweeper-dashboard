## Summary

Keep open: the Slack Block Kit table send-path work is still useful and not implemented on current main, but this PR still has blocking correctness issues and lacks external real behavior proof before merge.

## What This Changes

The PR wires Slack markdown table extraction into `sendMessageSlack`, sends single tables as Block Kit table attachments, falls back for multiple tables, changes Slack table defaults, and adds focused Slack/config tests.

## Best Possible Solution

Land a rebased Slack implementation that uses current plugin/default contracts, enables Block Kit table attachments only on metadata-consuming paths, reports truncation honestly, and includes focused tests plus real Slack proof.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review blockers: the PR head lets Slack slash replies receive `block` mode while that path only sends text chunks, and block mode removes inline table text unless metadata is consumed.

## Solution Assessment

No. The direction is maintainable, but the diff should scope block mode to `sendMessageSlack` or teach slash replies to send table metadata, and it should report any row evicted to fit a column truncation indicator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep slash replies out of block table mode:** `src/config/markdown-tables.ts:18`
  - body: Changing Slack's default table mode to `block` affects slash-command replies too. `deliverSlackSlashReplies` still renders with `markdownToSlackMrkdwnChunks(...)` and sends only text, so table metadata collected in block mode is never attached and table content can disappear from slash replies. Scope block mode to the send path or coerce slash replies back to `code` until they consume the metadata.
  - confidence: 0.91
- **[P2] Report rows evicted for column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: When column truncation is present and the table already fills the row cap, this branch pops the last data row before adding the indicator while `truncatedRows` remains zero. The output then reports only missing columns even though a row was dropped; count that eviction as row truncation or skip the indicator when no row slot is available.
  - confidence: 0.86
