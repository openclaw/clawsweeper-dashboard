## Summary

Keep open: the Slack table send path is not implemented on current main, but this PR head is conflicting and still has source-provable blockers plus no inspectable real Slack proof.

## What This Changes

The PR wires Slack markdown-table extraction into `sendMessageSlack`, adds Block Kit table attachment and fallback helpers/tests, and changes Slack table-mode defaults while carrying unrelated import/formatting churn.

## Best Possible Solution

Land a rebased Slack-owned implementation that scopes block table mode to attachment-capable paths or teaches slash replies to deliver table attachments, fixes truncation accounting, removes unrelated churn, and includes redacted Slack proof.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows slash replies pass block mode into a text-only renderer/respond path, and the table helper deterministically pops a row for full column-truncated tables. I did not run live Slack in this read-only review.

## Solution Assessment

No. The direction is useful, but the maintainable path is to rebase over current main, scope block rendering to Slack paths that can send table attachments or add attachment delivery to slash replies, and fix truncation accounting before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep slash replies out of block table mode:** `src/config/markdown-tables.ts:17-18`
  - body: Setting Slack's default table mode to `block` also feeds slash-command replies, but that path still renders with `markdownToSlackMrkdwnChunks` and sends `respond({ text })` only. Mixed replies with markdown tables can lose the table unless slash replies consume table metadata or coerce back to `code`.
  - confidence: 0.91
- **[P2] Report row loss when adding column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: For a table that already fills Slack's row cap and only exceeds the column cap, this `pop()` removes the last data row before adding the indicator while `truncatedRows` remains zero. Count that eviction, skip the indicator when full, or reserve capacity so row loss is not hidden.
  - confidence: 0.86
