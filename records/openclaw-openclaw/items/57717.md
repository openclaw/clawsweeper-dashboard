## Summary

Keep open: the Slack table-send feature is not implemented on current main, and this PR still has source-proven correctness blockers plus missing real Slack behavior proof before merge.

## What This Changes

The PR adds Slack markdown-table extraction, Block Kit table attachments for single-table sends, multi-table code fallback, Slack table-mode default changes, and focused Slack/config tests, alongside unrelated import/formatting churn.

## Best Possible Solution

Land a rebased Slack implementation that scopes block table mode to metadata-consuming send paths or equips slash replies to send table attachments, reports truncation accurately, removes unrelated churn, and includes redacted Slack proof.

## Reproduction Assessment

Yes. Source inspection on the PR head shows Slack slash replies receive block table mode while still responding with text only, and the column-only row-cap truncation path can be reasoned directly from the table helper.

## Solution Assessment

No. The direction is plausible, but the current diff should scope block mode or add metadata delivery for slash replies, fix truncation reporting, and provide real Slack proof before it is the maintainable merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep slash replies out of block table mode:** `src/config/markdown-tables.ts:18`
  - body: Setting Slack's default table mode to block also feeds block mode into slash-command replies. That path still renders with `markdownToSlackMrkdwnChunks` and responds with text only; because block mode collects tables as metadata instead of inline table text, slash replies can drop table content. Scope block mode to `sendMessageSlack` or coerce slash replies back to code until they consume table metadata.
  - confidence: 0.91
- **[P2] Report row loss when adding column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: For a table that already fills Slack's row cap and only exceeds the column cap, this `pop()` removes the last data row before adding the indicator while `truncatedRows` remains zero. The output reports omitted columns but hides that a row was dropped; count that eviction, skip the indicator when full, or reserve capacity explicitly.
  - confidence: 0.86
