## Summary

Keep open: current main still lacks Slack table attachment delivery, and this PR remains a useful implementation candidate, but the current head has source-reproducible Slack blockers and missing real Slack behavior proof before merge.

## What This Changes

The PR wires markdown table extraction into Slack outbound sends so one table can be posted as a Block Kit table attachment, adds multi-table fallback/tests, changes Slack table-mode defaults, and includes unrelated formatting/import churn outside Slack.

## Best Possible Solution

Land a rebased Slack implementation that scopes block mode to metadata-consuming send paths or teaches slash replies to send table attachments, reports truncation accurately, removes unrelated churn, and includes focused tests plus redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection on the PR head shows Slack slash replies receive Slack's resolved block table mode but only send text, while block mode collects table metadata instead of rendering inline table text.

## Solution Assessment

No. The direction is plausible, but the current diff should scope block mode or add metadata delivery for slash replies, fix truncation reporting, and provide real Slack proof before it is the maintainable merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep slash replies out of block table mode:** `src/config/markdown-tables.ts:17-18`
  - body: Setting Slack's default table mode to `block` lets slash-command replies receive block mode too. That path still renders with `markdownToSlackMrkdwnChunks(...)` and responds with text only, while block mode collects tables as metadata instead of inline text, so table content can be dropped from slash replies. Scope block mode to `sendMessageSlack` or coerce slash replies back to `code` until they consume table metadata.
  - confidence: 0.91
- **[P2] Report rows evicted for column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: When a table already fills the row cap and only columns are truncated, this branch pops the last data row before adding the indicator while `truncatedRows` stays zero. The output then reports missing columns but hides that a row was dropped; count the eviction as row truncation or skip the indicator when no row slot remains.
  - confidence: 0.86
