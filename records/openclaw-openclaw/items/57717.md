## Summary

Keep this PR open. Current main still does not implement the Slack markdown-table Block Kit send path, and the PR remains useful source work, but the proposed patch is not merge-ready because Slack block table mode would leak into slash-command replies that do not consume extracted table metadata, and one truncation path can silently replace a data row with an indicator.

## What This Changes

The PR adds Slack markdown table extraction, Block Kit table attachments in `sendMessageSlack`, fallback helpers/tests, and changes Slack table-mode defaults, with some unrelated formatting/generated-file churn.

## Best Possible Solution

Land a narrow, rebased Slack implementation that consumes block-mode table metadata only on send paths that actually emit Block Kit table attachments, keeps safe text rendering for slash and non-Slack paths, and covers single-table, multi-table, media, truncation, and fallback behavior with targeted tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main sends Slack markdown tables through `markdownToSlackMrkdwnChunks` only, and the PR's global Slack block default would route slash replies into a renderer that drops block-mode table text without sending the collected metadata.

## Solution Assessment

No. The core idea is the right direction, but the PR should scope block mode to the Slack message send path or teach slash replies to consume table metadata, and it should report any row evicted to make room for a truncation indicator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep Slack slash replies out of block table mode:** `src/config/markdown-tables.ts:16`
  - body: Adding Slack to `DEFAULT_TABLE_MODES` makes `resolveMarkdownTableMode` return `"block"` for every Slack caller. The slash reply path still renders through `markdownToSlackMrkdwnChunks(..., { tableMode })`, and block mode removes tables into metadata that this path never sends, so slash-command replies containing tables can lose the table content. Scope block mode to `sendMessageSlack` or coerce slash replies back to `code` until they attach the collected table metadata.
  - confidence: 0.91
- **[P2] Report row loss when replacing a capped row with the indicator:** `extensions/slack/src/block-kit-tables.ts:96-99`
  - body: When column-only truncation is present and the table already fills Slack's row cap, this `rows.pop()` removes the last data row before adding the indicator, but `truncatedRows` is still zero so the indicator only mentions columns. That silently drops a row; reserve/count the indicator as row truncation or skip it when there is no room.
  - confidence: 0.86
