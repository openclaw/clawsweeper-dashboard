## Summary

Keep open: the PR targets a real Slack table rendering gap and builds on the merged table-block seam, but the current diff still risks losing tables in Slack slash replies, under-reports row truncation in one Block Kit edge case, and lacks inspectable real Slack behavior proof.

## What This Changes

The PR wires Slack markdown table extraction into `sendMessageSlack`, sends single tables as Block Kit table attachments, falls back for multiple tables, changes Slack table defaults, and adds focused Slack/config tests.

## Best Possible Solution

Land a rebased Slack implementation that enables Block Kit table attachments only on paths that consume table metadata, keeps slash and non-Slack text rendering safe, reports truncation honestly, and includes focused tests plus real Slack proof.

## Reproduction Assessment

Yes. Source inspection shows current main lacks table attachment delivery, and the PR head routes Slack slash replies into block table mode without a metadata-consuming attachment path.

## Solution Assessment

No. The direction is maintainable, but the diff should scope block mode to `sendMessageSlack` or update slash replies to consume table metadata, and it should count any row evicted to fit a truncation indicator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep slash replies out of block table mode:** `src/config/markdown-tables.ts:18`
  - body: Adding Slack to the default block table modes makes slash-command replies receive `"block"` too, but `deliverSlackSlashReplies` still renders with `markdownToSlackMrkdwnChunks(...)` and never sends collected table metadata. In block mode the IR removes table text, so slash replies containing tables can lose content; scope block mode to `sendMessageSlack` or coerce slash replies back to `code` until they attach tables.
  - confidence: 0.91
- **[P2] Report rows evicted for column indicators:** `extensions/slack/src/block-kit-tables.ts:100-102`
  - body: When column truncation is present and the table already fills the row cap, this branch pops the last data row before adding the indicator while `truncatedRows` remains zero. The output reports only missing columns even though a row was dropped; count the eviction as row truncation or skip the indicator when no row slot is available.
  - confidence: 0.86
