## Summary

Keep open. The PR fixes a real provider-precedence gap, but it also changes shipped fallback behavior for configured web-search providers, so it needs maintainer compatibility approval or a compatibility-preserving update before merge.

## What This Changes

This PR makes agent `web_search` pass explicit `tools.web.search.provider` through provider selection, reloads missing explicit providers from plugin discovery, documents hard-route behavior, and adds regression tests.

## Best Possible Solution

Have maintainers explicitly approve hard-route/fail-closed `web_search` provider semantics, or require a compatibility-preserving fallback default before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main lets runtime web-search metadata win before configured provider routing, and the latest release still tests fallback for an unknown configured provider; I did not run a live repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the compatibility choice is accepted. The code is narrow if hard-route semantics are approved, but preserving fallback by default is safer until maintainers accept the runtime failure mode.
