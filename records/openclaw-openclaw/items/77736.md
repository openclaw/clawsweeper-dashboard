## Summary

Keep this PR open, but do not merge it as-is: it fixes a real provider-routing gap, while still changing shipped fallback behavior and conflicting with the runtime-first search direction in a related open PR.

## What This Changes

This PR forwards explicit `tools.web.search.provider` into agent `web_search`, reloads missing explicit providers from plugin discovery, documents hard-route behavior, and adds regression tests.

## Best Possible Solution

Land a maintainer-approved routing contract that forwards explicit provider IDs without surprising upgrades, or explicitly approve hard-route/fail-closed behavior while reconciling the runtime-first agent search boundary.

## Reproduction Assessment

Yes, at source level: current main does not pass configured provider IDs from the agent tool into `runWebSearch`, and the PR body shows a production-module terminal proof after the patch. I did not run a live gateway or third-party search repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The provider forwarding is narrow, but hard-route/fail-closed behavior and the agent search runtime-first boundary need explicit approval or adjustment before this is the best fix.
