## Summary

Keep open for maintainer review: the focused fix is well supported by source inspection, tests, and terminal proof, but it deliberately makes explicit web-search provider config fail closed and needs maintainer acceptance before merge.

## What This Changes

The PR makes explicit `tools.web.search.provider` routing authoritative for agent `web_search`, adds narrow runtime/provider-discovery fallback for explicit misses, and documents/tests the fail-closed behavior.

## Best Possible Solution

Land the focused fix if maintainers accept explicit `web_search` provider config as a hard route, keeping the new regression coverage and docs for stale metadata and missing runtime-registry cases.

## Reproduction Assessment

Yes. Source inspection on current main shows the agent wrapper does not pass the configured provider as `providerId`, and the runtime context checks stale runtime metadata before explicit config.

## Solution Assessment

Yes, with a maintainer compatibility call. Passing the explicit config value as `providerId` and reloading plugin providers only for explicit runtime misses is narrow; preserving silent fallback is the safer alternative only if maintainers reject hard routing.
