## Label Justifications

- `P2`: This is a focused provider-routing bug fix with limited blast radius, but it affects a user-facing configuration path.
- `merge-risk: 🚨 compatibility`: Fail-closed explicit provider routing can break setups that previously worked only because OpenClaw silently fell back to another provider.
- `merge-risk: 🚨 auth-provider`: The PR deliberately changes provider and credential routing precedence for `web_search` when explicit config and runtime metadata disagree.

## Summary

Keep open for maintainer review: current main still lets stale runtime web-search metadata outrank an explicit provider in the agent path, while this PR fixes that path with focused code, tests, docs, and sufficient terminal proof. The remaining decision is whether maintainers accept explicit `web_search` provider config as a hard route that fails closed when unavailable.

## What This Changes

The PR makes explicit `tools.web.search.provider` values authoritative for agent `web_search` routing, adds fallback plugin-provider discovery for explicit runtime misses, and documents/tests the fail-closed behavior.

## Best Possible Solution

Land the focused PR if maintainers accept explicit `web_search` provider config as a hard route, preserving the new docs and regression coverage for stale metadata and missing runtime-registry provider cases.

## Reproduction Assessment

Yes. Source inspection on current main shows the agent wrapper does not pass the configured provider as `providerId`, and the runtime context checks stale runtime metadata before explicit config.

## Solution Assessment

Yes, with a maintainer compatibility call. Passing the explicit config value as `providerId` and reloading plugin providers only for explicit runtime misses is a narrow fix; preserving silent fallback would be the safer alternative only if maintainers reject hard routing.
