## Summary

Keep open: the PR addresses a real source-level provider precedence gap, but it is not merge-ready because it changes shipped fallback behavior for configured web_search providers and still conflicts with the runtime-first agent search direction in https://github.com/openclaw/openclaw/pull/88684.

## What This Changes

This PR forwards configured `tools.web.search.provider` into agent `web_search`, reloads missing explicit providers from plugin discovery, documents hard-route behavior, and adds regression tests.

## Best Possible Solution

Land one maintainer-approved web_search routing contract that forwards explicit provider IDs downstream while preserving or explicitly approving fallback behavior and reconciling with the runtime-first agent gate.

## Reproduction Assessment

Yes, at source level: current main does not pass configured provider IDs from the agent tool into `runWebSearch`, and release coverage proves the shipped fallback behavior that this PR changes. I did not run a live gateway or third-party search repro in this read-only review.

## Solution Assessment

No, not yet. Forwarding the configured provider is a plausible narrow fix, but the hard-route/fail-closed behavior and agent runtime-first boundary need maintainer approval or adjustment before this is the best solution.
