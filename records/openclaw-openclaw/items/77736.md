## Summary

Keep open: current main still has the explicit provider routing bug, and this PR is a narrow fix with the earlier proof, changelog, and regression-test gaps addressed; it should proceed through exact-head CI and maintainer review rather than cleanup closure.

## What This Changes

The PR makes explicit `tools.web.search.provider` values authoritative for agent `web_search` routing, adds a plugin-provider reload fallback when the runtime set misses that explicit provider, and updates changelog plus regression coverage.

## Best Possible Solution

Land the narrow routing fix after exact-head CI completes, preserving explicit config precedence with the added regression tests and changelog entry.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path with `tools.web.search.provider` set to a custom provider while stale runtime metadata selects another provider and the runtime provider set lacks the custom provider. Current main does not pass `providerId` and lets runtime metadata choose discovery mode first.

## Solution Assessment

Yes: making the configured provider id authoritative at the agent wrapper/context boundary and reloading plugin providers only for explicit-provider misses is the narrow maintainable fix. A broader registry policy change is not needed for this reported path.
