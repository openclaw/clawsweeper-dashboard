## What This Changes

The PR makes configured `tools.web.search.provider` values authoritative for agent `web_search`, adds fallback provider discovery for explicit runtime misses, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land a rebased version of this explicit-provider precedence fix with the regression coverage, while making the stale or unavailable configured-provider behavior an intentional maintainer choice.

## Reproduction Assessment

Yes, source-reproducible. On current main, configure `tools.web.search.provider` for a custom provider while `runtimeWebSearch.selectedProvider` points at stale metadata; the agent wrapper does not pass the configured provider as `providerId`, and the runtime context checks runtime metadata first.

## Solution Assessment

Yes, with one compatibility caveat. Passing the configured provider id at the agent boundary and giving config precedence in runtime context is the narrow fix, but maintainers should explicitly accept the fail-closed behavior for unavailable configured providers before merge.
