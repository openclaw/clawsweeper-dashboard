## Summary

This PR needs revision before merge: it can replace successful existing Telegram delivery with a hard failure when a rich payload is rejected, and its grammY capability check repeatedly probes unsupported custom endpoints. The separate production hotfix report is useful concept evidence, but it does not prove the current branch head across the expanded transport surfaces.

## What This Changes

The PR adds raw Bot API 10.1 rich-message send, edit, and draft transports across Telegram outbound replies, pairing, stream previews, and progress flows, with selective fallback to established HTML/plain methods.

## Best Possible Solution

Ship a bounded single-chunk final-message rich path with safe pre-delivery rejection fallback and per-client endpoint capability state, then expand edit, pairing, preview, and draft surfaces only after parameter-parity and live Telegram proof.

## Reproduction Assessment

Yes for the existing lack of native rich-table rendering: current main uses the established Telegram HTML/plain path, while Bot API 10.1 now offers rich markdown tables. No real setup currently proves the complete PR head.

## Solution Assessment

No. The upstream rich-message method is the appropriate primitive, but enabling it broadly across dynamically exposed raw methods and several unproven delivery surfaces is less safe than a bounded final-message rollout.
