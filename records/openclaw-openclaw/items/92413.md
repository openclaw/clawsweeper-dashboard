## Summary

Keep this PR open, but do not merge it yet: grammY’s dynamic raw API makes every method appear available, while the fallback classifier does not recognize the normal 404 response from Bot API endpoints that have not upgraded to 10.1, so existing custom-endpoint users can lose formatted-message delivery.

## What This Changes

The PR adds raw Telegram Bot API 10.1 rich-message send, edit, and draft transports with fallback to existing HTML/plain behavior across normal delivery, DM pairing, streaming preview, and tool-progress paths.

## Best Possible Solution

Keep the temporary shim narrow, classify realistic unsupported-method or unsupported-parameter responses across all rich operations, preserve unrelated errors, and prove both Bot API 10.1 success and lagging-endpoint fallback on real Telegram setups.

## Reproduction Assessment

Yes, from source: use a custom or lagging apiRoot that returns the normal 404 response for sendRichMessage; grammY still exposes the dynamic raw function, but the PR’s narrow regex does not classify that response for fallback. A live endpoint reproduction was not run.

## Solution Assessment

No, not as written. A temporary dependency-lag shim is reasonable, but its capability detection and fallback must protect existing endpoint compatibility before it becomes the preferred transport.
