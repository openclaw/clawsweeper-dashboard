## Summary

The exact head still broadens Telegram delivery across five transport families, and the central fallback defects from the prior review remain. Live proof demonstrates rich single-chunk send/edit and visible table rendering, but it does not establish safe delivery when a supported rich method rejects a payload or justify changing pairing, preview, and progress-draft transports.

## What This Changes

Adds a typed Telegram Bot API 10.1 rich-message compatibility layer and adopts it for final replies, direct send/edit, DM pairing replies, stream previews, and native tool-progress drafts, with selected fallback to existing transports.

## Best Possible Solution

Limit rich-message adoption to the proven single-chunk Markdown send/edit path, and implement an explicit rich → existing HTML → plain fallback sequence that preserves reply, topic, link-preview, notification, and markup parameters; leave pairing, previews, and progress drafts on their current transports until independently required and proven.

## Reproduction Assessment

Yes for the patch defect: expose a rich method that rejects a supported payload with a non-404 Bot API error while the existing HTML sender succeeds; source inspection shows the exact-head final-reply path rethrows before legacy fallback. The requested rich-table capability itself is a feature, not a current-main bug.

## Solution Assessment

No. A typed compatibility shim is reasonable, but adopting it across five transport families before proving fallback behavior is broader than necessary; the narrow single-chunk final send/edit path with explicit rich-to-HTML-to-plain fallback is safer and easier to maintain.
