## Summary

Keep this PR open. Current main still has the clone-based hook wrapper while the bootstrap hook contract and consumer expect mutable context, so the proposed fix is useful and narrowly targeted; it still needs the required changelog entry and normal CI/Testbox proof before merge.

## What This Changes

The PR changes `src/plugins/registry.ts` so plugin hook handlers receive the original hook context with temporary `pluginConfig` injection, and adds bootstrap-hook regression coverage in `src/agents/bootstrap-hooks.test.ts`.

## Best Possible Solution

Land this PR or an equivalent narrow patch after adding a single-line Unreleased changelog entry and completing CI/Testbox validation, preserving mutable hook-context behavior while keeping `pluginConfig` scoped to each handler call.

## Reproduction Assessment

Yes. Register an `agent:bootstrap` hook through `api.registerHook` that assigns `event.context.bootstrapFiles`, then call the bootstrap resolver; current main sends the assignment to the wrapper's cloned context while core reads the original event context afterward.

## Solution Assessment

Yes. Reusing the original context and restoring `pluginConfig` in `finally` is the smallest compatible repair for the documented mutable hook contract; replacing mutation with a return-value API would be a larger plugin-contract change.
