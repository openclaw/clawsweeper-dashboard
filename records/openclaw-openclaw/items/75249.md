## Summary

Keep this PR open. Current main still clones plugin hook context before dispatch, which drops mutable `agent:bootstrap` context reassignments, while the PR’s code/test direction matches the documented mutable hook contract; it needs the required changelog entry and normal CI/maintainer review before merge.

## What This Changes

The PR changes `src/plugins/registry.ts` so registered plugin hook handlers receive the original context with temporary `pluginConfig` injection, and adds `src/agents/bootstrap-hooks.test.ts` regression coverage for bootstrap mutation preservation and cleanup behavior.

## Best Possible Solution

Land this PR or an equivalent narrow change after adding the changelog entry and CI proof, preserving mutable hook-context behavior while restoring/removing per-handler `pluginConfig` after each plugin hook call.

## Reproduction Assessment

Yes. Register an `agent:bootstrap` hook through `api.registerHook` that assigns `event.context.bootstrapFiles`, then call `applyBootstrapHookOverrides`; current main sends that assignment to the cloned wrapper context while core reads the original event context afterward.

## Solution Assessment

Yes for the code direction. Passing the original context and restoring `pluginConfig` in `finally` is the narrowest compatible fix for the existing mutable hook contract; replacing the API with immutable return values would be a larger product/API change.
