## Summary

Keep open. Current main still captures the original provider message ID when enqueueing, but collect-mode batching and queued followup execution do not propagate it to the collected prompt or runtime `currentMessageId`; the two linked fix attempts were closed unmerged.

## What This Changes

Preserve original platform `message_id` for busy-session queued messages so native reply APIs can target the queued message.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime `currentMessageId`, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu tenant. Queue a Feishu group-topic message while a collect-mode run is active: the queued run captures `messageId`, but collect drain creates a synthetic followup without it and the runner never passes it to `currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path because Feishu already records original inbound IDs and consumes `currentMessageId` for native reply threading.
