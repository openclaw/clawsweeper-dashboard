## Summary

Keep open. Current main and latest release still lose the queued provider message ID across collect-mode synthetic followups and followup delivery, so Feishu/Lark native reply APIs can still receive unusable placeholder IDs.

## What This Changes

Queued messages should preserve the original platform message_id so agents can reply to the specific queued platform message.

## Best Possible Solution

Preserve the normalized original queued messageId through collect drain, synthetic FollowupRun/currentMessageId, embedded runtime context, and followup delivery, with regression coverage and coordination with the related open threading work.

## Reproduction Assessment

Yes. A high-confidence source path is to enqueue same-route collect/followup messages with messageId and inspect that the collected synthetic followup, queued embedded runtime call, and final delivery threading call do not carry that ID.

## Solution Assessment

Yes. The best fix is shared queue/followup metadata propagation with tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
