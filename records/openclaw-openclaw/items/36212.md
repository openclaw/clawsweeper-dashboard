## Summary

Keep open. Current main still captures queued platform IDs at enqueue, but collect-mode batching and queued runtime/delivery paths omit them where agent tools and reply-threading need `currentMessageId`, so the Feishu/Lark reply-target problem remains valid.

## What This Changes

The issue requests preserving original platform message IDs for queued followup messages so platform-native reply APIs can target the queued message.

## Best Possible Solution

Preserve the normalized original queued `messageId` in shared queue/followup propagation, expose per-item IDs safely where the agent can reference them, and pass the ID into queued runtime and delivery `currentMessageId` paths without Feishu-specific logic or new configuration.

## Reproduction Assessment

Yes. A high-confidence source path is to enqueue same-route collect-mode followups with `messageId`, then inspect that the collected synthetic followup plus queued runtime and delivery calls omit that ID.

## Solution Assessment

Yes. The narrowest maintainable fix is shared queue/followup metadata propagation because Feishu already records original inbound IDs and downstream native reply targeting already consumes reply IDs/current-message context.
