## Label Justifications

- `impact:message-loss`: Queued replies can lose the platform reply target, causing channel-native replies to be sent without the intended message context or fail against provider APIs.

## Summary

Keep open. Current main still loses the unambiguous queued `messageId` across collect-mode batching: enqueue records the platform ID, but the collect drain creates a synthetic followup without carrying that field, so queued reply APIs cannot rely on `currentMessageId` for the original platform message.

## What This Changes

Queued messages should retain the original platform `message_id` so agents can reply to specific queued messages through channel-native APIs.

## Best Possible Solution

Add a narrow queue-layer fix that preserves per-item platform `messageId` through collect batches and keeps `currentMessageId` populated for unambiguous single queued turns, with focused queue and Feishu regression tests.

## Reproduction Assessment

Yes for source reproduction: enqueue records `FollowupRun.messageId`, but collect-mode batching drains a synthetic run that omits it. A live Feishu reproduction was not run in this read-only pass.

## Solution Assessment

Yes, the best fix belongs in the shared queue/followup layer rather than in Feishu-only workaround code. The existing Feishu prompt hint is useful context but does not preserve the generic runtime metadata path.
