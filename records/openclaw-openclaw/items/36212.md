## Summary

Keep open. Current main still captures the original provider message ID when a followup is queued, but collect-mode drain does not carry that ID into the collected prompt or synthetic followup run, and queued followup execution does not pass it as runtime `currentMessageId`; the two linked fix PRs were closed unmerged.

## What This Changes

The issue asks for queued Feishu/Lark messages to retain their original platform `message_id` so native reply APIs can target the queued message.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime `currentMessageId`, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, as a source-reproducible path rather than a live Feishu run: enqueue a Feishu group-topic message while collect-mode is active, then observe that current main captures `FollowupRun.messageId` but drops it from collect prompt metadata and `runEmbeddedPiAgent.currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path because Feishu already records original inbound IDs and uses runtime `currentMessageId` for native reply/thread targeting.
