## Summary

Keep open. Current main still captures the original provider message ID at enqueue, but the collect/followup path does not propagate it into trusted queued-item metadata, runtime `currentMessageId`, or final reply threading; the related fix PRs are closed and unmerged.

## What This Changes

Queued/busy message handling should preserve the original platform `message_id` so agents can use platform-native reply APIs for queued messages.

## Best Possible Solution

Add shared queue/runtime propagation for original queued message IDs, including trusted per-item collect metadata and `currentMessageId` forwarding for unambiguous queued runs, without Feishu-specific policy or new configuration.

## Reproduction Assessment

Yes, as a source-level reproduction. Queue an inbound Feishu/group message while a run is active in collect mode: the queued run captures `messageId`, but the collect drain and followup runner do not pass that ID into prompt metadata or `currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup path; Feishu already preserves the platform ID at ingress, so a Feishu-only send workaround would leave the runtime contract gap in place.
