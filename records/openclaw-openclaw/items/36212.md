## Summary

Keep open. Current main still captures the inbound Feishu provider id and stores it on the queued run, but shared collect, summary-drain, runtime, and followup-delivery paths do not carry that id through for prompt metadata, runtime `currentMessageId`, or reply threading; the directly related fix PR is closed and unmerged.

## What This Changes

Queued busy-session messages should carry the original provider `message_id` through collect/followup runtime and delivery so platform reply APIs can target them.

## Best Possible Solution

Implement a narrow, channel-agnostic queue/followup propagation fix that preserves trusted queued `messageId` through collect prompts, summary drains, embedded runtime `currentMessageId`, and followup delivery threading without adding Feishu-only policy or new config.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to enqueue busy-session followup runs with provider `messageId` in collect or summary-drain mode, then inspect the synthetic followup and runtime call; current main drops the id before prompt metadata, runtime `currentMessageId`, and delivery threading.

## Solution Assessment

Yes. The narrow maintainable fix is in shared queue/followup propagation because Feishu ingress already supplies the provider id and the loss happens in core followup plumbing before channel delivery.

## Repair Work Prompt

Fix #36212 with a narrow channel-agnostic queue/followup message-id propagation change. Observable bug: when a busy agent drains queued messages, the original provider `message_id` is available at ingress and initial queue creation but is not preserved through collect-mode prompts, summary-drain followups, embedded runtime `currentMessageId`, or followup delivery threading, so platform-native reply APIs such as Feishu `im.message.reply` cannot target queued messages. Expected boundary: normalize/preserve queued `FollowupRun.messageId`; include trusted per-item `message_id` metadata in collect prompts when available; carry an appropriate queued `messageId` through collect and overflow-summary synthetic followups; pass it to `runEmbeddedPiAgent` as `currentMessageId`; and make followup delivery apply reply threading with that id. Do not add a Feishu-only workaround, new config option, queue dedupe policy change, routing policy change, or channel adapter API change unless a regression proves it is required. Add/update focused regression tests and a user-facing changelog fix entry. Related refs: #36212 and closed unmerged PR #36235.
