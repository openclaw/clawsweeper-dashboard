## Summary

Keep open. Current main still preserves the inbound Feishu provider id at ingress and initial queue creation, but the collect drain, overflow-summary drain, followup runtime, and followup delivery paths do not carry that id through as queued-message metadata or `currentMessageId`. The directly related PR #36235 is closed and unmerged, so there is no merged canonical fix.

## What This Changes

Preserve the original platform `message_id` for queued followup and collect messages so agents can use native reply APIs on delayed queued turns.

## Best Possible Solution

Implement a narrow, channel-agnostic queue/followup continuity fix: normalize queued `FollowupRun.messageId`, include trusted per-item `message_id` metadata in collect prompts, preserve `messageId` through collect and overflow-summary drains, pass it to `runEmbeddedPiAgent` as `currentMessageId`, and use it during followup delivery threading. Keep Feishu-specific code limited to existing ingress/reply behavior unless a focused regression proves otherwise.

## Reproduction Assessment

Yes. A code-level reproduction is to enqueue busy-session followup runs with provider `messageId` in collect or summary-drain mode, then inspect the generated followup and runtime call; current main drops the id before prompt metadata, runtime `currentMessageId`, and delivery threading. I did not run tests because this review was read-only.

## Solution Assessment

Yes. The best fix is a small core queue/followup propagation change rather than a Feishu-only workaround, because the missing id is lost in shared followup plumbing before channel delivery.
