## Summary

Keep open. Current main still captures the original provider message ID when queueing, but collect-mode drain and queued followup execution still drop it before the runtime and Feishu reply tooling can use it; the earlier fix attempt at https://github.com/openclaw/openclaw/pull/36235 was closed unmerged.

## What This Changes

The issue requests preserving the original platform message_id through queued collect/followup delivery so platform-native reply APIs can target queued messages.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime context, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, from source inspection. Queue an inbound Feishu/group message while a run is active in collect mode: the queued run captures messageId, but current collect batching emits a synthetic followup without that ID and the runner never passes it to currentMessageId.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path; Feishu already preserves the original event ID before queueing, so a Feishu-only workaround would leave the runtime contract gap in place.
