## Summary

Keep open. Current main and the latest release still preserve the platform message ID only at enqueue time, then drop it across collect batching and followup delivery, so the Feishu/Lark queued-reply problem remains source-reproducible; the closed prior fix was not merged and the open handoff PR only covers part of the path.

## What This Changes

The issue requests preserving original platform message IDs through queued-message followup handling so channel-native reply APIs can target queued messages.

## Best Possible Solution

Keep this as the canonical queued-message-ID bug and preserve the normalized queued `messageId` through collect prompt metadata, synthetic followup/current-message context, embedded runtime context, and final followup delivery with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main stores the queued followup's original `messageId`, but collect batching and followup delivery do not pass that ID into reply-targeting surfaces; no live Feishu tenant run was attempted in this read-only review.

## Solution Assessment

Yes. The maintainable fix is shared queue/followup metadata propagation with regression tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
