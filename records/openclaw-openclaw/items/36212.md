## Label Justifications

- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main still captures the original inbound message ID when a followup is enqueued, but the collect drain, embedded followup runtime call, and final followup delivery path do not consistently carry that ID into reply-targeting surfaces, so the Feishu/Lark production problem remains a valid source-reproducible bug.

## What This Changes

The issue asks OpenClaw to preserve queued messages' original platform `message_id` so native platform reply APIs can target queued messages instead of placeholder IDs.

## Best Possible Solution

Keep this as the canonical queued-message-id bug and preserve the normalized queued `messageId` through collect prompt metadata, synthetic followup/current-message context, embedded runtime context, and final followup delivery with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main stores a queued followup's original `messageId`, but the collect drain, embedded runtime call, and followup delivery path omit that ID before reply targeting can use it.

## Solution Assessment

Yes. The maintainable fix is shared queue/followup metadata propagation with regression tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
