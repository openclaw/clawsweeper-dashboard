- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main and the latest release still capture the original provider message ID at enqueue, but collect-mode draining, queued embedded execution, and final followup delivery do not carry that ID into the reply-threading surfaces Feishu/Lark native replies need.

## What This Changes

Preserve original platform message IDs through queued followup and collect-mode metadata so agents can use platform-native reply APIs for queued messages.

## Best Possible Solution

Preserve the normalized original queued messageId through shared FollowupRun, collect drain, embedded followup runtime, and final delivery threading paths, with regression coverage and no Feishu-specific configuration.

## Reproduction Assessment

Yes. A high-confidence source path is to enqueue same-route collect-mode followups with messageId and inspect that the collected synthetic followup, queued runtime call, and final delivery threading call omit that ID.

## Solution Assessment

Yes. The narrow maintainable fix is shared queue/followup metadata propagation because Feishu already records original inbound IDs and downstream reply targeting already consumes currentMessageId or explicit reply IDs.
