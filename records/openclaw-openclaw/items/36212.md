## Label Justifications

- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main still captures the original queued provider message ID at enqueue, but the collect drain, queued runtime, and followup delivery paths do not carry that ID to the reply-threading surfaces Feishu/Lark native replies need; the reporter also confirmed this is still affecting production after the stale ping.

## What This Changes

Preserve original platform message IDs through queued and collect followup paths so native reply APIs can target the queued message.

## Best Possible Solution

Preserve the normalized original queued messageId through shared queue and followup propagation, expose it safely per collected item, and feed it into queued currentMessageId/reply-threading paths without Feishu-specific logic or new configuration.

## Reproduction Assessment

Yes. A high-confidence source path is to enqueue same-route collect-mode followups with messageId and inspect that the collected synthetic followup, queued runtime call, and final delivery threading call omit that ID.

## Solution Assessment

Yes. The narrow maintainable fix is shared queue/followup metadata propagation because Feishu already records original inbound IDs and downstream reply targeting already consumes currentMessageId or explicit reply IDs.
