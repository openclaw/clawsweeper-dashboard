## Summary

Keep open. Current main still captures the original provider message ID when enqueueing followups, but collect-mode batching and queued runtime execution do not propagate it into the collected prompt or `currentMessageId`; the linked fix attempts were closed unmerged.

## What This Changes

Preserve original provider `message_id` for queued messages so platform-native replies can target the original message after the agent drains a busy queue.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime `currentMessageId`, with focused regression coverage and no Feishu-specific workaround or new config.

Yes, from source inspection rather than a live Feishu tenant. Queue a Feishu group-topic message while collect-mode is active: enqueue captures `messageId`, but collect drain creates a synthetic followup without it and the runner never passes it to `currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in shared queue/followup propagation because Feishu already records original inbound IDs and its action path consumes `currentMessageId` for native reply targeting.
