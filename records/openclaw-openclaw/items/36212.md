## Summary

Current main still drops queued platform message identity when collect-mode batches are synthesized, and an open closing PR now owns the bounded fix. Keep this issue open until that PR or an equivalent implementation merges; its focused tests are useful, but the linked PR still lacks live Feishu reply proof.

## What This Changes

Preserve the original platform message ID and reply target when queued messages are collapsed into collect-mode or overflow-summary followup runs.

## Best Possible Solution

Land the provider-neutral metadata handoff in https://github.com/openclaw/openclaw/pull/91272 after confirming the latest-item selection invariant across both drain paths and adding redacted live Feishu proof when feasible; then close this issue through the merged PR.

## Reproduction Assessment

Yes at source level with high confidence: queued records contain the platform identifier, current collect synthesis omits it, and downstream threading context depends on that identifier. This review did not execute a live Feishu tenant reproduction.

## Solution Assessment

Yes in direction. Preserving existing provider-neutral metadata at the queue-drain boundary is narrower and more maintainable than adding Feishu-specific fallback behavior; maintainers still need to affirm the latest-item targeting rule for a multi-message batch.
