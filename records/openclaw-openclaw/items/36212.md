## Label Justifications

- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main still stores the original inbound message ID on queued followups, but collect-mode synthetic followups, queued embedded runtime calls, and final followup delivery do not carry that ID through to the reply-threading path; the reporter also confirmed ongoing Feishu/Lark production impact after stale marking.

## What This Changes

The issue asks OpenClaw to preserve the original platform message ID through queued-message handling so agents can use native reply APIs for queued messages.

## Best Possible Solution

Preserve the normalized original queued messageId through collect drain, synthetic FollowupRun/currentMessageId, embedded runtime context, and followup delivery, with regression coverage and coordination with the related open threading work.

## Reproduction Assessment

Yes, at source level. Enqueue stores `FollowupRun.messageId`, but collect drain, queued embedded runtime params, and followup delivery omit it before `applyReplyThreading` can set `replyToId`.

## Solution Assessment

Yes. The best fix is shared queue/followup metadata propagation with tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
