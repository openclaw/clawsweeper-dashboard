## Label Justifications

- `P2`: This is a valid queued-message reply-target bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: The issue causes queued replies to lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main and the latest shipped tag still capture the original queued provider `messageId` at enqueue, but collect-mode batching, queued runtime setup, and followup delivery do not propagate it into the places Feishu/Lark reply tooling needs it.

## What This Changes

Queued followup messages should preserve the original platform `message_id` so agents can use platform-native reply APIs for queued messages.

## Best Possible Solution

Preserve the normalized original queued `messageId` through shared queue/followup propagation, expose it safely per collected item, and pass it into queued runtime and delivery `currentMessageId` paths without Feishu-specific logic or new configuration.

## Reproduction Assessment

Yes. A high-confidence source path is to enqueue same-route collect-mode followups with `messageId`, then inspect that the collected synthetic followup plus queued runtime and delivery calls omit that ID.

## Solution Assessment

Yes. The narrowest maintainable fix is shared queue/followup metadata propagation because Feishu already records original inbound IDs and downstream native reply targeting already consumes `currentMessageId` or explicit reply IDs.
