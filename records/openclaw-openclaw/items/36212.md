## Label Justifications

- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main and the latest release still store the original queued message ID, but they do not carry it through collect-mode prompt metadata, embedded followup runtime context, or final followup delivery where reply targeting is resolved.

## What This Changes

The issue asks for queued followup messages to preserve the original platform `message_id` so channel-native reply APIs can target the queued message.

## Best Possible Solution

Preserve the normalized queued `messageId` through collect per-item metadata, synthetic followup/currentMessageId context, embedded runtime params, and final followup delivery, with focused regression coverage across queue drain, followup runner, and reply threading.

## Reproduction Assessment

Yes, at source level. Current main stores `FollowupRun.messageId` at enqueue, but collect drain, queued embedded runtime params, and followup delivery omit it before platform reply targeting can use it.

## Solution Assessment

Yes. The maintainable fix is shared queue/followup metadata propagation with tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
