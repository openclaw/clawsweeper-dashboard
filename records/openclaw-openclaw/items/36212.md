## Summary

Keep open. Current main still captures the original provider message ID when queueing, but collect-mode batching and queued followup execution still drop it before the runtime and Feishu reply tooling can use it.

## What This Changes

Queued messages should preserve each original platform message_id so platform-native reply APIs can target queued Feishu/Lark messages.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime currentMessageId, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu tenant. Queue a Feishu group/topic message while a collect-mode run is active: the queued run captures messageId, but the collect drain creates a synthetic followup without messageId and the runner never passes it to currentMessageId.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path because Feishu already records the original inbound ID and its reply tooling consumes currentMessageId.
