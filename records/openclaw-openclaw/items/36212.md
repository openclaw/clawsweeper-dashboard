- `P2`: This is a valid queued-message reply-targeting bug with limited blast radius to busy agents and threading-capable channels.
- `impact:message-loss`: Queued replies can lose or mis-target native reply context when platform APIs reject placeholder message IDs.

## Summary

Keep open. Current main still captures the original inbound message ID on queued followups, but the collect drain, queued embedded runtime call, and final followup delivery path do not carry that ID through to the reply-targeting surfaces; the related open PR only covers part of the handoff.

## What This Changes

The issue asks OpenClaw to preserve original platform message IDs through queued-message delivery so channel-native reply APIs can target the original message.

## Best Possible Solution

Keep this as the canonical queued-message-id bug and preserve the normalized queued messageId through collect prompt metadata, synthetic FollowupRun/currentMessageId, embedded runtime context, and final followup delivery with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main stores FollowupRun.messageId at enqueue, but collect drain, queued embedded runtime params, and followup delivery omit it before the platform reply target can be preserved.

## Solution Assessment

Yes. The maintainable fix is shared queue/followup metadata propagation with tests; a Feishu-specific workaround would leave the same broken contract for other threading-capable channels.
