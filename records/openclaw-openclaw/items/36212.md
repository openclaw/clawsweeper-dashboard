## Summary

Keep open: current main still stores the inbound platform `messageId` on the queued run but drops it before collect-mode followup execution, so queued Feishu/Lark replies can still lose the native reply target.

## What This Changes

Queued followup messages should preserve the original platform `message_id` so native reply APIs can target queued messages.

## Best Possible Solution

Preserve a trusted queued platform `messageId` through individual, collect, and summary followup drains, pass it to the embedded runtime as `currentMessageId`, and cover the behavior with focused queue and followup-runner regression tests.

## Reproduction Assessment

Yes from source inspection: current main captures the inbound `messageId`, then drops it in the collect drain and never passes it as `currentMessageId` to followup execution. I did not run live Feishu or Vitest proof because this review was read-only.

## Solution Assessment

Yes, the narrow fix is to wire the existing `FollowupRun.messageId` into the existing embedded-runtime `currentMessageId` contract. No channel API, queue policy, or product decision is needed.
