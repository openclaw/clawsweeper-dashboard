## Summary

Keep this issue open: current main and the latest release still drop queued collect-mode `messageId` metadata, and an open closing PR already owns the focused fix path.

## What This Changes

Queued collect-mode followups should preserve the original platform `message_id` so platform-native reply APIs can target queued messages.

## Best Possible Solution

Land or explicitly close the open linked PR, then let this issue close once the queue metadata fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows `messageId` is available on queued runs before collect mode, but the current collect drain does not copy it to the synthesized followup that the runner uses for `currentMessageId`.

## Solution Assessment

Yes. Carrying the latest queued `messageId` through the existing collect drain is the narrow owner-boundary fix; adding Feishu-specific reply behavior would be the wrong layer.
