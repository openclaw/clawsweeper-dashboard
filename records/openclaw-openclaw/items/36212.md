## Summary

Keep open. Current main still has a source-visible gap: queued runs capture the inbound provider message ID, but the collect/followup path does not carry it into the trusted runtime `currentMessageId` context that message/reaction tools use for native reply targets; the related fix PR was closed unmerged.

## What This Changes

Queued messages should preserve the original platform `message_id` so agents can use platform-native reply APIs for busy-agent followups.

## Best Possible Solution

Add shared queue/runtime propagation for original queued message IDs: expose trusted per-item metadata in collect prompts and pass an unambiguous queued message ID into `runEmbeddedPiAgent.currentMessageId`, without Feishu-specific policy or new config.

## Reproduction Assessment

Yes, from source inspection. Queue an inbound Feishu/group message while a run is active in collect mode: `FollowupRun.messageId` is captured, but collect drain emits a new run without `messageId`, and the followup runner never passes `currentMessageId` to the embedded runtime.

## Solution Assessment

Yes, the reporter’s suggested direction is the narrow maintainable fix. The shared queue/followup path should preserve trusted queued IDs and tests should cover collect batching plus runtime `currentMessageId`, rather than adding Feishu-only reply workarounds.
