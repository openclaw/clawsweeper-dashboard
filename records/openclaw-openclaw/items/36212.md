## Label Justifications

- `impact:message-loss`: The issue was about losing the usable platform message identity for queued inbound messages, which prevented message-specific native replies.

## Summary

Current main now preserves queued platform message IDs through the collect and overflow-summary drain paths and forwards them as the follow-up current message ID, so the reported Feishu reply-target problem is implemented on main after the latest release.

## What This Changes

Preserve original platform message IDs for queued messages drained into follow-up turns so channel-native reply APIs can target the original message.

## Best Possible Solution

Keep the current-main queue delivery-context implementation from the merged reply-route fix and close this issue as implemented until a new concrete regression appears.

## Reproduction Assessment

Yes for the original released behavior by source inspection: the latest release’s drain rebuilt queued follow-ups without copying the source message ID. It no longer reproduces on current main because the drain and follow-up runner now carry that ID forward.

## Solution Assessment

Yes. Preserving the existing queued metadata at the queue-drain synthesis boundary is the narrow maintainable fix, and it avoids Feishu-specific reply policy in core.
