## Summary

Keep open. Current main still captures the inbound provider message ID when enqueueing a followup, but collect/synthetic drain paths do not preserve it on the emitted followup run and the queued followup runner does not pass it as `currentMessageId`; the related fix PR was closed unmerged.

## What This Changes

Preserve original platform `message_id` through queued collect/followup runs so channel-native reply APIs can target queued messages.

## Best Possible Solution

Implement shared queued-message-id propagation through collect/synthetic followups into runtime context and trusted per-item prompt metadata, without Feishu-specific workaround logic or new configuration.

## Reproduction Assessment

Yes, from source inspection. Queue an inbound Feishu/group message while an active run is busy in collect mode: the original `messageId` is captured, then the collect drain emits an aggregate followup without it and the followup runner never forwards it to `runEmbeddedPiAgent.currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared queue/followup runtime path: preserve trusted queued IDs through batching and pass them to runtime/tool context, rather than adding a Feishu-only send workaround.
