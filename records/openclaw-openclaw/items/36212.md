## Summary

Keep open. Current main still captures the original provider message ID when enqueueing, but collect-mode batching and queued followup execution do not propagate that ID into the collected prompt or runtime `currentMessageId`; the linked fix PR was closed unmerged.

## What This Changes

Queued Feishu/Lark followup messages should preserve the original platform `message_id` so native reply APIs can target the queued message.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime `currentMessageId`, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, as a source-reproducible path rather than a live Feishu run. Queue a Feishu group-topic message while a collect-mode run is active: enqueue captures `messageId`, but collect drain and queued runner do not propagate it to the prompt or `currentMessageId`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path because Feishu already records original inbound IDs and consumes `currentMessageId` for native reply threading.
