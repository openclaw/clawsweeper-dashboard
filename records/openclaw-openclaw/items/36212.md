## Summary

Keep open. Current main still captures queued provider message IDs, but collect-mode batching drops them from the collected prompt and queued runtime paths still omit the ID where tools and reply-threading expect `currentMessageId`, so the Feishu/Lark reply-target problem remains valid.

## What This Changes

The issue requests preserving original platform `message_id` values across queued-message delivery so agents can use platform-native reply APIs for queued messages.

## Best Possible Solution

Preserve the normalized original queued `messageId` in shared queue/followup propagation, expose per-item IDs in collect-mode metadata, and pass the ID into queued runtime and delivery `currentMessageId` paths without adding Feishu-specific logic or new configuration.

## Reproduction Assessment

Yes, as a source-reproducible path: enqueue same-route collect-mode followups with `messageId`, then inspect that the collected synthetic followup and queued runtime/delivery calls omit that ID. A live Feishu tenant run would strengthen platform proof but is not needed to identify the source defect.

## Solution Assessment

Yes. The narrowest maintainable fix is in shared queue/followup metadata propagation because Feishu already records original inbound IDs and already uses runtime `currentMessageId` for native reply targeting.
