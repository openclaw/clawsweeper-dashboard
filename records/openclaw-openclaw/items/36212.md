## Summary

Keep open. Current main still captures the provider message ID when queuing a followup, but the collect/followup path does not propagate that ID into trusted per-item metadata or `runEmbeddedPiAgent.currentMessageId`, and the related fix PRs were closed unmerged.

## What This Changes

Preserve original platform `message_id` values for queued messages so agents can target platform-native reply APIs after busy-session drains.

## Best Possible Solution

Add shared queue/runtime propagation for original queued message IDs, including trusted per-item collect metadata and `currentMessageId` forwarding for unambiguous queued runs, without Feishu-specific policy or new configuration.

## Reproduction Assessment

Yes, from source inspection. Queue an inbound Feishu/group message while a run is active in collect mode: the queued run captures `messageId`, but current collect batching emits a followup without that ID and the followup runner never passes it to `currentMessageId`.

## Solution Assessment

Yes, the narrow maintainable fix is in the shared queue/followup path. Feishu already preserves the original event ID at ingress, so adding Feishu-only reply workarounds would leave the common runtime contract gap in place.
