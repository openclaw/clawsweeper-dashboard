## Best Possible Solution

Patch the shared queue/followup propagation path to preserve normalized original message IDs into collect prompt metadata and queued runtime `currentMessageId`, with focused regression coverage and no Feishu-specific workaround or new config.

## Repair Work Prompt

Fix this issue by preserving original platform message IDs through queued followup execution. Current main captures `FollowupRun.messageId` in `src/auto-reply/reply/get-reply-run.ts`, but collect-mode synthetic runs in `src/auto-reply/reply/queue/drain.ts` drop it and `src/auto-reply/reply/followup-runner.ts` does not pass it to `runEmbeddedPiAgent.currentMessageId`. Implement the fix in the shared queue/followup path, not as a Feishu-specific workaround and without adding config. Normalize blank IDs away, preserve trusted IDs in collect prompt metadata if exposing them to the agent, and forward the resolved queued ID as runtime `currentMessageId`. Add regression coverage for collect prompt/messageId propagation and followup-runner `currentMessageId` forwarding; update `CHANGELOG.md` because this is user-visible. Use https://github.com/openclaw/openclaw/pull/36235 only as prior art, not as an automatic cherry-pick.
