## Summary

Keep open. Current main still captures the original provider message ID at enqueue time, but collect-mode batching and queued followup execution still drop it before the runtime and Feishu reply tooling can use it; the prior focused fix attempt was closed unmerged.

## What This Changes

Queued Feishu/Lark messages should retain the original platform message_id through busy-agent queueing so native reply APIs can target the specific queued message.

## Best Possible Solution

Patch the shared queue/followup path to preserve normalized original message IDs through collect prompts and queued runtime currentMessageId, with focused regression coverage and no Feishu-specific workaround or new config.

## Reproduction Assessment

Yes, from source inspection rather than a live Feishu tenant. Queue a Feishu group-topic message while a collect-mode run is active: the queued run captures messageId, but the collect drain creates a synthetic followup without messageId and the runner never passes it to currentMessageId.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared queue/followup propagation path because Feishu already records original inbound IDs and consumes currentMessageId for native reply threading.
