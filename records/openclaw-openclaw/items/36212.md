## Summary

Keep this issue open. Current main still records the original Feishu provider message ID at ingress and on the initial queued run, but collect-mode batching, overflow summary drains, followup runtime setup, and followup delivery do not carry that ID through to queued-message metadata or `currentMessageId`; the directly related fix PR is closed and unmerged.

## What This Changes

Queued followup messages should preserve the original platform `message_id` so agents can use platform-native reply APIs for each queued message.

## Best Possible Solution

Keep the fix channel-agnostic in the core queue/followup path: preserve normalized queued `messageId` through collect and summary drains, expose trusted per-item `message_id` in collect prompt blocks, pass it into runtime as `currentMessageId`, and apply it during followup delivery reply threading without adding Feishu-specific core policy.
