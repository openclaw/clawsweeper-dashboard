## Summary

Keep open. Current main still has the exact-origin bug this PR targets, the proposed fix is narrow and well-scoped, and the remaining work is maintainer rebase/consolidation plus validation rather than stale cleanup.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to normalize opaque URL origins through a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated gateway patch that symmetrically preserves hosted opaque origins for exact matching, keeps wildcard policy separate, and retains WebSocket handshake regression coverage.

## Reproduction Assessment

Yes, source-level. Current main stores URL.origin and compares against parsedOrigin.origin, while Node reports chrome-extension:// origins as origin=null, so an exact configured extension origin cannot match.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes incoming and configured origins consistently, preserves exact allowlist semantics, and leaves scheme wildcard behavior to the separate policy issue.
