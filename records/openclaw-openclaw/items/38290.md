## What This Changes

The PR updates Gateway browser-origin allowlist matching so exact custom-scheme origins such as `chrome-extension://<id>` can be accepted, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land a narrow Gateway origin normalization fix that only accepts host-bearing opaque custom schemes by exact configured origin, preserves existing configured-origin exactness, and keeps scheme wildcard support in the separate product/security discussion.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe show current main still converts `chrome-extension://...`, `app://localhost`, and `tauri://localhost` to `URL.origin === "null"`, so exact configured entries cannot match.

## Solution Assessment

No. The right fix is a narrow `parseOrigin` fallback for host-bearing custom schemes, but this branch also reparses configured allowlist entries and permits empty-host opaque fallback keys.
