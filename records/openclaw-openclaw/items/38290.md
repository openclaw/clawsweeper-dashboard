## Summary

Keep open: the exact custom-scheme origin bug is real and the screenshots are sufficient proof, but the patch broadens configured `gateway.controlUi.allowedOrigins` entries across a Gateway browser-origin trust boundary and the branch is currently conflicting.

## What This Changes

The PR changes Gateway browser-origin allowlist matching so exact hosted custom-scheme origins such as `chrome-extension://<id>` can pass, and adds focused origin-check plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated exact opaque-origin fix that keeps configured `allowedOrigins` as exact trimmed/lowercased strings, applies `scheme://host` fallback only to hosted custom-scheme request origins, includes unit plus handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin` and compares it to exact `allowedOrigins`; Node reports hosted custom schemes such as `chrome-extension://` with `origin:"null"`, so exact entries cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader URL origins.
