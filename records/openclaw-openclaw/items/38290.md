## Summary

Keep open. Current main still has the exact custom-scheme origin allowlist bug, and this PR has useful proof plus handshake coverage, but the diff introduces a security-sensitive allowlist widening by parsing configured entries and stripping path/query/userinfo before matching.

## What This Changes

The PR updates Gateway origin normalization and tests so exact `chrome-extension://<id>` Control UI WebSocket origins can match `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, borrowing this PR's handshake proof and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.

## Review Findings
