## Summary

Keep open. The reported Gateway bug is still present on current main and this PR targets the right code path with usable real-world proof, but the patch should be tightened so malformed `allowedOrigins` entries with paths, query strings, fragments, or credentials do not collapse into broader trusted origins.

## What This Changes

The PR changes Gateway browser-origin matching to allow exact opaque-scheme origins such as `chrome-extension://<id>` and adds focused origin-check plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated exact opaque-origin normalization fix in Gateway, retaining WebSocket handshake coverage and explicit allowlist semantics while leaving scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted opaque-scheme origins such as `chrome-extension://...` into `URL.origin === "null"` and compares that against the exact configured allowlist entry used by the Gateway WebSocket path.

## Solution Assessment

No, not as written. The parser-level fallback is the right direction, but configured allowlist entries should be validated as origins instead of allowing path/query/fragment/credential-bearing URLs to collapse into broader trusted origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reject path-bearing allowlist entries:** `src/gateway/origin-check.ts:45`
  - body: `normalizeAllowedOrigin` now parses every configured value and matches on the parsed origin. That turns entries like `https://control.example.com/specific-path`, `https://user:pass@control.example.com`, or `https://control.example.com?x=1` into `https://control.example.com`, so a malformed value that used to match nothing now trusts the whole browser origin. Please reject non-origin strings before adding them to the allowlist, or preserve the old exact behavior for standard-origin entries.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff preserves token handling and dependencies, but the new allowlist-entry normalization can broaden malformed origin config values.

- **[medium] Malformed allowlist values become trusted origins:** `src/gateway/origin-check.ts:45`
  - body: Because configured entries are parsed and reduced to `URL.origin`, values containing paths, queries, fragments, or credentials can become a trusted bare origin. This is a Gateway browser-origin trust boundary, so invalid origin strings should fail closed.
