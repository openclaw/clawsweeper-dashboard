The PR changes Gateway Control UI origin matching to compare exact opaque-scheme origins such as `chrome-extension://<id>` using scheme-plus-host normalization, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one rebased exact opaque-origin normalization fix in Gateway, preferably retaining this PR's WebSocket handshake coverage, while leaving scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Solution Assessment

Yes for the exact-origin bug. Symmetric normalization of incoming and configured opaque origins is the narrow maintainable fix; broader wildcard matching belongs in https://github.com/openclaw/openclaw/issues/39739.
