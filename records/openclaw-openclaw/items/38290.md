## Summary

Keep open: current main still has the exact custom-scheme origin bug and this PR has useful real proof plus handshake coverage, but the patch should not merge as written because it broadens configured standard URL allowlist entries by parsing them into whole origins.

## What This Changes

The PR changes Gateway browser-origin normalization and adds unit plus WebSocket handshake tests so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, keeps unit and WebSocket handshake coverage for hosted opaque schemes, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses hosted custom-scheme origins to `URL.origin === "null"` and then compares that parsed value against exact configured allowlist entries, so an exact `chrome-extension://<id>` entry cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for incoming hosted opaque origins is the right narrow direction, but configured allowlist entries should remain exact raw strings instead of being parsed and stripped to whole origins.

## Review Findings
