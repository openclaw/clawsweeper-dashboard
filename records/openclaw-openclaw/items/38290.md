## Summary

Keep open: current main still has the exact custom-scheme origin matching bug, but this PR is not merge-ready because it broadens configured `allowedOrigins` entries through URL parsing and now conflicts with current main.

## What This Changes

The PR adds hosted custom-scheme origin normalization for Gateway browser-origin allowlist matching plus focused unit and WebSocket handshake tests.

## Best Possible Solution

Land one consolidated parser-level fix that falls back to `scheme://host` only for hosted custom-scheme request origins, keeps configured allowlist entries exact, adds regression coverage for standard URL path entries, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with `origin:"null"`, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. The incoming-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader origins.
