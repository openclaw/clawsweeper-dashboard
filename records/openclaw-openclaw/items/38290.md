## Summary

Keep open: this PR addresses a valid current and shipped Gateway bug, and the attached proof is sufficient, but the diff should not merge until it preserves the exact configured allowlist trust boundary.

## What This Changes

The PR adds custom-scheme fallback origin matching for `chrome-extension://` Gateway browser-origin checks and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist strings, falls back to `scheme://host` only for hosted custom-scheme request origins, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.12` compare exact configured strings against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with `origin:"null"`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. The incoming hosted custom-scheme fallback is the right direction, but configured allowlist entries should remain exact trimmed lowercase strings instead of being parsed into broader origins.
