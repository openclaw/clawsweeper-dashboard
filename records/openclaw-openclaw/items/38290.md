## Summary

This remains a valid focused Gateway bug fix, but it should not merge as written because the PR parses configured allowlist entries and broadens standard-origin config entries with paths, queries, or fragments into whole-origin matches. The supplied screenshots are sufficient real behavior proof for the Chrome-extension sidepanel scenario, and the remaining blocker is a narrow security-boundary repair plus branch refresh or consolidation with overlapping custom-origin work.

## What This Changes

The PR changes Gateway browser-origin matching and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can authenticate WebSocket browser clients.

## Best Possible Solution

Land one consolidated exact hosted custom-scheme origin fix that preserves configured allowlist entries as exact strings while only adding the safe `scheme://host` fallback for parsed request origins whose `URL.origin` is `"null"`.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings to `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with origin `"null"`, so exact configured extension origins cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right narrow solution, but parsing configured allowlist entries changes existing exact-string semantics for standard origins and should be avoided or explicitly approved with tests.
