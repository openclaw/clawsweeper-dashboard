## Summary

Keep open. Current main and the latest release still reject exact opaque extension/custom origins, and this PR remains a plausible narrow fix with sufficient real behavior proof, but it is currently conflicting and overlaps another open fix path that maintainers should consolidate.

## What This Changes

The PR changes Gateway browser-origin matching so exact opaque extension origins can match `gateway.controlUi.allowedOrigins`, with origin-check and WebSocket handshake regression tests.

## Best Possible Solution

Land one consolidated exact opaque-origin normalization fix with focused matcher and WebSocket regression coverage, while leaving scheme wildcard semantics to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` with `URL.origin === "null"` and then compares that parsed value against the exact allowlist entry used by the Gateway WebSocket handshake path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming exact opaque origins is the narrow maintainable fix; broader scheme wildcard matching should stay in the separate security/product discussion.
