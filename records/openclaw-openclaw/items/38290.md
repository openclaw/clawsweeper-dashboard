## Summary

Keep open: the current-main bug is valid and this PR has useful proof plus coverage, but the diff loosens configured allowlist semantics by parsing config entries into origins, so it needs a small security-sensitive fix and rebase before merge.

## What This Changes

The PR changes Gateway browser-origin matching so exact chrome-extension origins in gateway.controlUi.allowedOrigins can match and adds focused origin-check plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated Gateway fix that preserves hosted opaque-origin matching for request origins while accepting only literal configured origins or the global '*' entry, leaving scheme-wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as chrome-extension://... into URL.origin === "null" and then compares that value against exact configured allowedOrigins in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The parser-side fallback for hosted opaque origins is the right direction, but configured allowedOrigins should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject path-bearing allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: `normalizeAllowedOrigin` now parses configured entries through `URL` and compares the serialized origin. A value like `https://control.example.com/path` or `chrome-extension://id/page` will authorize `https://control.example.com` or `chrome-extension://id`, even though docs require full origins and current main only matched the literal configured string; reject or ignore non-origin allowlist strings instead of stripping them.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has a concrete Gateway origin-allowlist regression because configured non-origin URLs can become authorized origins after parsing.

Concerns:

- **[medium] Configured URLs are stripped into allowed origins:** `src/gateway/origin-check.ts:47`
  - body: Parsing configured allowedOrigins through URL makes path, query, and userinfo-bearing entries authorize the stripped origin on a security-sensitive Control UI WebSocket boundary. Keep matching strict for configured entries and add negative coverage for malformed or non-origin allowlist values.
  - confidence: 0.86
