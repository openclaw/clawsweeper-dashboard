## What This Changes

The PR changes Gateway browser-origin matching so exact non-standard origins such as `chrome-extension://<id>` can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake regression tests.

## Best Possible Solution

Land the exact opaque-origin matching fix after `normalizeAllowedOrigin` preserves fail-closed behavior for path, query, and hash-bearing entries and the branch is refreshed against current main.

## Reproduction Assessment

Yes at source level: current main compares configured `allowedOrigins` to `URL.origin`, and Node reports `chrome-extension://...` origins as `null`, so exact extension entries cannot match. I did not run the Gateway Vitest lane in this read-only review.

## Solution Assessment

No, not as submitted. The opaque-origin fallback is the right repair shape, but the configured allowlist entry path must reject path/query/hash values before normalization so the browser-origin boundary does not widen on upgrade.

Full review comments:

- **[P1] Reject path-bearing allowlist entries before normalizing:** `src/gateway/origin-check.ts:48`
  - body: `normalizeAllowedOrigin()` reparses configured allowlist entries and returns the parsed match origin. That means an existing value like `https://control.example.com/path?x#y`, which currently fails closed because it does not equal the browser `Origin`, would become `https://control.example.com` and allow the whole origin. Since this field is documented as full origins only and gates browser access to the Gateway, reject path/query/hash-bearing entries before normalizing while keeping the exact `chrome-extension://...` fix.
