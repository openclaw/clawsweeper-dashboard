## Summary

Keep open: current main and the latest release still reject exact opaque-scheme origins, but this PR should not land as-is because it reparses configured allowlist entries and needs stronger runtime proof for this browser-origin authorization boundary. The overlapping https://github.com/openclaw/openclaw/pull/73511 is open, but it is not a safe superseding target yet because it also lacks real behavior proof and still has review follow-up.

## What This Changes

This PR changes Gateway browser-origin matching so opaque schemes such as Chrome extension origins normalize to `scheme://host`, and adds unit plus WebSocket handshake tests for exact extension-origin allowlisting.

## Best Possible Solution

Land a refreshed narrow parser fix that applies `scheme://host` fallback only to incoming opaque origins while preserving exact trimmed lowercase configured allowlist strings and adding fail-closed regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main compares configured entries against `URL.origin`, and a Node probe shows the reported opaque/custom schemes serialize that origin as `null` before the exact allowlist check.

## Solution Assessment

No. The parser-local fix direction is right, but this branch also reparses configured allowlist entries and can authorize base origins that previously failed closed.
