## Summary

Keep open: the exact extension-origin bug is still present on current main and the latest release, and this PR has useful real-behavior-backed coverage, but the patch broadens configured allowlist entries and needs a focused repair before merge.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://<id>` allowlist entries can pass and adds origin-check plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist strings, falls back to `scheme://host` only for incoming hosted custom-scheme origins, keeps literal `Origin: null` and wildcard semantics unchanged, and includes unit plus handshake coverage.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node returns `null` for hosted custom schemes such as `chrome-extension://`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trimmed lowercase strings instead of being parsed into broader origins.
