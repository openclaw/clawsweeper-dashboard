## Summary

Keep open: the underlying Gateway origin bug is valid and the PR has real after-fix proof, but the patch is not merge-ready because it changes configured allowlist entries from exact strings into parsed origins in a security-sensitive browser-origin boundary.

## What This Changes

The PR updates Gateway browser-origin normalization and adds unit plus WebSocket handshake coverage so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can connect.

## Best Possible Solution

Land one consolidated Gateway origin-check fix that normalizes request origins for hosted custom schemes while preserving configured `allowedOrigins` as exact trimmed/lowercased entries, with regression tests for extension allow/reject and standard-origin path non-matches.

## Reproduction Assessment

Yes, at source level: current main compares exact configured entries to `URL.origin`, and Node returns `"null"` for `chrome-extension://...`, so exact extension origins cannot match. I did not run the Vitest lane in this read-only review, but the Node URL contract and current source make the failing path clear.

## Solution Assessment

No, not as written: the request-origin fallback is the right narrow fix, but reparsing configured allowlist entries broadens exact config semantics. The safer solution is to keep config entries exact and normalize only the parsed request origin for hosted custom schemes.
