## Summary

Keep this PR open: it addresses a real Gateway browser-origin bug with useful proof, but the current patch is not merge-ready because it broadens configured `allowedOrigins` entries across a security boundary and the branch now conflicts with main.

## What This Changes

The PR normalizes opaque custom-scheme Gateway browser origins such as `chrome-extension://<id>` for allowlist matching and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated parser-level fix that preserves configured allowlist entries as exact trimmed lowercase strings, falls back to `scheme://host` only for hosted custom-scheme request origins, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, source-reproducible. Current main compares exact configured strings against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with `origin:"null"`, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. The incoming-origin fallback is the right direction, but configured allowlist entries should stay exact instead of being parsed into broader origins.
