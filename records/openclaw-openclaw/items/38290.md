## Summary

Keep open. Current main still has the exact opaque-origin allowlist bug, and this PR is a focused, correct fix with real extension connection proof rather than stale cleanup material.

## What This Changes

The PR changes Gateway browser-origin matching so exact non-standard origins such as `chrome-extension://<id>` are normalized consistently and covered by unit plus WebSocket handshake tests.

## Best Possible Solution

Land this exact-origin normalization fix or the overlapping broader custom-scheme fix after current-main validation, while keeping scheme wildcard support in the separate policy discussion.

## Reproduction Assessment

Yes. Current main plus the Node URL probe gives a high-confidence source reproduction: `chrome-extension://...` parses with `URL.origin === "null"`, then `checkBrowserOrigin` compares that value to the exact configured allowlist entry.

## Solution Assessment

Yes. Normalizing opaque origins to exact `scheme://host` values on both sides is the narrowest maintainable fix because it preserves explicit allowlist enforcement and avoids adding wildcard or host-header fallback behavior.
