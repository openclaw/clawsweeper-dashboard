## Summary

Keep open. Current main still compares `gateway.controlUi.allowedOrigins` against `URL.origin`, which is `"null"` for `chrome-extension://`, `app://`, and similar opaque-origin schemes, so exact extension allowlist entries still cannot match. The PR is a narrow, appropriate fix and should be rebased/reviewed rather than closed by cleanup.

## What This Changes

The PR changes the gateway browser-origin matcher to use a scheme/host fallback for opaque non-standard URL schemes and adds unit plus WebSocket handshake tests for exact `chrome-extension://` allowlisting.

## Best Possible Solution

Land this PR after rebase, or land an equivalent narrow gateway patch that symmetrically normalizes incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins`, using `URL.origin` for standard origins and a `protocol//host` fallback only for opaque-origin schemes. Keep scheme wildcard support separate in #39739.

## Reproduction Assessment

Yes. On current main, `new URL("chrome-extension://abcdefghijklmnop").origin` is `"null"`, and `checkBrowserOrigin` compares that value against raw lowercased allowlist entries, so an exact `chrome-extension://...` allowlist entry cannot match.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix: normalize both sides consistently while preserving exact allowlist matching and literal `Origin: null` rejection. Wildcard patterns such as `chrome-extension://*` remain a separate product/security decision in #39739.
