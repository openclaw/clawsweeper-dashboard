## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` allowlist entries, and this PR is a narrow, valid fix path; it should be rebased and reconciled with the overlapping custom-origin PR before landing rather than closed.

## What This Changes

The PR changes Gateway browser-origin matching to normalize opaque hosted origins with a `scheme://host` fallback and adds unit plus WebSocket handshake regression tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that symmetrically normalizes hosted opaque origins for exact allowlist matching, keeps wildcard policy separate, and preserves the WebSocket handshake regression test.

## Reproduction Assessment

Yes, at source level. Current main still parses `chrome-extension://abcdefghijklmnop` to comparable origin `"null"`, then compares that against the exact configured extension origin, so the allowlist cannot match.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes incoming and configured origins consistently, preserves exact allowlist semantics, and leaves scheme wildcard behavior to https://github.com/openclaw/openclaw/issues/39739.
