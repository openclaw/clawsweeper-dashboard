## Summary

Keep open: current main and v2026.6.6 still have the exact opaque-origin allowlist bug, but this PR is not merge-ready because it reparses configured `allowedOrigins` and can turn previously fail-closed strings into authorized base origins.

## What This Changes

The PR changes gateway browser-origin matching to allow exact opaque-scheme origins such as `chrome-extension://<id>` and adds origin-check plus WebSocket auth tests.

## Best Possible Solution

Land a narrow fix that applies `scheme://host` fallback only to incoming opaque origins while preserving exact trimmed lowercase config matching and fail-closed tests for malformed or non-origin allowlist entries.

## Reproduction Assessment

Yes, at source level. Current main parses opaque origins through `URL.origin`, which returns `null`, then compares that value against literal configured allowlist entries.

## Solution Assessment

No, not in its current shape. The exact-origin direction is right, but reparsing configured allowlist values is broader than necessary and weakens fail-closed config semantics.
