## Summary

Keep open. The underlying Gateway bug is still present on current main and this PR has useful coverage, but the current patch would silently broaden path/query-bearing `allowedOrigins` entries into whole-origin matches and the branch is now merge-conflicting with current main.

## What This Changes

The PR normalizes Gateway browser-origin matching for exact `chrome-extension://<id>` allowlist entries and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that preserves hosted custom-scheme origins for exact allowlists, keeps configured allowlist entries origin-exact, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and compares that value against exact configured `allowedOrigins`.

## Solution Assessment

No, not as written. Symmetric custom-scheme normalization is the right direction, but configured allowlist entries must not be normalized in a way that strips paths or queries and widens a misconfiguration into a full-origin allow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject non-origin allowlist entries:** `src/gateway/origin-check.ts:45`
  - body: Running configured `allowedOrigins` through `parseOrigin` strips paths, queries, and fragments before comparison. A misconfigured entry like `https://control.example.com/path` currently matches nothing, but this patch would normalize it to `https://control.example.com` and accept that whole origin, silently widening the Gateway origin policy; keep the custom-scheme fallback while rejecting or ignoring entries that are not exact origins.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or script changes, but the origin-policy change can broaden path/query-bearing allowlist entries into whole-origin allows.

Concerns:

- **[medium] Allowlist URL paths are stripped into origins:** `src/gateway/origin-check.ts:45`
  - body: `normalizeAllowedOrigin` reuses `parseOrigin`, so configured values with paths, queries, or fragments are converted to `url.origin` before matching. That changes an invalid/non-matching explicit allowlist entry into an accepted full-origin rule, which matters because this code protects browser-origin access to the Gateway.
  - confidence: 0.86
