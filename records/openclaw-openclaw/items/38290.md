## What This Changes

The PR changes Gateway browser-origin allowlist matching to support exact custom-scheme origins such as `chrome-extension://<id>` and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated exact opaque-origin fix that keeps configured `allowedOrigins` exact, applies a `scheme://host` fallback only to hosted custom-scheme request origins, includes unit and handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin` and compares it to exact `allowedOrigins`; Node reports hosted custom schemes such as `chrome-extension://` with `origin:"null"`, so exact entries cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader URL origins.
