## Summary

Keep this PR open: it targets a valid current Gateway origin bug and includes useful real behavior proof, but the diff still has a blocking allowlist-normalization security bug before it can merge.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://` origins can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literal origins or `*`, and leave scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current `main` turns hosted custom schemes into `URL.origin === "null"` and then compares that value against the exact configured allowlist entry in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path or userinfo.
