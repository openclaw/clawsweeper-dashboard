## Summary

This PR should stay open: current main still cannot match exact opaque browser origins such as `chrome-extension://...` against `gateway.controlUi.allowedOrigins`, and the proposed exact-origin normalization remains a valid narrow fix. The branch needs maintainer choice and rebase/current-head validation because overlapping origin-normalization work is also open.

## What This Changes

The PR changes Gateway browser-origin matching so exact non-standard extension origins can be allowlisted, with unit and WebSocket handshake regression coverage.

## Best Possible Solution

Land one exact opaque-origin normalization fix with matcher and WebSocket regression coverage, while keeping scheme wildcard support in https://github.com/openclaw/openclaw/issues/39739.

Yes, at source level. Current main parses `chrome-extension://...` with `URL.origin === "null"` and compares that parsed value against the exact allowlist entry used by the Gateway WebSocket browser-origin path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming exact opaque origins is the narrow maintainable fix; broader scheme wildcard matching should remain in the separate product/security discussion.
