## Summary

Keep open: current main still has the exact custom-scheme origin allowlist bug, but this PR should not merge as written because it broadens configured allowlist entries by parsing standard URL strings into whole-origin matches.

## What This Changes

The PR changes Gateway origin normalization and adds unit and WebSocket handshake tests so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, covers allowed and rejected hosted custom-scheme origins plus a WebSocket handshake, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` to `origin="null"` and then compares that parsed value against exact configured strings, so an exact extension origin cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for hosted opaque request origins is the right direction, but configured standard allowlist entries should remain exact raw strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Current main keeps configured entries as exact lowercased strings and rejects that non-origin entry; keep the opaque-origin fallback without broadening standard entries.
