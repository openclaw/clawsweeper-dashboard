## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` allowlist entries, and this PR remains a focused fix candidate rather than stale cleanup or something already implemented.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to preserve exact opaque extension origins and adds unit plus WebSocket handshake regression coverage.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that symmetrically preserves hosted opaque origins for exact matching, retains rejection of literal `Origin: null`, and keeps the WebSocket handshake regression coverage.

Yes. Source inspection plus the Node URL probe show current main compares the configured exact extension origin against `URL.origin === "null"`, so the allowlist entry cannot match.

## Solution Assessment

Yes. The proposed normalization is the narrow maintainable direction for exact hosted extension origins, with wildcard support left to the separate security/product discussion.
