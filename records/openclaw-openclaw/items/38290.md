## Summary

Keep open: current main still has the exact custom-scheme origin bug, but this PR should not merge as written because it widens configured allowlist entries on a security-sensitive Gateway boundary.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, borrowing this PR's handshake coverage and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origin literals instead of being parsed as arbitrary URLs and normalized by stripping path or userinfo.

## Review Findings
