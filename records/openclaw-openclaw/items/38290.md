## Summary

Keep open. Current main still has the exact opaque-origin allowlist bug, while this PR is a focused fix with regression coverage and no blocking review findings; the remaining step is maintainer review/consolidation with the overlapping custom-scheme PR and fresh validation.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to preserve exact hosted opaque origins such as `chrome-extension://<id>` and adds unit plus WebSocket handshake regression tests.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that symmetrically preserves hosted opaque origins for exact matching, keeps literal `Origin: null` rejected, leaves wildcard policy to the separate issue, and retains regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main compares configured exact extension origins against `URL.origin`, and the Node URL probe shows extension origins produce `origin=null`, so the exact allowlist entry cannot match.

## Solution Assessment

Yes. Symmetric `protocol//host` normalization for hosted opaque origins is the narrow maintainable fix, while scheme wildcard support remains a separate Gateway trust-policy decision.
