## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` allowlist entries through the Gateway origin checker, and this PR is a focused valid fix with sufficient screenshot proof; the remaining work is maintainer review, rebase/consolidation with the overlapping custom-origin PR, and fresh validation.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to normalize opaque URL origins through a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that symmetrically normalizes hosted opaque origins, keeps wildcard policy separate, and retains the handshake regression test.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main stores `URL.origin`, and Node reports exact Chrome extension origins as `"null"`, so the exact configured allowlist value cannot match. I did not run the Vitest lane in this read-only review.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes both incoming and configured hosted opaque origins consistently, preserves exact allowlist semantics, and leaves scheme wildcard behavior to https://github.com/openclaw/openclaw/issues/39739.
