## Summary

Keep open: this is still a valid Gateway bug on current main, and the PR is a focused exact-origin fix with useful regression coverage. It should not be cleanup-closed; the remaining maintainer work is to rebase or consolidate it with the overlapping custom-origin PR and run current Gateway validation.

## What This Changes

The PR normalizes Gateway browser-origin matching so exact opaque origins such as `chrome-extension://<id>` can be matched in `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake tests.

## Best Possible Solution

Land one exact opaque-origin normalization path with focused matcher and WebSocket regression coverage, while keeping scheme wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `url.origin`, Node returns `"null"` for `chrome-extension://...`, and the WebSocket browser-origin handshake routes through that exact allowlist comparison.

## Solution Assessment

Yes, with a rebase. Normalizing hosted opaque origins back to exact `scheme://host` matching is the narrow maintainable fix; broader wildcard semantics should stay in the separate policy issue.
