## Summary

Keep open: current main and the latest release still have the exact opaque-origin allowlist bug, but this PR should not merge as-is because its allowlist-entry normalization can silently turn path-bearing configured URLs into host-wide allowed origins and the branch is currently conflicting with main.

## What This Changes

The PR changes gateway browser-origin matching so exact non-standard origins such as `chrome-extension://<id>` can be accepted from `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake coverage.

## Best Possible Solution

Rebase the branch onto current main, keep the exact opaque-origin fallback for both request and configured origins, and reject configured allowlist entries that are not exactly origin strings; keep scheme-level wildcard support in the separate security/product issue.

## Reproduction Assessment

Yes. Current main compares configured entries with `URL.origin`, and a Node URL probe confirms `chrome-extension://<id>` and `app://localhost` serialize to `origin=null`, so exact configured custom origins cannot match today.

## Solution Assessment

No, not exactly as submitted. The core idea is the right narrow fix, but the implementation should fail closed for configured values that are not exact origins before it changes a security-sensitive allowlist surface.
