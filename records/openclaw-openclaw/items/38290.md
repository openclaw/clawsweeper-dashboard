## Summary

Keep open. Current main and the latest release still reject exact `chrome-extension://<id>` allowlist entries, and this PR remains a narrow, correct fix with real behavior proof; it needs maintainer rebase/consolidation rather than cleanup closure.

## What This Changes

The PR normalizes opaque Gateway browser origins so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake regression tests.

## Best Possible Solution

Land one exact opaque-origin normalization path with focused matcher and WebSocket regression coverage, while keeping scheme wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `url.origin`, Node returns `"null"` for `chrome-extension://...`, and the Gateway WebSocket handshake routes browser-origin connects through that exact allowlist comparison.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming exact opaque origins is the narrow maintainable fix; broader `chrome-extension://*` semantics should remain in the separate policy issue.
