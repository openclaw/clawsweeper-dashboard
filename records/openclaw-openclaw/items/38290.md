## Summary

Keep open: current main still rejects exact `chrome-extension://<id>` allowlist entries, and this PR remains the narrow, non-obsolete fix path rather than something already implemented or superseded.

## What This Changes

The PR changes gateway browser-origin allowlist matching to normalize opaque URL origins with a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Rebase and land this PR, or an equivalent narrow gateway patch that symmetrically normalizes incoming origins and configured allowlist entries while leaving scheme wildcard support to #39739.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe show current main parses `chrome-extension://abcdefghijklmnop` to comparable origin `"null"`, while the exact allowlist entry remains `chrome-extension://abcdefghijklmnop` and cannot match.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes both sides consistently, preserves exact allowlist semantics, and does not add wildcard handling or change token/device auth.
