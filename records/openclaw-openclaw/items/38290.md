## Summary

Keep open: current main still rejects exact `chrome-extension://` allowlist entries, and this PR remains a narrow, correct fix that needs maintainer rebase/consolidation before landing.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to preserve opaque-origin `scheme://host` values and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that preserves hosted opaque-origin exact matching, keeps wildcard policy separate, and retains the WebSocket handshake regression test.

## Reproduction Assessment

Yes, source-level. Current main parses `chrome-extension://abcdefghijklmnop` to `url.origin === "null"`, then compares that value to exact configured allowlist strings in the Gateway WebSocket origin-check path.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes incoming and configured origins consistently while preserving exact allowlist semantics and leaving scheme wildcards to the separate policy item.
