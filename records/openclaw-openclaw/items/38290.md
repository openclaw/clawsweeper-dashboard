## Summary

Keep open: the underlying Gateway origin bug is still valid on current main and the PR has sufficient real behavior proof, but the patch currently broadens configured `allowedOrigins` entries across a browser-origin trust boundary and the branch is conflicting.

## What This Changes

This PR changes `checkBrowserOrigin` so custom-scheme browser origins such as `chrome-extension://<id>` can match `gateway.controlUi.allowedOrigins`, and adds unit plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist strings, falls back to `scheme://host` only for incoming hosted custom-scheme origins, adds no-broadening coverage and handshake coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve exact allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: `normalizeAllowedOrigin()` parses configured entries through `parseOrigin()`, so a value like `https://trusted.example/path` becomes `https://trusted.example` and starts authorizing the whole origin. Keep the custom-scheme request-origin fallback without broadening configured allowlist semantics.
  - confidence: 0.9
