## Summary

Keep open. Current main still collapses exact `chrome-extension://<id>` origins to `"null"` before allowlist matching, while this PR is a narrow and correct fix with useful regression coverage; the remaining work is maintainer handling for conflicts and overlap with the newer custom-scheme PR.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to normalize opaque-origin schemes through a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated gateway patch that symmetrically preserves hosted opaque origins for exact matching, keeps wildcard policy separate, and retains the WebSocket handshake regression test.

## Reproduction Assessment

Yes, source-level. Current main parses `chrome-extension://abcdefghijklmnop` to comparable origin `"null"`, while the allowlist stores the exact extension origin string and the Gateway WebSocket path uses that helper for browser-origin handshakes.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes incoming and configured origins consistently, preserves exact allowlist semantics, and leaves scheme wildcard behavior to the separate policy issue.
