## Summary

Keep open: current main still cannot match an exact `chrome-extension://<id>` entry in `gateway.controlUi.allowedOrigins`, and this PR is a narrow fix with focused coverage and sufficient real behavior proof.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to preserve exact hosted opaque origins and adds unit plus WebSocket handshake regression coverage.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that preserves exact hosted opaque origins symmetrically, keeps literal `Origin: null` rejected, and leaves scheme wildcard support to the separate product/security discussion.

## Reproduction Assessment

Yes. Source inspection on current main shows `checkBrowserOrigin` compares `allowedOrigins` against `URL.origin`, and the Node URL probe shows `chrome-extension://...` produces `origin === "null"`, so an exact allowlist entry cannot match.

## Solution Assessment

Yes. The PR's symmetric `scheme://host` fallback for parsed request and configured allowed origins is the narrow maintainable fix; broader patterns such as `chrome-extension://*` remain separate at https://github.com/openclaw/openclaw/issues/39739.
