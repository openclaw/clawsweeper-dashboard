## Summary

Keep open: current main and the latest release still have the opaque-origin allowlist bug, and this PR contains useful focused work, but it is not merge-ready because it reparses configured allowlist entries, lacks diagnostic runtime proof for a browser-origin security boundary, and is currently conflicting against main.

## What This Changes

The PR changes Gateway browser-origin allowlist matching for opaque schemes such as `chrome-extension://`, normalizes configured allowlist entries through the same parser, and adds origin-check plus WebSocket handshake tests.

## Best Possible Solution

Land a current-main patch that applies `scheme://host` fallback only to incoming opaque origins, preserves exact configured allowlist strings, adds fail-closed regression tests, and includes redacted runtime handshake proof.

## Reproduction Assessment

Yes. Current main compares configured entries against `URL.origin`, and the Node probe shows custom schemes serialize that origin as `null` before exact allowlist matching.

## Solution Assessment

No. The incoming-origin fallback is the right direction, but parsing configured allowlist entries broadens fail-closed config semantics; the safer fix keeps configured values literal and only changes parsed incoming opaque origins.
