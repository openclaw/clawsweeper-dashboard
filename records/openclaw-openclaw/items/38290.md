## Summary

Keep open: current main still rejects exact `chrome-extension://<id>` origins because it compares the allowlist to `URL.origin === "null"`, and this PR is a narrow, correct fix with sufficient real behavior proof and no blocking review findings.

## What This Changes

This PR changes Gateway browser-origin allowlist normalization so exact opaque-scheme origins can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake coverage for `chrome-extension://` origins.

## Best Possible Solution

Land this exact-origin normalization fix or the overlapping custom-scheme equivalent after rebase and validation, while keeping scheme wildcard support in the separate policy issue.

## Reproduction Assessment

Yes, at source level. Node returns `URL.origin === "null"` for `chrome-extension://...`, and current main compares that value directly to exact `gateway.controlUi.allowedOrigins` entries.

## Solution Assessment

Yes. Normalizing both request and configured origins through URL parsing plus a `scheme://host` fallback is the narrow maintainable fix for exact opaque-scheme origins without adding wildcard or auth-policy changes.
