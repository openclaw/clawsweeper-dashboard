## Summary

Keep open: current main still rejects exact hosted custom-scheme origins, and this PR contains useful source work plus real screenshot proof, but the patch widens configured allowlist semantics on a security-sensitive Gateway boundary.

## What This Changes

The PR changes Gateway origin normalization and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can pass browser WebSocket origin checks.

## Best Possible Solution

Land a consolidated Gateway fix that preserves hosted opaque request origins as `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, with negative coverage for path, query, and userinfo entries.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
