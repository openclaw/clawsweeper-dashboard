## Summary

Keep open: current main still collapses exact `chrome-extension://<id>` origins to `URL.origin === "null"`, and this PR is a narrow valid fix with sufficient screenshot proof, but the branch conflicts with current main and overlaps another open Gateway-origin PR.

## What This Changes

The PR normalizes Gateway browser-origin matching for exact `chrome-extension://<id>` allowlist entries and adds unit plus WebSocket handshake regression coverage.

## Best Possible Solution

Land one rebased exact opaque-origin normalization fix in Gateway, keeping the WebSocket handshake coverage, while leaving scheme-level wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

Yes for the exact-origin bug. Symmetric normalization of incoming and configured opaque origins is the narrow maintainable fix; broader wildcard matching belongs in https://github.com/openclaw/openclaw/issues/39739.
