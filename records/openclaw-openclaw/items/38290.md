## Summary

Keep open: the underlying Gateway origin bug is valid and current main still rejects exact hosted opaque origins, but this PR widens a security-sensitive allowlist by parsing configured entries and stripping path/userinfo before matching.

## What This Changes

This PR changes Gateway browser-origin matching so exact `chrome-extension://` entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while accepting only literal configured origins or `*`, borrowing this PR's handshake coverage and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path or userinfo.
