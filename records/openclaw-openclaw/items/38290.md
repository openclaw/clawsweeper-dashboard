## Summary

Keep open: current main and the latest release still have the exact `chrome-extension://` allowlist miss, and this PR is a narrow, well-scoped fix with real behavior proof. The remaining blocker is maintainer consolidation with the overlapping open Gateway-origin PR and a conflict/rebase before landing.

## What This Changes

The PR changes Gateway browser-origin matching to normalize opaque URL-scheme origins to exact `scheme://host` values and adds unit plus WebSocket handshake coverage for allowlisted `chrome-extension://` origins.

## Best Possible Solution

Land one exact opaque-origin normalization fix in Gateway with focused unit and WebSocket handshake coverage, while keeping scheme-wildcard semantics isolated to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node parses `chrome-extension://...` to `URL.origin === "null"`, and current main compares that parsed value against the exact configured allowlist entry used by the Gateway browser-origin WebSocket path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming opaque origins is the narrow maintainable fix; broader wildcard or pattern matching should remain in the separate policy discussion at https://github.com/openclaw/openclaw/issues/39739.
