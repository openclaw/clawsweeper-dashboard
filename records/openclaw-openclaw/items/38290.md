## Summary

Keep open: current main still has the exact-origin bug this PR is trying to fix, but the PR diff still widens configured allowlist semantics by parsing configured entries as arbitrary URLs and stripping path, query, and userinfo before matching on a security-sensitive Gateway origin boundary.

## What This Changes

The PR changes Gateway origin parsing and tests so exact `chrome-extension://<id>` browser-origin allowlist entries can pass Control UI WebSocket origin checks.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, borrowing this PR's handshake coverage and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
