## Summary

Keep open. Current main and the latest release still reject exact hosted custom-scheme origins, while this PR is a valid narrow fix with unit, WebSocket handshake coverage, and sufficient after-fix screenshot proof; it is not safe to close before an equivalent patch lands.

## What This Changes

The PR normalizes Gateway browser-origin matching for exact non-standard origins such as chrome-extension://<id> and adds origin-check plus WebSocket handshake regression tests.

## Best Possible Solution

Land one rebased exact opaque-origin normalization fix in Gateway, keeping the WebSocket handshake coverage, while leaving scheme-level wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores URL.origin and compares parsedOrigin.origin to gateway.controlUi.allowedOrigins, while Node reports chrome-extension:// and similar hosted non-special schemes with URL.origin === "null".

## Solution Assessment

Yes for the exact-origin bug. Symmetric normalization of incoming and configured opaque origins is the narrow maintainable direction; broader chrome-extension://* matching remains a separate policy decision in https://github.com/openclaw/openclaw/issues/39739.
