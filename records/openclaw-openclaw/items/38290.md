## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` allowlist entries because the Gateway compares against `URL.origin === "null"`, and this PR is a focused implementation with real screenshot proof rather than cleanup-close material.

## What This Changes

The branch updates Gateway browser-origin matching to normalize exact non-standard origins such as `chrome-extension://<id>` to `scheme://host`, with unit and WebSocket handshake regression tests.

## Best Possible Solution

Land this or the newer equivalent exact opaque-origin normalization fix with regression coverage, keeping scheme wildcard support in the separate Gateway policy discussion.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://abcdefghijklmnop` to `URL.origin === "null"` and then compares that value against exact `gateway.controlUi.allowedOrigins` entries; I did not run Vitest in this read-only review.

## Solution Assessment

Yes. Symmetric normalization of parsed request origins and configured allowlist entries is the narrow fix for exact opaque-origin matching; scheme-level wildcards should remain separate.
