## Summary

Keep open: this fixes a real Gateway origin-matching bug and has convincing after-fix proof, but the branch also broadens configured allowlist semantics in a security-sensitive path and needs a narrow repair before merge.

## What This Changes

The PR changes Gateway origin normalization so exact `chrome-extension://<id>` allowlist entries can match and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Keep configured `allowedOrigins` entries exact, add the opaque-origin `scheme://host` fallback only for parsed request origins, and consolidate the focused Chrome-extension coverage with the overlapping custom-origin PR path.

## Reproduction Assessment

Yes, as a source-level reproduction: current main compares `allowedOrigins` to `URL.origin`, and Node returns `"null"` for `chrome-extension://<id>`, so exact extension origins cannot match. I did not run the Vitest lane in this read-only review.

## Solution Assessment

No, not as written: the request-origin fallback is the right narrow shape, but normalizing configured allowlist entries through URL parsing broadens standard-origin config semantics. The safer fix preserves exact config entries and adds the opaque-origin fallback only on the request-origin side.
