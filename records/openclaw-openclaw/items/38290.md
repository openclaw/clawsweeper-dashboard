## Summary

Keep open. The underlying Gateway bug is still valid on current main and the PR includes sufficient real-use proof, but the patch introduces a security-sensitive allowlist widening by parsing configured entries and matching their stripped origins.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://` entries in `gateway.controlUi.allowedOrigins` can match, with focused unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or validated origin-only strings, borrowing this PR's handshake coverage and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject path-bearing allowlist entries:** `src/gateway/origin-check.ts:47`

  - confidence: 0.86
