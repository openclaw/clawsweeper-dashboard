## Summary

Keep open: current main and the latest release still have the exact-origin custom-scheme bug, and this PR contains useful proof, but the patch is not safe to merge as written because it broadens configured allowlist interpretation and permits empty-host opaque origins.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://<id>` values in `gateway.controlUi.allowedOrigins` can pass WebSocket allowlist checks, with unit and handshake coverage.

## Best Possible Solution

Land a current-main Gateway origin-check fix that normalizes only incoming hosted opaque origins, keeps configured allowlist strings exact except `*`, rejects empty-host opaque origins, and covers the WebSocket plus HTTP caller paths with focused tests.

## Reproduction Assessment

## Solution Assessment

No, not as written. The parser-local fallback is the right layer, but this PR should preserve exact configured allowlist strings and reject empty-host opaque origins before merge.
