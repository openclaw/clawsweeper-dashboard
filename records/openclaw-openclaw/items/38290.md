## Summary

Keep open. Current main still has the reported gateway origin-matching bug, and the PR is a narrow patch for the right surface: it changes exact allowlist normalization for opaque non-standard origins and adds focused unit plus WebSocket handshake coverage. I did not find a blocking correctness or security finding in the proposed diff.

## What This Changes

The PR changes gateway browser-origin allowlist matching to use a protocol/host fallback for opaque non-standard origins and adds unit plus WebSocket handshake tests for exact chrome-extension allowlisting.

## Best Possible Solution

Land this PR after rebase, or land an equivalent narrow gateway patch that symmetrically normalizes incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins`, using `URL.origin` for standard origins and a protocol/host fallback only for opaque-origin schemes. Keep scheme wildcard support as the separate product/security decision tracked in #39739.

## Reproduction Assessment

Yes. On current main, evaluating `new URL("chrome-extension://abcdefghijklmnop").origin` yields `"null"`, while the configured allowlist entry remains `"chrome-extension://abcdefghijklmnop"`, so the current `checkBrowserOrigin` exact comparison cannot match.

## Solution Assessment

Yes. The proposed approach is the narrowest maintainable fix I see: normalize both sides consistently while preserving exact allowlist semantics, literal `Origin: null` rejection, and existing token/device auth behavior.
