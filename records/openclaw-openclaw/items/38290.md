## Summary

Keep open: current main and the latest release still reject exact chrome-extension origins, while this PR is a narrow correct fix that needs maintainer rebase/consolidation before landing.

## What This Changes

The PR normalizes opaque Gateway browser origins so exact chrome-extension entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake regression tests.

## Best Possible Solution

Land one exact opaque-origin normalization path with focused matcher and WebSocket regression coverage, while keeping scheme wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node reports `URL.origin === "null"` for `chrome-extension://...`, current main compares that parsed value against the exact allowlist entry, and the Gateway WebSocket handshake routes browser-origin clients through the same helper.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming exact opaque origins is the narrow maintainable fix; broader `chrome-extension://*` or scheme-wildcard behavior belongs in the separate product/security discussion.
