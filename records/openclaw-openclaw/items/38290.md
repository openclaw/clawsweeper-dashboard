## Summary

Keep open: current main still has the exact custom-scheme origin bug, but this PR should not merge as written because it broadens configured allowlist entries across the Gateway browser-origin trust boundary.

## What This Changes

The PR changes Gateway browser origin normalization so exact chrome-extension:// origins can match gateway.controlUi.allowedOrigins and adds focused unit plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, adds chrome-extension/app/tauri/electron regression coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares raw allowedOrigins entries to new URL(origin).origin, and Node returns "null" for hosted custom schemes such as chrome-extension:// and tauri://, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming opaque-origin requests is the right narrow direction, but configured allowlist entries should not be parsed in a way that strips paths, queries, or fragments into broader origin grants.
