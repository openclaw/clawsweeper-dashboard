## Summary

Keep open. Current main still rejects exact opaque-scheme origins such as `chrome-extension://<id>` in `gateway.controlUi.allowedOrigins`, and this PR is a narrow, correct fix with inspected real behavior proof; it needs maintainer landing/rebase because it conflicts with main and overlaps https://github.com/openclaw/openclaw/pull/73511.

## What This Changes

The branch changes Gateway browser-origin matching to normalize opaque-scheme origins to scheme plus host and adds unit plus WebSocket handshake tests for exact `chrome-extension://<id>` allowlist entries.

## Best Possible Solution

Land one rebased exact opaque-origin normalization fix in Gateway, preferably retaining this PR's WebSocket handshake coverage, while keeping scheme wildcard policy separate in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted opaque schemes such as `chrome-extension://<id>` into `URL.origin === "null"` and then compares that value against the exact configured allowlist entry in the Gateway WebSocket path.

## Solution Assessment

Yes for the exact-origin bug. Symmetric normalization of incoming and configured opaque origins is the narrow maintainable fix; broader wildcard matching belongs in https://github.com/openclaw/openclaw/issues/39739.
