## Summary

Keep open: current main still collapses exact `chrome-extension://` origins through `URL.origin`, while this PR remains a focused fix with unit, handshake, and inspected real-behavior proof. The overlapping https://github.com/openclaw/openclaw/pull/73511 should be reconciled during maintainer landing rather than used to close this PR.

## What This Changes

The PR changes gateway browser-origin allowlist matching to normalize opaque URL origins via a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated gateway patch that symmetrically normalizes incoming and configured hosted opaque origins, keeps wildcard policy separate, and retains the handshake regression test.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe show current main parses `chrome-extension://abcdefghijklmnop` to comparable origin `"null"`, while the allowlist still contains the exact extension origin string.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes both sides consistently, preserves exact allowlist semantics, and leaves scheme wildcard behavior to the separate policy issue.
