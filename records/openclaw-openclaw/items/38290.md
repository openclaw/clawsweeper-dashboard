## Summary

Keep open: current main and the latest release still have the exact opaque-origin allowlist bug, but this PR is not merge-ready because it reparses configured allowlist entries on a browser-origin authorization boundary, has only screenshot proof for browser-runtime behavior, and is currently dirty against main.

## What This Changes

The PR changes Gateway browser-origin matching so exact opaque origins such as `chrome-extension://<id>` can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake tests.

## Best Possible Solution

Land a refreshed narrow gateway parser fix that applies `scheme://host` fallback to incoming opaque origins only, preserves exact configured allowlist strings, and includes fail-closed tests plus redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main compares configured entries against `URL.origin`, and a Node probe shows the reported opaque/custom schemes serialize that origin as `null` before the exact allowlist check.

## Solution Assessment

No. The parser-local fix direction is right, but this branch also reparses configured allowlist strings and can change fail-closed configs into accepted origins; the safer fix normalizes only incoming opaque origins.
