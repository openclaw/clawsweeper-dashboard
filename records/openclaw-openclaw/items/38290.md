## Summary

Keep open: current main and the latest release still have the exact custom-scheme origin bug, but this PR broadens allowlist semantics by parsing configured entries and needs a narrow repair or consolidation before merge.

## What This Changes

The branch changes Gateway origin matching so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist strings, keeps custom-scheme allow/reject coverage plus the useful WebSocket handshake test, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses hosted custom-scheme origins to `URL.origin === "null"` and then compares that parsed value against exact configured allowlist entries, so an exact `chrome-extension://<id>` entry cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for incoming hosted opaque origins is the right narrow direction, but configured allowlist entries should remain exact raw strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect
