## Summary

Keep open. Current main still has the exact-extension-origin allowlist bug and this PR is the right overall direction, but the patch should guard the opaque-origin fallback so hostless schemes are not collapsed into the same synthetic origin.

## What This Changes

The PR changes Gateway browser-origin matching to use a `scheme://host` fallback for opaque-origin schemes and adds chrome-extension unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one current-main exact-origin normalization fix that preserves this PR's chrome-extension handshake coverage, guards hostless opaque origins, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node reports hosted non-special schemes like `chrome-extension://...` with `URL.origin === "null"`, and current main compares that parsed value against exact allowlist strings.

## Solution Assessment

No, not quite as written. The parser-level fallback is the right maintainable shape, but it should synthesize `scheme://host` only when a host exists so hostless opaque schemes remain rejected or unmatched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Guard the opaque-origin fallback with a host:** `src/gateway/origin-check.ts:17`
  - body: The fallback runs for every URL whose `origin` is `"null"`, including hostless schemes such as `file:///tmp/a` and `data:...`. Because allowlist entries use the same normalization, `allowedOrigins: ["file:///trusted"]` and an incoming `Origin: file:///other` both become `file://` and pass. Please only synthesize `scheme://host` when `url.host` is non-empty.
  - confidence: 0.84

## Security Review

  - body: The new `url.origin === "null"` fallback returns `scheme://` even when `url.host` is empty, so different hostless opaque URLs can normalize to the same allowlist key on a browser-origin security check.
