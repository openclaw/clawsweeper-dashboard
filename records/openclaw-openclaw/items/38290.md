## Summary

Keep open: current main still rejects exact custom-scheme browser origins, and the PR has useful proof and coverage, but the diff introduces a security-relevant broadening of configured `allowedOrigins` entries and needs a focused repair before merge.

## What This Changes

The PR changes Gateway browser-origin normalization so exact `chrome-extension://` origins can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, adds custom-scheme and no-broadening regression coverage, keeps the real extension proof, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares `allowedOrigins` against `new URL(origin).origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://` and `tauri://`, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming opaque-origin requests is the right narrow direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.

## Review Findings
