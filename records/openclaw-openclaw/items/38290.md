## Summary

Keep open. The reported Gateway bug is valid and the PR has useful real-behavior proof, but the current diff introduces a security-sensitive allowlist widening by parsing configured `allowedOrigins` as arbitrary URLs and matching the stripped origin.

## What This Changes

The PR changes Gateway origin normalization and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can pass browser WebSocket origin checks.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, with negative coverage for path, query, and userinfo-bearing allowlist values.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.
