## Summary

Keep open: the root Gateway origin bug is valid and is not fixed on current main or the latest release, but this branch is not merge-ready because it reparses configured allowlist entries, lacks diagnostic real behavior proof for a security-boundary change, and is currently conflicting.

## What This Changes

This PR changes Gateway origin matching so non-standard browser origins such as chrome-extension://<id> can match gateway.controlUi.allowedOrigins and adds unit plus WebSocket handshake tests.

## Best Possible Solution

Land a refreshed narrow gateway parser fix that applies the opaque-origin fallback only to incoming request origins, preserves exact configured allowlist strings, adds fail-closed tests, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.6 parse URL.origin and compare exact configured entries, while a Node probe shows chrome-extension://, app://, and tauri:// serialize their origin as null.

## Solution Assessment

No. The parser-local fallback direction is right, but this branch also reparses configured allowlist entries; the safer fix keeps config literals exact and normalizes only incoming opaque origins.
