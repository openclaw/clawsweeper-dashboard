## Summary

Keep open. The underlying Gateway bug is still present on current main and the PR is useful, but the current diff broadens configured allowlist semantics at a browser-origin trust boundary and needs a narrow repair before merge.

## What This Changes

The PR changes Gateway origin normalization and tests so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins` during browser WebSocket handshakes.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, handles hosted opaque custom schemes, ports unit and handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses hosted custom schemes to `URL.origin === "null"` and then compares that parsed value against exact configured allowlist strings, so exact extension/custom-scheme origins cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for hosted opaque request origins is the right direction, but configured standard allowlist entries should remain exact strings instead of being parsed into broader origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve exact allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: Parsing configured allowlist entries through `parseOrigin()` strips path, query, and hash, so an operator entry like `https://trusted.example/path` now authorizes all of `https://trusted.example`. Current main keeps configured entries as exact lowercased strings and rejects that non-origin entry; keep the opaque-origin fallback without broadening standard entries.
