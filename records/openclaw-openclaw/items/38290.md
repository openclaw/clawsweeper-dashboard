## Summary

Keep open: current main and the latest stable tag still compare configured origins to `URL.origin`, so exact `chrome-extension://<id>` and similar opaque origins remain broken. This PR is a narrow, valid fix path, but it needs maintainer handling because the head is conflicting and overlaps another open exact custom-origin PR.

## What This Changes

The PR adds symmetric fallback normalization for opaque Gateway browser origins and adds unit plus WebSocket handshake regression tests for exact `chrome-extension://<id>` allowlist entries.

## Best Possible Solution

Land one exact opaque-origin normalization path with focused matcher and WebSocket regression coverage, while keeping scheme wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin`, Node returns `"null"` for `chrome-extension://...`, and the Gateway WebSocket handshake routes through that exact allowlist comparison.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming opaque exact origins is the narrow maintainable fix; broader `chrome-extension://*` matching should stay in the separate policy issue.
