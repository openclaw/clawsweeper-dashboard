## Summary

Keep open: the underlying Gateway origin bug is valid and the contributor proof is sufficient, but the patch is not merge-ready because it broadens configured allowlist entries across a browser-origin trust boundary and the branch currently conflicts with main. The useful Chrome-extension handshake coverage should either be repaired here or consolidated with https://github.com/openclaw/openclaw/pull/73511.

## What This Changes

The branch changes Gateway browser-origin matching so exact `chrome-extension://<id>` entries can match `gateway.controlUi.allowedOrigins`, with origin-check and WebSocket handshake tests.

## Best Possible Solution

Land one consolidated exact hosted custom-scheme origin fix that preserves configured allowlist strings as exact entries, keeps literal `Origin: null` rejected, includes unit plus handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin` and compares it to exact `allowedOrigins`; Node reports hosted custom schemes such as `chrome-extension://` with `origin:"null"`, so exact entries cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader URL origins.
