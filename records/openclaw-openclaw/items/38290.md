## Summary

Keep this PR open: current main still has the exact extension/custom-scheme origin bug, and the PR contains useful tests and real proof, but the patch should not merge as written because it broadens configured allowlist entries at a Gateway origin-policy boundary and is currently conflicting with main.

## What This Changes

The branch changes Gateway origin normalization and tests so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins` during browser WebSocket handshakes.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, covers allowed and rejected hosted custom-scheme origins, ports the WebSocket handshake coverage, and leaves scheme wildcards to the separate product/security discussion.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` to `origin="null"` and then compares that parsed value against exact configured strings, so an exact extension origin cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for hosted opaque-origin request Origins is the right direction, but configured standard allowlist entries should remain exact strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

Current main keeps configured entries as exact lowercased strings and rejects that non-origin entry; keep the opaque-origin fallback without broadening standard entries.
  - confidence: 0.88
