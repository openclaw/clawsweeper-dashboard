## Summary

Keep open. Current main and the latest release still collapse exact chrome-extension origins to the opaque URL origin string, and this PR is a narrow, correct fix with relevant unit and WebSocket handshake coverage; maintainer consolidation is needed because it overlaps the newer custom-scheme origin PR.

## What This Changes

The PR normalizes Gateway browser origins so exact non-standard extension origins such as `chrome-extension://<id>` can match `gateway.controlUi.allowedOrigins`, with origin-check and WebSocket handshake regression tests.

## Best Possible Solution

Land one consolidated exact opaque-origin normalization fix with focused matcher and WebSocket regression coverage, while leaving scheme wildcard semantics to `https://github.com/openclaw/openclaw/issues/39739`.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` with `URL.origin === "null"` and then compares that parsed value against the exact allowlist entry used by the Gateway WebSocket handshake path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming exact opaque origins is the narrow maintainable fix; broader scheme wildcard matching should stay in the separate security/product discussion.
