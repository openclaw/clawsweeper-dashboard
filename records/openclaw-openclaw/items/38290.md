## Best Possible Solution

Land one consolidated exact opaque-origin fix that keeps configured `allowedOrigins` as exact trimmed/lowercased strings, applies `scheme://host` fallback only to hosted custom-scheme request origins, adds unit and handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin` and compares it to exact `allowedOrigins`; Node reports hosted custom schemes such as `chrome-extension://` with `origin:"null"`, so exact entries cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader URL origins.
