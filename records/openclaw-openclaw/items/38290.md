## Summary

Keep open. Current main and the latest release still reject exact `chrome-extension://<id>` allowlist entries, and this PR has useful real-world proof plus focused tests, but the patch should not merge as written because it broadens configured standard URL allowlist entries into full-origin grants.

## What This Changes

The PR changes Gateway origin normalization and tests so exact `chrome-extension://<id>` browser origins can match `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, covers allowlisted and rejected hosted custom-scheme origins, ports the useful WebSocket handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` through `URL.origin === "null"` and compares that parsed value against explicit `gateway.controlUi.allowedOrigins` entries, so the exact allowlist cannot match.

## Solution Assessment

No, not as written. The custom-scheme fallback is the right direction, but configured allowlist entries should remain exact strings for standard origins instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve exact allowlist semantics:** `src/gateway/origin-check.ts:44`
  - body: Parsing configured allowlist entries with `parseOrigin()` changes existing exact matching: `https://trusted.example/path` now normalizes to `https://trusted.example` and authorizes that whole origin, where current main rejects the non-origin string. Keep the fallback for opaque incoming origins, but do not strip path/query/hash from configured standard URL entries.
  - confidence: 0.86
