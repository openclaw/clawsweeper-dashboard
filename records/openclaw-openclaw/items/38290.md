## Summary

Keep open. Current main still has the exact opaque-origin mismatch, and this PR is a narrow, reviewable fix with no blocking code findings; it should be rebased and rerun rather than cleaned up or closed.

## What This Changes

The PR changes Gateway browser-origin allowlist matching so exact non-standard origins such as `chrome-extension://<id>` normalize to `scheme://host`, with unit and WebSocket handshake regression tests.

## Best Possible Solution

Land this or an equivalent exact opaque-origin normalization fix in the Gateway origin checker, keeping scheme-level wildcard support tracked separately.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe show that current main parses `chrome-extension://...` to `URL.origin === "null"` and then compares that value against exact `gateway.controlUi.allowedOrigins` entries, so the configured extension origin cannot match.

## Solution Assessment

Yes. Symmetric normalization of parsed request origins and configured allowlist entries is the narrow maintainable fix; `chrome-extension://*` wildcard support should remain in the separate policy/security discussion.
