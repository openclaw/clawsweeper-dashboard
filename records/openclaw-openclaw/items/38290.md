## Summary

Keep open. Current main and v2026.5.7 still reject exact `chrome-extension://` custom-scheme origins because `URL.origin` becomes `"null"`; this PR remains a valid narrow fix with sufficient screenshot proof, but it overlaps https://github.com/openclaw/openclaw/pull/73511 and needs maintainer consolidation before landing.

## What This Changes

The PR changes Gateway origin matching to normalize opaque browser origins for exact allowlist comparison and adds origin-check plus WebSocket handshake tests for `chrome-extension://` origins.

## Best Possible Solution

Land one narrow opaque/custom-scheme origin normalization path with unit and WebSocket regression coverage, while keeping scheme-wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node returns `URL.origin === "null"` for `chrome-extension://...`, current main compares that parsed value to the exact allowlist entry, and the Gateway WebSocket handshake uses this helper.

## Solution Assessment

Yes. Exact `scheme://host` fallback for opaque origins is a narrow maintainable fix because it feeds the existing explicit allowlist gate without adding wildcard or auth bypass semantics.
