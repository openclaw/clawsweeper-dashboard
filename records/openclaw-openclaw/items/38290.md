## Summary

Keep open. The Gateway bug is real on current main, and the PR has useful proof and handshake coverage, but the patch is not merge-ready because it broadens configured allowlist entries at a browser-origin trust boundary.

## What This Changes

The PR updates Gateway origin normalization and tests so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins` during browser WebSocket handshakes.

## Best Possible Solution

Land one consolidated parser-level fallback for hosted opaque origins that preserves configured allowlist strings, ports the WebSocket handshake regression coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` through `URL.origin === "null"` and compares that parsed value against exact `gateway.controlUi.allowedOrigins` entries, so the exact allowlist cannot match.

## Solution Assessment

No, not as written. The hosted opaque-origin fallback is the right direction, but configured allowlist entries should remain exact lowercased strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Keep the opaque-origin fallback for incoming origins, but do not strip path/query/hash from configured allowlist entries.
