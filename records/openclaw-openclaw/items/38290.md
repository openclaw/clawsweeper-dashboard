## Summary

Keep open: current main still has the exact extension-origin allowlist bug and this PR has useful proof and tests, but the patch widens configured allowlist entries by parsing them as URLs before matching, so it needs repair or replacement before merge.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://` entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, borrowing this PR's handshake coverage and leaving scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject path-bearing allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: `normalizeAllowedOrigin` now parses configured entries through `URL` and compares the serialized origin. A value like `https://control.example.com/path` or `chrome-extension://id/page` will authorize `https://control.example.com` or `chrome-extension://id`, even though docs require full origins and current main only matched the literal configured string; reject or ignore non-origin allowlist strings instead of stripping them.
  - confidence: 0.86
