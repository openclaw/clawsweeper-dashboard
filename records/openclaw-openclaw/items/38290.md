## Summary

Keep open: current main still rejects exact hosted custom-scheme origins, but this PR should not merge as written because it parses configured allowlist entries as URLs and can authorize a stripped origin from a path, query, or userinfo-bearing value.

## What This Changes

The PR changes Gateway origin normalization and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can pass browser WebSocket origin checks.

## Best Possible Solution

Land a consolidated Gateway fix that normalizes hosted opaque request origins to `scheme://host` while keeping configured `allowedOrigins` strict literals or `*`, with negative coverage for path, query, and userinfo-bearing allowlist values.

## Reproduction Assessment

Yes, at source level. Current main turns hosted custom schemes such as `chrome-extension://...` into `URL.origin === "null"` and then compares that value against exact configured `allowedOrigins` in the Gateway WebSocket path.

## Solution Assessment

No, not as written. The request-origin fallback is the right direction, but configured `allowedOrigins` should remain strict origins instead of being parsed as arbitrary URLs and normalized by stripping path, query, or userinfo.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject path-bearing allowlist entries:** `src/gateway/origin-check.ts:47`

A value like `https://control.example.com/path` or `chrome-extension://id/page` will authorize the stripped origin, even though docs require full origins and current main only matched the literal configured string; reject or ignore non-origin allowlist strings instead of stripping them.
  - confidence: 0.86
