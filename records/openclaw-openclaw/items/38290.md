## Summary

Keep open. Current main still has the exact-origin bug, and this PR is a narrow, plausible fix with real behavior proof; the remaining work is maintainer consolidation/rebase because the head is dirty and overlaps with https://github.com/openclaw/openclaw/pull/73511.

## What This Changes

The PR changes Gateway browser-origin matching to normalize opaque URL origins via a scheme/host fallback and adds unit plus WebSocket handshake tests for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that preserves exact hosted opaque-origin matching, keeps scheme wildcard policy separate, and retains the WebSocket handshake regression test.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe show current main parses chrome-extension://abcdefghijklmnop to origin "null" while the configured allowlist keeps the exact chrome-extension:// string, so the exact match cannot succeed.

## Solution Assessment

Yes. Symmetric normalization through a scheme/host fallback is a narrow maintainable fix for exact hosted opaque origins, while wildcard support remains a separate product/security decision at https://github.com/openclaw/openclaw/issues/39739.
