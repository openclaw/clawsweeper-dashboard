## Summary

Keep open: current main and the latest release still reject exact host-bearing opaque origins such as `chrome-extension://<id>`, and this PR targets the right Gateway boundary, but the submitted patch is not merge-ready because it broadens configured allowlist entries and conflicts with newer origin hardening on main.

## What This Changes

The PR changes Gateway browser-origin matching to normalize host-bearing opaque schemes such as `chrome-extension://<id>` for exact `gateway.controlUi.allowedOrigins` entries, with unit and WebSocket handshake coverage.

## Best Possible Solution

Refresh onto current main, accept exact canonical host-bearing opaque origins, reject non-canonical configured origins before comparison, and leave scheme wildcard support to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes. Current source plus Node URL behavior show that `chrome-extension://<id>` and `app://localhost` parse to `URL.origin === "null"`, so exact configured strings cannot match the current literal allowlist set.

## Solution Assessment

No, not as submitted. Symmetric opaque-origin normalization is the right core idea, but configured entries must be constrained to canonical origins before normalization so invalid or path-bearing values do not gain broader authority.
