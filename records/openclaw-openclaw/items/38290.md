## Summary

Keep open: the underlying Gateway custom-scheme origin bug is still present on current main and this PR has useful real behavior proof, but the patch currently broadens configured allowlist entries across a Gateway browser-origin trust boundary.

## What This Changes

The PR changes Gateway browser-origin matching so exact `chrome-extension://<id>` entries can pass `gateway.controlUi.allowedOrigins`, with origin-check unit coverage and a WebSocket handshake test.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist strings, falls back to `scheme://host` only for incoming hosted custom-scheme origins, adds no-broadening coverage and handshake coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node returns `null` for hosted custom schemes such as `chrome-extension://`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Keep the custom-scheme request-origin fallback without broadening configured allowlist semantics.
