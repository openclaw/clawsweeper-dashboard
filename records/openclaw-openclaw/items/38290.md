## Summary

Keep open. Current main still has the exact custom-scheme origin bug and this PR targets the right Gateway path, but the patch is not mergeable as written because it broadens configured allowlist entries by parsing them into whole origins.

## What This Changes

The PR normalizes both incoming Gateway browser origins and configured allowedOrigins, then adds unit and WebSocket handshake tests for chrome-extension:// allowlisting.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, keeps unit and WebSocket handshake coverage for hosted opaque schemes, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node returns URL.origin === "null" for chrome-extension:// and other hosted non-special schemes, while current checkBrowserOrigin compares that parsed origin to exact allowedOrigins entries.

## Solution Assessment

No, not as written. Falling back for incoming opaque-origin requests is the right direction, but configured allowlist entries should remain exact raw strings so paths, queries, and fragments are not stripped into broader origin grants.

## Review Findings

Overall correctness: patch is incorrect
