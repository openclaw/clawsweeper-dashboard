## Summary

Keep open: the exact extension-origin bug is valid and current main still rejects `chrome-extension://<id>` allowlist entries, but this branch should not merge as written because it broadens configured allowlist entries by reparsing them as URL origins.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to accept exact hosted custom-scheme origins such as `chrome-extension://<id>` and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one focused Gateway fix that preserves configured `allowedOrigins` as exact trimmed/lowercased entries, applies the `scheme://host` fallback only to parsed request origins with `URL.origin === "null"`, and carries regressions for extension success plus path-bearing config rejection.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured `allowedOrigins` against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://<id>` as origin `"null"`, so the exact entry cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right narrow fix, but reparsing configured allowlist entries changes the existing exact-string contract and should be replaced with exact config matching plus focused regressions.
