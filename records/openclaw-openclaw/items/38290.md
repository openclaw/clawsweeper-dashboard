## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` entries because the Gateway origin checker compares against `URL.origin`, and this PR is a narrow, well-supported fix that still needs current-head rebase/CI and maintainer landing.

## What This Changes

The branch normalizes non-standard Gateway browser origins for exact allowlist matching and adds unit plus WebSocket handshake regression coverage.

## Best Possible Solution

Rebase and land exact non-standard-origin normalization with focused origin-check and WebSocket regression tests, while keeping scheme wildcard semantics in https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin`, Node returns `"null"` for `chrome-extension://...`, and the WebSocket handshake routes through that exact allowlist comparison.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming opaque exact origins is the narrow maintainable fix; broader `chrome-extension://*` matching should stay in the separate policy issue.
