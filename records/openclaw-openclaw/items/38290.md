## Summary

Keep open: current main still has the exact custom-origin allowlist bug and the PR has credible real-behavior proof, but the patch should not broaden parseable non-origin allowlist entries into whole-origin matches. The branch is also currently conflicting with main.

## What This Changes

The PR updates Gateway browser-origin matching so exact `chrome-extension://<id>` allowlist entries normalize through `scheme://host`, with unit and WebSocket handshake coverage.

## Best Possible Solution

Land an equivalent Gateway origin-matching fix that preserves exact origin-only allowlist semantics, covers opaque-scheme and standard-origin cases, and leaves scheme wildcard policy to the separate canonical issue.

## Reproduction Assessment

Yes, at source level. Current main compares configured entries to `URL.origin`, and Node returns `"null"` for `chrome-extension://...` and other custom schemes, so exact configured origins cannot match without the fallback.

## Solution Assessment

Not quite. The core `scheme://host` fallback is the right narrow direction, but the current patch should reject or ignore configured values that are not exact browser origins instead of normalizing full URLs into broader allowed origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Preserve exact origin-only allowlist semantics:** `src/gateway/origin-check.ts:47`
  - body: `normalizeAllowedOrigin` now parses every configured value and keeps only `URL.origin`, so an existing inert entry like `https://control.example.com/app` starts authorizing the whole `https://control.example.com` origin. The docs say this setting accepts full origins only, and current main compares the configured string exactly; please reject/drop values with path/search/hash or otherwise preserve exact origin-only matching while adding the opaque-scheme fallback.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff is scoped to Gateway origin auth, but the allowlist normalization can broaden malformed URL entries unless validation remains origin-only.

Concerns:

- **[medium] Allowlist entries with paths are broadened:** `src/gateway/origin-check.ts:47`
  - body: Because configured entries are reduced through `parseOrigin(...).matchOrigin`, a value containing a path, query, fragment, or default-port spelling can begin matching a broader browser origin that did not match before. That weakens the hardening boundary for operators who misconfigured a URL instead of an origin.
  - confidence: 0.87
