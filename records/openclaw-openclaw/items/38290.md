## Summary

Keep open: the underlying Gateway bug is still valid on current main and the PR has sufficient real behavior proof, but the patch should not merge as written because it broadens configured allowlist entries by parsing them through URL.origin.

## What This Changes

The PR changes Gateway browser-origin normalization so exact chrome-extension origins can match gateway.controlUi.allowedOrigins and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, adds custom-scheme and no-broadening regression coverage, keeps the inspected real extension proof, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares raw allowedOrigins entries to new URL(origin).origin, and Node returns "null" for hosted custom schemes such as chrome-extension:// and tauri://, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming opaque-origin requests is the right narrow direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve exact allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: `normalizeAllowedOrigin()` parses configured entries through `parseOrigin()`, so a value like `https://trusted.example/path` becomes `https://trusted.example` and starts authorizing the whole origin. Keep the opaque request-origin fallback without broadening configured allowlist semantics.
  - confidence: 0.9
