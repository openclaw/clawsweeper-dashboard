## Summary

Keep open. Current main and v2026.5.12 still reject exact `chrome-extension://<id>` and other hosted custom-scheme origins because the matcher compares the allowlist to `URL.origin === "null"`; this PR is a narrow, correct fix with unit and WebSocket handshake coverage, but the branch is now merge-conflicting and overlaps another open custom-origin PR.

## What This Changes

The PR normalizes opaque hosted origins to `scheme://host` for exact Gateway origin allowlist matching and adds chrome-extension unit plus WebSocket handshake regression tests.

## Best Possible Solution

Land one consolidated parser-level fix on current main that preserves hosted opaque origins for exact allowlist matching, keeps scheme wildcard policy in https://github.com/openclaw/openclaw/issues/39739 separate, and retains unit plus WebSocket handshake coverage.

## Reproduction Assessment

Yes, at source level. Node reports `chrome-extension://...`, `app://...`, and `tauri://...` with `URL.origin === "null"`, and current `checkBrowserOrigin` compares that value against exact allowlist entries.

## Solution Assessment

Yes. Normalizing both the incoming Origin and configured allowlist entries to `scheme://host` only for opaque hosted schemes is the narrow maintainable fix, and it avoids adding wildcard or Host-header policy changes.
