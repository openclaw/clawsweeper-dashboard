## Summary

Keep open: current main and the latest release are still affected, and this PR has useful proof and coverage, but the patch should not merge until it preserves exact configured allowlist semantics and is reconciled with the overlapping custom-scheme PR.

## What This Changes

The PR adds an opaque-origin fallback for Gateway origin matching plus unit and WebSocket handshake tests for allowlisted `chrome-extension://` browser origins.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist entries, normalizes hosted opaque-origin request origins to `scheme://host`, covers allow and reject cases in unit plus handshake tests, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node returns `URL.origin === "null"` for `chrome-extension://...`, and current main compares that parsed value against exact `gateway.controlUi.allowedOrigins` entries before rejecting the WebSocket handshake.

## Solution Assessment

No, not as written. The opaque-origin fallback is the right direction, but configured standard URL allowlist entries should not be parsed in a way that strips path, query, or hash and grants the whole origin.
