## Summary

Keep open: current main still has the exact extension-origin allowlist bug, and this PR targets the right Gateway surface, but the current patch needs one security-sensitive correctness fix before merge.

## What This Changes

The PR normalizes Gateway browser-origin matching for opaque non-standard origins such as `chrome-extension://<id>` and adds origin-check plus WebSocket handshake regression tests.

## Best Possible Solution

Land one rebased Gateway fix that preserves exact hosted opaque origins, keeps non-origin allowlist entries from broadening trust, retains WebSocket handshake coverage, and leaves scheme-wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin`, Node reports hosted non-standard schemes with `origin === "null"`, and the current allowlist comparison therefore misses exact `chrome-extension://<id>` entries.

## Solution Assessment

No, not as currently patched. Symmetric opaque-origin normalization is the right direction, but parsing every configured allowlist entry into `URL.origin` can turn invalid URL-with-path entries into broader origin grants; standard entries should remain origin-only or be rejected when not exact origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep allowlist entries from broadening to origins:** `src/gateway/origin-check.ts:47`
  - body: Parsing every configured `allowedOrigins` entry drops path/search/hash/default-port details before matching. A config like `https://control.example.com/admin` or a copied `chrome-extension://.../page.html` now accepts the whole origin, while current docs require origin-only entries and current behavior rejects those invalid values. Preserve exact configured origin semantics or reject non-origin strings, and only synthesize `scheme://host` for the opaque-origin case being fixed.
  - confidence: 0.86
