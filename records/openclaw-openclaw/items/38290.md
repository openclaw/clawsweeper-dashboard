## Summary

Keep open: current main still has the exact custom-scheme origin bug, and this PR is useful, but the branch is not mergeable as written because it broadens configured `allowedOrigins` entries across a Gateway trust boundary.

## What This Changes

The PR changes Gateway browser-origin parsing so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins` and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated exact opaque-origin normalization fix that preserves exact configured allowlist semantics, covers `chrome-extension`, `app`, `tauri`, and `electron` positive and negative cases, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares raw allowlist entries to `new URL(origin).origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://`, `app://`, and `tauri://`.

## Solution Assessment

No, not as written. The incoming-origin fallback is the right narrow direction, but configured allowlist entries should not be parsed in a way that strips paths, queries, or fragments into broader origin grants.

## Review Findings
