## Summary

Keep open: the underlying exact extension-origin allowlist bug is still present on current main, but this PR currently broadens Gateway allowlist semantics and is merge-conflicting, so it needs a focused repair before merge.

## What This Changes

The PR changes Gateway browser-origin matching to allow exact `chrome-extension://<id>` origins and adds origin-check plus WebSocket handshake coverage.

## Best Possible Solution

Land a consolidated parser-level fix that adds exact custom-scheme request-origin matching, preserves raw exact allowlist semantics, adds no-broadening regression coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares configured raw entries to `URL.origin`, and Node returns `null` for hosted custom schemes such as `chrome-extension://` and `app://`, so exact entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trimmed/lowercased strings instead of being parsed into broader origins.

## Review Findings
