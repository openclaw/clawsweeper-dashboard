## Summary

Keep open: the Gateway custom-scheme origin bug is real and this PR has useful proof and coverage, but the patch currently broadens configured allowlist semantics across a browser-origin trust boundary.

## What This Changes

The PR updates Gateway browser-origin normalization and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can pass WebSocket browser-origin checks.

## Best Possible Solution

Land one consolidated exact custom-scheme origin fix that applies the `scheme://host` fallback only to parsed request origins with `URL.origin === "null"`, preserves configured `allowedOrigins` as exact trimmed/lowercased strings, and carries this PR's Chrome-extension handshake coverage.

## Reproduction Assessment

Yes, at source level. Current main compares configured exact strings to `URL.origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://...`, so an exact extension origin cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right narrow fix, but parsing configured allowlist entries changes exact-string config semantics and should be avoided or explicitly approved with security-focused tests.
