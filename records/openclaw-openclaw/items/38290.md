## Summary

Keep open. The underlying Gateway bug is still present on current main and the PR has real after-fix proof, but the patch broadens configured origin allowlist semantics and needs a small security-sensitive correction before merge.

## What This Changes

The branch adds opaque-origin normalization to Gateway browser-origin matching and adds unit plus WebSocket handshake tests for exact `chrome-extension://` origins.

## Best Possible Solution

Land a consolidated Gateway fix that preserves exact configured allowlist strings, falls back to `scheme://host` only for incoming hosted opaque origins, keeps scheme wildcards separate, and retains the WebSocket regression test.

## Reproduction Assessment

Yes, at source level. Current main stores `URL.origin` as the comparable value, and Node reports `chrome-extension://abcdefghijklmnop` as origin `"null"`, so an exact configured extension origin cannot match.

## Solution Assessment

No, not as written. The narrow fix should change incoming origin normalization while keeping configured `allowedOrigins` exact, or otherwise reject non-origin config strings instead of silently widening them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep configured allowedOrigins exact:** `src/gateway/origin-check.ts:62`
  - body: Mapping configured allowlist entries through `parseOrigin` changes the config contract: `https://control.example.com/path` now becomes `https://control.example.com`, and hostless opaque values like `foo:bar` collapse to `foo://`. That can turn an invalid or more specific entry into a broader trusted browser origin; keep config entries as trimmed/lowercased exact origins and apply the opaque-origin fallback to the incoming Origin value instead.
  - confidence: 0.86
