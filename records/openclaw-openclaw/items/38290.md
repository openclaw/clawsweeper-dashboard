## Summary

Keep open: current main still rejects exact hosted custom-scheme origins, and this PR has useful real Chrome extension proof plus focused Gateway coverage, but the patch broadens configured allowlist entries across a browser-origin trust boundary.

## What This Changes

The PR changes Gateway browser-origin normalization so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can match, with unit and WebSocket handshake tests.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, adds custom-scheme and no-broadening coverage, keeps the real extension proof, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares configured exact strings against `URL.origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://` and `tauri://`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right narrow direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Keep the custom-scheme request-origin fallback without broadening configured allowlist semantics.
