## Summary

Keep open. The reported Gateway exact-origin bug is real and the PR has sufficient after-fix behavior proof, but the patch should not merge as written because it also normalizes configured allowlist entries and can broaden entries with paths, queries, or fragments into whole-origin grants.

## What This Changes

The PR changes Gateway browser-origin matching to normalize opaque `chrome-extension://` origins to `scheme://host` and adds unit plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, adds custom-scheme unit and WebSocket handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Node returns `URL.origin === "null"` for hosted non-special schemes, and current `checkBrowserOrigin` compares that parsed value to exact configured allowlist strings.

## Solution Assessment

No, not as written. Falling back for incoming opaque-origin requests is the right direction, but configured allowlist entries should remain raw exact strings so path/query/hash-bearing values do not become broader origin grants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

Current main keeps configured entries as exact lowercased strings; keep the opaque request-origin fallback without normalizing standard allowlist entries.
