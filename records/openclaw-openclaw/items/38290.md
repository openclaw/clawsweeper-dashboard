## Summary

Keep open: current main still rejects exact `chrome-extension://<id>` origins despite explicit `gateway.controlUi.allowedOrigins`, and this PR is a focused implementation with matching regression coverage. The remaining blocker is merge readiness, not product validity: the branch is conflicting and overlaps with another open custom-scheme-origin PR.

## What This Changes

The PR changes Gateway browser-origin allowlist matching to compare opaque URL origins by a `scheme://host` fallback and adds unit plus WebSocket handshake coverage for exact Chrome extension origins.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that symmetrically normalizes incoming and configured hosted opaque origins, keeps wildcard policy separate, rejects literal `Origin: null`, and retains both unit and handshake regression coverage.

## Reproduction Assessment

Yes. Source inspection plus a Node URL probe reproduce the mismatch: current main parses `chrome-extension://abcdefghijklmnop` to comparable origin `"null"`, while the allowlist keeps the exact extension origin string.

## Solution Assessment

Yes. The proposed fix is the narrow maintainable path because it changes only the origin-normalization seam, applies the same normalization to configured and incoming origins, and leaves scheme wildcard support to https://github.com/openclaw/openclaw/issues/39739.
