## Summary

Keep open: the reported exact extension-origin bug is still present on current main and this PR contains useful fix/proof, but the branch is not merge-ready because it also changes configured allowlist entries from exact strings into parsed URL origins on a Gateway browser-origin trust boundary.

## What This Changes

The PR changes Gateway origin matching so non-standard browser origins such as `chrome-extension://<id>` can match `gateway.controlUi.allowedOrigins` and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Keep configured `allowedOrigins` entries exact, add the opaque-origin `scheme://host` fallback only for parsed request origins with a non-empty host, and carry over the Chrome-extension handshake coverage while coordinating with the overlapping custom-origin PR.

## Reproduction Assessment

Yes, as a source-level reproduction: current main compares configured entries to `URL.origin`, and Node returns `"null"` for `chrome-extension://<id>`, so exact extension origins cannot match. I did not run Vitest in this read-only review.

## Solution Assessment

No, not as written: the request-origin fallback is the right narrow fix, but parsing configured allowlist entries broadens standard-origin config semantics. The safer path keeps config entries exact and applies the fallback only to request origins.
