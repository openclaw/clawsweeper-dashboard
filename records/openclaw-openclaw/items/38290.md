## Summary

Keep open: current main still has the exact custom-origin allowlist bug and this PR has useful proof plus focused coverage, but it should not merge as written because it changes configured allowlist entry semantics on a Gateway browser-origin trust boundary.

## What This Changes

The branch changes Gateway origin matching so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins`, with unit and WebSocket handshake tests.

## Best Possible Solution

Land one consolidated exact opaque-origin normalization fix that preserves configured `allowedOrigins` entries as exact strings, adds Chrome-extension/custom-scheme allow and reject coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level: current main compares configured entries to `URL.origin`, and Node returns `"null"` for `chrome-extension://<id>`, so exact extension origins cannot match. I did not run Vitest in this read-only review.

## Solution Assessment

No, not as written: the request-origin fallback is the right narrow fix, but parsing configured allowlist entries broadens standard-origin config semantics. The safer path keeps config entries exact and applies the fallback only to request origins.
