## Summary

Keep open: the underlying Gateway origin bug is still present on current main, but this PR needs changes because it parses configured allowlist entries into whole URL origins and can widen what an operator intended to trust.

## What This Changes

The PR changes Gateway origin normalization and adds unit plus WebSocket handshake coverage so exact `chrome-extension://<id>` origins can match `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist semantics, normalizes hosted opaque request origins only, includes unit and WebSocket coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` to `URL.origin === "null"` and compares that parsed value against exact configured allowlist strings, so an exact extension origin cannot match.

## Solution Assessment

No, not as written. Falling back to `scheme://host` for hosted opaque request origins is the right direction, but configured allowlist entries should remain exact raw strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve exact allowlist entries:** `src/gateway/origin-check.ts:46`
  - body: Parsing configured allowlist entries through `parseOrigin()` strips paths, queries, and hashes, so an operator entry like `https://trusted.example/path` now authorizes the whole `https://trusted.example` origin. Keep the opaque-origin fallback for the request side without broadening standard configured entries.
