## Summary

Keep open. Current main still collapses exact `chrome-extension://<id>` origins to `"null"` before the allowlist comparison, and this PR is a narrow fix with inspected real behavior proof; the remaining blocker is that the branch currently conflicts with main.

## What This Changes

The PR normalizes opaque browser origins to `scheme://host` for exact allowlist matching and adds unit plus WebSocket handshake regression coverage.

## Best Possible Solution

Rebase the exact-origin normalization onto current main, keep it separate from the scheme-wildcard policy request, and land it with the unit and WebSocket handshake regression tests.

## Reproduction Assessment

Yes, at source level: Node returns `"null"` for `chrome-extension://...` URL origins, current main stores that parsed value, and the allowlist compares against it. I did not run Vitest because this review is read-only.

## Solution Assessment

Yes. Symmetric normalization of incoming and configured exact opaque origins is the narrow maintainable fix; wildcard patterns such as `chrome-extension://*` should stay in the separate policy discussion.
