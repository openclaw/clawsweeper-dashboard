## Summary

Keep open: the exact custom-scheme origin bug is valid and still present on current main, but this PR is not merge-ready because it broadens configured `allowedOrigins` by parsing them into URL origins at a Gateway trust boundary.

## What This Changes

The PR adds custom-scheme origin normalization for Gateway browser-origin allowlist matching, plus unit and WebSocket handshake tests for `chrome-extension://` origins.

## Best Possible Solution

Land one consolidated parser-level fix that falls back to `scheme://host` only for hosted custom-scheme request origins, keeps configured `allowedOrigins` as exact normalized strings, adds unit and handshake regression coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with `origin:"null"`, so exact allowlist entries cannot match.

## Solution Assessment

No, not as written. The incoming-origin fallback is the right direction, but configured allowlist entries should stay exact trimmed lowercase strings instead of being parsed into broader origins.
