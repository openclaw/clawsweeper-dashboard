## Summary

Keep open. Current main still rejects exact hosted custom-scheme origins, and this PR targets the right Gateway surface with sufficient real behavior proof, but the proposed allowlist normalization broadens configured standard URL entries and needs a narrow repair before merge.

## What This Changes

The PR changes Gateway browser-origin normalization so exact `chrome-extension://<id>` `gateway.controlUi.allowedOrigins` entries can pass, with origin-check unit coverage and a WebSocket handshake test.

## Best Possible Solution

Land one consolidated parser-level fix that preserves exact configured allowlist semantics, adds custom-scheme and no-broadening coverage, keeps the real extension proof, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node returns `"null"` for hosted custom schemes such as `chrome-extension://`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trim/lowercase strings instead of being parsed into broader origins.
