## Summary

Keep open: current main still has the exact custom-scheme origin allowlist bug and this PR has useful proof and coverage, but the implementation currently broadens Gateway browser-origin allowlist semantics by parsing configured entries.

## What This Changes

The branch adds fallback origin matching for hosted non-standard schemes such as `chrome-extension://...` and adds focused origin-check plus WebSocket handshake tests.

## Best Possible Solution

Land one consolidated parser-level fix that keeps configured allowlist strings exact, falls back to `scheme://host` only for incoming hosted custom-scheme origins, preserves existing auth and wildcard behavior, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings against `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with `origin:"null"`, so those exact allowlist entries cannot match.

## Solution Assessment

No, not as written. Falling back for incoming hosted custom-scheme origins is the right direction, but configured allowlist entries should remain exact trimmed lowercase strings instead of being parsed into broader origins.
