## Summary

Keep this PR open, but it is not merge-ready as written: the exact custom-scheme origin bug is real and the proof is useful, while the implementation currently broadens configured allowlist semantics across a Gateway browser-origin trust boundary.

## What This Changes

The PR changes Gateway browser-origin normalization and tests so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` can pass WebSocket browser-client authentication.

## Best Possible Solution

Land one consolidated exact custom-scheme origin fix on current main that preserves configured allowlist entries as exact strings, keeps scheme wildcard policy in https://github.com/openclaw/openclaw/issues/39739 separate, and carries both unit and WebSocket handshake coverage.

## Reproduction Assessment

Yes, at source level. Current main compares exact configured strings to `URL.origin`, and Node reports hosted custom schemes such as `chrome-extension://...` with origin `"null"`, so exact configured extension origins cannot match.

## Solution Assessment

No, not as written. The request-origin fallback is the right narrow fix, but parsing configured allowlist entries changes existing exact-string semantics and should be avoided or explicitly approved with security tests.
