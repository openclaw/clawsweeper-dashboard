## Summary

Keep open. Current main and the latest release still reject exact `chrome-extension://<id>` allowlist entries, and this PR is a narrow correct fix with sufficient real behavior proof; the remaining blocker is maintainer consolidation because the branch is conflicting and overlaps https://github.com/openclaw/openclaw/pull/73511.

## What This Changes

The branch changes Gateway browser-origin matching to normalize opaque URL-scheme origins to exact `scheme://host` values and adds unit plus WebSocket handshake coverage for allowlisted `chrome-extension://` origins.

## Best Possible Solution

Ship one exact opaque-origin normalization fix in Gateway with focused unit and WebSocket handshake coverage, then keep scheme-wildcard semantics isolated to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` with `URL.origin === "null"` and compares that parsed value against the exact allowlist entry used by the Gateway WebSocket browser-origin path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming opaque origins is the narrow maintainable fix; broader wildcard or pattern matching should remain in the separate policy discussion at https://github.com/openclaw/openclaw/issues/39739.
