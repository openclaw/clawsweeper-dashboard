## Summary

Keep open: this is a valid Gateway origin-matching bug and the PR targets the right code path, but the patch currently broadens configured standard allowlist entries at a browser-origin trust boundary.

## What This Changes

The PR changes Gateway browser-origin matching so exact non-standard origins such as `chrome-extension://<id>` can be allowlisted and adds focused unit plus WebSocket handshake coverage.

## Best Possible Solution

Land one consolidated Gateway parser fix that preserves exact configured allowlist semantics, covers allowed and rejected hosted custom-scheme origins, ports the useful WebSocket handshake coverage, and leaves scheme wildcards to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses hosted custom/extension schemes with `URL.origin === "null"` and compares that parsed value against exact allowlist entries, so an exact `chrome-extension://<id>` entry cannot match.

## Solution Assessment

No, not as written. The hosted opaque-origin fallback is the right direction, but configured standard allowlist entries should remain exact trimmed/lowercased strings instead of being parsed and stripped to whole origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve exact allowlist entries:** `src/gateway/origin-check.ts:47`
  - body: Parsing configured allowlist entries through `parseOrigin()` strips path, query, and hash, so an operator entry like `https://trusted.example/path` now authorizes `https://trusted.example`. Current main keeps configured entries as exact lowercased strings and rejects that non-origin entry; keep the opaque-origin fallback without broadening standard entries.
  - confidence: 0.86
