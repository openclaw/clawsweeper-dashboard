## Summary

Keep open: current main and the latest release still reject exact `chrome-extension://<id>` allowlist entries, and this PR is a focused fix with useful regression coverage. It should not be cleanup-closed, but it needs maintainer consolidation because the head is conflicting and overlaps https://github.com/openclaw/openclaw/pull/73511.

## What This Changes

The branch changes Gateway origin matching to normalize opaque URL-scheme origins to exact `scheme://host` values and adds unit plus WebSocket handshake coverage for allowlisted `chrome-extension://` origins.

## Best Possible Solution

Land one exact opaque-origin normalization fix in Gateway with focused unit and WebSocket handshake coverage, while keeping scheme-wildcard semantics isolated to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current main parses `chrome-extension://...` to `URL.origin === "null"` and then compares that value against the exact configured allowlist entry used by the Gateway browser-origin WebSocket path.

## Solution Assessment

Yes. Symmetric normalization of configured and incoming opaque origins is the narrow maintainable fix; broader wildcard or pattern matching should remain in the separate policy discussion at https://github.com/openclaw/openclaw/issues/39739.
