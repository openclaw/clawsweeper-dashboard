## Summary

Keep open. Current main still rejects exact `chrome-extension://<id>` allowlist entries, and this PR is a narrow fix with adequate real behavior proof; it needs rebase/consolidation with the overlapping Gateway PR and fresh validation, not closure.

## What This Changes

The PR updates Gateway origin matching to compare hosted opaque origins like `chrome-extension://<id>` via a scheme/host fallback and adds unit plus WebSocket handshake coverage.

## Best Possible Solution

Land this PR or an equivalent consolidated Gateway patch that preserves exact hosted opaque-origin matching, retains negative coverage, and leaves scheme wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, as a source-level reproduction: current main compares an exact allowlist string to `URL.origin`, and Node returns `"null"` for `chrome-extension://abcdefghijklmnop`. The WebSocket handshake path uses that same helper for browser-origin clients.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I see because it normalizes incoming and configured origins symmetrically, keeps exact allowlist semantics, and does not implement wildcard policy.
