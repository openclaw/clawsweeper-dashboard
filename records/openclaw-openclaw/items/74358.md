## Summary

Keep open. Current main still has credible Slack paths where `streaming.preview.toolProgress: false` hides tool-progress lines without preserving in-flight answer preview delivery, and the latest issue discussion explicitly separates this failure from the open flat-DM PR #56544.

## What This Changes

The issue asks Slack to keep verbose/live answer preview streaming active while hiding tool-progress messages when `streaming.preview.toolProgress` is `false`.

## Best Possible Solution

Slack should treat answer-preview delivery and tool-progress visibility as separate concerns: `toolProgress: false` should hide progress lines only, while object-form streaming config still produces in-flight answer preview through the appropriate native or draft path. Regression tests should cover the reporter's config and docs/help should align with the shared streaming contract.

## Reproduction Assessment

Yes, at source level. The reporter provided exact object-form config, and current main still defaults that config to partial/native streaming while leaving answer partial delivery dependent on native/draft gates that are not covered for `toolProgress: false`.

## Solution Assessment

Yes, the maintainable solution is a narrow Slack-owned repair with tests and docs alignment. Relying on #56544 alone or changing only `suppressDefaultToolProgressMessages` would not cover the separate disabled-progress preview-delivery path.
