## Summary

Keep open. Current main still matches the reported Slack failure shape: object-form `streaming.preview.toolProgress: false` defaults to partial mode with native transport enabled, Slack only wires draft partial updates when native streaming is not in use and preview streaming is enabled, and the Slack tests/docs do not cover the disabled-tool-progress object-config path. The later discussion correctly separates this from the flat-DM thread gate in #56544, which is sibling work rather than a complete fix for this report.

## What This Changes

The issue requests that Slack preserve in-flight answer preview text when `streaming.preview.toolProgress` is false, while hiding tool-progress lines instead of making the preview silent.

## Best Possible Solution

Slack should treat answer-preview delivery and tool-progress visibility as independent controls: `toolProgress: false` should hide progress labels, not block in-flight answer text, with regression coverage for object-form Slack config, native-streaming versus draft-preview fallback, and flat-DM sibling behavior. Slack docs/config help should be reconciled with the shared streaming contract, and #56544 should remain sibling context rather than the canonical fix for this issue.

## Reproduction Assessment

Yes. The reporter supplied a stable config and current main has a source-level reproduction path: object-form Slack streaming defaults to partial/native, the draft partial callback is skipped when `useStreaming` is true, and tests do not cover `toolProgress: false`.

## Solution Assessment

Yes for keeping this open and repairing Slack-owned dispatch behavior, but no for a one-line change to `suppressDefaultToolProgressMessages` alone. The safer fix is to decouple partial answer preview updates from the tool-progress flag, then add focused Slack regression tests and docs/config-help updates.
