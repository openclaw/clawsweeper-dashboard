## Summary

Keep open. The reporter confirmed a stable Slack repro with object-only `streaming.preview.toolProgress: false`, and current main still resolves that config into Slack partial/native streaming while the Slack dispatch path lacks a covered guarantee that answer preview delivery stays live when tool-progress preview lines are disabled. The latest discussion also correctly treats #56544 as sibling flat-DM gate work, not a complete fix for this silent multi-tool preview case.

## What This Changes

The issue requests restoring Slack verbose in-flight answer previews when `streaming.preview.toolProgress` is `false`, without showing tool-progress lines or making the experience equivalent to final-only delivery.

## Best Possible Solution

Slack should make answer-preview delivery and tool-progress visibility independent: object-form `streaming.preview.toolProgress: false` should preserve in-flight answer text while hiding interim tool/progress lines, with Slack tests covering native and draft fallback paths plus docs/config help aligned to the shared streaming contract.

## Reproduction Assessment

Yes. The reporter supplied a concrete config and stable Slack repro, and current source inspection shows that object-only config defaults to partial/native streaming while Slack's dispatch wiring still lacks coverage for answer-preview-on/tool-progress-off behavior.

## Solution Assessment

Unclear for the issue's line-only suggested fix. The maintainable solution is a focused Slack dispatch and docs fix that decouples partial answer delivery from tool-progress rendering, rather than only changing `suppressDefaultToolProgressMessages`.
