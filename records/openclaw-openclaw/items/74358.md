## Summary

Keep open. Current main still does not prove that Slack answer preview delivery is independent from `streaming.preview.toolProgress`: object-form Slack streaming defaults to `partial` with native transport enabled, draft partial updates are skipped whenever native streaming is active, and the draft fallback tests still do not cover `toolProgress: false`. The related flat-DM PR is sibling context, not a complete fix for this reported failure mode.

## What This Changes

Restore Slack in-flight answer preview for `streaming.preview.toolProgress: false` while keeping tool-progress preview lines hidden.

## Best Possible Solution

Slack dispatch should treat `toolProgress: false` as a visibility setting for progress lines only, while preserving answer preview delivery for object-form streaming configs across native streaming and draft fallback. Add focused regression coverage for the reporter's config and align Slack docs/config help with the shared streaming contract.

## Reproduction Assessment

Yes. The reporter supplied stable Slack repro steps and current source shows the object-form config defaults and callback gating needed to reproduce at source level; the fastest live confirmation is toggling only `toolProgress` from false to true and checking whether preview returns.

## Solution Assessment

Unclear for the issue's suggested one-line direction. The maintainable fix should separate answer-preview delivery from tool-progress visibility rather than only changing the `suppressDefaultToolProgressMessages` expression.
