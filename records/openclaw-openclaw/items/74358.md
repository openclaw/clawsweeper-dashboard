## Summary

Keep open. Current main still has a Slack streaming path where `streaming.preview.toolProgress: false` is treated as progress-line visibility, but answer-preview delivery is not independently guaranteed for the reporter's object-only config; the related open flat-DM PR is sibling context, not a complete fix for this failure mode.

## What This Changes

The issue asks Slack preview streaming to continue showing in-flight answer text when `streaming.preview.toolProgress` is `false`, while hiding tool-progress lines.

## Best Possible Solution

Slack should route answer partial-preview updates independently from tool-progress visibility, then add regression coverage for object-form `streaming.preview.toolProgress: false` so progress lines stay hidden while in-flight answer text still appears. Slack docs and config help should be aligned with the shared streaming contract.

## Reproduction Assessment

Yes, at source level. The reporter provided a stable config and current main still has callback wiring and tests that do not guarantee answer preview delivery when `toolProgress` is false; a live Slack run was not performed.

## Solution Assessment

Yes, the maintainable direction is a narrow Slack streaming fix that separates answer-preview delivery from progress-line visibility. Updating only #56544's flat-DM gate or only toggling `suppressDefaultToolProgressMessages` would not cover the reported disabled-progress path.
