## Summary

Keep open: the PR targets a real Dashboard reconnect/busy-state bug that is not fully implemented on current main, but the branch cannot merge as-is because it lacks real behavior proof and still has source-level correctness blockers.

## What This Changes

The branch adds active chat run metadata to Gateway hello snapshots, restores Control UI chat busy state on reconnect, keeps abortable runs busy, changes chat-tab session selection, and injects restart/shutdown interruption notices with protocol and test updates.

## Best Possible Solution

Rebase or replace the branch on current main, keep active-run hello recovery deterministic, preserve current restart-sentinel fallback behavior, and require redacted real Dashboard/Gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main clears local chat run state on Gateway hello while the hello snapshot has no active-run recovery field, and the paired Dashboard busy-state issue remains open. I did not run a live browser/Gateway session in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR must choose recovered runs deterministically, preserve fallback notices when transcript injection fails, rebase over newer restart/session code, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` is built from a run-id keyed map, so a session can have more than one active entry. Returning the first matching session can restore an older run; later events for the newer run then fail the run-id guard and leave stale busy or stream state. Prefer the current host run when still active, otherwise choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve the restart fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, path resolution, transcript-file creation, or transcript injection before it returns `false`. Because callers enqueue the fallback system event only on a `false` return, an exception after consuming the sentinel drops the restart notice; catch helper errors and return `false`.
  - confidence: 0.92
