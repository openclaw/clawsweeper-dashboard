## Summary

Keep open: current main still has the Dashboard chat reconnect busy-state gap, but this external PR is not ready to merge because it lacks real behavior proof and still has two concrete correctness blockers in the recovery and restart-notice paths.

## What This Changes

The PR adds Gateway active chat-run snapshots, Dashboard reconnect busy-state recovery, chat-tab session preservation, and restart/shutdown interruption notices with focused Gateway/UI tests.

## Best Possible Solution

Rebase or replace the branch with deterministic active-run recovery and exception-safe restart-notice fallback, then require redacted real Dashboard/Gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection on current main. The hello handler clears chatRunId/chatStream after reconnect, the snapshot has no activeChatRuns payload, and the current hasActiveRun fallback only partially restores Stop while Queue/send still depend on local run-id state.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch must choose recovered runs deterministically, preserve fallback notices on injection errors, and be reconciled with current-main recovery code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: activeChatRuns can contain multiple entries for the same session because the server snapshots every run-id entry from chatAbortControllers. Returning the first match can recover an older run, causing later events for the newer run to fail the run-id guard and leave stale busy/stream state; keep the current host run when present, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Preserve the restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, path resolution, file creation, or transcript append, but callers only enqueue the fallback system event when it returns false. Catch helper errors and return false so a consumed sentinel does not lose the user-visible restart notice.
  - confidence: 0.92
