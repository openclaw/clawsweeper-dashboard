## Summary

Keep open: current main still lacks active run recovery in the hello snapshot and still clears dashboard chat run state after hello, so the reported bug remains; the PR direction is useful but needs a rebase plus deterministic recovery, exception-safe restart fallback, and a required changelog entry before merge.

## What This Changes

The PR adds active chat run data to Gateway hello snapshots, restores Control UI chat busy state on reconnect/tab navigation, keeps abortable runs busy in chat controls, and adds shutdown/restart interruption notices with protocol client and test updates.

## Best Possible Solution

Land a rebased implementation that keeps the additive activeChatRuns protocol contract, restores the correct active dashboard run deterministically, keeps controls busy while abort is available, integrates restart/shutdown notices with current transcript and delivery helpers, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Related issue #41064 gives focused dashboard steps, and current main shows hello snapshots lack activeChatRuns while the Control UI hello handler clears chatRunId and chatStream on reconnect.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch should choose recovered runs deterministically, preserve restart-notice fallback on transcript errors, add the required changelog entry, and rebase onto current main’s newer restart/transcript paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: activeChatRuns can contain multiple entries for the same session because the server tracks active runs by runId. Returning the first match can recover an older run, causing later events for the newer run to fail run-id guards and leave stale busy or stream state; prefer the current host run if still active, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Preserve restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, transcript path resolution, file creation, or append. Because callers only enqueue the system-event fallback when this helper returns false, an exception after the sentinel is consumed drops the restart notice; catch helper errors and return false.
  - confidence: 0.92
