## Summary

Keep open: current main has partial Stop recovery, but the PR still targets real Dashboard reconnect/restart-notice gaps and is not obsolete. The branch is currently conflicting, lacks required real behavior proof, and still has two concrete correctness blockers before merge.

## What This Changes

The PR adds active chat run metadata to Gateway hello snapshots, restores Dashboard chat state from reconnect snapshots, keeps abortable runs busy, changes sidebar chat session selection, and injects restart/shutdown interruption notices with tests and generated protocol updates.

## Best Possible Solution

Rebase or replace the branch on current main, preserve the shipped session-scoped Stop recovery, add deterministic active-run recovery for reconnects, keep restart-notice fallback behavior intact, and require redacted real Dashboard/Gateway proof before merge.

## Reproduction Assessment

Yes. The source path is high-confidence: current main clears local chat run state on hello/reconnect while the hello snapshot lacks active-run metadata, and the related Dashboard busy-state issue remains open.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR must choose recovered runs deterministically, preserve fallback notices when transcript injection fails, rebase over newer restart/session code, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server tracks active runs by run ID. Returning the first matching entry can restore an older run, causing later events for the newer run to fail run-ID guards and leave stale busy or stream state; prefer the current host run when still active, otherwise choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve the restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, transcript path resolution, or file creation before returning `false`. Because callers only enqueue the fallback system event when this helper returns `false`, an exception after consuming the sentinel drops the restart notice; catch helper errors and return `false`.
  - confidence: 0.92
