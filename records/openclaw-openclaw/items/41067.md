## Summary

Keep open: this is a valid Gateway Dashboard reliability fix direction, but the current PR still has merge-blocking review findings, lacks after-fix real behavior proof, and is conflicting with newer main work that only partially covers the symptom.

## What This Changes

The PR adds Gateway hello active chat-run snapshots, Dashboard chat run recovery and busy-state handling, restart interruption transcript notices, generated protocol updates, and focused tests.

## Best Possible Solution

Rebase or replace the branch on current main, keep the useful active-run recovery direction, make recovered-run selection deterministic, make restart-notice fallback exception-safe, and require redacted real reconnect/restart proof before merge.

## Reproduction Assessment

Yes, at source level: current main clears chatRunId/chatStream during Gateway hello while the snapshot has no activeChatRuns field, so reconnect cannot restore the active run identity. I did not run a live browser/Gateway reconnect or restart session in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR head needs deterministic active-run selection, exception-safe restart fallback behavior, a current-main rebase, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server snapshots every tracked abort controller. Returning the first match can recover an older run ID, causing later deltas or finals for the newer run to fail the run-id guard and leave stale busy state; keep the current run when present, otherwise choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Fallback when restart notice injection fails:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, path resolution, file creation, or transcript append setup. The fallback callers only enqueue a system event when this helper returns `false`, so a thrown error can consume the sentinel and drop the user-visible restart notice; catch helper errors and return `false`.
  - confidence: 0.92
