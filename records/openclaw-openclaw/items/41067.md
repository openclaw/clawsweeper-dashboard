## Summary

Keep open: this is a plausible fix for an open Dashboard/Gateway bug cluster, but the PR is currently conflicting, lacks required real behavior proof, and still has two blocking correctness defects in the proposed patch.

## What This Changes

The PR adds active chat-run data to Gateway hello snapshots, restores Dashboard chat run/busy state across reconnects, adjusts chat navigation/composer behavior, and injects restart interruption notices into transcripts.

## Best Possible Solution

Rebase or replace the branch on current main, keep active-run recovery deterministic and restart-notice fallback exception-safe, then require redacted real Dashboard reconnect/restart proof before merge.

## Reproduction Assessment

Yes, at source level: seed chatRunId/chatStream, process a Gateway hello after reconnect, and current main clears the run while the snapshot lacks an active-run recovery field. I did not run a live browser/Gateway reconnect or restart session in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR head needs deterministic recovered-run selection, exception-safe restart fallback behavior, a current-main rebase, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: The server snapshots every chatAbortControllers entry, so activeChatRuns can contain multiple runs for the same session. Returning the first match can recover an older run ID; later deltas or finals for the newer run then fail the run-id guard and leave stale busy state. Keep the current run when present, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Catch restart notice injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, path resolution, file creation, or transcript append setup. Both fallback callers only enqueue the system event when the helper returns false, so a thrown error can consume the sentinel and drop the user-visible restart notice. Catch helper errors and return false.
  - confidence: 0.92
