## Summary

Keep open: the PR targets a real source-reproducible Dashboard/Gateway recovery bug, but current main has not implemented the central hello-snapshot recovery, and the branch still needs deterministic run selection, restart-notice fallback preservation, a rebase over newer recovery code, and real behavior proof.

## What This Changes

This PR adds active chat-run metadata to Gateway hello snapshots, restores Dashboard chat busy state from reconnect snapshots, keeps abortable runs busy in the composer, preserves the last chat session in sidebar navigation, and writes interruption notices on gateway restart/shutdown.

## Best Possible Solution

Rebase or replace the branch on current main with deterministic active-run recovery and robust restart-notice fallback, then require redacted real Dashboard/Gateway reconnect/restart proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main clears chatRunId/chatStream on Gateway hello while SnapshotSchema has no activeChatRuns recovery field. I did not run a live browser/Gateway repro in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR must deterministically choose the recovered run, preserve fallback notices on injection errors, and rebase over current restart/session recovery before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server snapshots every run-id entry from `chatAbortControllers`. Returning the first matching array entry can recover an older run, so later events for the newer run fail the run-id guard and leave stale busy/stream state. Prefer the current host run if still present, otherwise choose the highest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve the restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, path resolution, file creation, or transcript append, but callers enqueue the fallback only when it returns `false`. Catch helper errors and return `false`; otherwise a consumed sentinel can lose the user-visible restart notice.
  - confidence: 0.92
