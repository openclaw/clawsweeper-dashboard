## Summary

Keep open: current main still lacks the dashboard reconnect recovery this PR targets, and the branch is useful, but it has two concrete correctness blockers before merge.

## What This Changes

The PR adds active chat run data to Gateway hello snapshots, restores dashboard chat busy state across reconnects, adjusts chat navigation/busy controls, and adds shutdown/restart interruption notices with protocol client and test updates.

## Best Possible Solution

Land a rebased implementation that keeps the additive activeChatRuns protocol contract, restores the active dashboard run deterministically, keeps controls busy while abort is available, and makes restart/shutdown notices durable without losing fallback delivery on transcript errors.

## Reproduction Assessment

Yes. Related issue #41064 gives focused dashboard steps, and current main source shows hello snapshots lack activeChatRuns while the Control UI hello handler clears chatRunId/chatStream after reconnect.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch should choose recovered runs deterministically and make restart-notice fallback exception-safe before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server tracks active runs by runId. Returning the first matching entry can recover an older run, causing later events for the newer run to fail run-id guards and leave stale busy or stream state; prefer the current host run if still active, otherwise choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, transcript path resolution, file creation, or append. Because callers only enqueue the system-event fallback when this helper returns `false`, an exception after the sentinel is consumed drops the restart notice; catch helper errors and return `false`.
  - confidence: 0.92
