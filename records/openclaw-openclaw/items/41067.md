## Summary

Keep open: the PR still targets a real current-main Dashboard reconnect bug, but it is not merge-ready because it lacks required real behavior proof and still has correctness issues in active-run selection and restart-notice fallback handling.

## What This Changes

This PR adds active chat-run metadata to Gateway hello snapshots, restores Dashboard chat busy state on reconnect/tab return, preserves the last active chat session, and writes interruption notices during restart/shutdown.

## Best Possible Solution

Rebase or replace the branch on current main with deterministic active-run hello/reconnect recovery, robust restart-notice fallback integrated with the current restart-sentinel queue, and redacted real Dashboard/Gateway proof.

## Reproduction Assessment

Yes, source-reproducible: current main clears chatRunId/chatStream on Gateway hello while the hello snapshot has no active-run recovery field. I did not run a live browser/Gateway timing repro in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the PR needs deterministic recovered-run selection, fallback-safe restart notice handling, rebase over current restart-sentinel architecture, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server snapshots runs by run id. Returning the first matching entry can recover an older run, causing later events for the newer run to fail the run-id guard and leave stale busy state; prefer the current host run if still present, otherwise choose the highest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve the restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, path resolution, file creation, or transcript append, but callers only enqueue the fallback when it returns `false`. Catch helper errors and return `false` so a consumed sentinel still produces a user-visible fallback notice.
  - confidence: 0.92
