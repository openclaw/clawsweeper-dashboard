No, not as currently proposed. The direction is maintainable, but the PR head needs deterministic recovered-run selection, exception-safe restart fallback behavior, a current-main rebase, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: activeChatRuns can contain multiple entries for the same session because the server snapshots every chatAbortControllers entry. Returning the first match can recover an older run ID, causing later deltas/finals for the newer run to fail the run-id guard and leave stale busy state; keep the current run when present, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Catch restart notice injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, path resolution, file creation, or transcript append setup. The fallback callers only enqueue the system event when this helper returns false, so a thrown error can consume the sentinel and drop the user-visible restart notice; catch helper errors and return false.
  - confidence: 0.92
