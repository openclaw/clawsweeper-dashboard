## Summary

Keep open: the PR targets a real current-main Dashboard/Gateway chat recovery bug that is not fully implemented on main, but the proposed patch still has two correctness blockers and lacks required real behavior proof before merge.

## What This Changes

The PR adds Gateway active chat-run snapshots, Dashboard reconnect busy-state recovery, chat-tab session preservation, and restart/shutdown interruption notices with focused Gateway/UI tests.

## Best Possible Solution

Rebase or replace the branch with deterministic active-run recovery and exception-safe restart-notice fallback, align it with current session-run/restart-sentinel code, then require redacted real Dashboard/Gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main clears chatRunId/chatStream on Gateway hello, lacks an activeChatRuns recovery field, and only partially recovers Stop through sessions.list while Queue/send still depend on local run-id state.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch must choose recovered runs deterministically, preserve fallback notices on injection errors, and be rebased over current recovery code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: activeChatRuns can contain multiple entries for the same session because the server snapshots every run-id entry from chatAbortControllers. Returning the first match can recover an older run, causing later events for the newer run to fail the run-id guard and leave stale busy/stream state; keep the current host run when present, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Preserve the restart-notice fallback on injection errors:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, path resolution, file creation, or transcript append, but callers only enqueue the fallback system event when it returns false. Catch helper errors and return false so a consumed sentinel does not lose the user-visible restart notice.
  - confidence: 0.92
