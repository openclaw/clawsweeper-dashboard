## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate restart abort cleanup to restart closes:** `src/gateway/server-close.ts:121-144`
  - body: This loop runs for every gateway close, even when `restartExpectedMs` is null. A normal SIGTERM/SIGINT shutdown would abort active chat runs and write a "gateway restart" interruption notice, while current main only performs restart-run drain/abort work inside the restart path.
  - confidence: 0.89
- **[P1] Filter active-run snapshots before sending hello:** `src/gateway/server/ws-connection/message-handler.ts:103-108`
  - body: `activeChatRuns` is built from every abort-controller entry. On current main that map can include hidden Control UI runs, aborted/non-projected entries, and agent-kind entries; exposing all of them lets the UI adopt a run whose chat events it should not receive.
  - confidence: 0.86
- **[P1] Fall back when transcript setup fails:** `src/gateway/server-restart-sentinel.ts:29`
  - body: `injectRestartNoticeIntoSession` awaits transcript setup before returning a boolean, but callers only enqueue the system-event fallback when it returns `false`. If path/file setup throws, the consumed restart sentinel notice is dropped instead of falling back.
  - confidence: 0.84
- **[P2] Pick the recovered run deterministically:** `ui/src/ui/app-gateway.ts:125-126`
  - body: `resolveActiveChatRunForSession` returns the first matching snapshot entry for a session. Because the server tracks runs by runId, duplicate active entries for the same session can occur; adopting an older run causes newer deltas/finals to be ignored by run-id checks.
  - confidence: 0.81
