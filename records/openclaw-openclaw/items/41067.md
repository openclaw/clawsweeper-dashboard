## Summary

Keep open. Current main still has the dashboard reconnect busy-state gap, so the PR is addressing a real bug, but the branch is not merge-ready because it can recover the wrong active run when more than one run exists for a session and can drop restart notices when transcript injection throws.

## What This Changes

The PR adds active chat run metadata to Gateway hello snapshots, updates Control UI reconnect/navigation/busy-state handling, and adds shutdown/restart interruption notices with generated protocol client and test updates.

## Best Possible Solution

Land a rebased implementation that keeps the additive `activeChatRuns` protocol/schema/generated-client contract, recovers the correct active run deterministically, keeps chat controls busy while abort is available, and integrates shutdown/restart notices with the current restart-sentinel and transcript append paths.

## Reproduction Assessment

Yes. Related issue #41064 gives focused dashboard steps, and current main source confirms hello snapshots do not expose active runs while the Control UI hello handler clears the busy latch after reconnect.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the branch needs deterministic active-run selection, exception-safe restart-notice fallback, and rebase work against current main before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered chat run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `activeChatRuns` can contain multiple entries for the same session because the server tracks active runs by runId. Returning the first matching entry can recover an older run; later events for the newer run then fail the run-id guard and leave stale busy or stream state. Prefer the existing host run if still active, otherwise choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Preserve fallback when transcript injection fails:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: `injectRestartNoticeIntoSession` can throw from session lookup, transcript path resolution, or file creation, but callers only enqueue the system-event fallback when this helper returns `false`. Once the sentinel is consumed, that throw drops the restart notice. Catch helper errors and return `false` so the fallback still runs.
  - confidence: 0.92
