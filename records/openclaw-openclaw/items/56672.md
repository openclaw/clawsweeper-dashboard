## Summary

Keep open: the branch is useful but not merge-ready because current main shows gateway delete already emits `session_end` while the PR manually emits it too, and the cron helper can still miss or duplicate terminal paths; the PR also lacks required real behavior proof.

## What This Changes

The PR adds `session_end` plugin hook emissions for failed subagent spawn cleanup and cron isolated-agent runs, plus runtime exports, hook JSDoc, and cron test-harness mocks.

## Best Possible Solution

Revise this into a single-owner lifecycle implementation: gateway deletion should own deleted subagent `session_end`, and cron should use one idempotent terminal finalizer covering success, skip/preflight, abort, and thrown outcomes exactly once, with focused tests and docs for the chosen hook contract.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR diff shows the failed-spawn cleanup path calls `sessions.delete` after the new manual emission, while cron prepared-failure and post-persist throw paths can still bypass or duplicate the added helper.

## Solution Assessment

No. The hook expansion is reasonable, but this patch should use the existing deletion lifecycle owner and an idempotent cron terminal path instead of ad hoc emissions across multiple exits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete`. On current main, `sessions.delete` always calls `emitGatewaySessionEndPluginHook` for deleted rows before using `emitLifecycleHooks` only for the unbound lifecycle event, so listeners can get duplicate `session_end` events; the manual payload also uses the idempotency key rather than the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767-783`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but `prepareCronRunContext` can assign `runSessionId` and `runSessionKey` before returning `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns that prepared result before any new emission, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalize throws:** `src/cron/isolated-agent/run.ts:1095`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
