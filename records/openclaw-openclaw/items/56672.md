## Summary

Keep open. The PR has useful intent for the still-open lifecycle hook requests, but it should not merge as-is: current main already emits `session_end` when failed subagent cleanup deletes the provisional session, while the proposed manual emission uses the run/idempotency id and can duplicate delivery; the cron addition also does not cover every terminal run path.

## What This Changes

The PR adds `session_end` hook emissions for failed subagent spawn cleanup and cron isolated-agent completion, updates plugin hook comments, and adds an Unreleased changelog entry.

## Best Possible Solution

Implement `session_end` as an exactly-once lifecycle signal from the owning lifecycle paths: rely on the existing session deletion emission for failed provisional subagent sessions, and add a cron terminal emission that covers success, abort, and executor failure after a run session exists and is persisted, with focused regression tests and updated hook docs/changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available from current main plus the PR diff: failed-spawn cleanup still reaches `sessions.delete`, which already emits `session_end`, while cron abort/error returns can bypass the proposed hook block.

## Solution Assessment

No. The proposed approach is not the best fix because it adds an ad hoc failed-spawn emission with the wrong id and duplicate delivery risk, and it does not centralize cron terminal emission across every already-started exit path.
