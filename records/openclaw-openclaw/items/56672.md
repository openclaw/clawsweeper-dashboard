## Summary

Keep open. The hook direction is useful, but the patch is not merge-ready because it duplicates failed-spawn `session_end` delivery and still misses or double-emits cron terminal paths.

## What This Changes

The PR adds `session_end` hook emissions to failed subagent spawn cleanup and isolated cron run completion paths, updates runtime barrel exports, hook type docs, test mocks, and the changelog.

## Best Possible Solution

Centralize `session_end` in owning lifecycle paths: let gateway session deletion own failed-spawn cleanup, and make cron emit exactly once for every allocated run session, including prepared failures, after the terminal outcome path is settled.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR manually emits failed-spawn `session_end` before `sessions.delete`, while current main already emits it there; cron prepare failures and post-finalize throws expose the missing and duplicate cron cases.

## Solution Assessment

No. The desired hook expansion is reasonable, but this implementation should use one-shot lifecycle ownership instead of adding ad hoc emissions around existing delete and cron finalization paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let session delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1193-1202`
  - body: This block runs when `emitLifecycleHooks=false` and then still calls `sessions.delete`. On current main, `sessions.delete` emits `session_end` for every deleted session before it honors `emitLifecycleHooks` for the separate unbound/subagent lifecycle event, so listeners can receive duplicate end events; the manual event also uses `childIdem` instead of the stored session id. Remove the manual emission or make one owner emit the hook with the real session entry id.
  - confidence: 0.95
- **[P2] Cover prepared cron failures before returning:** `src/cron/isolated-agent/run.ts:770-785`
  - body: `prepareCronRunContext` allocates a run session before model selection and provider preflight can return `ok:false`, but the new helper only accepts a `PreparedCronRunContext` and the caller returns those terminal results before any hook call. Those skipped/error cron runs still never fire `session_end`, so plugins waiting on the end signal can stall on failure paths.
  - confidence: 0.91
- **[P2] Keep cron session_end one-shot after delivery errors:** `src/cron/isolated-agent/run.ts:1098`
  - body: `finalizeCronRun` now emits `session_end` before delivery and result resolution finish, while the outer catch also emits it for the same prepared context. If delivery or later finalization throws after the first emission, one cron run sends two end events; track an emitted flag or move the hook into a single terminal finalization path.
  - confidence: 0.86
