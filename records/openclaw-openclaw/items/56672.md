## Summary

Keep open: the feature is still useful, but this branch has lifecycle correctness blockers and no after-fix real behavior proof from a plugin observing the new hook paths.

## What This Changes

The PR adds plugin `session_end` emissions for failed subagent spawn cleanup and cron isolated-agent terminal paths, exports hook helpers through runtime barrels, and updates hook JSDoc/test-harness mocks.

## Best Possible Solution

Use a single-owner lifecycle contract: gateway deletion owns deleted subagent `session_end`, and cron has one idempotent terminal finalizer covering success, prepared failure, abort, and thrown outcomes exactly once, with focused tests and real plugin proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of current main plus the PR diff shows the duplicate delete path, prepared cron failure gap, and cron double-emission risk. I did not run tests because this is a read-only review and local test commands would create artifacts.

## Solution Assessment

No. Extending the lifecycle hook is reasonable, but this patch adds ad hoc emissions across multiple owners instead of a one-shot lifecycle finalization contract.

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete` for the same child session.   - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767-783`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but preparation can assign `runSessionId` and `runSessionKey` before returning `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns that prepared result before this helper can run, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalization errors:** `src/cron/isolated-agent/run.ts:1095`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
