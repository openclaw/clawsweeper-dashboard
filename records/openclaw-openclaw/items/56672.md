## Summary

Keep open: the PR is aimed at a real lifecycle-hook gap and already incorporates useful review feedback, but the latest failed-spawn change still double-emits `session_end` on current main because `sessions.delete` emits that hook independently of `emitLifecycleHooks`. The branch also needs rebasing onto the current cron/hook-type layout before it can be safely merged.

## What This Changes

The PR adds `session_end` emissions for failed subagent spawn cleanup and cron isolated-agent termination, documents the trigger cases, and adds an Unreleased changelog entry.

## Best Possible Solution

Implement `session_end` as an exactly-once lifecycle signal from the owning paths: rely on or deliberately fix the `sessions.delete` contract for failed provisional subagent sessions, add cron terminal emission in the current prepare/execute/finalize flow after a persisted run session exists, and update hook docs/tests/changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with both `subagent_ended` and `session_end` hooks registered, the PR manually emits `session_end` when `emitLifecycleHooks` is false, then current-main `sessions.delete` emits `session_end` again for the same deleted child session.

## Solution Assessment

No. The direction is right, but this is not the best mergeable form because it relies on `emitLifecycleHooks` suppressing `session_end` when current main does not, and it needs to be rebased into the current cron/hook-type structure.
