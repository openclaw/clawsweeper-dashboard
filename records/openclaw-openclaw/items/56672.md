## Summary

Keep open: the PR targets a real plugin lifecycle gap, but the current diff still has source-proven hook ordering/idempotency bugs and lacks contributor real behavior proof, so it is not ready to merge or close.

## What This Changes

The PR adds `session_end` hook emissions for failed subagent spawns and cron isolated-agent run termination, plus runtime exports and hook JSDoc/test-harness mocks.

## Best Possible Solution

Keep the PR open and revise it so `session_end` is a one-shot lifecycle event: gateway deletion should own deleted subagent sessions, and cron should have one terminal finalizer that covers success, skip, error, abort, and thrown outcomes exactly once.

## Reproduction Assessment

Yes. Source inspection against current main plus the PR diff gives a high-confidence path: failed-spawn cleanup manually emits before `sessions.delete`, while cron prepared failures and post-finalize throws still bypass or duplicate the new cron emission.

## Solution Assessment

No. The requested hook expansion is reasonable, but this patch should use existing lifecycle owners and an idempotent cron terminal path instead of ad hoc emissions across multiple exits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false, then still calls `sessions.delete`. Current main's delete handler already emits `emitGatewaySessionEndPluginHook` for deleted rows before using `emitLifecycleHooks` for only the unbound lifecycle event, so listeners can receive duplicate `session_end` events; the manual payload also uses the idempotency key instead of the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:768-784`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but `prepareCronRunContext` can return `ok: false` after assigning `runSessionId`/`runSessionKey` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns those results before any new emission, so terminal skipped/error cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.9
- **[P2] Keep cron session_end one-shot after finalize throws:** `src/cron/isolated-agent/run.ts:1096`
  - body: `finalizeCronRun` now emits after persistence, but payload resolution and delivery work later in the same function can still throw. The outer catch also emits for the same prepared context, so a post-persist delivery/runtime error can send two `session_end` events for one cron run.
  - confidence: 0.86
