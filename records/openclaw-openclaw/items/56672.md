## Summary

Keep open: the hook direction is useful, but the current PR still has lifecycle correctness blockers around duplicate failed-spawn events and incomplete/duplicate cron terminal events.

## What This Changes

The PR adds `session_end` emissions for failed subagent starts and cron isolated runs, exports hook helpers through runtime barrels, updates the cron test harness, hook type docs, and changelog.

## Best Possible Solution

Make `session_end` a one-shot owner-lifecycle event: let gateway deletion own deleted subagent sessions, and add a single cron finalization path that covers every terminal run outcome exactly once.

## Reproduction Assessment

Yes. Source and diff inspection gives a high-confidence path: failed-spawn cleanup manually emits before `sessions.delete`, while cron prepared failures and post-finalize throws still bypass or duplicate the new cron emission.

## Solution Assessment

No. The requested hook expansion is reasonable, but the current implementation should be centralized in existing lifecycle owners and guarded so each terminal session emits `session_end` exactly once.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1197`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false, then still calls `sessions.delete`. On current main, `sessions.delete` emits `session_end` for any deleted row before it checks `emitLifecycleHooks`, so listeners get duplicate end events; the manual event also uses the idempotency key instead of the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767-783`
  - body: The helper only accepts `PreparedCronRunContext`, so all `!prepared.ok` returns still bypass `session_end`. `prepareCronRunContext` has already assigned `runSessionId`/`runSessionKey` before model-selection and provider-preflight failures, leaving terminal skipped/error runs without the lifecycle event this PR is meant to provide.
  - confidence: 0.9
- **[P2] Keep cron session_end one-shot after finalize errors:** `src/cron/isolated-agent/run.ts:878`
  - body: `finalizeCronRun` now emits `session_end` before later payload resolution and delivery work that can throw, while the outer catch also emits for the same prepared context. A delivery/runtime error after this line will therefore send two `session_end` events for one cron run.
  - confidence: 0.84
