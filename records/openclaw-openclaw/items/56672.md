## Summary

Keep open. Current main still lacks the requested cron and failed-subagent `session_end` coverage, and this PR is useful, but the latest diff is not ready because it can double-fire failed-spawn hooks, uses the run/idempotency seed as a session id, and still misses cron preparation exits.

## What This Changes

The PR adds `session_end` emissions for failed subagent starts and cron isolated-agent exits, exports hook helpers through runtime barrels, documents the hook, and adds a changelog entry.

## Best Possible Solution

Repair the PR so `session_end` is emitted exactly once by the owning lifecycle path with the real session id, including terminal cron preparation exits, and cover those paths with focused hook tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: failed forked subagent spawns can hit the new manual hook and then `sessions.delete`, while cron model/preflight failures return after creating a run session id but before any new `session_end` call.

## Solution Assessment

No. The feature direction is right, but the current patch should centralize subagent emission with `sessions.delete` semantics and add a cron preparation-exit path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Avoid double-firing failed-spawn session_end:** `src/agents/subagent-spawn.ts:1188-1190`
  - body: This manual call runs when `emitLifecycleHooks` is false, but `sessions.delete` still emits the gateway `session_end` hook for any deleted entry with a `sessionId`; `emitLifecycleHooks` only gates the subagent unbound cascade. A failed forked spawn already has a child session id, so plugins can receive two `session_end` events for the same ended session.
  - confidence: 0.94
- **[P2] Use the child session id in failed-spawn payloads:** `src/agents/subagent-spawn.ts:1197`
  - body: The payload passes `childIdem`, which is the agent RPC idempotency/run seed, as `sessionId`. When a child entry exists, the hook contract expects the stored transcript session id, so plugins correlating lifecycle events or loading session metadata cannot resolve this event reliably.
  - confidence: 0.88
- **[P2] Emit session_end for cron preparation exits:** `src/cron/isolated-agent/run.ts:1085`
  - body: The new helper is only called after `prepareCronRunContext` succeeds, but that function creates `runSessionId`/`runSessionKey` and then can return terminal error/skipped results on model selection or provider preflight failures. Those cron runs still end without `session_end`, leaving the lifecycle gap for failure paths that can block queue or memory cleanup.
  - confidence: 0.87
