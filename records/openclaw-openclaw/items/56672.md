## Summary

Keep open for normal PR review. The lifecycle direction is useful, but the latest branch is still dirty against main, lacks real behavior proof, and introduces source-visible session lifecycle edge cases around duplicate subagent end events and incomplete cron terminal coverage.

## What This Changes

The PR adds `session_end` emissions for failed subagent spawn cleanup and cron isolated-agent terminal paths, exports hook helpers through runtime barrels, and updates hook JSDoc/test-harness mocks.

## Best Possible Solution

Use a single-owner lifecycle finalizer: let gateway deletion own deleted subagent `session_end`, add one idempotent cron terminal helper covering success, abort, prepared failures, and thrown finalization paths exactly once, then prove it with focused tests and redacted plugin output.

## Reproduction Assessment

Yes for the PR findings by source inspection: current main plus the PR diff show `sessions.delete` already emits deleted `session_end`, prepared cron failures return before the new helper, and catch/finalize can both emit after post-persist failures. I did not run tests because this is a read-only review.

## Solution Assessment

No. Extending the lifecycle hook is a reasonable direction, but this patch adds ad hoc emissions across multiple owners instead of one idempotent lifecycle finalization contract.

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete` for the same child session. - **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:768-783`

  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalization errors:** `src/cron/isolated-agent/run.ts:1096`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
