## Summary

Keep open. The PR addresses a useful hook gap and the latest rebase narrows the earlier drift, but the current patch still has lifecycle correctness bugs: failed subagent cleanup can double-fire `session_end`, and cron preparation failures still bypass the new hook path.

## What This Changes

The PR adds `session_end` emissions for failed subagent starts and isolated cron completion paths, exports hook helpers through runtime barrels, updates the cron test harness, hook type docs, and changelog.

## Best Possible Solution

Centralize `session_end` emission in the owning lifecycle paths: let gateway session deletion handle provisional subagent cleanup, and make cron emit exactly once for every terminal isolated run session with focused regression coverage.

## Reproduction Assessment

Yes. Source and diff inspection gives a high-confidence reproduction: the PR adds a manual failed-spawn `session_end` before `sessions.delete`, while current `sessions.delete` already emits `session_end`, and cron preparation failures return after creating a run session but before any new helper call.

## Solution Assessment

No. The hook direction is good, but the implementation should avoid duplicating the existing delete lifecycle and should use one cron finalization path that covers every terminal prepared run exactly once.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let session delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1192-1197`
  - body: The new `if (!emitLifecycleHooks)` block manually emits `session_end` and then still calls `sessions.delete`. On current main, `sessions.delete` always emits `session_end` after deleting the row; `emitLifecycleHooks` only controls the later `subagent_ended` unbind hook. This gives listeners duplicate end events, and the manual event uses the idempotency/run seed instead of the stored session id. Remove the manual emission or make the cleanup path genuinely one-shot with the real session entry id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767`
  - body: The new helper is only called after `prepared.ok`, but `prepareCronRunContext` has already created `runSessionId`/`runSessionKey` before model-selection and provider-preflight failures return `ok: false`. Those terminal error/skipped cron runs still finish without `session_end`, so plugins can stall on exactly the failure paths this PR is meant to cover.
  - confidence: 0.9
- **[P2] Keep cron session_end one-shot after finalize errors:** `src/cron/isolated-agent/run.ts:878`
  - body: `finalizeCronRun` now emits `session_end` before delivery/result resolution completes, and the outer catch also emits it for the same prepared context. If delivery or later finalization throws after the first emission, the catch path sends a second `session_end` for one cron run. Track a one-shot emission or move it into a single finalization path.
  - confidence: 0.83
