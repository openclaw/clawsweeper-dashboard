## Summary

Keep open. The feature is still plausible, but this branch still has correctness blockers in the failed-subagent cleanup and cron terminal paths, and the external contributor has not provided real behavior proof showing a plugin observing the new hook.

## What This Changes

The PR adds `session_end` plugin hook emissions for failed subagent spawns and cron isolated-agent termination, with runtime exports, test-harness mocks, and hook JSDoc.

## Best Possible Solution

Revise this branch into a one-shot lifecycle implementation: gateway deletion should own deleted subagent sessions, and cron should use one idempotent terminal finalizer that covers success, skip, error, abort, and thrown outcomes exactly once.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR diff shows the failed-spawn cleanup path calls `sessions.delete` after the new manual emission, while cron preflight early returns and post-persist throw paths bypass or duplicate the added helper; I did not run tests because this was a read-only review.

## Solution Assessment

No. The requested hook expansion is reasonable, but this patch should use existing lifecycle owners and an idempotent cron terminal path instead of ad hoc emissions across multiple exits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete`. On current main, `sessions.delete` always calls `emitGatewaySessionEndPluginHook` for deleted rows before using `emitLifecycleHooks` only for the unbound lifecycle event, so a listener can get duplicate `session_end` events; the manual payload also passes the idempotency key rather than the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:768-784`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but `prepareCronRunContext` can assign `runSessionId`/`runSessionKey` and then return `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns `prepared.result` before any new emission, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.9
- **[P2] Keep cron session_end one-shot after finalize throws:** `src/cron/isolated-agent/run.ts:1096`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery work can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.86
