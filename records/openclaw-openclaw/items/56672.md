## Summary

Keep open: the goal is useful, but source inspection of the current PR head still shows duplicate or wrong-id failed-spawn emissions, cron prepared-failure gaps, and cron double-emission paths, and the external PR lacks real behavior proof.

## What This Changes

The PR adds plugin `session_end` emissions for failed subagent spawn cleanup and isolated cron-agent terminal paths, exports hook helpers through runtime barrels, updates hook JSDoc, and extends cron test-harness mocks.

## Best Possible Solution

Use a single-owner lifecycle design: gateway deletion should own deleted subagent `session_end`, and cron should use one idempotent terminal finalizer that covers success, prepared failure, abort, and thrown outcomes exactly once, with focused tests and real plugin proof.

## Reproduction Assessment

Yes for the PR review findings: current-main source plus the PR head show the duplicate delete path, prepared cron failure gap, and cron double-emission risk. I did not run live behavior because this is a read-only review and tests would create artifacts.

## Solution Assessment

No. Extending lifecycle signaling is reasonable, but this patch adds ad hoc emissions across multiple owners instead of a one-shot finalization contract for deletion and cron terminal paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1197-1202`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete` for the same child session. Current main's delete path already emits the plugin `session_end` for deleted rows before applying `emitLifecycleHooks` only to the unbound lifecycle event, so listeners can receive duplicate end events; the manual payload also uses the idempotency key instead of the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:771`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but preparation can assign `runSessionId` and `runSessionKey` before returning `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns that prepared result before this helper can run, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalization errors:** `src/cron/isolated-agent/run.ts:1099`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
