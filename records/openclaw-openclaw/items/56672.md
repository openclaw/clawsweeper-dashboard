## Summary

Keep open. The latest PR shape is useful, but the failed-spawn subagent addition still introduces a correctness regression: it manually emits `session_end` even though current `sessions.delete` already emits the plugin hook for deleted sessions, and it uses the run idempotency key instead of the stored session id.

## What This Changes

Adds `session_end` hook emission for failed subagent spawn cleanup and cron isolated-agent termination, exports hook helpers through runtime barrels, documents hook timing, and adds a changelog entry.

## Best Possible Solution

Keep the cron hook direction, but make subagent failed-spawn cleanup emit `session_end` exactly once with the stored session id and cover both lifecycle paths with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR diff plus current main: the new block emits `session_end` before `sessions.delete`, while current `sessions.delete` already emits the plugin hook using the stored session entry id.

## Solution Assessment

No. The lifecycle signal is the right product direction, but this patch should delegate failed-spawn deletion to the existing gateway hook path or share a helper that reads the real session entry id instead of sending a second event.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Let session deletion own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1191-1196`
  - body: `sessions.delete` already emits the plugin `session_end` hook whenever it deletes a stored entry, even when `emitLifecycleHooks` is false. This added block fires a second event before deletion and uses `childIdem` as `sessionId`, while the provisional session entry has its own generated session id, so hook consumers can receive one uncorrelatable end event plus the real delete event.
