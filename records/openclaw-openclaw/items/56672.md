## Summary

Keep open. This is a plausible lifecycle-hook implementation candidate, but the current patch is not merge-ready because source review shows `session_end` can be duplicated, missed, or emitted twice in edge paths, and the external PR still has no real behavior proof from a plugin observing the new hook.

## What This Changes

The PR adds typed plugin `session_end` emissions for failed subagent spawn cleanup and cron isolated-agent terminal paths, plus runtime exports, hook JSDoc, and test-harness mocks.

## Best Possible Solution

Use a single-owner lifecycle contract: `sessions.delete` owns deleted subagent `session_end`, and cron emits through one idempotent terminal helper covering success, prepared failure, abort, and thrown outcomes exactly once, with focused tests and real plugin proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of current main plus the PR diff shows the duplicate delete path, prepared cron failure gap, and cron double-emission risk. I did not run tests because this is a read-only review and local test commands would create artifacts.

## Solution Assessment

No. Extending lifecycle coverage is reasonable, but this patch adds ad hoc emissions across multiple owners instead of one idempotent terminal session lifecycle contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1190-1196`
  - body: This new block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete` for the same child session. Current main's delete path already emits the plugin `session_end` for deleted rows before applying `emitLifecycleHooks` only to the unbound lifecycle event, so listeners can receive duplicate end events; the manual payload also uses the spawn idempotency key instead of the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767-783`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but preparation can assign `runSessionId` and `runSessionKey` before returning `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns that prepared result before this helper can run, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalization errors:** `src/cron/isolated-agent/run.ts:1095`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
