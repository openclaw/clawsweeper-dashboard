## Summary

Keep open. The PR targets a real lifecycle gap, but the current head still has blocking hook ownership bugs and lacks required real behavior proof from a plugin observing the new hook.

## What This Changes

The PR adds plugin `session_end` hook emissions for failed subagent spawn cleanup and cron isolated-agent runs, exports hook helpers through runtime barrels, updates hook JSDoc, and adds cron test-harness mocks.

## Best Possible Solution

Use a single-owner lifecycle design: gateway deletion should own deleted subagent `session_end`, and cron should use one idempotent terminal finalizer covering success, prepared failure, abort, and thrown outcomes exactly once, with focused tests and real plugin proof.

## Reproduction Assessment

Yes for the PR review findings. Source inspection of current main plus the PR diff shows the duplicate delete path and cron terminal-path gaps; I did not run live behavior because this was a read-only review.

## Solution Assessment

No. Expanding the hook is reasonable, but this patch should not add ad hoc emissions across multiple owners when existing delete and cron terminal paths need one-shot ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Let sessions.delete own failed-spawn session_end:** `src/agents/subagent-spawn.ts:1187-1204`
  - body: This block emits `session_end` when `emitLifecycleHooks` is false and then still calls `sessions.delete`. Current main's `sessions.delete` already emits the plugin `session_end` for deleted rows before applying `emitLifecycleHooks` only to the unbound lifecycle event, so listeners can receive duplicate end events; the manual payload also uses the idempotency key instead of the stored session id.
  - confidence: 0.94
- **[P2] Emit cron session_end for prepared failures:** `src/cron/isolated-agent/run.ts:767-783`
  - body: `emitCronSessionEnd` only accepts `PreparedCronRunContext`, but `prepareCronRunContext` can assign `runSessionId` and `runSessionKey` before returning `ok: false` for model-selection or provider-preflight failures. `runCronIsolatedAgentTurn` returns that prepared result before any new emission, so those terminal cron runs still miss the hook this PR is meant to provide.
  - confidence: 0.88
- **[P2] Keep cron session_end one-shot after finalize failures:** `src/cron/isolated-agent/run.ts:1095`
  - body: `finalizeCronRun` emits immediately after persistence, but later payload resolution and delivery dispatch can still throw. The outer catch also emits for the same prepared context, so a post-persist failure can deliver two `session_end` events for one cron run.
  - confidence: 0.9
