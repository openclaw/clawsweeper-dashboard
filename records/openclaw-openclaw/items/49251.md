## Summary

Keep open: current main has provider cooldown, fallback, quota-suspension, active-run steering, and in-process command queues, but it still does not provide a durable, user-visible pending-prompt queue that replays API-limit-blocked prompts after capacity returns or after gateway restart.

## What This Changes

The issue requests queuing prompts that cannot be handled immediately because provider API limits exhaust the current and fallback models, with user-visible waiting state and automatic resume.

## Best Possible Solution

Design a core-owned, SQLite-backed pending prompt/retry state that composes with model fallback, quota suspension, session visibility, and restart recovery; do not add JSONL sidecar queue files.

## Reproduction Assessment

Yes from source at medium confidence: the current active-session steering queue, command lane queue, and quota suspension state do not persist a blocked prompt payload for replay across gateway restart. I did not run a live provider-limit reproduction.

## Solution Assessment

No: the JSONL queue proposed in discussion conflicts with OpenClaw's SQLite-first runtime state policy. The safer solution is a core-owned persisted queue/retry design integrated with session suspension and model fallback.
