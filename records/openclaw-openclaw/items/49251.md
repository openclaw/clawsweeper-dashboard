## Summary

Keep open: current main has in-process followup queues plus provider cooldown/session suspension, but it does not yet provide a durable capacity-wait prompt queue with visible waiting state and automatic replay; the related PRs are open partial slices.

## What This Changes

Request to preserve prompts that cannot run because all configured models are API-limited, show them as waiting, and resume them automatically when capacity returns.

## Best Possible Solution

Keep this as the canonical design tracker for a SQLite-backed capacity-wait queue integrated with provider cooldowns, user-visible status, and exactly-once replay.

## Reproduction Assessment

No live provider-limit reproduction was run. Source inspection does show the missing behavior: followup queues are process-local and rate-limit exhaustion returns a cooldown message instead of a durable queued prompt.

## Solution Assessment

Unclear pending maintainer design. A durable queue is the right direction, but the discussion's JSONL sidecar sketch conflicts with the repo's SQLite-first runtime-state policy; the best path is a maintained SQLite-backed capacity-wait design.
