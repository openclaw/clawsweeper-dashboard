## Summary

Keep open. Current main has in-process command/follow-up queues, model fallback, cooldowns, and quota-suspension metadata, but it still does not persist and automatically replay the original API-limit-blocked prompt after restart or capacity recovery.

## What This Changes

Request durable queuing, visible waiting status, and automatic resume for prompts that cannot be handled immediately because the current and fallback models are API-limited.

## Best Possible Solution

Extend the session suspension, follow-up queue, and session store paths with a maintainer-approved durable suspended-prompt record, visible waiting state, replay scheduler, idempotency guard, and sensitive-payload retention policy.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks a durable capacity-blocked prompt payload and replay path.

## Solution Assessment

Unclear as a standalone JSONL queue. The safer direction is to reuse the existing session store, quotaSuspension, follow-up queue, and durable delivery contracts with explicit replay and privacy policy.
