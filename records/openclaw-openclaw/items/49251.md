## Summary

Keep open. Current main has in-process command/followup queues plus quota-suspension and lane auto-resume primitives, but it does not yet durable-store and automatically retry the prompt that hit provider/model capacity exhaustion; related issue https://github.com/openclaw/openclaw/issues/62540 was already closed in favor of this tracker, while https://github.com/openclaw/openclaw/issues/75648 covers a narrower Retry-After/error-metadata gap.

## What This Changes

Request durable, visible queueing and automatic resumption for prompts that cannot run because all provider/model options are rate-limited.

## Best Possible Solution

Extend the existing session suspension and queue contracts with an owner-reviewed durable suspended-prompt record that can safely replay the blocked work, show waiting status, and resume when capacity returns.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has in-process queues and quota-suspension state, but not a durable retry record for the triggering capacity-blocked prompt.

## Solution Assessment

No. The JSONL sketch in the discussion is useful context, but the maintainable fix should build on the existing session store, quotaSuspension, command-lane, and followup-queue contracts with explicit persistence and replay policy.
