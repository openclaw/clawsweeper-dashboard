## Label Justifications

- `P2`: This is a meaningful reliability improvement for capacity-blocked prompts, but it is feature-level design work rather than an urgent current regression.
- `impact:message-loss`: The reported failure mode is a user prompt appearing sent while no model receives or completes it after capacity exhaustion or restart.
- `impact:session-state`: The missing durable queued-prompt record is session/runtime state that can disappear when held only in process memory.
- `impact:auth-provider`: The trigger is provider/model capacity exhaustion across fallback and auth-profile cooldown paths.

## Summary

Keep open. Current main has in-process command/follow-up queues, persisted quota-suspension metadata, and a durable restart-continuation queue, but it still does not persist the original capacity-blocked prompt payload or replay it automatically after provider capacity returns; https://github.com/openclaw/openclaw/issues/62540 was closed in favor of this tracker, while https://github.com/openclaw/openclaw/issues/75648 remains a narrower Retry-After/error-metadata follow-up.

## What This Changes

The issue requests durable queuing, user-visible waiting state, and automatic replay for prompts that cannot run immediately because all provider/model options are rate-limited or exhausted.

## Best Possible Solution

Extend the existing session suspension, follow-up queue, and durable session-delivery contracts with an owner-reviewed suspended-prompt record, replay scheduler, visible waiting state, and sensitive-payload handling; keep Retry-After propagation scoped to https://github.com/openclaw/openclaw/issues/75648.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has in-process queues and quota-suspension metadata, but not a durable prompt replay record for the capacity-blocked turn.

## Solution Assessment

Unclear as a standalone JSONL queue. The maintainable path should reuse the session store, quotaSuspension, command-lane, follow-up queue, and durable session-delivery contracts with explicit persistence, replay, idempotency, and sensitive-payload policy.
