## Summary

Keep open: current main has rate-limit cooldowns, fallback summaries, and quota suspension metadata, but it still does not provide a durable, user-visible pending-prompt queue that can replay work after provider capacity returns or the gateway restarts.

## What This Changes

The issue requests preserving prompts that cannot run because all provider/model options are API-limited, showing them as waiting, and resuming them automatically when capacity is available.

## Best Possible Solution

Add one core-owned pending-prompt/run persistence contract that uses canonical session storage, coordinates with auth cooldown/quota suspension, exposes waiting status, and replays safely after restart without sidecar JSONL transcript writes.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the current queue UI/runtime state is in memory while fallback exhaustion persists suspension metadata rather than a replayable prompt payload.

## Solution Assessment

No, the JSONL queue sketch from discussion is not the best repository-shaped solution. The safer path is canonical session/SQLite-backed pending-run state that respects existing fallback, auth cooldown, and transcript append contracts.
