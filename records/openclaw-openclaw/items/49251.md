## Summary

Keep open. Current main still handles all-model API-limit or cooldown exhaustion by rotating profiles/models and then returning visible retry-later failure text, while the existing queues are either in-process serialization or narrower restart-delivery storage. The related duplicate #62540 was closed to consolidate the same remaining pause/retry behavior here, so this issue remains the canonical tracker.

## What This Changes

The issue requests preserving prompts blocked by provider/API limits as visible queued work and automatically resuming them once model capacity returns.

## Best Possible Solution

Add a core durable pending-turn queue for provider capacity/API-limit exhaustion that records the original prompt, session/route, provider/model/fallback state, retry_at, retry count, and idempotency key; exposes a waiting state to users; and replays after cooldown or restart without duplicating final replies. The implementation should coordinate with the existing session-delivery queue or intentionally generalize it rather than adding an isolated ad hoc JSONL path.

## Reproduction Assessment

Yes. A source-level reproduction is all configured model candidates or auth profiles being rate-limited/cooldowned: current main walks fallback/cooldown handling, returns final rate-limit text, and marks the reply operation failed instead of persisting a waiting prompt.

## Solution Assessment

Yes on the product direction, but the exact implementation is not settled. The JSONL sketch from the discussion should be translated into OpenClaw's core state/recovery conventions and reconciled with the open pending-inbound/followup persistence PRs.
