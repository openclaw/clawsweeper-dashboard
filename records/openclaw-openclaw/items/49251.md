## Summary

Keep open. Current main has in-process command/chat queues plus model retry/fallback/cooldown handling, but it does not implement the requested durable, user-visible queue that automatically resumes prompts when all model capacity is unavailable.

## What This Changes

Request durable queueing, visibility, and automatic later processing for prompts that cannot be answered immediately because provider/API limits exhaust the current and fallback models.

## Best Possible Solution

Define a durable pending-run queue with idempotent replay, retry metadata, visible user status, and clear integration with existing session lanes, model fallback/cooldowns, restart recovery, and #75648’s Retry-After/timeout work.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/design request. Source inspection shows current main has in-process queues and failover/error surfacing, but no durable capacity-blocked prompt queue contract to reproduce against.

## Solution Assessment

Unclear until maintainers choose the durable queue semantics. A JSONL per-session queue is plausible, but the best implementation needs decisions on storage format, replay idempotency, retry timing, UI status, and overlap with provider Retry-After handling.
