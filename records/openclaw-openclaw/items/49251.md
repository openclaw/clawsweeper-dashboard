## Summary

Keep open. Current main still has process-local command/follow-up queues, provider/model fallback, and persisted quota-suspension metadata, but it does not persist and automatically replay the original capacity-blocked user prompt requested here.

## What This Changes

The issue requests durable queuing, user-visible waiting state, and automatic replay for prompts that cannot be answered immediately because all configured model/provider capacity is exhausted.

## Best Possible Solution

Extend the existing session suspension, follow-up queue, and durable delivery contracts with a maintainer-approved suspended-prompt record, replay scheduler, visible waiting state, and sensitive-payload policy.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks a durable capacity-blocked prompt payload and replay path.

## Solution Assessment

Unclear as the standalone JSONL queue suggested in discussion. The safer direction is to reuse the existing session store, quotaSuspension, follow-up queue, and durable delivery contracts with explicit replay and privacy policy.
