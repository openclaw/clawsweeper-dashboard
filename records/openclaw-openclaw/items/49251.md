## Summary

Keep open. Current main has in-process active-run/followup queueing, model failover/cooldowns, and limited restart recovery for interrupted transcript-locked sessions, but it does not durably queue a prompt and automatically resume it when all model capacity is rate-limited; #75648 is narrower and #62540 is closed.

## What This Changes

The issue requests durable, visible queuing and automatic resumption of prompts that cannot be handled immediately because all available models are API-limited.

## Best Possible Solution

Design a core-owned durable pending-turn queue with status visibility and bounded scheduled retry on provider rate-limit exhaustion, reusing existing session/run metadata and preserving idempotency across restarts.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current queue/failover behavior, but there is no existing documented contract that promises durable rate-limit prompt resumption.

## Solution Assessment

No. Current main’s failover/cooldown path and in-process followup queue are useful building blocks, but they are not the requested durable, visible, automatically resumed pending-prompt queue.
