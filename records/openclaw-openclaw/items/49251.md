## Summary

Keep open: current main has active-run in-memory queues plus provider retry/cooldown/fallback mitigation, but it still does not provide a durable, user-visible pending-prompt queue that preserves API-limit-blocked work and replays it after capacity returns.

## What This Changes

The issue requests durable, user-visible queueing and automatic resumption for prompts blocked by API/provider limits.

## Best Possible Solution

Define one core SQLite/session-backed pending-run contract that records blocked prompts or steps, surfaces waiting status, and resumes through the existing retry/fallback/wake paths, carrying over the concrete 429/Retry-After criteria from the related closed and open threads.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for durable capacity-wait queueing. Source inspection gives high confidence that current main has only process-local active-run queues plus cooldown/fallback handling, not the requested durable replay queue.

## Solution Assessment

Unclear as an implementation until maintainers choose the contract. The requested outcome is reasonable, but the JSONL sidecar suggested in discussion is not the best shape under the repository's SQLite-first state policy.
