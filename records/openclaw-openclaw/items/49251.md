## Summary

Keep open: current main and the latest release still have in-process prompt/follow-up queues, model fallback/cooldown summaries, and quota-suspension metadata, but not a durable, user-visible, replayable prompt queue for API-limit-blocked work. The related cooldown/circuit-breaker PRs are open and not safe canonical replacements for this broader prompt-preservation request.

## What This Changes

The issue requests preserving prompts that cannot run because all available models are API-limited, showing them as waiting, and resuming them automatically when capacity returns.

## Best Possible Solution

Add one core-owned, SQLite/session-backed pending-run contract that records waiting status and replay metadata, coordinates with auth cooldowns and quota suspension, and resumes exactly once after restart or capacity recovery.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the current prompt queues are process-local and quota suspension persists provider/model metadata rather than replayable prompt payloads.

## Solution Assessment

No, the sidecar JSONL queue sketch from discussion is not the best repo-shaped solution. The safer path is canonical SQLite/session-backed pending-run state that respects existing fallback, auth cooldown, quota suspension, and transcript append contracts.
