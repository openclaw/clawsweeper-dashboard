## Label Justifications

- `P2`: This is a meaningful reliability improvement for capacity-blocked prompts, but it is feature-level design work rather than an urgent confirmed regression.
- `impact:message-loss`: The reported failure mode is a prompt appearing sent while no model ultimately receives or completes it after capacity exhaustion.

- `impact:auth-provider`: The trigger is provider/model capacity exhaustion across fallback and auth-profile cooldown paths.

## Summary

Keep open. Current main has in-process queues, provider/model fallback, persisted quota-suspension metadata, and durable final-reply delivery retry, but it still does not persist and automatically replay the original capacity-blocked user prompt requested here.

## What This Changes

The issue requests preserving API-limit-blocked prompts in a visible queue and resuming them automatically when provider capacity returns.

## Best Possible Solution

Extend the existing session suspension, follow-up queue, and durable session-delivery contracts with an owner-reviewed suspended-prompt record, replay scheduler, visible waiting state, and sensitive-payload handling.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has in-process queues and quota-suspension metadata, but not a durable capacity-blocked prompt replay record.

## Solution Assessment

Unclear as the standalone JSONL queue suggested in discussion. The maintainable path should reuse the existing session store, quotaSuspension, command-lane, follow-up queue, and durable delivery contracts with explicit replay and privacy policy.

## Risks / Open Questions

- A durable prompt queue would persist private prompt and channel context, so retention, redaction, permissions, and cleanup policy need explicit owner review.
- Replay must be route-aware and idempotent to avoid duplicate prompts, stale session context, or wrong-channel delivery after restart or capacity recovery.
- Open PRs such as https://github.com/openclaw/openclaw/pull/64127, https://github.com/openclaw/openclaw/pull/81046, and https://github.com/openclaw/openclaw/pull/81054 cover adjacent cooldown/circuit-breaker pieces but do not replace this durable prompt-payload queue design.
