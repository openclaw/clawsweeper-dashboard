## Summary

Keep open. Current main has visible in-memory follow-up queues and provider rate-limit cooldown/fallback handling, but it does not implement the requested durable capacity-waiting prompt queue with automatic replay; related #75648 covers timeout/429 surfacing and does not replace this broader queue/recovery request.

## What This Changes

Request to preserve prompts that cannot run because all configured models are API-limited, show them as queued, and automatically retry when capacity returns.

## Best Possible Solution

Define a first-class durable inbound/session-run queue for capacity-limited prompts, reusing existing cooldown metadata and session delivery primitives only where their contracts fit, then expose queue status in gateway/UI/channel surfaces with regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because the requested durable capacity queue is not an existing documented contract. Source inspection does show current queues are either in-memory follow-up queues or restart/outbound delivery queues, not the requested provider-limit replay mechanism.

## Solution Assessment

Unclear until maintainers define the product contract. The direction is reasonable, but the maintainable solution should be a designed durable run/prompt queue rather than stretching the UI follow-up queue or outbound delivery queue beyond their current scopes.
