## Summary

Keep open: current main has model cooldown, quota-suspension, and in-process queue paths, but source inspection does not show a durable, user-visible pending-prompt queue that preserves prompts blocked by provider capacity and replays them after capacity returns or after gateway restart.

## What This Changes

The issue requests preserving prompts in a visible queue when API limits prevent any current or fallback model from responding, then automatically resuming them later.

## Best Possible Solution

Define one SQLite-backed pending prompt/run queue contract that coordinates with model fallback, auth cooldowns, quota suspension, gateway restart recovery, and user-visible status before implementing replay.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection shows current main persists cooldown and suspension metadata plus process-local queues, but not the replayable blocked prompt payload requested here.

## Solution Assessment

No for the JSONL sketch in the comment: the maintainable direction is a canonical SQLite-backed queue/replay contract aligned with session recovery and fallback state, not a new sidecar file format.
