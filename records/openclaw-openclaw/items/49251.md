## Summary

Keep open: current main has provider retry caps, auth/profile cooldowns, quota suspension, and process-local prompt queues, but it still does not provide a durable, user-visible pending-prompt queue that preserves API-limit-blocked work and replays it after capacity returns.

## What This Changes

Request durable queueing, visible waiting status, and automatic resume for prompts that cannot be served because all configured models or fallbacks are API-limited.

## Best Possible Solution

Define a single SQLite-backed capacity-wait/run replay contract that preserves prompt payloads securely, surfaces waiting status, and resumes without duplicate replies.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection on current main shows the existing cooldown, suspension, and queue paths are not a durable replayable prompt queue.

## Solution Assessment

Yes for the product direction, but no for the JSONL sidecar sketch in the discussion. The maintainable implementation should use the repository's canonical SQLite-backed state model and integrate with fallback, cooldown, restart recovery, and user-visible status.
