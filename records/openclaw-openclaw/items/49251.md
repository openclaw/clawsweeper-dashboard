## Summary

Keep open: current main and the latest release have in-process followup queueing, bounded same-model rate-limit retry, auth-profile cooldowns, and quota-suspension metadata, but the requested durable visible queue and automatic capacity-resume behavior remains missing and product-sensitive.

## What This Changes

The issue requests preserving API-limit-blocked prompts as visible queued work and resuming them automatically once provider capacity is available.

## Best Possible Solution

Keep this as the canonical capacity-wait/resume design issue, land or replace narrower SQLite-backed queue and cooldown pieces as they become ready, and define one user-visible retry/resume contract across provider cooldowns, model fallback, and session state.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection gives medium confidence that current main has cooldown, short retry, suspension, and in-memory queue pieces, but no durable visible queue that stores and later resumes the original prompt after all models are capacity-blocked.

## Solution Assessment

Unclear as an implementation shape. The issue direction is useful, but the JSONL-style suggestion in discussion is not the best final shape under current policy; the durable state should be SQLite-backed and integrated with provider cooldown/session recovery rather than added as a parallel sidecar.
