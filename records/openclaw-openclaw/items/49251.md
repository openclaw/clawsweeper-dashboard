## Summary

Keep open: current main has auth-profile cooldowns, quota-suspension metadata, restart/final-delivery recovery, and active-run steering queues, but it still does not provide a durable, user-visible provider-capacity prompt queue that stores the blocked prompt and replays it once capacity returns.

## What This Changes

The issue requests preserving prompts in a visible queue and automatically resuming them when provider/API capacity limits prevent any configured model from responding.

## Best Possible Solution

Define one SQLite-backed provider-capacity wait contract that stores pending prompts securely, surfaces waiting status, resumes once capacity returns, and prevents duplicate final replies.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection on current main shows existing cooldown, suspension, and queue paths are not a durable replayable prompt queue.

## Solution Assessment

Yes for the requested product direction, but no for the JSONL sidecar sketch in the discussion. The maintainable implementation should use OpenClaw's canonical SQLite-backed state model and integrate with fallback, cooldown, restart recovery, and user-visible status.
