## Summary

Keep open: current main still has rate-limit cooldowns, lane suspension metadata, in-process active-run queues, and partial queued-delivery recovery, but it does not yet provide a durable, user-visible provider-capacity prompt queue that stores the blocked prompt and replays it once capacity returns.

## What This Changes

Request durable queuing, visible waiting state, and automatic resume for prompts that cannot be handled because all configured model candidates are API-limited.

## Best Possible Solution

Define one SQLite-backed provider-capacity wait/replay contract that integrates model fallback, cooldown scope, follow-up queue persistence, lane suspension, user-visible waiting status, and exactly-once final delivery.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection does show that current main has cooldown, suspension, and in-memory queue paths but no durable replayable provider-capacity prompt queue.

## Solution Assessment

Unclear as an implementation shape. The requested behavior is valuable, but the durable design should use OpenClaw's SQLite-backed state model and reconcile overlapping fallback, cooldown, restart-recovery, and follow-up queue work.
