## Summary

Keep open: current main still has process-local followup queuing plus provider cooldown/suspension visibility, but no durable capacity-wait queue that preserves API-limit-blocked prompts, shows them as waiting, and replays them when provider capacity returns.

## What This Changes

The issue requests durable queuing, user-visible waiting status, and automatic replay for prompts that cannot run immediately because provider API limits exhaust all available model options.

## Best Possible Solution

Keep this as the canonical maintainer design item until a SQLite-backed capacity-wait queue preserves pending prompts, exposes waiting status, and replays them when provider/model capacity returns.
