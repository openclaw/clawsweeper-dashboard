## Summary

Keep this open: current main has in-process follow-up queues, provider retry-hint parsing, auth-profile cooldowns, and model fallback, but it does not yet preserve a capacity-blocked prompt as a durable visible queue item and automatically resume it when provider capacity returns. A closed duplicate already points back to this issue as the canonical tracker, while the related open PRs are conflicting or blocked on proof/design concerns.

## What This Changes

Request to preserve prompts in a visible queue and automatically resume them when API limits prevent the current and fallback models from responding immediately.

## Best Possible Solution

Use this as the canonical design thread for a durable, visible capacity-wait queue backed by canonical runtime state, integrated with provider cooldown metadata and session wake scheduling.

## Reproduction Assessment

Partly. Current source and docs show rate-limit cooldown/fallback behavior and in-process follow-up queues, but no durable capacity-wait prompt queue or automatic resume path; I did not run a live API-limit reproduction.

## Solution Assessment

No. The JSONL durable queue suggested in discussion is not the best fit for current repository policy; the better path is a canonical SQLite/session scheduler design that preserves provider cooldown and fallback recovery semantics.
