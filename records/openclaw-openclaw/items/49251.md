## Summary

Keep open: current main has process-local followup queues, provider cooldown/fallback handling, and separate restart-continuation delivery recovery, but it does not durably preserve a prompt blocked by provider capacity as a visible waiting item and replay it when capacity returns. The related PRs are still open and only cover slices of the problem.

## What This Changes

Request durable, user-visible queueing and automatic replay for prompts that cannot run immediately because all configured provider/model options are rate-limited or exhausted.

## Best Possible Solution

Keep this as the canonical maintainer design item and converge on a SQLite-backed durable waiting/retry model that connects provider cooldowns, visible queue status, and exact-once prompt replay.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The current-main source path is clear enough to show the missing behavior: provider/fallback exhaustion records cooldown or surfaces an error, while followup queues remain process-local and are not durable capacity-wait replay items.

## Solution Assessment

Unclear as an implementation plan: the desired behavior is aligned, but the JSONL sidecar idea in the discussion conflicts with the repo's database-first state policy. A shared SQLite queue integrated with provider cooldown and visible status surfaces is the safer direction.
