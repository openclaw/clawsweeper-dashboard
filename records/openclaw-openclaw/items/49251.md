## Summary

Keep open. Current main has in-process command/follow-up queues plus persisted quota-suspension metadata, but it still lacks a durable queued-prompt record and automatic replay for the prompt that exhausts all provider/model capacity; related issue https://github.com/openclaw/openclaw/issues/62540 was closed in favor of this tracker, while https://github.com/openclaw/openclaw/issues/75648 remains a narrower Retry-After/error-metadata follow-up.

## What This Changes

The issue requests that prompts blocked by provider/API capacity limits remain queued, visible to the user, and automatically resumed once capacity is available.

## Best Possible Solution

Extend the existing session suspension, command-lane, and follow-up queue contracts with an owner-reviewed durable suspended-prompt record, recovery/replay policy, visible waiting state, and sensitive-payload handling.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has in-process queues and quota-suspension state, but not a durable retry record for the triggering capacity-blocked prompt.

## Solution Assessment

No. The JSONL sketch in the discussion is useful context, but the maintainable fix should build on the existing session store, quotaSuspension, command-lane, and follow-up queue contracts with explicit persistence and replay policy.
