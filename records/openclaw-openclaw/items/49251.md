## Summary

Keep open. Current main has in-process command and follow-up queues plus persisted quota-suspension metadata, but it still does not persist the capacity-blocked prompt payload or replay it automatically after restart/capacity recovery; related https://github.com/openclaw/openclaw/issues/62540 was closed in favor of this tracker, while https://github.com/openclaw/openclaw/issues/75648 remains a narrower Retry-After/error-metadata follow-up.

## What This Changes

Request durable, visible queuing and automatic resumption for prompts that cannot be answered immediately because all provider/model capacity is exhausted.

## Best Possible Solution

Extend the existing session suspension, command-lane, and follow-up queue contracts with an owner-reviewed durable suspended-prompt record, recovery/replay policy, visible waiting state, and sensitive-payload handling.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has in-process queues and quota-suspension metadata, but not a durable prompt replay record for the capacity-blocked turn.

## Solution Assessment

Unclear as a one-off JSONL queue. The maintainable path should reuse the session store, quotaSuspension, command-lane, and follow-up queue contracts with explicit persistence, replay, and sensitive-payload policy.
