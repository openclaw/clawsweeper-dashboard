## Summary

Keep open: current main still only has process-local followup queues and provider cooldown/session suspension state; it returns a rate-limit/cooldown failure message when all fallback models fail, but it does not durably preserve the blocked prompt, expose it as waiting, and replay it when capacity returns. The related open PRs cover restart persistence, cooldown persistence, or quota classification slices, but they are partial or conflicting and do not replace this canonical capacity-wait design item.

## What This Changes

The issue requests preserving prompts that cannot run because all models are API-limited, showing them as waiting, and resuming them automatically once capacity returns.

## Best Possible Solution

Keep this as the canonical design tracker for a SQLite-backed capacity-wait queue that stores blocked prompt payloads, exposes waiting status, and wakes/replays the prompt once provider or model capacity returns.

## Reproduction Assessment

Source-reproducible: current main shows process-local followup queues plus provider cooldown/session suspension state, but no durable API-limit prompt replay path. I did not execute a live provider-limit failure during this read-only review.
