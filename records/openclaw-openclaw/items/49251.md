## Summary

Keep open: current main has short-window rate-limit retry, model/profile fallback, cooldown probing, and quota-suspension metadata, but it still does not provide a durable, user-visible provider-capacity prompt queue that stores the blocked prompt and replays it once capacity returns.

## What This Changes

The issue asks OpenClaw to queue prompts when all configured models are blocked by API limits, show the waiting state, and resume automatically when capacity returns.

## Best Possible Solution

Define one SQLite-backed provider-capacity wait/replay contract that stores pending prompts securely, surfaces waiting status, resumes once capacity returns, and prevents duplicate final replies.

## Reproduction Assessment

No high-confidence live provider-limit reproduction was run. Source inspection on current main shows existing cooldown, retry, suspension, and queue paths are not a durable replayable prompt queue.

## Solution Assessment

Yes for the durable queue direction, but no for the JSONL sidecar sketch in the discussion. The maintainable implementation should use OpenClaw's canonical SQLite-backed state model and integrate with fallback, cooldown, restart recovery, and user-visible status.
