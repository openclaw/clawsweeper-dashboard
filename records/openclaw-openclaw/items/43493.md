## Summary

Keep open: current main still does not provide a setup-time metadata prompt for custom providers, and this PR remains the focused implementation candidate, but the branch has one correctness bug that should block merge.

## What This Changes

The PR adds a custom-provider-only model metadata prompt to `openclaw configure --section model`, with helper code and tests for editing configured model display name, API adapter, reasoning, input modalities, context window, max tokens, and token pricing.

## Best Possible Solution

Fix the API Auto clearing edge case and add a regression test, then let maintainers decide whether to land this focused prompt while broader auto-detection and runtime-warning work stays with https://github.com/openclaw/openclaw/issues/48338.

## Reproduction Assessment

Yes for the review finding: source inspection on the PR branch shows selecting Auto leaves `result.api` unset while the final merge preserves the old model-level `api`. I did not run the interactive wizard in this read-only review.

## Solution Assessment

No as written. The prompt is a reasonable narrow direction, but the Auto option must either clear the model-level API override or be relabeled so it does not claim provider inheritance.

## Repair Work Prompt

On PR https://github.com/openclaw/openclaw/pull/43493, fix the model metadata API adapter Auto behavior. The UI label says `Auto (inherit from provider)`, so selecting Auto for an existing configured model with a model-level `api` override must remove/clear that model-level override so catalog resolution falls back to the provider-level `api`. Add a focused regression test in `src/commands/model-metadata.test.ts` that starts with a custom model containing `api`, selects `__auto`, and verifies the saved model no longer has a model-level `api`. Keep the prompt gated to `authChoice === "custom-api-key"`; do not reintroduce compat prompts or broaden the custom-provider UX. Run the focused model metadata test and `pnpm check:changed`. No changelog request for the contributor; maintainers can handle release notes if the feature lands.
