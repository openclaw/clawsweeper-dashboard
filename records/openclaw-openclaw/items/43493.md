## Summary

Keep open: current main still lacks an interactive custom-provider metadata step, and this PR is useful but not merge-ready because the new save path can leave stale model API overrides and erase tiered pricing.

## What This Changes

This PR adds a custom-provider-only metadata wizard after model configuration and tests helpers for collecting, editing, and saving model metadata.

## Best Possible Solution

Land a current-main implementation that collects custom-provider metadata during setup while preserving existing model overrides unless the user explicitly changes or clears them, with focused tests for API inheritance and pricing preservation.

## Reproduction Assessment

Yes for the review findings: source inspection shows Auto returns no api field before merging over the old model, and cost assignment rebuilds cost without tieredPricing. For the feature behavior, the PR body and screenshot show the new custom-provider metadata prompt after setup.

## Solution Assessment

No, not as-is. The direction is useful, but the implementation should preserve or explicitly clear nested/optional model fields correctly and be rebased into the current custom-provider configure flow.

## Review Findings

- **[P2] Clear the model API when Auto is selected:** `src/commands/model-metadata.ts:131-132`
  - body: Selecting Auto produces no `api` field in `updated`, and the later merge keeps `...entry.model`, so an existing model-level API override survives. The UI says inherit from provider, but the saved config keeps routing this model through the old adapter.
