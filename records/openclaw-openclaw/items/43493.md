## Summary

Keep open: current main still does not add an interactive custom-provider metadata prompt, and this PR remains a plausible implementation path, but it has one blocking correctness issue in the new API-adapter prompt semantics.

## What This Changes

The PR adds a custom-provider-only model metadata prompt and tests for editing display name, API adapter, reasoning, input modalities, token limits, and pricing during model configuration.

## Best Possible Solution

Fix the Auto adapter inheritance behavior with regression coverage, then land or replace the guarded custom-provider metadata prompt while leaving broader auto-detection and runtime-warning work to the related umbrella issue.

## Reproduction Assessment

Yes for the review finding. PR-head source shows Auto leaves updated.api absent while the final merge preserves entry.model.api, and current main treats absent model.api as provider-level inheritance.

## Solution Assessment

No as-is. The custom-provider metadata prompt is a narrow maintainable direction, but the Auto option must either clear the model-level adapter override or be relabeled before merge.
