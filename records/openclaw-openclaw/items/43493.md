## Summary

Keep open: the feature is still useful and the contributor’s screenshot is adequate real behavior proof, but the patch has one source-backed correctness issue where the new “Auto (inherit from provider)” adapter option cannot clear an existing model-level API override.

## What This Changes

This PR adds a custom-provider-only metadata prompt to `openclaw configure --section model` for editing model display name, API adapter, reasoning, input modes, context window, max output tokens, and pricing during setup.

## Best Possible Solution

Land the setup-time metadata prompt after fixing the API adapter inheritance bug, then leave broader auto-detection and runtime overflow copy to the open custom-provider UX follow-up if maintainers want more than guided setup metadata.

## Reproduction Assessment

Yes for the blocking review finding. A config with an existing model-level `api` override and provider-level `api` will keep using the stale model override after selecting Auto, because the PR omits `updated.api` and then spreads over `entry.model`.

## Solution Assessment

No, not as-is. The overall feature direction fits the custom-provider setup surface, but the Auto adapter option must delete or explicitly clear model-level `api` so it really inherits from the provider.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear the API override when Auto is selected:** `src/commands/model-metadata.ts:131-133`
  - body: Selecting `Auto (inherit from provider)` currently leaves `result.api` unset, but the save path merges over `...entry.model`, so any existing model-level `api` override is preserved. Since runtime resolves `model.api ?? provider.api`, users cannot actually return a model to provider inheritance through the new wizard even though the UI says they can.
  - confidence: 0.89
