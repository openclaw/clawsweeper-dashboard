## Summary

Keep open: current main still does not implement the custom-provider metadata prompt, and the related custom-provider context UX issues remain open. The PR has useful proof and no security concern, but it still has a blocking source-level correctness bug in the API-adapter Auto path.

## What This Changes

The PR adds a custom-provider model metadata prompt to `openclaw configure --section model`, plus helper functions and unit tests for editing configured model definitions.

## Best Possible Solution

Keep the custom-provider metadata prompt direction, fix the API inheritance semantics with regression coverage, and then let maintainer review decide whether the prompt should also move into the shared onboard path.

## Reproduction Assessment

Yes for the review finding. Source inspection of PR head shows Auto writes no `api`, the save path preserves `entry.model.api`, and current main only inherits the provider API when the model-level API field is absent.

## Solution Assessment

No as-is. The narrow maintainable fix is to make `Auto (inherit from provider)` remove an existing per-model `api` override before saving, or change the label and tests to state that Auto keeps the current override.
