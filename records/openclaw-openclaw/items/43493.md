## Summary

Keep open: current main still lacks the custom-provider metadata prompt and the linked context-window UX issue explicitly points to this PR as the narrow implementation candidate. The branch has real CLI proof and the earlier bot findings were mostly addressed, but the patch still has a P2 correctness issue in the API-adapter “Auto” path.

## What This Changes

The PR adds a custom-provider metadata wizard after model authentication so users can edit model limits, adapter, modalities, and pricing during configuration.

## Best Possible Solution

Land or replace this PR with a guarded custom-provider metadata prompt that preserves current auth behavior, correctly supports provider-level API inheritance, and keeps broader auto-detection/runtime wording in the linked umbrella issue.

## Reproduction Assessment

Yes for the review finding: source inspection shows selecting `Auto (inherit from provider)` produces no `api` update, but the save path preserves the old `entry.model.api`. For the feature itself, current main source shows the metadata prompt is absent and the PR screenshot shows the proposed CLI behavior.

## Solution Assessment

No, not as-is. The custom-only prompt direction is reasonable, but the patch should clear a model-level API override when Auto is selected and cover that path with a focused test.
