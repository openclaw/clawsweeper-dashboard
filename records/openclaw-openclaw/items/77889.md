## Summary

Keep open; this maintainer-labeled PR is not a cleanup-close candidate, and the diff introduces a sensitive-media persistence regression by staging all media before WebChat's existing `sensitiveMedia` gate.

## What This Changes

The PR adds a WebChat reply-media normalizer, calls it from Gateway WebChat media transcript/final display paths, adds regression coverage, and updates the changelog.

## Best Possible Solution

Keep the outbound normalizer seam, but skip or non-persistently pass through `sensitiveMedia` payloads before copying local files, with a regression for sensitive local media.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: a WebChat reply payload pointing at an agent `codex-home` image reaches managed image display while current allowed roots exclude that directory; I did not live-run it in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No; staging through the shared outbound normalizer is the right boundary, but the patch must preserve the live-only `sensitiveMedia` contract before copying local files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Skip sensitive media before staging display copies:** `src/gateway/server-methods/chat-reply-media.ts:30`
  - body: `sensitiveMedia` is documented as live-only and the existing WebChat display path skips it when `includeSensitiveMedia` is false, but this helper now sends every payload through `createReplyMediaPathNormalizer` first. That normalizer copies allowed local paths into managed media, so a sensitive local image can be written under state media even though WebChat later suppresses it. Filter sensitive payloads or add a non-persisting path before staging.
