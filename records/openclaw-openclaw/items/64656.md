## Summary

## What This Changes

## Best Possible Solution

Rebase and narrow into maintainer-reviewed pieces: decide the failover-state contract first, then thread LCM identity consistently through all context-engine lifecycle calls and keep only overflow recovery changes that improve on main with focused tests and redacted live proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Sanitize legacy auto overrides before selection:** `src/auto-reply/reply/agent-runner-execution.ts:596-597`
  - body: `sanitizeFailoverPollutedEntry` runs only inside fallback persistence, after model selection has already consumed legacy `modelOverrideSource: "auto"` state. If that fallback succeeds, persistence may never run, so polluted sessions can remain pinned; migrate or clear the auto override before `createModelSelectionState` resolves the run model.
  - confidence: 0.88
- **[P2] Preserve legacy memory identity on first reset:** `src/gateway/session-reset-service.ts:603-606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` falls back to `randomUUID()`, which creates a new LCM conversation instead of preserving the legacy session identity. Seed the preserved key from the existing session identity on first reset.
  - confidence: 0.9
- **[P2] Thread memory keys through attempt lifecycle calls:** `src/agents/pi-embedded-runner/run.ts:632`
  - body: Adding `memoryConversationKey` to the attempt params is not enough because bootstrap, assemble, afterTurn, ingest, and turn maintenance in `attempt.ts` still pass `params.sessionKey` to the context-engine helpers. Compaction can then operate on a different conversation than the prompt being assembled and finalized.
  - confidence: 0.86
- **[P2] Clear failover status when the primary recovers:** `src/auto-reply/status.ts:814-815`
  - body: `/status` treats `entry.failoverState.active` as current, but the PR only clears that state on reset or explicit user model changes. A later successful primary run can still show an old temporary failover as active; clear it when selection returns to the primary.
  - confidence: 0.8
