## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread memory keys through attempt lifecycle calls:** `src/agents/pi-embedded-runner/run.ts:635`
  - body: Adding `memoryConversationKey` to the top-level run call is not enough because `attempt.ts` still calls bootstrap, assemble, loop hooks, and finalization with only `params.sessionKey`. Once compaction uses the memory key, those lifecycle calls can operate on a different conversation from the compaction path.
  - confidence: 0.88
- **[P2] Preserve legacy memory identity on first reset:** `src/gateway/session-reset-service.ts:606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` falls back to `randomUUID()`, which rotates the LCM identity instead of preserving the legacy session identity promised by the comment. Seed the preserved key from the existing session identity on first reset.
  - confidence: 0.9
- **[P2] Sanitize legacy auto overrides before model selection:** `src/auto-reply/reply/agent-runner-execution.ts:615`
  - body: `sanitizeFailoverPollutedEntry` is applied inside fallback persistence, after model selection can already consume a legacy `modelOverrideSource: "auto"` entry. Polluted sessions can still run the fallback first; the migration needs to happen before selection resolves the run model.
  - confidence: 0.82
- **[P2] Clear failover status when the primary succeeds again:** `src/auto-reply/status.ts:814-815`
  - body: `/status` treats `entry.failoverState.active` as current, but the PR only clears that state on reset or explicit user model changes. A later successful primary run can still display an old temporary failover as active.
  - confidence: 0.8
