## Summary

Keep open: current main covers important auto-failover reset/primary-retry overlap, but this PR still contains unshipped LCM identity and overflow-recovery changes plus blocking correctness and proof gaps, so it is not closeable as implemented or mostly implemented.

## What This Changes

The PR moves fallback persistence into `SessionEntry.failoverState`, adds `memoryConversationKey` for context-engine identity, and adds overflow diagnostics plus a circuit breaker in the embedded runner.

## Best Possible Solution

Rebase and narrow the branch into maintainer-reviewed pieces: preserve the shipped source-field failover policy unless owners approve `failoverState`, thread LCM identity consistently, and land overflow handling only with focused tests plus redacted live proof.

The underlying live failover, LCM, and overflow incidents are not proven because live verification remains unchecked.

## Solution Assessment

No, not as-is. The branch mixes a session contract change, context-engine identity semantics, and overflow recovery while leaving lifecycle and state-consistency defects; a narrower maintainer-reviewed implementation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move legacy failover cleanup before model selection:** `src/auto-reply/reply/agent-runner-execution.ts:611-612`
  - body: `sanitizeFailoverPollutedEntry` runs only when persisting a fallback, after model selection has already consumed legacy `providerOverride`/`modelOverride` state. A migrated session with `modelOverrideSource: "auto"` can still take a fallback turn before cleanup, so clear or migrate auto overrides before `createModelSelectionState` resolves the run model.
  - confidence: 0.84
- **[P2] Preserve legacy LCM identity on first reset:** `src/gateway/session-reset-service.ts:603-606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` falls back to `randomUUID()`, which rotates legacy sessions away from their current context-engine identity even though reset is meant to preserve continuity. Seed the preserved key from the existing session identity instead of minting a new one on first reset.
  - confidence: 0.88
- **[P2] Thread memory keys through attempt lifecycle calls:** `src/agents/pi-embedded-runner/run.ts:632`
  - body: Passing `memoryConversationKey` into `runEmbeddedAttemptWithBackend` is not enough unless the attempt code forwards it to bootstrap, assemble, afterTurn, ingest, and maintenance. Otherwise compaction can use the memory key while the active attempt lifecycle still uses the runtime session key, so the context engine may compact or ingest a different conversation than the one being assembled.
  - confidence: 0.82
- **[P2] Clear stale failover status after primary recovery:** `src/auto-reply/status.ts:810-812`
  - body: `/status` treats `entry.failoverState.active` as current, but the PR only clears that state on reset or explicit user model changes. Once a later run succeeds on the primary, status can still report an old temporary failover as active; clear the state when selection returns to the primary.
  - confidence: 0.8
