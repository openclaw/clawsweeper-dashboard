## Summary

Keep open. Current main has already covered the reset-cleanup overlap, but this PR still contains unique LCM identity and overflow-recovery work and remains blocked by PR-head consistency bugs, merge conflicts, and missing real behavior proof.

## What This Changes

The PR moves auto-failover state into `SessionEntry.failoverState`, adds `memoryConversationKey` for context-engine/LCM identity across `/new` and `/reset`, and adds overflow diagnostics plus retry limiting in the embedded runner.

## Best Possible Solution

Rebase and narrow the branch: keep the shipped reset cleanup, decide the failover-state contract explicitly, thread LCM identity through every context-engine lifecycle call, and validate any overflow breaker with focused tests plus redacted live proof.

## Reproduction Assessment

Yes for the PR review findings: PR-head source inspection shows the memory-key routing gap, first-reset identity rotation, late sanitization, and stale status path. The underlying live failover, LCM, and overflow incidents are not high-confidence reproduced because live verification remains unchecked.

## Solution Assessment

No, not as-is. The PR mixes session contract changes, context-engine identity semantics, and overflow recovery while current main already carries part of the reset cleanup; a narrower maintainer-reviewed implementation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread memory keys through attempt lifecycle calls:** `src/agents/pi-embedded-runner/run.ts:635`
  - body: Passing `memoryConversationKey` into `runEmbeddedAttemptWithBackend` is ineffective unless `attempt.ts` forwards it to bootstrap, assemble, afterTurn, ingest, and maintenance. As written, those helpers fall back to `params.sessionKey`, while run-level compaction can use the memory key, so compaction can target a different conversation than the prompt lifecycle.
  - confidence: 0.88
- **[P2] Preserve legacy memory identity on first reset:** `src/gateway/session-reset-service.ts:603-606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` falls back to `randomUUID()`, which creates a new LCM conversation instead of preserving the legacy session identity. Seed the preserved key from the existing session identity on first reset.
  - confidence: 0.9
- **[P2] Sanitize legacy auto overrides before selection:** `src/auto-reply/reply/agent-runner-execution.ts:596-597`
  - body: `sanitizeFailoverPollutedEntry` runs inside fallback persistence, after model selection has already consumed a legacy `modelOverrideSource: "auto"` override. If the fallback succeeds without persistence, polluted sessions can remain pinned; clear or migrate auto overrides before selection resolves the run model.
  - confidence: 0.86
- **[P2] Clear failover status when the primary recovers:** `src/auto-reply/status.ts:814-815`
  - body: `/status` treats `entry.failoverState.active` as current, but the PR only clears that state on reset or explicit user model changes. A later successful primary run can still show an old temporary failover as active; clear it when selection returns to the primary.
  - confidence: 0.8
