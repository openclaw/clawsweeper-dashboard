## Summary

Keep open for maintainer review: current main shipped overlapping failover/reset fixes, but this PR still proposes unique LCM identity and overflow behavior while carrying source-level defects and no real behavior proof.

## What This Changes

The branch moves auto-failover persistence into `SessionEntry.failoverState`, adds `memoryConversationKey` for context-engine identity across `/new` and `/reset`, and adds embedded-runner overflow diagnostics plus a circuit breaker.

## Best Possible Solution

Rebase and split into maintainer-reviewed pieces: keep the shipped source-field failover policy unless owners approve `failoverState`, thread LCM identity consistently, and land overflow handling only with focused tests plus redacted live proof.

## Reproduction Assessment

Yes for the PR review findings: PR-head source inspection shows the sanitization timing, legacy reset identity rotation, incomplete memory-key routing, and stale status path. The underlying live failover, LCM, and overflow incidents are not proven because live verification remains unchecked.

## Solution Assessment

No, not as-is. The branch mixes a session contract change, context-engine identity semantics, and overflow recovery while leaving lifecycle and state-consistency defects; a narrower maintainer-reviewed implementation is safer.

Full review comments:

- **[P2] Move auto-override sanitization before selection:** `src/auto-reply/reply/agent-runner-execution.ts:596-597`
  - body: `sanitizeFailoverPollutedEntry` runs only while persisting a fallback, after model selection has already consumed legacy `providerOverride`/`modelOverride` state. A migrated session with `modelOverrideSource: "auto"` can still take one fallback turn before cleanup, so clear or migrate the auto override before `createModelSelectionState` resolves the run model.
  - confidence: 0.86
- **[P2] Seed reset memory keys from the legacy session:** `src/gateway/session-reset-service.ts:603-606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` falls back to `randomUUID()`, rotating legacy sessions away from their existing context-engine identity even though reset is meant to preserve continuity. Seed the preserved key from the existing session identity instead of minting a new one on first reset.
  - confidence: 0.88
- **[P2] Keep context-engine keys consistent across lifecycle calls:** `src/agents/pi-embedded-runner/run.ts:879`
  - body: Compaction now uses `contextEngineSessionKey`, but the attempt lifecycle and runtime context still pass the runtime `sessionKey`. Once `memoryConversationKey` exists, compaction can operate on a different conversation than bootstrap/assemble/afterTurn/ingest, so thread the same identity through all lifecycle calls.
  - confidence: 0.84
- **[P2] Clear stale failover status after primary recovery:** `src/auto-reply/status.ts:814-815`
  - body: `/status` treats `entry.failoverState.active` as current, but the PR only clears that state on reset or explicit user model changes. Once a later normal run succeeds on the primary, status can still report an old temporary failover as active; clear the state when selection returns to the primary.
  - confidence: 0.8
