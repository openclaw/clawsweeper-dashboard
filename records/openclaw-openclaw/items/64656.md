Keep open for maintainer review: current main overlaps parts of the reported behavior, but this PR is not implemented on main and the proposed branch still has blocking consistency problems in context-engine identity, legacy reset continuity, and failover status state.

## What This Changes

The PR changes fallback persistence to a structured session failover state, adds a memory conversation key for context-engine identity, and extends embedded-runner overflow diagnostics and recovery.

## Best Possible Solution

Rebase and narrow the branch under maintainer review: keep the shipped source-field reset cleanup unless maintainers choose a versioned failoverState contract, route LCM-specific conversation semantics through the context-engine/plugin lifecycle where possible, and only retain overflow diagnostics or breaker changes that improve the existing bounded recovery path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use one context-engine key for every lifecycle call:** `src/agents/pi-embedded-runner/run.ts:876`
  - body: The PR routes overflow compaction through memoryConversationKey, but the attempt lifecycle still bootstraps, assembles, finalizes, ingests, and maintains with the runtime sessionKey. Once the new key exists, compaction can operate on a different conversation than the one being assembled and finalized, so overflow recovery may compact the wrong thread.
  - confidence: 0.9
- **[P2] Preserve legacy identity on first reset:** `src/gateway/session-reset-service.ts:604-606`
  - body: For existing sessions without memoryConversationKey, /reset falls back to randomUUID(). That rotates legacy sessions away from their existing context-engine identity even though reset is meant to preserve continuity; seed the preserved key from the existing session identity instead.
  - confidence: 0.88
- **[P2] Clear stale failover state after primary recovery:** `src/auto-reply/reply/agent-runner-execution.ts:126-138`
  - body: The new failoverState is shown as active temporary failover, but normal model selection does not read it on the next turn and the PR only clears it on reset or explicit user model changes. After a later primary run succeeds, status can still report an old fallback as active.
  - confidence: 0.78
