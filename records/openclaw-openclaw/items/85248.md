## Summary

The latest PR now covers both OpenAI-compatible completions parser paths and addresses the linked unintended-tool-execution bug, so I do not see a blocking code finding. It should still stay open for maintainer acceptance of the compatibility tradeoff: provider `finish_reason` becomes authoritative, so non-conforming providers that send real tool calls with `stop` or `length` would have those tool calls stripped.

## What This Changes

The PR strips accumulated `toolCall` blocks from OpenAI-compatible streaming outputs when the mapped stop reason is not `toolUse`, with regression tests for the gateway transport parser and registered provider parser.

## Best Possible Solution

Land the mirrored parser fix after maintainers explicitly accept the provider-finish-reason contract, keeping the regression tests for both gateway transport and registered provider paths.
