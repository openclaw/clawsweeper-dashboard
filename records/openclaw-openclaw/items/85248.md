## Summary

Keep open: the gateway-side fix and real behavior proof are strong, but the registered `openai-completions` provider parser still has the same `delta.tool_calls` plus non-tool `finish_reason` mismatch and no sibling regression coverage.

## What This Changes

The PR strips accumulated OpenAI-compatible completions `toolCall` blocks when the gateway transport receives a non-tool-use finish reason and adds regression tests for that parser.

## Best Possible Solution

Make the OpenAI-completions stream contract consistent across both parsers: non-tool finish reasons strip final tool-call blocks, `tool_calls` preserves them, and focused tests prove both gateway and registered-provider paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: both parsers can accumulate `delta.tool_calls` while `finish_reason` maps to a non-tool stop reason. The contributor also provided after-fix Docker gateway logs, though I did not run live provider proof in this read-only review.

## Solution Assessment

No, not yet. The added gateway guard is the right direction, but the best fix mirrors or shares the behavior with the registered `openai-completions` provider path and tests both surfaces before merge.
