## Summary

Keep open: the patch is focused and appears correct from source and regression-test review, but the PR still needs after-fix real behavior proof for the reported provider path and maintainer acceptance of the finish-reason compatibility tradeoff.

## What This Changes

The PR adds a post-stream guard that removes OpenAI-compatible completions `toolCall` blocks unless the final stop reason maps to `toolUse`, plus regression tests for stop, tool-call, and no-tool-call streams.

## Best Possible Solution

Land the guard and regression tests once redacted live gateway/provider proof demonstrates the reported mismatch no longer executes unintended tools and maintainers explicitly accept the finish-reason compatibility contract.

## Reproduction Assessment

Yes, at source level: current main appends `delta.tool_calls` into final content even when the final `finish_reason` maps to `stop`, and the PR's mocked stream reproduces that shape. The exact live provider plus patched gateway path has not yet been shown publicly.

## Solution Assessment

Mostly yes: using `finish_reason: "tool_calls"` as the execution gate matches the OpenAI-style contract and preserves the normal tool-call path. The remaining question is compatibility for non-compliant providers, so maintainer acceptance and live proof should precede merge.
