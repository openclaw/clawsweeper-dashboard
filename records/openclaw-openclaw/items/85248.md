## Summary

The patch is focused and appears correct from source review, but this external PR still needs real provider proof and a maintainer call on making `finish_reason` authoritative for OpenAI-compatible tool execution.

## What This Changes

The PR adds an inverse stop-reason guard in `src/agents/openai-transport-stream.ts` to remove streamed `toolCall` blocks unless the final stop reason maps to `toolUse`, plus three completions-stream regression tests.

## Best Possible Solution

Land the guard with the regression tests after redacted live-provider proof and explicit maintainer acceptance of the finish-reason compatibility tradeoff.

## Reproduction Assessment

Yes from source inspection: current main appends `delta.tool_calls` even when the final mapped stop reason is `stop`, and the proposed mocked stream exercises that path. I did not run the live llama.cpp/Qwen scenario in this read-only review.

## Solution Assessment

Mostly yes: using the provider finish reason as the tool-use gate matches the documented OpenAI contract and the patch preserves explicit `tool_calls` finishes. The remaining question is compatibility for non-compliant providers, so maintainer acceptance and live proof should precede merge.
