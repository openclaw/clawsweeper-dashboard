## Label Justifications

- `P1`: The PR addresses unintended tool execution in OpenAI-compatible agent runs, which can affect real tool-running workflows.

## Summary

Keep open for maintainer review: the patch appears technically correct and covers both OpenAI-compatible parser paths, but merging makes `finish_reason` authoritative and will discard accumulated tool-call deltas unless the provider reports tool use.

## What This Changes

The PR strips accumulated `toolCall` blocks from OpenAI-compatible streaming completions when the final stop reason is not tool use, and adds regression tests for the gateway transport and registered provider parser.

## Best Possible Solution

Land this guard if maintainers accept `finish_reason` as authoritative for OpenAI-compatible tool use; otherwise add a provider compat escape hatch or narrower contract with focused tests before merge.

## Reproduction Assessment

Yes. The current-main source path accumulates `delta.tool_calls` independently of the final non-tool stop reason, and the contributor supplied Docker gateway proof using replayed SSE that exercises the mismatch.

## Solution Assessment

Yes, if maintainers accept the OpenAI-compatible contract that executable tool calls require a tool-use finish reason. The safer alternative for broader compatibility is an explicit provider compat option rather than preserving ambiguous default execution.
