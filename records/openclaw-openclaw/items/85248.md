## Summary

Keep open for human review: source inspection supports the reported bug and the patch direction, but this compatibility-sensitive provider semantic still needs maintainer signoff and stronger patched-gateway real behavior proof before merge.

## What This Changes

The PR adds a post-stream OpenAI-completions transport guard that removes `toolCall` blocks when the final stop reason is not `toolUse`, plus regression tests for stop, tool-call, and no-tool-call streaming cases.

## Best Possible Solution

Land the focused guard only after maintainers accept `finish_reason` as authoritative for OpenAI-compatible tool use and the PR includes patched-gateway proof for both spurious stop/length streams and normal `tool_calls` streams.

## Reproduction Assessment

Yes. By source inspection, feeding `processOpenAICompletionsStream` a stream with `delta.tool_calls` and `finish_reason: "stop"` on current main leaves a final `toolCall` block, and the agent loop executes final message tool calls; I did not run tests because this review is read-only.

## Solution Assessment

Yes, with a maintainer compatibility caveat. The guard follows the OpenAI/OpenClaw contract that intentional tool calls finish with `tool_calls`, but a maintainer should explicitly accept the stricter behavior for non-conforming providers and require real patched-gateway proof.
