## Summary

Keep open: this is a useful fix shape for the OpenAI-completions transport path, but it is not merge-ready because the registered built-in provider stream still has the same mismatched `delta.tool_calls` plus non-tool finish reason behavior and the posted real-behavior proof is still indirect.

## What This Changes

The PR adds a final stop-reason guard and regression tests to the OpenClaw OpenAI-completions transport stream so streamed `toolCall` blocks are removed when the provider finishes with a non-tool-use reason.

## Best Possible Solution

Land one consistent fix that makes both OpenAI-completions stream implementations drop tool-call blocks unless the final finish reason maps to `toolUse`, with focused coverage for both paths and redacted patched runtime proof.

## Reproduction Assessment

Yes. Current source shows a high-confidence mockable path where `delta.tool_calls` creates `toolCall` content while a later or same chunk `finish_reason: "stop"` leaves the final message non-tool-use; I did not execute tests in this read-only review.

## Solution Assessment

No as submitted. The transport guard is the right narrow shape for one surface, but the same rule must cover the registered OpenAI-completions provider stream or be backed by proof that the affected route never uses it.
