## Summary

Keep open: the targeted OpenClaw package-name normalization is useful and the PR supplies real CLI proof, but the current diff strips `-plugin` from every manifestless package-derived id and can break existing third-party package packs that currently load as `<name>-plugin`.

## What This Changes

The PR refactors plugin package id-hint derivation to strip `-provider` and `-plugin` suffixes through a shared helper and adds discovery coverage for `@openclaw/<id>-plugin` package names.

## Best Possible Solution

Normalize official OpenClaw package-role names while preserving existing third-party manifestless package ids, with focused regression tests for both paths.

## Reproduction Assessment

Yes: source inspection shows current main strips `-provider` but not `-plugin` in the package-derived fallback, and the PR body supplies an after-fix CLI run for the intended `@openclaw/brave-plugin` case. I did not run commands because this review is read-only.

## Solution Assessment

No: the normalization is aimed at the right package convention, but applying it to every manifestless package-derived id is not the safest compatibility shape. The narrow fix is to scope the new `-plugin` stripping or add a compat path that preserves existing third-party ids.
