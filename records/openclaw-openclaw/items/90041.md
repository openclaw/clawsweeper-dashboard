## Summary

Keep open: the reported subagent completion delivery problem is real and the PR has useful live Telegram proof, but the branch is not merge-ready because it bypasses the configured message_tool_only visible-reply contract, changes lifecycle terminal semantics without proving downstream consumers, and introduces an empty-reply fallback ordering bug.

## What This Changes

The PR changes subagent announce prompting and delivery so subagent completions can bypass message_tool_only, reclassifies unresolved tool-boundary run endings as lifecycle errors, and updates related tests.

## Best Possible Solution

Preserve message_tool_only as the default visible-reply contract, route subagent completions through the parent/message tool or document an explicit maintainer-approved exception, and split or prove the lifecycle reclassification across consumers before merge.

## Reproduction Assessment

Yes, at source level: current main code and tests show message_tool_only completion handoff requires message.send evidence and returns message_tool_delivery_missing when omitted. I did not run tests in this read-only review, but the PR body and Mantis recording provide after-fix Telegram evidence.

## Solution Assessment

No: the symptom is real, but bypassing the configured message-tool-only contract and reclassifying tool-boundary endings as lifecycle errors is broader than the proven bug. The safer fix is to honor the existing tool-only contract or make a documented maintainer-approved exception, then prove lifecycle changes separately.
