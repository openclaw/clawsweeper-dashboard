## Summary

Keep this PR open: it targets a real subagent completion message-loss regression and has strong live Telegram proof, but the current patch is not merge-ready because it bypasses the configured message-tool-only delivery contract and broadens lifecycle error semantics.

## What This Changes

The branch changes subagent announce prompting, completion delivery gating, empty-reply handling, lifecycle terminal events, and related tests for message-tool-only configured chats.

## Best Possible Solution

Preserve message_tool_only as the default visible-reply contract, route subagent completion delivery through the parent/message tool or add an explicit maintainer-approved exception, and keep lifecycle terminal semantics compatible while fixing empty fallback handling.

## Reproduction Assessment

Yes: current main source and tests show message-tool-only completion handoff requires sourceReplyDeliveryMode and returns message_tool_delivery_missing when the parent omits message.send. I did not run tests in this read-only review, but the PR body and Mantis artifact provide live Telegram candidate evidence.

## Solution Assessment

No: the branch proves the message-loss symptom, but bypassing the configured message-tool-only contract and reclassifying tool-boundary endings as lifecycle errors is broader than needed. The safer path is to honor the existing tool-only contract or make any exception explicit and maintainer-approved.
