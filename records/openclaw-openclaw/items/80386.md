## Summary

Keep open: this targets a valid self-update bug, but the current diff moves the ephemeral-prefix refusal into a shared resolver that manager detection still depends on, so a trapped install can skip self-update before it can relocate.

## What This Changes

The PR adds ephemeral per-Node prefix detection to the global update resolver, regression coverage for Homebrew/nvm/asdf/volta/fnm/n layouts, and a changelog entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so ephemeral installs still classify as npm, the update argv uses PATH npm, and coverage proves the full self-update path reaches global update.

## Reproduction Assessment

Yes, source-reproducible: current main can select an adjacent npm from a per-Node package prefix, and the linked issue has terminal evidence of the dangling Homebrew install. I did not run a destructive brew-upgrade repro in this read-only review.

## Solution Assessment

No. Refusing the per-version npm is the right direction, but the current patch does it inside a shared resolver used for manager detection, so the fix should separate ownership detection from preferred install-command selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning null here also feeds detectGlobalInstallManagerForRoot, whose final npm fallback is resolvePreferredNpmCommand(pkgRoot). For the reported Cellar trap, PATH npm points at the stable global root rather than the current trapped package root, so detection returns null and runSelfUpdate skips before it can relocate the install. Split ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
