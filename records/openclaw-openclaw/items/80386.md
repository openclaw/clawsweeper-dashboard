## Summary

Keep open: the PR targets a real source-reproducible update bug, but the current patch makes the reported trapped install fail manager detection and skip the global update path it is meant to repair.

## What This Changes

The PR adds per-Node prefix detection in `src/infra/update-global.ts`, regression coverage in `src/infra/update-global.test.ts`, and a changelog entry for update relocation after Node manager bumps.

## Best Possible Solution

Keep the PR open and split npm ownership detection from preferred-command selection so ephemeral installs still classify as npm while the update argv uses PATH `npm`, then add coverage for detection or runner behavior and refresh real terminal proof.

## Reproduction Assessment

Yes. The linked issue has a concrete Homebrew dangling-symlink repro, and current main source shows the resolver can select the adjacent Cellar npm; I did not run a destructive `brew upgrade` locally.

## Solution Assessment

No. Rejecting the per-version npm is the right direction, but doing it inside `resolvePreferredNpmCommand` also breaks manager detection, so the self-update flow can skip before it reaches the intended fallback install.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For the reported Cellar install, PATH `npm root -g` points at the stable Homebrew prefix, not the current Cellar package root; then this final preferred-npm check is null and `runSelfUpdate` falls through to `status: "skipped", reason: "not-git-install"`. That means the trapped install never reaches `global update`, so separate npm ownership detection from preferred command selection and add a regression for that path.
  - confidence: 0.91
