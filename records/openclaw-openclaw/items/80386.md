## Summary

Keep open: the PR targets a real update bug, but the current patch disables the npm ownership fallback that the self-update path needs to recognize a trapped global npm install before it can relocate it.

## What This Changes

The PR adds per-Node ephemeral npm-prefix detection for Homebrew Cellar, nvm, asdf, volta, fnm, and n layouts, plus focused update-global tests and a changelog entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so trapped npm installs still classify as npm, while the actual reinstall command falls back to PATH npm for known per-Node ephemeral prefixes.

## Reproduction Assessment

Yes. Source inspection shows the PR regression: a trapped install whose PATH npm root points elsewhere loses the final npm ownership fallback, so `runSelfUpdate` can skip before global update.

## Solution Assessment

No. Refusing per-version npm is the right direction, but doing it inside the shared resolver breaks manager detection; the safer fix separates ownership detection from command selection and adds runner-level coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning null here affects both install argv selection and `detectGlobalInstallManagerForRoot`'s final npm fallback. For the reported trapped install, PATH npm root points at a stable prefix, so detection misses the current package root; with this fallback disabled, `runSelfUpdate` returns skipped before running global update. Split ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
