## Summary

Keep open: the PR targets the right updater surface, but it is not merge-ready because the broad per-version-prefix refusal can break nvm-style installs that rely on the owning npm, and the supplied proof still does not show the patched self-update relocation path.

## What This Changes

This PR changes global self-update npm resolution to refuse Homebrew Cellar and common Node version-manager per-version npm prefixes for reinstall argv, preserves npm ownership detection, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the Homebrew Cellar escape, but either preserve owning npm for nvm/asdf/fnm/n-style installs or prove and test a stable/writable replacement root before changing those existing update paths.

## Reproduction Assessment

Yes. The original Homebrew Cellar failure is source-reproducible from current main's adjacent npm selection and the linked issue's dangling-symlink evidence; the PR-introduced nvm risk is also source-reproducible from the new guard plus the documented /usr/local staging failure in https://github.com/openclaw/openclaw/issues/78775.

## Solution Assessment

No. Refusing ephemeral prefixes is a good direction for Homebrew Cellar, but applying the same fallback to nvm/asdf/fnm/n without proving a stable PATH npm is not the safest maintainable fix.
