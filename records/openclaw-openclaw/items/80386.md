## Summary

Keep open: the reported bug is valid and the PR targets the right self-update surface, but the patch disables the npm ownership fallback that lets trapped installs reach the global update path.

## What This Changes

The PR adds per-Node ephemeral npm-prefix detection to update command selection, regression coverage for Homebrew/nvm/asdf/volta/fnm/n layouts, and a changelog entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so trapped installs still classify as npm, then use PATH npm only for the reinstall command when the owning prefix is ephemeral.

## Reproduction Assessment

Yes. Source inspection reproduces the PR regression: a trapped install whose PATH npm root points elsewhere loses the final npm ownership fallback and runSelfUpdate skips before global update.

## Solution Assessment

No. Refusing per-version npm is the right direction, but doing it inside the shared resolver breaks manager detection; the safer fix separates ownership detection from command selection and adds runner-level coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning null here affects both install argv selection and detectGlobalInstallManagerForRoot's final npm fallback. For the reported trapped install, PATH npm root points at a stable prefix, so detection misses the current package root; with this fallback disabled, runSelfUpdate returns skipped/not-git-install before running global update. Split ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
