## Summary

Keep open: the PR targets a real update bug, but the current patch would make trapped npm installs undetectable before self-update can relocate them, and the supplied proof still does not show the patched `openclaw update` path exercising the resolver.

## What This Changes

The PR adds per-Node ephemeral-prefix detection to npm global update command selection, tests install argv for Homebrew Cellar and common Node version-manager layouts, and adds a changelog entry.

## Best Possible Solution

Separate npm ownership detection from reinstall command selection so trapped installs still classify as npm while the reinstall command uses PATH npm for known per-Node ephemeral prefixes, then add runner-level regression coverage and real self-update proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can identify the adjacent npm-owned package root, while the PR moves the ephemeral-prefix guard into that same resolver so the trapped install can fall through to `not-git-install` before relocation.

## Solution Assessment

No. Refusing the per-version npm for reinstall is the right direction, but doing it inside the shared resolver is not the narrowest maintainable fix because that resolver is also used for npm ownership detection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning null here also disables `detectGlobalInstallManagerForRoot`'s final npm ownership fallback. In the reported trapped layout, PATH npm resolves to the stable prefix rather than the current Cellar package root, so `runSelfUpdate` sees no global manager and returns `skipped` / `not-git-install` before reinstalling. Split ownership detection from the reinstall command and cover that path.
  - confidence: 0.91
