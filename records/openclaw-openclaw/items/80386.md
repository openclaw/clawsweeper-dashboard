## Summary

Keep open: the bug is valid and the PR targets the right resolver, but the current patch returns `null` from a shared helper that manager detection still needs, so a trapped install can skip global update instead of relocating.

## What This Changes

This PR adds per-Node install-prefix detection for Homebrew Cellar and Node version-manager layouts so update install commands fall back to PATH `npm`, with regression coverage and a changelog entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so ephemeral installs still classify as npm, the update argv uses PATH `npm`, and coverage proves the full self-update path reaches global update.

## Reproduction Assessment

Yes, source-reproducible. The linked issue has concrete Homebrew dangling-symlink evidence, and source inspection shows both the original adjacent-npm selection and the PR-introduced manager-detection failure path.

## Solution Assessment

No. Refusing per-version npm is the right direction, but doing it inside the shared resolver breaks install-manager detection; ownership detection and command selection should be separated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For the reported Cellar install, PATH `npm root -g` points at the stable Homebrew prefix, not the current Cellar package root; then this final preferred-npm check is null and `runSelfUpdate` falls through to `status: "skipped", reason: "not-git-install"`. Split npm ownership detection from preferred command selection and cover that path.
