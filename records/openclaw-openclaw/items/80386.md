## Summary

Keep open: the linked bug is real and the PR is focused, but the current patch still makes trapped per-Node installs undetectable by returning null from the same npm resolver used for manager detection, so the self-update path can skip before it reaches the intended fallback install.

## What This Changes

This PR adds ephemeral per-Node npm prefix detection to the update global resolver, regression coverage for Homebrew/nvm/asdf/volta/fnm/n layouts, and a changelog entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so ephemeral installs still classify as npm, the update argv uses PATH `npm`, and coverage proves the full self-update path reaches global update.

## Reproduction Assessment

Yes. The linked issue has concrete Homebrew dangling-symlink evidence, and current main source shows the resolver can select the adjacent Cellar npm; I did not run a destructive brew upgrade locally.

## Solution Assessment

No. Refusing the per-version npm is the right direction, but doing it inside the shared resolver also breaks manager detection, so the fix should separate ownership detection from command selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For the reported Cellar install, PATH `npm root -g` points at the stable Homebrew prefix, not the current Cellar package root; then this final preferred-npm check is null and `runSelfUpdate` falls through to `status: "skipped", reason: "not-git-install"`. That means the trapped install never reaches global update, so split npm ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
