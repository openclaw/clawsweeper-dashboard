## Summary

Keep this PR open: it targets a real self-update trap, but the current diff can make trapped npm installs undetectable and cause `openclaw update` to skip instead of relocating the install.

## What This Changes

The PR adds per-Node ephemeral npm prefix detection to global update command selection, tests Homebrew/nvm/asdf/volta/fnm/n layouts, and adds a changelog fix entry.

## Best Possible Solution

Split npm ownership detection from install-command selection so trapped installs still classify as npm, update argv uses PATH `npm`, and coverage proves the self-update path reaches global update.

## Reproduction Assessment

Yes, as a source reproduction. The linked issue provides concrete Homebrew dangling-symlink evidence, and current main shows the resolver can select the adjacent Cellar npm; I did not rerun a destructive brew upgrade locally.

## Solution Assessment

No. Refusing the per-version npm is the right direction, but doing it inside the shared resolver also breaks manager detection, so the fix should separate ownership detection from command selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For a Cellar install, PATH `npm root -g` points at the stable Homebrew prefix rather than the current Cellar package root; after this guard the final preferred-npm check is also null, so `runSelfUpdate` falls through to `status: "skipped"` / `reason: "not-git-install"` instead of relocating the install. Split ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
