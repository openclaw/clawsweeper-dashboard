## Summary

Keep open: the linked bug is valid and current main still lacks the per-Node prefix escape, but this PR still breaks the trapped-install path by removing the shared npm ownership fallback that self-update needs before it can run the global reinstall.

## What This Changes

The PR adds per-Node ephemeral-prefix detection to update-global, changes npm global install argv to fall back to PATH npm for those layouts, adds unit coverage, and adds a changelog entry.

## Best Possible Solution

Separate npm ownership detection from reinstall command selection so trapped installs still classify as npm while the reinstall command uses PATH npm for known ephemeral prefixes, then cover that path at manager detection or runner level.

## Reproduction Assessment

Yes. Source inspection shows current main can pick an adjacent per-version npm, and PR-head source shows the proposed guard would make detectGlobalInstallManagerForRoot return null in the trapped layout before self-update can reinstall.

## Solution Assessment

No. Refusing per-version npm is the right direction, but doing it inside the shared resolver is not the narrowest maintainable fix because that resolver is also used as npm ownership detection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning null here also disables detectGlobalInstallManagerForRoot's final npm ownership fallback. In the reported trapped layout, PATH npm resolves to the stable prefix rather than the current Cellar package root, so runSelfUpdate sees no global manager and returns skipped/not-git-install before reinstalling. Split ownership detection from the command used for reinstall and cover that path.
  - confidence: 0.91
