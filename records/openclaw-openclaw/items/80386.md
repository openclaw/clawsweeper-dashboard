## Summary

Keep open: the bug and direction are valid, but the PR currently breaks the trapped-install detection path before the intended fallback update can run, and the supplied proof does not show the patched self-update flow.

## What This Changes

The PR adds per-Node npm-prefix detection, switches those updates to PATH `npm`, adds unit coverage, and records the fix in the changelog.

## Best Possible Solution

Split adjacent npm ownership detection from preferred update-command selection so ephemeral installs still classify as npm, then prove the runner reaches `global update` while using PATH `npm`.

## Reproduction Assessment

Yes. The linked issue gives a concrete Homebrew dangling-symlink repro, and current main source shows the resolver can select the adjacent Cellar npm; I did not run the destructive `brew upgrade` path locally.

## Solution Assessment

No. Refusing the per-version npm is the right direction, but returning `null` inside the shared resolver also disables npm install-manager detection, so the self-update flow can skip before the fallback install runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For the reported Cellar install, PATH `npm root -g` points at the stable Homebrew prefix, not the current Cellar package root; with this guard the final preferred-npm check is also null, so `runSelfUpdate` falls through to `status: "skipped", reason: "not-git-install"` instead of relocating the install. Split ownership detection from update-command selection and cover that path.
