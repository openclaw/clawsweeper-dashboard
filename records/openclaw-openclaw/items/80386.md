## Summary

Keep open: the PR targets a real update bug, but the current patch puts the ephemeral-prefix guard in a shared helper used for manager detection, so a trapped install can become `unknown` and skip the self-update path before reaching the intended PATH-npm fallback.

## What This Changes

The PR adds per-Node ephemeral npm prefix detection in `src/infra/update-global.ts`, regression coverage for Homebrew/nvm/asdf/volta/fnm/n layouts, and a changelog entry.

## Best Possible Solution

Split npm ownership detection from preferred command selection so ephemeral installs still classify as npm, while the actual install argv uses PATH `npm`, then add coverage for `detectGlobalInstallManagerForRoot` or `runGatewayUpdate` reaching `global update`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Cellar package root with PATH npm pointing at the stable Homebrew prefix makes manager detection rely on `resolvePreferredNpmCommand`, and the PR makes that fallback return `null`.

## Solution Assessment

No. Rejecting the per-version npm is the right direction, but putting the guard inside `resolvePreferredNpmCommand` also changes install-manager detection; a narrower solution should separate detection from command selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep trapped npm installs detectable:** `src/infra/update-global.ts:510-511`
  - body: Returning `null` here also feeds `detectGlobalInstallManagerForRoot`. For the reported Cellar install, PATH `npm root -g` points at the stable Homebrew prefix, not the current Cellar package root; after this guard, the final preferred-npm check is also null, so `runGatewayUpdate` returns `status: "skipped"` with `reason: "not-git-install"` before it can run the intended PATH-npm reinstall. Split npm ownership detection from preferred command selection and cover that path.
  - confidence: 0.91
