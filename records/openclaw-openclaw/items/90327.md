## Summary

This PR should stay open as a valid regression fix: current main rejects missing `content-type` streamed responses before the OpenAI SDK can parse them, while the SDK and upstream Codex HTTP stream path both parse SSE from the body without a response-header gate. The supplied live gateway logs are enough real behavior proof for this non-visual transport fix, with the remaining concern being maintainer acceptance of the shared OpenAI-compatible guard relaxation.

## What This Changes

The PR relaxes the OpenAI provider streamed-response content-type guard to allow missing `content-type` headers and adds a regression test for the ChatGPT Codex endpoint shape.

## Best Possible Solution

Land the missing-header tolerance with the regression test while preserving rejection for present non-stream content types; narrow it to the ChatGPT Codex route only if maintainers do not want the shared diagnostic behavior changed.

## Reproduction Assessment

Yes from source and report evidence, but not from a local live credentialed repro: current main rejects a streamed OK response with missing `content-type` before the OpenAI SDK SSE parser can read the body. The related issues and PR proof show this on v2026.6.1 ChatGPT-subscription models.

## Solution Assessment

Yes, this is an acceptable narrow fix for the observed regression because it preserves rejection for present bad content types and matches both OpenAI SDK and Codex body-based SSE parsing. The main alternative is narrowing the missing-header exception to ChatGPT Codex only if maintainers want to preserve old diagnostics for custom endpoints.
