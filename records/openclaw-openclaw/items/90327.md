## Summary

Keep open for maintainer review. The patch looks like a focused and likely correct regression fix: current main and v2026.6.1 reject missing streamed response content-type before the OpenAI SDK/Codex stream parser, while the PR allows the missing header and keeps present non-stream types rejected. The remaining decision is the shared compatibility and diagnostic tradeoff for custom OpenAI-compatible endpoints, not a concrete code defect.

## What This Changes

The PR relaxes the OpenAI SDK streamed-response content-type guard to allow missing or empty content-type headers and adds a ChatGPT Codex regression test.

## Best Possible Solution

Land this guard relaxation if maintainers accept the shared diagnostic tradeoff; otherwise narrow the missing-header allowance to the ChatGPT Codex host/API while keeping present non-stream content types rejected.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.1 reject a stream:true OK response with a body and missing content-type before the SDK parser; the PR body also supplies live gateway before/after logs for the ChatGPT OAuth path.

## Solution Assessment

Yes, this is an acceptable focused fix if maintainers accept the shared guard scope. The safer alternative is a host/API-specific missing-header exception, but dependency and native Codex source both support parsing the stream body without relying on the response content-type header.
