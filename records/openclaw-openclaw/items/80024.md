## Summary

Keep open: the bug fix direction is valid, but the proposed patch introduces a native approval rendering regression by summarizing the same command text twice in Matrix/Telegram-style prompts. The linked user problem remains active and should be fixed in this PR rather than closed.

## What This Changes

The branch adds long exec approval command summaries across approval presentation surfaces, carries gateway `allowedDecisions` into the Control UI modal, and updates docs, changelog, and tests.

## Best Possible Solution

Keep the PR open and make command summarization happen at exactly one presentation boundary, or make the summarizer idempotent, while preserving the full sanitized approval record and the `allowedDecisions` UI behavior.

## Reproduction Assessment

Yes from source inspection: on PR head, `buildPendingApprovalView` returns summarized `view.commandText`, and Matrix/Telegram pass that into a reply builder that now summarizes again. The helper logic also reproduces the marker changing from first 5 of 9 lines to first 5 of 11 lines after a second pass.

## Solution Assessment

No: the intended fix is the right narrow area, but this implementation should avoid applying the display summary twice. A safer version keeps one canonical summary boundary, adds regression coverage for the native text path, and then provides direct after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid summarizing native approval text twice:** `src/infra/approval-view-model.ts:78`
  - body: This makes `view.commandText` an already summarized string, but the Matrix and Telegram native handlers pass that value into `buildExecApprovalPendingReplyPayload`, and this PR now summarizes that parameter again. Long commands lose the original truncation marker and report inflated line counts, so keep the view text raw for downstream summarizers or make the summary helper idempotent before merging.
