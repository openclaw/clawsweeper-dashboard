## Summary

Keep this PR open. It targets a real current-main audit gap, but the patch duplicates only part of runtime `tools.byProvider` resolution and still misses documented/runtime provider alias, canonical-key, and provider-profile behavior; it also lacks required after-fix real behavior proof.

## What This Changes

The PR adds `tools.byProvider` handling to small-model security-audit exposure checks and adds unit tests for exact model-deny matching and non-matching model scoping.

## Best Possible Solution

Share or exactly mirror the runtime provider-policy resolver in the audit path, add alias/canonical/profile regression coverage, and include redacted real `openclaw security audit --deep` proof before merge.

## Reproduction Assessment

Yes by source inspection: current main computes `models.small_params` exposure without passing model identity into `resolveToolPolicies`, while runtime policy has a separate provider/model layer. A live repro would run `openclaw security audit --deep` with a small model and matching `tools.byProvider` deny, but I did not execute it because this review is read-only.

## Solution Assessment

No. The direction is right, but the narrow maintainable fix should reuse or match the runtime byProvider resolver so alias normalization, canonical precedence, and provider profiles cannot diverge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Mirror runtime byProvider key normalization:** `src/security/audit-extra.summary.ts:67-74`
  - body: This lookup only probes literal lowercase object keys. Runtime policy normalizes provider aliases and canonical keys, so configs such as a `bedrock/...` byProvider entry can deny tools at runtime while this audit still reports them exposed. Reuse/export the runtime lookup or add equivalent normalization and alias/canonical tests here.
  - confidence: 0.9
- **[P2] Apply byProvider profiles when computing exposure:** `src/security/audit-extra.summary.ts:100-103`
  - body: `tools.byProvider[provider].profile` is part of the documented runtime filtering order, but this path feeds the matched entry through `pickSandboxToolPolicy`, which drops profile-only policies. A small model restricted by a provider-scoped `minimal` profile can still look web-exposed in the audit.
  - confidence: 0.8
