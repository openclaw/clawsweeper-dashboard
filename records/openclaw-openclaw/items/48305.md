## Summary

Keep open: current main still drops config-reload restart reasons, and this PR contains the useful pass-through, but the proposed restart-infra diagnostics add unsafe stack-trace logging and the user-facing fix still needs changelog and focused coverage before merge.

## What This Changes

The PR passes config-reload reason strings into gateway restart emission/deferral and adds restart-entry logging for emitted and missing restart reasons.

## Best Possible Solution

Land a narrow current-main patch that passes `config-reload: ${reasons}` into both reload restart paths, adds focused reason-propagation coverage, and records the fix in `CHANGELOG.md` without stack-trace diagnostics.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: trigger a config reload plan with `restartReasons: ["gateway.port"]`; the handler computes the string but emits or defers the restart without passing it to restart infra.

## Solution Assessment

No, not as currently proposed. The config-reload pass-through is the right narrow fix, but the missing-reason stack-trace diagnostic is overbroad and current main already has generic reason plumbing.
