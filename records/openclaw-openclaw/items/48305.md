## Summary

Keep open: current main still drops config-reload restart reasons at the gateway restart call sites, but this PR also adds unsafe missing-reason stack-trace logging and is missing required release-note coverage.

## What This Changes

The PR threads config-reload reason strings into gateway restart calls and adds restart-entry logging plus missing-reason diagnostics in the restart helper.

## Best Possible Solution

Land a narrow current-main patch that passes `config-reload: ${reasons}` into both gateway restart paths, adds a focused reason-propagation assertion, omits stack-trace diagnostics, and records the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: trigger a config reload plan with `restartReasons: ["gateway.port"]`; the handler computes the reason but emits or defers the restart without passing it to restart infra.

## Solution Assessment

No, not as currently proposed. The config-reload call-site pass-through is the right narrow fix, but the stack-trace diagnostic is overbroad and current main already has the generic restart-reason plumbing.
