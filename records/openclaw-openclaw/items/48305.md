## Summary

Keep open: the config-reload reason loss is still source-reproducible on current main, but this PR should not merge as-is because it adds full stack-trace logging for missing restart reasons and has no after-fix real behavior proof from the external contributor.

## What This Changes

The PR passes config-reload restart reasons into deferred and immediate gateway restart paths and adds restart emission/missing-reason diagnostics in restart infra.

## Best Possible Solution

Land a current-main patch that passes `config-reload: ${reasons}` into both config-reload restart paths and adds focused coverage, while leaving restart infra’s generic reason plumbing free of stack-trace diagnostics.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: a reload plan with `restartReasons: ["gateway.port"]` computes `reasons`, then calls the restart infra without passing that value; I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. The config-reload pass-through is the right narrow fix, but the missing-reason stack-trace diagnostic should be removed and the patch should be rebased onto current restart reason plumbing.
