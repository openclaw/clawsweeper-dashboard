## Summary

Current `main` already has restart-reason plumbing, but the config-reload caller still drops the computed reason, so this PR is not obsolete. It should stay open, but it is not merge-ready because the branch conflicts with current restart code, adds stack-trace logging, and lacks real behavior proof.

## What This Changes

The PR passes config-reload restart reasons into gateway restart emission and adds restart emission diagnostics.

## Best Possible Solution

Apply only the config-reload reason handoff on top of current main’s restart plumbing, avoid stack-trace logging, add focused immediate/deferred config-reload coverage, and require redacted live gateway log proof before merge.

## Reproduction Assessment

Yes, via source inspection. Current main computes config-reload restart reasons but still calls the deferred and immediate restart emitters without passing that reason, while the run loop can consume an emitted restart reason when one is present.

## Solution Assessment

No, not as proposed. Passing the config-reload reason through is the narrow right fix, but the branch should be rebased onto current main and should drop the stack-trace warning rather than adding broad diagnostics to restart logs.
