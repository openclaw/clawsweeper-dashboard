## Summary

Keep this PR open, but do not merge it as-is: current main still drops the config-reload reason at the restart handoff, while the PR also adds unsafe stack-trace diagnostics and lacks required real behavior proof.

## What This Changes

The PR passes config-reload restart reasons into gateway restart emission and adds restart emission/missing-reason diagnostics in restart infra.

## Best Possible Solution

Carry forward the narrow current-main patch that passes `config-reload: ${reasons}` into both restart paths, add focused coverage, and leave generic restart infra free of stack-trace diagnostics.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: a reload plan with `restartReasons` computes `reasons`, then calls the restart infra without passing that value; I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. The config-reload pass-through is the narrow maintainable fix, but the missing-reason stack-trace diagnostic should be removed and the branch should be rebased onto current restart reason plumbing.
