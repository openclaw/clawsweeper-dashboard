## Summary

Keep open: the PR is not obsolete because current main still computes config-reload restart reasons and then drops them before emitting the restart. It should not merge as-is because the branch also adds a broad missing-reason stack-trace warning, and it does not address the linked report's observed `gateway-tool` `reason=none` log or later Windows missing-config root-cause discussion.

## What This Changes

The PR branch passes `config-reload: ...` reason strings into deferred and immediate gateway restarts and adds restart-entry logging/warnings in the restart helper.

## Best Possible Solution

Refresh or replace this with a narrow patch that passes `config-reload: ${reasons}` from `requestGatewayRestart` into both deferred and immediate restart emissions, adds a focused test that observes the propagated reason, and handles the #48205 Windows/gateway-tool breadcrumb separately if that remains a product goal.

## Reproduction Assessment

Yes. A source-level reproduction is clear: invoke the config reload restart path with `restartReasons: ["gateway.port"]` on current main; it computes `gateway.port` but emits or defers the restart without passing a reason.

## Solution Assessment

No, not as currently proposed. Passing the config-reload reason at the two call sites is the right small fix, but adding a full stack-trace warning for every missing reason is too broad and does not solve the linked `gateway-tool` `reason=none` report.
