## Summary

Keep this PR open: the current `main` still drops config-reload restart reasons, and the PR targets the right gateway call sites, but it should not merge as-is because it adds a broad missing-reason stack-trace warning and lacks the required user-facing changelog entry.

## What This Changes

The PR passes `config-reload: ...` reason strings into deferred and immediate gateway restarts and adds restart-entry logging and missing-reason diagnostics in the restart helper.

## Best Possible Solution

Land a narrow current-main patch that passes `config-reload: ${reasons}` into both the deferred and immediate restart emissions, adds a focused reason-propagation assertion, omits stack-trace diagnostics, and adds the required changelog entry. Treat the linked `gateway-tool` `reason=none` breadcrumb as a separate follow-up if maintainers still want tool-call attribution beyond caller-supplied reasons.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current `main`: trigger the config reload restart plan with `restartReasons: ["gateway.port"]`; the handler computes `gateway.port` but emits or defers the restart without passing a reason.

## Solution Assessment

No, not as currently proposed. The config-reload call-site change is the narrow maintainable fix, but adding a full stack-trace warning for every missing reason is too broad and does not directly solve the linked gateway-tool report.
