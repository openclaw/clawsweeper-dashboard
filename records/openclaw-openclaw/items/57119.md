## Summary

Keep open. Current main still treats `skills.entries.<name>.enabled:false` as skill eligibility/prompt visibility, while filesystem tools and `exec` are governed by workspace/tool/sandbox policy and do not receive a disabled-skill realpath deny boundary.

## What This Changes

The issue asks OpenClaw to make `skills.entries.<name>.enabled:false` block filesystem reads and script execution for that skill, not only hide it from skill discovery/prompt surfaces.

## Best Possible Solution

Define the `enabled:false` contract explicitly; if it is access control, enforce one centralized realpath-aware disabled-skill policy across filesystem, sandbox sync, and exec paths, otherwise clarify docs/UI and direct operators to sandbox/tool policy for isolation.

## Reproduction Assessment

Yes, from source inspection. With a disabled skill and filesystem/exec tools still allowed, current main filters the skill catalog but does not pass a disabled-skill denylist into read/write/edit/apply_patch or exec execution paths.

## Solution Assessment

Unclear. The requested enforcement is a plausible security direction, but the maintainable fix depends on maintainer intent for whether the existing toggle is an access-control boundary or only a visibility switch.
