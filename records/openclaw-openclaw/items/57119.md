## Summary

Keep open. Current main and the latest release still treat `skills.entries.<key>.enabled: false` as skill eligibility/prompt filtering, not as a disabled-skill filesystem or exec deny boundary; the remaining decision is a security/product contract choice rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to make disabled skills inaccessible to read and exec tool paths instead of only removing them from skill matching and prompts.

## Best Possible Solution

Define the disabled-skill security contract explicitly, then enforce it through one shared skill-access policy used by read, exec/sandbox execution, and docs.

## Reproduction Assessment

Yes, source-level: disabled skills are filtered out of eligibility, while read and exec access still use generic tool and sandbox policy rather than a disabled-skill deny boundary. I did not run a live exploit path in this read-only review.

## Solution Assessment

Unclear as a concrete implementation. Gateway-level filtering is plausible, but the best maintainable fix is likely a central skill-access policy after maintainers confirm the intended security contract and upgrade behavior.
