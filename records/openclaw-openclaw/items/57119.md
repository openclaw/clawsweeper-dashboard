## Summary

Keep this issue open. Current main still treats `skills.entries.<key>.enabled: false` as eligibility/prompt status rather than a filesystem or process-execution deny boundary, and the provided ClawSweeper comment already reached the same keep-open conclusion on the same security-sensitive gap.

## What This Changes

Request that disabling a skill also block read/write/edit/apply_patch and exec access to that skill directory, instead of only removing the skill from automatic skill eligibility and prompts.

## Best Possible Solution

Define the skill disablement contract explicitly. If it is meant as access control, implement a centralized disabled-skill realpath policy and enforce it at every filesystem and exec boundary, with docs and tests across host, sandbox `none`/`ro`/`rw`, managed/bundled/plugin/workspace skills, and link escapes; if it is visibility-only, rename or document the setting and route operators to tool policy, workspace-only FS, sandboxing, OS permissions, or containers for enforcement.

## Reproduction Assessment

Yes. A high-confidence repro can be built by creating a workspace skill, setting `skills.entries.<key>.enabled: false`, confirming the prompt omits it, then invoking `read` or `exec` directly against its path; I did not run it because this review is read-only, but current source and existing workspaceOnly=false tests show the missing deny boundary.

## Solution Assessment

Unclear until maintainers choose the product/security semantics. Gateway-level path filtering is the narrowest technical direction if disabled skills are meant to be access-controlled, but it needs a centralized realpath deny policy rather than ad hoc checks in one tool.
