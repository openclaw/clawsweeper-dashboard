## Summary

Keep open. Current main and v2026.6.10 still treat `skills.entries.<key>.enabled: false` as skill eligibility/prompt filtering, while read and exec remain governed by generic filesystem, sandbox, approval, and tool policy controls rather than a disabled-skill access boundary.

## What This Changes

The issue requests making disabled skills inaccessible to agent read and exec paths, not only hidden from automatic skill matching.

## Best Possible Solution

Define the disabled-skill security contract explicitly, then enforce it through a central skill-access policy that is reused by prompt exposure, filesystem tools, exec/sandbox execution, Gateway/UI status, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main filters disabled skills out of eligibility and prompt/status surfaces, while read and exec access still use generic filesystem, sandbox, approval, and safe-bin policy instead of disabled skill paths. I did not run a live exploit because this review is read-only.

## Solution Assessment

Unclear as an implementation path: the security direction is valid, but the best fix needs maintainer agreement on whether `enabled:false` becomes a hard compatibility boundary or whether a separate strict policy is needed.
