## Summary

Keep open. Current main and latest release still treat `enabled:false` as skill eligibility and visibility filtering, not as an enforceable filesystem or execution boundary, and the remaining decision is a security/product contract rather than cleanup.

## What This Changes

The issue requests making `skills.entries.<name>.enabled:false` enforce a real skill access boundary that blocks reads, writes, and script execution under disabled skill directories.

## Best Possible Solution

Define the disabled-skill security contract explicitly; if it is access control, enforce one centralized realpath-aware policy across skill loading/sync, file tools, exec, sandbox/remote paths, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main filters disabled skills out of prompt/status eligibility but does not carry disabled skill realpaths into file-tool, sandbox-sync, or exec deny policy.

## Solution Assessment

Unclear. The requested enforcement is plausible, but maintainers need to decide whether `enabled:false` is an access-control contract or a visibility switch before implementing a centralized boundary.
