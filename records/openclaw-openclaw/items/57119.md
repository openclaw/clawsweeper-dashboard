## Summary

Keep this issue open. Current main still treats `skills.entries.<key>.enabled: false` as skill eligibility and prompt/command visibility, while filesystem and exec enforcement remains governed by workspace/tool/sandbox policy rather than any disabled-skill realpath deny boundary.

## What This Changes

Request enforcement so disabled skills cannot be read or executed manually through agent filesystem or exec tools.

## Best Possible Solution

Define the `enabled: false` contract explicitly; if it is access control, enforce a centralized disabled-skill realpath deny policy across filesystem tools and exec, otherwise rename or document the toggle as visibility-only and point operators to sandbox, tool deny, workspace-only, OS permission, or container isolation controls.

## Reproduction Assessment

Yes, from source inspection: disablement filters skill eligibility and prompt exposure, while filesystem and exec paths do not receive or enforce a disabled-skill directory policy. I did not run a live exploit path because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear; the issue lists plausible enforcement strategies, but the best fix depends on maintainer security/product intent for the existing toggle. A narrow maintainable implementation would centralize path denial before all filesystem and exec entry points rather than adding per-tool ad hoc checks.
