## Summary

Keep open. Current main and v2026.6.10 still treat `skills.entries.<key>.enabled: false` as skill eligibility and prompt filtering, not as a disabled-skill filesystem or exec deny boundary; this remains a security/product decision rather than stale cleanup.

## What This Changes

The issue requests that disabling a skill also block reading that skill's files and executing its scripts, instead of only hiding the skill from prompt/eligibility surfaces.

## Best Possible Solution

Define the disabled-skill security contract explicitly, then enforce it through one shared skill-access policy used by read, exec/sandbox execution, skill materialization, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: disabled skills are filtered out of eligibility and visible prompt/status surfaces, while read and exec access still use generic filesystem, sandbox, approval, and safe-bin policy rather than a disabled-skill deny boundary. I did not run a live exploit path in this read-only review.

## Solution Assessment

Unclear as a concrete implementation: the requested security direction is valid, but the best fix needs a central access policy and maintainer confirmation of the compatibility contract before code lands.
