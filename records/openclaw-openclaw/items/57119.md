- `impact:security`: The issue is about whether disabled skill directories remain readable or executable by agent tools despite the admin-facing disabled state.

## Summary

Keep open. Current main still treats `skills.entries.<name>.enabled` as a skill selection and prompt-surface filter, not as a filesystem or process-execution deny boundary; changing that would create a new security/product contract for tool sandboxing rather than a stale cleanup close.

## What This Changes

The issue requests that disabled skills be inaccessible to agent read/write/exec tool paths, not merely removed from automatic skill discovery and prompting.

## Best Possible Solution

Define the disabled-skill security contract explicitly, then enforce it at the tool boundary with tests that cover both skill discovery and direct file/process access to disabled skill directories.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: disabled skills are filtered from the active registry, while the Codex file/shell tool registration path does not consult disabled skill roots. I did not run a live exploit because this review is read-only and the remaining behavior is clear from current main.

## Solution Assessment

Unclear as a final implementation, because the report correctly identifies the gap but the safest fix requires maintainers to choose the permanent sandbox boundary and compatibility semantics. A narrow code change should follow that decision rather than silently changing all disabled-skill filesystem access.
