## Summary

Keep open. Current main and the latest release still treat `enabled:false` as skill eligibility/prompt filtering, not as a realpath-aware file or process boundary; changing that contract is security-sensitive and needs maintainer product/security direction.

## What This Changes

Requests that `skills.entries.<name>.enabled:false` block manual reads and script execution under disabled skill directories, not only remove skills from auto-matching.

## Best Possible Solution

Define the `enabled:false` contract explicitly; if it is access control, enforce one centralized realpath-aware disabled-skill policy across skill sync, filesystem tools, exec, docs, and regression tests, otherwise clarify the docs/UI and direct operators to tool policy and sandbox isolation.

## Reproduction Assessment

Yes, from source inspection. A disabled skill is filtered out of prompt/status eligibility, but current main still does not carry disabled skill realpaths into sandbox sync, filesystem tools, or exec as a deny boundary.

## Solution Assessment

Unclear. The requested enforcement is plausible, but the best maintainable fix depends on whether maintainers want `enabled:false` to be a hard access-control contract or a visibility switch backed by separate sandbox/tool policies.
