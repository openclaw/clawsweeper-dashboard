## Summary

Keep open. Current main still treats `skills.entries.<key>.enabled: false` as a skill eligibility/prompt/env control, not as a host filesystem or exec deny boundary, so this is security-sensitive product work rather than stale cleanup.

## What This Changes

The issue requests making `skills.entries.<name>.enabled: false` enforce a read, write, and exec denial boundary for that skill directory.

## Best Possible Solution

Define the supported disabled-skill security contract, then implement enforcement at the chosen boundary with focused read/write/exec tests, docs that distinguish prompt hiding from sandbox denial, and upgrade-safe behavior for existing configs.

## Reproduction Assessment

Yes, source-level: current main filters disabled skills out of skill eligibility, prompts, env handling, and sandbox materialization, while host read and exec are governed by the generic tool/sandbox policy instead of a disabled-skill denylist. I did not run a live malicious-skill scenario in this read-only review.

## Solution Assessment

Unclear as an implementation path: hardening `enabled: false` directly may be desirable, but it changes config semantics and needs maintainer security/product approval before deciding between strict default enforcement, an explicit quarantine mode, or broader capability-policy work.
