## Summary

Keep open. Current main still treats `skills.entries.<name>.enabled:false` as skill eligibility and prompt visibility, while filesystem, sandbox sync, and `exec` paths do not enforce a disabled-skill realpath boundary; the security-sensitive contract needs maintainer direction before implementation.

## What This Changes

Request that `skills.entries.<name>.enabled:false` become an enforced access-control boundary blocking reads and script execution under disabled skill directories.

## Best Possible Solution

Define the `enabled:false` contract explicitly; if it is access control, enforce one centralized realpath-aware disabled-skill policy across filesystem tools, sandbox sync, and exec paths, otherwise clarify docs/UI and direct operators to sandbox and tool policy for isolation.

## Reproduction Assessment

Yes, from source inspection. A disabled skill is filtered out of eligible prompt/snapshot entries, but current main does not pass disabled-skill realpaths into filesystem tools, sandbox sync, or exec as a deny boundary.

## Solution Assessment

Unclear. The requested enforcement is plausible, but the maintainable fix depends on maintainer/security intent for whether the existing toggle should become access control or remain a visibility switch with clearer documentation.
