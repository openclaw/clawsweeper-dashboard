## Summary

Keep open. Current main still treats `skills.entries.<key>.enabled: false` as skill eligibility, prompt, and sandbox materialization filtering, not as a host read or exec deny boundary.

## What This Changes

Request to make disabling a skill prevent agents from reading or executing files inside that skill directory, not only hide it from skill selection.

## Best Possible Solution

Define the disabled-skill security contract, then enforce it at the chosen read/write/exec/sandbox boundary with regression tests and docs distinguishing prompt hiding from access denial.

## Reproduction Assessment

Yes, source-level: current main filters disabled skills out of eligible skill snapshots and sandbox sync, while host read and exec follow generic tool policies and do not receive disabled skill deny roots. No live exploit run was performed because this review is read-only.

## Solution Assessment

Unclear as a concrete implementation: hardening `enabled: false` may be the right end state, but maintainers need to choose whether it is strict default behavior, opt-in quarantine, or broader capability-policy work.
