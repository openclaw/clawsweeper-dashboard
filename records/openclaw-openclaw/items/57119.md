## Summary

Keep open. Current main and v2026.5.22 still treat `skills.entries.<name>.enabled: false` as skill eligibility and prompt/status filtering, not as a hard filesystem or shell-execution deny boundary.

## What This Changes

The issue asks OpenClaw to make disabled skills inaccessible to agent read and exec paths instead of only hiding them from skill auto-selection.

## Best Possible Solution

Define the disabled-skill security contract explicitly; if it is access control, enforce one centralized realpath-aware policy across skill discovery/sync, file tools, exec, sandbox/remote paths, docs, and regression tests.

## Reproduction Assessment

Yes, at source level. Current main filters disabled skills out of eligibility and prompt snapshots, while file-tool and exec construction do not carry disabled skill realpaths into a deny policy.

## Solution Assessment

Unclear. The requested enforcement is plausible and aligned with safe defaults, but maintainers need to decide whether `enabled:false` is an access-control contract or a visibility switch before implementing the boundary.
