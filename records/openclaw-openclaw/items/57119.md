## Summary

Keep open. Current main and the latest release still treat `skills.entries.<key>.enabled: false` as skill eligibility/prompt filtering, not as a read/exec deny boundary; adjacent fixes cover stale snapshots or env overrides but do not resolve this security-contract question.

## What This Changes

Requested behavior: enforce disabled skills as inaccessible to direct file reads and script execution, not only absent from automatic skill matching.

## Best Possible Solution

Make the disabled-skill security contract explicit, then enforce the chosen policy at the shared skill access/tool boundary with regression tests and docs that distinguish prompt hiding from access denial.

## Reproduction Assessment

Yes, source-level: current main filters disabled skills out of snapshots and prompt/sandbox materialization, while read and exec access are governed by generic filesystem, sandbox, and approval policy rather than disabled skill paths. No live exploit run was performed in this read-only review.

## Solution Assessment

Unclear as a concrete implementation: gateway-level filtering is plausible, but the best durable fix is likely a central skill-access policy consumed by read, sandbox materialization, and exec approval paths after maintainer security/product direction.
