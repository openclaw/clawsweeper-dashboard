## Summary

Keep open. Current main and the latest release still treat `enabled:false` as skill eligibility and prompt/status filtering, not as a filesystem or exec deny boundary, and the remaining decision is a security/product contract rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to enforce disabled skills as an access-control boundary that blocks reading, writing, syncing, and executing files under disabled skill directories.

## Best Possible Solution

Define the disabled-skill security contract explicitly; if it is access control, enforce one centralized realpath-aware policy across skill sync, file tools, exec, sandbox/remote paths, docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. Current main filters disabled skills out of prompt/status eligibility but does not carry disabled skill realpaths into file-tool, sandbox-sync, or exec deny policy.

## Solution Assessment

Unclear. The requested enforcement is plausible, but maintainers need to decide whether `enabled:false` is an access-control contract or a visibility switch before implementing a centralized boundary.
