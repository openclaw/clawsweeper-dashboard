## Summary

Keep open. Current main still treats unset and `OPENCLAW_RAW_STREAM=0` as disabled at the parser level, but the macOS LaunchAgent path can still preserve effective service environment through generated wrapper/env-file metadata, and normal restart does not refresh that metadata on the successful `kickstart -k` path. The docs still only describe how to enable raw-stream logging, not disable values or LaunchAgent precedence.

## What This Changes

The issue reports that macOS LaunchAgent-managed raw-stream logging can continue appending after `OPENCLAW_RAW_STREAM` is unset or set to `0`.

## Best Possible Solution

Keep this open for a focused macOS service-env and docs fix: document truthy and disabled raw-stream values; clarify precedence among `launchctl` inherited env, state-dir `.env` / config durable env, and generated LaunchAgent wrapper files; make restart/install/doctor detect or repair stale raw-stream service metadata, or print a precise reinstall instruction; and cover the disable path with a regression test.

## Reproduction Assessment

Yes. The issue gives a concrete LaunchAgent reproduction path, and current code supports the same failure mechanism through process-env-gated raw-stream writes plus persistent wrapper service env; I did not run a live macOS launchd repro in this read-only Linux checkout.

## Solution Assessment

Unclear. The requested direction is valid, but the maintainable fix should be a narrow service-env refresh/diagnostic plus docs update, not a change to the raw boolean parser, which already treats `0` and unset as disabled.
