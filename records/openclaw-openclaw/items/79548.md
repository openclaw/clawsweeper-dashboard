## Summary

Keep open: this targets a real ACP delivery gap, but the patch is not merge-ready because the fallback trigger does not match the real dispatcher contract, the final-only kind change can regress final-mode TTS, the new binding route can broadcast to every active binding, and no after-fix real behavior proof is attached.

## What This Changes

The PR changes ACP final-only projection to emit buffered text as block delivery, adds a session-binding routeReply fallback when dispatcher sends return false, and adds spawn-child delivery regression tests.

## Best Possible Solution

Implement a scoped persistent ACP delivery path that is keyed to the bound requester conversation and the actual async delivery lifecycle, with regression coverage using real dispatcher semantics and redacted live channel proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for live behavior: source inspection supports the suspected missing binding fallback, but the PR test simulates `send*()` returning false rather than the current dispatcher’s post-complete behavior.

## Solution Assessment

Is this the best way to solve the issue? No; the proposed fallback is tied to a synchronous false return that real teardown does not produce, and the binding route should resolve a single scoped destination or fail closed instead of broadcasting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Trigger fallback on real dispatch failure:** `src/auto-reply/reply/dispatch-acp-delivery.ts:450`
  - body: This fallback only runs when `sendToolResult`/`sendBlockReply`/`sendFinalReply` returns false. The real `ReplyDispatcher` returns false only before enqueue when normalization drops the payload; after `markComplete()` it still queues normalized payloads and records async transport errors later, so the parent-ended path described here will not enter this branch. Please trigger the binding route from the actual disposed/failed delivery signal or move delivery onto a persistent bound dispatcher.
  - confidence: 0.93
- **[P2] Preserve final-mode TTS for final-only output:** `src/auto-reply/reply/acp-projector.ts:303`
  - body: Changing final-only flushes from `final` to `block` means `maybeApplyAcpTts` skips them when the configured TTS mode is `final`, because that path returns early for non-final kinds. Users with final-only TTS would lose synthesized audio for ACP completion text. Keep final TTS semantics, or separate the delivery kind needed for fallback from the TTS kind.
  - confidence: 0.88
- **[P1] Fail closed on ambiguous bindings:** `src/auto-reply/reply/bind-aware-dispatch.ts:60-78`
  - body: This loops over every active binding for the session and sends the same payload to each. A persistent ACP session can have multiple active bindings, and existing bound-delivery routing avoids choosing without requester context. Broadcasting fallback output can duplicate or leak a private child update to unrelated bound chats; resolve one requester-matching binding or fail closed.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete privacy risk by routing fallback payloads to every active session binding without requester scoping or ambiguity handling.

Concerns:

- **[medium] Broadcasts fallback payloads to all active bindings:** `src/auto-reply/reply/bind-aware-dispatch.ts:60`
  - body: `deliverViaSessionBindings` iterates all active bindings for a session and calls `routeReply` for each. If an ACP session is bound in more than one conversation, output from one chat can be sent to other bound chats; the route should use requester context or fail closed on ambiguity.
  - confidence: 0.85
