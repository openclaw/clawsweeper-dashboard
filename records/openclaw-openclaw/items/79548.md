## Summary

Keep open: this is a plausible bug-fix direction, but the current PR still has blocking routing correctness issues and lacks required after-fix real Telegram proof.

## What This Changes

The branch adds a bind-aware ACP delivery fallback through session bindings, restores final_only ACP flushes as final replies, adds spawn-child regression tests, and updates the changelog.

## Best Possible Solution

Rework the fallback to use the existing channel delivery-target resolver, preserve threadId/parent conversation routing, prevent duplicate final fallback after bind-routed blocks, then add redacted live Telegram proof and focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the live Telegram report, because no redacted real Telegram before/after proof is attached; the remaining PR defects are source-reproducible from the routeReply target and failedCounts/final fallback paths.

## Solution Assessment

Is this the best way to solve the issue? No. The fallback direction is useful, but the current implementation bypasses channel target resolution and can duplicate visible block output after a failed block count; the safer solution is to reuse the existing delivery-target contract and update visible-delivery accounting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve bound conversation targets before routing:** `src/auto-reply/reply/bind-aware-dispatch.ts:100`
  - body: `deliverViaSessionBindings` passes the bound `conversationId` directly as `routeReply.to` and drops `parentConversationId`/`threadId`. Topic bindings need the existing delivery-target resolver so Telegram receives the parent chat as `to` and the topic as `threadId`; otherwise the fallback can fail or route to the wrong surface for the supergroup topic case this PR is meant to fix.
  - confidence: 0.91
- **[P2] Avoid final fallback after recovered block delivery:** `src/auto-reply/reply/dispatch-acp-delivery.ts:458-461`
  - body: The new fallback triggers on any prior failed count, but `settleVisibleText` still treats earlier block/final failures as visible-delivery failure. If the first post-teardown block fails and later blocks are bind-routed successfully, `finalizeAcpTurnOutput` can send the accumulated visible block text again as a final fallback, duplicating already-delivered content.
  - confidence: 0.82
