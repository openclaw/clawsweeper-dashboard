## Summary

## What This Changes

The branch adds a bind-aware ACP fallback for spawn-child replies after dispatcher failures, changes final-only ACP text flushing to final replies, adds focused tests, and edits CHANGELOG.md.

## Best Possible Solution

Keep the bugfix direction, but rebase onto current main, route bindings through routeFromConversationRef/routeToDeliveryFields, separate bind-routed visible accounting from dispatcher-failure fallback accounting, remove the direct changelog edit, and require real Telegram proof.

## Reproduction Assessment

Yes at source level, but not with live proof in this read-only review. Current main still routes direct ACP child output through the parent dispatcher while ReplyDispatcher records async transport failures in getFailedCounts, and the PR's own failing-path test models the reported failure.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix must reuse the existing conversation route projection contract and prevent successful bind-routed blocks from triggering the failed-dispatcher final fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route bound replies through the channel target resolver:** `src/auto-reply/reply/bind-aware-dispatch.ts:97-105`
  - body: The fallback sends `to` as `binding.conversation.conversationId` and drops `parentConversationId`. Telegram topic bindings resolve through the channel contract to the parent chat plus `threadId`; bypassing that projection means the reported supergroup topic can still fail or misroute.
  - confidence: 0.92
- **[P1] Exclude bind-routed blocks from dispatcher-failure replay:** `src/auto-reply/reply/dispatch-acp-delivery.ts:469-475`
  - body: After a bind-aware fallback succeeds, this call returns `delivered = true` for a block even though no dispatcher send was queued. `settleVisibleText` still compares against the prior dispatcher `failedCounts.block/final`, so a first failed block followed by successful bind-routed blocks can be repeated as the final accumulated fallback.
  - confidence: 0.86
