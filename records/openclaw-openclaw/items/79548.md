## Summary

Keep open: the PR targets a plausible ACP/Telegram delivery bug, but the current diff can drop later ACP payloads after an unrelated dispatcher failure and can miss multi-account bindings; it also still lacks required real Telegram behavior proof.

## What This Changes

The PR adds a bind-aware ACP delivery fallback through session bindings, updates final-only ACP delivery assertions, adds spawn-child delivery regression coverage, and adds a changelog entry.

## Best Possible Solution

Keep the scoped fallback direction, but make binding fallback opportunistic rather than a hard drop path, resolve requester accounts through the same configured-default account contract, then require redacted live Telegram proof before merge.

## Reproduction Assessment

Unclear for a live reproduction: current source supports the async failedCounts failure model and the PR simulates it, but no real Telegram before/after proof is attached. The review findings are source-reproducible from the PR head and current dispatcher/binding contracts.

## Solution Assessment

No: the direction is reasonable, but this patch should not hard-drop later payloads when no binding resolves and should use the configured default account for requester matching. A narrower maintainable fix would try the binding route only when it can resolve or succeed, otherwise preserve the normal dispatcher path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fall back to the dispatcher when no binding resolves:** `src/auto-reply/reply/dispatch-acp-delivery.ts:464-475`
  - body: This branch runs after any prior dispatcher failure, but deliverViaSessionBindings returns false for no-active-binding/no-requester-match and the code never attempts the original dispatcher. createReplyDispatcher catches one delivery error and keeps the send chain usable, so a transient ACP send failure without a matching binding now makes all later payloads hard-drop. Only bypass the dispatcher when a binding route is actually available/succeeds, or fall back to the normal send path.
  - confidence: 0.88
- **[P2] Use the configured default account for requester matching:** `src/auto-reply/reply/dispatch-acp.ts:372-378`
  - body: When ctx.AccountId is absent this hard-codes requesterConversation.accountId to default. Other ACP binding code resolves cfg.channels[channel].defaultAccount first, and the bound router requires the requester account to match the binding account. In a multi-account Telegram config with a non-default defaultAccount, the fallback fails closed and drops the payload even though the session has a valid binding.
  - confidence: 0.86
