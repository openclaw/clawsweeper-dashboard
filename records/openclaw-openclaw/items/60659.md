## Summary

Keep open. Current main and latest release still only expose global, chat-type, and channel-scoped reset policy, so per-agent or per-binding reset remains a real product/config decision with this issue as the canonical tracker.

## What This Changes

Request per-agent or per-binding session reset policy so one-shot workflows can reset independently of conversational agents.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence, aligned types/schema/help/docs, and regression tests covering binding or agent overrides over channel/type/global defaults.

## Reproduction Assessment

Yes, at source level: current types, strict schemas, docs, and runtime resolver show no accepted or applied per-agent or per-binding reset-policy shape.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should be one scoped precedence model rather than parallel agent and binding shortcuts.
