## Summary

Keep open. Current main and v2026.6.10 still only expose global, chat-type, and channel-scoped reset policy, so per-agent or per-binding reset remains a real public config and precedence decision.

## What This Changes

The issue requests a per-agent or per-binding session reset policy so independent workflows can reset sessions without changing global, type-wide, or channel-wide reset behavior.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence and aligned types, schema, help, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: current types, strict schemas, docs, and runtime reset resolution show no accepted or applied per-agent or per-binding reset-policy shape.

## Solution Assessment

Unclear until maintainers choose the public API shape; the maintainable fix should be one scoped precedence model rather than parallel agent and binding shortcuts.
