## Summary

Keep open. Current main still has only global, chat-type, and channel reset policy inputs; agent entries and route-binding session blocks do not accept reset policies, so this remains a scoped config feature that needs maintainer API and precedence design.

## What This Changes

The issue requests a new per-agent or per-binding session reset policy so one-shot workflows can reset independently from conversational agents.

## Best Possible Solution

Add an additive scoped reset-policy contract with explicit precedence, aligned types, strict schemas, generated config metadata, public docs, and focused runtime tests coordinated with the related per-group tracker.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that current main does not accept the proposed agent or binding reset-policy shapes and cannot select reset policy from those scopes.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should define one scoped precedence model instead of adding an isolated shortcut.
