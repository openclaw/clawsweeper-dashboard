## Summary

Keep open. Current main and the latest release still expose reset policy only at global, chat-type, and channel scope; agent entries and route-binding session blocks do not accept a reset policy, so this remains a valid scoped session lifecycle feature that needs maintainer config/API precedence design.

## What This Changes

Request adding reset-policy overrides at the agent and/or route-binding scope so one-shot workflows can start fresh without changing global or channel-wide session behavior.

## Best Possible Solution

Define one additive scoped reset-policy contract with explicit precedence, then align config types, strict schemas, generated metadata/help, public docs, and focused runtime tests across agent and binding scopes while coordinating the overlap with https://github.com/openclaw/openclaw/issues/8122.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main does not accept or apply the proposed agent- or binding-scoped reset policy shapes.

## Solution Assessment

Unclear until maintainers choose the public API shape and precedence model. The requested direction is reasonable, but the maintainable fix should be a single scoped resolver rather than isolated agent-only or binding-only shortcuts.
