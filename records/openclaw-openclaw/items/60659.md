## Summary

Keep open. Current main and v2026.6.8 still only resolve session reset policy at global, chat-type, and channel scope; per-agent and per-binding reset policy remains a real public config/API decision.

## What This Changes

The issue requests agent-level or binding-level session reset policy so different workflows can choose independent reset behavior instead of sharing global, type, or channel policy.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence, aligned TypeScript types, strict schema, generated config metadata/help, public docs, and focused runtime tests.

## Reproduction Assessment

Yes at source level: current main's types, strict schemas, docs, and runtime resolver show no accepted or applied per-agent/per-binding reset-policy shape. I did not run a live config parse because this read-only review avoided commands that could write artifacts.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should be one scoped precedence model rather than parallel agent and binding shortcuts.
