## Summary

Keep open. Current main still only supports session reset policy at the global, chat-type, or channel level; agent entries and route-binding session blocks remain strict and do not accept the requested scoped reset policy.

## What This Changes

Request per-agent or per-binding session reset configuration so one-shot workflow agents can start fresh sessions without affecting conversational agents.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence, aligned TypeScript types, strict Zod schemas, generated config metadata/help, public docs, and focused runtime tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main accepts only global/type/channel reset policy and strict schemas reject the proposed per-agent or per-binding reset shapes.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public API shape; the direction is reasonable, but the maintainable fix should define one scoped precedence model rather than an isolated shortcut.
