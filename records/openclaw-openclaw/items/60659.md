## Summary

Keep open. Current main and v2026.6.10 still only expose global, chat-type, and channel-scoped session reset policy, so per-agent or per-binding reset remains a real product/config decision with this issue as the canonical tracker.

## What This Changes

Request per-agent or per-binding session reset configuration so different workflows can choose fresh-session or long-lived-session behavior independently.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence, aligned types/schemas/help/docs, and focused runtime tests for binding or agent overrides over channel/type/global defaults.

## Reproduction Assessment

Yes, at source level: current types, strict schemas, docs, and runtime resolver show no accepted or applied per-agent or per-binding reset-policy shape. This is a missing feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should be one scoped precedence model rather than parallel agent and binding shortcuts.
