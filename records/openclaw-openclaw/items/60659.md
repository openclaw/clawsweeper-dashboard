## Summary

Keep open. Current main still models session reset policy only at global, chat-type, and channel scopes; agent entries and route-binding session config do not accept reset policy, so this remains a feature/config design request rather than cleanup-close material.

## What This Changes

The issue requests per-agent or per-binding session reset configuration so different workflows can choose independent session lifecycles.

## Best Possible Solution

Add an additive scoped reset-policy contract for agents and/or route bindings with explicit precedence, schema/types/help/docs updates, and focused runtime tests coordinated with the per-group tracker.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that current main does not accept the proposed agent or binding reset-policy shapes and cannot select reset policy from those scopes.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should define one precedence model instead of adding an isolated shortcut.
