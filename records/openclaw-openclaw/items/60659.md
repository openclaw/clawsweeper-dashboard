## Summary

Keep open. Current main still exposes reset policy only at global, chat-type, and channel scopes; route bindings only support `dmScope`, and no merged or canonical item now owns the broader per-agent/per-binding reset policy request.

## What This Changes

Request to add agent- or route-binding-level session reset policy so isolated workflows can reset per message or after short idle periods without changing other agents.

## Best Possible Solution

Define an additive route-aware reset-policy override contract after agent and binding resolution, document precedence, and cover config schemas, generated metadata, runtime resolution, routing, and session tests while preserving existing defaults.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that strict schemas do not accept `agents.list[].session.reset` or `bindings[].session.reset`, and runtime reset resolution has no effective agent or binding input.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is a generic routed reset-policy resolver with explicit precedence, not an ACP-only workaround or a channel-specific patch.
