## Summary

Keep open. Current main and the latest release still only resolve session reset policy at global, chat-type, or channel scope; per-agent and per-binding reset policy remains a real public config/API decision.

## What This Changes

Request per-agent or per-binding session reset configuration so one-shot workflows can avoid carrying transcript state across unrelated inbound messages.

## Best Possible Solution

Define one additive scoped reset-policy model with explicit precedence, aligned TypeScript types, strict Zod schema, generated config metadata/help, public docs, and focused runtime tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the current types, strict schemas, docs, and runtime resolver show no accepted or applied per-agent/per-binding reset-policy shape.

## Solution Assessment
