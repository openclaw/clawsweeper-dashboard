## Summary

Keep open. Current main still exposes session reset policy only at global, chat-type, and channel scopes; neither agent entries nor route bindings accept a reset-policy block, so this remains a valid feature/config design request.

## What This Changes

Request per-agent or per-route-binding session reset policy so one-shot workflow agents can avoid carrying conversation history while other agents keep continuity.

## Best Possible Solution

Add an additive scoped reset-policy contract for agents and/or route bindings with explicit precedence, schema/types/help/docs updates, and focused runtime tests coordinated with the open per-group tracker.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that current main does not accept the proposed agent or binding reset-policy shapes and cannot select reset policy from those scopes.

## Solution Assessment

Unclear until maintainers choose the public API shape. The requested direction is reasonable, but the maintainable fix should define one precedence model instead of adding an isolated shortcut.
