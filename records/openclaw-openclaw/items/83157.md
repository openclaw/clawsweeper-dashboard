## Label Justifications

- `impact:session-state`: The reported failure is session/transcript state drift during reseed, where stored recent turns are not carried into the recovered CLI session prompt.

## Summary

Keep open: current main still prefix-slices over-cap CLI reseed history, and the open linked PR is the right implementation path for this valid session-state bug.

## What This Changes

The issue requests changing CLI reseed prompt truncation so over-cap rendered history keeps the most recent turns instead of the oldest prefix.

## Best Possible Solution

Review and land the linked PR or an equivalent tail-slice regression-tested fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass chronological rendered history longer than `maxHistoryChars` with a latest-turn marker; current main slices from the head, so the latest marker is absent from `<conversation_history>`.

## Solution Assessment

Yes. Keeping the recent tail and moving the marker to describe older dropped turns is the narrow maintainable fix; the linked PR implements that direction with regression tests.
