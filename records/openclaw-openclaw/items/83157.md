## Summary

Keep open: current main still has the prefix slice, and an open same-author PR is already the right implementation path for this valid session-reseed bug.

## What This Changes

The issue requests that CLI reseed history truncation keep the most recent rendered transcript tail instead of dropping it when `maxHistoryChars` is exceeded.

## Best Possible Solution

Review and land the paired PR, then close this issue once the suffix-slice behavior and regression test are on main.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: pass chronological rendered history longer than `maxHistoryChars` with a latest marker, and current main slices from the head so the marker is absent. I did not run a live CLI reseed in this read-only review.

## Solution Assessment

Yes, keeping the recent tail and moving the marker to describe older dropped turns is the narrow maintainable fix. The open paired PR implements that direction, so the issue should stay open until that fix is merged or replaced.
