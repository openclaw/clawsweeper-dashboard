## Summary

The branch is a focused attempt to improve short-window throttling recovery, but it is not merge-ready: the new classifier lets generic `retry-after` override explicit daily/weekly/monthly or usage-limit wording, so terminal quota failures can spend an extra 60 seconds on the same unavailable provider before existing profile/model fallback. The PR also lacks required after-fix real-provider proof, and direct Codex inspection confirms retry/backoff is a dependency-owned concern that must be accounted for rather than assumed.

## What This Changes

The PR adds three deterministic 10s/20s/30s same-provider/model/profile retries for classified short-window assistant-stage rate limits before existing auth-profile rotation or model fallback.

## Best Possible Solution

Give explicit long-window quota, usage-limit, and subscription signals unconditional precedence; limit same-model retry to unambiguous RPM/TPM/per-minute throttles, account for dependency retry budgets, and prove both recovery and immediate terminal escalation in a real run.

## Reproduction Assessment

Yes at source level: use a rate-limit assistant error containing both `daily quota exceeded` and `retry-after 3600`; the branch classifier selects the same-model wait before existing fallback. A real-provider reproduction has not been supplied.

## Solution Assessment

No in its current form. A bounded same-model retry can be the right fix for explicit minute-scale throttles, but generic `retry-after` must not override terminal quota wording, and the total cross-layer retry budget must be demonstrated.
