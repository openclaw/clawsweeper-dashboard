## Summary

The PR is still useful, but it is not merge-ready: the new classifier treats a bare `429 quota exceeded` message as short-window because `429` itself satisfies the short-window hint, contradicting the stated escalation policy. The contributor also has not supplied real-provider proof for the added retry latency.

## What This Changes

The PR adds three deterministic same-provider/model retries of 10, 20, and 30 seconds for assistant errors classified as short-window rate limits before auth-profile rotation or model fallback.

## Best Possible Solution

Restrict same-model retries to explicit minute-scale signals such as RPM, TPM, per-minute, or valid retry timing—not a bare 429—then prove the total transport-plus-runner latency, abort behavior, recovery, and eventual fallback with a real provider trace.

## Reproduction Assessment

Yes. From the branch source, `429 quota exceeded` matches both the long-window expression and the short-hint expression, so the long-window exclusion does not apply and the same-model retry path is taken.

## Solution Assessment

No, not yet. Same-model waiting can be a valid fix for explicit minute-scale throttling, but a bare HTTP status must not override quota-exhaustion wording, and the combined provider/runner retry budget needs real behavior proof.
