## Summary

This is a useful reliability direction, but the patch is not merge-ready: it reparses provider text at the failover consumer, treats any `retry-after` wording as a short window, and lacks required real-provider proof. Current main does not already implement the behavior, so the PR should remain open for a bounded redesign and validation.

## What This Changes

The PR adds three abortable same-model retries at 10, 20, and 30 seconds for assistant errors classified as short-window rate limits before auth-profile rotation or model fallback.

## Best Possible Solution

Extend the canonical provider-error result with a closed rate-limit subtype or parsed retry timing, pass that prepared fact into failover, and cap same-model waiting from the actual short window before proving both recovery and hard-limit escalation on real providers.

## Reproduction Assessment

No, not at high confidence: source inspection and unit tests establish the proposed branch, but the PR does not reproduce a real provider short-window failure and recovery on current main.

## Solution Assessment

No; the retry policy is plausible, but reparsing free-form text in the failover consumer is not the narrowest maintainable solution, and unparsed `retry-after` duration makes the short-window decision unsafe.
