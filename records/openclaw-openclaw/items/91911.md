## Summary

This PR is still useful but not merge-ready: the current head keeps the two concrete failover defects from the prior review and still lacks real-provider proof for the new default retry behavior.

## What This Changes

The branch adds bounded same-provider/model/profile retries for assistant-stage short-window rate-limit errors before auth-profile rotation or model fallback.

## Best Possible Solution

Keep the same-model retry idea only if classification is bounded to proven short windows, traces represent the actual retry action, and redacted real-provider proof covers recovery and hard-quota escalation.

## Solution Assessment

No, not in its current form. Same-model retry is a plausible repair, but arbitrary `Retry-After` matching and inaccurate trace results make the implementation broader and less observable than the maintainable fix.

Full review comments:

- **[P2] Parse Retry-After before treating it as a short window:** `src/agents/embedded-agent-runner/run/assistant-failover.ts:41-43`
  - body: `retry-after` is accepted as a short-window hint without parsing the interval. A provider error like `429 rate_limit_exceeded; Retry-After: 3600` will therefore wait and retry the same unavailable model three times before profile/model fallback. Parse and bound the duration, or require an explicit RPM/TPM/per-minute signal when no structured short interval is available.
  - confidence: 0.96

  - body: The new `same_model_rate_limit` retry kind falls through to `rotate_profile`, although this path intentionally keeps the same provider, model, and profile. That makes execution traces report a rotation that never happened, so diagnostics cannot tell whether the new retry policy actually ran.
  - confidence: 0.98
