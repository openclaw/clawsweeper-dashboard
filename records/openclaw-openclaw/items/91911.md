## Summary

Keep open: the same-layer retry idea is plausible, but the PR leaves the new rate-limit retry budget run-scoped and never resets it after recovery, so later independent short-window 429s in the same embedded run can skip the intended same-model retry path. The external PR also has only test/lint verification, not real after-fix behavior proof.

## What This Changes

The PR adds bounded same-model assistant-stage retries for short-window rate-limit messages before profile rotation/model fallback, with helper and failover tests.

## Best Possible Solution

Keep the assistant failover retry path, but scope or reset the same-model rate-limit budget to consecutive short-window failures, preserve quota/profile fallback behavior, add focused regression coverage for the reset case, and require redacted real-run proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection shows current main immediately rotates/escalates assistant-stage rate_limit failures, and the PR uses mocked tests rather than a live provider short-window 429 run.

## Solution Assessment

No as submitted; the assistant failover layer is a reasonable place for this behavior, but the retry budget needs to be scoped or reset after recovery and the live 429 recovery path needs real proof.
