## Summary

Keep this PR open. The core repair direction is useful and addresses the reported trailing assistant-error loop, but the current tail scan still misses poisoned transcripts when metadata-only rows sit after the errored assistant, and the branch is missing required release-note/changelog cleanup.

## What This Changes

The PR updates session-file repair to drop trailing errored assistant entries, adds a dropped-tail counter and regression tests, and documents the new transcript-hygiene rule.

## Best Possible Solution

Revise the repair pass to reason over the replay-visible tail using SessionManager semantics: preserve metadata-only rows, treat them as transparent when finding a trailing errored assistant, reconnect preserved metadata to the dropped entry's parent, keep replay-visible summary/custom-message rows intact, and add coverage plus changelog/docs updates before the changed gate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session JSONL chain user -> errored assistant -> model_change or thinking_level_change; this PR stops at the metadata row, while SessionManager uses that row as the leaf and does not append it as a replay message, leaving the assistant as the final model-visible turn.

## Solution Assessment

No. The repair-file boundary is the narrowest maintainable place to heal persisted poisoned transcripts, but the current implementation is not the best form because it treats all non-message tail rows as blockers instead of distinguishing transparent metadata from replay-visible entries and preserving the tree correctly.
