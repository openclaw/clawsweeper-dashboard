## Summary

Keep open: current main has structured all-model exhaustion handling and safer built-in fallback text, but it still has no supported configurable user-facing message for billing or rate-limit exhaustion. The related open plugin-hook work does not change default user-facing reply behavior or add a `messages.*` config override.

## What This Changes

The issue requests a `messages` config key for a custom friendly reply when all configured models are exhausted by billing or rate-limit caps.

## Best Possible Solution

Add a documented optional `messages` key for terminal billing/rate-limit model exhaustion, applied only after structured fallback classification and leaving current built-in sanitized messages as the default.

## Reproduction Assessment

Yes for the missing config surface: source inspection shows the strict `messages` schema and type omit this key, while the reply path still selects built-in text from structured fallback reasons. This is a feature/config gap rather than a broken existing contract.

## Solution Assessment

Yes in direction, but it needs maintainer product approval for the public config contract. A narrow documented `messages` override tied to terminal billing/rate-limit exhaustion is the maintainable path; plugin hooks alone would not provide the requested built-in user-facing config.
