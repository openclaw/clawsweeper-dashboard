## Summary

Keep open. Current main and the latest shipped release have structured fallback classification and safer built-in billing/rate-limit copy, but they still do not provide a public `messages.*` setting or plugin/API contract for operator-supplied all-models-exhausted text.

## What This Changes

The issue requests an operator-configurable user-facing message when every configured model is exhausted by expected billing, quota, or rate-limit failures.

## Best Possible Solution

Choose one public customization surface, then route billing/rate-limit all-models-exhausted replies through it with schema, exported types, docs/help, and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.5.20` have structured fallback summaries and fixed billing/rate-limit reply branches, while the strict messages schema and exported messages type contain no custom exhaustion-copy field.

## Solution Assessment

Unclear. A `messages.modelExhausted` key is plausible, but maintainers should decide whether core config, the pending terminal-failure hook path, or the broader provider-error disclosure policy is the durable solution.
