## Summary

Keep open. Current main has safer built-in billing and rate-limit fallback-exhaustion copy, but it still does not expose an operator-configurable user-facing message in the strict config schema, exported message type, config help, or docs.

## What This Changes

Requests a public `messages.modelExhausted`-style config key for friendly user-facing copy when all configured models are exhausted by billing or rate-limit failures.

## Best Possible Solution

Choose one public customization surface, then wire it through the terminal fallback-exhaustion reply path with schema, exported types, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Current-main source inspection shows the fallback-exhaustion path is classifiable but uses hard-coded billing/rate-limit text, while the strict message config contract has no custom-copy field; I did not run a live quota-exhaustion provider scenario.

## Solution Assessment

Unclear. A `messages.modelExhausted` key is plausible, but maintainers should first decide whether core config or a terminal-failure plugin hook is the cleaner long-term customization surface.
