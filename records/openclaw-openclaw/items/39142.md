## Summary

Keep open. Current main still lacks a supported `messages.modelExhausted` or equivalent override, the strict `messages` schema would reject the proposed key, and the before-reply fallback exhaustion path still chooses built-in billing, rate-limit, or generic text. The adjacent open plugin-hook PR is complementary and explicitly does not change default user-facing fallback messaging.

## What This Changes

The issue requests a configurable user-facing message, such as `messages.modelExhausted`, for all-model fallback exhaustion caused by billing or rate-limit limits.

## Best Possible Solution

Ship a supported optional `messages.modelExhausted` string config key at the before-reply terminal fallback collapse point, scoped to structured billing/rate-limit exhaustion, while leaving plugin observability hooks as complementary rather than required for ordinary operators.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can inject a `FallbackSummaryError` with billing/rate-limit attempts into `runAgentTurnWithFallback`; current tests already exercise adjacent pure billing and mixed-cause paths and show there is no configured-message branch.

## Solution Assessment

Yes. A first-class config key in the existing message config surface is the narrowest maintainable fix because the failure path already has structured attempt reasons and the current plugin hook work does not change default visible replies.
