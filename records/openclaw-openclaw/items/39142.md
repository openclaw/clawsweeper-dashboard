## Summary

Keep open: current main has safer built-in fallback text for terminal billing/rate-limit exhaustion, but it still does not expose a supported configurable `messages.modelExhausted` override. I also did not find an open PR or canonical issue that implements this exact config surface.

## What This Changes

The issue requests a configurable `messages.modelExhausted` user-facing reply when terminal model fallback exhaustion is caused by billing or rate-limit failures.

## Best Possible Solution

Add a documented optional `messages.modelExhausted` string and apply it only to terminal fallback exhaustion caused by billing/rate-limit style reasons, preserving current sanitized built-in messages when unset or unrelated.

## Reproduction Assessment

Yes. Source inspection on current main shows the proposed key is absent from the strict `messages` schema, while terminal `FallbackSummaryError` handling still selects built-in billing, rate-limit, or generic text.

## Solution Assessment

Yes in direction. A first-class `messages` config key is the narrowest maintainable path because the reply path already has structured fallback reasons and `agent_end` is observation-only after the turn.
