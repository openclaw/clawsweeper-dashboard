## Summary

Keep this PR open: it targets a real current-main bug in OpenAI-completions model-param propagation, but the patch currently introduces a provider-payload alias regression and the supplied real-behavior proof does not demonstrate the embedded Pi path that exposed the bug.

## What This Changes

Adds a shared helper that applies OpenAI-completions model params to direct transport payloads and embedded Pi payload patching.

## Best Possible Solution

Use one shared OpenAI-completions payload normalizer that translates or reserves existing OpenClaw stream-param aliases before generic passthrough, applies the resolved token cap after generated defaults, and adds focused direct plus embedded payload coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows direct OpenAI-completions payload construction ignores model.params, and the embedded extra-params path does not apply model.params.max_completion_tokens to the final payload.

## Solution Assessment

No. Applying resolved params is the right direction, but verbatim copying must normalize or reserve documented aliases such as maxTokens/topP and should be covered by focused payload tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize stream-param aliases before copying params:** `src/agents/openai-completions-model-params.ts:58`
  - body: This helper copies every non-reserved `model.params` key directly into the OpenAI-completions JSON. Existing docs and runtime code support config params like `maxTokens` and `topP` as OpenClaw stream options, so resolved models carrying those params would now send non-OpenAI fields such as `maxTokens`/`topP` alongside the real wire fields. Please normalize these aliases or keep them reserved before the generic passthrough.
