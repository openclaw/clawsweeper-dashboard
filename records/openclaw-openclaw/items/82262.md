## Summary

Keep open for maintainer handling: the linked bug is real, but this draft should not merge as-is because the helper can overwrite request-specific token/runtime overrides, it lacks inspectable real-behavior proof, and https://github.com/openclaw/openclaw/pull/82244 already covers the same bug with tests and proof.

## What This Changes

The branch adds an OpenAI-completions model-param helper and applies it to direct request construction and embedded Pi payload patching.

## Best Possible Solution

Use the canonical alias-normalization approach from https://github.com/openclaw/openclaw/pull/82244 or an equivalent fix: normalize token aliases before request construction, preserve explicit request overrides, and keep regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main resolves params but only forwards camelCase `maxTokens` in the embedded wrapper and `options.maxTokens || model.maxTokens` in direct completions; I did not run the live QQBot/DashScope path.

## Solution Assessment

No. This patch applies model params as a late payload overwrite, which breaks request precedence and does not normalize `maxTokens`; the narrower maintainable fix is alias normalization before final payload construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve request overrides when applying model params:** `src/agents/openai-completions-model-params.ts:58`
  - body: The helper is called after request options have already shaped the payload, and the embedded wrapper patches the final payload after stream options are merged. Because this assignment unconditionally writes model defaults back onto the payload and deletes the opposite token alias, `params.max_completion_tokens = 64000` will overwrite an explicit per-request cap instead of preserving request precedence.
  - confidence: 0.9
- **[P2] Normalize maxTokens instead of forwarding it:** `src/agents/openai-completions-model-params.ts:58`
  - body: Agent docs advertise `params.maxTokens` as a supported model param. This helper only treats `max_completion_tokens` and `max_tokens` as aliases, so `params.maxTokens` reaches this assignment as a raw top-level `maxTokens` field in an OpenAI-compatible request instead of becoming the provider's token-cap wire field.
  - confidence: 0.86
