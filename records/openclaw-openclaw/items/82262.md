## Summary

Keep open. The PR targets a real source-reproducible bug, but the current branch still has a blocking payload-normalization regression and the supplied proof does not demonstrate the embedded Pi path or final token cap.

## What This Changes

Adds a shared OpenAI-completions model-param helper, calls it from direct transport and embedded Pi payload patching, and adds focused payload regression tests.

## Best Possible Solution

Use one shared OpenAI-completions normalizer that translates OpenClaw aliases before generic passthrough, then require redacted embedded-path proof showing the final payload cap.

## Reproduction Assessment

Yes. Source inspection shows current main ignores model.params in direct OpenAI-completions payload construction and the embedded wrapper does not apply model.params.max_completion_tokens to the final payload; I did not run a live provider repro in this read-only review.

## Solution Assessment

No. Applying resolved params in both paths is the right direction, but the current helper needs to normalize or reserve documented aliases such as maxTokens/topP before copying arbitrary params into the provider payload.
