## Summary

Keep open: current main still has a keyed-state trust gate that can reject a non-bundled WhatsApp plugin without trusted official install provenance, and the Docker-specific config/provenance behavior is not fully covered by the broader open community keyed-store request.

## What This Changes

The issue reports that the external WhatsApp plugin starts natively but exits in Docker when its plugin runtime calls keyed storage.

## Best Possible Solution

Keep this open until maintainers decide and document the supported Docker path for official external channel plugins, then either make `openclaw plugins install`/install records work reliably in Docker or link this to a narrower canonical plugin-runtime capability fix.

## Reproduction Assessment

Unclear.
