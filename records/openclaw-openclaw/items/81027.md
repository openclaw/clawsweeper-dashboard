## Summary

Keep open: the PR targets a plausible gateway abort bug, but the patch is not mergeable because the hard-kill listener is not wired to the actual model-fetch abort signal, it changes runId semantics used for gateway event routing, and the external proof is only mocked test output.

## What This Changes

The PR exports the dispatcher destroy helper, wires fetchWithSsrFGuard to destroy the active dispatcher on abort, changes retry run IDs, and adds an OpenAI HTTP abort test.

## Best Possible Solution

Revise the PR to compose the actual fetch abort signal, keep runId stable across retries, and add targeted dispatcher teardown coverage plus redacted real behavior proof from an OpenClaw run.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection shows a plausible abort path and the PR includes before logs, but the after-fix evidence is mocked tests rather than a real Google/fallback run.

## Solution Assessment

No. The narrow maintainable fix is to hard-kill the dispatcher from the composed signal actually used by fetch, keep runId as the stable external routing key, and use a separate attempt identifier only for diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire hard-kill teardown to the actual fetch abort signal:** `src/infra/net/fetch-guard.ts:373-375`
  - body: `fetchWithSsrFGuard` listens only to the top-level guard signal, but the model transport passes the request abort signal inside `RequestInit` and calls the guard without `params.signal`. In that path the dispatcher can still take the normal close path when the real request is aborted, so this does not reliably fix the reported zombie socket. Compose the timeout signal with `init.signal` and use that same signal for fetch and teardown.
  - confidence: 0.88
- **[P1] Keep retry attempts on the original runId:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: Changing retry attempts to emit under `${runId}-retry-N` breaks gateway state keyed by the original idempotency key: `agent.wait`, abort state, agent-run context, and subscription events all use the original runId. Retried attempts would publish terminal/tool events under an unregistered id while clients keep waiting on the original run. Keep `runId` stable and add a separate attempt id for tracing if needed.
  - confidence: 0.91
