## Summary

Keep open: the guarded-fetch teardown direction may be useful, but the current PR still changes retry runIds in a way that conflicts with current run-scoped event and prompt-cache contracts, and the supplied runtime logs do not prove dispatcher teardown or fallback survival after the abort window.

## What This Changes

The PR adds abort-triggered dispatcher destruction to guarded fetch, exports a broader dispatcher teardown helper, changes embedded retries to use suffixed runIds, and adds a gateway abort test.

## Best Possible Solution

Keep the network-level abort teardown, keep runId stable across retries, and add focused dispatcher-destruction or fallback-survival coverage plus stronger redacted runtime proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection and supplied logs support a plausible current-main path: guarded fetch closes dispatchers gracefully while Undici exposes destroy as the abrupt abort primitive, but the delayed fallback crash was not reproduced here.

## Solution Assessment

No. The network teardown boundary is the narrow maintainable direction, but changing retry attempt runIds is not part of that fix; use a separate attempt id/index if tracing needs disambiguation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep retry attempts on the original runId:** `src/agents/pi-embedded-runner/run.ts:1344`

Keep runId stable and add a separate attempt id if needed.
