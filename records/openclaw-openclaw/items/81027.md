## Summary

Keep open: the guarded-fetch socket teardown direction is plausible, but the PR is not mergeable because it changes retry run IDs that gateway routing treats as stable, and the external after-fix proof is only mocked test output.

## What This Changes

The PR exports `destroyDispatcher`, wires `fetchWithSsrFGuard` to destroy the active undici dispatcher on abort, adds a gateway abort test, and suffixes embedded retry attempt run IDs.

## Best Possible Solution

Keep the hard-kill behavior inside the guarded-fetch network primitive, preserve stable user-request run IDs across retries, add focused dispatcher-teardown regression coverage, and require redacted real after-fix evidence before merge.

## Reproduction Assessment

No high-confidence full reproduction is established here. Source inspection and the PR’s logs make the current-main socket cleanup concern plausible, but the 97-second Google fallback failure was not reproduced against current main and the after-fix proof is mocked.

## Solution Assessment

No. The fetch-guard destroy-on-abort approach may be the right narrow fix, but changing retry `runId` is not safe because gateway routing and abort state are keyed by that ID; use a separate attempt identifier if retry tracing is needed.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep retry events on the original run ID:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: `runEmbeddedPiAgent` is inside the retry loop, and this change sends later attempts under `runId-retry-N`. Gateway chat state registers and broadcasts under the original client run ID, then looks up event routing by `evt.runId`, so retry events can miss the original chat mapping and appear as a separate run or lose abort state. Keep `runId` stable and add a separate attempt identifier if tracing needs one.
  - confidence: 0.9

  - body: The new test only checks that the agent abort signal becomes aborted after a client disconnect, which is already covered by the existing non-streaming disconnect test. It would pass without the new `destroyDispatcher` integration, so it does not guard the zombie-socket fix; add a focused fetch/undici seam assertion that the active dispatcher is destroyed on abort.
  - confidence: 0.84
