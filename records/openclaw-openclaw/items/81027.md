## Summary

Keep open: the network-level abort teardown is a plausible bug fix, but this PR is not ready because it still changes retry run identity and the supplied proof/test coverage do not demonstrate the claimed dispatcher teardown or fallback survival.

## What This Changes

The PR adds abort-triggered dispatcher destruction to guarded fetch, exports dispatcher destruction, alters embedded retry run IDs, and adds a gateway client-disconnect test.

## Best Possible Solution

Keep the fix at the guarded-fetch dispatcher teardown boundary, preserve stable runId for the user request, add focused dispatcher-destruction or fallback-survival coverage, and require stronger redacted runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection and the supplied before logs make the reported zombie-socket/fallback failure plausible because current main uses graceful dispatcher close while Undici distinguishes graceful close from abrupt destroy.

## Solution Assessment

No. The network teardown boundary is the narrow maintainable direction, but changing retry runId is not part of that fix and the added test should prove dispatcher teardown or fallback survival rather than only another client-disconnect abort signal.
