## Summary

Keep open: the PR targets a plausible gateway abort/fallback bug, but the current patch still misses the model transport abort path and changes a stable gateway routing key, so it should not merge as-is.

## What This Changes

The PR exports `destroyDispatcher`, destroys the active SSRF-guarded dispatcher on abort, changes embedded retry run IDs, and adds a gateway HTTP abort test.

## Best Possible Solution

Keep the hard-kill fix focused in the network primitive, compose the actual fetch abort signal with timeout handling, preserve `runId` as the stable routing key, and add targeted dispatcher teardown coverage plus redacted real runtime proof.

## Reproduction Assessment

No high-confidence live reproduction of the original Google fallback failure was established. Source inspection does show a credible current-main path through `RequestInit.signal` and stable `runId` routing that the PR must handle.

## Solution Assessment

No. The hard-kill direction is plausible, but this implementation should compose the real fetch abort signal and keep `runId` stable, using a separate attempt identifier only for diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire teardown to the composed fetch abort signal:** `src/infra/net/fetch-guard.ts:373-375`
  - body: The teardown listener is attached to the guard's top-level signal, but model transport aborts arrive on `RequestInit.signal` and this caller does not pass `params.signal`. Compose `currentInit.signal` with the timeout/guard signal and use that same signal for fetch and dispatcher teardown, or this can still miss the abort path the PR is meant to fix.
  - confidence: 0.88
- **[P1] Keep retries on the original runId:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: `runId` is registered and waited on as the gateway routing key, while embedded subscriptions and agent events emit the supplied value. Suffixing retry attempts with `-retry-N` makes later attempt events look like a different unregistered run, so keep `runId` stable and add a separate attempt id if needed.
  - confidence: 0.91
