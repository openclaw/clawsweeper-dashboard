## Summary

This PR should stay open because the narrow abort-time dispatcher teardown is plausible and useful, but the current branch still changes retry run identity in a way that can break run-scoped state. The supplied EC2 logs are real runtime evidence, but they do not yet show dispatcher destruction, fallback survival, or the absence of a delayed upstream abort after the relevant wait window.

## What This Changes

The PR adds abort-triggered dispatcher destruction to guarded SSRF fetch cleanup, exports a more permissive `destroyDispatcher`, changes embedded retry attempts to suffixed runIds, and adds a gateway abort test.

## Best Possible Solution

Keep the abort teardown fix at the network primitive, preserve the original runId across retries, and add focused coverage or runtime proof that an aborted guarded fetch destroys the dispatcher without later disrupting fallback execution.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection and supplied before logs make the current-main failure plausible because guarded fetch uses graceful dispatcher close on cleanup, but the delayed fallback crash was not independently reproduced.

## Solution Assessment

No. Abrupt dispatcher teardown on abort is the right narrow boundary, but changing retry runIds is not necessary for that fix and introduces a separate behavioral contract risk.
