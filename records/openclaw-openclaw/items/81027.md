## Summary

Keep open: the dispatcher teardown direction is plausible, but the current PR still changes the stable runId contract and does not prove the hard-kill/fallback behavior it claims to fix.

## What This Changes

The branch adds hard-destroy-on-abort handling to guarded fetch dispatchers, exports `destroyDispatcher`, changes embedded retry `runId`s, and adds a gateway disconnect test.

## Best Possible Solution

Keep the narrow `fetchWithSsrFGuard` abort teardown, keep `runId` stable across retries, add focused dispatcher/fallback coverage, and require stronger redacted runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection plus supplied logs make the graceful-close abort path plausible, but the exact delayed fallback crash was not reproduced against current main.

## Solution Assessment

No. The network-level dispatcher destroy on abort is the maintainable boundary, but changing retry `runId`s is outside that fix and the added test does not cover the socket teardown behavior.
