## Summary

Keep open: the guarded-fetch socket teardown direction is plausible, but this PR should not merge as submitted because it changes retry attempt run IDs and can break gateway chat/event routing keyed to the original client-visible run ID.

## What This Changes

The PR exports dispatcher destruction, destroys guarded-fetch dispatchers when an AbortSignal fires, adds a gateway abort test, and changes embedded retry attempts to suffix runId values.

## Best Possible Solution

Keep the abort hard-kill fix at the guarded-fetch/dispatcher boundary, preserve the stable user/client runId across retries, and add a separate attempt identifier plus focused dispatcher-abort coverage if per-attempt tracing is needed.

## Reproduction Assessment

No, this review did not run a live current-main reproduction. Source inspection and the PR's before logs make the path credible: current fetchWithSsrFGuard has no AbortSignal listener that destroys the active dispatcher immediately.

## Solution Assessment

No, not as submitted. The network teardown direction matches Undici's close/destroy contract, but suffixing runId for retries breaks the existing gateway chat-run key contract and the test should cover dispatcher teardown directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Keep retry attempts on the original runId:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: Changing retry attempts to pass `${params.runId}-retry-N` into the backend breaks the existing chat/event routing contract. chat.send registers the active run under the original clientRunId, and server-chat only remaps events when `chatRunState.registry.peek(evt.runId)` matches; retry lifecycle/tool events with the suffixed ID can bypass that remap and surface or clean up under the wrong run. Keep `runId` stable and add a separate attempt id if tracing needs per-attempt identity.
