## Summary

Keep this PR open for author/maintainer follow-up: the network-abort direction is plausible, but the current diff introduces a retry runId routing regression that can prevent fallback/retry chat events from remapping back to the original run, and the supplied real behavior proof does not yet demonstrate the actual timeout/fallback socket path.

## What This Changes

The PR wires `fetchWithSsrFGuard` abort signals to destroy SSRF dispatchers, exports `destroyDispatcher` with a null/close fallback, changes embedded retry runIds, and adds a gateway HTTP disconnect test.

## Best Possible Solution

Keep the abort teardown work scoped to `fetchWithSsrFGuard`/`destroyDispatcher`, preserve the stable user-visible runId across retries, and add focused coverage that proves dispatcher destruction on abort without relying on runId mutation.

## Reproduction Assessment

Yes from source for the introduced regression: a chat send that reaches `runLoopIterations > 1` would emit attempt events under a suffixed runId while gateway chat state and run context remain keyed to the original client runId. I did not run a live repro because this was a read-only review.

## Solution Assessment

No. The dispatcher abort hook is the right boundary to explore, but changing `runId` is not the narrow maintainable fix; use a separate attempt identifier for tracing and test the dispatcher teardown directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep retry attempts on the stable runId:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: When `runLoopIterations > 1`, this sends later attempts through `runEmbeddedAttemptWithBackend` as `${params.runId}-retry-N`. Gateway chat links and run contexts are registered under the original client runId, and `server-chat` only remaps/finalizes lifecycle and assistant events when `evt.runId` matches that registry entry. A fallback retry would therefore emit under an unknown runId, losing the final/live chat remap and leaving the original chat run stuck instead of recovering.
