## Summary

Keep open: the network-level abort teardown is a plausible fix direction, but this PR still changes the stable `runId` contract on retries and the supplied EC2 logs do not prove dispatcher teardown or fallback survival after the delayed socket window.

## What This Changes

This PR exports dispatcher destruction, destroys the active guarded-fetch dispatcher on abort, suffixes embedded retry `runId`s, and adds an OpenAI HTTP client-disconnect test.

## Best Possible Solution

Keep the abort teardown at the guarded-fetch dispatcher boundary, keep `runId` stable across retries, add focused dispatcher-destruction or fallback-survival coverage, and require stronger redacted runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection plus supplied before logs support a plausible current-main path: guarded fetch currently performs graceful dispatcher close while Undici exposes abrupt destroy for aborted work.

## Solution Assessment

No. Destroying the guarded-fetch dispatcher on abort is the narrow maintainable boundary, but changing retry `runId`s is outside that fix and the new test does not assert dispatcher teardown or fallback survival.
