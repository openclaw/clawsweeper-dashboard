## Summary

Keep open: the network-level abort teardown is a plausible fix direction, but this PR should not merge as-is because it changes retry attempt runIds and the supplied real-behavior proof does not show dispatcher teardown, fallback survival, or a post-delay no-zombie observation.

## What This Changes

The PR adds abort-triggered dispatcher destruction in `fetchWithSsrFGuard`, exports `destroyDispatcher`, changes embedded retry attempts to suffixed runIds, and adds an OpenAI HTTP abort test.

## Best Possible Solution

Keep the network-level abort teardown, keep `runId` stable across retries, add focused dispatcher-destruction or fallback-survival coverage, and require stronger redacted runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection and supplied before logs support a plausible current-main path: guarded fetch closes dispatchers gracefully while Undici exposes `destroy()` for abrupt abort, but the delayed fallback crash was not reproduced here.

## Solution Assessment

No. The network teardown boundary is the narrow maintainable direction, but changing retry attempt `runId`s is not part of that fix; use a separate attempt id or attempt index if tracing needs disambiguation.
