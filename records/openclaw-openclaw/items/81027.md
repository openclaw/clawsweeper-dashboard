## Summary

Keep open: the network-level abort teardown is a plausible fix direction, but this PR still introduces a runId contract regression and the supplied real-environment proof does not yet show dispatcher teardown or fallback survival.

## What This Changes

The PR adds abort-time dispatcher destruction to SSRF-guarded fetch, exports `destroyDispatcher`, changes retry attempt runIds, and adds a gateway client-disconnect test.

## Best Possible Solution

Keep runId stable across retry attempts, apply abort-time dispatcher destruction at the guarded-fetch boundary, and add focused coverage plus stronger redacted runtime proof that demonstrates teardown or fallback survival.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported zombie-socket failure. Source inspection and supplied logs support a plausible current-main path, and the PR-introduced runId regression is source-verifiable.

## Solution Assessment

No. The network-level teardown is the narrow maintainable boundary, but suffixing retry runIds breaks existing run-scoped contracts; use a separate attempt id or index for tracing if needed.

- **[P2] Keep retry attempts on the original runId:** `src/agents/pi-embedded-runner/run.ts:1344`

Keep `params.runId` stable and add a separate attempt id if disambiguation is needed.
