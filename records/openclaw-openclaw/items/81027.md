## Summary

Keep open. The guarded-fetch abort teardown direction is plausible, but this PR is not merge-ready because it also changes retry attempt `runId` routing and the supplied proof/test coverage do not show the claimed hard-kill socket behavior.

## What This Changes

The PR exports dispatcher destruction, wires `fetchWithSsrFGuard` to destroy the active dispatcher on abort, adds a gateway abort test, and changes embedded retry attempts to use suffixed `runId` values.

## Best Possible Solution

Keep the abort hard-kill fix inside the guarded-fetch/dispatcher boundary, preserve stable `runId` routing, and add focused regression coverage that observes dispatcher destruction or fallback survival after abort.

## Reproduction Assessment

No high-confidence runtime reproduction was established. Source inspection shows current main only performs graceful dispatcher close on guarded-fetch release, and the Undici contract explains why `destroy()` is the relevant abort primitive, but the full 97-second fallback failure was not reproduced here.

## Solution Assessment

No. The network teardown approach is directionally right, but changing retry attempt `runId` is not the narrow fix and the regression test should exercise dispatcher teardown rather than duplicate client-disconnect abort propagation.
