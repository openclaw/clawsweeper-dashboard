## Summary

Keep open: the network-level abort teardown is a plausible bug fix direction, but the branch still changes retry run identity in a way that conflicts with run-scoped caches/events, and the supplied proof does not demonstrate dispatcher teardown or fallback survival.

## What This Changes

The PR adds abort-triggered Undici dispatcher destruction to guarded SSRF fetches, exports `destroyDispatcher`, changes retry attempt `runId`s, and adds a gateway HTTP abort test.

## Best Possible Solution

Keep the fix at the guarded network teardown boundary, preserve stable `runId` across retries, add direct dispatcher-destruction or fallback-survival coverage, and require after-fix proof showing no delayed abort cascade.

## Reproduction Assessment

No high-confidence live reproduction was established here. Current source plus the supplied before logs make the bug path plausible because guarded fetch currently uses graceful dispatcher close while Undici distinguishes graceful close from abrupt destroy, but the delayed provider/fallback failure was not replayed.

## Solution Assessment

No. Network-level dispatcher destruction on abort is the narrow maintainable direction, but changing retry `runId`s is not part of that fix and coverage should exercise the dispatcher teardown or fallback survival path directly.
