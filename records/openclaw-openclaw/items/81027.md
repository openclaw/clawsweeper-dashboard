## Summary

Keep open: the guarded-fetch hard-kill direction is plausible and the PR now supplies real logs, but the patch changes retry attempt run IDs in a way that conflicts with current gateway chat/event routing. The added regression test also repeats existing disconnect coverage without exercising dispatcher teardown.

## What This Changes

The PR exports dispatcher destruction from SSRF utilities, wires guarded fetch aborts to destroy the active dispatcher, adds a gateway disconnect test, and changes embedded retry attempts to use suffixed run IDs.

## Best Possible Solution

Keep the dispatcher hard-destroy fix scoped to guarded fetch, keep runId stable across retries, introduce separate attempt metadata only if diagnostics need it, and add a focused dispatcher-teardown regression test before merge.

## Reproduction Assessment

Yes for the blocking review issue by source inspection: chat.send registers the client runId, server-chat remaps by exact evt.runId, and the PR emits retry events under suffixed ids. The original zombie-socket symptom was not locally reproduced; the PR supplies before/after copied live logs.

## Solution Assessment

No. Destroying the guarded-fetch dispatcher on abort is a reasonable fix direction, but changing runId to isolate retries is not the narrow maintainable solution; keep runId stable and carry attempt identity separately if needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep retry events on the stable runId:** `src/agents/pi-embedded-runner/run.ts:1339`
  - body: The chat pipeline registers and remaps live events by the original client runId. Passing `${params.runId}-retry-N` into `runEmbeddedAttemptWithBackend` means retry/fallback events are emitted under an unregistered id, so `server-chat.ts` will not find the chat link or session context and WebChat/control subscribers can miss or mis-key retry output. Keep `runId` stable and add a separate attempt id for diagnostics if needed.
  - confidence: 0.92

  - body: This test repeats the existing non-streaming disconnect assertion and never exercises `fetchWithSsrFGuard` or a dispatcher with `destroy`, so removing the new hard-kill code would not fail it. Add a focused guarded-fetch/undici seam test that observes dispatcher destruction on abort, or drop the duplicate e2e coverage.
  - confidence: 0.9
