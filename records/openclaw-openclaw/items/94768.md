## Label Justifications

- `P1`: The reported bug leaked plugin approval prompts across agent/channel boundaries in a live messaging workflow.
- `impact:security`: The issue concerns approval prompt metadata being visible outside the requesting agent's intended channel.
- `impact:message-loss`: The failure mode is misrouted duplicate approval messages through unrelated Telegram bot connections.

## Summary

Current main implements the route-field fix for the reported plugin approval leak: before_tool_call approvals now carry trusted channel, target, account, and thread metadata into plugin.approval.request, and the merged gateway proof covers the no-route behavior. The fix is on main only and has not shipped in the latest stable release.

## What This Changes

The issue reports that plugin approval prompts from one agent can be delivered through unrelated Telegram bots or channels in a multi-agent setup sharing the same chat.

## Best Possible Solution

Keep the merged turn-source routing implementation on main and verify the next release includes it before telling v2026.6.8 users the fix is shipped.

## Reproduction Assessment

Yes for the reported release/source path: v2026.6.8 omitted turn-source route fields from before_tool_call plugin approvals, and current main adds them. I did not rerun a live two-bot Telegram scenario in this read-only review.

## Solution Assessment

Yes. The merged fix uses the existing HookContext and plugin approval contract to carry trusted channel/account/thread metadata, which is narrower and cleaner than adding a separate agent-id visibility rule.
