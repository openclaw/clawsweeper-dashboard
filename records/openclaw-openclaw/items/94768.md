## Summary

Keep open: current main and the latest release still omit trusted turn-source routing fields from before_tool_call plugin approval requests, leaving the reported Telegram/native approval misrouting path plausible. The open route-forwarding PR is the current candidate fix path, while the direct broadcast-leak PR closed unmerged.

## What This Changes

The issue reports that plugin approval prompts can be delivered through unrelated connected channel clients instead of only the requesting agent's channel.

## Best Possible Solution

Merge or replace the focused turn-source routing fix so trusted channel, target, account, and thread metadata reaches plugin.approval.request, then verify native Telegram delivery only targets the originating bot/account.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection supports the reported path: before_tool_call plugin approvals omit route metadata while native approval runtimes can receive plugin approval events and must self-filter by request binding.

## Solution Assessment

No, the issue's agent-id visibility suggestion is not the best standalone fix. The cleaner path is to carry trusted turn-source channel/account/thread metadata through the existing plugin approval contract and prove Telegram multi-agent isolation.
