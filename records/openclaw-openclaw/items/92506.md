## Summary

This remains a valid released regression on current main. The previous fix added the Codex rate-limit hook, but no-profile sessions still fail before reaching it: chat status requires an OAuth/token auth label, while CLI status lacks the authoritative Codex runtime-route context needed to select synthetic auth safely.

## What This Changes

Restore OpenAI/Codex quota-window usage in Telegram `/status` and CLI `status --usage` when the Codex runtime uses synthetic app-server auth without a saved OpenAI profile.

## Best Possible Solution

Carry the authoritative effective runtime-auth route into usage selection: chat status should trust its resolved Codex harness even without a local auth label, while CLI status should request synthetic usage only when the configured OpenAI route is the usable Codex app-server route.

## Reproduction Assessment

Yes at high confidence from source and the detailed v2026.6.6 host evidence: without a saved profile, chat status fails its auth-label gate and CLI lacks runtime-route context. This read-only review did not execute the reporter's live VPS setup.

## Solution Assessment

Yes, provided the fix uses the authoritative runtime route rather than unconditional provider synthesis. That preserves direct OpenAI behavior and reuses the existing Codex plugin hook instead of creating a parallel quota path.
