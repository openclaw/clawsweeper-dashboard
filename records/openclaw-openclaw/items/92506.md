## What This Changes

The issue reports that Telegram `/status` and CLI `status --usage` omit OpenAI/Codex usage windows when the Codex runtime is usable through synthetic `codex-app-server` auth but no local OpenAI auth profile exists.

## Best Possible Solution

Land or replace the route-aware fix in https://github.com/openclaw/openclaw/pull/92520 so status usage trusts a usable Codex app-server runtime route without changing direct OpenAI/API-key behavior.

## Reproduction Assessment

Yes. The current source path shows chat status gates synthetic usage on an OAuth/token label and CLI status lacks effective runtime-route auth context, matching the reporter's v2026.6.6 live evidence.

## Solution Assessment

Yes, the route-aware fix direction is the best shape. It reuses the existing Codex provider usage hook while avoiding unconditional synthetic usage for direct OpenAI or API-key-backed contexts.
