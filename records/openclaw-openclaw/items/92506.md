## Summary

The report remains necessary: current `main` still gates provider-usage plugin auth on saved or direct credential sources, so a usable synthetic `codex-app-server` route with no local OpenAI profile is omitted from CLI and chat status. The prior merged fix restored the provider hook and rate-limit reader but did not make this synthetic-only state eligible to reach that hook.

## What This Changes

Restore OpenAI/Codex quota usage in Telegram `/status` and CLI status when the Codex runtime uses synthetic app-server auth without a saved local OpenAI profile.

## Best Possible Solution

Extend the existing provider-usage auth eligibility decision to admit the activated synthetic Codex app-server route, then use the existing OpenAI-context/Codex-hook path and prove unrelated credential-less plugins remain skipped.

## Reproduction Assessment

Yes. The reporter provides a concrete `v2026.6.6` live state, and current source shows that enabling credential-source gating prevents plugin auth resolution when synthetic Codex auth is usable but no local profile exists.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to make the existing auth-source gate recognize the established synthetic Codex route; adding configuration, duplicating rate-limit fetching, or disabling the gate globally would be worse.
