## Summary

Keep open: current main still has a source-reproducible gap for the reported clean-but-mismatched memory status after a provider cutover. Status-only managers can treat existing metadata as clean while `--deep` initializes the new provider and keeps stale vector dimensions from the old index.

## What This Changes

The issue requests builtin memory cutover handling so switching from OpenAI embeddings to Ollama cannot leave a clean-looking mixed-dimension SQLite index.

## Best Possible Solution

Make status, deep probes, and live reload validate stored metadata against the active provider/model/providerKey/scope and mark mismatches dirty or rebuild before reporting the index clean.

## Reproduction Assessment

No. No end-to-end Docker/Ollama repro was run, but source inspection shows a current-main path where status-only managers treat existing metadata as clean while deep status can initialize the new provider and retain stale vector dimensions.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing full-reindex metadata comparison for status/reload validation, then add a regression test for stale provider or vector-dimension metadata.
