## Summary

Keep open. Current main still has a source-reproducible path where a status-only manager can combine the active Ollama provider/model with vector dimensions loaded from older index metadata and report `Dirty: no`; the earlier Ollama timeout issue was fixed separately and does not cover this provider/vector-space cutover state.

## What This Changes

Ensure memory provider/model/vector-dimension cutovers invalidate or rebuild the built-in SQLite memory index instead of reporting a clean mixed vector state.

## Best Possible Solution

Make memory-core validate stored index identity before reporting clean status or using vector search, and force a safe dirty/reindex state whenever active provider, model, provider key, or vector dimensions no longer match the stored SQLite metadata.

## Reproduction Assessment

Yes, at source level. A high-confidence path is to start with a 1536-dimensional OpenAI-indexed SQLite store, switch config to a 768-dimensional Ollama embedding model, then run `openclaw memory status --deep` or a search before a full reindex has completed; current status code can report clean while combining active provider data with stored old vector dimensions.

## Solution Assessment

Yes. The narrow maintainable fix is in memory-core metadata validation and status/search gating, not a new config knob or provider-specific workaround.
