## Summary

Keep open: current main still has a source-reproducible path for the reported clean-but-mismatched memory status. Status-only managers treat any existing index metadata as clean, reuse persisted vector dimensions, and `memory status --deep` can initialize the new provider without running the provider/model/providerKey mismatch check used by the indexing path.

## What This Changes

The issue reports that switching built-in memory search from OpenAI embeddings to Ollama can leave the SQLite index reporting the new provider/model while retaining stale 1536-dimensional OpenAI vectors and `Dirty: no`.

## Best Possible Solution

Centralize the provider/model/providerKey/scope validation so status, deep probes, and live reload mark mismatched indexes dirty or invalid and guide a safe rebuild before reporting the index clean.

## Reproduction Assessment

No high-confidence end-to-end live reproduction was run. Source inspection shows a clear current-main path for the visible status inconsistency: status-only managers can keep stale metadata dimensions clean while `--deep` initializes the new provider without syncing.

## Solution Assessment

Yes, the best direction is the narrow status/reload validation fix: reuse the existing reindex metadata comparison rather than adding a new config option or changing provider defaults.
