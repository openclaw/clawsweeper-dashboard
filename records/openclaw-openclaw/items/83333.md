## Summary

Keep open: current main still has a source-reproducible path for the reported clean-but-mismatched memory status. The indexing path compares stored provider/model/providerKey metadata, but the status-only path can still read stale vector dimensions from existing metadata, initialize the new provider during `--deep`, and report `Dirty: no` without running that mismatch check.

## What This Changes

Investigate a reported memory-core bug where switching `agents.defaults.memorySearch` from OpenAI embeddings to Ollama can leave the SQLite memory index marked clean while still showing stale OpenAI vector dimensions.

## Best Possible Solution

Make status, deep probes, and live reload use the same provider/model/providerKey/scope validation as the reindex path, marking mismatches dirty and guiding or triggering a safe rebuild before reporting the index clean.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run, but source inspection gives a clear current-main reproduction path for the visible status inconsistency: status-only managers treat any existing metadata as clean while `--deep` can initialize the new provider and keep stale vector dimensions.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing reindex metadata comparison for status and reload validation; closing as fixed would be premature because that comparison is not applied to the reported clean status path.
