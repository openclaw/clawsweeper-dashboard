## Summary

Keep open. Current main still truncates the stale-gateway ancestor exclusion to the direct parent on macOS, while the code comments, tests, and WeChat docs define the intended safety invariant as excluding ancestors; the reporter’s follow-up narrows this to a latent cleanup correctness bug rather than the live SIGTERM loop now tracked by #76259.

## What This Changes

Extend stale-gateway cleanup ancestor exclusion on macOS so a nested helper cannot leave a gateway grandparent in the stale PID kill list.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, keep the existing Linux `/proc` path, and cover the macOS grandparent case with focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes, source-level. Stub `process.platform` to `darwin`, set `process.ppid` to a plugin-host PID, feed lsof output containing a gateway grandparent, and current main will return that grandparent because the helper exits before any transitive walk.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix; changing cleanup call-site policy or folding in #76259 would mix separate process-kill issues.
