## Summary

Keep open. Current `main` and the latest `v2026.5.7` release still stop `getSelfAndAncestorPidsSync()` at `process.ppid` on non-Linux platforms, so a macOS nested caller can leave a gateway grandparent outside the stale-process exclusion set; the reporter's follow-up correctly narrows this to a latent correctness bug while https://github.com/openclaw/openclaw/issues/76259 owns the live SIGTERM loop.

## What This Changes

The issue asks OpenClaw to walk macOS/BSD ancestor PIDs, not only Linux `/proc`, before stale gateway cleanup decides which listener PIDs are safe to signal.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk in `getSelfAndAncestorPidsSync()`, preserve the Linux `/proc` path, and cover the macOS grandparent case with targeted regression tests plus a changelog entry.

## Reproduction Assessment

Yes, source-level. Stub `process.platform` to `darwin`, set the parent PID to a plugin host, and feed lsof output containing a gateway grandparent; current `main` returns before any transitive walk, leaving that gateway PID in the stale list.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix and avoids changing stale-cleanup call-site policy.
