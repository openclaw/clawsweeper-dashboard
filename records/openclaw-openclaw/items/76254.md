## Summary

Keep open. Current main still stops the stale-gateway ancestor exclusion after `process.ppid` on macOS, so a nested macOS caller can still classify a gateway grandparent as stale; the reporter's later update lowers the severity because the original live SIGTERM loop now appears tracked separately by #76259.

## What This Changes

Request extending stale-gateway cleanup's self-and-ancestor PID exclusion so macOS grandchild callers do not treat a gateway grandparent as stale.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, keep the fix at the stale-PID filter layer, and cover the macOS grandparent case with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can stub `process.platform` to `darwin`, set `process.ppid` to a plugin-host PID, feed lsof output containing a gateway grandparent, and observe that current main returns the grandparent as a stale PID.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix; it avoids call-site policy changes and preserves the existing Linux `/proc` path.
