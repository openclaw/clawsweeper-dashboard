## Summary

Keep open. Current main still documents and implements the non-Linux ancestor walk as self plus direct parent only, so a nested macOS caller can still leave a gateway grandparent in the stale-PID kill list; the reporter's follow-up usefully separates this latent correctness bug from the active SIGTERM loop now tracked by #76259.

## What This Changes

The issue asks OpenClaw to make macOS stale-gateway cleanup exclude the full caller ancestor chain, not only the current process and direct parent.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, preserve the existing Linux `/proc` path, and cover the macOS grandparent case with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can stub `process.platform` to `darwin`, set the parent PID override to a plugin-host PID, feed lsof output containing a gateway grandparent, and observe that current main returns the grandparent as stale.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix; it keeps policy at the stale-PID filter layer and can reuse the existing `ps -p <pid> -o ppid=` precedent.

Current `getSelfAndAncestorPidsSync` in `src/infra/restart-stale-pids.ts` stops at `process.ppid` on macOS/non-Linux, so a nested caller such as gateway -> plugin-host -> sidecar can leave the gateway grandparent in the stale-PID kill list. Add a bounded Darwin/BSD parent-PID walk, likely using `ps -p <pid> -o ppid=` with a short timeout and the existing `MAX_ANCESTOR_WALK_DEPTH`, while preserving the Linux `/proc` path and Windows behavior.
