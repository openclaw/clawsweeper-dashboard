## Summary

Keep open. Current main still truncates `getSelfAndAncestorPidsSync` after `process.ppid` on macOS, so nested Darwin callers can leave a gateway grandparent outside the stale-PID exclusion set; the reporter’s follow-up only moves the live SIGTERM loop to #76259, not this latent correctness bug.

## What This Changes

Request a macOS/BSD transitive ancestor PID walk in stale-gateway cleanup so gateway grandparents are excluded before SIGTERM.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, preserve the Linux `/proc` path, and cover the macOS grandparent case with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can stub the platform to `darwin`, set the parent PID to an intermediate helper, feed lsof output containing the gateway grandparent, and observe that current main returns that grandparent as stale.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper is the narrowest maintainable fix because all stale-gateway cleanup callers already rely on that helper before termination.
