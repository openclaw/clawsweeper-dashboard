## Summary

Keep open. Current main still returns from `getSelfAndAncestorPidsSync` after `process.ppid` on macOS, so a nested helper can leave a gateway grandparent in the stale PID list; the reporter’s later comment narrows this to a latent cleanup correctness bug while the live SIGTERM loop is tracked separately by #76259.

## What This Changes

The issue requests extending macOS stale-gateway ancestor exclusion beyond the direct parent so nested helper callers cannot SIGTERM a live gateway grandparent.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside the existing ancestor-exclusion helper, keep the Linux `/proc` path, and cover the macOS grandparent case with targeted regression tests.

## Reproduction Assessment

Yes, source-level. Stub `process.platform` to `darwin`, stub `process.ppid` to a plugin host, and return lsof output containing a gateway grandparent; current main leaves that grandparent in the stale PID list because the helper returns before a transitive walk.

## Solution Assessment

Yes. Extending `getSelfAndAncestorPidsSync` with a bounded macOS/BSD parent lookup is the narrowest maintainable fix and avoids changing stale-cleanup call-site policy.

## Repair Work Prompt

Fix the macOS/Darwin stale-gateway ancestor exclusion bug reported at https://github.com/openclaw/openclaw/issues/76254. `getSelfAndAncestorPidsSync` currently walks transitive ancestors only on Linux via `/proc`, so a Darwin caller nested under `gateway -> plugin-host -> sidecar` can still leave the gateway grandparent in the stale PID list. Add a bounded BSD/macOS parent-PID lookup, likely using `ps -p <pid> -o ppid=` with a short timeout, while preserving the current Linux `/proc` path and existing Windows behavior unless a clearly generic helper makes Windows safer too. Add targeted regression coverage in `src/infra/restart-stale-pids.test.ts` for the Darwin grandparent case and keep #68451 direct-parent behavior intact. Add a user-facing `CHANGELOG.md` entry. Do not change `cleanStaleGatewayProcessesSync` public signature, port polling budgets, lsof command behavior, or unrelated `killProcessTree` behavior tracked separately by #76259.
