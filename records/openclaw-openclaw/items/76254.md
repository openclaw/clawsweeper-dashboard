## Summary

Keep open. Current main still explicitly limits the stale-gateway ancestor exclusion to self plus direct parent on macOS, while the documented and shipped invariant says cleanup must exclude the caller's ancestors; the reporter's later update lowers this from the immediate SIGTERM root cause to a latent correctness bug.

## What This Changes

The issue requests a bounded macOS/BSD transitive parent-PID walk for stale gateway cleanup so grandparent gateway processes are excluded like Linux ancestors.

## Best Possible Solution

Add a bounded macOS/BSD parent-PID reader to the existing ancestor exclusion helper, cover the Darwin grandparent case with focused tests, and leave the separate kill-tree regression in #76259 to its own fix.

## Reproduction Assessment

Yes. A focused unit reproduction can stub Darwin platform and parent PID state, feed lsof output containing a gateway grandparent, and observe that current main returns that grandparent as stale; the reporter also supplied a live macOS trace, later narrowed to a separate immediate killer.

## Solution Assessment

Yes. Extending the existing stale PID ancestor-exclusion helper is the narrow maintainable fix because it preserves the generic invariant at the filter layer without adding plugin-specific policy or changing unrelated kill-tree behavior.

In `src/infra/restart-stale-pids.ts`, extend `getSelfAndAncestorPidsSync` so Darwin/BSD can walk beyond `process.ppid` using a bounded, timeout-protected parent-PID lookup such as `ps -p <pid> -o ppid=`.
