## Summary

Keep open. Current main still limits macOS ancestor exclusion to `process.ppid`, while the cleanup/signaling path depends on that exclusion set before sending gateway signals; the reporter’s later update lowers the severity by separating the live Telegram SIGTERM cycle into #76259, but it does not invalidate this latent macOS grandparent bug.

## What This Changes

The issue requests a macOS/BSD transitive parent-PID walk for stale gateway cleanup and related gateway signaling so nested sidecar callers do not signal a gateway grandparent.

## Best Possible Solution

Add a bounded macOS/BSD parent-PID walk to the existing ancestor-exclusion helper, cover the macOS gateway-grandparent case in targeted tests, and keep the fix scoped away from the separate kill-tree and lsof-argv issues.

## Reproduction Assessment

Yes. A focused unit reproduction can stub the platform to `darwin`, set the parent PID to a plugin host, feed lsof output containing an openclaw gateway grandparent, and observe that current main includes the grandparent in the candidate list; the reporter also supplied live launchd evidence but later tied the active SIGTERM loop to #76259.

## Solution Assessment

Yes. Extending the existing exclusion helper is the narrow maintainable solution because cleanup and unmanaged signaling already flow through listener discovery; the safer boundary is to avoid folding #76259 or #70664 into this PR unless a small shared test helper is unavoidable.
