## Summary

Keep open. Current main still returns the ancestor exclusion set after only `process.ppid` on macOS, so nested macOS callers remain outside the shipped #68451 fix; the prior merged fix explicitly scoped transitive walking to Linux and documented macOS/Windows as direct-parent only.

## What This Changes

Request extending stale-gateway cleanup ancestor exclusion on macOS so grandchild sidecar callers do not SIGTERM the gateway they descend from.

## Best Possible Solution

Add a bounded macOS/BSD parent-PID walk to the stale PID ancestor exclusion path, cover the macOS grandparent case with targeted tests, and consider sharing that guard with verified gateway signaling paths mentioned by the report.

## Reproduction Assessment

Yes. A focused unit reproduction can stub `process.platform` to `darwin`, set `process.ppid` to a plugin-host PID, feed lsof output containing a gateway grandparent, and observe that current main returns the grandparent as stale; the reporter also supplied a live launchd/Telegram trace.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: extend the existing ancestor-exclusion invariant on macOS instead of changing supervisor call sites or adding plugin-specific policy.

## Repair Work Prompt

Fix the macOS stale-gateway ancestor exclusion gap reported in https://github.com/openclaw/openclaw/issues/76254. Current `getSelfAndAncestorPidsSync` in `src/infra/restart-stale-pids.ts` walks transitive ancestors only on Linux via `/proc`; on macOS it stops at `process.ppid`, so a grandchild sidecar can leave the gateway grandparent killable. Add a bounded macOS/BSD parent PID reader, preferably using a small timeout around `ps -p <pid> -o ppid=` or an existing safer platform helper if one exists, and use it in the existing `MAX_ANCESTOR_WALK_DEPTH` loop for `darwin` while preserving Windows behavior unless you choose a well-tested Windows ancestor API. Add regression coverage that simulates `darwin` with a direct parent plugin-host and gateway grandparent, proves the grandparent is excluded from `findGatewayPidsOnPortSync`, and keeps unrelated stale PIDs killable. Review whether `signalVerifiedGatewayPidSync` / verified listener paths need the same self-plus-ancestor guard because the issue mentions them, but do not broaden into unrelated lifecycle refactors. Treat #70664 as adjacent context only; do not solve the lsof command-name mismatch unless it is required to make this fix correct. Add a single-line user-facing `CHANGELOG.md` fix entry if the code path changes. Do not add dependencies, do not change public CLI contracts, and keep the fix in the stale PID/gateway process area.
