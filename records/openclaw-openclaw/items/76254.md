## Summary

Keep open. Current main still truncates the stale-gateway ancestor exclusion at `process.ppid` on macOS, so the latent grandparent-kill correctness gap remains even though the reporter later separated the original live SIGTERM loop into #76259.

## What This Changes

Requested a bounded macOS/BSD ancestor PID walk for stale gateway cleanup so nested callers do not classify a gateway grandparent as stale.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, keep the fix at the stale-PID filter layer, and cover the macOS grandparent case with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction can stub `process.platform` to `darwin`, set the parent PID to a plugin-host process, feed lsof output containing a gateway grandparent, and observe that current main returns the grandparent as stale.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix; #76259 should remain a separate kill-tree issue.

## Repair Work Prompt

Fix the macOS/BSD stale-gateway ancestor exclusion gap reported in https://github.com/openclaw/openclaw/issues/76254. Current main only walks transitive ancestors on Linux via `/proc`; on Darwin it excludes only `process.pid` and `process.ppid`, so a `gateway -> plugin-host -> sidecar` caller can still classify the gateway grandparent as stale. Implement a bounded, failure-tolerant Darwin/BSD parent lookup inside `src/infra/restart-stale-pids.ts`, likely using `spawnSync("ps", ["-p", String(pid), "-o", "ppid="])` with a short timeout. Keep the existing Linux `/proc` path, keep Windows behavior unless you deliberately add equivalent coverage, do not add plugin-specific IDs or call-site policy changes, and keep the cleanup decision in the stale-PID filter layer. Add targeted tests in `src/infra/restart-stale-pids.test.ts` for a Darwin grandparent exclusion, lookup failure/truncation, and cycle/depth bounds. Add a user-facing changelog fix entry. Treat #76259 and #70664 as related context but separate issues, not part of this fix.
