## Summary

Keep open. Current main still limits stale-gateway ancestor exclusion to self plus direct parent on macOS, so a nested caller can leave a gateway grandparent in the stale PID list; the reporter's follow-up narrows this to a latent correctness bug while the live SIGTERM loop is tracked separately by #76259.

## What This Changes

The issue asks for macOS/BSD stale-gateway cleanup to exclude the full caller ancestor chain, not only the direct parent, before signaling gateway PIDs.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk inside `getSelfAndAncestorPidsSync`, preserve the existing Linux `/proc` path, and cover the macOS grandparent case with focused regression tests and a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: stub `process.platform` to `darwin`, set `process.ppid` to a plugin host, and feed lsof output containing a gateway grandparent; current main excludes only the direct parent and returns the grandparent as stale.

## Solution Assessment

Is this the best way to solve the issue? Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is narrower than changing cleanup call-site policy, and `src/infra/ports-inspect.ts` already uses the same `ps -p <pid> -o ppid=` pattern.
