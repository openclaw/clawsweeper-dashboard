## Summary

Keep open: current main appears to have fixed the cold registry preload gap inside `resolvePluginTools`, but the documented `tools.alsoAllow` path for optional plugin tools is still source-reproducible as unavailable to effective/native tool discovery because plugin materialization only reads `allow`.

## What This Changes

Request to make optional plugin tools that appear in catalog available to `tools.effective`, `tools.invoke`, and native agent tool paths, and to align optional-tool allowlist docs/config semantics.

## Best Possible Solution

Make documented `alsoAllow` entries participate in optional plugin tool discovery without converting them into a restrictive core-tool allowlist, or update the public Lobster docs if maintainers intentionally require explicit `allow`.

## Reproduction Assessment

Yes, source-level. Following the documented `tools.alsoAllow: ["lobster"]` path leaves `pluginToolAllowlist` empty because `collectExplicitAllowlist` reads only `allow`, and optional plugin tools are dropped when the allowlist is empty.

## Solution Assessment

No for closing this issue yet. The resolver cold-load fix is the right narrow solution for the preload half, but the remaining best fix is to align `alsoAllow`-based optional plugin discovery with the public docs or explicitly change those docs.
