## Summary

Keep open: current main still has a source-backed cold-runtime gap where catalog preloads optional plugin tool runtime state but the effective/invoke/native tool assembly path only resolves already-loaded plugin registries. The related `alsoAllow`/`llm-task` failure is partly covered by #74019/#74139, but that does not supersede this registry preload bug.

## What This Changes

Make optional plugin tools that appear in `tools.catalog` available to `tools.effective`, `tools.invoke`, and native agent tool surfaces when the tool is allowed.

## Best Possible Solution

Add a shared, narrow preload step before plugin tool resolution for agent/gateway execution surfaces, then align the Lobster optional-tool docs with the supported allowlist behavior or the separately landed `alsoAllow` fix.

## Reproduction Assessment

Yes, source-level. Current main shows catalog preloading the optional plugin tool registry while `createOpenClawTools`/gateway execution paths only call the resolver, which reads loaded registries and can return no optional tools from a cold runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to centralize or reuse the catalog-style preload before execution/effective/native plugin tool resolution, while leaving broader `alsoAllow` semantics to the existing allowlist repair track.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76616. Optional plugin tools such as `llm-task` and `lobster` can appear in `tools.catalog` because catalog preloads the standalone plugin tool registry, but `createOpenClawTools` / `resolveGatewayScopedTools` / `tools.effective` / `tools.invoke` can resolve plugin tools without that preload and miss allowlisted optional tools from a cold runtime. Add a shared preload in the execution/effective/native plugin tool resolution path, preferably near `resolveOpenClawPluginToolsForOptions` so `createOpenClawTools` and gateway callers share it. Preserve plugin disablement, workspace/runtime config selection, `authProfileStore` provider checks, and `allowGatewaySubagentBinding`. Add regression coverage that resets plugin runtime state and proves an allowlisted optional plugin tool is present in the effective/invoke/native tool path without calling `tools.catalog` first. Also update Lobster/llm-task docs only as needed to avoid the unsupported `tools.profile="full"` + `tools.alsoAllow` promise; coordinate with #74019/#74139 and do not broaden disabled-tool allowlist semantics. Add a user-facing `CHANGELOG.md` fix entry. Do not change unrelated plugin API contracts or add a config option.
