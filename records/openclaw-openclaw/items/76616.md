## Summary

Keep open: current main appears to have fixed the cold partial-registry path, but the documented `tools.alsoAllow` path still does not feed optional plugin discovery for gateway/agent execution surfaces, so the issue has remaining actionable work.

## What This Changes

The issue reports optional plugin tools visible in catalog but missing from `tools.effective`, `tools.invoke`, and native agent tool assembly, and asks to align execution preloading plus optional-tool allowlist semantics.

## Best Possible Solution

Keep the main-only cold-registry fix and decide the remaining config contract explicitly: either make profile-stage `alsoAllow` feed optional plugin discovery for execution surfaces or update the Lobster/llm-task docs to require explicit `allow` where that is the supported contract.

## Reproduction Assessment

Yes, source-level. A config following the current Lobster docs with only `tools.alsoAllow: ["lobster"]` reaches gateway/native plugin discovery through `collectExplicitAllowlist`, which reads only `allow`, and optional plugin tools are then filtered out unless the allowlist contains `*`, `group:plugins`, the plugin id, or the tool name.

## Solution Assessment

No, current main is only a partial solution. The partial-registry cold-load fix is narrow and well covered, but the documented `alsoAllow` path still needs a maintainer decision or follow-up before this issue is fully resolved.
