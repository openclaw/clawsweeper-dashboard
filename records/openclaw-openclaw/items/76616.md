## Summary

Keep open: current main still has a source-backed mismatch where `tools.catalog` preloads optional plugin tool runtime state, but the effective/invoke/native execution path resolves plugin tools without the same preload.

## What This Changes

The issue asks OpenClaw to make optional plugin tools such as `llm-task` and `lobster` available to `tools.effective`, `tools.invoke`, and native agent tool assembly when they are shown in the plugin tool catalog and allowed by policy.

## Best Possible Solution

Add a shared catalog-style preload before execution/effective/native plugin-tool resolution, cover it with focused gateway and agent regression tests, and align the optional-tool docs with the supported allowlist behavior.

## Reproduction Assessment

Yes, source-level. Current main shows catalog preloading optional plugin tool runtime state before resolution while the effective/invoke/native path reaches `resolvePluginTools` through `createOpenClawTools` without that preload.

## Solution Assessment

Yes for the runtime bug: the narrow maintainable fix is to reuse the existing standalone plugin-tool preload in the shared execution resolver before `resolvePluginTools`. Broader `alsoAllow` semantics should be split into docs or policy work rather than bundled into the runtime fix.
