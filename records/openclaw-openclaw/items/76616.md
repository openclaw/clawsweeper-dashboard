## Summary

Keep open: current main still has a source-backed cold-runtime gap where catalog preloads optional plugin tool runtime state, but effective/invoke/native tool assembly resolves plugin tools without the same preload.

## What This Changes

Optional plugin tools such as `llm-task` and `lobster` should be available through `tools.effective`, `tools.invoke`, and native agent tool assembly when they appear in `tools.catalog` and are explicitly allowed.

## Best Possible Solution

Add a shared or call-site preload before execution/effective/native plugin tool resolution, then align the Lobster optional-tool docs with the supported allowlist behavior without changing broader `alsoAllow` semantics unless maintainers choose that separately.

## Reproduction Assessment

Yes, source-level. Current main shows catalog preloading the standalone plugin registry before resolution, while `createOpenClawTools` and the gateway effective/invoke path resolve optional plugin tools without that preload.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the catalog-style preload before plugin tool resolution for execution surfaces; broader `tools.profile="full"` plus `alsoAllow` semantics should remain separate docs or policy work.
