## Summary

Current main already closes the reported catalog/effective/invoke gap: non-core `tools.invoke` names are carried into Gateway tool resolution, optional plugin resolution now has a standalone runtime fallback, and the `full`/`alsoAllow` policy paths now produce explicit allowlist entries for optional plugin tools.

## What This Changes

The issue requests that optional plugin tools listed in `tools.catalog`, including `llm-task` and `lobster`, also be available through `tools.effective`, `tools.invoke`, and native agent tool assembly.

## Best Possible Solution

Keep the current main behavior: request-scoped plugin tool allowlisting plus resolver-level cold runtime loading avoids catalog-only visibility without adding broad eager plugin preloads.

## Reproduction Assessment

Yes, source-level. The original failure path is visible from the reported release behavior, and current main shows the corrected `tools.invoke`/effective allowlist path plus resolver cold-load fallback and regression coverage.

## Solution Assessment

Yes. The current implementation is narrower than always preloading in `createOpenClawTools`: it carries explicit requested tools through policy resolution and loads only scoped plugin runtimes when needed.
