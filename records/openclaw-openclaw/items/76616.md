## Summary

Current main implements the missing execution-path behavior: plugin tool resolution now cold-loads a standalone runtime registry when scoped plugin tools are absent from loaded registries, and current policy/tests cover optional tools through wildcard and `alsoAllow` paths. This is present on main but not proven shipped in v2026.5.2.

## What This Changes

The issue asks OpenClaw to make optional plugin tools shown by `tools.catalog` available through `tools.effective`, `tools.invoke`, and native agent tool assembly, with clear optional-tool allowlist semantics.

## Best Possible Solution

Keep the current shared resolver behavior and ship it in the next release; no separate core issue remains for the catalog/effective/invoke mismatch.

## Reproduction Assessment

Yes for the original release behavior: the reporter supplied concrete Gateway calls and the discussion/source review identified the catalog preload versus execution resolver split. Current main no longer has a high-confidence failing path because the shared resolver and existing tests now cover cold optional-plugin loading and `alsoAllow`/wildcard policy.

## Solution Assessment

Yes. The current main approach is the narrow maintainable fix: make the shared plugin-tool resolver load missing scoped runtime registries and keep allowlist semantics in the existing policy layer, rather than adding a separate product/config mode.
