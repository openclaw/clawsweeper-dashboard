## Summary

Keep open: current main still has a source-backed mismatch where `tools.catalog` preloads optional plugin tool runtime state but the effective/invoke/native assembly path does not. The related #74019/#74139 allowlist-guard work is useful context, but it does not supersede this registry preload gap.

## What This Changes

The issue reports that optional plugin tools can appear in `tools.catalog` but remain unavailable to `tools.effective`, `tools.invoke`, and native agent tool surfaces.

## Best Possible Solution

Add a shared preload step before plugin tool resolution for native agent assembly, `tools.effective`, and `tools.invoke`, then align Lobster/llm-task docs with the supported allowlist behavior or the separate `alsoAllow` repair.

## Reproduction Assessment

Yes, source-level. Current main shows catalog preloading the standalone plugin tool registry while `createOpenClawTools` and the gateway execution path only call the resolver, which can return no optional tools from a cold runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or centralize the catalog-style preload before execution/effective/native plugin tool resolution, while leaving broader `alsoAllow` semantics to the existing allowlist repair track.

Optional plugin tools such as `llm-task` and `lobster` can be visible in `tools.catalog` but unavailable to `tools.effective`, `tools.invoke`, and native agent tool assembly when the execution path starts from a cold plugin runtime. Add a narrow shared preload before plugin tool resolution in the execution/effective/native path, reusing `resolveOpenClawPluginToolInputs`/`ensureStandalonePluginToolRegistryLoaded` or an equivalent helper so catalog and execution semantics stay aligned.
