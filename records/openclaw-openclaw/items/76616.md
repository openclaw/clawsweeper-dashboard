## Summary

Current main now covers the reported catalog/effective/invoke mismatch in the shared plugin-tool resolver, and the allowlist semantics called out in the report are also covered by current source and regression tests.

## What This Changes

The issue asks OpenClaw to make optional plugin tools shown in `tools.catalog` available through `tools.effective`, `tools.invoke`, and native agent tool assembly, and to align optional-tool allowlist semantics.

## Best Possible Solution

Keep the shared resolver behavior on main and let a future release carry it forward; no separate core issue is needed for the already-covered optional-tool availability path.

## Reproduction Assessment

No current-main failing path remains from source inspection. The original 2026.5.2 steps are well described, but current main cold-loads selected plugin registries and treats `alsoAllow`/`full` as allowlists that can admit optional plugin tools.

## Solution Assessment

Yes. Fixing the shared resolver and policy helpers is the narrowest maintainable solution because it covers catalog, effective, invoke, and native agent assembly without duplicating preload logic at each call site.
