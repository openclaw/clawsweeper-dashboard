## Summary

Keep open: the PR addresses the core URL-to-local-path bug and now has scoped terminal proof, but the explicit `--model` remote path still discards fetched MIME/filename metadata before provider dispatch, so remote PNG/WebP inputs can be sent as the wrong media type. The linked issue also covers broader MiniMax/image-tool behavior, so the closing reference should not land unless maintainers agree the broader issue is fully fixed.

## What This Changes

The PR changes image describe CLI/runtime handling so HTTP(S) `--file` values are passed as `MediaUrl` remote attachments, with focused CLI/runtime tests for configured and explicit-model flows.

## Best Possible Solution

Preserve URL inputs as `MediaUrl`, carry fetched MIME and filename through the explicit-model remote path, add regression coverage for that contract, and avoid auto-closing the broader linked issue unless the remaining MiniMax/image-tool report is resolved.

## Reproduction Assessment

Yes. Source inspection shows current main `path.resolve`s HTTP(S) `--file` values, and the contributor's terminal proof shows the branch preserves the URL in the no-provider image describe path.

## Solution Assessment

No. The `MediaUrl` boundary is the right fix direction, but the explicit `--model` branch should pass `media.mime` and `media.fileName` from the remote fetch result into the provider/generic image runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve fetched MIME for remote model describes:** `src/media-understanding/runtime.ts:181`
  - body: In the new `mediaUrl` branch, `cache.getBuffer()` returns the remote content type or detected MIME and filename, but the patch keeps only `.buffer` and later calls `describeImage` with `mime: params.mime` plus `path.basename(params.filePath)`. For `--file https://...png --model provider/model` with no MIME, the generic image runtime falls back to `image/jpeg`, so strict providers can receive PNG/WebP inputs with the wrong declared type. Keep the full media result and pass `media.mime` and `media.fileName` into the image request.
  - confidence: 0.88
