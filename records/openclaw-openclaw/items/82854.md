## Summary

Keep this PR open: it targets a valid source-reproducible CLI URL handling bug, but it is draft, lacks real behavior proof, and the explicit `--model` remote-URL path drops fetched MIME/filename before calling the image provider.

## What This Changes

The branch changes the image describe CLI and media-understanding runtime so HTTP(S) `--file` inputs are passed as `MediaUrl` remote attachments, with focused CLI/runtime tests.

## Best Possible Solution

Preserve URL inputs as `MediaUrl` for both configured and explicit-model image describe paths, carry fetched MIME/filename into provider calls, add regression coverage for that contract, and attach a real remote-URL CLI proof.

## Reproduction Assessment

Yes, by source inspection: current `runImageDescribe` calls `path.resolve(filePath)` for every `--file`, so an HTTP(S) image URL becomes a local-looking path before media-understanding dispatch. I did not run the CLI because this review is read-only.

## Solution Assessment

No. Preserving HTTP(S) as `MediaUrl` is the right boundary, but this patch should also carry the remote fetch result's MIME/filename through the explicit `--model` path and needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve fetched MIME for remote model describes:** `src/media-understanding/runtime.ts:181`
  - body: In the new `mediaUrl` branch, `cache.getBuffer()` returns the remote content type/detected MIME and filename, but the code keeps only `.buffer` and later calls `describeImage` with `mime: params.mime` plus `path.basename(params.filePath)`. For `--file https://...png --model provider/model` with no MIME, the generic image runtime falls back to `image/jpeg`, so the remote URL path can still send PNG/WebP inputs with the wrong declared media type. Keep the buffer result and pass `media.mime` and `media.fileName` into the image request.
  - confidence: 0.87
