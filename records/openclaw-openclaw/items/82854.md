## Summary

Keep open: the PR targets a valid source-reproducible CLI URL handling bug and now has terminal proof for the scoped behavior, but the explicit-model remote path drops fetched media metadata and can still send non-JPEG URLs with the wrong MIME.

## What This Changes

The PR changes image describe CLI/runtime handling so HTTP(S) `--file` values are passed as `MediaUrl` remote inputs instead of being resolved as local filesystem paths.

## Best Possible Solution

Preserve URL inputs as `MediaUrl` in both configured and explicit-model image describe paths while carrying fetched MIME and filename into provider calls.

## Reproduction Assessment

Yes. Source inspection of current main shows `runImageDescribe` calls `path.resolve(filePath)` for every `--file`, and the contributor's before output matches that URL-to-local-path failure mode.

## Solution Assessment

No. Preserving HTTP(S) inputs as `MediaUrl` is the right fix boundary, but the explicit `--model` branch should retain the full `MediaBufferResult` and pass `media.mime` and `media.fileName` into `describeImage`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve fetched MIME for remote model describes:** `src/media-understanding/runtime.ts:181`
  - body: When `params.mediaUrl` is set, `cache.getBuffer()` returns the fetched MIME and filename, but the new branch keeps only `.buffer`. The later provider call still uses `mime: params.mime` and `path.basename(params.filePath)`, so `--file https://...png --model provider/model` can send a PNG/WebP URL as the default `image/jpeg`. Keep the media result and pass its MIME and filename through.
