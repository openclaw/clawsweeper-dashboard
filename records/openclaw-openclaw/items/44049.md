## Summary

Keep open: the ACP client duplicate-`acp` bug is still present on current main and the PR's production change is narrow, but the added regression test bypasses the changed args builder and would not catch a revert of the fix.

## What This Changes

The PR changes ACP client server argument construction to drop one redundant leading `acp` token before prepending the canonical subcommand and adds a regression test.

## Best Possible Solution

ACP client argument construction should produce exactly one canonical `openclaw acp` subcommand, with regression coverage that reaches the args builder path rather than only the lower spawn materializer.

## Reproduction Assessment

Yes. Source inspection on current main shows `buildServerArgs` always prepends `acp`, so the reported `serverArgs: ['acp']` path deterministically becomes `['acp', 'acp']` before spawn; I did not execute it because this was a read-only review.

## Solution Assessment

No as submitted. The production normalization is the narrow maintainable fix, but the regression test should exercise `buildServerArgs` through `createAcpClient` or a focused helper before merge.
