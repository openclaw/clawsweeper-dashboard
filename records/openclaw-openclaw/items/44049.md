## Summary

Keep open. Current main still prepends `acp` unconditionally in the ACP client spawn args, so the underlying bug remains; this PR's production change is useful and narrow, but the submitted regression test bypasses the changed builder path and the user-visible fix still needs a changelog entry before merge.

## What This Changes

The PR changes ACP client command construction to drop one leading `acp` from forwarded `serverArgs` before prepending the canonical subcommand, and adds a spawn-invocation test for that case.

## Best Possible Solution

Land a narrow ACP client fix that normalizes one redundant leading `acp` in forwarded server args, tests the real builder or an end-to-end mocked spawn path, preserves ACP server/session/permission behavior, and records the user-visible fix in `CHANGELOG.md`.

Yes. Source inspection gives a high-confidence reproduction path: `openclaw acp client --server openclaw --server-args acp --server-verbose` reaches `buildServerArgs`, which prepends `acp` to caller-provided `['acp']` before spawn.

## Solution Assessment

No, not as submitted. The production normalization is the right direction, but the regression test needs to exercise `buildServerArgs` or `createAcpClient`, and the branch needs the required changelog entry.
