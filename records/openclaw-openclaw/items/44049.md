## Summary

Keep open: the production normalization is narrow and supported, but the added regression test does not exercise the changed ACP argument builder and would stay green if the fix were reverted.

## What This Changes

The PR removes one leading `acp` token from ACP client `serverArgs` before prepending the canonical subcommand and adds a helper-level regression test.

## Best Possible Solution

Land the one-token ACP argument normalization with regression coverage at the builder or end-to-end spawn boundary, while preserving `--server-verbose`, ACP server behavior, and session semantics.

## Reproduction Assessment

Yes. Current main prepends `acp` to unnormalized `serverArgs`, so a caller passing `["acp", "--verbose"]` reaches the spawn path as duplicated argv; I did not run the CLI in this read-only review.

## Solution Assessment

No, not fully as proposed. The production normalization is the narrow maintainable fix, but the regression test should cover `buildServerArgs` directly or assert the `createAcpClient` spawn argv instead of calling the post-builder helper.
