## Summary

Keep this PR open: the production change is a narrow fix for a source-proven ACP client argument duplication bug, but the added regression test does not exercise the changed builder path and should block merge until repaired.

## What This Changes

The PR removes one leading forwarded `acp` token from ACP client `serverArgs` before prepending the canonical ACP subcommand, and adds a regression test for that duplicated-argument case.

## Best Possible Solution

Land the one-token ACP argument normalization with regression coverage at `buildServerArgs` or the `createAcpClient` spawn boundary, while preserving `--server-verbose`, ACP server behavior, and session semantics.

## Reproduction Assessment

Yes. Current main prepends `acp` to unnormalized forwarded `serverArgs`, so a caller passing `["acp", "--verbose"]` reaches the spawn path as duplicated argv; I did not run the CLI in this read-only review.

## Solution Assessment

No, not fully as proposed. The production normalization is the narrow maintainable fix, but the regression test should cover `buildServerArgs` directly or assert the `createAcpClient` spawn argv instead of calling the post-builder helper.
