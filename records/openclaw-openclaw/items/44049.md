## Summary

Keep open. Current main still has the ACP duplicated-argument path, and the production fix is narrowly aimed at that bug, but the PR is not ready because its added regression test exercises a lower helper after args are already built and would not fail if the normalization were removed.

## What This Changes

The PR normalizes ACP client forwarded server args by dropping one leading `acp` token before prepending the canonical subcommand and adds a test.

## Best Possible Solution

Land the one-token ACP argument normalization with regression coverage at `buildServerArgs` or the `createAcpClient` spawn boundary while preserving verbose handling, env stripping, Windows wrapper behavior, and session semantics.

## Reproduction Assessment

Yes from source inspection. Current main prepends `acp` to unnormalized forwarded args and then sends those args through the real spawn path; I did not run the CLI because this was a read-only review.

## Solution Assessment

No, not fully as proposed. The production normalization is the narrow maintainable fix, but the regression test should cover the builder or `createAcpClient` spawn argv instead of the lower-level spawn materializer.
