## Summary

Keep this PR open: the production fix is narrow and matches a real ACP client argument bug, but the added regression test still bypasses the changed argument builder and would not catch the duplication returning.

## What This Changes

This PR changes ACP client server-argument construction to remove one leading forwarded `acp` token before prepending the canonical subcommand and adds a regression test.

## Best Possible Solution

Keep the one-token normalization, but land it with regression coverage at the builder or end-to-end client spawn boundary.

## Reproduction Assessment

Yes. Source inspection shows current main prepends `acp` to raw forwarded `serverArgs`, then passes the built argv to the spawn path, so `serverArgs: ["acp", ...]` becomes duplicated.

## Solution Assessment

No, not as proposed. The production edit is the narrow maintainable fix, but the proposed regression test bypasses the changed builder and should be replaced with builder or end-to-end spawn coverage.
