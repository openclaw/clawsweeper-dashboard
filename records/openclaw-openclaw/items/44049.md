## Summary

## What This Changes

The PR drops one leading forwarded `acp` token from ACP client `serverArgs` before prepending the canonical ACP subcommand, and adds a regression test for that case.

## Best Possible Solution

The maintained end state is one canonical ACP subcommand in spawned OpenClaw commands, covered at the ACP client construction boundary and reflected in release notes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw acp client --server openclaw --server-args acp --server-verbose` forwards a leading `acp` into `buildServerArgs`, which prepends another `acp` before spawn.

## Solution Assessment

No, not as submitted. The production normalization is the right narrow direction, but the branch needs a regression test that exercises `buildServerArgs` or the real `createAcpClient` spawn path and a required changelog entry.
