## Summary

Keep open: current main still builds duplicated ACP argv for callers that pass a leading `acp`, and no merged PR supersedes the bug. The production change is narrow, but the added regression test exercises the post-builder spawn helper rather than the argument builder that the PR changes.

## What This Changes

The PR drops one leading `acp` token from ACP client `serverArgs` before prepending the canonical ACP subcommand, and adds a test intended to cover that case.

## Best Possible Solution

Land the one-token ACP argument normalization with regression coverage at the builder or end-to-end client spawn boundary, while preserving `--server-verbose`, ACP server behavior, and session semantics.

## Reproduction Assessment

Yes. Current main prepends `acp` to unnormalized `serverArgs`, so a caller passing `["acp", "--verbose"]` produces duplicated argv before the ACP server spawn; I did not run the CLI in this read-only review.

## Solution Assessment

No, not fully as proposed. The production normalization is the narrow maintainable fix, but the regression test should cover `buildServerArgs` directly or assert the `createAcpClient` spawn argv instead of calling the post-builder helper.
