## Summary

Keep this PR open: the ACP argv duplication is still present on current main and the production change is narrow, but the added regression test does not exercise the changed `buildServerArgs` path and would stay green if the fix were reverted.

## What This Changes

The PR drops one leading `acp` token from ACP client `serverArgs` before prepending the canonical subcommand, and adds a spawn invocation test.

## Best Possible Solution

Land the one-token normalization with regression coverage at `buildServerArgs` or the `createAcpClient` spawn boundary while preserving verbose handling, env stripping, Windows wrapper behavior, and session semantics.

## Reproduction Assessment

Yes. Source inspection of current main shows the reported `--server-args acp --server-verbose` path flows through CLI option forwarding into `buildServerArgs`, producing duplicated `acp` before spawn; I did not execute the CLI in this read-only review.

## Solution Assessment

No, not fully as proposed. The production normalization is the right narrow fix, but the regression test is attached to a helper that receives already-built args and should instead cover the builder or the `createAcpClient` spawn argv.
