Keep open: the reported ACP client bug still exists on current main, and the production fix is narrow, but this PR should not merge as-is because its added regression test bypasses the normalization code and the branch is currently conflicting.

## What This Changes

The PR changes ACP client server-argument construction to drop one leading `acp` from forwarded args and adds a regression test.

## Best Possible Solution

Land a rebased or replacement patch that keeps the narrow leading-`acp` normalization and adds regression coverage through `buildServerArgs` or `createAcpClient`.

## Reproduction Assessment

Yes. Source inspection on current main shows `buildServerArgs` prepends `acp` to caller-provided `serverArgs`, so `--server-args acp --server-verbose` becomes `openclaw acp acp --verbose`.

## Solution Assessment

No, not as submitted. The production normalization is the right narrow fix, but the regression test should exercise the argument builder or end-to-end client spawn path instead of an already-built spawn invocation.
