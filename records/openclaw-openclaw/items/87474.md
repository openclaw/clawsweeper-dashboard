## Summary

Keep open. The PR targets a real linked Control UI bug, but as written it gates existing session-scoped abort UI on local stream/sending state and would hide Stop for recovered or disconnected active sessions; the external PR proof is also only build/typecheck/tests.

## What This Changes

The branch changes `ui/src/ui/views/chat.ts` to require local send/stream activity before showing in-progress status, queue steering, or Stop controls.

## Best Possible Solution

Split visible busy/status from abort capability: keep the completed-run composer idle while preserving session-scoped Stop/Steer for real active recovered runs, backed by focused tests and redacted Control UI proof.

## Reproduction Assessment

No live reproduction was run in this read-only review, but the source path is clear: current main can force in-progress/Stop from `canAbort` even when local stream/sending is idle. The linked issue provides concrete Windows Chrome/OpenClaw 2026.5.26 reproduction steps.

## Solution Assessment

No. The badge direction is narrow, but gating all abort affordances on `sending || stream !== null` is not the best fix because it drops the existing session-scoped abort behavior.

- **[P2] Preserve recovered active-run abort controls:** `ui/src/ui/views/chat.ts:1627`
