## Summary

Keep open. Current main still accepts ACP child runs before the initial task is durable in the child transcript, and `sessions_history` reads transcript messages rather than ACP task metadata, so the accepted-but-empty-history failure remains unresolved.

## What This Changes

The issue asks for ACP Codex child sessions to persist or expose the initial long task once accepted, instead of returning an accepted child session whose transcript can remain empty.

## Best Possible Solution

Persist the initial ACP task as a child transcript user message, or expose an equivalent queued-task event through `sessions_history`, before an accepted child can stall silently.

## Reproduction Assessment

Yes, at source level. The issue gives a concrete ACP Codex contrast repro, and current main still accepts the child before any transcript row is durable while ACP transcript persistence happens only after `runTurn` completes.

## Solution Assessment

Yes. The narrow maintainable fix is to move initial-task durability ahead of ACP runtime execution, or make the queued task visible through the same history surface, with dedupe coverage for successful turns.

## Repair Work Prompt

Fix the ACP child-session durability bug from https://github.com/openclaw/openclaw/issues/47951. Current main can return an accepted ACP child session before the initial task is durable in the child transcript, and the ACP command path only persists the user row after `runTurn` completes. Make accepted ACP child tasks visible through `sessions_history` before a stalled or failing runtime can leave an empty transcript, preferably by persisting the initial task as the child transcript user row before dispatch or by routing an equivalent queued-task event through the same history surface. Avoid duplicating the prompt after successful ACP turns. Keep the fix in generic ACP/session paths, not Codex-specific policy, and do not add a new config option. Add focused regression coverage for stalled/failing ACP dispatch and successful-turn dedupe. Add a `CHANGELOG.md` fix entry because this is user-facing reliability behavior.
