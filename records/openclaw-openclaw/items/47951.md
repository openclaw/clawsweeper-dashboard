## Summary

Keep open. Current main still has the same accepted-before-durable-history shape: ACP spawn returns an accepted child session after gateway dispatch, while the ACP command path only writes the child transcript user row after `runTurn` completes. The related closed #63702 covers a narrower Codex ACP internal-error path and does not supersede this empty-transcript durability bug.

## What This Changes

The issue requests that accepted ACP Codex long/complex child sessions start reliably or at least persist the initial user task in the child transcript for debugging and recovery.

## Best Possible Solution

Make the ACP spawn/agent contract durable before an accepted child can go silent: persist the initial task as a child transcript user row, or expose an equivalent first-task event through `sessions_history`, before `runTurn` can stall. The fix should dedupe post-turn persistence and add regression coverage for stalled/failing ACP dispatch so successful turns do not duplicate the prompt.

## Reproduction Assessment

Yes. The issue provides a concrete ACP Codex contrast repro, and current source confirms a design-level reproduction path where `accepted` can be returned before any child transcript message exists; I did not run the live repro because the review is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to move initial task durability ahead of the ACP runtime turn, or make the queued task visible through the same history surface; relying on the background task ledger alone is not enough because `sessions_history` reads transcript messages.
