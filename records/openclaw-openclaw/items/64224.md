## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, so a billing-disabled LiteLLM profile can block sibling models that share the same profile; the remaining work is a product policy choice for proxy aggregators versus direct provider account exhaustion.

## What This Changes

Request model-scoped or configurable billing cooldown behavior so one upstream LiteLLM model returning a 402 does not disable every model behind the same LiteLLM auth profile.

## Best Possible Solution

Add a documented proxy-aware billing cooldown policy for LiteLLM-style aggregators while preserving profile-wide disables for direct provider account or key exhaustion, with regression coverage for shared-profile sibling models.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed one LiteLLM auth profile with active disabledUntil and disabledReason: "billing", configure sibling LiteLLM fallback models, and current fallback code skips those same-provider candidates.

## Solution Assessment

No. The current profile-wide billing disable is documented and defensible for direct provider keys, but it is not the best fit for proxy aggregators; the safer solution is an explicit proxy-aware or operator-configured policy rather than globally changing billing failures to model-scoped behavior.
