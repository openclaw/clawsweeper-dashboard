## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, and that profile-wide state blocks sibling LiteLLM models sharing the same profile; this needs maintainer policy for proxy aggregators versus direct provider billing exhaustion.

## What This Changes

The issue requests model-scoped or configurable billing cooldown behavior for LiteLLM-style proxy providers so one upstream 402 does not disable all sibling proxy models.

## Best Possible Solution

Add an explicit LiteLLM/proxy-aware billing cooldown policy while preserving profile-wide disables for direct provider account or key exhaustion, then document it and add regression coverage for shared-profile sibling models.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models, and current fallback code skips those same-provider candidates.

## Solution Assessment

No. The current profile-wide billing disable is defensible for direct provider keys, but it is not the best fit for proxy aggregators; the safer fix needs an explicit proxy-aware or operator-configured policy instead of a global billing-scope change.
