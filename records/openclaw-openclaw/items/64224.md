## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, so one billing failure on a shared LiteLLM profile can block sibling models; the remaining work is a maintainer policy decision for proxy aggregators versus direct-provider key exhaustion.

## What This Changes

Request model-scoped or configurable billing cooldown behavior so a 402 from one upstream behind LiteLLM does not disable every model sharing the LiteLLM auth profile.

## Best Possible Solution

Define an explicit proxy/operator billing cooldown policy for aggregator providers such as LiteLLM, preserve account/key disables for direct providers, and add docs plus regression coverage for shared-profile sibling models.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a shared `litellm:default` auth profile with active `disabledUntil` and `disabledReason: "billing"`, then configure sibling LiteLLM fallback models and observe current fallback code skip them as same-provider billing candidates.

## Solution Assessment

No for the current behavior as a complete solution. Profile-wide billing disables are defensible for direct provider keys, but proxy aggregators need an explicit policy rather than a global change that would weaken direct-provider billing protection.
