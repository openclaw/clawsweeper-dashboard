## Summary

Keep open. Current main still treats billing-classified failures as profile-wide disabled state, while the only model-scoped bypass is for rate-limit cooldowns; LiteLLM proxy aggregation needs an explicit maintainer policy rather than conservative cleanup closure.

## What This Changes

The issue requests LiteLLM/proxy-aware billing cooldown behavior so one upstream model's billing failure does not disable every model sharing the same LiteLLM auth profile.

## Best Possible Solution

Add an explicit LiteLLM/proxy-aware billing cooldown policy, likely provider-owned or opt-in, while preserving profile-wide disables for direct provider account/key exhaustion and covering it with docs plus regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a single `litellm:default` profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models sharing that profile, and the current fallback path treats every sibling as unavailable.

## Solution Assessment

No. The current profile-wide billing disable is consistent with direct-provider billing docs, but it is not the best solution for proxy aggregators; maintainers need to choose a LiteLLM/proxy policy rather than globally changing billing semantics.
