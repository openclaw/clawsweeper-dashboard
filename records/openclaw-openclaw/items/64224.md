## Summary

Keep open. Current main still treats billing failures as auth-profile-wide disabled state, and the existing model-scoped bypass applies only to rate-limit cooldowns, so LiteLLM/proxy users can still have sibling models blocked after one upstream billing-shaped failure. The fix needs maintainer policy because direct-provider billing exhaustion and proxy-upstream billing failures should not be changed with the same global rule.

## What This Changes

The issue asks for billing cooldowns behind LiteLLM/proxy providers to avoid disabling every model on the shared provider/profile after one upstream model returns a billing error.

## Best Possible Solution

Add an explicit LiteLLM/proxy-aware billing cooldown policy, preferably provider-owned or opt-in, with focused fallback/auth-profile tests and docs while preserving profile-wide disables for direct-provider billing failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: classify a LiteLLM 402 `insufficient_balance` response as billing, persist `litellm:default` with active `disabledUntil`/`disabledReason`, then observe `isProfileInCooldown` and fallback decisions block sibling models on that shared profile.

## Solution Assessment

No. The current profile-wide billing disable is intentional for direct providers, but it is not the best fit for aggregator proxies; the safer solution is an explicit LiteLLM/proxy policy or provider-owned seam instead of a global semantic change.
