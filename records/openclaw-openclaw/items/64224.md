## Summary

Keep open. Current main and v2026.5.19 still classify generic HTTP 402 responses as billing, persist billing failures as profile-wide disabled auth state, and use one LiteLLM provider/auth profile for multiple proxied models, so the reported proxy/upstream billing-scope problem remains unresolved.

## What This Changes

The issue asks OpenClaw to avoid disabling an entire LiteLLM provider profile when only one upstream model behind the proxy returns a billing error.

## Best Possible Solution

Define an additive provider/proxy failure-scope contract that keeps direct-provider account billing lockouts profile-wide while allowing aggregator providers such as LiteLLM to scope upstream failures to the affected model or declared upstream bucket.

## Reproduction Assessment

Yes, by source inspection rather than a live LiteLLM setup: HTTP 402 maps to billing, billing writes profile-wide disabled state, and LiteLLM models share the same provider auth profile.

## Solution Assessment

No, not as a blanket model-scoped billing or zero-backoff change. The maintainable fix needs an explicit proxy/upstream billing-scope contract so direct-provider account billing lockouts keep their intended behavior.
