- `impact:auth-provider`: The issue directly concerns provider auth-profile cooldown state, provider routing, and model availability behind a proxy.

## Summary

Keep open. Current main and v2026.5.18 still treat generic or insufficient-balance HTTP 402 responses as billing and persist billing failures as profile-wide disables, so the LiteLLM proxy case is not cleanup-closeable; the fix needs a maintainer decision on proxy/upstream billing scope.

## What This Changes

The issue asks OpenClaw to avoid disabling every model behind a LiteLLM/proxy provider when one upstream model returns a billing-style 402, or to add a way to disable/tune that cooldown behavior.

## Best Possible Solution

Define an explicit provider/proxy failure-scope contract for billing errors, including LiteLLM upstream-model versus proxy-key/account scope, while preserving profile-wide billing disables for direct providers and documenting any config or SDK seam.

## Reproduction Assessment

Yes by source inspection rather than a live LiteLLM setup. A generic or insufficient-balance HTTP 402 classifies as billing, `markAuthProfileFailure` persists `disabledUntil` for the shared profile, and sibling models cannot bypass active `disabledUntil`.

## Solution Assessment

No as a blanket model-scoped billing change. The maintainable solution is an explicit proxy/upstream billing-scope policy or config/SDK seam that keeps direct-provider account billing lockouts profile-wide.
