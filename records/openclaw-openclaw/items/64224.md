## Summary

Keep open. Current main and the latest shipped release still classify generic or insufficient-balance HTTP 402 failures as billing and persist billing failures as a profile-wide disabled state; solving the LiteLLM proxy case needs an explicit proxy billing policy rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid disabling every model behind a LiteLLM proxy when one upstream model returns a billing-style 402, or to make billing cooldowns disableable/configurable.

## Best Possible Solution

Add an explicit LiteLLM/proxy-owned billing-cooldown policy or plugin/SDK seam that can scope upstream failures while preserving profile-wide billing disables for direct providers.

## Reproduction Assessment

Yes, by source inspection rather than a live LiteLLM setup. A 402 insufficient-balance style response classifies as billing, `markAuthProfileFailure` writes `disabledUntil` for the shared profile, and sibling models cannot bypass active `disabledUntil`.

## Solution Assessment

No, not as a blanket model-scoped billing change. The safer maintainable direction is an explicit LiteLLM/proxy policy or config/SDK seam that keeps direct-provider billing lockouts profile-wide.
