## Summary

Keep open. Current main still routes LiteLLM-style 402/insufficient-balance responses through the generic billing path, then stores the result as profile-wide disabled state, while the only model-scoped bypass remains rate-limit-only.

## What This Changes

The issue requests model-scoped or proxy-aware billing cooldown behavior so a LiteLLM upstream 402 does not disable every model sharing the same auth profile.

## Best Possible Solution

Add an explicit provider-owned or opt-in proxy billing-cooldown policy that can scope LiteLLM upstream failures without changing profile-wide disables for direct provider billing failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a LiteLLM 402 insufficient-balance response classifies as billing, `markAuthProfileFailure` persists `disabledUntil` on the shared profile, and sibling models fail `isProfileInCooldown` because billing disables are profile-wide.

## Solution Assessment

No, not as a global model-scoped billing change. The safer solution is an explicit LiteLLM/proxy policy or provider SDK seam that preserves direct-provider billing lockouts.
