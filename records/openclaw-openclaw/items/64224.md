## Summary

Keep open. Current main still treats billing failures as auth-profile-wide disabled state and skips sibling same-provider models, while LiteLLM is documented as a single provider that can route multiple backend models; the remaining work needs a proxy-aware cooldown policy decision rather than cleanup closure.

## What This Changes

Request model-scoped or configurable billing cooldown behavior so one upstream 402 behind a LiteLLM proxy does not disable every model sharing the same provider auth profile.

## Best Possible Solution

Define and implement an explicit proxy/operator billing-cooldown scope for LiteLLM-style aggregators while preserving profile-wide billing disables for direct provider key or account exhaustion, then document it and add regression coverage for shared-profile sibling models.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models that share that profile, and current fallback code skips those candidates.

## Solution Assessment

No. The current profile-wide billing disable is defensible for direct providers, but it is not the narrowest fit for proxy aggregators; maintainers should choose an explicit proxy-aware policy before implementation.
