## Summary

Keep open. Current main still routes LiteLLM-style 402 or insufficient-balance failures through the generic billing path and persists them as profile-wide disabled state, while the model-scoped bypass remains rate-limit-only. This needs maintainer policy for proxy billing semantics rather than cleanup closure.

## What This Changes

Request model- or proxy-aware billing cooldowns so a 402 from one upstream model behind LiteLLM does not disable the whole shared LiteLLM auth profile.

## Best Possible Solution

Add an explicit provider-owned or opt-in proxy billing-cooldown policy that can scope LiteLLM upstream failures without changing direct-provider billing lockouts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: LiteLLM-style 402 or insufficient-balance text classifies as billing, `markAuthProfileFailure` stores `disabledUntil` on the shared profile, and sibling models fail `isProfileInCooldown` while that disabled window is active.

## Solution Assessment

No, not as a global model-scoped billing change. The safer path is an explicit LiteLLM/proxy policy or SDK seam that preserves profile-wide billing disables for direct providers.
