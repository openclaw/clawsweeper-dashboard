## Summary

Keep open. Current main still treats billing-classified failures as profile-wide disabled state, and the documented/tested fallback behavior skips sibling LiteLLM models that share that profile; changing this needs maintainer policy for proxy aggregators versus direct provider billing exhaustion.

## What This Changes

Request billing cooldowns for LiteLLM/proxy setups to avoid disabling every model behind one shared provider profile when one upstream model returns a billing error.

## Best Possible Solution

Add an explicit LiteLLM/proxy-aware billing cooldown policy, likely provider-owned or opt-in, while preserving profile-wide disables for direct provider account or key exhaustion and covering it with docs plus regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models sharing that profile, and current fallback code skips those same-provider candidates.

## Solution Assessment

No. The current profile-wide billing disable is consistent with direct-provider billing docs, but it is not the best fit for proxy aggregators; the safer fix needs an explicit LiteLLM/proxy policy rather than globally changing billing semantics.
