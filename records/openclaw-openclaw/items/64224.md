## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, and active `disabledUntil` blocks sibling LiteLLM models that share the same profile; the remaining decision is the policy for proxy aggregators versus direct provider billing exhaustion.

## What This Changes

Request proxy-aware or model-scoped billing cooldown behavior, or a supported disable/zero-backoff option, so one upstream 402 behind LiteLLM does not disable every model on the shared provider profile.

## Best Possible Solution

Add an explicit proxy/operator billing cooldown policy for LiteLLM-style aggregators while preserving profile-wide disables for direct provider account or key exhaustion, then document the config and add regression coverage for shared-profile sibling models.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models, and current fallback code skips those same-provider candidates.

## Solution Assessment

No. The current profile-wide billing disable is defensible for direct provider keys, but it is not the best fit for proxy aggregators; the safer fix is an explicit proxy-aware or operator-configured policy instead of globally changing billing failures to model-scoped behavior.
