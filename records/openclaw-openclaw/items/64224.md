## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, so one LiteLLM proxy profile can block sibling models after an upstream 402; the related billing-message path is now covered on main, but cooldown scope and a supported zero/disable opt-out remain unresolved.

## What This Changes

The issue asks OpenClaw to avoid disabling every model behind a shared LiteLLM/proxy auth profile when one upstream model returns a billing error, either by model-scoping billing failures or allowing billing backoff to be disabled.

## Best Possible Solution

Add an explicit proxy-aware billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented opt-in zero/disable control for operators who manage failover externally. Preserve profile-wide disables for direct provider account/key exhaustion, update docs/config help, and add regression coverage for one shared LiteLLM profile serving multiple models.

## Reproduction Assessment

Yes. Seed a single LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, then request a sibling LiteLLM model through fallback; current source and tests show same-provider siblings are skipped.

## Solution Assessment

No. The current profile-wide billing disable is conservative for direct provider keys, but it is not the best fit for proxy aggregators; the safer maintainable direction is a deliberate proxy policy or explicit operator opt-out with guardrails.
