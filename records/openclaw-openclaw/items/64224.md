## Summary

Keep open. Current main and v2026.5.18 still classify generic HTTP 402 responses as billing, persist billing failures as profile-wide `disabledUntil`, and LiteLLM setup uses a shared provider profile, so the reported proxy/upstream billing-scope problem remains a real product-policy gap rather than a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to avoid disabling every model behind a LiteLLM proxy when one upstream model returns a billing error.

## Best Possible Solution

Define a provider/proxy failure-scope contract for billing errors that keeps direct-provider account/key billing disables profile-wide while allowing aggregator providers such as LiteLLM to scope upstream failures to the affected model or declared bucket.

## Reproduction Assessment

Yes, by source inspection rather than a live LiteLLM setup: generic HTTP 402 classifies as billing, billing writes a profile-wide disabled window, and different LiteLLM models share the same provider auth profile.

## Solution Assessment

No, not as a blanket model-scoped billing change. The maintainable solution needs an explicit proxy/upstream billing-scope policy or SDK/provider contract that does not break direct-provider account billing lockouts.
