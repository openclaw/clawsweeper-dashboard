## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, so a single LiteLLM proxy profile can block sibling models after one upstream billing error. The related #66315 work in the issue context addressed user-facing billing-message classification only; it did not change cooldown scope or add a supported zero/disable opt-out.

## What This Changes

Request model-scoped or proxy-aware billing cooldowns, or a supported billing-backoff opt-out, so one upstream model behind a LiteLLM proxy does not disable the shared provider profile.

## Best Possible Solution

Add a deliberate proxy-aware billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented opt-in disable/zero backoff for operators who manage failover externally. The implementation should preserve profile-wide billing disables for true key/account exhaustion and add focused regression coverage for one shared LiteLLM profile serving multiple models.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to seed a single LiteLLM auth profile with a billing `disabledUntil` after one model failure, then request a sibling LiteLLM model; current source and tests show that same-provider billing siblings are skipped.

## Solution Assessment

No. The current profile-wide billing disable is intentionally conservative for direct provider keys, but it is not the best fit for proxy aggregators; a scoped proxy policy or explicit operator opt-out is the safer maintainable direction.
