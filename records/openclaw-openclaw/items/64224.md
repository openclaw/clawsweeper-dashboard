## Summary

Keep open. Current main still turns billing-classified failures into auth-profile-wide disabled state, so a single LiteLLM proxy profile can block sibling models; the related billing-message fix addressed user-facing classification, not cooldown scope or a supported zero/disable opt-out.

## What This Changes

The issue requests model-scoped or configurable billing cooldown behavior so a LiteLLM proxy profile is not disabled for all models after one upstream model returns a billing error.

## Best Possible Solution

Add an explicit proxy-aware billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented opt-in zero/disable control for operators who manage failover externally. Preserve profile-wide disables for direct provider account/key exhaustion, update config docs, and add regression coverage for one shared LiteLLM profile serving multiple models.

## Reproduction Assessment

Yes. Seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models, and current source plus `model-fallback.test.ts` show the sibling model is skipped as the same provider.

## Solution Assessment

No. The current profile-wide billing disable is conservative for direct provider keys, but it is not the best fit for proxy aggregators; the maintainable fix should be an explicit proxy/operator policy rather than silently changing all billing failures to model-scoped behavior.
