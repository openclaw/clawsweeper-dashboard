## Summary

Keep open. Current main still treats billing-classified failures as auth-profile-wide disabled state, so one LiteLLM proxy profile can block sibling models; the related closed billing-message work did not change cooldown scope or add a supported zero/disable opt-out.

## What This Changes

The issue requests proxy-aware billing cooldown behavior so a billing failure from one LiteLLM upstream model does not disable the shared provider/auth profile for all sibling models.

## Best Possible Solution

Add an explicit proxy-aware billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented opt-in zero/disable control for operators who manage failover externally, while preserving profile-wide disables for direct provider account/key exhaustion.

## Reproduction Assessment

Yes. Seed one LiteLLM auth profile with active `disabledUntil` and `disabledReason: "billing"`, configure sibling LiteLLM fallback models, and current source plus `model-fallback.test.ts` show the sibling model is skipped as the same provider.

## Solution Assessment

No. The current profile-wide billing disable is conservative for direct provider keys, but it is not the best fit for proxy aggregators; the maintainable fix should be an explicit proxy/operator policy rather than silently making all billing failures model-scoped.
