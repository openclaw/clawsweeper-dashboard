## Summary

Keep open. Current main still treats billing failures as auth-profile disabled state and skips same-provider fallback models behind the same LiteLLM provider/profile; changing that needs maintainer policy for proxy aggregators versus direct provider billing exhaustion.

## What This Changes

The issue requests LiteLLM/proxy-aware billing cooldown behavior so one upstream model's billing error does not disable every model sharing the same provider auth profile.

## Best Possible Solution

Add an explicit LiteLLM/proxy-aware billing cooldown policy, preferably provider-owned or opt-in, while preserving direct-provider billing disables and adding focused fallback, auth-profile, LiteLLM, and docs coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mark `litellm:default` with active `disabledUntil`/`disabledReason: "billing"`, configure sibling LiteLLM fallback models sharing that profile, and current fallback skips the sibling candidates.

## Solution Assessment

No. The current profile-wide billing disable is consistent with direct-provider billing docs/tests, but it is not the best fit for aggregator proxies; the safer direction is an explicit LiteLLM/proxy policy instead of a global billing semantic change.
