## Summary

Keep open. Current main still omits the `file` alias from the read/write/edit display resolver and read-start warning guard, and the issue is already paired with an open closing PR.

## What This Changes

The issue requests that Read, Write, and Edit tool summaries show the file path when tool arguments use the `file` alias.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/77312 as a narrow display-layer compatibility fix, or close it and align diagnostics/tests around canonical `path` only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `{ file: "example.txt" }` into Read/Write/Edit display resolution or the embedded read-start handler, and current main has no branch that reads that value.

## Solution Assessment

Unclear until maintainers settle the alias contract. If display-layer alias tolerance is intended, the paired PR’s narrow display-only approach is maintainable; execution alias behavior should remain separate.
