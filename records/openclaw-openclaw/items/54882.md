Keep open. Current main and the latest release still omit the `file` alias from the Read/Write/Edit display and read-start diagnostic paths, and an open unmerged closing PR already tracks the focused implementation path.

## What This Changes

Show Read/Write/Edit verbose tool summaries with file path detail when tool arguments use the `file` alias.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/77312 is reviewed; either land a narrow display/diagnostic fix with coverage and real behavior proof, or standardize display, diagnostics, tests, and expectations on canonical `path` only.

## Reproduction Assessment

Yes, at source level. Passing `{ file: "example.txt" }` into the current Read/Write/Edit display path cannot resolve a path because `record.file` is not read, and the read-start guard also ignores that alias.

## Solution Assessment

Unclear. A narrow resolver/display fix is plausible, but maintainers should first settle whether display and diagnostics intentionally tolerate aliases now that execution validation requires canonical `path`.
