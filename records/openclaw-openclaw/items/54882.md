## Summary

Keep open. Current main still omits the `file` alias from the Read/Write/Edit display resolver and read-start diagnostic path, and an open PR with closing syntax is already paired with this issue.

## What This Changes

The issue requests that verbose/block-streaming Read, Write, and Edit tool summaries show the file path when tool args use the `file` alias.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/77312 as a narrow display-layer compatibility fix, or close it and align diagnostics/docs around canonical `path` only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `{ file: "example.txt" }` into Read/Write/Edit display resolution or the read-start handler, and current main has no branch that reads that value.

## Solution Assessment

Unclear until maintainers settle the alias contract. If display-layer alias tolerance is intended, the paired PR’s narrow display-only approach is the maintainable fix; execution alias behavior should remain separate.
