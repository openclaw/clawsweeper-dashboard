## Summary

Keep open. Current main and the latest release still omit the `file` alias from the read/write/edit display path resolver and the read-start diagnostic guard, and an open PR with closing syntax is already paired with this issue, so cleanup closure would be premature.

## What This Changes

The issue requests consistent verbose/block-streaming file path display when read/write/edit tool arguments use the `file` alias instead of canonical `path`.

## Best Possible Solution

End with one consistent path-parameter contract across runtime validation, verbose display summaries, read-start diagnostics, and regression tests: either land an updated #55121-style compatibility fix or intentionally document and enforce canonical `path` everywhere.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass `{ file: "example.txt" }` into read/write/edit display resolution or the read-start handler and current main has no branch that reads that value, while focused tests only cover `file_path` or `path`.

## Solution Assessment

Unclear. Adding `file` to display and diagnostics is the narrowest fix if legacy alias display tolerance is desired, but current runtime validation deliberately rejects those aliases, so maintainer review should settle that contract before merge or closure.
