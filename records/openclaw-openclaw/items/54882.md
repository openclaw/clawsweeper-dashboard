## Summary

Keep open. Current main and the latest release still omit the `file` alias from the Read/Write/Edit display path resolver and read-start guard, while https://github.com/openclaw/openclaw/pull/77312 is an open closing PR for the same display-layer bug.

## What This Changes

The issue asks Read/Write/Edit verbose tool summaries to show the file path when tool arguments use the `file` alias.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/77312 and either land a narrow display/diagnostic fix with regression coverage and real behavior proof, or explicitly standardize display, diagnostics, and tests on canonical `path` only.

## Reproduction Assessment

Yes, at source level. Passing `{ file: "example.txt" }` into the current Read/Write/Edit display path cannot resolve a path because `record.file` is not read, and the embedded read-start guard also ignores that alias.

## Solution Assessment

Unclear. The shared display resolver change is narrow and plausible, but read-start diagnostics should stay aligned with the canonical execution contract unless maintainers intentionally support aliases end to end.
