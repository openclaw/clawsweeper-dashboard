The issue reports that verbose Read/Write/Edit tool summaries omit the file path when tool arguments use the `file` alias.

## Best Possible Solution

Keep this issue open while https://github.com/openclaw/openclaw/pull/77312 is reviewed; either land a narrow display/diagnostic fix with regression coverage and real behavior proof, or explicitly align display, diagnostics, tests, and expectations on canonical `path` only.

## Reproduction Assessment

Yes, at source level. Passing `{ file: "example.txt" }` into the current Read/Write/Edit display path cannot resolve a path because `record.file` is not read, and the read-start warning guard also ignores that alias.

## Solution Assessment

Unclear. The narrow resolver and diagnostic fix in the paired PR is plausible, but maintainers should first settle whether display/diagnostics intentionally tolerate legacy aliases now that execution validation requires canonical `path`.
