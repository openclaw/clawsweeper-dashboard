## Summary

Keep open. Current main still omits the `file` alias from the read/write/edit display resolver and read-start warning guard, and an open unmerged PR with closing syntax is already tracking the implementation path. Runtime validation now requires canonical `path`, so maintainers should decide whether display/diagnostics remain alias-tolerant before closing this.

## What This Changes

Ensure Read/Write/Edit verbose tool summaries and read-start diagnostics show the file path when tool args use the `file` parameter alias.

## Best Possible Solution

Settle the path-parameter contract in one place: either land an updated #55121-style display/diagnostic alias fix with regression coverage for `file` and `filePath`, or intentionally standardize on canonical `path` by aligning diagnostics, tests, and user-facing expectations.
