## Summary

Keep open. Current main still omits the `file` alias in the display and read-start diagnostic paths, and open unmerged closing PRs already track the implementation path.

## What This Changes

The issue asks for read/write/edit verbose tool summaries to show file paths when tool arguments use the `file` alias.

## Best Possible Solution

Review and land one focused compatibility PR that aligns the resolver, read-start diagnostics, generated display config, and regression coverage, or deliberately standardize on canonical `path` with matching diagnostics and expectations.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass `{ file: "README.md" }` into read/write/edit display resolution or the read-start handler and current main has no branch that reads `record.file`.

## Solution Assessment

Unclear. Adding `file` to the existing display alias handling is the narrow fix if compatibility is desired, but current execution validation is canonical-path only, so PR review should settle that contract.
