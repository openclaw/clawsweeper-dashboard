## Summary

Keep open: current main still has no opt-in vi/vim modal mode for the TUI composer, and the linked implementation PR was closed unmerged after the author concluded the behavior belongs in the upstream editor component rather than OpenClaw's shortcut wrapper.

## What This Changes

Request an opt-in vi/vim-style modal keybinding mode for the TUI input editor.

## Best Possible Solution

Add first-class vi-mode support in the editor layer, then expose a default-off OpenClaw config surface and TUI docs once mode semantics and Escape behavior are settled.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, not broken existing behavior. Source, docs, dependency-type inspection, and related PR context give high confidence that current main has no supported in-place vi/vim input mode.

## Solution Assessment

Unclear until maintainers choose the owner boundary and config shape. The narrowest maintainable direction appears to be editor-level vi support rather than reimplementing a partial vi parser in OpenClaw's shortcut wrapper.
