## Summary

Keep open: current main and v2026.5.18 still do not expose an opt-in vi/vim modal input mode, and the related closed implementation PR explicitly deferred the work to the pi-tui editor layer rather than landing it in OpenClaw.

## What This Changes

Add an opt-in vi/vim modal keybinding mode to the OpenClaw TUI input editor, likely controlled by config or environment.

## Best Possible Solution

Implement first-class vi-mode support at the editor layer, then expose a default-off OpenClaw config/docs surface once supported commands and Escape semantics are defined.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source, docs, latest-release, dependency-source, and related PR checks give high confidence that current main has no supported in-place vi/vim input mode.

## Solution Assessment

Unclear until maintainers choose the owner boundary and config shape. The narrowest maintainable direction appears to be editor-level vi support in pi-tui, followed by a small OpenClaw opt-in surface, rather than a partial CustomEditor key intercept layer.
