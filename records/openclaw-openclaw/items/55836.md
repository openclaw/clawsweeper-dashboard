## Summary

Keep open: current main and the latest release still do not provide first-class vi/vim modal editing for the TUI input editor. The companion implementation at https://github.com/openclaw/openclaw/pull/55975 was closed unmerged, and the current extension editor API is not wired into the active `openclaw tui` editor path, so this still needs maintainer product/API direction rather than cleanup closure.

## What This Changes

The issue requests an opt-in vi/vim-style modal keybinding mode for the OpenClaw TUI input editor, likely through TUI config or an environment switch.

## Best Possible Solution

Keep this open until maintainers choose whether vi behavior belongs in pi-tui, OpenClaw core, or a real active editor-component plugin path, then approve a default-preserving public configuration shape.

## Reproduction Assessment

Yes. Current-main source and the latest release snapshot have no TUI vi-mode config, no modal editor state, no vi motion handling, and no pi-tui vi-mode option passed into `CustomEditor`.

## Solution Assessment

Unclear. The requested outcome is reasonable, but the best implementation path needs maintainer direction because a hand-rolled OpenClaw parser may be the wrong layer compared with first-class pi-tui support or a fully wired editor-component plugin seam.
