## Summary

Keep open: current main and v2026.6.8 still lack first-class vi/vim modal editing for the TUI input editor, and the related implementation PR was closed unmerged after the author identified the editor ownership boundary as unresolved.

## What This Changes

The issue requests an optional vi/vim-style modal keybinding mode for the TUI input editor, likely controlled by TUI config or an environment setting.

## Best Possible Solution

Keep this open until maintainers choose the pi-tui versus OpenClaw owner boundary and the public config/API shape, then implement a default-preserving vi mode with focused editor, PTY, config, and docs coverage.

## Reproduction Assessment

Yes. The missing capability is source-reproducible because current main and v2026.6.8 have no TUI vi-mode config, no modal editor state, no vi motion handling, and no pi-tui vi-mode option wired into `CustomEditor`.

## Solution Assessment

Unclear. The requested outcome is reasonable, but the proposed config/env switch and a hand-rolled `CustomEditor` parser are not clearly the best path until maintainers choose whether vi behavior belongs upstream in pi-tui or in OpenClaw.
