## Summary

Keep open: current main and v2026.5.18 still do not provide an opt-in vi/vim modal input mode, and the closed implementation PR left the owner-boundary question unresolved rather than landing a fix.

## What This Changes

Request to add a configurable vi/vim keybinding mode for the TUI input editor, with normal/insert modes and standard vi motions inside the composer.

## Best Possible Solution

Add first-class vi-mode support in the pi-tui Editor layer, then expose a default-off OpenClaw config and docs surface once Escape behavior and supported vi commands are defined.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, not broken existing behavior. Source, docs, latest-release, dependency-source, and related PR checks give high confidence that current main has no supported in-place vi/vim input mode.

## Solution Assessment

Unclear until maintainers choose the owner boundary and config shape. The narrowest maintainable direction appears to be editor-level vi support in pi-tui, followed by a small OpenClaw opt-in surface, rather than a partial CustomEditor key intercept layer.
