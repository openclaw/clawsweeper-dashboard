## Label Justifications

- `P2`: The issue describes a real operational gateway availability failure on Linux, but it is bounded to managed service lifecycle behavior and needs policy review before implementation.
- `impact:crash-loop`: Unexpectedly restarting the gateway can reclaim port 18789 and break the intended supervised service, which is a process-level availability problem.

## Summary

Keep open. Current `openclaw status` looks read-only, but current doctor/readiness/start paths can still restart an installed stopped Linux user-systemd gateway unit, and changing that default needs a maintainer lifecycle-policy decision because recovery starts are partly documented and tested today.

## What This Changes

The issue asks OpenClaw helper and repair commands to avoid starting a stopped and disabled user-level Linux systemd gateway unit unless the operator explicitly requests lifecycle startup.

## Best Possible Solution

Define the Linux service lifecycle contract so status stays non-mutating, helper recovery does not override disabled or masked operator intent, and explicit start/install/restart flows remain the only paths that bring a disabled user unit back with docs and regression coverage.

## Reproduction Assessment

No high-confidence exact reproduction for the current `openclaw status` sequence: current source makes status look read-only and no live Ubuntu/systemd run was performed. A closely related current-main helper/doctor path is source-reproducible because stopped installed services can flow into `systemctl --user restart`.

## Solution Assessment

Unclear. A broad new flag is probably not the first fix; the safer direction is to distinguish read-only diagnostics from explicit lifecycle repair and teach Linux systemd handling to respect disabled or masked units.
