## Summary

Keep open. Current main and the latest release still contain source-confirmed helper/start paths that can restart a stopped installed Linux user-systemd gateway service, while the exact `openclaw status` sequence appears read-only and needs live policy/repro follow-up.

## What This Changes

The issue reports that helper commands may restart a stopped and disabled user-level Linux systemd `openclaw-gateway` service.

## Best Possible Solution

Define the Linux service lifecycle contract so diagnostic/status paths stay read-only and helper repair/recovery paths respect disabled or masked user-systemd operator intent unless an explicit lifecycle command is used.

## Reproduction Assessment

No high-confidence live reproduction was run, and the exact `openclaw status` sequence does not reproduce from source inspection. A closely related current-main helper/start path is source-reproducible because stopped installed services can flow to `systemctl --user restart`.

## Solution Assessment

Unclear; the issue identifies a real lifecycle gap, but the fix should follow a maintainer decision on whether interactive or `--yes` repairs may start disabled units. A broad new flag is not necessarily the best first step if the service state can distinguish disabled or masked operator intent.
