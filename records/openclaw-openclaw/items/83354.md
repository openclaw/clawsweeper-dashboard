## Summary

Keep open. Current main does not show the exact `openclaw status` sequence as mutating, but helper readiness and doctor repair paths can still start or restart an installed stopped Linux user-systemd gateway, so the operator-intent lifecycle question remains unresolved.

## What This Changes

The issue reports that helper commands can restart a stopped and disabled user-level Linux systemd `openclaw-gateway` service, unexpectedly reclaiming port 18789.

## Best Possible Solution

Define the Linux service lifecycle contract so diagnostics stay non-mutating, helper recovery does not override disabled or masked operator intent, and only explicit lifecycle commands can bring a disabled unit back with clear docs and regression coverage.

## Reproduction Assessment

No for the exact `openclaw status` sequence: current source makes status look read-only and no live Ubuntu systemd run was performed. Yes at source level for closely related helper/doctor paths: a stopped installed service can still flow into `systemctl --user restart`.

## Solution Assessment

Unclear. A broad new flag is probably not the first fix; the safer direction is to distinguish diagnostic/helper recovery from explicit lifecycle commands and teach Linux systemd handling to respect disabled or masked units.
