## Summary

Keep open. Current main still has helper and doctor paths that can start or restart an installed stopped Linux user-systemd gateway, while the exact `openclaw status` resurrection path was not confirmed and needs a maintainer lifecycle-policy decision plus live proof.

## What This Changes

The issue asks OpenClaw helper and diagnostic commands to stop auto-starting a Linux user-systemd gateway after an operator has stopped and disabled it.

## Best Possible Solution

Define the Linux gateway service lifecycle contract so read-only diagnostics never mutate service state, helper recovery respects disabled or masked operator intent, and explicit lifecycle commands remain the only default start path with docs and regression coverage.

## Reproduction Assessment

No high-confidence exact reproduction for the `openclaw status` sequence was established. Source inspection does show current helper and doctor paths where an installed stopped user-systemd service can flow into `systemctl --user restart`, which systemd starts when inactive.

## Solution Assessment

Unclear because no patch is proposed and the right boundary is a lifecycle policy choice. The safer direction is to separate read-only diagnostics from explicit service lifecycle commands and then cover disabled or masked Linux systemd units in tests/docs.
