## Summary

Keep open: current main has made plain `openclaw status` read-only, but helper/repair paths still deliberately start or restart an installed stopped gateway service, and the linked dueling-systemd PR is still open with unresolved service-ownership risk.

## What This Changes

The issue asks OpenClaw helper commands to respect an operator-stopped and disabled user-level `openclaw-gateway` systemd unit instead of silently starting it again.

## Best Possible Solution

Settle the Linux gateway lifecycle policy so read-only helpers never start services, repair helpers require explicit operator consent or an explicit start mode, and disabled user units are not treated as ordinary recoverable stopped services.

## Reproduction Assessment

Yes, source-level only: current main still has readiness and doctor paths that can start or restart an installed stopped gateway service, but I did not live-run the reported Linux systemd sequence and plain `openclaw status` now appears read-only.

## Solution Assessment

No current implementation is the best fix yet. The likely fix belongs in the gateway service readiness/doctor lifecycle boundary, but maintainers need to choose the default operator-intent rule and any explicit start flag before code changes are safe.
