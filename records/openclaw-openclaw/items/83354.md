## Label Justifications

- `P2`: This is a normal-priority Linux gateway lifecycle bug with a limited but real operational blast radius during installs, cutovers, and handoffs.
- `impact:crash-loop`: Unexpectedly resurrecting a stopped gateway can make the intended gateway unable to bind port 18789 and create process-level availability failures.

## Summary

Keep open. Current main does not prove the exact `openclaw status` sequence still auto-starts a disabled user unit, but multiple helper/recovery paths still flow from stopped installed systemd services into `systemctl --user restart`, so the Linux gateway lifecycle policy still needs maintainer handling.

## What This Changes

The issue asks helper commands to respect explicitly stopped or disabled user-level systemd gateway units instead of restarting them implicitly.

## Best Possible Solution

Define the Linux gateway service lifecycle contract so diagnostic commands stay read-only, helper recovery does not override disabled or masked operator intent, and only explicit start/install/restart flows can reactivate a disabled unit with docs and regression coverage.

## Reproduction Assessment

No high-confidence exact reproduction for the issue's `openclaw status` sequence: current source makes that path look diagnostic and no live systemd run was performed. A closely related current-main helper/doctor/configure path is source-reproducible because stopped installed services can flow into `systemctl --user restart`.

## Solution Assessment

Unclear. The suggested explicit flag may be one option, but the maintainable fix is first to separate diagnostic/helper recovery from explicit lifecycle commands and define how disabled and masked units should be treated.
