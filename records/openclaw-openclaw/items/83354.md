## Summary

Keep open: current main and v2026.6.8 still contain helper/doctor paths that can start or restart an installed stopped gateway service, while the exact `openclaw status` path now looks read-only and the open dueling-systemd PR does not settle this disabled-user-unit operator-intent contract.

## What This Changes

The issue requests that helper commands stop silently restarting a user-level `openclaw-gateway` systemd unit after an operator has stopped and disabled it.

## Best Possible Solution

Define the Linux gateway lifecycle contract so diagnostic commands stay non-mutating, helper recovery respects disabled or masked user-systemd units, and only explicit lifecycle actions start those units with docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. The adjacent helper/doctor path is source-reproducible because an installed stopped service can flow through readiness or doctor recovery into `service.restart`, while the exact `openclaw status` path appears read-only on current main.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. The safer solution is to distinguish diagnostics/helper recovery from explicit service lifecycle commands and respect disabled or masked units, rather than closing this as covered by adjacent dueling-service work.
