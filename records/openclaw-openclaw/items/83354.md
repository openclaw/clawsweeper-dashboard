## Summary

Keep open. Current main still has lifecycle helper paths that can start or reinstall a stopped managed gateway, while the exact `openclaw status`/`configure` resurrection path is not source-confirmed, so this needs maintainer lifecycle-policy review and live Linux systemd proof rather than cleanup closure.

## What This Changes

The issue asks OpenClaw helper and diagnostic commands to respect an operator-stopped and disabled Linux user-level `openclaw-gateway` systemd unit instead of starting it again implicitly.

## Best Possible Solution

Define the Linux gateway lifecycle contract so status/configure stay non-mutating, doctor/dashboard recovery is explicit, and disabled or masked user-systemd units are respected with docs and regression coverage.

## Reproduction Assessment

No for the exact `openclaw status`/`configure` sequence: current source makes those paths look read-only. Yes at source level for adjacent helper/doctor recovery, because current code can call service start/restart or install paths for stopped managed gateways without distinguishing disabled operator intent.

## Solution Assessment

Unclear. The maintainable direction is to separate diagnostic commands from lifecycle recovery and make recovery explicit while preserving intentional start/install/restart commands, but maintainers need to choose how disabled versus masked systemd units should behave.
