## Label Justifications

- `P2`: The issue is a normal-priority Linux gateway lifecycle bug with real operational impact but bounded platform and service-manager scope.
- `impact:crash-loop`: A resurrected user-level gateway can reclaim the port and participate in repeated restart or service contention loops.

## Summary

Keep open: current main and v2026.6.8 still contain helper/doctor/start paths that can restart an installed stopped gateway service, while the related dueling-systemd PR is still open and does not fully settle the operator-intent policy for disabled user units.

## What This Changes

The issue asks OpenClaw helper commands to respect explicitly stopped/disabled user-level systemd gateway units instead of restarting them silently.

## Best Possible Solution

Define the Linux gateway lifecycle contract so read-only helpers never start services, repair helpers require explicit operator consent or an explicit start mode, and disabled user units are not treated as ordinary recoverable stopped services.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection does show current-main recovery paths that restart installed stopped services, and current tests encode that behavior.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. The likely fix belongs in the gateway readiness, doctor, and service/systemd boundary rather than a one-off helper patch.
