## Summary

Keep open. Current main and the latest release still have helper/doctor paths that can start an installed stopped gateway service through the systemd restart path, while the exact `openclaw status` path appears read-only; this needs a maintainer lifecycle-policy decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw helper commands to stop silently restarting a Linux user-systemd `openclaw-gateway` unit after an operator stopped and disabled it.

## Best Possible Solution

Define the Linux gateway lifecycle contract so diagnostic commands stay non-mutating, helper recovery and doctor respect disabled or masked user-systemd units, and only explicit lifecycle commands start those units with docs and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. The adjacent helper/doctor path is source-reproducible because an installed stopped service can flow through `runDaemonStart`, `startGatewayService`, and `systemctl --user restart`; the exact `openclaw status` path appears read-only on current main.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. The safer solution is to distinguish diagnostic/helper recovery from explicit service lifecycle commands and respect disabled or masked units, rather than closing this as covered by adjacent dueling-service work.
