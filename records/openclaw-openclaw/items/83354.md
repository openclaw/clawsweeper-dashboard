- `P2`: This is a normal-priority Linux gateway lifecycle bug with real operational impact but bounded platform and service-manager scope.
- `impact:crash-loop`: A resurrected user-level gateway can reclaim the port and participate in repeated service contention or restart loops during cutovers.
- `maturity:stable`: The issue affects M4 stable surfaces: CLI Gateway Service Management and Linux Gateway host Gateway Runtime and Service Control.

## Summary

Keep open: current main and v2026.6.10 still contain readiness, doctor, and configure paths that can start or restart an installed stopped user-systemd gateway, while plain status now appears read-only and the open dueling-systemd PR only partially overlaps this operator-intent issue.

## What This Changes

The issue asks helper commands to respect an explicitly stopped or disabled user-level openclaw-gateway systemd unit instead of silently starting it again.

## Best Possible Solution

Define a Linux gateway lifecycle contract where read-only helpers never start services, repair/setup helpers require explicit operator intent, and disabled user units are not treated as ordinary recoverable stopped services.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection and current tests show installed stopped services are still started or restarted by readiness and doctor repair paths.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. The likely fix belongs across readiness, doctor, configure, and systemd service ownership semantics rather than a one-line status patch.
