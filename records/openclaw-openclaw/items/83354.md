## Summary

Keep open. Current main does not support closing this as fixed: `openclaw status` itself looks read-only, but the readiness and doctor helper paths can still prompt or `--yes` into starting an installed stopped gateway service, which maps to a Linux user-systemd `restart`.

## What This Changes

The issue asks helper and diagnostic commands to respect explicitly stopped/disabled user-level Linux systemd gateway units instead of starting them again.

## Best Possible Solution

Define the Linux service lifecycle contract so diagnostic status stays non-mutating, helper recovery does not override disabled or masked operator intent, and only explicit lifecycle flows can start a disabled user unit with docs and regression coverage.

## Reproduction Assessment

No high-confidence exact reproduction for the `openclaw status` sequence: current source makes status look read-only. A source-level helper/doctor path remains because stopped installed services can flow into `systemctl --user restart`.

## Solution Assessment

Unclear. A broad new flag is probably not the first fix; maintainers should first decide how helper recovery distinguishes disabled operator intent from explicit lifecycle commands.
