## Summary

Keep open: current main still runs Telegram/channel polling in the gateway process, and this request is a broad gateway architecture/concurrency feature that needs maintainer design review rather than cleanup closure.

## What This Changes

Requests worker-thread or multi-process channel isolation, or minimum polling backpressure, so multi-agent Telegram/channel polling cannot monopolize the gateway event loop.

## Best Possible Solution

Triage this with the open Telegram starvation reports, define the supported channel concurrency/backpressure model, and only then decide whether worker/process isolation is needed beyond a Telegram-owned scheduler or global polling budget.

## Reproduction Assessment

No high-confidence live reproduction was established for the 16-bot starvation metrics. Source inspection does confirm the current in-process Telegram polling architecture and the absence of worker/process isolation on current main.

## Solution Assessment

Unclear: worker-thread or multi-process isolation may be a valid end state, but it is not proven to be the narrowest maintainable first fix. A safer path is to settle Telegram/channel backpressure and global polling policy, then evaluate whether gateway isolation is still required.
