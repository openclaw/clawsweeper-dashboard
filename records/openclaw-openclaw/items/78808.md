## Summary

Keep open: current main still documents and implements Telegram long polling inside the Gateway process, and the requested worker/process isolation is broad Gateway architecture work rather than conservative cleanup.

## What This Changes

Request a multi-threaded or multi-process Gateway architecture so channel polling and agent work do not compete on one Node.js event loop.

## Best Possible Solution

Define the supported Gateway concurrency model first, then split focused fixes for Telegram/channel backpressure and broader agent/Gateway scheduling with measured live proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection confirms the in-process Gateway and Telegram polling architecture, but the reported 16-bot ELU and delay metrics were not rerun on current main.

## Solution Assessment

Unclear: worker-thread offload may be valid, but it is not obviously the narrowest first fix. Maintainers should decide the concurrency model and likely add explicit channel backpressure before choosing worker/process isolation.
