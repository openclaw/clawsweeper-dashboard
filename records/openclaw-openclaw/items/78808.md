## Summary

Keep open: current main still runs Gateway queues and Telegram long polling in-process, with no worker/process isolation or cross-account getUpdates budget. The startup fanout cap from the related closed request helps only startup probes; the remaining request is a broad Gateway/channel concurrency design that needs maintainer architecture review.

## What This Changes

The issue requests a multi-threaded or multi-process Gateway/channel architecture, or at least cross-channel polling limits, to reduce single-event-loop contention under multi-agent and multi-channel load.

## Best Possible Solution

Keep this as the architecture tracker, split it from the already-fixed startup fanout issue, and design measured fixes for Telegram active-poll backpressure plus broader Gateway scheduling or worker/process isolation where evidence supports it.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm the current in-process Gateway/Telegram polling path and the absence of a global active getUpdates budget or worker/process isolation.

## Solution Assessment

Unclear: worker-thread offload may be a valid end state, but it is not proven to be the narrowest first fix. A safer path is to define the supported concurrency/backpressure model, separate Telegram-specific active polling limits from general agent/Gateway scheduling, and validate each change under load.
