## Summary

Keep open: current main still runs Telegram polling and agent/Gateway work inside the Gateway process, and the requested worker/process isolation is not implemented or safely redundant with the narrower open symptom reports. This needs maintainer architecture and product review rather than cleanup closure.

## What This Changes

Request to move Gateway channel polling and multi-agent concurrency away from a single Node event loop, especially for many Telegram bot accounts.

## Best Possible Solution

Define the supported Gateway/channel concurrency model, split Telegram polling backpressure from broader agent-prep/WebSocket latency work, and land measured fixes with live high-load proof.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported 16-bot timing numbers. Source inspection does confirm the current in-process Gateway, Telegram polling, and agent queue architecture that the report is about.

## Solution Assessment

Unclear: worker-thread offload may be a valid direction, but it is a broad architectural change rather than an obvious narrow fix. The safer path is a maintainer design that first defines Telegram/channel backpressure and then decides whether worker or process isolation is still required.
