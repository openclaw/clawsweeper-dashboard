## Summary

Keep open: current main still runs Telegram channel polling inside the gateway process, and the requested worker/process isolation is a broad Gateway architecture change that needs maintainer product and ownership review rather than cleanup closure.

## What This Changes

Request a multi-threaded or multi-process Gateway architecture, with channel polling offloaded or globally throttled so many Telegram bots do not starve multi-agent work.

## Best Possible Solution

Triage this alongside https://github.com/openclaw/openclaw/issues/78695, first defining the supported concurrency model and likely adding Telegram/channel backpressure before considering a broader worker/process split.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection confirms the current in-process Telegram polling architecture, and the linked report provides plausible 16-bot reproduction data, but the specific starvation metrics were not rerun.

## Solution Assessment

Unclear: worker-thread offload may be a valid direction, but it is not obviously the narrowest first fix. A safer maintainable path is to make the Telegram/channel concurrency and backpressure policy explicit, then decide whether Gateway worker/process isolation is still needed.
