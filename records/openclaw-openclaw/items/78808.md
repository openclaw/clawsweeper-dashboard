## Summary

Keep open: current main still runs Telegram polling and channel startup inside the gateway/plugin task path, with no worker-thread or process isolation for this surface. The linked symptom reports remain open and the requested fix is a broad Gateway/channel architecture decision, not conservative cleanup.

## What This Changes

Request worker/process isolation or channel polling caps so multi-agent gateway/channel load does not starve the single Node event loop.

## Best Possible Solution

Keep this as a Gateway/channel architecture tracker, linked to the Telegram-specific and broader event-loop reports, then split maintainer-approved work into measured concurrency/backpressure and isolation PRs.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported 16-bot/proxy starvation metrics. Source inspection does confirm the current in-process Gateway/Telegram polling architecture that the feature request targets.

## Solution Assessment

Unclear: worker-thread offload may be valid, but it is not obviously the narrowest maintainable first step. Maintainers should define channel concurrency/backpressure and Gateway isolation boundaries before implementation.
