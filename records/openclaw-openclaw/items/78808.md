## Summary

Keep open: current main still uses in-process Gateway queues and in-process Telegram polling rather than worker/process isolation. The request is a broad Gateway/channel architecture change with active related symptom trackers, so it needs maintainer product and ownership review rather than cleanup closure.

## What This Changes

The issue requests moving channel polling or broader Gateway work off the single Node.js event loop via worker threads or processes, with bounded channel polling as a fallback.

## Best Possible Solution

Keep this as an architecture tracker, resolve the narrower Telegram startup/polling and agent-prep latency bugs with measured proof, then decide whether worker/process isolation is still needed.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported 16-bot/proxy starvation metrics. Source inspection does confirm the requested architecture gap: Gateway queues and Telegram polling still run in-process on current main.

## Solution Assessment

Unclear: worker-thread or process isolation may be valid, but it is not proven to be the narrowest maintainable fix. A safer path is to define channel backpressure and Gateway scheduling policy first, then evaluate isolation with measured latency proof.
