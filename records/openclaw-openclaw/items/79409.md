## Summary

Keep open: current main still has a source-reproducible Feishu command-authorization path where admitted slash commands can reach `CommandAuthorized=false`, and open fix PRs still reference this issue with closing syntax rather than having shipped.

## What This Changes

The issue asks Feishu text slash commands such as `/compact`, `/new`, and `/stop` to be intercepted by the Gateway instead of being delivered to the agent as plain text.

## Best Possible Solution

Land a narrow command-authorization fix that lets admitted Feishu senders run text slash commands when no explicit command allowlist is configured, while preserving explicit allowlists and owner-only command enforcement.

## Reproduction Assessment

Yes, source-reproducible: current source and Feishu tests show admitted group slash commands without configured command allowlists can carry `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands. I did not run a live Feishu bot workspace.

## Solution Assessment

Unclear for the open PRs: setting `CommandSource` alone does not address the false `CommandAuthorized` path. The safer fix should adjust the Feishu ingress authorization call site or shared command-gating contract for unconfigured command allowlists, with regression coverage for explicit allowlists.
