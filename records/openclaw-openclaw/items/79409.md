## Label Justifications

- `P1`: Feishu users can lose the in-channel recovery path for wedged sessions, which is a broken channel workflow with real user impact.
- `impact:message-loss`: The reported slash-command messages produce no useful command acknowledgement or command effect in Feishu.
- `impact:session-state`: `/compact`, `/new`, and `/stop` are session recovery or mutation commands, so the failure directly affects session state.

## Summary

Keep open: current main still leaves the Feishu text-command path unresolved for the reported `/compact`, `/new`, and `/stop` class, and the related open PRs are not merged canonical fixes for this issue.

## What This Changes

Feishu should execute `/compact`, `/new`, and `/stop` text slash commands instead of forwarding them to the agent as ordinary text.

## Best Possible Solution

Implement a current-main structured text-command turn and authorization path for Feishu recovery commands, with focused Feishu `/compact`, `/new`, and `/stop` coverage and a visible denial path for unauthorized senders.

## Reproduction Assessment

Yes, by source inspection: current Feishu inbound context carries `CommandBody` but not explicit text-slash command metadata, and the shared command gate still denies unconfigured access-group command authorizers. I did not run a live Feishu workspace reproduction.

## Solution Assessment

Unclear. The issue direction is valid, but the stale `CommandSource`-only PR is not the best current-main shape; the fix should use the structured `CommandTurn` path and settle the session-recovery authorization boundary.
