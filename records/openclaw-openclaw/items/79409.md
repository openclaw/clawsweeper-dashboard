## Summary

Keep open. Current main still contains the source path described by the reporter: Feishu can dispatch a slash-command body with `CommandAuthorized: false`, and the shared reply runner silently returns no reply for whole-message control commands in that state. An open PR already references this issue, and the broader shared command-gating fix is also tracked by an open related PR, so this should not be closed before the fix path is resolved.

## What This Changes

Feishu text slash commands such as `/compact`, `/new`, and `/stop` should be handled as OpenClaw commands instead of reaching the agent as plain text or producing no reply.

## Best Possible Solution

Review and land a narrow command-authorization fix that makes already-admitted Feishu senders command-authorized when no command allowlist is configured, while preserving explicit allowlist enforcement; close this issue only after that fix merges.

## Reproduction Assessment

Yes, at source level. Current main still routes Feishu slash-command text into the shared reply path with `CommandAuthorized: false` when no command allowlist is configured, and the reply guard then returns no response for whole-message control commands.

## Solution Assessment

Unclear whether the currently linked Feishu-only PR is sufficient. The reporter's later root-cause analysis and current source point to command authorization, so the safer fix should address the shared gate or Feishu's command authorization handoff with regression coverage.
