## What This Changes

The PR changes qa-lab Mantis Discord smoke API body reads from unbounded `response.text()` to the shared byte-limited response reader, with regression tests for oversized, under-cap, and malformed Discord responses.

## Best Possible Solution

Land the bounded-read hardening with the existing SDK helper once required checks pass and maintainers accept the 16 MiB fail-closed oversized-response behavior.

## Reproduction Assessment

Yes, from source inspection: current main reads every Mantis Discord smoke API response with `await response.text()` after the guarded fetch, so an oversized body is buffered before JSON parsing. I did not run live Discord credentials in this read-only review, and a hostile oversized Discord body is not practical to reproduce against the real service.

## Solution Assessment

Yes: this is the narrow maintainable fix because it caps the single shared Discord smoke API read point with the existing SDK helper while preserving current parsing, artifact writing, redaction, and release cleanup. A custom reader or a broader Mantis refactor would add more surface without improving this specific hardening.
