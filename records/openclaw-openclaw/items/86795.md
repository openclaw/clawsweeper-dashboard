## Summary

Keep open. The patch is focused and improves the no-budget save path, but it is not merge-ready because the disk-budget path still eagerly collects preserve keys when a configured budget is under limit, and the external proof is focused tests rather than real workload evidence.

## What This Changes

The PR lazily collects session maintenance preserve keys in `saveSessionStoreUnlocked` and adds pruning integration tests for skipped and executed preserve providers.

## Best Possible Solution

Defer preserve-provider collection until prune/cap or disk-budget entry eviction actually needs it, with regression coverage for under-budget `maxDiskBytes` stores and preserved stale/cap behavior.

## Reproduction Assessment

No live/profiler reproduction was run in this review. Source inspection is high-confidence: current main collects preserve keys before deciding whether maintenance needs them, and the PR still does so for configured under-budget disk budgets.

## Solution Assessment

No. The lazy wrapper is the right direction for prune/cap paths, but the disk-budget call needs to defer collection until after the budget preflight and preferably until entry eviction is necessary.

Full review comments:

- **[P2] Defer preserve scans until disk-budget eviction needs them:** `src/config/sessions/store.ts:438`
  - body: This still calls `getPreserveSessionKeys()` whenever `maxDiskBytes` is configured, before `enforceSessionDiskBudget` checks whether the store is over budget.
