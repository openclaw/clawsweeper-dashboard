## Summary

Keep open: the PR’s one-flag change should not merge because it collapses the documented `showOk` and `showAlerts` contracts, and the author has agreed the current branch is wrong as-is. Current `main` still needs an output-aware webchat projection if maintainers want `showAlerts=false` honored without regressing default OK suppression.

## What This Changes

The PR changes webchat heartbeat chat suppression from `showOk` to `showAlerts` and updates nearby gateway tests to match that behavior.

## Best Possible Solution

Implement output-aware heartbeat chat projection: OK-only responses should follow `showOk`, non-OK alerts should follow `showAlerts`, and focused tests should cover default, alerts-off, OKs-on/alerts-off, and alerts-on/OKs-off cases.

## Reproduction Assessment

Yes, at source level. With the PR, default `showOk=false` and `showAlerts=true` makes the gate return false and expose ACK-like output, while `showOk=true` and `showAlerts=false` suppresses OK-only acknowledgments that were explicitly enabled.

## Solution Assessment

No. A single `showAlerts` gate is not the best fix because it collapses two documented visibility controls; the safer solution needs to classify the heartbeat output and apply the matching flag.
