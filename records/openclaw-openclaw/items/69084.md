## Summary

Keep open: the branch is not safe to merge because it collapses the documented `showOk`/`showAlerts` split, while current `main` still needs an output-aware heartbeat chat fix rather than this one-flag swap.

## What This Changes

The PR swaps webchat heartbeat chat suppression from `showOk` to `showAlerts` and rewrites adjacent gateway tests around that behavior.

## Best Possible Solution

Implement output-aware heartbeat chat projection: OK-only replies follow `showOk`, alert content follows `showAlerts`, and gateway tests cover defaults, alerts-off, OK-enabled/alerts-off, and the inverse mixed config.

## Reproduction Assessment

Yes, at source level. Current `main` shows the one-flag gate that misses `showAlerts=false` alert suppression, and the PR diff shows the inverse one-flag gate that would leak default `HEARTBEAT_OK` output and suppress OK-only configs.

## Solution Assessment

No. A single global `showAlerts` gate is not the narrow maintainable fix because the documented contract requires classifying heartbeat output before applying `showOk` or `showAlerts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep heartbeat ACK and alert visibility separate:** `src/gateway/server-chat.ts:61`
  - body: Changing this gate to `!visibility.showAlerts` makes the default `showOk=false` / `showAlerts=true` path expose `HEARTBEAT_OK`, and makes `showOk=true` / `showAlerts=false` suppress OK-only acknowledgments that the config explicitly enabled. The webchat projection needs to classify heartbeat output and apply `showOk` to OK-only replies and `showAlerts` to alert content.
