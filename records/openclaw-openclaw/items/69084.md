## Summary

Keep open. The PR targets a real heartbeat visibility bug, but the patch is incorrect because it collapses the documented `showOk` and `showAlerts` contracts; the author later agreed the branch should not merge as-is.

## What This Changes

The PR changes gateway webchat heartbeat chat filtering from `showOk` to `showAlerts` and updates two gateway agent-event tests to expect that behavior.

## Best Possible Solution

Rework the gateway heartbeat chat projection so ACK-only output follows `showOk`, non-OK alert content follows `showAlerts`, and tests cover default, alerts-off, OK-only, and mixed configurations.

## Reproduction Assessment

Yes, at source level. With default visibility `showOk=false` and `showAlerts=true`, the PR gate would return false and allow `HEARTBEAT_OK` through; with `showOk=true` and `showAlerts=false`, it would suppress OK acknowledgments even though OKs are enabled.

## Solution Assessment

No. A single `showOk` to `showAlerts` swap is not the best fix because it collapses separate documented flags; the safer fix should classify heartbeat output before applying the relevant visibility flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep heartbeat ACK and alert visibility separate:** `src/gateway/server-chat.ts:61`
  - body: Changing this gate to `!visibility.showAlerts` makes the default `showOk=false` / `showAlerts=true` path expose `HEARTBEAT_OK`, and makes `showOk=true` / `showAlerts=false` suppress OK-only acknowledgments. The docs/defaults treat `showOk` as OK visibility and `showAlerts` as non-OK alert visibility, so the webchat projection needs to classify heartbeat output and apply both flags separately.
  - confidence: 0.91
