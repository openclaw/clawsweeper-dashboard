## Summary

Keep open: the symptom is narrow and real, but this PR's current patch regresses the documented heartbeat visibility contract. Current main and the docs treat `showOk` and `showAlerts` as separate controls, and the author's latest comment agrees the single-flag swap should not merge as-is.

## What This Changes

The PR changes Gateway webchat heartbeat suppression from `showOk` to `showAlerts` and updates adjacent gateway agent-event tests around heartbeat chat output.

## Best Possible Solution

Implement webchat heartbeat projection that classifies heartbeat output first: ACK-only output remains controlled by `showOk`, non-OK alert text remains controlled by `showAlerts`, and focused tests cover default, alerts-off, OK-only, and mixed configurations.

## Reproduction Assessment

Yes. The existing gateway test harness models `showOk=false` and `showAlerts=true`; applying this PR's gate would make suppression false and let ACK-like heartbeat output pass through. The mixed `showOk=true, showAlerts=false` case follows from the same boolean gate and would suppress OKs users explicitly enabled.

## Solution Assessment

No. The single `showOk` to `showAlerts` swap is not the best fix because it collapses two documented contracts into one decision; the safer solution is to classify ACK versus alert output and apply the corresponding visibility flag.

## Review Findings

Please classify ACK-only versus alert heartbeat output and apply `showOk` and `showAlerts` separately.
