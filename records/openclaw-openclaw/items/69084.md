## Summary

Keep open and do not merge as-is: the PR collapses the documented `showOk` and `showAlerts` contracts into one gate, which would regress default heartbeat ACK suppression and OK-only configurations; the author later agreed the patch should not merge in its current form.

## What This Changes

The PR changes gateway webchat heartbeat chat filtering from `showOk` to `showAlerts` and rewrites adjacent gateway tests to expect showAlerts-based suppression.

## Best Possible Solution

Rework gateway heartbeat chat projection so OK-only output follows `showOk`, non-OK alert content follows `showAlerts`, and focused tests cover default, alerts-off, OK-only, and mixed configurations.

## Reproduction Assessment

Yes, at source level. With default `showOk=false` and `showAlerts=true`, the PR gate would allow `HEARTBEAT_OK` through; with `showOk=true` and `showAlerts=false`, it would suppress OK-only acknowledgments that the config explicitly enabled.

## Solution Assessment

No. A single `showOk` to `showAlerts` swap is not the best fix because it collapses two documented flags; the safer fix should classify heartbeat output before applying the relevant visibility flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep heartbeat ACK and alert visibility separate:** `src/gateway/server-chat.ts:61`
  - body: Changing this gate to `!visibility.showAlerts` makes the default `showOk=false` / `showAlerts=true` path expose `HEARTBEAT_OK`, and makes `showOk=true` / `showAlerts=false` suppress OK-only acknowledgments. The docs/defaults treat `showOk` as OK visibility and `showAlerts` as non-OK alert visibility, so the webchat projection needs to classify heartbeat output and apply both flags separately.
