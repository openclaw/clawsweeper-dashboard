## Summary

Keep open: this is a plausible gateway heartbeat-visibility bug, but the PR's single `showOk` to `showAlerts` gate swap is incorrect and the author has agreed it should not merge as-is.

## What This Changes

The PR changes webchat heartbeat chat suppression from `showOk` to `showAlerts` and updates adjacent gateway tests to expect showAlerts-based suppression.

## Best Possible Solution

Replace the one-flag gate with output-aware heartbeat chat projection: OK-only responses follow `showOk`, non-OK alerts follow `showAlerts`, and tests cover default, alerts-off, OK-only, and mixed configurations.

## Reproduction Assessment

Yes, at source level. With the PR, default `showOk=false` and `showAlerts=true` makes the heartbeat gate return false and lets `HEARTBEAT_OK` bypass stripping; with `showOk=true` and `showAlerts=false`, OK-only acknowledgments are suppressed even though `showOk` enabled them.

## Solution Assessment

No. A single `showAlerts` gate is not the best fix because it collapses two documented visibility contracts; the safer repair should classify the heartbeat output and apply the matching flag.

## Review Findings
