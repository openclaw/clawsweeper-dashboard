## Summary

Close as implemented on current main: the shipped release named in the report predates the session-lock ownership fix, while current main contains the owned transcript-write publication/reacquire path plus Telegram/direct event guardrails that address the reported takeover shape.

## What This Changes

The issue reports a release regression where Telegram direct and terminal turns abort with EmbeddedAttemptSessionTakeoverError after orphaned-user repair, leaving users without replies.

## Best Possible Solution

Keep the current main ownership-aware session-lock implementation, ship it in the next release, and use narrower open issues only for distinct remaining cron or provider-fallback trigger paths.

## Reproduction Assessment

Yes for the shipped regression: the issue includes deterministic v2026.5.20 logs, and v2026.5.20 lacks the owned-write publication path that current main adds. I did not run a live Telegram/macOS repro against current main because the reviewed source and merged regressions cover the fixed path.

## Solution Assessment

Yes. The current main approach keeps the takeover fence for external edits while allowing only explicit OpenClaw-owned same-process transcript writes, which is the narrow maintainable fix for this bug class.
