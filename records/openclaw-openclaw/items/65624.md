## Label Justifications

- `P1`: This is an auth-sensitive Mattermost slash-command security issue with a remaining cleartext callback token exposure path.
- `impact:security`: The issue concerns command-token transport and authorization boundaries for Mattermost slash callbacks.

## Summary

Keep open. Current main and the latest release still derive and register a plain-HTTP Mattermost native slash callback when `channels.mattermost.commands.callbackUrl` is omitted; the cross-trigger token replay part is already hardened, and the remaining transport issue is paired with open PR https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

The issue asks OpenClaw to harden Mattermost native slash-command callbacks so omitted/default callback URLs do not expose reusable command tokens.

## Best Possible Solution

Review, rework, or replace the linked Mattermost plugin fix so non-HTTPS slash callback transport is rejected according to the agreed policy while preserving current command-specific token validation, docs, tests, and real Mattermost/gateway proof.

## Reproduction Assessment

Yes, at source level. Enable Mattermost native slash commands without channels.mattermost.commands.callbackUrl; current main derives an http:// callback URL and proceeds into registration after only the warning path.

## Solution Assessment

Yes for the remaining direction, but not by closing this issue now. The narrow maintainable solution is a Mattermost plugin fix that rejects insecure callback transport before registration while keeping the current per-command/current-command token gate.
