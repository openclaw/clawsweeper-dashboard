## Label Justifications

- `P1`: The remaining live issue is an auth-sensitive Mattermost slash-command path that can expose reusable command tokens over cleartext transport.
- `impact:security`: The report concerns callback transport exposing command tokens and weakening the command execution authorization boundary.

## Summary

Keep open. Current main and v2026.5.18 still derive and register a plain-HTTP Mattermost native slash callback when `channels.mattermost.commands.callbackUrl` is omitted; the cross-trigger token replay portion is already hardened, and the remaining transport issue is paired with the open closing PR at https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

Mattermost native slash-command callbacks should fail closed for non-HTTPS derived callback URLs while preserving command-specific token validation.

## Best Possible Solution

Review, land, or replace the linked Mattermost plugin fix so insecure derived slash callbacks fail closed, docs/tests reflect the upgrade behavior, and current command-specific token validation remains intact.

## Reproduction Assessment

Yes, at source level. Enable Mattermost native slash commands without `channels.mattermost.commands.callbackUrl`; current main derives an `http://` callback and proceeds into registration after only the warning path.

## Solution Assessment

No, current main is not the complete solution because only the token-replay half is fixed. The narrow maintainable path is to rework or land the linked Mattermost plugin fix for HTTPS fail-closed callback transport with clear compatibility docs.
