## Summary

Keep open: current main still derives and registers `http://` Mattermost slash-command callbacks when `commands.callbackUrl` is omitted. The cross-command token replay portion is already improved on main, but the remaining cleartext transport issue is still valid and is paired with open fix PR https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

Mattermost native slash commands are reported to derive insecure HTTP callback URLs when no explicit HTTPS callback URL is configured, exposing reusable command tokens.

## Best Possible Solution

Land or replace the linked Mattermost plugin fix so native slash-command registration rejects non-HTTPS callback URLs before registration while retaining the current per-command token validation.

## Reproduction Assessment

Yes at source level: enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and current `resolveCallbackUrl()` derives `http://...` while the monitor passes it into registration. I did not run a live Mattermost roundtrip in this read-only review.

## Solution Assessment

Yes for the remaining issue shape: fail closed before registration for non-HTTPS callbacks and keep the existing per-command token check. The open linked PR is the right review target, but its fail-closed compatibility impact still needs maintainer/security review.
