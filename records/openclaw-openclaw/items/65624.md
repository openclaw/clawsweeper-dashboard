## Summary

Keep open: current main and the latest stable release still derive and register an `http://` Mattermost slash-command callback when `commands.callbackUrl` is omitted. The cross-command token-replay portion appears addressed on main, but the remaining cleartext transport issue is still valid and is already paired with open fix PR https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

The issue requests hardening Mattermost native slash commands so omitted callback URLs do not expose reusable command tokens over cleartext HTTP and captured tokens cannot be replayed across commands.

## Best Possible Solution

Land a narrow Mattermost plugin security fix that rejects non-HTTPS derived slash callbacks before registration, preserves the current per-command token validation, and documents the HTTPS callback upgrade path.

## Reproduction Assessment

Yes at source level: enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and current `resolveCallbackUrl()` derives `http://...` while `registerMattermostMonitorSlashCommands()` continues registration. I did not run a live Mattermost roundtrip in this read-only review.

## Solution Assessment

Yes for the remaining scope: fail closed before Mattermost registration when the derived callback is not HTTPS, while keeping the current per-command token check. The open linked PR is the right review target, but its compatibility/security tradeoff still needs maintainer handling.
