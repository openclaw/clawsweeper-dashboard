## Summary

Keep open: current main and the latest release still derive and register `http://...` Mattermost slash-command callbacks when `commands.callbackUrl` is omitted, while the older cross-command token replay path is already hardened. The remaining transport fix is paired with an open closing PR and needs maintainer/security review for the fail-closed upgrade impact.

## What This Changes

The issue asks OpenClaw to stop registering Mattermost native slash-command callbacks over non-HTTPS derived URLs and to preserve per-command callback token binding.

## Best Possible Solution

Review and land or replace the linked Mattermost security PR with a narrow HTTPS fail-closed transport fix that preserves current per-command token validation and documents the upgrade path for existing HTTP callback deployments.

## Reproduction Assessment

Yes, source-reproducible. Current main still derives `http://...` when `commands.callbackUrl` is omitted and proceeds to registration; I did not run a live Mattermost network capture in this read-only review.

## Solution Assessment

Unclear until maintainers accept the compatibility tradeoff. A narrow Mattermost-plugin HTTPS fail-closed gate appears to be the right fix shape, but it should land through the linked PR or a replacement after security and upgrade review.
