## Summary

Keep open: current main and the latest stable release still derive and register an HTTP Mattermost slash-command callback when `commands.callbackUrl` is omitted, while the closing fix remains open at https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

The issue reports insecure Mattermost native slash-command callback URL derivation when no explicit HTTPS callback URL is configured.

## Best Possible Solution

Land or replace the linked Mattermost plugin security PR with a narrow fix that rejects unsafe non-HTTPS slash-command callbacks before registration while preserving current per-command token validation.

## Reproduction Assessment

Yes at source level: enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and current `resolveCallbackUrl()` derives `http://...` while the monitor proceeds to registration. I did not run a live Mattermost roundtrip in this read-only review.

## Solution Assessment

Yes for the remaining issue shape: the maintainable fix belongs in the Mattermost monitor registration path, failing closed before registering unsafe callback URLs while keeping the existing per-command token validation. The open linked PR is the current implementation path, so this issue should stay open until that PR lands or is replaced.
