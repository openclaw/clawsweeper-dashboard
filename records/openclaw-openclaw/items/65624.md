## Summary

Keep open. Current main has addressed the cross-trigger token replay half, but Mattermost native slash commands still derive an `http://` callback URL when `channels.mattermost.commands.callbackUrl` is omitted and then proceed with registration; https://github.com/openclaw/openclaw/pull/65655 still tracks the remaining fail-closed work.

## What This Changes

The issue asks OpenClaw to fail closed for insecure Mattermost native slash callback URLs and prevent captured command tokens from being replayed across slash triggers.

## Best Possible Solution

Land or rework a narrow Mattermost plugin fix on current main that rejects non-HTTPS resolved slash callback URLs before command registration, keeps current per-command/current-token validation, and updates focused docs/tests.

## Reproduction Assessment

Yes. A source-level current-main path is clear: enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and `resolveCallbackUrl` returns `http://...` which the monitor passes into registration after only the suspicious-URL warning path.

## Solution Assessment

Yes for the remaining direction, but not by merging the stale branch as-is. The maintainable path is a small Mattermost-plugin-local HTTPS fail-closed guard reconciled with the current https://github.com/openclaw/openclaw/pull/72923 command-validation design.
