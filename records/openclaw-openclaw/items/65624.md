## Summary

Current main still derives `http://...` for omitted Mattermost slash-command callback URLs, while the older cross-command token replay portion is already hardened. This should stay open because the remaining fail-closed transport behavior is security- and upgrade-sensitive and is already paired with an open linked fix PR.

## What This Changes

Request to stop Mattermost native slash-command registration from defaulting to cleartext callback URLs and to prevent captured command tokens from being reused across slash triggers.

## Best Possible Solution

Review and land or replace the linked Mattermost security PR with a narrow HTTPS fail-closed transport fix that preserves the current per-command token validation and documents the upgrade behavior for existing HTTP callback deployments.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `commands.callbackUrl` still resolves to `http://...` and registration continues after only a suspicious-loopback warning; I did not run a live Mattermost network capture in this read-only review.

## Solution Assessment

Unclear until maintainers decide the compatibility tradeoff. A narrow fail-closed HTTPS transport gate in the Mattermost plugin appears to be the right fix shape, but it should land through the linked PR or a replacement after security/upgrade review.
