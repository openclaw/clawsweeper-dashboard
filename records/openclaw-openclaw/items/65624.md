## Summary

Keep open: current main and the latest stable release still derive and register a cleartext Mattermost slash-command callback when `commands.callbackUrl` is omitted. The cross-command token-replay portion is addressed on main, but the remaining cleartext transport issue is still valid and already has an open linked fix PR at https://github.com/openclaw/openclaw/pull/65655.

## What This Changes

The issue reports that Mattermost native slash commands derive a plain-HTTP callback URL when no explicit HTTPS callback URL is configured, exposing reusable command tokens on the callback transport.

## Best Possible Solution

Review and land, revise, or replace https://github.com/openclaw/openclaw/pull/65655 with a narrow Mattermost plugin security fix that rejects unsafe derived callbacks before registration while preserving current per-command token validation and documenting the HTTPS upgrade path.

## Reproduction Assessment

Yes at source level: enable Mattermost native slash commands, omit `channels.mattermost.commands.callbackUrl`, and current `resolveCallbackUrl()` derives `http://...` while `registerMattermostMonitorSlashCommands()` continues registration. I did not run a live Mattermost roundtrip in this read-only review.

## Solution Assessment

Yes for the remaining security shape: fail closed before Mattermost registration when the derived callback is not HTTPS, while keeping the current per-command token check. The linked PR is the right implementation target, but its compatibility and proof tradeoffs still need maintainer handling.
