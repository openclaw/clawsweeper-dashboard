## Summary

Keep open: current main and the latest release still derive an `http://` Mattermost slash-command callback when `commands.callbackUrl` is omitted, while the cross-trigger token replay part is already addressed on main and in `v2026.6.6`. The open linked fix PR still owns the remaining HTTPS fail-closed decision, so this issue should not be closed before that PR is reviewed, narrowed, merged, or explicitly rejected.

## What This Changes

The issue reports that Mattermost native slash commands still derive an HTTP callback URL when no explicit HTTPS callback is configured, exposing reusable command tokens in transit.

## Best Possible Solution

Keep this issue open as the security tracking item until maintainers review and either land or replace the linked HTTPS fail-closed fix, while preserving the per-command token validation already present on main.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: omit `channels.mattermost.commands.callbackUrl`, enable native Mattermost slash commands, and `resolveCallbackUrl()` produces an `http://` URL that the monitor passes into command registration. I did not run a live Mattermost server in this read-only review.

## Solution Assessment

Unclear whether the current linked PR is the best final patch as-is, because main already has the token-binding portion and the remaining change is the HTTPS fail-closed behavior. The best direction is a focused maintainer-reviewed fix for the non-HTTPS derived callback path with explicit upgrade messaging for affected Mattermost operators.
