## What This Changes

The PR changes `isServerErrorMessage` so a pure `status: internal server error` message is classified as a server error after the status phrase scrubs to an empty string.

## Best Possible Solution

Land the narrow classifier change after adding a regression test for the exact pure `status: internal server error` input and a single-line Unreleased Fixes changelog entry, while preserving the existing prefixed plain-status rejection and the mixed/code:500 retry behavior.

Yes. Static inspection plus a read-only regex check reproduces current-main behavior: the exact raw message scrubs to an empty string and the existing non-empty guard returns false before timeout failover can classify it.

## Solution Assessment

Yes, with completion work. The proposed empty-scrub branch is the narrowest code fix for the pure-status case and avoids matching un-scrubbed input, but it should be paired with focused test coverage and the required changelog entry before merge.
