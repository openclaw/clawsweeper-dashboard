## Summary

Keep open. The diagnostic goal is useful, but the patch adds a config-specific warning in a generic unauthorized SIGUSR1 path, so it can mislead operators and does not prove the config/update restart path it claims to address; the supplied proof is unit-harness output only.

## What This Changes

Adds a second gateway warning and test assertion for the unauthorized SIGUSR1 restart branch, telling operators to run `openclaw gateway restart` or use a service manager.

## Best Possible Solution

Keep the additive recovery hint, but make it accurate for any ignored unauthorized SIGUSR1 or propagate a concrete restart reason before naming config/update as the cause.

## Reproduction Assessment

Yes by source inspection, not by a live run. Current main's unauthorized SIGUSR1 branch emits only the existing warning, and the PR places the new config-specific text in that same generic branch.

## Solution Assessment

No. Adding actionable restart guidance is reasonable, but the proposed wording overclaims the cause; generic blocked-restart wording or reason-aware logging would be narrower and more maintainable.

- **[P2] Avoid claiming every ignored SIGUSR1 came from config:** `src/cli/gateway-cli/run-loop.ts:718-722`
