## Summary

Keep this PR open, but the current patch needs one wording fix before merge: the added warning is useful in spirit, but it attributes every unauthorized SIGUSR1 rejection to a config change even though the branch is shared by non-config restart signals too.

## What This Changes

The PR adds a second gateway warning and a focused run-loop test assertion when an unauthorized SIGUSR1 restart is ignored.

## Best Possible Solution

Keep the additive recovery hint, but make it generic or derive the cause from the actual restart reason before logging it.

## Reproduction Assessment

Yes, source-level: the existing run-loop test drives the unauthorized SIGUSR1 branch, and current source shows that branch is shared rather than config-specific. I did not run the test or a live gateway in this read-only review.

## Solution Assessment

No. The narrow maintainable fix is to keep the actionable restart command but avoid saying a config change caused the ignored signal unless the code has that reason available.
