## Summary

Keep open: this PR is a focused security fix for the linked skill SecretRef leak, and I found no blocking diff-level correctness issue. It still needs maintainer/security review because it deliberately changes generic `exec` inheritance of active skill env keys and intersects the open skill env usability thread.

## What This Changes

The PR adds a `stripInheritedKeys` input to host exec env sanitization, passes active skill env keys from generic `exec`, and adds sanitizer plus exec regression tests.

## Best Possible Solution

Land this PR or an equivalent narrow fix after maintainer/security signoff, then define any separate skill-owned env-bearing command path without letting generic `exec` inherit unrelated skill secrets.

## Reproduction Assessment

Yes, source-backed. Current main writes active skill env keys into `process.env` and generic host `exec` builds child env from that process env without receiving the active skill strip set; the PR body adds after-fix terminal proof against the production env override and sanitizer modules.

## Solution Assessment

Yes for the narrow leak fix: reusing the active skill env registry at the host exec sanitizer boundary is small and matches the ACP precedent. The broader skill CLI usability boundary still needs maintainer/security signoff rather than more code churn in this PR.
