## Summary

Keep open. Current main still routes Google/Gemini reasoning output to the tagged `<think>/<final>` contract while the direct Google transport uses native function-calling payloads, so the reported provider/prompt conflict remains actionable.

## What This Changes

The issue asks OpenClaw to stop Google/Gemini tagged reasoning prompts from causing native tool calls to be emitted as plain text.

## Best Possible Solution

Direct Google Gemini/Gemma API turns with native function calling should avoid a tagged-only text contract, while any Google CLI/replay-only tag requirements stay isolated behind provider-owned hooks and regression tests.

## Reproduction Assessment

Yes for the source-level failure path: current main still resolves Google direct/replay-family reasoning to tagged, injects the disputed prompt wording, and separately sends/parses native Google tool calls. A live model run remains useful for the Gemma 4-specific behavior noted in the follow-up comment.

## Solution Assessment

Yes, the best direction is a narrow provider-owned fix: change Google direct reasoning-output mode or clarify the tagged prompt around native function calls, rather than parsing prose tool-call text into real calls or adding core Google-specific policy.
