## Summary

Keep open: this is a plausible mitigation, but it is not merge-ready because it changes shared HTTP/1 dispatcher policy while the provider transport path still rethrows the first transient failure into fallback, and the proof does not show the affected Anthropic fallback improving.

## What This Changes

Adds a 25s `keepAliveMaxTimeout` default to shared undici HTTP/1 dispatcher construction and updates network tests to expect it.

## Best Possible Solution

Handle transient pre-response socket or fetch failures once in `buildGuardedModelFetch` before cross-provider fallback, or land the shared dispatcher cap only with maintainer acceptance and affected-gateway proof.

## Reproduction Assessment

Yes for the source-level path: current main routes Anthropic Messages through guarded fetch, rethrows the first pre-response transport failure, and the linked issue shows that failure advancing to model fallback. I did not live-reproduce the multi-hour stale-socket trigger.

## Solution Assessment

No. The cap is a plausible mitigation for server keep-alive hint reuse, but the narrow maintainable fix is a bounded same-provider retry in guarded model transport before fallback runs.
