## Summary

Keep open: the PR targets a real current-main DEP0190 path, but its current command construction loses forwarded argument boundaries for safe arguments containing spaces. Existing Windows cmd.exe helper coverage points to a narrower repair before merge.

## What This Changes

The PR refactors scripts/ui.js so Windows shell-runner calls fold args into a single command string before spawn/spawnSync, and adds prepareSpawnCommand unit coverage.

## Best Possible Solution

Route Windows .cmd/.bat/.com UI runner calls through the existing cmd.exe wrapper and argument escaping seam, preferably buildCmdExeCommandLine() or the pnpm-runner helper, while keeping shell:false and adding regression coverage for no DEP0190-triggering args array plus spaced forwarded arguments.

## Reproduction Assessment

Yes. Current main has a direct source reproduction path on Windows with Node v24 and a pnpm.cmd runner, and the PR regression is reproducible by forwarding a safe argument containing spaces such as a Vitest -t pattern.

## Solution Assessment

No. Folding raw args with args.join(" ") removes the deprecation warning but is not the best fix because it preserves neither argument boundaries nor the existing shared cmd.exe escaping behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve forwarded argument boundaries:** `scripts/ui.js:91`
  - body: The shell path folds the command with `${quotedCmd} ${args.join(" ")}`. Safe forwarded args can contain spaces because `assertSafeWindowsShellArgs` allows them, so `node scripts/ui.js test -- -t "path with spaces"` reaches cmd.exe unquoted and is split before pnpm. Use `buildCmdExeCommandLine()` or the pnpm-runner wrapper so each argument is escaped while still avoiding `shell:true` with a separate args array.
  - confidence: 0.86
