## Summary

Keep open: current main and the latest release still have the retained-write early-return leak, but this PR is not merge-ready because the deferred abort-release callback bypasses session-fence release semantics and still lacks after-fix real behavior proof.

## What This Changes

The PR adds retained-lock idle callbacks in embedded attempt abort/dispose release paths so the underlying session lock releases after an active retained write drains.

## Best Possible Solution

Update the deferred abort-release path to run the same fenced release after retained writes drain, add active-write abort/dispose regression coverage, and attach redacted after-fix runtime proof.

## Reproduction Assessment

Yes at source level: current main returns without releasing when waitForRetainedLockIdle() is false, while self-held lock files are not reclaimed. I did not run a live rapid session-switch repro in this read-only review.

## Solution Assessment

No as merge-ready. Deferring release until retained writes drain is the right direction, but the abort path must preserve the normal fence semantics and still needs focused regression coverage plus after-fix proof.
