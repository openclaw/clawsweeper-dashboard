## Summary

Keep this PR open: it targets a real embedded-attempt session-lock leak that current main and v2026.6.8 still leave reachable, but the submitted deferred callback is not merge-ready because it can bypass the normal session-fence handoff and still lacks after-fix real behavior proof.

## What This Changes

The PR adds deferred retained-lock idle callbacks in embedded attempt abort/dispose release paths so an active write can drain before the held session file lock is released.

## Best Possible Solution

Defer the same fence-preserving graceful release work until the retained write drains, add focused active-write abort/dispose regression coverage, and include redacted after-fix proof from the real rapid session-switching flow or an equivalent real-component lock harness.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 return early when waitForRetainedLockIdle() is false in the release/dispose paths, matching the reported active-write lock leak. I did not run a live GUI rapid-switching reproduction in this read-only review.

## Solution Assessment

No, not as-is. Deferred release is the right shape for preserving write serialization, but this implementation needs to preserve the existing fence handoff rather than only calling release after idle.
