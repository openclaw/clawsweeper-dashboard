## Summary

Keep open: the PR targets a plausible orphan-lock failure, but the proposed fix is unsafe because it releases the shared session write lock while the retained write is still active and there is no real behavior proof.

## What This Changes

The PR adds force-release branches in embedded attempt session-lock teardown and abort/prompt release paths when retained-lock idle waiting returns false.

## Best Possible Solution

Preserve session write serialization while fixing the orphan-lock teardown path: defer cleanup until active retained writes complete or install a post-write release handoff, with regression coverage for dispose/abort during an active retained write.

## Reproduction Assessment

No high-confidence real repro is present for the original orphan-lock report. Source inspection confirms the active-write early-return path and the PR-introduced release race, but no live sub-agent run or focused test was provided.

## Solution Assessment

No. The current branch addresses lock availability by force-releasing the active lock, but the best fix should keep the serialization invariant and move release ownership to a post-write or drain path.

- **[P1] Keep retained writes locked until they drain:** `src/agents/embedded-agent-runner/run/attempt.session-lock.ts:805-807`
  - body: Here `waitForRetainedLockIdle()` can only return `false` while this async context is already inside the active retained write.
