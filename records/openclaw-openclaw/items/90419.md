## Summary

Keep open: the lock-leak bug is still present on current main/latest release, but this PR is not merge-ready because the deferred release path drops the existing session-fence handoff and lacks after-fix real behavior proof.

## What This Changes

The PR adds deferred retained-lock idle callbacks in embedded attempt abort/dispose release paths so an active write can finish before the held session file lock is released.

## Best Possible Solution

Defer the same fence-preserving graceful release work until the retained write drains, add focused regression coverage for the active-write abort/dispose path, and include redacted after-fix proof from the real rapid session-switching flow.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 return early when waitForRetainedLockIdle is false in the release/dispose paths, matching the reported active-write lock leak. I did not run a live GUI rapid-switching reproduction in this read-only review.

## Solution Assessment

No, not as-is. Deferred release is the right shape for preserving write serialization, but the implementation needs to preserve the existing fence handoff rather than only calling release after idle.
