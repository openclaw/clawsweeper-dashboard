## Summary

Keep open. Current main does not implement this cron diagnostics surface, and the PR is useful, but it needs changes before merge because diagnostic summary ordering can report an earlier agent warning instead of the delivery error that made the run fail; it also still needs the required changelog entry.

## What This Changes

The PR adds cron run diagnostic metadata and threads it through isolated execution, job state, run logs, Gateway cron schemas, Swift protocol models, and `cron show` output.

## Best Possible Solution

Keep the additive diagnostics design, but make the visible summary prefer the status-determining or latest severe error, add regression coverage for delivery failure after earlier agent diagnostics, and add the active changelog bullet before maintainer review and gates.

## Reproduction Assessment

Not applicable for the original feature request, but yes for the blocking review finding: the PR diff statically shows `mergeCronRunDiagnostics` keeping the first summary while `finalizeCronRun` passes agent diagnostics before delivery errors.

## Solution Assessment

No, not as currently proposed. The overall additive metadata approach is maintainable, but the summary selection should reflect the final failing condition and the user-facing change needs the required changelog entry.
