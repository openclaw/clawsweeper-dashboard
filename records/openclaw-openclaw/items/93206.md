## Summary

Keep open: the PR is a plausible wiring change, but it is not merge-ready because it relies on a disabled-thinking payload for MiniMax M2.x while the linked report and current MiniMax docs say M2.x thinking cannot be disabled. The direct infer path still needs hosted thinking-before-text handling plus SDK family docs/contract alignment.

## What This Changes

Adds a MiniMax simple-completion stream hook to the `minimax-fast-mode` provider stream family and a regression test asserting direct MiniMax M2.7 completions send `thinking: { type: "disabled" }`.

## Best Possible Solution

Revise the fix to handle hosted MiniMax thinking-before-text responses on the direct infer path, preferably through the shared transport/parser path or equivalent provider-owned handling, and document/test the expanded SDK family contract.

## Reproduction Assessment

No hosted current-main reproduction was run in this read-only review. The linked issue provides a concrete `openclaw infer model run` command and response shape, and source inspection shows the missing direct MiniMax wrapper path on current main.

## Solution Assessment

No: adding the disabled-thinking wrapper is only a partial mitigation because MiniMax's M2.x Anthropic contract says thinking cannot be disabled. The safer fix is to preserve text after thinking blocks in the real hosted direct infer response path and keep the SDK family docs aligned.
