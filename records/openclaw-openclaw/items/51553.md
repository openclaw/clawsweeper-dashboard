## Summary

Keep this PR open. Current main still has the stale-cache read in `updateLastRoute`, and the PR's cache-bypass change is narrow, consistent with adjacent locked writers, and not superseded by a current-main implementation.

## What This Changes

The PR changes `updateLastRoute` in `src/config/sessions/store.ts` to call `loadSessionStore(storePath, { skipCache: true })` while holding `withSessionStoreLock`, with the existing concurrent-writer comment pattern.

## Best Possible Solution

Land this PR or an equivalent current-main patch, ideally with a focused regression that warms the session-store cache, performs an external same-key-preserving write, calls `updateLastRoute`, and verifies unrelated external fields are preserved.

## Reproduction Assessment

Yes. A high-confidence reproduction can mirror the existing stale-cache test shape: warm the cache, externally rewrite the session JSON while preserving the cache key, then call `updateLastRoute` and observe current main saving over fields from the external write. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Passing `{ skipCache: true }` inside the session-store lock is the narrowest maintainable fix because the other lock-holding writers already use that pattern to avoid clobbering concurrent writers; adding a regression test would improve coverage but the proposed code direction is correct.
