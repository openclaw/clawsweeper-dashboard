## Summary

Keep open: the visual cleanup is still relevant on current main, but the submitted diff is not mergeable as-is because it can discard explicit labels, lacks real behavior proof, and now conflicts with the current session-label owner modules.

## What This Changes

The PR filters `webchat:g-agent-...` synthetic session names out of Control UI session label resolution and adds helper/chat selector tests for fallback labels.

## Best Possible Solution

Apply a displayName-only synthetic-name filter in the current `session-display.ts`/`session-controls.ts` path, preserve explicit labels, and cover both hidden synthetic display names and preserved synthetic-looking labels.

## Reproduction Assessment

Yes, source-level: current main returns any non-key `row.displayName`, and the selector delegates non-key display names into that resolver. I did not run a browser repro in this read-only review.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is to filter only transport-derived `displayName` values in the current owner modules while leaving explicit `label` values authoritative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit labels when filtering synthetic names:** `ui/src/ui/app-render.helpers.ts:883-895`
  - body: The new filter is applied to `row.label` as well as `row.displayName`, but `label` is explicit session metadata persisted through `sessions.patch`. A user-authored label like `webchat:g-agent-control-tower-main` would be discarded and replaced with fallback text, so filter only transport-derived display names.
