## Summary

Keep open: the UI cleanup is source-reproducible on current main, but the submitted patch still filters explicit session labels and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds synthetic webchat `g-agent` session-name filtering in Control UI label helpers and tests fallback labels in helper and chat selector tests.

## Best Possible Solution

Port a displayName-only synthetic-name filter into `ui/src/ui/chat/session-controls.ts`, preserve explicit `label` values, and cover both hidden synthetic display names and preserved explicit labels.

## Reproduction Assessment

Yes, source-level: current main returns any non-key `row.displayName` and the selector delegates such rows into that resolver. I did not run a browser repro in this read-only review.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is to filter only transport-derived `displayName` values in the current `session-controls.ts` owner module while leaving explicit labels authoritative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit labels when filtering synthetic display names:** `ui/src/ui/app-render.helpers.ts:883-895`
  - body: The new filter is applied to `row.label` as well as `row.displayName`, but `label` is explicit session metadata persisted through `sessions.patch`. A user-authored label like `webchat:g-agent-control-tower-main` would be discarded and replaced with fallback text, so the synthetic-name filter should only apply to transport-derived display names.
  - confidence: 0.9
