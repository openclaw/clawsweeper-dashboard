## Summary

Keep this PR open, but it is not merge-ready: the reported UI label problem is source-reproducible on current main, while the submitted patch drops explicit session labels and still lacks after-fix real behavior proof.

## What This Changes

The branch hides `webchat:g-agent-...` synthetic transport names from Control UI session selector labels and adds helper/chat view tests for fallback labels.

## Best Possible Solution

Port a displayName-only synthetic transport-name filter into `ui/src/ui/chat/session-controls.ts`, keep explicit `label` values authoritative, and cover both hidden synthetic display names and preserved explicit labels.

## Reproduction Assessment

Yes, source-level: current main returns a non-key `displayName` such as `webchat:g-agent-control-tower-main` through `resolveSessionDisplayName`, and the selector path delegates rows with display names into that resolver. I did not run a browser repro in this read-only review.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is to filter only transport-derived `displayName` values in the current `session-controls.ts` module while preserving explicit labels persisted through `sessions.patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit labels when filtering synthetic names:** `ui/src/ui/app-render.helpers.ts:883-895`
  - body: The new filter is applied to `row.label` as well as `row.displayName`, but `label` is explicit session metadata persisted through `sessions.patch`. A user-authored label like `webchat:g-agent-control-tower-main` would be discarded and replaced with fallback text, so the synthetic-name filter should apply to transport-derived display names only.
  - confidence: 0.9
