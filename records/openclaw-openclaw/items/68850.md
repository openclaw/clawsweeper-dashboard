## Summary

Keep open: the underlying UI cleanup is still useful on current main, but the submitted patch has a correctness blocker because it filters explicit labels as if they were transport-derived display names, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a Control UI filter for webchat `g-agent` transport-derived session names and tests selector fallback labels.

## Best Possible Solution

Port a displayName-only synthetic transport filter to `ui/src/ui/chat/session-controls.ts`, preserve explicit `label` values, add focused regression coverage for both hidden synthetic display names and preserved synthetic-shaped labels, and keep the Telegram topic-name bridge tracked separately.

## Reproduction Assessment

Yes, source-level: a current-main session row with key `agent:control-tower:main` and displayName `webchat:g-agent-control-tower-main` still flows through `resolveSessionDisplayName` into the selector label path. I did not run a browser repro because this review was read-only.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is to filter only synthetic transport-derived `displayName` values in the current session-controls module while keeping explicit labels authoritative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit labels when filtering synthetic display names:** `ui/src/ui/app-render.helpers.ts:883-895`
  - body: The new filter is applied to `row.label` as well as `row.displayName`, but labels are explicit session metadata persisted by `sessions.patch`. A user-chosen label like `webchat:g-agent-control-tower-main` would be discarded and replaced with fallback text, so the synthetic-name filter should only apply to transport-derived display names.
  - confidence: 0.9
