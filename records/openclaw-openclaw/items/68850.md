## Summary

Keep open: the PR targets a real current-main UI label leak, but the submitted diff still drops explicit user-authored labels and needs to be ported to the current session-controls module; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR filters webchat `g-agent` synthetic transport names from Control UI session selector labels and adds focused helper/UI tests for fallback labels.

## Best Possible Solution

Apply a displayName-only synthetic transport-name filter in `ui/src/ui/chat/session-controls.ts`, keep explicit `label` values authoritative, and add regression coverage for both hidden synthetic display names and preserved explicit labels.

## Reproduction Assessment

Yes, source-level: current main's `resolveSessionDisplayName` returns a non-key displayName such as `webchat:g-agent-control-tower-main` before falling back, and `resolveSessionScopedOptionLabel` routes that row into the resolver. I did not run a browser repro in this read-only review.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is to filter only transport-derived `displayName` values in the current session-controls module while preserving explicit labels persisted through `sessions.patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit labels when filtering synthetic display names:** `ui/src/ui/app-render.helpers.ts:883-895`
  - body: The new filter is applied to `row.label` as well as `row.displayName`, but `label` is explicit session metadata persisted through `sessions.patch`. A user-authored label like `webchat:g-agent-control-tower-main` would be discarded and replaced with fallback text, so the synthetic-name filter should only apply to transport-derived display names.
  - confidence: 0.9
