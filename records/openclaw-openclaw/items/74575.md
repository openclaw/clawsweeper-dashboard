## Summary

Keep open. Current main has guided onboarding, section-based configuration, plugin config prompts, schema-backed config editing, and Crestodian typed setup/repair, but it does not provide the requested assistant-style recommender that interviews the user, probes host capabilities, proposes best-practice settings, and revisits recommendations after new features ship.

## What This Changes

Request a built-in automation that interviews users, checks host capabilities such as CPU, memory, and GPU, then recommends and applies best-practice OpenClaw configuration over time.

## Best Possible Solution

Define a scoped next-generation setup experience that builds on onboarding, `openclaw configure`, plugin setup metadata, config schema lookup, and Crestodian typed operations. It should gather user goals, use consented local capability discovery, allow plugins to contribute recommendation metadata through documented contracts, and keep writes explicit, audited, reversible, and documented.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is high-confidence from docs and source inspection: the existing flows configure known sections and typed operations, but do not perform an interview-driven host-capability recommendation pass.

## Solution Assessment

No, not as an unscoped built-in config writer linked to the normal main agent. A safer solution is a maintainer-scoped design that extends typed setup/config/plugin metadata while preserving explicit approval, audit, rollback, and security boundaries.
