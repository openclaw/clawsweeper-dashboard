## Summary

Keep open. Current main has guided onboarding, `openclaw configure`, plugin setup metadata, and Crestodian typed setup/repair, but it still does not provide the requested assistant-style configuration recommender that interviews the user, probes host resources such as CPU/memory/GPU, proposes best-practice settings, and revisits recommendations as features change. This is broad product and security design work, not a cleanup-close or safe automated repair.

## What This Changes

Request for a built-in guided configuration automation that asks goal-oriented questions, inspects host capabilities, recommends OpenClaw settings, and can update recommendations as new features ship.

## Best Possible Solution

Define a scoped next-generation setup UX that builds on onboarding, `openclaw configure`, plugin setup metadata, and Crestodian typed operations: gather user goals, use consented host/resource discovery, let plugins contribute schema-backed recommendation metadata where appropriate, and keep persistent writes explicit, audited, reversible, and documented.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is high-confidence from source and docs inspection: onboarding, configure, plugin prompts, and Crestodian do not provide a goal-driven host-capability recommender.

## Solution Assessment

No, not as a blind built-in config writer linked to the normal agent. The safer path is a maintainer-scoped design that extends typed setup/config surfaces and plugin metadata while preserving explicit approval, audit, and rollback boundaries.
