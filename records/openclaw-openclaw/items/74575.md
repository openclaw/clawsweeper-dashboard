## Summary

Keep open. Current main has guided onboarding, section-based `openclaw configure`, plugin field prompts, doctor repairs, and Crestodian typed setup, but it still does not implement the requested assistant-style configuration recommender that interviews the user, probes host resources, proposes best-practice settings, and revisits recommendations when features change. The existing issue discussion already reached the same keep-open direction, and the remaining work is broad product and security design rather than conservative cleanup.

## What This Changes

Request a built-in assistant-style configuration wizard that interviews users, probes host resources, recommends best-practice settings, and updates configuration as features change.

## Best Possible Solution

Define a scoped next-generation setup UX that builds on current onboarding, `openclaw configure`, doctor, plugin manifest/setup metadata, and Crestodian typed operations: gather user goals, perform consented host/resource discovery, allow plugins to contribute schema-backed recommendation hints where appropriate, and keep persistent writes explicit, audited, reversible, and documented.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is high-confidence from source and docs inspection: the existing onboarding/configure/Crestodian/plugin setup surfaces do not provide a goal-driven host-capability recommender.

## Solution Assessment

No, not as a broad built-in config writer linked to the normal agent. The safer direction is a maintainer-scoped design around typed setup operations, explicit approval/audit/rollback, and plugin-owned recommendation metadata rather than hidden autonomous config mutation.
