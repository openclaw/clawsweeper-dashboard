## Summary

Keep open. Current main has guided onboarding, section-based `openclaw configure`, plugin setup prompts, and Crestodian typed setup/repair, but it still does not provide the requested assistant-style configuration recommender that interviews the user, probes host capabilities, proposes best-practice settings, and revisits recommendations as OpenClaw changes. This is broad product and security design work, not a cleanup close or safe automated repair.

## What This Changes

Request to add a built-in guided configuration automation that can run basic host checks, ask goal-oriented questions, recommend OpenClaw settings, and retune configuration as features evolve.

## Best Possible Solution

Define a scoped next-generation setup UX that builds on onboarding, `openclaw configure`, plugin setup metadata, and Crestodian typed operations: gather user goals, use consented resource discovery, let plugins contribute schema-backed recommendation metadata where appropriate, and keep persistent writes explicit, audited, reversible, and documented.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is high-confidence from source and docs inspection: onboarding, configure, plugin prompts, and Crestodian do not provide a goal-driven host-capability configuration recommender.

## Solution Assessment

No, not as a blind built-in config writer linked to the normal agent. The safer path is a maintainer-scoped design that extends typed setup/config surfaces and plugin metadata while preserving explicit approval, audit, and rollback boundaries.

## Risks / Open Questions

- The product contract is undefined: an assistant that recommends or writes persistent configuration needs clear authority, approval, rollback, and audit boundaries.
- Host probing for CPU, memory, GPU, installed tools, and credentials can expose local machine details and should be explicitly consented and minimized.
- Plugin/provider recommendation metadata is not currently a generic contract, so implementing this narrowly in core could bypass the plugin boundary or hard-code owner-specific defaults.
