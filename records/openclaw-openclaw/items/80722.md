## Summary

Keep open: current `main` still prints the restart warning unconditionally after `openclaw config set`, while the existing gateway reload contract and docs classify `agents.list` / agent-model changes as hot-reloadable in the default mode. The fix should make the CLI notice follow the existing reload plan instead of adding an agentRuntime-only heuristic.

## What This Changes

The issue requests that `openclaw config set` avoid or rephrase its restart warning for `agents.list[N].model.*` edits that apply without a gateway restart.

## Best Possible Solution

Make `config set` post-write messaging reuse the existing reload-plan semantics so hot-safe agent/model edits do not tell operators to restart, while mixed or restart-required edits still do.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a successful `config set agents.list[0].model.primary ...` write will hit the unconditional success log, while the current reload planner and tests classify `agents.list` as hot-reloadable.

## Solution Assessment

No. The agentRuntime-only suggested fix is not the best boundary because current OpenClaw already decides reload behavior by changed path and `gateway.reload.mode`; the CLI warning should follow that shared contract.
