## Label Justifications

- `P2`: This is a meaningful session-routing feature with clear multi-agent workflow impact, but it is not an urgent regression or outage.
- `impact:session-state`: The issue changes which cron, DM, and group turns share transcripts, prompt cache, queue state, and session lifecycle.
- `impact:security`: Named inbound routing could expose private or group context across chat and automation boundaries unless the trust model is explicit.

## Summary

Keep open: current main and latest v2026.6.6 still only support cron `session:<id>` targets plus DM-scoped/identity-linked inbound routing; they do not provide a per-agent inbound route-to-named-session contract. The related shared-session issues and open groupScope PR are useful context, but none is a merged or complete replacement for this cron-plus-inbound named-session request.

## What This Changes

The issue requests configurable per-agent inbound routing into a canonical named session so cron and human conversations can share transcript and prompt-cache state.

## Best Possible Solution

Design one additive, opt-in canonical session-routing contract with isolated defaults, validated named-session targets, origin-preserving delivery metadata, documented reset/security semantics, and coordinated scope with the related shared-session issues.

## Reproduction Assessment

Yes at source level: current main deterministically derives inbound keys from DM scope or channel/peer identity, and the requested named inbound target is absent from schema, types, and route resolution. No live iMessage setup was exercised in this read-only review.

## Solution Assessment

Unclear as proposed: `agents.list[].session.routeInboundTo` is plausible, but maintainers should choose whether the permanent abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
