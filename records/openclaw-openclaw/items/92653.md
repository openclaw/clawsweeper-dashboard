## Label Justifications

- `P2`: This is a meaningful session-continuity feature with clear workflow impact, but it is not an urgent regression or outage.
- `impact:session-state`: The issue changes which cron, DM, and group turns share transcripts, prompt cache, queue state, and session lifecycle.
- `impact:security`: Named inbound routing can expose chat or cron context across trust boundaries unless the opt-in and delivery-origin rules are explicit.

## Summary

Keep open: current main still has cron `session:<id>` targets plus DM-only routing controls, but no per-agent inbound route-to-named-session contract for DMs or groups. This remains a valid session-routing feature that needs maintainer product and security design before implementation.

## What This Changes

The issue requests per-agent inbound routing into a named canonical session so cron jobs and human chat messages can share one transcript and prompt-cache context.

## Best Possible Solution

Design an additive, opt-in canonical session-routing contract with isolated defaults, validated named-session targets, origin-preserving delivery metadata, reset/security docs, and coordinated scope with the broader shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives inbound session keys from dmScope/identityLinks or channel/peer identity, and the strict schemas cannot express a named canonical inbound target.

## Solution Assessment

Unclear as proposed. `agents.list[].session.routeInboundTo` is plausible, but maintainers should decide whether the permanent abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
