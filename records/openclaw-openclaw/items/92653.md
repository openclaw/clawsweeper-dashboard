- `impact:session-state`: The request changes which cron, DM, and group turns share transcript, prompt cache, queue state, and lifecycle.

## Summary

Keep open: current main and v2026.6.8 still lack a per-agent inbound route-to-named-session contract, and the related shared-session issues and groupScope PR overlap without superseding this cron-plus-inbound use case.

## What This Changes

The issue requests a per-agent config contract that routes inbound channel messages into a named persistent session shared with cron jobs.

## Best Possible Solution

Design one additive, opt-in canonical session-routing contract with validated named targets, origin-preserving delivery metadata, documented reset/security semantics, and coordinated scope with the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives inbound session keys from DM scope, identity links, or channel/peer identity, while cron can bind to an explicit named session and strict schemas cannot express a per-agent inbound named-session target.

## Solution Assessment

Unclear as proposed. `agents.list[].session.routeInboundTo` is plausible, but maintainers should first decide whether the durable abstraction belongs on agent config, route bindings, identity/session aliases, or a broader shared-session model.
