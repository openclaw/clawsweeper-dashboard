## Summary

Keep open: current main and v2026.6.6 still support cron `session:<id>` targets plus DM-scoped routing, but there is no per-agent inbound route-to-named-session contract for DMs or groups. This is a valid session-routing feature that needs product and security design before implementation.

## What This Changes

Request an opt-in per-agent configuration that routes inbound DMs or group messages into a named persistent session so human chats and cron jobs share one transcript and prompt cache.

## Best Possible Solution

Design one additive, opt-in canonical session-routing contract with isolated defaults, validated named-session targets, origin-preserving delivery metadata, documented reset/security semantics, and coordinated scope with the related shared-session issues.

## Reproduction Assessment

Yes at source level for the feature gap: current main deterministically derives inbound session keys from DM scope or channel/peer identity and cannot express a named canonical inbound target. This is a missing capability rather than a broken existing contract.

## Solution Assessment

Unclear as proposed: `agents.list[].session.routeInboundTo` is plausible, but maintainers should decide whether the permanent abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
