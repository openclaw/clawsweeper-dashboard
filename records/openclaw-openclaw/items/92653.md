## Summary

Keep open: current main and v2026.6.8 still support cron `session:<id>` targets plus DM-oriented session controls, but they do not provide a per-agent inbound route-to-named-session contract for DMs or groups. The prior discussion and related open trackers are useful design inputs, but no current implementation or canonical replacement has landed.

## What This Changes

The issue requests per-agent configuration that routes inbound channel messages into a named persistent session so cron jobs and human conversations can share transcript context.

## Best Possible Solution

Design one additive canonical session-routing contract with isolated defaults, validated named targets, origin-preserving delivery metadata, reset/security docs, and coordination with the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 deterministically derive inbound keys from DM scope or channel/peer identity, while the strict schema cannot express the requested per-agent named inbound session target.

## Solution Assessment

Unclear as proposed. `agents.list[].session.routeInboundTo` is plausible, but maintainers should first decide whether the durable abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
