## Summary

Keep open: current main and v2026.6.6 still support cron named sessions plus DM scoping, but they do not provide a per-agent inbound route-to-named-session contract. The request is valid and source-reproducible, but it needs maintainer product and security design before implementation.

## What This Changes

The issue requests a per-agent config surface that routes inbound DMs and group messages into a named persistent session so cron and human interactions can share transcript context.

## Best Possible Solution

Design one additive canonical session-routing abstraction with isolated defaults, validated named-session targets, origin-preserving delivery metadata, documented reset/security semantics, and coordination with the related shared-session issues.

## Reproduction Assessment

Yes at source level: current route resolution deterministically derives inbound session keys from dmScope, identityLinks, channel, and peer identity, and there is no typed named-session target. I did not run live iMessage/cron E2E because the source-level feature gap is direct and no patch is under review.

## Solution Assessment

Unclear as proposed: agents.list[].session.routeInboundTo is plausible, but maintainers should choose whether the permanent abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
