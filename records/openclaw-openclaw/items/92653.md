## Summary

Keep open: current main and v2026.6.8 still lack a per-agent inbound route-to-named-session contract, and the related shared-session issues overlap without superseding this cron-plus-inbound use case.

## What This Changes

Request a per-agent config surface that routes inbound DMs or groups into a named persistent session so cron and human turns share transcript and prompt-cache context.

## Best Possible Solution

Design one additive, opt-in canonical session-routing contract with validated named targets, origin-preserving delivery metadata, documented reset/security semantics, and coordinated scope with the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives inbound session keys from DM scope or channel/peer identity, while the strict config schemas cannot express a named per-agent inbound session target.

## Solution Assessment

Unclear as proposed. agents.list[].session.routeInboundTo is plausible, but maintainers should first decide whether the durable abstraction belongs on agent config, route bindings, identity/session aliases, or a broader shared-session model.
