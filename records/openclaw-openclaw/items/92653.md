## Summary

Keep open: current main and v2026.6.9 still do not provide a per-agent inbound route-to-named-session contract, and the related shared-session/group-scope items overlap without covering the cron-plus-inbound named-session use case.

## What This Changes

Request to let each agent route inbound DMs or groups into a configured named persistent session so cron jobs and human conversations share one transcript and prompt cache.

## Best Possible Solution

Design one additive, opt-in canonical session-routing contract with validated named targets, origin-preserving delivery metadata, documented reset/security semantics, and coordinated scope with the related shared-session trackers.

## Reproduction Assessment

Yes, at source level. Current main deterministically derives inbound session keys from DM scope, identity links, or channel/peer identity, while cron can bind to an explicit named session and strict schemas cannot express a per-agent inbound named-session target.

## Solution Assessment

Unclear as proposed. `agents.list[].session.routeInboundTo` is plausible, but maintainers should first choose whether the durable abstraction belongs on agent config, route bindings, identity/session aliases, or a broader shared-session model.
