- `P2`: This is a meaningful session-routing feature with clear user value, but it is not an urgent regression or outage.
- `impact:session-state`: The request changes which cron, DM, and group turns share transcripts, prompt cache, queue state, and session lifecycle.
- `impact:security`: The requested cross-surface context sharing can expose private or group conversation state unless trust boundaries are explicit.

## Summary

Keep open: current main and v2026.6.6 support cron `session:<id>` targets plus DM scoping/identity links, but they still lack a per-agent inbound route-to-named-session contract for DMs or groups. This is a valid core session-routing feature, but it needs maintainer product and security design before implementation.

## What This Changes

Route inbound DM and group messages for an agent into a named persistent session so cron and human turns can share one transcript and prompt cache.

## Best Possible Solution

Design an additive opt-in canonical session-routing contract with isolated defaults, validated named-session targets, origin-preserving delivery metadata, documented reset/security semantics, and coordination with the related shared-session trackers.

## Reproduction Assessment

Yes at source level: current main derives inbound session keys from `dmScope`/`identityLinks` for direct peers or channel/peer identity for non-direct peers, and the strict schema cannot express a named inbound target. This is a feature gap rather than a failing existing contract.

## Solution Assessment

Unclear as proposed: `agents.list[].session.routeInboundTo` is plausible, but maintainers should decide whether the durable abstraction belongs in agent config, route bindings, identity links, channel groups, or a typed session-alias model.
