## Label Justifications

- `P2`: The request is a normal-priority session-routing improvement with real multi-agent workflow impact but no current emergency regression.
- `impact:session-state`: The issue is about intentionally sharing or separating transcript and prompt-cache state across inbound and cron sessions.
- `impact:security`: Changing inbound routing can merge conversations across peers or channels, so it needs explicit security and isolation design.

## Summary

Keep open: current main still supports cron named sessions and DM-scoped routing, but it does not provide the requested per-agent inbound route-to-named-session contract. The issue remains a valid feature/design item because its named cron plus inbound-session convergence is distinct from the broader identity and group/channel shared-session trackers.

## What This Changes

Request per-agent inbound routing to a named persistent session so cron jobs and inbound channel conversations can share one transcript and prompt-cache context.

## Best Possible Solution

Design an additive, opt-in canonical session-routing contract with validated targets, origin-preserving delivery metadata, reset semantics, and a clear relationship to identity links and shared-channel routing.

## Reproduction Assessment

Yes. Source inspection is enough: cron can use `session:<name>`, while inbound routing still derives sessions from `dmScope`, identity links, and peer shape with no named inbound target field.

## Solution Assessment

Unclear as proposed. `agents.list[].session.routeInboundTo` is plausible, but maintainers should decide whether the permanent abstraction belongs on agent config, route bindings, identity links, channel groups, or a typed session-alias model.
