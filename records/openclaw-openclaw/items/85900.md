## Summary

Keep open: current main and v2026.6.9 now protect pending stream creation, but cron still uses the explicit payload timeout as both the model stream idle ceiling and the cron wall-clock watchdog, with no merged per-candidate stream-stall/fallback-budget policy. This remains the best canonical tracker for the payload-pinned cron fallback gap.

## What This Changes

The issue reports that payload-pinned cron model calls can hang on provider stream stalls without advancing to the configured fallback chain before the cron timeout kills the run.

## Best Possible Solution

Define and implement a separate failover-worthy stream-stall attempt budget for cron model candidates that leaves enough wall-clock budget for configured fallbacks before the cron watchdog fires.

## Reproduction Assessment

Yes at source level: current main shows cron can apply the explicit payload timeout as both the LLM idle ceiling and the cron wall-clock watchdog. I did not run a live stalled-provider replay in this read-only review.

## Solution Assessment

No, the issue is not fully solved on current main. The first-event idle wrapper is useful, but the best fix still needs a separate per-candidate stall/fallback budget or equivalent cron deadline policy so fallbacks have time to run.
