## Label Justifications

- `P1`: The report describes a real scheduled-agent workflow missing recovery when a provider stalls, despite configured healthy fallback models.
- `impact:crash-loop`: The affected cron run can hang until a watchdog terminates the job instead of completing through fallback.
- `impact:auth-provider`: The remaining problem is model/provider fallback routing and timeout classification for stalled streams.

## Summary

Close as superseded by the existing idle-timeout configuration thread: the timeout propagation part is already merged, while the remaining stream-stall/fallback gap is the same missing independent model-stream idle or first-chunk timeout contract tracked in the canonical open issue.

## What This Changes

The issue reports that cron jobs with a payload-pinned model do not advance to the configured fallback chain when the provider opens a stream but sends no first chunk.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/78361 as the canonical place to decide the idle or first-chunk timeout contract, then implement the chosen timeout hierarchy so stalled model streams fail early enough for cron fallbacks to run before the outer job budget is exhausted.

## Reproduction Assessment

Yes, at source level: current main wires payload model fallbacks but uses the same explicit cron timeout as the LLM idle watchdog ceiling and the outer cron job timeout, so a no-chunk stream can exhaust the job budget before a fallback has useful time to run. I did not replay the live MiniMax outage.
