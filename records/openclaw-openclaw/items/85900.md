## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron's expected output and delivery path by timing out the job before a fallback response is produced.
- `impact:auth-provider`: The issue is about provider/model routing and a configured model fallback chain not engaging under a stalled primary.

## Summary

Keep open: current main still has the same source-level race where an explicit cron `timeoutSeconds` can become both the silent stream idle ceiling and the outer cron watchdog budget, so a no-first-chunk primary may consume the whole cron run before configured fallbacks get a useful attempt.

## What This Changes

The issue asks OpenClaw to treat no-first-chunk cron model stream stalls as retryable candidate failures so configured model fallbacks can run before the cron watchdog kills the job.

## Best Possible Solution

Define a cron-specific per-candidate first-chunk or stream-idle budget that is shorter than the cron wall-clock watchdog while preserving explicit long provider/run timeouts, then cover payload `model` plus configured fallbacks plus a no-first-chunk stream in regression tests.

## Reproduction Assessment

Yes at source level, but not with a live provider replay. A synthetic provider that opens a stream and never yields while a cron has `payload.model`, `timeoutSeconds`, and configured fallbacks should exercise the same race between the stream idle timer and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The behavior should be fixed, but the maintainable solution needs a deliberate split between retryable candidate-idle failures and the whole cron budget instead of blindly shortening all explicit cron/provider timeouts.
