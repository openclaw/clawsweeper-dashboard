## Label Justifications

- `P1`: A scheduled agent workflow can miss a full run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported timeout drops the cron job's expected output or delivery before any fallback response is produced.

## Summary

Keep open: current main and v2026.6.9 still use the cron payload timeout as both the outer cron watchdog and the cron-triggered model stream idle ceiling, and there is no merged per-candidate first-chunk/fallback-budget fix. The issue remains the best canonical tracker for the stream-stall fallback gap.

## What This Changes

The issue reports that cron jobs with a payload-pinned model do not trigger configured model fallbacks when the provider stream opens but stalls before the first chunk.

## Best Possible Solution

Define and implement a provider/attempt stream-stall budget that fails the current candidate and advances fallback before the cron wall-clock deadline, while preserving explicit slow-provider timeouts and caller-abort semantics.

## Reproduction Assessment

Yes at source level, but not with a live stalled-provider replay. A cron job with payload.model, configured fallbacks, and an opened-but-silent provider stream can use the same explicit timeout for the stream watchdog and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout hierarchy. Treating no-first-chunk stalls as failover-worthy matches the documented fallback contract, but the implementation should avoid shortening all cron or provider timeouts unconditionally.
