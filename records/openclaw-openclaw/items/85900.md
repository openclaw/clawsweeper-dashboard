## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron's expected output or delivery path by timing out before a fallback response is produced.
- `impact:auth-provider`: The issue is about provider/model routing and configured fallback candidates not engaging under a stalled primary model.

## Summary

Keep open: current main wires cron model fallbacks and idle-timeout detection, but explicit cron `timeoutSeconds` can still become both the silent-stream idle ceiling and the outer cron wall-clock watchdog, so a no-first-chunk primary can consume the run before fallback gets a useful attempt.

## What This Changes

The issue reports that cron jobs with a payload-pinned primary model can miss configured fallback models when the primary provider opens a stream but never emits a first chunk.

## Best Possible Solution

Define a per-candidate stream-idle or first-chunk budget for cron fallback attempts that is shorter than the cron wall-clock watchdog while preserving explicit long provider/run timeouts, then add focused regression coverage for the no-first-chunk fallback case.

## Reproduction Assessment

Yes at source level, but not with a live provider replay. A synthetic provider that opens a stream and never yields while a cron has `payload.model`, `timeoutSeconds`, and configured fallbacks should exercise the same idle-timeout versus outer-watchdog race.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The current implementation is not enough for this report, and the safer fix should separate retryable candidate-idle failures from the whole cron budget rather than simply shortening all explicit cron/provider timeouts.
