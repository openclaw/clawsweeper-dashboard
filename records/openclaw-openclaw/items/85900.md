## Label Justifications

- `P2`: The report describes a bounded cron reliability bug that can miss scheduled runs but does not make the core runtime unusable.
- `impact:crash-loop`: The affected behavior is a hung/stalled cron run that eventually times out under the scheduler watchdog.
- `impact:auth-provider`: The bug is in model/provider fallback routing after a provider stream stalls.

## Summary

Close as implemented: current main and v2026.5.22 already carry cron `timeoutSeconds` into the embedded runner, convert an opened-but-silent model stream into an LLM idle timeout, and route that timeout through the existing model fallback chain.

## What This Changes

The issue reports that cron jobs with a payload-pinned model can hang on an opened-but-silent provider stream without engaging the configured fallback chain.

## Best Possible Solution

Keep the shipped generic idle-timeout and model-fallback implementation; if this recurs on v2026.5.22 or newer, reopen with a fresh run log showing the stalled provider path.

## Reproduction Assessment

No current-main failing reproduction is established. The report targets 2026.5.20, while current main and v2026.5.22 source/tests show the opened-but-silent stream path aborts via LLM idle timeout and becomes fallback reason `timeout`.

## Solution Assessment

Yes. The current implementation is the maintainable path because it reuses the shared LLM idle-timeout wrapper and model fallback machinery instead of adding cron-specific retry logic.
