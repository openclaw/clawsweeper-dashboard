## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a stalled primary provider consumes the cron budget before healthy fallback models are tried.
- `impact:message-loss`: The reported failure drops the cron's expected output path by ending the job before a fallback response is produced.
- `impact:auth-provider`: The issue is about provider/model routing and configured model fallback behavior under a stalled primary.

## Summary

Keep open: current main still has the timeout race described in the report. Cron payload `timeoutSeconds` is used both as the outer cron watchdog budget and, for explicit cron runs, as the stream idle-timeout ceiling, so a no-first-chunk primary can consume the whole cron run before configured fallbacks get a useful attempt.

## What This Changes

The issue requests that cron jobs with a payload-pinned primary model engage their configured fallback chain when the primary provider opens a stream but never yields a chunk.

## Best Possible Solution

Define a per-candidate stream-idle or first-chunk budget for cron fallback attempts that fires before the cron wall-clock watchdog while preserving explicit long provider/run timeouts, then add regression coverage for payload `model` plus configured fallbacks plus a no-first-chunk stream.

## Reproduction Assessment

Yes at source level, but not as a live-provider replay. A synthetic provider that opens a stream and never yields under an isolated cron with payload `model`, `timeoutSeconds`, and configured fallbacks should exercise the same timeout race.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The durable fix should make a retryable candidate-idle or first-chunk failure fire before the whole cron wall-clock budget is exhausted, without globally shortening legitimate long cron/provider calls.
