## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a primary provider stalls, even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron's expected output/delivery path by timing out the job before a fallback response is produced.
- `impact:auth-provider`: The issue is about provider/model routing and the configured model fallback chain not engaging under a stalled primary.

## Summary

Keep open: current main has cron fallback and stream-idle-timeout machinery, but source review shows the explicit cron `timeoutSeconds` can also become the first silent stream's idle-timeout ceiling while the outer cron watchdog uses the same budget. That leaves a plausible current-main path where a no-first-chunk primary consumes the whole cron run before the configured fallback chain gets a useful attempt.

## What This Changes

Treat a cron payload-pinned primary model that opens a stream but sends no chunks as a retryable candidate failure so configured fallback models can run before the cron timeout aborts the job.

## Best Possible Solution

Define a per-candidate stream-idle or first-chunk budget for cron fallback attempts that is shorter than the cron wall-clock watchdog while preserving explicit long provider/run timeouts, then add regression coverage for payload `model` plus configured fallbacks plus a no-first-chunk stream.

## Reproduction Assessment

Yes at source level, but not with a live provider replay. A synthetic provider that opens a stream and never yields while a cron has `payload.model`, `timeoutSeconds`, and configured fallbacks should exercise the same race between the stream idle timer and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The current implementation is not enough for this report, but the safest fix should separate retryable candidate-idle failures from the whole cron budget instead of simply shortening all explicit cron/provider timeouts.
