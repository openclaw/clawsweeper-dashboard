## Label Justifications

- `P2`: This is a real cron/model fallback bug with production impact, but the blast radius is bounded to scheduled runs that hit a stalled provider window.
- `impact:auth-provider`: The affected behavior is provider/model routing: a pinned primary model stall prevents the configured fallback model chain from being used.

## Summary

Keep open: current main still maps an explicit cron payload timeout to the stream idle watchdog itself, so a no-first-chunk stream can lose the race to the outer cron watchdog before the fallback chain has useful time to advance. The docs promise configured fallback chains still apply for cron-selected primary models, so this remains a real cron/model-routing bug rather than stale support noise.

## What This Changes

The issue asks OpenClaw cron runs with a payload-pinned model to treat a stalled stream before the first chunk as a model failure so the configured fallback chain can run before the cron watchdog kills the job.

## Best Possible Solution

Define a model-call idle or first-chunk budget for cron that fails a stalled provider before the outer cron timeout, preserves explicit slow-provider opt-ins, and adds regression coverage proving a payload-pinned primary stall advances to configured fallbacks.

## Reproduction Assessment

Yes, at source level. Current main forwards cron `timeoutSeconds` into the embedded stream idle timeout and also uses it as the outer cron watchdog budget, so a no-first-chunk provider stall has no guaranteed smaller failure window before cron cancellation; I did not replay the live MiniMax outage.

## Solution Assessment

Unclear. Treating no-first-chunk stream silence as fallback-eligible is the right direction, but maintainers need to choose the supported timeout hierarchy or config surface so slow providers are not broken while cron fallbacks get time to run.
