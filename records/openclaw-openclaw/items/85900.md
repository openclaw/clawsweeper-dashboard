## Label Justifications

- `P1`: A scheduled agent workflow can miss a complete run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron job's expected output or delivery by timing out before a fallback response is produced.
- `impact:auth-provider`: The issue concerns provider/model routing and the configured model fallback chain not engaging under a stalled primary.

## Summary

Keep open: current main and v2026.6.6 have partial no-first-chunk idle-timeout coverage, but source review still shows explicit cron `timeoutSeconds` can be the same ceiling for the stream-idle watchdog and the outer cron wall-clock timeout, leaving no reliable fallback attempt when the first candidate stalls.

## What This Changes

The issue reports that cron jobs with a payload-pinned primary model can exhaust the cron timeout on an opened-but-silent provider stream without engaging configured model fallbacks.

## Best Possible Solution

Keep this as the canonical stream-stall cron fallback issue and define a per-candidate first-chunk or idle budget that is shorter than the cron wall-clock watchdog while preserving explicit long provider/run timeout semantics.

## Reproduction Assessment

Yes at source level, but not with a live stalled-provider replay. A cron with `payload.model`, configured fallbacks, and a provider stream that opens but never yields can hit the same idle-timeout and cron-watchdog budget before fallback gets a useful attempt.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The current implementation is only a partial fix; the safer direction is a per-candidate stream-stall budget that leaves room for fallback rather than shortening all cron or provider timeouts unconditionally.
