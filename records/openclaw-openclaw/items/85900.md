## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron's expected output or delivery path by timing out before a fallback response is produced.
- `impact:auth-provider`: The issue is about model/provider routing and the configured model fallback chain not engaging under a stalled primary.

## Summary

Keep open: current main still lets an explicit cron `timeoutSeconds` become both the silent-stream idle ceiling and the outer cron watchdog budget, so a no-first-chunk primary model can consume the run before configured fallback models get a useful attempt.

## What This Changes

The issue asks cron payload model overrides to treat no-first-chunk stream stalls as retryable model failures so configured fallback chains can run before the cron watchdog ends the job.

## Best Possible Solution

Introduce a cron-safe per-candidate no-first-chunk or stream-idle budget shorter than the job wall-clock budget, with regression coverage proving configured fallbacks get a real attempt while explicit long provider and run timeouts remain respected.

## Reproduction Assessment

Yes at source level, but not with a live provider replay. A synthetic provider that opens a stream and never yields while a cron has `payload.model`, `timeoutSeconds`, and configured fallbacks should exercise the same race between the stream idle timer and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The maintainable fix should separate retryable candidate-idle failures from the whole cron wall-clock budget instead of simply shortening all explicit cron or provider timeouts.
