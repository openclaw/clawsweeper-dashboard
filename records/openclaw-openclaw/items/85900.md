## Label Justifications

- `P1`: A scheduled agent workflow can hang until the cron watchdog and miss the run instead of using healthy configured provider fallbacks.
- `impact:crash-loop`: The reported failure is a long-running cron/model-call hang that ends in a watchdog timeout.
- `impact:message-loss`: The missed cron run can suppress the scheduled result or failure output the user expected.
- `impact:auth-provider`: The affected code path is provider/model fallback routing for a pinned cron model.

## Summary

Keep open: current main still uses the cron payload timeout as both the per-model idle window and the outer job watchdog, so a silent first candidate can consume the whole run budget before meaningful fallback. Related issues cover broader timeout/config behavior, but none clearly owns this narrow current cron fallback failure.

## What This Changes

The issue reports that cron jobs with a pinned payload model do not advance to configured fallbacks when the provider opens a stream but stalls before the first chunk.

## Best Possible Solution

Define a cron model-attempt or first-byte timeout budget that fails a stalled candidate before the cron job watchdog, while preserving explicit slow-provider opt-ins and proving that later fallback candidates make real requests before the outer abort.

## Reproduction Assessment

Yes, source-reproducible: current code sends the same payload timeout to the model idle watchdog and the cron job watchdog, then checks the cron abort signal before each fallback candidate. I did not replay the live MiniMax stall.

## Solution Assessment

No, current main is not the best solution because the fallback machinery recognizes idle timeouts but can reach them at the same deadline as the outer cron abort. The maintainable fix needs a distinct attempt/first-byte budget or explicit timeout precedence that leaves room for fallback.
