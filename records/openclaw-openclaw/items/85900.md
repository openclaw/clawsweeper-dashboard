## Label Justifications

- `P1`: A scheduled agent workflow can miss a whole run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported failure drops the cron job's expected output or delivery by timing out before a fallback response is produced.
- `impact:auth-provider`: The issue concerns provider/model routing and the configured model fallback chain not engaging under a stalled primary.

## Summary

Keep open: current main has partial no-first-chunk idle-timeout coverage, but an explicit cron `payload.timeoutSeconds` can still be both the stream-idle ceiling and the outer cron wall-clock deadline, so fallback is not reliably given time to engage for the reported stalled primary model case.

## What This Changes

The issue reports that cron jobs with a payload-pinned model can hang on a provider stream that opens but sends no first chunk, preventing the configured fallback chain from engaging before the cron watchdog kills the job.

## Best Possible Solution

Keep this as the canonical cron stream-stall fallback issue until maintainers settle a timeout hierarchy that gives each candidate a shorter first-chunk or idle budget than the cron wall-clock deadline and lands proof for a stalled provider stream with configured fallbacks.

## Reproduction Assessment

Yes at source level, but not with a live stalled-provider replay in this review. A cron `agentTurn` with `payload.timeoutSeconds`, a primary model, and fallbacks can still use the same budget for stream idle detection and the outer cron wall-clock abort.

## Solution Assessment

Unclear until maintainers choose the timeout policy. The narrowest maintainable direction appears to be a per-candidate first-chunk/idle budget that leaves room for fallback, rather than shortening all cron or provider timeouts unconditionally.
