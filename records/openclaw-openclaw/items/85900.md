## Label Justifications

- `P1`: A scheduled agent workflow can miss its run when a stalled primary provider consumes the cron deadline despite configured healthy fallbacks.
- `impact:message-loss`: The reported failure drops the cron job's expected response or delivery before any fallback model can produce output.
- `impact:auth-provider`: The affected behavior is provider/model fallback routing and timeout classification for model calls.

## Summary

Keep open: current main and v2026.6.9 still tie cron `payload.timeoutSeconds` to both the outer cron watchdog and the cron-triggered model stream idle ceiling, while duplicate reports already point to this issue as the canonical stream-stall fallback tracker.

## What This Changes

The issue reports that cron agent turns with a payload-pinned model do not advance to configured model fallbacks when the provider opens a stream but never sends the first chunk.

## Best Possible Solution

Define and implement a maintainer-approved timeout hierarchy that lets stream-stall provider attempts fail and advance through model fallbacks before the cron wall-clock deadline expires, while preserving explicit slow-provider timeouts and caller-abort semantics.

## Reproduction Assessment

Yes, at source level: current cron and embedded-runner code show a payload `timeoutSeconds` can become both the cron wall-clock timeout and the cron-triggered stream idle ceiling. I did not run a live stalled-provider replay in this read-only review.

## Solution Assessment

Unclear until maintainers choose the timeout hierarchy. Treating no-first-chunk stalls as failover-worthy matches the documented fallback contract, but the implementation should avoid shortening all cron or provider timeouts unconditionally.
