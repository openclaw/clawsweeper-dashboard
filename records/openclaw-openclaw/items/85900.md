## Label Justifications

- `P1`: A scheduled agent workflow can miss a full run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported timeout drops the cron job's expected output or delivery before a fallback response is produced.
- `impact:auth-provider`: The affected surface is provider/model fallback routing and timeout classification for model calls.

## Summary

Keep open: current main and v2026.6.8 have partial stream-idle coverage, but a cron `timeoutSeconds` value can still be both the stream-idle ceiling and the outer cron wall-clock watchdog, so a no-first-chunk stall can consume the run budget before fallback gets a useful chance.

## What This Changes

The issue asks for cron payload-pinned model runs to treat opened-but-silent provider streams as failover-worthy failures so configured fallback models are attempted before the cron job times out.

## Best Possible Solution

Define and implement a provider/attempt stream-stall budget that fails the current candidate before the cron wall-clock deadline while preserving documented cron fallback semantics.

## Reproduction Assessment

Yes at source level, but not with a live stalled-provider replay. A cron job with `payload.model`, configured fallbacks, and an opened-but-silent provider stream can use the same explicit timeout for the stream watchdog and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout hierarchy. The desired behavior matches the documented fallback contract, but the implementation should avoid shortening all cron or provider timeouts unconditionally.
