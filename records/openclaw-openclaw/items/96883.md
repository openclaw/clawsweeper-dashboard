## Summary

Keep open: the patch appears technically sound and well-proven, but it has a protected maintainer label and changes Gateway auth, managed MCP startup, and cron authorization-boundary behavior that need maintainer acceptance before merge.

## What This Changes

The branch adds a local agent-runtime identity token path, injects a managed MCP session-key environment value for the ACPX openclaw-tools bridge, and scopes Gateway cron CRUD/history calls to the calling agent while preserving unscoped operator behavior.

## Best Possible Solution

Land the scoped mitigation only after maintainers accept the Gateway auth and managed MCP env contracts, while keeping the broader scheduling-authority design in https://github.com/openclaw/openclaw/issues/71712.

## Reproduction Assessment

Yes, at source level. Current main exposes the openclaw-tools MCP cron path and Gateway cron CRUD without a verified agent runtime identity, while PR head threads and enforces caller identity through local Gateway context.

## Solution Assessment

Mostly yes. The agent tool, Gateway client, and Gateway server are the right enforcement layers for this scoped mitigation, but maintainers still need to accept the new auth/env contract and decide how it relates to the broader scheduling-authority RFC.

## Risks / Open Questions

- The new optional Gateway connect auth field intentionally fails closed when an older Gateway rejects it, so mixed-version local backend cron calls need maintainer-accepted compatibility behavior.
- OPENCLAW_TOOLS_MCP_AGENT_SESSION_KEY becomes required for the openclaw-tools MCP server path, so direct or unmanaged invocations that used to start without session context need an explicit support decision or documentation.
- The PR changes a model-mediated cron authorization boundary; a mistake could expose another agent's jobs, hide same-agent jobs, or deny valid operations.
- The broader scheduling-authority RFC remains open for playbooks, quotas, expiry policy, provenance receipts, and execution lineage, so this should be treated as a scoped mitigation rather than the final scheduling design.
