## Summary

## What This Changes

The PR adds caller-scoped cron RPC parameters, threads the calling agent identity from agent-tool and ACPX MCP sessions, enforces scoped cron CRUD/run/history in Gateway, updates generated Swift models, and expands focused tests.

## Best Possible Solution

Land this as a scoped mitigation only after maintainers explicitly accept the callerScope protocol and managed MCP env contract, while keeping the broader scheduling-authority design in https://github.com/openclaw/openclaw/issues/71712.

## Reproduction Assessment

Yes, at source level. Current main exposes the openclaw-tools MCP cron tool without an agent session key, while PR head threads callerScope through the agent tool and Gateway path and requires a managed session key for that MCP server.

## Solution Assessment

Mostly yes. The agent tool and Gateway are the right layers for deriving and enforcing cron caller scope while preserving unscoped operator RPC behavior, but maintainers must accept the public callerScope/env contract or choose an internal context alternative.
