## Summary

Keep this PR open. The cron scoping approach is useful and mostly in the right owner boundary, but PR head is not merge-ready because it leaves ACPX per-session MCP delegates unbounded and changes the Gateway protocol without committing generated protocol outputs; compatibility, authorization, and proof questions still need maintainer handling.

## What This Changes

The branch adds `callerScope` to Gateway cron RPCs, enforces same-agent cron access for agent-tool calls, and passes ACPX OpenClaw-tools MCP session identity through a managed environment variable.

## Best Possible Solution

Keep the server-side cron scoping design, but bound or clean up the ACPX per-session delegate lifecycle, commit regenerated protocol outputs, and have maintainers explicitly accept the protocol/env compatibility contract before merge.

## Reproduction Assessment

Yes, at source level. Current main exposes the OpenClaw-tools MCP cron tool without caller identity, while PR head carries `callerScope` through the tool and Gateway; the remaining defects are also visible from PR-head source and CI logs.

## Solution Assessment

No, not yet. The tool and Gateway are the right layers for scoped cron enforcement, but the ACPX bridge needs lifecycle ownership for scoped delegates and the protocol schema change needs generated client follow-through.
