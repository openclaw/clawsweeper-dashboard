## Summary

Keep open. Current main still has a source-visible path where a token-only WebChat URL without an explicit session resets the selected session to `main`, while Gateway restart recovery only resumes sessions that match the stale-lock/running-session contract; the focused linked fix PR was closed unmerged because the reconnect signal needs maintainer direction.

## What This Changes

The issue asks OpenClaw to restore the existing WebChat session after a Gateway crash or restart instead of showing a new empty session while old transcripts remain on disk.

## Best Possible Solution

Define a reconnect/session-known contract that preserves the selected WebChat session after Gateway restart without reviving stale sessions on fresh token-only launches, then cover URL hydration and restart recovery with focused regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main path: token-only URL hydration resets persisted WebChat session selection to `main`, while restart recovery only resumes marked aborted running sessions.

## Solution Assessment

Unclear until maintainers choose the reconnect signal. The prior token-equality approach was closed because it could weaken the token-only dashboard launch contract; explicit `session=` intent or a distinct reconnect marker looks safer.
