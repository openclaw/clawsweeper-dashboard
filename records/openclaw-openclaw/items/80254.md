## Summary

Current main should stay open: WebChat has a documented session-continuity contract and code that preserves a history-backed session id, but the reconnect/default-selection paths can still reset the selected session to `main` and the restart-recovery path only resumes sessions marked as running and `abortedLastRun`. The related open PRs are adjacent, not a merged canonical fix for this crash/restart selection loss.

## What This Changes

The issue reports that after a Gateway crash/restart, WebChat selects or creates a new session instead of restoring the existing session transcript on disk.

## Best Possible Solution

Preserve and resolve the last selected WebChat session on Gateway reconnect when its session store row or transcript still exists, while keeping restart-aborted run recovery within the existing session-store and stale-lock contracts.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main path: WebChat has a documented stored-conversation contract, but reconnect/auth/default normalization can reset selection to `main` while restart recovery only resumes marked running sessions.

## Solution Assessment

Yes, a focused fix in the WebChat session-selection/reconnect path is the narrowest maintainable direction. Gateway-side recovery changes should be limited to existing session-store metadata and should not add a new migration or global startup heuristic.
