## Summary

This should stay open because current main has restart-recovery machinery and persisted session selection, but source inspection shows plausible paths where a WebChat reconnect can fall back to the main/default session instead of restoring the existing session whose transcript remains on disk. I did not find a merged canonical fix or duplicate that fully owns this exact Gateway-crash/WebChat restore symptom.

## What This Changes

The issue asks for WebChat/Gateway restart recovery to reconnect to the existing session and preserve conversation history instead of showing a newly created empty session after a crash.

## Best Possible Solution

Restore the previously selected WebChat session after Gateway reconnect when its session store row or transcript still exists, and cover that path with focused UI/restart-recovery regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible reproduction path around Gateway reconnect plus persisted session selection resetting to main/default while restart recovery only resumes sessions that satisfy the stale-lock/running-session contract.

## Solution Assessment

The best fix appears to be preserving and resolving the last selected WebChat session on reconnect if the existing session is still present, rather than selecting or creating a fresh dashboard session. Any gateway-side repair should stay within the existing session-store and restart-recovery contracts.
