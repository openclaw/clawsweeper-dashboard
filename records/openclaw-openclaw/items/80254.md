## Summary

Keep this issue open. Current main and the latest release still contain the token-only URL path that can reset a persisted WebChat session to `main`, and the linked fix attempt at https://github.com/openclaw/openclaw/pull/80359 was closed unmerged after the author agreed that token equality is not a safe reconnect signal.

## What This Changes

Restore the previously selected WebChat session after a Gateway crash or restart instead of landing in a fresh `main` session while the old transcript remains on disk.

## Best Possible Solution

Define a safe reconnect/session-known signal, such as an explicit `session=` on reconnect URLs or a distinct reconnect marker, then preserve the selected WebChat session only for that path while keeping ordinary token-only dashboard opens on `main`.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a focused path: current URL hydration resets token-only, no-session links to `main`, and the existing test asserts that behavior, but the full Windows WebChat crash/restart flow was not replayed.

## Solution Assessment

No, the closed PR's token-equality approach is not the best fix because ordinary dashboard launches can reuse the same token. The safer solution is to make reconnect intent explicit before changing session preservation behavior.
