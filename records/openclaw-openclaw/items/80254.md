## Summary

Keep open. Current main still has the Control UI token-only URL path that can reset a persisted WebChat session to `main`, and the attempted fix in https://github.com/openclaw/openclaw/pull/80359 was closed unmerged after the reconnect signal was judged too broad.

## What This Changes

The issue asks OpenClaw to restore the previously selected WebChat session after a Gateway crash/restart instead of landing in a new/default session while the old transcript remains on disk.

## Best Possible Solution

Define a narrow reconnect/session-intent contract, preferably explicit `session=` on reconnect URLs or a separate reconnect marker, then add focused Control UI and restart-recovery regression coverage while preserving fresh token-only dashboard launches to `main`.

## Reproduction Assessment

No for the full Windows WebChat Gateway crash/restart flow; yes at source level for the likely selection-loss path. Seeding a persisted non-main session and loading a token-only same-gateway URL still drives `applySettingsFromUrl()` to reset the session to `main`.

## Solution Assessment

No current patch is available, and the previous token-equality fix is not the best solution because ordinary dashboard launches can reuse the same token. A safer fix needs explicit session intent or a distinct reconnect signal before code should change.
