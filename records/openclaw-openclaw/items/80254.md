## What This Changes

The issue reports that after a Gateway crash/restart, WebChat reconnects into a new `main` session instead of restoring the previously selected session whose transcript still exists on disk.

## Best Possible Solution

Preserve the previously selected WebChat session on real reconnect/restart while keeping token-only fresh-launch behavior well-defined and covered by focused regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection and the existing focused test give a credible current-main reproduction path for the token-only reconnect resetting the selected session to `main`, but not for the full Windows crash/restart sequence.

## Solution Assessment

Unclear until the linked PR is reviewed against the prior token-only launch contract. The maintainable solution should distinguish reconnect restoration from fresh tokenized opens rather than dropping the guardrail wholesale.
