## Summary

This should stay open: current main still resets token-only same-gateway Control UI URLs to `main`, the focused fix PR was closed unmerged after its reconnect signal was judged unsafe, and no merged canonical fix now owns the Windows WebChat crash/restore symptom.

## What This Changes

The issue asks Gateway/WebChat to restore the previously selected session after an unclean Gateway crash instead of opening a new main session while transcript files remain on disk.

## Best Possible Solution

Define a clear reconnect/session-known contract, such as explicit `session=` on reconnect URLs or a dedicated reconnect marker, then preserve the existing selected WebChat session when the session store or transcript still exists and cover it with focused UI and restart-recovery tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does reproduce the likely URL-hydration failure path: a persisted non-main WebChat session plus a token-only same-gateway URL currently resets selection to `main`.

## Solution Assessment

Unclear for implementation until maintainers choose the reconnect contract. The closed token-equality PR was not the best fix because ordinary dashboard launches can reuse the same token, so an explicit session intent or distinct reconnect marker is safer.
