## Summary

Keep open. Current main has partial WebChat session-continuity and restart-recovery machinery, but it does not provide a high-confidence fix for this Gateway-crash case: the UI can still reset a token-only reconnect to `main`, and backend restart recovery only resumes sessions that were explicitly marked as running and `abortedLastRun`.

## What This Changes

The issue asks OpenClaw to restore the existing WebChat session and conversation history after a Gateway crash/restart instead of creating or selecting a fresh session.

## Best Possible Solution

Restore the previously selected WebChat session after Gateway reconnect when its persisted selection, session store row, or transcript still exists, and cover that path with focused UI/restart-recovery regression tests while leaving broad graceful-restart manifest policy under the existing umbrella.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main path: token-only reconnect can reset selection to `main`, while restart recovery ignores intact transcripts unless their store row is running and marked `abortedLastRun`.

## Solution Assessment

No, current main is not a complete fix for the reported crash restore path. The narrow maintainable fix is to preserve or re-resolve the last selected WebChat session before `chat.history` reloads, without turning this into the broader graceful-restart manifest project.
