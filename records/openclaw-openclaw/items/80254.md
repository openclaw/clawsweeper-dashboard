## Summary

Keep open: current main still contains the WebChat token-only URL path that resets a persisted selected session to `main`, and the focused repair PR for this issue was closed unmerged after review found its token-equality reconnect signal unsafe for normal dashboard launches.

## What This Changes

The issue reports that after a Gateway crash/restart, WebChat opens a new session instead of restoring the previously active session whose transcript still exists.

## Best Possible Solution

Define a safe reconnect/session contract, such as explicit `session=` on reconnect URLs or a separate reconnect marker, that restores the selected WebChat session after Gateway restart without weakening fresh token-only dashboard launches.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible path: token-only WebChat URL hydration resets a persisted non-main session to `main`, while restart recovery only resumes sessions that are still marked running and aborted.

## Solution Assessment

No active patch is currently the best solution. The closed PR’s token-equality approach was too broad; the safer path is a maintainer-approved explicit session or reconnect signal with focused regression coverage.
