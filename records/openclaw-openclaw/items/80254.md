## Summary

Keep this open: current main still has a source-visible recovery gap around crash-residue session locks, and the linked fixing PR is open, unmerged, and currently conflicting rather than landed.

## What This Changes

The issue asks OpenClaw to restore the existing WebChat/Gateway session after an unclean Gateway crash instead of showing a fresh empty session while old transcript files remain on disk.

## Best Possible Solution

Land or replace the linked Gateway startup/session-lock recovery fix so crash-residue locks can safely mark and recover the interrupted main session, then verify the WebChat reconnect path keeps the recovered session selected.

## Reproduction Assessment

Yes at source level: current main only marks lock-interrupted sessions from cleaned locks, while young orphan crash locks can be preserved and therefore never reach the marker. I did not run a live Windows WebChat crash/restart reproduction.

## Solution Assessment

Yes, the best current direction is to repair the Gateway startup/session-lock recovery boundary rather than add a new user option or UI workaround. The linked PR is the active implementation path, but it needs rebase/review and WebChat validation before this issue can close.
