## Summary

This should stay open: current main still resets token-only WebChat URLs without an explicit session back to `main`, and the focused follow-up PR was closed unmerged after the reconnect contract concern was acknowledged.

## What This Changes

The issue reports that WebChat creates or selects a new session after a Gateway crash instead of restoring the previously active session whose transcript remains on disk.

## Best Possible Solution

Define a narrow reconnect/session-known signal, such as an explicit `session=` URL or distinct reconnect marker, then update Control UI hydration and tests so crash/restart reconnects preserve the selected WebChat session without weakening fresh token-only launches to `main`.

## Reproduction Assessment

Partially. Source inspection and existing tests reproduce the key reset path for token-only WebChat URLs with a persisted session, but I did not run the full Windows 11 Gateway crash/restart flow.

## Solution Assessment

No. The closed follow-up PR’s token-equality approach is not the best fix because ordinary dashboard launches can reuse the same token; a safer solution needs explicit session intent or a separate reconnect marker.
