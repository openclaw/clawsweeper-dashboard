## Summary

Keep open. Current main still has the token-only WebChat URL path that resets persisted session selection to `main`, and the only focused fix PR for this issue was closed unmerged after its reconnect signal was judged too broad.

## What This Changes

The issue reports that after a Gateway crash/restart, WebChat opens a new session instead of restoring the previously selected session whose transcript still exists on disk.

## Best Possible Solution

Define a safe reconnect/session-known contract that preserves the selected WebChat session after Gateway restart without weakening the existing token-only dashboard launch-to-main behavior, then cover it with focused Control UI tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show a credible current-main path: persisted session selection is loaded, token-only URL hydration runs before gateway connection, and the current test asserts that this path resets the session to `main`.

## Solution Assessment

No current proposed patch is the best solution. The closed PR’s token-equality approach was rejected as too broad, so maintainers should choose an explicit reconnect/session signal before implementation proceeds.
