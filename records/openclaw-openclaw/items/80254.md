Keep open. Current main still preserves the token-only reset-to-main behavior that can explain the reported WebChat restore failure, and the focused follow-up PR closed unmerged because its reconnect signal conflicted with the existing dashboard/session contract.

## What This Changes

The issue asks Gateway/WebChat restart flows to restore the previously selected session and conversation after an unclean Gateway crash instead of opening a fresh main session.

## Best Possible Solution

Define a narrow reconnect/session-known signal, then preserve the last selected WebChat session on reconnect while keeping fresh token-only dashboard launches on `main`.

## Reproduction Assessment

No. I did not run a live Windows 11 Gateway crash/restart flow; source inspection gives a credible path through token-only URL hydration resetting to `main` while restart recovery only resumes marked aborted running sessions.

## Solution Assessment

Unclear. The closed token-equality approach is not the best fix because it can weaken the existing token-only dashboard contract; an explicit `session=` reconnect URL or distinct reconnect marker would be safer.
