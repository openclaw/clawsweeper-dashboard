## What This Changes

The issue asks for WebChat/Gateway restart recovery to restore the previously selected session and history instead of selecting a newly created session.

## Best Possible Solution

Define an explicit reconnect/session intent, such as preserving or adding `session=` on reconnect URLs or a distinct reconnect marker, while keeping fresh token-only dashboard launches on `main`, then cover it with focused UI and Gateway regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible current-main path: a token-only same-gateway URL without `session` resets the selected WebChat session to `main`, matching the reported intact transcript but wrong UI selection after restart.

## Solution Assessment

No, the already-attempted token-equality fix is not the best final shape because normal dashboard launches can reuse the same token. A safer solution needs an explicit reconnect/session signal and regression coverage around both reconnect and fresh-launch behavior.
