## Summary

This should stay open. Current main still resets token-only WebChat URLs without an explicit session back to `main`, and the attempted fix was closed unmerged after review found the token-equality approach conflicted with ordinary dashboard launch behavior.

## What This Changes

Restore the selected WebChat session after a Gateway crash/restart instead of opening a new `main` session while old transcript files still exist.

## Best Possible Solution

Define a narrow reconnect/session-known signal, such as carrying explicit `session=` when WebChat should preserve selection, then restore that session on reconnect and add focused UI/restart-recovery regressions.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible path: persisted same-gateway session state can be loaded, then token-only URL hydration still resets the selected session to `main`.

## Solution Assessment

Unclear for implementation. The desired behavior is right, but the closed PR shows token equality alone is not the best fix because ordinary dashboard launches also use token-only URLs.
