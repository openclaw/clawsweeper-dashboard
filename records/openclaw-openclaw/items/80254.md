## Summary

Keep this open: current main still resets token-only WebChat URL hydration to `main`, the focused test still locks that behavior, and the only direct fix PR was closed unmerged because its token-equality reconnect signal would weaken the existing fresh dashboard launch contract.

## What This Changes

The issue asks for WebChat/Gateway restart recovery to reconnect to the existing on-disk session instead of opening a fresh main session after an unclean Gateway restart.

## Best Possible Solution

Define an explicit reconnect/session-intent contract, such as carrying `session=` when reconnect should preserve a selected WebChat session, then update URL hydration and focused tests without weakening fresh token-only dashboard opens to `main`.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible path: token-only URL hydration still resets a persisted selected session to `main`, and the current test suite explicitly asserts that behavior.

## Solution Assessment

No for the closed token-equality approach. The safer direction is an explicit session/reconnect signal that preserves intended crash-reconnect sessions while keeping ordinary token-only dashboard launches on `main`.
