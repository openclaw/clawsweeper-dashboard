## Summary

Keep this open: current main and the latest release still reset a token-only same-gateway WebChat reconnect to `main`, which matches the reported post-crash empty-session symptom. A narrow related fix exists in a closed unmerged PR, but it is not on main.

## What This Changes

The issue requests restoring the previously selected WebChat session after a Gateway crash/restart instead of opening a fresh empty session.

## Best Possible Solution

Land a narrow Control UI fix that preserves the selected session for matching-token same-gateway reconnects, keeps fresh token-only launches resetting to `main`, and adds focused regression coverage.

## Reproduction Assessment

Yes for the core reset path: current source and tests show a token-only URL with an existing selected session is reset to `main`. I did not run a full Windows Gateway crash replay.

## Solution Assessment

No, current main is not the best end state because it still resets matching-token reconnects. The narrow maintainable fix is to distinguish matching-token reconnects from fresh token-only launches and cover both cases in tests.
