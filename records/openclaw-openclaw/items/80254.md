## Summary

Keep this open: the report is fresh, external, unprotected, and current main still has credible source-level paths where WebChat reconnect/re-auth can fall back to the main/default session while restart recovery only resumes sessions that were explicitly marked aborted from stale locks. I did not find a merged canonical fix; the related PRs are adjacent and still open.

## What This Changes

Gateway crash restart should restore the existing WebChat session and history instead of selecting or creating a new empty session.

## Best Possible Solution

Restore the previously selected WebChat session after Gateway reconnect when its session-store row or transcript still exists, and keep restart resume within the existing stale-lock/aborted-run contract.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible path through token/session selection resetting to main plus restart recovery skipping unmarked running sessions, but a focused regression or Windows/WebChat repro should pin the exact branch.

## Solution Assessment

Yes, the best fix appears to be a narrow restore/preserve path for the last selected WebChat session on reconnect when the session still exists. Avoid broad recovery semantics that resume arbitrary unmarked sessions or weaken the intentional token reset behavior.
