## Summary

This should stay open: current main still resets token-only Control UI/WebChat URLs without an explicit session back to `main`, and the linked fix attempt was closed unmerged after identifying that token equality is not a safe reconnect signal. No merged canonical fix currently owns this exact Gateway-crash/WebChat session-restore symptom.

## What This Changes

The issue asks WebChat to restore the previously selected session and conversation history after a Gateway crash/restart instead of opening a fresh session.

## Best Possible Solution

Choose a durable reconnect/session-known contract, then preserve the last selected WebChat session only for genuine reconnects while keeping ordinary token-only dashboard launches on `main`.

## Reproduction Assessment

No, not for the full reported Windows Gateway crash/restart flow. Source inspection does show a credible current-main path where token-only URL hydration resets persisted session selection to `main` before gateway connection.

## Solution Assessment

No current proposed patch is ready to merge. The prior token-equality approach was rejected as too broad, so maintainers should first choose the reconnect signal and then implement the narrow UI/session fix.
