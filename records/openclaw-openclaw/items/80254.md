## Summary

Keep this open: current main still has a credible WebChat reconnect path that can reset persisted session selection to `main`, and the focused linked fix was closed unmerged because its token-equality signal could regress normal dashboard launches.

## What This Changes

The issue asks WebChat/Gateway to restore the previously selected session after an unclean Gateway restart instead of opening a new empty session.

## Best Possible Solution

Define a durable reconnect intent, such as explicit `session=` on reconnect/open URLs or a distinct non-token reconnect marker, then update Control UI hydration and regression tests without weakening fresh dashboard launch behavior.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a credible path: stored session selection and token are loaded before URL hydration, then a same-gateway token-only URL resets the selection to `main`.

## Solution Assessment

Unclear. The closed token-equality fix is not the best solution because ordinary dashboard launches can use the same token; maintainers should choose an explicit reconnect/session-intent contract first.
