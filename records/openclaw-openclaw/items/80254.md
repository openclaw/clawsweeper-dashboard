## Summary

Keep this open: current main still resets token-only WebChat URLs to `main`, and the direct fix attempt was closed unmerged after the review found its token-equality reconnect signal could regress ordinary dashboard launches.

## What This Changes

The issue reports that after a Gateway crash/restart, WebChat opens a new empty session instead of reconnecting to the previously selected session whose transcript remains on disk.

## Best Possible Solution

Define an explicit reconnect/session-known signal, such as carrying `session=` on reconnect URLs or adding a distinct reconnect marker, then preserve the previous WebChat session only on that signal while keeping fresh token-only dashboard launches on `main`.

## Reproduction Assessment

No live crash/restart reproduction was run. Source inspection and the current focused unit test show a high-confidence selection-loss path where token-only WebChat URL hydration resets an existing saved session to `main`.

## Solution Assessment

No, the closed token-equality fix is not the best solution because it conflicts with the established token-only dashboard launch contract. A maintainer-chosen reconnect signal or explicit `session=` URL is the safer path.
