## Summary

Keep open: current main still has the token-only WebChat URL path that resets the selected session to `main`, and the linked attempted fix was closed unmerged after the reconnect signal was judged unsafe for ordinary dashboard launches. The related open PRs are broad or about queued messages, so they do not supersede this crash-recovery report.

## What This Changes

The issue requests restoring the previously active WebChat session after a Gateway crash/restart instead of opening a fresh main session while the old transcript remains on disk.

## Best Possible Solution

Define a narrow reconnect/session-known signal, such as an explicit `session=` on reconnect URLs or an equivalent non-token marker, then preserve the selected WebChat session only for that path while keeping fresh token-only dashboard launches on `main`.

## Reproduction Assessment

No high-confidence live reproduction was run for the full Windows Gateway crash/restart flow. Source inspection does show current main still resets token-only WebChat URLs without `session` to `main`, and the current test suite locks in that behavior.

## Solution Assessment

Unclear until maintainers choose the reconnect/session intent contract. The closed linked PR was not the best final solution because token equality alone could preserve stale non-main sessions for ordinary dashboard launches.
