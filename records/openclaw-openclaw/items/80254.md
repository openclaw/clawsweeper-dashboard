## Summary

Keep open: current main still contains the token-only WebChat URL path that can reset session selection to `main`, the direct fix PR was closed unmerged after its reconnect signal was judged unsafe, and the related open PRs are adjacent rather than canonical fixes for this crash/restart restore symptom.

## What This Changes

The issue asks WebChat/Gateway restart to restore the previously selected session and conversation history after an unclean Gateway crash instead of opening a fresh session.

## Best Possible Solution

Define a safe reconnect/session-known contract, likely explicit `session=` on reconnect URLs or a distinct reconnect marker, then preserve the selected WebChat session only when that contract is present and covered by regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run for the full Windows 11 crash/restart path. Source inspection gives a credible path: current main still resets token-only WebChat auth URLs without `session` to `main`, and the existing test asserts that reset.

## Solution Assessment

Unclear. Preserving the previous session is the right product outcome, but the attempted token-equality fix was closed unmerged because it could weaken the existing dashboard token-only launch contract; maintainers should choose an explicit session or reconnect signal first.
