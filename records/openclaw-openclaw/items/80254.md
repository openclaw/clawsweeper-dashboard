## Summary

Keep this open: current main still has a source-visible path that resets token-only WebChat URLs without an explicit session back to `main`, and the focused follow-up at https://github.com/openclaw/openclaw/pull/80359 was closed unmerged after the contributor agreed token equality was not a safe reconnect signal. I did not find a merged duplicate or canonical open item that fully owns this exact Gateway-crash/WebChat session-restore symptom.

## What This Changes

The issue asks for WebChat to restore the previously selected session after an unclean Gateway restart instead of opening a new main/default session while the old transcript remains on disk.

## Best Possible Solution

Define an explicit reconnect/session intent, preferably by carrying `session=` on reconnect or adding a distinct non-token reconnect marker, then preserve the selected WebChat session and cover it with focused settings and Gateway restart-recovery regression tests.

## Reproduction Assessment

No high-confidence live reproduction was run for the full Windows 11 Gateway crash flow. Source inspection does show the token-only URL path resetting a persisted WebChat session to `main`, and the existing unit test currently locks in that reset behavior.

## Solution Assessment

Unclear as an implementation path: the closed token-equality fix is not the best solution because ordinary dashboard launches can carry the same token. A maintainer should choose an explicit reconnect/session signal before another focused fix PR is written.
