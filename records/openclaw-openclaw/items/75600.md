## Summary

Keep open. The latest PR head is active and current `main` does not already implement the bounded finalize-retry contract, but the patch still has a blocking retry-normalization bug: retry metadata replaces already-merged revise instructions before the harness receives them.

## What This Changes

The PR adds bounded `before_agent_finalize` retry metadata, run-context cleanup hardening, bounded plugin cleanup timeouts, Plugin SDK docs/API-baseline/changelog updates, and focused lifecycle contract tests.

## Best Possible Solution

Keep this as a focused generic Plugin SDK hardening slice, but compose the merged revise `reason` with the bounded retry instruction during normalization, add a regression test for mixed legacy revise plus retry hooks, then proceed through final SDK contract review and exact-head checks.

## Reproduction Assessment

Yes. A focused unit test with two `before_agent_finalize` hooks, one returning a legacy `reason` and another returning `retry.instruction`, reproduces the bug because PR-head normalization returns only the retry instruction.

## Solution Assessment

No. The generic host-hook boundary is the right place for the feature, but the current normalization is not the best fix because it replaces other merged instructions instead of preserving them while applying the retry budget.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve merged revise reasons for retry decisions:** `src/agents/harness/lifecycle-hook-helpers.ts:164`
  - body: When any hook supplies `retry`, `normalizeBeforeAgentFinalizeResult` returns only `retryInstruction`. `mergeBeforeAgentFinalize` may already have combined `result.reason` from other revise hooks, so a legacy revise hook plus a bounded retry hook will drop the legacy instruction before the harness gets it. Compose the merged reason with the retry instruction while applying the budget.
