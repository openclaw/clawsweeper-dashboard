## Summary

Keep open. Current main still lacks this Plugin SDK retry/run-context hardening, and the PR has one concrete P2 correctness issue in multi-plugin retry merging that should be fixed before merge.

## What This Changes

The PR adds bounded `before_agent_finalize` retry metadata/budgeting, terminal run-context cleanup guards, bounded plugin cleanup timeouts, stale run-context API guards, docs, changelog, API-baseline, and focused lifecycle tests.

## Best Possible Solution

Keep the focused SDK hardening direction, but adjust retry merging so independent plugin retry requests are not collapsed behind an exhausted first retry key; then re-run the focused hook/runtime contract tests and normal changed-surface gates.

## Reproduction Assessment

Yes. A focused unit test can register two `before_agent_finalize` hooks in one run where both return valid retry metadata, consume the first hook's `maxAttempts`, then verify the second hook's still-valid retry is honored instead of the merged result becoming `continue`.

## Solution Assessment

No, not as submitted. The overall seam placement is maintainable, but the merge logic should preserve or evaluate later retry metadata instead of letting one spent retry budget suppress another plugin's revision request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Honor later retry requests when the first retry is spent:** `src/plugins/hooks.ts:341`
  - body: When multiple finalize hooks return `revise` with retry metadata, the merge keeps only `acc.retry` when it is valid. Once that first retry key exceeds `maxAttempts`, harness normalization returns `continue` for the whole merged decision, so a later plugin's valid retry request is ignored even though its reason was merged. Preserve/evaluate later retry metadata instead of letting an exhausted first plugin suppress it.
  - confidence: 0.84
