## Summary

Keep open. The latest PR head fixes the earlier malformed retry metadata merge issue and includes docs/changelog coverage, but it still has a blocking run-context cleanup regression: repeated terminal lifecycle events can clear the timeout-expired marker and let an older, already-timed-out terminal handler write run context after forced cleanup.

## What This Changes

The PR adds bounded `before_agent_finalize` retry metadata, per-run retry budgets, terminal run-context cleanup hardening, bounded plugin cleanup timeouts, SDK docs/API-baseline/changelog updates, and focused lifecycle contract tests.

## Best Possible Solution

Keep this as a focused Plugin SDK hardening slice, but preserve the terminal-timeout expired marker once a run has exceeded the grace period, add a fake-timer regression for repeated terminal events after timeout, then rerun the targeted hook/runtime contract tests and exact-head SDK/lint checks before merge.

## Reproduction Assessment

Yes. A focused fake-timer contract test can emit a terminal event, advance past `PLUGIN_TERMINAL_EVENT_CLEANUP_WAIT_MS`, emit a second terminal event for the same run, then release the first handler and assert that its `ctx.setRunContext` cannot recreate run context.

## Solution Assessment

No. The retry and cleanup direction is the right generic SDK seam, but the current repeated-terminal handling is not the narrowest safe implementation because it reopens a closed-run write gate after timeout; keeping the expired marker until runtime reset or a fresh run identity is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep expired terminal cleanup markers across repeats:** `src/plugins/host-hook-runtime.ts:62`
  - body: `markPluginRunClosed` deletes `terminalEventCleanupExpiredRunIds` every time a terminal lifecycle event is emitted. If the first terminal handler has already timed out, a repeated `lifecycle:end`/`error` clears the expired marker while that old handler's `ctx.setRunContext` still checks it dynamically, so the late handler can recreate per-run context after forced cleanup. Keep the expired marker for that run, or otherwise key the gate per terminal wait, and add a repeated-terminal regression test.
  - confidence: 0.89
