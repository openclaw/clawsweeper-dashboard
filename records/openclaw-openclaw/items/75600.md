## Summary

Keep open. Current main still lacks the bounded `before_agent_finalize` retry contract and run-context cleanup hardening, so the PR is useful, but the current PR head still has a blocking retry-merge bug: later plugin retry requests can be hidden behind an exhausted first retry key.

## What This Changes

The PR adds bounded Plugin SDK finalize-retry metadata and budgeting, terminal run-context cleanup guards, bounded plugin cleanup timeouts, docs/changelog/API-baseline updates, lifecycle tests, and small Matrix/onboarding compile fixes.

## Best Possible Solution

Keep the focused Plugin SDK hardening direction, but change retry merging so independent plugin retry candidates are preserved or evaluated until one with remaining budget is found; then rebase onto current main and rerun the focused hook/runtime contract checks.

## Reproduction Assessment

Yes. Register two `before_agent_finalize` hooks in one run, have both return valid retry metadata, consume the first hook's `maxAttempts`, then verify the second hook's still-valid retry is honored instead of the merged decision becoming `continue`.

## Solution Assessment

No, not as submitted. The seam placement is maintainable, but selecting only `acc.retry` before budget evaluation lets one exhausted plugin suppress a later plugin's revision request; preserving retry candidates or evaluating budget during merge is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Honor later retry requests after a spent retry:** `src/plugins/hooks.ts:341`
  - body: When multiple finalize hooks return `revise` with retry metadata, this merge keeps `acc.retry` and drops `next.retry`. After that first retry key exceeds `maxAttempts`, harness normalization returns `continue` for the whole merged result, so a later plugin's still-valid retry is never evaluated. Preserve retry candidates or choose the first candidate that still has budget.
  - confidence: 0.86
