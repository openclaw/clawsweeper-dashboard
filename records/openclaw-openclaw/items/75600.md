## Summary

Keep open. Current main does not implement the bounded finalize retry contract or the new run-context cleanup hardening, and the current PR head still has two concrete P2 correctness blockers in retry merging and sticky registry retirement.

## What This Changes

The PR adds bounded `before_agent_finalize` retry metadata plus Plugin SDK run-context and cleanup lifecycle hardening, with docs, changelog, API baseline, and focused tests.

## Best Possible Solution

Keep the focused Plugin SDK hardening direction, but fix retry metadata selection so independent plugin retries are not collapsed, and make registry retirement reversible or activation-token based before rerunning the focused hook/runtime contract tests and normal changed-surface gates.

## Reproduction Assessment

Yes. The retry bug can be reproduced with two `before_agent_finalize` hooks where the first retry key has exhausted `maxAttempts` and the second has a valid retry; the registry bug can be reproduced by activating registry A, switching to registry B, restoring registry A, then calling a captured API path that uses `setRunContext`.

## Solution Assessment

No, not as submitted. The seam placement is maintainable, but the retry merge must preserve/evaluate later retry metadata and the stale-registry guard must not permanently disable a registry after it becomes active again.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor later retry requests after one retry key is spent:** `src/plugins/hooks.ts:341`
  - body: When multiple finalize hooks return `revise` with retry metadata, this merge keeps only the first normalized retry. If that first retry key has already exceeded `maxAttempts`, harness normalization returns `continue` for the whole merged decision and drops a later plugin's still-valid retry instruction. Merge or evaluate retry metadata so one exhausted key cannot suppress an independent later revise request.
  - confidence: 0.84
- **[P2] Unretire registries when they become active again:** `src/plugins/runtime.ts:133-134`
  - body: `setActivePluginRegistry` now permanently adds the previous registry to `retiredRegistries`, but never removes a registry when it is set active again. Flows that restore a previous registry in the same process will still fail `isPluginRegistryRetired(registry)`, so `setRunContext` and `clearRunContext` stay disabled even though that registry is active.
  - confidence: 0.86

## Repair Work Prompt

Fix the two correctness blockers in openclaw/openclaw PR https://github.com/openclaw/openclaw/pull/75600. Preserve the focused Plugin SDK hardening scope. First, update `before_agent_finalize` retry merging/normalization so a spent retry key from one plugin cannot suppress a later plugin's valid retry request; add a targeted multi-plugin test that consumes the first retry budget and verifies the second retry is still honored. Second, make registry retirement reversible or activation-token based so a registry restored via `setActivePluginRegistry` is allowed to commit run-context side effects again while stale captured APIs from truly replaced/unloaded registries remain blocked; add focused coverage for restoring a previous registry. Keep the existing changelog entry unless the public surface changes beyond this PR's current wording. Do not add unrelated workflow, dependency, package, generated, or product-scope changes.
