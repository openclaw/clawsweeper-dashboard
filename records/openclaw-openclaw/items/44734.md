## Summary

Keep open: current main and the latest release still advertise `.ogg`/`.opus` multimodal audio indexing and forward raw inline audio data to Gemini without provider-specific codec validation or transcoding. The issue discussion’s main-confirmation matches the current source path, but the exact fix needs maintainer judgment because Google’s public audio docs distinguish OGG Vorbis from Opus.

## What This Changes

Handle OGG/Opus audio in Gemini-backed multimodal memory indexing instead of letting unsupported raw audio fail during indexing.

## Best Possible Solution

Add provider-aware audio compatibility for Gemini multimodal memory: detect unsupported Opus/containers, either transcode to a Gemini-supported format or skip with a clear warning, and align docs plus focused regression tests.

## Reproduction Assessment

Yes, at source level: current main accepts `.ogg`/`.opus`, builds inline-data embedding parts from raw bytes, and sends them to Gemini unchanged. A live API repro was not run because no Gemini key was available, but the issue discussion already reports a main confirmation.

## Solution Assessment

Unclear. Provider-aware validation is necessary, but maintainers should decide whether advertised OGG/Opus support means transcoding to a supported codec or narrowing the supported-format contract and skipping unsupported audio with a warning.
