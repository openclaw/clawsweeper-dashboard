## Summary

Keep open. Current main still advertises and accepts `.ogg`/`.opus` audio for multimodal memory indexing, but the Gemini embedding path forwards the detected MIME and bytes without a provider-specific allowlist, transcode step, or graceful unsupported-format handling.

## What This Changes

The issue reports that OGG/Opus audio files configured through `memorySearch.extraPaths` are accepted for multimodal memory indexing but rejected by Gemini embedding as unsupported audio input.

## Best Possible Solution

Define a provider-aware multimodal memory audio contract: either support OGG/Opus through an explicit, tested transcode path or stop advertising/indexing unsupported raw formats and warn users with clear docs.

## Reproduction Assessment

Yes, at source level. Current main accepts `.ogg`/`.opus`, preserves the detected MIME, and sends it to Gemini as inline data with no provider-specific validation; I did not make a live Gemini API call.

## Solution Assessment

Unclear. Auto-transcoding may be the most user-friendly fix, but it adds ffmpeg/dependency and provider-policy surface; provider-aware validation plus docs may be the narrower first step if maintainers do not want transcoding in memory indexing.
