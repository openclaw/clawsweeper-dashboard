## Summary

Keep open. Current main still routes `.ogg` and `.opus` multimodal memory files into Gemini `inlineData` without provider format validation or transcoding, while the docs still advertise those audio extensions as supported.

## What This Changes

The issue requests making OGG/Opus audio files in `memorySearch.extraPaths` index successfully with Gemini multimodal memory, or otherwise narrowing the documented support contract.

## Best Possible Solution

Define a provider-aware multimodal memory audio contract: either explicitly transcode OGG/Opus into a Gemini-supported format, or stop advertising/indexing unsupported raw formats and warn users with matching docs.

## Reproduction Assessment

Yes, at source level. Current main classifies `.ogg` and `.opus` as multimodal audio, preserves the detected MIME in the embedding input, and forwards it unchanged to Gemini; I did not establish a live 400 in this environment.

## Solution Assessment

Unclear. Auto-transcoding would satisfy the user expectation but adds dependency and provider-policy surface; provider-aware validation plus docs is narrower but changes the advertised support promise.
