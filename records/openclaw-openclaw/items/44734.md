## Summary

Keep open. Current main and the latest release still advertise and accept OGG/Opus multimodal memory audio, preserve the detected MIME type, and pass it unchanged to the Gemini embedding request; the remaining fix needs a maintainer choice between narrowing/skipping unsupported audio and owning transcoding.

## What This Changes

The issue asks OpenClaw to handle OGG/Opus audio in Gemini multimodal memory indexing instead of letting the Gemini embedding request fail.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: skip or reject unsupported audio with clear diagnostics and aligned docs, or add an explicitly approved and tested transcode path if maintainers want OGG/Opus to remain supported.

## Reproduction Assessment

Yes, at source level: the scanner accepts `.ogg` and `.opus`, stores the detected audio MIME type, and passes it unchanged to Gemini inlineData. A live Gemini credentialed run was not performed in this read-only review.

## Solution Assessment

Unclear. The source bug is clear, but the best fix still requires choosing between provider-aware filtering/diagnostics and an explicitly owned transcoding path.
