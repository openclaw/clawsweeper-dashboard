## Summary

Keep open. Current main still advertises OGG/Opus as supported multimodal memory audio, builds raw inline-data embedding inputs from those files, and the Gemini embedding adapter forwards the MIME/data without provider-specific format validation or transcoding; the remaining decision is how maintainers want to handle unsupported Gemini audio formats.

## What This Changes

The issue reports that OGG/Opus audio files are treated as indexable for memory multimodal indexing but fail when sent to Gemini embeddings.

## Best Possible Solution

Define a provider-aware multimodal memory contract: keep generic file discovery separate from Gemini’s supported MIME/codec set, then either transcode OGG/Opus through an approved optional media path or skip/document unsupported formats with clear diagnostics.

## Reproduction Assessment

Yes, at source level but not through a live Gemini call. Current main advertises `.ogg`/`.opus`, classifies those files as audio, builds inline-data embedding inputs with detected MIME, and forwards them through Gemini unchanged; a commenter also confirmed the same path on main.

## Solution Assessment

Unclear. The narrow bug is real, but choosing between provider-aware skip/docs and ffmpeg-style transcoding needs maintainer product and dependency judgment before an automated repair should pick one.
