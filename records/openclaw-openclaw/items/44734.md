## Summary

Keep open. Current main and the latest release still advertise OGG/Opus as multimodal memory audio, preserve the detected raw MIME/data, forward that unchanged to Gemini embedding requests, and only suppress multimodal provider failures for size-limit errors; the related Gemini multimodal feature thread is broader and does not replace this narrower format-contract bug.

## What This Changes

The issue asks OpenClaw to stop failing Gemini multimodal memory indexing when `.ogg` or `.opus` audio files are included in `memorySearch.extraPaths`.

## Best Possible Solution

Define the provider-aware multimodal audio contract, then either transcode OGG/Opus into a Gemini-accepted representation with bounded dependency behavior or stop advertising raw Gemini support for those formats and skip them with clear warnings/docs.

## Reproduction Assessment

Yes, at source level. Enabling multimodal audio indexing for `.ogg` or `.opus` classifies the file, preserves its detected audio MIME, sends it to Gemini as raw inline data, and rethrows non-size provider errors; I did not run a live Gemini request.

## Solution Assessment

Unclear. Auto-transcoding is the most complete user-visible fix, but maintainers need to decide whether OpenClaw should add transcode dependency behavior or instead narrow the provider-specific supported-format contract.
