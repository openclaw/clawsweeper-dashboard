## Summary

Keep open. Current main and latest release still admit and document OGG/Opus audio for Gemini multimodal memory, then forward the detected MIME type unchanged to Gemini embeddings; the official embedding-specific contract lists MP3/WAV audio support, so the reported provider-format mismatch remains. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/embeddings?utm_source=openai))

## What This Changes

The issue asks OpenClaw to stop failing Gemini multimodal memory indexing for OGG/Opus audio files, either by handling the format or by correcting the advertised support.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: either narrow/skip unsupported audio with clear diagnostics or add an explicitly owned and tested transcode path, then align docs and regression coverage.

## Reproduction Assessment

Yes, at source level: current main accepts OGG/Opus as multimodal audio, stores the detected MIME type, and forwards it unchanged to Gemini inlineData. A live Gemini run with redacted credentials would still be useful before implementation.

## Solution Assessment

No, unconditional auto-transcoding is not clearly the best fix yet because it adds dependency and operator behavior. Maintainers should first choose whether Gemini should reject/skip unsupported audio with clear diagnostics or whether OpenClaw should own audio transcoding as a product feature.
