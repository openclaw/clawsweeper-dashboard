## Label Justifications

- `P2`: The issue breaks an opt-in memory-indexing workflow for documented audio formats but has a limited blast radius and a conversion workaround.
- `impact:session-state`: Affected audio content cannot enter the searchable memory index and therefore cannot be retrieved as intended.

## Summary

Current main still advertises OGG/Opus as supported multimodal-memory audio and forwards the raw MIME/data to Gemini without provider-format validation, so the reported contract mismatch remains unresolved. The broader https://github.com/openclaw/openclaw/issues/43117 tracks additional multimodal capabilities but does not replace this narrower format bug.

## What This Changes

Make Gemini multimodal memory indexing successfully process, or accurately reject and isolate, OGG/Opus audio files currently presented as supported.

## Best Possible Solution

Give embedding providers an explicit supported-MIME contract, validate each multimodal file before dispatch, and report or skip unsupported files without aborting the entire index; only add core-owned transcoding if maintainers explicitly accept its dependency and operational costs.

## Reproduction Assessment

Yes at source level: current main accepts and documents OGG/Opus, reads their raw bytes, and forwards their MIME types to Gemini without validation or conversion. A credentialed live Gemini request was not performed during this read-only review, so the exact remote error remains based on the issue evidence.

## Solution Assessment

The issue should remain open, but automatic transcoding is not yet proven to be the best fix. Provider-aware validation with isolated per-file failure is the narrowest maintainable default, while transparent transcoding requires an explicit product and dependency-ownership decision.
