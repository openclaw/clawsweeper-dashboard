## Label Justifications

- `P2`: This affects an opt-in Gemini multimodal memory indexing path with a conversion workaround rather than core runtime availability.
- `impact:session-state`: The failure can prevent selected audio files from being embedded into searchable memory state and may abort an indexing run.

## Summary

Keep open: current main and v2026.5.19 still advertise OGG/Opus as Gemini multimodal memory audio, preserve the detected MIME/raw bytes, and forward inline data to Gemini without provider-aware filtering or transcoding, while the current Gemini embedding docs list MP3/WAV as the supported audio formats. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/embeddings))

## What This Changes

Request to fix Gemini-backed multimodal memory indexing so OGG/Opus audio files under `memorySearch.extraPaths` do not make indexing fail.

## Best Possible Solution

Define a provider-aware multimodal audio contract: either transcode advertised-but-provider-unsupported audio to Gemini-supported MP3/WAV, or stop advertising those formats as Gemini-indexable and skip them with clear warnings and docs.

## Reproduction Assessment

Yes, at source level. Current main classifies `.ogg` and `.opus` as multimodal audio, preserves detected MIME/raw bytes, and forwards them unchanged to Gemini inline data; I did not run a live Gemini request.

## Solution Assessment

Unclear. The source defect is narrow, but maintainers need to choose whether OpenClaw should transcode unsupported audio or narrow the provider-specific supported-format contract with skip/warn behavior and docs.
