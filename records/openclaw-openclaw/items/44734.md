## Label Justifications

- `impact:session-state`: The bug affects which files become indexed memory context and therefore what later memory search can retrieve.

## Summary

Keep open. Current main and v2026.6.6 still classify OGG/Opus as supported Gemini multimodal audio, preserve the detected MIME type, and pass it unchanged to Gemini inlineData, while the official Gemini embedding docs list MP3/WAV as the supported audio formats. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/embeddings?utm_source=openai))

## What This Changes

Fix or document memory multimodal indexing so OGG/Opus audio under `memorySearch.extraPaths` does not fail when sent to Gemini embeddings.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: either reject or skip unsupported audio with clear diagnostics, or add an explicitly owned and tested transcode path, then align docs and regression coverage.

## Reproduction Assessment

Yes, at source level: current main accepts OGG/Opus as multimodal audio, stores the detected MIME type, and forwards it unchanged to Gemini inlineData. A live Gemini run with redacted credentials would still be useful before implementation.

## Solution Assessment

No, unconditional auto-transcoding is not clearly the best fix yet because it adds dependency and operator behavior. Maintainers should first choose whether Gemini should reject/skip unsupported audio with clear diagnostics or whether OpenClaw should own audio transcoding as a product feature.
