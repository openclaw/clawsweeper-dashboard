## Summary

Keep open. Current main still advertises `.ogg` and `.opus` as multimodal memory audio, preserves the detected MIME and raw bytes, forwards them unchanged to Gemini embedding `inlineData`, and rethrows non-size provider errors, so the reported failure remains source-reproducible.

## What This Changes

The issue requests that multimodal memory indexing avoid Gemini embedding failures for OGG/Opus audio files discovered through `memorySearch.extraPaths`.

## Best Possible Solution

Define a provider-aware multimodal audio contract: either transcode Opus/unsupported OGG into a Gemini-supported representation with live proof, or stop advertising/indexing unsupported raw audio and emit a clear skip or warning.

## Reproduction Assessment

Yes, at source level. Enabling multimodal audio indexing for `.ogg` or `.opus` classifies the file, preserves its detected audio MIME, sends it to Gemini as raw `inlineData`, and rethrows non-size provider errors; I did not run a fresh live Gemini request.

## Solution Assessment

Unclear. Auto-transcoding would preserve the advertised support but adds dependency and runtime policy, while provider-aware filtering plus docs is narrower but changes the current support contract.
