## Label Justifications

- `P2`: This is a normal-priority memory indexing regression with a bounded provider/format surface and a user workaround of converting audio to MP3 or WAV.
- `impact:session-state`: The bug affects which audio files become indexed memory context and therefore what later memory search can retrieve.

## Summary

Keep open. Current main and v2026.6.6 still accept OGG/Opus as multimodal memory audio, preserve the detected MIME type, and forward it unchanged to Gemini embeddings, while Gemini Embedding docs list MP3/WAV as the supported audio formats for that endpoint.

## What This Changes

Memory multimodal indexing should handle OGG/Opus audio for Gemini instead of sending unsupported raw audio MIME data to the embedding API.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: either reject or skip unsupported audio with clear diagnostics, or add an explicitly owned and tested transcode path, then align docs and regression coverage.

## Reproduction Assessment

Yes, at source level: current main accepts OGG/Opus as multimodal audio, stores the detected MIME type, and forwards it unchanged to Gemini inlineData. A live Gemini run with redacted credentials would still be useful before implementation.

## Solution Assessment

No automatic fix is clearly best yet. Maintainers should first choose whether Gemini should reject or skip unsupported audio with clear diagnostics, or whether OpenClaw should own audio transcoding as a product feature.
