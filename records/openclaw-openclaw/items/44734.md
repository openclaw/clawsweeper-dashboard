## Summary

Keep open: current main still advertises OGG/Opus as multimodal memory audio inputs and forwards their detected MIME types directly into Gemini embedding requests, so the reported provider rejection remains a real product gap rather than cleanup-closeable.

## What This Changes

The issue asks OpenClaw to handle OGG/Opus audio during Gemini-backed multimodal memory indexing or document the provider format limit.

## Best Possible Solution

Decide the Gemini multimodal memory contract, then implement provider-aware handling that either converts unsupported audio before embedding or skips it with clear diagnostics and docs aligned to the chosen behavior.

## Reproduction Assessment

Yes, from source inspection and the confirming issue comment. Current main selects `.ogg`/`.opus`, resolves them to audio MIME types, and sends those MIME types unchanged to Gemini embedding inline data.

## Solution Assessment

Unclear. The code path is narrow, but the best maintainable fix depends on whether maintainers want transcoding support or provider-specific format filtering plus docs updates.
