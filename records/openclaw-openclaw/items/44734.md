## Summary

Keep open. Current main and the latest release still accept OGG/Opus as multimodal memory audio, preserve the detected MIME type, and send it unchanged to Gemini embeddings; the narrower audio-format defect is not superseded by the broader multimodal memory tracker.

## What This Changes

The issue asks OpenClaw to stop failing Gemini multimodal memory indexing when `memorySearch.extraPaths` contains OGG/Opus audio files.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: either skip/reject unsupported audio with clear diagnostics and aligned docs, or add an explicitly approved and tested transcode path if maintainers want OGG/Opus to remain supported.

## Reproduction Assessment

Yes, at source level: current main accepts `.ogg` and `.opus`, stores the detected audio MIME type, and passes it unchanged to Gemini `inlineData`. I did not run a live Gemini account because this read-only review has no provider credentials.

## Solution Assessment

Unclear. The source bug is clear, but the best fix still requires choosing between provider-aware filtering/diagnostics and an explicitly owned transcoding path.
