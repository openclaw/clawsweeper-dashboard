## Summary

Keep open. Current main and v2026.5.12 still advertise `.ogg` and `.opus` as multimodal memory audio, preserve the detected raw MIME/data, forward it unchanged to Gemini embedding requests, and only suppress multimodal provider failures for size-limit errors; the related open feature request is broader and does not replace this narrower format bug.

## What This Changes

The issue asks OpenClaw to handle OGG/Opus audio in Gemini-backed multimodal memory indexing instead of failing after advertising those formats as supported.

## Best Possible Solution

Define a provider-aware multimodal audio contract: either transcode OGG/Opus into a Gemini-accepted representation with bounded dependency behavior, or stop advertising raw Gemini support for those formats and skip them with a clear warning/docs update.

## Reproduction Assessment

Yes, at source level. Enabling multimodal audio indexing for `.ogg` or `.opus` classifies the file, preserves its detected audio MIME, sends it to Gemini as raw `inlineData`, and rethrows non-size provider errors; I did not run a live Gemini request.

## Solution Assessment

Unclear. Auto-transcoding is the most complete user-visible fix, but maintainers need to decide whether OpenClaw should add transcode dependency behavior or instead narrow the provider-specific supported-format contract.
