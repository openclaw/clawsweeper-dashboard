## Summary

Keep open: current main and v2026.5.22 still advertise and accept `.ogg`/`.opus` multimodal memory audio, preserve the detected MIME/raw bytes, and pass them unchanged into Gemini inline data. The bug remains source-reproducible, but the fix needs a maintainer decision between transcoding unsupported Opus audio and narrowing the provider-specific support contract with skip/warn behavior plus docs.

## What This Changes

The issue asks memorySearch multimodal indexing to handle OGG/Opus audio files under `extraPaths` without Gemini rejecting the embedding request.

## Best Possible Solution

Define a provider-specific Gemini multimodal audio contract, then add tests and docs for either transcoding unsupported Opus audio to a Gemini-supported format or rejecting/skipping it before upload with a clear warning.

## Reproduction Assessment

Yes, at source level. Current main classifies `.ogg` and `.opus` as multimodal audio, preserves the detected MIME/raw bytes, and forwards them unchanged to Gemini inline data; I did not run a live Gemini request.

## Solution Assessment

Unclear. The narrow defect is real, but maintainers need to choose whether OpenClaw should own transcoding unsupported Opus audio or narrow the provider-specific supported-format contract with skip/warn behavior and docs.
