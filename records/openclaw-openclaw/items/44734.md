## Summary

Keep open. Current main and v2026.6.6 still document and admit OGG/Opus audio for Gemini multimodal memory, but the runtime forwards the detected MIME type unchanged to Gemini embeddings, whose embedding-specific docs list only MP3/WAV audio support; the broader tracker at https://github.com/openclaw/openclaw/issues/43117 remains open but treats this as a narrower audio-format follow-up.

## What This Changes

The issue reports that Gemini multimodal memory indexing accepts OGG/Opus audio from memorySearch.extraPaths and then fails when Gemini rejects that format.

## Best Possible Solution

Make Gemini multimodal memory provider-aware: either stop advertising/indexing unsupported OGG/Opus for Gemini or add an explicitly owned, tested transcode path, then align docs and regression coverage.

## Reproduction Assessment

Yes, at source level: current main accepts OGG/Opus as multimodal audio, stores the detected MIME type, and forwards it unchanged to Gemini inlineData. A live Gemini run would still be useful before choosing the final behavior.

## Solution Assessment

No, unconditional auto-transcoding is not clearly the best fix yet because it adds dependency and operator behavior. The safer maintainer path is to decide whether Gemini should reject/skip unsupported audio with clear diagnostics or whether OpenClaw should own audio transcoding as a product feature.
