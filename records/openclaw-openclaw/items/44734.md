## Summary

Keep open. Current main still advertises OGG/Opus audio for Gemini multimodal memory, preserves the detected MIME and raw bytes, and forwards them to Gemini without provider-specific format validation; unsupported-format provider errors still abort indexing unless they look like size-limit errors.

## What This Changes

The issue requests that Gemini multimodal memory indexing handle OGG/Opus audio from `memorySearch.extraPaths` without failing, either through supported-format handling/transcoding or accurate documentation.

## Best Possible Solution

Add provider-aware Gemini multimodal audio handling: do not send unsupported Gemini Embedding 2 audio inputs as raw inline data, make unsupported-format multimodal failures non-fatal for the rest of the index, and update docs/tests to describe raw supported formats. A later transcode path can be added behind existing trusted media plumbing if maintainers want OGG/Opus files to embed rather than be skipped.

## Reproduction Assessment

Yes. The source-level reproduction is clear: configure Gemini `gemini-embedding-2-preview` with multimodal audio extra paths containing `.ogg` or `.opus`, and current main will select the file, preserve its MIME/raw bytes, and forward it to Gemini inline without a supported-format gate.

## Solution Assessment

No current patch is under review. The narrowest maintainable fix is provider-aware format validation or a typed unsupported-format skip for Gemini, with docs/tests; full ffmpeg transcoding is a larger follow-up because it changes local binary execution and indexing semantics.
