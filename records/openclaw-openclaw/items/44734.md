## Summary

Keep open. Current main and v2026.6.8 still accept OGG/Opus as multimodal memory audio, preserve the detected MIME type, and forward it unchanged to Gemini embeddings, while the current Gemini Embedding 2 docs list MP3/WAV as the supported audio formats.

## What This Changes

The issue asks OpenClaw to stop failing multimodal memory indexing when Gemini-backed audio inputs are OGG/Opus files.

## Best Possible Solution

Make Gemini multimodal memory provider-aware, then align docs and regression coverage with the chosen behavior: clear skip/reject diagnostics for unsupported raw audio, or an explicitly owned and tested transcode path.

## Reproduction Assessment

Yes at source level: current main accepts OGG/Opus as multimodal audio, stores the detected MIME type, and forwards it unchanged to Gemini inlineData. I did not run a live Gemini request because this read-only review has no provider credentials path.

## Solution Assessment

Unclear for automation: the source bug is real, but the best fix depends on whether maintainers want provider-aware rejection/skipping or an owned audio transcoding capability. A docs-only change would avoid overpromising but would not satisfy the reported expectation that OGG/Opus files embed successfully.
