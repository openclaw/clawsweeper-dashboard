## Label Justifications

- `impact:session-state`: Unsupported audio can fail memory indexing and affect which extraPath files become searchable memory context.

## Summary

Keep open. Current main and v2026.6.10 still advertise and accept OGG/Opus as Gemini multimodal memory audio, preserve the detected audio MIME type, and pass it unchanged to Gemini inlineData, while Google's Gemini embeddings docs list audio support as MP3/WAV only. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/embeddings?utm_source=openai))

## What This Changes

Fix multimodal memory indexing so Gemini does not reject OGG/Opus audio discovered through memorySearch.extraPaths.

## Best Possible Solution

Make the Gemini multimodal memory path provider-aware: either reject or skip unsupported audio with clear diagnostics and aligned docs, or add an explicitly approved and tested transcode path if OGG/Opus should remain supported.

## Reproduction Assessment

Yes, at source level: current main accepts .ogg and .opus, stores the detected audio MIME type, and passes it unchanged to Gemini inlineData. I did not run a live Gemini account in this read-only review, so this is source- and provider-doc-backed rather than live-provider-backed.

## Solution Assessment

Unclear. The source bug is clear, but the best fix still requires a maintainer decision between provider-aware filtering/diagnostics and an explicitly owned transcoding path.
