## Summary

Keep open. Current main still advertises and accepts `.ogg`/`.opus` as multimodal memory audio, preserves the detected MIME bytes, forwards them unchanged into Gemini embedding `inlineData`, and only swallows size-limit failures, so the reported provider-format failure is still a valid source-level bug.

## What This Changes

The issue asks OpenClaw to make OGG/Opus audio files in `memorySearch.extraPaths` work with Gemini multimodal memory indexing instead of failing provider embedding requests.

## Best Possible Solution

Define a provider-aware multimodal audio contract: either transcode OGG/Opus into a Gemini-supported representation with live proof, or narrow the advertised formats and skip or warn on unsupported raw audio consistently.

## Reproduction Assessment

Yes, at source level. Enabling multimodal audio indexing for an `.ogg` or `.opus` file on current main classifies the file, preserves its detected MIME, sends it as Gemini `inlineData`, and rethrows non-size provider errors; I did not run a fresh live Gemini request.

## Solution Assessment

Unclear. Auto-transcoding would preserve the advertised support but adds dependency and runtime policy, while provider-aware filtering plus docs is narrower but changes the current support promise.
