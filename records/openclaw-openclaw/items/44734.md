## Summary

Keep open. Current main and v2026.5.18 still classify `.ogg` and `.opus` as multimodal memory audio, preserve the detected raw MIME/data, forward it unchanged to Gemini embedding requests, and rethrow non-size provider errors; the broader open tracker at https://github.com/openclaw/openclaw/issues/43117 does not replace this narrower format-contract bug.

## What This Changes

The issue reports that multimodal memory indexing accepts OGG/Opus audio but forwards it to Gemini Embedding as unsupported raw audio, causing indexing failures.

## Best Possible Solution

Define a provider-aware multimodal audio contract: either transcode OGG/Opus into a Gemini-accepted representation with bounded dependency behavior, or stop advertising raw Gemini support for those formats and skip them with clear warnings and docs.

## Reproduction Assessment

Yes, at source level. Current main classifies `.ogg` and `.opus` as multimodal audio, preserves the detected MIME and original bytes, forwards them unchanged to Gemini inline data, and rethrows non-size provider errors; I did not run a live Gemini request.

## Solution Assessment

Unclear. Auto-transcoding is the most complete user-visible fix, but maintainers need to decide whether OpenClaw should add transcode dependency behavior or instead narrow the provider-specific supported-format contract.
