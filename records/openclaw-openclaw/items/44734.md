## Summary

Keep open. Current main and the latest shipped release still advertise `.ogg`/`.opus` audio multimodal memory indexing and forward detected MIME bytes unchanged into Gemini structured embedding requests, so the reported provider rejection remains unresolved.

## What This Changes

The issue asks OpenClaw to handle OGG/Opus audio files during Gemini-backed multimodal memory indexing instead of failing with a provider 400.

## Best Possible Solution

Define provider-aware multimodal audio handling: either transcode OGG/Opus into a Gemini-supported format with live proof, or stop accepting and advertising those raw formats and warn or skip consistently.

## Reproduction Assessment

Yes, at source level. Enable multimodal audio indexing for an `.ogg` or `.opus` file with Gemini Embedding 2 and current main classifies it, preserves the detected MIME, sends it as Gemini `inlineData`, and rethrows non-size provider errors; I did not establish a fresh live 400.

## Solution Assessment

Unclear. Auto-transcoding would satisfy the advertised behavior but adds dependency and runtime policy, while provider-aware validation plus docs is narrower but changes what OpenClaw promises to index.
