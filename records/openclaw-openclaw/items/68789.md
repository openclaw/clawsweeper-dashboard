## Summary

Keep open. Current main has related Slack stability work, but it still does not show a Slack send-path recovery or diagnostic for fresh `WebClient` `getaddrinfo EAI_AGAIN slack.com` failures; this is distinct from the older crash-loop fix because the reported gateway stays running while sends continue to fail.

## What This Changes

The issue asks OpenClaw to recover from or clearly diagnose intermittent Slack WebClient DNS failures when host DNS and host Node HTTPS checks are healthy.

## Best Possible Solution

Add a Slack-owned, narrowly scoped recovery layer for pre-connect DNS request errors in the WebClient send path, with diagnostics that identify Slack WebClient versus Socket Mode failures. Keep broad write retries disabled unless the replacement retry is constrained to safe DNS/request-error cases.

## Reproduction Assessment

Unclear. The report has credible live evidence, and current code can be unit-reproduced by injecting a Slack WebAPI request error with nested `EAI_AGAIN`, but no deterministic live gateway repro has been isolated.

## Solution Assessment

Yes, a narrow Slack send-path fix is the best direction. A broad SDK retry policy for all Slack writes is riskier because `chat.postMessage` is non-idempotent; DNS-only retry plus clearer logging is the safer maintainable path.

In the Slack plugin, add a Slack-owned helper around the WebClient send path that detects pre-connect DNS/request failures from Slack WebAPI request errors, especially nested `.original`, `.cause`, or `.data` values with `code`/`errno` `EAI_AGAIN` or `ENOTFOUND`, `syscall: getaddrinfo`, and `hostname` under `slack.com`.
