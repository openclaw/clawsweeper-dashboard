## Summary

Keep open: current main still has the non-abortable trajectory flush path, and #77154 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports that a 50 MiB runtime trajectory sidecar makes `pi-trajectory-flush` time out and then continue blocking the gateway event loop.

## Best Possible Solution

Review and land #77154, or close it with a narrow replacement that bounds runtime trajectory flushing without relying on a timeout knob alone.

## Reproduction Assessment

No, I did not establish a high-confidence live 50 MiB reproduction. Source inspection does show the current-main path where the 10s cleanup race returns while the trajectory writer can keep draining its queued chain.

## Solution Assessment

Yes, the bounded runtime trajectory flush direction in #77154 is the narrow maintainable path. An environment timeout increase alone would not solve the background queue drain that current main leaves running.
