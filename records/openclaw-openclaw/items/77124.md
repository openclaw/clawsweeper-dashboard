## Summary

Keep open: current main still has the reported timeout shape, and the linked reports are related but do not supersede this trajectory-specific failure mode.

## What This Changes

The issue reports that a large trajectory sidecar can make `pi-trajectory-flush` time out while the underlying flush continues saturating the gateway event loop.

## Best Possible Solution

Make trajectory flushing bounded and low-impact in the core recorder/writer path, with regression coverage, while leaving broader session-list and compaction work to the related issues.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current-main path where the cleanup timeout returns without stopping the trajectory flush and the writer continues draining its queued chain.

## Solution Assessment

Yes for a narrow bounded/abortable trajectory flush fix; no for the env-timeout workaround by itself, because increasing the timeout would not stop background work from monopolizing the event loop.

## Repair Work Prompt

Fix the trajectory flush event-loop stall reported here. Keep scope to Pi/Core trajectory runtime and the queued file writer: ensure the cleanup timeout bounds the actual trajectory flush work, for example by adding abort/stop semantics to `runAgentCleanupStep` and the trajectory writer/recorder, or by making the recorder drop queued writes once the runtime file cap or timeout is reached. Do not make an environment timeout knob the primary fix, and do not redesign `sessions.list` or compaction. Add regression tests for a timed-out trajectory flush that does not continue draining indefinitely and for oversized trajectory queues/file caps. Add a user-facing `CHANGELOG.md` entry.
