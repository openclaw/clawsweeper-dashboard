## Summary

Current main still accepts and resolves `memorySearch.outputDimensionality`, but the OpenAI memory embedding path drops that value before building inline and batch OpenAI embedding requests. The issue should stay open because the bug remains and open PR #55507 is still the target-repo implementation candidate using closing syntax for this issue.

## What This Changes

The issue requests wiring `memorySearch.outputDimensionality` through the OpenAI memory embedding path so OpenAI embeddings requests include the configured `dimensions` value.

## Best Possible Solution

Keep the issue open and land an updated #55507 or replacement patch that threads `memorySearch.outputDimensionality` into the OpenAI embedding client, sends OpenAI `dimensions` for inline and batch `/v1/embeddings` requests when configured, includes the dimension in provider cache/reindex identity, and adds regression coverage for those paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction follows the configured field through schema and memory-search resolution, then shows the OpenAI inline and batch request builders never include `dimensions`, matching the reported 1536-dimension behavior for `text-embedding-3-small`.

## Solution Assessment

Yes, the narrow maintainable solution is provider-local: carry the resolved dimension through the OpenAI embedding client and request builders, with cache-key/reindex and test coverage, instead of adding a core migration or changing unrelated providers.
