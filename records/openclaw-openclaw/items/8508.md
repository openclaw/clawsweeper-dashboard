## Summary

This feature remains necessary: current main and v2026.6.5 support only one configured acknowledgement reaction, while contextual selection and cross-channel fallback behavior remain unresolved product decisions.

## What This Changes

Allow acknowledgement reactions to vary from a configured emoji set or be selected contextually for each inbound message.

## Best Possible Solution

Preserve the existing singular `ackReaction` behavior, then add one additive, typed selection contract after maintainers choose deterministic versus model-driven semantics, with channel-owned validation/fallbacks and aligned config schema, SDK, docs, and bundled-channel coverage.

## Reproduction Assessment

Yes at source level: current main demonstrably accepts and resolves only one acknowledgement reaction string, so neither the proposed array nor contextual selector currently exists.

## Solution Assessment

Unclear: native shared support is the appropriate boundary, but the proposed alternatives have materially different latency, predictability, configuration, and channel-compatibility implications that require a product decision first.
