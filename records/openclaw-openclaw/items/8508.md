Keep this open: current main and v2026.5.20 still support only one resolved ack reaction value, while multiple or contextual acknowledgement reactions remain an unresolved public config and channel behavior design.

## What This Changes

Allow acknowledgement reactions to be chosen from multiple configured emojis or dynamically/contextually instead of using one fixed `ackReaction` string.

## Best Possible Solution

After maintainer API review, add a backward-compatible `ackReactions` array or named selector/strategy contract while preserving `ackReaction`, then update schemas, docs, SDK/channel seams if needed, and focused bundled-channel tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main and v2026.5.20 only support a singular `ackReaction` config/runtime path and no `ackReactions` array or contextual selector.

## Solution Assessment

Unclear until maintainers choose the behavior model. The safest direction is a backward-compatible config/API expansion rather than changing existing `ackReaction` semantics in place.
