## Summary

Keep open: current main still only supports a single configured ack reaction plus lifecycle status reactions, so the requested configured list or contextual selector remains unimplemented and needs maintainer API/product direction.

## What This Changes

Allow automatic ack reactions to vary from a configured emoji list or contextual per-message selector instead of one fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then design a backward-compatible ack-selection contract covering config shape, selection timing, SDK/channel-feedback seams, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection verifies current main has only the existing single-value ack path rather than a failing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native support is the right surface if bundled channels should vary automatic ack reactions, but the implementation needs an explicit strategy and per-channel validation rather than a narrow bug fix.
