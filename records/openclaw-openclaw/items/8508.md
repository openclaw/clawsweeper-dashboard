## Summary

Keep this open: current main and the latest release still expose one configured ack reaction value plus lifecycle status reactions, not an `ackReactions` array, selector strategy, or model/context-driven pre-ack choice. The merged per-channel ackReaction work is useful related context, but it only made the single value configurable by channel/account, so the remaining request needs a maintainer product/API decision.

## What This Changes

Request configurable or contextual ack reaction selection so the agent can vary the acknowledgement emoji instead of always using one fixed value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract and add a backward-compatible `messages.ackReactions` array or selector/strategy contract only after maintainer API review, with schema, docs, SDK/channel-feedback, and bundled-channel tests updated together.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main and v2026.5.12 only support a singular `ackReaction` config/runtime path and no dynamic or array-based ack selector.

## Solution Assessment

Unclear until maintainers choose the behavior model. The safest direction is a backward-compatible config/API expansion rather than changing existing `ackReaction` semantics in place.
