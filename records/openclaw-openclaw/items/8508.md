## Summary

Keep open: current main still implements a single configured `ackReaction` string with per-channel/account overrides and lifecycle status reactions, but no configured list, selector, random chooser, or contextual per-message ack reaction path exists.

## What This Changes

The issue requests configurable ack reaction arrays or dynamic/contextual per-message selection for automatic inbound ack reactions.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then design an additive ack-selection contract covering config shape, timing, SDK/channel-feedback seams, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection of current main shows only the existing single-value ack path rather than a failing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native support is likely the right surface if bundled channels should vary automatic ack reactions, but the selection strategy and per-channel validation need explicit design first.
