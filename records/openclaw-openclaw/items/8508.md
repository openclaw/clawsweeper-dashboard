## Summary

Keep open: current main still exposes a single ack-reaction value, and the requested configurable list or contextual selector is not implemented. The only directly matching implementation attempt, https://github.com/openclaw/openclaw/pull/6790, was closed unmerged, so this remains a feature/API design item rather than a cleanup-close candidate.

## What This Changes

Request configurable lists or contextual selection for ack-reaction emojis instead of one fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or selector contract after maintainers choose selection semantics and per-channel fallback rules.

## Reproduction Assessment

## Risks / Open Questions

- Selection semantics are unresolved: deterministic rotation, random choice, rule-based selection, and model-driven choice have different latency and predictability tradeoffs because ack reactions are sent before normal reply generation.
- Channel emoji contracts differ across Slack shortcodes, Discord unicode/custom emoji, Telegram supported reactions, WhatsApp object config, and iMessage tapbacks, so validation and fallback behavior need explicit design.
