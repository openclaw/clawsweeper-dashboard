## Summary

Keep open: current main still only supports single-string ack reaction resolution plus lifecycle status reactions; the requested configured list or contextual chooser is not implemented and needs API/product design.

## What This Changes

The issue requests configurable or contextual automatic acknowledgement reactions instead of one fixed `ackReaction` emoji.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then design an additive ack-selection contract covering config shape, timing, SDK/channel-feedback seams, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and current-main source/docs inspection shows only the existing single-value ack path rather than a failing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native channel/config support is likely the right surface if automatic ack reactions should vary, but the selection strategy and per-channel validation need explicit design first.

## Risks / Open Questions

- Selection policy is unresolved: deterministic rotation, random choice, rule-based choice, and model-driven choice have different latency and predictability tradeoffs because ack reactions are sent before normal reply generation.
- Channel reaction formats differ across Slack shortcodes, Discord unicode/custom emoji, Telegram supported reactions, WhatsApp ack objects, Matrix reactions, and BlueBubbles tapbacks, so validation and fallback behavior need explicit design.
