## Summary

Current main still exposes ack reactions as a single configured value plus lifecycle status reactions; the requested configurable list or contextual selector remains unimplemented and needs maintainer API/product direction.

## What This Changes

Request to let automatic ack reactions vary by configured emoji list or contextual agent/channel selection instead of always using one fixed ack emoji.

## Best Possible Solution

Keep the existing `ackReaction` string contract, then add a backward-compatible ack selection design that defines the new config shape, selection timing, SDK/channel-feedback seam, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection confirms current main has only the existing single-value ack path rather than a failing implementation to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native support is the right surface if bundled channels should vary automatic ack reactions, but the design needs an explicit strategy and per-channel validation rather than a narrow bug fix.
