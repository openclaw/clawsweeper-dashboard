## Summary

Current main still supports only one configured acknowledgment emoji, and no related merged or viable open PR implements an array or contextual chooser. The request remains useful but needs an explicit maintainer decision because deterministic rotation, random selection, rules, and model-driven selection have materially different latency, predictability, and cross-channel behavior.

## What This Changes

Add configurable multiple or context-sensitive acknowledgment reaction emojis instead of one fixed acknowledgment reaction.

## Best Possible Solution

Preserve `ackReaction` as the backward-compatible default, then—if maintainers approve the feature—add one typed, deterministic selection policy at the shared channel-feedback boundary with explicit per-channel validation and fallback behavior; avoid a pre-ack model call unless its added latency and nondeterminism are an intentional product choice.

## Reproduction Assessment

Not applicable: this issue requests behavior beyond the documented single-value acknowledgment contract rather than reporting a failure of current behavior. Source inspection provides high confidence that the requested selector is still absent.

## Solution Assessment

No single proposed approach is yet the best solution: model-driven choice is broader and slower than necessary, while an array alone leaves selection and fallback semantics undefined. A deterministic typed policy is the narrowest maintainable direction if maintainers approve the capability.
