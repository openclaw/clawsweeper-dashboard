## Summary

Keep open: current main and the latest release still expose only a singular acknowledgement reaction value, while array-based or contextual per-message ack selection remains an unresolved config/channel API design.

## What This Changes

The issue requests configurable or dynamic acknowledgement reaction emojis so an agent can vary the pre-processing reaction instead of always using one fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible `ackReactions` or selector/strategy contract only after maintainers choose selection semantics and per-channel fallback rules.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug repro, but yes at source level: current main and `v2026.6.5` only expose one `ackReaction` string and no list or contextual selector.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the behavior model; the safest direction is a backward-compatible config/API expansion rather than changing the existing single-string semantics in place.
