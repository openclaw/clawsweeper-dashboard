## Summary

Keep this open: current main and the latest release still expose a single configured ack reaction plus lifecycle status reactions, with no `ackReactions` array, selector strategy, or model/context-driven pre-ack choice. This is a public config/API design request rather than conservative cleanup.

## What This Changes

The issue requests configurable or dynamic acknowledgement reactions so agents can vary the emoji used before processing a message instead of always using one fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible `messages.ackReactions` array or selector/strategy contract with schema, docs, SDK/channel seams, and focused bundled-channel tests after maintainer API review.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main and `v2026.5.19` only support a singular `ackReaction` config/runtime path and no dynamic or array-based ack selector.

## Solution Assessment

Unclear until maintainers choose the behavior model. The safest direction is a backward-compatible config/API expansion rather than changing existing `ackReaction` semantics in place.
