## Summary

Keep open. Current main still exposes one configured ack reaction value plus lifecycle status-reaction states; it does not have an `ackReactions` array, selector/strategy contract, or message-context/model-driven pre-ack hook, and the prior issue discussion reached the same keep-open conclusion.

## What This Changes

The issue requests configurable or contextual acknowledgement reaction emojis instead of a single fixed ack reaction.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or named selector/strategy API with aligned schemas, docs, metadata, SDK contracts, and bundled-channel tests after maintainers choose the behavior model.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The capability gap is high-confidence from current-main source inspection because the schema, resolver, helper, docs, and channel call sites all carry one ack value and expose no array or selector.

## Solution Assessment

Unclear until maintainers choose the API direction. A typed, backward-compatible config array or strategy seam is likely safer than a hidden model hook because acknowledgement reactions are sent before normal reply generation.
