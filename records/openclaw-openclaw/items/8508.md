## Summary

Keep open. Current main still exposes a single ack reaction value through config, schema, resolver, SDK feedback helpers, docs, and bundled channel call sites; there is no array, selector, random/contextual strategy, or pre-ack model hook. The prior maintainer/Codex comment reached the same conclusion, and the remaining work needs an API/product decision before implementation.

## What This Changes

Allow ack reactions to use a configured list or dynamic/contextual emoji selection instead of one fixed configured emoji.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or explicit selector/strategy contract once maintainers choose the behavior model. The implementation should align exported config types, zod/generated schemas, help metadata, public docs, SDK/channel-feedback contracts if third-party channels need the seam, and focused bundled-channel tests for platform validation and fallback behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The capability gap is high-confidence from inspection: current main only validates and resolves one `ackReaction` string, and a targeted search found no dynamic or array-based ack reaction surface.

## Solution Assessment

Unclear until maintainers choose the API direction. A backward-compatible config array or named strategy seam is likely safer than a hidden model hook because it keeps pre-reply acknowledgement predictable and testable across channel-specific emoji rules.
