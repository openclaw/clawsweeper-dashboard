## Summary

Keep open. Current main still exposes one ack reaction value through config, schema, resolver, SDK helpers, docs, and bundled channel call sites; the requested array/contextual behavior remains a new cross-channel API decision.

## What This Changes

Request native support for configurable arrays or contextual selection of ack reaction emojis instead of one fixed ack reaction per channel/account.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or explicit selector/strategy contract with aligned config types, schemas, SDK docs, generated metadata, and bundled-channel tests after maintainers choose the behavior model.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The capability gap is high-confidence from current-main source inspection: the schema, resolver, helper, docs, and channel call sites all carry one ack value and expose no array or selector.

## Solution Assessment

Unclear until maintainers choose the API direction. A backward-compatible config array or named strategy seam is likely safer than a hidden model hook because it keeps pre-reply acknowledgement predictable and testable across channel-specific emoji rules.
