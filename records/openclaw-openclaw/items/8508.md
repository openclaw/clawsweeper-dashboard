## Summary

Keep open: current main still exposes a single ack reaction value plus lifecycle status reactions, and the requested configured list or contextual per-message selector remains unimplemented and needs maintainer API/product direction.

## What This Changes

Add native support for choosing ack reaction emojis from a configured list or per-message contextual selector instead of a single fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then design an additive ack-selection contract covering config shape, timing, SDK/channel-feedback seams, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and current-main source inspection shows only the existing single-value ack path rather than a failing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native support is likely the right surface if bundled channels should vary automatic ack reactions, but the selection strategy and per-channel validation need explicit design first.
