## Summary

Keep open: current main still lacks multiple or contextual ack-reaction selection, and the prior discussion reached the same conclusion. This is a feature/config design item, not a broken existing contract or cleanup-close candidate.

## What This Changes

Request native support for selecting ack reaction emojis from a configured set or dynamically/contextually per inbound message.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or selector contract after maintainers choose selection semantics and per-channel fallback rules.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main gap: the config, schema, resolver, docs, and SDK helper surface still model a single ack reaction string.

## Solution Assessment

Unclear until maintainers choose the product/API direction. The narrow maintainable path is a backward-compatible config/selector design rather than treating this as a small bug fix.
