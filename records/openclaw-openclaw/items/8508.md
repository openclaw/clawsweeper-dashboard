## Summary

Keep open: current main still models ack reactions as one configured/resolved value, and dynamic or list-based selection is a new config/API design item rather than conservative cleanup.

## What This Changes

The issue requests configurable or contextual ack reaction emojis, such as an `ackReactions` array or per-message selector, instead of one fixed ack reaction.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or selector contract after maintainers choose selection semantics and per-channel fallback rules.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source inspection on current main gives a high-confidence current gap because config, schema, resolver, docs, tests, and helpers still model one ack reaction value.

## Solution Assessment

Unclear until maintainers choose deterministic, random, rule-based, or model-driven selection. The maintainable direction is a backward-compatible config or selector contract with per-channel validation, not a narrow bug fix.
