## Summary

Keep open: current main still exposes only a single-string ack reaction path, while the requested configured list or contextual chooser remains unimplemented and needs maintainer API/product direction.

## What This Changes

The issue requests configurable or dynamic ack reactions so agents can choose from multiple emojis or contextually select reactions before processing messages.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then design an additive ack-selection contract covering config shape, timing, SDK/channel-feedback seams if needed, docs, schemas, and bundled-channel fallback tests.

## Reproduction Assessment

Not applicable. This is a feature request, and current-main source/docs inspection shows the existing single-value ack path rather than broken existing behavior to reproduce.

## Solution Assessment

Unclear until maintainers choose the API. Native channel/config support is likely the right surface if automatic ack reactions should vary, but the selection strategy and per-channel validation need explicit design first.
