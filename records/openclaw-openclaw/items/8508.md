## Summary

Keep open: current main and v2026.6.5 still provide only a single acknowledgement reaction value, while the selection strategy and cross-channel fallback contract remain unresolved product decisions.

## What This Changes

Add configurable multiple acknowledgement reactions or contextual per-message reaction selection instead of always using one fixed emoji.

## Best Possible Solution

Preserve the existing ackReaction string as the compatible default, then add one maintainer-approved selection strategy with canonical config, deterministic behavior, and documented per-channel validation and fallback.

## Reproduction Assessment

Yes, at source level: current main accepts, resolves, sends, and removes one reaction string and exposes no array, strategy, or message-context input. This proves the feature gap rather than a failure of existing behavior.

## Solution Assessment

No final best approach has been selected yet: configured deterministic selection is narrower and more predictable than a model-driven pre-ack hook, but maintainers must decide the intended semantics and channel fallback contract.
