## Summary

Keep open: current main and the latest release still model ack reactions as a single resolved value, while the requested array/contextual selector is a new cross-channel config/API design.

## What This Changes

Request native support for configurable multiple ack reaction emojis or contextual per-message ack reaction selection instead of one fixed `ackReaction` value.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or selector contract once maintainers choose selection semantics and per-channel fallback rules.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source inspection gives a high-confidence current gap because config, schema, resolver, docs, tests, and helpers still model one ack reaction value.

## Solution Assessment

Unclear until maintainers choose deterministic, random, rule-based, or model-driven selection. The maintainable direction is a backward-compatible config or selector contract with per-channel validation, not a narrow bug fix.
