## Summary

Keep open. Current main still exposes a single ack reaction value through config, schema, resolver, SDK helper, docs, tests, and bundled channel call sites; the requested array/contextual behavior remains a new cross-channel API and product decision.

## What This Changes

Request configurable or contextual ack reaction selection, such as an `ackReactions` array or agent-selected emoji, instead of one fixed pre-processing ack emoji.

## Best Possible Solution

Preserve the existing `ackReaction` string contract, then add a backward-compatible array or named selector/strategy contract with aligned config types, schemas, SDK docs, generated metadata, and bundled-channel tests after maintainers choose the behavior model.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The capability gap is high-confidence from current-main source inspection because the schema, resolver, helper, docs, and channel call sites all carry one ack value and expose no array or selector.

## Solution Assessment

Unclear until maintainers choose the API direction. A backward-compatible config array or named strategy seam is likely safer than a hidden model hook because pre-reply acknowledgement should stay predictable and testable across channel-specific emoji rules.
