## Summary

Keep open. Current `main` has some Codex app-server idle-timeout and TUI stale-streaming guards, but the source still allows the reported kind of terminal-state divergence and still avoids automatic active-run recovery in the relevant diagnostic paths.

## What This Changes

The issue reports that native Codex harness app-server timeout/close behavior can leave TUI turns stuck, keep active embedded runs from recovery, and create mismatched trajectory versus persisted session status.

## Best Possible Solution

Harden native Codex app-server lifecycle so timeout or closed-client paths reach one terminal outcome, clear active run and lane state after bounded cleanup, align trajectory and persisted session status, and keep the TUI responsive under gateway backlog.

## Reproduction Assessment

Unclear/source-reproducible: the issue has detailed live logs, and current source still contains matching status-divergence and non-aborting active-run recovery paths. I did not establish a deterministic current-main live reproduction of the macOS TUI keyboard freeze.

## Solution Assessment

Unclear: no patch is proposed. The maintainable direction should start at Codex harness lifecycle and bounded active-run recovery policy; TUI stale-state handling alone is not enough.
