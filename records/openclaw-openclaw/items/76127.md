## Summary

Keep open. Current main has related Codex app-server timeout and TUI stale-stream guards, but the source still has paths matching the report: diagnostic recovery observes active embedded runs by default, Codex can record completion before an interrupted session end, and aborted lifecycle ends persist as timeout.

## What This Changes

The issue requests deterministic Codex app-server timeout/recovery behavior so TUI turns, trajectory events, and persisted session state stay consistent after app-server client closure.

## Best Possible Solution

Add targeted native Codex harness lifecycle hardening so app-server close/timeout reaches exactly one terminal outcome, clears active run/lane state after bounded cleanup, keeps trajectory and persisted session status aligned, and preserves TUI responsiveness under gateway backlog.

## Reproduction Assessment

Unclear: the report provides detailed logs and a plausible live pattern with trivial prompts, but there is no deterministic local/unit reproduction in current main. The current source explains several observed states, especially observe-only recovery and timeout persistence.

## Solution Assessment

Unclear: there is no proposed patch yet. A maintainable fix should start in the Codex harness/session lifecycle boundary and add focused regression coverage before treating this as a generic TUI reset problem.
