## Summary

Keep open. Current main still contains the source paths matching the report: stuck-session recovery observes active embedded runs by default, Codex can record a successful model completion before ending the session as interrupted, and gateway persistence still maps aborted lifecycle ends without `stopReason: "aborted"` to `timeout`; adjacent TUI stale-streaming fixes do not prove this Codex app-server lifecycle issue is resolved.

## What This Changes

The issue reports native Codex harness turns getting stuck or internally inconsistent after Codex app-server closes, including TUI unresponsiveness and trajectory/session status divergence.

## Best Possible Solution

Harden native Codex app-server lifecycle so close/timeout reaches one terminal outcome, clears active run and lane state after bounded cleanup, aligns trajectory and persisted session status, and keeps TUI input state usable while gateway work is backed up.

## Reproduction Assessment

Unclear/source-reproducible. The report has detailed live logs and current main still has source paths for observe-only recovery and completion/interrupted/timeout divergence, but I did not establish a deterministic current-main live reproduction of the full TUI freeze.

## Solution Assessment

Unclear because there is no proposed patch yet. The maintainable direction should start at the Codex app-server lifecycle and active-run recovery boundary before treating this as only a TUI reset problem.
