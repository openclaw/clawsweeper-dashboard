## Summary

Keep open. Current main has partial post-release handling for closed Codex app-server clients, but the remaining source still preserves the `active_embedded_run` observe-only recovery path and can still persist an aborted-but-successful Codex lifecycle as `timeout`, which matches the report's core state inconsistency.

## What This Changes

The issue asks OpenClaw to make native Codex app-server harness turns fail or complete deterministically, clear active embedded-run/session state after app-server close or timeout, keep TUI responsive, and record effective `serviceTier` evidence.

## Best Possible Solution

Keep the issue open and land a targeted Codex lifecycle/recovery fix that gives app-server close/timeout/abort paths one terminal outcome, clears active run/lane state, persists session status consistently with trajectory completion, and records effective `serviceTier` in Codex artifacts.

## Reproduction Assessment

Unclear for the full live TUI freeze because this read-only review did not run a macOS gateway/TUI load repro. Yes for the source-level mismatch: current main still has an observe-only active-run recovery branch and maps Codex aborted lifecycle completion to persisted `timeout`.

## Solution Assessment

Unclear. Current main’s closed-client restart/retry changes are a good partial direction, but the remaining fix likely needs maintainer judgment on when stuck-session recovery may abort or force-clear active Codex harness runs.
