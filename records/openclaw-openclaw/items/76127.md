## Summary

Keep open. Current main still preserves the reported source-level failure shapes: observe-only recovery while an active embedded run exists, Codex completion followed by interrupted session end, and gateway persistence mapping aborted lifecycle end without `stopReason` to timeout; newer TUI stale-streaming guards and the Codex 0.128 dependency do not prove the app-server/TUI freeze is fixed.

## What This Changes

The issue asks OpenClaw to make native Codex app-server TUI turns timeout and recover deterministically, clear active embedded run state, keep trajectory/session status consistent, and keep the TUI responsive after app-server client closure.

## Best Possible Solution

Harden native Codex app-server run lifecycle so app-server close/timeout converges to one terminal outcome with explicit status, bounded cleanup clears active run/lane state, persisted session state matches trajectory, and the TUI receives a responsive terminal state.

## Reproduction Assessment

Unclear/source-reproducible: the live report has detailed logs, and current source can still produce the observe-only recovery, completion/interrupted split, and timeout persistence mapping. I did not establish a deterministic current-main live reproduction for the full TUI freeze.

## Solution Assessment

Unclear: there is no proposed patch yet. A TUI-only reset would be too narrow because the strongest evidence sits at the Codex app-server/session lifecycle and active-run recovery boundary.
