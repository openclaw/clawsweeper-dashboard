## Summary

Keep this PR open: the WebChat reliability direction is useful, but the current diff still introduces source-reproducible regressions by dropping drained main-session system events and by adding a global GPT-5 acknowledgement instruction outside the WebChat receipt path.

## What This Changes

The PR adds WebChat preflight acknowledgement and pending-user-message continuity, delays brief Control UI disconnect cleanup, adjusts reset transcript allocation and preflight compaction, threads toolsAllow, updates GPT-5 prompt copy, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the WebChat reliability work, but preserve queued system-event delivery, keep receipt/acknowledgement behavior scoped to the WebChat gateway/UI path, and rebase over current main's related WebChat fixes.

## Reproduction Assessment

Yes. On PR head, a main-session run where drainFormattedSystemEvents returns a block will consume the events and then suppress them from commandBody/followupRun; current main has focused coverage that those queued events must reach commandBody.

## Solution Assessment

No. The reliability approach is reasonable, but dropping drained runtime events and using a provider-wide GPT-5 acknowledgement prompt are broader than the WebChat receipt fix; the narrower solution is gateway/UI preflight state with existing auto-reply event delivery preserved.
