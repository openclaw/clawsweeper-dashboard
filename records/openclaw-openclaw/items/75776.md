## Summary

Keep open: the WebChat reliability work is still useful, but the latest branch drops queued main-session system events after they are destructively drained, so it should not merge until that delivery path is preserved.

## What This Changes

The PR adds WebChat accepted-send acknowledgement, pending-turn continuity, delayed disconnect cleanup, safer reset transcript handling, earlier compaction gates, and focused tests.

## Best Possible Solution

Preserve queued system-event delivery for main WebChat sessions, filter only specific generated noise before draining or display, then rebase and validate the WebChat reliability changes against current main.

## Reproduction Assessment

Yes. On PR head, enqueue or mock a generic system event for `agent:main:main`, run `runPreparedReply`, and assert it reaches `runReplyAgent` command text or `followupRun.prompt`; source inspection shows the new suppression flag drops it after the queue is consumed.

## Solution Assessment

No. The broader reliability direction is reasonable, but suppressing every drained system-event block for the main session is broader than needed; the safer fix is to preserve event delivery and filter only unwanted generated acknowledgement/noise.
