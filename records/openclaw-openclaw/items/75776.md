## Summary

Keep this PR open: the latest head still has two source-level pending-history bugs that can make the new WebChat continuity path drop a pending prompt during refresh.

## What This Changes

The PR adds WebChat preflight acknowledgements, pending user-turn history continuity, delayed Control UI disconnect cleanup, reset transcript handling, earlier preflight compaction, changelog coverage, and focused tests.

## Best Possible Solution

Fix the pending-history overlay to use canonical session/run identity and preserve distinct repeated user turns, then review overlap with #74733 before landing.

## Reproduction Assessment

Yes. Source inspection gives two high-confidence paths: send the same text twice within the matcher window and refresh history while the second turn is pending, or send through the canonical main session and refresh history through the `main` alias.

## Solution Assessment

No. The general direction is maintainable, but the current pending overlay still matches by fuzzy text/time and raw request key, which is not reliable enough for the WebChat continuity bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep repeated pending prompts visible:** `src/gateway/server-methods/chat.ts:1045-1046`
  - body: This matcher treats a prior user message with the same normalized text and a timestamp within an hour as the pending turn. If a user sends the same prompt again after the first one is persisted, a `chat.history` refresh while the second run is still pending skips the overlay and the new prompt disappears until persistence catches up. Match by run/message identity when available, or use a stricter sequence/timestamp check.
  - confidence: 0.91
- **[P2] Use the canonical key for pending history:** `src/gateway/server-methods/chat.ts:1885`
  - body: `chat.send` stores pending messages under the canonical session key from `loadSessionEntry`, but `chat.history` passes the raw request key into the pending overlay. Supported aliases like `main` can load the same `agent:main:main` session while failing this pending-store comparison, so alias clients miss the pending prompt overlay this PR is adding.
  - confidence: 0.87
