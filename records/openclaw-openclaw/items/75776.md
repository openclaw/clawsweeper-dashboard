## Summary

Keep open: the WebChat reliability direction is still relevant, but the PR has two source-backed correctness bugs in the pending-history overlay that can still let accepted user turns disappear during refresh windows.

## What This Changes

The branch adds Gateway pending-user/history and preflight acknowledgement handling for WebChat, delayed Control UI disconnect cleanup, reset transcript and preflight compaction changes, prompt/tool allowlist plumbing, a changelog entry, and focused tests.

## Best Possible Solution

Fix the pending-history overlay so it is keyed by canonical session/run identity and preserves repeated same-text turns, then review overlap with #74733 before landing the WebChat reliability changes.

## Reproduction Assessment

Yes. Source inspection shows current main can accept `chat.send` without an early visible chat event, and the PR-introduced matcher has clear source-level reproductions for repeated same-text prompts and alias session keys during `chat.history` refresh.

## Solution Assessment

No, not yet. The approach is maintainable, but the pending overlay must use canonical session matching and avoid fuzzy text/time dedupe before it can reliably solve the visible-continuity bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep repeated pending prompts visible:** `src/gateway/server-methods/chat.ts:1045-1046`
  - body: This treats any prior user message with the same normalized text and a timestamp within an hour as the pending turn. If a user sends the same prompt again after the first one is persisted, a `chat.history` refresh while the second run is still pending will skip the overlay and the new prompt disappears until persistence catches up. Match pending turns by run/message identity when available, or use a stricter timestamp/sequence check.
  - confidence: 0.91
- **[P2] Use the canonical key for pending history:** `src/gateway/server-methods/chat.ts:1885`
  - body: `chat.send` stores pending messages under the canonical session key from `loadSessionEntry`, but `chat.history` passes the raw request key here. Supported aliases like `main` can load the same `agent:main:main` session while failing this pending-store comparison, so alias clients miss the pending prompt overlay this PR is adding.
  - confidence: 0.87
