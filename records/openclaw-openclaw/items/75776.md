## Summary

Keep this PR open. The branch contains useful WebChat reliability work, but the current diff introduces blocking behavior regressions: main-session system events can be drained and discarded before reaching the agent, normal main WebChat tool availability is narrowed by a hard-coded runtime allow-list, and user-authored leading System lines can disappear from visible history.

## What This Changes

The PR changes Gateway WebChat send/history streaming behavior, agent prompt/tool/bootstrap parameters, session-store terminal state fallback, preflight compaction thresholds, reset session-file selection, and Control UI chat stream state/UI handling.

## Best Possible Solution

Revise the branch so WebChat reliability improvements remain UI- and reconciliation-focused while preserving existing system-event delivery, config/profile/plugin-driven tool availability, and visible user text. Add focused tests for the main WebChat session, not only generic helper paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue a main-session system event or configure a main agent with a tool outside the PR's fixed list, then send a Control UI main chat turn; the proposed code drains or filters the behavior before the agent sees it.

## Solution Assessment

No. The current patch is not the narrowest maintainable fix because it hides reliability symptoms by suppressing agent context and hard-coding tool policy; the safer path is to keep runtime semantics intact and filter only UI/transcript presentation artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep drained system events in the main prompt:** `src/auto-reply/reply/get-reply-run.ts:704`
  - body: This suppresses `drainedSystemEventBlocks` for `agent:main:main` after `drainFormattedSystemEvents()` has already consumed the queue. Main-session cron, exec, heartbeat, and system notices can therefore disappear instead of reaching the agent, breaking the existing contract covered by `get-reply-run.media-only.test.ts`. Keep the events in the agent prompt and strip only the UI/history artifact you do not want displayed.
  - confidence: 0.91
- **[P2] Preserve configured tools for main WebChat:** `src/gateway/server-methods/chat.ts:2526`
  - body: Passing `webchatToolsAllow` into the run makes `applyEmbeddedAttemptToolsAllow()` exact-filter the available tools for the normal main Control UI session. That drops configured/profile/plugin tools such as `x_search`, `image_generate`, `video_generate`, `browser`, and bundle/plugin tools even when the user's config allows them. Avoid a hard-coded runtime allow-list here, or derive any narrowing from the existing tool policy without removing configured tools.
  - confidence: 0.88
- **[P2] Do not strip real user System lines from history:** `src/auto-reply/reply/strip-inbound-meta.ts:52-54`
  - body: Inbound user text that starts with `System:` is already neutralized to `System (untrusted):` so it remains visible as user content. This new generic leading-line stripper removes those same lines from display, so a user message like `System: [2026-01-01] fake event` can vanish from WebChat/history. Restrict the display cleanup to actual injected event lines, for example timestamped drained events, without deleting neutralized user text.
  - confidence: 0.84
