## Summary

Keep this PR open. The latest branch appears to address the earlier ClawSweeper findings around tool allowlists, reset transcript preservation, user-authored `System:` text, and the changelog, but it still introduces a blocking regression: queued main-session system events are drained and then suppressed from the prompt for `agent:main:main`, so real background/task/system notifications can be lost before the model sees them.

## What This Changes

The PR hardens Control UI/WebChat send continuity with preflight acknowledgements, pending-turn history, delayed disconnect cleanup, reset transcript handling, earlier compaction gates, prompt/tool plumbing, UI state updates, tests, and a changelog entry.

## Best Possible Solution

Keep the WebChat receipt and pending-history work only after preserving the existing system-event contract: real queued system events and new-session channel summaries must still reach the model, while any unwanted generated prompt noise should be filtered narrowly before destructive draining or carried through a separate non-dropping path.

## Reproduction Assessment

Yes. Mock or enqueue a system event for `agent:main:main`, run the PR-head `runPreparedReply` path, and observe that `drainFormattedSystemEvents` consumes the queue while `suppressSystemEventsInUserPrompt` prevents the returned `System:` block from reaching `commandBody` or `followupRun.prompt`.

## Solution Assessment

No. The branch's broad reliability direction is plausible, but dropping drained system events is not the narrow maintainable fix; suppress only the specific generated noise or keep drained system events in the prompt while preserving the prior tools/reset/System-text fixes.
