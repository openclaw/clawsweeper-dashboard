## Summary

Keep open: current main does not already contain the PR's central WebChat mechanisms, but the branch still needs current-head real WebChat proof and has a blocking duplicate-acknowledgment concern in the proposed prompt overlay.

## What This Changes

The PR adds WebChat preflight acknowledgements, pending user-turn history continuity, delayed disconnect cleanup, session reset/session-store/compaction fixes, prompt overlay updates, and focused tests.

## Best Possible Solution

Keep the server-side receipt and pending-history approach only after current-head real WebChat proof, and remove or scope the model-level acknowledgement so the receipt remains ephemeral.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main WebChat reproduction was established here; source inspection and related reports support the failure mode, but the supplied proof does not show the real runtime path.

## Solution Assessment

Is this the best way to solve the issue? No as currently proposed; the gateway/UI pending-history pieces are plausible, but the global GPT-5 acknowledgement should be removed or scoped so it cannot duplicate the ephemeral WebChat receipt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid prompting a second durable acknowledgment:** `src/agents/gpt5-prompt-overlay.ts:60-62`
  - body: The branch already sends an ephemeral `preflight` chat event and the UI avoids persisting that text on final. Adding this instruction to the global GPT-5 stable prefix tells the model itself to emit a visible acknowledgement for the same turn, so a normal GPT-5 WebChat send can show the ephemeral receipt and then stream/persist a second acknowledgement as assistant output. Keep this behavior in the gateway/UI receipt path or otherwise scope it away from the model prompt.
  - confidence: 0.84
