## Summary

Keep this PR open. The WebChat reliability direction is plausible and several earlier concerns appear addressed, but current PR head still introduces a prompt-event regression: main WebChat drains queued system events and then suppresses the drained block before it reaches the model.

## What This Changes

The PR adds WebChat preflight acknowledgements, pending user-turn history continuity, delayed Control UI disconnect cleanup, reset transcript selection fixes, earlier preflight compaction, prompt/tool plumbing, UI state updates, tests, and a changelog entry.

## Best Possible Solution

Keep the WebChat receipt, pending-history, reset, compaction, and UI reliability work only after preserving the existing system-event contract: real queued events and main-session channel summaries must still reach commandBody and followupRun.prompt, while any unwanted generated prompt noise should be filtered narrowly before destructive draining or represented through a non-dropping path.

## Reproduction Assessment

Yes. On PR head, run a focused runPreparedReply test with sessionKey agent:main:main and drainFormattedSystemEvents returning a System block; commandBody and followupRun.prompt will omit it because suppressSystemEventsInUserPrompt is true after the event queue has already been drained.

## Solution Assessment

No. The broad reliability fix is maintainable, but dropping all drained main-session system events is not the best fix; the safer path is to preserve event delivery and filter only the specific generated noise the PR intended to hide.
