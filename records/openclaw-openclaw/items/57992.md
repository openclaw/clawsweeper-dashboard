## Summary

Keep open. Current main has generic lifecycle error propagation for failed child runs, but the Docker sandbox create/start path still directly awaits container creation without targeted retry, gateway logging, or a registry/session failure record that would satisfy this Docker-specific zombie report.

## What This Changes

The issue requests that Docker sandbox container creation failures during `sessions_spawn` for non-main subagents be logged, surfaced in subagent/session status, and moved to an error state instead of leaving zombie runs.

## Best Possible Solution

Add focused sandboxed-subagent startup handling that logs sanitized Docker create/start failures, records a terminal failure visible to subagent/session tooling, and retries once only where sandbox isolation remains intact.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only pass because the reporter's Docker API failure was transient. The source path is reproducible by forcing Docker create/start failure after an accepted sandboxed native `sessions_spawn`.

## Solution Assessment

Yes, the best fix direction is narrow: make the existing sandboxed child startup either progress or fail visibly with Docker context. Disabling `non-main` sandboxing or changing the spawn contract would be a broader product change.
