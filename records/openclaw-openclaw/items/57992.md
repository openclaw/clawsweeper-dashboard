## Summary

Keep open. Current main has better generic lifecycle/error propagation for failed subagent runs than the original report describes, but the Docker sandbox creation path still has no create/start retry, no Docker-specific gateway log, no failed sandbox registry entry, and no targeted regression proving a transient container creation failure under `agents.defaults.sandbox.mode: "non-main"` is surfaced promptly rather than looking like a stuck run.

## What This Changes

The issue asks OpenClaw to make Docker sandbox container creation failures during sandboxed `sessions_spawn` visible, terminal, and preferably retried instead of leaving a phantom running subagent.

## Best Possible Solution

Add a deterministic regression for a native subagent spawned under `agents.defaults.sandbox.mode: "non-main"` where Docker create/start fails after the spawn is accepted, then make that failure produce a sanitized gateway/sandbox log entry, a persisted failed subagent outcome visible through `/subagents info` and session status, and no indefinite active run. A single retry should be limited to clearly transient Docker create/start failures and must not change sandbox policy, container privileges, or mount semantics.

## Reproduction Assessment

No live high-confidence reproduction is available because the reported Docker failure is transient. There is a clear deterministic reproduction path for a fix PR: mock the Docker create/start call or sandbox backend to fail after an accepted `sessions_spawn` and assert the child run terminalizes with a visible error.

## Solution Assessment

Yes, the requested direction is the right maintenance path: handle the failure at the sandbox backend and async subagent lifecycle boundary. The safest implementation is a narrow forced-failure regression plus sanitized logging/status propagation, with any retry constrained to Docker create/start rather than broad gateway retry behavior.
