## Summary

Keep open. Current main still accepts the subagent run before the async child runner resolves the Docker sandbox, and the Docker create/start path has no targeted retry, failure registry entry, or Docker-specific error surfacing for `subagents list`; related #39248 overlaps but does not cover this distinct container-creation failure requirement.

## What This Changes

The issue asks OpenClaw to log Docker sandbox container creation failures during sandboxed `sessions_spawn`, surface them in session/subagent status, fail the run promptly, and ideally retry once.

## Best Possible Solution

Add a focused regression that forces Docker create/start to fail during a sandboxed subagent run, then log a sanitized Docker error, persist a failed child run/session status visible to subagent tooling, and add a bounded create/start retry without weakening sandbox isolation.

## Reproduction Assessment

No live high-confidence reproduction is available from the current review because the reporter's Docker API failure was transient. The code provides a clear deterministic reproduction path by mocking Docker create/start failure after an accepted sandboxed `sessions_spawn`.

## Solution Assessment

Yes, the best path is a narrow sandboxed-subagent fix rather than closing as duplicate or stale. #39248 should stay linked for broader non-main sandbox startup coverage, but this issue should track Docker failure surfacing and retry behavior.
