## Summary

Keep open. Current main has some generic lifecycle error propagation, but the Docker create/start path still lacks targeted retry, gateway logging, sandbox registry failure state, and Docker-specific error detail in subagent list/status output.

## What This Changes

Ensure Docker sandbox container create/start failures during sandboxed `sessions_spawn` subagent startup are logged, surfaced in subagent/session state, and fail the run instead of leaving it running.

## Best Possible Solution

Add focused sandboxed-subagent startup handling that logs sanitized Docker create/start failures, persists terminal run/session state visible to subagent tooling, and retries once only where it preserves sandbox isolation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction was established because the reported Docker API failure was transient, but current source gives a deterministic mock path by forcing Docker create/start failure after an accepted sandboxed native `sessions_spawn`.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to make the existing sandboxed child startup either progress or fail visibly with Docker-specific context, not to disable `non-main` sandboxing or change the spawn contract.
