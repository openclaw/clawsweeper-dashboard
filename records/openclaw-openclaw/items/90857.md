## Summary

Keep open: current main only supports time-window inbound debounce and per-channel timing; the requested count-triggered early flush is not present in schema, docs, SDK/runtime debounce behavior, or a merged related item, so this needs maintainer product/API direction.

## What This Changes

Request adding a `messages.inbound.triggerCount` config that flushes a debounced inbound batch immediately after N messages while retaining `debounceMs` as the fallback timeout.

## Best Possible Solution

Have maintainers choose one canonical inbound aggregation contract; if count-trigger flush is accepted, add it through the existing `messages.inbound` schema/docs and shared debouncer/SDK path with tests for count flush, timer fallback, media/control bypass, and per-channel behavior.

## Solution Assessment

No, not yet: the proposed `triggerCount` option is plausible, but it is not proven as the best canonical contract given existing time-based debounce, collect queues, and related adaptive/aggregation proposals. Maintainers should decide the config/API shape before implementation.
