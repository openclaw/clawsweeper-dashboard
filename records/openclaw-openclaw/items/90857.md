## Summary

Keep open: current main and the latest release still implement only time-window inbound debounce, while the requested count-trigger behavior would add new public config and plugin-SDK semantics that need maintainer product/API direction. The related continuation-marker proposal at https://github.com/openclaw/openclaw/issues/9120 overlaps on multi-message batching, but it does not own this exact count-threshold trigger shape.

## What This Changes

Request a `messages.inbound.triggerCount` option that flushes inbound debounce immediately after a configured number of messages arrive within the debounce window.

## Best Possible Solution

Keep the time-only behavior as the default, then design one canonical generic inbound batching policy before adding `triggerCount` across config schema, SDK exports, docs, runtime, and focused channel tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a report of broken existing behavior. Source inspection confirms current main only supports timer-based inbound debounce.

## Solution Assessment

Unclear: `triggerCount` is a plausible shape, but it is not proven to be the best product/API answer because it overlaps with existing debounce, queue collect behavior, and the related continuation-marker proposal.
