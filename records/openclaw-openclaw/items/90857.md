## What This Changes

The issue requests `messages.inbound.triggerCount` so inbound debounce can flush immediately after a configured number of same-window messages instead of waiting only for `debounceMs`.

## Best Possible Solution

Design a generic debounce policy at the core/SDK boundary, keep the current time-only behavior as the default, and add schema, docs, and focused tests only after maintainers approve the new config surface.
