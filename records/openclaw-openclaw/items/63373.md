## Summary

Keep open: current main still lacks coverage for the reported uppercase `[TOOL_CALL]...[/TOOL_CALL]` transcript block shape. Existing fixes cover XML-style tool scaffolding, downgraded `[Tool Call: ...]` text, and a WebChat transcript-repair leak, but Telegram delivery still delegates to the plain-text outbound sanitizer, which only strips HTML-like internal tags and would not remove the exact non-JSON bracketed payload from the report.

## What This Changes

The report asks OpenClaw to prevent raw internal `[TOOL_CALL]` and `[TOOL_RESULT]` transcript blocks from reaching user-visible Telegram replies after transcript corruption or repair.

## Best Possible Solution

Add a shared user-visible sanitizer for uppercase square-bracket internal transcript blocks, cover the exact non-JSON Telegram payload from the report, and apply it on the shared assistant-visible and final outbound delivery boundaries rather than only inside Telegram.

## Reproduction Assessment

Yes for the sanitizer gap: source inspection shows the exact `[TOOL_CALL] {tool => ...} [/TOOL_CALL]` shape is not matched by current assistant-visible, bracketed-block, outbound, or Telegram sanitizers. The full live race that produced the malformed transcript remains unclear.

## Solution Assessment

Yes, the requested shared outbound guard plus repair-before-delivery direction is the narrowest maintainable solution. A Telegram-only filter would leave the same internal transcript marker family exposed on other messaging surfaces.
