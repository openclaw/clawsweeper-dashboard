## Summary

Keep open: current main still exposes managed skill paths outside the workspace while `tools.fs.workspaceOnly` guards `read` to the workspace root, and no canonical PR or narrower duplicate appears to own this exact skill-read gap. The remaining work changes a host filesystem isolation boundary, so it needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue requests that agents using `tools.fs.workspaceOnly: true` can still read prompt-listed managed skill files located outside the workspace.

## Best Possible Solution

Add an audited skill-root read mechanism for exactly the visible skill roots, preserving workspace write isolation and documenting the policy contract with regression coverage for managed, workspace, sandbox, and symlink cases.

## Reproduction Assessment

Yes. From source inspection, configure `tools.fs.workspaceOnly: true`, expose a managed skill under `~/.openclaw/skills`, and the prompt can tell the agent to read a path that the workspace guard rejects before the read tool runs.

## Solution Assessment

Unclear. The suggested allowlist, auto-exempt, or prompt-remap approaches could each work, but the best maintainable fix needs maintainer/security agreement on the filesystem isolation contract.
