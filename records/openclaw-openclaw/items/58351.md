## Summary

Keep open. Current main still has a real gap: shared managed skills can be listed in the prompt at paths outside the agent workspace, while `tools.fs.workspaceOnly: true` wraps the `read` tool with a workspace-root guard that rejects those paths. The existing sandbox skill-sync path only helps copied sandbox workspaces and does not provide a general read-only skill exception or prompt rewrite for host workspace-only agents.

## What This Changes

The issue requests that agents using `tools.fs.workspaceOnly: true` can still load shared `SKILL.md` files, or receive workspace-safe skill paths, without manually copying every skill into each workspace.

## Best Possible Solution

Add a generic, audited way for visible skill instructions to be read while preserving workspace write isolation. The maintainable direction is either to map/copy the exact visible skills into a workspace-safe location before emitting prompt paths, or to add a narrowly scoped read-only skill-root allowlist with realpath containment checks, config schema/docs, and focused tests for managed, workspace, and sandboxed agents.

## Reproduction Assessment

Yes. Configure an agent with `tools.fs.workspaceOnly: true`, make a managed skill visible from `~/.openclaw/skills/<name>/SKILL.md`, and ask the agent to read the prompt-listed skill path; current code expands `~` and rejects the path before the read because it is outside the workspace root.

## Solution Assessment

Unclear. The issue's `allowRead` and auto-exempt suggestions would address the symptom, but the safest final shape needs maintainer/security review because it changes what `workspaceOnly` means for shared host paths.
