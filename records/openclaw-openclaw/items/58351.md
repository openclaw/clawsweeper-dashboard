## Summary

Keep open: current main still has the reported gap where prompt-visible managed skill paths can live outside the agent workspace while `tools.fs.workspaceOnly` wraps `read` in a workspace-root guard. The recent `+1` comments and prior ClawSweeper review reinforce that multiple users are hitting the same behavior, and the remaining fix changes a filesystem isolation boundary that needs maintainer/security choice.

## What This Changes

Allow agents configured with `tools.fs.workspaceOnly: true` to read visible shared skill files without copying every skill into each workspace.

## Best Possible Solution

Add an audited path mapping or narrow read-only exception for exactly the visible skill roots while preserving workspace write isolation, then document and test managed, workspace, and sandboxed agent cases.

## Reproduction Assessment

Yes. From source inspection, configure an agent with `tools.fs.workspaceOnly: true`, expose a managed skill from `~/.openclaw/skills/<name>/SKILL.md`, and ask the agent to read the prompt-listed location; the guard rejects that path as outside the workspace root before `read` runs.

## Solution Assessment

Unclear. The suggested `allowRead` or auto-exempt approaches would address the symptom, but the best maintainable fix needs maintainer/security agreement on whether to use a read allowlist, prompt path remap, or workspace copy strategy.
