## Summary

Keep open: current main still exposes prompt-visible shared skill paths while `tools.fs.workspaceOnly` wraps host `read` in a single workspace-root guard. Multiple users have confirmed the impact, and the fix changes a filesystem isolation boundary that needs maintainer/security choice.

## What This Changes

Allow agents using `tools.fs.workspaceOnly: true` to read prompt-advertised shared skill files without copying them into every workspace.

## Best Possible Solution

Add a maintainer-approved narrow path mapping or read-only exception for exactly prompt-visible skill roots while preserving workspace write isolation, realpath containment, and coverage for host and sandboxed runs.

## Reproduction Assessment

Yes. By source inspection, configure a host agent with `tools.fs.workspaceOnly: true`, expose a managed skill from `~/.openclaw/skills/<name>/SKILL.md`, and ask the agent to read the prompt-listed location; current code emits that location and guards `read` against only the workspace root.

## Solution Assessment

Unclear. The suggested approaches would address the symptom, but the maintainable solution needs maintainer/security agreement on a read allowlist, prompt remap/copy strategy, or broader filesystem ACL contract.
