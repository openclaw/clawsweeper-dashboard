## Summary

Keep open for maintainer review: the PR addresses a real current-main Telegram recovery gap and outbound gateway timeout mismatch with focused tests and sufficient redacted live output, with no line-level correctness finding found.

## What This Changes

The PR classifies Telegram Bot API 5xx responses as recoverable outside send context, raises the outbound message gateway default timeout to 75s, adds focused tests, and updates Telegram troubleshooting docs.

## Best Possible Solution

Land the classifier and timeout alignment after CI and explicit maintainer acceptance of the broader 75s outbound gateway default, while preserving conservative non-idempotent send retries.

## Reproduction Assessment

Yes, source-level: current main preserves getUpdates error_code, then exits the isolated polling worker when the classifier does not treat a 502 as recoverable; current main also times outbound gateway callers at 10s while Telegram sends can run for 60s. I did not run a live Telegram 502 replay in this read-only review.

## Solution Assessment

Yes, with one maintainer-visible default decision: the classifier is the narrow owner boundary for Telegram retry classification, and the timeout change aligns the gateway caller with bounded provider sends. The safer alternative, if maintainers want old fail-fast behavior elsewhere, is to split the timeout by outbound operation rather than changing the shared helper default.
