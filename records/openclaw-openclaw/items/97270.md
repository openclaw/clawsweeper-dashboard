Keep open: this is a focused Telegram reliability fix that is not already on current main and the central behavior is supported by source inspection. The patch is small, keeps non-idempotent sends on the conservative boundary, and I found no blocking correctness or security findings.

## What This Changes

The branch classifies Telegram Bot API 5xx responses as recoverable outside send context, adds classifier regression tests, and documents Bad Gateway polling/startup troubleshooting.

## Best Possible Solution

Land the narrow classifier, test, and docs change after normal CI and maintainer review, while preserving the existing safe-send retry boundary for visible messages.

## Reproduction Assessment

Yes, at source level: current main converts getUpdates 502 responses into errors with error_code, then the isolated worker exits if the classifier does not treat them as recoverable. I did not induce a live Telegram 502 during this read-only review.

## Solution Assessment

Yes. The central classifier is the right narrow layer because polling, webhook/setup, and unknown control-plane callers already share it, and the PR explicitly keeps send context conservative instead of adding worker-only retry logic.
