## Summary

Keep open. The classifier change is the right narrow runtime surface for Telegram polling/setup recovery, but the new troubleshooting bullet overstates command-menu retry behavior, which is not wired through this classifier.

## What This Changes

The branch makes Telegram Bot API 5xx errors recoverable for non-send contexts, adds classifier regression tests, and documents Bad Gateway retry behavior.

## Best Possible Solution

Keep the classifier/test fix, and revise the Telegram troubleshooting bullet so it names only the runtime paths that actually retry 5xx unless command-menu retry support is added with tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main preserves Bot API error_code 502 in the worker, but the recoverability classifier does not treat it as retryable, so the worker throws. I did not induce a live Telegram 5xx during review.

## Solution Assessment

Mostly yes: the central classifier is the narrowest maintainable code surface and keeps send context conservative. The docs portion is not yet the best solution because it claims command-menu retry behavior the code does not implement.
