## Summary

Keep open. The report is fresh and now has strong profiling evidence tying the TUI CPU spin to repeated model discovery/provider plugin normalization, while current main still lacks the proposed normalization cache. Open PR #75503 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for the OpenClaw TUI high-CPU busy loop to be fixed, especially around repeated model/plugin resolution during TUI startup and context-window warmup.

## Best Possible Solution

Use #75503 as the active implementation path: review and land a narrow cache or equivalent fix that prevents repeated provider/plugin manifest resolution during model context warmup, then re-check idle TUI CPU and leave shutdown/orphan handling to the separate TUI signal issues.

## Reproduction Assessment

Yes. The issue has high-confidence reproduction data from Linux and macOS plus profiling that maps to the current-main code path; this read-only review did not live-run the TUI, but the source path still matches the reported hot loop.

## Solution Assessment

Yes as a direction, pending PR review. Caching `normalizeDiscoveredPiModel` or otherwise reusing provider/plugin resolution results is the narrow maintainable fix for the profiled hot path; broader registry caching can remain a follow-up if residual CPU remains.
