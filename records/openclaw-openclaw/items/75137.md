## Summary

## What This Changes

Report that `openclaw-tui` pegs a CPU core at idle or during turns because context/model/plugin discovery work repeats during TUI startup and rendering.

## Best Possible Solution

Keep the current main warmup-specific normalization bypass; track separate TUI orphan/exit lifecycle behavior in its own linked reports.

## Reproduction Assessment

Yes. The issue discussion provides concrete `openclaw tui` and `openclaw chat` repros with CPU samples plus native, async, and filesystem traces; the current-main source path maps directly to that context-window warmup chain.

## Solution Assessment

Yes. The main fix is narrower than caching all model normalization: context-window warmup only needs context metadata, so disabling full provider/plugin normalization there removes the expensive hook path while preserving provider-owned token metadata.
