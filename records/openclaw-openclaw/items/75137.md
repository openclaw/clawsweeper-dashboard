## Summary

Keep open. The report is fresh, concrete, and still matches current main: model discovery still normalizes each discovered model through plugin hooks without the cache proposed by the open closing PR, and that PR has not merged yet. The issue comments add useful profiling that narrows the hot path from TUI rendering to repeated model/plugin normalization work.

## What This Changes

The issue asks OpenClaw to stop `openclaw-tui` from consuming roughly a full CPU core while idle or during message turns in `openclaw chat` / `openclaw tui`.

## Best Possible Solution

Keep this issue as the active tracker until #75503 or an equivalent narrow fix lands, with a regression test proving repeated model normalization does not re-enter plugin resolution for the same discovered model objects. Treat TUI orphan/shutdown behavior in #75289 and broader plugin-loader gateway loops in #73532 as related but separate follow-ups unless the merged fix proves they share the same remaining root cause.

## Reproduction Assessment

Yes. The issue gives exact `openclaw chat` steps and process CPU evidence, and follow-up comments add macOS/Linux-style profiling, fs tracing, and a caller chain through model discovery and plugin manifest loading.

## Solution Assessment

Yes, the currently linked fix direction appears to be the narrowest maintainable path: cache repeated `normalizeDiscoveredPiModel` results at the model-discovery boundary rather than changing the TUI render loop. It still needs normal PR review and merge before this issue should close.
