## Summary

Keep open. The report is fresh, concrete, and still maps to current main: `chat` is still the local embedded TUI entry point, the TUI still has active timer/render/shutdown surfaces, and the open #75294 work targets terminal-hangup shutdown for #75289 rather than proving the idle open-window CPU loop is fixed.

## What This Changes

The issue asks for the local TUI process launched by `openclaw chat` or `openclaw tui --local` to stop pegging CPU while idle or during turns.

## Best Possible Solution

Keep this as the active local TUI idle-CPU tracker, collect `perf top`, `strace -f`, or a V8 CPU profile from an idle affected `openclaw-tui`, then fix the narrowest proven loop in `src/tui`, `src/tui/embedded-backend.ts`, or the embedded plugin/runtime path. Review #75294 for the related shutdown/orphan case, but do not treat it as sufficient for this idle-window report without profiling or regression proof.

## Reproduction Assessment

Yes for the reported environment: the issue gives exact `openclaw chat` steps plus `top`/`ps` proof for the `openclaw-tui` process. It was not independently reproduced on current main during this read-only review, so profiling evidence is still needed before a targeted fix.

## Solution Assessment

No implemented solution is proven on current main. The best next step is to profile the idle TUI process and separate any terminal-hangup shutdown fix from the open-window idle CPU loop.
