## Summary

Keep open. The report is fresh, concrete, and still has an open closing PR (#75503); current main does not include the proposed normalization cache, and the linked PR should be reviewed and merged or rejected before this issue is closed.

## What This Changes

The issue reports that `openclaw-tui` can peg a CPU core at idle or during TUI startup/turns because repeated context-window/model discovery work re-enters plugin manifest/provider resolution.

## Best Possible Solution

Land #75503 or an equivalent narrow fix that de-duplicates per-model provider/plugin normalization during TUI context-window warmup, verify both local and gateway TUI CPU settles near idle, and then let the closing PR close this issue.

## Reproduction Assessment

Yes. The issue has exact TUI commands and `top`/`ps` output, and follow-up comments add native samples, fs traces, and async stack traces that identify the repeated model/plugin resolution path.

## Solution Assessment

Yes, the linked cache direction appears to be the narrowest maintainable fix for this specific hot path because it avoids broad TUI render-loop changes and preserves plugin/provider contracts. It still needs normal PR review and merge proof before the issue should close.
