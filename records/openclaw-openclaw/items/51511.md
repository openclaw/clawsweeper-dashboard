## Summary

Keep this open. Current main still exposes Cursor as an ACP harness target in the system prompt and bundled ACP router when ACP spawning is available, while the tests cover positive ACP routing and availability gating rather than the reported negative case of an incidental Cursor link staying in the existing custom Feishu session.

## What This Changes

The issue requests that Cursor-related text or links must not override a custom Feishu agent session or clear its conversation context unless the user explicitly opts into Cursor/ACP routing.

## Best Possible Solution

Add an explicit-intent boundary for ACP harness routing: incidental Cursor links or text in an existing custom-agent conversation should remain in the current session, while requests that clearly say to run, use, open, spawn, bind, resume, or continue work in Cursor/ACP should keep the current ACP path. The fix should preserve current slash-command and explicit ACP behavior while adding regression coverage for the negative Cursor-link case.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the current evidence because the report has no logs and I did not run a live Feishu/ACP session. There is a high-confidence source-level risk path: current prompt and skill text still allow Cursor ACP routing when ACP is available, and no negative regression test covers incidental Cursor URLs.

## Solution Assessment

Unclear for the reporter's suggested opt-in config because removing Cursor ACP from default routing would be broader than necessary and could break explicit harness workflows. The narrower maintainable solution is to guard ACP routing on explicit user intent and test that incidental Cursor references do not spawn or bind ACP sessions.
