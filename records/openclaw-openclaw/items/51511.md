## Summary

Keep this open: current main still exposes Cursor as an ACP harness target through the agent prompt and bundled ACP router, and the available tests prove positive ACP routing/gating rather than the reported negative case where an incidental Cursor link stays in the existing custom Feishu session.

## What This Changes

The issue reports a custom Feishu agent session being taken over by Cursor/ACP support after a Cursor-related GitHub link is pasted.

## Best Possible Solution

Add an explicit-intent boundary for ACP harness routing so incidental Cursor links or mentions remain in the existing custom-agent conversation, while clear requests to run, bind, spawn, resume, or continue work in Cursor/ACP keep the current ACP path.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available because the report has no logs and I did not run a live Feishu/ACP session. The source-level risk path is clear: Cursor ACP routing is still present and no negative test covers incidental Cursor URLs or text.

## Solution Assessment

No for the broad suggested opt-in/removal as the default repair, because current docs and tests intentionally support explicit ACP harness routing. The narrower maintainable fix is an explicit-intent guard plus regression coverage for the incidental Cursor-link case.
