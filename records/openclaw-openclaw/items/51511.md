## Summary

Keep this open: current main still advertises Cursor as an ACP harness route in the agent prompt and bundled ACP router, while the tests/docs cover explicit ACP routing rather than the reported negative case where an incidental Cursor link stays in the existing custom Feishu session.

## What This Changes

The issue asks OpenClaw to prevent Cursor/ACP support routing from taking over a custom Feishu agent session when a Cursor-related link or keyword is mentioned.

## Best Possible Solution

Add an explicit-intent boundary for ACP harness routing so incidental Cursor links or mentions remain in the existing custom-agent conversation, while clear requests to run, bind, spawn, resume, or continue work in Cursor/ACP keep the supported ACP path.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available because the report lacks logs and I did not run a live Feishu/ACP session. Source inspection does show the remaining risk path: Cursor ACP routing is still prompt/skill-visible and no negative test covers incidental Cursor URLs or text.

## Solution Assessment

No for the broad suggested removal or config switch as the default repair, because current docs intentionally support explicit ACP harness routing. The narrower maintainable fix is an explicit-intent guard plus regression coverage for the incidental Cursor-link case.
