## Summary

Keep open. Current main still preserves Claude CLI no-output/overall timeouts as structured timeout failures, but the non-verbose external-chat reply path collapses them into the generic Telegram-style error instead of surfacing the CLI timeout class.

## What This Changes

The issue asks Telegram/status surfaces to distinguish Claude CLI no-output timeouts from generic external-chat failures.

## Best Possible Solution

Add a narrow timeout classification path for known CLI FailoverError timeout failures in external-chat and diagnostic surfaces, while leaving timeout-default policy changes out of this bug fix.

## Reproduction Assessment

Yes. A focused unit test can reject the agent run with a claude-cli FailoverError(reason=timeout) in a non-verbose external chat context and observe that current main returns the generic failure copy.

## Solution Assessment

Yes for the narrow fix: classify typed CLI timeout failures before the generic raw-error fallback and keep unknown errors generic. Changing default timeout values or Opus/resume policy would be a separate product decision.
