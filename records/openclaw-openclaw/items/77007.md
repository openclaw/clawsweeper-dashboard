## Summary

Keep open. Current main still preserves Claude CLI timeout failures as structured timeout errors, but normal external-chat replies only surface details in verbose mode and otherwise fall back to the generic Telegram-style failure copy; the linked open PR #77015 explicitly targets this issue and should be reviewed or landed before closing.

## What This Changes

The issue asks OpenClaw to surface Claude CLI no-output and overall timeout failures distinctly in Telegram/status instead of collapsing them into generic channel error text.

## Best Possible Solution

Review and land or close #77015; the narrow maintainable fix is to classify OpenClaw-generated CLI timeout templates before the generic external-chat fallback and cover it with focused regression tests.

## Reproduction Assessment

Yes. A focused unit test can make a non-verbose external-chat agent run fail with a Claude CLI FailoverError timeout message and observe current main returning the generic failure copy.

## Solution Assessment

Yes for the narrow fix. Classifying known internal CLI timeout failures before the generic fallback preserves unknown-error safety while avoiding a broad timeout-default or diagnostics redesign.
