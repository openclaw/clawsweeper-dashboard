## Summary

Keep open. Current main still preserves CLI timeout failures upstream but collapses them to generic external-chat copy when verbose failure details are off; linked PR #77015 is an open implementation candidate and should be reviewed or landed before closing this issue.

## What This Changes

Surface Claude CLI no-output and overall timeout failures in Telegram/external chat instead of replacing them with generic runner failure text.

## Best Possible Solution

Land a narrow fix like #77015 that classifies known CLI timeout failures before the generic external-chat fallback, then track broader diagnostics or timeout-default policy separately if maintainers want it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: reject an external chat agent run with a CLI timeout message while verbose details are off, and current main routes it through the generic fallback.

## Solution Assessment

Yes for the narrow bug. Classifying OpenClaw-generated CLI timeout strings before the generic fallback is the smallest maintainable fix; changing watchdog defaults or adding broader diagnostics should be separate work.
