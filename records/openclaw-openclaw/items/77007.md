## Summary

Keep open. Current main still routes Claude CLI timeout details into the generic external-chat failure copy when verbose mode is off, even though the CLI runner preserves the timeout as a structured `FailoverError`; #75264 is related but tracks a more specific Telegram + CLI tool-stall symptom.

## What This Changes

The issue requests clearer Telegram/status diagnostics for Claude CLI no-output timeouts instead of a generic channel error.

## Best Possible Solution

Keep this open and add a narrow timeout classification path that surfaces CLI timeout context in existing external-chat and diagnostic surfaces without changing default timeout policy.

## Reproduction Assessment

Yes, by source inspection and existing unit-test shape: a CLI timeout becomes a FailoverError(reason=timeout), then the external failure builder returns the generic message when verbose is off. A focused regression test can reproduce it by rejecting the agent run with a claude-cli timeout FailoverError.

## Solution Assessment

Yes for a narrow fix: classify typed CLI timeout failures before the generic raw-error fallback and keep unknown errors generic. Changing default timeout values would be a separate product decision and should not be bundled into the bug fix.
