## Summary

Keep open. Current main still preserves Claude CLI no-output/overall timeouts as structured timeout failures, but the non-verbose external-chat path collapses them into the generic user-facing error; #75264 is related but narrower.

## What This Changes

The issue requests that Claude CLI no-output timeouts surface as distinct, actionable Telegram/status diagnostics instead of the generic external-chat failure message.

## Best Possible Solution

Add a narrow typed timeout classification path for CLI backend failures in external-chat replies, and leave timeout-default policy changes as a separate maintainer decision if still desired.

## Reproduction Assessment

Yes by source inspection: a claude-cli no-output timeout becomes a FailoverError(reason=timeout), then the non-verbose external-chat failure builder returns the generic message. A focused unit test can reproduce it by rejecting the agent run with that timeout FailoverError.

## Solution Assessment

Yes for the narrow bug: classify typed CLI timeout failures before the generic raw-error fallback and keep unknown errors generic. Changing default timeout values would be a separate product decision and should not be bundled into the bug fix.
