## Summary

Keep open. Current main still collapses Claude CLI timeout failures into the generic external-chat reply, and open PR #77015 already proposes a closing fix, so this issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

Request clear Telegram/status messaging for Claude CLI no-output or overall turn timeouts instead of the generic external-chat failure copy.

## Best Possible Solution

Land a narrow external-chat classifier for OpenClaw-generated CLI timeout errors with regression coverage, and leave timeout-default/status UX policy changes to follow-up if maintainers want them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reject the embedded run with a Claude CLI timeout FailoverError in a non-verbose external chat context and current main returns GENERIC_EXTERNAL_RUN_FAILURE_TEXT.

## Solution Assessment

Yes for the narrow fix. Classify known OpenClaw-generated CLI timeout strings before the generic fallback and keep raw unknown failures behind verbose mode; broader diagnostics/default timeout changes are separate product work.
