## Summary

Keep open. Current main still preserves Claude CLI timeout failures as timeout-shaped errors, but the normal non-verbose external-chat reply path collapses them into generic failure copy, and #77015 is an open closing PR for the narrow fix.

## What This Changes

The issue asks OpenClaw to surface Claude CLI no-output and turn timeouts distinctly in Telegram/status instead of the generic external-chat failure message.

## Best Possible Solution

Land or revise #77015 to add a safe typed timeout reply for internally generated Claude CLI timeout failures, then split any broader diagnostics or timeout-default policy work into focused follow-ups.

## Reproduction Assessment

Yes, source-level. A focused unit test can reject the agent run with a claude-cli timeout FailoverError or fallback summary in a non-verbose external-chat context and observe current main returning the generic failure text.

## Solution Assessment

Yes for the narrow bug. Classifying known internally generated CLI timeout failures before the generic external-chat fallback is the smallest maintainable fix; changing timeout defaults or broader status surfaces should stay separate.
