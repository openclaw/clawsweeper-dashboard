## Summary

Keep open. Current main still preserves Claude CLI timeouts as structured timeout failures but collapses them to generic external-chat copy when verbose details are off, and open PR #77015 is already the closing implementation candidate.

## What This Changes

The issue asks Telegram/external-chat replies for claude-cli no-output and overall timeouts to show an actionable timeout class instead of the generic failure text.

## Best Possible Solution

Land or revise PR #77015, or an equivalent narrow patch, to classify known claude-cli timeout failures before the generic external-chat fallback and cover it with regression tests.

## Reproduction Assessment

Yes, source-backed. A focused unit test can reject the agent run with a claude-cli FailoverError timeout in a non-verbose external-chat context and observe current main returning the generic failure copy.

## Solution Assessment

Yes for the narrow bug fix. The maintainable path is to classify typed/internal CLI timeout messages before the generic fallback while leaving timeout-default policy and broader diagnostics to separate follow-up work.
