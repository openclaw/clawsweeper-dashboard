Full review comments:

- **[P2] Forward caller aborts through guarded browser fetch:** `extensions/browser/src/browser/client-fetch.ts:190-191`
  - body: `fetchWithSsrFGuard` builds the actual request signal from its top-level `signal` option and overwrites `init.signal` when `timeoutMs` is present.   - confidence: 0.9
- **[P1] Avoid reintroducing the generic Telegram SDK facade:** `src/plugin-sdk/telegram.ts:25-26`
  - body: Current main removed `src/plugin-sdk/telegram.ts` and only lists narrower Telegram SDK entrypoints. Adding promotion exports through this private facade expands the plugin API surface and conflicts with the current SDK boundary; use the setup contract or a maintainer-approved narrow entrypoint instead.
  - confidence: 0.89
- **[P2] Use the narrow account-id facade in setup contracts:** `extensions/telegram/src/setup-contract.ts:1`
  - body: `shared.ts` imports this setup contract on the Telegram runtime setup path, and `openclaw/plugin-sdk/setup` re-exports setup wizard/helper modules. This code only needs `DEFAULT_ACCOUNT_ID` and `normalizeAccountId`, so importing from the narrow routing facade preserves the hot-path boundary.
  - confidence: 0.86
- **[P2] Sync unloaded Telegram promotion fallback keys:** `extensions/telegram/src/setup-contract.ts:7-14`
  - body: The PR expands Telegram promotion keys to include token/account/webhook/streaming fields, but the PR-head unloaded setup fallback still only allows `botToken` and `tokenFile`. Setup paths that use that fallback can leave newly declared per-account transport fields at the root instead of moving them with the promoted account.
  - confidence: 0.78
