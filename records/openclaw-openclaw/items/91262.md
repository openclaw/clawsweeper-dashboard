## Summary

Keep open: current main still passes build-all TypeScript node steps through `--experimental-strip-types` unchanged, while this PR adds a narrow fallback for Node builds where native TypeScript stripping is unavailable. The branch remains useful, has focused regression coverage and sufficient real-host proof, and I found no blocking patch defect.

## What This Changes

The PR adds a build-all helper that rewrites eligible Node TypeScript script steps to `node --import tsx` when the current Node binary lacks native TypeScript support, plus focused regression tests.

## Best Possible Solution

Land the narrow build-all fallback after ordinary maintainer review and current-head validation; keep direct package-script parity separate unless maintainers decide those commands need the same compatibility behavior.

## Reproduction Assessment

Yes. Current main passes build-all `.ts` node steps through native strip-types unchanged, and the PR body supplies real Debian/Node proof where `process.features.typescript` is false; this read-only review did not rerun the failing build locally.

## Solution Assessment

Yes for the reported `pnpm build` failure. The fallback belongs in `resolveBuildAllStep`, which owns build-all command selection, while rewriting direct package scripts would be broader and should remain separate unless maintainers want that scope.
