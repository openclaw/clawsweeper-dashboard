## Summary

Keep open because current main still implements the reported Discord behavior. A non-empty `guilds.*.channels` map makes unmatched channels resolve to a denied channel config, and message preflight drops those messages before guild-level `requireMention` inheritance can apply; the related items are broader, pattern-specific, or already closed around the opposite fail-closed behavior.

## What This Changes

Change Discord guild `channels` config so listed entries act as additive per-channel overrides that inherit guild defaults instead of making unlisted channels implicitly denied.

## Best Possible Solution

Keep this open as a Discord plugin access-control design task: separate additive channel overrides from explicit channel allowlisting, define compatibility semantics for explicit deny and wildcard/default entries, and update runtime, doctor/security guidance, docs, and regression coverage across Discord ingress paths together.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance matches the requested UX, but applying it globally is not clearly the safest fix because existing configurations may rely on omission-as-deny; a compatibility mode, explicit deny/default entry, or migration path needs maintainer direction.
