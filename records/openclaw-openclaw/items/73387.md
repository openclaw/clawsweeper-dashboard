## Summary

Keep open. Current main still resolves unmatched Discord guild channel entries to a denied channel config, and the message preflight, voice, reaction, and native-command paths still reject that result, so the requested additive override behavior is not implemented. Related #69748 and #8972 are broader follow-ups, while closed PR #54353 pursued stricter allowlist behavior and was not merged.

## What This Changes

Change Discord guild `channels` configuration semantics so listed channels can act as per-channel overrides while unlisted channels inherit guild-level defaults instead of being implicitly denied.

## Best Possible Solution

Keep the fix in the bundled Discord plugin and make the semantics explicit: unlisted channels should inherit guild defaults for the reported open-guild override use case, while explicit `enabled: false`, strict allowlist-mode denial, and existing native command, reaction, voice, component, and preflight authorization guarantees remain covered by docs and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance is the narrow solution for the reported override use case, but applying it to all non-empty channel maps would be a security-sensitive compatibility change; the safer design is to separate override semantics from strict allowlist/explicit-deny semantics.
