## Summary

Keep open because current main still implements the documented implicit allowlist for non-empty Discord guild channel maps, and changing it would alter Discord access-control behavior for existing deployments.

## What This Changes

The issue requests Discord guild `channels` entries to behave as additive per-channel overrides so omitted channels inherit guild-level defaults instead of being denied.

## Best Possible Solution

Keep this as a Discord plugin access-control design task that separates per-channel overrides from explicit channel allowlisting, with compatibility handling, docs, security guidance, and regression coverage updated together.

## Reproduction Assessment

Yes. A source-level reproduction is a guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it before mention policy inheritance.

## Solution Assessment

Unclear. Additive inheritance solves the reported configuration UX, but a direct global semantic flip is not clearly the best fix because current documented behavior can be used as an access-control allowlist.
