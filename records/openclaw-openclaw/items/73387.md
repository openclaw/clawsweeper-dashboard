## Summary

Keep open. Current main still treats a non-empty Discord guild `channels` map as an implicit channel allowlist: unmatched channels resolve to `{ allowed: false }` and guild preflight drops them before mention/default inheritance can apply. Docs and tests still document/assert that behavior, while the related open issues are broader policy features and the related PR was closed unmerged.

## What This Changes

Request to make Discord guild `channels` entries additive per-channel overrides so omitted channels inherit guild-level defaults instead of being silently denied.

## Best Possible Solution

Define an explicit Discord access model that separates per-channel overrides from channel allowlisting: omitted channels can inherit guild defaults for the open-guild use case, while explicit `enabled: false`, wildcard entries, strict allowlist behavior, docs, doctor guidance, and regression coverage stay clear across messages, voice, native commands, and components.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance matches the requested UX, but applying it globally is not obviously the safest fix because existing configs may depend on omission-as-deny; a maintainer-approved compatibility mode or migration path is safer.
