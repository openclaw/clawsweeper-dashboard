## Summary

Keep open. Current main still implements the documented Discord `channels` block as an implicit channel allowlist, and the related open issues are broader access-control designs while the related PR is closed unmerged. The requested semantic change can broaden Discord guild ingress, so it needs maintainer access-control direction rather than conservative cleanup closure.

## What This Changes

The issue requests changing Discord guild `channels` entries from an implicit allowlist into additive per-channel overrides where non-listed channels inherit guild-level defaults.

## Best Possible Solution

Define a Discord access model that separates channel allowlisting from per-channel overrides, with explicit deny or strict-allowlist compatibility preserved, then update message, voice, reaction, native-command, docs, doctor guidance, and regression coverage together.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance solves the reported UX, but making it the global meaning of `channels` is not clearly the safest fix because current configs may rely on omitted channels as deny entries; an opt-in mode, migration, or separate override key is safer until maintainers choose the compatibility policy.
