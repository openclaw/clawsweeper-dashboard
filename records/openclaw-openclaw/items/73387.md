## Summary

Keep open because current main still implements Discord `guilds.*.channels` as an implicit channel allowlist: an unmatched channel in a non-empty channel map resolves to denied and is dropped before guild-level mention defaults can apply. The related open items cover broader access-list redesign and pattern matching, while the closed PR tracked the opposite fail-closed behavior, so this compatibility/security decision remains unresolved.

## What This Changes

Change Discord guild `channels` config so listed entries act as additive per-channel overrides that inherit guild defaults instead of denying unlisted channels.

## Best Possible Solution

Define a Discord-owned access-control model that separates additive channel overrides from explicit channel allowlisting, then update runtime, schema/types, docs, doctor/security guidance, and regression coverage across all Discord ingress paths together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord guild with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance addresses the reported configuration UX, but applying it globally is not clearly the safest fix without maintainer-approved compatibility behavior for existing omission-as-deny configs and explicit deny/default entries.
