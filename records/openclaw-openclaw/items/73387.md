## Summary

Keep open because current main still denies omitted Discord guild channels when a non-empty `guilds.*.channels` map exists, including before guild-level mention defaults can apply; changing that behavior would be a security-sensitive compatibility change.

## What This Changes

Request that Discord guild `channels` entries behave as additive per-channel overrides instead of making all unlisted channels implicitly denied.

## Best Possible Solution

Define an explicit compatibility-safe Discord channel access model that preserves deliberate allowlisting while adding documented additive override semantics where maintainers want inheritance.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance solves the reported configuration UX, but the safest maintainable fix likely needs an explicit mode, migration path, or compatibility guard rather than silently changing all existing `channels` maps.
