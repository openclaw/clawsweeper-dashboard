## Summary

Keep open because current main still implements the reported Discord omission-as-deny behavior: a non-empty guild `channels` map turns unmatched channels into denied channel config, and message preflight drops them before guild-level `requireMention` inheritance can apply. The related open issues are broader access-control designs, and the related PR was closed unmerged, so this specific semantic change remains unresolved and needs maintainer direction because it can widen Discord ingress.

## What This Changes

The issue asks for Discord `guilds.*.channels` entries to behave as additive per-channel overrides so omitted channels inherit guild defaults instead of being implicitly denied.

## Best Possible Solution

Separate per-channel overrides from explicit channel allowlisting in the Discord config model: make inheritance, explicit deny, wildcard/default channel entries, docs, doctor/security guidance, and regression coverage consistent across message, voice, reaction, native-command, and component ingress.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance matches the requested UX, but applying it globally is not clearly the safest fix because existing configurations may rely on omission-as-deny; a maintainer-approved compatibility mode or migration path is safer.
