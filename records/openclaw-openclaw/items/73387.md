## Summary

Keep open. Current main still treats a non-empty Discord guild `channels` map as an implicit channel allowlist: unmatched channels resolve to denied channel config and guild preflight drops them before guild-level `requireMention` inheritance can matter. The related open items are broader Discord access-control designs, and the related PR was closed unmerged, so this specific semantic change remains unresolved.

## What This Changes

The issue asks for Discord guild `channels` entries to behave as additive per-channel overrides so omitted channels inherit guild-level defaults instead of being silently denied.

## Best Possible Solution

Define a Discord access model that separates per-channel overrides from channel allowlisting: omitted channels can inherit guild defaults for open-guild setups, while explicit denial, wildcard entries, strict allowlist behavior, docs, doctor guidance, and regression coverage remain clear across messages, voice, reactions, components, and native commands.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance matches the requested UX, but applying it globally is not obviously the safest fix because existing configs may depend on omission-as-deny; a maintainer-approved compatibility mode or migration path is safer.
