## Summary

Keep open because current main still implements the reported Discord omission-as-deny behavior. A non-empty `guilds.*.channels` map resolves unmatched channels to a denied channel config, and message preflight drops those messages before guild-level `requireMention` inheritance can apply; related items are broader or closed unmerged, so this specific compatibility/security decision remains unresolved.

## What This Changes

Change Discord guild `channels` config so entries act as additive per-channel overrides that inherit guild defaults instead of making unlisted channels an implicit allowlist denial.

## Best Possible Solution

Separate per-channel overrides from explicit channel allowlisting in the Discord config model, with maintainer-approved compatibility semantics for explicit deny, wildcard/default channel entries, migration or doctor guidance, docs, and regression coverage across all Discord ingress paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance matches the requested UX, but applying it globally is not clearly the safest fix because existing configurations may rely on omission-as-deny; a compatibility mode, explicit deny/default entry, or migration path needs maintainer direction.
