## Summary

Keep open. Current main still treats a non-empty Discord guild `channels` map as an implicit channel allowlist: unmatched channels resolve to a denied config and are dropped in preflight, while docs and tests still describe/assert that behavior. The related open items are broader policy features, and the closed related PR was not merged, so this specific semantic change remains unresolved.

## What This Changes

Request Discord guild `channels` entries to act as additive per-channel overrides so unlisted channels inherit guild-level defaults instead of being denied.

## Best Possible Solution

Define an explicit Discord access model that separates per-channel overrides from channel allowlisting: unlisted channels can inherit guild defaults for the open-guild use case, while `enabled: false`, strict allowlist behavior, docs, doctor guidance, and regression coverage remain clear across messages, voice, reactions, components, and native commands.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear. Additive inheritance is the narrow behavior requested here, but applying it globally is not obviously the safest fix because existing configs may depend on omission-as-deny; a maintainer-approved compatibility mode or migration path is safer.
