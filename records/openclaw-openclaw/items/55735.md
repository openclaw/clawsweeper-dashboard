## Summary

Keep open. Current main has useful generic guardrails for checking docs, source, config schema, status, and mutable version facts, but it does not implement the requested version-aware self-knowledge, agent-readable change notes, general `openclaw explain` surface, or known-behavior registry.

## What This Changes

Request to make OpenClaw agents diagnose post-upgrade behavior with current-version awareness through injected change notes, an explain command, or an authoritative known-behavior registry.

## Best Possible Solution

Define a core, authoritative diagnostic path for upgrade questions: agents should identify the running OpenClaw version, consult current docs/source/config schema plus a maintainer-owned release/change-note source, and expose any topic explanation or behavior registry only after the command/data model is decided.

## Reproduction Assessment

Not applicable as a feature request rather than a single bug. The current-main gap is source-verifiable: existing prompts and docs provide generic live-check guidance, but no version-change notes, general explain command, or known-behavior registry are present.

## Solution Assessment

Unclear. The issue’s direction is valid, but the best implementation needs maintainer product judgment between prompt injection, docs/changelog retrieval, a new CLI surface, and a registry so OpenClaw does not add stale or cache-hostile context.
