## Summary

Keep this PR open. Current main still has the iOS chat attachment problem this PR targets, and the branch is directionally useful, but the proposed implementation still uses a width-only transcoder path while promising a true long-edge cap, so portrait and narrow-tall images can exceed the stated 1600 px limit. The remaining blockers are narrow and mechanical: fix the long-edge calculation/tests, clean up the unreachable safety-net comment, and add the required changelog entry.

## What This Changes

The PR adds iOS chat image preprocessing that re-encodes selected attachments as resized JPEGs through JPEGTranscoder, strips source metadata, runs processing off the MainActor, adds alpha flattening in JPEGTranscoder, and adds Swift unit tests for the new processor.

## Best Possible Solution

Land a corrected branch that gives chat uploads a true longest-edge resize policy while keeping JPEGTranscoder's existing width-based contract safe for PhotoCapture, adds portrait and narrow-tall coverage, preserves metadata stripping and alpha flattening, and records the user-facing iOS chat change in the active changelog. Broader gateway/server image configurability should remain with the related media-limit issues rather than being folded into this PR.

## Reproduction Assessment

Yes. A source-level reproduction is any portrait or narrow-tall image whose oriented width is at or below 1600 while height is greater than 1600, or a 3000x4000 portrait where width-based scaling can produce roughly 1600x2133; the contributor's posted device evidence also matches that width-only path.

## Solution Assessment

No, not yet. Reusing JPEGTranscoder off the MainActor is the right maintainable direction, but the PR needs an additive true-long-edge mode or equivalent adapter calculation plus cleanup of the misleading safety-net comment before it matches the stated behavior.
