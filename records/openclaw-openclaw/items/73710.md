## Summary

Keep this PR open: it addresses a real iOS chat attachment gap and current main still sends raw selected image bytes, but the branch is not merge-ready because the documented 1600 px longest-edge policy is still wired through a width-only transcoder contract.

## What This Changes

The PR adds OpenClawKit iOS chat image preprocessing that re-encodes selected attachments as bounded JPEGs via JPEGTranscoder, strips source metadata, updates ChatViewModel, and adds Swift tests.

## Best Possible Solution

Land a corrected branch that applies a true longest-edge cap while preserving the centralized JPEGTranscoder reuse, metadata stripping, alpha flattening, and off-MainActor processing; add portrait/oriented coverage plus a single-line changelog entry, and let #68524 close only after merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any portrait or narrow-tall image whose oriented width is at or below 1600 while height is greater than 1600; the existing JPEGTranscoder contract and 1000x2000 test expectation prove that path.

## Solution Assessment

No, not yet. Reusing JPEGTranscoder is the right maintainable direction, but the PR needs an explicit longest-edge mode or equivalent target-size calculation before it matches the requested behavior.
