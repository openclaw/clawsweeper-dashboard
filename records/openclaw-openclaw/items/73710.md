## Summary

Keep this PR open, but do not merge it as-is. The change addresses a real iOS chat attachment gap and the privacy direction is sound, but the latest head still does not implement the promised true long-edge cap, and the branch needs a small cleanup plus the required changelog entry.

## What This Changes

The PR adds iOS chat image preprocessing that re-encodes selected images through `JPEGTranscoder` off the MainActor, stores JPEG output in pending attachments, adds alpha flattening, and adds Swift tests for resize, byte budget, metadata stripping, and alpha behavior.

## Best Possible Solution

Land a corrected branch that gives iOS chat uploads a real longest-edge cap while preserving centralized JPEGTranscoder reuse, metadata stripping, alpha flattening, and off-MainActor processing; add portrait/narrow-tall/oriented coverage and a single-line changelog entry, and keep the linked #68524 workflow open until the corrected PR merges.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any portrait or narrow-tall image whose oriented width is at or below 1600 while height is greater than 1600; the current width-only transcoder contract and existing `1000x2000` test expectation prove that path.

## Solution Assessment

No, not yet. Reusing `JPEGTranscoder` is the right maintainable direction, but the branch needs an explicit long-edge mode or equivalent adapter calculation plus tests before it matches the requested behavior.
