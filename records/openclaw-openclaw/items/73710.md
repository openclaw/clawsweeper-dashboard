## Summary

Keep open. The PR addresses a real iOS chat attachment gap and the latest head fixes the earlier long-edge/changelog issues, but it still needs changes because sending can race ahead of asynchronous image processing and one source comment still describes a removed 5 MB hard gate.

## What This Changes

The PR adds iOS chat attachment preprocessing that converts selected images to resized JPEGs with source metadata stripped, extends JPEGTranscoder for long-edge and alpha handling, adds Swift tests, and updates CHANGELOG.

## Best Possible Solution

Keep the JPEGTranscoder-based approach, add pending attachment-processing state so send waits or is disabled until selected images are appended, update the stale budget wording, and then land after targeted Swift and device validation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original gap: addImageAttachment rejects images above 5 MB before transformation and performSend base64-encodes stored bytes directly; the PR-only send race is also source-reproducible from the queued public add method plus append-after-processing flow.

## Solution Assessment

No, not yet. Reusing JPEGTranscoder with a true long-edge cap is the narrow maintainable direction, but the branch should prevent sends while image processing is pending and fix the stale hard-gate wording before merge.
