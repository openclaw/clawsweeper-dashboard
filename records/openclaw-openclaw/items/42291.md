## Label Justifications

- `P1`: The report describes a broken model-auth workflow that can consume OAuth refresh tokens and leave gateway model calls failing for real remote-node users.
- `impact:data-loss`: The wrong-host login can rotate or consume refresh-token material into a store that the gateway does not use, making the intended credential state effectively lost.
- `impact:security`: Provider OAuth credential material can cross the wrong node/gateway trust boundary.
- `impact:auth-provider`: The failure directly affects provider OAuth state, model auth routing, and recovery from `refresh_token_reused` errors.

## Summary

Keep open: current main and the latest release still run provider auth login against the local agent auth store without a remote node-host guard, and the only open closing PR is a conflicting fork PR that maintainers cannot directly edit.

## What This Changes

The issue requests preventing `openclaw models auth login` on remote node hosts from silently writing OAuth credentials into the node-local auth profile store instead of the gateway.

## Best Possible Solution

Land a focused preflight that blocks or warns before OAuth starts on non-loopback remote node-host gateway configs, with regression coverage for remote-node and supported local cases.

## Reproduction Assessment

Yes. Source inspection shows the login command resolves a local `agentDir`, runs provider auth, and persists returned profiles there while node-host config can point at a non-loopback gateway.

## Solution Assessment

Yes. Blocking or warning before auth starts on a remote node host is the narrowest maintainable fix; automatic gateway credential forwarding would need a larger credential-transport API and security design.
