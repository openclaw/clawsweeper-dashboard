## Summary

Keep this issue open. Current main still runs `openclaw models auth login` through the locally resolved agent auth store, has no remote node-host guard or gateway credential write path, and PR #42381 remains the active implementation candidate with closing syntax for this bug.

## What This Changes

The issue reports that `openclaw models auth login --provider openai-codex` on a remote node host writes OAuth credentials to the node-local auth store instead of the gateway store used for model calls.

## Best Possible Solution

Keep the issue open and review PR #42381 or an equivalent focused fix that blocks or clearly warns before local OAuth writes on remote node hosts. A later sync path should first add an explicit authenticated gateway auth-profile mutation API with scoped permissions.

## Reproduction Assessment

Yes. The current source path shows the login command resolves and writes a local `agentDir`, and existing tests assert that local write behavior; the report's remote-node failure follows from that code path without needing to consume a live OAuth token.

## Solution Assessment

Unclear for direct sync, but the conservative short-term direction is clear: prevent the silent local write from remote node hosts and tell users to authenticate on the gateway. Direct gateway sync should wait for a designed, permissioned mutation surface.
