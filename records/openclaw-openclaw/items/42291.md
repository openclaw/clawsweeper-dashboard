## Label Justifications

- `P1`: A successful-looking OAuth login can immediately break model authentication on a remote gateway for affected users.
- `impact:data-loss`: The refresh credential can be consumed locally without updating the gateway's canonical credential state.
- `impact:security`: OAuth credentials are persisted at the wrong machine boundary, creating a sensitive credential-handling failure.

## Summary

The report remains valid on current main, and the linked OpenClaw fixes all closed unmerged. A live fork PR remains paired with this issue, so it should stay open as the canonical tracker rather than spawning duplicate work.

## What This Changes

Prevent remote node-host OAuth login from rotating credentials into a node-local auth store that the gateway does not use.

## Best Possible Solution

Add a pre-OAuth guard in the models auth command that reads canonical node-host configuration, blocks non-loopback remote gateways with actionable gateway-host instructions, and proves local or loopback login remains unchanged.

## Reproduction Assessment

Yes. Current main persists successful login results locally, node-host configuration independently identifies the remote gateway, and no check joins those facts before OAuth starts.

## Solution Assessment

Yes, the blocking guard is the narrowest maintainable solution. It prevents token consumption before persistence without inventing a credential-sync protocol and preserves normal local login.
