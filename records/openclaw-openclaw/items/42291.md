## Label Justifications

- `P1`: A successful-looking OAuth login on a remote node host can consume or rotate credentials while leaving the gateway unable to authenticate model calls.
- `impact:data-loss`: The usable refresh credential can be consumed on the wrong host without updating the gateway's canonical auth state.
- `impact:security`: The report is about OAuth secret material being persisted and rotated across the wrong machine boundary.
- `impact:auth-provider`: The affected workflow is provider OAuth login and gateway model authentication.

## Summary

Keep open. Current main and the latest release still persist `models auth login` results through the local agent auth store, and the open linked fork PR is the active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue asks OpenClaw to prevent `openclaw models auth login` on a remote node host from silently writing OAuth credentials to the node-local auth store instead of the gateway-owned auth store.

## Best Possible Solution

Land a focused pre-OAuth guard that detects non-loopback node-host gateway config, tells the user to run auth on the gateway host, preserves local/loopback auth login, and avoids a new credential-forwarding protocol.

## Reproduction Assessment

Yes. Source inspection shows current main persists login profiles through the local `agentDir`, while node-host config can identify a non-loopback gateway and is not consulted before OAuth starts.

## Solution Assessment

Yes. A pre-OAuth blocking guard is the narrowest maintainable fix because it prevents token consumption before local persistence without inventing a new gateway credential-sync protocol.
