## Label Justifications

- `P1`: The report describes a real remote gateway auth workflow where OAuth credentials can be consumed or stored on the wrong host and break model calls.
- `impact:data-loss`: A consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue is about OAuth credential placement across a node-host and gateway trust boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main still has the reported wrong-host auth path, and an open linked PR remains the active implementation path rather than proof that the issue is fixed.

## What This Changes

The issue asks for `openclaw models auth login` on a remote node host to avoid silently storing OAuth credentials in the node-local auth profile store instead of the gateway-owned store.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42381 with a focused pre-auth guard that blocks non-loopback node-host auth login before provider auth starts and tells the user to run auth on the gateway host.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a node host can persist a non-loopback gateway in `node.json`, while current `models auth login` still resolves a local `agentDir` and writes auth profiles through that local store.

## Solution Assessment

Yes. A pre-auth remote-node guard with gateway-host guidance is the narrow maintainable fix; automatic credential sync would require a larger gateway auth-profile API and token-safety design.
