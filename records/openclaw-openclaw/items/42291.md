## Label Justifications

- `P1`: The report describes a real remote node-host auth workflow where OAuth refresh material can be consumed locally while gateway model calls keep failing.
- `impact:data-loss`: The flow can consume or rotate refresh-token material into the wrong store, leaving the intended gateway credential unusable.
- `impact:security`: Provider OAuth credential material can be written across the wrong node/gateway trust boundary.
- `impact:auth-provider`: The failure directly affects provider OAuth state, model auth routing, and recovery from `refresh_token_reused` errors.

## Summary

Keep open: current main and the latest release still persist provider auth login results to the local agent auth store, while the only open closing PR is cross-repository, conflicting, and unmerged.

## What This Changes

The issue reports that `openclaw models auth login` on a remote node host stores OAuth credentials in the node-local auth profile instead of making them available to the gateway where model calls run.

## Best Possible Solution

Land a focused auth-login preflight that detects remote node-host gateway config and blocks or clearly redirects login before OAuth starts, with regression coverage for non-loopback and supported loopback cases.

## Reproduction Assessment

Yes, at source level. Current main resolves a local `agentDir`, runs provider auth, and upserts returned credentials there while node-host config can persist a non-loopback gateway target.

## Solution Assessment

Yes, a pre-OAuth remote-node guard is the narrowest maintainable fix. Full gateway credential forwarding would require a larger API design and is not needed to stop the wrong-host token write.
