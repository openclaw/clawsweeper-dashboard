## Label Justifications

- `P1`: The issue describes an urgent model-auth workflow where a remote node can consume OAuth refresh credentials locally while gateway model calls continue to fail.
- `impact:data-loss`: The reported flow consumes or rotates refresh-token material into the wrong local store, leaving the gateway without the usable credential.
- `impact:security`: OAuth credential material can be written onto a node host that is not the gateway host intended to use the model credentials.
- `impact:auth-provider`: The failure sits directly on provider auth profile storage and OpenAI/Codex credential routing for model calls.

## Summary

Keep open. Current main and the latest release still run provider auth login through the local agent auth store without checking node-host gateway state, and the only open closing PR is a conflicting fork PR rather than a landable upstream fix.

## What This Changes

The issue reports that `openclaw models auth login` on a remote node host stores/consumes OAuth credentials locally instead of making them usable by the gateway.

## Best Possible Solution

Land a focused upstream fix that detects non-loopback node-host gateway config before provider auth begins, blocks with clear gateway-host guidance or implements reviewed gateway credential forwarding, and covers tokenless configs plus shared loopback semantics in regression tests.

## Reproduction Assessment

Yes at source level: `modelsAuthLoginCommand` writes returned provider profiles to the local `agentDir` and does not inspect node-host gateway config before auth starts. I did not run the live OAuth flow, but the current code path matches the reported wrong-host write.

## Solution Assessment

Yes, the narrow maintainable direction is to stop or redirect auth before token material is minted on a remote node host. Full gateway credential forwarding would need a broader protocol/security design, so a fail-closed guard with clear guidance is the safer first fix.
