## Label Justifications

- `P1`: A remote node can consume or store OAuth/provider credentials locally while gateway model calls continue failing, which breaks a real auth-provider workflow.
- `impact:data-loss`: The reported flow can consume or rotate refresh-token material into the wrong local store, leaving the gateway without the usable credential.
- `impact:security`: Provider OAuth credential material can be written onto a node host that is not the intended gateway credential boundary.
- `impact:auth-provider`: The failure sits directly on provider auth profile storage and model credential routing for gateway-backed calls.

## Summary

Keep open. Current main and v2026.6.6 still persist provider login profiles through the local agent auth store without checking node-host gateway state, and the only closing PR is open in a fork, conflicting, and not a clean upstream landing path.

## What This Changes

The issue asks `openclaw models auth login` on a remote node host to avoid silently writing OAuth/provider credentials to the node-local auth store instead of the gateway host.

## Best Possible Solution

Land a focused upstream fix that detects non-loopback remote node-host config before provider auth starts, blocks with clear gateway-host guidance or implements reviewed credential forwarding, and covers both tokenless node configs and accepted loopback forms in regression tests.

## Reproduction Assessment

Yes at source level: `modelsAuthLoginCommand` writes returned provider profiles to the local `agentDir` and does not inspect node-host gateway config before auth starts. I did not run the live OAuth flow, but current source matches the reported wrong-host write.

## Solution Assessment

Yes, the narrow maintainable direction is to stop or redirect auth before token material is minted on a remote node host. Full gateway credential forwarding would need broader protocol/security review, so a fail-closed guard with clear guidance is the safer first repair path.
