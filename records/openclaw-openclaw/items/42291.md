## Label Justifications

- `P1`: The reported remote gateway auth workflow can consume OAuth credentials on the wrong host and break model calls for real users.
- `impact:data-loss`: A consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue is about OAuth credential placement across a node-host and gateway trust boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main and the latest tagged release still route `models auth login` through the local agent auth store, and the open linked implementation PR is the active fix path rather than proof that this issue is resolved.

## What This Changes

The issue reports that provider OAuth login on a remote node host stores credentials on the node instead of the gateway that performs model calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42381 with a focused pre-auth guard that blocks non-loopback node-host auth login before provider auth starts and directs the user to run auth on the gateway host.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a node host can persist a non-loopback gateway in `node.json`, while current `models auth login` still resolves a local `agentDir` and writes auth profiles through that local store.

## Solution Assessment

Yes. A pre-auth remote-node guard with gateway-host guidance is the narrow maintainable fix; automatic credential sync would require a larger gateway auth-profile API and token-safety design.
