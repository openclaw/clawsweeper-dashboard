## Label Justifications

- `P1`: The reported remote gateway auth workflow can consume or store OAuth credentials on the wrong host and break real model calls.
- `impact:data-loss`: A consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue concerns OAuth credential placement across a node-host and gateway trust boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main and the latest release still route `models auth login` through the local agent auth store, and the open linked PR remains the canonical implementation path rather than a merged fix.

## What This Changes

The issue requests preventing `openclaw models auth login` on a remote node host from consuming or writing OAuth credentials only to the node-local auth profile store instead of the gateway.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42381 with a focused pre-auth guard that blocks non-loopback node-host auth login before provider auth, uses the shared loopback helper, and covers tokenless remote plus accepted loopback cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a node host can persist a non-loopback gateway in `node.json`, while current `models auth login` still writes returned provider profiles through the local `agentDir`.

## Solution Assessment

Yes. A pre-auth remote-node guard with gateway-host guidance is the narrow maintainable fix; automatic credential sync would require a larger gateway auth-profile API and token-safety design.
