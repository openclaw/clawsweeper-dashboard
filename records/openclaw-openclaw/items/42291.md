## Label Justifications

- `P1`: The reported remote gateway auth workflow can consume or store OAuth credentials on the wrong host and break model calls for real users.
- `impact:data-loss`: A consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue concerns OAuth credential placement across a node-host and gateway trust boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main still persists `models auth login` results through the local agent auth store, and the live closing PR remains open rather than merged.

## What This Changes

Remote node-host `openclaw models auth login` should not silently write OAuth credentials to node-local auth profiles when model calls run through a remote gateway.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42381 with a focused pre-auth guard that blocks all non-loopback node-host auth login before provider auth and directs users to run auth on the gateway host.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a node host can persist a non-loopback gateway in `node.json`, while current `models auth login` still writes provider profiles through the local `agentDir`.

## Solution Assessment

Yes. A pre-auth remote-node guard with gateway-host guidance is the narrow maintainable fix; automatic credential sync would require a larger gateway auth-profile API and token-safety design.
