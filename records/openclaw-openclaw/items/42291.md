## Label Justifications

- `P1`: The report describes a real remote gateway auth workflow where OAuth credentials can be consumed or stored on the wrong host and break model calls.
- `impact:data-loss`: The consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue is about OAuth credential placement across a node host and gateway boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main still allows `models auth login` to run provider auth and persist credentials through the local agent auth store without checking remote node-host state, and the open closing PR remains the active fix path.

## What This Changes

The issue asks OpenClaw to stop silently writing OAuth credentials to a node-host local auth store when model calls are owned by a remote gateway.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/42381 with a focused pre-auth guard that blocks non-loopback node-host auth login before provider auth starts and tells the user to run auth on the gateway host.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: node-host config can persist a non-loopback gateway in `node.json`, while current `models auth login` resolves a local `agentDir` and writes auth profiles without checking that state.

## Solution Assessment

Yes. A pre-auth remote-node guard with gateway-host guidance is the narrow maintainable fix; automatic credential sync would require a larger gateway auth-profile API and token-safety design.
