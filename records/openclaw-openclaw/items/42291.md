## Label Justifications

- `P1`: The reported remote gateway auth workflow can consume or store OAuth credentials on the wrong host and break model calls for real users.
- `impact:data-loss`: A consumed refresh token can leave the gateway with stale or unusable credential state.
- `impact:security`: The issue concerns OAuth credential placement across a node-host and gateway trust boundary.
- `impact:auth-provider`: The failure is in provider OAuth login, auth-profile storage, and model-provider credential routing.

## Summary

Keep open: current main and the latest release still route `models auth login` through the local agent auth store, and the focused closing PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to prevent `openclaw models auth login` on a remote node host from silently saving OAuth credentials to the node-local auth profile store instead of gateway-owned auth state.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42381 with a pre-auth guard that blocks non-loopback node-host auth login before provider auth, reuses shared loopback semantics, and tells users to run auth on the gateway host.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a remote node host can persist a non-loopback gateway in `node.json`, while current `models auth login` still writes provider profiles through the local `agentDir`.

## Solution Assessment

Yes, with the current PR repaired. A pre-auth remote-node guard is the narrow maintainable fix; automatic credential syncing would require a larger gateway auth-profile API and token-safety design.
