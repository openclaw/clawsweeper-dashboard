## Summary

Keep open: current main and the latest release still run provider auth login through the local agent auth store, and the focused closing PR at https://github.com/openclaw/openclaw/pull/42381 is still open, conflicting, and unmerged.

## What This Changes

The issue asks OpenClaw to prevent `openclaw models auth login` on a remote node host from silently writing OAuth credentials to the node-local auth store instead of the gateway-owned store.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/42381, or replace it with a focused guard that blocks non-loopback remote node-host auth login before provider auth runs and tells users to authenticate on the gateway host.

## Reproduction Assessment

Yes by source inspection: a remote node host can persist non-loopback gateway state in `node.json`, while current `models auth login` still writes returned provider profiles through the local `agentDir`. I did not run a live OAuth flow in this read-only review.

## Solution Assessment

Yes in direction: a pre-auth remote-node guard is the narrow maintainable fix; automatic credential syncing would require a larger gateway auth-profile API and token-safety design.
