## Label Justifications

- `P1`: The report describes a real remote node-host provider-auth workflow where OAuth refresh material can be consumed locally while gateway model calls keep failing.
- `impact:data-loss`: The flow can consume or rotate refresh-token material into the wrong store, leaving the intended gateway credential unusable.
- `impact:security`: Provider OAuth credential material can be written across the wrong node/gateway trust boundary.
- `impact:auth-provider`: The failure directly affects provider OAuth state, model auth routing, and recovery from `refresh_token_reused` errors.

## Summary

Keep open: current main and the latest release still write provider auth login results to the local agent auth store with no remote node-host guard, and the open closing PR has not merged.

## What This Changes

The issue asks `openclaw models auth login` on a remote node host to avoid silently storing OAuth credentials on the node-local auth store instead of the gateway that runs model calls.

## Best Possible Solution

Land a focused auth-login preflight that detects remote node-host gateway config and blocks or warns before OAuth starts, with regression coverage for remote and loopback cases.

## Reproduction Assessment

Yes, at source level. Current main resolves a local `agentDir`, runs provider auth, and upserts returned credentials there while node-host config can point at a non-loopback gateway.

## Solution Assessment

Yes, the narrowest maintainable solution is to block or warn before auth starts on remote node hosts. Full gateway credential forwarding would be a larger API design and is not needed to stop the wrong-host write.
