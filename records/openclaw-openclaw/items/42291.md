## Label Justifications

- `P1`: The issue describes a broken remote node-host auth workflow that can consume OAuth refresh tokens and leave gateway model calls failing.
- `impact:data-loss`: The flow can consume or rotate refresh-token material into the wrong store, making the intended gateway credential unusable.

- `impact:auth-provider`: The failure directly affects provider OAuth state, auth-profile storage, and model-call credential resolution.

## Summary

Keep open: current main and latest release still write provider auth login results through the local agent auth store, while node-host config can point at a non-loopback gateway; the only open closing PR is a conflicting, non-maintainer-editable fork PR.

## What This Changes

The issue asks OpenClaw to prevent `openclaw models auth login` on remote node hosts from silently storing OAuth credentials on the node instead of the gateway.

## Best Possible Solution

Land a focused pre-auth guard that detects non-loopback node-host gateway config and blocks or warns before provider OAuth starts, with regression coverage for remote and accepted loopback cases.

## Reproduction Assessment

Yes, at source level. Current main resolves a local agent directory and upserts provider auth profiles there, while node-host config separately records a non-loopback gateway target.

## Solution Assessment

Yes, a pre-auth remote-node guard is the narrowest maintainable fix. Automatic credential forwarding to the gateway would be a larger protocol and security design.
