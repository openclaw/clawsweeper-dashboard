## Summary

Keep open: current main and v2026.6.10 still run provider auth login against the resolved local agent auth store with no remote node-host guard, and the only open closing PR is an external fork PR that is conflicting and not maintainer-editable.

## What This Changes

The issue asks `openclaw models auth login` on a remote node host to avoid silently writing OAuth credentials to the node-local store, either by forwarding auth to the gateway or warning/blocking before token consumption.

## Best Possible Solution

Land a focused preflight that detects remote node-host gateway config before provider auth starts, uses the shared gateway loopback semantics, blocks or clearly warns for non-loopback node-host auth login, and covers both blocked and local allow cases in tests.

## Reproduction Assessment

Yes, at source level. Current main resolves a local `agentDir`, runs provider auth, and upserts returned credentials there while node-host config can point at a non-loopback gateway.

## Solution Assessment

Yes for the narrow guard direction, with one maintainer choice remaining. Blocking or warning before auth starts is smaller and safer than adding gateway credential forwarding, but maintainers should explicitly own the fail-closed behavior.
