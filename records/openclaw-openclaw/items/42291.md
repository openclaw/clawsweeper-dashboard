## Label Justifications

- `P1`: The report describes a broken provider-auth workflow that can consume OAuth refresh material locally while gateway model calls continue failing.
- `impact:data-loss`: Refresh-token material can be rotated or consumed into the wrong local store, leaving the intended gateway credential unusable.
- `impact:security`: Provider OAuth credential material crosses the intended node/gateway boundary and lands on the node host.
- `impact:auth-provider`: The failure directly affects provider OAuth state, model auth routing, and recovery from `refresh_token_reused` errors.

## Summary

Keep open: current main and v2026.6.8 still persist provider OAuth results into the locally resolved agent auth store without checking node-host remote gateway config. The same-repo fix attempts are closed unmerged, and the only open closing PR is cross-repository, conflicting, and not maintainer-editable.

## What This Changes

The issue reports that `openclaw models auth login` on a node host connected to a remote gateway silently stores OpenAI OAuth credentials in the node-local auth store instead of the gateway-owned store.

## Best Possible Solution

Land a narrow pre-OAuth guard or warning in `models auth login` that detects node-host configs targeting non-loopback gateways and prevents local credential writes, with regression coverage for remote and loopback cases.

## Reproduction Assessment

Yes, at source level. `modelsAuthLoginCommand` resolves a local `agentDir`, runs provider OAuth, and `persistProviderAuthResult` upserts OAuth profiles into that local directory while `node.json` can identify a non-loopback gateway.

## Solution Assessment

No current-main solution exists yet. The safest maintainable fix is a preflight before provider OAuth starts; automatic gateway credential forwarding would need a designed gateway credential API first.
