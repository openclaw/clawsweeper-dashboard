## Summary

Keep open: the PR targets a real Telegram cron topic-delivery bug, but the branch is stale and the proposed pre-parser still misses the documented `chatId:topicId` shorthand, so isolated cron delivery can still lose the thread ID. It also lacks after-fix real Telegram proof for a topic-routing transport change.

## What This Changes

The PR adds Telegram slash target parsing, pre-parses cron Telegram topic targets, threads config into outbound target parsing, propagates delivery errors into cron run state, and updates docs/tests.

## Best Possible Solution

Refresh the fix onto current main and add one canonical isolated cron target-normalization path that covers `:topic`, numeric-colon shorthand, and slash forms, with focused regression tests and redacted real Telegram topic proof.

## Reproduction Assessment

Yes from source inspection, without running a mutating test: current main and the PR path both show that target-thread extraction must happen before loaded-plugin parsing for isolated cron, and the proposed pre-parser omits the numeric-colon format. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is right, but the implementation should be refreshed onto current main and cover every supported Telegram topic target shape through one maintained parsing path rather than a partial duplicate.
