## Summary

Keep open: the PR targets a real Telegram cron forum-topic delivery gap, but it is not merge-ready because it lacks real Telegram proof, is currently conflicting, and has correctness gaps in the proposed parsing/finalization path.

## What This Changes

The PR adds Telegram cron announce forum-topic parsing, legacy slash target support, delivery-error propagation, docs, and regression tests.

## Best Possible Solution

Rebase the useful pieces onto current main, cover every documented target format in one parser path, keep text finalization before threaded direct delivery, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main lacks slash-format parsing and the PR's own pre-parser omits the documented shorthand format. I did not run a live Telegram cron job in this read-only review.

## Solution Assessment

No, not as submitted. The direction is useful, but the patch should preserve the existing text finalization path for threaded deliveries, parse all documented target formats, resolve conflicts, and add real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve text finalization for threaded cron delivery:** `src/cron/isolated-agent/delivery-target.ts:100`
  - body: By injecting an explicit thread ID for forum-topic targets, this path makes `dispatchCronDelivery` take the direct delivery branch, which skips `finalizeTextDelivery`. That helper is where descendant/subagent output replaces interim parent text, so text-only cron jobs that fan out can announce the placeholder instead of the final reply in the Telegram topic.
  - confidence: 0.86
- **[P2] Parse the documented shorthand before plugin bootstrap:** `src/cron/isolated-agent/delivery-target.ts:24-30`
  - body: The pre-parser handles `chatId:topic:topicId` and `chatId/topicId`, but not `chatId:topicId`. In the isolated no-plugin-bootstrap case described by the PR, that documented shorthand still falls through without an extracted thread ID, so the docs claim that all listed formats work is false.
  - confidence: 0.9
