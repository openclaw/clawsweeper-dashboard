## Summary

Keep open. Current main still has the Signal daemon restart race, and this PR is a useful, narrow Signal-owned fix, but it still lacks the required user-facing changelog entry before merge; after that it should go through normal maintainer review, CI, and live Signal restart validation.

## What This Changes

The PR makes the Signal plugin's auto-started signal-cli daemon shutdown awaitable with SIGTERM-to-SIGKILL escalation, awaits monitor cleanup on abort/finally, and adds daemon/autostart regression tests.

## Best Possible Solution

Land the Signal plugin lifecycle fix after adding release-note coverage: keep daemon shutdown awaitable and idempotent, wait for signal-cli exit after SIGTERM, escalate after a bounded timeout, preserve the focused regression tests, and avoid changing core channel APIs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is Signal autoStart plus a channels.signal config reload: current main awaits channel stop before start, but the Signal monitor resolves after a fire-and-forget SIGTERM instead of waiting for the old daemon process to exit.

## Solution Assessment

Yes for the code direction. Signal owns this child-process lifecycle, and docs say OpenClaw owns signal-cli when autoStart is enabled; the narrow maintainable solution is an awaitable Signal daemon stop awaited by monitor cleanup, with changelog and live validation before merge.
