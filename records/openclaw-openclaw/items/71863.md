## Summary

Keep open for maintainer review. The patch appears correct and well-scoped, but it intentionally changes Signal daemon restart teardown from fire-and-forget SIGTERM to awaited exit with SIGKILL escalation, so the remaining question is maintainer acceptance of that availability policy rather than contributor repair.

## What This Changes

The branch makes Signal daemon stop async and idempotent, waits for child-process exit, escalates to SIGKILL after 1500 ms, awaits monitor cleanup, and adds daemon/monitor regression tests.

## Best Possible Solution

Land this or an equivalent Signal-owned lifecycle fix once maintainers accept the timeout/escalation policy, requesting one live Signal hot-reload run or a Signal-level final timeout only if that proof or policy gap matters before merge.

## Reproduction Assessment

Yes. Source-level reproduction is clear: current main sends SIGTERM from Signal daemon stop without waiting for child exit, and Signal hot reload stops then immediately starts the channel.

## Solution Assessment

Yes, with a policy caveat. The PR fixes the implicated Signal-owned daemon lifecycle and monitor teardown directly instead of adding a gateway-wide delay, new config option, or broad unrelated iMessage/Gmail cleanup; maintainers still need to accept the timeout/escalation behavior.
