## Summary

Keep open. Current main still has the synchronous Signal daemon stop path that the PR is trying to fix, and the PR remains an active, focused implementation candidate with no definite blocking correctness or security finding from this review.

## What This Changes

The PR makes Signal daemon shutdown awaitable, adds a bounded SIGTERM-to-SIGKILL fallback, awaits monitor cleanup, and adds daemon/monitor regression coverage.

## Best Possible Solution

Land a Signal-owned lifecycle fix that makes daemon stop idempotent and awaitable, waits for `signal-cli` exit after SIGTERM, escalates after a bounded timeout, and keeps the regression tests local to the Signal plugin without changing core channel restart semantics.

## Reproduction Assessment

Yes. The high-confidence code path is a Signal autostart config reload: current main awaits channel stop before restart, but the Signal monitor currently resolves cleanup after a fire-and-forget SIGTERM instead of waiting for the daemon child to exit.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because Signal owns the `signal-cli` daemon lifecycle, while the gateway already has a generic stop-then-start restart contract.
