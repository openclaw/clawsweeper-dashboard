## Summary

Keep open: the reported no-final-reply symptom is source-reproducible, but current main documents group/channel normal final replies as tool-only unless configured otherwise or sent through `message.send`. This PR still broadens that shared-room visibility boundary for @mentions and lacks after-fix real behavior proof, so it needs human product/security review before merge.

## What This Changes

The PR adds `WasMentioned` handling to source reply delivery so mentioned group/channel turns resolve from `message_tool_only` to `automatic`, with unit tests for default, requested, and policy cases.

## Best Possible Solution

If maintainers want an @mention exception, implement it as a documented policy that preserves any explicit opt-out semantics they require, updates channel coverage/docs, and includes redacted live group/channel proof; otherwise keep the current tool-only contract and route visible answers through `message.send` or `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

Yes, source-reproducible: current main resolves mentioned group/channel turns to `message_tool_only` through existing pre-resolution/requested-mode paths, and dispatch suppresses normal final replies. I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not as written. The resolver is a plausible policy hook, but unconditionally overriding requested/configured tool-only delivery is broader than the current documented contract and needs maintainer approval plus docs and real behavior proof.
