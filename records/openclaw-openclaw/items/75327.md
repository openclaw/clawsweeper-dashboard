## Summary

Keep open: the discussion usefully ties the symptom to the shared resolver and dispatch suppression path, but this PR still changes a documented shared-room visibility boundary, overrides explicit/requested tool-only delivery on @mention, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `WasMentioned` handling to source reply delivery resolution so group/channel @mentions change `message_tool_only` replies to `automatic`, with focused unit coverage.

## Best Possible Solution

Decide the @mention visibility rule explicitly; if accepted, preserve any intended explicit tool-only opt-outs, update public group/channel docs and channel coverage, and require redacted live proof from at least one affected real channel.

## Reproduction Assessment

Yes for the current behavior from source inspection: mentioned group/channel turns still resolve to `message_tool_only` and dispatch skips normal final replies while suppression is active. I did not establish a live channel reproduction in this read-only review.

## Solution Assessment

No, not as written. The resolver is a plausible policy plane, but this patch overrides explicit/requested tool-only delivery and conflicts with the current documented shared-room visibility contract without maintainer policy acceptance.
