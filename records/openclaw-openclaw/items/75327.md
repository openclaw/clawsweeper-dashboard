## Summary

Keep open: the reported no-visible-reply path is source-reproducible, but this PR is still not merge-ready because it broadens the current shared-room visibility boundary, can bypass direct-chat/tool-only policy, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds `WasMentioned` to source reply delivery resolution so mentioned group/channel turns switch from `message_tool_only` to `automatic`, with resolver and visibility-policy unit tests.

## Best Possible Solution

Decide the @mention visibility policy explicitly; if accepted, implement a group/channel-scoped path that preserves explicit tool-only policy where intended, update docs/tests across affected channels, and require redacted real-channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main resolves mentioned group/channel turns to `message_tool_only` under the documented default, and dispatch skips ordinary final replies while suppression is active. I did not establish a live channel reproduction in this read-only review.

## Solution Assessment

No, not as merge-ready. The resolver is a plausible policy plane, but this patch bypasses current tool-only semantics too broadly and needs maintainer policy approval, docs follow-through, and real behavior proof.
