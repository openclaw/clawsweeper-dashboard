## Summary

Keep this PR open for maintainer review, but do not merge it as-is. The branch changes group/channel @mentions from the current documented tool-only visible-reply contract to automatic final delivery, including when a caller explicitly requested `message_tool_only`; current main instead preserves that mode unless the message tool is unavailable, and the adjacent merged fix already covers the silent-delivery failure through message-tool availability and fallback handling.

## What This Changes

The PR adds `WasMentioned` to the source reply delivery context and tests so group/channel @mentions convert `message_tool_only` delivery, including requested modes, to `automatic`.

## Best Possible Solution

Keep the current tool-only group/channel default unless maintainers explicitly approve an @mention automatic-delivery exception. If approved, implement the exception with clear precedence for explicit requested modes, update group/channel and Discord docs plus `CHANGELOG.md`, and add channel-level coverage for the affected Discord/Slack-style pre-resolution path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: current main source/tests show mentioned Discord guild messages still pre-resolve to `message_tool_only`, and a focused resolver/dispatch case with `ChatType: "group"`, `WasMentioned: true`, and requested `message_tool_only` demonstrates the branch behavior this PR changes.

## Solution Assessment

Is this the best way to solve the issue? No: the PR bypasses explicit tool-only delivery and conflicts with the documented group/channel privacy contract; the safer existing path is message-tool delivery with automatic fallback only when the message tool is unavailable, unless maintainers choose and document a new exception.
