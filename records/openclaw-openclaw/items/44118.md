## Summary

Keep this PR open: it targets a real Feishu command-routing gap, but the current diff uses a coarse slash-token detector to bypass an operator-configured mention gate before real command authorization is known, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes Feishu group-message handling so slash-like commands can bypass `requireMention` and activate broadcast routing without an explicit bot mention.

## Best Possible Solution

Implement the Feishu bypass through the shared mention-gating semantics: normalize the Feishu probe body, require an actual authorized control command before skipping `requireMention`, reuse that decision for broadcast activation, and add false-positive, unauthorized, and broadcast regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main returns at the Feishu group activation gate before command probing for unmentioned `requireMention=true` messages, and the PR's changed path would let the false-positive-prone preauthorization helper bypass that gate.

## Solution Assessment

No. The direction is useful, but the patch should not use `shouldComputeCommandAuthorized` as the activation bypass; it should bypass only after resolving an authorized control command and then share that result with broadcast routing.
