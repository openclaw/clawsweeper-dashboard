## Summary

Current main has the transcript-persistence fix: fallback retries keep retry context model-visible, but once the current user turn has actually been persisted, the retry attempt suppresses the duplicate user-message write that caused synthetic retry text to appear as user-authored history.

## What This Changes

The issue asks for fallback retries to preserve the original user prompt in persisted transcript history instead of recording synthetic retry text.

## Best Possible Solution

Keep the shipped separation: preserve retry context for the model, suppress duplicate retry user-message persistence, and leave separate retry-marker/session-history tuning to https://github.com/openclaw/openclaw/issues/69086.

## Reproduction Assessment

No, not as a current-main failure. Source inspection shows the old transcript-persistence path is now guarded by current-turn append tracking and one-shot retry user-message suppression, with focused regression tests covering that boundary.

## Solution Assessment

Yes for this transcript issue. Returning only the original body would remove useful retry context from the model, while the current implementation keeps model retry context without persisting a synthetic retry turn as user-authored history.
