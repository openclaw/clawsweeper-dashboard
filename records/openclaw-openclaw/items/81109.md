## Summary

Keep open: current main still has a source-proven path where the Codex harness default resolves a direct WebChat turn to message-tool-only delivery, and an open linked fix PR uses closing syntax for this issue. The later comments and the alternative renderer PR make this an active maintainer choice, not a cleanup close.

## What This Changes

The issue reports that Codex harness message-tool visible-reply defaults can suppress automatic final delivery for direct internal WebChat turns, leaving users with a receipt-like response instead of the answer.

## Best Possible Solution

Review and land one WebChat-specific fix path that preserves external message-tool direct delivery while ensuring internal WebChat shows the substantive assistant reply.

## Reproduction Assessment

Yes. Source inspection gives a concrete current-main path: an unset visibleReplies direct WebChat/Codex turn can inherit `message_tool`, resolve to `message_tool_only`, and suppress queued final delivery while WebChat is non-deliverable.

## Solution Assessment

Unclear which implementation maintainers should prefer, because both https://github.com/openclaw/openclaw/pull/81110 and https://github.com/openclaw/openclaw/pull/81144 target the bug with different semantics. A prompt/tool-description-only fix looks weaker than a delivery-policy or renderer change that preserves external routes.
