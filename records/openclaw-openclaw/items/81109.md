Current main implements the maintainer-chosen fix through the merged internal UI message-tool sink, so Codex WebChat/TUI message-tool-only replies now project back into the active UI turn instead of disappearing or surfacing only a send receipt.

## What This Changes

The issue asks OpenClaw to keep direct Codex WebChat replies visible when the Codex harness defaults source-visible replies to the message tool.

## Best Possible Solution

Keep the shipped https://github.com/openclaw/openclaw/pull/81586 design: Codex keeps using message-tool-only source replies, while WebChat and TUI satisfy targetless current-run sends without becoming outbound delivery channels.

## Reproduction Assessment

Yes for the original bug: the linked merged PR includes live proof of a targetless Codex message.send from an internal UI session succeeding and being visible in chat history. On current main, the failure no longer reproduces because the guarded internal UI sink handles that path.

## Solution Assessment

Yes. The merged fix is the maintainable product shape chosen in discussion: preserve the Codex harness message-tool contract, add a narrow current-run sink for WebChat/TUI, and leave explicit external channel routing unchanged.
