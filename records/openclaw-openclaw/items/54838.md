## Summary

Keep open: current main has the expansion-state seam but still hardcodes Control UI auto-expansion off and has no persisted `chatExpandToolCalls` setting, while this PR's branch has concrete review blockers before it can be merged safely.

## What This Changes

The PR adds a persisted `chatExpandToolCalls` Control UI setting, desktop/mobile toolbar buttons, render plumbing, English copy, and fixture updates so tool-call details expand by default.

## Best Possible Solution

Refresh the feature against current main by wiring a persisted browser preference into `autoExpandToolCalls` state seeding, with consistent toolbar disabled copy, generated i18n outputs, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks the setting and hardcodes `autoExpandToolCalls` off, while the PR branch's direct `open` binding can be observed by manually opening/collapsing a tool disclosure and triggering a chat rerender.

## Solution Assessment

No. The feature direction is useful, but the best implementation is to feed the setting into the current expansion-state map rather than binding `<details open>` directly, then complete the toolbar, i18n, changelog, and test follow-through.
