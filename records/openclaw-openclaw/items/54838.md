## Summary

Keep open: the requested Control UI feature is not implemented on current main, but the latest PR head has blocking source-level defects, is still wired around an older direct-`open` rendering approach instead of main's expansion-state seam, and lacks required real behavior proof.

## What This Changes

The PR adds a persisted `chatExpandToolCalls` Control UI setting, desktop/mobile toolbar controls, render plumbing, English copy, and fixture updates to expand webchat tool-call output by default.

## Best Possible Solution

Rebase onto current main, store the preference in `UiSettings`, pass it into the existing `autoExpandToolCalls`/`syncToolCardExpansionState` path, preserve manual disclosure toggles, regenerate i18n outputs, and attach redacted real webchat proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the latest PR head has a malformed lit template in `grouped-render.ts`, and its direct `open` bindings would force disclosure state on rerenders instead of preserving current-main manual toggles. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix is to wire the new preference into current main's expansion-state map rather than binding `<details open>` directly.
