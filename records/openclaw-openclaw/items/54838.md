## Summary

Keep open: current main still lacks a user-facing expand-by-default preference, but this PR should not merge as-is because it bypasses the current stateful expansion path and misses required UI follow-through.

## What This Changes

The PR adds a persisted Control UI `chatExpandToolCalls` setting, desktop/mobile toolbar toggles, render plumbing, English copy, and fixture updates for expanding tool-call output by default.

## Best Possible Solution

Refresh the feature on current main by adding a persisted browser preference that feeds the existing expansion-state map, with consistent toolbar disabled copy, generated i18n outputs, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on the PR branch, manually open or collapse a tool disclosure and then trigger a streaming/history rerender; the new `?open` binding reapplies the global preference and overrides the manual state.

## Solution Assessment

No. The feature direction is right, but the maintainable fix is to wire the new setting into current main's `autoExpandToolCalls` state seeding rather than binding `<details open>` directly, then finish the toolbar, i18n, and changelog requirements.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve manual tool expansion state:** `ui/src/ui/chat/grouped-render.ts:565`
  - body: Binding `open` directly to the preference makes every rerender force the disclosure state. With the default off, a manually opened tool section can collapse again during streaming/history refresh; with it on, a manual collapse can be undone. Feed the preference into the existing expansion-state map instead.
  - confidence: 0.9
- **[P3] Disable the expand toggle consistently:** `ui/src/ui/app-render.helpers.ts:311-324`
  - body: The expand button only disables when tool calls are hidden, so it stays interactive during onboarding while the neighboring thinking/tool-call controls are locked. When tool calls are manually hidden, the desktop title also says setup is disabled. Align the disabled condition and use hidden-tool-call copy.
  - confidence: 0.83
- **[P3] Regenerate locale outputs:** `ui/src/i18n/locales/en.ts:384`
  - body: This adds a new English UI string but does not update the generated locale bundles or `.i18n` metadata required by the Control UI i18n pipeline. `pnpm ui:i18n:check` should stay clean after the feature lands.
  - confidence: 0.8

  - body: The new toolbar toggle is a user-facing Control UI feature, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for this kind of feature work.
  - confidence: 0.82
