## Summary

Keep open. The requested Control UI feature is still not implemented on current main, but this PR cannot merge as-is because the latest head has a malformed grouped-render expression, still bypasses main's tool-expansion state seam, and lacks required real behavior proof.

## What This Changes

The branch adds a persisted `chatExpandToolCalls` Control UI setting, desktop/mobile toolbar buttons, chat render plumbing, English i18n, and fixture updates.

## Best Possible Solution

Rebase the branch, add the persisted preference by feeding it into main's `autoExpandToolCalls`/`syncToolCardExpansionState` path, regenerate Control UI i18n outputs, and provide redacted real webchat proof with tool messages.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the malformed grouped-render ternary and the direct `open` binding path that would reset manual disclosure state on rerender.

## Solution Assessment

No. The feature direction is reasonable, but the implementation should reuse main's expansion-state map instead of binding `<details open>` directly, and the branch must first be syntactically repaired.
