Keep this PR open, but it needs changes before merge. The requested Control UI preference is valid and current main still lacks a persisted expand-by-default setting, but the patch is stale against the current stateful tool-expansion path and would regress manual expand/collapse behavior during rerenders.

## What This Changes

The PR adds a browser-local `chatExpandToolCalls` setting, desktop/mobile Control UI toolbar buttons, render plumbing, English copy, and fixture/test updates so tool-call output can expand by default.

## Best Possible Solution

Refresh the implementation on current main by adding the persisted browser preference, wiring it into the existing `autoExpandToolCalls` and `syncToolCardExpansionState` path, preserving manual disclosure state across rerenders, fixing accurate disabled-state copy for onboarding versus hidden tool calls, and adding the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on the PR branch, render a tool disclosure with the preference false, manually open it, then trigger a streaming/history rerender; Lit reapplies `?open=false` and collapses it again.

## Solution Assessment

No. The feature direction is right, but the patch should use the current stateful expansion seam instead of binding `open` directly, and it should align toolbar disabled states plus changelog policy before merge.
