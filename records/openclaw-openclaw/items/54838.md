## Summary

Keep open: current main still lacks the persisted Control UI preference, and the PR is useful source work, but the branch needs changes before merge because it forces disclosure state on rerender, has inconsistent toolbar disabled/copy behavior, and misses required i18n/changelog follow-through.

## What This Changes

The PR adds a persisted `chatExpandToolCalls` browser setting, desktop/mobile chat toolbar toggles, render plumbing, English copy, and test fixture updates for expanding Control UI tool-call output by default.

## Best Possible Solution

Refresh the feature against current main by wiring the persisted preference into `autoExpandToolCalls`/`syncToolCardExpansionState`, aligning toolbar disabled/copy behavior, regenerating i18n outputs, adding a changelog entry, and covering the state behavior with focused tests.

## Reproduction Assessment

Yes for the PR defect. Source inspection shows the PR binds `<details open>` to a global setting, so a streaming or history rerender can overwrite manual expansion/collapse state.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable solution is to seed current main's expansion-state map from the setting rather than binding the DOM `open` attribute directly, then finish the UI/i18n/changelog work.
