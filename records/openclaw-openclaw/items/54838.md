## Summary

Keep this PR open, but it is not merge-ready: the feature is useful and matches several Control UI tool-output requests, yet the current head is conflicting, lacks real behavior proof, and introduces concrete WebChat rendering regressions against the current expansion-state design.

## What This Changes

The PR adds a localStorage-backed `chatExpandToolCalls` setting, desktop/mobile chat toolbar toggle, render plumbing, English i18n text, and fixture updates so Control UI can expand tool-call output by default.

## Best Possible Solution

Rebase onto current main, route any default expansion through the existing `autoExpandToolCalls`/`tool-expansion-state` seam, decide how a persisted UI toggle coexists with session verbose level, and then add real Control UI proof.

## Reproduction Assessment

Yes, for the PR defects by source inspection: the head has a malformed grouped-render ternary and direct `<details open>` binding, while current main uses a disclosure-state map. I did not run a live browser session in this read-only review.

## Solution Assessment

No. The user-facing idea is reasonable, but the current implementation bypasses main's state map and competes with the verbose-level expansion path; the safer solution is to integrate through the existing expansion seam after a maintainer source-of-truth decision.
