## Summary

Keep open: current main still lacks the persisted Control UI expand-by-default preference, and this PR is useful but needs implementation and policy repairs before merge.

## What This Changes

The PR adds a persisted Control UI chat setting and desktop/mobile toolbar toggle to expand tool-call output by default, then threads that setting through chat rendering and tests.

## Best Possible Solution

Refresh the feature on current main by storing a browser preference that feeds the existing `autoExpandToolCalls` expansion-state seam, with consistent toolbar disabled copy, generated locale outputs, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the PR defect: manually open or collapse a tool disclosure, then trigger a streaming/history rerender and the direct `?open` binding reapplies the global preference.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable fix is to wire the new setting into current main's expansion-state map instead of binding `<details open>` directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve manual tool expansion state:** `ui/src/ui/chat/grouped-render.ts:565`
  - body: Binding `open` directly to the preference makes rerenders force every disclosure state. With the default off, a manually opened tool section can collapse again during streaming/history refresh; with it on, a manual collapse can be undone. Feed the preference into the existing expansion-state map instead.
  - confidence: 0.9
- **[P3] Disable the expand toggle consistently:** `ui/src/ui/app-render.helpers.ts:311-324`
  - body: The expand button only disables when tool calls are hidden, so it stays interactive during onboarding while neighboring chat controls are locked. When tool calls are manually hidden, the desktop title also says setup is disabled; align the disabled condition and use hidden-tool-call copy.
  - confidence: 0.83
- **[P3] Regenerate locale outputs:** `ui/src/i18n/locales/en.ts:384`
  - body: This adds a new English UI string without the generated locale bundles and `.i18n` metadata required by the Control UI i18n pipeline, so `pnpm ui:i18n:check` would be expected to fail after the feature lands.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `ui/src/ui/app-render.helpers.ts:308`
  - body: The new toolbar toggle is a user-facing Control UI feature, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for this kind of feature work.
  - confidence: 0.82
