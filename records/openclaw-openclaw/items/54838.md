## Summary

Keep open: current main still lacks a persisted Control UI preference for expanding tool calls by default, and this PR is useful but needs changes before merge because it bypasses the current expansion-state seam, has inconsistent toolbar disabled copy, and misses required i18n/changelog follow-through.

## What This Changes

The PR adds a persisted Control UI chat setting and desktop/mobile toolbar toggle to expand tool call and tool output sections by default, with render plumbing, English copy, and test fixture updates.

## Best Possible Solution

Refresh the feature on current main by adding a persisted browser preference that feeds `autoExpandToolCalls`/`syncToolCardExpansionState`, with aligned toolbar UX, generated locale updates, a changelog entry, and focused UI tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on the PR branch, render a tool disclosure with the preference false, manually open it, then trigger a streaming/history rerender; Lit reapplies `?open=false` and collapses it again.

## Solution Assessment

No. The feature direction is correct, but the implementation should seed the current expansion-state map instead of binding `open` directly, and it should align toolbar disabled states plus release/i18n policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve manual tool expansion state:** `ui/src/ui/chat/grouped-render.ts:565`
  - body: Binding `open` directly to the global preference makes each rerender force the disclosure state. With the default off, a manually opened tool section can collapse again during streaming/history rerenders; with it on, manual collapse can be undone. Seed the existing expansion-state map instead.
  - confidence: 0.89
- **[P3] Disable the expand toggle consistently:** `ui/src/ui/app-render.helpers.ts:311-324`
  - body: The expand button stays enabled during onboarding because `showToolCalls` is forced true there, and when tool calls are manually hidden the desktop title says setup is disabled. Align the disabled condition with the neighboring controls and use separate hidden-tools copy.
  - confidence: 0.8

  - body: This adds a new English UI string but does not update the generated locale bundles or `.i18n` metadata required by the Control UI i18n pipeline, so the i18n check should report drift.
  - confidence: 0.78
