## What This Changes

The PR adds a persisted Control UI `chatExpandToolCalls` browser setting, desktop/mobile toolbar buttons, render plumbing, English i18n, and fixture updates for expanding tool-call sections by default.

## Best Possible Solution

Add the persisted preference and toolbar control by feeding it into main's `autoExpandToolCalls` expansion-state map, regenerate i18n outputs, and provide redacted real Control UI proof with tool messages.

## Reproduction Assessment

Yes. Source inspection of the PR head gives a high-confidence reproduction path for the blocker: the latest grouped-render source is syntactically broken around the markdown/tool-card conditional, and the direct `open` bindings would override manual disclosure state on rerender.

## Solution Assessment

No. The feature direction is reasonable, but the current patch should be rewritten on top of main's expansion-state map instead of direct `<details open>` bindings, with the syntax error, disabled copy, mobile behavior, and generated i18n outputs fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restore the markdown branch before rendering tool cards:** `ui/src/ui/chat/grouped-render.ts:746-747`
  - body: The latest commit leaves a `${hasToolCards ? ...}` interpolation between `: markdown` and `: nothing`, so this file is no longer valid TypeScript/Lit syntax and the Control UI cannot build. Put the markdown rendering branch back and render tool cards after the conditional.
  - confidence: 0.97
- **[P2] Seed the expansion map instead of binding open:** `ui/src/ui/chat/grouped-render.ts:568`
  - body: Binding `open` directly to the new preference reapplies that preference on every render, so a streaming/history rerender can collapse a section the user opened or reopen one they collapsed. Feed the setting into the existing per-session expansion-state map instead.
  - confidence: 0.9
- **[P3] Use the right disabled state and copy:** `ui/src/ui/app-render.helpers.ts:324`
  - body: When tool calls are manually hidden, the desktop title still says the control is disabled during setup, and the mobile control is not locked during onboarding like the neighboring toggles. Split hidden-tool-call copy from onboarding copy and share the disabled condition on both controls.
  - confidence: 0.84
- **[P3] Regenerate locale outputs:** `ui/src/i18n/locales/en.ts:387`
  - body: This adds an English Control UI string without updating the generated locale bundles and `.i18n` metadata required by the scoped UI i18n pipeline, so the i18n generated outputs will drift after merge.
  - confidence: 0.8
