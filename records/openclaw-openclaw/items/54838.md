## Summary

Keep open: the feature request is valid and current main still lacks a persisted Control UI preference, but this PR is not merge-ready because it binds disclosure state directly, misses required i18n/changelog follow-through, and has toolbar disabled-copy defects.

## What This Changes

The PR adds a persisted `chatExpandToolCalls` setting, desktop/mobile toolbar controls, render plumbing, English copy, and test fixture updates to expand Control UI tool-call output by default.

## Best Possible Solution

Refresh the feature on current main by adding a persisted browser preference that feeds the existing `autoExpandToolCalls` / `syncToolCardExpansionState` path, with accurate toolbar disabled copy, generated i18n updates, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear on the PR branch: with the preference false, manually open a tool disclosure and trigger a streaming/history rerender; Lit reapplies `?open=false` and collapses it again.

## Solution Assessment

No. The feature direction is right, but the implementation should seed the current expansion-state map instead of binding `open` directly, and it should align toolbar disabled states plus release/i18n policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve manual tool expansion state:** `ui/src/ui/chat/grouped-render.ts:565`
  - body: Binding `open` directly to the global preference makes every rerender force the DOM disclosure state. With the default off, a manually opened tool section can collapse again during streaming/history rerenders; with it on, manual collapse can be undone. Seed the expansion state instead of binding the `<details>` attribute directly.
  - confidence: 0.89
- **[P3] Disable the expand toggle consistently:** `ui/src/ui/app-render.helpers.ts:311-324`
  - body: The expand button stays enabled during onboarding because `showToolCalls` is forced true there, and when tool calls are manually hidden the desktop title says setup is disabled. Align the disabled condition with the neighboring controls and use separate hidden-tools copy.
  - confidence: 0.8

  - body: This adds a new English UI string but does not update the generated locale bundles or `.i18n` metadata required by the Control UI i18n pipeline. `pnpm ui:i18n:check` will report drift until the generated outputs are committed.
  - confidence: 0.78

  - body: This is a user-facing Control UI feature, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires one for user-visible feature work, with contributor credit where appropriate.
