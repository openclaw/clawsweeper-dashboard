## What This Changes

The issue reports that Feishu/Lark setup on Synology Docker became unavailable after upgrade because unsafe plugin directory permissions make the plugin unavailable, while safer permissions appear to block setup config writes.

## Best Possible Solution

Keep the plugin permission boundary safe while giving Docker/NAS users a documented, tested setup path that does not require world-writable plugin directories.

## Reproduction Assessment

Partially: current and latest-release source show a world-writable plugin path can still block plugin discovery when chmod repair fails, which matches the reported unavailable-plugin shape. I did not live-reproduce the Synology Docker write-permission loop.

## Solution Assessment

No; broadly allowing 777 is not clearly the best fix because it weakens a plugin execution security boundary. A safer fix would address Docker/NAS ownership and config-write setup, or add an explicit maintainer-approved trusted override with strong diagnostics.
