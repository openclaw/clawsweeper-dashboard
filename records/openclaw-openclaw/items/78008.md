## Summary

Keep open: current main still blocks world-writable non-bundled plugin paths, and the Feishu setup flow can still emit the reported `feishu plugin not available` note when the scoped plugin cannot load. The requested 0777 trust override conflicts with the documented plugin path-safety model and needs maintainer judgment.

## What This Changes

The issue asks for Feishu setup in Docker/Synology to recover from plugin permission blocking after upgrade, possibly by letting trusted plugins use 0777 permissions.

## Best Possible Solution

Keep world-writable plugin execution blocked, but add a supported setup/doctor repair path that tightens trusted installed Feishu plugin and state-directory permissions while preserving config writes.

## Reproduction Assessment

No, not end-to-end: I did not run Synology Docker or the Feishu QR setup. Source inspection does give a clear reproduction slice for the blocking behavior: non-bundled 0777 plugin paths are rejected and the setup flow can emit the reported availability note when the scoped plugin is not loaded.

## Solution Assessment

No, not as requested: a user-controlled 0777 bypass is not the narrowest maintainable fix because current docs and code intentionally block world-writable plugin paths. A safer fix would repair or diagnose the installed plugin/config ownership loop instead of relaxing the loader gate.
