## Summary

Keep open: current main has not implemented the requested TUI follow-up busy state, and this draft PR is not merge-ready because it scans rendered assistant text while the real `sessions_yield` signal is hidden metadata.

## What This Changes

The branch adds an `awaiting follow-up` TUI activity state, marker-based live final/history restore detection, and focused tests in TUI event/session actions.

## Best Possible Solution

Expose a typed follow-up-wait signal from the sessions_yield custom/metadata path to both live TUI finals and history restore, while keeping hidden context out of rendered assistant text and adding regression coverage plus changelog.

## Reproduction Assessment

Yes. Source inspection shows `sessions_yield` writes a hidden `openclaw.sessions_yield` custom message, `chat.history` projects only display messages, and the current TUI final-without-message path finalizes as `idle`.

## Solution Assessment

No. The proposed marker scan is not the best fix because real sessions carry the signal outside rendered assistant text; a typed metadata/projection signal is the narrower maintainable repair.
