## Summary

Keep open: current main still drops yielded follow-up waits back to idle, but this draft branch remains blocked by missing real TUI proof and two correctness issues in the proposed implementation.

## What This Changes

The branch adds an `awaiting follow-up` TUI activity state, derives it from a `sessions_yield` marker in final output/history, renders it as busy, and adds focused tests.

## Best Possible Solution

Drive `awaiting follow-up` from structured yielded/liveness metadata and hidden `openclaw.sessions_yield` history records, then cover event handling, history restore, clearing, and real TUI behavior proof.

## Reproduction Assessment

Yes, source-level: current main finalizes successful chat finals and lifecycle `end` events to `idle` while `sessions_yield` persists hidden context and emits yielded/liveness metadata. I did not run a live TUI in this read-only review.

## Solution Assessment

No. The visible busy state is the right direction, but the branch should consume structured yielded signals and hidden history state instead of scanning visible assistant text for a hard-coded marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the structured yielded signal:** `src/tui/tui-event-handlers.ts:268-269`
  - body: The new check only sets `awaiting follow-up` when finalized assistant text contains the literal context marker. Real `sessions_yield` persists that context as hidden `openclaw.sessions_yield` state and emits `yielded`/`livenessState` metadata, so yielded runs can still finalize or restore as idle when the marker is not visible text.
  - confidence: 0.9
