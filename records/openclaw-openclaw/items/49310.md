## Summary

Keep open. This is a plausible TUI bug fix, but the patch is not merge-ready because it scans a user-visible assistant marker that current `sessions_yield` stores as hidden custom context, the added tests are stale against current TUI types/package imports, and the PR lacks real behavior proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, detects the `sessions_yield` context marker in final/history assistant text, and adds focused TUI tests.

## Best Possible Solution

Rebase and rework the TUI status change to consume an explicit yielded/paused session signal, likely the `pauseReason`/lifecycle state from https://github.com/openclaw/openclaw/pull/75662, then prove the real TUI waiting state with redacted terminal output or a short recording.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows current TUI has no `sessions_yield` busy state and the runner stores the marker as hidden custom context, so the failure mode is plausible but not proven from a real setup in this review.

## Solution Assessment

No. Parsing final assistant text is not the best fix because current `sessions_yield` state is hidden custom/runtime metadata; the safer maintainable path is to use explicit yielded or paused session state and update the TUI tests against current types.
