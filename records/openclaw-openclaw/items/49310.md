## Summary

Keep open: the PR targets a real TUI yielded-wait gap, but the patch still keys the state off literal display text instead of the structured yielded/liveness contract, the added tests reference stale or missing types, and the external PR still lacks real TUI behavior proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI busy state by scanning final/history text for a `sessions_yield` context marker, renders that state as busy, and adds focused TUI tests.

## Best Possible Solution

Drive the TUI wait indicator from structured yielded/liveness or paused-session state, then align that behavior with the broader parent-session progress contract before merging.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful TUI chat finals to `idle` while `sessions_yield` persists hidden context and emits yielded/liveness metadata that the TUI does not consume.

## Solution Assessment

No. A visible busy state is the right direction, but hard-coded marker scanning is brittle; live events and history restore should consume structured yielded/liveness or paused-session state.
