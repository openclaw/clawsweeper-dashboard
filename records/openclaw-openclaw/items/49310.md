Keep open: the TUI follow-up busy-state bug is source-reproducible on current main, but this PR should not merge as-is because it relies on visible marker text instead of the structured yielded/follow-up signals and still lacks real TUI behavior proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, detects the `sessions_yield` marker in final/history text, and adds focused TUI tests.

## Best Possible Solution

Consume structured yielded/paused lifecycle metadata for live events and hidden `openclaw.sessions_yield` custom history records on restore, then prove the visible busy state with focused tests and redacted real TUI output.

Yes, source-level: current main finalizes successful TUI final and lifecycle-end events to `idle`, while the runner emits yielded/paused metadata and persists hidden `sessions_yield` context. I did not run a live TUI in this read-only review.
