## Summary

Keep this PR open because it addresses a real current-main TUI gap, but the proposed patch is not merge-ready: it keys follow-up busy state off duplicated assistant text instead of the typed lifecycle and persisted session-yield signals that the runtime already emits.

## What This Changes

The draft PR adds an `awaiting follow-up` TUI activity state, detects a hard-coded `sessions_yield` context marker in final assistant/history text, updates the status renderer, and adds focused event/history tests.

## Best Possible Solution

Revise the TUI follow-up status so live runs derive the pending wait from stable terminal lifecycle metadata (`yielded: true`, `livenessState: "paused"`, and/or `stopReason: "end_turn"`), history restore detects the persisted `openclaw.sessions_yield` custom context through a shared helper or constant, and stale follow-up state is cleared on continuation, session switch, and restores without a pending marker.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an active TUI run that receives a terminal lifecycle event with `{ phase: "end", yielded: true, livenessState: "paused" }`, or a restored history containing the hidden `openclaw.sessions_yield` custom message; current main maps the live event to `idle` and ignores the custom history marker.

## Solution Assessment

No. The PR touches the right TUI surfaces, but matching a duplicated literal in assistant text is not the best fix when current runtime code already provides typed lifecycle metadata and a persisted custom context message.

Reloading real session history can therefore miss the pending follow-up state even though the marker exists in the transcript.
  - confidence: 0.84
