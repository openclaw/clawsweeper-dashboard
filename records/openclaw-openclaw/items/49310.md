## Summary

Keep open: the bug remains source-reproducible on current main, but this draft patch is not merge-ready because it keys off a literal visible marker instead of the structured yielded and hidden-history signals, and it lacks real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI busy state, detects a `sessions_yield` marker in final/history text, and adds focused TUI tests.

## Best Possible Solution

Have the TUI derive awaiting-follow-up from lifecycle yielded/paused metadata and hidden `openclaw.sessions_yield` history records, then verify the real TUI with redacted runtime output.

## Reproduction Assessment

Yes, source-level: current main sets active chat finals and lifecycle `end` events to idle even though the runner emits `yielded: true` / `livenessState: "paused"` and persists hidden sessions_yield context. I did not run a live TUI.

## Solution Assessment

No: scanning final assistant text is not the narrow reliable fix because the durable signals are structured lifecycle metadata and hidden custom history records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the structured yielded signal:** `src/tui/tui-event-handlers.ts:265-266`
  - body: The new check only sets `awaiting follow-up` when finalized assistant text contains the literal context marker. Real `sessions_yield` terminal events carry `yielded: true` / `livenessState: "paused"`, so yielded runs without that visible prose can still finalize as idle.
  - confidence: 0.9
- **[P2] Restore from the hidden sessions_yield history record:** `src/tui/tui-session-actions.ts:367`
  - body: History restore now looks only at the last assistant text, but `sessions_yield` persists its follow-up marker as a hidden custom `openclaw.sessions_yield` message. Reloading a real yielded session can miss that custom state and clear the TUI back to idle.
  - confidence: 0.88
