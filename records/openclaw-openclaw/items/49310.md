## Summary

Keep open, but this draft PR is not merge-ready: the visible busy-state direction is plausible, but the patch relies on scanning rendered marker text, misses the structured yielded/liveness and hidden history contract, uses a stale TUI package name in added tests, and still lacks after-fix real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, detects a `sessions_yield` marker in final output and history, renders it as busy, and adds focused TUI tests.

## Best Possible Solution

Drive the TUI wait indicator from structured `yielded`/`livenessState` and hidden or persisted `sessions_yield` session state, fix the test imports, add real TUI proof, and coordinate broader progress/streaming semantics with https://github.com/openclaw/openclaw/issues/58067.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful final chat events and lifecycle `end` events to `idle` while sessions_yield emits structured yielded/liveness metadata and hidden context that the current TUI does not render as a wait state.

## Solution Assessment

No. The visible busy state is the right direction, but the PR should consume structured yielded metadata and hidden or persisted session state instead of scanning display text for a hard-coded marker.
