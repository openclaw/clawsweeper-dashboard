## Summary

Keep open: the reported TUI state problem is still plausible on current main, but this draft PR is not ready because it lacks real after-fix TUI proof and relies on visible text instead of the structured sessions_yield metadata and hidden history record.

## What This Changes

The PR adds an `awaiting follow-up` TUI busy state, detects a `sessions_yield` marker in final assistant text/history restore, and adds focused TUI tests.

## Best Possible Solution

Have the TUI derive awaiting-follow-up from lifecycle `yielded`/`paused` metadata and hidden `openclaw.sessions_yield` history records, keep that state visibly busy, and clear it after the follow-up completes.

## Reproduction Assessment

Yes, source-level: current main has final and lifecycle terminal paths that set the TUI to idle while the runner emits `yielded: true` and `livenessState: "paused"`. I did not run a live TUI reproduction.

## Solution Assessment

No: the proposed text-marker scan is not the best fix because the reliable signals are structured lifecycle metadata and the hidden custom sessions_yield history record.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the structured yielded signal:** `src/tui/tui-event-handlers.ts:265-266`
  - body: The new check only sets `awaiting follow-up` when finalized assistant text contains the literal context marker. Real yielded runs carry `yielded: true` / `livenessState: "paused"`, and the marker is persisted separately, so yielded turns can still finalize as idle.
  - confidence: 0.9
- **[P2] Restore from the hidden sessions_yield record:** `src/tui/tui-session-actions.ts:367`
  - body: History restore now scans only the last assistant text, but sessions_yield persists its follow-up marker as a hidden custom `openclaw.sessions_yield` message. Reloading a real yielded session can miss that state and clear the TUI back to idle.
  - confidence: 0.88
