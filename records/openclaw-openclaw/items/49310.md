## Summary

Keep open: the PR targets a real TUI follow-up wait bug, but it is not merge-ready because it keys the busy state off visible marker text instead of the structured yield contract, adds tests with a stale TUI package reference, and lacks real TUI proof.

## What This Changes

The branch adds an `awaiting follow-up` TUI busy state, derives it from a hard-coded `sessions_yield` marker in final output/history, updates status rendering, and adds focused TUI tests.

## Best Possible Solution

Drive `awaiting follow-up` from lifecycle `yielded`/`livenessState` and hidden `openclaw.sessions_yield` or paused-session state, fix the test package reference, and keep broader progress/streaming work coordinated with https://github.com/openclaw/openclaw/issues/58067.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful final chat events and lifecycle `end` events to `idle` while sessions_yield emits structured yielded/liveness metadata and hidden context that the current TUI does not render as a wait state.

## Solution Assessment

No. The visible busy state is the right direction, but the PR should consume structured yielded metadata and hidden history/session state instead of scanning display text for a hard-coded marker.
