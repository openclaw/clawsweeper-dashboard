## Summary

Keep open: current main still has the TUI idle-state gap for yielded follow-up waits, but this draft PR relies on brittle marker-text detection instead of the structured yielded/liveness metadata and still lacks real behavior proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state based on a literal `sessions_yield` context marker in final assistant text/history and updates focused TUI tests.

## Best Possible Solution

Keep the TUI busy whenever structured terminal metadata or hidden session history says a `sessions_yield` wait is still pending, with focused tests and redacted real TUI proof.

## Reproduction Assessment

Yes, source-level: current main finalizes successful chat finals and lifecycle end events to `idle` while the runner persists hidden `sessions_yield` context and emits yielded/liveness metadata. I did not run a live TUI in this read-only review.

## Solution Assessment

No. The PR should consume structured yielded/paused lifecycle metadata for live events and hidden `openclaw.sessions_yield` custom history records on restore, rather than scanning visible assistant text for a hard-coded marker.
