## Summary

Keep open: the TUI idle gap is source-reproducible, but this PR keys the new busy state off a literal assistant-text marker that current `sessions_yield` stores as hidden custom state, so real yielded runs can still miss the state and the PR also lacks inspectable real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, marker-based live/history detection for `sessions_yield` waits, and focused TUI tests.

## Best Possible Solution

Consume a structured yielded/paused session signal in both TUI live events and history restore, then verify it with focused tests plus a real redacted TUI smoke showing the busy state until the follow-up arrives.

## Reproduction Assessment

Yes, source-reproducible. Current main finalizes successful chat finals to `idle` and history restore ignores custom messages, while `sessions_yield` persists its context marker as hidden custom state and runner metadata already carries yielded/paused state; no live TUI run was executed in this read-only review.

## Solution Assessment

No. The assistant-text marker check is not the narrowest reliable fix because the real yield path stores the marker as hidden custom state; the safer fix is to consume structured yielded/paused metadata or the actual custom history record.
