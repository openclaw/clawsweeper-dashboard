## Summary

Keep open. Current main still drops yielded follow-up waits back to idle, so the PR addresses a real TUI gap, but the proposed patch is not merge-ready because it keys live and restored state off hard-coded assistant text instead of the existing lifecycle metadata and hidden persisted yield context.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, detects a `sessions_yield` context marker in final assistant/history text, updates the status renderer, and adds focused event/history tests.

## Best Possible Solution

Land a revised TUI fix that treats yielded/paused terminal lifecycle events as a busy follow-up wait, restores that state from the hidden `openclaw.sessions_yield` custom history record, clears stale follow-up state on continuation/session changes, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an active TUI run receiving a terminal lifecycle event with `{ phase: "end", yielded: true, livenessState: "paused" }`, or a restored history containing a hidden `openclaw.sessions_yield` custom message; current main maps the live event to `idle` and ignores the custom history marker.

## Solution Assessment

No. The PR touches the right TUI surfaces, but matching duplicated assistant text is not the narrowest maintainable fix when current runtime code already exposes typed lifecycle metadata and a persisted custom context record.
