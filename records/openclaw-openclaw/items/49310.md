## Summary

Keep open: the TUI idle-looking gap after `sessions_yield` still looks like a valid current-main problem, but this draft patch appears stale against the current sessions_yield transcript shape and would not reliably detect real yielded runs.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, infers it from the `sessions_yield` context marker in final/history assistant text, and adds focused TUI tests.

## Best Possible Solution

Represent the yielded-follow-up wait using a current-main signal shared by live chat and history restore, keep it busy until a follow-up arrives or the session changes, and add real-shape regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main stores the sessions_yield marker in a hidden custom message while TUI final handling and history restore look only at chat final assistant text or rendered assistant history, so the idle gap is source-reproducible without a live run.

## Solution Assessment

No. The patch watches final/history assistant text for a marker, but current main strips the aborted assistant artifact and persists the marker as hidden custom context, so the repair should consume a reliable yielded-run or history signal instead.
