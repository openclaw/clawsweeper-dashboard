## Summary

Keep open: the underlying TUI busy-state gap is source-reproducible, but this draft PR detects a hidden `sessions_yield` marker from assistant text that the current live and history paths do not expose.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, marker-based detection in live final events and history restore, and focused TUI tests.

## Best Possible Solution

Derive `awaiting follow-up` from the actual `sessions_yield` custom/metadata signal for both live finals and history reloads, keep hidden context out of rendered assistant text, and add regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_yield` stores the marker as a hidden custom transcript entry, `chat.history` projects only display messages, and the TUI final-without-message path currently finalizes as `idle`.

## Solution Assessment

No. Matching a literal marker in rendered assistant text is not the best fix because current main carries the signal as hidden custom metadata; a typed or projected follow-up-wait signal is the safer repair.
