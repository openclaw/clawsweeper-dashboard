## What This Changes

The branch adds an `awaiting follow-up` TUI activity state, sets it from final assistant/history text containing the `sessions_yield` marker, updates status rendering, and adds focused TUI tests.

## Best Possible Solution

Drive the TUI wait indicator from structured yielded/liveness or paused-session state, restore it from hidden/custom/session state, and keep broader progress streaming coordinated with the open tracker.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful TUI chat events to idle while `sessions_yield` emits yielded/liveness metadata and hidden context that the current TUI does not render as a wait state.

## Solution Assessment

No. The visible busy state is the right direction, but the PR should consume structured yielded/session state instead of scanning display text for a hard-coded marker, and the tests need the current TUI package.
