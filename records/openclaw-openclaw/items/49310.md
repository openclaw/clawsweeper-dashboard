## Summary

Keep open: the PR targets a real TUI status gap, but the patch keys off a marker in rendered assistant text while current `sessions_yield` persists that marker as a hidden custom transcript message that the TUI history projection does not expose.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity status, detects a `sessions_yield` marker in assistant text, restores or clears that status during history load, and adds focused TUI tests.

## Best Possible Solution

Rework the PR to derive `awaiting follow-up` from the real `sessions_yield` event or transcript metadata contract, keep the hidden context text out of the chat log, cover the actual custom-message history shape, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_yield` stores the marker as a hidden `custom_message`, `chat.history` drops that entry, and the TUI final-without-message path still finalizes as `idle`.

## Solution Assessment

No. Matching a literal marker in rendered assistant text is not the narrowest maintainable fix because the current runner/history path does not expose that marker as assistant text; a typed or projected follow-up-wait signal would be safer.
