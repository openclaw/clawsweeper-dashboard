## Summary

Keep this PR open for author revision: the reported TUI idle gap is source-reproducible, but the proposed fix is not reliable because it keys off visible assistant text instead of the structured `sessions_yield` state that current main already emits and persists.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state by detecting a `sessions_yield` marker in final assistant/history text, and adds focused TUI tests.

## Best Possible Solution

Consume structured yielded/paused lifecycle metadata for live events and hidden `openclaw.sessions_yield` custom history records on restore, then prove the visible busy state with focused tests and redacted real TUI output.

Yes, source-level: current main finalizes successful TUI final and lifecycle-end events to `idle`, while the runner emits yielded/paused metadata and persists hidden `sessions_yield` context. I did not run a live TUI in this read-only review.

No. Scanning final assistant text is not the narrow reliable fix because the real yield state is carried by lifecycle metadata and hidden custom history, not guaranteed visible prose.
