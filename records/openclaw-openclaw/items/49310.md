## Summary

Keep open: current main still has a source-level TUI yielded-wait visibility gap, but this draft PR is not merge-ready because it relies on visible marker text instead of structured yielded signals and still lacks real behavior proof.

## What This Changes

The draft PR adds an `awaiting follow-up` TUI busy state, detects a literal `sessions_yield` context marker in final/history text, and adds focused tests.

## Best Possible Solution

Represent yielded follow-up waits through typed lifecycle/history metadata in the TUI, coordinated with the broader status-contract work in https://github.com/openclaw/openclaw/issues/56856 and https://github.com/openclaw/openclaw/issues/58067.

## Reproduction Assessment

Yes, source-level: current main finalizes chat finals and lifecycle end events to `idle` while `sessions_yield` persists hidden custom context and emits `yielded`/`livenessState`. I did not run a live TUI in this read-only review.

## Solution Assessment

No. The PR should consume structured yielded/paused lifecycle metadata and hidden `openclaw.sessions_yield` history records instead of scanning visible assistant text for a hard-coded marker.
