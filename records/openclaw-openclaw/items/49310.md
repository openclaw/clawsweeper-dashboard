## Summary

Keep open: this is a plausible narrow bug fix, but the current draft still keys off visible marker text instead of the structured yielded contract, adds test code that references a stale TUI package name, and lacks contributor-supplied real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, derives it from `sessions_yield` marker text in final assistant output/history, renders it as busy, and adds focused tests.

## Best Possible Solution

Drive `awaiting follow-up` from structured yielded/liveness metadata and hidden session-yield history state, fix the test package reference, and add focused tests plus redacted real TUI proof.

## Reproduction Assessment

Yes, source-level: current main finalizes successful chat finals and lifecycle end events to `idle` while `sessions_yield` emits yielded/liveness metadata and persists hidden custom context. I did not run a live TUI in this read-only review.

## Solution Assessment

No. The visible busy state is the right direction, but the implementation should consume structured yielded signals and hidden history state instead of scanning visible assistant text for a hard-coded marker.
