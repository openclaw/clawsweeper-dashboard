## Summary

Keep open. The source-level bug is still plausible on current main, but this draft PR is not merge-ready because it derives the wait state from visible marker text instead of structured yielded/session state, still has a stale test package reference, and lacks real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state, detects a `sessions_yield` context marker in final assistant/history text, renders that as a busy status, and adds focused TUI tests.

## Best Possible Solution

Drive the TUI wait indicator from structured yielded/liveness or paused-session state and coordinate it with the broader open issue tracking parent-session progress visibility.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful TUI chat events to `idle` while the yield path carries structured `yielded`/`livenessState` metadata and hidden `openclaw.sessions_yield` context that the TUI does not currently map to a visible wait state.

## Solution Assessment

No. A visible busy state is the right user-facing direction, but this PR should consume structured yield/session state rather than scanning display text for a hard-coded context marker.
