## Summary

Keep open, but not merge-ready: the source-level bug is real, while this patch relies on marker text instead of structured yield/liveness state and still lacks real TUI proof.

## What This Changes

The PR adds an `awaiting follow-up` TUI activity state by scanning final/history text for a `sessions_yield` marker, updates busy-status rendering, and adds focused TUI tests.

## Best Possible Solution

Drive the TUI wait indicator from structured yielded/liveness or paused-session state, then align it with the broader parent-session progress contract before merging.

## Reproduction Assessment

Yes, at source level. Current main finalizes successful TUI chat finals to `idle` while `sessions_yield` persists hidden context and emits yielded/liveness metadata that the TUI does not consume.

## Solution Assessment

No. A visible busy state is the right direction, but hard-coded marker scanning is brittle; live events and history restore should consume structured yield/liveness state.
