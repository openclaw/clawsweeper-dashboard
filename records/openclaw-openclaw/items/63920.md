## Label Justifications

- `impact:message-loss`: Slack delivers shortcut interactions, but OpenClaw currently has no listener that turns them into system events or agent turns.

## Summary

Keep this open: current main and the latest release still wire Slack block actions and modal lifecycle events, but not Bolt shortcut events, so registered Slack global and message shortcuts still have no OpenClaw system-event path. The request fits the Slack plugin boundary and is narrow, with the main implementation constraints being immediate Slack ACK and a clear no-channel routing choice for global shortcuts.

## What This Changes

Expose Slack global shortcut and `message_action` payloads through the Slack channel plugin as routed, compact system events.

## Best Possible Solution

Add a Slack-plugin-owned shortcut handler that ACKs immediately, filters/authenticates like other interactive events, emits compact sanitized system events for global and message shortcuts, and updates focused Slack tests and docs without expanding into full modal/workflow APIs.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: current Slack interaction registration has block-action and modal handlers, while `rg` finds no `app.shortcut`/`message_action` path in the Slack plugin or tests.

## Solution Assessment

Yes. A Slack-plugin-local handler is the narrowest maintainable solution; it should reuse the existing sanitizer/session/event patterns and explicitly handle immediate ACK plus global-shortcut routing.
