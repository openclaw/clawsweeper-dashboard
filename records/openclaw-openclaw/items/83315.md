## Summary

Keep open: the linked bug is valid and this PR is directionally useful, but it should not merge as-is because the widened copy rule also preserves blocked prototype keys and the supplied proof is still unit-test/check output rather than a real doctor repair run.

## What This Changes

The PR changes the legacy web search doctor migration to preserve unknown record-valued `tools.web.search` entries and adds a regression test for that case.

## Best Possible Solution

Land one focused fix for https://github.com/openclaw/openclaw/issues/83287 that preserves unknown search records, drops blocked prototype keys, and proves a real doctor migration.

## Reproduction Assessment

Yes, source-level. Current main reaches the migration loop for a config with `tools.web.search.apiKey` plus a custom record, then omits that custom record because only `openaiCodex` records or non-record values are copied.

## Solution Assessment

No, not as-is. Preserving unknown records is the right direction, but the maintainable fix also needs the existing blocked-prototype-key guard pattern and real doctor migration proof.
