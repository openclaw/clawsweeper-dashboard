## Summary

Keep open. Current main still accepts and acknowledges model switches without comparing the carried-over session context against the target model's effective budget or the reported account-tier TPM/rate-limit risk; existing context safeguards run later in the prompt path, and related #58957 remains broader and open rather than a merged superseding fix.

## What This Changes

Request that `/model` warn or offer compaction when the current session context is large relative to the target model's context or rate-limit characteristics, including fallback and per-model context override considerations.

## Best Possible Solution

Add a shared model-switch context-risk warning used by WebChat and messaging directives. It should resolve the target model's effective `contextTokens`/`contextWindow`/`maxTokens`, compare that with the freshest prompt or transcript usage available for the current session, and recommend `/compact` or `/new` before treating the switch as safe; account-tier TPM prediction and automatic compaction should be explicit follow-up policy decisions.

## Reproduction Assessment

Yes for the observable missing warning: inspect or create a large-context session, invoke `/model <provider/model>` through WebChat or directive-only chat, and current main will persist and acknowledge the switch without a switch-time context-risk comparison. The exact HTTP 429 case depends on provider account tier and quota state.

## Solution Assessment

Unclear as written. A switch-time warning is the right product direction, but the safe first fix should avoid automatic compaction and account-specific quota prediction until maintainers choose thresholds and UX policy.
