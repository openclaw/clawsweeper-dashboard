## Summary

Keep open. Current main still makes browser screenshots model-visible while also exposing the screenshot path to the trusted tool-result media delivery path, and the Pinchy comment corroborates real shared-channel impact; the related open PR at https://github.com/openclaw/openclaw/pull/75762 adds another media hint rather than a suppression control.

## What This Changes

Add a privacy control so browser screenshot results can remain visible to the model without being automatically attached to visible chat replies.

## Best Possible Solution

Add a central tool-result media visibility gate before queued media reaches channel replies, preserving browser screenshots for the model while giving maintainers an explicit default and suppression surface.

## Reproduction Assessment

Yes, by source inspection. A browser screenshot calls imageResultFromFile, imageResult stores details.media.mediaUrl, browser is trusted for local tool-result media, and the subscribe pipeline queues that media into visible replies without a privacy gate.

## Solution Assessment

Unclear on the exact product shape. The safest boundary is before queuePendingToolMedia or reply consumption, but maintainers need to choose whether the control is a screenshot parameter, tool-level policy, channel policy, or global default.
