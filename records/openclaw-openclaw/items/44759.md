## Summary

Keep open. Current main still preserves browser screenshot images for model vision and also turns the screenshot file path into trusted tool-result media that is queued into outbound replies; the provided Pinchy comment is useful corroboration that this affects real shared-channel browser automation workflows.

## What This Changes

Add a way to keep browser screenshot tool images visible to the model while suppressing automatic delivery of those screenshots to chat channels.

## Best Possible Solution

Implement a core tool-result media visibility policy that can suppress channel delivery while preserving model-visible tool content. The narrow first step should support browser screenshot private delivery, with tests proving suppressed screenshots keep the image block for the model and do not enter pending channel media, while intentionally public/generated media still delivers.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to invoke browser screenshot in a channel session: current main returns details.media.mediaUrl for the screenshot path, trusts browser media, queues it, and tests assert queued tool media is attached or flushed to outbound replies.

## Solution Assessment

Yes, the reporter's per-call private screenshot option is a good minimal fix, but it should be implemented through the shared pending-tool-media pipeline so the visibility decision is explicit before queuePendingToolMedia runs.
