## Summary

Keep this PR open, but it is not merge-ready: the feature is still absent from current main and the PR’s useful embedded-runner path does not yet provide a consistent access-control contract for CLI/loopback-backed `/v1/responses` runs. The contributor fixed the Greptile-reported memory/runtime group docs, but the remaining security-boundary and real-proof blockers still stand.

## What This Changes

The PR adds an optional `/v1/responses` `tool_deny` request field, documents it, and threads it from OpenResponses ingress into embedded PI tool construction.

## Best Possible Solution

Define one run-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical tool catalog, and require live gateway proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the field is accepted and forwarded through generic OpenResponses ingress, while CLI runner and loopback tool resolution have no per-request deny input. I did not run a live gateway repro because this was a read-only review.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable part of the solution, but the public API must either enforce denial on every supported backend or reject unsupported backends fail-closed.
