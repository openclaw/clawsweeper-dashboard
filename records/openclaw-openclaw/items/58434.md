## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested `/v1/responses` `tool_deny` surface, and the embedded-runner direction is plausible, but the patch appears to introduce a TypeScript parameter mismatch on the CLI-runner path and it omits the required changelog entry for a user-facing feature.

## What This Changes

The PR adds an optional `tool_deny` request field to `/v1/responses`, threads it through Gateway and agent run parameters into the embedded tool policy pipeline, and documents per-request built-in tool or group denial.

## Best Possible Solution

Land a small additive OpenResponses extension only after the tool-deny override is consistently typed and enforced across supported run paths, the release notes document the new request field, and focused schema/tool-policy tests prove the intended ephemeral deny behavior.

## Reproduction Assessment

Yes. The primary review finding is reproducible by comparing the PR-added `toolDenyOverride` property on the `runCliAgent` object literal with the current `RunCliAgentParams` definition, which lacks that property.

## Solution Assessment

No, not as-is. Reusing the existing deny/group-expansion pipeline is the right narrow implementation for embedded runs, but the PR needs a type-safe CLI boundary and changelog coverage before it is a maintainable solution.
