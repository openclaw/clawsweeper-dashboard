## Summary

Keep open: this is a useful OpenResponses feature, but the PR is not merge-ready because the request-scoped deny contract is only wired into embedded tool construction, the CLI/loopback path can still present a false access-control guarantee, and real gateway proof is missing.

## What This Changes

Adds an optional `/v1/responses` `tool_deny` request field, documents it, and threads it from gateway ingress into embedded PI tool policy construction.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical catalog, and require real gateway proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows `tool_deny` is accepted and forwarded from OpenResponses ingress, while CLI runner and loopback tool resolution have no matching per-request deny input. I did not run a live gateway repro because this was a read-only review.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable part of the solution, but the public API must either enforce denial on every supported backend or reject unsupported backends fail-closed.
