## Summary

Keep open: the API direction is useful, but the patch is not merge-ready because the proposed deny list is not enforced or rejected across all supported run backends, the new docs overstate group semantics, and contributor-side real gateway proof is still missing.

## What This Changes

The PR adds a `tool_deny` array to the `/v1/responses` request schema/docs and forwards it through agent command options into embedded PI tool-policy filtering.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical catalog, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR accepts and forwards `tool_deny`, while current CLI-backed tool resolution has no matching per-request deny input or fail-closed guard. I did not run live gateway proof because this was a read-only review and the contributor has not provided real runtime evidence.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable piece, but the public API must either enforce denial on every supported backend or reject unsupported backends fail-closed.
