## Summary

Keep open: current main does not implement this field, and the PR still has blocking enforcement, documentation, and proof gaps for an access-control-looking API.

## What This Changes

The PR adds an optional `tool_deny` field to `/v1/responses`, forwards it through agent command execution into embedded PI tool construction, and documents per-request built-in tool denial.

## Best Possible Solution

Define one run-scoped deny contract that is enforced for every supported `/v1/responses` backend or rejected fail-closed, with canonical group documentation and live gateway proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the field is accepted and forwarded through generic ingress, but only the embedded runner consumes it and CLI/loopback tool resolution has no per-request deny input.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is reasonable, but the public API must either enforce denial on all supported backends or fail closed and must document exact group semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override for CLI-backed runs:** `src/gateway/openresponses-http.ts:448`
  - body: This accepts `tool_deny` on the generic `/v1/responses` path, but the PR only passes the override into embedded PI runs. If the request dispatches to a CLI backend, the field is silently ignored and the original tools remain available, so this needs to be threaded through or rejected fail-closed.
  - confidence: 0.92
- **[P2] Document the actual `group:openclaw` scope:** `docs/gateway/openresponses-http-api.md:145`
  - body: The docs say `group:openclaw` denies all built-in tools, but runtime builds that group only from catalog entries marked `includeInOpenClawGroup`, excluding filesystem tools and `exec`/`process`. Callers using this as an access-control shortcut could leave tools enabled unexpectedly.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff is deny-only and adds no dependencies, but incomplete enforcement and overstated group docs create false-assurance risk for a tool-access-control API.

Concerns:

  - body: The PR advertises per-request tool denial, but CLI runner preparation and loopback tool resolution have no per-request deny input, so callers could believe tools are unavailable while CLI-backed runs still expose them.
