## Summary

Keep open: the direction is useful, but the PR is not merge-ready because it lacks real gateway proof and still creates a false access-control guarantee for request-scoped tool denial across backend paths.

## What This Changes

The PR adds an optional OpenResponses `tool_deny` request field, documents it, and threads it into embedded agent tool-policy construction.

## Best Possible Solution

Define one request-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical catalog, and require live gateway proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the new field is accepted and forwarded, while the CLI and loopback tool paths do not receive an equivalent deny policy. I did not run live gateway proof in this read-only review, and the PR body says live gateway E2E was not verified.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable part of the solution, but the public API must enforce denial on every supported backend or reject unsupported backends fail-closed.
