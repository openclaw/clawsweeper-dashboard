## Summary

This PR should stay open: the central feature is not on current main, but the patch still lacks real runtime proof and leaves the new deny override unenforced or not fail-closed on CLI-backed OpenResponses runs.

## What This Changes

The PR adds an OpenClaw-specific `tool_deny` request field for `/v1/responses`, threads it through gateway and embedded-agent tool construction, and documents per-request built-in tool denial.

## Best Possible Solution

Add one run-scoped deny policy for `/v1/responses` that is enforced or rejected fail-closed across embedded and CLI-backed runs, with canonical group documentation and focused coverage.

## Reproduction Assessment

Yes for the review blockers: source inspection shows current main rejects `tool_deny`, and the PR’s CLI path still lacks a deny override contract or fail-closed preparation step. There is not yet live gateway proof of the after-fix behavior.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable direction, but the patch must either enforce or reject the deny override on CLI-backed runs and describe the exact group semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override in CLI runs:** `src/agents/command/attempt-execution.ts:477`
  - body: The PR passes `toolDenyOverride` into `runCliAgent`, but `RunCliAgentParams` and `prepareCliRunContext` still only know about `toolsAllow`. CLI-backed OpenResponses requests would not enforce `tool_deny` through the loopback MCP tool list unless this path consumes the deny list or fails closed when it is set.
  - confidence: 0.93
- **[P2] Document the actual openclaw group membership:** `docs/gateway/openresponses-http-api.md:145`
  - body: The new docs describe `group:openclaw` as all built-in tools, but runtime builds that group only from tools marked `includeInOpenClawGroup`, excluding filesystem tools plus `exec` and `process`. Callers using this as an access-control group could leave built-in tools exposed that they expected to deny.
  - confidence: 0.86
