## Summary

Keep open; current main does not implement `tool_deny`, and this PR is still useful but should not merge as-is because the CLI runner boundary is not typed/enforced and the user-facing API feature lacks the required changelog entry.

## What This Changes

The PR adds an optional `/v1/responses` `tool_deny` request field, threads it into Gateway and embedded agent tool filtering, and documents the OpenClaw-specific per-request deny list.

## Best Possible Solution

Land the additive OpenResponses extension only after request-level deny filtering is consistently typed and enforced across supported runner paths, documented accurately, and covered by focused schema/tool-policy tests plus release notes.

## Reproduction Assessment

Yes. Current main rejects `tool_deny` through the strict request schema, and the PR blocker is reproducible by comparing the PR-added `runCliAgent` argument with the current `RunCliAgentParams` type and CLI loopback tool-resolution path.

## Solution Assessment

No, not as-is. Reusing the existing policy pipeline is the right narrow embedded-run approach, but the PR needs type-safe CLI-runner behavior and changelog coverage before it is maintainable.
