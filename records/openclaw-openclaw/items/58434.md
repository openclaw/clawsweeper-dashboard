## Summary

Keep open: current main still has no `tool_deny` support, and this PR is not merge-ready because the deny override is not enforced or rejected for CLI-backed OpenResponses runs, the new docs overstate `group:openclaw`, and contributor proof is only tests/CI.

## What This Changes

The PR adds an optional `tool_deny` request field to `/v1/responses`, threads it into the gateway and embedded PI tool-policy path, and documents per-request built-in tool denial.

## Best Possible Solution

Implement one run-scoped deny policy for `/v1/responses` that is enforced or rejected fail-closed on embedded and CLI-backed runs, with canonical group documentation and focused gateway/tool-policy coverage.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the proposed field would enter the generic agent command path, while CLI backend preparation has no deny override and only fails closed for `toolsAllow`. The intended user behavior still lacks live gateway proof.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable direction, but the PR must enforce or reject the deny override on CLI-backed runs and document the exact group semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override in CLI-backed runs:** `src/gateway/openresponses-http.ts:445`
  - body: The PR forwards `toolDenyOverride` from the OpenResponses ingress into `agentCommandFromIngress`, but the command can dispatch to CLI backends, where `RunCliAgentParams` and `prepareCliRunContext` still only know how to fail closed for `toolsAllow`. A CLI-backed request can therefore accept `tool_deny` while still exposing the unchanged loopback tool list; pass/enforce the deny list there or reject these runs fail-closed.
  - confidence: 0.92
- **[P2] Document the actual openclaw group scope:** `docs/gateway/openresponses-http-api.md:145`
  - body: The new docs describe `group:openclaw` as all OpenClaw built-in tools, but runtime builds that group only from catalog entries marked `includeInOpenClawGroup`, excluding `read`, `write`, `edit`, `apply_patch`, `exec`, and `process`. Callers using this as an access-control shortcut could leave built-in tools enabled unexpectedly.
  - confidence: 0.86
