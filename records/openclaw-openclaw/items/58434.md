## Summary

Keep open: the feature direction is useful, but the PR is not merge-ready because the public `tool_deny` API lacks real gateway proof and the patch can imply a tool access-control boundary that CLI/loopback-backed runs and the documented `group:openclaw` semantics do not actually enforce.

## What This Changes

The PR adds a `tool_deny` array to the `/v1/responses` request schema/docs and forwards it through agent command options into embedded PI tool-policy filtering.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical catalog, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible for the review blockers: the PR accepts and forwards `tool_deny`, while current CLI/loopback tool resolution has no matching per-request deny input. I did not run live gateway proof because this was a read-only review and the contributor has not provided real runtime evidence.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable piece, but the public API must either enforce denial on every supported backend or reject unsupported backends fail-closed.
