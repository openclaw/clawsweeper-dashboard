## Summary

Keep open: current main does not implement `tool_deny`, and the PR still has merge-blocking access-control correctness issues plus missing real gateway proof. The earlier Greptile documentation finding for `group:memory` and `group:runtime` was addressed in the PR, but source inspection still shows unsupported CLI-backed runs can accept the deny request without enforcing it.

## What This Changes

The PR adds a `tool_deny` request field to `/v1/responses`, forwards it through gateway and embedded-agent tool construction, and documents per-request built-in tool denial.

## Best Possible Solution

Implement one run-scoped deny contract that is enforced for every supported `/v1/responses` backend or rejected fail-closed, with canonical group documentation and live gateway proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the new field would be accepted and forwarded to the generic agent command path while CLI preparation and loopback tool resolution have no deny override. No live gateway reproduction was provided for the intended behavior.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is reasonable, but the PR must enforce or fail closed on CLI-backed runs and document exact group semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override in CLI-backed runs:** `src/gateway/openresponses-http.ts:445`
  - body: The PR forwards `toolDenyOverride` into `agentCommandFromIngress`, but the command can dispatch to CLI backends where `RunCliAgentParams` and `prepareCliRunContext` only know how to fail closed for `toolsAllow`. A CLI-backed `/v1/responses` request can therefore accept `tool_deny` while still exposing the unchanged loopback tool list; pass/enforce the deny list there or reject these runs fail-closed.
  - confidence: 0.92
- **[P2] Document the actual openclaw group scope:** `docs/gateway/openresponses-http-api.md:145`
  - body: The new docs describe `group:openclaw` as all OpenClaw built-in tools, but runtime builds that group only from catalog entries marked `includeInOpenClawGroup`, excluding `read`, `write`, `edit`, `apply_patch`, `exec`, and `process`. Callers treating this as an access-control shortcut could leave built-in tools enabled unexpectedly.
  - confidence: 0.86

## Security Review

Status: needs_attention

  - body: The PR advertises per-request tool denial, but CLI runner preparation and MCP loopback resolution have no per-request deny input, so callers could believe tools are unavailable while CLI-backed runs still expose them.
