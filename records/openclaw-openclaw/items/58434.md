## Summary

Keep open: the feature is not on current main, and the PR still has merge-blocking correctness/security gaps plus missing real gateway proof.

## What This Changes

This PR adds an optional `tool_deny` request field to `/v1/responses`, forwards it through agent command and embedded PI tool construction, and documents the new per-request deny list.

## Best Possible Solution

Define one run-scoped deny contract that is enforced for every supported `/v1/responses` backend or rejected fail-closed, with canonical group documentation and live gateway proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the new field would be accepted and forwarded to the generic agent command path while CLI preparation and loopback tool resolution have no deny override. No live gateway reproduction was provided for the intended behavior.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is reasonable, but the PR must enforce or fail closed on CLI-backed runs and document exact group semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override for CLI-backed runs:** `src/gateway/openresponses-http.ts:445`
  - body: This forwards `toolDenyOverride` into `agentCommandFromIngress`, but that command can dispatch to CLI backends where `RunCliAgentParams` and `prepareCliRunContext` only know how to fail closed for `toolsAllow`. A CLI-backed `/v1/responses` request can therefore accept `tool_deny` while still exposing the unchanged loopback tool list; thread the deny list through that path or reject these runs fail-closed.
  - confidence: 0.92
- **[P2] Document the actual `group:openclaw` scope:** `docs/gateway/openresponses-http-api.md:145`
  - body: The new docs describe `group:openclaw` as all OpenClaw built-in tools, but runtime builds that group only from catalog entries marked `includeInOpenClawGroup`, excluding filesystem tools plus `exec` and `process`. Callers treating this as an access-control shortcut could leave built-in tools enabled unexpectedly.
  - confidence: 0.86

## Security Review
