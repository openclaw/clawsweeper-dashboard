## Summary

The PR should stay open: it adds the public API field, but enforcement only reaches embedded PI tool construction while CLI/loopback tool surfaces do not receive or honor the per-request deny contract. The second commit resolves Greptile’s memory/runtime group comment, but `group:openclaw` is still documented more broadly than the canonical catalog supports.

## What This Changes

The PR adds an OpenClaw-specific `tool_deny` request field for `/v1/responses`, forwards it through gateway agent ingress into embedded PI tool filtering, and documents per-request tool denial.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` execution backend either enforces or rejects fail-closed, document tool groups from the canonical catalog, and require redacted live gateway proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR forwards `tool_deny` into embedded PI only, while CLI/loopback tool resolution has no matching request-scoped deny input. I did not run live gateway proof because this was a read-only review and the contributor has not provided it.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable component, but the public API needs backend-wide enforcement or fail-closed rejection before it is the best maintainable solution.
