## Summary

This PR is useful but not merge-ready: it adds a public tool access-control field without proving or enforcing the deny contract across every supported `/v1/responses` backend, and contributor-side real behavior proof is still missing.

## What This Changes

The PR adds a `tool_deny` request field to `/v1/responses`, documents it, and threads it into embedded PI tool-policy filtering.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` backend enforces or rejects fail-closed, document group semantics from the canonical catalog, and require real gateway proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows `tool_deny` is accepted and forwarded from OpenResponses ingress, while CLI runner and loopback tool resolution have no matching per-request deny input. I did not run a live gateway repro because this was a read-only review.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable piece, but the public API must either enforce denial on every supported backend or reject unsupported backends fail-closed.
