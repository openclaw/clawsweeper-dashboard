## Summary

Keep open with blocking review findings. Current main still lacks the requested `tool_deny` request field, and the PR is directionally useful, but the branch is not merge-ready because it threads the override into the CLI runner without a matching typed/enforced CLI contract and omits the required user-facing changelog entry. ## What This Changes

The PR adds an optional OpenClaw-specific `tool_deny` array to `/v1/responses`, threads it through Gateway and embedded agent tool creation, applies it as a deny-only tool-policy step, and documents the request field.

## Best Possible Solution

Land the additive OpenResponses extension after the deny override is type-safe and consistently enforced across supported run paths, release notes cover the new request field, and focused tests prove the override is ephemeral and deny-only.

## Reproduction Assessment

Yes. The main blocker is reproducible by comparing the PR-added `toolDenyOverride` property on the `runCliAgent({...})` call with the current `RunCliAgentParams` definition, which lacks that property; current main also strictly rejects `tool_deny` at the request schema.

## Solution Assessment

No, not as-is. Reusing the existing group expansion and deny pipeline is the narrow maintainable solution for embedded runs, but the PR must either implement the same contract for CLI-backed runs or explicitly keep the override out of that path before merge.
