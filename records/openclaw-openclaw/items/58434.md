## Summary

Keep open: the embedded PI filtering path is plausible, but the public `/v1/responses` field still has security-boundary blockers because unsupported backends are not shown to enforce or reject the override, the `group:openclaw` docs overstate the canonical group, and real gateway behavior proof is still missing.

## What This Changes

The PR adds an optional OpenClaw-specific `tool_deny` request field for `/v1/responses`, forwards it into embedded PI tool filtering, and documents per-request tool denial.

## Best Possible Solution

Define a run-scoped deny contract that every supported `/v1/responses` execution backend either enforces or rejects fail-closed, document tool groups from the canonical catalog, and require redacted live gateway proof before merge.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. The review blockers are source-reproducible because the PR only threads the override into embedded PI, while current CLI and gateway loopback tool-resolution paths have no matching request-scoped deny input.

## Solution Assessment

No, not as-is. Reusing the embedded tool-policy pipeline is a reasonable component, but the public API needs backend-wide enforcement or fail-closed rejection and canonical group documentation before it is the maintainable solution.
