## What This Changes

The PR adds an optional `/v1/responses` `tool_deny` request field, threads it into OpenResponses and embedded agent tool filtering, and documents OpenClaw-specific per-request deny groups.

## Best Possible Solution

Land the additive OpenResponses extension after the deny override is consistently typed and enforced across embedded and CLI-backed tool-resolution paths, documented accurately, covered by focused tests, and recorded in the changelog.

Yes. Current main rejects `tool_deny` through the strict schema, and the PR blocker is reproducible by comparing the PR-added `runCliAgent` argument with the current `RunCliAgentParams` type and loopback MCP tool-resolution path.

## Solution Assessment

No, not as-is. Reusing the existing embedded policy pipeline is the narrowest maintainable approach for embedded runs, but the PR must also handle CLI-backed runs and release notes before merge.
