## Summary

Keep this PR open, but do not merge as-is: the embedded-runner path is plausible, while CLI-backed OpenResponses runs still lack enforceable deny handling and the docs overstate `group:openclaw` for a tool-access-control feature.

## What This Changes

The PR adds an OpenClaw-specific `/v1/responses` `tool_deny` array, threads it into gateway and embedded agent tool filtering, and documents per-request tool denial.

## Best Possible Solution

Implement `tool_deny` as a single run-scoped policy contract that embedded runs and CLI-backed loopback runs either honor consistently or reject fail-closed, with docs describing exact group expansion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main rejects `tool_deny`, and the PR blocker is visible from the added CLI argument plus the missing CLI runner type and loopback deny inputs.

## Solution Assessment

No, not as-is. Reusing the embedded policy pipeline is a reasonable seam, but the feature must cover or fail closed on CLI-backed runs and fix the `group:openclaw` documentation before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enforce the deny override in CLI runs:** `src/agents/command/attempt-execution.ts:477`
  - body: The PR passes `toolDenyOverride` into `runCliAgent`, but `RunCliAgentParams` does not declare or consume that field and CLI preparation only fails closed for `toolsAllow`. CLI-backed OpenResponses runs would still expose denied tools through the loopback tool list unless this path is wired through or rejected when `tool_deny` is present.
  - confidence: 0.93
- **[P2] Document the actual openclaw group membership:** `docs/gateway/openresponses-http-api.md:145`
  - body: The new docs say `group:openclaw` means all OpenClaw built-in tools, but runtime builds that group only from definitions marked `includeInOpenClawGroup`; filesystem and most runtime tools are separate groups. Callers using it as an all-built-ins deny would still leave tools such as `read`, `write`, `edit`, `exec`, and `process` outside that group.
  - confidence: 0.84
