## Summary

Keep open: current main still lacks MiniMax TTS response-envelope error handling, and the linked closing PR is open, conflicting, and still needs normal proof/merge handling before this issue can close.

## What This Changes

The issue reports that MiniMax TTS quota exhaustion does not fall back to a configured secondary TTS provider such as Microsoft.

## Best Possible Solution

Review, update, and land or supersede the provider-local MiniMax envelope guard in the linked PR, with regression tests and redacted real fallback proof, while leaving the generic TTS fallback loop unchanged.

## Reproduction Assessment

Yes, at source level: current main accepts an HTTP 200 MiniMax TTS response with non-zero base_resp.status_code when data.audio is non-empty because it never inspects the envelope. I did not run a live MiniMax quota flow in this read-only review.

## Solution Assessment

Yes for the likely code direction: checking MiniMax’s response envelope inside extensions/minimax/tts.ts is the narrow plugin-owned fix that lets the existing dispatcher catch the error and try fallback providers. The issue should remain open until the linked PR or an equivalent fix lands.
