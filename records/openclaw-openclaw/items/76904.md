## Summary

Keep open: the documented generic TTS fallback path still appears to catch provider errors and continue to configured providers, but the report lacks the exact MiniMax quota response and caller surface needed to identify the bypass.

## What This Changes

The issue requests TTS failover from MiniMax to a configured Microsoft or other fallback provider when MiniMax quota or usage limit is exceeded.

## Best Possible Solution

Reproduce the MiniMax quota-limit path with the requested diagnostics, then patch the specific caller or provider path that bypasses `synthesizeSpeech()` fallback and add focused regression coverage.

## Reproduction Assessment

No, we do not have a high-confidence way to reproduce the issue yet. Source inspection shows generic configured-provider fallback should work, but the report does not identify the caller surface, config shape, `/tts status` output, quota response, or explicit-selection state.

## Solution Assessment

Unclear. The existing generic fallback loop is the right maintainable design, so the safer path is to prove whether a MiniMax quota response or a `disableFallback` caller path is bypassing it before changing code.
