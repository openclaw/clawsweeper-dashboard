## Summary

Keep open: TTS fallback is documented existing behavior, but current main and v2026.5.2 both show the generic synthesis loop catching provider errors and continuing through configured providers; the report still needs the exact caller surface, sanitized config, `/tts status` output, and MiniMax quota response to identify the bypass.

## What This Changes

The issue reports that MiniMax TTS quota exhaustion does not fall back to a configured Microsoft or other secondary TTS provider.

## Best Possible Solution

Reproduce the MiniMax quota path with the reported config and caller surface, then fix only the specific provider or caller path that bypasses the existing fallback loop and add regression coverage around that path.

## Reproduction Assessment

No, we do not have a high-confidence way to reproduce the issue yet. Source inspection shows the generic configured-provider fallback path should continue after MiniMax provider errors, but the report lacks the exact TTS surface, sanitized config, `/tts status` output, and MiniMax quota response.

## Solution Assessment

Unclear. The existing generic fallback loop is the right maintainable design, so the safest next step is to prove whether MiniMax quota handling or a fallback-disabled caller path is bypassing it before changing code.
