## Summary

Keep open: TTS fallback is documented existing behavior, and current main still has a generic provider-error fallback loop, but the report lacks the exact surface, config, and MiniMax quota payload needed to identify the bypass.

## What This Changes

The issue asks OpenClaw to fall back from MiniMax TTS to a configured secondary TTS provider such as Microsoft when MiniMax quota is exceeded.

## Best Possible Solution

Capture the failing MiniMax quota attempt through `/tts status` or the exact gateway/command path, then patch only the caller or provider path proven to bypass the existing fallback contract.

## Reproduction Assessment

No. Source inspection gives a plausible bypass area, but the issue lacks the TTS surface, sanitized config, `/tts status`, and MiniMax quota HTTP body needed to reproduce a current-main failure.

## Solution Assessment

Unclear. The generic fallback loop appears to be the right design, so the maintainable fix is to patch the specific proven bypass rather than broaden fallback semantics for explicit selections without evidence.
