## Summary

Keep open: current main and the latest stable release still let MiniMax TTS treat an HTTP-200 response with nonzero MiniMax envelope status and non-empty audio as success, so the shared fallback loop never advances to Microsoft or another configured provider. The only linked repair PR identified the same narrow provider-owned fix but was closed unmerged.

## What This Changes

The issue requests that configured TTS fallback run when MiniMax TTS quota or code-plan usage is exhausted.

## Best Possible Solution

Add MiniMax-owned response-envelope validation before audio decoding so quota or billing failures throw into the existing configured-provider fallback loop, with focused regression coverage for nonzero base_resp plus placeholder audio.

## Reproduction Assessment

Yes, at source level: a MiniMax HTTP-200 response with nonzero base_resp.status_code and non-empty data.audio reaches the current audio decode path and resolves, so the fallback loop cannot catch a provider error. I did not run a live MiniMax quota-limit request.

## Solution Assessment

Yes, the best fix is a narrow MiniMax provider repair: throw on nonzero response-envelope status before decoding audio and leave the generic speech-core fallback loop unchanged. Adding a core fallback mode or new config option would be broader than the observed bug requires.
