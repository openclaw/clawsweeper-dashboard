## Summary

Keep open: current main and v2026.6.6 still parse MiniMax TTS HTTP 200 responses without checking `base_resp`, so quota envelopes with placeholder audio can be treated as success and skip the documented fallback chain. The linked fix PR identified the right provider-local direction, but it was closed unmerged.

## What This Changes

The issue asks OpenClaw to fall back from MiniMax TTS to the configured secondary speech provider when MiniMax code-plan usage or quota is exceeded.

## Best Possible Solution

Add a provider-local MiniMax TTS `base_resp.status_code` guard before decoding `data.audio`, with regression tests for non-zero envelope status both with and without placeholder audio; leave the shared fallback loop unchanged.

## Reproduction Assessment

Yes, at source level: a mocked HTTP 200 MiniMax TTS response with non-zero `base_resp.status_code` and non-empty `data.audio` would resolve on current main because `extensions/minimax/tts.ts` reads only `data.audio`. I did not run a live MiniMax quota/billing account reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is provider-local envelope validation before audio decoding, because shared speech-core fallback already advances when a provider throws. A core fallback rewrite would be broader and would not address the MiniMax-specific success envelope contract as cleanly.
