## Summary

Keep open: current main still treats an HTTP 200 MiniMax TTS response with non-empty `data.audio` as success without checking `base_resp.status_code`, so the existing fallback loop will not advance to Microsoft or another configured provider for that quota-envelope failure. The linked closing PR is open and not merged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that configured fallback TTS providers are not tried when MiniMax TTS hits a code-plan or quota limit.

## Best Possible Solution

Land or supersede the provider-local MiniMax envelope guard from the linked PR with regression coverage and redacted real fallback proof, while leaving the generic speech-core fallback loop unchanged.

## Reproduction Assessment

Yes at source level: a mocked HTTP 200 MiniMax TTS response with non-zero `base_resp.status_code` and non-empty `data.audio` would pass the current provider path as success, preventing fallback. I did not run a live MiniMax quota-flow reproduction in this read-only review.

## Solution Assessment

Yes for the code direction: checking MiniMax's response envelope inside `extensions/minimax/tts.ts` is the narrow provider-owned fix that lets the existing dispatcher catch the error and try fallback providers. The issue should stay open until the linked PR or an equivalent fix merges.
