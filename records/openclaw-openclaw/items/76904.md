## Summary

Keep open: current main still lacks MiniMax TTS response-envelope handling, and the linked closing PR is open, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that MiniMax TTS quota or plan-limit responses are treated as success, preventing configured TTS fallback providers from being tried.

## Best Possible Solution

Add a MiniMax provider-local `base_resp.status_code` guard before audio decoding, keep the shared fallback loop unchanged, and cover the HTTP-200 envelope-error case with provider tests plus redacted real fallback proof.

## Reproduction Assessment

Yes, at source level: a mocked HTTP 200 MiniMax TTS response with non-zero `base_resp.status_code` and non-empty `data.audio` would resolve on current main because the provider never reads `status_code`. I did not run a live MiniMax quota or billing-account flow in this read-only review.

## Solution Assessment

Yes for the implementation direction: checking MiniMax's response envelope in `extensions/minimax/tts.ts` is the narrow provider-owned fix and matches adjacent MiniMax providers. The issue should not close until the linked PR or an equivalent patch lands.
