## Summary

Keep open: the PR addresses the reported CLI-backed cron timeout clamp, but it should not merge as-is because the new arithmetic makes configured watchdog `maxMs` values stop acting as hard upper bounds.

## What This Changes

The PR changes CLI no-output watchdog arithmetic and adds focused tests so resumed CLI-backed cron runs can scale past the default 180-second ceiling for large `timeoutMs` values.

## Best Possible Solution

Scale only the default or unconfigured resume watchdog ceiling for large explicit cron timeouts while preserving configured `noOutputTimeoutMs` and `maxMs` as hard bounds, with regression coverage for both paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a resumed CLI-backed cron run with `timeoutMs` above 600000 computes a larger ratio value but current main clamps it to the default 180000 ms resume profile max.

## Solution Assessment

No. The proposed direction is right for the default resume profile, but the current patch applies scaling to every `profile.maxMs`, including explicit user-configured upper bounds.
