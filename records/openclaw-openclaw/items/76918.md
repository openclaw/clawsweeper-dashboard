## Summary

Keep open: the PR addresses a valid CLI-backed cron timeout bug, but the patch should not merge as-is because it turns explicit watchdog `maxMs` overrides from upper bounds into scalable floors.

## What This Changes

The PR changes CLI no-output watchdog timeout calculation and adds focused unit tests so resumed CLI-backed cron runs can scale past the default 180s ceiling when a large `timeoutMs` is supplied.

## Best Possible Solution

Preserve explicit `maxMs` as a hard upper bound while scaling only the default/unconfigured resume ceiling, then add regression coverage for both the cron resume case and configured `maxMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a resumed CLI-backed cron run with `timeoutMs` above 600000 currently computes a larger ratio value but clamps it to the 180000 ms resume profile max.

## Solution Assessment

No. The PR fixes the default 180-second clamp, but it is not the best form because it also overrides user-configured `maxMs` values that the config contract defines as upper bounds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve explicit watchdog maxMs overrides:** `src/agents/cli-runner/reliability.ts:76-77`
  - body: This makes every `maxMs` scalable by comparing it with `computed`, so a backend that deliberately sets `reliability.watchdog.resume.maxMs` below the ratio-derived value no longer gets that upper bound. For example, `maxMs: 240_000`, `timeoutMs: 1_800_000`, and the default resume ratio now returns 540s instead of the configured 240s cap. Please distinguish the default ceiling from an explicit override, and add coverage for configured `maxMs`.
