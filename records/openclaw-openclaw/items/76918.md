## Summary

Keep open. The PR targets the right CLI resume watchdog path, but the patch makes configured `maxMs` stop acting as an upper bound whenever the computed ratio exceeds it, so it should not merge as-is.

## What This Changes

The PR changes CLI no-output watchdog arithmetic and adds reliability tests so large resume timeouts can scale beyond the default 180s ceiling.

## Best Possible Solution

Preserve explicit operator watchdog caps while allowing the default resume ceiling to scale with explicit cron/run timeouts, with regression tests for both default scaling and configured `maxMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: a resumed CLI run with `timeoutMs=1_800_000` computes 540000 ms but current main clamps it to the resume `maxMs` of 180000 ms.

## Solution Assessment

No. The direction is maintainable for the CLI default case, but the implementation must distinguish the default ceiling from an explicitly configured `maxMs` upper bound and include coverage for that contract.
