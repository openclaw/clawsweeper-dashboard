## Summary

Keep open: the PR targets a valid CLI-backed cron timeout bug, but the patch should not merge as-is because it turns documented `maxMs` overrides into scalable floors and misses required changelog coverage.

## What This Changes

The PR changes CLI no-output watchdog timeout calculation and adds focused tests so resumed CLI-backed cron runs can scale beyond the default 180-second ceiling for large `timeoutMs` values.

## Best Possible Solution

Preserve explicit `maxMs` as a hard upper bound while scaling only the default or unconfigured resume ceiling for long explicit cron timeouts, with regression tests for both paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main passes cron `timeoutMs` into the CLI runner, then resumed CLI watchdogs compute a larger ratio value but clamp to the default 180000 ms max; the PR diff also source-reproduces the configured `maxMs` regression.

## Solution Assessment

No. The intended default scaling is the right direction, but this implementation also defeats user-configured `maxMs` upper bounds and needs the required changelog entry.
