## Summary

Keep this PR open, but it needs changes before merge: source inspection confirms the underlying cron/CLI watchdog bug, while the patch currently breaks the documented `maxMs` upper-bound contract and misses required release-note coverage.

## What This Changes

The PR changes CLI no-output watchdog arithmetic and adds resolver tests so explicit large cron timeouts can lengthen resumed CLI watchdogs beyond the 180-second default.

## Best Possible Solution

Land a focused CLI watchdog fix that scales only the unconfigured resume default for explicit cron timeouts while keeping configured `maxMs` values as hard caps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a resumed CLI-backed cron run with `timeoutMs` above 600000 computes a larger ratio value but current main clamps it to the 180000 ms resume profile max.

## Solution Assessment

No. The PR fixes the default 180-second clamp, but it is not the best form because it turns every `maxMs` into a floor instead of preserving explicit upper-bound overrides.
