## Summary

Keep open: this PR is a plausible feature implementation, but it still has blocking config-routing and documentation/contract mismatches, and it lacks real after-fix behavior proof. Current main does not already implement the Weibull model, so this is normal PR review rather than a cleanup close.

## What This Changes

The PR adds an opt-in Weibull temporal decay model for memory-core scoring, new memory-search config/schema fields, generated schema output, a design doc, and targeted tests.

## Best Possible Solution

Land a revised opt-in memory-core implementation only if maintainers accept the feature: keep exponential as default, expose one documented memorySearch config path, thread model and shape through resolved config into scoring, align validation/docs/runtime semantics, and include focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the PR defect. Source inspection on the PR head shows model and weibullShape validate in the runtime schema, but resolved memory-search config still returns only enabled and halfLifeDays, so a configured Weibull model would not reach runtime scoring.

## Solution Assessment

No as-is. The core math is plausible, but the patch is not the narrowest maintainable solution until the public config contract, plugin metadata, docs, validation behavior, and proof all describe the same runtime path.

- **[P1] Thread decay model through resolved config:** `src/config/zod-schema.agent-runtime.ts:759-760`
  - body: The schema now accepts temporalDecay.model and temporalDecay.weibullShape, but ResolvedMemorySearchConfig and mergeConfig still preserve only enabled and halfLifeDays. A user can configure model: "weibull" successfully and still get the exponential default at runtime.
  - confidence: 0.94
- **[P2] Use a single temporal-decay config surface:** `extensions/memory-core/openclaw.plugin.json:25-44`
  - body: This adds plugins.entries.memory-core.config.temporalDecay, but memory scoring is resolved from agents.*.memorySearch.query.hybrid.temporalDecay. Unless runtime code reads this plugin field, users configuring it get an accepted no-op; otherwise remove it and keep the documented memorySearch path.
  - confidence: 0.88
- **[P2] Correct the beta less-than-one guidance:** `docs/pr-weibull-decay.md:40`
  - body: For 0 < beta < 1, (t / halfLife)^beta is larger before half-life and smaller after half-life, so decay is faster early and slower late. The added table says the inverse, which would make users tune the wrong curve.
  - confidence: 0.95
- **[P2] Align schema validation with shape normalization:** `src/config/zod-schema.agent-runtime.ts:760`
  - body: The runtime helper clamps out-of-range shapes, but the config schema rejects the same values with min/max validation while the PR text and docs describe clamping/self-healing. Either accept and normalize those values or document strict config validation instead.
  - confidence: 0.9
