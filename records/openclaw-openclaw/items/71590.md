## Summary

Keep open. This PR is a plausible opt-in memory feature, but the current patch is not merge-ready: the PR-head schema accepts Weibull config that resolved memory-search config still drops, it adds a second plugin config path that scoring does not read, beta-less-than-one docs remain inverted, and external real behavior proof is missing.

## What This Changes

Adds an opt-in Weibull temporal decay model to memory-core scoring with config/schema fields, generated schema updates, a design doc, and targeted tests.

## Best Possible Solution

Land a revised opt-in memory-core implementation only if maintainers accept the feature, with one documented memorySearch config path, full resolved-config plumbing, aligned validation/docs/runtime semantics, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes for the PR defect. Source inspection of the PR head shows the schema accepts model and weibullShape while resolved memory-search config still returns only enabled and halfLifeDays, so configured Weibull mode would not reach scoring.

## Solution Assessment

No as-is. The core math is plausible, but the public config contract, plugin metadata, docs, validation behavior, and runtime plumbing need to describe the same path before this is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Thread decay model through resolved config:** `src/config/zod-schema.agent-runtime.ts:759-760`
  - body: The schema now accepts temporalDecay.model and temporalDecay.weibullShape, but PR-head ResolvedMemorySearchConfig and mergeConfig still preserve only enabled and halfLifeDays. Users can configure `model: "weibull"` successfully and still get the exponential default at runtime.
  - confidence: 0.94
- **[P2] Use a single temporal-decay config surface:** `extensions/memory-core/openclaw.plugin.json:25-44`
  - body: This adds `plugins.entries.memory-core.config.temporalDecay`, but memory scoring is resolved from `agents.*.memorySearch.query.hybrid.temporalDecay`. Unless runtime code reads this plugin field, users configuring it get an accepted no-op; otherwise remove it and keep the documented memorySearch path.
  - confidence: 0.88
- **[P2] Correct the beta less-than-one guidance:** `docs/pr-weibull-decay.md:40`
  - body: For `0 < beta < 1`, `(t / halfLife)^beta` is larger before half-life and smaller after half-life, so decay is faster early and slower late. The added table says the inverse, which would make users tune the wrong curve.
  - confidence: 0.95
- **[P2] Align schema validation with shape normalization:** `src/config/zod-schema.agent-runtime.ts:760`
  - body: The runtime helper clamps out-of-range shapes, but the config schema rejects the same values with min/max validation while the PR text and docs describe clamping and self-healing. Either accept and normalize those values or document strict config validation instead.
  - confidence: 0.9
