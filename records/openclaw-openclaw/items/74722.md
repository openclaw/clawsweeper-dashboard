The issue requests optional language-aware TTS provider/model/voice routing for inbound voice replies, driven by detected input or reply language while preserving current fixed TTS behavior when disabled.

## Best Possible Solution

Keep this open for maintainer product/API review, then implement a generic speech language-routing contract that preserves current fixed TTS fallback behavior when disabled.
