## Summary

Keep open. Current main still has modality-aware auto-TTS but no public config, schema, runtime parameter, STT metadata, or provider-selection step that can route TTS provider/model/voice by input or reply language.

## What This Changes

Add language-aware routing so inbound or reply language can select the TTS provider, model, and voice while preserving existing `messages.tts.auto: "inbound"` and per-agent override behavior.

## Best Possible Solution

Design a backward-compatible routing contract that preserves optional STT language metadata, supports input/reply/auto source precedence, deep-merges with existing overrides, resolves routes before synthesis, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for the detected or inferred language to change provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS routing contract, not a provider-specific workaround or prompt-only convention.
