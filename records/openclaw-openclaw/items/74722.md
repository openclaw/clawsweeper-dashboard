## Summary

Keep open. Current main still lacks a public language-aware TTS routing contract, does not preserve detected STT language metadata, and resolves spoken replies through the existing fixed provider/model/voice path.

## What This Changes

Request optional language-aware TTS routing so inbound voice replies can select provider, model, and voice from detected input or reply language while preserving existing fallback behavior.

## Best Possible Solution

Add a documented, backward-compatible language-routing contract that carries optional STT language metadata into TTS resolution, chooses routes before synthesis, and falls back to the existing provider/model/voice path.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed TTS voice and send voice inputs in multiple languages; current main has no code path that changes TTS provider/model/voice based on detected language.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS routing contract, not a provider-specific workaround or prompt-only convention.
