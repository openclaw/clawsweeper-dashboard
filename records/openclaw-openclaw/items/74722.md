## Summary

Keep open. Current main still has fixed TTS provider/model/voice selection, `inbound` only gates whether audio is generated, and STT outputs do not preserve detected language metadata for a routing layer.

## What This Changes

Request language-aware TTS provider/model/voice routing for auto-TTS replies based on inbound or reply language.

## Best Possible Solution

Choose and document a backward-compatible routing contract that preserves optional STT language metadata and lets TTS select provider/model/voice by language while keeping provider-specific policy out of unrelated core paths.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main lacks language routing and only uses inbound audio state to decide whether to synthesize audio.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested config is plausible, but the open pre-TTS hook discussion may be the narrower extension-friendly solution for per-message voice routing.
