## Summary

Keep open. Current main still lacks the requested language-aware TTS routing contract: `messages.tts` has no route/schema surface, inbound auto-TTS only gates whether audio is synthesized, provider/model/voice selection remains config, preference, persona, and directive driven, and audio transcription metadata does not preserve detected language for TTS routing. The CausaMem comment is memory-related context, not a TTS implementation or duplicate.

## What This Changes

Add a backward-compatible language-aware TTS routing layer that selects TTS provider, model, and voice from detected input or reply language while preserving current auto-TTS and per-agent override behavior.

## Best Possible Solution

Design a generic TTS language-routing contract in the shared TTS config/runtime path, preserve optional detected-language metadata from STT/preflight, apply routing before provider synthesis, and document/test fallback behavior across global, per-agent, channel, account, preference, persona, and directive layers.

## Reproduction Assessment

Yes for the current limitation. Source and docs show `auto: "inbound"` only controls whether synthesis runs, while no input/reply language is available to the provider routing path; no live STT/TTS credentials are needed to verify the missing contract.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is sound, but the best fix should be a generic core speech-routing contract rather than a provider-specific workaround, prompt convention, or external-only plugin.
