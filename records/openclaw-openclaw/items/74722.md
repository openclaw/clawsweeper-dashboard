## Summary

Keep open: current main does not implement language-aware TTS routing, and the request is a real feature/config design across TTS config, STT metadata, and plugin SDK boundaries rather than an obsolete or duplicate cleanup item.

## What This Changes

Add language-aware provider/model/voice routing for auto-TTS so inbound voice replies can synthesize with a voice matched to the detected input or reply language.

## Best Possible Solution

Design a backward-compatible language metadata contract first, then add a documented `messages.tts` routing surface that can deep-merge through agent/channel/account overrides and fall back to the existing provider/model/voice behavior when detection or matching fails.

## Reproduction Assessment

Not applicable: this is a new multilingual routing capability, not a bug in an existing documented behavior. Current-main source confirms `auto: "inbound"` only controls whether to generate audio after inbound audio, not which language-specific voice/model/provider to use.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact config shape and source precedence need maintainer design because the change crosses TTS config, STT provider result metadata, auto-reply context, and plugin SDK compatibility.
