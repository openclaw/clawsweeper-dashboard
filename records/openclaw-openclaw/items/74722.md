## Summary

Keep open. Current main still resolves TTS from static global, agent, channel, account, persona, local preference, and directive inputs; it does not expose language-routing config or preserve detected STT language metadata for TTS route selection.

## What This Changes

Add optional language-aware routing for auto-TTS so inbound or reply language can select the TTS provider, model, and voice while preserving current fallback behavior.

## Best Possible Solution

Define a backward-compatible generic TTS route-resolution contract that carries optional STT/input/reply language metadata into synthesis, documents fallback precedence, and composes with existing global, agent, channel, account, persona, local preference, and directive overrides.

## Reproduction Assessment

Yes, at source level. With auto-TTS in inbound mode, current main gates on inboundAudio and then resolves provider/model/voice from static config and overrides without any input-language, reply-language, or detected STT language signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
