## Summary

Keep open. Current main still has modality-aware auto-TTS, but no public config, schema, runtime parameter, STT metadata, or provider-selection step that can route TTS provider/model/voice by input or reply language.

## What This Changes

Add backward-compatible language-based provider/model/voice routing for auto-TTS, using detected input or reply language and composing with existing global and per-agent TTS config.

## Best Possible Solution

Design a generic, backward-compatible routing contract that preserves optional STT language metadata, supports input/reply/auto source precedence, resolves routes before synthesis, deep-merges with existing overrides, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure messages.tts.auto: "inbound" with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for detected or inferred language to change provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
