## Summary

Keep open. Current main still lacks a public language-aware TTS routing contract, does not preserve detected STT language in the audio transcription result, and resolves TTS provider/model/voice without any input- or reply-language routing step.

## What This Changes

Add optional language-aware routing so auto-generated TTS replies can choose provider/model/voice from detected input or reply language while preserving existing fixed-voice behavior as fallback.

## Best Possible Solution

Design a backward-compatible routing contract that preserves optional detected-language metadata, supports input/reply/auto source precedence, deep-merges overrides, resolves language routes before synthesis, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for detected language to change provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS routing contract, not a provider-specific workaround or prompt-only convention.
