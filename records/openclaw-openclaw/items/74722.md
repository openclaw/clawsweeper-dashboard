## Summary

Keep open. Current main still has inbound-audio auto-TTS gating plus static provider/persona/preference selection, but no language-routing config, preserved STT detected-language metadata, or input/reply language signal passed into TTS synthesis; this is valid feature/API design work rather than a cleanup close.

## What This Changes

Add optional language-aware TTS routing so inbound voice replies choose provider/model/voice from detected input or reply language while preserving current fallback behavior.

## Best Possible Solution

Define a backward-compatible generic TTS route-resolution contract that carries optional STT/input/reply language metadata into synthesis, documents fallback precedence, and composes with existing agent, channel, account, persona, provider, and directive overrides.

## Reproduction Assessment

Yes, at source level. With messages.tts.auto set to inbound, current main only gates on inboundAudio and then resolves provider/model/voice without any input-language, reply-language, or detected STT metadata signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
