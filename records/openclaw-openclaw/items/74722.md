## Summary

Keep open. Current main still supports only modality-aware auto-TTS and fixed provider/persona/provider-fallback selection; there is no public config/schema, STT language metadata, or runtime route-resolution step for choosing provider/model/voice by input or reply language.

## What This Changes

Add optional language-aware TTS provider/model/voice routing for inbound voice replies while preserving existing `messages.tts.auto: "inbound"` modality behavior.

## Best Possible Solution

Add a generic, backward-compatible TTS route-resolution contract that can preserve optional STT language metadata, resolve input/reply/auto language sources before synthesis, deep-merge with existing overrides, and document fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection does confirm the current limitation: `auto: "inbound"` receives only modality state and TTS provider selection has no language input.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
