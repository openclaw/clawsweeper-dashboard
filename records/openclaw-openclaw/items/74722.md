## Summary

Keep open. Current main has modality-aware auto-TTS, but it still has no public config/schema, STT metadata field, or runtime provider-selection step that can route TTS provider/model/voice by input or reply language; the existing follow-up comment is consistent with that, and the later CausaMem memory link does not resolve this TTS routing request.

## What This Changes

Request optional language-aware TTS routing so inbound voice replies can select provider/model/voice from detected input or reply language while preserving existing auto-TTS behavior.

## Best Possible Solution

Add a generic, backward-compatible TTS route-resolution contract that preserves optional STT language metadata, supports input/reply/auto source precedence, resolves routes before synthesis, deep-merges with existing overrides, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure messages.tts.auto: "inbound" with one fixed provider/voice and send voice inputs in different languages; current main passes only inboundAudio into auto-TTS and selects provider/model/voice without any language signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
