## Summary

Keep open. Current main still has modality-aware auto-TTS, but no public config/schema, STT metadata field, or runtime provider-selection step that can route TTS provider/model/voice by input or reply language; the prior ClawSweeper comment is still consistent with current source, and the later CausaMem comment does not resolve this TTS routing request.

## What This Changes

Request optional language-aware TTS routing so inbound voice replies can select provider/model/voice from detected input or reply language while preserving existing auto-TTS behavior.

## Best Possible Solution

Add a generic, backward-compatible TTS route-resolution contract that preserves optional STT language metadata, supports input/reply/auto source precedence, resolves routes before synthesis, deep-merges with existing overrides, and documents fallback behavior.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows auto-TTS only receives inboundAudio and the fixed provider resolution path has no language signal or route table.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
