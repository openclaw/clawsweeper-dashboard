## Summary

Keep open. Current main still has only fixed/preference-based TTS provider selection, the strict TTS config has no language-routing contract, and audio transcription results do not preserve detected language for routing.

## What This Changes

The issue requests language-aware TTS provider/model/voice routing for inbound voice replies while preserving current fixed-voice behavior as the fallback.

## Best Possible Solution

Design a backward-compatible generic contract that preserves optional detected-language metadata, resolves input/reply/auto route precedence before synthesis, deep-merges overrides, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure messages.tts.auto:"inbound" with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for detected language to alter provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is generic STT language metadata plus TTS route resolution, not a provider-specific workaround or prompt-only convention.
