## Summary

Keep open. Current main still has only fixed/provider-preference TTS selection plus inbound-audio modality gating; it lacks a public language-routing config contract, does not carry detected STT or reply language into the TTS request, and has no language-derived provider/model/voice routing step. The request crosses config schema, STT metadata, TTS precedence, docs, tests, and plugin-facing behavior, so it should get maintainer API/product design before implementation.

## What This Changes

Add optional language-aware TTS routing so auto-TTS voice replies can choose provider, model, and voice from detected input or reply language while preserving current fallback and per-agent override behavior.

## Best Possible Solution

Design a backward-compatible TTS routing contract that preserves optional transcription language metadata, supports input/reply/auto language sources, deep-merges global, agent, channel, and account TTS settings, routes provider/model/voice before synthesis, and documents/tests fallback when detection or route matching fails.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed provider/voice and send voice inputs in multiple languages; current main has no code path for input or reply language to affect provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable solution should be a generic STT metadata plus TTS routing contract, not a provider-specific workaround or prompt-only convention.
