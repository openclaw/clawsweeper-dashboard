## Summary

Keep open. Current main still has static TTS provider/persona/preference routing plus inbound-audio modality gating, but no language-routing config, preserved detected-language metadata, or route-source contract for TTS synthesis.

## What This Changes

The issue requests language-aware TTS route selection for inbound voice replies so different input or reply languages can use different TTS providers, models, or voices.

## Best Possible Solution

Define a backward-compatible generic TTS route-resolution contract that carries optional STT/input/reply language metadata into synthesis and composes with existing per-agent, channel, account, persona, provider, and directive overrides.

## Reproduction Assessment

Yes, at source level. With messages.tts.auto set to inbound, current main gates on inboundAudio and then resolves provider/model/voice without any input or reply language signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
