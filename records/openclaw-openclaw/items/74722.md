## Summary

Keep open. Current main still lacks a public language-routing config, STT language metadata preservation, and any TTS provider/model/voice resolution step based on input or reply language; the existing per-agent TTS support only covers static overrides.

## What This Changes

Request adds optional language-aware TTS provider/model/voice routing for auto-TTS replies, using detected input or reply language with fallback behavior.

## Best Possible Solution

Define a backward-compatible generic TTS route-resolution contract that preserves optional STT language metadata, supports input/reply/auto source precedence, resolves provider/model/voice before synthesis, and documents fallback behavior.

## Reproduction Assessment

Yes, at source level. Configure messages.tts.auto: "inbound" with one fixed provider/voice and send multilingual voice inputs; current main gates on inboundAudio but has no language signal in provider selection or synthesis.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract rather than a provider-specific workaround or prompt-only convention.
