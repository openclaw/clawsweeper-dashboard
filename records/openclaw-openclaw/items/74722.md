## Summary

Keep open. Current main has fixed TTS provider/model/voice resolution with static per-agent/channel/account overrides, but it does not expose language-routing config, preserve detected STT language in transcription results, or route TTS from input/reply language; the related open issue covers per-agent STT overrides, not this dynamic language-aware TTS routing request.

## What This Changes

Request optional language-aware TTS provider/model/voice routing for audio replies based on detected input or reply language, while preserving the existing fixed TTS fallback behavior.

## Best Possible Solution

Design a backward-compatible generic speech-routing contract that can carry detected or inferred language into TTS provider/model/voice selection, with explicit fallback to the current fixed provider/voice path and focused tests/docs for upgrade behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main lacks the requested language-routing config and detected-language metadata path.

## Solution Assessment
