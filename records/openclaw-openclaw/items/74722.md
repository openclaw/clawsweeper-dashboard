## Summary

Keep open. Current main still implements only static TTS provider/model/voice selection through global, agent, channel, account, prefs, and directive layers; it does not carry detected language metadata from STT or route TTS by input/reply language.

## What This Changes

The issue requests language-aware TTS provider, model, and voice routing for inbound voice replies based on detected input or reply language.

## Best Possible Solution

Maintainers should first choose the speech-routing contract and owner boundary, then implement language metadata propagation plus TTS route selection with fallback to the current fixed provider/voice behavior.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level confirmation: current main lacks the requested language-routing config and detected-language result metadata. The relevant checks are `TtsConfig`, `resolveEffectiveTtsConfig`, auto-TTS attachment, and audio transcription result types.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic speech language-routing and metadata contract, not a one-off provider-specific route table without maintainer API/product review.
