## Summary

Keep open. Current main still supports modality-aware auto-TTS, but it has no public config, schema, runtime context, STT metadata, or provider-resolution step that can choose TTS provider/model/voice by input or reply language.

## What This Changes

Add language-aware TTS routing so inbound voice replies can choose provider, model, and voice from detected input or reply language while preserving existing fallback behavior.

## Best Possible Solution

Design a generic, backward-compatible routing contract that carries optional STT/input or reply language metadata into speech-core, resolves provider/model/voice before synthesis, preserves fallback behavior, and documents agent override merging.

## Reproduction Assessment

Yes. Configure messages.tts.auto as inbound with one fixed provider/voice, send voice inputs in multiple languages, and current main has no code path for detected or inferred language to change provider/model/voice selection.

## Solution Assessment

Unclear for the exact JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
