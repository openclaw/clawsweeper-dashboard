## Summary

Keep open. Current main still lacks a public language-aware TTS routing contract, does not preserve detected STT language in the media-understanding output, and routes auto-TTS from fixed config, prefs, persona, directives, and provider fallback rather than input or reply language.

## What This Changes

Add language-aware TTS routing so inbound voice replies can select provider, model, and voice from detected input or reply language while preserving existing fixed TTS behavior as fallback.

## Best Possible Solution

Design a backward-compatible TTS routing contract that preserves optional detected-language metadata from transcription/preflight, supports input/reply/auto source precedence, deep-merges global and per-agent TTS settings, resolves language routes before synthesis, and documents/tests fallback behavior when detection or route matching fails.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed provider and voice, send voice inputs in multiple languages, and current main has no code path for input or reply language to affect provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable path is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
