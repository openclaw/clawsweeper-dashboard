## Summary

Keep open. Current main has modality-aware auto-TTS, but no public routing config, STT language metadata, or TTS provider/model/voice selection step that can switch by input or reply language; this needs maintainer API/config design before implementation.

## What This Changes

Add language-aware TTS provider/model/voice routing for auto-TTS replies using detected input or reply language while preserving current fallback behavior.

## Best Possible Solution

Add a generic, backward-compatible TTS route resolver that preserves optional STT language metadata, supports input/reply/auto precedence, composes with existing deep-merged overrides, and falls back to the current provider/model/voice when no route matches.

## Reproduction Assessment

Yes. Configure messages.tts.auto: "inbound" with one fixed provider/voice and send audio in different languages; current-main source passes only inboundAudio into auto-TTS and resolves a single active provider before synthesis.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic language metadata plus TTS route-resolution contract, not provider-specific switching or prompt-only behavior.
