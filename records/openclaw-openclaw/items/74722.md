## Summary

Keep open. Current main still lacks a public language-aware TTS routing contract, does not carry input or reply language into auto-TTS synthesis, and resolves provider/model/voice without a language-derived routing step. The request is valid, but it crosses public config, STT metadata, provider routing precedence, docs, and plugin-facing behavior, so it needs maintainer API/product design before implementation.

## What This Changes

Request language-aware TTS provider, model, and voice routing for automatic inbound voice replies using detected input or reply language while preserving current fallback behavior.

## Best Possible Solution

Design a backward-compatible TTS routing contract that preserves optional detected-language metadata from transcription/preflight, supports input/reply/auto source precedence, deep-merges global and per-agent TTS settings, routes provider/model/voice before synthesis, and documents/tests fallback behavior when detection or route matching fails.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with a fixed provider and voice, send voice inputs in different languages, and current main has no code path for input or reply language to change provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable solution should be a generic STT metadata plus TTS routing contract rather than a provider-specific workaround or prompt-only convention.
