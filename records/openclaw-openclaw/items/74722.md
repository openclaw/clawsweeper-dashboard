## Summary

Keep open. Current main supports static TTS provider/model/voice resolution with global, agent, channel, account, prefs, and inline override layers, but it still has no language-routing config, no detected-language field in transcription results, and no route lookup that can choose TTS provider/model/voice from input or reply language.

## What This Changes

Request adds optional language-aware TTS provider/model/voice routing for inbound voice replies using detected input or reply language.

## Best Possible Solution

Design a generic speech-routing contract that carries detected or inferred language into TTS provider/model/voice selection while preserving the current fixed-provider fallback path when routing is absent or fails.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main lacks the requested language-routing config and detected-language metadata handoff.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic speech-routing and language metadata contract, not a provider-specific Piper-style route or prompt-only workaround.
