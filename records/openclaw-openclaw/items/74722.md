## Summary

Keep open. Current main still supports modality-aware auto-TTS, but it has no public config/schema, STT metadata propagation, or synthesis-time provider selection path that can route provider/model/voice by input or reply language.

## What This Changes

Add language-aware TTS routing so auto-TTS can select provider/model/voice from detected input or reply language while preserving current fallback behavior.

## Best Possible Solution

Add a backward-compatible TTS routing contract that carries optional STT detected-language metadata, resolves input/reply/auto source precedence before synthesis, deep-merges through existing TTS override layers, and documents fallback behavior.

## Reproduction Assessment

Yes. Configure messages.tts.auto: "inbound" with one fixed provider/voice, send voice inputs in multiple languages, and current main only passes an inboundAudio boolean plus reply text into synthesis, so language cannot affect provider selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS routing contract, not a provider-specific workaround or prompt-only convention.
