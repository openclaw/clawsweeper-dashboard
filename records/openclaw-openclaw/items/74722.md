## Summary

Keep open. Current main still supports modality-aware Auto-TTS, but there is no public config/schema/runtime path that can preserve detected language metadata or route TTS provider/model/voice by input or reply language.

## What This Changes

Add optional language-aware TTS routing so inbound multilingual voice replies can select provider, model, and voice from detected input or reply language while preserving existing fallback behavior.

## Best Possible Solution

Design a backward-compatible generic routing contract that preserves optional STT language metadata, supports input/reply/auto source precedence, deep-merges with existing overrides, resolves routes before synthesis, and documents fallback behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `messages.tts.auto: "inbound"` with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for detected or inferred language to alter provider/model/voice selection.

## Solution Assessment

Unclear for the exact JSON shape in the request. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
