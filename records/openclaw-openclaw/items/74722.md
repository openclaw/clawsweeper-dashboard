## Summary

Keep open. Current main supports inbound auto-TTS and per-agent/provider overrides, but it does not define language-aware routing, preserve detected STT language metadata, or route TTS provider/model/voice by input or reply language.

## What This Changes

Request optional language-aware routing of TTS provider, model, and voice for auto-TTS replies based on detected input or reply language.

## Best Possible Solution

Design an additive language-routing contract around speech-core provider selection and media-transcription metadata, then document and test fallback behavior across global, agent, channel, and account TTS overrides.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, not broken documented behavior. Source inspection confirms current main only gates auto-TTS on inbound audio and resolves configured provider/voice settings.

## Solution Assessment

Unclear until maintainers choose the API shape. The direction is plausible, but the sample config should not be implemented verbatim before deciding where detected language metadata lives and how routes compose with existing TTS precedence.
