## Summary

Keep open. Current main and v2026.6.10 still choose TTS provider/model/voice from static TTS config or directives; they do not preserve detected STT language or apply per-turn language-aware routing.

## What This Changes

Request adds optional language-aware TTS routing so inbound voice replies can select provider, model, and voice from detected input or reply language.

## Best Possible Solution

Keep this issue open until maintainers choose the speech-routing contract, then implement a bounded schema/runtime/docs/test path that preserves current static TTS behavior when routing is disabled.

## Reproduction Assessment

Yes, at source level. Current main has inbound audio gating and static TTS config resolution, but no language-routing config or detected-language propagation into the TTS call.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable solution needs a maintainer-approved routing contract for input-vs-reply language source, fallback semantics, and plugin/provider boundaries before implementation.
