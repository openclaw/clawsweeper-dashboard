## Summary

Keep open. Current main still has static TTS provider/persona/preference selection plus inbound-audio modality gating, but no language-routing config, preserved STT language metadata, or input/reply language signal in TTS synthesis.

## What This Changes

The issue requests language-aware TTS provider/model/voice routing for inbound voice replies based on detected input or reply language.

## Best Possible Solution

Define a backward-compatible generic TTS route-resolution contract that carries optional STT/input/reply language metadata into synthesis, documents fallback precedence, and composes with existing per-agent, channel, account, persona, and provider overrides.

## Reproduction Assessment

Yes, at source level. With messages.tts.auto set to "inbound", current main only gates on inboundAudio and then resolves TTS provider/model/voice without any input or reply language signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
