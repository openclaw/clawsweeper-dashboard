## Summary

Keep open. Current main still lacks a language-aware TTS routing contract: TTS config accepts fixed provider/provider-config fields only, STT/audio transcription results do not carry detected-language metadata, and auto-TTS synthesis receives only a boolean inbound-audio signal before resolving one provider chain.

## What This Changes

Add optional language-aware TTS routing so inbound voice replies can choose TTS provider, model, and voice from detected input or reply language while preserving the current fixed-provider fallback behavior.

## Best Possible Solution

Design a backward-compatible public contract that preserves optional detected-language metadata, resolves input/reply/auto route precedence before synthesis, composes with provider/persona/directive fallback behavior, and documents global plus per-agent override semantics.

## Reproduction Assessment

Yes. Configure `messages.tts.auto: "inbound"` with one fixed provider/voice, send voice inputs in different languages, and current main has no code path for detected language to affect provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT language metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
