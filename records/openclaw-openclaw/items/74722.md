## Summary

Keep open. Current main still lacks a public language-aware TTS routing contract, does not pass input or reply language into automatic TTS synthesis, and preserves audio transcription output as text plus model metadata only; the request is valid but needs maintainer API/product design before an implementation PR is queued.

## What This Changes

Request optional language-aware TTS provider, model, and voice routing for auto-TTS replies using inbound or reply language, with backward-compatible global and per-agent configuration.

## Best Possible Solution

Design a backward-compatible TTS routing contract, preserve optional detected-language metadata from audio transcription/preflight, route provider/model/voice before synthesis across global, per-agent, channel, account, preference, persona, and directive layers, then document and test fallback behavior.

## Reproduction Assessment

Yes for the current limitation. Source and docs show auto: "inbound" only gates whether audio is synthesized, and the runtime has no input or reply language path into provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The right solution is a generic TTS/STT metadata and routing contract rather than a provider-specific workaround or prompt-only convention.
