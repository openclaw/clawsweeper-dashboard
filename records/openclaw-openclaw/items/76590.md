## Summary

Keep open. The hash-epoch approach is useful for the sandbox rollout, but this PR prelabels the new hash epoch as the `--init` rollout while the actual `--init` flag remains in #74083, so it can make that rollout miss containers if merged and used first.

## What This Changes

The PR adds a Docker create-args epoch to sandbox and browser config hashes, stamps it as a Docker label, updates hash/recreate tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Co-land the epoch bump with the actual Docker create-arg change, or use a neutral bootstrap epoch here and require the `--init` PR to bump it when that flag lands.

## Reproduction Assessment

Yes, source-level. Current main reuses containers by a config hash that excludes create-arg-only changes, and the PR diff shows the final `--init` epoch can be written into that hash before `--init` exists.

## Solution Assessment

No, not as currently sequenced. The design is narrow, but the epoch value must not claim the `--init` rollout before the actual flag is in the same effective update path.
