## Label Justifications

- `P1`: The linked sandbox issue can exhaust process slots in real long-running agent sandboxes, and this PR’s rollout defect can affect those workflows during upgrade.
- `merge-risk: 🚨 compatibility`: The PR changes persisted container hash compatibility, so existing containers labelled with the old hash can be treated as stale even with no create-args change.

## Summary

Keep open: the create-args epoch is a useful companion to the sandbox zombie-process fix, but this branch is not merge-ready because the neutral bootstrap value participates in the config hash and can recreate existing containers before any real flag change, the branch is currently conflicting, and external real Docker proof is missing.

## What This Changes

The PR adds a sandbox Docker-create-args epoch to sandbox and browser config hashes, stamps it as a container label, updates hash/recreate tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased sandbox fix that pairs the first create-args epoch participation with the actual Docker flag change, preserves current main hash inputs, removes the release-owned changelog edit, and includes real Docker proof.

## Reproduction Assessment

Yes for a source-level path: current main hashes do not include createArgsEpoch, while this PR makes the bootstrap epoch required and feeds it into expectedHash. I did not run a live Docker upgrade/recreate workload in this read-only review.

## Solution Assessment

No. The epoch mechanism is useful, but as a separate bootstrap-only PR it invalidates existing hashes without a real create-args change; the safer fix is to pair participation with the first actual flag change or keep the bootstrap out of the hash until then.
