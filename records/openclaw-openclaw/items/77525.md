## Summary

Keep open: current main still has the reported failure shape, with `/status` rendering unknown context when fresh totals are absent and the xAI provider wrapper not normalizing Grok usage into the existing fresh-token path.

## What This Changes

The issue requests that xAI/Grok sessions show actual `/status` context usage instead of `?/1.0m` when the context window is known.

## Best Possible Solution

Populate xAI/Grok usage into the existing `lastCallUsage` or `promptTokens` accounting path while preserving the stale-total guard in the status renderer.

## Reproduction Assessment

Yes from source inspection for the visible symptom: a Grok session with known `contextTokens` but no fresh `totalTokens` will render `?/1.0m`. I did not run a live xAI request here, so the provider payload gap is source-reproducible rather than end-to-end reproduced.

## Solution Assessment

Yes on direction: the narrow maintainable fix is to make xAI usage metadata feed the existing fresh context accounting path. Loosening `/status` to infer from stale cumulative totals would regress the stale-total guard fixed by related status work.
