## Summary

Keep open: the issue is a valid status-token accounting bug, and current main does not show a clear post-release fix for the xAI/Grok usage path that the reporter captured.

## What This Changes

The issue requests real context-token usage in `/status` for xAI/Grok sessions instead of `?/1.0m`.

## Best Possible Solution

Normalize xAI Responses completed-event usage into the existing last-call prompt-token accounting path without relaxing the stale-total guard.

## Reproduction Assessment

Yes from source plus reporter-supplied live payloads, but not from a local end-to-end run. Current main will render `?/1.0m` when xAI usage reaches the session store as zero/missing prompt tokens, and the issue shows that behavior on 2026.5.3-1.

## Solution Assessment

Yes on direction: fix the xAI/Responses usage capture path, not the renderer. The renderer is correctly refusing to invent a context total when the stored usage snapshot is stale.
