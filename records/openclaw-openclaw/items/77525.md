## Summary

Keep open: the report still points to a real xAI/Grok usage-capture gap rather than a status-rendering or context-window lookup problem, and current `main` does not show a newer source change beyond the reported release that would make the raw xAI usage reach fresh session totals.

## What This Changes

Grok/xAI sessions should show actual `/status` context usage instead of `?/1.0m` when xAI Responses returns token usage.

## Best Possible Solution

Capture xAI Responses usage before it is flattened to zeros and feed it into the existing fresh `lastCallUsage`/`promptTokens` accounting path, leaving the status renderer's stale-total guard intact.

## Reproduction Assessment

Yes from source inspection and supplied artifacts, but not from a live current-main Grok run here. The reported raw xAI usage shape is supported by the central normalizer, while the current persistence path still renders unknown when no fresh usage snapshot reaches it.

## Solution Assessment

Yes on direction: the narrow fix should happen upstream of `buildStatusMessage`, in the xAI/pi stream or subscription usage-capture path. Loosening the renderer would regress the deliberate stale-total behavior fixed by earlier status issues.

Keep the status renderer's stale-total guard unchanged. First trace whether the usage is available on pi assistant events, `assistantMessageEvent`, stream records, or the xAI provider wrapper. Add regression coverage for the xAI-shaped usage event and the persisted status/session total. Do not add config flags, do not synthesize totals from stale cumulative `inputTokens + outputTokens`, and do not change xAI catalog context-window values. Add a `CHANGELOG.md` fix entry if the patch changes user-facing behavior.
