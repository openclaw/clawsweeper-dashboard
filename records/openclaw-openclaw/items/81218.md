## Summary

Keep open: current main still short-circuits self-hosted provider discovery whenever an explicit vLLM/SGLang provider config exists, while the shared wildcard model policy says `provider/*` entries should keep provider-scoped catalogs dynamic. The fix looks narrow in shared provider discovery, with vLLM/SGLang tests and docs updates.

## What This Changes

Make vLLM/SGLang `provider/*` allowlist entries discover models from explicitly configured self-hosted provider endpoints.

## Best Possible Solution

Preserve exact `provider/model` entries as manual allowlists, but when `agents.defaults.models` contains `provider/*`, run the provider catalog against the configured endpoint so advertised models appear consistently across CLI, Gateway, and channel model views.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure `models.providers.vllm` or `models.providers.sglang` plus `agents.defaults.models["provider/*"]`, then the shared helper returns before the provider builder can query `/models`. I did not run a live endpoint in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is in the shared self-hosted discovery path, distinguishing wildcard-driven dynamic discovery from exact manual model entries and passing the explicit endpoint details into the existing provider builders.
