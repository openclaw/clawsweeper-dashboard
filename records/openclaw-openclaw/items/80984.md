## Summary

Keep open: current main still lacks a typed no-prior-chat cron delivery error, and the empty-session `last` path can still expose generic channel-selection text in cron preview/task errors. The adjacent webchat PR and implicit-route PRs address related but different surfaces, not this diagnostic cleanup.

## What This Changes

Request a typed, actionable cron delivery error for `delivery.channel: "last"` or an omitted channel when the selected session has no prior chat route.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list wording for `last` without session history, while preserving explicit targets, configured-channel fallback, and related admission-time guards.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still reaches generic channel selection and exposes its wrapped error.

## Solution Assessment

Yes, with the seam kept on the cron side. The maintainable fix is to detect the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` and surface a typed actionable error without moving session-history semantics into the generic channel selector.
