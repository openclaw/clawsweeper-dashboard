## Summary

Keep open: current main and v2026.6.6 still do not define a typed no-prior-chat error, and the no-history/no-fallback cron path can still surface the generic channel-selection failure. Related open cron PRs change nearby delivery semantics but do not own this typed diagnostic request.

## What This Changes

Request a typed, actionable cron delivery error for `delivery.channel: "last"` or omitted channel when the resolved session has no prior chat route and no fallback channel.

## Best Possible Solution

Settle the nearby cron delivery semantics first, then add a small typed no-prior-chat error in the cron delivery resolver with preview/runtime tests if `last` remains a fail-closed route.

## Reproduction Assessment

Yes, by source inspection: a `last` cron delivery with no session `lastChannel` asks generic channel selection and can return the generic no-configured-channels error. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers settle the nearby cron delivery behavior changes. If `last` remains fail-closed, the narrow resolver-side typed error with preview/runtime tests is the maintainable fix.
