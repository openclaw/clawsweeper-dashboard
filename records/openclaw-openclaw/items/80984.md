## What This Changes

The issue requests a typed, actionable cron delivery error when `delivery.channel` is `last` or omitted but the selected session has no previous chat route and no fallback channel can be resolved.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list wording for unresolved `last` session history, while preserving explicit channels, provider-prefixed targets, single-configured-channel fallback, and `delivery.mode: "none"`.

## Reproduction Assessment

Yes, source-level: a cron announce job with omitted or `last` channel, no selected session `lastChannel`, and no resolvable fallback still flows through generic channel selection and returns the wrapped generic selector error. I did not execute a live cron job or tests in this read-only review.

## Solution Assessment

Yes, with the seam kept cron-side. Detecting the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` is narrower than teaching generic channel selection about session history, and it can preserve the existing fallback and explicit-target behavior.
