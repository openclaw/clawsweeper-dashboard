## Summary

Keep open: current main still lacks the requested typed no-prior-chat error. The cron resolver falls through to generic channel-selection wrapping for the `last`/empty-session case, and the related webchat PR explicitly leaves this issue as follow-up work.

## What This Changes

Request a typed cron delivery error for `delivery.channel: "last"` when the selected session has no prior chat history and no fallback channel can be resolved.

## Best Possible Solution

Teach cron delivery resolution to return a typed no-prior-chat error for `last` or omitted channel when the selected session lacks `lastChannel` and fallback selection cannot resolve, then cover the preview/list-facing message with focused tests.

## Reproduction Assessment

Yes, by source inspection. An announce delivery with `channel: "last"` or no channel, no selected session `lastChannel`, and no resolvable fallback enters `resolveDeliveryTarget`, falls into channel selection, and currently produces generic or plain `Error` text rather than the requested typed no-prior-chat error.

## Solution Assessment

Yes. A narrow cron-delivery typed error is the maintainable direction because `channel-selection` does not know whether the failure came from empty session history, while `resolveDeliveryTarget` already has that context.
