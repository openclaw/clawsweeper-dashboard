## Summary

Keep open: current main still lacks the requested typed/actionable no-prior-chat diagnostic, and the empty-session `last` path still wraps generic channel-selection errors into cron preview/list output. The related open PRs address adjacent webchat or ambiguous multi-channel admission failures, not this session-state-specific error family.

## What This Changes

Request a typed, actionable cron delivery error when `delivery.channel` is `last` or omitted and the selected session has no prior chat route or resolvable fallback.

## Best Possible Solution

Add a cron-side typed no-prior-chat diagnostic in `resolveDeliveryTarget` and preview/list output while preserving explicit channel, provider-prefixed target, single-configured-channel fallback, and adjacent webchat/ambiguous-route behavior.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted or `last` channel, no selected session `lastChannel`, and no resolvable fallback still reaches generic channel selection and returns a wrapped generic resolver error.

## Solution Assessment

Yes. A narrow cron-side typed error is the maintainable fix because the root cause depends on session history; putting it only in generic channel selection would mix session-state semantics into config/plugin channel selection.
