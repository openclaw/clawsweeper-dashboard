## Summary

Keep open: current main still lacks the requested typed no-prior-chat cron delivery error, and the `last`/empty-session path still reaches generic channel selection before returning a wrapped generic error. The related open webchat PR explicitly leaves this `delivery.channel="last"` case out of scope.

## What This Changes

Add a typed cron delivery error for `delivery.channel: "last"` or inferred last-channel delivery when the selected session has no prior chat and no fallback channel.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list message for `last` without session history, while preserving configured-channel fallback and explicit target resolution.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still flows through `resolveMessageChannelSelection({ cfg })` and returns the generic wrapped channel-selection error.

## Solution Assessment

Yes, with a seam adjustment. The safest fix is to detect the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` and use a typed actionable error there; putting it only in generic channel selection would mix config/plugin selection with session history semantics.
