## Summary

Keep open: current main still lacks a typed no-prior-chat cron delivery error, and the empty-session `last` route still falls through generic channel selection before surfacing a generic preview/task error. The adjacent webchat PR is still open and explicitly leaves this session-history case out of scope.

## What This Changes

The issue requests an actionable typed cron delivery error for `delivery.channel="last"` or omitted channel when the selected session has no prior chat route and no fallback channel can resolve.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list message for `last` without session history, while preserving explicit channels, provider-prefixed targets, single-configured fallback, stored delivery context, and allowlist rerouting.

## Reproduction Assessment

Yes, source-level. A cron announce route with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still reaches `resolveMessageChannelSelection({ cfg })` and returns a generic wrapped channel-selection error.

## Solution Assessment

Yes, with a cron-boundary seam. The maintainable fix is to detect the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` rather than making generic channel selection know about session history.
