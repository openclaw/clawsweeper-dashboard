## Summary

Keep open: current main still lacks the requested typed no-prior-chat cron delivery error, and the fresh-session `last` path still falls through to generic channel selection and preview output. The linked webchat PR is open and explicitly leaves this `delivery.channel="last"` case out of scope.

## What This Changes

Add a typed, actionable cron delivery error for omitted or `last` delivery when the selected session has no prior chat route and no fallback channel is resolvable.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list message for `last` without session history, while preserving explicit targets, provider-prefixed targets, and configured-channel fallback behavior.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still flows through `resolveMessageChannelSelection({ cfg })` and returns the generic wrapped channel-selection error.

## Solution Assessment

Yes, with a seam adjustment. The safest fix is to detect the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` and use a typed actionable error there rather than mixing session history semantics into the generic channel selector.
