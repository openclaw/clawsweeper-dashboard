## Summary

Keep open: current main still does not provide the requested typed no-prior-chat cron delivery error; the `last`/empty-session path still falls through generic channel selection and surfaces a wrapped generic channel-selection message. The linked webchat PR is adjacent and explicitly leaves this `delivery.channel="last"` case out of scope.

## What This Changes

Request a typed, actionable cron delivery error for `delivery.channel: "last"` or omitted channel when the selected session has no prior chat route and no fallback channel resolves.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list message for `last` without session history, while preserving configured-channel fallback, provider-prefixed targets, and explicit delivery target resolution.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still flows through generic `resolveMessageChannelSelection({ cfg })` and returns the wrapped channel-selection error.

## Solution Assessment

Yes, with a seam adjustment. The safest fix is to detect the session-state-specific no-prior-chat condition in `resolveDeliveryTarget` and use a typed actionable error there; putting it only in generic channel selection would mix config/plugin selection with session history semantics.
