## Summary

Keep open. Current main still routes the empty-session `delivery.channel: "last"` path through generic channel-selection errors, and the linked webchat PR explicitly leaves this follow-up out of scope.

## What This Changes

Requests a typed, actionable cron delivery error when `delivery.channel` is `last` or omitted but the resolved session has no prior deliverable chat route.

## Best Possible Solution

Add a focused typed/actionable diagnostic at the cron delivery target boundary while preserving provider-prefixed targets, single-configured-channel fallback, explicit channel errors, and current delivery semantics.

## Reproduction Assessment

Yes, by source inspection. A `last` or omitted channel with no session `lastChannel` flows through `resolveMessageChannelSelection` and the preview/list surface displays the resulting generic error.

## Solution Assessment

Yes. A narrow typed cron diagnostic is the maintainable fix; it should not change fallback routing or turn this into a broader channel-selection policy change.
