## Summary

Keep this open. Current main still has the no-prior-chat cron path flowing into generic channel selection, and the linked webchat PR explicitly treats this as separate follow-up work rather than a fix for this issue.

## What This Changes

The issue requests a distinct typed/actionable error for cron announce delivery when `delivery.channel` is `last` or omitted and the resolved session has no prior chat route.

## Best Possible Solution

Add the typed/actionable no-prior-chat diagnostic in the cron delivery resolution path, preserve existing fallback selection semantics, and cover the runtime plus preview/list output with focused tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is an isolated cron announce target with `delivery.channel` omitted or `last`, an empty relevant session entry, and no resolvable fallback channel; current main returns the generic selector error through `resolveDeliveryTarget` and `resolveCronDeliveryPreview`.

## Solution Assessment

Yes, with the seam kept narrow. The best fix is to distinguish the session-history miss in `resolveDeliveryTarget` or an explicitly typed helper around that path, while leaving generic channel-selection errors for non-cron callers unchanged.
