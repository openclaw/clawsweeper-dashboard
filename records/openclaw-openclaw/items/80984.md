## Summary

Keep open: current main still routes the no-prior-chat `delivery.channel="last"` cron case through generic channel selection and raw preview/list error text. The related webchat PR at https://github.com/openclaw/openclaw/pull/80985 is open and explicitly leaves this `last` case out of scope.

## What This Changes

Requests a typed, actionable cron delivery error when implicit or `last` delivery has no previous chat route for the selected session.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and preview/list wording for `last` without session history, while preserving explicit channel/target handling and configured-channel fallback.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still flows through generic channel selection and returns a wrapped generic error.

## Solution Assessment

Yes, with a seam adjustment. The maintainable fix is to detect the session-history-specific no-prior-chat condition in `resolveDeliveryTarget`, not to make the generic channel selector infer session-store semantics.
