## Summary

Keep open: current main still lacks the requested typed no-prior-chat cron delivery error, and the empty `last` session-history path still falls through to generic channel selection before surfacing a wrapped generic error. The open webchat PR is adjacent but explicitly out of scope, and the open implicit-last admission PR does not replace the runtime/preview diagnostic requested here.

## What This Changes

The issue requests a typed, actionable cron delivery error for `delivery.channel="last"` when the selected session has no stored prior chat route.

## Best Possible Solution

Add a cron-side typed/actionable no-prior-chat route error and preview/list message for unresolved `last` session history, while preserving explicit channels, provider-prefixed targets, single-configured-channel fallback, and configured fallback routes.

## Reproduction Assessment

Yes, source-level. A cron announce route with omitted or `last` channel, no selected session `lastChannel`, and no usable fallback still reaches generic channel selection and returns a wrapped generic error.

## Solution Assessment

Yes, with the cron resolver as the clean seam. The no-prior-chat condition is session-state-specific, so detecting it in `resolveDeliveryTarget` is safer than pushing session semantics into generic channel selection.
