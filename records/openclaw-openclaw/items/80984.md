## Summary

Keep open: current main still lacks the requested typed no-prior-chat cron delivery error. The `delivery.channel: "last"`/empty-session path still falls through to generic channel selection and the preview surface shows the wrapped generic error; the linked webchat PR is open and explicitly leaves this case out of scope.

## What This Changes

Request a typed, actionable cron delivery error for omitted or `last` delivery channels when the selected session has no prior chat route and no fallback channel is resolvable.

## Best Possible Solution

Add a narrow cron-side typed no-prior-chat route error and clear preview/list wording while preserving explicit targets, provider-prefixed targets, and configured-channel fallback behavior.

## Reproduction Assessment

Yes, source-level. A cron announce target with omitted/`last` channel, no selected session `lastChannel`, and no resolvable fallback still reaches `resolveMessageChannelSelection({ cfg })` and returns the wrapped generic selector error.

## Solution Assessment

Yes, with the error created at the cron delivery seam. Detecting the no-prior-chat condition in `resolveDeliveryTarget` keeps session-history semantics out of the generic config/plugin channel selector.
