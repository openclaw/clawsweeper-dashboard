## Summary

Keep open: the PR addresses a real current-main config writeback bug, but the branch is conflicting and still has a writer correctness gap around readable validation-invalid disk configs plus no after-fix real behavior proof.

## What This Changes

The PR updates config writeback to use the latest valid on-disk snapshot as the runtime projection base and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one readable current disk snapshot for both projection and persistence, preserves recoverable disk content after validation failures, and keeps explicit deletions routed through `unsetPaths`.

## Reproduction Assessment

Yes, source-level: set a stale runtime/source snapshot, add `session.dmScope` on disk, then call `writeConfigFile(loadConfig())`; current main projects through the stale source snapshot and merge-patch deletes can remove disk-only fields. I did not run tests because this review is read-only.

## Solution Assessment

No as currently patched. The PR targets the right writer path, but the maintainable fix should use readable disk snapshots even when validation fails and should share that snapshot with the inner persistence write.
