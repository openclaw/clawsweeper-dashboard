## Summary

Keep open. Current main still has the stale runtime-source projection path, and this PR is a focused fix attempt, but the branch is conflicting, lacks real behavior proof, and still has a validation-failure gap in the writer logic; the broader overlapping config-preservation PR is open, not merged.

## What This Changes

The PR changes exported config writeback to prefer a freshly read on-disk snapshot as the projection base during runtime snapshot writeback and adds a `session.dmScope` stale-snapshot regression test.

## Best Possible Solution

Land a rebased writer-level fix that uses one current existing/readable disk snapshot as the projection and persistence base, preserves readable disk content across validation failures, and keeps focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: with runtime/source snapshots active, adding `session.dmScope` directly on disk after the source snapshot and then calling `writeConfigFile(loadConfig())` still projects through `runtimeConfigSourceSnapshot` on current main. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. The PR targets the right writer path, but the maintainable fix should reuse one existing/readable disk snapshot for both projection and persistence and avoid stale runtime-source fallback solely because validation failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection bases when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but the inner writer rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, the projection base and persistence base can diverge, leaving this writeback path with another avoidable key-loss window.
  - confidence: 0.74
