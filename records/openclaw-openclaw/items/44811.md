## Summary

Keep open: current main still has the writer-level stale runtime projection path, while this PR is not merge-ready because the latest diff leaves readable-but-invalid disk configs on the stale fallback, rereads disk independently, and lacks real behavior proof.

## What This Changes

This PR changes runtime config writeback to project runtime-derived writes onto the latest valid existing disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that reads one current existing disk snapshot, preserves readable disk content even when validation fails, passes that snapshot to the inner writer, and keeps focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: set runtime/source snapshots, add `session.dmScope` directly on disk after the source snapshot, then call `writeConfigFile(loadConfig())`; the wrapper still projects through `runtimeConfigSourceSnapshot` and the merge-patch write treats omitted keys as deletions.

## Solution Assessment

No. The PR targets the correct writer path, but the `valid && exists` guard and independent inner reread leave correctness gaps; the narrower maintainable solution is to reuse one readable existing disk snapshot for both projection and persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection base when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, `nextCfg` can omit a key present in the inner base and the merge-patch write can treat that as a deletion.
  - confidence: 0.74
