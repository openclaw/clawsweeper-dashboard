## Summary

Keep open: this PR targets a real current-main config writeback bug, but the branch is not merge-ready because it conflicts with main, lacks after-fix real behavior proof, and still has writer correctness gaps around invalid-but-readable disk snapshots and double-reading the projection base.

## What This Changes

The PR changes exported config writeback to project runtime-derived writes onto a freshly read disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one current existing disk snapshot for both projection and persistence, preserves readable disk content even when validation fails, and keeps focused regression coverage for stale runtime snapshot writeback.

## Reproduction Assessment

Yes, by source inspection: set runtime/source snapshots, add `session.dmScope` directly on disk after the source snapshot, then call `writeConfigFile(loadConfig())`. Current main projects through the stale `runtimeConfigSourceSnapshot`, and merge-patch deletion semantics can remove the disk-only key.

## Solution Assessment

No. The PR targets the correct writer path, but the `valid && exists` guard and independent inner reread leave correctness gaps; the narrower maintainable fix is to reuse one readable existing disk snapshot for both projection and persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection bases when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, `nextCfg` can omit a key present in the inner base and the merge-patch write can treat that as a deletion.
  - confidence: 0.74
