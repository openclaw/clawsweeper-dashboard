## Label Justifications

- `P1`: The PR targets a real config writeback path that can silently drop session and auth-provider configuration during gateway/runtime workflows.
- `merge-risk: 🚨 compatibility`: The patch changes core persisted-config writeback behavior and can affect existing config preservation, validation, and upgrade paths.
- `merge-risk: 🚨 session-state`: The motivating failure drops `session.dmScope`, which controls DM session isolation and routing behavior.
- `merge-risk: 🚨 auth-provider`: The same runtime/source projection path preserves SecretRef-style provider auth values, so writer mistakes can affect auth/provider persistence.

## Summary

Keep open: current main still has the stale runtime snapshot writeback path and this PR is a focused fix attempt, but the branch is conflicting, lacks after-fix real behavior proof, and leaves a readable-invalid config path that can still drop disk-only keys.

## What This Changes

The PR changes `writeConfigFile` to prefer a latest on-disk snapshot as the runtime projection base and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one readable existing disk snapshot for both projection and persistence, preserves recoverable disk content after validation failures, and keeps focused stale-runtime-snapshot regression coverage.

## Reproduction Assessment

Yes, source-level: set a stale runtime/source snapshot, add `session.dmScope` on disk, then call `writeConfigFile(loadConfig())`; current main projects through the stale source snapshot and merge-patch deletes can remove disk-only fields. I did not run tests because this review is read-only.

## Solution Assessment

No as currently patched. The PR targets the right writer path, but the maintainable fix should use readable disk snapshots even when validation fails and should share that snapshot with the inner persistence write.
