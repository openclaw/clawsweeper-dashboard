## Summary

Keep open: the PR targets a real config-loss bug and adds useful focused coverage, but the branch is conflicting, lacks external real-behavior proof, and still has writer correctness gaps around invalid disk snapshots and duplicate snapshot reads.

## What This Changes

The PR changes runtime config writeback to project onto the latest valid on-disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one readable existing disk snapshot for projection and persistence, preserves recoverable disk content even after validation failures, and keeps focused stale-runtime-snapshot regression coverage.

## Reproduction Assessment

Yes, source-level: set a stale runtime/source snapshot, add `session.dmScope` on disk, then call `writeConfigFile(loadConfig())`; current main projects the runtime patch onto the stale source snapshot and can emit merge-patch deletes for disk-only fields. I did not run tests because this review was read-only.

## Solution Assessment

No as currently patched. The PR targets the right writer path, but the maintainable fix should use readable disk snapshots even when validation fails and should share that snapshot with the inner persistence write.
