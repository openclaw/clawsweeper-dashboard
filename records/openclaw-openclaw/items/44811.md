## Summary

Keep open. Current main still has the stale runtime-source projection path, and this PR is aimed at a real config data-loss bug, but the current branch is conflicting, lacks after-fix real behavior proof, and leaves writer correctness gaps in the changed code.

## What This Changes

The PR changes exported config writeback to project runtime-derived writes onto a freshly read disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one current existing/readable disk snapshot as both projection and persistence base, preserves readable disk content when validation fails, and keeps focused regression coverage for stale runtime snapshot writeback.

## Reproduction Assessment

Yes, by source inspection: with runtime/source snapshots active, add `session.dmScope` directly on disk after the source snapshot, then call `writeConfigFile(loadConfig())`. Current main computes a runtime patch and applies it to the stale `runtimeConfigSourceSnapshot` before persisting.

## Solution Assessment

No. The PR targets the correct writer path, but the maintainable fix should reuse one existing/readable disk snapshot for both projection and persistence and avoid stale runtime-source fallback solely because validation failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection bases when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, `nextCfg` can omit a key present in the inner base and the merge-patch write can treat that as a deletion.
  - confidence: 0.74
