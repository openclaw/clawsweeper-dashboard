## Summary

Keep open. The PR targets a real current-main config writeback bug and adds a useful dmScope regression test, but it should not merge as written because readable validation-invalid disk configs can still be projected from the stale runtime source snapshot, the write path still uses two different disk snapshots, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR changes config writeback so runtime-derived writes prefer a fresh existing disk snapshot as the projection base and adds a regression test preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a focused config-writer fix that chooses one current existing disk snapshot for the whole writeback, uses its readable source/resolved payload when validation fails after parsing, falls back to the runtime source snapshot only for missing or genuinely unreadable files, keeps the dmScope stale-snapshot regression, adds invalid-readable and single-snapshot coverage, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction is to create a runtime snapshot from an older source config, add `session.dmScope` directly on disk after that snapshot, then call `writeConfigFile(loadConfig())`; current main projects the runtime patch onto the stale source snapshot and can delete the disk-only key.

## Solution Assessment

No, not as written. The PR is in the right narrow area, but the `latestSnapshot.valid` guard and second disk read leave data-loss paths in the same preservation contract, and the required changelog entry is missing.

Full review comments:

- **[P2] Use readable snapshots even when validation fails:** `src/config/io.ts:1519-1523`
  - body: The new branch only uses `latestSnapshot.resolved` when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails schema/plugin validation, the snapshot still carries the readable disk source, but this code falls back to the stale runtime source snapshot, so runtime writeback can still delete disk-only keys in the same `session.dmScope` scenario.
  - confidence: 0.86
- **[P3] Reuse the snapshot used for projection:** `src/config/io.ts:1518`
  - body: The added read computes `nextCfg` from one disk snapshot, but the inner writer then reads the config file again because no `baseSnapshot` is passed. A key added between those reads is absent from `nextCfg`, so the inner merge patch can treat it as a deletion; thread the same snapshot through the existing base-snapshot path or otherwise keep the write based on one snapshot.
  - confidence: 0.72

  - body: This fixes a user-facing config persistence bug affecting `session.dmScope`, but the PR changes only code and tests. OpenClaw policy requires a single-line `CHANGELOG.md` entry for user-facing fixes, credited to the contributor.
