## Summary

Keep open: this PR targets a real config writeback bug that still exists on current main, but the proposed patch is not merge-ready because it leaves a validation-invalid disk snapshot gap, keeps a double-read race in the write path, and lacks real after-fix behavior proof.

## What This Changes

The PR changes runtime-derived config writes to project onto the latest valid disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased config-writer fix that reads one current existing disk snapshot, uses its resolved source content for projection even when validation fails, passes that snapshot as `baseSnapshot` into the inner writer, and keeps focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set runtime/source snapshots, add `session.dmScope` directly on disk after that source snapshot, then call `writeConfigFile(loadConfig())`; current main still projects through the stale runtime source snapshot, though this read-only review did not execute the test.

## Solution Assessment

No. The PR targets the correct writer path, but the `valid && exists` guard and second disk read leave correctness gaps; the safer solution is to reuse one readable disk snapshot for both projection and persistence.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.89

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection base when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Pass the projection snapshot into the inner writer:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, `nextCfg` can omit a key present in the inner base and the merge-patch write can treat that as a deletion.
  - confidence: 0.74
