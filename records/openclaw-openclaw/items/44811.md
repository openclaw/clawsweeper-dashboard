## Summary

Keep open. Current main still has the stale runtime-source projection path and no merged superseding fix, so the reported config-loss bug remains valid; the PR is not merge-ready because it lacks real behavior proof and still has writer-logic findings.

## What This Changes

The PR changes runtime config writeback to use a freshly read disk snapshot as the projection base and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one current existing/readable disk snapshot for both projection and persistence, preserves readable disk content across validation failures, and keeps focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows that with runtime/source snapshots active, current main can project a runtime-derived write through stale `runtimeConfigSourceSnapshot`, and the merge-patch contract can delete disk-only keys such as `session.dmScope`; I did not run live gateway proof in this read-only review.

## Solution Assessment

No. The PR targets the right writeback path, but the fix should use readable existing disk snapshots even after validation failures and should pass the same snapshot into the inner writer instead of rereading disk.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: This only switches projection bases when the latest snapshot is both valid and existing. If the file exists and parses but fails validation, it still falls back to the stale runtime source snapshot, so disk-only keys can be dropped by the writeback this PR is meant to fix.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads the disk snapshot to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. A file change between those reads can make the projection base and persistence base diverge, leaving an avoidable key-loss window.
  - confidence: 0.74
