## Label Justifications

- `P1`: The item concerns config writeback that can silently drop session/auth-provider configuration during real gateway/runtime workflows.
- `merge-risk: 🚨 compatibility`: The PR changes core config writeback semantics and can affect existing persisted config upgrade/runtime behavior.
- `merge-risk: 🚨 session-state`: The motivating failure drops `session.dmScope`, which controls session isolation behavior.
- `merge-risk: 🚨 auth-provider`: The same runtime/source projection path preserves SecretRef-style provider auth values, so writer mistakes can affect auth/provider configuration persistence.

## Summary

Keep open. This is a real config-persistence bug fix, but the current PR still has writer-logic gaps and no after-fix real behavior proof, so it is not ready for merge or cleanup-close.

## What This Changes

The PR changes runtime config writeback to project runtime-derived writes onto a freshly read disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one current readable existing disk snapshot for projection and persistence, preserves disk content across validation failures, and keeps focused regression coverage for the stale runtime snapshot case.

## Reproduction Assessment

Yes. Source inspection shows current main projects runtime-derived writes through stale `runtimeConfigSourceSnapshot` and then applies `null` merge-patch deletes, which can remove disk-only fields such as `session.dmScope`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No as currently patched. The PR targets the right writer path, but the maintainable fix should use a readable existing disk snapshot even when validation fails and should reuse that snapshot for the inner persistence write.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: This only switches projection bases when the latest snapshot is both valid and existing. If the file exists and parses but fails validation, it still falls back to the stale runtime source snapshot, so disk-only keys can be dropped by the writeback this PR is meant to fix.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads the disk snapshot to build `nextCfg`, but `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. A file change between those reads can make the projection base and persistence base diverge, leaving an avoidable key-loss window.
  - confidence: 0.74
