## Summary

Keep open. The bug is still source-reproducible on current main, but this PR is not merge-ready because it still has a readable-but-invalid config gap, an avoidable double-read window, merge conflicts, and only test/CI proof rather than after-fix real behavior proof.

## What This Changes

The PR changes config writeback to project runtime-derived writes onto a freshly read disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased writer-level fix that uses one current existing/readable disk snapshot as both the projection and persistence base, preserves readable disk content across validation failures, and keeps focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows that with runtime/source snapshots active, a disk-only key such as `session.dmScope` added after the source snapshot can be deleted because current main applies a merge patch to the stale `runtimeConfigSourceSnapshot`.

## Solution Assessment

No. The PR targets the right writer path, but the fix should reuse one readable disk snapshot for projection and persistence and should not fall back to stale runtime-source data solely because validation failed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection bases when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.88
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but the inner writer rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, the projection base and persistence base can diverge, leaving this writeback path with another avoidable key-loss window.
  - confidence: 0.74
