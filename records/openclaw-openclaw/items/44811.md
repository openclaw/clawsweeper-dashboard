## Summary

Keep open: the PR targets a real config writeback bug and adds useful regression coverage, but the current diff still has a data-loss edge for readable validation-invalid configs, a second-read TOCTOU window, and no required changelog entry.

## What This Changes

The PR changes runtime-derived config writes to project onto a fresh existing disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a focused config-writer fix that chooses one current existing disk snapshot for the whole writeback, preserves readable disk payloads when validation fails after parsing, falls back only for missing or genuinely unreadable files, keeps stale-snapshot regression coverage, adds edge-case coverage, and includes a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create a runtime snapshot from older source config, add `session.dmScope` directly on disk afterward, then call `writeConfigFile(loadConfig())`; source inspection shows current main can project the write onto the stale source snapshot.

## Solution Assessment

No. The PR is in the right narrow area, but it should use one disk snapshot consistently, preserve readable validation-invalid snapshots, and add the required changelog entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches to the latest disk payload when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.87
- **[P3] Reuse the projection snapshot for the inner write:** `src/config/io.ts:1518`
  - body: The outer read computes `nextCfg` from one disk snapshot, but the inner writer reads the config file again because no `baseSnapshot` is passed. A key added between those reads can be absent from `nextCfg` and then treated as a deletion against the second snapshot.
  - confidence: 0.74
- **[P3] Add the required changelog entry:** `src/config/io.ts:1517`
  - body: This fixes a user-facing config persistence bug affecting `session.dmScope`, but the PR only changes code and tests. OpenClaw policy requires a single-line `CHANGELOG.md` entry for user-facing fixes, credited to the contributor.
  - confidence: 0.9
