## Summary

Keep open. The PR addresses a real config writeback bug that current main still has in the exported writer path, but the proposed fix is incomplete and the external PR still lacks after-fix real behavior proof.

## What This Changes

This PR changes exported runtime-derived config writes to project onto the latest valid existing disk snapshot and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a rebased config-writer fix that uses one current existing readable disk snapshot for both projection and persistence, preserves validation-invalid parsed payloads, falls back only for missing or genuinely unreadable files, and keeps targeted regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set a runtime snapshot from older source config, add `session.dmScope` directly on disk, then call `writeConfigFile(loadConfig())`; current main still projects through the stale runtime source snapshot, though I did not execute the test in this read-only review.

## Solution Assessment

No. The PR targets the right writer path, but the `valid && exists` guard and second disk read leave correctness gaps; the safer solution is to choose one readable disk snapshot and thread it through the write.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use readable disk snapshots after validation failures:** `src/config/io.ts:1519-1523`
  - body: The new branch only switches projection base when `latestSnapshot.valid && latestSnapshot.exists`. If the file exists, parses, and then fails config validation, the snapshot still carries readable disk source content, but this code falls back to the stale runtime source snapshot and can still delete disk-only keys such as `session.dmScope`.
  - confidence: 0.87
- **[P3] Pass the projection snapshot into the inner writer:** `src/config/io.ts:1518`
  - body: The wrapper reads disk to build `nextCfg`, but then `io.writeConfigFile` rereads the file because no `baseSnapshot` is passed. If the config changes between those reads, `nextCfg` can omit a key present in the inner base and the merge-patch write can treat that as a deletion.
  - confidence: 0.74
