## Summary

Keep open for normal maintainer landing: the shipped `/models` stall remains on current `main`, the PR fixes it at the canonical registry owner boundary, focused tests cover cache partitioning and invalidation, and no blocking correctness or security defect was found.

## What This Changes

The PR adds bounded, clone-isolated caches for plugin setup manifest and resolved registries, integrates lifecycle invalidation, and adds focused cache regression tests.

## Best Possible Solution

Land the bounded owner-level cache after maintainer confirmation that every supported plugin metadata mutation uses the existing lifecycle clear; keep request-specific configuration passed to hooks at execution time rather than adding caller-specific caches.

## Reproduction Assessment

Yes, at source-reproducible confidence. Current `main` repeatedly resolves the setup registry from per-model visibility checks, and the supplied live benchmark exercises those same production calls, though this review did not independently run the long benchmark.

## Solution Assessment

Yes. Eliminating recomputation once at the canonical setup-registry owner is cleaner and more comprehensive than adding another hot-path exception, and the key, clone, boundedness, and lifecycle tests address the material correctness risks.
