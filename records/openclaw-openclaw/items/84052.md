## Summary

This should stay open: the fix targets a real source-reproducible media filename bug, but the current patch truncates valid URL basenames that contain escaped separators and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch decodes percent-encoded URL path basenames for remote media fallback filenames, preserves malformed escapes, and adds focused media tests plus a changelog entry.

## Best Possible Solution

Keep the URL-basename decode fix, but preserve decoded escaped separators by replacing or sanitizing them instead of re-baselining, add tests for %2F and %5C, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection and a read-only Node probe: current main keeps My%20Report.pdf encoded because URL.pathname is passed directly to basenameFromAnyPath. I did not run the Vitest lane because this review was read-only.

## Solution Assessment

No, not as written: the decode direction is right, but re-running basenameFromAnyPath after decoding loses valid escaped slash/backslash content. The safer fix is to decode the isolated basename and replace decoded separators with safe filename characters, with regression tests for both ordinary spaces and encoded separators.
