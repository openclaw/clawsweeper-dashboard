## What This Changes

The branch changes `sessions_send` timeout handling and A2A provenance, gateway agent session metadata persistence, outbound mirror session routing, focused tests, and adds unrelated pending-review template docs.

## Best Possible Solution

Rework the PR around the current main contracts: narrow the timeout floor to actual cross-agent cases, let channel resolvers own outbound session selection, coordinate broader timeout policy with the canonical tracker, and split the unrelated template docs into separate work.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a medium-confidence reproduction path for the claimed timeout/routing gaps and high-confidence proof for the introduced review findings; no live long-running multi-agent setup was run.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is plausible, but the patch over-applies the 45s timeout floor, overrides channel-resolved outbound routes, bundles unrelated docs, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply the timeout floor only to real cross-agent sends:** `src/agents/tools/sessions-send-tool.ts:66-67`
  - body: `resolveAgentIdFromSessionKey` falls back to `main`, and this call path normally has both session keys, so `requesterAgentId && targetAgentId` makes every non-zero `sessions_send` wait at least 45s. Same-agent or legacy-key sends with an explicit short timeout will now hang for 45s instead of failing quickly; gate this on an actual requester/target agent mismatch or agent-key check.
  - confidence: 0.94
- **[P1] Preserve the resolved outbound route for different targets:** `src/infra/outbound/message-action-threading.ts:88-95`
  - body: When `currentSessionKey` is present, this rewrites the resolver's channel-specific session key to the current run session before persistence and before setting `__sessionKey`. A main/webchat session sending to a QQ, Discord, or similar room would persist that external route metadata onto the current session and route follow-up policy through the wrong session context; leave target matching to the resolver's existing current-session recovery logic.
  - confidence: 0.91
