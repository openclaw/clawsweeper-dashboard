## Summary

Keep open. The PR is a plausible bug-fix direction, but it cannot merge as-is because two changed branches introduce timeout and routing regressions, the branch is conflicting, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds a 45s `sessions_send` timeout floor, removes some inter-session provenance fields, sanitizes webchat-backed main-session route metadata, rewrites outbound mirror routes to `currentSessionKey`, adds regression tests, and adds unrelated pending-review template docs.

## Best Possible Solution

Rebase and split the branch, then preserve explicit short waits unless the send is truly cross-agent, keep resolved outbound route keys unless the target is proven to be the current conversation, and handle the pending-review docs as a separate docs change.

## Reproduction Assessment

Yes at source level for the review blockers: the timeout resolver predicate is always truthy for normal keys, and the outbound route rewrite is visible directly in the PR branch. I did not run a live multi-agent gateway setup.

## Solution Assessment

No. The direction addresses real A2A/routing pain, but this patch is not the best implementation because it broadens all nonzero sends to at least 45s and overwrites resolved cross-target route keys after the resolver has already chosen them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply the timeout floor only to true cross-agent sends:** `src/agents/tools/sessions-send-tool.ts:70-71`
  - body: `resolveAgentIdFromSessionKey` returns a non-empty default agent ID even for same-agent or legacy session keys, so this condition makes every nonzero `sessions_send` wait at least 45s. Callers that intentionally use a short timeout for same-agent/session-local sends will now block much longer than requested; compare normalized requester and target agents and only floor the real cross-agent path.
  - confidence: 0.94
- **[P1] Preserve resolved routes for cross-target sends:** `src/infra/outbound/message-action-threading.ts:89-95`
  - body: When `currentSessionKey` is present, this rewrites the route chosen for the explicit outbound target before persisting it. A send from a main/webchat session to a QQ, Discord, or similar external room can therefore store that external route metadata on the current session key and make later routing/policy decisions follow the wrong context; only reuse `currentSessionKey` when the resolved target is the same conversation.
  - confidence: 0.92
