## Summary

Keep open. Current main still lacks the PR's central A2A timeout, webchat main-session metadata cleanup, and outbound continuity changes, but the submitted patch is not merge-ready because the timeout floor is over-scoped and the outbound rewrite can persist cross-channel route metadata onto the active session key.

## What This Changes

The PR adds a 45 second A2A `sessions_send` timeout floor, removes some inter-session provenance propagation, clears stale QQ/group metadata from webchat-backed main sessions, rewrites outbound mirror routes to `currentSessionKey`, adds focused regression tests, and also adds unrelated pending-review policy text to docs templates.

## Best Possible Solution

Keep the runtime bug-fix thread open, but revise it into a narrow gateway/A2A change: scope timeout flooring to true cross-agent sends, sanitize webchat-backed main session metadata with explicit regression coverage, preserve outbound session continuity only when the resolved route is known to match the active session, and split the docs-template policy into a separate docs PR.

## Reproduction Assessment

Yes for the PR review blockers: source inspection gives a high-confidence reproduction path. A same-agent `sessions_send` with a short non-zero timeout now satisfies both truthy agent-id checks, and the PR's own outbound test demonstrates a main `currentSessionKey` plus QQ target being rewritten before metadata persistence.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best implementation because it over-applies the timeout floor, unconditionally rewrites outbound route identity, and mixes unrelated docs-template policy into the runtime fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope the timeout floor to real cross-agent sends:** `src/agents/tools/sessions-send-tool.ts:70-71`
  - body: `resolveAgentIdFromSessionKey()` always returns an id, falling back to `main`, so this condition is true for same-agent and legacy/local session keys too. A caller that explicitly sets `timeoutSeconds: 5` for a local send will now wait up to 45 seconds, which regresses fast-fail behavior outside the A2A case this PR is trying to protect.
  - confidence: 0.93
- **[P1] Keep resolved outbound keys for cross-target sends:** `src/infra/outbound/message-action-threading.ts:89-95`
  - body: This rewrites every resolved route to `currentSessionKey` before calling `ensureOutboundSessionEntry`. When the current main/webchat session sends to a QQ, Discord, or other external target, the external route's `from`/`to`/`chatType` metadata will be persisted onto the current session instead of the channel-specific route key, so later routing and policy can follow the wrong channel context.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The outbound mirror rewrite can persist external channel metadata onto the active session key, which is a routing and privacy-boundary concern.

Concerns:

- **[medium] Avoid cross-channel metadata persistence:** `src/infra/outbound/message-action-threading.ts:89`
  - body: The PR rewrites the resolved route key to `currentSessionKey` before persistence; `ensureOutboundSessionEntry` then records external `From`, `To`, `ChatType`, and `Provider` data under that key. For a main/webchat run sending to an external room, this can pollute the active session's later routing context.
  - confidence: 0.87
