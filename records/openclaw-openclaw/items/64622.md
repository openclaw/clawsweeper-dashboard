## Summary

Keep this PR open, but it is not ready to merge: the current branch still has two blocking routing/timeout correctness issues, a routing/privacy-sensitive persistence concern, and no after-fix real A2A/gateway behavior proof from the contributor.

## What This Changes

The branch changes `sessions_send` timeout/provenance handling, gateway main-session metadata persistence, outbound mirror session-key handling, focused regression tests, and two docs templates.

## Best Possible Solution

Rework this into smaller runtime fixes: preserve resolved outbound route identity, handle timeout semantics through the canonical timeout trackers, keep only proven webchat metadata cleanup if still needed, and split unrelated docs-template policy into separate docs work.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the timeout floor applies beyond true cross-agent sends and the outbound route identity is rewritten before persistence. No high-confidence live reproduction is present for the original multi-agent runtime symptoms.

## Solution Assessment

No. The direction is plausible, but this patch is not the best fix because it broadens timeout semantics, rewrites cross-target route identity before persistence, mixes unrelated docs policy, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope the timeout floor to true cross-agent sends:** `src/agents/tools/sessions-send-tool.ts:70-71`
  - body: `resolveAgentIdFromSessionKey()` always returns a non-empty ID by falling back to the default agent, so this guard applies the 45s floor to same-agent and local sends too. A caller that requested a short non-zero timeout would now wait up to 45 seconds outside the A2A case this PR is trying to protect.
  - confidence: 0.93
- **[P1] Preserve resolved route keys for cross-target sends:** `src/infra/outbound/message-action-threading.ts:89-95`
  - body: When `currentSessionKey` is present, this branch rewrites the resolved external route before `ensureOutboundSessionEntry`. Cross-target sends can then persist external `from`/`to`/`chatType` metadata under the current main/webchat session, corrupting later routing and policy decisions.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a routing/privacy-sensitive persistence concern by writing external-route metadata under the active session key.

- **[medium] Avoid cross-channel metadata persistence:** `src/infra/outbound/message-action-threading.ts:89`
