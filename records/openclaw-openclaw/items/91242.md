## Label Justifications

- `P2`: This is a concrete Slack channel improvement with security-boundary implications, but it is not an urgent regression or core outage.
- `impact:security`: The issue is about changing Slack ingress verification from OpenClaw-owned Slack signature/app-token checks to an explicitly trusted upstream boundary.

## Summary

Keep open: current main and the latest release still only support Slack Socket Mode and HTTP Request URLs, while the requested trusted-upstream mode is a new security-sensitive Slack plugin transport that needs maintainer product/security decisions.

## What This Changes

Add an opt-in `channels.slack.mode: "trusted-upstream"` that accepts sidecar-verified Slack Socket Mode-style envelopes over HTTP without requiring OpenClaw to hold Slack app/signing credentials.

## Best Possible Solution

Design this inside the Slack plugin as an explicit opt-in trusted ingress mode with mandatory upstream assertion, source/loopback guardrails, clear token-conflict behavior, docs/audit/doctor coverage, and tests for the accept and reject paths.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection show the current supported Slack ingress modes are only Socket Mode and HTTP Request URLs.

## Solution Assessment

Unclear. A Slack-plugin mode is plausible, but maintainers still need to decide the mode name, required header/value contract, source and loopback constraints, stale-event policy, audit/doctor surface, and Slack SDK dispatch contract.
