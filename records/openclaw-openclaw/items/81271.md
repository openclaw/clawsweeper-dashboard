## Summary

Keep open: current main and the latest release still require explicit exec host/node selection, while sender/device-derived default node routing remains a new command-execution policy with security and configuration implications.

## What This Changes

The issue requests per-sender exec routing so a TUI/client on the same device as a connected node can default shell execution to that node in multi-node setups.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current source and docs define explicit node selection, and this issue asks for a new sender-derived routing policy. Source inspection does confirm the requested behavior is absent on current main.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox interaction, and fallback behavior before code changes.
