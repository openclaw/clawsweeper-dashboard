## Summary

Keep open: current main still lacks sender/device-derived exec node routing; explicit node selection remains the documented contract, and automatic per-sender defaults need product and security-policy design before implementation.

## What This Changes

Add an opt-in per-sender/per-device exec routing mode so a TUI on a node host can default exec to that same node in multi-node Gateway setups.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy in the exec/node routing layer that reuses Gateway device identity and node presence while preserving current explicit host/node defaults.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a requested new routing policy, not broken existing behavior. Current source and docs provide a high-confidence behavior check that `host=auto` resolves to sandbox or Gateway and node routing requires explicit node selection.

## Solution Assessment

Unclear until maintainers choose the policy. Same-device matching is plausible, but the maintainable path should be opt-in and define identity trust, multiple-node resolution, sandbox interaction, and fallback behavior first.
