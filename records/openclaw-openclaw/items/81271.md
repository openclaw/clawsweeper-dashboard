## Summary

Keep open: current main still supports explicit exec node routing only, while sender/device-derived default routing remains a new exec-routing policy with config and security implications.

## What This Changes

Request default exec routing to the node whose `device.id` matches the TUI/client sender in multi-node Gateway setups.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current docs and source define `host=auto` as sandbox-or-Gateway routing and require explicit `host=node` plus node selection. The issue asks for new behavior rather than reporting a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox interactions, and fallback behavior before code changes.
