## Summary

Keep open: current main still only supports explicit exec host/node selection, while sender-device-derived default routing is a new exec-routing policy with security and config-contract implications.

## What This Changes

The issue requests opt-in per-sender exec routing so a TUI connected from the same device as a node can default shell execution to that node.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current docs and source define `host=auto` as sandbox-or-Gateway routing and require explicit `host=node` plus node selection. Source inspection does confirm the requested capability is not present on current main.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox interactions, and fallback behavior before code changes.
