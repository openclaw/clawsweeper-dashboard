## Summary

Keep open: current main and v2026.6.10 still require explicit node selection for multi-node exec routing, while sender/device-derived defaults remain an unimplemented security-sensitive feature decision.

## What This Changes

The issue requests per-sender exec routing so a TUI sharing a device identity with a node can default shell execution to that same node.

## Best Possible Solution

If maintainers accept the direction, add an opt-in sender-device routing mode that carries authenticated requester device identity into exec default resolution, preserves explicit overrides, and documents/tests fallback behavior.

## Reproduction Assessment

Yes from source, but not from a live multi-node setup in this review. Current exec node resolution requires an explicit node id when multiple nodes are available and does not use sender device identity.

## Solution Assessment

Unclear: per-sender routing is plausible, but maintainers first need to choose the opt-in config/API, override precedence, and security contract for automatic exec target selection.
