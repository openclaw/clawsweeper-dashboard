## Summary

Keep open: current main still defines exec node routing as an explicit config/session/per-call choice, while this issue asks for a new sender-device-derived default. That changes exec target policy and command-execution trust boundaries, so it needs maintainer product and security-policy review rather than conservative cleanup closure.

## What This Changes

Request automatic exec routing to the node sharing the sender/TUI device identity, with fallback to existing gateway or configured defaults.

## Best Possible Solution

Design an explicit opt-in sender-device exec routing policy that reuses Gateway device identity and node presence while preserving today's explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current source and docs define `host=auto` as sandbox-or-Gateway routing and require explicit node selection for node exec. The requested behavior is a new routing mode, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy. Same-device matching is plausible, but the maintainable solution should be opt-in and define identity trust, multi-node resolution, sandbox interactions, and fallback behavior before implementation.
