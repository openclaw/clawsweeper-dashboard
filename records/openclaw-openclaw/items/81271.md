## Summary

Keep open: current main still has only explicit exec host/node routing, while the requested sender/device-derived default is a new exec-routing policy and config surface.

## What This Changes

Request automatic per-sender exec routing so a TUI sharing a device id with a connected node defaults exec to that node, with fallback to current behavior.

## Best Possible Solution

Design an opt-in sender-device routing policy that reuses authenticated Gateway device identity and node presence while preserving the current explicit host/node model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current docs and source define `host=auto` as sandbox-or-gateway routing and require explicit `host=node` plus node selection. The issue asks for a new routing mode rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but it should be opt-in and define trust, multiple-node resolution, sandbox interactions, and fallback behavior before code changes.
