## Summary

Keep open: current main still implements and documents explicit node routing only, while the requested sender/device-derived default routing is a new exec-routing capability with security and product-policy implications.

## What This Changes

Request opt-in per-sender exec routing so a TUI connected from the same device as a node runs shell commands on that device's node by default in multi-node Gateway setups.

## Best Possible Solution

Design an explicit opt-in sender-device exec routing policy in the core Gateway/exec selection path, document it beside the existing `host`/`node` rules, and preserve explicit node binding as the safe default behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: the reported Gateway execution matches the documented current `host=auto` and explicit-node routing contract. Source and docs provide high-confidence evidence that the requested per-sender default is absent rather than broken.

## Solution Assessment

Unclear until maintainers choose the policy: same-device fallback is plausible, but the best fix needs explicit opt-in config, sender identity propagation, security review, and deterministic fallback rules.
