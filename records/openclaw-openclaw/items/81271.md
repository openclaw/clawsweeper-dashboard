## Summary

Keep open: current main documents and implements explicit node routing only, while the requested sender/device-derived default routing is a new exec-routing capability with product and security implications.

## What This Changes

The issue requests per-sender exec routing so TUI clients in multi-node setups can default shell execution to a node on the same device as the sender.

## Best Possible Solution

Design an explicit, opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the existing explicit `host`/`node` safety model and documenting fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: the reported behavior matches the current documented contract where `host=auto` resolves to gateway without a sandbox and node execution requires explicit `host=node` plus node selection. Source and docs confirm the requested behavior is absent rather than broken.

## Solution Assessment

Unclear until maintainers choose the policy: the proposed same-device fallback is plausible, but it needs explicit rules for opt-in config, sender device identity propagation into exec target selection, and fallback when multiple roles or nodes match.
