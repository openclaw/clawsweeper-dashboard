## Summary

Keep open: current main still supports explicit exec node selection only, while sender/device-derived default routing would be a new security-sensitive exec policy rather than a cleanup-close candidate.

## What This Changes

The issue requests opt-in routing that makes exec calls from a TUI/operator sender run on a node associated with the same device in multi-node Gateway setups.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current source and docs define explicit exec routing, and the issue asks for a new routing mode rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy; same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox interactions, and fallback behavior before code changes.
