## Summary

Keep open: current main and the latest release still only support explicit exec node routing, while sender/device-derived routing is a new exec policy with command-execution trust and config-contract implications.

## What This Changes

Request automatic exec routing to the node whose device identity matches the TUI/client sender in multi-node Gateway deployments.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current docs and source define `host=auto` as sandbox-or-Gateway routing and require explicit `host=node` plus node selection. The request is for new routing behavior rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox interactions, and fallback behavior before code changes.
