## Summary

Keep open: current main still documents and implements explicit node routing only, while sender/device-derived default exec routing is a new capability with product and command-execution trust-boundary implications.

## What This Changes

Request to add per-sender/per-device exec routing so a TUI connected from the same device as a node defaults shell commands to that node.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that carries authenticated operator device identity into exec target resolution, selects one matching connected node safely, and documents fallback plus approval behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: current source and docs show `host=auto` resolves to sandbox or gateway and node execution requires explicit `host=node` plus node selection. This is a missing feature, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation needs explicit opt-in config, authorization boundaries, ambiguous-match handling, and sandbox/elevated interaction rules.
