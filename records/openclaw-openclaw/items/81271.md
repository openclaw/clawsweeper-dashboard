## Summary

Keep open: current main and v2026.6.8 still require explicit exec node selection through config, session/per-call overrides, or agent config; sender/device-derived exec routing remains a new security-sensitive product decision.

## What This Changes

The issue requests per-sender exec node routing so a TUI on the same device as a node can default shell execution to that node in multi-node setups.

## Best Possible Solution

If maintainers accept the direction, add opt-in sender-device routing that carries authenticated requester device identity into exec default resolution, matches only same-device node roles, preserves explicit overrides, and documents/tests fallback behavior.

## Reproduction Assessment

Yes from source, but not from a live multi-node setup in this review: current exec node resolution requires an explicit node id when multiple nodes are available and does not use sender device identity.

## Solution Assessment

Unclear: per-sender routing is a plausible implementation shape, but maintainers first need to decide the opt-in config/API, override precedence, and security contract for automatic exec target selection.
