## Summary

Keep open: current main and the latest release still resolve exec targets from explicit session, agent, global, or per-call settings only; sender/device-derived default node routing remains a new security-sensitive exec policy decision.

## What This Changes

The issue requests per-sender exec node routing so a TUI on the same device as a node can default shell execution to that node in multi-node Gateway setups.

## Best Possible Solution

Design an explicit opt-in routing policy that reuses Gateway device identity and node presence while preserving current default exec semantics and documenting fresh-install plus upgrade behavior.

## Reproduction Assessment

Not applicable as a broken existing contract, but source inspection gives a high-confidence path for the reported gap: current main lacks sender-device-derived exec routing and requires explicit host/node selection.

## Solution Assessment

Unclear; same-device matching is plausible, but the best fix needs maintainer policy on opt-in defaults, sender identity trust, multiple-node resolution, sandbox interaction, and fallback behavior before code changes.
