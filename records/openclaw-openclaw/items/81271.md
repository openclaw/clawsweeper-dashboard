## Summary

Keep open: current main still only supports explicit exec node selection, and sender/device-derived exec routing remains a new product and security policy for command execution rather than an implemented cleanup target.

## What This Changes

The issue requests opt-in per-sender exec routing so a TUI connected from the same device as a node can default exec calls to that node in multi-node setups.

## Best Possible Solution

Design an explicit opt-in sender-device routing policy that reuses Gateway device identity and node presence while preserving the current explicit host/node safety model by default.

## Reproduction Assessment

Not applicable as a bug reproduction: current docs and source define `host=auto` as sandbox-or-Gateway routing and require explicit node selection for multi-node node exec. The requested behavior is a new routing mode, not a broken current contract.

## Solution Assessment

Unclear until maintainers choose the policy: same-device matching is plausible, but the implementation should be opt-in and define identity trust, multiple-node resolution, sandbox/elevated interactions, and fallback behavior first.
