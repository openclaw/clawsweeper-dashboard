## Summary

Keep open: current main and v2026.5.7 still catalog Kimi Coding as `kimi-code`, while Kimi's current third-party setup docs use `kimi-for-coding`; the TLS and cached-endpoint claims remain unproven and should not drive a broader transport/config rewrite yet.

## What This Changes

The issue reports that Kimi Code usage in v2026.5.7 fails due to Node TLS errors, a cached local proxy base URL, config-file ambiguity, and Kimi provider/model behavior.

## Best Possible Solution

Make `kimi/kimi-for-coding` the bundled Kimi Code default while preserving legacy refs, update docs/tests, and handle TLS or config-repair behavior only after live evidence proves those separate failures.

## Reproduction Assessment

No, not for the full issue. Source inspection gives a credible path for the model-id drift because OpenClaw sends `model.id`, but the TLS and cached-endpoint failures need a Kimi API key plus the reporter's config state to reproduce with high confidence.

## Solution Assessment

No for the curl-proxy workaround as a core fix. The narrower maintainable repair is to align the bundled Kimi catalog/docs/tests with Kimi's current model id while preserving explicit baseUrl/header overrides and legacy aliases.
