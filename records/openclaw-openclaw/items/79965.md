## Summary

Keep open for maintainer triage, but this is not a safe automated fix: current main already has a native Kimi provider with the expected endpoint, honors explicit baseUrl overrides, and a direct Node 22.22.x probe did not reproduce the reported TLS or empty-JSON failure.

## What This Changes

The issue reports Kimi Code failures in v2026.5.7 involving Node TLS to api.kimi.com, a stale local-proxy baseUrl in config, config-file naming confusion, and native-provider override behavior.

## Best Possible Solution

If a real Node/Undici/Kimi TLS failure is reproduced, fix the provider transport or document a Node/runtime workaround; otherwise keep the built-in kimi/kimi-code path and guide users to remove stale baseUrl overrides with the config CLI.

## Reproduction Assessment

No. A direct Node v22.22.2 request to the reported Kimi endpoint returned normal 401 JSON on three attempts, while the full authenticated OpenClaw model turn and the reporter's exact local config state were not available.

## Solution Assessment

Unclear. The curl proxy workaround is useful local evidence, but baking a proxy or automatic baseUrl repair into core is not the narrowest maintainable solution without a reproduced transport or config-write bug.
