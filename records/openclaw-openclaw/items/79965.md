## Summary

Keep open: current main still defaults the bundled Kimi Code model to `kimi-code`, while Kimi's official docs say third-party tools should send `kimi-for-coding`; the TLS symptom did not reproduce in unauthenticated Node probes, and the config-path/baseUrl concerns are partly already covered by current code and docs. ([kimi.com](https://www.kimi.com/code/docs/en/?utm_source=openai))

## What This Changes

The issue reports Kimi Code usage in v2026.5.7 failing due to TLS errors, a stale local `baseUrl`, config-file confusion, and a requested `kimi/kimi-for-coding` setup.

## Best Possible Solution

Align the bundled Kimi Code catalog, onboarding defaults, docs, and tests to `kimi/kimi-for-coding` while preserving legacy refs and explicit overrides; handle TLS or stale local config only with live evidence or a narrower canonical follow-up.

## Reproduction Assessment

No for a full live reproduction of the issue: the TLS failure did not reproduce without a Kimi key. Yes at source level for the model-id drift, because current main sends the bundled `model.id` and still defaults Kimi Code to `kimi-code`.

## Solution Assessment

Yes for a narrow catalog/docs/test repair that changes the default model id while preserving legacy aliases and explicit overrides. No for adopting the curl-proxy workaround in core; transport or cache changes need live provider evidence or separate issue ownership.
