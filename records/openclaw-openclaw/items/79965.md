## Summary

Keep open: current main and v2026.5.7 still default Kimi Code to `kimi-code`, while Kimi's current docs require third-party tools to send `kimi-for-coding`; the TLS and stale-local-proxy claims are not live-proven here, and the config/baseUrl concerns are partly already covered by current source/docs.

## What This Changes

The issue reports broken Kimi Code use in OpenClaw 2026.5.7, including TLS failures, a stale local proxy baseUrl, config-file confusion, and native Kimi provider/model behavior.

## Best Possible Solution

Make `kimi/kimi-for-coding` the bundled primary Kimi Code model while preserving legacy refs as compatibility aliases, and leave TLS or cache-marker changes to live-proven or already paired follow-ups.

## Reproduction Assessment

No for the full issue as written: the authenticated TLS failure was not reproduced without a Kimi key, and unauthenticated Node TLS/fetch probes succeeded. Yes at source level for the model-id drift: current catalog/docs/tests still use `kimi-code`, and the transport sends `model.id` to Kimi.

## Solution Assessment

No for the proposed curl-proxy workaround as the repository fix. The narrow maintainable path is to align the bundled Kimi catalog/onboarding/docs/tests with Kimi's stable `kimi-for-coding` model id while preserving explicit baseUrl/header overrides and legacy model refs.
