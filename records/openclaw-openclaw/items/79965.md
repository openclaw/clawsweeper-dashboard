## Summary

Keep open: current main still ships Kimi Code as `kimi-code`, while the official Kimi Code docs say third-party tools must send `kimi-for-coding`; the TLS claim did not reproduce in unauthenticated Node probes, and the config/baseUrl concerns are partly already covered by current code/docs. ## What This Changes

The issue reports that the bundled Kimi Code provider in v2026.5.7 fails due to TLS errors, stale local baseUrl config, config-file ambiguity, and Kimi endpoint/model mismatches.

## Best Possible Solution

Make `kimi/kimi-for-coding` the bundled primary Kimi Code model while preserving legacy refs, update docs/tests, and handle any remaining TLS failure only with live authenticated evidence.

## Reproduction Assessment

No, not as a full live reproduction: there is no Kimi API key available, and unauthenticated Node TLS/fetch probes did not reproduce the TLS failure. Source plus official Kimi docs do give a credible reproduction path for the model-id drift because OpenClaw sends `model.id` and currently defaults it to `kimi-code`.

## Solution Assessment

No for the proposed curl-proxy workaround; the maintainable OpenClaw fix is to align the bundled Kimi catalog/docs/tests with Kimi's stable model id while preserving explicit baseUrl/header overrides and legacy aliases.
