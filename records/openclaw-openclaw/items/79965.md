## Summary

Keep open: current main and v2026.5.7 still default bundled Kimi Code to `kimi-code`, while Kimi's current third-party-tool docs require `kimi-for-coding`; the reported TLS failure was not reproduced by unauthenticated Node probes, and the config/baseUrl parts are partly already covered by source and docs.

## What This Changes

The issue reports that native Kimi Code usage in OpenClaw 2026.5.7 is blocked by TLS errors, stale local baseUrl config, config-file ambiguity, and provider/model contract drift.

## Best Possible Solution

Align the bundled Kimi Code catalog, onboarding, docs, and tests on `kimi/kimi-for-coding` while preserving legacy `kimi-code` and `k2p5` refs; handle TLS or user-local stale baseUrl only with separate live evidence.

## Reproduction Assessment

No, not for the full issue: there is no authenticated Kimi API key and unauthenticated Node probes did not reproduce the TLS failure. The model-id drift is source-reproducible because OpenClaw sends `model.id` and current Kimi docs require `kimi-for-coding`.

## Solution Assessment

No for the proposed curl-proxy workaround as the primary fix; the narrow maintainable path is to fix the bundled Kimi model catalog and docs first, then only address TLS/config repair with live authenticated evidence.
