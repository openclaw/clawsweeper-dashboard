## Summary

Keep open: current main and v2026.5.7 still ship Kimi Code as `kimi/kimi-code`, while Kimi's current third-party-tool docs require request `model: "kimi-for-coding"`; the TLS symptom did not reproduce in an unauthenticated Node probe, and the config/baseUrl parts are partly covered by current code/docs.

## What This Changes

The issue asks OpenClaw to make native Kimi Code work without a curl proxy by addressing TLS failures, stale local baseUrl config, config-file confusion, and correct Kimi endpoint/model defaults.

## Best Possible Solution

Align the bundled Kimi provider and docs with `kimi/kimi-for-coding` while keeping legacy `kimi/kimi-code` and `kimi/k2p5` compatibility, then require live authenticated evidence before changing TLS or cache-control behavior.

## Reproduction Assessment

No, not for the full reported issue: there is no Kimi API key available and the unauthenticated TLS probe succeeded. There is a source-backed reproduction path for the model-id drift because current main sends `model.id` and the bundled Kimi default remains `kimi-code` while Kimi documents `kimi-for-coding`.

## Solution Assessment

No for the curl-proxy workaround as a repository fix. The narrow maintainable path is to update the Kimi catalog/onboarding/docs/tests for the stable model id while preserving explicit provider overrides and legacy aliases; TLS/cache changes need live proof first.
