## Summary

Keep open: current `main` and the latest release protect the supported `update.run` path better, but the exec preflight guard still does not block raw global OpenClaw package-manager upgrades. This issue also has a protected `maintainer` label, a MEMBER author, and an open closing PR, so it needs maintainer review rather than cleanup closure.

## What This Changes

The issue requests hardening so agent-initiated self-updates use `update.run` / `openclaw update` or otherwise guarantee a gateway restart instead of raw global package-manager upgrades.

## Best Possible Solution

Review and either refresh/land https://github.com/openclaw/openclaw/pull/78696 or implement the same exec-preflight guard in `src/infra/exec-control-command-guard.ts`, while preserving `update.run` / `openclaw update` as the restart-owning self-update path.

## Reproduction Assessment

Yes, by source and report evidence: current `main` lets exec launch shell commands after only the approval/channel-login guard, while the issue logs show raw `npm install -g openclaw@latest` followed by stale chunk import failures. I did not run a live service-managed host upgrade in this read-only review.

## Solution Assessment

Yes, the best narrow fix is to enforce the boundary at exec command preflight while preserving `update.run` / `openclaw update` as the restart-owning path. The open linked PR targets that behavior, but it needs refresh against current `main`.
