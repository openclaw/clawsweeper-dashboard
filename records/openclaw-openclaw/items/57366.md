## Summary

Keep open: the PR addresses the earlier parser edge cases, but the fix is still incomplete because it only changes the rawBody placeholder while the separate media resolver can still attach allowed Teams CDN emoji sprites. It also lacks after-fix real Teams behavior proof.

## What This Changes

The PR adds Teams CDN emoji alt-text extraction to the MS Teams attachment placeholder path, plus parser tests and a changelog entry.

## Best Possible Solution

Share a pure Teams emoji classification helper between placeholder generation and inbound media resolution so pure emoji becomes Unicode text and produces no media payload, while mixed or real attachments keep the existing fallback behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds rawBody from the placeholder helper and separately sends the same attachments to `resolveMSTeamsInboundMedia()`, whose downloader re-extracts inline HTML image URLs. That makes a pure Teams CDN emoji image still eligible for media attachment even after the PR's placeholder fast path.

## Solution Assessment

No. Returning Unicode in rawBody is the right direction, but the fix also needs to suppress pure Teams emoji CDN images in inbound media resolution so the agent does not still receive sprite media.
