## Summary

Keep open: the branch is useful and now has a changelog entry, but it only fixes the text placeholder path while the same Teams CDN emoji `<img>` can still flow through inbound media download and reach the agent as media. It also still needs after-fix real Teams behavior proof before merge.

## What This Changes

The PR adds Microsoft Teams emoji-CDN extraction in the attachment placeholder helper, focused unit tests, and a changelog entry.

## Best Possible Solution

Share the pure-emoji classification with inbound media resolution so emoji-only Teams attachments produce Unicode text and no media payload, while mixed content keeps the existing image/document fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Teams `text/html` attachment containing only a `statics.teams.cdn.office.net` emoji `<img>` is counted as an inline image on current main, and the message handler uses that placeholder and media payload for the agent context when message text is empty.

## Solution Assessment

No, not yet. Converting the placeholder to Unicode is the right direction, but the fix also needs to skip emoji-only CDN images in the inbound media download path so the agent does not still receive a media attachment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Skip emoji-only attachments in media resolution:** `extensions/msteams/src/attachments/html.ts:179-181`
  - body: This fast-path only changes the placeholder returned for `rawBody`; the same attachments are later passed to `resolveMSTeamsInboundMedia`, which still calls `downloadMSTeamsAttachments` and `extractInlineImageCandidates`. Because the default media allowlist includes `statics.teams.cdn.office.net`, an emoji-only Teams message can still attach the sprite as `MediaPath`, so the agent may still see an image despite the text placeholder being `👋`. Reuse the pure-emoji detection before media download, with a regression test for the final media payload.
  - confidence: 0.88
