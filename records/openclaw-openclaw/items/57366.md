## Summary

This PR should stay open: the direction is useful, but the prepared head still only fixes the Teams emoji placeholder path while leaving emoji CDN sprites eligible for media download, and after-fix real Teams proof is still missing.

## What This Changes

Adds Teams emoji-CDN alt-text extraction to `buildMSTeamsAttachmentPlaceholder`, focused attachment tests, and a changelog entry.

## Best Possible Solution

Share the pure-emoji classification with inbound media resolution so emoji-only Teams attachments become Unicode text with no media payload, while mixed content keeps the existing image/document fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main a Teams `text/html` attachment containing only a `statics.teams.cdn.office.net` emoji image is counted as inline media, then the message handler also resolves media from the same attachments.

## Solution Assessment

No, not yet. Converting the placeholder to Unicode is the right direction, but the fix also needs to skip emoji-only CDN images in inbound media resolution so the agent does not still receive an image payload.

## Review Findings

Overall correctness: patch is incorrect
