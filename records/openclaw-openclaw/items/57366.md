## Summary

This should stay open; the proposed fix is useful but incomplete because it changes only the Teams placeholder text while the same emoji CDN image can still flow through inbound media resolution. The external PR also still lacks after-fix real Teams behavior proof.

## What This Changes

The PR adds Teams CDN emoji extraction in `buildMSTeamsAttachmentPlaceholder`, focused attachment parser tests, and a changelog entry.

## Best Possible Solution

Share the pure-emoji classification with inbound media resolution so pure Teams emoji becomes Unicode text and produces no media payload, while mixed content keeps the existing image/document fallback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path. On current main, a Teams `text/html` attachment containing a `statics.teams.cdn.office.net` emoji `<img>` is counted as an inline image, and the same allowed CDN URL can be downloaded as inbound media.

## Solution Assessment

No, not yet. Converting the placeholder to Unicode is the right direction, but the patch also needs to skip emoji-only CDN images in inbound media resolution so the agent does not still receive an image payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip emoji-only attachments before media download:** `extensions/msteams/src/attachments/html.ts:179-181`
  - body: This fast path only changes the placeholder returned for `rawBody`; the same attachments are later passed to `resolveMSTeamsInboundMedia`, which still calls `downloadMSTeamsAttachments` and `extractInlineImageCandidates`. Because the default media allowlist includes `statics.teams.cdn.office.net`, an emoji-only Teams message can still attach the sprite as `MediaPath`, so the agent may still see an image despite the text placeholder being the emoji. Reuse the pure-emoji detection before media download and add a regression test for the final media payload.
  - confidence: 0.88
