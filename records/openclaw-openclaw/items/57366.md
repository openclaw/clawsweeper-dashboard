## Summary

Keep open: this is a useful focused MS Teams bug fix, but the prepared head still only changes the text placeholder path while the same Teams emoji CDN image can continue through inbound media download, and the external after-fix real behavior proof gate is still missing.

## What This Changes

The PR adds Teams CDN emoji `<img>` alt extraction to the MS Teams attachment placeholder, parser tests, and a changelog entry.

## Best Possible Solution

Share the pure-emoji classification with inbound media resolution so pure Teams emoji becomes Unicode text and produces no media payload, while mixed or real attachments keep the existing fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main counts Teams emoji CDN `<img>` tags as inline images, and the separate media resolver can still download the same allowed CDN URL.

## Solution Assessment

No, not yet. Converting the placeholder to Unicode is the right direction, but the patch also needs to skip emoji-only CDN images in inbound media resolution so the agent does not still receive an image payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip emoji-only attachments before media download:** `extensions/msteams/src/attachments/html.ts:179-181`
  - body: This fast path only changes the placeholder returned for `rawBody`; the same attachments are later passed to `resolveMSTeamsInboundMedia`, which still calls `downloadMSTeamsAttachments` and `extractInlineImageCandidates`. Because the default media allowlist includes `statics.teams.cdn.office.net`, an emoji-only Teams message can still attach the sprite as `MediaPath`, so the agent may still see an image despite the text placeholder being the emoji. Reuse the pure-emoji detection before media download and add a regression test for the final media payload.
  - confidence: 0.88
