## Summary

Keep open. The parser addition is useful and the earlier parser-review issues are addressed, but current main shows this PR only changes the RawBody placeholder path; the same pure Teams emoji HTML still flows into inbound media resolution, so the agent can still receive a downloaded emoji sprite media payload, and external real Teams after-fix proof is still missing.

## What This Changes

The PR adds Teams CDN emoji `img` alt-text extraction for the MS Teams attachment placeholder, plus focused parser tests and a changelog entry.

## Best Possible Solution

Use one pure Teams emoji classification path before both RawBody placeholder generation and inbound media resolution/download, with tests proving pure emoji yields Unicode text and no media payload while mixed or real attachments keep fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a pure Teams `text/html` emoji `img` with empty text becomes placeholder text through the PR, but the unchanged media resolver still extracts and downloads the same allowed CDN `img` source.

## Solution Assessment

No. The placeholder parser direction is maintainable, but the best fix must also suppress pure Teams emoji sprites before inbound media download and prove the live Teams payload no longer contains sprite media.
