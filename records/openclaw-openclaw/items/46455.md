## Summary

Keep this PR open for maintainer handling. The branch has addressed some earlier review concerns, but it still weakens current poll-intent validation and handles empty Discord component skeletons above the parser boundary, leaving native Discord/component-data paths uncovered.

## What This Changes

The PR changes poll intent detection and message send sanitization, adds helpers to strip poll/component parameters, and updates outbound message tests for auto-populated poll and component fields.

## Best Possible Solution

Keep the shipped zero-default poll tolerance, preserve corrective validation for explicit poll-shaped sends unless maintainers intentionally change that policy, and handle empty Discord component skeletons in the Discord parser/runtime so all native and channelData consumers are covered.

## Reproduction Assessment

Yes. The poll regression is source-reproducible from current helper/send-validation tests, and the Discord failure is source-reproducible by passing an empty `components.modal.fields` payload through `readDiscordComponentSpec()`.

## Solution Assessment

No. The branch contains useful context, but the safer maintainable fix is narrower: do not globally redefine poll intent to only `pollQuestion`, and move empty Discord component tolerance to the parser boundary instead of only the shared `message.send` runner.
