## Summary

This PR can close as superseded: the bug cluster is real, but the later #75652 PR is the better canonical fix path because it covers both semantic poll-default handling and Discord parser-level empty component normalization.

## What This Changes

The PR changes shared message send handling to strip model-filled poll and empty component parameters, updates poll intent detection, and adds/updates tests around send validation.

## Best Possible Solution

Use #75652 as the canonical maintained fix path, keeping shipped zero-default tolerance while moving remaining poll-default and empty Discord component handling to the semantic helper/parser boundaries.

## Reproduction Assessment

Yes. Source inspection shows current main still rejects send when hasPollCreationParams is true, and the Discord direct send path can still pass an empty modal skeleton to a parser that throws; I did not run tests in this read-only review.

## Solution Assessment

No. This PR contains useful work, but the narrower maintained solution is #75652 because it keeps option/question poll intent semantics and normalizes Discord empty components at the parser path instead of only in the shared runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve option-only poll intent:** `src/poll-params.ts:49-52`
  - body: Changing hasPollCreationParams to require pollQuestion means a send request with pollOption but no question is treated as harmless noise and can be stripped/allowed. Current main treats non-empty poll options as poll intent, and the later canonical fix keeps pollQuestion or pollOption as the semantic gate, so this branch would silently degrade malformed poll requests into plain sends.
  - confidence: 0.83
- **[P2] Normalize empty Discord components in the parser path:** `src/infra/outbound/message-action-runner.ts:724`
  - body: Calling stripEmptyComponents only from the shared message runner misses direct Discord sendMessage paths, which read ctx.params.components and call readDiscordComponentSpec directly. Empty modal skeletons can still throw there, so the #43015 failure is not fully fixed unless the Discord parser/runtime boundary treats empty component specs as absent.
  - confidence: 0.86
