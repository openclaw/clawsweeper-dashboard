## Summary

Keep this PR open, but the current patch should not merge as written. The reported leak path is real on current main, yet the PR uses raw text-prefix filtering that can suppress legitimate visible messages, and the sanitizer has since moved to the shared chat-display projection path on main.

## What This Changes

The PR adds chat.history filtering for synthetic compaction markers, pre-compaction memory-flush prompts, and post-compaction context-refresh messages, plus targeted Gateway sanitizer and history tests.

## Best Possible Solution

Port the fix onto the shared chat-display projection path, filter synthetic compaction records by structural metadata, and hide memory/post-compaction maintenance entries only when they carry explicit internal provenance or equivalent trusted metadata. Add regressions that prove those internal artifacts are hidden while user-authored text with the same prefixes remains visible.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a transcript with a type: "compaction" entry or the documented maintenance prompt strings and call chat.history; current main projects the synthetic compaction message and does not filter the memory/post-compaction marker text. The PR-introduced false positive is also reproducible from the diff with a normal user message beginning with either filtered prefix.

## Solution Assessment

No. The PR fixes a real class of visible artifacts, but raw prefix filtering is not the narrowest maintainable solution because it lacks provenance and can hide legitimate visible messages; structural metadata or inputProvenance-based filtering is safer.
