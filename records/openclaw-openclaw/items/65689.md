## Summary

Keep open: this PR addresses a real current-main gap because Swift chat history currently drops the peer `errorMessage` field, but the branch renders that field too broadly for any empty or non-text message. That conflicts with existing stopReason-gated error handling and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `errorMessage` to the Swift chat message model, propagates it through chat UI copy/reconcile paths, renders it as fallback text, and adds a ChatViewModel history-refresh test.

## Best Possible Solution

Preserve `errorMessage` through the Swift model and copy sites, but render it only for assistant turns with normalized `stopReason == "error"` and no visible text, with regression coverage for stale success and tool-only rows.

## Reproduction Assessment

Yes. A chat-history payload with an assistant message containing `content: []`, `stopReason: "error"`, and a non-empty `errorMessage` follows the current decoder/rendering path and remains invisible; a successful or tool-only row with stale `errorMessage` shows the PR regression risk.

## Solution Assessment

No. Preserving the field is the right direction, but rendering must be gated to true assistant error turns so successful, silent, and tool-only messages are not rewritten as provider failures.
