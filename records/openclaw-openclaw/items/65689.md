## Summary

Keep this PR open: the Swift ChatUI `errorMessage` path is still not implemented on current main, but the submitted patch is not mergeable because it fails the Swift build, renders stale error text too broadly, and lacks after-fix real behavior proof.

## What This Changes

The PR adds Swift ChatUI decoding, propagation, fallback rendering, and a history-refresh regression test for provider error turns carried in `errorMessage`.

## Best Possible Solution

Preserve peer `errorMessage` through the Swift model and copy paths, then render it only for assistant turns with normalized `stopReason == "error"` and no visible text, with stale-error/tool-only regression coverage.

## Reproduction Assessment

Yes, at source level. A current-main Swift chat-history payload with `role: "assistant"`, `content: []`, `stopReason: "error"`, and non-empty `errorMessage` decodes to no visible text, and the PR head also has a live CI compile failure at the changed decoder line.

## Solution Assessment

No. The intended fix boundary is appropriate, but the current implementation should avoid reading `self` before initialization and should display peer error text only for true assistant error turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Initialize before reading self in the decoder:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198-199`
  - body: The new fallback closure reads `self.errorMessage` before `self.content` has been initialized in `init(from:)`, and the live macOS Swift job fails with `'self' used before all stored properties are initialized`. Decode into a local string/helper before assigning stored properties.
  - confidence: 0.98
- **[P2] Gate synthetic error content to error turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198-204`
  - body: The decoder creates visible text from `errorMessage` whenever decoded content is empty. Successful silent rows can carry stale/background `errorMessage`, so fallback text should require an assistant error turn, such as normalized `stopReason == "error"`.
  - confidence: 0.93
- **[P2] Restrict error fallback text to failed turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatView.swift:442`
  - body: `primaryText` falls back to `message.errorMessage` whenever no text blocks are found. Tool-call-only or otherwise successful assistant messages can then display stale provider failures as the reply; gate this fallback to assistant error turns.
  - confidence: 0.91
