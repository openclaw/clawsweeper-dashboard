## Summary

Keep open: this addresses a real Swift chat-history bug, but the current PR head fails the macos-swift build, renders peer `errorMessage` too broadly, and still needs real after-fix behavior proof from the external contributor.

## What This Changes

This PR adds `errorMessage` to Swift chat messages, renders fallback provider error text for empty assistant turns, propagates the field through ChatView/ChatViewModel copies, and adds a ChatViewModel regression test.

## Best Possible Solution

Preserve peer `errorMessage` through the Swift model and copy paths, then render it only for assistant turns with normalized `stopReason == "error"` and no visible text, with stale-error/tool-only regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has no Swift `OpenClawChatMessage.errorMessage` decoding and `ChatView.primaryText(in:)` only joins text content, while the backend stream-error contract keeps detailed provider errors in peer `errorMessage`.

## Solution Assessment

No. Preserving and rendering `errorMessage` is the right direction, but this implementation must compile and must limit fallback rendering to true assistant error turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Initialize decoder locals before building fallback content:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198-199`
  - body: The new fallback closure reads `self.errorMessage` before `self.content` has been initialized in `init(from:)`, and the macos-swift job fails with `'self' used before all stored properties are initialized`. Decode `errorMessage` into a local or helper before assigning stored properties.
  - confidence: 0.98
- **[P2] Gate decoder fallback to assistant error turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198-204`
  - body: The decoder creates visible text from `errorMessage` whenever decoded content is empty. Successful or silent rows can carry stale/background `errorMessage`, so synthesize fallback text only for assistant turns whose normalized `stopReason` is `error`.
  - confidence: 0.93
- **[P2] Gate primaryText fallback to true error turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatView.swift:442-445`
  - body: `primaryText` falls back to `message.errorMessage` whenever no text blocks are found. Tool-call-only or otherwise successful assistant messages can then display stale provider failures as the reply; require an assistant error turn before using this fallback.
  - confidence: 0.91
