## Summary

Keep open. The PR addresses a real Swift chat-history gap, but the current patch renders any preserved `errorMessage` as visible assistant text, which conflicts with current backend/session-history contracts that only true assistant error turns should surface peer error detail.

## What This Changes

The PR adds `OpenClawChatMessage.errorMessage` decoding/encoding and copy-site propagation, renders `errorMessage` as fallback chat text when content has no visible text, and adds a ChatViewModel history-refresh regression test.

## Best Possible Solution

Preserve `errorMessage` in the Swift chat message model and through copy/reconcile sites, but synthesize or render fallback text only for assistant messages with normalized `stopReason == "error"` and no visible text. Add regression coverage for the intended empty error-turn case, stale successful turns, and tool-only/non-text messages, plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A chat-history payload containing an assistant message with `content: []`, `stopReason: "error"`, and a non-empty `errorMessage` reproduces the current-main visibility gap; a non-error or tool-only assistant row with `errorMessage` reproduces the PR regression risk.

## Solution Assessment

No. The preservation part is the right direction, but the rendering fallback is too broad; the maintainable solution is to gate display to true assistant error turns and leave successful, silent, and tool-only rows untouched.
