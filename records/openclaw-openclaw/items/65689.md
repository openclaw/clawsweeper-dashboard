## Summary

Keep open: the latest head fixes the earlier Swift initialization and stale-error gating problems, but it still misses current main's sentinel-backed stream-error shape and the external-PR real behavior proof gate is still unmet.

## What This Changes

This PR adds Swift chat-message `errorMessage` decoding/encoding, renders it as fallback assistant text for empty assistant error turns, propagates the field through chat copy paths, adds regression coverage, and updates the changelog.

## Best Possible Solution

Decode and preserve `errorMessage`, then prefer it over empty or `STREAM_ERROR_FALLBACK_TEXT` content only for assistant `stopReason == "error"` turns, with regression coverage for both payload shapes and real macOS chat-history proof.

## Reproduction Assessment

Yes, at source level. Current main decodes `OpenClawChatMessage` without `errorMessage` and reads only text content in Swift, while the backend stores detailed provider failures in peer `errorMessage`.

## Solution Assessment

No. Preserving and gated rendering of `errorMessage` is the right direction, but this branch only falls back when decoded content is empty and therefore misses current stream-error rows that carry the canonical sentinel text plus `errorMessage`.
