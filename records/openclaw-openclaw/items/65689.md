## Summary

Keep open: the fix direction is useful, but the PR currently fails the macOS Swift build, still renders raw `errorMessage` without the backend stop-reason guard, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `errorMessage` to the Swift chat message model, propagates it through Swift chat UI copy/reconcile paths, renders it as fallback assistant text, and adds a history-refresh regression test.

## Best Possible Solution

Preserve `errorMessage` through the Swift model and copy sites, but synthesize visible fallback text only for assistant turns with normalized `stopReason == "error"` and no visible text, with regression coverage for stale success and tool-only rows.

## Reproduction Assessment

Yes, at source level. A chat-history payload with an assistant message containing `content: []`, `stopReason: "error"`, and non-empty `errorMessage` follows the current decoder/rendering path and remains invisible, while stale non-error `errorMessage` rows show the PR regression risk.

## Solution Assessment

No. The proposed direction is narrow and appropriate, but the current patch does not compile and must gate display to true assistant error turns to preserve the backend contract and prior stale-error fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Initialize before reading self in the decoder:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198`
  - body: The new fallback closure reads `self.errorMessage` before `self.content` has been initialized in `init(from:)`, and the live macOS Swift CI job fails with `'self' used before all stored properties are initialized`. Use a local decoded error string/helper instead of capturing `self` before all stored properties are assigned.
  - confidence: 0.98
- **[P2] Gate synthetic error content to error turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatModels.swift:198-204`
  - body: Even after the compile fix, the decoder creates visible text from `errorMessage` whenever decoded content is empty. Core repair and extraction paths only treat peer `errorMessage` as displayable for assistant `stopReason == "error"`; without that guard, a successful silent row with stale error metadata becomes a visible failure.
  - confidence: 0.93
- **[P2] Restrict error fallback text to failed turns:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatView.swift:442`
  - body: `primaryText` falls back to `message.errorMessage` whenever no text blocks are found. Tool-call-only or otherwise successful assistant messages can carry stale background errors, so this can render a provider failure as the assistant reply after a successful turn; gate the fallback to assistant error turns.
  - confidence: 0.91
