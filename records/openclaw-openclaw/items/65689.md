## Summary

This PR should stay open: current main still lacks Swift chat-history `errorMessage` rendering, but the latest PR head still misses the current stream-error sentinel shape and lacks required after-fix real behavior proof.

## What This Changes

The PR adds Swift chat-message `errorMessage` decoding and propagation, fallback rendering for empty assistant error turns, regression tests, and a changelog entry.

## Best Possible Solution

Preserve peer `errorMessage` through the Swift model and render it for assistant error turns even when content contains the stream-error sentinel, with regression coverage for both empty-content and sentinel-content histories plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main has no Swift message-level `errorMessage` decoding, while backend error turns keep detailed provider text in peer `errorMessage` and may use replay-safe sentinel content.

## Solution Assessment

No. The PR is the right direction for empty-content error turns, but it should also treat the current stream-error sentinel as fallback content so the detailed provider error is what appears in chat history.
