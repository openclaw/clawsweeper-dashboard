## Summary

Keep this PR open. Current main still does not preserve `errorMessage` on the Swift chat history message model, so the user-visible gap remains real, but the proposed fallback is too broad and can render stale/background provider errors as successful assistant replies. The existing review comments are supported by current main’s explicit `#13935` guardrails.

## What This Changes

The PR adds `errorMessage` to `OpenClawChatMessage`, passes it through Swift chat copy sites, renders fallback error text when normal message text is empty, and adds a ChatViewModel regression test for history refresh after an error turn.

## Best Possible Solution

Preserve `errorMessage` in the Swift chat message model and pass it through all copy/reconcile/merge sites, but render or synthesize fallback text only for assistant messages with `stopReason == "error"` and no visible text. Add regression coverage for the intended empty error-turn history case plus stale-success and tool-only `errorMessage` cases.

## Reproduction Assessment

Yes. A history payload containing an assistant message with empty `content`, `stopReason: "error"`, and non-empty `errorMessage` reproduces the current-main visibility gap; a non-error or tool-only message with stale `errorMessage` reproduces the PR regression risk.

## Solution Assessment

No. The PR is directionally correct, but the safest maintainable solution is a narrower gated fallback keyed to assistant error turns, with tests proving successful/tool-only messages do not surface stale provider errors.
