## Label Justifications

- `P2`: This is a real credential-rotation UX and auth-provider improvement with limited blast radius, not an emergency outage.
- `impact:security`: The issue concerns API key rotation, stale credentials, and whether OpenClaw should copy or fall back between secret sources.
- `impact:auth-provider`: The affected behavior is provider credential precedence across auth profiles, config env credentials, and SecretRef-backed auth.

## Summary

Keep open: current main and v2026.5.19 still have no bulk auth sync, auto-sync, or warning path for env-backed key rotation across agent auth profiles. The closed explicit-provider-key bug and the newer all-agent active-profile migration issue are adjacent, but neither supersedes this env/preference-shadowing workflow decision.

## What This Changes

The issue requests a CLI command, warning, fallback, or auto-sync behavior for rotating provider API keys when per-agent auth profiles shadow updated config env credentials.

## Best Possible Solution

Define an explicit, non-destructive credential-rotation policy, likely a dry-run-capable sync or stronger shadow-warning flow, while preserving documented profile precedence unless security owners deliberately change it.

## Reproduction Assessment

Yes, at source level. Current main applies config env values, but profile-first runtime auth resolution and tests show stored auth profiles still shadow env credentials unless a caller explicitly opts into env-first resolution.

## Solution Assessment

Unclear until maintainer/security review chooses the policy. The issue lists several viable directions, and copying secrets, warning, explicit sync, or fallback-on-401 each carries different credential-authority tradeoffs.
