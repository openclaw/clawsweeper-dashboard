## Summary

Keep open. Current main still has profile-first auth resolution with env-first only as an opt-in path, `REF_SHADOWED` remains audit-only, and no bulk sync or rotate command is registered; the fix direction needs maintainer/security policy because it changes secret storage and credential precedence.

## What This Changes

Request a user-facing way to sync, rotate, or warn about provider API keys when per-agent auth profiles shadow env/config credentials.

## Best Possible Solution

Define a non-destructive credential-rotation policy, likely a dry-run-capable sync or doctor repair plus clearer shadow warnings, while preserving or explicitly revising documented auth-profile precedence.

## Reproduction Assessment

Yes, at source level. Current source and tests show stored auth profiles win over env credentials by default, `REF_SHADOWED` only warns during audit, and no sync/rotate command is registered.

## Solution Assessment

Unclear. Auto-sync, explicit sync, warning-only behavior, and fallback-on-401 all have different security and precedence tradeoffs, so maintainers should choose the credential authority first.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because API-key rotation affects where secrets are stored and which credential source wins at runtime.

Concerns:

- **[medium] Stale stored credentials can shadow rotated env credentials:** `src/agents/model-auth.ts:688`
  - body: Normal runtime resolution can select a stored auth profile before the updated env/config credential, so users may believe rotation succeeded while OpenClaw continues trying the old stored credential.
  - confidence: 0.9
- **[medium] Bulk sync could expand secret copies without policy:**
  - body: A sync command or auto-sync path could write credential material across multiple agent stores unless maintainers define a narrow ref-based, dry-run-capable, or otherwise non-destructive policy.
  - confidence: 0.82
