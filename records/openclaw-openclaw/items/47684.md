## Label Justifications

- `P2`: This is a real credential-rotation UX and auth-provider improvement with limited blast radius, not an emergency outage.
- `impact:security`: The issue concerns API key rotation, stale credentials, and whether OpenClaw should copy or fall back between secret sources.
- `impact:auth-provider`: The affected behavior is provider credential precedence across auth profiles, config env credentials, and SecretRef-backed auth.

## Summary

Keep open: current main and the latest release still preserve profile-first auth resolution for env-backed provider keys, and there is no bulk auth sync/rotation command; the remaining work is a security-sensitive product choice rather than conservative cleanup.

## What This Changes

Request a bulk credential rotation/sync or warning flow so config env API key updates do not stay shadowed by per-agent auth profiles.

## Best Possible Solution

Define an explicit, non-destructive credential-rotation policy, likely a dry-run-capable sync or stronger shadow warning flow, while preserving documented profile precedence unless security owners deliberately change it.

## Reproduction Assessment

Yes, at source level. Current main applies config env values, but profile-first runtime auth resolution and tests show stored auth profiles still shadow env credentials unless a caller explicitly opts into env-first resolution.

## Solution Assessment

Unclear. The issue lists several viable directions, and maintainers need to choose whether credential rotation should copy secrets, warn, offer an explicit sync, or alter fallback/precedence semantics.

## Security Review

Status: needs_attention

Summary: Credential rotation policy is security-sensitive because a fix could copy API keys across agent stores or change which credential source wins at runtime.

  - body: A sync or auto-sync implementation could write credential material across multiple agent stores unless maintainers require a narrow, auditable, dry-run-capable, or SecretRef-aware flow.
