## Summary

Keep open: current source supports the false-healthy status path, but I did not establish a live Slack reproduction for the reported Socket Mode delivery stall, and the safest fix needs owner judgment on transport heartbeat versus event-delivery liveness.

## What This Changes

The issue reports that Slack Socket Mode can remain connected and healthy while direct-message events stop reaching OpenClaw and `lastInboundAt` stops advancing.

## Best Possible Solution

Slack should either publish real Socket Mode transport heartbeat into `lastTransportActivityAt` or add a Slack-owned delivery-liveness design with live QA proof when history shows missed `message.im` events.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection does show a current-main path where Slack can remain healthy with stale `lastInboundAt` because Slack does not publish `lastTransportActivityAt`, but the reported missed-DM delivery stall was not reproduced against a real Slack workspace.

## Solution Assessment

Unclear. The generic stale-socket mechanism already exists, but deciding whether Slack should use SDK ping/pong, WebSocket envelope activity, history reconciliation, or another canary is a product/owner choice because quiet workspaces must not be marked broken.
