## Summary

Keep open: this is a fresh, concrete Slack Socket Mode bug report, and current main still has a status blind spot where Slack can be marked connected/healthy while inbound dispatch has not advanced. I did not find a merged fix or exact canonical duplicate; the related Slack start-account stall issue is worth linking, but this report preserves a distinct false-healthy DM delivery symptom.

## What This Changes

The issue asks OpenClaw to detect or recover when Slack Socket Mode remains connected/healthy but realtime DM events stop dispatching to OpenClaw.

## Best Possible Solution

Add Slack-owned event-delivery liveness or recovery that can distinguish a stale Socket Mode dispatch path from an idle workspace, then degrade/reconnect or surface diagnostics when Web API-visible DMs are not reaching the handler.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current-main status blind spot: Slack can remain connected/healthy while lastInboundAt only changes after Bolt dispatch reaches OpenClaw, and the probe only validates Web API auth.

## Solution Assessment

Unclear. The right fix is likely not a simple lastInboundAt idle timeout, because docs and code intentionally avoid using quiet app-level events as transport liveness; maintainers should choose a Slack-specific delivery probe, transport instrumentation, or recovery path that avoids false positives.
