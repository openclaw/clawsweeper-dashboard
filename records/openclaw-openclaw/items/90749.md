## Summary

Keep open: the supplied live proof now covers the OpenAI/Twilio path, but the patch still removes the existing provider-neutral local speech cancellation path and relies on an OpenAI-style `input_audio_buffer.speech_started` event, which can regress Google Gemini Live or other realtime voice-call providers.

## What This Changes

The PR moves voice-call realtime barge-in cancellation from local MuLaw speech detection to provider `input_audio_buffer.speech_started`, gates it on active output audio, and updates the realtime handler test and proof text.

## Best Possible Solution

Keep the new provider VAD cancellation for OpenAI, but preserve a provider-neutral barge-in fallback or add a typed cross-provider speech-start contract gated on active output audio and covered by a non-OpenAI regression test.

## Reproduction Assessment

Yes for source-level review: current main cancels from the local MuLaw detector, while the PR only cancels on provider `input_audio_buffer.speech_started`. I did not run tests because this was a read-only review.

## Solution Assessment

No; moving cancellation entirely to an OpenAI event is too narrow for the documented provider surface. The maintainable fix is to keep provider VAD handling while retaining a provider-neutral active-output barge-in path.
