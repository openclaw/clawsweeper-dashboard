## Summary

Keep this PR open: current main still does not cancel the voice-call Talk turn from provider `input_audio_buffer.speech_started`, and the latest branch addresses the earlier SDK/catalog documentation gap with focused runtime, protocol, docs, and regression coverage.

## What This Changes

The branch adds optional `emitsSpeechStartedEvent` realtime provider capability metadata, marks OpenAI as emitting provider speech-start events, routes voice-call barge-in cancellation through provider speech-start events when available, preserves local MuLaw fallback detection for other providers, and adds docs/tests.

## Best Possible Solution

Land the scoped provider-speech-start cancellation path with the additive capability contract, keeping the local MuLaw fallback for providers that do not advertise the event and relying on the required checks before merge.

## Reproduction Assessment

Yes. Current main shows provider `input_audio_buffer.speech_started` only ensuring a Talk turn in the voice-call handler, while the PR body supplies a redacted live Twilio/OpenAI call log showing provider barge-in clearing queued audio and emitting `turn.cancelled` after the patch.

## Solution Assessment

Yes. The PR puts telephony playback clearing and Talk cancellation in the voice-call transport owner while using an optional provider capability instead of hard-coding OpenAI or disabling local fallback for providers that need it.
