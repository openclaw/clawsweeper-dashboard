## Summary

Keep this PR open, but not merge-ready: the OpenAI/Twilio provider-VAD path is useful, yet the fallback gate is too broad and can break existing Google realtime voice-call barge-in cancellation.

## What This Changes

The PR moves realtime voice-call barge-in cancellation to provider `input_audio_buffer.speech_started`, guards cancellation on active output audio, keeps local MuLaw detection as a fallback/diagnostic path, and updates realtime handler tests.

## Best Possible Solution

Keep provider-VAD cancellation for OpenAI-style events, but preserve or replace the local fallback for providers whose barge-in signal is `onClearAudio`/interrupted rather than `speech_started`.

## Reproduction Assessment

Yes. Source inspection shows a `supportsBargeIn: true` provider without `speech_started` skips the PR's local fallback while its provider clear-audio path does not cancel the Talk turn.

## Solution Assessment

No. The OpenAI event path is the right direction, but using broad provider barge-in capability as the suppressor is not the narrowest maintainable fix across sibling providers.

Full review comments:

- **[P1] Preserve fallback for providers without speech-start events:** `extensions/voice-call/src/webhook/realtime-handler.ts:894-896`
  - body: This disables local MuLaw cancellation whenever `capabilities.supportsBargeIn` is true, but that capability is broader than the `input_audio_buffer.speech_started` event this handler now depends on. The Google realtime provider declares `supportsBargeIn: true` and handles interruption by calling `onClearAudio()` from `serverContent.interrupted`, without emitting that bridge event, so Google voice-call sessions can stop cancelling the active Talk turn. Keep the local fallback for providers that do not emit the provider speech-start event, or map the provider clear/interrupted signal to the same cancellation path before suppressing the fallback.
