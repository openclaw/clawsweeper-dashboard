## Summary

Keep open; current main still does not cancel voice-call realtime output on provider `input_audio_buffer.speech_started`, while the PR supplies a focused implementation and sufficient live-call proof. The remaining decision is maintainer acceptance of the new plugin-visible realtime provider capability/catalog field before merge.

## What This Changes

The PR moves voice-call realtime barge-in cancellation to provider speech-start events, preserves local MuLaw fallback for providers without that event, exposes an optional provider capability through Talk catalog/protocol, and updates docs/tests.

## Best Possible Solution

Land the scoped provider speech-start cancellation after maintainer acceptance of the additive provider capability contract, preserving local fallback for providers that omit the flag.

## Reproduction Assessment

Yes. Current main source shows provider `input_audio_buffer.speech_started` only starts or ensures a Talk turn, while the PR body supplies redacted live Twilio/OpenAI logs showing queued audio cleared and `turn.cancelled` after the patch.

## Solution Assessment

Yes, pending maintainer API acceptance. The fix stays in the voice-call transport owner, uses provider metadata instead of hard-coding OpenAI, and preserves local fallback behavior for providers without server speech-start events.
