## Summary

Keep open: the latest head fixes the earlier sibling-provider fallback issue, but it still gates provider barge-in clearing on Talk output state instead of telephony playback state, so queued phone audio can keep playing when the user interrupts after `response.done`.

## What This Changes

The PR moves voice-call realtime barge-in cancellation to provider `input_audio_buffer.speech_started`, adds an `emitsSpeechStartedEvent` provider capability for OpenAI, preserves local fallback for providers without that event, and updates focused tests.

## Best Possible Solution

Track or query transport playback/queued-audio state for barge-in clearing, while keeping provider-VAD cancellation and the non-speech-start provider fallback covered by regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: queue assistant audio, receive provider `response.done` before the pacer drains, then receive provider `input_audio_buffer.speech_started`; PR head returns before clearing audio because `talk.outputAudioActive` is false.

## Solution Assessment

No. Moving OpenAI-style barge-in to provider VAD is the right direction, but using Talk output activity as the clear gate is too narrow for paced phone playback; the fix should use playback/queue state or otherwise preserve clearing until the transport is drained.
