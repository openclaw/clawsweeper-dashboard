## Summary

Keep open: the PR targets a real voice-call barge-in bug, but the current patch removes the provider-agnostic local cancellation path and relies on an OpenAI-style provider event, which can regress Google/non-OpenAI realtime voice-call sessions. It also lacks after-fix real call proof.

## What This Changes

The branch moves voice-call realtime barge-in cancellation from local MuLaw speech detection to provider `input_audio_buffer.speech_started` events, guards cancellation on active assistant audio, and updates the realtime handler test.

## Best Possible Solution

Keep provider VAD cancellation for OpenAI but preserve or replace the local detector with a provider-independent fallback, then add regression coverage for OpenAI-style events and a provider without that event plus redacted live-call proof.

## Reproduction Assessment

Yes, source-reproducible: current main cancels on local MuLaw speech detection, while the PR version only logs local detection. Docs and Google provider code show Google is a supported realtime voice provider that does not emit the OpenAI `input_audio_buffer.speech_started` event.

## Solution Assessment

No, this is a plausible OpenAI fix but not the best provider-boundary fix yet. The safer solution keeps provider VAD cancellation when available while retaining a provider-independent fallback or adding a typed cross-provider speech-start contract.
