## Summary

Keep open; the central bug is not already fixed on current main, and this PR still bypasses existing OpenAI interruption-disable settings by cancelling voice-call output on the raw provider speech-start event before the OpenAI bridge applies that policy.

## What This Changes

The PR moves realtime voice-call barge-in cancellation to provider speech-start events, adds optional speech-start capability metadata through the Talk catalog/protocol, keeps a local MuLaw fallback for providers without that event, and updates tests/docs.

## Best Possible Solution

Land after provider speech-start cancellation preserves the resolved interruption policy, or after the OpenAI bridge emits a more specific policy-approved event, and maintainers accept the additive capability contract.

## Reproduction Assessment

Yes. Current main only ensures a Talk turn on provider `input_audio_buffer.speech_started`, while the PR body supplies redacted live Twilio/OpenAI logs for the fixed behavior and source inspection reproduces the PR's disabled-interruption regression path.

## Solution Assessment

No, not as submitted. Moving cancellation to the provider speech-start boundary is the right layer, but it must preserve existing OpenAI interruption-disable settings before the new capability contract is safe to merge.
