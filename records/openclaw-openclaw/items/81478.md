## Summary

Keep open: current main still has no per-agent `memorySearch.qmd.command` field and the strict agent memory-search schema would reject it. Existing QMD state is already per-agent, and the related memory-wiki and direct-read reports cover adjacent isolation surfaces, so this is a real config/API and trust-boundary decision rather than cleanup closure.

## What This Changes

Add an agent-level `agents.list[].memorySearch.qmd.command` override so different agents can route built-in memory search through different QMD executables or wrappers.

## Best Possible Solution

Decide whether per-agent QMD wrapper support belongs in the memory config contract; if accepted, add it consistently to types, zod schema, labels/help, docs, backend resolution, manager identity, and focused QMD tests while preserving existing per-agent XDG/QMD state isolation.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection does give a high-confidence configuration check: the requested per-agent `qmd.command` path is absent and would be rejected by the strict schema today.

## Solution Assessment

Unclear: the exact command override is plausible, but current main already gives each agent an isolated QMD home. The safer maintainer decision is whether users need a first-class per-agent executable, an environment override, or clearer support guidance.
