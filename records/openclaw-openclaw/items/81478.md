## Summary

Keep open: current main already isolates QMD state per agent, but it still does not accept or resolve a per-agent `qmd.command`, so the remaining request needs maintainer config/API judgment rather than cleanup closure.

## What This Changes

The issue requests allowing `agents.list[].memorySearch.qmd.command` to override the global QMD command so individual agents can use separate QMD wrappers.

## Best Possible Solution

Decide whether per-agent QMD wrapper support belongs in the memory config contract; if accepted, add it consistently to types, strict schema, labels/help, docs, backend resolution, and QMD manager tests while preserving existing per-agent QMD state isolation.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection confirms the requested `agents.list[].memorySearch.qmd.command` path is absent and would be rejected by strict validation today.

## Solution Assessment

Unclear: the exact per-agent `command` override is plausible, but current main already isolates QMD state per agent. The maintainable path is a maintainer decision on whether custom wrappers need a first-class per-agent executable setting or a different supported boundary.
