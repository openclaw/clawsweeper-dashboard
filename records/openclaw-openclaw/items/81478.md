## Summary

Keep open: current main already provides per-agent QMD state isolation, but the requested per-agent `qmd.command` field is still absent and strict config validation would reject it. The related memory-wiki and agent-privacy issues are adjacent rather than exact duplicates, so this needs maintainer config/API judgment rather than cleanup closure.

## What This Changes

Request adding `agents.list[].memorySearch.qmd.command` so a specific agent can use a different QMD executable or wrapper for built-in memory tools.

## Best Possible Solution

Decide whether per-agent QMD wrapper support belongs in the memory config contract; if accepted, add it consistently to types, zod schema, help/labels, docs, backend resolution, and QMD manager tests while preserving the existing per-agent XDG isolation.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source inspection does confirm the requested `agents.list[].memorySearch.qmd.command` path is absent and would be rejected by the strict schema today.

## Solution Assessment

Unclear: adding the exact command override is one plausible solution, but current main already handles per-agent QMD index isolation through per-agent XDG/QMD directories. The maintainable path is to first decide whether custom wrappers need a first-class per-agent executable or environment override.
