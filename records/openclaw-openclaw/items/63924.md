## Summary

Keep this PR open for maintainer handling. Current main already contains the Feishu schema part and has separate current behavior for provider auth aliases and skill containment, but the PR still proposes provider and skill changes that conflict with current contracts, including a security-sensitive symlink containment bypass.

## What This Changes

The PR replaces Feishu's hand-written channel config schema, changes health snapshots to read best-effort config, collapses Volcengine/BytePlus plan provider IDs during normalization, and lets skill paths fall back to unresolved paths when realpath containment fails.

## Best Possible Solution

Split the work. Keep the Feishu schema generation already present on main, handle any health freshness issue as a focused current-main patch with a runtime-snapshot regression test, keep plan provider IDs distinct while using manifest providerAuthAliases for shared credentials, and preserve fail-closed realpath containment for skills unless a reviewed canonicalization design covers symlink aliases safely.

## Reproduction Assessment

Yes for the blocking PR findings: the current manifests/docs show Volcengine and BytePlus plan providers are distinct providers, and the PR's skill fallback would return an unresolved path after realpath containment fails. The original health report is only partially reproducible from static code inspection and still needs a focused gateway runtime test.

## Solution Assessment

No. The PR is not the best way to solve the issues because it mixes unrelated fixes, includes one already-implemented Feishu change, and changes provider identity plus skill containment in ways that conflict with current contracts; a split, current-main repair is safer.

  - body: This moves `volcengine-plan` and `byteplus-plan` into generic provider normalization.   - confidence: 0.9
