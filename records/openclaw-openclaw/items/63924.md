## Summary

Keep open, but do not merge as-is. Current main already has the Feishu schema generation piece, while the submitted provider and skill-path changes would regress current contracts: plan providers are manifest-distinct and skill discovery intentionally enforces realpath containment.

## What This Changes

The PR replaces Feishu's hand-written config schema, changes gateway health snapshots to read best-effort config, collapses Volcengine/BytePlus plan provider IDs during normalization, and lets symlinked skills fall back to unresolved paths when realpath containment fails.

## Best Possible Solution

Split the work. Keep the already-landed Feishu schema generation, make any health freshness change as a focused current-main patch with a pinned-runtime regression test, keep plan provider IDs distinct while relying on manifest auth aliases, and handle symlink canonicalization only through a reviewed fail-closed design with escape tests.

## Reproduction Assessment

Yes for the blocking review findings: current manifests/tests show plan providers are distinct, and the workspace skill tests exercise symlink escapes resolving outside the configured root. The original health stale-config report still needs a focused runtime-snapshot regression test before landing that part.

## Solution Assessment

No. The PR bundles one already-implemented change with a plausible health fix and two changes that conflict with current provider and skill-security contracts; a narrower split is the maintainable path.

## Review Findings

Full review comments:

- **[P2] Keep plan provider IDs distinct:** `src/agents/model-selection.ts:94-100`
  - body: This collapses `volcengine-plan` and `byteplus-plan` into their base provider IDs in generic normalization. Current manifests define those plan providers as separate runtime/catalog providers with distinct coding base URLs, sharing only auth through `providerAuthAliases`, so model/catalog lookups can be routed through the wrong provider.

  - body: When a skill path's realpath escapes the configured root, this returns the unresolved path because the textual path is inside the root. Subsequent loading can follow that symlink and load an outside `SKILL.md`, bypassing the containment boundary covered by current symlink-escape tests.
  - confidence: 0.9
