## Summary

Keep open. Current main still has the reported shared embedded-run gap: trusted image tool output without `MEDIA:` text or `details.path` is not promoted to pending media, while the original tool-call args are available but not handed to the extractor; open PR #50359 remains the focused implementation path.

## What This Changes

The issue requests preserving media from `read` image tool results into Feishu outbound replies when image content exists but `details.path` is missing.

## Best Possible Solution

Review and land PR #50359, or an equivalent narrow core fix, so trusted image tool results can fall back to original `path`/`file_path`/`filePath` args before the existing pending-media merge and Feishu send path runs.

## Reproduction Assessment

Yes, by source inspection. Current main has a path where image content without `MEDIA:` text or `details.path` returns no media, while `startArgs` are available but not passed to the extractor.

## Solution Assessment

Yes. The targeted PR direction is the narrowest maintainable fix because downstream pending-media merge and Feishu `mediaUrls` delivery already exist; a Feishu-only outbound change would miss the shared embedded-run extraction gap.
