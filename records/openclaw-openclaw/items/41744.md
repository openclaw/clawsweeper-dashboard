## Summary

Keep open. Current main still extracts tool-result media only from structured media, MEDIA text, or details.path, and handleToolExecutionEnd does not pass the original read tool args into that extractor; PR #50359 is still the right focused implementation path to review.

## What This Changes

The issue reports that Feishu auto-replies can lose media from read image tool results before the final outbound payload, especially when image content exists but details.path is missing and the original tool-call args contain the file path.

## Best Possible Solution

Land a narrow, trust-scoped fallback in the embedded tool-result media path: pass tool_execution_start args into media extraction, use path/file_path/filePath only when trusted image content exists and no explicit media path is present, then rely on the existing pending-media merge and Feishu send path. Consolidate or close redundant older PRs after the reviewed fix lands.

## Reproduction Assessment

Yes. The current source and tests give a high-confidence static reproduction: image content without MEDIA text or details.path returns no media, and the handler has startData.args available but does not pass it to the extractor.

## Solution Assessment

Yes. The targeted PR direction is the narrowest maintainable fix because the downstream final-payload merge and Feishu send path already exist; broad Feishu-specific channel changes would miss the shared embedded-run extraction gap.
