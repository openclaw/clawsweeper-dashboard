## Summary

Keep open. Current main and the latest release still have the shared embedded-run media promotion gap: image tool results without `MEDIA:` text or `details.path` do not promote the original tool-call path into pending media, while an open targeted PR already covers the narrow fix.

## What This Changes

The issue asks OpenClaw to preserve media from Feishu embedded `read` image tool results into the final outbound `mediaUrls` payload when the tool result lacks `details.path`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50359, or an equivalent narrow core change, so trusted image tool results can fall back to original tool-call args before the existing pending-media merge and Feishu dispatch paths run.

## Reproduction Assessment

Yes, by source inspection. Current main still has a path where trusted image content without `MEDIA:` text or `details.path` yields no media, even though `startData?.args` is available in `handleToolExecutionEnd`.

## Solution Assessment

Yes. Passing original tool-call args into the shared media extractor is the narrow maintainable fix; a Feishu-only send change would miss the upstream embedded-run promotion gap.
