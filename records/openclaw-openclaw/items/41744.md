## Summary

Keep open. Current main still misses the read-image case where the tool result has image content but no `MEDIA:` text or `details.path`, and open PR #50359 remains the focused implementation path to review.

## What This Changes

Preserve media from Feishu embedded read-image tool results through final outbound payload assembly, including cases where the path is only present in original tool-call args.

## Best Possible Solution

Review and land PR #50359, or an equivalent narrow core fix, so trusted image tool results can fall back to original `path`/`file_path` args and then use the existing pending-media merge and Feishu send path.

## Reproduction Assessment

Yes, by source inspection. Current main has a clear path where image content without `MEDIA:` text or `details.path` returns no media, while the original read args are available but not passed to the extractor.

## Solution Assessment

Yes. The targeted PR direction is the narrowest maintainable fix because downstream final-payload merge and Feishu media sending already exist; a Feishu-only outbound change would miss the shared embedded-run extraction gap.
