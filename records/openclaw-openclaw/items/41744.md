## Summary

Keep open: current main and the latest release still miss the pathless read-image fallback, and an open linked PR already owns the intended fix path. The issue should not be closed or sent to a second repair lane while that PR is pending review and proof.

## What This Changes

The issue reports that Feishu final replies lose image media from read-tool image results before outbound delivery.

## Best Possible Solution

A narrow shared embedded-tool media fix should promote trusted read-image paths from original tool start args into pending media and preserve them into final channel payloads without adding Feishu-specific policy.

## Reproduction Assessment

Yes, at source level. Current main still has a test asserting that image content without `details.path` extracts no media URLs, and the read tool details contract does not carry the original path.

## Solution Assessment

Yes, the best fix is in the shared embedded tool-result media pipeline. Feishu already sends structured media when present, so a channel-specific workaround would be the wrong layer.
