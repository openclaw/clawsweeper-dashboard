## Summary

Keep open: current main and latest v2026.6.10 still leave pathless read-image tool results without outbound media, and an open but conflicting PR already targets this gap.

## What This Changes

The issue asks OpenClaw to preserve media from Feishu read-image tool results into the final outbound payload, including cases where the result has image content but no `details.path`.

## Best Possible Solution

Retarget or replace the linked fix in the active `embedded-agent-subscribe.*` media pipeline, preserve the trusted-media gate, and verify that Feishu receives `mediaUrls` for pathless read-image results.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current `read` can emit image blocks without a path, and the current extractor/test still leave that shape without media URLs. I did not run a live Feishu E2E in this read-only review.

## Solution Assessment

Unclear for the open PR as submitted because it is conflicting, but the right fix belongs in the shared embedded-agent media bridge rather than Feishu transport. The Feishu adapter already sends media when the final payload contains `mediaUrls`.
