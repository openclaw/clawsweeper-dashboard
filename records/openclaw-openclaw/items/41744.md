## Summary

Keep open. Current main still has the shared embedded-run media promotion gap described in the discussion: trusted image tool output without `MEDIA:` text or `details.path` is not promoted to pending media, while the original tool-call args are available but not passed into media extraction.

## What This Changes

The issue reports that Feishu replies can lose image media from `read` tool results before the final outbound payload receives `mediaUrl` or `mediaUrls`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50359, or an equivalent narrow core fix, so trusted image tool results can fall back to original tool args before the existing pending-media merge and Feishu delivery path runs.

## Reproduction Assessment

Yes, by source inspection. Current main still has a path where trusted image content without `MEDIA:` text, image-block `path`, or `details.path` yields no media, even though the original tool-call args are available in `handleToolExecutionEnd`.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because downstream pending-media merge and Feishu `mediaUrls` dispatch already exist; a Feishu-only send change would miss the shared embedded-run extraction gap.
