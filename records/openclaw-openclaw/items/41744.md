## Summary

Keep open. Current main still has the embedded-run media promotion gap for trusted image tool results that lack `MEDIA:` text and `details.path`, and the remaining central fix is already covered by the open targeted PR.

## What This Changes

The issue reports that Feishu replies can drop media from trusted `read` image tool results before final outbound payload assembly.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50359 or an equivalent core fallback so trusted image tool results can derive media from original tool-call path args before the existing pending-media merge and Feishu dispatch run.

## Reproduction Assessment

Yes. By source inspection, current main still has a path where trusted image content without `MEDIA:` text or `details.path` produces no media, even though `handleToolExecutionEnd` has the original start args available.

## Solution Assessment

Yes. Passing original tool-call args into the shared media extractor is the narrow maintainable fix; a Feishu-only send change would miss the upstream embedded-run promotion gap and other channels using the same pipeline.
