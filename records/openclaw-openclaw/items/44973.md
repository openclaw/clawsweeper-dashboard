## Summary

Keep open: current main still has the duplicate cron system-event path, and this PR is a plausible narrow fix, but it is not merge-ready because real behavior proof is missing and the stale branch needs current-main test expectation updates.

## What This Changes

The PR tags main-session cron system events with a `:main-system-event` context-key suffix, skips those events when building heartbeat cron reminder prompts, and adds a regression test.

## Best Possible Solution

Rebase the branch, update the affected cron context-key expectations, keep this marker-and-filter fix or an equivalent narrow dedupe, and merge only after redacted real cron/heartbeat proof confirms the duplicate wrapper is gone while ordinary cron reminders still wrap.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: main-session cron queues a `cron:` system event, heartbeat peeks and wraps it, and reply setup drains the same queue into `System` lines.

## Solution Assessment

Mostly yes. The marker-and-skip approach is a narrow maintainable fix for duplicated model input, but the branch needs current-main test updates, conflict resolution, and real behavior proof before merge.
