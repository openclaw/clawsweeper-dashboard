## Summary

Current main still has the requested privacy gap, but the PR needs a rebase and security/product review before it is mergeable: it adds a user-facing `privacy.redactPII` option without the full config contract, misses newer LLM-visible metadata fields, and uses unkeyed hashes for low-entropy identifiers. The contributor also has not supplied real after-fix behavior proof beyond unit-test output.

## What This Changes

The PR adds a `privacy.redactPII` config knob that threads into inbound prompt metadata builders to hash, pseudonymize, or strip selected OpenClaw-injected inbound metadata before it reaches LLM context, with focused tests.

## Best Possible Solution

Land a rebased privacy implementation that updates the typed/generated/documented config contract, covers every current LLM-visible system-injected metadata field, and either uses keyed pseudonyms or clearly narrows the documented privacy guarantee.

## Reproduction Assessment

Yes, by source inspection. Current main still serializes raw identifying inbound metadata into LLM-visible prompt context, and the focused verification path is the inbound metadata builder tests plus a real configured run showing redacted prompt context.

## Solution Assessment

No, not as-is. The direction is useful, but the patch is not the best maintainable solution until it is rebased onto the v2 metadata shape, completes the config contract, and resolves keyed versus unkeyed pseudonym semantics.

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, or package supply-chain changes, but the privacy-sensitive redaction feature uses reversible unkeyed hashes for low-entropy identifiers.

Concerns:
