## Summary

Keep open: current main still lacks this privacy mode and the duplicate issue was already closed in favor of this PR, but the patch is not merge-ready because it is stale against current prompt metadata contracts, has incomplete config follow-through, needs a privacy/security decision, and lacks real behavior proof.

## What This Changes

The PR adds a `privacy.redactPII` config option and redaction/pseudonymization logic for OpenClaw-injected inbound metadata in LLM prompt context, with focused inbound metadata tests.

## Best Possible Solution

Land a rebased privacy implementation that redacts every current prompt-visible metadata path in user context, keeps the trusted system prefix cache-stable, aligns the full config contract, and either keys or clearly documents pseudonymization before merge.

## Reproduction Assessment

Yes, source-reproducible: current main still serializes raw identifying inbound metadata through `buildInboundUserContextPrefix`. I did not run a live configured OpenClaw prompt because this review is read-only.

## Solution Assessment

No, not as-is. The direction is useful, but the patch needs a current-main rebase, config-contract follow-through, coverage for newer metadata fields, real behavior proof, and a maintainer/security decision on keyed versus unkeyed pseudonyms.

Overall confidence: 0.9

Full review comments:

- **[P2] Complete the privacy config contract:** `src/auto-reply/reply/get-reply-run.ts:268`
  - body: `cfg` is typed as `OpenClawConfig`, but this PR only adds `privacy.redactPII` to the zod schema. Add the exported config type plus the generated/help/docs config surfaces so this public key is valid everywhere it is consumed.
  - confidence: 0.9
- **[P2] Keep chat IDs out of the system prefix:** `src/auto-reply/reply/inbound-meta.ts:81-84`
  - body: Current main intentionally keeps conversation IDs out of trusted system metadata for prompt-cache stability. Hashing `chat_id` still makes the system prefix vary by conversation, so redaction should happen in user-context metadata instead.
  - confidence: 0.88
- **[P2] Pass redaction through the current prefix options:** `src/auto-reply/reply/get-reply-run.ts:303`
  - body: On current main, `buildInboundUserContextPrefix` takes envelope formatting as the second argument and source-reply options as the third. After rebasing, this second-argument redaction path would not enable user-context redaction unless the current options shape is extended correctly.
  - confidence: 0.86
- **[P2] Redact the current group identity fields:** `src/auto-reply/reply/inbound-meta.ts:191-194`
  - body: This branch strips older group labels, but current main also emits `group_members`, `topic_name`, and chat-window sender labels in prompt-visible user context. Those fields need stripping or pseudonymization or `privacy.redactPII=true` can still leak participant or topic identity.
  - confidence: 0.86
- **[P2] Key low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7-10`
  - body: `hashId` uses bare truncated SHA-256 for values that can include phone-like identifiers. Prompt logs can contain reversible pseudonyms for low-entropy IDs, so use an operator-secret HMAC or narrow and document the privacy guarantee before merging.
  - confidence: 0.85
