## Summary

Keep open. Current main still exposes the LLM-visible inbound metadata this PR targets, but the submitted patch is stale/incomplete for the current config contract, current metadata fields, and privacy-strength semantics.

## What This Changes

The PR adds a `privacy.redactPII` option, threads it into inbound prompt metadata builders, redacts selected chat/sender/group/reply/forward/history metadata, and adds focused inbound metadata tests.

## Best Possible Solution

Land a rebased implementation that adds the typed/generated/documented config contract, redacts every current human-identifying system-injected metadata field, and either keys low-entropy pseudonyms or clearly documents the weaker deterministic-label guarantee.

## Reproduction Assessment

Yes. Current-main source and tests show raw sender, chat, group, topic, reply, forward, and history metadata are serialized into LLM-visible user context; the focused verification path is the inbound metadata test suite plus new redaction cases.

## Solution Assessment

No as-is. The direction is useful and scoped, but the patch is not the narrowest maintainable solution until it matches the current config contract, current metadata set, and agreed privacy-strength semantics.

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain changes, but the privacy-sensitive redaction feature uses reversible unkeyed hashes for low-entropy identifiers.

Concerns:

- **[medium] Reversible phone-number pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7`
  - body: Phone-like sender identifiers are hashed with unsalted, truncated SHA-256. Anyone with prompt output can precompute likely phone numbers and recover or correlate identities, which is too weak for a PII-redaction setting unless maintainers intentionally accept and document that limitation.
