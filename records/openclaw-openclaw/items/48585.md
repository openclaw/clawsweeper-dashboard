## Summary

Keep open: current main still has the LLM-visible metadata exposure this PR targets, and the canonical issue was closed in favor of this PR, but the branch is conflicting/stale and has blocking correctness, security, config-contract, and real-behavior-proof gaps.

## What This Changes

The branch adds a `privacy.redactPII` config option and threads redaction into inbound prompt metadata builders/tests to hash or strip sender, chat, group/thread, reply, forward, and history metadata.

## Best Possible Solution

A rebased implementation should keep volatile IDs in user-context only, align schema/types/docs/generated config surfaces, cover current prompt-visible fields, and use or document keyed pseudonym semantics.

## Reproduction Assessment

Yes, by source inspection: current main's `buildInboundUserContextPrefix` emits raw chat, sender, group/thread, reply, forward, and history metadata into prompt-visible blocks. I did not run a live gateway flow.

## Solution Assessment

No, not as-is. The direction is useful, but the current PR needs a rebase, full config-contract follow-through, broader current-field coverage, and a security/product decision on pseudonym strength.

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the privacy feature introduces unkeyed truncated SHA-256 pseudonyms for low-entropy identifiers.

Concerns:

- **[medium] Reversible low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7`
  - body: Phone-like sender identifiers are hashed with unsalted, truncated SHA-256. Anyone with prompt output can precompute likely phone numbers and recover or correlate identities unless maintainers explicitly accept and document that weaker privacy guarantee or switch to keyed pseudonyms.
