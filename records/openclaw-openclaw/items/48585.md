## Summary

Keep open. Current main still has the underlying LLM prompt privacy gap, but this PR is not merge-ready because the branch predates current inbound metadata hardening, omits the full config contract, and introduces reversible low-entropy pseudonyms under a PII-redaction setting.

## What This Changes

The PR adds a `privacy.redactPII` config option, threads it into inbound prompt metadata builders, pseudonymizes or strips selected chat/sender/group/reply/forward/history metadata, and adds focused inbound metadata tests.

## Best Possible Solution

Land a rebased implementation that keeps current main's prompt-cache separation, adds the typed/generated/documented config contract, redacts every current human-identifying system-injected metadata field including `group_members` and `topic_name`, and either uses keyed pseudonyms or documents a weaker deterministic-label guarantee clearly.

## Reproduction Assessment

Yes. Current-main source and tests show raw sender, chat, group, topic, reply, forward, and history metadata are serialized into LLM-facing user context; the focused verification path is the existing inbound metadata test suite plus new redaction cases.

## Solution Assessment

No as-is. The direction is focused and useful, but the submitted patch is stale against current main and incomplete for config contracts, newer metadata fields, and privacy-strength semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Add the privacy config contract:** `src/auto-reply/reply/get-reply-run.ts:268`
  - body: `cfg` is typed as `OpenClawConfig`, but the PR only adds `privacy.redactPII` to the Zod schema. Strict typecheck will reject `cfg.privacy`, and the generated/docs config surface stays stale unless the config type and config docs/schema outputs are updated with the new key.
  - confidence: 0.9
- **[P2] Strip current group identity fields:** `src/auto-reply/reply/inbound-meta.ts:191-194`
  - body: The redacted conversation block strips older group labels, but current main also emits human-readable `group_members` and `topic_name` beside those fields. After rebasing, `privacy.redactPII=true` would still leak group participants or forum topic names unless those fields are stripped or pseudonymized too.
  - confidence: 0.84
- **[P2] Key low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7-10`
  - body: `hashId` uses bare truncated SHA-256 for values that can include phone-like identifiers. Those identifiers have a small search space, so prompt logs still contain reversible pseudonyms under a setting named PII redaction; use an operator-secret HMAC or narrow the exposed guarantee before merging.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain changes, but the privacy-sensitive redaction feature uses reversible unkeyed hashes for low-entropy identifiers.

Concerns:

- **[medium] Reversible phone-number pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7`
  - body: Phone-like sender identifiers are hashed with unsalted SHA-256 and truncated. Anyone with prompt output can precompute likely phone numbers and recover or correlate identities, which is too weak for a feature exposed as PII redaction unless maintainers intentionally accept and document that limitation.
  - confidence: 0.82
