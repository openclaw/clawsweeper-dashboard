## Summary

## What This Changes

The PR adds a `privacy.redactPII` config option and redaction/pseudonymization logic for OpenClaw-injected inbound metadata in LLM prompt context, with focused inbound metadata tests.

## Best Possible Solution

Land a rebased, maintainer-approved privacy implementation that keeps trusted system prefixes cache-stable, redacts every current prompt-visible metadata path in user context, aligns the full config contract, and keys or documents pseudonymization.

## Reproduction Assessment

Yes, source-reproducible: current main still serializes raw identifying inbound metadata through `buildInboundUserContextPrefix`. I did not run a live configured OpenClaw prompt because this review is read-only.

## Solution Assessment

No as-is. The direction is useful, but the PR needs a current-main rebase, full config-contract follow-through, coverage for newer prompt-visible metadata fields, real behavior proof, and a maintainer/security decision on keyed versus unkeyed pseudonyms.

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the privacy feature needs attention because low-entropy identifiers use unkeyed truncated hashes and the public privacy contract is incomplete.

Concerns:

- **[medium] Reversible low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7`
  - body: Phone-like sender identifiers are hashed with unsalted, truncated SHA-256. Anyone with prompt output can precompute likely phone numbers and recover or correlate identities unless maintainers explicitly accept and document that weaker privacy guarantee.
  - confidence: 0.85

  - body: The PR exposes a public `privacy.redactPII` switch but does not align the exported config type and related config surfaces, which can make the privacy control fail or be misunderstood by operators.
