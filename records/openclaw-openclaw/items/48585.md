## What This Changes

The PR adds a `privacy.redactPII` config option and redacts or pseudonymizes selected inbound metadata before it is injected into LLM prompt context.

## Best Possible Solution

Rebase and redesign the privacy path to redact all current user-context metadata, keep trusted system metadata cache-stable, align the full config contract, and settle keyed pseudonym semantics before merge.

## Reproduction Assessment

Yes, source-reproducible: current main's `buildInboundUserContextPrefix` still serializes raw identifying inbound metadata into prompt-visible user context. I did not run a live configured OpenClaw prompt in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the patch needs a current-main rebase, full config contract follow-through, broader metadata coverage, real behavior proof, and a maintainer/security decision on keyed versus unkeyed pseudonyms.

  - body: Current main intentionally keeps conversation IDs out of trusted system metadata for prompt-cache stability. Hashing `chat_id` still makes the system prefix vary by conversation, so redaction should happen in user-context metadata instead.
  - confidence: 0.88
- **[P2] Thread redaction through the current prefix options:** `src/auto-reply/reply/get-reply-run.ts:306`
  - body: On current main, `buildInboundUserContextPrefix` takes envelope formatting as the second argument and source-reply options as the third. After rebasing, this second-argument redaction path would not enable user-context redaction unless the current options shape is extended correctly.
  - confidence: 0.86
- **[P2] Redact current prompt-visible identity fields:** `src/auto-reply/reply/inbound-meta.ts:191-194`
  - body: This branch strips older group labels, but current main also emits `group_members`, `topic_name`, chat-window sender labels, and reply-chain sender fields in prompt-visible user context. Those need stripping or pseudonymization or the privacy option will still leak participant or topic identity.
  - confidence: 0.86
- **[P2] Use keyed pseudonyms for low-entropy identifiers:** `src/auto-reply/reply/inbound-meta.ts:7-9`
  - body: `hashId` uses bare truncated SHA-256 for values that can include phone-like identifiers. Prompt logs can contain reversible pseudonyms for low-entropy IDs, so use an operator-secret HMAC or narrow and document the privacy guarantee before merging.
  - confidence: 0.85
