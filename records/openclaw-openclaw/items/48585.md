## Summary

Keep this PR open. Current main still has no `privacy.redactPII` surface and still injects raw human-identifying inbound metadata into the LLM-visible user context, so the PR is not obsolete; however, the branch is not merge-ready because it misses the typed/generated config contract, needs a rebase over the current inbound-metadata shape, and has a real privacy-strength concern around low-entropy hashed identifiers.

## What This Changes

The PR adds a `privacy.redactPII` config option, threads it into inbound prompt metadata builders, pseudonymizes or strips selected prompt-visible chat/sender/group/reply/forward/history metadata, and adds focused inbound metadata tests.

## Best Possible Solution

Land a rebased implementation that preserves main's prompt-cache separation and current inbound metadata hardening, adds the full typed/generated/documented config contract, redacts every current human-identifying system-injected metadata field covered by the feature, and clearly defines whether the pseudonyms are keyed privacy protection or weaker deterministic labels.

## Reproduction Assessment

Yes. For current-main behavior, source and tests show raw sender, chat, group, reply, forward, and history metadata are serialized into LLM-facing context; the review did not execute tests because the task is read-only.

## Solution Assessment

No as-is. The direction is focused and useful, but the current patch is not the narrowest maintainable solution until it updates config contracts, rebases over current prompt metadata, covers newer human-readable fields, and resolves the pseudonymization-strength concern.

- **[P2] Add the privacy config contract:** `src/auto-reply/reply/get-reply-run.ts:268`
  - body: `cfg` is typed as `OpenClawConfig`, but this PR only adds `privacy.redactPII` to the Zod schema. Since `OpenClawConfig` has no `privacy` member, strict typecheck rejects this read, and the generated schema/docs surface stays stale unless the config type and generated config outputs are updated with the new key.
  - confidence: 0.9
- **[P2] Strip current group identity fields:** `src/auto-reply/reply/inbound-meta.ts:191-194`
  - body: The redacted conversation block strips `group_subject`, `group_channel`, `group_space`, and `thread_label`, but current main also emits human-readable `group_members` and `topic_name` next to these fields. After rebasing, `privacy.redactPII=true` would still leak group participants or forum topic names unless those fields are stripped or pseudonymized too.
  - confidence: 0.82
- **[P2] Key low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:9`
  - body: `hashId` uses bare truncated SHA-256 for values that include phone-like identifiers. Those identifiers have a small search space, so prompt logs still contain reversible pseudonyms under a setting named `redactPII`; use an operator-secret HMAC or narrow the exposed privacy guarantee before merging.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces no supply-chain changes, but the privacy-sensitive pseudonymization uses bare truncated SHA-256 for low-entropy identifiers and needs maintainer attention before merge.

Concerns:

- **[medium] Reversible phone-number pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:9`
  - body: Phone-like sender identifiers are hashed with unsalted SHA-256 and truncated. Anyone with prompt output can precompute likely phone numbers and recover or correlate identities, which is too weak for a feature exposed as PII redaction unless the limitation is intentionally accepted and documented.
