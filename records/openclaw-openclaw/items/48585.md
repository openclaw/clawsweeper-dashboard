## Summary

Keep open: the privacy direction is still useful and current main has not implemented `privacy.redactPII`, but this PR is not mergeable as-is because it is stale against the current prompt metadata shape, leaves the public config contract incomplete, needs a security/product decision on pseudonym strength, and still lacks real behavior proof.

## What This Changes

The PR adds a `privacy.redactPII` config path plus inbound prompt metadata hashing/stripping logic and tests across `get-reply-run`, `inbound-meta`, and the zod config schema.

## Best Possible Solution

Land a rebased privacy implementation that redacts every current prompt-visible metadata path, keeps volatile IDs out of the system prefix, aligns the full config contract, and documents or strengthens the pseudonymization guarantee.

## Reproduction Assessment

Yes, by source inspection rather than a runtime run: current main still serializes raw identifying inbound metadata through `buildInboundUserContextPrefix`. I did not run a live configured OpenClaw prompt because this was a read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the patch needs a current-main rebase, config-contract follow-through, coverage for newer metadata fields, real behavior proof, and a maintainer/security decision on keyed versus unkeyed pseudonyms.

  - body: Current main intentionally excludes conversation IDs from the system prompt for prompt-cache stability. Reintroducing even hashed `chat_id` into `buildInboundMetaSystemPrompt` still varies by conversation, so the redaction should happen in user-context metadata instead.
  - confidence: 0.88
- **[P2] Pass redaction through the current prefix options:** `src/auto-reply/reply/get-reply-run.ts:303`
  - body: On current main, `buildInboundUserContextPrefix` takes envelope formatting as the second argument and source-reply options as the third.   - confidence: 0.86
- **[P2] Redact the current group identity fields:** `src/auto-reply/reply/inbound-meta.ts:191-194`
  - body: This branch strips older group labels, but current main also emits `group_members`, `topic_name`, and chat-window sender labels in prompt-visible user context. Those fields need stripping or pseudonymization or `privacy.redactPII=true` can still leak participant or topic identity.
  - confidence: 0.86

  - body: `hashId` uses bare truncated SHA-256 for values that can include phone-like identifiers. Prompt logs can contain reversible pseudonyms for low-entropy IDs, so use an operator-secret HMAC or narrow and document the privacy guarantee before merging.
  - confidence: 0.85
