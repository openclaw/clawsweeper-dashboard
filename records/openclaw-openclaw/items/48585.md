## Summary

Keep open: current main still lacks the requested `privacy.redactPII` behavior, and this PR is useful but incomplete/stale against the current inbound metadata and config surfaces.

## What This Changes

The PR adds a `privacy.redactPII` option, threads it into inbound prompt metadata builders, pseudonymizes or strips selected LLM-visible inbound metadata, and adds focused inbound metadata tests.

## Best Possible Solution

Land a rebased privacy implementation that updates the typed/generated/documented config contract, covers every current LLM-visible system-injected metadata field, and settles keyed pseudonyms versus a clearly documented weaker guarantee.

## Reproduction Assessment

Yes by source inspection. Current main clearly serializes raw sender, chat, group/thread, reply, forward, and history metadata into LLM-visible user context, and the focused verification path is the inbound metadata test suite plus new redaction cases.

## Solution Assessment

No as-is. The direction is well-scoped, but the patch is not yet the best maintainable solution because it misses current config contract outputs, current metadata fields, and agreed privacy-strength semantics.

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, or package supply-chain changes, but the privacy-sensitive redaction feature uses reversible unkeyed hashes for low-entropy identifiers.

Concerns:

- **[medium] Reversible low-entropy pseudonyms:** `src/auto-reply/reply/inbound-meta.ts:7`
