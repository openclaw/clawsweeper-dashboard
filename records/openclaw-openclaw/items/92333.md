## Summary

OpenClaw’s supported `config.patch` path already recursively preserves sibling model fields, while the reported switch occurs in the separate AutoClaw desktop client and no destructive OpenClaw request is shown; the actionable change is therefore in that client unless a future report demonstrates a gateway contract violation.

## What This Changes

Report that AutoClaw desktop primary-model selection removes the existing `agents.defaults.model.fallbacks` configuration.

## Best Possible Solution

Keep OpenClaw’s recursive merge-patch contract unchanged and update AutoClaw’s model selector to send only `agents.defaults.model.primary`; open a new OpenClaw issue with the exact RPC method, raw payload, base hash, and replace paths if that request still removes `fallbacks`.

## Reproduction Assessment

No. The supported current and shipped `config.patch` implementation preserves nested sibling fields, and the issue does not include the external AutoClaw client’s actual RPC method or payload.

## Solution Assessment

No. Adding a core merge workaround would duplicate or weaken an already-correct contract; the narrow fix is for the external client to send a primary-only nested patch.
