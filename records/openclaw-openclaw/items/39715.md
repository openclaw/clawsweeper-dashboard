## Summary

Keep open. The production fix is still aimed at a real background exec output-loss path, but the latest PR head introduces a likely blocking test hang: switching from fake to real timers discards the queued fake `setImmediate` that the new supervisor drain is waiting on.

## What This Changes

This PR adds a post-settle supervisor I/O drain before stdout/stderr snapshots, hardens cancellation/finalization around that drain, adds regression coverage, and updates the changelog for background sandbox exec output loss.

## Best Possible Solution

Keep the narrow supervisor-level drain and hardening approach, but repair the fake-timer tests so the queued `setImmediate` is deterministically drained without discarding pending fake timers.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The linked report gives a concrete Bun/Docker background exec path, and current main shows child `close` resolution followed by immediate aggregate snapshotting without a drain turn.

## Solution Assessment

No, not as currently patched. The production direction is narrow and maintainable, but the latest test repair uses `vi.useRealTimers()` in a way that can discard the pending fake `setImmediate` required for `run.wait()` to resolve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drain fake immediates before restoring real timers:** `src/process/supervisor/supervisor.test.ts:166`
  - body: These tests switch back to real timers after `advanceTimersByTimeAsync()` has already let `run.wait()` queue the new supervisor `setImmediate`. Vitest’s `useRealTimers()` uninstalls fake timers and discards scheduled timers, so that queued drain never fires and `await exitPromise` can hang. Keep fake timers and explicitly run the pending immediate, or avoid faking `setImmediate` for these tests.
  - confidence: 0.9
