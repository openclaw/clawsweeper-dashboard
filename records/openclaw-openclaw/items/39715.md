## Solution Assessment

No, not yet: the drain-boundary idea is narrow, but the current placement exposes an incoherent kill/remove state before final output publication. A safer fix preserves the drain while making the completing state explicit or unobservable to process actions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep sessions out of fallback removal while draining:** `src/process/supervisor/supervisor.ts:229-234`
  - body: By finalizing the supervisor record and deleting the active run before the new `setImmediate`, there is a full event-loop turn where `process remove` still finds the background `ProcessSession` but `cancelManagedSession()` returns false because the supervisor record is already `exited`. That path falls back to `killProcessTree()`, marks/deletes the session, and the later `managedRun.wait()` continuation can re-add the supposedly removed session as finished. Keep this draining state out of the fallback kill/remove path or treat it as already completing.
  - confidence: 0.89
