## Summary

Keep open: the feature is not already on current main, but this PR is not merge-ready because the terminal failure hook exposes a security-sensitive failure message outside the existing conversation-access gate and the new public hook surface lacks docs/changelog coverage.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, wires them into embedded model failover and terminal before-reply failure paths, and adds focused hook tests.

## Best Possible Solution

Keep the narrow hook direction only after the terminal payload/access-control contract is settled, then document both hooks, add the required changelog entry, rebase onto current main, and validate the focused plugin/failover tests.

## Reproduction Assessment

Yes. The issue is reproducible by inspecting the PR diff where `finalMessage: message` is emitted, current-main fallback summary construction from per-attempt errors, and the current conversation hook registry gate.

## Solution Assessment

No, not as currently written. The hook seam is reasonable, but terminal failure details should either be sanitized to non-sensitive fields or protected by the same conversation-access policy before merge.
