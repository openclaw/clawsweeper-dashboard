## Summary

Keep open: the feature is useful and the supplied terminal proof is enough for the proof gate, but the current PR head still publishes the failed attempt as `provider`/`model` for `model_failover` while the PR discussion and docs describe those fields as the current or selected fallback target.

## What This Changes

Adds typed `model_failover` and `model_failure_terminal` plugin hooks, wires them into embedded failover and terminal before-reply failure paths, and updates docs, config help, changelog, and focused tests.

## Best Possible Solution

Land the hook API after the failover event carries selected fallback target facts from the outer fallback step, or explicitly narrow the public contract so plugins do not infer a selected target that is not present.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible: the PR head passes the failed attempt as both source and `provider`/`model` before `runWithModelFallback` advances to `nextCandidate`.

## Solution Assessment

No, not yet. The hook runner, terminal gate, and fire-and-forget shape are reasonable, but the `model_failover` payload should not expose duplicate failed-model fields under a target contract that operators expect to identify the selected fallback.
