## Summary

Keep open. The PR adds a useful plugin observability surface and has sufficient terminal proof, but the latest head still has a blocking public API semantics issue: prompt-side failover publishes the failed attempt as both source and target, so plugins cannot reliably tell which fallback was selected.

## What This Changes

Adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and terminal before-reply failure paths, and updates docs, config help, tests, and changelog.

## Best Possible Solution

Align `model_failover` with the existing fallback-step facts so plugin authors can distinguish failed source and selected fallback target without log scraping.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible from PR head: prompt failover passes the failed attempt as both source and target while current main’s outer fallback layer already has `nextCandidate` facts.

## Solution Assessment

No, not yet. The hook runner, terminal gate, and fail-open plumbing are reasonable, but the public payload should not imply a selected target when prompt fallback currently emits A/A instead of A/B.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
