## Summary

Keep open: the hook plumbing and conversation-access gate look useful, and the stale Greptile `stage` concern appears addressed, but the current PR still has a blocking public API semantics issue for `model_failover`.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and terminal failure paths, and updates plugin docs, config help, tests, and changelog.

## Best Possible Solution

Land this only after the hook payload/timing contract clearly distinguishes failed source from selected fallback target, or explicitly narrows the public docs/types to failed-attempt telemetry.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible from PR head: the prompt failover event copies the failed attempt into both target and source fields while current main's outer fallback layer has `nextCandidate` facts.

## Solution Assessment

No, not yet. The hook runner, terminal gate, and fail-open plumbing are reasonable, but the public `model_failover` payload should not imply a selected target when prompt fallback currently emits A/A instead of A/B.
