## Summary

Keep open: this is a useful feature PR with supplied real-behavior proof and green relevant CI, but the public `model_failover` contract still has a correctness/API issue because PR head can publish the failed prompt model as both the source and target while the docs and discussion describe `provider`/`model` as the current or selected target.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and terminal before-reply paths, and updates plugin docs, config help, tests, and changelog.

## Best Possible Solution

Settle the public hook contract before merge: either emit selected fallback target facts from the outer fallback seam, or explicitly narrow the docs/types so `model_failover` is failed-attempt telemetry rather than selected-target telemetry.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible from PR head: the prompt failover call passes the same failed provider/model into both target and source fields while current main's outer fallback layer has `nextCandidate` facts.

## Solution Assessment

No, not yet. The terminal hook gate and fire-and-forget plumbing are reasonable, but the public `model_failover` payload/timing needs a maintainer API decision before it is safe to publish.
