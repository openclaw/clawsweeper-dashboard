## Summary

Keep open. Current main does not implement these model failover hooks, and the PR addresses a real observability gap from the linked feature request, but the proposed public plugin API still has a blocking security/data-access issue: the terminal failure hook forwards terminal error text without being covered by the existing conversation-access gate. It also needs docs and changelog coverage before merge.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from failover-decision and terminal before-reply failure paths, and adds focused hook-runner tests.

## Best Possible Solution

Keep the narrow hook API if maintainers accept the product direction, but align the terminal failure payload with the existing plugin data-access model, document both hooks and their access requirements, add the required changelog entry, and validate the focused plugin/failover tests before merge.

## Reproduction Assessment

Yes. The review path is high-confidence: inspect the PR diff's terminal emission, confirm current main's conversation hook gate and fallback summary construction, and verify current main lacks the new hook names.

## Solution Assessment

No, not as currently written. The hook direction is a reasonable core/plugin seam, but the terminal failure hook should be sanitized or gated like other conversation/error-observation hooks before becoming public.
