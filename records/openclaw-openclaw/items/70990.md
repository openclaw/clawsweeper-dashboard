## Summary

Keep open: the PR is useful and current main does not have this hook surface, but the current head still has a blocking public payload semantics issue for `model_failover`. The docs and discussion describe `provider`/`model` as the selected/current fallback target, while the prompt-failure emission path still supplies the failed attempt as both source and target.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and before-reply terminal failure paths, gates terminal failure payloads behind conversation access, and updates docs, tests, and changelog.

## Best Possible Solution

Land the hook API only after the public payload either carries selected fallback target facts from the outer fallback step or explicitly narrows the docs/types so plugins do not infer a selected target that is not present.

## Reproduction Assessment

Yes for the review finding: source inspection of the current PR head shows prompt failover sets `provider`/`model` and `sourceProvider`/`sourceModel` from the same failed attempt before the outer fallback loop advances. This is not a current-main bug repro because the hook is a new feature.

## Solution Assessment

No, not yet. The hook runner, terminal failure gate, and fire-and-forget shape are reasonable, but the selected-target semantics must be corrected or explicitly narrowed before this becomes a public plugin API.
