## Summary

Keep open: the feature direction is useful, but the PR still has a public hook-contract blocker. The `model_failover` docs promise a current/selected target while the prompt-side emission still passes the failed provider/model as both target and source, and the branch also carries unrelated changelog entries.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and terminal before-reply failure paths, and updates plugin docs, config help, tests, and changelog.

## Best Possible Solution

Finalize the public hook contract first: either emit selected fallback target facts from the outer fallback seam, or narrow the docs/types/proof so `model_failover` is explicitly failed-attempt telemetry; then remove unrelated changelog entries.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The blocking issue is source-reproducible from PR head: prompt failover passes the same failed provider/model into both target and source fields while current main has selected `nextCandidate` facts in the outer fallback layer.

## Solution Assessment

No, not yet. The terminal hook access gate and fire-and-forget plumbing are reasonable, but the `model_failover` payload/timing semantics need maintainer API judgment before publishing.
