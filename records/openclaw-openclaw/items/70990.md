## Summary

Keep open: current main still lacks these hooks and this PR carries unique feature work, but the proposed `model_failover` payload currently exposes the failed candidate as `provider`/`model` even though the docs and discussion describe those fields as the current or selected fallback target.

## What This Changes

Adds typed `model_failover` and `model_failure_terminal` plugin hooks, wires them into embedded failover and terminal before-reply failure paths, and updates docs, config help, changelog, and focused tests.

## Best Possible Solution

Land a hook contract that uses the existing fallback-step facts for selected target information while keeping terminal failure summaries behind the existing conversation-access gate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. The payload mismatch is source-reproducible because the PR emits `model_failover` inside the failed inner attempt before `runWithModelFallback` advances to the next candidate.

## Solution Assessment

No, not yet. The hook runner and gating approach are narrow, but the `model_failover` payload must not present the failed model as the selected fallback target under the documented contract.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Populate the fallback target before publishing the hook event:** `src/agents/pi-embedded-runner/run/failover-observation.ts:123-126`
