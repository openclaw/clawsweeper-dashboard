## Best Possible Solution

Keep the new hook API only if maintainers accept the public contract, and emit one terminal-failure event from the shared exhaustion handling for thrown summaries and exhausted results across direct, auto-reply, and follow-up runner paths.

## Solution Assessment

No: the hook API is a plausible direction, but the implementation should be anchored at the shared terminal exhaustion handling instead of only the thrown-error branch. That shape avoids publishing a hook that misses a central all-models-failed path.
