## Summary

Keep open: the PR is useful and has real terminal proof, but one public hook-contract issue remains because `model_failover` still reports the failed prompt attempt as both the source and target for fallback decisions.

## What This Changes

Adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from embedded failover and terminal before-reply failure paths, and updates docs, config help, tests, and changelog.

## Best Possible Solution

Align the new hook contract with existing fallback-step facts: carry the selected next fallback target into the hook payload, or explicitly document `provider`/`model` as the failed current attempt and add separate target fields when available.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible: PR head passes the failed prompt attempt as both `provider`/`model` and `sourceProvider`/`sourceModel`, while the outer fallback layer already has `nextCandidate` target facts.

## Solution Assessment

No, not yet. The hook runner, terminal gate, and fail-open plumbing are reasonable, but the public `model_failover` payload should not imply a selected target when prompt fallback events currently publish A/A instead of A/B.
