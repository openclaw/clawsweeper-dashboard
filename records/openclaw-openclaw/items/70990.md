## Summary

Keep open: the hook direction is useful and the latest head addresses the earlier main auto-reply findings, but the new `model_failover` contract is still one-sided because queued follow-up runs share the fallback runner and do not receive the selected next candidate.

## What This Changes

This PR adds `model_failover` and `model_failure_terminal` Plugin SDK hooks, wires them into embedded failover and terminal auto-reply failure paths, updates docs/generated baselines, and adds regression tests.

## Best Possible Solution

Land the hook contract only after all Gateway fallback paths that can emit `model_failover` either propagate the selected target or are explicitly documented as out of scope, with the terminal hook remaining conversation-access gated.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the remaining defect: queued follow-up runs call the same fallback orchestrator but do not pass the selected next candidate into `runEmbeddedAgent`. I did not execute the queued-follow-up scenario in this read-only review.

## Solution Assessment

No, not yet: the hook design is plausible, but the implementation is not the best complete fix while a sibling Gateway fallback path can emit the new hook without the selected fallback target. The safer pre-merge repair is to wire the same next-candidate context through `followup-runner.ts` or explicitly scope the contract away from that path.
