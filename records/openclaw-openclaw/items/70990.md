## Summary

Keep open. The PR is useful and current main still lacks these first-class failover hooks, but the proposed terminal hook exposes a new plugin data surface before the access-control and documentation contract are settled.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, emits them from failover and terminal before-reply failure paths, and adds focused hook tests.

## Best Possible Solution

Keep the narrow hook direction only after maintainers choose the terminal payload/access contract, then document the public API, add the changelog entry, and validate the focused hook/failover paths.

## Reproduction Assessment

Yes. Source and PR-diff inspection reproduces the concern: the PR emits `finalMessage: message`, current main gates only existing conversation hooks, and fallback summaries are built from per-attempt provider error text.

## Solution Assessment

No, not as currently written. The hook API is a reasonable seam, but the terminal event should either be conversation-gated or reduced to a sanitized/minimal payload before becoming public.
