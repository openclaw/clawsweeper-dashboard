## Summary

Keep open: the feature direction and supplied live WhatsApp proof are useful, but latest head still has blocking routing/data-loss defects plus generated-artifact and documentation/changelog cleanup before merge.

## What This Changes

The PR adds native WhatsApp location sends through message.send, Gateway send params, outbound/action routing, protocol models, CLI flags, and focused tests.

## Best Possible Solution

Land one coordinated native location implementation after routing every fallback through registered Gateway surfaces, rejecting ambiguous payloads consistently, and updating the documented user-facing contract.

## Reproduction Assessment

Yes for the PR defects: source inspection of latest head shows the core fallback calling an unregistered `location` RPC and the action-runner `send` path dropping mixed message/media fields when coordinates are present. Current-main reproduction is not applicable because this PR adds the location-send capability.

## Solution Assessment

No. The feature direction is reasonable, but the patch should use the existing registered `send` RPC for core fallback location delivery, reject mixed location/text/media payloads in every entry path, and keep generated/docs/changelog surfaces aligned before merge.

Overall confidence: 0.93

  - body: `sendLocation()` still dispatches Gateway method `location`, but the Gateway exposes location delivery under the existing `send` handler and does not register or scope a standalone `location` method.   - confidence: 0.97

Tool and CLI callers can get a successful location send while their accompanying text/media is silently dropped; reject the mixed payload here too, matching the Gateway send handler.
  - confidence: 0.91
