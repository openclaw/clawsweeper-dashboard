## Summary

Keep open: the revision addresses the earlier duplicate-user risk, but the embedded assistant dedupe can still skip a needed gap-fill when the current reply repeats an older assistant turn, and live behavior proof is still missing.

## What This Changes

The PR widens agent-command transcript persistence and compaction to embedded runner results, adds an embedded assistant-only gap-fill with dedupe, updates tests, and adds a changelog entry.

## Best Possible Solution

Keep the embedded assistant-only approach, but make dedupe prove the matching assistant is the current transcript tail or current-turn entry, add repeated-reply regression coverage, and then provide live embedded-session proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence path: embedded runs are marked "embedded" while current main only mirrors "cli" transcripts; no live WebChat or Feishu reproduction was run here.

## Solution Assessment

Is this the best way to solve the issue? No; the assistant-only gap-fill direction is appropriate, but the dedupe must be tied to the actual transcript tail/current turn rather than any older matching assistant text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require the dedupe match to be the transcript tail:** `src/agents/command/attempt-execution.ts:231-232`
  - body: In embedded gap-fill mode this check uses `readLatestAssistantTextFromSessionTranscript()`, which skips backward over newer user/tool rows until it finds any assistant. If the current embedded turn persisted its user/tool rows but lost an assistant reply whose text repeats an older answer, this branch treats the older assistant as already persisted and skips the gap-fill, leaving the reported missing assistant entry unfixed. Please only suppress the append when the matching assistant is actually the current transcript tail or current turn.
