## Summary

Keep open: current main still has the Discord announce account-routing gap, and open PR #73975 already targets this issue with closing syntax. The resolver still returns the Discord key-derived fallback before session hydration when the channel metadata lacks the opt-in flag, so the stored session account identity is not recovered on this path.

## What This Changes

The issue requests restoring Discord A2A announce delivery so it uses the target session's stored accountId instead of falling back to the default account.

## Best Possible Solution

Review and land PR #73975, or an equivalent narrow Discord-owned change, so Discord announce target resolution opts into session hydration and regression tests prove deliveryContext.accountId or lastAccountId reaches the final gateway send.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Discord session key resolves to a fallback target without accountId, Discord metadata does not request session lookup, and the A2A send forwards that missing accountId.

## Solution Assessment

Yes. The narrowest maintainable direction is the existing Discord-specific opt-in plus regression coverage, rather than changing announce lookup defaults for every channel.
