## Summary

Keep open. Current main still has only bundled, plugin, managed-user, and workspace hook sources, and hook loading/disable behavior remains controlled by user configuration. This is not a ClawHub-only request because OS-owned discovery roots, disable immunity, policy-first ordering, and audit-visible inventory require core trust-boundary design.

## What This Changes

Request to add an admin-managed machine-wide hook policy tier that loads before user-controlled hooks, cannot be disabled by user config or CLI toggles, and exposes policy-aware CLI inventory.

## Best Possible Solution

Define a maintainer-approved core policy-hook architecture, then implement hardened POSIX and Windows discovery, a policy source type with deterministic highest precedence, disable immunity, policy-aware list/info output, fail-open handler error handling with honored deny decisions, docs, changelog, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing capability is high-confidence from current source and docs because all internal hook discovery and enablement paths remain bundled, plugin, managed-user, or workspace scoped and user-config gated.

## Solution Assessment

Unclear until maintainers settle the security/product design. If OpenClaw accepts mandatory fleet governance hooks, the core hook loader, dispatcher, config schema, and hooks CLI are the narrowest maintainable layer because current plugin and hook-pack surfaces cannot enforce OS-owned roots or user-disable immunity.
