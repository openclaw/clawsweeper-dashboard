Keep open: current main contains the earlier instructions fix, but the reported local-only Codex failure is still plausible because the local probe path uses a different simple-completion transport/request shape than the gateway path that succeeds.

## What This Changes

The issue reports that local `openai-codex/gpt-5.5` and `gpt-5.4` model probes on v2026.5.7 return a generic no-text error while gateway and agent paths succeed.

## Best Possible Solution

Align the local Codex model-probe path with the gateway/OpenClaw Codex Responses request contract, then prove `--local` returns text for `openai-codex/gpt-5.5` without changing non-Codex probes.

## Reproduction Assessment

No, not as a live current-main run in this checkout. The reporter gives a concrete release repro and current source shows the local Codex path still differs from the working gateway path, so it is source-reproducible but not live-reproduced here.

## Solution Assessment

Yes, the best fix direction is narrow: keep `infer model run --local`, but make `openai-codex-responses` local probes use the same sanitized Codex Responses request semantics as the successful runtime path or an equivalent provider-owned simple-completion seam.
