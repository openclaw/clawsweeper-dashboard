## Summary

Keep open. The NUL-prefix echo bug is still valid on current main, but this PR should not merge as-is because it strips NUL bytes from every position and needs to be composed with the current leading-only corruption-marker normalizer plus a changelog entry.

## What This Changes

The PR changes iMessage sent-message echo-cache normalization to strip NUL bytes and adds regression tests for single and repeated leading NUL-prefixed echoes.

## Best Possible Solution

Repair this as a narrow iMessage plugin fix by adding U+0000 to the existing leading corruption-marker normalization, adding regression tests for one and multiple leading NUL bytes plus an interior-NUL non-match, and adding a single-line changelog fix entry while leaving TTL, message-id matching, echo scope, and unrelated channel logic unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main caches clean outbound text, but normalizeEchoTextKey does not strip a leading U+0000 from the reflected inbound text, so cache.has misses for the reported NUL-prefixed echo path.

## Solution Assessment

No, not as currently proposed. The narrow maintainable fix is to compose with current main's leading-only marker normalizer instead of deleting every NUL byte, because global deletion can collapse distinct message bodies inside the echo TTL.
