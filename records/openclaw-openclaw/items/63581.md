## Summary

Keep open. Current main still misses the NUL-prefix echo-cache case, but this PR removes NUL bytes globally and should be narrowed before merge so interior text remains distinguishable.

## What This Changes

The PR changes iMessage sent-message echo-cache normalization to strip NUL bytes and adds regression coverage for NUL-prefixed echoed text.

## Best Possible Solution

Keep this as a narrow iMessage plugin fix by adding U+0000 to the existing leading-only corruption-marker normalization, adding prefix and interior-NUL regression coverage, and documenting the user-visible fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main caches clean outbound text, but a reflected inbound body prefixed with U+0000 is not normalized to the same key, so `cache.has()` misses on the text fallback path.

## Solution Assessment

No. The current PR addresses the reported prefix miss but is not the best form because global NUL deletion can collapse distinct message bodies inside the echo TTL; leading-only marker normalization is the narrower maintainable solution.
