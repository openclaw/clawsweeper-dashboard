## Summary

Keep open. Current main and the latest release still lack U+0000 handling in iMessage echo-cache text normalization, so the central fix remains useful; the branch needs repair before merge because it strips interior NUL bytes and does not compose with the now-merged leading-marker normalizer.

## What This Changes

The PR changes iMessage echo-cache text normalization to remove NUL bytes before trimming and adds tests for single and repeated leading NUL-prefixed echoes.

## Best Possible Solution

Repair this as a narrow iMessage plugin fix by adding U+0000 to the existing leading corruption-marker normalization, adding regression tests for one and multiple leading NUL bytes plus an interior-NUL non-match, and leaving echo scope, TTL, message-ID behavior, and unrelated channel logic unchanged.

Yes. ## Solution Assessment
