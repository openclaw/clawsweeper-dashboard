## Summary

Keep open: current main still lets accumulated replay state from earlier tool activity flow into the failed attempt, and the silent-error retry helper rejects attempts whose replay metadata has `hadPotentialSideEffects`. The latest release has the same relevant shape, and the related issues cover adjacent retry or diagnostic gaps rather than this exact prior-side-effect blocker.

## What This Changes

Fix the embedded runner so transient empty/non-visible provider errors still retry after prior session tool calls when the failed current turn itself had no side effects.

## Best Possible Solution

Keep this issue open and change the retry/replay-state boundary so empty/non-visible provider-error retries are blocked by side effects from the failed current turn, not by unrelated earlier session tool history.

## Reproduction Assessment

Yes at source level: current main still passes accumulated replay state into each attempt, and the silent-error retry rejects attempts whose replay metadata reports prior side effects. I did not run a live OpenRouter 5xx reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is to separate current-attempt side-effect evidence from accumulated replay safety for this retry path, preserving the guard for side effects in the failed turn itself.
