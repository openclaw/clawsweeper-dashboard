## Summary

This PR should stay open: the feature is plausible, but the current implementation bypasses the existing subagent session-visibility scope and the no-argument listing can miss existing named sessions. It also lacks after-fix proof from a real Gateway/channel setup, so it is not merge-ready.

## What This Changes

The PR adds a `/resume [session-label]` Gateway chat command, shared binding helper code, focused command tests, and slash-command documentation for named session resume.

## Best Possible Solution

Keep the PR open and make `/resume` reuse the existing `/focus` visibility boundary for both list and resolve, fix bounded named-session listing, update focused tests, and add real channel proof before merge.

## Reproduction Assessment

Yes. The review findings are source-reproducible: the PR calls `sessions.list`/`sessions.resolve` without the existing `spawnedBy` subagent visibility scope, and it filters named sessions after a capped list response.

## Solution Assessment

No. The maintainable path is to share the current `/focus` controller/visibility scoping and make the named-session list bounded by named rows rather than by all recent sessions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope `/resume` lookups to visible child sessions:** `src/auto-reply/reply/commands-subagents/action-resume.ts:104-107`
  - body: `/resume <label>` resolves with only `{ label, agentId }`, unlike `/focus`, which passes `spawnedBy` for subagent requesters before `sessions.resolve`. A subagent with child-control scope can therefore bind to any same-agent labeled session, and the no-arg list has the same visibility gap for labels/previews; compute the controller before list/resolve and apply the same requester-visible scope.
  - confidence: 0.88
- **[P2] Return named sessions before applying the 100-row cap:** `src/auto-reply/reply/commands-subagents/action-resume.ts:51`
  - body: `sessions.list` applies the requested limit before returning rows, but this code filters out unnamed sessions only after receiving the newest 100. If those rows are unnamed, `/resume` can say there are no named sessions even when older named sessions exist; page until enough named rows are found or add a server-side named-session filter.
  - confidence: 0.84
