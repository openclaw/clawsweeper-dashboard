## Summary

Keep open. Current main still lacks Active Memory-owned configuration or a reservation policy for splitting recall between memory and wiki results; the runtime has only a final `maxSummaryChars` summary cap plus shared `memory_search corpus=all` result sorting/slicing, so the reported wiki starvation path remains plausible and actionable.

## What This Changes

Request configurable Active Memory recall allocation for memory result count, memory snippet length, wiki result count, and wiki character budget, or an equivalent proportional budget split.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the backward-compatible final injected-summary cap, and updates config schema, manifest hints, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A focused mocked test can register memory hits plus memory-wiki supplement hits, call `memory_search` with `corpus=all` and a small `maxResults`, and observe wiki hits being dropped after the combined score sort/slice; static source inspection confirms that path on current main.

## Solution Assessment

Unclear. The proposed knobs would address the operator need, but the narrowest maintainable fix may be a balanced or reserved per-source recall policy with optional caps rather than exposing every internal snippet and wiki formatting budget directly.
