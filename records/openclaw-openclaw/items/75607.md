## Summary

Keep open. Current main still lacks Active Memory-owned recall allocation controls or a wiki reservation; the only exposed Active Memory cap is the final `maxSummaryChars`, while the shared memory/wiki paths still merge sources by score and slice under one result cap. The existing ClawSweeper keep-open comment remains consistent with current main.

## What This Changes

The issue requests configurable Active Memory recall allocation so memory snippets and wiki/vector search results do not compete under one hardcoded summary budget.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and updates config schema, manifest UI hints, docs, focused tests, and changelog together.

## Reproduction Assessment

Yes. A focused mocked test can register memory hits plus memory-wiki supplement hits, execute `memory_search` with `corpus=all` and a small `maxResults`, and observe wiki hits being dropped by the combined score sort/slice; current source confirms that path.

## Solution Assessment

Unclear. The proposed knobs would address the operator need, but the narrowest maintainable fix may be an additive balanced recall policy with optional per-source caps rather than exposing every internal snippet and wiki formatting budget directly.

Fix the Active Memory recall allocation issue reported at https://github.com/openclaw/openclaw/issues/75607. Current main only exposes `maxSummaryChars`, while `memory_search` with `corpus=all` merges memory hits and memory-wiki supplement hits, sorts by score, and slices once, allowing memory hits to crowd out wiki context.
