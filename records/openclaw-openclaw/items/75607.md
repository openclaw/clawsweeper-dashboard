## Summary

Keep open. Current main still has only a final Active Memory summary cap and the shared memory/wiki search path still sorts and slices both corpora under one result limit, so wiki supplement hits can still be crowded out before the capped recall summary is injected.

## What This Changes

Request to make Active Memory’s memory/wiki recall allocation configurable, including memory result count, snippet length, wiki top-k, and wiki formatting budget.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are available, keeps maxSummaryChars as the final injected-summary cap, and documents minimal optional tuning fields.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits, return higher-scored memory hits through memory_search with corpus=all and a small maxResults, and assert that the current combined sort/slice path drops wiki coverage.

## Solution Assessment

Yes in direction, but the safest implementation is a narrow plugin-owned split or reservation policy with shared memory-core support only where needed, not exposing every low-level formatter constant as public API.
