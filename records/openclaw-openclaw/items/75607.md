## Summary

Keep open. Current main still exposes only the total Active Memory summary cap, and the shared `memory_search corpus=all` path can sort and slice combined memory/wiki results before Active Memory produces its bounded summary, so the requested split or reservation policy is still missing.

## What This Changes

Request to make Active Memory's memory/wiki recall allocation configurable, or otherwise reserve budget so wiki results are not crowded out by memory hits.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements exist, exposes only the minimal documented tuning fields needed, and keeps `maxSummaryChars` as the final injected-summary cap.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register wiki supplement hits, return higher-scored memory hits through `memory_search corpus=all` with a small result limit, and the combined sort/slice path can drop wiki coverage before Active Memory summarizes.

## Solution Assessment

Unclear until maintainers choose the public surface. A split or reservation policy is the right direction, but exposing every formatter constant would likely be broader than necessary.
