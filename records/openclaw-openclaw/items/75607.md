## Summary

Keep open. Current main still lacks the requested Active Memory recall budget split: there are no `maxMemoryResults`, `memorySnippetChars`, `wikiTopK`, or `wikiMaxChars` config fields, Active Memory only exposes the final `maxSummaryChars` cap plus adjacent QMD search-mode tuning, and the shared `memory_search corpus=all` path still merges memory and wiki supplement hits under one result cap instead of reserving wiki coverage.

## What This Changes

The issue requests configurable Active Memory recall budgeting so memory snippets and wiki/vector results do not crowd out wiki context under one shared summary/result budget.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that deterministically preserves wiki coverage when wiki supplements are enabled, while keeping existing global `memory.qmd.limits.*` behavior intact and documenting the new operator-facing tuning path.

## Reproduction Assessment

No high-confidence live reproduction was run. Static inspection does confirm the requested knobs are absent and that the current shared `memory_search corpus=all` code can drop wiki hits after score sorting and a single combined result cap.

## Solution Assessment

Unclear. The proposed knobs are plausible, but the narrowest maintainable fix may be an Active Memory scoped balanced retrieval policy or tool-call default rather than exposing every low-level per-source field directly.
