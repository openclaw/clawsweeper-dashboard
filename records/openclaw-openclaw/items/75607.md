## Summary

Keep open. Current main still lacks configurable Active Memory recall allocation or a wiki reservation, and the shared memory/wiki search path still applies one combined score sort and result cap rather than preserving per-source coverage.

## What This Changes

Request to make Active Memory recall allocation configurable for memory result count, memory snippet length, wiki result count, and wiki formatting budget, or otherwise split the recall budget proportionally.

## Best Possible Solution

Add a backward-compatible Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps maxSummaryChars as the final injected-summary cap, and documents the supported tuning knobs across config schema, manifest UI hints, docs, tests, and changelog.

## Reproduction Assessment

Yes. A focused mocked test can register memory hits plus a memory-wiki supplement, call memory_search with corpus=all and a small maxResults, and observe lower-scored wiki hits being dropped by the combined sort/slice path; static source inspection confirms that path on current main.

## Solution Assessment

Unclear. Configurability is the right direction, but the narrowest maintainable fix may be an additive balanced/reserved recall policy with optional per-source caps rather than exposing every internal snippet and wiki formatting budget directly.
