## Summary

Keep open. Current main still does not expose the requested Active Memory recall budget split knobs, and the shared `memory_search` path can still combine memory and wiki supplement hits under one sort and result cap, so wiki coverage is not deterministically preserved when memory hits dominate.

## What This Changes

The issue requests configurable Active Memory recall budget allocation between memory snippets and wiki/vector search context.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps existing configs backward compatible, and documents the operator tuning path across plugin schema, docs, tests, and changelog.

## Reproduction Assessment

Yes. A focused mocked test can register several higher-scored memory hits plus lower-scored wiki supplement hits, call `memory_search` with `corpus: "all"` and a small `maxResults`, and observe wiki hits being dropped after the current combined sort/slice path.

## Solution Assessment

Unclear. The proposed knobs are plausible, but the narrowest maintainable fix may be a deterministic reserved/balanced recall policy with optional config rather than directly exposing every internal low-level field.
