## Summary

Keep open. Current main still exposes only the final Active Memory summary cap, while shared `memory_search corpus=all` merges memory and wiki supplement hits and slices the combined list by score, so wiki coverage can still be crowded out before Active Memory injects the bounded summary.

## What This Changes

Request configurable Active Memory recall allocation between regular memory and Memory Wiki results, including result count, snippet length, wiki top-k, or wiki character-budget tuning.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements exist, keeps `maxSummaryChars` as the final injected-summary cap, and documents only the minimal tuning fields needed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: register wiki supplement hits, return higher-scored memory hits through `memory_search corpus=all` with a small result limit, and assert the current combined sort/slice path drops wiki coverage.

## Solution Assessment

Yes in direction. The best fix is a narrow plugin-owned split or reservation policy with generic memory-core support only where needed, rather than exposing every low-level formatter constant as public configuration.
