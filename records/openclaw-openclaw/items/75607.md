## Summary

Keep open. Current main still lacks Active Memory allocation knobs or a reservation policy; it exposes only the final summary cap, while shared `memory_search corpus=all` still sorts memory and wiki supplement hits together and slices the combined list, so wiki coverage can be crowded out under constrained result budgets.

## What This Changes

Request configurable Active Memory recall allocation between memory snippets and wiki/vectordb results.

## Best Possible Solution

Add an Active Memory-owned recall allocation or reservation policy that preserves wiki coverage when wiki supplements exist, keeps `maxSummaryChars` as the final injected-summary cap, and documents only the minimal tuning fields needed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register wiki supplement hits, return higher-scored memory hits through `memory_search corpus=all` with a small result limit, and the current combined sort/slice path can drop wiki coverage before Active Memory injects the bounded summary.

## Solution Assessment

Yes in direction, but the exact knobs should stay narrow. The maintainable fix belongs in the Active Memory/memory plugin boundary, with generic memory-core support only where needed rather than exposing all internal formatter constants.
