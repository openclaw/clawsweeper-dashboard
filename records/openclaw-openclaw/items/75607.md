## Summary

Keep open. Current main has adjacent QMD/global memory-search limits, but the active-memory plugin still exposes only a single `maxSummaryChars` summary cap plus recent-turn and QMD search-mode tuning; it does not expose the requested per-source recall split or deterministic wiki reservation, and the shared `memory_search` path merges memory and wiki hits under one result cap.

## What This Changes

Request to make Active Memory’s recall budget split among memory and wiki results configurable, including memory result count, memory snippet length, wiki top-K, and wiki max characters.

## Best Possible Solution

Add an additive, plugin-owned tuning path that either reserves wiki coverage during Active Memory recall or exposes a documented balanced shared-search policy, while preserving existing `memory.qmd.limits` behavior for normal memory search.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does confirm the requested config knobs are absent and that shared memory/wiki results are currently merged under one cap when `memory_search corpus=all` is used.

## Solution Assessment

Unclear. The proposed knobs are plausible, but the narrowest maintainable fix may be an active-memory scoped balanced `memory_search`/corpus policy rather than exposing every low-level per-source field directly.
