## Summary

Keep open. Current main still lacks Active Memory recall split controls: the plugin exposes only a final summary cap and QMD search-mode tuning, while the shared memory/wiki search path still merges both sources under one sorted result cap with no wiki reservation.

## What This Changes

Request to make Active Memory's memory/wiki recall allocation configurable or balanced so wiki context is not starved by memory snippets under the active-memory summary budget.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps existing memory-core behavior compatible for normal `memory_search`, and documents the operator-facing tuning path.

## Reproduction Assessment

Yes for the source-level failure mode: a targeted test can register memory and wiki supplement hits, call `memory_search` with `corpus=all` and a small cap, and observe wiki hits compete in the same sorted/sliced result set. A full live reproduction with a real wiki vault was not run.

## Solution Assessment

Unclear whether exposing every proposed low-level knob is the best API. The narrowest maintainable fix is likely an Active Memory scoped balanced retrieval policy with additive config fields only where operators genuinely need tuning.
