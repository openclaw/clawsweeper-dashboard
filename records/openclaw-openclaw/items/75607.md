## Summary

Keep open. Current main still exposes only a final Active Memory summary cap, while shared `memory_search corpus=all` still merges memory and wiki supplement hits under one sorted result limit, so wiki coverage can still be crowded out before the active-memory summary is injected.

## What This Changes

Request configurable Active Memory recall allocation for memory result count, snippet size, wiki top-k, and wiki character budget so memory and wiki results both survive the summary cap.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements are available, keeps `maxSummaryChars` as the final injection cap, and exposes only minimal documented tuning fields needed by operators.

## Reproduction Assessment

Yes. A focused mocked test can make `memory_search corpus=all` return enough higher-scored memory hits plus valid wiki supplement hits, then assert the current combined sort/slice path drops wiki coverage before Active Memory summarizes under `maxSummaryChars`.

## Solution Assessment

Yes in direction, but the safest fix is a narrow source split or reservation policy with generic memory-core supplement support, not a broad set of low-level formatter constants unless maintainers choose that API.
