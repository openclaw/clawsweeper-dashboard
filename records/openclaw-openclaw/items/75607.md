## Summary

Keep open. Current main still exposes only a final Active Memory summary cap and shared memory/wiki results are still sorted and capped together, so wiki context can be crowded out under the reported default budget.

## What This Changes

Request to make Active Memory's memory/wiki recall allocation configurable or balanced so wiki hits are not crowded out under the final summary budget.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements are enabled, keeps maxSummaryChars as the final injected-summary cap, and documents minimal optional tuning.

## Reproduction Assessment

Yes. A focused mocked test can return higher-scored memory hits plus lower-scored wiki supplement hits through memory_search with corpus=all and a small maxResults, then observe wiki hits dropped by the combined sort/slice path.

## Solution Assessment

Yes in direction, with a narrower maintainable implementation. Prefer a balanced default or source reservation with minimal config over directly exposing every internal formatter budget unless tests show each knob is necessary.
