## Summary

Keep open. Current main still lacks an Active Memory recall budget split or per-source recall knobs, and the shared memory/wiki search path can still drop wiki supplement hits after a single score sort and combined result cap.

## What This Changes

Request configurable Active Memory recall allocation between memory snippets and wiki supplement results.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that deterministically preserves wiki coverage when wiki supplements are enabled, keeps existing global memory and QMD limits compatible, and documents the operator tuning path without exposing unnecessary low-level internals.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits plus memory hits, call memory_search with corpus: "all" and a small maxResults, and verify that lower-scored wiki hits are dropped after the combined score sort and cap; static source inspection confirms that path.

## Solution Assessment

Unclear. The requested config keys are reasonable, but the narrowest maintainable solution may be a balanced/reserved recall policy with optional Active Memory config rather than exposing every internal budget field directly.
