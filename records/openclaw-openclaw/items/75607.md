## Summary

Keep open. Current main still lacks an Active Memory recall budget split or per-source recall knobs, and the shared memory/wiki search path can still drop wiki supplement hits after a single score sort and combined result cap.

## What This Changes

Request to make Active Memory recall allocation configurable or balanced across memory results, memory snippet length, wiki result count, and wiki formatting budget.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that deterministically preserves wiki coverage when wiki supplements are enabled, while keeping existing global memory/QMD limits compatible and documenting the operator tuning path.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits plus memory hits, call `memory_search` with `corpus: "all"` and a small `maxResults`, and observe wiki hits being dropped after the combined sort/slice; static source inspection confirms that path.

## Solution Assessment

Unclear. The proposed knobs are plausible, but the narrowest maintainable fix may be a balanced Active Memory recall policy with optional config rather than exposing every internal low-level field directly.
