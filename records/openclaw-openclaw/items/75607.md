## Summary

Keep open. Current main still exposes only a final Active Memory summary cap while shared memory/wiki recall can sort and slice combined results under one result limit, so wiki coverage can still be crowded out before the injected summary is built.

## What This Changes

Request to make Active Memory recall allocation configurable or otherwise reserve budget so wiki recall is not crowded out by memory snippets under the shared summary cap.

## Best Possible Solution

Add an Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements are available, keeps maxSummaryChars as the final injected-summary cap, and documents minimal optional tuning fields.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits, return enough higher-scored memory hits through memory_search with corpus=all and a small maxResults, and assert that the current combined sort/slice path drops wiki coverage.

## Solution Assessment

Yes in direction, with one caveat: the safest repair is a narrow plugin-owned split or reservation policy with a generic memory-core seam only where needed, rather than exposing every formatter constant as public API.
