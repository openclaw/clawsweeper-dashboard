## Summary

Keep open. Current main still lacks an Active Memory recall-allocation config or reservation policy, and source inspection shows wiki supplement hits can still be crowded out by the combined `memory_search corpus=all` ranking path before the final summary cap is applied.

## What This Changes

Expose a configurable Active Memory recall allocation so memory snippets and wiki supplement hits share the recall budget without wiki results being crowded out.

## Best Possible Solution

Add an Active Memory-owned source allocation or reservation policy that keeps wiki supplements visible under shared recall, expose only the minimal tuning surface needed, and update docs, tests, and changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register wiki supplement hits, return higher-scored memory hits through `memory_search corpus=all` with a small result limit, and the current combined sort/slice can drop wiki coverage before Active Memory applies `maxSummaryChars`.

## Solution Assessment

Yes in direction, but the exact proposed knobs are not necessarily the best API. A narrow plugin-owned allocation or reservation policy is safer than exposing all low-level formatter constants as public config.
