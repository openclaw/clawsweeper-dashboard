## Summary

Keep open. Current main at 61fc62a still exposes only the final Active Memory `maxSummaryChars` cap, while the shared `memory_search` `corpus=all` path still merges memory and wiki hits by score under one result cap, so wiki coverage can be crowded out before the recall summary is produced.

## What This Changes

Make Active Memory's memory/wiki recall allocation configurable or otherwise balanced so wiki results still contribute under the summary cap.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and documents minimal optional tuning config.

## Reproduction Assessment

Yes. A focused mocked test can register a memory-wiki supplement, return higher-scored memory hits, call `memory_search` with `corpus: "all"` and a small `maxResults`, and observe wiki hits dropped by the combined sort/slice before Active Memory can summarize them.

## Solution Assessment

Unclear for the exact proposed knob set. A balanced per-source default plus a small set of optional caps is safer than making all low-level formatter budgets public API.
