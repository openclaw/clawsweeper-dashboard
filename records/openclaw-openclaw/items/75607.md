## Summary

Keep open. Current main still exposes only the final Active Memory `maxSummaryChars` cap, while shared memory/wiki recall still sorts and slices combined hits under one result cap, so wiki results can still be crowded out before the injected recall summary is produced.

## What This Changes

The issue requests configurable Active Memory recall allocation so memory snippets do not crowd out wiki/vector search context under the final summary cap.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and documents a small optional tuning surface.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits, return higher-scored memory hits through `memory_search` with `corpus=all` and a small `maxResults`, then observe the wiki hits dropped by the combined sort/slice path.

## Solution Assessment

Yes, if solved as a narrow plugin-owned policy with shared memory-core support only where needed. A balanced per-source default plus a small set of optional caps is safer than making all low-level formatter budgets public API.
