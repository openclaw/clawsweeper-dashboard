## Summary

Keep open. Current `main` still exposes only the final Active Memory `maxSummaryChars` cap and still merges memory/wiki results under one `maxResults` slice, so wiki supplement hits can be crowded out before the recall summary is produced.

## What This Changes

Request configurable Active Memory recall allocation so memory result count, memory snippet size, wiki top-k, and wiki formatting budget do not starve wiki context under the summary cap.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and documents only the smallest useful optional tuning knobs.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits, return higher-scored memory hits through `memory_search` with `corpus=all` and a small `maxResults`, then observe the wiki hits dropped by the combined sort/slice path.

## Solution Assessment

Unclear as proposed. A balanced default or per-source reservation with a few optional caps is safer than exposing every low-level formatter budget as public plugin config.
