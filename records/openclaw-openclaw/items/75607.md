## Summary

Keep open. Current main still has no configurable Active Memory memory/wiki recall allocation, and the relevant memory/wiki retrieval paths still share one score sort and result cap, so the reported tuning gap remains actionable rather than implemented or obsolete.

## What This Changes

Add configurable Active Memory recall allocation or sizing so memory and wiki search results can both contribute to the injected recall summary.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps maxSummaryChars as the final injected-summary cap, and updates config schema, manifest UI hints, docs, focused tests, and changelog together.

## Reproduction Assessment

Yes. A focused mocked test can register higher-scored memory hits plus lower-scored wiki supplement hits, call memory_search with corpus=all and a small maxResults, and observe wiki hits dropped by the combined sort/slice path shown on current main.

## Solution Assessment

Unclear. The requested knobs would solve operator tuning, but the narrowest maintainable fix may be a balanced per-source recall policy with optional stable caps rather than exposing every internal snippet and wiki formatting budget directly.
