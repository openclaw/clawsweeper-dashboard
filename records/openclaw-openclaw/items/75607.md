## Summary

Keep open. Current main at 3980eaa still exposes only the final `maxSummaryChars` cap for Active Memory, and the memory/wiki search path still merges sources by score under one result cap, so wiki results can be crowded out before the recall summary is produced.

## What This Changes

Request to make Active Memory recall allocation between memory search and wiki results configurable or balanced instead of letting memory results fill the shared summary budget.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and documents minimal optional tuning config.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits and return higher-scored memory hits through `memory_search` with `corpus=all` and a small `maxResults`, then observe the wiki hits dropped by the combined sort/slice path.

## Solution Assessment

Yes, if solved as a narrow plugin-owned policy with shared memory-core support only where needed. A balanced per-source default plus a small set of optional caps is safer than making all low-level formatter budgets public API.
