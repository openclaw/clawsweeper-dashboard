## Summary

Keep open. Current main still has no configurable Active Memory memory/wiki recall allocation, and the shared `memory_search corpus=all` path can still let one source family crowd out the other before the Active Memory summary is injected.

## What This Changes

The issue requests configurable Active Memory recall allocation knobs for memory result count, memory snippet length, wiki top-k, and wiki character budget, or an equivalent proportional split.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps `maxSummaryChars` as the final injected-summary cap, and updates config schema, manifest UI hints, docs, focused tests, and changelog together.

## Reproduction Assessment

Yes. Static inspection shows `memory_search corpus=all` merges memory and wiki supplement results under one score sort and cap, while Active Memory exposes only `maxSummaryChars`; a focused mocked test can seed higher-scored memory hits plus lower-scored wiki hits and observe the wiki results dropped.

## Solution Assessment

Unclear. The requested knobs would solve operator tuning, but the narrowest maintainable fix may be a balanced per-source recall policy with additive optional caps rather than directly exposing every internal snippet and wiki formatting budget.
