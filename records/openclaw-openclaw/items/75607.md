## Summary

Keep open. Current main still lacks configurable Active Memory recall allocation or a wiki reservation, and the shared memory/wiki search path still merges both sources under one combined score sort and result cap, matching the prior ClawSweeper keep-open discussion rather than making this obsolete.

## What This Changes

Request to make Active Memory's recall budget split between memory results and wiki results configurable, or otherwise reserve a fair share for wiki context under the existing summary cap.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage whenever wiki supplements are enabled, keeps maxSummaryChars backward-compatible as the final injected-summary cap, and documents the supported tuning knobs across config schema, manifest UI hints, docs, tests, and changelog.

## Reproduction Assessment

Yes. A focused mocked test can register memory hits plus memory-wiki supplement hits, call memory_search with corpus=all and a small maxResults, and observe wiki hits being dropped after the combined sort/slice; static source inspection confirms that path on current main.

## Solution Assessment

Unclear. The proposed knobs would address the operator need, but the narrowest maintainable fix may be an additive balanced recall policy with optional per-source caps rather than exposing every internal snippet and wiki formatting budget directly.
