## Summary

Keep open. Current main still lacks Active Memory-owned recall allocation knobs or a reservation policy, and the shared memory/wiki search paths still merge sources under one score sort and result cap, so wiki context can be crowded out by memory hits.

## What This Changes

The issue requests configurable or balanced Active Memory recall allocation between memory results and wiki supplement context.

## Best Possible Solution

Add an Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps maxSummaryChars as the final injected-summary cap, and documents minimal optional tuning config.

## Reproduction Assessment

Yes. A focused mocked test can return higher-scored memory hits plus lower-scored wiki supplement hits through memory_search with corpus=all and a small maxResults, then observe wiki hits dropped after the combined sort/slice path shown in current source.

## Solution Assessment

Yes, if implemented as a narrow plugin-owned policy with generic memory-core support where needed. Directly exposing every low-level formatter budget may be less maintainable than a balanced per-source default plus optional caps.
