## Summary

Keep open. Current main still lacks configurable Active Memory recall allocation or a deterministic wiki reservation, and the shared `memory_search` path can still merge memory hits and wiki supplement hits under one score sort and one result cap.

## What This Changes

Add configurable or balanced Active Memory recall allocation so memory results do not starve wiki corpus results under the shared recall budget.

## Best Possible Solution

Add an additive Active Memory-owned recall policy that preserves wiki coverage when wiki supplements are enabled, keeps existing `maxSummaryChars` behavior compatible, and documents operator tuning in the manifest, docs, and changelog.

## Reproduction Assessment

Yes. A focused mocked test can register wiki supplement hits plus higher-scored memory hits, call `memory_search` with `corpus: "all"` and a small `maxResults`, and observe wiki hits being dropped after the combined sort/slice path.

## Solution Assessment

Unclear. The proposed knobs are plausible, but the narrowest maintainable fix may be a balanced or reserved per-source recall policy with optional config rather than exposing every low-level internal field directly.

## Repair Work Prompt

Fix the Active Memory recall starvation reported in https://github.com/openclaw/openclaw/issues/75607. Current main lets the hidden Active Memory sub-agent reach wiki content only through shared `memory_search`/`memory_get`; `memory_search corpus=all` merges memory hits and wiki supplement hits, sorts once, and slices once by `maxResults`, so memory hits can starve wiki coverage. Implement a narrow additive Active Memory-owned recall allocation policy that preserves wiki coverage when wiki supplements are enabled. Prefer a maintainable balanced/reserved per-source policy with optional config over exposing brittle internals unless the existing config patterns make explicit fields the clearest path. Keep backward compatibility for existing `maxSummaryChars`, do not move owner-specific wiki policy into generic core beyond the minimal shared seam needed, and do not change unrelated memory/dreaming behavior. Update the Active Memory manifest schema/UI hints, docs at `docs/concepts/active-memory.md`, and `CHANGELOG.md` with proper non-bot attribution. Add focused tests proving wiki hits survive memory pressure and that config normalization/schema accepts any new knobs.
