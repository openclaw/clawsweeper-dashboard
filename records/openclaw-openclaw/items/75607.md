## Summary

Keep open. Current main still lacks Active Memory configuration for memory/wiki recall allocation, and the shared memory/wiki search path still applies one combined score sort and result cap that can crowd out wiki supplement hits.

## What This Changes

Make Active Memory's memory/wiki recall allocation configurable or balanced so wiki results are preserved under the active-memory summary budget.

## Best Possible Solution

Add a default-compatible Active Memory recall policy that reserves wiki coverage when wiki supplements are enabled, with any memory-core support kept generic and config schema, manifest UI hints, docs, tests, and changelog updated together.

## Reproduction Assessment

Yes. A focused mocked test can seed higher-scored memory hits plus lower-scored wiki supplement hits, call memory_search with corpus=all and a small maxResults, and observe wiki hits dropped after the combined sort/slice path.

## Solution Assessment

Yes, if implemented as an Active Memory-owned, default-compatible policy with source reservation before the final maxSummaryChars truncation; a prompt-only instruction would not be deterministic enough.
