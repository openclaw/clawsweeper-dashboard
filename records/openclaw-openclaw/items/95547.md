## Summary

Keep open: current main and v2026.6.10 contain partial mitigation for replay-safe stdio Codex closes and Gateway restart recovery marking, but the reported long maintenance turn can still end as a generic Codex app-server client-close failure and the graceful restart/follow-up contract remains unsettled.

## What This Changes

The issue requests durable, user-visible recovery and clearer classification when long Codex-backed messaging turns are interrupted by app-server closure or Gateway SIGTERM/restart.

## Best Possible Solution

Define one maintainer-owned lifecycle contract that separates intentional Gateway shutdown from unexpected Codex app-server loss and guarantees a durable recovery, status, or confirmation message for long channel turns without replaying unsafe side effects.

## Reproduction Assessment

No live current-main reproduction was run. The issue is source-reproducible for the generic client-close classification and narrow retry boundary because current code still maps that path to generic failure text while restart recovery is separate and partial.

## Solution Assessment

No. Current main is only a partial mitigation: it retries a narrow replay-safe stdio case and separately marks restart-aborted sessions, but it does not fully classify intentional Gateway shutdown at the Codex failure site or guarantee a durable long-turn follow-up.
