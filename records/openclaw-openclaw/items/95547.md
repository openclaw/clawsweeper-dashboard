## Summary

Keep open: current main has narrow Codex client-close retry and Gateway restart-recovery pieces, but source and tests still show non-replay-safe client closes can surface the generic Codex error and the restart path remains a policy-sensitive recovery flow rather than the complete graceful maintenance-turn recovery requested.

## What This Changes

This issue requests graceful handling of long Codex-backed channel turns when the app-server closes or the Gateway restarts, including clearer interruption classification and useful recovery/follow-up messaging.

## Best Possible Solution

Define one Codex-backed channel recovery contract that classifies expected Gateway restart separately from unexpected app-server loss, resumes only replay-safe turns, and otherwise delivers a clear recovery or unresumable notice tied to the interrupted session.
