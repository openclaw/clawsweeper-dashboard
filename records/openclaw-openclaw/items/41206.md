## Summary

Keep this PR open, but it is not mergeable as-is. The directory-backed cron store is still absent from current main and this PR is the active implementation candidate for https://github.com/openclaw/openclaw/issues/20950, but the branch is conflicting, lacks real behavior proof, and has concrete data-integrity and filename-containment defects.

## What This Changes

The PR adds exported helpers and tests for an opt-in `cron/jobs/*.json` directory store with one JSON file per cron job while leaving the existing `jobs.json` store helpers in place.

## Best Possible Solution

Rework or replace this PR under https://github.com/openclaw/openclaw/issues/20950 with a backward-compatible directory-backed cron definition store that uses stable contained filenames, preserves `jobs-state.json`, wires Gateway/CronService add-update-remove paths, and updates schema, help, doctor, docs, and focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head writes filenames from mutable `job.name`, deletes by filename-derived id, and none of the new directory helpers are called by the current cron load/persist/add/update/remove paths. I did not run tests because this review was read-only.

## Solution Assessment

No. The maintainable solution needs an integrated storage contract and runtime wiring, not helper exports alone, and it must preserve the current definition/runtime state split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds the path from `job.name`, but names are mutable and not guaranteed unique while cron job ids are the stable identity. Two jobs with the same name, or a rename during update, can overwrite or leave duplicate persisted files.
  - confidence: 0.95
- **[P1] Delete files by the persisted job id:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files saved by this PR are named from the job name. A normal job such as id `brew-1` and name `Daily Brew` saves `daily-brew.json`, so delete by id returns false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory store into cron runtime:** `src/cron/store.ts:181-188`
  - body: The new directory-store functions are exported but no current cron startup, load, add, update, or remove path calls them. With this patch, `CronService` still loads and persists the aggregate `jobs.json`, so the advertised directory store is not active for real cron operations.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar contract:** `src/cron/store.ts:248`
  - body: The new writer serializes the full `CronJob`, including runtime fields, into the per-job definition file. Current main split runtime metadata into `jobs-state.json`, so a mergeable directory backend needs to keep `state` and `updatedAtMs` out of user-tracked definition files.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds local cron job file-writing code with a concrete filename containment flaw.

Concerns:

- **[medium] Constrain directory job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to the raw input when a name/id contains no alphanumeric characters. Values such as `../` can then be joined into the jobs path and write outside `cron/jobs/`, so the filename component should be generated from a constrained stable id and checked to remain inside the jobs directory.
