## Summary

Keep open. Current main still persists cron through `jobs.json` plus `jobs-state.json`, while this PR is not merge-ready because the new directory-store helpers are not wired into live Gateway/CLI persistence and introduce filename-keying, deletion, runtime-state, and path-safety problems.

## What This Changes

This PR adds exported helpers and tests for an optional `cron/jobs/` per-job JSON directory store while keeping the existing `jobs.json` store available.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: define activation and precedence, keep `jobs.json` compatibility, key files by stable collision-safe job IDs, preserve the runtime state sidecar or an equivalent non-config state store, wire Gateway/CLI/doctor/docs behavior, and cover migration, overlap, rename, delete, malformed filenames, symlinks, and sidecar semantics with focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a job with id `brew-1` and name `Daily Brew` is saved as `daily-brew.json`, while deletion by `brew-1` compares against the filename-derived `daily-brew`; current main also has no live service path that loads `cron/jobs/`.

## Solution Assessment

No. The directory-store direction may be useful, but this patch is not the best implementation because it adds unreachable helpers, keys files by mutable names, serializes runtime state into job config files, and leaves the public config/docs/doctor contract undefined.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported, but the PR never switches Gateway, service add/update/remove, CLI, config, or doctor paths to use it. Creating `cron/jobs/` would not affect runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Use immutable IDs for job filenames:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives filenames from `job.name`, but cron job names are mutable and nonunique while service-created IDs are stable UUIDs. Duplicate names can overwrite each other, and renames can leave multiple files for the same job.
  - confidence: 0.96
- **[P1] Delete files by persisted job IDs:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested ID with the filename-derived ID, but files written by this patch are name-derived. A normal job whose stable ID differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, `slugifyCronJobName` returns the raw input. Values containing path separators can then be joined into the output path, so the fallback must be safe and the resolved path must stay under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the entire `CronJob` into each per-job file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.85
