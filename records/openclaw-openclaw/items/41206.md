## Summary

Keep this PR open for maintainer/contributor follow-up, but it should not merge as-is: current main still lacks directory-backed cron jobs, the canonical tracker remains open, and this branch has unresolved storage-contract bugs, a filename-containment security concern, and no real behavior proof.

## What This Changes

The PR adds directory cron-store helpers and tests for storing one cron job JSON file per job under `cron/jobs/` alongside the existing `jobs.json` store.

## Best Possible Solution

Rework or replace this PR under https://github.com/openclaw/openclaw/issues/20950 with a backward-compatible directory-backed cron definition store that uses stable contained filenames, preserves runtime state, wires Gateway and CronService load/add/update/remove paths, and updates schema/help/doctor/docs plus focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head saves filenames from mutable names, deletes by filename-derived ids, writes runtime state into definition files, and current cron runtime paths still only use `loadCronStore` and `saveCronStore`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best merge path until the storage contract is settled, runtime paths are wired, filenames are stable and contained, state sidecar behavior is preserved, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but names are mutable and not unique while cron job ids are the stable identity. Renames or duplicate names can overwrite jobs or leave duplicate persisted files for the same id.
  - confidence: 0.95
- **[P1] Delete files by the persisted job id:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files saved by this PR are named from the job name. A normal id/name mismatch makes delete return false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory store into cron runtime:** `src/cron/store.ts:196-197`
  - body: The new directory-store helpers are exported but no cron startup, load, add, update, or remove path calls them. With this patch, real cron operations still use the aggregate `jobs.json` store.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar contract:** `src/cron/store.ts:248`
  - body: The writer serializes the full `CronJob`, including runtime fields, into each per-job definition file. Current main stores runtime metadata in `jobs-state.json`, so the directory backend needs to preserve or deliberately replace that contract.
  - confidence: 0.86
- **[P2] Constrain generated job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to raw input when no slug is produced. Path-like values made only of separators can escape the intended `cron/jobs/` file component, so generated filenames need a constrained stable id plus containment check.
  - confidence: 0.82
