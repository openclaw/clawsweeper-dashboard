## Summary

Keep open: current main still lacks directory-backed cron storage and this PR remains the active implementation candidate for the canonical cron jobs.d request, but the head is not mergeable as-is because source inspection confirms unresolved data integrity, runtime wiring, state-contract, filename containment, and proof gaps.

## What This Changes

The PR adds helpers and tests for storing cron jobs as one JSON file per job under `cron/jobs/` while keeping the existing aggregate `jobs.json` path.

## Best Possible Solution

Rework or replace this PR under https://github.com/openclaw/openclaw/issues/20950 with a backward-compatible directory-backed cron definition store that uses stable contained filenames, preserves `jobs-state.json`, wires Gateway and CronService load/add/update/remove paths, and updates schema/help/doctor/docs plus focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head saves filenames from mutable names, deletes by filename-derived ids, writes runtime state into definition files, and current cron runtime paths still only use `loadCronStore` and `saveCronStore`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best merge path until the storage contract is settled, the runtime paths are wired, filenames are made stable and contained, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but names are mutable and not unique while cron job ids are the stable identity. Renames or duplicate names can overwrite jobs or leave duplicate persisted files for the same id.
  - confidence: 0.95
- **[P1] Delete files by the persisted job id:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files saved by this PR are named from the job name. A normal id/name mismatch makes delete return false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory store into cron runtime:** `src/cron/store.ts:196-197`
  - body: The new directory-store helpers are exported but no cron startup, load, add, update, or remove path calls them. With this patch, real cron operations still use the aggregate `jobs.json` store.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar contract:** `src/cron/store.ts:248`
  - body: The writer serializes the full `CronJob`, including runtime fields, into each per-job definition file. Current main stores runtime metadata in `jobs-state.json`, so the directory backend needs to preserve or deliberately replace that contract.
  - confidence: 0.86
- **[P2] Constrain generated job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to raw input when the slug is empty. Values made only of path separators can be joined into `cron/jobs/`, so the filename should be generated from a constrained stable id and checked for containment.
  - confidence: 0.82
