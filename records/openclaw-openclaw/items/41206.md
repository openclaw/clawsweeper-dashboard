## Summary

Keep open. The directory-store idea is still useful and related #20950 remains open, but this PR is not merge-ready against current main: the added helpers are not wired into live cron persistence, the file keying can corrupt or orphan jobs, and current main now has a `jobs-state.json` sidecar contract this patch would need to preserve.

## What This Changes

The PR adds exported cron directory-store helpers and tests for loading, saving, removing, and migrating one JSON file per job under `cron/jobs/`.

## Best Possible Solution

Rework the feature on current main as a first-class cron storage backend with id-keyed files, explicit activation and precedence, preserved `jobs-state.json` semantics, documented symlink behavior, and aligned CLI/doctor/docs/tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a job with id `brew-1` and name `Daily Brew` writes `daily-brew.json`, while deletion by id compares against `daily-brew` and leaves the file behind.

## Solution Assessment

No. The direction is plausible for the GitOps use case, but this patch is not the best current solution because it is not wired into live persistence and does not preserve current state-sidecar semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported and tested, but Gateway startup, add/update/remove, CLI, doctor, and docs still use the flat `loadCronStore`/`saveCronStore` path. Creating `cron/jobs/` would not affect scheduler behavior.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the filename from mutable, non-unique `job.name`. Duplicate names can overwrite each other, and renames can leave multiple files for the same stable job id.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files written by this patch are name-derived. Normal jobs whose stable id differs from the display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, the helper returns raw input. Values containing path separators can escape `cron/jobs/` unless the fallback is safe and the resolved path is checked.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main keeps runtime fields in `jobs-state.json`, but this writer serializes the full `CronJob` into each per-job config file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.86
- **[P2] Support or document symlinked job files:** `src/cron/store.ts:211`
  - body: The related GitOps/NixOS use case depends on symlinkable job definitions, but `entry.isFile()` skips symlinks before parsing. The backend should either support symlinks with a clear containment policy or reject them with a documented diagnostic.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds a local cron filesystem write path with a concrete filename containment issue; it does not add dependencies, CI, package resolution, or secret handling.

Concerns:

- **[medium] Sanitize directory job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` returns raw input when no alphanumeric slug remains, and `saveCronJobFile` joins that value into the output path. A local cron job name or id containing path separators can escape `cron/jobs/` unless the fallback is safe and the resolved path is checked.
