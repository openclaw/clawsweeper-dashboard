## Summary

This PR should stay open for maintainer review: the directory-store feature is still absent on current main, but the branch has concrete correctness and security blockers before it can be merged.

## What This Changes

The PR adds directory-based cron store helpers and tests for storing each cron job as a separate JSON file under `cron/jobs/` while preserving `jobs.json` compatibility.

## Best Possible Solution

Rework this into a first-class cron storage backend with id-keyed contained filenames, explicit precedence and symlink behavior, state-sidecar preservation, Gateway/CLI/doctor/docs wiring, and regression coverage.

## Reproduction Assessment

Yes for the PR review findings: static source inspection of the branch shows the name-derived save path, id-derived delete path, raw fallback filename, and missing live-service wiring. This is a feature PR, not a current-main bug report.

## Solution Assessment

No. The one-file-per-job direction fits the open GitOps request, but this patch is not yet the narrowest maintainable implementation because it adds helper APIs and tests without integrating the runtime store contract safely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` and the per-job writer are exported and tested, but the live cron service still loads and saves only the flat `jobs.json` path. Creating `cron/jobs/` would not affect Gateway or CLI add/update/remove behavior, so the feature is not actually activated.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the output filename from `job.name`, which is mutable and not unique, while runtime jobs have stable ids. Duplicate names can overwrite each other, and renames can leave multiple files for the same job id on reload.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the id inferred from the filename, but files written by this patch are name-derived. A normal job whose id differs from its display name will not be removed from the directory store.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, the helper returns the raw input and the save path joins it into `cron/jobs/`. Names or ids containing path separators can escape the jobs directory unless the fallback is sanitized and the resolved path is constrained.
  - confidence: 0.9
- **[P2] Preserve runtime state outside job config files:** `src/cron/store.ts:248`
  - body: Current main strips runtime-only fields from `jobs.json` and writes them to `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job config file. Wiring it in would make runtime execution state user-editable config again.
  - confidence: 0.86
- **[P2] Define symlink handling before loading jobs:** `src/cron/store.ts:211`
  - body: The linked GitOps use case depends on symlinkable job definitions, but the loader skips directory entries unless `entry.isFile()` is true. The backend should either support symlinks with containment checks or reject them with a documented diagnostic.
  - confidence: 0.82
