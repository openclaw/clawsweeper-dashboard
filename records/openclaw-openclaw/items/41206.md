## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Key directory job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but names are mutable and not unique while cron job ids are stable UUIDs. Renaming a job or creating two jobs with the same name can leave stale files or overwrite a different job, so directory storage needs id-keyed filenames or an id-based cleanup scan before writing.
  - confidence: 0.95
- **[P1] Delete directory files by persisted job ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with `jobIdFromFilename(entry.name)`, but files written by this PR are named from the job name. A normal id/name mismatch makes removal return false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory backend into cron runtime paths:** `src/cron/store.ts:196-197`
  - body: The PR exports a directory loader/writer, but cron startup, add, update, and remove still use the aggregate `loadCronStore`/`saveCronStore` path. Real `openclaw cron` operations would continue using `jobs.json`, so the advertised directory store is not active.
  - confidence: 0.9
- **[P2] Preserve the split runtime-state sidecar:** `src/cron/store.ts:248`
  - body: The writer serializes the full `CronJob`, including runtime fields, into each per-job definition file. Current main intentionally strips runtime fields from `jobs.json` and stores them in `jobs-state.json`, so the directory backend must preserve or explicitly replace that contract.
  - confidence: 0.86
- **[P2] Constrain generated job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to raw input when no safe slug can be generated. Path-like all-symbol names or ids can then become unconstrained path components under `cron/jobs/`, so generated filenames need a safe stable id and a containment check.
  - confidence: 0.82
