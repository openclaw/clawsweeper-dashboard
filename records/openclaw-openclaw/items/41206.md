## Summary

Keep this PR open, but do not merge it as-is. Current main still only supports the flat `jobs.json` cron store, while this PR is dirty against main and its directory-store helpers remain unwired plus affected by filename, deletion, state-sidecar, and path-safety bugs already surfaced in review discussion.

## What This Changes

This PR adds exported helpers and tests for an optional `cron/jobs/` per-job JSON directory store while keeping the existing `jobs.json` store available.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: define activation and precedence, preserve `jobs.json` compatibility and the `jobs-state.json` sidecar contract, key files by stable collision-safe job ids, wire Gateway/CLI/doctor/docs, and test migration, symlinks, malformed files, rename/delete, overlap, and permissions behavior.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no `cron/jobs/` load/save path, and on the PR head a job with id `brew-1` and name `Daily Brew` writes `daily-brew.json` but deletion by `brew-1` compares against the filename-derived `daily-brew`.

## Solution Assessment

No. The directory-store direction may be useful, especially for the open GitOps request, but this patch is not the best implementation because it is stale, unreachable from runtime paths, uses mutable names as storage keys, bypasses the split-state contract, and leaves path safety unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported, but Gateway startup, cron add/update/remove, CLI, config, and doctor paths still use the flat `loadCronStore`/`saveCronStore` path. Creating `cron/jobs/` would not affect runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the filename from `job.name`, but cron job names are mutable and nonunique while service-created ids are stable UUIDs. Duplicate names can overwrite each other, and renames can leave multiple files for the same job.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files written by this patch are name-derived. A normal job whose stable id differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, `slugifyCronJobName` returns the raw input. Values containing path separators can then be joined into the output path, so the fallback must be safe and the resolved path must stay under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job config file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.86
