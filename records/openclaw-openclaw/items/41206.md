## Summary

Keep this PR open, but it is not merge-ready: current main still only loads and persists the flat cron store, this branch is the active attempt for the related directory-store feature, and the diff has blocking persistence, id-keying, and filename-containment issues.

## What This Changes

The PR adds directory-based cron store helpers and tests for one JSON file per cron job under `cron/jobs/` while preserving `jobs.json` compatibility.

## Best Possible Solution

Rework this as a first-class cron storage backend with id-keyed contained filenames, explicit load/write precedence, state-sidecar preservation, documented symlink and migration behavior, plus aligned Gateway, CLI, doctor, docs, and regression tests.

## Reproduction Assessment

Yes for the PR review findings: static diff inspection is enough to reproduce the name-derived save path versus id-derived delete path and the missing live-service wiring. This is a new feature patch, not a current-main bug reproduction.

## Solution Assessment

No. The directory-store direction is reasonable for the GitOps use case, but this patch is not the best current implementation because it adds helpers/tests without integrating the live scheduler/persist paths or settling the storage contract safely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported and tested, but current runtime paths still call the flat `loadCronStore` and `saveCronStore` backend. Creating `cron/jobs/` would not affect Gateway, CLI add/edit/remove, or reload behavior.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives filenames from mutable, non-unique `job.name`. Duplicate names can overwrite jobs, and renames can leave multiple files for the same stable job id.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files written by this patch are name-derived. A normal job whose id differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, the helper returns the raw input. Values containing path separators can escape `cron/jobs/` unless the fallback is sanitized and the resolved path is checked.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main stores runtime fields in `jobs-state.json`, but this writer serializes the full `CronJob` into each per-job config file. Wiring it in would make runtime execution state user-editable config again.
  - confidence: 0.86
- **[P2] Define symlink handling before loading jobs:** `src/cron/store.ts:211`
  - body: The related GitOps use case depends on symlinkable job definitions, but `entry.isFile()` skips symlinks before parsing. The backend should either support symlinks with containment checks or reject them with a documented diagnostic.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds a local cron file writer whose fallback filename can contain path separators and escape the jobs directory.

Concerns:

- **[medium] Sanitize directory job filenames:** `src/cron/store.ts:151`
