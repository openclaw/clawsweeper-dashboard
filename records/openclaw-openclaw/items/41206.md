The PR adds exported helpers and tests for loading, migrating, saving, and removing individual cron job JSON files under `cron/jobs/`.

No. The direction is useful, but this implementation is not the best fix until it is rebased and reworked around stable id-keyed files, current split-state persistence, explicit activation/config semantics, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wire the directory store into runtime operations:** `src/cron/store.ts:196`
  - body: The new directory-store helpers are exported but not called by Gateway startup or cron add/update/remove, which still pass a single `storePath` through `persist`. As written, creating `cron/jobs/` will not make the scheduler load or write per-job files, so the central feature is not active.
  - confidence: 0.94
- **[P1] Key directory files by immutable job ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds the filename from `job.name`, which is mutable and not unique. Two jobs with the same name can overwrite each other, and a rename writes a second file for the same id; use a stable contained id-based filename or remove the old id-matched file before writing.
  - confidence: 0.97
- **[P1] Delete files by persisted job ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id to the filename-derived slug, but files saved from human names commonly have a slug different from the stable job id. Removal then returns false and leaves the job file behind; load the candidate file and match `job.id`, or make filenames id-keyed.
  - confidence: 0.96
- **[P1] Preserve the split runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips runtime fields from `jobs.json` and stores `updatedAtMs` plus `state` in `jobs-state.json`, but this writer serializes the whole job into each per-job file and the directory loader never merges the sidecar. That can regress runtime-state behavior and mis-associate next-run or running state after upgrade.
  - confidence: 0.9
- **[P2] Reject filenames that cannot be safely slugged:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, this returns the raw name or id, so values containing parent-directory segments or path separators can escape the intended `jobs/` directory once used in `path.join`. Reject unsafe values or fall back to a sanitized stable id instead.
  - confidence: 0.88
