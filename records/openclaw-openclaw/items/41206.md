## Summary

Keep this PR open, but it is not merge-ready: current main still lacks the directory-backed cron store, while the PR only adds helper-level APIs/tests and has unresolved storage, security, runtime-wiring, conflict, and real-proof blockers.

## What This Changes

The PR adds cron store helpers and tests for loading, writing, deleting, and migrating one JSON file per cron job under `cron/jobs/` while keeping `jobs.json` available.

## Best Possible Solution

Rework or replace this with a first-class backward-compatible cron directory store that is wired into Gateway and CLI persistence, uses contained id-keyed filenames, preserves runtime state outside user config, defines precedence and symlink behavior, and updates schema/help, doctor, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Yes for the PR review findings: static inspection of the PR diff and current main shows helper-only wiring, name-derived save paths, filename-derived delete paths, full-job serialization, and unsafe raw filename fallback. This is a feature PR, not a current-main bug report.

## Solution Assessment

No. The one-file-per-job direction matches the open GitOps request, but this patch is not the narrowest maintainable solution until runtime wiring, id-keyed containment, state separation, storage policy, docs/schema/doctor coverage, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` and the per-job writer are exported and tested, but the live cron service still loads and saves only the flat `jobs.json` path. Creating `cron/jobs/` would not affect Gateway or CLI add/update/remove behavior, so the feature is not actually active.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the output filename from `job.name`, which is mutable and not unique, while runtime jobs have stable ids. Duplicate names can overwrite each other, and renames can leave multiple files for the same job id on reload.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the id inferred from the filename, but files written by this patch are name-derived. A normal job whose id differs from its display name will not be removed from the directory store.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, the helper returns raw input and the save path joins it into `cron/jobs/`. Names or ids containing path separators can escape the jobs directory unless the fallback is sanitized and the resolved path is constrained.
  - confidence: 0.9
- **[P2] Preserve runtime state outside job config files:** `src/cron/store.ts:248`
  - body: Current main strips runtime-only fields from `jobs.json` and writes them to `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job config file. Wiring it in would make runtime execution state user-editable config again.
  - confidence: 0.86
