## Summary

Keep this PR open, but it is not currently mergeable. Current main still lacks directory-backed cron jobs and https://github.com/openclaw/openclaw/issues/20950 remains the canonical tracker, while this PR is the active implementation candidate with blocking correctness/security findings and missing real behavior proof.

## What This Changes

The PR adds exported cron directory-store helpers and tests for per-job JSON files under `cron/jobs/`, including filename slug/disable handling and a migration helper.

## Best Possible Solution

Rework or replace this with a backward-compatible directory-store backend wired into Gateway and CLI persistence, using stable id-keyed contained filenames, preserving `jobs-state.json`, and documenting deterministic precedence, migration, doctor, and validation behavior.

## Reproduction Assessment

Yes for the PR findings: static diff and current-main source inspection show helper-only wiring, name-derived save paths, filename-derived delete paths, full-job serialization, and unsafe raw filename fallback. This is a feature PR review rather than a current-main bug reproduction.

## Solution Assessment

No. One-file-per-job storage is a useful direction, but this patch is not the best mergeable solution until runtime wiring, stable contained filenames, state separation, storage policy, docs/schema/doctor coverage, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` and the per-job writers are exported and tested, but Gateway/CronService still load and save only the configured `jobs.json` path. Creating `cron/jobs/` would not affect live add/update/remove/startup behavior, so the advertised feature remains inactive.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the output filename from `job.name`, which is mutable and not unique, while cron jobs already have stable ids. Duplicate names can overwrite one another, and renames can leave stale files that reload as duplicate jobs.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id to an id inferred from the filename, but this PR writes files using the job name. A normal job whose UUID id differs from its display name will not be removed from the directory store.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, the helper returns raw input and the writer joins it into `cron/jobs/`. Names or ids with path separators can escape the jobs directory unless the fallback is sanitized and the resolved path is constrained.
  - confidence: 0.9
- **[P2] Preserve runtime state outside job files:** `src/cron/store.ts:248`
  - body: Current main strips runtime-only fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job config file. Wiring this in would make runtime execution state user-editable config again.
  - confidence: 0.86
