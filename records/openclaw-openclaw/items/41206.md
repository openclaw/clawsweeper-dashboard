## Summary

Keep this PR open, but it is not mergeable as-is. Current main and the canonical tracker at https://github.com/openclaw/openclaw/issues/20950 still need directory-backed cron storage, while this PR has unresolved runtime wiring, filename/id, state-sidecar, security, conflict, and real-proof blockers.

## What This Changes

The PR adds directory-store cron helpers and tests for storing one cron job per JSON file under `cron/jobs/` while preserving the existing aggregate `jobs.json` path.

## Best Possible Solution

Rework or replace this PR under https://github.com/openclaw/openclaw/issues/20950 with a backward-compatible directory-backed cron definition store that wires Gateway and CronService load/add/update/remove paths, uses stable contained filenames, preserves `jobs-state.json`, and updates schema/help/doctor/docs plus focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head saves filenames from mutable `job.name`, deletes by filename-derived id, writes full runtime state into definition files, and current cron runtime paths still only call `loadCronStore`/`saveCronStore`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best merge path until it is integrated into the real cron storage contract, reconciled with current main, and backed by real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but names are mutable and not unique while cron job ids are the stable identity. Renames or duplicate names can overwrite jobs or leave duplicate persisted files for the same id.
  - confidence: 0.95
- **[P1] Delete files by the persisted job id:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files saved by this PR are named from the job name. A normal job such as id `brew-1` and name `Daily Brew` saves `daily-brew.json`, so delete by id returns false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory store into cron runtime:** `src/cron/store.ts:196-197`
  - body: The new directory-store helpers are exported but no current cron startup, load, add, update, or remove path calls them. With this patch, `CronService` still uses the aggregate `jobs.json` store, so the advertised directory store is not active for real cron operations.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar contract:** `src/cron/store.ts:248`
  - body: The new writer serializes the full `CronJob`, including runtime fields, into each per-job definition file. Current main splits runtime metadata into `jobs-state.json`, so a directory backend needs to keep `state` and `updatedAtMs` out of user-tracked definition files or explicitly replace that contract.
  - confidence: 0.86
- **[P2] Constrain generated job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to raw input when the slug is empty, so names or ids made only of path separators can be joined into the jobs path and write outside the intended `cron/jobs/` directory. Generate a constrained stable filename and verify containment before writing.
  - confidence: 0.82
