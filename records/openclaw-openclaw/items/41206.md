## Summary

Keep open. Main does not have per-job cron files yet, and this PR is not merge-ready because its new helpers are not wired into the live cron load/save path and the proposed filename strategy can corrupt or misplace job files.

## What This Changes

The PR adds directory-store cron helpers and tests for loading, saving, removing, and migrating one JSON file per cron job under `cron/jobs/`.

## Best Possible Solution

Rework this as a fresh cron storage design against current main: preserve `jobs.json` compatibility and `jobs-state.json` semantics, use stable id-keyed filenames, define opt-in/precedence clearly, and cover CLI, doctor, docs, changelog, and regression tests.

## Reproduction Assessment

Yes for the PR defects from source inspection. Current main only loads and saves the configured flat store file, while the PR’s save/remove mismatch follows directly from `saveCronJobFile` using `job.name` and `removeCronJobFile` comparing filename-derived ids.

## Solution Assessment

No. Helper-only directory support is not the narrowest maintainable solution; the feature needs an integrated storage abstraction that preserves the current state split and uses stable, path-safe filenames.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wire the directory store into cron persistence:** `src/cron/store.ts:187`
  - body: The new directory loader is never used by the live cron service: current loading still goes through `loadCronStore(state.deps.storePath)`, and no save path calls the new per-file writer/remover. Creating `~/.openclaw/cron/jobs/` would therefore not make the gateway load or persist these files, so the advertised feature remains inert.
  - confidence: 0.95
- **[P1] Use stable job ids for filenames:** `src/cron/store.ts:235`
  - body: `saveCronJobFile` derives the filename from mutable, non-unique `job.name`, but removal matches filenames by job id. A normal UUID-backed job named `Daily Brew` saves as `daily-brew.json` and will not be removed by UUID; renames or duplicate names can also leave stale files or overwrite another job.
  - confidence: 0.97
- **[P2] Keep slugs inside the jobs directory:** `src/cron/store.ts:236`
  - body: The slug helper falls back to the raw name/id when it contains no ASCII alphanumerics, and that value is joined directly into the output path. A value with `..` path segments can escape `jobsDir` before the write, so the fallback needs to be path-segment safe or rejected.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention: no dependency or workflow changes were found, but the new job-file write path can use an unsanitized raw fallback slug outside `cron/jobs`.

Concerns:

- **[medium] Raw slug fallback can escape the jobs directory:** `src/cron/store.ts:236`
  - body: `slugifyCronJobName` may return the raw input when slugification is empty, and `saveCronJobFile` passes that value into `path.join(jobsDir, ...)`. A job name made only of path separators or `..` segments can target a path outside the intended jobs directory.
  - confidence: 0.86
