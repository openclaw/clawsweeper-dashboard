## Summary

Keep this PR open, but it is not merge-ready: current main and v2026.5.12 still lack the per-job cron directory store, the open canonical tracker is https://github.com/openclaw/openclaw/issues/20950, and this branch still has blocking correctness, security, and real-behavior-proof gaps already reflected in prior review discussion.

## What This Changes

Adds exported cron directory-store helpers and tests for storing each cron job as its own JSON file under `cron/jobs/` while preserving the existing `jobs.json` path.

## Best Possible Solution

Rework or replace this under https://github.com/openclaw/openclaw/issues/20950 with a backward-compatible directory-backed cron definition store keyed by stable contained ids, preserving `jobs-state.json`, wiring Gateway/CronService load/add/update/remove paths, and updating schema, docs, doctor behavior, migration, and focused tests.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch saves filenames from mutable names, deletes by filename-derived ids, writes full runtime state into definition files, and current cron runtime paths still only use aggregate `loadCronStore`/`saveCronStore`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best merge path until stable contained filenames, runtime wiring, state-sidecar behavior, docs/schema/doctor/migration expectations, and real behavior proof are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but names are mutable and not unique while cron job ids are the stable identity. Renames or duplicate names can overwrite jobs or leave duplicate persisted files for the same id.
  - confidence: 0.95
- **[P1] Delete files by persisted job ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the id inferred from the filename, but files saved by this PR are named from the job name. A normal id/name mismatch makes delete return false and leaves the job file behind.
  - confidence: 0.96
- **[P2] Wire the directory store into runtime paths:** `src/cron/store.ts:196-197`
  - body: The new directory-store helpers are exported but no cron startup, load, add, update, or remove path calls them. With this patch, real cron operations still use the aggregate `jobs.json` store.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar writes:** `src/cron/store.ts:248`
  - body: The writer serializes the full `CronJob`, including runtime fields, into each per-job definition file. Current main stores runtime metadata in `jobs-state.json`, so the directory backend needs to preserve or deliberately replace that contract.
  - confidence: 0.86
- **[P2] Constrain generated job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to raw input when no slug can be generated. Path-like values made only of separators can escape the intended `cron/jobs/` file component, so generated filenames need a constrained stable id plus a containment check.
  - confidence: 0.82
