## Summary

Keep open, but do not merge yet. This is still the live implementation attempt for cron directory-store support, but static review finds unreachable runtime persistence, unstable filename keying, delete/rename corruption, split-state regression risk, and a concrete filename containment issue.

## What This Changes

The PR adds exported helpers and tests for an optional per-job cron directory store under `cron/jobs/` while leaving the existing `jobs.json` store in place.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: define activation and precedence, preserve `jobs.json` compatibility and the runtime-state sidecar, use stable collision-safe ids for filenames, handle or explicitly reject symlinks, and update Gateway, CLI, doctor, docs, and tests together.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on the PR head, a job with id `brew-1` and name `Daily Brew` writes `daily-brew.json`, then deletion by `brew-1` compares against the filename-derived `daily-brew` and misses the file; current main also has no runtime call path that loads `cron/jobs/`.

## Solution Assessment

No. The directory-store direction is plausible for the related GitOps use case, but this implementation is not the best fix because it is helper-only, name-keyed, stale against the split-state contract, and lacks filename containment safeguards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported, but Gateway startup, cron add/update/remove, CLI, config, docs, and doctor paths still use the flat `loadCronStore`/`saveCronStore` path. Creating `cron/jobs/` would not affect runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the filename from `job.name`, but cron job names are mutable and nonunique while service-created ids are stable UUIDs. Duplicate names can overwrite each other, and renames can leave multiple files for the same job.
  - confidence: 0.96
- **[P1] Delete job files by persisted ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, but files written by this patch are name-derived. A normal job whose stable id differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, `slugifyCronJobName` returns the raw input. Values containing path separators can then be joined into the output path, so the fallback must be safe and the resolved path must stay under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job config file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.86
- **[P2] Support or explicitly reject symlinked job files:** `src/cron/store.ts:211`
  - body: The related GitOps/NixOS use case depends on symlinking job definitions into the store, but `entry.isFile()` skips symlinks before parsing. Either support symlinked JSON files with a clear containment policy or reject them with a documented diagnostic.
  - confidence: 0.82
