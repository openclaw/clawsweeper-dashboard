## Summary

Keep open. Current main still resolves and writes doctor config through the invoking process environment without a doctor preflight comparison against the running service/gateway config path, and #63268 remains an open paired implementation PR for this issue.

## What This Changes

The issue requests a guard or warning so `openclaw doctor --fix` does not silently create and write a minimal root-shell config when a managed gateway is using a different home/config path.

## Best Possible Solution

Review and land #63268 or a replacement that compares doctor’s planned write path with a generic service or running-gateway config-path signal, emits clear `OPENCLAW_CONFIG_PATH` remediation, and preserves existing config resolution semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The reporter provided concrete systemd/root-shell steps, and current source shows doctor resolves and writes the invoking environment’s config path without comparing it to the service config path.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A focused doctor preflight warning/guard is the right narrow direction, but the implementation should avoid hard-coded home guesses and reuse a generic service or running-gateway config signal where possible.
