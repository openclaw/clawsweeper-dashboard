## Summary

Current main still lacks the reported doctor preflight guard, and the issue is explicitly paired with an open, unmerged implementation PR. The source path still lets `doctor --fix` resolve and write through the invoking shell environment without comparing that path to the running gateway or service config path.

## What This Changes

The issue requests a guard or warning so `openclaw doctor --fix` does not silently create or repair a root-shell config that diverges from the managed gateway's real config path.

## Best Possible Solution

Review and land #63268 or a replacement that compares doctor's planned write path with a generic running gateway or service config-path signal, emits a clear `OPENCLAW_CONFIG_PATH` remediation, and preserves existing config resolution semantics.

## Reproduction Assessment

Yes. The reporter provided concrete root-shell/systemd-HOME mismatch steps, and current source still shows doctor resolving and writing through the invoking environment without a running-service path comparison; I did not execute destructive `doctor --fix` during this read-only review.

## Solution Assessment

Unclear. A focused doctor preflight warning is the right narrow direction, but the safest implementation should reuse generic service or running-gateway config metadata rather than relying only on hard-coded alternate home directory probes.
