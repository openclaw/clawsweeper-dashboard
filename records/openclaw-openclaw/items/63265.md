## Summary

Keep open: current main still lacks a doctor preflight/write guard for the reported CLI-vs-managed-service config path divergence, and the issue remains paired with an open, unmerged implementation PR.

## What This Changes

Add a `doctor --fix` warning or guard so a root shell does not silently create or repair a different config path than the running Gateway service uses.

## Best Possible Solution

Review and land #63268 or a replacement that compares doctor’s planned write path with the managed/running Gateway config path and emits actionable `OPENCLAW_CONFIG_PATH` remediation without changing normal config resolution semantics.

## Reproduction Assessment

Yes. The reporter gave concrete root-shell/systemd-HOME mismatch steps, and current source still resolves and writes doctor config through the invoking environment without a service-path comparison; I did not run destructive `doctor --fix` in this read-only review.

## Solution Assessment

Unclear. A focused doctor warning or guard is the right narrow fix, but the safest implementation should use a generic service or running-gateway config-path signal rather than probing a small list of alternate home directories.
