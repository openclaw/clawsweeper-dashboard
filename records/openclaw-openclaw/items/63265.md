## Summary

Keep open: current main still lacks a doctor preflight guard for the reported CLI-vs-service config path divergence, and the issue is paired with open, unmerged implementation PR #63268.

## What This Changes

`openclaw doctor --fix` should detect or warn before writing a caller-selected root config path that differs from the running service or gateway config path.

## Best Possible Solution

Review and land #63268 or a replacement that compares doctor’s planned write path with a generic service or running-gateway config-path signal, emits an actionable `OPENCLAW_CONFIG_PATH` remediation, and preserves existing config resolution semantics.

## Reproduction Assessment

Yes. The reporter provided concrete root-shell/systemd-HOME mismatch steps, and current source still shows doctor resolving and writing through the invoking environment without a running-service path comparison; I did not run destructive `doctor --fix` during this read-only review.

## Solution Assessment

Unclear. A focused doctor warning/guard is the right narrow direction, but the safest implementation should reuse service command environment or running-gateway metadata rather than hard-coded alternate home directory probes.
