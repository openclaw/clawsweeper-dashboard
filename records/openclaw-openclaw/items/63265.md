## What This Changes

The issue asks for `openclaw doctor --fix` to warn or guard when its resolved config path differs from the running gateway or managed-service config path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/63268 or a replacement that compares doctor’s resolver-backed write path with a verified service or running-gateway config path before repair writes.

Yes, source-backed. The reporter gives a concrete root-shell/systemd `HOME` mismatch flow, and current source shows doctor resolves and writes config from the invoking process environment without a doctor-time service or running-gateway config-path comparison.

## Solution Assessment

Yes for the issue direction, but the paired PR still needs review. A focused warning or guard is the narrow maintainable fix, preferably using existing resolver and service/running-gateway context rather than hard-coded alternate home probes.
