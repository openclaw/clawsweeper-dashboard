## Summary

Keep open: current main does not already include this Workboard dialog accessibility hardening, and the PR has a blocking focus-restore bug caused by recreating Lit ref callbacks on each render.

## What This Changes

The PR adds Workboard modal and drawer dialog roles, focus trap/return handling, inert background behavior, screen-reader labels, CSS support, and focused UI tests.

## Best Possible Solution

Keep the Workboard accessibility direction, but make the dialog ref/cleanup lifecycle stable across Lit rerenders and cover close-after-edit for both modal and drawer paths.

## Reproduction Assessment

Yes. Source inspection plus Lit's ref contract shows that typing in an open Workboard dialog triggers a rerender, the inline ref is treated as changed, and the saved return-focus target can be overwritten before close.

## Solution Assessment

No. The UI owner boundary is right, but the current implementation is not the best fix until dialog refs or cleanup guards are stable across normal Lit rerenders.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep the return-focus target stable across rerenders:** `ui/src/ui/views/workboard.ts:542`
  - body: This inline `ref` callback is recreated on every render. Lit calls the previous callback with `undefined` when the ref identity changes, so a normal dialog update like typing in the title field runs `restoreWorkboardFocus()`, clears the launcher target, and then the new ref can recapture the focused input as the return target. When the modal later closes, that input is disconnected and focus is not restored to the opener. Use a stable ref callback or guard cleanup so ordinary rerenders do not reset return focus, and add a rerender-before-close regression.
  - confidence: 0.91
