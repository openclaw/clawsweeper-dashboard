## Summary

Keep open: current main still has the unbounded tools-manager release JSON read and the implementation direction is sound, but this PR is not merge-ready because the added proof/test bypasses the changed tools-manager path and the new fail-closed 1 MiB cap needs maintainer acceptance.

## What This Changes

The PR replaces tools-manager's GitHub latest-release `response.json()` call with a 1 MiB `readResponseWithLimit` read and adds a bounded-read proof test.

## Best Possible Solution

Keep the shared bounded-reader approach, but prove `ensureTool()` or `getLatestVersion()` with normal and oversized streamed release responses before merging with an explicit cap decision.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` both read tools-manager GitHub latest-release success responses through `response.json()`, so the unbounded success-body path is source-reproducible without running an OOM scenario.

## Solution Assessment

Yes for the implementation layer: `getLatestVersion()` owns this parse and `readResponseWithLimit` is the established bounded-reader contract. It is not merge-ready until the proof/test drives the changed production path and maintainers accept or tune the cap.
