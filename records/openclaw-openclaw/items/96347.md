## Summary

Keep open: the code change is a narrow, correct hardening of the tools-manager GitHub release JSON read, and current main plus v2026.6.10 still use unbounded `response.json()`. The remaining blocker is merge readiness: the external-PR proof is still code/tests/CI only, without PR-body terminal output, logs, or a real after-fix harness run showing the changed path rejecting an oversized response.

## What This Changes

The PR replaces `getLatestVersion()`'s GitHub latest-release `response.json()` call with a 1 MiB `readResponseWithLimit` read before `JSON.parse`, and adds helper-level bounded-read tests.

## Best Possible Solution

Keep the shared bounded-reader implementation, add PR-body real behavior proof for normal fd/rg release JSON and an oversized response, then let maintainers accept the 1 MiB cap before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.10 call `response.json()` on the GitHub latest-release body, and a Node stream probe showed `Response.json()` consumes the full finite stream before resolving. I did not run a full tools-manager download harness because this review is read-only.

## Solution Assessment

Yes for the code shape: `getLatestVersion()` owns this GitHub API parse, and `readResponseWithLimit` is the existing bounded-reader contract. The PR is not merge-ready until the external real-behavior proof and 1 MiB threshold are maintainer-accepted.
