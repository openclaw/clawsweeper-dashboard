## Best Possible Solution

Keep the shared-reader fix shape, then add focused after-fix proof showing an oversized streamed GitHub release JSON response is rejected under the cap while normal latest-release JSON still parses.

## Reproduction Assessment

I did not run an oversized tools-manager harness in this read-only review.

## Solution Assessment

Yes, the code shape is the best narrow fix: `getLatestVersion()` owns the GitHub API parse and `readResponseWithLimit` is the existing bounded-reader contract. The PR still needs real behavior proof before merge.
