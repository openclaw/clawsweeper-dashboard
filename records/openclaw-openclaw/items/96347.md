## Summary

Keep this PR open: the code change is a narrow and likely correct hardening of the tools-manager GitHub release JSON path, but it is not merge-ready until the contributor proof covers the production path or live GitHub release response behavior and maintainers explicitly accept the new 1 MiB fail-closed cap.

## What This Changes

The PR replaces the tools-manager GitHub latest-release `response.json()` parse with a 1 MiB `readResponseWithLimit` read before `JSON.parse`, and adds helper-level bounded-read proof tests.

## Best Possible Solution

Keep the shared bounded-reader approach, add production-path or live GitHub release proof for normal fd/rg responses plus an oversized streamed response, then merge once maintainers accept the 1 MiB threshold.

## Reproduction Assessment

Yes. Current main and the latest release both parse the tools-manager GitHub latest-release body with `response.json()`, so the unbounded read path is source-reproducible without needing a live oversized response.

## Solution Assessment

Yes for the code shape: `getLatestVersion()` owns this parse and `readResponseWithLimit` is the established bounded-reader contract. The PR still needs stronger production-path or live API proof and explicit cap acceptance before merge.
