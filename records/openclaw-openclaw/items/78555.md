## Summary

Keep open: the reported CLI crash is real and the PR has sufficient real-environment proof, but the branch still adds an ungated v4 `secrets.resolve` result field that current v4 clients reject under `additionalProperties: false`, so it needs a narrow compatibility repair before merge.

## What This Changes

The branch adds `createIfMissing` to `secrets.resolve` assignments, applies flagged sibling-ref CLI secret assignments with `setPathCreateStrict`, updates generated protocol/checksum surfaces, and adds CLI regression coverage plus a changelog entry.

## Best Possible Solution

Preserve the sibling-ref create-missing CLI behavior, but derive the create decision from local secret target metadata or explicitly version/gate any wire-visible protocol change.

## Reproduction Assessment

Yes. Current main is source-reproducible: gateway assignments are applied with `setPathExistingStrict`, while Google Chat sibling-ref configs can validly omit `channels.googlechat.serviceAccount`; the PR body also supplies Pi 5 before/after terminal proof.

## Solution Assessment

No, not as written. The sibling-ref-only create path is the right narrow fix, but adding an ungated v4 result field is not the safest compatibility path; derive it locally or version/gate the wire contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep createIfMissing out of the v4 result schema:** `src/gateway/protocol/schema/secrets.ts:21`
  - body: This adds `createIfMissing` to a v4 `secrets.resolve` assignment, but existing v4 CLIs validate assignments with `additionalProperties: false` before applying them. A newer gateway returning this field can make an older CLI reject the whole payload as invalid, even when the resolved leaf already exists, so derive this decision client-side from the local target shape or version/gate the protocol field.
  - confidence: 0.86
