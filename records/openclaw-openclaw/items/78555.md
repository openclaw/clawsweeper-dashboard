## Summary

Keep open: the PR targets a real CLI SecretRef crash, but the current patch still introduces a protocol compatibility regression for strict older `secrets.resolve` clients.

## What This Changes

The PR adds a `createIfMissing` assignment signal for `secrets.resolve`, uses it in CLI SecretRef application for missing `sibling_ref` leaves, and updates tests, Swift protocol models, and the changelog.

## Best Possible Solution

Keep the sibling-ref create-on-apply behavior, but derive it from existing CLI target metadata or gate/version the response field so older strict clients never receive unknown assignment properties.

## Reproduction Assessment

Yes, at source level. Current main has a documented read-only SecretRef status path, a Google Chat `sibling_ref` target whose resolved leaf can be absent, and an unconditional `setPathExistingStrict` apply that throws on that missing leaf.

## Solution Assessment

No, not as-is. The behavior fix is narrow, but the wire-field approach should be made protocol-compatible before merge, preferably by CLI-side inference from target metadata or by explicit capability/version gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate createIfMissing before emitting it:** `src/secrets/command-config.ts:82`
  - body: This field is emitted over `secrets.resolve`, but existing CLI builds validate assignments with `additionalProperties: false` before applying them. A newer gateway that sends `createIfMissing` makes those clients reject the whole response as invalid, so please gate/version this signal or infer the `sibling_ref` create behavior on the CLI side without adding an unconditional response field.
  - confidence: 0.9
