## Summary

Keep this PR open: it fixes a valid CLI SecretRef crash and has sufficient real behavior proof, but the current diff still introduces a protocol compatibility regression by emitting a new strict `secrets.resolve` response property without version or capability gating.

## What This Changes

The PR adds an optional `createIfMissing` flag to `secrets.resolve` assignments, uses it to create missing sibling-ref CLI SecretRef leaves, updates Swift protocol models, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the sibling-ref CLI create-on-apply behavior, but land it with a protocol-compatible signal or client-side inference so version-skewed clients keep accepting `secrets.resolve` responses.

## Reproduction Assessment

Yes, at source level and with PR-provided terminal proof. Current main validates a strict gateway payload, applies assignments with `setPathExistingStrict`, and Google Chat can target an absent sibling-ref `serviceAccount` path from `serviceAccountRef`-only config.

## Solution Assessment

No, not as-is. The behavioral fix is narrow and appropriate, but the protocol field should be gated/versioned or inferred client-side before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
