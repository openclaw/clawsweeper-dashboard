## Summary

Keep open: the reported CLI SecretRef bug is valid and the PR has sufficient real-world proof, but the current patch still introduces an unconditional `createIfMissing` field in the strict `secrets.resolve` response shape, which can break older clients talking to a newer gateway.

## What This Changes

The PR adds a `createIfMissing` assignment signal for `secrets.resolve`, uses it to create missing `sibling_ref` resolved-value leaves in CLI SecretRef application, updates regression tests, generated Swift protocol models, and the changelog.

## Best Possible Solution

Keep the sibling-ref create-on-apply behavior, but derive it from existing CLI target metadata or gate/version the response field so strict older clients never receive unknown assignment properties.

## Reproduction Assessment

Yes, at source level. Current main has a documented read-only SecretRef status path, a Google Chat `sibling_ref` target whose resolved leaf can be absent, and an unconditional `setPathExistingStrict` apply that throws on that missing leaf.

## Solution Assessment

No, not as-is. The behavioral fix is narrow, but the wire-field approach should be made protocol-compatible before merge, preferably by CLI-side inference from target metadata or explicit capability/version gating.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate createIfMissing before emitting it:** `src/secrets/command-config.ts:82`
  - body: This flag is now serialized on `secrets.resolve` assignments, but existing CLI builds validate assignments with `additionalProperties: false` before applying them. A newer gateway returning `createIfMissing` makes those clients reject the response, so gate/version this signal or infer the sibling-ref create behavior on the CLI side without adding an unconditional response field.
  - confidence: 0.9
