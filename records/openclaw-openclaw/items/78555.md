## Summary

Keep open: the PR targets a valid CLI SecretRef crash and includes sufficient real behavior proof, but it still emits a new strict-schema protocol field that older clients will reject.

## What This Changes

The branch adds an optional `createIfMissing` marker to `secrets.resolve` assignments, uses `setPathCreateStrict` for marked sibling-ref CLI SecretRef application, and updates tests, Swift protocol models, and the changelog.

## Best Possible Solution

Keep the sibling-ref create-on-apply behavior, but infer it from existing CLI target metadata or gate/version the response field so strict older clients never receive unknown assignment properties.

## Reproduction Assessment

Yes, at source level. Current main documents read-only SecretRef status/channel degradation, Google Chat exposes a `sibling_ref` resolved-value path that can be absent, and the CLI currently applies gateway assignments with `setPathExistingStrict`.

## Solution Assessment

No, not as-is. The behavioral fix is narrow and appropriate, but the protocol signal should be derived client-side or gated/versioned before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:
