## Summary

Keep open. Current main still has the CLI apply path using `setPathExistingStrict` for gateway-returned assignments, while Google Chat registers `serviceAccount` as a `sibling_ref` target whose resolved-value leaf can intentionally be absent; the PR’s additive `createIfMissing` flag and CLI apply branch address that path without changing `secret_input` strictness.

## What This Changes

The PR adds an optional `createIfMissing` field to `secrets.resolve` assignments, propagates it from sibling-ref secret targets, applies those assignments with `setPathCreateStrict` in the CLI, regenerates the Swift protocol model and generated baseline hashes, and adds CLI regression tests.

## Best Possible Solution

Land this narrow protocol-compatible CLI apply fix after normal maintainer and CI checks, preserving strict existing-path behavior for unflagged assignments.

## Reproduction Assessment

Yes. Source inspection shows current main discovers the Google Chat sibling-ref target while the CLI then writes the resolved value with `setPathExistingStrict`, which throws when the resolved-value leaf is absent; the PR body also supplies before/after real command output.

## Solution Assessment

Yes. The PR uses an additive protocol hint derived from the existing `secretShape` contract and centralizes the CLI apply choice, which is narrower than changing all command secret assignments to create missing paths.
