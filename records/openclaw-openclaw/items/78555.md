## Summary

Keep open for normal PR review/merge. The patch addresses a real current-main CLI SecretRef apply bug with a narrow additive protocol flag, and I found no blocking correctness issue in the diff.

## What This Changes

The PR adds an optional `createIfMissing` flag to `secrets.resolve` assignments for `sibling_ref` targets and uses it in CLI command secret resolution, with Swift protocol, generated-baseline hash, changelog, and regression test updates.

## Best Possible Solution

Land the additive protocol and CLI apply fix once required checks finish, preserving strict existing-path behavior for non-`sibling_ref` SecretRef assignments.

## Reproduction Assessment

Yes. Current main uses `setPathExistingStrict` for gateway assignments, and Google Chat’s `sibling_ref` contract can legitimately omit `channels.googlechat.serviceAccount` when only `serviceAccountRef` is configured; the PR body also includes live before/after CLI proof.

## Solution Assessment

Yes. The per-assignment flag is an additive protocol hint derived from the generic secret target shape, avoiding a Google Chat special-case while keeping older gateways and ordinary `secret_input` assignments strict.
