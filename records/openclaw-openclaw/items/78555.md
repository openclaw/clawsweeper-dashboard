## Summary

Keep open: this PR fixes a valid, source-reproducible CLI SecretRef bug and includes sufficient real terminal proof, but the current diff still sends an un-gated `secrets.resolve` response field that strict older clients reject.

## What This Changes

The PR adds a `createIfMissing` marker to `secrets.resolve` assignments, applies marked `sibling_ref` SecretRefs with `setPathCreateStrict` in CLI flows, updates Swift protocol models and tests, and adds a changelog entry.

## Best Possible Solution

Keep the sibling-ref create-on-apply behavior, but derive the create decision in the CLI from existing target metadata or add explicit protocol gating/versioning before any new response field is emitted.

## Reproduction Assessment

Yes, at source level. Current main has a documented read-only SecretRef status contract, Google Chat exposes a `sibling_ref` target whose resolved-value path can be absent, and the CLI currently applies gateway assignments with `setPathExistingStrict`.

## Solution Assessment

No, not as-is. The behavioral fix is the right narrow direction, but the protocol-field approach needs compatibility repair or client-side inference because existing clients reject unknown assignment properties.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate createIfMissing before emitting it:** `src/secrets/command-config.ts:82`
