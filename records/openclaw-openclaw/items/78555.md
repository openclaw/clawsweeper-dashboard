## Summary

Keep open: this is a valid bug-fix PR with sufficient real-world terminal proof, but the current diff still has a blocking protocol-compatibility regression because it emits an ungated `createIfMissing` field in `secrets.resolve` responses that strict older CLI clients reject as an invalid payload.

## What This Changes

The PR adds sibling-ref-aware CLI SecretRef assignment handling by carrying `createIfMissing` through `secrets.resolve` assignments and using `setPathCreateStrict` for those assignments.

## Best Possible Solution

Keep the sibling-ref create-on-apply behavior, but derive it from existing CLI target metadata or version/capability-gate the response field so strict older clients never receive unknown assignment properties.

## Reproduction Assessment

Yes, at source level. Current main validates `secrets.resolve` payloads and then applies assignments with `setPathExistingStrict`, while Google Chat has a `sibling_ref` target whose resolved-value path can be absent in legacy ref-only config.

## Solution Assessment

No, not as-is. The behavioral fix is narrow and appropriate, but the new protocol signal must be inferred client-side or gated/versioned before merge.
