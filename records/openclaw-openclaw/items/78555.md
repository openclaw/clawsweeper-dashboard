## Summary

Keep open: the reported CLI crash is valid and the PR’s sibling_ref create behavior is useful, but the patch still adds an ungated property to a strict v4 `secrets.resolve` result, which can break older v4 TypeScript clients talking to a newer gateway.

## What This Changes

The PR adds a `createIfMissing` hint to command secret assignments and uses it so CLI secret resolution can create missing sibling_ref resolved-value leaves.

## Best Possible Solution

Keep the sibling_ref create behavior, but make it v4-compatible by deriving it from target metadata in the CLI or by explicitly versioning/gating the new protocol field with regenerated clients and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows gateway assignments are applied with `setPathExistingStrict`, while Google Chat sibling_ref configs can validly omit `channels.googlechat.serviceAccount`; I did not run live commands in this read-only review.

## Solution Assessment

No, not as written. The create-on-sibling-ref behavior is the right narrow fix, but the current patch changes a strict v4 wire payload without compatibility handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate the new secrets.resolve assignment field:** `src/gateway/protocol/schema/secrets.ts:21`
  - body: This adds `createIfMissing` to a v4 `secrets.resolve` result while existing v4 TypeScript clients validate assignments with `additionalProperties: false`. A newer gateway can make an older CLI reject the whole payload before applying any assignment, including sibling_ref assignments whose value leaf already exists. Please derive this on the client side, gate emission by negotiated support, or bump/version the protocol with the generated clients and docs updated.
  - confidence: 0.86
