## Summary

Keep open: the reported CLI crash is still valid on current main, and the PR has strong real-behavior proof, but the current patch adds an ungated v4 gateway result field that existing v4 clients can reject before applying any assignment.

## What This Changes

The branch adds a `createIfMissing` flag to `secrets.resolve` assignments, applies flagged `sibling_ref` CLI SecretRef assignments with `setPathCreateStrict`, regenerates protocol/baseline artifacts, and adds CLI regression coverage.

## Best Possible Solution

Land a rebased or fresh PR that keeps the v4 wire schema unchanged and derives create-missing behavior from local `sibling_ref` target metadata, with focused tests for sibling-ref creation and strict non-sibling assignments.

## Reproduction Assessment

Yes. Current main is source-reproducible: Google Chat can validly have `serviceAccountRef` without `serviceAccount`, while the CLI applies returned assignments with `setPathExistingStrict`; the PR body and comments add Pi 5 and x86_64 terminal proof of the crash and after-fix exit-0 behavior.

## Solution Assessment

No, not as written. The create-missing behavior is the right fix boundary, but carrying it as an ungated v4 wire field is not the safest contract path; local derivation or an explicit protocol version path is safer.
