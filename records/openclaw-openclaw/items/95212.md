## What This Changes

The PR changes QMD memory backend command resolution so unquoted Windows drive-letter and UNC `memory.qmd.command` paths bypass POSIX shell-argument parsing, and adds two regression tests.

## Best Possible Solution

Land one canonical fix that preserves Windows absolute QMD command paths at the backend-config boundary, keeps regression coverage, and includes redacted Windows QMD status/index/search proof.

## Reproduction Assessment

Yes. Source inspection and the linked issue show an unquoted Windows absolute `memory.qmd.command` enters `splitShellArgs`, whose unquoted backslash branch consumes separators before the resolved command reaches QMD spawn resolution.

## Solution Assessment

Yes, this is the right boundary: `memory.qmd.command` is an executable path, so preserving Windows absolute paths before POSIX-style shell parsing is narrower than changing shared shell parsing. It still needs real Windows behavior proof and should be reconciled with the overlapping open PR.
