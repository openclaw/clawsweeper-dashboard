## Summary

Keep open: current main and v2026.6.8 still carry the assistant-media append path that can create a second gateway-injected media transcript row, while the exact raw MEDIA replacement remains only in the active but unmerged PR at https://github.com/openclaw/openclaw/pull/75727.

## What This Changes

The issue asks for WebChat to deliver only one assistant message when a reply contains a MEDIA directive instead of rendering both the raw provider turn and the gateway-injected media turn.

## Best Possible Solution

Land or replace the active gateway/WebChat fix so the latest matching raw MEDIA assistant turn is rewritten into the normalized media-bearing message instead of appending an extra assistant-media row.

## Reproduction Assessment

Yes at source level: current main can append a gateway-injected assistant-media row for a media-bearing agent payload while the raw provider-authored MEDIA assistant turn remains renderable. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes, the best fix shape is to rewrite or replace the latest matching raw MEDIA assistant turn at the transcript boundary rather than adding UI-only suppression. The active PR follows that direction but still needs to become landable or be replaced.
