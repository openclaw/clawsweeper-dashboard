## Summary

Keep open. Current main still has a source-backed duplicate path for model-authored WebChat MEDIA replies: WebChat renders raw MEDIA directives from the Pi-persisted assistant turn, while the gateway also appends a separate `gateway-injected` `:assistant-media` assistant transcript supplement for media-bearing agent-run payloads. The reporter added a MEDIA-only confirmation and another user confirmed the issue, and the earlier closed stale-media duplicate does not cover this raw MEDIA case.

## What This Changes

The issue asks OpenClaw to stop WebChat from delivering both the raw MEDIA assistant message and the gateway-injected assistant-media message for the same assistant output.

## Best Possible Solution

Make the gateway/WebChat transcript path choose a single display owner for model-authored MEDIA directives, with regression coverage for agent-run MEDIA-only and MEDIA-plus-text replies.

## Reproduction Assessment

Yes, at source level. WebChat parses raw MEDIA from Pi-persisted assistant transcript text, and the gateway agent-run path independently appends a `:assistant-media` assistant message for the same media-bearing payload; I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to keep WebChat media supplements only where the gateway owns the displayed message, and skip or suppress the duplicate supplement when the normal Pi assistant turn already carries a renderable MEDIA directive.
