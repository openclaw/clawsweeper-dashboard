## Summary

Keep open. Current `main` still has a source-level path where a parsed `MEDIA:` reply can append a separate `gateway-injected` assistant-media transcript entry while the raw assistant `MEDIA:` turn remains renderable in WebChat, and the focused fix is still the open https://github.com/openclaw/openclaw/pull/75727 rather than merged main behavior.

## What This Changes

The issue reports duplicate WebChat delivery when an assistant reply containing a `MEDIA:` directive is both rendered from raw transcript text and appended as a normalized gateway media message.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75727 with a narrow gateway transcript fix that rewrites or suppresses the matching raw `MEDIA:` assistant turn instead of appending a second visible assistant-media turn.

## Reproduction Assessment

Yes, source-level. Current `main` still has the media-bearing reply supplement append path, raw `MEDIA:` assistant text is independently renderable in WebChat, and attachment-bearing duplicates are not collapsed by the text-only duplicate guard.

## Solution Assessment

Yes. The maintainable fix belongs at the gateway transcript boundary by reconciling the raw `MEDIA:` turn with the normalized media message, not by adding UI-only duplicate hiding.
