## Summary

Keep open: current main still has a plausible duplicate-delivery path where assistant MEDIA text is parsed into a media payload and webchat appends a gateway-injected assistant-media transcript entry, while the display projection/UI do not suppress the original raw MEDIA assistant message.

## What This Changes

The issue asks webchat to deliver only one visible assistant message when a reply contains a MEDIA directive.

## Best Possible Solution

Keep the processed gateway-injected media message for webchat display and suppress the paired raw model MEDIA message from chat history/live rendering without changing transcript persistence or media access validation.

## Reproduction Assessment

Yes, at source level: an assistant MEDIA reply is parsed into mediaUrls and webchat appends a `:assistant-media` injected message, while the original MEDIA text remains renderable and no projection filter handles the pair. A live webchat run would be the final runtime proof for the fix PR.

## Solution Assessment

Yes, the maintainable fix is to suppress the duplicate display projection or live delivery for the raw MEDIA model message when a paired gateway-injected media message exists, preserving the processed media message and the underlying transcript.
