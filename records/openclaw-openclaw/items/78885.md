## Summary

Keep open: current main still has a source-level path that can create a gateway-injected `assistant-media` assistant transcript entry for an agent-run media payload while WebChat can already render the original `MEDIA:` assistant text, matching the reported duplicate delivery.

## What This Changes

The issue reports duplicate WebChat assistant messages when a model response contains a `MEDIA:` directive because both the original model transcript and a gateway-injected media transcript entry render the same media.

## Best Possible Solution

Keep Pi as the owner of normal agent-run assistant transcript entries and only inject WebChat media supplements when they add non-duplicative content such as TTS/audio or non-agent command media.

## Reproduction Assessment

Yes, at source level: a WebChat agent-run final payload carrying parsed media can reach `appendWebchatAgentMediaTranscriptIfNeeded`, while the original assistant `MEDIA:` transcript is also renderable by the UI. A fix PR should still add an executable regression test or live proof for the full path.

## Solution Assessment

Yes, the narrowest maintainable fix is to stop mirroring ordinary model-visible MEDIA replies into a second assistant transcript entry, while preserving the existing supplement cases that are not already represented by the model transcript.
