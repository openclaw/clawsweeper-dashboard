## Summary

Current main still converts account-only channel configuration changes into whole-channel restarts, and the discussion includes credible production evidence of an in-flight Telegram reply being lost during that restart window. This issue should remain open while its same-author closing PR, https://github.com/openclaw/openclaw/pull/43938, is reviewed and either landed or rejected.

## What This Changes

Use account-scoped channel restarts for account-only configuration changes while retaining full-channel restarts for global or mixed changes.

## Best Possible Solution

Carry account IDs through the canonical reload plan and handler, collapse mixed changes to a full-channel restart, preserve existing reload safeguards, and validate the result with focused tests plus a real multi-account Telegram reload scenario.

## Reproduction Assessment

Yes at source level with high confidence. Current-main code and tests deterministically route account-only configuration changes to whole-channel stop/start, and the discussion supplies timestamped production delivery-loss evidence; this review did not perform a fresh live run.

## Solution Assessment

Yes in direction: preserving account identity through the existing account-aware lifecycle is the narrowest maintainable fix and avoids introducing queues, provider-specific retries, or new configuration. The linked implementation still needs proof that mixed-change precedence and existing reload safeguards remain correct.
