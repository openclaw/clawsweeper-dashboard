## Summary

Keep open. Current main still collapses account-scoped channel config changes into whole-channel restart targets, and an open implementation PR already uses closing syntax for this issue but has not merged.

## What This Changes

The issue asks the gateway reload planner to restart only the affected channel account for paths like `channels.<channel>.accounts.<accountId>.*` while preserving full-channel restarts for global or mixed channel changes.

## Best Possible Solution

Land a corrected account-scoped reload implementation that carries account targets through the planner, hot reload, plugin reload, and `secrets.reload` restart/rollback paths while preserving full-channel precedence and active-work deferral.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a change to `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, enters the channel-only `restartChannels` set, and is applied through channel-only stop/start calls; the issue comment adds matching runtime logs.

## Solution Assessment

Yes. Account-scoped reload planning through the existing account-aware lifecycle is the narrow maintainable direction, but the linked implementation still needs to cover all reload-plan consumers and preserve active-work deferral before merge.
