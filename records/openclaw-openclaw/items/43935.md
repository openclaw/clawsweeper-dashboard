## Summary

Current main still converts account-only channel configuration changes into whole-channel restarts, while the discussion provides credible production evidence that this disruption can lose an in-flight Telegram reply. The same-author closing PR remains open, so this issue should stay open until that implementation is reviewed and either landed or rejected.

## What This Changes

Request account-scoped channel hot reloads for changes under `channels.<channel>.accounts.<accountId>.*`, while retaining full restarts for channel-global or mixed changes.

## Best Possible Solution

Land a focused generic account-aware reload plan that deduplicates affected accounts, gives a full-channel restart precedence for global or mixed changes, preserves existing deferral and error handling, and proves active Telegram traffic remains deliverable.

## Reproduction Assessment

Yes at source level: an account-prefixed plugin change currently collapses into a whole-channel restart, and the discussion supplies timestamped production delivery-loss evidence. This read-only review did not execute a live current-main reproduction.

## Solution Assessment

Yes, directionally: extending the generic gateway reload plan is narrower and more maintainable than Telegram-specific handling. The implementation must also preserve full-channel precedence, active-work deferral, plugin coordination, and restart-failure behavior.
