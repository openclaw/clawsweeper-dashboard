## Summary

Keep open: current main and the latest release still route account-scoped channel config changes through channel-wide restart planning, and an open linked PR is the active implementation candidate rather than a merged fix.

## What This Changes

The issue requests account-scoped gateway hot reload for `channels.<channel>.accounts.<accountId>.*` changes so one account restart does not restart the whole channel.

## Best Possible Solution

Land a narrow account-scoped reload implementation that uses the existing account-aware channel lifecycle, preserves full-channel restarts for global or mixed changes, and then close this issue from the merged PR.

## Reproduction Assessment

Yes at source level: current main maps `channels.telegram.accounts.<id>.*` through the broad Telegram reload prefix into `plan.restartChannels`, and the handler restarts the whole channel. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, account-scoped reload planning is the narrowest maintainable fix direction for this issue. The best implementation should be rebased onto the current reload handler shape and prove both account-only and mixed full-channel precedence paths.
