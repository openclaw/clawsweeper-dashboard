## Summary

Keep open. Current main still represents channel reload targets as whole-channel restarts, so an account-scoped Telegram config path still resolves to restarting the full Telegram plugin; the already-open PR #43938 is the paired implementation path and uses closing syntax for this issue.

## What This Changes

Account-scoped channel config changes such as `channels.telegram.accounts.<accountId>.*` should restart only the affected account instead of restarting the whole channel.

## Best Possible Solution

Review, land, or replace PR #43938 with a current-main implementation that adds account-scoped restart targets to `GatewayReloadPlan`, promotes mixed or channel-global changes to full-channel restarts, updates hot reload and secrets reload consumers, preserves active-work deferral, and adds focused planner and handler regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, the planner adds `telegram` to `restartChannels`, and the handler calls channel stop/start without an account id; the discussion also includes matching runtime logs.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable fix for the reported blast radius; Telegram reply retry or drain hardening should remain a separate follow-up only if failures still reproduce after reload granularity is fixed.
