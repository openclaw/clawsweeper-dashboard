## Summary

Current main and the latest release still route account-only channel config changes through whole-channel restart planning and whole-channel stop/start, while the issue has an open same-author closing PR that has not merged. The issue remains necessary until that PR is repaired, proven, and landed or explicitly rejected.

## What This Changes

The issue asks the gateway hot-reload planner to restart only the affected channel account for strictly account-scoped config changes instead of restarting the entire channel.

## Best Possible Solution

Carry account IDs through the canonical reload plan and handler, collapse mixed account/global changes back to a full-channel restart, preserve existing skip/deferral/failure-isolation behavior, and validate with focused tests plus real multi-account Telegram reload proof.

## Reproduction Assessment

Yes at source level with high confidence. Current main deterministically widens account-only reload paths into whole-channel stop/start, and the issue discussion adds timestamped production delivery-loss evidence; I did not perform a fresh live Telegram run.

## Solution Assessment

Yes in direction: preserving account identity through the existing account-aware lifecycle is the narrowest maintainable fix and avoids adding queues, provider-specific retry policy, or new config. The linked PR still needs conflict repair and real behavior proof before it can close the issue.
