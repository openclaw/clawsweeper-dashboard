## Summary

Keep open. Current main still collapses channel account config changes into whole-channel restart targets, and the open implementation PR #43938 uses closing syntax for this issue, so the issue should stay open until that PR is landed, replaced, or closed.

## What This Changes

The issue requests account-scoped gateway channel reloads for changes under `channels.<channel>.accounts.<accountId>.*` instead of restarting the whole channel.

## Best Possible Solution

Carry account-scoped restart targets through reload planning and each channel-restart consumer, while preserving full-channel precedence for channel-global, plugin reload, and mixed changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, the planner adds `telegram` to `restartChannels`, and the hot reload handler calls channel stop/start without an account id; the discussion also includes matching runtime logs.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable fix for the reported blast radius; Telegram reply retry or drain hardening should remain separate unless failures still reproduce after reload granularity is fixed.
