## Summary

Keep open. Current main still routes account-scoped channel config paths through channel-wide restart targets, and the issue is paired with an open implementation PR that uses closing syntax but has not merged.

## What This Changes

The issue requests account-scoped gateway reloads for `channels.<channel>.accounts.<accountId>.*` changes instead of restarting the entire channel.

## Best Possible Solution

Land a corrected account-scoped reload implementation that carries account targets through the planner, hot reload, plugin reload, and secrets reload rollback paths while preserving full-channel precedence and active-work deferral.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, enters the channel-only `restartChannels` set, and is applied through channel-only stop/start calls; the issue discussion adds matching runtime logs.

## Solution Assessment

Yes. Account-scoped reload planning through the existing account-aware lifecycle is the narrow maintainable direction, but the implementation must update every reload-plan consumer rather than only the primary hot-reload handler.
