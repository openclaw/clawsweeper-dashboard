## Summary

Keep open. Current main still collapses account-scoped channel config changes into whole-channel restart targets, and the linked implementation PR is open and unmerged with prior review blockers.

## What This Changes

The issue requests account-scoped gateway reloads for `channels.<channel>.accounts.<accountId>.*` changes instead of restarting the entire channel.

## Best Possible Solution

Land a corrected account-scoped reload implementation that carries account targets through hot reload and `secrets.reload` rollback paths, preserves full-channel precedence and active-work deferral, and includes regression plus real gateway/channel proof.

## Reproduction Assessment

Yes. At source level, `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad `channels.telegram` reload prefix, the planner adds only `telegram` to `restartChannels`, and hot reload calls channel stop/start without an account id; the issue discussion also includes matching runtime logs.

## Solution Assessment

Yes, account-scoped reload planning through the existing account-aware channel lifecycle is the narrow maintainable direction. The current linked PR is not merge-ready until it preserves deferral behavior and updates every reload-plan consumer, especially `secrets.reload`.
