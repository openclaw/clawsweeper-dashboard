## Summary

Keep open. Current main still maps account-scoped channel config changes into whole-channel restart targets, and open PR #43938 already uses closing syntax for this issue, so this should stay open until that PR lands, is replaced, or is closed.

## What This Changes

The issue asks the gateway reload planner to restart only the affected channel account for strictly account-scoped `channels.<channel>.accounts.<accountId>.*` config changes.

## Best Possible Solution

Extend reload planning with account-scoped restart targets and route those through existing account-aware channel lifecycle methods, while preserving full-channel precedence for channel-global, mixed, plugin-reload, and explicitly channel-wide changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: `channels.telegram.accounts.engineer.dmPolicy` matches the broad Telegram reload prefix, the plan adds only `telegram` to `restartChannels`, and the hot reload handler restarts Telegram without an account id.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable fix for the reported blast radius; broader Telegram reply retry or drain work should remain separate unless failures still reproduce afterward.
