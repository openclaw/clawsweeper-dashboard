- `impact:message-loss`: The reporter's follow-up logs include failed final Telegram sends during the full-channel restart window.

## Summary

Keep open: current main and the latest release still collapse account-scoped channel-account config edits into whole-channel restarts, and the linked implementation PR is open and conflicting rather than merged. The issue has source-backed evidence plus user logs showing message delivery loss during that broader reload window.

## What This Changes

The issue asks the gateway reload planner to restart only the affected channel account for `channels.<channel>.accounts.<accountId>.*` config changes instead of restarting the whole channel.

## Best Possible Solution

Refresh, review, and land a focused reload-planning fix that carries account ids through the gateway reload plan and uses account-scoped channel lifecycle calls while preserving full-channel restarts for channel-global or mixed changes.

## Reproduction Assessment

Yes at source level: current main still maps account config paths into channel-level restart actions and the reload handler restarts the whole channel. I did not run a live Telegram traffic repro in this read-only review.

## Solution Assessment

Yes, account-scoped reload planning is the narrow maintainable fix because the lower channel manager already supports optional account ids. The linked PR is the right candidate to refresh and review before this issue closes.
