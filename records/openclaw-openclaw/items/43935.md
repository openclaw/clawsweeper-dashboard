## Summary

Keep this issue open: current main and the latest release still route account-scoped channel config changes through channel-wide restarts, and the issue is already paired with an open closing PR that should be refreshed/reviewed rather than replaced by cleanup closure.

## What This Changes

The issue reports that account-scoped channel config changes under `channels.<channel>.accounts.<accountId>.*` trigger a full channel restart instead of an account-scoped restart, widening the disruption window for active traffic.

## Best Possible Solution

Land a refreshed version of the account-scoped reload fix, preserving full-channel restarts for global or mixed changes and proving both planner behavior and the hot-reload handler account targets.

## Reproduction Assessment

Yes at source level: current main maps `channels.telegram.accounts.*` through the broad `channels.telegram` reload prefix into a channel-wide restart, and the handler calls `stopChannel`/`startChannel` without an account id. I did not run live Telegram traffic in this read-only review.

## Solution Assessment

Yes, the account-scoped reload direction is the narrow maintainable fix because `server-channels.ts` already supports account-scoped lifecycle calls; durable outbound retry or drain behavior can remain separate if live proof shows another failure mode.
