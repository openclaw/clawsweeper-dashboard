- `P1`: The report ties a gateway reload granularity bug to live channel traffic disruption and dropped Telegram replies during active use.
- `impact:message-loss`: The issue and follow-up comment include Telegram `sendMessage` and final reply failures after an account-scoped config patch triggered a whole-channel restart.

## Summary

Keep open: current main and the latest release still collapse account-scoped channel config changes into whole-channel reloads, and the same-author closing PR remains open and conflicting rather than merged.

## What This Changes

The issue asks the gateway reload planner to restart only the affected channel account for `channels.<channel>.accounts.<accountId>.*` changes instead of restarting every account on that channel.

## Best Possible Solution

Land or replace the linked narrow gateway fix so reload planning carries account-scoped restart targets and falls back to whole-channel restarts only for channel-global or mixed changes.

## Reproduction Assessment

Yes for the gateway reload path: a `channels.telegram.accounts.<accountId>.*` change matches the broad channel reload prefix, the plan stores only the channel ID, and the hot reload handler calls whole-channel start/stop. I did not run a live Telegram delivery-loss repro in this read-only review.

## Solution Assessment

Yes, the linked approach of carrying account-scoped restart targets through the reload plan is the narrowest maintainable fix because the channel manager already supports account-scoped lifecycle calls. The PR still needs conflict resolution and real behavior proof before this issue can close.
