## Summary

Keep open: current main and the latest release still route account-scoped channel config changes through whole-channel restart planning, and the linked fixing PR is open rather than merged. The issue has source-level reproduction plus a user log showing dropped Telegram replies during the restart window, so conservative cleanup should leave it open for the linked PR or a replacement fix.

## What This Changes

The issue asks the gateway reload planner to restart only the changed channel account for `channels.<channel>.accounts.<accountId>.*` edits instead of restarting the entire channel.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/43938`, or a rebased replacement, with account-scoped reload planning, regression coverage for account-only versus channel-global/mixed changes, and real Telegram proof for the delivery-loss scenario.

## Reproduction Assessment

Yes at source level: current main maps `channels.telegram.accounts.*` changes to a channel restart and the reload handler restarts the whole channel. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, account-scoped reload planning is the narrowest maintainable fix for the reported blast radius. The linked PR should still prove that channel-global and mixed changes keep the full-channel path, and that Telegram delivery improves in a real setup.
