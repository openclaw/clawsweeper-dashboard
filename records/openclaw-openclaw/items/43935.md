## Label Justifications

- `P1`: This is a broken gateway/channel hot-reload path that can interrupt active Telegram traffic and drop in-flight replies.
- `impact:message-loss`: The issue includes failed Telegram `sendMessage` and final reply logs immediately after an account-scoped config edit restarted the whole channel.

## Summary

Keep open: current main and the latest release still collapse account-scoped channel config paths into whole-channel restarts, and the direct fix PR is still open and conflicting rather than merged.

## What This Changes

The issue requests account-scoped gateway hot reloads for `channels.<channel>.accounts.<accountId>.*` config changes so unrelated accounts on the same channel are not restarted.

## Best Possible Solution

Land or replace the direct account-scoped reload fix so the planner carries channel/account restart targets and full-channel restarts still win for channel-global or mixed changes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: an account-scoped Telegram path matches the broad `channels.telegram` reload prefix, becomes a channel-only `restartChannels` entry, and the handler performs whole-channel stop/start.

## Solution Assessment

Is this the best way to solve the issue? Yes; carrying account-scoped restart targets through the generic gateway planner and handler is the narrow maintainable fix because the lower channel manager already accepts optional account ids.
