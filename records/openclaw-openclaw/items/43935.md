## Best Possible Solution

Review, rebase or repair, and prove the linked account-scoped reload PR while preserving full-channel restart precedence for channel-global or mixed changes.

## Reproduction Assessment

Yes, at source level: current main maps account-scoped channel config paths into a channel-only restart set and the handler calls whole-channel stop/start. I did not run a live Telegram repro in this read-only review, but the issue discussion includes concrete live failure logs on the same path.

## Solution Assessment

Yes, the best fix direction is to carry account ids through reload planning and use the existing account-scoped channel lifecycle, while collapsing to full-channel restart for channel-global or mixed changes. The linked PR is the right implementation candidate, but it still needs review, mergeability repair, and real behavior proof before this issue can close.
