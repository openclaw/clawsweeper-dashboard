## Summary

Current main still lacks a landed, shared guard that strips standalone OpenAI-style `tool_calls` JSON before Telegram partial-preview delivery, and the linked fix attempts for this exact leak closed unmerged. This is still valid but should stay with maintainer review because it is privacy-sensitive and overlaps an open adjacent suppression PR.

## What This Changes

The issue asks OpenClaw to prevent raw internal `tool_calls` JSON from appearing in Telegram direct-chat replies during background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware sanitizer for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with regressions for complete, incomplete, inline, fenced, ordinary JSON, and phase-marked tool-use cases.

## Reproduction Assessment

Yes. The live provider race is not minimized, but the source-level path is clear: a visible lead-in followed by bare `{"tool_calls":[...]}` can bypass the current shared sanitizer and reach `onPartialReply`.

## Solution Assessment

Yes. A narrow shared sanitizer plus preview-surface guard is the best maintainable fix; provider-specific tinyrouter handling or relying only on phase-marked suppression would leave the unphased plain-text payload class exposed.
