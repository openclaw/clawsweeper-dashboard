## Summary

Keep open. Current main still does not strip bare `{"tool_calls":[...]}` JSON from the shared assistant-visible sanitizer, and the streaming preview path can still pass cleaned text to channel previews without a bare-JSON guard. The new commentary-phase suppression on main is useful, but it only covers phase-marked commentary/tool-use messages and does not prove this unphased plain-text payload class is fixed.

## What This Changes

The issue reports that bare OpenAI-style `tool_calls` JSON can be rendered to users during Telegram direct-chat background `exec` and `process poll` flows.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON and use it before both final visible delivery and partial preview callbacks, with regressions for complete, incomplete, inline, fenced, ordinary JSON, and phase-marked commentary cases.

## Reproduction Assessment

Yes. The live provider race is not minimized, but there is a high-confidence source reproduction: a visible lead-in followed by bare `{"tool_calls":[...]}` bypasses `stripDowngradedToolCallText()` and can flow through `handleMessageUpdate()` into `onPartialReply`.

## Solution Assessment

Yes. A narrow shared sanitizer plus partial-preview guard is the best maintainable fix; provider-specific tinyrouter handling or relying only on commentary-phase suppression would leave unphased plain-text payloads exposed.
