## Summary

Keep open. Current main still leaves the reported standalone OpenAI-style `tool_calls` JSON class outside the shared visible-text sanitizer and can forward assistant partial text into Telegram previews; the related fix attempts are closed/unmerged, while the open adjacent PR covers tool-use/commentary suppression rather than this bare-JSON payload path.

## What This Changes

The issue asks OpenClaw to prevent raw internal `tool_calls` JSON from appearing in user-visible Telegram replies during background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with focused regressions for complete payloads, incomplete fragments, ordinary JSON, code examples, and Telegram partial streaming.

## Reproduction Assessment

Yes. A visible lead-in followed by bare `{"tool_calls":[...]}` still bypasses `stripDowngradedToolCallText()` and can flow through `handleMessageUpdate()` to Telegram `onPartialReply`; I did not rerun the live Telegram/tinyrouter setup.

## Solution Assessment

Yes. The narrow maintainable fix is shared streaming-aware sanitization plus a preview-path guard; provider-specific TinyRouter handling or diagnostics-only warnings would leave the plain-text payload class exposed.
