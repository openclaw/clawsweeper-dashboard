## Summary

Keep open. Current main still has the same source-level leak path: standalone OpenAI-style `tool_calls` JSON is not stripped by the shared visible-text sanitizer and can still be forwarded through Telegram partial preview streaming; the related fix attempts are closed/unmerged or cover adjacent tool-use text suppression rather than this bare-JSON payload class.

## What This Changes

The issue asks OpenClaw to prevent raw internal `tool_calls` JSON from becoming user-visible during Telegram direct background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with focused regressions for complete, incomplete, ordinary JSON, code examples, and Telegram partial streaming.

## Reproduction Assessment

Yes. A visible lead-in followed by bare `{"tool_calls":[...]}` still bypasses the shared sanitizer and can flow from `handleMessageUpdate()` through Telegram `onPartialReply`; I did not rerun the live Telegram/tinyrouter setup.

## Solution Assessment

Yes. The narrow maintainable direction is a shared streaming-aware sanitizer plus a preview-streaming guard; provider-specific tinyrouter handling or diagnostics-only warnings would leave the plain-text payload class exposed.
