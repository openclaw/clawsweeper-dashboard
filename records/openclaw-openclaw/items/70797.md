## Summary

Keep open. Current main still has a source-reproducible path where bare OpenAI-style `tool_calls` JSON bypasses the assistant-visible sanitizer and can be forwarded through partial preview delivery; the commentary-phase suppression and fake-tool diagnostics cover adjacent cases but not this unphased plain-text payload class.

## What This Changes

The issue asks OpenClaw to prevent raw `tool_calls` JSON from appearing in Telegram/direct user replies during background exec/process polling.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with regressions for complete, incomplete, inline, fenced, ordinary JSON, and phase-marked commentary cases.

## Reproduction Assessment

Yes, as a source-level reproduction: a visible lead-in followed by bare `{"tool_calls":[...]}` bypasses `stripDowngradedToolCallText()` and can flow through `handleMessageUpdate()` to `onPartialReply`. I did not run the live Telegram/tinyrouter path in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is a shared sanitizer plus streaming-preview guard; provider-specific tinyrouter handling, diagnostics-only logging, or commentary-phase suppression would leave unphased plain-text payloads exposed.
