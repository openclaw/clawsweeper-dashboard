## Summary

Keep open. Current main still leaves the reported source-level leak path: standalone OpenAI-style `tool_calls` JSON is not stripped by the shared visible-text sanitizer and can still be forwarded through Telegram partial preview streaming.

## What This Changes

The issue reports raw OpenAI-style tool-call JSON becoming visible in Telegram partial streaming during background exec/process polling.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with focused regressions for complete, incomplete, ordinary JSON, inline/fenced examples, and commentary/final-answer phase handling.

## Reproduction Assessment

Yes, source-level. A visible lead-in followed by bare `{"tool_calls":[...]}` still bypasses the shared sanitizer and can flow through `handleMessageUpdate()` to Telegram `onPartialReply`; I did not run the live Telegram/tinyrouter path.

## Solution Assessment

Yes, the best direction is a narrow shared streaming-aware sanitizer plus a preview-streaming guard. Diagnostics-only warning, provider-specific tinyrouter handling, or commentary-phase suppression alone would leave the unphased plain-text payload class exposed.
