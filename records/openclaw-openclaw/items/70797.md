## Summary

Keep open. Current main still has the source-level leak path: bare OpenAI-style `tool_calls` JSON bypasses the shared visible-text sanitizer and can still be forwarded through the partial reply callback used by Telegram preview streaming.

## What This Changes

The issue reports raw OpenAI-style tool_call JSON becoming visible in Telegram direct-chat partial or final assistant replies after background exec/process polling.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with regressions for complete, incomplete, inline, fenced, ordinary JSON, and phase-marked commentary cases.

## Reproduction Assessment

Yes, source-level: a visible lead-in followed by bare `{"tool_calls":[...]}` still bypasses the shared sanitizer and can flow to `onPartialReply`. I did not run the live Telegram/tinyrouter path in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow shared sanitizer plus a preview-streaming guard. Provider-specific tinyrouter handling, diagnostics-only logging, or commentary-phase suppression alone would leave the unphased plain-text payload class exposed.
