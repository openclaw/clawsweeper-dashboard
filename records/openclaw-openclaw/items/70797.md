## Label Justifications

- `P1`: The report describes a recurring user-visible leak in a live Telegram agent workflow involving background exec/process polling.
- `impact:security`: The leaked payload can expose internal tool arguments, session IDs, and orchestration details to a chat user.
- `impact:message-loss`: The affected surface is channel-visible assistant delivery, where raw internal payloads can be emitted as user messages.

## Summary

Keep open. Current main and the latest stable release still leave the reported standalone OpenAI-style `tool_calls` JSON outside the shared visible-text sanitizer, and Telegram partial streaming still forwards assistant text into visible preview callbacks; the direct fix attempts were closed unmerged while the open adjacent PR covers a different tool-use/commentary leak class.

## What This Changes

The issue reports raw OpenAI-style `tool_calls` JSON becoming visible to Telegram users during background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware guard for standalone OpenAI-style `tool_calls` JSON before both final visible delivery and partial preview callbacks, with focused regressions and redacted live Telegram or equivalent channel proof.

## Reproduction Assessment

Yes, at source level: a visible lead-in followed by bare `{"tool_calls":[...]}` still bypasses `stripDowngradedToolCallText()` and can flow through `handleMessageUpdate()` to `onPartialReply`. I did not rerun the live Telegram/tinyrouter setup.

## Solution Assessment

Yes. The narrow maintainable fix is shared streaming-aware sanitization plus a preview-path guard; provider-specific TinyRouter handling or diagnostics-only warnings would leave the plain-text payload class exposed.
