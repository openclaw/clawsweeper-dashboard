## Summary

Keep this PR open: it targets a real P1 gateway restart-loop bug, but the new dueling detector is overbroad and can classify an unrelated marker-owned system unit as authoritative, which would let doctor remove a legitimate user-scope gateway unit.

## What This Changes

The PR adds Linux systemd gateway installation detection, doctor cleanup of simultaneous user/system units, service-mode dueling warnings, and regression tests.

## Best Possible Solution

Land a narrower systemd repair that only treats proven same-gateway user+system units as dueling, preserves intentional isolated gateways, and includes real Linux doctor cleanup proof.

## Reproduction Assessment

Yes for the source-level bug: current main's lifecycle detector returns the user unit first when both canonical scopes exist, so the dueling state cannot be surfaced there. I did not live-reproduce the WSL2 systemd restart loop in this read-only review.

## Solution Assessment

No: the direction is right, but the PR should not let the marker-owned system-unit fallback drive destructive dueling cleanup without proving it is the same gateway. A narrower canonical or same-port detector is the safer fix.
