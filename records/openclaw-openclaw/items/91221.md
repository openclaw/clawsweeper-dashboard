## Summary

Keep open: this PR addresses a real Linux systemd gateway restart-loop bug that is still present on current main and the latest release. I found no concrete code-review blocker, but the branch intentionally makes `doctor --fix` remove the same-name user-scope unit and treat the system-scope unit as authoritative, so maintainer acceptance of that upgrade policy is still needed.

## What This Changes

Adds Linux systemd dueling-scope detection, doctor cleanup of the redundant user-scope gateway unit, a service-mode warning, and regression tests.

## Best Possible Solution

Land one focused systemd fix for the linked restart cascade after maintainers choose the service-ownership policy and decide between this doctor-cleanup approach and the broader owner-selection approach in the related open PR.

## Reproduction Assessment

Yes. Current main still has a source-level reproduction because `findInstalledSystemdGatewayScope` returns the user unit before checking system-scope units, and the linked issue plus PR proof describe the live restart cascade; I did not run a live systemd reproduction in this read-only review.

## Solution Assessment

Yes, this is an acceptable fix shape if maintainers accept system-scope authority for same-name conflicts. It stays in the systemd/doctor service-management layer instead of changing Telegram behavior or stale-PID matching, but the related open PR presents a broader owner-selection alternative that maintainers should compare.
