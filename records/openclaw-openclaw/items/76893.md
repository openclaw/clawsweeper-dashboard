## Summary

Keep open; the PR addresses a real heartbeat loop exposure, but it does not yet cover the separate client-tool loop-detection path and leaves user-facing docs/changelog inconsistent with the new default.

## What This Changes

The PR adds a heartbeat-trigger default to tool-loop config resolution, threads it into OpenClaw coding tools, and adds resolver tests.

## Best Possible Solution

Apply the heartbeat-only default consistently to OpenClaw-managed and client-provided tool paths, document the heartbeat exception, add a changelog entry, and keep harder no-tools or token-cap changes as separate follow-ups.

## Reproduction Assessment

Yes, from source: current main resolves no loop-detection config for unconfigured runs, `detectToolCallLoop` defaults disabled, and both relevant resolver call sites omit heartbeat trigger information. The linked incident supplies real heartbeat-loop evidence, but I did not run a live model reproduction in this read-only review.

## Solution Assessment

No as submitted; enabling the existing guard for heartbeat runs is the narrow maintainable direction, but it must cover the client-tool resolver path and update docs/changelog before merge.
