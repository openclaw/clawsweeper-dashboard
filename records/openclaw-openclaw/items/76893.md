## Summary

Keep open: the PR targets a real heartbeat safety gap, but it does not apply the new heartbeat default to the separate client-tool loop-detection path and would leave user-facing docs/changelog inaccurate.

## What This Changes

The branch changes `resolveToolLoopDetectionConfig` to default loop detection on for heartbeat triggers, threads that trigger through OpenClaw coding tools, and adds focused resolver tests.

## Best Possible Solution

Apply the heartbeat-only default consistently to both core/plugin and client-tool loop detection, add regression coverage for both paths, and document the heartbeat exception while leaving hard no-tools and token-cap work as separate follow-ups.

## Reproduction Assessment

Yes. Source inspection shows no-config loop detection resolves disabled today, heartbeat trigger is available to the runner, and the linked incident context describes real heartbeat tool-call loops that this existing detector class is meant to bound.

## Solution Assessment

No as submitted. Enabling the existing detector for heartbeat runs is a narrow maintainable direction, but the PR must cover the separate client-tool resolver path and update docs/changelog before merge.
