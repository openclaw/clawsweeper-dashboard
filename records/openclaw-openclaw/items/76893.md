## Summary

Keep open: the PR addresses a real current-main safety gap, but it is not merge-ready because one heartbeat tool path still ignores the trigger and the user-facing default change is undocumented.

## What This Changes

The PR enables existing tool-loop detection by default for heartbeat-triggered runs when no explicit loop-detection setting exists, and adds resolver tests for that behavior.

## Best Possible Solution

Apply the heartbeat-only default consistently to all heartbeat tool-loop detection paths, add regression coverage for both core/plugin and client tools, and ship the behavior with docs and changelog updates; keep hard no-tools or token-cap work as separate follow-ups.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source: current main ignores the heartbeat trigger when resolving loop detection, no-config detection resolves disabled, and #21597 supplies real heartbeat loop incidents.

## Solution Assessment

Is this the best way to solve the issue? No as submitted; the runtime direction is narrow and maintainable, but it needs to cover the separate client-tool resolver path and update user-facing docs/changelog before merge.
