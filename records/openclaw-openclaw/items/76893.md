## Summary

Keep open: current main still does not implement the heartbeat-specific rolling loop-detection default, but this PR is not mergeable as submitted because it only updates the built-in/plugin tool path, misses other resolver paths and user-facing docs/help, and still lacks after-fix real heartbeat proof. ## What This Changes

The PR adds a heartbeat-trigger override intended to enable tool-loop detection by default for heartbeat runs and adds resolver tests for that override.

## Best Possible Solution

Implement the heartbeat default in the shared resolver on current main, pass the trigger through every loop-detection resolver path, update the public docs/config help, and keep the merged post-compaction guard as a separate safety net.

## Reproduction Assessment

Yes, from source: rolling loop detection still defaults off, and current main has no trigger-aware resolver path for heartbeat runs. I did not run a live heartbeat replay in this read-only review.

## Solution Assessment

No as submitted. The direction is narrow and plausible, but the patch must be rebased onto the shared resolver, cover client and tool-search paths, sync user-facing surfaces, and include real after-fix heartbeat proof before it is the best mergeable fix.
