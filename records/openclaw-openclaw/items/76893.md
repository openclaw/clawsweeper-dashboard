## Summary

Keep open; the patch takes a good narrow direction for the heartbeat burn described in #21597, but it still misses the separate client-tool loop-detection path and would leave user-facing docs/changelog inaccurate.

## What This Changes

The PR adds a trigger-aware heartbeat default to `resolveToolLoopDetectionConfig`, passes `options.trigger` through `createOpenClawCodingTools`, and adds resolver coverage for that default.

## Best Possible Solution

Apply the heartbeat-only default consistently to builtin/plugin and client-tool loop detection, document the heartbeat exception and explicit opt-out behavior, and leave hard no-tools or token-ceiling work to separate follow-ups.

## Reproduction Assessment

Yes, source-reproducible: current main carries heartbeat trigger context but the loop-detection resolver ignores it, and missing loop config resolves disabled. The linked #21597 supplies real heartbeat-loop incident evidence.

## Solution Assessment

No as submitted. The heartbeat-only default is a narrow maintainable fix direction, but the PR needs to cover the client-tool resolver path and update user-facing docs/changelog before merge.
