## Summary

Keep this PR open. Current main still has the prompter boundary and setup `.trim()` assumptions that the PR targets, and the proposed patch is a small hardening change with focused regression coverage; it is not obsolete or implemented on main.

## What This Changes

The PR normalizes undefined text prompt results in the Clack prompter and channel setup token/text-input paths, and adds regression tests for undefined text responses.

## Best Possible Solution

Land a narrow hardening change that makes `WizardPrompter.text` implementations consistently return strings, with regression tests for undefined text results; keep downstream guards only where they preserve existing setup semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR-targeted contract failure: a mocked `WizardPrompter.text` or Clack `text()` result of `undefined` reaches `rawValue.trim()` and `promptSingleChannelToken().trim()` on current main. The original live Clack-backed onboarding crash is less certain on current main because #67086 fixed a separate docsPath trim crash and Clack 1.2.0 finalizes undefined text input to an empty string.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: normalizing at `createClackPrompter` enforces the exported `WizardPrompter.text` contract, and the added tests are focused. The maintainer choice is whether the downstream fallback guards should stay or whether the patch should be narrowed to the prompter boundary plus tests.
