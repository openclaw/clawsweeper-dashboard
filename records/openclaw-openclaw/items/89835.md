## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not treat missing auth mode as OAuth usage:** `src/infra/provider-usage.limits.ts:27-31`
  - body: `agent-runner` passes only `runResult.meta?.requestShaping?.authMode` into the limits resolver, and current CLI/embedded runners omit that field for ordinary API-key/env turns. This branch then maps `undefined` to `oauth`, so an OpenAI API-key reply can pick up ambient OAuth/ChatGPT usage windows from `loadProviderUsageSummary` and show the wrong 📊 limits. Derive the active auth mode at snapshot time, or treat a missing signal as unknown/no usage.
  - confidence: 0.91
- **[P2] Remove the session-key usage snapshot fallback:** `src/auto-reply/reply/reply-usage-state.ts:46-55`
  - body: The new hook docs say `usageState` is absent on routed durable deliveries and recovered queue replays, but this lookup falls back to `sk:<sessionKey>` whenever a run id is missing. Any delivery without the original run id can therefore receive a recent run's snapshot for the same session, or a concurrent run's data, and render stale usage in plugins or the footer. Correlate by run id only for live delivery, or otherwise make the same-run ownership explicit and tested.
  - confidence: 0.86
- **[P2] Move template file polling out of reply delivery:** `src/auto-reply/usage-bar/template.ts:47-57`
  - body: `loadUsageBarTemplate` is called while building the `/usage full` reply, and it synchronously `statSync`s the configured path on every render, with a synchronous read when the mtime changes. A slow home directory, network mount, or blocked path can now stall visible replies for users who enable the feature. Resolve/cache templates through config lifecycle or another nonblocking owner path instead of polling from delivery.
  - confidence: 0.9
- **[P2] Document the new usage template contract:** `src/config/zod-schema.session.ts:160`
  - body: This line accepts a new public `messages.usageTemplate` config key, and the branch also adds `reply_payload_sending.usageState`, but the docs still only describe the existing `/usage full` footer and hook behavior. Operators and plugin authors need the template schema, fallback semantics, path behavior, and hook-field availability documented before depending on the new surface.
  - confidence: 0.82
