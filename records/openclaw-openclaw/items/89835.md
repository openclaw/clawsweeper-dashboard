## Summary

Strong live proof shows the configurable footer working on Telegram and Discord, but the branch is not merge-ready: a missing template path can repeatedly perform synchronous filesystem reads in reply delivery, and the new config/plugin contracts are incomplete and need explicit owner approval.

## What This Changes

The PR adds an opt-in core templating DSL for `/usage full`, a per-turn `reply_payload_sending.usageState` contract, credential-aware provider-limit data, template caching, configuration, and focused tests.

## Best Possible Solution

First land or reject the narrower usage-state contract with complete public documentation and delivery/auth tests; then make the renderer lifecycle-owned, document and validate the template format, and keep all filesystem loading out of reply delivery before reconsidering the core feature.

## Reproduction Assessment

Yes for the introduced defect: configure `messages.usageTemplate` to a missing or unreadable path, and source inspection shows every footer render re-enters synchronous file loading because no failure entry is cached. The requested capability itself is a feature, not a current-main bug.

## Solution Assessment

No. A data-only renderer is a plausible direction and the visible result is proven, but this branch combines multiple permanent contracts, violates the reply hot-path invariant for bad paths, and omits required public config/plugin contract work.
