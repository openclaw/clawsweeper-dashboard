## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Do not treat missing auth mode as OAuth usage:** `src/infra/provider-usage.limits.ts:29`
  - body: `toUsageCredentialType(undefined)` returns `oauth`, so `getProviderUsageLimitsCached("openai", { credentialType: undefined })` can return a fresh cached OAuth quota window before any credential re-check. Current CLI and embedded run metadata only sets `requestShaping.authMode` for auth-profile runs, so env/API-key OpenAI turns can hit this missing-auth path and show ChatGPT/OAuth limits incorrectly; missing auth should stay ineligible unless the resolved turn auth is explicitly OAuth/token/auth-profile, with a cached-value regression test.
  - confidence: 0.91
- **[P2] Move template file checks out of reply delivery:** `src/auto-reply/usage-bar/template.ts:45`
  - body: `loadUsageBarTemplate` still runs synchronous `statSync` on every configured `/usage full` reply and may `readFileSync` when the mtime changes. That means a slow, remote, or stuck template path can block visible reply delivery; load or refresh this template outside the hot reply path and hand the renderer a prepared snapshot.
  - confidence: 0.86
- **[P2] Document the usage template and usageState contracts:** `src/config/types.messages.ts:139-148`
  - body: This adds the public `messages.usageTemplate` config contract and the plugin-facing `reply_payload_sending.usageState` event shape, but the user/plugin docs still describe only the existing `/usage` behavior and reply-payload rewriting. Operators and plugin authors need the template schema, path/fail-open behavior, optional-live-only `usageState` semantics, and limits/auth caveats before relying on these surfaces.
  - confidence: 0.83
