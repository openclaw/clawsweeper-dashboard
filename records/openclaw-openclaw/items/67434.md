## Summary

Keep open. Current main still leaves the reported gap: GPT-5 OpenAI/OpenAI Codex strict-agentic runs auto-expose `update_plan`, and the default runtime paths I checked still do not cap pure consecutive `update_plan`-only activity before a non-plan tool resumes.

## What This Changes

The issue asks whether massive consecutive `update_plan` bursts on long GPT-5/Codex strict-agentic runs are expected or should be fixed separately from the earlier `update_plan` regression.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only activity as non-progress before hundreds of calls can accumulate, with focused regression coverage and docs/changelog updates for the operator-visible behavior.

## Reproduction Assessment

Yes, for the source-level bug shape: simulate a strict-agentic OpenAI/Codex GPT-5 run whose attempt records repeated `update_plan` tool metas with no non-plan tool activity and no assistant text, then assert the runner retries or blocks. A live provider replay was not run here.

## Solution Assessment

No patch is under review. The config workaround `tools.experimental.planTool=false` avoids the flood, but the narrow maintainable fix is a runtime guard plus tests rather than relying on operator opt-out.
