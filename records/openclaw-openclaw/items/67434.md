## Summary

Keep open. Current `main` still shows the reported runtime gap: strict-agentic GPT-5 OpenAI/OpenAI Codex runs can auto-expose `update_plan`, but the checked guards do not provide a default cap or retry path for a pure consecutive `update_plan`-only burst before real tool progress resumes.

## What This Changes

The issue reports excessive consecutive `update_plan` tool calls on long GPT-5/OpenAI Codex strict-agentic runs before normal tool use resumes.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only tool activity as non-progress and blocks or retries before a large burst can accumulate, while preserving `tools.experimental.planTool=false` and normal interleaved plan updates.

## Reproduction Assessment

Yes, for source-level reproduction: simulate a strict-agentic OpenAI/OpenAI Codex GPT-5 run with repeated `update_plan`-only tool activity and assert the runner blocks or retries before the next non-plan tool. I did not establish a live provider replay in this read-only review.

## Solution Assessment

No patch is under review. The narrow maintainable solution is a consecutive `update_plan` non-progress guard with focused regression coverage; the documented config opt-out is only an operator mitigation.
