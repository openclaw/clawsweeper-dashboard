## Summary

Keep this issue open. Current main still exposes `update_plan` automatically for strict-agentic OpenAI/OpenAI Codex GPT-5 runs and documents the opt-out, but the default runtime path still lacks a cap or block/retry guard for long consecutive `update_plan`-only tool bursts before the next real tool.

## What This Changes

The issue requests stronger runtime and documentation handling for excessive consecutive `update_plan` calls during long strict-agentic GPT-5/Codex runs.

## Best Possible Solution

Add a default-on runtime guard for strict-agentic OpenAI/OpenAI Codex GPT-5 runs that treats consecutive `update_plan`-only activity as non-progress, then retries or blocks before hundreds of plan-only tool calls can accumulate. Keep the documented `tools.experimental.planTool=false` opt-out as mitigation, but do not rely on it as the fix.

## Reproduction Assessment

Yes, for the source-level remaining gap: simulate a strict-agentic OpenAI/Codex GPT-5 run with repeated `update_plan`-only tool activity and assert that the runner retries or blocks before the next non-plan tool. A live GPT-5 reproduction was not run, but the current code path has no default cap for the reported shape.

## Solution Assessment

Not applicable to a current patch; the existing config opt-out is a workaround, not the best runtime solution. The narrow maintainable fix is a dedicated `update_plan`-only progress guard with regression coverage in the embedded Pi/tool-loop layer.
