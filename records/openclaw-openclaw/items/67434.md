## Summary

Keep open. Current main still leaves the reported path possible: GPT-5 OpenAI/OpenAI Codex runs can receive `update_plan` by default, empty `update_plan`-only turns still miss the planning-only retry helper, and rolling tool-loop detection remains opt-in; the closest fix path is still an open PR, not merged main behavior.

## What This Changes

The issue asks for runtime protection against excessive consecutive `update_plan`-only bursts on long GPT-5/OpenAI Codex strict-agentic runs.

## Best Possible Solution

Add default-on strict-agentic protection for consecutive `update_plan`-only non-progress, reset it on real non-plan tool progress, preserve `tools.experimental.planTool=false`, and cover empty tool-only bursts with regression tests.

## Reproduction Assessment

Yes, source-level: current main can still expose `update_plan` by default for supported GPT-5 OpenAI/OpenAI Codex runs, while empty `update_plan`-only turns fall through the planning-only retry helper and rolling loop detection is disabled unless configured. I did not establish a live provider replay.

## Solution Assessment

No patch is attached to this issue. The best path is a narrow runtime guard or a revised version of the open `update_plan` churn PR that preserves legitimate plan progress while blocking no-progress bursts.
