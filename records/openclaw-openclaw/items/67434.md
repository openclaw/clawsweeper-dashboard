## Summary

Keep open. Current main and the latest release still leave the reported default path possible: GPT-5 OpenAI/OpenAI Codex strict-agentic runs can receive `update_plan`, pure tool-only planning bursts still miss the planning-only retry guard, and the rolling loop detector is off by default; the related fix PR is still open and conflicted with an existing ClawSweeper blocker.

## What This Changes

The issue reports excessive consecutive `update_plan` calls during long GPT-5/OpenAI Codex tasks and asks whether this remains expected behavior or a bug needing tracking.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only activity as non-progress, resets on real non-plan tool progress, preserves `tools.experimental.planTool=false`, and includes focused regression coverage for pure tool-only bursts.

## Reproduction Assessment

Yes, source-level: strict-agentic OpenAI/OpenAI Codex GPT-5 runs can still expose `update_plan`, while a pure `update_plan`-only turn with no assistant text falls through the planning-only retry helper and the rolling loop detector is disabled unless configured. I did not establish a live provider replay.
