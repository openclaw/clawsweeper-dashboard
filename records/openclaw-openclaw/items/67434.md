## Summary

Keep open. Current `main` still matches the reported remaining runtime gap: GPT-5 OpenAI/OpenAI Codex strict-agentic runs can auto-expose `update_plan`, while the checked guards still do not enforce a default cap on pure consecutive `update_plan`-only tool activity before real tool progress resumes.

## What This Changes

The issue reports long GPT-5/OpenAI Codex tasks producing hundreds of consecutive `update_plan` tool calls before normal tool use resumes.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only tool activity as non-progress and blocks or retries before a large burst can accumulate, while preserving `tools.experimental.planTool=false` as an operator workaround.

## Reproduction Assessment

Yes, for source-level reproduction: simulate a strict-agentic OpenAI/OpenAI Codex GPT-5 run with repeated `update_plan`-only tool activity and assert the runner blocks or retries before the next non-plan tool. I did not establish a live provider replay in this read-only review.

## Solution Assessment

No current patch is under review. The documented config opt-out is only a mitigation; the narrow maintainable fix is a dedicated consecutive `update_plan` non-progress guard with focused regression coverage.
