## Summary

Keep open. Current main still auto-exposes `update_plan` for strict-agentic OpenAI/OpenAI Codex GPT-5 runs, but the default guards still do not cap or retry a pure consecutive `update_plan`-only burst before real tool progress. The open fix PR at https://github.com/openclaw/openclaw/pull/68673 is relevant but unmerged and previously reviewed as needing revision/proof, so it is not a reason to close this issue.

## What This Changes

The issue reports long GPT-5/Codex runs spending many consecutive tool turns on `update_plan` before resuming real tool use.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only activity as non-progress, resets on real non-plan tool progress, preserves `tools.experimental.planTool=false`, and includes focused regression coverage.

## Reproduction Assessment

Yes, source-level: simulate a strict-agentic OpenAI/OpenAI Codex GPT-5 run with repeated `update_plan`-only tool activity and no assistant text; current main does not trip planning-only retry, and generic loop detection is disabled unless configured. I did not establish a live provider replay.

## Solution Assessment

No patch is under review on this issue. The closest open PR, https://github.com/openclaw/openclaw/pull/68673, targets part of the loop hashing problem but is unmerged and has review blockers; the maintainable fix should be a narrow default strict-agentic no-progress guard with regression tests.
