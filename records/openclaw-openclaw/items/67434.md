## Summary

Keep open. Current main still matches the reported remaining gap: GPT-5 OpenAI/OpenAI Codex strict-agentic runs can auto-expose `update_plan`, and the default runtime paths I checked do not cap or retry pure consecutive `update_plan`-only bursts before a non-plan tool call.

## What This Changes

The issue asks OpenClaw to prevent or better document massive consecutive `update_plan` bursts on long GPT-5/OpenAI Codex strict-agentic runs.

## Best Possible Solution

Add a default-on runtime guard for strict-agentic OpenAI/OpenAI Codex GPT-5 runs that treats consecutive `update_plan`-only tool activity as non-progress and retries or blocks before hundreds of plan-only calls can accumulate. Keep `tools.experimental.planTool=false` as an operator workaround, but do not rely on it as the fix.

## Reproduction Assessment

Yes, for the source-level remaining gap: simulate a strict-agentic OpenAI/Codex GPT-5 run with repeated `update_plan`-only tool activity and assert that the runner retries or blocks before the next non-plan tool. A live provider replay was not run here.

## Solution Assessment

No current patch is under review. The documented config opt-out is a mitigation, while the maintainable fix is a dedicated consecutive `update_plan` non-progress guard with focused regression coverage.
