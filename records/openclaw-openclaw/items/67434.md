## Summary

Keep open. Current main and the latest stable release still auto-expose `update_plan` for GPT-5 OpenAI/OpenAI Codex strict-agentic runs, empty plan-only turns can miss the planning retry helper, and the generic loop detector is still disabled by default; the related implementation path remains an open PR rather than merged behavior.

## What This Changes

The issue reports excessive consecutive `update_plan` calls during long GPT-5/OpenAI Codex strict-agentic runs and asks whether runtime/docs should cap or clarify that behavior.

## Best Possible Solution

Add default-on strict-agentic protection for consecutive no-progress `update_plan`-only bursts, reset it on real non-plan tool progress, preserve `tools.experimental.planTool=false`, and fold or revise the open PR rather than creating a duplicate fix path.

## Reproduction Assessment

Yes, source-level. Current main can still auto-expose `update_plan` for supported GPT-5 OpenAI/OpenAI Codex runs, while empty plan-only tool turns and default-disabled loop detection leave the reported no-config burst path possible; I did not run a live provider replay.

## Solution Assessment

No patch is attached to this issue. The best path is to review and likely revise the open `update_plan` churn PR so it provides a default strict-agentic guard rather than relying only on opt-in rolling loop detection.
