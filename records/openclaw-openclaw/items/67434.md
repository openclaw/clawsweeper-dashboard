## Summary

Keep open. Current main still leaves the reported default path possible: GPT-5 OpenAI/OpenAI Codex runs can auto-receive `update_plan`, pure tool-only plan bursts do not trip the planning-only retry helper, and the generic loop detector remains disabled unless configured; the closest fix PR is still open and conflicted.

## What This Changes

The issue asks whether long GPT-5/OpenAI Codex runs should still be allowed to emit very large consecutive `update_plan` bursts before resuming real tool use.

## Best Possible Solution

Add default-on strict-agentic protection for consecutive `update_plan`-only non-progress, reset it on real non-plan tool progress, preserve `tools.experimental.planTool=false`, and cover empty tool-only bursts with regression tests.

## Reproduction Assessment

Yes, source-level: current main can still expose `update_plan` by default for supported GPT-5 OpenAI/OpenAI Codex runs, while an empty assistant-text `update_plan`-only turn falls through the planning-only retry helper and default loop detection is disabled.
