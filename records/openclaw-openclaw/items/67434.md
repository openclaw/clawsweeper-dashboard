## Summary

Keep open. Current main still leaves the reported default path possible: strict-agentic GPT-5 OpenAI/OpenAI Codex runs receive `update_plan`, but pure plan-tool churn with no visible assistant text is not caught by the planning-only retry path, and the rolling loop detector remains disabled by default. The related fix PR at https://github.com/openclaw/openclaw/pull/68673 is relevant but still open and previously reviewed as needing revision/proof, so it is not a close reason.

## What This Changes

The issue asks OpenClaw to prevent very large consecutive `update_plan`-only bursts during long GPT-5/Codex strict-agentic runs.

## Best Possible Solution

Add a default-on strict-agentic guard that treats consecutive `update_plan`-only activity as non-progress, resets on real non-plan tool progress, preserves `tools.experimental.planTool=false`, and includes focused regression coverage.

## Reproduction Assessment

Yes, source-level: a strict-agentic OpenAI/OpenAI Codex GPT-5 run can still expose `update_plan`, while a pure `update_plan`-only turn with no assistant text falls through planning-only retry and the rolling loop detector is disabled unless configured. I did not establish a live provider replay.

## Solution Assessment

No patch is under review on this issue. The closest open PR, https://github.com/openclaw/openclaw/pull/68673, targets part of the loop hashing problem but is unmerged and previously reviewed with a correctness/proof blocker; the maintainable fix should be a narrow default strict-agentic no-progress guard with tests.
