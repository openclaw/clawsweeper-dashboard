## Summary

Keep open: current main still has the unmarked unavailable-approval payload, so the PR is useful, but this head remains incomplete because it handles only one dispatch gate and still lacks after-fix real behavior proof.

## What This Changes

The PR adds `channelData.execApprovalUnavailable` to exec-approval-unavailable reply payloads, allows that marker through the group/native tool-delivery gate, and adds regression tests.

## Best Possible Solution

Treat unavailable approvals as deterministic approval metadata at every dispatch gate, add focused progress-suppression coverage, and require redacted real no-approval-route proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main builds unavailable approval replies as text-only, and the PR-marked payload can still be returned before dispatch when `suppressDefaultToolProgressMessages` is true because the second gate only exempts `execApproval`.

## Solution Assessment

No, not as-is. The marker is the right narrow fix, but it must be recognized by the progress-suppression gate as well, preferably through shared marker-check logic or matching checks with regression coverage.
