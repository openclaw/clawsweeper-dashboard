Keep open: current main still lacks the unavailable-approval marker, and the PR addresses a real source-reproducible delivery blackout. ## What This Changes

The PR adds `channelData.execApprovalUnavailable` metadata to unavailable exec-approval reply payloads and teaches the group/native tool-delivery filter to pass that marker, with focused tests.

## Best Possible Solution

Land a small adjusted patch that marks unavailable approval payloads and treats `execApprovalUnavailable` as deterministic approval metadata at every dispatch gate, with focused regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible. A PR-marked `channelData.execApprovalUnavailable` text payload passes the updated first filter, then can still be returned before dispatch when `suppressDefaultToolProgressMessages` is true because the later gate only exempts media, errors, and `channelData.execApproval`.

## Solution Assessment

No, not as-is. The marker is the narrow maintainable fix, but it also needs to be recognized by the progress-suppression gate, preferably through shared helper logic or matching checks plus regression coverage.
