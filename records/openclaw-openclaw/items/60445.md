## Summary

Keep open: the payload marker is useful, but this PR still leaves a later progress-suppression gate recognizing only `execApproval`, and the contributor has not supplied after-fix real behavior proof.

## What This Changes

The PR adds `channelData.execApprovalUnavailable` to unavailable exec-approval reply payloads, updates the group/native delivery allowlist, and adds unit coverage.

## Best Possible Solution

Treat `execApprovalUnavailable` as deterministic approval metadata at every dispatch gate, add focused regression coverage for progress suppression, and include redacted runtime proof from a real no-approval-route flow.

## Reproduction Assessment

Yes, source-reproducible. A marked `channelData.execApprovalUnavailable` text payload passes the PR-updated allowlist, then can still be returned before dispatch when `suppressDefaultToolProgressMessages` is true because the later gate only exempts `execApproval`.

## Solution Assessment

No, not as-is. The marker is the narrow maintainable fix, but it must be recognized by the progress-suppression gate as well, ideally through shared helper logic or matching checks plus regression coverage.
