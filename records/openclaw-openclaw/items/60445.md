## Summary

Keep open: the PR addresses a real exec-approval delivery gap, but it is not complete because marked unavailable payloads can still be dropped by the later progress-suppression gate, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `channelData.execApprovalUnavailable` to unavailable exec-approval reply payloads and allows that marker through the group/native tool-result delivery filter.

## Best Possible Solution

Land an adjusted version that marks unavailable approval payloads and treats that marker as deterministic approval metadata at every dispatch gate, with focused regression coverage; maintainer-owned release notes can be handled during landing.

## Reproduction Assessment

Yes, from source inspection. A PR-marked `channelData.execApprovalUnavailable` text payload passes the updated `resolveToolDeliveryPayload` branch, then can still be returned before dispatch by `suppressDefaultToolProgressMessages` because that later gate only exempts `channelData.execApproval`.

## Solution Assessment

No, not as-is. The marker approach is the narrow maintainable fix, but the marker needs to be recognized by the progress-suppression gate too, ideally via shared helper logic or matching checks plus a regression beside the existing pending-approval test.
