## Summary

Keep open. Current main still emits exec-approval-unavailable replies as plain text and only preserves pending `execApproval` payloads through the group/native tool-result gate, so this PR addresses a real gap. The patch direction is narrow and useful, but it misses the later `suppressDefaultToolProgressMessages` gate, which can still drop the newly marked unavailable payloads in preview/native flows.

## What This Changes

The PR adds `channelData.execApprovalUnavailable` to unavailable exec approval reply payloads, teaches the tool-result delivery filter to pass those marked payloads, and adds focused regression tests for the builder and dispatch path.

## Best Possible Solution

Land a small adjusted version that keeps the new unavailable-payload marker and treats it as a deterministic approval payload at every relevant delivery gate, including progress-suppression, with focused regression coverage. No new config or public API surface is needed for this fix.

## Reproduction Assessment

Yes. Current main can be inspected directly: unavailable approval payloads have no marker, and text-only tool results without `channelData.execApproval` are suppressed in group/native delivery; the remaining PR bug is reproducible by enabling `replyOptions.suppressDefaultToolProgressMessages` and sending a marked unavailable payload.

## Solution Assessment

No, not as-is. The marker approach is the narrow maintainable fix, but the implementation needs the same `execApprovalUnavailable` recognition in the later progress-suppression gate before it fully solves the blackout class it describes.
