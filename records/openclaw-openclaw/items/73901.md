## Summary

Keep open. Current main still rejects non-allowlisted agent-driven `config.patch` requests before the gateway write RPC, and the requested approval path would relax a shipped GHSA-related fail-closed boundary without an existing config approval contract.

## What This Changes

The issue requests an operator approval gate, plus optional expanded allowlist, for `gateway config.patch` writes outside the current safe path allowlist.

## Best Possible Solution

Design a config-specific approval policy that keeps reject-by-default behavior, separates always-allowed, promptable, and never-promptable paths, shows an operator-readable diff, and binds approval to the exact base hash and patch payload.

## Reproduction Assessment

Yes. Submit the agent-facing gateway tool with `action: "config.patch"` for `plugins.allow` or `memory.qmd.command`; current source rejects before forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for all non-allowlisted config patches is not the safest maintainable design; config mutation needs dedicated path classification, reviewed-diff approval, and explicit never-promptable paths.
