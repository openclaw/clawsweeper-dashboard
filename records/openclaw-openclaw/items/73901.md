## Summary

Keep this open. Current main still rejects non-allowlisted agent-driven `config.patch` before forwarding the gateway RPC, and the requested approval gate would relax a GHSA-related fail-closed model-to-operator config boundary that needs maintainer security design.

## What This Changes

Add an optional operator approval path or expanded allowlist for agent-driven `gateway config.patch` writes outside the current narrow allowlist.

## Best Possible Solution

Design a config-specific approval policy that preserves reject-by-default behavior, separates promptable from never-promptable paths, shows an operator-readable diff, and binds approval to the exact base hash and payload.

## Reproduction Assessment

Yes, from source inspection. Submit the agent-facing gateway tool with `action: "config.patch"` for `plugins.allow` or `memory.qmd.command`; current main rejects before forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for all non-allowlisted config patches is not the safest maintainable design; config mutation needs dedicated path classification, reviewed-diff approval, and explicit never-promptable paths.
