## Summary

Keep this open. Current main still rejects non-allowlisted agent-driven `config.patch` changes before the gateway RPC, and the requested approval gate would relax a recent GHSA-related fail-closed boundary that needs maintainer security design.

## What This Changes

The issue requests an optional operator approval gate for agent-driven `gateway config.patch` writes that target paths outside the current allowlist.

## Best Possible Solution

Design a config-specific approval policy that remains reject-by-default, separates promptable from never-promptable paths, renders an operator-readable diff, and binds approval to the exact base hash and patch payload.

## Reproduction Assessment

Yes, from source inspection. Submit the agent-facing gateway tool with `action: "config.patch"` for `plugins.allow` or `memory.qmd.command`; current main rejects before forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No, not as proposed. Reusing exec-style approval for all non-allowlisted config patches is too broad; the maintainable path is a dedicated config approval policy with explicit path classification and exact-diff approval binding.
