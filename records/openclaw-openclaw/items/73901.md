## Summary

Keep open. Current main still intentionally rejects non-allowlisted agent-driven config mutations, and the requested approval fallback would change a GHSA-related model-to-operator trust boundary rather than closing an obsolete or already-implemented gap.

## What This Changes

Request to add an operator approval gate for agent-driven `gateway config.patch` edits outside the current allowlist, plus an optional expanded allowlist for trusted paths.

## Best Possible Solution

Add a first-class config mutation approval policy: keep reject-by-default behavior, classify paths as always allowed, promptable, or never promptable, render an operator-readable diff, bind approval to the exact base hash and patch payload, and cover the agent tool, gateway write path, and channel approval delivery with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding the write RPC, and guard tests encode both blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for every non-allowlisted config patch is not the safest maintainable solution; config needs path classification, exact diff/base-hash binding, and explicit never-promptable paths.

## Risks / Open Questions

- A broad approval fallback could authorize plugin loading, tools/elevated policy, MCP, hooks, sandbox, browser, or gateway auth changes unless maintainers define promptable and never-promptable path classes first.
- A config approval flow that is not bound to the reviewed base hash, changed paths, and patch payload could apply a stale or altered mutation after the operator approves it.
- Telegram, Discord, and native approval clients do not currently have a dedicated config approval kind, so reused exec/plugin approval UX could hide config-specific risk details.
