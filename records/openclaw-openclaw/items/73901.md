## Summary

Keep this open. Current main still rejects non-allowlisted agent-driven `config.patch` before forwarding the write, and the approval infrastructure only exposes exec/plugin approvals, so the requested config-specific operator approval gate remains unimplemented and security-sensitive.

## What This Changes

Request adding an optional operator approval flow for agent-driven gateway `config.patch` changes outside the current allowlist, with support for an operator-configured expanded allowlist.

## Best Possible Solution

Keep the fail-closed default and design a dedicated config-patch approval policy with always-allowed, promptable, and never-promptable path classes, operator-readable diffs, exact payload/base-hash binding, and gateway/channel tests.

## Reproduction Assessment

Yes. Submit the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current source rejects it before forwarding, and guard tests cover those blocked paths.

## Solution Assessment

No. Reusing exec approval semantics for all non-allowlisted config patches is not the safest maintainable fix; config mutation needs a dedicated path classification and reviewed-diff approval contract.
