## Summary

Keep this open. Current main still intentionally rejects non-allowlisted agent-driven `gateway config.patch` changes, and the request asks to add a new approval path across a GHSA-related model-to-operator config mutation boundary rather than describing behavior that is already implemented or obsolete.

## What This Changes

Request an optional operator approval gate for agent-facing `gateway config.patch` changes outside the current allowlist, plus an operator-configurable expanded allowlist for paths that should not prompt.

## Best Possible Solution

Define a dedicated config mutation approval policy with explicit path classes: always allowed, promptable with reviewed before/after diff, and never promptable. The approval should be tied to the base hash, changed paths, and exact patch payload, with gateway protocol tests and docs covering channel delivery behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding the write RPC, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for all non-allowlisted config patches is not the best narrow fix; config mutation needs config-specific path classification, reviewed-diff approval, base-hash binding, and explicit never-promptable paths.

## Risks / Open Questions

- Relaxing the fail-closed allowlist without path classification could let a model prompt an operator into approving plugin loading, auth, MCP, hook, sandbox, tool-policy, browser, or credential-adjacent changes that should remain never-promptable.
- A config approval flow needs to bind the approved diff to the exact base hash and patch payload; otherwise stale or altered mutations could be applied after review.
- Channel-native approval UX differs across Telegram, Discord, Slack, and other surfaces, so a config-specific reviewed-diff prompt needs explicit delivery and fallback semantics.
