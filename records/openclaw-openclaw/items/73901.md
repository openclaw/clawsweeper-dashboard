## Summary

Keep open. Current main still intentionally fails closed for agent-driven `config.patch`/`config.apply` outside a narrow allowlist, and the request asks to change a GHSA-related model-to-operator trust boundary rather than describe behavior that is already implemented. The proposal is concrete and useful, but it needs maintainer security/product design before code work.

## What This Changes

Request an optional operator approval gate for agent-facing `gateway config.patch` changes outside the current allowlist, plus configurable expanded allowlist support.

## Best Possible Solution

Add a first-class config mutation approval policy: keep rejection as the default, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch, support an explicit reviewed allowlist, and cover agent-tool, gateway protocol, server, and channel/native approval tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects before forwarding, and guard tests encode those cases.

## Solution Assessment

No. Reusing exec approvals for all non-allowlisted config patches is not the narrowest maintainable solution; this needs config-specific approval semantics, path classification, exact diff/base-hash binding, and explicit never-promptable paths.
