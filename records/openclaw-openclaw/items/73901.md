## Summary

Keep open. Current main still intentionally rejects agent-driven `config.patch` and `config.apply` mutations outside a narrow allowlist, and the request asks to change a GHSA-related model-to-operator trust boundary rather than describe behavior that is already implemented or safely obsolete.

## What This Changes

The issue requests an operator approval gate for non-allowlisted `gateway config.patch` changes, plus an optional operator-configured expanded allowlist.

## Best Possible Solution

Add a first-class config mutation approval policy: keep reject-by-default behavior, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch payload, support an explicit reviewed allowlist, and cover Telegram/Discord/native delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding to the server RPC, and existing guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approvals for every non-allowlisted config patch is not the safest maintainable solution; this needs config-specific approval semantics with path classification, exact diff/base-hash binding, and explicit never-promptable paths.
