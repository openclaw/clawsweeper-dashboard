## Summary

Keep this open. Current main still intentionally blocks agent-driven `config.patch` for non-allowlisted paths, and the requested approval-based expansion would alter a GHSA-related model-to-operator config mutation boundary that needs maintainer security design.

## What This Changes

The issue requests an operator approval flow for `gateway config.patch` when a model attempts to change config paths outside the current narrow allowlist, with optional operator-configured allowlist expansion.

## Best Possible Solution

Design a config-specific approval policy that keeps fail-closed defaults, separates always-allowed, promptable, and never-promptable paths, presents an operator-readable diff, and binds approval to the exact base hash and patch payload.

## Reproduction Assessment

Yes. Submit the agent-facing gateway tool with `action: "config.patch"` for paths such as `plugins.allow` or `memory.qmd.command`; current main rejects the request before the gateway RPC, and guard coverage locks in those blocked cases.

## Solution Assessment

No. Reusing exec-style approval for every non-allowlisted config patch is not the narrowest safe solution; config mutation needs dedicated path classification, reviewed-diff approval, and explicit never-promptable paths.
