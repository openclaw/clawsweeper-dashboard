## Summary

Keep this open. Current main still rejects non-allowlisted agent-driven `config.patch` requests before the write RPC, and the requested approval gate would change a GHSA-related fail-closed boundary rather than finish conservative cleanup.

## What This Changes

Allow agent-facing `gateway config.patch` writes outside the current allowlist after optional operator approval, with a configurable expanded allowlist.

## Best Possible Solution

Design a config-specific approval policy that remains reject-by-default, classifies paths as always allowed, promptable, or never promptable, renders an operator-readable diff, and binds approval to the exact base hash and patch payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for every non-allowlisted config patch is not the safest maintainable solution; config mutation needs dedicated path classification, reviewed-diff approval, base-hash binding, and explicit never-promptable paths.
