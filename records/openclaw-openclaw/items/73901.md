## Summary

Keep open. Current main still intentionally rejects agent-driven `gateway config.patch` and `config.apply` mutations outside the narrow allowlist, and the requested approval fallback would change a GHSA-related model-to-operator trust boundary rather than reflect behavior that is already implemented or safely obsolete.

## What This Changes

The issue requests an optional operator approval flow for `gateway config.patch` paths outside the current allowlist, plus a configurable expanded allowlist for operator-approved paths.

## Best Possible Solution

Add a first-class config mutation approval policy while preserving reject-by-default behavior: classify paths as always allowed, promptable, and never promptable; show an operator-readable diff; bind approval to the exact base hash, changed paths, and patch payload; support an explicit reviewed allowlist; and cover the agent tool, gateway RPC, approval protocol, and channel/native delivery paths with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding to the server RPC, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approvals for every non-allowlisted config patch is not the safest maintainable solution; config writes need config-specific approval semantics, exact diff/base-hash binding, and explicit never-promptable path classes.

  - body: The server write path already requires a config base hash, but a future approval flow should bind operator approval to the reviewed base hash, changed paths, and patch/diff so a stale or altered request cannot apply a different mutation after approval.
