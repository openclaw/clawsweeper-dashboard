## Summary

Keep open. Current main still intentionally rejects non-allowlisted agent-driven config mutations, and the request asks to change a GHSA-related model-to-operator trust boundary rather than describing behavior that is already implemented or obsolete.

## What This Changes

Feature request asks for an optional operator approval gate that lets `gateway config.patch` apply non-allowlisted paths after human approval, with an optional expanded allowlist for trusted paths.

## Best Possible Solution

Add a first-class config mutation approval policy: keep reject-by-default behavior, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch payload, and cover Telegram/Discord/native delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding the write RPC, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for every non-allowlisted config patch is not the safest maintainable solution; config-specific approval semantics need path classification, exact diff/base-hash binding, and explicit never-promptable paths.
