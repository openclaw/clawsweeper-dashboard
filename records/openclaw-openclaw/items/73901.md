## Summary

Keep this open. Current main still intentionally rejects non-allowlisted agent-driven `config.patch` mutations, and the request asks to change a shipped security boundary rather than describing behavior that is already implemented or obsolete.

## What This Changes

Request to add an operator approval flow, plus optional expanded allowlist configuration, for `gateway config.patch` requests that target paths outside the current narrow allowlist.

## Best Possible Solution

Design a config-specific approval policy that keeps reject-by-default behavior, classifies paths as always allowed, promptable, or never promptable, shows an operator-readable diff, binds approval to the exact base hash and patch payload, and covers chat-channel delivery plus gateway protocol tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding the write RPC, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for every non-allowlisted config patch is not the safest maintainable path; config mutation needs dedicated path classification, reviewed-diff approval, base-hash binding, and explicit never-promptable paths.
