## Summary

Keep this open. Current main still rejects non-allowlisted agent-driven `config.patch` requests before forwarding the write RPC, and the requested approval path would change a GHSA-related fail-closed boundary rather than complete conservative cleanup.

## What This Changes

The issue requests an operator approval gate for agent-driven `config.patch` writes outside the current allowlist, with an optional expanded allowlist for trusted paths.

## Best Possible Solution

Design a config-specific approval policy that keeps reject-by-default behavior, classifies paths as always allowed, promptable, or never promptable, shows an operator-readable diff, and binds approval to the exact base hash and patch payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approval semantics for every non-allowlisted config patch is not the safest maintainable solution; config mutation needs dedicated path classification, reviewed-diff approval, base-hash binding, and explicit never-promptable paths.
