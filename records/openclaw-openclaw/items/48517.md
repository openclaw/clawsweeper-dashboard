## Summary

Keep open. Current main and v2026.5.22 still provide ordinary plugin hook veto/approval/parameter rewrite plus subagent delegation, but not a plugin-returned quarantine directive, constrained isolated execution contract, or structured trust metadata; this remains a security-sensitive Plugin SDK design item.

## What This Changes

The issue requests a Plugin SDK/runtime API that lets plugins request isolated execution for untrusted work and receive provenance or trust metadata for the result.

## Best Possible Solution

Keep this as a core Plugin SDK/security design item and split any future implementation into explicit contracts for isolation directives, enforced constraints, result provenance, failure handling, operator policy, docs, and regression tests.

## Reproduction Assessment

Not applicable: this is an additive Plugin SDK/security feature request, not a runtime bug. The gap is source-verifiable from the current hook result types, hook merge path, runtime consumer, plugin subagent runtime, docs, latest release tag, and open related proposal cluster.

## Solution Assessment

Unclear. A core Plugin SDK/runtime contract is the plausible layer for a trust-boundary feature, but maintainers still need to choose the directive shape, enforcement guarantees, operator trust/config model, failure behavior, and metadata standard before implementation would be safe.
