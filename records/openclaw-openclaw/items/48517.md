## Summary

Keep open. Current main and the latest release still support blocking, approval, parameter rewrite, ordinary subagent runs, and configured sandbox-required subagents, but they do not expose a plugin-returned quarantine directive, per-call capability constraints, or structured trust/provenance metadata for isolated results.

## What This Changes

Request a Plugin SDK/runtime API that lets plugins ask OpenClaw to run untrusted work in a constrained isolated context and return provenance or trust metadata.

## Best Possible Solution

Keep this as a core Plugin SDK/security design item for an additive isolation contract with enforceable constraints, explicit failure semantics, and structured trust/provenance metadata.

## Reproduction Assessment

Not applicable: this is an additive Plugin SDK/security feature request, not a runtime bug. The gap is source-verifiable from current hook result types, hook merging, before_tool_call runtime consumption, plugin subagent runtime, gateway dispatch, docs, and the latest release tag.

## Solution Assessment

Unclear. A core Plugin SDK contract is the right level for a trust-boundary feature, but maintainers still need to choose the directive shape, enforcement guarantees, failure behavior, and result metadata before implementation would be safe.
