## Summary

Keep open. Current main still exposes block, approval, and parameter-rewrite hooks plus ordinary subagent/session spawning, but it does not provide a plugin-returned quarantine directive, per-request capability constraints, or structured trust/provenance metadata for isolated results.

## What This Changes

Request an additive core/plugin API that lets plugins ask OpenClaw to run untrusted work in an isolated constrained session and return provenance/trust metadata.

## Best Possible Solution

Keep this as a core/plugin API design item for an additive, documented isolation contract with enforceable constraints, explicit failure semantics, and structured trust/provenance metadata coordinated with the related hook-enrichment work.

## Reproduction Assessment

Not applicable as a feature request. The source verification path is high-confidence: inspect the hook result type, hook runner, before_tool_call runtime, plugin subagent runtime, gateway subagent dispatch, sessions_spawn schema, and docs to confirm the requested contract is not present.

## Solution Assessment

Unclear. A core/plugin API is the right level for a trust-boundary feature, but the directive, enforcement guarantees, and result metadata shape need maintainer/security design before code is written.
