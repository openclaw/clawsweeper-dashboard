## Summary

Keep open. Current main and the latest release still expose `before_tool_call` as block/approval/param-rewrite policy plus ordinary subagent delegation, not a plugin-requestable quarantined execution context with enforced constraints and returned trust metadata; this remains a security-sensitive Plugin SDK design item.

## What This Changes

The issue requests a Plugin SDK/runtime contract that lets plugins ask OpenClaw to run untrusted work in an isolated constrained session and return provenance or trust metadata.

## Best Possible Solution

Keep this as a core Plugin SDK/security design item, then split implementation into targeted PRs for an additive isolation contract with enforced constraints, explicit failure semantics, operator policy, docs, tests, and structured trust/provenance metadata.

## Reproduction Assessment

Not applicable: this is an additive Plugin SDK/security feature request, not a runtime bug. The gap is source-verifiable from the current hook result types, hook merge path, runtime consumption, plugin subagent runtime, docs, tests, and latest release tag.

Unclear. A core Plugin SDK contract is the right layer for a trust-boundary feature, but maintainers still need to choose the directive shape, enforcement guarantees, operator trust/config model, failure behavior, and result metadata before implementation would be safe.
