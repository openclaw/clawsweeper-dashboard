## Summary

Keep open: current main and v2026.6.10 still provide before_tool_call blocking, approval, and param rewriting plus subagent isolation/sandbox primitives, but not a plugin-returned isolation directive with constrained execution and provenance/trust metadata. This remains a public Plugin SDK/runtime security-boundary design item, not conservative cleanup.

## What This Changes

Request a Plugin SDK/runtime hook directive that lets plugins run untrusted tool work in an isolated constrained context and receive provenance or trust metadata.

## Best Possible Solution

Design an additive Plugin SDK/runtime isolation contract that reuses sessions_spawn, sandbox, tool-policy, session, and provenance primitives with explicit enforceable guarantees before exposing it to plugins.

## Reproduction Assessment

Not applicable: this is a feature/API design proposal rather than a broken existing behavior report. Source and release inspection confirm the requested hook result shape and trust metadata contract are absent on current main and v2026.6.10.

## Solution Assessment

Unclear as an implementation path: the direction fits OpenClaw's security and plugin-extension goals, but maintainers still need to choose the API shape, enforcement model, and trust metadata contract. The safest path is to reuse existing sessions_spawn, sandbox, tool-policy, and provenance primitives rather than adding a parallel isolation stack.
