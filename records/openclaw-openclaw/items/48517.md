## Summary

Keep open: current main has adjacent hook blocking/approval, subagent isolation modes, and sandbox enforcement, but it does not provide a plugin-requestable isolation directive that runs untrusted work with constrained capabilities and returns host-authored quarantine/provenance metadata. This remains security-sensitive plugin API and product design work, not conservative cleanup.

## What This Changes

Requests a plugin hook/runtime capability for plugins to ask OpenClaw to run untrusted work in an isolated, constrained session and return provenance or trust metadata.

## Best Possible Solution

Define a maintainer-approved host-authored isolation contract across hook results, subagent/runtime APIs, capability constraints, result metadata, and enforcement before implementing it as a plugin SDK/security boundary.

## Reproduction Assessment

Yes at source level. Current hook result, hook event, plugin subagent runtime, and subagent spawn schemas show adjacent controls but no plugin-requestable quarantine directive with constrained capabilities and returned provenance metadata.

## Solution Assessment

Unclear as an implementation until maintainers settle the security contract. The direction fits the right core/plugin API boundary, but the best fix needs coordinated design with the related hook-enrichment and provenance issues first.
