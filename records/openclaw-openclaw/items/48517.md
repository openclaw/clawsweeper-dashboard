## Summary

Keep open. Current main still has ordinary before_tool_call block/approval/param-rewrite behavior and generic subagent delegation, but no plugin-returned isolation directive, constrained quarantined execution path, or structured trust/provenance result contract.

## What This Changes

Request an additive core/plugin API that lets hooks ask OpenClaw to run untrusted operations in a constrained isolated context and return trust/provenance metadata.

## Best Possible Solution

Keep this as a core/plugin API design item for an additive, documented isolation contract with enforceable constraints, explicit failure semantics, and structured trust/provenance metadata coordinated with the related hook-enrichment work.

## Reproduction Assessment

Not applicable as a feature request. The source verification path is high-confidence: inspect the hook result type, hook runtime, plugin subagent runtime, gateway subagent dispatch, and sessions_spawn schema to confirm the requested contract is not present.

## Solution Assessment

Unclear. A core/plugin API is the right level for this trust-boundary feature, but the exact directive and result metadata shape needs maintainer/security design before implementation.
