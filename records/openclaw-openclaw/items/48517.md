## Summary

Keep open. Current main still has block, approval, and parameter-rewrite hooks plus ordinary subagent/session spawning, but it does not expose a plugin-returned quarantine directive, per-request capability constraints, or structured trust/provenance metadata for isolated results.

## What This Changes

Request plugin-requestable isolated execution for untrusted work, with constraints and provenance metadata returned through `before_tool_call` or a dedicated hook.

## Best Possible Solution

Keep this as a core Plugin SDK/security design item for an additive, documented isolation contract with enforceable constraints, explicit failure semantics, and structured trust/provenance metadata.

## Reproduction Assessment

Not applicable as a feature request. The gap is source-verifiable by inspecting the hook result type, hook runner, before_tool_call runtime, plugin subagent runtime, gateway subagent dispatch, and plugin docs.

## Solution Assessment

Unclear. A core Plugin SDK contract is the right level for a trust-boundary feature, but maintainers need to choose the directive shape, enforcement guarantees, and result metadata before code is safe to write.
