## Summary

Keep open. Current main and the latest release still provide hook veto/approval/parameter rewrite plus ordinary subagent delegation, but not a plugin-returned quarantine directive, constrained execution context, or isolated-result trust metadata; this remains a security-sensitive Plugin SDK design item.

## What This Changes

Issue requests a Plugin SDK/runtime API that lets plugins request isolated execution for untrusted work and receive results with trust/provenance metadata.

## Best Possible Solution

Keep this as a core Plugin SDK/security design item and split any future implementation into explicit contracts for isolation directives, enforced constraints, result provenance, failure handling, operator policy, docs, and regression tests.

## Reproduction Assessment

Not applicable: this is an additive Plugin SDK/security feature request, not a runtime bug. The gap is source-verifiable from current hook result types, hook merge behavior, runtime consumers, subagent runtime types, docs, and the latest release tag.

## Solution Assessment

Unclear. A core Plugin SDK/runtime contract is the plausible layer for a trust-boundary feature, but maintainers still need to choose the directive shape, enforcement guarantees, operator trust/config model, failure behavior, and metadata standard before implementation would be safe.
