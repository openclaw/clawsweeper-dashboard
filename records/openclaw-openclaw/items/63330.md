## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Route restart continuation through the trusted gateway path:** `extensions/gateway-restart/index.ts:36-39`
  - body: This tool asks the model for `sessionKey` and later uses that value for an owner-trusted follow-up, while the current gateway restart path derives session context from runtime state, checks `commands.restart`, and writes the canonical sentinel. A parallel plugin restart authority can target the wrong session and bypass the existing restart boundary, so this should route through the trusted gateway restart/update path or wait for an explicit maintainer API decision.
  - confidence: 0.89
- **[P1] Preserve full follow-up run context:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: `buildFollowupRunForSession` rebuilds only a subset of `FollowupRun.run` and hardcodes owner/workspace/block behavior. The follow-up runner consumes omitted fields such as account, group, sender, channel context, cwd, skillsSnapshot, bashElevated, and reply-delivery mode, so cold plugin turns can resume with broader tools or the wrong workspace/delivery context.
  - confidence: 0.87
- **[P2] Use the canonical restart sentinel instead of a marker file:** `extensions/gateway-restart/index.ts:160-164`
  - body: This writes a new JSON restart marker and deletes it once `enqueueFollowupTurn()` returns true, but that only proves admission to the process-local follow-up queue. Current restart continuation already uses SQLite sentinel state plus expected session checks and queued session delivery, so a crash after enqueue can lose the callback and leaves two restart durability models to maintain.
  - confidence: 0.9
- **[P2] Declare the TypeBox runtime dependency:** `extensions/gateway-restart/package.json:7-14`
  - body: `index.ts` imports `@sinclair/typebox`, but the plugin package manifest has no dependency or peer dependency for it. An installed plugin outside the monorepo's hoisted root dependencies can fail to load before it registers any tool or service, so the owning plugin package needs to declare the runtime dependency.
  - confidence: 0.85
- **[P2] Keep plugin follow-up enqueue idempotent:** `src/plugins/runtime/runtime-followup.ts:30-37`
  - body: The public API documents `false` for deduped turns, but the implementation passes dedupe mode `"none"` into `enqueueFollowupRun`. Duplicate plugin retries or repeated startup service runs can therefore enqueue the same prompt multiple times for one session instead of coalescing it at the API boundary.
  - confidence: 0.78
