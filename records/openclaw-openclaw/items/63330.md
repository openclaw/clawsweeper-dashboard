## Label Justifications

- `P2`: This is a substantial plugin-runtime feature with bounded but important session and gateway restart impact.
- `merge-risk: 🚨 session-state`: The PR reconstructs cold followup runs from persisted session state and can mis-associate workspace, account, group, transcript, or elevated state if fields are incomplete.
- `merge-risk: 🚨 security-boundary`: The PR adds a plugin tool that can restart the gateway and run allowlisted commands, so trusted session routing and restart policy are security-sensitive.
- `merge-risk: 🚨 availability`: The new restart plugin calls `process.exit(0)` for the whole gateway and can affect every active session when enabled.

## Summary

Keep open. The followup-turn API is useful and not implemented on current main, but the current PR is not merge-ready because the new gateway-wide restart tool and cold-session run builder miss current security, session-state, package, and durability invariants.

## What This Changes

The PR adds `runtime.followup.enqueueFollowupTurn()` for plugin-scheduled hot/cold session turns and a disabled-by-default `gateway-restart` bundled plugin that exits the gateway and resumes a session after restart.

## Best Possible Solution

Land a narrower, tested followup-turn API only after cold-session reconstruction preserves current session invariants, and either harden or split the gateway-restart plugin so restart authorization, policy, and callback durability are clear before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the supported TypeBox dependency:** `extensions/gateway-restart/index.ts:4-5`
  - body: This new plugin imports `@sinclair/typebox`, but the repo and extension packages use the `typebox` package and the new package declares no dependency for `@sinclair/typebox`. A standalone plugin install can fail to load before registration, so switch to `typebox` and declare the runtime dependency like sibling plugins.
  - confidence: 0.93
- **[P1] Bind restarts to the trusted runtime session:** `extensions/gateway-restart/index.ts:37-39`
  - body: `sessionKey` is accepted from model tool arguments and later persisted into the restart marker, so a tool call can target an arbitrary session for the post-restart callback. Register this as a context-aware plugin tool and use the runtime-provided session key instead of exposing session routing as model input.
  - confidence: 0.9
- **[P1] Honor the global restart-disable policy:** `extensions/gateway-restart/index.ts:164`
  - body: This tool schedules `process.exit(0)` without checking `commands.restart`, while the built-in gateway restart tool fails closed when that flag is false. Enabling this plugin would bypass an operator's configured restart ban, so check the resolved runtime config before writing the marker or exiting.
  - confidence: 0.88
- **[P1] Preserve the session workspace for cold followups:** `src/auto-reply/reply/queue/build-followup-run.ts:56-58`
  - body: The builder always uses `resolveAgentWorkspaceDir(cfg, agentId)` and never reuses `sessionEntry.spawnedWorkspaceDir` or `spawnedCwd`. Cold followups for spawned/subagent sessions can therefore run commands in the agent's default workspace instead of the session's actual working tree.
  - confidence: 0.9
- **[P1] Carry account and group policy context into the run:** `src/auto-reply/reply/queue/build-followup-run.ts:107-129`
  - body: The reconstructed run omits `agentAccountId`, `groupId`, `groupChannel`/`groupSpace`, and `bashElevated`, even though current followup runs use those fields for account-scoped config, group tool policy, media context, and elevated exec defaults. Cold plugin followups can therefore run with the wrong account or broader/missing policy context.
  - confidence: 0.86
- **[P1] Normalize the persisted session file path:** `src/auto-reply/reply/queue/build-followup-run.ts:82-85`
  - body: When `sessionEntry.sessionFile` exists, this path is trusted directly instead of passing through `resolveSessionFilePath(...)`, bypassing the current containment and stale-path fallback logic. Cold followups should normalize the stored path before using it as the transcript target.
  - confidence: 0.84
- **[P1] Keep restart callbacks durable until processed:** `extensions/gateway-restart/index.ts:221-223`
  - body: The watcher deletes the restart marker as soon as `enqueueFollowupTurn()` returns true, but that only confirms an in-memory enqueue. If the gateway exits again before the queue drains, the callback is lost and the resume-after-restart guarantee breaks.
  - confidence: 0.87
- **[P2] Avoid clobbering concurrent restart markers:** `extensions/gateway-restart/index.ts:160-162`
  - body: The tool writes every restart request to one marker file, so two invocations before the 500 ms exit timer fires can overwrite the first session's callback with the second. Use an atomic create/lock or a per-request durable entry so one session's restart resume is not silently dropped.
  - confidence: 0.8
- **[P2] Restore the mocked global timer:** `extensions/gateway-restart/index.test.ts:23-26`
  - body: This top-level `setTimeout` spy is never restored; `beforeEach` only clears calls. In the repo's non-isolated Vitest workers, later test files can inherit a fake timer implementation and become order-dependent.
  - confidence: 0.82
