## Summary

Keep open: the feature direction is plausible, but the branch still introduces a new public plugin runtime API plus restart and shell-command execution paths with unresolved owner-only, restart-policy, session-context, and durability blockers.

## What This Changes

The branch adds a plugin runtime follow-up turn API, a cold-session follow-up run builder, an opt-in gateway-restart plugin, tests/mocks, lockfile wiring, and small qa-lab/test changes.

## Best Possible Solution

If maintainers want this capability, split the generic follow-up API from the restart plugin, build post-restart continuation on the existing durable restart/session-delivery path, preserve full session policy context, and gate restart execution with owner-only plus commands.restart checks.

## Reproduction Assessment

Yes. The blocking issues are source-reproducible from the PR head against current main: the new restart tool lacks existing restart guardrails, cold follow-up runs omit current session-policy fields, and the qa-lab diff drops a still-supported concurrency path.

## Solution Assessment

No. A plugin follow-up helper may be useful, but reconstructing core reply state independently and adding a separate restart marker is not the narrowest maintainable path while current main already has durable restart continuation plumbing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark gateway_restart owner-only:** `extensions/gateway-restart/index.ts:89-95`
  - body: This tool controls process restart and allowlisted shell commands, but its registration omits ownerOnly. Current tool policy only auto-gates known core names such as gateway, cron, and nodes, so enabling this plugin can expose restart to non-owner tool users.
  - confidence: 0.95
- **[P2] Honor restart-disable policy before exiting:** `extensions/gateway-restart/index.ts:164`
  - body: The plugin writes its marker and schedules process.exit without checking commands.restart. The built-in gateway restart path rejects when that flag is false, so this bypasses an operator's explicit restart-disable policy.
  - confidence: 0.92
- **[P1] Preserve the session workspace for follow-up runs:** `src/auto-reply/reply/queue/build-followup-run.ts:57`
  - body: buildFollowupRunForSession always resolves the agent default workspace instead of preferring sessionEntry.spawnedWorkspaceDir. Cold follow-ups for spawned or subagent sessions can resume in the wrong repository and run commands against the wrong files.
  - confidence: 0.9
- **[P2] Carry session policy context into cold runs:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The reconstructed run omits agentAccountId, groupId/groupChannel/groupSpace, and bashElevated even though the follow-up runner and tool policy consume those fields. Cold plugin follow-ups can use the wrong account, miss group-scoped restrictions, or lose elevated execution defaults.
  - confidence: 0.88
- **[P2] Keep restart callbacks durable until processed:** `extensions/gateway-restart/index.ts:221-223`
  - body: enqueueFollowupTurn returning true only proves an in-memory queue accepted the item. Deleting the marker before the queued turn drains or delivers means a second crash can lose the callback permanently, unlike current main's durable restart-continuation queue.
  - confidence: 0.87
- **[P2] Forward multipass concurrency:** `extensions/qa-lab/src/multipass.runtime.ts:329-368`
  - body: The PR removes concurrency from the Multipass plan and stops appending --concurrency, while the qa-lab CLI still accepts and passes that option. A multipass run with a requested concurrency would silently use the default worker count.
  - confidence: 0.83
- **[P2] Use the supported TypeBox package:** `extensions/gateway-restart/index.ts:4-5`
  - body: Current bundled plugins import from typebox and the new plugin package declares no @sinclair/typebox dependency. A plugin-local install can fail to load this plugin before registration.
  - confidence: 0.87
- **[P2] Reject blank session keys before restarting:** `extensions/gateway-restart/index.ts:128-129`
  - body: The schema accepts any string and the tool writes parsed.sessionKey into the marker before exiting. A blank or whitespace session key can restart the gateway with no usable callback target, so validate it before pre-commands or process exit.
  - confidence: 0.84
- **[P3] Restore the global timer mock:** `extensions/gateway-restart/index.test.ts:24-26`
  - body: The test file replaces global setTimeout at module scope and only clears mocks in beforeEach. With non-isolated Vitest workers, later tests in the same process can inherit the mocked timer implementation.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces restart and shell-command execution paths that do not yet match current owner-only, restart-disable, durable continuation, and session-policy guardrails.

Concerns:

- **[high] Restart tool is callable by non-owner senders:** `extensions/gateway-restart/index.ts:89`
  - body: gateway_restart lacks ownerOnly: true, and generic tool policy will not infer owner-only status for this plugin-specific name. If enabled, non-owner users with plugin tool access could restart the gateway and run allowlisted pre-commands.
  - confidence: 0.95
- **[medium] Restart-disable policy is bypassed:** `extensions/gateway-restart/index.ts:164`
  - body: The plugin restart path does not check commands.restart=false before scheduling process exit, unlike the built-in gateway tool.
  - confidence: 0.92
- **[medium] Cold follow-ups can lose policy context:** `src/auto-reply/reply/queue/build-followup-run.ts:107`
  - body: The follow-up builder omits account, group, and bash elevation fields from reconstructed runs, which can change account-scoped behavior or skip group-scoped tool restrictions compared with the original session.
  - confidence: 0.88
- **[medium] Restart callback durability is weaker than main:** `extensions/gateway-restart/index.ts:221`
  - body: The watcher removes the marker after in-memory enqueue, so a crash before drain or delivery can lose the continuation that current main's durable restart-continuation path preserves.
  - confidence: 0.87
