## Summary

Keep open. Current main still does not expose the proposed `api.runtime.followup.enqueueFollowupTurn()` surface, so the PR is not obsolete or implemented, but the branch is currently merge-conflicted and introduces blocking correctness/security issues around gateway restart authority, durable restart continuation, and reconstructed session context.

## What This Changes

The branch adds a plugin runtime follow-up turn API, a cold-session follow-up run builder, an opt-in `gateway-restart` plugin, related tests/mocks, lockfile wiring, and small qa-lab/test adjustments.

## Best Possible Solution

Keep the PR open for maintainer API/product/security review. If the follow-up API is accepted, rebase it onto current main, build it on the durable session-delivery/restart-continuation path, preserve normal session run context, and either split the gateway-restart plugin or make it match the existing owner-only restart guardrails and tests.

## Reproduction Assessment

Yes. The review defects reproduce by source inspection: the PR head registers a restart tool without owner gating, exits without the current restart-disable check, deletes a marker after in-memory enqueue, and reconstructs follow-up runs without persisted workspace/account/group/elevated context.

## Solution Assessment

No. A generic plugin follow-up API may be the right product direction, but this implementation reconstructs core reply state independently and bundles a security-sensitive restart plugin instead of using current main’s durable restart continuation and guardrail mechanisms.

Full review comments:

- **[P2] Use the supported TypeBox package:** `extensions/gateway-restart/index.ts:4-5`
  - body: Current main’s bundled plugins import `Type` from `typebox`, while this new plugin imports `@sinclair/typebox`. After rebasing onto current main, this can fail or reintroduce the removed package surface; use the repo-supported `typebox` dependency like the other bundled plugins.
  - confidence: 0.86
- **[P1] Mark gateway_restart owner-only:** `extensions/gateway-restart/index.ts:89-95`
  - body: The new restart tool is not registered with `ownerOnly: true`. Current tool policy only auto-gates known core names such as `gateway`, so a non-owner sender could invoke a gateway-wide restart and any allowlisted pre-command when this plugin is enabled.
  - confidence: 0.94
- **[P2] Validate sessionKey before restarting:** `extensions/gateway-restart/index.ts:128-130`
  - body: The tool copies `parsed.sessionKey` into the marker and proceeds to pre-commands and restart without rejecting blank or whitespace session keys. A malformed call can restart the gateway with an unusable callback marker, leaving no session to resume.
  - confidence: 0.82
- **[P2] Honor commands.restart before exiting:** `extensions/gateway-restart/index.ts:164`
  - body: This path writes the marker and schedules `process.exit(0)` without checking the global `commands.restart` setting. The built-in gateway restart tool rejects when that flag is false, so this plugin bypasses an operator’s explicit restart-disable policy.
  - confidence: 0.91
- **[P1] Reuse the persisted session workspace:** `src/auto-reply/reply/queue/build-followup-run.ts:57`
  - body: `buildFollowupRunForSession` always uses the agent default workspace. Spawned sessions persist `spawnedWorkspaceDir`, so a cold follow-up after restart can resume in the wrong repository and run commands against the wrong files.
  - confidence: 0.9
- **[P2] Preserve account, group, and elevated context:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The reconstructed run omits `agentAccountId`, group fields, and `bashElevated`, even though follow-up execution and tool policy consume those fields. Cold plugin follow-ups can therefore use the wrong account, miss group-scoped policy, or lose elevated execution defaults compared with a normal turn.
  - confidence: 0.87
- **[P2] Keep restart callbacks durable until processed:** `extensions/gateway-restart/index.ts:221-223`
  - body: The watcher removes the restart marker as soon as `enqueueFollowupTurn()` returns true, but that only proves an in-memory queue accepted the item. A second crash before drain finishes loses the callback permanently, unlike current main’s durable restart-continuation queue.
  - confidence: 0.86
- **[P2] Forward multipass concurrency to the guest command:** `extensions/qa-lab/src/multipass.runtime.ts:355-368`
  - body: The PR removes `concurrency` from the multipass plan and stops appending `--concurrency`, while the qa-lab CLI still parses and passes that option. `--runner multipass --concurrency <n>` would be silently ignored, changing QA runtime behavior.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces gateway restart and shell-command execution paths that do not yet match current owner-only, restart-disable, and durable continuation guardrails.

Concerns:

- **[high] Restart tool is callable by non-owner senders:** `extensions/gateway-restart/index.ts:89`
  - body: `gateway_restart` lacks `ownerOnly: true`, and generic tool policy will not infer owner-only status for this plugin-specific name. If enabled, this exposes process restart and allowlisted shell command execution to non-owner users who can access plugin tools.
  - confidence: 0.94
- **[medium] Restart-disable policy is bypassed:** `extensions/gateway-restart/index.ts:164`
  - body: The plugin restart path does not check `commands.restart=false`, unlike the built-in gateway tool. Operators who disabled restart actions could still be restarted through this new plugin.
  - confidence: 0.91
- **[medium] Cold followups can lose authorization context:** `src/auto-reply/reply/queue/build-followup-run.ts:107`
  - body: The follow-up builder omits account, group, and elevated-exec context that normal turns preserve. Downstream runtime and tool policy consume those fields, so cold plugin follow-ups can execute under a different routing or policy context from the original session.
  - confidence: 0.86
