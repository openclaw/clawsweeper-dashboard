Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the duplicate gateway restart authority:** `extensions/gateway-restart/index.ts:89-95`
  - body: Current main already routes restart continuation through the `gateway` tool with `commands.restart`, trusted runtime session identity, and shared restart state. This new tool adds a separate marker, `process.exit`, command allowlist, and tool-supplied `sessionKey`, so restart policy and recovery can drift; reuse the existing owner path or wait for a maintainer-approved SDK seam.
  - confidence: 0.9
- **[P1] Preserve full cold-session run context:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The builder reconstructs only a subset of `FollowupRun` and forces `senderIsOwner`, while normal queued runs carry account, group, sender, skills, elevated/bash, workspace/cwd, reply, and streaming context used downstream. Cold plugin follow-ups can resume in the wrong workspace, bypass group/tool policy context, or deliver with different behavior than the original session.
  - confidence: 0.88
- **[P2] Use canonical restart state instead of a JSON marker:** `extensions/gateway-restart/index.ts:160-162`
  - body: Current main stores restart sentinel/intent/handoff state in shared SQLite, and repo policy keeps runtime state out of JSON sidecars. Writing `restart-pending.json` here creates a second restart persistence path that doctor, cleanup, migration, and restart recovery code will not own consistently.
  - confidence: 0.86
- **[P2] Keep callback state until the follow-up is processed:** `extensions/gateway-restart/index.ts:221-223`
  - body: `enqueueFollowupTurn()` returning `true` only confirms an in-memory queue insert, not that the follow-up turn has executed or been delivered. Deleting the marker immediately means a second restart or crash before the queue drains can lose the promised callback permanently.
  - confidence: 0.84
- **[P2] Enable dedupe for plugin follow-up enqueue:** `src/plugins/runtime/runtime-followup.ts:30-36`
  - body: The public API says `false` can mean the item was deduped, but this call passes dedupe mode `"none"`, so duplicate plugin retries or repeated external events can enqueue duplicate follow-up turns instead of being coalesced.
  - confidence: 0.82
