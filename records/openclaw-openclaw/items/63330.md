## Summary

Keep this PR open, but do not merge it as-is. Current main still does not provide the proposed generic plugin follow-up turn API, so the idea is not obsolete; however, the patch introduces owner-gating, restart-policy, durable-continuation, session-context, dependency, and qa-lab regressions on security-sensitive and public plugin API paths.

## What This Changes

The PR adds a plugin runtime follow-up turn API, a cold-session follow-up-run builder, an opt-in gateway-restart plugin, related tests/mocks, lockfile wiring, and small qa-lab/test adjustments.

## Best Possible Solution

Add a documented generic follow-up turn capability only after the API contract is accepted, and build it on the existing session delivery/follow-up machinery so cold turns preserve the same workspace, account, group, elevated, streaming, auth, and transcript normalization context as normal turns. Keep any restart implementation owner-only, commands.restart-aware, durable across crashes, and preferably split from the generic API if it needs separate security review.

## Reproduction Assessment

Yes. The blocking issues reproduce by source inspection against current main: the PR registers a process-restart tool without owner-only metadata, bypasses the built-in restart disable check, reconstructs cold runs without current run-context fields, and removes qa-lab concurrency forwarding.

## Solution Assessment

No. A generic follow-up helper may be a reasonable end state, but this implementation independently rebuilds core reply state and bundles a security-sensitive restart plugin instead of reusing current durable restart-continuation and session-delivery guardrails.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark gateway_restart owner-only:** `extensions/gateway-restart/index.ts:89-95`
  - body: `gateway_restart` is registered without `ownerOnly: true`, and current tool policy only auto-gates known core names like `gateway`. If this plugin is enabled, non-owner senders who can access plugin tools can restart the gateway and run allowlisted pre-commands.
  - confidence: 0.95
- **[P2] Honor commands.restart before exiting:** `extensions/gateway-restart/index.ts:164`
  - body: This path writes the marker and schedules `process.exit(0)` without checking the global `commands.restart` setting. The built-in gateway restart tool rejects when that flag is false, so this plugin bypasses an operator's explicit restart-disable policy.
  - confidence: 0.92
- **[P2] Reject blank session keys before restarting:** `extensions/gateway-restart/index.ts:128-130`
  - body: The schema accepts any string and the tool persists `parsed.sessionKey` before exiting. A blank or whitespace session key can restart the whole gateway with no usable callback target, so validate it before running commands or scheduling exit.
  - confidence: 0.85
- **[P1] Reuse the persisted session workspace:** `src/auto-reply/reply/queue/build-followup-run.ts:57`
  - body: `buildFollowupRunForSession` always resolves the agent default workspace, but spawned sessions persist `spawnedWorkspaceDir`. A cold follow-up after restart can resume in the wrong repository and run commands against the wrong files.
  - confidence: 0.9
- **[P2] Preserve session policy context for cold followups:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The reconstructed run omits account, group, and `bashElevated` context and hardcodes `blockReplyBreak`. Current follow-up execution consumes those fields, so cold plugin follow-ups can use the wrong account/tool policy/elevation defaults or stream differently than normal turns.
  - confidence: 0.88
- **[P2] Keep restart markers until continuation is durable:** `extensions/gateway-restart/index.ts:221-223`
  - body: The watcher deletes the marker as soon as `enqueueFollowupTurn()` returns true, but that only proves an in-memory queue accepted the item. A second crash before drain completion loses the callback permanently, unlike current main's durable restart-continuation queue.
  - confidence: 0.87
- **[P2] Forward multipass concurrency:** `extensions/qa-lab/src/multipass.runtime.ts:337-368`
  - body: The PR removes `concurrency` from the multipass plan and stops appending `--concurrency`, while the qa-lab CLI still parses and passes that option. `--runner multipass --concurrency <n>` would be silently ignored.
  - confidence: 0.84
- **[P2] Use the supported TypeBox package:** `extensions/gateway-restart/index.ts:4-5`
  - body: Current bundled plugins and package metadata use `typebox`, not `@sinclair/typebox`, and this new plugin package declares no runtime dependencies. Installed plugin loading can fail with a missing module; use the repo-supported package and declare any plugin-local dependency if needed.
  - confidence: 0.87
- **[P3] Restore the global timer mock:** `extensions/gateway-restart/index.test.ts:22-24`
  - body: The test file replaces global `setTimeout` at module scope and only clears mocks in `beforeEach`. With non-isolated Vitest workers, later tests in the same process can inherit the mocked timer implementation.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces a restart and shell-command execution plugin that does not yet match current owner-only and restart-disable guardrails.

Concerns:

- **[high] Restart tool is callable by non-owner senders:** `extensions/gateway-restart/index.ts:89`
  - body: `gateway_restart` lacks `ownerOnly: true`, and generic tool policy will not infer owner-only status for this plugin-specific name. If enabled, this exposes process restart and allowlisted shell command execution to non-owner users who can access plugin tools.
  - confidence: 0.95

  - body: The plugin calls `process.exit(0)` without checking `commands.restart`, while current core restart tooling treats that setting as an operator safety control.
  - confidence: 0.92
