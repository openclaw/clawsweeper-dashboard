## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Derive callbacks from trusted session identity:** `extensions/gateway-restart/index.ts:36-39`
  - body: The tool asks the model for `sessionKey` and later uses that value for an owner-trusted follow-up turn. Current gateway restart prefers the runtime `agentSessionKey` so model-supplied params cannot queue work into another session; this path should derive the callback target from trusted tool/session context or a maintainer-approved capability boundary.
  - confidence: 0.92
- **[P1] Use the canonical gateway restart path:** `extensions/gateway-restart/index.ts:160-164`
  - body: This writes its own JSON marker and exits the process, bypassing `commands.restart`, SIGUSR1 scheduling/coalescing, SQLite restart sentinel/handoff state, and durable restart continuation recovery. That creates a second restart authority with different policy and durability behavior.
  - confidence: 0.91
- **[P1] Preserve complete follow-up run context:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The reconstructed run omits fields current follow-up execution uses for routing and policy, including account, group/sender context, `cwd`, elevation, provenance, source-delivery mode, and block-break semantics. Cold resumes can run in the wrong workspace or with wrong delivery/policy behavior.
  - confidence: 0.9
- **[P2] Keep restart continuation durable until delivery:** `extensions/gateway-restart/index.ts:214-223`
  - body: `enqueueFollowupTurn()` only confirms the turn entered the in-memory follow-up queue, but the marker is deleted immediately afterward. A crash before drain or delivery can still lose the continuation; the restart path should use durable session-delivery semantics or keep retry state until delivery is acknowledged.
  - confidence: 0.86
