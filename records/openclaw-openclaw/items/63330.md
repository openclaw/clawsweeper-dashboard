Full review comments:

- **[P1] Remove the duplicate gateway restart authority:** `extensions/gateway-restart/index.ts:89-95`
  - body: Current main already routes restart continuation through the `gateway` tool and `/restart` path with `commands.restart`, trusted runtime session identity, and the shared restart sentinel. This new tool adds a separate marker, `process.exit`, command allowlist, and tool-supplied `sessionKey`, so restart policy and recovery can drift; reuse the existing owner path or wait for a maintainer-approved SDK seam.
  - confidence: 0.9
- **[P1] Preserve full cold-session run context:** `src/auto-reply/reply/queue/build-followup-run.ts:107-119`
  - body: The builder reconstructs only a subset of `FollowupRun` and forces `senderIsOwner`, while normal queued runs carry account, group, sender, skills, elevated/bash, workspace/cwd, reply, and delivery policy fields. Cold resumed turns can run under the wrong workspace, tool policy, account, or reply behavior unless this path preserves the same context normal follow-ups carry.
  - confidence: 0.87
- **[P2] Keep the restart marker until callback processing is durable:** `extensions/gateway-restart/index.ts:221-223`
  - body: `enqueueFollowupTurn()` returning true only confirms an in-memory enqueue. Deleting the marker here means a second restart or crash before the queue drains can permanently lose the promised callback, which breaks the resume-after-restart guarantee.
  - confidence: 0.83
- **[P2] Enable dedupe for plugin follow-up enqueue:** `src/plugins/runtime/runtime-followup.ts:30-36`
  - body: The public type text says `enqueueFollowupTurn` can return false when a run is deduped, but this call hard-disables dedupe with `"none"`. Repeated plugin retries or duplicate external events can therefore enqueue duplicate turns instead of coalescing safely.
  - confidence: 0.78
